## Summary

Keep this PR open. Current main still lacks the proposed plugin followup runtime API and bundled gateway-restart plugin, but the branch is broad, security-sensitive, and has blocking correctness/security issues around restart authorization, restart-disable policy, cold-session context preservation, and an unrelated qa-lab concurrency regression.

## What This Changes

This PR adds `api.runtime.followup.enqueueFollowupTurn()`, a cold-session followup run builder, a bundled `gateway-restart` plugin, related tests/mocks, and small qa-lab/test/generated-hash changes.

## Best Possible Solution

Keep the PR open while maintainers decide the generic plugin followup contract. If accepted, port it onto current main's durable session-delivery/restart-continuation and normal reply-run context preservation, and either split the gateway-restart plugin or make it match the built-in owner-only restart policy, durable sentinel semantics, docs, and focused tests.

## Reproduction Assessment

Yes. The main-branch absence is reproducible with targeted `rg`/source inspection, and the PR bugs are visible in the diff: enabling the plugin would register a non-owner-only restart tool, while cold followups from spawned/group/multi-account sessions would be built without the same context normal reply runs pass to `followup-runner`.

## Solution Assessment

No, not as currently proposed. A generic followup API may be the right direction, but this branch reconstructs session runs independently and bundles a high-risk restart tool; the safer path is to design the API against current durable delivery internals first, then implement the restart use case with the existing core guardrails.

Full review comments:

- **[P1] Mark gateway_restart owner-only:** `extensions/gateway-restart/index.ts:84-88`
  - body: The new `gateway_restart` tool is not registered with `ownerOnly: true`. Current tool policy only auto-gates known core names such as `gateway`, so when this opt-in plugin is enabled a non-owner sender can invoke a gateway-wide restart and any allowlisted pre-command.
  - confidence: 0.93

  - body: This execute path writes a marker and schedules `process.exit(0)` without checking the global `commands.restart` setting. The built-in gateway tool rejects restart when that flag is false, so this plugin bypasses an operator's explicit restart-disable policy.
  - confidence: 0.9
- **[P1] Reuse the persisted session workspace:** `src/auto-reply/reply/queue/build-followup-run.ts:55`
  - body: `buildFollowupRunForSession` always uses the agent default workspace. Spawned/subagent sessions persist `spawnedWorkspaceDir`, so a cold followup after restart can resume in the wrong repository and run commands against the wrong files.
  - confidence: 0.88
- **[P2] Preserve account, group, and elevated context:** `src/auto-reply/reply/queue/build-followup-run.ts:104-117`
  - body: The reconstructed run omits `agentAccountId`, group fields, and `bashElevated`, even though `followup-runner` forwards those fields into tool routing, group policy, and embedded exec defaults. Cold followups can therefore use the wrong account, miss group-scoped policy, or lose elevated exec state compared with a normal turn.
  - confidence: 0.86
- **[P2] Keep restart callbacks durable until processed:** `extensions/gateway-restart/index.ts:218-221`
  - body: The watcher deletes the restart marker as soon as `enqueueFollowupTurn()` returns true, but that only proves an in-memory followup queue accepted the item. A second crash or restart before drain finishes loses the callback permanently; current main's restart continuation path uses the durable session-delivery queue for this reason.
  - confidence: 0.84
- **[P2] Forward multipass concurrency to the guest command:** `extensions/qa-lab/src/multipass.runtime.ts:365`
  - body: The PR removes `concurrency` from the multipass plan and stops appending `--concurrency`, while the qa-lab CLI still parses and passes that option. `--runner multipass --concurrency <n>` would be silently ignored, changing QA runtime behavior.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a gateway restart and shell-command execution path that does not yet match existing owner-only, restart-disable, and session-context guardrails.

Concerns:

- **[high] Restart tool is callable by non-owner senders:** `extensions/gateway-restart/index.ts:84`
  - body: `gateway_restart` lacks `ownerOnly: true`, and the generic tool policy will not infer owner-only status for that plugin-specific name. If enabled, this exposes process restart and allowlisted shell command execution to non-owner users who can access plugin tools.
  - confidence: 0.93
- **[medium] Restart-disable policy is bypassed:** `extensions/gateway-restart/index.ts:91`
  - body: The plugin restart path does not check `commands.restart=false`, unlike the built-in gateway tool. Operators who disabled restart actions could still be restarted through this new plugin.
  - confidence: 0.9
- **[medium] Cold followups can lose authorization context:** `src/auto-reply/reply/queue/build-followup-run.ts:104`
  - body: The followup builder omits account, group, and elevated-exec context that normal turns preserve. Because downstream runtime/tool policy consumes those fields, cold plugin followups can execute under a different routing or policy context than the original session.
  - confidence: 0.84
