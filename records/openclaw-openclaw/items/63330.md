## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor commands.restart before exiting the gateway:** `extensions/gateway-restart/index.ts:164`
  - body: The built-in gateway restart tool refuses restart when `commands.restart=false`, and docs say that setting disables gateway restart tool actions. This new model tool writes the marker and calls `process.exit(0)` without checking that policy, so enabling the plugin reopens restarts in configurations that explicitly disabled them.
  - confidence: 0.91
- **[P2] Use or declare the packaged TypeBox dependency:** `extensions/gateway-restart/index.ts:4-5`
  - body: This extension imports `@sinclair/typebox`, but its package has no dependencies while the repo's packaged extension convention declares `typebox` and the dependency contract requires runtime imports to be declared. Import from `typebox` or add the correct dependency so the packaged plugin does not fail contract checks or external loading.
  - confidence: 0.86
- **[P2] Keep the marker until the callback is durable:** `extensions/gateway-restart/index.ts:221-223`
  - body: `enqueueFollowupTurn()` currently queues into the process-local followup map, but the watcher deletes the restart marker as soon as enqueue returns true. If the gateway exits again before drain/delivery, the marker is gone and the post-restart continuation is lost; keep the marker until durable delivery or use the persistent queue work.
  - confidence: 0.88
- **[P2] Preserve persisted session execution context:** `src/auto-reply/reply/queue/build-followup-run.ts:57`
  - body: The builder reconstructs cold followup runs from agent defaults and a subset of `SessionEntry`, but the runner consumes persisted context such as spawned workspace/cwd, group/account routing, bash elevation, and block break mode. Cold followups for spawned, group, or elevated sessions can resume in the wrong workspace or policy context.
  - confidence: 0.83
- **[P2] Forward multipass concurrency to qa suite:** `extensions/qa-lab/src/multipass.runtime.ts:367`
  - body: Current main accepts `createQaMultipassPlan({ concurrency })` and forwards `--concurrency` to the guest qa suite, and CLI callers still pass that option. The PR-head multipass plan drops the parameter and argument, so `qa suite --multipass --concurrency N` silently falls back to default concurrency.
  - confidence: 0.89
