- **[P1] Mark gateway_restart as owner-only:** `extensions/gateway-restart/index.ts:89-95`
  - body: This tool restarts the whole gateway and can run allowlisted shell precommands, but the registration does not set `ownerOnly: true`. Current tool policy only auto-gates known core names such as `gateway`, so non-owner senders with plugin-tool access could invoke `gateway_restart`.
  - confidence: 0.96

  - body: The execute path writes the marker and schedules `process.exit(0)` without checking `commands.restart`. That lets this plugin restart gateways even when operators disabled restart actions for the core gateway tool.
  - confidence: 0.94
- **[P1] Reuse the persisted session workspace:** `src/auto-reply/reply/queue/build-followup-run.ts:57`
  - body: Cold follow-ups rebuild `workspaceDir` from the agent default instead of preferring `sessionEntry.spawnedWorkspaceDir`. Resuming a spawned or isolated session can therefore run tools in the wrong repo/workspace after restart.
  - confidence: 0.91
- **[P1] Carry session policy context into cold followups:** `src/auto-reply/reply/queue/build-followup-run.ts:107-122`
  - body: The reconstructed run omits account, group, sender, and bash elevated context even though `followup-runner` forwards those fields into reply execution and effective tool policy. Cold plugin turns can run with the wrong account or skip group-scoped restrictions.
  - confidence: 0.88
- **[P2] Keep the restart marker until the callback is durable:** `extensions/gateway-restart/index.ts:221-223`
  - body: Deleting the marker immediately after `enqueueFollowupTurn()` only proves an in-memory queue accepted the item. A second restart or crash before drain/delivery loses the callback permanently, so the marker needs to survive until the continuation is durably processed or queue persistence lands.
  - confidence: 0.9
- **[P2] Forward multipass concurrency into the guest command:** `extensions/qa-lab/src/multipass.runtime.ts:329-341`
