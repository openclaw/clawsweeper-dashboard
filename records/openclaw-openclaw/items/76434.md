## Summary

Keep open. The PR adds useful WhatsApp configured-binding helper surface, but it does not fix the normal `type: "route"` WhatsApp routing path described in #75211, because inbound WhatsApp messages still route through `resolveAgentRoute` rather than the added channel-plugin `bindings` provider.

## What This Changes

The PR adds a WhatsApp channel `bindings` provider, conversation-id normalization helpers/tests, and a changelog entry for WhatsApp binding support.

## Best Possible Solution

Revise the PR so #75211 is handled at the actual WhatsApp route-binding boundary, with regression coverage for the reported config and clear docs or diagnostics for `accountId` versus `peer.id`; keep configured ACP provider support only if maintainers want that separate capability too.

## Reproduction Assessment

Yes, source-reproducible. WhatsApp inbound still calls `resolveAgentRoute` with the sender as `peer.id`, while this PR only adds `plugin.bindings` hooks that the normal route resolver does not use.

## Solution Assessment

No. This is not the best fix for the reported issue because it adds configured-binding provider hooks without changing the normal WhatsApp `type: "route"` path or adding an integration test that proves the reported route now reaches the restricted agent.
