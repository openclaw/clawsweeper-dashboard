## Summary

Keep open: the warning path looks aligned after the numeric fixture fix, but the configured-binding parity portion is still incomplete and security-sensitive because the active WhatsApp inbound handler never invokes the configured-binding route helpers.

## What This Changes

This PR adds a WhatsApp configured-binding provider and helper tests, adds a route-time warning for phone-shaped WhatsApp `match.accountId` rules without `match.peer`, and updates the changelog.

## Best Possible Solution

Keep the diagnostic if maintainers want it, and either fully wire WhatsApp inbound through the configured-binding route/runtime helpers with regression coverage or split/remove the provider-parity change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a WhatsApp `type: "acp"` configured binding can compile via the new provider, but current inbound processing still only uses `resolveAgentRoute` and never applies `resolveConfiguredBindingRoute`.

## Solution Assessment

No. The runtime diagnostic is a narrow fix for `accountId` versus `peer.id` misuse, but the configured-binding provider parity claim is not the best solution until WhatsApp inbound actually routes through the configured-binding helpers or that scope is split out.
