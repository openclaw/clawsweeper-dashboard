## Summary

Keep this PR open: the v2 diagnostic direction is plausible, but the patch still advertises WhatsApp configured-binding parity without applying configured bindings on the WhatsApp inbound route, and one new warning test is internally inconsistent with the predicate it adds.

## What This Changes

The PR adds WhatsApp configured-binding provider helpers/tests, a route-time warning for phone-shaped WhatsApp `match.accountId` rules without `match.peer`, and changelog entries.

## Best Possible Solution

Keep the runtime diagnostic with a valid numeric regression fixture, and either wire WhatsApp inbound through the existing configured-binding route/ready path or remove the provider-parity claims from this PR.

## Reproduction Assessment

Yes. Source inspection shows the configured-binding provider is added but the WhatsApp inbound route path still never calls the configured-binding route helper, and the new warning test fixture cannot satisfy the new phone predicate.

## Solution Assessment

No. A diagnostic warning may be the right low-risk fix for the `accountId` vs `peer.id` misuse, but the provider-parity half remains incomplete and the regression test needs to match the predicate it asserts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route configured bindings through WhatsApp inbound:** `extensions/whatsapp/src/channel.ts:119`
  - body: Adding the WhatsApp `bindings` provider only makes rules compilable; inbound messages still use `resolveAgentRoute` and the group-session wrapper without calling `resolveConfiguredBindingRoute`, unlike the existing configured-binding channels. A WhatsApp `type: "acp"` binding can therefore compile but still leave inbound delivery on the base/default route.
  - confidence: 0.9
- **[P2] Use a phone-shaped sender in the warning test:** `src/routing/resolve-route.test.ts:1286`
  - body: The new warning predicate only accepts an optional plus followed by digits, but this test uses the redacted value `+51XXXXXXXXX`. That means the warning branch is skipped and the assertion expecting `suspicious WhatsApp binding` will fail; use a digit-only E.164 fixture or relax the predicate intentionally.
  - confidence: 0.96
