## Summary

Keep open. The diagnostic direction is useful, but the PR still advertises WhatsApp configured-binding parity without applying configured/runtime binding routing in the WhatsApp inbound handler, and the warning can suggest an unmatchable peer id for formatted phone inputs.

## What This Changes

The PR adds WhatsApp configured-binding helpers/provider registration, a route-time warning for phone-shaped WhatsApp `match.accountId` rules without `match.peer`, tests, and changelog entries.

## Best Possible Solution

Keep the route-level diagnostic, normalize its suggested peer id, and either remove/split the WhatsApp configured-binding provider parity or complete inbound configured/runtime binding routing with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR adds a WhatsApp configured-binding provider, but the current WhatsApp inbound handler still calls only `resolveAgentRoute`, so configured ACP bindings are not enforced inbound.

## Solution Assessment

No. A diagnostic-only fix can be a narrow solution for `accountId` versus `peer.id` misuse, but the retained provider parity and changelog claim are not the best solution until inbound routing uses those helpers or that scope is split out.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route WhatsApp configured bindings through inbound:** `extensions/whatsapp/src/channel.ts:122-130`
  - body: The new provider only makes WhatsApp configured ACP bindings compilable. Inbound WhatsApp messages still resolve through `resolveAgentRoute` in `on-message.ts` and never call `resolveConfiguredBindingRoute` or `resolveRuntimeConversationBindingRoute`, so the advertised configured-binding parity can still leave traffic on the base/default route.
  - confidence: 0.91
- **[P2] Normalize the suggested peer id:** `src/routing/resolve-route.ts:564`
  - body: `looksLikeWhatsAppSenderPhone` accepts accountIds with spaces, dashes, and parentheses, but the warning echoes that raw value as `match.peer.id`. Route binding peer ids are only trimmed while WhatsApp inbound sender ids are normalized to E.164, so following the warning for `+1 (555) 123-4567` would still not match.
  - confidence: 0.86
