## Summary

Keep open. The routing diagnostic is a reasonable narrow addition, but this PR still adds and announces WhatsApp configured-binding provider parity while the current WhatsApp inbound path never applies configured or runtime conversation-binding routing.

## What This Changes

This PR adds a WhatsApp configured-binding provider/helpers/tests, a route-time warning for phone-shaped WhatsApp `match.accountId` rules without `match.peer`, and changelog entries.

## Best Possible Solution

Keep the diagnostic if desired, and either remove/split the provider-parity claim or fully wire WhatsApp inbound through configured/runtime binding routing with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: this PR makes WhatsApp configured bindings compilable, but current WhatsApp inbound still resolves only through `resolveAgentRoute`, so configured ACP bindings are not applied to inbound messages.

## Solution Assessment

No. The diagnostic is a narrow fix for `accountId` versus `peer.id` misuse, but the retained configured-binding provider and changelog claim are not the best solution until inbound routing uses those helpers or that scope is split out.
