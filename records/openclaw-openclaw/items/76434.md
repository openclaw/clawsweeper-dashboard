## Summary

Keep this PR open, but do not merge it as-is. The added WhatsApp provider is useful parity work, but current WhatsApp inbound routing still never consults the configured-binding route helper, and the diff still claims route-rule/#75211 behavior that the PR body now correctly says remains unresolved.

## What This Changes

The PR adds a WhatsApp channel configured-binding provider, conversation-id normalization helpers/tests, and a changelog entry for WhatsApp binding support.

## Best Possible Solution

Implement WhatsApp configured-binding support at the same inbound route boundary used by iMessage, BlueBubbles, LINE, Feishu, and Telegram, with regression coverage, while keeping #75211 open until the normal route-binding semantics are fixed or documented.

## Reproduction Assessment

Yes. Source inspection shows the PR adds provider hooks, but current WhatsApp inbound processing still calls only `resolveAgentRoute` and `resolveWhatsAppGroupSessionRoute`; a WhatsApp configured ACP binding would compile but not be applied to inbound routing.

## Solution Assessment

No. The provider helpers are necessary but not sufficient; the maintainable solution also needs WhatsApp inbound routing to call the configured-binding route/ready path, or the PR should narrow its claims to command-target normalization only.
