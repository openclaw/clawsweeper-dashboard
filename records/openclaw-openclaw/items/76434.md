## Summary

Keep open. The PR adds a WhatsApp configured-binding provider, but the reported bug is normal `type: "route"` WhatsApp routing; current source shows that path still goes through `resolveAgentRoute` and does not use the added provider hooks.

## What This Changes

The PR adds WhatsApp configured-binding helpers/tests, wires a `bindings` provider into the WhatsApp channel plugin, and adds a changelog entry for WhatsApp binding support.

## Best Possible Solution

Revise the PR around the actual WhatsApp normal route-binding boundary, with regression coverage for the reported configuration and clear docs or diagnostics for `accountId` versus `peer.id`; keep the configured-binding provider only if maintainers want that separate ACP capability.

## Reproduction Assessment

Yes, source-reproducible. Current main shows WhatsApp inbound messages still route through `resolveAgentRoute`, while the PR only adds channel-plugin configured-binding hooks that normal route resolution does not consult.

## Solution Assessment

No. This is not the best fix for the reported issue because it addresses the configured ACP/provider path without proving or changing the normal WhatsApp `type: "route"` path from the report.
