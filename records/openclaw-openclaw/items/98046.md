## Summary

Keep open: current main still preserves structured gateway auth failure details, but Android onboarding recovery copy does not render distinct next actions for non-pairing auth failures, and an open closing PR is now the active candidate rather than a reason to close early.

## What This Changes

Android setup should distinguish stale credentials, pending approval, expired QR/bootstrap setup, and scanner permission failures instead of showing a generic authentication-needed recovery state.

## Best Possible Solution

Land or replace the focused Android recovery mapping so existing gateway detail codes and scanner permission failures become actionable UI copy, while leaving new node-ID diagnostic plumbing to https://github.com/openclaw/openclaw/issues/98045 if needed.

## Reproduction Assessment

Yes at source level: current main preserves structured gateway auth details in `GatewayConnectionProblem`, but Android recovery UI still maps several non-pairing auth failures to generic copy. I did not run a live Android device reproduction.

## Solution Assessment

Yes, the best fix is to consume the existing structured gateway details in Android onboarding UI and tests. If exact node approval IDs require new gateway data, that narrower protocol/diagnostic work should stay with https://github.com/openclaw/openclaw/issues/98045.
