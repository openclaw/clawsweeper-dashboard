## Summary

Keep open: current main still preserves structured gateway auth details, but Android setup/status UI collapses several recovery causes into generic copy, and the related open items only cover narrower node approval or platform persistence problems.

## What This Changes

Request Android setup and recovery UI distinguish stale credentials, pending approval, QR/bootstrap expiry, and camera permission failures instead of generic authentication-needed copy.

## Best Possible Solution

Add a focused Android recovery-detail mapper that renders existing gateway detail codes and scanner permission failures with next actions, while leaving any new node-ID plumbing to the narrower gateway diagnostics issue if the current protocol cannot expose it.

## Reproduction Assessment

Yes at source level: Android preserves gateway auth detail codes in GatewayConnectionProblem, but the setup/recovery/status UI still maps non-pairing auth failures to generic copy. I did not run a live Android device reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is to consume the existing structured gateway error details in Android UI rather than inventing another auth path. If exact node approval IDs are missing from the current protocol, that part should stay with https://github.com/openclaw/openclaw/issues/98045 instead of being guessed in the app.
