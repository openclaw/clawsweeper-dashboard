## Summary

Keep open: current main and the latest release still contain the strict `reconcileUnknownSend` requirement for required durable sends, while Slack does not declare that capability. The exact reporter payload should be confirmed because current source and tests also say prepared message-tool sends default to best-effort, so a blanket removal could weaken the durable-delivery contract.

## What This Changes

The issue reports Slack channel `message(action="send")` failures when required durable delivery demands `reconcileUnknownSend`, which Slack does not implement.

## Best Possible Solution

Confirm the route that makes Slack message-tool sends required, then keep ordinary prepared Slack sends best-effort by default unless Slack gets a real `reconcileUnknownSend` implementation backed by tests.

## Reproduction Assessment

No high-confidence exact reproduction yet: source proves Slack required durable sends fail without `reconcileUnknownSend`, but the shown message-tool payload should default to best-effort in current code. A focused repro should capture the actual normalized `bestEffort`/`queuePolicy` values for the failing Slack call.

## Solution Assessment

Unclear: dropping the requirement everywhere is too broad, but preserving best-effort for ordinary prepared Slack sends matches the documented contract. A real Slack reconciliation implementation would also be valid if it can prove uncertain `chat.postMessage` outcomes.
