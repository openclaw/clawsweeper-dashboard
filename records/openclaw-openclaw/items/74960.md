## Summary

Current main and the shipped v2026.4.29 release address the matching stale/wrong-image symptom at the shared agent replay boundary: older media markers and inbound media refs are pruned before prompt construction so a previous image is not rehydrated as a fresh prompt image. The reporter only confirmed v2026.4.26/v2026.4.27, while the matching stale-media fix shipped afterward.

## What This Changes

The issue asks OpenClaw to preserve pasted or channel-provided image bytes so WebChat and WeChat images received by the agent match the image the user sent.

## Best Possible Solution

Keep the shipped v2026.4.29 stale-media-ref pruning behavior and direct any fresh reports to a new issue with confirmation on v2026.4.29 or newer plus captured current-turn bytes versus replayed prompt images.

## Reproduction Assessment

Yes for the stale/cached wrong-image symptom: current tests reproduce old [media attached] and media://inbound markers being present in replay history and verify they are pruned before later prompt construction. I did not run a live browser paste flow because this sweep is read-only.

## Solution Assessment

Yes. The fix belongs at the shared agent replay/prompt boundary rather than only in WebChat paste code, because the report also describes WeChat/openclaw-weixin and an older image resurfacing across channels.
