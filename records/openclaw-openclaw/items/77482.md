## Summary

Keep open: this PR targets a real doctor/auth bug and is a useful implementation candidate, but the new active-profile scan trusts malformed config shapes before the unknown-key repair can run, which can make `doctor --fix` crash on partially invalid configs.

## What This Changes

The PR adds doctor unknown-key cleanup logic that preserves active `auth.profiles` fields for configured providers, fallback model references, and `auth.order` entries, with focused regression tests.

## Best Possible Solution

Keep the retention behavior, but harden the new active-profile discovery against invalid config shapes and document the user-facing doctor fix in the changelog.

## Reproduction Assessment

Yes. The current-main bug is source-reproducible through `stripUnknownConfigKeys` deleting strict-schema unknown `auth.profiles.*` fields, and the PR regression can be reproduced with a unit fixture containing repairable unknown keys plus malformed provider or auth-order values.

## Solution Assessment

No, not as written. The retention direction is the right narrow fix, but the implementation should use `isRecord` and array guards around untrusted config before reading provider API keys or iterating auth-order profile lists.
