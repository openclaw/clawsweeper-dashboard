## What This Changes

The PR changes doctor unknown-key cleanup and tests to preserve unknown `auth.profiles` fields when a provider appears active through configured API keys, model fallbacks, or auth order.

## Best Possible Solution

Keep the current main metadata-repair implementation, close this superseded PR, and let a future release carry the #77400 fix.

## Reproduction Assessment

Yes, from source inspection and the linked #77400 review: the old cleanup path could strip unknown auth-profile fields before any active-profile guard. Current main now has a focused repair path plus regression coverage for provider API keys, fallback refs, and explicit profile refs.

## Solution Assessment

No, this PR is no longer the best way to solve the issue. Current main's `protectActiveAuthProfileConfig` is narrower and more maintainable because it repairs active metadata while preserving the schema contract that credential material lives in auth-profile storage, not unknown `openclaw.json` fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not leave schema-unknown auth secrets in config:** `src/commands/doctor-config-analysis.ts:200-205`
  - body: This branch skips deletion for active `auth.profiles.*` unknown keys, so `doctor --fix` can leave `auth.profiles.openai:default.apiKey` in `openclaw.json` even though the schema only allows profile metadata there and runtime credential resolution reads `auth-profiles.json`. The result is a still-invalid config with secret-like material preserved in the wrong surface instead of a repair.
  - confidence: 0.89
