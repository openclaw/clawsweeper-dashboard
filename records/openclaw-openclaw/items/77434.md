## Summary

Keep open: current main intentionally routes typed Control UI `/new` to New Chat/fresh dashboard session creation while `/reset` is the in-place reset path, so the requested change needs maintainer UX/product judgment rather than conservative cleanup.

## What This Changes

Request that webchat `/new` with `session.dmScope: "main"` reset the active session instead of creating a child dashboard session.

## Best Possible Solution

Maintainers should choose the Control UI slash-command contract, then align `app-chat`, session creation/reset handling, docs, and regression tests with that decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed Control UI `/new` dispatches `new-session`, which calls `sessions.create` and produces a dashboard child session with the previous session as parent.

## Solution Assessment

Unclear. Changing `/new` to reset would contradict current Control UI docs and tests, so the narrow maintainable path is a maintainer decision before implementation.
