## Summary

Keep open: current main still routes typed Control UI `/new` through dashboard session creation, while the Gateway/channel reset path resets the same session key in place. Because current Control UI docs and tests explicitly describe the special-case behavior, changing it needs maintainer UX/product judgment rather than conservative cleanup closure.

## What This Changes

Make typed `/new` in webchat reset the active session in place under shared `dmScope: "main"` instead of creating a child dashboard session.

## Best Possible Solution

Decide the intended Control UI semantics, then either route typed `/new` through the in-place reset path and update tests/docs, or make the dashboard-session split consistently documented as intentional.

## Reproduction Assessment

Yes, from source inspection: typed `/new` in webchat reaches `onSlashAction("new-session")`, which creates a dashboard child session, while `/reset` and Gateway/channel `/new` use the in-place reset path. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear: routing webchat `/new` to `sessions.reset` would be a narrow code change, but it would contradict current Control UI docs and tests. Maintainers should first choose whether webchat should preserve New Chat semantics or match channel reset semantics.
