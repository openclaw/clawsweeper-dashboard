## Summary

Keep open: current main reproduces the reported Control UI/WebChat path from source, but that path is documented and covered by tests as intentional New Chat behavior, so changing it needs maintainer UX/product direction rather than conservative bug cleanup.

## What This Changes

Make typed `/new` in WebChat/Control UI reset the current `dmScope: "main"` session instead of creating a fresh dashboard child session.

## Best Possible Solution

Maintainers should choose the Control UI command contract, then update the implementation, tests, and public docs together so `/new` and `/reset` have unambiguous browser-chat semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` dispatches the local `new-session` action, which calls `sessions.create` and produces an `agent:<agent>:dashboard:<uuid>` child when the current session exists.

## Solution Assessment

Unclear. The requested behavior contradicts current Control UI docs and tests, so the maintainable solution depends on a maintainer decision about whether browser typed `/new` should mean New Chat or in-place reset.
