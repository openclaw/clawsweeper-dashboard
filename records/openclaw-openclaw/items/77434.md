## Summary

Keep open: current main deliberately maps Control UI/WebChat typed `/new` to the New Chat dashboard-session creation path, while `/reset` is the documented in-place reset path; changing this needs maintainer UX/product direction.

## What This Changes

The issue requests that typing `/new` in Control UI/WebChat reset the active `dmScope: "main"` session in place instead of creating a child dashboard session.

## Best Possible Solution

Maintainers should define the Control UI slash-command contract, then either keep `/new` as New Chat and clarify the broader docs, or intentionally change the UI dispatch plus docs/tests so `/new` resets the active session under the chosen conditions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` is parsed as a local Control UI command, dispatches `new-session`, calls `sessions.create`, and receives an `agent:<agent>:dashboard:<uuid>` child key with the current session as parent.

## Solution Assessment

Unclear. Treating `/new` as an in-place reset would contradict current Control UI docs and tests, so the narrow maintainable path is a maintainer decision before changing code.
