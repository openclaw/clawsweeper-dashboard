## Summary

Keep open: current main intentionally treats typed Control UI/WebChat `/new` as New Chat/fresh dashboard session creation, while `/reset` is the documented in-place reset path; changing that needs maintainer UX/product direction rather than conservative bug cleanup.

## What This Changes

The issue asks for typed `/new` in WebChat/Control UI with `session.dmScope: "main"` to reset the active session instead of creating a child dashboard session.

## Best Possible Solution

Maintainers should choose the Control UI command contract, then align the local handler, docs, and tests with the selected `/new` and `/reset` behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` dispatches `new-session`, `createChatSession` calls `sessions.create` with the previous session as parent, and Gateway creates an `agent:<agent>:dashboard:<uuid>` child session.

## Solution Assessment

Unclear. The requested change would reverse documented and tested Control UI behavior, so the narrow maintainable path is a maintainer UX decision before implementation.
