## Summary

Keep open: current main intentionally routes Control UI/WebChat typed `/new` to New Chat/fresh dashboard session creation, while `/reset` is the documented in-place reset path, so changing this needs maintainer product/UX direction rather than conservative bug cleanup.

## What This Changes

Request that Control UI/WebChat `/new` reset the current `dmScope: "main"` session in place instead of creating a child dashboard session.

## Best Possible Solution

Maintainers should decide the Control UI slash-command contract, then align `app-chat`, session creation/reset handling, docs, and tests with that chosen behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` dispatches `new-session`, `createChatSession` passes the current session as parent, and `sessions.create` generates `agent:<agent>:dashboard:<uuid>`.

## Solution Assessment

Unclear. Changing `/new` to reset would contradict current docs and regression tests, so the narrow maintainable path is a maintainer decision before implementation.
