## Summary

Keep open: current `main` still intentionally routes Control UI/webchat typed `/new` to New Chat/dashboard-session creation while `/reset` is the documented in-place reset path, so changing this needs maintainer UX/product direction rather than conservative cleanup.

## What This Changes

The issue requests that Control UI/webchat typed `/new` reset the current main-scoped session instead of creating or switching to a dashboard child session.

## Best Possible Solution

Maintainers should choose the Control UI command contract, then update the handler, docs, and tests consistently: either keep `/new` as New Chat and steer users to `/reset`, or intentionally make `/new` reset the active session under the chosen conditions without creating dashboard children.

## Reproduction Assessment

Yes for the reported behavior. Source inspection shows typed `/new` dispatches `new-session`, creates `agent:<agent>:dashboard:<uuid>`, and preserves `/reset` as the separate in-place reset path.

## Solution Assessment

Unclear. The requested change contradicts current docs and regression tests, so the narrow maintainable solution depends on a maintainer decision about the intended Control UI UX before implementation.
