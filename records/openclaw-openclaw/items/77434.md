## Summary

Keep open: current main intentionally routes Control UI/WebChat typed `/new` to New Chat/fresh dashboard session creation, while `/reset` is the documented in-place reset path, so the requested behavior needs maintainer product/UX judgment rather than conservative bug cleanup.

## What This Changes

Request that WebChat `/new` reset the active `dmScope: "main"` session instead of creating a fresh dashboard child session.

## Best Possible Solution

Maintainers should decide the Control UI command contract: either keep `/new` as New Chat and steer users to `/reset`, or intentionally change the local handler plus docs/tests to reset the active session under the chosen conditions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` is parsed as a local Control UI command, dispatches `new-session`, then creates `agent:<agent>:dashboard:<uuid>` with the previous session as parent.

## Solution Assessment

Unclear. Changing `/new` to reset would contradict current docs and tests, so the narrow maintainable path is a maintainer decision on the intended Control UI UX before implementation.
