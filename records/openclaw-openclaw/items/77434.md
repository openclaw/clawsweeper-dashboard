## Summary

Keep open: current main still source-reproduces the Control UI `/new` behavior, but that behavior is intentionally documented and tested as New Chat while other surfaces describe `/new` as a reset alias, so this needs maintainer command-contract judgment rather than conservative cleanup.

## What This Changes

The issue asks for Control UI/WebChat `/new` to reset the current `dmScope: "main"` session instead of creating a fresh dashboard child session.

## Best Possible Solution

Maintainers should decide the Control UI command contract, then either keep `/new` as New Chat and clarify cross-surface help/docs, or intentionally change the local handler, docs, and tests so `/new` uses the in-place reset path under the chosen conditions.

## Reproduction Assessment

Yes. Source inspection on current main shows typed `/new` is intercepted as a local Control UI command, dispatches `new-session`, and creates `agent:<agent>:dashboard:<uuid>` with the previous session as parent; `/reset` is the in-place `chat.send` path.

## Solution Assessment

Unclear. Changing `/new` to reset would align some channel/ACP/TUI semantics but contradict current Control UI docs and tests, so maintainer UX direction is needed before a narrow patch.
