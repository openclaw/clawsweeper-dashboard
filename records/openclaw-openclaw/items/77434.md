## Summary

Keep open: current main still creates or switches to a Control UI dashboard session for typed `/new`, and changing that to an in-place reset would reverse documented/tested Control UI behavior rather than close a conservative cleanup item.

## What This Changes

The issue asks for Control UI/webchat `/new` under `session.dmScope: "main"` to reset the active session instead of creating or switching to a dashboard child session.

## Best Possible Solution

Maintainers should explicitly choose the Control UI command contract, then align `app-chat`, `createChatSession`, docs, and tests so `/new` and `/reset` have unambiguous, intentionally different or intentionally shared behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: typed `/new` dispatches `new-session`, `createChatSession` calls `sessions.create`, and the Gateway returns a dashboard session key rather than resetting the active key.

## Solution Assessment

Unclear. Changing `/new` to reset may be reasonable, but it contradicts current Control UI docs, tests, and the recently merged create-and-switch direction, so the best next step is maintainer UX/product direction before implementation.
