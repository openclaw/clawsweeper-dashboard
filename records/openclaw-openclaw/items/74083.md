## Summary

Keep open. The PR targets a real sandbox PID 1 problem and `--init` is the right Docker mechanism for new containers, but the patch does not force existing normal or browser sandbox containers through recreation and it is missing the required changelog entry.

## What This Changes

The PR adds Docker `--init` to sandbox container creation arguments and extends create-args tests to assert the flag is present after `docker create`.

## Best Possible Solution

Enable `--init` and pair it with an intentional create-args/hash epoch included in both normal sandbox and sandbox-browser hash inputs, plus focused recreate coverage and a single-line changelog fix entry.

## Reproduction Assessment

Yes for the source-level behavior: current `main` creates long-lived Docker sandboxes without `--init`, and related issue #68691 supplies observed PPID=1 zombie evidence. A live zombie workload was not run in this read-only pass.

## Solution Assessment

No, not as written. Adding Docker `--init` is the right mechanism for newly created containers, but without a hash/epoch rollout existing matching containers can keep running with the old PID 1 behavior.
