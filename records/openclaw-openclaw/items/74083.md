## Summary

Keep open. The PR uses the right Docker mechanism for newly created sandbox containers, but it is incomplete because existing Docker sandbox and sandbox-browser containers can be reused without `--init` when their config hash still matches, and the user-facing sandbox fix also needs a changelog entry.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and extends create-args tests to assert the flag is present after `docker create`.

## Best Possible Solution

Enable Docker `--init` for sandbox creation and pair it with an intentional rollout mechanism, preferably a clearly named create-args/hash epoch included in both normal sandbox and sandbox-browser hash inputs, plus focused recreate/hash coverage and a single-line changelog fix entry.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init`, runs `sleep infinity`, and related issue #68691 provides observed PPID=1 zombie evidence. I did not run a live Docker zombie workload in this read-only review.

## Solution Assessment

No, not as written. Adding `--init` is the right mechanism for newly created containers, but the PR needs a hash/epoch rollout or explicit recreate path so existing affected runtimes do not stay unchanged.
