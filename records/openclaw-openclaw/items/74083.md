## Summary

Keep open. The PR targets a real sandbox PID 1 problem and uses the right Docker mechanism for newly created containers, but current main still reuses matching-hash sandbox runtimes, so the patch does not reliably move existing affected containers onto Docker init.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and adds create-args test coverage asserting the flag appears after `docker create`.

## Best Possible Solution

Enable Docker `--init` for sandbox creation and pair it with an intentional rollout mechanism, preferably a create-args/hash epoch included in both normal sandbox and sandbox-browser hash inputs, with focused recreate/hash tests and a user-facing changelog entry.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init`, runs `sleep infinity` as the long-running command, and related issue #68691 provides observed PPID 1 zombie evidence. I did not execute a live zombie workload in this read-only pass.

## Solution Assessment

No, not as written. Adding `--init` is the right Docker mechanism for newly created containers, but the PR needs a hash/epoch rollout or explicit recreate path so existing affected runtimes do not keep running unchanged.
