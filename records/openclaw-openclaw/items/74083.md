- `P1`: The linked production report describes sandbox PID-1 zombies that can exhaust process slots and break normal agent command execution.
- `merge-risk: 🚨 compatibility`: The PR changes Docker sandbox creation defaults without invalidating existing sandbox config hashes or documenting an upgrade path.

## Summary

This PR is still useful but not merge-ready: it adds the right Docker init flag for newly created sandbox containers, but it does not invalidate the sandbox config hash, so existing long-lived containers can keep running without init after upgrade.

## What This Changes

The branch adds Docker `--init` to sandbox `docker create` arguments and updates create-args tests to assert the flag is emitted after `create`.

## Best Possible Solution

Land or replace this with one coordinated sandbox rollout that adds Docker `--init`, bumps a Docker create-args hash epoch, preserves current hash inputs, covers fresh and existing-container paths, and includes redacted real Docker proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: current main creates Docker sandboxes without `--init`, runs `sleep infinity` as PID 1, and can keep prune-disabled containers alive indefinitely; I did not run a live zombie-producing Docker workload in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. Docker `--init` is the right low-level fix for new sandbox containers, but the best fix also invalidates sandbox hashes so existing long-lived containers pick it up.
