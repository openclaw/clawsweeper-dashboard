## Summary

Keep open: current main and the latest checked release still lack Docker `--init` on sandbox-created containers, so this useful fix is not obsolete. The patch is not merge-ready because it changes only future `docker create` calls; existing long-lived or prune-disabled containers can keep the old PID 1 unless the create-args hash rollout is coordinated, and the external PR still lacks real Docker behavior proof.

## What This Changes

The PR adds Docker `--init` to sandbox container create arguments and updates create-args tests to assert that the flag is present after `docker create`.

## Best Possible Solution

Land or replace this with one coordinated sandbox rollout that adds Docker `--init`, bumps a Docker create-args hash epoch, covers fresh and existing-container paths, and includes redacted real Docker proof.

## Reproduction Assessment

Yes, source-level: current main creates Docker sandboxes without `--init`, runs `sleep infinity` as the container command, and can keep prune-disabled containers alive indefinitely. I did not run a live zombie-producing Docker workload in this read-only review.

## Solution Assessment

No, not as-is. Docker `--init` is the right low-level fix for new sandbox containers, but the best fix also invalidates sandbox hashes so existing long-lived containers pick it up.
