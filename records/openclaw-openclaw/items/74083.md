## Summary

Keep open. The PR targets a real Docker sandbox zombie-process problem, but the current patch is incomplete because it only changes newly created containers and does not force existing matching-hash sandbox or browser containers through recreation.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and updates sandbox create-args tests to assert the flag is present after `docker create`.

## Best Possible Solution

Enable Docker `--init` and pair it with an intentional create-args/hash epoch or equivalent recreation path for both normal sandbox and sandbox-browser containers, with focused tests and a changelog entry.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init`, runs `sleep infinity` as the long-lived container command, and related issue #68691 provides observed PPID=1 zombie evidence. I did not execute a live Docker workload in this read-only pass.

## Solution Assessment

No, not as written. Adding `--init` is the right mechanism for newly created containers, but the PR needs a rollout mechanism so existing affected runtimes do not keep running without Docker init.
