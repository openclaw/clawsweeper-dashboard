## Summary

Keep this PR open: the direct Docker `--init` change is useful, but it is not merge-ready because existing long-lived sandboxes can keep their old config hash and never recreate into the new PID 1 behavior.

## What This Changes

The PR adds Docker `--init` to sandbox `docker create` arguments and updates create-args tests to assert the flag.

## Best Possible Solution

Land or replace this with a coordinated sandbox rollout that adds `--init`, bumps a Docker create-args hash epoch, covers fresh and existing-container paths, and includes redacted real Docker proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main creates Docker sandboxes without `--init`, runs `sleep infinity` as PID 1, and can keep prune-disabled containers alive indefinitely. I did not run a live zombie-producing Docker workload in this read-only review.

## Solution Assessment

No, not as-is. Docker `--init` is the right low-level fix for new sandbox containers, but the best fix also invalidates sandbox hashes so existing long-lived containers pick it up.
