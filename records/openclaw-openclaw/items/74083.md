## Summary

Keep open. The PR uses the right Docker mechanism for newly created sandbox containers, but it is incomplete because existing Docker sandboxes can be reused under the same config hash and therefore keep the old PID 1 until a recreate path is triggered or clearly documented.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and unit coverage asserting the flag is emitted after `docker create`.

## Best Possible Solution

Land the Docker `--init` change with focused create-args coverage, add an explicit Docker sandbox runtime/create-args epoch into the config-hash path so affected existing containers go through the existing recreate or hot-container hint flow, and add the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init`, runs `sleep infinity`, and reuses matching config-hash containers. I did not run a live zombie workload in this read-only review, but the code path and Docker CLI contract are enough to reproduce the missing rollout behavior.

## Solution Assessment

No, not as written. Adding `--init` is the right mechanism for new containers, but the PR is not complete until existing reused containers are forced or guided through recreation and the release/changelog surface is updated.
