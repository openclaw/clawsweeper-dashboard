## Summary

Keep open. The Docker `--init` direction matches the dependency contract and the production evidence, but this branch only affects newly created containers; existing long-lived sandbox and sandbox-browser containers can keep the old PID 1 because the config hash does not change, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and updates create-args tests to assert the flag is present.

## Best Possible Solution

Land Docker `--init` together with a deliberate create-args epoch bump wired into both normal and browser sandbox hashes, then verify the recreated container path with redacted live Docker process proof.

## Reproduction Assessment

Yes, source-level: current main creates sandbox containers as `sleep infinity` without `--init`, the hash cannot change for a create-args-only flag, and https://github.com/openclaw/openclaw/issues/68691 includes PPID=1 zombie evidence. I did not run a live stress workload in this read-only review.

## Solution Assessment

No, not as written. Docker `--init` is the right mechanism for new containers, but the PR must also bump or add a create-args hash epoch so existing long-lived containers are recreated, likely coordinated with https://github.com/openclaw/openclaw/pull/76590.
