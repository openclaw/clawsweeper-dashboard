## Summary

Keep open. The PR targets a real Docker sandbox PID-1 reaping problem and uses Docker's intended `--init` mechanism for newly created containers, but the patch is incomplete because existing matching-hash sandbox and sandbox-browser containers can continue running without init. It also lacks the required changelog entry for a user-facing sandbox fix.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and updates create-args tests to assert the flag appears after `docker create`.

## Best Possible Solution

Enable Docker `--init` for sandbox creation and pair it with an intentional rollout mechanism, preferably a clearly named create-args/hash epoch included in both normal sandbox and sandbox-browser hash inputs, plus focused recreate/hash coverage and a single-line changelog fix entry.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init`, runs `sleep infinity` as the long-running command, and related issue #68691 provides observed PPID=1 zombie evidence. I did not execute a live zombie workload in this read-only pass.

## Solution Assessment

No, not as written. Adding `--init` is the right mechanism for newly created containers, but the PR needs a hash/epoch rollout or explicit recreate path so existing affected runtimes do not keep running unchanged.
