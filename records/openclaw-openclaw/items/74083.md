## Summary

Keep open. The PR targets a real Docker sandbox PID-1 gap and `--init` is the right Docker mechanism for newly created containers, but the current diff does not change the normal or browser sandbox hash inputs, so existing affected containers can be reused unchanged; it also lacks the required changelog entry.

## What This Changes

The PR adds Docker `--init` to sandbox container create args and extends create-args tests to assert the flag is present after `create`.

## Best Possible Solution

Enable Docker `--init` and pair it with a named create-args/hash epoch used by both normal and browser sandbox hashes, focused recreate coverage, and one Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes for source-level reproduction: current main creates long-lived Docker sandbox containers without `--init`, and related issue #68691 provides observed PPID=1 zombie evidence. I did not run a live Docker zombie workload in this read-only review.

## Solution Assessment

No, not as written. Adding `--init` is the narrow Docker-level fix for new containers, but the PR also needs an intentional rollout mechanism so existing affected containers are recreated through the established hash path.
