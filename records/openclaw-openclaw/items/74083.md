## Summary

Keep open: the sandbox PID-1 problem is real and this PR contains useful work, but it is not merge-ready because it only changes future Docker creates, does not invalidate the sandbox config hash for existing long-lived containers, lacks real Docker proof, and is currently conflicting.

## What This Changes

The PR adds Docker `--init` to sandbox container create arguments and updates the create-args test to assert the flag is present after `docker create`.

## Best Possible Solution

Land or replace this with one coordinated sandbox rollout that adds Docker `--init`, bumps a Docker create-args hash epoch, covers fresh and existing-container paths, and includes redacted real Docker proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 create Docker sandboxes without `--init`, run `sleep infinity` as the container command, and can keep prune-disabled containers alive indefinitely. I did not run a live zombie-producing Docker workload in this read-only review.

## Solution Assessment

No, not as-is. Docker `--init` is the right low-level fix for new sandbox containers, but the best fix also invalidates sandbox hashes so existing long-lived containers pick it up.
