## Summary

Keep this PR open. The `--init` mechanism is the right Docker-level fix for newly created containers, and the related zombie-process report provides useful context, but the patch still misses rollout for already-created sandbox containers that are reused by the current config-hash path.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and updates sandbox create-args coverage to assert the flag is present after `docker create`.

## Best Possible Solution

The maintainable end state is to enable Docker `--init` for newly created sandbox containers and pair it with an intentional rollout mechanism, preferably a create-args/hash epoch included in the sandbox and sandbox-browser hash inputs so existing affected containers are recreated through the existing mismatch path.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init` and runs `sleep infinity` as the long-running command, while the linked report shows orphaned zombies under PPID 1. I did not run a live zombie workload in this read-only pass.

## Solution Assessment

No, not as written. Adding `--init` is the right mechanism for new containers, but the patch is incomplete until it either changes the relevant config hash/epoch or documents an explicit recreate requirement for existing runtimes.
