## Summary

Keep open. The PR uses the right Docker mechanism for newly created sandboxes, and related issue #68691 now has reporter confirmation of production impact, but the patch still misses the config-hash rollout needed to recreate long-lived existing containers and also lacks the required changelog entry.

## What This Changes

The PR adds Docker `--init` to sandbox `docker create` arguments and extends create-args tests to assert the flag appears after `create`.

## Best Possible Solution

Enable Docker `--init` and roll it out through an explicit init/create-args epoch included in both normal sandbox and sandbox-browser hash inputs, with focused recreate coverage and a one-line changelog fix entry.

## Reproduction Assessment

Yes for the source-level behavior: current main creates long-running Docker sandboxes without `--init`, Docker documents that `--init` reaps processes, and #68691 provides observed PPID=1 zombie evidence. I did not run a live zombie workload in this read-only pass.

## Solution Assessment

No, not as written. Docker `--init` is the narrow maintainable mechanism for new containers, but the patch must intentionally change the normal and browser sandbox hash inputs so affected long-lived containers are recreated.
