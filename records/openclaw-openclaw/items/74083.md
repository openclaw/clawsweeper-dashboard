## Summary

Keep open. The PR targets a real sandbox zombie-process bug and Docker `--init` is the right mechanism for newly created containers, but the patch is not merge-ready because existing sandbox and sandbox-browser containers are reused when their config hashes still match, so affected long-running runtimes may never receive the new init process.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and updates sandbox create-args coverage to assert the flag appears after `create`.

## Best Possible Solution

Enable Docker `--init` for newly created sandbox containers and pair it with an intentional rollout mechanism, preferably a create-args/hash epoch included in both core sandbox and sandbox-browser hash inputs, plus a focused regression test and changelog entry.

## Reproduction Assessment

Yes for the source-level behavior: current main creates Docker sandbox containers without `--init`, runs `sleep infinity`, and related issue #68691 reports orphaned zombies under PPID 1. I did not run a live zombie-producing workload in this read-only pass.

## Solution Assessment

No, not as written. Adding `--init` is the right primitive, but the patch is incomplete until existing affected containers are forced through the existing mismatch/recreate path or the recreate requirement is made explicit.
