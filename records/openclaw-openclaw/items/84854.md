## Label Justifications

- `P1`: The current sanitizer can deterministically break persisted config and OAuth keyring paths for real containerized deployments that set XDG config roots.
- `impact:data-loss`: The reported failure mode stores credentials/config outside the persisted XDG root, causing loss across restarts.
- `impact:security`: The affected code is the host exec environment security policy and any fix must preserve the override-blocking boundary.

## Summary

Keep open: current main still strips inherited `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` from host-exec subprocess environments, while repository docs and downstream OpenClaw tools rely on operator-set XDG config roots. The report is source-reproducible and appears to need a focused security-policy fix, not cleanup closure.

## What This Changes

The issue asks OpenClaw to preserve operator-inherited XDG Base Directory config variables for subprocesses while still rejecting agent-supplied XDG env overrides.

## Best Possible Solution

Add `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` to the inherited allowlist with explicit rationale and regression coverage, while keeping them blocked as untrusted per-call overrides.

## Reproduction Assessment

Yes, at source level: current policy derivation marks inherited `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` as blocked, and the host exec path feeds `process.env` through that sanitizer before spawning. I did not run an end-to-end gateway command in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to allow only trusted inherited XDG config vars while preserving override rejection and adding policy/rationale/parity tests. A broad sanitizer bypass or removal from the override blocklist would be the wrong direction.
