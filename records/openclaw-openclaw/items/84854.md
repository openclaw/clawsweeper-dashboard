## Label Justifications

- `P1`: The report describes deterministic loss of tool config and OAuth keyring persistence for real containerized deployments on the latest release.
- `impact:data-loss`: The affected XDG config root can cause subprocess tools to write credentials and config outside the persisted directory, losing them on restart.
- `impact:security`: The affected code is the host exec environment sanitizer and any fix changes the security boundary between trusted inherited env and untrusted per-call overrides.

## Summary

Keep open: current main still strips inherited `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` from host exec subprocesses, and the report is well supported by current source, tests, related history, and downstream tool behavior.

## What This Changes

The issue asks OpenClaw to preserve operator-inherited XDG Base Directory config environment variables for subprocesses while still blocking untrusted per-call overrides.

## Best Possible Solution

Add the XDG Base Directory variables to the trusted inherited exception policy, keep untrusted override blocking for dangerous XDG overrides, update TS/Swift parity and baseline tests, and prove a host-exec subprocess receives operator-set XDG config roots.

## Reproduction Assessment

Yes, at source level: current main derives both XDG config vars into `blockedInheritedKeys`, and host exec passes `process.env` through that sanitizer before spawning commands. I did not run a live agent subprocess in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable direction is a policy-level inherited allowlist change plus tests, not a new config knob or bypass. The fix should be reviewed as security-sensitive because it adjusts host exec env inheritance.
