## Label Justifications

- `P1`: The report describes deterministic credential/config persistence breakage for real containerized deployments on the latest main and release.
- `impact:data-loss`: Stripping the inherited config root can make tools write OAuth keyrings and config outside the persisted directory, losing them on restart.
- `impact:security`: The affected surface is the host exec environment sanitizer and any fix changes trusted inheritance versus untrusted override semantics.

## Summary

Keep open: current main and the latest release still strip inherited `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` from host exec subprocesses, and the report is backed by source, tests, shipped behavior, and downstream tool code.

## What This Changes

The issue asks OpenClaw to preserve operator-set XDG Base Directory variables for subprocess inheritance while continuing to block untrusted per-call XDG overrides.

## Best Possible Solution

Add the XDG Base Directory variables to the explicit trusted-inherited exception policy, keep per-call override blocking intact, update generated Swift parity and baseline tests, and prove a host exec subprocess receives operator-set XDG config roots.

## Reproduction Assessment

Yes, at source level: current main derives `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` into `blockedInheritedKeys`, and the host exec sanitizer drops matching inherited base-env keys before subprocess execution. I did not run a live gateway subprocess in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a policy-level inherited allowlist change plus parity and regression tests, not a new config knob or sanitizer bypass. The fix should be reviewed as security-sensitive because it adjusts host exec env inheritance.
