## Label Justifications

- `P1`: The report describes deterministic loss of subprocess config and OAuth keyring persistence for real containerized deployments.
- `impact:data-loss`: Stripping `XDG_CONFIG_HOME` can make tools write credentials and config outside the persisted root, causing loss after restart.
- `impact:security`: The affected code is the host exec environment sanitizer and the repair must preserve the security boundary between trusted inherited env and untrusted overrides.

## Summary

Keep open: current `main` and the latest shipped tag still strip inherited `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` from host exec subprocesses, and an open fixing PR explicitly closes this issue, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue asks OpenClaw to preserve operator-set XDG Base Directory environment variables for host exec subprocesses while continuing to block untrusted per-call overrides.

## Best Possible Solution

Review and land or reject the linked fix, with regression coverage for trusted inherited XDG vars and blocked per-call XDG overrides on the host exec path.

## Reproduction Assessment

Yes, at source level: current `main` derives both XDG config vars into `blockedInheritedKeys`, and host exec passes `process.env` through that sanitizer before spawning commands. I did not run a live agent subprocess in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a policy-level inherited allowlist change plus tests, not a new config knob or bypass. The linked PR appears to be the right implementation path to review before closing this issue.
