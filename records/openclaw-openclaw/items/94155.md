## Summary

Keep open: current main and v2026.6.8 still explain the reported mismatch. Plain `cron list` filters to enabled jobs while `cron status` counts all loaded SQLite rows, and the related open PR only adds a hint around disabled-job hiding rather than settling the list contract.

## What This Changes

The issue asks for `cron list` after JSON-to-SQLite migration to return the same complete job set that `cron status`, the dashboard, and the SQLite table report.

## Best Possible Solution

Decide the plain `cron list` contract, then align the CLI/RPC defaults, docs, tests, and any disable-time hint so list, status, dashboard, and SQLite counts are no longer contradictory.

## Reproduction Assessment

Yes, at source level: current main filters default `cron.list` and `listPage` to enabled jobs, while `cron.status` counts all loaded jobs. I did not run a live current-main repro because this review is read-only, but the shipped report plus source path are enough to explain the mismatch.

## Solution Assessment

Unclear until maintainers choose the contract. Making plain list include all jobs is the narrow code fix for the docs and report, but keeping enabled-only listing and correcting docs/help around `--all` may be the safer compatibility path.
