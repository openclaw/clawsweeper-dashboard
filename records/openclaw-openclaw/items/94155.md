## Summary

Keep open: current main and the latest release still have the reported cron-list visibility mismatch. Source explains the 9-of-41 result when only 9 migrated rows are enabled, but the fix needs a maintainer decision because code/help/tests and public docs disagree on whether plain `cron list` means enabled jobs only or all matching jobs.

## What This Changes

Plain `cron list` should return every migrated SQLite cron job instead of only enabled jobs, matching `cron status` and the `cron_jobs` table.

## Best Possible Solution

Maintainers should choose and document one cron-list contract, then align CLI/RPC defaults, docs/help, tests, and the related hint PR around either enabled-only plus `--all` or all jobs by default.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and only 9 enabled rows, `cron.status` counts all rows while default CLI/RPC `cron.list` uses `includeDisabled: false` and the service filters disabled jobs before returning total/hasMore.

## Solution Assessment

Unclear until maintainers choose the contract. The code change is bounded, but the best fix depends on whether OpenClaw wants plain `cron list` to mean all jobs, enabled jobs, or only an operator view exposed by `--all`.
