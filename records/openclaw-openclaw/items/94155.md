## Summary

Keep open: current main and the latest release still allow the reported list/status mismatch because status counts every loaded SQLite cron row while plain CLI/RPC list defaults to enabled-only results. The related open PR is useful mitigation context, but it preserves the enabled-only default instead of settling the public list contract.

## What This Changes

The issue asks for `cron list` to return all migrated SQLite cron jobs so its count matches `cron status` and the SQLite `cron_jobs` table.

## Best Possible Solution

Maintainers should choose the public `cron list` default, then align the CLI/RPC behavior, docs, tests, UI expectations, and the related disable-hint PR around that contract.

## Reproduction Assessment

Yes, source-level. With 41 loaded SQLite rows and only 9 enabled rows, `cron status` reports the full store while default `cron list` and `listPage` return only enabled jobs unless `--all`, `includeDisabled`, or `enabled: all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The implementation change is bounded, but the best fix depends on whether plain CLI/RPC `cron list` should mean all jobs or enabled jobs only.
