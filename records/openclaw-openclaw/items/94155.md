## Summary

Keep open: current main and v2026.6.9 still preserve the source-level list/status mismatch. `cron status` counts all loaded SQLite cron rows, while default CLI/RPC `cron list` returns enabled jobs unless `--all`, `includeDisabled`, or `enabled: all` is used, and the related open PR only mitigates the disabled-job confusion without settling the default contract.

## What This Changes

The issue reports that after cron storage migration to SQLite, `cron list` returns fewer jobs than `cron status` and the backing SQLite store.

## Best Possible Solution

Maintainers should choose the public cron list contract, then align CLI/RPC behavior, UI defaults, docs, tests, and the disable-hint PR around one default.

## Reproduction Assessment

Yes, at source level. With loaded SQLite rows where only a subset is enabled, `cron status` reports the full store while default `cron list` and `listPage` return enabled jobs unless `--all`, `includeDisabled`, or `enabled: all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The implementation change may be bounded, but the best fix depends on whether plain CLI/RPC `cron list` should mean all jobs or enabled jobs only.
