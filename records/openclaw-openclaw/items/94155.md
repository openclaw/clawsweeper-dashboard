## Summary

Keep open: current main and v2026.6.8 still have a real cron-list visibility mismatch. The code path can explain the reported 9-of-41 result when only 9 rows are enabled, but changing plain `cron list` semantics is compatibility-sensitive because CLI help/tests and an open related PR treat disabled-job hiding as intentional while public docs say plain list shows all matching jobs.

## What This Changes

The issue reports that after legacy cron jobs are migrated into SQLite, plain `cron list` returns only 9 of 41 stored jobs while `cron status`, SQLite, and the Control UI show all 41.

## Best Possible Solution

Maintainers should decide the contract first: either make plain CLI/RPC `cron.list` truly return all jobs and update tests/docs, or keep the enabled-only default and correct the docs/help/reporting around `--all` for SQLite stores.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and only 9 enabled rows, `cron.status` counts all rows while default `cron.list`/`listPage` filters to enabled jobs unless `includeDisabled` or `--all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The narrow code fix is small, but the best solution depends on whether OpenClaw wants default `cron list` to mean all jobs, enabled jobs, or only a documented `--all` operator view.
