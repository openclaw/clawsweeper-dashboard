## Summary

Keep open: current main and v2026.6.9 still allow the reported list/status mismatch because status counts every loaded SQLite cron row while default CLI/RPC list paths filter to enabled jobs. The related open PR is useful mitigation context, but it deliberately preserves the enabled-only default instead of settling whether plain cron list should show all jobs.

## What This Changes

The issue asks for `cron list` after JSON-to-SQLite migration to return every stored cron job rather than only the enabled subset.

## Best Possible Solution

Maintainers should choose the public cron list contract, then align CLI/RPC behavior, UI defaults, docs, tests, and the related disable-hint PR around one unambiguous default.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and only 9 enabled rows, `cron status` reports the full store while default `cron list` and `listPage` return only enabled jobs unless `--all`, `includeDisabled`, or `enabled: all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The implementation change may be bounded, but the best fix depends on whether plain CLI/RPC `cron list` should mean all jobs or enabled jobs only.
