## Summary

Keep open: current main and v2026.6.10 still have a source-level list/status mismatch because status counts every loaded SQLite cron row while the default CLI/RPC list path filters to enabled jobs unless the caller asks for all jobs.

## What This Changes

The issue asks for plain `cron list` to return every migrated SQLite cron job so its count matches `cron status` and the `cron_jobs` table.

## Best Possible Solution

Maintainers should choose the public cron list contract, then align CLI/RPC behavior, docs, tests, and the related disable-hint PR around that single default.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and only 9 enabled rows, current `cron status` reports the full store while default `cron list` and `listPage` return only enabled jobs unless `--all`, `includeDisabled`, or `enabled: all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The code change may be bounded, but the best fix depends on whether plain CLI/RPC `cron list` should mean all jobs or enabled jobs only.
