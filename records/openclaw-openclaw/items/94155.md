## Summary

Keep open: current main and v2026.6.9 still allow the reported mismatch because status counts all loaded SQLite cron rows while plain CLI/RPC list paths filter to enabled jobs unless the caller asks for all jobs. The related open PR is useful mitigation context, but it preserves the enabled-only default instead of resolving the public `cron list` contract.

## What This Changes

The issue asks for `cron list` to return all migrated SQLite cron jobs consistently with `cron status` and the SQLite store.

## Best Possible Solution

Maintainers should choose the public `cron list` contract, then align CLI/RPC behavior, dashboard defaults, docs, tests, and the related disable-hint PR around one unambiguous default.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and only 9 enabled rows, `cron status` reports the full store while default `cron list` and `listPage` return only enabled jobs unless `--all`, `includeDisabled`, or `enabled: all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The docs support the reporter's all-jobs expectation, while current code and the related PR treat enabled-only listing as intentional behavior.
