## Summary

Keep open: current main and v2026.6.10 still have the reported list/status mismatch when some migrated cron jobs are disabled, because status counts all loaded jobs while plain CLI/RPC list defaults to enabled-only output. The remaining blocker is the public `cron list` contract, not missing source evidence.

## What This Changes

The issue asks for `cron list` to return the complete SQLite cron job set after migration, matching `cron status` and the database row count.

## Best Possible Solution

Pick one public `cron list` contract and align CLI/RPC behavior, docs, and regression coverage around it; if all-jobs is the contract, plain list should include disabled jobs or otherwise expose an explicit default filter in the response.

## Reproduction Assessment

Yes, source-level: a store with 41 total jobs and only 9 enabled jobs will produce `cron status.jobs === 41` while plain CLI `cron list` sends `includeDisabled: false` and `listPage()` defaults to enabled-only. I did not run a live migrated store because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public contract. Returning all jobs by default matches current docs and the issue expectation, but it changes the shipped enabled-only behavior, so the maintainable fix must include a contract decision plus aligned docs and tests.
