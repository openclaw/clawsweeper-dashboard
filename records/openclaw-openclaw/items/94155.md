## Summary

Keep open: current main and the latest release still explain the reported mismatch because plain `cron list` returns enabled jobs while `cron status` counts all loaded SQLite-backed jobs. The remaining question is the intended CLI/RPC contract, not whether the report has gone away.

## What This Changes

The issue reports that `cron list` returns only enabled jobs after cron SQLite migration while status and the SQLite table show more stored jobs.

## Best Possible Solution

Maintainers should choose the cron list contract first: either make plain CLI/RPC list return all jobs and update tests/docs, or keep the enabled-only default and correct docs/help/reporting around `--all`.

## Reproduction Assessment

Yes, at source level: with disabled jobs in the loaded SQLite-backed store, `cron.status` counts all rows while default `cron.list` and `listPage` filter to enabled jobs before returning `total` and `hasMore`. I did not run a live reproduction against the reporter's database.

## Solution Assessment

Unclear until maintainers choose the contract. The narrow code fix is small if docs are authoritative, but preserving enabled-only behavior and fixing documentation may be the better compatibility path.
