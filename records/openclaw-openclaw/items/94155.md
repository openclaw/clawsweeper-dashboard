## Summary

Keep open: current main and the latest release still allow this mismatch, because status counts every loaded SQLite cron row while default list paths filter to enabled jobs. The related open PR https://github.com/openclaw/openclaw/pull/78139 is useful context, but it intentionally preserves the enabled-only default instead of settling whether plain list should show all jobs.

## What This Changes

Make plain cron list return all SQLite-backed cron jobs after migration instead of only enabled jobs, matching cron status and the cron_jobs table.

## Best Possible Solution

Maintainers should choose the cron list contract, then align CLI/RPC behavior, docs, tests, and the related hint PR so default list semantics are unambiguous.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and only 9 enabled rows, cron status reports the full store while default cron list/listPage filters disabled jobs unless includeDisabled, enabled=all, or --all is used.

## Solution Assessment

Unclear until maintainers choose the contract. The implementation change is probably small, but the best fix depends on whether plain CLI/RPC cron list should mean all jobs or enabled jobs only.
