## Summary

Keep open: current main and v2026.6.8 still allow the reported 9-of-41 shape when only 9 migrated SQLite rows are enabled, because plain `cron list` filters disabled jobs while status counts all loaded jobs. The remaining work is not a safe automatic fix because current CLI help/tests and the related open PR preserve enabled-only default list semantics, while current docs say plain list shows all matching jobs.

## What This Changes

The issue reports that `cron list` shows only enabled cron jobs after SQLite migration even though `cron status` and SQLite contain all jobs.

## Best Possible Solution

Maintainers should choose the public contract first: either make plain CLI/RPC `cron list` return all jobs and update tests/help, or keep enabled-only as the default and correct docs/reporting around `--all` for migrated SQLite stores.

## Reproduction Assessment

Yes, at source level. With 41 loaded SQLite rows and 9 enabled rows, `cron.status` counts all loaded rows while default `cron.list`/`listPage` filters to enabled jobs unless `includeDisabled`, `enabled: "all"`, or CLI `--all` is used.

## Solution Assessment

Unclear until maintainers choose the contract. The code fix is small, but the best solution depends on whether OpenClaw wants default `cron list` to mean all jobs, enabled jobs, or only a documented operator view behind `--all`.
