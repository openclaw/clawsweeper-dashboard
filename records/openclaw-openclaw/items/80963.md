## Summary

Keep open: current main still routes local OpenAI Codex OAuth through the browser URL path before manual fallback when the fixed callback listener is unavailable. The closest fix PR, https://github.com/openclaw/openclaw/pull/80971, closed unmerged after review found it also needed to preserve a manually usable authorization URL.

## What This Changes

Request clearer handling of OpenAI Codex OAuth when localhost:1455 is occupied, avoiding misleading browser callback failures while moving directly to manual entry with diagnostics.

## Best Possible Solution

Add a provider-owned localhost:1455 availability check that suppresses automatic browser opening on EADDRINUSE, explains the conflict, shows a diagnostic command, immediately prompts manual entry, and still exposes the authorization URL for manual opening.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned OAuth helper still calls onAuth after bind failure, so OpenClaw opens/logs the browser URL and then waits before manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is a local occupied-port preflight or equivalent immediate manual path in the OpenAI provider/plugin wrappers; it should not add a configurable callback port or change token exchange behavior.
