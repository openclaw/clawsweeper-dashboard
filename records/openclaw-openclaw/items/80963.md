## Summary

Keep open: current main and the latest release still expose the OpenAI Codex authorize URL through the local browser opener even when the upstream helper cannot bind localhost:1455, then wait on the delayed local manual fallback. The exact-scope fixing PR was closed unmerged, and the open callback-settled issue covers a different remote prompt lifecycle feature.

## What This Changes

The issue asks OpenClaw to avoid opening the OpenAI Codex browser OAuth URL when localhost:1455 is already occupied and to prompt for manual OAuth entry with clear diagnostics immediately.

## Best Possible Solution

Add a provider-owned busy-port guard in both OpenAI Codex OAuth wrappers that gives clear localhost:1455 diagnostics, preserves the manual authorization URL, and prompts immediately without changing redirect URI, token exchange, remote flow, or config surface.

## Reproduction Assessment

Yes, with source-level confidence: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned helper still invokes onAuth with the authorize URL after bind failure, and current OpenClaw treats that as browser-started before delayed manual fallback. I did not execute the full browser login in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a local-only busy-port preflight or onAuth guard in the OpenAI provider path. The prior PR’s approach was close, but a correct repair must also show or log the manual authorization URL instead of dropping it.
