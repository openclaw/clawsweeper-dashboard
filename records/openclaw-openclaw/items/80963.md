## Summary

Keep open: current main still routes the local OpenAI Codex OAuth flow through a browser URL even when the fixed callback listener is unavailable, so the reported confusing state-failure path remains source-reproducible. The prior related fallback PRs were closed unmerged and do not resolve the narrower UX issue on main.

## What This Changes

Requests clearer immediate handling for OpenAI Codex OAuth localhost:1455 bind conflicts so users are not led into a misleading browser callback state failure before manual paste.

## Best Possible Solution

Add a local-only localhost:1455 availability check before starting the OpenAI Codex browser flow; on EADDRINUSE, explain the conflict, show a diagnostic command or safe owner details, and immediately prompt for manual redirect/code while preserving normal, remote/VPS, TLS preflight, and fixed redirect-URI behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned upstream helper still calls `onAuth` after bind failure, and OpenClaw then waits the local fallback delay before manual paste.

## Solution Assessment

Yes. The narrow maintainable fix is a plugin-owned local port preflight or equivalent immediate manual path before invoking the upstream browser flow; a configurable callback port is not needed for this report.
