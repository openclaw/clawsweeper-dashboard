## Summary

Keep open: current main still lacks a localhost:1455 busy-port preflight or equivalent guard before exposing the OpenAI Codex browser authorization URL, so the reported occupied-port OAuth state confusion remains source-reproducible. The exact-scope fixing PR was closed unmerged, and the open callback-settled issue is a separate remote prompt lifecycle feature.

## What This Changes

OpenAI Codex OAuth should avoid automatically opening the localhost:1455 browser callback when that port is already occupied and should give immediate manual-entry diagnostics instead.

## Best Possible Solution

Add a local-only occupied-port guard in the OpenAI provider path that gives clear diagnostics, preserves the manual auth URL, and prompts for redirect/code immediately without changing the fixed redirect URI or remote flow.

## Reproduction Assessment

Yes. Source inspection plus the pinned @earendil-works/pi-ai 0.74.1 contract gives a high-confidence path: occupy 127.0.0.1:1455 and run OpenAI Codex OAuth login, and the wrapper still opens/logs the authorize URL before delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is a provider-owned busy-port preflight or onAuth guard that avoids opening the occupied localhost redirect while still showing the manual authorization URL; a configurable callback port is not needed.
