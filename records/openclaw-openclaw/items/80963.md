## Summary

Keep open: current main still opens/logs the OpenAI Codex OAuth browser URL after the upstream localhost:1455 bind fallback, so the confusing wrong-process callback path remains source-reproducible. The same-scope fix PR was closed unmerged, and the newer remote callback-settled issue tracks a different remote prompt lifecycle problem.

## What This Changes

The issue asks OpenClaw to make occupied localhost:1455 OpenAI Codex OAuth conflicts explicit, avoid opening the misleading browser callback path, and prompt for manual redirect/code entry immediately.

## Best Possible Solution

Add a local-only occupied-port path that avoids calling the browser opener, explains the localhost:1455 conflict, shows the manual authorization URL and diagnostic command, prompts immediately for redirect/code, and preserves remote/VPS, TLS preflight, token exchange, and fixed redirect-URI behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned upstream helper still calls onAuth after bind failure, and OpenClaw then opens/logs the URL before waiting for the local fallback delay.

## Solution Assessment

Yes. The narrow maintainable fix is a plugin-owned local port preflight or equivalent no-browser branch before the local opener runs; it should preserve a manual authorization URL rather than adding a configurable callback port.
