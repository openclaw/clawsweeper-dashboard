## Summary

Keep open: current main still lets the pinned OpenAI Codex OAuth helper call OpenClaw's local `onAuth` handler after a localhost:1455 bind failure, so the browser/open-url path can still point at the occupied callback port before manual paste. The later fix attempt at https://github.com/openclaw/openclaw/pull/80971 was closed unmerged, and the adjacent open remote-prompt issue tracks a different callback-settled contract.

## What This Changes

The issue asks OpenAI Codex OAuth to avoid opening a browser flow to localhost:1455 when that fixed callback port is already occupied, and to switch immediately to clearer manual-entry guidance.

## Best Possible Solution

Keep this issue open and land a narrow OpenAI-plugin-owned occupied-port path that suppresses automatic browser opening, preserves a manual authorization URL, explains the port conflict, and prompts immediately without changing the fixed redirect URI.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455, run `openclaw models auth login --provider openai-codex`, and the pinned helper still calls `onAuth` with the authorize URL after bind failure, causing OpenClaw to open/log it and wait the local fallback delay.

## Solution Assessment

Yes. The narrow maintainable fix is a local busy-port preflight or equivalent busy-port flag in the OpenAI Codex OAuth wrapper that suppresses `openUrl`, logs manual-open instructions and diagnostics, and prompts immediately while preserving normal local, remote/VPS, TLS preflight, redirect URI, and token exchange behavior.
