## Summary

Keep open: current main and the latest release still lack a localhost:1455 busy-port preflight before exposing the OpenAI Codex browser authorization URL, so the reported confusing occupied-port path remains source-reproducible. The exact-scope fix PR was closed unmerged, and the open callback-settled issue covers a different remote prompt lifecycle problem.

## What This Changes

The issue requests clearer OpenAI Codex OAuth behavior when localhost:1455 is already occupied, avoiding a misleading browser callback state failure and prompting manual entry immediately.

## Best Possible Solution

Add a local-only occupied-port preflight or equivalent onAuth guard that shows clear diagnostics, preserves a manual authorize URL, suppresses the automatic browser opener for the busy-port case, and prompts for redirect/code immediately without changing the fixed redirect URI or adding a callback-port setting.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned helper still calls onAuth after bind failure, and current OpenClaw then treats the browser flow as started before delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable solution is a provider-owned local port preflight or onAuth guard that avoids opening the occupied localhost redirect while still showing a manual authorization URL and prompt; a configurable callback port is not needed.
