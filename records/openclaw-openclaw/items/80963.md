## Summary

Keep open: current main and the latest shipped tag still expose the OpenAI Codex browser authorization URL through the local `onAuth` path after the dependency's callback listener fails, so an occupied localhost:1455 can still produce the reported confusing browser state failure before delayed manual fallback.

## What This Changes

The issue asks OpenAI Codex OAuth to handle a busy localhost:1455 callback port with clear diagnostics and immediate manual paste guidance instead of opening a misleading browser callback flow.

## Best Possible Solution

Add a local-only occupied-port guard in both OpenAI Codex OAuth wrappers that keeps the fixed redirect URI, skips automatic `openUrl` when localhost:1455 is busy, logs the manual authorization URL plus `lsof` diagnostic guidance, and prompts for manual entry immediately.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy `127.0.0.1:1455`, run `openclaw models auth login --provider openai-codex`, and the pinned helper still calls `onAuth` before the delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is an OpenAI provider-owned busy-port guard that avoids automatic browser opening but preserves manual URL exposure; a configurable callback port is unnecessary.
