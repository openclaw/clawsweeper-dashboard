## Summary

Keep open: current main still allows the occupied localhost:1455 path to open the browser URL and then wait before manual paste, so the narrower confusing state-failure UX remains source-reproducible. The related port-conflict PRs were closed unmerged or addressed only adjacent remote/manual-routing behavior, and the newer OpenAI auth-default change does not remove this local port-conflict path.

## What This Changes

The issue requests clearer OpenAI Codex OAuth handling when localhost:1455 is occupied so the CLI does not open a misleading browser callback and instead guides manual paste immediately.

## Best Possible Solution

Add a local-only occupied-port preflight or equivalent bind-failure signal before opening the OpenAI Codex browser URL; on EADDRINUSE, explain the conflict, show a diagnostic command or safe owner details, and prompt for manual redirect/code immediately while preserving the fixed redirect URI, normal local OAuth, remote/VPS, TLS preflight, and device-code flows.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455, run `openclaw models auth login --provider openai-codex`, and the pinned helper can still call onAuth with the browser URL while OpenClaw then waits the local fallback delay before manual paste.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned local port-conflict detection or an equivalent upstream bind-failure signal that switches directly to manual guidance; a configurable callback port is unnecessary for this report.
