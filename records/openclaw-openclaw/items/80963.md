## Summary

Keep open: current main and v2026.5.18 still lack a busy-port guard before local OpenAI Codex OAuth exposes the authorize URL to the browser opener, and the upstream helper still calls `onAuth` after a localhost:1455 listen error. The exact-scope fixing PR at https://github.com/openclaw/openclaw/pull/80971 was closed unmerged, so this remains a narrow source-reproducible bug.

## What This Changes

The issue requests clearer OpenAI Codex OAuth handling when the fixed localhost callback port 1455 is already occupied.

## Best Possible Solution

Implement an OpenAI-provider-owned localhost:1455 availability check or equivalent `onAuth` guard that suppresses automatic browser opening, shows clear port-conflict guidance plus the manual authorize URL, and keeps the bundled runtime copy aligned.

## Reproduction Assessment

Yes. Source inspection against current main and `@earendil-works/pi-ai@0.75.1` gives a high-confidence path: occupy 127.0.0.1:1455, run the OpenAI Codex OAuth login, and the helper still calls local `onAuth` before delayed manual fallback.

## Solution Assessment

Yes. The best fix is a narrow provider-owned busy-port guard or preflight that avoids opening the occupied localhost callback while preserving manual URL/code entry; a configurable callback port is not needed for this report.
