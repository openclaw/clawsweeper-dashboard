## Summary

Keep open: current main still exposes the OpenAI Codex authorization URL through the local browser opener when the fixed localhost:1455 callback cannot bind, so the confusing wrong-local-process callback path remains source-reproducible. The exact-scope fixing PR was closed unmerged, and the other open related item tracks a separate remote prompt lifecycle feature.

## What This Changes

The issue requests clearer OpenAI Codex OAuth handling when localhost:1455 is occupied, avoiding an automatic browser redirect to the wrong local process and moving directly to manual entry with diagnostics.

## Best Possible Solution

Add a provider-owned occupied-port guard that keeps the fixed redirect URI, suppresses automatic browser opening only when localhost:1455 is busy, shows clear diagnostics plus the manual authorization URL, and prompts for redirect/code immediately.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned helper still calls onAuth with the authorization URL after the bind failure, and current OpenClaw opens/logs that URL before delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is a local occupied-port preflight or equivalent onAuth guard in the OpenAI provider path, preserving manual URL exposure and avoiding a configurable callback port or broader OAuth ownership change.
