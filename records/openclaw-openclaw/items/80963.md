## Summary

Keep open: current main still exposes the OpenAI Codex authorization URL through the local browser path after the upstream helper has failed to bind localhost:1455, so the reported confusing occupied-port flow remains source-reproducible. The exact-scope fixing PR was closed unmerged, and the open callback-settled issue covers a different remote prompt lifecycle problem.

## What This Changes

The issue asks OpenClaw to handle a busy localhost:1455 during OpenAI Codex OAuth with clear diagnostics and immediate manual entry instead of opening a browser flow that redirects to the wrong process.

## Best Possible Solution

Add a local-only occupied-port preflight or equivalent onAuth guard in the OpenAI Codex OAuth wrapper and bundled runtime that gives clear diagnostics, preserves the manual authorization URL, and prompts immediately without changing the fixed redirect URI or remote flow.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455, run `openclaw models auth login --provider openai-codex`, and the pinned helper still calls `onAuth` before manual input while OpenClaw opens/logs the browser URL and waits before prompting.

## Solution Assessment

Yes. The best fix is a narrow provider-owned busy-port preflight or onAuth guard that avoids opening the occupied localhost redirect, keeps the manual authorize URL available, and leaves remote/VPS OAuth behavior unchanged.
