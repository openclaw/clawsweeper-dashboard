## Summary

Keep open: current main still lacks a localhost:1455 busy-port preflight or equivalent guard before exposing the OpenAI Codex browser authorization URL, so the reported confusing occupied-port path remains source-reproducible. The nearest fixing PR was closed unmerged, and the open related callback-settled issue tracks a different remote prompt lifecycle feature.

## What This Changes

The issue requests clearer local OpenAI Codex OAuth handling when the fixed localhost:1455 callback port is already occupied.

## Best Possible Solution

Add a local-only occupied-port preflight or equivalent guard that gives clear diagnostics, preserves the manual auth URL, and prompts for redirect/code immediately without changing the fixed redirect URI or remote flow.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned helper still calls onAuth after bind failure, and current OpenClaw then treats the browser flow as started before delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable solution is a plugin-owned local port preflight or onAuth guard that avoids opening the occupied localhost redirect while still showing the manual URL and prompt; a configurable callback port is not needed.
