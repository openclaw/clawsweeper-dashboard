## Summary

Keep open: current main still exposes the OpenAI Codex OAuth browser URL through the local opener path after the fixed localhost callback listener is unavailable, so the occupied-port state-failure UX remains source-reproducible. The same behavior is present in the latest release, and the direct fix PR was closed unmerged.

## What This Changes

The issue requests clearer OpenAI Codex OAuth handling when localhost:1455 is already occupied: diagnose the conflict, avoid misleading browser callback errors, and prompt for manual OAuth immediately.

## Best Possible Solution

Add a local-only localhost:1455 availability check or equivalent preflight before automatic browser opening; on EADDRINUSE, explain the conflict, show the lsof diagnostic, expose the authorize URL for manual use, and prompt immediately without changing redirect URI, token exchange, remote flow, or TLS preflight behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the pinned upstream helper still calls `onAuth` after bind failure, and current OpenClaw then follows the local browser-open plus delayed manual fallback path.

## Solution Assessment

Yes. The narrow maintainable repair is plugin-owned occupied-port handling before the automatic local opener, while preserving manual URL visibility; a configurable callback port or broader OAuth rewrite is unnecessary for this bug.

## Repair Work Prompt

Fix the occupied localhost:1455 OpenAI Codex OAuth UX bug tracked by https://github.com/openclaw/openclaw/issues/80963. Current main still lets the upstream helper call `onAuth` after the local callback bind fails, which opens/logs the browser URL and then waits 15 seconds before manual entry. Add a local-only availability check or equivalent guarded path in `src/plugins/provider-openai-codex-oauth.ts` and the bundled copy `extensions/openai/openai-codex-oauth.runtime.ts` before invoking the automatic local opener. On EADDRINUSE, print a clear port-conflict diagnostic including `lsof -nP -iTCP:1455 -sTCP:LISTEN`, explain that `Invalid or expired OAuth state` can come from the wrong local listener, suppress automatic `openUrl`, still show/log the authorize URL for manual opening, and prompt for manual redirect/code immediately. Do not add a configurable callback port, change the fixed redirect URI, change token exchange/refresh semantics, regress remote/VPS OAuth behavior, alter TLS preflight behavior, import core private helpers into the plugin runtime, or broaden provider API. Add/update focused regression tests in `src/plugins/provider-openai-codex-oauth.test.ts`, including busy-port/no-opener/manual-URL-available/immediate-prompt behavior and preservation of the existing delayed local fallback when the browser flow really starts. Add a user-facing changelog entry unless maintainer policy says to handle it at landing.
