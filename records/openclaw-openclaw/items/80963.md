## Summary

Keep open: current main and the latest release still allow the occupied-port path to call the browser opener before delayed manual fallback, so the confusing localhost:1455 state-failure path remains source-reproducible. The focused fixing PR was closed unmerged, and the open related callback-settled issue tracks a different remote prompt lifecycle problem.

## What This Changes

The issue asks OpenClaw to handle an occupied OpenAI Codex OAuth callback port with clear diagnostics and immediate manual entry instead of opening a browser URL that redirects to localhost:1455.

## Best Possible Solution

Add a plugin-owned local localhost:1455 availability guard that suppresses the browser opener on EADDRINUSE, prints an explicit diagnostic and lsof command, and immediately prompts for manual redirect/code without changing the fixed redirect URI or remote flow.

## Reproduction Assessment

Yes. Source inspection plus the pinned @earendil-works/pi-ai 0.75.1 contract gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the dependency still calls onAuth after listen failure, so OpenClaw opens/logs the browser URL before delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is to guard the OpenAI Codex local callback path before invoking the browser opener, keep the manual paste flow, and avoid adding a configurable redirect port or changing token exchange semantics.
