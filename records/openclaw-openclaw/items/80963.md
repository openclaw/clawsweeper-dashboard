## Summary

Keep open: current main still exposes the local OpenAI Codex OAuth browser URL before manual fallback when the fixed callback listener is unavailable, and the linked fix PR was closed unmerged after a review blocker, so the narrow UX bug remains.

## What This Changes

The issue requests clearer OpenAI Codex OAuth handling when `localhost:1455` is occupied, avoiding misleading browser state failures and prompting manual paste immediately.

## Best Possible Solution

Add a local-only `localhost:1455` availability check in the OpenAI-owned OAuth wrappers that suppresses automatic browser opening on `EADDRINUSE`, shows clear conflict diagnostics plus a manual authorization URL, and prompts immediately without changing the fixed redirect URI or token exchange.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: bind `127.0.0.1:1455` and run `openclaw models auth login --provider openai-codex`; the pinned helper still calls `onAuth` after bind failure and OpenClaw then follows the delayed local manual fallback path.

## Solution Assessment

Yes. The narrow maintainable fix is an OpenAI plugin-owned busy-port preflight or equivalent immediate manual path; a configurable callback port is outside this report and unnecessary for the registered redirect URI constraint.
