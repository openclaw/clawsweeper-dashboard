## Summary

Keep open: current main still lets the upstream OpenAI Codex OAuth helper call the local browser opener after a localhost:1455 bind failure, so the confusing wrong-process callback path remains source-reproducible. The direct fix PR was closed unmerged after review feedback, leaving this issue still actionable.

## What This Changes

The issue asks OpenClaw to avoid opening the OpenAI Codex OAuth browser URL when localhost:1455 is already occupied and instead give immediate manual-entry guidance.

## Best Possible Solution

Add a local-only localhost:1455 availability preflight or equivalent guard in the OpenAI plugin OAuth path, suppress automatic browser opening on EADDRINUSE, preserve the authorization URL for manual use, and prompt immediately with clear diagnostics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455, run `openclaw models auth login --provider openai-codex`, and the pinned helper still invokes OpenClaw's onAuth URL handler before manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-owned occupied-port handling before exposing the browser opener, while preserving manual URL/code entry and avoiding a configurable callback port.
