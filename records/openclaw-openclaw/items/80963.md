## Summary

Keep open: current main still routes the local OpenAI Codex OAuth path through the upstream browser auth callback after localhost:1455 bind failure, then waits before manual entry, so the confusing occupied-port state-failure path remains source-reproducible. The same-scope fix PR was closed unmerged after review found it dropped the manual authorization URL, so there is no merged implementation to close this issue against.

## What This Changes

Request clearer OpenAI Codex OAuth handling when localhost:1455 is occupied: avoid opening a misleading browser callback to the wrong local process and prompt for manual redirect/code immediately with diagnostics.

## Best Possible Solution

Implement a plugin-owned local localhost:1455 availability guard before the local browser opener runs; on EADDRINUSE, keep the authorize URL available for manual opening, print clear conflict and diagnostic guidance, skip openUrl, and prompt for manual redirect/code immediately in both the core wrapper and bundled OpenAI runtime copy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: occupy 127.0.0.1:1455, run `openclaw models auth login --provider openai-codex`, and the pinned upstream helper still emits onAuth with the authorize URL before OpenClaw waits for the local manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is an OpenAI-plugin-local port availability gate or equivalent onAuth guard that avoids the wrong-process browser callback while preserving the manual URL; a configurable callback port is outside this report.
