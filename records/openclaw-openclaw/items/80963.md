## Summary

Keep open: current main still exposes the OpenAI Codex authorization URL through the local browser opener after the upstream helper fails to bind localhost:1455, so the confusing wrong-process callback path remains source-reproducible. The exact-scope fixing PR was closed unmerged, and the open related callback-settled issue covers a different remote prompt lifecycle problem.

## What This Changes

The issue asks OpenClaw to avoid opening the OpenAI Codex OAuth browser flow on an already-occupied localhost:1455 callback port and instead provide clearer immediate manual-auth diagnostics.

## Best Possible Solution

Add a local-only localhost:1455 occupied-port preflight or equivalent onAuth guard in the OpenAI provider and bundled runtime copy, emit clear conflict diagnostics plus a manual authorization URL, and prompt immediately without changing the fixed redirect URI or token exchange flow.

## Reproduction Assessment

Yes by source inspection. Occupying 127.0.0.1:1455 makes the pinned helper fall back while still invoking onAuth with the authorize URL, and current OpenClaw then opens/logs that URL before the delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned busy-port handling that avoids openUrl on the occupied callback port while preserving the manual URL; a configurable callback port is not needed.
