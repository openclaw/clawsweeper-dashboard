## Summary

Keep open: current main still exposes the OpenAI Codex browser authorization URL after the fixed localhost callback cannot bind, so the occupied-port state-error path remains source-reproducible. The exact-scope fix PR was closed unmerged, and the remaining open related issue covers a different remote prompt lifecycle feature.

## What This Changes

Improve OpenAI Codex OAuth handling when localhost:1455 is occupied so the CLI avoids a misleading browser callback failure, gives clear diagnostics, and prompts for manual entry immediately.

## Best Possible Solution

Add a provider-owned busy-port guard that suppresses automatic opening when localhost:1455 is occupied, preserves/logs the manual authorization URL, gives the lsof diagnostic, prompts immediately, and mirrors the behavior in the bundled OpenAI runtime copy with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: occupy 127.0.0.1:1455 and run `openclaw models auth login --provider openai-codex`; the dependency still calls onAuth after bind failure and OpenClaw then opens/logs the browser URL before delayed manual fallback.

## Solution Assessment

Yes. The narrow maintainable fix is an OpenAI provider local-port guard or equivalent onAuth guard that avoids `openUrl` for the occupied callback while still showing the authorization URL for manual use; a configurable callback port is not needed.
