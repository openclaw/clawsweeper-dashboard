## Summary

Keep open: current main and the latest release still expose the OpenAI Codex authorization URL through the local browser opener after the helper fails to bind localhost:1455, so the confusing occupied-port path remains source-reproducible. The exact-scope fixing PR was closed unmerged, and the open remote callback-settled issue tracks a different prompt lifecycle problem.

## What This Changes

The issue asks for clearer OpenAI Codex OAuth behavior when localhost:1455 is occupied so the CLI avoids opening a misleading browser redirect and prompts for manual OAuth immediately.

## Best Possible Solution

Add a local-only occupied-port preflight or equivalent guard in the OpenAI Codex OAuth wrapper and bundled runtime copy that gives clear diagnostics, preserves the manual authorization URL, and prompts for redirect/code immediately without changing the fixed redirect URI, token exchange, or remote flow.

## Reproduction Assessment

Yes, by source inspection. Occupying 127.0.0.1:1455 makes the pinned helper return a null callback server but still call onAuth with the authorization URL, and current OpenClaw then opens/logs that URL before the delayed manual fallback path.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned local port preflight or onAuth guarding that avoids opening the wrong localhost callback while still exposing the manual authorization URL; no configurable callback port or new product policy is needed.
