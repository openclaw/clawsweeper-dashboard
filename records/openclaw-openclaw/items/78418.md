## Summary

Keep open: the destructive runtime auto-clobber path has been moved out of hot reload, but current main and the latest release still start the config watcher and read/validate changed config before `gateway.reload.mode="off"` suppresses reload actions. The remaining request is a real operator-control/config contract decision, not an implemented cleanup item.

## What This Changes

The issue requests a supported way to disable the Gateway runtime config-file watcher while leaving startup config validation unchanged.

## Best Possible Solution

Choose one documented runtime watcher opt-out contract, wire it through schema/help/docs and focused reload tests, and keep Gateway startup validation fail-closed regardless of watcher behavior.

## Reproduction Assessment

Yes, at source level. Current main still starts the config watcher unconditionally, reads and validates changed snapshots, and only suppresses reload actions later when `gateway.reload.mode="off"`; I did not run a live filesystem repro in this read-only review.

## Solution Assessment

Unclear. A watcher opt-out is plausible, but maintainers should choose between changing `off` semantics and adding one explicit opt-out, with a bias against adding both config and env surfaces without approval.
