## Summary

Keep open: current main and the latest release no longer auto-clobber invalid watched config edits, but they still start the config watcher and read/validate changes before `gateway.reload.mode="off"` suppresses reload actions, so the remaining watcher opt-out is real product/config-surface work.

## What This Changes

The issue requests a supported operator switch to disable the runtime config-file watcher and validation path while keeping gateway startup validation unchanged.

## Best Possible Solution

Choose one documented runtime watcher opt-out contract, update schema/help/docs and reload tests around that contract, and keep startup validation fail-closed regardless of watcher behavior.

## Reproduction Assessment

Yes, at source level: current main still starts the watcher unconditionally and reaches `gateway.reload.mode="off"` only after reading/validating and planning against a valid changed snapshot. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Unclear. A watcher opt-out is plausible, but the best solution needs maintainer choice between changing `off` semantics and adding one new explicit opt-out, with a strong bias against adding both config and env surfaces without approval.
