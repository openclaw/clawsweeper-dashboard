## Summary

Keep open: the destructive watcher clobber path has been moved out of hot reload, but current main and the latest release still start the config watcher and read/validate config changes before `gateway.reload.mode="off"` suppresses reload actions. The remaining request is a real config/product decision, not a stale implemented bug.

## What This Changes

The issue requests a supported operator opt-out that disables the runtime config-file watcher entirely while keeping gateway startup validation unchanged.

## Best Possible Solution

Choose one documented runtime watcher opt-out contract, wire it through schema/help/docs and config-reload tests, and keep gateway startup validation fail-closed regardless of watcher behavior.

## Reproduction Assessment

Yes, at source level: current main still starts the config watcher unconditionally, reads and validates changed snapshots, and only suppresses reload actions later when `gateway.reload.mode="off"`. I did not run a live gateway filesystem repro in this read-only review.

## Solution Assessment

Unclear. A watcher opt-out is plausible, but the best product shape needs maintainer choice between changing `off` semantics and adding one explicit opt-out, with a bias against adding both config and env surfaces without approval.
