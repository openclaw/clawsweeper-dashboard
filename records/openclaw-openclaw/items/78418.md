## Summary

Keep open: current main and v2026.6.10 no longer auto-restore invalid reload edits, but they still start the config watcher and read/validate changes before `gateway.reload.mode="off"` suppresses reload actions. The remaining request is a valid operator-control feature and config contract decision, not an implemented cleanup close.

## What This Changes

The issue requests a supported way for operators to disable runtime config-file watching and validation while keeping startup validation unchanged.

## Best Possible Solution

Define one documented runtime watcher opt-out contract, wire it through schema/help/docs and reload tests, and keep Gateway startup validation fail-closed regardless of watcher behavior.

## Reproduction Assessment

Yes, at source level: current main still starts the watcher unconditionally, reads and validates changed snapshots, and only suppresses reload actions later when `gateway.reload.mode="off"`. I did not run a live filesystem gateway repro in this read-only review.

## Solution Assessment

Unclear. A watcher opt-out is plausible, but the best product shape requires maintainer choice between aligning `gateway.reload.mode="off"` with the docs and adding a narrower explicit opt-out surface.
