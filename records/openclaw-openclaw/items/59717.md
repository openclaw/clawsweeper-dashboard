## Summary

Keep open, but the patch is not mergeable as-is: it advertises an ACPX `allow` policy in OpenClaw wrapper config/docs without matching `acpx` runtime support, dangerous-config guard coverage, authoritative setup docs, or real behavior proof.

## What This Changes

This PR adds `allow` as an ACPX `nonInteractivePermissions` value in config validation, manifest schema, parser tests, and ACP agent docs.

## Best Possible Solution

If maintainers want this policy, implement and release it in `openclaw/acpx` first, then bump the OpenClaw pin, add dangerous-flag metadata, update the setup docs, and prove a no-TTY write/exec run; otherwise keep `permissionMode=approve-all` as the supported full-approval path.

## Reproduction Assessment

Yes for the source-level config gap: current main rejects `nonInteractivePermissions=allow`. No for the promised runtime behavior: the pinned and current upstream `acpx` runtime still contracts `deny|fail` and no proof shows no-TTY auto-approval.

## Solution Assessment

No. Validating and documenting `allow` only in the OpenClaw wrapper is not the best fix because permission behavior is owned by `acpx` and the current supported full-approval path is `permissionMode=approve-all`.
