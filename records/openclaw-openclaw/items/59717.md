## Summary

Keep open: this is a security-sensitive feature request, and the PR currently accepts/documents `nonInteractivePermissions=allow` in OpenClaw without matching support in the pinned `acpx` runtime, dangerous-config metadata, current setup docs, or real no-TTY behavior proof.

## What This Changes

The PR adds `allow` to ACPX `nonInteractivePermissions` validation, manifest schema, documentation, and parser tests.

## Best Possible Solution

Decide whether ACPX should support a separate trusted non-interactive allow policy; if yes, implement and release it in `openclaw/acpx`, pin that release here, mark the config as dangerous, update the setup docs, and prove a no-TTY write/exec run.

## Reproduction Assessment

Yes for the source-level config gap: current main rejects `nonInteractivePermissions=allow`, and the PR would accept it. No for the promised runtime behavior: the pinned `acpx` runtime still contracts `deny`/`fail` and its no-TTY path denies or cancels non-`fail` prompts.

## Solution Assessment

No. Accepting `allow` only in the OpenClaw wrapper is not the best fix because the permission behavior is owned by the `acpx` runtime and needs matching release, dangerous-config metadata, docs, and real behavior proof.
