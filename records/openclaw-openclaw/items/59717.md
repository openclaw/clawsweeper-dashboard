## Summary

Keep open: this is a plausible ACPX feature proposal, but the current patch is not mergeable because it only expands OpenClaw validation/docs while the pinned acpx runtime contract still supports `deny`/`fail`, and the new trusted auto-approval mode is missing the matching dangerous-config guard and real behavior proof.

## What This Changes

The PR adds `allow` as an ACPX `nonInteractivePermissions` value in the OpenClaw plugin schema, manifest, docs, and config parser tests.

## Best Possible Solution

Decide whether ACPX should support a separate trusted non-interactive allow policy; if yes, implement and release it in `openclaw/acpx`, pin that release, mark the OpenClaw config as dangerous, update the setup docs, and prove a no-TTY write/exec run.

## Reproduction Assessment

Yes for the source-level config gap: current main rejects `nonInteractivePermissions=allow`, and the PR would accept it. The promised no-TTY auto-approval behavior is not reproduced because acpx v0.7.0 still only contracts `deny`/`fail`.

## Solution Assessment

No. Accepting `allow` only in OpenClaw validation is not the best fix because the behavior is owned by the acpx runtime and needs matching dangerous-config metadata, docs, and runtime proof.
