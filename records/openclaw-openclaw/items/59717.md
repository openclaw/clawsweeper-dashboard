## Summary

Keep open: the branch is not merge-ready because it accepts and documents `allow` without matching acpx runtime support, dangerous-config guard coverage, current setup docs, or after-fix real behavior proof.

## What This Changes

The PR adds `allow` to ACPX `nonInteractivePermissions` config/schema/docs and adds parser tests for accepting it and rejecting unknown values.

## Best Possible Solution

Keep the existing `permissionMode=approve-all` path unless maintainers decide a separate non-interactive `allow` policy is worth implementing end to end in acpx and OpenClaw.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main rejects `nonInteractivePermissions=allow`, while the PR would pass that value to acpx 0.7.0, whose published type/source still only define `deny|fail` and do not auto-approve unknown values.

## Solution Assessment

No. A validation-only enum/docs change is not the best fix; either keep the documented `permissionMode=approve-all` path or make `allow` a complete acpx runtime feature with OpenClaw dangerous-config guards and current docs.
