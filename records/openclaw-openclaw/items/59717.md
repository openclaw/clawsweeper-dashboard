## Summary

Keep open and do not merge as-is: the PR targets a real ACPX permission gap, but it only expands OpenClaw validation/docs while the pinned runtime contract still needs allow support and the new auto-approval mode needs dangerous-config guardrails.

## What This Changes

The PR adds `allow` to ACPX `nonInteractivePermissions` config/schema/docs and adds parser tests for accepting `allow` and rejecting unknown values.

## Best Possible Solution

Implement or consume acpx runtime support for `allow`, then update OpenClaw schema, manifest dangerousFlags, current setup docs, runtime tests, security tests, and changelog together.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: current main rejects `allow`, while the PR would accept and forward it to the pinned acpx runtime; the focused runtime check should run `approve-reads` plus `nonInteractivePermissions=allow` without a TTY and verify write/exec prompts are approved.

## Solution Assessment

No. Expanding OpenClaw validation alone is not the narrowest maintainable fix because the behavior depends on the acpx runtime package and on the existing dangerous-config contract.
