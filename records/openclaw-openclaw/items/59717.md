## Summary

Keep open. Current main still only supports `deny`/`fail`, so the contributor PR is addressing a real gap from #59663, but the patch is not merge-ready: it makes OpenClaw accept and document `allow` without matching support in the pinned/upstream acpx runtime contract, and it omits the dangerous-config guard for a trusted auto-approval policy.

## What This Changes

The PR adds `allow` to ACPX `nonInteractivePermissions` config/schema/docs and adds config-parser tests for accepting `allow` and rejecting unknown values.

## Best Possible Solution

Implement `allow` first in the acpx runtime package or explicitly map it to a supported runtime behavior, then update OpenClaw's ACPX config schema, manifest enum, dangerousFlags contract, security tests, runtime behavior tests, and the current ACP setup docs together.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: current main rejects `allow`, while the PR would accept it and forward it to an acpx runtime whose reviewed contract only supports `deny`/`fail`; a focused runtime test should run `approve-reads` plus `nonInteractivePermissions=allow` without a TTY and verify write/exec prompts are actually approved.

## Solution Assessment

No. The PR only expands OpenClaw validation and docs, but the maintainable fix has to include dependency-backed runtime support and dangerous-config guardrails before users can rely on the new value.
