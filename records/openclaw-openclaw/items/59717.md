## Summary

Keep open: the PR is focused, but it is not mergeable as-is because it exposes a new trusted auto-approval value before the pinned ACPX runtime and OpenClaw dangerous-config contract support that behavior. The linked issue was already closed around the existing `permissionMode=approve-all` path, so this needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

The PR adds `allow` to ACPX `nonInteractivePermissions` config validation, manifest schema, docs, and parser tests.

## Best Possible Solution

If maintainers want this separate mode, implement or consume published acpx runtime support first, then update the OpenClaw schema, manifest dangerous flags, setup docs, runtime/security tests, and release notes together; otherwise keep using the documented `permissionMode=approve-all` path.

## Reproduction Assessment

Yes. A high-confidence source path exists: current main rejects `allow`, while the PR would accept and forward it to pinned `acpx` 0.7.0, whose published contract still lists only `deny` and `fail`; a focused no-TTY ACPX runtime run is still needed for live after-fix proof.

## Solution Assessment

No. Expanding OpenClaw validation and docs alone is not the best fix because the behavior depends on upstream acpx runtime support and the existing dangerous-config guardrails.
