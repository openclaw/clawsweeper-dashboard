## Summary

Keep open. Current main still normalizes `mode` and `apiKey` aliases but not snake_case `api_key`, while both eligibility and runtime resolution require canonical `key` or `keyRef`. The related fix PR #57418 was closed unmerged, so there is no implemented or shipped fix to close against.

## What This Changes

The issue asks OpenClaw to stop silently ignoring `api_key` in `auth-profiles.json` by documenting, accepting, or validating that field for API-key credentials.

## Best Possible Solution

Add a narrow persisted auth-profile normalization change that maps `api_key` to canonical `key` only when `key` is absent, preserve canonical-field precedence, add a regression case beside the existing `mode/apiKey` alias tests, and keep docs describing `key` as the canonical field.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: load an API-key auth profile containing only `api_key`; current normalization leaves `key` undefined, and both eligibility and runtime resolution then treat the profile as missing credentials.

## Solution Assessment

Yes. The narrowest maintainable fix is a load-time alias in `normalizeRawCredentialEntry` plus regression coverage; broader unknown-field validation can be considered separately but is not required to fix the reported failure.
