## Summary

Keep open. Current main documents the canonical `key` field and still normalizes camelCase `apiKey`, but a raw auth profile that only uses snake_case `api_key` is still loaded without mapping it to `key`, then filtered as missing credentials; the exact fix PR was closed unmerged and no open replacement appears to target this issue.

## What This Changes

The issue asks OpenClaw to document, accept, or validate `api_key` in `auth-profiles.json` so API-key profiles do not fail silently when users use the snake_case field instead of canonical `key`.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` during auth-profile store load when `key` is absent, preserve canonical-field precedence, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a profile with only `api_key` is loaded without canonical `key`, marked `missing_credential`, filtered from auth order, and never resolved by `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending `normalizeRawCredentialEntry` to map `api_key` to `key` when canonical `key` is absent is the narrowest maintainable fix because it uses the existing alias-normalization point and preserves canonical precedence.
