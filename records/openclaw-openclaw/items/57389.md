## Summary

Keep open. Current `main` still normalizes only `apiKey` to `key`, while auth-profile eligibility and runtime resolution require canonical `key` or `keyRef`; the related exact-fix PR #57418 is closed unmerged.

## What This Changes

The issue requests that `auth-profiles.json` entries using `api_key` either be accepted as an alias for `key`, documented clearly, or reported with validation instead of failing silently.

## Best Possible Solution

Normalize `api_key` to canonical `key` during auth-profile store load when `key` is absent, preserve canonical-field precedence, and add a focused regression test plus changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a loaded profile with only `api_key` is not normalized to `key`, is marked `missing_credential`, and never reaches a usable `resolveApiKeyForProvider` result.

## Solution Assessment

Yes. The narrowest maintainable fix is to extend the existing `normalizeRawCredentialEntry` alias normalization rather than teaching every resolver or eligibility check about another raw persisted field.
