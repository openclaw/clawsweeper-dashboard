## Summary

Keep open. Current main still normalizes `apiKey` but not snake_case `api_key`, while auth-profile eligibility and API-key resolution require canonical `key` or `keyRef`; the exact related PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to stop silently ignoring `api_key` fields in `auth-profiles.json`, preferably by documenting, accepting, or validating that shape.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` during auth-profile store load when `key` is absent, preserve canonical-field precedence, and cover the load plus resolver path with focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: an `auth-profiles.json` profile with only `api_key` loads without mapping to `key`, is classified as `missing_credential`, and is filtered before `resolveApiKeyForProvider` can return it.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing `normalizeRawCredentialEntry` alias-normalization point, rather than teaching each eligibility and resolver path about another raw persisted field.
