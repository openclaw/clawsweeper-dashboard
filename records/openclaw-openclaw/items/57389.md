## Summary

Keep open. Current main and the latest release still normalize `mode` and `apiKey` but not snake_case `api_key`, while the auth-profile eligibility and resolver paths only consume canonical `key`/`keyRef`; the exact related fix at https://github.com/openclaw/openclaw/pull/57418 was closed unmerged.

## What This Changes

Accept, document, or validate `api_key` in `auth-profiles.json` API-key credentials so profiles are not silently skipped.

## Best Possible Solution

Extend `normalizeRawCredentialEntry` to map raw `api_key` to canonical `key` only when `key` is absent, preserve canonical precedence, and add parser plus model-auth regression coverage.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` normalizes without `key`, fails API-key eligibility as `missing_credential`, and cannot resolve in `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing alias normalizer is the narrowest maintainable fix because current main already uses that compatibility point for adjacent `mode` and `apiKey` inputs while preserving canonical `key` precedence.
