## Summary

Keep open. Current main still normalizes `apiKey` but not snake_case `api_key`, while eligibility and runtime resolution require canonical `key` or `keyRef`; the related exact-fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to accept, document, or validate `api_key` fields in `auth-profiles.json` instead of silently skipping those API-key profiles.

## Best Possible Solution

Normalize `api_key` to canonical `key` during auth-profile store load when `key` is absent, preserve canonical-field precedence, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a profile loaded with only `api_key` is not normalized to `key`, is marked `missing_credential`, and is filtered out before `resolveApiKeyForProvider` can return it.

## Solution Assessment

Yes. The narrowest maintainable fix is to extend the existing `normalizeRawCredentialEntry` alias-normalization point rather than teaching every eligibility and resolver path about another raw persisted field.
