## Summary

Keep open. Current main still normalizes `apiKey` but not snake_case `api_key`, while the selection path requires canonical `key` or `keyRef`; the exact fix PR is closed unmerged and no open replacement PR appears to reference this issue.

## What This Changes

The issue requests documented, validated, or compatible handling for `api_key` in `auth-profiles.json` API-key profiles.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` during auth-profile store load when `key` is absent, preserve canonical-field precedence, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an `auth-profiles.json` profile with only `api_key` is loaded without mapping it to `key`, marked `missing_credential`, and filtered before `resolveApiKeyForProvider` can return it.

## Solution Assessment

Yes. The narrow maintainable fix is to extend `normalizeRawCredentialEntry`, the existing alias-normalization point, instead of teaching eligibility and resolver paths about another raw persisted field.
