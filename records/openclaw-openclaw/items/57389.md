## Summary

Keep open. Current main still normalizes `apiKey` to `key` but not snake_case `api_key`, while the eligibility and resolver paths only consume canonical `key`/`keyRef`; the related fix PR added the exact alias but was closed unmerged.

## What This Changes

The issue asks OpenClaw to avoid silently ignoring `api_key` in `auth-profiles.json` API-key profiles, either by documenting, accepting, or validating that field.

## Best Possible Solution

Extend `normalizeRawCredentialEntry` to map raw `api_key` to canonical `key` only when `key` is absent, preserve canonical precedence, and add focused parser plus model-auth regression coverage.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` is parsed without `key`, fails API-key eligibility as `missing_credential`, and cannot resolve in `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing alias normalizer is the narrowest maintainable fix because current main already uses that compatibility point for adjacent `mode` and `apiKey` inputs while preserving canonical `key` precedence.
