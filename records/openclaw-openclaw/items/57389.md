## Summary

Current main still has the same gap: it normalizes `apiKey` to `key`, but not `api_key`, and the resolver/eligibility path only reads canonical `key`/`keyRef`. The closed fix PR added the right alias but was not merged, so this remains a narrow auth-profile compatibility bug rather than cleanup-close material.

## What This Changes

The issue requests that `auth-profiles.json` stop silently ignoring `api_key` API-key credentials by documenting, accepting, or validating the field.

## Best Possible Solution

Extend the existing auth-profile alias normalizer to map raw `api_key` to canonical `key` only when `key` is absent, preserve canonical precedence, and add focused regression coverage for parsing and model-auth resolution.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` is normalized without `key`, fails API-key eligibility as `missing_credential`, and cannot resolve in `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending `normalizeRawCredentialEntry` is the narrowest maintainable fix because current main already uses that function for adjacent `mode` and `apiKey` compatibility, and canonical `key` precedence can remain unchanged.
