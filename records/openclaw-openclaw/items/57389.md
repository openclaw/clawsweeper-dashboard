## Summary

Keep open: current main still normalizes `mode` and camelCase `apiKey`, but not snake_case `api_key`, while eligibility and resolution only consume canonical `key`/`keyRef`; the exact related implementation was closed unmerged.

## What This Changes

The issue asks OpenClaw to stop silently ignoring `api_key` in `auth-profiles.json` by documenting, accepting, or validating that spelling.

## Best Possible Solution

Extend the existing auth-profile alias normalizer to map raw `api_key` to canonical `key` only when `key` is absent, then add parser and model-auth regression coverage.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` normalizes without `key`, fails eligibility as `missing_credential`, and cannot resolve through `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing alias normalizer is the narrowest maintainable fix because current code already uses that compatibility seam for adjacent `mode` and `apiKey` inputs while preserving canonical `key` precedence.
