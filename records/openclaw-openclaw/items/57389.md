## Summary

Keep open. Current main still normalizes camelCase `apiKey` but not snake_case `api_key`, so an otherwise valid `api_key`-only auth profile is loaded, then treated as missing credentials; the exact fix PR was closed unmerged and no open replacement appears to cover this alias.

## What This Changes

The issue requests that `auth-profiles.json` stop silently ignoring API-key credentials written with `api_key` instead of canonical `key`.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` in `normalizeRawCredentialEntry` when `key` is absent, preserve canonical-field precedence, and add focused regression coverage plus the usual user-facing changelog entry.

## Reproduction Assessment

Yes. A profile with `type: "api_key"`, `provider`, and only `api_key` loads without canonical `key`, fails `resolveAuthProfileEligibility` as `missing_credential`, and is filtered out before `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing auth-profile alias normalizer is the narrowest maintainable fix because it matches the already-merged `apiKey -> key` compatibility path and avoids changing resolver semantics.
