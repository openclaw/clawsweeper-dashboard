## Summary

Keep open: current `main` and v2026.5.20 still normalize `mode` and camelCase `apiKey`, but not snake_case `api_key`, while eligibility and runtime resolution only consume canonical `key`/`keyRef`; the exact related fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to accept or clearly diagnose `api_key` in `auth-profiles.json` API-key credentials instead of silently requiring canonical `key`.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` only when `key` is absent, preserve canonical precedence, and add parser plus model-auth regression coverage.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` normalizes without `key`, fails eligibility as `missing_credential`, and cannot resolve through `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing alias normalizer is the narrowest maintainable fix because current code already uses that compatibility seam for adjacent `mode` and `apiKey` inputs while preserving canonical `key` precedence.
