- `impact:auth-provider`: The issue affects model provider API-key resolution through `auth-profiles.json`.

## Summary

Keep open: current `main` and v2026.5.22 still normalize `mode` and camelCase `apiKey`, but not snake_case `api_key`, while auth-profile eligibility and runtime resolution only consume canonical `key`/`keyRef`; the linked fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to document, validate, or accept snake_case `api_key` in `auth-profiles.json` instead of silently skipping API-key profiles that use it.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` only when `key` is absent, preserve canonical precedence, and add focused parser plus resolver regression coverage while keeping `key` as the documented canonical field.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` normalizes without `key`, fails eligibility as `missing_credential`, and cannot resolve through `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing alias normalizer is the narrowest maintainable fix because current code already uses that compatibility seam for adjacent `mode` and `apiKey` inputs while preserving canonical `key` precedence.
