## Summary

Keep open. Current main and the latest release still normalize `mode` and camelCase `apiKey` but not snake_case `api_key`, while auth-profile eligibility and runtime resolution only consume canonical `key`/`keyRef`; the exact related fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to stop silently ignoring `api_key` in `auth-profiles.json` by documenting, validating, or normalizing it to the canonical `key` field.

## Best Possible Solution

Normalize raw `api_key` to canonical `key` only when `key` is absent, preserve canonical precedence, and add focused parser plus model-auth regression coverage.

## Reproduction Assessment

Yes, by source inspection. A profile with `type: "api_key"`, `provider`, and only `api_key` normalizes without `key`, fails eligibility as `missing_credential`, and cannot resolve through `resolveApiKeyForProfile`.

## Solution Assessment

Yes. Extending the existing alias normalizer is the narrowest maintainable fix because current main already uses that compatibility seam for adjacent `mode` and `apiKey` inputs while preserving canonical `key` precedence.
