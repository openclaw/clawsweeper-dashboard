## Summary

Keep open: the PR targets a useful cron enhancement, but it is not ready because it timestamps repeated alerts from emit time instead of event time, leaves the single-failure failure-destination alert unchanged, lacks a changelog entry, and does not show after-fix real behavior proof.

## What This Changes

The PR adds ISO timestamp text to repeated cron failure alerts and schedule-error auto-disable notifications in the cron service.

## Best Possible Solution

Land a narrow cron change that timestamps all cron failure notification bodies from recorded run/event times, adds focused regression coverage, and records the user-facing change in the changelog.

## Reproduction Assessment

Not applicable as a feature request. Source inspection of current main confirms the repeated failure alert, single-failure failure-destination alert, and auto-disable notification templates omit event timestamps today.

## Solution Assessment

No. The direction is right, but the maintainable fix should pass explicit event times such as result.endedAt and evt.runAtMs into both alert builders instead of formatting state.deps.nowMs() at send time.
