## Summary

Keep open: the prior ClawSweeper blockers still match the latest PR head, with source-only proof, send-time timestamps instead of explicit event times, the single-failure path still uncovered, and no changelog entry for the user-facing alert text change.

## What This Changes

The PR adds ISO timestamps from `state.deps.nowMs()` to repeated cron failure/skipped alerts and schedule-error auto-disable notifications, plus one updated failure-alert test expectation.

## Best Possible Solution

Land a narrow cron change that threads explicit run/event times into every failure notification body, covers repeated, single-failure, and auto-disable paths with tests, and records the user-facing text change in `CHANGELOG.md`.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current cron failure notification templates omit event timestamps, and the linked request identifies the affected paths.

## Solution Assessment

No. The direction is useful, but the safer fix should pass `result.endedAt` or `evt.runAtMs` into the alert builders instead of formatting `state.deps.nowMs()` at delivery/application time, and it should cover the single-failure path and changelog.
