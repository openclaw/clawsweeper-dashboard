## Summary

Keep open: the feature direction is useful, but this PR is not merge-ready because it timestamps one alert path with send/apply-time, misses the single-failure failure-destination alert from the linked report, lacks the required changelog entry, and still needs real after-fix behavior proof.

## What This Changes

The PR adds ISO timestamp text to repeated cron failure alerts and schedule-error auto-disable notifications.

## Best Possible Solution

Land a narrow cron change that timestamps all cron failure notification bodies from explicit run/event times, adds focused regression coverage, and records the user-facing alert-text change in the changelog.

## Reproduction Assessment

Not applicable as a feature request. Source inspection of current main confirms the repeated failure alert, single-failure failure-destination alert, and schedule auto-disable notification templates omit event timestamps today.

## Solution Assessment

No. The direction is useful, but the safer fix should thread explicit event times such as result.endedAt and evt.runAtMs into all alert builders instead of formatting state.deps.nowMs() at send/application time.
