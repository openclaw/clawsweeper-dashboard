## Summary

Keep open: the PR addresses part of the requested cron alert text, but it is not ready to merge because it stamps repeated alerts with alert-emission time, leaves the first-failure notification body untimestamped, and does not show after-fix patched alert output.

## What This Changes

The PR adds ISO timestamp text to repeated cron failure/skipped alerts and schedule-error auto-disable notifications, and updates failure-alert test expectations.

## Best Possible Solution

Thread an explicit cron event time through repeated alerts, first-failure failure-destination/fallback messages, and auto-disable notifications, then cover those paths with focused cron and gateway tests plus patched runtime proof.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main's repeated-alert, first-failure, and auto-disable templates omit event-time text in the user-facing message body.

## Solution Assessment

No. The maintainable fix is to use explicit event timestamps such as `result.startedAt`/`result.endedAt` or `evt.runAtMs` across all affected alert paths, rather than formatting `nowMs()` in only part of the surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the cron event time, not alert emission time:** `src/cron/service/timer.ts:444-445`
  - body: The new timestamp is formatted from `state.deps.nowMs()` inside `emitFailureAlert`. `onTimer` runs due jobs through workers and only applies completed results after `Promise.all`, so a fast failed job can be stamped after later jobs finish instead of with its own run event time. Thread the selected event time, such as `result.startedAt` or `result.endedAt`, into this helper.
  - confidence: 0.91
- **[P2] Timestamp the first-failure notification body:** `src/cron/service/timer.ts:447`
  - body: This PR updates the repeated `failureAlert` text, but the linked feature also covers the first-failure `failureDestination` and primary fallback messages. `dispatchCronFailureDestinationNotifications` still sends `Cron job "..." failed: ...` text without using `evt.runAtMs`, so delayed first-failure alerts would remain misleading.
  - confidence: 0.89
