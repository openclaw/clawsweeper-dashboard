## Summary

Keep open: current main still omits event-time text from cron failure notifications, and this PR targets the right surface, but the proposed run-failure timestamp is taken from alert application time instead of the completed run time.

## What This Changes

The PR adds an ISO timestamp to cron repeated-failure alert text and schedule-error auto-disable notifications.

## Best Possible Solution

Land a narrow cron change that passes the actual run completion time into failure-alert formatting, timestamps schedule auto-disable notifications at the compute error, adds focused regression tests, and records the user-facing change in the changelog.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: the repeated-failure and auto-disable alert builders omit event-time text today.

## Solution Assessment

No. The direction is right, but the run-failure alert should use result.endedAt or another explicit event time instead of state.deps.nowMs() inside the alert builder.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the completed run time for alert timestamps:** `src/cron/service/timer.ts:444-447`
  - body: The timestamp is formatted from `state.deps.nowMs()` when the alert is emitted. Due jobs run concurrently and results are applied only after all workers finish, so a fast failed job can receive a timestamp delayed by another long-running job, which is the ambiguity this feature is meant to fix. Pass `result.endedAt` into the alert path and format that instead.
  - confidence: 0.9
