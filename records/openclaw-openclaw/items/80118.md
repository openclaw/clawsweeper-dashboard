## Summary

Keep open: current main still appears to ignore `tools.byProvider` when `collectSmallModelRiskFindings()` computes small-model web/browser exposure, but the requested downgrade/exclusion needs maintainer/security policy clarification because current tests intentionally keep small models critical unless sandboxing is also enabled.

## What This Changes

The issue asks for `openclaw security audit --deep` to resolve per-provider/model `tools.byProvider` denies when computing `models.small_params` exposure and severity.

## Best Possible Solution

Have the audit compute each small model's web/browser exposure through the same provider/model-scoped effective tool-policy path as runtime, then align tests and docs on whether per-model web denial alone downgrades the finding or sandboxing remains required.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: the collector's policy list omits `tools.byProvider`, while runtime policy resolution handles provider/model keys separately; I did not run the packaged CLI because this review had to keep the checkout read-only.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the full requested downgrade: reusing runtime policy resolution is the narrow fix for the incorrect `web=[...]` exposure, but changing CRITICAL eligibility conflicts with current tests and needs maintainer/security policy confirmation.
