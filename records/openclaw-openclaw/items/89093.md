## Summary

Keep open for maintainer review: the PR addresses the source-backed supervised auto-update failure with a shared managed-service handoff and sufficient Linux systemd package proof, but merging changes startup update/restart availability behavior and needs maintainer acceptance of the remaining platform proof gap.

## What This Changes

The branch moves the managed-service update handoff into `src/infra`, re-exports it for gateway `update.run`, and changes startup auto-update to use that handoff with channel/timeout forwarding and focused tests.

## Best Possible Solution

Land the shared managed-service handoff after maintainers accept the current Linux package proof plus focused cross-supervisor tests, or request one more packaged launchd/Scheduled Task proof before merge if that availability risk needs direct coverage.

## Reproduction Assessment

Yes. Current main and v2026.6.1 show startup auto-update spawning the update child with inherited env, while the CLI package-update guard rejects gateway-service marker env.

## Solution Assessment

Yes. Reusing the managed-service handoff is a better fix than only stripping markers because it avoids replacing the package tree from a live gateway child and aligns startup auto-update with `update.run`; the remaining question is platform proof depth.
