## Summary

Keep this PR open for maintainer review: the branch addresses a real shipped supervised auto-update failure and I did not find a blocking line-level defect, but the live service restart/handoff timing remains an availability risk for maintainers to accept.

## What This Changes

The branch routes supervised startup auto-updates through the managed-service handoff path, shares that handoff from src/infra, forwards update channel and timeout arguments, and updates focused startup and handoff tests.

## Best Possible Solution

Land the shared managed-service handoff after maintainers accept the remaining platform proof gap, or request one more packaged launchd/Scheduled Task proof before merge if that availability risk needs direct coverage.

## Reproduction Assessment

Yes. Current main and v2026.6.1 show startup auto-update spawning the update child with inherited env, while the CLI package-update guard rejects gateway-service marker env.

## Solution Assessment

Yes. Reusing the managed-service handoff is a better fix than only stripping markers from the direct child because it avoids replacing the package tree while the live gateway can still lazy-load old dist files and aligns startup auto-update with update.run.
