## Summary

Keep open: the reported WhatsApp idle-flap bug is credible and the supplied real logs are sufficient, but the current patch is not merge-ready because review identified a real timeout-policy regression plus lifecycle and coverage gaps.

## What This Changes

The PR adds a five-minute WhatsApp `unavailable` presence keepalive and changes the web message-idle watchdog default from 30 minutes to 4 hours.

## Best Possible Solution

Implement the presence keepalive as a controller-owned lifecycle policy with separate recent-inbound timeout, quiet-session cap, and keepalive cadence, backed by focused fake-timer tests.

## Reproduction Assessment

Yes. A low-traffic WhatsApp Web account left idle while watching gateway logs for status 428/499 is a high-confidence path; this pass did not run a live account, but current source and the supplied before/after logs support it.

## Solution Assessment

No. The presence keepalive direction is good, but the current implementation should split the timeout policies and move timer ownership into `WhatsAppConnectionController` before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Split the watchdog timeout policies:** `extensions/whatsapp/src/auto-reply/monitor.ts:261`
  - body: Changing the fallback `messageTimeoutMs` to 4h also feeds `WhatsAppConnectionController`'s `appSilenceTimeoutMs = messageTimeoutMs * 4`, so quiet sessions with transport frames now wait 16h before the app-silence safety net fires. It also makes the recent-inbound recovery path use 4h instead of the shorter message watchdog. Please pass separate defaults for recent-inbound silence, quiet-session app-silence, and presence cadence.
  - confidence: 0.95
- **[P2] Move the keepalive timer into the controller lifecycle:** `extensions/whatsapp/src/session.ts:221-225`
  - body: The interval is created in `createWaSocket()` and only cleared from a Baileys `connection.update: close` event, while `closeCurrentConnection()` and `shutdown()` can close sockets directly and only clear controller-owned timers. That can leave the presence interval firing against a closed or replaced socket if the close event is delayed or absent, so the timer should be owned and cleared by the controller lifecycle.
  - confidence: 0.86
