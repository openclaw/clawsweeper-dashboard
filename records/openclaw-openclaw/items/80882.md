## Summary

Keep open: this is a fresh draft bug-fix PR, and current main still has the 30-minute WhatsApp message-idle default without a periodic socket-level presence keepalive. I found no blocking correctness issue in the diff; the main remaining gap is normal PR readiness and preferably focused regression coverage for the timer behavior.

## What This Changes

Adds a 5-minute WhatsApp `unavailable` presence interval while the Baileys socket is open and raises the default auto-reply message watchdog from 30 minutes to 4 hours.

## Best Possible Solution

Land the focused socket/watchdog fix after the draft is ready, ideally with a regression test that proves the `unavailable` keepalive starts on open, stops on close, and leaves explicit `tuning.messageTimeoutMs` overrides intact.

## Reproduction Assessment

Yes, with a real low-traffic WhatsApp account left idle past 30 minutes; I did not rerun that live path, but current main source plus the supplied before/after logs give a coherent reproduction path for the 499 watchdog flap and Baileys 428 close.

## Solution Assessment

Yes, the patch is a narrow owner-local fix: it uses Baileys' existing presence API on the existing authenticated socket, preserves `markOnlineOnConnect: false`, and keeps the existing tuning override. A focused regression test would make it safer to merge but does not reveal a blocking diff bug.
