## Summary

Keep open. The presence-keepalive direction is plausible and backed by real logs, but the watchdog default change interacts with the current transport-aware controller and would stretch the normal app-silence recovery window far beyond the PR's stated 4-hour safety net.

## What This Changes

The PR adds a periodic WhatsApp Web `unavailable` presence keepalive and changes the default auto-reply message watchdog from 30 minutes to 4 hours.

## Best Possible Solution

Keep the WhatsApp-owned presence keepalive if maintainers accept that protocol behavior, but keep the watchdog safety window aligned with the current transport/app-silence model and add focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw side at source level: current main still has a 30-minute `messageTimeoutMs` default and no periodic presence keepalive. The exact WhatsApp-server 428 flap is supported by contributor live logs and Baileys source mapping, but I did not reproduce it with a real account here.

## Solution Assessment

No, not as-is. The presence keepalive is a narrow candidate fix, but the 4-hour default must be reconciled with the controller's existing 4x app-silence cap or it weakens stuck-session recovery to about 16 hours on the normal quiet-session path.

## Review Findings

Overall correctness: patch is incorrect
