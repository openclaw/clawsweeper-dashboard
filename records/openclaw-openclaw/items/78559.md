## Summary

Keep open. The PR targets a real Discord final-send bug, but the current diff still lets broadly marked tool payloads skip Discord-specific trace/channel scrubbing, and the external PR still lacks inspectable real Discord proof.

## What This Changes

The PR marks Discord tool-kind deliveries with `channelData.openclawDiscordToolProgress`, preserves marked text in the final Discord sanitizer, adjusts marker-only `channelData` handling, and adds focused Discord monitor/delivery tests.

## Best Possible Solution

Preserve only known-safe Discord tool-progress labels or keep Discord trace/channel scrubbing active for marked payloads, add an unsafe marked-payload regression, and attach redacted real Discord proof.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main routes final Discord replies through `sanitizeDiscordFrontChannelText`, whose regex strips `Read` and `Exec` labels at final delivery; I did not run a live Discord bot in this read-only review.

## Solution Assessment

No. The final-send boundary is the right place to fix this, but the current implementation is too broad because every tool delivery can skip Discord-specific trace and channel-line scrubbing.
