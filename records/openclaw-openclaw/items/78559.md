## Summary

Keep open: this PR targets a real Discord final-send sanitizer bug, but the current diff still has a blocking sanitizer/security regression and lacks inspectable real Discord proof.

## What This Changes

The PR adds a Discord `channelData` marker for tool-kind replies, preserves marked tool-progress text in the final reply sanitizer, adjusts marker-only `channelData` handling, and adds focused Discord monitor tests.

## Best Possible Solution

Preserve only known-safe Discord tool-progress labels or keep Discord trace/channel scrubbing active for marked payloads, add unsafe-marked-payload coverage, and require redacted real Discord proof before merge.

## Reproduction Assessment

Yes, as a source-level reproduction: current main routes final Discord replies through `sanitizeDiscordFrontChannelText`, whose regex strips `📖 Read:` and `🛠️ Exec:` progress-shaped lines. I did not run a live Discord bot in this read-only review.

## Solution Assessment

No. The final-send boundary is the right place to fix this, but this implementation is too broad because every marked tool delivery skips Discord-specific trace and channel-line scrubbing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
