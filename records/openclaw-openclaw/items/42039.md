## Summary

Keep open: current main and the latest release still leave the requested cron catch-up controls as internal service deps rather than accepted public config, and the linked duplicates make this the canonical tracker. The remaining decision is product/API shape, because exposing restart missed-run behavior changes scheduler policy and public config surface.

## What This Changes

The issue requests public cron config for restart missed-job replay limits and staggering so operators can reduce duplicate or bursty catch-up deliveries.

## Best Possible Solution

Define a compatibility-safe public missed-run policy, then wire schema, config help/docs, gateway construction, runtime behavior, and focused restart-catch-up tests in one change.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the config gap: the requested keys exist only as internal cron service deps, are absent from the public config type and strict schema, and are not wired from gateway config into `CronService`.

## Solution Assessment

Unclear. Exposing the internal knobs is plausible, but the best fix should first decide the public missed-run policy semantics, especially whether disabling catch-up means defer, skip to next natural schedule, or per-job policy.
