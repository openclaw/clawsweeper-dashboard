## Summary

Keep open: current main still has the public configuration gap, and the related duplicate issues were closed in favor of this issue as the canonical tracker. The straightforward “expose the internal fields” fix is not clearly sufficient because `maxMissedJobsPerRestart: 0` currently defers all missed jobs instead of skipping catch-up, so maintainers should choose the public missed-run policy before implementation.

## What This Changes

The issue requests public cron configuration for missed-job restart catch-up limits and stagger timing so operators can reduce or disable restart catch-up bursts.

## Best Possible Solution

Define a backward-compatible cron missed-run policy that distinguishes run, defer/stagger, and skip-to-next-natural-slot behavior, then update the config type, strict schema, help/docs, gateway wiring, and restart-catch-up tests together.

## Reproduction Assessment

Yes for the reported config gap: current source shows the public `CronConfig` type and strict zod schema do not accept the requested keys, while the internal deps still exist. The larger real-world duplicate-message scenario was not live-run in this read-only review, but the startup catch-up code path and tests make the behavior source-reproducible.

## Solution Assessment

No. Directly exposing the two internal dependency fields is not the best complete solution because current `maxMissedJobsPerRestart: 0` semantics defer all missed jobs instead of skipping catch-up; the safer direction is an explicit public missed-run policy with tests for skip versus defer.
