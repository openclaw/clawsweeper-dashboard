## Summary

Keep open: current main and v2026.5.22 still keep the restart catch-up knobs internal, the public cron config rejects the requested keys, and the reporter confirmed this remains relevant on May 20, 2026. The user problem is real, but the fix needs a cron policy/API decision rather than a literal expose-only patch.

## What This Changes

Expose cron restart missed-job controls or an equivalent missed-run policy through public cron configuration so operators can avoid duplicate catch-up deliveries after gateway restarts.

## Best Possible Solution

Define a backward-compatible missed-run policy, then update cron config types, strict schema/help, docs, gateway wiring, optional per-job behavior, and restart catch-up regression tests together.

## Reproduction Assessment

Yes, source-level. Current main rejects the requested public cron keys, does not wire them into CronService, and still replays or defers startup missed jobs without a public skip policy.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because the current internal max=0 behavior defers eligible jobs instead of disabling catch-up.
