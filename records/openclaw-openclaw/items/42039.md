## Summary

Keep this open: current main still exposes the requested missed-job controls only as internal `CronServiceDeps`, not as public cron config, and the central disable-catch-up behavior still needs an API decision because the current internal `maxMissedJobsPerRestart: 0` path would defer missed jobs rather than skip them.

## What This Changes

The issue requests public cron configuration for missed-job catch-up limits and staggering after gateway restart.

## Best Possible Solution

Add a backward-compatible cron missed-run policy through the public config surface, with `CronConfig`, strict zod validation, generated help/labels, docs, gateway wiring, and focused restart-catch-up tests updated together; define skip/freshness semantics explicitly instead of relying on the current internal max-immediate deferral behavior.

## Reproduction Assessment

Yes, source-level. Current main shows the config keys are absent from `CronConfig` and the strict zod schema, while startup still invokes missed-job replay and tests assert immediate replay or staggered deferral.

## Solution Assessment

No, not as written. Exposing the current internal knobs is incomplete because `maxMissedJobsPerRestart: 0` would defer missed jobs into near-future catch-up slots, so maintainers should choose an explicit skip or freshness policy before implementation.
