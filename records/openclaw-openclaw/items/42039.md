## Summary

Keep open. Current main and v2026.6.6 still keep the restart catch-up limit and stagger as internal cron service dependencies only, while duplicate reports and the closed attempted PR leave this issue as the canonical public cron missed-run policy decision.

## What This Changes

Expose public cron missed-run catch-up controls so users can cap, stagger, or skip overdue jobs after a gateway restart.

## Best Possible Solution

Define a cron-owned missed-run policy that directly expresses replay, skip, bounded-freshness, and stagger behavior, then align CronConfig, strict schema, gateway wiring, protocol/job policy if chosen, docs, and restart catch-up tests.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence path: persist an enabled recurring cron job with a past nextRunAtMs and start CronService; current startup catch-up replays or defers it without a public skip/freshness policy.

## Solution Assessment

No, not as stated yet. Raw exposure of the internal deps is only a plausible implementation detail; the maintainable fix should choose an explicit public missed-run policy because zero immediate jobs currently means deferred replay, not skip.
