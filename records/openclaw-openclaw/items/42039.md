## Summary

Keep open. Current main still keeps the missed-job catch-up knobs internal, and the requested public config behavior needs a maintainer policy decision because the existing internal `maxMissedJobsPerRestart: 0` semantics would defer all missed jobs rather than skip catch-up entirely.

## What This Changes

Expose cron missed-job catch-up controls such as `maxMissedJobsPerRestart` and `missedJobStaggerMs` in public cron configuration, with possible skip-catch-up behavior for user-facing jobs.

## Best Possible Solution

Define the public missed-run policy first, then add schema/types/help/docs/tests and gateway wiring for either the existing internal knobs plus clear semantics or a more explicit global/per-job skip-catch-up option.

## Reproduction Assessment

Yes for the missing config surface: current main's strict cron schema and `CronConfig` omit both requested fields, and startup still calls `runMissedJobs`. I did not execute tests because this review is read-only.

## Solution Assessment

No for a purely mechanical exposure of the current deps. The narrow maintainable fix needs an explicit policy for whether zero means skip catch-up, defer all catch-up, or use a separate `skipMissedRuns`/per-job option.
