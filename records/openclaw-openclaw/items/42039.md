## Summary

Keep open. Current main and the latest release still have only internal restart catch-up limit/stagger controls, while duplicate reports and the closed attempted PR point back to this issue as the canonical public cron missed-run policy decision.

## What This Changes

The issue requests public cron configuration for controlling missed-job restart catch-up limits/staggering and optionally skipping catch-up for user-facing jobs.

## Best Possible Solution

Define a cron-owned missed-run policy that directly expresses replay, skip, or bounded-freshness intent, then align `CronConfig`, strict schema/help/labels, gateway wiring, protocol/job policy if needed, docs, and restart catch-up tests.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: persist an enabled recurring job with a past `nextRunAtMs`, start `CronService`, and the current startup path replays or defers it without a public skip/freshness policy.

## Solution Assessment

No, not as stated yet. Raw exposure of the internal deps is only a plausible implementation detail; the maintainable fix should choose an explicit public missed-run policy because immediate-limit zero currently means deferred replay, not skip.
