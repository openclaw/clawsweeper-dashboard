## Summary

Keep open: current main still lacks the requested public cron config keys or an equivalent missed-run skip policy, while related duplicate reports were closed in favor of this issue as the canonical tracker. Directly exposing the two internal knobs is not clearly sufficient because `maxMissedJobsPerRestart: 0` currently defers missed jobs instead of skipping catch-up.

## What This Changes

Expose public cron missed-run controls in config so operators can tune or avoid startup catch-up duplicates and provider-rate bursts.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update the config type, strict schema, generated help/labels, docs, gateway wiring, and restart catch-up tests together.

## Reproduction Assessment

Yes for the public config gap: current source shows the strict public config surface does not accept the requested keys while the internal service deps still exist. The larger duplicate-message scenario was not live-run, but the startup catch-up path and tests make the behavior source-reproducible.

## Solution Assessment

No for the literal direct-expose proposal. The safer solution is an explicit missed-run policy because the existing internal zero-max behavior defers missed jobs instead of skipping them.
