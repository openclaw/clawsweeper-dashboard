## Summary

Keep open: current main and the latest release still lack the requested public cron config keys, and related duplicate reports were closed in favor of this issue as the canonical tracker. The requested direct plumbing is not clearly the right fix because the current internal `maxMissedJobsPerRestart: 0` behavior defers missed jobs instead of skipping catch-up.

## What This Changes

The issue requests public cron configuration for missed-job replay limits and restart staggering after gateway restarts.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update the config type, strict schema, generated labels/help, docs, gateway wiring, and restart catch-up tests together.

## Reproduction Assessment

Yes for the public config gap: current source shows the strict public `CronConfig` schema does not accept the requested keys while the internal service deps still exist. The larger duplicate-message scenario was not live-run, but the startup catch-up path and tests make the behavior source-reproducible.

## Solution Assessment

No. Directly exposing the two internal dependency fields is not the best complete solution because `maxMissedJobsPerRestart: 0` currently defers all missed jobs; the safer direction is an explicit public missed-run policy with tests for skip versus defer semantics.
