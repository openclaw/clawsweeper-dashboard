Keep open: current main and v2026.5.22 still keep the requested cron restart catch-up controls internal, the strict public cron schema would reject the proposed config keys, and the remaining behavior needs a maintainer policy decision rather than a narrow auto-fix.

## What This Changes

Expose or replace the internal cron missed-job restart catch-up controls with public configuration so users can stagger or skip catch-up runs after gateway restart.

## Best Possible Solution

Define a backward-compatible missed-run policy, then update cron config types, strict schema/help, docs, gateway wiring, optional per-job behavior, and restart catch-up regression tests together.

## Reproduction Assessment

Yes, at source level. Current main rejects the requested public cron keys, does not wire them into CronService, and still replays or defers startup missed jobs without a public skip policy.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because current internal max=0 behavior defers eligible jobs instead of disabling catch-up.
