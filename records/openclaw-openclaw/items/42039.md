## Summary

Keep open: current main still lacks the requested public cron missed-job controls or an equivalent skip policy, and related duplicate reports were closed in favor of this issue as the canonical tracker. Directly exposing the internal knobs is not clearly safe because the current zero-max path defers missed jobs rather than skipping catch-up.

## What This Changes

Expose cron missed-job catch-up controls in public configuration so users can limit, stagger, or skip replayed jobs after gateway restart.

## Best Possible Solution

Define a backward-compatible missed-run policy, then update CronConfig, strict schema/help/docs, gateway wiring, protocol if per-job controls are added, and restart catch-up tests together.

## Reproduction Assessment

Yes for the public config gap: current main shows strict public cron config does not accept the requested keys while the internal service deps still exist. The broader duplicate-message scenario is source-reproducible through the startup catch-up path and tests, but was not live-run.

## Solution Assessment

No for the literal direct-expose proposal. The narrow maintainable solution should define explicit missed-run semantics first because the existing maxMissedJobsPerRestart: 0 behavior defers eligible missed jobs instead of skipping catch-up.
