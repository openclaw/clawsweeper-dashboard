## Summary

Keep open: current main and latest release still do not expose these cron restart catch-up controls through public config, and the safest fix needs a maintainer policy choice because the raw internal limit/stagger knobs do not fully match the requested skip-catch-up behavior.

## What This Changes

The issue requests public cron configuration for missed-job restart catch-up limits and staggering, with optional per-job skip behavior to avoid duplicate user-facing deliveries after gateway restarts.

## Best Possible Solution

Define an explicit cron startup catch-up policy that preserves today's default behavior while adding opt-in skip/defer/stagger semantics with schema, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing-config part: source inspection shows the strict public `cron` schema and `CronConfig` omit the requested keys, while startup still evaluates missed jobs. I did not live-replay the reporter's 13-job restart scenario.

## Solution Assessment

No clear best fix yet: directly exposing the internal knobs is only a plausible path, because current `maxMissedJobsPerRestart: 0` behavior would defer rather than skip missed runs. The better solution is a named catch-up policy with explicit default and opt-in semantics.
