## Summary

Keep open: current main still keeps the requested restart catch-up controls internal, strict public cron config rejects those keys, and recent related closures point back to this issue as the canonical cron missed-run policy decision.

## What This Changes

Expose missed-job restart catch-up controls in public cron config so operators can stagger or disable overdue cron replay after gateway restarts.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update config types, strict schema/help, docs, gateway wiring, optional per-job protocol, and restart catch-up tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows the public cron config omits and strictly rejects the requested keys, while startup catch-up still replays or defers overdue jobs and the restart-catchup test asserts immediate replay.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because maxMissedJobsPerRestart: 0 currently defers eligible jobs instead of skipping catch-up.
