## Summary

Keep this open: current main and the latest release still expose the missed-job controls only as internal CronService dependencies, while startup catch-up remains active and the public config/API decision is unresolved.

## What This Changes

Expose cron missed-job catch-up controls such as maxMissedJobsPerRestart and missedJobStaggerMs in public configuration so users can reduce or disable startup replay.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update CronConfig, strict schema, generated help/labels, docs, gateway wiring, and restart-catchup tests together.

## Reproduction Assessment

Yes, from source inspection: the public CronConfig and strict zod cron schema omit the requested keys, while startup still calls runMissedJobs and tests assert overdue jobs replay on start. I did not execute the gateway because this review is read-only, but the code path is clear.

## Solution Assessment

No, exposing only the existing internal deps is not enough because maxMissedJobsPerRestart: 0 would defer missed jobs rather than disable catch-up. The better solution is an explicit missed-run policy with schema/docs/tests and clear global versus per-job semantics.
