## Summary

Keep open: current main still has `maxMissedJobsPerRestart` and `missedJobStaggerMs` only as internal cron service dependencies, while the public cron config type, strict Zod schema, generated metadata, and docs omit them. The requested disable behavior also needs maintainer API judgment because `maxMissedJobsPerRestart: 0` would currently defer missed jobs to near-future staggered runs, not skip catch-up until the next scheduled slot.

## What This Changes

Expose cron missed-job startup catch-up controls in public configuration so users can throttle or disable restart catch-up bursts.

## Best Possible Solution

Define a public cron missed-run policy that can skip catch-up until the next scheduled slot or throttle catch-up, then wire the chosen fields through `CronConfig`, Zod/schema metadata, Gateway construction, focused cron tests, and docs.

## Reproduction Assessment

Yes, from source inspection: current main has internal missed-job catch-up deps but no public config type/schema/docs or Gateway mapping for the requested keys. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written: exposing `maxMissedJobsPerRestart` verbatim would help throttle immediate runs, but it would not disable catch-up under the current deferred-overflow implementation. The maintainable fix should first settle the public catch-up policy and names.
