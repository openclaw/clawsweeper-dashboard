## Summary

Keep open: current main still lacks public cron missed-job controls or an equivalent skip/freshness policy, and duplicate reports were closed in favor of this issue as the canonical tracker. The request needs maintainer API/product review because exposing the raw internal knobs would add public config while leaving the important skip-versus-defer semantics unresolved.

## What This Changes

Expose cron startup missed-job controls in public configuration so users can limit, stagger, or skip catch-up runs after gateway restart.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update CronConfig, strict schema/help/docs, gateway wiring, protocol if per-job controls are added, and restart catch-up tests together.

## Reproduction Assessment

Yes. The public config gap is source-reproducible: current main’s strict CronConfig schema excludes the requested keys while internal CronServiceDeps still uses them, and restart catch-up tests assert overdue job replay/deferral behavior.

## Solution Assessment

No for the literal direct-expose proposal. The safer solution is to define explicit missed-run semantics first, because maxMissedJobsPerRestart: 0 currently defers eligible missed jobs rather than skipping catch-up.
