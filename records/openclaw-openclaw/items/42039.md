## Summary

Keep open: current main still has the reported public cron configuration gap, and the related duplicate reports/closed partial PR point back here as the canonical cron missed-run policy decision.

## What This Changes

Expose public cron restart missed-job controls for maximum immediate catch-up runs, catch-up staggering, and possibly per-job skip behavior.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update CronConfig, strict schema/help/docs, gateway wiring, per-job protocol if chosen, and restart catch-up tests together.

## Reproduction Assessment

Yes. This is source-reproducible: current main rejects the requested public cron keys, does not wire them through the gateway, and still runs or defers missed jobs on startup without a public skip policy.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because maxMissedJobsPerRestart: 0 currently means defer eligible missed jobs, not skip catch-up.
