## Summary

Keep open: current main and the latest release still keep the missed-job max/stagger controls internal, while closed related reports point here as the canonical cron missed-run policy/API decision.

## What This Changes

Expose cron missed-job restart controls in public configuration so users can limit, stagger, or disable startup catch-up behavior.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update CronConfig, strict schema/help/docs, gateway wiring, optional per-job protocol, and restart catch-up tests together.

## Reproduction Assessment

Yes. Source inspection on current main shows the strict cron config/schema/help/protocol omit these keys while startup still invokes runMissedJobs, so a user config with these fields would be rejected before reaching CronService.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because the current internal max/stagger controls implement replay/deferral, not a clear public skip-catch-up contract.
