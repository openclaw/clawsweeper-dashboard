## Summary

Current main still lacks a public missed-run policy: restart catch-up consumes internal-only limit and stagger dependencies, while the public cron type, schema, gateway wiring, and docs expose no corresponding control. Related duplicate reports were deliberately consolidated into this issue, so meaningful product work remains.

## What This Changes

Expose public controls for limiting, staggering, or skipping overdue cron-job replay after gateway restart.

## Best Possible Solution

Define one backward-compatible missed-run policy in cron-owned config that directly expresses replay, skip, or bounded-freshness intent; align type, strict schema/help, gateway wiring, docs, and restart tests, adding per-job control only if the global policy cannot cover established use cases.

## Reproduction Assessment

Yes. Current source and restart tests provide a high-confidence path: persist an enabled recurring job with a past nextRunAtMs, start CronService, and observe startup replay or staggered deferral rather than a configurable skip policy.

## Solution Assessment

No, the proposed raw field exposure is not yet the best proven solution. The API should express missed-run intent directly because the existing maximum distinguishes immediate replay from deferral, not replay from skipping, and global versus per-job behavior needs an explicit product decision.
