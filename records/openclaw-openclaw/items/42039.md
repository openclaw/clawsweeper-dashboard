## Summary

Keep open: current main and the latest release still lack the requested public cron missed-job controls or an equivalent missed-run policy, and related duplicate reports were closed in favor of this issue as the canonical tracker. The literal direct exposure needs maintainer API/product review because the current internal maxMissedJobsPerRestart: 0 path defers missed jobs into staggered future ticks rather than skipping catch-up.

## What This Changes

The issue asks to expose cron missed-job catch-up controls in public config so users can limit, stagger, or disable restart catch-up runs.

## Best Possible Solution

Define a backward-compatible missed-run policy, then update CronConfig, strict schema/help/docs, gateway wiring, protocol if per-job controls are added, and restart catch-up tests together.

## Reproduction Assessment

Yes for the public config gap: current main source shows strict public cron config does not accept the requested keys while the internal service deps still exist. ## Solution Assessment

No for the literal direct-expose proposal. The safer solution is to define explicit missed-run semantics first, because the current maxMissedJobsPerRestart: 0 behavior defers eligible missed jobs instead of skipping catch-up.
