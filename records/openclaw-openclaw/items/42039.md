## Summary

Keep open: current main still keeps the missed-job restart controls internal, strict public cron config rejects the requested keys, and the reporter confirmed this remains relevant on May 20, 2026. Related duplicate skip/catch-up reports point back to this issue as the canonical API/product decision, and the literal expose-only proposal is not enough because the current internal max=0 behavior would defer missed jobs rather than skip catch-up.

## What This Changes

Expose cron missed-job restart controls through public configuration and consider an optional per-job catch-up skip policy.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update config types, strict schema/help, docs, gateway wiring, optional per-job protocol, and restart catch-up tests together.

## Reproduction Assessment

Yes, source-level. Current main rejects the requested public cron keys, does not wire them into CronService, and still replays or defers startup missed jobs without a public skip policy.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because the current internal max=0 behavior defers eligible jobs instead of disabling catch-up.
