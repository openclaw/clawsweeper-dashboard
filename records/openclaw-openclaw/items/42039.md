## Summary

Keep open. Current main still keeps `maxMissedJobsPerRestart` and `missedJobStaggerMs` as internal cron service dependency knobs only; the public cron config type, strict validation schema, generated schema/help metadata, Gateway wiring, and docs still do not expose them. The exact PR attempt is closed and unmerged, the open adjacent PR is only a partial stagger proposal, and the newer overlapping issue does not supersede this more specific tracker.

## What This Changes

Expose cron startup missed-job catch-up controls so users can configure the max immediate replay count and missed-job stagger delay from `openclaw.json`.

## Best Possible Solution

Keep this as the canonical specific tracker for public cron missed-job catch-up controls. A complete implementation should add backward-compatible global cron config fields, preserve current defaults, define `maxMissedJobsPerRestart: 0` as an actual skip-to-next-scheduled-time behavior if that is the accepted public contract, wire the settings into Gateway cron construction, update generated config/help/docs/changelog, and add regression tests for zero, positive limits, stagger timing, and interrupted-startup behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction for the config gap: the requested keys are absent from `CronConfig`, strict zod validation, generated schema/help metadata, Gateway dep wiring, and docs while the internal deps remain in use.

## Solution Assessment

Yes in direction, but not as a bare pass-through. Exposing these controls is the narrow maintainable fix for this issue, provided the implementation explicitly handles the requested zero/disable semantics instead of relying on the current internal deferred-overflow behavior.
