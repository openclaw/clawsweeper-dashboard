## Summary

Keep open: current main still has the reported public cron config gap and no equivalent skip/freshness policy for missed restart runs. The related closed duplicate reports and the blocked partial PR point back to this issue as the canonical place for the cron policy decision.

## What This Changes

Expose or replace cron missed-job restart controls so users can prevent or stagger overdue cron runs after gateway restart.

## Best Possible Solution

Define a backward-compatible cron missed-run policy, then update CronConfig, strict schema/help/docs, gateway wiring, and restart catch-up tests together.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main’s strict public cron config excludes the requested keys while internal CronServiceDeps still uses them, and startup catch-up runs without a public missed-run policy gate.

## Solution Assessment

No for the literal direct-expose proposal. The safer solution is to design explicit missed-run semantics first, because maxMissedJobsPerRestart: 0 currently defers eligible missed jobs rather than skipping catch-up.
