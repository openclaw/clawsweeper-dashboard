Keep open: current main and the latest release still have the same source-level gap: the restart catch-up knobs exist only as internal `CronServiceDeps`, not as accepted public cron config. ## What This Changes

Request to expose `maxMissedJobsPerRestart` and `missedJobStaggerMs` under public cron configuration, with an optional per-job skip-catch-up policy for user-facing jobs.

## Best Possible Solution

Keep this as the canonical cron missed-run policy tracker, then choose a compatibility-safe public API before wiring schema, help/docs, Gateway construction, runtime semantics, and focused restart-catch-up tests in one change.

## Reproduction Assessment

Yes. Source inspection reproduces the config gap: the requested keys are present only in internal cron service deps, absent from public `CronConfig` and strict zod config, and not wired from Gateway config into `CronService`.

## Solution Assessment

Unclear. Exposing the internal knobs is a plausible implementation, but the best solution should first define the public missed-run policy semantics, especially whether disabling catch-up means staggered deferment or advancing to the next natural schedule.
