## Summary

Keep open: current main and v2026.5.20 still keep the restart catch-up controls internal, the public cron config schema rejects the requested keys, and the reporter confirmed this remains relevant on May 20, 2026. This is a real cron delivery policy problem, but the fix needs maintainer/API direction rather than a literal expose-only patch.

## What This Changes

Expose cron restart missed-run catch-up controls in public configuration so operators can limit, stagger, or skip overdue job replay after gateway restart.

## Best Possible Solution

Define a backward-compatible missed-run policy, then update cron config types, strict schema/help, docs, gateway wiring, optional per-job behavior, and restart catch-up regression tests together.

## Reproduction Assessment

Yes, source-level. Current main rejects the requested public cron keys, does not wire them into CronService, and still replays or defers startup missed jobs without a public skip policy.

## Solution Assessment

No for the literal expose-only proposal. The safer solution is an explicit missed-run policy because the current internal max=0 behavior defers eligible jobs instead of disabling catch-up.
