## Summary

Current main still reads the installed plugin index synchronously before plugin loader cache lookup, so this is a valid narrow performance bug rather than a cleanup close.

## What This Changes

Avoid redundant synchronous installed-plugin-index reads on hot plugin tool/runtime registry resolution paths when the loader cache can be reused.

## Best Possible Solution

Keep installed-index changes visible, but move install-record loading out of loader-cache hit paths by reusing an invalidated load-context/install-record fingerprint or an explicit current metadata snapshot.

## Reproduction Assessment

Yes from source inspection: a warm `loadOpenClawPlugins()` cache still computes `resolvePluginLoadCacheContext()` first, and that performs the sync installed-index read before `getReusableCachedPluginRegistry()`. I did not run a write-producing benchmark in this read-only review.

## Solution Assessment

The reported process-lifetime memoization is not the safest exact fix because installed-index writes need to stay observable. The narrow maintainable fix is to skip the sync read on proven cache hits while tying any cached install-record state to existing plugin loader/index invalidation semantics.
