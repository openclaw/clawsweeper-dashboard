## Summary

Keep open: current main still reads the installed-plugin index synchronously while building the plugin loader cache context before checking the runtime loader LRU, so the reported hot-path cost is source-confirmed. The fix should avoid the extra read without adding a hidden lifetime metadata cache.

## What This Changes

Avoid redundant synchronous installed-plugin index reads before plugin loader LRU cache lookup during resolvePluginTools/runtime loading.

## Best Possible Solution

Thread already-owned metadata snapshot install records, or an equivalent explicit installRecords input, into runtime loader cache-key construction so warm loader hits avoid rereading the installed index while preserving install/config invalidation.

## Reproduction Assessment

Yes, by source inspection. resolvePluginTools can fall through to loadOpenClawPlugins, which calls resolvePluginLoadCacheContext and synchronously reads installed records before the PluginLoaderCacheState cache lookup.

## Solution Assessment

No, not as the report's process-lifetime memoization. The safer maintainable fix is to reuse explicit caller-owned snapshot/installRecords for runtime load options and keep the installed-index freshness contract intact.

## Repair Work Prompt

Fix the current issue in a focused PR. Current main builds resolvePluginTools runtime load options from a PluginMetadataSnapshot, but loadOpenClawPlugins/resolvePluginLoadCacheContext still calls loadInstalledPluginIndexInstallRecordsSync before checking PluginLoaderCacheState. Add a narrow internal seam to pass already-loaded snapshot.index.installRecords, or equivalent explicit installRecords, through PluginLoadOptions/runtime load options into resolvePluginLoadCacheContext so cache-key construction and discovery use caller-owned records and warm loader cache-hit paths do not reread the installed index. Do not add a hidden process-lifetime installed-index cache, env/config cache knob, or broad cache-size workaround. Preserve docs/plugins/architecture-internals.md metadata freshness rules and existing cache partitioning by install record paths/env. Add regression coverage that observes/mocks loadInstalledPluginIndexInstallRecordsSync and proves a warm resolvePluginTools or loadOpenClawPlugins path with caller-owned records avoids the sync reader, while callers without explicit records still read as before. Add a CHANGELOG.md fix entry if the maintainers treat the perf improvement as user-facing.
