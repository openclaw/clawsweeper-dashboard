## Summary

Keep open: the bug fix is directionally right and the PR includes real copied live-output proof, but the new direct-delivery matcher can still bypass parent-owned ACP suppression for a different thread that shares the same parent channel target.

## What This Changes

The PR changes ACP reply dispatch so parent-owned ACP child sessions can deliver replies when the inbound turn matches their saved delivery context, with unit coverage for bound Discord thread delivery and off-thread suppression.

## Best Possible Solution

Land a tightened version that bypasses suppression only for compatible channel/account/thread-or-target routes, with regression coverage for same-target but different-thread turns.

## Reproduction Assessment

Yes, source-reproducible: current main unconditionally suppresses parent-owned ACP child user delivery, and the PR body gives a real Discord/ACP follow-up path plus added unit coverage for the primary case. I did not run local tests because this review is read-only.

## Solution Assessment

No as written; the maintainable fix is to keep the direct bound-thread bypass but reject explicit thread mismatches before falling back to target equality.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require thread match before using the target fallback:** `src/auto-reply/reply/dispatch-from-config.ts:154`
  - body: When both the saved delivery context and the incoming turn have thread ids, a mismatch still falls through to `deliveryTo === turnTo`. For threaded channels where `OriginatingTo` is the parent channel and `MessageThreadId` selects the thread, a turn from another thread in the same channel would unsuppress a parent-owned ACP child and send its output visibly there. Return false on explicit thread mismatch before the `to` fallback, and add a same-target/different-thread regression test.
  - confidence: 0.9
