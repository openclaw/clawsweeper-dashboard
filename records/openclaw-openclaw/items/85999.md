## What This Changes

The issue reports a v2026.5.22 regression where provider-auth prewarm starves the gateway event loop during startup, delaying channel handshakes and inbound messages.

## Best Possible Solution

Make provider-auth prewarm cooperative enough that channel timers and inbound message dispatch keep running during startup, while preserving the prepared auth-state cache semantics for model/status hot paths.

## Reproduction Assessment

Yes, there is a high-confidence source-backed reproduction path: restart a multi-provider gateway on a small Linux host and send an inbound channel message during the provider-auth prewarm window. I did not run that live VM scenario in this read-only review.

## Solution Assessment

No current fix is present. The maintainable fix should change the provider-auth warm path to avoid long event-loop monopolization rather than adding a user-facing workaround knob.
