## Summary

Keep open: current main still runs provider auth pre-warm in the gateway process, and the reported post-ready event-loop starvation is not fully solved by the shipped v2026.5.22 mitigations or the small main-only read-only catalog change.

## What This Changes

The issue reports a v2026.5.22 regression where provider auth pre-warm monopolizes the Node event loop after gateway startup and causes channel handshakes and early inbound messages to time out or stall.

## Best Possible Solution

Keep provider auth state warming useful for model-listing performance, but move or scope the expensive work so channel startup and early inbound delivery stay responsive, with proof for constrained Linux startup and focused regression coverage around event-loop responsiveness.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The reporter provided two stable-release production restarts with matching prewarm metrics, and current main still contains the same in-process provider-auth warm path that can block channel timers.

## Solution Assessment

Unclear until owner review chooses the exact mitigation. The current main-only read-only catalog change is not enough by itself; a safer fix needs to preserve auth-state semantics while proving channel handshakes are not starved.
