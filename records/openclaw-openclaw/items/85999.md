## Label Justifications

- `P1`: The report describes a latest-release regression that breaks channel handshakes and delays inbound messages after gateway restarts.
- `impact:message-loss`: Inbound messages sent during the startup prewarm window can be delayed enough to miss channel timers or arrive late.

## Summary

Keep open: current main still schedules provider-auth prewarm as a gateway lifetime sidecar, and the warm path still performs per-agent auth-store/external-CLI discovery before serial provider checks. The related https://github.com/openclaw/openclaw/pull/86003 is an open draft mitigation for operator controls and explicitly does not close the underlying event-loop starvation bug.

## What This Changes

The issue reports provider-auth prewarm monopolizing the gateway event loop during startup and delaying or breaking channel handshakes.

## Best Possible Solution

Make provider-auth prewarm cooperative or move its slow external-auth work off the gateway event loop, while preserving the prepared auth-state cache semantics for model/status hot paths.

## Reproduction Assessment

Yes, there is a high-confidence source-backed reproduction path: restart a gateway with provider-auth prewarm enabled and send an inbound channel message during the warm window. I did not run the live VM/macOS restart scenario in this read-only review.

## Solution Assessment

No current fix is present. The best repair is to make the warm path cooperative/cached enough that channel timers continue to run, not only to add an operator disable knob.
