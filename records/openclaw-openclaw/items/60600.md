## Summary

Keep open. Current main still puts the Gemini native-PDF API key in the request URL, and this PR is a narrow, directionally correct fix that matches Google's REST API-key guidance and existing Google provider patterns. It is not superseded by the broader stale native-PDF SSRF PR, which remains a separate security thread. citeturn0search1

## What This Changes

This PR changes `geminiAnalyzePdf` to call Gemini `:generateContent` without a `?key=` query parameter and sends the normalized API key in the `x-goog-api-key` header instead.

## Best Possible Solution

Rebase and land the narrow header-auth fix for Gemini native PDF, ideally with a focused unit assertion that the fetch URL has no sensitive `key` query parameter and the request headers carry `x-goog-api-key`. Keep broader native-PDF SSRF/proxy hardening separate from this XS auth-placement change.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `geminiAnalyzePdf` builds a Gemini URL containing `?key=` before `fetch`; a focused unit test can stub `fetch` and inspect the URL and headers without live credentials.

## Solution Assessment

Yes for the production code change. Moving the API key from the URL to `x-goog-api-key` is the narrowest maintainable fix for this PR's stated problem; adding regression coverage would make it safer to land.
