## Summary

Keep open for a narrow docs fix before merge. The code direction matches a real current-main behavior, and Greptile's observability concern was addressed, but the PR introduces a user-facing env var that disables state-dir permission hardening without documenting the safe-use and secret-storage implications.

## What This Changes

The PR adds an `OPENCLAW_SKIP_STATE_DIR_HARDEN` opt-out around config state-dir chmod hardening, a regression test for preserving a `0o2770` shared state dir, and a changelog entry.

## Best Possible Solution

Land the narrow opt-out after adding user-facing docs that frame it as an advanced multi-uid deployment override while keeping default `0o700` hardening as the normal path.

## Reproduction Assessment

Yes. Current main directly reproduces the mode change: a Linux config write against the default state/config dir calls the hardening helper and chmods group/other-accessible dirs to `0o700`; the PR's added unit test is the right focused check for preserving `0o2770` when the env var is set.

## Solution Assessment

Yes, with the docs addition. An opt-in env guard inside the existing hardening helper is narrower and safer than changing the default mode or moving config paths, provided the security tradeoff is documented.
