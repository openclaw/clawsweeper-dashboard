## Summary

Keep open: the proposed opt-out is incomplete because current config writes still pass `dirMode: 0o700` to the atomic writer, whose pinned implementation chmods the parent directory. The PR also needs real multi-uid behavior proof and explicit maintainer/security approval for the state-directory hardening exception.

## What This Changes

The PR adds an `OPENCLAW_SKIP_STATE_DIR_HARDEN` env-var guard and accepted-env log to config directory hardening, plus a regression test and changelog entry.

## Best Possible Solution

If maintainers accept this deployment mode, implement one explicit operator opt-out that covers both the manual hardening helper and the atomic writer's directory mode, align audit/doctor messaging, and verify it in a real multi-uid setup.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the PR defect: the guard returns from `tightenStateDirPermissionsIfNeeded`, but `writeConfigFile` still reaches `replaceFileAtomic({ dirMode: 0o700 })`, and fs-safe 0.2.4 chmods the parent directory.

## Solution Assessment

No. Guarding only the helper is not the narrow complete fix; the accepted opt-out needs to flow through the atomic write path and be reconciled with security audit/repair behavior.

## Review Findings
