## Summary

Keep open: this targets a real multi-UID state-directory problem, but the patch is not merge-ready because the new opt-out only skips the pre-write hardener while current main still calls `replaceFileAtomic` with `dirMode: 0o700`, and fs-safe chmods the parent directory to that mode during the actual config write.

## What This Changes

The PR adds an opt-in `OPENCLAW_SKIP_STATE_DIR_HARDEN` env var, logs when it is accepted, adds a config-write unit test for preserving a setgid state directory, and adds a changelog entry.

## Best Possible Solution

Make the permission policy cover the whole config write path, including fs-safe atomic replacement, and land it only with maintainer security approval plus real multi-UID deployment proof.

## Reproduction Assessment

Yes at source level: current main writes config through `replaceFileAtomic` with `dirMode: 0o700`, and the fs-safe implementation chmods the parent directory to that mode. I did not run a live multi-UID Fly/Kubernetes deployment in this read-only review.

## Solution Assessment

No. The proposed env guard is only a plausible partial fix because it skips `tightenStateDirPermissionsIfNeeded` but not the later atomic writer chmod, so the best fix must cover the whole config write boundary or choose a different maintainer-approved policy.
