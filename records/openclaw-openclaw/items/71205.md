## Summary

Keep open for maintainer/security review. Current main clearly hardens the state/config directory on config writes and the PR is a narrow opt-in escape hatch, but it adds a user-facing permission-hardening bypass that needs explicit approval and durable operator docs before merge.

## What This Changes

The PR adds an `OPENCLAW_SKIP_STATE_DIR_HARDEN` env guard and accepted-env log in config IO, a regression test preserving a `0o2770` shared state dir, and a changelog entry.

## Best Possible Solution

Keep default state-dir hardening, accept the opt-in bypass only as a documented advanced deployment escape hatch, and leave broader permission configurability to #56263.

## Reproduction Assessment

Yes. Source inspection on current main shows config writes call the hardening helper and chmod matching state/config dirs with group/other bits to `0o700`; the existing test asserts that default behavior.

## Solution Assessment

Mostly yes, but not merge-ready. A narrow opt-in env guard is the maintainable sidecar escape hatch, but it should land only with maintainer/security approval and docs rather than as an undocumented permissions-policy exception.
