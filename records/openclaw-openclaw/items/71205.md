## Summary

Keep open. Current main still chmods the default state-backed config directory to 0o700 on config writes, so the reported multi-uid sidecar problem remains real; this PR is a narrow opt-in implementation with direct regression coverage and no blocking code-review finding from this pass.

## What This Changes

The PR adds a truthy `OPENCLAW_SKIP_STATE_DIR_HARDEN` guard with accepted-env logging before config state-dir chmod hardening, adds a POSIX regression test preserving a 0o2770 state directory, and records an Unreleased changelog entry.

## Best Possible Solution

Land a maintainer-approved version if OpenClaw wants this operational exception: keep default state-dir hardening unchanged, allow skipping only through a trusted operator environment variable, log accepted use, document the multi-uid shared-volume case if maintainers want a stable operator reference, and retain tests for both default 0o700 hardening and setgid preservation with the opt-out enabled.

## Reproduction Assessment

Yes. On current main, a POSIX default state/config dir with group or other permission bits reaches `tightenStateDirPermissionsIfNeeded` through `writeConfigFile` and is chmodded to 0o700; the PR's added test mirrors the reported 0o2770 multi-uid case with the skip env enabled.

## Solution Assessment

Yes for the implementation shape. A truthy env guard before stat/chmod is the narrowest maintainable code path that preserves current secure defaults while supporting explicitly managed multi-uid shared-volume deployments.
