## Summary

Keep open: the PR targets a real multi-uid state-directory permission problem, but it is not merge-ready because the new guard only skips `tightenStateDirPermissionsIfNeeded` while current main’s atomic config writer still chmods the config directory to `0o700`. ## What This Changes

The PR adds an opt-in `OPENCLAW_SKIP_STATE_DIR_HARDEN` env var to bypass config state-dir chmod hardening, plus a config write regression test and changelog entry.

## Best Possible Solution

Do not merge as-is; if maintainers want this operator escape hatch, make it cover the complete config write path including fs-safe atomic replacement, align doctor/audit/docs expectations, and require redacted real-environment proof before merge.

## Reproduction Assessment

Yes for source-level behavior: current main hardens a group-accessible config/state directory to `0o700`, and the patched helper would not stop the later fs-safe atomic writer from chmodding that same directory. I did not run a live multi-uid container in this read-only review.

## Solution Assessment

No. The PR is a plausible mitigation, but it is too narrow on current main because the atomic writer still receives `dirMode: 0o700`; the best fix needs a maintainer-approved policy that covers the complete config write path.
