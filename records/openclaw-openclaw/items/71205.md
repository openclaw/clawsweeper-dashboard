## Summary

Keep open: the deployment problem is plausible, but this branch is not mergeable as-is because current main still chmods the config parent through the atomic writer after the new guard returns, and the security/policy plus real-environment proof gates remain unresolved.

## What This Changes

The PR adds an `OPENCLAW_SKIP_STATE_DIR_HARDEN` env-var guard and accepted-env log around config directory hardening, a regression test for preserving a 02770 state directory, and a changelog entry.

## Best Possible Solution

If maintainers accept this deployment mode, implement one explicit operator opt-out that covers both the manual hardening helper and the atomic writer's directory mode, then align audit/doctor messaging and verify it in a real multi-uid setup.

## Reproduction Assessment

Yes. Source inspection of current main plus the pinned `@openclaw/fs-safe` 0.2.4 source gives a high-confidence path: `writeConfigFile` reaches `replaceFileAtomic({ dirMode: 0o700 })`, which chmods the parent directory even if the proposed guard returns.

## Solution Assessment

No. Guarding only `tightenStateDirPermissionsIfNeeded` is not enough on current main; the fix needs to thread the accepted opt-out through the atomic-write directory mode path and reconcile the security audit/repair contract.
