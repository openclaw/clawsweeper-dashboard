## Summary

Keep open. The PR addresses a real multi-uid permissions problem, but the current diff is incomplete because current main still chmods the config parent directory through the atomic writer path; the proposed hardening opt-out also needs maintainer/security approval and real behavior proof before merge.

## What This Changes

The PR adds an `OPENCLAW_SKIP_STATE_DIR_HARDEN` environment-variable opt-out in config IO, plus a regression test and changelog entry for preserving a group-writable setgid state directory.

## Best Possible Solution

If maintainers accept this deployment mode, implement one explicit operator-approved permissions policy or opt-out that covers both config hardening and atomic writer directory modes, then align audit/doctor/fix messaging and verify it in a real multi-uid setup.

## Reproduction Assessment

Yes. Source inspection on current main plus `@openclaw/fs-safe` 0.2.4 shows `writeConfigFile` reaches `replaceFileAtomic({ dirMode: 0o700 })`, whose implementation chmods the parent directory, so the proposed guard alone does not preserve a `02770` state dir.

## Solution Assessment

No. Guarding only `tightenStateDirPermissionsIfNeeded` is not sufficient; the maintainable fix needs to cover the atomic writer directory mode path and reconcile the security audit/repair contract.
