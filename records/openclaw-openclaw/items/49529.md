## Best Possible Solution

Rebase or replace the PR with a current-main implementation that keeps the strict provider config/schema/docs/tests, moves Google-specific payload injection into the Google provider/provider-hook path, fixes wrapper reapplication leakage, and includes redacted real Gemini runtime proof.

The blocking wrapper bugs are source-reproducible from the PR branch because wrappers persist across repeated `applyExtraParamsToAgent` composition paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Scope tool_stream injection to Z.AI payloads:** `src/agents/pi-embedded-runner/extra-params.ts:468`
  - body: This wrapper is installed when the provider normalizes to Z.AI, but the wrapper itself patches every later payload and is never removed. If the same agent is reused for another provider, strict OpenAI-compatible backends can receive an unexpected top-level `tool_stream` field and reject the request.
  - confidence: 0.86
- **[P2] Unwrap nested safety wrappers before reapplying:** `src/agents/pi-embedded-runner/extra-params.ts:335-337`
  - body: `stripGoogleSafetySettingsWrapper` only peels a top-level safety wrapper. In real runs, `run/attempt.ts` adds more wrappers after `applyExtraParamsToAgent`, so a later apply can leave the old safety wrapper nested inside and let stale thresholds populate `config.safetySettings` before the new wrapper can update or clear them.
  - confidence: 0.83
