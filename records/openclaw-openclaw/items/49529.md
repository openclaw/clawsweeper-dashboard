## Summary

Keep this PR open. Current main still does not accept or apply Gemini `safetySettings`, and the PR targets a real user-facing gap, but the proposed patch is not merge-ready because it can leave stale safety and `tool_stream` payload wrappers active across later runs.

## What This Changes

The PR adds `models.providers.*.safetySettings` config/schema/docs/generated baselines/tests and injects configured Gemini safety thresholds into `google-generative-ai` payloads, while also touching Z.AI `tool_stream` handling and scoped Vitest config setup.

## Best Possible Solution

Land a narrowed implementation that keeps `safetySettings` provider-scoped, strictly validates the supported Gemini categories and thresholds, injects them only for the active `google-generative-ai` provider while preserving explicit request payload values, excludes `google-gemini-cli`, documents the config, and avoids unrelated Z.AI/Vitest changes unless they are separately justified.

## Reproduction Assessment

Yes. On current main, adding `models.providers.<provider>.safetySettings` to a Gemini provider is enough to reproduce the gap because the strict provider schema omits the field and source search shows no runtime payload injector.

## Solution Assessment

No, not as currently proposed. The provider-scoped config direction is maintainable, but the patch needs to rebuild or unwrap payload wrappers from a stable base and keep Z.AI `tool_stream` mutations scoped so stale wrappers cannot affect later provider runs.
