Keep this PR open: the feature is useful and still missing from current main, but the branch should not merge as-is because it lacks real Gemini behavior proof and still has blocking payload/wrapper issues on a stale provider-runtime architecture.

## What This Changes

The branch adds `models.providers.*.safetySettings` config/schema/docs/baselines/tests and generic agent stream wrappers that inject Gemini safety thresholds, along with related Z.AI `tool_stream` wrapper changes.

## Best Possible Solution

Rebase onto current main and implement Gemini safety defaults in the Google provider/native transport path, preserving explicit per-request settings and adding redacted real Gemini runtime proof.

## Solution Assessment

No. The config/schema direction is useful, but the current branch injects the wrong request shape for current OpenClaw's native Google transport and carries stale generic wrapper behavior that should be provider-owned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Write safetySettings to the Gemini request body:** `src/agents/pi-embedded-runner/extra-params.ts:313-320`
  - body: This wrapper only creates `payload.config.safetySettings`, but current OpenClaw's Google transport sends a REST `GenerateContentRequest` with top-level fields such as `generationConfig`, `tools`, and `toolConfig`. The configured safety settings therefore do not reach the native Gemini request path and the new config can be accepted without taking effect.
  - confidence: 0.88
- **[P1] Scope tool_stream injection to Z.AI payloads:** `src/agents/pi-embedded-runner/extra-params.ts:468`
  - body: This branch installs the Z.AI `tool_stream` wrapper from the generic extra-params path, but the wrapper itself patches every later payload and is never stripped. If the same agent is reused for another provider, strict OpenAI-compatible backends can receive an unexpected top-level `tool_stream` field and reject the request.
  - confidence: 0.86
- **[P2] Unwrap nested safety wrappers before reapplying:** `src/agents/pi-embedded-runner/extra-params.ts:333-337`
  - body: `stripGoogleSafetySettingsWrapper` only peels a top-level safety wrapper. In real runs, later wrappers are added after `applyExtraParamsToAgent`, so a later apply can leave the old safety wrapper nested inside and let stale thresholds populate `config.safetySettings` before the new wrapper can update or clear them.
  - confidence: 0.8
