## Best Possible Solution

Rework or replace the branch with a provider-owned Google implementation that adds strict config, writes top-level REST `safetySettings`, preserves explicit request settings, avoids unrelated Z.AI churn, and includes real Gemini proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Write safetySettings to the Gemini request body:** `src/agents/pi-embedded-runner/extra-params.ts:313-320`
  - body: This wrapper creates `payload.config.safetySettings`, but OpenClaw’s current Google transport sends a REST `GenerateContentRequest` where `safetySettings` is a top-level field. The new config can validate successfully without changing the native Gemini request that is actually sent.
  - confidence: 0.88
- **[P1] Scope tool_stream injection to Z.AI payloads:** `src/agents/pi-embedded-runner/extra-params.ts:468`
  - body: The branch installs the Z.AI `tool_stream` wrapper from the generic extra-params path, and the helper writes `tool_stream=true` to every payload it sees. If the same agent later runs another provider through a retained wrapper, strict OpenAI-compatible backends can receive an unexpected top-level field and reject the request.
  - confidence: 0.86
- **[P2] Unwrap nested safety wrappers before reapplying:** `src/agents/pi-embedded-runner/extra-params.ts:333-337`
  - body: `stripGoogleSafetySettingsWrapper` only removes a top-level safety wrapper. Current run setup adds wrappers after `applyExtraParamsToAgent`, so an older safety wrapper can remain nested and populate stale thresholds before the new wrapper can update or clear them.
  - confidence: 0.8
- **[P3] Support the civic-integrity safety category:** `src/config/types.models.ts:59-64`
  - body: The new config type is strict but exposes only four categories, while the current Gemini API reference lists `HARM_CATEGORY_CIVIC_INTEGRITY` as supported. Users who need that threshold cannot configure it through the new field.
  - confidence: 0.76
