## Summary

Keep open: this PR has a protected `maintainer` label and touches a gateway authorization boundary. The security intent is valid, but current `main` already centralizes write-scoped reset protection through the session reset authorization path, while this PR adds a second hard-coded pre-dispatch guard and lacks real behavior proof.

## What This Changes

The PR rejects `chat.send` `/new` and `/reset` requests without `operator.admin`, extracts the reset-command regex, and adds a gateway regression test.

## Best Possible Solution

Rebase onto current `main`, keep the centralized reset authorization path as the source of truth, and only add a narrow error/reporting change if maintainers explicitly want `chat.send` reset attempts to fail closed at the RPC layer.

## Reproduction Assessment

Yes, by source inspection: the PR's new guard checks only `RESET_COMMAND_RE`, while reset behavior is configured through `session.resetTriggers` and current main now authorizes that centralized path. I did not run the gateway because this was a read-only review.

## Solution Assessment

No, not as-is. The maintainable solution is to keep reset authorization in the session reset pipeline so default and configured triggers share one policy, then decide separately whether a non-admin `chat.send` reset should return an RPC error or proceed as a normal message without rotation.
