## Summary

Keep open. Current main still feeds the `before_prompt_build` prompt into Active Memory `queryMode: "message"` without a cap or slim-intent boundary, and the linked closing fix at https://github.com/openclaw/openclaw/pull/92099 is still open and unmerged.

## What This Changes

The issue requests a native Active Memory boundary so `queryMode: "message"` does not pass a full assembled request envelope into the hidden recall runner.

## Best Possible Solution

Make Active Memory message mode consume a true latest user message or a bounded equivalent before the recall prompt is built, with docs, diagnostics, and regression coverage matching the chosen contract.

## Reproduction Assessment

Yes. Source inspection shows `before_prompt_build` passes `event.prompt` into Active Memory as `latestUserMessage`, and message mode returns that text unchanged; I did not run the long Telegram session, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. The issue correctly identifies the broken docs/runtime contract, but maintainers still need to choose whether the best permanent fix is the linked config cap or a slimmer runtime intent boundary.
