## Summary

Keep open. Current main and the latest release still pass the full `before_prompt_build` prompt into Active Memory `message` mode, while the linked fix path is still open and needs maintainer product/API review.

## What This Changes

The issue asks OpenClaw to bound or slim Active Memory `queryMode: "message"` input so the hidden recall runner does not receive full assembled request envelopes.

## Best Possible Solution

Resolve one canonical Active Memory fix that derives or caps a current-user-message value before `buildQuery`, includes docs/logging, proves the large Telegram/direct envelope path, and then closes this same-root cluster after merge.

## Reproduction Assessment

Yes. Source inspection shows current main passes `event.prompt` into Active Memory and `message` mode returns it unchanged; the reporter also supplied v2026.6.5 logs with 500000-character `queryChars`, but this read-only pass did not run live Telegram.

## Solution Assessment

Unclear. A plugin-local cap or slim current-intent boundary is the right area, but maintainers still need to choose between the competing config-cap and extraction/hook-contract shapes before this is fully solved.
