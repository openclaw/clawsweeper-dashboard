## Label Justifications

- `P2`: Oversized hidden recall prompts can cause substantial latency, cost, context pressure, and timeouts, but the blast radius is limited to the optional Active Memory workflow.
- `impact:session-state`: The defect mis-associates a full assembled session/request envelope with the latest-user input to a hidden memory run.

## Summary

Keep open. Current main and v2026.6.5 still forward the full `before_prompt_build` prompt unchanged in Active Memory message mode, while the closing implementation at https://github.com/openclaw/openclaw/pull/92099 remains open and unmerged.

## What This Changes

Bound or slim the latest-user input sent to the Active Memory blocking recall sub-agent when `queryMode` is `message`.

## Best Possible Solution

Establish one canonical small-input contract for message mode, add diagnostics and regression coverage, and require redacted long-session proof that the hidden query is bounded without breaking useful recall.

## Reproduction Assessment

Yes at source level: current main directly forwards `event.prompt` unchanged in message mode, and the reporter provides correlated production query-size, token-usage, transcript-size, and timeout evidence. This read-only review did not run a fresh live Telegram reproduction.

## Solution Assessment

Unclear. Bounding the input is necessary, but maintainers must choose whether the durable solution is a fixed plugin invariant, an operator-configurable cap, or a generic slim-intent hook field before accepting the linked implementation.
