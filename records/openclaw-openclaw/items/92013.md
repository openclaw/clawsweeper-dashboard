## Summary

Keep open. Current main and v2026.6.6 still have the oversized Active Memory message-mode path, and the linked closing fix at https://github.com/openclaw/openclaw/pull/92099 is open, unmerged, and currently conflicting.

## What This Changes

Active Memory message mode is reported to send a full assembled request envelope to the hidden recall runner instead of a bounded latest user message.

## Best Possible Solution

Choose the durable boundary for Active Memory input, then land a focused fix that uses the actual latest user intent or a bounded fallback and proves the large-envelope case no longer reaches the recall runner.

## Reproduction Assessment

Yes from source: current main passes `event.prompt` into Active Memory message mode, and `buildQuery()` returns that value unchanged. The reporter also supplied v2026.6.5 logs showing 500K-character Active Memory input, but this read-only review did not run a live Telegram reproduction.

## Solution Assessment

Unclear as a final product choice. A latest-message cap is a plausible mitigation, but maintainers still need to choose whether the best contract is a new plugin config key, a slim current-user-intent hook field, or an equivalent bounded default with diagnostics.
