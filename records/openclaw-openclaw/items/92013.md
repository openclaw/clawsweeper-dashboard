## Label Justifications

- `impact:session-state`: The defect mis-associates the assembled session/request envelope with the latest user intent passed into memory recall.

## Summary

Keep open. Current main still passes the assembled `before_prompt_build` prompt into Active Memory as the latest user message, and message mode returns that input unchanged; the open closing PR has not merged, while an older issue and PR cover the same root cause but do not override the rule to keep this issue open until its linked fix lands or is redirected.

## What This Changes

Bound Active Memory’s message-mode recall input to the actual latest user intent instead of forwarding a potentially huge assembled request envelope.

## Best Possible Solution

Consolidate on one PR that derives a canonical latest user intent from structured hook data, keeps the existing bounded search-query path, and adds truncation only if live long-message evidence establishes a separate need; validate bounded hidden-runner input and preserved recall before landing.

## Reproduction Assessment

Yes, at source level with high confidence: current main passes `event.prompt` into message mode unchanged, matching the reporter’s correlated large `queryChars` and timeout evidence. A live current-main Telegram reproduction was not run during this read-only review.

## Solution Assessment

The root fix is to supply the real latest user intent, not merely cap an incorrectly assembled envelope. The optional cap requires a maintainer product decision, and the overlapping PRs should be consolidated rather than creating parallel policies.
