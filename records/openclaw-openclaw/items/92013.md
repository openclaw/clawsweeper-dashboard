## Summary

Keep open. Current main still passes the assembled `before_prompt_build` prompt into Active Memory message mode and `buildQuery()` returns that value unchanged, the latest release `v2026.6.6` has the same behavior, and the closing fix at https://github.com/openclaw/openclaw/pull/92099 is open, unmerged, and currently conflicting.

## What This Changes

Request a native Active Memory cap or slim current-user-intent boundary so `queryMode: "message"` cannot feed long assembled request envelopes into the hidden recall runner.

## Best Possible Solution

Review and resolve https://github.com/openclaw/openclaw/pull/92099, with maintainer agreement on the durable contract and real large-envelope proof, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes from source: current main forwards `event.prompt` into message mode and `buildQuery()` returns that value unchanged. The reporter also supplied `v2026.6.5` logs showing 500K-character Active Memory input, but this read-only review did not run a fresh live Telegram reproduction.

## Solution Assessment

Unclear as a final product choice. The linked PR is a plausible bounded fix, but maintainers still need to choose whether the durable contract should be a plugin `messageMaxChars` config option, a slim hook/runtime intent field, or both.
