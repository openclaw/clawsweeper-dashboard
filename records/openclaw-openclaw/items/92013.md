## Summary

Keep open. Current `main` and shipped v2026.6.5 both contradict the documented message-mode contract: the assembled prompt is forwarded unchanged as the latest user message, so the reported oversized hidden recall input remains possible.

**Likely related people**
- Ayaan Zaidi — high confidence; introduced the current Active Memory query and prompt-hook implementation in commit `7200c316e53172179a723603ca8b785eab1e7411`.

## What This Changes

Provide Active Memory message mode with bounded current-user intent instead of the complete assembled request envelope.

## Best Possible Solution

Add a generic optional prepared current-user-intent field to `before_prompt_build`, populate it before envelope assembly across runner paths, and make Active Memory message mode prefer it with a bounded fallback.

## Reproduction Assessment

Yes. Current-main source establishes that the assembled prompt is forwarded unchanged in message mode, and the reporter's persisted transcript and query-size measurements corroborate the resulting context pressure; this review did not run a fresh live Telegram reproduction.

## Solution Assessment

The slim prepared-intent field is the best architectural direction because it repairs the semantic boundary without adding another operator setting. The exact public Plugin SDK field and fallback semantics still need maintainer product/API judgment.
