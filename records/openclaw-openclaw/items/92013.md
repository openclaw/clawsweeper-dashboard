## Summary

Keep open. Current main and the latest release still pass the `before_prompt_build` prompt into Active Memory and `buildQuery()` returns it unchanged for `queryMode: "message"`, while the docs promise latest-message-only behavior; the closing fix PR is open but unmerged.

## What This Changes

The issue requests that bundled Active Memory stop feeding full assembled request envelopes into `queryMode: "message"` recall, either by capping the latest-message input or by passing a slim user-intent field.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/92099 after the config/default decision and real behavior proof, or replace it with a slim-intent implementation that preserves the documented small-message contract.

## Reproduction Assessment

Yes, source-reproducible: the hook passes `event.prompt` into message-mode query construction and `buildQuery()` returns it unchanged. I did not run a fresh Telegram session, but the reporter supplied redacted runtime logs showing 500K-character Active Memory input.

## Solution Assessment

Unclear as a final product choice. The requested direction is valid, but maintainers still need to choose whether the best fix is the open plugin-owned config cap or a broader runtime-provided slim-intent field.
