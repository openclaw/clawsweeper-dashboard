## Summary

Keep open. Current main and v2026.6.9 still feed `event.prompt` unchanged into Active Memory message-mode recall, and the open closing pull request at https://github.com/openclaw/openclaw/pull/92099 remains unmerged while maintainers choose the cap-versus-slim-intent contract.

## What This Changes

Active Memory message-mode recall should stop sending full assembled request envelopes into the hidden recall runner and use a bounded or slim latest-user intent instead.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/92099 with a maintainer-approved Active Memory input contract that bounds message-mode recall input and preserves useful diagnostics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure Active Memory with `queryMode: "message"` and pass a large assembled prompt through `before_prompt_build`; current main returns that prompt unchanged as the recall query. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Unclear whether the linked config-cap PR is the best final shape, because it expands plugin config surface while a runtime slim-intent field may better match the documented contract. The best solution needs maintainer choice plus proof on the long Telegram/direct session path.
