## Summary

Keep open. Current main and v2026.6.6 still send the full `before_prompt_build` prompt into Active Memory `message` mode, and the linked closing fix is open, conflicting, and still needs maintainer review rather than issue cleanup.

## What This Changes

Active Memory `queryMode: "message"` should bound or slim the latest-message input so assembled request envelopes are not forwarded into the hidden recall runner.

## Best Possible Solution

Review and resolve https://github.com/openclaw/openclaw/pull/92099, or replace it with a maintainer-approved slim-intent/cap fix, then close this issue with merge and release provenance.

## Reproduction Assessment

Yes, from source: current main passes `event.prompt` into Active Memory and `message` mode returns it unchanged. The reporter also supplied OpenClaw 2026.6.5 logs showing 500000-character Active Memory input, but this read-only pass did not run a live Telegram reproduction.

## Solution Assessment

Unclear as a final product choice. A cap is a plausible bounded fix, but maintainers still need to decide whether the durable contract should be a new `messageMaxChars` option or a slim current-user-intent hook field.
