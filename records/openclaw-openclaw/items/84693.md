## Summary

Keep this PR open for maintainer review: the patch has credible live proof and no blocking correctness finding from this read-only pass, but it changes hot session cache/snapshot behavior and still needs session/cache owner acceptance before merge.

## What This Changes

The PR interns repeated session skills prompts, adds immutable session-store snapshot/entry APIs through core and plugin runtime, caps serialized cache state, and migrates several read-only session lookups off mutable full-store clones.

## Best Possible Solution

Land the compatibility-preserving snapshot-read and capped-cache path after session/cache owner acceptance and completed CI; track any stricter loadSessionStore deprecation or guardrail plan separately.

No local high-confidence current-main reproduction was run in this read-only review. Source inspection shows the full-store clone and unbounded serialized-cache paths, and the PR body supplies credible live before/after heap evidence for the high-session workload.

## Solution Assessment

Yes, mostly: preserving loadSessionStore compatibility while adding immutable snapshot reads and bounding serialized cache state is the maintainable fix shape. Any stricter deprecation of mutable full-store reads should remain a separate maintainer API decision.
