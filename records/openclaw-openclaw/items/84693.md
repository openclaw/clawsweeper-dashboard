## Summary

Keep open for normal maintainer review: the patch looks broadly correct and has credible live heap proof, but it changes session-store cache behavior in agent/cron/plugin runtime paths and adds public SDK read APIs that should be documented before merge.

## What This Changes

The PR adds immutable session-store snapshot/entry read APIs, bounds serialized session-store cache retention, interns repeated large skills prompts, and migrates several read-only session lookups away from full mutable store clones.

## Best Possible Solution

Land the compatibility-preserving snapshot-read path after documenting the SDK helpers and maintainer acceptance of the high-load cache behavior; keep any stricter loadSessionStore deprecation plan separate.

## Reproduction Assessment

No local high-confidence current-main reproduction was run in this read-only review. Source inspection shows the full-store clone path, and the PR body supplies credible live before/after heap evidence for the high-session workload.

## Solution Assessment

Yes, mostly: preserving loadSessionStore compatibility while adding immutable snapshot reads and bounding serialized cache state is a maintainable fix shape. The new public SDK helpers should be documented, and deprecation or guardrail policy should remain a separate maintainer decision.
