## Summary

Keep open: this is a plausible focused fix for current main’s clone-heavy session store, with real high-load proof and targeted validation, but the new public SDK read surface needs documentation before merge and the session/cache risk deserves maintainer review.

## What This Changes

The PR interns repeated session prompt strings, adds immutable session-store snapshot/entry read APIs, bounds the serialized cache, and migrates several read-only session/delivery paths away from full mutable store clones.

## Best Possible Solution

Land the non-breaking snapshot-read path after documenting the new SDK helpers and having a session/cache owner accept the high-load cache behavior; keep mutable `loadSessionStore()` compatible while migrating hot read-only callers.

## Reproduction Assessment

No local high-confidence reproduction was established in this read-only review. The current-main clone path and the PR body’s live heap before/after data make the memory-retention scenario credible.

## Solution Assessment

Yes, mostly: preserving `loadSessionStore()` compatibility while adding immutable snapshot reads is the safest maintainable direction. The public SDK docs should be updated, and any stricter deprecation plan can be a separate maintainer decision.
