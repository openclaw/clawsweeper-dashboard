## Summary

Keep open: this is a credible active fix for high-load session-store memory retention, with strong live heap evidence and no blocking correctness finding, but it should get SDK documentation and session/cache owner review before merge.

## What This Changes

The PR adds immutable session-store snapshot/entry read APIs, bounds serialized session-store caching, interns repeated skills prompt strings, and migrates several agent/cron/delivery read paths away from mutable full-store clones.

## Best Possible Solution

Land the compatibility-preserving snapshot-read path after documenting the new SDK helpers and maintainer acceptance of the high-load cache behavior; keep any stricter loadSessionStore deprecation plan separate.

## Reproduction Assessment

No local high-confidence reproduction was established in this read-only review. Source inspection shows current main clones full session stores through loadSessionStore, and the PR body supplies credible live before/after heap evidence for the high-session workload.

## Solution Assessment

Yes, mostly: preserving loadSessionStore compatibility while adding immutable snapshot reads and bounding serialized cache state is a maintainable fix shape. The new public SDK helpers should be documented, and any stricter deprecation policy should be reviewed separately.
