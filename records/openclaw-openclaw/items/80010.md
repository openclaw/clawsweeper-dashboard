## Summary

Keep open: current main still has a source-level Discord `/voice` routing collision, and the linked implementation PR was closed unmerged, so there is no merged fix to close against.

## What This Changes

The issue reports that Discord `/voice status/list/chat` routes to status-like Talk voice output instead of a usable voice-selection flow or outbound audio.

## Best Possible Solution

Restore the documented split so Discord `/voice` reaches TTS, Discord `/talkvoice` manages Talk voice selection, and one-shot audio guidance points to `/voice audio <text>`.

## Reproduction Assessment

Yes, source-level: current main can route Discord native `/voice` through the Talk voice plugin before normal TTS handling. I did not run a live Discord session, so the observed UI response is not independently reproduced.

## Solution Assessment

Yes for the narrow fix: command matching should preserve the documented Discord `/voice` TTS and `/talkvoice` Talk voice split. Changing `/voice chat <text>` or adding `/vc chat` is not the best fix without a separate maintainer UX decision.
