## Summary

Keep open: current main still has a source-level Discord `/voice` command-routing problem, and an open linked PR explicitly targets this issue, so the issue should stay open until that implementation path is resolved.

## What This Changes

The issue reports that Discord `/voice list` does not present usable voices and `/voice chat <text>` does not send outbound audio on OpenClaw 2026.5.7.

## Best Possible Solution

Restore the documented split: Discord `/voice` should run TTS, Discord `/talkvoice` should manage Talk voice selection, and one-shot audio guidance should point to `/voice audio <text>` unless maintainers explicitly choose a new alias.

## Reproduction Assessment

Yes, source-level: current main shows Discord native `/voice` can be matched as the Talk voice plugin command before TTS handling. I did not run a live Discord session, so the exact UI output is not independently reproduced.

## Solution Assessment

Yes for the narrow fix: command matching should preserve the documented Discord `/voice` TTS and `/talkvoice` Talk voice split. Adding `/vc chat` or changing `/tts chat` semantics is not the best fix without separate maintainer UX approval.
