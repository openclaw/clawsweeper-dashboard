## Summary

Keep open: current main and latest release still show the Discord native `/voice` routing path that can let Talk voice plugin matching preempt the documented TTS command. The linked implementation attempt at https://github.com/openclaw/openclaw/pull/80070 is closed unmerged and did not land the narrow routing fix.

## What This Changes

The issue reports Discord `/voice list` not providing a usable voice-selection flow and `/voice chat <text>` not sending outbound audio.

## Best Possible Solution

Make Discord native dispatch resolve provider-native plugin commands without letting Talk voice's canonical `/voice` preempt TTS; keep `/talkvoice` for Talk voice and documented `/voice audio <text>` for one-shot TTS.

## Reproduction Assessment

Yes, source-level: current main and `v2026.5.12` still call plugin command matching for Discord native `/voice` without a Discord-native surface filter, while docs define `/voice` as Discord TTS and `/talkvoice` as Talk voice. I did not run a live Discord reproduction.

## Solution Assessment

Yes for the narrow repair: fix Discord native command matching so the documented command split is preserved. No for reviving the closed `/vc list/chat` expansion or silently adding a new `/voice chat <text>` audio alias without maintainer UX approval.
