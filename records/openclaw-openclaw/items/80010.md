## Summary

Keep open: current main still has a source-level Discord `/voice` routing collision where the native TTS alias can be handled as the Talk voice plugin command, and the only linked PR was closed unmerged after proposing a different `/vc` surface.

## What This Changes

The issue asks for Discord `/voice list` to provide a usable voice-selection flow and `/voice chat <text>` to send outbound audio instead of status-like output.

## Best Possible Solution

Restore the documented split so Discord `/voice` dispatches TTS, Discord `/talkvoice` manages Talk voice selection, and one-shot audio guidance remains `/voice audio <text>` unless maintainers approve a new alias.

## Reproduction Assessment

Yes, source-level: current main can route Discord native `/voice` through the Talk voice plugin before TTS because the dispatcher falls back when `matchPluginCommand('/voice')` succeeds. I did not run a live Discord session, so this is not a live UI reproduction.

## Solution Assessment

Yes for the narrow repair: command matching should preserve the documented Discord `/voice` TTS and `/talkvoice` Talk voice split. Adding `/vc chat` or changing `/tts chat` semantics is not the best fix without separate maintainer UX approval.
