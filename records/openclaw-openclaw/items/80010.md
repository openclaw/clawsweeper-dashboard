## Summary

Keep open: current main still has a real Discord native command routing problem. `/voice` is the documented Discord native name for TTS, but the native dispatcher can match the Talk voice plugin's canonical `/voice` command before it reaches the TTS command, while `/voice chat <text>` is not the documented one-shot audio action.

## What This Changes

The issue reports that Discord `/voice status/list/chat` does not provide a usable Talk voice-selection flow or one-shot outbound audio reply.

## Best Possible Solution

Restore the documented Discord command split: `/voice` should run TTS, `/talkvoice` should manage Talk voice selection, and help/errors should point one-shot audio users to `/voice audio <text>` without changing `/tts chat` semantics.

## Reproduction Assessment

Yes for the command-routing mismatch: current source shows Discord native `/voice` can be intercepted by the Talk voice plugin before TTS handling. No live Discord run is attached, so the exact UI text is not independently reproduced.

## Solution Assessment

Yes for the documented contract: a narrow routing/test fix should keep Discord `/voice` on TTS and `/talkvoice` on Talk voice. The safer alternative to adding `/voice chat <text>` is to preserve the existing `/tts chat` override and improve guidance toward `/voice audio <text>`.
