## Summary

Keep open: current main still has a source-level Discord native `/voice` routing collision where the Talk voice plugin can intercept Discord's documented TTS `/voice` command before the TTS path runs. The previously linked fix PR is closed unmerged and did not resolve the narrow documented command split.

## What This Changes

The issue requests a usable Discord `/voice` voice-selection flow and outbound audio for `/voice chat <text>` instead of status-like Talk voice output.

## Best Possible Solution

Restore the documented split so Discord native `/voice` reaches the TTS command path, Discord native `/talkvoice` manages Talk voice selection, and help or validation points one-shot audio users to `/voice audio <text>`.

## Reproduction Assessment

Yes, source-level: current main still allows Discord native `/voice ...` to be matched as the Talk voice plugin command before the TTS path. I did not run live Discord, so this is not a full end-to-end reproduction.

## Solution Assessment

Yes for the narrow fix: preserve the documented Discord `/voice` TTS and `/talkvoice` Talk voice split in native command matching. Adding `/vc chat`, `/vc list`, or new `/voice chat` semantics is broader than the reported routing regression.
