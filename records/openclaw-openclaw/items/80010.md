## What This Changes

The issue reports Discord native `/voice` returning Talk voice status/list behavior instead of usable TTS audio output.

## Best Possible Solution

Preserve the documented split: Discord native `/voice` should execute TTS, Discord native `/talkvoice` should manage Talk voice selection, and one-shot audio should remain `/voice audio <text>`.

## Reproduction Assessment

Yes, source-level: current main shows Discord native `/voice` can still match the Talk voice plugin command before built-in TTS handling. I did not run a live Discord session, so the exact UI output is not independently reproduced here.

## Solution Assessment

Yes for the narrow fix: native Discord dispatch should distinguish provider-native plugin names from canonical plugin command names so `/voice` remains TTS and `/talkvoice` remains Talk voice config. Adding new `/vc chat` or `/voice chat <text>` semantics would be a separate UX decision.
