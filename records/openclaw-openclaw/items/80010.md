## Summary

Keep open. Current main and v2026.5.7 expose overlapping voice command surfaces: Talk voice selection is `/voice` as a plugin/text command but `/talkvoice` natively on Discord, while Discord native `/voice` is TTS and uses `audio` for one-off synthesis. The report identifies a real Discord command-UX gap, but the expected `/voice chat <text>` behavior is not an already-defined bug contract.

## What This Changes

Requests Discord `/voice list` to provide usable voice selection and `/voice chat <text>` to send an outbound audio reply instead of returning status/help text.

## Best Possible Solution

Clarify and align the Discord voice/TTS command contract so `/voice` help distinguishes TTS audio, Talk voice selection, and `/vc` voice channels, then add any maintainer-approved aliases or selectable voice UI with tests and docs.

## Reproduction Assessment

Unclear for the full issue. Source inspection gives a high-confidence path for why `/voice chat <text>` does not produce audio, but the reported `/voice list` usability problem needs the missing screenshots or live Discord output to distinguish provider failure from command-surface confusion.
