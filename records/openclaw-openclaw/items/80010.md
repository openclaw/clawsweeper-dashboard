## Summary

Keep open: current main still shows a real Discord command-surface mismatch. Discord documents native `/voice` as the TTS command, but the native dispatch path can build `/voice ...` and match the Talk voice plugin before TTS handling; the literal `/voice chat <text>` expectation also conflicts with the documented `/tts chat` session override semantics.

## What This Changes

The issue asks for Discord `/voice list` to provide usable voice selection and `/voice chat <text>` to send outbound audio instead of status-like output.

## Best Possible Solution

Align Discord command routing with the documented split: native `/voice` should run TTS actions such as `/voice audio <text>`, native `/talkvoice` should run Talk voice list/set, and help/errors should make that distinction clear.

## Reproduction Assessment

Yes for the source-level command routing mismatch: current main can build a Discord native `/voice ...` prompt that matches the Talk voice plugin before TTS handling. No live Discord reproduction was established because the screenshots/logs are absent and no Discord credentialed run was used.

## Solution Assessment

Unclear for the literal requested behavior: making `/voice chat <text>` synthesize audio would conflict with the current TTS chat override contract. The narrower maintainable direction is to restore the documented `/voice` TTS and `/talkvoice` Talk split, then let maintainers decide whether to add any extra alias.
