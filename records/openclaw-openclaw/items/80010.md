## Summary

Keep open: current main still has the Discord native `/voice` routing problem, and the only linked implementation PR is closed unmerged. The narrow root cause is that Discord native dispatch checks plugin commands without Discord channel context, allowing the Talk voice plugin's canonical `/voice` command to preempt the documented Discord TTS `/voice` surface.

## What This Changes

The report asks for Discord `/voice list` to provide a usable voice-selection path and `/voice chat <text>` to produce outbound audio instead of Talk voice status-style output.

## Best Possible Solution

Keep this open and fix Discord native dispatch to use channel-aware plugin command matching so `/voice` stays on TTS while `/talkvoice` owns Talk voice config.

## Reproduction Assessment

Yes, source-level: current main registers Discord TTS as `/voice`, registers Talk voice as canonical `/voice` with Discord alias `/talkvoice`, and then matches plugin commands without Discord channel context. I did not run a live Discord session, so the visible Discord output is not independently observed.

## Solution Assessment

Yes for the narrow repair: pass Discord channel context into plugin matching at native command definition and dispatch. Adding `/vc chat/list` or treating `/voice chat <text>` as a new audio alias is not the best fix without separate maintainer UX approval.
