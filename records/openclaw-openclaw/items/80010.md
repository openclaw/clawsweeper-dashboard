## Summary

Keep open: current main and the latest release still let the Talk voice plugin's canonical `/voice` intercept Discord native `/voice` before TTS, and the only related PR is closed unmerged.

## What This Changes

Issue reports that Discord `/voice list` is not usable and `/voice chat <text>` does not produce outbound audio.

## Best Possible Solution

Make Discord native plugin-command matching use the Discord native alias surface so `/voice` dispatches TTS and `/talkvoice` dispatches Talk voice, with regression coverage for both paths.

## Reproduction Assessment

Yes, source-level: current main maps Discord TTS to `/voice`, registers Talk voice as `/voice` with Discord alias `/talkvoice`, then matches plugin commands without provider-native context. I did not run a live Discord session.

## Solution Assessment

Yes for the narrow repair: preserve the documented `/voice` TTS and `/talkvoice` Talk voice split. Expanding `/vc` or adding `/voice chat <text>` is not the best fix without separate UX approval.
