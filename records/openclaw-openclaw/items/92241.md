## Label Justifications

- `P1`: The report describes a connected gateway silently dropping inbound channel messages after update or rollback until an operator restarts it.
- `impact:message-loss`: The affected Signal inbound path can receive transport messages without surfacing a reply to the agent or user.
- `impact:crash-loop`: The stale process remains alive but runtime imports fail until restart, making this a gateway availability failure mode.

## Summary

Keep open: current main and v2026.6.6 still have unguarded reply-dispatch lazy imports that can hit removed hashed chunks after a package swap, and the open PR for this report remains unmerged with proof/design blockers.

## What This Changes

The issue asks OpenClaw to stop update or rollback package swaps from silently dropping inbound Signal messages by restarting, warning visibly, or refusing an unsafe update exit.

## Best Possible Solution

Use one lifecycle-owned stale-dist recovery and update policy that covers outer reply dispatch plus inner lazy loaders, emits supervisor-neutral diagnostics, and is proven with a live package-swap inbound-message test.

## Reproduction Assessment

Yes, source-reproducible with high confidence: npm global rollback without a gateway restart can remove hashed dist chunks while current reply dispatch still uses lazy imports from the old process graph. I did not run the live WSL2/systemd/Signal scenario in this read-only review.

## Solution Assessment

No merged solution is present. The open reply-local PR is a plausible diagnostic improvement, but the best fix needs maintainer choice on the process-lifecycle recovery boundary and live package-swap proof.
