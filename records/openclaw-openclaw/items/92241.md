## Summary

Current main still permits stale bundled imports when a direct global package replacement bypasses the managed updater, and the open linked PR remains the active implementation path but has unresolved lifecycle-coverage and real-proof blockers. Likely related people include myps6415 for gateway package-swap lifecycle work, martingarramon for prior reply-loader recovery work, and Shakker for recent reply-pipeline maintenance.

## What This Changes

Prevent stale bundled module paths after an in-place update or rollback from silently dropping inbound gateway messages.

## Best Possible Solution

Centralize narrowly scoped OpenClaw dist-rotation handling at a gateway lifecycle or outer reply-dispatch boundary, provide supervisor-neutral restart guidance or controlled process recovery, prevent repeated retries against permanently missing chunks, and prove the direct global rollback with an actual inbound message.

## Reproduction Assessment

Yes at high source confidence: replacing a global `dist/` tree under a running process leaves already-loaded code able to request old content-hashed chunks through later lazy imports. The exact WSL2 rollback and Signal-delivery sequence was not independently executed during this read-only review.

## Solution Assessment

No, the current linked loader-local patch is not yet the best solution because it covers only selected imports, retries an unrecoverable loader after clearing it, and lacks process-level recovery. A lifecycle-owned or outer-dispatch solution with live rollback proof is safer and more complete.
