## Summary

Keep open. Current main has meaningful update/restart hardening, but it does not fully settle the reported manual update/rollback stale-dist inbound-drop path, and multiple open PRs still directly claim to close this issue with different fix shapes.

## What This Changes

The issue reports that after an update or rollback, a still-running gateway can retain stale hashed dist import paths and silently drop inbound Signal messages until the gateway is restarted.

## Best Possible Solution

Land a maintainer-approved fix that either prevents live gateways from running against a replaced dist tree or makes stale-dist import failures operator-visible across the affected inbound lazy-loader/update paths, with live update or rollback proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The source path is plausible on current main because several inbound reply lazy imports still lack the visible stale-dist handling discussed in the issue, while the supported update.run path is separately hardened.

## Solution Assessment

Unclear whether any current open PR is the best fix. A warning-only change may reduce operator confusion, but the maintainable end state likely needs maintainer choice across update-owned restart/handoff, postinstall warning, and runtime-visible stale-import failure handling.
