## Summary

Keep open. Current main still recognizes base `openai-codex/gpt-5.2` as a modern OpenAI Codex model, but the provider-owned xhigh allowlist omits that exact model id, so the thinking profile still will not advertise `xhigh` for the reported provider/model pair. The one-off fix PRs referenced in the discussion are closed unmerged, and #37804 is a broader RFC rather than a replacement for this concrete missing-model case.

## What This Changes

Request to allow `xhigh` thinking for the OpenAI Codex provider's base `gpt-5.2` model id.

## Best Possible Solution

Make the OpenAI Codex provider policy explicit for base `gpt-5.2`: either add it to `OPENAI_CODEX_XHIGH_MODEL_IDS` with focused regression coverage for the exposed thinking levels and hint text, or document that only the codex-suffixed GPT-5.2 route is xhigh-capable. Keep #37804 as the broader follow-up for forward-compatible provider model capability drift.

## Reproduction Assessment

Yes. The source-level reproduction is deterministic: inspect or call the OpenAI Codex provider `resolveThinkingProfile`/`listThinkingLevels` path for `openai-codex` + `gpt-5.2`; the current allowlist cannot match that id, so `xhigh` is not exposed.

## Solution Assessment

Unclear until maintainers confirm the base OpenAI Codex GPT-5.2 capability. If confirmed, the narrow maintainable fix is provider-owned: add `gpt-5.2` to the OpenAI Codex xhigh model ids and cover it with focused tests; otherwise the better solution is explicit docs/config behavior saying it is high-only.
