## Summary

Keep open: current main and the latest release still omit base `openai-codex/gpt-5.2` from the OpenAI Codex xhigh thinking profile while recognizing that model as a modern Codex model. The linked one-off PRs were closed unmerged, and the broader RFC has closed without replacing this concrete provider-policy decision.

## What This Changes

Add base OpenAI Codex `gpt-5.2` to the provider-owned xhigh thinking profile so `openai-codex/gpt-5.2` accepts `thinking=xhigh`.

## Best Possible Solution

Decide the OpenAI Codex provider capability policy, then either add `gpt-5.2` to `OPENAI_CODEX_XHIGH_MODEL_IDS` with focused regression coverage or document base OpenAI Codex GPT-5.2 as high-only.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the reported gate: `openai-codex/gpt-5.2` routes through the OpenAI Codex thinking profile, and that profile's xhigh allowlist lacks `gpt-5.2`, so `listThinkingLevels("openai-codex", "gpt-5.2")` will not expose `xhigh`. I did not run live provider validation in this read-only review.

## Solution Assessment

Unclear. The narrow code change is an allowlist entry plus a regression test if maintainers confirm upstream xhigh support; otherwise the maintainable answer is to leave the policy as-is and document the high-only behavior.
