## Summary

Close as cannot reproduce: current main no longer has a supported OpenAI Codex `gpt-5.2` path where `xhigh` should be enabled. The provider now suppresses stale Codex OAuth `gpt-5.1*`, `gpt-5.2*`, and `gpt-5.3*` refs and directs users to current Codex routes instead.

## What This Changes

Request to add `openai-codex/gpt-5.2` to the OpenAI Codex `xhigh` thinking allowlist.

## Best Possible Solution

Keep stale Codex OAuth `gpt-5.2` refs suppressed and route users to `openai-codex/gpt-5.5` or `openai/gpt-5.5` with the native Codex runtime unless maintainers intentionally re-enable the retired model family.

## Reproduction Assessment

No. On current main, the requested user path is obsolete: `openai-codex/gpt-5.2` is suppressed as a stale Codex OAuth model before it should be treated as an xhigh-capable supported model.

## Solution Assessment

Yes. The current policy is narrower and more maintainable than adding another allowlist entry: reject the stale model with a replacement route rather than advertise `xhigh` for a model the provider docs say not to configure.
