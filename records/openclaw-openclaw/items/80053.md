## Summary

Keep open: the reported native subagent `cwd` bug is real and the PR targets the right path, but the current patch introduces a workspace mismatch for inline attachments and still lacks after-fix real behavior proof.

## What This Changes

The PR threads `sessions_spawn.cwd` into native `spawnSubagentDirect`, gives explicit cwd precedence over target-agent workspace inheritance, adds a resolver unit test, and adds a changelog entry.

## Best Possible Solution

Resolve the final child workspace once, validate/canonicalize explicit native cwd against the intended requester/target workspace policy, reuse that workspace for attachments and spawned metadata, keep Gateway `agent` params schema-clean, and add focused plus real-behavior proof.

## Reproduction Assessment

Yes, source-reproducible: current main reads `cwd`, forwards it only to ACP, and native cross-agent spawns still resolve to the target agent workspace instead of the explicit cwd. I did not run the macOS/Telegram canary in this read-only review.

## Solution Assessment

No. Forwarding cwd is the right direction, but the patch should compute and validate a single final child workspace before attachment materialization and keep the Gateway payload boundary intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reuse the resolved cwd for attachments:** `src/agents/subagent-spawn.ts:1059`
  - body: This lets the child run from `params.cwd`, but attachments are still materialized before that workspace is resolved and `materializeSubagentAttachments` writes them under the target agent workspace. A cross-agent `sessions_spawn({ runtime: "subagent", cwd, attachments: [...] })` will tell the child the files are at a relative path in its cwd even though they were written elsewhere. Compute the final child workspace first and use that same value for attachment materialization and spawned metadata.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch changes native spawned-run workspace selection and needs bounded cwd validation before merge.

Concerns:

- **[medium] Constrain native cwd overrides:** `src/agents/subagent-spawn.ts:1059`
  - body: The PR stores a tool-call `cwd` as the native spawned workspace, and the Gateway then uses stored spawned workspace metadata as the child runtime workspace. Prior work kept spawned workspace overrides internal; this change should validate or canonicalize the requested cwd against the approved requester/target workspace boundary before trusting arbitrary tool input.
  - confidence: 0.78
