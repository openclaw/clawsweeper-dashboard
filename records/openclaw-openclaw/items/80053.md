## Summary

Keep open: the PR targets a real source-reproducible native subagent `cwd` bug, but the proposed fix turns a model-supplied tool argument into the child workspace without validating the workspace boundary, and it still lacks after-fix real behavior proof.

## What This Changes

This PR threads `sessions_spawn.cwd` into native subagent spawning, adds a workspace-inheritance unit test, and adds a changelog fix entry.

## Best Possible Solution

Implement native subagent `cwd` as a bounded workspace override: canonicalize and allow only approved requester/target workspace roots, keep `spawnedWorkspaceDir` internal-only, add tool-level and spawn-level regression coverage, then require a real same-workspace canary proof.

## Reproduction Assessment

Yes, source-level: current main reads `cwd` in `sessions_spawn`, sends it to ACP only, and native cross-agent `spawnSubagentDirect` resolves to the target workspace. I did not run a live Telegram/macOS canary in this read-only review.

## Solution Assessment

No. The PR fixes the missing parameter threading, but it is not the best mergeable form because it makes model-supplied `cwd` an unvalidated native child workspace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate native cwd before making it the child workspace:** `src/agents/subagent-spawn.ts:1069-1070`
  - body: This makes a model-provided tool argument the `spawnedWorkspaceDir` without checking that it is an allowed requester or target workspace. That value is persisted through `sessions.patch` and later used as the runtime workspace, so a caller could point a native subagent at an arbitrary host path instead of the intended workspace boundary. Please canonicalize and constrain the path, or reject it, before passing it to `resolveSpawnedWorkspaceInheritance`.
  - confidence: 0.88

## Security Review

Status: needs_attention
