## Summary

Keep open: current main and the latest release still use the old Matrix cross-signing UIA fallback that ignores MAS reset stages, UIA sessions, and reset URLs. The same reporter's open implementation candidate at https://github.com/openclaw/openclaw/pull/74509 is the right path, but it has follow-up review work before this issue should close.

## What This Changes

The issue asks the Matrix bootstrap flow to handle MAS-backed cross-signing UIA by preserving session data, surfacing reset URLs, and keeping MSC3967 first-publish working for PAT-authenticated bots.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/74509 with a Matrix plugin-owned fix that parses UIA challenges, preserves UIA sessions, surfaces MAS reset URLs, keeps the MSC3967 upload path available, and updates focused tests plus operator docs.

## Reproduction Assessment

Yes, at source level: current main's `authUploadDeviceSigningKeys` callback can be exercised with no configured password while null and dummy requests reject. That path throws the reported password fallback error and never inspects the MAS UIA body, reset URL, or session.

## Solution Assessment

Yes in direction: the stage-aware Matrix plugin fix described by the issue is the narrow maintainable path. The issue should stay open until the active PR lands or is superseded, because current main still has the old behavior.
