## Summary

Keep open. Current main still has the reported Matrix cross-signing UIA fallback path, the docs and tests still describe/assert that old behavior, and the same reporter has an open paired implementation candidate in #74509. This should stay open until that PR lands, is replaced, or maintainers choose a different Matrix plugin fix.

## What This Changes

The issue reports that Matrix cross-signing bootstrap fails on MAS-fronted Synapse for PAT-authenticated bot accounts and asks OpenClaw to handle MSC3967 first-publish plus MAS cross-signing reset UIA cleanly.

## Best Possible Solution

Land or supersede #74509 with a Matrix plugin-owned fix that parses UIA 401 bodies, preserves UIA sessions, reports a typed MAS reset URL for operator approval, keeps MSC3967 first-publish uploads available, and updates focused tests plus Matrix operator docs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to invoke the current authUploadDeviceSigningKeys callback with no configured password while no-auth and dummy requests reject; current tests assert the same misleading fallback error, and the issue supplies the concrete MAS 401 body shape.

## Solution Assessment

Yes at the design level, but it is not implemented on main. Handling this inside the Matrix plugin cross-signing bootstrap path plus a homeserver capability probe is the narrow maintainable direction; #74509 is the active candidate to review, land, or replace.
