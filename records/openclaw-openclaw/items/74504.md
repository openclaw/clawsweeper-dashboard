## Summary

Keep open. Current `main` still follows the reported Matrix cross-signing UIA fallback path, docs and tests still describe the old no-auth/dummy/password behavior, and the same author has an open implementation PR (#74509) for this exact MAS cross-signing path, so this issue should remain open until that PR lands or is superseded.

## What This Changes

The issue asks the Matrix plugin to handle MAS-fronted Synapse cross-signing bootstrap by preserving MSC3967 first-publish behavior, parsing `org.matrix.cross_signing_reset` UIA challenges, surfacing the MAS reset URL, and avoiding the misleading password-only recovery message.

## Best Possible Solution

Land or supersede #74509 with a Matrix plugin-owned fix that parses Matrix UIA 401 responses, threads the UIA session through supported stages, reports a typed MAS reset URL for operator approval, keeps MSC3967 first-publish uploads available, and updates focused crypto-bootstrap tests plus Matrix operator docs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to invoke the current `authUploadDeviceSigningKeys` callback with no configured password while no-auth and dummy requests reject; current tests assert the same misleading password fallback, and the issue supplies the concrete MAS 401 response shape.

## Solution Assessment

Yes at the design level, but it is not implemented on `main`. Handling this in the Matrix plugin cross-signing UIA callback plus a homeserver-capability probe is the narrowest maintainable direction; #74509 is the active candidate to review, land, or replace.
