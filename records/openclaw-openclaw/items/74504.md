## Summary

Keep open: current main and the latest release still use the old Matrix cross-signing UIA fallback, and the paired implementation PR remains open with review follow-up. This is a valid Matrix E2EE bug, but closing should wait until the linked fix lands or is superseded.

## What This Changes

The issue reports that Matrix cross-signing bootstrap fails on MAS-fronted Synapse when a PAT-authenticated bot needs MSC3967 first-publish or MAS reset UIA handling.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/74509 with a Matrix plugin-owned fix that parses UIA challenges, preserves sessions, supports MAS reset URLs and stable m.oauth, keeps MSC3967 first-publish working, and updates focused tests plus operator docs.

## Reproduction Assessment

Yes, at source level: current main's authUploadDeviceSigningKeys callback can be exercised with no password while no-auth and dummy requests reject, causing the reported password fallback error without reading the MAS UIA body, reset URL, or session. I did not repeat a live MAS homeserver run in this read-only review.

## Solution Assessment

Yes in direction: the stage-aware Matrix plugin fix in the paired PR is the narrow maintainable path. The issue should remain open until that PR lands or is replaced, because current main and v2026.5.12 still have the old behavior.
