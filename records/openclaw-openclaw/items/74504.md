## Summary

Keep open. Current main still follows the reported Matrix cross-signing UIA fallback path, does not parse MAS `org.matrix.cross_signing_reset` challenges or UIA sessions, and still has docs/tests for the old no-auth/dummy/password behavior. The issue is also paired with open implementation PR #74509, so it should stay open until that PR lands or is superseded.

## What This Changes

The issue requests a Matrix plugin fix for MAS-fronted Synapse cross-signing bootstrap so OpenClaw preserves MSC3967 first-publish behavior, parses UIA 401 challenges, surfaces MAS reset URLs, and stops suggesting password UIA when only the MAS reset stage is advertised.

## Best Possible Solution

Land or supersede #74509 with a Matrix plugin-owned fix that parses Matrix UIA 401 responses, threads the UIA session through supported stages, reports a typed MAS reset URL for operator approval, keeps MSC3967 first-publish uploads available, and updates focused crypto-bootstrap tests plus Matrix operator docs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to invoke the current `authUploadDeviceSigningKeys` callback with no configured password while no-auth and dummy requests reject; current tests assert the same misleading password fallback, and the issue supplies the concrete MAS 401 response shape.

## Solution Assessment

Yes at the design level, but it is not implemented on main. Handling this in the Matrix plugin's cross-signing UIA callback plus a homeserver-capability probe is the narrowest maintainable path; #74509 is the active candidate to review, land, or replace.
