## Summary

Keep this PR open: the reported reset bug is source-reproducible and the patch goes in the right direction, but the current subagent predicate would still preserve stale CLI bindings for ordinary dashboard child sessions and the external real behavior proof is not inspectable.

## What This Changes

The PR imports `clearAllCliSessions`, clears CLI resume bindings during gateway `sessions.reset` for entries it classifies as non-subagents, and adds reset hook tests for main-session clearing plus spawned-subagent preservation.

## Best Possible Solution

Clear CLI resume bindings for user-initiated resets on non-orchestration sessions, preserve them only for actual spawned subagent flows, cover main/dashboard/subagent reset cases, and attach redacted real claude-cli after-fix proof.

## Reproduction Assessment

Yes at source level: current main copies CLI binding fields during gateway reset, and the CLI runner later uses those fields to choose resume args. I did not execute a live claude-cli reset in this read-only review.

## Solution Assessment

No as written: clearing bindings on reset is the right direction, but `parentSessionKey` alone is not a safe spawned-subagent predicate. The narrower fix is to preserve only actual spawned/orchestration sessions and add dashboard-child coverage.
