## Summary

Keep open: source inspection confirms the current reset bug and the PR targets the right gateway state path, but the branch still misclassifies ordinary parent-linked dashboard sessions as spawned subagents and lacks inspectable real behavior proof.

## What This Changes

The PR clears persisted CLI session bindings during gateway resets for non-spawned sessions and adds reset hook tests for main-session clearing and spawned-subagent preservation.

## Best Possible Solution

Clear CLI resume bindings for user-initiated resets on every non-orchestration session, preserve bindings only for actual spawned subagent reset flows, cover main/dashboard/subagent cases, and attach redacted real claude-cli after-fix proof.

## Reproduction Assessment

Yes at source level: current main copies CLI binding fields during gateway reset, and the CLI runner later uses those fields to choose resume args. I did not execute a live claude-cli reset because this review is read-only.

## Solution Assessment

No as written: clearing bindings on reset is the right direction, but `parentSessionKey` alone is not a safe spawned-subagent predicate. The narrower fix is to preserve only actual spawned/orchestration sessions and add dashboard-child coverage.
