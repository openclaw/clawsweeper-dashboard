## Summary

Keep open: this is a focused bugfix and current main still accepts inverted explicit usage ranges; I found no blocking patch defect, but the stricter gateway RPC behavior needs maintainer acceptance before merge.

## What This Changes

The PR adds handler-level validation and tests so `usage.cost` and `sessions.usage` reject explicit `startDate`/`endDate` pairs where `startDate` is after `endDate`.

## Best Possible Solution

Land the narrow handler-entry validation once maintainers accept the RPC compatibility change; otherwise require an explicit compatibility-preserving design before merge.

## Reproduction Assessment

Yes. Current main parses both explicit dates and returns the range without checking order, so an inverted pair reaches the usage loaders as a backwards window; the PR body also includes terminal negative-control and after-fix proof against the real handlers.

## Solution Assessment

Yes. Validating at the two gateway handler entries is the narrowest maintainable fix because those handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would either broaden its contract or hide RPC error decisions in a helper.
