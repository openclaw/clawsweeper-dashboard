## Summary

No blocking code findings found. This PR should stay open for maintainer review because it is a focused fix with sufficient proof, but it deliberately changes accepted gateway RPC behavior from successful empty/backwards responses to `INVALID_REQUEST`.

## What This Changes

The PR adds handler-level validation and tests so `usage.cost` and `sessions.usage` reject explicit `startDate` values after `endDate` before reaching cost or session loaders.

## Best Possible Solution

Land this PR as-is if maintainers accept fail-closed validation for inverted explicit usage ranges; otherwise ask for a compatibility-preserving policy before changing the RPC contract.

## Reproduction Assessment

Yes, source-reproducible. Current main parses two explicit valid dates and forwards the resulting window without checking order, and the PR body includes terminal proof with a parent negative control showing the backwards window reaching loaders.

## Solution Assessment

Yes. Handler-entry validation is the narrowest maintainable fix because those handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would broaden a shared helper contract used by tests and internal callers.
