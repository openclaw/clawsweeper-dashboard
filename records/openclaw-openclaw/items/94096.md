## Summary

Keep open: the patch appears focused and correct with sufficient real handler proof, but it intentionally changes two gateway RPCs from successful reversed date windows to fail-closed `INVALID_REQUEST`, so maintainers should accept that compatibility change before merge.

## What This Changes

The branch adds a handler-level inverted explicit `startDate`/`endDate` check for `usage.cost` and `sessions.usage` plus focused gateway usage tests.

## Best Possible Solution

Land the focused handler validation after maintainers accept that reversed explicit usage date ranges should be invalid gateway RPC requests.

## Reproduction Assessment

Yes. Current main source computes explicit start/end date milliseconds and passes that range to cost/session loaders without an order check; the PR body also includes pre-patch backwards-window output.

## Solution Assessment

Yes, with maintainer compatibility acceptance. The handler-boundary guard covers both affected RPCs before I/O and avoids changing the public schema or duplicating checks in lower-level loaders.
