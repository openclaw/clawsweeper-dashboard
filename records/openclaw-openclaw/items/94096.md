## Summary

Keep open for maintainer review: the bug and fix are well supported, but the PR intentionally changes two gateway RPCs from successful empty/backwards-window responses to `INVALID_REQUEST` for reversed explicit date ranges.

## What This Changes

The branch adds a shared date-range helper, handler guards, and tests so `usage.cost` and `sessions.usage` reject valid explicit dates when `startDate` is after `endDate`.

## Best Possible Solution

Land this guard and its focused tests after explicit maintainer acceptance of the stricter gateway RPC contract; otherwise ask for a compatibility-preserving design.

## Reproduction Assessment

Yes. Source inspection shows current main builds an inverted `parseDateRange` window for valid reversed explicit dates and passes it into both usage loaders; the PR body also includes a parent negative control and patched terminal proof.

## Solution Assessment

Yes, pending maintainer acceptance of the compatibility change. Handler-level validation is the narrowest maintainable shape because these handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would require a broader parser return contract.
