## Summary

Keep this PR open for maintainer review: current main still accepts inverted explicit usage date ranges, and the patch is a narrow, well-covered fix, but landing it intentionally changes the gateway RPC compatibility contract from a successful empty/backwards response to `INVALID_REQUEST`.

## What This Changes

The PR adds inverted explicit `startDate`/`endDate` validation to `usage.cost` and `sessions.usage`, plus helper and handler-level regression tests.

## Best Possible Solution

Land the narrow handler-level validation once maintainers accept the stricter RPC contract and the proof gate is aligned with the already-supplied terminal evidence.

## Reproduction Assessment

Yes. Source inspection on current main shows `parseDateRange` does not compare explicit `startDate` and `endDate`, so both handlers can pass a backwards window to their loaders; the PR body also includes a negative-control terminal proof for that behavior.

## Solution Assessment

Yes, subject to maintainer acceptance of the stricter contract. The handler-entry guard is the narrowest maintainable fix because the handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would widen its internal contract or hide the RPC error decision.
