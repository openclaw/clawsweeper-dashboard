## Summary

Keep this PR open: the patch appears correct and well-proven, but it intentionally changes accepted gateway RPC behavior and still needs maintainer acceptance of that fail-closed compatibility choice.

## What This Changes

The PR adds validation and tests so `usage.cost` and `sessions.usage` reject explicit `startDate` values after `endDate` with `INVALID_REQUEST` before loading usage data.

## Best Possible Solution

Land the handler-level validation if maintainers agree inverted explicit usage ranges are invalid requests; otherwise keep current behavior until a compatibility-preserving policy is chosen.

## Reproduction Assessment

Yes. Current main's `parseDateRange` can return a backwards explicit window when `startDate` parses after `endDate`, and the PR body includes terminal negative-control proof showing that window reaches the loaders.

## Solution Assessment

Yes. Handler-entry validation is the narrowest maintainable fix because those handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would broaden a shared helper's contract.
