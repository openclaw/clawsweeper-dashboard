- `merge-risk: 🚨 compatibility`: Existing clients that send reversed explicit usage dates currently receive a successful response, while this PR returns `INVALID_REQUEST`.

## Summary

Current main does not have the inverted-range guard; the PR is focused and source/proof support the bug fix, but it intentionally changes a shipped gateway RPC response from success to `INVALID_REQUEST`, so it should stay open for maintainer acceptance rather than cleanup-close.

## What This Changes

The PR adds `hasInvertedExplicitDateRange` and uses it in `usage.cost` and `sessions.usage` so valid-but-reversed explicit date ranges return `INVALID_REQUEST` before usage loaders run.

## Best Possible Solution

Land the focused handler-boundary validation if maintainers accept reversed explicit usage date ranges as invalid requests; otherwise pause for an explicit compatibility or deprecation path.

## Reproduction Assessment

Yes. Current main parses both explicit valid dates, does not check ordering, and passes the resulting reversed `startMs`/`endMs` into the usage loaders; the PR body also includes real-handler before/after terminal proof.

## Solution Assessment

Yes, with maintainer compatibility acceptance. The handler-boundary guard is narrower than changing the public schema and avoids duplicating checks in lower-level cost/session loaders while covering both affected RPCs before I/O.
