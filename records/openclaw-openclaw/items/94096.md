## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main parses both explicit dates and returns the range without checking order, so an inverted pair reaches the usage loaders as a backwards window. I did not run a local repro because this review was read-only, but the PR body includes after-fix and negative-control terminal proof against the real handlers.

## Solution Assessment

Yes, this is the best narrow fix shape: validate at the two gateway handler entries where `INVALID_REQUEST` responses are already produced, while preserving `parseDateRange`'s internal return contract. Moving ordering into `parseDateRange` would either need a broader result type or hide the error-response decision from the RPC boundary.
