## What This Changes

The PR centralizes usage date-range validation in `resolveDateRange` and updates tests so `usage.cost` and `sessions.usage` return `INVALID_REQUEST` for explicit `startDate` values after `endDate` before loaders run.

## Best Possible Solution

Land the centralized `resolveDateRange` validation after explicit maintainer acceptance of the stricter gateway RPC contract; otherwise ask for a compatibility-preserving design.

## Reproduction Assessment

Yes. Source inspection shows current main builds an inverted `parseDateRange` window for valid reversed explicit dates and passes it into both usage loaders; the PR body also includes a parent negative control and patched terminal proof.

## Solution Assessment

Yes, pending maintainer compatibility acceptance. Centralizing the invalid-date and inverted-range checks in `resolveDateRange` is cleaner than duplicating handler guards, and it preserves forward ranges, defaults, and existing date-interpretation behavior.
