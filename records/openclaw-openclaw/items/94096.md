## Summary

Keep this PR open: current main still accepts inverted explicit usage date ranges, and the patch is a focused gateway RPC validation fix with adequate proof. The remaining blocker is maintainer acceptance of the stricter fail-closed RPC compatibility behavior.

## What This Changes

The PR adds handler-level validation and tests so `usage.cost` and `sessions.usage` reject explicit `startDate`/`endDate` ranges where `startDate` is after `endDate`.

## Best Possible Solution

Land this focused validation if maintainers accept inverted explicit usage ranges as invalid requests; otherwise require a compatibility-preserving or opt-in strict path.

## Reproduction Assessment

Yes. Source inspection shows current main builds a date range for two explicit valid dates without checking order, and the PR body includes terminal negative-control proof showing backwards windows reaching the loaders before the patch.

## Solution Assessment

Yes, subject to maintainer acceptance of the compatibility change. Handler-entry validation is the narrowest maintainable fix because those handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would broaden a shared helper contract.
