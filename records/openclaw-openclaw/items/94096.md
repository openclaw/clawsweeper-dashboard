## Summary

Keep open: the patch addresses a real gateway usage validation gap with sufficient terminal proof and no blocking code defect, but it intentionally changes reversed explicit usage ranges from successful RPC responses to `INVALID_REQUEST`, so maintainer acceptance of that compatibility change is still needed.

## What This Changes

The PR adds handler-level validation and tests so `usage.cost` and `sessions.usage` reject explicit valid dates when `startDate` is after `endDate`.

## Best Possible Solution

Land the guard and tests if maintainers accept stricter gateway RPC semantics; otherwise preserve the current success contract with a documented compatibility design.

## Reproduction Assessment

Yes. Source inspection shows current main builds an inverted `parseDateRange` window for valid reversed explicit dates and passes it into both usage loaders; the PR body also includes a parent negative control and patched terminal proof.

## Solution Assessment

Yes, pending maintainer acceptance of the compatibility change. Handler-level validation is the narrow fix because these handlers already own `INVALID_REQUEST` responses, while changing `parseDateRange` would require a broader parser return contract.
