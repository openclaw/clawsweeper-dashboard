## Summary

Keep open for maintainer acceptance. The patch is focused and appears correct, and the PR body now has sufficient real handler proof, but merging intentionally changes the gateway RPC contract from accepting reversed explicit usage dates to returning `INVALID_REQUEST`.

## What This Changes

The branch adds `hasInvertedExplicitDateRange`, calls it from `usage.cost` and `sessions.usage`, and adds focused gateway usage tests.

## Best Possible Solution

Accept the stricter usage RPC contract and land this focused handler-boundary validation, unless maintainers explicitly want a compatibility or deprecation path for reversed date windows.

## Reproduction Assessment

Yes. Current source shows `parseDateRange` can produce `startMs > endMs` for valid reversed dates, and both affected handlers pass that computed range to their loaders after only unparseable-date validation.

## Solution Assessment

Yes, with a maintainer compatibility decision. The handler-boundary guard is narrower than changing the public schema and avoids duplicating the check in lower-level cost/session loaders while covering both affected RPCs before I/O.
