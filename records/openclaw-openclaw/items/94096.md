## Summary

## What This Changes

The PR adds gateway usage validation and focused tests so `usage.cost` and `sessions.usage` reject explicit `startDate` values after `endDate`.

## Best Possible Solution

Land the handler-level guard with the focused tests if maintainers accept fail-closed semantics for inverted explicit usage ranges; otherwise ask for a compatibility-preserving design.

## Reproduction Assessment

Yes from source inspection: current main returns an inverted `parseDateRange` window when both explicit dates are valid and reversed, then both affected handlers pass that window into usage/session loading. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes, pending maintainer acceptance of the compatibility change. Handler-level validation is the narrow owner-boundary fix because these handlers already own `INVALID_REQUEST` responses and the shared parser contract can stay unchanged.
