## What This Changes

The PR adds gateway usage validation so explicit `startDate` after `endDate` is rejected for `usage.cost` and `sessions.usage`.

## Best Possible Solution

Land the narrow handler-level guard if maintainers accept the stricter gateway RPC contract for inverted explicit usage ranges.

## Reproduction Assessment

Yes. Current main’s `parseDateRange` returns a backwards explicit range without checking order, and both handlers pass that range into usage/session loaders.

## Solution Assessment

Yes. The validation sits at the two gateway handler entries that own `INVALID_REQUEST` responses and leaves the shared parser’s existing range-return contract unchanged; changing the parser itself would broaden the helper contract unnecessarily.
