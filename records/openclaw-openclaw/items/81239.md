## What This Changes

The PR changes `/models` provider visibility and list construction so CLI runtime aliases appear as picker providers, adds Claude CLI allowlist seeding, tests, and a changelog entry.

## Best Possible Solution

Restore CLI runtime picker entries through the model-catalog and visibility contracts, keeping provider-specific allowlists plugin-owned and preserving auth/config gating.

## Reproduction Assessment

Yes. Source inspection shows current main filters CLI runtime aliases out of `/models`, and the inspected Telegram screenshots show the intended after-fix picker path on the PR.

## Solution Assessment

No. The visibility predicate change is plausible, but the raw catalog and Claude allowlist additions must preserve auth/config visibility and avoid importing Anthropic plugin data into core.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve auth visibility for CLI picker entries:** `src/auto-reply/reply/commands-models.ts:190-211`
  - body: Both new CLI population paths run after `visibleCatalog` and call `add()` or `addRawModelRef()` directly. `add()` does not call `createProviderAuthChecker`, so unauthenticated `claude-cli`, `codex-cli`, or `google-gemini-cli` rows can appear in the default `/models` picker when the catalog or hardcoded allowlist contains them. Gate these additions through the same auth/config visibility decision before adding them.
  - confidence: 0.91
- **[P2] Move Claude CLI allowlist seeding out of core:** `src/auto-reply/reply/commands-models.ts:1`
  - body: `commands-models.ts` is shared core auto-reply code, but this import pulls Anthropic plugin-specific allowlist data from `extensions/anthropic` and bakes one bundled provider's model policy into the generic picker. Put the Claude CLI rows behind plugin-owned catalog metadata or a generic provider catalog seam, then let the picker consume the catalog normally.
  - confidence: 0.86
