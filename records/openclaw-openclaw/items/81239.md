## Summary

Keep open: the CLI picker regression is real and the Telegram proof is sufficient, but the PR still bypasses the provider auth/config visibility gate for added CLI rows and pulls Anthropic plugin-owned catalog data directly into core.

## What This Changes

The PR changes /models provider filtering and model-list construction so CLI runtime providers appear as picker entries, adds Claude CLI allowlist seeding, tests, and a changelog entry.

## Best Possible Solution

Restore CLI runtime discoverability through the existing visible-catalog/auth contract and a plugin-owned catalog or manifest seam, while keeping the bare codex compatibility alias hidden.

## Reproduction Assessment

Yes. Current main source shows CLI runtime aliases are filtered out of /models, and the inspected Telegram screenshots show the intended after-fix claude-cli picker path, though I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. Narrowing the legacy predicate is plausible, but the added raw catalog and Claude allowlist paths must preserve auth/config visibility and use a plugin-owned catalog surface rather than hardcoding Anthropic internals in core.
