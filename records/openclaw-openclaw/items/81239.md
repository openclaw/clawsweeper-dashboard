## Summary

Keep open: this is a plausible fix for the linked regression, but latest head currently fails model-picker CI because the global legacy predicate exposes CLI alias refs outside `/models`, and the attached Telegram proof predates the final catalog seam.

## What This Changes

The PR changes runtime-alias visibility, adds CLI runtime provider catalog/auth handling for `/models`, contributes Claude CLI catalog rows from the Anthropic plugin, updates command tests, and adds a changelog entry.

## Best Possible Solution

Keep the provider-owned catalog/auth fix, but scope CLI alias provider visibility to the channel `/models` picker and preserve legacy hiding/fallback behavior in setup and config pickers; then refresh latest-head Telegram proof.

## Reproduction Assessment

Yes. Current main is source-reproducible: CLI runtime aliases are classified as legacy and filtered before `/models` provider data is returned; PR CI also reproduces a new model-picker regression at latest head.

## Solution Assessment

No for the current patch shape. The provider-owned catalog/auth direction is reasonable, but changing the global legacy predicate is too broad; the safer fix is a `/models`-specific visibility path that does not alter setup/default/allowlist picker semantics.
