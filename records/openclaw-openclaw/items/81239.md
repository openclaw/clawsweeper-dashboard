## Summary

Keep open: the regression is real and the latest head fixes the prior unauthenticated-provider visibility blocker, but the PR still hard-codes an Anthropic plugin allowlist into core `/models` instead of routing it through a plugin catalog/public seam.

## What This Changes

The PR narrows legacy runtime provider detection, adds CLI runtime catalog and Claude CLI allowlist population for `/models`, updates command tests, and adds a changelog entry.

## Best Possible Solution

Keep the predicate and auth-gate fix, but move provider-specific CLI model seeding into the Anthropic plugin catalog or another public plugin seam so `/models` consumes generic catalog entries.

## Reproduction Assessment

Yes, by source inspection: current main classifies all CLI runtime aliases as legacy and the picker hides legacy providers. The contributor's Telegram proof also demonstrates the affected click-through path after applying the PR.

## Solution Assessment

No. The visibility/auth direction is right, but importing an Anthropic plugin allowlist directly into core is not the narrowest maintainable solution; the allowlist should enter through plugin-owned catalog metadata or a public seam.
