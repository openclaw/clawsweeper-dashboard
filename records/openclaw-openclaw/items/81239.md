## Summary

Keep open: the narrow legacy-alias predicate change is plausible and the author added inspectable Telegram proof, but the follow-up picker-data changes bypass the documented model allowlist/provider-auth path and hard-code Anthropic plugin catalog policy into core, so this PR should not merge as-is.

## What This Changes

The branch makes CLI runtime aliases visible as `/models` picker providers, sources CLI runtime rows from the model catalog and Claude CLI allowlist, and updates tests plus the changelog.

## Best Possible Solution

Expose CLI runtime choices through the existing visibility/auth contract, with provider-owned catalog data or a typed catalog seam for full CLI lists, while keeping catalog refresh tracked separately.

## Reproduction Assessment

Yes, source-level: current main filters every legacy runtime alias from `/models`, including CLI aliases, and the PR's screenshots show the intended Telegram-visible after state. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No: narrowing the legacy predicate is a plausible minimal fix, but bypassing `visibilityPolicy`/auth for CLI rows and importing an Anthropic plugin constant into core is not the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep CLI picker rows inside the model allowlist:** `src/auto-reply/reply/commands-models.ts:126-127`
  - body: This exemption skips the only `visibilityPolicy.allows(...)` check for `claude-cli`, `codex-cli`, and `google-gemini-cli`; the following unfiltered catalog pass and Claude allowlist seed can therefore show models outside `agents.defaults.models` and providers without usable auth. Telegram treats `byProvider` as selectable and persists the choice, so restricted configs will show buttons that are outside the documented allowlist contract. Please keep CLI rows on the same visibility/auth path, using `provider/*` or provider-owned catalog data when the full CLI list is actually allowed.
  - confidence: 0.9
- **[P2] Move provider catalog policy out of core:** `src/auto-reply/reply/commands-models.ts:1`
  - body: `commands-models.ts` is core auto-reply code, but this import makes it depend on Anthropic's plugin-local allowlist and hard-codes one bundled provider as a special picker source. That violates the plugin boundary and leaves other CLI runtimes asymmetric; the full Claude list should come through manifest/provider catalog metadata or a typed plugin/runtime seam instead of a relative `extensions/anthropic` import.
  - confidence: 0.82
