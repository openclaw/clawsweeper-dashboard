## Summary

Keep open; this PR addresses the reported default env SecretRef shape, but it is not merge-ready because the string-only normalizer ignores every configured SecretRef and can fall through to an unrelated `DISCORD_BOT_TOKEN`. Current main source, tests, and Discord docs confirm both the original crash path and the broader token-resolution contract.

## What This Changes

The PR switches Discord token normalization from strict SecretRef resolution to string-only normalization and updates Discord token/client tests for unresolved SecretRef fallback behavior.

## Best Possible Solution

Update Discord token resolution to inspect SecretRefs: resolve supported env refs by provider/id, preserve account-level SecretRef behavior, and avoid selecting unrelated fallback tokens for configured but unresolved non-env refs; add focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `resolveDiscordToken` sends a raw SecretRef into the strict helper, and the current unit test asserts the unresolved SecretRef throw.

## Solution Assessment

No. The proposed fix is too broad because it treats every SecretRef as missing, so it only fixes the special `DISCORD_BOT_TOKEN` fallback case while breaking or masking docs-supported file/exec, alternate env, and account-level SecretRefs.
