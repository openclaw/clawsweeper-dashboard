## Summary

Keep open. The PR addresses the current Discord SecretRef startup crash and the earlier changelog/non-env fallback feedback, but the updated fallback gate still bypasses configured SecretRef provider policy for matching env refs.

## What This Changes

The PR changes Discord token resolution, tests, and changelog so unresolved top-level `DISCORD_BOT_TOKEN` SecretRefs can fall through to the existing env fallback while non-matching/account SecretRefs report no token.

## Best Possible Solution

Make Discord mirror Telegram-style env SecretRef handling: validate provider source/default/allowlist before using `process.env`, preserve non-env/account intent, and keep the focused regression tests plus changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `resolveDiscordToken` sends a raw SecretRef into the strict helper, and the current unit test asserts the unresolved SecretRef throw; I did not run tests because this review is read-only.

## Solution Assessment

No. The repair direction is narrow, but this PR's matching-env fallback checks only `source` and `id`; the maintainable fix should also enforce SecretRef provider/default/allowlist semantics before reading the fallback env var.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate SecretRef providers before env fallback:** `extensions/discord/src/token.ts:65`
  - body: This fallback gate only checks that the unresolved ref is `env` and points at `DISCORD_BOT_TOKEN`. If the ref uses a non-default provider that is missing, configured with another source, or has an allowlist that excludes this id, the PR still reads `process.env.DISCORD_BOT_TOKEN` and starts the bot with a token the configured SecretRef provider would reject. Please validate the provider/default/allowlist contract, as Telegram does, before allowing the env fallback.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff still has a credential-policy regression risk because a configured Discord SecretRef can be bypassed by the fallback env token.

Concerns:

- **[medium] SecretRef provider policy can be bypassed:** `extensions/discord/src/token.ts:65`
  - body: A matching `env:...:DISCORD_BOT_TOKEN` ref falls through to `process.env` without checking whether that provider is configured as env, is the default env provider, or allowlists the id, which can bypass operator secret-provider policy and start the wrong bot token.
  - confidence: 0.86
