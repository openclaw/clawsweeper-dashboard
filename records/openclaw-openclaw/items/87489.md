## Summary

Keep this PR open: it addresses the main timeout path and includes useful real-behavior proof, but the latest head still checks the stored hard deadline only after a full `agent.wait`, so resumed or retried runs can exceed `runTimeoutSeconds` before parent wake/cleanup.

## What This Changes

The PR makes explicit subagent `runTimeoutSeconds` produce terminal timeout outcomes, adjusts timeout announce cleanup for delete mode, and adds focused registry/announce regressions.

## Best Possible Solution

Honor the stored timeout deadline before issuing or retrying `agent.wait`, or pass only the remaining time, while preserving the new terminal timeout outcome and delete-cleanup retry behavior.

## Reproduction Assessment

Yes, from source inspection: current main retries plain `agent.wait` timeouts without terminal snapshot fields, and the PR’s remaining defect is source-reproducible because the hard-deadline check runs after the full wait begins. I did not run a live current-main repro because this review is read-only.

## Solution Assessment

No, not yet: the main fix direction is right, but the PR must honor elapsed timeout before each wait/resume/retry rather than only after `agent.wait` returns. The narrower maintainable repair is to compute the remaining deadline at the registry wait boundary and cover the resumed/retried case.
