## Summary

Keep open. Current main still has the misleading infer audio/image empty-result wording, and this PR is a valid narrow fix direction, but it still needs branch-specific regression assertions, the required user-facing changelog entry, and fresh head validation before merge.

## What This Changes

The PR adds a helper that checks whether any media-understanding provider for a capability is configured, uses it to choose clearer image/audio infer CLI errors on empty results, and relaxes two existing CLI tests to accept the new messages.

## Best Possible Solution

Land a focused CLI fix that keeps the provider-configuration check at the empty-result boundary, preserves the old empty-provider-response wording when a provider is actually configured, adds explicit tests for configured-provider and no-provider cases for both image and audio, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. Current main can be reproduced from source by following the no-provider media-understanding path: no provider entries resolve, the runtime returns no text, and the CLI throws the old path-based image/audio error; linked issue #73569 also gives concrete shell commands for a default no-provider setup.

## Solution Assessment

Mostly yes. The PR's code direction uses the same media-understanding registry and generic config/env checks as the provider listing surface, which is the narrow maintainable place to distinguish no-provider from empty provider output; the safer completion is to tighten the tests and changelog before merge.
