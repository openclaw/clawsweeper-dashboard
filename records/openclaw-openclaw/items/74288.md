## Summary

Keep open. The linked CLI bug is still present on current main, and this PR is a focused fix in the right area, but the current head has a blocking type issue, weakened regression assertions, and no required changelog entry for a user-facing CLI fix.

## What This Changes

The PR adds a provider-availability helper to `src/cli/capability-cli.ts` and changes the image describe/audio transcribe empty-result paths and tests to emit missing-provider messages when no matching provider is configured.

## Best Possible Solution

Land a focused CLI fix that keeps the provider-configuration check at the image/audio empty-result boundary, uses the existing `MediaUnderstandingCapability` contract, preserves the old empty-provider-response wording when a configured provider returns no text, adds branch-specific tests for configured and unconfigured providers, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The linked bug provides concrete no-provider CLI commands, and current main source shows both image describe and audio transcribe still throw the old path-based messages when the media-understanding runtime returns no text.

## Solution Assessment

No as written. The implementation direction is narrow and maintainable, but the helper must be typed correctly, the tests must assert the new no-provider branches directly, and the required changelog entry must be added before merge.
