## Summary

Keep this PR open. Current main still has the misleading path-based no-text errors, and the PR is a narrow, appropriate direction, but it still needs the required changelog entry and branch-specific tests that would fail if the new missing-provider paths regressed.

## What This Changes

The PR adds a media-understanding provider-configuration helper, uses it to choose clearer `infer image describe` and `infer audio transcribe` empty-result errors, and updates the related CLI tests.

## Best Possible Solution

Land one focused CLI fix that distinguishes missing media-understanding provider configuration from a configured provider returning empty text, preserves the old empty-result wording for configured providers, adds explicit coverage for both image and audio branches, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. The linked issue gives concrete no-provider CLI steps, and current main source shows the no-provider media-understanding path returns no text and is converted into the old path-based image/audio errors.

## Solution Assessment

Mostly yes. Checking configured providers at the empty-result boundary reuses the existing provider-listing/config semantics and is narrow; the safer completion is to make the tests assert each branch directly and add the required changelog entry.
