## Summary

Keep open: current main and the latest release still send `dimensions` once on the direct memory-lancedb OpenAI-compatible path, while this PR is the canonical focused fix with sufficient real behavior proof; the remaining question is maintainer acceptance of the retry/truncate compatibility policy and sibling-provider consistency.

## What This Changes

The PR changes memory-lancedb direct OpenAI-compatible embedding calls to retry without `dimensions` after a clear provider field rejection, locally truncate the fallback vector, tighten dimensions validation coverage, and update plugin docs wording.

## Best Possible Solution

Land or explicitly decline the plugin-local fallback after maintainers choose the compatibility rule, and decide whether the shared OpenAI-compatible embedding provider should be aligned.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` still send `dimensions` once with no retry, and the inspected terminal screenshots show a live NVIDIA-compatible endpoint rejecting `body.dimensions` on main and succeeding on the branch.

## Solution Assessment

Yes, if maintainers want direct memory-lancedb recovery from dimensions-field rejection. If the intended product rule is provider-wide consistency, the safer solution is to align the shared OpenAI-compatible embedding provider first.
