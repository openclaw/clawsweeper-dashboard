## Summary

Keep open for maintainer review: current main and the latest release still have the direct memory-lancedb dimensions-forwarding behavior, and the PR has sufficient real proof with no concrete correctness or security finding. The remaining blocker is maintainer acceptance of the retry-without-dimensions plus local-truncation compatibility policy.

## What This Changes

The PR changes memory-lancedb direct OpenAI-compatible embeddings to retry without `dimensions` after recognized field rejection, locally truncate fallback vectors, and update dimensions docs and regression coverage.

## Best Possible Solution

Land the direct-client fallback only if maintainers accept that compatibility policy, and separately decide whether the shared OpenAI-compatible provider should adopt the same retry/truncation contract.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` still forward `dimensions` once with no retry, and the PR body includes terminal screenshots showing a live NVIDIA-compatible endpoint rejecting `body.dimensions` on main and succeeding on the branch.

## Solution Assessment

Yes, if maintainers want the direct memory-lancedb path to recover from dimensions-field rejection. A broader shared-provider policy would be cleaner if the same behavior should apply to every OpenAI-compatible embedding route.
