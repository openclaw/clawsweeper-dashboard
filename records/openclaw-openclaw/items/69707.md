## Summary

Keep open: the PR fixes a real rejecting-provider path, but it is not merge-ready because it drops shipped server-side `dimensions` pass-through for direct OpenAI-compatible embeddings and can break existing `memory-lancedb` setups.

## What This Changes

The PR changes `memory-lancedb` direct OpenAI-compatible embeddings to omit upstream `dimensions`, truncate returned vectors locally, validate invalid dimensions, and update tests/docs around that behavior.

## Best Possible Solution

Preserve shipped server-side `dimensions` behavior by default and add a narrow no-dimensions retry/local-truncation path only for providers that reject that field, with executed tests for both paths.

## Reproduction Assessment

Yes. Current main source shows direct memory-lancedb embeddings forwarding `dimensions`, and the inspected PR screenshot shows a real NVIDIA-compatible endpoint rejecting `body.dimensions` during `ltm search`.

## Solution Assessment

No. Unconditionally omitting `dimensions` fixes rejecting providers but is not the safest fix because it removes a shipped server-side dimensions path; a compatibility-preserving fallback is the better shape.
