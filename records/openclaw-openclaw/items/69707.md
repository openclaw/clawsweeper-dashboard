## Summary

Keep open for maintainer review: current main and the latest release still forward `dimensions` once on the direct `memory-lancedb` OpenAI-compatible path, while this PR has sufficient live proof and focused tests for a retry/truncate fix that needs explicit compatibility-policy acceptance before merge.

## What This Changes

The PR changes `memory-lancedb` direct OpenAI-compatible embedding calls to retry without `dimensions` on clear field rejection, locally truncate the fallback vector, tighten dimensions validation/tests, and update plugin docs wording.

## Best Possible Solution

Land this only after maintainers accept the direct-client retry/truncate compatibility policy and decide whether the shared OpenAI-compatible provider should use the same contract.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` still send `dimensions` once with no retry, and the inspected terminal screenshots show a live NVIDIA-compatible endpoint rejecting `body.dimensions` on main and succeeding on the branch.

## Solution Assessment

Yes, if maintainers want direct memory-lancedb recovery from dimensions-field rejection. If the intended product rule is provider-wide consistency, the safer solution is to align the shared OpenAI-compatible embedding provider too.
