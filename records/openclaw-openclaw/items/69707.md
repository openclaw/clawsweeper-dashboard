## Summary

Keep open: current main and the latest release still forward `dimensions` once on the direct `memory-lancedb` OpenAI-compatible path, while this PR appears to implement the narrow retry/truncation fix with sufficient real behavior proof. The remaining blocker is maintainer acceptance of the compatibility policy and sibling-provider consistency, not a contributor-facing correctness finding.

## What This Changes

The PR changes `memory-lancedb` direct OpenAI-compatible embedding requests to retry without `dimensions` after provider field rejection, locally truncate fallback vectors, validate dimensions, and update tests/docs.

## Best Possible Solution

Land the direct-client fallback only after maintainers accept the retry/truncate compatibility policy and decide whether the shared OpenAI-compatible provider should adopt the same contract.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` still forward `dimensions` once with no retry, and the inspected terminal screenshots show a live NVIDIA-compatible endpoint rejecting `body.dimensions` on main and succeeding on the branch.

## Solution Assessment

Yes, if maintainers want the direct memory-lancedb path to recover from dimensions-field rejection. If the intended product rule is provider-wide consistency, the cleaner solution is to define the same retry/truncation contract in the shared OpenAI-compatible embedding provider too.
