## Label Justifications

- `P2`: The PR targets a bounded memory-lancedb compatibility bug with plugin-local user impact.
- `merge-risk: 🚨 compatibility`: Existing direct OpenAI-compatible memory-lancedb configs would retry and locally truncate after dimensions-field rejection instead of returning unavailable memory immediately.

## Summary

Keep open: current main and the latest release still forward configured embedding dimensions once with no fallback, while this PR has useful proof and a focused implementation. The remaining blocker is maintainer acceptance of the retry-without-dimensions plus local-truncation compatibility policy, especially because the sibling shared OpenAI-compatible embedding provider remains different.

## What This Changes

The PR changes memory-lancedb direct OpenAI-compatible embeddings to retry without `dimensions` after recognized field rejection, locally truncate fallback vectors, update dimension wording, and add regression coverage.

## Best Possible Solution

Land the narrow direct-client behavior only if maintainers accept that compatibility policy, and separately decide whether the shared OpenAI-compatible provider should adopt the same retry/truncation contract.

## Reproduction Assessment

Yes, with high-confidence source and contributor proof rather than a local rerun. Current main and `v2026.6.10` forward `dimensions` once with no retry, and the inspected terminal screenshots show a live NVIDIA-compatible endpoint rejecting `body.dimensions` on main and succeeding on the branch.

## Solution Assessment

Yes with maintainer acceptance. The patch is a narrow direct-client fix that preserves server-side dimensions for accepting providers, but retry-without-dimensions plus local prefix truncation is a compatibility policy choice and the shared-provider divergence needs owner signoff.
