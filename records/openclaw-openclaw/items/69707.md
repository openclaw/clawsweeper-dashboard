## Summary

Keep open: this PR addresses a real memory-lancedb direct OpenAI-compatible embedding bug with strong before/after terminal proof, but the fallback detector is broad enough to mask unsupported dimension-value errors unless maintainers narrow or explicitly accept that compatibility policy.

## What This Changes

The PR changes memory-lancedb’s direct OpenAI-compatible embedding client to retry without `dimensions` on provider rejection, locally truncate fallback vectors, and add docs/tests for dimension validation and fallback behavior.

## Best Possible Solution

Land a narrow direct-client fix that retries only on clear unsupported-field errors, preserves server-side dimensions for accepting providers, fails loudly on invalid dimension values or too-short fallback vectors, and keeps the docs/tests aligned with that contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: current main and the latest release both forward `dimensions` without a retry, and the inspected terminal proof shows NVIDIA rejecting `body.dimensions` in the documented direct endpoint setup; I did not rerun live credentials locally.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the direct-client retry is a plausible scoped fix, but the error detector should be narrowed or explicitly accepted so unsupported dimension-value errors do not get silently converted into local truncation.
