## Summary

This PR should stay open: it fixes a real proof-gate automation gap for private maintainers, but the current diff mints a GitHub App token in a `pull_request_target` workflow without constraining the token to the membership read permission the checker needs.

## What This Changes

The PR adds GitHub App token minting to the real-behavior-proof workflow, checks `openclaw/maintainer` team membership before enforcing proof, and adds helper tests.

## Best Possible Solution

Add `permission-members: read` to both `actions/create-github-app-token@v3` steps, keep the trusted base-checkout and fallback behavior, then merge after targeted proof-policy tests and CI confirm maintainer-skip and fallback behavior.

## Reproduction Assessment

Yes. Current-main source shows a `CONTRIBUTOR` PR author with no proof section is still treated as external and fails the proof gate, which matches the private-maintainer failure mode described by the PR.

## Solution Assessment

Mostly yes. A direct maintainer-team lookup is the narrowest fix and avoids labeler ordering races, but the workflow must request only `permission-members: read` before merge.
