## Summary

Keep this PR open. Current main still has no `maxCostPerRun` or `cost-cap-exceeded` implementation, so the feature is not obsolete, but the proposed patch is not merge-ready: the public config schema would still reject the new key, the estimate prices a smaller context and sometimes the wrong model, and the guard is heartbeat-local despite the PR claiming embedded-run coverage. Prior Greptile and ClawSweeper comments pointed at the same risk area, and current source inspection confirms the blockers remain relevant.

## What This Changes

This PR adds a `heartbeat.maxCostPerRun` TypeScript config field, a pre-model cost estimate and skip branch in `runHeartbeatOnce`, model-pricing helpers, and heartbeat cost-cap tests.

## Best Possible Solution

Keep the PR open for maintainer-approved revision or replacement. A complete implementation should add the public config key to runtime and generated schemas, docs, and changelog; estimate against the same effective model and prompt context that will be sent to the provider; enforce the cap at the shared embedded-run boundary or explicitly cover cron variants; and include focused tests for heartbeat, cron, per-agent model overrides, transcript-heavy sessions, invalid cap values, and zero-cost models.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main's strict heartbeat schemas omit `maxCostPerRun`, and the PR diff places the estimate before the later effective model/context path used by `getReplyFromConfig`.

## Solution Assessment

No. The cost-cap direction is useful, but this patch is not the best solution because it adds only a type-level config field, prices an incomplete prompt/model, and does not enforce the cap at the shared embedded-run surface promised by the PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add maxCostPerRun to runtime schemas:** `src/config/types.agent-defaults.ts:269`
  - body: Adding only the TypeScript field does not make the config key usable. `HeartbeatSchema` and the generated schema are strict, so normal config validation will still reject `heartbeat.maxCostPerRun` until the zod schema, generated schema/help surface, docs, and validation coverage are updated.
  - confidence: 0.93
- **[P1] Estimate the full prompt context before skipping:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The cap is priced from `ctx.Body` alone, but default heartbeat runs can also send session transcript plus system/bootstrap/tool context. A long-lived session can pass this check and still make the expensive call the cap is meant to prevent, so the estimate needs to use the effective prompt context or a conservative explicit overhead.
  - confidence: 0.94
- **[P2] Use the effective agent model for pricing:** `src/infra/heartbeat-runner.ts:725`
  - body: When `heartbeat.model` is unset, this prices a global default instead of the per-agent model resolved later by `getReplyFromConfig()`. Multi-agent configs can therefore bypass the cap for an expensive agent model or skip a cheap one by pricing the wrong model.
  - confidence: 0.91
- **[P2] Move the cap to the shared embedded-run boundary:** `src/infra/heartbeat-runner.ts:722-724`
  - body: The PR describes dispatch-level coverage for heartbeat, cron, and future triggers, but the guard is inserted inside `runHeartbeatOnce()`. Isolated cron and other embedded-run entry points can still dispatch without this check, so the cap should live at a shared boundary or have equivalent cron coverage and tests.
  - confidence: 0.86
- **[P2] Avoid importing plugin internals from core tests:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: The new core infra test deep-imports Telegram and WhatsApp files from `extensions/*/src`. That crosses the core/plugin boundary and can pull plugin runtime dependencies into core lanes; use public plugin/API fixtures or a generic registry seam instead.
  - confidence: 0.82
- **[P3] Make the zero-cap test match its asserted behavior:** `src/infra/heartbeat-runner.cost-cap.test.ts:330-341`
  - body: The test name says a free catalog model proceeds with `maxCostPerRun: 0`, but it uses a non-free `gemini-2.0-flash` model and asserts a skip. Either use a truly zero-cost catalog model for the stated case or rename the test to cover cheap non-free models at a zero cap.
  - confidence: 0.86
