## Summary

Keep open: the cost-cap feature is still absent on main and addresses real cost-control reports, but this PR has blocking correctness and public-contract gaps, so it should not merge as-is.

## What This Changes

This PR adds a `heartbeat.maxCostPerRun` config type, a pre-model estimated-cost skip guard in `runHeartbeatOnce`, pricing helpers, and cost-cap tests.

## Best Possible Solution

Add the public config key across schemas, docs, generated help, and changelog, then enforce the cap at a shared embedded-run boundary using the effective model and a conservative estimate of the full prompt/transcript context with heartbeat and cron coverage.

## Reproduction Assessment

Yes. Static source review is high-confidence: current main lacks `maxCostPerRun`, and the PR diff conflicts with current strict heartbeat schemas plus downstream model/session resolution.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it prices a partial prompt with a pre-resolution model inside heartbeat-specific code and leaves the public config schemas/docs incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add maxCostPerRun to runtime schemas:** `src/config/types.agent-defaults.ts:269`
  - body: Adding the TypeScript field alone does not make the config key usable. `HeartbeatSchema` is strict and the generated schema sets `additionalProperties: false`, so normal config validation will still reject `heartbeat.maxCostPerRun` until the zod schema, generated schema/help surface, docs, and coverage are updated.
  - confidence: 0.93
- **[P1] Estimate the actual embedded prompt before skipping:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The guard prices only `ctx.Body`, but non-isolated heartbeat runs can also send bootstrap/system/tool context and the persisted session transcript. A long-lived session can pass this check and still make the expensive call the cap is meant to prevent.
  - confidence: 0.94
- **[P2] Price the effective model selected for the run:** `src/infra/heartbeat-runner.ts:725`
  - body: When `heartbeat.model` is unset, the estimate runs before `getReplyFromConfig` resolves the session/agent model and heartbeat override path. Multi-agent setups can therefore skip or allow a run based on a different model than the one actually dispatched.
  - confidence: 0.91
- **[P2] Enforce the cap at the shared embedded-run boundary:** `src/infra/heartbeat-runner.ts:722-724`
  - body: The PR promises dispatch-level coverage for heartbeat, cron, and future triggers, but the guard is inserted inside `runHeartbeatOnce`. Isolated cron and later embedded-run entry points can still dispatch without this check unless enforcement moves to a shared boundary or equivalent trigger-specific coverage is added.
  - confidence: 0.86
- **[P2] Keep core tests out of plugin internals:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: The new core infra test deep-imports Telegram and WhatsApp files from `extensions/*/src`. That crosses the core/plugin boundary and can pull plugin runtime dependencies into core lanes; use public plugin/API fixtures or a generic registry seam instead.
  - confidence: 0.82
- **[P3] Fix the zero-cap test case:** `src/infra/heartbeat-runner.cost-cap.test.ts:330-341`
  - body: The test name says a free catalog model proceeds with `maxCostPerRun: 0`, but it uses non-free `gemini-2.0-flash` and asserts a skip. Either use a truly zero-cost catalog model for the stated case or rename the test to cover cheap non-free models at a zero cap.
  - confidence: 0.86
