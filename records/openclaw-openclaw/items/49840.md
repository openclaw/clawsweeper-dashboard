## Summary

Keep open. Current main still lacks a native per-run embedded cost cap, but this PR is a new config/API feature with unresolved correctness blockers and missing contributor real-behavior proof before it can merge.

## What This Changes

The PR adds a heartbeat.maxCostPerRun type field, a prompt-size cost estimate and skip path in runHeartbeatOnce, pricing helpers, and new cost-cap tests.

## Best Possible Solution

Define a schema-backed maxCostPerRun contract and enforce it at the shared embedded-run boundary using the effective model and a conservative full prompt/transcript estimate, with docs, focused heartbeat/cron coverage, and real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new feature/config request. Source inspection shows current main lacks the feature, and the merge blockers are visible in the PR diff and current runtime path.

## Solution Assessment

No. The direction is useful, but a heartbeat-local prompt-only precheck is not the best maintainable fix for a public embedded-run cost cap; schema-backed shared-boundary enforcement with effective-model and fuller-context pricing is safer.

Full review comments:

- **[P2] Add maxCostPerRun to the runtime schema:** `src/config/types.agent-defaults.ts:266-272`
  - body: Adding the TypeScript field alone does not make the config key usable. HeartbeatSchema is strict, so normal config validation will still reject heartbeat.maxCostPerRun until the zod schema and generated/help/docs surfaces are updated together.
  - confidence: 0.93
- **[P1] Estimate the full embedded prompt before skipping:** `src/infra/heartbeat-runner.ts:724-728`
  - body: This guard prices only ctx.Body, but non-isolated heartbeat runs can include system/bootstrap/tool context and the persisted session transcript. A long-lived session can pass this cap and still make the expensive call the feature is meant to prevent.
  - confidence: 0.94
- **[P2] Price the effective dispatched model:** `src/infra/heartbeat-runner.ts:724-728`
  - body: When heartbeat.model is unset, this precheck falls back to resolveDefaultModelId(cfg) before getReplyFromConfig resolves the session agent and effective model. Multi-agent setups can skip or allow a run based on a different model than the one actually dispatched.
  - confidence: 0.91
- **[P2] Enforce the cap at the shared embedded-run boundary:** `src/infra/heartbeat-runner.ts:719-724`
  - body: The PR promises heartbeat, cron, and future trigger coverage, but the guard is inserted inside runHeartbeatOnce. Isolated cron and later embedded-run entry points can still dispatch without this cap unless enforcement moves to a shared boundary or equivalent trigger coverage is added.
  - confidence: 0.87
- **[P2] Keep core cost-cap tests out of plugin internals:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: This core infra test deep-imports Telegram and WhatsApp files from extensions/*/src. That crosses the core/plugin boundary and can pull plugin runtime internals into core lanes; use public plugin fixtures or a generic registry seam instead.
  - confidence: 0.84
