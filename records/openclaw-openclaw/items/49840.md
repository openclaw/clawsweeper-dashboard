## Summary

Keep this PR open: current main still does not implement a per-run embedded cost cap, and the proposed patch has blocking API and runtime correctness gaps around schema acceptance, prompt/model estimation, and shared trigger coverage.

## What This Changes

This PR adds a `heartbeat.maxCostPerRun` TypeScript config field, a pre-model estimated-cost skip path in `runHeartbeatOnce`, model-pricing helpers, and heartbeat cost-cap tests.

## Best Possible Solution

A complete solution should add the public config key across runtime/generated schemas, docs, and changelog, then enforce the cap at a shared embedded-run boundary using the effective model and a conservative estimate of the full prompt/transcript context, with heartbeat and cron coverage.

## Reproduction Assessment

Yes. Static review is high-confidence: current main still lacks the feature, and the PR diff can be checked against strict schema validation plus the current prompt/model dispatch flow.

## Solution Assessment

No. The cost-cap direction is useful, but this patch is not the best implementation because it adds only a type-level config key, estimates before effective model/context resolution, and does not place enforcement at the shared embedded-run surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add maxCostPerRun to runtime schemas:** `src/config/types.agent-defaults.ts:269`
  - body: Adding the TypeScript field alone does not make the config key usable. `HeartbeatSchema` is strict and the generated schema sets `additionalProperties: false`, so normal config validation will still reject `heartbeat.maxCostPerRun` until the zod schema, generated schema/help surface, docs, and coverage are updated.
  - confidence: 0.93
- **[P1] Estimate the actual embedded prompt before skipping:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The guard prices only `ctx.Body`, but non-isolated heartbeat runs can also send bootstrap/system/tool context and the persisted session transcript. A long-lived session can pass this check and still make the expensive call the cap is meant to prevent.
  - confidence: 0.94
- **[P2] Price the effective model selected for the run:** `src/infra/heartbeat-runner.ts:725`
  - body: When `heartbeat.model` is unset, this prices a default model before `getReplyFromConfig` applies session, parent, or channel model overrides. Multi-agent setups can therefore skip or allow a run based on a different model than the one actually dispatched.
  - confidence: 0.91
- **[P2] Move the cap out of heartbeat-only dispatch:** `src/infra/heartbeat-runner.ts:722-724`
  - body: The PR promises dispatch-level coverage for heartbeat, cron, and future triggers, but the guard is inserted inside `runHeartbeatOnce`. Isolated cron and later embedded-run entry points can still dispatch without this check, so enforcement needs a shared boundary or equivalent trigger-specific coverage.
  - confidence: 0.86
- **[P2] Avoid importing plugin internals from core tests:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: The new core infra test deep-imports Telegram and WhatsApp files from `extensions/*/src`. That crosses the core/plugin boundary and can pull plugin runtime dependencies into core lanes; use public plugin/API fixtures or a generic registry seam instead.
  - confidence: 0.82
- **[P3] Make the zero-cap test match its behavior:** `src/infra/heartbeat-runner.cost-cap.test.ts:330-341`
  - body: The test name says a free catalog model proceeds with `maxCostPerRun: 0`, but it uses non-free `gemini-2.0-flash` and asserts a skip. Either use a truly zero-cost catalog model for the stated case or rename the test to cover cheap non-free models at a zero cap.
  - confidence: 0.86
