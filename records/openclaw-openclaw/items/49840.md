  - body: The branch only adds the TypeScript field, but `HeartbeatSchema` is strict and still stops at `skipWhenBusy`. A config using the PR's example key can be rejected before the runner ever sees the cap.
  - confidence: 0.94
- **[P1] Estimate the full prompt context before skipping:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The guard prices only `ctx.Body`. Default non-isolated heartbeat runs also carry bootstrap/system/tool context and the persisted session transcript into the model call, so a large existing session can pass the cap and still dispatch the expensive run.
  - confidence: 0.94

  - body: When `heartbeat.model` is unset, the precheck uses `resolveDefaultModelId(cfg)`, but `getReplyFromConfig` resolves the per-agent default and heartbeat override after deriving `agentId`. Multi-agent configs can be allowed or skipped against a different model from the one called.
  - confidence: 0.91
- **[P2] Move the cap to the embedded-run boundary:** `src/infra/heartbeat-runner.ts:719-724`
  - body: The PR says the guard covers heartbeat, cron, and future triggers, but the new check runs only inside `runHeartbeatOnce`. Isolated cron jobs still call `runEmbeddedPiAgent` directly with no cost-cap path.
  - confidence: 0.89
- **[P2] Keep core tests on public plugin seams:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: This core infra test deep-imports Telegram and WhatsApp implementation/runtime files from `extensions/*/src`. That crosses the plugin boundary; use public plugin fixtures, SDK facades, or a generic registry seam instead.
  - confidence: 0.84

  - body: The hardcoded table covers Claude 4 and `claude-haiku`, but common `claude-3-*` IDs do not match those prefixes. Without catalog pricing, those models fall to the expensive default and can be skipped far earlier than intended.
  - confidence: 0.78
