## Summary

Keep this PR open. Current main still does not implement a per-run embedded cost cap, and the closed tracking issue points to this PR as the remaining canonical work, but the submitted patch has blocking correctness and config-surface gaps that should be resolved before merge.

## What This Changes

This PR adds a `heartbeat.maxCostPerRun` type field, a heartbeat pre-dispatch cost estimate/skip branch returning `cost-cap-exceeded`, model-pricing helpers, and new heartbeat cost-cap tests.

## Best Possible Solution

Keep the PR open for revision or a maintainer-approved replacement. A complete fix should add the accepted config key to the runtime schema, generated schema/help, and docs; estimate against the same effective model and prompt context that will be sent to the provider; enforce the cap at the shared embedded-run boundary or all intended trigger paths; and cover heartbeat, cron, invalid values, zero/free models, per-agent model overrides, and transcript-heavy sessions with focused tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main rejects unknown heartbeat config keys through strict schemas, and the PR patch visibly estimates only `ctx.Body` before the later model-selection path runs.

## Solution Assessment

No. The cost-cap direction is useful, but this patch is not the best implementation because it validates only a TypeScript type, estimates a smaller prompt than the actual provider call, prices a possibly different model, and places the guard in the heartbeat-specific path despite the linked issue asking for embedded-run coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add maxCostPerRun to runtime schemas:** `src/config/types.agent-defaults.ts:269`
  - body: Adding only the TypeScript field does not make the config key usable: `HeartbeatSchema` and the generated config schema are strict, so users who add `heartbeat.maxCostPerRun` through normal config validation will still hit an unknown-property path. Please add the zod field, regenerate the schema/help/docs surface, and cover validation.
  - confidence: 0.93
- **[P1] Estimate the full prompt context before skipping:** `src/infra/heartbeat-runner.ts:724`
  - body: The cap is priced from `ctx.Body` alone, but default heartbeat runs also send the session transcript plus system, bootstrap, and tool context unless `isolatedSession` is enabled. A long-lived session can pass this check and still make the expensive call the cap is meant to prevent, so the estimate needs to use the same effective prompt context or an explicit conservative overhead.
  - confidence: 0.94
- **[P2] Use the effective agent model for pricing:** `src/infra/heartbeat-runner.ts:725`
  - body: When `heartbeat.model` is unset, this prices `resolveDefaultModelId(cfg)`, which only looks at global defaults. The actual call resolves the per-agent model and aliases later inside `getReplyFromConfig()`, so multi-agent configs can either bypass the cap for an expensive agent model or skip a cheap one by pricing the wrong model.
  - confidence: 0.91
- **[P2] Enforce the cap outside the heartbeat-only path:** `src/infra/heartbeat-runner.ts:722-724`
  - body: The guard is inserted inside `runHeartbeatOnce()`, so isolated cron jobs and future embedded trigger paths still reach their embedded runner without this check even though the PR describes a dispatch-level cap. Move the check to a shared embedded-run boundary or add equivalent cron coverage and tests.
  - confidence: 0.86
- **[P2] Avoid importing plugin internals from core tests:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: The new core infra test imports `extensions/telegram/src/*` and `extensions/whatsapp/src/*` directly. That crosses the repo's core/plugin boundary and can pull plugin runtime dependencies into core test lanes; use public plugin/API test fixtures or a generic test registry seam instead.
  - confidence: 0.82
