  - body: Adding only the TypeScript field leaves the public config invalid. `HeartbeatSchema` is strict and the schema/help/inventory surfaces do not include `maxCostPerRun`, so normal config validation can reject the example key instead of honoring it.

  - body: The guard prices only `ctx.Body`, but non-isolated heartbeat runs can include system/bootstrap/tool context and persisted session messages at the LLM boundary. A long transcript can pass this cap and still dispatch the expensive call the feature is meant to prevent.
  - confidence: 0.94

  - body: When `heartbeat.model` is unset, this precheck uses `resolveDefaultModelId(cfg)` instead of `getReplyFromConfig`'s per-agent `resolveDefaultModel` path. Agent-scoped model overrides can therefore be allowed or skipped based on a different model than the run uses.
  - confidence: 0.91
- **[P2] Enforce the cap at the shared embedded-run boundary:** `src/infra/heartbeat-runner.ts:719-724`
  - body: The PR says this covers heartbeat, cron, and future triggers, but the guard is inside `runHeartbeatOnce`. Isolated cron turns call `runEmbeddedPiAgent` through their own executor, so those embedded runs still have no per-run cap.
  - confidence: 0.89
- **[P2] Keep core tests out of plugin internals:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: This new core infra test deep-imports Telegram and WhatsApp implementation files from `extensions/*/src`. That crosses the repo's core/plugin boundary; use public plugin fixtures, SDK facades, or a generic registry seam instead.
