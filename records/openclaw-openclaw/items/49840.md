## Summary

Keep open. Current main still lacks a native `maxCostPerRun` cap, and this active feature PR is the canonical thread, but the branch has blocking correctness gaps in the config contract and cost-estimation path plus missing real behavior proof.

## What This Changes

The branch adds a heartbeat `maxCostPerRun` config type, a pre-`getReplyFromConfig` estimated-cost skip in `runHeartbeatOnce`, model-pricing helpers, and cost-cap tests.

## Best Possible Solution

Revise the PR so the config schema, type, docs, and audit inventory all expose the same option; estimate against the actual resolved model and relevant input context; clearly scope heartbeat versus cron coverage; then prove the behavior with a real config-driven run.

## Reproduction Assessment

Not applicable as a feature request rather than a current-main bug report. Source inspection does give high confidence that current main lacks the requested cap and that the PR's proposed config path is incomplete.

## Solution Assessment

No. The current branch is a plausible starting point, but adding a TS-only config field and pricing only `ctx.Body` is not the narrowest maintainable way to deliver a trustworthy cost cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire the new heartbeat key into the schema:** `src/config/types.agent-defaults.ts:263-270`
  - body: Adding only the TypeScript field leaves `HeartbeatSchema` as a strict object without `maxCostPerRun`, so a real user config containing the documented key will be rejected before the runner can read it. Add the zod schema/docs/audit coverage with this type entry so the option is actually usable.
  - confidence: 0.95
- **[P1] Include persisted session context in the estimate:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The guard prices only `ctx.Body`, but default heartbeat turns run with a session key and the embedded runner replays the persisted transcript/system/tool context. A long main session can stay under the prompt-only estimate while the actual model call is far above the cap, so this does not bound the expensive runs it is meant to prevent.
  - confidence: 0.91
- **[P2] Price the same model the runner will execute:** `src/infra/heartbeat-runner.ts:726-728`
  - body: When `heartbeat.model` is unset, this uses `resolveDefaultModelId(cfg)`, but `getReplyFromConfig` resolves the effective model with `agentId` and can also apply channel or stored session overrides. In multi-agent or overridden sessions, the cap can price a cheap default while the run executes an expensive model, or skip a cheap run priced as the wrong default.
  - confidence: 0.89
