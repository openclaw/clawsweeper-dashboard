## Summary

Keep open. The feature is still useful and not implemented on current main, but this PR is not merge-ready because the public config surface is incomplete, the guard estimates only a partial heartbeat prompt with a pre-resolution model, and enforcement is heartbeat-specific despite the requested embedded-run scope.

## What This Changes

The PR adds a `heartbeat.maxCostPerRun` config type, a pre-model estimated-cost skip guard in `runHeartbeatOnce`, pricing helpers, and cost-cap tests.

## Best Possible Solution

Define a schema-backed public `maxCostPerRun` contract and enforce it at a shared embedded-run boundary using the effective model and a conservative estimate of the full prompt/transcript context, with docs and focused heartbeat/cron coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new feature/config request. Static source review is high-confidence that current main lacks the feature and that the PR’s merge blockers are present in the diff.

## Solution Assessment

No. The direction is useful, but this implementation is not the best fix because it prices a partial prompt with a pre-resolution model inside heartbeat-specific code and leaves the public schema/docs surface incomplete.

Full review comments:

- **[P2] Add maxCostPerRun to the runtime config schema:** `src/config/types.agent-defaults.ts:269`
  - body: Adding the TypeScript field alone does not make the config key usable. `HeartbeatSchema` is strict, so normal config validation will still reject `heartbeat.maxCostPerRun` until the zod schema, generated/help surfaces, docs, and coverage are updated.
  - confidence: 0.93
- **[P1] Estimate the full embedded prompt before skipping:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The guard prices only `ctx.Body`, but non-isolated heartbeat runs can also include system/bootstrap/tool context and persisted session transcript. A long-lived session can pass this cap and still make the expensive call the feature is meant to prevent.
  - confidence: 0.94
- **[P2] Price the model that will actually be dispatched:** `src/infra/heartbeat-runner.ts:725`
  - body: When `heartbeat.model` is unset, this precheck falls back to `resolveDefaultModelId(cfg)` before `getReplyFromConfig` resolves the session agent and effective model. Multi-agent setups can therefore skip or allow a run based on a different model than the one dispatched.
  - confidence: 0.91
- **[P2] Move enforcement to the shared embedded-run boundary:** `src/infra/heartbeat-runner.ts:722-724`
  - body: The PR promises heartbeat, cron, and future trigger coverage, but the guard is inserted inside `runHeartbeatOnce`. Isolated cron and later embedded-run entry points can still dispatch without this cap unless enforcement moves to a shared boundary or equivalent trigger coverage is added.
  - confidence: 0.87
- **[P2] Keep core tests out of plugin internals:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: This core infra test deep-imports Telegram and WhatsApp files from `extensions/*/src`. That crosses the core/plugin boundary and can pull plugin runtime internals into core lanes; use public plugin fixtures or a generic registry seam instead.
  - confidence: 0.84
