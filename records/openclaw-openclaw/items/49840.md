## Summary

Keep open. Current main still lacks a native per-run cost cap and the closed feature issue was intentionally superseded by this PR, but the patch is not merge-ready: it adds a new public config knob without the runtime schema/docs/help surface, estimates only part of the actual embedded request, prices before effective model resolution, and does not enforce at the shared cron/heartbeat embedded-run boundary.

## What This Changes

This PR adds a `heartbeat.maxCostPerRun` config type, a pre-model estimated-cost skip guard in `runHeartbeatOnce`, pricing helpers, and cost-cap tests.

## Best Possible Solution

Define and document a `maxCostPerRun` contract, add the config key across schema/help/docs/tests, then enforce it at a shared embedded-run boundary using the resolved effective model and a conservative estimate of the assembled prompt/session context for heartbeat and cron paths.

## Reproduction Assessment

Yes. Static source review is high-confidence: current main lacks `maxCostPerRun`, and the PR diff conflicts with current strict heartbeat schemas plus downstream model/session resolution.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it prices a partial prompt inside heartbeat-specific code before effective model selection and leaves the public config surface incomplete.

- **[P2] Add maxCostPerRun to runtime schemas:** `src/config/types.agent-defaults.ts:262-269`
  - body: Adding the TypeScript field alone does not make the config key usable. `HeartbeatSchema` is strict, so normal config validation will still reject `heartbeat.maxCostPerRun` until the zod schema and generated help/docs surfaces are updated.
  - confidence: 0.93
- **[P1] Estimate the assembled embedded request:** `src/infra/heartbeat-runner.ts:724-728`
  - body: The guard prices only `ctx.Body`, but non-isolated heartbeat runs can also send bootstrap/system/tool context and the persisted session transcript. A long-lived session can pass this check and still make the expensive call the cap is meant to prevent.
  - confidence: 0.94
- **[P2] Price the resolved run model:** `src/infra/heartbeat-runner.ts:725`
  - body: When `heartbeat.model` is unset, this estimates with `resolveDefaultModelId(cfg)` before `getReplyFromConfig` resolves the session/agent model. Multi-agent setups can skip or allow a run based on a different model than the one actually dispatched.
  - confidence: 0.91
- **[P2] Move enforcement to the shared embedded-run boundary:** `src/infra/heartbeat-runner.ts:722-724`
  - body: The PR promises heartbeat, cron, and future trigger coverage, but the check is inserted only in `runHeartbeatOnce`. Isolated cron and later embedded-run entry points can still dispatch without this cap unless enforcement moves to a shared boundary or equivalent trigger-specific coverage is added.
  - confidence: 0.86
- **[P2] Keep core tests on public plugin seams:** `src/infra/heartbeat-runner.cost-cap.test.ts:2-5`
  - body: This core infra test deep-imports Telegram and WhatsApp implementation files from `extensions/*/src`. That crosses the core/plugin boundary and can pull plugin runtime dependencies into core lanes; use public plugin/API fixtures or a generic registry seam instead.
  - confidence: 0.82
