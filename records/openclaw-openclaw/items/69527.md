## Summary

Keep open. Current `main` still drops fallback-attempt diagnostics from `infer model run --json`, and `models status` still builds the double-prefixed `openrouter/openrouter/auto` label reported here. Open PR #74028 addresses the related local `--model` override bug in #73921, but it does not make this issue closeable because the fallback-attempt and OpenRouter preset/status parts remain unresolved.

## What This Changes

The issue requests that `openclaw infer model run` honor a configured OpenRouter primary model, handle OpenRouter preset refs, and show fallback attempts when failover occurs.

## Best Possible Solution

Land or incorporate the focused local override fix from PR #74028, then complete the remaining model-run diagnostics work: thread fallback-attempt metadata into local and gateway `model.run` JSON envelopes, fix the OpenRouter status label formatting, and add OpenRouter `openrouter/auto` plus `@preset/...` regression coverage in provider-owned seams without hardcoding OpenRouter policy into generic core fallback logic.

## Reproduction Assessment

Yes for the current diagnostics and status-label gaps: source inspection on current-main-equivalent blobs shows `attempts: []` is still hard-coded in both `model.run` transports while the fallback layer records attempts, and `models status` still concatenates a double-prefixed OpenRouter label. The live OpenRouter preset failure remains partially reproducible only from the reporter's logs and needs provider-backed verification.

## Solution Assessment

No patch is under review for this issue. The best fix is a narrow CLI/runtime metadata propagation and display-formatting change, coordinated with PR #74028 for the local override half, rather than a broad fallback-policy rewrite or OpenRouter-specific core migration.
