## Summary

Keep open: the PR addresses a real Control UI metadata leak, but the current diff filters empty normalized history items before canvas tool previews are lifted, which can hide canvas-only assistant output that current main can render.

## What This Changes

The PR strips sender metadata from Control UI chat history/live stream display, omits operator UI sender identity fields in gateway `chat.send`, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the metadata stripping and gateway sender changes, but filter metadata-only history without removing empty assistant anchors needed for lifted canvas previews.

## Reproduction Assessment

Yes. Source inspection on current main shows active streams render raw `props.stream` and `chat.send` populates sender identity from the Control UI client; the PR body also supplies after-fix source-level terminal output for the metadata stream/history path.

## Solution Assessment

No. The fix direction is correct, but the empty-message filter is placed too early and should run after canvas lifting or be narrowed to metadata-only entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve canvas anchors before filtering empty messages:** `ui/src/ui/chat/build-chat-items.ts:310-312`
  - body: The new history filter runs before `liftedCanvasSources` attaches canvas previews to the nearest assistant message. An assistant turn with no visible text but a canvas-render tool result used to remain as the anchor and then receive the canvas block; after this check it is skipped, so `findNearestAssistantMessageIndex` returns null and the preview disappears. Please delay the empty-message filter until after canvas lifting or make it target only metadata-only entries.
  - confidence: 0.83
