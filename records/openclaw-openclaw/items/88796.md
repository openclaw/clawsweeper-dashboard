## What This Changes

The PR makes Discord message search accept `query` or `content`, infer `guildId` from a Discord channel when omitted, normalize `channel:` targets, and avoid injecting the session channel when explicit filters or guild scope are present.

## Best Possible Solution

Land this after redacted live Discord guild-channel proof shows a model/tool search without `guildId` resolving the channel to a guild, preserving explicit channel filters, and returning the expected search behavior; otherwise a maintainer should explicitly accept the proof risk.

## Reproduction Assessment

Yes, from source and linked report evidence: current main requires `guildId` before Discord `searchMessages` can run, while the model-visible/search entry path can naturally provide only `query`. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Yes, this is a good bounded fix shape: it stays in the Discord plugin action/runtime layer, keeps the guild-scoped REST boundary, reuses existing target normalization and read authorization, and avoids broadening explicit filters. Adding a new shared `currentGuildId` context field would be broader than needed for this PR.
