## Summary

This PR should stay open: it targets a real Discord search bug that still exists on current main and the latest release, but the current patch can narrow explicit guild-wide searches to the active session channel and still lacks live Discord/API proof for the changed path.

## What This Changes

The PR makes Discord message search accept query/content, infer a missing guild id from a channel or session channel, normalize channel ids, and adds regression tests.

## Best Possible Solution

Gate the session-channel fallback on both missing guildId and missing channel filters, keep explicit guildId/no-filter searches guild-wide, add that regression test, then require redacted live Discord proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.1 require guildId before Discord search can proceed, matching the linked live logs in https://github.com/openclaw/openclaw/issues/88790. The PR-introduced scope regression is also source-reproducible from the adapter fallback and the runtime guild-wide contract, though I did not run live Discord.

## Solution Assessment

No as written: the omitted-guildId direction and channel-id normalization are useful, but the adapter fallback is too broad. The narrow maintainable fix is to use the session channel only when guildId and explicit channel filters are absent.
