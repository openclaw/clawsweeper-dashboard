## Summary

Keep open. The PR targets a real Discord search bug, but it is not merge-ready because the new session-channel fallback can silently narrow explicit guildId-only searches and the supplied proof still does not show the fixed path in a live Discord bot/API flow.

## What This Changes

The PR makes Discord message search accept query/content aliases, infer guildId from channel info, normalize channel IDs, and adds regression tests for the new search behavior.

## Best Possible Solution

Preserve guild-wide search semantics when guildId is explicit, use the session channel only to infer guildId when needed, and add live redacted Discord proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main requires guildId for Discord searchMessages, and the linked issue includes live logs where the model sends query without guildId and receives ToolInputError: guildId required. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No, not as written: the channel-info fallback is a plausible fix layer, but the tool entry point should not inject the session channel when guildId is already explicit. The safer fix keeps guildId-only searches guild-wide and uses the session channel only when it is needed to infer the missing guildId.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep guildId-only searches guild-wide:** `extensions/discord/src/actions/handle-action.guild-admin.ts:442-444`
  - body: When `message.search` is called from a Discord session with an explicit `guildId` but no `channelId`/`channelIds`, this fallback injects `ctx.toolContext.currentChannelId`. The runtime then sends `channelIds` to `searchMessagesDiscord`, so a previously guild-wide search is silently narrowed to the current channel and can miss valid results from other allowed channels. Gate the fallback on `!guildId` and add a regression test for explicit `guildId` plus Discord `toolContext` with no channel filters.
  - confidence: 0.92
