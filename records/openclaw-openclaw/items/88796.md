## Summary

Keep this PR open: it fixes the reported missing-guild path in a plausible place, but the current patch also silently narrows existing explicit-guild Discord searches to the current channel and still lacks real Discord search proof.

## What This Changes

The PR changes Discord message search handling to accept `query`/`content`, infer `guildId` from channel context or provided channel ids, normalize channel targets, and add regression tests.

## Best Possible Solution

Preserve guild-wide behavior when `guildId` is explicit, use the session channel only to infer a missing `guildId`, keep the normalization/query-alias fixes, and add redacted live Discord proof before merge.

## Reproduction Assessment

Yes, from source: current `main` requires `guildId` for Discord `searchMessages`, and the linked issue includes live logs where the model invoked `search` with `query` but no `guildId`. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No: the runtime fallback and query alias are plausible, but the action-layer fallback is too broad because it changes explicit-guild searches with no channel filters. The safer fix is to use the current channel only when `guildId` is missing and preserve explicit filters exactly.
