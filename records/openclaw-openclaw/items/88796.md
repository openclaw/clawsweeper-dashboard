## Summary

Keep open: the PR addresses a real Discord search bug, but the current branch still introduces a merge-blocking scoping regression and the external real-behavior proof is still weaker than the gate requires.

## What This Changes

The PR makes Discord message search accept query/content aliases, infer a missing guildId from channel info, normalize Discord channel IDs, and adds focused Discord action/runtime tests.

## Best Possible Solution

Preserve guild-wide semantics when guildId is explicit, use the session channel only to infer a missing guildId, keep read-target authorization intact, and add redacted live Discord proof before merge.

## Reproduction Assessment

Yes. Current main source requires guildId for Discord searchMessages and the linked issue includes live logs where the model sends query without guildId and receives ToolInputError: guildId required; I did not run a live Discord repro in this read-only review.

## Solution Assessment

No, not as written. The channel-info fallback is a plausible fix layer, but the tool entry point should not inject the session channel when guildId is already explicit because that changes a guild-wide search into a channel-scoped search.

Full review comments:

- **[P1] Preserve guild-wide searches when guildId is explicit:** `extensions/discord/src/actions/handle-action.guild-admin.ts:442-444`
  - body: When `message.search` is called from a Discord session with an explicit `guildId` but no channel filters, this fallback injects `ctx.toolContext.currentChannelId`. The runtime then sends that value as `channelIds`, so a previously guild-wide search is silently narrowed to the current channel and can miss valid results from other allowed channels. Gate the fallback on `!guildId` and add the focused regression test.
