Keep open: the PR is a useful partial Discord search fix, but it still lets prefixed channel targets reach the Discord search `channel_id` payload, and the contributor proof is a local simulation rather than a live Discord/OpenClaw search.

## What This Changes

The PR changes Discord message search to infer a missing `guildId` from channel context, accept `query`/`content` aliases, and add runtime regression tests.

## Best Possible Solution

Normalize Discord search channel filters before both authorization and `searchMessagesDiscord` dispatch, assert that behavior in the runtime test, then require redacted live Discord/OpenClaw proof before merge.

## Reproduction Assessment

Yes from source inspection: PR head builds `channelIdList` from raw `channelId`/`channelIds` but passes that same list to `searchMessagesDiscord`, whose sender appends each value as `channel_id`. The new `channel:C1` test only checks guild lookup, not the final search payload.

## Solution Assessment

No. The PR is a plausible partial fix, but the maintainable fix is to normalize channel targets before every Discord API boundary, including the search filter payload.
