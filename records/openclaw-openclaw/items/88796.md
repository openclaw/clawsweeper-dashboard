## Summary

Keep open: the PR addresses a valid source-reproducible Discord search bug, but the current branch has a blocking search-scope regression because it injects the current session channel even when `guildId` was explicitly supplied for a guild-wide search. It also still needs real Discord behavior proof before merge.

## What This Changes

The PR makes Discord message search accept `query`/`content`, infer a missing `guildId` from channel info, normalize channel IDs, and adds regression coverage.

## Best Possible Solution

Preserve the existing explicit `guildId` guild-wide search path, use the session-channel fallback only when `guildId` and explicit channel filters are absent, then add focused regression coverage and redacted real Discord run proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and `v2026.6.1` require `guildId` for Discord search, and the linked issue includes live logs for the failing natural `query` call. I did not run a live Discord session in this read-only review.

## Solution Assessment

No, not as currently written: the runtime guild inference and channel ID normalization are acceptable, but the entry-point fallback must not narrow explicit guild-wide searches. The safer fix is to inject the session channel only when `guildId` is absent and no explicit channel filter exists.
