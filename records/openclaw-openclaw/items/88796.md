## Summary

Keep open: the PR addresses the source-reproducible Discord search missing-guildId path, but the current head can narrow existing explicit guildId-only searches to the current Discord channel and still lacks redacted live Discord bot/API proof.

## What This Changes

The PR changes Discord search action handling to accept `query` or `content`, infer missing `guildId` from channel/session context, normalize channel IDs, and add regression tests.

## Best Possible Solution

Keep the channel-info `guildId` fallback, make session-channel injection conditional on missing `guildId`, and require redacted live Discord search proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main requires `guildId` before any fallback, and PR head shows Discord session context can inject `channelId` into explicit `guildId`-only searches. I did not run a live Discord bot session in this read-only review.

## Solution Assessment

No as written: resolving missing `guildId` from channel info is an acceptable narrow fix, but the wrapper fallback must not alter explicit `guildId`-only searches. A longer-term `currentGuildId` context seam could be cleaner, but it is not required to repair this PR safely.
