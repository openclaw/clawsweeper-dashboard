## What This Changes

The PR makes Discord message search accept `query` or `content`, infer a missing `guildId` from normalized channel context, avoid injecting the session channel when explicit filters or guild scope exist, and adds Discord action/runtime regression tests.

## Best Possible Solution

Land this PR only after redacted live Discord guild-channel proof shows search succeeds without explicit `guildId` and preserves explicit channel filters, or after maintainers explicitly accept the live-proof gap.

## Reproduction Assessment

Yes. Current main and `v2026.6.1` require `guildId` for Discord search at the action entry point/runtime, so the linked query-only in-channel search fails before it can reach the Discord search API.

## Solution Assessment

Mostly yes. The PR stays inside the Discord plugin action/runtime boundary and reuses existing channel resolution and read-target authorization; the remaining issue is proof of the live Discord behavior, not a better code location.
