## Summary

Keep open: this targets a real Discord search bug and the prior changelog issue is gone, but the current patch can add the current session channel to searches that already supplied explicit `channelIds`, and live Discord/OpenClaw proof is still missing.

## What This Changes

The PR updates Discord search handling to infer a missing `guildId` from channel info, accept `query`/`content` aliases, normalize search channel filters, and add regression tests.

## Best Possible Solution

Preserve explicit search filters, use the session channel only when no explicit `channelId` or `channelIds` exists, keep the runtime guild inference/normalization, add the missing regression test, and then provide redacted live Discord proof.

## Solution Assessment

No, not as currently written. Runtime guild inference is the right boundary, but the tool entry point conflates session fallback with explicit `channelIds` filters and needs that scoped repair before merge.

Full review comments:

- **[P1] Do not append the session channel to explicit channelIds searches:** `extensions/discord/src/actions/handle-action.guild-admin.ts:437-441`
  - body: For existing calls that pass `guildId` plus `channelIds` from a Discord session, this fallback now also sets `channelId` to the current session channel. The runtime builds its search filter from `channelIds` plus `channelId`, so a request scoped to explicit channels will also search the current channel. Only use the session channel when neither `channelId` nor `channelIds` was provided, or use it for guild inference without adding it to the filter.
