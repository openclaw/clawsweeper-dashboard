## Summary

Keep open: the PR fixes part of the missing `guildId` path, but the current head still introduces a blocking scope regression for explicit guild-wide Discord searches and the supplied proof is not live Discord behavior proof.

## What This Changes

The PR changes Discord search action handling to infer missing `guildId` from channel context, accept `query`/`content` aliases, normalize channel IDs, and add regression tests.

## Best Possible Solution

Keep the runtime-side `guildId` inference, but only inject the session channel when `guildId`, `channelId`, and `channelIds` are all absent, then prove the fixed path in a redacted live Discord guild-channel search.

## Reproduction Assessment

Yes. From source inspection, `message.search` with `{ guildId, query }` in a Discord session now receives `currentChannelId`, and the runtime converts that into a `channel_id` search filter.

## Solution Assessment

No. The fallback direction is useful for omitted-`guildId` calls, but the best fix must preserve explicit guild-wide searches and then show live Discord proof.
