## Summary

Keep this PR open: it addresses a real Discord search bug, but the latest head still changes explicit `guildId` guild-wide searches into current-channel searches inside Discord sessions, and the supplied proof does not show a live Discord bot/API search after the patch.

## What This Changes

The PR makes Discord message search accept `query`/`content`, infer missing `guildId` from a channel or session channel, normalize channel IDs, and adds focused action/runtime tests.

## Best Possible Solution

Gate the session-channel fallback on both missing `guildId` and missing channel filters, keep explicit guild-wide search behavior intact, add the adapter regression test, and land only after redacted live Discord proof is supplied.

## Reproduction Assessment

Yes for source-level reproduction: current main and the latest release require `guildId` before Discord search can proceed, matching the linked live logs. The PR-introduced scope regression is also source-reproducible from the adapter fallback and runtime guild-wide contract, though I did not run live Discord.

## Solution Assessment

No as written: the omitted-`guildId` direction is right, but the adapter fallback is too broad because explicit `guildId` with no channel filters should remain guild-wide. The narrow repair is to use the session channel only when `guildId` and channel filters are absent.
