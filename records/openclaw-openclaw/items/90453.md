## Summary

Keep open: this is a useful candidate fix for the memory safe-reindex race, but the current patch is not merge-ready because it leases the vector/FTS query helpers while leaving earlier DB-backed search readiness reads outside the active-reader barrier.

## What This Changes

The PR adds memory-core reindex/read coordination, stable status snapshots during safe reindex, post-swap metadata rewrite, and race coverage for memory search/status.

## Best Possible Solution

Use one consistent reader lease over search readiness and query execution, or move the reindex switch so live readers never observe the temporary DB.

No as submitted: the owner boundary is right, but the patch protects the actual query helpers while leaving some DB-backed readiness reads outside the active-reader barrier.

- **[P1] Lease the readiness reads too:** `extensions/memory-core/src/memory/manager.ts:612`
  - body: `drainReindex()` only waits for a reindex that is already registered; because each `await` yields, another sync can start immediately after the drain resolves and before `hasIndexedContent()`, `refreshIndexIdentityDirty()`, or FTS availability checks run. Those reads are not counted in `activeIndexReaders`, so `runInPlaceReindex()` can still swap `this.db` to the temp DB underneath them and reproduce the false missing-metadata/empty-index path this PR is meant to close.
