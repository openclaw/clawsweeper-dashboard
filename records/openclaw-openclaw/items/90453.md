## Summary

Keep open: current main still has the safe-reindex temp-DB exposure in the live memory manager, and this PR adds a focused manager-level latch/snapshot with deterministic race coverage. I did not find a line-level correctness bug in the patch, but maintainers still need to accept the search-wait availability tradeoff and refresh the branch against newer memory-core main commits.

## What This Changes

The PR adds memory-core reindex latching, active-reader tracking, stable status snapshots, a post-swap metadata rewrite, and regression tests for safe full reindex/search/status concurrency.

## Best Possible Solution

Land one current-main-rebased manager-level mid-reindex guard that preserves true mismatch fail-closed behavior, coordinates with the open stale-handle work at https://github.com/openclaw/openclaw/pull/90727, and explicitly accepts the search-wait semantics.

## Reproduction Assessment

Yes from source inspection: current main can start async sync before search identity/query reads, safe reindex swaps this.db to a temp DB, and the tool wrapper fails closed on status identity. I did not run a live current-main reproduction, but the PR tests force the relevant temp-DB window.

## Solution Assessment

Yes for the mid-build safe-reindex race: the manager-level drain/snapshot approach is more maintainable than a tool-wrapper-only mitigation that preserves hits after a bad status read. The remaining decision is whether maintainers accept waiting through reindex and how to sequence this with https://github.com/openclaw/openclaw/pull/90727.
