## Summary

Keep open. Current main still uses the 30s watchdog and still emits the normal watchdog system warning, while the merged adjacent work fixed tool/lifecycle and reconnect false-idle cases without deciding the timeout/suppression UX this PR proposes.

## What This Changes

The PR raises the TUI streaming watchdog default from 30s to 120s, removes the normal user-visible watchdog warning, and updates watchdog tests to expect silent resets.

## Best Possible Solution

Have a maintainer decide the intended UX after the current reconnect/proof-of-life fixes: either land a rebased version with the required changelog entry, or revise toward a softer deduped warning/configurable threshold so genuine silent drops remain diagnosable without noisy false alarms.

## Reproduction Assessment

Yes. The current fake-timer tests reproduce the watchdog by sending a delta, advancing past `streamingWatchdogMs`, and asserting both the idle reset and `streaming watchdog` system message; the PR changes those assertions to expect no message.

## Solution Assessment

Unclear. The patch is a narrow implementation of the requested quieter behavior, but complete suppression may not be the best maintainer choice given open related requests for configuration, deduplication, and a less alarming warning.
