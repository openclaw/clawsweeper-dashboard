## Summary

Keep this PR open: the main FD-fan-out fix is well targeted and the earlier fallback/type-check findings appear addressed, but the native macOS watcher path can lose watch-driven indexing if the watched memory directory is deleted and recreated.

## What This Changes

The PR routes memory-core directory watch paths to native recursive `fs.watch` on macOS/Windows, keeps file and non-native-platform paths on chokidar, closes native watchers during manager teardown, and expands watcher tests.

## Best Possible Solution

Add native-watch reattachment or an equivalent parent/chokidar fallback for memory-root replacement, keep Linux on the existing chokidar path, and treat broader `sync.watch` default policy as separate work.

## Reproduction Assessment

Yes. The PR body provides a concrete current-main macOS gateway probe where one `memory_search` over a 12,391-file memory tree opens 12,392 REG FDs before the patch and 1 after; I did not rerun that live probe in this read-only review.

## Solution Assessment

No, not as-is. The backend split is the right narrow shape for the FD fan-out, but it should preserve watch coverage when the native-watched memory root is replaced before this becomes the default macOS path.

- **[P2] Handle memory root replacement for native watchers:** `extensions/memory-core/src/memory/manager-sync-ops.ts:509-512`
  - body: This switches the macOS memory directory from chokidar to native `fs.watch(dir, { recursive: true })`, but Node documents that on macOS the watcher remains bound to the original inode after the watched path is deleted and recreated. With `intervalMinutes` defaulting to `0`, `rm -rf memory && mkdir memory` can leave new files under `memory/` unindexed until restart. Please reattach or fall back when the watched root is replaced, or keep a parent/chokidar watcher for that case.
