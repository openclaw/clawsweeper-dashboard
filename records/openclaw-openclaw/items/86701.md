- `merge-risk: 🚨 session-state`: Changing memory watcher backends can suppress or delay memory reindexing when watcher coverage fails or falls back incorrectly.
- `merge-risk: 🚨 availability`: The non-native Node recursive watcher path can allocate O(tree) watches on Linux-like platforms and may hit watch/resource limits in large memory trees.

## Summary

Keep open: the macOS FD proof is strong and the latest commit restores watcher coverage after native watcher errors, but the patch still enables Node recursive `fs.watch` on Linux/AIX/IBMi where Node uses a JS recursive watcher that allocates per-path watches before OpenClaw's ignore predicate can prune the tree.

## What This Changes

The PR splits memory-core watch paths so directories use recursive `fs.watch` with chokidar fallback while file paths remain on chokidar, adds native watcher teardown, and expands watcher configuration tests.

## Best Possible Solution

Land the native watcher approach after gating non-native recursive platforms back to chokidar, or after adding a pruned recursive backend plus focused Linux proof; leave the broader `sync.watch` default policy to https://github.com/openclaw/openclaw/issues/71335.

## Solution Assessment

- **[P2] Gate recursive fs.watch to native platforms:** `extensions/memory-core/src/memory/manager-sync-ops.ts:494-496`
  - body: On Node 22.19, recursive `fs.watch` is native only on macOS and Windows; Linux/AIX/IBMi route through `internal/fs/recursive_watch`, which walks every descendant and calls `fs.watch` for each file or directory. Because this line sends all memory directory paths there and only applies `shouldIgnoreMemoryWatchPath` in the event callback, large ignored or non-md trees can still allocate O(tree) watches and hit inotify/watch limits. Keep those platforms on chokidar or add a pruned recursive backend before enabling this path.
