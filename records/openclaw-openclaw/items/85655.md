## Summary

Current main and the latest release already route active Memory Wiki bridge status/doctor/import calls through Gateway RPC, so the CLI sees the runtime memory plugin context instead of an empty standalone registry.

## What This Changes

Make Memory Wiki bridge status/import see public artifacts exported by the active memory plugin instead of reporting zero artifacts in CLI context.

## Best Possible Solution

Keep the shipped Gateway-RPC routing implementation and handle the still-open direct-fallback PR as superseded by that contract-safe fix.

## Reproduction Assessment

Yes. The report gives a concrete bridge-mode config and prior linked reports/PRs confirm the standalone CLI path could see an empty memory capability while the Gateway runtime saw artifacts; current main no longer uses that failing path for active bridge reads.

## Solution Assessment

Yes. Routing active bridge status/doctor/import through the Gateway is the maintainable fix because it uses the registered runtime memory plugin context instead of adding a memory-core-specific fallback that bypasses the plugin contract.
