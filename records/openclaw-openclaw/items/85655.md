## Summary

Keep open: current main and the latest release still route memory-wiki bridge status/import through the active memory public-artifact helper, which returns an empty list when the memory capability is absent. Open candidate PRs now cover the likely fix, but they are unmerged and still need maintainer boundary/proof review.

## What This Changes

The issue asks for `openclaw wiki status` and `openclaw wiki bridge import` to see active memory plugin public artifacts instead of reporting zero when direct memory-core listing succeeds.

## Best Possible Solution

Land a maintainer-approved active-memory public-artifact handoff through the public capability/loader lifecycle boundary with redacted live CLI proof, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level: current bridge status/import uses the active public-artifact helper, and that helper returns `[]` when no memory capability provider is registered. The closed later beta duplicate gives additional live-style evidence of the same host-helper versus active-helper split.

## Solution Assessment

Unclear as a final implementation path: the issue is valid, but maintainers need to choose between repairing the generic capability lifecycle and accepting a narrower fallback. A direct private memory-core bypass is not the safest default without that boundary decision.
