## Summary

Keep this open: current main has mitigations for slow Gateway handshakes and a supported service-wrapper path, but there is not enough evidence that it fixes the reported WSL2 TUI freeze on 2026.4.29. The service-repair part also appears to hinge on whether a manual `sg docker -c ...` systemd command should be treated as supported or external customization.

## What This Changes

The issue reports an OpenClaw 2026.4.29 TUI freeze on Windows 11/WSL2 during initial connection, plus service/status/doctor regressions around Linux group-switch wrappers for Docker orchestration.

## Best Possible Solution

Keep the issue open, first isolate the TUI freeze on Windows 11/WSL2 against current main, then either fix the TUI/Gateway startup path or split the service-wrapper concern into a narrower decision about supporting `sg docker` wrappers versus directing users to `openclaw gateway install --wrapper` or `OPENCLAW_SERVICE_REPAIR_POLICY=external`.

## Reproduction Assessment

No: the issue gives plausible WSL2 steps and #75806 corroborates the same 2026.4.29 symptom, but this review did not live-run a WSL2 TUI or a matching systemd unit on current main.

## Solution Assessment

Unclear: current main's handshake/readiness mitigation and documented wrapper API are good directions, but they do not conclusively solve the frozen TUI or prove the manual group-switch wrapper should be treated as canonical.
