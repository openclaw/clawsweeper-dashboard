## Summary

Keep open: current main has relevant TUI/Gateway responsiveness and service-wrapper mitigations, but they do not conclusively fix the reported Windows 11/WSL2 `connecting | idle` freeze on 2026.4.29. The service-repair part also needs maintainer judgment about whether manually embedded `sg docker -c ...` systemd commands should be supported or treated as external service customization.

## What This Changes

The issue requests fixing a Windows 11/WSL2 TUI freeze during initial Gateway handshake and preserving Docker group-switch service wrappers through status and doctor repair flows.

## Best Possible Solution

Keep the issue open and first reproduce the TUI freeze on Windows 11/WSL2 against both 2026.4.29 and current main. Once isolated, fix the narrow TUI/Gateway event-loop or handshake path if it still reproduces, and split the service-wrapper concern into either documented `--wrapper` guidance, `OPENCLAW_SERVICE_REPAIR_POLICY=external` guidance, or an explicit maintainer-approved enhancement for recognizing shell-embedded `sg docker` launchers.

## Reproduction Assessment

No: the issue gives plausible WSL2 steps and #75806 corroborates the same current-release TUI symptom, but there are no logs and no current-main WSL2 live run proving the freeze still reproduces or has been fixed. The best focused reproduction is to run a healthy Gateway and `openclaw tui` in WSL2 while checking input responsiveness, Ctrl+C exit, Gateway event-loop readiness, and `sessions.list` latency.

## Solution Assessment

Unclear: current main's readiness wait, handshake timeout plumbing, bounded session-list reads, and documented wrapper API are appropriate mitigations, but they do not prove the reported WSL2 deadlock is solved or that manual `sg docker -c ...` units should become a supported service shape. The safer path is to isolate the TUI failure first and handle the service-wrapper policy as a separate maintainer decision if needed.
