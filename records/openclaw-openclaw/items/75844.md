## Summary

Keep open: current main contains relevant TUI/Gateway/session-list and service-wrapper mitigations, but the WSL2/macOS TUI hang is still corroborated by recent comments and I could not prove that current main or v2026.5.2 fixes the startup deadlock or the raw `sg docker` service-wrapper audit path end to end.

## What This Changes

The issue reports a regression where `openclaw tui` freezes during initial Gateway handshake on WSL2/macOS, with related doctor/status problems around a Docker group-switch systemd wrapper.

## Best Possible Solution

Confirm the TUI startup hang on current main, then fix the narrow blocking path and split the raw Docker group-switch unit behavior into either documented executable-wrapper guidance or explicit support in service parsing/audit.

## Reproduction Assessment

No: the report and related comments give plausible WSL2/macOS symptoms, and source inspection shows the affected startup and service-audit paths, but there is no high-confidence current-main reproduction of the TUI deadlock.

## Solution Assessment

Unclear: the current readiness and sessions-list mitigations target plausible causes, but the best final fix depends on live reproduction and on whether maintainers want to support raw `sg docker -c ...` systemd units or require an executable `OPENCLAW_WRAPPER` path.
