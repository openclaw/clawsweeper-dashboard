## Summary

The latest head appears to address the previous literal RFC2544 blocker: the direct-upload sink now validates HTTP(S), runs the shared SSRF preflight before token lookup, applies the fake-IP DNS allowance only for non-literal hostnames, and keeps literal IP hosts on the default block policy. This PR remains useful because current main still forwards direct media URLs to QQ without a sink guard, but the maintainer/security-boundary labels and focused proof gaps mean it should stay open for maintainer review and normal merge gates.

## What This Changes

This PR adds QQBot `MediaApi.uploadMedia()` direct URL validation and focused tests for public HTTP(S), invalid schemes, SSRF-blocked targets, literal special-use IPs, and fake-IP DNS compatibility.

## Best Possible Solution

Land the sink-level HTTP(S) and SSRF preflight after maintainer security/compatibility review and required checks confirm the direct URL, fake-IP DNS, and literal-host split.

## Solution Assessment
