## Summary

Keep open: this is a useful QQBot security hardening PR, but it is not merge-ready because the current patch still narrows shipped direct media URL behavior and only provides test/mock proof rather than real behavior proof.

## What This Changes

The PR adds a QQBot `MediaApi.uploadMedia()` direct URL preflight using the SDK SSRF resolver and adds focused tests for direct URL uploads.

## Best Possible Solution

Keep the sink-level SSRF preflight, but preserve shipped public `http://` and `https://` direct URL behavior by default, or land HTTPS-only only with explicit maintainer/security acceptance plus docs, tests, and upgrade proof.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.5.22 send `opts.url` directly to the QQ media API after the outbound layer accepts both `http://` and `https://` remote sources. I did not run a live QQ upload.

## Solution Assessment

No as written: the `MediaApi.uploadMedia()` sink is the right ownership boundary, but the patch should preserve shipped public HTTP(S) behavior or get explicit maintainer/security acceptance for HTTPS-only behavior.
