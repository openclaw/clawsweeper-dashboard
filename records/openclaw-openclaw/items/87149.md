## Summary

Keep open. The QQBot media upload sink hardening is directionally useful, but the patch changes default direct URL upload compatibility without a maintainer-approved upgrade/config decision and has only mocked/unit proof so far.

## What This Changes

The PR adds a QQBot `MediaApi.uploadMedia()` direct-upload URL preflight using the existing SSRF policy and adds focused Vitest coverage for accepted, non-HTTPS, and policy-rejected URLs.

## Best Possible Solution

Land a maintainer-approved QQBot URL media policy that hardens direct uploads while explicitly handling existing `urlDirectUpload` users and the open `mediaAllowHosts` policy decision with focused upgrade tests.

## Reproduction Assessment

Yes by source inspection: current main forwards `opts.url` directly into the QQ upload body, and the PR’s mocked tests show rejected URLs failing before token lookup or API calls. I did not run a live QQ upload.

## Solution Assessment

Unclear as a merge shape. The upload sink is the right choke point for validation, but the proposed hard fail-closed behavior needs explicit maintainer approval or a config/upgrade path before it is the safest solution.
