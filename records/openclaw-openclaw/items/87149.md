## Summary

This PR is still useful because current main forwards QQBot direct media URLs without a sink preflight, but it is not ready to merge: the new guard preserves fake-IP DNS compatibility by also allowing literal 198.18.0.0/15 upload URLs, which keeps a special-use direct URL forwardable to QQ.

## What This Changes

The PR adds QQBot `MediaApi.uploadMedia()` direct URL validation plus focused regression tests for public HTTP(S), invalid schemes, SSRF-blocked targets, and fake-IP behavior.

## Best Possible Solution

Keep the sink-level HTTP(S) and SSRF preflight, but reject special-use literal URL hosts before applying the fake-IP DNS allowance to non-literal public hostnames, then update the tests to cover that split.

## Reproduction Assessment

Yes. Source inspection is enough: the PR passes `parsed.hostname` to the SSRF helper with `allowRfc2544BenchmarkRange: true`, and the added test expects a literal `https://198.18.0.42/...` URL to reach the QQ upload request body.

## Solution Assessment

No. The sink-level guard is the right place, but the implementation conflates fake-IP DNS answers with literal special-use URL hosts; the narrow fix is to reject literals first and keep the fake-IP allowance only for hostname resolution.
