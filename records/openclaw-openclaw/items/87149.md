- `merge-risk: 🚨 compatibility`: The PR adds stricter validation to an existing direct-upload input path, so existing public URL setups can fail during upgrade.

## Summary

Keep open. The sink-level SSRF guard is the right direction and the contributor supplied usable runtime proof, but the current patch has a merge-blocking compatibility/message-delivery risk because the default generic SSRF preflight can reject public direct-upload URLs in fake-IP or special-use DNS environments while the fallback path is QQ/Tencent allowlisted.

## What This Changes

The PR adds QQBot direct media URL validation in `MediaApi.uploadMedia()` and new regression tests for HTTP(S), invalid URL, and SSRF allow/block behavior.

## Best Possible Solution

Land a sink-level SSRF guard that preserves documented public HTTP(S) direct uploads, with tests for normal public URLs and fake-IP/special-use DNS behavior or an explicit maintainer-approved fail-closed decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `main` places `opts.url` into the QQ upload request body without a sink-level SSRF check before token lookup. I did not run a live QQ upload, but the current code path is direct and the PR proof exercises the changed sink.

## Solution Assessment

No. The sink-level guard is the maintainable place to fix this, but using the default generic DNS policy as-is can break public direct uploads in fake-IP DNS environments unless maintainers explicitly choose that fail-closed behavior.
