## Summary

Keep open. The sink-level QQBot media URL guard is the right direction and the contributor has now provided substantive runtime proof, but the PR still over-broadly allows literal RFC 2544 fake-IP URLs to be forwarded to QQ instead of limiting that exception to DNS answers for public hostnames.

## What This Changes

The PR adds QQBot direct media URL validation in `MediaApi.uploadMedia()` and adds focused regression tests for allowed public HTTP(S), invalid/non-HTTP, SSRF-blocked, and fake-IP direct URL cases.

## Best Possible Solution

Keep the sink-level guard, but block literal private/internal/special-use URL hosts and allow the RFC 2544 exception only for resolver answers from non-IP public hostnames before considering the security/compatibility tradeoff ready to merge.

## Reproduction Assessment

## Solution Assessment

No. Guarding at `MediaApi.uploadMedia()` is the right sink, but the current implementation should preserve fake-IP DNS compatibility without allowing literal RFC 2544 special-use URLs to be forwarded to QQ.
