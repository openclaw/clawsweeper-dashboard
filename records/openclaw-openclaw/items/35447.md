## Summary

Keep open: current main still has global cron reads and mutations, while the linked implementation PRs are closed unmerged and the remaining related issues only cover overlapping or narrower parts of the same isolation problem.

## What This Changes

Request per-agent/session isolation for cron job listing, update, remove, and manual run operations, with an admin override path.

## Best Possible Solution

Add a reviewed cron authorization seam that derives caller agent/session from trusted gateway or agent-tool context, applies it consistently to list/listPage/runs/update/remove/run/enqueueRun, preserves legacy/admin behavior by explicit policy, and includes regression tests plus user-facing docs/changelog.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create jobs with different agentId/sessionKey values, then current cron.list/listPage returns all matching enabled/query filters and cron.update/remove/run act by id without ownership checks. I did not execute it because this review must keep the checkout read-only.

## Solution Assessment

Unclear. The suggested service-layer filtering is directionally right, but the best fix needs maintainer approval for the trusted caller identity source, admin bypass semantics, and how this relates to the broader per-agent storage and per-session tool-policy threads.

## Security Review

Status: needs_attention

Summary: Current main still exposes a cross-session cron visibility and mutation risk wherever multiple owner-capable sessions share the scheduler.

Concerns:

- **[medium] Cron mutations are not ownership-gated:** `src/cron/service/ops.ts:350`
  - body: The service and gateway paths update, remove, and run jobs by id without comparing the stored agentId/sessionKey to the caller, so a caller with cron access can target another session's job.
  - confidence: 0.91
- **[medium] Cron listings are globally visible:** `src/gateway/server-methods/cron.ts:203`
  - body: cron.list/listPage and the all-runs job-name map read the global job set without caller filtering, exposing job names and metadata across sessions.
  - confidence: 0.88
