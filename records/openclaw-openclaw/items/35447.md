## Summary

Keep open. Current main still stamps cron jobs with agent/session ownership metadata but does not enforce that metadata in cron list, update, remove, run, or queued manual-run paths; the linked implementation PRs are closed unmerged and the related open items are narrower or broader follow-ups rather than replacements.

## What This Changes

Add per-agent/session isolation for cron job visibility and mutations, with an admin override path for trusted operators.

## Best Possible Solution

Implement a server-side cron caller-context contract: derive trusted agentId/sessionKey and admin override from gateway/agent context, enforce it in CronService list/listPage/update/remove/run/enqueueRun, define legacy-job behavior, and add service, gateway, protocol, and cron-tool regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create cron jobs with different agentId/sessionKey values, then call cron.list or cron.update/remove/run through the current service or gateway path; those signatures still provide no caller context, so the operations are global.

## Solution Assessment

Yes, the requested server-side direction is the right boundary. The maintainable fix should derive trusted caller identity in gateway/agent context and pass it into the cron service rather than treating client-provided ownership filters as authorization.
