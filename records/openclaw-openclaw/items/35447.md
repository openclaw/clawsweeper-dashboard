## Summary

Keep open: current main still has global cron reads and id-only mutations, while the two implementation PRs linked from the discussion are closed unmerged and the related open issues do not fully supersede this combined visibility/mutation isolation request.

## What This Changes

Request adds per-agent/session authorization so cron list, update, remove, and run only operate on caller-owned jobs, with an explicit admin bypass.

## Best Possible Solution

Add a reviewed cron authorization seam that derives caller agent/session from trusted gateway or agent-tool context, applies it consistently to list/listPage/runs/update/remove/run/enqueueRun, preserves legacy/admin behavior by explicit policy, and adds regression tests plus docs/changelog.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: create two jobs with different agentId/sessionKey values, then list/listPage returns global jobs and update/remove/enqueueRun act by id without caller ownership checks.

## Solution Assessment

Unclear. Service-layer filtering is directionally right, but the maintainable fix needs maintainer approval for trusted caller identity, admin bypass policy, and how this intersects with broader per-session tool policy work.
