## Summary

Keep open. Current main still stores cron ownership metadata but does not thread trusted caller context through the cron service, gateway protocol, gateway handlers, or cron tool mutations, so the reported cross-agent/session visibility and mutation problem remains. The related implementation PRs are closed unmerged; #49175 is narrower run-history leakage, #49183 is broader policy discussion, and #26370 overlaps but is storage-focused rather than a replacement for this caller-context contract.

## What This Changes

Add per-agent/session isolation to cron so job listing, updates, removals, manual runs, and related visibility/mutations are scoped to the caller unless a trusted admin override is used.

## Best Possible Solution

Implement a core cron caller-context contract: derive trusted caller identity and admin capability at the gateway boundary, thread it through the protocol and `CronService`, scope `list`/`listPage`, authorize `update`/`remove`/`run`/`enqueueRun`, define legacy-job behavior, and cover the service, gateway, and cron-tool paths with focused regression tests. Keep #49175 as the run-history name-leak follow-up and #49183 as the broader tool-policy design discussion.
