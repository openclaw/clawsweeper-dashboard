## Summary

Keep open. Current main still injects a fresh `Current time:` line into heartbeat, isolated cron, session-reset, and post-compaction turn content, so the requested cache-stability cleanup remains valid and is not fully covered by the broader cache-affinity issue.

## What This Changes

The issue requests moving exact wall-clock current-time text out of cache-sensitive recurring prompt prefixes while keeping stable timezone context.

## Best Possible Solution

Land a focused cleanup that removes or delays exact wall-clock injection in the named per-turn payload builders while preserving timezone-only system context and any date substitution behavior that agents still need.

## Reproduction Assessment

Yes. Source inspection on current main shows the heartbeat body, isolated cron `commandBody`, session-reset prompt, and post-compaction refresh block still receive a freshly formatted `Current time:` line, with tests asserting that behavior.

## Solution Assessment

Yes. Targeting these prompt builders is the narrowest maintainable path; the broader cross-run cache-affinity work in #62809 should remain separate.

Fix https://github.com/openclaw/openclaw/issues/71973 by moving volatile exact wall-clock `Current time:` injection out of cache-sensitive heartbeat, isolated cron, session-reset, and post-compaction prompt prefixes. Keep stable timezone metadata in `src/agents/system-prompt.ts` and preserve intentional date substitution semantics, but stop appending a fresh exact-time line into `src/infra/heartbeat-runner.ts`, `src/cron/isolated-agent/run.ts`, `src/auto-reply/reply/session-reset-prompt.ts`, and `src/auto-reply/reply/post-compaction-context.ts` unless the replacement is late-bound and does not disturb the stable prompt prefix. Update existing regression tests that currently assert `Current time:` in those payloads. Do not add a new config option, provider-specific policy, or the broader cache-affinity work from #62809. Add a single-line changelog entry if the behavior change is user-visible.
