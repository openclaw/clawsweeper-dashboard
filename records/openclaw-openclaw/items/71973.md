## Summary

Keep open: current main and v2026.6.10 still route volatile `Current time` / `Reference UTC` text into heartbeat, isolated cron, post-compaction, reset, and memory-flush prompt content, so the central cache-stability concern is still live and needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests moving exact wall-clock time out of cache-sensitive prompt prefixes while preserving stable timezone context.

## Best Possible Solution

Define one model-visible timestamp and cache-boundary contract, then update the shared current-time helper, heartbeat, cron, reset, memory-flush, post-compaction callers, docs, and regression coverage together.

## Reproduction Assessment

Yes at source level: current main still routes exact `Current time` / `Reference UTC` blocks into heartbeat, isolated cron, session reset, memory flush, and post-compaction model-bound prompt text. I did not run a live provider cache experiment.

## Solution Assessment

Unclear. Moving exact time later is plausible, but maintainers need to choose whether exact time belongs in late runtime metadata, user message text, `session_status`, or a separate trigger/current timestamp contract.
