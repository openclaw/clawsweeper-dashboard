## Summary

Keep open. Current main still generates and injects an exact wall-clock `Current time:` line into the reported heartbeat, isolated cron, post-compaction refresh, and bare reset paths; the stable system prompt is already timezone-only, but that does not resolve the narrower per-turn prompt-placement concern. The broader #62809 cache-affinity issue is related but does not cover all of these timestamp injection surfaces.

## What This Changes

Request to reduce prompt-cache churn by moving exact wall-clock `Current time:` lines out of cache-sensitive heartbeat, cron, post-compaction, and reset prompt content while preserving stable timezone metadata.

## Best Possible Solution

Make a provider-aware prompt composition change that keeps timezone-only context in the stable system prompt, includes exact wall-clock time only on turns that need it, and places that volatile value below provider cache checkpoints or in late-bound runtime/event content with regression coverage for heartbeat, isolated cron, post-compaction refresh, session reset, and OpenAI/Anthropic-family payload shaping.

## Reproduction Assessment

Yes. Source and test inspection on current main deterministically follows the `Current time:` helper into the reported heartbeat, isolated cron, post-compaction, and reset paths; tests currently assert that those prompts contain the volatile line. Tests were not executed because this review was read-only.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right for cache efficiency, but the narrow maintainable fix needs prompt/cache owner judgment about which runtime turns still require exact time and where each provider can receive volatile content without harming cache reuse.
