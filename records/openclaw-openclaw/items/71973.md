## Summary

Keep open. Current main still emits an exact wall-clock `Current time:` line through the reported heartbeat, isolated cron, post-compaction refresh, and session reset paths; the broader #62809 cache-affinity issue does not supersede this narrower timestamp-placement concern.

## What This Changes

The issue requests moving exact wall-clock current-time injection out of cache-sensitive prompt prefixes while keeping stable timezone metadata available.

## Best Possible Solution

Make a provider-aware prompt composition change that preserves timezone-only stable system context, keeps exact wall-clock time only on turns that need it, and places that volatile value below provider cache checkpoints or in late-bound runtime/event content with regression coverage for heartbeat, isolated cron, post-compaction refresh, session reset, and OpenAI/Anthropic-family payload shaping.

## Reproduction Assessment

Yes. A read-only source and test inspection on current main follows the helper to the heartbeat, isolated cron, post-compaction refresh, and session reset paths, and existing tests assert `Current time:` is present there.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely correct, but the maintainable solution needs prompt/cache owner judgment about which runtime turns still require exact time and where each provider can receive volatile content without harming cache reuse.
