## Summary

Keep open: current main and v2026.6.8 still build exact wall-clock `Current time` / `Reference UTC` text into heartbeat, isolated cron, session-reset, and post-compaction model-bound prompt content. The merged fixes around stale timestamps and cron cache affinity address adjacent problems, but they do not move volatile time out of the cache-sensitive prompt surface.

## What This Changes

Move exact per-turn current-time text out of cache-sensitive prompt prefixes for heartbeat, cron, and post-compaction flows while preserving stable timezone context.

## Best Possible Solution

Define one model-visible timestamp and cache-boundary contract, then update the shared current-time helper, heartbeat, cron, reset, post-compaction callers, docs, and regression tests together.

## Reproduction Assessment

Yes, at source level. Current main still routes exact `Current time` / `Reference UTC` blocks into heartbeat, cron, reset, and post-compaction model-bound prompt text; I did not run a live provider cache experiment.

## Solution Assessment

Unclear. Moving exact time later is plausible, but the best fix needs a maintainer decision about whether exact time belongs in late runtime metadata, user message text, `session_status`, or a separate trigger/current timestamp contract.
