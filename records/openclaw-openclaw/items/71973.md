## Summary

Keep open: current `main` and the latest release still build exact `Current time` / `Reference UTC` text into heartbeat, isolated cron, session-reset, post-compaction, and memory-flush prompt content. Related fixes improved stale timestamps and cron cache affinity, but they did not move volatile wall-clock time out of the cache-sensitive model-bound surface.

## What This Changes

The issue asks OpenClaw to keep volatile exact wall-clock time out of cache-sensitive prompt prefixes for heartbeat, cron, and post-compaction automation turns.

## Best Possible Solution

Define one model-visible timestamp and cache-boundary contract, then update the shared current-time helper, heartbeat, cron, reset, memory-flush, post-compaction callers, docs, and regression tests together.

## Reproduction Assessment

Yes, at source level. Current main still routes exact `Current time` / `Reference UTC` blocks into heartbeat, isolated cron, session reset, memory flush, and post-compaction model-bound prompt text; I did not run a live provider cache experiment.

## Solution Assessment

Unclear. Moving exact time later is plausible, but maintainers need to choose whether exact time belongs in late runtime metadata, user message text, `session_status`, or a separate trigger/current timestamp contract.
