## Summary

Keep open: current `main` still routes exact wall-clock `Current time` / `Reference UTC` text through heartbeat, cron/system-event, and post-compaction prompt bodies, so the cache-sensitive prompt-placement cleanup remains valid and is not covered by the merged UTC-format PR or the closed cron-affinity issue.

## What This Changes

Request to reduce prompt-cache churn by moving exact per-turn wall-clock time out of cache-sensitive heartbeat, cron/system-event, and post-compaction prompt prefixes while keeping stable timezone awareness.

## Best Possible Solution

Make the stable timezone/system context remain cacheable, then move exact wall-clock time into a late-bound event/runtime segment only for turns that need it, with regression coverage for heartbeat, system-event/cron, and post-compaction prompt assembly.

## Reproduction Assessment

Yes, source inspection reproduces the reported placement: current `main` still appends exact `Current time` / `Reference UTC` text into heartbeat, cron/system-event, and post-compaction prompt bodies. I did not run a live cache-hit experiment, so the cache-cost impact remains inferred from the prompt construction path.

## Solution Assessment

Unclear until maintainers choose the cache boundary: the requested direction is plausible and well scoped, but the best fix must preserve time awareness for automation turns while avoiding volatile bytes in stable prefixes.
