## Summary

Keep open. Current main still matches the report: exact wall-clock `Current time:` text is generated and injected into heartbeat, isolated cron, post-compaction refresh, and session-reset prompt paths, while the stable system prompt only carries timezone metadata. The related #62809 is broader cache-affinity work and does not supersede this narrower timestamp-placement issue.

## What This Changes

The issue asks OpenClaw to keep stable timezone metadata cacheable while moving exact per-turn wall-clock time out of cache-sensitive prompt prefixes for recurring automation and context-refresh turns.

## Best Possible Solution

Make a provider-aware prompt composition change that keeps timezone metadata in stable system context, preserves exact wall-clock time only for turns that require it, and ensures volatile timestamps do not land in repeated cached prefixes or provider cache checkpoints. Regression coverage should cover heartbeat, isolated cron, post-compaction refresh, session reset, and Anthropic/OpenAI-family payload shaping.

## Reproduction Assessment

Yes. A read-only source check on current main follows the reported paths and finds `Current time:` injected into heartbeat, isolated cron, post-compaction refresh, and reset prompts, with existing tests asserting those lines.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right, but deciding which runtime turns still need exact time and where to place it relative to provider cache boundaries needs prompt/cache owner design.
