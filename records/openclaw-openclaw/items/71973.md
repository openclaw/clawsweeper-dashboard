## Summary

Keep open. Current main still appends exact wall-clock `Current time:` lines into heartbeat, isolated cron, post-compaction refresh, and bare reset prompt bodies, while the adjacent open #62809 does not replace this timestamp-placement question.

## What This Changes

Move exact wall-clock time out of cache-sensitive prompt prefixes for heartbeat, cron, automation, and post-compaction turns while preserving stable timezone awareness.

## Best Possible Solution

Define a provider-aware runtime timestamp contract that keeps exact wall-clock time below cache checkpoints or in late-bound runtime/event content only when needed, while preserving date grounding for startup, daily-memory, heartbeat, and cron workflows.

## Reproduction Assessment

Yes. Source inspection on current main deterministically follows `resolveCronStyleNow()` into heartbeat, isolated cron, post-compaction refresh, and bare reset prompt paths, and existing focused tests assert `Current time:` in those prompts; tests were not run because this review was read-only.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right for cache efficiency, but the maintainable solution needs prompt/cache owner judgment about which runtime turns still require exact time and where each provider can receive volatile content without reducing prefix-cache reuse.
