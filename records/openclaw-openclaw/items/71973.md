## Summary

Keep open. Current main still matches the report: exact wall-clock `Current time:` text is generated and injected into heartbeat, isolated cron, post-compaction refresh, and session reset prompt paths, while the stable system prompt keeps only timezone metadata. The related #62809 is broader OpenAI cron cache-affinity work and does not supersede this narrower timestamp-placement issue.

## What This Changes

The issue requests moving exact wall-clock time injection out of cache-sensitive prompt prefixes for heartbeat, cron, and post-compaction automation turns.

## Best Possible Solution

Make a provider-aware prompt composition change that preserves stable timezone metadata in cacheable system context, keeps exact wall-clock time only on turns that require it, and places volatile timestamps below provider cache checkpoints or in late-bound event/runtime content. Regression coverage should cover heartbeat, isolated cron, post-compaction refresh, session reset, and Anthropic/OpenAI-family payload shaping.

## Reproduction Assessment

Yes. A read-only source reproduction follows the current-time helper into heartbeat, isolated cron, post-compaction refresh, and session reset paths, with tests asserting those `Current time:` lines on current main.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely correct, but the best implementation needs prompt/cache owner design because exact time is intentionally used in several automation and startup paths.
