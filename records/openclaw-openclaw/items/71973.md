## Summary

Keep open. Current main still appends exact wall-clock `Current time:` lines into heartbeat, isolated cron, post-compaction refresh, and bare reset prompt bodies, so the narrower cache-placement concern remains unresolved; related #62809 overlaps cron cache affinity but does not cover all timestamp-injection surfaces.

## What This Changes

Request moving exact wall-clock current-time injection out of cache-sensitive heartbeat, cron, and post-compaction prompt prefixes while retaining stable timezone metadata.

## Best Possible Solution

Introduce provider-aware prompt composition that keeps exact wall-clock time below cache checkpoints or in late-bound runtime/event content only when needed, while preserving a non-cache-busting way for startup and daily-memory flows to obtain the current date.

## Reproduction Assessment

Yes. Source inspection on current main deterministically follows `resolveCronStyleNow()` into the heartbeat, isolated cron, post-compaction refresh, and bare reset prompt paths, and existing focused tests assert those prompts contain `Current time:`; tests were not run because this review was read-only.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right for cache efficiency, but the maintainable solution needs prompt/cache owner judgment about which runtime turns still require exact time and where each provider can receive volatile content without reducing prefix-cache reuse.
