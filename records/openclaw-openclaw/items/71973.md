## Summary

Keep open. Current main still injects a fresh `Current time:` line into heartbeat, isolated cron, and post-compaction turn payloads, so the narrow cache-stability cleanup requested here remains valid even though the stable system prompt is already timezone-only.

## What This Changes

Request moving exact wall-clock current-time lines out of cache-sensitive heartbeat, cron, and post-compaction prompt prefixes while keeping stable timezone context.

## Best Possible Solution

Update the affected per-turn payload builders so exact wall-clock time is omitted, delayed, or tool-provided unless required, while keeping timezone-only system context and preserving date substitution semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows the heartbeat `Body`, isolated cron `commandBody`, and post-compaction refresh block still receive a freshly formatted `Current time:` line, with committed tests asserting that behavior.

## Solution Assessment

Yes. Targeting these per-turn prompt builders is the narrowest maintainable fix because the system-prompt cache boundary and `session_status` exact-time path already exist; provider cache machinery does not need to change for this issue.

Fix https://github.com/openclaw/openclaw/issues/71973 by moving volatile exact `Current time:` injection out of cache-sensitive heartbeat, isolated cron, and post-compaction prompt prefixes. Keep the stable system prompt timezone-only contract, preserve `YYYY-MM-DD` date substitution and session reset semantics, and use `session_status` or a late/gated event-local location for exact time when a turn truly needs it. Update focused tests that currently assert the volatile line in heartbeat, isolated cron, and post-compaction payloads; add regression coverage proving cache-sensitive prefixes stay stable across different `nowMs` values.
