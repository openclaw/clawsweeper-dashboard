## Summary

Keep open. Current main still builds an exact wall-clock `Current time:` line and places it into the heartbeat, isolated cron, post-compaction refresh, and bare reset prompt paths; the existing stable system-prompt timezone work does not resolve this narrower per-turn cache-placement concern. The related #62809 cache-affinity issue overlaps the cron/cache theme, but it does not cover all of these timestamp injection surfaces.

## What This Changes

The issue asks OpenClaw to improve prompt-cache reuse by keeping volatile wall-clock time out of cache-sensitive prompt prefixes while preserving stable timezone context.

## Best Possible Solution

Make a provider-aware prompt composition change that keeps timezone-only context in the stable system prompt, includes exact wall-clock time only on turns that need it, and places that volatile value below provider cache checkpoints or in late-bound runtime/event content with regression coverage for heartbeat, isolated cron, post-compaction refresh, session reset, and OpenAI/Anthropic-family payload shaping.

## Reproduction Assessment

Yes. Source inspection on current main deterministically follows `resolveCronStyleNow()` into the heartbeat, isolated cron, post-compaction, and bare reset paths, and the existing tests assert those prompts contain the volatile `Current time:` line. Tests were not executed because this review was read-only.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right for cache efficiency, but the maintainable solution needs prompt/cache owner judgment about which runtime turns still require exact time and where each provider can receive volatile content without reducing prefix-cache reuse.
