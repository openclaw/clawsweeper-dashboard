## Summary

This issue should stay open: current main still adds exact per-run wall-clock text to several automation and refresh prompt bodies, while the cache-stable system prompt only carries timezone metadata and points agents to `session_status`. The adjacent cache-affinity issue and stale-timestamp PR cover related but different problems, so they do not supersede this volatile-prefix cleanup.

## What This Changes

Move exact wall-clock `Current time:` injection out of cache-sensitive prompt prefixes while preserving stable timezone context for automation and refresh turns.

## Best Possible Solution

Move exact wall-clock values to `session_status` or another late, turn-specific surface only where required, keep timezone-only system context cacheable, and leave broader cross-run provider affinity to https://github.com/openclaw/openclaw/issues/62809.

## Reproduction Assessment

Yes. Source inspection on current main shows the exact-time helper is still called from heartbeat, isolated cron, post-compaction, reset, and memory flush prompt builders, and tests still assert those volatile lines in the generated bodies.

## Solution Assessment

Yes. The narrow maintainable direction is to change the shared helper contract and named prompt builders so exact time is late or on-demand, while preserving stable timezone context and not treating the broader cron cache-affinity issue as the same fix.
