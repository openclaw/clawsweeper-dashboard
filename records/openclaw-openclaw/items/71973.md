## Summary

Keep open. Current main still injects a fresh `Current time:` line into the heartbeat body, isolated cron prompt body, and post-compaction refresh block, while docs only prove the stable system prompt was already cleaned up.

## What This Changes

Move volatile exact `Current time:` injection for heartbeat, cron, and post-compaction turns out of cache-sensitive prompt prefixes while preserving stable timezone/date guidance.

## Best Possible Solution

Target the per-turn heartbeat, isolated cron, and post-compaction payload builders so exact wall-clock time is omitted, delayed, or tool-provided unless a turn truly needs it, while keeping the stable system prompt timezone-only contract.

## Reproduction Assessment

Yes. Source inspection and committed tests show the named flows still put a freshly formatted `Current time:` line into the actual agent turn payloads.

## Solution Assessment

Yes, the requested direction is the right narrow fix: current main already has a system-prompt cache boundary, so changing these owner-area per-turn call sites is safer than reworking core prompt-cache machinery.

## Repair Work Prompt

Fix openclaw/openclaw#71973 by reducing volatile exact wall-clock `Current time:` injection in heartbeat, isolated cron, and post-compaction agent-turn payloads. Keep the existing stable system prompt timezone-only behavior. Scope the change to `src/agents/current-time.ts`, `src/infra/heartbeat-runner.ts`, `src/cron/isolated-agent/run.ts`, `src/auto-reply/reply/post-compaction-context.ts`, and their focused tests/docs. Preserve `YYYY-MM-DD` date substitution for post-compaction and any behavior needed for daily memory/startup flows, but avoid putting minute-level wall-clock text near the front of recurring cache-sensitive prompts unless the turn explicitly requires current time. Do not implement the broader cross-run `prompt_cache_key` affinity requested in #62809. Update tests that currently assert exact `Current time:` injection, update date/time or prompt-caching docs if behavior changes, and add a user-facing changelog entry if the change is visible as a perf/fix behavior change.
