## Summary

Keep open. Current main still appends an exact wall-clock `Current time:` line into heartbeat, isolated cron, post-compaction refresh, and bare reset prompts; the stable system prompt is timezone-only, but that does not resolve the narrower per-turn cache-placement concern. The related #62809 report is still open and mainly covers isolated-cron cache affinity, so it does not supersede all timestamp-injection surfaces here.

## What This Changes

Optimize prompt composition so exact wall-clock time is not injected into cache-sensitive prompt prefixes for heartbeat, cron, and post-compaction automation turns.

## Best Possible Solution

Make a provider-aware prompt composition change that keeps timezone-only context in the stable system prompt, includes exact wall-clock time only for turns that need it, and places that volatile value below provider cache checkpoints or in late-bound runtime/event content with coverage for heartbeat, isolated cron, post-compaction refresh, session reset, and OpenAI/Anthropic-family payload shaping.

## Reproduction Assessment

Yes. Source inspection on current main deterministically follows `resolveCronStyleNow()` into heartbeat, isolated cron, post-compaction refresh, and bare reset prompts, and existing tests assert those prompts contain `Current time:`. Tests were not executed because this review was required to stay read-only.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right for cache efficiency, but the narrow maintainable solution needs prompt/cache owner judgment about which runtime turns still require exact time and where each provider can receive volatile content without harming cache reuse.
