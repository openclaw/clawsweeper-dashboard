## Summary

Keep this issue open. Current main still starts Feishu accounts through the gateway's concurrent per-account startup path, and the existing sequential Feishu preflight only covers the aggregate no-accountId monitor path. The linked implementation PRs are still the right place for maintainer review rather than closing this before a fix lands.

## What This Changes

The issue requests serializing or staggering Feishu startup bot-info probes so multiple configured accounts do not concurrently hit Feishu token and bot-info APIs during gateway startup.

## Best Possible Solution

Keep the fix inside the Feishu plugin: add serialization or bounded throttling for startup bot-identity probes across concurrent gateway startAccount calls, preserve degraded/background recovery behavior, and add regression coverage for the actual per-account gateway startup path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure multiple Feishu accounts, start the Feishu channel through gateway whole-channel startup, and observe that gateway Promise.all can enter per-account monitor startup concurrently; live Feishu credentials are needed to reproduce the external API timeout itself.

## Solution Assessment

Yes, the narrow maintainable direction is Feishu-owned serialization or throttling around startup identity probes rather than a core-wide channel startup policy change. PR #63481 is the closest direct fix candidate, but it should be reconciled with PR #67466 before merge.
