## Summary

Keep open. Current main still fans out gateway account startup concurrently, and Feishu's per-account startup path bypasses the aggregate sequential preflight, so the reported bot-info probe burst remains plausible; the linked open PRs are implementation candidates, not a reason to close before merge.

## What This Changes

The issue reports that multiple Feishu accounts start bot identity and tenant-token probes concurrently, causing one startup bot-info probe to time out and recover later by background retry.

## Best Possible Solution

Feishu should centralize startup identity-probe throttling so both aggregate monitor startup and gateway per-account startAccount calls share the same protection, with regression coverage for the gateway fan-out path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure multiple Feishu accounts and start the Feishu channel through gateway whole-channel startup, which enters concurrent startAccount calls that each fetch bot identity directly; live credentials are only needed to observe Feishu's actual timeout.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin around startup bot-identity probing, while core-wide serialization of channel account startup would be broader and could slow unrelated channels.
