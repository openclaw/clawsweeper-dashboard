## Summary

Keep this PR open: the earlier missing type import is fixed, but the patch now risks killing a valid Chrome launch by turning the HTTP discovery gate into a one-shot HTTP+WebSocket diagnostic gate before the existing post-launch readiness wait can run. The supplied proof is also insufficient because it exercises copied decision logic, not the patched OpenClaw browser launch path.

## What This Changes

This PR changes `launchOpenClawChrome` to run `diagnoseChromeCdp` after launch polling, use that result for readiness, and reuse the same diagnostic result in failure text.

## Best Possible Solution

Keep the single-source diagnostic idea, but preserve the documented two-stage startup contract: avoid contradictory failure text without killing HTTP-discovered launches before the configured CDP WebSocket readiness window, then add focused coverage and real launch proof.

## Reproduction Assessment

Yes at source level: current main can decide launch failure from `isChromeReachable` while formatting the thrown message from `diagnoseChromeCdp`, and the linked report gives repeated cold-start symptoms. I did not live-reproduce the intermittent macOS race.

## Solution Assessment

No: the direction is right, but this implementation is not the best mergeable shape because it requires one immediate WebSocket diagnostic after the first HTTP discovery success and bypasses the existing post-launch readiness budget.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the post-launch CDP readiness window:** `extensions/browser/src/browser/chrome.ts:552-557`
  - body: The diagnostic now runs once immediately after an HTTP-only polling loop that can break on the first `/json/version` success. The documented contract gives WebSocket readiness a separate `localCdpReadyTimeoutMs` window after launch, so an HTTP-ready/WS-warming Chrome can now hit `!finalDiagnostic?.ok`, get SIGKILLed, and never reach `waitForCdpReadyAfterLaunch`. Keep the contradiction fix, but preserve the two-stage wait by only using the diagnostic for final failure text/ok override or by polling the diagnostic under the right readiness budget.
  - confidence: 0.9
