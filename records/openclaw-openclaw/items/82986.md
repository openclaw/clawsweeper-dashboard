## Summary

Keep open: the fix direction matches the source-reproducible browser launch bug, but this PR is not mergeable because the new `ChromeCdpDiagnostic` annotation is not imported and the supplied proof only runs a standalone logic script, not the changed OpenClaw browser launch path.

## What This Changes

The PR changes `launchOpenClawChrome` to run one `diagnoseChromeCdp` check after launch polling and use that same result for both readiness and failure text.

## Best Possible Solution

Keep the single-diagnostic readiness approach, import or derive the diagnostic type correctly, and verify the actual browser launch path with redacted real output before merge.

## Reproduction Assessment

Yes at source level: current main can make the launch decision from `isChromeReachable` while formatting the error from `diagnoseChromeCdp`, and the related report gives repeated macOS cold-start symptoms. I did not live-reproduce the intermittent macOS launch race here.

## Solution Assessment

The proposed direction is the right narrow fix, but this implementation is not yet the best mergeable form because it misses a local type import and lacks proof from the actual changed OpenClaw path.
