## Summary

Keep open. Current main appears to have fixed the accidental WhatsApp allowlist pairing-reply leak, but the requested denied-inbound hook and explicit suppression/policy surface are still absent and need privacy-sensitive maintainer design.

## What This Changes

The issue requests a denied-inbound hook that can suppress default outbound behavior and a per-channel pairing-reply suppression option for unauthorized messages.

## Best Possible Solution

Design a generic, privacy-minimized channel access-denial extension point with typed reason/default-action data and a clear suppress-or-replace contract; add config only if it fits that policy model.

## Reproduction Assessment

Not applicable as a bug reproduction for the remaining request. Source and docs inspection on current main show no unauthorized/access-denied hook and no suppression or denial-response config option, while the earlier WhatsApp allowlist pairing-reply leak is covered by current source and tests.

## Solution Assessment

Unclear until maintainers choose the API and privacy model. Current main takes the right direction for the leak by making allowlist denial silent, but a generic denied-inbound policy hook is likely safer than a one-off suppressPairingReply flag.
