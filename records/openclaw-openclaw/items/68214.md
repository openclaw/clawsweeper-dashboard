## Summary

Keep open. Current main fixes the reported accidental WhatsApp allowlist pairing-reply leak, but the remaining request is still a new privacy-sensitive denial hook and optional suppression/config policy surface that is not implemented or documented.

## What This Changes

Request an unauthorized-inbound hook event with suppress-default semantics plus a per-channel pairing-reply suppression option for denied WhatsApp DMs.

## Best Possible Solution

Design one generic, privacy-reviewed channel access-denial extension point with typed reason/default-action data, minimized identifiers by default, trusted-plugin semantics, and a clear suppress-or-replace result contract; fold any config-only silent posture into that same policy model if maintainers want it.

## Reproduction Assessment

Yes for the remaining feature gap: current source and docs show no unauthorized/access-denied hook, no suppress-default denial result contract, and no documented WhatsApp suppression config. The original live WhatsApp leak was not rerun, and current tests now cover the fixed allowlist and historical-message leak paths.

## Solution Assessment

Unclear until maintainers settle the API and security design. A standalone `suppressPairingReply` flag would address one symptom, but the maintainable direction is a generic privacy-safe denial policy hook coordinated with related unpaired-response work.
