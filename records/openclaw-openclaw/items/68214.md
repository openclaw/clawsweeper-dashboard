## Summary

Keep open. Current main fixes the accidental WhatsApp allowlist pairing-reply leak, but the requested unauthorized-inbound hook and suppress-or-replace default outbound policy surface are still absent; related #39065 and closed #42438 do not supersede this issue.

## What This Changes

The issue requests a channel access-denial hook event plus an operator config option to suppress pairing replies for unauthorized inbound senders without disabling legitimate allowlisted DMs.

## Best Possible Solution

Design a generic, privacy-reviewed channel access-denial extension point: typed denial reason/default-action payload, hashed or minimized sender identity by default, clear trusted-plugin opt-in semantics, and a suppress-or-replace result contract. Then decide whether a config-only silent posture belongs in that same contract or in the existing unpaired-response work.

## Reproduction Assessment

Partially. Current-main source and tests give a high-confidence path for the remaining missing surface: hook types, hook docs, and WhatsApp config schema lack an access-denial hook or suppress-default contract; the original 2026.4.12 live WhatsApp incident was not rerun.

## Solution Assessment

No. Current main is a good fix for the accidental non-pairing WhatsApp reply leak, but it is not the best complete solution for this issue because operators still cannot observe or control denied inbound handling through a documented hook/API.
