## Summary

Keep open. Current main appears to have fixed the accidental WhatsApp allowlist pairing-reply leak, but the requested unauthorized-denial hook and denial/pairing-reply policy surface are still absent; the older related request was closed into this issue, and the newer Telegram request is a narrower companion rather than a replacement.

## What This Changes

Add a privacy-aware unauthorized inbound hook/event plus a per-channel or per-account way to suppress pairing replies for denied senders.

## Best Possible Solution

Design a privacy-minimized channel access-denial event/policy contract, keep pairing replies opt-in to pairing mode, and add config, docs, and focused channel tests only after maintainers choose the cross-channel versus channel-owned scope.

## Reproduction Assessment

Not applicable as a bug reproduction for the remaining request. Source inspection on current main shows the earlier WhatsApp allowlist pairing-reply leak is covered, while the requested hook/config surface simply does not exist yet.

## Solution Assessment

Unclear until maintainers choose the API and privacy model. The unauthorized hook/event direction is plausible, but a one-off suppressPairingReply flag is likely weaker than a generic denial-response contract with minimized metadata.
