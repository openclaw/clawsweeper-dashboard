## Summary

Keep open. Current main appears to handle the accidental WhatsApp allowlist pairing-reply path correctly, but the requested unauthorized-denial hook and explicit denial/pairing-reply policy surface are still absent and need privacy-sensitive maintainer design.

## What This Changes

Request to add an unauthorized-inbound hook that can suppress default denial replies plus a channel/account pairing-reply suppression config option.

## Best Possible Solution

Design a generic, privacy-minimized channel access-denial extension point with typed reason/default-action data and a clear suppress-or-replace contract; add config only if it fits that policy model.

## Reproduction Assessment

Not applicable as a bug reproduction for the remaining request. Source and docs show the requested hook/config surface is absent, while current source and tests cover silent allowlist blocking rather than the old accidental pairing-reply leak.

## Solution Assessment

Unclear until maintainers choose the API and privacy model. Current main takes the right direction for the leak by making allowlist denial a block/drop path, but a generic denied-inbound policy hook is likely safer than a one-off suppressPairingReply flag.
