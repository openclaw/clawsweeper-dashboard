## Summary

Keep open. Current main fixes the accidental WhatsApp allowlist pairing-reply leak and adds a historical-message pairing-reply guard, but it still does not expose the requested unauthorized-inbound hook or a documented suppress/replace default-outbound policy surface.

## What This Changes

Request an access-denial hook for rejected inbound messages plus a per-channel way to suppress pairing replies for unauthorized senders without disabling legitimate allowlisted DMs.

## Best Possible Solution

Design a generic, privacy-reviewed channel access-denial extension point with typed reason/default-action data, minimized sender identity by default, trusted-plugin semantics, and a suppress-or-replace result contract; decide whether a config-only silent posture belongs in that same contract or in the related unpaired-response work.

## Reproduction Assessment

Yes for the remaining missing feature surface: current source/docs/schema checks show no access-denial hook, no suppress-default denial result contract, and no documented WhatsApp suppress-pairing config. The original live WhatsApp leak was not rerun, but current tests cover the fixed allowlist and historical-message paths.

## Solution Assessment

Unclear until maintainer API/security design is done. A standalone `suppressPairingReply` flag would be narrower than the hook ask, but the durable solution should make pairing replies opt-in to pairing-mode decisions and expose a privacy-safe denial policy hook.
