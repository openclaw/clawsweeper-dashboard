## Summary

Keep open. Current main fixes the accidental WhatsApp allowlist pairing-reply leak, but the requested unauthorized-denial hook and config/policy surface are still absent and need privacy-sensitive maintainer design.

## What This Changes

Request an unauthorized-inbound hook event plus a channel-level way to suppress pairing replies for denied senders.

## Best Possible Solution

Design one generic, privacy-minimized channel access-denial extension point with typed reason/default-action data and a clear suppress-or-replace contract; fold any config-only silent posture into that same policy if maintainers want it.

## Reproduction Assessment

Not applicable as a bug reproduction for the remaining request. The feature gap is high-confidence from source: current hook names/docs have no unauthorized/access-denied event and the WhatsApp schema has no suppression or denial-response option.

## Solution Assessment

Unclear until maintainers choose the API and privacy model. Current main takes the right direction for the leak by making pairing replies opt-in to pairing mode, but the new denial hook/config surface should be designed generically rather than as a one-off symptom flag.
