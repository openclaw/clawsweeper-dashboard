## Summary

Keep open. Current main covers the accidental WhatsApp allowlist pairing-reply leak path, but the requested unauthorized-denial hook and operator-facing denial/pairing-reply policy surface are still absent and need maintainer API/privacy design.

## What This Changes

The issue asks for an unauthorized-inbound hook plus a per-channel or per-account config surface to suppress pairing replies for denied inbound messages.

## Best Possible Solution

Design a privacy-minimized channel access-denial event or policy contract, keep pairing replies opt-in to pairing mode, and add config, docs, and focused channel tests after maintainers choose the cross-channel versus channel-owned scope.

## Reproduction Assessment

Not applicable as a bug reproduction for the remaining request. Source inspection on current main shows the earlier WhatsApp allowlist pairing-reply leak path is covered, while the requested hook/config surface does not exist.

## Solution Assessment

Unclear until maintainers choose the API and privacy model. The unauthorized hook/event direction is plausible, but a standalone suppressPairingReply flag is likely weaker than a generic denial-response contract with minimized metadata.
