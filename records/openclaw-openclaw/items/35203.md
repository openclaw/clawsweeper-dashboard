## Summary

Keep this open for explicit maintainer product and security direction: current main provides spawning, direct session messaging, lifecycle hooks, memory plugins, and usage telemetry, but it does not implement learned capability routing, an authorized shared blackboard, cross-agent memory tiers, or budget enforcement. Focused open items now cover parts of the proposal, yet no safe canonical replacement covers the remaining routing and shared-memory design, so closing this umbrella would lose meaningful unresolved work.

## What This Changes

The issue proposes capability-aware agent routing, a shared blackboard, tiered cross-agent memory, and token-budget governance.

## Best Possible Solution

Keep this as an umbrella design record, route cost enforcement to https://github.com/openclaw/openclaw/issues/42475 and conversational coordination to the focused open RFCs, then require separate threat-modeled proposals for learned routing and cross-agent memory authorization using plugin-owned APIs and canonical SQLite storage.

## Risks / Open Questions

- Automatic capability scoring lacks a trustworthy success signal and could reinforce incorrect routing or biased historical outcomes.
- Team/global memory and blackboard injection can expose private or sensitive information unless authorization, provenance, revocation, and tenant boundaries are defined before storage or retrieval.
- A monolithic four-layer implementation would create multiple new config, persistent-data, runtime, and UI surfaces at once, making upgrade and security review impractical.
- The proposed dedicated blackboard sidecar conflicts with the repository's canonical SQLite storage policy and would introduce a second state path.
- Automatic promotion, expiry, and reaction-based decay can silently alter durable knowledge; the comments show different entry types require different lifecycle semantics.
