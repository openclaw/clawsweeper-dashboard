## Summary

Keep open. Current main handles fetched Feishu quoted content only when `message.parent_id` is present, while the issue discussion identifies a separate quote-only target that current code still does not model; the active related PR improves parent-id quoted context but does not settle that exact payload contract.

## What This Changes

Request Feishu quote/reply events to include the referenced quoted or parent message content in the agent-facing input.

## Best Possible Solution

Verify the real Feishu quote payload, then add a Feishu-plugin-local quote target that falls back to `parentId`, preserves context-visibility filtering, and includes quote-only regression coverage.

## Reproduction Assessment

Yes, at source level for a quote-only event shape: if Feishu sends only a quote target outside `message.parent_id`, current main has no modeled field and skips the fetch. The exact vendor payload still needs official schema or redacted live-event proof.

## Solution Assessment

Unclear until the payload contract is verified. Fetching the referenced message through the existing Feishu `getMessageFeishu` path is the right owner boundary, but maintainers should not guess an undocumented quote field without live or SDK evidence.
