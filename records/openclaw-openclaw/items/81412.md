## Summary

Keep open: current main and the latest release already handle exact and trailing cron `NO_REPLY` suppression, but they intentionally preserve non-trailing `NO_REPLY` mentions and do not expose the requested cron flag/config policy. This is a product/config feature request rather than an already-implemented bug fix.

## What This Changes

The issue asks for an opt-in cron delivery policy that suppresses announce delivery when `NO_REPLY` appears anywhere in the final response.

## Best Possible Solution

Choose a documented opt-in cron delivery policy, likely a per-job delivery filter, and align CLI, protocol schema, docs, and focused cron delivery tests without changing the existing default exact/trailing behavior implicitly.

## Reproduction Assessment

Yes, source-reproducible. Current code and tests show exact/trailing cron suppression exists, while non-trailing `NO_REPLY` mentions remain deliverable and no requested flag/config field exists.

## Solution Assessment

Unclear. The requested behavior is plausible as an opt-in cron policy, but changing default handling or adding regex/config knobs needs maintainer product judgment because current tests deliberately preserve non-trailing mentions.
