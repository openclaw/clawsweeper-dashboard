## Summary

Keep open: the protected maintainer label requires explicit handling, and review found a blocking correctness concern in the experimental BlueBubbles SQLite dedupe path.

## What This Changes

Adds `PluginStateKeyedStore.registerIfAbsent`, a bounded plugin-state capacity override, docs/changelog entries, and an experimental BlueBubbles opt-in for SQLite-backed inbound dedupe.

## Best Possible Solution

Keep the store primitive direction, but make BlueBubbles pending claims recoverable before the opt-in path can drop replayed messages as duplicates.

## Reproduction Assessment

Yes, the review finding is source-reproducible: the PR writes `{ status: "claimed" }` before finalization and later returns `duplicate` for any live row. I did not execute tests because this review was required to stay read-only.

## Solution Assessment

No, not as-is for the BlueBubbles opt-in path. The SQLite primitive is narrow, but the consumer should distinguish pending claims from committed entries or use a short recoverable claim lease before dropping replays.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Handle stale claimed rows before dropping replays:** `extensions/bluebubbles/src/inbound-dedupe.ts:316-323`
  - body: The SQLite path persists a live `claimed` row before the message is finalized, but a later delivery treats any existing row as a duplicate. If the gateway exits after `registerIfAbsent` succeeds and before `finalize` or `release` runs, BlueBubbles can replay the GUID and this opt-in path will drop it for the full 7-day TTL even though the message was never processed. Please distinguish pending claims from committed entries, for example with a short claim lease or reclaim path, before returning `duplicate`.
  - confidence: 0.86
