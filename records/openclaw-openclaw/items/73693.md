## Summary

Keep open: the PR targets a real ACP timeout-control bug that is still visible in current source and related live reports, but the submitted patch needs changes before merge because it can make rejected permission controls run as silent no-ops and its changelog still carries stale acpx version wording.

## What This Changes

This PR removes ACP timeout backend mirroring, reroutes `/acp timeout` through local runtime options, adds JSON-RPC detail extraction and fail-soft unsupported-control handling, forwards acpx capability input, and updates docs, tests, and changelog.

## Best Possible Solution

Keep the manager-owned timeout watchdog, remove or skip the redundant backend timeout mirror, route timeout commands through local runtime options, and keep permission/security controls visible when a backend rejects them.

## Reproduction Assessment

Yes, source and related live reports give a high-confidence reproduction path: current main still builds and applies a `timeout` config option before ACP turns, while Claude ACP reports in https://github.com/openclaw/openclaw/issues/78546 show that key being rejected. I did not run a live ACP harness in this read-only review.

## Solution Assessment

No, not as submitted. Removing the timeout mirror and localizing `/acp timeout` is the right direction, but fail-softing all unsupported config/control rejections needs narrowing or user-visible handling for permission controls, and the changelog must stop claiming stale dependency-backed behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep permission-control failures visible:** `src/acp/control-plane/manager.runtime-controls.ts:151-155`
  - body: The new fail-soft branch treats any classified unsupported config rejection as skippable, so a backend rejecting `approval_policy` or a persisted permission profile would now run the turn with only a verbose log. Limit fail-soft behavior to non-security optional controls such as timeout, or surface/block rejected permission controls so users do not think a stricter policy was applied.
  - confidence: 0.88
- **[P2] Correct the stale acpx version claim:** `CHANGELOG.md:21`
  - body: The added changelog says advertised-key reconciliation is plumbed through bumped `acpx@0.6.2`, but this branch does not change package or lock files, and current main already pins `acpx` 0.7.0. Reword this as wrapper handle forwarding/current dependency behavior, or remove the stale version claim.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff can make rejected permission controls non-fatal, so a session may proceed under backend defaults without user-visible notice.

Concerns:

- **[medium] Rejected permission controls can become silent no-ops:** `src/acp/control-plane/manager.runtime-controls.ts:151`
  - body: `permissionProfile` is mapped to `approval_policy`; with this patch, an unsupported-config rejection for that key matches the fail-soft classifier and only logs via `logVerbose`, so a requested stricter permission profile can be ignored while the ACP turn proceeds.
