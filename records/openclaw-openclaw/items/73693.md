## Summary

Keep open. Current main still has the ACP timeout/control rejection behavior, and this PR addresses the right surfaces, but the branch is not merge-ready because its changelog says `acpx@0.6.2` is bumped even though the bundled plugin dependency remains pinned to `acpx` 0.6.1 and the PR does not change package or lockfile metadata.

## What This Changes

The PR removes ACP timeout config mirroring, routes timeout commands through local runtime options, adds JSON-RPC error-detail extraction and classified fail-soft control handling, forwards acpx capability input, and updates ACP tests, docs, and changelog.

## Best Possible Solution

Correct the changelog dependency wording, then land this focused ACP fix or a clean superseding PR with the targeted ACP/acpx tests and a live smoke confirming timeout stays local and unsupported backend controls no longer collapse turns into opaque errors.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction path: timeout runtime options are converted into a backend `timeout` config key and applied before the turn, while the PR body and linked issue include a direct JSON-RPC rejection from `claude-agent-acp` v0.31+ for that unsupported key.

## Solution Assessment

Mostly yes for the runtime behavior: removing redundant timeout round-trips, keeping local timeout enforcement, forwarding capabilities input, and fail-softing only classified unsupported controls is a narrow maintainable fix. As submitted, the changelog is not the best solution because it overstates dependency-backed behavior that is not in this PR.
