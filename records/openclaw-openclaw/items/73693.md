## Summary

Keep open: the PR targets a real ACP timeout-control failure and addresses the earlier changelog version issue, but it still lacks after-fix real behavior proof and introduces unredacted diagnostic surfaces that need correction before merge.

## What This Changes

The PR removes ACP timeout config mirroring, reroutes `/acp timeout` through local runtime options, adds fail-soft handling for unsupported ACP controls, forwards acpx capability input, bumps acpx to 0.7.0, and updates tests/docs/changelog.

## Best Possible Solution

Land one current-main-compatible ACP timeout-control fix that keeps the manager-owned timeout watchdog, skips or filters unsupported backend timeout controls, preserves redacted diagnostics, and includes live Claude ACP proof plus focused regression tests.

## Reproduction Assessment

Yes, source/report-reproducible. Current main still emits the timeout config pair, and linked live diagnostics show Claude ACP rejecting that unsupported key while direct ACPX works; I did not run a live harness in this read-only review.

## Solution Assessment

No, not yet. The core direction is plausible, but the PR needs redaction fixes and after-fix live behavior proof before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Redact skipped-control diagnostics before logging:** `src/acp/control-plane/manager.runtime-controls.ts:155`
  - body: This warning includes the raw config value and raw backend error detail. `/acp set` and `backendExtras` can carry backend-specific values, so unsupported-control logs should pass both the value and extracted detail through the existing redaction helper before writing verbose logs.
  - confidence: 0.87
- **[P2] Redact JSON-RPC detail before sending replies:** `src/acp/runtime/error-text.ts:53`
  - body: `detailLine` is appended directly to user-facing ACP error text, and that formatter is used by `/acp` command failures and ACP dispatch failure replies. Upstream `data.details` can contain arbitrary backend text, so this should use the existing redaction path before reaching channel replies.
  - confidence: 0.84
