## Summary

Keep this PR open, but do not merge it as-is. Current main does not already implement the requested gateway context headers, and the PR is useful source work, but the diff still needs maintainer decisions around opt-in/trust boundaries and has concrete fallback and turn-id correctness bugs.

## What This Changes

The PR adds OpenClaw session, attempt, fallback, and per-turn context headers to embedded LLM requests, propagates fallback state through model-fallback callers, and adds `isFallback` to plugin LLM output events.

## Best Possible Solution

Adopt a maintainer-approved observability design that makes LLM request context propagation explicit and trust-scoped, derives fallback state from the fallback candidate selection rather than callback count, keeps a per-run monotonic LLM request counter across embedded retries, and lands focused tests plus docs/changelog coverage coordinated with #69463 and #22021.

## Reproduction Assessment

Yes. The main issues are reproducible by static code tracing: the PR merges new headers into `params.model.headers`, current fallback code can skip candidates before the run callback, and embedded retry paths can restart before an assistant message is persisted.

## Solution Assessment

No. The direction is useful, but the current PR is not the best safe implementation because it exposes stable identifiers without an opt-in trust boundary and derives some metadata from counters that do not match provider request semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate context headers to trusted gateway targets:** `src/agents/pi-embedded-runner/run/attempt.ts:2179-2193`
  - body: The new `gatewayContextHeaders` are merged into `params.model.headers`, so any provider transport that forwards model headers can receive stable OpenClaw session/run IDs during normal provider calls, not only through an operator-configured AI gateway. That broadens outbound metadata and can leak cross-request correlation identifiers; make this opt-in or restrict it to trusted gateway/proxy targets before merge.
  - confidence: 0.86
- **[P2] Use fallback runner options for CLI fallback status:** `src/agents/agent-command.ts:520`
  - body: `runWithModelFallback` now computes fallback state at the candidate level in the PR, but this path still forwards `params.isFallbackRetry`, which counts only executed callbacks. If the primary candidate is skipped before invocation due to cooldown, the first executed fallback is labeled primary in the new header and hook metadata. Pass the fallback runner option through this call path instead.
  - confidence: 0.88
- **[P2] Keep turn IDs unique across embedded retries:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2475`
  - body: `llmTurnCount` is local to one `runEmbeddedAttempt` and is seeded from persisted assistant messages. Inner retry paths can restart the attempt before an assistant message is written, so the next HTTP request can reuse the same `{runId}-turn-{index}` value even though the PR promises a unique turn ID per LLM HTTP request. Store the counter at the enclosing run level or otherwise carry it across retries.
  - confidence: 0.82
