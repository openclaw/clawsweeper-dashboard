## Summary

Keep open. The observability goal is plausible, but the PR is not merge-ready because it sends privacy-sensitive identifiers unconditionally, can duplicate model-call IDs across retries, still has fallback-state edge cases, and lacks real behavior proof.

## What This Changes

The PR adds fixed OpenClaw session, run, fallback, turn-index, and turn-id headers to outbound LLM requests and adds `isFallback` to LLM output hook events.

## Best Possible Solution

Define one documented opt-in or trusted-target-scoped LLM request-correlation contract, use bounded identifiers, carry fallback state from the actual fallback runner, and allocate retry-stable model-call IDs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug because this is a feature PR, but the blocking PR issues are source-reproducible from the diff and current fallback/retry paths.

## Solution Assessment

Is this the best way to solve the issue? No. The request-correlation direction is useful, but this implementation should first establish a privacy-safe opt-in boundary and fix fallback plus per-request identity semantics.
