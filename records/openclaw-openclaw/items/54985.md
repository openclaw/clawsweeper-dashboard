## Summary

Keep open. The observability goal is plausible, but the PR is not merge-ready because it sends provider-bound identifiers without an opt-in trust boundary, can duplicate request identifiers across retry paths, still mislabels some fallback executions, lacks real behavior proof, and is now conflict-stale against current main.

## What This Changes

The PR adds fixed OpenClaw session, run, fallback, turn-index, and turn-id headers to outbound LLM requests and exposes fallback status on LLM output hook events.

## Best Possible Solution

Define one documented opt-in or trusted-target LLM request-correlation contract with bounded identifiers, use run-scoped diagnostic call IDs and actual fallback runner state, and coordinate it with https://github.com/openclaw/openclaw/pull/69463 plus https://github.com/openclaw/openclaw/issues/22021.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug because this is a feature PR; yes for the blocking PR defects, which are source-reproducible from the diff and current fallback/retry paths.

## Solution Assessment

Is this the best way to solve the issue? No. The request-correlation direction is useful, but the implementation should first establish an opt-in privacy boundary, use retry-stable request IDs, and carry fallback state from the actual fallback runner.
