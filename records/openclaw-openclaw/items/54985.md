## Summary

Keep open: the observability direction is plausible, but the PR is not merge-ready because it sends raw session/run identifiers to provider-bound requests without an opt-in boundary, can reuse turn IDs across retry paths, still has fallback-state edge cases, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds fixed OpenClaw session, attempt, fallback, turn-index, and turn-id headers to outbound LLM requests and exposes fallback status on LLM output hook events.

## Best Possible Solution

Define one documented opt-in or trusted-target LLM request-correlation contract with bounded identifiers, retry-stable per-call IDs, and candidate-level fallback/model metadata coordinated with the open tracing and actual-model work.

## Solution Assessment

Is this the best way to solve the issue? No. The request-correlation direction is useful, but the implementation should first establish an opt-in privacy boundary, use retry-stable request IDs, and carry fallback state from the actual fallback runner.

  - confidence: 0.84

  - body: This path passes fallback status from the executed-attempt counter, but the fallback runner can skip primary candidates before invoking this callback.   - confidence: 0.82
