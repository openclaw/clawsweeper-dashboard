## Summary

Keep open. The observability direction is useful, but this PR still has blocking privacy and correctness issues, overlaps with an opt-in tracing proposal, and has no inspectable real-behavior proof from a gateway/provider setup.

## What This Changes

The PR injects X-OpenClaw session, run, fallback, and turn headers into embedded LLM requests, threads fallback state through model-fallback callers, and adds fallback metadata to plugin LLM output events.

## Best Possible Solution

Define one documented, privacy-safe request-correlation contract that is opt-in or trusted-target scoped, uses bounded identifiers, carries fallback state from actual model selection, and assigns unique per-model-call IDs across retries.

## Reproduction Assessment

Yes for the blocking PR issues: source inspection shows the headers merge into model headers, fallback candidates can be skipped before the run callback, and the turn counter is scoped to one embedded attempt. This is a feature PR, so there is no current-main bug to reproduce.

## Solution Assessment

No. The requested request-correlation capability is useful, but this patch is not the best solution until maintainers define the privacy/API boundary and the implementation fixes fallback and per-request identity semantics.

  - body: The turn counter resets for each embedded attempt and seeds IDs from persisted assistant messages, so retries before an assistant message is written can reuse the same X-OpenClaw-Turn-Id. Carry a run-scoped request counter or diagnostic call id across retries.
  - confidence: 0.82
