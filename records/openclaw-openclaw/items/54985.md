## Summary

Keep open: the gateway-correlation feature is plausible, but this PR still crosses a privacy-sensitive provider boundary without an opt-in or trusted target, has incorrect fallback/turn identity semantics in retry paths, and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds session, run, fallback, and turn HTTP headers to embedded LLM requests, propagates fallback state through model-fallback callers, and exposes `isFallback` on LLM output plugin events.

## Best Possible Solution

Define one documented, privacy-safe LLM request-correlation contract that is opt-in or trusted-target scoped, uses bounded identifiers, carries fallback state from actual model selection, and assigns unique per-model-call IDs across retries.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. Yes for the blocking PR issues: source inspection shows the headers cross into model headers, fallback candidates can be skipped before callbacks, and the turn counter is local to one embedded attempt.

## Solution Assessment

No. The request-correlation direction is useful, but the proposed implementation should first establish a privacy/API boundary and fix fallback plus per-request identity semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate context headers to trusted targets:** `src/agents/pi-embedded-runner/run/attempt.ts:2179-2193`
  - body: The new gateway context headers are merged into model headers before session creation, so stable session and run identifiers can be sent to ordinary provider transports. Gate this behind an explicit opt-in or trusted gateway target, or send bounded non-sensitive identifiers before crossing the provider boundary.
  - confidence: 0.9
- **[P2] Propagate fallback state from run options:** `src/agents/agent-command.ts:520`
  - body: This passes `isFallback` from the callback-count-derived `isFallbackRetry`, but `runWithModelFallback` can skip primary candidates before invoking the callback. Use the fallback runner's run options so the first executed fallback is not labeled as primary.
  - confidence: 0.87
- **[P2] Keep model-call IDs unique across retries:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2475`
  - body: The turn counter is scoped to one `runEmbeddedAttempt` and seeds IDs from persisted assistant messages, so retries before an assistant message is written can reuse the same `X-OpenClaw-Turn-Id`. Carry a run-scoped request counter or diagnostic call id across retries.
  - confidence: 0.82
