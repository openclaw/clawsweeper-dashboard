## Summary

Keep open. The tracing direction is useful, but this PR is not merge-ready because it sends stable OpenClaw identifiers across the provider boundary without an opt-in/trusted-target contract, can reuse model-call IDs across retries, can mislabel fallback state, has no real behavior proof, and currently conflicts with main.

## What This Changes

This PR adds fixed OpenClaw session, attempt, fallback, turn-index, and turn-id headers to outbound LLM requests and exposes fallback status on LLM output hook events.

## Best Possible Solution

Define one documented opt-in or trusted-target LLM request-correlation contract with bounded identifiers, run-scoped model-call IDs, and actual fallback runner state, coordinated with the open tracing and actual-model metadata work.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. The blocking PR defects are source-reproducible from the diff and current fallback/retry paths.

## Solution Assessment

No. The request-correlation direction is useful, but the implementation should first establish an opt-in privacy boundary, use retry-stable call IDs, carry fallback state from the actual fallback runner, and rebase onto current main.
