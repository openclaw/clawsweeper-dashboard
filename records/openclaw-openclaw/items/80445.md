## Summary

Keep open: the PR is aimed at a real duplicate-delivery bug, but the current patch can still suppress a final reply on a different visible route when account/thread metadata is present on only one side, and the supplied proof is mocked-only.

## What This Changes

The PR threads same-turn explicit `message.send` records through agent runs, suppresses exact duplicate final deliveries on the same route, adds in-process explicit-send idempotency, and expands focused regression tests.

## Best Possible Solution

Land the exact duplicate suppression and explicit-send idempotency only after one-sided account/thread metadata is treated as non-matching, live redacted Gateway/Discord proof is supplied, and detached longwork ownership remains tracked separately at https://github.com/openclaw/openclaw/issues/80450.

## Reproduction Assessment

Yes, from source inspection for the same-turn path: current main can execute an explicit visible `message.send` and then independently deliver the same final payload. The replayed explicit-send shape is supported by the contributor's live observation but was not live-reproduced here.

## Solution Assessment

No, not as currently implemented. Exact visible-text suppression plus direct-send idempotency is the right narrow direction, but route matching must reject one-sided account/thread metadata before merge.

## Review Findings
