## Summary

Keep open: current main does not have the same-turn duplicate-send guard, and this PR is on the right delivery surface, but current head can still suppress an in-flight replay before the original send succeeds and the supplied proof is synthetic/mock-only.

## What This Changes

The PR adds same-turn explicit `message.send` tracking, duplicate final-delivery suppression, explicit-send idempotency propagation, and regression tests across agent delivery, message-tool, and outbound delivery paths.

## Best Possible Solution

Keep the exact route/text duplicate-final predicate, revise outbound idempotency to wait for or share the in-flight result or only suppress after successful send memory, then add redacted live Gateway/Discord proof or an explicit maintainer override.

## Reproduction Assessment

Yes, source-reproducible: current main has separate visible `message.send` and final durable delivery paths with no same-turn tracker. I did not run a live Discord reproduction.

## Solution Assessment

No, not yet. Exact normalized text plus full route equality is the right suppression shape, but the idempotency helper should not mark an in-flight replay as suppressed before the original send has succeeded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Wait for in-flight idempotent sends before suppressing:** `src/infra/outbound/send-idempotency.ts:119-120`
  - body: When a matching idempotency key is already in flight, the helper returns the duplicate value immediately. If the first send then fails before any platform result is remembered, the replay has already been suppressed and no queue/platform retry remains, so the visible message can be dropped; share or wait on the in-flight result, or suppress only after a successful send is remembered.
