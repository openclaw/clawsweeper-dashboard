## Summary

Keep this PR open: it is a plausible targeted fix for duplicate visible deliveries, but the latest head still has a route-matching correctness bug and the supplied proof is isolated mocked test output rather than live Gateway/Discord behavior.

## What This Changes

The PR adds same-turn explicit message-send tracking, exact normalized visible-text final-delivery suppression, in-process outbound idempotency for replayed explicit sends, and focused regression tests.

## Best Possible Solution

Land this direction only after one-sided account/thread route metadata is treated as non-matching, a regression covers that boundary, redacted live Gateway/Discord proof is supplied, and detached longwork ownership remains tracked separately at https://github.com/openclaw/openclaw/issues/80450.

## Reproduction Assessment

Yes, from source inspection: current main can execute a visible `message.send` through `runMessageActionForTool` and then independently deliver the same final reply through `sendDurableMessageBatch`. I did not live-reproduce the Discord replay case.

## Solution Assessment

No, not as currently implemented. Exact visible-text suppression plus explicit-send idempotency is the right narrow direction, but route matching must reject one-sided account/thread metadata before merge.
