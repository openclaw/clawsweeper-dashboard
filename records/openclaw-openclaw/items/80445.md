## Summary

Keep open: the duplicate-delivery bug is plausible from current source, but the patch can suppress near-duplicate final replies, still needs real Discord/gateway proof, and has failing validation on the PR head.

## What This Changes

The branch adds same-turn explicit message-send tracking through gateway/agent run context and suppresses final delivery when it matches a current-channel send, with focused delivery tests.

## Best Possible Solution

Keep the generic delivery-boundary approach, but restrict suppression to exact normalized or structurally proven duplicates, add a near-duplicate regression test, attach redacted live Discord/gateway proof, and rerun validation.

## Reproduction Assessment

Yes, at source level: current main has independent current-channel message.send execution and final durable delivery with no same-turn duplicate tracker. A model turn that sends the same Discord text via message.send and then returns it as final text would traverse both paths.

## Solution Assessment

No as written. The delivery boundary is a reasonable place to solve this, but the fuzzy text predicate is broader than duplicate suppression and the new route normalization currently fails lint.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Avoid suppressing near-duplicate final replies:** `src/agents/command/delivery.ts:313-320`
  - body: The duplicate guard treats long replies with 90% substring containment or 94% token overlap as duplicates. A same-target explicit send followed by a final answer that adds a short warning or correction to a long report would be suppressed, so the channel never receives the new information; restrict this to exact normalized or structurally proven duplicates.
  - confidence: 0.88

Narrow to string/number/nullish inputs or reuse an existing route normalizer in both new helper copies.
