## Summary

Keep this PR open, but it should not merge yet: the intended duplicate-delivery fix is plausible, while the current predicate can suppress non-identical final replies and the PR only provides typecheck/unit-test proof.

## What This Changes

The PR threads a same-turn explicit message-send tracker through gateway agent runs and suppresses final outbound delivery when it matches a current-channel explicit send, with focused delivery tests.

## Best Possible Solution

Tighten suppression to exact normalized duplicates or another proof-safe structural duplicate check, keep the route/media/thread guards, add a near-duplicate regression test, and attach redacted live Discord or gateway proof.

## Reproduction Assessment

Yes from source inspection, though not from a live run: current main has independent current-channel message-tool sends and final durable delivery with no same-turn duplicate tracker. A model turn that sends the same Discord text via `message.send` and then returns it as the final payload would traverse both send paths.

## Solution Assessment

No as written. The generic delivery boundary is a reasonable place to solve this, but the fuzzy content match is broader than duplicate suppression and should be narrowed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid suppressing near-duplicate final replies:** `src/agents/command/delivery.ts:313-320`
  - body: The new duplicate guard treats long replies with only 94% token overlap, or a 90% substring match, as duplicates. Please restrict this to exact normalized duplicates, or otherwise prove the extra-content case cannot be dropped.
