## Summary

Keep this PR open: the duplicate-delivery direction is useful, but the patch still has blocking correctness issues and the supplied proof is only isolated mocked-test output, not real Gateway/Discord behavior proof.

## What This Changes

The PR tracks same-turn current-channel `message.send` calls, suppresses exact duplicate final delivery, adds explicit-send idempotency, and adds focused delivery/idempotency regression tests.

## Best Possible Solution

Keep the exact-duplicate suppression and explicit-send idempotency direction, but require route matching to treat one-sided account/thread metadata as non-matching, preserve current source-reply-mode plumbing, and add redacted real Gateway/Discord proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main can execute a visible `message.send` through the message tool and later send final delivery through `sendDurableMessageBatch` in the same turn, although I did not live-reproduce the Discord duplicate.

## Solution Assessment

No, not yet. Exact normalized visible-text suppression plus explicit-send idempotency is the narrow maintainable direction, but the patch still needs stricter one-sided route comparison and must preserve current `sourceReplyDeliveryMode` forwarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require one-sided route fields to mismatch:** `src/agents/command/delivery.ts:287-294`
  - body: The duplicate guard still treats account/thread as matching when only one side has the field. A thread-targeted `message.send` can therefore suppress a same-text final reply whose delivery route has no thread, even though those are different visible surfaces; require equality whenever either route field is present and add a one-sided route regression.
  - confidence: 0.88
- **[P2] Preserve source reply mode when adding send tracking:** `src/agents/tools/message-tool.ts:1005`
  - body: This branch's merge result drops current main's `sourceReplyDeliveryMode` argument from the preceding `runMessageActionForTool` call. `runMessageAction` uses that option to route WebChat/TUI `message_tool_only` sends into the internal source reply sink, so keep forwarding it while adding the explicit-send tracking.
  - confidence: 0.91
