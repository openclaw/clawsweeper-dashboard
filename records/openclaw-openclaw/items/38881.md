## Summary

Keep open. The product gap is real and current main still does not switch the final reply run to `agents.defaults.imageModel`, but this branch is not merge-ready because it switches after media understanding, uses generic media fields as image detection, treats unknown catalog entries as non-vision, and ignores valid image-model fallbacks.

## What This Changes

The branch adds a get-reply helper and Vitest coverage to switch image-bearing replies from a non-vision active model to `agents.defaults.imageModel.primary`.

## Best Possible Solution

Rework this as a current-main reply/model-selection change that resolves the image-capable run model before media understanding, uses normalized attachment kind metadata for actual image inputs, shares the existing alias/catalog/fallback behavior for `imageModel`, and leaves unknown or explicitly overridden models unchanged unless capability is known.

## Reproduction Assessment

Yes. The merge-blocking problems are reproducible by source inspection: the PR calls the switch after media understanding, checks only generic path fields, and relies on catalog `undefined` as non-vision while current tests show non-image `MediaPaths`.

## Solution Assessment

No. The direction is useful, but this patch duplicates partial model and media selection logic instead of using the existing normalized attachment and image-model fallback contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move the switch before media understanding:** `src/auto-reply/reply/get-reply.ts:303-314`
  - body: The helper is called only after `applyMediaUnderstanding` has already run with the pre-switch model, so image turns on text-only defaults can first run separate image understanding and then run the final reply on the vision model. Resolve the switch before media understanding, or skip/reconcile media understanding after switching.
  - confidence: 0.92
- **[P2] Use typed attachment detection for images:** `src/auto-reply/reply/get-reply.ts:87-89`
  - body: `MediaPath` and `MediaPaths` are generic media fields: current chat.send paths also carry PDFs and other files, while URL-only media can arrive through `MediaUrl`/`MediaUrls`. This can switch non-image messages to the image model and miss URL-only image inputs; use normalized attachment kind checks instead.
  - confidence: 0.9
- **[P2] Require a known non-vision current model:** `src/auto-reply/reply/get-reply.ts:95-99`
  - body: When the active provider/model is absent from the catalog, `modelSupportsVision(undefined)` is false and this path silently overrides custom or newly configured vision-capable models. Treat unknown current capability as no-switch unless another explicit source proves it cannot accept images.
  - confidence: 0.88
- **[P2] Honor imageModel fallbacks:** `src/auto-reply/reply/get-reply.ts:101-103`
  - body: The existing `imageModel` contract supports `primary` plus `fallbacks`, including fallback-only configurations, but this helper reads only `imageModel.primary` and returns early when it is empty. Use the same candidate/fallback resolution path so valid fallback entries can be selected.
  - confidence: 0.86
