## Summary

Keep this PR open. The requested behavior is still not implemented on current main, but the proposed patch is not merge-ready because it switches after media understanding has already run, uses generic media-path fields as image detection, assumes unknown catalog entries are non-vision, and ignores valid imageModel fallback-only configurations.

## What This Changes

The PR adds a get-reply auto-switch helper and Vitest coverage to route image-bearing replies to agents.defaults.imageModel.primary when the selected model lacks vision support.

## Best Possible Solution

Rework this in the reply/model-selection path so the model is resolved before media understanding, image inputs are detected through normalized attachment type metadata, imageModel primary and fallbacks are resolved through the existing alias/catalog/fallback helpers, unknown model capability does not cause a silent override, and integration coverage proves the final run receives the vision model only for actual image inputs.

## Reproduction Assessment

Yes. The merge-blocking issues are reproducible by source inspection against the PR head and current-main attachment tests: MediaPaths can be non-image, MediaUrls are valid attachment inputs, and the PR call runs after media understanding.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation because it duplicates or misorders the media-understanding path and bypasses existing attachment and image-model fallback contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move the switch before media understanding:** `src/auto-reply/reply/get-reply.ts:303-314`
  - body: The helper is called after media understanding has already run, so image turns on text-only defaults can first run the separate image-understanding model and then run the final reply on the vision model. Resolve the switch before applyMediaUnderstandingIfNeeded, or explicitly skip/reconcile media understanding after switching so one turn does not analyze the same image twice.
  - confidence: 0.92
- **[P2] Use typed attachment detection for images:** `src/auto-reply/reply/get-reply.ts:86-90`
  - body: MediaPath/MediaPaths are generic media fields: current chat.send tests route PDFs through them, while URL-only media uses MediaUrl/MediaUrls. This condition can switch non-image messages to imageModel and miss URL-only image messages; use the normalized attachment helpers and image-kind checks instead.
  - confidence: 0.9
- **[P2] Require known non-vision capability before switching:** `src/auto-reply/reply/get-reply.ts:94-99`
  - body: When the active provider/model is absent from the catalog, modelSupportsVision(undefined) is false and the helper silently overrides custom or newly configured vision-capable models. Treat an unknown current model as no-switch unless another explicit capability source proves it cannot accept images.
  - confidence: 0.88
- **[P2] Honor imageModel fallbacks:** `src/auto-reply/reply/get-reply.ts:101-105`
  - body: The existing image-model fallback contract accepts primary and fallbacks, including fallback-only configurations, but this helper returns early when imageModel.primary is empty. Use the same imageModel candidate resolution/fallback path so valid fallback entries can be selected.
  - confidence: 0.86
