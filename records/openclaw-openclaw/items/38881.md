## Summary

Keep open. The underlying documented bug is valid, but current main still lacks final-run imageModel auto-switching and this PR has blocking correctness issues in call order, attachment detection, unknown-model handling, and imageModel fallback handling.

## What This Changes

The PR adds a get-reply helper and tests to switch image-bearing replies from a non-vision active model to `agents.defaults.imageModel.primary`.

## Best Possible Solution

Rework reply model selection so image-capable run-model resolution happens before media understanding, uses normalized image attachment metadata, shares the existing imageModel primary/fallback candidate behavior, and preserves explicit overrides.

## Reproduction Assessment

Yes. Source inspection is enough: current docs promise imageModel fallback routing, while current `getReplyFromConfig` passes the default provider/model to media understanding before later model resolution and has no final-run switch.

## Solution Assessment

No. The branch's direction is useful, but the switch should run before media understanding, use typed image attachment data, require a known non-vision current model, and reuse existing primary/fallback imageModel candidate resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move the switch before media understanding:** `src/auto-reply/reply/get-reply.ts:303-314`
  - body: The helper is called only after `applyMediaUnderstanding` has already run with the pre-switch active model. Image turns on text-only defaults can therefore run separate image understanding and then run the final reply on the vision model, adding avoidable latency/cost and conflicting context; resolve the switch before media understanding or skip/reconcile that pass after switching.
  - confidence: 0.92
- **[P2] Use typed image attachment detection:** `src/auto-reply/reply/get-reply.ts:87-89`
  - body: `MediaPath` and `MediaPaths` are generic media fields that can carry PDFs/audio/video, while URL-only images can arrive through `MediaUrl`/`MediaUrls`. This can switch non-image messages to the image model and miss supported image inputs; use normalized attachment kind or MIME metadata instead.
  - confidence: 0.9
- **[P2] Require a known non-vision current model:** `src/auto-reply/reply/get-reply.ts:95-99`
  - body: When the active provider/model is absent from the catalog, `modelSupportsVision(currentModelEntry)` returns false and this path silently overrides custom or newly configured vision-capable models. Treat unknown current capability as no-switch unless an explicit source proves it cannot accept images.
  - confidence: 0.88
- **[P2] Honor imageModel fallbacks:** `src/auto-reply/reply/get-reply.ts:101-103`
  - body: The existing `imageModel` contract supports `primary` plus `fallbacks`, including fallback-only configurations, but this helper reads only `imageModel.primary` and returns early when it is empty. Use the same candidate/fallback resolution path so valid fallback entries can be selected.
  - confidence: 0.86
