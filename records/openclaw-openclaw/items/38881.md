## Summary

This PR should stay open for rework: it targets a real docs-backed gap from #38796, but the branch’s helper is too late in the reply pipeline and duplicates partial media/model-routing logic that current main already handles more carefully elsewhere.

## What This Changes

The branch adds `resolveImageModelAutoSwitch()` and Vitest coverage to route image-bearing replies from a non-vision active model to `agents.defaults.imageModel.primary`.

## Best Possible Solution

Rework reply model selection on current main so image-capable run-model resolution happens before media understanding, uses normalized image attachment metadata, shares the existing imageModel primary/fallback candidate behavior, and preserves explicit model overrides.

## Reproduction Assessment

Yes. Source inspection on current main shows the documented `imageModel` contract is not applied to the final reply model, and the PR defects are visible from the changed call order and helper logic.

## Solution Assessment

No. The direction is right, but this implementation is not the best fix because it runs after media understanding, treats generic media paths as images, assumes unknown active models are non-vision, and ignores fallback-only imageModel configurations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move the switch before media understanding:** `src/auto-reply/reply/get-reply.ts:303-314`
  - body: The helper is called only after `applyMediaUnderstanding` already ran with the pre-switch model at lines 206-212. Image turns on text-only defaults can therefore run separate image understanding and then run the final reply on the vision model, adding avoidable latency/cost and conflicting context; resolve the switch before media understanding or skip/reconcile media understanding after switching.
  - confidence: 0.92
- **[P2] Use typed attachment detection for images:** `src/auto-reply/reply/get-reply.ts:87-89`
  - body: `MediaPath` and `MediaPaths` are generic media fields: current chat.send paths also carry PDFs and other files, while URL-only media can arrive through `MediaUrl`/`MediaUrls`. This can switch non-image messages to the image model and miss URL-only images; use normalized attachment kind checks instead.
  - confidence: 0.9
- **[P2] Require a known non-vision current model:** `src/auto-reply/reply/get-reply.ts:95-99`
  - body: When the active provider/model is absent from the catalog, `modelSupportsVision(undefined)` is false and this path silently overrides custom or newly configured vision-capable models. Treat unknown current capability as no-switch unless another explicit source proves it cannot accept images.
  - confidence: 0.88
- **[P2] Honor imageModel fallbacks:** `src/auto-reply/reply/get-reply.ts:101-103`
  - body: The existing `imageModel` contract supports `primary` plus `fallbacks`, including fallback-only configurations, but this helper reads only `imageModel.primary` and returns early when it is empty. Use the same candidate/fallback resolution path so valid fallback entries can be selected.
  - confidence: 0.86
