## Best Possible Solution

## Reproduction Assessment

## Solution Assessment

No. Parser-level validation is the right boundary, but the latest implementation uses plain `Error`, which Commander does not treat as an invalid option argument.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Use Commander invalid-argument errors:** `extensions/memory-core/src/cli.ts:71`
  - body: Commander only formats parser failures as option errors when the thrown error has code `commander.invalidArgument`; plain `Error` falls through from `_callParseArg`. With this head, invalid numeric options can reject with the raw helper error instead of the promised `option ... argument ... is invalid` CLI error, and the tests only assert the substring. Please use `InvalidArgumentError` or an equivalent Commander-compatible error for both helpers and assert the CLI-level error path.
