## Summary

Keep open. The PR fixes a real memory CLI numeric parsing bug, and the updated PR body gives usable live-output proof for the fail-fast behavior, but current head still throws plain `Error` from Commander option parsers, so the CLI does not get Commander invalid-argument formatting.

## What This Changes

The PR replaces permissive numeric coercion in memory-core and memory-wiki CLI options with finite/range validation and adds focused regression tests for malformed values.

## Best Possible Solution

Keep parser-level validation, but make both helpers throw Commander-compatible invalid-argument errors without adding runtime `commander` imports, then assert the CLI-level option error and land with clear release-note context for malformed numeric arguments.

## Reproduction Assessment

Yes. Current main source uses raw `Number(value)` for the affected memory and wiki CLI options, so values like `nope` and `Infinity` can become NaN or Infinity before runtime handling.

## Solution Assessment

No. Parser-level validation is the right boundary, but plain `Error` does not satisfy Commander's invalid-argument contract; the narrow fix is a local Commander-compatible error shape plus CLI-level assertions.
