## Summary

Keep open: this is a valid user-visible reminder-guard bug and the PR branch is useful, but the patch is not merge-ready because it reads a new public config key without adding the typed config contract or regenerated config metadata. Maintainers also still need to choose this config opt-out direction over the related delivery-stripping approach in #39330.

## What This Changes

This PR adds a `messages.unscheduledReminderNote` boolean gate around the unscheduled-reminder note append path, adds the key to the session Zod schema and field labels, and adds focused reminder-guard unit tests.

## Best Possible Solution

Settle the reminder-guard UX, then land one complete implementation: either a fully typed, documented, generated, changelogged, and tested `messages.unscheduledReminderNote` config option, or the competing strip/system-annotation path that prevents the internal note from appearing as agent-authored user text.

## Reproduction Assessment

Yes. Current main has a focused source and test path: a reply payload such as `I'll remember that` or `I'll remind you tomorrow morning` with `successfulCronAdds: 0` and no same-session active cron reaches `appendUnscheduledReminderNote`, and existing tests assert the appended-note behavior.

## Solution Assessment

Unclear. A default-on config opt-out can address the reported pain, but this branch has not completed the public config contract and maintainers should deliberately choose it over #39330's delivery-stripping approach before merge.
