## Summary

This should stay open: the item has the protected `maintainer` label, and current `main` still has a source-reproducible iMessage group-drop path when the group registry check runs without the existing `hasGroupAllowFrom` bypass.

## What This Changes

The issue requests visible diagnostics, and possibly startup or doctor warnings, for iMessage group messages dropped when `groupPolicy=allowlist` is configured without `channels.imessage.groups` entries.

## Best Possible Solution

Keep this open for an owner decision, then add focused iMessage coverage and either warn clearly on the misconfiguration or align the iMessage call with the shared `hasGroupAllowFrom` contract.

## Reproduction Assessment

Yes, by source inspection: current `main` can allow the sender through `groupAllowFrom` and then still drop the same group message at the registry gate because iMessage does not pass `hasGroupAllowFrom` into `resolveChannelGroupPolicy`. I did not run a live macOS iMessage monitor in this read-only review.

## Solution Assessment

Unclear. The proposed warning and doctor checks would improve observability, but the narrowest maintainable fix may be to use the existing shared bypass plus a regression test, with a warning reserved for truly empty group allowlists.
