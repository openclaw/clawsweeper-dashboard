## Summary

Keep open. The report is credible and not fully superseded: the local embedded TUI path can still reach gateway-dependent scope-upgrade failures without a clear pairing-required backoff contract, while the related approval-race/recovery work remains open in https://github.com/openclaw/openclaw/issues/74484.

## What This Changes

The issue reports that embedded TUI scope-upgrade retries generate fresh pending approval request IDs fast enough to make manual device approval unreliable.

## Best Possible Solution

Add bounded local embedded TUI handling for pairing-required scope upgrades so a turn stops or backs off without churning requests, while keeping approval semantics aligned with the canonical pairing recovery contract.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The release steps and source path are strong enough to make this source-reproducible: `openclaw chat` runs local embedded TUI, gateway scope upgrades create request IDs, and pending replacements can invalidate earlier IDs.

## Solution Assessment

Unclear. TUI backoff on pairing-required looks like the narrowest safe fix, while adding an atomic `--latest` approval path needs maintainer approval because it changes an auth-sensitive approval contract.
