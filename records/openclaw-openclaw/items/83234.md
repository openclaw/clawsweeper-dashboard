## Summary

Keep open: the prior token-scope fix for https://github.com/openclaw/openclaw/issues/78727 does not cover the reported retry/id churn. Current source still replaces a pending request whenever the next request has a different or narrower scope set than the already-merged pending request, which can keep invalidating the approval ID during fast TUI retries.

## What This Changes

The issue reports that TUI scope-upgrade retries keep replacing pending request IDs, making manual approval unreliable on v2026.5.12.

## Best Possible Solution

Stabilize the existing pending scope-upgrade request across equivalent, subset, or already-merged retry surfaces, while preserving exact-ID approval and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: seed one pending upgrade, then repeat requestDevicePairing for the same device/publicKey with a subset or differently merged scope set; current code takes the replacement path and issues a new requestId. I did not run the full packaged TUI flow.

## Solution Assessment

No for the --latest auto-approve direction: that would change approval semantics on a security-sensitive surface. The narrower fix is to make pending-request reconciliation idempotent for already-covered scope upgrade retries and keep explicit approval intact.
