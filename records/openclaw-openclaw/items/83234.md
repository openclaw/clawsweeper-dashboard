## Label Justifications

- `P1`: The report describes a recent release regression that can block first-run TUI tool use until the user escapes a fast approval retry loop.

## Summary

Keep open. The report describes a real approval-race surface: current pairing reconciliation still replaces a pending request when a later retry asks for a narrower or otherwise different scope set, while the CLI's implicit `--latest` path remains preview-only and cannot atomically approve the moving request.

## What This Changes

The issue asks OpenClaw to stop TUI scope-upgrade retries from churning pending device approval request IDs, either by backing off/reusing pending approvals or by adding an atomic latest-approval flow.

## Best Possible Solution

Stabilize pending approval IDs when an existing request already covers a later retry, or make the TUI pause while that pending scope upgrade exists; keep `--latest` preview-only unless maintainers explicitly approve an atomic safety contract.

## Reproduction Assessment

Yes, at source level. If a pending request has been merged to a broader role/scope set and a retry arrives with a narrower set for the same device, current code fails exact-match refresh and replaces the pending `requestId`; I did not run the full TUI loop live in this read-only review.

## Solution Assessment

Unclear. Backing off or reusing already-covered pending requests looks like the safer narrow fix, while atomic `--latest` approval needs an explicit maintainer decision because the CLI currently keeps implicit selection preview-only for safety.
