## Summary

Keep open. Current `main` still matches both reported hot paths, and an open closing PR already exists, so this issue should stay open until that PR or an equivalent fix merges.

## What This Changes

This issue requests eliminating redundant per-request plugin registry loads in web-provider resolution and empty-candidate plugin auto-enable materialization on ARM64.

## Best Possible Solution

Land a focused fix that reuses the active gateway registry for runtime web-provider mapping, filters by requested plugin ids afterward, and adds focused regression coverage for both web-provider active-registry reuse and empty-candidate auto-enable materialization without weakening configured-entry allowlist behavior.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence on current `main`: the web-provider resolver still falls through to `loadOpenClawPlugins` on scoped compatibility misses, and empty-candidate auto-enable materialization still resolves the manifest registry first.

## Solution Assessment

Yes, directionally. Reusing the already-loaded active registry and post-filtering providers is the narrowest maintainable fix for the request-time web-provider path; the empty-candidate optimization is appropriate if it preserves configured allowlist materialization semantics.
