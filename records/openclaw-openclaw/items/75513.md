## Summary

Keep open. Current main still matches the two remaining hot paths described in the report, and the issue is now paired with open PR #75514, so it should stay open until that PR or an equivalent fix lands. The adjacent capability-provider active-registry path is already covered on main, which narrows the remaining work to web-provider registry reuse and a safe auto-enable no-candidate fast path.

## What This Changes

The issue reports request-time ARM64 slowdowns because scoped web-provider resolution and empty auto-enable materialization can reload plugin registries instead of reusing or short-circuiting existing state.

## Best Possible Solution

Review and land PR #75514, or an equivalent narrow patch, so runtime web-provider resolution reuses the active gateway registry with post-filtering and auto-enable materialization avoids unnecessary manifest work without changing configured-entry allowlist behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: scoped web-provider load options with onlyPluginIds miss active-registry compatibility and fall through to loadOpenClawPlugins, while applyPluginAutoEnable still detects candidates and then resolves the manifest registry again during materialization.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix. The issue itself should remain open while PR #75514 is reviewed because the linked PR is the active implementation candidate and has not merged.
