## Summary

Keep open. Current main still contains the two reported hot paths, and the issue is paired with open closing PR #75514, so it should remain open until that PR or an equivalent fix is merged or closed.

## What This Changes

The issue requests removing redundant per-request plugin registry loads by reusing the active runtime registry for web-provider resolution and avoiding a second manifest-registry load when auto-enable candidates are empty.

## Best Possible Solution

Review and land #75514 or an equivalent focused fix, with regression coverage for active-registry web-provider reuse and empty-candidate auto-enable behavior that preserves configured-entry allowlist semantics.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: current main can resolve web providers with a narrower onlyPluginIds set than the active gateway registry key and fall back to loadOpenClawPlugins, and applyPluginAutoEnable can resolve the manifest registry twice when detection returns no candidates.

## Solution Assessment

Yes, with one caveat. Reusing the active registry and filtering providers afterward is the narrowest maintainable web-provider fix; the empty-candidate optimization is appropriate only if it preserves the existing configured-entry allowlist behavior.
