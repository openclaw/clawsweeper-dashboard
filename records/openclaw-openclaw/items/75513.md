## Summary

Keep open. Current main still contains the two reported hot paths, and the issue is paired with open closing PR #75514, so it should remain open until that PR or an equivalent focused fix is merged or closed.

## What This Changes

The issue requests eliminating redundant per-request `loadOpenClawPlugins` calls in web-provider resolution and empty-candidate auto-enable materialization on ARM64.

## Best Possible Solution

Review and land #75514 or an equivalent focused fix, with regression coverage that reuses the active gateway registry for web-provider lookups while filtering by requested plugin ids, and skips the second manifest-registry load only when empty-candidate auto-enable semantics remain unchanged.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main can miss active-registry compatibility when web-provider `onlyPluginIds` differs from the startup registry key, and `applyPluginAutoEnable` still materializes after empty detection by resolving the manifest registry again.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix. Reusing the already-loaded active registry and filtering afterward fits the existing mapper contract; the auto-enable fast path is appropriate if it does not bypass configured-entry allowlist preservation.
