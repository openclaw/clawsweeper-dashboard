## Summary

Keep open. Current main still matches the two remaining hot paths described in the report: scoped web-provider resolution can miss the active gateway registry and fall through to a fresh plugin load, and the auto-enable apply path still resolves the manifest registry again during materialization even when detection returned no candidates. The adjacent capability-provider active-registry path appears already covered on current main, so the remaining repair is narrow and suitable for a focused performance PR.

## What This Changes

Request to reduce ARM64 request latency by reusing the active gateway plugin registry for scoped web provider resolution and avoiding redundant plugin auto-enable manifest-registry materialization when there are no candidates.

## Best Possible Solution

Implement a small core performance fix: have runtime web-provider resolution reuse the active gateway registry and filter by requested plugin ids when the active registry is available, and add a no-candidate auto-enable fast path only when it preserves existing configured-entry allowlist behavior. Cover both with targeted regression tests and a user-facing changelog entry crediting the reporter.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: current main builds scoped web-provider load options with `onlyPluginIds`, while the loader only reuses the active registry on cache-key compatibility and tests assert scoped `onlyPluginIds` is not active-registry compatible; the auto-enable double-resolution path is also directly visible in the call graph.

## Solution Assessment

Yes, directionally. Reusing the already-loaded gateway registry before fallback loading is the narrowest maintainable fix for the request-time web-provider cost, while the auto-enable fast path should be constrained by tests so it does not bypass configured plugin-entry allowlist materialization.
