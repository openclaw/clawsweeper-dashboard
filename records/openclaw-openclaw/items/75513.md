## Summary

Keep open. Current main still contains the two reported request-time plugin-load paths, and this issue is paired with open, unmerged PR #75514 using closing syntax, so the issue should remain open until that PR is reviewed and either landed or closed.

## What This Changes

Avoid redundant per-request `loadOpenClawPlugins` work during web-provider resolution and empty-candidate plugin auto-enable materialization on ARM64 deployments.

## Best Possible Solution

Review and land PR #75514, or replace it with an equivalent narrow fix, adding regression tests for both the active-registry web-provider fast path and the empty-candidate auto-enable path plus a credited changelog entry. This issue should close only after that fix merges.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: scoped web-provider load options cannot match the startup registry cache key, and the auto-enable apply path still resolves materialization after detection even when no candidates are present.

## Solution Assessment

Yes, directionally. Reusing the active gateway registry and filtering by plugin id is the narrow fix for the request-time web-provider load, and an empty-candidate auto-enable shortcut is reasonable if tests preserve configured-entry allowlist behavior.
