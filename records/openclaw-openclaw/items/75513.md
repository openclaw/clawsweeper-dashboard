## Summary

Keep open. Current main still contains the reported web-provider compatibility-miss fallback and the empty-candidate auto-enable registry resolution, and the issue is paired with open closing PR #75514, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests reducing ARM64 per-request latency by avoiding redundant `loadOpenClawPlugins` calls in web-provider resolution and empty-candidate plugin auto-enable materialization.

## Best Possible Solution

Review and land #75514 or an equivalent focused fix, with regression coverage for active-registry web-provider reuse and empty-candidate auto-enable behavior that preserves configured-entry allowlist semantics.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: current main can resolve web providers with a narrower `onlyPluginIds` set than the active gateway registry key and fall back to `loadOpenClawPlugins`, and auto-enable materialization still resolves the manifest registry before checking for empty candidates.

## Solution Assessment

Yes, with one caveat. Reusing the active registry and filtering providers afterward is the narrowest maintainable web-provider fix; the empty-candidate optimization is appropriate only if it preserves the existing configured-entry allowlist behavior.
