## Summary

Keep open. Current main covers several adjacent Telegram media-loss paths, but local/generated Markdown image targets still remain text and never reach media normalization, so the reported ordinary-reply failure is not fully resolved.

## What This Changes

The issue asks for Telegram ordinary assistant replies with text plus local/generated image content to deliver both the text and the image attachment.

## Best Possible Solution

Add a narrow, access-controlled promotion path for Telegram/local generated Markdown images that accepts only approved managed, sandbox, or workspace media and then reuses the existing reply media normalizer; keep arbitrary host paths as text.

## Reproduction Assessment

Yes, source-level. A Telegram opt-in reply such as `![x](/root/.openclaw/media/tool-image-generation/example.png)` still stays text because the parser only lifts remote Markdown images, and current tests explicitly assert local Markdown targets produce no `mediaUrls`.

## Solution Assessment

Unclear. The requested direction is right for generated media, but broadening Markdown parsing to arbitrary local paths would be unsafe; the maintainable fix is an approved allowlist that feeds the existing access-controlled media normalizer.
