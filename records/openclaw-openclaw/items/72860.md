## Summary

Keep open. Current main fixes remote Markdown-image extraction and adjacent Telegram media-loss paths, but local/generated Markdown image targets still remain text before reply media normalization can attach them.

## What This Changes

The issue asks for Telegram ordinary assistant replies with text plus generated/local images to preserve attachments through Markdown parsing, final-payload suppression, and Telegram delivery.

## Best Possible Solution

Add a narrowly scoped local Markdown-image promotion path for approved generated, managed, workspace, or sandbox targets, then reuse the existing reply media normalizer and access checks.

## Reproduction Assessment

Yes. A source-level reproduction is a Telegram opt-in reply such as `![x](./generated.png)`: current parsing leaves it as text with no `mediaUrls`, and tests explicitly assert local Markdown targets are not lifted.

## Solution Assessment

Unclear. The requested direction is right, but the safest implementation should not broaden parsing to arbitrary local paths; it should pass only approved local candidates into the existing guarded normalization path.
