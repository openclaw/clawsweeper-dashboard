## Summary

Keep open. Current main has fixed adjacent media-loss paths, including preserving unsent media after block-streamed text and sending media-bearing segmented Telegram finals through the payload path, but the reported local/generated Markdown-image path is still not implemented: the shared parser only lifts remote HTTPS Markdown images and current tests still assert local Markdown image targets remain plain text with no media URLs.

## What This Changes

The issue requests preserving image attachments for Telegram ordinary assistant replies, especially local/generated Markdown image content, when text is also delivered.

## Best Possible Solution

Define a narrow safe-local-media rule for generated or agent-workspace Markdown image destinations, route accepted paths through the existing reply media normalizer and media access checks, and keep unsafe absolute, traversal, home-dir, and host file URLs rejected with focused parser, final-payload, and Telegram delivery regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the parser/final-payload path with extractMarkdownImages enabled and a local Markdown image target such as `![x](/root/.openclaw/media/tool-image-generation/example.png)`; current tests assert the result remains text with mediaUrls undefined.

## Solution Assessment

Unclear. The issue's requested direction is valid, but the best fix is not to lift all local Markdown image paths; maintainers should first define the safe generated/local path boundary and then reuse existing media normalization and access controls.
