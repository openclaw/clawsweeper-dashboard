## Summary

Keep open. Current main covers adjacent remote Markdown-image extraction, block-stream final payload preservation, and Telegram media delivery, but local/generated Markdown image targets still remain text before media normalization; changing that policy needs a narrow security-approved allowlist.

## What This Changes

Allow ordinary Telegram assistant replies containing local/generated Markdown image syntax to deliver the image attachment instead of only the text.

## Best Possible Solution

Add a narrowly scoped local/generated Markdown-image promotion path that accepts only approved media candidates and then reuses the existing access-controlled reply media normalizer.

## Reproduction Assessment

Yes, source-level. In Telegram opt-in delivery, `![x](./generated.png)` reaches the Markdown parser, but only `http(s)` candidates are promoted, and current tests assert local Markdown image targets keep `mediaUrls` undefined.

## Solution Assessment

Unclear. The requested direction is right for generated media, but the maintainable fix should not promote arbitrary local paths; it should define an approved local-media allowlist and keep existing access checks in the delivery normalizer.
