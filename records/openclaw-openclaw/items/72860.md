## Summary

Keep open. Current main has adjacent media-loss fixes for Telegram, including remote Markdown-image extraction and preserving unsent media after streamed text, but the specific local/generated Markdown-image path remains rejected by the shared parser and its tests. The remaining fix needs a maintainer/security decision on which local paths may be promoted to outbound media.

## What This Changes

Allow Telegram ordinary assistant replies containing text plus generated/local Markdown image references to deliver the image attachment instead of only the text.

## Best Possible Solution

Define a narrow safe-local-media rule for generated OpenClaw media and allowed workspace/sandbox paths, then let accepted Markdown-image targets flow into mediaUrls so the existing reply media normalizer and media access checks can stage or reject them consistently.

## Reproduction Assessment

Yes. A source-level reproduction is to run the parser/final-payload path with extractMarkdownImages enabled and a local/file Markdown image target; current tests assert it remains plain text with mediaUrls undefined.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix is not to lift all local Markdown images; it should reuse the existing managed-media and access-control path after maintainers define the safe target set.
