## Summary

Keep open. Current main still leaves the local/generated Markdown-image case unresolved: explicit `MEDIA:`/`mediaUrls` paths and Telegram media delivery are covered, but local Markdown image targets are still treated as text and changing that crosses the local-media trust boundary.

## What This Changes

The issue asks OpenClaw to preserve image attachments on ordinary Telegram assistant replies, including generated/local Markdown image outputs, rather than delivering only text.

## Best Possible Solution

Define a safe generated/managed-local-media promotion policy, or require generated images to flow through explicit `MEDIA:`/`mediaUrls`, then add parser, reply-pipeline, Telegram delivery tests, and redacted live Telegram proof.

## Reproduction Assessment

Yes, source-level. A Telegram reply containing a local Markdown image target still remains text because the parser only lifts remote Markdown image candidates, and current tests explicitly assert local targets produce no `mediaUrls`.

## Solution Assessment

Unclear as a direct arbitrary local Markdown lift. The maintainable path needs a maintainer-approved generated/local media allowlist tied to existing file-read and media type checks, or explicit `MEDIA:`/`mediaUrls` metadata from the image-producing path.
