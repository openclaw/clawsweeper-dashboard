## Summary

Keep open. Current main fixed adjacent Telegram/media-loss paths, but the core local/generated Markdown image case is still intentionally rejected: Telegram opts into Markdown image extraction, yet the shared parser only lifts remote HTTPS Markdown image destinations and tests still assert local Markdown image targets remain plain text without media URLs.

## What This Changes

The issue asks OpenClaw to preserve image attachments for ordinary Telegram assistant replies that combine text with local/generated Markdown image references.

## Best Possible Solution

Define a narrow safe-local Markdown image rule, likely limited to managed generated media and agent/workspace paths that already pass reply media normalization, then route accepted targets through the existing media access checks while keeping file URLs, traversal, home-dir, arbitrary absolute host paths, and loopback/private remote URLs rejected unless explicitly allowed by existing policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse or build reply payloads with extractMarkdownImages enabled and a local Markdown image target such as `![x](file:///etc/passwd)` or a generated absolute path; current tests assert those remain text with no mediaUrls.

## Solution Assessment

Unclear. The requested direction is valid, but lifting all local Markdown image paths would be too broad; the maintainable fix needs a reviewed safe-path boundary and should reuse the existing reply media normalizer and access checks.
