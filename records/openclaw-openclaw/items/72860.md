## Summary

Keep open. Current main still leaves local/generated Markdown image targets as ordinary text even when Telegram opts into Markdown image extraction, so the reported normal assistant reply path can still fail to produce mediaUrls; adjacent block-stream media preservation and Telegram media delivery once mediaUrls exist are covered.

## What This Changes

Allow ordinary Telegram assistant replies containing text plus a local/generated Markdown image to deliver the image attachment instead of only the text.

## Best Possible Solution

Define a narrow safe-local Markdown image promotion rule for opt-in channels that reuses reply media normalization and outbound media access checks while continuing to reject arbitrary host paths, traversal, file URLs, home paths, UNC paths, and unsafe remote URLs.

## Reproduction Assessment

Yes. A source-level reproduction is parsing a Telegram-enabled reply like `![x](/tmp/generated.png)` with `extractMarkdownImages: true`; current parser/tests leave local targets as text with mediaUrls undefined.

## Solution Assessment

Unclear. Promoting local/generated Markdown images is the right product direction, but the best implementation needs maintainer-owned local-media policy rather than simply accepting all local parser destinations.
