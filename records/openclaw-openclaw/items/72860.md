## Summary

Keep open. Current main and the latest shipped release still only promote remote Telegram Markdown images into reply media, while local/generated Markdown image targets remain text; adjacent block-stream and Telegram delivery loss paths are covered, but the local Markdown promotion boundary remains unresolved.

## What This Changes

The issue asks OpenClaw to preserve image attachments in ordinary Telegram assistant replies that include text plus local or generated image media, especially when images appear as Markdown.

## Best Possible Solution

Add a narrowly scoped, maintainer-approved local/generated Markdown image promotion path for Telegram that only accepts safe media candidates and then reuses the existing reply media normalizer and Telegram media delivery tests.

## Reproduction Assessment

Yes, source-level. A Telegram opt-in reply like `![x](./generated.png)` still remains text because the parser only lifts remote Markdown images, and current tests explicitly assert local Markdown targets produce no `mediaUrls`.

## Solution Assessment

Unclear as written. The report points at the right remaining gap, but the maintainable fix should not accept arbitrary local Markdown paths; it needs an approved local/generated media allowlist tied into the existing access-controlled normalizer.
