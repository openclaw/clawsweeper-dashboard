## Summary

Keep open. Current main has fixed adjacent media-loss behavior, but it still does not implement the reported local/generated Markdown-image path: Markdown image extraction remains remote-only, and tests explicitly assert local Markdown image targets stay plain text without media URLs.

## What This Changes

The issue requests that ordinary Telegram assistant replies preserve image attachments when reply text includes local or generated Markdown image syntax.

## Best Possible Solution

Define a narrow allow rule for generated or managed OpenClaw media paths in Markdown image destinations, feed accepted targets through the existing reply media normalizer and access checks, and keep arbitrary absolute paths, file URLs, traversal, home-dir, loopback, and host-secret paths rejected with focused parser, final-payload, and Telegram delivery coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse or build a final reply payload with extractMarkdownImages enabled and a local target such as `![x](/root/.openclaw/media/tool-image-generation/example.png)`; current parser and final-payload tests show local targets produce no mediaUrls.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix is not to promote every local Markdown image path; it should be limited to safe generated/managed media and reuse the existing media access normalization path.
