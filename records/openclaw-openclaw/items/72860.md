## Summary

Keep open. Current main covers adjacent Telegram media-loss cases, but the reported local/generated Markdown image path still does not become outbound media: Telegram opts into Markdown image extraction, while the shared parser only lifts validated remote HTTP(S) Markdown image destinations and current tests still assert local Markdown image targets remain plain text.

## What This Changes

The issue asks OpenClaw to preserve image attachments in ordinary Telegram assistant replies, especially text plus local/generated Markdown image replies that currently deliver only text.

## Best Possible Solution

Define a narrow safe-local Markdown image rule that only promotes managed generated media and already-authorized workspace or sandbox media through the existing reply media normalizer and outbound media access checks, while continuing to reject arbitrary file URLs, traversal, home-dir paths, unsafe absolute host paths, UNC paths, data URLs, and private or loopback remote URLs unless existing policy explicitly allows them.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse or build reply payloads with extractMarkdownImages enabled and a local Markdown image target; current tests assert those inputs keep mediaUrls undefined.

## Solution Assessment

Unclear. The issue’s direction is valid, but accepting all local Markdown image paths would be too broad; the maintainable fix should reuse the existing reply media path normalizer and access checks with an explicitly reviewed safe-path boundary.
