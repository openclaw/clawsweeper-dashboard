## Summary

Keep open. Current main covers adjacent Telegram text-plus-media and block-stream final delivery paths, but the reported local/generated Markdown image case still remains: Telegram opts into Markdown image extraction, while the shared parser only lifts validated remote HTTPS Markdown image targets and tests still assert local Markdown image paths remain plain text with no media URLs.

## What This Changes

Request to make Telegram ordinary assistant replies preserve image attachments from text-plus-local/generated Markdown image output instead of delivering only the text.

## Best Possible Solution

Add a narrow safe-local Markdown image rule for Telegram and other opt-in channels that lifts only approved managed/generated, workspace, or sandbox-mapped image paths into mediaUrls, then routes them through the existing reply media normalizer and outbound local-media access checks while continuing to reject arbitrary host paths, file URLs, traversal, home-directory paths, UNC paths, and unsafe remote URLs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Telegram-enabled reply such as `![x](/tmp/generated.png)` with `extractMarkdownImages: true`; current parser tests show local targets remain text with mediaUrls undefined, so the Telegram delivery path never receives media to send.

## Solution Assessment

Unclear. Promoting generated/local Markdown images is the right direction for the bug, but accepting all local paths would be unsafe; the maintainable fix should reuse the existing local media normalization and access policy rather than broadening the parser directly.
