## Summary

Keep open. Current main has fixed adjacent media-loss paths, but the central reported local/generated Markdown-image path still remains remote-only: Telegram opts into Markdown image extraction, yet the shared parser only promotes http(s) Markdown image targets and tests still assert local targets stay plain text with no media payload.

## What This Changes

The issue asks OpenClaw to preserve image attachments for ordinary Telegram assistant replies that contain text plus local/generated image references, especially Markdown image syntax.

## Best Possible Solution

Define a safe local/generated media rule for Markdown image destinations, route accepted paths through the existing reply media path normalizer and outbound media access checks, and keep unsafe file URLs, traversal, home-directory, arbitrary absolute host paths, and loopback/private remote URLs rejected with focused parser, final-payload, and Telegram delivery coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Telegram final reply with extractMarkdownImages enabled and a local Markdown image target such as `![x](/root/.openclaw/media/tool-image-generation/example.png)`; current code leaves it as text, and current tests assert local Markdown image targets have mediaUrls undefined.

## Solution Assessment

Unclear. The issue's direction is valid, but the best fix is not to lift every local Markdown image destination; the maintainable path is a constrained safe-local media policy that reuses existing normalization and access checks.
