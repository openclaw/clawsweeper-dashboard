## Summary

Keep open. Current main covers the adjacent Telegram media-delivery and block-stream final-payload paths, but local/generated Markdown image targets still remain plain text and never reach the reply media normalizer.

## What This Changes

Make Telegram ordinary assistant replies deliver image attachments when a text reply also contains local/generated Markdown image content.

## Best Possible Solution

Add a narrowly scoped local/generated Markdown-image promotion path for Telegram opt-in replies that only admits managed media, sandbox/workspace paths, or existing read-approved media, then reuse the current media normalizer.

## Reproduction Assessment

Yes, source-level. With Telegram Markdown-image extraction enabled, a reply like `![x](./generated.png)` reaches the parser, but current code only promotes http(s) Markdown destinations and tests assert local targets produce no `mediaUrls`.

## Solution Assessment

Unclear. Promoting generated/local Markdown images is directionally right, but the best fix must use a managed/sandbox/workspace allowlist rather than accepting arbitrary local-looking Markdown paths.
