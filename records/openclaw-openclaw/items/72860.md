## Summary

Keep open. Current main covers the adjacent auto-parse, block-stream final-media preservation, and Telegram media-bearing payload delivery paths, but it still intentionally promotes only remote Markdown image targets; local/generated Markdown image promotion needs a maintainer-approved local-media policy.

## What This Changes

The issue asks OpenClaw to preserve Telegram ordinary assistant text-plus-image replies, including local/generated Markdown image attachments, instead of delivering only text.

## Best Possible Solution

Define a safe generated or managed-local-media promotion policy, or require generated images to flow through explicit MEDIA/mediaUrls metadata, then add parser, reply-pipeline, Telegram delivery tests, and redacted live Telegram proof.

## Reproduction Assessment

Yes, source-level. With Telegram-style `extractMarkdownImages: true`, current main still leaves local Markdown image targets as text because the parser only lifts remote Markdown image candidates and the tests assert local targets produce no mediaUrls.

## Solution Assessment

Unclear as a direct arbitrary local Markdown lift. The maintainable path needs an approved generated/local media allowlist tied to existing file-read and media type checks, or explicit MEDIA/mediaUrls metadata from the image-producing path.
