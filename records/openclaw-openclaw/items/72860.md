## Summary

Keep open. Current main still leaves the reported local/generated Markdown-image path unresolved: Telegram opts into Markdown image extraction, but the shared parser only promotes validated remote http(s) image destinations, so local Markdown images remain text and never reach the existing local-media normalizer.

## What This Changes

Deliver images from normal Telegram assistant replies when the reply contains text plus local/generated Markdown image syntax.

## Best Possible Solution

Add a narrowly scoped safe-local Markdown image promotion path for opt-in channels that accepts only approved managed/generated, workspace, or sandbox-mapped image paths and then reuses the existing reply media normalizer and outbound local-media access checks.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Telegram opt-in reply such as `![x](./generated.png)` with `extractMarkdownImages: true`; current code leaves local targets as text with `mediaUrls` undefined, so Telegram never receives an attachment.

## Solution Assessment

Unclear. Promoting local Markdown images is the right direction, but the maintainable fix should not broaden the parser to arbitrary local-looking paths; it should route candidates through the existing local media policy.
