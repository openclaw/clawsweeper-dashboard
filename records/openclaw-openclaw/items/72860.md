## Summary

Keep open. Current main covers adjacent Telegram text-plus-media delivery and block-stream final media preservation, but the reported local/generated Markdown image path still remains text-only because Markdown image extraction only promotes approved remote HTTPS targets.

## What This Changes

Ensure ordinary Telegram assistant replies with text plus local/generated Markdown image attachments deliver both the text and the image.

## Best Possible Solution

Add an opt-in safe-local Markdown image rule that lifts only approved managed/generated, workspace, or sandbox-mapped image paths into mediaUrls, then route them through the existing reply media normalizer while rejecting arbitrary host paths, file URLs, traversal, home-directory paths, UNC paths, and unsafe remote URLs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Telegram-enabled reply like `![x](/root/.openclaw/media/tool-image-generation/example.png)` with `extractMarkdownImages: true`; current code only promotes remote http(s) Markdown targets and tests assert local targets remain plain text with no mediaUrls.

## Solution Assessment

Unclear until the local attachment exposure policy is decided. Promoting generated/local Markdown images is the right direction for the user-visible bug, but the maintainable fix should reuse existing local media normalization and access checks instead of broadening the parser to arbitrary local paths.
