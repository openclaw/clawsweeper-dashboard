## Summary

Keep open. Current main and the latest shipped release still only lift remote Telegram Markdown images into media; local/generated Markdown image targets remain text, while the adjacent auto-parse, block-stream, and Telegram media-bearing delivery paths are now covered.

## What This Changes

The issue asks OpenClaw to stop Telegram ordinary assistant replies from dropping image attachments, especially local/generated image targets emitted in Markdown.

## Best Possible Solution

Define a safe local/generated Markdown image promotion policy, or route generated images through existing MEDIA/mediaUrls metadata, then add parser, reply-pipeline, Telegram delivery regression coverage plus redacted live Telegram proof.

## Reproduction Assessment

Yes, source-level. A Telegram opt-in reply containing a local Markdown image target still remains text because the parser only lifts remote Markdown image candidates, and current tests explicitly assert local targets produce no mediaUrls.

## Solution Assessment

Unclear as a direct arbitrary local Markdown lift. The maintainable path needs a maintainer-approved local/generated media allowlist tied to existing media access checks, not a blanket acceptance of local Markdown destinations.
