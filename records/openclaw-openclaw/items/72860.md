## Summary

Keep open. Current main fixed adjacent Telegram media-loss paths, but the remaining local/generated Markdown-image path is still not implemented: Telegram opts into Markdown-image extraction, while the shared parser still promotes only validated remote HTTP(S) Markdown images and the regression tests still assert local Markdown-image targets remain text-only.

## What This Changes

Normal Telegram assistant replies containing text plus local/generated Markdown image attachments should deliver both the text and the image.

## Best Possible Solution

Add a narrow reply-pipeline implementation that promotes only safe generated/OpenClaw-managed Markdown-image destinations into media payloads, then routes them through the existing reply media normalizer/access boundary. Keep arbitrary local, file-style, private-network, and unsafe targets blocked, and cover parser, final payload, block-stream, and Telegram delivery behavior with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is to parse or build reply payloads with `extractMarkdownImages: true` and a local/generated Markdown-image destination; current code leaves it as text with no `mediaUrls`, and existing tests assert the same local-target behavior.

## Solution Assessment

Unclear. The requested direction is valid, but broadly lifting all local Markdown-image targets in `src/media/parse.ts` would not be the best fix; maintainers should first define the safe generated/local media boundary and reuse the existing media normalization path.
