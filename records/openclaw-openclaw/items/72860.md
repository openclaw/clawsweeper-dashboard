## Summary

Keep open. Current main covers adjacent Telegram text-plus-media delivery paths, but the reported local/generated Markdown image case still remains because Telegram opts into Markdown image extraction while the shared parser only promotes remote HTTPS Markdown images and current tests assert local Markdown image targets stay as text without media URLs.

## What This Changes

Ensure ordinary Telegram assistant replies containing text plus local/generated Markdown image attachments deliver both the text and image instead of silently dropping the attachment.

## Best Possible Solution

Add a narrow safe-local Markdown image rule that routes accepted local/generated image destinations through the existing reply media normalizer and outbound local-media access checks, while continuing to reject arbitrary host paths, file URLs, traversal, home-directory paths, UNC paths, loopback/private remote URLs, and other untrusted targets.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Telegram-enabled reply with extractMarkdownImages and a local Markdown image target such as `![x](/tmp/generated.png)`; current parser tests show local targets remain text with no mediaUrls.

## Solution Assessment

Unclear. Promoting generated/local Markdown images is the right direction, but accepting all local paths would be unsafe; the maintainable fix should reuse the existing reply-media path normalization and access policy rather than adding a broad parser bypass.
