## Summary

Keep open. Current main covers adjacent remote Markdown-image extraction and streamed final-media preservation, but the reported local/generated Markdown image path still remains blocked by the shared parser and existing tests assert that behavior.

## What This Changes

The issue requests that ordinary Telegram assistant replies preserve local/generated Markdown image attachments alongside delivered text.

## Best Possible Solution

Add an opt-in safe-local Markdown image rule that lifts only approved generated, workspace, sandbox, or managed-media paths and routes them through existing reply media normalization and access checks.

## Reproduction Assessment

Yes. A source-level reproduction is a Telegram-enabled reply like `![x](/tmp/generated.png)` with `extractMarkdownImages: true`; current parser tests show local targets remain text with `mediaUrls` undefined.

## Solution Assessment

Unclear. Promoting local/generated Markdown images is the right direction, but the best fix must reuse the local media access policy instead of broadening the parser to arbitrary host paths.
