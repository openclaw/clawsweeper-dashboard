## Summary

Keep open. Current main still only promotes remote Markdown image targets into media; local/generated Markdown targets remain visible text, while the adjacent auto-parse, block-stream suppression, and Telegram media-bearing send paths are now covered.

## What This Changes

The issue requests that Telegram ordinary assistant replies preserve text plus local/generated image attachments instead of delivering only the text.

## Best Possible Solution

Define a safe generated/managed-local-media promotion policy, or require generated images to flow through existing MEDIA/mediaUrls metadata, then add parser, reply-pipeline, Telegram delivery tests, and redacted live Telegram proof.

## Reproduction Assessment

Yes, source-level. A Telegram opt-in reply containing a local Markdown image target still stays text because the parser only lifts remote Markdown image candidates, and current tests explicitly assert local targets produce no mediaUrls.

## Solution Assessment

Unclear as a direct arbitrary local Markdown lift. The maintainable path needs a maintainer-approved generated/local media allowlist tied to existing file-read and media type checks, or explicit MEDIA/mediaUrls metadata from the image-producing path.
