## Summary

Keep open. Current main covers several adjacent Telegram media-loss paths, but the reported local/generated Markdown image case still remains: Telegram opts into Markdown image extraction while the shared parser only promotes validated remote HTTP(S) Markdown image destinations, and current tests still assert local Markdown image targets remain text with no media URLs.

## What This Changes

The issue requests fixing Telegram ordinary assistant replies so text plus local/generated image attachments survive the normal Markdown-image, block-streaming, and lane-splitting reply pipeline.

## Best Possible Solution

Define a narrow safe-local Markdown image rule for managed generated media and agent/workspace paths that already pass reply media normalization and outbound media access checks, while continuing to reject arbitrary file URLs, traversal, home-dir paths, unsafe absolute host paths, UNC paths, data URLs, and private or loopback remote URLs unless existing policy explicitly allows them.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse or build reply payloads with extractMarkdownImages enabled and a local Markdown image target; current tests assert file/absolute local targets remain text and produce no mediaUrls.

## Solution Assessment

Unclear. The issue's direction is valid, but accepting all local Markdown image paths would be too broad; the maintainable fix should reuse the existing reply media path normalizer and media access checks with an explicitly reviewed safe-path boundary.
