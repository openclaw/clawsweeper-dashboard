## Summary

Keep open. Current main covers the adjacent Telegram media-loss paths for remote Markdown images, block-stream final payloads, and Telegram media delivery, but local/generated Markdown image targets still remain text and never reach the reply media normalizer.

## What This Changes

The issue requests that ordinary Telegram assistant replies preserve text plus image attachments, including generated/local Markdown image outputs, through parsing and delivery.

## Best Possible Solution

Add a narrowly scoped Telegram/local generated Markdown-image promotion path that only accepts approved managed, sandbox, or workspace media and then reuses the existing reply media normalizer.

## Reproduction Assessment

Yes, source-level. A Telegram opt-in reply like `![x](./generated.png)` still stays text because the parser only lifts remote Markdown images, and current tests explicitly assert local Markdown targets produce no `mediaUrls`.

## Solution Assessment

Unclear. The report's direction is right for generated media, but the best fix should not broaden the parser to arbitrary local-looking paths; it should allow only approved local media candidates through the existing access-controlled normalizer.
