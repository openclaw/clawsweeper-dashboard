## Summary

Keep open. Current main still leaves the reported local/generated Markdown-image path unresolved: Telegram opts into Markdown image extraction, but the shared parser only promotes validated remote HTTPS Markdown image targets, and tests still assert local targets remain plain text with no media URLs.

## What This Changes

Request to preserve Telegram ordinary assistant reply image attachments when text and local/generated Markdown image output are delivered together.

## Best Possible Solution

Add a narrow safe-local Markdown image rule for opt-in channels that lifts only approved managed/generated, workspace, or sandbox-mapped image paths into mediaUrls, then routes them through existing reply media normalization and outbound local-media access checks.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Telegram-enabled reply such as `![x](/tmp/generated.png)` with `extractMarkdownImages: true`; current parser tests show local targets remain text with mediaUrls undefined, so Telegram never receives media to send.

## Solution Assessment

Unclear. Promoting generated/local Markdown images is the right direction for the bug, but accepting all local paths would be unsafe; the maintainable fix should reuse existing local media normalization and access policy rather than broadening the parser directly.
