## Summary

Keep open: current main and v2026.6.8 still route ordinary inbound runs through a per-session lane and then the shared process-wide `main` lane, while the requested per-agent command-lane/config surface remains missing and needs maintainer product direction.

## What This Changes

The issue requests per-agent command lanes or equivalent per-agent concurrency isolation so long-running work for one agent does not starve other agents on a shared gateway.

## Best Possible Solution

Define one scheduler-owned lane isolation model that keeps per-session serialization, preserves a real gateway-wide cap, uses server-controlled lane ids, and adds aligned schema, docs, and fresh-install/upgrade tests for per-agent or grouped lane routing.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still queue ordinary runs through a per-session lane and then the process-wide `main` lane when no explicit lane is supplied. I did not run a live 30-agent Feishu deployment, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No, not as an automatic fix. Per-agent isolation is a plausible direction, but the best fix requires a maintainer-owned scheduler/config decision that preserves global caps and avoids unsafe request-controlled lane overrides.
