## Summary

Keep open: current main still uses the shared foreground `main` lane for ordinary inbound agent work, the requested per-agent command-lane config is absent, and the open closing PR is conflicting with unresolved resource-control and security review concerns.

## What This Changes

The issue requests per-agent command lane isolation so long-running work for one agent cannot starve other agents sharing a gateway.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, bounded lane names, focused config/docs/tests, and a preserved global resource cap.

## Reproduction Assessment

Yes for source-level capability proof: current main routes ordinary auto-reply and embedded runs to `CommandLane.Main` unless a caller supplies a lane, and config/docs expose only global foreground concurrency. The reported Feishu deployment latency has logs but was not live-reproduced.

## Solution Assessment

Unclear as proposed. Per-agent command lanes are plausible, but the maintainable fix needs agreed config semantics, server-owned lane selection, bounded lane creation, and a global resource cap before implementation lands.
