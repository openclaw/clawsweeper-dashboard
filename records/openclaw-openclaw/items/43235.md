## Summary

Keep open: current main still routes ordinary inbound agent work through the shared foreground `main` lane, the requested per-agent command-lane config surface is absent, and the open closing PR remains conflicting with design and security concerns.

## What This Changes

The issue requests per-agent or per-agent-group foreground command lanes so long-running work by one agent cannot starve other agents in multi-agent deployments.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, bounded lane names, focused config/docs/tests, and a preserved global resource cap.

## Reproduction Assessment

Yes for source-level behavior: current main routes ordinary auto-reply and embedded runs to `CommandLane.Main` unless an explicit lane is passed, and config/docs expose only `agents.defaults.maxConcurrent`. The reported Feishu deployment latency has logs but was not live-reproduced.

## Solution Assessment

Unclear as proposed. Per-agent command lanes are plausible, but the maintainable fix needs agreed config semantics, server-owned lane selection, bounded lane creation, and a global resource cap before implementation lands.
