## Summary

Keep open. Current main still sends ordinary foreground agent work through the process-wide main lane unless a caller supplies a lane, and the open closing PR is still conflicting with unresolved resource-control and design review feedback.

## What This Changes

Request per-agent command lanes/configuration so multi-agent deployments can isolate heavy agents from the shared foreground lane.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent lane routing, focused concurrency/docs coverage, and a preserved global resource cap.

## Reproduction Assessment

Yes for source behavior: ordinary runs without an explicit lane still share CommandLane.Main, and the config/docs surface exposes only the global agents.defaults.maxConcurrent path. The exact Feishu deployment latency was not live-reproduced.

## Solution Assessment

Unclear. Per-agent command lanes are a plausible direction, but the maintainable solution needs server-owned lane selection, bounded lane creation/resource caps, and agreed config semantics before implementation.
