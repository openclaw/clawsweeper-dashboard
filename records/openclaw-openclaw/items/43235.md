## Summary

Keep open: current main and the latest release still route ordinary foreground agent work through the shared `main` lane, and the linked closing PR remains open and conflicting, with unresolved resource-control concerns in review.

## What This Changes

Requested support for per-agent command lanes so long-running work from one agent does not starve other agents sharing the same gateway.

## Best Possible Solution

Land a maintained per-agent/default lane design that keeps lane selection server-owned, bounds configured lane names, preserves a global resource cap, and covers config, gateway, auto-reply, docs, tests, and upgrade behavior.

## Reproduction Assessment

Yes for source-level capability proof: current main routes ordinary auto-reply and embedded runs to `CommandLane.Main` unless a caller supplies a lane, and config/docs expose only global foreground concurrency. The reported Feishu deployment latency has detailed logs but was not live-reproduced.

## Solution Assessment

Unclear as currently proposed. Per-agent command lanes are a plausible direction, but the linked PR should be repaired or replaced to settle server-owned lane selection, bounded lane creation, and global cap semantics before merge.
