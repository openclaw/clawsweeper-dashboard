## Summary

Keep open. Current main still routes ordinary foreground agent work through the shared `main` lane when no caller lane is supplied, there is no per-agent command-lane config surface, and the open linked implementation PR still needs maintainer design/security review rather than a new cleanup close.

## What This Changes

Request support for per-agent command lanes or concurrency budgets so long-running work from one agent does not starve other agents in multi-agent deployments.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, focused concurrency docs/tests, and a preserved global resource cap.

## Reproduction Assessment

Yes for source behavior: ordinary runs without an explicit lane still share `CommandLane.Main`, and the config/docs surface exposes only global `agents.defaults.maxConcurrent`. The exact Feishu `@all` latency was not live-reproduced.

## Solution Assessment

Unclear as proposed. Per-agent command lanes are a plausible direction, but the maintainable solution needs server-owned lane selection, bounded lane creation/resource caps, and agreed config semantics before implementation.
