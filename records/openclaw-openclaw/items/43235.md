## Label Justifications

- `P2`: Shared multi-agent gateways can see multi-minute response delays, but this is a bounded scheduler/config improvement rather than an emergency outage.
- `impact:other`: The concrete impact is cross-agent queue starvation and severe response latency, which does not fit the narrower owned impact categories.

## Summary

Keep open. Current main and v2026.6.6 still route ordinary foreground agent work through one process-wide `main` lane, and the open linked implementation PR is conflicting with unresolved design and security-review concerns.

## What This Changes

Support per-agent command lanes so slow work for one agent does not consume the shared foreground lane used by other agents.

## Best Possible Solution

Adopt one scheduler-owned foreground composition: session serialization, optional agent or agent-group isolation, a preserved gateway-wide cap, and aligned schema, docs, fresh-install tests, and upgrade behavior.

## Reproduction Assessment

Yes, at source level. Current main still routes foreground work through a per-session lane and then the shared `main` lane, while the reporter and Feishu commenter provide logs that match that path; I did not run a live Feishu deployment.

## Solution Assessment

Unclear. Per-agent isolation is likely the right capability, but the config shape, global-cap semantics, network-facing lane trust boundary, and linked PR conflicts need maintainer decision before this is the best safe fix.
