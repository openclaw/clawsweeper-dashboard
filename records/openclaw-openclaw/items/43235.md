## Summary

Keep open: current main and the latest shipped release still route ordinary foreground agent work through the shared `main` lane, and the open closing PR has unresolved resource-control/security concerns plus conflicts.

## What This Changes

The issue requests configurable per-agent command lanes or equivalent per-agent concurrency isolation so heavy work for one agent does not consume the shared foreground `main` lane for every agent.

## Best Possible Solution

Land a maintained per-agent/default lane design that keeps lane selection server-owned, bounds configured lane names, preserves a global resource cap, and covers config, gateway, auto-reply, docs, and upgrade behavior.

## Reproduction Assessment

Yes for source-level capability proof: current main routes ordinary auto-reply and embedded runs to `CommandLane.Main` unless a caller supplies a lane, and config/docs expose only global foreground concurrency. The reported Feishu deployment latency has detailed logs but was not live-reproduced.

## Solution Assessment

Unclear as currently proposed. Per-agent command lanes are a plausible direction, but the existing linked PR should be repaired or replaced to settle server-owned lane selection, bounded lane creation, and global cap semantics before merge.
