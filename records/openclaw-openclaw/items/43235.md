## Label Justifications

- `P2`: Shared multi-agent gateways can experience multi-minute delays for unrelated agents, but the impact is bounded to deployments exercising substantial concurrent foreground work.
- `impact:other`: The concrete impact is cross-agent queue starvation and severe response latency, which does not fit the narrower owned impact categories.

## Summary

Keep open: current main and v2026.6.5 still route ordinary foreground agent work through one process-wide `main` lane, and the linked closing PR remains open and conflicting. The request is valid, but maintainers must decide a canonical configuration and scheduler composition that adds agent isolation without bypassing the gateway-wide resource cap.

## What This Changes

Support per-agent foreground command-lane isolation and concurrency budgets while preserving session ordering and a gateway-wide concurrency cap.

## Best Possible Solution

Use one scheduler-owned composition across every foreground embedded-run entry point: serialize by session, optionally limit by canonical agent identity, and always acquire a process-wide total-cap slot; expose the smallest config surface needed and document fresh-install and upgrade semantics.

## Reproduction Assessment

Yes, from source with high confidence: current foreground runs use the shared process-wide `main` lane, and the supplied deployments show corresponding `lane=main` waits of up to several minutes. A live 30-agent load test was not performed.

## Solution Assessment

No final solution is approved yet. Per-agent isolation is directionally appropriate, but the implementation must preserve a real global cap, cover all foreground entry points, and converge on one minimal config contract rather than creating independent scheduling pools.
