## Summary

Keep open: current main and v2026.6.6 still route ordinary inbound agent work through a per-session lane and then the shared `main` lane, with no supported per-agent command-lane or per-agent foreground concurrency config. The request is source-proven and useful, but it needs maintainer product direction because the fix adds public scheduling/config surface and must preserve a global resource cap and trusted server-owned lane selection.

## What This Changes

The issue requests per-agent command lane/concurrency isolation so heavy work from one agent does not consume the shared main lane for all agents.

## Best Possible Solution

Adopt one maintainer-approved, server-owned lane policy that can isolate selected agents or agent groups while preserving per-session ordering, a gateway-wide cap, strict schema/docs, upgrade-safe defaults, and focused tests for cross-agent parallelism plus same-agent serialization.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 still default foreground embedded runs to the process-wide `main` lane and expose only `agents.defaults.maxConcurrent` as the top-level agent run cap; I did not run a live Feishu deployment.

## Solution Assessment

Unclear as proposed. Per-agent isolation is a plausible core scheduler capability, but the config shape, global-cap semantics, related channel/workload lane overlap, and network-facing lane trust boundary need maintainer direction before an implementation is safe.
