## Label Justifications

- `P2`: Shared multi-agent gateways can see multi-minute response delays, but this is a bounded scheduler/config improvement rather than an emergency outage.
- `impact:other`: The concrete impact is cross-agent queue starvation and severe response latency, which does not fit the narrower owned impact categories.

## Summary

Keep open. Current main and latest release still route ordinary foreground agent work through one process-wide main lane, while the linked implementation PR is open, conflicting, and has unresolved design/security concerns, so cleanup closure is not safe.

## What This Changes

The issue requests per-agent command lane isolation so heavy work from one agent cannot consume the shared foreground lane budget for all agents in a multi-agent gateway.

## Best Possible Solution

Adopt one scheduler-owned foreground composition: session serialization, optional agent-level isolation, and a preserved gateway-wide cap, with schema/docs/tests covering fresh and upgrade behavior.

## Reproduction Assessment

Yes, from source with high confidence: current main still queues a foreground run into a session lane and then a global lane that defaults to main, while gateway startup only installs one process-wide main concurrency cap. I did not run a live Feishu deployment, but the issue and comment logs match that path.

## Solution Assessment

Unclear. Per-agent isolation is likely the right capability, but the config/API shape, global-cap semantics, and linked PR security concerns need maintainer decision before this becomes the best safe fix.
