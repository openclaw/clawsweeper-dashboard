## Summary

Keep open. Current main still routes inbound agent turns through a per-session lane and then a shared global `main` lane, with only defaults-level concurrency exposed; the Feishu discussion adds concrete recent `lane=main` wait evidence. Open PR #73991 directly targets this issue, so this should remain open until that PR lands or maintainers choose a different implementation.

## What This Changes

Request opt-in per-agent command lanes and concurrency so long-running work from one agent does not block other agents sharing the current global main lane.

## Best Possible Solution

Add a backwards-compatible core queue/config path that can derive an effective command lane from agent/default configuration, apply concurrency to those derived lanes, preserve per-session serialization, and document/test the default shared behavior plus opt-in per-agent isolation.

## Reproduction Assessment

Yes. The issue and comment provide a concrete Feishu multi-agent reproduction with `lane=main` waits, and current main source confirms all default inbound runs still share the same global lane after per-session serialization.

## Solution Assessment

Yes, the narrow maintainable direction is an opt-in core queue/config change rather than a channel-specific workaround or separate gateway deployment. Closing before the linked implementation PR merges would not solve the reported production behavior.
