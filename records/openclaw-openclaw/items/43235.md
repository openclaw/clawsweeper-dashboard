## Summary

Keep open. Current main still routes default inbound agent turns through a per-session lane and then a shared process-wide `main` lane; the Feishu comment adds concrete recent `lane=main` wait evidence, and open PR #73991 explicitly targets this issue but has not merged.

## What This Changes

Request opt-in per-agent or per-agent-group command lanes/concurrency so one heavy agent does not starve other agents in a shared gateway.

## Best Possible Solution

Land a backwards-compatible core queue/config change that keeps shared `main` as the default, derives opt-in effective lanes from agent/default config, applies lane caps at gateway startup/reload, and updates docs plus regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows each default inbound run is serialized per session and then enqueued into the shared global `main` lane, while the Feishu comment gives concrete multi-bot steps and `lane=main` wait logs; I did not run a live Feishu load test.

## Solution Assessment

Yes. The narrow maintainable direction is an opt-in core queue/config path rather than a Feishu-specific workaround or separate gateway deployment; PR #73991 matches that direction but remains open and dirty.
