## Summary

Keep open. Current main and the latest release still route ordinary ingress agent runs through the shared default `main` lane unless a caller explicitly supplies a lane, and an open PR already tracks this exact request but has unresolved review/security-sensitive design issues.

## What This Changes

Request per-agent command lanes so long-running work for one agent does not consume the shared global main lane and delay other agents.

## Best Possible Solution

Review and repair or replace the open per-agent command-lane PR with a backward-compatible server-owned config contract, focused lane/concurrency tests, docs, and a preserved global resource cap.

## Reproduction Assessment

Yes for the current source behavior: ordinary ingress runs without an explicit lane share `CommandLane.Main`, and current config has no per-agent command-lane setting. No live Feishu deployment reproduction was run for the reported wait durations.

## Solution Assessment

Unclear as implemented by the open PR until maintainer review resolves lane ownership, ingress override policy, and default concurrency inheritance. The requested direction is reasonable, but the safest solution is server-controlled per-agent lane configuration with bounded resource semantics.
