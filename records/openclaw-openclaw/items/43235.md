## Summary

Keep open. Current main still routes ordinary foreground agent work through the shared `main` lane, has no per-agent command-lane config surface, and the linked implementation PR remains open with unresolved design/security concerns.

## What This Changes

The issue requests per-agent or per-agent-group command lanes so long-running work from one agent does not monopolize the shared foreground lane for other agents.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, bounded lane names, focused config/docs/tests, and a preserved global resource cap.

## Reproduction Assessment

Yes for source-level behavior: ordinary channel runs still resolve to `CommandLane.Main` unless a caller supplies a lane, and the public config surface exposes only global `agents.defaults.maxConcurrent`. The exact Feishu `@all` latency was not live-reproduced.

## Solution Assessment

Unclear as proposed. Per-agent command lanes are plausible, but the maintainable fix needs agreed config semantics, server-owned lane selection, bounded lane creation, and global resource caps before implementation lands.
