## Summary

Keep open. Current `main` and `v2026.5.12` still route ordinary foreground work to the shared `main` lane unless an explicit lane is supplied, and the open implementation PR that references this issue still needs maintainer review for lane ownership and resource-control semantics.

## What This Changes

Request support for per-agent or per-agent-group command lanes so long-running work from one agent does not block other agents in multi-agent deployments.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with a backward-compatible, server-owned per-agent lane config, focused concurrency tests, docs, and a preserved global resource cap.

## Reproduction Assessment

Yes for the current source behavior: ordinary foreground runs without an explicit lane still share `CommandLane.Main`, and the config/docs surface has no per-agent command-lane option. The exact Feishu wait-time numbers were not live-reproduced.

## Solution Assessment

Unclear. The per-agent lane direction is reasonable, but the maintainable fix needs maintainer decisions around server-owned lane selection, request-lane trust, and global resource caps before the linked PR can land.
