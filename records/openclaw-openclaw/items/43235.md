## Label Justifications

- `P2`: The issue has concrete shared-gateway latency impact, but it is a bounded scheduler/config feature gap rather than an emergency outage.
- `impact:other`: The missing isolation can let one busy agent degrade unrelated agents on a shared gateway, which is meaningful operator impact outside the narrower owned impact labels.

## Summary

Keep open: current main and v2026.6.10 still route ordinary inbound agent work through a per-session lane and then the shared `main` lane, and the only linked implementation PR was closed unmerged after security, compatibility, and proof blockers.

## What This Changes

Request per-agent command lanes or concurrency so long-running work for one agent does not consume the shared main lane and delay other agents.

## Best Possible Solution

Keep this as the canonical foreground per-agent lane issue, then design a server-owned per-agent lane/concurrency model that preserves the global cap, updates schema/docs, and covers all foreground embedded-run entry points.

## Reproduction Assessment

Yes, at source level: ordinary channel runs still enter `session:<key>` and then the shared `main` global lane, while the issue and Feishu comment logs match that bottleneck. I did not run a live multi-agent Feishu deployment in this read-only review.

## Solution Assessment

Unclear as originally proposed. Per-agent isolation is the right capability, but maintainers still need to choose a server-owned config shape and security boundary rather than relying on arbitrary request-supplied lanes.
