## Summary

Keep open: current main still uses the shared foreground `main` lane for ordinary inbound agent work, the requested per-agent command-lane config is absent, and the open closing PR at https://github.com/openclaw/openclaw/pull/73991 is still conflicting with unresolved resource-control/security review concerns.

## What This Changes

The issue requests configurable per-agent command lanes so one agent's long-running work does not consume the shared foreground lane for all agents.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, bounded lane names, focused config/docs/tests, and a preserved aggregate resource cap.

## Reproduction Assessment

Yes for source-level capability proof: current main routes ordinary auto-reply and embedded runs to `CommandLane.Main` unless a caller supplies a lane, and config/docs expose only a global foreground cap. The reported Feishu deployment latency has logs but was not live-reproduced.

## Solution Assessment

Unclear as proposed. Per-agent command lanes are plausible, but the maintainable fix needs agreed config semantics, server-owned lane selection, bounded lane creation, and a global resource cap before implementation lands.
