## Summary

Keep open. Current main and v2026.6.6 still route ordinary foreground agent work through one process-wide main lane, and the prior implementation PR was closed unmerged after security/design concerns.

## What This Changes

The issue asks for per-agent command lanes so one busy agent cannot consume shared foreground lane capacity for every other agent in a multi-agent gateway.

## Best Possible Solution

Add a maintainer-approved scheduler/config design that preserves the existing global cap while adding scoped per-agent or explicit lane isolation with upgrade-safe schema, docs, and tests.

## Reproduction Assessment

Yes, at source level. Current main still routes ordinary foreground work through a per-session lane and then the shared main lane, and the reporter plus Feishu commenter provide logs that match that path; I did not run a live Feishu deployment.

## Solution Assessment

No confirmed best fix yet. Per-agent isolation is a plausible capability, but the config shape, global-cap semantics, and network-facing lane trust boundary need maintainer decision before another PR is queued.
