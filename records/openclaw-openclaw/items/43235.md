## Summary

Keep open. Current main and the latest release still route ordinary foreground agent work through the shared `main` lane unless a caller supplies an explicit lane, and https://github.com/openclaw/openclaw/pull/73991 is an open implementation candidate with unresolved review/security design issues.

## What This Changes

Request support for per-agent or agent-group command lanes so heavy agents do not starve other agents on the shared foreground lane.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with a backward-compatible, server-owned per-agent lane config, focused concurrency tests, docs, and a preserved global resource cap.

## Reproduction Assessment

Yes for the current source behavior: ordinary foreground runs without an explicit lane still share `CommandLane.Main`, and the config/schema/docs have no per-agent command-lane surface. The exact Feishu deployment latency was not live-reproduced.

## Solution Assessment

Unclear until maintainer review resolves lane ownership, ingress override policy, and default/global cap semantics. The requested direction is reasonable, but the maintainable solution should keep lane selection server-owned and bounded.
