## Summary

Keep open. Current main still routes normal inbound agent work through the shared `main` lane, the requested per-agent command-lane config surface is absent, and the directly linked implementation PR remains open with unresolved design/security review concerns.

## What This Changes

The issue requests per-agent command lanes so long-running work from one agent does not consume the shared foreground lane for every agent in a multi-agent deployment.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, bounded lane names, focused config/docs/tests, and a preserved global resource cap.

## Reproduction Assessment

Yes for source-level behavior: current main still routes ordinary auto-reply work to `CommandLane.Main` and exposes only global `agents.defaults.maxConcurrent`. The exact Feishu `@all` latency was not live-reproduced.

## Solution Assessment

Unclear as proposed. Per-agent command lanes are plausible, but the maintainable fix needs agreed config semantics, server-owned lane selection, bounded lane creation, and a global resource cap before implementation lands.
