## Summary

Keep open: current main and v2026.6.8 still route foreground inbound runs through a per-session lane and then the process-wide `main` lane, with no per-agent command-lane config. This remains the canonical tracker because earlier duplicates were closed into it and the only linked implementation PR was closed unmerged after security, proof, and merge-readiness blockers.

## What This Changes

The issue requests per-agent or per-agent-group command lanes so long-running work for one agent does not starve other agents on a shared gateway.

## Best Possible Solution

Keep this as the canonical foreground per-agent lane issue, then design a server-owned per-agent lane/concurrency config that preserves the process-wide cap, avoids arbitrary client lane overrides, updates schema/docs, and covers all foreground embedded-run entry points.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 route foreground runs through a session lane plus global `main`, while the strict agent schema and docs expose no per-agent foreground lane or cap; I did not run a live Feishu deployment.

## Solution Assessment

Unclear as originally proposed. Per-agent isolation is the right capability, but maintainers still need to choose the server-owned config shape and security boundary; the previous PR demonstrated why arbitrary request lane overrides are not safe.
