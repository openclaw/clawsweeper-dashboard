## Label Justifications

- `P2`: Shared multi-agent gateways can see multi-minute response delays, but this is a bounded feature/reliability improvement rather than an emergency outage.
- `impact:other`: The concrete impact is cross-agent queue starvation and severe response latency, which does not fit the narrower owned impact categories.

## Summary

Current main and v2026.6.6 still route ordinary foreground agent work through one process-wide `main` lane, so the requested per-agent isolation remains necessary. The linked closing PR is open, conflicting, and has unresolved design/security review concerns, so this issue should stay open for maintainer product/API direction rather than be swept closed.

## What This Changes

Request support for per-agent command lanes so heavy work from one agent does not consume the shared main lane for all agents in multi-agent gateway deployments.

## Best Possible Solution

Adopt one scheduler-owned composition for foreground runs: session serialization, optional agent-level isolation, and a preserved gateway-wide cap, with the minimal config surface and docs/tests for fresh install and upgrade behavior.

## Reproduction Assessment

Yes, from source with high confidence: foreground runs resolve a session lane and then a global lane that defaults to `main`, and gateway startup only sets one process-wide `main` concurrency from `agents.defaults.maxConcurrent`. The issue and follow-up comment also include field logs showing `lane=main` waits in multi-agent Feishu deployments.

## Solution Assessment

Unclear as an implementation path: per-agent isolation is directionally right, but the best fix needs maintainer agreement on a minimal config shape and must preserve the global resource cap while covering all foreground entry points.
