## Summary

Keep open: current main and the latest release still route ordinary inbound agent work through the shared foreground `main` lane, and the open closing PR is not merged and still has unresolved design/security concerns.

## What This Changes

The issue requests per-agent or grouped command lanes so long-running work for one agent does not occupy the same global foreground lane used by other agents.

## Best Possible Solution

Review and repair or replace https://github.com/openclaw/openclaw/pull/73991 with server-owned per-agent/default lane routing, bounded lane names, focused config/docs/tests, and a preserved global resource cap.

## Reproduction Assessment

Yes for source-level behavior: ordinary embedded and auto-reply runs still use the process-wide `main` lane unless an explicit lane is supplied, and config/docs expose only the global cap. The exact Feishu deployment latency was supported by user logs but not live-reproduced in this review.

## Solution Assessment

Unclear as proposed. Per-agent lanes are plausible, but the maintainable fix needs agreed config semantics, server-owned lane selection, bounded lane creation, and a total resource cap before implementation lands.
