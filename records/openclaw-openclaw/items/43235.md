## Label Justifications

## Summary

Keep open. Current main and v2026.6.6 still route ordinary foreground agent work through one process-wide `main` lane, and the linked closing PR is open, conflicting, and has unresolved design/security review concerns, so this needs maintainer scheduler/config direction rather than cleanup closure.

## What This Changes

Request support for per-agent or per-agent-group command lanes so heavy agents cannot starve other agents on the shared foreground main lane.

## Best Possible Solution

Adopt one scheduler-owned foreground composition: session serialization, optional agent-level isolation, and a preserved gateway-wide cap, with minimal config/schema/docs/tests covering fresh and upgrade behavior.

## Reproduction Assessment

Yes, from source with high confidence: foreground runs still take a session lane and then a global lane that defaults to `main`, and gateway startup only applies one process-wide `agents.defaults.maxConcurrent` cap. I did not run a live Feishu deployment, but the issue and comment include field logs matching the source path.

## Solution Assessment

Unclear. Per-agent isolation is likely the right capability, but the current linked PR is not yet the best safe path until maintainers choose the config/API shape and global-cap semantics.
