## Summary

Keep open: current main and the latest release still route ordinary foreground agent work through a per-session lane and then the shared `main` lane, while the linked implementation PR was closed unmerged after security, compatibility, proof, and mergeability blockers.

## What This Changes

The issue requests per-agent command lanes or per-agent concurrency so heavy agents in shared gateways do not starve unrelated agents on the global foreground lane.

## Best Possible Solution

Keep this as the canonical foreground per-agent lane tracker, then design a server-owned lane/concurrency model that preserves the global cap, updates schema/docs, and covers all foreground embedded-run entry points.

## Reproduction Assessment

Yes, at source level: current foreground runs still compose `session:<key>` plus the shared `main` lane, and the issue/comment logs match that bottleneck. I did not run a live multi-agent Feishu deployment in this read-only review.

## Solution Assessment

Unclear as proposed: per-agent isolation is the right capability, but maintainers still need to choose the server-owned config shape, global-cap semantics, and security boundary.
