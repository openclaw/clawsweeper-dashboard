## Summary

Keep open: current main and v2026.6.8 still route ordinary agent runs through a per-session lane and then the shared `main` global lane, with no per-agent lane or per-agent concurrency config. The prior implementation PR was closed unmerged after security/design blockers, so the remaining work needs maintainer scheduler/config direction rather than cleanup.

## What This Changes

The issue requests per-agent command lane isolation so one busy agent cannot consume the shared main lane for all agents in a multi-agent gateway.

## Best Possible Solution

Define one server-owned lane isolation model for foreground agent work, with validated config, preserved global caps, docs, upgrade behavior, and tests proving cross-agent parallelism plus same-agent/session serialization.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 route ordinary runs through `session:<key>` and then the shared `main` lane, and the Feishu logs match that bottleneck path; no live Feishu deployment was run in this read-only review.

## Solution Assessment

Unclear as proposed. Per-agent isolation is a plausible capability, but the exact public config shape and global-cap semantics need maintainer product direction before a safe implementation.
