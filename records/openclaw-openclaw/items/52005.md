## Summary

Keep open. Current main still has identity-based Discord bound-thread webhook suppression and shared message-id dedupe, while the issue discussion includes a live PluralKit repro shape and the paired implementation PR is closed unmerged.

## What This Changes

The issue reports that PluralKit-originated messages in bound Discord threads can be handled as duplicate inbound turns with possible duplicate context pressure and lingering typing indicators.

## Best Possible Solution

Add Discord-owned canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer explicit provider metadata when available; otherwise use a short-lived fallback scoped to Discord bound-thread agent-session traffic, with regression coverage that suppressed duplicates do not start or extend typing indicators.

## Reproduction Assessment

Yes. The issue gives live reproduction steps, and a deterministic test can feed a bound-thread original/proxy pair with different Discord message ids and missing or nonmatching bound-webhook metadata through preflight/dispatch to verify whether two turns or typing callbacks are produced.

## Solution Assessment

Yes. A narrow Discord provider/preflight fix is the best direction; broad shared content-based inbound dedupe would risk changing unrelated channels and should be avoided unless a generic contract need is proven.
