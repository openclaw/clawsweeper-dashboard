## Summary

Keep open. Current main still lacks a Discord-bound PluralKit original/proxy canonicalization path: the existing suppression is webhook-id based, shared inbound dedupe keys on Discord message id, and the linked implementation PR is closed unmerged.

## What This Changes

Ensure PluralKit-originated messages in Discord bound threads behave as one inbound turn without duplicate run pressure or lingering typing indicators.

## Best Possible Solution

Add Discord-owned preflight canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer an explicit Discord or PluralKit signal; if unavailable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and prevent suppressed duplicates from starting or extending typing indicators.

## Reproduction Assessment

Yes. The issue gives a live repro shape, and a focused regression can feed a bound-thread original/proxy pair with different Discord message ids and missing or nonmatching webhook metadata through Discord preflight/dispatch to verify that only one turn and no extra typing callback are produced.

## Solution Assessment

Yes. A narrow Discord provider/preflight fix is the best direction; broad shared content-based inbound dedupe would risk cross-channel behavior and should be avoided unless a generic contract need appears.
