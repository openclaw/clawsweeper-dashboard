## Label Justifications

- `P2`: The issue has source-level support and real multi-agent usability impact, but it requires product/API design rather than an urgent runtime hotfix.
- `impact:session-state`: The durable-rule half concerns session-level natural-language training diverging from durable workspace and memory state.
- `impact:message-loss`: The multi-mention half can leave explicitly mentioned bots silent or route a visible reply to only one active bot.

## Summary

Keep open: current main and the latest shipped tag still leave both requested product gaps open. Feishu has source-confirmed partial failures around explicit multi-bot mentions and one-visible-responder broadcast semantics, while memory promotion still writes general long-term memory rather than a governed durable rule layer with precedence or supersession.

## What This Changes

The issue requests an optional durable natural-language rule layer plus explicit group-chat semantics where every explicitly mentioned bot replies once and unmentioned bots stay silent.

## Best Possible Solution

Define a maintainer-approved durable rule-precedence layer and explicit all-mentioned-bots routing contract, then implement it through memory promotion, channel dispatch, focused tests, docs, and live multi-bot proof.

## Reproduction Assessment

Yes, at source level: current Feishu broadcast creates one visible dispatcher while observers cannot send visible replies, and memory promotion persists general MEMORY.md snippets without a governed rule layer. I did not run a live Feishu/Lark multi-bot group.

## Solution Assessment

No, not as an autonomous fix from the issue text. The maintainable path is a product/API decision first, then a scoped implementation that preserves existing broadcast and memory compatibility while adding explicit rule precedence and routing semantics.
