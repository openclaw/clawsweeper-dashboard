## Summary

Keep open for maintainer product review: current main has memory promotion, active-memory hooks, and Feishu broadcast/session handling, but not the requested governed natural-language rule layer with explicit supersession semantics or a channel contract that every explicitly mentioned bot replies once.

## What This Changes

Request a durable natural-language rule-promotion layer plus explicit multi-bot mention semantics for group chats.

## Best Possible Solution

Define the product/API boundary first, then split implementation into a governed memory/plugin rule-promotion design and a narrow channel contract for explicit multi-bot mentions with live Feishu/Lark proof.

## Reproduction Assessment

Source-level yes for the missing governed rule layer and for current Feishu broadcast having one visible active dispatcher. No live Feishu/Lark multi-bot reproduction was run in this read-only review.

## Solution Assessment

Unclear as a single implementation: the requested direction is plausible, but it spans memory authority, plugin APIs, workspace-file precedence, and channel routing. The safer solution is a maintainer-approved design split before code work.
