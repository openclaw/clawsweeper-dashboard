## Summary

Keep open: current main and v2026.5.22 still leave the central product request unresolved. Feishu broadcast still gives only one routed agent a visible reply dispatcher, memory promotion still writes through MEMORY.md/DREAMS.md rather than a governed durable rule-precedence layer, and related PRs cover narrower listen-only or self-mention pieces without settling the requested all-mentioned-bots contract.

## What This Changes

Request durable natural-language rule promotion with explicit conflict handling, plus group-chat semantics where every explicitly mentioned bot replies once and unmentioned bots stay silent.

## Best Possible Solution

Define a maintainer-approved durable rule-precedence and Feishu all-mentioned-bots routing contract, preferably through memory/plugin-owned seams where possible, then implement focused tests, docs, and live multi-bot proof.

## Reproduction Assessment

Yes at source level: current Feishu broadcast creates one visible dispatcher while observer agents cannot send visible replies, and memory promotion persists candidates into MEMORY.md without a governed durable rule layer. I did not run a live Feishu/Lark multi-bot group.

## Solution Assessment

No, not as an autonomous fix from the issue text. The maintainable path is a product/API decision first, then a scoped implementation that preserves compatibility while adding explicit rule precedence and routing semantics.
