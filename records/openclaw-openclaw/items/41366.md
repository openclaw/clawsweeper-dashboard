## Summary

Keep open: current main and the latest shipped tag still leave the broad product request unresolved. Source and tests show Feishu broadcast currently gives only one agent a visible reply dispatcher, and current memory promotion still writes general candidates into MEMORY.md rather than a governed durable rule layer with explicit precedence or supersession.

## What This Changes

Requests durable natural-language rule promotion and precedence plus group-chat semantics where every explicitly mentioned bot replies once and unmentioned bots stay silent.

## Best Possible Solution

Define a maintainer-approved durable rule-precedence layer and explicit all-mentioned-bots routing contract, then implement it through memory promotion, channel dispatch, focused tests, docs, and live multi-bot proof.

## Reproduction Assessment

Yes, at source level: current Feishu broadcast creates one visible dispatcher while observer agents cannot send visible replies, and memory promotion persists general MEMORY.md candidates without a governed rule layer. I did not run a live Feishu/Lark multi-bot group.

## Solution Assessment

No, not as an autonomous fix from the issue text. The maintainable path is a product/API decision first, then a scoped implementation that preserves compatibility while adding explicit rule precedence and routing semantics.
