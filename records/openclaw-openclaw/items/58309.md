## Summary

Keep open. Current main still has the raw JSON parse behavior this PR targets, but the proposed branch needs maintainer/security review before merge because it broadens into device identity key-file writes and introduces a world-readable fallback mode in that new helper.

## What This Changes

The branch wraps selected JSON.parse sites in archive, Tailscale, and outbound delivery queue helpers with contextual errors and adds a synchronous atomic write helper for device identity files.

## Best Possible Solution

Keep the useful JSON diagnostic work, but finish it as a narrow infra fix: make the device identity write helper private-by-default or reuse the existing secure JSON write helper, harden the remaining Tailscale funnel parse path consistently, add focused regression coverage, and include the required user-facing changelog entry if maintainers keep the behavior in this PR.

## Reproduction Assessment

Yes. Current main shows direct `JSON.parse` calls in the targeted helpers, so malformed archive JSON, corrupt queue files loaded through `failDelivery()` or `loadPendingDelivery()`, and malformed Tailscale JSON output are enough to exercise the reported diagnostic gap.

## Solution Assessment

No. The proposed direction is reasonable, but this exact patch is not the best final form until the key-material helper cannot default to world-readable mode and the remaining Tailscale funnel JSON parse path, tests, and changelog coverage are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use a private default for key-material writes:** `src/infra/device-identity.ts:47`
  - body: The new helper is documented for cryptographic key material but falls back to `0o644`. Current callers pass `0o600`, but any later call without options would create identity data with a world-readable mode, so the helper should default to `0o600` or require callers to supply a mode.
  - confidence: 0.83

## Security Review
