## Summary

Keep open. Current main still has the raw JSON parse paths this PR is trying to improve, but the proposed patch is not merge-ready because it also changes device identity key-material persistence and introduces a private-key write helper with an unsafe fallback mode.

## What This Changes

The PR wraps selected JSON.parse calls in archive, Tailscale, and outbound delivery queue helpers with contextual errors and adds a synchronous atomic write helper for device identity files.

## Best Possible Solution

Keep the useful JSON diagnostic work, but finish it as a narrow infra fix: make the identity write helper private-by-default or reuse the existing secure atomic write helper, harden the remaining Tailscale funnel parse path, add focused malformed-JSON regression coverage, and include a changelog entry if maintainers keep this as a user-facing bug fix.

## Reproduction Assessment

Yes. Current main shows direct JSON.parse calls in the targeted helpers, so malformed archive JSON, corrupt outbound queue files, and malformed Tailscale status/whois/funnel JSON output provide a high-confidence reproduction path for the diagnostic gap.

## Solution Assessment

No. The direction is reasonable, but this exact patch is not the best final form until the key-material helper cannot default to world-readable mode and the remaining Tailscale funnel parse path, tests, and changelog coverage are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Default key-material writes to private mode:** `src/infra/device-identity.ts:47`
  - body: atomicWriteFileSync() is documented for cryptographic key material but falls back to `0o644`. Current callers pass `0o600`, but any later caller that omits the option would create identity data with world-readable permissions, so the helper should default to `0o600` or require a mode.
  - confidence: 0.83
