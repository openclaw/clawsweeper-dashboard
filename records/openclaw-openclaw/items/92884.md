## What This Changes

The issue reports `openclaw config validate` rejecting a replacement Slack plugin's schema-backed `channels.slack.threadGuard` field even though `config schema` exposes it.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/92902; do not close this issue or queue a duplicate repair while that candidate fix is open.

## Reproduction Assessment

Yes, at source level: current main validates `channels.<id>` through collected channel schema metadata, and the current collector only protects closer-origin schemas, leaving same-origin replacement schemas order-dependent. I did not run a failing CLI reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow metadata collector repair with regression coverage so `config schema` and `config validate` agree on the active replacement channel schema. A separate validation mode or global Slack schema relaxation would create a second source of truth.
