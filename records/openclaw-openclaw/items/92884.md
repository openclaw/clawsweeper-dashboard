## Summary

Keep open: the report matches OpenClaw's documented plugin-owned channel schema contract, and current source still has validation paths that can reject a plugin-owned extension before the active channel manifest schema is allowed to own the channel shape.

## What This Changes

The issue asks `openclaw config validate` to accept plugin-owned `channelConfigs.<id>.schema` extensions for existing channel keys such as `channels.slack.threadGuard`.

## Best Possible Solution

Make `config validate` use the same plugin-owned channel schema source that `config schema` exposes for existing bundled channel ids, with a Slack-style regression for `threadGuard` and no global relaxation of bundled schemas.

## Reproduction Assessment

Yes, by source inspection there is a credible current-main path where bundled channel validation can run before plugin-aware schema ownership and emit the reported additional-properties error. I did not run a live packaged repro because the report omits a redacted minimal config/plugin fixture.

## Solution Assessment

Yes, the narrowest fix is to align `config validate` with manifest-owned channel schema resolution rather than adding a separate validation mode or relaxing Slack's bundled schema globally.
