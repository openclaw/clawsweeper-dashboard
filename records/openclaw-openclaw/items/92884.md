## Summary

Keep open: the report describes a real config-validation contract gap, and https://github.com/openclaw/openclaw/pull/92902 is still open as the matching fix candidate. Current main still routes validation through channel schema metadata collection, while the documented manifest contract says `channelConfigs.<channel-id>.schema` owns validation for `channels.<channel-id>`.

## What This Changes

The issue asks for `config validate` to honor plugin-provided `channelConfigs` schemas for replacement channel plugins instead of rejecting plugin-owned properties such as `channels.slack.threadGuard`.

## Best Possible Solution

Review and land or explicitly close https://github.com/openclaw/openclaw/pull/92902; the end state should use one active channel schema selection path for both `config schema` and `config validate`, while preserving strict bundled validation when no replacement owns the channel.

## Reproduction Assessment

Yes, source-level: the reporter supplied a redacted replacement Slack fixture and current main shows validation chooses schemas through the collector that lacks same-origin `preferOver` handling. I did not execute a live failing command in this read-only review.

## Solution Assessment

Yes: the narrow collector-level repair in the linked PR is the best path because both schema generation and validation consume collected channel metadata. A separate validation mode or Slack-only relaxation would create another source of truth.
