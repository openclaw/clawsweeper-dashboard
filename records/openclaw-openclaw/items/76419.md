## Summary

Keep open. Current main still turns Telegram `ReplyToBody` into an untrusted context block, then the embedded runner strips that block out of the submitted user prompt as `runtimeContext`; the live report says prompt-local runtime context is not reliably visible for the current reply turn.

## What This Changes

Telegram replies/quotes should preserve the structured replied-message body in the current model turn after the runtime-context prompt split.

## Best Possible Solution

Thread explicit untrusted reply/quote metadata through current-turn model input separately from internal runtime context, with regression coverage proving `OPENCLAW_INTERNAL_CONTEXT` stays hidden.

## Reproduction Assessment

Yes for source-level reproduction: a Telegram reply with `ReplyToBody` creates a replied-message block in the effective prompt, while the clean transcript prompt causes `resolveRuntimeContextPromptParts` to move that block into `runtimeContext`. I did not perform the live Telegram provider run in this read-only review.

## Solution Assessment

No, the current implementation is not the best way to solve this issue because it treats explicit reply context like internal runtime context. The safer fix is to classify inbound reply/quote context separately and include it in the current model input without re-exposing arbitrary internal runtime data.
