## Summary

Keep open. Current main still routes the rendered Telegram `ReplyToBody` block out of the submitted user prompt and only injects it as prompt-local runtime system context; the fresh live report says that path is not reliable, so this is not safe to close as implemented by earlier related reviews.

## What This Changes

Telegram reply/quote context should remain available to the assistant on the current turn after the runtime-context prompt split.

## Best Possible Solution

Thread explicit untrusted reply/quote metadata through the current-turn model input while keeping internal runtime context hidden from provider user turns and visible transcripts.

## Reproduction Assessment

Yes for source-level reproduction: a Telegram turn with `ReplyToBody` builds a prompt containing the replied-message block, then `resolveRuntimeContextPromptParts` strips that block into `runtimeContext` and submits only the clean transcript prompt. I did not perform the live provider reproduction in this read-only review.

## Solution Assessment

No, the current implementation is not enough to close this report: prompt-local system-context injection may protect transcripts, but it does not preserve reply/quote context adjacent to the current user turn as expected. The safer repair is to separate explicit inbound untrusted context from internal runtime context instead of treating both as the same prompt remainder.

## Repair Work Prompt

Fix the Telegram/current-turn reply context regression reported in https://github.com/openclaw/openclaw/issues/76419. Source inspection shows Telegram populates `ReplyToBody`, `buildInboundUserContextPrefix` renders a `Replied message (untrusted, for context)` block, but `transcriptPrompt` is only the clean current user text, so `resolveRuntimeContextPromptParts` moves the reply block into `runtimeContext` and `run/attempt.ts` submits only the clean prompt. Implement a narrow fix that preserves explicit inbound reply/quote context for the current model turn while keeping arbitrary internal OpenClaw runtime context hidden from provider user turns and visible transcripts. Prefer a typed/separate prompt part or explicit untrusted inbound-context marker over broad string matching when feasible. Add regression coverage proving a reply body is available to the current model input and that internal runtime context remains hidden/queued. Add a single-line `CHANGELOG.md` fix entry; do not add a config option, do not change Telegram auth/routing, and do not reintroduce hidden internal runtime context into user prompts.
