## Summary

Keep open. Current main still builds Telegram reply metadata upstream, but the embedded runner strips the rendered replied-message block out of the submitted user prompt and sends only the clean transcript prompt, while the report provides fresh live evidence that prompt-local runtime system context is not reliable enough for this case.

## What This Changes

Restore Telegram reply/quote context so the model receives the quoted message in the same current turn after the runtime-context prompt split.

## Best Possible Solution

Thread explicit untrusted reply/quote context through the current-turn model input while keeping internal runtime context hidden from provider user turns and visible transcripts.

## Reproduction Assessment

Yes for source-level reproduction: a Telegram turn with `ReplyToBody` builds a replied-message block in `queuedBody`, then the runtime-context splitter removes that block because `transcriptPrompt` is only the clean current message. I did not run a live provider test in this read-only review.

## Solution Assessment

No, the current implementation is not the best fix for this issue because it treats explicit inbound reply context the same as hidden internal runtime context. The safer repair is to distinguish user-context reply/quote metadata from internal runtime context instead of moving all prompt remainder back into the user turn.

## Repair Work Prompt

Fix the Telegram reply/quote context regression tracked at https://github.com/openclaw/openclaw/issues/76419. Current main builds `ReplyToBody` correctly and renders it through `buildInboundUserContextPrefix`, but `resolveRuntimeContextPromptParts` treats the rendered replied-message block as hidden runtime context because `transcriptPrompt` is only the clean current message. Preserve explicit inbound reply/quote user-context for the current model turn while keeping arbitrary internal runtime context hidden from provider user turns and visible transcripts. Prefer a structured boundary over fragile string matching if feasible; do not add a config flag, do not move all runtime context back into user prompts, and do not change unrelated channel behavior. Add focused regression coverage for the prompt split and the Telegram/auto-reply path, and add a user-facing `CHANGELOG.md` fix entry.
