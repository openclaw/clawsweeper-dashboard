## Summary

Close: current main already serializes OAuth refresh per provider/profile, and the fix is also present in shipped releases, including the latest stable release.

## What This Changes

The issue requests single-flight OAuth refresh so concurrent runtime turns do not spend the same rotating refresh token in parallel.

## Best Possible Solution

Keep the centralized OAuthManager queue plus global refresh lock as the canonical runtime fix; open a narrower follow-up only if a separate non-runtime direct-refresh path still has concrete concurrency evidence.

## Reproduction Assessment

No current-main repro: the relevant runtime path now serializes same-profile OAuth refresh and the regression test asserts a four-agent expired-token burst performs exactly one refresh call. I did not run tests because this review is read-only, but the source and shipped test coverage are direct.

## Solution Assessment

Yes: the current implementation is the right shape because it combines an in-process per-provider/profile queue with a cross-agent file lock and lets later callers adopt the fresh persisted credential.
