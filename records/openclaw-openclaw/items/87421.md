## Label Justifications

- `P1`: The report describes an always-on runtime auth failure that forces manual OAuth reauthentication under normal concurrent load.
- `impact:auth-provider`: The affected path is OAuth refresh, provider credential routing, and refresh-token reuse behavior.

## Summary

Current main already solves the reported OAuth refresh storm for the provider-auth runtime path: refreshes are queued per provider/profile and guarded by a shared refresh lock, xAI uses that provider hook, and the latest release contains the same implementation.

## What This Changes

The issue reports concurrent OAuth refresh attempts for the same provider/profile revoking rotated refresh-token families and asks for per-profile single-flight refresh behavior.

## Best Possible Solution

Keep the current generic auth-manager serialization and close this duplicate; new evidence should arrive as a fresh issue with v2026.6.6 logs showing parallel refresh calls.

## Reproduction Assessment

No current-main failing repro remains. The old race is covered by source-level regression tests that model concurrent callers sharing an expired OAuth profile and assert the fixed single-flight behavior.

## Solution Assessment

Yes. The implemented fix is at the generic auth-profile owner boundary for provider runtime refresh, while the Codex app-server direct-refresh path has its own scoped queue and upstream Codex external-auth contract support.
