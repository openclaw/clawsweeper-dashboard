## Label Justifications

- `P1`: The reported race can invalidate rotating OAuth refresh tokens and break provider access for always-on runtime users.
- `impact:auth-provider`: The issue is about OAuth refresh token rotation, provider auth profiles, and re-authentication failures.

## Summary

Current main and the latest stable release already serialize OAuth refreshes for the reported provider-auth path: generic auth-profile refreshes are queued per provider/profile and guarded by a cross-agent lock, xAI registers its refresh hook through that generic provider path, and Codex app-server scoped refreshes have their own per-store queue.

## What This Changes

The issue asks for OAuth credential refreshes to be serialized per provider/profile so concurrent runtime turns do not reuse a rotating refresh token.

## Best Possible Solution

Keep the current generic auth-manager serialization and provider-owned refresh hooks; the Miles keepalive sidecar remains a reasonable local workaround for older deployments but is no longer needed as a core fix path on current OpenClaw.

## Reproduction Assessment

Yes, as a source-level regression path: the current tests model concurrent callers sharing an expired OAuth profile and assert the fixed behavior. I did not force a live xAI refresh because this read-only review should not spend real credentials.

## Solution Assessment

Yes. The implemented fix sits at the generic auth-profile runtime owner, keeps xAI refresh provider-owned through `refreshOAuth`, and separately queues the Codex app-server scoped direct-refresh path.
