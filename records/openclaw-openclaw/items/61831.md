## Summary

Keep open. Current main has an optional sandbox-browser path that can isolate browser containers by agent or session when sandboxing is enabled, but ordinary multi-agent browser use still defaults to the shared host `openclaw` browser profile, so the requested default per-agent Chrome/profile isolation is not implemented.

## What This Changes

The issue requests fully isolated browser instances per agent so tabs, cookies, localStorage, and concurrent MCP/browser commands cannot leak or race across agents.

## Best Possible Solution

Define the multi-agent browser contract explicitly, then implement a default-safe isolation path, likely agent-scoped managed profiles or an automatic sandbox-browser mode for agents that get the browser tool, while preserving explicit `profile="user"` access as an opt-in shared-session escape hatch.

## Reproduction Assessment

Yes. With the default sandbox mode off and two agents allowed to call `browser` without `target` or `profile`, both calls follow the host target and shared `openclaw` profile path shown in current source.

## Solution Assessment

Unclear. The existing sandbox browser is a useful building block, but it is opt-in and does not by itself satisfy the reported default multi-agent behavior; the best fix needs maintainer choice on default isolation, migration, and documentation.
