## Summary

Keep open. Current main still gives ordinary non-sandboxed browser tool calls the shared host browser control path and the shared default `openclaw` profile; the newer sandbox-browser path can isolate by agent/session, but it remains opt-in rather than the default requested here.

## What This Changes

The issue requests default per-agent Chrome process/profile isolation for the browser tool so simultaneous agents cannot share tabs, cookies, localStorage, or MCP tab state.

## Best Possible Solution

Define the intended multi-agent browser isolation contract, then implement default-safe per-agent host profiles or automatic sandbox-browser isolation while preserving an explicit opt-in path for shared `profile="user"` or shared `openclaw` behavior.

## Reproduction Assessment

Yes. With sandboxing off and two agents allowed to call `browser` without `target` or `profile`, current source sends both calls to the host browser control service with `profile` omitted, which resolves to the shared `openclaw` profile.

## Solution Assessment

Unclear. The existing sandbox-browser path is a useful building block, but it is opt-in; the maintainable fix needs a maintainer decision on defaults, migration, lifecycle cleanup, and documentation.
