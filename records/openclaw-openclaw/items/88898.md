The branch adds a `dispatchReplyFromConfig` guard plus a regression test so `messages.suppressToolErrors` also suppresses `isError` tool-result progress payloads.

## Best Possible Solution

Land the narrow dispatch guard after maintainer review, while keeping broader `all | final | off` warning policy and dedicated warning-routing decisions in their existing open product issues.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.8 forward `isError` tool-result progress before any `messages.suppressToolErrors` gate, and the PR test/proof exercises the configured suppression path while preserving final delivery.

## Solution Assessment

Yes. The guard belongs at the dispatch-owned visible progress boundary before both source callback and dispatcher delivery; lower warning-payload generation already honors the same config, and broader delivery-mode/routing policy is separate follow-up work.
