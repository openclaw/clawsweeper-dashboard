The PR adds a dispatch-time guard so messages.suppressToolErrors drops error tool-result progress callbacks before source callback or channel delivery, plus a focused regression test.

## Best Possible Solution

Land this patch or an equivalent dispatch-site guard after normal maintainer merge handling, while leaving broader warning-routing product requests to their own issues.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: dispatchReplyFromConfig can receive an isError tool-result payload with messages.suppressToolErrors true and still forward/deliver it because the callback path lacks the config gate.

## Solution Assessment

Yes. The guard belongs at the core progress-dispatch point because payload generation already handles terminal warning payloads and channel adapters should not infer product policy from raw error text.
