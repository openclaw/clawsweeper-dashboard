## Summary

Keep this PR open: current main still leaks `isError` tool-result progress under the existing `messages.suppressToolErrors` setting, while the PR adds a narrow guard in the central dispatch path with focused coverage and no concrete code-review findings.

## What This Changes

The PR adds a config-gated guard in `dispatchReplyFromConfig` to drop error tool-result progress payloads when `messages.suppressToolErrors` is true and adds a regression test for that path.

## Best Possible Solution

Land the focused dispatch guard once maintainers are satisfied with the unrelated CI state, preserving default behavior while making the existing opt-in suppression cover visible tool-error progress.

## Reproduction Assessment

Yes. Current main can forward an `isError` tool-result through verbose progress delivery before any `messages.suppressToolErrors` guard in that path; I did not run local tests because this review is read-only.

## Solution Assessment

Yes. Guarding the central dispatch progress path before both callback forwarding and channel dispatch is the narrowest maintainable fix; changing only terminal warning payloads or channel adapters would leave a second policy path or duplicate suppression logic.
