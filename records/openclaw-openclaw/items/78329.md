## Summary

Keep this PR open: the controller-level ownership tracking is pointed at the right surface, but the current branch still converts an explicit open `plugins.allow: []` into a restrictive allowlist when enabling a plugin.

## What This Changes

The branch adds Control UI config-controller tracking for automatically added plugin allowlist entries, controller regression coverage, and an Unreleased changelog entry.

## Best Possible Solution

Keep the controller ownership bookkeeping, but only auto-add and auto-remove plugin ids for existing non-empty restrictive allowlists, and add a regression test that `plugins.allow: []` stays open.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the PR branch appends into `plugins.allow: []`, while current docs and runtime treat an empty allowlist as open; this read-only review did not execute the PR branch.

## Solution Assessment

Is this the best way to solve the issue? No: the ownership-tracking approach is maintainable, but it must preserve explicit empty allowlists by guarding auto-add behavior to non-empty restrictive lists.
