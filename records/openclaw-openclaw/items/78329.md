## Summary

Keep open: the PR now has real browser proof and the changelog entry is present, but the patch still converts explicit empty `plugins.allow` configs into restrictive allowlists, which contradicts current docs and runtime behavior.

## What This Changes

The branch adds Control UI config-controller ownership tracking for auto-added plugin allowlist entries, controller regression coverage, and an Unreleased changelog entry.

## Best Possible Solution

Keep the controller-level ownership bookkeeping, but limit auto-add and auto-remove behavior to already restrictive non-empty allowlists and add a regression test that `plugins.allow: []` stays open.

Do we have a high-confidence way to reproduce the issue? Yes at source level: the PR appends to any array `plugins.allow`, while current docs and runtime only treat non-empty allowlists as restrictive; this read-only review did not execute the PR branch.

## Solution Assessment

Is this the best way to solve the issue? No: the ownership tracking is a good fit for the Control UI controller, but it must preserve explicit empty allowlists by only auto-appending for non-empty restrictive allowlists.
