## Summary

Keep open: the PR addresses a valid current-main auth compatibility bug, but the patch has a blocking secret-scrubbing gap in the legacy alias normalization path.

## What This Changes

The PR normalizes legacy OpenAI Codex OAuth auth-profile aliases, adds a mode-mismatch missing-auth hint, regression tests, and a changelog entry.

## Best Possible Solution

Merge a revised version that keeps the alias normalization and diagnostic but also removes or scrubs consumed legacy token aliases, with regression coverage for alias-only profile persistence.

## Reproduction Assessment

Yes. Source inspection of current main shows alias-only OpenAI Codex OAuth fields are preserved but not converted before PI conversion, and stale api_key profile metadata rejects a stored OAuth profile before the generic missing-auth error.

## Solution Assessment

No, not yet. The normalization and diagnostic are the right narrow direction, but the implementation should also drop consumed legacy alias fields or extend persistence scrubbing so OpenAI Codex OAuth token redaction still holds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Drop legacy OAuth alias fields after normalizing:** `src/agents/auth-profiles/persisted.ts:588`
  - body: After `normalizeLegacyOpenAiCodexOAuthAliases` copies `access_token`/`refreshToken` into canonical fields, the parsed credential still carries the original alias keys. On normal loads with `rewriteInlineOAuthSecrets`, that makes the OpenAI Codex secret rewrite create an `oauthRef` but keep the legacy token aliases in `auth-profiles.json`, because the sanitizer only deletes `access`, `refresh`, and `idToken`. Please remove the consumed aliases or extend the sanitizer and add coverage for alias-profile rewrite.
  - confidence: 0.83
