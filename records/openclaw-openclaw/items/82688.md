## Summary

Keep open. Current main still has the reported launcher comparison path: the packaged launcher compares Node's active compile-cache directory to OpenClaw's unversioned scoped root, so an already-enabled cache under the intended root can still take the respawn branch.

## What This Changes

The issue requests fixing packaged launcher compile-cache detection so OpenClaw does not respawn every run when the active Node cache directory is already under the desired OpenClaw cache root.

## Best Possible Solution

Update the packaged launcher to treat Node's versioned active cache directory under the resolved OpenClaw cache root as already correct, and add launcher e2e coverage that observes the packaged respawn marker rather than only cache enablement.

## Reproduction Assessment

Yes, via source inspection and a focused Node probe rather than a full e2e run. Node reports the active compile-cache directory as a versioned child, while current main compares it to the unversioned OpenClaw-scoped desired root and therefore takes the respawn branch.

## Solution Assessment

Yes. The requested containment or logical-root comparison is the narrow maintainable fix, provided it preserves the first respawn from unrelated inherited cache roots and adds a regression test for packaged respawn idempotence.
