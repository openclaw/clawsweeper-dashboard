## Summary

Keep open: current main and the latest release still use exact path equality for the packaged compile-cache respawn guard, and a scratch packaged fixture reproduced the repeated respawn. The issue is also paired with an open fixing PR that uses closing syntax, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that the packaged launcher can respawn on every run because it compares Node's versioned active compile-cache directory with OpenClaw's unversioned desired cache root.

## Best Possible Solution

Review and land the open fixing PR, or replace it with an equivalent narrow fix that recognizes Node's versioned child under the desired packaged cache root and adds a two-launch packaged regression test.

## Reproduction Assessment

Yes. A scratch packaged fixture copied from current main launched twice with the same NODE_COMPILE_CACHE root and printed the packaged respawn marker on both runs.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the packaged launcher guard and launcher E2E coverage; closing this issue before the linked PR merges would be premature.
