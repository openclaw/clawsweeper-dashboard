## Summary

Keep open: current main reproduces the reported launcher behavior. The packaged bootstrap compares Node's active versioned compile-cache directory with OpenClaw's unversioned desired root, and a current-main packaged fixture respawned on both repeated launches.

## What This Changes

Avoid unnecessary repeated packaged launcher respawns by recognizing Node's active versioned compile-cache child as belonging to the intended OpenClaw cache root.

## Best Possible Solution

Teach the packaged launcher to compare logical compile-cache roots while preserving OpenClaw's package-scoped cache policy, then cover the two-launch packaged fixture path.

## Reproduction Assessment

Yes. A current-main packaged fixture launched twice with `NODE_COMPILE_CACHE` set reported the packaged compile-cache respawn marker on both runs, matching the source-level versioned-child versus root comparison.

## Solution Assessment

## Repair Work Prompt

Fix the packaged launcher compile-cache respawn guard reported at https://github.com/openclaw/openclaw/issues/82688. In `openclaw.mjs`, account for Node returning a versioned child from `module.getCompileCacheDir()` and avoid respawning when the active cache is already under the intended OpenClaw package-scoped compile-cache root. Preserve source-checkout cache disabling and the package version/install-marker scoping that prevents stale compiled chunks after updates. Add focused regression coverage in `test/openclaw-launcher.e2e.test.ts` using a packaged fixture that prints `OPENCLAW_PACKAGED_COMPILE_CACHE_RESPAWNED`, launch it twice with the same compile-cache configuration, and prove the already-correct path does not respawn repeatedly. Touch `src/entry.compile-cache.ts` / `src/entry.compile-cache.test.ts` only if shared helper behavior changes. Add a user-facing changelog fix entry.
