## What This Changes

The issue reports that internal `workspaceDir` metadata can leak into Gateway `method: "agent"` params during subagent spawn, causing Gateway schema validation to reject the request.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/79821 by landing or closing it; keep `workspaceDir` as internal spawn/session metadata rather than expanding the public Gateway agent params schema.

## Reproduction Assessment

Yes at source level: injecting `workspaceDir` into public Gateway agent params hits a schema with `additionalProperties: false`, and the issue includes the observed rejection. Current `main` strips the field from the subagent Gateway call, so I did not establish a failing current-main runtime path.

## Solution Assessment

Yes. The narrow maintainable solution is to keep workspace metadata internal and add the linked spawn assembly regression guard, not to make `workspaceDir` a public Gateway `agent` parameter.
