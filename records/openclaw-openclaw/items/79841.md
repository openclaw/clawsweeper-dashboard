## Summary

Keep open: the report is concrete and a live open PR explicitly closes it with a regression guard, so this issue should remain open until that PR is resolved. Current main already treats `workspaceDir` as internal spawn metadata rather than a public Gateway `agent` param.

## What This Changes

The issue reports a subagent spawn boundary bug where internal `workspaceDir` metadata can leak into Gateway `method="agent"` params and be rejected by the strict agent schema.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/79821; if it lands and no unique runtime gap remains, close this issue with that PR as proof.

## Reproduction Assessment

Yes at source level, but not as a current-main end-to-end failure: injecting `workspaceDir` into Gateway `agent` params hits a strict schema that does not allow that field, while current subagent source strips it before the Gateway call. I did not run the reporter's live Telegram/macOS canary.

## Solution Assessment

Yes: keeping `workspaceDir` internal via spawn/session metadata is the narrow owner-boundary fix, rather than expanding the public Gateway schema. The linked PR adds the missing regression guard and should be handled before this issue is closed.
