## Summary

Keep open: this issue is still paired with an open closing PR that adds the exact regression guard, and source review confirms the reported boundary: Gateway rejects `workspaceDir` as a public `agent` param while current spawn code strips it internally.

## What This Changes

The issue reports that internal workspace metadata can leak into Gateway `method: "agent"` params during native subagent spawn, causing schema rejection.

## Best Possible Solution

Review and either land or explicitly close the linked regression-guard PR; if it lands, close this issue as implemented while keeping `workspaceDir` out of the public Gateway schema.

## Reproduction Assessment

Yes, at source level: adding `workspaceDir` to public Gateway `agent` params violates `AgentParamsSchema` and the handler returns `invalid agent params`. I did not run a live canary, and current main's subagent spawn code strips the field before the Gateway call.

## Solution Assessment

Yes. Keeping workspace metadata internal and guarding the spawn request boundary is narrower and safer than expanding the public Gateway `agent` schema for an internal bookkeeping field.
