## Summary

Keep open: the reported Gateway boundary is valid and current source preserves it, but this issue is still paired with an open closing PR that adds the explicit regression guard, so the issue should resolve through that PR rather than be auto-closed early.

## What This Changes

The issue asks for subagent spawn workspace metadata to remain internal instead of leaking `workspaceDir` into Gateway `method: "agent"` params.

## Best Possible Solution

Keep `workspaceDir` as internal `spawnedWorkspaceDir` and registered-run metadata, then land or close the linked regression-guard PR without expanding public Gateway agent params.

## Reproduction Assessment

Yes, at source level: `AgentParamsSchema` rejects additional properties and does not define `workspaceDir`, so forwarding it through `method: "agent"` reproduces the reported validation failure. I did not run the live macOS/Telegram canary in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve workspace metadata internally and strip it before the public Gateway request; the open linked PR is the right path for the missing regression guard.
