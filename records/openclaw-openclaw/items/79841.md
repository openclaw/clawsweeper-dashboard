## Summary

Keep open: an open linked PR uses closing syntax for this issue, so the issue should stay open until that PR is reviewed, merged, superseded, or closed. Source inspection confirms the intended boundary is real: Gateway `agent` params reject `workspaceDir`, while current subagent spawn code keeps workspace metadata internal before calling the Gateway.

## What This Changes

Prevent subagent workspace metadata from leaking into Gateway `method: "agent"` params while preserving it internally for spawned session bookkeeping.

## Best Possible Solution

Keep `workspaceDir` as internal spawn/session metadata, land or explicitly supersede the linked regression PR, then close this issue once the canonical guard is resolved on main.

## Reproduction Assessment

Yes, at source level: a Gateway `agent` payload containing `workspaceDir` is rejected by the strict schema, matching the reporter’s observed error. I did not establish a live current-main leak path because current main now strips the field before the Gateway call.

## Solution Assessment

Yes. Keeping `workspaceDir` internal via `spawnedWorkspaceDir` and excluding it from public `AgentParamsSchema` is the narrow maintainable fix; expanding the public schema would be the riskier direction.
