## Summary

Keep open: this is a valid bug report, but it is already paired with the open closing PR https://github.com/openclaw/openclaw/pull/79821, so cleanup should wait for that PR to merge or for a maintainer to close/supersede it. Current main already keeps `workspaceDir` out of Gateway `agent` params, and the linked PR adds the narrower regression guard.

## What This Changes

The issue asks OpenClaw to prevent internal `workspaceDir` metadata from being forwarded in Gateway `method: "agent"` params during subagent spawn while preserving it for session bookkeeping.

## Best Possible Solution

Keep `workspaceDir` as internal spawn/session metadata, and resolve this issue by reviewing and landing or closing https://github.com/openclaw/openclaw/pull/79821.

## Reproduction Assessment

Yes, source inspection gives a high-confidence failure path: the Gateway `AgentParamsSchema` rejects unexpected public fields, so leaked `workspaceDir` is invalid. I did not establish a failing current-main subagent path because current main now strips that field before the Gateway call.

## Solution Assessment

Yes. Keeping `workspaceDir` internal and stripping it before public Gateway `agent` params is the narrowest maintainable fix; expanding the public schema would expose runtime bookkeeping unnecessarily.
