## Summary

Keep open because the report describes a real Gateway/subagent boundary and an open closing PR is already attached; the issue should close after that PR lands or maintainers explicitly decide current main’s existing internal-metadata path is enough.

## What This Changes

The issue reports that subagent spawn request construction can leak internal `workspaceDir` into Gateway `method: "agent"` params, causing schema validation failure.

## Best Possible Solution

Keep `workspaceDir` as internal spawned-session metadata, land or close the linked regression-guard PR after review, and avoid expanding the public Gateway `agent` params schema.

## Reproduction Assessment

Yes for the schema rejection path: source shows public Gateway `agent` params reject an unexpected `workspaceDir`. I did not establish that current main’s subagent spawn path still leaks it, because current source strips the field before the Gateway call.

## Solution Assessment

Yes. Keeping workspace/cwd metadata in internal spawned-session bookkeeping and adding a focused regression guard is narrower and safer than making `workspaceDir` public Gateway API.
