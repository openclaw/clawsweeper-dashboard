## Summary

Keep open: current main shipped the broader durable delivery queue and WhatsApp reconnect drain, but it does not implement this PR's configurable WhatsApp send-retry surface, and the proposed retry classifier still misses the plain reconnect-window errors cited in the discussion.

## What This Changes

The PR adds a WhatsApp retry config/schema/docs surface and wraps text, media, and poll outbound sends with exponential-backoff retry helpers.

## Best Possible Solution

If maintainers still want configurable WhatsApp send retry, rebase it onto current main as a WhatsApp-plugin-owned complement to the durable queue, classify only true pre-send/readiness failures, and document how retry differs from queued recovery.

## Reproduction Assessment

Yes for the review finding. The proposed helper can be source-reproduced by throwing a plain `Error("No active WhatsApp Web listener")` or `Error("Connection Closed")`; without an allowlisted `code` or `name`, it will not retry.

## Solution Assessment

No, not as proposed. Configurable retry may still be useful, but the PR needs to rebase against the shipped durable queue and cover the actual reconnect-window readiness errors without broad duplicate-delivery risk.
