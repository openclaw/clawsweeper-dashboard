## Summary

Keep open: current main still lacks the requested configurable WhatsApp send retry surface, so this PR is not obsolete, but the branch is not merge-ready because it lacks real behavior proof and misses the listener-unavailable reconnect-window error it is meant to cover.

## What This Changes

The PR adds WhatsApp channel/account retry config and wraps text, media, and poll sends with a shared outbound retry helper, plus generated config docs and retry tests.

## Best Possible Solution

If maintainers want immediate retry as a durable-queue complement, rebase it onto current main, keep it plugin-owned, explicitly retry safe pre-send/readiness failures, update current schema/docs artifacts, and prove a real WhatsApp resend.

## Reproduction Assessment

Yes for the review finding via source inspection: the send path can throw a plain listener-unavailable Error before platform send, while the PR classifier only retries code/name allowlists. I did not run a live WhatsApp repro in this read-only review.

## Solution Assessment

No, not as proposed. Configurable retry may still be useful, but it should be rebased onto the current durable queue design and cover safe listener-readiness failures before enabling a default retry surface.
