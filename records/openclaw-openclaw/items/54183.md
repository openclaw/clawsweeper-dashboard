## Summary

Keep this PR open. Current main shipped the durable outbound queue/reconnect drain for persistent WhatsApp loss, but it does not implement the adjacent configurable immediate send-retry surface, and the PR still has a correctness blocker plus a stale rebase/product decision against that shipped queue behavior.

## What This Changes

The PR adds a WhatsApp `retry` config surface and wraps text, media, and poll outbound sends with exponential-backoff retry helpers, plus schema, generated config docs, and tests.

## Best Possible Solution

If maintainers still want this surface, rebase it onto current main as a WhatsApp-plugin-owned complement to the durable queue, classify only real pre-send or listener-readiness failures, and document retry versus queued recovery behavior.

## Reproduction Assessment

Yes. The review finding is reproducible from the proposed helper by throwing a plain `Error("No active WhatsApp Web listener")` or `Error("Connection Closed")`; without an allowlisted `code` or `name`, the default retry path will not retry.

## Solution Assessment

No, not as-is. Configurable retry may still be useful, but the PR needs to rebase against the shipped durable queue and cover the actual reconnect-window failures it claims to fix without broad duplicate-delivery risk.
