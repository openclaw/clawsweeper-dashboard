## Summary

Keep open: current main still lacks the requested configurable WhatsApp retry surface, but this PR is not ready because its retry classifier misses the message-only reconnect/listener failures it is meant to cover and maintainers have already called the retry surface adjacent to the shipped durable queue work.

## What This Changes

The branch adds a WhatsApp retry config/schema/docs surface and wraps WhatsApp text, media, and poll outbound sends with shared exponential-backoff retry helpers and tests.

## Best Possible Solution

Rebase on current main, decide whether configurable immediate retry should complement durable delivery, then narrowly classify pre-send/listener-readiness failures and document how it differs from queued recovery.

## Reproduction Assessment

Yes for the review finding: source inspection shows plain `Error("No active WhatsApp Web listener")` or `Error("Connection Closed")` has no allowlisted `code` or `name`, so the PR helper returns false. I did not run tests because this review is read-only.

## Solution Assessment

No, not as proposed. Configurable retry may still be useful, but the PR needs a product boundary with the durable queue and a classifier that covers true pre-send/readiness failures without broad duplicate-delivery risk.
