## Summary

Keep open. Current main has adjacent durable WhatsApp delivery recovery, but it still lacks this configurable channel/account retry surface; the linked feature issue was also closed with this PR as the canonical remaining item. The PR is not merge-ready because the retry classifier misses the plain listener-unavailable error path it is meant to cover, and contributor real-behavior proof is still missing.

## What This Changes

The PR adds channel/account WhatsApp retry config, a shared retry helper wrapping text/media/poll sends, generated config docs, and retry tests.

## Best Possible Solution

If maintainers want immediate retry as a durable-queue complement, rebase this onto current main, keep the config/plugin ownership local to WhatsApp, retry safe pre-send listener-readiness failures, refresh generated schema/docs, and add real WhatsApp after-fix proof.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR head still throws a plain listener-unavailable Error before platform send while the retry classifier only accepts code/name allowlists. I did not run a live WhatsApp repro in this read-only review.

## Solution Assessment

No, not as proposed. Configurable retry may still be useful, but it should be rebased onto the current durable queue design and explicitly cover safe listener-readiness failures before enabling a default retry surface.
