## Summary

Keep this PR open, but do not merge it as-is. Current main still does not expose configurable WhatsApp send retry, and the linked production reports show a real reliability problem, but current main now has durable outbound queue and reconnect drain behavior that changes the product tradeoff; the PR’s retry classifier also misses the plain reconnect-window errors the PR is intended to cover.

## What This Changes

The PR adds a WhatsApp `retry` config surface and wraps text, media, and poll outbound sends with exponential-backoff retry helpers, plus schema, generated config docs, and tests.

## Best Possible Solution

If maintainers still want configurable immediate retries, rebase onto current main and make the retry layer a WhatsApp-plugin-owned complement to the durable queue: cover text/media/poll and deps-backed paths, classify real pre-send or no-listener reconnect failures, avoid duplicate-delivery-prone ambiguous sends, and document immediate retry versus queued recovery.

## Reproduction Assessment

Yes. The review finding can be reproduced with the PR helper by throwing a plain `Error("No active WhatsApp Web listener")` or `Error("Connection Closed")`: because those errors have no allowlisted `code` or `name`, the PR’s default retry path will not retry.

## Solution Assessment

No, not as-is. A configurable retry surface may still be useful, but the current PR needs maintainer agreement against the shipped queue/reconnect behavior and must classify the actual reconnect-window failures it claims to fix.
