## Summary

Keep this PR open. Current main has shipped the broader durable outbound queue and WhatsApp reconnect drain, but it still does not implement the PR's configurable immediate WhatsApp send retry surface, and the proposed retry classifier misses the reconnect-window errors discussed in the thread.

## What This Changes

This PR adds a WhatsApp `retry` config surface and wraps text, media, and poll outbound sends with exponential-backoff retry helpers, plus schema, generated docs, and tests.

## Best Possible Solution

If maintainers still want configurable immediate retries, rebase onto current main and make the retry layer a WhatsApp-plugin-owned complement to the durable queue: cover text/media/poll, classify real pre-send or no-listener reconnect failures, avoid ambiguous duplicate-delivery cases, and document retry versus queued recovery.

## Reproduction Assessment

Yes. The review finding is reproducible against the PR helper by throwing a plain `Error("No active WhatsApp Web listener")` or `Error("Connection Closed")`; because those errors have no allowlisted `code` or `name`, the default retry path will not retry.

## Solution Assessment

No, not as-is. Configurable retry may still be useful, but this PR must rebase against the shipped durable queue behavior and classify the actual reconnect-window failures it claims to fix.
