## Summary

Keep this PR open, but do not merge it as-is: the configurable WhatsApp retry surface is still absent from current main, while current main already shipped durable queue/reconnect recovery and this branch misses the plain reconnect-window errors it claims to cover.

## What This Changes

The PR adds a WhatsApp retry config surface, retry helper, text/media/poll send wrappers, schema and generated-doc updates, and retry-focused tests.

## Best Possible Solution

If maintainers still want configurable immediate retries, rebase onto current main and make the retry layer a WhatsApp-plugin-owned complement to the durable queue, limited to actual pre-send/no-listener reconnect failures and documented against queued recovery.

## Reproduction Assessment

Yes. The review finding is reproducible by running the PR helper with plain Error("No active WhatsApp Web listener") or Error("Connection Closed"); those errors have no allowlisted code or name, so the PR's default retry path skips them.

## Solution Assessment

No, not as-is. Configurable retry may still be useful, but the branch must rebase against shipped durable queue behavior and classify the actual reconnect-window failures before it is a maintainable fix.
