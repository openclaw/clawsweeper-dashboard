## Summary

Keep open: current main still lacks the requested WhatsApp retry config, but this PR is not merge-ready because the retry classifier misses message-only listener/readiness failures, lacks after-fix real behavior proof, and needs maintainer review against the durable queue now on main.

## What This Changes

The PR adds configurable WhatsApp outbound send retry, schema/docs/generated metadata, retry helpers, and tests wrapping text, media, and poll sends.

## Best Possible Solution

If maintainers want this surface, rebase it onto current main as a WhatsApp-plugin-owned complement to the durable queue, retry only true pre-send/readiness failures, and document how it differs from queued recovery.

## Reproduction Assessment

Yes for the review finding. Source inspection shows a plain `Error("No active WhatsApp Web listener")` or similar message-only readiness error lacks an allowlisted code/name and will not retry.

## Solution Assessment

No, not as proposed. Configurable retry may still be useful, but it should be rebased onto the durable queue design and classify only safe pre-send/readiness failures before adding a default config surface.
