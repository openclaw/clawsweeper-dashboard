## Summary

Keep open: current main shipped the durable WhatsApp outbound queue/reconnect drain for the persistent-loss class, but configurable WhatsApp send retry is not implemented and this PR still has merge-blocking review findings plus product/architecture questions.

## What This Changes

The PR adds a WhatsApp retry config/schema/docs surface and wraps text, media, and poll outbound sends with exponential-backoff retry helpers.

## Best Possible Solution

If maintainers still want configurable WhatsApp send retry, rebase it onto current main as a WhatsApp-plugin-owned complement to the durable queue, use focused SDK subpaths, classify only true pre-send/readiness failures, and document how retry differs from queued recovery.

## Reproduction Assessment

Yes for the review findings. Source inspection shows the proposed classifier skips message-only reconnect-window errors, and the plugin import guard matches the added `infra-runtime` import.

## Solution Assessment

No, not as proposed. Configurable retry may still be useful, but this branch needs to rebase against the shipped durable queue and fix the boundary/import and classifier blockers first.
