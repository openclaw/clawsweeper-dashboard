## Summary

Keep this PR open, but do not merge it as-is. Current main still does not expose configurable WhatsApp send retry, so the PR is not obsolete; however the proposed retry classifier misses the plain reconnect-window errors the discussion says it should cover, and maintainers should first decide how this adjacent retry surface should coexist with the shipped durable delivery queue.

## What This Changes

This PR adds configurable WhatsApp outbound send retry with schema/docs metadata, shared retry helpers, production/test outbound wiring, and retry-focused tests.

## Best Possible Solution

If maintainers still want configurable immediate retry, keep it WhatsApp-plugin-owned, rebase it onto the shipped durable queue, classify actual no-listener/pre-send reconnect failures, avoid ambiguous post-send retries, and document how retry differs from queued recovery.

## Reproduction Assessment

Yes. The review finding can be reproduced by applying the PR helper logic to a plain `Error("No active WhatsApp Web listener")` or `Error("Connection Closed")`; because those errors have no allowlisted `code` or `name`, the default retry path will not retry them.

## Solution Assessment

No, not as-is. Configurable retry may still be a useful complement, but the current PR misses the reported reconnect-window error shape and needs maintainer agreement on how it should sit beside the shipped durable queue.
