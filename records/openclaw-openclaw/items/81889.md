Keep open. Current main is still QR-only for WhatsApp linking, and the related closed items only fixed QR delivery or removed a broken old pairing-code path; adding phone-pairing code login is new auth-surface work that needs maintainer product/security review.

The issue requests a WhatsApp login alternative that uses Baileys phone-pairing codes instead of QR scanning.

## Best Possible Solution

If maintainers accept the product direction, implement method-aware WhatsApp login in the WhatsApp plugin-owned auth/gateway/agent/wizard surfaces with phone validation, Baileys timing guards, method-aware status copy, regression tests, and live proof.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, CLI, gateway schema, agent-tool tests, and docs all show the current main behavior is QR-only, so the missing pairing-code capability is a feature gap rather than broken existing behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The requested Baileys primitive is real, but the maintainable solution should be a method-aware WhatsApp login contract across plugin, CLI, gateway/UI, agent tool, setup, and status copy rather than a narrow local shim.
