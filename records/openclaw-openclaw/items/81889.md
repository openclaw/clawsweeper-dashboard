## Summary

Keep open. Current main is still QR-only for WhatsApp device linking, the recent implementation attempt at https://github.com/openclaw/openclaw/pull/83482 was closed unmerged after security/API-boundary review, and the related open docs PR only documents QR limitations rather than adding the requested auth method.

## What This Changes

The issue requests WhatsApp phone-pairing-code login via Baileys `requestPairingCode` as an alternative to QR scanning.

## Best Possible Solution

If maintainers accept the product direction, implement pairing-code login as an additive WhatsApp-owned auth method across CLI, setup, gateway/UI, agent tool, docs, and status copy, with phone validation, Baileys timing guards, redaction, regression tests, and live WhatsApp proof.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, CLI, gateway schema, and agent-tool inspection show the existing product is QR-only, so the report is a missing capability rather than broken documented behavior.

## Solution Assessment

Unclear until maintainers approve the product shape. Baileys supports the primitive, but the maintainable OpenClaw solution should be method-aware and plugin-owned instead of a narrow global `--phone` shim or a standalone auth-dir script.
