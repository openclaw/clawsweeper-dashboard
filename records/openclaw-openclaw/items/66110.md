## Summary

Keep open: the requested CLI/env override is plausible, but this PR should not merge as-is because routed commands can still ignore the new flag, daemon status can report the wrong service log path, and contributor real behavior proof is missing. The linked config-path bug was already closed separately, while the dedicated `--log-file` surface still needs maintainer approval.

## What This Changes

The PR adds a root `--log-file <path>` option, an `OPENCLAW_LOG_FILE` override for logger/log-path resolution, daemon status reporting coverage, docs, and tests.

## Best Possible Solution

If maintainers want this override, land it as an approved CLI/env feature with route-first flag promotion, service-env-aware log-path resolution for status, focused regression tests, docs, and redacted real CLI/status proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runMain` can return through `tryRouteCli()` before Commander `preAction`, and status already has a service env that the PR resolver ignores. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The direction may be maintainable if maintainers approve the new surface, but this patch needs earlier root-option promotion and explicit env plumbing through log-path resolution/status first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Promote --log-file before routed commands:** `src/cli/program/preaction.ts:99-102`
  - body: The new flag is promoted only in Commander `preAction`, but `runMain` returns from `tryRouteCli()` before Commander parses routed commands. Paths such as `status` and `config get` can therefore accept the advertised root flag while silently ignoring it.
  - confidence: 0.94
- **[P2] Resolve status log paths from the service env:** `src/logging/log-file-path.ts:42`
  - body: `resolveConfiguredLogFilePath()` now reads `OPENCLAW_LOG_FILE` from the caller process, while daemon status computes `mergedDaemonEnv` for the inspected service. When those envs differ, status can show the wrong file sink; thread an explicit env through the resolver and pass `mergedDaemonEnv`.
  - confidence: 0.9
