## Summary

Keep open. Current main still lacks the root log-file/env override, and the PR addresses a real user-facing logging gap, but the proposed patch still has route-first and daemon-status correctness issues that should be fixed before merge.

## What This Changes

The PR adds a root `--log-file` option, `OPENCLAW_LOG_FILE` log-file precedence, daemon status coverage, and docs/tests for log-file override behavior.

## Best Possible Solution

Revise the branch so root log-file parsing and env promotion happen before route-first execution, log-file path resolution accepts an explicit env with `process.env` as the default, daemon status passes the merged service env, and docs/tests/changelog describe one consistent precedence model.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `openclaw --log-file /tmp/custom-openclaw.log status` on the PR branch: route-first execution can return before Commander preAction sets `OPENCLAW_LOG_FILE`; a second focused path is daemon status when the service env and caller env differ.

## Solution Assessment

No. The requested direction is narrow and maintainable, but the current patch needs earlier root-option promotion and explicit service-env threading before it reliably solves the advertised behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Promote --log-file before routed commands:** `src/cli/program/preaction.ts:96-99`
  - body: The new env var is set only from Commander `preAction`, but `runMain` can return from `tryRouteCli()` before Commander parses route-first commands. Routed paths such as `status`, `gateway status`, and `config get/unset` can therefore accept the new root flag while never setting `OPENCLAW_LOG_FILE`, so the advertised global override is silently ignored there.
  - confidence: 0.94
- **[P2] Resolve status log paths from the service environment:** `src/logging/log-file-path.ts:40`
  - body: `resolveConfiguredLogFilePath()` now reads `OPENCLAW_LOG_FILE` from `process.env`, but daemon status is reporting on the gateway service and already computes `mergedDaemonEnv`. When the caller shell and service env differ, status can show the wrong file sink; pass an explicit env through the resolver and call it with the merged service env.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `docs/help/environment.md:118`
  - body: This adds user-facing behavior and docs for `OPENCLAW_LOG_FILE` and root `--log-file`, so repo policy requires a `CHANGELOG.md` entry before landing. Add one single-line fix entry under the active release section with appropriate human contributor credit.
  - confidence: 0.86
