## Summary

Keep open: current main has the shipped `logging.file` fix, but this PR adds a separate root CLI/env override surface that still needs maintainer approval, real behavior proof, and fixes for two source-proven correctness issues before merge.

## What This Changes

The PR adds a root `--log-file <path>` option, `OPENCLAW_LOG_FILE` log-path resolution, daemon status reporting, docs, and regression tests.

## Best Possible Solution

If maintainers want this override, implement it as an approved CLI/env feature with route-first flag promotion, service-env-aware log-path resolution for status, focused tests/docs, and redacted real CLI/status proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: routed commands can return before Commander `preAction`, and status already has a service env that the PR resolver ignores. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The direction may be maintainable if maintainers approve the new surface, but this patch needs earlier flag promotion and explicit env plumbing through log-path resolution/status first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Promote --log-file before routed commands:** `src/cli/program/preaction.ts:99-102`
  - body: The new flag is promoted only in Commander `preAction`, but `runMain` can return through `tryRouteCli()` before Commander parses routed commands. Paths such as `status` and routed `config get` can accept the advertised root flag while silently ignoring it.
  - confidence: 0.94
- **[P2] Resolve status log paths from the service env:** `src/logging/log-file-path.ts:41`
  - body: `resolveConfiguredLogFilePath()` reads `OPENCLAW_LOG_FILE` from the caller process, while daemon status already computes the inspected service env. When those envs differ, status can show the wrong file sink; thread an explicit env through the resolver and pass the daemon env.
  - confidence: 0.9
