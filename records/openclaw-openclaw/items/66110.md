## Summary

Keep open: current main still lacks the requested root/env log-file override, and the branch has merge-blocking correctness issues around route-first commands and daemon status environment resolution. The related issue was closed for the existing `logging.file` config bug only, so this PR is now a separate root flag/env override feature with test-only proof.

## What This Changes

The PR adds a root `--log-file <path>` option, an `OPENCLAW_LOG_FILE` override in logger/log-path resolution, daemon status reporting changes, and related docs/tests.

## Best Possible Solution

Treat `--log-file`/`OPENCLAW_LOG_FILE` as a separate feature: if maintainers want it, promote the root option before route-first execution, let log-path helpers accept an explicit env, pass `mergedDaemonEnv` from daemon status, and require redacted real CLI/status proof before landing.

## Reproduction Assessment

Yes. A high-confidence source reproduction is `openclaw --log-file /tmp/custom.log status` on the PR branch: `runMain` can return through `tryRouteCli()` before Commander `preAction` promotes the flag. The daemon-status mismatch is also source-reproducible because the resolver reads the caller env instead of the inspected service env.

## Solution Assessment

No. The direction may be maintainable if maintainers approve the new override surface, but this implementation needs earlier root-option promotion and explicit env plumbing through log-path resolution/status first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Promote --log-file before routed commands:** `src/cli/program/preaction.ts:96-99`
  - body: The branch only sets `OPENCLAW_LOG_FILE` from Commander `preAction`, but `runMain` returns from `tryRouteCli()` before Commander parses route-first commands. After adding `--log-file` as a root option, routed paths such as `status`, `gateway status`, and `config get/unset` can accept the flag but never promote it, so the advertised global override is ignored.
  - confidence: 0.94
- **[P2] Resolve status log paths from the service env:** `src/logging/log-file-path.ts:40`
  - body: `resolveConfiguredLogFilePath()` now reads `OPENCLAW_LOG_FILE` from the caller process, while `gatherDaemonStatus()` computes a merged daemon/service environment for the gateway being inspected. When those envs differ, status can show the wrong file sink; accept an explicit env in the resolver and pass `mergedDaemonEnv`.
  - confidence: 0.9
