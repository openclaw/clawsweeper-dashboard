## Summary

Keep open. Current main still lacks root/env log-file overrides, so the PR addresses a real logging gap, but the current PR head has two merge-blocking correctness issues around route-first CLI commands and daemon-status environment resolution.

## What This Changes

The branch adds a root `--log-file <path>` option, an `OPENCLAW_LOG_FILE` override path, logger/log-path precedence updates, daemon status coverage, CLI parser tests, and logging/environment docs.

## Best Possible Solution

Revise the PR so root log-file parsing and env promotion happen before route-first execution, the log-file env resolver accepts an explicit env with `process.env` as the default, daemon status passes the merged service env, and the focused CLI/logger/status/docs coverage remains aligned with that precedence model.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `openclaw --log-file /tmp/custom.log status`: current PR code promotes the flag only in Commander preAction, but `runMain` returns through `tryRouteCli()` for `status` before Commander parsing.

## Solution Assessment

No. The requested direction is sound, but the current implementation is not the best fix because it misses route-first commands and makes log-path resolution depend only on the current process env instead of the inspected service env.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Promote --log-file before routed commands:** `src/cli/program/preaction.ts:96-99`
  - body: The new env var is set only from Commander `preAction`, but `runMain` returns from `tryRouteCli()` before Commander parses. Routed paths like `status`, `gateway status`, and `config get/unset` will still ignore `openclaw --log-file /tmp/x ...`, so the advertised global override is unreliable.
  - confidence: 0.94
- **[P2] Thread service env into log path resolution:** `src/logging/log-file-path.ts:40`
  - body: `resolveConfiguredLogFilePath()` now reads `OPENCLAW_LOG_FILE` from the current process env, but daemon status is reporting the gateway service and already computes `mergedDaemonEnv`. When the CLI env and service env differ, `status.logFile` can show the wrong file sink.
  - confidence: 0.9
