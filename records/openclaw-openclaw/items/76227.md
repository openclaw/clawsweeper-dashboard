## Summary

Keep open: the PR targets a real hardcoded completion-cache timeout surface, but the new resolver accepts partial numeric prefixes, which can turn common invalid-looking values into very small timeouts instead of falling back safely.

## What This Changes

The PR adds `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS`, shares a resolver between the post-update and doctor completion-cache spawn paths, adds resolver unit tests, and updates the changelog.

## Best Possible Solution

Land the timeout knob with strict positive-safe-integer parsing, extra invalid-value tests, and a short CLI docs note while preserving the 30000ms default and both spawn-site hookups.

## Reproduction Assessment

Yes for the review finding: a direct Node check shows the proposed `parseInt` logic accepts partial prefixes such as `60_000` -> `60` and `1e5` -> `1`. The original slow-hardware timeout is supported by the linked #75955 context but was not locally reproduced.

## Solution Assessment

No, not as written. The env-only override is the right narrow boundary, but the resolver should validate the whole value, preferably with the existing strict positive-integer helper, before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the timeout env var before accepting it:** `src/cli/completion-runtime.ts:36`
  - body: `Number.parseInt` accepts partial prefixes, so values like `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS=60_000` or `1e5` become `60` and `1` instead of falling back. That makes the slow-machine escape hatch fail almost immediately for common malformed inputs, so validate the whole value or reuse the strict positive-integer parser before returning it.
