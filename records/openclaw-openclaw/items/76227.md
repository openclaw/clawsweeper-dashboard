## Summary

Keep open: current main still hardcodes the 30-second completion-cache spawn timeout, and this PR is a focused implementation for making that budget configurable. It still needs contributor-side proof that the actual update/doctor parent spawn path honors the env var, and the docs line should not refer to a nonexistent `openclaw doctor completion` command.

## What This Changes

The PR adds an `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS` env-var resolver, wires it into the update and doctor completion-cache `spawnSync` calls, and documents/tests the strict parsing behavior.

## Best Possible Solution

Choose the canonical timeout policy, keep the shared resolver if the env-var knob is accepted, correct the docs wording, and require redacted terminal/live output from the update or doctor parent-spawn path before merge.

## Reproduction Assessment

Yes for the source-level failure mode: current main passes a fixed 30_000 ms timeout into both affected `spawnSync` calls. I did not locally reproduce a >30s cache generation, but related PR evidence reports a direct cache write taking about 36s while the shipped 30s parent spawn returns ETIMEDOUT.

## Solution Assessment

Mostly yes: a shared strict resolver is maintainable and preserves default behavior. The docs should name the real doctor surface, and maintainers need to decide whether this env knob is preferable to the overlapping fixed-120s approach.
