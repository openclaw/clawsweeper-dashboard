## Summary

Keep open: current main still uses a fixed 30s completion-cache spawn timeout, and this PR has sufficient terminal proof for the configurable-timeout path, but the diff still documents an invalid `openclaw doctor completion` command and overlaps with another open timeout-policy PR.

## What This Changes

The branch adds an env-var-controlled completion-cache spawn timeout, applies it in update and doctor cache refresh paths, documents it, updates changelog/generated baseline hashes, and adds resolver tests.

## Best Possible Solution

Land one coherent completion-cache timeout policy after fixing the doctor wording; if maintainers prefer an operator knob, keep this shared resolver and both spawn-site updates, otherwise consolidate with the fixed-cap/progress approach in https://github.com/openclaw/openclaw/pull/72910.

## Reproduction Assessment

No high-confidence live timeout was reproduced in this read-only review. Source inspection confirms both affected current-main spawn sites still use a fixed 30000ms timeout, and related PR context documents slow real environments hitting that ceiling.

## Solution Assessment

Mostly yes: a shared strict env-var resolver is a narrow maintainable way to let slow hosts raise the budget while preserving the default. The docs/comment wording should not name `openclaw doctor completion`, and maintainers should choose how this relates to the overlapping fixed-cap PR.
