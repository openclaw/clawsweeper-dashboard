## Summary

Keep open: current main still hardcodes the 30000ms completion-cache spawn timeout in both affected paths, and the PR is a narrow useful operator escape hatch. The remaining issue is a small docs/comment wording bug that names an unregistered `openclaw doctor completion` command.

## What This Changes

The PR adds `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS`, a shared strict parser for completion-cache spawn timeouts, updates the update and doctor spawn paths, and adds parser tests, docs, changelog, and generated baseline hash updates.

## Best Possible Solution

Keep the shared strict timeout resolver and default behavior, correct the doctor wording in docs and nearby comments, then land after normal current-head validation.

## Reproduction Assessment

No high-confidence current-main timeout failure was reproduced in this read-only review. Source inspection does confirm both affected spawn sites still use a fixed 30000ms timeout, and the PR body plus https://github.com/openclaw/openclaw/pull/75955 provide the slow-hardware context.

## Solution Assessment

Mostly yes. A shared strict env-var resolver is the narrow maintainable way to add an operator override while preserving defaults, but the docs should refer to `openclaw doctor` rather than `openclaw doctor completion`.
