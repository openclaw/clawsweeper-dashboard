## Summary

Keep open. The timeout knob is useful and current main still has the hardcoded 30000ms spawn timeout, but the PR’s resolver accepts malformed values as very small timeouts, so it should not merge as written.

## What This Changes

The PR adds `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS`, routes the update and doctor completion-cache spawn sites through a shared timeout resolver, adds resolver tests, and updates the changelog.

## Best Possible Solution

Land the timeout override with strict positive-safe-integer parsing, invalid-looking numeric test cases, and a short CLI docs note while preserving the default 30000ms behavior and both spawn-site hookups.

## Reproduction Assessment

Yes for the review finding: a direct Node check reproduces the proposed parser accepting partial numeric prefixes. The underlying slow-machine timeout motivation is supported by the related #75955 context but was not locally reproduced here.

## Solution Assessment

No, not as written. The env-only override is the right narrow boundary, but the resolver should validate the whole value, preferably by reusing `parseStrictPositiveInteger`, before returning it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the timeout env var before accepting it:** `src/cli/completion-runtime.ts:36`
  - body: `Number.parseInt` accepts partial prefixes, so values like `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS=60_000`, `1e5`, or `1.5` become `60`, `1`, and `1` instead of falling back. That makes the slow-machine escape hatch fail almost immediately for common malformed inputs, so validate the whole value or reuse the strict positive-integer parser before returning it.
  - confidence: 0.91
