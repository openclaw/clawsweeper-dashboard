## Summary

Keep this PR open: current main still hardcodes the completion-cache spawn timeout, and this branch is a focused, backwards-compatible way to let slow hosts raise that budget. There are no blocking code-review findings, but maintainers should choose how to consolidate this with the overlapping completion-cache timeout and doctor fast-path PRs before merge.

## What This Changes

The branch adds a shared `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS` resolver, wires it into update and doctor completion-cache subprocess timeouts, and adds resolver tests plus docs and changelog text.

## Best Possible Solution

Land one consolidated completion-cache refresh policy that keeps default behavior compatible, gives slow hosts enough budget or a documented override, preserves the fast plugin-skip path for host-owned cache refreshes, and covers update plus doctor with focused tests and docs.

Yes for the source-level behavior: current main has hardcoded 30000 ms spawn timeouts in the update and doctor completion-cache paths. The slow-host timeout itself is supported by linked reports and contributor Pi proof, but I did not locally reproduce a real ETIMEDOUT on current main.

## Solution Assessment

Unclear: the PR implementation is focused and appears correct, but the best product direction still needs maintainer choice between an env knob, a larger default with progress output, the doctor fast path, or a consolidated combination.
