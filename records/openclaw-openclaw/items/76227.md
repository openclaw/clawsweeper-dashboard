## Summary

Keep open: current main still hardcodes the 30-second parent-spawn timeout, and this PR is a plausible narrow implementation, but merge readiness still depends on maintainer acceptance of a new env-var config surface and proof that the update or doctor parent spawn path honors it.

## What This Changes

The PR adds `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS`, a shared timeout resolver, update/doctor spawn wiring, parser tests, docs, and a changelog entry.

## Best Possible Solution

Choose one canonical timeout policy, then land it with parent-spawn proof, targeted wiring coverage, docs, and changelog alignment.

## Reproduction Assessment

No high-confidence current-main live repro was established for the slow-hardware timeout. Source inspection does confirm both parent spawn sites still use a hardcoded 30_000 ms timeout, and related reports show the same class of timeout failure on slower installs.

## Solution Assessment

Unclear for merge readiness: the shared strict resolver is maintainable if maintainers want an env-var knob, but the repository already has an overlapping fixed-timeout/progress PR and the contributor proof should cover the parent spawn boundary.
