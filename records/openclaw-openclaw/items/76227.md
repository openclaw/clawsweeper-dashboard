## Summary

Keep open. The PR addresses a current-main timeout boundary that is still present in both update and doctor completion-cache paths, but the added docs still name an invalid `openclaw doctor completion` command and maintainers still need to choose how this overlaps with https://github.com/openclaw/openclaw/pull/72910.

## What This Changes

The PR adds `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS`, a shared strict timeout resolver, uses it for update and doctor completion-cache spawns, and adds docs, changelog, and parser tests.

## Best Possible Solution

Land one coherent completion-cache timeout policy after fixing the doctor wording; either keep this shared strict env-var resolver or consolidate around the fixed-cap/progress approach in https://github.com/openclaw/openclaw/pull/72910.

## Reproduction Assessment

No high-confidence live timeout was reproduced here. Source inspection confirms current main still uses fixed `30000` ms timeouts in both affected spawn sites, and the PR plus related timeout work provide real slow-host context.

## Solution Assessment

Mostly yes: a shared strict resolver is a narrow maintainable way to let slow hosts raise the budget while preserving the default. The docs should not name `openclaw doctor completion`, and maintainers should decide how this relates to https://github.com/openclaw/openclaw/pull/72910.
