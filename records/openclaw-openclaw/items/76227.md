## Summary

Keep open: current main still hardcodes the 30000ms completion-cache spawn timeout in both update and doctor paths, while this active PR provides a focused configurable-timeout implementation with sufficient terminal proof. The remaining concrete defect is a docs wording bug, plus maintainers still need to choose how this relates to the overlapping fixed-timeout/progress PR.

## What This Changes

The PR adds `OPENCLAW_COMPLETION_CACHE_WRITE_TIMEOUT_MS`, a shared strict timeout resolver, tests, docs, changelog text, and wires update and doctor completion-cache spawns through it.

## Best Possible Solution

Land one coherent completion-cache timeout policy after fixing the docs wording; either keep this shared strict env-var resolver or explicitly choose the fixed-timeout/progress approach in the overlapping PR.

## Reproduction Assessment

No high-confidence live timeout was reproduced in this review. Source inspection shows current main still uses fixed 30000ms spawn timeouts in both affected paths, and linked PR/issue evidence documents real ETIMEDOUT behavior on slow hosts.

## Solution Assessment

Mostly yes, but it still needs maintainer policy choice. A shared strict resolver is narrow and default-preserving, while the overlapping PR offers a fixed 120s timeout plus progress output; the invalid doctor command wording should be fixed either way.
