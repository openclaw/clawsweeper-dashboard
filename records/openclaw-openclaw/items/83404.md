## Summary

Keep open: current main still re-enters the same queued follow-up transcript prompt for each fallback candidate and lacks assistant-error-stub suppression, while the linked fix PR remains open and unmerged. The steer-mode update broadens this into a failover transcript-corruption bug rather than making the original report obsolete.

## What This Changes

The issue reports that model failover retries can duplicate transcript entries, corrupting session history in followup mode and later also in steer mode.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83417 or an equivalent replacement that makes fallback transcript persistence idempotent for queued user messages and failed assistant stubs, then keep the group-chat context issue separate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: configure followup or steer with multiple fallback models, force failed fallback candidates, and inspect session JSONL; this review did not run the live OpenRouter/channel path.

## Solution Assessment

Is this the best way to solve the issue? Yes: the fix should be at the fallback/session-persistence boundary so retries do not re-record the same logical turn, with focused regression tests and live proof before the issue closes.
