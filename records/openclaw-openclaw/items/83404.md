## Summary

Keep open: current main still sends the same queued transcript prompt through each model-fallback candidate without a cross-candidate transcript-persistence latch, and the linked fix PR that closes this issue is open but not merged.

## What This Changes

The issue reports that model fallback retries corrupt session transcripts by re-persisting queued follow-up user messages and assistant error stubs during failed attempts.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/83417, with regression coverage proving followup and steer mode fallback candidates do not duplicate transcript entries.

## Reproduction Assessment

Yes at source level: current main re-passes the queued transcript prompt through each fallback candidate and only has a one-shot user-message suppression flag. I did not run a live OpenRouter/channel reproduction in this read-only review.

## Solution Assessment

Yes: the linked PR's direction of latching queued user persistence across fallback candidates and suppressing nonterminal assistant error persistence is the narrowest maintainable fix. The issue should remain open until that PR is merged or replaced.
