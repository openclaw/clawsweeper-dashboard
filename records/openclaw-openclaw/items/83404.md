## Summary

Keep open: the report targets a documented follow-up queue mode, and current source shows queued follow-up prompts re-enter the model fallback loop without clear per-logical-message transcript persistence idempotency.

## What This Changes

The issue asks OpenClaw to record queued follow-up user messages only once per logical queued turn, even when model failover retries the run across fallback candidates.

## Best Possible Solution

Make queued follow-up transcript persistence idempotent per logical queued message or run across fallback candidates, with regression coverage for a failing primary candidate and a fallback attempt.

## Reproduction Assessment

Yes at source level with medium confidence: followup queueing and fallback retry paths are present, and the same queued prompt is re-entered for fallback candidates. A live provider/channel reproduction was not run during this read-only review.

## Solution Assessment

Yes: the narrow fix should be an idempotency guard at the queued-turn or fallback boundary, not a new queue mode or OpenRouter-specific retry policy change.
