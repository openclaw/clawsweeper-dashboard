## Summary

Keep open: current main has queue-time message-id dedupe, but the queued follow-up run can still re-enter the model fallback loop with the same prompt/transcript prompt and no durable per-logical-message transcript persistence guard across fallback candidates.

## What This Changes

The issue asks OpenClaw to persist queued follow-up user messages only once when model failover retries the same logical follow-up run.

## Best Possible Solution

Add queued-follow-up transcript idempotency across fallback candidates using the queued run's stable logical message identity, with regression coverage that proves the queued user turn is written once after a failed primary and successful fallback.

## Reproduction Assessment

Yes at source level with medium confidence: follow-up queueing, fallback retry, and transcript persistence paths are present, and the same queued prompt is passed into each fallback candidate without a durable queued-message persistence key. A live OpenRouter/channel reproduction was not run during this read-only pass.

## Solution Assessment

Yes: the narrow fix should make transcript persistence idempotent at the queued-turn or fallback boundary. A new queue mode or OpenRouter-specific retry policy would not address the core replayed transcript write path.
