## Summary

Keep open. Current main still publishes a device-paired node's declared command list before approved node pairing exists, `node.invoke` still authorizes from that live list, and the prior paired PR was closed unmerged after a real concern about preserving same-connection post-approval command availability.

## What This Changes

The issue asks OpenClaw to prevent `node.invoke` from executing node commands until node-pair approval has established trust for that node.

## Best Possible Solution

Maintainers should either restore a fail-closed pre-approval command gate that also refreshes or sources commands after same-connection approval, or explicitly update the pairing docs, security guidance, changelog expectation, and tests to document the current behavior as intentional.

## Reproduction Assessment

Yes. The source path is high-confidence: connect a device-paired node without approved node pairing, observe declared commands on the live session, then `node.invoke` authorizes from `NodeSession.commands` with only the global allowlist.

## Solution Assessment

No merged solution is present. The closed PR's one-line empty-command approach blocked pre-approval exposure but left a same-connection post-approval refresh gap, so a revised fix or explicit policy/doc correction is needed.
