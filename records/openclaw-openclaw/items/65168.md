## Summary

Keep open. Current main still exposes a device-paired node's declared command list before an approved node-pair record exists, and this issue is already paired with open unmerged PR #65169 using closing syntax.

## What This Changes

The issue requests fail-closed node command exposure and `node.invoke` behavior until node-pair approval exists.

## Best Possible Solution

Review and land PR #65169 or an equivalent narrow fix if node-pair approval is intended to gate command exposure; otherwise update pairing docs, security guidance, changelog expectations, and regression tests together.

## Reproduction Assessment

Yes. The current source path plus `server.roles-allowlist-update.test.ts` give a high-confidence reproduction: connect a device-paired-only node with declared commands, observe those commands on the live session before node-pair approval, then `node.invoke` authorizes from that session command list.

## Solution Assessment

Unclear until maintainers confirm the intended policy. If the 2026.3.31 command-gating docs are authoritative, failing closed before node-pair approval is the narrowest maintainable fix; if the handshake/trusted-operator docs are authoritative, the fix should be coordinated docs and test correction instead of code-only hardening.
