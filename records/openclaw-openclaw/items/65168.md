## Summary

Keep open. Current main still publishes a device-paired node's declared commands before an approved node-pair record exists, `node.invoke` still authorizes from that live session command list, and an open unmerged PR already targets this issue with closing syntax.

## What This Changes

The issue reports that `node.invoke` remains reachable for allowlisted commands on a reconnecting device-paired node before `node.pair.approve` creates an approved node-pair record.

## Best Possible Solution

Review and land PR #65169 or an equivalent narrow fix if node-pair approval is intended to gate command exposure; otherwise update pairing docs, security guidance, changelog expectations, and tests together to make the trusted-operator contract explicit.

## Reproduction Assessment

Yes. The current source path and the existing `server.roles-allowlist-update.test.ts` case give a high-confidence reproduction: connect a device-paired-only node with declared commands, observe those commands in the live session before node-pair approval, then `node.invoke` authorizes from that live command list.

## Solution Assessment

Unclear until maintainers resolve the intended policy. If the pairing docs and changelog are authoritative, fail-closing the command set before node-pair approval is the narrowest maintainable fix; if the security-doc trusted-operator stance is authoritative, the safer solution is a coordinated docs/test correction rather than a code-only patch.
