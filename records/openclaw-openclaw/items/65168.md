## Summary

Keep open. Current main still publishes declared commands for a device-paired node before node-pair approval exists, `node.invoke` still authorizes from that live command list, and the issue is already paired with open unmerged PR #65169 using closing syntax.

## What This Changes

The issue asks OpenClaw to fail closed so reconnecting nodes cannot expose `node.invoke` commands until node-pair approval exists.

## Best Possible Solution

Review PR #65169 or an equivalent narrow fix. If the documented node-pairing gate is intended, publish an empty command set before node-pair approval and add regression coverage for `node.invoke`; if the current trusted-operator behavior is intentional, align the pairing docs, security guidance, and tests together.

## Reproduction Assessment

Yes. The current source path plus the `server.roles-allowlist-update.test.ts` case give a high-confidence reproduction: connect a device-paired-only node with declared commands, observe those commands on the live node session before node-pair approval, then `node.invoke` authorizes from that session list.

## Solution Assessment

Unclear until maintainers settle the intended security contract. A fail-closed command set before node-pair approval is the narrowest code fix if the docs are authoritative; otherwise the safer resolution is an explicit docs/test update rejecting node-pairing as a pre-command gate.
