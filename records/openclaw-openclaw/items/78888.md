## Summary

Close as duplicate/superseded: current main still uses JSON/JSONL session persistence, but the remaining database-backed session storage design is already covered by broader open storage trackers, and concrete sessions.list latency is tracked separately.

## What This Changes

The issue requests replacing per-agent JSON/JSONL session persistence with a per-agent SQLite database containing sessions, messages, and metadata tables.

## Best Possible Solution

Keep the database-backed storage design centralized in https://github.com/openclaw/openclaw/issues/13440 and https://github.com/openclaw/openclaw/issues/63492; if SQLite is chosen, implement it through a versioned persistence contract with migration, rollback, transcript semantics, benchmarks, and docs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. The duplicate status is source- and discussion-backed, while concrete performance reproduction is already tracked on https://github.com/openclaw/openclaw/issues/75839.

## Solution Assessment

Yes for cleanup: closing this as superseded keeps storage architecture discussion on the broader canonical issues. Implementing a standalone SQLite replacement from this issue would fragment migration, transcript, config, and durability decisions.
