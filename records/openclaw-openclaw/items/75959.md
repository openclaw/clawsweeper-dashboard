## Summary

Current main already implements the requested Kimi Code Plan image path: the bundled Kimi provider uses the Kimi coding endpoint, an approved `claude-code/0.1.0` User-Agent, declares image input for `kimi-code`, and the Anthropic Messages transport preserves image blocks for image-capable models.

## What This Changes

Request image-capable Kimi Code Plan requests with a Kimi-whitelisted User-Agent and correct endpoint so attached images do not fail with HTTP 403.

## Best Possible Solution

Keep the bundled Kimi Coding provider on the image-capable Anthropic-compatible path; a new issue should include a latest-main live failure log only if Kimi still rejects image turns with this configuration.

## Reproduction Assessment

Yes for the source-level path: the report provides a concrete failing request, and current main traces to a Kimi provider that sends image-capable Anthropic Messages requests with the approved User-Agent. A live vendor smoke was not feasible because no Kimi key is present in the environment.

## Solution Assessment

Yes. The implemented provider-local fix is narrower than a core change: Kimi-specific endpoint, headers, auth choice, and model capabilities stay inside the Kimi plugin while the shared transport handles image blocks generically.
