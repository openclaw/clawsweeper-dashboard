## Summary

Keep open. Current main already uses the Kimi-whitelisted User-Agent, but the bundled Kimi provider and onboarding still route image-capable `kimi-code` defaults through `anthropic-messages` and the old coding base URL; open PR #76002 is the active closing path and should be reviewed before this issue closes.

## What This Changes

The issue asks OpenClaw to make Kimi Code Plan image attachments work by routing the bundled Kimi provider through Kimi's OpenAI-compatible coding endpoint while retaining an accepted User-Agent.

## Best Possible Solution

Land a narrow Kimi plugin change that updates the default catalog and onboarding API/base URL to Kimi's OpenAI-compatible coding endpoint, preserves explicit user overrides, and updates the existing Kimi provider tests plus changelog.

## Reproduction Assessment

Yes, by source inspection and the reporter's curl evidence. Current main publishes `kimi-code` and `k2p5` as image-capable while defaults and onboarding still select `anthropic-messages` at `https://api.kimi.com/coding/`, which sends Anthropic image blocks instead of the working OpenAI `image_url` payload.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned default `api` and `baseUrl` changes in `extensions/kimi-coding` with override behavior preserved; a core transport rewrite or new user-facing config knob is not needed.
