## Summary

Current main implements the requested Kimi Code Plan image path well enough: the bundled Kimi provider now advertises image-capable `kimi-code`/`k2p5` models and sends Kimi's whitelisted `User-Agent: claude-code/0.1.0` on the Kimi Coding endpoint. The proof change is on main after `v2026.4.29`, so this is main-only until the next release.

## What This Changes

Request support for image attachments with Kimi Code Plan by adjusting the Kimi provider endpoint/header behavior so `kimi-code` vision requests no longer fail with 403.

## Best Possible Solution

Keep the Kimi Coding provider-owned defaults as the source of truth: image-capable Kimi models, the Kimi-approved User-Agent, and the existing Anthropic Messages transport; only add an OpenAI `/chat/completions` route later if live evidence shows the current transport still fails.

## Reproduction Assessment

Yes. The issue gives a concrete Kimi Code Plan image request failure tied to the old User-Agent, and current-main source now satisfies the two relevant conditions: whitelisted User-Agent and image-capable model metadata.

## Solution Assessment

Yes. The current implementation is the narrow provider-owned fix: it changes Kimi plugin defaults and preserves the Anthropic Messages transport that existing Kimi Coding tool-compatibility code supports, instead of adding a broader core transport switch.
