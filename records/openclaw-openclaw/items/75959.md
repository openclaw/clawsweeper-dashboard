## Summary

Keep open. Current main partially addresses the reported User-Agent problem, but the Kimi provider still advertises image input while using the Anthropic-compatible request path, and the current vendor docs raise endpoint, model-id, and client-identity questions that need a deliberate provider fix rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make Kimi Code Plan image attachments work by using a Kimi-compatible request endpoint, image payload format, and User-Agent.

## Best Possible Solution

Make the bundled Kimi provider's advertised image capability match a vendor-compliant, live-verified request path, or stop advertising image input until Kimi Code images work without regressing Kimi tool and thinking behavior.

## Reproduction Assessment

Unclear for the live HTTP failure because no Kimi key was available. The source-level request path is clear: current main sends Kimi image-capable model traffic through the Anthropic-compatible `/v1/messages` transport rather than the OpenAI `chat/completions` image format described in the issue.

## Solution Assessment

Unclear. The requested endpoint switch may be directionally useful for images, but the safest solution needs maintainer review because current docs/tests intentionally use the Anthropic-compatible path and vendor docs add model-id and User-Agent constraints.
