## Summary

Keep open: the PR fixed the earlier hard-coded provider-family and changelog concerns, but the latest hook-owned cache still risks reusing provider hook output across context inputs the hook contract exposes. The supplied benchmark is useful but stale because it was run at `65aba3cccf8a7ff93321e1dbeac75746011dc6de`, while the current head is `c3346ebaf984ec781218a81ee30e574a1690c29c` after cache-key changes.

## What This Changes

The PR adds an in-memory LRU cache for provider-normalized agent tool parameters, exports provider hook identity lookup, expands cache tests, and adds a changelog entry.

## Best Possible Solution

Land the optimization only after cacheability and key material are owned by the provider hook contract, or the cache is restricted to hooks whose inputs are fully captured, with refreshed current-head real behavior proof.

## Reproduction Assessment

Not applicable as a perf PR rather than a bug report. Source inspection confirms the patch targets the embedded agent tool-normalization path, and the PR body supplies a microbenchmark for an older head.

## Solution Assessment

No. Caching cloned normalized parameters is a reasonable direction, but the current generic cache key should not assume provider hook output is independent of context inputs that the hook contract exposes.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Key schema-cache results by hook context inputs:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:140-153`
