## Summary

Keep open: current PR head still has a correctness issue where the cache-key path re-resolves the provider plugin while the normalization path uses the prepared runtime handle, so cached parameters can be keyed or bypassed by a different hook owner than the one that produced them.

## What This Changes

The PR adds provider-owned caching for normalized tool schema parameters, a `resolveToolSchemaCacheKey` provider hook, docs/API baseline updates, tests, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Thread the prepared runtime handle through cache-key resolution, add prepared or mismatched-provider regression coverage, then continue maintainer review of the new provider SDK hook.

## Reproduction Assessment

Yes for the review finding: current PR source shows cache-key resolution omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache design is conservative, but the cache-key resolver must use the same `ProviderRuntimePluginHandle` that owns normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache-key path calls `resolveProviderToolSchemaNormalizeCacheKey` without `params.runtimeHandle`, but the normalization path below passes `runtimeHandle` into `normalizeProviderToolSchemasWithPlugin`. In prepared runtime-plan turns, this can key or bypass the cache using a re-resolved provider plugin and then store parameters normalized by the prepared handle's plugin; thread the same handle into key resolution and cover a prepared or mismatched-provider case.
  - confidence: 0.9
