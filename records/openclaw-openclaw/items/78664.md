## Summary

Keep open: the performance direction is plausible and the supplied terminal proof is adequate, but the latest PR head still has a blocking correctness issue where cache-key resolution can use a different provider plugin than the prepared handle used for normalization.

## What This Changes

The PR adds a provider-owned cache for normalized tool schema parameters in embedded agent turns, plus SDK/docs/tests, an API baseline update, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Thread the prepared `ProviderRuntimePluginHandle` into cache-key resolution, add a prepared-only or mismatched-provider regression test, then continue maintainer API review of the SDK hook.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows cache-key resolution omits the prepared provider handle while normalization uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache should be keyed through the same prepared provider handle that owns normalization, preferably by extending the resolver params and reusing the existing prepared-handle helper pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache-key path calls `resolveProviderToolSchemaNormalizeCacheKey` without `params.runtimeHandle`, but normalization below passes that handle into `normalizeProviderToolSchemasWithPlugin`. In prepared turns this can key or bypass the cache using one provider plugin and then store normalized parameters from another; please thread the prepared handle into cache-key resolution and cover a prepared-only or mismatched-provider case.
  - confidence: 0.88
