## Summary

Keep open: the refreshed proof now matches the current PR head, but the cache key still assumes provider hook output is independent of public hook context fields that are not fully keyed.

## What This Changes

This PR adds an in-memory LRU cache for provider-normalized embedded-agent tool parameters, exports provider hook identity lookup, adds cache coverage, and updates the changelog.

## Best Possible Solution

Make provider-hook cacheability explicit and keyed by hook-owned inputs, or scope the cache to hooks whose context dependencies are fully captured before landing.

## Reproduction Assessment

Not applicable as a perf PR rather than a bug report. Source inspection and the current-head PR benchmark cover the targeted normalization hot path.

## Solution Assessment

No. Caching cloned parameters is plausible, but the cache should not reuse provider hook output unless the hook contract owns the cache key or the implementation restricts caching to fully captured inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Key schema cache results by hook-owned context:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:140-153`
  - body: `buildProviderToolSchemaCacheKey` can reuse normalized parameters when public hook context fields outside the selected provider/model strings have changed. Since provider hooks receive `config`, `env`, `workspaceDir`, and the full `model`, a third-party hook can legitimately vary normalization on those inputs and get stale parameters on the second call; make the hook provide the cache key/cacheability decision or restrict caching to hooks whose inputs are fully captured.
