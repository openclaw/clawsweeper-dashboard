## Summary

Keep open: the PR still has a blocking correctness issue because cache-key resolution re-resolves the provider plugin instead of using the prepared runtime handle that produces the cached normalized parameters.

## What This Changes

The PR adds a provider-owned normalized tool-schema parameter cache for embedded-agent turns, exposes a `resolveToolSchemaCacheKey` provider hook, documents the SDK contract, refreshes API baseline data, and adds tests plus an `OPENCLAW_TOOL_SCHEMA_CACHE=0` kill switch.

## Best Possible Solution

Keep the provider-owned schema-parameter cache design, but thread the prepared provider runtime handle into cache-key resolution and add a regression test for a prepared-handle versus registry-resolution mismatch.

## Reproduction Assessment

Yes for the review finding: PR-head source shows the cache-key resolver omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness path, not a current-main end-user bug reproduction.

## Solution Assessment

No, not as currently written. The cache design is reasonable, but the resolver must share the prepared provider handle with the normalizer before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`

A prepared runtime-plan turn can therefore key, hit, or store under one provider hook while normalizing with another; pass the prepared handle into cache-key resolution and cover the mismatch case.
