## Summary

Keep this PR open, but it still needs changes before merge: the latest head resolves the cache key by re-resolving the provider plugin, while the normalization call uses the prepared runtime handle, so prepared turns can key or bypass the cache with one provider and store another provider's normalized parameters.

## What This Changes

The PR adds a provider-owned cache for normalized tool schema parameters, a `resolveToolSchemaCacheKey` provider SDK hook, docs/API baseline/changelog updates, and cache behavior tests.

## Best Possible Solution

Thread the prepared runtime handle through cache-key resolution, add prepared or mismatched-provider regression coverage, and then continue with maintainer review of the new provider SDK hook.

## Reproduction Assessment

Yes for the review finding: latest PR source shows cache-key resolution omits the prepared provider handle while normalization uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache design is maintainable, but the cache-key resolver must use the same `ProviderRuntimePluginHandle` that owns normalization.
