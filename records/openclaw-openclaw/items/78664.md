## Summary

Keep this PR open: the cache direction has supplied proof, but current head still resolves the cache key without the prepared provider runtime handle while normalization uses that handle, so a prepared turn can key one provider and store another provider's normalized parameters.

## What This Changes

This PR adds a provider-owned cache for normalized tool schema parameters in embedded agent turns, exposes and documents a `ProviderPlugin.resolveToolSchemaCacheKey` SDK hook, and updates tests, API baseline, and changelog.

## Best Possible Solution

Resolve the cache key through the same `ProviderRuntimePluginHandle` used for normalization, preserve hook-owned opt-in semantics, add a prepared-handle regression test, then continue maintainer API review.

## Reproduction Assessment

Yes for the review finding: source inspection at current PR head shows cache-key resolution omits the prepared provider handle while the normalization call uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache design is maintainable, but cache-key resolution must use the same prepared provider handle that owns normalization before this is safe to merge.
