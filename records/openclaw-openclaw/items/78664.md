## Summary

Keep open: the provider-owned cache direction is useful and the PR supplies real behavior proof, but the current patch still has a blocking prepared-runtime ownership mismatch in the cache-key path.

## What This Changes

The PR adds a provider-owned cache for normalized tool schema parameters, a provider SDK cache-key hook, tests, docs, API baseline updates, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Thread the prepared `ProviderRuntimePluginHandle` through cache-key resolution, use the same provider handle for key ownership and normalization, and add regression coverage for prepared or mismatched-provider handles.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows cache-key resolution omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness path, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The conservative cache design is reasonable, but the cache-key resolver must use the same prepared provider runtime handle that owns schema normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache key is resolved by re-looking up the provider plugin, but normalized parameters are produced with `params.runtimeHandle`. Prepared runtime-plan turns can therefore key or bypass the cache using one provider plugin while storing parameters produced by another; pass the same prepared handle into cache-key resolution and cover the mismatch case.
