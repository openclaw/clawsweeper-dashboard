## Summary

Keep open: the branch still has a source-level correctness blocker in the provider-owned cache path. Cache-key resolution can use a different provider plugin than normalization because the prepared runtime handle is not threaded into the new resolver.

## What This Changes

The PR adds a provider-owned cache for normalized embedded-agent tool schema parameters, a plugin SDK cache-key hook, tests, docs, an API baseline refresh, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Keep the provider-owned schema-parameter cache design, but make cache-key resolution use the exact prepared provider runtime handle used for normalization and add regression coverage for a handle/registry mismatch.

## Reproduction Assessment

Yes for the review finding: PR-head source shows cache-key resolution omits `runtimeHandle` while normalization still uses it. This is a PR-introduced correctness path, not a current-main end-user bug reproduction.

## Solution Assessment

No, not as currently written. The cache can remain provider-owned, but the resolver must share the prepared provider handle with the normalizer before this is the narrow maintainable fix.

## Repair Work Prompt

Repair https://github.com/openclaw/openclaw/pull/78664 by preserving provider runtime-handle ownership in the new tool-schema normalization cache. The PR currently resolves the cache key without `params.runtimeHandle` while normalization uses that handle, so a prepared runtime-plan turn can hit/store under one provider hook and normalize under another. Update `resolveProviderToolSchemaNormalizeCacheKey` and its call sites to accept/use the same prepared `ProviderRuntimePluginHandle` as normalization, likely via `ensureProviderRuntimePluginHandle`, and add a regression test that fails when registry resolution differs from the supplied runtime handle. Keep cached values limited to cloned schema parameters; do not cache tool objects, execute closures, guards, approval state, or policy. Do not broaden the cache or remove the `OPENCLAW_TOOL_SCHEMA_CACHE=0` switch. The changelog entry already exists unless the repair changes user-facing behavior beyond the current PR.
