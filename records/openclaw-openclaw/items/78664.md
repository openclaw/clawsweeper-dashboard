## Summary

Keep open: the performance direction is useful and the proof is meaningful, but current PR head still resolves the cache key without the prepared provider handle while normalization uses that handle, so cache ownership can diverge in prepared runtime turns.

## What This Changes

The PR adds a provider-owned tool-schema normalization cache for embedded agent turns, exports and documents a provider SDK cache-key hook, adds tests, and updates the changelog/API baseline.

## Best Possible Solution

Resolve the cache key through the same ProviderRuntimePluginHandle used for normalization, add prepared-only or mismatched-provider regression coverage, then continue maintainer review of the SDK hook.

## Reproduction Assessment

Yes for the review finding: source inspection at current PR head shows cache-key resolution omits the prepared provider handle while the normalization call uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache direction is maintainable, but cache-key resolution must use the same prepared provider handle that owns normalization.
