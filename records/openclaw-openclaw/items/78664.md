## Summary

Keep this PR open: the branch addresses the earlier review direction by making caching hook-owned, but the current head still resolves the cache-key hook without the prepared provider runtime handle that the normalization path uses.

## What This Changes

This PR adds a provider-owned cache for normalized tool schema parameters, exposes `resolveToolSchemaCacheKey` in the provider SDK, documents the hook, adds tests, and adds `OPENCLAW_TOOL_SCHEMA_CACHE=0` as a rollback switch.

## Best Possible Solution

Thread the prepared provider runtime handle through cache-key resolution, add a prepared or mismatched-provider regression test, then continue maintainer review of the new provider SDK hook.

## Reproduction Assessment

Yes for the review finding: PR source shows cache-key resolution omits `runtimeHandle` while normalization uses it, and current main has a prepared runtime-plan handle contract. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The conservative cache design is close, but the cache-key resolver must use the same `ProviderRuntimePluginHandle` that owns normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`

In prepared runtime-plan turns, this can key or bypass the cache using a re-resolved provider plugin and then store parameters normalized by a different prepared plugin; thread the same handle into cache-key resolution and cover that case.
