## Summary

Keep open: the hot-path proof is sufficient and the clone-safety tests cover the intended cache behavior, but the implementation hard-codes provider families in the embedded runner and the user-facing perf change is missing the required changelog entry.

## What This Changes

The PR adds an LRU cache for provider-normalized tool parameters, keys it by provider/model/schema/hook identity, adds an `OPENCLAW_TOOL_SCHEMA_CACHE=0` kill switch, and expands runtime tests.

## Best Possible Solution

Keep the optimization direction, but make cache eligibility/keying provider-owned or otherwise generic to the hook contract, then add a focused changelog entry and keep the clone-safety coverage.

## Reproduction Assessment

Not applicable as a perf PR rather than a bug report. Source inspection shows the changed function is on the runtime-plan tool normalization path, and the PR body supplies a terminal microbenchmark for the hot path.

## Solution Assessment

No. Caching cloned normalized parameters is a reasonable approach, but this implementation should not encode bundled provider families in the embedded runner and also needs the required changelog entry.
