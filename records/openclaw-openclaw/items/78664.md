## Summary

Keep this PR open, but it still needs changes before merge: the cache-key path can resolve a different provider plugin from the one used for normalization when a prepared runtime handle is present.

## What This Changes

The PR adds provider-owned caching for normalized tool-schema parameters in embedded agent turns, exposes a provider SDK cache-key hook, updates plugin docs/API baselines/changelog, and adds cache behavior tests.

## Best Possible Solution

Use the same prepared provider runtime handle for cache-key resolution and normalization, add a prepared or mismatched-provider regression test, then continue maintainer review of the new provider SDK hook.

## Reproduction Assessment

Yes for the review finding: PR head source shows cache-key resolution omits the prepared provider handle while normalization uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache design is conservative, but cache-key resolution must use the same `ProviderRuntimePluginHandle` that owns normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`

## Repair Work Prompt

On PR https://github.com/openclaw/openclaw/pull/78664, fix the provider tool-schema cache-key ownership bug. `normalizeProviderToolSchemas` currently calls `resolveProviderToolSchemaNormalizeCacheKey` without the prepared `runtimeHandle` but then calls `normalizeProviderToolSchemasWithPlugin` with that handle. Update the provider-runtime/cache-key resolver path so it accepts and honors `ProviderRuntimePluginHandle` via the same prepared-handle contract used by other provider hook helpers, then add a regression test for prepared or mismatched provider handles. Keep the cache conservative: cache only cloned schema `parameters`, do not cache tool objects, execute closures, approval state, guards, or broader tool materialization. Do not broaden the public API beyond what is needed for the existing hook-owned cache design, and do not add another changelog entry unless the repair changes user-facing behavior beyond the existing PR entry.
