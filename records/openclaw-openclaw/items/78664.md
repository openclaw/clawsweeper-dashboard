## Summary

Keep open: the performance/API direction is plausible, but the patch has a concrete cache-key ownership bug where the new key resolver can choose a different provider plugin than the prepared handle used for normalization.

## What This Changes

The PR adds a provider-owned tool-schema normalization cache for embedded agent turns, an SDK cache-key hook, an env rollback switch, docs/changelog/API-baseline updates, and focused tests.

## Best Possible Solution

Keep the provider-owned, parameter-only cache design, but make cache-key resolution use the same prepared provider handle as normalization before maintainer API review and CI gating.

## Reproduction Assessment

Yes for the review finding: source inspection traces a prepared `runtimeHandle` into tool normalization while the new cache-key path omits it. Not applicable as an end-user current-main bug because this PR is a performance/API enhancement.

## Solution Assessment

No, not as currently written. The provider-owned cache is a narrow design, but the cache-key resolver must share the prepared provider handle with the normalizer to avoid mismatched plugin ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The runtime plan passes a prepared `runtimeHandle` into tool normalization, and `normalizeProviderToolSchemasWithPlugin` uses that handle to choose the exact plugin. This new cache-key call re-resolves without `params.runtimeHandle`, so a turn can key the cache with one provider plugin and then normalize/store parameters from another prepared plugin; prepared-only handles also silently bypass the cache. Please thread the same handle through cache-key resolution and cover the mismatch with a test.
  - confidence: 0.84
