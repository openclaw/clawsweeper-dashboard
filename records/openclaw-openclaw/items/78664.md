## Summary

Keep open: the provider-owned cache is a useful performance/API enhancement, but latest head still resolves the cache key without the prepared provider runtime handle, so the cache can be keyed from a different provider plugin than the one used for normalization.

## What This Changes

The branch adds provider-owned caching for normalized tool schema parameters, an SDK `resolveToolSchemaCacheKey` hook, an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch, docs/changelog/API-baseline updates, and focused tests.

## Best Possible Solution

Thread the prepared `ProviderRuntimePluginHandle` into cache-key resolution and add regression coverage for prepared-only or mismatched provider handles, then continue normal SDK/API and CI review.

## Reproduction Assessment

Yes for the review finding: source inspection traces a prepared `runtimeHandle` into tool normalization while the new cache-key path omits it. This is not a current-main end-user bug; it is a correctness issue introduced by this performance/API PR.

## Solution Assessment

No, not as currently written. The provider-owned, parameter-only cache is the right narrow direction, but cache-key resolution must use the same prepared provider handle as normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache-key path re-resolves the provider plugin without `params.runtimeHandle`, while the normalization path below uses that prepared handle. In a prepared turn this can key the cache from one provider plugin and then store normalized parameters from another, or bypass caching when the prepared plugin is not discoverable through the broad resolver. Please thread the same handle through cache-key resolution and add a mismatch/prepared-only regression test.
