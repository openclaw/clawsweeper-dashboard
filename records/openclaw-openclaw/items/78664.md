## Summary

Keep open: the cache-key path still resolves the provider plugin without the prepared runtime handle while the normalization path uses it, so prepared turns can cache or bypass against the wrong provider owner before merge.

## What This Changes

The branch adds provider-owned caching for normalized embedded-agent tool schema parameters, exports a provider SDK cache-key hook, documents it, updates the changelog/API surface, and adds focused tests.

## Best Possible Solution

Thread the prepared `ProviderRuntimePluginHandle` into cache-key resolution, add prepared-only or mismatched-provider regression coverage, then continue normal SDK/API and CI review.

## Reproduction Assessment

Yes for the review finding: source inspection shows the new cache-key path omits the prepared provider handle while the normalization path uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The provider-owned, parameter-only cache is a maintainable direction, but cache-key ownership must come from the same prepared provider handle as normalization.

## Review Findings
