## Summary

Keep this PR open, but it still needs a narrow correctness fix before merge: the cache-key path can resolve a different provider hook than the prepared handle used for normalization.

## What This Changes

The PR adds a provider-owned normalized tool-schema-parameter cache, a provider SDK cache-key hook, tests, docs/API baseline updates, a changelog entry, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Keep the provider-owned normalized-parameters cache, but thread the prepared provider runtime handle through cache-key resolution and leave broader gateway TTFT work under https://github.com/openclaw/openclaw/issues/80131.

## Reproduction Assessment

Yes for the review finding: PR-head source shows cache-key resolution omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness path, not a current-main end-user bug reproduction.

## Solution Assessment

No, not as currently written. The cache design is appropriately provider-owned, but cache-key resolution must use the same prepared provider handle as normalization before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: `normalizeProviderToolSchemas` resolves the cache key without `params.runtimeHandle`, but cached parameters are produced through `normalizeProviderToolSchemasWithPlugin` with that handle. A prepared runtime-plan turn can therefore key, hit, or store under one provider hook while normalizing with another; pass the prepared handle into cache-key resolution and cover the mismatch case.
