## Summary

The PR should stay open because the latest head still resolves the cache key through a fresh provider lookup while normalization uses the prepared runtime handle, so prepared runtime-plan turns can cache under a different provider/plugin than the one that produced the normalized parameters.

## What This Changes

The PR adds a provider-owned cache for normalized tool schema parameters, a `ProviderPlugin.resolveToolSchemaCacheKey` hook, tests, docs/API baseline updates, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` kill switch.

## Best Possible Solution

Keep the provider-owned cache design, but resolve the cache key through the same prepared provider runtime handle that owns normalization and add regression coverage for mismatched prepared-vs-resolved providers.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows cache-key resolution omits the prepared runtime handle while the normalization call uses it. This is a PR-introduced correctness path, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache is conservative in what it stores, but cache ownership and normalization ownership must come from the same `ProviderRuntimePluginHandle`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache key is still resolved by re-looking up the provider plugin, while normalized parameters are produced with `params.runtimeHandle` later in the same function. Prepared runtime-plan turns can therefore key or bypass the cache using one plugin while storing parameters produced by another; pass the prepared handle into cache-key resolution and cover the mismatch case.
