## Summary

Keep this PR open; it still has a P2 correctness issue because cache-key ownership can diverge from the prepared provider runtime handle, so a cache entry can be keyed by one provider hook and populated from another. The provider-owned cache direction and supplied terminal proof look useful, but this blocker should be repaired before merge.

## What This Changes

The branch adds provider-owned caching for normalized tool schema parameters, a new `resolveToolSchemaCacheKey` SDK hook, an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch, docs, tests, a changelog entry, and API baseline updates.

## Best Possible Solution

Keep the provider-owned cache design, but make cache-key resolution use the same prepared `ProviderRuntimePluginHandle` as normalization and add regression coverage for mismatched prepared-handle cases.

## Reproduction Assessment

Yes for the review finding: PR-head source shows cache-key resolution omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness path, not a current-main end-user bug reproduction.

## Solution Assessment

No, not as currently written. The cache should stay provider-owned, but the key resolver must share the prepared provider handle with the normalizer before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache key is resolved by re-looking up the provider plugin and does not receive `params.runtimeHandle`, while cached parameters are produced through `params.runtimeHandle`. A prepared runtime-plan turn can therefore key/hit/store under one provider hook while normalizing with another; pass the prepared handle into cache-key resolution and cover the mismatch case.

## Repair Work Prompt

Address https://github.com/openclaw/openclaw/pull/78664 with a narrow repair. Thread `runtimeHandle` from `normalizeProviderToolSchemas` into `resolveProviderToolSchemaNormalizeCacheKey` so cache-key ownership and `normalizeProviderToolSchemasWithPlugin` ownership come from the same `ProviderRuntimePluginHandle`; use `ensureProviderRuntimePluginHandle` or equivalent in the resolver instead of re-resolving independently. Add regression coverage where normal provider lookup would find one provider hook but a prepared runtime handle points to another, and assert the cache cannot key/hit/store under the wrong hook.
