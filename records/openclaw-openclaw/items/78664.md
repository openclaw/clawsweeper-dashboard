## Summary

Keep open: the performance goal is plausible and the PR includes benchmark proof, but the patch can reuse cached schemas across distinct provider hook/config/workspace contexts because the process-global key omits provider-runtime identity.

## What This Changes

The PR adds a provider tool-schema parameter cache, test-only cache controls and coverage, an `OPENCLAW_TOOL_SCHEMA_CACHE=0` kill switch, and a changelog entry.

## Best Possible Solution

Keep the optimization, but make cache eligibility and keys provider-runtime aware so entries cannot cross distinct hook implementations, then rebase and rerun the touched core/runtime gates.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR cache is process-global and its key omits provider-runtime inputs even though hook resolution uses config, workspace, env, and plugin state. I did not run tests because this review is read-only.

## Solution Assessment

No. Caching provider-normalized schemas is reasonable, but the maintainable path is to scope eligibility/keying to the resolved provider hook or provider-runtime control-plane identity instead of hard-coding provider families in the embedded runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Include provider hook identity in the cache key:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:159-173`
  - body: The cache is process-global, but this key only includes provider/model/schema data. `normalizeProviderToolSchemasWithPlugin` resolves the actual normalization hook from config, workspace, env, active registry, and plugin settings, so two contexts with different provider hooks can share this key and the second one will return cached parameters without running its hook. Please include the provider-runtime/control-plane identity in the key, or move cache eligibility/keying into the provider-owned hook.
  - confidence: 0.88
