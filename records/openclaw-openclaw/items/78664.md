## Summary

Keep this PR open: the cache direction is focused and has terminal proof, but the latest diff still resolves the cache owner separately from the prepared provider runtime handle that owns normalization, so a prepared-runtime turn can cache under the wrong hook owner.

## What This Changes

This PR adds provider-owned caching for normalized tool schema parameters, exposes `resolveToolSchemaCacheKey`, documents the hook, updates tests and SDK baselines, and adds `OPENCLAW_TOOL_SCHEMA_CACHE=0` as a rollback switch.

## Best Possible Solution

Thread the prepared `ProviderRuntimePluginHandle` through cache-key resolution, add a mismatch regression test, and then land the conservative parameters-only cache if maintainer API review accepts the new hook.

## Reproduction Assessment

Yes for the PR-introduced review finding: source inspection at head `21dc618c56696d204b97282e7f3bc5be50d3aa10` shows cache-key resolution does not receive `params.runtimeHandle` while normalization does. This is not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The parameters-only cache is a reasonable narrow design, but cache-key ownership and normalization ownership must come from the same prepared provider handle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache key is resolved by looking up the provider plugin without `params.runtimeHandle`, but the normalized parameters are produced with that prepared handle. Prepared runtime-plan turns can therefore key or bypass the cache using one plugin while storing parameters produced by another; pass the same prepared handle into cache-key resolution and cover the mismatch case.
