## Summary

Latest PR head still has a blocking correctness gap: cache-key resolution can use a broadly re-resolved provider plugin while normalization uses the prepared provider handle. The supplied terminal proof is adequate for the intended performance behavior, and the security pass found no supply-chain concern.

## What This Changes

The PR adds a hook-owned LRU cache for provider-normalized tool schema parameters in embedded agent turns, exports and documents an SDK cache-key hook, adds regression tests, and updates the API baseline and changelog.

## Best Possible Solution

Use the same prepared ProviderRuntimePluginHandle for cache-key resolution and schema normalization, add a prepared-only or mismatched-provider regression test, then continue normal maintainer review of the SDK hook.

## Reproduction Assessment

Yes for the review finding: source inspection at latest PR head shows the cache-key path omits the prepared provider handle while the normalization path uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache direction is maintainable, but the cache key must be resolved through the same prepared provider handle that owns normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache-key path re-resolves the provider plugin without `params.runtimeHandle`, while normalization below uses that prepared handle. In a prepared turn this can key the cache from one provider plugin and then store normalized parameters from another, or bypass caching when the prepared plugin is not discoverable through the broad resolver; please thread the same handle through cache-key resolution and add prepared-only or mismatch coverage.
  - confidence: 0.88
