## Summary

Keep open: the proof and overall cache direction are plausible, but PR head still lets cache-key resolution use a different provider plugin than normalization because the prepared runtime handle is not passed into the cache-key resolver.

## What This Changes

The PR adds a provider-owned cache for normalized embedded-agent tool schema parameters, exposes a plugin SDK cache-key hook and rollback env var, and updates docs, tests, API baseline, and changelog.

## Best Possible Solution

Keep the provider-owned schema-parameter cache design, but make cache-key resolution use the same prepared provider runtime handle as normalization and add mismatch regression coverage.

## Reproduction Assessment

Yes for the review finding: PR-head source shows cache-key resolution omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness path, not a current-main end-user bug reproduction.

## Solution Assessment

No, not as currently written. The cache can remain provider-owned, but the key resolver must share the prepared provider handle with the normalizer before this is the narrow maintainable fix.
