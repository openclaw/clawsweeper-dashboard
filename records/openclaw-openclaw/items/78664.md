## Summary

Keep open: the cache design is useful, but the PR still has a correctness blocker because cache-key ownership can diverge from the prepared provider runtime handle that actually performs normalization.

## What This Changes

The PR adds provider-owned caching for normalized tool schema parameters, a `resolveToolSchemaCacheKey` provider hook, `OPENCLAW_TOOL_SCHEMA_CACHE=0`, tests, docs, API baseline hashes, and a changelog entry.

## Best Possible Solution

Thread the prepared provider runtime handle through cache-key resolution so cache ownership and normalization ownership come from the same plugin, then add regression coverage for a prepared or mismatched-provider case.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows cache-key resolution omits the prepared runtime handle while the normalization call uses it. This is a PR-introduced correctness path, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache design is conservative, but the cache-key resolver must use the same `ProviderRuntimePluginHandle` that owns normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache key is resolved by re-looking up the provider plugin, but the normalized parameters are produced with `params.runtimeHandle`. Prepared runtime-plan turns can therefore key or bypass the cache using one plugin while storing parameters produced by another; pass the same prepared handle into cache-key resolution and cover the mismatch case.
  - confidence: 0.9
