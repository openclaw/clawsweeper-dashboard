## Summary

Keep this PR open, but it still needs changes before merge. The cache direction is useful and tied to the open repeated tool-schema normalization report, but latest head still resolves the cache key without the prepared provider runtime handle that the actual normalization call uses.

## What This Changes

The PR adds a provider-owned cache for normalized tool schema parameters during embedded agent turns, exposes a plugin SDK cache-key hook, updates docs/API baseline/changelog, and adds cache behavior tests.

## Best Possible Solution

Resolve cache keys through the same provider runtime handle used for normalization, keep cached values limited to cloned schema parameters, and leave API-surface approval to maintainers.

## Reproduction Assessment

Yes for the review finding: latest PR source shows cache-key resolution omits the prepared provider handle while the normalization call uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The cache design is maintainable, but the cache-key resolver must use the same ProviderRuntimePluginHandle that owns normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache-key path calls `resolveProviderToolSchemaNormalizeCacheKey` without `params.runtimeHandle`, but the normalization below passes `runtimeHandle` into `normalizeProviderToolSchemasWithPlugin`. In prepared turns this can key or bypass the cache using one provider plugin and then store normalized parameters from another; thread the prepared handle into cache-key resolution and cover a prepared-only or mismatched-provider case.
  - confidence: 0.89
