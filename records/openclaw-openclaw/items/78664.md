## Summary

Keep open: the current PR head still resolves the provider tool-schema cache key without the prepared runtime handle, while normalization uses that handle, so the previous P2 correctness issue remains.

## What This Changes

The branch adds provider-owned caching for normalized tool schema parameters, a provider SDK cache-key hook, tests, docs/API baseline updates, a changelog entry, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Keep the provider-owned cache design, but make cache-key resolution use the same prepared `ProviderRuntimePluginHandle` as normalization and add regression coverage for mismatched prepared-handle cases.

## Reproduction Assessment

Yes at source level for the review finding: PR-head source shows cache-key resolution omits `runtimeHandle` while normalization uses it. I did not run a live repro because this is a PR-introduced correctness path, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The provider-owned cache is a maintainable direction, but the key resolver must share the prepared provider handle with the normalizer before the patch is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the prepared provider handle for cache keys:** `src/agents/pi-embedded-runner/tool-schema-runtime.ts:196-202`
  - body: The cache key is resolved without `params.runtimeHandle`, but cached parameters are produced through `normalizeProviderToolSchemasWithPlugin` with `params.runtimeHandle`. Runtime plans can carry a prepared provider handle that differs from a fresh registry lookup, so this can store or hit parameters under the wrong provider hook; pass the prepared handle into cache-key resolution and cover the mismatch case.
  - confidence: 0.91
