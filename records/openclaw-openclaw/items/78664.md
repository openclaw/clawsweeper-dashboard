## Summary

Keep open: the latest head still resolves the cache key through a provider re-lookup path while normalization uses the prepared provider handle, so prepared turns can cache or bypass against a different provider than the one doing normalization.

## What This Changes

The PR adds a provider-owned cache for normalized embedded-agent tool schema parameters, a `resolveToolSchemaCacheKey` provider SDK hook, tests, docs, API baseline, changelog, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` kill switch.

## Best Possible Solution

Use the same prepared `ProviderRuntimePluginHandle` for cache-key resolution and normalization, with regression coverage for prepared-only or mismatched-provider handles; then continue SDK/API review.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows the cache-key path omits the prepared provider handle while the normalization path uses it. This is a PR-introduced correctness issue, not a current-main end-user bug.

## Solution Assessment

No, not as currently written. The provider-owned, parameter-only cache is a maintainable direction, but cache-key ownership must come from the same prepared provider handle as normalization.

## Review Findings
