## Summary

Keep open: the provider-owned cache is a useful performance direction, but current PR head still keys the cache through a fresh provider lookup while normalization uses the prepared runtime handle. That can mix cache ownership in prepared runtime-plan turns.

## What This Changes

The PR adds a provider-owned normalized tool-schema parameter cache, an optional provider SDK cache-key hook, tests, docs/API baseline updates, a changelog entry, and an `OPENCLAW_TOOL_SCHEMA_CACHE=0` rollback switch.

## Best Possible Solution

Keep the hook-owned schema-parameter cache design, but align cache-key ownership with the prepared provider runtime handle and leave broader gateway TTFT work under https://github.com/openclaw/openclaw/issues/80131.

## Reproduction Assessment

Yes for the review finding: PR-head source shows cache-key resolution omits `runtimeHandle` while normalization uses it. This is a PR-introduced correctness path, not a current-main end-user bug reproduction.

## Solution Assessment

No, not as currently written. The cache should remain provider-owned, but the resolver needs to share the prepared provider handle with the normalizer before this is the narrow maintainable fix.
