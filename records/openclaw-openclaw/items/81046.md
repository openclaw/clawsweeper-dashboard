## Summary

Keep open: current main does not already implement this feature, and the PR still needs changes because it fails the session-entry slot type contract, has a remaining session-agent identity gap for some session fallback callers, lacks real behavior proof, and overlaps with another open same-author implementation.

## What This Changes

The PR adds session-store-backed model exhaustion cooldowns, a new `agents.defaults.modelExhaustionRetryMinutes` config field, `SessionEntry.exhaustedModels`, reset clearing, schema help/labels, and focused tests.

## Best Possible Solution

Keep one canonical session-cooldown implementation that updates the session entry slot contract, threads explicit session agent identity through all session fallback callers, preserves custom agent directories, clears cooldowns on reset, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes for the review findings: PR CI logs show the `SessionEntry` slot type failure, and source inspection shows session fallback callers with explicit `agentId` still route the new cooldown resolver through `agentDir` inference. I did not run tests because this review is read-only.

## Solution Assessment

No. Session-scoped cooldown persistence is plausible, but this branch needs the slot contract update, explicit session-agent plumbing across callers, reconciliation with the overlapping PR, and real behavior proof before merge.

  - body: Adding `exhaustedModels` to `SessionEntry` trips the existing exhaustive reserved-slot assertion: CI reports `Type '"exhaustedModels"' does not satisfy the constraint 'never'` in `src/plugins/session-entry-slot-keys.ts`, so production/test type checks cannot pass until this key is reserved or the contract is updated.
  - confidence: 0.96

  - body: The new resolver falls back to deriving the store agent id from `agentDir` when `sessionAgentId` is missing. Auto-reply runs already carry `run.agentId` but only pass `agentDir` through fallback options, so configured arbitrary agent directories can send cooldown reads/writes to the wrong session store. Pass explicit session agent identity from those callers instead of inferring it from the filesystem path.
  - confidence: 0.86
