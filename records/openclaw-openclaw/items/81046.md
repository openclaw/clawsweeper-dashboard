## Summary

Keep open: the latest branch fixes the earlier schema and locked-write concerns, but the session cooldown path is still disconnected from production fallback callers/session identity and lacks required real behavior proof.

## What This Changes

The PR adds session-store backed exhausted-model cooldowns to `runWithModelFallback`, a new `agents.defaults.modelExhaustionRetryMinutes` config field/schema entry, `SessionEntry.exhaustedModels`, and focused Vitest coverage.

## Best Possible Solution

Choose one canonical session-cooldown implementation that threads `sessionId` plus the real `sessionAgentId` or `sessionKey` through fallback, clears cooldown maps on session rollover, documents the config surface, keeps focused regression coverage, and includes redacted live proof.

## Reproduction Assessment

Yes. Source inspection is enough: the production `agent-command` fallback call does not pass `sessionId`, and the PR derives a session-store agent id from `path.basename(agentDir)` even though resolved agent dirs end in `/agent`.

## Solution Assessment

No. Session-scoped cooldowns are plausible, but this branch should not merge until it uses the real production session identity, handles reset rollover, aligns config help, reconciles the related PR, and supplies real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Thread real session identity into cooldown persistence:** `src/agents/model-fallback.ts:939-943`
  - body: The new session cooldown resolver can only work when `sessionId` is present and the correct session-store agent id is used. The main `agent-command` fallback caller does not pass `sessionId`, and this resolver derives `agentId` from `path.basename(agentDir)`, which is `agent` for normal resolved dirs like `.../agents/main/agent`. Real agent runs will therefore skip the new persistence path or look in the wrong store; pass `sessionId` plus the real `sessionAgentId`/`sessionKey` from callers instead.
  - confidence: 0.94
- **[P2] Clear cooldowns when the session rolls over:** `src/agents/model-fallback.ts:245`
  - body: The PR body says a reset clears the marker, but existing session reset/update flow reuses the same store entry and spreads old fields while replacing `sessionId`. Once `exhaustedModels` exists, a `/reset` or idle rollover can carry active cooldowns into the fresh session and skip models there too; drop this map when the stored session id changes or bind it to the session generation.
  - confidence: 0.86
- **[P3] Document the cooldown config key in schema hints:** `src/config/zod-schema.agent-defaults.ts:58`
  - body: The strict schema now accepts `agents.defaults.modelExhaustionRetryMinutes`, but the public schema help/label tables are not updated, so generated config hints will expose an undocumented knob. Add the matching `FIELD_HELP` and `FIELD_LABELS` entries with the default and scope.
  - confidence: 0.78
