## Summary

Keep open: current main still lacks launch-time forwarding of an explicit sub-agent provider/model into the child gateway `agent` call, and an open closing PR already owns the candidate fix. The issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue asks for `sessions_spawn` with an explicit model to make the spawned child actually run on that requested provider/model instead of silently using the default DeepSeek route.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95436, or an equivalent narrow fix that forwards only explicit `sessions_spawn.model` provider/model into the child gateway `agent` launch and authorizes that override at the gateway boundary.

## Reproduction Assessment

Yes for source reproduction: current main resolves and persists the requested model, but the child gateway `agent` launch does not carry the explicit provider/model fields that the gateway override path consumes. I did not rerun the reporter's live QQ/Qwen/DeepSeek environment.

## Solution Assessment

Yes, the narrowest maintainable fix appears to be forwarding the resolved explicit provider/model at the existing sub-agent gateway launch boundary and authorizing it there. A new config option or fallback policy would be broader than the reported bug needs.
