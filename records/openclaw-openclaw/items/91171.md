## What This Changes

The issue reports that explicit `sessions_spawn.model` is resolved in the spawn response but not used by the actual child sub-agent run, causing fallback to the default provider/model.

## Best Possible Solution

Land one focused fix that forwards the resolved provider/model through the authorized child gateway `agent` launch while preserving current session override persistence and default-spawn behavior.

## Reproduction Assessment

Yes, source-level: current main accepts and resolves `sessions_spawn.model`, but the child gateway `agent` launch still omits provider/model while the gateway only honors authorized request-level overrides. I did not run the live QQ/Qwen/DeepSeek scenario.

## Solution Assessment

Yes, the best fix is a narrow agents/gateway repair that forwards the resolved model with the existing authorization seam. Adding a new config option or changing global fallback policy would be the wrong layer.
