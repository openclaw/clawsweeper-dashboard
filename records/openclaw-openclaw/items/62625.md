## Summary

Keep open. Current main and the latest stable release still have the reported memory-core session reconciliation gap: clean `session-start` and `watch` syncs skip session files unless a full reindex, forced or targeted sync, or in-process dirty event is already present, so the user-visible `memory_search` miss remains unresolved. The merged #39732 fix covers only the full-reindex/config-drift path.

## What This Changes

The issue requests startup or missed-event reconciliation for memory-core session transcript indexing so changed session JSONL files are discovered without requiring `openclaw memory index --force`.

## Best Possible Solution

Add a bounded memory-core reconciliation path that enumerates `listSessionFilesForAgent()`, compares current transcript files against stored `sessions` source state, marks changed or missing files dirty or syncs them directly, and makes disk-vs-index drift visible enough that `Dirty: no` is not mistaken for verified freshness.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: after restart or a missed transcript update, `sessionsDirty=false` and `sessionsDirtyFiles` is empty; `warmSession()` triggers `reason: "session-start"`, the gate returns false, and `syncSessionFiles()` is skipped. The reporter and dwbutler also supplied concrete production observations on 2026.3.28 and 2026.4.12.

## Solution Assessment

Yes. A bounded reconciliation scan in the memory-core owner module is the narrowest maintainable fix; simply removing the `session-start`/`watch` exclusion would be riskier because it could rescan all session transcripts on hot paths.
