## Summary

Keep open: this PR is aimed at the right Codex app-server sandbox-policy problem, but the patch currently introduces a P2 permission-widening regression in `/codex permissions default`, lacks after-fix real behavior proof, and carries the protected `maintainer` label.

## What This Changes

The PR adds Codex app-server requirements.toml sandbox-policy parsing and permission provenance across runtime option resolution, thread bindings, conversation controls, bound turns, diagnostics, tests, and the changelog.

## Best Possible Solution

Keep the workspace-policy support, but treat requirements files without `allowed_sandbox_modes` as no sandbox restriction for `/codex permissions default` or preserve the existing workspace-write default, add regression coverage, then provide redacted live/terminal proof for restricted and unrestricted Codex app-server paths.

## Reproduction Assessment

Yes, by source inspection. On the PR head, a requirements file without `allowed_sandbox_modes` still creates a policy object, and `/codex permissions default` then resolves that object to `danger-full-access` instead of current main's `workspace-write` default.

## Solution Assessment

No. The overall direction is maintainable, but the proposed implementation should not merge until the no-allowlist requirements path preserves restricted defaults and the contributor adds real behavior proof.
