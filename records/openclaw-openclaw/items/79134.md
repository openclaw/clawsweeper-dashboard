## Summary

Keep open: the latest PR head fixes the earlier no-allowlist default-permission regression and now includes acceptable copied terminal proof, but two P2 permission-policy issues remain in the new requirements parser/path handling and the PR has the protected `maintainer` label.

## What This Changes

The PR adds Codex app-server requirements.toml sandbox-policy parsing, permission provenance in thread bindings, policy-aware `/codex` permission/runtime behavior, tests, and a changelog entry.

## Best Possible Solution

Keep the requirements-policy support, but make OpenClaw read the same platform-specific requirements file and TOML semantics as Codex, then add regression coverage for Windows path selection and commented allowlist entries.

## Reproduction Assessment

Yes, by source inspection. The PR parser tokenizes raw TOML text after `allowed_sandbox_modes = [` and the PR default path is Unix-only, while Codex's own loader uses real TOML parsing and a Windows ProgramData requirements path.

## Solution Assessment

No. The direction is maintainable, but the current implementation should match Codex's real loader contract instead of hand-parsing comments and hard-coding the Unix system requirements path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Ignore TOML comments in sandbox allowlists:** `extensions/codex/src/app-server/requirements-policy.ts:78-80`
  - body: This parser scans the raw array text for quoted or bare tokens, so a normal TOML comment such as `# "DangerFullAccess"` is still recorded as allowed. Codex parses this file as TOML, so OpenClaw can think full access is allowed, send it, and then have the app-server reject the turn instead of falling back to workspace-write.
  - confidence: 0.94
- **[P2] Use Codex's Windows requirements path:** `extensions/codex/src/app-server/requirements-policy.ts:3`
  - body: The new default only probes `/etc/codex/requirements.toml`, but Codex uses `%ProgramData%\OpenAI\Codex\requirements.toml` on Windows. Managed Windows hosts with an `allowed_sandbox_modes` policy will be missed and OpenClaw will keep sending full access unless users discover and set the OpenClaw-specific override.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff changes sandbox permission policy and still has two medium-risk policy-detection gaps.

Concerns:

- **[medium] Commented allowlist can widen perceived sandbox policy:** `extensions/codex/src/app-server/requirements-policy.ts:78`
  - body: The raw token scanner treats sandbox modes inside TOML comments as active allowlist entries, which can bypass the fallback that should prevent full-access requests on restricted hosts.
  - confidence: 0.94
- **[medium] Windows policy file is not read by default:** `extensions/codex/src/app-server/requirements-policy.ts:3`
  - body: The default path misses Codex's Windows ProgramData requirements file, so managed Windows sandbox restrictions are not applied unless a user sets the OpenClaw override.
  - confidence: 0.92
