## Summary

Keep open: the feature is useful, but the PR still misreports some installed channel plugins as installable catalog rows and lacks the required docs/changelog for a new user-facing CLI contract.

## What This Changes

The PR adds a `configure surface` subcommand, setup-surface JSON exporter/types, provider auth surface metadata, generated SDK baselines, and focused tests for external provider/channel setup UIs.

## Best Possible Solution

Keep the PR open, fix installed-state serialization for catalog-derived channels, add the required user docs/changelog, then have a maintainer validate the stable setup-surface contract before merge.

## Reproduction Assessment

Yes. From source, the PR treats every catalog entry outside `registry.channelSetups` as uninstalled even though current catalog discovery can include installed manifest entries that did not register a setup surface.

## Solution Assessment

No. The exporter should derive installed/disabled/load-error state from registry/plugin/catalog provenance instead of hardcoding catalog-only rows as installable, and the new public CLI contract needs docs and changelog before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve installed catalog channels:** `src/setup/surface-export.ts:267-268`
  - body: `listChannelPluginCatalogEntries()` includes manifest-discovered installed plugins, not only remote install suggestions. Since this branch hardcodes those rows as `installed: false` and `installable: true`, an installed-but-disabled or load-failed channel that is absent from `registry.channelSetups` is exported as something to install, and it is dropped entirely under `--installed-only`. Derive installed state from the catalog/registry provenance before serializing the fallback row.
  - confidence: 0.86
- **[P2] Document the new stable CLI contract:** `src/cli/program/register.configure.ts:83-86`
  - body: This registers a new user-facing `openclaw configure surface` command and stable JSON contract, but the PR does not update `CHANGELOG.md` or the configure CLI docs. Repo policy requires user-facing feature changes to be documented/changeloged, and consumers need the version, flags, and schema expectations before depending on this as a stable surface.
  - confidence: 0.8
