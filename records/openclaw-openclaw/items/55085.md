## Summary

Keep open. Current main still lacks the proposed static setup-surface export, but this PR is not merge-ready because the export activates plugin loading, section parsing is still raw-argv based, and the branch is stale against current generated-baseline and public-doc policy.

## What This Changes

The PR adds a `configure surface` CLI subcommand, setup-surface descriptor/export types, provider auth surface metadata, command descriptor updates, generated API artifacts, and targeted tests.

## Best Possible Solution

Settle a documented, versioned setup-discovery contract that uses non-activating manifest/setup-entry metadata, then ship it with current Plugin SDK baseline hashes, CLI docs, changelog, and focused plugin-boundary tests.

## Reproduction Assessment

Yes. Source inspection is sufficient for the PR findings: current main has no static `configure surface` command, while the PR head uses `activate: true` for the exporter and raw `program.rawArgs` scanning for section options.

## Solution Assessment

No. The direction may be useful, but the implementation should avoid active plugin loading, use Commander’s parsed option values, and align with current generated-baseline/docs/changelog requirements before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep surface export on a non-activating plugin load:** `src/setup/surface-export.ts:297`
  - body: `configure surface` is advertised as a read-only metadata export, but this call takes the active plugin-load path. Current loader behavior can clear/restore global plugin state, run plugin registration, emit active-load warnings, and activate the loaded registry, so a metadata command can trigger side effects. Use a non-activating manifest/setup-entry snapshot path instead.
  - confidence: 0.86
- **[P2] Use parsed options for surface sections:** `src/cli/program/register.configure.ts:95-99`
  - body: The action ignores Commander’s parsed `opts.section` and reconstructs values from raw argv. Valid invocations with earlier option values like `configure` can anchor the scan to the wrong token and silently export the default sections instead of the requested subset. Pass `opts.section` through directly and test the global-option collision.
  - confidence: 0.78
- **[P2] Update the tracked Plugin SDK API baseline:** `src/plugins/types.ts:253-266`
  - body: Adding `ProviderAuthMethod.surface` changes a public Plugin SDK type, but the PR updates old full JSON/JSONL baseline artifacts rather than the currently tracked `docs/.generated/plugin-sdk-api-baseline.sha256`. On current main those full baselines are gitignored inspection files, so the API check will still report drift until the tracked hash is regenerated after rebase.
  - confidence: 0.82
- **[P3] Document and release-note configure surface:** `src/cli/program/register.configure.ts:84-93`
  - body: This adds a public CLI command and machine-consumable setup contract, but the PR does not update the configure CLI docs or changelog. Downstream UI authors need a stable reference for the command, flags, and JSON contract in the same change that introduces it.
  - confidence: 0.8
