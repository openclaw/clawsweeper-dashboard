## Summary

Keep open. Current main has adjacent typed Gateway wizard-session support, but it does not implement this PR's static `configure surface` file export, and the proposed public CLI/Plugin SDK contract still needs maintainer/API revision because the exporter activates plugins while advertised as read-only.

## What This Changes

The PR adds a `configure surface` CLI subcommand, setup-surface descriptor/export types, provider auth surface metadata, command descriptor and generated SDK baseline updates, and targeted tests.

## Best Possible Solution

Settle the setup discovery contract, then implement it as a documented, versioned, non-activating export that reuses manifest/setup-entry metadata where possible and includes schema, CLI docs, changelog, Plugin SDK/API baseline checks, and plugin-boundary tests.

## Reproduction Assessment

Yes. For the PR findings, source inspection is sufficient: the PR head passes `activate: true` in `exportSetupSurface`, and current loader code shows activation clears/activates global plugin state and can stage bundled runtime dependencies; current-main absence was verified by focused source search.

## Solution Assessment

No. The requested static discovery seam may be useful, but this implementation is not the best current solution because it mixes read-only metadata export with active plugin loading and lacks the public docs/changelog expected for a new CLI and Plugin SDK contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep surface export on a non-activating plugin load:** `src/setup/surface-export.ts:297`
  - body: `configure surface` is advertised as a read-only metadata export, but this call uses the active plugin-load path. In current loader semantics, activation can clear/restore process-global plugin state, activate registries, and stage bundled runtime dependencies, so external setup UIs can trigger side effects while only asking for JSON metadata. Use an existing non-activating metadata/setup-entry snapshot path instead.
  - confidence: 0.84
- **[P2] Use parsed options for surface sections:** `src/cli/program/register.configure.ts:95-99`
  - body: The action ignores Commander’s parsed `opts.section` and reconstructs section values from `program.rawArgs`. That makes a machine-consumed command depend on brittle token scanning even though Commander already parsed the repeatable option; pass `opts.section` through directly and cover the collision case in the registration test.
  - confidence: 0.74
- **[P3] Document the public configure surface command:** `src/cli/program/register.configure.ts:84-93`
  - body: This adds a new public CLI and setup contract, but the PR does not update the configure docs or changelog. Users and downstream UI authors will not have a stable reference for the new command, flags, or JSON contract unless this ships with docs and a release note.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a read-only CLI export but builds it through active plugin loading, which is a concrete security and supply-chain concern for a metadata command.

Concerns:

- **[medium] Read-only export activates plugin code:** `src/setup/surface-export.ts:297`
  - body: `exportSetupSurface` passes `activate: true` to `loadOpenClawPlugins`. Current loader semantics allow active loads to mutate process-global plugin registration state and stage bundled runtime dependencies, so running a metadata export can execute plugin code and dependency setup beyond the stated read-only purpose.
