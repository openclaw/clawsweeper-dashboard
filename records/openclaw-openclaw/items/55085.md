## Summary

Keep open: current main does not provide the exact `configure surface` export, but the PR still needs maintainer API/product direction, real behavior proof, and fixes for setup-state semantics before it is mergeable.

## What This Changes

The PR adds `openclaw configure surface --json-out`, setup-surface JSON types/export logic, provider API-key surface metadata, CLI registration, tests, and generated Plugin SDK API baseline updates.

## Best Possible Solution

Decide the canonical public setup-discovery contract; if this CLI export is accepted, derive state from existing setup discovery/auth status, preserve installed/configured/disabled/error semantics, document the v1 schema publicly, and require redacted real command proof before merge.

## Reproduction Assessment

Not applicable; this is a feature/API-surface PR rather than a bug report. Source inspection shows current main has Gateway wizard RPC but no exact `configure surface` exporter.

## Solution Assessment

Unclear; a file-backed discovery export is plausible, but it overlaps with the documented Gateway wizard RPC path and the current schema loses important setup state. The safer path is to settle the public contract first, then preserve status semantics and docs in the implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve installed catalog state in surface exports:** `src/setup/surface-export.ts:267`
  - body: This catalog-only branch passes `installed: false` for every catalog entry not present in `registry.channelSetups`. Current setup discovery has a separate installed-catalog bucket for manifest-discovered plugins that did not load or register setup, so the export can tell UIs to install an already-present disabled or broken plugin instead of showing installed-but-unavailable state.
  - confidence: 0.86
- **[P2] Expose provider configured state:** `src/setup/surface-types.ts:47-61`
  - body: The provider surface schema has auth methods and env vars but no configured/status field, even though the command's advertised contract is to tell UIs whether setup already exists. Consumers cannot distinguish ready providers from unauthenticated ones without reimplementing provider auth detection outside the stable surface.
  - confidence: 0.78
- **[P3] Document the stable setup surface:** `src/cli/program/register.configure.ts:83-93`
  - body: This registers a stable CLI command and v1 JSON contract, but the PR does not add public configure docs or schema docs. External UI authors should not have to infer flags, output shape, and compatibility expectations from source before depending on the surface.
  - confidence: 0.9
