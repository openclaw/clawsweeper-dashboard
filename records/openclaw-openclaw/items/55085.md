## Summary

Keep open: current main still lacks this exact static `configure surface` export, but the PR is a new CLI/API feature with unresolved contract questions, missing real behavior proof, and blocking correctness gaps in the proposed schema and option handling.

## What This Changes

The PR adds a `configure surface` CLI subcommand, setup-surface JSON types/exporter, provider API-key surface metadata, command registration, tests, and generated Plugin SDK API baselines.

## Best Possible Solution

Decide the canonical public setup-discovery contract; if this CLI export is accepted, preserve installed/configured/disabled/error semantics, expose provider configured status, document the v1 schema, and require redacted real command proof before merge.

## Reproduction Assessment

Not applicable; this is a feature/API-surface PR rather than a bug report. Source inspection shows current main has Gateway wizard RPC but no `configure surface` exporter.

## Solution Assessment

Unclear; a file-backed discovery export is plausible, but this implementation overlaps with Gateway wizard RPC and loses important setup state. The safer path is to settle the public contract first, then fix state semantics, docs, and proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Read surface sections from parsed options:** `src/cli/program/register.configure.ts:95-99`
  - body: The action ignores Commander's parsed `opts.section` and reparses `program.rawArgs`. If a global option value or output filename matches `configure` or `surface`, the parser can anchor to the wrong token and silently pass `[]`, causing a requested subset such as `--section channels` to export both sections.
  - confidence: 0.82
- **[P2] Preserve installed catalog state:** `src/setup/surface-export.ts:263-268`
  - body: This catalog branch sets `installed: false` for any catalog entry not in `registry.channelSetups`. Current setup discovery has a separate installed-catalog bucket for manifest-installed plugins whose setup/runtime did not load, so the export can tell UIs to install an already-present disabled or broken plugin.
  - confidence: 0.88
- **[P2] Expose provider configured status:** `src/setup/surface-types.ts:47-61`
  - body: The provider entry type has env vars and methods but no `configured` or status field, although the PR contract says UIs need to know whether auth exists. Consumers would have to reimplement provider auth detection to distinguish ready providers from unauthenticated ones.
  - confidence: 0.8
- **[P3] Document the setup surface contract:** `src/cli/program/register.configure.ts:83-93`
  - body: This registers a stable command and v1 JSON schema but only adds CLI help text. External UI authors need public docs for flags, the file-output contract, status semantics, and schema compatibility before depending on this surface.
  - confidence: 0.9
