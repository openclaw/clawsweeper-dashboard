## Summary

Keep open: the protected `maintainer` label blocks automatic cleanup, and current `main` still routes every non-empty `imsg rpc` stderr line to ERROR with no benign AddressBook filter.

## What This Changes

The issue asks to downgrade or filter known-benign Apple AddressBook stderr emitted by the long-running `imsg rpc` child so it no longer appears as gateway ERROR noise.

## Best Possible Solution

Add a small tested classifier at the iMessage RPC client boundary that routes only documented benign Apple framework chatter to a non-error log path while leaving malformed JSON, RPC errors, process exits, and unknown stderr visible as errors.

## Reproduction Assessment

Yes from source, but not from a live macOS run: current `main` routes every non-empty `imsg rpc` stderr line through `runtime.error()`, so the reported AddressBook line would be emitted at ERROR if `imsg` writes it. A live proof would require an enabled iMessage channel on macOS and waiting for the Apple AddressBook flush.

## Solution Assessment

Mostly yes: a narrow known-benign stderr downgrade is more maintainable than suppressing all stderr, but the implementation should use the runtime/logging surface that exists today and avoid adding a config option until multiple patterns need operator control.
