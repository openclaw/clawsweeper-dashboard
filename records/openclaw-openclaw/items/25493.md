## Summary

Keep open: current main and v2026.6.6 still hardcode OpenClaw-owned Safety and Self-Update guidance, while supported customization provides whole-prompt replacement, hooks, or three narrow provider-owned section overrides. Both direct implementation PRs closed without merge, and the remaining design requires maintainer product/security judgment rather than conservative cleanup.

## What This Changes

Allow users to edit or replace OpenClaw-owned system-prompt sections, potentially through workspace templates such as SAFETY.md.

## Best Possible Solution

Define one canonical, documented section-override contract with a bounded allowlist, deterministic precedence, secure defaults, and explicit operator opt-in; otherwise document whole-prompt replacement and hooks as the supported customization boundary.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken established behavior. Current source, tests, and v2026.6.6 provide high-confidence proof that section-level editing is not presently supported.

## Solution Assessment

Unclear: the requested flexibility is reasonable, but neither prior workspace-file nor config proposal established a maintainer-approved ownership, precedence, and security model. A single bounded contract is safer than adding another parallel prompt mutation path.
