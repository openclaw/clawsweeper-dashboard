## Summary

Keep open. The Telegram dedupe refactor is coherent, but the patch should not merge as-is because it exposes a raw sync plugin-state factory on a public SDK doctor subpath, bypassing the existing trusted-plugin and runtime-bound plugin-id state boundary.

## What This Changes

This PR replaces Telegram's local dispatch replay dedupe buckets with the SDK claimable persistent dedupe helper and adds doctor migration support for legacy JSON and bucketed plugin-state entries.

## Best Possible Solution

Keep plugin-state access runtime-bound and implement Telegram's legacy bucket import through a narrow core/doctor migration helper or explicitly approved scoped SDK API, then rerun focused Telegram, state-migration, and plugin SDK API checks.

## Reproduction Assessment

Yes, from source inspection: the PR head exports the raw sync plugin-state store factory on a public SDK subpath, while current main only exposes plugin state through a runtime proxy that enforces trust and binds the caller plugin id.

## Solution Assessment

No. Reusing SDK claimable dedupe for Telegram is the right direction, but the current solution expands the SDK state boundary too broadly; a scoped doctor migration helper or owner-approved API would be safer.
