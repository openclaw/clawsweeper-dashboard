## Summary

This PR remains necessary because current main still lacks a pruning lifecycle event, but the branch is not merge-ready: its normal extension path cannot obtain the session identity it requires, and awaiting arbitrary internal hook handlers can stall context construction. The MEMBER author association and protected `maintainer` label also require explicit maintainer handling rather than cleanup closure.

## What This Changes

Adds a `session:prune` internal lifecycle event with pruning counts, tool names, session metadata, documentation, and tests while converting context pruning to an asynchronous path.

## Best Possible Solution

Implement the event on the current `src/agents/agent-hooks/context-pruning` path using a documented canonical session identifier, a typed minimal payload, and delivery semantics that cannot block context preparation, with focused extension-to-hook and non-settling-handler coverage.

## Solution Assessment

No. A pruning event is a reasonable solution, but it should be implemented at the current agent-hook owner boundary with supported session identity and without allowing optional observers to block context preparation.
