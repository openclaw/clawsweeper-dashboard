Full review comments:

- **[P1] Use supported session identity for prune hooks:** `src/agents/pi-extensions/context-pruning/extension.ts:29-30`
  - body: `ExtensionContext` does not expose `sessionKey` or `sessionId`, and the current runner's context factory does not attach those fields. Because `emitPruneHook` returns early without a key, the normal extension path can prune successfully while silently never emitting the new lifecycle event; derive identity from a supported source such as `ctx.sessionManager.getSessionId()` plus the canonical session key contract, and cover the full extension path in a test.
  - confidence: 0.9
- **[P1] Bound prune hook delivery off the hot path:** `src/agents/pi-extensions/context-pruning/pruner.ts:292`
  - body: This awaits `triggerInternalHook` from context pruning, but internal hook handlers are awaited sequentially without timeout. A slow or never-settling hook handler can block context construction before the model call, so use a bounded/non-blocking hook helper or an explicit timeout instead of coupling pruning completion to arbitrary hook work.
  - confidence: 0.87
- **[P2] Sanitize the emitted tool names:** `src/agents/pi-extensions/context-pruning/pruner.ts:289`
  - body: The new payload forwards raw `toolName` strings into hook context. Tool names can come from stored tool-result and provider/replay data, and hook consumers commonly log or serialize context; normalize, cap, and document these fields as untrusted before exposing them as a hook contract.
  - confidence: 0.82
- **[P3] Document the prune payload contract:** `docs/automation/hooks.md:235`
  - body: The docs add the event name but omit the context fields that make the event useful. Please document the supported payload shape, trust expectations, and whether fields such as `toolNames` are sanitized or optional so hook authors know what they can rely on.
  - confidence: 0.78
