Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Wire pruning hooks to a real session identity:** `src/agents/pi-extensions/context-pruning/extension.ts:29-30`
  - body: The extension passes `sessionKey` and `sessionId` by casting arbitrary extras from `ctx`, while `emitPruneHook` returns early when both are absent. If the normal host context does not carry those undocumented fields, pruning still happens but `session:prune` never fires; derive the identity from a documented runner/session contract and add an extension-level test that proves the real path emits.
  - confidence: 0.84
- **[P1] Do not let hook handlers hang context pruning:** `src/agents/pi-extensions/context-pruning/pruner.ts:292-294`
  - body: This awaits `triggerInternalHook`, whose current implementation awaits handlers sequentially without an overall timeout. A slow or never-settling hook can now stall context building for any request that prunes; make prune hook delivery bounded or intentionally fire-and-forget with clear ordering semantics.
  - confidence: 0.88
- **[P2] Sanitize tool names before exposing them to hooks:** `src/agents/pi-extensions/context-pruning/pruner.ts:331-333`
  - body: The hook payload forwards `msg.toolName` values directly into `context.toolNames`. Tool names can come from persisted transcript/tool-call repair paths and are later likely to be logged or indexed by hook handlers, so normalize, cap, and restrict the values before exposing them.
  - confidence: 0.8
- **[P3] Document the prune payload shape:** `docs/automation/hooks.md:235`
  - body: The docs add the event name but not the fields hook authors can rely on. Since the payload is the feature's API, document `softTrimmedCount`, `hardClearedCount`, `toolNames`, `sessionId`, and `prunedAt` or the final approved shape.
  - confidence: 0.92

## Security Review
