Adding `session:prune` only to the old module will not cover the live runtime after this branch is rebased, so the feature needs to be reapplied to the current `agent-hooks` pruning files and tests.
  - confidence: 0.96
- **[P1] Source session identity from a typed runtime contract:** `src/agents/pi-extensions/context-pruning/extension.ts:29-30`
  - body: The extension reads `sessionKey` and `sessionId` through casts on `ExtensionContext`, but the current context object is built from typed getters and does not expose those fields. If this shape is carried forward, the normal extension path can silently skip every prune hook because `hookSessionKey` stays undefined.
  - confidence: 0.9
- **[P1] Bound prune hook delivery before awaiting handlers:** `src/agents/pi-extensions/context-pruning/pruner.ts:293`
  - body: `triggerInternalHook` awaits each registered handler in order and has no timeout. Awaiting it from context pruning means a slow or hung hook handler can block model-context preparation and stall agent turns, so this needs timeout-bounded or nonblocking delivery with focused coverage.
  - confidence: 0.93
- **[P1] Sanitize or omit tool names before emitting hook metadata:** `src/agents/pi-extensions/context-pruning/pruner.ts:289`
  - body: The new payload forwards raw `toolName` strings from pruned tool-result messages to hook handlers. Since handlers may log, store, or forward this metadata, the hook contract should normalize, cap, and document these names or omit them by default.
  - confidence: 0.84
- **[P2] Document the prune payload contract:** `docs/automation/hooks.md:235`
  - body: The docs add the event name but not the context fields, timing, privacy limits, or whether pushed messages are ignored for this lifecycle event. Hook authors need that contract before the event is published.
  - confidence: 0.88
