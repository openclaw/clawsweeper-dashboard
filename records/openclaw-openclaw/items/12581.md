Full review comments:

- **[P1] Source session identity from a real contract:** `src/agents/pi-extensions/context-pruning/extension.ts:29-30`
  - body: `ExtensionContext` in the PR's `@mariozechner/pi-coding-agent@0.52.9` contract, and current main's replacement context, do not expose `sessionKey` or `sessionId`. These casts therefore produce `undefined` on the normal extension path, and `emitPruneHook` returns early whenever pruning happens without those values, defeating the new hook outside direct pruner tests. Plumb a typed session identity from the runner/runtime boundary and add an integration test through `contextPruningExtension`.
  - confidence: 0.91
- **[P1] Bound prune hook delivery before awaiting it:** `src/agents/pi-extensions/context-pruning/pruner.ts:293`
  - body: `emitPruneHook` now awaits `triggerInternalHook`, and the hook runner awaits each handler sequentially without a timeout. A hook handler that performs slow or hanging I/O can now stall context pruning and block the model-call setup path. Keep pruning best-effort by using a bounded wait or non-blocking delivery with rejection handling.
  - confidence: 0.84
- **[P2] Sanitize or drop tool names before emitting:** `src/agents/pi-extensions/context-pruning/pruner.ts:331-332`
  - body: This forwards raw `ToolResultMessage.toolName` strings into `event.context.toolNames`. Tool names can come from transcript/tool-call metadata and should be treated as untrusted before they reach user-installed hooks that may log or export them. Use counts only, or normalize, cap, and document the field as untrusted metadata.
  - confidence: 0.82
- **[P3] Document the prune hook payload:** `docs/automation/hooks.md:235`
  - body: The new docs entry only says when `session:prune` fires, but the payload shape is the useful contract for hook authors. Add the supported context fields and safety caveats so consumers know what is stable and what must be escaped or ignored.
  - confidence: 0.76
