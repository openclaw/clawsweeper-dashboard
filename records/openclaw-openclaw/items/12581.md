## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire pruning hooks to documented session identity:** `src/agents/pi-extensions/context-pruning/extension.ts:29-30`
  - body: The context handler reads `sessionKey` and `sessionId` via casts, but the `@earendil-works/pi-coding-agent@0.75.4` `ExtensionContext` contract does not provide those fields. Through the normal extension path they can be undefined, making `emitPruneHook` return early on every prune and defeating the event.
  - confidence: 0.9
- **[P1] Do not let hook handlers hang context pruning:** `src/agents/pi-extensions/context-pruning/pruner.ts:293-294`
  - body: This awaits `triggerInternalHook` while pruning context, and the current dispatcher awaits each handler with no timeout. A slow or never-settling hook can block context construction and the reply that triggered pruning, so delivery should be bounded or explicitly non-blocking.
  - confidence: 0.88
- **[P2] Sanitize tool names before exposing them to hooks:** `src/agents/pi-extensions/context-pruning/pruner.ts:331-333`
  - body: The hook payload forwards raw `ToolResultMessage.toolName` values. Those names come from transcript/runtime data and are not normalized for hook/log use here; normalize to a strict short identifier, cap length, or omit them so hooks do not receive unsafe metadata.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention: the diff exposes pruning metadata to hooks and awaits hook handlers from a pruning hot path without a bounded contract.

Concerns:

- **[medium] Normalize pruning hook tool names:** `src/agents/pi-extensions/context-pruning/pruner.ts:331`
  - body: Raw tool names are forwarded into `event.context.toolNames`; hook handlers commonly log or serialize context, so the payload should use a strict normalized and capped identifier or omit tool names by default.
