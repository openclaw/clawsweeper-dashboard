## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass session identity through a real context contract:** `src/agents/pi-extensions/context-pruning/extension.ts:29-30`
  - body: `ExtensionContext` does not declare `sessionKey` or `sessionId`, and `createContext()` does not define them, so these casts produce `undefined` in the normal context-pruning path. Because `emitPruneHook` now returns early without a hook session key, the new lifecycle event can silently never fire in production; add an owned session identity field/source and cover the extension-to-pruner path.
  - confidence: 0.9
- **[P1] Keep pruning from waiting forever on hooks:** `src/agents/pi-extensions/context-pruning/pruner.ts:291`
  - body: `triggerInternalHook` awaits every registered handler sequentially and has no timeout. Awaiting it from context pruning means a slow or hung hook can block model context preparation for any turn that prunes; make this emission fire-and-forget or time-bounded before merging.
  - confidence: 0.86
- **[P2] Sanitize or omit tool names before hook emission:** `src/agents/pi-extensions/context-pruning/pruner.ts:288`
  - body: This exposes `msg.toolName` values directly in `event.context.toolNames`. Default pruning allows all tool names, and repair paths can synthesize tool results from assistant-provided tool-call names, so hook logs/automation can receive unbounded or control-character data; normalize, cap, allowlist, or keep the payload to counts only.
  - confidence: 0.78
- **[P3] Document the prune hook payload contract:** `docs/automation/hooks.md:235`
  - body: The docs add `session:prune` but do not list the context fields handlers can rely on. Since the payload is the feature's main value, document the exact fields and trust/escaping expectations before publishing the hook surface.
  - confidence: 0.82
