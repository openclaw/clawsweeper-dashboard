## Summary

Keep this protected maintainer PR open, but do not merge the current branch: it targets the pre-refactor pruning paths, conflicts with current main, lacks real-runtime proof, and introduces an unbounded hook-handler wait in the agent context-building path.

## What This Changes

Adds a `session:prune` internal lifecycle event with prune counts, tool names, session metadata, and deterministic timing, while converting context pruning and its extension callback to asynchronous execution.

## Best Possible Solution

Implement the lifecycle signal at the current `src/agents/agent-hooks/context-pruning` boundary, preserve the supported synchronous transform contract, carry a typed prune-result summary to a lifecycle owner, and emit through non-blocking or explicitly time-bounded delivery using canonical session identity.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new lifecycle capability. Source inspection does provide high-confidence evidence that current main prunes tool results without emitting the requested dedicated event.

## Solution Assessment

No. Adding a prune event is appropriate, but awaiting arbitrary handlers inside a now-obsolete pruning transform is not the narrowest or safest implementation; the event should be ported to the current owner boundary with bounded delivery.
