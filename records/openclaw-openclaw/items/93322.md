## Label Justifications

- `P2`: This is a normal-priority correctness bug in status reporting with limited blast radius, not a core runtime outage.
- `impact:session-state`: The issue is about status surfaces disagreeing with the session-selected model after a session-level `/model` switch.
- `impact:auth-provider`: The wrong provider usage line can make an OpenAI-selected session display DeepSeek balance/auth usage information.

## Summary

Keep open: current main has a partial mitigation that clears stale runtime model fields when `/model` applies, but the status renderer still derives usage from the active runtime provider when stale runtime fields are present, and two open closing PRs now own the focused fix path.

## What This Changes

`/status` and `session_status` should show usage for the session-selected model/provider after a `/model` switch instead of the configured default or stale runtime provider.

## Best Possible Solution

Review and land one focused fix PR that makes `/status` and `session_status` resolve usage/auth/context from the selected session model, with regression coverage for separate and provider-qualified override shapes.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: pass a selected OpenAI override plus stale DeepSeek runtime fields into the current status path, and usage provider selection follows `modelRefs.active`. I did not run the live Telegram scenario in this read-only review.

## Solution Assessment
