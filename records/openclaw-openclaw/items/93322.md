## Summary

Keep open: current `main` still routes status usage through the active/runtime model path, while two open PRs use closing syntax for this exact bug. The issue should close only after one fix PR lands.

## What This Changes

The issue reports that `/status` and `session_status` show usage for the configured default provider instead of the session-selected provider after `/model` changes the live session model.

## Best Possible Solution

Land the strongest reviewed fix PR for the status renderer, then close this issue with the merge commit and leave the broader session model/auth refactor tracked separately.

## Reproduction Assessment

Yes, at source level: use a DeepSeek configured/default model with an OpenAI `providerOverride`/`modelOverride` and a stale DeepSeek runtime model entry, then current `buildStatusText` derives the usage provider from `modelRefs.active`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes: the maintainable fix is to resolve the selected session model once and use that selected provider/model for status usage/auth/context lookups while keeping fallback/runtime display semantics separate. The broader session resolver refactor can remain a follow-up.
