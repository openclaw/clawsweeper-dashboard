- `impact:auth-provider`: The visible usage/auth summary can be resolved from the wrong provider after a session model switch.

## Summary

Keep open: current main and the latest release still route `/status` usage through the active runtime provider when stale runtime model fields remain, while two open mergeable PRs now target this exact bug.

## What This Changes

The issue asks for `/status` and `session_status` usage/auth metadata to follow the session-selected `/model` provider instead of the configured default or stale runtime provider.

## Best Possible Solution

Review and land one focused fix that resolves the usage/auth/status lookup from the selected session model unless runtime fallback or alias provenance is explicitly authoritative, then close this issue from the merged PR.

## Reproduction Assessment

Yes, at source level. A session entry with `providerOverride: "openai"`/`modelOverride: "gpt-5.5"` plus stale `modelProvider: "deepseek"`/`model` still feeds `modelRefs.active`, and `buildStatusText` uses that active provider for usage lookup.

## Solution Assessment

No, current main is only a partial mitigation. The best fix is to move the status usage/auth lookup boundary to the selected session model while preserving explicit fallback/alias runtime provenance.
