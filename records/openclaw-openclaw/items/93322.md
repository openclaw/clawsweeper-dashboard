## Summary

Keep open: current main and the latest release still route `/status` provider usage through the active runtime provider when stale runtime model fields are present, and two open closing PRs now own the focused fix path.

## What This Changes

The issue asks `/status` and `session_status` usage/auth details to follow the session-selected model after `/model`, not the configured default or stale runtime provider.

## Best Possible Solution

Land one focused status resolver PR that derives usage/auth/context lookup from the selected session model unless a real fallback or runtime alias is authoritative.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a selected OpenAI override plus stale DeepSeek `sessionEntry.modelProvider/model` into `buildStatusText`; current main resolves the usage provider from `modelRefs.active`.

## Solution Assessment

Yes. A focused resolver fix in the status path is the narrowest maintainable fix for this issue; the broader session model/auth state refactor can remain separate.
