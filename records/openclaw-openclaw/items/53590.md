## Summary

Keep open. Current main has partial adjacent fixes, but it still lacks the requested generic startup or doctor warning when configured model IDs become stale after an update; the only catalog/allowlist warning path is scoped to `hooks.gmail.model`, and the open related PR is still only partial doctor coverage.

## What This Changes

Request startup, doctor, and changelog diagnostics when configured model IDs become invalid or renamed after an OpenClaw npm update.

## Best Possible Solution

Add one shared configured-model diagnostics layer used by both Gateway startup and `openclaw doctor`: collect configured refs from defaults, fallbacks, per-agent settings, hooks, channel overrides, tools, and media settings; resolve aliases consistently; warn only when catalog evidence is authoritative; and document or changelog retired shipped model slugs when OpenClaw changes them.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: configure an invalid `agents.defaults.model.primary`, start the Gateway or run doctor, and current main has no generic code path that validates that ref against the catalog outside `hooks.gmail.model`.

## Solution Assessment

No for the current implementation. Runtime `model_not_found` handling and hook-scoped validation are useful, and PR #40355 is useful partial work, but the maintainable fix should centralize configured-ref collection and catalog status checks rather than add another one-off warning path.
