## Summary

Close as implemented: current main already has a memory plugin slot, memory capability registration APIs, and documented local/external memory backend paths; an Awareness-specific integration would fit as a plugin rather than new core architecture.

## What This Changes

The issue requests a pluggable persistent memory backend so agent context can be shared across sessions, machines, and external stores.

## Best Possible Solution

Keep the shipped memory slot and SDK capability model in core, and route service-specific backends such as Awareness through an external memory plugin published via ClawHub.

## Reproduction Assessment

Not applicable: this is a feature and architecture request, not a failing current-main behavior. Verification is source/docs/history based: the memory slot, SDK API, and backend docs are present on current main.

## Solution Assessment

Yes: the shipped plugin slot and memory capability API are the narrow maintainable solution. A specific cloud backend should be implemented as a plugin instead of adding a provider-specific core path.
