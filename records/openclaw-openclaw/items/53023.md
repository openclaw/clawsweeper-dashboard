## Summary

Keep open: current main and the latest release still preserve one active run per session lane, do not expose the requested session-concurrency or auto-yield config keys, and the related PR/issues only cover separate slices of this umbrella.

## What This Changes

Request configurable per-session lane concurrency, automatic yielding during long tool execution, and configurable lane-wait warning thresholds.

## Best Possible Solution

Keep this as the umbrella tracker while maintainers decide same-session concurrency semantics, review or replace the session-concurrency PR, leave warning-threshold work to its narrower tracker, and design auto-yield around explicit session-state safety rules.

## Reproduction Assessment

Yes, at source level. New session lanes still default to one active run, embedded runs still enter `session:<key>`, the requested config keys are absent, and yield still depends on an explicit `sessions_yield` tool call.

## Solution Assessment

Unclear as one implementation. Session concurrency, lane-wait diagnostics, and auto-yield are separable changes, and auto-yield needs maintainer decisions around cancellation, transcript ordering, and session mutation safety.
