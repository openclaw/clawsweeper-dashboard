## Summary

Keep open. The issue requests configurable per-session lane concurrency, auto-yield behavior, and adjustable lane-wait diagnostics so busy same-session/chat workflows are not serialized indefinitely.

## Best Possible Solution

Keep this as the product/design tracker for session concurrency and auto-yield, while letting the narrower lane-wait threshold proceed separately and requiring an explicit maintainer-approved session ordering and migration/config contract before implementation.

## Risks / Open Questions

- Allowing more than one active run in a session can race transcript ordering, session writes, active-run steering, cancellation, and tool side effects unless maintainers define the concurrency contract first.
- Auto-yield needs product semantics for when an in-flight run may be interrupted, whether tools are abortable, how queued prompts are merged, and how it differs from the existing explicit sessions_yield tool.
- The lane-wait warning threshold is separable and already tracked by #14747/#86676, so bundling it with session concurrency could mix a low-risk diagnostic knob with a higher-risk session-state change.
