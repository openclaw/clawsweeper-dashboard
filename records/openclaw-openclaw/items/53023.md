## Summary

Keep open. Current main still serializes each session lane at one active run, documents that per-session serialization, lacks the requested config keys, and only supports explicit `sessions_yield`; related open work covers only parts of the umbrella.

## What This Changes

Request configurable per-session lane concurrency, automatic yield-after-time, and a configurable lane-wait warning threshold for interactive agent sessions.

## Best Possible Solution

Keep this as the umbrella tracker while maintainers review #63864 for session concurrency, keep or consolidate #14747 for warning thresholds, and design auto-yield with explicit cancellation and transcript-ordering semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two runs for the same `session:<key>` still share a lane whose default `maxConcurrent` is 1, and queued work still uses the 2-second warning default unless the caller passes an override.

## Solution Assessment

Unclear as one implementation. Session concurrency and warning thresholds are separable config work, while auto-yield needs maintainer runtime design around cancellation, transcript ordering, and session mutation safety.
