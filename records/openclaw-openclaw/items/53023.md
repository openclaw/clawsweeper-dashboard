## Summary

Keep open: current main and v2026.5.22 still serialize normal session lanes at one active run, keep the default lane-wait warning at two seconds, lack the requested agent-defaults config keys, and only yield through explicit `sessions_yield`; the related open PR and issues cover slices rather than the full scheduler decision.

## What This Changes

Request configurable same-session lane concurrency, automatic yield or preemption after long tool execution, and configurable lane-wait diagnostics for interactive agents.

## Best Possible Solution

A maintainer-owned scheduler design should split the umbrella into backwards-compatible session lane concurrency, a diagnostic-threshold config contract, and explicit auto-yield semantics with transcript-ordering and safety tests.

## Reproduction Assessment

Yes, at source level: current main still routes embedded runs through `session:<key>` lanes capped at one active task and lacks the requested config keys. I did not run a live multi-minute queue-delay reproduction in this read-only review.

## Solution Assessment

Unclear as one implementation. Configurable session concurrency, diagnostic thresholds, and auto-yield are separable changes, and auto-yield needs maintainer decisions around cancellation, transcript ordering, and session mutation safety.
