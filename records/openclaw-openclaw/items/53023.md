## Summary

Keep open. Current main and the latest release still serialize session lanes at one active run, have no public session-concurrency or auto-yield config, and only the narrower lane-wait diagnostic threshold is covered by an open unmerged PR; the remaining scheduler behavior needs maintainer product and session-state design.

## What This Changes

Requests configurable per-session lane concurrency, automatic yielding after long tool execution, and a configurable lane-wait warning threshold.

## Best Possible Solution

Keep this open as the umbrella design tracker: land or replace the narrow diagnostics-threshold PR separately, and require a maintainer-approved session scheduler contract before implementing session concurrency or auto-yield.

## Reproduction Assessment

Yes, at source level. Current main still creates session lanes with one active task, routes embedded runs through those lanes, and exposes only explicit sessions_yield rather than an automatic yield policy.

## Solution Assessment

Unclear. The requested knobs are plausible, but the best solution needs maintainer decisions on session ordering, cancellation, transcript safety, and public config shape before code should be written.
