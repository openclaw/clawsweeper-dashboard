## Summary

Keep open. Current main still lacks the requested configurable per-session lane concurrency, automatic yield-after-time policy, and configurable lane-wait warning threshold; the related open PR covers only the session-concurrency slice, while the warning threshold and auto-yield semantics remain separate maintainer-design work.

## What This Changes

The issue requests agent defaults for per-session concurrency, automatic yielding after a tool-execution duration, and configurable lane-wait warning thresholds to reduce interactive session delays.

## Best Possible Solution

Keep this as the umbrella tracker: review or rework #63864 for the configurable session-concurrency slice, keep or consolidate #14747 for the lane-wait diagnostic threshold, and design auto-yield in the embedded runner/tool loop with explicit cancellation, transcript, ordering, and safety semantics before implementation.

## Reproduction Assessment

Yes. Current source and docs provide a high-confidence reproduction path: two runs targeting the same `session:<key>` still enter a lane whose default `maxConcurrent` is 1, and queued work can still emit the 2-second lane wait warning unless an explicit per-call override is supplied.

## Solution Assessment

Unclear as a single implementation. Configurable session concurrency and warning thresholds are separable, but automatic yielding should be designed as an embedded-runner policy with explicit safety semantics rather than added as a simple timeout knob.
