## Summary

Keep open: current main and v2026.5.22 still serialize normal session lanes, do not expose the requested session concurrency, auto-yield, or lane-wait warning config keys, and the linked work only covers separate slices of the broader scheduler decision.

## What This Changes

The issue requests configurable per-session lane concurrency, automatic yielding during long tool execution, and a configurable lane-wait warning threshold for interactive agents.

## Best Possible Solution

Keep this as the umbrella tracker while maintainers review or replace the session-concurrency PR, keep the warning-threshold slice on its narrower tracker unless folded into one design, and specify auto-yield semantics before implementation.

## Reproduction Assessment

Yes, at source level. Current main still creates session lanes with `maxConcurrent: 1`, routes embedded runs through `session:<key>`, lacks the requested config keys, and only yields when `sessions_yield` is explicitly called.

## Solution Assessment

Unclear as one implementation. Session concurrency, diagnostic thresholds, and auto-yield are separable changes, and auto-yield needs maintainer decisions around cancellation, transcript ordering, and session mutation safety.
