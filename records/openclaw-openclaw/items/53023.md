## Summary

Keep open. Current main and latest release v2026.6.8 still serialize embedded runs through per-session lanes with no requested session-concurrency or auto-yield config; existing steering and interrupt modes are useful partial mitigations, but the remaining work is a core scheduler/product decision rather than conservative cleanup.

## What This Changes

The issue requests configurable per-session lane concurrency, automatic yielding during long tool-heavy turns, and a configurable lane-wait warning threshold.

## Best Possible Solution

Keep this as the umbrella product/design tracker, review the lane-wait threshold slice separately, and only implement per-session concurrency or auto-yield after maintainers settle safe session ordering, cancellation, transcript, and config semantics.

## Reproduction Assessment

Yes, source-level for the current limitation: current main still serializes session lanes at one active run, lacks the requested config keys, and only supports explicit sessions_yield plus queue steer/interrupt modes. I did not run a live timing reproduction in this read-only review.

## Solution Assessment

Unclear. The requested knobs are plausible, but the best solution needs maintainer decisions about whether concurrency, steering/interrupt improvements, explicit yield guidance, or a narrower auto-yield mechanism is the safest product surface.
