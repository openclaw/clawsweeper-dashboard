- `impact:session-state`: The request changes same-session lane concurrency and yielding behavior, which directly affects transcript ordering and active session state.

## Summary

Keep open: current main and the latest release still serialize each session lane at one active run, expose only global/subagent concurrency, keep the default lane-wait warning fallback at 2 seconds, and rely on explicit `sessions_yield`. The related session-concurrency PR and warning-threshold issue cover separate slices, while auto-yield still needs a product/runtime decision.

## What This Changes

Requests configurable same-session lane concurrency, automatic yielding after tool runtime, and a configurable lane-wait warning threshold.

## Best Possible Solution

Keep this as the umbrella tracker while maintainers review or replace https://github.com/openclaw/openclaw/pull/63864 for opt-in session concurrency, leave warning-threshold work to https://github.com/openclaw/openclaw/issues/14747, and design auto-yield with explicit cancellation, transcript-ordering, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still creates session lanes with `maxConcurrent: 1`, routes embedded runs through `session:<key>`, lacks the requested config keys, and only yields when the model explicitly calls `sessions_yield`.

## Solution Assessment

Unclear as a single implementation. Session concurrency, warning thresholds, and auto-yield are separable changes, and auto-yield needs maintainer decisions around cancellation, transcript ordering, and session mutation safety.
