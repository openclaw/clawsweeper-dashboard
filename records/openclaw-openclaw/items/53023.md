## Label Justifications

- `impact:session-state`: The request changes per-session lane concurrency and yielding behavior, which directly affects session ordering, transcript state, and active-run state.

## Summary

Keep open: current `main` and the latest shipped release still serialize session lanes at one active run, lack the requested config keys, keep the default lane-wait warning fallback at 2 seconds, and only support explicit `sessions_yield`; the linked PR and warning-threshold issue cover separate slices rather than the full umbrella.

## What This Changes

The issue requests configurable per-session lane concurrency, automatic yielding during long tool execution, and a configurable lane-wait warning threshold.

## Best Possible Solution

Keep this as the umbrella tracker while maintainers review or replace the session-concurrency PR, leave the warning threshold to the narrower tracker, and design auto-yield with explicit cancellation and transcript-ordering semantics before implementation.

## Reproduction Assessment

Yes, at source level. New session lanes still default to one active run, embedded runs still enter `session:<key>`, the requested config keys are absent, and yielding remains an explicit tool callback rather than an automatic policy.

## Solution Assessment

Unclear as one patch. Session concurrency, lane warning thresholds, and auto-yield are separable changes, and auto-yield especially needs maintainer decisions around cancellation, transcript ordering, and session mutation safety.
