## Summary

This PR targets a real `message.send` buffer attachment bug on current main, but it should stay open because the proposed runtime path can hand gateway-executed plugins a caller-local temp file, decodes caller-controlled base64 before the size check, and still lacks inspectable real channel proof.

## What This Changes

The PR hydrates base64 buffer inputs for `message.send`, materializes them as media via `saveMediaBuffer`, relaxes buffer-only send validation, exports attachment size-limit resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated implementation that carries base64/media safely across the gateway boundary or materializes it in the gateway/channel runtime, prechecks size before decode, preserves media policy limits, and includes live channel proof.

## Reproduction Assessment

Yes, at source level. Current main advertises `buffer` on `message.send`, but ordinary `send` is excluded from attachment hydration and the runner only builds media from URL/path fields.

## Solution Assessment

No. Hydrating `send` is the right direction, but materializing the buffer to caller-local storage before gateway execution and decoding before a size precheck are not the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers in the gateway runtime:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local temp path and stores that path in `params.media` before delivery. On current main, gateway-executed plugin sends forward `params` over `message.action`, and Telegram runs message actions in the gateway, so the gateway can receive a caller-local file path it cannot read. Keep the buffer in the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit is applied in `saveMediaBuffer`. Add an encoded-length estimate or equivalent precheck before allocating the decoded buffer, then keep the decoded-byte check.
  - confidence: 0.78

## Security Review

Status: needs_attention
