## Summary

Keep open: this targets a real current-main message attachment bug, but the branch still needs real channel proof and has a gateway-boundary delivery defect plus an oversized-base64 availability concern before it is merge-ready.

## What This Changes

The branch enables `message.send` buffer hydration, writes hydrated buffers to outbound media files before delivery, relaxes buffer-only sends, and adds outbound runner/params regression tests.

## Best Possible Solution

Land one consolidated `message.send` attachment fix that preserves buffers across the actual delivery runtime boundary, rejects oversized input before decode, keeps per-channel media limits, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level. Current main advertises `buffer` on `message.send`, but ordinary `send` is excluded from attachment hydration and does not promote direct buffers into media delivery.

## Solution Assessment

No. Hydrating send buffers is the right direction, but this branch materializes them before the gateway boundary and applies size policy only after decode; the safer fix keeps buffer handling with the delivery runtime and rejects oversized input early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep send buffers across the gateway boundary:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local temp file and stores that path in `params.media` before gateway plugin delivery can forward the action over `message.action`. Telegram resolves `send` actions to gateway execution, so the gateway may receive a caller-local path it cannot read; keep the buffer in the request or save it after the gateway receives it.
  - confidence: 0.86
- **[P3] Precheck base64 size before decoding:** `src/infra/outbound/message-action-runner.ts:413`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit is applied in `saveMediaBuffer`. Add an encoded-length estimate or equivalent precheck before allocating the decoded buffer, then keep the decoded-byte check.
  - confidence: 0.78
