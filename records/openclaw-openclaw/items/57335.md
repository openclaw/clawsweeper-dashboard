## Summary

Keep open. The current-main bug is source-reproducible and this PR addresses the right surface, but the proposed implementation still materializes caller-provided buffers to a local temp path before gateway-executed plugin dispatch, so remote gateway channels can receive a path they cannot read.

## What This Changes

The PR enables `message.send` buffer hydration, saves hydrated buffers as outbound media paths, exports attachment size-limit resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated send-buffer implementation that normalizes base64/data-URI buffers, enforces encoded and decoded size limits, materializes media only where the delivery runtime can read it, preserves current reply/send media semantics, and adds gateway-dispatch plus channel-level regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `message.send` advertises `buffer`, the hydrator excludes `send`, and the runner only routes URL/path media, so a buffer-only send cannot become an attachment.

## Solution Assessment

No. The hydration direction is right, but the branch currently writes the buffer to a local temp file before gateway dispatch; the safer fix is to carry the buffer over the gateway RPC or materialize it inside the gateway/channel runtime, with base64 size checks before decode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers where the gateway can read them:** `src/infra/outbound/message-action-runner.ts:423-424`
  - body: This saves the hydrated buffer to a local temp file and then forwards that path through the normal `params.media` flow. For gateway-executed channels such as Telegram, the channel runtime runs in the gateway process, so a CLI/local temp path may not exist there; keep the buffer in the gateway request or materialize it inside the gateway/runtime before handing it to the plugin.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: The new send-buffer path decodes `params.buffer` with `Buffer.from(..., "base64")` before the media-store size limit runs. An over-limit encoded payload can still allocate memory and CPU before rejection; add an encoded-length precheck and keep the decoded-byte check via `saveMediaBuffer`.
  - confidence: 0.78
