## Summary

Keep this PR open. Current main still has the reported schema/runtime mismatch, and the PR is useful source work, but it is not merge-ready because the proposed buffer materialization happens before gateway plugin dispatch and can hand gateway-executed channels a client-local media path.

## What This Changes

The PR allows `message.send` calls with non-empty base64 `buffer` args to run attachment hydration, materializes hydrated buffers into outbound temp media for delivery, exports attachment size-limit resolution, and adds unit/runner regression tests.

## Best Possible Solution

Land one shared send-buffer implementation that normalizes base64/data-URI buffers, enforces channel/account media limits, materializes media only where the delivery runtime can read it, supports both local/core and gateway-executed plugin channels, and adds regression coverage for local and gateway-dispatched sends.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the bug path: `message.send` accepts `buffer`, hydration returns early for `send`, and the send runner only treats URL/path fields as media.

## Solution Assessment

No. The PR is close, but materializing the buffer into a local temp path before gateway plugin dispatch is not the safest maintainable fix for gateway-executed channels; the buffer should either cross the gateway RPC or be materialized on the gateway side.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Materialize send buffers on the gateway side:** `src/infra/outbound/message-action-runner.ts:422-423`
  - body: This saves the base64 payload to a local temp path before plugin gateway dispatch can run. Gateway-executed channels such as Telegram forward `params.media` to the gateway and read only `media`/`mediaUrl`, so a CLI or remote send with `{buffer, message}` can hand the gateway a client-local path it cannot read. Keep the buffer over the gateway RPC or materialize it after the action is executing where the channel runtime can access the file.
  - confidence: 0.78
