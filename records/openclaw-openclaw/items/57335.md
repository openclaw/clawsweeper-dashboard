## Summary

Keep open. The bug is source-reproducible on current main, but this PR still needs real behavior proof and has a blocking gateway-boundary issue plus a low-severity base64 size precheck concern.

## What This Changes

The PR hydrates base64 `buffer` attachments for outbound `message.send`, materializes them as media files, relaxes buffer-only sends, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated fix that carries base64 media through the correct runtime boundary, prechecks size before decode, preserves media policy limits, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level. Current main advertises `buffer` on `message.send`, but ordinary `send` is excluded from attachment hydration and the runner never promotes direct buffers into media delivery.

## Solution Assessment

No. Hydrating send buffers is the right direction, but this branch materializes before gateway delivery and decodes caller-controlled base64 before a size precheck; a safer implementation should keep materialization on the runtime that will read the media and reject oversized input early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers after the gateway boundary:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local temp path and stores that path in `params.media` before downstream plugin delivery. Gateway-executed plugin sends forward the current params over `message.action`, and Telegram runs message actions in the gateway, so the gateway can receive a caller-local file path it cannot read. Keep the buffer in the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit is applied in `saveMediaBuffer`. Add an encoded-length estimate or equivalent precheck before allocating the decoded buffer, then keep the decoded-byte check.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or secret-handling change, but it decodes caller-controlled base64 before enforcing media size limits.

Concerns:

- **[low] Inline base64 can allocate before size enforcement:** `src/infra/outbound/message-action-runner.ts:414`
  - body: The send-buffer path calls `Buffer.from(params.buffer, "base64")` before `saveMediaBuffer` receives the resolved max-byte limit, so oversized payloads can consume memory and CPU before rejection.
