## Summary

Keep open: the reported bug is valid and the PR moves in the right direction, but the current branch still has a gateway-boundary bug, needs a base64 size precheck, and lacks required after-fix real behavior proof.

## What This Changes

The PR hydrates base64 buffer attachments for `message.send`, materializes hydrated buffers as outbound media paths, relaxes send validation for buffer media, exports attachment size-limit resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated send-buffer implementation that normalizes base64/data-URI buffers, enforces encoded and decoded size limits, materializes media where the delivery runtime can read it, and includes shared runner plus channel-visible regression proof.

## Reproduction Assessment

Yes, at source level. Current main advertises `buffer` for `message.send`, but the attachment hydrator excludes `send` and the runner only consumes media/path URL fields, so a buffer-only send cannot become delivered media.

## Solution Assessment

No. Hydrating `send` is the right direction, but this branch saves a local temp path before gateway dispatch; a safer fix should carry the buffer across the gateway RPC or materialize it inside the gateway/channel runtime, with a size check before decode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers in the gateway runtime:** `src/infra/outbound/message-action-runner.ts:412-423`
  - body: This saves the hydrated buffer to a local temp file and forwards that path through `params.media`. For gateway-executed channels such as Telegram, the plugin handler runs in the gateway process, so a caller-local path may not exist there; keep the buffer in the gateway request or save it after crossing the gateway boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:413`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before `saveMediaBuffer` enforces `maxBytes`. An oversized encoded payload can still allocate memory and CPU before rejection, so add an encoded-length precheck and keep the decoded-byte check.
  - confidence: 0.78
