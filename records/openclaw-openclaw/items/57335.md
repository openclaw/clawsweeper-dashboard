## Summary

Keep this PR open. Current main still has the schema/runtime mismatch: `message.send` advertises base64 `buffer` attachments, but the shared hydration gate still excludes `send`, and the runner/delivery paths only treat URL/path media as sendable attachments. The PR is still relevant, with overlap to #63578 and #53641 rather than proof that the change has shipped.

## What This Changes

This PR adds send-action base64 buffer hydration/materialization for outbound message attachments and regression tests around the params and runner media paths.

## Best Possible Solution

Land one canonical shared outbound fix that hydrates `send` media hints including `buffer`, materializes base64 through the existing media store with channel/account limits enforced before or during decode, forwards the resulting media path through plugin and core send paths, and keeps focused regression coverage for buffer-only and buffer-with-message sends.

## Reproduction Assessment

Yes. The issue is reproducible from current-main source by invoking `message.send` with `buffer`: the schema accepts it, hydration returns before processing `send`, and the runner never converts `params.buffer` into `media`. A live channel would confirm delivery, but it is not required to prove the mismatch.

## Solution Assessment

Unclear. The PR’s direction is the right shared outbound layer and the later commits address earlier routing/limit review comments, but the base64 materialization path still needs maintainer review for bounded decode/validation and coordination with #63578.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Bound base64 decoding before materializing media:** `src/infra/outbound/message-action-runner.ts:410`
  - body: The new send path decodes the full caller-supplied base64 string with `Buffer.from` before `saveMediaBuffer` enforces the per-channel/account `maxBytes` limit. A request larger than the configured media limit can still allocate the full decoded buffer before being rejected, so check the encoded length or decode under the resolved bound before materialization.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The patch expands runtime handling of caller-provided base64 attachments and needs a bounded decode/validation check before merge.

Concerns:

  - body: The proposed `handleSendAction` path decodes `params.buffer` into a `Buffer` before `saveMediaBuffer` checks `maxBytes`. Oversized base64 can therefore allocate memory before the intended channel/account media limit rejects it.
