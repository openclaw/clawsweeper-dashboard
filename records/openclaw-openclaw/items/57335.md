## Summary

Keep open: this targets a real `message.send` attachment bug, but the current branch is not merge-ready because it can materialize buffers on the wrong side of the delivery boundary, decodes before size enforcement, and still lacks after-fix real channel proof.

## What This Changes

The PR changes outbound `message.send` so non-empty base64 buffers are hydrated, saved as outbound media, treated as media content, and covered by params/runner tests.

## Best Possible Solution

Land one consolidated `message.send` attachment fix that keeps inline buffers valid across the actual delivery runtime or rejects unsupported fields before success, prechecks size before decode, preserves channel limits, and includes real channel proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: current main accepts direct `buffer`/filename/MIME fields for ordinary `message.send`, but the shared hydrator excludes `send` and the send payload path ignores direct buffer fields.

## Solution Assessment

No. Hydrating `send` buffers is the right direction, but this branch materializes buffers before the delivery runtime and checks size only after decode; the safer fix keeps buffer handling with the delivering runtime and rejects oversized input early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep send buffers across the delivery boundary:** `src/infra/outbound/message-action-runner.ts:397-424`
  - body: The patch saves `params.buffer` to a local temp file and rewrites `params.media` before plugin/gateway delivery. Gateway-backed channels such as Telegram execute in the gateway runtime, so a caller-local path may be unreadable there; keep the inline buffer through the delivery boundary or materialize it in the runtime that sends it.
  - confidence: 0.86
- **[P3] Precheck base64 size before decoding:** `src/infra/outbound/message-action-runner.ts:413-414`
  - body: `Buffer.from(params.buffer, "base64")` allocates the decoded payload before `saveMediaBuffer()` applies the resolved media limit. Add an encoded-length estimate or equivalent precheck before decoding, then keep the decoded-byte check.
  - confidence: 0.78
