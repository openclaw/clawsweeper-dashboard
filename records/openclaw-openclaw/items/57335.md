## Summary

Keep open: current main still has the `message.send` buffer schema/runtime mismatch, but this PR is not merge-ready because live channel proof is missing and the patch still has delivery-boundary and pre-decode allocation risks.

## What This Changes

The PR lets `message.send` hydrate non-empty base64 buffers, saves them as outbound media, relaxes buffer-only validation, and adds outbound params/runner tests.

## Best Possible Solution

Land one consolidated `message.send` attachment fix that preserves inline buffers across the actual delivery boundary or rejects unsupported fields before success, prechecks size before decode, keeps channel media limits, and proves live delivery.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main advertises `buffer` on `message.send`, but the shared hydrator excludes ordinary `send` and the send payload path ignores direct buffer fields.

## Solution Assessment

No. Hydrating `send` buffers is the right direction, but this branch materializes buffers before the delivery boundary and applies size policy only after decode; the safer fix keeps buffer handling with the delivering runtime and rejects oversized input early.

  - body: The patch saves `params.buffer` to a local temp file and rewrites `params.media` before plugin/gateway delivery.
