## Summary

Keep open: this targets a real current-main schema/runtime mismatch, but the latest branch still has blocking gateway-boundary, size-enforcement, and real-behavior-proof gaps. The overlapping open work at https://github.com/openclaw/openclaw/pull/63578 should be consolidated rather than landing this branch independently.

## What This Changes

The PR hydrates `message.send` base64 buffers, saves them through `saveMediaBuffer` as `params.media`, relaxes empty-message validation for buffer media, exports attachment max-byte resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated implementation that carries base64/media safely across the gateway boundary or materializes it inside the gateway/channel runtime, prechecks encoded and decoded size limits, resolves the overlap with https://github.com/openclaw/openclaw/pull/63578, and includes live channel proof.

## Reproduction Assessment

Yes, at source level. Current main exposes `buffer` on `message.send`, but the hydrator excludes `send` and the runner only builds media from URL/path fields, so a buffer-only send cannot become delivered media.

## Solution Assessment

No. Hydrating `send` is the right direction, but this PR saves the buffer to caller-local temp media before gateway dispatch and decodes before a size precheck; the safer fix is to carry the buffer through RPC or save it in the gateway/channel runtime with limit checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers in the gateway runtime:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local path before `runGatewayPluginMessageActionOrNull` forwards `params` to `message.action`. Telegram and other gateway-executed plugins run in the gateway process, so forwarded `params.media` can point at a caller-local file the gateway cannot read. Keep the buffer in the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit can reject it in `saveMediaBuffer`. Oversized encoded input can still allocate memory and CPU first, so add an encoded-length estimate/precheck and keep the decoded-byte check.
  - confidence: 0.78
