## Summary

Keep open: current main still has a real source-level `message.send` buffer-attachment bug, but this branch needs contributor real behavior proof and has a blocking gateway-boundary issue plus a low-severity base64 size-precheck concern.

## What This Changes

The PR hydrates `message.send` base64 buffers, materializes them through `saveMediaBuffer`, relaxes send validation for buffer media, exports attachment max-byte resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated send-buffer implementation that normalizes base64/data-URI input, enforces encoded and decoded size limits, carries the media across the gateway boundary safely, and includes real channel-visible proof.

## Reproduction Assessment

Yes, at source level. Current main advertises `buffer` for `message.send`, but the hydrator excludes `send` and the runner only consumes media/path URL fields, so a buffer-only send cannot become delivered media.

## Solution Assessment

No. Hydrating `send` is the right direction, but saving the buffer before the gateway boundary is not the narrowest safe fix; carry the buffer through gateway RPC or materialize it in the gateway/channel runtime, with a size precheck before decode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers in the gateway runtime:** `src/infra/outbound/message-action-runner.ts:412-423`
  - body: The new path writes the hydrated buffer to a local media file before `runGatewayPluginMessageActionOrNull` forwards `params` over `message.action`. Gateway-executed channels such as Telegram run the plugin handler in the gateway process, so a caller-local path may be unreadable; keep the buffer in the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:413`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit is enforced by `saveMediaBuffer`. An oversized encoded payload can still allocate memory and CPU before rejection, so add an encoded-length precheck and keep the decoded-byte check.
  - confidence: 0.78
