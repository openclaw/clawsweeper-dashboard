## Summary

Keep open. Current main still has the advertised `message.send` buffer-attachment gap, and this PR is useful but not merge-ready because it lacks after-fix real behavior proof and introduces a gateway/materialization concern plus a low-severity base64 sizing concern.

## What This Changes

The PR changes outbound `message.send` handling so a base64 `buffer` is hydrated, materialized as media, accepted without text, and covered by new outbound tests.

## Best Possible Solution

Land one consolidated implementation that carries base64 media safely across the gateway boundary or materializes it in the gateway runtime, prechecks size before decode, preserves media policy limits, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level. Current main advertises `buffer` on `message.send`, but ordinary `send` is excluded from attachment hydration and the runner only builds media from URL/path fields.

## Solution Assessment

No. Hydrating `send` buffers is the right direction, but this PR materializes the buffer before gateway delivery and decodes caller-controlled base64 before a size precheck; the safer fix should keep the buffer available to the gateway-side runtime and reject oversized encoded input early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers after the gateway boundary:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local temp path and stores that path in `params.media` before downstream plugin delivery. On current main, gateway-executed plugin sends forward the current params over `message.action`, and Telegram runs message actions in the gateway, so the gateway can receive a caller-local file path it cannot read. Keep the buffer in the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit is applied in `saveMediaBuffer`. Add an encoded-length estimate or equivalent precheck before allocating the decoded buffer, then keep the decoded-byte check.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or secret-handling change, but it decodes caller-controlled base64 before enforcing media size limits.

Concerns:

- **[low] Inline base64 can allocate before size enforcement:** `src/infra/outbound/message-action-runner.ts:414`
  - body: The send-buffer path calls `Buffer.from(params.buffer, "base64")` before `saveMediaBuffer` receives the resolved max-byte limit, so oversized payloads can consume memory and CPU before rejection.
  - confidence: 0.78
