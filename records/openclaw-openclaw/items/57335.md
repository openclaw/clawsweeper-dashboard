## Summary

Keep open: this targets a real `message.send` schema/runtime mismatch, but the branch is not merge-ready because it lacks real channel proof, conflicts with current main, overlaps the related send-media PR, and introduces delivery-boundary plus pre-decode allocation risks.

## What This Changes

The PR hydrates non-empty base64 `buffer` values for `message.send`, materializes them as outbound media paths, relaxes buffer-only validation, and adds outbound params/runner tests.

## Best Possible Solution

Land one consolidated `message.send` attachment contract that preserves buffers across local/gateway delivery or rejects unsupported fields before success, prechecks size before decode, keeps channel media limits, and proves live delivery.

## Reproduction Assessment

Yes, source-level. Current main advertises `buffer` on `message.send`, but the shared hydrator excludes ordinary `send` and the send payload path ignores direct buffer fields.

## Solution Assessment

No. Hydrating `send` buffers is the right direction, but this branch materializes buffers before the delivery boundary and enforces size only after decode; the safer fix consolidates boundary-safe buffer delivery with the overlapping send-media work.
