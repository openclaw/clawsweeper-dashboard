## Summary

Keep open. The bug is valid on current main and this PR targets the right contract gap, but the branch still has a gateway-boundary correctness problem, lacks inspectable real behavior proof, and overlaps with another open send-media hydration PR.

## What This Changes

The PR hydrates `message.send` base64 buffers, materializes them to outbound media paths, relaxes empty-message validation for buffered sends, exports attachment max-byte resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated implementation that carries base64/media safely across the gateway boundary or materializes it inside the gateway/channel runtime, prechecks encoded and decoded size limits, and includes redacted live channel proof.

## Reproduction Assessment

Yes, at source level. Current main exposes `buffer` on `message.send`, but the hydrator excludes `send` and the runner never consumes `params.buffer`, so a buffer-only send cannot become delivered media.

## Solution Assessment

No. Hydrating `send` is the right direction, but this branch saves a caller-local temp file before gateway dispatch and decodes base64 before a size precheck; the safer fix is to carry or materialize the buffer on the gateway/channel side with limit checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers in the gateway runtime:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local path before gateway plugin dispatch forwards `params` over `message.action`. Telegram and other gateway-executed plugins can then receive a `params.media` path that exists only in the caller process or host, so the advertised live-channel attachment path can still fail. Carry the buffer through the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before `saveMediaBuffer` applies the resolved max-byte limit. Oversized encoded input can allocate memory and CPU before rejection, so add an encoded decoded-size estimate/precheck before decoding and keep the decoded-byte check.
  - confidence: 0.78
