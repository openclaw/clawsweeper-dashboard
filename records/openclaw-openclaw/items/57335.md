## Summary

Keep open, but do not merge as-is. The PR addresses a real `message.send` schema/runtime mismatch, but it still lacks after-fix real behavior proof and the current implementation has gateway-boundary and base64 size-precheck concerns that should be resolved before landing.

## What This Changes

The PR lets `message.send` buffer args enter attachment hydration, materializes base64 buffers to outbound media files, exports attachment size-limit resolution, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated implementation that carries base64/media safely across the gateway boundary or materializes it inside the gateway/channel runtime, prechecks encoded and decoded size limits, resolves the overlap with https://github.com/openclaw/openclaw/pull/63578, and includes redacted live channel proof.

## Reproduction Assessment

Yes, at source level. Current main exposes `buffer` on `message.send`, but the hydrator excludes `send` and the runner only builds media from URL/path fields, so a buffer-only send cannot become delivered media.

## Solution Assessment

No. Hydrating `send` is the right direction, but this PR materializes the buffer before the gateway boundary and decodes before a size precheck; the safer fix should carry the buffer through RPC or save it in the gateway/channel runtime with limit checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Materialize send buffers in the gateway runtime:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local path before gateway-executed channels can receive the action. On current main, gateway-mode plugin actions forward `params` over `message.action`, and Telegram resolves message actions to gateway execution, so `params.media` can point at a caller-local file the gateway cannot read. Keep the buffer in the gateway request or save it after crossing that boundary.
  - confidence: 0.86
- **[P3] Bound base64 input before decoding it:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before `saveMediaBuffer` can enforce the resolved media limit. Oversized encoded input can still allocate memory and CPU first, so add an encoded-length estimate or streaming/precheck before decoding, then keep the decoded-byte check.
  - confidence: 0.78
