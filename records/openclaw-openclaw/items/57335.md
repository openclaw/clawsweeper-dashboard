## Summary

Keep open: this PR targets a real `message.send` schema/runtime mismatch, but it still lacks real channel proof and materializes buffers into caller-local temp paths before gateway delivery, which can still drop Telegram/gateway attachments; it also decodes caller-controlled base64 before size enforcement.

## What This Changes

The branch hydrates non-empty base64 buffers for `message.send`, materializes them as outbound media temp files, relaxes buffer-only validation, and adds outbound params/runner tests.

## Best Possible Solution

Land one consolidated `message.send` attachment contract that preserves inline buffers across local/gateway delivery or rejects unsupported fields before success, prechecks size before decode, keeps channel media limits, and proves live delivery.

## Reproduction Assessment

Yes, source-level. Current main advertises `buffer` on `message.send`, but the shared hydrator excludes ordinary `send` and the send/Telegram paths read media references rather than direct buffer fields.

## Solution Assessment

No, not yet. Hydrating buffers is the right direction, but this branch materializes them before the delivery boundary and enforces size only after decode; the safer fix keeps buffer handling with the actual delivery runtime and proves it live.
