## Summary

Keep open: current main still has the advertised `message.send` buffer attachment mismatch, but this PR should not merge until the gateway-boundary bug, base64 size precheck, missing real behavior proof, and overlap with the other send-media PR are resolved.

## What This Changes

The branch hydrates base64 buffers for `message.send`, materializes them as outbound media paths, relaxes buffer-only sends, and adds outbound regression tests.

## Best Possible Solution

Land one consolidated `message.send` attachment fix that carries buffer media through the runtime that performs delivery, rejects oversized input before decode, preserves media policy limits, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level: current main advertises `buffer` on `message.send`, but ordinary `send` is excluded from attachment hydration and the runner does not promote direct buffers into media delivery. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. Hydrating send buffers is the right direction, but this branch materializes before the gateway boundary and decodes caller-controlled base64 before a size precheck; a safer fix keeps buffer handling in the delivery runtime and rejects oversized input early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep send buffers across the gateway boundary:** `src/infra/outbound/message-action-runner.ts:413-424`
  - body: The new path saves `params.buffer` to a local temp file and stores that path in `params.media` before gateway plugin delivery can forward the action over `message.action`. Telegram resolves message actions to gateway execution, so the gateway may receive a caller-local path it cannot read; keep the buffer in the request or save it after the gateway receives it.
  - confidence: 0.86
- **[P3] Precheck base64 size before decoding:** `src/infra/outbound/message-action-runner.ts:414`
  - body: `Buffer.from(params.buffer, "base64")` decodes caller-controlled input before the resolved media limit is applied in `saveMediaBuffer`. Add an encoded-length estimate or equivalent precheck before allocating the decoded buffer, then keep the decoded-byte check.
  - confidence: 0.78
