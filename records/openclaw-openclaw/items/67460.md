## Summary

Keep open: this PR is still the focused implementation candidate for the closed feature request, but it is not merge-ready because it lacks real behavior proof, has blocking correctness/security findings, and currently conflicts with main.

## What This Changes

The branch adds an opt-in shared mention-gating suppression flag, enables it across several channel plugins, changes Discord preflight, updates a PDF model test constant, and adds postinstall cleanup for stale dist plugin node_modules.

## Best Possible Solution

Keep the explicit-target precedence rule, but base it on typed per-channel configured-agent target facts, preserve Discord role context, harden or drop the postinstall cleanup, and reconcile the Discord portion with the related routing work.

## Reproduction Assessment

Yes, at source level. Current main lacks the requested always-on suppression path, and the PR-head regressions are visible from the shared helper, channel call sites, Discord preflight payload, and postinstall code.

## Solution Assessment

No. A shared mention-gating helper is a plausible home, but this implementation should use an explicit configured-agent target signal and should not mix an unhardened postinstall cleanup into the routing feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use an agent-target signal before suppressing:** `src/channels/mention-gating.ts:133-139`
  - body: The suppression path treats `hasAnyMention` as proof that another OpenClaw agent was targeted, but Slack and Telegram pass generic platform mentions. In an always-on room, a normal human/user mention can drop the message even though no configured agent was targeted.
  - confidence: 0.91
- **[P1] Restore member roles in the preflight payload:** `extensions/discord/src/monitor/message-handler.preflight.ts:1086`
  - body: This return payload no longer includes `memberRoleIds`, while `DiscordMessagePreflightContext` requires it and downstream context/routing forwards it into `MemberRoleIds`. That loses sender role context for role-gated behavior.
  - confidence: 0.93
- **[P1] Validate dist roots before recursive deletion:** `scripts/postinstall-bundled-plugins.mjs:673`
  - body: This recursive `rmSync` path only checks existence and Dirent metadata. Reuse the existing lstat/realpath package-root checks before deleting `dist/extensions/*/node_modules` so a redirected dist tree cannot delete outside the package.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR adds a postinstall recursive delete path without the package-root realpath checks used by existing dist pruning.

Concerns:

- **[high] Postinstall recursive deletion lacks root validation:** `scripts/postinstall-bundled-plugins.mjs:673`
  - body: `pruneBundledDistPluginNodeModules` removes `dist/extensions/*/node_modules` recursively during postinstall without proving that `extensionsDir` and the target resolve under the OpenClaw package root. A symlinked or redirected dist tree could delete outside the install directory.
  - confidence: 0.88
