## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not suppress on generic platform mentions:** `src/channels/mention-gating.ts:138-139`
  - body: The new core gate treats `hasAnyMention && !effectiveWasMentioned` as "another agent was mentioned," but the wired channel callers populate `hasAnyMention` from generic platform mentions such as Slack `<@user>`, Telegram `@username`, Teams mention entities, WhatsApp mentioned JIDs, and LINE mention metadata. In an always-on group, a normal message that mentions a human or role now sets `suppressedByOtherAgentMention` and is dropped even though no other OpenClaw agent was targeted. The fix needs an other-agent-specific fact, or only channels that can prove an OpenClaw agent target should opt in.
  - confidence: 0.92
- **[P1] Keep dist pruning inside the package root:** `scripts/postinstall-bundled-plugins.mjs:672-673`
  - body: `pruneBundledDistPluginNodeModules` walks whatever `extensionsDir` points at and then recursively removes `<entry>/node_modules` without validating that `dist/extensions` itself is a real directory under the installed package. If `dist/extensions` is a symlink or redirected path, postinstall can delete directories outside the OpenClaw install tree; current main's dist cleanup already uses realpath/lstat boundary checks, so this branch should not add a weaker deletion path.
  - confidence: 0.88
- **[P1] Restore Discord member roles in the preflight context:** `extensions/discord/src/monitor/message-handler.preflight.ts:1065`
  - body: The PR branch's Discord preflight return payload omits `memberRoleIds`, while current main carries that field into the built context so downstream routing, command authorization, and prompt context can see the sender's roles. Merging this stale return shape would lose role context for Discord inbound messages; add the field back or rebase onto the current `buildDiscordMessagePreflightContext` path.
  - confidence: 0.83
