## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve Gateway auth before trust RPCs:** `src/discord/monitor/trust-command.ts:63-68`
  - body: The Discord command builds a raw `GatewayClient` with only URL and scopes. In token/password or device-auth deployments this sends no usable auth, so `/trust` and `/untrust` fail for authorized owners. Reuse the normal Gateway bootstrap/call path and request the needed admin scope there.
  - confidence: 0.92
- **[P1] Require stable owner IDs for trust commands:** `src/discord/monitor/trust-command.ts:35-51`
  - body: This gate uses the Discord account chat allowlist and still permits dangerous name/tag matching. Normal chat allowlists are broader than owner-only admin identity, and name matching is not safe for a command that grants unrestricted exec. Use the owner identity source and require explicit stable IDs.
  - confidence: 0.9
- **[P2] Hide expired windows from trust status:** `src/gateway/server-methods/exec-approvals.ts:212-218`
  - body: The status RPC returns any cached trust window without checking `expiresAt`. Enforcement ignores expired windows, but clients can still show an active trust state after expiry until another path notices it. Filter expired windows or revoke them before responding.
  - confidence: 0.86
- **[P2] Ignore stale trust responses after agent switches:** `src/tui/tui.ts:766-767`
  - body: `refreshTrustWindow` requests the current agent's status, but writes the result unconditionally when the promise resolves. If the user changes agents while the request is in flight, the old response can populate the footer for the new agent and the 30s throttle delays correction.
  - confidence: 0.83
- **[P2] Count commands that finish after window end:** `src/infra/trust-audit.ts:148-149`
  - body: Audit entries are appended after command completion, but the summary drops entries with `ts > endedAt`. A command started during trust and finishing after expiry or revocation is then omitted from the revocation summary, undercounting commands and failures.
  - confidence: 0.78
- **[P3] Report untrust failure messages from the RPC result:** `src/discord/monitor/trust-command.ts:248-250`
  - body: The server returns `{ ok: false, message }` for untrust failures, but this branch reads `result.summary`, so common failures such as no active window display `unknown error`. Include `message` in the result type and reply with it.
  - confidence: 0.8
