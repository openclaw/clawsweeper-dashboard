## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Move Discord trust commands into the plugin:** `src/discord/monitor/provider.ts:91`
  - body: Current main removed the `src/discord` shim tree and builds the active provider from `extensions/discord/src`. Adding trust commands under `src/discord` means this branch will not rebase into the current plugin-owned Discord runtime or register the commands in the active provider.
  - confidence: 0.93
- **[P1] Resolve Gateway auth before invoking trust RPCs:** `src/discord/monitor/trust-command.ts:59-69`
  - body: `callGatewayRpc` constructs a `GatewayClient` from URL plus requested scopes, but the normal Gateway call path resolves token/password auth, TLS fingerprinting, device identity, method support, and timeouts first. In authenticated deployments, `/trust` and `/untrust` can fail or use the wrong connection semantics.
  - confidence: 0.91
- **[P1] Require stable owner IDs for trust commands:** `src/discord/monitor/trust-command.ts:48-51`
  - body: The owner gate rejects wildcard allowlists but still enables name/tag matching through `dangerousNameMatching`. For unrestricted exec elevation, mutable names and tags are not safe authorization material; this should require explicit stable Discord user IDs.
  - confidence: 0.9
- **[P2] Filter expired trust windows from status:** `src/gateway/server-methods/exec-approvals.ts:212-218`
  - body: The status RPC returns the cached trust window without checking `expiresAt`. Enforcement ignores expired windows, but clients can still show an active trust state after expiry until another path notices and cleans it up.
  - confidence: 0.84
- **[P2] Count commands that finish after the window ends:** `src/infra/trust-audit.ts:148-149`
  - body: Audit entries are appended after command completion, but the summary filter drops entries with timestamps after `endedAt`. A command started during trust and finishing after expiry or revocation is omitted from the summary, undercounting trust-window activity.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds temporary unrestricted exec authority and still has concrete owner-authorization and Gateway-auth wiring concerns.

Concerns:

- **[high] Trust authorization permits mutable-name matching:** `src/discord/monitor/trust-command.ts:48`
  - body: The Discord trust command allows `allowListMatches` with dangerous name/tag matching enabled, so non-ID allowlist entries can authorize unrestricted exec elevation. This needs stable owner IDs only.
  - confidence: 0.9
- **[medium] Discord trust RPC bypasses normal Gateway auth resolution:** `src/discord/monitor/trust-command.ts:59`
  - body: The Discord command creates `GatewayClient` directly instead of using the established Gateway call path that resolves credentials, TLS, device identity, method support, and timeouts, making the admin trust RPC unreliable in authenticated deployments.
  - confidence: 0.91
