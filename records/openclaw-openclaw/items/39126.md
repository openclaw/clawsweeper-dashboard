## Summary

Keep this PR open for maintainer/security review: current main does not implement trust windows, the branch is stale against the current Discord plugin layout, and the proposed temporary unrestricted exec surface still has blocking security and correctness defects.

## What This Changes

The PR adds time-bounded exec trust windows across exec approval resolution, CLI/Gateway RPCs, Discord slash commands, TUI/status display, Swift protocol models, and JSONL audit logging.

## Best Possible Solution

Choose a canonical trust-window design, likely starting from the narrower core/CLI/Gateway work in #46956 and the product scope in #62284, then add Discord/channel UX in the current `extensions/discord` plugin after auth, owner identity, expiry/status, audit semantics, docs, and protocol models are settled.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The blocking defects are source-reproducible from the PR diff and current main: current main lacks trust windows, and the PR targets stale Discord paths while adding security-sensitive trust RPCs.

## Solution Assessment

No. This broad patch is not the best solution as-is because it spans many surfaces, targets the old Discord tree, and still has admin-auth and owner-identity defects; a narrower security-reviewed implementation should land first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move Discord trust commands into the plugin:** `src/discord/monitor/provider.ts:94`
  - body: Current main removed the `src/discord` shim tree and builds the active Discord provider from `extensions/discord/src`. Registering `/trust` here resurrects a stale path, so the command will not be part of the current Discord plugin after rebase. Move the implementation into `extensions/discord` and follow the plugin boundary.
  - confidence: 0.91
- **[P1] Resolve Gateway auth before trust RPCs:** `src/discord/monitor/trust-command.ts:63-68`
  - body: This creates a raw `GatewayClient` with only URL and scopes. The normal Gateway call path resolves token/password auth, TLS, device identity, timeouts, and method support first; without that, `/trust` and `/untrust` fail or use unintended cached device-token scopes in authenticated deployments.
  - confidence: 0.92
- **[P1] Require stable owner IDs for trust commands:** `src/discord/monitor/trust-command.ts:48-51`
  - body: The code says trust commands require specifically listed owner IDs, but then enables dangerous name/tag matching. For unrestricted exec elevation, mutable display names and tags are not safe authorization material; require explicit stable Discord IDs only.
  - confidence: 0.9
- **[P2] Hide expired windows from trust status:** `src/gateway/server-methods/exec-approvals.ts:212-218`
  - body: The status RPC returns any cached trust window without checking `expiresAt`. Enforcement ignores expired windows, but clients can still show active trust after expiry until another path notices it. Filter or revoke expired windows before responding.
  - confidence: 0.86
- **[P2] Ignore stale trust responses after agent switches:** `src/tui/tui.ts:766-767`
  - body: `refreshTrustWindow` requests status for the current agent but writes the result unconditionally when the promise resolves. If the user switches agents while the request is in flight, the footer can show the old agent's trust state until the throttle permits another fetch.
  - confidence: 0.83
- **[P2] Count commands that finish after window end:** `src/infra/trust-audit.ts:148-149`
  - body: Audit entries are appended after command completion, but the summary drops entries with `ts > endedAt`. A command started during trust and finishing after expiry or revocation is omitted from the summary, undercounting commands and failures.
  - confidence: 0.78
- **[P3] Report untrust failure messages from the RPC result:** `src/discord/monitor/trust-command.ts:248-250`
  - body: The server returns `{ ok: false, message }` for untrust failures, but this branch only reads `summary`, so normal failures such as no active window display `unknown error`. Include `message` in the result type and reply with it.
  - confidence: 0.8
