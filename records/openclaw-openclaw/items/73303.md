## Summary

Keep open. Current main still allows a managed Gateway restart to wait through the default 300000 ms restart drain before closing the old process, so the reported 3-4 minute SIGTERM-to-new-start gap remains plausible even though recent releases bounded previously indefinite waits and improved some startup/first-response latency paths.

## What This Changes

The issue requests investigation and a fix/workaround for macOS LaunchAgent Gateway restarts on v2026.4.26 that can wait several minutes before the replacement process starts, with related concerns about plugin teardown and first-response latency after model switches.

## Best Possible Solution

Define an explicit operator-managed restart contract, then implement an observable fast path for service restarts with clear safe-drain versus force-restart semantics, progress logging, and configurable timeout behavior so LaunchAgent restarts do not appear stuck while reply-delivery guarantees remain intentional.

## Reproduction Assessment

Yes for the likely mechanism: current source and fake-timer tests show scheduled and SIGTERM-intent restarts can wait 300000 ms while active work remains. The exact reporter's macOS/plugin environment was not live-reproduced.

## Solution Assessment

No for the current behavior. Main has bounded an indefinite wait and improved adjacent startup/performance paths, but it still permits the several-minute managed-restart delay described here; the safer fix needs a maintainer decision on service-restart policy before changing defaults.
