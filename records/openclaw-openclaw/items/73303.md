## Summary

Keep open. Current main still lets managed service restarts wait through the default 300000 ms restart drain while active tasks, embedded runs, or bundled runtime-dependency installs remain active; the newer package-update restart bypass helps `update.run` package swaps, but it does not change the LaunchAgent/SIGTERM service-restart path described here.

## What This Changes

Request to investigate and reduce multi-minute macOS LaunchAgent Gateway restart delays on v2026.4.26, including shutdown drain timing, plugin/channel teardown logging, and patch/workaround status.

## Best Possible Solution

Define the operator restart contract explicitly, then implement a fast and observable service-restart path with clear safe-drain versus force-restart semantics, progress logging, and documented timeout controls so LaunchAgent restarts do not appear stuck while reply-delivery guarantees remain intentional.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely mechanism: current source and fake-timer tests show restarts can wait 300 seconds while active work remains, and a restart intent makes SIGTERM enter that draining restart path. The exact macOS host/plugin mix was not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. Current main bounds the old indefinite wait and adds a narrow package-update bypass, but a normal managed service restart can still wait several minutes; the safer next step is an explicit maintainer policy decision for operator restarts before changing defaults.
