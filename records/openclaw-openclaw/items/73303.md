## Summary

Keep open. Current main still permits the reported LaunchAgent restart symptom because service-manager SIGTERM restarts enter the normal Gateway restart drain, and the default drain remains 300000 ms when active tasks, embedded runs, or bundled runtime-dependency installs are present. The v2026.4.29 change bounded previously unbounded waits to five minutes, but that does not resolve a 3-4 minute operator restart delay.

## What This Changes

The issue asks OpenClaw to investigate and reduce macOS Gateway restart hangs where a managed LaunchAgent restart can wait several minutes before the new process starts.

## Best Possible Solution

Define a clear operator restart contract for managed Gateway services, then implement a fast and observable service-restart path with explicit safe-drain versus force-restart semantics, progress logging, and documented timeout controls so LaunchAgent restarts do not appear stuck while reply-delivery guarantees remain intentional.

## Reproduction Assessment

Yes for the likely mechanism: current source and fake-timer tests show restarts can wait 300 seconds while active work remains, and a restart intent makes SIGTERM enter that draining restart path. The exact macOS host/plugin timeline was not live-reproduced.

## Solution Assessment

No. Current main bounds the old indefinite wait and adds a narrower package-update bypass, but normal managed service restarts can still wait several minutes; the safer next step is a maintainer decision on operator restart defaults and force semantics.
