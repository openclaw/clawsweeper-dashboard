## Summary

Keep open. Current main still permits the reported several-minute macOS LaunchAgent restart gap because a service-manager Gateway restart writes a restart intent, SIGTERM is treated as a draining restart, and active tasks, embedded runs, or bundled runtime-dependency installs can hold shutdown for the default 300000 ms. The shipped v2026.4.29 work bounds previously indefinite waits and package-update restarts, but it does not eliminate the 3-4 minute operator restart delay described here.

## What This Changes

The issue asks OpenClaw to investigate and reduce macOS LaunchAgent Gateway restarts that linger for several minutes before a fresh process starts, especially around shutdown drain and plugin/channel teardown.

## Best Possible Solution

Define a clear managed Gateway restart contract, then implement a fast and observable operator restart path with explicit safe-drain versus force-restart controls, progress logging, and documented timeout knobs so LaunchAgent restarts do not appear stuck while delivery guarantees remain intentional.

## Reproduction Assessment

Yes for the likely mechanism: current source and tests show service-manager SIGTERM restarts enter the normal restart drain and can wait 300 seconds while active work remains. The exact reporter macOS/plugin timeline was not live-reproduced.

## Solution Assessment

No. Current main bounds the old indefinite wait and gives package-manager update.run a bypass, but normal managed service restarts can still wait several minutes; the safer next step is maintainer product judgment on restart defaults and force semantics.
