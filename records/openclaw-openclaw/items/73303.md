## Summary

Keep open. Current main has update-specific restart improvements and better drain logging, but a managed macOS LaunchAgent restart can still turn SIGTERM into the normal restart drain and wait up to the default 300000 ms before the old process exits and launchd starts the replacement.

## What This Changes

The issue reports multi-minute macOS LaunchAgent Gateway restart delays on v2026.4.26, with related startup and first-response latency reports.

## Best Possible Solution

Define the managed Gateway restart contract, then add an explicit service-manager/operator restart mode with observable fast-drain or force-restart semantics while keeping graceful drains for config reloads and normal in-process restarts.

## Reproduction Assessment

Yes for the likely mechanism: current source and tests show a loaded service-manager restart writes restart intent, SIGTERM consumes it as a restart, and active work can hold the drain for 300000 ms. The reporter's exact macOS/plugin timing was not live-reproduced.

## Solution Assessment

Unclear because there is no proposed patch and the remaining change is a lifecycle policy decision. A narrow technical fix is possible, but maintainers need to choose whether LaunchAgent restarts should bypass, shorten, or preserve active-work drains.
