## Summary

Keep open. Current main still allows the reported several-minute managed macOS Gateway restart gap: service-manager restarts write a restart intent, SIGTERM is converted into a draining restart, and active command tasks, embedded runs, or bundled runtime-deps installs can hold shutdown for the default 300000 ms. The v2026.4.29 work bounds formerly indefinite waits and improves observability, but it does not make operator LaunchAgent restarts fast by default.

## What This Changes

The issue asks whether v2026.4.26 macOS LaunchAgent Gateway restarts can block for several minutes before a new process starts, especially around plugin/channel teardown and startup latency.

## Best Possible Solution

Define the managed Gateway restart contract, then implement an explicit fast operator restart path with visible progress, safe-drain versus force-restart semantics, documented timeout knobs, and coverage for LaunchAgent restarts so restarts do not appear stuck while delivery guarantees remain intentional.

## Reproduction Assessment

Yes for the likely mechanism: current source and tests show service-manager SIGTERM restarts enter the normal restart drain and can wait 300 seconds while active work remains. The exact reporter machine/plugin log sequence was not live-reproduced.

## Solution Assessment

No for the current behavior. Current main bounds the old indefinite wait and gives package-update restarts a bypass, but normal managed service restarts can still wait several minutes; changing that default needs a product/lifecycle decision around reply delivery and restart force semantics.
