## Summary

Current main and the shipped v2026.5.2 source now address the reported restart-drain pain with explicit `gateway restart --force` and `--wait` controls, blocker logging, and documentation, while preserving the default active-work drain as intentional reply-safety behavior.

## What This Changes

The issue reports that macOS LaunchAgent Gateway restarts on v2026.4.26 can wait several minutes before a replacement process starts when active work is present.

## Best Possible Solution

Keep the shipped lifecycle split: default restarts drain active work to protect replies, while operators use the documented `gateway restart --force` or `--wait` controls when they need a prompt managed restart.

## Reproduction Assessment

Yes for the likely mechanism: current source and tests show a SIGTERM restart intent can still drain active tasks and embedded runs for the default 300000 ms budget. The reporter's exact macOS plus Weixin/Telegram environment was not live-reproduced in this read-only review.

## Solution Assessment

Yes. The current implementation is a maintainable balance: it keeps the default drain for reply safety, adds explicit operator `--force` and `--wait` controls, and documents when to use the immediate path instead of changing default restart semantics for everyone.
