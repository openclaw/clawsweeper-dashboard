## Summary

Keep open. Current main still allows the reported managed macOS Gateway restart delay: a service restart writes restart intent, the running Gateway turns the following SIGTERM into a draining restart, and the default drain budget remains 300000 ms when active tasks or embedded runs are present.

## What This Changes

The issue asks OpenClaw to investigate and fix v2026.4.26 macOS LaunchAgent Gateway restarts that can wait roughly 3-4 minutes before the replacement process starts.

## Best Possible Solution

Define the managed Gateway restart contract explicitly, then implement a fast and observable operator restart path with safe-drain versus force-restart semantics, progress logging, and documented timeout knobs so LaunchAgent restarts do not appear stuck while reply-delivery guarantees remain intentional.

## Reproduction Assessment

Yes for the likely mechanism: current source and tests show a service-manager SIGTERM restart enters the normal restart drain and can wait 300000 ms while active work remains. The exact reporter macOS/plugin timeline was not live-reproduced.

## Solution Assessment

Unclear until maintainers decide the lifecycle policy. The narrow code path is identifiable, but the safest implementation depends on whether an operator restart should prioritize fast relaunch or graceful completion of active work.
