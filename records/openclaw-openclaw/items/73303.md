## Summary

Keep open. Current main still allows a managed Gateway restart to spend up to the default 300000 ms drain budget before closing the old process, which matches the reported multi-minute LaunchAgent pre-start gap. Recent main has better bounds, progress logging, and runtime-deps performance work, but the default restart policy still needs maintainer judgment.

## What This Changes

The issue reports macOS LaunchAgent Gateway restarts in v2026.4.26 taking several minutes before a replacement process starts while active work or plugin/runtime-deps activity may be draining.

## Best Possible Solution

Define the operator restart lifecycle policy explicitly, then implement a clear graceful-drain versus force-restart path with progress diagnostics and documented timeout semantics so routine managed restarts are prompt without silently discarding active work guarantees.

## Reproduction Assessment

Yes for the likely mechanism: current main code and fake-timer tests show SIGTERM with a Gateway restart intent enters the restart drain path and can wait for the 300000 ms default while active work remains. The exact reporter machine and plugin combination were not live-reproduced.

## Solution Assessment

No. The current implementation bounds an infinite wait and adds visibility, but it still permits the reported multi-minute pre-start delay; a better solution needs a maintainer-approved default policy or explicit force path.
