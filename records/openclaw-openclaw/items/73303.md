## Summary

Keep open. Current main still allows a managed Gateway restart to wait through the default 300000 ms restart drain when active tasks, embedded runs, or bundled runtime-dependency installs are present; recent runtime-deps and update.run mitigations reduce adjacent causes but do not remove the LaunchAgent/SIGTERM service-restart wait described here.

## What This Changes

The issue asks OpenClaw to investigate and reduce macOS LaunchAgent Gateway restarts that can wait several minutes before the replacement process begins, especially around shutdown, plugin teardown, and v2026.4.26 regressions.

## Best Possible Solution

Define the operator restart contract explicitly, then implement a fast and observable service-restart path with separate safe-drain versus force-restart semantics, progress logging, and documented timeout controls so LaunchAgent restarts do not appear stuck while reply-delivery guarantees remain intentional.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely mechanism: current source and fake-timer tests show managed restarts can wait 300 seconds while active work remains, and SIGTERM with a restart intent enters that path. The exact macOS host/plugin combination was not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. Current main bounds indefinite waits and adds a narrow update.run package-swap bypass, but ordinary managed service restarts can still wait several minutes; the safer next step is a maintainer policy decision for operator restarts before changing defaults.
