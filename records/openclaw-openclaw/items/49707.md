## Summary

Keep open. The bug appears source-reproducible, but this PR is incomplete: it accepts zero-valued usage at one guard while later totals and persistence paths still treat all-zero usage as missing, and the PR has no real behavior proof.

## What This Changes

The PR removes one nonzero-usage guard in embedded PI session token accounting so zero-valued provider usage can be recorded.

## Best Possible Solution

Track whether provider usage was observed separately from whether any counter is nonzero, return zero-valued totals when usage was observed, and add focused regression coverage through embedded subscription and session persistence.

## Reproduction Assessment

Yes, from source inspection. A normalized all-zero provider usage object is rejected by `hasNonzeroUsage` and, even with the PR guard change, `getUsageTotals()` still returns `undefined` for all-zero accumulated totals.

## Solution Assessment

No. The PR changes only the first guard; the maintainable fix needs an explicit “usage observed” state carried through totals and persistence so zero can remain distinct from missing data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Preserve all-zero usage through totals:** `src/agents/pi-embedded-subscribe.ts:274-276`

A provider response such as `{ input: 0, output: 0, total: 0 }` can be accepted here, but `getUsageTotals()` still returns `undefined` unless some accumulated counter is `> 0`, so downstream `attemptUsage` remains missing and the display keeps treating usage as unknown. Track whether usage was observed separately from whether any counter is nonzero, and return zero-valued totals when observed.
