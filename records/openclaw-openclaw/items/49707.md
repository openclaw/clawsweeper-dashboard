## Summary

Keep open: the reported zero-usage problem is still valid, but this PR is incomplete and currently lacks after-fix real behavior proof.

## What This Changes

The PR removes the nonzero-usage guard in `recordAssistantUsage()` so zero-valued provider usage can be accepted by the subscription path.

## Best Possible Solution

Carry an explicit observed-usage signal through subscription aggregation, runner metadata, session persistence, and status formatting so valid reported zeros render as fresh `0/context` while absent usage remains unknown.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass normalized `{ input: 0, output: 0, total: 0 }` through the assistant usage path; current nonzero-only gates and zero-collapsing totals drop it before status can render a fresh zero.

## Solution Assessment

No. Changing one old subscription guard is not the best fix because current main still has zero-dropping gates in `resolveAssistantUsage()`, `getUsageTotals()`, the runner accumulator, and session persistence.
