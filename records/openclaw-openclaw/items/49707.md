## Summary

Keep open. The PR is a plausible bug fix, but it still does not solve the zero-valued usage path on current main: all-zero usage is rejected by the embedded resolver/totals and session persistence semantics, and the PR has no real after-fix behavior proof.

## What This Changes

The PR removes the `hasNonzeroUsage` import and changes `recordAssistantUsage` in `src/agents/pi-embedded-subscribe.ts` to accept any normalized usage object.

## Best Possible Solution

Implement a presence-aware usage snapshot that distinguishes provider-reported zero from missing usage across embedded aggregation, persistence, and status display, with focused regression coverage and redacted real output; keep broader local-estimation work in https://github.com/openclaw/openclaw/issues/54996.

## Reproduction Assessment

Yes, by source inspection. Current main normalizes zero values but still gates embedded resolution, aggregation, and persistence on nonzero counters, so an all-zero provider usage object does not become a fresh token snapshot.

## Solution Assessment

No. Relaxing only the first guard is not the narrow maintainable fix; the code needs explicit observed-usage state carried through totals, persistence, and display handoff.
