## Summary

Keep open: this is a focused bug-fix attempt, but the patch does not fix the zero-valued usage path end to end, the branch is merge-conflicting, and the PR still lacks real after-fix `/status` proof from a provider/setup that returns all-zero usage.

## What This Changes

The PR removes `hasNonzeroUsage` from `src/agents/pi-embedded-subscribe.ts` and changes assistant usage recording to accept any normalized usage object so zero-valued provider usage can be shown in `/status`.

## Best Possible Solution

Track explicit usage presence separately from nonzero token totals through embedded resolution, accumulation, session persistence, and status rendering, while preserving unknown only for genuinely absent or stale usage and adding focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. A zero-valued normalized usage object is still rejected by `resolveAssistantUsage()`, converted to `undefined` by `getUsageTotals()`, or treated as missing by persistence/status paths; I did not run a live Qwen session.

## Solution Assessment

No. The one-line guard swap is not the best fix because downstream code still uses nonzero totals as the existence signal, so the maintainable fix needs an explicit usage-present signal carried through the full pipeline.
