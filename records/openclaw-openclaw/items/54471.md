## What This Changes

The PR adds a `system_event` handler in `startAcpSpawnParentStreamRelay()` to emit non-empty event text to the parent ACP session.

## Best Possible Solution

Update this PR in place to relay sanitized current ACP runtime status text from the `acp` stream, count it as visible progress/resume state, preserve assistant/lifecycle behavior, add focused parent-relay tests, and keep broader tool/result/thinking visibility with https://github.com/openclaw/openclaw/issues/47596.

Yes at source level. Current main emits ACP status/progress through `stream: "acp"` runtime events and the relay logs/counts those events without enqueueing their text to the parent session; I did not run a live ACP spawn in this read-only review.

## Solution Assessment

No. The branch handles a separate `system_event` stream and misses progress/stall state; the narrower maintainable fix is to relay sanitized current ACP runtime status text from the existing ACP event contract and cover it in the parent-relay tests.
