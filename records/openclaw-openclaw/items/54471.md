## Summary

Keep open: the reported ACP parent-stream bug is source-reproducible, but this PR is not merge-ready because current ACP `status` text is still dropped before the relay and the added relay branch would not update stall/resume accounting. The contributor also has not provided after-fix real behavior proof from a real ACP spawn.

## What This Changes

The PR adds a `system_event` branch to `startAcpSpawnParentStreamRelay()` so non-empty text can be emitted to the parent session with a fallback context key.

## Best Possible Solution

Emit or translate non-empty ACP runtime status/progress text into untrusted parent-session system events at the ACP command boundary, count that visible text as relay progress for stall/resume state, preserve existing assistant/lifecycle behavior, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current ACP runtime events include `type: "status"` text, while the ACP command callback returns before emitting anything for non-`text_delta` events, so a `sessions_spawn({ runtime: "acp", streamTo: "parent" })` status or clarifying event can be dropped before the relay.

## Solution Assessment

No. Listening for an agent-event stream named `system_event` inside the relay is not enough when the actual ACP `status` source is never emitted there, and relayed system text should also refresh stall/resume state.
