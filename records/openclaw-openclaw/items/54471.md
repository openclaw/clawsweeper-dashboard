## Summary

Keep open: the reported ACP parent-stream bug is still plausible, but this PR is not merge-ready because current main routes ACP status/progress text through `stream: "acp"`, while the branch listens for `system_event`, misses stall-state bookkeeping, lacks after-fix real behavior proof, and is currently conflicting.

## What This Changes

The PR adds a `system_event` branch to `startAcpSpawnParentStreamRelay()` that emits non-empty event text to the parent session with a fallback ACP context key.

## Best Possible Solution

Update this PR in place to relay current `acp` `runtime_event` status text as parent system events, count emitted status text as progress/resume activity, preserve assistant/lifecycle behavior, add focused tests, and include redacted real ACP spawn proof.

## Reproduction Assessment

Yes, at source level: ACP `status` events carry text, `agent-command.ts` emits them on the `acp` stream, and the parent relay currently records that stream as diagnostics/progress state without enqueueing the text to the parent session. I did not run a live ACP spawn in this read-only review.

## Solution Assessment

No. The PR listens for a new `system_event` stream instead of relaying the current `acp` `runtime_event` status text, and it omits the stall/resume bookkeeping used for visible assistant output.
