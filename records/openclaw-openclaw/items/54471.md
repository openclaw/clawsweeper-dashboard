## Summary

## What This Changes

The PR adds a `system_event` branch to the ACP spawn parent relay so non-empty event text can be emitted to the parent session with a fallback context key.

## Best Possible Solution

Update this PR in place to relay sanitized current ACP runtime status text from the `acp` stream, count it as visible progress/resume state, preserve assistant/lifecycle behavior, add focused relay coverage, and then provide real ACP spawn proof.

## Reproduction Assessment

Yes, at source level. Current main emits ACP status/progress through `stream: "acp"` runtime events and the relay logs/counts those events without enqueueing their text to the parent session; I did not run a live ACP spawn in this read-only review.

## Solution Assessment

No. A separate `system_event` branch does not match the current ACP event source, and the safer narrow fix is to relay sanitized ACP runtime status text from the existing `acp` event contract while updating stall/resume state.
