## Summary

Keep open: current main still drops ACP status/progress text before the parent relay, and this PR only adds a relay-side listener that has no current ACP status producer and does not update stall accounting. The existing Greptile and ClawSweeper comments were useful context and align with the remaining relay-progress defect.

## What This Changes

The PR adds a `system_event` branch in `startAcpSpawnParentStreamRelay()` so non-empty system-event text is emitted to the parent session.

## Best Possible Solution

Map non-empty ACP runtime `status` text into untrusted parent-session system events, treat relayed system text as progress for stall/resume state, preserve existing assistant/lifecycle behavior, and add focused tests plus a user-facing changelog fix.

## Reproduction Assessment

Yes, at source level. During a `sessions_spawn({ runtime: "acp", streamTo: "parent" })` turn, an `AcpRuntimeEvent` with `type: "status"` and text is dropped in `agent-command.ts` before `startAcpSpawnParentStreamRelay()` can emit it.

## Solution Assessment

No. A relay listener for `event.stream === "system_event"` alone is not sufficient because current ACP status text is never emitted on that stream, and relayed system text should also refresh the relay's stall/resume state.
