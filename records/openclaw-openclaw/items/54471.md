## Summary

Keep open: the PR targets a real documented ACP parent-streaming bug, but the patch is incomplete because current ACP status text is dropped before this new relay branch can see it, and the branch does not update stall/resume progress state.

## What This Changes

The PR adds a `system_event` handler to `startAcpSpawnParentStreamRelay()` so non-empty event text can be emitted to the parent session.

## Best Possible Solution

Map non-empty ACP runtime `status` text into untrusted parent-session system events before it is dropped, count visible system text as relay progress for stall/resume state, preserve assistant/lifecycle behavior, and add focused ACP relay tests.

## Reproduction Assessment

Yes, at source level. Current ACP runtime events include `type: "status"` text, while `agent-command.ts` returns before emitting anything for non-`text_delta` events, so a `sessions_spawn({ runtime: "acp", streamTo: "parent" })` status or clarifying event is dropped before the relay.

## Solution Assessment

No. A relay branch for `event.stream === "system_event"` alone is not sufficient because the actual ACP status source is not emitted on that stream, and emitted system text should also refresh the relay's stall/resume accounting.

## Review Findings

Runtime `status` text therefore still never reaches this handler, so the reported clarifying/progress messages remain dropped unless the actual status source is emitted or translated first.
  - confidence: 0.88

A run that only produces status or clarifying messages can show that text and then still enqueue the no-output warning after the timeout, which creates contradictory parent-session updates.
  - confidence: 0.9
