## Summary

Keep open. The branch targets a real ACP parent-relay gap, but the diff only adds a relay-side listener for a stream that ACP status/progress does not currently emit, and it would leave visible system events outside the stall/resume accounting.

## What This Changes

The PR adds a `system_event` branch in `startAcpSpawnParentStreamRelay()` to emit non-empty system-event text to the parent session.

## Best Possible Solution

Map non-empty ACP runtime status/progress text into untrusted parent-session system events, treat that text as relay progress for stall/resume state, preserve existing assistant/lifecycle/commentary behavior, and add focused tests plus a user-facing changelog fix.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an ACP `runTurn` event with `type: "status"` during a `streamTo: "parent"` spawn; current main and the PR branch return before emitting an agent event for that status text.

## Solution Assessment

No. The proposed relay listener is not the best fix because the actual ACP status source is dropped upstream, and relayed system text should also update the same progress timers as assistant output.
