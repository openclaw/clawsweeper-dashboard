## Summary

Keep open. Current main still lacks a first-class remote bundle MCP caller-context opt-in, and this PR is a plausible implementation with no definite correctness finding, but forwarding session/caller headers to remote MCP endpoints is a trust-model change that should get maintainer review and exact-head validation before merge.

## What This Changes

The PR adds per-server `mcp.servers.<name>.injectCallerContext` support, caller-header injection for trusted remote bundle MCP servers, placeholder env gating, config schema metadata, changelog, and focused bundle MCP tests.

## Best Possible Solution

Ship the owner-managed, URL-bound opt-in only after maintainers approve the trust boundary and exact-head validation is green, keeping plugin self-opt-ins ignored and config/schema/changelog behavior aligned.

## Reproduction Assessment

Yes. Current main can be verified statically: there is no `injectCallerContext` config field and `OPENCLAW_MCP_*` env is loopback-only; the PR behavior is reproducible by generating bundle MCP config with a remote URL plus the opt-in and inspecting emitted headers.

## Solution Assessment

Yes, with merge-readiness caveats. A per-server owner-managed URL-bound opt-in is narrower than a global flag or plugin-declared trust, but it still needs maintainer approval for session forwarding and validation before merge.
