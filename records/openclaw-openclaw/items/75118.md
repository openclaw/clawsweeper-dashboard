## Summary

Keep open. Current main still lacks the opt-in caller-context path for remote bundle MCP servers, while this PR adds a plausible owner-managed implementation but is dirty/unmergeable and touches a security-sensitive caller/session forwarding boundary that needs maintainer trust-model review before merge.

## What This Changes

The PR adds a per-server `mcp.servers.<name>.injectCallerContext` config flag, caller-header injection for trusted remote bundle MCP URLs, placeholder environment gating, config schema/type metadata, changelog coverage, and bundle MCP tests for Claude, Codex, Gemini, and prepare paths.

## Best Possible Solution

If maintainers accept the trust model, rebase the branch onto current main, keep the owner-managed per-server opt-in with exact URL matching and plugin-supplied opt-ins ignored, run the targeted bundle MCP/config docs checks, then run the changed gate before merge.

## Reproduction Assessment

Yes. Current main can be verified statically: there are no `injectCallerContext` code paths, and `OPENCLAW_MCP_*` env is only populated for loopback runtime; the PR behavior can be reproduced by generating bundle MCP config for a remote URL with `injectCallerContext: true` and inspecting emitted headers.

## Solution Assessment

Yes, with merge-readiness caveats. The owner-managed, URL-bound opt-in is the narrowest maintainable shape found here; a global flag or plugin-self-declared trust would be riskier, but the PR still needs conflict resolution, maintainer security review, and validation.
