## Summary

Keep open. Current main does not implement this config key or remote bundle MCP caller-context injection, and this PR is the active implementation for the linked feature issue, but the latest diff has a caller-context trust-boundary bug and failing PR checks that should block merge.

## What This Changes

The PR adds an optional `mcp.servers.<name>.injectCallerContext` field, injects OpenClaw caller headers into opted-in remote bundle MCP server configs, populates related placeholder env, updates generated config metadata/changelog text, and adds bundle MCP tests.

## Best Possible Solution

Keep the per-server opt-in direction, but make the trusted unit the owner-managed remote server definition itself: require the trusted layer to include a non-empty remote `url`, keep plugin self-opt-in ignored, preserve the disabled-tools env gate, add a regression test for same-name merge layers, and land only after CI and maintainer trust-boundary review are green.

## Reproduction Assessment

Yes. Current main reproduces the feature gap by configuring a remote bundle MCP server with the proposed flag and observing that no config key or injection pass exists; the PR-specific security bug can be reproduced with an existing `--mcp-config` server URL plus owner config for the same name that sets only `injectCallerContext: true`.

## Solution Assessment

No, not as written. The per-server opt-in is the right maintainable shape, but the implementation should bind trust to an owner-supplied remote URL server rather than trusting a name after unrelated merge layers have supplied transport details.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Tie caller-context trust to an owner-supplied URL:** `src/agents/bundle-mcp-config.ts:94-95`
  - body: This marks a server name trusted whenever owner config has `injectCallerContext: true`, even if that owner-managed entry has no `url`. Since `prepareCliBundleMcpConfig` first reads an existing MCP config and then merge-patches bundle/owner config over it, an earlier layer can provide the URL for the same name and still receive `x-session-key` plus caller headers. Require the trusted layer itself to be a remote URL server, or carry trusted server objects through the merge, before injecting caller context.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete caller/session forwarding concern because trust is tracked by server name instead of by the owner-supplied remote server definition.

Concerns:

- **[medium] Name-only trust can leak caller headers:** `src/agents/bundle-mcp-config.ts:94`
  - body: A config entry with only `injectCallerContext: true` adds the name to the trusted set; after merge patches, a URL from an earlier config layer with the same name can receive `x-session-key` and caller identifiers. The opt-in should require the trusted layer itself to declare the remote URL that will receive the headers.
  - confidence: 0.84
