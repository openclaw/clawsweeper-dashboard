Current main still lacks the requested remote bundle MCP caller-context opt-in, and the PR is not obsolete; the latest head also appears to address the earlier name-only trust leak by binding trust to an owner-declared URL. ## Best Possible Solution

If maintainers accept the trust model, land a rebased version that keeps caller-context injection owner-managed, URL-bound, disabled-tools gated, documented in the config surface, and covered by targeted bundle MCP tests.

Yes, with merge-readiness caveats. A per-server owner-managed URL-bound opt-in is the narrow maintainable shape for this feature; the safer alternatives are exactly the constraints the latest PR adds, namely ignoring plugin self-opt-in and refusing name-only trust.
