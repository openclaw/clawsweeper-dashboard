## Summary

Keep open: current main still lacks the requested opt-in, and this PR is a plausible implementation, but it is not mergeable as-is because it still has correctness findings, missing real behavior proof, and a security-sensitive trust-model decision around remote caller/session headers.

## What This Changes

The PR adds a default-off `mcp.servers.<name>.injectCallerContext` option that injects OpenClaw caller headers into owner-trusted remote bundle MCP servers across CLI and embedded Pi paths, with schema, changelog, and tests.

## Best Possible Solution

Rebase onto current main, keep the per-server owner-trusted exact-URL opt-in, fix header preservation and schema metadata, then require targeted tests plus redacted real CLI or remote MCP proof before maintainer/security signoff.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug, but yes: the feature gap is source-verifiable on current main because the typed config, Zod schema, bundle merge path, and CLI env preparation do not implement the requested opt-in.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The per-server exact-URL opt-in is the right general shape, but the patch still corrupts some Pi headers, misses schema source metadata, and needs real proof plus security review.
