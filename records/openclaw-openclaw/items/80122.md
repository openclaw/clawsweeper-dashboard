## Summary

Keep this PR open: the direction matches a real Codex app-server media-delivery gap, but the proposed scanner can turn a successful Codex turn into a failed attempt on one unreadable generated-artifact subtree, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR snapshots Codex app-server `generated_*` directories, appends newly written deliverable artifacts to the final reply media payload, adds fallback captions for artifact-only replies, and covers the path with Codex app-server tests.

## Best Possible Solution

Keep the Codex media bridge, but make native-artifact discovery best-effort, prefer protocol-backed paths such as `imageGeneration.savedPath` where available, and require a real Codex app-server delivery proof before merge.

## Reproduction Assessment

Yes from source inspection: current `main` only forwards dynamic-tool media telemetry from Codex app-server attempts, while upstream Codex has native generated-image item paths that OpenClaw does not currently attach. I did not live-run Codex in this read-only review.

## Solution Assessment

No, not yet: attaching native generated artifacts is the right owner-boundary direction, but the scanner must not fail completed turns on incidental filesystem errors, and the PR needs real behavior proof from an actual Codex app-server run.
