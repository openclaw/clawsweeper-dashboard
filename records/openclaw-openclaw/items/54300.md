## Summary

Keep open. Current main still has the two narrow hardening gaps from the report: Docker label names are interpolated into a Go template without validation, and the thread-ownership plugin sends raw channel/thread tokens as URL path segments. The TLS/private-WS and Control UI dangerous flags are intentional, documented break-glass behavior, so changing those needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue asks to harden Docker label template construction, Slack thread-ownership ownership URL construction, and gateway/Control UI break-glass bypass mechanisms.

## Best Possible Solution

Keep the issue open for maintainer security triage. Land focused validation/encoding fixes for Docker label reads and thread-ownership ownership URLs, then decide separately whether the private-WS and Control UI break-glass mechanisms should be redesigned.

## Reproduction Assessment

Yes for source-level reproduction: static inspection shows Docker label input is interpolated into a Go template, and a URL constructor check shows raw thread-ownership path segments can normalize outside the intended route. Full exploitability is limited by current hardcoded Docker callers and normal Slack ID shapes.

## Solution Assessment

Unclear for the grouped issue as a whole. Label validation and path-segment encoding are narrow maintainable fixes for the first two findings, while replacing documented break-glass flags with an allowlist is a security/product policy decision.
