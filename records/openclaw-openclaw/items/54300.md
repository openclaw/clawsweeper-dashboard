## Summary

Keep open. Current main still has the two narrow hardening gaps from the prior review: Docker label names are interpolated into an inspect Go template without validation, and thread-ownership still builds a forwarder URL from raw path segments. The private-WS and Control UI dangerous flags are documented, tested break-glass behavior, so changing those should go through maintainer security/product triage instead of cleanup closure.

## What This Changes

The issue requests low-risk hardening for Docker label inspection templates, thread-ownership forwarder URL path construction, and operator TLS/auth break-glass bypasses.

## Best Possible Solution

Keep this open for maintainer security triage. Land focused hardening for Docker label-name validation and thread-ownership URL segment encoding or strict validation, then handle private-WS and Control UI break-glass policy as a separate secops/product decision with docs and tests updated together.

## Security Review

Status: needs_attention

Summary: Two concrete low-risk hardening gaps remain; the TLS/auth bypass concerns are documented operator break-glass controls that need maintainer policy review.

Concerns:

- **[low] Validate Docker label names before templating:** `src/agents/sandbox/docker.ts:196`
  - body: readDockerContainerLabel interpolates an arbitrary string into a Docker Go template. Current callers are hardcoded, but the helper should reject labels outside Docker label-key characters before constructing `docker inspect -f`.
  - confidence: 0.9
- **[low] Encode thread ownership URL path segments:** `extensions/thread-ownership/index.ts:178`
  - body: thread-ownership constructs the forwarder ownership URL from raw channel/thread tokens, and current token normalization can preserve slash or dot-segment values. Encoding or strict validation would prevent path rewriting before the SSRF guard fetch.
  - confidence: 0.9
