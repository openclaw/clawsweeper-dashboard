## Summary

Keep open. Current main still has the two narrow hardening gaps called out in the prior ClawSweeper review: Docker label names are interpolated into a Go template without validation, and the thread-ownership plugin builds a forwarder URL from raw path segments. The TLS/private-WS and Control UI flags remain documented, tested break-glass behavior, so changing those needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue asks to harden Docker label template construction, thread-ownership forwarder URL path construction, and documented TLS/Control UI break-glass bypasses.

## Best Possible Solution

Keep the issue open for security triage. Land narrow validation and path-segment encoding fixes for the Docker helper and thread-ownership plugin, then decide separately whether the private-WS and Control UI break-glass mechanisms should be redesigned or left as documented operator policy.

## Reproduction Assessment

Yes for source-level reproduction: static inspection shows the Docker label value is interpolated into a Go template, and a URL constructor check demonstrates raw thread path segments can normalize outside the ownership route. Runtime exploitability is limited or context-dependent because current Docker callers are hardcoded and normal Slack IDs are canonical.

## Solution Assessment

Unclear for the grouped issue as a whole. Strict Docker label-name validation and `encodeURIComponent` for ownership path segments are narrow maintainable fixes for the first two findings, while replacing the break-glass flags is a maintainer security/product decision.
