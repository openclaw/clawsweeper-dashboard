## Summary

Keep open. Current main still leaves the two narrow hardening gaps from the previous review: Docker label names are interpolated into a Go template without validation, and the thread-ownership plugin builds a forwarder path from raw channel/thread tokens. The TLS/private-WS and Control UI dangerous flags are documented, tested break-glass behavior, so changing those should go through maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue requests low-severity hardening for Docker label template construction, thread-ownership forwarder URL path construction, and TLS/Control UI break-glass bypass disclosure.

## Best Possible Solution

Keep the issue open for security triage. Land narrow validation/encoding fixes for Docker label reads and thread-ownership ownership URLs, then separately decide whether the private-WS and Control UI break-glass mechanisms need a policy redesign.

## Reproduction Assessment

Yes for source-level reproduction: the raw thread URL path can be shown to normalize away from the ownership route, and static inspection shows Docker label input is interpolated into a Go template. Full exploitability is limited/unclear because current Docker callers are hardcoded and normal Slack IDs are canonical.

## Solution Assessment

Unclear for the grouped issue as a whole. Strict label validation and path-segment encoding are narrow maintainable fixes for the first two findings, while replacing documented break-glass flags with a narrower allowlist is a product/security decision rather than an automatic cleanup change.
