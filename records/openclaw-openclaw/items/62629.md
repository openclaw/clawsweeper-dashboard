## Summary

Keep open. Current main still has no explicit profile-to-provider-model mapping for `agents.defaults.models`; exact model-ref behavior means the reported suffix can still become the runtime model id, while stripping after `:` would break supported model ids that already use colons.

## What This Changes

Request to support multiple `agents.defaults.models` entries for one upstream model by using unique suffixed config keys while sending the base model id and per-variant params to the provider.

## Best Possible Solution

Add a documented, backward-compatible model profile routing contract that separates selectable OpenClaw config keys from provider-facing model ids and keeps schema, docs, pickers, allowlist checks, params lookup, session persistence, and runtime resolution aligned.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a suffixed `agents.defaults.models` key with an alias, select that alias, and the current alias/runtime paths preserve the suffixed model id through provider model construction.

## Solution Assessment

No. Delimiter stripping is not the best solution because `:` is already meaningful in supported model ids; an explicit route/profile mapping is safer and matches the related open design work.
