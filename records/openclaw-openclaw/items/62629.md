## Summary

Keep open. Current main still treats `agents.defaults.models` keys as exact model refs, so a suffixed key can continue through alias/session resolution as the runtime model id; the requested capability is valid, but delimiter stripping is unsafe because supported model ids already use `:`.

## What This Changes

The issue requests a way to define multiple `agents.defaults.models` entries for the same provider model with different params while sending the unsuffixed provider model id to the API.

## Best Possible Solution

Add an explicit, documented variant mapping for `agents.defaults.models` so the unique OpenClaw config key can point to a provider-facing model id while preserving per-variant params. The change should update schema/help/docs and keep `/model`, Control UI/WebChat picker selection, allowlist checks, extra params lookup, and runtime provider resolution aligned.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a suffixed `agents.defaults.models` key with an alias, select that alias or key, and current alias/runtime resolution can carry the suffixed model id into provider model construction.

## Solution Assessment

No. Stripping after `:` is not the best solution because `:` is already meaningful in supported provider model ids; an explicit variant-to-provider-model mapping is safer and preserves exact model-id semantics.
