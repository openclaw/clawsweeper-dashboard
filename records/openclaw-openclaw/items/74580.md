## Summary

Keep open. Current main still treats non-bundled `agent_end` as a raw conversation hook, and there is no `allowStateAccess` or state-only `agent_end` payload; the existing agent-event subscription path is a possible workaround, but the issue discussion correctly narrows the remaining ask to a documented, lightweight turn-ended/state contract that does not grant conversation content access.

## What This Changes

Add a state-only plugin permission or equivalent lifecycle signal so non-bundled plugins can observe agent turn completion without granting raw conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive state-only turn-ended/lifecycle contract that exposes only run/session/state/timing fields, with contract tests proving no conversation text reaches state-only consumers.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is rejected on current main, and `src/plugins/loader.test.ts:6795` codifies that path.

## Solution Assessment

Unclear. A new `allowStateAccess` flag is plausible, but exposing the existing `agent_end` event under that flag would be unsafe because it carries `messages`; a sanitized event or explicitly documented lifecycle subscription contract is the safer maintainable direction.
