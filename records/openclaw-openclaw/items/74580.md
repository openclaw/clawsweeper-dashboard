## Summary

Keep open. Current main still treats `agent_end` as a conversation-content hook, still exposes raw `messages` on that event, and has no `allowStateAccess` or explicit state-only turn-ended hook contract. The linked OpenClaw Interaction Bridge and follow-up comment are useful evidence of a real external-plugin use case, but this is a public Plugin SDK permission-boundary decision rather than a conservative cleanup close.

## What This Changes

The issue requests a state-only plugin hook permission or equivalent lifecycle signal so external plugins can observe agent turn state without full conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive state-only turn-ended/lifecycle SDK contract that exposes only run, session, phase, outcome, and timing fields, with contract tests proving no prompt, response, history, or message array reaches state-only consumers.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence reproduction path: a non-bundled plugin registering `api.on("agent_end", ...)` without `hooks.allowConversationAccess=true` receives no typed hook registration, and `src/plugins/loader.test.ts` codifies that behavior.

## Solution Assessment

Unclear. `allowStateAccess` is plausible, but the current `agent_end` payload cannot be reused safely because it includes raw messages; a sanitized turn-ended event or a clearly documented lifecycle subscription contract would be the safer maintainable direction.
