## Summary

Keep open. Current main still treats non-bundled `agent_end` as a conversation-content hook, the existing `agent_end` payload still includes raw `messages`, and there is no state-only hook policy or documented lightweight turn-ended contract. The issue is valid plugin API work, but it changes a security-sensitive permission boundary and needs maintainer direction before implementation. https://github.com/openclaw/openclaw/issues/74580

## What This Changes

Request a state-only plugin permission or equivalent sanitized turn-ended lifecycle contract so non-bundled plugins can track agent state without full conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add a documented, tested minimal lifecycle/turn-ended contract that exposes only session/run/state/timing fields. If maintainers choose `registerAgentEventSubscription` as the intended path, document that recipe for state-display plugins and add contract tests proving no conversation text reaches the state-only path.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is blocked by the current registry path, and `src/plugins/loader.test.ts` codifies that behavior.

## Solution Assessment

Unclear. `allowStateAccess` is plausible, but the existing `agent_end` payload includes raw messages, so a sanitized turn-ended event or explicitly documented lifecycle subscription path is safer than relaxing the current gate.
