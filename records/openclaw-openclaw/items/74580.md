## Summary

Keep open. Current main still lacks `allowStateAccess` or a documented state-only `agent_end` payload; the existing sanitized agent-event subscription path is relevant, but the discussion raises contract, cadence, and permission-boundary questions that need maintainer direction.

## What This Changes

Request a state-only Plugin SDK permission or contract so external plugins can observe `agent_end`-style lifecycle state without granting raw conversation access.

## Best Possible Solution

Define an additive Plugin SDK state-only lifecycle contract, or explicitly bless and document the existing sanitized lifecycle subscription, while keeping raw `agent_end.messages` behind `allowConversationAccess`.

## Reproduction Assessment

Yes. Source inspection on current main shows `agent_end` is in `CONVERSATION_HOOK_NAMES`, non-bundled typed hooks in that set are rejected without `plugins.entries.<id>.hooks.allowConversationAccess=true`, and loader coverage asserts that behavior.

## Solution Assessment

Unclear. A state-only path is reasonable, but maintainers should choose between documenting the existing sanitized lifecycle subscription, adding a sanitized turn-ended hook, or adding `allowStateAccess`; exposing current `agent_end.messages` under a weaker flag would not be safe.
