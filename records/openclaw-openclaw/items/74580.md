## Summary

Keep open. Current main still requires full conversation access for non-bundled `agent_end`, the `agent_end` payload still carries raw `messages`, and there is no `allowStateAccess` or documented state-only turn-ended plugin contract. The linked OpenClaw Interaction Bridge is useful evidence of the affected external-plugin use case, but the remaining fix is a security-sensitive public Plugin SDK decision rather than a stale cleanup item.

## What This Changes

Request a state-only plugin hook permission or equivalent lifecycle signal so external plugins can observe agent turn completion without granting conversation-content access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive state-only turn-ended/lifecycle Plugin SDK contract that exposes only run/session/state/timing fields, with contract tests proving no prompt, response, history, or message array reaches state-only consumers.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is rejected on current main, and `src/plugins/loader.test.ts:5223` codifies that behavior.

## Solution Assessment

Unclear. A new `allowStateAccess` flag is plausible, but reusing the existing `agent_end` payload under that flag would be unsafe because it includes `messages`; a sanitized event or explicitly documented lifecycle subscription contract is the safer maintainable direction.
