## Summary

Keep open. Current main still blocks non-bundled `agent_end` without `allowConversationAccess`, and that remains security-relevant because the hook payload includes raw `messages`; the remaining request is a public Plugin SDK permission/API design decision, not stale cleanup.

## What This Changes

The issue requests a state-only plugin hook permission or equivalent lightweight lifecycle signal so external plugins can observe agent turn completion without full conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive state-only turn-ended/lifecycle SDK contract that exposes only `runId`, `sessionKey`, phase, timing, and terminal status, with tests proving no prompts, responses, history, tool payloads, or `messages` reach state-only consumers.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is blocked by `registerTypedHook`, and `src/plugins/loader.test.ts` codifies that behavior.

## Solution Assessment

Unclear. `allowStateAccess` is plausible, but reusing the current `agent_end` payload would be unsafe because it includes `messages`; a sanitized event or explicitly supported lifecycle subscription contract is the safer direction.
