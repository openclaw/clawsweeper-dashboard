## Summary

Keep open. Current main still treats `agent_end` as a raw-conversation hook and blocks non-bundled registration without `allowConversationAccess`, while the existing sanitized agent-event subscription surface is only a possible workaround, not a documented lightweight turn-ended/state-only Plugin SDK contract.

## What This Changes

Request a state-only agent lifecycle permission or documented turn-ended plugin contract so external plugins can observe turn completion without raw conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive state-only lifecycle/turn-ended Plugin SDK contract that exposes only run/session/state/timing fields, with tests proving no prompt, response, history, or message array reaches state-only consumers.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is rejected by the current registry path, and `src/plugins/loader.test.ts` asserts that behavior.

## Solution Assessment

Unclear. `allowStateAccess` is plausible, but applying it to the current `agent_end` event would be unsafe because that event contains `messages`; a sanitized lifecycle/turn-ended event or explicitly documented `registerAgentEventSubscription` contract is the safer direction.
