## Summary

Keep open. Current main still gates raw `agent_end` behind `allowConversationAccess`, no `allowStateAccess` or state-only `agent_end` payload exists, and the linked plugin use case shows a real need for turn-end state without raw conversation access.

## What This Changes

Add a state-only Plugin SDK lifecycle or `agent_end` access path so external plugins can observe turn completion without granting raw conversation access.

## Best Possible Solution

Define an additive, documented Plugin SDK turn-ended or lifecycle contract that exposes only session/run/state metadata while keeping raw `agent_end.messages` behind `allowConversationAccess`.

## Reproduction Assessment

Yes for the current limitation. Source inspection and the loader regression test show a non-bundled plugin registering `agent_end` is blocked unless `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Solution Assessment

Unclear. `allowStateAccess` is plausible, but maintainers should first choose whether the durable contract is a sanitized `agent_end`, a new lifecycle hook, or explicit lifecycle subscription guidance.

## Security Review

Status: needs_attention

Summary: The requested feature changes a plugin permission boundary and must preserve the raw conversation gate.

Concerns:

- **[medium] Keep state-only access away from raw messages:** `src/plugins/hook-types.ts:272`
  - body: `agent_end` currently carries `messages`, so any state-only permission must split or sanitize the payload before allowing non-bundled plugins without `allowConversationAccess`.
  - confidence: 0.97
- **[medium] Constrain lifecycle subscription guidance:** `src/plugins/host-hooks.ts:106`
  - body: Agent event subscriptions can receive all streams when `streams` is omitted, so a state-tracking recipe should require `streams: ["lifecycle"]` and avoid assistant/tool/content streams as a permission bypass.
  - confidence: 0.84
