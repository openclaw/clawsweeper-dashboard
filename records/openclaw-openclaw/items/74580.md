## Summary

Keep open: current main and v2026.6.8 still treat `agent_end` as a raw conversation hook with no state-only permission surface, while the reporter's follow-up narrows the useful request to state-only `agent_end` access. This remains a security-sensitive plugin API/product decision, not conservative cleanup.

## What This Changes

The issue requests a state-only permission or sanitized lifecycle contract so external plugins can observe agent turn completion without receiving raw conversation content.

## Best Possible Solution

Have maintainers choose and document a secure plugin contract: `allowStateAccess` with `agent_end.messages` omitted, a separate state-only plugin hook/event, or an explicitly documented lifecycle subscription path, with schema, docs, and contract tests aligned.

## Reproduction Assessment

Yes, source-based. Current main and v2026.6.8 still classify `agent_end` as a conversation hook, expose `messages`, block non-bundled registration unless `allowConversationAccess=true`, and have no `allowStateAccess` schema/config surface.

## Solution Assessment

Unclear until maintainers choose the contract. `allowStateAccess` is plausible, but a sanitized `agent_end`, a new state-only hook, or a documented lifecycle subscription path could be better depending on the desired plugin security model.
