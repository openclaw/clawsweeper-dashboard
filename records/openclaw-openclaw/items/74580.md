## Summary

Keep open. Current main still blocks non-bundled `agent_end` unless `allowConversationAccess=true`, the existing `agent_end` payload still includes raw `messages`, and there is no state-only permission or documented lightweight turn-ended plugin contract. The linked hardware/display plugin is useful evidence of real demand, but this is a security-sensitive plugin API boundary that needs maintainer contract choice before implementation.

## What This Changes

Request a granular state-only permission or equivalent lifecycle/turn-ended hook so plugins can observe agent turn completion/state without raw conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive sanitized lifecycle/turn-ended contract that exposes only fields such as run id, session key, phase, timing, and success/error state, with contract tests proving no conversation text reaches state-only consumers.

## Reproduction Assessment

Yes. Source and test inspection provide a high-confidence reproduction path: a non-bundled plugin registering `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is blocked, and `src/plugins/loader.test.ts:6795` codifies that behavior.

## Solution Assessment

Unclear. The proposed `allowStateAccess` flag is plausible, but because the current `agent_end` event includes raw `messages`, the safer maintainable solution is a sanitized turn-ended/lifecycle contract or a clearly documented event subscription path rather than widening the existing payload.

## Risks / Open Questions

- A state-only permission implemented by simply allowing the current `agent_end` hook would expose `messages` to non-bundled plugins without full conversation access.
- The right public contract is unsettled: `allowStateAccess`, a sanitized `agent_end` variant, or documented `registerAgentEventSubscription` lifecycle semantics each have different compatibility and performance tradeoffs.
- The per-run versus per-turn cadence and global-session behavior of the lifecycle event subscription need explicit documentation before it can fully replace `agent_end` for this use case.
