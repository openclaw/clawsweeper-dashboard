## Summary

Keep open. Current main still has the reported permission gap: non-bundled `agent_end` remains gated as a conversation hook, its payload still contains raw `messages`, and there is no `allowStateAccess` or documented state-only turn-ended hook. The linked OpenClaw Interaction Bridge example is a valid plugin use case, but changing this is security-sensitive public plugin API work, so a maintainer should choose the sanitized lifecycle contract before implementation.

## What This Changes

Add a state-only permission or event path so non-bundled plugins can observe agent turn completion without granting full conversation hook access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document an additive state-only turn-ended lifecycle contract that exposes only safe fields such as run id, session key, phase, timing, success, and error summary, with tests proving no conversation text reaches state-only consumers.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is rejected by current `registerTypedHook` behavior, and `src/plugins/loader.test.ts:6795` codifies that path.

## Solution Assessment

Unclear for the exact proposed `allowStateAccess` flag. A state-only contract is the right direction, but reusing the current `agent_end` event under a weaker permission is not safe because that payload carries `messages`; a sanitized event or documented lifecycle subscription contract is safer.
