## Summary

Close: current main and v2026.6.5 already route `/clear` and `/reset` through Gateway reset, rotate/archive the active transcript, clear non-subagent CLI resume bindings, and avoid re-reading archived transcript files as active prompt context.

## What This Changes

The issue asks for `/clear` and `/reset` to stop stale `.jsonl` transcripts from being reused after session clearing because they appear to drive unexpected token spend.

## Best Possible Solution

Keep the shipped reset behavior: rotate the OpenClaw session file, archive old transcripts, clear non-subagent CLI resume bindings, and treat hard-delete retention as a separate policy only if new evidence shows archived files are still re-ingested.

## Reproduction Assessment

No; I did not reproduce the reported failure on current main. Source inspection and existing regression tests show current `/clear` and `/reset` paths rotate/archive transcripts and clear stale non-subagent CLI resume bindings.

## Solution Assessment

Yes; the current fix is the maintainable boundary because reset updates Gateway session state, rotates the active transcript, clears provider resume state, and leaves archived transcripts available without making them active prompt context.
