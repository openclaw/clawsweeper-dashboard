## Summary

Keep open. Current main disproves the simplest version of the report because normal `/new` and `/reset` rotate the active session to a new transcript and archive the old one, while `/clear` is not a default reset trigger; however, related source paths and open work show archived transcripts can still be re-ingested by memory/Dreaming surfaces, so this should be clarified or routed rather than closed.

## What This Changes

The issue reports that `/clear` or `/reset` leave stale `.jsonl` transcript files that are later included in prompts, causing unnecessary token use.

## Best Possible Solution

First narrow the affected surface: if the token burn is memory-core Dreaming archive ingestion, route it through https://github.com/openclaw/openclaw/issues/90313 and its PRs; if ordinary chat prompts include old history after `/reset`, collect a concrete session store/transcript/prompt proof and fix the active prompt assembly or session lookup path without deleting retained archives.

## Reproduction Assessment

Partly. Source inspection reproduces an archived reset/deleted transcript ingestion path for memory-core Dreaming, but I did not find proof that ordinary `/reset` chat prompts attach the old active `.jsonl`; the issue needs the exact surface, config, and prompt evidence.

## Solution Assessment

Unclear. Deleting transcript archives is not the best default because current code intentionally archives them for usage, hooks, and history; the maintainable fix should target the confirmed ingestion or prompt-assembly path after the surface is narrowed.
