## Summary

Keep open: the PR is useful, but the latest head still compares raw trimmed agent IDs while cron create/update normalize IDs through the shared agent-id normalizer. That can make the new filter miss equivalent jobs, especially mixed-case default-agent jobs, before this lands.

## What This Changes

This PR adds an optional `agentId` filter to `cron.list`/cron service listPage, threads it through the gateway handler, adds regression tests, and updates the changelog.

## Best Possible Solution

Normalize the requested filter and effective job owner with the same `normalizeAgentId` contract used by cron create/update, keep omitted `agentId` backward-compatible, and preserve #77118 for tool/CLI follow-ups unless maintainers explicitly decide the gateway foundation is enough.

## Reproduction Assessment

Yes. Source inspection on PR head shows the new filter is only trimmed before comparison, while current cron create/update paths normalize agent IDs; a request such as `cron.list({ agentId: "MAIN" })` can miss jobs stored as `main`.

## Solution Assessment

No. The additive gateway field is the right shape, but matching should reuse the existing agent-id normalizer before merge so the new API follows the same ownership semantics as job creation and updates.
