## Summary

Keep open: the additive gateway/API direction is sound, but the proposed service predicate would hide valid default-agent jobs whose stored cron records omit `agentId`, and the PR still risks closing the broader #77118 follow-up scope early.

## What This Changes

The PR adds an optional `agentId` field to cron list paging and gateway `cron.list`, filters `listPage` by that field, adds regression tests, and updates the changelog.

## Best Possible Solution

Normalize both the requested filter and each job's effective owner, falling back missing job owners to the cron service default agent, then keep the tool/CLI follow-ups tracked separately if they remain outside this PR.

## Reproduction Assessment

Yes, by source inspection: current main allows main-session default-agent jobs with `agentId` omitted, and the PR predicate would filter those out for `cron.list({ agentId: "main" })`. I did not execute tests in this read-only review.

## Solution Assessment

No. The schema and handler threading are the right narrow direction, but the service filter must use the existing default-agent fallback semantics before this is safe to merge.
