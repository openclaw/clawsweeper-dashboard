## Summary

Keep this PR open: the gateway-side `agentId` filter is the right foundation, but the current head still compares raw agent-id strings where existing cron paths normalize IDs, and the changelog names a non-existent `cron.list_page` API. The linked #77118 issue is also broader than this gateway-only patch because tool auto-fill and CLI scoping remain follow-ups.

## What This Changes

The PR adds an optional `agentId` parameter to the cron list protocol, gateway handler, and cron service list pagination, with service tests and a changelog entry.

## Best Possible Solution

Land the additive gateway filter after normalizing every compared agent ID with the existing helper, fixing the changelog wording, and keeping #77118 open or clearly scoped for the remaining tool and CLI work.

## Reproduction Assessment

Yes. Source inspection on current main shows no `agentId` field in the cron list schema, no handler plumbing, and no owner filter in `listPage`; the PR diff also makes the raw-comparison regression clear without executing tests.

## Solution Assessment

No, not as written. The additive protocol/service direction is maintainable, but the filter should use the same agent-id normalization as create/update paths and the changelog should name only the real `cron.list` API.

## Review Findings

Overall correctness: patch is incorrect
