## Summary

Keep this PR open: it adds the right gateway-side shape for scoped cron listing, but the latest patch still compares the new `agentId` filter as a raw trimmed string while existing cron ownership paths normalize agent IDs before storing or comparing them.

## What This Changes

The PR adds an optional `agentId` field to the gateway `cron.list` contract, threads it into cron `listPage`, adds list filtering tests and Swift protocol artifacts, and records a changelog entry.

## Best Possible Solution

Normalize both the requested filter and each effective job owner with the existing agent-id helper, then land the additive protocol change while keeping any remaining tool/CLI default-scoping work explicitly tracked.

## Reproduction Assessment

Yes, from source inspection: current main rejects `agentId` in `CronListParamsSchema` and `cron.list`/`listPage` never apply an owner predicate. I did not run tests because this review is read-only and `pnpm` is not available on PATH in this checkout.

## Solution Assessment

No, not yet. The additive gateway field is the right narrow direction, but the filter should use the same `normalizeAgentId` semantics as cron create/update and default-agent checks before comparing owners.

## Review Findings

Overall correctness: patch is incorrect
