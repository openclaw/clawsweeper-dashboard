## Summary

Keep this PR open: the additive gateway-side `agentId` filter is the right general direction for the related multi-agent cron issue, but the current filter would hide valid default-agent jobs that have no stored `agentId`.

## What This Changes

This PR adds an optional `agentId` field to the `cron.list` gateway schema/handler and cron service `listPage` filtering, with regression tests and a changelog entry.

## Best Possible Solution

Keep the additive gateway contract, but make `agentId=<defaultAgentId>` include jobs whose stored `agentId` is missing and add regression coverage for explicit and implicit default-agent jobs.

## Reproduction Assessment

Yes, from source inspection: current main lacks the requested `agentId` list filter, and the PR's proposed comparison would drop default-agent jobs that current code allows to omit `agentId`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The additive gateway parameter is a maintainable boundary, but the service filter needs to compare against the effective job agent, treating missing `job.agentId` as the cron service default agent.

## Review Findings
