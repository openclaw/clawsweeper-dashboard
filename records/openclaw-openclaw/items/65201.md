## Summary

Keep open. Current main still has both doctor Gateway auth paths warning on any SecretRef-backed `gateway.auth.token` before using the SecretRef-aware resolver, and the overlapping repair PRs are open rather than merged.

## What This Changes

The issue reports that `openclaw doctor` warns that a SecretRef-backed Gateway auth token is unavailable even when secrets audit, reload, and Gateway runtime auth show the token resolves.

## Best Possible Solution

Land one focused fix, likely via the open repair PRs or a consolidated replacement, that makes both interactive and lint doctor Gateway auth checks use the SecretRef-aware token resolver, keeps unresolved refs read-only, and covers resolved env/file refs plus explicit exec semantics.

## Reproduction Assessment

Yes, at source level. Configure local token auth with a resolvable `gateway.auth.token` SecretRef and run doctor; current main detects the ref, then `resolveGatewayAuth()` drops SecretRef values and the warning branch remains reachable.

## Solution Assessment

Mostly yes. Reusing the existing SecretRef-aware Gateway token resolver in both doctor paths is the narrow maintainable fix; exec SecretRef wording or opt-in behavior should be consolidated before merge.
