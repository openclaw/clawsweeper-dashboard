## Summary

Keep open. Current main still sends doctor Gateway auth through the generic resolver that does not resolve configured SecretRef values, then emits the reported unavailable warning when `gateway.auth.token` is a SecretRef and no plaintext token is found.

## What This Changes

Request that `openclaw doctor` stop warning that a SecretRef-managed Gateway auth token is unavailable when the configured SecretRef resolves successfully.

## Best Possible Solution

Land a focused target-repo fix, possibly split from #67463, that resolves `gateway.auth.token` through `resolveGatewayAuthTokenForService` before doctor emits the unavailable warning, while preserving the warning for genuinely unresolved SecretRefs and adding a doctor regression test.

## Reproduction Assessment

Yes. Configure local token auth with a resolvable file or env SecretRef for `gateway.auth.token`, avoid a plaintext token path that satisfies `resolveGatewayAuth`, then run doctor; the current static path still reaches the warning branch without resolving the configured SecretRef.

## Solution Assessment

No current-main solution is in place. The maintainable fix is to use the existing SecretRef-aware Gateway token helper in `runGatewayAuthHealth`, not only the generic auth resolver.
