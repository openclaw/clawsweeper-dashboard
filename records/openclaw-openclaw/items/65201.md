## Summary

Keep open. Current main still runs doctor Gateway auth through `resolveGatewayAuth`, which treats a configured SecretRef token as absent, then emits the reported unavailable warning before using the existing SecretRef-aware token resolver; an open PR already contains the focused repair.

## What This Changes

The issue reports that `openclaw doctor` warns that a SecretRef-managed `gateway.auth.token` is unavailable even when secrets audit and gateway runtime can resolve it.

## Best Possible Solution

Review and land the existing doctor-only SecretRef fix so `runGatewayAuthHealth` resolves `gateway.auth.token` through the SecretRef-aware helper before warning, while preserving the warning for genuinely unresolved SecretRefs.

## Reproduction Assessment

Yes. Configure local token auth with a resolvable env or file SecretRef for `gateway.auth.token` and run `openclaw doctor`; source inspection shows current main reaches the warning branch without resolving that SecretRef first.

## Solution Assessment

Yes, the open PR's approach is the narrowest maintainable direction: use the existing SecretRef-aware Gateway token helper only on the token-needed doctor path, then keep the current warning for unresolved SecretRefs.
