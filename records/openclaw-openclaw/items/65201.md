## Summary

Keep open. Current main still emits the Gateway auth SecretRef unavailable warning before materializing a resolvable SecretRef-backed token, and active PRs already cover the repair area.

## What This Changes

Doctor reports a Gateway auth SecretRef token as unavailable even when secrets audit and the gateway runtime can resolve it.

## Best Possible Solution

Land one focused doctor fix that resolves safe Gateway token SecretRefs only after token auth is required, preserves read-only behavior for unresolved refs, and reports exec refs as opt-in or unverified rather than unavailable.

## Reproduction Assessment

Yes. Configure local token auth with a resolvable env or file SecretRef for `gateway.auth.token` and run `openclaw doctor`; current source detects the SecretRef, then `resolveGatewayAuth()` ignores SecretRef values and reaches the warning branch.

## Solution Assessment

Mostly yes. Reusing the existing SecretRef-aware Gateway token resolver after `needsToken` is the narrow fix for env/file refs, while exec refs should use an explicit opt-in or unverified diagnostic rather than implicit command execution.
