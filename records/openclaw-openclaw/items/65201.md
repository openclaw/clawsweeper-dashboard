## Summary

Keep open. Current main still contains the reported false-positive path: doctor sees a configured Gateway auth SecretRef, resolves auth through the non-SecretRef-aware `resolveGatewayAuth` path, and emits the unavailable warning even though a configured SecretRef can resolve through the existing Gateway token helper. The latest shipped release snapshot has the same branch, and the related implementation PRs are not merged fixes in the target repository.

## What This Changes

Suppress the doctor Gateway auth warning when `gateway.auth.token` is SecretRef-managed but resolvable through the configured secrets provider.

## Best Possible Solution

Land a focused target-repo fix, possibly split from #67463, that resolves `gateway.auth.token` through the existing SecretRef-aware Gateway token helper before doctor emits the unavailable warning. Keep the warning for genuinely unresolved SecretRefs and add a doctor regression test for a resolvable env or file SecretRef.

## Reproduction Assessment

Yes. Configure local Gateway token auth with `gateway.auth.token` as a resolvable configured SecretRef and no ambient `OPENCLAW_GATEWAY_TOKEN`, then run doctor; the current static path still reaches the unavailable warning because `runGatewayAuthHealth` does not call the SecretRef-aware resolver.

## Solution Assessment

No current-main solution is in place. Passing only `env` through `resolveGatewayAuth` would cover some env-token cases but still misses configured providers such as file refs; the maintainable fix is to call `resolveGatewayAuthToken` or the service helper before deciding the SecretRef is unavailable.
