## Summary

Keep open: the scoped provider-side AbortError fix is directionally right, but the shared guard now depends on callers threading `abortSignal`, and the isolated-cron fallback caller still does not do that.

## What This Changes

The PR narrows model fallback AbortError rethrowing, forwards compaction abort signals into fallback, and adds model-fallback regression tests.

## Best Possible Solution

Thread `params.abortSignal` through the isolated-cron `runWithModelFallback` call and add a focused regression before merge; keep broader terminal-abort work in a separate maintainer decision path.

## Reproduction Assessment

Yes, source-reproducible. The PR guard falls through when `params.abortSignal` is absent, and current cron code can throw an `AbortError` from an internally aborted embedded run while not forwarding the signal to `runWithModelFallback`.

## Solution Assessment

No, not yet. The shared fallback boundary is the right layer, but the best fix must update every production caller that carries a cancellation signal; isolated cron remains unthreaded.
