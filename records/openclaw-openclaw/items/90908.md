## Summary

Keep this PR open: the provider-side AbortError retry change is pointed at the right fallback boundary, but the latest cron follow-up still does not stop real cron timeout/cancellation paths because the forwarded signal is not recognized as terminal for the string/plain-Error cron abort shape.

## What This Changes

The PR changes model fallback abort classification, forwards abort signals for compaction and isolated cron fallback calls, and adds focused regression tests.

## Best Possible Solution

Keep the narrow provider-side AbortError repair, but make cron cancellation produce or map to a terminal fallback-abort shape and prove an aborted cron run cannot reach a second candidate.

## Reproduction Assessment

Yes, source-reproducible. On the PR head, cron forwards the signal but still throws a plain `Error(params.abortReason())`; current fallback only treats named terminal Error reasons or AbortError-with-aborted-signal as terminal before timeout normalization.

## Solution Assessment

No, not yet. The shared fallback boundary is the right fix location, but the cron part must pass a terminal abort shape or expand terminal detection for canonical cron timeout reasons before this is the best fix.
