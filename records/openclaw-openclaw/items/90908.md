## Summary

Keep open: the provider-side AbortError fallback fix is useful, but the PR is not merge-ready because the cron cancellation path still throws a plain timeout-text Error that can be normalized as retryable fallback, and the supplied proof does not show cron cancellation stopping before another candidate runs.

## What This Changes

The PR changes model fallback abort classification, forwards abort signals through compaction and isolated cron fallback calls, and adds focused regression tests.

## Best Possible Solution

Keep the provider-side AbortError guard, make cron cancellation produce or map to a terminal fallback-abort shape, and add proof/tests that a cancelled cron run cannot reach a second candidate.

## Reproduction Assessment

Yes, source-reproducible. On the PR head, cron forwards the abort signal but still throws a plain Error from abortReason(), while current fallback normalization treats timeout text as retryable failover.

## Solution Assessment

No, not yet. The shared fallback boundary is the right place for the provider-side AbortError repair, but the cron cancellation part must use terminal abort detection before this is the best fix.
