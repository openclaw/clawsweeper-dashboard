## Summary

Keep this PR open: it fixes the reported provider-side AbortError path in one direction, but the new no-fallback classifier also treats upstream abort-signal cancellations as provider aborts because it only checks `replyOperation.result`; real after-fix behavior proof is still insufficient.

## What This Changes

This PR changes model fallback and external-run failure handling so provider-side AbortErrors can try fallback models or surface a visible group/channel error instead of being treated as user cancellations.

## Best Possible Solution

Preserve cancellation semantics by checking the operation abort signal/result through a shared guard, then add redacted after-fix runtime proof for the provider-abort path.

## Reproduction Assessment

Yes. Source inspection shows current main rethrows plain provider AbortError before fallback and then silences generic group/channel failures; the PR also has unit-level coverage for the intended provider-abort path.

## Solution Assessment

No. The fallback change is plausible, but the no-fallback classifier should distinguish provider aborts from upstream abort-signal cancellations before this is the best fix.
