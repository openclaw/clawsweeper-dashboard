## Summary

Keep open: the patch helps the configured-fallback success case, but it does not yet cover the no-fallback or final-candidate provider AbortError path that can still become a generic group/channel failure, and the external PR only provides unit-test proof.

## What This Changes

The PR changes model fallback AbortError handling to rethrow only when the external abort signal is already aborted and adds unit coverage for user aborts versus provider-side AbortErrors with a configured fallback.

## Best Possible Solution

Classify provider-side AbortErrors without an external abort as retryable or visible provider failures even when no fallback remains, while preserving true user/gateway abort rethrow and covering both fallback and no-fallback cases.

## Reproduction Assessment

Yes. Source inspection shows a provider-side AbortError with no successful fallback still reaches the last-candidate raw throw and then the generic group/channel silent-reply path; I did not run tests because this was a read-only review.

## Solution Assessment

No. Gating on the external abort signal is a useful part of the fix, but it is too narrow because the final/no-fallback provider AbortError path remains generic and silent, and the new test only proves the configured-fallback success case.
