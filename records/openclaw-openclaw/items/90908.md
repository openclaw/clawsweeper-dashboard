## Summary

Keep this PR open: the core bug is real, but the latest patch introduces a blocking over-broad timeout classification in the FallbackSummaryError path and still lacks concrete after-fix real behavior proof from a patched runtime.

## What This Changes

The PR changes model fallback and external-run failure handling so provider-side AbortErrors can fall through to fallback or visible group/channel error replies instead of being treated as cancellations.

## Best Possible Solution

Keep the fallback rethrow guard, narrow exhausted-summary detection to abort-text failures with a regression for ordinary timeouts, then require redacted after-fix channel/runtime proof before merge.

## Reproduction Assessment

Yes for source reproduction: current main rethrows non-timeout AbortErrors before fallback and generic non-direct failures resolve to SILENT_REPLY_TOKEN. The PR body also includes redacted before-fix trajectory logs, but I did not run a live provider/channel repro in this read-only pass.

## Solution Assessment

No as currently written: the fix is in the right general layer, but the FallbackSummaryError branch treats any all-timeout summary as provider-side abort. The safer solution is to require abort-like error text for timeout attempts and preserve existing generic timeout silence unless maintainers intentionally change that policy.
