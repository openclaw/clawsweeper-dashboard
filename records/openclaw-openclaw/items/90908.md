## Summary

The final comparison found a concrete blocker: current main already retries the reported `AbortError("This operation was aborted")` because it normalizes to a timeout failover, while this PR removes the remaining guard for unclassified raw `AbortError`s and therefore broadens retry behavior beyond the demonstrated bug. The useful cancellation repair should stay, but the raw-abort rethrow contract should be preserved unless maintainers explicitly approve the broader semantics.

## What This Changes

The PR adds a pre-normalization external-cancellation guard, removes the existing unclassified raw-abort rethrow, forwards compaction’s abort signal, and adds model-fallback regression tests.

## Best Possible Solution

Keep the pre-normalization `AbortError` plus `abortSignal.aborted` cancellation guard and compaction signal forwarding, then retain the existing post-normalization rethrow for raw AbortErrors that do not coerce to a known failover reason.

## Reproduction Assessment

Yes for the actual cancellation gap: abort the caller signal with an `AbortError` reason and reject the active model attempt with an abort-shaped error; current main can normalize it and continue fallback because `isTerminalAbort` does not recognize that reason. The claimed provider exact-message failure itself is not reproduced because current main already normalizes that message into fallback.

## Solution Assessment

No as currently written. The pre-normalization external-cancellation guard and compaction forwarding are the right repair, but deleting the post-normalization guard changes unrelated raw-abort semantics without proof.
