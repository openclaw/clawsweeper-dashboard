## Summary

Keep open: the shared fallback fix is useful and the supplied terminal proof is acceptable, but the latest head still leaves the isolated-cron outer fallback caller without the abort signal that the new guard depends on.

## What This Changes

The PR changes model fallback AbortError classification, forwards compaction abort signals into fallback, and adds focused model-fallback regression coverage.

## Best Possible Solution

Forward `params.abortSignal` into the isolated cron `runWithModelFallback` call and add a focused cron regression that fails if an aborted cron run advances to another fallback candidate.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the new guard depends on `params.abortSignal?.aborted`, and the isolated cron outer `runWithModelFallback` call on the PR head does not pass `params.abortSignal`. The inner cron runners receive the signal, so this is a boundary omission rather than missing input state.

## Solution Assessment

No, the shared helper change is plausible but too narrow as submitted. The maintainable fix is to update each cancellation-aware fallback caller, and the cron caller is still missing that handoff.
