## Label Justifications

- `P1`: The PR targets and risks core agent/channel reply behavior where fallback mistakes can silence or incorrectly continue user-visible runs.
- `merge-risk: 🚨 message-delivery`: A cancellation-shaped AbortError can now continue to another fallback candidate and potentially produce a reply after the original operation should have stopped.
- `merge-risk: 🚨 availability`: Continuing fallback after a cancellation-shaped AbortError can waste provider calls and keep agent work running after cancellation.

## Summary

Keep this PR open, but do not merge it as-is: the intended provider-side abort retry is plausible, yet the patch also drops the existing plain no-signal AbortError stop path and can continue fallback after cancellation-shaped errors in callers without an abort signal.

## What This Changes

The PR narrows model fallback AbortError rethrowing to aborted external signals, forwards compaction abortSignal into model fallback, and adds model-fallback regression tests.

## Best Possible Solution

Keep provider-side abort-text retry behavior while preserving plain no-signal AbortError cancellation, either by retrying only classified provider/timeout aborts or by threading abortSignal through every fallback caller before relaxing the guard.

## Reproduction Assessment

Yes from source inspection, not from a local test run. Current main has a no-signal AbortError non-fallback regression test, while the PR rewrites that case to require an aborted signal and leaves runWithImageModelFallback without any signal path.

## Solution Assessment

No. The provider-side retry direction is reasonable, but this implementation is too broad because it removes a current cancellation guard instead of limiting fallback to explicitly classified provider/timeout aborts.
