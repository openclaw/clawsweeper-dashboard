## Summary

Keep open, but this PR is not mergeable as written. Current main already includes the config-schema part via https://github.com/openclaw/openclaw/pull/71247, while the remaining runtime cap behavior is still useful and now overlaps https://github.com/openclaw/openclaw/issues/71136; this branch misses compaction/extension paths, has an override-precedence bug, and lacks real behavior proof.

## What This Changes

The PR adds `agents.list[].contextTokens`, teaches `resolveContextWindowInfo` about an optional `agentId`, passes it in the embedded runner, and adds context-window guard tests.

## Best Possible Solution

Centralize positive finite token-limit resolution with source metadata, clamp known agent/model/discovery limits consistently, and feed that result through run, compaction, extension factories, session persistence, and status while keeping the accepted schema/docs contract from current main.

## Reproduction Assessment

Not applicable as a user bug reproduction, but the patch defects are source-reproducible. The branch only threads agent context through the main run path, while compaction and extension budget paths still lack the agent-specific cap and the guard can fall back to defaults despite a per-agent value.

## Solution Assessment

No, this is not the best way to solve it as written. The maintainable direction is a shared resolver and full runtime call-site threading, coordinated with https://github.com/openclaw/openclaw/issues/71136 rather than a partial `agentId` patch.
