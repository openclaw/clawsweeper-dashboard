## Summary

Keep this PR open: the per-agent contextTokens direction has useful user support, but current main only has the schema/config half and this branch still leaves precedence plus compaction/extension budget paths incomplete.

## What This Changes

The PR adds `agents.list[].contextTokens` schema support, adds an optional `agentId` to context-window resolution, passes it through one embedded-runner path, and adds resolver tests.

## Best Possible Solution

Land a rebased implementation that treats explicit `agents.list[].contextTokens` as the authoritative per-agent override and threads the effective session agent id through main run setup, direct and queued compaction, context-pruning, and compaction-safeguard setup.

## Reproduction Assessment

Yes for the review findings. Source inspection of current main and the PR diff shows the patch passes agent identity through only one runtime path and lets `agents.defaults.contextTokens` override a larger per-agent value.

## Solution Assessment

No. The requested direction is sound, but the patch is incomplete; the safer solution is a rebased agent-aware budget resolver used by every context-budget consumer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread agent ids through every context budget path:** `src/agents/pi-embedded-runner/run.ts:392`
  - body: Passing `agentId` only into the main run resolver leaves direct compaction, queued/context-engine compaction, context-pruning, and compaction-safeguard setup on agent-blind resolution. Those paths still size budgets from defaults/model metadata, so an agent-specific cap is ignored when compaction or pruning thresholds are computed.
  - confidence: 0.88
- **[P2] Treat per-agent contextTokens as the override:** `src/agents/context-window-guard.ts:50-52`
  - body: The new lookup returns the per-agent value only when it is lower than the base window; otherwise it falls through to `agents.defaults.contextTokens`. With defaults=100k, agent=196k, and model=180k, the resolver still returns 100k, so an agent cannot replace a stricter global cap.
  - confidence: 0.9
