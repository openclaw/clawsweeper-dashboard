## Summary

Keep open. The PR is aimed at a useful per-agent context-budget feature, but current `main` only has the schema/persistence portion while this branch still applies the runtime cap in only one path and has unresolved precedence/contract gaps.

## What This Changes

The PR adds `agents.list[].contextTokens` schema support, threads an optional `agentId` into one embedded-runner context-window resolution call, and adds guard tests for per-agent/default precedence.

## Best Possible Solution

Keep the accepted schema from current main, then centralize positive finite token-budget resolution with source metadata and feed the same result through run, compaction, extension factories, status/session metadata, and related max-token paths.

## Reproduction Assessment

Not applicable as a user-bug reproduction, but the patch defects are source-reproducible. Source inspection shows the branch only threads agent identity through one run path while compaction and extension budgets still lack a per-agent cap, and the resolver can still fall through to a stricter default despite a per-agent value.

## Solution Assessment

No, this is not the best way to solve the issue as written. The maintainable path is a shared resolver plus full runtime call-site threading, coordinated with https://github.com/openclaw/openclaw/issues/71136 rather than a partial one-call-site override.
