## Summary

Keep open for now: the config-schema part was later landed by https://github.com/openclaw/openclaw/pull/71247, but this PR still proposes a distinct embedded-runner context-window behavior and is not safe to merge as written. The patch leaves per-agent caps out of compaction/extension paths, treats the per-agent value as only a lower cap instead of an override over `agents.defaults.contextTokens`, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `agents.list[].contextTokens`, threads `agentId` into one embedded-runner context-window guard call, and adds unit coverage for per-agent/default cap resolution.

## Best Possible Solution

Rebase on current main, keep the shipped schema/config-contract implementation, and rework the runtime path so one resolved per-agent context budget is applied consistently to embedded run, explicit compaction, context pruning, and compaction safeguard setup.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR defects: a config with `agents.defaults.contextTokens` lower than a matching agent’s `contextTokens` would still fall back to the default cap, and current compaction/extension call sites do not receive agent identity. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is useful, but this patch is not the best implementation until it reuses the already-merged schema support and carries the resolved per-agent budget through every context-window consumer instead of patching only one runner call site.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Make the per-agent value override the default cap:** `src/agents/context-window-guard.ts:49-50`
  - body: The new branch only returns `contextTokens` from the matching agent when it is lower than the model/base window. If the global default is stricter, a larger per-agent value still falls through and gets capped by `agents.defaults.contextTokens`, so the per-agent override cannot actually override the default as the PR promises.
  - confidence: 0.93
- **[P2] Thread the agent cap through all context-window consumers:** `src/agents/pi-embedded-runner/run.ts:392`
  - body: This updates the main run setup only, but explicit compaction plus context-pruning and compaction-safeguard extension setup still call the context-window resolver without agent identity. A configured per-agent cap would therefore be ignored in the compaction paths that this feature is meant to protect.
  - confidence: 0.88
- **[P2] Regenerate the config contract for the new field:** `src/config/zod-schema.agent-runtime.ts:772`
  - body: Adding a strict config field in `AgentEntrySchema` also needs the generated config schema, exported config type, and generated baseline/docs artifacts kept in sync. Without those updates, the public config contract drifts even if direct Zod parsing accepts the new key.
  - confidence: 0.84
