## Summary

Keep open. The feature remains useful, and user discussion supports the per-agent precedence model, but this branch is incomplete: current main already contains the schema portion, the PR threads agent identity through only one runtime path, and no after-fix real behavior proof is posted.

## What This Changes

The PR adds `agents.list[].contextTokens` schema support, threads `agentId` into one embedded-runner context-window resolver call, and adds resolver tests for per-agent/default precedence.

## Best Possible Solution

Land a rebased implementation that treats explicit `agents.list[].contextTokens` as the per-agent budget source ahead of defaults across run setup, direct and queued compaction, context-pruning, and compaction-safeguard setup, while preserving known model-window safety clamps.

## Reproduction Assessment

Yes for the review findings. Source inspection of the PR diff and current main shows the branch passes agent identity through only one runtime path and lets `agents.defaults.contextTokens` override some explicit per-agent values.

## Solution Assessment

No. The direction is sound, but the patch needs to be rebased onto current main and converted into an agent-aware context-budget resolver used by every runtime, compaction, pruning, and safeguard consumer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread agent ids through every context budget path:** `src/agents/pi-embedded-runner/run.ts:392`
  - body: Passing `agentId` only to this main run resolver leaves direct compaction, queued compaction, context-pruning, and compaction-safeguard setup on agent-blind context-window resolution. Those paths still compute thresholds from defaults or model metadata, so an agent-specific cap is ignored in the flows this feature is meant to protect.
  - confidence: 0.9
- **[P2] Honor explicit per-agent caps over defaults:** `src/agents/context-window-guard.ts:50-52`
  - body: The new lookup returns the per-agent value only when it is lower than the base window, then still applies `agents.defaults.contextTokens`. With a larger explicit per-agent budget and a smaller global default, the default still wins, so the override cannot actually override defaults.
  - confidence: 0.9
