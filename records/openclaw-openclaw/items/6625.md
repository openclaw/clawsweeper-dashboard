## Summary

Keep open. Current main and the latest shipped release still expose only hard sub-agent run timeouts plus post-timeout partial-progress reporting; there is no config key, `sessions_spawn` parameter, native path, or ACP path that warns a child run before the hard timeout.

## What This Changes

The issue requests a configurable pre-timeout warning/grace period for sub-agents so child agents can checkpoint work before `runTimeoutSeconds` aborts them.

## Best Possible Solution

Define a documented sub-agent timeout-warning contract under the existing sub-agent config/tool surfaces, deliver it consistently for native and ACP runs before the hard timeout, and preserve current post-timeout recovery behavior.

## Reproduction Assessment

Not applicable as a runnable bug reproduction; this is a feature/config request. Source and docs show the current supported behavior is a hard run timeout plus post-timeout partial-progress reporting, not a pre-timeout warning.

## Solution Assessment

Yes at the product-direction level, but not as an automatic narrow bug fix. The maintainable solution needs a deliberate native and ACP lifecycle contract rather than a prompt workaround or local timeout wrapper.
