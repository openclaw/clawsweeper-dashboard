## Summary

Keep open. Current main still has hard sub-agent run timeouts, post-timeout recovery, and timeout-only docs/config, but no child-facing pre-timeout warning or grace-period delivery path.

## What This Changes

Request a configurable pre-timeout warning/grace period for sub-agents so they can checkpoint progress before `runTimeoutSeconds` terminates the run.

## Best Possible Solution

Define one lifecycle-owned pre-timeout child notification contract shared by native and ACP sub-agent runs, with conservative default behavior plus focused tests and docs.

## Reproduction Assessment

Yes, at the source level: current main has hard sub-agent deadlines and post-timeout recovery, but no warning input, config field, or scheduled child notification before termination. I did not run a live timeout scenario during this read-only review.

## Solution Assessment

Unclear. A pre-timeout notification directly addresses the reported loss mode, but the standalone config shape is not yet proven to be the narrowest solution; maintainers should choose the shared native/ACP lifecycle policy first.
