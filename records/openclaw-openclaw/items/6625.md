## Summary

Keep open: current main still supports hard sub-agent timeouts and post-timeout partial-progress reporting, but I found no configurable pre-timeout warning/grace surface or scheduled warning delivery path for native or ACP sub-agents.

## What This Changes

Request a configurable warning that reaches a sub-agent before `runTimeoutSeconds` kills the run, so the child can checkpoint progress.

## Best Possible Solution

Define one documented timeout-warning contract and implement deadline-relative warning delivery through shared native and ACP child-session control paths while preserving existing hard-timeout and post-timeout recovery behavior.

## Reproduction Assessment

Not applicable: this is an enhancement request, not a broken documented behavior. Source inspection on current main confirms timeout and post-timeout recovery paths exist, but no pre-timeout warning/grace surface is present.

## Solution Assessment

Unclear until maintainers settle the public config/tool API, default lead time, and child-control semantics. A shared native/ACP warning path is safer than adding ad hoc prompt injection at only one spawn call site.
