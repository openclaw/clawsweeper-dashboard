## Summary

Keep open. Current main and the latest release still provide hard sub-agent run timeouts and post-timeout handling, but they do not expose or schedule a pre-timeout checkpoint warning for native or ACP child agents.

## What This Changes

Request a configurable pre-timeout warning/grace period so sub-agents can checkpoint work before `runTimeoutSeconds` terminates the run.

## Best Possible Solution

Define one maintainer-approved pre-timeout child notification contract shared by native and ACP sub-agent runs, with focused tests and docs while preserving existing hard-timeout and post-timeout recovery behavior.

## Reproduction Assessment

Yes, at the source level: current main has hard `runTimeoutSeconds` and post-timeout handling, but no warning config/tool input or scheduled child notification before termination. I did not run a live timeout scenario during this read-only review.

## Solution Assessment

Unclear as proposed. A pre-timeout child notification directly targets the reported loss mode, but the config/API shape, default policy, and native/ACP delivery semantics need maintainer design before implementation.
