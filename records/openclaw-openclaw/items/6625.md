- `impact:data-loss`: The issue is about losing unsaved code, research, generated content, and other work when a child run is killed at timeout.

## Summary

Keep open. Current main and the latest release still expose only hard sub-agent run timeouts plus post-timeout recovery; there is no config key, `sessions_spawn` parameter, native runtime path, or ACP runtime path that warns the child before the timeout.

## What This Changes

The issue requests a configurable pre-timeout warning/grace period for sub-agents so they can checkpoint work before `runTimeoutSeconds` kills the run.

## Best Possible Solution

Define a documented sub-agent timeout-warning contract on the existing config/tool surfaces, deliver it consistently before the hard timeout in native and ACP runs, and preserve current post-timeout recovery.

## Reproduction Assessment

Not applicable as a runnable bug reproduction because this is a feature/config request. Source inspection gives high confidence that current behavior is hard timeout plus post-timeout capture, not a pre-timeout warning.

## Solution Assessment

Yes at the product-direction level, but not as an automatic narrow bug fix. The maintainable path is a deliberate native and ACP lifecycle contract rather than prompt text or a local timeout wrapper.
