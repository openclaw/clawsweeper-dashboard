- `impact:data-loss`: The request is specifically about avoiding loss of unsaved code, research, generated content, and accumulated work when a child run times out.
- `impact:session-state`: The affected behavior is child-agent session and transcript state being aborted without a pre-timeout checkpoint opportunity.

## Summary

Keep open. Current main and v2026.5.19 still expose hard sub-agent run timeouts and post-timeout partial-progress recovery, but no configurable pre-timeout warning or child-session checkpoint prompt before abort.

## What This Changes

The issue requests a configurable pre-timeout warning for sub-agents so child runs can checkpoint work before `runTimeoutSeconds` kills them.

## Best Possible Solution

Define a documented sub-agent timeout-warning contract, likely under `agents.defaults.subagents` plus an optional `sessions_spawn` override, and deliver it consistently across native and ACP runs while preserving existing post-timeout recovery.

## Reproduction Assessment

Not applicable as a runnable bug reproduction; this is a feature/config request. Source and docs inspection on current main and v2026.5.19 show no schema, docs, or runtime path for a pre-timeout warning.

## Solution Assessment

Yes at the product-direction level, but not as an automatic narrow bug fix. The maintainable solution needs a deliberate lifecycle contract across native and ACP sub-agent runs rather than a prompt workaround or local timeout wrapper.
