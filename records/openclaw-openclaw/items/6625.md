## Summary

Keep open. Current main and v2026.6.6 still expose and enforce hard sub-agent `runTimeoutSeconds` behavior plus post-timeout recovery, but no native/ACP pre-timeout warning, grace contract, or warning configuration exists.

## What This Changes

Request a pre-timeout warning or grace mechanism for sub-agents approaching `runTimeoutSeconds` so they can checkpoint work before termination.

## Best Possible Solution

Define a lifecycle-owned pre-timeout notification contract for native and ACP sub-agent runs, use a safe default warning offset where possible, and add a public override only if maintainers confirm the operator need.

## Reproduction Assessment

Yes, at source level: current main accepts and enforces a hard sub-agent deadline but exposes no warning input, config field, or scheduled child notification before termination. No live timeout scenario was run during this read-only review.

## Solution Assessment

Unclear as proposed. A pre-timeout notification addresses the reported loss mode, but the standalone config key is not yet proven to be the narrowest solution; maintainers should first choose the shared native/ACP lifecycle policy.
