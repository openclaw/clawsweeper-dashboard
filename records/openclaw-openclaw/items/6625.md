- `impact:session-state`: The affected behavior is child-agent session/transcript state being aborted without a pre-timeout checkpoint opportunity.

## Summary

Keep open. Current main and the latest release still only expose hard sub-agent run timeout behavior plus post-timeout partial-progress recovery; there is no config key, `sessions_spawn` parameter, native runtime path, or ACP runtime path that warns the child before timeout.

## What This Changes

Request a configurable pre-timeout warning or grace period for sub-agents so they can checkpoint before `runTimeoutSeconds` kills the run.

## Best Possible Solution

Define a documented sub-agent timeout-warning contract on the existing sub-agent config/tool surfaces, deliver it consistently before the hard timeout in native and ACP runs, and preserve current post-timeout recovery.

## Reproduction Assessment

Not applicable as a runnable bug reproduction because this is a feature/config request. Source inspection gives high confidence that current behavior is hard timeout plus post-timeout capture, not a pre-timeout warning.

## Solution Assessment

Yes at the product-direction level, but not as an automatic narrow bug fix. The maintainable path is a deliberate native and ACP lifecycle contract rather than a prompt workaround or local timeout wrapper.
