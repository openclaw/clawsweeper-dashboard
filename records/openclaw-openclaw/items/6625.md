## Label Justifications

- `P2`: The feature would reduce lost progress in bounded long-running sub-agent tasks but does not represent an emergency core outage.
- `impact:data-loss`: A hard timeout can discard child work that has not yet been persisted or returned to the requester.
- `impact:session-state`: The request concerns preserving useful child-session progress before terminal timeout state is recorded.

## Summary

Keep open. Current main and v2026.6.5 implement hard sub-agent deadlines and post-timeout recovery, but neither warns the active child before termination; no related issue or PR currently owns that remaining feature.

## What This Changes

Request a configurable warning to sub-agents shortly before their hard run timeout so they can checkpoint progress.

## Best Possible Solution

Add a lifecycle-owned pre-deadline warning under the canonical sub-agent configuration, deliver it through a child-runtime control path supported by native and ACP runs, cancel it on terminal completion, and preserve existing post-timeout partial-progress recovery.

## Reproduction Assessment

Yes at source level: current main carries only runTimeoutSeconds through native and ACP spawn and converts the stored deadline directly into terminal timeout handling, with no scheduled child warning. No live long-running agent was executed during this read-only review.

## Solution Assessment

The lifecycle goal is appropriate, but the proposed top-level sessions key is not the best-established owner; agents.defaults.subagents is the canonical timeout surface, and a per-spawn override should be added only if maintainers explicitly want that public API.

## Risks / Open Questions

- A warning timer must remain correct across gateway restarts and persisted run deadlines rather than existing only as an in-memory spawn timer.
- Native and ACP children have different steering and cancellation semantics, so a warning could race hard timeout, terminal completion, cleanup, or a naturally completed run.
- Adding both a global setting and per-spawn override would enlarge an already compatibility-sensitive public config and tool contract without a settled product need for both.
