## Summary

Keep open. Current main and v2026.6.6 still expose and enforce hard sub-agent run deadlines, plus post-timeout recovery/announcement behavior, but they do not warn the child before the deadline so it can checkpoint active work.

## What This Changes

The issue requests a configurable pre-timeout warning/grace period for sub-agents before `runTimeoutSeconds` terminates the child run.

## Best Possible Solution

Define one lifecycle-owned pre-timeout notification policy for native and ACP sub-agent runs, preferably deriving a safe default from the configured deadline unless maintainers approve a public override.

## Reproduction Assessment

Yes, at the source level: current main has a hard sub-agent deadline and post-timeout recovery surfaces, but no warning input, config field, or scheduled child notification before termination. I did not run a live timeout scenario during this read-only review.

## Solution Assessment

Unclear. A pre-timeout notification directly addresses the reported loss mode, but the proposed standalone config key is not yet proven to be the narrowest solution; maintainers should first choose the shared native/ACP lifecycle policy.
