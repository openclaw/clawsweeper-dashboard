## Summary

Keep open. Current main and v2026.6.8 still have configurable hard sub-agent timeouts and post-timeout handling, but no pre-timeout warning/grace message, warning config key, or child-session notification path before termination.

## What This Changes

The issue requests a configurable pre-timeout warning for sub-agent runs so child agents can checkpoint work before `runTimeoutSeconds` kills the run.

## Best Possible Solution

Define one maintainer-approved pre-timeout child notification contract shared by native and ACP sub-agent runs, with focused tests and docs while preserving existing post-timeout recovery.

## Reproduction Assessment

Yes, at the source level: current main has hard sub-agent deadlines and post-timeout handling, but no warning input, config key, or scheduled child notification before termination. I did not run a live timeout scenario during this read-only review.

## Solution Assessment

Unclear. A pre-timeout child notification directly targets the reported loss mode, but maintainers still need to choose the API/config shape, default policy, and native/ACP delivery semantics.
