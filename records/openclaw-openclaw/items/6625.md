## Summary

Keep open: current main still has hard sub-agent run timeouts and post-timeout partial-progress recovery, but no pre-timeout warning/grace mechanism. This is a new core sub-agent lifecycle/config/API feature, not an obsolete or already-implemented report.

## What This Changes

Request a configurable pre-timeout warning/grace period for sub-agents so they can checkpoint before `runTimeoutSeconds` aborts them.

## Best Possible Solution

Add a documented timeout-warning/grace contract with config and optional per-spawn override, delivered through native and ACP child-session lifecycle paths while preserving hard timeout and current post-timeout recovery.

## Reproduction Assessment

Not applicable: this is an enhancement request, not broken documented behavior. Source inspection against current main shows hard run timeout plumbing and post-timeout partial-progress extraction, but no pre-timeout warning surface to reproduce.

## Solution Assessment

Unclear until maintainers settle the lifecycle contract. A core child-session control/message path that works for native and ACP runs is safer than ad hoc prompt injection at one call site.
