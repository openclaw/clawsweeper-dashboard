## Summary

Keep open: current main still has hard sub-agent run timeouts and post-timeout partial-progress reporting, but no configurable pre-timeout warning/grace delivery path. This remains feature/API lifecycle work that needs maintainer product direction rather than a conservative cleanup close.

## What This Changes

The issue requests a configurable warning delivered to sub-agents shortly before `runTimeoutSeconds` kills the run so they can checkpoint progress.

## Best Possible Solution

Define an accepted sub-agent timeout-warning contract, add documented config and optional per-spawn surface if approved, then schedule warning delivery before hard timeout in native and ACP paths while preserving post-timeout recovery.
