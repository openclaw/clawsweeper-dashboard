## Summary

Keep open: current main and the latest shipped release still only expose hard sub-agent run timeouts plus post-timeout partial-progress recovery, with no configurable pre-timeout warning or grace-period delivery path. Prior review comments made the same distinction, and this recheck against current main confirms the remaining work is a feature/API and lifecycle design decision.

## What This Changes

Request a configurable pre-timeout warning/grace period for sub-agent runs so child agents can checkpoint before `runTimeoutSeconds` stops them.

## Best Possible Solution

Define an accepted sub-agent timeout-warning contract, add documented config and optional per-spawn surface if approved, then schedule warning delivery before hard timeout in native and ACP paths while preserving existing post-timeout recovery.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection give high confidence that current main lacks the requested config/tool parameters and pre-timeout warning path.

## Solution Assessment

Unclear. A pre-timeout warning is plausible, but maintainers need to settle the warning contract, defaults, override surface, and native/ACP delivery semantics before implementation.
