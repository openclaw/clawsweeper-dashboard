- `impact:data-loss`: The request is specifically about avoiding loss of unsaved code, research, generated content, and accumulated work when a child run times out.
- `impact:session-state`: The affected behavior is child-agent session and transcript state being aborted without a pre-timeout checkpoint opportunity.

## Summary

Keep open. Current main and v2026.5.20 still only expose hard `runTimeoutSeconds` plus post-timeout partial-progress handling; there is no config, tool parameter, or native/ACP runtime path that warns a child agent before the hard timeout, so this needs maintainer product/API review.

## What This Changes

The issue requests a configurable pre-timeout warning/grace period for sub-agents before `runTimeoutSeconds` aborts the child run.

## Best Possible Solution

Define a documented sub-agent timeout-warning contract, likely under `agents.defaults.subagents` with an optional `sessions_spawn` override, then deliver it consistently for native and ACP runs while preserving current post-timeout recovery.

## Reproduction Assessment

Not applicable as a runnable bug reproduction; this is a feature/config request. Source, docs, tests, and v2026.5.20 inspection show no schema, docs, or runtime path for a pre-timeout warning.

## Solution Assessment

Yes at the product-direction level, but not as an automatic narrow bug fix. The maintainable solution needs a deliberate lifecycle contract across native and ACP sub-agent runs rather than a prompt workaround or local timeout wrapper.
