## Summary

Keep open. Current main still exposes hard sub-agent run timeouts and post-timeout partial-progress recovery, but no configurable pre-timeout warning/grace path; the config/API shape and native/ACP delivery semantics need maintainer design.

## What This Changes

The issue requests a configurable warning to sub-agents shortly before `runTimeoutSeconds` expires so they can checkpoint progress before the hard timeout.

## Best Possible Solution

Add first-class core lifecycle support for a documented grace/warning setting and optional per-spawn override, with safe native and ACP delivery before the hard timeout fires.

## Reproduction Assessment

Yes for the reported missing capability: source inspection on current main shows the tool schema, strict config schema, native spawn path, ACP spawn path, docs, and timeout tests expose hard timeout/post-timeout behavior only. This is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear. The requested warning direction is reasonable, but the best implementation should be a designed lifecycle/control-message path across native and ACP runs rather than an ad hoc prompt or send workaround.
