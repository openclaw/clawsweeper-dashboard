## Summary

Keep open: current main and the latest shipped release still expose hard sub-agent run timeouts and post-timeout partial-progress recovery, but no configurable pre-timeout warning/grace delivery path. This remains a feature/API and lifecycle design decision rather than a narrow existing-contract bug.

## What This Changes

Request a configurable warning period before `runTimeoutSeconds` aborts a sub-agent so the child can checkpoint work before the hard timeout.

## Best Possible Solution

Define an accepted sub-agent timeout-warning contract, add documented config and optional per-spawn surface if approved, then schedule warning delivery before hard timeout in native and ACP paths while preserving existing post-timeout recovery.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection give high confidence that current main lacks the requested config/tool parameters and pre-timeout warning path.

## Solution Assessment

Unclear. A pre-timeout warning is plausible, but maintainers need to settle the config namespace, default behavior, per-spawn override surface, message semantics, and native/ACP delivery path before implementation.
