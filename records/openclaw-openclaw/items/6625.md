## Label Justifications

- `P2`: This is a meaningful sub-agent reliability improvement with bounded blast radius, but existing timeout recovery prevents it from being an emergency.
- `impact:data-loss`: A hard timeout can discard uncheckpointed code, research, generated content, or other work that exists only in the active child run.
- `impact:session-state`: The requested warning is intended to let a child preserve useful session progress before its terminal timeout state.

## Summary

Keep open. Current main and v2026.6.5 enforce hard sub-agent deadlines and preserve limited post-timeout output, but neither warns the active child before termination; no canonical related issue or PR owns the missing behavior.

## What This Changes

Add a configurable warning delivered to a sub-agent shortly before its hard run timeout so it can checkpoint unfinished work.

## Best Possible Solution

Define one lifecycle-owned pre-timeout notification contract for native and ACP runs, derive a safe default warning offset from the configured deadline, and add a public override only if a concrete operator need justifies the extra config surface.

## Reproduction Assessment

Yes at the source level: current main accepts and enforces a hard sub-agent deadline but exposes no warning input, config field, or scheduled child notification before termination. No live timeout scenario was run during this read-only review.

## Solution Assessment

Unclear. A pre-timeout notification directly addresses the reported loss mode, but the proposed standalone config key is not yet proven to be the narrowest solution; maintainers should first select the shared native/ACP lifecycle policy.
