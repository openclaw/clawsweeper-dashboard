## Summary

Keep open: current main still exposes hard sub-agent run timeouts plus post-timeout partial-progress recovery, but it does not provide a configurable pre-timeout warning/grace delivery path. This remains sub-agent lifecycle/API work that needs maintainer product direction.

## What This Changes

Request a configurable warning/grace period that lets sub-agents checkpoint shortly before `runTimeoutSeconds` aborts the run.

## Best Possible Solution

Define an accepted sub-agent timeout-warning contract, add documented config and optional per-spawn surface if approved, then schedule warning delivery before hard timeout in native and ACP paths while preserving existing post-timeout recovery.

## Risks / Open Questions

- A correct design needs maintainer decisions on config namespace, defaults, per-spawn override shape, warning text/control semantics, and whether native and ACP child sessions can reliably receive the warning before cleanup.

- This overlaps with broader subagent delivery and timeout reliability work, so the implementation should not create a second lifecycle contract that conflicts with durable delivery, restart, steering, or heartbeat policy.
