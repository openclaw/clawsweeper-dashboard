## Summary

The report is valid on current main, with one correction: an absent `openclaw.extensions` field is treated as `missing` and does not duplicate, but an empty or invalid extension entry plus a missing declared `main` does produce two failures for one plugin.

## What This Changes

The issue asks for the plugin payload smoke check to stop after an empty or invalid extension-entry failure so it does not also report a missing main entry for the same plugin.

## Best Possible Solution

Add a targeted `continue` after recording an empty or invalid extension-entry failure, and add a focused regression test for that combined malformed-plugin case.

## Reproduction Assessment

Yes from source inspection: `openclaw.extensions: []` or an invalid extension entry plus a declared missing `main` takes both failure paths. An absent `openclaw.extensions` field alone is not the right repro because current manifest resolution returns `missing`, not `empty`.

## Solution Assessment

Yes: adding `continue` after the terminal empty/invalid extension-entry failure is the narrowest maintainable fix, while keeping the valid-extension path’s additive main-entry check intact.

## Repair Work Prompt

Fix the duplicate plugin payload smoke-check diagnostics reported in https://github.com/openclaw/openclaw/issues/83891. In `src/cli/update-cli/plugin-payload-validation.ts`, stop processing the current plugin after recording a `missing-extension-entry` failure for `extensionResolution.status === "invalid" || extensionResolution.status === "empty"`, without changing the existing additive `missing-main-entry` check after a valid extension resolution. Add a focused regression test in `src/cli/update-cli/plugin-payload-validation.test.ts` that creates a package with `openclaw.extensions: []` or an invalid extension entry plus a declared missing `main`, then asserts exactly one failure for that plugin. Do not broaden update/restart policy or change post-core convergence warning formatting beyond the duplicate-failure fix.
