## Summary

Keep this open: current main and the latest release still lack a completion-origin route lease or equivalent fallback, and the issue is now paired with an open fixing PR that uses closing syntax but has not merged yet.

## What This Changes

The issue reports that isolated cron announce completion can lose an explicit `delivery.channel` before final completion delivery, causing `Channel is required` even after the target run succeeds.

## Best Possible Solution

Land a maintainer-approved route-lease or equivalent completion-origin fix that keeps the resolved cron delivery route available through delayed completions and retires it under the shared background-delivery lifecycle.

## Reproduction Assessment

Yes at source level, but not as an independently run live repro in this read-only review. Current main resolves explicit cron delivery targets but later completion-origin resolution only reads requester session delivery context, matching the reported route-loss seam.

## Solution Assessment

Yes, keeping this issue open is the right handling because https://github.com/openclaw/openclaw/pull/93110 is an open implementation candidate, not a merged fix. The safest implementation shape is the maintainer-requested generic route lease owned by the shared background completion lifecycle.
