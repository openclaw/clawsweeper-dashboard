## Summary

Keep this open: current main still lacks the durable completion-route fallback described by the issue and the directly linked fix PRs were closed unmerged. The report is source-plausible and fits a real core message-delivery bug, but the likely fix adds shared persisted route/lifecycle state, so it needs maintainer design review before another automated fix attempt.

## What This Changes

The issue asks OpenClaw to preserve an isolated cron job's explicit announce `delivery.channel` through the final completion-announcer/controller-return path.

## Best Possible Solution

Land a maintainer-reviewed core fix that records the resolved completion route at cron task preparation/dispatch time and lets the final announcer recover that route when session `deliveryContext` is missing, reusing existing state surfaces if maintainers prefer that over a new table.

## Reproduction Assessment

Yes, at source level and from issue evidence: current main has a session-entry based announcer origin path and lacks the task-route fallback proposed by the closed candidate fixes. I did not run a live isolated cron/webchat repro in this read-only review.

## Solution Assessment

Unclear. The durable task-completion-route design appears aimed at the right boundary, but because it adds persisted shared state and lifecycle semantics, maintainer review should confirm the route key, retirement, and pruning contract before implementation.
