## Label Justifications

- `P1`: The report describes lost completion announcements for successful isolated cron runs, which can break a real async agent/channel workflow.
- `impact:message-loss`: The affected completion path can drop or fail the final outbound announcement even after target work completes.

## Summary

Keep open: current main still shows the resolved cron delivery target is used for planning but is not written into the isolated session state that later completion announce paths depend on. The linked fix PR was closed unmerged after a maintainer confirmed the bug but rejected that branch’s lifecycle approach, and the other open cron announce PR is an adjacent admission-time guard rather than this completion-origin fix.

## What This Changes

The issue reports that isolated cron completion announce delivery loses an explicit `delivery.channel` before final outbound delivery, causing completed isolated runs to fail with `Channel is required`.

## Best Possible Solution

Implement a shared requester-route lease owned by the background completion/delivery lifecycle so explicit cron delivery targets remain available through final announce delivery and are retired after every supported completion settles.

## Reproduction Assessment

We have a high-confidence source-level reproduction boundary, but not a fresh live reproduction from this read-only review. Current main lacks the deliveryContext write-back, the linked PR’s focused proof hit the same path, and a maintainer confirmed the routing bug before closing that PR as the wrong lifecycle shape.

## Solution Assessment

No, the previously proposed narrow cron write-back is not the best final solution. The maintainable path is the maintainer-described generic requester-route lease with lifecycle retirement across cron, subagent, ACP, and media completion delivery.
