## Label Justifications

- `P1`: The report describes a live isolated cron workflow where successful work can lose its final controller callback.

## Summary

The ordinary isolated-cron delivery path on current main preserves an explicit channel, but subagent completion uses separately captured requester-origin metadata; the issue identifies a credible uncovered boundary where an isolated cron session can spawn work without carrying its explicit destination into the final completion announcement. The item is distinct from the older planning-level report and remains necessary.

## What This Changes

Fix isolated cron terminal completion announcements so an explicit `delivery.channel` survives through descendant/controller-return delivery.

## Best Possible Solution

Carry the already-resolved cron destination into descendant requester-origin metadata at spawn time, then add a regression test that exercises an isolated cron with explicit `webchat` delivery through terminal descendant completion and verifies exactly one callback reaches the intended requester.

## Reproduction Assessment

Yes at source level with medium confidence: current main has separate cron delivery-target and subagent requester-origin contracts, and focused coverage for their explicit-channel handoff is absent. A live current-main reproduction was not run, and the reporter’s local artifacts were unavailable.

## Solution Assessment

Yes in direction: propagate the prepared destination into the existing requester-origin contract rather than adding another fallback, configuration option, or parallel delivery mechanism. The exact assignment point should be selected after a focused spawn-path test demonstrates the failure.
