## Summary

Keep this PR open. Current main still routes the WebChat New Chat affordance through `/new`, and the PR correctly moves toward the existing `sessions.create` gateway contract, but the current PR head can still overwrite composer edits made while the create/refresh request is pending. The PR is maintainer-authored, labeled `maintainer`, and opted into `clawsweeper:automerge`, so the right path is another narrow repair and exact-head review rather than cleanup closure.

## What This Changes

The PR rewires Control UI/WebChat New Chat to call `sessions.create`, refresh session options, switch to the created dashboard session, add focused session tests, and add a changelog entry.

## Best Possible Solution

Keep the narrow `sessions.create` approach, but before auto-switching compare the current draft and attachments with the pre-await snapshots. If either changed, leave the user in the current session with the refreshed session list and current composer intact so the new session remains selectable without losing edits.

## Reproduction Assessment

Yes. On PR head `521b99e381dc`, mock `createSessionAndRefresh` to mutate `state.chatMessage` or replace `state.chatAttachments` before resolving; `createChatSession` will still switch sessions and restore the older pre-await values.

## Solution Assessment

No. The backend contract and PR scope are right, but the implementation is not yet the safest solution because it treats session/run state as stale-aware while ignoring composer edits made during the async create/refresh window.
