## Summary

Keep this PR open. The change is useful and current main still routes WebChat New Chat through the old `/new` reset path, but the latest PR head can overwrite draft or attachment edits made while `sessions.create` is pending, so it is not ready for automerge yet.

## What This Changes

This PR rewires Control UI/WebChat New Chat from a `/new` reset to a `sessions.create` plus refresh-and-switch flow, adds focused session helper tests, and adds a changelog entry.

## Best Possible Solution

Keep the focused `sessions.create` integration, but add a pre-switch composer guard: if the current draft or attachments differ from the pre-await snapshots, leave the user in the current session with the current composer intact while keeping the refreshed session list so the newly created session remains selectable.

## Reproduction Assessment

Yes. On PR head `521b99e381dc`, mock `createSessionAndRefresh` to mutate `state.chatMessage` or replace `state.chatAttachments` before resolving; `createChatSession` still switches sessions and restores the older pre-await values.

## Solution Assessment

No. The backend contract and PR direction are right, but the implementation is not yet the safest narrow solution because it guards session/run state while ignoring composer edits made during the async create/refresh window.
