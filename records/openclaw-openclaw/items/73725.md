## Summary

Keep open. Current main still routes the Control UI/WebChat New Chat button through the in-place `/new` reset path, so the PR is useful and not obsolete, but the current PR head still has a P2 stale-composer bug that can overwrite edits made while session creation is pending. The PR is also maintainer-authored/labeled and opted into ClawSweeper automerge, so this should remain open for repair and exact-head gating rather than cleanup closure.

## What This Changes

The PR changes Control UI/WebChat New Chat to call `sessions.create`, refresh and switch to the created dashboard session, keep explicit `/new` reset semantics, add focused session/helper tests, and add a changelog entry.

## Best Possible Solution

Keep the narrow `sessions.create` direction, but add a stale-composer guard before auto-switching. If the draft text or attachments changed while create/refresh was pending, leave the user in the current session with the current composer intact and keep the refreshed session list so the new session is selectable.

## Reproduction Assessment

Yes. On the PR head, mock `createSessionAndRefresh` to mutate `state.chatMessage` or replace `state.chatAttachments` before resolving; `createChatSession` will still switch and restore the older pre-await values.

## Solution Assessment

No. The PR uses the right backend contract and scope, but it is not yet the safest implementation because it treats session/run state as stale while ignoring composer edits made during the async create/refresh window.

- **[P2] Preserve composer edits made while creating a session:** `ui/src/ui/app-render.helpers.ts:651-653`
