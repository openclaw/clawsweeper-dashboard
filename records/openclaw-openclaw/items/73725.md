## Summary

Keep this PR open. Current main still routes WebChat New Chat through the `/new` reset path, so the PR is addressing real remaining behavior, but the repaired head can still overwrite composer edits made while `sessions.create`/`sessions.list` is pending. The PR is also maintainer-labeled and member-authored, so cleanup closure is not appropriate.

## What This Changes

The PR changes Control UI/WebChat New Chat to create and switch to a dashboard session via `sessions.create`, adds focused session/controller coverage, and updates the changelog.

## Best Possible Solution

Keep the narrow create/switch direction, but add a stale-composer guard before auto-switching: if the draft text or attachments changed while creation was pending, leave the user in the current session with their current composer intact and refresh the session list so the created session remains selectable. Add focused regression coverage for delayed draft and attachment edits, then rerun targeted tests and the changed gate.

## Reproduction Assessment

Yes. On the PR head, mock `createSessionAndRefresh` to change `state.chatMessage` or replace `state.chatAttachments` before resolving; `createChatSession` will still call `switchChatSession` and then restore the older pre-await values.

## Solution Assessment

No. The overall approach is the right narrow fix, but it is not yet the safest implementation because it only treats session changes and active runs as stale, not composer edits made during the async create/refresh window.
