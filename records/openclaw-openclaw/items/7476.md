## Summary

Keep this issue open: current main contains a tested low-level WhatsApp sticker sender for QA, but normal outbound WebP media still follows the generic image path and the supported WhatsApp action surface still has no sticker intent. The closed implementation proposals remain useful context, especially the reporter’s live test and the later encoding cautions, but neither proposal merged and no canonical replacement owns the remaining work.

## What This Changes

Add a supported outbound WhatsApp sticker path that sends sticker media as a native sticker instead of a generic image.

## Best Possible Solution

Expose one explicit typed sticker intent in the bundled WhatsApp plugin, reuse the existing canonical `sendSticker` sender, normalize or reject unsupported static and animated inputs deterministically, preserve ordinary WebP image behavior, and cover action discovery, dispatch, media preparation, Baileys payload selection, live delivery, and public docs.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: normal outbound WebP enters the generic image branch, the action runtime lacks sticker dispatch, and the docs expose only inbound sticker handling.

## Solution Assessment

No, the prior empty-caption and file-shape inference is not the best permanent solution because it changes existing media semantics and does not settle encoding requirements. The narrow maintainable direction is an explicit typed sticker intent backed by the existing canonical sender and a defined normalization policy.
