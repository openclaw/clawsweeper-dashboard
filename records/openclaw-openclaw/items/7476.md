## Summary

Keep this issue open. The latest release contains a native WhatsApp sticker transport primitive, but it is only wired into QA Lab; normal agent/CLI message actions and the standard outbound send contract still cannot express sticker intent, so the requested user capability remains missing.

## What This Changes

Add first-class outbound WhatsApp sticker sending so native stickers can be sent instead of routing WebP media as ordinary images.

## Best Possible Solution

Add one explicit plugin-owned sticker intent to the normal message surface, pass it through the WhatsApp outbound contract to the existing `sendSticker` primitive, define accepted static and animated WebP handling, and document and live-test that user path.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 expose sticker sending only through QA infrastructure, while normal WhatsApp action discovery and outbound options have no sticker intent. A live WhatsApp account remains necessary for final end-to-end acceptance proof.

## Solution Assessment

No, MIME, size, and caption auto-detection alone is not the best solution because it creates ambiguous image behavior and does not settle encoding requirements. An explicit sticker intent routed to the existing plugin-owned sender is the narrower maintainable design.
