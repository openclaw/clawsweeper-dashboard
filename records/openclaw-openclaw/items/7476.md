## Summary

Keep this issue open: current main still lacks an outbound WhatsApp sticker contract, WebP media still routes through generic image sending, and the two related implementation PRs are closed without merge. The latest discussion also adds real encoding-policy concerns, so this needs maintainer product review before a safe implementation is queued.

## What This Changes

The issue requests outbound WhatsApp sticker send support instead of delivering sticker WebP files as ordinary WhatsApp image attachments.

## Best Possible Solution

Implement first-class outbound WhatsApp sticker support in the bundled plugin with explicit sticker intent, scoped WebP preservation or re-encoding policy, Baileys sticker payload delivery, aligned action/tool/docs behavior, focused regression tests, and live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main routes outbound `image/webp` through the generic image payload and has no WhatsApp sticker option or action to select a Baileys sticker payload. I did not run a live WhatsApp device send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract and encoding policy. An explicit `action=sticker` or `asSticker`/`sendAsSticker` intent with scoped validation or re-encoding is safer than relying on automatic WebP detection alone.
