## Summary

Current main implements sticker payload sending only as a low-level WhatsApp QA-driver capability; the normal message-tool send/action contract, outbound option plumbing, and user documentation still lack sticker support, so the requested user behavior remains necessary. The reporter’s working Baileys test and the encoding cautions are useful design evidence, while both implementation PRs are closed unmerged and no open canonical replacement owns the work.

## What This Changes

Add supported outbound WhatsApp sticker sending through an explicit message action or send option instead of delivering eligible media as ordinary images.

## Best Possible Solution

Add an explicit WhatsApp-owned sticker intent that reuses the existing canonical `sendSticker` method, defines whether inputs must already be valid stickers or are normalized by the plugin, preserves ordinary image behavior, and ships focused contract/pipeline tests, WhatsApp docs, and live transport proof.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction: normal WhatsApp action discovery and outbound options lack sticker intent, so a WebP sent through the ordinary path cannot select the existing low-level `sendSticker` method.

## Solution Assessment

No, automatic WebP/size/caption inference is not the best permanent solution because it overloads existing image semantics and does not settle encoding validity. An explicit plugin-owned action or typed option with documented normalization rules is the narrowest maintainable direction.

## Risks / Open Questions

- The public contract is unresolved: a dedicated `sticker` action is explicit and typed, while a generic `sendAsSticker` field would expand shared message-tool/API surface and automatic WebP inference could silently change existing image sends.
- Arbitrary input conversion needs a defined policy for dimensions, static versus animated WebP, animation chunks, size ceilings, and metadata; routing bytes as `{ sticker }` alone may produce rejection or inconsistent client rendering.
- The June QA merge can create false confidence because it proves a low-level transport path, not a supported end-user workflow.
