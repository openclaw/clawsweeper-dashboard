## Summary

Current main has an internal WhatsApp `sendSticker` primitive for QA/runtime tests, but the user-facing send path still does not expose or dispatch outbound stickers: `send`/`upload-file` WebP media still goes through `active.sendMessage` and the generic image payload, while WhatsApp actions advertise no `sticker` action. This issue should stay open for maintainer product direction on the public outbound contract and sticker validation/encoding behavior.

## What This Changes

Request outbound WhatsApp sticker support so bots can send sticker messages instead of delivering eligible WebP files as regular image attachments.

## Best Possible Solution

Implement a first-class WhatsApp outbound sticker contract in the bundled WhatsApp plugin, route explicit sticker intent through Baileys sticker payloads, keep normal image sends unchanged, and document/test the chosen validation or re-encoding behavior.

## Reproduction Assessment

Yes, at source level: current main exposes no WhatsApp `sticker` action or sticker send option, and the normal outbound media path still sends `image/webp` through the generic image payload. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability belongs in the WhatsApp plugin, but explicit sticker intent is safer than silently treating qualifying WebP images as stickers because it avoids misclassifying ordinary image sends.
