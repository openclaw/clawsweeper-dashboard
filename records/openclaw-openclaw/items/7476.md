## Summary

Keep this issue open. Current main still lacks an outbound WhatsApp sticker send contract and still routes outbound WebP/image media through the generic image path; the related implementation PRs are closed and unmerged, and the latest discussion raises a real maintainer choice around sticker intent and encoding policy.

## What This Changes

Request outbound WhatsApp sticker send support, either through a dedicated sticker action, an explicit send option, or WebP sticker detection.

## Best Possible Solution

Implement first-class outbound sticker support in the bundled WhatsApp plugin with explicit sticker intent, scoped WebP preservation or re-encoding policy, Baileys `{ sticker: buffer }` delivery, aligned action/tool/docs behavior, focused regression tests, and live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main routes outbound `image/webp` through the generic `image/*` payload and has no WhatsApp sticker option or action to select a Baileys sticker payload. I did not run a live WhatsApp device send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. An explicit `action=sticker` or `asSticker`/`sendAsSticker` intent with scoped validation or re-encoding is safer than relying on automatic WebP detection alone.
