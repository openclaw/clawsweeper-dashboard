## Summary

Keep this issue open. Current main still lacks an outbound WhatsApp sticker send contract and still routes outbound WebP/image media through the generic image path; the two related implementation PRs are closed and unmerged, and the discussion raises a real maintainer choice around explicit intent and sticker encoding policy.

## What This Changes

Request outbound WhatsApp sticker send support via a dedicated action, send option, or safe WebP sticker routing.

## Best Possible Solution

Implement first-class outbound sticker support in the bundled WhatsApp plugin with explicit sticker intent, scoped WebP preservation or re-encoding, Baileys `{ sticker: buffer }` delivery, aligned action/tool/docs behavior, focused regression tests, and live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main routes outbound `image/webp` through the generic `image/*` payload and has no WhatsApp sticker option or action to select a Baileys sticker payload. I did not run a live WhatsApp device send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. An explicit `action=sticker` or `asSticker` intent with scoped validation/re-encoding is safer than relying on automatic WebP detection alone.
