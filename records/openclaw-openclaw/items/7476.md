## Summary

Keep this open: current main still has no outbound WhatsApp sticker send contract or payload branch, and the related implementation PRs remain closed without merge. The source gap is clear, but the latest discussion makes the public action/option and WebP encoding policy a maintainer decision before repair automation is safe.

## What This Changes

Request to add outbound WhatsApp sticker sending so explicit sticker intent or qualifying WebP media is delivered as a WhatsApp sticker payload instead of a regular image.

## Best Possible Solution

Implement first-class outbound WhatsApp sticker support in the bundled WhatsApp plugin with explicit sticker intent, scoped WebP preservation or re-encoding policy, Baileys `{ sticker: buffer }` delivery, aligned action/tool/docs behavior, focused regression tests, and live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main routes outbound `image/webp` through the generic image payload and has no WhatsApp sticker option or action to select a Baileys sticker payload. I did not run a live WhatsApp device send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract and encoding policy. An explicit `action=sticker` or `asSticker`/`sendAsSticker` intent with scoped validation or re-encoding is safer than relying on automatic WebP detection alone.
