## Summary

Close as implemented: current main and the latest release no longer let this empty tool-only path fall through to zero Telegram delivery. The literal `Done.` fallback is not current policy, but the reported no-message outcome is handled by the shared embedded-run incomplete-turn payload before Telegram's empty-payload skip can apply.

## What This Changes

The issue requests a visible Telegram acknowledgement when an assistant turn completes tool calls but produces no final assistant text.

## Best Possible Solution

Keep the shipped shared runner behavior: surface a visible terminal warning for no-output tool turns, while leaving any future success-style acknowledgement as a separate UX/product discussion.

## Reproduction Assessment

No, current main does not reproduce the reported silent drop from source inspection. The embedded runner turns the empty no-committed-delivery result into a visible payload before Telegram delivery sees it.

## Solution Assessment

Yes, for the no-delivery bug: the current shared runner fix is narrower than a Telegram-only patch and avoids pretending success when the model produced no final answer. A literal `Done.` acknowledgement would be a separate UX choice, not required to fix the silent drop.
