## Summary

Current main already prevents the reported Telegram direct silent-drop path by sending a Telegram-owned visible fallback when an automatic direct-chat dispatch completes with no delivered reply; the remaining difference is copy preference, not missing delivery.

## What This Changes

The issue requests a visible completion acknowledgement for Telegram turns where tools run successfully but the assistant produces no final text.

## Best Possible Solution

Keep the current main behavior: Telegram direct chats should receive the existing channel-level visible fallback for empty automatic replies, while any request to change the fallback copy to `Done.` can be handled as a separate UX preference.

## Reproduction Assessment

No, not against current main from source trace: the shared runner can return no payloads, but Telegram direct dispatch detects the no-delivery result and sends a projected visible fallback. The remaining message text is not exactly the proposed `Done.`, but the reported no-message outcome is covered.

## Solution Assessment

Yes, the current channel-level fallback is the better maintainable fix because it handles any Telegram automatic turn that ends without visible delivery, not only runs where tool metadata is available in the shared runner.
