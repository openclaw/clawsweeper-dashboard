## Summary

Keep open: current main still has a credible LINE target-routing bug. The LINE plugin strips `line:*:` prefixes when it owns the send, but core can persist/replay raw durable targets and cron/session-key inference can reconstruct a lowercased LINE recipient from normalized session keys, matching the reported failure mode.

## What This Changes

Fix LINE long-running delivery recovery so case-sensitive LINE recipient IDs are preserved and `line:*:` prefixes are not sent to the LINE push API as `to` values.

## Best Possible Solution

Normalize LINE targets at the channel-owned delivery boundary and avoid reconstructing LINE recipients from lowercased session-key fragments; store or reuse a case-preserving delivery context with focused regression coverage.

## Reproduction Assessment

No live current-main reproduction was run, but source inspection gives a clear reproduction path for the lowercase failure: LINE peer IDs are lowercased in session keys and cron fallback can reuse that fragment as `delivery.to`. The prefix-in-queue symptom is also source-visible, but the final failed send path needs a targeted recovery test because the LINE plugin normally strips prefixes.

## Solution Assessment

Yes, the best fix is a narrow routing/normalization repair rather than a new setting. The repair should keep core plugin-agnostic, use the LINE plugin's existing target normalization or stored delivery context, and add LINE-specific regression tests for durable recovery and session-key fallback.
