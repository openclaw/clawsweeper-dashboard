## Summary

Keep open: current main still has a credible LINE target-routing bug, especially for case-sensitive LINE IDs reconstructed from lowercased session keys. The prefixed durable target is stored in the queue, but the final prefix failure needs a focused recovery test because the LINE send helper normally strips `line:*:` prefixes before calling the API.

## What This Changes

The issue reports that LINE post-reply-token and delivery-recovery sends can use lowercased or prefixed recipients, causing LINE push delivery to fail while OpenClaw marks the message delivered.

## Best Possible Solution

Preserve the original LINE webhook target for delayed/durable sends and make recovery pass through the channel-owned target normalization boundary, while avoiding reconstruction of LINE recipients from lowercased session-key fragments.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a credible reproduction path for the lowercase failure: LINE peer IDs are lowercased in session keys, then cron fallback can reuse that fragment as `delivery.to`; the prefixed queue symptom is visible in durable storage but needs a targeted send/recovery test because `send.ts` normally strips prefixes.

## Solution Assessment

Yes, a narrow routing/normalization repair is the best direction. The fix should keep core plugin-agnostic, reuse the LINE plugin's target normalization or a stored case-preserving delivery context, and add LINE-specific regression coverage rather than adding a setting.
