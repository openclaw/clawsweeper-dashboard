## Summary

Keep open: current main still has a source-reproducible LINE recipient normalization failure, and the open linked fix PR explicitly closes this issue but has not merged.

## What This Changes

The issue reports that LINE durable delivery recovery and post-reply-token push paths can send lowercased or prefixed LINE recipient IDs, causing rejected LINE push requests while OpenClaw reports delivery success.

## Best Possible Solution

Review and land the linked LINE send-boundary normalization fix after real LINE behavior proof, then close this issue once the fix is merged and, ideally, shipped.

## Reproduction Assessment

Yes for a source-level reproduction path: current main can lowercase LINE peer IDs into session keys, cron fallback copies that fragment into `delivery.to`, and recovery replays queued `entry.to` verbatim. I did not run a live LINE account reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is at the LINE channel send boundary: strip OpenClaw target prefixes and canonicalize LINE-shaped U/C/R IDs before calling the LINE API, without teaching core LINE-specific casing rules.
