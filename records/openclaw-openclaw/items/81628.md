Keep open: current main still has a source-visible LINE recipient routing bug, and two open fix PRs already reference this issue with closing syntax, so the issue should remain open until one fix lands or is rejected.

## What This Changes

The issue requests a fix for LINE long-running and delivery-recovery push paths sending lowercased or session-prefixed recipients to the LINE Messaging API.

## Best Possible Solution

Land one focused fix that keeps LINE recipient casing/prefix handling at the LINE-owned delivery boundary or preserves case-exact delivery context, with targeted regression coverage and real LINE proof.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: a LINE peer ID can be lowercased into the session key, cron fallback can copy that fragment to `delivery.to`, and delivery recovery replays queued `entry.to` unchanged. I did not run a live LINE API reproduction in this read-only review.

## Solution Assessment

Yes, a narrow LINE-owned normalization or case-preserving delivery-context fix is the maintainable direction; core should not grow broader LINE-specific policy if the plugin boundary can handle the wire-level recipient safely.
