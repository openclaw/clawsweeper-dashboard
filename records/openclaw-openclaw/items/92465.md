## Summary

The report remains actionable because the affected 2026.6.1 release already contained the same bounded embedding batches as current main, yet its indexing run still produced the 431 response. Current recovery logic does not recognize `431 request_headers_too_large`, but the exact provider response should be captured before assuming ordinary body splitting is sufficient.

## What This Changes

Prevent large builtin-memory indexing runs from failing with OpenAI `431 request_headers_too_large`, while preserving one-pass indexing for large corpora.

## Best Possible Solution

Capture the redacted failing OpenAI embedding response on current main, then make the canonical inline batch path adaptively split the specific recoverable 431 shape and return an actionable terminal error when a single-item request still fails.

## Reproduction Assessment

Yes at medium confidence: the report supplies an exact released image, command, provider, scale threshold, and successful chunked control, while source confirms that HTTP 431 is not handled by the existing split policy. A current-main live OpenAI run is still needed for high-confidence reproduction.

## Solution Assessment

Partly: internal adaptive batching is the correct product boundary, but adding a blind 431 regex alone is not yet the best fix because the response identifies headers while the corpus is transmitted in the request body.
