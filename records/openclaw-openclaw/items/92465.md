## Summary

The report is credible for v2026.6.1, and current main now limits builtin-memory embedding bodies to small batches, but that unreleased change does not explicitly recognize OpenAI HTTP 431 and cannot prove that request-body size caused a header-size response. Keep this issue open for a redacted current-main reproduction that identifies the oversized header before choosing between an adapter fix and an exact 431 regression repair.

## What This Changes

Prevent large OpenAI-backed builtin-memory imports from failing with HTTP 431, using automatic batching or actionable diagnostics.

## Best Possible Solution

Capture a redacted current-main OpenAI response plus request-header names and approximate sizes, then fix the header-producing provider adapter if a header grows with corpus size; if OpenAI emits 431 as a request-size proxy, add the exact error to the canonical split classifier with a focused regression test and actionable terminal output.

## Reproduction Assessment

No. The report gives a concrete v2026.6.1 Docker procedure, but current main has since added body batching and the available evidence does not reproduce or explain the exact HTTP 431 header failure.

## Solution Assessment

Unclear. Centralized batching is the narrowest solution for oversized embedding bodies, but an HTTP 431 requires identifying the actual oversized header before treating additional chunking as the best fix.
