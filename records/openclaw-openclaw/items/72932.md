## Summary

Keep this PR open. Current main has already absorbed the safer session-routing behavior, but the PR still contains an unmerged fast-startup change and its branch has concrete correctness/operability blockers before it can be used or replaced.

## What This Changes

The PR changes CLI and gateway session-key routing for `agentId` plus `sessionId`, adds regression tests, defers selected gateway startup work behind `OPENCLAW_GATEWAY_FAST_CONFIG`, and changes Bonjour unhandled-rejection cleanup lifetime.

## Best Possible Solution

Rebase or replace the PR as a narrow current-main change: keep the session routing behavior already present on main, add any fast-startup path with explicit rejected-task logging and isolated tests, and only touch Bonjour in `extensions/bonjour` if a remaining current-main cleanup bug is still reproducible.

## Reproduction Assessment

Yes. Static reproduction is high-confidence from the PR diff: send gateway `agent` params with both a named `agentId` and a fresh `sessionId`, and the PR clears `requestedSessionKey` while the downstream dispatch path lacks an agent id. The background-maintenance observability bug is reproducible by making either deferred startup task reject under the PR's fast-startup mode.

## Solution Assessment

No. This PR is not the best complete solution as written because it is partly superseded by current main, introduces a named-agent routing regression on its gateway path, hides deferred startup failures, and patches a Bonjour file that no longer owns the behavior.

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve agent routing for fresh session-id requests:** `src/gateway/server-methods/agent.ts:506-510`
  - body: When `sessionId` is present, this branch leaves `requestedSessionKey` undefined; the later gateway dispatch on the PR branch does not forward `agentId`, so a fresh request for a named agent can run against the default agent/store. Forward the validated agent id or derive an explicit `agent:<id>:explicit:<sessionId>` key here.
  - confidence: 0.91
- **[P2] Log background startup maintenance failures:** `src/gateway/server-startup-plugins.ts:72-73`
  - body: The fast-startup branch consumes rejections with `Promise.allSettled`, but the callback ignores rejected results. Operators get no warning when deferred channel maintenance or session migration fails, so inspect the settled results and log rejected reasons.
  - confidence: 0.88
- **[P2] Clean up the Bonjour rejection handler on startup errors:** `src/infra/bonjour.ts:332-333`
  - body: The PR registers the unhandled-rejection handler before `createCycle()`, but the error path only restores console logging. If cycle creation or listener setup throws before an advertiser is returned, the process-level handler leaks until process exit.
  - confidence: 0.84
- **[P3] Restore fast-startup env in the test:** `src/gateway/server-startup-post-attach.test.ts:307`
  - body: This test sets `OPENCLAW_GATEWAY_FAST_CONFIG` and deletes it only on the happy path. If an import or assertion throws first, later tests inherit fast gateway mode; save the previous value and restore it in a `finally`.
  - confidence: 0.83
- **[P3] Measure plugin trace stages from their own start:** `src/gateway/server-startup-plugins.ts:82`
  - body: Several trace calls pass the bootstrap start time into a helper whose `stage_ms` is computed from the supplied start time, so those labels report elapsed-since-bootstrap rather than each stage duration. Capture a local timestamp for each measured stage.
  - confidence: 0.78
