## Summary

Keep open. Current main already contains the safer session-id routing and the Bonjour cleanup has moved into the bundled Bonjour plugin, but the PR still contains an unmerged fast-startup change and its head has concrete functional regressions that should be fixed before merge.

## What This Changes

The PR changes CLI and gateway session-key routing for `agentId` plus `sessionId`, adds regression tests, defers selected gateway startup work behind `OPENCLAW_GATEWAY_FAST_CONFIG`, and adjusts Bonjour unhandled-rejection cleanup lifetime.

## Best Possible Solution

Rebase or replace the PR with a smaller patch against current main: keep the current session-id resolver and ingress forwarding behavior, implement the optional fast-startup path with explicit logging for background failures, avoid stale Bonjour-core edits unless the current plugin still needs a fix, and add the required changelog entry.

## Reproduction Assessment

Yes. The main PR regression is reproducible by static call-path inspection on the provided PR head: send gateway `agent` params with both `agentId` and a fresh `sessionId`, and the PR clears `requestedSessionKey` while dispatching without a preserving `agentId`. The startup observability bug is reproduced by making deferred maintenance reject under fast mode and observing that `Promise.allSettled` consumes the rejection without logging it.

## Solution Assessment

No. This PR is not currently the best way to solve the issue because it mixes a partially superseded session fix, an unlogged deferred-startup branch, and a stale Bonjour file path; a rebased/split fix is safer.
