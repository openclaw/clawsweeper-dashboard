## Summary

The report is credible and should stay open: current main rejects the first bad non-default-agent `sessionTarget="main"` job, but source inspection did not show an intent-level supersession path that would suppress a later stale replay after a corrected job succeeds.

## What This Changes

The issue requests deduplicating cron validation-failure announcements by reminder intent so a corrected `cron.add` or `cron.update` suppresses the stale failed attempt.

## Best Possible Solution

Add regression-covered cron failure supersession so a successful later schedule with the same reminder intent marks the earlier failed announcement as superseded, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a live high-confidence reproduction. Source inspection confirms the reported validation failure and the absence of an obvious intent-level supersession check, but the exact replay source still needs a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is a plausible direction, but maintainers should first choose the canonical intent key and pin the stale replay producer before changing cron state or heartbeat delivery semantics.
