## Summary

Keep open: current main still has the reported non-default-agent `sessionTarget="main"` rejection, and the inspected cron/system-event/heartbeat paths show only local text/event filtering rather than an intent supersession mechanism for a stale failure after a corrected schedule succeeds.

## What This Changes

The issue requests cron failure replay deduplication by reminder intent so a corrected schedule can supersede an older validation failure before it is announced.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update the cron tool guidance so non-default agents are steered to isolated agent-turn jobs before the first failure.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default `sessionTarget="main"` rejection and the absence of visible cron intent supersession, but the stale replay path still needs a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should first choose the canonical key and pin the stale failure producer before changing cron state or heartbeat replay semantics.
