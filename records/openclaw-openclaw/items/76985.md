## Summary

Keep open: current main still has the reported non-default-agent `sessionTarget="main"` rejection and I did not find a cron/commitment supersession path that would retire an earlier stale reminder-style failure after a corrected cron job succeeds.

## What This Changes

The issue asks cron failure replay to suppress stale validation failures once a corrected schedule with the same reminder intent succeeds.

## Best Possible Solution

Add regression-covered supersession between failed reminder attempts and later successful cron schedules, using a maintainer-approved intent key, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default `sessionTarget="main"` rejection and the absence of a cron/commitment supersession path, but no failing end-to-end replay test was established in this read-only review.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should first choose whether cron creation, commitment extraction, or heartbeat delivery owns the supersession state.
