## Summary

Keep open: current main still has the precursor cron.add rejection and the failed mutating-tool replay state only clears a matching action fingerprint, without a maintainer-approved cron intent key that would supersede a stale failed schedule after a corrected same-intent job succeeds.

## What This Changes

The issue requests deduplicating cron announce-on-failure replay by stable schedule intent so corrected cron.add or cron.update attempts suppress stale failure announcements.

## Best Possible Solution

Define a maintainer-approved cron intent identity at the failed tool-mutation replay boundary, then test that a corrected same-intent cron.add supersedes the stale failure while distinct schedules still warn.

No high-confidence live reproduction was run. Source inspection confirms the precursor validation failure, rejection-before-storage behavior, and missing cron intent supersession path.

## Solution Assessment

Unclear: intent deduplication is directionally right, but maintainers need to choose the stable cron intent contract and whether it belongs in generic failed mutating-tool replay or cron-specific normalization.
