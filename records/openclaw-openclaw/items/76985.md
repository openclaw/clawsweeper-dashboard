## Summary

Keep open: current main still has the non-default-agent `sessionTarget="main"` rejection and I did not find a supersession path that dismisses an earlier inferred/reminder replay when a later corrected `cron.add` succeeds.

## What This Changes

Request deduping stale cron failure-replay announcements by reminder intent when a corrected cron schedule succeeds.

## Best Possible Solution

Add regression-covered supersession between failed inferred reminder/cron replay entries and later successful cron schedules using a documented intent key, and update cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection on current main verifies the invalid `sessionTarget="main"` rejection, the absence of same-intent supersession, and the replay-capable commitment/heartbeat paths, but no focused failing test was run.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and the exact seam between cron job creation, commitment extraction, and heartbeat delivery before changing behavior.
