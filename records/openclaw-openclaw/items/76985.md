Keep open: current main still has the non-default `sessionTarget="main"` cron validation failure, and I did not find an intent-level supersession path that would suppress a stale failed cron add/update after a corrected schedule lands. The related closed report covered guidance around `current` vs `isolated` and was self-closed as skill territory, but it does not cover this replay-dedup behavior.

## What This Changes

The issue requests deduplicating cron add/update failure replay by schedule intent so a corrected cron schedule suppresses a stale validation rejection.

## Best Possible Solution

Add cron-specific stable intent supersession at the failure replay layer that actually owns the stale announcement, then cover failed-then-corrected scheduling with a focused regression or live proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run. Source inspection confirms the precursor validation failure and shows no cron-intent supersession path, but the scheduled-fire replay needs a focused test or live repro.

## Solution Assessment

Unclear as an implementation plan: deduping by intent is the right product shape, but the fix should land where failed cron mutations are persisted or replayed, not blindly in cron job storage.
