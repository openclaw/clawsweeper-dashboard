## Summary

Keep open: current main still rejects the reported non-default-agent `sessionTarget="main"` case, and I found no cron intent or supersession state that would retire a stale failure after a corrected schedule succeeds.

## What This Changes

Deduplicate cron failure replay announcements by reminder intent so a later successful corrected schedule suppresses a stale validation rejection.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and clarify the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default `sessionTarget="main"` rejection and the absence of visible intent supersession, but the stale replay path still needs a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and state boundary before changing queued system-event or heartbeat replay behavior.
