## Summary

Current main still has the reported non-default-agent `sessionTarget="main"` rejection and I did not find any cron intent/supersession mechanism that would retire an older failure after a corrected schedule is created.

## What This Changes

Deduplicate stale cron validation-failure announcements when a later successful schedule represents the same reminder intent.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by an approved reminder intent, and update cron tool guidance so non-default agents avoid the invalid `main` target path up front.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default-agent rejection and lack of intent-level supersession, but a focused current-main test is still needed to pin the stale replay producer.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and pin the producer before changing cron state or heartbeat delivery semantics.
