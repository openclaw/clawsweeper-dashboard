## Summary

This issue should stay open: current main still rejects the reported non-default-agent `sessionTarget="main"` cron add, and the inspected cron/system-event/heartbeat paths still do not have a schedule-intent supersession mechanism that would retire a stale failure after a corrected job succeeds.

## What This Changes

The issue requests cron failure replay deduplication so a corrected schedule with the same reminder intent suppresses an earlier validation failure announcement.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the reported validation failure and the absence of cron intent supersession, but a focused failing replay test still needs to pin the producer path.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and whether the fix belongs in cron state, tool-error recovery, or heartbeat/system-event replay before changing delivery semantics.
