## Summary

Keep open. Current main documents the `sessionTarget` constraint and has cron failure-announcement plumbing, but I found no cron intent/supersession state that would suppress an older failed add/update once a later schedule with the same intent succeeds.

## What This Changes

Request deduplicating cron add/update validation-failure announcements when a later successful job schedules the same agent/name/time intent.

## Best Possible Solution

Add internal cron intent supersession for failed add/update attempts and cover it with a regression test so a corrected same-intent schedule prevents stale failure delivery without changing the session-target rules.

## Reproduction Assessment

No high-confidence runtime reproduction was run. Source inspection confirms the reported validation rejection and the absence of a superseded-intent check in the failure-announcement path, but the full replay timing was not exercised.

## Solution Assessment

Likely yes: deduping or superseding by a stable internal intent key is the narrowest maintainable direction. The implementation should stay internal and avoid adding config or relaxing `sessionTarget` validation.

Implement a narrow internal cron failure-announcement supersession path so a failed `cron.add` or `cron.update` validation for an intended schedule is skipped if a later successful job lands with the same intent, such as agent id plus normalized job name plus scheduled fire time. Keep the existing `sessionTarget="main"` validation intact, do not add a user-facing config option, and avoid changing unrelated cron retry or delivery semantics.
