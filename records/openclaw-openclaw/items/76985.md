## Summary

The report should stay open: current main still has the reported non-default-agent `sessionTarget="main"` rejection and I did not find a cron intent/supersession mechanism that would retire an earlier stale failure after a corrected job succeeds.

## What This Changes

The issue asks for failed `cron.add` or `cron.update` announcements to be deduplicated by reminder intent when a later corrected schedule succeeds.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default `sessionTarget="main"` rejection and the absence of cron intent supersession, but the stale replay producer still needs a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and pin the replay producer before changing cron state or heartbeat delivery semantics.
