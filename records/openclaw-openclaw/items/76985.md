## Summary

Keep open: current main still has the precursor non-default `sessionTarget="main"` cron validation failure and I did not find a source-backed cron-intent supersession path that would suppress stale failed cron mutations after a corrected schedule lands. The related closed report is orthogonal skill/error-copy territory, and the merged reply-dedupe PR does not cover this cron replay path.

## What This Changes

The issue requests intent-level deduplication so corrected cron schedules suppress stale announce-on-failure replays from earlier failed `cron.add` or `cron.update` attempts.

## Best Possible Solution

Add stable cron intent supersession at the failure replay layer that actually owns the stale announcement, then cover failed-then-corrected scheduling with a focused regression or live proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run. Source inspection confirms the precursor validation failure and shows no cron-intent supersession path, but the scheduled-fire replay still needs a focused test or live repro.

## Solution Assessment

Unclear as an implementation plan: deduping by intent is the right product shape, but the fix should land where failed cron mutations are persisted or replayed, not blindly in cron job storage.
