## Summary

Keep open: current main still has the precursor non-default-agent cron rejection, rejected cron.add calls do not become stored jobs, and the generic failed mutating-tool replay state still lacks a cron intent key that would supersede a stale failure after a corrected same-intent schedule succeeds.

## What This Changes

The issue requests cron announce-on-failure replay deduplication so a stale failed scheduling attempt is superseded by a later successful schedule with the same intent.

## Best Possible Solution

Define a stable cron intent supersession contract at the failed cron tool-mutation replay boundary, then add tests proving a corrected same-intent cron.add clears the stale failure without suppressing distinct failed schedules.

## Reproduction Assessment

No high-confidence live reproduction was established; source inspection confirms the precursor validation failure, rejection-before-storage behavior, and missing cron intent supersession in failed mutating-tool state.

## Solution Assessment

Unclear: intent deduplication is directionally right, but the safest fix likely belongs in failed cron tool-mutation replay semantics and needs maintainer-approved intent rules.
