## Summary

The issue should stay open: current main still rejects the reported non-default-agent main-session cron add, and I found no intent/supersession state that would retire the earlier failure after a corrected schedule succeeds.

## What This Changes

A failed cron add or update can leave a stale failure announcement that should be deduplicated when a later corrected cron job represents the same operator intent.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update cron tool guidance for non-default agents.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction in this read-only pass. Source inspection verifies the validation failure and missing intent supersession, but the stale replay needs a focused test around the failure-to-announcement producer.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should first choose the canonical key and whether supersession belongs in cron creation, embedded-run failure payloads, or heartbeat system-event consumption.
