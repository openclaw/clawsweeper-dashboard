## Summary

Keep this open: current main still has the non-default-agent `sessionTarget="main"` rejection and I did not find an intent or superseded-failure mechanism that would retire a stale cron failure after a corrected schedule succeeds.

## What This Changes

Deduplicate stale cron failure announcements by logical schedule intent after a corrected `cron.add` or `cron.update` succeeds.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live end-to-end reproduction. Source inspection verifies the initial rejection and the lack of visible supersession in the relevant cron add, event queue, heartbeat replay, and failure-notification paths.

## Solution Assessment

Unclear. Intent-key dedupe is a plausible fix, but maintainers should choose the canonical key and the exact replay entry to supersede before changing cron failure delivery semantics.
