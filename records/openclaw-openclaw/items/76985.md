## Summary

This should stay open: current main still has the reported non-default-agent `sessionTarget="main"` rejection, and the inspected failure paths only clear the same mutating action, not a later corrected cron schedule with the same reminder intent.

## What This Changes

Deduplicate or supersede stale cron failure announcements when a corrected `cron.add` or `cron.update` with the same reminder intent succeeds.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection confirms the non-default `sessionTarget="main"` rejection and the absence of cron intent supersession, but a focused failing test still needs to pin the stale replay behavior.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and confirm the replay producer before changing cron state or heartbeat delivery semantics.
