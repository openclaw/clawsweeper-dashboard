## Summary

## What This Changes

Request deduplication or supersession for stale failed cron add/update announcements when a later corrected schedule represents the same cron intent.

## Best Possible Solution

Define a stable cron intent supersession contract at the failed cron tool-mutation replay boundary, with tests proving a corrected same-intent cron.add clears the stale failure without suppressing distinct failed schedules.

## Reproduction Assessment

No live reproduction was run, but source inspection confirms the precursor validation failure, rejection-before-storage behavior, and lack of a cron-intent supersession key in the failed tool-error clearing path.

## Solution Assessment

The requested intent dedupe is directionally right, but the best fix should likely be in failed cron tool-mutation replay semantics rather than cron job storage, with maintainer-approved intent rules.
