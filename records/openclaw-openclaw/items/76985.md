## Summary

Keep this open: current main still has the non-default-agent `sessionTarget="main"` rejection path and no cron intent/supersession mechanism that would retire an earlier failure after a corrected schedule succeeds.

## What This Changes

The issue requests that cron announce-on-failure replay skip stale validation failures when a later successful schedule represents the same reminder intent.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and keep the cron tool guidance aligned with the non-default-agent `sessionTarget` constraint.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the reported validation rejection and the absence of later-success supersession, but this read-only sweep did not exercise the full scheduled failure-replay path.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and pin the replay producer before changing cron state or heartbeat delivery semantics.
