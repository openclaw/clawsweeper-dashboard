## Summary

Keep this open: current main still has the reported non-default-agent `sessionTarget="main"` rejection, and I did not find cron failure replay state that can mark an earlier failure as superseded after a corrected schedule succeeds.

## What This Changes

Request cron failure replay dedupe by scheduling intent so corrected `cron.add` or `cron.update` calls suppress stale failure announcements.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and add the non-default-agent `sessionTarget` constraint to cron tool guidance.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default-agent rejection and the absence of visible intent supersession, but the stale replay producer needs a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and pin the replay producer before changing cron state or heartbeat delivery semantics.
