## Summary

This should stay open: current main still has the reported non-default-agent `sessionTarget="main"` rejection and I did not find an intent-level supersession path that would retire an earlier cron failure after a corrected schedule succeeds.

## What This Changes

The issue requests deduplicating stale cron failure announcements when a later corrected `cron.add` or `cron.update` creates the same scheduled reminder intent.

## Best Possible Solution

Add regression-covered cron failure supersession so a later successful schedule for the same reminder intent marks the earlier failure announcement obsolete, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection confirms the reported validation failure and the absence of a cron intent supersession check, but the stale replay should still be pinned with a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and confirm the exact stale replay producer before changing cron failure delivery semantics.
