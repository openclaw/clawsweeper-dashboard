## Summary

Keep open: current main still has the non-default-agent `sessionTarget="main"` rejection and I did not find an intent/supersession mechanism that would retire an earlier stale cron failure after a corrected schedule succeeds under a new job ID.

## What This Changes

The issue requests suppressing stale cron failure announcements when a later corrected `cron.add` or `cron.update` represents the same reminder intent.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update cron tool/docs guidance so non-default agents choose isolated agent-turn jobs before hitting the main-session guard.

## Reproduction Assessment

No, not as a live high-confidence reproduction. Source inspection verifies the non-default `sessionTarget="main"` rejection and the absence of intent/superseded state in the visible queue/replay paths, but I did not establish a failing current-main replay test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical intent fields and the owner layer before changing cron state or heartbeat delivery semantics.
