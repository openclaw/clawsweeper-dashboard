## Summary

Keep open: current main still has the reported cron validation failure path and I did not find an intent-level supersession path that would suppress a stale failed cron add/update after a corrected schedule lands. The related closed issue covered misleading sessionTarget guidance and was self-closed, but it does not cover this replay-dedup behavior.

## What This Changes

Request deduping cron add/update failure replay by cron intent so a later successful schedule suppresses stale rejection announcements.

## Best Possible Solution

Add intent-level supersession for failed cron add/update attempts at the layer that persists or replays the failure, and cover failed-then-corrected scheduling with a regression test that proves the stale rejection is not delivered.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this read-only review. Source inspection confirms the precursor validation failure and the absence of an obvious cron-intent supersession path, but the scheduled-fire replay needs a focused test or live repro.

## Solution Assessment

The proposed intent-dedup direction looks like the right shape, but the fix should land where failed mutating cron attempts are actually persisted or replayed. Per-job cron failure alerts are probably the wrong layer because invalid cron.add does not create a stored job.
