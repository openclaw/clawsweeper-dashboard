## Summary

Keep this open: current main still has the non-default-agent `sessionTarget="main"` rejection and I did not find cron intent/supersession state that would retire an earlier failure after a corrected job succeeds.

## What This Changes

Request to supersede stale cron validation-failure announcements when a later successful schedule with the same reminder intent is created.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the non-default `sessionTarget="main"` rejection and the absence of intent supersession, but the stale replay path still needs a focused failing test.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and confirm the replay producer before changing cron state or heartbeat delivery semantics.
