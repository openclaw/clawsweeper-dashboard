## Summary

Keep open: current main still appears to reject the reported bad non-default-agent `sessionTarget="main"` cron add, but I did not find a cron intent-level supersession path that would retire a stale failure after a later corrected schedule succeeds.

## What This Changes

Request cron failure replay suppression when a later corrected `cron.add` or `cron.update` with the same reminder intent succeeds, plus clearer non-default-agent `sessionTarget` guidance.

## Best Possible Solution

Add regression-covered cron failure supersession keyed to a maintainer-approved reminder intent, and update the cron tool guidance so non-default agents avoid the invalid `main` target path.

## Reproduction Assessment

No, not as a live high-confidence reproduction. Source inspection confirms the reported validation failure and the absence of an obvious intent-level supersession check, but a focused test is still needed to pin the stale replay producer.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should first choose the canonical intent key and decide whether the fix belongs in cron state, embedded tool-error replay, or system-event delivery.
