## Summary

Keep this open: current main still rejects non-default-agent `sessionTarget="main"` cron creation and I did not find an intent/supersession mechanism that would retire an earlier cron failure event after a corrected schedule succeeds.

## What This Changes

Request that cron failure replay announcements be superseded when a corrected `cron.add` or `cron.update` with the same reminder intent is successfully scheduled.

## Best Possible Solution

Add regression-covered cron failure supersession keyed by a maintainer-approved reminder intent, and update the cron tool guidance for non-default agents.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection verifies the reported validation rejection and the absence of an intent/supersession hook, but I did not establish a failing current-main run that replays the stale failure after the corrected job succeeds.

## Solution Assessment

Unclear. Intent-key dedupe is plausible, but maintainers should choose the canonical key and pin the exact replay producer before changing cron state or heartbeat delivery semantics.
