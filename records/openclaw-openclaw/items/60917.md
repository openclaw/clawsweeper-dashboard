## Summary

Keep open. Current main still serializes the dashboard form payload without an original-aware redaction-sentinel filter, while gateway writes still fail closed when `__OPENCLAW_REDACTED__` cannot be restored from the original config. The linked focused fix in #60971 is closed unmerged, so the reported save failure remains a valid narrow UI bug.

## What This Changes

The issue requests that dashboard form saves omit or restore redacted `gateway.remote.token` placeholders so unrelated settings can save when the original config has no remote token.

## Best Possible Solution

Add a UI-side, original-aware sanitization step in the dashboard form save path: drop redaction sentinels for sensitive paths that are absent from the original snapshot/raw config, preserve placeholders that correspond to existing secrets so the gateway can restore them, and keep the backend reserved-sentinel rejection unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to put `__OPENCLAW_REDACTED__` at `configForm.gateway.remote.token` while the original snapshot lacks that path, then save in form mode; current UI serialization includes the sentinel and the gateway restore path rejects it as unrestorable.

## Solution Assessment

Yes. The narrowest maintainable fix is a UI-side original-aware filter before form-mode `config.set`/`config.apply`; relaxing `restoreRedactedValues` would weaken the intentional fail-closed contract for reserved sentinel strings.
