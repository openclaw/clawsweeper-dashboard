## Summary

Keep open: the feature is not on current main and this is still a plausible implementation path, but the branch is not ready because it leaves public config, gateway, UI, and current docs surfaces out of sync and lacks real behavior proof.

## What This Changes

The PR adds an opt-in `readonly` tool profile across the core tool catalog, runtime config parsing, gateway catalog typing, tests, and documentation.

## Best Possible Solution

Add `readonly` only as a complete accepted policy surface: align catalog policy, public types/schema, gateway protocol schemas, Quick Settings, current docs, focused tests, and product/security agreement on the allowed tool set.

## Reproduction Assessment

Yes for the review blockers. Static source inspection of current main and the PR diff shows that the branch advertises `readonly` through catalog/default-profile paths while shared config types, gateway profile-result schemas, Quick Settings, and current docs still enumerate only the old profiles.

## Solution Assessment

No, not as submitted. The direction is reasonable, but the maintainable solution is a complete additive contract/UI/docs update with accepted read-only membership and real behavior proof, not a catalog-local profile addition with partial gateway typing.
