## Summary

## What This Changes

The PR adds an opt-in `readonly` tool profile across the core tool catalog, runtime config parsing, gateway catalog typing, tests, and docs.

## Best Possible Solution

Accept `readonly` only as a complete additive policy surface: align catalog policy, public config types/schema, gateway protocol schemas, UI pickers, canonical docs, focused tests, and product/security agreement on the allowed read set.

## Reproduction Assessment

Yes for the review blockers. Static source inspection of current main and the PR diff shows the branch advertises `readonly` in catalog paths while shared config types, gateway profile-result schemas, Quick Settings, and canonical docs still enumerate only the old profiles.

## Solution Assessment

No, not as submitted. The direction may be reasonable, but the maintainable solution is a complete additive contract/UI/docs update with accepted read-only membership and real behavior proof, not a partial catalog-local profile addition.
