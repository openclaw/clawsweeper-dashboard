## Summary

Keep open. The underlying block-stream final-payload bug is valid and still source-reproducible on current main, but this PR is not mergeable as-is because its per-payload `hasSentPayload()` filter can re-send coalesced streamed text and the PR lacks inspectable real behavior proof.

## What This Changes

The PR changes block-stream final payload filtering so `hasSentPayload()` drops only streamed payloads and updates the matching unit test.

## Best Possible Solution

Rebase or replace this with a current-main repair that delivers genuinely unsent post-tool final text while preserving coalesced-stream suppression, direct-send dedupe, media stripping, and error fallback behavior.

## Reproduction Assessment

Yes, at source level. Current main's `buildReplyPayloads` test still expects a successful streamed turn to drop a text final even when `hasSentPayload()` returns false; I did not run live channel validation in this read-only review.

## Solution Assessment

No as submitted. The direction is valid, but the PR's per-payload `hasSentPayload()` filter can duplicate coalesced streamed text and is stale against current media/error fallback paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Preserve coalesced stream suppression:** `src/auto-reply/reply/agent-runner-payloads.ts:208-212`
  - body: This `hasSentPayload()`-only filter checks each final payload independently. When block coalescing sends chunks such as `A` and `B` as one outbound `A\n\nB` payload, later split final payloads `A` and `B` do not match the coalesced content key or the full streamed-text comparison, so they can be sent again at turn end. Keep coverage-aware suppression for text already covered by the stream while allowing genuinely new post-tool final text through.
  - confidence: 0.86
