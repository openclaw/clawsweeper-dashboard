## Summary

Keep open: current main and the latest release still suppress ordinary text-only final payloads after a successful block stream, but this submitted branch is not merge-ready because its PR-head `hasSentPayload`-only filter can duplicate coalesced streamed chunks and still needs real behavior proof.

## What This Changes

The branch changes `buildReplyPayloads` so block-streaming final payloads are filtered through `blockReplyPipeline.hasSentPayload()` instead of being dropped wholesale, and updates the focused unit test accordingly.

## Best Possible Solution

Update the fix on current main so text-only final payloads are delivered only when not covered by streamed or directly sent content, while preserving the existing coalescing, media, error, fallback-notice, and route-aware dedupe protections.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.6.6` still suppress text-only final payloads after a successful non-aborted block stream, and the focused test still expects that drop; I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is valid, but the submitted PR-head filter is too broad for coalesced streams and needs to be reworked against current main's richer dedupe/media/error handling.
