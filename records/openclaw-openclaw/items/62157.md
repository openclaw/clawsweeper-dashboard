## Summary

Keep open. The PR targets a real block-streaming final-delivery problem that current main still has at the payload-filter level, but the submitted patch is not merge-ready: GitHub reports the branch as dirty/unrebaseable against main, and the hasSentPayload-only filter can resend already-streamed coalesced text.

## What This Changes

The PR removes successful block-stream final-payload suppression in buildReplyPayloads and updates the unit test so only payloads reported by hasSentPayload are dropped.

## Best Possible Solution

Rework this on current main as a content-coverage-aware block-stream final dedupe: deliver genuinely unsent post-tool final text, keep successful-stream suppression for coalesced content that was already flushed, and preserve the current media/error fallback behavior with regression coverage.

## Reproduction Assessment

Yes at function level. Current main's buildReplyPayloads still drops text-only final payloads when the block pipeline reports a successful stream, and the checked-in unit test asserts that behavior even when hasSentPayload returns false.

## Solution Assessment

No as submitted. The direction is useful, but a plain hasSentPayload-only filter is not the safest fix because it can re-deliver coalesced streamed chunks; the fix should distinguish content already covered by the stream from genuinely new final text.
