## Summary

Keep open. Current main still drops unsent text-only final payloads after successful block streaming, so the reported delivery bug remains, but this PR's hasSentPayload-only filter is unsafe because coalesced streamed text can be re-sent as split final payloads and a current-main repair must preserve the newer media/error fallback behavior.

## What This Changes

The PR removes successful block-stream final-payload suppression in buildReplyPayloads and updates a unit test so hasSentPayload decides which final payloads are dropped.

## Best Possible Solution

Rework this on current main as content-coverage-aware final-payload dedupe: deliver unsent post-tool text, keep already-streamed/coalesced text suppressed, and retain the existing media/error fallback paths.

## Reproduction Assessment

Yes. At function level, the checked-in buildReplyPayloads test shows successful block streaming drops a text-only final even when hasSentPayload returns false; the duplicate risk follows from the coalescer and hasSentPayload coverage logic.

## Solution Assessment

No as submitted. The direction is valid, but a per-payload hasSentPayload-only filter is not the narrowest maintainable fix because it can re-deliver text already covered by a coalesced block stream.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Preserve coalesced stream suppression:** `src/auto-reply/reply/agent-runner-payloads.ts:208-212`
  - body: This new blockStreamingEnabled branch checks each final payload independently with hasSentPayload. When block coalescing sends chunks like A and B as one outbound `A\n\nB` payload, final split payloads A and B do not match the coalesced content key or full streamed-text coverage, so they can pass through and be sent again at turn end. Keep coverage-based suppression for text already covered by the stream while allowing genuinely new post-tool final text through.
