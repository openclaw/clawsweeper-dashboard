## Summary

Keep open: current main still has the source-level final-text drop, but this PR’s `hasSentPayload()`-only fix can re-send already streamed coalesced text and still lacks inspectable real behavior proof.

## What This Changes

The PR changes block-streaming final-payload filtering so final payloads are filtered by `hasSentPayload()` instead of being dropped wholesale after a successful stream.

## Best Possible Solution

Rework the fix on current main so genuinely unsent post-tool final text is delivered while coalesced-stream, media, error, rich-content, and direct-send dedupe behavior is preserved, then attach redacted live proof.

## Reproduction Assessment

Yes, at source level. Calling `buildReplyPayloads` with block streaming enabled, `didStream() === true`, `isAborted() === false`, `hasSentPayload() === false`, and a text-only final payload still returns no reply payloads on current main.

## Solution Assessment

No as submitted. The per-payload `hasSentPayload()` filter can duplicate already streamed coalesced chunks, so the maintainable fix needs coverage-aware suppression while allowing genuinely new final text through.
