## Summary

Keep open: the reported delivery bug is still source-reproducible on current main, but this PR’s `hasSentPayload()`-only filter can resend coalesced streamed text and still needs inspectable real behavior proof before merge.

## What This Changes

The PR removes successful-stream final-payload suppression in `buildReplyPayloads` and updates the focused unit test to keep unsent final text after block streaming.

## Best Possible Solution

Rework the fix on current main so genuinely unsent post-tool final text is delivered while coalesced-stream, media, error, and direct-send dedupe behavior is preserved, then attach redacted live proof.

## Reproduction Assessment

Yes, at source level. Current main still drops text-only final payloads after a successful block stream, and the focused test explicitly expects that drop even when `hasSentPayload()` would not match the final text.

## Solution Assessment

No as submitted. Per-payload `hasSentPayload()` filtering is too weak for coalesced streams; the safer fix needs coverage-aware suppression for already-streamed text while allowing genuinely new final text through.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve coalesced stream suppression:** `src/auto-reply/reply/agent-runner-payloads.ts:208-212`
  - body: This `hasSentPayload()`-only filter checks each final payload independently. When block coalescing sends chunks such as `A` and `B` as one outbound `A\n\nB` payload, later split final payloads `A` and `B` do not match the coalesced content key or full streamed text, so they can be sent again at turn end. Keep coverage-aware suppression for text already covered by the stream while allowing genuinely new post-tool final text through.
  - confidence: 0.86
