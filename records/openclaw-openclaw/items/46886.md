## Label Justifications

- `P2`: This is a real multi-agent routing bug with limited blast radius and a clear focused implementation path.
- `impact:message-loss`: Wake events intended for a non-default agent can be routed to the wrong agent, so the intended target never receives the event.
- `impact:session-state`: Misrouted wake events can update the default agent session while leaving the intended agent session unchanged.

## Summary

Keep open. Current main still has a source-reproducible wake-targeting gap: wake accepts and forwards `sessionKey` in several paths, but `agentId` is not typed or forwarded end to end, and the direct agent `cron` tool still calls wake with only `{ mode, text }`.

## What This Changes

The issue asks for cron wake events to accept an optional `agentId` so multi-agent wakes target the intended non-default agent instead of the default agent.

## Best Possible Solution

Land one refreshed fix, likely https://github.com/openclaw/openclaw/pull/83738 or a replacement, that threads `agentId` and `sessionKey` through the wake schema, Gateway handler, CronService contract, timer, direct agent `cron` tool, CLI/generated clients, docs, and focused regressions while preserving current `sessionKey` behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows wake callers and service boundaries accept or forward only `sessionKey`, so an `agentId` wake target is dropped before reaching the existing cron target resolver.

## Solution Assessment

Yes. The maintainable fix is additive `agentId` plumbing through the existing wake/cron target contracts; the open PRs should be refreshed, consolidated, or replaced rather than opening another competing branch.
