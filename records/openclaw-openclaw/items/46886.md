## Label Justifications

- `P2`: This is a real multi-agent wake-routing bug with clear affected files and limited blast radius.
- `impact:message-loss`: A wake intended for a non-default agent can route to the default agent, leaving the intended target without the event.
- `impact:session-state`: Misrouted wake events can update the default agent session while the intended agent session misses the expected state change.

## Summary

Keep open. Current main and latest v2026.5.20 still leave the agent-facing cron wake path without an explicit `agentId`: the direct tool call sends only `mode` and `text`, while the validated Gateway/CronService wake path can carry `sessionKey` but not `agentId`; the two related open PRs are implementation candidates, not merged fixes.

## What This Changes

The issue asks for cron wake events to accept and forward an optional agent target so non-default agents can be woken instead of routing through the default agent.

## Best Possible Solution

Land or consolidate one additive fix that threads `agentId` through the cron tool wake action, `WakeParamsSchema`, Gateway handler, CronService contract, and timer enqueue/heartbeat calls while preserving existing `sessionKey` and no-origin default behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the direct cron wake path sends only `{ mode, text }`, and the validated Gateway/CronService/timer wake path can carry only `sessionKey`, not explicit `agentId`; I did not run a live multi-agent route in this read-only review.

## Solution Assessment

Yes. The maintainable direction is additive target plumbing through the existing wake contracts, with the linked open PRs reviewed, refreshed, or consolidated rather than opening a competing repair branch.
