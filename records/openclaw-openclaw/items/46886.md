## Summary

Keep open. Current main still drops direct cron wake target metadata before the request reaches the Gateway cron service, so direct cron wake calls cannot reliably target a non-default agent even though scheduled cron job execution already carries agent and session targeting.

## What This Changes

Add optional agentId/sessionKey targeting to the cron wake action so manual wake events can route to a specific agent/session instead of the default agent.

## Best Possible Solution

Land one focused additive wake-target contract update that threads optional agentId/sessionKey through the cron tool, Gateway wake schema/handler, CronServiceContract, service wake/wakeNow/timer wake path, docs/changelog, and regression coverage while preserving default behavior when no target is supplied.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: call cron wake with target metadata and the tool/Gateway/service path narrows the request back to mode/text before enqueueing an unscoped system event.

## Solution Assessment

Yes. The best fix is an additive target-plumbing change through the existing wake contract and cron target-resolution path; using sessions_send remains a workaround, not a replacement for the missing wake target support.
