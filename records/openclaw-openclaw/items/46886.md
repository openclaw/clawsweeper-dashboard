## Summary

Keep open. Current `main` still drops target data on the direct cron wake path before it reaches cron execution, even though scheduled cron jobs and the Gateway cron adapter already support agent/session targeting. An overlapping open PR, #57199, covers the lower system wake contract but is unmergeable and does not add cron tool forwarding, so it is not enough to close this issue.

## What This Changes

The issue requests optional `agentId` and `sessionKey` support for the direct `cron` tool `wake` action so wake events can target a specific agent/session instead of the default agent.

## Best Possible Solution

Add an additive target contract for direct wake calls and thread optional `agentId`/`sessionKey` through the cron tool, Gateway `WakeParams`/handler, `CronServiceContract`, `CronService.wake`, `ops.wakeNow`, and `timer.wake`, reusing the existing Gateway cron target resolver and preserving default no-target behavior and `WakeParams` forward compatibility.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: call the cron tool with `action: "wake"` plus `agentId` or `sessionKey`; current main drops those fields in the tool branch and then queues an unscoped system event and heartbeat wake.

## Solution Assessment

Yes. The requested target plumbing is the narrowest maintainable fix because the scheduled cron path and Gateway cron adapter already support the target semantics; the fix should avoid regressing the existing forward-compatible wake schema behavior and should be coordinated with #57199.
