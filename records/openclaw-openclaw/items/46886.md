## Summary

Keep open. Current main still drops direct cron wake target data before it reaches cron execution, while scheduled cron jobs and the Gateway cron adapter already have agent/session target plumbing; the related open PRs overlap adjacent wake paths but do not fully resolve this cron tool path.

## What This Changes

Allow the cron tool `wake` action to accept and forward optional `agentId` and `sessionKey` so manual wake events can target non-default agents or sessions.

## Best Possible Solution

Add an additive target contract for direct wake calls and thread optional `agentId`/`sessionKey` from the cron tool through Gateway wake validation/handling and the cron service wake stack, preserving current default behavior when no target is supplied.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: invoke the cron tool with `action: "wake"`, `text`, `mode`, and `agentId` or `sessionKey`; current main only forwards `{ mode, text }` and then queues an unscoped system event.

## Solution Assessment

Yes, the requested target plumbing is the narrowest maintainable fix because scheduled cron jobs and the Gateway cron adapter already support the same target semantics; the implementation should be coordinated with #57199 because it overlaps the lower wake contract.
