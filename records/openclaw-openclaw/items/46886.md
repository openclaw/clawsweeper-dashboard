## Summary

Keep open. Current main still drops target data on the direct cron wake path before it reaches cron execution, while scheduled cron jobs and the Gateway cron adapter already support agent/session targeting. Open PR #57199 overlaps the shared wake path but is unmergeable and still does not cover the cron tool wake branch, so this issue is not resolved on main.

## What This Changes

The issue requests optional `agentId` and `sessionKey` support for the direct `cron` tool `wake` action so wake events can target a specific agent/session instead of the default agent.

## Best Possible Solution

Add an additive target contract for direct wake calls and thread optional `agentId`/`sessionKey` through the cron tool, Gateway `WakeParams`/handler, `CronServiceContract`, `CronService.wake`, `ops.wakeNow`, and `timer.wake`, reusing the existing Gateway cron target resolver and preserving default no-target behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: call the cron tool with `action: "wake"` plus `agentId` or `sessionKey`; current main drops those fields in the tool branch and then queues an unscoped system event and heartbeat wake.

## Solution Assessment

Yes. The requested target plumbing is the narrowest maintainable fix because scheduled cron jobs and the Gateway cron adapter already have target semantics; the implementation should be coordinated with #57199 and keep `WakeParams` forward compatible.
