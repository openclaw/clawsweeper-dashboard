## Summary

Keep this issue open: current main and the latest release still leave the cron tool wake path without an agent target, and the active fix is the open linked PR rather than merged code.

## What This Changes

The issue asks for cron wake routing to accept or infer an agent/session target instead of always waking the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83738, or replace it with an equivalent focused fix that threads the originating `sessionKey` and `agentId` through cron tool, gateway wake, protocol clients, cron service, and targeted regression proof.

## Reproduction Assessment

Yes, at source level: on current main, `cron` tool `action: "wake"` sends only `{ mode, text }`, while the gateway/schema/service path lacks an `agentId` field. A wake from a non-default agent session therefore has no target identity to preserve through this path.

## Solution Assessment

Yes for keeping this issue open: the linked PR is the right active repair path because it updates the tool, protocol, gateway, service, tests, and proof together. The issue should close only after that PR or an equivalent replacement lands.
