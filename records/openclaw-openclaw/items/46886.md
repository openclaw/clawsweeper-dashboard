## Summary

Keep open. Current main still exposes cron `wake` as `{ mode, text }` only and drops any target before the service wake path, while active related PRs cover lower-level wake/session-key work rather than the cron tool `agentId` surface reported here.

## What This Changes

The issue requests targetable cron wake events so a non-default agent can be woken instead of always routing through the default agent.

## Best Possible Solution

Accept or reject the lower-level wake `sessionKey` contract first, then extend the cron tool to infer or accept the intended target and pass it through the gateway/service wake path with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: the cron tool, gateway handler, service contract, and timer wake implementation have no typed path for `agentId` or `sessionKey`. I did not establish a live failing multi-agent run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the supported target contract. The narrow implementation should likely build on the active typed `sessionKey` work, but adding cron-tool `agentId` semantics directly is a product/API choice.
