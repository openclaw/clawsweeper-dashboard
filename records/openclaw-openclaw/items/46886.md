## Summary

Keep open. Current `main` still lacks an `agentId` wake target through the direct agent `cron` tool and the Gateway/CronService wake contract; the related open PR at https://github.com/openclaw/openclaw/pull/57199 covers part of the surface but is conflicting and still needs refresh/proof.

## What This Changes

The issue asks for cron wake actions to accept an `agentId` target so direct wake events can route to a non-default agent.

## Best Possible Solution

Add `agentId` end to end through the wake protocol, Gateway handler, CronService contract, timer forwarding, agent `cron` tool, generated clients, docs, and focused regression tests while preserving current `sessionKey` behavior.

## Reproduction Assessment

Yes. Source inspection on current `main` shows direct `cron` wake calls and the wake service boundary still accept or forward only `sessionKey`, not `agentId`.

## Solution Assessment

Yes. The maintainable direction is an additive target-plumbing update through the existing wake contract, coordinated with or replacing https://github.com/openclaw/openclaw/pull/57199 so the current `sessionKey` semantics are not regressed.
