## Summary

Keep open. Current main still drops `agentId` on the wake path: the direct agent `cron` tool, wake protocol validation surface, Gateway handler, CronService contract, and timer implementation all forward only `sessionKey`; the related open PR covers much of the desired plumbing but is conflicting, lacks accepted real behavior proof, and does not cover the direct agent `cron` tool wake path.

## What This Changes

The issue requests adding `agentId` targeting to cron/system-event wake so wake events can route to a non-default agent instead of always using the default agent.

## Best Possible Solution

Add `agentId` end to end through the wake schema, Gateway handler, CronService contract, timer forwarding, direct agent `cron` tool, generated clients, docs, and focused regression tests while preserving current `sessionKey` behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows direct `cron` wake calls and the wake service boundary still accept or forward only `sessionKey`, not `agentId`, so an `agentId` wake target is dropped before reaching the existing cron target resolver.

## Solution Assessment

Yes for the requested direction, but not by merging the current related PR as-is. The maintainable fix is an additive wake-target plumbing update coordinated with or replacing https://github.com/openclaw/openclaw/pull/57199 so current `sessionKey` semantics and generated surfaces are preserved.
