## Summary

Keep open. Current main has picked up sessionKey support for wake/session routing, but the reported agentId path is still missing from the cron tool and the wake service contract, so cron wake events still cannot directly target a non-default agent by agentId.

## What This Changes

The issue requests optional agentId support for cron wake events so a wake can target a non-default agent instead of the configured default agent.

## Best Possible Solution

Land or replace a narrow additive change that adds agentId to the wake contract and cron tool while preserving current sessionKey behavior, generated protocol surfaces, docs, and focused wake routing tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cron({ action: "wake", agentId: "local-dev" }) cannot work today because the cron tool sends only mode/text and the downstream wake contract has no agentId field.

## Solution Assessment

Partly. Adding agentId end to end is the right direction, but the implementation should be refreshed around the current sessionKey semantics and include the cron tool surface, not only the system CLI path.
