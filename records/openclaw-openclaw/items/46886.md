## Summary

Keep open. Current main still drops direct cron wake target data at the cron tool, Gateway wake handler, and CronService wake path; scheduled cron jobs and the Gateway cron adapter already carry agent/session targets, while the related open PRs overlap adjacent wake routing but are not merged and do not conclusively cover cron tool forwarding.

## What This Changes

Request to let the cron wake action accept optional `agentId` and `sessionKey` so direct wake events can target a non-default agent or session instead of falling back to the default agent.

## Best Possible Solution

Thread optional `agentId` and `sessionKey` through the cron tool wake schema, Gateway wake validation/handler, and `CronService.wake`/`wakeNow`, reusing the existing Gateway cron target resolution and preserving default behavior when no target is supplied.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: call the cron tool wake action with `text`, `mode`, and target fields; current main only forwards `{ mode, text }` and the service queues an unscoped system event.

## Solution Assessment

Yes. The narrow maintainable fix is additive target plumbing through the existing wake contracts, coordinated with #57199 rather than adding a fallback or new configuration policy.
