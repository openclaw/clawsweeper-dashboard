## Label Justifications

- `P2`: This is a real multi-agent cron wake routing bug with a clear but bounded gateway/cron surface.
- `impact:message-loss`: A wake intended for a non-default agent can be routed to the default lane, so the intended agent never receives the event.
- `impact:session-state`: Misrouted wake events can update or pollute the default agent session while leaving the intended agent/session unchanged.

## Summary

Keep open. Current main and the latest release still do not carry `agentId` through direct cron wake; the source path can forward only `sessionKey`, while the related canonical implementation PR is open and draft rather than merged.

## What This Changes

The issue asks OpenClaw to let cron wake target a non-default agent by accepting and carrying `agentId`, optionally with `sessionKey`, through the wake path.

## Best Possible Solution

Land one coordinated additive fix, likely through https://github.com/openclaw/openclaw/pull/83738 or a narrow replacement, that threads `agentId` through the cron tool, wake protocol, Gateway handler, CronService contract, timer enqueue/heartbeat calls, CLI/docs, and focused tests while preserving no-origin behavior.

## Reproduction Assessment

Yes, at source level. Current main sends only `{mode, text}` from the cron tool and the validated Gateway/CronService/timer path can forward only `sessionKey`, not `agentId`; no live multi-agent route was run in this read-only review.

## Solution Assessment

Yes, the maintainable direction is additive routing through the existing wake and cron target contracts. The linked PR is the best current vehicle, but it should be reviewed or replaced before closing this issue.
