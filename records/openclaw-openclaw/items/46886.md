## Summary

Keep open. Current main still drops direct cron wake target data at the tool, Gateway wake handler, and CronService wake path, and the latest shipped tag shows the same Gateway/timer behavior; the related system-event wake PR remains open and the hook-routing PR is adjacent rather than a cron-tool fix.

## What This Changes

The issue asks for cron wake actions to accept optional agentId/sessionKey data so multi-agent wake events can target a non-default agent or session.

## Best Possible Solution

Land one focused, additive wake-target contract update that threads optional agentId/sessionKey through the cron tool, Gateway wake schema/handler, CronServiceContract, service wake/wakeNow/timer wake path, docs/changelog, and focused regression tests while preserving default main-session behavior.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the problem: call cron wake with target metadata and the tool/Gateway/service path narrows the call back to mode/text before enqueueing an unscoped system event.

## Solution Assessment

Yes. The maintainable direction is additive target plumbing through the existing wake contract and shared cron target resolution; the current open PR should be refreshed or replaced rather than worked around with a fallback policy.
