## Label Justifications

- `P2`: This is a real multi-agent routing bug with limited blast radius and a clear focused implementation path.
- `impact:message-loss`: Wake events intended for a non-default agent can be delivered to the wrong agent session, effectively losing the intended message for the target agent.
- `impact:session-state`: Misrouted wake events can pollute the default agent session and leave the intended agent session without the expected state update.

## Summary

Keep open. Current `main` still drops `agentId` on the wake path: the direct agent `cron` tool, wake protocol schema, Gateway wake handler, CronService contract, and timer implementation only type or forward `sessionKey`, while the lower cron target resolver already understands `agentId`.

## What This Changes

The issue asks for cron wake/system-event wake calls to accept and forward an optional `agentId` so wake events can target a non-default agent.

## Best Possible Solution

Add `agentId` end to end through the wake schema, Gateway handler, CronService contract, timer forwarding, direct agent `cron` tool, generated clients, docs, and focused regression tests while preserving current `sessionKey` behavior.

## Reproduction Assessment

Yes. Source inspection on current `main` shows wake callers and service boundaries accept or forward only `sessionKey`, so any `agentId` wake target is dropped before reaching the existing cron target resolver.

## Solution Assessment

Yes. The maintainable fix is additive `agentId` plumbing through the existing wake/cron target contracts; the related PR should be refreshed or replaced because it conflicts and misses the direct agent `cron` tool path.
