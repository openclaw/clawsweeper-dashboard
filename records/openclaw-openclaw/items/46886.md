## Label Justifications

- `P2`: This is a real multi-agent wake-routing bug with limited blast radius and clear affected files.
- `impact:message-loss`: A wake intended for a non-default agent can be routed to the default agent, so the intended target never receives the event.
- `impact:session-state`: Misrouted wake events can pollute the default agent session while leaving the intended agent session without the expected state update.

## Summary

Keep open. Current main still cannot carry `agentId` through cron wake: the tool, protocol, Gateway handler, service contract, and timer wake path only forward `sessionKey`, and the related implementation PRs are still open rather than merged.

## What This Changes

The issue requests optional `agentId` support for cron wake so wake events can target a non-default agent instead of always falling back to the default agent.

## Best Possible Solution

Land one coordinated additive fix that threads `agentId` through the direct cron tool wake action, `WakeParamsSchema`, Gateway handler, CronService contract, timer enqueue/heartbeat calls, and CLI surface while preserving current `sessionKey` and no-origin behavior.

## Reproduction Assessment

Yes, at source level. Current main shows the direct cron wake path sends only `{ mode, text }`, and the validated Gateway/CronService/timer wake path can carry only `sessionKey`, not `agentId`; I did not run a live multi-agent route in this read-only review.

## Solution Assessment

Yes. The maintainable fix is additive `agentId` plumbing through the existing wake/cron target contracts, with the linked open PRs reviewed, refreshed, or consolidated rather than opening a competing repair branch.
