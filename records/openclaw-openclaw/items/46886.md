## Summary

Keep open. Current main and the latest shipped release still cannot carry an `agentId` through the direct wake path; the related open PR covers part of the change but is conflicting and still misses direct agent `cron` tool follow-through.

## What This Changes

Request to let cron wake/system-event wake target a non-default agent by accepting and forwarding an optional `agentId`.

## Best Possible Solution

Add `agentId` end to end through the wake protocol, Gateway handler, CronService contract, timer forwarding, agent `cron` tool, generated clients, docs, and focused regression tests while preserving current `sessionKey` behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the direct `cron` wake path and wake service boundary still accept or forward only `sessionKey`, not `agentId`.

## Solution Assessment

Yes. The maintainable solution is an additive target-plumbing update through the existing wake contract, coordinated with or replacing the related open PR so current `sessionKey` behavior is preserved.
