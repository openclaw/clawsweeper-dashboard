## Summary

Keep open. Current main still drops target data on the direct cron wake path before it reaches the cron service, so direct wake events remain scoped to the default/main agent while scheduled cron jobs already preserve agent/session targets.

## What This Changes

The issue requests optional `agentId` and `sessionKey` support for the direct `cron` tool `wake` action so wake events can target a specific agent/session instead of the default agent.

## Best Possible Solution

Add an additive target contract for direct wake calls and thread optional `agentId`/`sessionKey` through the cron tool, Gateway `WakeParams`/handler, `CronServiceContract`, `CronService.wake`, `ops.wakeNow`, and `timer.wake`, reusing existing target resolution and preserving default no-target behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call the cron tool with `action: "wake"` plus `agentId` or `sessionKey`; current main drops those fields in the tool branch, handler, and service wake path before queuing an unscoped event.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing wake plumbing and target resolver semantics rather than inventing a separate routing path for the cron tool.
