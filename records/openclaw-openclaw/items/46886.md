## Label Justifications

- `P2`: The bug affected cron wake routing in multi-agent and non-default-session setups but was limited to a bounded workflow.
- `impact:message-loss`: Wake-triggered output could fail to return to the intended conversation or be routed elsewhere.
- `impact:session-state`: The wake event was previously associated with the wrong agent or session context.

## Summary

Current main now routes cron wake events with the originating or explicitly supplied session and agent identity, fixing the default-agent misrouting reported here; the implementation is not yet in the latest release.

## What This Changes

Allow cron wake events to target the originating or explicitly selected agent and session instead of always routing through the default agent.

## Best Possible Solution

Keep the canonical origin-aware wake routing now on main and let it ship in the next release without adding a parallel targeting mechanism.

## Reproduction Assessment

Yes at source level. The old path omitted agent targeting, the merged related PR supplied live before-and-after transport proof, and current focused tests cover originating, explicit, and agent-only wake targets.

## Solution Assessment

Yes. The implementation carries canonical session and agent identity through the existing tool, gateway, cron service, heartbeat, and delivery-context boundaries rather than introducing a competing routing path.
