## Summary

Keep open: this PR fixes a real dispatcher hook bypass, but the current patch is not merge-ready because it can make `message_sending` run twice on Telegram native-command replies and the supplied proof does not cover that downstream visible path. Several open sibling PRs target the same bug, so maintainers should pick one canonical branch and require exactly-once hook behavior before merge.

## What This Changes

The PR composes channel/custom `beforeDeliver`, `reply_payload_sending`, and `message_sending` in the plain and buffered auto-reply dispatchers, with regression tests for custom `beforeDeliver` paths.

## Best Possible Solution

Land one canonical hook-composition fix that runs `message_sending` exactly once per visible reply across plain, buffered, Telegram native, and downstream delivery paths, with focused tests and real Telegram or equivalent live proof.

## Reproduction Assessment

Yes. Current main deterministically skips `message_sending` whenever `dispatcherOptions.beforeDeliver` is present, and Telegram supplies such hooks; this review also found a source-level duplicate-hook path introduced by the PR.

## Solution Assessment

No. The shared dispatcher layer is the right place to fix the bypass, but this patch needs exactly-once ownership so paths like Telegram native-command delivery do not run `message_sending` both before and inside `deliverReplies`.
