## Summary

Keep open: the underlying `message_sending` bypass is real, but this PR is not merge-ready because it unconditionally adds the hook where Telegram already runs it at delivery time, so visible replies can be rewritten, cancelled, or audited twice; the supplied terminal proof also does not cover the buffered Telegram path where that matters.

## What This Changes

The branch changes auto-reply dispatcher hook composition so channel/custom `beforeDeliver` is followed by `reply_payload_sending` and `message_sending`, with dispatcher regression tests for plain and buffered paths.

## Best Possible Solution

Land one canonical fix that restores `message_sending` for configured `beforeDeliver` paths while giving Telegram and other delivery-time hook owners exactly one `message_sending` pass, with focused tests and real Telegram or equivalent proof.

## Reproduction Assessment

Yes. Source inspection shows current `main` and `v2026.6.6` skip `message_sending` when `dispatcherOptions.beforeDeliver` is present, and the PR introduces a source-level duplicate Telegram path through dispatcher `beforeDeliver` followed by `deliverReplies`.

## Solution Assessment

No. The shared dispatcher seam is the right layer for the bypass, but this implementation is too broad because Telegram already runs `message_sending` at delivery time; the best fix needs explicit exactly-once ownership.
