## Summary

Keep open: the linked `message_sending` bypass is real, but this PR is not merge-ready because adding the hook to every configured dispatcher chain can double-run Telegram delivery hooks that already call `message_sending`.

## What This Changes

The PR changes shared auto-reply dispatcher hook composition so channel/custom `beforeDeliver` runs before `reply_payload_sending` and `message_sending`, with regression coverage in dispatcher tests.

## Best Possible Solution

Land one canonical fix that restores the configured dispatcher hook chain while guaranteeing `message_sending` runs exactly once per visible reply across shared dispatchers and Telegram delivery paths.

## Reproduction Assessment

Yes. Current main deterministically omits `message_sending` when `dispatcherOptions.beforeDeliver` is present, and the PR's duplicate Telegram path is traceable from the new dispatcher hook into Telegram `deliverReplies`.

## Solution Assessment

No. The shared dispatcher layer is the right place to fix the bypass, but this patch needs exactly-once hook ownership so Telegram paths do not run `message_sending` both before and inside delivery.
