## Summary

Keep this PR open, but do not merge it as-is: the dispatcher bypass is real, yet the patch unconditionally adds `message_sending` to configured dispatcher hooks while Telegram already runs `message_sending` in `deliverReplies`, creating a concrete duplicate-hook path for safety, rewrite, cancel, and audit plugins.

## What This Changes

The PR composes custom dispatcher `beforeDeliver` hooks with `reply_payload_sending` and `message_sending` in both buffered and plain auto-reply dispatchers, with regression tests for custom hook paths.

## Best Possible Solution

Land one canonical fix for https://github.com/openclaw/openclaw/issues/92374 that runs `message_sending` exactly once per visible outbound reply across shared dispatch, Telegram delivery-owned sends, and Telegram preview-finalized streaming.

## Reproduction Assessment

Yes. Current-main source shows configured `beforeDeliver` excludes `message_sending`, and Telegram supplies configured `beforeDeliver`; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not as-is. The shared dispatcher seam is the right layer for the generic bypass, but this implementation needs exactly-once ownership so Telegram paths that already run `deliverReplies` do not invoke `message_sending` twice.
