## Summary

Keep open: the dispatcher bypass is real, but this PR is not merge-ready because it adds `message_sending` to configured dispatcher hooks without resolving Telegram's existing delivery-time hook owner, so Telegram replies can run the hook twice.

## What This Changes

This PR composes channel/custom dispatcher `beforeDeliver` hooks with `reply_payload_sending` and `message_sending` in the shared auto-reply dispatchers and adds regression tests.

## Best Possible Solution

Land one canonical fix for https://github.com/openclaw/openclaw/issues/92374 that runs `message_sending` exactly once per visible outbound reply across shared dispatcher, Telegram delivery-owned sends, and Telegram preview-finalized streaming.

## Reproduction Assessment

Yes. Current-main source deterministically selects the configured `beforeDeliver` branch that excludes `message_sending`, and Telegram supplies that hook; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not as-is. The shared dispatcher seam is the right layer for the bypass, but this implementation needs exactly-once ownership so Telegram paths that already run `deliverReplies` do not invoke `message_sending` twice.
