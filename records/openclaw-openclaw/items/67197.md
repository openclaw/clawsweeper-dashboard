## Summary

Current main now solves the central non-verbose fallback visibility problem: fallback and recovery notices are built as normal reply-status payloads outside the verbose-only path, covered by e2e tests, documented, and shipped in v2026.5.18. The requested default-false config key did not land, but the observable user problem is implemented well enough to close this narrower issue.

## What This Changes

The issue requests an opt-in setting that shows model fallback notices in normal non-verbose replies.

## Best Possible Solution

Keep the shipped default visible fallback/recovery notices, and handle any future opt-out, approval, hook, or context-reinjection policy in the broader fallback UX threads.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source, docs, tests, merged PR provenance, and the shipped v2026.5.18 tag give high-confidence proof that non-verbose fallback and recovery notices are now visible.

## Solution Assessment

Yes for this issue's central user problem. The merged implementation reuses fallback transition state and normal reply delivery to show once-per-state notices; the proposed default-false config knob is a separate policy choice rather than a blocker for closing this narrower request.
