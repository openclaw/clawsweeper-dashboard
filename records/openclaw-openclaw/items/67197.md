## Summary

Keep open. Current main still tracks fallback transitions but only prepends the user-facing fallback and fallback-cleared reply notices when verbose mode is enabled; the requested independent opt-in config is absent from the config and docs surface, and the issue is paired with open implementation PR #69399 in the provided GitHub context.

## What This Changes

Requests a default-false opt-in config that shows model fallback and fallback-cleared notices even when verbose mode is off.

## Best Possible Solution

Review #69399 and either land it or request a revised implementation that adds a default-false agent-defaults option, updates config types/schema/help/docs, and lets `runReplyAgent` prepend fallback and fallback-cleared notices when verbose mode is enabled or the opt-in is true while preserving once-per-active-fallback behavior.

## Reproduction Assessment

Yes. The current e2e tests already exercise a mocked fallback transition with verbose off and assert no fallback notice text, which reproduces the reported gap without needing live provider failures.

## Solution Assessment

Yes in overall shape: reusing the existing transition tracking and notice formatter behind a default-false opt-in is the narrowest maintainable fix. The exact config key remains a maintainer product/API decision.
