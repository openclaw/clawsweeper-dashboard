## Summary

Keep open. This is a draft, maintainer-labeled feature PR with explicit product/API questions, and source review found a blocking correctness issue in the new channel policy scanner even though the submitted real-behavior proof is sufficient for the claimed CLI loop.

## What This Changes

The branch adds a bundled policy plugin, `openclaw policy` generate/check commands, doctor `--lint` structured health checks and repairs, channel conformance checks, oc-path document helpers, docs, labels, and tests.

## Best Possible Solution

Land this only after maintainers approve the policy/doctor-health seam and the implementation filters actual channel entries from channel metadata, with focused coverage for `channels.defaults`, `channels.modelByChannel`, and invalid severity values.

## Reproduction Assessment

No bug-report reproduction is applicable, but the main correctness issue is source-reproducible: `scanPolicyChannels()` walks every key under `channels`, while current config defines non-channel keys there. The submitted terminal-style proof is sufficient for the happy-path denied-Telegram lint/fix/lint loop.

## Solution Assessment

No. The policy direction may be viable, but the implementation should use canonical channel ids or channel metadata instead of treating every `channels.*` key as a provider, and `policy check` should validate severity options consistently with `doctor --lint`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter channel metadata out of policy entries:** `extensions/policy/src/policy-generate.ts:64-65`
  - body: `scanPolicyChannels()` currently walks every top-level key under `cfg.channels` and records it as a channel provider. Current OpenClaw config also stores metadata there, including `channels.defaults` and `channels.modelByChannel`, so workspaces that use those normal config keys will get fake policy entries and drift hashes for non-channel data. Filter to real channel ids before generating entries or computing the channel policy hash.
  - confidence: 0.9
- **[P3] Reject invalid policy severity values:** `extensions/policy/src/cli.ts:133`
  - body: `policy check --severity-min warn` or any other typo currently falls back to `info` because the parsed value is coalesced to the default. That makes CI use a different threshold than requested, while `doctor --lint` correctly errors on invalid severity. Return an error for invalid non-empty values before running checks.
  - confidence: 0.82
