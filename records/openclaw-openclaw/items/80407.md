## Summary

Keep open: this is a draft feature PR with the protected maintainer label, and current main does not already contain the proposed policy/doctor-health system. The happy-path proof is useful, but the patch still has source-reproducible correctness issues in channel scanning and severity parsing.

## What This Changes

The branch adds a bundled policy plugin with `policy check`, `policy.jsonc` channel conformance evidence, doctor lint health checks, opt-in channel disable repair, oc-path document helpers, docs, labels, and tests.

## Best Possible Solution

Approve the policy/doctor-health contract first, then land the channel policy slice only after it filters real channel entries and validates CLI severity input consistently.

## Reproduction Assessment

Yes for the review findings: source inspection against current main shows normal channel metadata under `channels`, and the PR scanner walks every `channels.*` key as a provider. No bug-report reproduction is applicable because this is a new feature PR.

## Solution Assessment

No. The direction may be viable if maintainers approve it, but the implementation should use canonical channel ids or channel metadata instead of raw `channels` keys, and `policy check` should reject invalid severity values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter channel metadata before building policy evidence:** `extensions/policy/src/policy-state.ts:57-68`
  - body: `scanPolicyChannels()` iterates every key in `cfg.channels` and treats the key as a provider. Current config also stores metadata such as `channels.defaults` and `channels.modelByChannel`, so normal workspaces will get fake channel evidence, drift hashes, and possible findings for non-channel data. Filter to real configured channel entries or resolved channel registry metadata before hashing/checking.
  - confidence: 0.91
- **[P3] Reject invalid policy severity values:** `extensions/policy/src/cli.ts:98`
  - body: `policy check --severity-min warn` or any typo falls back to `info` because the parse result is coalesced to the default. That makes automation enforce a different threshold than requested; mirror `doctor --lint` and return an error for invalid non-empty severity values before running checks.
  - confidence: 0.84
