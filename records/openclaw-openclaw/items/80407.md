## Summary

Keep this PR open for maintainer review: it is a draft, maintainer-labeled feature proposal with real product architecture questions, and the current patch still has blocking correctness/security findings.

## What This Changes

The PR adds a bundled policy plugin, `openclaw policy check`, doctor lint/repair health plumbing, channel conformance checks, docs, tests, and an oc-path document API seam.

## Best Possible Solution

Have maintainers decide the policy/doctor-health architecture, then make policy fail closed on malformed artifacts and align targeted repair behavior with the documented safety model before rerunning focused CLI/docs proof.

## Reproduction Assessment

Yes for the review findings: source inspection at PR head shows malformed policy diagnostics are discarded and `doctor --fix --only` is accepted but not honored. The broader PR is a new feature proposal, so there is no current-main bug reproduction path for the product request itself.

## Solution Assessment

No as-is: the proposed direction may be acceptable, but the implementation should fail closed on malformed policy files and either wire repair selectors through or reject them outside lint mode. The larger policy/doctor layering still needs maintainer judgment.

## Review Findings

Overall correctness: patch is incorrect

  - body: When `policy.jsonc` has JSONC parse errors, this branch returns an empty finding set, so `policy check` can emit `ok: true` and doctor lint will not block even though the policy artifact could not be evaluated. Add an explicit parse/diagnostic finding and cover invalid policy input so conformance fails closed.
  - confidence: 0.93
- **[P2] Honor repair selectors or reject them:** `src/cli/program/register.maintenance.ts:31-32`

Either thread selected check IDs into structured repairs or fail fast when these selectors are used without `--lint`, so a targeted policy repair command cannot silently run unrelated repairs.
  - confidence: 0.86
