## Summary

Keep open: the PR addresses a real CLI JSONL parser bug, but the patch only updates the full-buffer parser while current main's runtime path can prefer the streaming parser output first. It also still needs after-fix real behavior proof before merge.

## What This Changes

The PR adds terminal-field extraction to `parseCliJsonl()`, parser tests for `task_complete`/`final_answer` precedence over commentary, and a changelog entry.

## Best Possible Solution

Share the terminal-field priority logic between full-buffer and streaming JSONL parsing, add a regression through the streaming parser or runner path, then land after real Codex/OpenAI CLI JSONL proof is attached.

## Reproduction Assessment

Yes, source-level. The linked report gives a concrete JSONL shape, and current main shows the runner can consume `createCliJsonlStreamingParser().getOutput()` before the full-buffer parser, so the PR's fix can be bypassed.

## Solution Assessment

No, not as-is. The parser direction is right, but the maintainable fix should make terminal-field selection common to both JSONL parsing paths or otherwise ensure the runner uses the terminal-aware result after finish.
