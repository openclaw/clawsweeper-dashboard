## Summary

This PR remains useful and should stay open: current `main` still has the phase-blind CLI JSONL parser behavior reported in the linked issue, while the PR is a focused implementation candidate with sufficient parser/Codex CLI proof. The only remaining blocker is maintainer acceptance of the compatibility change for commentary-only generic JSONL output.

## What This Changes

The PR updates the CLI JSONL full-buffer and streaming parsers to prefer `task_complete.last_agent_message` and `phase:"final_answer"` fields over commentary text, with parser regression tests.

## Best Possible Solution

Land this parser fix after maintainers accept terminal-field and final-answer phase precedence as the JSONL reply-selection contract, while keeping the broader inter-tool channel-leak policy work on https://github.com/openclaw/openclaw/issues/25592 separate.

## Reproduction Assessment

Yes. The linked report provides a sanitized JSONL shape, and current main source shows the parser ignores `task_complete.last_agent_message` and `phase:"final_answer"` while collecting generic item text; I did not run tests because this review was read-only.

## Solution Assessment

Yes for the narrow parser bug. The shared CLI JSONL parser is the right boundary because both streaming and completed CLI runner paths consume it before final text is transformed and delivered; the remaining question is maintainer acceptance of the commentary-only compatibility behavior.
