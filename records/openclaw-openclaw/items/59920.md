## Label Justifications

- `P2`: This is a normal-priority bug fix for a bounded CLI fallback parser path that can affect user-visible replies but does not make the core runtime unusable.
- `merge-risk: 🚨 compatibility`: The PR changes behavior for existing JSONL backends that emit explicit commentary-phase `item.text`, replacing old fallback delivery with suppressed/empty output.
- `merge-risk: 🚨 message-delivery`: The parser change can intentionally suppress commentary-only JSONL text and changes which record is delivered as the final channel-visible reply.

## Summary

Keep this PR open: current main still lacks the terminal-field precedence for non-Claude CLI JSONL output, while the PR adds the behavior in the shared parser and covers both full-buffer and streaming paths. I found no blocking correctness or security finding, but the merge does intentionally change how explicit commentary-phase JSONL is delivered, so maintainers should accept that compatibility/message-delivery tradeoff before landing.

## What This Changes

The PR updates `src/agents/cli-output.ts` so CLI JSONL parsing prefers `task_complete.last_agent_message`, then `phase:"final_answer"`, and skips explicit commentary-phase fallback text, with regression tests for buffered and streaming parser paths.

## Best Possible Solution

Land the parser-layer fix if maintainers accept the explicit contract that commentary-phase JSONL is not user-visible final output and terminal/final-answer fields are the source of truth.

## Reproduction Assessment

Yes, source-reproducible: current main's parser ignores `task_complete.last_agent_message` and `phase:"final_answer"` for generic non-Claude JSONL, while the issue fixture contains those terminal records. I did not run a local test because this review is read-only, but the failing path is clear from `src/agents/cli-output.ts`.

## Solution Assessment

Yes. The PR fixes the shared parser boundary used by both streaming stdout and buffered stdout callers, and it matches the existing phase-aware invariant elsewhere in OpenClaw and the sibling Codex SDK contracts.
