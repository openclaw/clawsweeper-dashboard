## Label Justifications

- `P2`: This is a focused user-visible parser bug fix for a fallback reply path with limited code surface but real channel-delivery impact.
- `merge-risk: 🚨 compatibility`: Changing explicit commentary-phase JSONL from fallback reply text to empty output can alter behavior for existing custom JSONL CLI backends.
- `merge-risk: 🚨 message-delivery`: The changed precedence can decide whether channel-visible fallback delivery sends final answer text, commentary text, or no text.

## Summary

This PR is still necessary: current main remains phase-blind for the generic CLI JSONL fallback, while the branch adds narrow terminal-field precedence for both full-buffer and streaming parser paths. I found no blocking correctness or security finding, but the commentary-suppression change is compatibility and message-delivery sensitive enough that maintainers should explicitly accept it before landing.

## What This Changes

The PR updates `src/agents/cli-output.ts` and parser tests so CLI JSONL parsing prefers `task_complete.last_agent_message`, then `phase: "final_answer"`, before generic non-commentary item text.

## Best Possible Solution

Land the narrow parser fix once maintainers accept the explicit-commentary compatibility tradeoff and the required checks are green or documented unrelated.

## Reproduction Assessment

Yes, at source level: current main's generic JSONL fallback collects message-like `item.text` without checking phase, and the PR adds fixtures where commentary loses to `final_answer` and `task_complete`. I did not run tests because this review was read-only.

## Solution Assessment

Yes, this is the best narrow layer for the reported bug: the shared CLI JSONL parser is where both full-buffer and streaming fallback output lose terminal semantics. Runner-level `--output-last-message` wiring was considered separately in https://github.com/openclaw/openclaw/pull/66081, but it does not cover the app-server-style `task_complete` and `final_answer` JSONL shapes handled here.
