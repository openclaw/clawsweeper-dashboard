## Summary

Keep open. The PR targets a real CLI JSONL parser bug that current main still has, but as-is it only fixes the full-buffer parser while current main can prefer the streaming JSONL parser output first, so the production path can still bypass the new terminal-field priority.

## What This Changes

The PR adds terminal-field extraction to `parseCliJsonl()`, parser tests for Codex/OpenAI JSONL `task_complete` and `final_answer` precedence over commentary, and a changelog entry.

## Best Possible Solution

Share the terminal-field priority logic between the full-buffer and streaming JSONL parsers, add a regression through the streaming parser or runner path, and land after real Codex/OpenAI CLI JSONL proof is attached.

## Reproduction Assessment

Yes, source-level. Current main still has the phase-blind `item.text` JSONL fallback, and the linked report plus synthetic fixtures describe a concrete stream where commentary can beat terminal fields; I did not run tests in this read-only review.

## Solution Assessment

No, not as-is. The parser direction is right, but current main can consume `createCliJsonlStreamingParser().getOutput()` before the full-buffer parser, so the fix should cover that path too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Apply terminal priority to the streaming parser:** `src/agents/cli-output.ts:554-555`
  - body: Current main now prefers `createCliJsonlStreamingParser().getOutput()` over `parseCliOutput()` for JSONL backends after the process exits. This patch only teaches `parseCliJsonl()` about `task_complete.last_agent_message` and `final_answer`, so a stream with commentary `item.text` plus a later terminal payload can still return the streaming parser's commentary and bypass the new priority rules. Please share this extraction with the streaming parser or make the runner use the full parsed result after finish, with a regression for that path.
