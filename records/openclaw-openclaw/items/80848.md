## Summary

Keep this PR open: the code change looks like a focused fix for a real Telegram context-isolation bug, but it still needs after-fix real behavior proof and a security-audit disposition before merge.

## What This Changes

The PR filters Telegram cached conversation context so messages before the latest built-in `/new` or non-soft `/reset` in the same chat/topic are not selected, adds a regression test, and updates the changelog.

## Best Possible Solution

The durable end state is a Telegram context boundary that prevents pre-reset cached topic messages from becoming active prompt context while preserving intentional soft-reset behavior.

## Reproduction Assessment

Yes, at source level: current main selects recent and reply-target cache windows without a reset-boundary check, then sends them as Telegram Conversation context. I did not run live Telegram or local tests in this read-only review.

## Solution Assessment

Mostly yes: filtering in the message-cache selector is narrow and covers both recent and reply-target selected context with a regression test. The safer long-term refinement would be to pass configured reset triggers into this boundary logic if alias parity is required.
