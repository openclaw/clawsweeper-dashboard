## Summary

Keep open: current main and the latest release still have no persisted main-session restart-recovery retry budget, and the remaining work is now represented by open candidate PRs rather than merged code.

## What This Changes

The issue requests a durable cross-boot retry budget and quarantine behavior for wedged main-session restart recovery so the gateway cannot re-enter the same recovery loop after every reboot.

## Best Possible Solution

Land one canonical fix that persists main-session recovery attempts and quarantine state at the session-store boundary, then close this issue and supersede the duplicate candidate PRs.

## Reproduction Assessment

Yes, at source level: current main can recover any running main-session row with abortedLastRun=true without consulting a persisted attempt ceiling. I did not run the full Telegram/systemd production loop.

## Solution Assessment

Yes for the issue direction: a durable main-session retry budget plus quarantine marker is the narrowest maintainable survivability fix. The implementation decision now is which open candidate PR maintainers want as the canonical landing path.
