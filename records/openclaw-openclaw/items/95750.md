## Label Justifications

- `P1`: The issue reports a restart-recovery loop that can exhaust supervisor restarts and break real channel/main-session workflows until operator intervention.
- `impact:crash-loop`: The reported failure mode repeatedly restarts the gateway until `systemd` refuses further restarts.
- `impact:session-state`: The bug is centered on persisted session recovery state in `sessions.json` and repeated recovery eligibility across boots.
- `impact:message-loss`: Wedged channel main-session turns can be retried or killed repeatedly without delivering a stable user-facing outcome.

## Summary

Keep open: current `main` still retries any running main-session row with `abortedLastRun: true` on each startup without a persisted attempt ceiling, while the subagent side already has a durable wedged-state guard that can be used as the model for the remaining main-session fix.

## What This Changes

The issue asks OpenClaw to add a persistent cross-boot retry budget and quarantine/wedge behavior for restart-recovery of wedged main sessions, with the same survivability concern checked against subagent recovery.

## Best Possible Solution

Add a durable main-session recovery budget/wedged-state guard, reuse the existing retry constant or a narrow internal constant, clear recovery eligibility after the ceiling, and cover repeated-boot failure plus successful new-turn cleanup with focused tests; keep trigger-layer work in the linked issues separate.

## Reproduction Assessment

Yes, at source level: seed a running main-session row with `abortedLastRun: true` and a resumable transcript, make the recovery `agent` call fail, then invoke startup recovery from a fresh scheduler context; current main has no persisted attempt ceiling. I did not run the full Telegram/systemd production loop.

## Solution Assessment

Yes for the core direction: a durable main-session retry budget/wedged marker is the narrowest survivability fix and matches the existing subagent recovery pattern. Adding a new config knob is not necessary for the first repair unless maintainers explicitly want operator override.
