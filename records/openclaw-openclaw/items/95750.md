## Label Justifications

- `P1`: The report describes a real latest-release gateway restart death loop that can exhaust a systemd user service and block channel work until manual repair.
- `impact:crash-loop`: The core failure mode is repeated gateway restart recovery until supervisor restart limits are exhausted.
- `impact:session-state`: The affected state is persisted session metadata, including `abortedLastRun`, `restartRecoveryRuns`, and any future recovery tombstone.
- `impact:message-loss`: The loop starts from wedged channel main-session turns and can prevent further Telegram or group-topic delivery until the session store is repaired.

## Summary

Keep open: current main and v2026.6.9 still let restart-aborted main sessions re-enter recovery on every boot with only a process-local retry budget, while subagent orphan recovery already has a durable tombstone pattern that main-session recovery lacks.

## What This Changes

The issue requests a durable cross-boot retry budget or quarantine for restart-aborted main-session recovery so wedged sessions cannot repeatedly restart the gateway.

## Best Possible Solution

Add a durable main-session restart-recovery budget and quarantine/tombstone path, align it with the existing bounded subagent recovery behavior, and cover mark, resume, overrun, and next-turn cleanup paths without adding a new config knob unless maintainers explicitly want one.

## Reproduction Assessment

Yes at source level: a current-main running session with `abortedLastRun: true` remains eligible on every fresh boot, while `MAX_RECOVERY_RETRIES` is only process-local. I did not run the live Telegram/systemd reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation shape: the proposed durable counter is plausible and mirrors the missing main-session guard, but maintainers should choose the stored state and quarantine-clearing semantics before a fix lands.
