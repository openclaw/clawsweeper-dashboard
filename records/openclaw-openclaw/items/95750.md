## Summary

Keep open: current main still has the main-session recovery gap, and three open PRs already reference this issue as their closing fix candidates, so the issue should stay open until one canonical PR lands or the PRs are closed.

## What This Changes

The issue asks for a durable cross-boot retry budget and quarantine state for restart-aborted main sessions so wedged rows cannot repeatedly restart-loop the gateway.

## Best Possible Solution

Land one canonical fix PR that persists the retry budget and quarantine state at the SessionEntry recovery owner boundary, clears it on successful recovery or a real new user turn, and then close superseded duplicate PRs.

## Reproduction Assessment

Yes, at source level: seed a running main-session row with `abortedLastRun: true` and a resumable transcript, make the recovery `agent` call fail, and a fresh process gets a new in-process retry budget because no durable attempt ceiling exists. I did not run the full Telegram/systemd loop.

## Solution Assessment

Yes for the core direction: a durable session-store retry budget plus quarantine marker is the narrowest survivability fix and matches the existing subagent recovery pattern. A new config knob should stay out of the initial repair unless maintainers explicitly want operator override.
