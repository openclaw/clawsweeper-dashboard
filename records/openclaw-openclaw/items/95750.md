## Label Justifications

- `P1`: The issue reports a real restart-recovery loop that can exhaust supervisor restarts and break channel/main-session workflows until operator intervention.
- `impact:session-state`: The bug is centered on persisted `sessions.json` recovery state and repeated eligibility across gateway boots.
- `impact:message-loss`: Wedged channel main-session turns can keep re-entering recovery without delivering a stable user-facing result.
- `impact:crash-loop`: The reported failure repeatedly restarts the gateway until the supervisor refuses further starts.

## Summary

Keep open: current main and the latest release still recover any running main-session row with `abortedLastRun: true` without a persisted cross-boot attempt ceiling, while the open fix candidates are not merged implementation proof.

## What This Changes

Add a durable cross-boot retry budget and quarantine path for main-session restart recovery so wedged sessions cannot restart-loop the gateway.

## Best Possible Solution

Land one narrow recovery fix that adds optional persisted main-session attempt/quarantine state after the existing routing, owner, and dedupe guards, keeps the subagent guard intact, and validates repeated-boot and upgrade behavior.

## Reproduction Assessment

Yes, at source level: seed a running main-session row with `abortedLastRun: true`, make resume fail, then start recovery from a fresh process context; current main has no persisted attempt ceiling. I did not run the full Telegram/systemd production loop.

## Solution Assessment

Yes for direction: a durable main-session recovery budget/quarantine marker is the narrow survivability fix and matches the existing subagent recovery pattern. A new operator config knob is not necessary for the first repair unless maintainers explicitly request it.
