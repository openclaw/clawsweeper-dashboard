## Summary

Keep open: current main still lacks skill-integrity verification, and this PR remains non-mergeable because the proposed security check has fail-open gaps and needs a scope/rebase decision.

## What This Changes

The PR records SHA-256 metadata for ClawHub skill installs, adds verification helpers and security-audit findings, and adds a skill-integrity proposal document.

## Best Possible Solution

Land a rebased, security-reviewed implementation that records stable full-content hashes for ClawHub-installed skills, reports mismatch, missing-hash, and verifier-error states, hashes runtime-relevant files except explicit OpenClaw metadata, and aligns tests/docs/changelog with the final audit-versus-loader scope.

## Solution Assessment

No. The install/audit direction is plausible, but this is not the best mergeable form until it fails closed on verifier errors, deliberately covers hidden runtime files, rebases onto current main, and settles loader-versus-audit policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Report verifier errors in audit findings:** `src/security/audit-extra.async.ts:1350-1358`
  - body: `verifySkillIntegrity` returns `status: "error"` for read/hash failures, but this collector only handles missing hashes and mismatches. A tampered or unreadable skill can therefore make integrity verification fail open while `security audit` stays clean; emit a finding for error results.
  - confidence: 0.9
- **[P2] Hash runtime-relevant hidden skill files:** `src/agents/skills-clawhub.ts:320`
  - body: The PR records hashes with `computeSkillFingerprint`, which skips every dot-prefixed file or directory. If a skill uses a hidden runtime file, post-install tampering of that file will not change the recorded/verifier digest; exclude only explicit OpenClaw metadata such as `.clawhub`.
  - confidence: 0.86
- **[P3] Align the lockfile version contract:** `docs/proposals/skill-integrity-verification.md:36-44`
  - body: The proposal documents a version-2 lockfile, but the implementation still types and reads the lockfile as `version: 1`. Either keep the documented format as a version-1 extension or implement read/write support for the version-2 schema so future readers do not split on incompatible assumptions.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The PR adds a security check, but the proposed implementation has concrete fail-open integrity gaps.

Concerns:

- **[medium] Integrity errors are silently ignored:** `src/security/audit-extra.async.ts:1350`
  - body: Hash/read failures are represented as `status: "error"`, but the audit collector never turns them into findings, so compromised or unreadable skill contents can avoid the intended warning.
  - confidence: 0.9
- **[medium] Hidden skill contents are outside the digest:** `src/agents/skills-clawhub.ts:320`
  - body: The install and verification hashes skip dot-prefixed files and directories, so hidden runtime files can be changed after install without triggering a mismatch.
  - confidence: 0.86
