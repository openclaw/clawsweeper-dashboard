## Summary

Keep open. This is a plausible security feature, but current main still lacks native skill integrity verification, this PR is conflicting and has fail-open review findings, and the broader skill-signature trust model is already being tracked at https://github.com/openclaw/openclaw/issues/12507.

## What This Changes

The branch records SHA-256 fingerprints for ClawHub skill installs, adds integrity verifier helpers and security-audit findings, wires the non-deep audit collector, and adds a proposal document.

## Best Possible Solution

Land a rebased, security-reviewed implementation aligned with https://github.com/openclaw/openclaw/issues/12507 that records trusted skill integrity metadata, reports mismatch, missing-hash, and verifier-error states, hashes runtime-relevant contents except explicit OpenClaw metadata, and defines loader policy.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows verifier error results are ignored and dot-prefixed contents are excluded from the digest used by verification. I did not run a mutating install/audit repro because this review is read-only.

## Solution Assessment

No. The install/audit direction is plausible, but this is not the best mergeable form until verifier errors become findings, runtime-relevant hidden contents are covered, the branch is rebased, real behavior proof is added, and maintainers settle the trust-model scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Report verifier errors in audit findings:** `src/security/audit-extra.async.ts:1350-1358`
  - body: `verifySkillIntegrity` can return `status: "error"` for read or hash failures, but this collector only emits findings for missing hashes and mismatches. A compromised or unreadable skill can therefore make verification fail open while security audit stays clean; emit an audit finding for error results.
  - confidence: 0.9
- **[P2] Include hidden files in skill integrity fingerprints:** `src/agents/skills-clawhub.ts:526`
  - body: The new verifier uses `computeSkillFingerprint` as the tamper-detection digest, but that traversal skips every dot-prefixed file and directory. If a skill uses hidden runtime files, post-install tampering there still verifies as ok; exclude only explicit OpenClaw metadata such as `.clawhub`.
  - confidence: 0.86
- **[P3] Keep lockfile version semantics consistent:** `src/agents/skills-clawhub.ts:36-38`
  - body: The PR stores `sha256` and `scannedAt` fields while the lockfile type and reader still declare only version 1, even though the proposal documents this as a version-2 schema. Either document this as a version-1 additive extension or update the reader/writer migration semantics so future readers do not disagree about the persisted format.
  - confidence: 0.72
