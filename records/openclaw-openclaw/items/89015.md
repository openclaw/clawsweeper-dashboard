## Summary

Keep open. This PR targets a real, source-reproducible QQ Bot token-fetch failure that current main and the latest release still have, but the SSRF relaxation should be tightened before merge because the new policy is not actually hostname-scoped across guarded-fetch redirects.

## What This Changes

The PR adds an RFC2544 benchmark-range SSRF policy to the QQ Bot token fetch call and updates the token manager tests to expect that policy.

## Best Possible Solution

Keep the per-call QQ Bot token fix, but make the RFC2544 exception truly host-bound, for example with a bots.qq.com hostname allowlist or by disabling redirects for this token POST, and cover that constrained policy in the token tests before merge.

## Reproduction Assessment

Yes from source inspection: current main and v2026.5.28 call fetchWithSsrFGuard without a policy, while the SDK blocks 198.18.0.0/15 unless allowRfc2544BenchmarkRange is set. I did not run a live QQ fake-IP setup in this read-only review.

## Solution Assessment

No as written. The per-call token manager is the right layer, but the policy should match the host-scoped security claim before merge.
