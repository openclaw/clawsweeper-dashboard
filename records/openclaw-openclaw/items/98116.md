## What This Changes

iOS Chat should avoid transiently rendering duplicate final assistant replies when live `chat.final` and canonical `session.message` events for the same turn arrive in either order.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/98117 with coverage for both event orders and repeated identical assistant replies; close this issue only after a fix merges.

## Reproduction Assessment

Yes. Source inspection on current main shows a live `chat.final` assistant row and a canonical `session.message` row can both be appended while timestamp-sensitive dedupe keeps them distinct until history refresh.

## Solution Assessment

Yes. The narrow maintainable fix belongs in shared `OpenClawChatViewModel` reconciliation by treating live final rows as provisional for the current turn, not by changing Gateway transport or globally deduping by text.
