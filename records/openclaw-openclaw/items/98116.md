## What This Changes

Reports a shared iOS/macOS Chat UI reconciliation bug where live `chat.final` and canonical `session.message` events can briefly render the same assistant final response twice before history refresh.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/98117, preserving regression coverage for both event orders and repeated identical assistant replies before closing this issue.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main appends a live final assistant row and a later canonical `session.message` row, while the identity/dedupe keys are timestamp-sensitive. I did not run Swift tests or an iOS device repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable solution is to treat live final assistant rows as provisional for the current turn and adopt/skip them when the canonical transcript row arrives, not to globally dedupe by text. The linked pull request implements that direction and adds focused regression cases, but it still needs normal PR review and CI completion.
