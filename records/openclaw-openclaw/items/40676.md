## Summary

Keep this PR open. The underlying Discord voice bug still appears valid on current main, but this branch is not merge-ready because it targets the old `src/discord/**` layout and its stale-connection/session-key changes introduce concrete regressions.

## What This Changes

The PR adds Discord voice auto-join on gateway RESUMED events, destroys a stale `@discordjs/voice` connection before joining, then adds an eager autoJoin call plus a voice-specific session key and system prompt in legacy `src/discord/**` files.

## Best Possible Solution

Land a rebased Discord plugin fix that adds a RESUMED listener beside the READY listener, registers it through `extensions/discord/src/monitor/provider.ts`, and uses an account-scoped `@discordjs/voice` group for stale connection lookup/join cleanup. Keep voice turns on the existing routed session key unless a separate, tested design changes Discord session semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The linked bug gives concrete `voice.autoJoin` plus health-monitor restart steps, and current-main code confirms RESUMED dispatch exists while voice autoJoin is READY-only and the join path has no stale registry cleanup; I did not run a live Discord gateway test in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The intended RESUMED autoJoin and stale cleanup direction is right, but the patch needs to move to `extensions/discord/**`, scope `@discordjs/voice` connection groups per account/manager, and drop the unrelated session-key/system-prompt changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Scope stale cleanup to this voice manager:** `src/discord/voice/manager.ts:412-416`
  - body: `getVoiceConnection(guildId)` reads the default process-wide voice registry. OpenClaw supports multiple Discord accounts, so destroying that connection can disconnect another account that is currently joined to the same guild. Use an account/manager-scoped voice group for both lookup and `joinVoiceChannel` before destroying anything.
  - confidence: 0.9
- **[P2] Keep voice session keys routable for approvals:** `src/discord/voice/manager.ts:660-668`
  - body: Changing the route session key from `:channel:` to `:voice:` means Discord approval routing can no longer extract the originating channel/group from the session key. Approval prompts that target the current channel can then fail to deliver for voice-triggered commands.
  - confidence: 0.82
- **[P2] Preserve shared voice and text session context:** `src/discord/voice/manager.ts:660-668`
  - body: Rewriting spoken turns into a separate `:voice:` session splits them away from the text channel session created for the same voice channel. Users alternating speech and text in the channel would lose shared history/tool context, which is unrelated to the RESUMED/stale-connection fix.
  - confidence: 0.78
