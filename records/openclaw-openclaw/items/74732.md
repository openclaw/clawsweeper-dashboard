## Summary

Keep open. Current main still lacks oMLX-specific memorySearch documentation, and the issue is paired with an open, unmerged closing PR from the same author, so the right cleanup is to review that PR rather than close the issue early. ## What This Changes

Request to document oMLX as an Apple Silicon MLX/OpenAI-compatible memorySearch embedding provider, with an optional first-class `omlx` provider adapter for friendlier defaults.

## Best Possible Solution

Land or close PR #74761 after maintainer review of the recipe accuracy, keeping oMLX on the existing OpenAI-compatible embedding path. If a native `omlx` adapter is desired later, handle it as a separate provider/plugin product decision rather than blocking the documentation fix.

## Reproduction Assessment

Not applicable as a docs/feature request. The documentation gap is directly verifiable on current main by inspecting the memory-search, memory-config, and OpenAI provider docs, while the generic runtime path is visible in the OpenAI embedding provider and remote embedding client.

## Solution Assessment

Yes for the docs portion: documenting `provider: "openai"` with `memorySearch.remote.baseUrl` is the narrowest maintainable solution because the runtime already supports OpenAI-compatible embedding endpoints. A native oMLX provider would be a separate UX enhancement, not the prerequisite fix.
