## Summary

Keep this open because current main has the generic OpenAI-compatible memory embedding path needed for oMLX, but it still lacks the oMLX-specific documentation requested here. PR #74761 is an open, unmerged closing PR for the docs gap, so this issue should remain open until that PR is reviewed and either merged or closed.

## What This Changes

The issue asks OpenClaw to document, and optionally add first-class support for, oMLX as an Apple Silicon MLX memorySearch embedding provider using its OpenAI-compatible local endpoint.

## Best Possible Solution

Review and land PR #74761 if the oMLX recipe is accurate, keeping oMLX on the existing OpenAI-compatible embedding path. Treat a native `omlx` provider as a separate follow-up only if maintainers want vendor-specific defaults or diagnostics.

## Reproduction Assessment

Not applicable as a docs/feature request. The docs gap is reproducible by source search on current main, and the generic runtime path can be inspected through the OpenAI embedding adapter and remote client code.

## Solution Assessment

Yes for the docs portion: a focused recipe is the narrowest maintainable fix because runtime support already exists through the OpenAI-compatible embedding adapter. A native provider would be a broader UX/product choice, not a prerequisite for solving the reported dead ends.
