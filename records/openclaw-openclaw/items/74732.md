## Summary

Keep open. Current main still lacks oMLX-specific memorySearch documentation, and the linked open PR #74761 is already the focused implementation candidate for the docs portion; the issue should wait for that PR to be reviewed, landed, or closed instead of being closed early.

## What This Changes

Request to document, and optionally natively support, oMLX as an Apple Silicon MLX memorySearch embedding provider through its OpenAI-compatible local endpoint.

## Best Possible Solution

Ship a focused docs recipe that keeps oMLX on the existing `provider: "openai"` embedding path, warns against using the Ollama adapter, includes the local `/v1` base URL and model example, and tells users to reindex after changing embedding models or vector spaces. Treat a native `omlx` adapter as a later UX enhancement if maintainers want it.

## Reproduction Assessment

Not applicable as a docs/feature request. The docs gap is directly verifiable on current main by the absence of oMLX-specific references, while the reported runtime path is supported by the generic OpenAI-compatible embedding code.

## Solution Assessment

Yes for the docs portion. Reviewing and landing PR #74761 is the narrowest maintainable fix; adding a native provider adapter would be broader product work, not a prerequisite for resolving the current documentation dead-end.
