## Summary

Keep open: current main still lacks the requested MCP Apps gateway RPC surface, but this active feature PR is not merge-ready because the branch regresses current loopback owner/hook protections and lacks contributor real behavior proof.

## What This Changes

The PR adds MCP Apps gateway WebSocket methods, HTTP resources/list/read support, a ui:// resource registry, _meta.ui schemas, docs, and tests.

## Best Possible Solution

Rebase the feature onto current main and implement it through token-derived owner context, owner-only filtering, shared hook-aware tool invocation, scoped resource semantics, focused regression tests, and live MCP/gateway client proof.

## Reproduction Assessment

Yes for the review defects: source inspection of PR head versus current main shows the owner-token, owner-only filtering, hook blocking, and abort-propagation paths that the branch bypasses. The requested MCP Apps protocol itself is a new feature, not a bug reproduction case.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the narrowest maintainable solution until it preserves current MCP loopback security semantics and resolves the gateway/resource boundary with maintainers.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive owner status from loopback bearer tokens:** `src/gateway/mcp-http.request.ts:139-140`
  - body: Current main derives senderIsOwner from separate owner and non-owner loopback bearer tokens and ignores spoofed owner headers. This branch validates one token and then trusts x-openclaw-sender-is-owner, so a non-owner loopback caller can self-declare owner and reach owner-only tools.
  - confidence: 0.9

  - body: Current main applies applyOwnerOnlyToolPolicy before caching and schema generation. This branch caches next.tools directly and builds schemas from that unfiltered list, allowing non-owner MCP callers to list or execute owner-only tools such as cron, gateway, or nodes.
  - confidence: 0.93
- **[P1] Run before-tool hooks for HTTP tool calls:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The modified HTTP tools/call handler calls tool.execute directly. That skips current before-tool-call hooks used for blocks, approval flows, parameter rewrites, and abort propagation before loopback tool execution.
  - confidence: 0.91
- **[P1] Run before-tool hooks for WS tool calls:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new mcp.tools.call handler also calls tool.execute directly. Since this creates another gateway tool-execution surface, it needs the same before-tool-call hook behavior as the existing loopback and tools.invoke paths before merge.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch adds MCP tool-execution surfaces while regressing current loopback owner-token filtering and hook-based execution protections.
