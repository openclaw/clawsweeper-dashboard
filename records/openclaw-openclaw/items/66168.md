## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Derive owner status from loopback bearer tokens:** `src/gateway/mcp-http.request.ts:139-140`
  - body: The PR validates a single bearer token and then trusts `x-openclaw-sender-is-owner` in `resolveMcpRequestContext`. Current main derives owner status from separate owner/non-owner bearer tokens, so this lets a non-owner loopback caller self-declare owner and reach owner-only tools.
  - confidence: 0.92
- **[P1] Filter owner-only tools before caching:** `src/gateway/mcp-http.runtime.ts:63-64`
  - body: This cache stores `next.tools` and builds schemas from that unfiltered list. Current main applies `applyOwnerOnlyToolPolicy` before caching/schema generation, so non-owner MCP callers can list or execute owner-only tools when owner context is false or undefined.
  - confidence: 0.93
- **[P1] Run before-tool hooks for HTTP calls:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The modified HTTP `tools/call` path directly executes the tool. Current main runs `runBeforeToolCallHook` first and passes adjusted params plus the request abort signal, so this skips hook blocks, approvals, param rewrites, loop detection, and cancellation.
  - confidence: 0.91
- **[P1] Run before-tool hooks for WS calls:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new `mcp.tools.call` WebSocket handler also directly executes the tool. Since this is a new gateway tool-execution surface, it needs the same hook-aware execution behavior and abort handling before merge.
  - confidence: 0.89
- **[P1] Scope MCP resources to the caller before reading:** `src/gateway/mcp-http.runtime.ts:77-78`
  - body: Resource sync unions every active cache entry into one process-global registry, and list/read then serve that registry without session filtering. Because resources can be file-backed, one session can enumerate or read another session's synced `ui://` resource.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch adds MCP tool-execution and resource-read surfaces while regressing current owner-token, owner-only filtering, hook, abort, and resource-isolation protections.

Concerns:

- **[high] Owner status is header-derived:** `src/gateway/mcp-http.request.ts:139`
  - body: The PR validates one bearer token but derives `senderIsOwner` from `x-openclaw-sender-is-owner`; current main derives owner status from separate owner/non-owner bearer tokens, so owner status becomes spoofable on the loopback surface.
  - confidence: 0.92
- **[high] Owner-only tools can leak through the cache:** `src/gateway/mcp-http.runtime.ts:63`
  - body: The PR caches and builds schemas from unfiltered tools without `applyOwnerOnlyToolPolicy`, allowing non-owner MCP callers to see or invoke tools current main filters out.
  - confidence: 0.93
- **[high] HTTP tool calls bypass hooks:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The modified HTTP handler directly executes tools, skipping hook-based blocks, approval handling, parameter rewrites, loop detection, and abort propagation that current main applies before execution.
  - confidence: 0.91
- **[high] WS tool calls bypass hooks:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new WebSocket handler executes tools directly, creating another tool-execution surface without the current before-tool-call protections or abort handling.
  - confidence: 0.89
- **[high] Resource registry is process-global:** `src/gateway/mcp-http.runtime.ts:77`
  - body: The PR syncs all active cache entries into one registry and serves resources without caller filtering; file-backed resources make this a cross-session data exposure risk.
  - confidence: 0.82
