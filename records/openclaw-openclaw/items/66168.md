Full review comments:

- **[P1] Route WS tool calls through the MCP policy hook:** `src/gateway/server-methods/mcp.ts:146`
  - body: `mcp.tools.call` executes the resolved tool directly, while current main’s MCP loopback runs `runBeforeToolCallHook` before `tool.execute` so policy/audit hooks can block or rewrite the call. This new WS execution path should use the same hook context before invoking the tool, otherwise a client with write scope can bypass the policy boundary that existing MCP calls receive.
  - confidence: 0.9
- **[P1] Scope MCP resources before listing or reading them:** `src/gateway/mcp-app-resources.ts:169-170`
  - body: The resource APIs return from a process-global registry and the schema says `sessionKey` is not used, but registered resources can be file-backed and are populated from session-scoped tool resolution. With concurrent sessions, a caller can enumerate or read another session’s registered `ui://` resource by URI; resource ownership needs to be part of the registry/read path before this is exposed.
  - confidence: 0.87
