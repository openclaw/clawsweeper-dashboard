## Summary

Keep open: the requested MCP Apps gateway protocol is a real feature and is not implemented on current main, but this PR should not land as-is because it bypasses current MCP loopback ownership, owner-only filtering, tool-hook, abort, and resource-isolation protections, and it still lacks real behavior proof.

## What This Changes

Adds MCP Apps gateway WebSocket methods, HTTP loopback resource support, UI metadata schemas, an in-process ui:// resource registry, docs, and tests.

## Best Possible Solution

Rework or replace the branch on current main so MCP Apps gateway support reuses token-derived ownership, owner-only filtering, hook-aware execution with abort handling, and caller-scoped resources, then coordinate it with the active MCP Apps/resource PRs and add live client proof.

## Reproduction Assessment

Yes for the review defects: PR-head source inspection against current main shows the owner-token, owner-only filtering, hook, abort, and resource-scope paths the branch bypasses. The requested MCP Apps gateway protocol itself is a new feature, so there is no broken current-main behavior to reproduce.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the narrowest maintainable implementation until it preserves current MCP loopback security semantics, scopes resources by caller/session, and proves the runtime path in a real setup.

## Security Review

Status: needs_attention

Summary: The diff adds MCP tool-execution and resource-read surfaces while bypassing current ownership, owner-only filtering, hook, abort, and resource-isolation protections.

Concerns:

- **[high] Owner status is header-derived:** `src/gateway/mcp-http.request.ts:139`
  - body: The PR validates a single bearer token path but then derives senderIsOwner from x-openclaw-sender-is-owner; current main derives owner status from separate owner/non-owner bearer tokens, making owner status spoofable on the loopback surface.
  - confidence: 0.92
- **[high] Owner-only tools can leak through the cache:** `src/gateway/mcp-http.runtime.ts:63`
  - body: The PR caches and builds schemas from unfiltered tools without applyOwnerOnlyToolPolicy, allowing non-owner MCP callers to see or invoke tools current main filters out.
  - confidence: 0.93
- **[high] HTTP tool calls bypass hooks:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The modified HTTP handler directly executes tools, skipping hook-based blocks, approval handling, parameter rewrites, loop detection, and abort propagation that current main applies before execution.
  - confidence: 0.91
- **[high] WS tool calls bypass hooks:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new WebSocket handler executes tools directly, creating another tool-execution surface without the current before-tool-call protections or abort handling.
  - confidence: 0.89
- **[high] Resource registry is process-global:** `src/gateway/mcp-http.runtime.ts:77`
  - body: The PR syncs all active cache entries into one registry and serves resources without caller filtering; file-backed resources make this a cross-session data exposure risk.
  - confidence: 0.82
