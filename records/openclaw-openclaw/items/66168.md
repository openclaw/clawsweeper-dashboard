## Summary

Keep open: current main does not yet implement the requested MCP Apps gateway methods, but the PR is not merge-ready because its stale loopback changes bypass current owner-token filtering and before-tool-call hook protections.

## What This Changes

The PR adds gateway MCP Apps WebSocket methods, HTTP resource support, resource registry/types/schemas, `_meta.ui` metadata, docs, and tests.

## Best Possible Solution

Rebase the feature onto current main and keep the MCP Apps protocol additive while preserving token-derived owner context, owner-only filtering, before-tool-call hooks, abort propagation, and focused regression tests for both HTTP and WS call paths.

## Reproduction Assessment

Yes. The reproduction path is code-level and testable: current `src/gateway/mcp-http.test.ts` asserts owner-only filtering and hook blocking, while the PR head removes those protections from the modified HTTP path and omits them from the new WS path.

## Solution Assessment

No. The feature direction may be valid, but this PR is not the narrowest maintainable solution until it is rebased through the current MCP loopback security model instead of replacing it with direct execution paths.

## Security Review
