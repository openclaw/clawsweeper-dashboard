Full review comments:

- **[P1] Derive owner status from bearer tokens:** `src/gateway/mcp-http.request.ts:139-140`
  - body: This branch validates a single loopback bearer token and then trusts `x-openclaw-sender-is-owner` to decide owner status. Current main derives owner/non-owner status from distinct bearer tokens, so merging this would let any bearer holder mark itself as owner and expose owner-only tools.

  - body: `tools/call` executes the tool directly with caller arguments. Current main runs `runBeforeToolCallHook` first and forwards the request abort signal, so this bypasses plugin approvals/vetoes, parameter rewrites, loop detection context, and cancellation for the HTTP MCP path.
  - confidence: 0.94
- **[P1] Run hooks before WS MCP tool execution:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new WebSocket `mcp.tools.call` path also calls `tool.execute` directly. Gateway-exposed tool execution needs the same before-tool-call hook and abort behavior as the existing HTTP/gateway invocation paths, otherwise a new client surface can bypass current tool policy.
  - confidence: 0.92
- **[P1] Scope resource reads to the requesting session:** `src/gateway/server-methods/mcp.ts:245`
  - body: `mcp.resources.read` resolves any URI from a process-global registry after cache sync, without checking whether that resource belongs to the caller's session/account. Because the registry can contain file-backed resources from other active sessions, callers can enumerate/read resources outside their own context.
  - confidence: 0.88
