Full review comments:

- **[P1] Run MCP tool calls through the policy hook:** `src/gateway/server-methods/mcp.ts:146`
  - body: This new WS path executes the selected tool directly. Existing gateway/MCP tool execution runs the before-tool-call hook so policy, audit, approval, and cancellation context are applied before a plugin tool runs; bypassing that path lets `mcp.tools.call` behave differently from the supported tool execution surfaces.
  - confidence: 0.92
- **[P1] Do not regress loopback tool-call hooks:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The PR also changes the existing HTTP loopback `tools/call` handler to call `tool.execute` directly. Current main runs `runBeforeToolCallHook` and passes hook context/signal into execution, so existing loopback clients would lose the current policy and cancellation behavior after merge.
  - confidence: 0.94
- **[P1] Scope resources before exposing the registry:** `src/gateway/mcp-app-resources.ts:169`
  - body: `listResources()` and `resolveResourceContent()` read from one process-global registry while the schema accepts but ignores `sessionKey`. After one session syncs MCP app resources, another gateway or loopback caller can enumerate and read those resources, including file-backed entries, even though current loopback tool resolution is context-sensitive.
  - confidence: 0.88
- **[P2] Preserve the full loopback cache context:** `src/gateway/mcp-http.runtime.ts:34-40`
  - body: The new cache key drops current channel/thread/message/audio, inbound event kind, reply delivery mode, and agent identity that current main carries into MCP loopback resolution. That can reuse a tool catalog or synced app resources across channel contexts that are currently kept distinct.
  - confidence: 0.84
