## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Route mcp.tools.call through tool policy hooks:** `src/gateway/server-methods/mcp.ts:146`
  - body: This new gateway tool-execution surface calls `tool.execute` directly. Existing MCP loopback and `tools.invoke` paths run `runBeforeToolCallHook` first so plugin approvals, trusted tool policies, adjusted params, diagnostics, and cancellation semantics are preserved; this path would let a WRITE-scope MCP Apps caller execute the same tool without those checks.
  - confidence: 0.94
- **[P1] Scope resource reads to the requested session:** `src/gateway/server-methods/mcp.ts:207`
  - body: `mcp.resources.list` returns the process-global registry after only syncing the caller's tool cache. Because the registry also supports file-backed resources, one READ-scope client can enumerate or read `ui://` resources left by another session/owner; key the registry by session or make these resources explicitly process-static before exposing them through session-shaped methods.
  - confidence: 0.9
- **[P2] Use the gateway runtime config snapshot:** `src/gateway/server-methods/mcp.ts:64`
  - body: Each MCP handler reloads config with `loadConfig()` instead of using `context.getRuntimeConfig()`. Gateway request handlers normally use the active runtime snapshot so reload lifecycle, cache invalidation, and method behavior stay consistent; rereading config in this hot path can scope tools against a different config than the running gateway.
  - confidence: 0.84
