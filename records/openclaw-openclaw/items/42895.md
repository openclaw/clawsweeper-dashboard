## Summary

Keep this PR open. Current main and the latest release still use the old home-directory fallback chain with no guarded Android/Termux `PREFIX` fallback, while the PR remains a focused fix for the linked Termux startup crash and has addressed the earlier review concerns around tilde expansion and non-Termux Android chroots.

## What This Changes

The PR adds a guarded Termux `PREFIX`-derived home fallback to `src/infra/home-dir.ts`, adds regression tests, and updates the environment docs and changelog.

## Best Possible Solution

Rebase and land this focused implementation, or a maintainer-equivalent patch, in the core home resolver with the same precedence guarantees: `OPENCLAW_HOME`, `HOME`, `USERPROFILE`, guarded Termux `PREFIX`, then `os.homedir()`. Keep the regression tests and docs/changelog update, and explicitly decide whether the memory-host SDK resolver should be aligned in the same PR or tracked separately.

## Reproduction Assessment

Yes. The linked bug gives real Termux logs, and current-main code inspection gives a focused unit-level reproduction: with `HOME` and `USERPROFILE` unset, Termux-like `PREFIX`/`ANDROID_DATA`, and `homedir()` returning `/home`, `resolveEffectiveHomeDir()` still returns `/home` on main.

## Solution Assessment

Yes for the core OpenClaw startup/config/workspace path. The PR changes the implicated resolver, preserves existing env precedence, covers the tilde-expansion edge case raised in review, and rejects generic Android chroot prefixes; the safer remaining step is rebase plus fresh validation, not a different design.
