## Summary

Keep open. Current main still has the reported usage aggregation bug, and this PR targets the right area, but the branch is conflicting, lacks real behavior proof, and needs rework to preserve current cache-aware/family usage semantics without introducing unbounded request-time loading.

## What This Changes

This PR changes gateway and Control UI usage aggregation/ranking so zero-cost token-heavy channels or agents surface and `sessions.usage` aggregates are intended to cover all matched sessions while returned rows stay limited.

## Best Possible Solution

Rebase or split a narrow fix on current main so aggregates and zero-cost rankings use all matched sessions, returned rows stay limited, cache-aware loading survives, overview diagnostics share one explicit scope, and redacted real usage proof is added.

## Reproduction Assessment

Yes. Source inspection on current main shows that creating more matching sessions than `limit` causes the handler to slice `mergedEntries` before the only aggregation loop, so totals can only include the returned subset.

## Solution Assessment

No. The direction is right, but the current PR shape needs a clean rebase and a narrower implementation that preserves current cache/family behavior and keeps overview diagnostics on one scope.
