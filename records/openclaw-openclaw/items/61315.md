## Summary

Keep this PR open. The underlying usage-dashboard bug is still present on current main and the PR has useful direction, but the current diff still leaves inconsistent overview scope in the UI and regresses the no-overview session-limit warning fallback; recent Usage Mosaic changes on main also make maintainer/author rework safer than closing or replacing it automatically.

## What This Changes

The PR changes gateway sessions.usage aggregation/types plus Control UI usage helpers/rendering/tests to add matched-session overview metadata, token-aware channel/agent ranking, token-primary Top Agents/Top Channels display, and regression coverage for limit-trimmed aggregates.

## Best Possible Solution

Rework the branch into a consistent usage contract: limit should trim only returned rows, overview totals/aggregates/diagnostics should share the same documented matched-session scope, old responses without overview should keep a cap-warning fallback, zero-cost channel/agent rankings should stay visible by token signal, and channel attribution should stay coordinated with #52521 rather than being mixed into this PR.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to seed more matched sessions than the requested limit with zero-cost token-heavy usage, then compare sessions.usage totals/aggregates and unfiltered Control UI overview diagnostics against the returned session subset; the provided PR tests cover part of this path, and current-main code inspection confirms the limited-scope behavior.

## Solution Assessment

No. The PR's direction is maintainable, but the current patch is not the best final form because it still mixes full overview aggregates with limited-session peak/mosaic diagnostics and drops the no-overview limit-warning fallback; the safer alternative is an explicit overview-scope contract with compatible UI fallbacks and bounded aggregation I/O.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep overview diagnostics on the same scope:** `ui/src/ui/views/usage.ts:306-310`
  - body: When `useGatewayOverview` is true, these changed lines switch `activeAggregates` to gateway-wide data, but the render path still builds peak error hours and the mosaic from `aggregateSessions`, which is the limited returned session list. With `matchedSessions > returnedSessions`, the overview combines full totals/error-day data with peak/mosaic diagnostics from a truncated subset, so the cards can point users to the wrong hours. Feed those diagnostics from overview-scoped data or keep the whole overview on returned rows.
  - confidence: 0.84
- **[P2] Preserve the limit warning fallback:** `ui/src/ui/app-render-usage-tab.ts:26-28`
  - body: When `overview` is missing, this expression falls back to comparing `sessions.length` to itself, so it can never flag truncation. Mixed-version or cached `sessions.usage` responses without overview metadata would lose the previous `>= 1000` warning and make capped Usage data appear complete.
  - confidence: 0.9
