## Summary

Keep open: current main still has the reported usage aggregation gaps, and this external PR is a useful bug-fix attempt, but the latest branch still mixes matched-session overview data with limited-row diagnostics, loses the legacy cap warning fallback, and lacks required real behavior proof.

## What This Changes

The PR changes gateway `sessions.usage` and Control UI usage helpers to add overview metadata, aggregate matched sessions beyond the returned limit, rank zero-cost channel/agent usage by tokens, update insight card display, and add regression tests.

## Best Possible Solution

Rework the usage contract so `limit` trims only returned rows, overview totals and diagnostics share one matched-session scope, legacy no-overview cap warnings remain, zero-cost token-heavy rankings stay visible, and channel attribution aligns with the linked attribution PR.

## Reproduction Assessment

Yes, as a source-level path: seed more matched sessions than `limit`, include zero-cost token-heavy channel or agent usage, and include peak-hour data outside the returned slice. Current main aggregates only `limitedEntries` and cost-sorts zero-cost rankings; I did not run tests in this read-only review.

## Solution Assessment

No. The direction is useful, but this patch is not the best final solution because it leaves full-scope overview cards next to limited-row peak-hour/mosaic diagnostics and removes the no-overview cap-warning fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep overview diagnostics on the matched-session scope:** `ui/src/ui/views/usage.ts:309-313`
  - body: In the no-filter path, this switches `activeAggregates` to gateway-wide data, but the later peak-hour and mosaic calls still use `aggregateSessions`, which is only the limited returned rows when `matchedSessions > returnedSessions`. That can show full overview totals beside diagnostics from a truncated slice, so the diagnostics need the same matched-session scope or an explicit limited-data fallback.
  - confidence: 0.86

  - body: When `overview` is absent, this expression falls back to comparing `sessions.length` with itself, so capped responses without the new field can no longer show the previous `>= 1000` warning. Keep a legacy fallback for mixed-version or cached responses that do not include `overview`.
  - confidence: 0.88
