## Summary

This PR still targets a real unresolved usage-dashboard bug, so it should not be cleaned up as obsolete. Current main still caps `sessions.usage` aggregates by `limit`, but this stale branch is dirty, lacks real behavior proof, and introduces request-path availability and compatibility blockers that need a refresh before merge.

## What This Changes

The PR changes `sessions.usage` aggregation/ranking and Control UI usage insight rendering so zero-cost token-heavy channels/agents stay visible and aggregates are intended to cover all matched sessions while `limit` only trims returned rows.

## Best Possible Solution

Refresh or replace the branch on current main with a bounded, cache-aware fix for https://github.com/openclaw/openclaw/issues/76496 that keeps `limit` scoped to returned rows, preserves older-response UI fallback behavior, and includes real Gateway/Control UI proof.

## Reproduction Assessment

Yes from source. Current main slices `mergedEntries` into `limitedEntries` before loading and aggregating session usage, so date ranges with more matches than the requested limit undercount aggregates; this read-only review did not execute a live repro.

## Solution Assessment

No as-is. The fix direction is right, but the branch is not the best implementation because it scans every matched session synchronously and weakens older-response UI fallback behavior; the safer fix should reuse current cache/background aggregation paths.
