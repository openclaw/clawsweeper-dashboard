## Summary

Keep open: this PR still targets real current-main usage aggregation and ranking problems, but the branch is now conflicting, lacks real behavior proof, and needs rework around cache-aware loading and consistent overview scope before it is merge-ready.

## What This Changes

The branch changes gateway `sessions.usage` and Control UI usage helpers to rank zero-cost agents/channels by tokens, add an overview response scope, and compute aggregates from all matched sessions while returning a limited session list.

## Best Possible Solution

Rebase or split the gateway aggregate-limit fix onto current main while preserving cacheStatus, family grouping, and cache-aware loading, then align the Control UI ranking and overview diagnostics with focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes from source inspection, though I did not execute a live repro in this read-only pass. Current main slices `mergedEntries` before the only usage-loading and aggregation loops, so more matching sessions than `limit` produces capped totals.

## Solution Assessment

No, not in the current branch shape. The right fix is to decouple aggregate scope from returned-row paging while preserving current cache-aware loading, warning fallback, and a single consistent UI diagnostics scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve bounded cache-aware usage loading:** `src/gateway/server-methods/usage.ts:613`
  - body: Changing the loop to walk every `mergedEntries` item means a request with `limit: 50` can now load usage for every matched session before returning the first page. On current main this also conflicts with the cache-aware `loadSessionCostSummaryFromCache` path and `cacheStatus`; keep aggregate computation from becoming unbounded foreground I/O.
  - confidence: 0.85
- **[P2] Keep overview diagnostics on one scope:** `ui/src/ui/views/usage.ts:306-311`
  - body: The no-filter view switches `activeAggregates` to gateway-wide aggregates, but `buildPeakErrorHours(...)` and the Usage Mosaic still use `aggregateSessions`, which is only the returned limited rows when `matchedSessions > returnedSessions`. That mixes full totals with partial hourly diagnostics and can point users at the wrong peak hours or mosaic buckets.
  - confidence: 0.82
- **[P2] Preserve the limit-warning fallback:** `ui/src/ui/app-render-usage-tab.ts:24-26`
  - body: When `overview` is absent, this expression falls back to comparing `sessions.length` with itself, so it can never show the capped-result warning. Mixed-version or older gateway responses without `overview` would silently look complete even when the 1000-row cap was hit.
  - confidence: 0.9
