## Summary

Keep this PR open: current main still has the reported usage aggregation/ranking problem, but the branch is not merge-ready because it lacks real behavior proof and still has scope-consistency regressions in the usage overview UI.

## What This Changes

The PR updates gateway `sessions.usage`, shared usage aggregators, and Control UI usage views/tests to rank zero-cost channel/agent usage by tokens and add matched-session overview metadata while limiting returned rows.

## Best Possible Solution

Rework the usage contract so `limit` trims only returned rows while overview totals, insight diagnostics, warnings, and channel attribution all use explicit and consistent scopes.

## Reproduction Assessment

Yes, source-level: seed more matched sessions than `limit` and include zero-cost token-heavy channel or agent usage; current main aggregates only `limitedEntries` and cost-sorts zero-cost rankings.

## Solution Assessment

No. The direction is useful, but this patch is not the best final solution because it leaves full-scope overview aggregates next to limited-row peak-hour/mosaic diagnostics and removes the no-overview cap-warning fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep overview diagnostics on the matched-session scope:** `ui/src/ui/views/usage.ts:309-313`
  - body: When the no-filter view uses gateway-wide `activeAggregates`, the peak-hour and Usage Mosaic calls below still use `aggregateSessions`, which is only the limited returned row list when `matchedSessions > returnedSessions`. That can show full overview totals beside diagnostics from a truncated subset, so these diagnostics need the same matched-session scope or a clear limited-data fallback.
  - confidence: 0.86
- **[P2] Preserve the no-overview limit warning fallback:** `ui/src/ui/app-render-usage-tab.ts:26-28`
  - body: When `overview` is absent, this expression falls back to comparing `sessions.length` with itself, so capped legacy or cached responses can no longer show the previous `>= 1000` warning. Keep a fallback for responses that do not include `overview`.
  - confidence: 0.88
