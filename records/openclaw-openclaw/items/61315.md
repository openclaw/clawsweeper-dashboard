The PR updates `sessions.usage` and the Control UI usage view so aggregate totals cover all matched sessions while returned rows stay limited, and zero-cost channel/agent rankings can surface token-heavy entries.

## Best Possible Solution

Rebase or split a narrow fix on current main so aggregates and zero-cost rankings use all matched sessions, returned rows stay limited, cache-aware family loading survives, overview diagnostics share one explicit scope, and redacted real usage proof is added.

## Reproduction Assessment

Yes. Source inspection on current main shows that creating more matching sessions than `limit` causes the handler to slice `mergedEntries` before the only usage aggregation path, so totals can only include the returned subset.

## Solution Assessment

No. The direction is right, but the current PR shape needs a clean rebase and narrower implementation that preserves current cache/family behavior and keeps overview diagnostics on one scope.
