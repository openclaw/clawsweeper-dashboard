## Summary

Keep open: current main still has the limited-scope and cost-ranked usage behavior this PR is trying to fix, but the branch still mixes overview scopes in the Control UI and regresses the legacy limit warning fallback, so it is useful but not ready to merge.

## What This Changes

The PR updates gateway `sessions.usage`, shared usage aggregation helpers, Control UI usage rendering, and regression tests to add overview metadata, token-aware channel/agent ranking, token-primary insight cards, and full-matched-session aggregates while limiting returned session rows.

## Best Possible Solution

Keep the branch direction but rework it into a consistent usage contract: `limit` should trim only returned rows, overview totals/aggregates/diagnostics should share one documented matched-session scope, older responses without `overview` should preserve the cap warning fallback, token-heavy zero-cost channel/agent rankings should stay visible, and any full-range summary reads should be bounded or cached. Coordinate origin-channel attribution separately with #52521.

## Reproduction Assessment

Yes. A high-confidence path is to seed more matched sessions than the requested `limit`, including zero-cost token-heavy channel/agent usage and peak-hour data outside the returned slice, then compare `sessions.usage` totals/aggregates and the Control UI overview against the returned session subset; current main code inspection and the PR tests cover the core setup.

## Solution Assessment

No. The PR's core approach is plausible, but the current patch is not the best final solution because it still mixes full overview aggregates with limited-row diagnostics and loses the no-overview limit-warning fallback; a safer fix should make overview scope explicit and preserve compatibility behavior.
