## Summary

Keep open: the underlying usage aggregation bug is still present on current main, and this PR is a plausible fix path, but it still needs contributor real behavior proof and has two unresolved UI scope/fallback regressions.

## What This Changes

The PR changes gateway `sessions.usage` aggregation/ranking, shared usage-ranking helpers, Control UI usage rendering/types, and regression tests for zero-cost channel/agent visibility and limit-scoped aggregates.

## Best Possible Solution

Finish a usage contract where `limit` trims only returned rows while totals, rankings, warnings, duration, peak-hour, and mosaic diagnostics use explicit matched or limited scopes, then add focused tests and redacted real dashboard proof.

## Reproduction Assessment

Yes, source-level: seed more matched sessions than `limit` and include zero-cost token-heavy channel or agent usage. Current main aggregates only `limitedEntries` and cost-sorts the relevant rankings, so the failure is clear without running tests in this read-only review.

## Solution Assessment

No. The PR direction is useful, but this patch still mixes full-scope overview aggregates with limited-row diagnostics and drops the no-overview cap-warning fallback; the safer fix is to align each metric with an explicit scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

  - body: When the no-filter view uses gateway-wide `activeAggregates`, the peak-hour and Usage Mosaic paths still derive from `aggregateSessions`, which is only the limited returned row list when `matchedSessions > returnedSessions`. That can show full overview totals beside diagnostics from a truncated subset.
  - confidence: 0.86

  - confidence: 0.88
