## Summary

Keep open. Current main still has the usage aggregation/ranking bug, and this PR is useful but not merge-ready because it lacks real behavior proof and still has two UI scope/fallback regressions.

## What This Changes

The PR changes `sessions.usage` aggregation/ranking, shared usage ranking helpers/types, Control UI usage overview rendering, and regression tests for limit-scoped totals plus zero-cost channel/agent visibility.

## Best Possible Solution

Revise or split the PR so gateway aggregates use all matched sessions while `limit` only pages returned rows, zero-cost rankings surface token-heavy channels/agents, and UI diagnostics/warnings use explicit matched-vs-returned scopes with focused tests and real proof.

## Reproduction Assessment

Yes, source-reproducible. Current main slices `mergedEntries` before the aggregation loop and cost-sorts channels, so a range with more matching sessions than `limit` plus a zero-cost token-heavy channel or agent can undercount totals and hide the token-heavy entry.

## Solution Assessment

No. The PR addresses valid bugs, but the current diff still mixes full-scope overview aggregates with limited-row peak-hour/mosaic diagnostics and regresses the no-overview limit-warning fallback.
