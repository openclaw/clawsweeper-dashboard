## Summary

Keep open: the PR targets a real usage-dashboard aggregation bug, but it still has two blocking P2 correctness regressions and lacks required real behavior proof from the contributor’s environment.

## What This Changes

The branch changes gateway `sessions.usage` aggregation, shared usage-ranking helpers, Control UI usage rendering/types, and regression tests so zero-cost token-heavy channel/agent usage can surface and aggregates can cover all matched sessions despite a returned-row limit.

## Best Possible Solution

Rework the usage contract so `limit` trims only returned rows while overview totals, warnings, peak-hour/mosaic diagnostics, and channel/agent ranking all use explicit consistent scopes, then add real dashboard proof.

## Reproduction Assessment

Yes, source-level: seed more matched sessions than `limit` and include zero-cost token-heavy channel or agent usage. Current main aggregates only `limitedEntries` and cost-sorts zero-cost rankings, so the reported bug is clear from source and tests even though I did not run tests in this read-only review.

## Solution Assessment

No. The direction is useful, but this patch still mixes full-scope aggregates with limited-row diagnostics and regresses the no-overview cap warning fallback; the safer fix is to carry matched-scope diagnostics or make limited scope explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep overview diagnostics on the matched-session scope:** `ui/src/ui/views/usage.ts:309-313`
  - body: When the no-filter view uses gateway-wide `activeAggregates`, the peak-hour and Usage Mosaic calls below still use `aggregateSessions`, which is only the limited returned row list when `matchedSessions > returnedSessions`. That can show full overview totals beside diagnostics from a truncated subset, so these diagnostics need the same matched-session scope or a clear limited-data fallback.
  - confidence: 0.86
- **[P2] Preserve the no-overview limit warning fallback:** `ui/src/ui/app-render-usage-tab.ts:26-28`
  - body: When `overview` is absent, this expression falls back to comparing `sessions.length` with itself, so capped legacy or cached responses can no longer show the previous `>= 1000` warning. Keep a fallback for responses that do not include `overview`.
  - confidence: 0.88
