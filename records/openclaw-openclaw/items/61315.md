## Summary

Keep open: current main still caps `sessions.usage` totals and aggregate buckets at `limit`, so the underlying bug remains useful to fix. This PR is not merge-ready as written because it predates the current cache-backed usage path, can reintroduce large request-path transcript scans, lacks real behavior proof, and leaves UI overview compatibility/scope gaps.

## What This Changes

The PR changes gateway and Control UI usage aggregation so session totals/insights include all matched sessions while the returned session list remains limited, ranks zero-cost channel/agent usage by tokens, adds an `overview` response field, and adds Vitest regression coverage.

## Best Possible Solution

Land a current-main-native fix that aggregates all matched sessions through the existing cache/background summary path, limits only returned rows, keeps UI overview diagnostics consistent, and preserves fallback behavior for older gateway responses.

## Reproduction Assessment

Yes from source for the central bug: current main slices to `limitedEntries` before loading and aggregating `sessions.usage`. I did not run artifact-producing tests in this read-only review.

## Solution Assessment

No as-is. The best fix is to preserve current main's durable/background cache path while computing aggregates from all matched sessions and keeping UI diagnostics/fallbacks on one consistent scope.
