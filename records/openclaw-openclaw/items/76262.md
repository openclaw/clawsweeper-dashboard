## Summary

Keep open. The draft SDK migration has useful direction, but it currently regresses current-main MS Teams webhook auth, proactive routing metadata, dual route fallback, and Docker port behavior; the PR body also acknowledges the custom JWT and tenant/AAD shims still need to be re-applied.

## What This Changes

The PR migrates the Microsoft Teams plugin toward `@microsoft/teams.apps`, replacing adapter, webhook auth/routing, proactive send, streaming, card-action, test, and Docker surfaces.

## Best Possible Solution

Keep the draft open and layer the SDK migration only after preserving current JWT audience/app-binding validation, proactive conversation reference metadata, dual route registration, and Docker/docs port alignment, followed by targeted MS Teams validation and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to compare the PR patch with current main's `sdk.test.ts` global-audience tests, `buildConversationReference` proactive metadata tests, docs/default port values, and the pinned SDK source.

## Solution Assessment

No. Full SDK adoption may be the right direction, but this patch is not the best mergeable solution until it preserves current main's compatibility and security invariants; an incremental migration with the shims retained is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the Teams JWT validator:** `extensions/msteams/src/monitor.ts:247-253`
  - body: This switches the webhook to SDK-owned validation through `ExpressAdapter`, but the pinned SDK validator's default audience list omits `https://api.botframework.com`. Current main explicitly accepts those Bot Framework global-audience tokens with app binding, so valid Teams webhooks can start returning 401.
  - confidence: 0.92
- **[P1] Forward proactive send routing metadata:** `extensions/msteams/src/send.ts:371-372`
  - body: Calling `app.send(baseRef.conversation.id, activity)` discards the stored tenantId, aadObjectId, user, bot, and serviceUrl metadata that current main threads through the proactive reference. That metadata is covered by the #58774 regression tests because missing tenant routing can produce HTTP 403 for proactive Teams sends.
  - confidence: 0.94
- **[P2] Expose the port MS Teams actually listens on:** `docker-compose.yml:46`
  - body: The compose file publishes container port 3000, but the plugin and docs still default `channels.msteams.webhook.port` to 3978. A default Docker Compose deployment would expose a port where the MS Teams webhook is not listening.
  - confidence: 0.87
- **[P2] Keep the standard /api/messages fallback:** `extensions/msteams/src/monitor.ts:253`
  - body: Passing only `messagingEndpoint: configuredPath` leaves the SDK with a single configured route. Current main also registers `/api/messages`, so deployments with a custom path that still receive Bot Framework traffic on the standard endpoint would start returning 404.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff changes the MS Teams webhook authentication boundary and should not merge until the current custom validator invariants are preserved or explicitly replaced.

Concerns:

- **[medium] MS Teams JWT validation path changed:** `extensions/msteams/src/monitor.ts:247`
  - body: The patch removes current main's custom Bot Framework JWT validator and delegates to SDK validation with a different default audience contract; this can reject valid global-audience tokens and changes a security-sensitive webhook boundary.
  - confidence: 0.9
- **[medium] SDK route owns body parsing before equivalent auth proof:** `extensions/msteams/src/monitor.ts:253`
  - body: Current main performs full JWT validation before JSON body parsing and uses the shared webhook body limit; the SDK `ExpressAdapter` route parses JSON internally, so the migration needs owner review to prove equivalent ingress hardening.
  - confidence: 0.78
