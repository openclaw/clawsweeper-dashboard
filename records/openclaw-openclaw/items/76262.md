## Summary

Keep open. The SDK migration is active and has supplied live proof, but current head still replaces SDK sign-in defaults and routes several proactive/update paths through the app-level Teams service URL instead of the stored conversation reference.

## What This Changes

Migrates the Microsoft Teams plugin to @microsoft/teams.apps 2.0.10, replacing Bot Framework adapter/auth/streaming/invoke/proactive messaging code and updating tests, Docker defaults, package metadata, and plugin contracts.

## Best Possible Solution

Keep the PR open for author and Teams-domain maintainer changes that preserve SDK SSO fallback semantics and stored serviceUrl routing, or explicitly approve a different cloud routing policy before rebase and CI.

## Reproduction Assessment

Yes. The blockers are source-reproducible from PR head and @microsoft/teams.apps@2.0.10 source: the PR registers user sign-in routes that replace system defaults, and app.send/app.reply build refs from app-level serviceUrl.

## Solution Assessment

No. The SDK migration direction is plausible, but this is not the safest merge path until SSO consent fallback and stored serviceUrl compatibility are preserved or explicitly accepted by maintainers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep SDK sign-in defaults active:** `extensions/msteams/src/monitor.ts:448-452`
  - body: These user routes replace the SDK system routes. The custom handler returns void, so consent-required token exchange failures are wrapped as a generic 200 instead of the SDK's 412 fallback; persist tokens from the SDK signin event without replacing the defaults, or return the exact SDK semantics.
  - confidence: 0.93
- **[P2] Preserve the stored serviceUrl for reply fallbacks:** `extensions/msteams/src/messenger.ts:535-536`
  - body: The revoked-context fallback now calls app.reply/app.send with only a conversation id. The SDK then uses the app-level serviceUrl, whereas current behavior carries the stored conversation reference serviceUrl, so regional or sovereign threaded replies can be sent to the wrong Connector endpoint.
  - confidence: 0.86
- **[P2] Preserve the stored serviceUrl for proactive activity sends:** `extensions/msteams/src/send.ts:440`
  - body: This builds baseRef from the stored conversation reference but discards baseRef.serviceUrl when calling app.send. Polls, adaptive cards, and file-consent cards sent out of context should use a ref-scoped SDK/client path that carries the stored Connector serviceUrl.
  - confidence: 0.85
- **[P2] Preserve the stored serviceUrl for edits and deletes:** `extensions/msteams/src/send.ts:606`
  - body: The edit/delete path calls app.api.conversations.activities(...) rooted at the app's configured serviceUrl instead of the stored conversation serviceUrl. Proactive updates and deletes can therefore target the wrong Connector endpoint for existing conversations.
  - confidence: 0.84
