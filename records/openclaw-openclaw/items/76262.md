## Summary

Keep open. The PR is active and useful, but current head still has source-level blockers around Teams SSO fallback semantics, proactive Connector routing, legacy JWT audience compatibility, and dependency freshness scope.

## What This Changes

The PR migrates the Microsoft Teams plugin to `@microsoft/teams.apps`, rewiring auth, streaming, invoke handling, proactive sends, tests, dependencies, and Docker defaults.

## Best Possible Solution

Land a compatibility-preserving Teams SDK migration that keeps SDK-owned sign-in semantics or exact 412 parity, preserves current auth audience and routing contracts unless maintainers approve a documented deprecation, and scopes dependency freshness exceptions to reviewed versions.

## Reproduction Assessment

Yes. The blockers are source-reproducible: the PR head registers replacement sign-in routes, `app.send`/`app.api` use SDK app-level routing, and the auth test asserts rejection of an audience current main accepts.

## Solution Assessment

No. The SDK migration direction is plausible, but this is not the safest merge shape until it truly preserves SDK sign-in defaults or 412 parity, keeps current proactive routing/auth compatibility, and narrows the dependency freshness bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep SDK sign-in system routes active:** `extensions/msteams/src/monitor.ts:448-452`
  - body: The PR still registers user routes for `signin.token-exchange` and `signin.verify-state`. The Teams SDK removes its system route when a user route has the same name, so this disables the SDK 200/412 consent fallback that the PR body and latest proof say remain active. Remove these routes and persist tokens from the SDK `signin` event, or implement exact SDK fallback parity explicitly.
  - confidence: 0.94
- **[P2] Preserve stored serviceUrl for proactive calls:** `extensions/msteams/src/send.ts:440`
  - body: This path sends through `app.send(baseRef.conversation.id, activity)`, but SDK `App.send` builds the proactive reference from the app-level serviceUrl. Current main's adapter uses the stored `conversationReference.serviceUrl`, so regional, sovereign, or tenant-specific Connector endpoints can be ignored; the same routing issue applies to edit/delete and proactive thread sends.
  - confidence: 0.9
- **[P2] Do not drop the legacy Bot Framework audience silently:** `extensions/msteams/src/auth-coverage.test.ts:108-121`
  - body: This new test locks in rejecting `aud=https://api.botframework.com`, while current main explicitly accepts that audience and recent history/changelog call it supported compatibility. Upgrading existing bots can turn valid current deployments into 401s unless this gets a shim, a config-gated deprecation, or an explicit maintainer security decision.
  - confidence: 0.84
- **[P3] Scope the Teams SDK release-age bypass:** `pnpm-workspace.yaml:22`
  - body: Adding unversioned `@microsoft/teams.apps` to `minimumReleaseAgeExclude` bypasses the workspace freshness guard for future SDK releases, not just the reviewed 2.0.10 bump. Use an exact-version exception or remove it once the normal release-age window has elapsed.
  - confidence: 0.82
