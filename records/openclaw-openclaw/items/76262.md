## Label Justifications

- `P1`: The remaining blocker can break real Teams delivery during upgrade for non-public or custom Connector deployments.

## Summary

Keep open: the latest cloud/serviceUrl work covers message-tool proactive operations, but the revoked or contextless reply fallback still sends through the Teams SDK app-level service URL without the new stored-serviceUrl guard, leaving the remaining non-public Teams routing blocker in place.

## What This Changes

The branch migrates the msteams plugin to `@microsoft/teams.apps`, replacing custom adapter, JWT, streaming, and invoke plumbing while adding cloud/serviceUrl config, docs, Docker, dependency, and test updates.

## Best Possible Solution

Apply the cloud/serviceUrl boundary check at the shared app-level proactive send seam before every `app.send` or `app.reply` fallback, while keeping public cloud as the default and the SDK-based migration direction.

## Reproduction Assessment

Yes from source for the remaining PR defect: the ctx-less/revoked fallback calls `app.reply` or `app.send` directly, while the new guard is only in `resolveMSTeamsSendContext`. I did not live-reproduce a sovereign-cloud tenant failure.

## Solution Assessment

No. The SDK cloud/serviceUrl direction is maintainable, but the guard needs to live at the shared app-level proactive fallback boundary rather than only in the explicit send-context path.
