## Summary

Keep this PR open: the latest head fixes the earlier SDK sign-in-route issue and includes structured live Teams proof, but source and dependency review still show a proactive `serviceUrl` compatibility regression plus an unscoped dependency freshness bypass.

## What This Changes

The PR migrates the Microsoft Teams plugin from Bot Framework adapter plumbing to `@microsoft/teams.apps`, rewiring auth, streaming, invoke handling, proactive sends/edits/deletes, tests, Docker defaults, and package metadata.

## Best Possible Solution

Land a Teams SDK migration that preserves current Connector routing compatibility, or introduces a documented explicit cloud/serviceUrl configuration, and scopes dependency freshness exceptions to reviewed versions.

## Reproduction Assessment

Yes. The remaining blockers are source-reproducible: PR head calls SDK app-level proactive routing while current main carries the stored conversation reference, and pnpm documents that package-name release-age excludes apply to all versions.

## Solution Assessment

No. The SDK migration direction is plausible and the SSO proof now matches the code, but the branch should preserve current serviceUrl/cloud routing compatibility and narrow the freshness exception before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the stored serviceUrl for proactive sends:** `extensions/msteams/src/send.ts:440`
  - body: `app.send()` builds its outgoing reference from the SDK app's single configured service URL, so this drops the stored `conversationReference.serviceUrl` that current main passes through `continueConversation`. Sovereign, GCC/DoD, regional, or tenant-specific Connector endpoints can misroute proactive sends, edits, deletes, and threaded fallbacks unless the SDK call is scoped to the stored service URL or an explicit cloud/service URL config preserves the old routing contract.
  - confidence: 0.9
- **[P3] Scope the Teams SDK freshness bypass:** `pnpm-workspace.yaml:22`
  - body: pnpm applies `minimumReleaseAgeExclude` by package name to all future versions, and this package-wide entry bypasses the workspace freshness guard beyond the reviewed 2.0.10 bump. Use an exact-version exception such as `@microsoft/teams.apps@2.0.10`, add any matching exact companion package needed, or remove the exception once the normal age window has passed.
  - confidence: 0.85
