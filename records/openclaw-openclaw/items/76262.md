## Summary

Keep open for normal PR review: this is a recent, substantive Microsoft Teams SDK migration that is not on current main, has supplied live Teams proof and green checks, and I found only one low-priority correctness issue in the new User-Agent plumbing.

## What This Changes

The PR migrates the bundled Microsoft Teams plugin from custom Bot Framework plumbing to `@microsoft/teams.apps`, updating Teams streaming, invoke, SSO, proactive send/edit/delete paths, docs, Docker port mapping, dependency metadata, and msteams tests.

## Best Possible Solution

Land the SDK migration after fixing or removing the unsupported User-Agent claim, while preserving the documented public-cloud boundary and the existing auth/invoke/streaming regression coverage.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR passes the OpenClaw User-Agent through the SDK ClientOptions path, and the pinned SDK source overwrites that header. I did not independently reproduce the broader Teams live flows; those are covered by supplied live proof and CI.

## Solution Assessment

Mostly yes: moving the Teams plugin onto the supported SDK is a maintainable direction and prior reviewer blockers appear addressed. The User-Agent observability addition should use a SDK-supported merged-client path or drop the claim before merge.
