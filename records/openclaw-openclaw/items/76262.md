## Summary

Keep open: current PR head still changes established Teams webhook authentication, request parsing, edit/delete routing, and dependency freshness policy in ways that need maintainer review before merge.

## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework/JWT/REST/streaming plumbing to `@microsoft/teams.apps`, updates Teams send/edit/delete, streaming, Adaptive Card, Docker, dependency, and test surfaces.

## Best Possible Solution

Keep the SDK migration under review and preserve the current Teams auth, pre-parse validation, stored service URL, visible allowlist diagnostics, and dependency freshness contracts unless maintainers explicitly approve a documented compatibility change.

## Reproduction Assessment

Yes for the review blockers: source inspection of current main, the PR head, and `@microsoft/teams.apps@2.0.10` provides a high-confidence path. I did not run a live Teams tenant, so this is source-reproducible rather than live reproduced.

## Solution Assessment

No. The SDK migration may be a reasonable direction, but this patch is not yet the narrowest maintainable solution because it changes established auth, request parsing, routing, diagnostic, and supply-chain contracts without preserving parity or an explicit maintainer decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the Bot Framework token contract:** `extensions/msteams/src/monitor.ts:252-255`
  - body: Delegating inbound validation to the SDK route drops the current custom validator that accepts `https://api.botframework.com` global-audience tokens with app binding. `@microsoft/teams.apps@2.0.10` adds the v1 issuer path, but its service-token validator still does not add that audience, so deployed bots using tokens current main accepts can start returning 401.
  - confidence: 0.9
- **[P2] Keep full JWT validation before body parsing:** `extensions/msteams/src/monitor.ts:252-255`
  - body: The retained pre-gate only checks for a Bearer-shaped header, while the SDK `ExpressAdapter` registers `express.json()` before `HttpServer.authorize()`. This removes current main's full JWT-before-body-parse behavior and explicit Teams webhook body limit for attacker-supplied Bearer requests.
  - confidence: 0.92
- **[P2] Use stored service URLs for edits and deletes:** `extensions/msteams/src/send.ts:606`
  - body: Edits and deletes now call the SDK API client with only the conversation ID, but the SDK app API is rooted at its configured/default service URL rather than the stored inbound Teams `serviceUrl`. Regional or tenant-specific Connector conversations can therefore target the wrong endpoint even when the stored reference has the correct URL.
  - confidence: 0.83
- **[P2] Keep allowlist resolution failures visible:** `extensions/msteams/src/monitor.ts:198`
  - body: This downgrades Graph allowlist resolution failures from `runtime.error` to `runtime.log`. Current main intentionally logs these as errors because unresolved Graph IDs leave the Teams allowlist degraded, so operators may miss access-control misconfiguration.
  - confidence: 0.9
- **[P2] Narrow the Teams SDK release-age bypass:** `pnpm-workspace.yaml:22`
  - body: The new `@microsoft/teams.*` pattern bypasses the release-age guard for every future package in that scope, not just the reviewed SDK version in this migration. Use a narrower exception or wait for the normal release-age window unless maintainers explicitly approve the broader supply-chain policy change.
  - confidence: 0.88
