## Summary

Keep open. This draft is useful, and the author explicitly called out SDK-side context, but source review still shows regressions in the Microsoft Teams webhook auth, pre-parse validation, route fallback, and proactive-reference contracts, and the external PR lacks inspectable real behavior proof.

## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework adapter/JWT/streaming plumbing to `@microsoft/teams.apps`, rewiring auth, routing, streaming, polls, send/edit/delete paths, tests, and Docker port metadata.

## Best Possible Solution

Keep the draft open and either layer the SDK migration behind current auth, proactive-reference, route, and changelog contracts with proof, or split a narrower owner-approved migration that documents any intentional compatibility retirements.

## Reproduction Assessment

Yes. A high-confidence source reproduction compares PR head with current main and the published `@microsoft/teams.apps@2.0.9` source: the PR routes through SDK auth and send paths that omit current main's validator ordering and stored-reference metadata; no live Teams setup was run.

## Solution Assessment

No. Full SDK adoption may still be a good direction, but this patch is not the best merge path until it preserves current compatibility/security contracts or carries explicit owner-approved retirements backed by inspectable Teams proof.
