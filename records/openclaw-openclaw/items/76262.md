## Summary

Keep open. This draft PR changes the security-sensitive Microsoft Teams SDK boundary and still regresses current main's JWT audience contract, proactive routing metadata, `/api/messages` fallback route, and Docker port default; the author already acknowledged the JWT and proactive metadata items as rebase regressions to reapply.

## What This Changes

The PR migrates the MS Teams plugin from custom Bot Framework adapter/JWT/proactive/streaming plumbing to `@microsoft/teams.apps` SDK primitives and adjusts related tests plus Docker metadata.

## Best Possible Solution

Keep the draft open and reconcile the SDK migration by preserving current JWT validation invariants, proactive tenant/AAD routing, dual route registration, and Docker/docs port alignment before targeted MS Teams validation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to compare the PR diff with current main's `sdk.test.ts`, `messenger.test.ts`, route registration, docs/default port values, and the pinned SDK 2.0.9 source.

## Solution Assessment

No. Full SDK adoption may be the right direction, but this patch is not the best mergeable solution until it retains the current compatibility and security shims; an incremental migration with those invariants preserved is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the Teams JWT validator:** `extensions/msteams/src/monitor.ts:246-253`
  - body: This switches the webhook to SDK-owned validation through `ExpressAdapter`, but the pinned SDK validator does not include `https://api.botframework.com` in its default audience list. Current main explicitly accepts those Bot Framework global-audience tokens with app binding, so valid Teams webhooks can start returning 401.
  - confidence: 0.92
- **[P1] Forward proactive send routing metadata:** `extensions/msteams/src/messenger.ts:538`
  - body: Calling `params.app.send(conversationId, activity)` rebuilds the send reference without the stored tenantId, aadObjectId, user, and serviceUrl metadata that current main threads through `buildConversationReference`. That metadata is covered by the #58774 regression tests because missing tenant routing can produce HTTP 403 for proactive Teams sends.
  - confidence: 0.94
- **[P2] Expose the MS Teams webhook default port:** `docker-compose.yml:46`
  - body: The compose file publishes container port 3000, but the plugin and docs still default `channels.msteams.webhook.port` to 3978. A default Docker Compose deployment would expose a port where the MS Teams webhook is not listening.
  - confidence: 0.87
- **[P2] Keep the standard `/api/messages` fallback:** `extensions/msteams/src/monitor.ts:252`
  - body: Passing only `messagingEndpoint: configuredPath` leaves the SDK with a single configured route. Current main also registers `/api/messages`, so deployments with a custom path that still receive Bot Framework traffic on the standard endpoint would start returning 404.
  - confidence: 0.83

## Security Review

  - confidence: 0.9
