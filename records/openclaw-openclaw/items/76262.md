## Label Justifications

- `P1`: This migration affects real Microsoft Teams reply delivery and can break non-public Teams deployments at runtime.
- `merge-risk: 🚨 compatibility`: The PR changes the upgrade contract from per-conversation stored Connector serviceUrls to SDK app-level endpoint selection.

- `merge-risk: 🚨 auth-provider`: The SDK cloud/serviceUrl choice controls Bot Framework auth/token/cloud routing for sovereign Teams environments.

## Summary

Keep open: the Teams SDK migration has strong public-cloud proof and several prior blockers are resolved, but the latest head still leaves ctx-less/revoked inbound reply fallback paths using SDK app-level sends without the new serviceUrl boundary guard, so non-public Teams deployments can misroute instead of failing closed.

## What This Changes

This PR migrates the msteams plugin to `@microsoft/teams.apps`, replaces custom adapter/JWT/streaming/invoke plumbing, adds Teams cloud/serviceUrl config and docs, and updates Docker, dependency, and regression-test coverage.

## Best Possible Solution

Keep the SDK migration direction, but make one shared msteams app-level send boundary validate stored conversation serviceUrl against the configured SDK cloud/serviceUrl before every proactive fallback can call `app.send` or `app.reply`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure the SDK app for default public cloud with a stored non-public Teams serviceUrl, then trigger a ctx-less or revoked reply fallback that calls `sendMSTeamsMessages` without passing through `resolveMSTeamsSendContext`. I did not live-reproduce a sovereign-cloud tenant failure.

## Solution Assessment

No, the current PR is not yet the best safe solution because the new boundary check is not shared by every app-level send path. The safer fix is one common validated send/reply helper used by explicit sends, revoked-context replies, queued fallbacks, and typing fallbacks.
