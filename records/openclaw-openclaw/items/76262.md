## Summary

Keep open. ## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework adapter/JWT/streaming plumbing to `@microsoft/teams.apps` SDK routes, auth, streaming, proactive sends, docs, Docker, dependency metadata, and tests.

## Best Possible Solution

Preserve upgrade-safe Teams routing by either keeping stored serviceUrl routing for proactive operations or adding explicit cloud/serviceUrl configuration with fail-closed diagnostics and focused tests before replacing the old adapter path.

## Reproduction Assessment

Yes for the remaining compatibility risk from source: current main uses each stored Teams `serviceUrl`, while the PR calls app-level SDK send/update/delete APIs. I did not have a live GCC/DoD tenant repro, so the cloud-specific failure remains source-proven rather than live-reproduced.

## Solution Assessment

No, not yet. The SDK migration is a maintainable direction, but the safest merge path needs either preserved per-conversation serviceUrl routing or an explicit fail-closed cloud/serviceUrl contract before replacing the current adapter behavior.
