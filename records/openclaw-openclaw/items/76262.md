## Summary

Keep this PR open: the Teams SDK migration has useful fixes and solid live Teams proof, but current head still has a blocking serviceUrl boundary gap in one SDK app-level send path. The new cloud/serviceUrl validator is only wired into the explicit send-context path, while feedback reflection follow-ups now call `app.send(...)` directly and can bypass the fail-closed protection added for non-public or custom Teams Connector endpoints.

## What This Changes

The PR migrates the msteams plugin from Bot Framework adapter/custom JWT/streaming/invoke plumbing to `@microsoft/teams.apps`, adds Teams cloud/serviceUrl config and docs, rewires streaming/invoke/proactive send paths, and updates Docker, tests, and package metadata.

## Best Possible Solution

Keep the SDK migration direction, but require every SDK app-level proactive send path to share the same cloud/serviceUrl boundary check or use a scoped SDK client before merge, with focused tests for the feedback reflection path.

## Reproduction Assessment

Yes for a source-level path, but not with a live non-public Teams tenant in this review. Trigger a negative-feedback reflection follow-up with a stored conversation serviceUrl whose host differs from the configured SDK serviceUrl and the current code reaches `app.send` without the new validator.

## Solution Assessment

No, not yet. The SDK migration and cloud/serviceUrl config direction is maintainable, but the implementation must apply the same fail-closed boundary to every SDK app-level proactive send before merge.
