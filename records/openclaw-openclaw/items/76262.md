## Summary

Keep open: this draft SDK migration still changes MS Teams auth, proactive-send, route, and Docker defaults in ways that regress current main, and the PR body acknowledges some compatibility shims remain unresolved.

## What This Changes

The PR migrates the MS Teams plugin from custom Bot Framework adapter/JWT/proactive/streaming plumbing to `@microsoft/teams.apps` SDK primitives, updates related tests, and tweaks Docker metadata.

## Best Possible Solution

Keep the draft open and reconcile the SDK migration so it preserves current JWT validation, proactive routing metadata, standard route fallback, and Docker/docs port alignment before owner review and Teams validation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to compare the PR head against current main's MS Teams JWT tests, proactive send reference builder, route registration, and Docker/docs port defaults.

## Solution Assessment

No. Full SDK adoption may still be the right direction, but this patch is not mergeable until it preserves the existing compatibility and security invariants or a maintainer explicitly accepts replacing them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the Teams JWT validator:** `extensions/msteams/src/monitor.ts:247-253`
  - body: This delegates webhook validation to the SDK route and removes current main's custom Bot Framework validator. Current main explicitly accepts `https://api.botframework.com` audience tokens with app binding and tenant-scoped issuers, so valid Teams webhooks can start returning 401.
  - confidence: 0.92
- **[P1] Forward proactive send routing metadata:** `extensions/msteams/src/messenger.ts:537-538`
  - body: Calling `params.app.send(conversationId, activity)` rebuilds the proactive send from only a conversation ID and activity, dropping the stored `tenantId`, `aadObjectId`, `user`, and `serviceUrl` metadata that current main threads through `buildConversationReference`. That metadata is required by the #58774 regression tests to avoid Teams HTTP 403s; the same pattern appears in the feedback-reflection proactive path.
  - confidence: 0.94
- **[P2] Keep the standard webhook route fallback:** `extensions/msteams/src/monitor.ts:253`
  - body: Passing only `messagingEndpoint: configuredPath` leaves the SDK with a single configured route. Current main also registers `/api/messages`, so deployments with a custom path that still receive Bot Framework traffic on the standard endpoint would start returning 404.
  - confidence: 0.84
- **[P2] Expose the MS Teams webhook default port:** `docker-compose.yml:46`
  - body: The compose file publishes container port 3000, but the MS Teams plugin and docs still default `channels.msteams.webhook.port` to 3978. A default Docker Compose deployment would expose a port where the webhook is not listening.
  - confidence: 0.87
