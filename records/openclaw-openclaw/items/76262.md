## Summary

Keep open: the Teams SDK migration is still useful, but current head is not merge-ready because its proactive SDK send path drops the tenant and recipient metadata that current main preserves for existing Teams deployments.

## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework plumbing to `@microsoft/teams.apps`, adding SDK cloud/service URL config, docs, dependency changes, Docker updates, and broad Teams tests.

## Best Possible Solution

Land the SDK migration only after the new SDK proactive paths preserve the existing tenant/channelData/recipient contract, keep the cloud/serviceUrl guard, and rebase cleanly onto current main.

## Reproduction Assessment

Yes from source: the PR calls SDK `app.send`/`app.reply`, and the upstream SDK constructs its own outbound reference without OpenClaw's tenant/aadObjectId metadata. I did not live-reproduce the resulting Teams 403 on this branch, but current main and merged PR history show that metadata is required for the existing proactive-send fix.

## Solution Assessment

No, not yet: moving to the SDK is a maintainable direction, but the current implementation bypasses a shipped proactive-send compatibility contract. The safer solution is to keep the SDK path while explicitly preserving tenant/channelData/recipient metadata and testing the actual outbound activity.
