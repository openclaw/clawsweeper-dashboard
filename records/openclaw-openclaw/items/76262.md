## Summary

Keep this PR open: the migration still changes security-sensitive Teams webhook validation and proactive routing contracts without parity code or inspectable live proof.

## What This Changes

The PR migrates the msteams plugin from OpenClaw's local Bot Framework/JWT/REST/streaming adapter code to `@microsoft/teams.apps`, updating streaming, card actions, proactive sends, Docker port exposure, dependency metadata, and tests.

## Best Possible Solution

Keep the migration open until it preserves current webhook JWT/body-limit behavior, fallback routing, stored serviceUrl/tenant/recipient metadata for proactive sends and edits/deletes, narrows the dependency-age exception, and includes redacted live Teams proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of PR head plus `@microsoft/teams.apps@2.0.10` shows deterministic auth/body-parsing and proactive-routing parity losses. I did not run a live Teams setup.

## Solution Assessment

No. SDK adoption may be a good direction, but this implementation is not yet the narrowest maintainable fix because it removes established auth, body-limit, fallback-route, routing metadata, and supply-chain guardrails without parity code or inspectable live proof.

- **[P1] Preserve OpenClaw's Bot Framework JWT contract:** `extensions/msteams/src/monitor.ts:252-255`
  - body: This delegates inbound webhook validation to the SDK route. In `@microsoft/teams.apps@2.0.10`, `ServiceTokenValidator` still uses the cloud token issuer and default client-id audiences, so it does not preserve current main's Bot Framework global audience, tenant-scoped Entra issuer, and app-id binding checks; valid deployed Teams webhooks can regress to 401.

  - body: `sendProactively` now calls `app.send`/`app.reply` with only the normalized conversation id. The SDK builds a fresh reference from its default service URL and does not include the stored serviceUrl, tenant id, Teams user id, or AAD object id that current main forwards, so revoked-context and proactive thread replies can 403 or route through the wrong Connector endpoint.
  - confidence: 0.94
- **[P1] Preserve stored routing metadata in proactive tools:** `extensions/msteams/src/send.ts:440`
  - body: This helper backs proactive poll/card/file sends and now calls `app.send(baseRef.conversation.id, activity)` directly. That bypasses the current adapter path that rehydrates serviceUrl, tenant, recipient id, and AAD object id from the stored conversation reference, so message-tool sends can regress outside the reply fallback path too.
  - confidence: 0.92
- **[P2] Keep full auth before SDK body parsing:** `extensions/msteams/src/monitor.ts:252-255`
  - body: The retained pre-gate only checks for a Bearer-shaped header, then the SDK `ExpressAdapter` installs `express.json()` before `HttpServer.authorize`. This drops current main's full JWT-before-body-parsing behavior and its explicit webhook body limit for Bearer-shaped requests.
  - confidence: 0.91
- **[P2] Register the standard Bot Framework fallback route:** `extensions/msteams/src/monitor.ts:253-255`
  - body: Passing only `messagingEndpoint: configuredPath` makes the SDK register a single POST route. Current main also registers `/api/messages` when a custom path is configured, so existing Azure Bot endpoints still posting to the standard Bot Framework path can start getting 404s.
  - confidence: 0.88
- **[P2] Use the conversation service URL for edits and deletes:** `extensions/msteams/src/send.ts:606`
  - body: The edit/delete paths call the global `app.api` client with only the conversation id. That client is constructed from the SDK app's default service URL, while current main sends edits/deletes through the stored conversation serviceUrl, so regional Teams conversations can update or delete against the wrong Connector endpoint.
  - confidence: 0.87
- **[P2] Narrow the Teams release-age exception:** `pnpm-workspace.yaml:22`
  - body: Adding `@microsoft/teams.*` to `minimumReleaseAgeExclude` bypasses the package-age guard for every future Teams SDK package. Use a version-scoped exception or wait out the age gate unless an owner approves the broader supply-chain policy change.
  - confidence: 0.82
- **[P2] Keep allowlist resolution failures visible:** `extensions/msteams/src/monitor.ts:198`
  - body: This downgrades allowlist resolution failures from `runtime.error` back to `runtime.log`. Current main intentionally logs these at error level because Graph-resolved allowlists may be missing after a resolution failure, leaving the Teams bot in a degraded routing/access-control state.
  - confidence: 0.84
