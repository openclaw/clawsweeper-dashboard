## Summary

Keep open: the SDK migration has useful live proof, but current PR head still has source-reproducible auth and Connector routing regressions, and a member requested-changes review remains active.

## What This Changes

The PR migrates the Microsoft Teams plugin from Bot Framework adapter/auth/streaming plumbing to `@microsoft/teams.apps`, updates Teams invoke/streaming/proactive messaging code, dependencies, Docker defaults, and tests.

## Best Possible Solution

Land the SDK migration only after preserving auth-before-body hardening and per-conversation Connector serviceUrl routing, then get Teams-domain maintainer approval on the rebased diff.

## Reproduction Assessment

Yes. The remaining blockers are source-reproducible from PR head and `@microsoft/teams.apps@2.0.10`: JSON parsing precedes auth, and `App.send`/`reply` use app-level `serviceUrl` while PR calls pass only conversation ids.

## Solution Assessment

No. The migration direction is plausible, but the current patch is not the best fix until it preserves the existing webhook hardening and stored Connector routing contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move JSON parsing back behind auth:** `extensions/msteams/src/monitor.ts:268`
  - body: This installs `express.json()` before the Bearer gate, so unauthenticated JSON requests are parsed before they are rejected. Current main rejects missing Bearer headers and validates JWT before body parsing; keep that ordering while still applying an explicit size limit before the SDK route can parse.
  - confidence: 0.9
- **[P2] Preserve stored serviceUrl for proactive sends:** `extensions/msteams/src/send.ts:440`
  - body: This rebuilds `baseRef` but then calls `app.send` with only the conversation id. In `@microsoft/teams.apps@2.0.10`, `App.send` builds the outgoing reference from `this.api.serviceUrl`, so regional, sovereign, or tenant-specific conversations can be sent through the wrong Connector endpoint instead of the stored `ref.serviceUrl`.
  - confidence: 0.88
- **[P2] Preserve stored serviceUrl in reply fallback:** `extensions/msteams/src/messenger.ts:535-536`
  - body: The revoked-context fallback now routes through `app.reply`/`app.send` with only `baseRef.conversation.id`. Those SDK helpers use the app-level serviceUrl, so channel thread fallbacks can lose the inbound conversation's Connector endpoint even though `baseRef.serviceUrl` is available.
  - confidence: 0.86
- **[P2] Preserve stored serviceUrl for edits and deletes:** `extensions/msteams/src/send.ts:606-646`
  - body: The edit/delete paths call `app.api.conversations.activities(...)`, whose client is rooted at the app-level serviceUrl created in the SDK App constructor. Proactive edits and deletes should use the stored conversation `serviceUrl`, otherwise non-default Connector environments can fail or target the wrong endpoint.
  - confidence: 0.86
