## Summary

Keep open. The SDK migration is useful and has real Teams recordings, but current head still has blocking regressions around stored `serviceUrl` routing, file-consent invoke responses, pre-auth parsing order, and the package freshness bypass; a maintainer requested-changes review also remains active.

## What This Changes

The PR migrates the Microsoft Teams plugin to `@microsoft/teams.apps` 2.0.10, updating streaming, Adaptive Cards, proactive sends/edits/deletes, Docker port defaults, dependencies, tests, and some unrelated formatting churn outside the plugin.

## Best Possible Solution

Land the SDK migration only after it preserves per-conversation Connector routing, SDK invoke response semantics, pre-auth webhook hardening, and exact package freshness policy, followed by a Teams-domain review and the msteams changed gates.

## Reproduction Assessment

Yes for the review blockers: source inspection of PR head plus `@microsoft/teams.apps@2.0.10` shows the changed routing, invoke-response, parsing, and package-policy paths. I did not replay a live Teams tenant locally.

## Solution Assessment

No. The SDK migration may be the right direction, but this PR is not yet the safest merge path because it drops or weakens current-main contracts that are still required for compatibility and operations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move the bearer gate before JSON parsing:** `extensions/msteams/src/monitor.ts:262`
  - body: This installs `express.json({ limit })` before the Bearer-header gate, so even requests with no `Authorization` header are parsed up to the webhook body limit before they are rejected. Current main rejects missing Bearer before body parsing; keep that cheap gate first, then install the limited parser before the SDK route so the SDK parser remains bounded.
  - confidence: 0.86
- **[P2] Preserve stored serviceUrl for proactive replies:** `extensions/msteams/src/messenger.ts:535-536`
  - body: `app.reply()` and `app.send()` build outbound references from the app-level API client service URL, not from `baseRef.serviceUrl`. For regional, sovereign, or tenant-specific Connector URLs, revoked-context fallbacks and proactive thread replies can target the default Teams Connector instead of the endpoint captured from the inbound conversation.
  - confidence: 0.88
- **[P2] Preserve stored serviceUrl for edits and deletes:** `extensions/msteams/src/send.ts:606-607`
  - body: The edit/delete helpers now call the app-level `app.api`, which is rooted at the SDK App's default service URL. These operations should use the stored conversation reference's `serviceUrl`, as current main does, or they can fail or route to the wrong Connector endpoint for non-public-cloud conversations.
  - confidence: 0.86
- **[P2] Return file-consent responses through an SDK invoke route:** `extensions/msteams/src/file-consent-invoke.ts:137`
  - body: Sending `{ type: "invokeResponse" }` through `context.sendActivity()` no longer returns the HTTP invoke response; it is treated as an outbound activity. The file-consent flow should return the SDK invoke response from a `file.consent` handler or equivalent route, otherwise large-file consent callbacks can time out or post invalid Bot Connector traffic.
  - confidence: 0.84
- **[P2] Pin the Teams freshness bypass to the reviewed version:** `pnpm-workspace.yaml:22`
  - body: Adding `@microsoft/teams.apps` exempts every future version of that package from the repository's `minimumReleaseAge` guard. Since this PR needed an early 2.0.10 adoption, scope the bypass to the reviewed version, or remove it once the age window has passed.
  - confidence: 0.82
