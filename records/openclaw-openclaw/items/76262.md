## Summary

Keep open: the SDK migration is useful and not obsolete, but current head still bypasses OpenClaw Teams sender/channel policy on the new SDK SSO signin event and card.action poll vote routes before writing token or poll state. I do not repeat the reported app.reply blocker because the published @microsoft/teams.apps@2.0.10 source and types expose App.reply.

## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework plumbing to @microsoft/teams.apps, updating webhook auth, streaming, invoke routes, proactive send/edit/delete paths, Docker/docs, dependencies, and tests.

## Best Possible Solution

Keep the SDK migration direction, but route every typed Teams SDK event/invoke path through the shared msteams sender/channel policy before sensitive state writes, then rerun focused msteams checks and final maintainer review.

## Reproduction Assessment

Yes for the review defects by source inspection: PR head saves SSO tokens and records card.action poll votes before any resolveMSTeamsSenderAccess gate, while current main gates sign-in invokes and poll votes through the normal handler. I did not run live Teams.

## Solution Assessment

No. The SDK migration direction is maintainable and @microsoft/teams.apps@2.0.10 does expose app.reply, but the typed SDK routes need to share the same sender/channel authorization gates before token persistence and poll mutation.

Reuse the shared ingress access check before recording the vote.
