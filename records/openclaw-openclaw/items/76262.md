## Summary

Keep this PR open: the migration is plausible and includes real Teams proof, but source and SDK inspection show blocking auth, webhook hardening, service URL routing, and supply-chain policy regressions that need maintainer review before merge.

## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework/JWT/REST/streaming plumbing to `@microsoft/teams.apps` 2.0.x, rewires Teams send/edit/delete and Adaptive Card handling, updates Docker port exposure, and changes dependency metadata.

## Best Possible Solution

Keep the SDK migration open and preserve the current Teams auth, webhook hardening, stored service URL, allowlist-error visibility, and dependency policy contracts unless maintainers explicitly approve and document a compatibility break.

## Reproduction Assessment

Yes for the review blockers: source inspection of current main, the PR head, and `@microsoft/teams.apps@2.0.10` provides a high-confidence reproduction path. I did not run a live Teams tenant, so this is source-reproducible rather than live reproduced.

## Solution Assessment

No. The SDK migration may be a good direction, but this patch is not yet the narrowest maintainable solution because it changes established auth, request parsing, routing, and supply-chain contracts without preserving parity or an explicit maintainer decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the Bot Framework token contract:** `extensions/msteams/src/monitor.ts:252-255`
  - body: Delegating inbound validation to the SDK route drops the current custom validator that accepts `https://api.botframework.com` global-audience tokens with app binding and tenant-scoped Entra issuers. Current main has tests for those token shapes, so deployed bots can start returning 401 unless this compatibility break is explicitly accepted and documented.
  - confidence: 0.9
- **[P2] Keep full JWT validation before body parsing:** `extensions/msteams/src/monitor.ts:252-255`
  - body: The retained pre-gate only checks for a Bearer-shaped header, while the SDK `ExpressAdapter` installs `express.json()` before `HttpServer.authorize()`. That removes current main's full JWT-before-body-parse behavior and explicit Teams webhook body limit for attacker-supplied Bearer requests.
  - confidence: 0.92
- **[P2] Use stored service URLs for edits and deletes:** `extensions/msteams/src/send.ts:606`
  - body: Edits and deletes now call the SDK API client with only the conversation ID, but SDK sends use its global `api.serviceUrl` default rather than the stored inbound Teams `serviceUrl`. Regional or tenant-specific Connector conversations can therefore target the wrong endpoint even when the stored conversation reference has the correct service URL.
  - confidence: 0.83
- **[P2] Keep allowlist resolution failures visible:** `extensions/msteams/src/monitor.ts:198`
  - body: This downgrades Graph allowlist resolution failures from `runtime.error` to `runtime.log`. Current main intentionally logs these as errors because unresolved Graph IDs leave the Teams allowlist in a degraded state, so operators may miss access-control misconfiguration.
  - confidence: 0.9
- **[P2] Narrow the Teams SDK release-age bypass:** `pnpm-workspace.yaml:22`
  - body: The new `@microsoft/teams.*` pattern bypasses the release-age guard for every future package in that scope, not just the reviewed SDK version in this migration. Use a narrower exception or wait for the normal release-age window unless maintainers explicitly approve the broader supply-chain policy change.
  - confidence: 0.88
