## Summary

Keep this PR open: the Teams SDK migration is useful and has strong public-cloud proof, but current head still has app-level Teams sends that bypass the new serviceUrl safety boundary for non-public/cloud-mismatch cases.

## What This Changes

The PR migrates the msteams plugin to `@microsoft/teams.apps`, rewires Teams streaming/invoke/proactive delivery, adds Teams cloud/serviceUrl config, and updates docs, tests, Docker, and dependency locks.

## Best Possible Solution

Land the SDK migration only after every app-level Teams proactive send/reply/update/delete path, including feedback reflection and revoked-context fallback delivery, shares the same cloud/serviceUrl boundary guard and focused regression coverage.

## Reproduction Assessment

Yes from source inspection: feedback invokes carry activity.serviceUrl into conversationRef, but sendReflectionFollowUp calls app.send directly, while the upstream SDK app.send uses the app-level serviceUrl. A focused unit test can reproduce a mismatched non-public serviceUrl not failing closed before the send.

## Solution Assessment

No: the SDK migration direction is reasonable, but the current implementation only applies the new serviceUrl guard to send-context proactive resolution. The maintainable fix is to centralize or reuse that guard for every app-level Teams send fallback before merge.
