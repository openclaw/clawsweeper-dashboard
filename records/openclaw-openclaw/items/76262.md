## Summary

Keep open. This SDK migration is not merge-ready because it drops current MS Teams auth and proactive-send invariants, adds a Docker port mapping that does not match the plugin default, and the PR body already calls out failing MS Teams tests that still need owner attention.

## What This Changes

The PR rebases a large MS Teams SDK migration onto current main, replacing the local adapter/streaming paths with @microsoft/teams.apps App/ExpressAdapter APIs while updating send, monitor, poll, feedback, tests, and Docker surfaces.

## Best Possible Solution

Keep the PR open for MS Teams owner review and preserve current auth, proactive routing metadata, fallback route, port, and regression-test invariants before considering merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send a Bot Framework webhook token with aud=https://api.botframework.com or perform a proactive send from a stored ref containing tenantId/aadObjectId; current main has comments/tests for both, and this PR removes the corresponding paths.

## Solution Assessment

No. Migrating to the Teams SDK App APIs may be the right direction, but this patch is not the safest implementation because it replaces proven auth and proactive-send behavior without preserving the current compatibility fixes or passing the targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the Teams JWT validator:** `extensions/msteams/src/monitor.ts:247-253`
  - body: This switches inbound webhook auth to the SDK's ExpressAdapter path and removes the custom Bot Framework JWT validator. The 2.0.9 SDK validator's default audience list does not include `https://api.botframework.com`, while current main has explicit coverage for accepting those global-audience tokens with app binding, so valid Teams webhooks can start returning 401.
  - confidence: 0.91
- **[P1] Forward proactive send routing metadata:** `extensions/msteams/src/send.ts:371-372`
  - body: Calling `app.send(baseRef.conversation.id, activity)` drops the `tenantId`, `aadObjectId`, user, and serviceUrl metadata that `buildConversationReference` just recovered. Current main added that metadata because Bot Framework proactive sends fail with HTTP 403 without it, so proactive messages, polls, cards, edits, and deletes can regress.
  - confidence: 0.93
- **[P2] Expose the port MS Teams actually listens on:** `docker-compose.yml:43`
  - body: The compose file now publishes container port 3000 for MS Teams, but the plugin still defaults `channels.msteams.webhook.port` to 3978 and the docs/config examples use 3978. A default Docker Compose deployment would expose a port where the Teams webhook is not listening.
  - confidence: 0.86
- **[P2] Keep the standard /api/messages fallback:** `extensions/msteams/src/monitor.ts:253`
  - body: Passing only `messagingEndpoint: configuredPath` leaves route registration to a single SDK endpoint. Current main registers both the configured path and `/api/messages`, so deployments that configured a custom path but still receive Bot Framework traffic on the standard endpoint would start returning 404.
  - confidence: 0.82
