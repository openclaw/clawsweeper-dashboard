## Label Justifications

- `P1`: The PR is a large Teams channel migration with a remaining upgrade blocker that can break real non-public Teams deployments after merge.
- `merge-risk: 🚨 compatibility`: Merging can change existing Teams deployments from per-conversation stored serviceUrl routing to SDK app-level routing without an upgrade guard.
- `merge-risk: 🚨 message-delivery`: The unsupported cloud mismatch can drop or misroute proactive Teams sends, edits, deletes, polls, file-consent cards, and queued replies.
- `merge-risk: 🚨 auth-provider`: The migration moves Teams auth, token, SSO, and Connector endpoint selection onto SDK app configuration, so cloud-specific auth and routing need explicit configuration before merge.

## Summary

Keep open: the branch has strong public-cloud Teams proof and the cleanup/CI work looks good, but a P1 upgrade blocker remains because proactive Teams operations now use the SDK app-level service URL without a configured cloud/serviceUrl guard, while current main routes through each stored conversation serviceUrl.

## What This Changes

Migrates the msteams plugin from custom Bot Framework adapter, JWT, streaming, and invoke plumbing to `@microsoft/teams.apps` SDK APIs, with related docs, Docker, dependency, and test updates.

## Best Possible Solution

Add explicit `channels.msteams` cloud/serviceUrl configuration, pass it into the Teams SDK App, keep public cloud as the default, and validate stored conversation serviceUrls before proactive operations with focused upgrade tests.

## Reproduction Assessment

Yes from source for the remaining blocker: current main sends through each stored Teams `serviceUrl`, while the PR uses SDK app-level send/update/delete APIs. I did not live-reproduce a GCC/GCC High/DoD tenant failure, so the non-public cloud failure remains source-proven rather than live-proven.

## Solution Assessment

No, not merge-ready yet. The SDK migration direction is maintainable, but replacing per-conversation serviceUrl routing without an explicit cloud/serviceUrl config and fail-closed guard is not the safest upgrade path.
