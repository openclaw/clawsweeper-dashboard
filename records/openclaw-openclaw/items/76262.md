## Summary

Keep open: the Teams SDK migration has useful live proof and green CI, but the current diff still has source-proven blockers around per-conversation Connector serviceUrl routing and the dependency freshness exception.

## What This Changes

The PR migrates the Microsoft Teams plugin from legacy Bot Framework adapter plumbing to `@microsoft/teams.apps`, rewiring auth, streaming, invoke handling, proactive sends/edits/deletes, tests, Docker defaults, and package metadata.

## Best Possible Solution

Land the SDK migration only after proactive send/edit/delete/thread fallback preserve the stored serviceUrl or a documented explicit cloud/serviceUrl replacement is accepted, and scope the release-age exception to reviewed versions.

## Reproduction Assessment

Yes. Source inspection of the PR head and `@microsoft/teams.apps@2.0.10` shows the stored serviceUrl is not carried into proactive `app.send`/`app.reply`/`app.api` calls, and pnpm docs show the freshness exclude applies to every package version.

## Solution Assessment

No. The SDK migration direction is plausible and the live proof is useful, but this diff removes an existing per-conversation routing contract and broadens the dependency freshness bypass before that compatibility decision is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the stored serviceUrl for proactive sends:** `extensions/msteams/src/send.ts:440`
  - body: `app.send()` builds its Bot Framework reference from the SDK app's configured `serviceUrl`, not the stored `conversationReference.serviceUrl` captured from the inbound activity. Current main requires that stored serviceUrl for proactive sends, so this can silently route sends through the public/default Connector endpoint instead of a sovereign, regional, or tenant-specific endpoint; the same issue carries into the edit/delete and proactive thread fallback paths using `app.api`/`app.reply`.
  - confidence: 0.9
- **[P3] Scope the Teams SDK freshness bypass:** `pnpm-workspace.yaml:22`
  - body: Adding bare `@microsoft/teams.apps` to `minimumReleaseAgeExclude` exempts every future release of that package, not only the reviewed `2.0.10` bump. Since pnpm supports version-qualified excludes, keep the release-age bypass pinned to the exact version needed here or remove it once the package is old enough.
  - confidence: 0.92
