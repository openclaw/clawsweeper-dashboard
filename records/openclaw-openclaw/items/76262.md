## Summary

Keep open: the Teams SDK migration has useful live proof, but maintainer-requested blockers remain around webhook hardening, serviceUrl routing, invoke responses, streaming fallback, allowlist diagnostics, and dependency freshness policy.

## What This Changes

Migrates the Microsoft Teams plugin from custom Bot Framework/JWT/REST/streaming plumbing to `@microsoft/teams.apps`, updating Teams sends, streaming, Adaptive Cards, Docker, dependencies, and tests.

## Best Possible Solution

Land the SDK migration only after it preserves the current webhook hardening, stored serviceUrl routing, invoke response semantics, stream failure fallback, allowlist diagnostics, and package freshness policy.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head, current main, and `@microsoft/teams.apps@2.0.10` shows the changed auth/body-parse ordering, default serviceUrl routing, invoke-response path, stream error handling, and release-age bypass. I did not replay a live Teams tenant locally.

## Solution Assessment

No. The SDK migration may be the right long-term direction, but this PR is not yet the narrowest maintainable merge path because it changes security-sensitive and compatibility contracts that current main deliberately preserves.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve pre-auth JWT validation and body limits:** `extensions/msteams/src/monitor.ts:263-268`
  - body: This delegates the Teams route to the SDK after only checking for a Bearer-shaped header. The SDK's ExpressAdapter installs `express.json()` on the route before HttpServer authorizes the request, so invalid Bearer requests are parsed before full JWT validation and without the current `MSTEAMS_WEBHOOK_MAX_BODY_BYTES` handling.
  - confidence: 0.9
- **[P2] Route proactive sends with the stored serviceUrl:** `extensions/msteams/src/send.ts:440`
  - body: `app.send()` rebuilds the Bot Framework reference from the app-level API client service URL, not from the stored conversation reference. That can route proactive sends to `https://smba.trafficmanager.net/teams` even when the inbound conversation came from a regional or tenant-specific serviceUrl that current main preserves.
  - confidence: 0.86
- **[P2] Route edits and deletes with the stored serviceUrl:** `extensions/msteams/src/send.ts:606-607`
  - body: The SDK API client used here is rooted at the app default service URL, so proactive update/delete calls can target the wrong Connector endpoint for conversations whose stored reference carries a different serviceUrl. Keep the per-conversation serviceUrl when constructing these API calls.
  - confidence: 0.84
- **[P2] Return invoke responses through SDK invoke handlers:** `extensions/msteams/src/file-consent-invoke.ts:137`
  - body: Sending `{ type: "invokeResponse" }` through the adapted turn context is no longer the same as returning the HTTP invoke response from the SDK handler. File-consent and signin invokes can still time out or send a bogus outbound activity instead of acknowledging Teams within the invoke request.
  - confidence: 0.8
- **[P2] Fallback when streaming fails after partial output:** `extensions/msteams/src/reply-stream-controller.ts:168-174`
  - body: If `stream.emit()` or `stream.close()` fails for a non-cancel reason after tokens have been emitted, the error is rethrown while `preparePayload()` suppresses normal block delivery. That can leave users with a truncated stream and no final message; preserve a failure latch that lets the final reply fall back to block delivery.
  - confidence: 0.78
- **[P2] Narrow the Teams release-age bypass:** `pnpm-workspace.yaml:22`
  - body: The wildcard exemption allows every future `@microsoft/teams.*` package to bypass the repository's minimum release-age guard. Limit this to the exact SDK packages or otherwise keep the freshness policy intact for unrelated packages under the scope.
  - confidence: 0.88
- **[P2] Keep allowlist resolution failures visible:** `extensions/msteams/src/monitor.ts:212`
  - body: Current main logs Teams allowlist resolution failures at error level because unresolved Graph IDs degrade access control. Downgrading this to `runtime.log` makes a security-relevant startup failure easy to miss in normal operations.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR changes security-sensitive webhook parsing/auth ordering and package freshness policy, and it lowers visibility for allowlist resolution failures.
