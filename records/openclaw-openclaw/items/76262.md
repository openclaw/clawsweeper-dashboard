## Summary

Keep this PR open. The SDK migration is useful and has live proof, but the current patch still introduces source-reproducible regressions in webhook auth ordering and stored Connector serviceUrl routing, and an active member review still requests changes on the security-sensitive Teams path.

## What This Changes

The branch migrates the Microsoft Teams plugin to `@microsoft/teams.apps` 2.0.10, replacing custom Bot Framework auth/adapter/streaming/invoke/proactive messaging code and updating Docker, package metadata, and tests.

## Best Possible Solution

Land the SDK migration only after preserving auth-before-body hardening and per-conversation Connector serviceUrl routing, then get Teams-domain maintainer approval on the narrowed diff.

## Reproduction Assessment

Yes. The blockers are source-reproducible from the PR head and `@microsoft/teams.apps@2.0.10`: JSON parsing is installed before auth, and `app.send`/`app.reply`/`app.api` use the app-level serviceUrl instead of the stored conversation serviceUrl.

## Solution Assessment

No. The SDK migration direction is plausible, but the best fix must keep a limited parser behind the Bearer gate and use a serviceUrl-scoped Connector client for proactive sends, edits, and deletes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move JSON parsing back behind the Bearer gate:** `extensions/msteams/src/monitor.ts:268`
  - body: This installs `express.json({ limit })` before the Bearer-token gate, so unauthenticated webhook traffic is parsed before rejection. Current main rejects missing Bearer headers before body parsing; keep the limit, but put the parser after the cheap auth gate and before the SDK route.
  - confidence: 0.9
- **[P2] Preserve the stored serviceUrl for proactive sends:** `extensions/msteams/src/send.ts:440`
  - body: This rebuilds `baseRef` but then calls `app.send` with only the conversation id. The SDK builds its Connector reference from the app-level serviceUrl, so regional or sovereign conversations can be routed through the wrong endpoint instead of the stored conversation `serviceUrl`.
  - confidence: 0.88
- **[P2] Preserve the stored serviceUrl for edits and deletes:** `extensions/msteams/src/send.ts:606`
  - body: The edit/delete paths also rebuild `baseRef` but call the app-level API client. These operations need to target the stored conversation `serviceUrl`, matching current main’s REST client behavior for proactive Connector calls.
  - confidence: 0.88
- **[P2] Preserve the stored serviceUrl for proactive thread replies:** `extensions/msteams/src/messenger.ts:535-536`
  - body: The threaded fallback uses `app.reply(baseRef.conversation.id, ...)`, which internally calls `app.send` and uses the app-level serviceUrl. Channel thread replies should stay scoped to the stored conversation `serviceUrl` instead of relying on the default Connector endpoint.
  - confidence: 0.86
