## Summary

Keep open: the current head still replaces OpenClaw's Microsoft Teams auth and routing contracts with SDK defaults that drop accepted JWT issuer/audience coverage and stored serviceUrl/tenant/recipient metadata, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR migrates the Microsoft Teams plugin from local Bot Framework/JWT/REST/streaming adapter code to `@microsoft/teams.apps`, updating streaming, polls, proactive sends, edit/delete paths, Docker port exposure, dependencies, and tests.

## Best Possible Solution

Keep the PR open until the migration preserves current Teams webhook auth, body-size/auth ordering, fallback endpoint, serviceUrl/tenant/recipient routing, edit/delete routing, dependency-age policy, changelog expectations, and redacted live Teams proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of PR head plus `@microsoft/teams.apps@2.0.10` shows deterministic paths for JWT compatibility loss, auth-after-body parsing, and proactive routing metadata loss. I did not run a live Teams setup.

## Solution Assessment

No. SDK adoption may be a reasonable direction, but this implementation is not yet the safest fix because it changes established auth, routing, edit/delete, fallback-route, and supply-chain contracts without parity code and inspectable live proof.

  - confidence: 0.92

  - body: The retained pre-gate only checks for a Bearer-shaped header, then the SDK `ExpressAdapter` installs `express.json()` before `HttpServer.authorize`. This drops current main's full JWT-before-body-parsing behavior and the explicit 1 MB webhook body limit for Bearer-shaped requests.
  - confidence: 0.92

  - confidence: 0.88

  - confidence: 0.87
