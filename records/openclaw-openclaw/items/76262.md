## Summary

Keep this PR open: current main does not already contain the Teams SDK migration, and the active PR still has two source-backed authorization regressions in the new typed SDK routes. I did not carry forward the latest `app.reply` blocker because the published `@microsoft/teams.apps@2.0.10` tarball exposes `App.reply(...)` in both declarations and runtime JS.

## What This Changes

The PR migrates the Microsoft Teams plugin from custom Bot Framework plumbing to `@microsoft/teams.apps`, updating webhook auth/dispatch, streaming, proactive sends/edits/deletes, SSO, file consent, poll actions, Docker/docs, tests, and plugin dependencies.

## Best Possible Solution

Keep the SDK migration open, add the missing sender/channel authorization checks to the SSO token persistence and poll vote routes, then re-run focused msteams tests plus refreshed live Teams proof before maintainer merge review.

## Reproduction Assessment

Not applicable as a standalone issue, but yes for the PR review blockers: the SSO and poll authorization gaps are source-reproducible by comparing the PR head typed routes with the existing guarded message/sign-in paths on current main.

## Solution Assessment

No. The SDK migration direction is plausible, and the `app.reply` concern does not reproduce against the published SDK source, but the branch is not mergeable until the new typed routes preserve the established Teams sender/channel authorization policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Authorize Teams SSO events before saving tokens:** `extensions/msteams/src/monitor.ts:447-490`
  - body: The new SDK `signin` event path saves delegated tokens directly, while the old sign-in invoke path first ran `isSigninInvokeAuthorized(...)` to enforce the Teams DM, group sender, and team/channel gates. Persisting SSO state for a Teams user OpenClaw would otherwise reject is an authorization regression; reuse that gate before `tokenStore.save(...)`.
  - confidence: 0.88
- **[P2] Authorize card actions before recording poll votes:** `extensions/msteams/src/monitor.ts:365-384`
  - body: `card.action` now records poll votes directly and the catch-all skips `adaptiveCard/action`, so poll votes bypass the normal message path's sender/channel allowlist checks before mutating poll state. Run the same authorization gate, and ideally confirm the invoke conversation matches the stored poll conversation, before `recordVote(...)`.
  - confidence: 0.9
