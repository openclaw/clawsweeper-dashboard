## Summary

Keep open: current main has some process-tree protections, but the launcher/compile-cache respawn path still uses blocking child respawns without the existing signal bridge, so the reported orphan pattern is still source-reproducible.

## What This Changes

The issue asks for `openclaw security audit --deep` and the background `process kill` path to terminate any spawned worker descendants when the launcher receives SIGTERM or is killed.

## Best Possible Solution

Centralize launcher respawn child management so all OpenClaw CLI respawns forward shutdown signals or terminate the child process group, then add regression coverage for SIGTERM and background-session cancellation.

## Reproduction Assessment

Yes, at source level: the launcher respawn paths use `spawnSync` without the existing child-process bridge, which can reproduce the SIGTERM-to-wrapper orphan pattern. I did not run a live macOS audit process in this read-only pass.

## Solution Assessment

No, current main is not yet the best solution because only the normal CLI respawn path uses `attachChildProcessBridge`. The maintainable fix is a shared respawn helper or equivalent tree/signal handling for both `openclaw.mjs` and the TS compile-cache respawn path.
