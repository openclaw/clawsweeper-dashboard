## Summary

Keep open: current main still has unbridged synchronous launcher and compile-cache respawn paths, while the documented process contract and the normal CLI respawn path already require child signal bridging, so the reported orphan pattern remains source-reproducible.

## What This Changes

The issue asks for `openclaw security audit --deep` and background-session termination to cleanly stop any worker or respawned child instead of leaving an orphaned high-CPU process.

## Best Possible Solution

Centralize launcher respawn child management so every OpenClaw CLI respawn forwards shutdown signals or terminates the child process group safely, then add regression coverage for SIGTERM/background-session cancellation.

## Reproduction Assessment

Yes, at source level: the launcher and compile-cache respawn paths still use blocking `spawnSync` without the existing child-process bridge, which can leave the respawned child alive when the wrapper receives SIGTERM. I did not run the live macOS audit workload in this read-only review.

## Solution Assessment

No, current main is not yet the best solution because only the normal CLI respawn path uses `attachChildProcessBridge`. The narrow maintainable fix is a shared respawn helper or equivalent signal/tree handling for `openclaw.mjs` and `src/entry.compile-cache.ts`, without special-casing security audit.
