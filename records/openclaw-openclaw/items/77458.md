## Summary

Keep open: current main still has unbridged launcher and compile-cache respawn paths that can leave the reported security-audit child alive when the wrapper is terminated, even though the normal CLI respawn path already has a signal bridge.

## What This Changes

The issue asks for `openclaw security audit --deep` shutdown to terminate its worker child when the launcher or background session is killed.

## Best Possible Solution

Centralize launcher respawn child management so all OpenClaw CLI respawns forward shutdown signals or terminate the intended child process group, with regression coverage for wrapper SIGTERM/background-session cancellation.

## Reproduction Assessment

Yes, at source level: the root launcher and TS compile-cache respawn helpers still use blocking `spawnSync` without the existing child-process bridge, matching the wrapper-killed-child-survives pattern. I did not perform a live macOS audit run in this read-only pass.

## Solution Assessment

No, current main is not yet the best solution because only the normal CLI respawn path bridges shutdown signals. The narrow maintainable fix is to share or replicate bridged respawn handling for the compile-cache launcher paths while preserving service-managed process-group safeguards.
