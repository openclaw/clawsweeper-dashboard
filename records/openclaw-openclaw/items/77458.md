## Summary

Keep open: current main still has unbridged launcher and compile-cache respawn paths, so the reported SIGTERM-to-wrapper orphan pattern remains source-reproducible despite existing process-tree protections elsewhere.

## What This Changes

The issue asks OpenClaw to ensure `openclaw security audit --deep` cleans up its child worker when the launcher or background session is killed.

## Best Possible Solution

Centralize CLI launcher respawns so the root launcher and TypeScript compile-cache path forward termination signals or terminate the child process group, with regression coverage for SIGTERM and background-session cancellation.

## Reproduction Assessment

Yes, at source level: current main still has unbridged `spawnSync` launcher/compile-cache respawn paths while the local docs require child-process bridging for CLI respawns. I did not run the live macOS audit session in this read-only review.

## Solution Assessment

No, current main is not yet the best solution because only the normal CLI respawn path uses `attachChildProcessBridge`. The narrow maintainable fix is a shared respawn helper or equivalent signal/process-group handling for both launcher respawn paths.
