## Summary

Keep open: current main and the latest release still prune context without emitting a pruning lifecycle hook, and the linked implementation PR remains open, conflicting, and blocked on the hook contract.

## What This Changes

The issue requests a context-pruning lifecycle hook that fires after pruning and exposes safe metadata for external indexing or memory workflows.

## Best Possible Solution

Keep this as the canonical request and resolve it through the linked PR or a replacement that emits bounded, sanitized pruning metadata after successful pruning, with tests and public docs.

## Reproduction Assessment

Yes, at source level. Current main prunes through the context extension and pruner without emitting a pruning hook or returning hook metadata, and the hook catalogs and docs omit a pruning event.

## Solution Assessment

Unclear until maintainers choose the hook contract. A metadata-only `session:prune` lifecycle event looks like the narrowest maintainable direction, while raw pruned-content indexing needs explicit privacy, trust, timeout, and size-limit decisions.
