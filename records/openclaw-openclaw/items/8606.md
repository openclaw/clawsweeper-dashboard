## Summary

Keep open: current main and the latest release still prune context without emitting a pruning lifecycle hook, and the live linked pull request that explicitly closes this issue remains open, conflicting, and blocked on the hook contract.

## What This Changes

The issue requests a new context-pruning lifecycle hook that fires after pruning and exposes safe metadata so external memory/indexing handlers can react without polling.

## Best Possible Solution

Keep this as the canonical request and resolve it through the linked PR or a replacement that emits bounded, sanitized pruning metadata only after successful pruning, with focused tests and public hook documentation.

## Reproduction Assessment

Yes. At source level, current main prunes through the context extension and pruner without emitting a pruning hook or returning hook metadata; the hook catalogs and docs also omit such an event.

## Solution Assessment

Unclear until maintainers choose the hook contract. A metadata-only `session:prune` lifecycle event looks like the narrowest maintainable direction, while raw pruned-content indexing needs explicit privacy, trust, timeout, and size-limit decisions.
