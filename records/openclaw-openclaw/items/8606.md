## Summary

Keep open: current main and the latest release still have context pruning as a message-rewrite path with no `session:prune` or `context:prune` hook, and the open linked implementation PR is unmerged and still needs maintainer product/security review.

## What This Changes

The issue requests a lifecycle hook after context pruning, with metadata about what was trimmed so external memory or indexing can run without polling.

## Best Possible Solution

Keep this issue open and land a maintainer-approved pruning lifecycle event on the active `agent-hooks/context-pruning` path, with bounded hook execution, privacy-safe metadata, focused tests, and public docs for the final contract.

## Reproduction Assessment

Yes, source-level. Current main’s active pruning extension and pruner rewrite messages only, while hook catalogs and docs have no pruning lifecycle event or payload metadata.

## Solution Assessment

Yes in direction, but not as an automatic repair. The maintainable solution is a narrow, reviewed hook contract on the active pruning path; the stale linked PR should be rebased or replaced rather than using the old path as-is.
