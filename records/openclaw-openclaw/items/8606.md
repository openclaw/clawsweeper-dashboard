## Summary

Keep open: current main still lacks a context-pruning lifecycle hook, and https://github.com/openclaw/openclaw/pull/12581 explicitly closes this issue but remains open, conflicting, and unmerged. This is a new hook/API contract with privacy-sensitive payload choices, so it is not a conservative cleanup-close candidate.

## What This Changes

Request a context-pruning lifecycle hook that fires after pruning with metadata about what was trimmed so handlers can index or react to pruned content.

## Best Possible Solution

Add a narrow, summary-only `session:prune`-style event in `src/agents/pi-hooks/context-pruning/*` after successful pruning, with focused tests and public hook documentation; use, rebase, replace, or close https://github.com/openclaw/openclaw/pull/12581 as the implementation path.

## Reproduction Assessment

Yes, at source level: current main prunes and returns rewritten messages, while source, hook catalogs, tests, and docs expose no `session:prune` or `context:prune` event. This is a feature gap rather than broken existing behavior.

## Solution Assessment

Unclear until maintainers settle the hook contract and privacy boundary. A narrow summary-only pruning lifecycle event appears maintainable, but the open implementation candidate needs current-main review, rebase, or replacement.

## Security Review

Status: needs_attention

Summary: A pruning lifecycle hook can persist transcript or tool-output details unless its payload is intentionally summary-only and treats metadata strings as untrusted.

Concerns:

- **[medium] Keep pruning hook payloads summary-only:** `src/agents/pi-hooks/context-pruning/pruner.ts:287`
  - body: A hook that forwards raw pruned tool output or unsanitized tool metadata could persist secrets, local paths, private endpoints, or log-injection content into user hooks, indexes, or telemetry. The contract should default to safe summary metadata and require explicit maintainer approval for raw-content access.
  - confidence: 0.88
