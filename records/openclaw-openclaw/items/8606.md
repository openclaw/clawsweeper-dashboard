## Summary

Current main and the latest release still lack a context-pruning lifecycle hook, while the directly linked implementation PR remains open and conflicted; keep the issue open for maintainer-led contract design and a current-main implementation.

## What This Changes

Expose a lifecycle hook after context pruning so handlers can react to safely described trimmed content without polling.

## Best Possible Solution

Define a typed, best-effort plugin lifecycle event at the current `src/agents/agent-hooks/context-pruning/` boundary, default to privacy-safe summary metadata, document its timing and failure semantics, and update or replace the linked PR against current main.

## Reproduction Assessment

Yes, at source level: current main's context-pruning extension returns rewritten messages without emitting a lifecycle event, and the typed hook catalog has no pruning hook. This is a missing capability rather than a failing existing contract.

## Solution Assessment

Yes in architectural direction, but not yet in final contract: a reusable plugin hook is preferable to core indexing or polling, while maintainers must decide privacy, payload, timing, and observer-failure semantics.
