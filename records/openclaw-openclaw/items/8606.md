## Summary

Current main and the latest release still prune tool-result context without emitting a pruning lifecycle event, while the open linked implementation PR remains the canonical work path and requires maintainer resolution of its conflicts, proof, and sensitive payload contract.

## What This Changes

Requests a lifecycle hook that reports what context pruning removed so external indexing can react without scheduled polling.

## Best Possible Solution

Refresh or replace the linked PR with a post-success pruning event that carries explicit session attribution and minimal safe metadata by default; expose removed content only through a separately approved, documented, and tested sensitive-data contract.

## Reproduction Assessment

Yes, at source level. Current main has a concrete pruning path, and the handler, pruner return type, hook catalogs, tests, and docs consistently show no lifecycle event or metadata payload for pruning consumers.

## Solution Assessment

The proposed lifecycle point is appropriate, but sending raw pruned content is not the best default. A metadata-only event is the narrowest maintainable base contract unless maintainers explicitly approve privileged content access for plugins.
