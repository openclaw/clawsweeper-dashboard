## Summary

Keep open: current main and the latest release still lack a context-pruning lifecycle hook, and the linked implementation path remains open, conflicting, and in need of maintainer/security contract review.

## What This Changes

Request adding a context-pruning lifecycle hook that fires after pruning with metadata so external indexing can react without scheduled polling.

## Best Possible Solution

Keep this issue open and resolve it through the linked PR or a maintainer-approved replacement that emits bounded, sanitized pruning metadata only after successful pruning and documents the event contract.

## Reproduction Assessment

Yes, at source level: current main prunes context through the session extension and pruner without emitting a pruning hook or returning hook metadata. I did not run a live agent scenario because the missing hook surface is directly visible in source and docs.

## Solution Assessment

Unclear until maintainers choose the hook contract. A metadata-only `session:prune` event is the likely narrow direction, while raw pruned-content indexing needs explicit privacy, trust, timeout, and sanitization decisions.
