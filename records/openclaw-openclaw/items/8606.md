## Summary

Keep open. Current main still lacks a context-pruning lifecycle hook or pruning metadata for hook consumers, and open PR #12581 explicitly targets this issue, so the issue should remain open until that PR or a maintainer-approved replacement lands.

## What This Changes

The issue requests a lifecycle hook such as `session:prune` or `context:prune` that fires after context pruning and provides metadata about trimmed content for indexing or search handlers.

## Best Possible Solution

Add a narrow, documented pruning lifecycle event from the context-pruning owner path after successful pruning. It should expose safe summary metadata by default, avoid leaking full pruned tool output unless explicitly designed, include focused pruning and hook tests, and document the supported event contract.

## Reproduction Assessment

Yes. On current main, inspect the pruning extension/pruner and hook catalogs: pruning rewrites and returns messages, while targeted searches across source and docs find no `session:prune` or `context:prune` event.

## Solution Assessment

Unclear until the open implementation PR is reviewed. A `session:prune` lifecycle event with safe metadata is the narrow maintainable direction, but maintainers still need to decide whether it is internal-only, plugin-exposed, and what payload is safe.
