## Summary

Keep open: current main still lacks a context-pruning lifecycle hook, and an open implementation PR with closing syntax already targets the request. The remaining work is a new hook contract with privacy-sensitive payload choices, so this is not a cleanup-close candidate.

## What This Changes

The issue requests a lifecycle hook that fires after context pruning and exposes safe metadata about pruned content for downstream indexing.

## Best Possible Solution

Add a narrow, summary-only pruning lifecycle event after successful pruning in src/agents/pi-hooks/context-pruning/*, with focused tests and public hook documentation; review, rebase, replace, or close https://github.com/openclaw/openclaw/pull/12581 as the implementation path.

## Reproduction Assessment

Yes, at source level: current main prunes and returns rewritten messages, while source, hook catalogs, tests, and docs expose no session:prune or context:prune event. This is a feature gap rather than broken existing behavior.

## Solution Assessment

Unclear until maintainers settle the hook contract and payload privacy boundary. A narrow session:prune-style event with safe summary metadata appears maintainable, but the open implementation candidate needs current-main review, rebase, or replacement.

## Security Review

Status: needs_attention

Summary: A pruning hook can persist transcript or tool-output details unless the payload is intentionally summary-only and treats metadata strings as untrusted.

Concerns:

- **[medium] Keep pruned-content payloads summary-only:** `src/agents/pi-hooks/context-pruning/pruner.ts:287`
  - body: A pruning hook that forwards raw pruned tool output or unsanitized tool metadata could persist secrets, local paths, private endpoints, or log-injection content into user hooks, indexes, or telemetry. The contract should default to safe summary metadata and require explicit maintainer approval for raw-content access.
  - confidence: 0.88
