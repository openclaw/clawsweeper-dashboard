## Summary

Keep open: current main and the latest release still prune by returning rewritten messages without a pruning lifecycle hook or metadata payload, and a linked open PR explicitly targets this issue but remains unmerged and conflicting.

## What This Changes

Request to add a context-pruning lifecycle hook that reports pruning activity so external indexing or memory workflows can react without polling.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/12581 with a maintainer-approved session:prune-style event that emits bounded summary metadata after successful pruning, documents the internal/plugin visibility boundary, and covers pruning plus hook-delivery tests.

## Reproduction Assessment

Yes, at source level. Current main rewrites and returns pruned messages without hook emission or pruning metadata, and the hook catalog/docs still omit a pruning lifecycle event.

## Solution Assessment

Unclear until maintainers choose the event shape and visibility. The narrow maintainable path is a safe summary-metadata hook in the existing context-pruning path, with docs stating whether it is internal-only or plugin-visible.

## Security Review

Status: needs_attention

Summary: The requested lifecycle hook is security-sensitive because pruning metadata or content can expose private tool output to hooks, logs, or external indexes.

Concerns:

- **[medium] Define pruning payload privacy boundary:** `src/agents/pi-hooks/context-pruning/pruner.ts:287`
  - body: A pruning hook should default to bounded summary metadata; forwarding raw pruned content or unsanitized tool names would expand the sensitive transcript and tool-output surface and needs explicit opt-in, documentation, and retention guidance.
  - confidence: 0.88

  - body: The current internal hook runner awaits registered handlers sequentially, so a future pruning hook should be bounded or explicitly non-blocking before it sits in the context-building path.
  - confidence: 0.82
