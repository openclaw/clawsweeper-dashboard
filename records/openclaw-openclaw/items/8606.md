## Summary

Keep open: current main still lacks a context-pruning lifecycle hook, and the live closing PR remains open, conflicting, and unmerged. The request is a new hook/API contract with privacy-sensitive payload choices, so it is not a conservative cleanup-close candidate.

## What This Changes

The issue requests a lifecycle hook fired after context pruning so external handlers can index or otherwise react to pruned content.

## Best Possible Solution

Add a narrow, summary-only `session:prune`-style event in the current `src/agents/pi-hooks/context-pruning/*` path, with focused tests and public hook documentation; use, rebase, replace, or close the existing closing PR as the implementation path.

## Reproduction Assessment

Yes, at source level: current main prunes and returns rewritten messages, while source, hook catalogs, tests, and docs expose no `session:prune` or `context:prune` event. This is a feature gap rather than broken existing behavior.

## Solution Assessment

Unclear until maintainers settle the hook contract and privacy boundary. A narrow summary-only pruning lifecycle event appears maintainable, but the open implementation candidate needs current-main review, rebase, or replacement.
