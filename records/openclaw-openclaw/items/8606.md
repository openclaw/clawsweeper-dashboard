## Summary

Keep open: current main and v2026.6.10 still prune context without emitting a pruning lifecycle hook, and the linked implementation PR is open, conflicting, and blocked on security/product contract review.

## What This Changes

Request to emit a context/session prune hook with metadata when context pruning trims old tool outputs so downstream handlers can preserve searchable access.

## Best Possible Solution

Keep this as the canonical request and resolve it through the linked PR or a replacement that emits bounded, sanitized pruning metadata only after successful pruning, with focused tests and public hook docs.

## Reproduction Assessment

Yes. At source level, current main prunes through the context extension and pruner without emitting a pruning hook or returning hook metadata; the hook catalogs and docs also omit such an event.

## Solution Assessment

Unclear until maintainers choose the hook contract. A metadata-only `session:prune` lifecycle event looks like the narrowest maintainable direction, while raw pruned-content indexing needs explicit privacy, trust, timeout, and size-limit decisions.
