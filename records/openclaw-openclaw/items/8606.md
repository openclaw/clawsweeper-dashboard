## Label Justifications

- `P2`: The missing event is a normal-priority extension-point improvement with a bounded affected surface and an existing polling workaround.
- `impact:session-state`: The requested hook reports when and how active session context is rewritten by pruning.
- `impact:security`: A pruning payload can expose sensitive tool-result content unless the public contract is deliberately constrained and reviewed.

## Summary

Current main still lacks a pruning lifecycle event, and the open maintainer-authored implementation PR remains the canonical but conflicting path. The issue should stay open until maintainers refresh and approve that PR or replace it, with an explicit decision on whether the hook exposes safe summary metadata or pruned content.

## What This Changes

Request a lifecycle hook that reports what tool-output context pruning removed so plugins can trigger external indexing without polling.

## Best Possible Solution

Provide one canonical post-success pruning lifecycle event at the current agent-hook boundary, with a typed, privacy-reviewed payload and documented plugin exposure; keep external indexing and long-term memory storage in plugins rather than core.

## Reproduction Assessment

Yes, at source level. Current main’s pruning handler returns rewritten messages without emitting a lifecycle event, and the hook catalogs and docs contain no pruning hook.

## Solution Assessment

The lifecycle-hook direction is the narrowest maintainable solution, but the proposal is not complete until maintainers decide whether the event is internal-only or public to plugins and what sensitive data, if any, it may carry.
