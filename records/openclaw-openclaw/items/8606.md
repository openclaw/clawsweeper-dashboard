## Summary

Current main still lacks any pruning lifecycle emission, and the explicitly linked implementation PR remains open rather than merged. The request is still necessary, but the permanent contract needs maintainer judgment because the reporter wants pruned content for indexing while the safer candidate exposes metadata only.

## What This Changes

Add a post-pruning lifecycle hook with details about tool-output context removed from active model input.

## Best Possible Solution

Define a stable pruning lifecycle contract at `src/agents/agent-hooks/context-pruning/*`, emit it after every successful soft-trim or hard-clear path, and expose privacy-safe metadata by default; any raw-content indexing capability should use an explicit protected plugin/storage contract rather than placing sensitive content directly in a general hook payload.

## Reproduction Assessment

Yes at source level. Current main has a clear pruning entry point and pruner contract, and neither emits or returns data for a pruning lifecycle event.

## Solution Assessment

Partly. Emitting at the canonical pruning boundary is the narrowest maintainable design, but directly including pruned content is not yet the best solution without an explicit security and storage contract; metadata-only emission is safer but does not completely fulfill the stated indexing use case.
