## Summary

Keep open: current main still prunes by returning rewritten messages without a pruning lifecycle hook or metadata payload, and an open linked PR already targets the requested feature but needs maintainer judgment on the hook contract and privacy boundary.

## What This Changes

Request a context-pruning lifecycle hook that fires after pruning and exposes enough metadata for downstream indexing or search workflows.

## Best Possible Solution

Land or replace the linked pruning-hook PR with a maintainer-approved `session:prune`-style event that emits bounded summary metadata only after successful pruning, documents the internal/plugin visibility boundary, and covers pruning plus hook delivery tests.

## Reproduction Assessment

Yes, at source level. Current main rewrites and returns pruned messages without hook emission or pruning metadata, and the hook catalog/docs still omit a pruning lifecycle event.

## Solution Assessment

Unclear until maintainers choose the event shape. The narrow maintainable path is a safe summary-metadata hook in the existing context-pruning path, with docs stating whether it is internal-only or plugin-visible.
