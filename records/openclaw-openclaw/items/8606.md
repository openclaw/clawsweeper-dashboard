## Summary

Keep open: current main and the latest release still have no context-pruning lifecycle hook, and the direct implementation candidate remains open and unmerged.

## What This Changes

The issue requests a context-pruning lifecycle hook that fires after pruning and exposes safe metadata about trimmed tool output for downstream indexing or memory workflows.

## Best Possible Solution

Keep this issue open and resolve it through the linked PR or a maintainer-approved replacement that emits bounded, sanitized pruning metadata only after successful pruning and documents the event contract.

## Reproduction Assessment

Yes, at source level: current main's pruning extension and pruner complete pruning without emitting a pruning hook or returning hook metadata. I did not run a live agent scenario because the missing hook surface is directly visible in source and docs.

## Solution Assessment

Unclear until maintainers choose the hook contract. A metadata-only `session:prune` event is the likely narrow direction, while raw pruned-content indexing needs explicit privacy, trust, timeout, and sanitization decisions.
