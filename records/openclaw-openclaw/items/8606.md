## Summary

Current main still provides no context-pruning lifecycle signal, and the open implementation PR remains the canonical path. The request fits OpenClaw’s plugin-extension direction, but maintainers must approve whether the supported event is internal, plugin-facing, or both, and ensure its payload cannot expose sensitive pruned content.

## What This Changes

Add a lifecycle hook after context pruning with metadata about trimmed tool outputs so external indexing can react without scheduled polling.

## Best Possible Solution

Add one canonical, privacy-preserving pruning lifecycle event at `src/agents/agent-hooks/context-pruning`, expose it through the supported plugin hook contract when external consumers require it, and keep indexing or long-term-memory behavior in a plugin rather than core.

## Reproduction Assessment

Yes, at the source level. Following the current context handler through `pruneContextMessages` shows that pruning rewrites messages without emitting an event, and the typed hook catalogs and public docs contain no pruning lifecycle hook.

## Solution Assessment

Yes in direction, but the final contract needs maintainer judgment. A generic privacy-safe hook is the narrowest maintainable solution; the existing PR should be adapted or replaced rather than embedding indexing logic or raw pruned content in core.
