## Summary

Current main still lacks the requested context-pruning lifecycle event, and the open maintainer-authored implementation PR remains unmerged. This issue is still necessary until maintainers approve the event and privacy contract and land that PR or a replacement.

## What This Changes

Add a lifecycle hook after context pruning so external handlers can observe what was trimmed and support workflows such as indexing.

## Best Possible Solution

Keep indexing in a plugin and expose a post-success pruning lifecycle contract with explicit privilege and privacy semantics; use bounded summary metadata by default, adding content access only through a deliberate secure contract if maintainers accept that risk.

## Reproduction Assessment

Yes at source level: current main's pruning entry point and pruner are directly traceable, and neither emits nor returns a pruning lifecycle event or payload metadata. This is a feature gap rather than a failing existing contract, so a runtime failure reproduction is not applicable.

## Solution Assessment

The lifecycle-hook direction is the correct architectural seam, but exposing pruned content is not yet a safe final contract. Maintainers must decide internal versus public plugin exposure and whether handlers receive summary metadata, privileged content references, or no raw content.
