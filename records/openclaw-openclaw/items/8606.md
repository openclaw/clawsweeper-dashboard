## Summary

Keep open: current main and v2026.5.20 still prune by returning rewritten messages without hook emission or pruning metadata, while the linked implementation PR remains open and needs maintainer review for the hook contract and security boundary.

## What This Changes

The issue requests a context-pruning lifecycle hook that emits safe metadata about what was pruned so downstream handlers can index or react to pruned content.

## Best Possible Solution

Land or replace the linked pruning-hook PR with a maintainer-approved `session:prune`-style event that emits bounded summary metadata only after successful pruning, documents whether it is internal-only or plugin-visible, and adds focused pruning and hook tests.

## Solution Assessment

Unclear until maintainers choose the event shape. The narrow maintainable path is a safe summary-metadata hook in the existing context-pruning path, with docs stating whether it is internal-only or plugin-visible.
