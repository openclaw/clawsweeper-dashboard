## Summary

Keep open: current main and the latest release still prune context without emitting a pruning lifecycle hook, and the direct implementation path remains an open PR that needs maintainer/security contract review.

## What This Changes

Expose a lifecycle hook when context pruning trims tool outputs so a handler can act on safe pruning metadata.

## Best Possible Solution

Keep this issue as the canonical request and resolve it through the linked PR or a replacement that emits bounded, sanitized pruning metadata only after successful pruning, with focused tests and public hook docs.

## Reproduction Assessment

Yes. At source level, current main prunes through the context extension and pruner without emitting a pruning hook or returning hook metadata; the hook catalogs and docs also omit such an event.

## Solution Assessment

Unclear until maintainers choose the hook contract. A metadata-only `session:prune` event looks like the narrowest maintainable direction, while raw pruned-content indexing needs explicit privacy, trust, timeout, and size-limit decisions.
