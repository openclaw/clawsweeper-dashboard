## Summary

Current main still prunes tool-result context without emitting a pruning lifecycle event, so the requested extension point remains necessary. The linked maintainer-associated implementation at https://github.com/openclaw/openclaw/pull/12581 is still open and should be reviewed rather than bypassed; maintainers must also decide whether the event exposes only safe summary metadata or sensitive pruned content.

## What This Changes

Add a lifecycle hook after context pruning so extensions can react to details about trimmed tool outputs.

## Best Possible Solution

Keep indexing plugin-owned and land one canonical post-prune event at the pruning owner boundary, with an explicitly approved data-minimized payload and clear documentation of whether raw pruned content is intentionally unavailable or exposed only through a separate trusted contract.

## Reproduction Assessment

Yes. Current main has a clear context-pruning path, and source plus hook/documentation searches show with high confidence that no pruning lifecycle event is emitted.

## Solution Assessment

Partly. Emitting after successful pruning at the pruning owner boundary is the narrowest maintainable mechanism, but exposing the content needed for indexing is not automatically the best contract because a metadata-only event is materially safer.
