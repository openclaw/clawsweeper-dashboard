## Label Justifications

- `P2`: This is a normal-priority feature request for context-pruning observability with limited blast radius but real hook-contract and privacy design work.
- `impact:session-state`: The request concerns context pruning, session memory visibility, and downstream indexing after active context changes.
- `impact:security`: A pruning lifecycle hook can expose transcript or tool-output metadata to handlers and external indexes if the payload boundary is not defined carefully.

## Summary

Keep open: current main still prunes by returning rewritten messages without a pruning lifecycle hook or metadata payload, and the linked implementation PR at https://github.com/openclaw/openclaw/pull/12581 remains open and needs maintainer judgment on the hook contract and privacy boundary.

## What This Changes

The issue requests a context-pruning lifecycle hook that fires after pruning with metadata about what was trimmed so downstream handlers can index or otherwise react to pruned content.

## Best Possible Solution

Land or replace the linked pruning-hook PR with a maintainer-approved `session:prune`-style event that emits bounded summary metadata after successful pruning, documents the internal/plugin visibility boundary, and covers pruning plus hook-delivery tests.

## Reproduction Assessment

Yes, at source level. Current main rewrites and returns pruned messages without hook emission or pruning metadata, and the hook catalog/docs still omit a pruning lifecycle event.

## Solution Assessment

Unclear until maintainers choose the event shape and visibility. The narrow maintainable path is a safe summary-metadata hook in the existing context-pruning path, with docs stating whether it is internal-only or plugin-visible.
