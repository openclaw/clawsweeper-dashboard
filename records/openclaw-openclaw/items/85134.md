## Summary

Keep open: the Slack thread-mute direction is useful, but this patch is not merge-ready because the new persistent mute side effect runs before Slack sender authorization, and the PR still lacks real Slack behavior proof.

## What This Changes

The PR adds Slack shared-thread mute detection and persistent per-thread mute state, wires it into `prepareSlackMessage`, and adds unit coverage for detection, persistence, and prepare behavior.

## Best Possible Solution

Keep the thread-mute feature in the Slack prepare path, but perform mute record/clear/drop only after Slack ingress sender authorization, add denied-sender regression coverage, and require redacted live Slack proof before merge.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR records mute at `prepare.ts:851` before the channel sender gate runs later in the same function. I did not run a live Slack reproduction, and the PR’s real behavior proof is still pending.

## Solution Assessment

No as written. A pre-model Slack mute gate is the right broad boundary, but it must sit after sender authorization so persistent thread state cannot be changed by users OpenClaw would otherwise drop.
