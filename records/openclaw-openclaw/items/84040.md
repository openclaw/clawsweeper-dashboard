## Label Justifications

- `P1`: The report describes a live spawn-authorization boundary where wildcard permission can create unintended agent identities with inherited tool access.
- `impact:security`: The issue is about authorization and scoping semantics for which agent IDs a caller may spawn.
- `impact:session-state`: The reported behavior persists child sessions and agent homes under unintended identities.

## Summary

Keep open: current main still accepts any normalized `agentId` when `allowAgents` contains `*`, and the source, tests, and docs show this is a real security-sensitive policy boundary that needs maintainer approval before changing compatibility behavior.

## What This Changes

Request to make `sessions_spawn` reject unknown `agentId` targets when `allowAgents` contains `*`, instead of auto-provisioning default-configured subagents, with an optional compatibility escape hatch and docs/tests.

## Best Possible Solution

Make wildcard spawn targeting validate against configured agents by default, coordinate that with env allowlist work, and document/test an explicit compatibility escape hatch only if maintainers approve it.

## Reproduction Assessment

Yes, from source inspection: the wildcard branch returns ok before any configured-agent membership check, direct and ACP call sites do not pass configured IDs, and current tests expect wildcard or allowlisted unconfigured targets to be accepted. I did not run a live spawn because this review was read-only.

## Solution Assessment

Partly. Registry validation before spawn is the right safety boundary, but changing default wildcard semantics and adding a `strictRegistry` escape hatch need maintainer product/security approval because current docs and tests define `*` as any.
