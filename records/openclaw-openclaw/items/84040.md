## Label Justifications

- `P1`: The issue describes an agent-scoping boundary where wildcard spawn permission can create unintended identities with the requester's tool surface.
- `impact:security`: The affected behavior is authorization/scoping for which agent IDs a caller may spawn.
- `impact:session-state`: The reported behavior persists sessions and agent homes under unintended identities.

## Summary

Keep open: current main still accepts `allowAgents: ["*"]` before any configured-agent registry check, and the existing tests/docs describe that wildcard as accepting any target. The report is security-relevant and source-reproducible, but the fix changes a documented compatibility contract and needs maintainer/security product judgment before automation should implement it.

## What This Changes

The issue asks for `sessions_spawn` wildcard targeting to validate unknown `agentId` values against the configured agent registry, with a possible compatibility opt-out.

## Best Possible Solution

Make wildcard spawn targeting an explicit maintainer-approved contract, thread configured IDs through native and ACP spawn gates, and update tests/docs for both the default behavior and any approved compatibility escape hatch.

## Reproduction Assessment

Yes. Source inspection on current main shows the wildcard branch returns ok before a configured-agent membership check, and current tests expect wildcard to accept an unconfigured target; I did not run a live spawn in this read-only review.

## Solution Assessment

Partly. Registry validation before spawn is the right safety boundary, but default-on strictness and the proposed opt-out flag need maintainer/security approval because current docs and tests define `*` as any.
