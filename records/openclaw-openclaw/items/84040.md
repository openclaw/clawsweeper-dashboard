## Label Justifications

- `P1`: A caller already allowed to use `sessions_spawn` with wildcard policy can create arbitrary agent identities under requester permissions, which is an urgent boundary and observability problem.
- `impact:security`: The issue is about agent identity/scoping and tool-permission boundaries for subagent spawning.
- `impact:session-state`: Unknown ids can create hidden agent homes and transcripts outside the configured agent roster.

## Summary

Keep open: current `main` still accepts wildcard `sessions_spawn` targets without registry validation, and the report identifies a real source-level security/session-state concern. The fix should not be treated as routine cleanup because current docs/tests encode lax wildcard behavior, so maintainers need to choose the compatibility and security policy before implementation.

## What This Changes

The issue requests registry validation for `sessions_spawn.agentId` when `subagents.allowAgents` contains `"*"`, plus compatibility handling for any legacy auto-provision behavior.

## Best Possible Solution

Make spawn enforcement and `agents_list` discovery share one target-registry contract, with schema, docs, tests, and rejection paths that avoid creating agent directories or transcripts for rejected ids.

## Reproduction Assessment

Yes, at source level: with `allowAgents: ["*"]`, `resolveSubagentTargetPolicy` returns ok before comparing the target id to configured agents, and the existing allowlist test expects an unconfigured `beta` target to be accepted. I did not run a live spawn because this review is read-only, but the source path and tests are direct.

## Solution Assessment

Unclear as an implementation plan: stricter registry validation is the right direction for the reported boundary, but the proposed strict-by-default plus opt-out knob changes current docs/tests and needs an explicit compatibility/security decision. The safer end state is one shared policy contract for discovery and spawn, then tests for default and any approved legacy mode.
