## Label Justifications

- `P1`: The report describes a live spawn authorization boundary where wildcard permission can create unintended agent identities with tool access.
- `impact:security`: The issue is about which agent IDs a caller is authorized to spawn under `sessions_spawn`.
- `impact:session-state`: The reported behavior persists child sessions and agent homes under unintended identities.

## Summary

Keep open: current main still accepts wildcard `allowAgents` before any configured-agent membership check, and the fix changes a security-sensitive spawn policy that current docs/tests also describe as `*` meaning any target.

## What This Changes

Require `sessions_spawn` wildcard allowlists to target only configured agents instead of accepting arbitrary agent IDs that auto-provision default subagents.

## Best Possible Solution

Make a maintainer-owned wildcard contract decision, then implement registry validation across native and ACP spawn gates with docs, tests, upgrade behavior, and any explicit compatibility escape hatch.

## Reproduction Assessment

Yes. Source inspection shows the wildcard branch returns ok before configured-agent membership is checked, and current tests expect wildcard and allowlisted-but-unconfigured targets to be accepted.

## Solution Assessment

Partly. Registry validation before spawn is the right safety boundary, but default-on strictness and a `strictRegistry` compatibility option need maintainer approval because current docs and tests define `*` as any.
