## Label Justifications

- `P1`: The issue describes an agent-scoping boundary where wildcard spawn permission can create unintended identities with tool access.
- `impact:security`: The report concerns authorization semantics for which agent IDs a caller may spawn.
- `impact:session-state`: The behavior can persist sessions and agent homes under unintended identities.

## Summary

Keep open: current main still has the wildcard path accepting arbitrary normalized agent IDs, and the security/scoping concern is real. The fix shape needs maintainer product/security judgment because current docs and tests also define `allowAgents: ["*"]` as accepting any target.

## What This Changes

The issue asks OpenClaw to reject `sessions_spawn` targets that are not present in the configured agent registry when `allowAgents` contains `*`, with a documented compatibility escape hatch.

## Best Possible Solution

Make wildcard spawn targeting validate against the configured agent registry by default, thread configured IDs through native and ACP gates, and document an explicitly approved compatibility escape hatch with upgrade coverage.

## Reproduction Assessment

Yes. Source inspection shows the wildcard branch returns ok before any configured-agent membership check, and current tests expect wildcard to accept an unconfigured target; I did not run a live spawn because this review is read-only.

## Solution Assessment

Partly. Registry validation before spawn is the right safety boundary, but the default-on strictness and opt-out flag need maintainer approval because current tests and docs define `*` as any.
