## Label Justifications

- `P1`: The report describes a live agent-scoping boundary where wildcard spawn permission can create unintended identities with tool access.
- `impact:security`: The issue is about authorization and scoping semantics for which agent IDs a caller may spawn.
- `impact:session-state`: The reported behavior persists agent homes and transcripts under unintended identities.

## Summary

Keep open: current main still accepts wildcard subagent targets before any configured-agent membership check, while docs/tests describe `allowAgents: ["*"]` as permissive, so this is a valid security/session-state issue that needs maintainer product and security review.

## What This Changes

The issue requests that `sessions_spawn` treat `allowAgents: ["*"]` as any registered agent rather than any arbitrary agent ID, with compatibility handling, docs, and tests.

## Best Possible Solution

Define the wildcard contract explicitly, then implement registry validation before both native and ACP spawns with regression coverage and a documented compatibility escape hatch only if maintainers approve that upgrade path.

## Reproduction Assessment

Yes from source inspection: the wildcard branch returns `ok: true` before any configured-agent membership check, and current tests expect wildcard to accept an unconfigured target. I did not run a live spawn because this review was read-only.

## Solution Assessment

Partly: registry validation before spawn is the right safety boundary, but changing default wildcard semantics and adding an opt-out setting need maintainer approval because current docs and tests define `*` as any target ID.
