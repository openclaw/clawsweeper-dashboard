## Label Justifications

- `P1`: The report describes an agent-scoping boundary where a wildcard spawn permission can create unintended identities with the requester’s tool surface.
- `impact:security`: The issue is about authorization semantics for which agent ids a caller may spawn under `sessions_spawn`.
- `impact:session-state`: The reported behavior persists child sessions and agent homes under unintended agent identities.

## Summary

Keep open: current main and the latest release still allow the reported wildcard path, and the issue is security-relevant. The fix should not be automated yet because current tests and docs also define `allowAgents: ["*"]` as accepting any target, so tightening it needs a maintainer product/security decision and an upgrade story.

## What This Changes

Request to make `sessions_spawn` reject unknown `agentId` values under `allowAgents: ["*"]`, with an explicit compatibility path for current auto-provisioning behavior.

## Best Possible Solution

Validate wildcard native and ACP subagent targets against the configured agent registry by default, document the contract, and include an explicit compatibility escape hatch only if maintainers choose to preserve auto-provisioning.

## Reproduction Assessment

Yes. Source inspection shows the wildcard branch returns ok before registry membership is checked, and current tests expect `allowAgents: ["*"]` to accept an unconfigured target; I did not run a live spawn because this review is read-only.

## Solution Assessment

Partly. Registry validation is the right safety boundary, but making it default and adding a `strictRegistry` escape hatch changes documented behavior and needs maintainer product/security approval.
