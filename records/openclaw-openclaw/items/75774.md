## Summary

Current main covers the reported MCP stdio child-process leak path: configured MCP runtimes are reused per session/config, stale and in-flight runtimes are disposed, stdio transports kill the process tree on close, and gateway reload/shutdown/session cleanup paths retire cached runtimes. The report was observed on 2026.4.21; the proof is present on current main but not in the latest tagged release.

## What This Changes

Report that a single configured local `link-cli --mcp` MCP server should not spawn duplicate child processes across gateway restart/reload and normal session or cron activity, and should reap child processes cleanly.

## Best Possible Solution

Keep the current-main lifecycle model: reuse one session/config-scoped MCP runtime, dispose stale runtimes on config/session/run/gateway lifecycle boundaries, and kill stdio server process trees on close. No Link-specific core special case is needed.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path for 2026.4.21 using one `mcp.servers.link-cli` entry, in-process gateway restart, normal isolated cron/session activity, and `pgrep`/`ps` evidence of accumulating children; this read-only review verified the current-main fix paths in source, tests, docs, and changelog rather than running the live container repro.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it addresses the generic MCP lifecycle and process-supervision boundary instead of adding a Link-specific workaround.
