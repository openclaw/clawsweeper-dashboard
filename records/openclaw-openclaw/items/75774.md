## Summary

Current `main` already addresses this MCP stdio child-process leak class: session MCP runtimes are reused by session/config, stale and in-flight runtimes are disposed, isolated run paths opt into run-end cleanup, gateway reload/shutdown disposes cached runtimes, and stdio transport close reaps lingering process trees. The Link-specific report matches the same lifecycle surface as earlier MCP child accumulation reports, with current proof on `main` but not in the latest provided release `v2026.4.29`.

## What This Changes

The issue reports that one configured `link-cli --mcp` stdio MCP server can accumulate duplicate child processes across gateway restart/session activity instead of being reused or reaped.

## Best Possible Solution

Keep the current session-scoped MCP runtime design: reuse per session/config, retire run-scoped runtimes when isolated work ends, dispose cached runtimes on MCP config reload and gateway shutdown, and reap stdio process trees on close. If a future report shows duplicate `link-cli --mcp` children on this `main` or a release containing it, it should be opened as a fresh runtime-specific regression with process tree samples and gateway logs.

## Reproduction Assessment

Yes for the originally reported behavior: configure one local stdio MCP server, exercise gateway restart plus isolated cron/session activity, and watch `pgrep`/`ps` for a rising `link-cli --mcp` child count. I did not live-run it here, but the report includes concrete commands and process/memory evidence, and current `main` now has targeted lifecycle coverage for that path.

## Solution Assessment

Yes. The current implementation is the maintainable fix boundary: it avoids duplicate startups by reusing session/config runtimes, handles in-flight races, and cleans up at run, session, config reload, and shutdown boundaries instead of trying to force one global MCP process across all OpenClaw sessions.
