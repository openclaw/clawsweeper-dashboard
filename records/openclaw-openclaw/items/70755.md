## Summary

Keep this issue open. Current main still authenticates BlueBubbles webhook requests with the BlueBubbles server/API password, the public docs still show that password in the webhook URL, and open PR #70757 is the paired implementation candidate using closing syntax for this issue.

## What This Changes

The issue requests a dedicated BlueBubbles inbound webhook secret so OpenClaw no longer reuses the BlueBubbles server/API password for public webhook authentication.

## Best Possible Solution

Land PR #70757 or an equivalent maintainer-approved patch that adds a distinct BlueBubbles webhook credential, authenticates inbound webhooks only against that credential, keeps the API password reserved for BlueBubbles server access, and updates setup, docs, SecretRef metadata, and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure channels.bluebubbles.password, send POST /bluebubbles-webhook?password=<that value> with a caller-controlled handle.address, and current main authenticates before allowlist, pairing, command gating, and agent delivery consume message.senderId.

## Solution Assessment

Yes. The dedicated inbound webhook secret is the narrow maintainable fix; moving the same API password from query string to header would reduce URL leakage but would not separate the public webhook credential from the reusable BlueBubbles API credential.

## Security Review

Status: needs_attention

Summary: This issue reports a concrete BlueBubbles webhook authentication boundary problem that is still present on current main.

Concerns:

- **[high] Separate BlueBubbles webhook auth from API password:** `extensions/bluebubbles/src/monitor.ts:185`
  - body: Current main accepts guid/password query parameters and x-password-style headers, then compares that public webhook token to target.account.config.password. Because docs publish the same value in the webhook URL and the payload supplies sender identity used by allowlist, pairing, command routing, and agent context, inbound webhook auth should use a separate credential.
