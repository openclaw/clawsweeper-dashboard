## Summary

Keep this issue open. Current main still authenticates BlueBubbles webhook requests with the configured BlueBubbles server/API password, docs still publish that password in the webhook URL, and open PR #70757 is the paired implementation candidate for https://github.com/openclaw/openclaw/issues/70755.

## What This Changes

The issue requests a dedicated BlueBubbles inbound webhook secret so OpenClaw no longer reuses the BlueBubbles server/API password for public webhook authentication.

## Best Possible Solution

Land PR #70757 or an equivalent maintainer-approved fix that adds a distinct BlueBubbles webhook credential, authenticates inbound webhooks only against that credential, keeps the API password reserved for BlueBubbles server access, and updates setup, docs, SecretRef metadata, and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure channels.bluebubbles.password, send POST /bluebubbles-webhook?password=<that value> with a caller-controlled handle.address, and current main authenticates before the allowlist/pairing flow consumes message.senderId.

## Solution Assessment

Yes. A dedicated inbound webhook secret is the narrow maintainable fix; moving the same API password from the query string to a header would reduce URL leakage but would not separate the public webhook credential from the reusable BlueBubbles API credential.
