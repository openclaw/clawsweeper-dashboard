## Summary

Keep this issue open. Current main still authenticates BlueBubbles webhooks with the BlueBubbles server/API password, docs still tell operators to publish that password in the webhook URL, and open PR #70757 explicitly closes this issue but has not merged.

## What This Changes

Request to separate BlueBubbles inbound webhook authentication from the BlueBubbles server/API password by adding a dedicated webhook secret and updating setup, docs, config metadata, and tests.

## Best Possible Solution

Ship a maintainer-approved fix that introduces a dedicated BlueBubbles webhook secret, authenticates inbound webhooks only against that secret, keeps the BlueBubbles API password reserved for server access, and updates setup, SecretRef metadata, docs, and regression tests together.

## Reproduction Assessment

Yes. Configure BlueBubbles with channels.bluebubbles.password, send a POST to /bluebubbles-webhook?password=<password> with a caller-controlled handle.address, and current main authenticates before sender allowlist and command-gating logic consumes that sender identity.

## Solution Assessment

Yes. A distinct inbound webhook secret is the narrow maintainable solution; moving the same password to a header would reduce URL leakage but would not separate the public webhook credential from the BlueBubbles server credential.
