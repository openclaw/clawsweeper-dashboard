## Summary

Keep open. The PR targets a real resolver bug, but the proposed alias-collision guard overmatches documented legacy provider aliases and should be narrowed before merge.

## What This Changes

The PR preserves raw custom `models.providers` keys when their normalized provider alias differs, adds `kimi-code` regression tests, and adds a changelog entry for the linked cron model-resolution bug.

## Best Possible Solution

Keep the user-facing fix, but narrow raw-key preservation to exact entries that are truly dispatchable custom inline providers and add coverage proving legacy aliases still normalize when no custom API owner exists.

## Reproduction Assessment

Yes. The resolver path is deterministic: current main returns null for an exact configured provider without provider-level `api`, then `parseModelRef` normalizes `kimi-code` to `kimi`; the PR’s tests cover the selection layer, though not a full cron dispatch.

## Solution Assessment

No as written. Preserving raw keys for every alias collision is broader than the documented model-provider contract; the safer fix is to detect custom ownership without capturing legacy compatibility aliases.
