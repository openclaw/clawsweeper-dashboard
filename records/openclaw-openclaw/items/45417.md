## Summary

Keep this PR open. Current main still lacks immediate TUI rendering for inbound user turns, but the PR's current approach does not cover the external-channel ingress path described by the linked report and introduces a broader raw-message broadcast surface.

## What This Changes

The PR adds a `user-message` chat event emitted from `chat.send`, extends the TUI chat handler and protocol schema to accept it, adds TUI regression tests, and includes unrelated Anthropic model-selection regression coverage.

## Best Possible Solution

Keep the PR open, but redirect the implementation toward the existing transcript stream: have TUI `--session` subscribe to `sessions.messages.subscribe`, render `session.message` user turns for the active session with local echo/dedupe protection, and add focused gateway/TUI coverage. If maintainers prefer a new chat event, emit it from the shared ingress/transcript pipeline with explicit protocol docs and subscriber scoping.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: current main loads history but the TUI neither subscribes to nor handles `session.message`, and the PR diff emits `user-message` only from `chat.send`; a live Telegram run was not needed to verify that mismatch.

## Solution Assessment

No. The proposed patch is not the best fix as written because it creates a new global `chat` event for raw user text while the repo already has a subscription-gated transcript event path designed for session messages.

Full review comments:

- **[P1] Emit user turns from the ingress transcript path:** `src/gateway/server-methods/chat.ts:1274-1275`
  - body: The new `user-message` event is emitted only inside `chat.send`, but the reported Telegram/external-channel flow enters through the shared channel ingress/`dispatchInboundMessage` path and does not call this RPC. As a result, `openclaw tui --session ... --deliver` still misses the primary remote user turn this PR is meant to show; wire the TUI to `session.message` or emit from the shared transcript/ingress path instead.
  - confidence: 0.91
- **[P2] Target raw user-message events to session subscribers:** `src/gateway/server-methods/chat.ts:1274`
  - body: `context.broadcast("chat", userMessagePayload)` sends the raw inbound message to every operator read WebSocket, leaving unrelated clients to discard by session key. The existing `session.message` path is explicitly subscription-gated, so raw user turns should stay on that targeted path or an equivalent scoped event.
  - confidence: 0.86
