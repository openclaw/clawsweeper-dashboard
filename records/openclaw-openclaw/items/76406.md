## Summary

Keep open: the PR targets a real gateway hot path, but it should not merge as-is because the shared gateway catalog loader also serves the documented `models.list` full-catalog view and the changed call is not type-compatible with the local loader signature.

## What This Changes

The branch makes gateway model catalog refresh call `loadModelCatalog` in read-only mode and adds a persisted `models.json` fast path that skips provider plugin augmentation for read-only catalog loads.

## Best Possible Solution

Keep the performance optimization, but split the fast read-only picker/configured path from the full diagnostic catalog path so `models.list` with `view: "all"` keeps returning the complete catalog.

## Reproduction Assessment

Yes. The regression is source-reproducible from the PR diff plus current docs/tests: `view: "all"` is defined as the full catalog, while the PR routes the shared gateway catalog load through a read-only fast path that can return only persisted/configured rows.

## Solution Assessment

No. The optimization is in the right area, but applying `readOnly` at the shared gateway catalog cache is too broad; the safer fix is a separate read-only/default picker path or a view-aware loader that preserves the full path for diagnostics.

## Review Findings

- **[P2] Preserve the full catalog for `view: "all"`:** `src/gateway/server-model-catalog.ts:46`
  - body: This shared loader is also what `models.list` uses for `view: "all"`, which current docs/tests define as the full Gateway catalog.   - confidence: 0.93

  - body: `LoadModelCatalog` is still declared as a function that accepts only `{ config }`. Passing the object literal `{ config, readOnly: true }` through that type should fail TypeScript excess-property checking, so the PR needs to widen the injected loader type or avoid passing unsupported params through this wrapper.
  - confidence: 0.9
