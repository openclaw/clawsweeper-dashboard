## Summary

Keep open: the issue has a protected maintainer label and current source still shows a credible diagnostics gap for Slack/channel callbacks, especially the lack of a channel-level trace scope and unparented `openclaw.message.processed` OTel span creation.

## What This Changes

The issue asks for Slack/channel message diagnostics to be correlated into one trace across inbound message processing, harness execution, model usage, model calls, and reply delivery.

## Best Possible Solution

Keep this open for a maintainer-owned fix that creates or preserves one trusted per-inbound-message trace context in the shared channel dispatch path, parents message lifecycle and delivery spans, and proves the Slack/OTel waterfall with regression coverage and live Tempo proof.

## Reproduction Assessment

Yes, source-level: the Slack inbound path dispatches without its own diagnostic trace scope, embedded agent traces fall back to a new root when no active scope exists, and `openclaw.message.processed` is exported without parent context. A live Tempo run would still be needed before closing or landing a fix.

## Solution Assessment

Yes on direction, but no patch is proposed here: the best fix should live in the shared channel/message diagnostic path rather than a Slack-only workaround, with the OTel exporter honoring trusted parent context for message lifecycle spans.
