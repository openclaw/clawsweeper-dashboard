## Summary

This should stay open: current `main` still requires `hooks.token` as a literal string, explicitly rejects SecretRef objects at that path, and only treats `${ENV_VAR}` syntax as an audit exemption rather than runtime resolution. The remaining work is a useful operator/security-hardening feature, but choosing `tokenFile`, env interpolation, or SecretRef support is a public config and security-policy decision.

## What This Changes

The issue requests a secrets-out-of-config mechanism for `hooks.token`, such as `tokenFile`, env-var interpolation, or SecretRef support.

## Best Possible Solution

Add one documented, test-covered secrets-out-of-config path for hook ingress, likely `hooks.tokenFile` parity with channel tokens unless maintainers prefer SecretRef policy work, and keep audit/remediation docs aligned with the chosen contract.

## Reproduction Assessment

Yes for the current limitation by source inspection: current `main` has a literal-string-only resolver, a strict schema with no `tokenFile`, and an explicit SecretRef denylist for `hooks.token`. I did not run a live gateway curl repro because this read-only review forbids test or build commands that may write artifacts.

## Solution Assessment

Unclear which proposed option is best without maintainer direction. `tokenFile` parity appears to be the narrowest low-surprise implementation, while env interpolation and SecretRef support change broader config or policy semantics.
