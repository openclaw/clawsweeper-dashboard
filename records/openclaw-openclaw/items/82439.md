## Summary

Current main and shipped releases already support the central non-plaintext path requested here: generic `${VAR}` environment substitution is applied to any config string before hooks auth reads `hooks.token`. `tokenFile` and SecretRef remain unsupported for this field, but the issue asked for env-ref or tokenFile, and the env-ref path is already the supported remediation.

## What This Changes

Add or identify a non-plaintext secret source for `hooks.token`, such as env references, `tokenFile`, or SecretRef support.

## Best Possible Solution

Keep the shipped `${ENV_VAR}` config-substitution path as the supported non-plaintext source for `hooks.token`; handle audit false positives or a future `tokenFile` field as separate focused work.

## Reproduction Assessment

No. The plaintext-only behavior does not reproduce from current source: config reads resolve `${VAR}` strings before `resolveHooksConfig` reads `hooks.token`, though `tokenFile` and SecretRef remain unsupported.

## Solution Assessment

Yes. The existing generic config-substitution contract is the narrowest maintainable solution for the env-ref part because it avoids a hooks-only special case; `tokenFile` or SecretRef parity would be a separate config/API expansion.
