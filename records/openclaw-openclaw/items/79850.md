## Summary

Keep open. Current main does not prove the reported `request.allowPrivateNetwork: true` path is broken, but the `.internal`/`hostnameAllowlist` behavior is still an active SSRF policy question and this issue is already paired with an open closing PR.

## What This Changes

The issue asks for model provider requests to `host.docker.internal` and other `.internal` hostnames to work when private-network access or the provider base URL host policy is configured.

## Best Possible Solution

Review the linked PR or a narrower replacement that proves the effective `request.allowPrivateNetwork: true` path and preserves reserved-host and metadata protections unless maintainers choose to broaden `hostnameAllowlist` semantics.

## Reproduction Assessment

Unclear. Current main source reproduces a block when only the auto-generated `hostnameAllowlist` is present, but the issue’s config does not show `request.allowPrivateNetwork: true` and source inspection shows that documented opt-in should skip the reserved/private checks.

## Solution Assessment

Unclear. A broad `hostnameAllowlist` bypass is not clearly the safest fix; the maintainable path is to validate the documented private-network opt-in and keep any policy expansion under explicit maintainer security review.
