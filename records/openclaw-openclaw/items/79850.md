## Summary

Keep open: the report points at a real SSRF-policy edge, but current source suggests the model-provider fetch path already skips hostname/IP checks when `models.providers.*.request.allowPrivateNetwork` is actually present, while the low-level exported classifier still blocks `.internal` names. Because this is SSRF guard behavior and the repro config omits the opt-in field, this needs maintainer/security review rather than automatic cleanup.

## What This Changes

Allow model-provider requests to configured `.internal` hosts such as `host.docker.internal` when the operator opts in with `request.allowPrivateNetwork`.

## Best Possible Solution

Add a focused regression for `host.docker.internal` model-provider fetches with `request.allowPrivateNetwork: true`, then make the narrowest SSRF-policy change that preserves metadata/internal hostname blocking for untrusted fetches.

## Reproduction Assessment

Unclear: source inspection confirms the low-level classifier blocks `.internal`, but the current model-provider fetch path appears to bypass that block when `request.allowPrivateNetwork: true` is actually attached. A live Docker provider repro or targeted regression test is still needed for the exact reported path.

## Solution Assessment

Unclear: the suggested change is plausible for the low-level helper but may be broader than needed for the model-provider path. A safer fix would first pin down whether the config opt-in is missing from the repro or dropped at runtime, then patch only that seam.
