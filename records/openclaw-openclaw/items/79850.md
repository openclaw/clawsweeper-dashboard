## Summary

Keep open. Current main still has an unresolved SSRF policy question around `.internal` hostnames and `hostnameAllowlist`, and two open PRs explicitly reference this issue with closing syntax, so the issue should remain open until one PR lands or maintainers reject the approach.

## What This Changes

The issue requests that SSRF guarded model fetches allow trusted Docker/private `.internal` provider hostnames such as `host.docker.internal` instead of blocking them before policy allowlists are honored.

## Best Possible Solution

Review the focused SSRF PRs and land the narrow fix that supports `host.docker.internal` under explicit trusted-provider semantics while preserving metadata and reserved-host protections.

## Reproduction Assessment

Unclear. Source inspection shows current main blocks `host.docker.internal` when only the auto-generated `hostnameAllowlist` is present, but the issue’s config omits the documented `request.allowPrivateNetwork: true` opt-in and source suggests that opt-in changes the guarded fetch path.

## Solution Assessment

Unclear. The open PRs are the right place to settle this, but the safest solution must avoid turning provider base-url allowlists into a general reserved-hostname/private-IP bypass.
