## What This Changes

The issue reports that guarded model-provider requests to Docker hostnames such as `host.docker.internal` are blocked by SSRF hostname/private-IP checks despite the intended private-network allowance.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/79862 lands or is replaced by a narrower fix that preserves the explicit private-network trust boundary.

## Reproduction Assessment

Unclear for the issue as written. The shown config lacks `models.providers.<id>.request.allowPrivateNetwork: true`, while current source appears to skip both reserved-host and DNS-answer private checks when that flag is actually present; the no-private-opt-in path is source-reproducible.

## Solution Assessment

No, closing this issue now is not the best path. The implementation should be decided in the linked PR, with maintainer/security review of whether `hostnameAllowlist` should imply private-network trust.
