## Summary

Keep open. Current main still has the SSRF hostname/private-DNS policy behavior under discussion, and https://github.com/openclaw/openclaw/pull/79862 is an open same-author PR that explicitly closes this issue, so this issue should stay open until that PR lands or maintainers choose a different scoped fix.

## What This Changes

Allow Docker/local model provider requests to `host.docker.internal` or other `.internal` hostnames when the provider SSRF policy is trusted for private-network access.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/79862 against the SSRF trust model, then either land a scoped fix that preserves explicit private-network trust or document the required `request.allowPrivateNetwork` path more clearly.

## Reproduction Assessment

Unclear. Source shows current main blocks `host.docker.internal` when only the auto-generated `hostnameAllowlist` is present, but the issue's config omits documented `models.providers.<id>.request.allowPrivateNetwork: true`, and source shows that explicit flag is threaded into model fetch policy.

## Solution Assessment

Unclear. The open PR's broad hostname-allowlist-as-private-trust direction fixes the reported Docker flow, but the safer maintainable path should keep private-network DNS bypasses tied to explicit `request.allowPrivateNetwork` unless maintainers intentionally change that contract.
