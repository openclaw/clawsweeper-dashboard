## Summary

Keep open. Current main still has a source-level blocking path for the shown `host.docker.internal` provider setup, and an open closing PR already targets this issue, so the issue should stay open until that PR is reviewed, merged, or closed.

## What This Changes

Allow Docker/provider hostnames such as `host.docker.internal` to pass SSRF checks when trusted by provider private-network or hostname policy.

## Best Possible Solution

Review the open closing PR and land only a fix that preserves explicit private-network and metadata SSRF boundaries, or intentionally documents a maintainer-approved `hostnameAllowlist` trust contract.

## Reproduction Assessment

Yes for the concrete shown config path by source inspection: current main auto-adds a provider `hostnameAllowlist`, then still blocks `.internal` names and private DNS answers when no explicit private-network skip applies. It is unclear for the title's explicit `allowPrivateNetwork` claim because current main does thread that flag into the skip path when configured.

## Solution Assessment

Unclear. The linked PR direction fixes the demonstrated Docker flow, but the safest maintainable solution must not turn every `hostnameAllowlist` entry into private-network trust without a maintainer security decision.
