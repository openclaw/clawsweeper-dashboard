## Summary

Keep open for maintainer triage. The reported surface is real, but current main appears to thread `models.providers.*.request.allowPrivateNetwork` into guarded model fetches and skip the reserved-host/private-IP checks when that documented opt-in is true; the reproduction config shown in the issue omits that nested option, so the claimed failure is not proven from the provided steps.

## What This Changes

The issue requests that model-provider requests to `host.docker.internal` and other `.internal` hostnames succeed when private-network access is explicitly enabled.

## Best Possible Solution

Confirm the failure with an effective config containing `models.providers.<id>.request.allowPrivateNetwork: true`; if it still fails, fix the SSRF policy with a focused `host.docker.internal` regression while preserving metadata and reserved-host protections.

## Reproduction Assessment

Unclear. The shown reproduction config does not include the documented nested `request.allowPrivateNetwork: true`, and source inspection shows current main should pass that opt-in through the guarded model fetch path.

## Solution Assessment

Unclear. The requested behavior is reasonable for explicit provider private-network opt-in, but the proposed broad hostname-allowlist ordering change is not the safest fix without a regression that preserves reserved-host SSRF protections.
