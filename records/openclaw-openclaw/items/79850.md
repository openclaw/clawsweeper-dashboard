## Summary

Keep open. Current main still contains the reported SSRF boundary, and an open linked PR already attempts a fix, but the remaining question is a maintainer/security policy decision about whether `hostnameAllowlist` should imply private-network trust.

## What This Changes

The issue requests that configured model provider requests to `.internal` hostnames such as `host.docker.internal` work through the SSRF guard when the provider target is trusted.

## Best Possible Solution

Review the linked fix PR and choose a security-preserving contract: either require explicit `request.allowPrivateNetwork` for `host.docker.internal` or intentionally bless host-scoped allowlists with regression tests and maintainer approval.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the exact `allowPrivateNetwork: true` claim, because the provided config omits that nested option; source inspection does show current main blocks `.internal` when only the auto-generated `hostnameAllowlist` is present.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Allowing `hostnameAllowlist` to bypass all private DNS checks would fix the Docker flow but needs maintainer/security agreement because it changes SSRF trust semantics.
