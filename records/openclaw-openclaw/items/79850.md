## Summary

Keep open: current main has not merged a fix for the allowlist-only `.internal` path, and https://github.com/openclaw/openclaw/pull/79862 is an open closing PR that should be reviewed or revised before this issue is resolved.

## What This Changes

The issue asks for model-provider requests to `host.docker.internal` and similar `.internal` hostnames to pass the SSRF guard when the provider is trusted for private-network access.

## Best Possible Solution

Review and either land a security-approved version of https://github.com/openclaw/openclaw/pull/79862 or narrow the fix to the documented `request.allowPrivateNetwork: true` trust boundary with regression coverage for `host.docker.internal`.

## Reproduction Assessment

Unclear for the exact title claim: the issue’s sample config omits nested `request.allowPrivateNetwork`, and current source appears to pass that opt-in through when present. The allowlist-only failure is source-reproducible because the provider `baseUrl` creates `hostnameAllowlist`, but current SSRF checks still block `.internal` and private DNS answers without explicit private-network trust.

## Solution Assessment

No for the broad allowlist bypass as currently proposed; it changes an SSRF trust boundary. The narrower maintainable path is to preserve explicit private-network opt-in unless maintainers approve a new `hostnameAllowlist` contract.
