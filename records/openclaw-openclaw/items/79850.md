## Summary

Keep open. Current main still contains the source path that blocks `host.docker.internal` when only the provider-generated `hostnameAllowlist` is present, but this issue is already paired with an open fixing PR that uses closing syntax and that PR has a security-policy review concern, so the issue should stay open until the linked PR is landed or replaced.

## What This Changes

The issue requests SSRF handling that permits configured model-provider hosts such as `host.docker.internal` through reserved-hostname and resolved-private-IP checks when the provider base URL or private-network policy makes that host trusted.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/79862 or replace it with a narrower fix that preserves the explicit private-network trust boundary while adding focused `host.docker.internal` regression coverage.

## Reproduction Assessment

Partly. Source inspection shows current main blocks `host.docker.internal` for the pasted config where only the auto-generated `hostnameAllowlist` is present, but it does not prove a failure when `request.allowPrivateNetwork: true` is actually set.

## Solution Assessment

Unclear. The linked PR fixes the observed Docker flow by broadening `hostnameAllowlist` semantics, but the safer maintainable direction is to keep private-network bypasses tied to explicit trust unless maintainers approve changing that contract.
