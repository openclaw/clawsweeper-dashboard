## Summary

Keep open. Current main still has the reported `.internal` hostname block path, and https://github.com/openclaw/openclaw/pull/79862 is an open same-author implementation PR with closing syntax and an unresolved SSRF policy review concern.

## What This Changes

The issue asks for Docker host-local model provider URLs such as `host.docker.internal` to work under the model-provider SSRF/private-network policy.

## Best Possible Solution

Resolve the policy question in https://github.com/openclaw/openclaw/pull/79862: either land a narrow fix with regression coverage that preserves explicit private-network trust, or document the required config if maintainers decide hostname allowlists must not imply private-network access.

## Reproduction Assessment

No, not for the exact `allowPrivateNetwork: true` claim; the shown config omits that nested option and source shows current main threads it through. Yes by source inspection for the no-explicit-opt-in path: `.internal` hostnames and private DNS answers remain blocked despite the provider-generated `hostnameAllowlist`.

## Solution Assessment

Unclear. The active PR fixes the reported Docker flow, but the broad `hostnameAllowlist` bypass is not clearly the safest maintainable SSRF contract without maintainer approval.
