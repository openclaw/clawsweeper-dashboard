## Summary

Keep open. Current main still blocks `host.docker.internal` when only the auto-generated `hostnameAllowlist` is present, but the documented private-DNS path is `models.providers.<id>.request.allowPrivateNetwork: true`; the same author has an open closing PR at https://github.com/openclaw/openclaw/pull/79862, so this issue should stay open while that security-policy decision is reviewed.

## What This Changes

Allow Docker host-local model provider URLs such as `host.docker.internal` to pass the SSRF guard when explicitly trusted instead of being blocked by reserved-hostname/private-IP checks.

## Best Possible Solution

Keep this issue open while https://github.com/openclaw/openclaw/pull/79862 is reviewed; either land a maintainer-approved policy change with SSRF regressions or document the exact `request.allowPrivateNetwork` Docker config if current behavior is intentional.

## Reproduction Assessment

Unclear for the exact `request.allowPrivateNetwork: true` claim because current source threads that option into the guarded model fetch path. Source inspection does reproduce the narrower hostnameAllowlist-only block for `host.docker.internal`.

## Solution Assessment

No, not as currently proposed in the paired PR without maintainer approval. The narrow maintainable path is to preserve explicit private-network trust and add focused coverage or documentation for Docker `host.docker.internal`.
