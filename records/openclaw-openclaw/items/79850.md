## Summary

Keep open: current main still has the SSRF policy ambiguity around `.internal` hostnames, and an open contributor PR already closes this issue, so the issue should remain open until that PR is reviewed, merged, or rejected.

## What This Changes

The issue asks for model provider requests to `host.docker.internal` and similar `.internal` hostnames to be allowed when the configured provider endpoint is treated as trusted by SSRF policy.

## Best Possible Solution

Review and either land or revise https://github.com/openclaw/openclaw/pull/79862 with focused SSRF regression coverage and clear policy semantics, then close the issue after merge.

## Reproduction Assessment

Partly: the no-explicit-opt-in path is source-reproducible because the provider base URL creates a `hostnameAllowlist`, but current main still blocks `.internal` before that allowlist affects the hostname/IP helper. The literal `allowPrivateNetwork: true` claim is not high-confidence because the issue config omits that nested field and current source appears to honor it.

## Solution Assessment

Unclear. The linked PR is the right place to resolve this, but maintainers should decide whether `hostnameAllowlist` alone may bypass reserved-host/private-IP checks or whether the exception must remain tied to explicit `request.allowPrivateNetwork: true`.
