## Summary

Keep open: the cleanup direction is useful, but the latest PR head still ignores configured session maintenance retention when deciding what to delete. The branch passes config into the new call sites, yet the sweeper resolves defaults instead of `cfg.session.maintenance`, so custom retention or disabled reset-archive cleanup can be violated.

## What This Changes

The PR adds gateway startup/hourly cleanup for stale session archive artifacts and orphaned atomic-write temp files, adds focused tests and a changelog entry, and includes broad formatting-only churn in unrelated UI/plugin files.

## Best Possible Solution

Land a narrowed proactive cleanup that passes existing `session.maintenance` settings into the sweeper, preserves default and configured store roots, covers configured/disabled retention in tests, and trims unrelated churn before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows the sweeper receives `cfg` but resolves maintenance with no input, while current main's resolver is explicitly parameterized by `SessionMaintenanceConfig`.

## Solution Assessment

No. Startup/hourly cleanup is a maintainable direction, but this implementation must call `resolveMaintenanceConfigFromInput(params.cfg?.session?.maintenance)` before it can safely honor user retention settings.
