## Summary

Keep open. Current main still lacks the startup/hourly session archive sweep that this PR targets, and the linked disk-bloat issue remains open, but the PR still needs real behavior proof and a maintainer decision because it deletes files even when session maintenance is configured as warn-only.

## What This Changes

The PR adds a gateway startup/hourly sweeper for stale `.deleted.*`, `.reset.*`, orphaned `.tmp`, and excess `.bak.*` session artifacts, plus tests, timer shutdown wiring, a changelog entry, and broad formatting churn.

## Best Possible Solution

Land or replace this with a focused proactive cleanup implementation that covers configured session-store roots, preserves or explicitly updates warn-mode semantics, trims unrelated churn, and includes redacted real startup/timer cleanup proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a gateway startup/hourly session artifact sweep, while existing cleanup paths do not proactively remove the full `.deleted.*`, `.reset.*`, orphan `.tmp`, and `.bak.*` artifact set for default warn-mode users.

## Solution Assessment

No, not as currently written. The cleanup direction is maintainable, but destructive startup/hourly cleanup should either respect `mode: "enforce"` or land with explicit maintainer-approved docs/product changes for warn-mode deletion semantics.
