## Summary

Keep open: current main still lacks the proposed proactive session archive cleanup, and the PR is useful for the linked disk-bloat bug, but the branch is stale against current main and has blocking review findings around the maintenance-config import and custom session-store discovery.

## What This Changes

The PR adds a gateway startup sweep and hourly timer to remove stale `.deleted.*`, `.reset.*`, orphaned UUID `.tmp`, and excess `.bak.*` session artifacts, with tests for those artifact classes.

## Best Possible Solution

Land a rebased, focused gateway/session maintenance fix that uses current main's config-aware session-store target discovery, imports the current maintenance runtime helper, wires the startup sweep into the current sidecar lifecycle, keeps the hourly timer clearable on shutdown, preserves active transcripts and young atomic-write temp files, and adds default-root plus custom `session.store` regression coverage with a user-facing changelog entry.

## Reproduction Assessment

Yes. The linked bug can be reproduced by leaving stale `.deleted.*`, `.reset.*`, UUID `.tmp`, or `.bak.*` artifacts under an agent sessions directory, and current-main source shows cleanup only runs through reactive session-store maintenance rather than a proactive startup/hourly sweep.

## Solution Assessment

No. The direction is maintainable, but this PR is not the best current fix until it is rebased onto current gateway startup code, imports the current maintenance runtime module, and scans configured session-store roots instead of only default state-dir agent sessions.
