## Summary

Keep this PR open: current main still lacks proactive session-archive cleanup and the PR addresses the linked disk-bloat bug, but the latest diff has a blocking data-deletion issue in the new numeric backup pruning path.

## What This Changes

The PR adds gateway startup and hourly cleanup for stale session archive artifacts, orphaned atomic-write temp files, and excess backups, with timer shutdown wiring, regression tests, a changelog entry, and broad formatting churn.

## Best Possible Solution

Land a narrow proactive cleanup that preserves existing retention semantics while restricting deletion to known OpenClaw session artifact names and configured store basenames.

## Reproduction Assessment

Yes. Source inspection of current main shows archive cleanup only runs from session-store write maintenance, while the PR’s new path can be checked directly against the added sweeper and startup wiring.

## Solution Assessment

No as currently written. Startup/hourly cleanup is the right direction, but the numeric backup matcher must be narrowed before this is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Constrain numeric backup pruning to session stores:** `src/gateway/session-archive-cleanup.ts:112`
  - body: The new sweep now scans `path.dirname(target.storePath)` for configured stores, and configured `session.store` paths can be arbitrary files. With the broad `/\.bak\.\d+$/` matcher, a custom store in a shared directory can cause unrelated backups such as `openclaw.json.bak.1` or other `*.bak.<digits>` files to be grouped and deleted after three entries. Please restrict numeric backup pruning to the configured store basename, or to the legacy `sessions.json.bak.<digits>` pattern, while keeping ISO transcript archive handling separate.
  - confidence: 0.84

## Security Review
