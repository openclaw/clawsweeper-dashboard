## Summary

Keep this PR open: it addresses a real, source-reproducible gateway status diagnostic bug and includes real macOS terminal proof, but the current token-flattening approach can hide malformed shell-wrapped service commands that only mention `gateway` outside the executed OpenClaw argv.

## What This Changes

The PR expands gateway service audit command tokens so shell-wrapped LaunchAgent payloads expose `gateway` and `--port`, with regression tests for a macOS wrapper and a `--gateway-url` near miss.

## Best Possible Solution

Keep the shell-wrapped LaunchAgent support, but parse the shell payload by executable command segment and validate the actual OpenClaw gateway argv and port before suppressing service-audit warnings.

## Reproduction Assessment

Yes. Current main passes LaunchAgent ProgramArguments into the service audit while `hasGatewaySubcommand` only checks exact top-level argv elements, and the LaunchAgent reader leaves generic `/bin/zsh -lc` payloads opaque.

## Solution Assessment

No. The direction is right, but flattening every shell word is broader than the maintainable fix; the safer solution is segment-aware shell parsing that validates the actual OpenClaw gateway argv.
