## Summary

Keep open. Current main still has a clear isolated-cron CLI lifecycle gap: the embedded cron runner opts isolated jobs into run-end MCP cleanup, but the CLI cron branch does not pass the available live-session or bundle-MCP cleanup flags despite the report’s production symptoms.

## What This Changes

The issue reports that frequent isolated cron agent turns on Railway leave stale OpenClaw worker processes behind, causing host load spikes and multi-minute responses.

## Best Possible Solution

Add isolated-cron CLI run-end cleanup using the existing CLI live-session and bundle-MCP cleanup controls, with regression coverage and a Docker/Railway-style stress check.

## Reproduction Assessment

Yes. The issue gives a concrete Railway/Docker reproduction path with worker counts, load averages, and cleanup recovery, and current source still shows the missing isolated-cron CLI cleanup path; I did not live-run Railway in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing one-shot CLI cleanup flags for isolated cron CLI runs instead of adding a broad host-level process reaper or external cleanup script.

## Repair Work Prompt

Fix issue #76171 on current main. The observable bug is that frequent isolated cron agent turns can leave stale OpenClaw worker/CLI child processes behind, causing high host load and slow responses. Keep the fix scoped to isolated cron CLI lifecycle cleanup: in src/cron/isolated-agent/run-executor.ts, pass the existing run-end cleanup controls to runCliAgent for isolated cron runs, aligned with the embedded runner's cleanupBundleMcpOnRunEnd behavior. Prefer cleanupCliLiveSessionOnRunEnd and cleanupBundleMcpOnRunEnd for sessionTarget === "isolated"; avoid adding a broad process reaper, new config option, or startup migration. Add/update regression tests that prove the CLI isolated cron path passes the cleanup flags, and keep existing non-isolated behavior intentional. Add a user-facing CHANGELOG.md fix entry. Validate with targeted cron/CLI tests and, if available, the Docker cron MCP cleanup smoke or an equivalent Railway-style stress check that records process command lines, worker counts, load average, and cleanup behavior.
