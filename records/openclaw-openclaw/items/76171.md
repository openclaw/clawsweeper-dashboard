## Summary

Keep open. Current main still has a plausible isolated-cron CLI lifecycle gap: embedded cron runs opt into run-end cleanup, but the CLI-runtime branch does not, while the report and follow-up comments describe current production worker accumulation under cron load.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating after cron/agent turns on Railway, causing high host load and multi-minute response latency.

## Best Possible Solution

Add isolated-cron CLI run-end cleanup using the existing CLI live-session and bundle-MCP cleanup controls, with regression coverage and Docker/Railway-style stress validation.

## Reproduction Assessment

Yes, the issue gives a high-confidence production reproduction path on Railway/Docker with multiple cron jobs over 30-60 minutes, process counts, load averages, and manual-cleanup recovery. I did not live-run Railway here, but current source still shows the missing isolated-cron CLI cleanup path.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing one-shot CLI cleanup flags for isolated cron CLI runs rather than adding a broad process reaper or external cleanup script.
