## Summary

Keep open. Current main has cleanup seams for timeouts, browser sessions, embedded bundled MCP, and explicit one-shot CLI cleanup, but the isolated cron CLI path still does not prove completed worker processes are reaped, and the report has concrete production load/process-count evidence plus a second confirmation.

## What This Changes

The issue asks for stale cron/agent worker processes to exit after each run so host load and response latency do not accumulate.

## Best Possible Solution

Implement bounded lifecycle cleanup for cron-launched isolated CLI/agent workers, preserving persistent sessions, then prove it with targeted regression tests and a Docker/Railway-style cron stress smoke.

## Reproduction Assessment

Yes. The issue gives a concrete production reproduction path on Railway/Docker with multiple cron jobs over 30-60 minutes, worker counts, load averages, and recovery after cleanup; this review did not live-run that environment.

## Solution Assessment

Unclear. The narrowest likely code path is isolated cron CLI lifecycle cleanup, but the stale process command lines should be confirmed before adding broader process-killing or orphan-reconciliation policy.
