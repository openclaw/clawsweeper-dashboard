## Label Justifications

- `P1`: The issue describes severe user-facing availability degradation from worker accumulation under normal cron/agent automation load.
- `impact:crash-loop`: The failure mode is runaway process accumulation, host saturation, memory pressure, and gateway stalls.

## Summary

Keep open: current main and the latest release still leave the isolated cron CLI branch outside the available run-end cleanup contract, while embedded isolated cron already requests bundled MCP cleanup and docs promise isolated cron runtime teardown.

## What This Changes

The issue asks OpenClaw to stop stale cron/agent worker processes from accumulating by ensuring one-shot worker runs exit and clean up after completion.

## Best Possible Solution

Fix the isolated cron CLI-backed run path so one-shot runs clean up CLI live-session and bundled MCP resources without closing a shared loopback server owned by another active run.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 still show the isolated cron CLI branch calling runCliAgent without the cleanup flags, while runCliAgent has explicit run-end cleanup behavior when those flags are set. I did not run a live multi-cron process-count reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to propagate the existing one-shot cleanup contract from isolated cron CLI execution and add focused regression coverage. The implementation should account for the adjacent shared loopback ownership risk before landing.
