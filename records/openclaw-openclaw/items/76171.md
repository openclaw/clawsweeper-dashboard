## Summary

Keep open. Current main still has a source-backed isolated-cron CLI lifecycle gap: the CLI branch does not request run-end cleanup, while the report and follow-up comments describe stale worker accumulation under cron load.

## What This Changes

The issue asks for OpenClaw cron worker processes to exit cleanly after each completed isolated agent turn instead of accumulating on the host.

## Best Possible Solution

Add run-end cleanup for isolated cron CLI runs using the existing CLI live-session and bundle-MCP cleanup flags, while preserving persistent current/custom cron session reuse.

## Reproduction Assessment

Yes. The issue gives a high-confidence Railway/Docker reproduction path over 30-60 minutes, and current main still shows the isolated-cron CLI branch missing the cleanup controls; I did not live-run Railway in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse existing run-end cleanup flags for isolated cron CLI calls rather than adding a broad host process reaper or cleanup script.

## Repair Work Prompt

Fix stale worker accumulation for isolated cron jobs by reusing existing CLI cleanup controls. In `src/cron/isolated-agent/run-executor.ts`, ensure the CLI-backed isolated cron path passes `cleanupCliLiveSessionOnRunEnd: true` and `cleanupBundleMcpOnRunEnd: true` when `job.sessionTarget === "isolated"`, while preserving warm CLI/MCP behavior for current/custom session targets. Add regression coverage in the isolated cron tests, likely `src/cron/isolated-agent/run.session-key-isolation.test.ts` or `src/cron/isolated-agent/run.skill-filter.test.ts`, proving isolated CLI cron sets both flags and persistent session targets do not. Do not add a host process reaper, cleanup script, config option, or broad process-kill policy. Add a single-line `CHANGELOG.md` fix entry for the user-facing cron worker leak. Validate with the targeted tests and, if available, a Docker/Railway-style smoke running multiple isolated cron jobs for 30-60 minutes while recording worker command lines, counts, load average, and cleanup behavior.
