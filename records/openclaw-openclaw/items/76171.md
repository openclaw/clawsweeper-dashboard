## Summary

Keep open. Current main and the latest release still leave CLI-backed isolated cron runs outside the existing run-end cleanup contract, the latest reporter still reproduces severe worker accumulation on v2026.5.22, and the overlapping MCP cleanup PR remains open and conflicting rather than a completed fix.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating after cron and agent turns, causing high host load, memory exhaustion, and very slow or stalled channel responses.

## Best Possible Solution

Reuse the existing run-end cleanup contract for one-shot isolated cron, heartbeat, and queued follow-up paths that can spawn CLI live sessions or bundled MCP runtimes, with regression coverage and real worker cleanup proof while preserving persistent session reuse.

## Reproduction Assessment

Yes, at source level. The issue provides concrete cron-load and latest-release logs, and current main still routes CLI-backed isolated cron runs through runCliAgent without the cleanup flags; I did not run the long host-load stress scenario.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing cleanup flags instead of adding a host reaper. The overlapping PR at https://github.com/openclaw/openclaw/pull/73919 needs refresh/repair and likely broader coverage before it can be the complete fix.
