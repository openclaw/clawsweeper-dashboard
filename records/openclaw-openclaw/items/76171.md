## Summary

Keep open. The report is fresh, detailed, and matches a plausible current-main gap: cron timeout cleanup exists, but successful CLI-backed agent child exits only dispose listeners and do not reap detached descendant process groups, which can leave stale workers behind on Docker-style hosts.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating after cron/agent turns on Railway, causing high host load and multi-minute response latency.

## Best Possible Solution

Add a targeted process-supervisor lifecycle fix that reaps detached CLI child process groups after terminal completion, with cron/CLI regression coverage and no broad cron scheduler rewrite.

## Reproduction Assessment

Yes. The report gives a concrete Railway cron-load reproduction, and current source supports a focused local harness: spawn a CLI child that exits while leaving a descendant in its process group, then verify the supervisor does not currently reap that descendant after a successful run.

## Solution Assessment

Unclear for the reporter's exact host until the survivor command line is confirmed, but the narrow maintainable direction is supervisor-owned detached child process-group cleanup rather than an external cleanup script or cron-level polling workaround.
