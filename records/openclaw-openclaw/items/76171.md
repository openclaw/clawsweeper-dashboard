## Summary

Keep open: current main and the latest release still show the isolated cron CLI branch using `runCliAgent` without the available run-end cleanup flags, while the docs and embedded branch say isolated cron should tear down bundled MCP resources.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating after cron/agent turns, causing high host load, memory pressure, slow responses, and gateway stalls.

## Best Possible Solution

Keep this issue open and fix the isolated cron CLI path so one-shot runs clean up CLI live-session and bundled MCP resources without closing a shared loopback server owned by another active run.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` still show the isolated cron CLI branch missing the cleanup flags while the cleanup contract exists and docs promise isolated cron runtime teardown. I did not run a live multi-cron process-count reproduction in this read-only review.

## Solution Assessment

No, current main is not yet the best solution because it only covers embedded isolated cron cleanup and outer session MCP retirement, leaving the CLI one-shot cleanup contract unwired. The safer fix is to propagate the cleanup flags with an ownership guard for shared MCP loopback reuse and add focused regression coverage.
