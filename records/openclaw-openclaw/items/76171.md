## Summary

Keep open. Current main has timeout cleanup, browser cleanup, and session-row reaping, but the cron CLI-provider path still does not show a completion-time cleanup path for stale live CLI/worker processes, and the issue has concrete production evidence plus corroborating comments.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating under Railway/Docker cron load, causing high host load and multi-minute response latency.

## Best Possible Solution

Add bounded lifecycle cleanup for completed isolated cron CLI worker processes, reusing existing cleanup controls where appropriate, and prove it with targeted unit coverage plus a Docker/Railway-style cron stress smoke.

## Reproduction Assessment

Yes. The issue gives a concrete production reproduction path on Railway/Docker with multiple cron jobs, a 30-60 minute accumulation window, worker counts, load averages, and manual-cleanup recovery; this read-only pass did not live-run that environment.

## Solution Assessment

No. The cleanup script workaround is not the best fix; the maintainable solution should live in isolated cron CLI lifecycle handling or process supervision, with guardrails for legitimate persistent sessions.
