## Summary

Keep open. Current main has one-shot cleanup controls for CLI live sessions and bundled MCP resources, but the isolated cron CLI path still does not pass those controls, while the report includes concrete production process-count/load evidence and follow-up confirmations from two other users.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating after frequent cron/agent turns on Railway/Docker, causing high host load and multi-minute response latency.

## Best Possible Solution

Wire isolated cron CLI runs into the existing one-shot cleanup contract, then prove that completed CLI cron turns close their live-session/MCP resources without changing persistent session semantics.

## Reproduction Assessment

Yes. The issue gives a concrete Railway/Docker reproduction path with worker counts, load averages, timing, and manual-cleanup recovery, and current source inspection shows a plausible CLI cron cleanup gap; I did not run a live Railway stress test in this read-only review.

## Solution Assessment

Yes, a narrow fix in the cron CLI handoff is the best first solution. It should use the existing cleanup flags rather than adding a broad host process sweeper or relying on an external cleanup script.

## Repair Work Prompt

Fix #76171 by wiring isolated cron CLI-provider runs into the existing one-shot cleanup contract. In `src/cron/isolated-agent/run-executor.ts`, when the `isCliProvider(...)` branch calls `runCliAgent`, pass cleanup flags for isolated cron runs: `cleanupCliLiveSessionOnRunEnd: params.job.sessionTarget === "isolated"` and `cleanupBundleMcpOnRunEnd: params.job.sessionTarget === "isolated"` unless code review shows a safer equivalent. Add or update focused tests proving an isolated CLI cron run passes both cleanup flags, while non-isolated/custom session cron behavior does not unexpectedly close persistent sessions. Do not add a host-wide process sweeper, do not kill arbitrary `openclaw` processes, and do not change process supervisor detachment policy unless separately justified by failing tests. Add a single-line `CHANGELOG.md` fix entry because this is user-facing latency/load behavior. Validate with focused tests around `src/cron/isolated-agent/run.session-key-isolation.test.ts`, `src/cron/isolated-agent/run.skill-filter.test.ts`, `src/agents/cli-runner.before-agent-reply-cron.test.ts`, and `src/agents/cli-runner.spawn.test.ts`; run `pnpm check:changed` in Testbox before handoff if available.
