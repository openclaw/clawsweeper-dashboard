## Summary

Keep open. Current main still has a concrete lifecycle gap for CLI-backed isolated cron runs: the embedded path opts into one-shot MCP cleanup, but the CLI path does not pass the existing cleanup flags that close live CLI sessions and bundle MCP resources after a temporary run.

## What This Changes

The issue requests that cron/agent-turn worker processes exit cleanly instead of accumulating stale OpenClaw workers and degrading host load/response time.

## Best Possible Solution

Wire existing one-shot CLI cleanup into isolated cron agent turns, add a focused regression proving the flags are passed, then validate with the Docker cron MCP cleanup/stress path before release.

## Reproduction Assessment

Yes. The issue gives a concrete Railway/Docker cron-rate reproduction with load averages, stale-worker counts, and recovery after manual cleanup; source inspection also gives a deterministic unit-level reproduction by asserting CLI isolated cron calls currently lack the cleanup flags.

## Solution Assessment

Yes, the narrowest maintainable fix appears to reuse the existing cleanupCliLiveSessionOnRunEnd and cleanupBundleMcpOnRunEnd flags for CLI-backed isolated cron runs rather than adding a generic worker-killing script.

In current main, `src/cron/isolated-agent/run-executor.ts` passes `cleanupBundleMcpOnRunEnd` for embedded isolated cron runs but does not pass `cleanupCliLiveSessionOnRunEnd` or `cleanupBundleMcpOnRunEnd` to `runCliAgent` in the CLI-provider branch. Reuse the existing `runCliAgent` cleanup flags for isolated cron one-shot runs; do not add a generic process-kill sweep, do not change main-session cron reuse semantics, and do not alter unrelated provider/session behavior. Add focused Vitest coverage that a CLI isolated cron run passes the cleanup flags while non-isolated continuation behavior remains intact.
