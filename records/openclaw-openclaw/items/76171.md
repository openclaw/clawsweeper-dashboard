## Summary

Keep open. Current main has timeout and isolated-run cleanup paths, but I did not find a source-backed lifecycle path that reaps completed-but-still-alive cron/CLI worker processes, and the report includes concrete production load/process-count evidence.

## What This Changes

The issue asks OpenClaw to stop accumulating stale worker processes after isolated cron or agent turns complete in Railway/Docker deployments.

## Best Possible Solution

Add bounded lifecycle cleanup for cron-launched CLI/agent worker processes, preserving legitimate persistent sessions, and prove it with an isolated-cron regression plus Docker/Railway-style stress validation.

## Reproduction Assessment

Yes. The issue gives a concrete production reproduction path on Railway/Docker with multiple cron jobs, a 30-60 minute accumulation window, worker counts, load averages, and recovery after manual cleanup; I did not live-run Railway in this read-only review.

## Solution Assessment

No. Current main's cleanup appears focused on timeouts, browser lifecycle, and embedded-run MCP cleanup, so the maintainable fix should be in cron agent handoff or process supervision rather than an external cleanup script.
