## Summary

Keep open. Current `main` still has the reported stale installed-plugin-index recovery gap: a disabled persisted plugin record can be accepted as a non-stale persisted registry, then later rehydrated during `includeDisabled` validation and surfaced as a fatal missing-manifest config error.

## What This Changes

The issue requests downgrade-safe handling for stale disabled plugin records in `plugins/installs.json`, plus clearer diagnostics and doctor repair when a bundled plugin is no longer present in the downgraded package.

## Best Possible Solution

Fix the generic plugin registry, config-validation, and doctor recovery path so missing disabled persisted records from `plugins/installs.json` are ignored or pruned before manifest loading, while enabled stale records still fail closed and diagnostics identify the state file, plugin id, and manifest path.

## Reproduction Assessment

Yes. Seed `plugins/installs.json` with a disabled plugin record whose `rootDir` or manifest path no longer exists, then run a config validation, status, or startup path that loads the plugin registry with `includeDisabled: true`; current source shows that record can still become a missing-manifest diagnostic.

## Solution Assessment

Yes. The narrow maintainable fix belongs in shared plugin registry/config-validation and doctor repair code, not in a `file-transfer` special case or a broad relaxation of manifest validation for enabled plugins.
