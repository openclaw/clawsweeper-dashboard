## Summary

Keep open. Current main still appears to allow a disabled stale installed-plugin-index record to survive the persisted-registry freshness check, then rehydrate it during config validation and fail on the missing manifest without identifying `plugins/installs.json` as the source.

## What This Changes

The issue reports that downgrading can leave a disabled stale `file-transfer` record in `plugins/installs.json` that blocks Gateway/CLI startup and misattributes the failure to `openclaw.json`.

## Best Possible Solution

Fix the generic plugin registry, config-validation, and doctor recovery path so missing disabled records from `plugins/installs.json` are ignored or pruned before manifest loading, while diagnostics name the state file, plugin id, and manifest path.

## Reproduction Assessment

Yes. Seed `plugins/installs.json` with a disabled record whose `rootDir` or `manifestPath` no longer exists, then exercise a config validation, status, or startup path that loads the registry with `includeDisabled: true`; current source shows that record can still reach manifest loading.

## Solution Assessment

Yes. The narrow maintainable direction is registry/index validation plus doctor cleanup and clearer diagnostics, not a `file-transfer` special case or a broad relaxation of manifest validation.
