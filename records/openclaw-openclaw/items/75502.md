## Summary

Keep open. Current main still has the stale installed-plugin-index recovery gap for disabled persisted records: the record can survive freshness checks, then get rehydrated for metadata/config validation and reported as a generic config problem instead of being pruned or attributed to `plugins/installs.json`.

## What This Changes

The issue requests that stale disabled bundled-plugin records in `~/.openclaw/plugins/installs.json` not block downgrade/startup and that diagnostics identify the installed-plugin index as the source.

## Best Possible Solution

Add a generic installed-index recovery path that ignores or prunes missing disabled records before manifest loading, while diagnostics and doctor name `plugins/installs.json`, the plugin id, and the missing manifest path.

## Reproduction Assessment

Yes. Seed `plugins/installs.json` with a disabled bundled record whose root or manifest path no longer exists, then run a metadata/config-validation startup path; current source shows that record can still be loaded with `includeDisabled: true`.

## Solution Assessment

Yes. The narrow maintainable fix is generic installed-index validation and doctor cleanup, not a `file-transfer` special case or relaxed validation for enabled plugins.
