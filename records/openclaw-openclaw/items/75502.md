## Summary

Keep open. Current main has partial stale managed-install cleanup, but the source path still allows a disabled persisted `plugins/installs.json` record with missing files to be reused for metadata/config validation and reported as a generic plugin config failure.

## What This Changes

The issue asks OpenClaw to tolerate or repair stale disabled bundled plugin records in `plugins/installs.json` after downgrading, and to point diagnostics at that state file instead of `openclaw.json`.

## Best Possible Solution

Add generic installed-index recovery that ignores or prunes missing disabled persisted records before any `includeDisabled` manifest load, while keeping enabled missing-plugin failures strict and naming `plugins/installs.json`, plugin id, and manifest path in diagnostics.

## Reproduction Assessment

Yes, from source inspection. Seed `plugins/installs.json` with a disabled persisted plugin record whose bundled root or manifest path no longer exists, then take a metadata/config-validation path that calls `loadPluginManifestRegistryForInstalledIndex({ includeDisabled: true })`.

## Solution Assessment

Yes, the right fix is a generic installed-index stale-record recovery path, not a `file-transfer` special case. The current main cleanup is useful but incomplete because it targets managed install records that match current bundled package metadata, not missing disabled persisted plugin-array records before manifest loading.
