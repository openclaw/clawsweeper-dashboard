## Summary

Keep open. Current `main` still matches the reported failure path: disabled persisted plugin-cache records are ignored by the stale-source guard, but config/status/doctor metadata paths later request disabled plugins and can rehydrate the missing manifest as a fatal config diagnostic. The reporter's clarification that the stale state lives in `plugins/installs.json` is consistent with the documented plugin index shape.

## What This Changes

Downgrading after `file-transfer` was installed should not let a disabled stale `plugins/installs.json` record block startup, status, or doctor repair, and diagnostics should name the real state file.

## Best Possible Solution

Add a generic plugin-registry recovery fix: treat missing persisted plugin files as stale even when the cached record is disabled, or skip only missing disabled cache records during manifest registry hydration while preserving durable install records. Pair that with a doctor repair/refresh path and diagnostics that identify `plugins/installs.json` as the source of the stale manifest path.

## Reproduction Assessment

Yes. Seed `plugins/installs.json` with a disabled `file-transfer` plugin cache entry pointing at a missing root or manifest, then run config validation/status/startup code that loads the installed-index registry with `includeDisabled: true`; current source shows it will attempt to load the missing manifest.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared plugin registry/config-validation and doctor recovery path, not in `file-transfer` itself or in a startup-time core migration keyed to that plugin id.
