## Summary

Keep open. Current main still has the downgrade recovery gap: a disabled persisted installed-plugin record with a missing manifest can survive stale-source checks, be rehydrated during metadata/config validation, and surface as a generic `plugins` config error instead of a `plugins/installs.json` repair.

## What This Changes

The issue requests downgrade-safe recovery for stale disabled bundled plugin records in `plugins/installs.json`, plus diagnostics that name the installed-plugin state file instead of only `openclaw.json`.

## Best Possible Solution

Add a generic installed-index recovery path that ignores or prunes missing disabled records before manifest loading, and make diagnostics/doctor name `plugins/installs.json`, the plugin id, and the missing manifest path.

Yes, from source inspection. Seed `plugins/installs.json` with a disabled record whose root or manifest path no longer exists, then run a metadata/config-validation startup path; current main can still load that disabled record with `includeDisabled: true`.

## Solution Assessment

Yes. The narrow maintainable fix is generic installed-index validation and doctor cleanup, not a `file-transfer` special case or relaxed validation for enabled plugins.
