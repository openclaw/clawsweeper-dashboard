## Summary

Keep open. Current main still has the reported stale plugin-index recovery gap: disabled persisted plugin records can avoid stale-source fallback, then config validation rehydrates disabled records and can turn a missing manifest from `plugins/installs.json` into a fatal config-facing error. The two user reports give a concrete downgrade path; the linked latency reports are related motivation but separate remaining work.

## What This Changes

The issue asks OpenClaw to tolerate or repair disabled stale `plugins/installs.json` plugin records after downgrades and to point diagnostics at the plugin index instead of only `openclaw.json`.

## Best Possible Solution

Fix the generic plugin registry, config-validation, and doctor recovery path so missing disabled persisted records from `plugins/installs.json` are ignored or pruned before manifest loading, while enabled stale records still fail closed and diagnostics identify the state file, plugin id, and manifest path.

## Reproduction Assessment

Yes. Seed the persisted plugin index with a disabled record whose `rootDir` or `manifestPath` no longer exists, then run a startup/status/config-validation path that loads the registry with `includeDisabled: true`; current source shows that candidate will still be loaded and reported as a missing manifest.

## Solution Assessment

Yes, the narrowest maintainable solution is in the shared plugin registry/config-validation and doctor repair paths. A `file-transfer` special case or broad manifest-validation relaxation would leave the same stale-index failure mode for other bundled or third-party plugins.

## Repair Work Prompt

Fix openclaw/openclaw#75502. Current main still trusts disabled stale persisted plugin index records: `hasMissingPersistedPluginSource` skips disabled records, config validation calls `loadPluginManifestRegistryForPluginRegistry` with `includeDisabled: true`, and installed-index reconstruction loads disabled records as manifest candidates. Implement generic recovery so missing disabled persisted records from `plugins/installs.json` are ignored or pruned before manifest loading/config validation/doctor repair; enabled missing records must still fail closed. Improve diagnostics to identify `plugins/installs.json`, plugin id, and manifest path for stale index records. Do not add a `file-transfer` special case, do not weaken manifest validation for enabled plugins, and keep third-party plugin index behavior compatible. Add focused tests for plugin registry snapshot fallback, installed-index manifest reconstruction, config validation diagnostics, gateway startup/status or doctor repair, and stale-plugin-config repair. Add a `CHANGELOG.md` fix entry crediting @lysopen and @WeipengMO.
