## Summary

Keep open. Current main still lets disabled stale persisted plugin-index records avoid the stale-source fallback, then rehydrates disabled records during config validation, so a missing manifest from `plugins/installs.json` can still become a fatal startup/doctor validation error with weak source attribution.

## What This Changes

The issue reports that downgrading from `2026.4.29` to `2026.4.27` leaves a disabled `file-transfer` install record in `plugins/installs.json`, causing startup, status, and doctor paths to fail with a misleading `openclaw.json` config error.

## Best Possible Solution

Fix the generic plugin-index recovery path so missing disabled persisted install records are ignored or pruned before config validation and doctor repair, diagnostics identify `plugins/installs.json` and the plugin id, and enabled stale plugin records still fail validation.

## Reproduction Assessment

Yes. Seed the persisted plugin index with a disabled record whose `rootDir` or `manifestPath` no longer exists, then run a validation/startup/status path that loads the registry with `includeDisabled: true`; current source shows it will attempt to load that missing manifest.

## Solution Assessment

Yes for the stale registry bug. The narrowest maintainable fix belongs in the shared plugin registry/config-validation and doctor repair path, not in a `file-transfer` special case or a change to the documented older-binary guard.

## Repair Work Prompt

Fix the stale disabled plugin-index downgrade/startup bug reported in https://github.com/openclaw/openclaw/issues/75502. Current main still allows disabled persisted records in `plugins/installs.json` to bypass stale-source fallback, then config validation rehydrates them with `includeDisabled: true` and can fail on a missing manifest. Implement a generic fix in the plugin index/registry/config-validation/doctor repair path: missing disabled persisted records should be ignored or pruned before fatal validation; missing enabled or configured plugin manifests must still fail closed; diagnostics should identify `plugins/installs.json` and the plugin id when stale persisted metadata is the source. Do not special-case `file-transfer`, do not weaken manifest validation for enabled plugins, and do not change the documented `OPENCLAW_ALLOW_OLDER_BINARY_DESTRUCTIVE_ACTIONS` guard without explicit maintainer approval. Add focused regression coverage for disabled stale persisted records and doctor/config validation recovery. Add a changelog entry if behavior changes are user-facing.
