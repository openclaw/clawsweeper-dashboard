## Summary

Keep open. Current main still has the reported generic recovery gap: a disabled persisted plugin-index record with missing files is not considered stale by the persisted-registry guard, but later config validation and doctor paths can still rehydrate disabled records with `includeDisabled: true` and turn the missing manifest into a fatal plugin diagnostic.

## What This Changes

The issue reports that a downgrade can leave a disabled stale `file-transfer` record in OpenClaw state `plugins/installs.json`, blocking startup and doctor with a misleading config-file error.

## Best Possible Solution

Fix the generic persisted plugin registry, config-validation, and doctor recovery paths so missing disabled records from state `plugins/installs.json` are ignored or pruned before manifest loading, while enabled missing manifests still fail closed and diagnostics name the state file, plugin id, and manifest path.

## Reproduction Assessment

Yes. Seed a valid persisted plugin index with a disabled record whose root or manifest path is gone, then run a config-validation, status, or doctor path that loads the plugin registry with `includeDisabled: true`; current source shows that candidate will still be manifest-loaded and reported as missing.

## Solution Assessment

Yes. The narrow maintainable fix is in shared plugin registry/config-validation/doctor recovery, not a `file-transfer` special case or a broad relaxation of manifest validation for enabled plugins.
