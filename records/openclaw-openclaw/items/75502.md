## Summary

Keep open. Current main still lets a disabled persisted installed-plugin record with missing files bypass stale-source detection, then later rehydrates it for metadata/config validation and reports the missing manifest as a generic plugin config error.

## What This Changes

The issue requests downgrade/startup recovery when `plugins/installs.json` contains a stale disabled bundled plugin entry whose manifest no longer exists.

## Best Possible Solution

Add generic installed-index recovery that ignores or prunes missing disabled records before manifest loading, and make doctor/config diagnostics name `plugins/installs.json`, the plugin id, and the missing manifest path.

## Reproduction Assessment

Yes, from source inspection and reporter-confirmed state. Seed `plugins/installs.json` with a disabled record whose root or manifest path no longer exists, then run a metadata/config-validation startup path that calls `loadPluginMetadataSnapshot` with `includeDisabled: true`.

## Solution Assessment

Yes. The narrow maintainable repair is generic installed-index validation/doctor cleanup with better diagnostics, not a `file-transfer` special case or relaxed validation for enabled plugins.
