## Summary

Keep open. Current main still has the reported installed-plugin-index recovery gap: disabled stale records can bypass stale-source detection, then later metadata/config validation rehydrates disabled entries and reports a missing manifest as a generic config problem.

## What This Changes

The issue requests downgrade-safe startup and doctor handling when `plugins/installs.json` contains a disabled stale bundled plugin record whose manifest no longer exists.

## Best Possible Solution

Add generic installed-index recovery that skips or prunes missing disabled records before manifest loading, reports `plugins/installs.json` with plugin id/path, and keeps enabled missing manifests strict.

Yes, from source inspection. Seed `plugins/installs.json` with a disabled record whose manifest path no longer exists, then run a metadata/config-validation startup path; current main can still include that disabled record and surface a generic missing-manifest config issue.

## Solution Assessment

Yes. The narrow maintainable solution is generic installed-index validation and doctor cleanup, not a `file-transfer` special case or relaxed validation for enabled plugins.
