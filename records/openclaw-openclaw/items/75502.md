## Summary

Keep open. Current main still lets disabled stale persisted plugin records bypass the stale-source fallback, then config validation rehydrates disabled records and can turn a missing manifest from the plugin index into a fatal config validation issue. The report is concrete, current, and narrow enough for a focused repair PR.

## What This Changes

The issue asks OpenClaw to tolerate or repair a disabled stale `file-transfer` entry in state-managed `plugins/installs.json` after downgrading from `2026.4.29` to `2026.4.27`, and to point diagnostics at the real state file.

## Best Possible Solution

Fix the generic plugin-index recovery path so missing disabled persisted `plugins` cache records are ignored or pruned before config validation and doctor repair, while enabled stale records still fail closed and diagnostics identify `plugins/installs.json` as the source of stale metadata.

## Reproduction Assessment

Yes. Seed the state plugin index with a disabled persisted record whose `rootDir` or `manifestPath` no longer exists, then run config validation or a startup/status path that loads the plugin registry with `includeDisabled: true`; current source shows it will attempt to load the missing manifest.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared plugin registry/config-validation and doctor repair path, not in a `file-transfer` special case or a startup-only migration.
