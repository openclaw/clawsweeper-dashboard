## Summary

Keep open. Current main still has the installed-plugin index recovery gap: a disabled persisted record whose files are missing can bypass stale-source detection, then be included in metadata/config validation and reported as a generic plugin config failure rather than as stale state from `plugins/installs.json`.

## What This Changes

Downgrade recovery should tolerate or repair stale disabled `file-transfer` entries in `plugins/installs.json` and report the real state file.

## Best Possible Solution

Add generic installed-index recovery that ignores or prunes missing disabled records before manifest loading, then make doctor and config diagnostics name `plugins/installs.json`, the plugin id, and the missing manifest path.

## Reproduction Assessment

Yes, source-reproducible. Seed `plugins/installs.json` with a disabled record whose `rootDir` or `manifestPath` no longer exists, then run a metadata/config-validation startup path; current main can still rehydrate that disabled record with `includeDisabled: true`.

## Solution Assessment

Yes. The narrow maintainable fix is generic installed-index validation plus doctor cleanup and better diagnostics, not a `file-transfer` special case or relaxed validation for enabled missing plugins.
