## Summary

Keep open. The stale downgrade path is still source-backed on current main: disabled persisted plugin records are skipped by stale-source detection, but later rehydrated for config validation with disabled entries included, so a missing manifest from `plugins/installs.json` can still become a fatal registry diagnostic. The latency reports are useful related context, but this item tracks a narrower startup/doctor recovery bug.

## What This Changes

The issue requests startup and doctor recovery when a downgrade leaves a disabled bundled plugin record in `plugins/installs.json` for a plugin the older package no longer ships.

## Best Possible Solution

Fix the generic persisted plugin registry and config-validation path so missing disabled records from `plugins/installs.json` are ignored or pruned before manifest loading, while enabled stale records still fail closed. Doctor should be able to repair the stale state before full validation aborts, and diagnostics should name `plugins/installs.json` when that file supplied the bad metadata.

## Reproduction Assessment

Yes. Seed `plugins/installs.json` with a disabled bundled record whose `rootDir` or `manifestPath` no longer exists, then run a config validation, status, or gateway startup path that loads the registry with `includeDisabled: true`; the current source shows that record is rehydrated and manifest-loaded.

## Solution Assessment

Yes. The narrow maintainable fix is generic installed-index stale-record handling plus doctor pre-validation repair, not a `file-transfer` special case or weakened validation for enabled plugins.
