## Summary

Keep open. Current main still has a source-visible recovery gap: disabled persisted installed-plugin records with missing files can bypass stale-source detection, then be included in metadata manifest hydration and surface as a generic config/plugin failure instead of being ignored or pruned.

## What This Changes

The issue requests downgrade/startup recovery for stale disabled bundled plugin entries in `plugins/installs.json`, with diagnostics that name the real state file.

## Best Possible Solution

Add generic installed-index recovery that ignores or prunes missing disabled records before manifest loading, and make doctor/startup diagnostics name `plugins/installs.json`, the plugin id, and the missing manifest path.

## Reproduction Assessment

Yes, from source inspection. Seed `plugins/installs.json` with a disabled record whose root or manifest path no longer exists, then run a metadata/config-validation startup path that calls `loadPluginMetadataSnapshot`; current main can still pass that record into manifest loading with `includeDisabled: true`.

## Solution Assessment

Yes. The narrow maintainable fix is generic installed-index validation and doctor cleanup, not a `file-transfer` special case or relaxed validation for enabled plugins.
