## Summary

Keep open. Current main still has the reported downgrade failure path: stale disabled records from the persisted plugin install index can remain selected, config validation includes disabled plugin records, missing manifests become registry errors under the generic `plugins` path, and gateway startup recovery only degrades issues scoped to `plugins.entries.*`.

## What This Changes

The issue asks OpenClaw to tolerate or repair stale disabled bundled-plugin entries in the managed plugin install index after downgrades and to point diagnostics at `plugins/installs.json`.

## Best Possible Solution

Fix the generic plugin install-index recovery path so missing manifests for disabled persisted records are inert or pruned before config validation can block startup, status, or doctor repair, and improve the diagnostic to name the managed install index when that state file supplied the missing manifest path.

## Reproduction Assessment

Yes. Seed `plugins/installs.json` with a disabled installed-index record whose `rootDir` and manifest no longer exist, then run a config-validation or startup path that loads the plugin registry with `includeDisabled: true`; the current source path will attempt to load the missing manifest and report a config issue.

## Solution Assessment

Yes, the narrow maintainable solution is to treat missing disabled persisted plugin-cache records as stale metadata in the registry/doctor path and to improve diagnostics for index-sourced manifest failures. A startup-time migration tied to `file-transfer` would be less maintainable and would violate the generic plugin boundary.
