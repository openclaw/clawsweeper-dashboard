## Summary

Keep open. The report maps to a current-main plugin registry/config-validation path: disabled persisted plugin-index records can still be rehydrated during validation, missing manifests become fatal config issues, and the diagnostic path does not identify `plugins/installs.json` as the source state file.

## What This Changes

The issue asks OpenClaw to tolerate or repair stale disabled bundled-plugin records in `~/.openclaw/plugins/installs.json` after downgrading, and to report the real state-file source for missing manifests.

## Best Possible Solution

Keep this open and fix the core plugin-index recovery path so stale disabled persisted records from older/newer bundled-plugin inventories cannot block startup, status, or doctor repair, while preserving install metadata needed by update/uninstall diagnostics.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes. The narrow maintainable solution is in the plugin registry/config-validation and doctor repair path: treat missing disabled persisted plugin-cache records as stale or inert during validation, and improve diagnostics to name `plugins/installs.json` when that state file supplied the missing manifest path.

## Repair Work Prompt

Fix openclaw/openclaw issue #75502. Current main can fail config validation/startup after downgrade when `~/.openclaw/plugins/installs.json` contains a persisted `plugins` cache entry for a missing bundled plugin such as `file-transfer` with `enabled:false`. The current path is: `loadPluginRegistrySnapshotWithMetadata` ignores missing files for disabled persisted records, `validateConfigObjectWithPlugins` requests `includeDisabled:true`, `loadPluginManifestRegistryForInstalledIndex` rehydrates the disabled record into a candidate, and `loadPluginManifest` emits a fatal missing-manifest diagnostic that points at config instead of the plugin index state. Make stale disabled persisted plugin-cache records inert or stale/derived during validation and startup, and ensure doctor/registry refresh can repair or prune them without losing durable `installRecords`. Improve the diagnostic when the missing manifest came from `plugins/installs.json`. Likely files: `src/plugins/plugin-registry-snapshot.ts`, `src/plugins/manifest-registry-installed.ts`, `src/config/validation.ts`, `src/plugins/installed-plugin-index-store.ts`, doctor/registry repair files if needed, and focused tests. Add a user-facing `CHANGELOG.md` fix entry with proper reporter credit. Do not change file-transfer path policy, plugin execution permissions, or broad plugin API contracts.
