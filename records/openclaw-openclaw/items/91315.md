## Summary

Keep open: current main and v2026.6.8 still have the reported embedded spawn gate, but this PR is not merge-ready because it exposes `sessions_spawn` from default config without proving local embedded execution and without mirroring per-agent allowlists.

## What This Changes

The PR changes `createOpenClawTools` so embedded sessions include `sessions_spawn` when `agents.defaults.subagents.allowAgents` is non-empty.

## Best Possible Solution

Gate embedded `sessions_spawn` on the same effective per-agent/default allowlist used by runtime validation and expose it only where a gateway or in-process dispatch path is present, or add local dispatch with focused tests and real TUI proof.

## Reproduction Assessment

Yes for a source-level path: current main sets local TUI into embedded mode and excludes `sessions_spawn` unless `allowGatewaySubagentBinding` is true. I did not run a live Windows/npm TUI reproduction.

## Solution Assessment

No. `createOpenClawTools` is a plausible materialization layer, but the condition should mirror effective requester policy and a supported spawn execution context rather than exposing a Gateway-dependent tool from default config alone.
