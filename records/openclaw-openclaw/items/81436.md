## Summary

Keep open; the PR is useful but not merge-ready because YAML edits on malformed YAML can throw through the CLI, and the restored YAML/.lobster capability plus plugin-local dependency still needs maintainer approval.

## What This Changes

The PR restores YAML/YML/.lobster support in the bundled oc-path plugin, adds `$first` positional addressing, updates docs/changelog, and adds YAML tests plus a plugin-local `yaml` dependency.

## Best Possible Solution

Fix malformed-YAML handling so parse errors return structured failures, then merge the plugin-local YAML adapter only if maintainers accept the restored scope and dependency.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review finding by source inspection: `parseYaml` can return a document with errors, and `yaml@2.9.0` throws when that document is stringified during edit cloning. For the feature request itself, reproduction is not applicable because current main intentionally lacks YAML support.

## Solution Assessment

Is this the best way to solve the issue? Not as-is; using the upstream YAML Document API inside the oc-path plugin is the right boundary if maintainers accept the feature, but malformed YAML must be rejected before edit/emit paths can throw raw exceptions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject invalid YAML before editing:** `extensions/oc-path/src/oc-path/yaml/edit.ts:201`
  - body: `parseYaml` records diagnostics but still returns a `Document` with `errors`, and `setYamlOcPath` later clones it via `doc.toString()`. In `yaml@2.9.0`, `Document.toString()` throws when errors are present, so `openclaw path set` on a malformed YAML file can crash instead of returning the structured `parse-error`/failure shape used by the other adapters. Gate YAML edits on parse diagnostics or convert this path into a structured failure before cloning.
  - confidence: 0.88
