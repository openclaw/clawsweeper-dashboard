## Summary

Keep open: this is an active draft, maintainer-labeled feature PR, and latest head fixes the earlier YAML value/empty-sequence findings but still has a YAML keyed-insertion sentinel bypass plus missing real behavior proof. Current main intentionally documents oc-path as markdown/jsonc/jsonl only, so re-adding YAML also needs maintainer scope approval.

## What This Changes

The PR adds YAML/YML/.lobster support to the oc-path plugin across parse, resolve, find, set, emit, tests, docs, changelog, and adds `$first` positional addressing plus a plugin-local `yaml` dependency.

## Best Possible Solution

Accept YAML support only after keyed writes preserve the sentinel guard, real CLI proof is supplied, and maintainers approve restoring YAML/.lobster and `$first` in oc-path.

## Reproduction Assessment

Yes for the review finding by source inspection: a keyed YAML insertion can write `marker.key` without scanning it, and default `emitYaml` returns edited `ast.raw`. I did not execute the PR branch in this read-only review.

## Solution Assessment

No, not as-is. The narrow code fix is to guard keyed insertion markers or strictly scan edited YAML bytes before returning them, followed by real behavior proof and maintainer approval for the expanded oc-path scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard YAML insertion keys before writing:** `extensions/oc-path/src/oc-path/yaml/edit.ts:64`
  - body: `insertYamlOcPath` only scans `newValue`, but map insertion writes `marker.key` through `cloned.setIn([...segments, marker.key], newValue)` and then returns rendered YAML as raw bytes. A path like `oc://x.yaml/config/+__OPENCLAW_REDACTED__` can therefore dry-run or write the sentinel literal as a key, bypassing the oc-path sentinel contract; guard keyed markers or scan the edited bytes before returning.
  - confidence: 0.9
