## Summary

Keep open: this draft, maintainer-labeled feature PR has sufficient real CLI proof and the prior sentinel object-key concern appears fixed, but latest head still misreports YAML file-root insertion for sequence and scalar roots.

## What This Changes

The PR adds YAML/YML/.lobster support to the bundled oc-path CLI/substrate, adds `$first` positional addressing, updates tests/docs/changelog, and adds the plugin-local `yaml` dependency.

## Best Possible Solution

If maintainers accept the restored YAML scope, fix the root insertion resolver to branch on YAML root map/sequence/scalar, add focused coverage, and keep the recursive sentinel guard from latest head.

## Reproduction Assessment

Yes for the review finding by source inspection: a file-root YAML insertion path resolves as `root`, and `resolveYamlInsertion` returns `yaml-map` without inspecting the root node kind. I did not execute the PR branch in this read-only review.

## Solution Assessment

No, not as-is. The narrow repair is to inspect `ast.doc.contents` with `isMap`/`isSeq` for YAML root insertion and return null for scalar roots, while maintainer review decides the broader YAML/.lobster and `$first` scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Check YAML root kind before resolving insertion:** `extensions/oc-path/src/oc-path/universal.ts:337`
  - body: `resolveYamlInsertion` treats a file-only YAML insertion target as a `yaml-map` whenever `resolveYamlOcPath` returns `root`, without inspecting `ast.doc.contents`. A root sequence such as `- a\n` would make `openclaw path resolve 'oc://items.yaml/+'` report a map insertion point, and a root scalar would report an insertion point even though `set` cannot write there. Branch on `isMap`/`isSeq` and return null for scalar roots.
  - confidence: 0.88
