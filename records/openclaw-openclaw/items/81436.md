## Summary

Keep open: this is an active maintainer-labeled feature PR with sufficient real CLI proof, but the latest head still has a source-reproducible malformed-YAML edit crash that should block merge.

## What This Changes

The PR restores YAML/YML/.lobster support in the bundled oc-path plugin, adds `$first` addressing, updates docs and changelog text, adds YAML tests, and declares a plugin-local `yaml` dependency.

## Best Possible Solution

Keep the YAML restoration inside the oc-path plugin boundary, reject YAML parse-error documents before edit/emit paths stringify them, then have maintainers approve the restored scope and dependency delta.

## Reproduction Assessment

Yes for the review finding by source inspection: malformed YAML can produce `doc.errors`, and the PR edit path calls `Document.toString()` on that document even though `yaml@2.9.0` throws in that state. For the YAML restoration itself, current main does not document YAML support, so it is feature-scope rather than a current-main bug reproduction.

## Solution Assessment

No, not as-is: using the plugin-local YAML document API is the right implementation boundary if maintainers accept the feature, but malformed YAML must return a structured `parse-error` before cloning or emitting. The safer fix is to gate `ast.doc.errors` in YAML set/insert paths and cover it in YAML and CLI tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject invalid YAML before editing:** `extensions/oc-path/src/oc-path/yaml/edit.ts:201`
  - body: `parseYaml` records diagnostics but still returns a `Document`, and `setYamlOcPath` later clones it with `doc.toString()`. In `yaml@2.9.0`, `Document.toString()` throws when parse errors are present, so `openclaw path set` on malformed YAML can crash instead of returning the structured `parse-error` shape used by the CLI. Gate YAML edits on parse diagnostics before cloning or convert this path into a structured failure.
  - confidence: 0.9
