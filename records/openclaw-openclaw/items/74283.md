## Summary

Keep open. Current main still has the direct approval follow-up fallback path that can emit unmatched `other` result text after a failed session resume, and the existing tests do not cover source-like unmatched text. Because the report describes a channel-delivery information leak, it needs maintainer/security follow-up rather than cleanup closure.

## What This Changes

The issue requests a guard against raw internal handler/source-like text being delivered through the exec approval follow-up direct fallback after session resume fails.

## Best Possible Solution

Add a fail-closed guard in the exec approval direct fallback so only recognized completion/denial payloads or otherwise verified safe reply text can be delivered directly, then trace and repair the producer that can supply source-like `resultText`. Add regression coverage for source-like unmatched payloads in the exec approval follow-up tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, a deliverable Telegram route, a rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `other` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Yes for immediate containment: a fail-closed direct-fallback guard is the narrowest maintainable fix for the channel delivery boundary. The producer repair should be handled in the same PR if quickly found, or as a linked follow-up after the leak boundary is closed.
