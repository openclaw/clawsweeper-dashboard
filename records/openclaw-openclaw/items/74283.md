## Summary

Keep open. Current main still has the approval-completion to session-resume to direct-fallback path described in the report: unmatched follow-up text is parsed as raw `kind: "other"`, passed through only the generic user-facing sanitizer, and sent to the originating channel when the follow-up agent resume fails. The issue and @martingarramon's comment correctly distinguish this from the separate #72184 delivery-queue restart recovery path.

## What This Changes

The issue requests hardening exec approval follow-up recovery so raw internal handler/source-like `resultText` cannot be delivered directly to Telegram after session resume failure.

## Best Possible Solution

Add a fail-closed guard at the exec approval direct-fallback boundary so only recognized structured completion/denial payloads, or otherwise verified safe reply text, can be delivered directly. Then trace the producer path and add regression coverage for the reported JS-like snippet shapes so internal handler state cannot become `resultText`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock `callGatewayTool("agent", ...)` to throw, call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, and JS-like unmatched `resultText`, and observe current main parse it as `kind: "other"` before sending the fallback prefix plus sanitized raw text.

## Solution Assessment

Unclear for the complete two-layer proposal. The fail-closed direct fallback guard is the narrow immediate containment, but the durable solution should also repair the producer that populates `resultText` with internal state.
