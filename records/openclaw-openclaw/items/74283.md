## Summary

Keep open. Current main still has the approval follow-up session-resume fallback path described in the report: unmatched result text is parsed as `other`, sanitized generically, and sent directly after resume failure, so this remains distinct from the #72184 recovery path.

## What This Changes

The issue asks OpenClaw to stop approval follow-up fallback delivery from posting raw internal handler/source-like text to Telegram when session resume fails.

## Best Possible Solution

Fail closed for unmatched/source-like approval result text before direct channel delivery, add regression coverage for the reported snippets, then trace and repair the producer that creates malformed `resultText`.

## Reproduction Assessment

Yes. A focused code-level reproduction is to mock `callGatewayTool("agent", ...)` to throw, call `sendExecApprovalFollowup` with a Telegram delivery route plus session key, and pass JavaScript-like unmatched `resultText`; current main can send that text via the direct fallback.

## Solution Assessment

Yes for immediate containment: guarding `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback` is the narrowest maintainable boundary fix. The producer-side repair should follow once maintainers identify where source-like `resultText` is created.
