## Summary

Keep open. Current main still has the approval-completion -> session-resume -> direct-fallback path described in the report, and unmatched approval result text can still be delivered after generic sanitization; this remains distinct from the #72184 delivery-recovery path.

## What This Changes

The issue asks OpenClaw to prevent exec-approval follow-up recovery from posting source-like internal handler text to chat when session resume fails.

## Best Possible Solution

Add a fail-closed guard in the exec approval direct fallback for unmatched `other` result text, then repair the producer that creates source-like `resultText`, with regression coverage for the reported snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a Telegram delivery route, a session key, a rejected `callGatewayTool("agent", ...)`, and JavaScript-like unmatched `resultText`; current main parses it as `other` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Yes for immediate containment: guarding `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback` is the narrowest delivery-boundary fix. The producer-side repair should follow once maintainers trace where the source-like result text is created.
