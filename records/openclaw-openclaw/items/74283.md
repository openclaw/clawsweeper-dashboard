## Summary

Keep open. Current main still has the reported approval-follow-up failure path: unexpected approval result text is classified as unstructured text and can be delivered by the direct fallback after session resume fails, while the newer delivery-recovery fixes address different retry/recovery surfaces rather than this boundary.

## What This Changes

The issue requests a guard and producer fix for `/approve` post-approval follow-up recovery so raw internal handler/source-like text is not sent to Telegram when session resume fails.

## Best Possible Solution

Add fail-closed containment at the exec approval direct fallback for unverified or source-like `other` payloads, then repair the producer so approval follow-up result text is either a known approval result or empty, with regression tests covering source-like snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a deliverable Telegram route, a session key, a throwing `callGatewayTool("agent", ...)`, and JavaScript-like unmatched `resultText`; current main routes that text through the `other` branch into direct fallback.

## Solution Assessment

Yes for the issue’s proposed two-layer direction: a fail-closed delivery-boundary guard is the narrow containment fix, and producer cleanup is the durable fix once maintainers confirm where the internal text is created.
