## Summary

Keep open. Current main still sends unmatched approval follow-up `resultText` through the direct external fallback after session-resume failure, while the related open prefix/retry issue covers adjacent UX/retry behavior rather than the source-like payload guard reported here.

## What This Changes

The issue reports that exec approval follow-up fallback delivery can send raw internal handler/source-like text to Telegram after session resume fails.

## Best Possible Solution

Add a fail-closed guard at the direct approval follow-up delivery boundary for unmatched/source-like text, cover the reported snippets with regression tests, and separately trace the producer that creates unsafe `resultText`.

## Reproduction Assessment

Yes, source-level: call `sendExecApprovalFollowup` with a deliverable Telegram route, a session key, a rejected `callGatewayTool("agent", ...)`, and source-like unmatched `resultText`; current main parses it as `other` and sends the sanitized raw text through the direct fallback.

## Solution Assessment

Yes for containment: the narrow maintainable boundary is `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback`, where unverified `other` text can be dropped or replaced with a generic internal-log-only stub before external delivery.
