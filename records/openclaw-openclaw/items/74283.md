## Summary

Keep open. Current main still has the approval-completion to session-resume to direct-fallback path described in the report: unmatched `resultText` is parsed as raw `kind: "other"`, sanitized only by the generic user-facing text sanitizer, and then sent directly to the originating channel when the follow-up agent resume fails. The report and follow-up comment also correctly distinguish this from #72184, which covered a different restart recovery path.

## What This Changes

Harden the exec approval post-approval follow-up fallback so raw internal handler/source-like text cannot be delivered to Telegram or other channel recipients after session resume failure.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only recognized structured completion/denial payloads, or otherwise verified safe user-facing text, can be delivered directly after session resume failure. Then trace and fix the producer path so internal handler state cannot become `resultText`, with regression coverage for the reported JS-like snippet shapes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text through the direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The Layer 1 fail-closed delivery guard is the narrow immediate containment, but the complete solution should also repair the upstream producer so internal handler state cannot enter `resultText`.
