## Summary

Keep open. Current main and the latest release still allow unmatched approval follow-up text to flow through the direct external fallback after session-resume failure, and the related open prefix/retry issue does not cover the source-like payload guard reported here.

## What This Changes

The issue reports that approval follow-up delivery can expose raw internal handler/source-like text through the direct fallback when agent session resume fails.

## Best Possible Solution

Add a fail-closed guard at the direct approval follow-up delivery boundary for unmatched/source-like text, cover the reported snippets with regression tests, and separately trace the producer that creates unsafe `resultText`.

## Reproduction Assessment

Yes, source-level: call `sendExecApprovalFollowup` with a deliverable Telegram route, a session key, a rejected `callGatewayTool("agent")`, and source-like unmatched `resultText`; current main parses it as `other` and can send the sanitized raw text through direct fallback. I did not run a live Telegram reproduction in the read-only checkout.

## Solution Assessment

Yes for containment: the narrow maintainable boundary is `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback`, where unverified `other` text can be dropped or replaced before external delivery. The producer that generated source-like `resultText` should be traced as a separate root-cause follow-up.
