## Summary

Keep open. Current main still has the approval-completion to session-resume to direct-fallback path described in the report: unrecognized approval result text is parsed as raw `other` text, sanitized only by the generic user-facing sanitizer, and delivered directly to the originating channel when the follow-up agent resume fails. The discussion correctly distinguishes this from #72184, which covered a different delivery-recovery restart path.

## What This Changes

Request fail-closed handling for exec approval post-approval direct fallback so unrecognized or internal handler `resultText` is not delivered to Telegram after session resume failure.

## Best Possible Solution

Add a fail-closed guard in the exec approval direct fallback so only recognized completion/denial payloads or otherwise verified safe reply text can be delivered directly, then trace and repair the producer that can populate `resultText` with handler state. Add regression coverage for source-like snippets in the exec approval follow-up tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Yes for immediate containment: a fail-closed direct-fallback guard is the narrowest maintainable fix for the delivery boundary. The producer repair should be a follow-up or same PR once the source of source-like `resultText` is identified.
