## Summary

Keep open. Current main still has the approval-completion to session-resume to direct-fallback path described in the report: unrecognized exec approval result text falls through as raw `other` text, receives only generic user-facing sanitization, and can be sent directly to the originating channel when the follow-up agent resume fails. The discussion correctly distinguishes this from #72184, which covered the delivery-queue restart recovery path rather than this exec approval follow-up fallback.

## What This Changes

Requests hardening the exec approval post-approval follow-up fallback so raw internal handler/source text is not sent to Telegram when session resume fails.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only recognized completion/denial payloads or otherwise verified safe reply text can be sent directly after session resume failure, then trace and repair the producer that lets internal handler state become `resultText`. Add focused regression tests for source-like `other` payloads in the exec approval follow-up tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text via direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal, but the immediate fail-closed direct-fallback guard is the narrowest maintainable containment. The producer repair should follow once the source of source-like `resultText` is identified.
