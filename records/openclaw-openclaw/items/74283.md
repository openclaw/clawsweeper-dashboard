## Summary

Keep open. Current main still allows the reported approval-completion -> session-resume -> direct-fallback path to deliver unrecognized follow-up text to the originating channel after the follow-up agent resume fails, and the provided issue discussion correctly distinguishes this from the earlier restart delivery-recovery fix.

## What This Changes

Prevent post-approval exec follow-up direct fallback delivery from sending raw internal or source-like handler text to Telegram after automatic session resume fails.

## Best Possible Solution

Add a fail-closed exec approval direct-fallback boundary that only delivers recognized completion/denial payloads or verified safe agent reply text; use generic unavailable copy for unsafe or unrecognized text, then trace and fix the producer path with regression coverage for the reported snippet shapes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendExecApprovalFollowup with a session key, a deliverable Telegram route, a rejected callGatewayTool("agent", ...), and JavaScript-like unstructured resultText; current main parses it as kind "other" and sends the sanitized raw text through the direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The direct fallback guard is the narrow immediate containment, while the producer repair should follow once maintainers identify where internal-state text enters resultText.
