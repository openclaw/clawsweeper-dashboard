## Summary

Keep open. Current main still has the approval-completion to session-resume to direct-fallback path described in the report: unrecognized `resultText` is parsed as `kind: "other"`, sanitized only by the generic user-facing sanitizer, and delivered directly to the originating channel if the follow-up agent resume fails. The provided follow-up comment from martingarramon matches the current code and correctly distinguishes this from #72184, which is the delivery-queue restart recovery path.

## What This Changes

The issue asks OpenClaw to prevent `/approve` exec follow-up fallback from delivering raw internal handler/source-like `resultText` and to repair the upstream producer path that creates that text.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only recognized completion/denial payloads or otherwise verified safe reply text can be sent directly after session resume failure, then trace and repair the producer that lets internal handler state become `resultText`. Add regression coverage for source-like snippets in the exec approval follow-up tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text via direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal, but the immediate fail-closed direct-fallback guard is the narrowest containment. The producer repair should follow once the source of source-like `resultText` is identified.
