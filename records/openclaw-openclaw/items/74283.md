## Summary

Keep open. Current main still has the approval-completion session-resume fallback path described in the report, and unmatched approval result text can still be delivered directly after generic sanitization; the discussion correctly distinguishes this from the older #72184 recovery-queue fix.

## What This Changes

The issue reports that exec approval follow-up fallback can send source-like internal handler text to Telegram after session resume fails.

## Best Possible Solution

Add a fail-closed guard for unsafe unmatched approval result text in the direct fallback boundary, then trace and fix the producer that creates source-like resultText, with regression coverage for the reported snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendExecApprovalFollowup with a Telegram delivery route, a session key, a rejected callGatewayTool("agent", ...), and JavaScript-like unmatched resultText; current main parses it as kind "other" and sends the sanitized raw text through direct fallback.

## Solution Assessment

Yes for immediate containment: guarding formatDirectExecApprovalFollowupText or sendDirectFollowupFallback is the narrowest maintainable delivery-boundary fix. The producer-side repair should follow so resultText is either a verified agent reply/result shape or empty.
