## Summary

Keep open. Current main still has the approval-completion to session-resume to direct-fallback path described in the report, and unrecognized `resultText` can still be sanitized generically and sent directly to the originating channel after the follow-up agent resume fails. This remains distinct from #72184, which covered delivery-queue restart recovery rather than the exec approval follow-up fallback.

## What This Changes

The issue requests fail-closed hardening for exec approval post-approval fallback delivery so raw internal handler/source-like snippets are not sent to Telegram when session resume fails.

## Best Possible Solution

Add a fail-closed boundary around direct exec approval fallback delivery so only structured, recognized approval completion/denial payloads or otherwise provenance-verified user-facing text can be delivered directly. Then fix the producer path that can populate `resultText` with internal handler state, with regression coverage for the reported source-like snippet shapes.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to mock `callGatewayTool("agent", ...)` to reject, call `sendExecApprovalFollowup` with a Telegram route and JavaScript-like unstructured `resultText`, and assert current main sends the fallback prefix plus sanitized raw text through `sendMessage`.

## Solution Assessment

Yes for the two-layer direction. The narrow containment is a fail-closed direct-delivery guard, while the more complete fix should also repair the upstream producer so internal handler state cannot enter `resultText`.
