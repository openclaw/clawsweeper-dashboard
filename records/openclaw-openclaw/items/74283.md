## Summary

Keep open. Current main still has the approval-completion -> session-resume -> direct fallback path described in the report: unmatched follow-up text is parsed as `kind: "other"`, passed through only the generic user-facing sanitizer, and sent to the originating channel when the follow-up agent resume fails. The reporter and @martingarramon correctly distinguish this from #72184, which covered the separate delivery-queue restart recovery path.

## What This Changes

The issue requests hardening of exec approval follow-up recovery so raw internal handler/source-like `resultText` cannot be delivered directly to Telegram after session resume failure.

## Best Possible Solution

Add a fail-closed guard at the exec approval direct-fallback boundary so only recognized structured completion/denial payloads or verified safe reply text can be sent directly. Then trace the producer path and add regression coverage for the reported JS-like snippet shapes so internal handler state cannot enter `resultText` in the first place.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock `callGatewayTool("agent", ...)` to throw, call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, and JS-like unmatched `resultText`, and observe current main parse it as `kind: "other"` before sending the fallback prefix plus sanitized raw text.

## Solution Assessment

Unclear for the full two-layer proposal. The fail-closed direct fallback guard is the narrow immediate containment, but the durable fix should also repair the producer that populates `resultText` with internal state.
