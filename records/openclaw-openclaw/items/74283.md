## Summary

Keep open. Current main still routes unmatched exec approval result text through the session-resume direct fallback after generic sanitization, so the reported `/approve` follow-up leak remains a distinct path from the closed delivery-recovery issue.

## What This Changes

The issue asks OpenClaw to prevent `/approve` post-approval fallback delivery from dumping source-like internal handler text into Telegram when session resume fails.

## Best Possible Solution

Add a fail-closed guard at the exec approval direct-delivery boundary for unmatched/source-like result text, with regression coverage for the reported snippets, then separately trace and repair the malformed `resultText` producer.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a Telegram delivery route, a session key, a rejected `callGatewayTool("agent", ...)`, and source-like unmatched `resultText`; current main formats that `other` text and sends it through the direct fallback.

## Solution Assessment

Yes for immediate containment: guarding `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback` is the narrowest maintainable delivery-boundary fix. The producer-side repair should follow once maintainers identify where source-like `resultText` is created.
