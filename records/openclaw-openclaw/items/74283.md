## Summary

Keep open. Current main still has the reported approval-completion to session-resume to direct-fallback path: unrecognized approval result text becomes raw `other` text, receives only generic user-facing sanitization, and can be sent directly to the channel after the follow-up agent resume fails.

## What This Changes

The issue requests preventing `/approve` post-approval follow-up fallback delivery from exposing raw internal handler/source-like text to Telegram after session resume failure.

## Best Possible Solution

Add a fail-closed guard in the exec approval direct fallback so only recognized completion/denial payloads or otherwise verified safe reply text can be delivered directly; replace unsafe/unrecognized source-like text with a generic unavailable message and add focused regression coverage, then repair the upstream producer that creates source-like `resultText`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, a rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `other` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Yes for immediate containment: a fail-closed direct-fallback guard is the narrowest maintainable fix for the channel-delivery boundary. The producer repair should follow in the same PR or a linked follow-up once the source of source-like `resultText` is identified.
