## Summary

Keep open. Current main still has the reported approval-completion -> session-resume -> direct-fallback path: unrecognized `resultText` is parsed as raw `kind: "other"`, sanitized only by the generic user-facing sanitizer, and sent to the originating channel when the follow-up agent resume fails. The issue and follow-up comment correctly distinguish this from #72184, which covered delivery-queue restart recovery rather than exec approval follow-up fallback.

## What This Changes

Prevent post-approval exec follow-up fallback delivery from sending raw internal handler/source-like text to Telegram after automatic session resume fails.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only recognized completion/denial payloads or otherwise verified safe reply text can be delivered directly after session resume failure. Then trace the producer path that lets internal handler state become `resultText` and add regression coverage for the reported source-like snippet shapes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, a deliverable Telegram route, a rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text through the direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The Layer 1 fail-closed direct-fallback guard is the narrow immediate containment, while the producer repair should follow once the source of internal-state `resultText` is identified.
