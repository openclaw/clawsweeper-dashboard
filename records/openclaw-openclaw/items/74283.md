## Summary

Keep open. Current main still allows unmatched approval follow-up text to reach the direct external fallback after a session-resume failure, and the related open prefix/retry issue does not cover the source-like `resultText` guard reported here.

## What This Changes

The issue asks OpenClaw to stop `/approve` post-approval follow-up recovery from sending source-like internal handler text to Telegram when session resume fails.

## Best Possible Solution

Add a fail-closed direct-delivery guard for unmatched approval follow-up text, add regression coverage for the reported source-like snippets, then separately trace the producer that can populate `resultText` with internal handler state.

## Reproduction Assessment

Yes, source-level: call `sendExecApprovalFollowup` with a deliverable Telegram route, a session key, a rejected `callGatewayTool("agent", ...)`, and source-like unmatched `resultText`; current main parses it as `other` and sends the sanitized raw text through the direct fallback. I did not run a live Telegram reproduction.

## Solution Assessment

Yes for containment: the narrow maintainable boundary is `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback`, where unverified `other` text can be dropped or replaced with a generic internal-log-only stub before external delivery.
