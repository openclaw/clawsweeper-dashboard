## Summary

Keep open. The patch appears to address the linked P1 webchat session-state failure with a narrow runner-to-gateway lifecycle marker and no blocking code findings, but the external PR proof is mock-only and the exact live provider/webchat failure remains unproven before merge.

## What This Changes

The PR marks exhausted embedded provider fallback errors as final lifecycle failures and has the gateway finalize those errors immediately while preserving retry grace for per-attempt fallback errors.

## Best Possible Solution

Land the narrow lifecycle-marker fix after real webchat/provider-failure proof or an explicit maintainer proof override confirms the session exits `running` and persists a failed terminal state.

## Reproduction Assessment

No live reproduction was established for the exact Linux/OpenAI OAuth failure. Source inspection does show the relevant path: current main defers lifecycle errors behind retry grace, fallback-step events can clear pending per-attempt errors, and fallback exhaustion can then return a final failure without a terminal lifecycle marker.

## Solution Assessment

Yes, this looks like the narrow maintainable fix shape: mark only fallback-exhausted failures in the runner and let the gateway preserve retry grace for unmarked per-attempt errors. The safer alternatives, such as disabling grace globally or patching only UI state, would either regress fallback retry behavior or leave backend session state wrong.
