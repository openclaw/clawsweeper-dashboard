## Summary

Keep open, but this PR should not merge as-is. The session-store performance direction is plausible, but the branch currently fails basic loader/typecheck expectations, bypasses the existing locked atomic session-store write contract, makes normal session reads destructive, loses manifest-only plugin metadata, and reintroduces runtime package-manager activity during plugin loading.

## What This Changes

The PR writes normalized and maintained session-store data from `loadSessionStore` and rewrites plugin loading to stage bundled runtime dependencies with `setImmediate`.

## Best Possible Solution

Keep the session-store performance fix as a narrow follow-up that persists normalization/maintenance only through the existing session write lock and atomic write helper, while preserving non-destructive read semantics. Drop the plugin runtime-dependency change from this PR; if plugin dependency latency still needs work, handle it in explicit install/update/doctor flows with docs, tests, and changelog coverage.

## Reproduction Assessment

Yes for the merge blockers: the PR head imports modules that return 404, removes a typed option still used by `src/plugins/tools.ts`, and contradicts current tests for normal session reads and manifest-only plugin metadata. The production gateway latency measurements were not reproduced here.

## Solution Assessment

No. The current PR is not the best way to solve the issue because it mixes a plausible session-store optimization with unsafe read-path writes, destructive read behavior, stale loader changes, and runtime dependency installation; the safer alternative is a focused session-store patch using the existing locked atomic write contract.
