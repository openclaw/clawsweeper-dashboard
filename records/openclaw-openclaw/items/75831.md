## Summary

Keep open. The PR targets plausible session-store and plugin-startup performance problems, but the branch should not merge as-is because it adds unsafe read-path writes, makes destructive maintenance run from ordinary reads, and reintroduces a runtime package-install path that current main has moved out of plugin loading.

## What This Changes

The PR changes `loadSessionStore` to write normalized and maintained session data back to `sessions.json`, and changes plugin loading to defer bundled runtime dependency staging while adjusting loader cache and module-loading metadata.

## Best Possible Solution

Land a narrower session-store fix that persists normalization or explicit maintenance through the existing locked atomic save path with regression tests; keep plugin dependency handling in the current install/update/doctor architecture and preserve manifest-only plugin metadata.

## Reproduction Assessment

Yes for the PR review findings: the diff and current tests provide a high-confidence source-level reproduction path. Targeted checks would include the session pruning integration test and plugin loader metadata tests, but they were not run because this review is read-only.

## Solution Assessment

No. The session-store direction is worth salvaging, but this implementation bypasses the concurrency contract and changes an obsolete, security-sensitive plugin dependency path instead of using the current architecture.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use the session-store lock for write-back:** `src/config/sessions/store-load.ts:138-139`
  - body: `loadSessionStore` is called by readers outside `withSessionStoreLock`, so writing the parsed snapshot directly can overwrite a newer save that lands after this read. Persisting normalization needs to go through the locked atomic update path or use a stale-snapshot guard before replacing the file.
  - confidence: 0.89
- **[P2] Keep destructive maintenance off ordinary reads:** `src/config/sessions/store-load.ts:148-150`
  - body: Removing the `runMaintenance` gate makes normal reads prune and cap session entries, but this path skips `saveSessionStore`'s active-session preservation, transcript/archive cleanup, disk-budget handling, and lock. A `sessions.list` read can now delete entries as a side effect.
  - confidence: 0.86
- **[P2] Keep runtime dependency staging inside the load boundary:** `src/plugins/loader.ts:1507-1518`
  - body: Deferring the installer with `setImmediate` lets plugin imports continue before missing deps are materialized, and install failures no longer flow through the surrounding plugin-load error handling. Cold bundled plugin loads can fail or race uncoordinated installs.
  - confidence: 0.87
- **[P2] Preserve manifest metadata without loading modules:** `src/plugins/loader.ts:1683-1684`
  - body: The patch removes the manifest snapshot fill before `loadModules: false` records are returned. Manifest-only callers would lose channel/provider/CLI/command metadata even though current tests cover that no-import snapshot behavior.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR changes a package-manager/runtime-dependency path and moves installation outside the guarded plugin-load boundary.

Concerns:
