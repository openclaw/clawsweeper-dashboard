## Summary

Keep open. The PR targets a plausible session-store performance problem, but the branch should not merge as-is because it breaks plugin loader type/build behavior, makes ordinary session reads destructive, bypasses the existing session-store write contract, drops manifest-only metadata, and reopens a security-sensitive runtime dependency policy path.

## What This Changes

The PR writes normalized and maintained session-store data from `loadSessionStore` and changes bundled plugin loading to defer runtime dependency installation with `setImmediate`.

## Best Possible Solution

Split the work. Keep a narrow session-store fix that persists normalization or explicit maintenance through the existing locked atomic write path with focused tests, and leave bundled plugin dependency repair in explicit install/update/doctor flows unless maintainers intentionally change that policy with docs and validation.

## Reproduction Assessment

Yes for the PR blockers: the PR head imports missing modules, removes a typed option used by `src/plugins/tools.ts`, contradicts current session maintenance tests, and drops manifest-only metadata covered by loader tests. The reported production gateway latency itself was not reproduced here.

## Solution Assessment

No. This PR is not the best way to solve the issue because it mixes a plausible session-store optimization with unsafe read-path writes and a plugin dependency policy change; the safer path is a focused session-store patch plus separate maintainer-owned runtime-dependency work if still needed.
