## Summary

Keep open. The PR targets a plausible session-store performance problem, but it should not merge as-is: the plugin half imports missing runtime-dependency modules and reintroduces package-manager work during plugin loading, while the session half makes ordinary reads destructive and bypasses the existing locked atomic write contract.

## What This Changes

The PR changes session-store loading to write normalized and maintained data back to `sessions.json`, and changes plugin loading to defer bundled runtime dependency installation with `setImmediate`.

## Best Possible Solution

Keep the session-store performance direction, but implement persistence through the existing locked atomic session-store write path and preserve non-destructive read semantics. Drop the plugin runtime-dependency change unless maintainers explicitly redesign the plugin dependency model; dependency convergence should remain in install/update/doctor flows.

## Reproduction Assessment

Yes for the merge blockers: the PR head imports module paths that return 404, and current tests encode the `loadSessionStore` read-maintenance contract that the patch removes. The production latency report is plausible but was not live-reproduced here.

## Solution Assessment

No. The current PR is not the safest fix because it bypasses session write locking/atomic writes and reintroduces runtime dependency installation; a narrower repair should use existing session-store helpers and leave plugin dependency repair out of runtime loading.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add the missing runtime-deps modules:** `src/plugins/loader.ts:30-44`
  - body: This import block references four `bundled-runtime-*` modules, but the PR only changes two files and those module paths are absent at the PR head. The branch will fail to build/typecheck before reaching the runtime behavior it is trying to change.
  - confidence: 0.96
- **[P2] Keep ordinary session reads non-destructive:** `src/config/sessions/store-load.ts:147-149`
  - body: Removing `runMaintenance` makes every uncached `loadSessionStore` call run enforcement when the store is over the cap. Current tests require normal reads to leave oversized stores untouched and only prune/cap when maintenance is explicitly requested, so read-only surfaces like listing sessions can now delete entries.
  - confidence: 0.91
- **[P2] Use the session store write lock for write-back:** `src/config/sessions/store-load.ts:137-138`
  - body: Writing `sessions.json` directly from the read path bypasses the existing session-store lock and atomic write helper, which are there to avoid clobbering concurrent writers. Persisting normalization or maintenance needs to go through that write contract instead of `fs.writeFileSync`.
  - confidence: 0.88
