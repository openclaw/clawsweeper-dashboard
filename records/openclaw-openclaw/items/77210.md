## Summary

Keep open: this PR fixes a real diagnostics regression, but the patch currently treats every `origin: "global"` diagnostics-id plugin as trusted. The `internalDiagnostics` grant should prove the official `@openclaw` package identity before merge.

## What This Changes

The PR widens `createServiceContext` so `diagnostics-otel` and `diagnostics-prometheus` receive `internalDiagnostics` for `bundled` or `global` origins, adds service tests for global/config origins, and adds a changelog fix entry.

## Best Possible Solution

Keep the regression fix, but carry verifiable official package identity into the service registration or an equivalent narrow predicate, grant only the two official diagnostics packages, and keep workspace/config/other global packages denied.

## Reproduction Assessment

Yes from source inspection. Current main routes diagnostics exporters through external package installs, while `services.ts` injects `internalDiagnostics` only for bundled services and both exporters exit when that context field is absent.

## Solution Assessment

No. Adding `global` to the origin check fixes the official install path, but the narrow maintainable fix should also verify the global plugin is the official `@openclaw` diagnostics package before exposing trusted diagnostic emit/subscribe hooks.
