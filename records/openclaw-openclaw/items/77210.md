## Summary

Keep this PR open: it targets a real diagnostics plugin regression, but the proposed trust check grants `internalDiagnostics` to any globally loaded plugin with the diagnostics service ID rather than proving it is the official `@openclaw` package.

## What This Changes

The PR widens the `internalDiagnostics` service-context gate from bundled diagnostics services to bundled or global diagnostics services, adds service tests for global/config origins, and adds a changelog entry.

## Best Possible Solution

Keep the diagnostics regression fix, but make the global-origin grant depend on an explicit official package identity such as the exact `@openclaw/diagnostics-*` package or official catalog/ClawHub source, with a denial test for a third-party global package using the same plugin/service ID.

## Reproduction Assessment

Yes, from source inspection: current `main` documents installing the diagnostics plugins through the external install path while `createServiceContext` only grants `internalDiagnostics` to `origin === "bundled"`. I did not execute the runtime path in this read-only review, but the code path matches the linked failure report.

## Solution Assessment

No. Accepting all `global` origins is too broad for an internal diagnostics capability; the narrower maintainable fix is to prove the global service comes from the exact official diagnostics package/source before injecting the capability.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require official package identity before granting diagnostics:** `src/plugins/services.ts:28`
  - body: This widens the internal diagnostics capability to every globally loaded plugin whose manifest/service ID is `diagnostics-otel` or `diagnostics-prometheus`. `origin: "global"` only proves it came from the global install/discovery path, and service registrations do not carry the npm package name; install validation also allows unscoped package-name matches, so a third-party global package with the same ID could receive the trusted diagnostics stream. Please gate global grants on the exact official package/source identity, not origin plus ID alone.
