## Summary

Keep open: current main has partial transparency and recovery for compaction, including opt-in notices, status/UI indicators, checkpoints, restore/branch RPCs, and plugin hooks, but it still lacks the requested preview/approve/reject workflow or configurable confirmation mode for automatic context compaction. The related context-engine and self-compaction items do not supersede this broader core UX/control request.

## What This Changes

Request to make automatic context truncation, summarization, and dropping user-visible and controllable with notifications, metadata, previews, confirmation modes, rollback, configuration, and developer hooks.

## Best Possible Solution

Design this as a core compaction UX/policy layer rather than a plugin-only feature: keep current automatic behavior as the default, add opt-in interactive preview/confirmation only where the requester can respond, reuse existing checkpoints for branch/restore, and document non-interactive fallback semantics clearly.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The current-main verification is high-confidence: source, schema, docs, and RPC inspection show notices and checkpoints exist, but preview/approve/reject confirmation modes are absent.

## Solution Assessment

Unclear as one patch: the requested direction is valid, but the safest solution needs maintainer product judgment around interactive versus unattended sessions. A phased core design built on existing notifyUser, checkpoint, and hook surfaces is safer than forcing confirmation into every automatic compaction path.
