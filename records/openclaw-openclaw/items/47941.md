## Summary

Keep open. Current main still lets a hook-provided bootstrap file with a valid path but missing name reach the context report with `name: file.name`, and the `/context` renderer prints that raw name. The provided GitHub context also shows open PR #48021 explicitly using closing syntax for this issue, so the right path is to review and land or close that PR before resolving this issue.

## What This Changes

Ensure `/context list` and related context breakdowns label hook-injected virtual bootstrap files from a non-empty name, path/basename, or stable placeholder instead of rendering literal `undefined`.

## Best Possible Solution

Review #48021 or an equivalent narrow fix, normalize bootstrap stat labels from non-empty name to path/basename to placeholder, and add a focused regression test for hook-injected path-only virtual bootstrap files before closing this issue.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: a bootstrap hook can append a path/content object without `name`, `sanitizeBootstrapFiles` preserves it, `buildBootstrapInjectionStats` returns `name: file.name`, and `/context` renders `f.name`. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Yes. Normalizing the label in `buildBootstrapInjectionStats` or the report-construction layer, with a regression test, is the narrowest maintainable solution; a UI-only fallback would not fix cached or JSON report consumers.
