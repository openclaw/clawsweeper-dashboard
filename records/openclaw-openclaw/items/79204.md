## Summary

Keep open: the security direction is valid, but the patch does not fully enforce the advertised owner/admin gate because the chosen helper still treats no-scope trusted identity requests as default-admin owner requests.

## What This Changes

This PR changes managed outgoing image byte serving to require `resolveOpenAiCompatibleHttpSenderIsOwner`, updates the managed-image tests, and adds a changelog fix entry.

## Best Possible Solution

Use a media-specific owner/admin predicate that does not accept requester-session self-declaration or implicit default-admin scopes as ownership, while preserving the intended shared-secret behavior and adding coverage with the real auth helper semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main bypasses the requester-session owner check for trusted-scope and device-token auth before serving managed image bytes. The proposed patch’s remaining gap is also source-visible through the helper’s default-scope owner behavior.

## Solution Assessment

No, the selected helper is not the best fix as used here because it still grants owner semantics through implicit default scopes on trusted identity requests. A narrower media-specific check or explicit-scope check would better match the PR’s stated security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid defaulting media fetches to owner:** `src/gateway/managed-image-attachments.ts:1023`
  - body: The new guard calls `resolveOpenAiCompatibleHttpSenderIsOwner`, but for non-shared-secret auth that helper derives ownership from trusted HTTP scopes, and missing `x-openclaw-scopes` falls back to the default scope set containing `operator.admin`.
