## Summary

Keep open. Current main still scopes Skill Workshop proposals, review context, storage, tool actions, and writes to the active workspace; the requested shared-skill targeting and cross-workspace promotion are valid feature work but expand write authority and cross-workspace review in a security-sensitive bundled plugin, so they need maintainer product/security direction before implementation.

## What This Changes

The issue requests opt-in Skill Workshop support for proposing/refining shared skills, promoting workspace skills to shared skill candidates, and detecting similar skill patterns across workspaces.

## Best Possible Solution

Stage this inside extensions/skill-workshop: first design explicit target/root metadata, separated workspace-vs-shared proposal storage, strict shared-root validation, manual-only shared applies, docs, and focused tests; then handle cross-workspace semantic promotion as a follow-up after privacy and retention rules are approved.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static inspection gives high confidence in the current limitation because the current source, docs, tests, tool schema, reviewer, and store are all workspace-scoped.

## Solution Assessment

Unclear. Shared targeting is plausible, but the proposed A/B/C package crosses shared write and cross-workspace privacy boundaries, so the safer solution is to split the design and get maintainer product/security approval before coding.
