## Summary

Keep open. Current main still scopes Skill Workshop writes, proposals, reviewer input, and proposal storage to one workspace; the requested shared-skill targeting and cross-workspace promotion are valid feature work, but they expand shared write authority and cross-workspace data handling, so maintainer product/security direction is needed before implementation.

## What This Changes

Request opt-in Skill Workshop support for shared-skill update proposals, cross-workspace pattern detection, and manual promotion from workspace skills to shared skill candidates.

## Best Possible Solution

Stage this inside extensions/skill-workshop: first design explicit target/root metadata, separated workspace-vs-shared proposal storage, strict shared-root validation, manual-only shared applies, docs, and focused tests; then treat semantic cross-workspace promotion as a follow-up after privacy and retention rules are approved.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static inspection gives high confidence in the current limitation because the source, docs, tests, tool schema, reviewer, and store are all workspace-scoped.

## Solution Assessment

Unclear. Shared targeting is plausible, but the requested A/B/C bundle crosses shared write and cross-workspace privacy boundaries, so the narrowest maintainable path is a split design with explicit maintainer product/security approval before coding.
