## Summary

Keep open. Current main still scopes Skill Workshop proposals, tool actions, reviewer context, proposal storage, and writes to the active workspace; the requested shared-skill targeting and cross-workspace promotion remain valid feature work, but they change shared write authority and cross-workspace data flow in a bundled plugin, so this needs maintainer product/security direction before implementation.

## What This Changes

Request opt-in Skill Workshop support for proposing shared-skill updates and promoting repeated workspace-skill patterns into shared skill candidates.

## Best Possible Solution

Stage this inside extensions/skill-workshop: first design explicit target/root metadata, separated workspace-vs-shared proposal storage, strict shared-root validation, manual-only shared applies, docs, and focused tests; then handle semantic cross-workspace promotion as a follow-up after privacy and retention rules are approved.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static inspection gives high confidence in the current limitation because the source, docs, tests, tool schema, reviewer, and store are all workspace-scoped.

## Solution Assessment

Unclear. Shared targeting is plausible, but the proposed A/B/C bundle crosses shared write and cross-workspace privacy boundaries, so the safer path is a split design with explicit maintainer product/security approval before coding.
