## Summary

Keep open. Current main does not implement the PR’s intended Play-flavor SMS/MMS enablement or MMS batching change, and the proposed patch should not merge as-is because it drops MMS candidates beyond the hydration cap before filters and pagination. The PR also predates the current Android flavor source-set split, so the Play permission change needs maintainer product/security review rather than a mechanical cleanup close.

## What This Changes

The PR enables SMS and call-log access for the Android Play flavor, updates SMS permission UI copy to mention MMS, and rewrites mixed SMS/MMS by-phone search to batch-hydrate MMS text/meta with a 50-row cap and timeout.

## Best Possible Solution

Decide the Play-vs-thirdParty sensitive-permission policy first. If maintainers want this behavior, implement it against the current SensitiveFeatureConfig/source-set architecture, keep Play Store and self-hosted build semantics explicit, preserve all MMS candidates while only limiting expensive hydration work, and add focused tests for more than 50 hydratable MMS rows with offsets and selective filters.

## Reproduction Assessment

Yes at code-review level. A by-phone includeMms search with more than 50 hydratable MMS rows and either offset beyond the cap or selective keyword/type/read filters would miss valid rows because the PR truncates mmsCandidates before filtering and pagination.

## Solution Assessment

No. The batching direction is useful, but this patch is not the best fix as-is because the cap discards data and the Play enablement targets an older BuildConfig/manifest-overlay model instead of current main’s flavor-owned source sets.
