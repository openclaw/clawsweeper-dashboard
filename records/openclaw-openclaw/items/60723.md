## Summary

Keep open: current main does not already contain the MMS batching improvement, but this PR is not ready because it changes the Play flavor's sensitive SMS/call-log policy and the proposed batching path drops MMS candidates before hydrated filters and pagination.

## What This Changes

The PR enables Play flavor SMS/MMS/call-log access, updates Android SMS wording, and rewrites by-phone MMS search hydration to batch MMS body/meta lookups.

## Best Possible Solution

Split the work: keep the Play-safe source-set policy unless explicitly approved, and land a narrow thirdParty MMS batching fix that preserves the candidate window with focused tests and real Android proof.

## Reproduction Assessment

Yes. Source review of PR head shows the candidate-loss path: `take(50)` runs before hydrated keyword/type/read filtering and pagination, so a by-phone `includeMms` query with later matching MMS rows can lose valid results.

## Solution Assessment

No. Batch MMS hydration is a reasonable direction, but this PR mixes it with a sensitive Play-policy change and implements the cap by discarding candidates instead of bounding expensive hydration while preserving searchable rows.
