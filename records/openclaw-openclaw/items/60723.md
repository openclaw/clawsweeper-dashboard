## Summary

Keep this PR open for maintainer handling: the MMS batching direction addresses a real current-main hotspot, but the patch is not mergeable as-is because it broadens Play flavor SMS/call-log access and introduces a candidate-truncation bug in the MMS query path.

## What This Changes

The PR enables SMS/MMS and call-log access for the Android Play flavor, updates SMS permission UI copy, and rewrites mixed SMS/MMS by-phone search to batch-hydrate MMS rows with a timeout.

## Best Possible Solution

Decide the Play-vs-thirdParty sensitive-permission policy first; if maintainers want the MMS performance fix, retarget it to the current thirdParty SmsManager without discarding candidates beyond the hydration cap.

## Reproduction Assessment

Yes at code-review level. A by-phone includeMms query with more than 50 hydratable MMS rows and an offset or selective hydrated filter can miss valid rows because the PR truncates candidates before filtering and pagination.

## Solution Assessment

No. Batch hydration is a good direction, but this PR is not the best fix as-is because it combines a sensitive Play policy change with a stale flavor model and a result-truncation bug.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve MMS candidates beyond the hydration cap:** `apps/android/app/src/main/java/ai/openclaw/app/node/SmsManager.kt:1061`
  - body: The new `take(MAX_MMS_HYDRATION_BATCH)` drops later MMS candidates before hydrated keyword/type/read filters and pagination run. For by-phone `includeMms` searches with more than 50 hydratable MMS rows, callers can get empty or incomplete pages even when later rows match; limit expensive hydration work without removing candidates from the result window.
