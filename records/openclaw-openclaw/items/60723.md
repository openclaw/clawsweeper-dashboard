## Summary

Keep open. The useful MMS batching/perf concern is not implemented on current main, but this branch is not mergeable as-is because it can drop MMS rows past the hydration cap, changes Play SMS/call-log policy, targets an older Android flavor model, and lacks real behavior proof.

## What This Changes

The PR enables SMS/MMS and call-log access for the Android Play flavor, updates SMS/MMS UI text, and batch-hydrates MMS rows in by-phone SMS/MMS search.

## Best Possible Solution

Retarget only the MMS batching fix to the current thirdParty SmsManager with candidate-window preservation and focused proof, and handle Play SMS/call-log access separately with explicit release-policy approval.

## Reproduction Assessment

Yes, at source-review level for the PR defect: PR head caps hydrated MMS candidates with `take(50)` before hydrated keyword/type/read filters and pagination.

## Solution Assessment

No. Batch MMS hydration is a reasonable direction, but this PR mixes it with a sensitive Play policy change, targets a stale flavor model, and drops candidates past the hydration cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve MMS rows beyond the hydration cap:** `apps/android/app/src/main/java/ai/openclaw/app/node/SmsManager.kt:1061`
  - body: The new `take(MAX_MMS_HYDRATION_BATCH)` drops every MMS candidate after the first 50 before hydrated keyword/type/read filters and pagination run.
