## Summary

Keep open: the MMS timeout fix is not implemented on current main and may still be useful, but this PR is not mergeable as-is because the batching path can drop valid MMS rows, it broadens Play flavor SMS/call-log access without policy approval, it lacks contributor real-behavior proof, and it now conflicts with the current Android flavor split.

## What This Changes

This PR enables Android Play flavor SMS/MMS/call-log access, updates SMS wording, and rewrites mixed by-phone MMS hydration to batch body/meta queries with a timeout.

## Best Possible Solution

Split the work: keep Play restricted unless explicitly approved, then port a candidate-preserving MMS batching fix to the current thirdParty SmsManager with focused tests and real Android proof.

## Reproduction Assessment

Yes. Source review of PR head shows `take(MAX_MMS_HYDRATION_BATCH)` runs before hydrated keyword/type/read filtering and pagination, so later matching MMS rows can disappear from an `includeMms` by-phone query.

## Solution Assessment

No. Batch hydration is a reasonable direction, but this PR mixes it with a sensitive Play-policy change and bounds work by discarding candidates instead of preserving the searchable window.
