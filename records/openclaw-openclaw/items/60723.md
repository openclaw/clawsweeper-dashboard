## Summary

Keep open, but this PR is not merge-ready: current main intentionally keeps Play SMS and call-log access disabled, the MMS timeout/performance work is not already implemented on main, and the proposed batch path has a source-reproducible truncation bug plus a security-sensitive Play policy change.

## What This Changes

The PR enables SMS/MMS and call-log access for the Android Play flavor, updates SMS permission UI copy, and rewrites mixed SMS/MMS by-phone search to batch-hydrate MMS rows with a timeout.

## Best Possible Solution

Split the MMS performance fix into the current thirdParty SmsManager, preserve candidates past any hydration budget, add focused regression and real-device proof, and only change Play SMS/call-log access after explicit maintainer approval.

## Reproduction Assessment

Yes at source-review level for the patch defect. The PR's capped `take(MAX_MMS_HYDRATION_BATCH)` runs before hydrated keyword/type/read filtering and pagination, so a by-phone `includeMms` query with more than 50 hydratable MMS rows can miss later valid rows.

## Solution Assessment

No. Batch MMS hydration is a reasonable direction, but this PR mixes it with a Play sensitive-permission policy change, targets a stale flavor model, and drops MMS candidates past the cap.
