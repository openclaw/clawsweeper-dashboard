## Summary

Keep open: current main does not include the MMS performance fix, but this branch is not mergeable as-is because it drops MMS candidates beyond the hydration cap, lacks real behavior proof, and bundles a Play sensitive-permission policy change.

## What This Changes

The PR enables SMS/MMS and call-log access for the Android Play flavor and rewrites mixed SMS/MMS by-phone search to batch-hydrate MMS rows.

## Best Possible Solution

Split the work: retarget MMS batching to the current thirdParty SmsManager with candidate-window preservation and focused proof, and handle Play SMS/call-log access only as a separate maintainer-approved policy change.

## Reproduction Assessment

Yes at source-review level for the PR defect. The proposed cap runs before hydrated keyword/type/read filters and pagination, so an `includeMms` by-phone search with more than 50 hydratable MMS rows can miss later valid rows.

## Solution Assessment

No. Batch MMS hydration is a reasonable direction, but this PR mixes it with a sensitive Play policy change, targets a stale flavor model, and drops candidates past the hydration cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve MMS rows beyond the hydration cap:** `apps/android/app/src/main/java/ai/openclaw/app/node/SmsManager.kt:1061`
  - body: The new `take(MAX_MMS_HYDRATION_BATCH)` drops every MMS candidate after the first 50 before hydrated keyword/type/read filters and pagination run.
