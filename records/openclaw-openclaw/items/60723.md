## Summary

Keep open for maintainer review: the MMS batching idea is still relevant, but the PR is not mergeable as-is because it broadens Play flavor SMS/call-log access, conflicts with the current source-set split, lacks real behavior proof, and introduces a lossy MMS hydration cap.

## What This Changes

The PR enables Android Play flavor SMS/MMS/call-log access, updates SMS wording, and rewrites mixed by-phone MMS hydration to batch body/meta queries with a timeout.

## Best Possible Solution

Split the work: keep Play restricted unless explicitly approved, then port a candidate-preserving MMS batching fix to the current thirdParty SmsManager with focused tests and real Android proof.

## Reproduction Assessment

Yes by source review for the PR-introduced regression: the PR caps `mmsCandidates` before hydrated keyword/type/read filters and pagination. I did not reproduce the original timeout on a real device, but current main still shows the per-row MMS hydration path.

## Solution Assessment

No. Batch hydration is a reasonable direction, but this PR mixes it with a sensitive Play-policy expansion and bounds work by discarding candidates; the safer fix is thirdParty-only and candidate-preserving unless maintainers approve a Play policy change.
