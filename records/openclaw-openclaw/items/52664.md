## Summary

Keep open. The feature is not on current main and the PR still has a correctness blocker in the steer path, plus the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds an optional `rawBody` field to plugin hook events, threads it from channel reply runs through agent execution and followup queues, and updates docs, API baseline, changelog, and focused tests.

## Best Possible Solution

Land an additive plugin hook API only after the `agent_end` contract cannot report stale clean text for steered messages, and after a real plugin/channel run proves the field observed by hooks.

## Reproduction Assessment

Yes for the review finding: source inspection shows the steer path queues only text into the active run while the PR reports `params.rawBody` from the original run at `agent_end`. I did not run tests because this review is constrained to read-only inspection.

## Solution Assessment

No. The additive hook field is reasonable, but exposing it on `agent_end` while steered user messages can be appended without updating or invalidating `rawBody` is not the narrowest maintainable contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid stale rawBody on steered agent_end events:** `src/agents/pi-embedded-runner/run/attempt.ts:2998`
  - body: In `queue.mode=steer`, an overlapping user turn is queued into the active embedded run as text only, so the active run's `params.rawBody` still describes the original turn. Adding that value to `agent_end` means plugins can see `event.messages` containing the steered user message while `event.rawBody` points at older input, which breaks the memory/analytics use case this field is meant to support. Please either carry the steered raw input through the queue contract or omit/mark `rawBody` when the run has steered messages.
  - confidence: 0.84
