## Summary

Keep this PR open: the additive hook API is useful, but the current diff can report stale `agent_end.rawBody` for steered turns and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `rawBody` to plugin hook event types and threads channel message text through reply, queue, embedded runner, docs, generated SDK hash, tests, and changelog.

## Best Possible Solution

Keep the additive hook field, but align `rawBody` semantics across immediate, queued, collected, and steered runs before landing with real plugin/channel proof.

## Reproduction Assessment

Yes for the review finding by source inspection: steer queues only a prompt string while `agent_end` would emit the original run's `params.rawBody` with the final messages. I did not execute a live steer session in this read-only review.

## Solution Assessment

No as currently proposed. Exposing `rawBody` is a reasonable API direction, but the `agent_end` behavior needs steer-aware metadata, omission, or clearer narrowed semantics so plugins do not pair final messages with stale input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid reporting stale rawBody after steered turns:** `src/agents/pi-embedded-runner/run/attempt.ts:2998`
  - body: `agent_end` runs after mid-stream steer messages can be appended to `messages`, but this new field is still the original run parameter. In `queue.mode=steer`/`steer-backlog`, a second user turn can appear in `event.messages` while `event.rawBody` remains the previous turn, so plugins that prefer the new field for memory or analytics can pair the assistant response with stale input. Carry raw text through the steer queue, or omit/replace `rawBody` for `agent_end` once a steered message is accepted.
  - confidence: 0.86
