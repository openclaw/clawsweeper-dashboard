## Summary

Keep the PR open, but not merge-ready: the additive hook API is useful and current main does not already provide it, yet the implementation gates `rawBody` only by provider string and misses internal/provenance-marked webchat turns.

## What This Changes

Adds optional `rawBody` to plugin hook events and threads it through immediate, queued, collect, deferred, and steer reply paths.

## Best Possible Solution

Expose `rawBody` only for direct external channel user turns, keep inter-session/internal-system origins undefined, and add focused regression coverage before maintainer API acceptance.

## Reproduction Assessment

Yes. From source, gateway `chat.send` can build `webchat` contexts with `InputProvenance`, while the PR only checks `Provider` before emitting `rawBody`.

## Solution Assessment

No. The threading shape is close, but the best fix must gate the public field on actual external channel user provenance instead of provider string alone.

- **[P1] Gate rawBody to direct user-channel turns:** `src/auto-reply/reply/get-reply-run.ts:662`
  - body: This only excludes heartbeat/cron/exec by `Provider`.
