## Summary

Keep open: current main still lacks the requested hook field, and the PR remains useful, but the latest head still has a blocking agent_end semantics issue for steered turns plus no contributor-provided real behavior proof.

## What This Changes

The PR adds an optional rawBody field to before_prompt_build, before_agent_start, and agent_end hook events, threads it through reply and embedded-runner execution, and updates docs, generated SDK baseline hash, tests, and changelog.

## Best Possible Solution

Land the additive hook field only after agent_end cannot pair stale rawBody with final messages from steered turns, and after a real plugin/channel run proves the hook payload observed by plugins.

## Reproduction Assessment

Yes for the review finding by source inspection: steer queues only prompt text into the active embedded run while the PR would emit the original run-level rawBody on agent_end. I did not execute a live steer session in this read-only review.

## Solution Assessment

No as currently proposed. Exposing rawBody is a reasonable API direction, but agent_end needs steer-aware metadata, omission, or invalidation so plugins do not pair final messages with stale input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid reporting stale rawBody after steered turns:** `src/agents/pi-embedded-runner/run/attempt.ts:2998`
  - body: In queue.mode=steer or steer-backlog, overlapping user text is queued into the active embedded run as prompt text only, but agent_end would emit params.rawBody from the original run. That lets event.messages include the steered user message while event.rawBody points at older input, breaking the memory and analytics use case this field is meant to support. Carry raw text through the steer queue, or omit/mark rawBody for agent_end once a steered message is accepted.
  - confidence: 0.86
