## Summary

Keep open: current main does not expose `rawBody` on these plugin hook events, and the PR is a plausible additive API, but the latest head still has a stale `agent_end.rawBody` case for active-run steering and lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `rawBody` to `before_prompt_build`, `before_agent_start`, and `agent_end` hook event types, threads it through reply and embedded-run queue paths, and updates hook docs, tests, SDK baseline hashes, and changelog.

## Best Possible Solution

Land the additive hook API after `agent_end` cannot expose stale run-level `rawBody` for runs that accepted steering, and after redacted real plugin/channel output proves the field in a real setup.

## Reproduction Assessment

Yes, by source inspection for the review finding: active `queue.mode=steer` sends only prompt text through the steering queue, while the PR’s `agent_end` payload reports the original run-level `params.rawBody`. I did not run a live steer session in this read-only review.

## Solution Assessment

No. Exposing `rawBody` is a reasonable API direction, but `agent_end` should omit or mark `rawBody` after accepted steering, or carry raw input metadata through the steer queue before landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid reporting stale rawBody after steered turns:** `src/agents/pi-embedded-runner/run/attempt.ts:2998`
  - body: When another message is steered into an active Pi run, the steer path queues only `followupRun.prompt`, so the final `messages` snapshot can include later user text. This added `agent_end.rawBody` still reports `params.rawBody` from the original turn, which can make memory or analytics plugins pair the final conversation with stale input; omit it after accepted steering or carry raw metadata through the steer queue.
  - confidence: 0.86
