## Summary

Keep open because the branch still adds a useful public plugin hook field that current main lacks; merge should wait for real behavior proof and a small docs clarification, not cleanup closure.

## What This Changes

The PR adds optional `rawBody` to prompt-build, legacy agent-start, and agent-end hook events, threading it from channel reply input through reply queues and embedded-runner hook payloads with docs, changelog, API hash, and gating tests.

## Best Possible Solution

Land a rebased version after maintainer API review, a docs clarification for run-level steered-message semantics, and real channel/plugin proof showing `rawBody` on user-originated runs and absence on system-event runs.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source inspection shows current main lacks this hook field and the PR diff adds it to the intended hook surfaces.

## Solution Assessment

Mostly yes after one docs correction: an additive optional hook field is the narrow maintainable direction, but the public contract should say `rawBody` is run-level and does not update for steered messages.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Clarify steered-message rawBody semantics:** `docs/plugins/hooks.md:254-256`
  - body: The code only passes `rawBody` when a run is created; the steer path queues later text with `queueEmbeddedPiMessageWithOutcome(..., followupRun.prompt, ...)` and no raw metadata. For a user-originated run, `agent_end` can therefore still carry the initiating turn's `rawBody` while `messages` includes steered text, so this docs sentence should clarify run-level semantics instead of implying steered messages produce an absent `rawBody`.
  - confidence: 0.82
