## Summary

Keep open: the feature is useful and current `main` does not already expose `rawBody` on these plugin hook events, but PR head still has two merge-blocking contract issues around input provenance and CLI-backed hook delivery.

## What This Changes

The branch adds optional `rawBody` to `before_prompt_build`, `before_agent_start`, and `agent_end` hook events and threads it through reply, followup, collect, steer, embedded-agent, docs, and focused tests.

## Best Possible Solution

Gate `rawBody` to direct external-user channel input, forward it through every hook runner path that handles channel-originated runs, update docs/tests for provenance-marked absence, and then let maintainers decide whether to accept the additive plugin API.

## Reproduction Assessment

Yes. Source inspection on PR head shows `rawBodyForPluginEvent` ignores input provenance and the CLI-backed prompt-build hook preparation path never receives `rawBody`; I did not run tests because this is a read-only review.

## Solution Assessment

No. Threading clean user text through the reply/run path is the right shape, but it is not the best solution until provenance and sibling hook surfaces share the same contract.
