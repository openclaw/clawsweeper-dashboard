## Summary

Keep open. Current main does not expose `rawBody` on plugin hook events, but the latest PR still has a blocking stale-`rawBody` problem for active-run steering and lacks required real behavior proof.

## What This Changes

The PR adds optional `rawBody` to `before_prompt_build`, `before_agent_start`, and `agent_end` hook events, threads it through reply and embedded-run params/queues, and updates docs, SDK baseline hashes, tests, and changelog.

## Best Possible Solution

Land the additive hook API after `agent_end` cannot pair final messages with stale run-level `rawBody` and after redacted real-run proof shows plugins receiving the field.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: an active `queue.mode=steer` run queues only prompt text through `queueEmbeddedPiMessageWithOutcome`, while the PR’s `agent_end` payload uses the original run-level `params.rawBody`. I did not run a live steer session in this read-only review.

## Solution Assessment

No. Exposing `rawBody` is a reasonable API direction, but `agent_end` should omit or mark `rawBody` after accepted steering, or carry raw input metadata through the steer queue before landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid reporting stale rawBody after steered turns:** `src/agents/pi-embedded-runner/run/attempt.ts:2998`
  - body: When another channel message arrives while the same session is streaming, the steer path queues only `followupRun.prompt` and the embedded queue handle accepts text/options only. This new `agent_end` field then still emits the original run's `params.rawBody`, so `event.messages` can include the steered user text while `event.rawBody` points at the older input, which breaks the memory/analytics use case and contradicts the documented steered-message absence contract.
  - confidence: 0.86
