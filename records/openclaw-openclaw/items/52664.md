## Summary

Keep open: the additive plugin API is useful, but the latest PR head still can emit a run-level `rawBody` at `agent_end` while active-run steering appends later user text without raw metadata, so hook consumers can see `rawBody` that does not match `event.messages`; contributor real behavior proof is also still missing.

## What This Changes

The PR adds optional `rawBody` fields to plugin hook event types/docs and threads channel-derived raw user input through reply, queue, collect, and embedded agent run paths.

## Best Possible Solution

Land the rawBody API after hook event semantics match final message sets across normal, queued, collect, system-event, and active steering paths.

## Reproduction Assessment

Yes for the review finding by source inspection: current main queues only steered prompt text, while the PR emits the run-level `params.rawBody` at `agent_end`. I did not run a live channel/plugin reproduction in this read-only review.

## Solution Assessment

No as-is. The additive field is the right direction, but active-run steering needs unambiguous semantics and the contributor still needs to provide real behavior proof.
