## Summary

Keep open. Current main still does not expose `rawBody` on the requested plugin hook events, and the PR addresses a real plugin ergonomics gap backed by the `RawBody` precedent in #51502. The patch is not merge-ready because it forwards a raw body for every prepared reply run, including heartbeat/cron/system-event paths that the PR says should remain undefined, and it still needs release-note coverage for the public hook API change.

## What This Changes

The PR adds optional `rawBody` to `before_prompt_build`, legacy `before_agent_start`, and `agent_end` hook events and threads it through reply, embedded-runner, deferred follow-up, collect queue, and generated SDK baseline surfaces.

## Best Possible Solution

Land a rebased version that exposes `rawBody` only for user-originated channel runs across immediate, deferred follow-up, and collect paths, keeps CLI/heartbeat/cron/exec-event runs undefined, documents the hook field for plugin authors, and adds focused tests for channel raw text plus non-channel omission.

## Reproduction Assessment

Yes. The provided diff derives `rawBody` from `ctx.CommandBody ?? ctx.RawBody ?? ctx.Body` unconditionally, and current heartbeat coverage constructs a heartbeat prepared reply with all three fields set to the scheduler prompt.

## Solution Assessment

No. The additive hook field is the right direction, but the value should be gated before storage/forwarding so system-event and non-channel runs cannot be reported as raw user input.
