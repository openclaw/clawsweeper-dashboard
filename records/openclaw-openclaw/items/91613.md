## Summary

Keep open: current main still falls back from missing cron-specific delivery context to the shared main-session destination and then durably replays the stored destination after restart. The linked clean PR addresses a keyless implicit-cron slice, but it is unmerged and its stated trigger does not fully establish the reporter’s exact `sessionTarget: isolated` path or a general recovery-time safety rule.

## What This Changes

Prevent pre-restart outbound replies from being durably replayed into an incorrect Matrix room.

## Best Possible Solution

Bind cron delivery to trustworthy job or creator-session provenance before enqueue, fail closed when the destination comes only from an unrelated shared-session fallback, and define a narrow recovery-time freshness or destination-validity rule only if live validation shows residual stale entries can still cross conversation boundaries.

## Reproduction Assessment

Yes at source level: current main contains the shared-main fallback and durable replay interaction needed for wrong-target persistence. The exact full restart path with the reporter’s `sessionTarget: isolated` configuration has not been independently reproduced on current main.

## Solution Assessment

Unclear as a complete fix: refusing ambiguous shared-main targets before enqueue is safer and narrower than flushing the entire queue, but the linked PR must be reconciled with current main and tested against the reporter’s exact configuration before deciding whether recovery also needs a generic guard.
