## Summary

Keep open: current main and the latest release still have startup outbound delivery recovery that can replay pending delivery entries after a restart, and I did not find a source-level boot fence, TTL, or Matrix room/session mismatch guard that would make the reported stale Matrix drain obsolete.

## What This Changes

Prevent Gateway outbound reply queue entries from surviving restart and draining stale Matrix replies into the wrong room after isolated cron or session completion.

## Best Possible Solution

Add an explicit startup recovery safety policy for outbound delivery that prevents stale or mismatched Matrix queue entries from replaying across restart while preserving intentional durable sends with focused tests.

## Reproduction Assessment

No high-confidence live reproduction was established. The current source clearly replays pending outbound entries on startup, but the full wrong-room Matrix sequence still needs a focused restart/Matrix proof.

## Solution Assessment

Unclear: flushing all pending outbound entries on startup would stop this class of leak but can drop legitimate in-flight replies. The best fix likely needs a maintainer decision around boot fencing, TTL, and Matrix room/session binding validation.
