## Summary

Keep open: this PR targets a real restart-continuation/heartbeat race and now includes credible live proof, but the retry implementation conflicts with the durable queue’s five-retry cap, so the intended 20-attempt shutdown wait is not actually honored.

## What This Changes

The PR preserves routed Telegram restart continuations as trusted internal agent turns, retries busy shutdown replies, suppresses the competing heartbeat wake, exports the busy-reply text constant, and updates gateway tests plus the changelog.

## Best Possible Solution

Align the restart-continuation busy retry path with the durable queue retry contract, then cover the real queue max-retry behavior so Telegram topic continuations can wait through the intended post-reboot cleanup window.

## Reproduction Assessment

Yes from source plus supplied live logs: current main queues both the routed continuation and a generic heartbeat wake, and the PR body shows the pre-patch Telegram topic failure. I did not run a live Telegram/systemd reproduction in this read-only review.

## Solution Assessment

No, not yet. The direction is narrow and maintainable, but the busy-retry loop must either avoid consuming the generic queue retry cap or explicitly coordinate with that cap before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor the busy retry budget before failing the queue entry:** `src/gateway/server-restart-sentinel.ts:55`
  - body: The new loop advertises 20 busy retries, but each busy reply is persisted through `drainPendingSessionDeliveries`, whose production recovery path moves an entry to `failed/` once `retryCount` reaches 5 before invoking `deliver` again. After five busy responses the next drain removes the continuation, so attempts 6-20 never happen; the new test misses this because its mock drain has no max-retry check. Please either keep busy deferrals from consuming the generic queue retry cap or add a bounded max-retry override and cover it with the real recovery behavior.
  - confidence: 0.91
