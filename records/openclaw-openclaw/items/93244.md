## Summary

Keep this PR open for repair: the Codex watchdog direction matches the upstream notification contract, but the lightweight reply lane is effectively unreachable from the normal dispatch path because resolved delivery mode is treated as a caller opt-out.

## What This Changes

The PR adds a helper-driven lightweight reply lane for low-risk text replies and a Codex app-server initial-progress watchdog for accepted-but-silent turns.

## Best Possible Solution

Wire the lightweight decision into the resolved delivery-policy boundary so ordinary automatic replies can enter the lane while message-tool-only, command, media, link, unresolved-reply, and tool-dependent turns stay on the full path.

## Reproduction Assessment

Yes from source inspection: normal dispatch resolves and passes `sourceReplyDeliveryMode` into the resolver, while the new helper exits whenever that field is defined. I did not run tests because this review is read-only.

## Solution Assessment

No: the watchdog part is consistent with Codex protocol events, but the lightweight lane is wired at the wrong boundary. The safer fix is to distinguish explicit caller pins from computed automatic delivery mode, or make the admission decision after delivery policy is resolved.
