## Summary

Keep open. Current main appears to handle the exact app-mention seeding path, but source still shows a related Slack continuity gap for unseeded active channel turns where `replyToMode=first` can deliver a threaded reply from a flat channel session and the later inbound thread reply resolves to a thread-scoped session.

## What This Changes

The issue asks Slack routing to keep outbound-created thread replies and later inbound thread messages on the same OpenClaw session.

## Best Possible Solution

Define a Slack routing policy that keeps outbound-created thread replies and later inbound thread replies on one session without regressing intentionally flat channel-session behavior, then cover `replyToMode=first` plus automatic visible replies in Slack monitor tests.

## Reproduction Assessment

No high-confidence reproduction for the exact app-mention steps: current main source and tests seed mentioned top-level Slack turns into a thread session. Source inspection does show a medium-confidence variant for unseeded active channel turns where the first outbound reply can thread while the initial session stays channel-scoped.

## Solution Assessment

Unclear. The narrow fix is not just mechanical because current tests intentionally preserve flat top-level channel sessions; maintainers need to choose promotion, aliasing, or documented flat-thread behavior before implementation.
