## Summary

Keep open: current main still emits the reported wrapper-first LaunchAgent shape for managed service env, but the live macOS crash path is not proven and the proposed direct `EnvironmentVariables` workaround conflicts with the documented owner-only env-file design.

## What This Changes

The issue reports that `openclaw doctor --fix` and `openclaw update` rewrite the macOS LaunchAgent plist with a generated env-wrapper first in `ProgramArguments`, after which the gateway allegedly crashes under launchd.

## Best Possible Solution

Confirm the failure on macOS launchd, then make the generated env-file wrapper invocation launchd-safe while preserving owner-only service environment storage and adding regression coverage for doctor/update rewrites.

## Reproduction Assessment

Unclear: current main source reproduces the wrapper-first plist whenever managed service env exists, and doctor/update can rewrite through that path. I did not establish a live macOS launchd run proving Node parses the generated shell script instead of executing it.

## Solution Assessment

Unclear: direct `EnvironmentVariables` would address the reported plist shape, but it conflicts with test- and doc-backed owner-only env-file handling. If the crash reproduces, the safer fix is to preserve file-backed env storage and adjust the launchd invocation or wrapper semantics.
