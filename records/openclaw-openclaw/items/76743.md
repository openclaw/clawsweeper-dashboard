## Summary

Keep open: current main still emits the reported wrapper-first LaunchAgent shape for managed macOS service env, and the follow-up discussion confirms that shape is intentional in tests/docs but the live launchd crash path is not yet proven. This needs macOS maintainer validation and a fix shape that preserves the owner-only service-env design if the crash reproduces.

## What This Changes

The issue reports that `openclaw doctor --fix` and `openclaw update` rewrite the macOS gateway LaunchAgent with a generated env-wrapper as the first `ProgramArguments` entry, causing the gateway to crash on launch.

## Best Possible Solution

Confirm the failure on macOS launchd, then fix the LaunchAgent invocation while preserving owner-only service environment storage and add regression coverage for doctor/update rewrites.

## Reproduction Assessment

Unclear: current main source reproduces the reported wrapper-first plist shape whenever managed service env exists, but this review did not establish a high-confidence live path where launchd causes Node to parse the generated shell wrapper.

## Solution Assessment

Unclear: switching to inline `EnvironmentVariables` would likely fix the reported plist shape but would undo documented secret-handling behavior. If the crash reproduces, the narrower maintainable fix is to keep the owner-only env file and make the wrapper invocation launchd-proof.
