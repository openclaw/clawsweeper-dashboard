## Summary

Keep open: current main still writes the wrapper-first LaunchAgent shape when managed service environment exists, and v2026.5.2 shipped the same behavior. The remaining uncertainty is the live macOS launchd failure mode, because docs and tests currently make the owner-only env-file wrapper an intentional secret-handling contract.

## What This Changes

The issue asks doctor/update to stop rewriting macOS Gateway LaunchAgent plists into a wrapper-first `ProgramArguments` shape that allegedly makes Node parse the env-wrapper shell script and crash.

## Best Possible Solution

Confirm the live macOS failure, then preserve managed secrets outside plist metadata while making the generated wrapper invocation launchd-proof and covered by install, doctor, and update regression tests.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection reproduces the wrapper-first plist shape on current main and v2026.5.2 when managed service env exists, but this review did not prove the launchd path where Node parses the wrapper script.

## Solution Assessment

Unclear: the direct `EnvironmentVariables` workaround would address the visible plist shape but conflicts with the current owner-only secret-handling contract. If the crash reproduces, the safer solution is to keep the env file private and make wrapper execution robust under launchd.
