## Summary

Keep open: current main still writes the reported wrapper-first LaunchAgent plist shape when managed service environment values exist, but the live macOS crash mechanism is not proven and the suggested inline `EnvironmentVariables` workaround conflicts with the documented owner-only service-env design.

## What This Changes

The issue requests fixing `openclaw doctor --fix` and update-triggered LaunchAgent rewrites that put the generated env-wrapper at `ProgramArguments[0]` and allegedly crash the gateway on launch.

## Best Possible Solution

Confirm the live macOS launchd failure, then make the generated wrapper invocation launchd-proof while preserving owner-only env-file storage, with regression coverage for doctor and update rewrites.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection reproduces the reported wrapper-first plist shape on current main whenever managed service env exists, but there is not yet high-confidence live macOS proof that launchd causes Node to parse that wrapper.

## Solution Assessment

Is this the best way to solve the issue? Unclear: direct inline `EnvironmentVariables` would address the reported plist shape, but the safer maintainable direction is to preserve owner-only env storage unless maintainers intentionally change that security model.
