## Summary

Keep open: current main and v2026.5.2 do write the reported macOS LaunchAgent env-wrapper shape, but the final Node-parses-wrapper crash is not proven from source alone and the proposed direct EnvironmentVariables fix conflicts with the documented owner-only service-env design.

## What This Changes

The report asks for doctor/update LaunchAgent rewrites to stop placing the generated env-wrapper at ProgramArguments[0], keep the gateway launching on macOS, and avoid resetting throttling to 1 second.

## Best Possible Solution

Confirm the failure on macOS launchd, then fix the LaunchAgent invocation while preserving owner-only service environment storage and add regression coverage for doctor/update rewrites.

## Reproduction Assessment

Unclear: current main source reproduces the reported plist shape whenever managed service env exists, but I did not establish a live macOS crash where Node parses the generated wrapper. The crash path needs launchd validation because current source does not emit a separate Program key pointing at Node.

## Solution Assessment

Unclear: replacing the wrapper with inline EnvironmentVariables would address the reported plist shape but conflicts with the documented service-env secret handling. A safer fix, if the crash reproduces, would keep the owner-only env file and make the wrapper invocation launchd-proof with focused macOS coverage.
