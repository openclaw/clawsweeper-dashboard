## Summary

Keep open: current main and v2026.5.2 still generate wrapper-first macOS LaunchAgent plists when managed service env exists, while current docs/tests show that owner-only wrapper path is intentional for secret handling. The remaining gap is live macOS proof that launchd makes Node parse the wrapper script, plus a maintainer-approved fix shape that preserves the secret-handling contract.

## What This Changes

Fix `openclaw doctor --fix` and `openclaw update` so macOS LaunchAgent rewrites do not leave the gateway crashing from an incorrectly invoked env-wrapper script.

## Best Possible Solution

Confirm the live launchd failure on macOS, then keep owner-only managed env storage while making the wrapper invocation launchd-proof and covering install, doctor repair, and update refresh paths.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection on current main reproduces the wrapper-first `ProgramArguments` shape without a `Program` key when managed env exists, but it does not prove launchd makes Node parse the wrapper script.

## Solution Assessment

Unclear: the inline `EnvironmentVariables` workaround would address the visible plist shape but conflicts with the current secret-handling contract. A safer fix, if the crash reproduces, is to preserve owner-only env storage and make wrapper execution robust under launchd.
