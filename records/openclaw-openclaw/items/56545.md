## Summary

Keep open. The feature is still useful and current main does not implement failure-alert agent turns, but this PR is stale/incomplete across the current cron service, CLI, Gateway protocol, update normalizer, and agent-tool surfaces, so it should not merge as-is.

## What This Changes

The PR widens cron failure-alert and failure-destination mode types/config schemas to include `agent-turn`, adds a timer failure-alert branch that enqueues a system event and wakes heartbeat, and updates CLI help plus a gateway guard.

## Best Possible Solution

Land a rebased implementation that supports `failureAlert.mode: "agent-turn"` end to end across config, CLI, Gateway protocol, cron update normalization, agent tool schema, targeted heartbeat wake, focused tests, and changelog; keep `failureDestination` announce/webhook-only unless separately designed.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the advertised CLI value is rejected by the parser, Gateway schemas reject the value, cron update normalization drops it, and the PR's timer wake call does not match current main's cron service dependency contract.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best current fix because it widens public contracts before all entry points and dispatch paths agree; a narrow failure-alert-only implementation with targeted wake semantics is safer.
