## Best Possible Solution

Rebase onto current main and implement `failureAlert.mode: "agent-turn"` end to end with targeted heartbeat, untrusted event handling, focused tests, and explicit maintainer scope for whether `failureDestination` should support agent turns.

  - body: After widening `failureDestination.mode`, returning `null` for `agent-turn` makes the gateway treat it like no explicit failure destination and fall back to the primary announce path, which can send an outbound alert instead of the configured agent turn. Either remove this mode from failureDestination or implement that path end to end.
  - confidence: 0.89
- **[P1] Use the current targeted heartbeat wake contract:** `src/cron/service/timer.ts:353-354`
  - body: The branch calls `state.deps.requestHeartbeatNow`, but current `CronServiceDeps` exposes `requestHeartbeat`; the new wake also needs the failing job's agent/session target so a single cron failure does not wake unrelated agents. Rebase onto the current dependency contract and pass the target through the wake request.
  - confidence: 0.91
- **[P1] Downgrade cron failure text before injecting it:** `src/cron/service/timer.ts:352-353`
  - body: Cron error text can include tool, provider, or runtime output, but this branch enqueues it with only `{ agentId }`, which current gateway cron wiring treats as trusted owner content. Mark the event as untrusted, for example with `forceSenderIsOwnerFalse: true`, before waking the agent.
  - confidence: 0.86
