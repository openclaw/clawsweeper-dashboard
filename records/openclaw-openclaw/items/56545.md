## Summary

Keep open. The feature direction is useful and current main does not implement failure-alert agent turns, but this PR is incomplete against the current cron, CLI, Gateway protocol, and agent-tool surfaces, so it should not merge as-is.

## What This Changes

The PR widens cron failure-alert and failure-destination modes to include `agent-turn`, adds a timer branch that enqueues a system event and wakes heartbeat, and updates CLI help plus a gateway guard.

## Best Possible Solution

Land a rebased implementation that supports `failureAlert.mode: "agent-turn"` end to end across config, CLI, Gateway protocol, cron update normalization, agent tool schema, targeted heartbeat wake, tests, docs or help text, and changelog; keep `failureDestination` announce/webhook-only unless a separate dispatch design is approved.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the advertised CLI value is rejected by the parser, Gateway schemas reject the value, and the cron update normalizer drops it even if earlier entry points are fixed.

## Solution Assessment

No. The direction is reasonable, but this patch widens public contracts before all entry points and dispatch paths agree; the narrower maintainable fix is failure-alert-only agent-turn support with a targeted heartbeat wake and focused coverage.

  - body: The help text advertises `agent-turn`, but the action handler still throws unless the mode is `announce` or `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` command therefore fails before sending `cron.update`.
  - confidence: 0.96

  - body: The PR accepts `agent-turn` in cron config/types, but Gateway RPC schemas still validate failure alert and failure destination modes as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value are rejected.
  - confidence: 0.94
- **[P1] Preserve agent-turn in cron update patches:** `src/cron/service/jobs.ts:867-869`
  - body: Even after parser and protocol validation are updated, `mergeCronFailureAlert` rewrites any mode other than `announce` or `webhook` to `undefined`. Editing an existing job to `agent-turn` would not persist the requested mode.
  - confidence: 0.93
- **[P2] Expose agent-turn in the cron tool schema:** `src/agents/tools/cron-tool.ts:235`
  - body: The agent-facing cron tool schema still limits `failureAlert.mode` to `announce` and `webhook`. Agents using the tool cannot create or update jobs with the new failure-alert mode.
  - confidence: 0.9
- **[P2] Target the heartbeat wake to the failing job:** `src/cron/service/timer.ts:459`
  - body: The new branch enqueues the event for the job's agent but wakes heartbeat with only a reason. Untargeted wakes are global in the heartbeat runner, so a single cron failure can wake unrelated agents instead of the affected job session.
  - confidence: 0.9

  - body: The PR widens `failureDestination.mode` to `agent-turn`, but the failure-destination resolver and gateway dispatcher are still announce/webhook-shaped. Returning no plan for this mode can fall back to primary announce delivery, which contradicts the configured agent-turn destination.
  - confidence: 0.88

  - body: This is a user-facing cron feature with new CLI/config behavior, so repo policy requires an Unreleased changelog entry with contributor credit before merge.
