## Summary

Keep open. The feature is not implemented on current main, and this PR is still too partial to merge: the advertised mode is rejected or dropped by current CLI, Gateway schema, job normalization, and agent tool surfaces, and the branch uses an obsolete heartbeat dependency without real behavior proof.

## What This Changes

This PR adds an `agent-turn` cron failure-alert mode across cron/config types plus CLI help and a timer branch that would enqueue the failure into the agent session.

## Best Possible Solution

Land a rebased, failure-alert-focused implementation that updates config, CLI, Gateway protocol, job normalization, agent tool schema, docs, focused tests, and the current targeted `requestHeartbeat` path; keep `failureDestination` semantics separate unless maintainers choose that expansion.

## Reproduction Assessment

Yes. Source inspection against current main shows the advertised mode is rejected or dropped by the CLI, Gateway schema, job normalization, and agent tool schema, while the PR calls an obsolete heartbeat dependency.

## Solution Assessment

No. The feature direction is plausible, but this branch is too partial to merge; the narrower maintainable path is a rebased failure-alert implementation using the current targeted heartbeat contract and a separate decision on `failureDestination`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current targeted heartbeat API:** `src/cron/service/timer.ts:356`
  - body: The new branch calls `state.deps.requestHeartbeatNow`, but current main exposes `requestHeartbeat` and routes wake targeting through agent/session options. As written after rebase this will not typecheck, and an untargeted wake can run unrelated agents instead of only the failing cron job's target.
  - confidence: 0.94
- **[P1] Keep the failure-alert sender contract narrow:** `src/gateway/server-cron.ts:304-306`
  - body: The new guard compares `mode` with `"agent-turn"` inside `sendCronFailureAlert`, but that dependency still receives only `announce | webhook` after the timer branch returns early. This can produce a type error after rebase and does not implement the failure-destination path; remove the guard or widen and implement the contract end to end.
  - confidence: 0.86
- **[P1] Accept agent-turn before advertising it in cron edit:** `src/cli/cron-cli/register.cron-edit.ts:77`
  - body: The help text advertises `agent-turn`, but the action handler still rejects every value except `announce` and `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` path therefore fails before it sends `cron.update`.
  - confidence: 0.96
- **[P1] Update the Gateway failure-alert schema:** `src/gateway/protocol/schema/cron.ts:206`
  - body: The PR widens cron/config types to accept `agent-turn`, but `CronFailureAlertSchema` still validates the mode as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value are rejected at protocol validation.
  - confidence: 0.94
- **[P1] Preserve agent-turn when applying failure-alert patches:** `src/cron/service/jobs.ts:982`
  - body: Even if parser and protocol validation accept the value, `mergeCronFailureAlert` rewrites any mode other than `announce` or `webhook` to `undefined`. Editing an existing job to `agent-turn` would not persist the requested mode.
  - confidence: 0.93
- **[P2] Do not widen failureDestination without an end-to-end path:** `src/cron/delivery.ts:205-209`
  - body: The PR adds `agent-turn` to failure-destination types and then returns `null` for that mode. In the Gateway failure-destination dispatcher, `null` means no explicit failure destination and can fall back to the primary announce path, contradicting the configured `agent-turn` semantics.
  - confidence: 0.88
- **[P2] Expose the mode in the agent cron tool schema:** `src/agents/tools/cron-tool.ts:245`
  - body: The agent-facing cron tool schema still limits `failureAlert.mode` to `announce` and `webhook`. Agents using the tool cannot create or update jobs with the new failure-alert mode even after the config and CLI surfaces are updated.
  - confidence: 0.9
