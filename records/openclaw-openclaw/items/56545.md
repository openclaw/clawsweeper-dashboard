## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Update the parser before advertising agent-turn:** `src/cli/cron-cli/register.cron-edit.ts:80`
  - body: The help text now advertises `agent-turn`, but the action handler still rejects any `--failure-alert-mode` other than `announce` or `webhook`. The documented command in the PR body therefore fails at runtime instead of applying the new mode.
  - confidence: 0.97
- **[P1] Sync the gateway protocol before widening cron types:** `src/cron/types.ts:79`
  - body: This widens `CronFailureAlert.mode` to include `agent-turn`, but the gateway protocol schema still validates `failureAlert.mode` as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` calls carrying the new value will be rejected until `packages/gateway-protocol/src/schema/cron.ts` and its validators are updated.
  - confidence: 0.94
- **[P2] Do not treat agent-turn failureDestination as unset:** `src/cron/delivery.ts:209-210`
  - body: Returning `null` for `delivery.failureDestination.mode: "agent-turn"` makes the gateway interpret the explicit failure destination as absent, so it can fall through to the primary announce notification path instead of honoring the configured agent-turn semantics. Either implement this mode end-to-end for failureDestination or keep that surface announce/webhook-only.
  - confidence: 0.9
- **[P1] Target the heartbeat wake to the failing job:** `src/cron/service/timer.ts:356`
  - body: The new branch enqueues the event for the job agent but wakes heartbeat with only a reason. Current cron wake paths carry `agentId` and `sessionKey`; without that target, one cron failure can wake unrelated configured agents or sessions.
  - confidence: 0.86
