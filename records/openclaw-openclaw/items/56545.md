## Summary

Keep open: current main still lacks the requested cron failure-alert mode, and this PR remains the canonical implementation thread after the linked feature issue was closed in favor of it. The branch still has blocking stale-head and contract gaps before it can merge.

## What This Changes

The branch adds an `agent-turn` cron `failureAlert.mode` across cron/config types, config validation, delivery resolution, timer failure-alert dispatch, Gateway failure-alert handling, and CLI help.

## Best Possible Solution

Land a rebased, failure-alert-only implementation that supports `failureAlert.mode: "agent-turn"` end to end across config, CLI, Gateway protocol, update normalization, agent tool schema, targeted heartbeat wake, and focused tests; keep `failureDestination` announce/webhook-only unless separately designed.

## Reproduction Assessment

Yes. Source inspection against current main shows the advertised mode is still rejected or dropped in the CLI parser, Gateway schemas, cron update normalization, and agent tool schema, and the PR branch calls an obsolete heartbeat dependency.

## Solution Assessment

No. The feature direction is reasonable, but this patch is too partial and stale; the maintainable path is a rebased failure-alert-only implementation with targeted `requestHeartbeat` and no `failureDestination` widening unless that API is explicitly designed.

  - body: The new branch calls `state.deps.requestHeartbeatNow`, but current main's `CronServiceDeps` exposes `requestHeartbeat`. A rebase will fail typecheck here; use the typed wake request and scope it to the failing job's agent/session.

  - body: The option help advertises `agent-turn`, but the action handler still throws unless the mode is `announce` or `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` command therefore fails before sending `cron.update`.
  - confidence: 0.96
- **[P1] Sync the Gateway cron schemas:** `src/gateway/protocol/schema/cron.ts:206-217`
  - body: The PR widens cron config/types to accept `agent-turn`, but the Gateway RPC schemas still validate failure alert and failure destination modes as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value are rejected.
  - confidence: 0.94
- **[P1] Preserve agent-turn in cron update patches:** `src/cron/service/jobs.ts:935-936`
  - body: Even after parser and protocol validation are updated, cron update normalization rewrites any mode other than `announce` or `webhook` to `undefined`. Editing an existing job to `agent-turn` would not persist the requested mode.
  - confidence: 0.93
- **[P2] Keep failureDestination announce/webhook-only unless implemented:** `src/cron/delivery.ts:205-209`
  - body: The PR widens `failureDestination.mode` to `agent-turn`, but the failure-destination resolver and dispatcher are still announce/webhook-shaped. Returning no plan for this mode can fall back to primary announce delivery, contradicting the configured agent-turn semantics.
  - confidence: 0.88
- **[P2] Expose agent-turn in the cron tool schema:** `src/agents/tools/cron-tool.ts:235`
  - body: The agent-facing cron tool schema still limits `failureAlert.mode` to `announce` and `webhook`. Agents using the tool cannot create or update jobs with the new failure-alert mode.
  - confidence: 0.9
