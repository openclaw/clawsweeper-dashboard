## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Wire the advertised mode through CLI parsing:** `src/cli/cron-cli/register.cron-edit.ts:77`
  - body: The help text advertises `agent-turn`, but the command action still needs to accept and serialize that value wherever `--failure-alert-mode` is parsed. Without that, the documented CLI command cannot create the new mode.
  - confidence: 0.86
- **[P1] Update the gateway cron protocol schema:** `src/config/zod-schema.ts:556`
  - body: The PR adds `agent-turn` to cron config/runtime types, but gateway-backed `cron.add` and `cron.update` requests are still validated by the protocol schema that only accepts `announce` or `webhook`. The new mode must be added there, or the runtime/config type change is not usable through the public gateway contract.
  - confidence: 0.9
- **[P1] Do not accept agent-turn for failureDestination without routing:** `src/cron/delivery.ts:205-209`
  - body: Adding `agent-turn` to `failureDestination.mode` creates a second contract for failure routing, but this destination path is an outbound notification destination today. Either implement that path all the way through the Gateway finished-notification dispatcher or keep `failureDestination` limited to `announce` and `webhook` so configured failure alerts are not dropped or rerouted unexpectedly.
  - confidence: 0.88
- **[P1] Scope the wake to the cron failure target:** `src/cron/service/timer.ts:353-355`
  - body: The new agent-turn branch enqueues the event for a job agent but calls the heartbeat wake without an agent/session target. In multi-agent or session-targeted cron setups, the wake should be scoped through the same target resolution path as the event so one failed job does not wake unrelated agents or sessions.
  - confidence: 0.84
