## Summary

Keep open: current main does not implement the requested failure-alert mode, and this PR is still too partial/outdated to merge because it lacks real behavior proof and misses current CLI, protocol, persistence, and targeted wake contracts.

## What This Changes

The PR widens cron failure-alert and failure-destination mode types to include `agent-turn`, advertises it in `cron edit`, and adds a timer branch intended to enqueue a system event instead of sending a failure notification.

## Best Possible Solution

Land a rebased, failure-alert-focused implementation that updates CLI parsing, config and Gateway schemas, job normalization, agent tool schema/docs/tests, and the current targeted `requestHeartbeat` path; keep `failureDestination` expansion separate unless maintainers explicitly choose that API.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the advertised value is still rejected or dropped by CLI parsing, Gateway protocol validation, job normalization, and the current heartbeat dependency contract. I did not run runtime tests because this was a read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the narrowest maintainable solution until it is rebased and implements `failureAlert` end to end with the current targeted wake contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current targeted heartbeat contract:** `src/cron/service/timer.ts:356`
  - body: The new branch calls `state.deps.requestHeartbeatNow`, but current `CronServiceDeps` exposes `requestHeartbeat(opts)` and Gateway resolves agent/session targeting through that request. After rebase this will not typecheck, and the wake needs to target the failing job's agent/session instead of using the obsolete API.
  - confidence: 0.94
- **[P1] Accept agent-turn before advertising it in cron edit:** `src/cli/cron-cli/register.cron-edit.ts:77`
  - body: The help text advertises `agent-turn`, but the action handler still rejects every value except `announce` and `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` path therefore fails before sending `cron.update`.
  - confidence: 0.96
- **[P1] Update the Gateway failure-alert schema:** `src/config/zod-schema.ts:556`
  - body: The PR widens config types and config validation, but the Gateway `CronFailureAlertSchema` still validates the mode as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value will be rejected at protocol validation.
  - confidence: 0.94
- **[P1] Preserve agent-turn when applying failure-alert patches:** `src/cron/types.ts:75-76`
  - body: The type change says `agent-turn` is valid, but `mergeCronFailureAlert` still rewrites any mode other than `announce` or `webhook` to `undefined`. Editing an existing job to the new mode would not persist the requested value.
  - confidence: 0.93
- **[P2] Do not widen failureDestination without an end-to-end path:** `src/cron/delivery.ts:205-209`
  - body: The PR adds `agent-turn` to failure-destination types and then returns `null` for that mode. In the Gateway failure-destination flow, `null` means no explicit failure destination and can fall back to primary announce delivery, contradicting the configured agent-turn semantics.
  - confidence: 0.88
- **[P2] Keep the failure-alert sender contract narrow:** `src/gateway/server-cron.ts:303-306`
  - body: The new guard compares the sender's `mode` with `"agent-turn"`, but that dependency should only receive `announce | webhook` after the timer branch handles agent-turn directly. Remove the unreachable guard or widen and implement the sender contract end to end.
  - confidence: 0.86
