## Best Possible Solution

Land a current-main implementation that deliberately supports `agent-turn` for the chosen failure-alert scope across CLI, config, gateway protocol, tool schema, cron service normalization, targeted heartbeat wake, trust metadata, regression tests, and real behavior proof.

  - confidence: 0.93

  - body: Adding `agent-turn` to config validation is not enough: the gateway protocol schemas and cron tool schemas still expose only `announce | webhook`, so gateway-backed clients and agent tool calls will reject or omit the new value. The new mode needs to be added across those contracts with focused tests.
  - confidence: 0.9
- **[P1] Do not turn agent-turn failureDestination into fallback delivery:** `src/cron/delivery.ts:207`
  - body: `resolveFailureDestination` now accepts `agent-turn` and returns `null`, which the gateway failure-notification flow treats as no explicit failure destination and can fall back to the job's primary outbound announce target. A configured agent-turn destination should either be implemented end to end or rejected, not silently converted into different delivery semantics.
  - confidence: 0.88

  - body: Cron failure messages can include tool, provider, webhook, or runtime output, but this enqueue call does not mark the event as untrusted. Pass the appropriate trust metadata, such as `forceSenderIsOwnerFalse`, so external error text is not treated as owner/system context in the agent session.
