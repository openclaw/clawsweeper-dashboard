## Summary

Keep this PR open as the canonical implementation thread for the closed feature request, but it is not merge-ready: current main still lacks the requested cron failure-alert agent turn mode, while the branch is stale against the current cron/heartbeat contracts, has end-to-end acceptance gaps, lacks real behavior proof, and needs a trust-boundary fix.

## What This Changes

The PR adds an `agent-turn` cron failure-alert mode across cron config/types and attempts to enqueue failed cron alerts into the agent session instead of sending an outbound notification.

## Best Possible Solution

If maintainers want this feature, land a rebased implementation that supports `failureAlert.mode: "agent-turn"` across CLI, config, protocol, tool schemas, cron service normalization, targeted heartbeat wake, tests, and untrusted event handling; keep `failureDestination` support out unless that separate behavior is explicitly approved.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the branch's advertised mode is still rejected or omitted by current CLI, gateway protocol, and tool schemas, and the branch calls the obsolete heartbeat dependency. I did not run runtime tests because this review is read-only.

## Solution Assessment

No. The feature likely belongs in the cron/gateway boundary if accepted, but this branch is not the best implementation until it is rebased, scoped, schema-complete, targeted to the right heartbeat/session, and explicit about untrusted cron failure text.
