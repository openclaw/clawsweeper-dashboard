## Summary

Keep open: current main and the latest release still only support `announce` and `webhook` for cron failure alerts, and the linked feature issue was closed in favor of this PR. The branch is not merge-ready because the advertised `agent-turn` mode is incomplete across CLI parsing, gateway protocol validation, failure-destination routing, targeted wake behavior, tests, docs, and real behavior proof.

## What This Changes

The branch adds an `agent-turn` cron failure-alert mode across config/types/CLI help and tries to enqueue failed cron alerts into an agent session instead of sending outbound notifications.

## Best Possible Solution

Reroll this as a current-main implementation of `failureAlert.mode: "agent-turn"` that updates config, the gateway-protocol package, CLI parser/help, docs, and regression tests, targets both the system event and heartbeat to the job agent/session, and either leaves `failureDestination` unchanged or implements it end-to-end by explicit maintainer choice.

## Reproduction Assessment

Yes for the patch defects from source inspection: the CLI parser and gateway protocol still reject `agent-turn`, and the PR diff makes `failureDestination.mode: "agent-turn"` resolve to null. No live cron run proof was provided.

## Solution Assessment

No. The feature direction is useful and still canonical, but this branch is too narrow and stale; the best fix is a current-main reroll with one coherent API boundary, targeted wake routing, docs, and tests.
