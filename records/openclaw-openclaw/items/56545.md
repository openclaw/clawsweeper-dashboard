Reroll the feature on current main by implementing one canonical `agent-turn` failure-alert path in the current failure-alert/notification helpers, updating CLI, config schema, gateway protocol schema, docs, and tests together, and proving a real cron failure wakes only the intended agent/session.

Full review comments:

- **[P1] Align CLI parsing with the advertised mode:** `src/cli/cron-cli/register.cron-edit.ts:77`
  - body: This help text advertises `agent-turn`, but the action handler still accepts only `announce` or `webhook`, so the documented `openclaw cron edit <id> --failure-alert-mode agent-turn` command exits with an invalid-mode error instead of applying the feature.
  - confidence: 0.96
- **[P1] Update the gateway cron protocol schemas:** `src/config/zod-schema.ts:556`
  - body: The PR widens config/zod modes to include `agent-turn`, but gateway-backed `cron.add` and `cron.update` validation still accepts only `announce` or `webhook` in `packages/gateway-protocol/src/schema/cron.ts`, so clients cannot create or update jobs with the new mode.
  - confidence: 0.95
- **[P1] Do not treat agent-turn failure destinations as unset:** `src/cron/delivery.ts:205-209`
  - body: Returning `null` for `delivery.failureDestination.mode: "agent-turn"` makes the notification dispatcher behave as if no explicit failure destination was configured, which can fall back to primary announce delivery instead of routing the failure into the agent turn the user selected.
  - confidence: 0.9
- **[P1] Target the heartbeat wake to the failed job:** `src/cron/service/timer.ts:354`
  - body: The new branch enqueues the failure text for one job agent but wakes heartbeat through an untargeted `requestHeartbeatNow` call. In multi-agent setups this can wake unrelated agents, so the wake should carry the same agent/session target as the queued failure event.
  - confidence: 0.86
