Full review comments:

- **[P1] Accept the advertised mode in the edit parser:** `src/cli/cron-cli/register.cron-edit.ts:152`
  - body: The help text now tells users `--failure-alert-mode agent-turn` is valid, but the action handler still rejects anything other than `announce` or `webhook`. That makes the usage documented in the PR body fail before the cron patch is sent, so the CLI path for the feature is unusable.
  - confidence: 0.96
- **[P1] Sync the gateway cron schemas with the new mode:** `src/config/types.cron.ts:20`
  - body: This PR widens the config/types surface to include `agent-turn`, but the gateway cron schemas still validate `failureAlert.mode` and `delivery.failureDestination.mode` as only `announce | webhook`. `cron.add` and `cron.update` use those schemas, so gateway-backed clients will reject the new mode unless the protocol schemas and validator tests are updated too.
  - confidence: 0.92
- **[P1] Target the heartbeat wake to the failed job:** `src/cron/service/timer.ts:354`
  - body: The new branch enqueues the system event for `params.job.agentId`, but wakes heartbeat with only a reason. The heartbeat API supports `agentId` and `sessionKey`, and current gateway cron deps preserve untargeted wakes, so one cron failure can wake unrelated agents in multi-agent setups. Pass the same resolved job target into the wake.
  - confidence: 0.88
- **[P2] Remove or implement agent-turn failureDestination:** `src/cron/delivery.ts:209`
  - body: The PR also accepts `agent-turn` for `CronFailureDestination`, but `resolveFailureDestination` returns `null` for that mode. Downstream code treats `null` as no separate failure destination, so a configured `delivery.failureDestination.mode: "agent-turn"` can fall back to normal delivery instead of injecting an agent event. Keep the new mode on `failureAlert` only, or implement this route end-to-end.
  - confidence: 0.86
