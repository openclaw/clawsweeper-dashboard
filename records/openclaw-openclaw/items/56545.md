## What This Changes

The PR adds cron `failureAlert.mode: "agent-turn"` to config/types/help and attempts to route cron failures into the agent session with `enqueueSystemEvent` plus a heartbeat wake.

## Best Possible Solution

Rebase onto current main and implement `agent-turn` as a single current contract across CLI, config, gateway protocol, service merge/normalization, runtime wake targeting, tests, docs, and trust-boundary handling; keep `failureDestination` outbound-only unless maintainers explicitly approve agent-turn semantics there.

- **[P1] Accept the advertised agent-turn CLI mode:** `src/cli/cron-cli/register.cron-edit.ts:77`
  - body: This help text advertises `agent-turn`, but the `cron edit` action handler still rejects any `--failure-alert-mode` except `announce` or `webhook`. The documented command therefore fails before it can apply the new mode.
  - confidence: 0.96
- **[P1] Sync gateway and service validators with the new mode:** `src/config/zod-schema.ts:556`
  - body: Adding `agent-turn` to config validation is not enough: current gateway cron schemas still accept only `announce | webhook`, and service merge paths still strip any other failure-alert or failure-destination mode. `cron.add`/`cron.update` will reject or drop the advertised value.
  - confidence: 0.94
- **[P1] Do not treat agent-turn failureDestination as unset:** `src/cron/delivery.ts:205-207`
  - body: Returning `null` for `agent-turn` makes the notification dispatcher treat the job as having no explicit failure destination and fall back to the primary announce target. A configured agent-turn failure destination can therefore still send an outbound alert instead of triggering the agent.
  - confidence: 0.92
- **[P1] Use the current targeted heartbeat API:** `src/cron/service/timer.ts:353`
  - body: Current main no longer has the `requestHeartbeatNow` dependency used here; cron services receive `requestHeartbeat` with `source`, `intent`, `agentId`, and `sessionKey`. The wake also needs to target the failing job's agent/session instead of queuing an untargeted global wake.
  - confidence: 0.95
- **[P1] Wrap cron failure text as untrusted content:** `src/cron/service/timer.ts:352`
  - body: Cron failure text can include tool output, provider errors, fetched content, or other attacker-controlled strings. Injecting it directly as a trusted system event lets that text look like operational instruction to a tool-enabled agent; preserve an explicit untrusted-content boundary before waking the agent.
  - confidence: 0.86
