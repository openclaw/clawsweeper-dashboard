## Summary

Keep this PR open, but it is not mergeable as-is: current main still lacks the requested cron failure-alert `agent-turn` mode, while the branch is partial, stale against the current cron wake contract, and missing required real behavior proof.

## What This Changes

The PR adds an `agent-turn` cron failure-alert mode across cron/config types, config validation, CLI help, failure-destination planning, and timer delivery so failed cron jobs can enqueue a system event for the agent session.

## Best Possible Solution

Land a rebased, failure-alert-focused implementation that updates config, CLI parsing, Gateway protocol, job normalization, agent tool schema, docs, focused tests, and the current targeted heartbeat path; keep `failureDestination` expansion separate unless maintainers explicitly choose it.

## Reproduction Assessment

Yes. Source inspection against current main shows the advertised mode is rejected or dropped by CLI parsing, config/Gateway schemas, job normalization, and agent tool schema, while the PR calls an obsolete heartbeat dependency.

## Solution Assessment

No. The feature direction is plausible, but this branch is too partial and outdated to merge; the narrower maintainable path is a rebased failure-alert implementation using the current targeted wake contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current targeted heartbeat contract:** `src/cron/service/timer.ts:356`
  - body: The new branch calls `state.deps.requestHeartbeatNow`, but current `CronServiceDeps` exposes `requestHeartbeat(opts)` and Gateway resolves agent/session targeting through that request. After rebase this will not typecheck, and the wake needs to target the failing job's agent/session rather than use an obsolete API.
  - confidence: 0.94
- **[P1] Accept agent-turn before advertising it in cron edit:** `src/cli/cron-cli/register.cron-edit.ts:77`
  - body: The help text advertises `agent-turn`, but the action handler still rejects every value except `announce` and `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` path therefore fails before sending `cron.update`.
  - confidence: 0.96
- **[P1] Update the Gateway failure-alert schema:** `src/config/zod-schema.ts:556`
  - body: The PR widens cron/config types to accept `agent-turn`, but `CronFailureAlertSchema` still validates the mode as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value are rejected at protocol validation.
  - confidence: 0.94
- **[P1] Preserve agent-turn when applying failure-alert patches:** `src/cron/types.ts:75-76`
  - body: Even if parser and protocol validation accept the value, `mergeCronFailureAlert` still rewrites any mode other than `announce` or `webhook` to `undefined`. Editing an existing job to `agent-turn` would not persist the requested mode.
  - confidence: 0.93
- **[P1] Keep the failure-alert sender contract narrow:** `src/gateway/server-cron.ts:303-306`
  - body: The new guard compares `mode` with `"agent-turn"` inside `sendCronFailureAlert`, but that dependency still receives only `announce | webhook` after the timer branch returns early. Remove the guard or widen and implement the sender contract end to end.
  - confidence: 0.86
- **[P2] Do not widen failureDestination without an end-to-end path:** `src/cron/delivery.ts:205-209`
  - body: The PR adds `agent-turn` to failure-destination types and then returns `null` for that mode. In the Gateway failure-destination dispatcher, `null` means no explicit failure destination and can fall back to primary announce delivery, contradicting the configured `agent-turn` semantics.
  - confidence: 0.88
