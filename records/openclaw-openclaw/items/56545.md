## Summary

Keep open for contributor and maintainer follow-up: current main still lacks the requested cron `agent-turn` failure-alert mode and the paired issue was closed in favor of this PR, but the branch is stale, incomplete across cron/Gateway/tool contracts, and has no real behavior proof.

## What This Changes

The PR adds a cron `failureAlert.mode: "agent-turn"` value across cron/config types and config validation, plus a timer branch that enqueues failure text into the agent session and wakes the heartbeat.

## Best Possible Solution

Land a rebased, failure-alert-only implementation that supports `agent-turn` end to end across config, CLI, Gateway protocol, cron update normalization, agent tool schema, targeted heartbeat wake, docs, and focused tests; keep `failureDestination` announce/webhook-only unless separately designed.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the blocking gaps: the advertised value is still rejected by the CLI parser and Gateway schemas, cron update normalization drops it, and the PR calls an obsolete heartbeat dependency.

## Solution Assessment

No. The feature direction is plausible, but this branch is too partial and stale; the narrow maintainable path is a rebased `failureAlert` implementation using the current targeted `requestHeartbeat` contract and no `failureDestination` widening without explicit API design.

  - body: The added branch calls `state.deps.requestHeartbeatNow`, but current main's `CronServiceDeps` exposes `requestHeartbeat`. A rebase will not typecheck here, and the wake should carry the failing job's agent/session target instead of issuing an untargeted heartbeat.
  - confidence: 0.94

  - body: The help text advertises `agent-turn`, but the action handler still rejects every mode except `announce` and `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` command therefore fails before sending `cron.update`.
  - confidence: 0.96
- **[P1] Update the Gateway cron validation contract:** `src/gateway/protocol/schema/cron.ts:206-217`
  - body: The PR widens cron config/types to accept `agent-turn`, but the Gateway RPC schemas still validate failure-alert and failure-destination modes as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value are rejected.
  - confidence: 0.94
- **[P1] Preserve agent-turn when applying failure-alert patches:** `src/cron/service/jobs.ts:980-982`
  - body: Even after parser and protocol validation are updated, cron update normalization rewrites any failure-alert mode other than `announce` or `webhook` to `undefined`. Editing an existing job to `agent-turn` would not persist the requested mode.
  - confidence: 0.93
- **[P2] Do not widen failureDestination without an end-to-end path:** `src/cron/delivery.ts:205-209`
  - body: The PR widens `failureDestination.mode` to `agent-turn`, then returns `null` for that mode. That can be interpreted as no explicit failure destination and fall back to outbound announce delivery, contradicting the configured semantics.
  - confidence: 0.88
- **[P2] Expose the mode in the agent cron tool schema:** `src/agents/tools/cron-tool.ts:245`
  - body: The agent-facing cron tool schema still limits `failureAlert.mode` to `announce` and `webhook`. Agents using the tool cannot create or update jobs with the new failure-alert mode.
