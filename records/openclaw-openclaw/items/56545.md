## Summary

Keep open: this is still the canonical implementation thread for the requested cron failure-alert feature, but the branch is incomplete and stale against current main, with public contract gaps, an obsolete heartbeat API call, and missing real behavior proof.

## What This Changes

The PR adds a cron `failureAlert.mode: "agent-turn"` option across cron/config types and attempts to enqueue cron failure text into the agent session with an immediate heartbeat wake.

## Best Possible Solution

Land a rebased, failure-alert-only implementation that supports `failureAlert.mode: "agent-turn"` end to end across config, CLI, Gateway protocol, cron update normalization, agent tool schema, targeted heartbeat wake, docs, and focused tests; keep `failureDestination` announce/webhook-only unless separately designed.

## Reproduction Assessment

Yes. Source inspection against current main shows the advertised mode is still rejected or dropped by the CLI parser, Gateway schemas, cron update normalization, and agent tool schema, while the branch calls an obsolete heartbeat dependency.

## Solution Assessment

No. The feature direction is plausible, but this patch is too partial and stale; the narrower maintainable path is a rebased `failureAlert` implementation using the current targeted `requestHeartbeat` contract and no `failureDestination` widening without explicit API design.

- **[P1] Use the current cron heartbeat dependency:** `src/cron/service/timer.ts:356`
  - body: The new branch calls `state.deps.requestHeartbeatNow`, but current main's `CronServiceDeps` exposes `requestHeartbeat`. A rebase will fail typecheck here; use the typed wake request and target it to the failing job's agent/session.
  - confidence: 0.94
- **[P1] Accept agent-turn in the cron edit parser:** `src/cli/cron-cli/register.cron-edit.ts:125`
  - body: The option help advertises `agent-turn`, but the action handler still rejects every mode except `announce` and `webhook`. The documented `openclaw cron edit ... --failure-alert-mode agent-turn` command therefore fails before sending `cron.update`.
  - confidence: 0.96
- **[P1] Sync the Gateway cron schemas:** `src/gateway/protocol/schema/cron.ts:206-217`
  - body: The PR widens cron config/types to accept `agent-turn`, but the Gateway RPC schemas still validate failure alert and failure destination modes as only `announce | webhook`. Gateway-backed `cron.add` and `cron.update` requests carrying the new value are rejected.
  - confidence: 0.94
- **[P1] Preserve agent-turn in failure alert updates:** `src/cron/service/jobs.ts:980-982`
  - body: Even after parser and protocol validation are updated, cron update normalization rewrites any failure-alert mode other than `announce` or `webhook` to `undefined`. Editing an existing job to `agent-turn` would not persist the requested mode.
  - confidence: 0.93
- **[P2] Keep failureDestination modes supported end to end:** `src/cron/delivery.ts:205-209`
  - body: The PR widens `failureDestination.mode` to `agent-turn`, but the failure-destination resolver and dispatcher are still announce/webhook-shaped. Returning no plan for this mode can fall back to primary announce delivery, contradicting the configured agent-turn semantics.
  - confidence: 0.88
- **[P2] Expose agent-turn in the cron tool schema:** `src/agents/tools/cron-tool.ts:245`
  - body: The agent-facing cron tool schema still limits `failureAlert.mode` to `announce` and `webhook`. Agents using the tool cannot create or update jobs with the new failure-alert mode.
  - confidence: 0.9
