## What This Changes

The PR threads `agents.list[].identity.name` into runtime prompt params and renders it as the visible Runtime agent label while retaining the slot id as `agent_id`.

## Best Possible Solution

Land the narrow prompt plumbing that resolves `identity.name` through the existing agent config resolver, displays it first, and keeps the stable slot id for diagnostics.

## Reproduction Assessment

Yes, at source level: current main only carries `agentId` into runtime prompt metadata and renders it as `agent=...`, so a config with id `main` and identity name `Runt` cannot display `Runt` there. I did not run an interactive chat path in this read-only review.

## Solution Assessment

Yes. The diff uses the existing normalized agent resolver to read `identity.name`, renders it as the visible runtime agent label, and keeps `agent_id` for diagnostics; adding a broader prompt prefix/suffix config is not needed for this bug.
