## Summary

Keep this PR open: it targets the right runtime prompt bug and includes real after-fix output, but the new identity-name resolver bypasses OpenClaw's normalized agent-id lookup and can miss supported configured agents.

## What This Changes

The branch threads `agents.list[].identity.name` into runtime prompt params and renders it as `agent=...` while preserving the technical slot id as `agent_id=...`.

## Best Possible Solution

Use the existing normalized agent config resolver to carry `identity.name` into runtime metadata, then render the identity first while retaining the stable slot id for diagnostics.

## Reproduction Assessment

Yes, at source level: current main builds the Runtime line from `agentId` only, so a distinct configured `identity.name` cannot appear there. I did not run a live Telegram chat reproduction.

## Solution Assessment

No, not quite: threading `identity.name` is the right narrow fix, but the resolver should follow the existing normalized agent lookup contract so supported ids still resolve after normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use normalized agent lookup for identity names:** `src/agents/system-prompt-params.ts:69`
  - body: This direct trim comparison bypasses the canonical `normalizeAgentId`/`resolveAgentConfig` path. If config has a valid mixed-case id such as `Work`, runtime code normalizes the selected agent id to `work`, so this lookup returns no identity name and the Runtime line falls back to `agent=work`, leaving the reported identity drift unresolved for that supported config. Reuse `resolveAgentConfig(config, agentId)` or normalize both sides before reading `identity.name`.
  - confidence: 0.86
