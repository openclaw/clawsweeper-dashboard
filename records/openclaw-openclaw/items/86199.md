## Summary

Keep open: the issue is protected by maintainer author/label state, and current main still has the stale optimistic-submit path described by the report.

## What This Changes

The issue asks for TUI Esc abort and session switching to clear pending optimistic submit state so later prompts are not blocked as busy.

## Best Possible Solution

Clear pendingOptimisticUserMessage wherever the same run is successfully aborted and when switching sessions, then add focused TUI session-action regression coverage for pendingChatRunId plus pendingOptimisticUserMessage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start with pendingChatRunId and pendingOptimisticUserMessage set, let abortActive resolve successfully, and the code clears only pendingChatRunId while the submit guard still treats pendingOptimisticUserMessage as busy.

## Solution Assessment

Yes. The narrow maintainable fix is to reset the optimistic pending flag together with the pending run id on successful abort and session switch, with regression tests around that state tuple.
