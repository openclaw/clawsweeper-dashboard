## Summary

Keep open. The direction matches the linked watchdog request, but the patch has a concrete reliability bug: after one `systemd-notify` failure it disables all future watchdog pings even though `WatchdogSec` will then fail and restart the service; the external proof also does not exercise a real Linux systemd watchdog path.

## What This Changes

The branch adds a `systemd-notify` helper, wires gateway maintenance ticks to send readiness/watchdog notifications, renders `WatchdogSec`/`NotifyAccess` in generated units, and updates docs, tests, and changelog.

## Best Possible Solution

Keep the PR open, make watchdog notifications retryable or gate `WatchdogSec` on a notifier known to work, and add redacted Linux systemd proof for the heartbeat/restart path.

## Reproduction Assessment

Partially. Source inspection of current main shows the generated systemd unit lacks watchdog directives and the gateway tick lacks a systemd notification path, but there is no live reproduction here of a hung Linux systemd gateway being recovered.

## Solution Assessment

No. The proposed shape is close, but permanently disabling notifications after one failure conflicts with the systemd watchdog contract; the safer fix keeps heartbeats recoverable or only enables watchdog units when notification delivery is proven.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep watchdog notifications retryable after failures:** `src/infra/systemd-notify.ts:69-72`
  - body: With `WatchdogSec` in the generated unit, systemd expects future `WATCHDOG=1` pings and will fail/restart the service when they stop. Setting `disabled = true` after the first `systemd-notify` error means one transient failure, bad PATH lookup, or temporary notify-socket issue guarantees all later pings are skipped for an otherwise healthy gateway. Please keep retrying at a bounded cadence or avoid enabling `WatchdogSec` unless notification delivery is known to work.
  - confidence: 0.88
