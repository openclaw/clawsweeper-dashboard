## Summary

Keep this PR open: it addresses a valid linked systemd watchdog feature, but the patch uses an unsupported `systemd-notify --watchdog` option, so the real heartbeat path would fail and disable notifications. The supplied proof is also renderer/no-op terminal output, not Linux systemd heartbeat proof.

## What This Changes

Adds systemd watchdog directives to generated and documented gateway units, a `systemd-notify` helper, maintenance-tick notifications, tests, docs, and a changelog entry.

## Best Possible Solution

Use the supported `WATCHDOG=1` notification payload while keeping the narrow `WatchdogSec`/`NotifyAccess` unit change, then prove on Linux systemd that the user service accepts heartbeat pings from the 30s tick.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and dependency inspection: current main lacks the watchdog directives/notifier, and the PR's proposed watchdog path fails locally with `systemd-notify: unrecognized option '--watchdog'`; I did not freeze a live gateway.

## Solution Assessment

Is this the best way to solve the issue? No as written: using systemd's watchdog from the existing tick is likely the right narrow design, but the implementation must send `WATCHDOG=1` through `systemd-notify` and be validated in a real Linux systemd service.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send watchdog pings with WATCHDOG=1:** `src/infra/systemd-notify.ts:77`
  - body: The real heartbeat path currently runs `systemd-notify --watchdog`, but systemd-notify 255 and the latest freedesktop man page do not define that option; locally it exits with `unrecognized option '--watchdog'`. With `NOTIFY_SOCKET` set, the catch disables the notifier after the first ping, so the generated `WatchdogSec` unit would never receive keepalives. Send the supported `WATCHDOG=1` status variable instead and update the test to match.
  - confidence: 0.94
