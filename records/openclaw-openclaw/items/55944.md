## Summary

Keep open. The requested visionOS policy support is still missing on current main, but this PR is not ready as-is because it predates the current canonical platform-policy shape, bypasses the plugin-owned canvas defaults, misses audit coverage, lacks real behavior proof, and depends on the still-open visionOS app surface.

## What This Changes

The PR adds a `visionos` platform id, default command allowlist, spatial commands, platform matching, and device-family tokens in `src/gateway/node-command-policy.ts`.

## Best Possible Solution

Land a rebased gateway policy change that recognizes `visionos` through the current canonical platform model, keeps canvas defaults plugin-owned, updates audit and focused tests, and coordinates with the accepted visionOS app command surface.

## Reproduction Assessment

Yes, by source inspection. Current main omits `visionos` from `PlatformId` and canonical platform ids, so `platform: "visionos"` normalizes to `unknown`; I did not establish live Vision Pro hardware proof.

## Solution Assessment

No, not as proposed. Gateway platform classification is the right seam, but this branch needs a current-main rebase through plugin-owned canvas defaults, audit/test updates, real behavior proof, and maintainer approval of the new defaults.
