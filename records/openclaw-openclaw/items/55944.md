## Summary

Keep open. Current main still lacks a visionOS node command policy, but this PR cannot land as-is because it predates the canonical platform-id model, puts canvas defaults back into gateway core, omits audit/test follow-through, and still lacks contributor-provided real behavior proof.

## What This Changes

The PR adds a `visionos` platform id, spatial command defaults, platform matching, and device-family matching to `src/gateway/node-command-policy.ts`.

## Best Possible Solution

Land a rebased gateway policy change that adds `visionos` to the current canonical platform model, keeps plugin-owned defaults in plugins, updates audit/tests, and coordinates the default command surface with the visionOS app review.

## Reproduction Assessment

Yes, by source inspection: current main normalizes a non-empty platform string that is not a canonical id to `unknown`, so a `visionos` node would not receive a distinct policy. This is a new-platform feature gap rather than a validated regression in an existing supported platform.

## Solution Assessment

No, not as currently proposed. Gateway platform classification is the right area, but the patch should be rebased onto canonical platform ids, route canvas defaults through the canvas plugin, update audit/tests, and include real device proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Rebase platform matching onto canonical ids:** `src/gateway/node-command-policy.ts:149`
  - body: Current main no longer uses `PLATFORM_PREFIX_RULES`; it accepts exact canonical ids, native labels, and device-family fallbacks with consistency checks. Adding a `visionos` prefix rule on the stale branch will not apply cleanly and would miss the current family-validation path, so add `visionos` through `CANONICAL_PLATFORM_IDS`, `platformMatchesDeviceFamily`, and focused tests instead.
  - confidence: 0.9
- **[P2] Route canvas defaults through the canvas plugin:** `src/gateway/node-command-policy.ts:129`
  - body: Current main registers canvas node commands through the canvas plugin's `defaultPlatforms`, but this PR adds `...CANVAS_COMMANDS` directly inside gateway core. That grants canvas commands even when the canvas plugin is disabled and bypasses the plugin-owned policy boundary; widen the plugin policy type and add `visionos` through `extensions/canvas/index.ts` instead.
  - confidence: 0.88
- **[P3] Include visionOS in audit known-command defaults:** `src/gateway/node-command-policy.ts:128`
  - body: `listKnownNodeCommands` still enumerates only the existing platform ids. With new visionOS defaults, a valid `gateway.nodes.denyCommands` entry such as `spatial.mesh` can be reported as unknown or ineffective even though the policy would default-allow it.
  - confidence: 0.78
