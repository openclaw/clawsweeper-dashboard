## Summary

Keep open. The optimization targets a real hot path, but the patch caches replay policy by config only even though provider runtime lookup and replay hook context are env-sensitive; it also lacks the required changelog entry and the PR head is not green.

## What This Changes

The PR adds a WeakMap/Map cache around `resolveTranscriptPolicy` keyed by config object plus provider, model API, model id, and workspace inputs.

## Best Possible Solution

Keep the optimization only after the cache key matches provider hook runtime semantics, preserves env-sensitive plugin selection, has focused regression coverage, includes the changelog entry, and passes exact-head checks.

## Reproduction Assessment

Yes. A focused unit test can mock `resolveProviderRuntimePlugin` to return different `buildReplayPolicy` output when `env` changes while config/provider/model inputs stay the same; the patched cache would return the first policy because the bucket is only the config object.

## Solution Assessment

No. The safer solution is to key or invalidate on all hook-observable inputs, especially env/plugin-control-plane state, or skip the outer cache for custom env before adding the performance cache.
