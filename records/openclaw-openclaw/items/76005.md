## Summary

Keep open. The performance target is plausible, but the proposed cache boundary is narrower than the provider hook contract and current env-aware runtime-plugin cache, and the production perf change is missing the required changelog entry.

## What This Changes

The PR adds an in-process WeakMap/Map cache around `resolveTranscriptPolicy` in `src/agents/transcript-policy.ts` keyed by config plus provider/model/workspace inputs.

## Best Possible Solution

Keep the optimization only after the cache boundary matches provider hook semantics, add focused env-sensitive regression coverage plus a changelog entry, and then validate the exact PR head.

## Reproduction Assessment

Yes. A focused unit test can mock `resolveProviderRuntimePlugin` or `buildReplayPolicy` to vary by `env` while reusing the same config/provider/model inputs; the patched cache would return the first policy.

## Solution Assessment

No. A config-only outer cache is not the narrowest maintainable fix because current main treats provider runtime resolution as env/plugin-control-plane-sensitive; keying or bypassing the cache for custom env is safer.
