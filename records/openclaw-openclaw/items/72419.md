## Summary

Keep this PR open and do not merge as-is. The branch does not close the linked gateway authentication bug because the shared token/password backend self-pairing bypass remains, and it adds bootstrap tokens to the same pairing-skip path even though current code and docs route bootstrap tokens through constrained device pairing.

## What This Changes

The PR adds `bootstrap-token` as an auth method that can skip local backend self-pairing and adds helper-level tests plus an explanatory comment for that bypass.

## Best Possible Solution

Tighten backend self-pairing so a device-less backend session cannot rely only on client-declared backend identity plus shared token/password auth; if a bypass remains necessary, bind it to a server-verifiable backend-only credential and keep bootstrap tokens on the existing bootstrap pairing and handoff flow with regression tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: call `shouldSkipLocalBackendSelfPairing` with backend `gateway-client`, direct/local loopback locality, no Origin, `sharedAuthOk: true`, and `authMethod: "token"`; current main returns true, and the message handler uses that skip to avoid clearing unbound scopes.

## Solution Assessment

No. This PR is not the best fix because it leaves the shared-secret bypass in place and adds `bootstrap-token` to the bypass instead of enforcing a server-verifiable backend identity or using the constrained bootstrap pairing path.
