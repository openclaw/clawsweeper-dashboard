## Summary

Keep open: the PR does not resolve the linked gateway pairing-bypass bug because it preserves the token/password shared-secret backend skip and adds bootstrap-token to the same pairing-bypass path.

## What This Changes

The PR adds `bootstrap-token` as a backend self-pairing skip auth method and adds helper-level test/comment coverage.

## Best Possible Solution

Bind backend self-pairing bypass to a server-verifiable backend identity or a dedicated backend credential, and keep bootstrap-token handshakes on the existing bootstrap pairing path unless maintainers approve a broader exception with tests.

## Reproduction Assessment

Yes. The helper-level path reproduces it: backend `gateway-client`, loopback/local-equivalent locality, no Origin, valid shared token/password auth, and `sharedAuthOk: true` make the backend skip true; the message handler then avoids the normal scope-clearing/pairing path.

## Solution Assessment

No. The PR adds bootstrap-token to the bypass instead of removing or tightening the shared-secret backend condition, so it is not the narrowest maintainable security fix.
