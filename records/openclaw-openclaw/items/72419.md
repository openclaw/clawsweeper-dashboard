## Summary

Keep open: the PR is a security-sensitive gateway auth change, and the diff does not fix the linked shared-token backend self-pairing bypass while adding bootstrap-token to the same bypass path.

## What This Changes

The PR adds `bootstrap-token` as an auth method that can skip local backend self-pairing and adds helper-level tests plus a comment documenting that bypass.

## Best Possible Solution

Revise the gateway auth design so backend self-pairing is bound to a server-verifiable backend identity or a dedicated backend credential, and keep bootstrap-token handshakes on the existing bootstrap pairing and handoff path unless maintainers explicitly approve a broader backend exception with regression coverage.

## Reproduction Assessment

Yes. The helper-level path is enough to reproduce the reported condition: a loopback backend `gateway-client` with no Origin, shared token/password auth, and `sharedAuthOk: true` makes the self-pairing skip true, and the message handler uses that skip to preserve self-declared scopes.

## Solution Assessment

No. The proposed patch leaves the shared-token/password bypass intact and adds bootstrap-token to the same bypass condition instead of tightening the backend identity check.

## Review Findings
