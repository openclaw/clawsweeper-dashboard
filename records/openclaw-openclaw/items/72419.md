## Summary

Keep open: the latest PR moves the shared token/password backend bypass in the right direction, but it still treats `bootstrap-token` as a backend self-pairing bypass and is stale against current-main helper expectations, so it is not safe to merge without security-owner review and revision.

## What This Changes

The PR narrows local backend self-pairing skip from shared token/password auth to bootstrap-token or device-token credentials and updates helper tests, message-handler comments, and gateway security docs.

## Best Possible Solution

Remove shared token/password from the backend self-pairing bypass, keep bootstrap-token handshakes on the existing bootstrap pairing path unless secops explicitly approves a backend-only bootstrap exception, and rebase/update tests, docs, and changelog to match the chosen container-local contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main reproduces the shared-token backend skip, and the PR's new bootstrap-token branch would make `skipLocalBackendSelfPairing` true before `requirePairing` enforces the existing bootstrap guardrails.

## Solution Assessment

No. Removing shared token/password is the right direction, but adding bootstrap-token to the bypass is not the narrowest maintainable fix because bootstrap-token currently has its own bounded pairing flow.
