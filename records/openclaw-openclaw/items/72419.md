## Summary

Keep open: the reported shared-token/password backend pairing skip is still present on current main, but this PR should not merge as-is because it adds a bootstrap-token pairing bypass and leaves related contract/docs/changelog work incomplete.

## What This Changes

The PR changes the gateway backend self-pairing skip to reject shared token/password auth, allow bootstrap-token/device-token auth for direct-local backend clients, and updates helper tests plus gateway security docs.

## Best Possible Solution

Remove shared token/password from the backend bypass if maintainers accept the compatibility break, keep bootstrap-token on the existing bounded pairing/handoff path unless explicitly approved, and update helper tests, integration tests, docs, and changelog together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main returns true for the shared-token/password backend skip, while the PR patch would make bootstrap-token set the same skip before `requirePairing` runs.

## Solution Assessment

No. The shared-token/password removal is directionally aligned with the report, but adding bootstrap-token to the bypass is not the narrowest maintainable fix because bootstrap tokens already have a constrained pairing and handoff flow.
