## Summary

Keep open: current main still matches Feishu group @mentions against the receiving bot's own `botOpenId`, so the reported app-scoped `open_id` mismatch can still suppress the intended bot before dispatch; https://github.com/openclaw/openclaw/pull/40782 is an open closing PR, so this issue should stay open until that PR or a replacement lands.

## What This Changes

The issue requests a Feishu multi-bot group mention matcher that can recognize a bot when Feishu delivers app-scoped mention `open_id` values that differ from the receiving bot's self `botOpenId`.

## Best Possible Solution

Keep this as the canonical issue until https://github.com/openclaw/openclaw/pull/40782 or a replacement lands a Feishu-validated identity matcher across group admission, content stripping, rich-text post handling, mention-forward/debounce paths, with focused tests and redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention `open_id` differs from `botOpenId` makes `checkBotMentioned` return false, and the require-mention group path returns before dispatch. I did not run live Feishu validation in this read-only review.

## Solution Assessment

Unclear: the affected code path is narrow, but the durable identity source is not settled because the current probe result only supplies `botOpenId`/`botName` and the reporter says `bot/v3/info` did not return `union_id`. The best fix should use a Feishu-supported stable identifier or cross-app alias strategy and prove it in a real multi-bot setup.
