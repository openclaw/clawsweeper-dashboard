## Summary

Keep open: current main and v2026.6.10 still decide Feishu group mentions by the receiving bot's open_id only, so the app-scoped open_id mismatch can still suppress an explicitly mentioned bot. The linked same-author PR remains open, conflicting, and under human review, so this issue should stay open until that path lands or is replaced.

## What This Changes

The issue reports that Feishu group @mentions can be ignored when multiple bot apps share a group because event mention open_id values may not match the receiving bot's self open_id.

## Best Possible Solution

Land one current-main Feishu identity fix that proves a stable bot identity across text mentions, post mentions, mention-forward, debounce merging, slash-command stripping, tests, and redacted live multi-bot proof, then close this issue.

## Reproduction Assessment

Yes, at source level: a Feishu event whose mention open_id differs from botOpenId makes checkBotMentioned return false, which can keep the group requireMention gate from dispatching. I did not run a live multi-bot Feishu tenant.

## Solution Assessment

No landed solution exists yet. The open linked PR is a plausible candidate, but the best fix needs a current-main rewrite around a proven stable Feishu bot identity source before merge.
