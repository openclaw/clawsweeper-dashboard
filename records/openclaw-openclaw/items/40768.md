## Summary

Keep open: current main still gates Feishu group admission on a mention match against the receiving bot's own `botOpenId`, so an app-scoped mention `open_id` mismatch can still drop the intended Feishu bot message before dispatch; the open linked fix PR is an implementation candidate, not a close reason before it lands.

## What This Changes

The issue requests a Feishu mention matcher that recognizes the intended bot in multi-bot group chats even when Feishu delivers app-scoped mention `open_id` values that differ from the receiving bot's self `open_id`.

## Best Possible Solution

Keep this as the canonical issue until https://github.com/openclaw/openclaw/pull/40782 or a replacement lands a Feishu-validated identity matcher across the current group mention paths with focused regression coverage and redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention `open_id` differs from `botOpenId` makes `checkBotMentioned` return false, and the require-mention group path returns before dispatch. I did not run live Feishu validation in this read-only review.

## Solution Assessment

Unclear: the affected code path is narrow, but the durable identity source is not settled because `bot/v3/info` appears not to provide the proposed `union_id`. The best fix should use a Feishu-supported stable identifier or alias-resolution strategy and prove it with a real multi-bot Feishu setup.
