## Summary

Keep open: current main and v2026.6.8 still match Feishu bot mentions only against the receiving bot's `open_id`, so an app-scoped mismatch can still trip the `requireMention` gate and suppress the intended reply. The linked open fix PR remains the right review target, but it is not merged and still needs a verified Feishu identity source plus live multi-bot proof.

## What This Changes

The issue reports that Feishu group @mention filtering can drop messages when multiple bot apps share one group because mention `open_id` values are app-scoped and do not match the receiving bot's self `open_id`.

## Best Possible Solution

Keep this issue as the canonical Feishu multi-bot mention identity bug until a repaired PR uses a verified stable identity or cross-app mapping and proves detection, stripping, debounce, and live Feishu multi-bot behavior.

## Reproduction Assessment

No live high-confidence reproduction was performed in this review. The source path is clear: a Feishu event whose mention `open_id` differs from `botOpenId` makes `checkBotMentioned` false and can cause the group `requireMention` gate to return before dispatch.

## Solution Assessment

No, current main is not the best final state, and the linked PR is only a plausible direction until the identity source is verified. The best fix should choose a maintainable Feishu identity strategy and carry it through detection, stripping, debounce, tests, and live proof.
