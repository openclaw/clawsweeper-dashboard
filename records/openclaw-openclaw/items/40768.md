## Summary

Keep open: current main still recognizes Feishu group mentions by exact `open_id` equality, so the reported app-scoped ID mismatch remains source-reproducible; the issue is also paired with open PR #40782, which references this issue with closing syntax.

## What This Changes

Fix Feishu group mention detection so a bot is recognized when Feishu delivers app-scoped mention IDs that differ from the receiving bot's startup `botOpenId`.

## Best Possible Solution

Keep this open until PR #40782 is reviewed, repaired, or replaced with a Feishu-validated identity matcher that handles app-scoped mention IDs while preserving strict mention gating.

## Reproduction Assessment

Yes, at source level: a group event whose mention metadata contains an `open_id` different from the receiving account's `botOpenId` makes `mentionedBot=false`, and `requireMention` then drops the message. I did not live-test against Feishu.

## Solution Assessment

Unclear. The narrow fix belongs in Feishu mention identity matching, but the linked PR's `union_id`-from-bot-info direction conflicts with the reporter's note and current probe shape, which only exposes `botID`/`botName`.
