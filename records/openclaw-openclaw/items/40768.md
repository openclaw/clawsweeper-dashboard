## Summary

Keep open: current `main` still treats Feishu group mentions as addressed only when the mention `open_id` exactly equals the receiving bot's `botOpenId`, so the reported app-scoped mismatch remains source-reproducible. The issue is also paired with open implementation PR #40782, which references it with closing syntax and should be reviewed, repaired, or replaced before the issue is closed.

## What This Changes

Fix Feishu group mention matching so a bot is recognized when Feishu WebSocket mention `open_id` values are app-scoped and differ from that bot's self `open_id`.

## Best Possible Solution

Keep this open until #40782 is reviewed and either repaired or replaced with a Feishu-validated identity matcher that handles app-scoped mention IDs while preserving strict mention gating.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata contains an `open_id` different from the receiving account's `botOpenId` makes `checkBotMentioned()` return false, and `requireMention` then drops the message. I did not live-test against Feishu.

## Solution Assessment

Unclear. The fix should stay in Feishu mention identity matching, but the open PR's proposed `union_id` source conflicts with the reporter's note that `bot/v3/info` does not provide that field.
