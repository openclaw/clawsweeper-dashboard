## Summary

Keep open: current main still matches Feishu group @mentions against the receiving bot's own `botOpenId`, so the reported app-scoped `open_id` mismatch can still suppress the intended bot before dispatch; the linked open fix PR is the implementation path, not a reason to close before merge.

## What This Changes

The issue requests a Feishu multi-bot mention matcher that can recognize the intended bot when WebSocket mention `open_id` values do not match that bot's self-reported `botOpenId`.

## Best Possible Solution

Keep this as the canonical issue until https://github.com/openclaw/openclaw/pull/40782 or a replacement lands a Feishu-supported identity matcher or cross-app alias mapping across detection, stripping, rich-text, mention-forward, and debounce paths with focused tests and redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention `open_id` differs from `botOpenId` makes `checkBotMentioned` false, then the require-mention group path returns without dispatch. I did not run live Feishu validation in this read-only review.

## Solution Assessment

Unclear: current main is not the solution, and the open PR's union-id direction needs a proven Feishu bot identity contract before it can be considered durable. The maintainable fix should use a supported stable identifier or explicit cross-app mapping and carry it through all mention-related paths.
