## Summary

Keep open: current main and the latest release still match Feishu group mentions against the receiving bot's own `botOpenId`, so the reported app-scoped `open_id` mismatch can still suppress a valid targeted bot before dispatch; the open closing pull request is the active implementation path and needs maintainer review rather than a separate cleanup close.

## What This Changes

The issue reports that Feishu group @mentions can be dropped in multi-bot groups because mention `open_id` values are app-scoped and do not match the receiving bot's own `botOpenId`.

## Best Possible Solution

Keep this as the canonical issue until https://github.com/openclaw/openclaw/pull/40782 or a replacement lands a Feishu-supported identity matcher or cross-app alias strategy across admission, content stripping, rich-text post handling, mention-forward/debounce paths, with focused tests and redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention `open_id` differs from `botOpenId` makes `checkBotMentioned` return false, and the require-mention group path returns before dispatch. I did not run live Feishu validation in this read-only review.

## Solution Assessment

Unclear: the affected code path is narrow, but the durable identity source is not settled because current main only supplies `botOpenId`/`botName` and the linked investigation says the obvious `union_id` source was not available. The maintainable fix should first prove a Feishu-supported stable identifier or cross-app alias strategy, then propagate it consistently through all mention paths.
