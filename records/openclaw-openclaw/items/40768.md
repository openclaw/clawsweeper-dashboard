## What This Changes

Fix Feishu group @mention detection for multiple bot apps when app-scoped mention identifiers do not match the receiving bot’s probed identity.

## Best Possible Solution

Define mention targeting around a Feishu-documented identifier available for both the receiving bot account and event mentions, preserve exact `open_id` matching as the primary path, and verify that one targeted bot responds while every sibling bot ignores the same message in a real four-app group.

## Reproduction Assessment

Yes at source level: in a shared group with multiple Feishu apps, a mention identifier differing from the receiving bot’s probed `open_id` makes the current matcher report that the bot was not mentioned. Executing the full reproduction requires real multi-app Feishu credentials.

## Solution Assessment

No, the current `union_id` proposal is not yet the best proven solution because its account-side identity source is disputed; maintainers should first establish a common documented identifier and then require positive targeting and sibling-isolation proof.
