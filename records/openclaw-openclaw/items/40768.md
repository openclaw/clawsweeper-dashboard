## Summary

Keep open: current main still treats Feishu group mentions as a strict `open_id === botOpenId` match, so the reported app-scoped mismatch can still drop a real @mention; the linked implementation PR remains open and paused for human review.

## What This Changes

Feishu group @mentions should wake the addressed bot even when multiple Feishu bot apps in the group expose different app-scoped `open_id` values.

## Best Possible Solution

Keep this issue open until the linked PR is repaired or replaced with a Feishu-validated identity matcher that handles app-scoped mention IDs while preserving strict mention gating and command stripping.

## Reproduction Assessment

Yes, at source level: a current Feishu group event whose mention metadata contains an `open_id` different from the receiving account's `botOpenId` will set `mentionedBot=false`, and the existing `requireMention` group path then drops it. I did not perform live Feishu validation.

## Solution Assessment

Unclear: the fix should live in Feishu mention identity matching, but the open PR's proposed union-id source and several mention-sensitive paths still need validation or repair before it is the maintainable solution.
