## Summary

Keep open: current main and the latest release still compare Feishu group @mentions only against the receiving bot's stored botOpenId, so the reported app-scoped open_id mismatch remains source-reproducible. The open closing PR at https://github.com/openclaw/openclaw/pull/40782 is the implementation path, but it is unmerged and still needs Feishu identity-contract review plus real behavior proof.

## What This Changes

Report Feishu group @mentions being dropped when multiple bot apps share a group because app-scoped mention open_id values do not match the receiving bot's stored botOpenId.

## Best Possible Solution

Keep this as the canonical issue until the linked PR or a replacement lands a Feishu-validated identity matcher consistently across admission, mention stripping, rich-text posts, mention-forward/debounce handling, tests, and redacted live proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention.id.open_id differs from the stored botOpenId makes checkBotMentioned return false, and the requireMention group path returns before dispatch. I did not run live Feishu/Lark validation in this read-only review.

## Solution Assessment

Unclear: the affected path is narrow, but the durable identity source is not settled because current startup identity only carries botOpenId/botName and the reporter says bot/v3/info did not return union_id. The safest fix needs a Feishu-supported stable identifier or cross-app alias strategy proved in a real multi-bot setup.
