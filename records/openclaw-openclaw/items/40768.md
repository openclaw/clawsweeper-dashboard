## Summary

Keep open: current main and latest release still use Feishu mention `open_id` equality for the mention gate, so the app-scoped mismatch remains source-reproducible. The linked fix PR is open, conflicting, human-review paused, and not proof-positive, so this issue is still needed as the canonical tracker.

## What This Changes

The issue reports that Feishu group @mention detection can silently ignore the intended bot when multiple Feishu bot apps share one group because mention `open_id` values are app-scoped and do not match the receiving bot's stored `botOpenId`.

## Best Possible Solution

Keep this issue open as the canonical Feishu app-scoped mention bug until https://github.com/openclaw/openclaw/pull/40782 is rewritten, proven, and merged, or replaced by a maintainer-approved Feishu identity-resolution design.

## Reproduction Assessment

Yes at source level, but not live: if Feishu supplies a mention `open_id` that differs from the receiving bot's stored `botOpenId`, current `checkBotMentioned` returns false and mention-gated group handling can drop the message. A real Feishu multi-bot group reproduction is still missing from this review.

## Solution Assessment

No for immediate automation: current main does not fix the mismatch, and the open linked `union_id` fallback needs maintainer review because the reporter and prior PR review say the startup identity endpoint may lack `union_id`. The best fix is a proven Feishu-owned identity-resolution path with focused tests and live evidence.
