## Summary

Keep open. Current main still recognizes Feishu bot mentions by exact `open_id` comparison against the startup-resolved `botOpenId`, while this issue reports Feishu delivering mention IDs scoped to another app. The issue is also paired with open, unmerged PR #40782 using closing syntax, so it should stay open for PR review, refresh, replacement, or closure. https://github.com/openclaw/openclaw/issues/40768

## What This Changes

The issue requests Feishu group @mention detection to work when multiple Feishu bot apps share one group and the delivered mention `open_id` does not match the receiving bot's own app-scoped `botOpenId`.

## Best Possible Solution

Implement a Feishu-owned identity matching helper backed by a verified stable Feishu/Lark identifier or explicit cross-app alias mapping, store the needed per-account state in the monitor, reuse it across all mention-related paths, and add regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a group Feishu event whose mention `open_id` differs from the receiving account's `botOpenId` while `requireMention` is true; current parsing sets `mentionedBot=false`, and group handling returns before dispatch.

## Solution Assessment

Unclear for the linked union_id proposal. The requested behavior is valid, but the safest solution is a verified stable identity or alias model rather than assuming `bot/v3/info` can provide `union_id`.
