## Summary

Keep this PR open: the normalization is in the right layer and the supplied TUI proof is useful, but the patch still bypasses the existing redaction path for mapped result content and leaves adjacent dynamic-tool progress coverage inconsistent with the new result shape.

## What This Changes

The PR maps Codex app-server dynamic tool result `contentItems` into TUI-compatible `result.content[]`, strips protocol-only `contentItems` and `success` from emitted transcript payloads, and adds regression coverage.

## Best Possible Solution

Keep the result-shape normalization, but build the mapped transcript content from sanitized response data and update the adjacent dynamic-tool progress tests to assert the redacted `content[]` contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main emits Codex dynamic tool progress as `contentItems`/`success`, while TUI verbose-full output reads `result.content[]`; the PR screenshots also show before/after TUI behavior.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: normalizing at the Codex app-server event boundary is the right owner layer, but the implementation must preserve sanitizer output and align the existing dynamic-tool progress coverage before merge.

Full review comments:

- **[P1] Redact mapped tool-result content before broadcasting:** `extensions/codex/src/app-server/run-attempt.ts:315`
  - body: `toTranscriptToolResult()` sanitizes the original response, but then builds `content` from raw `response.contentItems`. That means any secret-like string returned by a dynamic tool bypasses `sanitizeCodexToolResponse()` and can be emitted through `emitCodexAppServerEvent()` to TUI or Control UI consumers. Build the mapped content from sanitized items or sanitize the mapped `content[]`, and keep the redaction regression assertion.
  - confidence: 0.93
- **[P2] Align the dynamic-tool progress result contract test:** `extensions/codex/src/app-server/run-attempt.ts:1699`
  - body: The PR intentionally removes protocol-only `success` from emitted transcript results, but the existing dynamic-tool progress test still expects `result.success === false`. Update that adjacent test to assert the new `content[]` shape and absence of protocol-only fields, otherwise the focused dynamic-tool lane remains failing or ambiguous.
  - confidence: 0.86
