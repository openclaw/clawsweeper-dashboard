## Summary

Keep open. Current main and the latest release still have adjacent safeguards, but they do not define the requested atomic intent-to-execution contract with mandatory commit or structured abort; this remains a core product/protocol design item rather than conservative cleanup.

## What This Changes

The issue requests a transaction-style run lifecycle that turns pre-action intent into guaranteed execution evidence plus a mandatory commit or structured abort outcome.

## Best Possible Solution

Define one additive public run-intent contract that reuses canonical terminal outcomes, carries structured abort blockers, and propagates consistently through gateway events, wait responses, session projection, plugin-visible APIs, docs, and regression tests.

## Reproduction Assessment

Yes for the contract gap: current source and v2026.6.6 show generic lifecycle events and terminal snapshots without mandatory intent-to-execution binding. The intermittent orphan-notice behavior itself was not live-reproduced because the report has no concrete trace or deterministic interruption point.

## Solution Assessment

No, the standalone transaction vocabulary is not yet the best fully specified implementation. The safer direction is an additive extension of the existing terminal-outcome architecture, not a parallel lifecycle state machine.
