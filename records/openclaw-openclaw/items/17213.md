## Summary

Keep open. Current main still has adjacent lifecycle, liveness, replay-invalid, abort snapshot, and SDK normalization safeguards, but it does not define the requested atomic intent-to-execution contract with mandatory commit-or-abort terminal states and structured abort blockers.

## What This Changes

The issue requests a transaction-style run lifecycle that guarantees announced action intent is followed by execution and an explicit commit or abort result.

## Best Possible Solution

Define an additive protocol-level run lifecycle contract that distinguishes intent, execution evidence, committed completion, and aborted/blocker completion, then wire it through event types, Gateway schemas, runtime emitters, wait behavior, SDK/client normalization, session state, docs, and regression tests without breaking existing lifecycle consumers.

## Reproduction Assessment

No: the issue does not include concrete interruption steps, logs, or a minimal runtime trace. Current-main source and docs do give high-confidence verification that the requested public lifecycle contract is absent.

## Solution Assessment

Unclear: the transaction-style lifecycle is a plausible maintainable direction, but the exact states, blocker payload, and backwards-compatible rollout need maintainer design before implementation. Another local heuristic would not solve the cross-surface contract gap.
