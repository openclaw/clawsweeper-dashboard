## Summary

Keep open. Current main and v2026.6.8 have Web Push subscription/test plumbing and inbound `message:received` hooks, but there is still no opt-in bridge that automatically sends browser push notifications for inbound channel messages.

## What This Changes

The issue requests an opt-in Control UI browser push notification trigger for inbound non-webchat channel messages.

## Best Possible Solution

Define one deliberate opt-in notification policy that connects inbound `message_received` events to Web Push with privacy-safe payloads, channel filters, config schema/docs, and focused tests, or first extend the plugin API if maintainers want this outside core.

## Reproduction Assessment

Not applicable: this is a feature request, not broken behavior under an existing contract. Source and release inspection give high-confidence absence proof for the automatic inbound-message-to-Web-Push trigger.

## Solution Assessment

Unclear until maintainer product/privacy review. The likely best implementation should reuse the existing `message_received` and `broadcastWebPush` foundations, but the exact config, payload redaction, and core-versus-plugin boundary need a decision first.
