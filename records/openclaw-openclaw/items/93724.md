- `impact:security`: The requested automatic Web Push path could expose inbound channel message content through browser/system notifications and needs privacy-safe opt-in design.
- `impact:other`: The user-visible impact is missed background-tab awareness for incoming channel messages, which is a concrete notification/ergonomics gap outside the narrower impact labels.

## Summary

Keep open. Current main and the latest release have Web Push subscription/test plumbing and inbound message hooks, but no automatic inbound-message Web Push trigger or opt-in config. This needs maintainer product/security/API scoping rather than cleanup closure because public plugin hooks can observe messages, but the supported docs do not expose a Web Push send API for a ClawHub-only implementation.

## What This Changes

Request built-in opt-in browser Web Push notifications for incoming non-webchat channel messages.

## Best Possible Solution

Define one intentional opt-in notification contract that reuses `message_received` and Web Push, with privacy-safe payload defaults, config/docs/tests, and a maintainer decision on whether the permanent owner is core or a new plugin-safe notification API.

## Reproduction Assessment

Not applicable: this is a feature request, not broken behavior with an existing contract. Source inspection gives high-confidence absence proof for an automatic `message_received` to Web Push trigger on current main and `v2026.6.9`.

## Solution Assessment

Unclear until maintainer product/API review. Directly calling `broadcastWebPush` from the message hook is plausible, but the best fix must settle privacy defaults, config shape, channel filtering, and whether a public plugin-safe notification API should own this instead.
