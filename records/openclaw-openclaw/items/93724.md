## Summary

Keep open. Current main and the latest release have Web Push subscription/test plumbing and inbound `message_received` hooks, but no built-in opt-in trigger that sends browser push notifications for incoming channel messages; the remaining work needs product, privacy, and API scoping rather than cleanup closure.

## What This Changes

Request to add configurable browser push notifications for inbound non-webchat channel messages in the Control UI.

## Best Possible Solution

Define one privacy-safe notification contract: either a core-owned opt-in inbound-message Web Push policy with clear payload/redaction/channel rules, or a public plugin/Gateway notification API that lets this live outside core while reusing the existing Web Push transport.

## Reproduction Assessment

Not applicable as a bug repro; this is a feature request rather than broken documented behavior. Source inspection gives high-confidence absence proof: current main has inbound hooks and Web Push subscription/test plumbing, but no path connects `message_received` to `broadcastWebPush`.

## Solution Assessment

Unclear until maintainer product review. The requested built-in config is plausible, but the best design may instead be a public plugin/Gateway notification API coordinated with the broader Control UI notification trackers.
