## Summary

Keep open. Current main and the latest release have Web Push subscription/test infrastructure and a `message:received` hook path, but they do not implement automatic browser notifications for inbound channel messages; the requested opt-in config/policy also needs maintainer product and plugin/API-boundary direction.

## What This Changes

Issue requests built-in browser Web Push notifications for inbound non-webchat channel messages with an opt-in config surface.

## Best Possible Solution

Maintain an open product/API decision: either add a narrow public Web Push notification seam that a hook/plugin can use, or implement a core opt-in inbound-message notification policy with explicit privacy defaults and tests.

## Reproduction Assessment

Yes, from source inspection for the feature gap: current main has Web Push send/test infrastructure and inbound message hooks, but no automatic Web Push trigger on inbound channel messages. This is not a failing existing contract reproduction.

## Solution Assessment

No, not yet. The requested built-in config is plausible, but the best fix depends on maintainer choice between a core opt-in policy and a supported plugin/API seam.
