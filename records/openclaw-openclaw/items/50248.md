## Summary

Keep open. Current main and v2026.5.20 still let `sessions cleanup --fix-missing` prune from one resolved persisted `sessionFile` path, so the stale-path plus existing canonical transcript case remains source-reproducible; https://github.com/openclaw/openclaw/pull/69954 is still open and conflicting with closing syntax for this issue.

## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` can delete fresh cron/session-store entries when `sessionFile` is stale even though the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/69954 with a current-main cleanup-service fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale metadata when safe, and preserves true-missing pruning behavior.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a missing stale generated `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single resolved-path cleanup check and would be pruned.

## Solution Assessment

Yes, the narrow fallback-and-repair direction is the maintainable fix for this cleanup bug. The issue should stay open until that behavior lands on current main through the linked PR or an equivalent replacement.
