## Summary

Keep open. Current main and the latest release still let `sessions cleanup --fix-missing` prune from one resolved persisted `sessionFile` path, while the linked pull request https://github.com/openclaw/openclaw/pull/69954 remains open and unmerged as the narrow implementation candidate.

## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` can prune fresh cron or heartbeat session entries when a stale persisted `sessionFile` is missing but a valid canonical transcript still exists.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/69954 with a focused current-main cleanup-service fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale generated `sessionFile` metadata when safe, preserves true-missing pruning, and adds regression coverage.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=A`, stale missing `sessionFile=B.jsonl`, and a live `A.jsonl` canonical transcript will still be pruned because current cleanup checks only the resolved persisted path.

## Solution Assessment

Yes, the narrow maintainable fix is a cleanup-service fallback and metadata repair, not a broad session-history redesign. The existing linked pull request captures the right direction but still needs current-main review or replacement before the issue can close.
