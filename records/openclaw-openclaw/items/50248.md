## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` can prune fresh cron or heartbeat sessions when stale `sessionFile` metadata points away from an existing canonical `<sessionId>.jsonl` transcript.

## Best Possible Solution

Land or replace the linked PR with a current-main `cleanup-service.ts` fix that checks the canonical `<sessionId>.jsonl` candidate, repairs stale `sessionFile` metadata, counts repair-only mutations, and preserves true-missing pruning.

## Reproduction Assessment

Yes, at source level. A row with a valid `sessionId`, a stale but structurally valid missing `sessionFile`, and an existing canonical `<sessionId>.jsonl` still follows the single resolved-path cleanup check and would be pruned; I did not run a live destructive cleanup repro.

## Solution Assessment

No, current main is not sufficient yet. The fallback-and-repair direction is the right narrow fix, but it needs to land in the current cleanup service through the linked PR or a replacement.
