## Summary

Keep open. Current main and the latest release still let `sessions cleanup --fix-missing` delete from one resolved persisted `sessionFile` path, so the stale-path plus existing canonical transcript case remains source-reproducible; https://github.com/openclaw/openclaw/pull/69954 is still open and explicitly tied to this issue.

## What This Changes

Prevent `openclaw sessions cleanup --fix-missing` from pruning valid fresh sessions when stored `sessionFile` metadata is stale but the canonical transcript still exists.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/69954 or replace it with the same focused cleanup-service fix: check the canonical `<sessionId>.jsonl` candidate before pruning, repair stale metadata when safe, and preserve true-missing pruning behavior.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a missing stale generated `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single resolved-path cleanup check and would be pruned.

## Solution Assessment

Yes. The narrow fallback-and-repair direction is the maintainable fix for this cleanup bug, and the issue should stay open until that behavior lands through the linked PR or an equivalent replacement.
