## Summary

Keep open. Current main still has the unsafe `sessions cleanup --fix-missing` stale-path behavior, and PR #69954 is open with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune valid fresh cron/run sessions when stored `sessionFile` metadata points at a stale missing transcript while the canonical `<sessionId>.jsonl` still exists.

## Best Possible Solution

Land PR #69954 if its behavior is acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical file exists, preserves true-missing pruning, and leaves broader doctor/orphan archival semantics to separate follow-up.

## Reproduction Assessment

Yes. A deterministic fixture can create a store entry with `sessionId=<id>`, a stale missing `sessionFile`, and an existing canonical `<id>.jsonl`; current main's cleanup source shows `--fix-missing` still prunes from the stale resolved path only.

## Solution Assessment

Yes. A narrow cleanup fallback and metadata repair is the best fix for this issue because producer-side rollover changes do not repair existing stale metadata, and broader orphan/archive behavior is a separate product surface.
