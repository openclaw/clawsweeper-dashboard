## Summary

Keep open: current main has partial producer-side fixes for stale `sessionFile` carryover, but `sessions cleanup --fix-missing` still resolves one persisted transcript path and prunes when that path is missing without checking the canonical `<sessionId>.jsonl`. The existing open closing PR at https://github.com/openclaw/openclaw/pull/69954 remains the implementation path, but it is conflicting and previously reviewed as needing a port into the cleanup service.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune fresh cron sessions when stored `sessionFile` metadata points at a stale path even though the canonical transcript file exists.

## Best Possible Solution

Land a current-main cleanup-service fix that keeps true-missing pruning, checks the canonical transcript path before deleting stale-metadata rows, repairs `sessionFile` when safe, and adds service-level regression coverage.

## Reproduction Assessment

Yes, at source level: a row with a valid `sessionId`, stale generated `sessionFile`, and existing canonical `<sessionId>.jsonl` still follows the single resolved-path cleanup check and can be pruned. I did not run the mutating cleanup command because this review is read-only.

## Solution Assessment

Yes for the requested cleanup fallback direction, but not by closing the issue now. The narrow maintainable fix belongs in `src/config/sessions/cleanup-service.ts`, not only in producer-side rollover or a stale command-local PR branch.
