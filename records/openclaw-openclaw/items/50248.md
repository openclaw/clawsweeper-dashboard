## Summary

Keep open: current main and v2026.6.6 still run `--fix-missing` through `cleanup-service.ts`, which resolves one persisted transcript path and prunes when that path is absent without checking the canonical `<sessionId>.jsonl`. The linked fix PR at https://github.com/openclaw/openclaw/pull/69954 remains the open implementation path, but it is not merged and is currently conflicting.

## What This Changes

Fix session cleanup so `--fix-missing` does not prune fresh cron or heartbeat sessions when stale `sessionFile` metadata points at a missing path but the canonical transcript exists.

## Best Possible Solution

Port the canonical transcript fallback and stale `sessionFile` repair into `src/config/sessions/cleanup-service.ts`, count repairs in cleanup summaries, add service-level regression coverage, and close this issue after the linked fix merges.

## Reproduction Assessment

Yes, at source level. A store entry with a stale but structurally valid missing `sessionFile` and an existing canonical `<sessionId>.jsonl` still follows the active cleanup-service single-path check and would be pruned.

## Solution Assessment

Yes for the narrow fallback-and-repair direction, but not as current main stands. The maintainable fix belongs in `cleanup-service.ts` with repair accounting and regression coverage rather than only in the stale command-local helper.
