## Summary

Keep open. Current `main` and the latest release still let `sessions cleanup --fix-missing` prune from one resolved `sessionFile` path, while the linked closing PR remains open and conflicting.

## What This Changes

The issue reports that stale `sessionFile` metadata can make `sessions cleanup --fix-missing` prune valid session entries whose canonical `<sessionId>.jsonl` transcript still exists.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/69954 with a current-main cleanup-service fix that checks canonical `<sessionId>.jsonl` before pruning, repairs stale `sessionFile` metadata, counts repair mutations, and preserves true-missing pruning.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a missing stale generated `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single resolved-path check in current main and would be pruned.

## Solution Assessment

Yes, the linked fallback-and-repair direction is the narrow maintainable fix for this cleanup bug. The issue should stay open until that PR or an equivalent current-main replacement lands because closing syntax does not resolve the issue before merge.
