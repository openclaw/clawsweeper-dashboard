## Summary

Keep open: current main and the latest release still prune from one resolved stale `sessionFile` path, while the explicit closing PR is open and unmerged.

## What This Changes

`sessions cleanup --fix-missing` can prune fresh or valid session-store entries when their persisted `sessionFile` path is stale even though the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Land or replace the focused cleanup repair so `--fix-missing` checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical file exists, and leaves broader doctor/orphan archival semantics to their own issue.

## Reproduction Assessment

Yes. Source inspection shows a stale-but-structurally-valid missing `sessionFile` is resolved and checked first, so cleanup can prune even when the canonical `<sessionId>.jsonl` exists.

## Solution Assessment

Yes, the narrow cleanup fallback/repair is the best immediate fix. The producer-side rollover mitigations on main reduce future drift but do not protect existing stale store entries during cleanup.
