## Summary

Keep open. Current main and the latest release still prune from a single resolved `sessionFile` path during `sessions cleanup --fix-missing`, while the open linked PR is unmerged and conflicting, so this issue remains the canonical tracker for the stale-transcript cleanup safety bug.

## What This Changes

The issue reports that session cleanup can prune fresh cron or heartbeat sessions when stored `sessionFile` metadata points at a missing stale transcript even though a valid canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Keep this issue as the canonical session-cleanup safety bug until a current-main PR ports the canonical transcript fallback into `src/config/sessions/cleanup-service.ts`, reports metadata repairs as mutations, and adds integration coverage for stale stored paths with live canonical transcripts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a stale but structurally valid `entry.sessionFile` resolves to the missing stored path, and cleanup prunes when `fs.existsSync(transcriptPath)` is false without checking the canonical session-id transcript.

## Solution Assessment

Yes on direction, not yet on landed implementation. The narrow cleanup-side fallback plus repair accounting is the maintainable fix; producer-side rollover mitigations alone do not protect existing stale rows.
