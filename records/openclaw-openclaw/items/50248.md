## Summary

Keep open: current main still resolves one persisted transcript path during `--fix-missing` cleanup and prunes when that path is absent, so the stale `sessionFile` plus live canonical transcript case is not implemented. The linked fix candidate is open and unmerged, but it now needs review or replacement against the current cleanup-service path.

## What This Changes

Prevent `sessions cleanup --fix-missing` from pruning valid cron or heartbeat sessions when `sessionFile` is stale but the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Port or land a current-main cleanup fallback that checks canonical session transcript candidates before pruning, repairs safe stale generated `sessionFile` metadata, and keeps broader doctor/orphan archive work separate.

## Reproduction Assessment

Yes, at source level. A stale but structurally valid `sessionFile` inside the sessions directory plus an existing canonical `<sessionId>.jsonl` still follows the current cleanup path to `prune-missing`; I did not run a mutating repro command in this read-only review.

## Solution Assessment

Yes for the narrow fallback direction, but it is not implemented on current main. The maintainable fix is to make cleanup use canonical transcript candidate semantics before pruning, while preserving true-missing cleanup behavior.
