## Summary

Keep open. Current main and latest release still resolve one persisted `sessionFile` during `sessions cleanup --fix-missing` and prune when that path is missing; the linked implementation candidate at https://github.com/openclaw/openclaw/pull/69954 is still open and conflicting, so the issue should not close before a current-main fix lands.

## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` can prune valid fresh cron or heartbeat session entries when `sessionFile` metadata is stale but the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/69954 with a current-main cleanup-service fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata, counts repair mutations, preserves true-missing pruning, and leaves broader doctor/orphan archive semantics to their own tracked follow-up.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a missing stale generated `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single resolved-path cleanup check in current main and would be pruned.

## Solution Assessment

Yes, the fallback-and-repair direction is the narrow maintainable fix for this cleanup bug, but it needs to land in the active cleanup service path before this issue can close.
