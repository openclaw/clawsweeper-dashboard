## Summary

Keep open: current main still prunes from one resolved persisted transcript path, so stale `sessionFile` metadata can still remove a valid session entry when the canonical `<sessionId>.jsonl` exists. The linked fix PR is open and unmerged, so this issue should remain the canonical tracking item.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune fresh cron or heartbeat sessions when stored `sessionFile` metadata points at a stale transcript path despite a valid sessionId transcript existing.

## Best Possible Solution

Land or replace the linked cleanup-service fix so `--fix-missing` checks the canonical transcript candidate before pruning, repairs stale generated `sessionFile` metadata when safe, and preserves true-missing/header-only pruning semantics.

## Reproduction Assessment

Yes, source-level. A store entry with a structurally valid but missing `sessionFile` and an existing canonical `<sessionId>.jsonl` still follows the cleanup path that checks only the resolved persisted path.

## Solution Assessment

No current-main fix is complete. Producer-side rollover mitigations help future writes, but the best fix for this issue is a cleanup-service fallback/repair path plus regression coverage.
