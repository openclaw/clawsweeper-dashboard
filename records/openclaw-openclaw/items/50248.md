## Summary

Keep open: current main has partial producer-side fixes for stale `sessionFile` carryover, but `sessions cleanup --fix-missing` still uses one resolved transcript path and can prune a row when that stored path is stale even if the canonical `<sessionId>.jsonl` exists. The linked closing PR is open, conflicting, and previously reviewed as needing a port into the cleanup service, so it is an implementation path rather than a reason to close this issue.

## What This Changes

Fix `sessions cleanup --fix-missing` so stale stored `sessionFile` metadata cannot cause valid fresh cron or heartbeat sessions with existing transcripts to be pruned.

## Best Possible Solution

Land or replace the linked PR with a current-main service-level fix that checks the canonical transcript before pruning, repairs stale `sessionFile` metadata, counts repair-only mutations, and preserves true-missing cleanup behavior.

## Reproduction Assessment

Yes, at source level. A row with a valid `sessionId`, a stale stored `sessionFile` path, and an existing canonical `<sessionId>.jsonl` still follows the single resolved-path cleanup check and would be pruned; I did not run a live destructive cleanup repro in this read-only review.

## Solution Assessment

No, current main is not sufficient yet. The fallback-and-repair direction is the right narrow fix, but it needs to live in `src/config/sessions/cleanup-service.ts`, include repair accounting, and land through the linked PR or a replacement.
