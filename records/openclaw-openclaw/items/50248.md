## Label Justifications

- `P2`: This is a bounded session cleanup safety bug with real history-loss risk but no current evidence of a core-runtime outage.
- `impact:data-loss`: Enforced cleanup can remove session-store references for transcript data that still exists on disk.
- `impact:session-state`: The bug is stale drift between session-store metadata and the actual transcript backing a session.

## Summary

Keep open. Current main and v2026.6.8 still let `sessions cleanup --fix-missing` trust one persisted `sessionFile` path and prune when that file is missing, while normal transcript readers already know how to try the canonical `<sessionId>.jsonl` candidate; the linked implementation PR remains open and unmerged.

## What This Changes

Fix `sessions cleanup --fix-missing` so stale stored `sessionFile` metadata cannot cause valid fresh cron or heartbeat sessions to be pruned when the canonical transcript exists.

## Best Possible Solution

Land or replace the linked cleanup fallback PR with a service-level fix that checks canonical transcript candidates, repairs safe stale generated paths, preserves true-missing pruning, and keeps broader doctor/orphan semantics on separate issues.

## Reproduction Assessment

Yes, at source level. In current main, a stale but structurally valid `entry.sessionFile` is resolved first and cleanup deletes the row when that one path is absent; a canonical `<sessionId>.jsonl` existing beside it is never checked.

## Solution Assessment

Yes for the narrow fallback-and-repair direction, but not until it lands in the active cleanup service. A command-only or automation-only change would be too narrow because current main routes cleanup through `src/config/sessions/cleanup-service.ts`.
