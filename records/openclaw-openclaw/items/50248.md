## Summary

Keep open. Current main still lets `sessions cleanup --fix-missing` prune from a single resolved `sessionFile` path, so already-stale store metadata can still remove a valid session entry even when the canonical `<sessionId>.jsonl` transcript exists; the open linked fix candidate is not merged.

## What This Changes

The issue asks for session cleanup to stop pruning fresh or valid sessions when stored `sessionFile` metadata is stale but the canonical transcript exists.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69954 if refreshed and acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale generated `sessionFile` metadata when that canonical file exists, and preserves true-missing cleanup behavior.

## Reproduction Assessment

Yes, at source level. A stale but structurally valid persisted `sessionFile` plus an existing canonical `<sessionId>.jsonl` still follows the current cleanup path to `prune-missing`; I did not run a mutating repro command in this read-only review.

## Solution Assessment

Yes, the narrow cleanup fallback/repair is the best remaining fix for already-stale stores. The producer-side rollover fixes on main reduce new bad metadata, but they do not make existing stale cleanup entries safe.
