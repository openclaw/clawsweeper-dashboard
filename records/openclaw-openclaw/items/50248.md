## Summary

Keep open. Current main and the latest release still let `sessions cleanup --fix-missing` prune from one resolved persisted `sessionFile` path, so stale metadata can still remove a valid session entry even when the canonical transcript exists; the direct fix PR is now closed unmerged.

## What This Changes

The issue asks OpenClaw to stop `sessions cleanup --fix-missing` from pruning fresh or valid sessions when `sessionFile` metadata is stale but a valid transcript exists under the current `sessionId`.

## Best Possible Solution

Land a focused cleanup-service fix that treats stale generated `sessionFile` metadata like the reader path does: check the canonical `<sessionId>.jsonl` before pruning, repair generated stale metadata when the canonical transcript exists, and keep true-missing pruning intact.

## Reproduction Assessment

Yes. The source path is high-confidence: create a store row with a current `sessionId`, a stale missing generated `sessionFile`, and a valid `<sessionId>.jsonl`; current `--fix-missing` resolves only the stale path and will mark the row `prune-missing`.

## Solution Assessment

Yes for the narrow fix direction, but it is not implemented yet. The best repair is in cleanup-service/test coverage, not broad automatic pruning or another producer-only rollover mitigation.
