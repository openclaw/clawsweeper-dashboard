## Summary

Keep open: current main and v2026.5.22 still let cleanup prune from one resolved persisted `sessionFile` path, and https://github.com/openclaw/openclaw/pull/69954 is still open with closing syntax for this issue.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can delete fresh session entries when `sessionFile` points to a stale missing path while the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/69954 with a cleanup-service fix that checks the canonical `<sessionId>.jsonl` candidate, repairs stale metadata when safe, and preserves true-missing pruning behavior.

## Reproduction Assessment

Yes, at source level. A row with `sessionId=<id>`, a stale missing persisted `sessionFile`, and an existing canonical `<id>.jsonl` still resolves only the stale path in cleanup and is deleted when `fs.existsSync` fails.

## Solution Assessment

Yes, the narrow fallback-and-repair direction is the maintainable fix for this cleanup bug. Producer-side mitigations help new writes but do not protect already-stale rows from `--fix-missing`.
