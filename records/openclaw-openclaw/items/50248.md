## Summary

Keep open. Current main still uses a single resolved transcript path for `sessions cleanup --fix-missing`, so stale persisted `sessionFile` metadata can still make cleanup prune a valid entry; the open linked PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for `sessions cleanup --fix-missing` to preserve fresh sessions when the stored `sessionFile` is stale but the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/69954, or replace it with an equivalent cleanup-service fix that checks the canonical transcript before pruning, repairs stale generated `sessionFile` metadata, and preserves true-missing pruning.

## Reproduction Assessment

Yes, at source level. A fixture with `sessionId=<id>`, a missing stale `sessionFile`, and an existing canonical `<id>.jsonl` would still follow the single stale-path check in current main and be pruned.

## Solution Assessment

Yes. A narrow cleanup fallback plus metadata repair is the maintainable fix for this issue because producer-side rollover hardening does not repair existing stale session-store entries.
