## Summary

Keep open. Current main still lets `sessions cleanup --fix-missing` prune from a single resolved persisted transcript path, so stale `sessionFile` metadata can still remove an entry even when the canonical `<sessionId>.jsonl` exists; PR #69954 is open with closing syntax and should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for session cleanup to avoid pruning fresh or valid cron/heartbeat sessions when persisted `sessionFile` metadata is stale but the real transcript still exists.

## Best Possible Solution

Land PR #69954 if its behavior is acceptable, or replace it with an equivalent focused cleanup fix that checks and repairs the canonical `<sessionId>.jsonl` path before pruning while preserving true-missing cleanup behavior.

## Reproduction Assessment

Yes. A deterministic fixture can create a store entry with `sessionId=<id>`, a stale missing `sessionFile`, and an existing canonical `<id>.jsonl`; current main's cleanup source still follows only the stale resolved path for `--fix-missing`.

## Solution Assessment

Yes. A narrow cleanup fallback and metadata repair is the best fit for this issue because producer-side rollover hardening does not protect existing stale session-store entries.
