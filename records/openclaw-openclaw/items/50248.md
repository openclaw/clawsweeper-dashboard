## Summary

Keep open. Current main and the latest release still let cleanup trust one resolved persisted transcript path, and the open linked PR is unmerged and conflicting, so the user-visible cleanup safety problem is not resolved yet.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune valid fresh cron or heartbeat sessions when stale `sessionFile` metadata points to a missing transcript while the canonical `sessionId` transcript exists.

## Best Possible Solution

Land or replace the linked PR with a current-main cleanup-service fix that checks the canonical transcript candidate before pruning, repairs stale generated `sessionFile` metadata, preserves true-missing pruning, and reports repair-only mutations accurately.

## Reproduction Assessment

Yes, at source level. A store entry with a stale but structurally valid `sessionFile` and an existing canonical `<sessionId>.jsonl` still reaches cleanup’s single-path prune decision on current main.

## Solution Assessment

Yes for the issue’s fallback direction, but not as closed work yet. The narrow maintainable solution is to port and land the canonical transcript fallback/repair in the current cleanup service, with repair accounting and regression coverage.
