## Summary

Keep open. Current main still checks one resolved transcript path during `sessions cleanup --fix-missing`, so stale persisted `sessionFile` metadata can still prune an entry whose canonical `<sessionId>.jsonl` exists; the open linked PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks cleanup to avoid pruning fresh cron or heartbeat sessions when stale `sessionFile` metadata points at a missing file but a valid canonical session transcript exists.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/69954, or replace it with an equivalent focused cleanup-service fix that checks the canonical transcript before pruning, repairs stale generated `sessionFile` metadata, and preserves true-missing pruning behavior.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a missing stale `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single stale-path check in current main and would be pruned.

## Solution Assessment

Yes. A narrow cleanup fallback plus metadata repair is the maintainable fix for this issue because producer-side rollover hardening does not repair existing stale session-store entries.
