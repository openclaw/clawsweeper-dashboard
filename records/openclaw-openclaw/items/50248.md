## Summary

Keep open. Current main still prunes missing transcripts from one resolved `sessionFile` path, while the linked closing implementation remains open and conflicting, so this issue should wait for https://github.com/openclaw/openclaw/pull/69954 or an equivalent fix to land.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune valid fresh cron or heartbeat sessions when stale `sessionFile` metadata points away from an existing canonical `<sessionId>.jsonl` transcript.

## Best Possible Solution

Port or land the fallback-and-repair behavior in the cleanup service: check the canonical `<sessionId>.jsonl` candidate before pruning, repair stale `sessionFile` metadata when that file exists, count repairs as mutations, and preserve true-missing pruning.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a missing stale `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single resolved-path prune branch on current main.

## Solution Assessment

Yes in direction, but not landed. The narrow maintainable fix is the linked PR’s canonical fallback and repair behavior, ported to the current cleanup-service implementation with explicit repair accounting.
