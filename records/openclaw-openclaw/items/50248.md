## Summary

Keep open. Current main still lets `--fix-missing` prune from one resolved persisted transcript path, so already-stale `sessionFile` rows can still be removed even when the canonical transcript exists; https://github.com/openclaw/openclaw/pull/69954 is the linked implementation candidate but remains open and conflicting.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune fresh cron or heartbeat session entries whose persisted `sessionFile` is stale even though a valid canonical transcript exists under the current `sessionId`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/69954 with a focused current-main cleanup-service fix that checks the canonical transcript before pruning, repairs stale generated `sessionFile` metadata, and preserves true-missing pruning.

## Reproduction Assessment

Yes, source-reproducible. Current main still prunes after checking only the resolved persisted transcript path, and the issue provides concrete stale `sessionFile` plus live canonical transcript examples.

## Solution Assessment

Yes for the narrow direction, but not solved yet. The maintainable fix is the cleanup-service canonical fallback plus metadata repair; producer-side rollover mitigation and automatic pruning do not fully address already-stale rows.
