## Summary

Keep open. Current main and the latest release still resolve one persisted `sessionFile` path during `sessions cleanup --fix-missing`, so stale metadata can still make cleanup prune a valid session entry; the linked open PR is the active fix candidate but is not merged.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune fresh cron or heartbeat session entries when stale `sessionFile` metadata points away from an existing canonical transcript.

## Best Possible Solution

Port or land a focused cleanup-service fix that checks and repairs the canonical `sessionId` transcript path before pruning stale `sessionFile` rows, preserves true-missing pruning, and leaves broader doctor/orphan archive semantics to their own tracked issue.

## Reproduction Assessment

Yes, source-reproducible: a store row with `sessionId`, stale in-dir `sessionFile`, and an existing canonical `<sessionId>.jsonl` still flows through one resolved path and is deleted when that stale path is missing. I did not run a filesystem repro because this review is read-only, but the current-main branch path is direct.

## Solution Assessment

No, current main is not the best end state because cleanup still lacks the canonical fallback even though producer-side rollover now avoids some new stale metadata. The best fix is a narrow cleanup-service repair with regression coverage, likely by porting or replacing the linked open PR.
