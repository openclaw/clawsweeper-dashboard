## Summary

Keep open. Current main still prunes `--fix-missing` entries from a single persisted transcript path, so a stale `sessionFile` can still delete a valid session entry when the canonical `<sessionId>.jsonl` exists; PR #69954 is the open implementation candidate and should be reviewed or replaced.

## What This Changes

The issue asks session cleanup to avoid pruning fresh or valid sessions when persisted `sessionFile` metadata is stale but a canonical transcript for the current `sessionId` exists.

## Best Possible Solution

Review and land PR #69954, or replace it with an equivalent focused cleanup-service fix that checks the canonical transcript candidate before deletion, repairs stale metadata, and preserves true-missing pruning.

## Reproduction Assessment

Yes. A deterministic fixture can create a store entry with `sessionId=<id>`, a stale missing `sessionFile`, and an existing canonical `<id>.jsonl`; current main's cleanup source still follows only the stale resolved path before pruning.

## Solution Assessment

Yes. A narrow cleanup fallback and metadata repair is the maintainable fix for this issue because producer-side rollover hardening does not repair existing stale session-store entries.
