## Summary

Keep open: current main and v2026.6.10 still let cleanup prune after checking only one resolved persisted transcript path, so the reported stale sessionFile/canonical transcript case remains possible. ## What This Changes

The issue reports that sessions cleanup can prune fresh cron or run sessions when stored sessionFile metadata points at a stale transcript path even though a valid transcript exists for the current sessionId.

## Best Possible Solution

Implement a focused cleanup-service fix that checks canonical transcript candidates before pruning, repairs stale generated sessionFile metadata when the canonical file exists, keeps true-missing pruning intact, and covers the stale-row case in regression tests.

## Reproduction Assessment

Yes, at source level: current main still resolves one persisted sessionFile path and prunes when that path is missing, while the issue gives concrete rows where the canonical sessionId transcript exists.

## Solution Assessment

Yes for the narrow fix direction. Cleanup should share transcript-candidate safety or explicitly check and repair the canonical sessionId transcript before pruning; write-side rollover fixes alone do not cover already-stale rows.
