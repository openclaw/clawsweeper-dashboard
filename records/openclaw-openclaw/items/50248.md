## Label Justifications

- `P2`: This is a normal-priority session cleanup safety bug with limited blast radius but real potential to hide valid session history.
- `impact:session-state`: The issue is about stale `sessionFile` metadata causing cleanup to remove valid session-store entries.
- `impact:data-loss`: The reported failure can make valid transcript-backed session history disappear from `sessions.json` when cleanup is enforced.

## Summary

The central cleanup bug still reproduces from source: cleanup resolves only the persisted `sessionFile` path and prunes when that path is missing, even though reader-side transcript candidate logic can prefer the canonical `sessionId` transcript. The linked fix PR is open and unmerged, so the issue should stay open rather than be closed as implemented.

## What This Changes

`sessions cleanup --fix-missing` can remove fresh or valid session-store entries when `sessionFile` points at a stale missing transcript while the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Land or replace the linked cleanup fix so `--fix-missing` checks and repairs the canonical `<sessionId>.jsonl` path before deleting entries, while keeping true-missing pruning and broader doctor/orphan archive behavior separate.

## Reproduction Assessment

Yes. A current-main source repro is to create a store row with `sessionId = X`, a stale in-dir `sessionFile` that is missing, and an existing `X.jsonl`; `pruneMissingTranscriptEntries` checks only the resolved stale path before pruning.

## Solution Assessment

Yes for the direction, not yet for the open branch. The best fix is the narrow cleanup-service fallback and repair against current main, plus accounting/tests for repair-only mutations; the existing open PR implements the idea but needs a port and proof before merge.
