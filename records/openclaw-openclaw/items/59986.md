Full review comments:

- **[P1] Claim callbacks before invoking handlers:** `src/plugins/interactive.ts:321-325`
  - body: This path only peeks at the dedupe cache before calling `onMatched` and the plugin handler, then records the key after the handler resolves. Two duplicate callbacks arriving together can both pass `peek` and run side effects; preserve current main's in-flight claim/commit/release pattern before invoking handlers.
  - confidence: 0.9
- **[P1] Wire Teams card submits into plugin dispatch:** `extensions/msteams/src/outbound.ts:45-49`
  - body: The new Teams card sends `Action.Submit` data with `oc: "interactive"`, but the Teams invoke wrapper only consumes `message/submitAction` when `actionName === "feedback"`; these buttons can render without ever reaching the registered plugin interaction handler.
  - confidence: 0.86
- **[P1] Encode Feishu card actions with the structured envelope:** `extensions/feishu/src/outbound.ts:101-107`
  - body: Feishu buttons are emitted with `oc: "interactive"`, while the card-action decoder treats structured actions only when `oc` is the `ocf1` envelope. These clicks fall back to a JSON text synthetic command instead of the generic interaction handler.
  - confidence: 0.88
- **[P1] Use the existing presentation pipeline for rich replies:** `extensions/msteams/src/outbound.ts:121-130`
  - body: Current main already has the portable `MessagePresentation` and channel `renderPresentation` path. Adding a second `InteractiveReply` projection in channel outbound creates duplicate rich-message semantics and will drift from the current adapter limits, fallback, and action model after rebase.
