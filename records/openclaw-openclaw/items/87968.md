## What This Changes

The PR changes the Control UI update controller to treat `ok: true` update responses as pending, fall back to `updateAvailable.latestVersion` when handoff responses omit `after.version`, and adds controller tests for that payload shape.

## Best Possible Solution

Merge the controller fix after redacted live managed-service Dashboard proof, or after a maintainer with that environment records a proof override.

## Reproduction Assessment

Yes from source inspection: current main checks `status === "ok"`, while the documented gateway handoff response is `ok: true` with `result.status: "skipped"`. I did not run a live managed-service Dashboard repro in this read-only review.

## Solution Assessment

Yes for the code shape: interpreting `ok: true` at the controller boundary matches the gateway contract, and using `updateAvailable.latestVersion` preserves post-restart verification when `after.version` is absent. The proposed solution is not merge-ready until real behavior proof or a maintainer override covers the managed-service UI flow.
