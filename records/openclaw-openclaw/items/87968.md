## What This Changes

The PR updates the Control UI update controller and tests so `update.run` `ok: true` managed-service handoff responses are treated as pending updates and can derive the expected post-restart version from `updateAvailable.latestVersion`.

## Best Possible Solution

Land this shape after live managed-service Dashboard update proof confirms the skipped banner is gone and version verification stays active, or after a maintainer records an explicit proof override.

## Reproduction Assessment

Yes. The current-main source path is reproducible by feeding `runUpdate` the documented Gateway handoff response: `ok: true`, `result.status: "skipped"`, and `reason: "managed-service-handoff-started"`; live managed-service reproduction was not performed in this read-only review.

## Solution Assessment

Yes for the code shape. Treating Gateway `ok: true` as pending success matches the server/docs contract, and the `updateAvailable.latestVersion` fallback is the narrowest way to keep the existing post-restart verifier active when the handoff payload has no `after.version`.
