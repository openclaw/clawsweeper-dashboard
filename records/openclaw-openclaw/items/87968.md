## Best Possible Solution

Land this UI fix only after redacted live managed-service Dashboard proof or an explicit maintainer proof override, while keeping the canonical issue open until the full flow is verified.

## Reproduction Assessment

Yes at source level: current main's gateway started-handoff payload has no `after.version`, while current `runUpdate` can only derive the expected version from `after.version`. I did not reproduce the live Dashboard managed-service update flow.

## Solution Assessment

Yes for the UI state layer: falling back to `updateAvailable.latestVersion` is the narrowest UI-side way to keep post-restart version verification active for the documented handoff payload. It does not settle the separate backend cwd or launchd update-contract questions.
