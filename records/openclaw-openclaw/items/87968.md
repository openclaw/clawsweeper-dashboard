## Summary

Current main now suppresses the erroneous skipped banner and tracks the asynchronous handoff, but this PR still adds a meaningful target-version check that is not yet on main; the patch looks correct, with live managed-service proof remaining the merge gate.

## What This Changes

The PR makes the Control UI derive the pending update’s expected version from `updateAvailable.latestVersion` when a managed-service handoff response omits `result.after.version`, with focused regression tests and handoff guidance text.

## Best Possible Solution

Retain the focused fallback and tests, then prove a real managed-service update reaches the final sentinel and verifies the installed version against the selected target without showing the skipped banner.

## Reproduction Assessment

Yes, from source with high confidence: the real handoff response has no `after.version`, while current main stores only that field as the expected version. The submitted tests reproduce the controller boundary, but the integrated live path was not run.

## Solution Assessment

Yes, provisionally: using the already-loaded update target is a narrow UI-side repair and preserves the existing Gateway contract; live validation should confirm the cached target stays aligned with the configured channel.
