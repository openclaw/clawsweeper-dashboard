## What This Changes

The PR updates Control UI `runUpdate` to use `updateAvailable.latestVersion` when a managed-service handoff lacks `result.after.version`, adds handoff guidance text, and adds controller regression tests.

## Best Possible Solution

Land this as a scoped UI verification-state fix only after real managed-service Dashboard proof or maintainer override, and keep the linked issue open until the backend handoff path is also resolved or proven unnecessary.

## Reproduction Assessment

Yes, source-reproducible: current main's Gateway handoff response has no `after.version`, and current `runUpdate` therefore leaves `pendingUpdateExpectedVersion` null. I did not run a live managed-service Dashboard update in this read-only review.

## Solution Assessment

Yes for the UI-state gap: using the existing `updateAvailable.latestVersion` is a narrow controller-side fallback because the Gateway cannot know the detached helper's final version at handoff start. It is not sufficient by itself to close the full linked Dashboard update issue while backend handoff work remains open.
