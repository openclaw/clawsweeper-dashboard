## Best Possible Solution

Land the UI fallback after redacted live managed-service Dashboard proof or a maintainer proof override, and let the linked issue close only once maintainers are satisfied the full Dashboard update flow is resolved.

## Reproduction Assessment

Yes at source level: current main's gateway returns a started managed-service handoff with no `after.version`, and current `runUpdate` therefore cannot set the expected final version from that payload. A live managed-service Dashboard repro was not established in this read-only review.

## Solution Assessment

Yes for the UI-side gap: using the already-known `updateAvailable.latestVersion` is a narrow maintainable fallback when handoff initiation cannot report an `after.version`. It is not proof-sufficient until the real managed-service Dashboard flow is validated or overridden by a maintainer.
