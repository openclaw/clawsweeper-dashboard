## Summary

No blocking code finding: this PR remains useful because current main handles started handoffs as pending but still loses the target version when the handoff response has no `after.version`. The remaining merge gate is live managed-service Dashboard proof or a maintainer proof override.

## What This Changes

The PR updates the Control UI update controller to derive the pending update version from `updateAvailable.latestVersion` for started managed-service handoffs and adds focused regression coverage.

## Best Possible Solution

Land the narrow UI fallback and tests once a maintainer accepts the proof gap, supplies managed-service Dashboard proof, or applies a proof override.

## Reproduction Assessment

Yes, source-reproducible: current main's gateway handoff response has no `after.version`, and current `runUpdate` therefore stores a null expected version for that path. I did not run a live managed-service update in this read-only review.

## Solution Assessment

Yes, the PR is the narrow UI-side fix for the remaining gap: use the already-known update target instead of inventing an `after` version before the detached helper has run. Changing the Gateway to report a final version at handoff start would be the wrong layer.
