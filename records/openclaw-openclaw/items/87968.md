## Summary

Keep open: current main still misclassifies a started managed-service update handoff as skipped, and this patch is a focused match for the gateway contract with targeted regression tests. There are no blocking code findings, but this external PR still needs live managed-service Dashboard proof or an explicit maintainer proof override before merge.

## What This Changes

The PR updates Control UI `runUpdate` to treat `ok: true` managed-service handoff responses as pending, derive the expected post-restart version from `updateAvailable`, and add regression coverage.

## Best Possible Solution

Land the focused UI fix after redacted live managed-service Dashboard proof or an explicit maintainer proof override confirms the handoff and reconnect path.

## Reproduction Assessment

Yes, this is source-reproducible with high confidence: current main’s gateway contract returns `ok: true` for a started handoff while current main’s UI only treats `status === "ok"` as pending. I did not run a live managed-service install locally.

## Solution Assessment

Yes, this appears to be the best code-level fix: `runUpdate` should trust the gateway payload `ok` field and reuse the existing `updateAvailable.latestVersion` state to keep post-restart verification active when the handoff response has no `after.version`. The remaining blocker is proof quality, not a different implementation layer.
