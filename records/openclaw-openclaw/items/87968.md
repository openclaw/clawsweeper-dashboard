## What This Changes

The PR updates `runUpdate` to use `updateAvailable.latestVersion` when a started managed-service handoff lacks `result.after.version`, and adds focused controller tests for that payload shape.

## Best Possible Solution

Land the narrow UI fallback and tests after live managed-service Dashboard proof or a maintainer proof override, while keeping separate gateway handoff failures in their own linked PRs.

## Reproduction Assessment

Yes for the source-level gap: current main's Gateway handoff payload has no `after.version`, and current `runUpdate` therefore leaves the expected version null. I did not establish a live managed-service Dashboard click-through repro in this read-only review.

## Solution Assessment

Yes, this is the best narrow fix for the PR's UI-side gap: the Gateway cannot know the final `after.version` at handoff start, while the UI already has the target version in `updateAvailable.latestVersion`. The remaining blocker is proof, not a better code location.
