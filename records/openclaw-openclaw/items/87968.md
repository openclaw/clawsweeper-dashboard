## Summary

Keep open: current main still routes a started managed-service handoff into the skipped-update banner, and this PR is a focused fix for that UI bug. I found no blocking code defect, but the external real-behavior proof is still insufficient because it uses a mocked gateway client rather than a live managed-service Dashboard flow or maintainer proof override.

## What This Changes

The PR changes the Control UI update controller to treat successful managed-service handoff responses as pending updates and adds regression coverage for the handoff payload.

## Best Possible Solution

Land the focused UI controller fix after redacted live managed-service Dashboard proof or an explicit maintainer proof override confirms the handoff starts without the skipped-update banner and version verification still runs.

## Reproduction Assessment

Yes from source: the gateway can return `ok: true` with `result.status: "skipped"` for started managed-service handoffs, while current main’s UI success branch requires `status === "ok"`. I did not run a live managed-service Dashboard reproduction.

## Solution Assessment

Yes: using the payload-level `ok` signal is the narrowest maintainable UI fix because the gateway already defines started handoff as `ok: true`. Special-casing only the handoff reason in the UI would duplicate gateway semantics and be more brittle.
