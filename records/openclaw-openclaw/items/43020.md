## Summary

Keep this issue open. Current main still has the Mattermost draft-preview overwrite path, and open PR #70508 is already the active implementation path for the same draft-boundary bug.

## What This Changes

Request that Mattermost edit-in-place streaming preserve each assistant/tool-call segment as its own visible post instead of overwriting one draft and batching final replies.

## Best Possible Solution

Keep this open until #70508 or an equivalent Mattermost-local fix rotates or finalizes the draft at assistant/reasoning/tool boundaries and adds focused regression coverage.

## Reproduction Assessment

Yes. The source-level path is high confidence: the inbound Mattermost run reuses one draftStream, disables core block streaming, and later assistant/tool boundaries update that same draft instead of rotating it.

## Solution Assessment

Yes. A Mattermost-local boundary fix using the existing forceNewMessage draft primitive is the narrowest maintainable direction; changing core block-streaming semantics would risk other channels.
