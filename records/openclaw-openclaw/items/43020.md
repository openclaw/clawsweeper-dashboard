## Summary

Keep this issue open. Current main still reuses one Mattermost draft preview across assistant/reasoning/tool boundaries while core block delivery is disabled, and open PR #70508 is already the active implementation path for this same overwrite/batched-final behavior.

## What This Changes

Mattermost edit-in-place streaming should keep each assistant text segment visible across tool-call boundaries instead of overwriting one draft preview and sending accumulated final replies at completion.

## Best Possible Solution

Land or replace #70508 with a Mattermost-local boundary fix that rotates/finalizes the draft at assistant, reasoning, and tool-start boundaries, with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main disables core block streaming for the Mattermost preview path, streams partial text into one draftStream, and never rotates that draft at assistant/reasoning/tool boundaries.

## Solution Assessment

Yes. A Mattermost-local boundary rotation using the existing draftStream.forceNewMessage primitive is the narrowest maintainable direction; changing core block-delivery semantics would risk other channels.
