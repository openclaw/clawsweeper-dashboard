## Summary

Keep this issue open. Current main still uses one Mattermost draft stream for the inbound run, disables core block streaming on that preview path, and does not rotate or seal the draft at assistant/reasoning/tool boundaries, so the reported overwrite and batched-final behavior remains source-reproducible; open PR #70508 is already the active implementation path.

## What This Changes

The issue asks Mattermost edit-in-place streaming to preserve each assistant segment in multi-tool-call responses instead of overwriting one draft post and posting all final messages at completion.

## Best Possible Solution

Keep this issue open until a Mattermost-local fix lands that rotates or finalizes the draft at assistant/reasoning/tool boundaries, prevents duplicate final delivery for preview-owned segments, and adds focused regression coverage for partial text, tool boundary, next assistant text, and final delivery.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: an inbound Mattermost turn calls onPartialReply into one draftStream, then a tool or assistant boundary occurs, then later partial text updates the same post because the boundary callbacks only reset lastPartialText.

## Solution Assessment

Yes, a Mattermost plugin fix is the narrowest maintainable direction because the overwrite happens in the Mattermost draft-preview callback wiring. A broader core block-streaming change would risk changing other channels that already manage draft boundaries differently.
