## Summary

Keep this issue open. Current main still reuses one Mattermost draft stream for the inbound run, disables core block streaming on that path, and only clears local partial-text state at assistant/reasoning boundaries while tool starts update the same draft post; open PR #70508 targets this bug but has not merged. https://github.com/openclaw/openclaw/issues/43020

## What This Changes

Mattermost edit-in-place streaming should preserve each assistant segment during multi-turn tool-call responses instead of overwriting the same draft post and batching finals at the end.

## Best Possible Solution

Land or replace a Mattermost-local fix that rotates or finalizes draft previews at assistant, reasoning, and tool boundaries, avoids duplicate final delivery for preview-owned segments, and adds focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level path exists on current main: an inbound Mattermost turn streams partial text into one draftStream, then assistant/reasoning/tool boundaries do not rotate that stream, so later partials keep editing the same post while final replies are deferred.

## Solution Assessment

Yes. A Mattermost plugin fix that calls the existing draft boundary primitive is the narrowest maintainable direction; changing core block-streaming delivery would risk other channels that manage preview boundaries differently.
