## Summary

Keep this issue open. Current main still uses one Mattermost draft stream for the inbound run, disables core block streaming on that path, and does not rotate or seal the draft at assistant/tool boundaries, so the reported multi-turn overwrite/batched-final behavior remains plausible. An open related PR, #70508, is already attempting the Mattermost-local fix, so the next step is review/land or replace that PR rather than close this issue.

## What This Changes

The issue requests that Mattermost edit-in-place streaming deliver each assistant segment in a multi-tool-call response as its own finalized post instead of overwriting one draft post and batching final messages at completion.

## Best Possible Solution

Keep this issue open until a Mattermost-local fix lands. The fix should finalize or rotate the draft preview when an assistant/tool/reasoning boundary arrives after visible streamed text, avoid duplicate final delivery for preview-delivered segments, and add a focused regression test covering partial text, tool boundary, next assistant text, and final delivery.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: an inbound Mattermost turn with preview streaming calls onPartialReply, then a tool or assistant boundary, then another partial reply. Current main keeps the same draftStream and only resets lastPartialText at those boundaries, so later updates target the same Mattermost post.

## Solution Assessment

Yes, a Mattermost plugin fix is the narrowest maintainable direction because the overwrite happens in the Mattermost draft-preview callback wiring. A broader core block-streaming change would risk affecting other channels that already handle draft boundaries differently.
