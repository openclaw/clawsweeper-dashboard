## Summary

Keep open: this is a plausible gateway sharing feature, but the PR still exposes bearer-token minting before the auth, delegation, lifecycle, and storage policy are complete, and the contributor has not supplied real gateway proof.

## What This Changes

The PR adds gateway session-share token RPC schemas, WRITE-scope method registration, handlers for creating and revoking time-limited tokens, a JSONL-backed in-memory token store, and focused token-store tests.

## Best Possible Solution

Complete this as a cohesive, security-reviewed gateway sharing feature or keep the methods unregistered while maintainers settle the broader token identity/session model in https://github.com/openclaw/openclaw/issues/43903.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the new RPCs mint tokens, while current main has no auth path that consumes them and the PR does not add one. I did not run a gateway because this is a read-only review and the gap is visible in the diff and current source.

## Solution Assessment

No. Exposing public mint/revoke methods before auth enforcement, delegation policy, lifecycle initialization, idempotency, compaction, and storage threat model are complete is not the narrowest maintainable solution.

- **[P2] Wire shared tokens into gateway auth:** `src/gateway/server-methods/session-share-tokens.ts:31-35`
  - body: `sessions.createSharedToken` returns a bearer token, but the patch does not update the WebSocket or HTTP connect auth path to resolve that token into read-only, session-bound access. As merged, this publishes an unusable token API and a security contract with no enforcement; wire the auth path in this PR or leave the method unregistered.
  - confidence: 0.91
- **[P2] Check session delegation before minting tokens:** `src/gateway/server-methods/session-share-tokens.ts:25-29`
  - body: The create handler accepts a caller-supplied `sessionKey` and mints a durable token without proving the session exists or that the caller may delegate read access to it. A WRITE-scope caller could pre-mint a token for another session before the follow-up auth path makes those tokens usable.
  - confidence: 0.88
- **[P2] Initialize token persistence during gateway startup:** `src/gateway/server-methods.ts:99`
  - body: Registering the handler does not enable the advertised JSONL persistence because `initSessionShareTokens` is never called from gateway startup. In a real gateway process, token creation remains memory-only and restart loading never happens.
  - confidence: 0.89
- **[P2] Compact expired token records automatically:** `src/infra/session-share-tokens.ts:121-122`
  - body: Token creation appends every token to JSONL, while expired entries are compacted only by explicit revoke or a manual `pruneExpiredTokens` call. Add a startup, periodic, or threshold compaction path before this becomes a long-lived credential file.
  - confidence: 0.84
- **[P3] Require idempotency for token creation:** `src/gateway/protocol/schema/sessions.ts:328-332`
  - body: The protocol docs require idempotency keys for side-effecting methods, but `sessions.createSharedToken` has no `idempotencyKey`. A retried request after a lost response can mint multiple valid bearer tokens for the same share action.
  - confidence: 0.78
