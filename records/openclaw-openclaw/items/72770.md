## Summary

Keep open: current main does not have session-share tokens, and the PR still exposes a public gateway bearer-token feature before the auth enforcement, session delegation, persistence lifecycle, and real behavior proof are complete.

## What This Changes

The PR adds gateway protocol schema/scope/handler entries plus a JSONL-backed in-memory store and tests for time-limited session-share tokens.

## Best Possible Solution

Land session sharing only as a complete gateway feature: session existence/delegation checks, read-only session-bound auth resolution, startup persistence initialization, automatic compaction, docs/client follow-through, and integration tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: the PR registers `sessions.createSharedToken` and returns a bearer token, while current gateway connect auth accepts only existing token/password/bootstrap/device paths and never resolves the new share token.

## Solution Assessment

No. Exposing mint/revoke methods before the auth boundary, session delegation policy, lifecycle, and protocol/docs follow-through are complete is not the narrowest maintainable solution; either complete those in this PR or leave the handlers unregistered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire shared tokens into gateway auth:** `src/gateway/server-methods/session-share-tokens.ts:32-35`
  - body: `sessions.createSharedToken` returns a bearer token, but the patch never updates the WebSocket/connect auth path to resolve that token into read-only, session-bound scopes. As merged, the public RPC mints unusable tokens; wire enforcement in this PR or keep the method unregistered until the feature is complete.
  - confidence: 0.91
- **[P2] Check session access before minting tokens:** `src/gateway/server-methods/session-share-tokens.ts:27-30`
  - body: The create handler accepts a caller-supplied `sessionKey` and mints a durable token without checking that the session exists or that the caller may delegate access to it. Add an explicit session existence and delegation check before issuing share tokens.
  - confidence: 0.88
- **[P2] Initialize token persistence during gateway startup:** `src/gateway/server-methods.ts:99`
  - body: Registering the handler does not enable the advertised JSONL persistence: `persistencePath` is set only by `initSessionShareTokens`, and the patch does not call that initializer from gateway startup. In a real gateway process, token creation stays memory-only and restart loading never happens.
  - confidence: 0.89
- **[P2] Compact expired token records automatically:** `src/infra/session-share-tokens.ts:128`
  - body: Every token creation appends a JSONL record, while expired entries are compacted only by explicit revoke or a manual `pruneExpiredTokens` call. Add a startup, periodic, or threshold compaction path so the file does not retain every token ever minted.
  - confidence: 0.84
- **[P3] Require idempotency for token creation:** `src/gateway/protocol/schema/sessions.ts:328-332`
  - body: Gateway protocol docs require idempotency keys for side-effecting methods, but `sessions.createSharedToken` has no `idempotencyKey`. A retried request after a lost response can mint multiple valid bearer tokens for the same share action.
  - confidence: 0.78
