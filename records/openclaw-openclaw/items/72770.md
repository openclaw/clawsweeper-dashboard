## Summary

Keep open. Current main does not already implement session sharing, and the PR is an active feature proposal, but the patch is not safe to merge as-is because it exposes token-minting RPCs before the gateway auth boundary, session delegation policy, persistence lifecycle, docs, and changelog are complete.

## What This Changes

The PR branch adds gateway protocol schemas, WRITE-scoped create/revoke RPCs, handler registration, and an in-memory/JSONL-backed session-share token store with infra tests.

## Best Possible Solution

Land session sharing as a complete gateway feature: token creation/revocation, session existence and delegation checks, websocket/auth resolution into read-only session-bound scopes, startup persistence initialization, automatic compaction, protocol docs, changelog, and integration tests for expiry, revocation, restart, and scoped read access.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: inspect the PR patch and current main to see the RPC returns a bearer token while no connect/auth path resolves that token, and the token store initializer is only exercised by tests.

## Solution Assessment

No. The current PR is not the best way to solve the feature because it publishes public gateway methods before the security boundary and production lifecycle are complete; the safer path is one coherent implementation or leaving the RPCs unregistered until auth is ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire shared tokens into gateway auth:** `src/gateway/server-methods/session-share-tokens.ts:27-36`
  - body: `sessions.createSharedToken` returns a bearer token, but the patch never teaches the websocket connect/auth path to resolve that token into read-only, session-bound scopes. Wire the auth middleware in this PR or leave the method unregistered until the token is usable.
  - confidence: 0.91
- **[P2] Check session access before minting tokens:** `src/gateway/server-methods/session-share-tokens.ts:27-31`
  - body: The create handler accepts any caller-supplied `sessionKey` and mints a durable token without checking that the session exists or that the caller may delegate access to it. Add the session existence/delegation rule before issuing bearer tokens.
  - confidence: 0.86
- **[P2] Initialize token persistence at startup:** `src/gateway/server-methods.ts:104`
  - body: Registering the handler does not enable the advertised JSONL persistence: `persistencePath` is only set by `initSessionShareTokens`, and this patch only calls that initializer from tests. In a real gateway process token creation stays memory-only and restart loading never happens.
  - confidence: 0.89
- **[P2] Compact expired tokens automatically:** `src/infra/session-share-tokens.ts:143-146`
  - body: Every token creation appends a JSONL record, while expired entries are compacted only by explicit revoke or a manual `pruneExpiredTokens` call. Add a startup, periodic, or threshold compaction path so the file does not retain every token ever minted.
  - confidence: 0.86
- **[P3] Require idempotency for token creation:** `src/gateway/protocol/schema/sessions.ts:327-334`
  - body: Gateway protocol docs require idempotency keys for side-effecting methods, but `sessions.createSharedToken` has no `idempotencyKey`. A retried request after a lost response can mint multiple valid bearer tokens for the same share action.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/method-scopes.ts:169-170`
  - body: This adds public gateway protocol methods, but the PR file list does not include `CHANGELOG.md`. Repo policy requires a user-facing feat change to add an active-version changelog entry with appropriate attribution before merge.
  - confidence: 0.84
