## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Validate session authority before minting share tokens:** `src/gateway/server-methods/session-share-tokens.ts:27-31`
  - body: This handler turns any validated `sessionKey` string into a bearer token without checking that the session exists or that the caller may share it. Once token auth is wired, a write-scoped caller could mint read tokens for sessions they should not expose, so resolve and authorize the session before creating the token.
  - confidence: 0.91
- **[P1] Store share tokens in SQLite-owned state:** `src/infra/session-share-tokens.ts:90`
  - body: This appends full bearer-token records to `session-share-tokens.jsonl` under the state directory. OpenClaw runtime state is SQLite-owned, and a revocable token store needs durable prune, revoke, backup, and upgrade semantics instead of a new JSONL sidecar.
  - confidence: 0.88
- **[P2] Move token schemas to the current protocol package:** `src/gateway/server-methods/session-share-tokens.ts:8-10`
  - body: The new handler imports validators from `../protocol/index.js`, and the PR updates the old `src/gateway/protocol` tree that current main no longer uses. Put the schemas and exports in `packages/gateway-protocol` and register the methods through the current descriptor table so clients and server policy share one source of truth.
  - confidence: 0.84
- **[P2] Keep token methods private until auth consumes them:** `src/gateway/server-methods-list.ts:99-100`
  - body: The PR advertises create/revoke methods, but the PR body says the auth middleware that would consume these tokens is a follow-up. That publishes a token API users cannot use for read-only access and commits the protocol before the security contract is reviewed.
  - confidence: 0.84
