## Summary

Keep this PR open for human review because it proposes a new public gateway bearer-token feature, but the patch exposes token creation before the auth, session-delegation, persistence lifecycle, and runtime proof are complete.

## What This Changes

The PR adds gateway schema/scope/handler entries plus an in-memory JSONL-backed token store and tests for time-limited read-only session-share tokens.

## Best Possible Solution

Land session sharing only as a complete gateway feature: session existence/delegation checks, read-only session-bound auth resolution, startup persistence initialization, automatic compaction, docs/client follow-through, and integration tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR returns bearer tokens from a public RPC, while current gateway auth has no path that resolves those tokens into read-only, session-bound scopes.

## Solution Assessment

No. Publishing public token-minting methods before the security boundary and lifecycle are complete is not the narrowest maintainable solution; either wire the full feature in this PR or leave the token store unregistered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire shared tokens into gateway auth:** `src/gateway/server-methods/session-share-tokens.ts:32-36`
  - body: sessions.createSharedToken returns a bearer token, but the patch never teaches the WebSocket/connect auth path to resolve it into read-only, session-bound scopes. As merged, the public RPC mints unusable tokens; wire enforcement in this PR or keep the method unregistered until the feature is complete.
  - confidence: 0.91
- **[P2] Check session access before minting tokens:** `src/gateway/server-methods/session-share-tokens.ts:27-30`
  - body: The create handler accepts a caller-supplied sessionKey and mints a durable token without checking that the session exists or that the caller may delegate access to it. Add an explicit session existence and delegation check before issuing share tokens.
  - confidence: 0.87
- **[P2] Initialize token persistence during gateway startup:** `src/gateway/server-methods.ts:102`
  - body: Registering the handler does not enable the advertised JSONL persistence: persistencePath is set only by initSessionShareTokens, and the patch does not call that initializer from gateway startup. In a real gateway process, token creation stays memory-only and restart loading never happens.
  - confidence: 0.89
- **[P2] Compact expired token records automatically:** `src/infra/session-share-tokens.ts:139`
  - body: Every token creation appends a JSONL record, while expired entries are compacted only by explicit revoke or a manual pruneExpiredTokens call. Add a startup, periodic, or threshold compaction path so the file does not retain every token ever minted.
  - confidence: 0.85
- **[P3] Require idempotency for token creation:** `src/gateway/protocol/schema/sessions.ts:328-332`
  - body: Gateway protocol docs require idempotency keys for side-effecting methods, but sessions.createSharedToken has no idempotencyKey. A retried request after a lost response can mint multiple valid bearer tokens for the same share action.
  - confidence: 0.78
