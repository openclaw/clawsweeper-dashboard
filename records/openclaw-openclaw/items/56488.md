## Summary

Keep open. Current main still has the same source-level due-selection gap: the normal timer scan can select a past-due terminal one-shot `at` job because it does not pass the existing completed-job guard, and this issue is already paired with an open closing PR that should be reviewed, rebased, landed, or replaced.

## What This Changes

`kind: "at"` cron jobs with `deleteAfterRun: true` should run once and then be removed instead of re-triggering from a stale past `nextRunAtMs`.

## Best Possible Solution

Use the existing terminal one-shot guard consistently for every non-forced due scan, including the normal timer and manual due path, with regression coverage for a successful `ok` one-shot job whose stale `nextRunAtMs` remains in the past while preserving transient-error retries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, and a past `nextRunAtMs`, then run the normal timer or non-forced due predicate; current main still treats it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard across non-forced due scans is the narrowest maintainable fix; adding a migration, retry policy, or new configuration knob would be broader than needed.
