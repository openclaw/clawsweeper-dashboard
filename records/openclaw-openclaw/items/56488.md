## Summary

Keep open: current main and the latest release still have source-visible cron scheduler gaps for completed one-shot jobs, and the only target-repo PR that directly tried the small guard fix was closed unmerged. An open same-repo PR appears to cover the buffered completion/deletion path, but it is still unmerged and needs maintainer review rather than cleanup closure.

## What This Changes

The issue reports that one-shot cron jobs with `deleteAfterRun: true` can repeatedly re-trigger after a successful run instead of being removed.

## Best Possible Solution

Keep this issue open and either rebase/review/land https://github.com/openclaw/openclaw/pull/81572 or land a narrow replacement that finalizes completed due-job outcomes promptly and prevents terminal one-shot `at` jobs left in the store from being selected again by the normal timer path.

## Reproduction Assessment

Yes. Source inspection shows ordinary due-job outcomes are still buffered until the worker batch drains, and terminal `at` jobs are skipped only when `skipAtIfAlreadyRan` is passed, while the normal timer path still omits that flag; no tests were run because this review is read-only.

## Solution Assessment

Yes, the narrow scheduler-layer fix is the right direction, but the active path should be the open focused due-outcome persistence PR or a small replacement rather than the closed mixed branch. The failed-run deletion policy should stay in the separate linked issue.
