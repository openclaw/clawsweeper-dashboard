## Summary

Keep open: current main still matches the source-level failure shape, and the strongest same-repo fix path is an open but conflicting PR rather than a merged implementation. The docs still define successful one-shot `--at` jobs as auto-deleting, so the reported behavior remains a valid core cron bug.

## What This Changes

The issue reports that one-shot `kind: at` cron jobs with `deleteAfterRun: true` can complete successfully and then re-trigger repeatedly instead of being removed.

## Best Possible Solution

Land or replace the focused scheduler fix that finalizes successful one-shot outcomes promptly and prevents terminal `at` jobs left in the store from being selected again by the normal timer path.

## Reproduction Assessment

Yes, at source level. Current main still leaves the normal timer path without the terminal one-shot guard and still delays ordinary outcome finalization until the due-job batch drains; no executable repro was run because this review is read-only.

## Solution Assessment
