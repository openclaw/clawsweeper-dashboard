## Label Justifications

- `P1`: Repeated one-shot cron execution can spawn duplicate isolated agent sessions and duplicate outbound posts for existing users.
- `impact:session-state`: The bug is about stale cron run state allowing an already-terminal one-shot job to be selected again.
- `impact:message-loss`: The report describes repeated generated Discord posts from one intended one-shot job, which is duplicate message delivery.

## Summary

Keep open: current main and the latest release still let the normal cron timer select a terminal one-shot `at` job without the completed-job guard that startup catch-up already uses, and the open fork PR that claims the fix is still unmerged and dirty.

## What This Changes

The issue requests that one-shot cron jobs with `deleteAfterRun` run once and not retrigger after a successful terminal run leaves stale scheduler state.

## Best Possible Solution

Make normal timer selection honor the existing terminal one-shot guard, preserve explicit retry exceptions, and add focused regression coverage for terminal `at` jobs left in the store after a run.

## Reproduction Assessment

Yes. Source inspection shows the normal `onTimer` path still omits `skipAtIfAlreadyRan`, while startup catch-up passes it, so an enabled terminal `at` job with a past `nextRunAtMs` can be selected again.

## Solution Assessment

No merged solution exists yet. The narrow fix shape is to reuse the existing terminal one-shot guard in normal timer selection, but the currently linked fork PR is dirty and not a clean landing path.
