## Label Justifications

- `P1`: The bug can repeatedly run one-shot cron jobs for real users, spawning duplicate agent sessions and outbound posts.
- `impact:message-loss`: The reported failure duplicated Discord output from a job intended to run exactly once.
- `impact:session-state`: The repeated cron fires created duplicate isolated agent sessions from stale scheduler state.

## Summary

Keep open. Current main still has the cron due-scan gap for completed one-shot `at` jobs, and the open closing-syntax implementation candidate at https://github.com/openclaw/openclaw/pull/56509 has not merged.

## What This Changes

The issue asks for `deleteAfterRun` one-shot `at` cron jobs to run once and be removed instead of retriggering from stale past `nextRunAtMs` state.

## Best Possible Solution

Land or replace a focused cron-only fix that applies the terminal one-shot guard to every non-forced due scan and adds regression coverage for successful completed `at` jobs with stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, no `runningAtMs`, and a past `nextRunAtMs`; current main's normal timer path can still treat it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for all non-forced due scans is the narrow maintainable fix; a migration, retry-policy change, or new config option would be broader than needed.
