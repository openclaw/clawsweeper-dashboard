## Summary

Keep open. Current main still lets normal cron timer and non-forced manual-due scans select a completed one-shot `at` job when `nextRunAtMs` remains in the past, and an open PR already targets this issue with closing syntax.

## What This Changes

The issue reports successful one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` re-firing after completion instead of being removed.

## Best Possible Solution

Land or replace the focused cron fix so every non-forced due scan skips terminal one-shot `at` jobs while preserving transient-error retry eligibility and adding regression coverage for successful `ok` jobs with stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, no `runningAtMs`, and a past `nextRunAtMs`; the normal timer or non-forced manual due path still treats it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for all non-forced due scans is the narrowest maintainable fix; a migration, retry policy change, or new config knob would be broader than needed.
