## Summary

Keep open. Current main still leaves the normal cron timer and manual due predicate able to select an enabled, past-due successful `kind: "at"` job when the terminal one-shot guard is not opted in; the linked closing PR should be reviewed, rebased, landed, or replaced before cleanup.

## What This Changes

The issue requests that one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` run once and then be deleted instead of re-triggering from a stale past `nextRunAtMs`.

## Best Possible Solution

Land or replace the focused cron fix so every non-forced due scan skips terminal one-shot `at` jobs while preserving transient-error retries and adding regression coverage for the successful `ok` stale-`nextRunAtMs` case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, and a past `nextRunAtMs`, then exercise the normal timer or non-forced due path; current main still treats it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for all non-forced due scans is the narrowest maintainable fix; a store migration or new policy knob would be broader than needed.
