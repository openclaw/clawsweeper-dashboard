## Summary

Keep open. Current `main` and the latest release still allow the normal cron timer and non-forced manual due path to select a completed one-shot `at` job with a stale past `nextRunAtMs`; the open closing PR at https://github.com/openclaw/openclaw/pull/56509 should be reviewed, split, landed, or replaced before this issue closes.

## What This Changes

The issue reports one-shot cron `at` jobs with `deleteAfterRun: true` repeatedly firing after a successful run instead of being deleted.

## Best Possible Solution

Apply the existing terminal one-shot guard to every non-forced due scan, including the normal timer and manual due path, while preserving transient-error retry eligibility and adding a regression for a successful `ok` job with a stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled one-shot `at` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, no `runningAtMs`, and a past `nextRunAtMs`; the normal timer or non-forced manual due path still treats it as due.

## Solution Assessment

Yes. Reusing the existing `skipAtIfAlreadyRan` terminal one-shot guard for all non-forced due scans is the narrowest maintainable fix; a migration, retry policy change, or new config option would be broader than needed.
