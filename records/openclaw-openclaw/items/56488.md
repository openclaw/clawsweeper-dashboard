## Label Justifications

- `P1`: The bug can repeatedly fire one-shot cron jobs for real users, spawning duplicate agent sessions and outbound posts.
- `impact:message-loss`: The reported failure duplicated Discord output from a job intended to run exactly once.
- `impact:session-state`: The repeated cron fires created duplicate isolated agent sessions from stale scheduler state.

## Summary

Keep open. Current main and the latest release still call `collectRunnableJobs` from the normal timer path without `skipAtIfAlreadyRan`, so completed one-shot `at` jobs with stale past `nextRunAtMs` remain source-reproducible; the open closing PR should be reviewed, refreshed, or replaced before this issue closes.

## What This Changes

Fix one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` so successful completed jobs cannot be selected again by normal timer due scans.

## Best Possible Solution

Land or replace a focused cron-only fix that applies the existing terminal one-shot guard to every normal due scan while preserving transient retry eligibility and adding regression coverage for successful completed `at` jobs with stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an enabled successful one-shot `at` job with `deleteAfterRun: true`, `lastStatus: "ok"`, no `runningAtMs`, and a past `nextRunAtMs` reaches `onTimer`'s unguarded due scan and can be selected again.

## Solution Assessment

Yes. Reusing the existing `skipAtIfAlreadyRan` guard for normal due scans is the narrow maintainable fix; the open linked PR is the current implementation candidate but needs maintainer review, refresh, or replacement before closure.
