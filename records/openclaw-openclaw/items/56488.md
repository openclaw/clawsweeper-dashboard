## Summary

Keep open. Current main and v2026.5.22 still leave the normal timer due scan without the existing terminal one-shot guard, so a completed `kind: "at"` job with stale past `nextRunAtMs` remains source-reproducible; the open linked implementation PR should be refreshed, narrowed, or replaced before this issue closes.

## What This Changes

The issue reports that `schedule.kind: "at"` cron jobs with `deleteAfterRun: true` can re-trigger after completion instead of being removed.

## Best Possible Solution

Land or replace with a focused cron-only fix that applies the existing already-run `at` guard to every non-forced due scan, preserves retry eligibility, and adds regression coverage for completed one-shot jobs with stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro state: persist an enabled completed `kind: "at"` job with `lastStatus: "ok"`, no `runningAtMs`, and a past `nextRunAtMs`; current `onTimer` can still pass it through the due-time check because it does not enable `skipAtIfAlreadyRan`.

## Solution Assessment

Yes. The existing `skipAtIfAlreadyRan` guard is the narrow maintainable mechanism, but the current open PR needs a focused refresh, split, or replacement before it is the right landing vehicle.
