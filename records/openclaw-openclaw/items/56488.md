## Summary

Keep open. Current main still leaves the terminal one-shot `at` guard opt-in, while the normal timer scan and non-forced manual due check can still select an enabled successful `at` job whose stale `nextRunAtMs` is in the past; linked PR #56509 is open and unmerged.

## What This Changes

The issue reports that successful one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` can remain runnable and fire repeatedly instead of being removed after the first run.

## Best Possible Solution

Land or replace the focused cron scheduler fix so every non-forced due scan skips terminal one-shot `at` jobs while preserving transient-error retries and adding regression coverage for successful `ok` runs.

## Reproduction Assessment

Yes. Persist an enabled `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, and a past `nextRunAtMs`, then exercise the normal timer or non-forced due path; current main treats that stale slot as due by source inspection.

## Solution Assessment

Yes. Reusing or centralizing the existing terminal one-shot guard for normal scheduler and non-forced manual due checks is the narrow maintainable fix; adding a config knob or store migration would be broader than needed.
