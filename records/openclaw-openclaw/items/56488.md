## Summary

Keep open. Current main still applies the completed one-shot guard only when that option is passed, while the normal scheduler tick and manual due predicate can still select a successful past-due `kind: "at"` job; the open closing PR should be reviewed, rebased, landed, or replaced before this issue closes.

## What This Changes

The issue requests that one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` run once and not re-trigger on later scheduler wakes after a successful completion.

## Best Possible Solution

Land or replace the focused cron fix so every non-forced due scan skips terminal one-shot `at` jobs while preserving transient-error retries, with regression coverage for a successful `ok` job whose stale `nextRunAtMs` remains in the past.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, and a past `nextRunAtMs`, then exercise the normal timer or non-forced due path; current main still treats it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for all non-forced due scans is the narrowest maintainable fix; a store migration or new policy knob would be broader than needed.
