## Summary

Keep open. Current main still has the source-reproducible normal timer gap for completed one-shot `at` jobs, and the closing-syntax implementation candidate is still open and unmerged.

## What This Changes

The issue reports that `kind: "at"` cron jobs with `deleteAfterRun: true` can keep re-triggering after a successful run instead of being deleted.

## Best Possible Solution

Land or replace a focused cron fix that applies the terminal one-shot guard to every non-forced due scan while preserving transient retry behavior, with regression coverage for successful completed `at` jobs whose `nextRunAtMs` remains in the past.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful one-shot `at` job with terminal status, no `runningAtMs`, and a past `nextRunAtMs`; the normal timer due scan can still treat it as runnable.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for non-forced due scans is the narrow maintainable fix; the linked PR at https://github.com/openclaw/openclaw/pull/56509 should be reviewed, refreshed, split, or replaced before landing.
