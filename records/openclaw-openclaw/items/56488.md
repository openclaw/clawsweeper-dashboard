## What This Changes

The issue reports that one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` can keep re-running after a successful first execution instead of being removed.

## Best Possible Solution

Land a narrow cron-only fix that applies the completed-one-shot guard to the normal timer path and adds regression coverage for a terminal `at` job with a past `nextRunAtMs`; repair or close the bundled conflicting PR before queueing another branch.

## Reproduction Assessment

Yes at source level: an enabled terminal `at` job with a past `nextRunAtMs` can be selected by normal `onTimer` because that path does not pass `skipAtIfAlreadyRan`. I did not run tests or a live gateway scenario because this review is read-only.

## Solution Assessment

No merged solution yet. The linked PR has the right cron call-site idea, but it is open, conflicting, and bundled with unrelated runtime changes, so the best fix is a narrow cron-only patch plus focused regression coverage.
