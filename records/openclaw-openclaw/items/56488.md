## Summary

Keep open. Current main still has the completed one-shot guard as an opt-in check, but the normal scheduler and due-run paths still reach past-due `at` jobs without that guard; linked PR #56509 also remains open and uses closing syntax for this issue, so the issue should stay open until a focused fix lands or the PR is superseded.

## What This Changes

The issue reports that one-shot cron `kind: "at"` jobs with `deleteAfterRun: true` can re-trigger repeatedly after a successful run instead of being removed.

## Best Possible Solution

Land a focused cron scheduler fix that applies the existing completed-one-shot guard to the normal timer scan and due/manual scan semantics, with regression coverage for an enabled successful `kind: "at"` job whose `lastStatus` is terminal and whose `nextRunAtMs` is still in the past.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to persist an enabled `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, a recorded `lastRunAtMs`, and a past `nextRunAtMs`, then exercise `onTimer` or the due-run path; current main still treats that persisted job as runnable.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix. Reusing the existing `skipAtIfAlreadyRan` terminal one-shot guard is preferable to adding a separate cleanup migration, but the final patch should also cover the public due/manual predicate rather than only one helper call site.
