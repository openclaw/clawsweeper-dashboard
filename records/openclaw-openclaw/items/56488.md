## Summary

Keep open. Current main and the latest release still leave the normal cron timer and manual due paths able to select a completed one-shot `at` job with a stale past `nextRunAtMs`; the open closing PR should be reviewed, split/landed, or replaced before this issue closes.

## What This Changes

The issue reports that successful one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` can run repeatedly instead of being deleted after the first run.

## Best Possible Solution

Land or replace the focused cron fix so every non-forced due scan skips terminal one-shot `at` jobs while preserving transient-error retry eligibility and adding a regression for a successful `ok` job with a stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, `lastRunAtMs`, no `runningAtMs`, and a past `nextRunAtMs`; the normal timer or non-forced manual due path still treats it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for all non-forced due scans is the narrowest maintainable fix; a migration, retry policy change, or new config knob would be broader than needed.
