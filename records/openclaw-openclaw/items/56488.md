## Summary

Keep open. Current main still has the normal cron due-scan gap for completed one-shot `at` jobs, and the linked closing-syntax fix remains open and unmerged.

## What This Changes

The issue asks for `kind: "at"` cron jobs with `deleteAfterRun: true` to run once and be deleted instead of repeatedly re-triggering from stale past scheduler state.

## Best Possible Solution

Land or replace a focused cron-only fix that applies the terminal one-shot guard to every non-forced due scan while preserving transient retry behavior, with regression coverage for successful completed `at` jobs with stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, terminal `lastStatus`, no `runningAtMs`, and a past `nextRunAtMs`; current main's normal timer path can still treat it as due.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for non-forced due scans is the narrow maintainable fix; the current linked PR should be refreshed or replaced before landing because it is open and conflicting.
