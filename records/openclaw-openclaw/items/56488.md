## Label Justifications

- `P1`: Repeated one-shot cron execution can spawn duplicate isolated agent sessions and duplicate outbound posts for existing users.
- `impact:session-state`: The bug is about stale cron run state allowing an already-terminal one-shot job to be selected again.
- `impact:message-loss`: The report describes repeated generated Discord posts from the same intended one-shot job, which is duplicate message delivery.

## Summary

Keep open: current main and v2026.6.8 still let the normal cron timer select terminal one-shot `at` jobs without the completed-job guard that startup catch-up already uses, and the open fork PR that claims the fix is still unmerged and dirty.

## What This Changes

The issue asks cron one-shot `at` jobs with `deleteAfterRun: true` to stop re-triggering from stale past-due state after their first terminal run.

## Best Possible Solution

Make normal timer selection honor the same completed-one-shot guard as startup catch-up, while preserving explicit retry exceptions and adding focused regression coverage for terminal `at` jobs left in the store.

## Reproduction Assessment

Yes, at source level: an enabled terminal `at` job with a past `nextRunAtMs` can still be selected by normal `onTimer` because that path omits `skipAtIfAlreadyRan`. I did not execute a live gateway repro in this read-only review.

## Solution Assessment

No merged solution exists yet. The narrow scheduler guard reuse is the right fix shape, but the current linked fork PR is dirty and not a clean landing path.
