## Label Justifications

- `P1`: The bug can repeatedly run one-shot cron jobs for real users, spawning duplicate agent sessions and outbound posts.
- `impact:message-loss`: The reported failure duplicated Discord output from a job intended to run once.
- `impact:session-state`: The repeated cron fires created duplicate isolated agent sessions from stale scheduler state.

## Summary

Keep open. Current main still has the source-level cron due-scan gap: the normal timer and manual due paths can treat a completed one-shot `at` job with stale past `nextRunAtMs` as runnable, and https://github.com/openclaw/openclaw/pull/56509 is still the open closing-syntax implementation candidate.

## What This Changes

The issue requests that one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` run once and then stop being selected for later scheduler ticks.

## Best Possible Solution

Land or replace a focused cron-only fix that applies the terminal one-shot guard to every non-forced due scan and adds regression coverage for successful completed `at` jobs with stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, no `runningAtMs`, and a past `nextRunAtMs`; current main's normal timer path still treats it as due because it does not pass `skipAtIfAlreadyRan`.

## Solution Assessment

Yes. Reusing the existing terminal one-shot guard for all non-forced due scans is the narrow maintainable fix; a migration, retry-policy change, or new config option would be broader than needed.
