## Summary

Keep open. Current main still lacks the Control UI composer suppression attributes and numeric-preservation regression coverage, and open PR #74369 is the active linked implementation path with closing syntax.

## What This Changes

The issue requests that Control UI chat preserve pure numeric input exactly instead of allowing values like `123456789` to become hyphen-grouped text before sending.

## Best Possible Solution

Land PR #74369, or an equivalent narrow Control UI composer fix, with payload-preservation tests and without Gateway-side deformatting of intentional hyphenated text.

## Reproduction Assessment

No, not as a high-confidence live reproduction: this pass did not run a Windows browser/IME stack. The source path is reproducible enough to keep open because current main forwards `textarea.value` through `chat.send` and lacks the plain-text suppression attributes.

## Solution Assessment

Yes: composer-level input attributes plus exact numeric payload regression coverage are the narrowest maintainable fix. Server-side deformatting would risk corrupting intentionally hyphenated user content.
