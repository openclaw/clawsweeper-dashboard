## What This Changes

The issue asks for Control UI chat composer numeric-only input such as `123456789` to be preserved exactly instead of being auto-formatted with separators.

## Best Possible Solution

Land or replace the narrow Control UI composer-boundary fix with regression coverage, preserving numeric text before `chat.send` rather than rewriting messages downstream.

## Reproduction Assessment

No, not with high confidence: this read-only review did not run a Windows/browser/IME live scenario. Source inspection does show the current composer stores textarea values without plain-text attributes and forwards the stored draft to `chat.send`.

## Solution Assessment

Yes, the best path appears to be the linked composer-boundary fix: prevent browser/OS rewriting before the draft is stored, with tests for attributes and numeric pass-through. Gateway-side deformatting would be riskier because it could corrupt intentional hyphenated or grouped text.
