## Summary

Keep open: current main still has a source-backed Telegram preview path that can treat `MEDIA:` directive finals as editable text before delivery normalization, even though docs and normal delivery support local `MEDIA:` attachments.

## What This Changes

Ensure Telegram direct-chat replies with local-path `MEDIA:` directives deliver attachments and strip directive lines instead of showing them as text.

## Best Possible Solution

Normalize or directive-parse the final payload before Telegram preview edit eligibility is computed, while preserving the existing media file-read and type checks in the delivery path.

## Reproduction Assessment

Yes, source-reproducible: with an active Telegram answer preview, a final payload whose text contains standalone `MEDIA:` lines reaches the preview finalizer before directive parsing and can be classified as text-only. I did not run a live Telegram bot in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to make the Telegram preview/finalization media check use the same directive-normalized payload that normal delivery uses. A safer alternative is not to relax media loading, but to fall back to normal media delivery whenever normalized final parts contain media.
