## Summary

Keep open: current main still has the source-level Telegram preview/finalization path reported here. The shared outbound path can parse `MEDIA:` directives into `mediaUrls`, but the Telegram lane still checks media presence on raw text before that normalization, so an active preview can finalize as text instead of sending attachments.

## What This Changes

Telegram replies containing local-path `MEDIA:` directives should be delivered as attachments and have directive lines stripped from visible text.

## Best Possible Solution

Normalize the final lane segment through the shared outbound payload plan or equivalent directive parser before Telegram decides it is text-only preview-finalizable, while leaving media file-read and type policy in the existing delivery/media loader boundary.

## Reproduction Assessment

Yes, at source level: a final answer segment containing only `MEDIA:/path` can reach `deliverLaneText`, whose media check reads only `payload.mediaUrl(s)` and can finalize the preview before `deliverReplies` parses directives. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make the Telegram preview/finalization media check use a directive-normalized payload shape before deciding whether to reuse the text preview. This avoids changing Telegram upload behavior, file trust policy, or adding configuration.
