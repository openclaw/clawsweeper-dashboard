## Summary

Keep open: current main still has a source-reproducible Telegram preview-finalization bug where directive-bearing MEDIA text can be treated as text-only before shared outbound normalization runs.

## What This Changes

The issue asks Telegram direct-chat replies with local-path MEDIA directives to send those files as attachments and hide the directive lines from visible text.

## Best Possible Solution

Normalize the Telegram lane segment payload before deciding preview edit eligibility, and keep file loading, media type checks, and attachment sends in the existing delivery path.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: a final answer segment containing only MEDIA text can reach createLaneTextDeliverer with no payload.mediaUrls, making hasMedia false before directive parsing. I did not run a live Telegram chat in this read-only review.

## Solution Assessment

Yes: the narrow fix is to make Telegram's preview/finalization media check use the same directive-normalized reply payload that final delivery already uses, rather than adding config or changing upload behavior.
