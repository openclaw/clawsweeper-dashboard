## Summary

Keep open: current main still treats `contextPruning.mode: "off"` as pruning-only, has no public compaction `off` or hard-limit-only mode, and the exact Anthropic 1M-context compaction storm remains unproven live.

## What This Changes

The issue requests clarification and a way to prevent compaction when context pruning is off, especially for Anthropic 1M-context API-key sessions with cache-ttl defaults and multiple auth profiles.

## Best Possible Solution

Make an explicit compaction policy decision: either document that pruning-off does not disable compaction, or add a public hard-disable/hard-limit compaction mode with schema, types, docs, upgrade guidance, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact Anthropic multi-profile compaction storm. Source inspection does confirm the remaining product gap: pruning-off is respected, compaction remains separate, and no public compaction-off or hard-limit-only mode exists.

## Solution Assessment

No, treating `contextPruning.mode: "off"` as an implicit compaction disable would conflate two separate contracts. The maintainable path is an explicit compaction policy decision or context-engine/plugin route with documented overflow semantics.
