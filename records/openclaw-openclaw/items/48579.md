## Summary

Keep open: current main still treats `contextPruning.mode: "off"` as pruning-only, compaction has no public `off` or hard-limit-only mode, and the exact Anthropic multi-profile compaction storm still needs live proof plus a product decision.

## What This Changes

The issue reports frequent compactions despite `contextPruning.mode: "off"` and asks for compaction trigger clarification, Anthropic cache/auth-profile behavior, and a way to fully disable compaction except at a hard limit.

## Best Possible Solution

Make an explicit compaction policy decision: document that pruning-off does not disable compaction, and if maintainers want hard-disable behavior, add a public compaction off or hard-limit-only mode with schema, types, docs, upgrade guidance, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact Anthropic multi-profile compaction storm. Source inspection does reproduce the contract gap: `contextPruning.mode: "off"` disables pruning only, while compaction remains a separate mode surface without public hard-off semantics.

## Solution Assessment

No, treating `contextPruning.mode: "off"` as an implicit compaction disable would conflate two separate contracts. The maintainable path is an explicit compaction policy decision with documented overflow behavior and tests.
