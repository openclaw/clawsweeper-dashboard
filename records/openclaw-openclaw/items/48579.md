## Summary

Keep open: current main still treats `contextPruning.mode: "off"` as pruning-only, while compaction remains a separate `default`/`safeguard` policy with no public hard-off or hard-limit-only mode.

## What This Changes

The issue asks for clarification and/or a way to fully disable compaction except at a hard context limit after frequent Anthropic API-key compactions despite `contextPruning.mode: "off"`.

## Best Possible Solution

Settle an explicit compaction contract first, then implement any hard-disable or hard-limit mode across config types, schema, docs, overflow recovery, focused tests, and live Anthropic proof.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact Anthropic multi-profile compaction storm. Source inspection confirms the contract gap: `contextPruning.mode: "off"` disables pruning only, while current compaction has no public hard-off mode.

## Solution Assessment

Unclear as an implementation path. The maintainable direction is an explicit compaction policy with documented overflow behavior, not making pruning-off implicitly disable a separate compaction engine.
