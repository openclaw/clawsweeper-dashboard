## Summary

Keep open: current main still respects explicit `contextPruning.mode: "off"`, but pruning and compaction remain separate systems, `compaction.mode` still has no `off` or hard-limit-only mode, and the low-context compaction trigger family is not proven fixed on main or in v2026.6.10.

## What This Changes

The issue requests investigation and repair for aggressive low-context compactions despite pruning being off and a high reserve floor, plus clearer compaction trigger semantics and a hard-limit-only disable option.

## Best Possible Solution

Fix any verified low-context compaction trigger accounting bug, persist a closed compaction trigger reason, and add hard-limit-only/off semantics only after maintainers accept the safety and config contract.

## Reproduction Assessment

No, not for the exact Anthropic API-key/cache-TTL sequence on current main. Source inspection does show a plausible low-context compaction path through successful-turn overflow accounting, and related issue comments provide real provider-independent examples.

## Solution Assessment

No, a pruning-defaults-only fix is not the best solution because explicit pruning off is already respected. The maintainable path is in compaction trigger accounting, durable trigger metadata, and a maintainer decision on hard-limit-only/off semantics.
