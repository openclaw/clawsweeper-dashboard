## Summary

Keep open: current main still treats `contextPruning.mode: "off"` as pruning-only, has no public compaction `off` or hard-limit-only mode, and intentionally keeps Pi auto-compaction enabled for ordinary Anthropic/default-mode runs. The exact Anthropic 1M-context compaction storm was not live-reproduced, so this needs maintainer runtime/product judgment rather than an auto-close or autonomous fix.

## What This Changes

The issue asks OpenClaw to clarify why compaction still runs with `contextPruning.mode: "off"` and to provide a way to disable compaction except at a hard context limit.

## Best Possible Solution

Make an explicit maintainer decision: either keep built-in compaction as summarization-only and route hard-disable policies to context-engine plugins, or add a documented `compaction` hard-disable/hard-limit mode with overflow behavior, schema/types/docs, upgrade guidance, and regression coverage; do not make `contextPruning.mode: "off"` implicitly disable compaction.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact Anthropic multi-profile compaction storm. Source inspection does confirm the remaining product gap: pruning-off is respected, compaction remains separate, and no public compaction-off or hard-limit-only mode exists.

## Solution Assessment

No, treating `contextPruning.mode: "off"` as an implicit compaction disable would conflate two separate contracts. The maintainable path is an explicit compaction policy decision or context-engine/plugin route with documented overflow semantics.
