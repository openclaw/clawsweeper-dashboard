## Summary

Keep open: current main respects `contextPruning.mode: "off"` as a pruning-only setting, but compaction remains a separate always-available runtime path with no public `off` or hard-limit-only mode, and the reported Anthropic multi-profile compaction storm still lacks live current-main proof.

## What This Changes

The issue reports aggressive low-usage compactions despite `contextPruning.mode: "off"` and requests clearer compaction triggers plus a way to disable compaction except near the hard context limit.

## Best Possible Solution

Make an explicit compaction policy decision: either document that pruning-off does not disable compaction, or add a public hard-disable/hard-limit compaction mode with schema, types, docs, upgrade guidance, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact Anthropic multi-profile compaction storm. Source inspection does reproduce the contract gap: `contextPruning.mode: "off"` disables pruning only, while default compaction remains separate and has no public `off` mode.

## Solution Assessment

No, treating `contextPruning.mode: "off"` as an implicit compaction disable would conflate two separate contracts. The maintainable path is an explicit compaction policy decision with documented overflow behavior and tests.
