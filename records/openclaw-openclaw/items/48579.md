## Summary

Keep open: current main and v2026.6.6 still treat context pruning and compaction as separate systems. The explicit pruning-off path is respected, but there is still no accepted hard-limit-only/off compaction mode, and related issues overlap without fully owning this Anthropic/cache-TTL configuration request.

## What This Changes

The issue requests clarification and a fix for frequent low-context compactions when Anthropic API-key sessions have `contextPruning.mode: "off"`, plus a way to disable compaction except at a hard limit.

## Best Possible Solution

Define the compaction trigger contract, persist enough trigger metadata to distinguish overflow/threshold/manual/background causes, and add a hard-limit-only or off mode only if maintainers accept that safety tradeoff.

## Reproduction Assessment

No for the exact Anthropic API-key/cache-TTL runaway path on current main. Source inspection does confirm the durable behavior that `contextPruning.mode: "off"` disables pruning only and does not disable the independent compaction system.

## Solution Assessment

No for a pruning-defaults-only fix: explicit pruning off is already respected. The maintainable solution belongs in compaction trigger accounting, trigger metadata, and a maintainer decision on any hard-limit-only config mode.
