- `impact:security`: The requested policy changes hardening around config, credentials, approvals, auth files, and other sensitive local state.

- `impact:auth-provider`: Device auth, credentials directories, auth profiles, and provider/config secrets are included in the reported permission surface.

## Summary

Keep open. Current main and the latest release still enforce private 0600/0700-style modes across config/state, security repair, cron, exec approvals, and fs-safe-backed private stores; the open related PRs cover only slices of the broader sensitive-state permission policy.

## What This Changes

The issue requests configurable or opt-out file and directory modes for OpenClaw-managed state so multi-user POSIX ACL/group setups are not repeatedly clamped back to owner-only permissions.

## Best Possible Solution

Keep private 0600/0700 defaults, then design a maintainer-approved opt-in sensitive-state permission policy that validates allowed modes, applies consistently across OpenClaw writers and fs-safe-backed stores, and updates audit, doctor, docs, and regression proof together.

## Reproduction Assessment

Yes, source-level. Current main still has deterministic chmod 0o700/0o600 paths and fs-safe private store calls that match the reported POSIX ACL-mask collapse, though I did not run a destructive live ACL repro in this read-only review.

## Solution Assessment

Unclear. Configurable 0640/0750-style behavior is directionally plausible, but the best solution is a unified opt-in sensitive-state permission policy rather than a narrow skip flag for one writer path.
