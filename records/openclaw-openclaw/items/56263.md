## Label Justifications

- `P1`: The issue describes a current regression that can make multi-user gateway and sidecar deployments unusable while touching sensitive local state permissions.
- `impact:security`: The requested change loosens or bypasses local file-mode hardening for OpenClaw-managed config, credentials, approvals, and state.
- `impact:auth-provider`: device-auth.json, credentials, provider config, and auth-adjacent state are part of the affected chmod surface.
- `impact:session-state`: The report specifically includes session transcripts and plugin/runtime state whose ACL access can be lost after writes.

## Summary

Keep open. Current main still repeatedly enforces private 0600/0700-style modes through config/state writers and the newer @openclaw/fs-safe storage paths, and the linked open PR only covers one state-directory hardening path rather than the broader ACL-mask breakage described in this issue.

## What This Changes

The issue requests an opt-in way to preserve group-readable/traversable permissions or skip chmod hardening for multi-user deployments that share OpenClaw state through POSIX groups or ACLs.

## Best Possible Solution

Define a maintainer-approved, opt-in permissions policy that keeps private defaults, validates allowed file and directory modes, preserves explicit ACL/group-sharing deployments, and updates writer helpers, fs-safe usage, audit, repair, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a group/ACL-readable state directory, trigger config or state writes, and current chmod calls in config IO, JSON helpers, security repair, and fs-safe-backed stores can restore 0700/0600-style private modes that collapse POSIX ACL masks.

## Solution Assessment

Unclear. An opt-in permission policy is the right direction, but a global skip flag is too narrow for the reported surface and too blunt for the security boundary unless audit, repair, validation, and docs are updated together.
