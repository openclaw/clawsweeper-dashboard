## Summary

Keep open for maintainer review: current main still has the SDK custom-tool descriptor crash path, and this focused PR adds a plausible hardening fix with regression proof, but it intentionally changes the exported `customTools` option from live caller-owned descriptors to process-stable snapshots.

## What This Changes

The branch snapshots SDK custom tool definitions at session ingress, drops unreadable descriptors, preserves callable receivers, and adds focused SDK regression coverage.

## Best Possible Solution

Land the ingress snapshot approach after maintainers explicitly accept process-stable SDK custom tool descriptor semantics and the draft/check gates are ready.

## Reproduction Assessment

Yes, at source level: current main maps `options.customTools` through `tool.name` and later reads descriptor metadata during registry refresh, so hostile getters can throw on the session setup path. I did not run tests because this review is read-only.

## Solution Assessment

Yes, if maintainers accept the SDK compatibility choice: snapshotting at SDK/session ingress prevents repeated unsafe descriptor reads and preserves callback receivers. The safer alternative for live-descriptor compatibility would be guarded reads on every refresh, but that keeps dynamic descriptors as part of the SDK contract.
