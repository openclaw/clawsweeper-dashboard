## Summary

Keep open for maintainer review: the patch appears source-reproducible and correctly scoped, but it is a maintainer-labeled draft that changes the exported SDK `customTools` option from live caller-owned descriptors to process-stable snapshots.

## What This Changes

The PR snapshots SDK custom tool definitions at session ingress, drops unreadable descriptors, preserves callable receivers, and adds SDK regression tests.

## Best Possible Solution

Land the ingress snapshot approach after maintainers explicitly accept process-stable SDK custom tool descriptors and the draft/required-check gates are ready.

## Reproduction Assessment

Yes, at source level: current main maps `options.customTools` through `tool.name` and later reads descriptor metadata in `refreshToolRegistry`, so hostile getters can throw on the session/tool-registry path. I did not run tests because this review is read-only.

## Solution Assessment

Yes, if maintainers accept the SDK compatibility choice: snapshotting at SDK/session ingress prevents repeated unsafe descriptor reads and preserves callback receivers. The safer alternative for live-descriptor compatibility would be targeted guarded reads on every refresh, but that keeps the dynamic descriptor contract alive.
