## What This Changes

This PR snapshots SDK custom tool definitions at session ingress, drops unreadable descriptors before registry reads, and preserves receivers for wrapped tool callbacks.

## Best Possible Solution

Land the narrow SDK snapshot path after maintainers explicitly accept process-stable customTools descriptor semantics, or intentionally switch to the broader session snapshot path once its proof and mergeability are resolved.

## Reproduction Assessment

Yes. Source inspection shows current main reads options.customTools through tool.name and later reads descriptor metadata during registry refresh, so hostile getters can throw on session setup without needing a broader harness run.

## Solution Assessment

Yes, if maintainers accept the SDK compatibility choice. Snapshotting at SDK/session ingress prevents repeated unsafe descriptor reads and preserves callback receivers; the safer alternative for live-descriptor compatibility is guarded reads on each refresh.
