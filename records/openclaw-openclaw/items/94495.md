## Summary

Keep open: the PR is a plausible timeout tweak, but the patch is not correct for the linked remote Ollama failure because it hard-codes a 5-minute fallback that overrides OpenClaw's existing inherited/global stream timeout path and does not address the reporter's configured-timeout case.

## What This Changes

The PR changes the native Ollama stream timeout resolver so a missing explicit request timeout becomes a fixed 300000 ms fallback before calling `fetchWithSsrFGuard`.

## Best Possible Solution

Preserve current timeout inheritance when no explicit provider timeout exists, then fix the configured remote-Ollama stream boundary with focused regression coverage and redacted live remote proof.

## Reproduction Assessment

No high-confidence live remote Ollama reproduction was run for the linked issue. The PR-introduced timeout regression is source-reproducible because the new fallback is passed as an explicit guarded-fetch timeout and bypasses the current global timeout fallback.

## Solution Assessment

No. This is a wrong-layer fallback for the reported case because the reporter already configured provider/run timeouts; the safer solution is to preserve inherited timeout behavior and fix the boundary where configured-timeout native Ollama streams stop progressing.
