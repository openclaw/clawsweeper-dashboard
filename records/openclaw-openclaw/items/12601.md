## Summary

Keep this open for maintainer/API review: current main still lacks Slack Canvas content reading, the prior implementation PR was closed unmerged as not merge-safe, and Slack’s current `canvases.sections.lookup` contract appears to return section identifiers rather than Canvas markdown content. ([docs.slack.dev](https://docs.slack.dev/reference/methods/canvases.sections.lookup/?utm_source=openai))

## What This Changes

Request support for reading Slack Canvas markdown content through the Slack message tool or a dedicated Canvas read action.

## Best Possible Solution

Keep the issue open for maintainer decision: either document/close the request if Slack confirms no supported content-read API, or add an opt-in Slack plugin action once a supported API can return bounded Canvas content with tests, docs, and live Slack proof.

## Reproduction Assessment

Yes for the source-level gap: current main has no `read-canvas` action, no Canvas dispatch/runtime path, and no `canvases:read` setup guidance. No live Slack workspace reproduction was run, and the remaining question is upstream API capability rather than whether OpenClaw currently exposes the feature.

## Solution Assessment

Unclear. A Slack plugin Canvas read action is the right OpenClaw boundary only if Slack exposes supported Canvas content reads; otherwise the maintainable solution is to document the upstream limitation or wait for a Slack API change instead of shipping a misleading lookup-only action.
