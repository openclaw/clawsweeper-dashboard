## Summary

Keep this open. Current main still has no Slack Canvas read surface: Slack action discovery, dispatch, runtime handling, file download behavior, setup scopes, docs, and contract tests cover Slack message/file/pin/reaction/member/emoji actions only. The related implementation PR #18752 was closed unmerged, and the maintainer comment already preserved this as bundled Slack plugin work rather than a resolved or ClawHub-only request.

## What This Changes

Add support for reading Slack Canvas markdown content through the bundled Slack message tooling or a dedicated Slack Canvas action/tool.

## Best Possible Solution

Implement this in the bundled Slack plugin, not core: add a bounded Canvas read action or dedicated Canvas tool, wire it through Slack action discovery/dispatch/runtime, add canvases:read setup and docs guidance, preserve existing token/account resolution, and cover the new behavior with focused Slack plugin tests plus live validation when available.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to request a Slack `read-canvas`-style message action on current main: discovery never advertises it, dispatch has no case for it, runtime has no Canvas handler, setup/docs omit `canvases:read`, and the existing file download path only handles private file URLs.

## Solution Assessment

Yes, the issue’s requested direction is the narrow maintainable solution if implemented inside `extensions/slack` with an additive message action or Slack-owned tool. The safer boundary is to keep core generic and avoid adding Slack-specific behavior outside the Slack plugin except for any required additive shared action-name contract.
