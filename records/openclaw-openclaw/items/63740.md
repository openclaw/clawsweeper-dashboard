## Summary

Keep open. Current main has packaging hardening for missing and stale `dist` paths, but it still records the packaged dist inventory as paths only, so a byte-modified expected `dist/cli/run-main.js` chunk would not be detected before the entrypoint dynamically imports it. The report remains security-sensitive because the root cause of the installed-code mutation is unproven and the current integrity checks do not rule it out.

## What This Changes

The issue asks OpenClaw to prevent or clearly repair a global install where an installed `dist/run-main-*.js` chunk was corrupted with a shell `tail` fragment and the CLI then crashed during ESM compilation.

## Best Possible Solution

Treat this as a package/runtime integrity investigation: audit update, postinstall, restart, logging, diagnostic, and plugin-runtime-dependency paths for writes under the installed package root, then add content-digest verification for packaged dist chunks with a clear startup or doctor repair path for corrupted expected files.

## Reproduction Assessment

Unclear. The issue has a high-signal corrupted installed-file snippet and startup stack, and a manually corrupted run-main chunk would still fail through the current dynamic import path, but there is not yet a high-confidence reproduction for OpenClaw itself writing the `tail` command into `dist`.

## Solution Assessment

No. The current path-only dist inventory is useful for missing/stale chunks, but it is not the best complete fix for this report because it cannot detect byte-modified expected chunks; hash-based verification plus a root-cause audit is the safer direction.
