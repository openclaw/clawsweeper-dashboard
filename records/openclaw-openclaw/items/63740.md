## Summary

Keep open. Current main and the latest shipped update hardening verify packaged dist membership by path and prune stale files, but they still do not byte-verify expected dist chunks before the CLI imports `run-main`, so the reported installed-code corruption remains a security-sensitive integrity investigation.

## What This Changes

The issue reports an installed `dist/run-main-*` chunk corrupted with a shell `tail` command, causing OpenClaw CLI startup to fail with `SyntaxError: Unexpected number`.

## Best Possible Solution

Audit package-root writes across runtime, update, postinstall, logging, diagnostics, and plugin flows, then add digest-backed packaged dist verification with a clear startup/update/doctor recovery path for modified expected files.

## Reproduction Assessment

Unclear for the original self-corruption root cause: the report has a concrete corrupted installed-file snippet, but I do not have a deterministic current-main path proving OpenClaw writes `tail` output into `dist`. Source inspection does show that a corrupted expected `run-main` chunk would still reach the dynamic import and fail before a content-integrity check.

## Solution Assessment

No. The current path-inventory hardening is useful for missing and stale files, but it is not the complete fix for byte-modified expected chunks; digest verification plus a root-cause write audit is the safer maintainable direction.
