## Summary

Keep open. Current main still detects packaged `dist` drift by path membership, but not byte changes inside an expected chunk, and the startup path still imports `run-main` without an installed-package integrity preflight; the reported installed-code mutation therefore remains a security-sensitive investigation rather than a completed fix.

## What This Changes

The issue reports an installed OpenClaw CLI chunk being modified with a `tail -n 200` shell fragment, causing startup to fail with `SyntaxError: Unexpected number`.

## Best Possible Solution

Add digest-backed package dist inventory verification before loading installed CLI chunks, then complete a focused audit of runtime, update, logging, diagnostic, and plugin flows for any write path that could target installed package files.

## Reproduction Assessment

Unclear for the original self-corruption root cause: the report has a concrete corrupted installed-file snippet, but I do not have a deterministic current-main path proving OpenClaw writes shell output into `dist`. Source inspection does show that a byte-modified expected `run-main` chunk is not caught by current path-only inventory checks before import.

## Solution Assessment

No. The current path-inventory hardening is useful for missing and stale files, but the narrow maintainable fix for this report is byte integrity verification plus a root-cause write audit and recovery policy.
