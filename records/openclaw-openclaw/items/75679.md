## Summary

Keep open. The PR is aimed at a valid linked bug, but the patch only canonicalizes the read/list/control side; the `sessions_spawn` writer path still records main-alias tool spawns under the raw `main` key, so the agent-facing list path can still miss a run after spawn. The diff also needs the required changelog entry for a user-facing fix.

## What This Changes

The PR canonicalizes main-session requester/controller keys for the subagents tool and `/subagents` command and adds focused regression tests for main-alias list routing.

## Best Possible Solution

Normalize main-session aliases consistently at both registry write and control/list lookup boundaries, while preserving compatibility for existing raw-alias entries, then cover the `sessions_spawn` main-alias spawn followed by list/control path and add the required changelog entry.

## Reproduction Assessment

Yes. A focused reproduction is to spawn through the `sessions_spawn` tool with `agentSessionKey: "main"`, then call `subagents(action="list")` from `agentSessionKey: "main"`; code inspection shows the writer stores `main` while the patched reader would look up `agent:main:main`.

## Solution Assessment

No. Canonicalizing only the read boundary is not the best fix because it leaves one current writer path and existing raw-alias entries behind; the safer fix is to canonicalize registry writes as well or make lookup deliberately handle both equivalent main keys without widening subagent scope.
