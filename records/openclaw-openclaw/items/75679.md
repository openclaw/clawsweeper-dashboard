## Summary

Keep open. The PR targets a valid subagent listing regression linked to #75593, but the proposed patch only canonicalizes the list/control read side; current registry write paths can still store the raw `main` alias, and exact-key lookup would then miss those runs after this patch. It also needs the required user-facing changelog entry before merge.

## What This Changes

The branch canonicalizes main-session requester/controller keys in subagent tool and slash-command list/control resolution and adds focused alias regression tests for those read paths.

## Best Possible Solution

Normalize requester/controller session keys symmetrically across subagent registry writes and reads, or make the list/control lookup include equivalent legacy raw-main aliases while new writes become canonical. Keep the focused tests, add reciprocal raw-main-to-canonical coverage, and add a single-line `CHANGELOG.md` fix entry with contributor credit.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to run `sessions_spawn` from a tool context with `agentSessionKey: "main"`, which stores a raw `main` requester/controller key, then list through the patched subagents read path, which canonicalizes the same logical caller to `agent:main:main` before exact-key filtering.

## Solution Assessment

No. Canonicalizing only the read boundary is not the best fix because the write paths still persist raw main aliases; the safer narrow solution is symmetric canonicalization or alias-compatible lookup plus reciprocal regression coverage.
