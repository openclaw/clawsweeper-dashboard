## Summary

Keep open. The PR targets the right subagent requester-key bug and the slash-command half is plausible, but the proposed list-side canonicalization is asymmetric with the existing sessions_spawn registry write path, so a run spawned under the raw logical main alias can still disappear from the agent-facing list/control tool after this patch. The user-facing fix also needs a changelog entry before merge.

## What This Changes

The PR canonicalizes main-session aliases for subagent list/control lookup in the agent tool and slash-command dispatch, with regression tests for alias-based listing.

## Best Possible Solution

Normalize requester/controller keys symmetrically across subagent registry writes and reads, or make the list/control lookup include equivalent legacy raw-main aliases while new writes become canonical. Keep the focused regression tests, add reciprocal coverage for runs spawned under the raw logical main alias, and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to spawn/register a subagent run through the agent-facing `sessions_spawn` path with `agentSessionKey: "main"`, then list through the patched `subagents` tool from the same logical main alias; the write path stores `main` while the patched read path looks for `agent:main:main`, so the exact-match filter returns no runs.

## Solution Assessment

No. Canonicalizing only the list/control boundary fixes one alias direction but breaks or leaves uncovered the reciprocal sessions_spawn tool path; the safer narrow fix is symmetric canonicalization or alias-compatible lookup plus reciprocal regression coverage.
