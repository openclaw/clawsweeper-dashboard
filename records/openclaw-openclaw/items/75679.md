## Summary

Keep open. The linked bug is valid and this PR targets the right lookup boundary, but the patch normalizes only read-side list/control keys while existing spawn registry writes can still store the raw logical main alias, leaving a reciprocal disappearance path. The user-facing fix also needs the required changelog entry before merge.

## What This Changes

The branch canonicalizes main-session aliases for subagent list/control requester keys and adds focused tool and slash-command regression tests.

## Best Possible Solution

Normalize requester/controller keys symmetrically across subagent registry writes and list/control reads, or make list/control lookup include equivalent legacy raw-main aliases while new writes become canonical. Keep the focused regression tests, add reciprocal coverage for runs spawned under raw `main`, and add a single-line `CHANGELOG.md` fix entry with contributor credit.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to spawn/register a subagent through a path using `agentSessionKey: "main"`, then list through the patched canonicalized main lookup; the write can store `main` while the read looks for `agent:main:main`, and exact-key filtering returns no runs.

## Solution Assessment

No. Canonicalizing only list/control lookups fixes one alias direction but leaves the reciprocal raw-main spawn/write path incompatible; the safer narrow solution is symmetric canonicalization or alias-compatible lookup with reciprocal tests.
