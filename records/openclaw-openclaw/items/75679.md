## Summary

Keep open. The linked bug is real and this PR moves in the right direction, but the patch only canonicalizes the read/list side while current spawn and registry writes can still persist raw `main` requester/controller keys that exact-key list/control lookups will miss. The branch also lacks the required changelog entry for a user-visible subagents fix.

## What This Changes

The PR canonicalizes main-session aliases for subagent list/control requester resolution and adds focused tool and slash-command regression tests.

## Best Possible Solution

Normalize requester/controller keys symmetrically across subagent registry writes and list/control reads, or make list/control lookup alias-compatible for equivalent legacy main-session keys while new writes use a canonical key. Keep the focused regression coverage, add reciprocal raw-main coverage, and add the changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to register or spawn a subagent from a tool context with `agentSessionKey: "main"`, then list through a path canonicalized to `agent:main:main`; current exact-key filtering leaves the raw-main run invisible.

## Solution Assessment

No. Canonicalizing only list/control requester keys is not the narrowest complete fix because it leaves raw-main registry writes incompatible; symmetric canonicalization or alias-set lookup is safer and backward-compatible.
