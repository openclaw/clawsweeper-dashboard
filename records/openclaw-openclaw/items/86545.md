The issue requests restoring isolated cron Discord announce delivery when the Discord bot token is SecretRef-backed from the gateway runtime environment.

## Best Possible Solution

Add a focused regression test and ensure isolated cron announce delivery uses the active secrets runtime snapshot for Discord account tokens without requiring users to move credentials into the service EnvironmentFile.

Yes, source-level: the exact error is thrown when Discord sees a configured_unavailable token, and isolated cron required delivery propagates that outbound failure as the run error. I did not run a live systemd Environment-only setup, so the environment-source distinction still needs direct proof.

## Solution Assessment

Yes, the maintainable fix is to restore active runtime snapshot use through the isolated cron direct-delivery path; documenting an EnvironmentFile workaround alone would conflict with the existing outbound SecretRef contract.
