## Summary

Keep open: this draft, maintainer-labeled stacked PR is still useful because current main still has the direct `updateSessionStore(...)` deleted-agent purge, and the PR moves that purge behind a locked storage-boundary helper. I found no blocking code defect, but the session-state risk, parent PR dependency, and real-behavior proof gap need maintainer handling before merge.

## What This Changes

The PR adds a deleted-agent session-entry purge operation to the session accessor/store boundary and routes `agents.delete` cleanup through it while extending focused tests and the boundary guard.

## Best Possible Solution

Land this only as part of the reviewed session lifecycle seam stack, with the locked current-state purge contract preserved and explicit proof that deleted-agent entries are removed while legacy/shared-store entries survive.

## Reproduction Assessment

Yes for the internal gap: current main still shows `purgeAgentSessionStoreEntries(...)` using a direct `updateSessionStore(...)` whole-store callback. This is a source-level reproduction of the migration-boundary issue, not a user-facing runtime repro.

## Solution Assessment

Yes, the locked current-state purge is the best fix shape I saw because it preserves the old delete-time owner scan while removing the caller-owned mutable whole-store callback. A pre-lock exact-entry removal plan would be weaker because updated entries could be skipped or judged from stale state.
