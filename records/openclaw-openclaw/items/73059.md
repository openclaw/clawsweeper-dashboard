## Summary

Keep open. Current main still uses a raw path-string direct-execution guard in `scripts/ui.js`, the gateway first-start Control UI auto-build still invokes that wrapper, and the latest release carries the same guard; the related Windows UI PR was adjacent and closed unmerged.

## What This Changes

The issue requests that `scripts/ui.js` run its build path when launched through a Windows global-link junction so first-start Control UI auto-build recreates `dist/control-ui/index.html`.

## Best Possible Solution

Canonicalize both `scripts/ui.js` entrypoint paths with realpath plus a resolved-path fallback, add focused regression coverage, and record the user-facing Windows build-wrapper fix in the changelog.

## Reproduction Assessment

Yes. The reporter supplied concrete Windows `pnpm link --global` junction steps, and current main still has the exact raw-path guard that would skip `main()` when the junction path differs from the real script path.

## Solution Assessment

Yes. Fixing direct execution in `scripts/ui.js` is the narrowest maintainable solution because both `pnpm ui:build` and gateway first-start auto-build depend on that wrapper; changing only the gateway caller would leave direct `node ...\scripts\ui.js build` broken.
