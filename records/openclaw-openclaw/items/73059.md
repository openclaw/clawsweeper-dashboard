## Summary

Keep open. Current main still has the path-string direct-execution guard in `scripts/ui.js`, the gateway first-start auto-build path still invokes that wrapper, and the latest release still contains the same guard; the related Windows UI PR was adjacent and closed unmerged.

## What This Changes

The issue asks OpenClaw to make junction-launched Windows source installs run the same Control UI build path as repo-local `node scripts/ui.js build`.

## Best Possible Solution

Normalize `scripts/ui.js` direct-execution detection through canonical realpath comparison with a resolved-path fallback, then cover linked or junctioned `argv1` in `test/scripts/ui.test.ts`.

## Reproduction Assessment

Yes. The reporter supplied concrete Windows `pnpm link --global` junction steps, and current main still compares `path.resolve(process.argv[1])` with `fileURLToPath(import.meta.url)`, which is the failing condition for junction-launched scripts.

## Solution Assessment

Yes. Fixing the direct-execution guard in `scripts/ui.js` is the narrowest maintainable solution; changing only the gateway caller would leave direct `node ...\scripts\ui.js build` broken.
