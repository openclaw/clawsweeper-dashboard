## Summary

Keep open. Current main still has the unresolved direct-execution guard in `scripts/ui.js`, so a junction-launched `node ...\scripts\ui.js build` can skip `main()` and leave the Control UI bundle missing; the adjacent Windows PR covered `pnpm.cmd` quoting/dependency install behavior and was closed unmerged, so it does not supersede this direct Node launch failure.

## What This Changes

The issue requests that `scripts/ui.js` execute the Control UI build path correctly when launched through a Windows global npm junction from a linked source checkout.

## Best Possible Solution

Normalize the `scripts/ui.js` direct-execution guard through realpath-on-both-sides comparison with a resolved-path fallback, then add a focused regression test that simulates a linked entry path. Keep the fix scoped to the wrapper and test coverage; a gateway-only workaround would leave direct script execution broken.

## Reproduction Assessment

Yes. The issue gives concrete Windows `pnpm link --global` junction steps, and current main still has the exact unresolved path comparison that can skip `main()` when the launch path and module realpath differ.

## Solution Assessment

Yes. A narrow guard normalization plus `test/scripts/ui.test.ts` regression coverage is the maintainable fix; changing Control UI app code or only masking the gateway symptom would be broader and less direct.
