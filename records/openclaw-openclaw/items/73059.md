## Summary

Keep open. Current main still has the path-string direct-execution guard in `scripts/ui.js`, the gateway first-start Control UI build path still invokes that wrapper, and the latest release tag still contains the same guard. The related Windows UI PR was closed unmerged and covered Windows `pnpm.cmd` quoting/dependency installation, not the direct junction-launched `node ...\scripts\ui.js build` failure mode described here.

## What This Changes

Fix Windows linked source installs so `scripts/ui.js` executes its build path when launched through an npm global junction and the Control UI auto-build recreates `dist/control-ui/index.html`.

## Best Possible Solution

Normalize the `scripts/ui.js` direct-execution check with canonical realpath comparison and a resolved-path fallback, then add focused regression coverage for a linked or junctioned entry path. Keep the fix in the wrapper path so both direct `node ...\scripts\ui.js build` and gateway auto-build are covered.

## Reproduction Assessment

Yes. The report gives concrete Windows `pnpm link --global` junction steps, and current source still compares the junction launch path directly against the real module path before deciding whether to call `main()`.

## Solution Assessment

Yes. A narrow wrapper guard fix plus `test/scripts/ui.test.ts` coverage is the maintainable solution; only changing the gateway caller would leave direct `node ...\scripts\ui.js build` broken.
