## Summary

Keep open. Current main still has the raw direct-execution guard that can miss a Windows junction-launched `scripts/ui.js`, and the gateway auto-build path still reports the missing Control UI bundle if that script exits without producing `dist/control-ui/index.html`. The related PR is adjacent Windows `pnpm.cmd` work, not the same direct `node ...\scripts\ui.js build` path.

## What This Changes

The issue requests that Windows source installs launched through the global npm junction run the same Control UI build path as repo-local `scripts/ui.js` execution.

## Best Possible Solution

Normalize the `scripts/ui.js` direct-execution check by comparing realpath-normalized entry and module paths, with a resolved-path fallback for missing paths, then add a focused regression test for a linked entry path and leave Control UI app code unchanged.

## Reproduction Assessment

Yes. The issue gives concrete Windows `pnpm link --global` junction steps, and current main still has the exact raw path comparison that can skip `main()` when the launch path and module path resolve differently.

## Solution Assessment

Yes. A narrow script guard fix plus `test/scripts/ui.test.ts` coverage is the maintainable solution; changing gateway error handling would only mask the broken direct CLI behavior.
