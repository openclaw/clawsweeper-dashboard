## Summary

Keep open. Current main still has the raw `scripts/ui.js` direct-execution guard described in the report, so a junction-launched `node ...\scripts\ui.js build` can skip `main()` instead of building Control UI assets; the gateway auto-build path still reports the missing-index symptom when the script exits without producing `dist/control-ui/index.html`. The related PR #52291 is adjacent Windows `pnpm.cmd` quoting/dependency-install work, closed unmerged, and does not cover direct `node` execution through the global npm junction.

## What This Changes

The issue requests that Windows linked source installs run the Control UI build path when `scripts/ui.js` is launched through the global npm junction, so first-start auto-build recreates `dist/control-ui/index.html`.

## Best Possible Solution

Normalize the `scripts/ui.js` direct-execution guard through `fs.realpathSync.native` or equivalent realpath-on-both-sides comparison with a resolved-path fallback, then add a focused regression test for a linked entry path. Keep the fix scoped to the wrapper and its test; avoid changing Control UI app code or package metadata unless the regression test proves another boundary is involved.

## Reproduction Assessment

Yes. The issue gives concrete Windows `pnpm link --global` junction steps, and current main still has the exact unresolved path comparison that would skip `main()` when the launch path and module realpath differ.

## Solution Assessment

Yes. A narrow guard normalization plus `test/scripts/ui.test.ts` regression coverage is the maintainable fix; a gateway-only workaround would leave direct `node ...\scripts\ui.js build` behavior broken.
