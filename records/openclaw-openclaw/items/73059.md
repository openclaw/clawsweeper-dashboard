The issue requests that Windows linked source installs run the Control UI build when `scripts/ui.js` is launched through the global npm junction.

## Best Possible Solution

Normalize `scripts/ui.js` direct-execution detection through canonical realpath comparison with a resolved-path fallback, then cover linked or junctioned `argv[1]` in `test/scripts/ui.test.ts`.

## Repair Work Prompt

Fix the Windows linked-source Control UI wrapper bug reported in https://github.com/openclaw/openclaw/issues/73059. Current `scripts/ui.js` only calls `main()` when `path.resolve(process.argv[1]) === fileURLToPath(import.meta.url)`, which can skip execution when `node %APPDATA%\npm\node_modules\openclaw\scripts\ui.js build` reaches the repo through a Windows junction. Keep the fix scoped to direct-execution detection in `scripts/ui.js`; prefer canonical realpath comparison with a resolved-path fallback, matching the pattern in `src/infra/is-main.ts`, and do not change gateway auto-build policy or UI build behavior. Add a focused regression test in `test/scripts/ui.test.ts` for a linked/symlinked or injected `argv[1]` path that resolves to the same script, plus any minimal helper export needed to test this without running the build. Add a user-facing `CHANGELOG.md` fix entry if repo policy requires it for this Windows behavior fix. Treat PR #52291 as adjacent context only; it addressed Windows `pnpm.cmd` quoting/dependency install behavior and was closed unmerged.
