## Summary

Keep open. Current `main` and the latest release still use the raw path-string direct-execution guard in `scripts/ui.js`, while the gateway first-start Control UI asset builder still depends on that wrapper; related PR #52291 was closed unmerged and covered a different Windows `pnpm.cmd` path issue.

## What This Changes

The issue requests that Windows junction-launched `scripts/ui.js build` execute the same Control UI build path as repo-local invocation.

## Best Possible Solution

Update `scripts/ui.js` to canonicalize the entrypoint comparison with a realpath fallback, add focused linked-entrypoint regression coverage, and preserve the existing Windows shell hardening.

## Reproduction Assessment

Yes, source-backed. The reporter supplied concrete Windows junction steps, and current `main` still has the exact raw path comparison that would skip `main()` when `argv[1]` and `import.meta.url` name the same file through different filesystem paths.

## Solution Assessment

Yes. Fixing `scripts/ui.js` is the narrowest maintainable solution because changing only the gateway caller would leave direct `node ...\scripts\ui.js build` invocations broken.

## Repair Work Prompt

Fix openclaw/openclaw#73059. In `scripts/ui.js`, make direct-execution detection robust when `process.argv[1]` reaches the script through a Windows junction, symlink, or global npm link. Canonicalize both `process.argv[1]` and `fileURLToPath(import.meta.url)` with `fs.realpathSync.native`, falling back to `path.resolve`, before comparing. Preserve import-as-module behavior and the existing Windows shell launcher safety/quoting behavior. Add focused regression coverage in `test/scripts/ui.test.ts`, preferably through an exported pure helper or a temporary linked-entrypoint fixture that proves a canonicalized linked path still executes the direct-entry path. Add a user-facing `CHANGELOG.md` fix entry. Do not change unrelated gateway asset lookup policy or broaden Windows shell argument handling beyond what this bug needs.
