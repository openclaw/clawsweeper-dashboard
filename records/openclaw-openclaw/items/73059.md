## Summary

Keep open. Current main still has the path-string direct-execution guard in `scripts/ui.js`, the gateway auto-build still calls that wrapper and can surface the reported missing-index symptom, and the latest release tag still contains the same guard. The related Windows UI PR was closed unmerged and addressed `pnpm.cmd` quoting/full dependency install behavior, not direct `node ...\scripts\ui.js` execution through a junction.

## What This Changes

Fix Windows linked source installs so a junction-launched `scripts/ui.js build` runs the Control UI build path and recreates `dist/control-ui/index.html`.

## Best Possible Solution

Normalize the `scripts/ui.js` direct-execution check through canonical realpath comparison with a resolved-path fallback, then add focused regression coverage that simulates a linked or junctioned entry path. Keep the behavior change scoped to the wrapper unless shared script guard coverage is added deliberately.

## Reproduction Assessment

Yes. The issue provides concrete Windows `pnpm link --global` junction steps, and current main still contains the direct path comparison that can skip `main()` when `process.argv[1]` is the junction path but `import.meta.url` resolves to the real module path.

## Solution Assessment

Yes. A narrow guard normalization plus `test/scripts/ui.test.ts` coverage is the maintainable fix; only working around the gateway auto-build caller would leave direct `node ...\scripts\ui.js build` broken.
