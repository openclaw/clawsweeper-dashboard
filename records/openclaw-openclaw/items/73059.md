## Summary

Keep open. Current main still has the direct-execution guard that compares the launched path string to `import.meta.url` without realpath normalization, so the Windows junction-launched source-install path described in the issue is still plausibly a no-op; the adjacent Windows UI build PR was closed unmerged and covered a different `pnpm.cmd` quoting/dependency-install failure mode.

## What This Changes

The issue requests that `scripts/ui.js` run its build path correctly when launched through a Windows global-link junction instead of silently skipping `main()`.

## Best Possible Solution

Normalize the `scripts/ui.js` direct-execution guard with a canonical realpath comparison and a resolved-path fallback, then add focused regression coverage for a linked or junctioned entry path while keeping the change scoped to the wrapper/test surface.

## Reproduction Assessment

Yes. The issue provides concrete Windows `pnpm link --global` junction steps, and current main still contains the exact path-string guard that can skip `main()` when the launch path and module realpath differ.

## Solution Assessment

Yes. A narrow guard normalization plus `test/scripts/ui.test.ts` regression coverage is the maintainable fix; changing only the gateway auto-build caller would leave direct `node ...\scripts\ui.js build` broken.
