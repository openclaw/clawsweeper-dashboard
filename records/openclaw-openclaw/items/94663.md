Full review comments:

- **[P1] Bound MCP base64 before decoding:** `src/agents/agent-bundle-mcp-materialize.ts:42-46`
  - body: MCP servers control `block.data`, but this path strips whitespace and decodes the whole string before the media byte cap in `saveMediaBuffer` can run. A large image/audio/blob result can allocate a second huge string plus a Buffer and exhaust the agent process before the intended limit throws; reject or estimate/canonicalize against `maxBytesForMime` before `Buffer.from`.
  - confidence: 0.88
- **[P2] Preserve MCP attachment order:** `src/agents/agent-bundle-mcp-materialize.ts:202-210`
  - body: `Promise.all` runs every content-block staging call in parallel while each call pushes into the shared `mediaAttachments` array. Since file saves can complete in any order, `details.media.attachments` can stop matching the original MCP `content[]` order, and downstream channel delivery sends the first URL with the caption and the rest sequentially; return indexed attachments and collect them in content order instead.
  - confidence: 0.82
