## Summary

Keep this open: current main and v2026.6.6 retry recognized short-window rate-limit failures, but a Bailian/Qwen-style HTTP 403 with `Model.AccessDenied` still follows the auth path unless a provider-owned classifier identifies it as a throttle. The broad provider retry-config request remains a product/config decision, and the related open PR https://github.com/openclaw/openclaw/pull/61443 explicitly keeps 403 client errors non-retryable.

## What This Changes

Request to add optional provider-level retry configuration for 403/429/503 rate-limit responses so the same model can wait and retry before model fallback is exhausted.

## Best Possible Solution

Recognize only a verified Qwen/Model Studio short-window throttling signature through the provider failover-classification seam, then reuse the existing bounded same-model retry path while leaving genuine HTTP 403 auth failures immediate.

## Reproduction Assessment

No high-confidence live reproduction is available against current main. Source inspection does show that a generic HTTP 403 stays on the auth path and therefore cannot reach the existing same-model rate-limit retry branch.

## Solution Assessment

No, the proposed broad status-code retry config is not currently the best-supported fix. A narrow provider-owned classifier for a verified Bailian throttle signature is safer and reuses the existing bounded retry policy.
