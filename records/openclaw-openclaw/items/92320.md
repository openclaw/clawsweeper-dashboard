## Summary

Keep this open for provider-owner and product review, but do not implement the proposed generic status-code retry configuration yet: current OpenClaw behavior matches Alibaba’s documented meaning of `403 Model.AccessDenied`, and the report lacks the complete live response needed to prove an undocumented throttling case.

## What This Changes

Add optional provider-level delay, retry-count, jitter, and status-code controls so selected transient failures retry the same model before fallback.

## Best Possible Solution

Keep documented access-denied responses fail-fast; capture a redacted live response and verify endpoint, plan, workspace, and model authorization, then add a narrow provider-owned structured throttle classification only if that evidence proves the documented contract is insufficient.

## Reproduction Assessment

No. The logs establish that OpenClaw receives and fails over on a 403, but they do not establish that Alibaba emitted a rate-limit response; the documented meaning of `Model.AccessDenied` is authorization failure, and the complete redacted response is missing.

## Solution Assessment

No. A generic configurable 403 retry would mask genuine permission failures and duplicate SDK retry ownership; a structured provider-specific classification is the narrower maintainable solution if live evidence supports it.
