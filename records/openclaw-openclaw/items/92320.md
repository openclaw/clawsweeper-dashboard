## Summary

Current main now retries the same model for bounded short-window rate limits before fallback, but the reported Bailian `403 Model.AccessDenied` still matches authorization handling rather than that retry lane. Keep this open for a provider-contract decision: a verified structured Bailian rate-limit signal should reuse the existing retry path, while raw configurable 403 retries would also delay genuine permission failures.

## What This Changes

Requests optional provider-level retry configuration for selected HTTP errors so a model can wait and retry before exhausting fallback models.

## Best Possible Solution

Define a narrow provider error classification for a documented Bailian short-window throttle signal and route it through the existing bounded same-model retry controller, without adding a user-configurable raw status-code list.

## Reproduction Assessment

Yes at source level: generic `403` and `access denied` signals enter the auth lane, while same-model retry only accepts selected short-window rate-limit signals. A live current-main Bailian response with full structured fields and headers has not been reproduced.

## Solution Assessment

No; arbitrary provider-configured HTTP status retries are broader than the reported semantic and would compete with existing SDK retry and failover policy. A structured Bailian classifier feeding the current bounded retry path is safer and smaller.
