## Summary

Keep this open: current main and v2026.6.6 retry classified short-window rate limits on the same model, but the reported Bailian-style HTTP 403 still follows the auth path unless a provider-owned signal reclassifies it; the requested provider retry config is broader than the safest fix and needs maintainer product judgment plus a redacted live payload.

## What This Changes

Add optional provider-level retry settings so configured provider statuses such as 403, 429, and 503 can retry the same model before fallback.

## Best Possible Solution

Recognize a verified provider-owned Bailian throttling signature through the existing error-classification seam, then reuse the bounded same-model short-window retry path without adding broad status-code retry configuration.

## Reproduction Assessment

No, not as a live current-main reproduction; the issue includes logs, and source inspection confirms that generic HTTP 403 auth classification bypasses the existing same-model rate-limit retry path.

## Solution Assessment

No, the proposed provider-level status-code retry config is broader than the safest available path; a verified provider-specific classification should feed the current bounded retry logic.
