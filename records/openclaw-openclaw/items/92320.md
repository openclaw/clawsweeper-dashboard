## Summary

Keep this open: current main and the latest release already retry short-window errors that are classified as rate limits, but the reported Bailian/DashScope HTTP 403 `Model.AccessDenied` path still classifies as auth unless a provider-owned signal reclassifies it. The broad provider-level retry config is still a product/config decision rather than an automatically safe fix.

## What This Changes

Add provider-level retry settings so configured HTTP 403/429/503 rate-limit responses wait and retry the same provider/model before fallback.

## Best Possible Solution

Add a narrow Qwen/Bailian provider-owned classifier for a verified short-window throttle signature, then reuse the existing bounded same-model retry path without adding generic status-code retry config.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction against current main. Source inspection does show that generic HTTP 403 access-denied responses classify as auth and therefore cannot reach the existing same-model rate-limit retry branch.

## Solution Assessment

No, the proposed general provider-level status-code retry config is broader than the best current fix. A provider-owned classifier for a verified Bailian throttle signature is narrower, preserves real 403 auth failures, and reuses the existing retry policy.
