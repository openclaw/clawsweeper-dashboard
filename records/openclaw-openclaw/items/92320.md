## Summary

Keep this open: current main now retries recognized short-window rate limits on the same model, but Bailian-style HTTP 403 responses still follow the auth path unless a trustworthy provider-specific signal reclassifies them. The reported gap therefore remains, although the proposed general retry configuration is broader than necessary and needs maintainer judgment plus a redacted live response payload.

## What This Changes

Requests optional provider-level retry controls for selected rate-limit status codes before model fallback proceeds.

## Best Possible Solution

Recognize only a verified Bailian short-window throttling signature through the provider error-classification seam, then reuse the existing bounded same-model Retry-After path while preserving genuine HTTP 403 authorization failures.

## Reproduction Assessment

No, there is not yet a high-confidence live reproduction against current main. Source inspection does confirm that a generic HTTP 403 follows the auth path and therefore bypasses the existing same-model rate-limit retry branch.

## Solution Assessment

No, the proposed general provider status-code retry configuration is not currently the best-supported solution. A narrow provider-owned classification for a verified throttling signature would reuse existing retry policy and avoid making unrelated HTTP 403 failures retryable.
