## Summary

Keep open: current main can retry recognized short-window rate-limit failures on the same model, but the reported HTTP 403 path still depends on a trusted provider-specific rate-limit classification. The broad provider-level retry config request is useful context, but it needs maintainer product judgment and a redacted current Bailian response payload before implementation.

## What This Changes

The issue requests optional provider-level retry settings for 403/429/503 rate-limit-style failures so OpenClaw waits and retries the same provider/model before exhausting fallbacks.

## Best Possible Solution

Verify the Bailian throttle payload, then classify only that provider-specific short-window signature as rate_limit through the provider hook so the existing bounded same-model retry path handles it without broad 403 retries.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection does confirm the core path: a generic HTTP 403 classifies as auth and bypasses same-model rate-limit retry unless a provider hook returns rate_limit.

## Solution Assessment

No, the proposed broad provider-level status-code retry config is not yet the best-supported fix. A narrow verified provider classifier should be considered first, with the broader retry config decision kept with maintainers.
