## Summary

Keep this open: current main and v2026.6.6 already have bounded same-model retries for recognized short-window rate limits, but the reported Bailian-style HTTP 403 shape still routes through auth unless a provider-owned classifier can safely reclassify it.

## What This Changes

The issue requests configurable provider-level retries for 403/429 rate-limit responses so same-provider fallback chains wait and retry instead of exhausting immediately.

## Best Possible Solution

Recognize only a verified Bailian/Qwen short-window throttling signature through the provider-owned failover classifier, then feed that into the existing bounded same-model retry path while preserving genuine HTTP 403 auth failures.

## Reproduction Assessment

Partly: the provided 403 log shape is source-reproducible because current code maps generic HTTP 403 to auth unless a provider hook returns rate_limit, so it bypasses same-model retry. A live current Bailian response payload is still missing.

## Solution Assessment

No: provider-level status-code retry configuration is broader than the safest fix. A narrow provider-owned classifier for a verified throttle signature would reuse the shipped retry policy without making unrelated 403 failures retryable.
