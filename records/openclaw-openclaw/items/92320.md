## Summary

This should stay open: current main already has bounded same-model retry for recognized short-window rate-limit failures, but the reported Bailian-style HTTP 403 path still classifies as auth unless a provider-owned classifier identifies it as throttling. The requested generic provider retry configuration is broader than the safest fix and needs maintainer/provider judgment plus a redacted live payload before implementation.

## What This Changes

Add provider-level retry settings so configured 403/429/503 rate-limit responses wait and retry the same model before fallback.

## Best Possible Solution

Verify a redacted Bailian/DashScope throttle payload and add a narrow provider-owned failover classification for that exact short-window signal, reusing the existing bounded same-model retry path.

## Reproduction Assessment

No high-confidence live reproduction is available against current main. Source inspection does confirm that a generic HTTP 403 remains on the auth path and bypasses the current same-model rate-limit retry unless a provider hook reclassifies it.

## Solution Assessment

No, the requested general provider status-code retry configuration is broader than the best fix. A verified provider-owned throttle classifier is the narrower maintainable path because it preserves hard-auth 403 behavior while reusing existing bounded retry logic.
