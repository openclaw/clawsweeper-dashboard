## Summary

Keep open: current main and v2026.6.6 already retry recognized short-window `rate_limit` failures on the same model, but the reported Bailian/DashScope-style HTTP 403 path still classifies as auth unless a provider-owned structured signal reclassifies it. The central 403 throttling gap remains, while the proposed broad provider retry config needs maintainer product judgment.

## What This Changes

Request to add provider-level retry controls so selected HTTP 403/429/503 rate-limit responses wait and retry the same provider/model before fallback.

## Best Possible Solution

Verify a narrow Bailian/Qwen/DashScope short-window throttling signature, classify it through the provider-owned failover hook, and reuse the existing bounded same-model retry path while preserving real 403 authorization failures.

## Reproduction Assessment

No, not as a live Bailian reproduction against current main. Source inspection does show the reported shape is plausible because a generic HTTP 403 `Model access denied` response follows the auth path and bypasses same-model rate-limit retry.

## Solution Assessment

No, the requested broad provider status-code retry config is not currently the best-supported fix. The safer path is a provider-owned classifier for a verified throttling signature that reuses the existing retry policy without making unrelated 403 failures retryable.
