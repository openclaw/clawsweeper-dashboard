## Summary

Current main solves the ordinary short-window 429 case, but it still classifies the reported unqualified Bailian 403 as authorization failure. Keep this issue open for a provider-contract and product-boundary decision: the evidence does not justify retrying arbitrary 403 responses, and the proposed generic retry configuration would overlap existing SDK, profile-rotation, and same-model retry policies.

## What This Changes

Add optional provider-level status-code, delay, and retry controls so transient 403/429 responses retry the same model before fallback.

## Best Possible Solution

Obtain a redacted full Bailian response or official error contract, then recognize only a stable transient discriminator at a provider-owned normalization/classification boundary and feed it into the existing bounded same-model retry loop.

## Reproduction Assessment

No, not at high confidence. Current source clearly sends an unqualified 403 down the authorization/fallback path, but the available log does not establish a reliable way to distinguish Bailian rate limiting from genuine access denial on current main.

## Solution Assessment

No. Arbitrary provider-level status-code retry configuration is broader and less safe than the existing provider-classification seam plus bounded same-model retry behavior; maintainers first need a verified transient signal and an ownership decision for generic custom providers.
