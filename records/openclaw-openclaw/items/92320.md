## Summary

The evidence supports keeping this open: current main now retries recognized short-window rate limits on the same model, but the reported Bailian `403 Model.AccessDenied` was classified as auth and is not proven to enter that path. The requested raw status-code configuration is broader than the safest fix because 403 can also mean a permanent model-access or credential denial.

## What This Changes

Add optional provider-level retry status codes, retry counts, delays, jitter, and Retry-After handling before advancing through model fallbacks.

## Best Possible Solution

Recognize a stable, provider-documented Bailian throttling discriminator and route it through the existing bounded, Retry-After-aware same-model retry path; add a new config surface only if that canonical behavior cannot safely cover the use case.

## Reproduction Assessment

No, not at high confidence. Source confirms that an auth-classified response bypasses current rate-limit retry, but reproducing the claimed bug requires a redacted complete Bailian error payload or live response that proves the 403 is throttling rather than access denial.

## Solution Assessment

No, the proposed arbitrary status-code retry block is not yet the best solution. The narrow maintainable path is to extend the existing error taxonomy and same-model retry mechanism using a documented provider signal, while preserving immediate failure for genuine 403 authorization errors.
