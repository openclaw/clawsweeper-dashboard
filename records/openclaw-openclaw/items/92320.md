## Summary

Current main confirms the reported design gap: generic HTTP 403 is classified as authentication and immediately enters profile/model failover, while no provider-level delayed retry policy exists. The issue should remain open for a maintainer product decision because the proposed arbitrary status-code configuration is broader and less safe than the newly available provider-owned error-classification seam.

## What This Changes

Add optional provider-level delayed retries for configured rate-limit status codes before model fallback.

## Best Possible Solution

Recognize DashScope’s stable rate-limit signal through the provider-owned classifier, then add a bounded shared model-request retry policy only if a live current-main trace proves the provider SDK cannot retry it; keep hard-auth 403 behavior unchanged and make retries obey the existing turn deadline and cancellation signal.

## Reproduction Assessment

Partly yes: current source proves that generic HTTP 403 becomes an auth failover reason, and the supplied production logs show the claimed chain exhaustion. A live Bailian/DashScope rate-limit response on current main is still needed to verify its stable code, headers, and timing.

## Solution Assessment

No, the proposed arbitrary status-code retry configuration is not yet the best solution. The narrower maintainable path is provider-owned classification of the specific DashScope rate-limit signal, followed by a separately approved bounded retry contract if SDK retry remains insufficient.
