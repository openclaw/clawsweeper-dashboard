## Summary

Current main still sends generic 403 responses down the authentication/fallback path and has no provider-configured delayed same-model retry, so the reported workflow gap remains. The issue should stay open for a provider-contract and API decision, but the proposed arbitrary status-code configuration is broader than necessary because current main already exposes provider-owned failover classification and bounded SDK retry controls.

## What This Changes

Requests optional provider-level retry policy for configured transient HTTP statuses, including provider-specific 403 rate limits, before model fallback.

## Best Possible Solution

Define a typed provider-owned transient-error signal for a documented Bailian throttling response, then perform a tightly bounded same-model retry using Retry-After or capped backoff while leaving generic 403 responses as authentication failures.

## Reproduction Assessment

Yes at source level: current main excludes 403 from SDK retry and normally classifies it as authentication before fallback. A high-confidence live reproduction of Bailian throttling specifically still requires a redacted structured response or official contract showing that this Model.AccessDenied variant is rate limiting.

## Solution Assessment

No, the proposed arbitrary configured status-code list is not the best solution because it can retry real authorization failures. A provider-owned, typed throttling classification plus bounded retry is narrower and consistent with the current architecture.
