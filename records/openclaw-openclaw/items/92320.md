## Label Justifications

## Summary

Current main now retries recognized short-window rate limits before fallback, but the reported Bailian `403 Model.AccessDenied` remains classified as authorization failure. The issue should stay open until a live response or official provider contract establishes a reliable transient signal; retrying raw 403 responses would also delay genuine permission failures.

## What This Changes

Add optional provider-level delay and retry controls for configured rate-limit status codes before model fallback.

## Best Possible Solution

Teach the canonical error classifier to recognize a documented, structured Bailian transient-throttling signal and reuse the existing bounded same-model retry path, while leaving ambiguous or genuine authorization 403 responses on immediate fallback or failure handling.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection confirms that an `auth`-classified 403 bypasses the same-model rate-limit retry lane, but the supplied evidence does not establish that the observed Bailian response is truly transient throttling.

## Solution Assessment

No, the proposed generic status-code configuration is not the best solution because it cannot safely distinguish rate limiting from genuine access denial. A narrow provider-contract-backed classification feeding the existing retry mechanism is safer and avoids a parallel retry policy.
