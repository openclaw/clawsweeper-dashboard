## Summary

Keep this open: current main and v2026.6.6 have bounded same-model retries for errors already classified as short-window rate limits, but the reported Bailian-style HTTP 403 still falls through the auth path without a verified provider-specific throttle signal. The requested broad provider retry config is not the right automatic fix yet because it could retry real permission failures.

## What This Changes

The issue requests optional provider-level retry settings so configured 403/429/503 rate-limit responses retry the same provider/model before exhausting fallback models.

## Best Possible Solution

Verify a redacted current Bailian response payload, then add a narrow provider-owned classifier for a trustworthy short-window throttle signature that reuses the existing bounded same-model retry path while preserving real 403 access-denied failures as auth.

## Reproduction Assessment

No high-confidence live reproduction is available from current main. Source inspection does show that a generic HTTP 403 remains on the auth path and therefore bypasses same-model rate-limit retry unless provider-specific classification reclassifies it.

## Solution Assessment

No, the proposed provider-level status-code retry config is not the best-supported solution yet. The safer path is a verified provider-owned throttle classifier that feeds the existing bounded retry owner without making all configured 403s retryable.
