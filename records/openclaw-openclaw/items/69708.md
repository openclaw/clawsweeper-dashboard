## Summary

Keep open. Current main still rejects persisted auth-profile entries whose credential `type` is `aws-sdk`, so the reported `invalid_type` drop remains present even though provider-level Bedrock `auth: "aws-sdk"` is implemented. The linked targeted PR was closed unmerged, and current tests do not cover SDK-managed auth-profile markers.

## What This Changes

The issue asks OpenClaw to accept or otherwise handle Bedrock auth-profile entries using `type: "aws-sdk"` so EC2 IMDS/default-chain authentication keeps working after startup.

## Best Possible Solution

Define the SDK-managed auth-profile marker contract in the auth-profile store: `aws-sdk` should no longer be treated as a genuine invalid credential for Bedrock, provider-level AWS SDK/default-chain runtime auth should remain unchanged, and regression tests should cover a canonical `amazon-bedrock:default` marker alongside real invalid entries.

## Reproduction Assessment

Yes. A stored profile with `type: "aws-sdk"` reaches `parseCredentialEntry`, misses the current three-value credential allow-list, is rejected with `invalid_type`, and is then dropped during `auth-profiles.json` load.

## Solution Assessment

Unclear for the exact fix shape. PR #69753's skip-marker approach is narrow and plausible, but the secops-owned durable-store contract should be confirmed before deciding whether to skip, migrate, or type the marker.
