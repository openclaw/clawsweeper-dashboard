## Summary

Keep open. Current main still hides `apply_patch` when `write` is denied, current tests assert that behavior, and open PR #76795 already references this issue with closing syntax, so this should stay open pending maintainer review of that policy change.

## What This Changes

The issue requests allowing `tools.deny: ["write"]` to hide the raw `write` tool while keeping `apply_patch` visible when it is otherwise enabled.

## Best Possible Solution

Review PR #76795 and decide the intended write-deny semantics; if decoupling is accepted, land the matcher, regression-test, and docs updates together, otherwise document that `write` deny blocks all file-mutation tools.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `isToolAllowedByPolicyName("apply_patch", { deny: ["write"] })` returns false by construction, and current tests assert that result.

## Solution Assessment

Unclear. The proposed direction in PR #76795 is narrow, but changing `write` deny semantics for a file-mutating tool needs an explicit maintainer/security decision before it is treated as the best fix.
