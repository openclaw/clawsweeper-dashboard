## Summary

Keep open. The report matches current main, but the current matcher and tests intentionally couple `write` policy to `apply_patch`, so changing it needs maintainer product/security review rather than conservative cleanup.

## What This Changes

The issue asks for `tools.deny: ["write"]` to hide the raw `write` tool while keeping `apply_patch` visible when `apply_patch` is otherwise enabled.

## Best Possible Solution

Maintainers should decide the intended policy meaning first; if decoupling is accepted, update the matcher, regression tests, and docs while preserving explicit `apply_patch` and `group:fs` deny behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `isToolAllowedByPolicyName("apply_patch", { deny: ["write"] })` returns false by construction, and current tests assert the same behavior.

## Solution Assessment

Unclear. Treating this as a straight bug fix is not the best path until maintainers decide whether `write` deny should mean only the raw tool or all file-writing capabilities.
