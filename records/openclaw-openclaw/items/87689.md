## Summary

Keep open: current main and v2026.6.6 still run memory-core Dreaming session transcript ingestion without a supported per-ingestion disable guard. The newer policy knob covers QMD/session-memory indexing checks, not Dreaming's own session-corpus ingestion path, so the remaining work is a maintainer product/config decision.

## What This Changes

The issue requests a supported config setting or durable sentinel that disables Dreaming/QMD session transcript ingestion during staged memory migrations.

## Best Possible Solution

Add one canonical memory-core Dreaming session-ingestion disable control with schema, docs, status or doctor coverage, preserving current defaults and aligning it with the broader session/cron exclusion policy.

## Reproduction Assessment

Yes, at source level: current main calls Dreaming session transcript ingestion in light/REM flows, writes session-corpus files, and existing tests assert those transcript snippets become promotion candidates. I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a final implementation: the operational need is valid, but maintainers should choose a canonical config, policy, or sentinel contract instead of preserving a local installed-runtime workaround.
