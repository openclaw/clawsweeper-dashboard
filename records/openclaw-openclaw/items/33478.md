## Summary

Keep open: current main has partial thread-routing hardening, but the requested structured sessions_send callback action and explicit announce routing/thread override surface are still absent, and maintainer discussion says this should remain a concrete handoff reliability requirement.

## What This Changes

Request structured sessions_send handoff actions and inherited or overridable channel/thread context so A2A reply/announce flows no longer rely on exact magic-token model output.

## Best Possible Solution

Define one narrow structured handoff outcome contract for sessions_send that preserves legacy exact-token behavior, reuses current deliveryContext/threadId routing, and coordinates with the broader parent-announce routing design instead of adding competing ad hoc modes.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 both lack the requested structured sessions_send parameters, and the A2A prompts/classifiers still depend on exact REPLY_SKIP/ANNOUNCE_SKIP text. I did not run a live Telegram handoff repro in this read-only review.

## Solution Assessment

Unclear: a structured outcome is likely the maintainable direction, but the exact public API should be decided with the related parent-routing and delivery-intent issues so OpenClaw gets one coherent handoff contract.
