## Summary

Keep this PR open: it adds the right ciao assertion classifier and direct suppression, but the advertise-failure path still leaves an unannounced service for the existing watchdog to recreate after the stuck-state timer, so it does not fully prevent the Bonjour restart loop described in the PR update.

## What This Changes

The PR adds a Bonjour ciao `no-valid-addresses` classifier, skips immediate recovery for that kind, adds classifier and advertiser regression tests, and records a changelog fix for IPv6-only interfaces.

## Best Possible Solution

Treat `no-valid-addresses` as a terminal mDNS-disable condition across unhandled rejection, advertise-failure, and watchdog paths, with timer-advanced regression coverage.

## Reproduction Assessment

Yes, from source inspection. Current main does not classify the ciao assertion, `@homebridge/ciao@1.3.7` throws the exact message, and the PR branch’s own unannounced advertise-failure setup would still reproduce a watchdog recreation after the timers advance.

## Solution Assessment

No. The classifier is the right narrow direction, but the proposed no-recovery branch is incomplete because the watchdog can still recreate the advertiser for the same stable `no-valid-addresses` condition.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stop watchdog restarts after no-valid-addresses failures:** `extensions/bonjour/src/advertiser.ts:547-551`
  - body: Skipping `requestCiaoRecovery` here only avoids the immediate restart. In the advertise-failure path the service remains `unannounced`, and the watchdog marks non-announced services on its 5s interval before calling `recreateAdvertiser` once `STUCK_ANNOUNCING_MS` elapses, so IPv6-only hosts can still churn the advertiser despite this guard. Mark this classification as a terminal disabled state, or otherwise exempt the cycle from watchdog recreation, and cover it by advancing the timers in the regression test.
  - confidence: 0.88
