## Summary

Keep open. Current main still supports only a run-level `heartbeat.model`: structured `HEARTBEAT.md` tasks have no typed or parsed model metadata, all due tasks are folded into one heartbeat prompt, and the reply path receives one model override for the whole run. The related reports only cover bugs where that run-level override was ignored, not per-task model selection.

## What This Changes

Request to let individual `HEARTBEAT.md` task blocks specify a model or primary-model escalation instead of using one `heartbeat.model` for the entire heartbeat run.

## Best Possible Solution

Define a documented optional task-level model or escalation field for structured `HEARTBEAT.md` tasks, then implement the chosen mixed-model execution contract with config/docs updates and regression tests for default, overridden, skipped, failed, and delivered heartbeat task runs.

## Reproduction Assessment

Yes for the capability gap: adding `model:` under a `tasks:` entry has no current parser or runner path, and due tasks are executed through one heartbeat reply call. This was verified by static inspection of the parser, runner, docs, and existing task/model override tests.

## Solution Assessment

Unclear until maintainers choose the contract. The narrow maintainable path is a first-class task metadata field plus explicit batching/failure semantics, not an ad hoc prompt keyword that the parser and timestamp code cannot reason about.
