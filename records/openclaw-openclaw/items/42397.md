## Summary

Keep open: current main and the latest release still only support a run-level `heartbeat.model`, while structured `HEARTBEAT.md` tasks have only `name`, `interval`, and `prompt` and are batched into one heartbeat turn. This remains a feature/API request that needs maintainer direction before implementation.

## What This Changes

The issue requests per-task model hints in `HEARTBEAT.md` so some heartbeat tasks can run on a different model than the run-level heartbeat model.

## Best Possible Solution

Define a documented `HEARTBEAT.md` task metadata contract, validate model references through existing model alias/provider resolution, and implement model-aware heartbeat execution with docs and regression coverage after batching versus split-turn semantics are approved.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence current-main proof that task metadata is limited to name/interval/prompt and heartbeat execution uses one model selection per turn.

## Solution Assessment

Unclear until maintainers choose the contract. A typed `model` field validated through normal model alias/provider rules is safer than a magic `ESCALATE_TO_PRIMARY` rerun path, but batching versus split-turn behavior needs product/API review.
