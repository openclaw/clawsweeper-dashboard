## Summary

Keep open: current main still parses `HEARTBEAT.md` tasks as name, interval, and prompt, batches due tasks into one heartbeat turn, and only supports one `heartbeat.model` per run. The related closed router issue covers per-turn plugin routing, but not this heartbeat-specific per-task execution model.

## What This Changes

The issue requests per-task model hints or escalation syntax in `HEARTBEAT.md` so selected periodic heartbeat tasks can run on a stronger model without changing the whole heartbeat run.

## Best Possible Solution

Define one canonical heartbeat task-routing contract, then either split model-tagged task groups safely or document a plugin/per-turn routing path if maintainers decide per-run escalation is the supported boundary.

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives a high-confidence current-main check that task-level model metadata is absent today.

## Solution Assessment

Unclear: the requested syntax is plausible, but the best maintainable fix depends on whether maintainers want core `HEARTBEAT.md` task metadata with split runs or a plugin-first per-turn routing pattern.
