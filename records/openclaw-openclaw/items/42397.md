## Summary

Keep open: current main and v2026.6.8 still support only a run-level heartbeat model, parse HEARTBEAT.md tasks as name/interval/prompt, and batch all due tasks into one heartbeat turn. The existing per-turn plugin routing seam is adjacent, but it cannot satisfy a mixed-model per-task heartbeat split by itself.

## What This Changes

Requests HEARTBEAT.md task-level model or escalation hints so selected heartbeat tasks can run on a different model than the run-level heartbeat.model.

## Best Possible Solution

Define one canonical heartbeat task-routing contract; if core supports it, split due tasks by resolved model with clear fallback and session semantics, otherwise document that only per-turn plugin routing is supported.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high-confidence proof that task-level model metadata is absent today.

## Solution Assessment

Unclear: the HEARTBEAT.md hint syntax is plausible, but a parser-only field would not work while due tasks are batched into one run. Maintainers need to choose core split-run semantics or explicitly leave this to per-turn plugin routing.
