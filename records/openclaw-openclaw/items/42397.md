## Summary

Keep open: current main and v2026.6.10 still only parse `name`/`interval`/`prompt`, batch all due `HEARTBEAT.md` tasks into one heartbeat turn, and apply `heartbeat.model` once to the whole run. The closed general router/plugin item is only a partial alternative because it can choose one model for a turn, not run selected heartbeat tasks on a different model while preserving the shared heartbeat session.

## What This Changes

The issue requests a way to assign a different model to individual `HEARTBEAT.md` heartbeat tasks instead of using one run-level heartbeat model.

## Best Possible Solution

Define one maintainer-approved heartbeat task-routing contract: either split or group due tasks by resolved model while preserving shared session and task-state semantics, or explicitly fold this into a broader native heartbeat-entry/model-routing design with docs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high-confidence proof that task-level model metadata and split execution are absent today.

## Solution Assessment

Unclear: the proposed `HEARTBEAT.md` hint syntax is plausible, but current main batches due tasks into one turn, so maintainers need to choose split-run semantics or a different canonical heartbeat/model-routing contract.
