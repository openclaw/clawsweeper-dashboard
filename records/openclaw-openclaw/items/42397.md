## Summary

Keep open: current main and the latest release still support only a whole-heartbeat-run model, while HEARTBEAT.md tasks carry no model metadata and due tasks are batched into a single heartbeat reply turn. The related merged work fixed run-level heartbeat model routing and added cost controls, but it does not implement the requested per-task override, so this remains a feature/API product decision.

## What This Changes

The issue requests per-task model hints in HEARTBEAT.md so selected heartbeat tasks can use a different model from the run-level heartbeat model.

## Best Possible Solution

Define a documented task-level model field in HEARTBEAT.md that resolves through the existing alias/provider model selection path, then implement grouped or split heartbeat execution after maintainers decide batching and timestamp semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection gives high confidence that current main lacks task-level model metadata and uses one model selection for the heartbeat turn.

## Solution Assessment

Unclear until maintainers choose the contract. A typed task-level model field resolved through existing alias/provider rules is safer than a magic rerun keyword, but batching versus split-turn behavior needs product/API review.
