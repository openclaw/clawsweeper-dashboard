## Summary

Keep open: current main and the latest release still support only a run-level heartbeat model override, while HEARTBEAT.md tasks have no task-level model metadata and are executed in one batched heartbeat turn. This remains a feature request that needs maintainer product/API direction before implementation.

## What This Changes

Request to let individual HEARTBEAT.md task blocks choose a different model than the run-level heartbeat.model.

## Best Possible Solution

Define a documented HEARTBEAT.md task metadata contract, validate model references through existing model alias/provider resolution, and implement model-aware heartbeat execution with docs and regression coverage once batching versus split-turn semantics are approved.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main shows HEARTBEAT.md task metadata is parsed only as name/interval/prompt and heartbeat execution still uses one model selection per turn.

## Solution Assessment

Unclear until maintainers choose the contract. A typed model field validated through normal model alias/provider rules is safer than a magic ESCALATE_TO_PRIMARY rerun path, but the batching versus split-turn behavior needs product/API review.
