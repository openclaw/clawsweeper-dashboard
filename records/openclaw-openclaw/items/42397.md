Keep open: current main and the latest release still support only run-level heartbeat model selection, while due `HEARTBEAT.md` tasks are batched into one heartbeat turn. The request remains a feature/product-direction item, not a conservative cleanup close.

## What This Changes

Request support for assigning a different model, or primary-model escalation hint, to individual heartbeat tasks defined in `HEARTBEAT.md`.

## Best Possible Solution

Define one canonical heartbeat task-routing contract that preserves existing run-level `heartbeat.model` behavior and only adds per-task model selection if maintainers accept the product/API surface.

Yes for source reproduction: current main parses heartbeat tasks without any task-level model field and runs due tasks through one heartbeat model override for the whole turn. This is not a bug reproduction because the requested behavior is a new capability.
