## Summary

Keep open: current main and v2026.5.20 still expose only run-level `heartbeat.model`, while `HEARTBEAT.md` task parsing and batching have no task-level model metadata. This remains a heartbeat product/API feature, not conservative cleanup.

## What This Changes

The issue requests per-task model selection metadata in `HEARTBEAT.md` so selected heartbeat tasks can run on a different model from the run-level heartbeat model.

## Best Possible Solution

Define a documented task-level model field in `HEARTBEAT.md` that resolves through existing alias/provider rules, then implement grouped or split heartbeat execution only after maintainers decide batching, timestamp, and delivery semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, and v2026.5.20 inspection give high confidence that current OpenClaw lacks task-level model metadata and uses one model selection for each heartbeat reply turn.

## Solution Assessment

Unclear until maintainers choose the contract. A typed task-level model field resolved through existing alias/provider rules is safer than a magic rerun keyword, but batching versus split-turn behavior needs product/API review.
