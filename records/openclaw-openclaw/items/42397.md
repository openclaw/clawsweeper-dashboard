## Summary

Keep open: current main and v2026.6.10 still support only a run-level heartbeat model, while HEARTBEAT.md due tasks are parsed as name/interval/prompt and batched into one heartbeat turn. The existing per-turn plugin hook and cron model overrides are useful adjacent paths, but they do not fully provide mixed-model HEARTBEAT.md task semantics without a maintainer product/API decision.

## What This Changes

The issue requests per-task model hints in HEARTBEAT.md so selected periodic heartbeat tasks can run on a stronger model without moving every heartbeat turn to that model.

## Best Possible Solution

Define one supported scheduled-work routing contract: if HEARTBEAT.md owns this, parse task model hints and group due tasks by resolved model with explicit session and fallback semantics; otherwise document cron/plugin routing as the supported alternative and close heartbeat-specific syntax as out of scope.

## Reproduction Assessment

Yes at source level for the missing capability: current HEARTBEAT.md tasks parse only name, interval, and prompt, and the runner sends all due tasks through one heartbeat model override.

## Solution Assessment

Unclear: per-task HEARTBEAT.md hints are plausible, but the best fix needs maintainer choice between heartbeat split-run semantics and the existing cron/plugin routing surfaces.
