## Summary

Keep open: current main and the latest release still support only a whole-heartbeat-run model, while HEARTBEAT.md tasks carry no model metadata and due tasks are batched into one reply turn. The related merged work fixed run-level heartbeat.model routing and adjacent cost controls, but it does not implement per-task heartbeat model selection, so this remains a product/API decision rather than conservative cleanup.

## What This Changes

The issue requests a HEARTBEAT.md task-level model hint so selected periodic heartbeat tasks can run on a different model than the rest of the heartbeat run.

## Best Possible Solution

Define a documented task-level model field in HEARTBEAT.md that resolves through the existing alias/provider model-selection path, then implement grouped or split heartbeat execution only after maintainers decide batching, timestamp, and delivery semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, and latest-release inspection give high confidence that current OpenClaw lacks task-level model metadata and uses one model selection for each heartbeat reply turn.

## Solution Assessment

Unclear until maintainers choose the contract. A typed task-level model field resolved through existing alias/provider rules is safer than a magic rerun keyword, but batching versus split-turn behavior needs product/API review.
