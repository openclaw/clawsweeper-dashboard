## Summary

Keep open: current main and the latest release still only support a single run-level heartbeat model, while per-task HEARTBEAT.md model metadata would require a new task contract and mixed-model execution semantics.

## What This Changes

The issue requests a per-task model hint or escalation marker inside HEARTBEAT.md so expensive periodic tasks can use a stronger model while routine heartbeat tasks stay on a cheaper model.

## Best Possible Solution

Define a documented task-level model field in HEARTBEAT.md that resolves through existing model alias/provider selection, then implement grouped or split heartbeat execution with explicit timestamp, transcript, and delivery semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence that current main lacks task-level model metadata and uses one model selection for each heartbeat reply turn.

## Solution Assessment

Unclear until maintainers choose the contract. A typed task-level model field resolved through existing alias/provider rules is safer than a magic rerun keyword, but batching versus split-turn behavior needs product/API review.
