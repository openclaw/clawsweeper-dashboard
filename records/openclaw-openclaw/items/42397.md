## Summary

Keep open: current main and v2026.6.10 still expose only a run-level heartbeat model override, while HEARTBEAT.md due tasks are parsed as name/interval/prompt entries and batched into one heartbeat turn. The request remains a feature and product/API decision, not a conservative cleanup close.

## What This Changes

Request to let individual HEARTBEAT.md task blocks choose a different model or escalate to the primary model instead of using one heartbeat.model for the whole heartbeat run.

## Best Possible Solution

Maintainers should define one canonical heartbeat task model-routing contract that preserves existing run-level heartbeat.model behavior, then update parser types, runner grouping or split-turn semantics, docs, and focused tests around that contract.

## Reproduction Assessment

Yes, source-level: current main's HeartbeatTask type/parser expose only name, interval, and prompt, and runHeartbeatOnce sends all due tasks through one heartbeatModelOverride. This is a feature absence, not a broken documented behavior.

## Solution Assessment

Unclear: the proposed HEARTBEAT.md model hint is plausible but not sufficient by itself because the runner currently batches due tasks into one model call. The best fix needs maintainer choice on syntax, alias/fallback behavior, and whether to split or group heartbeat turns by model.
