## Summary

Keep open. Current main intentionally resolves canonical `openai/gpt-*` Codex harness routes through the `openai-codex` auth-provider namespace, so that part of the audit is not itself evidence of a bad migration, but the report still identifies a valid user-facing gap: a direct smoke can return green after the configured primary route failed and fallback won.

## What This Changes

The issue reports that after updating to `2026.5.16-beta.5`, a migrated `openai/gpt-5.5` route failed while looking up `openai-codex` auth and the smoke passed only through Anthropic fallback.

## Best Possible Solution

Keep the canonical `openai/*` route plus `openai-codex` auth-profile compatibility, then improve smoke/status metadata so a primary-route failure and fallback winner are obvious and route broader persisted fallback audit work through the canonical observability issue.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source and docs do show that `openai-codex` auth lookup is expected for Codex-harness OpenAI routes, while the issue body provides credible beta-host evidence that the primary failure was hidden by fallback.

## Solution Assessment

No, the best fix is not a blanket migration away from `openai-codex` auth for `openai/gpt-*`. The maintainable path is to preserve the documented auth namespace and improve direct smoke/status/audit reporting for primary failure plus fallback success.
