## Summary

Keep open: the source-level ACP `final_only` bug is real, but this PR still force-flushes tool/status deliveries at intermediate `done` boundaries, has only synthetic projector terminal proof, and overlaps with the stronger open candidate at https://github.com/openclaw/openclaw/pull/84509.

## What This Changes

The branch changes the ACP reply projector so `final_only` text, output-limit state, and hidden-boundary separator state survive intermediate `done`/`error` resets, with regression tests for multi-invocation accumulation.

## Best Possible Solution

Land one canonical ACP projector fix that treats intermediate `final_only` `done` events as per-invocation boundaries, preserves accumulated text/tool/status/limit/separator state, flushes once at dispatch turn completion, and includes real ACP gateway or transport proof.

## Reproduction Assessment

Yes at source level: current main flushes and resets the ACP projector on every `done`/`error`, while dispatch also performs a later turn-level flush and docs describe `final_only` as turn-scoped buffering. I did not run tests in this read-only review.

## Solution Assessment

No. Preserving only final text while still force-flushing tool/status deliveries is not the clean turn-scoped `final_only` fix; the safer shape is to preserve all accumulated final-only state through intermediate `done` events and flush once at turn completion.
