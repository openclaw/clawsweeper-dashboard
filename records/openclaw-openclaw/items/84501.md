## Summary

Keep open: current main and v2026.6.6 still have the ACP `final_only` reset behavior, and the related sibling PRs are closed unmerged, but this branch is not merge-ready because it still drains tool/status output at intermediate `done` boundaries and lacks real ACP runtime proof.

## What This Changes

The PR changes the ACP reply projector so `final_only` text, output-limit state, and hidden-boundary separator state survive intermediate tool-call boundaries until turn-level flush.

## Best Possible Solution

Land one canonical ACP projector fix that preserves all `final_only` accumulated state through intermediate `done` events, flushes once at dispatch turn completion, and includes real ACP gateway or transport proof.

## Reproduction Assessment

Yes at source level: current main resets final-only text, counters, and hidden-boundary state on every ACP `done`/`error`, while dispatch has a later turn-level `projector.flush(true)` after `runTurn`. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR preserves final text and counters, but it still force-flushes tool/status deliveries at every intermediate `done`; the cleaner fix is to preserve the whole `final_only` turn-scoped buffer until the dispatch turn-level flush, with error behavior covered separately.
