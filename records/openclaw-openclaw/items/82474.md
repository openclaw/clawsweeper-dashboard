## Summary

Keep open: the HTTP error-mapping fix is useful, but the new actionable message points operators to `agents.defaults.agentRuntime.id`, which current runtime policy ignores, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds an OpenAI-compatible error mapper for unregistered agent harness errors, wires it into non-streaming and streaming `/v1/chat/completions`, and adds regression tests.

## Best Possible Solution

Keep the PR open, change the message to name supported provider/model-scoped runtime policy plus plugin install/enable guidance, and require redacted after-fix proof from a real gateway request.

## Reproduction Assessment

Yes. Source inspection shows `selectAgentHarness` throws the unregistered-harness error and current `openai-http.ts` falls through to generic 500/internal-error handling for that unmapped exception.

## Solution Assessment

No. Mapping the error at the HTTP boundary is a narrow maintainable direction, but the proposed message must point to current provider/model-scoped runtime policy instead of an ignored legacy key.
