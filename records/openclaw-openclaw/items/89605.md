## Summary

Keep this PR open: current main still returns `code: null` for timeout kills with no reported exit code, and this branch fixes that in the shared process helper with matching Linux/Windows tests, but the plugin-facing `null` to `124` result-shape change needs maintainer compatibility acceptance.

## What This Changes

The PR changes `runCommandWithTimeout` to return `code: 124` for timeout and no-output-timeout terminations when the child reports no numeric exit code, and updates process tests to expect that code.

## Best Possible Solution

Land the helper-level timeout result normalization after maintainer acceptance of the plugin-facing contract, while keeping the broader macOS launchd/startup issue open for its remaining symptoms.

## Reproduction Assessment

Yes from source inspection: current main can return `code: null` for timeout/no-output-timeout kills when no numeric child status is reported. I did not run a local current-main repro because this was a read-only review, but the code path is direct and the PR body provides after-fix live output.

## Solution Assessment

Yes, this is the right code layer because the shared process helper owns timeout result semantics and caller-by-caller normalization would leave the public runtime API ambiguous. The remaining question is compatibility acceptance, not a narrower mechanical fix.
