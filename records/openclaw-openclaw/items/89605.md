## Summary

Keep open: this PR fixes a real, source-reproducible timeout-result bug in the shared process runtime with no blocking correctness findings, but it changes a plugin-facing return-code semantic so maintainers should explicitly accept that compatibility tradeoff before merge.

## What This Changes

The branch changes `runCommandWithTimeout` to return exit code `124` for timeout and no-output-timeout results when the killed child reports `null` or `0`, and tightens Linux/Windows timeout tests to assert that exact code.

## Best Possible Solution

Land the shared process-runtime normalization once maintainers accept `124` as the canonical timeout/no-output-timeout code while preserving the existing `termination` and `signal` fields.

## Reproduction Assessment

Yes, from source inspection and a focused Node child-process probe: current main maps timeout `0` to `124` but leaves a SIGKILL timeout's `null` code unchanged.

## Solution Assessment

Yes, normalizing at the shared process-runtime boundary is the narrowest maintainable fix; caller-by-caller coercion would leave the plugin runtime contract inconsistent.
