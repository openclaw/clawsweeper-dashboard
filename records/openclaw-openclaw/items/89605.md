## Summary

Keep open for maintainer review: the patch is a narrow, source-backed fix for a real timeout-result edge case and is not on current `main`, but it intentionally changes a plugin-exposed runtime result from `code: null` to `code: 124` for some timeouts.

## What This Changes

The PR makes `runCommandWithTimeout` return `code: 124` for timeout and no-output-timeout kills when the child process reports no numeric exit code, and tightens Linux/Windows regression tests to expect 124.

## Best Possible Solution

Land this narrow process-runtime fix only after maintainer acceptance of the plugin-runtime compatibility change, while keeping the broader launchd/ClawKitchen work tracked in https://github.com/openclaw/openclaw/issues/73743.

## Reproduction Assessment

Yes. Source inspection shows a current-main timeout kill with no numeric child code can resolve `code: null`; the PR body also includes after-fix Docker output for a real timed-out child process.

## Solution Assessment

Yes, with maintainer acceptance. Centralizing timeout-code normalization in `runCommandWithTimeout` is narrower and less drift-prone than requiring every core or plugin caller to reinterpret `null` timeout results.
