## Summary

Keep open for maintainer review: the patch appears to fix the narrow timeout-code bug with no blocking findings and useful real behavior proof, but it changes the plugin-facing process result contract from `code: null` to `code: 124` for timeout kills, so maintainers should explicitly accept that compatibility impact before merge.

## What This Changes

The PR changes `runCommandWithTimeout` to normalize timeout and no-output-timeout kills with missing or zero child exit codes to `124`, with Linux and Windows regression assertions updated to require that exact code.

## Best Possible Solution

Land the narrow runtime fix after maintainers explicitly accept `124` as the plugin-facing timeout/no-output-timeout code, and leave the broader startup/fan-out/cache work tracked in https://github.com/openclaw/openclaw/issues/73743.

## Reproduction Assessment

Yes. Source inspection shows current main can preserve `code: null` for timeout kills because only `0` is normalized to `124`, and the PR's focused tests model that exact killed-child path.

## Solution Assessment

Yes, this is the narrow owner-boundary fix for the shared process helper; changing individual plugin callers would leave the plugin runtime API with the same ambiguous timeout result, but maintainers still need to accept the compatibility change.
