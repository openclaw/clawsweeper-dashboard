## Summary

Keep open: current main and the latest release still carry the macOS `NODE_USE_SYSTEM_CA=1` service default, OpenAI/Codex auth discovery is still on the affected runtime path, and no duplicate or merged fix owns this specific keychain failure.

## What This Changes

The issue reports that `NODE_USE_SYSTEM_CA=1` in the macOS Gateway/runtime environment breaks `openai-codex` keychain-backed auth reads and can block fresh runtime launches with `SecItemCopyMatching failed -50`.

## Best Possible Solution

Keep this open and fix the macOS runtime/auth environment boundary so Codex/OpenAI keychain-backed auth works under fresh service launches without regressing launchd TLS trust behavior.

## Reproduction Assessment

Partly: source inspection confirms current main and `v2026.5.28` propagate `NODE_USE_SYSTEM_CA=1` into macOS service envs and that `models auth list` reaches OpenAI/Codex external auth discovery. The exact `SecItemCopyMatching failed -50` throw still needs a live macOS reproduction because current no-prompt OpenClaw credential reads skip or catch direct keychain failures.

## Solution Assessment

Unclear: the reported bundled-CA workaround is a strong mitigation signal, but the best fix should be chosen after isolating whether the failing path is OpenClaw service env propagation, native Codex keyring access, or a child runtime secret path. A scoped env sanitization is safer than removing the macOS system-CA default globally without TLS upgrade proof.
