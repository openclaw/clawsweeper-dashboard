## Summary

Keep open: current main and latest release still combine a macOS `NODE_USE_SYSTEM_CA=1` service default with keychain-capable Codex auth reads, and no merged same-repo fix or canonical duplicate proves the reported macOS keychain failure is resolved.

## What This Changes

The issue reports that macOS `NODE_USE_SYSTEM_CA=1` breaks OpenAI Codex keychain-backed auth reads and can block fresh runtime launches with `SecItemCopyMatching failed -50`.

## Best Possible Solution

Reproduce the macOS keychain failure with live proof, then fix the narrow auth/startup boundary so Codex keychain reads or fresh launch auth do not inherit the problematic CA mode while preserving launchd TLS reliability.

## Reproduction Assessment

No high-confidence live reproduction is available in this review: the issue provides concrete macOS commands and current source still has the relevant startup/auth path, but the `SecItemCopyMatching failed -50` failure was not reproduced against current main.

## Solution Assessment

Unclear: the reported bundled-CA workaround is useful evidence, but flipping the global macOS CA default is not clearly the best fix because that default protects launchd TLS; a scoped auth/startup boundary fix needs maintainer choice and macOS proof.
