## What This Changes

The issue reports a macOS regression where `NODE_USE_SYSTEM_CA=1` makes legacy OpenAI Codex OAuth/keychain reads fail with `SecItemCopyMatching failed -50` and can block fresh runtime launches.

## Best Possible Solution

Keep this as a maintainer-owned macOS auth/startup bug, reproduce it with live keychain proof, then fix the narrow boundary that prevents Codex keychain reads from inheriting the problematic CA mode without regressing launchd TLS.

## Reproduction Assessment

Unclear for full live behavior: the source path still combines the macOS `NODE_USE_SYSTEM_CA=1` service default with keychain-capable Codex auth code, but I did not run the reported macOS keychain command or reproduce `SecItemCopyMatching failed -50` on current main.

## Solution Assessment

Unclear: flipping the CA default globally is not clearly the best fix because that default exists for launchd TLS reliability, while current no-prompt auth paths only partially avoid raw keychain access.
