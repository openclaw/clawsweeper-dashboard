## Summary

Keep this open: current main and the latest release still carry the macOS `NODE_USE_SYSTEM_CA=1` daemon/startup default, while current auth paths only partially reduce keychain exposure by using non-prompting external CLI reads. I do not see a merged fix or duplicate that fully owns the reported CA/keychain failure, and the remaining decision needs macOS live proof before changing launch TLS behavior.

## What This Changes

The issue reports that `NODE_USE_SYSTEM_CA=1` on macOS breaks `openai-codex` keychain-backed auth reads and can prevent fresh runtime launches with `SecItemCopyMatching failed -50`.

## Best Possible Solution

Keep the daemon TLS default and Codex auth/keychain boundary under maintainer review, then land a targeted macOS-proven fix or doctor/runtime migration that prevents Codex keychain auth from inheriting the problematic CA mode without breaking launchd TLS.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully; current main still contains the reported macOS `NODE_USE_SYSTEM_CA=1` startup default and direct Codex keychain read support, but the actual `SecItemCopyMatching` failure needs live macOS proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear; the external credential-delegation comment may be a useful workaround direction, but the maintainable OpenClaw fix needs a targeted auth/startup boundary decision rather than a broad CA default flip.
