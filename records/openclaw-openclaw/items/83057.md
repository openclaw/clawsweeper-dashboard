## Summary

Current main and v2026.5.12 already decouple Dreaming from heartbeat active-hours by running the managed Dreaming cron as an isolated agent turn, with doctor migration for stale persisted jobs; the 3 AM docs default is therefore consistent with runtime behavior now.

## What This Changes

The issue asks for Dreaming cron runs and documentation/defaults to stop being coupled to heartbeat active-hours quiet-hours gating.

## Best Possible Solution

Keep the shipped isolated-agent-turn Dreaming cron design and point stale installs at `openclaw doctor --fix` rather than adding a new Dreaming active-hours escape hatch.

## Reproduction Assessment

No. The reported current-main path is obsolete: source inspection shows new Dreaming cron jobs no longer enter the heartbeat active-hours path, and stale persisted jobs have a doctor migration to the same detached shape.

## Solution Assessment

Yes. The shipped detached cron implementation is the narrowest maintainable fix because Dreaming is background maintenance and should not inherit heartbeat notification quiet-hours policy.
