## Summary

Keep open: current main still contains the synchronous plugin index, manifest, and discovery path named in the profile, and the linked heartbeat cooldown fix does not address this plugin-discovery root cause. The report is actionable performance work, but the fix spans the plugin metadata/snapshot boundary and should get owner review rather than an automated narrow patch.

## What This Changes

The issue asks OpenClaw to remove synchronous plugin discovery/index/manifest I/O from Gateway hot paths because it blocks the event loop and delays replies.

## Best Possible Solution

Keep the issue open for a bounded plugin/gateway refactor that removes sync discovery from live Gateway hot paths while preserving manifest-first metadata, explicit snapshot invalidation, and plugin safety checks.

## Reproduction Assessment

No high-confidence local reproduction was run in this read-only review. Source inspection confirms the named synchronous discovery/index/manifest path on current main, but the reported 8s event-loop spike remains environment-dependent without the attached profile artifact or a live WSL2 run.

## Solution Assessment

No: the broad suggested conversion of every hot helper to async is probably not the safest standalone fix. The maintainable direction is to carry prepared plugin metadata/snapshots through Gateway flows and asyncify only the remaining unavoidable cold I/O with focused invalidation and startup tests.
