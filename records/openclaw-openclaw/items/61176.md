## Summary

Keep open. Current main still does not have this native Control UI session-sidebar surface, but this PR is not mergeable as-is because the diff has source-proven P1/P2 behavior bugs and lacks after-fix real behavior proof.

## What This Changes

The PR adds a Control UI session sidebar with grouped recent sessions, context usage indicators, a New Chat control, `/new` history metadata, and related UI/build refactors.

## Best Possible Solution

Repair and rebase the sidebar slice on current main, preserving the shipped New Chat create/switch path, using stable archived transcript references, decoupling session-sidebar state from tool-output sidebar state, and leaving broader directory/search/archive decisions to the canonical tracker.

## Reproduction Assessment

Yes, for the PR-introduced defects by source inspection. The transcript bug follows from copying `currentEntry.sessionFile` before reset archival renames it, and the shortcut/sidebar-state bugs follow from duplicate key handlers plus shared `sidebarOpen` state.

## Solution Assessment

No. The feature direction is useful, but the current patch should not merge until it is rebased onto current main and the transcript, shortcut, sidebar-state, and proof gaps are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the archived transcript path for /new history:** `src/gateway/session-reset-service.ts:458-463`
  - body: The preserved history row copies `currentEntry` before the reset flow archives the old transcript, so it can store a `sessionFile` path that is renamed moments later. Selecting that row can reopen an empty or missing transcript instead of the prior conversation.
  - confidence: 0.92
- **[P2] Stop the session shortcut from toggling twice:** `ui/src/ui/views/chat.ts:1144-1151`
  - body: This Ctrl/Cmd+` handler toggles the session sidebar inside the chat while the PR also adds a document-level handler for the same shortcut. When focus is in chat, the event bubbles and both handlers run, leaving the sidebar in its original state.
  - confidence: 0.9
- **[P2] Restore the prior sidebar after closing sessions:** `ui/src/ui/app.ts:795-797`
  - body: Closing the session sidebar also forces `sidebarOpen = false`, so a user who had a markdown or tool-output sidebar open loses that panel after briefly opening the session list. The session overlay should be decoupled from, or restore, the prior sidebar state.
  - confidence: 0.78
