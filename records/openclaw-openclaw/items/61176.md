## Summary

Keep open. The feature direction is still useful and not implemented by current main’s narrower header picker/New Chat flow, but the PR has source-reproducible correctness bugs and no inspectable real behavior proof.

## What This Changes

The PR adds a Control UI session sidebar, context usage meter threshold changes, a full-width New Chat action, 7-day ended-session visibility, and `/new` history metadata.

## Best Possible Solution

Rebase and narrow the sidebar slice on current main, reuse the shipped sessions.create/New Chat path, store stable archived transcript references, decouple session-sidebar state from tool-output sidebar state, and leave broader directory/search/archive decisions to the canonical tracker.

## Reproduction Assessment

Yes, for the PR-introduced defects by source inspection. The reset-history bug follows from copying currentEntry before transcript archival renames the file, and the shortcut/sidebar-state bugs follow from duplicate handlers plus shared sidebarOpen state.

## Solution Assessment

No. The feature direction is plausible, but this patch should not merge until it is rebased onto current main, the transcript/shortcut/sidebar-state defects are fixed, and contributor real behavior proof is provided.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the archived transcript path for /new history:** `src/gateway/session-reset-service.ts:454-456`
  - body: The preserved history row copies currentEntry, including sessionFile, before the reset flow archives and renames the old transcript. Selecting that row can reopen a missing or empty transcript instead of the previous conversation.
  - confidence: 0.92
- **[P2] Stop the session shortcut from toggling twice:** `ui/src/ui/views/chat.ts:1141-1142`
  - body: This Ctrl/Cmd+` handler toggles the session sidebar inside chat while app.ts also registers a document-level handler for the same shortcut. When focus is in chat, the event bubbles and both handlers run, leaving the sidebar in its original state.
  - confidence: 0.9
- **[P2] Restore the previous sidebar after closing sessions:** `ui/src/ui/app.ts:797-800`
  - body: Closing the session sidebar also forces sidebarOpen=false, so a user who had markdown or tool-output content open loses that panel after briefly opening the session list. The session overlay should be decoupled from, or restore, the prior sidebar state.
  - confidence: 0.78
