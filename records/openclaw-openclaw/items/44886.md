## Summary

Keep open. Current main still accepts and applies Telegram direct config for DM routing/prompt behavior, and global `tools.toolsBySender` is available as a partial workaround, but the specific `direct.<peer>.tools` / `toolsBySender` surface still is not reliably enforced because the shared policy context only recognizes group/channel session keys; the related fix PR is still open and unmerged.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` and `toolsBySender` should enforce per-DM-user tool restrictions, and to either fix enforcement or clarify the config surface.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender` while preserving existing provider-scoped sender-key semantics and adding regression plus live Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram direct config is accepted and selected, but direct session keys are filtered out before the shared tool-policy resolver can find a trusted policy scope.

## Solution Assessment

Yes for the related fix direction, but it is not on current main. The maintainable solution is to route the existing Telegram direct config surface through a direct/DM-aware shared policy path rather than documenting away an accepted security-sensitive config shape.
