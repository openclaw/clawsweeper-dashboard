## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` / `toolsBySender` and applies direct prompt config, but the shared tool-policy resolver only trusts group/channel session scopes, so Telegram DM tool denies are still not reliably enforced; the linked fix PR remains open and unmerged.

## What This Changes

The issue asks whether Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` is intended to enforce per-DM user tool policy and, if so, to fix the enforcement path.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a direct/DM-aware policy resolver that enforces Telegram `direct.<peer>` and wildcard `tools`/`toolsBySender` with regression coverage and real Telegram proof.

## Reproduction Assessment

Yes. By source inspection on current main, Telegram direct config is accepted and prompt settings are applied, but direct session keys do not produce trusted group/channel context for `resolveGroupToolPolicy`.

## Solution Assessment

Yes on the related fix direction, but it is not implemented on current main. The maintainable solution is to route the exposed Telegram direct config surface through a direct/DM-aware shared policy path rather than documenting away a security-sensitive config mismatch.
