## Summary

Keep open. Current main and the latest release still accept Telegram `direct.<peer>.tools`/`toolsBySender` config while the enforced tool-policy path is still group/channel-scoped; the related repair PR was closed unmerged and this remains the canonical issue.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` is intended to enforce user-specific tool permissions, and requests either runtime enforcement or schema/docs clarification.

## Best Possible Solution

Keep this as the canonical security-sensitive report until Telegram direct tool policy is either enforced through the shared resolver with regression and Telegram proof, or intentionally retired/clarified with schema/docs and migration support.

## Reproduction Assessment

Yes. Source inspection reproduces the mismatch: Telegram direct tool policy is accepted and direct config is consumed for DMs, but the mounted policy path only trusts group/channel contexts and reads `groups`.

## Solution Assessment

Unclear until maintainers choose the intended contract. The maintainable options are to make the canonical policy resolver direct-session aware, or deliberately retire/clarify the advertised direct tool-policy surface with compatible docs/schema/doctor handling.
