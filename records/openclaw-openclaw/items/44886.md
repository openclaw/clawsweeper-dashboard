## Summary

Keep open. Current main still accepts Telegram direct-message `tools` / `toolsBySender` config and applies the selected direct config to prompts, but the shared tool-policy resolver only derives trusted policy scope from group/channel session keys, so this remains a source-reproducible security bug with an open related fix PR still needing maintainer/security review.

## What This Changes

Reported that Telegram `direct.<peer>.tools` and `toolsBySender` appear accepted by config but are not reliably enforced as per-DM-user tool policy.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a rebased direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender`, preserves provider-scoped sender precedence, and includes focused regression plus real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Telegram DMs use direct-session keys, while the current shared policy parser and resolver only trust group/channel session context, so `direct.*.tools` is skipped before policy lookup.

## Solution Assessment

Yes for the related fix direction, but it is not landed. The maintainable solution is to route the accepted Telegram direct config surface through a direct/DM-aware shared policy path while preserving existing sender matching precedence.
