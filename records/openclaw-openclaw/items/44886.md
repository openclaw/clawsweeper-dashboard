## Summary

Keep open: current main and v2026.6.6 still accept Telegram direct `tools` config while resolving actual tool policy through group/channel-only paths, and the related fix PR is closed unmerged rather than a safe canonical landing path.

## What This Changes

The issue reports that Telegram direct-message `direct.<peer>.tools` config is schema-valid but not reliably enforced as a per-DM user tool policy.

## Best Possible Solution

Make the accepted Telegram direct `tools` and `toolsBySender` surface either enforce through a direct/DM-aware policy path or retire/migrate the schema surface with explicit maintainer security direction.

## Reproduction Assessment

Yes. Source inspection reproduces the mismatch: Telegram direct `tools` is accepted and typed, but direct session keys are rejected by the group-policy parser and the resolver reads only group config; I did not run a live Telegram DM.

## Solution Assessment

No existing merged solution is sufficient. The best implementation path is a refreshed direct/DM-aware resolver with focused Telegram regression coverage, unless maintainers decide to retire the accepted config surface with migration/docs instead.
