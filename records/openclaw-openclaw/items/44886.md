## Summary

Keep open. Current main still accepts and advertises Telegram direct `tools` / `toolsBySender`, and Telegram resolves direct DM config for prompt context, but mounted-tool filtering still goes through a group/channel-only resolver that drops `direct` session keys before policy lookup; open PR #53530 remains the active implementation candidate.

## What This Changes

Request confirmation or fix for Telegram per-DM `direct.<peer>.tools` so accepted direct-message tool policies are reliably enforced for mounted tools.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policies through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, sender overrides, and thread/parent fallback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: use a Telegram direct session key such as `agent:main:telegram:direct:<peer>` with `direct.<peer>.tools`, and the mounted-tool policy parser rejects `direct` before any direct policy lookup.

## Solution Assessment

Yes for the direction, but not yet in current main. The narrow maintainable fix is the shared direct-session policy resolver described by PR #53530, followed by Telegram direct-surface coverage.
