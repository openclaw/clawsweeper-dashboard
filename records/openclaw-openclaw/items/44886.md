## Summary

Keep open. Current main still accepts Telegram per-DM `tools`/`toolsBySender`, but direct Telegram session keys are rejected before mounted-tool policy lookup; the related fix PR is still open and conflicting, not merged.

## What This Changes

The issue asks OpenClaw to enforce `channels.telegram.accounts.<account>.direct.<peer>.tools` as per-DM tool policy, or clarify/remove that config surface if unsupported.

## Best Possible Solution

Route Telegram direct config through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, sender overrides, and thread/parent fallback; if unsupported, deprecate or remove the schema/type surface and document the supported policy path.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Telegram can route DMs as `agent:<agent>:telegram:direct:<peer>`, while the mounted-tool policy parser rejects `direct`/`dm` kinds before any direct `tools` lookup.

## Solution Assessment

Yes on direction, but it is not fixed on main. The maintainable fix is to land, repair, split, or replace the shared direct-session resolver work rather than add a separate Telegram-only filter.
