## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` config and applies direct prompt settings, but the enforcement path still derives only group/channel tool-policy context, so direct/DM session keys can bypass the configured per-DM tool policy; the related fix PR remains open and unmerged.

## What This Changes

The issue asks whether Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` is intended to enforce per-DM-user tool permissions and, if so, to fix that enforcement path.

## Best Possible Solution

Land or replace the related direct/DM-aware policy resolver so Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` are enforced with account, sender, thread/parent, explicit, and wildcard precedence covered by regression tests and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A Telegram direct session key such as `agent:main:telegram:direct:<peer>` cannot produce trusted tool-policy context even though `channels.telegram.accounts.<account>.direct.<peer>.tools` is accepted and direct prompt config is applied.

## Solution Assessment

Yes on direction: the narrow maintainable fix is to enforce the already-exposed Telegram direct config surface through a direct/DM-aware policy path. A docs-only clarification would leave an unsafe config contract, while https://github.com/openclaw/openclaw/pull/53530 needs refresh, review, and proof or a narrower replacement.
