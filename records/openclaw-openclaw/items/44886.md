## Summary

Keep open. Current main still accepts and advertises Telegram direct `tools` / `toolsBySender`, and applies the resolved direct config to prompt context, but mounted-tool filtering still goes through a group/channel-only resolver that rejects `direct` session keys before policy lookup; related PR #53530 remains the active implementation candidate.

## What This Changes

Request that Telegram `direct.<peer>.tools` and `toolsBySender` be enforced as per-DM tool policy or clarified as unsupported.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, sender overrides, and thread/parent fallback; if that surface is not intended, deprecate or remove the schema/types and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: Telegram can produce `agent:main:telegram:direct:<peer>` session keys while the mounted-tool policy parser rejects `direct` keys before any `direct.<peer>.tools` lookup.

## Solution Assessment

Yes for the direction, but not yet for current main. The maintainable fix is a shared direct-session policy resolver used by mounted tools; PR #53530 appears to track that path, but it is still unmerged.
