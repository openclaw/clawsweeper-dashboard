## Summary

Keep open: current `main` and v2026.6.8 still accept Telegram direct `tools` config while the enforced tool-policy path is group/channel-scoped, and the related repair PR is closed unmerged rather than a safe canonical fix path.

## What This Changes

The issue asks OpenClaw to enforce or clarify Telegram per-DM `direct.<peer>.tools` and `toolsBySender` policies that the schema and types currently accept.

## Best Possible Solution

Choose maintainer/security direction, then either enforce Telegram direct tools through a direct/DM-aware policy path with regression coverage or retire and migrate the accepted config surface.

## Reproduction Assessment

Yes, at source level. Current `main` accepts Telegram direct `tools` config, selects direct config for DMs, and then resolves actual tool policy through group/channel-only paths; I did not run a live Telegram DM.

## Solution Assessment

No merged solution is sufficient yet. The best implementation path is a refreshed direct/DM-aware resolver with focused Telegram coverage unless maintainers decide to retire the accepted config surface with migration and docs.
