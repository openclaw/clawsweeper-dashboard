## Summary

Keep open. Current main still accepts Telegram `direct.*.tools` and `toolsBySender` config, and Telegram direct config is used for prompts, but the shared tool-policy resolver still only derives trusted policy scope from group/channel session keys, leaving the reported DM tool restriction gap unresolved.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` config should be enforced as actual per-user tool policy and, if so, to fix the DM tool-policy path.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender`, preserves provider-scoped sender precedence, and includes focused regression plus redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts and selects Telegram direct config, but the shared resolver only trusts group/channel session context, so direct-session keys never reach a direct-policy lookup.

## Solution Assessment

Yes, the best direction is to route the accepted Telegram direct config surface through a direct/DM-aware shared policy path. The related PR is the right shape conceptually, but it still needs maintainer/security review and proof before it can close this issue.
