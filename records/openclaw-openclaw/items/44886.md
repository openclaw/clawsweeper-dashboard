## Summary

Keep open. Current `main` still exposes Telegram `direct.<peer>.tools` / `toolsBySender` in config and applies direct prompt config, but the shared tool-policy path only derives trusted policy context from group/channel sessions; the related fix PR is still open and unmerged.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` is intended to enforce real tool permissions, and to fix the runtime path or clarify the config surface.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender`, plus regression coverage and real Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current `main` shows Telegram accepts and selects direct config, but direct session keys are filtered out before the shared tool-policy resolver can find a trusted policy scope.

## Solution Assessment

Yes for the related fix direction, but it is not on current `main`. The maintainable solution is to route the existing Telegram direct config surface through a direct/DM-aware shared policy path rather than documenting away a security-sensitive mismatch.
