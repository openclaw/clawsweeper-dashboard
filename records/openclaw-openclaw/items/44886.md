## Label Justifications

- `P1`: A configured Telegram DM tool restriction can fail open at runtime, which is an urgent security-sensitive channel workflow bug.
- `impact:security`: The issue concerns tool-authorization boundaries between different Telegram DM users sharing one agent.

## Summary

Keep open. Current main and the latest shipped tag still accept Telegram `direct.*.tools` / `toolsBySender` and select per-DM direct config, but the shared tool-policy resolver only derives scope from group/channel session keys, so accepted Telegram DM policy can still fail to constrain the mounted tools; the related open PR is the active fix path, not a reason to close this issue.

## What This Changes

The issue asks whether Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` should enforce per-DM-user tool permissions and, if so, requests fixing the mismatch between accepted config and runtime tool filtering.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram exact-peer and wildcard `tools` / `toolsBySender`, preserves sender precedence and provider-scoped sender matching, and includes security-focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. This is source-reproducible on current main and `v2026.5.20`: Telegram accepts and selects direct config, while the policy conversation parser rejects `direct` session keys and the resolver remains group/channel scoped.

## Solution Assessment

Yes. The maintainable fix is to route the accepted Telegram direct config surface through a shared direct/DM-aware policy path rather than adding another Telegram-only filter or documenting away an advertised type/schema contract.
