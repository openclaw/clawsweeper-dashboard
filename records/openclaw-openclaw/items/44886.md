## Label Justifications

- `P1`: A configured Telegram DM tool restriction can fail open at runtime, which is an urgent security-sensitive channel workflow bug.
- `impact:security`: The issue concerns tool-authorization boundaries between different Telegram DM users sharing one agent.

## Summary

Keep open. Current main and the latest shipped tag still accept Telegram `direct.*.tools` / `toolsBySender` and select per-DM direct config, but the enforced tool-policy path remains group/channel scoped, so the accepted per-DM policy can still fail open; the related open PR is the active fix path, not a completed implementation.

## What This Changes

The issue asks OpenClaw to either enforce Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` as per-DM tool policy or clarify the config/schema contract.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram exact-peer and wildcard `tools` / `toolsBySender`, preserves sender precedence and provider-scoped matching, and includes focused security regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. This is source-reproducible on current main and `v2026.5.20`: Telegram accepts and selects direct config, while the trusted policy parser rejects `direct` session keys and the resolver remains group/channel scoped.

## Solution Assessment

Yes. The maintainable fix is to route the accepted Telegram direct config surface through a shared direct/DM-aware policy path rather than adding a Telegram-only filter or documenting away an advertised schema/type contract.
