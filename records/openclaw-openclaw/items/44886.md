## Summary

Keep open. Current main still exposes Telegram direct-message tool policy in config and applies direct prompt config, but the runtime tool-policy path derives only group/channel scope and drops direct/DM session keys before enforcement; the related implementation PR is still open and needs maintainer review rather than pre-closing this issue.

## What This Changes

The issue reports that Telegram per-DM `direct.<peer>.tools` and `toolsBySender` config is accepted and prompt overrides apply, but tool denies are not reliably enforced for direct-message sessions.

## Best Possible Solution

Wire Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` into a direct/DM-aware tool-policy resolver with account, sender, thread/parent, explicit, and wildcard precedence covered by regression tests.

## Reproduction Assessment

Yes, by source inspection. A Telegram direct session key such as `agent:main:telegram:direct:<peer>` cannot produce trusted tool-policy context even though `channels.telegram.accounts.<account>.direct.<peer>.tools` is accepted and other direct config is applied.

## Solution Assessment

Yes on direction: the narrow maintainable fix is to enforce the already-exposed Telegram direct config surface through a direct/DM-aware policy path. A docs-only clarification would leave a misleading security-relevant config shape, while the open related PR needs rebase, review, and proof before it can be the fix.
