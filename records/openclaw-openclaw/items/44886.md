## Summary

Keep open: current main and the latest release still accept Telegram `direct.<peer>.tools` config while the tool-policy resolver remains group/channel-scoped, and no merged PR now owns the fix. The duplicate report https://github.com/openclaw/openclaw/issues/85665 preserves matching evidence, while https://github.com/openclaw/openclaw/pull/53530 was closed unmerged and is not a safe canonical fix path.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` config should enforce actual per-user tool policy instead of only accepting the schema shape.

## Best Possible Solution

Make the runtime honor schema-accepted Telegram direct `tools` and `toolsBySender` through a direct/DM-aware resolver that preserves account, wildcard, thread-parent, and sender-policy precedence; if maintainers reject that contract, remove or migrate the accepted config surface instead of leaving it silently unenforced.

## Reproduction Assessment

Yes. Source inspection reproduces the mismatch: Telegram direct `tools` is accepted and typed, but the policy path rejects direct session keys and reads only group/channel config; I did not run a live Telegram DM.

## Solution Assessment

No existing merged solution is sufficient. The closed PR had the right general direction but is not the best landing path as submitted; the best fix is a refreshed direct/DM-aware resolver with focused Telegram regression coverage and security review.
