## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` is intended to enforce per-user tool permissions and, if so, to fix the schema/runtime mismatch.

## Best Possible Solution

Keep this as the canonical security-sensitive report until Telegram direct config is either wired into the shared tool-policy resolver with regression coverage or the accepted schema/docs surface is removed or clarified.

## Reproduction Assessment

Yes. Source inspection reproduces the mismatch: Telegram direct `tools` is accepted and direct config is resolved for DMs, but the mounted tool-policy path accepts only group/channel contexts and reads group policy config.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrow maintainable direction is to make the canonical policy resolver direct-session aware, or deliberately retire/clarify the advertised direct tool-policy surface with migration/docs support.
