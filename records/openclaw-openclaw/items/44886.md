## Summary

Keep open. Current main still accepts Telegram DM-level tool policy while the enforcement path resolves only group/channel contexts and group policy config, and the related repair PR is closed unmerged.

## What This Changes

The issue asks whether `channels.telegram.accounts.<account>.direct.<peer>.tools` should enforce per-DM Telegram tool permissions, or be removed/clarified if it is not supported.

## Best Possible Solution

Honor the accepted Telegram direct tool-policy surface through one canonical direct-session-aware resolver and typed channel/plugin seam, with regression coverage for exact peer, wildcard, account, sender, thread, and group-isolation cases.

## Reproduction Assessment

Yes at source level: current main accepts Telegram `direct.*.tools`, creates direct session keys, and then resolves tool policy through group/channel-only paths that never read the direct map. I did not run a live Telegram DM in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix is to extend the canonical policy resolver/channel contract for direct sessions rather than adding a one-off Telegram-only enforcement branch. Maintainers still need to decide the exact security-reviewed API shape before implementation.
