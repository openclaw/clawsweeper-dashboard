## Summary

Keep open. Current main still accepts and types Telegram direct `tools` / `toolsBySender`, but the mounted-tool policy path is still group/channel-oriented and rejects `direct` session keys before policy lookup; the linked implementation candidate remains open and unmerged.

## What This Changes

The issue asks whether Telegram direct-message `direct.<peer>.tools` should enforce per-user mounted tool policy, because config accepts it while runtime enforcement appears unreliable.

## Best Possible Solution

Route Telegram direct config through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, sender overrides, and thread/parent fallback; if maintainers decide that surface is unsupported, remove or deprecate the schema/types and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: Telegram can create `agent:main:telegram:direct:<peer>` session keys while the mounted-tool policy parser rejects `direct` session kinds before any direct `tools` lookup.

## Solution Assessment

Yes on direction, but not completed on main. The narrow maintainable path is to land, repair, split, or replace the linked shared direct-session policy resolver instead of adding a separate Telegram-only filter.
