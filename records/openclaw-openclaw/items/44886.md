## Summary

Keep open. Current main still accepts Telegram direct-message tool policy config and applies direct prompt config, but the runtime policy resolver still derives only group/channel scope and drops direct/DM session keys before enforcement; the related open PR is useful context but has not merged and still needs maintainer handling.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` should enforce per-DM-user tool permissions and, if so, to fix that enforcement path.

## Best Possible Solution

Land or replace the related direct/DM-aware policy resolver so Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` are enforced with account, sender, thread/parent, explicit, and wildcard precedence covered by regression tests.

## Reproduction Assessment

Yes, by source inspection. A Telegram direct session key such as `agent:main:telegram:direct:<peer>` cannot produce trusted tool-policy context even though `channels.telegram.accounts.<account>.direct.<peer>.tools` is accepted and direct prompt config is applied.

## Solution Assessment

Yes on direction: the narrow maintainable fix is to enforce the already-exposed Telegram direct config surface through a direct/DM-aware policy path. A docs-only clarification would leave an unsafe config contract, while https://github.com/openclaw/openclaw/pull/53530 needs rebase/proof/repair or replacement before it can close this issue.
