## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` / `toolsBySender` and applies the resolved direct config to DM prompt context, but mounted-tool filtering still routes through a group/channel-only policy path, and related PR #53530 remains the active implementation candidate.

## What This Changes

The issue asks OpenClaw to either enforce Telegram per-DM `direct.<peer>.tools` policy or clarify/remove the accepted config surface.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, sender overrides, and thread/parent fallback; if that surface is not intended, deprecate or remove the schema/types and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: Telegram DM direct config is accepted and used for prompt context, while the mounted-tool policy path rejects `direct` session keys before policy lookup.

## Solution Assessment

Unclear for the exact open PR until maintainer review, but the right solution direction is a shared direct-session policy resolver rather than more Telegram-only special casing or documenting an accepted-but-ignored authorization field.
