## Summary

Keep open. Current main still accepts and selects Telegram `direct.*.tools` config for DMs, but the shared tool-policy resolver only derives trusted policy scope from group/channel session keys, so the reported authorization gap is still present and the linked fix PR remains open.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` config is meant to enforce per-user tool policy and reports that accepted config is not reliably enforced at runtime.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender`, preserves sender precedence, and includes security-focused regression plus redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts and selects Telegram direct config, but the shared resolver returns no trusted policy context for `direct` session keys, so direct-session keys never reach direct-policy lookup.

## Solution Assessment

Yes, the best direction is to route the accepted Telegram direct config surface through a direct/DM-aware shared policy path. The linked PR is conceptually the right fix path, but it still needs maintainer/security review and proof before it can close this issue.
