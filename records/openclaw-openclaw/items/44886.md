## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` / `toolsBySender` in schema and uses the resolved direct peer config for Telegram DM prompt context, but mounted-tool filtering still goes through the group/channel-only policy path, so direct-session tool denies are not reliably enforced. The related implementation candidate #53530 remains open, so this should stay tracked for maintainer review rather than be closed or swept into a separate automated repair.

## What This Changes

The issue requests confirmation and enforcement of Telegram per-DM `direct.<peer>.tools` policy, or schema/docs clarification if that config surface is not intended to control mounted tools.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, sender overrides, and thread/parent fallback where appropriate. If Telegram direct tool policy is not intended, remove or deprecate that schema/type surface and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Telegram direct session key is rejected by the group/channel-only session parser before mounted-tool policy lookup, while the same direct peer config is accepted and consumed for prompt overrides.

## Solution Assessment

Yes for triage. Keeping this open while maintainers review, land, split, or replace #53530 is the narrowest maintainable path because the missing behavior is in an authorization-sensitive shared tool-policy resolver.
