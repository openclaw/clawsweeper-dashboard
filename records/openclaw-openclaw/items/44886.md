## Summary

Keep open. Current main still exposes Telegram direct-message tool policy in schema/types and resolves `direct` config for DMs, but the shared runtime policy resolver only derives group/channel context, so `direct.*.tools` is not enforced through the actual tool filter; the related implementation PR remains open and does not close the Telegram surface by itself.

## What This Changes

Make Telegram `direct.<peer>.tools` and `toolsBySender` enforce actual per-DM-user tool policy, or clarify the config contract if that surface is not intended.

## Best Possible Solution

Wire Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` into a direct/DM-aware tool-policy resolver with account, thread/topic, sender, and wildcard precedence coverage, then keep schema/docs/tests aligned with the enforced behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a Telegram direct session key such as `agent:main:telegram:direct:<peer>` cannot produce trusted policy context, even though `channels.telegram.accounts.<account>.direct.<peer>.tools` is accepted.

## Solution Assessment

Yes. The narrow maintainable solution is to enforce the already-exposed direct config surface through a direct/DM-aware policy path; a docs-only clarification would leave a misleading, security-relevant accepted config shape.
