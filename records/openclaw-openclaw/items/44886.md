## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` / `toolsBySender` and applies the selected direct config for prompts, but the shared enforcement path only derives trusted tool-policy scope from group/channel session keys; https://github.com/openclaw/openclaw/pull/53530 tracks the fix direction and remains open/conflicting.

## What This Changes

The issue asks whether Telegram DM `direct.<peer>.tools` and `toolsBySender` should enforce per-DM user tool policy, and reports that the accepted config is not reliably applied at runtime.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a rebased direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender` while preserving channel-scoped sender matching and regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Telegram DMs use `telegram:direct` session keys, while `parseRawSessionConversationRef` and `resolveGroupContextFromSessionKey` return no trusted policy scope for non-group/channel keys, so `direct.*.tools` is not read by the shared tool-policy path.

## Solution Assessment

Yes for the related fix direction, but it is not on current main. The maintainable solution is to route the existing Telegram direct config surface through a direct/DM-aware shared policy path, with global/per-agent `tools.toolsBySender` remaining the supported workaround until that lands.
