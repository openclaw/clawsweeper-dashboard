- `impact:security`: The issue concerns authorization boundaries for tool access between different Telegram DM users sharing one agent.

## Summary

Keep open. Current main and the latest release still accept and select Telegram `direct.*.tools` config for DMs, but the shared policy resolver only derives trusted scope from group/channel session keys, so the direct-message authorization gap remains; https://github.com/openclaw/openclaw/pull/53530 is the related open fix path.

## What This Changes

The issue requests confirmation and enforcement of Telegram per-DM-user `direct.<peer>.tools` policy, or schema/docs clarification if that config surface is not intended to be enforceable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender`, preserves sender precedence, and includes security-focused regression plus redacted Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts and selects Telegram direct config, but direct-session keys have kind `direct` and the shared resolver returns no trusted policy context for non-group/channel kinds.

## Solution Assessment

Yes, the right fix direction is to route the accepted Telegram direct config surface through a direct/DM-aware shared policy path. The global `tools.toolsBySender` surface is a useful workaround, but it does not resolve the accepted `direct.*.tools` mismatch.
