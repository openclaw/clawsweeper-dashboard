## Summary

Keep open. Current main still accepts Telegram direct-message tool policy in schema/types, but the runtime policy resolver derives only group/channel context, so Telegram `direct.*.tools` is not enforced through the actual tool filter; the closest related implementation remains an open PR and does not close the Telegram direct surface by itself.

## What This Changes

The issue asks whether Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` should enforce per-DM-user tool policy, and requests either enforcement or config/docs clarification.

## Best Possible Solution

Wire Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` into a direct/DM-aware resolver with account, thread, sender, and wildcard precedence coverage, while keeping schema/docs aligned with enforced behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the config accepts and resolves Telegram direct tool policy, but direct session keys cannot produce trusted policy context in the group-only resolver.

## Solution Assessment

Yes. The narrow maintainable solution is to enforce the already-exposed direct config surface through a direct/DM-aware policy path; a docs-only clarification would leave a misleading, security-relevant accepted config shape.
