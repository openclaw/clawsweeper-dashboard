## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` in config and applies direct-message prompt overrides, but the runtime tool-policy resolver only derives group/channel scope and drops direct-session keys before policy lookup; the related implementation PR remains open and should be reviewed rather than pre-closing this issue.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` should enforce actual per-user tool permissions and, if intended, to fix the enforcement path.

## Best Possible Solution

Wire Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` into a direct/DM-aware policy resolver with account, sender, thread/parent, explicit, and wildcard precedence covered by regression tests.

## Reproduction Assessment

Yes, by source inspection. A Telegram direct session key such as `agent:main:telegram:direct:<peer>` cannot produce trusted tool-policy context even though `channels.telegram.accounts.<account>.direct.<peer>.tools` is accepted and other direct config is applied.

## Solution Assessment

Yes. The narrow maintainable fix is to enforce the already-exposed direct config surface through a direct/DM-aware policy path; a docs-only clarification would leave a misleading security-relevant config shape.
