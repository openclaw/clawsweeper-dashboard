## Summary

Keep open. Current main and the latest release still accept Telegram per-DM tool policy in config while the runtime enforcement path remains group/channel-only, and the linked repair is closed unmerged, so this remains the canonical security-relevant bug.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` and `toolsBySender` should enforce per-DM tool permissions, or whether the accepted config surface should be clarified.

## Best Possible Solution

Add direct-session awareness to the canonical shared tool-policy resolver and Telegram adapter, with regression coverage for account scope, exact and wildcard peers, sender overrides, thread fallback, and group isolation.

## Reproduction Assessment

Yes, at source level: Telegram direct config accepts `tools`, but direct session keys are rejected by the group/channel parser and the remaining enforcement path reads only group policy. I did not run a live Telegram account in this read-only review.

## Solution Assessment

Yes in direction: the narrow maintainable fix is direct-session support in the canonical resolver plus a typed Telegram adaptation, not a one-off bypass. The closed related PR is not a viable landing branch because it was closed unmerged with proof and security concerns.
