## Label Justifications

- `P1`: A configured Telegram DM tool restriction can fail open at runtime, which is an urgent security-sensitive channel workflow bug.
- `impact:security`: The issue concerns tool-authorization boundaries between different Telegram DM users sharing one agent.

## Summary

Keep open. Current main and the latest release still accept Telegram `direct.*.tools` / `toolsBySender`, and Telegram still applies the selected direct config to prompts, but the enforced tool-policy resolver remains group/channel scoped and can miss DM direct policy; https://github.com/openclaw/openclaw/pull/53530 is still open and is the active repair path, not a merged fix.

## What This Changes

The issue requests reliable enforcement or clarification of Telegram per-DM `direct.<peer>.tools` and `toolsBySender` policy that is accepted by schema and config types.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram exact-peer and wildcard `tools` / `toolsBySender`, preserves provider-scoped sender matching, and includes focused security regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. This is source-reproducible on current main and `v2026.5.22`: Telegram accepts and selects direct config, while trusted policy parsing and resolution remain group/channel scoped.

## Solution Assessment

Yes. The maintainable fix is to route the accepted Telegram direct config surface through a shared direct/DM-aware policy path rather than adding a Telegram-only filter or documenting away an advertised schema/type contract.
