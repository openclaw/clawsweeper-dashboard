## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools` / `toolsBySender` and applies the selected direct config for prompts, but the shared tool-policy resolver still only trusts group/channel session scope; https://github.com/openclaw/openclaw/pull/53530 remains the open, conflicting fix path.

## What This Changes

Telegram direct-message config accepts `direct.<peer>.tools`, and the request is to make that accepted per-DM tool policy actually constrain mounted tools.

## Best Possible Solution

Route Telegram direct peer and wildcard config through a direct/DM-aware tool-policy resolver that preserves sender matching; keep global/per-agent `tools.toolsBySender` as the supported workaround until that lands.

## Reproduction Assessment

Yes, source-reproducible. Telegram DMs route as `direct` sessions, while the shared policy parser and resolver only trust `group`/`channel` scope, so `direct.*.tools` is not reached by the mounted-tool policy path.

## Solution Assessment

Yes for the related direction, but it is not landed on current main. The maintainable fix is to adapt the shared resolver for direct/DM policy scope while preserving existing provider-scoped sender matching and trust checks.
