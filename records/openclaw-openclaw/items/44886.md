## Summary

Keep open. Current main still accepts Telegram `direct.*.tools` and selects direct DM config, but the tool-policy resolver only derives policy scope from group/channel session keys, so the accepted per-DM tool restriction can still fail to affect the mounted tools; https://github.com/openclaw/openclaw/pull/53530 remains the related open fix path.

## What This Changes

The issue asks OpenClaw to enforce or clarify Telegram per-DM `direct.<peer>.tools` and `toolsBySender` policy for direct-message sessions.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/53530 with a current-main direct/DM-aware policy resolver that enforces Telegram direct peer and wildcard `tools` / `toolsBySender`, preserves sender precedence, and includes security-focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. This is source-reproducible: current main accepts and selects Telegram direct config, but direct session keys are not accepted by the policy conversation parser and the shared resolver falls back to group-only policy lookup.

## Solution Assessment

Yes. The maintainable fix is to route the accepted Telegram direct config surface through a direct/DM-aware shared policy path rather than adding another Telegram-only tool filter.
