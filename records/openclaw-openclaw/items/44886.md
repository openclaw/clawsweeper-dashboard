## Label Justifications

- `P1`: A schema-accepted per-DM tool deny policy can be silently ineffective in Telegram direct sessions, affecting a security-sensitive channel workflow for real operators.
- `impact:security`: The issue is about a configured tool-permission boundary not being enforced for Telegram DM users.

## Summary

Keep open. Current main and latest release still accept Telegram direct-message tool-policy config, but the agent tool-policy path still resolves only group/channel policy; the linked repair at https://github.com/openclaw/openclaw/pull/53530 is closed unmerged, so the reported security-relevant mismatch remains.

## What This Changes

The issue asks OpenClaw to either enforce Telegram per-DM `direct.<peer>.tools` policy reliably or clarify/remove the accepted config surface.

## Best Possible Solution

Honor Telegram `direct` and `direct["*"]` tool policies through the shared agent tool-policy path with explicit direct-vs-group precedence, account scoping, and regression coverage; only clarify/remove the schema if maintainers reject that config surface.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: direct DM config is accepted and used for prompts, while the runtime tool-policy resolver still reads only group/channel policy and excludes direct session keys.

## Solution Assessment

Is this the best way to solve the issue? Yes, enforcing the existing accepted `direct` tool-policy surface in the shared resolver/Telegram adapter is the narrow maintainable fix; a docs-only/schema rollback would be a fallback only if maintainers decide per-DM enforcement is not intended.
