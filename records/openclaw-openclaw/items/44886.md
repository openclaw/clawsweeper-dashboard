## Summary

Keep open. Current main still accepts Telegram `direct.*.tools` in schema/types, but the runtime tool-policy path remains group/channel-oriented, and the related direct-session resolver PR is still open and explicitly does not claim to close the Telegram `direct` surface yet.

## What This Changes

The issue asks OpenClaw to enforce Telegram per-DM `direct.<peer>.tools` policy, or clarify/remove the accepted config surface if it is not intended.

## Best Possible Solution

Reuse the shared direct/DM policy resolver direction, wire Telegram `direct.<peer>` and wildcard direct config into it with account/thread/sender precedence coverage, and keep schema/docs aligned with the enforced behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: Telegram accepts and resolves `direct.*.tools`, but direct session keys do not produce trusted policy context in `resolveGroupToolPolicy`, so the direct policy is not enforced through the actual tool filter.

## Solution Assessment

Yes. The narrow maintainable fix is to route Telegram direct-session policy through a direct/DM-aware resolver and add focused regression coverage; a docs-only clarification would leave an accepted security-relevant config surface misleadingly present.
