## Summary

Keep open: current main and v2026.6.8 still accept Telegram `direct.*.tools` and `toolsBySender` config while the inspected enforcement path resolves only group/channel-scoped policy, and the related repair PR is closed unmerged.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` is intended to enforce actual per-user tool permissions, and if so to fix the runtime mismatch or clarify the schema/docs.

## Best Possible Solution

Make Telegram `direct.*.tools` and `toolsBySender` either enforce through a direct/DM-aware policy path with focused regression coverage, or retire the accepted schema surface with an explicit migration/docs decision.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main accepts Telegram direct `tools`, selects direct config for DM prompt settings, and then resolves actual tool policy through group/channel-only paths. I did not run a live Telegram DM.

## Solution Assessment

Unclear until maintainers make the security/product decision. If the accepted surface remains supported, a direct/DM-aware resolver with Telegram regression coverage is the narrow maintainable fix; otherwise the schema/docs need retirement and migration handling.
