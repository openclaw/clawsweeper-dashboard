## Summary

Keep open. Current main still exposes Telegram direct `tools`/`toolsBySender` config and applies direct prompt settings, but the shared tool-policy resolver still only trusts group/channel session scopes, so a Telegram direct session cannot reliably enforce per-DM tool restrictions; the related fix PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to confirm and enforce Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` as per-DM user tool policy, or clarify the config surface if unsupported.

## Best Possible Solution

Land or replace the related direct/DM-aware policy resolver so Telegram `direct.<peer>` and wildcard direct `tools`/`toolsBySender` are enforced with account, sender, thread/parent, explicit, and wildcard precedence covered by tests and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A Telegram direct session key such as `agent:main:telegram:direct:<peer>` cannot produce trusted tool-policy context even though `channels.telegram.accounts.<account>.direct.<peer>.tools` is accepted and direct prompt config is applied.

## Solution Assessment

Yes on direction, but not complete on current main. The maintainable fix is to route the already-exposed Telegram direct config surface through a direct/DM-aware policy path rather than documenting away a security-sensitive config mismatch.
