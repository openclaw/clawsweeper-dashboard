## Summary

Keep this issue open: current main now provides canonical `sessionKey` correlation, but it explicitly does not propagate outbound `runId` and still offers no bounded tool-call/result evidence to the pre-delivery hook. The related merged work solved only the session identity subset, while the remaining capability needs a maintainer-owned plugin API and sensitive-data boundary decision.

## What This Changes

Expose stable session/run identity and bounded execution evidence to the pre-delivery outbound message hook so plugins can deterministically rewrite or block unsupported agent claims.

## Best Possible Solution

Add a generic, additive plugin contract that threads the canonical `runId` through every session-backed outbound path and supplies a minimal bounded/redacted evidence view or run-scoped lookup handle, while keeping verifier policy and claim auditing in external plugins.

## Reproduction Assessment

Yes, at the source level. Current main explicitly documents that outbound message hooks lack `runId`, and the outbound event type contains no bounded transcript, tool-call, or tool-result evidence; this is a missing capability rather than a regression requiring runtime reproduction.

## Solution Assessment

The plugin-boundary direction is appropriate, but the proposed full session history in every outbound event is not yet proven to be the safest or narrowest solution. Maintainers should choose between a bounded redacted evidence snapshot and a run-scoped lookup capability, then require consistent coverage across all delivery paths.

## Risks / Open Questions

- Exposing transcript or tool-result material to every plugin registered for outbound hooks could disclose credentials, private conversation content, tool arguments, filesystem paths, or other sensitive run data unless the view is deliberately bounded and redacted.
- Adding a second verification/interception path without defining its relationship to `before_agent_finalize`, `before_tool_call`, `after_tool_call`, and `agent_end` could create conflicting decisions or duplicated sources of truth.
- The contract must propagate the same run identity through shared delivery, direct replies, followups, cron, command, and harness paths; a one-sided implementation could silently mis-associate evidence or omit important sends.
