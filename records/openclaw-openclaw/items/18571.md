## Label Justifications

- `P2`: The request addressed a meaningful deterministic-verification capability but was not an emergency availability or data-loss incident.
- `impact:session-state`: The verification workflow depends on correlating the answer and evidence with the exact session and run.
- `impact:security`: Conversation and tool evidence can contain sensitive data and is protected by an explicit trusted-plugin permission.

## Summary

Current main and stable releases now solve the central verification problem through the safer `before_agent_finalize` lifecycle hook: trusted plugins receive exact run/session correlation, the natural final assistant answer, and the agent message snapshot, then can request a bounded additional model pass before the answer is accepted. This avoids coupling agent transcript evidence to channel-level delivery hooks; the remaining open gateway-send PR addresses a narrower delivery-cancellation workflow rather than this verification capability.

## What This Changes

Provide deterministic pre-delivery verification with stable run/session identity, tool-call/result evidence, and a bounded block-and-retry mechanism.

## Best Possible Solution

Keep deterministic claim verification on the trusted `before_agent_finalize` lifecycle boundary and reserve `message_sending` for channel-level rewrite, cancellation, and delivery policy.

## Reproduction Assessment

Yes. Current source and focused finalization tests provide a high-confidence path where a plugin inspects the natural final answer and agent message snapshot, then requests a bounded revision before acceptance.

## Solution Assessment

No. Adding agent transcript and tool history to a channel delivery hook would duplicate the shipped lifecycle capability and blur the agent-versus-transport ownership boundary; `before_agent_finalize` is the narrower maintainable solution.
