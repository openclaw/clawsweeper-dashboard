## Summary

Keep open: current main and the latest release have a cancellable `message_sending` hook, but shared outbound delivery still passes only routing context, has no outbound run identifier, and exposes no bounded transcript/tool-call/result evidence for deterministic verification. The related open PRs cover slices of session identity, send-path behavior, or agent context, but no merged mainline change implements the full API and security contract.

## What This Changes

Expose session/run identity and bounded tool-call/result context to the outbound `message_sending` hook so plugins can verify, annotate, or block messages before delivery.

## Best Possible Solution

Define one additive, permission-aware `message_sending` contract that consistently carries available session/run identity plus a bounded, redacted recent evidence view across shared outbound and direct reply paths.

## Reproduction Assessment

Yes, at source level: current main and `v2026.5.19` show shared outbound `message_sending` lacks `sessionKey`, `runId`, and verification evidence. This is an API feature gap rather than a failing documented behavior test.

## Solution Assessment

No for exposing raw session tool history exactly as proposed. The safer maintainable solution is a bounded, redacted, permission-aware evidence contract coordinated with the overlapping session-identity, outbound-send, and agent-context work.
