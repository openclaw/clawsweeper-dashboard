## Summary

Keep open: current main has a cancellable `message_sending` hook, but shared outbound delivery still passes only routing context, `OutboundSessionContext` has no `runId`, and there is no bounded transcript/tool-call/result evidence contract for deterministic verification. The related open PRs cover overlapping session-identity slices, not the full plugin hook API and privacy decision requested here.

## What This Changes

Request an outbound pre-delivery message hook/context that exposes outbound text, session/run identity, and bounded verification evidence so plugins can annotate or block unverifiable replies.

## Best Possible Solution

Define one additive, permission-aware plugin hook contract that consistently threads available session/run identity and exposes a bounded, redacted recent verification evidence view across shared outbound and direct reply delivery, with docs and regression tests.

## Reproduction Assessment

Yes for the missing API surface: source inspection on current main shows shared outbound `message_sending` receives only routing context, `OutboundSessionContext` lacks `runId`, and the sending event type lacks transcript/tool-call/tool-result evidence. This is a feature/API gap rather than a broken documented behavior repro.

## Solution Assessment

No for exposing raw session tool history exactly as proposed. The safer maintainable solution is a bounded, redacted, permission-aware evidence contract coordinated with the overlapping session-identity PRs.
