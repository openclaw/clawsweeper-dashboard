## Summary

Keep open: current main still has a cancellable `message_sending` hook, but it does not provide the requested outbound `runId` plus bounded verification evidence, and shared outbound delivery still does not thread `sessionKey`/`runId` into that hook context. Related PRs remain open and cover only parts of the identity or payload-hook problem, so this still needs maintainer API and security design.

## What This Changes

Request to expose outbound message hook session identity, run identity, and verification context so plugins can annotate or block messages before delivery.

## Best Possible Solution

Define one additive, permission-aware message hook contract that consistently threads available session/run identity and exposes a bounded redacted recent evidence view across shared outbound and direct reply paths.

## Reproduction Assessment

Yes, at source level: current main shows shared outbound `message_sending` lacks `sessionKey`, `runId`, and verification evidence, while the latest release has the same gap. This is an API feature gap rather than a failing documented behavior test.

## Solution Assessment

No for exposing raw session tool history exactly as proposed. The safer maintainable solution is a bounded, redacted, permission-aware evidence contract coordinated with the overlapping session-identity and outbound-send work.
