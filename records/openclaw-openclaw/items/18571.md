## Summary

Keep open. Current main has a `message_sending` hook that can rewrite or cancel outbound content, but shared outbound delivery still invokes it with only route fields and there is no bounded transcript/tool-call/result evidence contract for deterministic verification; the related open PRs cover only pieces of the session identity gap.

## What This Changes

The issue requests a pre-delivery outbound message hook with outbound text, routing identity, session/run correlation, recent tool-call evidence, and cancellation support for deterministic verification.

## Best Possible Solution

Define one additive plugin hook contract for outbound verification: thread available `sessionKey` and `runId` through shared outbound delivery and direct reply paths, expose a bounded recent transcript/tool-call/result evidence view with clear privacy limits, then update docs and regression tests around the canonical path.

## Reproduction Assessment

Yes. A source-level reproduction is to inspect or instrument `deliverOutboundPayloads` with a `message_sending` hook: current main calls the hook before delivery, but the hook context lacks `sessionKey`, `runId`, and any verification evidence payload.

## Solution Assessment

No, the current implementation is only a partial foundation. The narrow maintainable solution is an explicit additive hook-context contract plus targeted plumbing/tests, not relying on legacy metadata or unbounded raw transcript access.
