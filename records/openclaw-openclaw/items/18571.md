## Summary

Keep open: current main and v2026.6.5 include the cancellable `message_sending` hook and now thread `sessionKey`, but they still do not expose outbound `runId` or a bounded tool-call/result evidence view needed for deterministic pre-delivery verification.

## What This Changes

The issue requests outbound pre-delivery hook context that lets plugins inspect outgoing text, correlate it to session/run state, and block or annotate messages using recent tool-call/result evidence.

## Best Possible Solution

Define one additive, permission-aware outbound verification contract that threads available run identity and a bounded recent evidence view across shared outbound and direct reply paths, with docs and regression tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 show `message_sending` gets optional `sessionKey`, but not outbound `runId` or bounded tool-call/result evidence. This is an API feature gap rather than a failing documented runtime behavior test.

## Solution Assessment

No, not as raw session tool history exactly as proposed. The safer maintainable solution is a bounded, redacted, permission-aware evidence contract coordinated with the existing hook correlation surfaces.
