## Summary

Keep open: current main and the latest release still have a cancellable `message_sending` hook, but shared outbound delivery passes only channel/account/conversation context, `OutboundSessionContext` has no run identifier, and there is still no bounded transcript/tool-call/result evidence contract. The related open PRs cover identity or send-path slices, not the full API and security decision this request needs.

## What This Changes

Expose session/run identity and bounded verification evidence to outbound `message_sending` so plugins can inspect, annotate, or block outbound messages deterministically before delivery.

## Best Possible Solution

Define one additive, permission-aware `message_sending` contract that consistently carries available session/run identity plus a bounded, redacted recent evidence view across shared outbound and direct reply paths.

## Reproduction Assessment

Yes, at source level: current main and `v2026.5.19` show shared outbound `message_sending` lacks `sessionKey`, `runId`, and verification evidence. This is an API feature gap rather than a failing documented behavior test.

## Solution Assessment

No for exposing raw session tool history exactly as proposed. The safer maintainable solution is a bounded, redacted, permission-aware evidence contract coordinated with the overlapping session-identity and outbound-send work.
