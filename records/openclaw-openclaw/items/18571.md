## Label Justifications

- `P2`: The issue concerns a useful plugin verification capability with limited but real session/security impact.
- `impact:session-state`: The requested verification depends on correlating outbound claims with run/session state and transcript evidence.
- `impact:security`: The issue is about pre-delivery verification/guarding of potentially false or unsafe outbound agent claims.

## Summary

Current main and shipped releases solve the central pre-delivery deterministic verification need through the trusted `before_agent_finalize` lifecycle hook, which is a narrower agent-boundary solution than putting transcript/tool evidence on channel-level `message_sending`.

## What This Changes

The issue requests an outbound pre-delivery message hook context that exposes session/run identity and bounded evidence so verification plugins can inspect, annotate, or block outbound messages deterministically.

## Best Possible Solution

Keep deterministic claim verification on the trusted `before_agent_finalize` lifecycle boundary and leave `message_sending` focused on channel-level rewrite, cancellation, and delivery policy.

## Reproduction Assessment

Yes. Source and focused tests show the shipped path: `before_agent_finalize` receives the final answer plus run/session/message context before delivery and can request a bounded revision.

## Solution Assessment

Yes. The current lifecycle hook is the better owner-boundary solution because it verifies the agent’s natural final answer before acceptance without exposing raw transcript/tool evidence through every channel delivery hook.
