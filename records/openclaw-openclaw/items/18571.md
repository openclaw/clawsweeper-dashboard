## Summary

Keep open. Current main implements the `sessionKey` subset, but outbound `message_sending` still lacks `runId` and any bounded transcript/tool-call/result evidence, so the deterministic pre-delivery verification capability remains a product/security design item.

## What This Changes

Request to expose outbound message hook context with session/run identity and bounded verification evidence so plugins can deterministically inspect, rewrite, or block messages before delivery.

## Best Possible Solution

Keep this open until maintainers choose the canonical verification boundary and define a least-privilege hook payload for run correlation plus bounded evidence, if outbound delivery needs more than `before_agent_finalize`.

## Reproduction Assessment

Yes at source level: current main documents outbound `runId` as unplumbed and `message_sending` has no bounded transcript/tool-call/result evidence payload. This is a capability gap, not a failing existing documented behavior.

## Solution Assessment

No, the standalone evidence-rich `message_sending` design is not yet established as the best solution because `before_agent_finalize` already owns final-answer inspection and bounded revision semantics. Maintainers should decide whether to extend that boundary, add outbound run correlation, or define a separate delivery-policy contract.
