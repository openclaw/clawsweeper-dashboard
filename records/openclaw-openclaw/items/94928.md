## Label Justifications

- `P1`: The PR targets a real agent delivery regression where a generated reply can be hidden by a post-turn compaction failure.
- `merge-risk: 🚨 compatibility`: As written, the catch can change existing no-reply compaction failures from runtime errors to warning-only no-visible-payload success.

## Summary

Keep open with a blocking finding: the PR fixes the reported reply-present path, but the new catch is still broader than that path and can convert no-reply compaction failures into warning-only successful turns.

## What This Changes

The branch wraps post-turn `runCliTurnCompactionLifecycle` in `agentCommandInternal` with a warning-only catch and adds a regression test in compaction-rotation coverage.

## Best Possible Solution

Gate post-turn compaction soft-fail on actual generated or deliverable assistant text, rethrow no-reply compaction failures, and add a no-reply regression beside the new delivery regression.

## Reproduction Assessment

Yes. Source inspection shows current main awaits post-turn CLI compaction before delivery, while this PR catches failures after `persistedCliTurnTranscript`; `persistCliTurnTranscript` can return `persisted` without an assistant reply, and delivery treats empty payloads as successful suppression.

## Solution Assessment

No, not as submitted. The fix is in the right owner path, but it should only soften compaction failure after a generated or deliverable reply exists and should preserve fail-closed behavior when there is no reply.
