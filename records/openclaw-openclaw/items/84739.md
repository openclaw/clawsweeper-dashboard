## Label Justifications

- `P1`: The underlying issue is a security-sensitive provider-key exposure path, but the current PR also risks breaking active provider/model workflows.
- `merge-risk: 🚨 compatibility`: The diff changes the persisted `models.json` shape in a way upstream currently rejects for non-built-in providers with model rows.
- `merge-risk: 🚨 auth-provider`: The change removes provider auth metadata from the registry input without replacing the contract for custom and implicit provider auth.
- `merge-risk: 🚨 availability`: Invalid custom provider entries cause the upstream registry to drop custom models, making configured providers unavailable in registry-backed paths.

## Summary

Keep open: the security goal is valid and source-reproducible, but the patch currently breaks the upstream custom-provider contract for non-built-in providers with model rows and still needs real behavior proof before merge.

## What This Changes

The PR strips provider-level `apiKey` fields before writing generated `models.json` and updates agents model-config tests to expect omitted keys.

## Best Possible Solution

Keep real provider credentials out of persisted `models.json` while preserving the upstream registry contract for non-built-in providers with model rows.

## Reproduction Assessment

Yes, from source inspection: current main stringifies `finalProviders` into `models.json`, and existing tests assert provider `apiKey` markers are present. I did not run tests because this review is read-only.

## Solution Assessment

No, not as proposed: unconditionally deleting `apiKey` is the right security direction but violates the upstream custom-provider schema. The safer fix is to keep secrets out of the file while preserving registry-compatible auth metadata or an in-memory replacement path.
