## Summary

Keep open; this targets a real provider-secret exposure path, but the current diff breaks the Pi registry contract for non-built-in providers with model rows and still lacks real behavior proof.

## What This Changes

The branch strips provider-level `apiKey` before writing generated `models.json` and updates agents model-config tests to expect omitted keys.

## Best Possible Solution

Persist secret-free provider auth metadata or adapt the registry input so custom provider model rows remain valid, then land one canonical Layer 1 fix with redacted real-run proof.

## Reproduction Assessment

Yes, from source inspection: current main serializes `finalProviders` into `models.json`, and current tests assert provider `apiKey` markers are persisted. I did not run the Vitest lane because this review is read-only.

## Solution Assessment

No, not as proposed: stripping real credentials is the right direction, but unconditionally deleting `apiKey` violates the upstream registry contract for non-built-in providers with model rows. The safer path is a secret-free marker or adapter that keeps those rows valid.
