## What This Changes

The issue reports duplicate embedded PI run cleanup logging `reason=handle_mismatch` after terminal lifecycle cleanup clears the same run before the attempt `finally` cleanup.

## Best Possible Solution

Review and land the linked PR if its proof and regression coverage are acceptable, or apply the same narrow idempotent missing-handle clear in the run registry while preserving replacement-handle mismatch diagnostics.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: terminal lifecycle cleanup clears the active handle, then the attempt `finally` block clears the same handle again and reaches the mismatch branch because the active map entry is already gone. I did not run the live embedded-agent flow in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to make `clearActiveEmbeddedRun` treat the no-active-run case as idempotent while still logging `handle_mismatch` when a different active handle owns the session.
