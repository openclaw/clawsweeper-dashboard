## Summary

Keep open: the PR addresses a real pending-final recovery gap, but the new direct replay path can fall back to stale session route fields and send a frozen final reply to the wrong chat before clearing the durable pending state. It also lacks external real behavior proof.

## What This Changes

The PR adds direct restart recovery for terminal main sessions with `pendingFinalDelivery=true` by sending the captured final text through `message.action` and clearing pending-final fields after success.

## Best Possible Solution

Replay terminal pending finals only through an exact saved pending/restart recovery delivery context, leave contextless finals pending or mark them diagnostic-failed without sending, and require focused regression coverage plus redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level: on the PR branch, a terminal session with `pendingFinalDeliveryText` but no `pendingFinalDeliveryContext` and stale `lastChannel`/`lastTo` will enter the new non-running pending-final branch and send via the legacy session fallback. I did not run tests because this review is read-only and test execution would write artifacts.

## Solution Assessment

No. Direct terminal replay is a plausible owner-boundary fix, but the best version must require the frozen delivery context instead of falling back to broad session route metadata.
