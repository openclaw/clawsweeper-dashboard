## Summary

Keep open: current main still keeps `apply_patch` workspace-contained by default, media/PDF local reads narrow to workspace roots when `tools.fs.workspaceOnly` is active, and the related open PRs are partial or blocked rather than merged fixes for the reported v2026.6.6 regression.

## What This Changes

The issue asks OpenClaw to restore or explicitly configure outside-workspace filesystem access for `apply_patch`, write/edit, and local media verification paths such as `/tmp` and Desktop output directories.

## Best Possible Solution

Choose one maintainer-approved filesystem policy for trusted outside-workspace roots, then align write/edit/apply_patch and local media readback behavior with docs, upgrade notes, and Codex permission-profile mapping.

## Reproduction Assessment

Yes, partially by source inspection: `apply_patch` remains workspace-contained by default and media/PDF local roots narrow under workspace-only policy. The exact v2026.6.6 `write` failure still needs a live setup/config repro because host write is broad when workspace-only is unset.

## Solution Assessment

Unclear as a direct fix: restoring broad `/tmp` access by default may weaken secure defaults, while adding trusted roots or a broader `tools.fs` policy needs maintainer security/product approval. The narrowest maintainable path is to align the tool surfaces under an explicit trusted-root contract.
