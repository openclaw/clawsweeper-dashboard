## Summary

Keep open: current main and latest checked release still make `apply_patch` workspace-contained by default, while host `write` remains unrestricted when `tools.fs.workspaceOnly` is false, so the remaining problem is a security-sensitive compatibility/product decision rather than an already-fixed cleanup item.

## What This Changes

The issue reports a v2026.6.6 regression where `write` and `apply_patch` fail for `/tmp` or other absolute paths outside the agent workspace.

## Best Possible Solution

Preserve secure defaults and add or approve a clearly documented trusted outside-workspace path policy only after maintainer/security review and live upgrade proof.

## Reproduction Assessment

Yes, partially by source inspection: current main and v2026.6.8 keep apply_patch workspace-only by default; the write failure needs live runtime/config proof because host write remains unrestricted when filesystem workspace-only is false or unset.

## Solution Assessment

Unclear: unconditional `/tmp` restoration is not clearly the best fix because it weakens a documented boundary; maintainers should choose between explicit trusted roots, existing opt-out flags, or a compatibility mode with security and upgrade proof.
