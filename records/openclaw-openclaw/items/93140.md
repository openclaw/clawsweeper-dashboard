## Summary

Keep open: current main and v2026.6.6 intentionally keep `apply_patch` workspace-contained by default, while host `write` still appears intended to allow absolute paths when `tools.fs.workspaceOnly` is false or unset. The reported packaged `write` failure therefore needs maintainer reproduction with the reporter's effective config, tool profile, and sandbox mode before choosing a security-sensitive fix.

## What This Changes

The issue asks OpenClaw to restore or explicitly configure file-tool writes to `/tmp` and other absolute paths after a reported v2026.6.6 regression.

## Best Possible Solution

Keep the issue open, reproduce the v2026.6.6 host `write` failure with the reporter's effective config and sandbox mode, and route any new path allowlist design through the existing path-scoped permissions discussion.

## Reproduction Assessment

No, not end to end. Source inspection proves `apply_patch` defaults to workspace-only and `write` should allow host absolute paths when `tools.fs.workspaceOnly` is false or unset, but the reported packaged v2026.6.6 `write` failure still needs a concrete config/sandbox reproduction.

## Solution Assessment

Unclear. A narrow bug fix may be needed for `write` if the packaged release blocks `/tmp`, but restoring default `apply_patch` outside the workspace or adding `allowedPaths` is a security/product decision best coordinated with the existing path-scoped permissions design.
