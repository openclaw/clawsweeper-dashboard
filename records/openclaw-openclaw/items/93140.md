## Summary

Keep open: current main and v2026.6.8 do not implement the requested `/tmp`/outside-workspace compatibility path or a trusted `allowedPaths` config surface, and the open PathGuard PR is overlapping but not merged or safe to treat as the fix.

## What This Changes

The issue asks OpenClaw to restore `/tmp` and absolute-path writes for `write`/`apply_patch`, or add an explicit filesystem allowlist config for those paths.

## Best Possible Solution

Keep this open until maintainers choose a compatibility/security policy for `/tmp` and other trusted roots, then implement it with docs, upgrade guidance, and coverage across OpenClaw host tools, sandboxed tools, and Codex-native paths.

## Reproduction Assessment

Partially yes: current source and tests show `apply_patch` is workspace-only by default and only an explicit false opt-out restores outside-workspace patching. The `write` failure still needs a live config/runtime repro because host write remains unrestricted when `tools.fs.workspaceOnly` is false or unset.

## Solution Assessment

Unclear: unconditionally restoring `/tmp` by default is not clearly the best fix because it may weaken sandbox expectations. The safer path is a maintainer-approved trusted-root policy or compatibility mode with explicit docs and upgrade proof.
