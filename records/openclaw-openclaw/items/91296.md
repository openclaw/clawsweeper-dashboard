## Label Justifications

- `P1`: The PR targets and currently fails to fully close a live gateway self-update availability hazard for supervised git installs.
- `merge-risk: 🚨 availability`: The merged behavior could still rebuild `dist/` while a supervisor-restarted gateway is serving requests.
- `merge-risk: 🚨 compatibility`: The PR changes supervised git `update.run` from an in-process update result to a managed-handoff skipped response without matching docs.

## Summary

Keep open: the patch addresses the right gateway self-update hazard, but the current handoff still runs git updates through a CLI path that does not stop or suppress the managed service before rebuilding `dist/`, and the new supervised-git response shape is not documented.

## What This Changes

The PR routes supervised git-source `update.run` calls through the existing managed-service handoff path while keeping unsupervised git updates in process and adding focused regression coverage.

## Best Possible Solution

Make supervised git updates own the managed service lifetime through git rebuild, post-core convergence, sentinel write, restart, and health verification, then document the supervised-git handoff response.

## Reproduction Assessment

No live current-main reproduction was run in this read-only review. Source inspection gives a clear reproduction path: supervised git updates enter a handoff, the helper waits for parent exit, and the CLI git branch rebuilds without the package-only service stop.

## Solution Assessment

No; routing supervised git through the existing handoff is only a partial fix because the CLI path still does not stop or suppress the managed service for git rebuilds. The safer fix is to make the handoff own service lifetime through rebuild and health verification, then update docs.

- **[P1] Stop the restarted gateway before git handoff updates:** `src/gateway/server-methods/update.ts:148-151`
  - body: When this branch sends supervised git installs through the managed handoff, the helper waits for the parent to exit and then runs the normal CLI update. Update the docs once the final git handoff behavior is correct.
