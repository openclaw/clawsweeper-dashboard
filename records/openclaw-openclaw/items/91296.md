## Summary

Keep open for maintainer review: the patch appears correct and has strong live proof, but it intentionally changes the supervised git `update.run` response contract and managed-service lifecycle, so the remaining decision is compatibility and availability acceptance rather than automated repair.

## What This Changes

The PR routes supervised git-source Gateway `update.run` calls through the managed-service handoff path, expands CLI stop/restart handling to git updates, and updates focused tests and docs.

## Best Possible Solution

Merge after update/gateway owners accept the documented handoff response contract and service lifecycle change; otherwise revise the patch to preserve compatibility for callers that require the old in-process git result.

## Reproduction Assessment

Yes at source level: current main and v2026.6.1 route git installs through the live `runGatewayUpdate` path while package-manager installs use managed handoff. I did not independently reproduce the transient module-resolution failure in a live current-main service.

## Solution Assessment

Yes, if maintainers accept the contract change: reusing the existing managed-service handoff boundary is the cleanest fix shape, and an in-process guard would leave the supervisor relaunch window harder to reason about.
