## Summary

Keep this PR open for normal maintainer merge handling: current main still carries the stale Docker image artifact path, while the branch removes it in the runtime-assets stage with no blocking code-review findings. The remaining concern is intentional compatibility risk for downstream images that depended on private `/app/node_modules` paths.

## What This Changes

The PR updates `Dockerfile` to remove the nested stale `openclaw` package, its `.bin` shim, and its pnpm virtual-store package from the runtime-assets stage before the final image copies `node_modules`.

## Best Possible Solution

Keep the cleanup in the Docker image staging path and treat the stale nested package path as unsupported image debris while preserving the documented launcher path.

## Reproduction Assessment

Yes from source and reporter/PR Docker commands, though I did not run Docker in this read-only review. Current main copies `/app/node_modules` from `runtime-assets`, and current extension manifests plus `pnpm-lock.yaml` include the stale `openclaw@2026.5.28` dependency.

## Solution Assessment

Yes. Pruning the stale package and shim in `runtime-assets` is the narrowest Docker-image fix because it removes the artifact before the final `COPY` without changing extension package contracts or plugin dependency declarations.
