## Summary

Keep open: current main still lacks the requested generic configurable context-file elevation and Project Context exclusion surface. Codex-specific bootstrap routing partially overlaps the use case, but the default renderer, config schema, tests, and docs still preserve the flat Project Context model, and the linked implementation PR is closed unmerged.

## What This Changes

The issue requests configurable promotion of SOUL.md, IDENTITY.md, USER.md, and MEMORY.md into named prompt sections before Project Context, with matched files optionally excluded from the flat context block.

## Best Possible Solution

Maintainers should choose the supported prompt/context customization boundary, then implement deterministic per-file section mapping and Project Context exclusion through that boundary with focused schema, docs, and tests.

## Reproduction Assessment

Yes at source level: current main still renders stable context files through `buildProjectContextSection` under `# Project Context` and lacks the requested config/API surface. I did not run a live model session because this is a prompt-shape feature request rather than a deterministic runtime failure.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. A typed core config option is plausible, but a context-engine or plugin prompt contract may be safer if this should not become fixed core prompt policy.
