## Summary

Current main still lacks the requested configurable context-file elevation surface: bootstrap files are ordered but rendered as flat Project Context entries, and the issue remains a prompt/config product-boundary decision rather than cleanup-close material.

## What This Changes

The issue requests configurable elevation of selected bootstrap/context files such as SOUL.md, IDENTITY.md, USER.md, and MEMORY.md into dedicated prompt sections with optional exclusion from flat Project Context.

## Best Possible Solution

Choose one maintainer-approved prompt/context extension point, then implement deterministic per-file elevation and Project Context exclusion as either documented core config or a plugin/context-engine contract with focused prompt, config, and docs coverage.

## Reproduction Assessment

Yes for source behavior: current main can be inspected to show bootstrap files are sorted and rendered as flat Project Context entries, with tests asserting that flat rendering. There is no high-confidence runtime reproduction for the model-persona degradation itself because the requested change is a new configurable prompt capability.

## Solution Assessment

Unclear until maintainers choose the product boundary. The proposed `context.elevation` config is plausible, but VISION.md and AGENTS.md both push optional or policy-heavy behavior toward lean core seams and plugin/API discussions when current extension points are insufficient.
