## Summary

Keep open. Current main still sends image-tool inputs through `loadWebMedia()` optimization before the VLM call, optimizer/backend failures still propagate, and open PR #73451 uses closing syntax for this issue.

## What This Changes

The issue asks for JPEG/PNG image-tool inputs to reach the configured vision model instead of failing in OpenClaw's preprocessing pipeline.

## Best Possible Solution

Repair the `media-understanding-core` image runtime dependency path and add only a guarded loader-level fallback if maintainers accept preserving under-cap originals.

## Reproduction Assessment

Yes. Calling the image tool or `loadWebMedia()` on a valid image in a packaged install where `sharp` is unavailable remains a high-confidence path: current source still optimizes before model invocation and propagates optimizer failures.

## Solution Assessment

No. Current main is not the final fix, and PR #73451's direct optimizer fallback is less safe than fixing runtime dependency availability plus any fallback at the media-loader boundary.
