## Summary

Keep open. Current main improves the missing-runtime-dependency path and error detail, but the shared media loader still has no guard-aware original-image fallback when optimization itself throws, and the issue is already paired with open closing PR #74243 plus same-author PR #73451.

## What This Changes

The issue reports that image preprocessing fails with `Failed to optimize image` before configured vision providers receive valid JPEG/PNG media from LINE and other channels.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that returns the original image only when it is already valid and under cap, preserves original MIME/content type, and keeps HEIC conversion, decoded-pixel, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments provide concrete JPEG/PNG failures across LINE, WhatsApp, Telegram, macOS, and Linux, and current source confirms the image tool still reaches `loadWebMedia(...)` and then `optimizeImageToJpeg(...)` before any provider call.

## Solution Assessment

Yes, the guard-aware loader-level fallback proposed by #74243 appears to be the best narrow solution. Returning original bytes directly from `optimizeImageToJpeg(...)`, as in #73451, is riskier because it can mislabel MIME/content type and weaken guard boundaries.
