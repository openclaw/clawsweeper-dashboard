## Summary

Keep open. Current main mitigates the missing `sharp` packaged-runtime path, but the shared media loader still has no guard-aware fallback for valid under-cap originals when optimization itself throws, and this issue is explicitly paired with open closing PR #74243.

## What This Changes

The issue reports valid JPEG/PNG media failing before VLM/provider calls with `Failed to optimize image` in the shared media preprocessing path.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap original images, preserves the original MIME/content type, and keeps HEIC conversion, decoded-pixel limits, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG reproductions, and current source still has a path where optimizer failures propagate from `loadWebMedia()` before the provider call.

## Solution Assessment

Yes. The linked loader-level fallback is the narrowest maintainable direction because it can preserve MIME and safety boundaries; returning original bytes directly from `optimizeImageToJpeg()` would be broader and risks mislabeling non-JPEG input.
