## Summary

Keep open. Current main partially mitigates the packaged missing-sharp path by staging plugin runtime deps for `media-understanding-core`, but the shared media loader still has no guard-aware original-image fallback when optimization itself throws, and this issue is explicitly paired with open closing PR #74243.

## What This Changes

The issue requests that valid JPEG images sent through the image/media-understanding pipeline reach the configured VLM instead of failing inside OpenClaw's image optimization preprocessing.

## Best Possible Solution

Review and land #74243, or an equivalent `src/media/web-media.ts` change, so valid original images under the media cap survive optimizer backend failures while HEIC conversion, decoded-pixel limits, unknown dimensions, unsafe paths, and over-cap files continue to fail closed.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, macOS, and Linux, and current source shows optimizer failures can still propagate before the provider call; the exact missing-sharp packaging path is partly mitigated on main.

## Solution Assessment

Yes. A loader-level, guard-aware fallback is the best maintainable direction because it can preserve the original MIME/content type and safety checks; returning original bytes directly from `optimizeImageToJpeg()` would be broader and could mislabel non-JPEG input.
