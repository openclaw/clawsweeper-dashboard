## Summary

Keep open. Current main has a partial in-limit missing-optimizer fallback, but the packaged/global-install `sharp` availability gap remains open and this issue is still paired with the same author's open fix PR using closing syntax.

## What This Changes

The issue asks for JPEG/PNG media sent through channels such as LINE to reach the configured VLM instead of failing in OpenClaw image preprocessing with `Failed to optimize image`.

## Best Possible Solution

Resolve or supersede https://github.com/openclaw/openclaw/pull/73451, keep the current guarded loader fallback, and land or supersede the root `sharp` package-dependency fix tracked by https://github.com/openclaw/openclaw/issues/77760.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a packaged/global install without root `sharp` reaches bundled `media-understanding-core/image-ops.js`, which dynamically imports `sharp` before the VLM call; issue comments report the same failure across LINE, WhatsApp, Telegram, macOS, and Linux.

## Solution Assessment

Yes for the current maintenance direction, but not via the open paired PR as written. The safer solution is the already-merged guarded loader fallback plus a root package dependency fix, rather than returning raw bytes directly from `optimizeImageToJpeg` and risking MIME/cap/HEIC boundary drift.
