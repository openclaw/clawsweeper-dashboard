## Summary

Keep open. Current main still routes image-tool file inputs through shared media optimization before the provider call, and optimizer failures can still propagate as `Failed to optimize image`; open PR #74243 explicitly closes this issue and should remain the active implementation path until it lands.

## What This Changes

The issue requests that valid image-tool inputs not fail in shared preprocessing with `Failed to optimize image` before the configured VLM provider is called.

## Best Possible Solution

Land #74243 or an equivalent guarded `src/media/web-media.ts` fallback, then close this issue once the merged code preserves MIME type and keeps HEIC, pixel-limit, unsafe, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments provide concrete LINE, WhatsApp, Telegram, and macOS/Linux reproductions, and current source shows the same pre-provider `loadWebMedia()` path can still throw from optimization.

## Solution Assessment

Yes. A loader-level fallback is the narrowest maintainable direction because it can return valid under-cap originals without relabeling non-JPEG bytes or weakening image safety guards.
