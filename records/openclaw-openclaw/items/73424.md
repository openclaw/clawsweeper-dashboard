## Summary

Keep open. Current main has improved the missing-runtime-dependency and diagnostic paths, but the shared `loadWebMedia` image path still has no guard-aware fallback when optimization itself throws, and open PR #74243 explicitly closes this issue with that remaining implementation.

## What This Changes

The issue requests that valid JPEG images sent through channel/media preprocessing reach the configured image model instead of failing with `Failed to optimize image` before the provider call.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back to the original image only when it is already valid and under cap, while keeping guard failures closed and preserving original MIME/content type.

## Reproduction Assessment

Yes. The comments provide concrete JPEG/PNG reproductions across multiple channels and platforms, and current source shows optimizer failures still propagate from `loadWebMedia()` before the image provider call.

## Solution Assessment

Yes. The loader-level fallback proposed by #74243 is the narrowest maintainable direction because it can preserve MIME/type information and safety guards; returning original bytes directly from `optimizeImageToJpeg()` would be broader and risks mislabeling non-JPEG inputs.
