## What This Changes

The issue reports `Failed to optimize image` during JPEG/media preprocessing before VLM model calls, originally seen with LINE images and later confirmed across shared media paths.

## Best Possible Solution

Keep the safe loader-boundary fallback, resolve or supersede the paired PR, and land a canonical root package dependency fix for `sharp` before closing this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a package/global install without root `sharp` can reach `media-understanding-core/image-ops.ts`, which imports `sharp`, and current fallback only covers in-limit non-HEIC media-loader cases.

## Solution Assessment

Yes for the remaining direction, but not via the open broad passthrough PR. The safest solution is the existing loader-boundary fallback plus a root package `sharp` dependency fix; `optimizeImageToJpeg` should not return raw original bytes as JPEG without loader policy context.
