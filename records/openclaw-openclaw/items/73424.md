## Summary

Keep open. Current main still optimizes image-tool inputs through `loadWebMedia()` before the provider call, optimizer failures still propagate from `optimizeImageToJpeg()`, and the issue discussion has multiple channel-level reproductions tied to missing or broken `sharp` runtime availability. The paired open PR #73451 still references this issue with closing syntax, so this should stay open until a safe implementation lands or that PR is superseded.

## What This Changes

The issue requests fixing the image tool preprocessing path so valid JPEG/PNG media from channels reaches the configured VLM instead of failing with `Failed to optimize image`.

## Best Possible Solution

Land a guarded fix that reliably stages/loads `media-understanding-core` image runtime dependencies and, if retaining fallback behavior, does it at the media-loader boundary while preserving MIME type and fail-closed safety guards.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG failures across LINE, WhatsApp, Telegram, and Discord, and current source still shows optimizer/backend exceptions propagating before the provider call.

## Solution Assessment

No. Current main is not the final fix, and the open PR's optimizer-level fallback is less safe than a loader/runtime-dependency fix that preserves MIME type and existing guard boundaries.
