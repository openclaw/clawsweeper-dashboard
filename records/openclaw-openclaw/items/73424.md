## Summary

Keep open. Current main still routes image-tool and channel media through shared image optimization with no guard-aware fallback when optimization throws; the linked open PR #74243 is the active implementation candidate and explicitly closes this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports valid JPEG image analysis failing before the VLM call because shared image preprocessing throws `Failed to optimize image`.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back to the original image only after validating it is an image and under the active cap, preserves the original MIME/content type, keeps existing safety guards fail-closed, and adds regression coverage for optimizer backend failures.

## Reproduction Assessment

Yes. The issue comments provide concrete JPEG/PNG reproductions across several channels, and current source inspection shows optimizer failures still propagate from `loadWebMedia()` before the image provider call.

## Solution Assessment

Yes. A loader-level fallback like #74243 is the narrowest maintainable solution because it can preserve MIME/type metadata and resource guards; returning original bytes directly from `optimizeImageToJpeg()` would be broader and could mislabel non-JPEG inputs.
