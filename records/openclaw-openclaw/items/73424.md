## Summary

Keep open. Current main mitigates the missing `sharp` runtime-dependency path, but the shared media loader still has no guard-aware fallback when optimization itself throws, and open PR #74243 explicitly closes this issue, so the issue should remain open until that PR lands or is superseded.

## What This Changes

The issue asks for the built-in image tool and shared media preprocessing to stop failing with `Failed to optimize image` for valid JPEG/PNG inputs before the VLM provider call.

## Best Possible Solution

Keep the issue open and land #74243, or an equivalent `src/media/web-media.ts` change, so optimizer backend failures fall back only for valid under-cap original images while preserving MIME type and keeping HEIC conversion, decoded-pixel, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, Discord, macOS, and Linux, and current source shows optimizer failures can still propagate from `loadWebMedia()` before the provider call.

## Solution Assessment

Yes. The linked loader-level fallback direction is the narrowest maintainable fix because it can preserve MIME/content type and safety guards; returning original bytes directly from `optimizeImageToJpeg()` would be broader and can mislabel non-JPEG fallbacks.
