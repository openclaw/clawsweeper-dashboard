## Summary

Keep open. Current main mitigates the missing `sharp` runtime-dependency path, but the shared `loadWebMedia` image path still has no guard-aware fallback when optimization itself throws, and this issue is explicitly paired with open closing PR #74243.

## What This Changes

The issue requests that valid JPEG/PNG images sent through channels such as LINE, WhatsApp, Telegram, or Discord reach the image/VLM pipeline instead of failing during OpenClaw image preprocessing with `Failed to optimize image`.

## Best Possible Solution

Review and land #74243, or an equivalent `src/media/web-media.ts` change, so optimization backend failures fall back only for valid under-cap original images while preserving MIME type and keeping HEIC conversion, decoded-pixel limits, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, Discord, macOS, and Linux, and current source shows optimizer failures can still propagate from `loadWebMedia()` before the provider call.

## Solution Assessment

Yes. The guard-aware loader-level fallback proposed by #74243 is the narrowest maintainable fix because it can preserve MIME/content type and safety boundaries; returning original bytes directly from `optimizeImageToJpeg()` would be broader and risks mislabeling non-JPEG inputs.
