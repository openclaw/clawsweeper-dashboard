## Summary

Keep open. Current main still lets image optimizer failures propagate through `loadWebMedia(...)` before the provider call, and the issue is paired with open PR #73451, so this should remain tracked until that PR is reviewed, landed, or superseded.

## What This Changes

The issue asks OpenClaw to stop failing valid JPEG/PNG image-tool inputs with `Failed to optimize image` during shared media preprocessing before the configured vision model runs.

## Best Possible Solution

Review PR #73451 or replace it with a guard-aware loader-level fallback that preserves the original MIME type and only returns the original image when it is valid, under cap, and not blocked by HEIC, pixel-limit, or unsafe-dimension guards.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG failures across image-tool and channel paths, and current source still shows the image tool calling `loadWebMedia(...)` before the provider while optimizer errors can propagate as `Failed to optimize image`.

## Solution Assessment

No current-main fix fully solves the requested behavior. The narrow maintainable direction is the loader-level fallback described in the discussion, while the existing open PR needs review or supersession before the issue can close.
