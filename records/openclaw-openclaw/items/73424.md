## Summary

Keep open. Current main still runs image-tool inputs through `loadWebMedia()` before the VLM call and still lets optimizer failures throw from `optimizeImageToJpeg()`, while open PR #73451 is explicitly paired as the implementation candidate.

## What This Changes

The issue reports that valid JPEG/PNG images fail before the VLM call with `Failed to optimize image` in shared image preprocessing.

## Best Possible Solution

Review or supersede #73451 with a loader-level fallback that preserves MIME/content type and keeps HEIC conversion, pixel-limit, unsafe, and over-cap failures closed.

## Reproduction Assessment

Yes. The report and comments give concrete JPEG/PNG failures across LINE, WhatsApp, Telegram, and macOS/Linux installs, and current source shows a high-confidence forced path by making the optimizer backend fail before the provider call.

## Solution Assessment

No. Current main is not the final fix; a fallback inside `loadWebMedia()`/`optimizeAndClampImage()` is safer than returning original bytes from `optimizeImageToJpeg()` because it can preserve MIME and enforce existing safety guards.
