## Summary

Keep open. Current main has improved diagnostics and the runtime-dependency path, but the shared image preprocessing path can still propagate optimizer failures before the image provider call, and open PR #74243 explicitly closes this issue with the remaining guard-aware fallback work.

## What This Changes

The issue asks for valid JPEG/PNG images sent through channel media to survive the built-in image tool preprocessing pipeline instead of failing with `Failed to optimize image` before the VLM request.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back to the original image only when it is valid and under the applicable cap, while preserving MIME/content type and keeping HEIC conversion, pixel-limit, unknown-dimension, and over-cap guard failures closed.

## Reproduction Assessment

Yes. The issue discussion provides concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, macOS, and Linux, and current source shows optimizer failures can still propagate from `loadWebMedia()` before the image provider call.

## Solution Assessment

Yes. The loader-level fallback direction in #74243 is the narrowest maintainable solution because it can preserve original MIME/type metadata and media safety guards; returning original bytes directly from `optimizeImageToJpeg()` would be broader and risks mislabeling non-JPEG inputs.
