## Summary

Keep open. Current main still routes image-tool inputs through shared `loadWebMedia()` optimization before the provider call and still propagates optimizer failures; the open paired PR #73451 remains the implementation candidate to review, replace, or close.

## What This Changes

The issue requests that valid JPEG images sent through LINE reach the configured VLM instead of failing in OpenClaw's internal image preprocessing step.

## Best Possible Solution

Review, land, or supersede #73451 with a guard-aware `loadWebMedia()` fallback that preserves MIME/content type and keeps HEIC, pixel-limit, unsafe, and over-cap failures closed.

## Reproduction Assessment

Yes. The report and comments provide concrete JPEG/PNG failures across channels, and current source still shows optimizer exceptions propagating before the provider call when the backend cannot produce an image.

## Solution Assessment

No. Current main is not the final fix; a loader-level fallback is safer than returning original bytes from `optimizeImageToJpeg()` because it can preserve MIME and enforce the existing safety guards.
