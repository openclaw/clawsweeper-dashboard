## Summary

Keep open. Current main still sends non-data image-tool inputs through shared media preprocessing before the provider call, optimizer failures still propagate as `Failed to optimize image`, and the paired open PR #73451 references this issue with closing syntax.

## What This Changes

The issue requests that valid JPEG/PNG images sent through LINE and other channels reach the configured VLM instead of failing in OpenClaw's shared image optimization preprocessing.

## Best Possible Solution

Land or supersede #73451 with a guarded fix that reliably provides `media-understanding-core` image runtime dependencies and only falls back for safe under-cap originals while preserving MIME type and pixel/HEIC guards.

## Reproduction Assessment

Yes. The current source path is source-reproducible: valid image inputs reach `loadWebMedia()` before the provider call, and optimizer backend failures still propagate as `Failed to optimize image`; the issue discussion supplies concrete JPEG/PNG failures across several channels.

## Solution Assessment

No. Current main is not the final fix, and #73451's optimizer-level original-buffer fallback needs maintainer review because a loader/runtime-dependency fix with MIME preservation and fail-closed guards is safer.
