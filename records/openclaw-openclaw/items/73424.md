## Summary

Keep open. Current main mitigates the missing `sharp` runtime-dependency path, but the shared media loader still has no guard-aware fallback for valid under-cap images when optimization itself throws, and this issue is explicitly paired with open closing PR #74243.

## What This Changes

The issue reports that shared image preprocessing fails with `Failed to optimize image` before vision-model calls for valid JPEG/PNG media.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap original images, preserves the original MIME/content type, and keeps HEIC conversion, decoded-pixel limits, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, macOS, and Linux, and current source shows optimizer failures still propagate from `loadWebMedia()` before the provider call.

## Solution Assessment

Yes. The guard-aware loader-level fallback proposed by #74243 is the narrowest maintainable fix because it can preserve MIME and safety boundaries; returning original bytes directly from `optimizeImageToJpeg()` would be broader and risks mislabeling non-JPEG input.
