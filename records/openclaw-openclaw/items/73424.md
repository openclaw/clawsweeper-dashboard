## Summary

Keep open. Current main still sends image-tool inputs through shared media optimization without a valid-under-cap original-image fallback, and open PR #74243 explicitly closes this issue with the remaining implementation.

## What This Changes

Fix the built-in image tool/shared media preprocessing path so valid JPEG images do not fail with `Failed to optimize image` before the VLM call.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` fallback that only returns the original image when it is valid, under cap, and not a guard failure, while preserving original MIME/content type.

## Reproduction Assessment

Yes. The issue comments provide concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, and macOS/Linux, and current source shows optimizer exceptions still propagate through `loadWebMedia()` before the provider call.

## Solution Assessment

Yes. A loader-level fallback is the narrowest maintainable direction because it can preserve content type and keep HEIC, pixel-limit, unsafe, and over-cap failures closed; returning originals directly from `optimizeImageToJpeg()` would be broader.
