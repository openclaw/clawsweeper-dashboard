## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use a private gate for soul_update exposure:** `src/agents/embedded-agent-runner/run/attempt.ts:1347`
  - body: `inputProvenance.sourceTool` is accepted by the gateway protocol and normalized from request input, so using it as the only signal for `enableSoulUpdateTool` lets a caller mark a normal run as `soul_reflection` and expose the persistent SOUL.md write tool. Gate this with a private internal runner option or trusted handoff state that external requests cannot set.
  - confidence: 0.89
- **[P1] Run reflection only after delivery policy allows the turn:** `src/agents/agent-command.ts:899-900`
  - body: This hook fires before the existing `opts.deliver` send-policy check, so a direct delivered turn that should be rejected can still run a reflection sub-turn and append to SOUL.md first. Move reflection behind the same policy/validation gate, or make the reflection path share the deny decision before any model call or write.
  - confidence: 0.86
- **[P1] Deliver SOUL notices through the originating channel:** `src/agents/agent-command.ts:909`
  - body: The forced notice after an append is only `runtime.log(...)`, which reaches process output rather than Discord, Telegram, Slack, or HTTP callers. Either gate auto-update to local/TUI surfaces for now or send a portable channel-visible notice before allowing channel-triggered persistent writes.
  - confidence: 0.88
- **[P2] Keep SOUL read errors from aborting the main turn:** `src/agents/soul-reflection-runner.ts:64`
  - body: The before/after `readLastSoulRule()` calls sit outside the try/catch that is supposed to swallow reflection failures, and the helper can throw on ordinary read errors. Wrap those reads in the same containment path or make the helper return `null` on read failure so auto-update cannot reject the user's main turn.
  - confidence: 0.82
