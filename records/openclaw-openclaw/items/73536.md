## Summary

Keep open. Current main still omits `RequestOptions` on bundled MCP `callTool()` calls, and this PR now fixes the code path in the right direction while preserving the SDK's 60s default for unconfigured servers; however, it is not merge-ready because the public docs and required changelog entry are still missing, and exact-head CI is not green.

## What This Changes

The PR adds a resolved bundled-MCP `requestTimeoutMs` with a 60s SDK-default fallback, propagates it through MCP transport/session state, passes it to `Client.callTool()`, and adds transport-config regression tests.

## Best Possible Solution

Land this PR or an equivalent current-main patch after adding the public MCP documentation and changelog entry, preserving the 60s request-timeout fallback for unconfigured servers while applying an explicit operator timeout to long-running bundled MCP tool calls.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a bundled MCP server with an explicit timeout above 60,000 ms and call a tool that runs longer than 60s; current main passes no `RequestOptions.timeout`, so the MCP SDK default still applies.

## Solution Assessment

No, not as-is. The code approach is narrow and avoids the 30s default regression, but the mergeable solution should also update docs/changelog and clear exact-head CI; a separate public `requestTimeoutMs` key is broader product work, not required for this bug.
