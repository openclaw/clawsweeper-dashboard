## Summary

Keep open: this is the active implementation candidate for the bundled-MCP call timeout bug, and the code direction matches the MCP SDK contract, but it still needs public docs for the inherited timeout semantics and contributor-supplied real behavior proof before merge.

## What This Changes

The PR adds a bundled-MCP request timeout derived from explicit connectionTimeoutMs, propagates it through MCP transport/session state, passes it to Client.callTool() as RequestOptions.timeout, and adds resolver regression tests.

## Best Possible Solution

Land this PR or an equivalent patch after documenting the explicit connectionTimeoutMs request-timeout inheritance and 60s unset fallback, then require redacted after-fix runtime proof showing a long-running bundled MCP tool succeeds with an explicit timeout.

## Reproduction Assessment

Yes. On current main, a bundled MCP server with an explicit timeout above 60 seconds and a tool that responds after more than 60 seconds reaches Client.callTool without RequestOptions, so SDK 1.29.0 applies its 60-second default.

## Solution Assessment

Mostly yes, but not merge-ready. Threading a resolved request timeout into Client.callTool's third argument is the narrow code fix and preserves the unconfigured 60-second default, but the public docs and real runtime proof still need follow-through.

## Review Findings

Overall correctness: patch is incorrect
