## Label Justifications

- `P2`: The item concerns a bounded bundled MCP runtime bug affecting long-running tool calls, with limited blast radius.

## Summary

Current main has a same-or-better bundled MCP request-timeout implementation, and the fix shipped in v2026.6.6. This PR now targets the removed `pi-bundle` runtime path, is merge-conflicting, lacks real behavior proof, and its remaining timeout semantics are worse than the shipped `requestTimeoutMs`/`timeout` contract.

## What This Changes

The PR adds an internal per-call MCP request timeout derived from `connectionTimeoutMs`, propagates it through resolved transports, passes it to `Client.callTool()`, and adds transport-config regression tests.

## Best Possible Solution

Keep the shipped current-main `requestTimeoutMs` / `timeout` implementation and close this obsolete PR; future progress-reset or longer-default behavior should be a fresh narrow follow-up against `agent-bundle`.

## Reproduction Assessment

Yes. Source inspection plus the MCP SDK 1.29.0 contract show the old failure path: `Client.callTool()` without `RequestOptions.timeout` falls back to 60 seconds, while current main now passes the resolved per-server request timeout.

## Solution Assessment

No, this PR branch is no longer the best way to solve the issue. Current main shipped the maintainable fix in the renamed `agent-bundle` runtime with aligned schema/docs and a separate request-timeout config contract.
