## Summary

Keep open. Current main still omits the per-call MCP timeout, and the PR code now handles the earlier 30s default regression, but the branch is not mergeable while the public docs/changelog are missing and relevant CI checks are failing.

## What This Changes

The PR adds a bundled MCP request timeout derived from explicit `connectionTimeoutMs` or the SDK 60s default, threads it through transport/session resolution, passes it to `Client.callTool`, and adds config regression tests.

## Best Possible Solution

Land this PR or an equivalent current-main patch after documenting that explicit `connectionTimeoutMs` also controls bundled MCP tool-call request timeout, adding the required Unreleased changelog entry, and preserving the 60s request fallback for unconfigured servers.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: on current main, configure a bundled MCP server with `connectionTimeoutMs` above 60,000 ms and call a tool that runs longer than 60s; the runtime never passes that timeout to `Client.callTool()`, so the SDK default still applies.

## Solution Assessment

No, not as-is. The code direction is the narrow maintainable fix, but the mergeable solution also needs public docs/changelog updates and green checks; a new public `requestTimeoutMs` config key can remain a separate product decision.
