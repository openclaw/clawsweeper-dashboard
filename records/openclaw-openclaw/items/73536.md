## Summary

Keep this PR open. Current main still omits the MCP per-call timeout, and the PR head fixes the core code path with focused timeout-resolution tests, but it still needs user-facing docs and the required changelog entry before automerge.

## What This Changes

The PR derives a bundled-MCP per-call request timeout from explicit `connectionTimeoutMs`, preserves the SDK 60s default when unset, threads it through transport/session state, passes it to `Client.callTool`, and adds resolver regression tests.

## Best Possible Solution

Land this PR or an equivalent patch after documenting the explicit `connectionTimeoutMs` request-timeout inheritance with the 60s unset fallback and adding an Unreleased Fixes changelog entry with contributor credit.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure a bundled MCP server with an explicit timeout above 60,000 ms and call a tool that runs longer than 60s; current main passes no `RequestOptions.timeout`, so the pinned MCP SDK default still applies.

## Solution Assessment

No, not as merge-ready as-is. The TypeScript direction is narrow and matches the SDK contract, but the best mergeable solution also updates user-facing MCP docs and the required changelog entry.
