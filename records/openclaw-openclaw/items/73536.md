## Summary

Keep this PR open, but it still needs small docs and changelog follow-through before merge. The code direction matches the MCP SDK timeout contract and the updated head avoids the 30s default regression, while current main still omits per-call timeout options entirely.

## What This Changes

The PR adds bundled-MCP request-timeout resolution, propagates it through MCP transport/session state, passes it to `Client.callTool()` as request options, and adds transport-config regression tests.

## Best Possible Solution

Land this PR or an equivalent patch after documenting the explicit `connectionTimeoutMs` request-timeout inheritance and adding the required Unreleased Fixes changelog entry with contributor credit.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a bundled MCP server with an explicit timeout above 60,000 ms and call a tool that runs longer than 60s; current main passes no `RequestOptions.timeout`, so the SDK default still applies.

## Solution Assessment

No, not merge-ready yet. The code change is the narrow maintainable fix and preserves the SDK default when unset, but the user-facing config semantics and changelog need to be updated before merge.
