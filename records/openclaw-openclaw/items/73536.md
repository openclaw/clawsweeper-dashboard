## Summary

Keep open. The PR is a plausible focused fix for a real bundled-MCP timeout bug, but it is not merge-ready because the new request-timeout inheritance is not documented in the public config/docs surface and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a bundled-MCP request timeout derived from explicit connectionTimeoutMs, propagates it through MCP transport/session state, passes it to Client.callTool(), and adds resolver tests.

## Best Possible Solution

Land this PR or an equivalent patch after documenting the request-timeout inheritance/fallback semantics and adding redacted after-fix runtime proof for a long-running bundled MCP tool.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: configure a bundled MCP server with an explicit timeout above 60 seconds and call a tool that responds after more than 60 seconds; current main reaches Client.callTool without RequestOptions.timeout, so SDK v1.29.0 applies its 60-second default.

## Solution Assessment

Mostly yes, but not merge-ready. Threading a resolved request timeout into Client.callTool's third argument is the narrow code fix and preserves the unconfigured 60-second fallback, but the public docs/type help and real runtime proof still need follow-through.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Document request-timeout inheritance:** `src/agents/mcp-transport-config.ts:66-74`
  - body: This change makes explicit `connectionTimeoutMs` values also cap per-call bundled MCP tool requests while preserving the SDK 60s fallback when unset, but the public MCP docs and config type help still describe the field only as a connection timeout. Operators will not know this is the supported knob for long-running tools, and existing short connection budgets will unexpectedly cap tool calls; please update the public docs/help for both stdio and HTTP transports and spell out the unset fallback.
