## What This Changes

The PR adds a resolved bundled-MCP request timeout, propagates it through MCP transport/session state, passes it to `Client.callTool()` as request options, and adds transport-config regression tests.

## Best Possible Solution

Land this PR or an equivalent current-main patch after documenting the timeout semantics, adding the required Unreleased changelog entry, and clearing exact-head validation, while preserving the 60s request fallback for unconfigured servers and the explicit `connectionTimeoutMs` inheritance for long-running tool calls.

Yes. A high-confidence static reproduction exists: configure a bundled MCP server with an explicit timeout above 60,000 ms and call a tool that runs longer than 60s; current main passes no `RequestOptions.timeout`, so the MCP SDK default still applies.
