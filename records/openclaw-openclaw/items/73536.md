## Summary

Keep open. Current main still has the bundled MCP call-timeout bug, and this PR is the active implementation candidate, but it still needs public docs/changelog follow-through and exact-head CI cleanup before merge.

## What This Changes

The PR adds per-call bundled MCP request-timeout resolution, propagates it through transport/session state, passes it to `Client.callTool()`, and adds transport-config regression tests.

## Best Possible Solution

Land this PR or an equivalent current-main patch after adding the public MCP documentation and changelog entry, preserving the 60s request-timeout fallback for unconfigured servers while applying explicit operator timeout values to long-running bundled MCP tool calls.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a bundled MCP server with an explicit timeout above 60000 ms and call a tool that runs longer than 60s; current main passes no `RequestOptions.timeout`, so the MCP SDK default still applies.

## Solution Assessment

No, not as-is. The code direction is narrow and avoids the 30s default regression, but the mergeable solution should also update docs/changelog and clear exact-head CI; adding a separate public `requestTimeoutMs` key is broader product work and is not required for this bug.
