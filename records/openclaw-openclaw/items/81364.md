## Summary

Keep open: current main does not have the ClawHub install trust gate, so the PR is not obsolete, but the latest head still lets a ClawHub `blockedFromDownload` release proceed when `--acknowledge-clawhub-risk` is set. This is a MEMBER-authored, `maintainer`-labeled supply-chain PR, so it needs explicit maintainer/security handling before merge.

## What This Changes

The branch adds exact-release ClawHub trust checks and `--acknowledge-clawhub-risk` across plugin install/update/update/doctor/onboarding paths, with docs, tests, and rebase compatibility changes.

## Best Possible Solution

Land the centralized ClawHub trust gate after making `blockedFromDownload` return a non-overridable typed failure before archive download, while preserving acknowledgement for non-blocking risky states and warning-only pending or stale clean trust records.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: in the PR head, `blockedFromDownload` becomes a risk reason and `acknowledgeClawHubRisk` returns success for any risk reason, while the branch test expects archive download after acknowledging a blocked release.

## Solution Assessment

No. The trust-check placement is the right boundary, but blocked ClawHub releases should fail closed before the acknowledgement path; the acknowledgement flow should apply only to non-blocking risky states.

## Review Findings

Overall correctness: patch is incorrect
