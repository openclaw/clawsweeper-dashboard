## Summary

Keep open: this MEMBER-authored, maintainer-labeled security PR is not cleanup-closeable, and the current branch has a blocking supply-chain policy bug because `--acknowledge-clawhub-risk` can bypass `trust.blockedFromDownload` even though the new docs say blocked releases are refused.

## What This Changes

The PR reintroduces ClawHub exact-release trust checks before plugin install/update downloads, adds `--acknowledge-clawhub-risk` through CLI/update/doctor/onboarding paths, updates docs/tests/fixtures, and carries runtime-config compatibility fixes.

## Best Possible Solution

Land the centralized ClawHub trust gate after making `blockedFromDownload` non-overridable, while preserving pending/stale warnings and acknowledgement for non-blocking risky states.

## Reproduction Assessment

Yes, source-level: in the PR head, a trust response with `blockedFromDownload: true` reaches the acknowledgement bypass in `src/plugins/clawhub.ts`, and the branch test expects archive download after acknowledgement. I did not run tests because this review is read-only.

## Solution Assessment

No: checking trust before download is the right boundary, but blocked ClawHub releases should fail closed before the acknowledgement path. Pending/stale clean records can remain warnings, and non-blocking risky states can still use the acknowledgement flow.

## Review Findings

Overall correctness: patch is incorrect
