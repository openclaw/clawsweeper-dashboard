## Summary

Keep open: this protected MEMBER-authored PR is a plausible supply-chain hardening change, but the current head should not merge because ClawHub releases marked blocked can still be installed with the acknowledgement flag, contrary to the PR docs.

## What This Changes

The branch adds ClawHub exact-release trust checks and `--acknowledge-clawhub-risk` plumbing across plugin install, update, doctor, onboarding, and post-update sync paths, with docs, tests, and rebase fixes.

## Best Possible Solution

Land the centralized ClawHub trust gate after making `blockedFromDownload` a non-overridable install/update failure, while preserving acknowledgement for non-blocking risky states and warning-only pending or stale clean records.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: on the PR head, `blockedFromDownload` becomes a risk reason, `acknowledgeClawHubRisk` returns success for any risk reason, and the branch test expects download/install after acknowledging a blocked release.

## Solution Assessment

No. The trust-check placement before archive download is the right boundary, but blocked ClawHub releases should fail closed before the acknowledgement path; acknowledgement should apply only to non-blocking risky states.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep ClawHub download blocks non-overridable:** `src/plugins/clawhub.ts:565-566`

That lets automation download a release ClawHub explicitly marked blocked, while the added docs say OpenClaw refuses it; return a typed failure for blocked releases before the acknowledgement bypass.
  - confidence: 0.95
