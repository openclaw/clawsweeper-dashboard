## Summary

Keep open: this protected maintainer PR is still useful, but the current head has a blocking supply-chain bug because `--acknowledge-clawhub-risk` can still allow a ClawHub release marked `blockedFromDownload` to be downloaded, despite the added docs saying blocked releases are refused.

## What This Changes

The branch adds ClawHub exact-release trust checks and `--acknowledge-clawhub-risk` to plugin install/update/update/doctor/onboarding paths, updates docs/tests/fixtures, and includes rebased runtime-config, Matrix test-runtime, gateway lint, and prompt snapshot compatibility changes.

## Best Possible Solution

Land the centralized ClawHub trust gate after making `blockedFromDownload` fail closed before the acknowledgement path, while preserving acknowledgement for non-blocking risky states and warnings for pending or stale clean trust records.

## Reproduction Assessment

Yes, source-level: in the PR head, `blockedFromDownload` becomes a risk reason and `acknowledgeClawHubRisk` returns success for any risk reason. The branch test also expects archive download after acknowledging a blocked release.

## Solution Assessment

No: the trust-check placement is the right boundary, but blocked ClawHub releases should return a typed failure before the acknowledgement bypass. The acknowledgement flow should remain limited to non-blocking risky states.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

That lets automation download a release ClawHub explicitly marked blocked, contradicting the added docs and the fail-closed trust boundary; return a typed failure for blocked releases before the acknowledgement bypass.
