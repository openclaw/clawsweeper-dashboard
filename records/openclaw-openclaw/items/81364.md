## Summary

Keep open: this is a recent maintainer-labeled MEMBER PR, and the diff has a blocking security finding. The trust-check direction is useful, but blocked ClawHub download records must fail closed instead of being bypassable by the acknowledgement path.

## What This Changes

This PR adds ClawHub exact-release trust checks before plugin install/update downloads, acknowledgement handling across CLI/update/repair/onboarding flows, docs/tests, and narrow compatibility fixes in adjacent plugin/runtime paths.

## Best Possible Solution

Land the exact-release ClawHub trust gate after making blockedFromDownload an unconditional install/update failure and keeping acknowledgement only for non-blocking risky or stale/pending trust states.

## Reproduction Assessment

Yes. The source diff shows blockedFromDownload contributes to riskReasons, but the acknowledgement branch returns success before download; the added test also exercises blockedFromDownload with acknowledgement.

## Solution Assessment

No. The overall trust-check direction is appropriate, but the implementation must split hard ClawHub download blocks from acknowledgable risk warnings before it is safe to merge.
