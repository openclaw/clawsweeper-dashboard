## Summary

Keeping this open: current main still has the incompatible TypeScript and Swift device-identity schemas, and both paths can target the same state file when the macOS app inherits `OPENCLAW_STATE_DIR`. The report’s prompt-storm explanation is source-reproducible and not implemented on main.

## What This Changes

The issue asks OpenClaw to keep macOS app and CLI/gateway device identity stable when they share `OPENCLAW_STATE_DIR`.

## Best Possible Solution

Define one versioned cross-runtime identity/auth persistence contract, or explicitly split both macOS app identity and token files from CLI state, with migration and regression tests for TS-format and Swift-format files under shared state.

## Reproduction Assessment

Yes from source inspection, but not by live execution. On current main, a TS-format `identity/device.json` lacks Swift’s required `publicKey`/`privateKey` keys, while a Swift-format file lacks TS `version`/`publicKeyPem`/`privateKeyPem`, so each loader can silently regenerate when reading the other schema.

## Solution Assessment

Unclear until maintainers choose the persistence contract. A complete fix should account for both `identity/device.json` and `identity/device-auth.json`; changing only one filename could leave app and CLI token caching fighting over a single-device token store.
