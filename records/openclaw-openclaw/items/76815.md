## Summary

Keep open. Current main still matches the reported failure mode: the TS gateway/CLI and Swift app use different schemas for the same shared identity path when `OPENCLAW_STATE_DIR` is inherited, so one runtime can silently replace the other's device identity.

## What This Changes

The issue requests that the macOS app and CLI/gateway preserve a stable device identity instead of sharing incompatible `identity/device.json` schemas that can rotate IDs after relaunch.

## Best Possible Solution

Add a narrow cross-runtime identity compatibility contract so recognized TS and Swift identity files are loaded without silent regeneration, with regression fixtures covering both schemas under shared `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes, at source level. A TS-format identity file cannot decode as Swift `DeviceIdentity`, and a Swift-format identity file fails the TS schema checks, so alternating runtimes at the same `OPENCLAW_STATE_DIR` path can regenerate the identity.

## Solution Assessment

No, current main is not the best solution because both runtimes still silently regenerate on the other recognized schema. The narrow maintainable fix is dual-schema compatibility or an explicit non-colliding identity path, backed by TS and Swift tests.

## Repair Work Prompt

Fix openclaw/openclaw#76815. Current main lets the macOS Swift app and TS CLI/gateway share `<OPENCLAW_STATE_DIR>/identity/device.json` with incompatible schemas: TS expects `{ version, deviceId, publicKeyPem, privateKeyPem, createdAtMs }`, while Swift expects `{ deviceId, publicKey, privateKey, createdAtMs }`. Implement a narrow compatibility fix so neither runtime silently overwrites the other recognized OpenClaw identity schema. Prefer preserving the existing device ID by reading both schemas and writing a canonical compatible form only when safe; if a separate app identity path is chosen, make sure it cannot clobber the TS identity and that app device tokens stay tied to the app identity. Add regression coverage for TS handling of Swift raw-key fixtures and Swift/OpenClawKit handling of TS PEM fixtures under `OPENCLAW_STATE_DIR`. Keep `device-auth.json` semantics and pairing authorization policy unchanged. Add a user-facing `CHANGELOG.md` fix entry. Stop and ask if the fix would require a new config option or a security/product policy change.
