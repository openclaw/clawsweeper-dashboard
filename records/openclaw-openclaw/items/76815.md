## Summary

Keep open. Current main still has incompatible Swift and TypeScript identity-file schemas at the same `OPENCLAW_STATE_DIR`-derived path, and both device-token stores bind tokens to the active `deviceId`, so the reported device-ID churn is source-reproducible and actionable.

## What This Changes

The issue reports that macOS and TypeScript device identity stores can collide on the same state path with incompatible JSON schemas, causing device ID churn after relaunch.

## Best Possible Solution

Adopt one durable identity/auth persistence contract for shared state dirs, with regression coverage proving that an existing TS PEM identity is not silently overwritten by the Swift app and that stored device tokens remain bound to a stable `deviceId`.

## Reproduction Assessment

Yes, from source inspection. A TS-format `identity/device.json` under `OPENCLAW_STATE_DIR` lacks Swift's required `publicKey` and `privateKey` fields, so Swift decoding fails and `loadOrCreate()` generates a replacement identity; the TS loader then rejects the Swift raw-key shape for the same reason in reverse.

## Solution Assessment

Yes, the issue is pointing at the right layer. The safest end state is schema compatibility or a fully isolated app identity/auth store, plus a fail-closed path for recognized incompatible OpenClaw identity files instead of silent regeneration.
