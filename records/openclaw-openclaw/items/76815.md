## Summary

Keep this open: current main still has the reported Swift/TS identity schema mismatch, and the latest release tag shows the same incompatible loaders. The source path is narrow enough for a focused fix PR.

## What This Changes

The issue asks OpenClaw to stop the macOS app and CLI/gateway from overwriting each other’s shared device identity and causing repeated pairing prompts after relaunch.

## Best Possible Solution

Make macOS identity loading compatible with the shared TS identity path, or use a non-conflicting app identity file while refusing to overwrite a recognized OpenClaw identity in another schema; add regression coverage for both existing formats.

## Reproduction Assessment

Yes, from source inspection: a TS PEM identity in `OPENCLAW_STATE_DIR/identity/device.json` makes Swift decoding fail and regenerate, while a Swift raw-key identity makes the TS loader reject and regenerate. I did not run a live macOS relaunch loop in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to stop silent cross-schema overwrite at the shared identity path and cover the reconnect-token contract with tests. A broad pairing-policy change is not needed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76815. Current main has incompatible TS and Swift schemas for `OPENCLAW_STATE_DIR/identity/device.json`: TS expects `version/publicKeyPem/privateKeyPem`, Swift expects raw-base64 `publicKey/privateKey`, and both silently regenerate when the other schema exists. Implement the narrowest compatibility fix so the macOS app no longer overwrites or replaces a recognized CLI/gateway identity when launched with the shared state dir. Prefer supporting both existing on-disk schemas, including preserving stable `deviceId` and stored device tokens; if choosing a separate macOS identity file, explicitly avoid overwriting recognized TS PEM identities and add tests documenting the one-time behavior. Do not add a new config option or broaden pairing policy. Add regression coverage for Swift loading against a TS PEM-style file and for existing Swift-format identities, and update TS tests only if the TS loader behavior changes. Add a user-facing `CHANGELOG.md` fix entry. Validate with targeted Swift tests and `pnpm test src/infra/device-identity.test.ts` if TS is touched; before handoff run the repo changed gate in Testbox per policy.
