## Summary

Keep open: current main still has incompatible TS and Swift device identity schemas at the same `identity/device.json` path when `OPENCLAW_STATE_DIR` is shared, and token reuse depends on the resulting stable `deviceId`. The reported relaunch prompt storm is source-reproducible from the current code.

## What This Changes

The issue asks OpenClaw to stop the macOS app and TS/CLI gateway from overwriting each other’s device identity when they share `OPENCLAW_STATE_DIR`.

## Best Possible Solution

Make the macOS and TS identity stores recognize a single compatible on-disk identity contract, or use clearly separate non-colliding paths, while adding regression coverage that proves a shared `OPENCLAW_STATE_DIR` keeps the same `deviceId` across relaunches.

## Reproduction Assessment

Yes, from source inspection. With `OPENCLAW_STATE_DIR` pointed at a TS state dir containing a PEM-shaped `identity/device.json`, Swift decode fails and writes a new raw-key identity; the TS loader then rejects that raw-key file and can regenerate the PEM identity again.

## Solution Assessment

No patch is present to assess. The narrow maintainable fix is identity-schema compatibility or a deliberate non-colliding path, with tests covering both current file shapes and token reuse by stable `deviceId`.
