## What This Changes

The issue requests that macOS app and TypeScript/CLI gateway device identity persistence stop using incompatible schemas for the same state-dir identity file.

## Best Possible Solution

Make the cross-runtime identity contract explicit and covered so Swift and TypeScript either understand one shared on-disk schema or use non-conflicting identity paths without silently replacing a recognized OpenClaw identity file.

Yes from source inspection: set `OPENCLAW_STATE_DIR` for the macOS app to the CLI state dir, put either runtime's valid `identity/device.json` there, and the other runtime treats it as invalid and generates a different identity. I did not run a live macOS relaunch loop in this read-only review.

## Repair Work Prompt

Fix the current-main macOS/TypeScript device identity schema collision reported in https://github.com/openclaw/openclaw/issues/76815. The boundary is the shared state identity contract: `src/infra/device-identity.ts` currently writes `{version, deviceId, publicKeyPem, privateKeyPem, createdAtMs}` to `<OPENCLAW_STATE_DIR>/identity/device.json`, while `apps/shared/OpenClawKit/Sources/OpenClawKit/DeviceIdentity.swift` reads/writes `{deviceId, publicKey, privateKey, createdAtMs}` at the same path and silently regenerates on decode failure. Implement a narrow backwards-compatible fix so a recognized OpenClaw identity file is not silently replaced by the other runtime; prefer explicit compatibility/migration or a documented non-conflicting app path, and preserve device-token reuse semantics in `DeviceAuthStore`. Add focused regression coverage for the TS loader and Swift/OpenClawKit identity path. Update `CHANGELOG.md` because this is user-facing pairing behavior. Do not add a new config knob, do not change unrelated gateway pairing policy, and stop for maintainer review if canonicalizing the key format requires a broader product/security decision.
