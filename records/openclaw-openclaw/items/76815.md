## Summary

Keep open: current main still has incompatible TS and Swift identity-file schemas at the same shared state path, and the Swift loader silently creates a new identity when it cannot decode the TS PEM-form file.

## What This Changes

The issue asks OpenClaw to stop the macOS app and TS/CLI gateway clients from regenerating conflicting device identities when the macOS app is launched with a shared `OPENCLAW_STATE_DIR`.

## Best Possible Solution

Make the Swift/macOS identity and token persistence contract compatible with the TS gateway contract, or deliberately isolate both app identity and token files, with regression coverage for a shared `OPENCLAW_STATE_DIR`.

## Reproduction Assessment

Yes, by source inspection: a TS-format `identity/device.json` at the shared state path fails Swift `DeviceIdentity` decoding because it lacks `publicKey` and `privateKey`, so Swift falls through to `generate()` and overwrites the file with a different schema and device id. I did not execute the live macOS pairing loop in this read-only review.

## Solution Assessment

Unclear. The issue identifies the right failure boundary, but the best fix needs an owner decision between TS PEM compatibility/migration and fully separate app-scoped identity plus token storage.
