## Summary

Keep open. Current main still has the reported shared-path schema mismatch, and the issue includes concrete logs plus source pointers showing why relaunches can rotate the macOS device identity.

## What This Changes

The issue reports that macOS and TS identity stores can use the same `identity/device.json` path with incompatible persisted schemas, causing repeated new device IDs after relaunch.

## Best Possible Solution

Align TS and Swift identity persistence around one compatible on-disk contract, or a non-colliding migrated path, and add regression coverage that both runtimes behave safely when `OPENCLAW_STATE_DIR` points at the same state root.

## Reproduction Assessment

Yes, from source inspection. With a shared `OPENCLAW_STATE_DIR`, TS writes PEM fields to `identity/device.json`, Swift expects raw key fields at that same path, and each loader can reject the other schema and create a new identity.

## Solution Assessment

Yes. Treating this as a device identity compatibility bug is the narrow maintainable direction; a new config option is unnecessary if the stores stop colliding or learn a shared canonical schema.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76815 as a narrow device-identity persistence bug. Make the macOS Swift identity store and TS `src/infra/device-identity.ts` safe when `OPENCLAW_STATE_DIR` points both runtimes at the same state root: they must not overwrite a recognized OpenClaw identity schema with a fresh keypair, and relaunching the macOS app must keep the same `deviceId` and reusable `device-auth.json` token. Prefer a canonical shared schema or a deterministic compatibility/migration path; do not add a config flag, do not weaken key validation, and preserve existing private keys/tokens where feasible. Add TS and Swift regression coverage for TS-format and Swift-format identity files at `identity/device.json`, plus a user-facing changelog fix entry.
