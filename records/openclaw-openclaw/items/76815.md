## Summary

Keep open. Current main still has the reported macOS/OpenClawKit versus TS/CLI identity-file schema mismatch, and the source path can silently regenerate a different device ID when the app is pointed at the shared state directory.

## What This Changes

Request to make macOS/OpenClawKit and TS/CLI/gateway use a compatible device identity schema so app relaunch under a shared OPENCLAW_STATE_DIR reuses the same deviceId.

## Best Possible Solution

Unify the on-disk identity contract across TS and OpenClawKit with backward-compatible loading/migration, and fail clearly instead of silently regenerating when a recognized OpenClaw schema is present.

## Reproduction Assessment

Yes, source-reproducible: point OpenClawKit at a TS-created OPENCLAW_STATE_DIR and its decoder cannot read the PEM identity schema, so loadOrCreate falls through to a newly generated identity. I did not run a live macOS relaunch in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable fix is schema compatibility/migration in the identity loaders with regression coverage; moving only the identity path would still need care because the device-auth store is keyed by a single current deviceId.

## Repair Work Prompt

Fix the current-main bug where the macOS/OpenClawKit device identity store silently overwrites or rotates a TS/CLI PEM `identity/device.json` when `OPENCLAW_STATE_DIR` points at the shared state directory. Prefer a single backward-compatible identity contract: OpenClawKit must load a TS `{version:1, deviceId, publicKeyPem, privateKeyPem, createdAtMs}` identity without changing the deviceId, and the TS side must continue loading existing PEM identities; support or migrate existing raw OpenClawKit `{deviceId, publicKey, privateKey, createdAtMs}` identities without silent loss. Do not add a new config option, do not change gateway pairing policy, and stop for maintainer review if the fix requires redesigning device-auth storage beyond preserving existing tokens. Add regression coverage for the TS PEM fixture under `OPENCLAW_STATE_DIR`, the raw Swift legacy schema if touched, and token reuse across repeated loadOrCreate calls. Add a CHANGELOG.md fix entry because this is user-facing.
