## Summary

Keep open: current main still has the reported schema/path mismatch, and the issue includes enough source pointers and runtime evidence to treat this as a valid device-auth persistence bug rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to stop macOS app relaunches from generating new device identities when the app shares the CLI/gateway state directory.

## Best Possible Solution

Make macOS and TS agree on one documented identity persistence contract, or deliberately separate the app identity path, with regression coverage that prevents silent cross-runtime identity replacement.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: put a TS PEM identity at `$OPENCLAW_STATE_DIR/identity/device.json`, launch the Swift client with that same `OPENCLAW_STATE_DIR`, and Swift decode fails into identity regeneration; the reverse schema is also rejected by the TS loader.

## Solution Assessment

Yes, the issue's suggested direction is the narrow maintainable fix boundary. The safest implementation is schema compatibility or a non-colliding versioned path, plus no silent overwrite for recognized OpenClaw identity formats.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76815. The macOS OpenClawKit `DeviceIdentityStore` and TS `src/infra/device-identity.ts` both use `$OPENCLAW_STATE_DIR/identity/device.json` but currently use incompatible schemas, causing identity churn and repeated pairing when the app runs against the shared CLI/gateway state dir. Implement a narrow fix that prevents relaunches from replacing a valid OpenClaw identity: support/migrate the TS PEM schema in Swift and/or the Swift raw-key schema in TS, or move the macOS app to a clearly non-conflicting versioned identity path if that is the chosen local pattern. Do not silently overwrite a recognized OpenClaw identity schema; keep corruption recovery for truly invalid files. Add regression coverage for Swift reading a TS-format identity, TS behavior for the Swift/raw schema or the chosen non-collision path, and stable device-token reuse keyed by the same deviceId after relaunch. Update CHANGELOG.md for the user-facing pairing persistence fix. Avoid broad pairing-policy changes, new config flags, or unrelated gateway auth behavior.
