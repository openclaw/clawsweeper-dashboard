## Summary

Current main still matches the reported failure mode: the Swift app and TS gateway can target the same `identity/device.json` under `OPENCLAW_STATE_DIR` while expecting incompatible key fields, and stored device tokens are keyed by the resulting `deviceId`. This should stay open for a real cross-runtime persistence fix.

## What This Changes

Fix the macOS app and TS/CLI/gateway device identity stores so a shared `OPENCLAW_STATE_DIR` cannot silently rotate the app `deviceId` and invalidate approvals after relaunch.

## Best Possible Solution

Keep the issue open and converge on a single durable identity contract, preferably with regression coverage proving a TS PEM identity in `OPENCLAW_STATE_DIR` is reused or fails closed without changing `deviceId`.

## Reproduction Assessment

Yes, by source inspection: put a TS PEM-format `identity/device.json` under a state dir inherited by the Swift app and `DeviceIdentityStore.loadOrCreate()` cannot decode the missing `publicKey`/`privateKey` fields, so it falls through to generate a new identity. The reverse direction is also clear because the TS loader rejects a Swift raw-key file lacking `version`, `publicKeyPem`, and `privateKeyPem`.

## Solution Assessment

Yes, fixing identity persistence is the right layer; pairing retry behavior would only mask the rotated identity. The maintainable fix is to make the Swift and TS stores share a compatible schema, add migration/fail-closed behavior, or deliberately separate paths without silently overwriting recognized OpenClaw identity files.
