## Summary

Keep open: current main still has the reported cross-runtime identity schema mismatch, and v2026.5.2 shipped the same behavior. The bug is narrow and source-reproducible from the TS and Swift loaders plus the device-token store keyed by `deviceId`.

## What This Changes

Fix the macOS app and TS/CLI gateway device identity storage so app relaunches reuse the same device ID and stored approval tokens when sharing `OPENCLAW_STATE_DIR`.

## Best Possible Solution

Unify or compatibly migrate the Swift and TS identity formats for shared state, stop silent regeneration for recognized alternate OpenClaw schemas, and add cross-runtime regression coverage for identity and token reuse.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: put a TS-format PEM `identity/device.json` under a shared `OPENCLAW_STATE_DIR` and Swift decode fails because `publicKey`/`privateKey` are missing, so it generates and saves a new raw-key identity. Then TS treats that raw-key file as invalid because `version`/`publicKeyPem`/`privateKeyPem` are missing and regenerates again; I did not execute this write-based repro because this review is read-only.

## Solution Assessment

Yes, the best fix direction is a narrow cross-runtime compatibility/migration change rather than pairing-policy changes or a new setting. A separate identity file is only safe if the matching device-token store separation is handled too.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76815. Current main stores TS/CLI identity at `identity/device.json` as `{ version, deviceId, publicKeyPem, privateKeyPem, createdAtMs }`, while Swift `DeviceIdentityStore` uses the same file when `OPENCLAW_STATE_DIR` is set but expects `{ deviceId, publicKey, privateKey, createdAtMs }`; both sides silently regenerate on the other schema, and `DeviceAuthStore` tokens are keyed to the resulting `deviceId`. Implement a narrow compatibility fix, preferably by making shared OpenClawKit read/write or migrate the TS PEM schema while preserving legacy Swift raw identities, and update TS only as needed so a recognized OpenClaw alternate schema is not silently overwritten. Do not add a new config/env option, do not change gateway pairing authorization policy, and do not touch unrelated launchd update repair code unless a test proves it is part of this bug. Add Swift regression coverage for a shared `OPENCLAW_STATE_DIR` containing a TS PEM identity and TS regression coverage if TS learns the Swift raw schema; include a user-facing changelog fix entry.
