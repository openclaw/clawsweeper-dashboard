## Summary

Keep this open: current main still has the reported cross-runtime identity schema mismatch, and the source path explains the relaunch-to-new-device behavior without needing a new product decision.

## What This Changes

Make macOS app and CLI/gateway device identity persistence compatible when both use the same `OPENCLAW_STATE_DIR`, so relaunches reuse the approved `deviceId`.

## Best Possible Solution

Adopt one cross-runtime on-disk identity contract, or add a dual-read migration that preserves recognized OpenClaw identities and refuses silent regeneration when the other schema is present.

## Reproduction Assessment

Yes, from source inspection: seed `$OPENCLAW_STATE_DIR/identity/device.json` with the TS PEM schema and Swift will fail to decode it and generate a new raw-key identity; seed it with Swift raw keys and the TS loader falls through to regeneration. I did not run a live macOS relaunch because this review is read-only.

## Solution Assessment

Yes: fixing the identity store compatibility is the narrow maintainable solution. The repair should avoid a config knob and keep gateway pairing policy unchanged while preserving existing device-token state for the same `deviceId`.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76815 by making the macOS/OpenClawKit device identity store compatible with the TS gateway identity contract under a shared `OPENCLAW_STATE_DIR`. Expected behavior: when `<state>/identity/device.json` contains the TS PEM schema `{version:1, deviceId, publicKeyPem, privateKeyPem, createdAtMs}`, Swift loads it, derives raw public/private keys for gateway device auth, preserves the same `deviceId`, and does not overwrite or regenerate the file. If the TS side must also tolerate the Swift raw-key schema, add a generic migration/dual-reader without silently rotating any recognized OpenClaw identity. Add Swift regression coverage for a seeded TS PEM identity and repeat load/signature/device-auth token reuse; update TS tests if the TS reader changes. Add a single-line `CHANGELOG.md` fix entry. Do not add a config flag, do not change gateway pairing scopes/policy, and do not rotate existing tokens except when the identity truly cannot be parsed.
