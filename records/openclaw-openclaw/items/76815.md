## Summary

Keep open. Current main still has the cross-runtime identity-file mismatch and silent regeneration path, so the reported pairing churn remains actionable.

## What This Changes

The issue asks OpenClaw to keep macOS app and TS CLI/gateway device identities compatible, or non-conflicting, so app relaunches reuse the same deviceId and device-auth token.

## Best Possible Solution

Teach the TS and Swift identity stores to recognize and preserve the same on-disk identity contract, including legacy raw and PEM forms, without silently replacing a recognized OpenClaw identity file.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: seed `$OPENCLAW_STATE_DIR/identity/device.json` with the TS PEM schema, then call Swift `DeviceIdentityStore.loadOrCreate()` and it cannot decode that schema, so it generates and saves a new raw-key identity. I did not execute the app loop because this sweep was read-only.

## Solution Assessment

Yes, the best fix is a narrow compatibility repair in the existing identity stores, not a new flag or pairing-policy change. The repair should avoid silent regeneration for both recognized OpenClaw schemas and add cross-runtime regression coverage.
