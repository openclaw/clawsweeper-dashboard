## Summary

Keep open: current main still has incompatible TS and Swift schemas for the same shared `identity/device.json` path, and the token store is keyed by the resulting `deviceId`, so the reported prompt storm is source-reproducible.

## What This Changes

The issue reports that macOS and CLI/gateway write incompatible schemas to the shared device identity file, causing repeated new device IDs after app relaunch.

## Best Possible Solution

Unify the cross-runtime device identity contract, with compatibility for both existing schemas, so sharing `OPENCLAW_STATE_DIR` never silently regenerates or overwrites a valid OpenClaw identity.

## Reproduction Assessment

Yes from source inspection, though not by live macOS execution here. A TS-owned PEM-format `identity/device.json` under a shared `OPENCLAW_STATE_DIR` is not decodable by Swift, and the Swift path falls through to generating a different raw-key identity.

## Solution Assessment

Yes, the narrow maintainable fix is in the identity compatibility layer rather than pairing policy. Both runtimes should recognize valid OpenClaw identity files and avoid silent regeneration when an alternate known schema is present.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76815. Current main has incompatible device identity schemas at `identity/device.json`: TS `src/infra/device-identity.ts` uses `{ version, deviceId, publicKeyPem, privateKeyPem, createdAtMs }`, while Swift `apps/shared/OpenClawKit/Sources/OpenClawKit/DeviceIdentity.swift` uses `{ deviceId, publicKey, privateKey, createdAtMs }` and honors `OPENCLAW_STATE_DIR`. Implement a focused compatibility fix so the macOS/native store and TS store do not overwrite each other or silently generate new device IDs when sharing a state directory. Prefer supporting both existing schemas and migrating or preserving a single canonical schema only after validating that the stored public key still derives the stored deviceId. Do not weaken signature verification, deviceId/publicKey binding, token scope checks, or pairing policy. Add regression coverage for TS reading/handling the Swift schema if needed and Swift reading/handling the TS PEM schema, plus a token-store check that a stable deviceId keeps `device-auth.json` reusable. Add a user-facing `CHANGELOG.md` fix entry.
