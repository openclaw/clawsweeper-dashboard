## Summary

Keep open: current main still has incompatible TS and Swift device-identity schemas on the same shared state path, and the auth-token store depends on the resulting deviceId staying stable.

## What This Changes

The issue asks OpenClaw to keep macOS app and CLI/gateway device identities compatible so approvals survive relaunch when the app uses the shared state dir.

## Best Possible Solution

Make OpenClawKit compatible with the canonical TS PEM identity when sharing OPENCLAW_STATE_DIR, preserve or deliberately migrate legacy raw app identities, and add cross-runtime regression coverage for identity plus device-token reuse.

## Reproduction Assessment

Yes, from source inspection: a TS PEM identity at $OPENCLAW_STATE_DIR/identity/device.json cannot be decoded by the Swift raw-key Codable model, so Swift falls through to generation. I did not run the macOS app in this read-only review.

## Solution Assessment

Yes, a narrow compatibility/migration fix in OpenClawKit is the best direction; changing only the app identity path would still leave the shared device-auth store tied to a single deviceId unless token storage is separated too.
