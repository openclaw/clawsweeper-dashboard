## Summary

Current main still has the reported shared-state identity schema collision, so this should stay open and is a good focused repair candidate.

## What This Changes

The issue requests stable macOS app pairing when the app inherits `OPENCLAW_STATE_DIR` and shares state with the CLI/gateway.

## Best Possible Solution

Use one canonical shared on-disk identity/auth schema across TS and Swift, or isolate both macOS identity and auth-token files so relaunches cannot silently rotate device IDs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: TS writes a versioned PEM `identity/device.json`, Swift reads the same path as raw-key fields and regenerates on decode failure, and token lookup then misses because it is keyed by the new `deviceId`.

## Solution Assessment

Yes. The narrow maintainable fix is to remove the schema collision and preserve the existing device ID; retry/prompt changes would not fix approval persistence.

## Repair Work Prompt

Fix the macOS/Swift and TS device identity schema collision reported in https://github.com/openclaw/openclaw/issues/76815. Prefer a canonical versioned shared identity schema for `OPENCLAW_STATE_DIR/identity/device.json`: teach Swift `DeviceIdentityStore` to read/write the TS PEM schema and migrate old Swift raw-key identity records without changing the device ID; teach the TS loader to recognize/migrate old Swift raw-key records if needed so neither runtime silently regenerates on a recognized OpenClaw schema. Confirm `identity/device-auth.json` remains tied to the preserved device ID; do not introduce a new config option or weaken signature verification. Add focused Swift and TS regression coverage for a TS PEM file loaded by Swift, a Swift legacy raw-key file loaded/migrated, stable `deviceId` across repeated loads, and token reuse by `DeviceAuthStore`. Add a user-facing `CHANGELOG.md` fix entry. Stop and ask for maintainer direction if preserving one shared schema proves unsafe and the alternative would require separate macOS identity plus separate auth-token files.
