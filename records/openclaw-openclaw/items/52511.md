## Summary

Keep open. Current main has generic Control UI reconnect handling, bounded cached-device-token recovery, and clearer scope-upgrade/token-drift guidance, but there is still no targeted source/test/release proof that Control UI chrome-relay browser commands recover after Gateway restart or token rotation. The browser Gateway method remains `operator.admin`-scoped, which conflicts with the report's read/write-only expectation, and the directly linked retry PR was closed unmerged.

## What This Changes

The issue asks for Control UI browser commands over a Chrome relay profile to retain or refresh browser-control authorization after Gateway restart or device-token rotation without relying on repeated token rotation or Gateway restarts.

## Best Possible Solution

Keep this as the Control UI browser-permission tracker. The desired end state is an explicit current-main reproduction or regression test for Control UI `browser.request` after Gateway restart/token rotation, followed by targeted reconnect/token recovery or user-facing approval guidance, with a deliberate decision on whether browser control stays admin-only or gets a narrower permission model.
