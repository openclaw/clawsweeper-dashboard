## Summary

Keep open: current main has generic Control UI reconnect handling and bounded token-drift recovery, but I did not find proof that the reported Control UI chrome-relay browser flow recovers after Gateway restart or token rotation. The browser gateway method is still admin-scoped, so the report's read/write-only permission expectation conflicts with current source and tests.

## What This Changes

The issue asks for Control UI browser commands to keep working after Gateway restart or token rotation when the paired device has operator read/write permissions.

## Best Possible Solution

Keep this as the Control UI browser-permission tracker until maintainers either reproduce the current-main chrome-relay restart/token-rotation failure and add targeted recovery coverage, or explicitly document/adjust the admin-only browser-control permission model.

No. The issue has plausible 2026.3.13 Windows/chrome-relay steps and source-backed auth-policy conflicts, but I did not find a high-confidence current-main reproduction of Control UI browser commands failing after Gateway restart or token rotation.
