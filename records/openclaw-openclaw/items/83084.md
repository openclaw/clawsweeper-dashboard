## Summary

Keep open: the PR targets a real source-reproducible missing-footer path, but the patch is not merge-ready because it bypasses the message-tool-only privacy boundary with untrusted text scraping and still lacks live Telegram forum-topic proof.

## What This Changes

The PR adds post-loop `Usage:` footer extraction from suppressed message-tool-only final replies plus a regression test for standalone footer delivery.

## Best Possible Solution

Represent the response usage footer as trusted metadata or a dedicated payload at generation time, route only that trusted footer through suppression while preserving sendPolicy and room-event rules, and prove the result with redacted live Telegram forum-topic evidence.

## Reproduction Assessment

Yes, source-level: current main suppresses ordinary final payloads in `message_tool_only` while `appendUsageLine` adds the usage footer as plain final-reply text after the run. I did not establish a live Telegram forum-topic reproduction in this read-only review.

## Solution Assessment

No. The patch is aimed at the right missing-footer path, but text-pattern scraping is not a safe provenance boundary; a marked footer payload or metadata flag is the narrower maintainable fix.
