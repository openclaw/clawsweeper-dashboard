## Summary

Keep this PR open, but it should not merge as-is: the bug is source-reproducible and the PR targets the right dispatch area, but the proposed fix scrapes arbitrary suppressed reply text for a `Usage:` line and can expose part of a private final reply in message-tool-only group/channel mode.

## What This Changes

The PR adds post-loop extraction of a `Usage:` line from suppressed final replies in `src/auto-reply/reply/dispatch-from-config.ts` plus a regression test for message-tool-only delivery.

## Best Possible Solution

Mark the actual response-usage footer with explicit metadata or emit it as a dedicated suppression-allowed payload, while preserving sendPolicy and room-event privacy, then prove the Telegram forum-topic behavior live.

## Reproduction Assessment

Yes. Current source shows `message_tool_only` suppresses final payloads while the usage footer is appended later as ordinary payload text; I did not run a live Telegram forum-topic repro.

## Solution Assessment

No. The PR is aimed at the right bug, but scraping `Usage:` from suppressed private text is not a safe provenance boundary; the usage footer should be marked or emitted explicitly before bypassing suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid scraping suppressed replies for Usage lines:** `src/auto-reply/reply/dispatch-from-config.ts:1691-1696`
  - body: In message-tool-only mode, normal final answers are intentionally private unless the model uses the message tool. This block publishes any suppressed line that starts with `Usage: `, so a user- or model-produced private final answer like `Usage: rotate the token...` would leak into the group/channel, and room events also bypass the existing exclusion. Please carry explicit usage-footer provenance, such as metadata or a dedicated payload, before allowing delivery despite source suppression.
  - confidence: 0.88
