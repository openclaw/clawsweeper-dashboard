## Summary

Keep open: the linked usage-footer gap is source-reproducible, but this PR should not merge as written because it bypasses the message-tool-only privacy boundary by scraping untrusted suppressed text and it still lacks real Telegram proof.

## What This Changes

The PR adds a post-loop `Usage:` line extractor in `dispatch-from-config` plus a regression test to deliver a standalone usage footer when `message_tool_only` suppresses final replies.

## Best Possible Solution

Represent the response usage footer as trusted structured metadata or a dedicated payload generated by the runner, then allow only that marked footer through suppression while preserving sendPolicy and room-event privacy rules; if maintainers reject the runtime behavior, clarify the docs instead.

## Reproduction Assessment

Yes, source-level: current main suppresses ordinary final payloads in `message_tool_only` while `appendUsageLine` adds the usage footer as plain final-reply text after the run. I did not establish a live Telegram forum-topic reproduction.

## Solution Assessment

No. The PR targets the right missing-footer path, but text-pattern scraping is not a safe provenance boundary; a marked usage-footer payload is the narrower maintainable fix.
