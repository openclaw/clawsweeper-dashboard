## Summary

Keep open: the PR targets a real source-reproducible message-loss path, but the current patch is unsafe because it scrapes `Usage:` from suppressed private final text and can publish content that should stay private; after-fix live Telegram proof is also missing.

## What This Changes

The PR adds post-loop extraction of a `Usage:` footer from suppressed message-tool-only final replies and a regression test for that delivery path.

## Best Possible Solution

Represent the response usage footer as structured metadata or a dedicated payload generated in the runner, then allow only that trusted footer through suppression while preserving sendPolicy and room-event privacy rules.

## Reproduction Assessment

Yes, source-level: current main suppresses ordinary final payloads in `message_tool_only` while `appendUsageLine` adds the usage footer as plain final-reply text after the run. I did not establish a live Telegram forum-topic repro.

## Solution Assessment

No. The patch is aimed at the right missing-footer path, but text-pattern scraping is not a safe provenance boundary; a marked usage-footer payload is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve footer provenance before bypassing suppression:** `src/auto-reply/reply/dispatch-from-config.ts:1689-1694`
  - body: In `message_tool_only`, normal final text is intentionally private unless the agent uses the message tool. Matching `\nUsage: ` inside any suppressed reply can publish model/user content, and it also skips the room-event exclusion used by the existing suppression bypass. Carry explicit usage-footer metadata or emit a dedicated payload before calling `sendFinalPayload`.
  - confidence: 0.9
