## Summary

Keep open: current main still drops mixed-message `/model` directive state before model selection, but this PR has blocking correctness defects and lacks required real behavior proof from the contributor.

## What This Changes

The PR adds one-shot mixed-message `/model <provider/model> <message>` handling across reply directive parsing, run preparation, auth-profile propagation, docs, and regression tests.

## Best Possible Solution

Implement one-shot `/model` through an explicitly non-persistent selection path that strips directive text, applies provider/model/auth profile only to the current run, recomputes context and defaults from that selected model, and leaves stored session overrides untouched.

## Reproduction Assessment

Yes, source-reproducible. Current main parses `/model`, then clears directive state for mixed messages before model selection, so the run stays on the current session/default model while the directive text is stripped.

## Solution Assessment

No. The requested direction is right, but this patch calls `resolveContextTokens` without required `cfg` and `provider` inputs and recomputes through persistent session-selection state; a dedicated non-persistent path is safer.

## Review Findings

Overall confidence: 0.93

Full review comments:

- **[P1] Pass cfg and provider into one-shot context resolution:** `src/auto-reply/reply/get-reply-directives-apply.ts:277-280`
  - body: The new one-shot block calls `resolveContextTokens` with only `agentCfg` and `model`, but the helper requires `cfg`, `agentCfg`, `provider`, and `model`. This should fail typechecking and would also lose provider-specific context windows for the selected one-shot model.
  - confidence: 0.95
- **[P1] Avoid mutating stored auth profiles during one-shot recompute:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: After switching to the one-shot provider/model, this rebuild passes live `sessionEntry`, `sessionStore`, and `sessionKey` into `createModelSelectionState()`. That helper can clear a stored auth-profile override on provider mismatch, so a non-persistent one-shot can delete persistent session auth state.
  - confidence: 0.9
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This rebuild uses `hasModelDirective: false` while still passing session state, so a stored session model override can replace the rebuilt `modelState` before thinking/reasoning defaults are resolved. The run can use the one-shot model while defaults come from the old session model.
  - confidence: 0.9
