## Summary

Keep open: current main still drops mixed-message `/model` state, so this PR targets a real bug, but the branch is not ready because the one-shot refresh can mutate or reuse persistent session state and the PR lacks after-fix real behavior proof.

## What This Changes

The PR adds a one-shot `/model <provider/model> <message>` path, passes a per-message auth profile into reply execution, updates slash-command docs, and adds regression coverage for non-persistent model selection.

## Best Possible Solution

Keep the one-shot `/model` behavior, but compute defaults from a non-persistent one-shot selection state that cannot reapply or mutate stored session model/auth-profile overrides.

## Reproduction Assessment

Yes. Source inspection on current main shows mixed-message inline directives are stripped and then cleared, with `hasModelDirective` reset before model selection, so `/model openai/gpt-4.1-mini please sync now` runs on the existing/default model while removing the directive text.

## Solution Assessment

No. The proposed direction is right, but the current patch rebuilds model selection with live session state after the one-shot switch, which can corrupt persistent auth-profile state and compute thinking/reasoning defaults from the old session model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve session auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This rebuild runs after the one-shot model changes `provider`/`model`, but it still passes the live `sessionEntry` and `sessionStore`. `createModelSelectionState` clears an incompatible `authProfileOverride`, so a one-shot `/model other-provider/model message` can permanently delete the session's normal auth profile even though this path is supposed to be non-persistent.
  - confidence: 0.86
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:478-480`
  - body: The refreshed model state is created with `hasModelDirective: false` while still receiving the session override inputs. If the session already has a stored `/model` override, `createModelSelectionState` can reapply that stored model, so `resolveDefaultThinkingLevel()` and `resolveDefaultReasoningLevel()` use the session model instead of the one-shot model selected for this run.
  - confidence: 0.9
