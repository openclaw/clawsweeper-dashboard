## Summary

Keep this PR open: current main still drops mixed-message `/model` before model selection, so the bug is real, but the branch has unresolved session-state regressions, merge conflicts, and no after-fix real behavior proof.

## What This Changes

The PR preserves mixed-message `/model` as one-shot reply state, applies provider/model/auth-profile overrides for that run, updates slash-command docs, and adds regression coverage.

## Best Possible Solution

Rebase the branch and keep `/model <provider/model> <message>` as isolated per-message runtime state that does not reapply or mutate stored session model/auth-profile overrides, with regressions for existing session overrides.

## Reproduction Assessment

Yes. Source inspection on current main shows `/model ... <message>` is parsed and stripped, then `clearInlineDirectives()` removes model directive state before model selection can apply it; I did not run a live chat reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but the PR refreshes model selection through live session storage after the one-shot switch, so it can read or mutate persistent session state when the override should be non-persistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve session auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: After switching provider/model for the one-shot run, this refresh still passes `sessionEntry`, `sessionStore`, and `sessionKey` into `createModelSelectionState`. That helper clears incompatible stored auth profiles, so a mixed-message one-shot can permanently delete the session's saved auth profile even though the selection is supposed to be non-persistent.
  - confidence: 0.86
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:478-480`
  - body: The refreshed model state is created with `hasModelDirective: false` while still receiving session override inputs. If a stored `/model` override exists, thinking/reasoning defaults can come from the session model instead of the one-shot model used for this run.
  - confidence: 0.9
