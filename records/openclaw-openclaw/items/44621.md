## Summary

Keep open: current main still has the mixed-message `/model` gap and this PR remains the canonical implementation attempt, but the submitted branch is not merge-ready because it is conflicted, lacks real behavior proof, and introduces session/auth-provider isolation bugs.

## What This Changes

The PR adds one-shot mixed-message `/model <provider/model> <message>` handling through reply directives, optional per-message auth profile forwarding, slash-command docs, and regression tests.

## Best Possible Solution

Implement this on current main with request-local one-turn model/auth-profile state that never reads or mutates stored session overrides during the one-shot switch, then cover stored override interactions and add redacted real setup proof.

## Reproduction Assessment

Yes, at source level. Current main strips `/model ...` from mixed messages and then clears `hasModelDirective` before the model-selection call, so the selected provider/model cannot apply to that run.

## Solution Assessment

No. The target behavior is the right end state, but this patch rebuilds model selection through persisted session state after the one-shot switch; the safer fix is request-local state isolation with stored-override tests.

- **[P1] Preserve stored auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: After the one-shot switch changes `provider`/`model`, this rebuild passes live `sessionEntry`, `sessionStore`, and `storePath` into `createModelSelectionState`. That callee validates `sessionEntry.authProfileOverride` against the temporary provider and clears it on mismatch, so a single `/model other-provider/model message` can permanently remove the user's stored auth-profile override despite the feature being non-persistent.
  - confidence: 0.91

The run still uses the one-shot provider/model, but default thinking/reasoning can be resolved from the old session model instead of the selected one-shot model.
  - confidence: 0.88
