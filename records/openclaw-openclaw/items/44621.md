## Label Justifications

- `P2`: This fixes a real user-facing reply/model-selection bug with limited blast radius but important session and auth-provider implications.
- `merge-risk: 🚨 session-state`: The PR's one-shot refresh can read or mutate persisted session model/auth-profile state during a non-persistent override.
- `merge-risk: 🚨 auth-provider`: The PR threads per-message auth-profile selection and can clear or bypass stored auth-profile routing in the reply run.

## Summary

Keep open: the underlying bug still exists on current main, but this PR is not merge-ready because the proposed one-shot refresh can mutate stored auth/session state, can compute defaults from the wrong model, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

The PR preserves mixed-message `/model <provider/model> <message>` as a one-shot reply override, threads optional per-message auth-profile state, updates slash-command docs, and adds regression coverage.

## Best Possible Solution

Rebase onto current main, carry one-shot model/auth-profile selection as isolated run state that skips stored-session mutation, add stored model and auth-profile regression coverage, and provide redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows mixed `/model ... <message>` is parsed and stripped, then `clearInlineDirectives()` removes the model directive before model selection can apply it; I did not run a live chat reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but the PR rebuilds model selection through live session storage after the one-shot switch, so stored model and auth-profile state can affect or be mutated by a non-persistent override.

  - confidence: 0.9
