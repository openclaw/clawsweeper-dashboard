## Summary

Keep open: current main still lacks the reported mixed-message `/model` behavior, and this PR is the active implementation path, but the head has blocking correctness defects and still lacks real behavior proof from the contributor.

## What This Changes

The PR adds a one-shot inline `/model <provider/model> <message>` path through reply directive parsing, model selection, auth-profile propagation, docs, and regression tests.

## Best Possible Solution

Implement a dedicated non-persistent `/model ... <body>` selection path that strips the directive, applies provider/model/auth profile only to the current run, recomputes context and defaults from that selected runtime, and leaves stored session overrides untouched.

## Reproduction Assessment

Yes. Source inspection of current main shows `parseInlineDirectives()` extracts `/model`, then `resolveReplyDirectives()` replaces mixed-message directive state with `clearInlineDirectives()`, so the selected model is stripped from the prompt but never reaches model selection.

## Solution Assessment

No. The requested direction is right, but this patch is not the best implementation yet because it calls a helper with missing required inputs and recomputes through persistent session-selection state that can read or mutate stored overrides.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve session auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This rebuild runs after the one-shot model changes `provider`/`model`, but it still passes the live `sessionEntry` and `sessionStore`. `createModelSelectionState` clears an incompatible `authProfileOverride`, so a one-shot `/model other-provider/model message` can permanently delete the session's normal auth profile even though this path is supposed to be non-persistent.
  - confidence: 0.86
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:478-480`
  - body: The refreshed model state is created with `hasModelDirective: false` while still receiving the session override inputs. If the session already has a stored `/model` override, `createModelSelectionState` can reapply that stored model, so `resolveDefaultThinkingLevel()` and `resolveDefaultReasoningLevel()` use the session model instead of the one-shot model selected for this run.
  - confidence: 0.9
