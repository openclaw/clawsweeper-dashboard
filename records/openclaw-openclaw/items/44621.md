## Summary

Keep open: current main still has the reported mixed-message `/model` bug and this PR is the canonical fix candidate, but the branch is not mergeable as-is because it lacks external real behavior proof and introduces blocking one-shot model-selection defects.

## What This Changes

The PR adds one-shot mixed-message `/model <provider/model> <message>` handling across reply directives, auth-profile propagation, docs, and regression tests.

## Best Possible Solution

Implement one-shot `/model` through an explicitly non-persistent selection path that strips directive text, recomputes context/defaults from the selected provider/model, and preserves all session model/auth-profile overrides.

## Reproduction Assessment

Yes. Source inspection on current main shows `/model` is parsed and stripped, then mixed-message directive state is cleared before model selection, so the run stays on the session/default model.

## Solution Assessment

No. The requested direction is right, but this patch recomputes through persistent session-selection state and calls `resolveContextTokens` without required fields; a dedicated non-persistent path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass cfg and provider into one-shot context resolution:** `src/auto-reply/reply/get-reply-directives-apply.ts:277-280`
  - body: The new one-shot block calls `resolveContextTokens({ agentCfg, model })`, but the helper requires `cfg`, `agentCfg`, `provider`, and `model`. This should fail typechecking and also prevents provider-specific context tokens from being resolved for the selected model.
  - confidence: 0.95
- **[P1] Keep one-shot recompute from clearing stored auth profiles:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: After switching to the one-shot provider/model, this rebuild passes live `sessionEntry`, `sessionStore`, and `sessionKey` into `createModelSelectionState()`. For a cross-provider one-shot, that helper can clear the persistent auth-profile override, violating the non-persistent one-shot contract.
  - confidence: 0.9
- **[P2] Suppress stored overrides when rebuilding one-shot defaults:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This rebuild uses `hasModelDirective: false` while still passing session state, so a stored session model override can replace the rebuilt `modelState` before thinking/reasoning defaults are resolved. The run can use the one-shot model while defaults come from the old session model.
  - confidence: 0.9
