## Summary

Keep open: current main still has the reported `/model ... <message>` behavior gap, and this PR targets the right area, but the branch is not mergeable as-is because it lacks real behavior proof and introduces session-state regressions around one-shot model selection.

## What This Changes

The PR preserves mixed-message `/model` directives as one-shot provider/model/auth-profile state, threads that through reply execution, updates slash-command docs, and adds regression coverage.

## Best Possible Solution

Keep the one-shot `/model` behavior, but carry the selected model/auth profile as non-persistent per-message runtime state that does not reapply or mutate stored session overrides, with regression coverage for existing session model and auth-profile overrides.

## Reproduction Assessment

Yes. Source inspection on current main shows mixed `/model ... <message>` enters the inline-directive path, then `clearInlineDirectives(parsedDirectives.cleaned)` drops `hasModelDirective` before model selection, matching the reported fallback.

## Solution Assessment

No. The direction is right, but the patch rebuilds model selection against live session state after the one-shot switch, so it can mutate persistent auth-profile state and compute defaults from the wrong model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve session auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This refresh runs after the one-shot `/model` changes `provider`/`model`, but it still passes the live session entry and store. `createModelSelectionState` clears incompatible `authProfileOverride` values, so a one-shot switch to another provider can permanently delete the session's normal auth profile even though the path is meant to be non-persistent.
  - confidence: 0.86
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:478-480`
  - body: The refreshed model state is created with `hasModelDirective: false` while still receiving the session override inputs. If the session already has a stored `/model` override, that stored model can be reapplied, so thinking/reasoning defaults come from the session model instead of the one-shot model used for the run.
  - confidence: 0.9
