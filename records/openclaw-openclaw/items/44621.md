## Summary

Keep open: current main still has the mixed-message `/model` fallback bug, and this PR targets the right behavior, but the patch is not mergeable as written because it can reapply or mutate stored session state during a one-shot run and it lacks external real behavior proof.

## What This Changes

The PR preserves mixed-message `/model <provider/model> <message>` as one-shot reply state, threads provider/model/auth-profile selection into reply execution, updates slash-command docs, and adds regression tests.

## Best Possible Solution

Carry the selected model and auth profile as non-persistent per-message runtime state, recompute model-derived defaults from that state, and cover existing session model/auth-profile overrides with regressions.

## Reproduction Assessment

Yes. Source inspection on current main shows mixed `/model ... <message>` is stripped through `clearInlineDirectives(...)`, which clears model directive state before model selection can apply it.

## Solution Assessment

No. The direction is right, but the current patch rebuilds model selection against live session state after the one-shot switch, so it can mutate persistent auth-profile state and compute defaults from the wrong model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve session auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This refresh runs after the one-shot `/model` changes `provider`/`model`, but it still passes the live session entry and store. `createModelSelectionState` clears incompatible `authProfileOverride` values, so a one-shot switch to another provider can permanently delete the session's normal auth profile even though the path is meant to be non-persistent.
  - confidence: 0.86
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:478-480`
  - body: The refreshed model state is created with `hasModelDirective: false` while still receiving the session override inputs. If the session already has a stored `/model` override, that stored model can be reapplied, so thinking/reasoning defaults come from the session model instead of the one-shot model used for the run.
  - confidence: 0.9
