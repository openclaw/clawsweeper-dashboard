## Summary

Keep this PR open: current main still drops mixed `/model` directives, and the branch addresses the right bug, but the proposed implementation has blocking session-state/auth-profile defects and still lacks after-fix real behavior proof.

## What This Changes

The PR preserves mixed-message `/model <provider/model> <message>` as a one-shot reply override, threads per-message auth profile state, updates slash-command docs, and adds regression tests.

## Best Possible Solution

Rebase and repair the implementation so mixed `/model <provider/model> <message>` carries isolated per-message model/auth-profile state through the reply run, skips stored session override reads and mutations, and documents the non-persistent behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `/model ... <message>` is parsed and stripped, then `clearInlineDirectives()` removes the model directive before model selection can apply it; I did not run a live chat reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but the PR rebuilds model selection through live session storage after the one-shot switch, so stored model and auth-profile state can affect or be mutated by a non-persistent override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve session auth profiles during one-shot refresh:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: After switching provider/model for the one-shot run, this refresh still passes `sessionEntry`, `sessionStore`, and `sessionKey` into `createModelSelectionState`. That helper clears incompatible stored auth profiles, so a mixed-message one-shot can permanently delete the session's saved auth profile even though this selection is supposed to be non-persistent.
  - confidence: 0.86
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:478-480`
  - body: The refreshed model state is created with `hasModelDirective: false` while still receiving session override inputs. If a stored `/model` override exists, thinking/reasoning defaults can come from the session model instead of the one-shot model used for this run.
  - confidence: 0.9
