## Summary

Keep open: the PR targets a real delivery durability gap, but it still clears pending-final-delivery state before downstream user delivery succeeds, leaving the same loss window in restart, normal final-dispatch, and heartbeat replay paths.

## What This Changes

The PR adds pending final-delivery fields for session and subagent records, persists main-session final text, replays it during restart or heartbeat recovery, and adjusts subagent delivery retry state.

## Best Possible Solution

Keep the durable pending payload and retry metadata until the actual route/send success boundary observes delivery, then clear it from that success path and cover failure paths with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to simulate a route/send failure after restart recovery, after `runReplyAgent` returns, or after heartbeat replay returns; the PR has already cleared the only durable retry marker before the downstream delivery boundary can fail.

## Solution Assessment

No. The shared pending-delivery shape is a reasonable direction, but the clear point is too early; the safer implementation clears only after observed delivery success and preserves pending state on failures.
