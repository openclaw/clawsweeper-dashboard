## Summary

Keep open. The PR targets a real durability gap, and the linked #65037 discussion is useful prior context, but the current patch clears the new durable pending-final-delivery markers before the existing final-dispatch boundary has actually succeeded. It also lacks the required changelog entry for a user-visible delivery reliability change.

## What This Changes

The PR adds pending-final-delivery state for main sessions and subagent records, restart recovery replay for captured final payloads, heartbeat replay/deferral handling, and a focused main-session recovery test.

## Best Possible Solution

Keep the durable pending marker until the actual final delivery or agent-command delivery path reports success, preserve retry metadata on delivery failure, add focused failure-path tests for normal final dispatch and heartbeat replay, and add one Unreleased changelog line with appropriate contributor credit.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: the PR clears pendingFinalDelivery in runReplyAgent and heartbeat replay before dispatchReplyFromConfig reaches sendFinalPayload, so a route/send failure after resolver return leaves no retry marker.

## Solution Assessment

No. The direction is maintainable, but this patch clears durability state earlier than the current delivery boundary; the safer fix is to clear only after successful direct or queued final delivery and leave retry state intact on failure.
