## Summary

Keep open. The PR is aimed at a real durability gap, but the patch still clears the durable pending-final-delivery marker before the final payload has actually reached the user, and it is missing the required changelog entry for a user-visible reliability fix.

## What This Changes

The PR adds shared pending-final-delivery fields to session and subagent state, persists main-session final reply text, replays it through restart and heartbeat paths, and adds restart recovery coverage.

## Best Possible Solution

Keep the durable pending payload until actual final delivery succeeds, retain retry metadata on failures, and cover restart, heartbeat, and normal final-dispatch failure paths plus the changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to make routeReplyToOriginating or dispatcher.sendFinalReply fail after restart recovery, heartbeat replay, or runReplyAgent returns; the PR has already cleared the only durable retry marker before that boundary.

## Solution Assessment

No. The direction is maintainable, but clearing state from resolver/recovery code is too early; the safer solution is to clear only from the final delivery success boundary and leave the marker intact on failure.
