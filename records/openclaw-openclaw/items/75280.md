## Summary

Keep open: the PR improves durable final-delivery retry direction, but the latest diff still persists retry text before delivery suppression and visible-payload filtering, and it marks agent-command output even when no channel delivery was requested.

## What This Changes

The PR adds durable pending-final-delivery state for main and subagent sessions, restart/heartbeat replay paths, clearing after dispatch, focused tests, and a changelog entry.

## Best Possible Solution

Keep durable retry state, but persist only visible delivery-eligible final payloads when automatic channel delivery is intended, and clear markers only after proven route or queue success.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a message_tool_only or send-policy-denied turn that returns final text, or a reasoning final payload: the PR can persist that text before dispatch suppresses or filters it, and heartbeat replay returns it as ordinary text.

## Solution Assessment

No. The overall durable-retry direction is maintainable, but the capture and clear boundaries need to match the existing delivery contract instead of storing every final payload or local command result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate pending capture to deliverable automatic replies:** `src/auto-reply/reply/agent-runner.ts:1814`
  - body: The marker is written before dispatch knows whether automatic source delivery is suppressed. In message_tool_only or send-policy-denied turns, dispatch deliberately keeps final output private, so a restart after this write can make heartbeat replay text that should not be sent automatically.
  - confidence: 0.91
- **[P2] Filter hidden payloads before persisting retry text:** `src/auto-reply/reply/agent-runner.ts:1816-1818`
  - body: pendingText joins every final payload's text, but final dispatch skips isReasoning payloads. Heartbeat replay later returns plain text, so hidden reasoning or non-user-visible payloads can become visible unless the pending text is built from the same sendable final payload set.
  - confidence: 0.89
- **[P2] Only mark command payloads when delivery is requested:** `src/agents/agent-command.ts:1248`
  - body: This branch marks main-session agent command output for any payload, even when opts.deliver is false and the expected result is local or JSON output. A restart between the marker write and cleanup can replay local command output to the session's last channel.
  - confidence: 0.86
- **[P2] Clear command markers only after delivery succeeds:** `src/agents/agent-command.ts:1291`
  - body: The clear runs whenever deliverAgentCommandResult returns, but best-effort outbound delivery catches per-payload failures and returns after logging. That can erase the durable marker even though no route or queue success happened.
  - confidence: 0.82
