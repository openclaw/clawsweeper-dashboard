## Summary

Keep open. The latest branch fixes the earlier early-clear and changelog blockers, but the new durable capture point is still broader than the real delivery boundary and can persist suppressed, non-deliver, or hidden payload text for later heartbeat replay.

## What This Changes

The PR adds durable pending-final-delivery state for main/subagent sessions, replays main-session pending replies during restart/heartbeat recovery, clears markers after dispatch success, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the durable retry marker, but make its payload and intent match the actual automatic final-delivery path so suppressed, local-only, hidden, reasoning, error, and compaction payloads never become replayable.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a `message_tool_only` or send-policy-denied turn that returns final text: the PR writes `pendingFinalDeliveryText`, dispatch suppresses final delivery and never clears it, and a later heartbeat can return that text for user delivery.

## Solution Assessment

No. The overall direction is maintainable, but the capture gate is too broad; the safer fix is to persist only delivery-eligible final payloads when automatic final delivery is actually intended, then clear only after the real success boundary.

  - confidence: 0.88

  - body: This branch marks main-session agent command output as pending for any payload, even when `opts.deliver` is false and the expected result is local/JSON output. A restart between the marker write and cleanup can leave a durable marker that recovery sends to the session's last channel, so this should be gated on actual delivery intent.
  - confidence: 0.83
