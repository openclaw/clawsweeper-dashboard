## Summary

Keep open. The PR is useful source work for durable main-session final delivery, but it still clears the durable replay marker before the captured reply has actually reached the user on restart recovery, normal final dispatch, and heartbeat replay paths.

## What This Changes

The PR adds pending-final-delivery state to session/subagent records, persists main-session final reply text for recovery, replays it through restart/heartbeat paths, narrows heartbeat deferral, and adds main-session recovery coverage.

## Best Possible Solution

Keep the durable pending payload until route/send success is observed for restart, heartbeat, and normal final delivery, preserve retry metadata on failures, and document the user-visible reliability fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to simulate a route/send failure after restart recovery, runReplyAgent return, or heartbeat replay return; the PR has already cleared the only pending marker before the downstream delivery boundary can fail.

## Solution Assessment

No. The direction is maintainable, but the clearing point is too early; the safer fix is to clear only from the actual delivery success boundary and leave the marker intact on failure.
