## Summary

Keep open. Current main still lacks durable main-session pending-final-delivery recovery, so the PR is addressing a real gap, but the proposed patch clears the new retry markers before the actual delivery boundary and is missing the required user-facing changelog entry.

## What This Changes

The PR adds pending-final-delivery fields to session and subagent records, persists final payload text for main-session delivery/restart recovery, replays pending text on heartbeat/recovery paths, narrows heartbeat deferral, and adds a main-session recovery unit test.

## Best Possible Solution

Keep the durable pending marker until the actual final dispatch or agent-command delivery call reports success, preserve retry metadata on delivery failure, add focused failure-path tests for normal final delivery and heartbeat replay, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: with the PR patch, simulate a route/send failure after runReplyAgent or heartbeat replay returns its payload; the pending marker has already been cleared before sendFinalPayload can fail.

## Solution Assessment

No. The direction is maintainable, but the clearing point is too early; the safer fix is to clear only at the downstream delivery success boundary and leave the marker intact on route/send failure.
