## Summary

Keep open. The latest PR head still captures retry text before the existing delivery visibility policy decides whether a reply is sendable, and the command path still writes or clears retry markers outside confirmed channel-delivery semantics.

## What This Changes

The PR adds durable pending-final-delivery state for main sessions and subagents, heartbeat/restart replay paths, dispatch clearing, tests, and a changelog entry.

## Best Possible Solution

Keep durable retry state, but derive it from the same policy-eligible sendable payload set used by final dispatch and clear it only after confirmed queued or routed outbound delivery.

## Reproduction Assessment

Yes. Static reproduction paths are a message_tool_only or sendPolicy-denied final reply, a reasoning final payload, or an agent command run with deliver false or best-effort delivery failure; the PR stores or clears retry state before those paths are proven visible and delivered.

## Solution Assessment

No. Durable delivery retry is the right direction, but this patch places capture and cleanup outside the existing delivery contract; the safer implementation is to capture only dispatch-eligible payloads and clear only after confirmed outbound success.
