## Summary

Keep open. The durable-delivery direction is useful, but the patch still clears the retry marker before the actual user-delivery boundary in restart recovery, normal final dispatch, and heartbeat replay paths.

## What This Changes

The PR adds session/subagent pending-final-delivery state, main-session restart and heartbeat replay handling, and targeted restart recovery coverage for durable final response delivery.

## Best Possible Solution

Land a narrower durable-delivery patch that persists pending final payloads but clears them only at verified delivery success boundaries, with failure-path tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to simulate a route/send failure after restart recovery returns, after runReplyAgent returns, or after heartbeat replay returns; the PR clears the durable marker before those downstream delivery attempts can fail.

## Solution Assessment

No. Persisting the payload is the right direction, but the clear point is too early; the safer solution is to clear only when final channel or dispatcher delivery succeeds and to preserve retry metadata on failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep restart marker until delivery succeeds:** `src/agents/main-session-restart-recovery.ts:154-160`
  - body: `resumeMainSession` calls the agent API with `deliver: false`, then clears the pending final-delivery fields as soon as that call returns. That only proves the recovery turn was accepted, not that the captured final answer reached the user, so a failed later replay can still lose the only durable marker.
  - confidence: 0.9
- **[P2] Preserve pending state until final dispatch succeeds:** `src/auto-reply/reply/agent-runner.ts:1838-1847`
  - body: This clears `pendingFinalDelivery` immediately after `finalizeWithFollowup`, but that helper only returns the resolver value; final channel delivery happens later in `dispatchReplyFromConfig`. If route/send fails after the resolver returns, the retry marker has already been removed.
  - confidence: 0.92
- **[P2] Keep heartbeat replay marker until send succeeds:** `src/auto-reply/reply/get-reply.ts:321-341`
  - body: The heartbeat branch clears the pending fields before returning the replay text to the dispatcher. A later final delivery failure has nothing left to retry, which defeats the durable replay path this PR is adding.
  - confidence: 0.9

  - body: This is a user-visible delivery reliability fix, so repo policy requires a single-line Unreleased changelog entry under Fixes before merge.
