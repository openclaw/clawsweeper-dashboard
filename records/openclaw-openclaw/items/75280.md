## Summary

Keep open. The PR addresses a valid main-session delivery durability problem, but the current head still has blocking delivery-boundary issues: command retries persist raw payload text before sendable filtering, and heartbeat replay returns pending text without preserving the original delivery route.

## What This Changes

This PR adds durable pending-final-delivery state for sessions and subagents, restart/heartbeat retry handling, dispatch clearing, tests, and a changelog entry for main-session final delivery recovery.

## Best Possible Solution

Keep durable retry state, but persist only the already-filtered sendable outbound payload plus the original delivery context, then replay and clear it through the same delivery dispatcher used by the original final reply.

## Reproduction Assessment

Yes. Source inspection of PR head shows raw command payload capture before outbound filtering and heartbeat replay of pending text without route context; no live channel run is needed to identify those paths.

## Solution Assessment

No. Durable retry is the right direction, but this patch must preserve the original delivery route and reuse sendable-payload filtering before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Replay pending replies through the original route:** `src/auto-reply/reply/get-reply.ts:343`
  - body: The heartbeat replay path returns stored pendingFinalDeliveryText as a normal reply, while the PR does not populate or consume the original delivery context. A delivered main-session run can target a separate channel/recipient, but a later heartbeat may replay the stored text through whatever route the heartbeat resolves instead of the failed original target.
  - confidence: 0.87
- **[P2] Filter command retry text through sendable payloads:** `src/agents/agent-command.ts:1258-1261`
  - body: This persists every raw payload's text before deliverAgentCommandResult builds the outbound plan. That plan suppresses isReasoning payloads, so a restart or delivery failure after this write can make hidden reasoning text replay later as a plain heartbeat reply.
  - confidence: 0.88
