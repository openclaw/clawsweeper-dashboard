## Summary

Keep open. Current main does not implement durable main-session final delivery, and the PR is a useful scoped attempt, but the latest patch still clears the new pending-delivery marker before downstream final delivery has actually succeeded; it also lacks the required user-facing changelog entry.

## What This Changes

The PR adds shared pending-final-delivery state for subagent and main-session records, persists main-session final payload text, replays it through restart or heartbeat recovery paths, narrows heartbeat deferral, and adds restart recovery coverage.

## Best Possible Solution

Keep the PR open and finish it by clearing pending-delivery state only after the downstream final dispatch or agent-command delivery boundary reports success, preserving retry metadata on send failure, adding focused failure-path tests, and adding a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. With the PR head, simulate a route/send failure after `runReplyAgent` returns its final payload or after heartbeat replay returns `{ text }`; the pending marker has already been cleared before the actual send boundary can fail.

## Solution Assessment

No. The durable-delivery direction is maintainable, but the current clearing point is too early; the safer solution is to clear only at the downstream delivery success boundary and keep the marker intact on route/send failure.
