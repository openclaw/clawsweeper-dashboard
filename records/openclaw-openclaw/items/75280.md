## Summary

Keep open. The latest head addresses the earlier dispatch-boundary findings and adds the changelog entry, but it still has two concrete durable-delivery regressions around non-delivery restart recovery and non-heartbeat user turns.

## What This Changes

The PR adds durable pending-final-delivery fields, restart/heartbeat replay handling, dispatch-time marker clearing, subagent pending payload persistence, tests, and a changelog entry for main-session final reply recovery.

## Best Possible Solution

Keep the shared durable marker direction, but preserve existing pending payloads until actual routed/queued delivery success and replay or queue them before any new main-session turn can overwrite them.

## Reproduction Assessment

Yes. Seed a main session with pendingFinalDeliveryText, then either let restart recovery invoke the gateway with deliver:false or send a normal user message before heartbeat replay; the PR paths can clear or overwrite the marker before the captured answer reaches the user.

## Solution Assessment

No. The direction is maintainable, but the pending marker must be written and cleared only around real delivery attempts/success, with explicit handling for non-heartbeat user turns that arrive while a payload is pending.
