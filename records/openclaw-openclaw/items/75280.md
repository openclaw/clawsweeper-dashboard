## Summary

Keep open. The PR is useful and the latest head addresses some prior durable-delivery blockers, but it still stores/replays pending main-session text outside the original delivery route and command-payload visibility contract.

## What This Changes

The PR adds durable pending-final-delivery state, heartbeat/restart replay, dispatch clearing, subagent pending-delivery persistence, focused tests, and a changelog entry for main-session delivery recovery.

## Best Possible Solution

Keep durable retry state, but store only route-bound, sendable pending payloads and replay/clear them through the same delivery contract that the original final reply used.

## Reproduction Assessment

Yes. Source inspection shows command retries capture raw payload text before delivery filtering, and heartbeat replay returns pending text without using original delivery context; no live channel run was needed to identify these paths.

## Solution Assessment

No. Durable retry is the right direction, but this patch must preserve the original delivery route and reuse the same sendable-payload filtering before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Replay pending replies through the original route:** `src/auto-reply/reply/get-reply.ts:343`
  - body: The heartbeat replay path returns stored `pendingFinalDeliveryText` as a normal reply, while the main-session markers added by this PR do not save or use the original delivery context. A delivered CLI run can target Slack while sharing the agent's main session; after a failed send or restart, the next heartbeat for that session can post the stored text to whatever route the heartbeat resolves instead of the original target.
  - confidence: 0.87
- **[P2] Filter command retry text through sendable payloads:** `src/agents/agent-command.ts:1258-1261`
  - body: This persists every raw payload's text before `deliverAgentCommandResult` builds its outbound plan. That plan suppresses `isReasoning` payloads, so a restart or delivery failure after this write can make hidden reasoning text replay later as a plain heartbeat reply.
  - confidence: 0.88
