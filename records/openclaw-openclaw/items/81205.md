## Summary

Keep open: current main and the latest release still register `openclaw wiki` against one resolved memory-wiki vault path, with no CLI path/name/env selector for office/life vault maintenance. The request is useful, but it adds public CLI/env/config behavior and should stay with maintainer product review rather than conservative cleanup.

## What This Changes

The issue requests a way for `openclaw wiki` commands to target non-default memory-wiki vaults, such as `office` or `life`, via a path, name, or environment selector.

## Best Possible Solution

Define one maintainer-approved memory-wiki vault-selection contract for CLI/admin flows, coordinate it with per-agent vault isolation, then cover status, ingest, compile, lint, docs, and focused CLI tests.

## Reproduction Assessment

Yes, at source level. Current main resolves one memory-wiki `vault.path` before `openclaw wiki` subcommands run and does not register the requested path/name/env selector; I did not execute a live CLI command in this read-only review.

## Solution Assessment

Unclear. A CLI `--path` override may be the narrow cron-job fix, but named vaults and an environment default need maintainer agreement so CLI, tools, gateway methods, and per-agent vault isolation do not grow competing contracts.
