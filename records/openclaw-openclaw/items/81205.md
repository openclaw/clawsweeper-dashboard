## Summary

Keep open. Current main and the latest release still bind `openclaw wiki` to one resolved memory-wiki vault path, and the related open runtime-templating PR does not add the requested CLI-level vault selector.

## What This Changes

The issue requests per-invocation vault selection for `openclaw wiki`, such as `--path`, `--wiki`, or `OPENCLAW_WIKI_PATH`, so automation can operate multiple memory-wiki vaults.

## Best Possible Solution

Define one supported memory-wiki vault-selection contract, likely starting with a path override plus explicit precedence rules, then update the CLI implementation, docs, and focused CLI/config tests.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.5.7` shows `openclaw wiki` resolves one memory-wiki config and registers no `--path`, `--wiki`, or `OPENCLAW_WIKI_PATH` selector.

## Solution Assessment

Unclear. The need is concrete, but maintainers should choose the supported selector surface and precedence before implementation because arbitrary paths, named vaults, and env defaults have different safety and UX implications.
