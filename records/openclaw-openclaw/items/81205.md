## Summary

Keep open. Current main still does not provide a `openclaw wiki` CLI or environment selector for alternate memory-wiki vaults, and the adjacent open vault-templating PR targets runtime/tool context rather than this direct CLI UX.

## What This Changes

Request adding `openclaw wiki` vault selection through flags such as `--path`/`--wiki` or an environment default so maintenance commands can target office, life, or custom vaults.

## Best Possible Solution

Define a plugin-owned CLI vault-selection contract, then update `registerWikiCli`, lazy CLI metadata, config/schema or env handling as needed, docs, and focused tests in coordination with https://github.com/openclaw/openclaw/pull/67584.

## Reproduction Assessment

Yes, source-level. Current main registers `openclaw wiki` with one captured memory-wiki config, and there is no registered `--path`, `--wiki`, named-vault, or `OPENCLAW_WIKI_PATH` selector in code or docs.

## Solution Assessment

Unclear. A per-invocation path or named-vault override is plausible, but maintainers should choose the supported selector surface and precedence before implementation, especially relative to context-aware vault templating work.
