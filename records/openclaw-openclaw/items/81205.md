## Summary

Keep open. Current main and the latest shipped tag still wire `openclaw wiki` to one configured memory-wiki vault path, so the requested per-command vault selector remains a real feature/config request rather than cleanup-close material.

## What This Changes

Add per-invocation memory-wiki vault selection for commands such as status, ingest, compile, and lint.

## Best Possible Solution

Define a supported memory-wiki vault-selection contract, likely starting with a narrow per-invocation path override, then update CLI help, docs, and focused tests once maintainers choose path/env/named-vault precedence.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.5.7` shows wiki commands are bound to one resolved plugin config and no `--path`, `--wiki`, or `OPENCLAW_WIKI_PATH` selector is registered.

## Solution Assessment

Unclear. The need is concrete, but maintainers should choose the supported selector surface and precedence before implementation because `--path`, `--wiki`, and an environment default have different config and safety implications.
