## Summary

Keep open: current main and v2026.6.10 still resolve `openclaw wiki` against one memory-wiki vault path with no path, name, or environment selector. The request is useful, but it adds public CLI/env/config behavior and should stay with maintainer product review, coordinated with per-agent memory-wiki isolation.

## What This Changes

Request support for selecting a non-default memory-wiki vault from the `openclaw wiki` CLI, such as by path, vault name, or environment default.

## Best Possible Solution

Define one maintainer-approved memory-wiki vault-selection contract for CLI/admin flows, coordinate it with per-agent vault isolation, then cover status, ingest, compile, lint, docs, and focused CLI tests.

Yes, at source level: current main resolves one memory-wiki `vault.path` before wiki subcommands run and does not register the requested path, wiki-name, or env selector. I did not execute a live CLI command because this was a read-only review.

Unclear: a CLI `--path` override may be the narrow cron-job fix, but named vaults and an environment default need maintainer agreement so CLI, tools, gateway methods, and per-agent vault isolation do not grow competing contracts.
