## Summary

This PR addresses a credible Slack message-loss bug, but the post-render regex also rewrites backtick spans inside quoted fenced code blocks, so the patch is not safe to merge yet; the supplied formatter-only output also does not prove the fix in Slack itself.

## What This Changes

The PR post-processes Slack mrkdwn to replace inline-code delimiters with bold delimiters on blockquote lines, adds formatter regression tests, and adds a standalone reproduction script.

## Best Possible Solution

Apply the Slack workaround while blockquote and inline-code spans are still distinguishable in Markdown IR, preserving fenced code blocks, literal backticks, non-blockquote code, and chunk-size guarantees, then verify the result with a redacted real Slack send.

## Reproduction Assessment

Partly: the linked issue gives a credible controlled reproduction in a real Slack workspace, but this review did not establish a current-main live Slack reproduction and the PR only verifies local formatter output.

## Solution Assessment

No. A regex over rendered text loses the IR distinction between inline code and code-block content; a structure-aware rendering change is the narrower and safer fix.

- **[P1] Preserve backticks inside quoted fenced code blocks:** `extensions/slack/src/format.ts:155-156`
  - body: This regex runs after rendering and therefore treats literal backticks inside a blockquoted fenced code block as inline-code delimiters. Perform the substitution while the IR still distinguishes `code` from `code_block`, or explicitly exclude code-block ranges, and add a regression covering this case.
