## Summary

Keep open: current main still has the reported Control UI mismatch. The composer plus button is labeled as creating a new session, but the handler sends `/new` through the reset-confirmation path for the active chat, while the Gateway protocol already has a separate `sessions.create` concept for a real session entry.

## What This Changes

The issue requests clearer Control UI wording so the composer plus action is distinguished as resetting context rather than creating a separate session entry.

## Best Possible Solution

Update the Control UI so the existing composer plus action is explicitly presented as a reset-context action, including tooltip, aria label, confirmation text, and tests. If maintainers want a real new conversation container from the same area, that should be a separate `sessions.create`-backed control rather than leaving the reset action labeled as session creation.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main renders the composer plus as `New session`, sends `/new` with reset confirmation, and does not call `sessions.create` from that composer path.

## Solution Assessment

Yes for the reported UX bug. The narrowest maintainable solution is to relabel and test the existing reset-context behavior; adding a true new-session-entry action would be a separate product decision.
