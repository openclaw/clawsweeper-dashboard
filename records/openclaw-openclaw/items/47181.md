## Summary

Keep open: the feature is not on current main, and the PR still has source-level correctness blockers, missing inspectable real Azure proof, and an unresolved provider-ownership question because current main routes Foundry onboarding through the Microsoft Foundry plugin.

## What This Changes

This PR adds a core Azure Claude AI Foundry onboarding auth choice with CLI flags/env vars, credential storage, provider config wiring, docs, and tests.

## Best Possible Solution

Implement Azure Claude support in the Microsoft Foundry provider or a dedicated provider plugin with strict endpoint validation, generic token compatibility, docs, tests, and redacted live proof; keep the custom-provider workaround until then.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows generic `--token` input is ignored for the API-key-mapped Azure path, and malformed Azure Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow.

## Solution Assessment

No. The capability is useful, but this core auth branch is not the narrowest maintainable boundary while provider auth/onboarding is plugin-owned and Microsoft Foundry already owns Azure setup.
