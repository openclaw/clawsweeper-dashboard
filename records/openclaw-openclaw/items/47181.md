## Summary

Keep open: current main still does not implement the requested Azure Claude onboarding path, and this PR remains blocked by source-level correctness issues, missing real behavior proof, and an unresolved provider-ownership decision.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding path with CLI flags, env vars, base URL normalization, auth-profile storage, config wiring, docs, and tests.

## Best Possible Solution

Keep Azure Claude support provider-owned, with strict endpoint validation; land the narrower Microsoft Foundry guardrail if maintainers choose that immediate path, then add full Anthropic Foundry support in the owning provider or a dedicated plugin.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows generic `--token` is ignored for the API-key-mapped Azure path, and malformed Azure Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow.

## Solution Assessment

No. The capability is useful, but adding a core Azure-specific auth branch is not the narrowest maintainable solution while Microsoft Foundry owns the adjacent provider surface and a linked guardrail PR tracks the immediate bug.

  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` maps to `anthropic-azure-api-key`, but it resolves only `anthropicAzureApiKey`. Scripted setup with generic `--token` is ignored and falls back to env or prompt, unlike other API-key provider flows.
  - confidence: 0.9
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-63`
  - body: For full URL input, any path that does not end with `/anthropic` gets `/anthropic` appended. A pasted endpoint like `https://fabric-hub.services.ai.azure.com/anthropic/v1` becomes `.../anthropic/v1/anthropic`, storing a malformed base URL and deferring failure to runtime.
  - confidence: 0.88
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch checks only that the host ends with `.services.ai.azure.com`, so `https://fabric_hub.services.ai.azure.com/anthropic` can pass even though the bare resource path rejects underscores. Apply the same resource-label validation before accepting full URLs.
  - confidence: 0.86
