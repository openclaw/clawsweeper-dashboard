## Summary

Keep open: current main and the latest release do not implement this Azure Claude onboarding path, and the PR still has actionable P2 correctness blockers plus missing real behavior proof and unresolved provider-ownership direction.

## What This Changes

The PR adds a core Azure Claude AI Foundry onboarding auth choice with CLI flags, environment variables, credential storage, provider config wiring, docs, and tests.

## Best Possible Solution

Settle provider ownership first, then implement the capability in the Microsoft Foundry provider or a dedicated provider plugin with strict endpoint validation, generic token compatibility, docs, and redacted real Azure proof; keep the custom-provider workaround documented until then.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows generic `--token` input is ignored for the API-key-mapped Azure path, and malformed Azure Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow.

## Solution Assessment

No. The capability is useful, but adding a core `anthropic-azure` auth branch is not the narrowest maintainable direction while current main has plugin-owned provider auth choices and Microsoft Foundry already owns Azure onboarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor generic --token input for Azure API-key auth:** `src/commands/auth-choice.apply.api-providers.ts:176-178`
  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` maps to `anthropic-azure-api-key`, but it passes only `params.opts?.anthropicAzureApiKey` to the shared helper. Scripted generic `--token` setup is ignored and falls back to env or prompt, unlike the LiteLLM API-key path.
  - confidence: 0.9
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-62`
  - body: For full URL input, any path that does not already end in `/anthropic` gets `/anthropic` appended. A pasted endpoint like `https://fabric-hub.services.ai.azure.com/anthropic/v1` becomes `.../anthropic/v1/anthropic`, storing a malformed base URL instead of failing during onboarding.
  - confidence: 0.88
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch only checks the `.services.ai.azure.com` suffix, so `https://fabric_hub.services.ai.azure.com/anthropic` passes even though the bare resource path rejects underscores. Validate the resource label before accepting the full URL.
  - confidence: 0.86
