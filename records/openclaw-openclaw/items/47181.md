## Summary

Keep open for maintainer handling: the Azure Claude onboarding capability is not first-class on current main, but this PR still has correctness blockers, lacks inspectable real behavior proof, and overlaps an active Microsoft Foundry ownership discussion.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding path with CLI flags, env vars, URL normalization, credential storage, provider config, tests, and Anthropic provider docs.

## Best Possible Solution

Implement Azure Claude through the provider-owned path, with strict Foundry endpoint validation, generic API-key semantics, focused tests/docs, and redacted real Azure onboarding proof.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows the generic `--token` value is ignored for the mapped Azure API-key path, and malformed Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow in this read-only review.

## Solution Assessment

No. The capability is useful, but this patch adds a core Azure-specific branch with correctness blockers where the maintainable direction is provider-owned Microsoft Foundry support or a dedicated provider plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor generic token input for Azure API-key auth:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` normalizes to `anthropic-azure-api-key`, but it passes only `anthropicAzureApiKey` into the API-key resolver. Scripted setup that supplies the generic `--token` is ignored and falls back to env or prompt, unlike the existing API-key provider flow.
  - confidence: 0.9
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:60-62`
  - body: For full URL input, any path that does not end with `/anthropic` gets `/anthropic` appended. A pasted endpoint like `https://fabric-hub.services.ai.azure.com/anthropic/v1` becomes `.../anthropic/v1/anthropic`, storing a malformed base URL and deferring failure to runtime.
  - confidence: 0.88
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch only checks that the host ends with `.services.ai.azure.com`, so `https://fabric_hub.services.ai.azure.com/anthropic` can pass even though the bare resource path rejects underscores. Apply the same resource-label validation before accepting full URLs.
  - confidence: 0.86
