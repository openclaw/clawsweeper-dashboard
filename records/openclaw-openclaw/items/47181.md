## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire Azure setup into the current provider flow:** `src/commands/auth-choice-options.static.ts:23-30`
  - body: On current main, `applyAuthChoice` only invokes loaded plugin/provider setup flows and then returns the unchanged config for unmatched non-legacy choices. This static picker option can therefore appear in interactive onboarding while the Azure implementation added to `applyAuthChoiceApiProviders` is never called, so selecting it would not write the Azure profile or provider config. Move this into the current provider setup-flow contract or explicitly rewire the current applier.
  - confidence: 0.91
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:57-61`
  - body: The URL branch appends `/anthropic` to any path that does not already end with that suffix. Inputs like `https://fabric-hub.services.ai.azure.com/anthropic/v1` or `/foo` would be saved as malformed bases, and OpenClaw's Anthropic transport will later append `/v1/messages`, producing a bad runtime URL instead of failing onboarding.
  - confidence: 0.94
- **[P2] Validate the Azure resource label in URL input:** `src/commands/anthropic-azure-utils.ts:50-55`
  - body: Full URL input only checks that the hostname ends with `.services.ai.azure.com`, so a host such as `fabric_hub.services.ai.azure.com` passes even though the bare-resource path rejects underscores. Apply the same resource-label validation to the host branch so invalid Azure resource names fail during onboarding rather than at the first model request.
  - confidence: 0.92
