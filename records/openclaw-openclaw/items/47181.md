## Summary

Keep open: the Azure Claude onboarding feature is not implemented on current main, and this PR still has correctness blockers plus missing real behavior proof. The useful direction needs maintainer provider-ownership judgment because current Microsoft Foundry support is plugin-owned and OpenAI-compatible, while the linked guardrail work remains separate.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding path with CLI flags, env vars, URL normalization, credential storage, provider config, tests, and Anthropic provider docs.

## Best Possible Solution

Implement Azure Claude through the provider-owned Microsoft Foundry path or a dedicated provider plugin, with strict Foundry endpoint validation, generic API-key semantics, focused tests/docs, and redacted real Azure onboarding proof.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows generic `--token` is ignored for the mapped Azure API-key path, and malformed Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow.

## Solution Assessment

No. The capability is useful, but this patch adds a core Azure-specific branch with correctness blockers; the maintainable direction is provider-owned Microsoft Foundry support or a dedicated provider plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor generic token input for Azure API-key auth:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` maps to `anthropic-azure-api-key`, but it passes only `anthropicAzureApiKey` into API-key resolution. Scripted setup with `--token` is ignored and falls back to env or prompt, unlike other API-key provider flows.
  - confidence: 0.9
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:60-62`
  - body: For full URL input, any path that does not end with `/anthropic` gets `/anthropic` appended. A pasted endpoint like `https://fabric-hub.services.ai.azure.com/anthropic/v1` becomes `.../anthropic/v1/anthropic`, storing a malformed base URL and deferring failure to runtime.
  - confidence: 0.88
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch only checks that the host ends with `.services.ai.azure.com`, so `https://fabric_hub.services.ai.azure.com/anthropic` can pass even though the bare resource path rejects underscores. Apply the same resource-label validation before accepting full URLs.
  - confidence: 0.86
