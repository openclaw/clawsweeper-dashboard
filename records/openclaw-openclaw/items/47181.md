## Summary

Keep open: this is a useful Azure Claude onboarding feature, but the PR still has source-level P2 blockers, missing real behavior proof, merge conflicts, and an unresolved ownership question because current main routes provider onboarding through plugin-owned Microsoft Foundry surfaces rather than a new core auth branch.

## What This Changes

Adds a core Azure Claude AI Foundry onboarding auth choice with CLI/env inputs, base URL normalization, auth-profile storage, provider config wiring, docs, and tests.

## Best Possible Solution

Implement full Claude-on-Foundry support in the Microsoft Foundry provider or a dedicated provider plugin, with strict endpoint validation, generic token compatibility, and redacted real Azure proof; keep the custom-provider workaround documented until then.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows generic `--token` input is ignored for the API-key-mapped Azure path, and malformed Azure Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow.

## Solution Assessment

No. The capability is useful, but adding a core `anthropic-azure` auth branch is not the narrowest maintainable direction while current main has plugin-owned provider auth choices and Microsoft Foundry owns Azure onboarding.

  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` maps to `anthropic-azure-api-key`, but it resolves only `anthropicAzureApiKey`. Scripted setup with generic `--token` is ignored and falls back to env or prompt, unlike other API-key provider flows.

  - confidence: 0.88
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch checks only that the host ends with `.services.ai.azure.com`, so `https://fabric_hub.services.ai.azure.com/anthropic` can pass even though the bare resource path rejects underscores. Apply the same resource-label validation before accepting full URLs.
  - confidence: 0.86
