## Summary

Keep open: current main still lacks first-class Azure Claude onboarding, and the branch is not merge-ready because the PR diff accepts malformed Azure hosts/paths, ignores generic `--token` for the API-key alias path, and needs a maintainer decision on provider ownership now that onboarding auth choices are plugin-owned.

## What This Changes

The PR adds a hard-coded `anthropic-azure-api-key` onboarding choice with Azure AI Foundry CLI flags, env-var resolution, API-key profile storage, config wiring, docs, and non-interactive tests.

## Best Possible Solution

Implement Azure Claude through a provider-owned setup path, likely in `extensions/microsoft-foundry` if it owns Anthropic deployments or in a dedicated provider plugin if the API/auth shape is distinct, using manifest auth metadata, strict endpoint validation, generic token/SecretRef support, docs, tests, and a changelog entry.

## Reproduction Assessment

Yes. Static inspection of the PR head reproduces the malformed URL normalization and generic-token routing gaps, and current-main search verifies the requested first-class auth choice is absent.

## Solution Assessment

No. The requested UX is reasonable, but the current branch hard-codes a new core provider path even though current main has plugin-owned provider auth metadata and Microsoft Foundry ownership needs a maintainer decision.
