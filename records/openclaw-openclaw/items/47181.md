## Summary

Keep this PR open: the Azure Claude onboarding problem is valid and current main still lacks this first-class path, but the branch adds a core-owned provider/auth surface with remaining URL and token-handling bugs and needs provider-ownership review before merge.

## What This Changes

Adds an `anthropic-azure-api-key` onboarding choice with Azure Claude CLI flags, env-var handling, URL normalization, auth-profile/config wiring, docs, and non-interactive tests.

## Best Possible Solution

Implement Azure Claude through a provider-owned setup path, likely in `extensions/microsoft-foundry` if it owns Anthropic deployments or in a dedicated provider plugin if the API/auth shape is distinct, with strict host/path validation and generic token/SecretRef support.

## Reproduction Assessment

Yes. The PR diff gives static reproductions for malformed full URLs such as `https://fabric-hub.services.ai.azure.com/anthropic/v1`, invalid full-URL hosts such as `fabric_hub.services.ai.azure.com`, and `--auth-choice apiKey --token-provider anthropic-azure --token ...`; current-main search also verifies the first-class path is absent.

## Solution Assessment

No. The requested UX is reasonable, but adding a new hard-coded core provider path is not the narrowest maintainable solution now that provider auth choices, onboarding flags, and non-interactive setup can be owned by provider plugins.
