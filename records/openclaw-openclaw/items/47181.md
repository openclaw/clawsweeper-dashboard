## Summary

Keep open: current main still does not provide Azure Claude/Anthropic Foundry onboarding, and the PR remains blocked by correctness findings, missing real behavior proof, and a provider-ownership decision.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding choice with Azure Claude flags/env vars, base URL normalization, auth-profile storage, provider config wiring, docs, and focused onboarding tests.

## Best Possible Solution

Keep the full Azure Claude capability provider-owned: either add Anthropic Foundry support to `microsoft-foundry` with strict endpoint validation, or land the narrower guardrail in https://github.com/openclaw/openclaw/pull/72446 while documenting the custom-provider workaround.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows generic `--token` is ignored for the API-key-mapped Azure path, and malformed Azure Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow.

## Solution Assessment

No. The requested capability is useful, but adding a core Azure-specific auth branch is not the narrowest maintainable path while Microsoft Foundry owns the adjacent Azure provider surface and an open linked PR already tracks the immediate guardrail.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor generic token input for Azure API-key auth:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` maps to `anthropic-azure-api-key`, but it resolves only `anthropicAzureApiKey`. Scripted setup with generic `--token` is ignored and falls back to env or prompt, unlike other API-key provider flows.
  - confidence: 0.9
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-62`
  - body: For full URL input, any path that does not end with `/anthropic` gets `/anthropic` appended. A pasted endpoint like `https://fabric-hub.services.ai.azure.com/anthropic/v1` becomes `.../anthropic/v1/anthropic`, storing a malformed base URL and deferring failure to runtime.
  - confidence: 0.88
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch checks only that the host ends with `.services.ai.azure.com`, so `https://fabric_hub.services.ai.azure.com/anthropic` can pass even though the bare resource path rejects underscores. Apply the same resource-label validation before accepting full URLs.
  - confidence: 0.86
