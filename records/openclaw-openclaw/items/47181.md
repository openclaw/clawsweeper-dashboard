## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor generic --token input for Azure API-key auth:** `src/commands/auth-choice.apply.api-providers.ts:176-178`
  - body: When users run an API-key mapped flow such as `--auth-choice apiKey --token-provider anthropic-azure --token ...`, this branch is selected but the supplied `opts.token` is never passed to the API-key resolver. Scripted onboarding then falls back to env/prompt and can fail or store the wrong credential even though the user provided the key.
  - confidence: 0.93
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-62`
  - body: For full URL input, paths that do not already end with `/anthropic` are blindly suffixed, so `https://resource.services.ai.azure.com/anthropic/v1` becomes `/anthropic/v1/anthropic` and is saved into config. That should fail during onboarding instead of producing a runtime-only provider failure.
  - confidence: 0.92
- **[P2] Validate Azure resource labels in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-58`
  - body: The URL branch accepts any hostname ending in `.services.ai.azure.com`, so invalid resource labels such as underscores can be persisted even though the bare-resource path rejects them. Apply the same label validation to the URL host before writing provider config.
  - confidence: 0.89
