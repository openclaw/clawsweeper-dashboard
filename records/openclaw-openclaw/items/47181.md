## Summary

This PR should remain open because it adds a new Azure Claude onboarding feature that is not implemented on current main, and the PR head still has concrete correctness blockers plus missing inspectable real behavior proof. The related Microsoft Foundry Claude-routing issue and guardrail PR are also still open, so this needs maintainer/product handling rather than conservative cleanup closure.

## What This Changes

The PR adds an `anthropic-azure-api-key` onboarding choice with CLI flags, environment variables, Azure Foundry URL/model helpers, auth profile storage, non-interactive handling, tests, and Anthropic provider docs.

## Best Possible Solution

Keep Azure Claude support plugin-owned, either by extending the Microsoft Foundry provider with correct Anthropic routing or by adding a dedicated provider plugin, while preserving generic API-key semantics and strict Foundry endpoint validation.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows the generic `--token` value is not passed through and malformed Foundry URL forms can be accepted. I did not run a live Azure AI Foundry onboarding flow in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this PR adds a new core auth choice with duplicate Azure-specific handling where the maintainable endpoint is likely the Microsoft Foundry provider or a dedicated provider plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor generic token input for Azure API-key auth:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: This branch is also reached when `--auth-choice apiKey --token-provider anthropic-azure` normalizes to `anthropic-azure-api-key`, but it passes only `anthropicAzureApiKey` into `ensureApiKeyFromOptionEnvOrPrompt`. Scripted setup that supplies the generic `--token` is ignored and falls back to env or prompt, unlike the existing API-key provider flow.
  - confidence: 0.9
- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:60-62`
  - body: For full URL input, any path that does not end with `/anthropic` gets `/anthropic` appended. A pasted endpoint like `https://fabric-hub.services.ai.azure.com/anthropic/v1` becomes `.../anthropic/v1/anthropic`, storing a malformed base URL and deferring failure to runtime instead of failing during onboarding.
  - confidence: 0.88
- **[P2] Validate the resource label in URL hosts:** `src/commands/anthropic-azure-utils.ts:53-56`
  - body: The URL branch only checks that the host ends with `.services.ai.azure.com`, so `https://fabric_hub.services.ai.azure.com/anthropic` can pass even though the bare resource path rejects underscores. Apply the same resource-label validation before accepting full URLs.
  - confidence: 0.86
