## Summary

Keep this PR open. Current main still lacks a first-class Azure Claude onboarding path, so the contribution is not obsolete, but the branch is not mergeable as-is: it adds provider-specific behavior in core while current main has plugin-owned provider auth/setup seams, and the diff still has concrete URL-validation and generic-token routing bugs called out in review discussion.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding choice with Azure Claude CLI flags, environment-variable handling, URL normalization, auth-profile/config wiring, docs, and focused onboarding tests.

## Best Possible Solution

Implement Azure Claude through a provider-owned setup path, preferably by extending `extensions/microsoft-foundry` if it owns Azure AI Foundry Anthropic deployments or by adding a separate provider plugin if the API/auth shape is distinct. That implementation should publish manifest auth/CLI metadata, validate `.services.ai.azure.com` hosts and `/anthropic` paths strictly, preserve generic `apiKey`/SecretRef flows, document the provider path, and cover interactive, non-interactive, env-var, URL, resource-name, and ref-mode cases.

## Reproduction Assessment

Yes. The PR diff gives static reproductions for malformed full URLs such as `https://fabric-hub.services.ai.azure.com/anthropic/v1`, invalid full-URL hosts such as `fabric_hub.services.ai.azure.com`, and `--auth-choice apiKey --token-provider anthropic-azure --token ...`; current-main search also verifies the requested first-class path is absent.

## Solution Assessment

No. The user problem is valid, but the proposed core-owned auth choice is not the narrowest maintainable solution now that provider auth choices, onboarding flags, and non-interactive setup can be owned by provider plugins.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-62`
  - body: `normalizeAnthropicAzureBaseUrl` appends `/anthropic` to any full URL path that does not already end in that suffix. Inputs like `https://fabric-hub.services.ai.azure.com/anthropic/v1` would be stored as `.../anthropic/v1/anthropic`, so onboarding succeeds with a malformed runtime base URL instead of failing fast.
  - confidence: 0.92
- **[P2] Validate Azure resource labels in URL input:** `src/commands/anthropic-azure-utils.ts:58`
  - body: The full-URL branch only checks that the hostname ends with `.services.ai.azure.com`. A value such as `https://fabric_hub.services.ai.azure.com/anthropic` bypasses the bare-resource validation and persists an invalid Azure resource host, pushing the failure to runtime.
  - confidence: 0.9
- **[P2] Honor generic token input for API-key routing:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: `--auth-choice apiKey --token-provider anthropic-azure --token ...` normalizes into this Azure branch, but the branch reads only `anthropicAzureApiKey`. Existing scripted API-key flows that use generic `--token` will be ignored and fall back to env or prompts.
  - confidence: 0.86
