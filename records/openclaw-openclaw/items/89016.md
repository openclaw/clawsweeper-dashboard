## What This Changes

The PR materializes managed OpenAI transport tool descriptors before strict-mode resolution and request payload conversion, skips unreadable required descriptors, degrades unreadable descriptions to an empty string, and adds regression coverage for Responses and Chat Completions builders.

## Best Possible Solution

Land the focused managed OpenAI transport guard after maintainer review, and either explicitly exclude or separately track the sibling provider helper if maintainers want the same descriptor-hardening invariant there.

## Reproduction Assessment

Yes from source inspection, though I did not execute the repro in this read-only review. Current main reads tool descriptor getters directly in both managed OpenAI request builders, so a throwing name, description, or parameters accessor can abort request construction.

## Solution Assessment

Yes for the managed transport path: materializing descriptors once before strict resolution and payload mapping is the narrowest maintainable fix there. The sibling legacy provider converter remains the only scope question, not a defect in the changed lines themselves.
