## Summary

Keep open: current `main` and the latest stable release still gate the bundled OpenAI TTS provider's voice-note compatibility to Opus only, while the Telegram send path and shared media contract already treat MP3 as voice-compatible. The linked fix PR was closed unmerged, so no canonical implementation currently resolves this issue.

## What This Changes

The issue asks for OpenAI-compatible TTS configured with `responseFormat: "mp3"` to route Telegram replies as voice notes instead of audio-file attachments.

## Best Possible Solution

Implement a narrow source fix in the OpenAI speech provider so explicit MP3 voice-note output is marked compatible under the existing Telegram/shared media contract, while preserving unsupported formats as non-compatible and adding regression coverage.

## Reproduction Assessment

Yes, by source inspection. Telegram requests `voice-note`, OpenAI with configured MP3 returns `voiceCompatible: false`, speech-core leaves `audioAsVoice` false, and Telegram then takes the `sendAudio` fallback instead of `sendVoice`.

## Solution Assessment

Yes. The narrow provider compatibility change proposed in https://github.com/openclaw/openclaw/pull/80326 is the best direction; broad channel transcoding or a new config option is unnecessary for MP3, which the existing Telegram/media contract already accepts.
