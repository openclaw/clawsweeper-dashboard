## Summary

Keep open: current main still preserves an explicit OpenAI TTS `responseFormat: "mp3"` but marks voice-note output as voice-compatible only for Opus, so Telegram does not get the `audioAsVoice` path for this reported configuration. The linked replacement PR was closed unmerged, and the broader Telegram voice-note issue was closed before this provider-specific mismatch was fixed.

## What This Changes

The issue requests that OpenAI-compatible TTS MP3 output be treated as Telegram voice-note compatible instead of falling back to audio-file delivery.

## Best Possible Solution

Align the OpenAI TTS provider with the existing shared MP3 voice-media contract by accepting MP3 for voice-note compatibility, keep the Opus default and WAV rejection, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level: configure OpenAI TTS with `responseFormat: "mp3"` on Telegram, and current main resolves MP3 but leaves `voiceCompatible` false, so speech-core does not set `audioAsVoice`. I did not run a live Telegram/Piper repro in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix is provider-local: mark MP3 as voice-compatible only for OpenAI voice-note output where the shared Telegram/media contract already accepts MP3, with tests that keep WAV false.
