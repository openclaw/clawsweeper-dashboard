## Summary

Keep open: current main and the latest release still keep OpenAI-compatible TTS MP3 output out of the voice-note path, and an open linked PR already references this issue with closing syntax. The right cleanup is to review and land https://github.com/openclaw/openclaw/pull/83227 or an equivalent fix, then close this issue through that merge.

## What This Changes

Treat OpenAI-compatible TTS `responseFormat: "mp3"` as voice-compatible for Telegram voice-note delivery instead of falling back to audio-file delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83227, or an equivalent shared-helper provider fix with focused regressions, then let the issue close through the PR's closing reference.

## Reproduction Assessment

Yes, by source inspection. OpenAI configured with MP3 for a Telegram `voice-note` target returns `voiceCompatible: false`, speech-core therefore leaves `audioAsVoice` false, and Telegram falls back to `sendAudio`.

## Solution Assessment

Yes. Reusing the shared media compatibility helper at the OpenAI provider boundary is the narrowest maintainable fix, and the linked PR is already structured around that approach.
