## Summary

Keep open: current main and v2026.5.12 still leave explicit OpenAI `responseFormat: "mp3"` non-voice-compatible for Telegram, while the shared Telegram voice-media path accepts MP3 and the only linked fix PR closed unmerged.

## What This Changes

The issue requests that OpenAI-compatible TTS output configured as MP3 route to Telegram `sendVoice` as a native voice note instead of falling back to `sendAudio`.

## Best Possible Solution

Patch the OpenAI provider to treat explicit MP3 voice-note output as voice-compatible under the existing Telegram/shared media contract, with focused regression tests and redacted real Telegram proof before landing.

Yes, by source inspection. Configure OpenAI-compatible TTS with `responseFormat: "mp3"` on Telegram: the provider returns `voiceCompatible: false`, speech-core leaves `audioAsVoice` unset, and Telegram falls back from `sendVoice` to `sendAudio`.

## Solution Assessment

Yes. The narrow maintainable fix is to broaden only OpenAI `mp3` voice-note compatibility while preserving the Opus default, Groq/WAV handling, and the existing false result for non-compatible formats.
