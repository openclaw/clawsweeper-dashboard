## Summary

Keep open: current main and the latest release still mark OpenAI-compatible TTS `responseFormat: "mp3"` as not voice-compatible for voice-note targets, while the shared media/Telegram path already treats MP3 as voice-compatible. The linked fix PR is still open, so this issue should close only after that PR lands or maintainers reject that path.

## What This Changes

The issue asks for OpenAI-compatible TTS configured with `responseFormat: "mp3"` to route as a Telegram voice note instead of falling back to an audio-file attachment.

## Best Possible Solution

Review and land or explicitly reject the linked PR; the end state should reuse the shared media voice-compatibility helper for OpenAI speech output and keep focused provider, speech-core, and Telegram routing coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows OpenAI MP3 voice-note synthesis returns `voiceCompatible: false`, while the shared media helper and Telegram routing already support MP3 voice delivery; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes. The linked PR's approach of asking the shared media runtime whether the selected output extension is voice-compatible is narrower and less duplicative than hard-coding another OpenAI-specific format list.
