## Summary

Keep open: current main and the latest release still mark OpenAI TTS voice-note output as voice-compatible only for Opus, so configured MP3 still falls through before Telegram can use its MP3-capable sendVoice path. The related fix PR was closed unmerged, and no other canonical open item appears to own the remaining provider-specific mismatch.

## What This Changes

The issue asks OpenAI-compatible TTS `responseFormat: "mp3"` to be treated as Telegram voice-note compatible so it routes through `sendVoice` like Edge/Microsoft TTS.

## Best Possible Solution

Align the OpenAI speech provider's MP3 voice-note compatibility with the existing Telegram/media contract and add a focused regression test for configured MP3 voice-note synthesis.

## Reproduction Assessment

Yes, by source inspection. A Telegram TTS request targets `voice-note`, explicit OpenAI `responseFormat: "mp3"` is preserved, the provider reports `voiceCompatible: false`, and speech-core therefore does not set `audioAsVoice` for Telegram delivery.

## Solution Assessment

Yes. The narrow maintainable fix is to broaden the OpenAI provider compatibility check for MP3 where Telegram already accepts MP3 voice media, with regression coverage rather than a new config option or broad transcoding project.
