## Summary

Keep open: current main and the latest stable tag still mark OpenAI TTS voice-note output as compatible only for Opus, so configured MP3 output still misses Telegram voice delivery despite the shared Telegram/media path already accepting MP3.

## What This Changes

The issue requests that OpenAI-compatible TTS configured with `responseFormat: "mp3"` be treated as Telegram voice-note compatible so it routes through `sendVoice` instead of `sendAudio`.

## Best Possible Solution

Align OpenAI provider MP3 voice-note compatibility with the existing Telegram/media contract, keep non-compatible formats such as WAV false, and add focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: Telegram requests a `voice-note`, OpenAI preserves configured `mp3`, the provider returns `voiceCompatible: false`, speech core leaves `audioAsVoice` unset, and Telegram falls back to `sendAudio`. I did not run a live Telegram/Piper repro in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to broaden the OpenAI provider's voice-note compatibility to include MP3, matching the shared Telegram/media compatibility contract while preserving the existing false result for WAV and other unsupported formats.
