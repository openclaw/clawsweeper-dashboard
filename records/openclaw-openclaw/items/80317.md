## Summary

Keep open: current main still has the reported OpenAI-provider gate, so an explicit OpenAI `responseFormat: "mp3"` can synthesize MP3 for Telegram while failing to mark the result as voice-compatible, even though Telegram routing and shared media policy already accept MP3 voice sends.

## What This Changes

The issue asks OpenAI-compatible TTS MP3 output to be treated as Telegram voice-note compatible instead of falling back to `sendAudio`.

## Best Possible Solution

Update the OpenAI speech provider to use the same voice-compatible audio policy for MP3 and Opus on voice-note targets, add regression coverage, and avoid adding a new config knob or broad transcoding change.

## Reproduction Assessment

Yes, by source inspection: configure Telegram TTS through the OpenAI provider with `responseFormat: "mp3"`; the provider returns `voiceCompatible: false`, speech-core therefore does not set `audioAsVoice`, and Telegram routing falls back to `sendAudio`. I did not perform a live Telegram send in this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable fix is to mark MP3 as voice-compatible in the OpenAI speech provider for voice-note targets, matching shared Telegram media policy and the OpenRouter/DeepInfra OpenAI-compatible provider behavior while keeping WAV non-compatible.
