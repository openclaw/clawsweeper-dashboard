## Summary

Keeping this open because current main still has the OpenAI TTS MP3 voice-note mismatch, and an open linked fix PR now carries the proposed implementation. The source path is clear: MP3 is an allowed OpenAI speech response format and Telegram/shared media accepts MP3 for voice delivery, but the OpenAI provider only marks Opus responses as `voiceCompatible`, causing the Telegram path to fall back away from `sendVoice`.

## What This Changes

The issue requests that OpenAI-compatible TTS MP3 output be treated as voice-compatible for Telegram voice notes, matching the existing Edge TTS behavior.

## Best Possible Solution

Review and land the linked OpenAI provider fix with focused regression coverage, or close that PR if maintainers choose a broader conversion-based design instead.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Telegram requests a `voice-note`, OpenAI MP3 synthesis returns `voiceCompatible: false`, speech core leaves `audioAsVoice` false, and Telegram then cannot take the `sendVoice` route.

## Solution Assessment

Yes. Treating OpenAI MP3 as voice-compatible for voice-note delivery is the narrowest source-level fix if tests cover the provider gate and Telegram delivery assumptions; a broader transcoding design can remain separate if maintainers want it later.
