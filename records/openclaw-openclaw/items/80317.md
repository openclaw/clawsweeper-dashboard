## Summary

Keep open: current main and the latest release still mark OpenAI TTS `responseFormat: "mp3"` as not voice-compatible for voice-note targets, while the Telegram/shared media path already treats MP3 as voice-compatible. The linked fix PR at https://github.com/openclaw/openclaw/pull/83227 is still open, so this issue should stay open until that PR lands or maintainers reject that fix path.

## What This Changes

The issue requests that OpenAI-compatible TTS MP3 output be routed as Telegram voice delivery instead of falling back to Telegram audio-file delivery.

## Best Possible Solution

Review and land the focused linked fix, or explicitly close that PR with a replacement decision; after a merge, close this issue as implemented with commit and release provenance.

## Reproduction Assessment

Yes, by source inspection: an OpenAI TTS voice-note request with configured `responseFormat: "mp3"` returns `voiceCompatible: false`, and speech-core only marks Telegram TTS audio as voice delivery when that provider metadata is true. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow maintainable fix: align the OpenAI provider's MP3 compatibility metadata with the shared media/Telegram voice contract and add provider plus speech-core regressions. A broader automatic conversion feature is not required for this reported MP3 case.
