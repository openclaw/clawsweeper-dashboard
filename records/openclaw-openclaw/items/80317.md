## Summary

Keep open: current main and the latest stable release still mark OpenAI-compatible MP3 voice-note output as not voice-compatible, so Telegram falls back to audio-file delivery even though the shared Telegram/media path accepts MP3 for voice messages. The related fix PR was closed unmerged, and no open canonical replacement was found.

## What This Changes

Make OpenAI-compatible TTS with `responseFormat: "mp3"` route to Telegram voice-note delivery like Edge/Microsoft TTS instead of falling back to audio-file delivery.

## Best Possible Solution

Broaden the OpenAI provider voice-note compatibility check to include MP3, keep non-compatible formats such as WAV false, and add focused provider plus delivery regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configured MP3 is preserved, OpenAI returns `voiceCompatible: false` for voice-note unless the format is Opus, speech core leaves `audioAsVoice` false, and Telegram then uses the audio-file path.

## Solution Assessment

Yes. The narrow maintainable fix is provider-local: treat MP3 as voice-compatible for OpenAI voice-note synthesis because Telegram and OpenClaw's shared media contract already accept MP3 voice media, then cover the behavior with regression tests.
