## Summary

Keep open: current main and the latest release still gate OpenAI TTS voice compatibility to Opus only, while Telegram sendVoice and OpenClaw's shared media policy already accept MP3. The linked fix PR was closed unmerged, so the source-backed provider/delivery mismatch remains.

## What This Changes

OpenAI-compatible TTS configured with `responseFormat: "mp3"` should route Telegram voice-note output through `sendVoice` instead of falling back to `sendAudio`.

## Best Possible Solution

Land a narrow source fix with regression coverage that lets Telegram-compatible OpenAI MP3 TTS reach native voice-note delivery while preserving existing Opus defaults and unsupported-format safeguards.

## Reproduction Assessment

Yes, by source inspection. Configure OpenAI TTS with `responseFormat: "mp3"` on Telegram: OpenAI returns `voiceCompatible: false`, speech-core leaves `audioAsVoice` false, and Telegram then takes the `sendAudio` fallback despite MP3 being accepted by its voice send path.

## Solution Assessment

Yes, the narrow maintainable fix is to align OpenAI MP3 voice-note compatibility with the existing Telegram/shared-media contract and add focused tests. It should be implemented in source, not by patching generated output or adding a new config knob.
