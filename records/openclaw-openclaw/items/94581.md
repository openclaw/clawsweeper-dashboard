## Summary

Keep open: current main and the latest release still use a shared memory-lancedb store, schema, and CLI surface without agent-scoped filtering; the overlapping security tracker does not fully cover this issue's CLI filtering and migration asks, and prior implementation PRs closed unmerged.

## What This Changes

The issue requests agent-scoped memory-lancedb/memsearch recall, indexing/search CLI flags, and migration so memories do not leak between configured agents.

## Best Possible Solution

Design and implement memory-lancedb agent scoping inside the plugin boundary, with a managed storage and migration story that covers auto-recall, auto-capture, memory tools, ltm CLI, docs, and live multi-agent proof together.

## Reproduction Assessment

Yes, at source level: current memory-lancedb constructs one shared MemoryDB and calls search/store/delete from tools and hooks without an agent scope. I did not run a live two-agent LanceDB reproduction in this read-only review.

## Solution Assessment

Unclear for the exact requested shape: agent-scoped memory is the right problem to solve, but the directory layout, CLI flags, legacy-memory visibility, and migration behavior need maintainer-approved semantics before implementation.
