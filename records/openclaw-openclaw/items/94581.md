## Label Justifications

- `P1`: The issue reports a real multi-agent memory isolation failure that can expose one agent's remembered data inside another active agent workflow.
- `impact:security`: Shared memory-lancedb recall can make sensitive memories from one agent trust boundary visible to another agent context.
- `impact:session-state`: The affected surface is persistent memory and prompt context being associated with the wrong agent/session scope.

## Summary

Keep open: current main and the latest release still use one shared memory-lancedb table and database instance with no agent-scoped storage, recall filter, or plugin CLI agent selector. The older implementation attempts are closed unmerged, and the remaining fix needs maintainer product/security decisions about storage, migration, and management semantics.

## What This Changes

Request agent-scoped memsearch/memory-lancedb isolation with per-agent collections, agent-filtered auto-recall, CLI `--agent` selectors, and a migration path for existing global memories.

## Best Possible Solution

Land a maintainer-approved memory-lancedb isolation design that scopes auto-recall, auto-capture, tools, CLI/docs, and migration or doctor behavior together while preserving legacy behavior only where explicitly chosen.

## Reproduction Assessment

Yes, at source level: current main uses one shared memory-lancedb DB/table and both auto-recall and auto-capture call that shared DB without an agent scope. I did not run a live two-agent LanceDB setup in this read-only review.

## Solution Assessment

Unclear for the exact requested shape: per-agent isolation is the right problem, but directory layout, CLI flags, shared-memory behavior, and migration/default semantics need maintainer approval before this is the best implementation path.
