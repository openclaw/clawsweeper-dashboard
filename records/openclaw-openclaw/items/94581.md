## Label Justifications

- `P1`: The issue reports a real multi-agent memory isolation failure that can inject one agent's memories into another active agent workflow.
- `impact:security`: The reported behavior can expose memories from one agent trust domain inside another agent's context.
- `impact:session-state`: The affected surface is cross-agent memory recall/capture contaminating session context and agent state.

## Summary

Keep open: current main and the latest release still use one shared memory-lancedb store/table with no agent-scoping config or CLI agent selector, and the prior implementation PRs for this problem are closed unmerged.

## What This Changes

The issue requests agent-scoped memory-lancedb recall, indexing, and search so memories from one configured agent are not auto-injected into another agent's session.

## Best Possible Solution

Design and implement memory-lancedb agent scoping inside the plugin boundary, preserving single-agent/legacy behavior intentionally, adding migration or doctor handling for existing memories, and covering auto-recall, auto-capture, tools, CLI/docs, and tests together.

## Reproduction Assessment

Yes, at source level: current main constructs one shared memory-lancedb store and the tools/hooks call db.search, db.store, and db.delete without an agent scope. I did not run a live two-agent LanceDB reproduction in this read-only review.

## Solution Assessment

Unclear for the exact requested shape: per-agent filtering is the right problem to solve, but the requested directory layout and CLI flags need maintainer-approved storage and migration semantics before implementation.
