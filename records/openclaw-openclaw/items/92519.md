## Summary

Keep open: current main and v2026.6.8 contain startup restart recovery, but the latest field report shows a remaining recovery-completion/delivery gap, and source inspection supports that the recovery RPC can return after the accepted ack rather than after the recovered run finishes.

## What This Changes

The issue reports that a mid-turn gateway restart can leave a Telegram-backed session stuck running or busy after restart recovery, requiring `/new` or `/reset`.

## Best Possible Solution

Keep this issue as the focused tracker for recovered main-session runs that do not reach terminal completion or final channel delivery after restart; the fix should bind recovery to run completion/delivery state rather than only to an accepted agent RPC.

## Reproduction Assessment

Yes at source level. Current recovery can submit a synthetic recovery turn and clear the restart marker after the accepted agent RPC, while the gateway runs the actual turn asynchronously and the latest v2026.6.8 field report shows that accepted recovered work can still fail to produce a final delivery.

## Solution Assessment

Unclear as an implementation patch, because the existing no-diff linked PR is not a usable fix. The best direction is to verify terminal completion and delivery binding for recovered runs, then add regression coverage for restart recovery that does not stop at accepted RPC submission.
