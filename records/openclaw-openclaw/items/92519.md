## Summary

Current main and the v2026.6.6 release already implement startup recovery for running main sessions left without an active run after a gateway restart, so the reported permanent busy state is covered well enough to close.

## What This Changes

The issue asks for automatic recovery when a mid-turn gateway restart leaves a session permanently busy until `/new` or `/reset`.

## Best Possible Solution

Keep the shipped startup recovery path; if this still reproduces on v2026.6.6 or current main, handle it as a fresh regression with live restart logs and session-store evidence.
