## Label Justifications

- `P1`: A gateway restart can leave an active channel session unusable until manual reset, breaking a real user workflow.
- `impact:session-state`: The report concerns stale run and session lifecycle state surviving the process that owned it.
- `impact:message-loss`: New Telegram messages are reportedly assembled but never dispatched while the session remains wedged.
- `impact:crash-loop`: The availability failure is triggered by gateway restart and leaves the session effectively hung after the process returns.

## Summary

Current main still treats an aborted recovery with `released=0` as an aborted outcome and retries dispatch without proving the stale owner was removed; the open linked PR addresses one active-lane shape but has no real restart proof and explicitly leaves stale target binding unresolved, so this restart-specific report remains necessary.

## What This Changes

Ensure a session interrupted by a gateway restart automatically becomes dispatchable again instead of remaining permanently busy until `/new` or `/reset`.

## Best Possible Solution

Extend the canonical gateway and recovery lifecycle so an interrupted process cannot leave stale run or lane ownership, then add a restart-to-next-dispatch regression scenario that verifies the same session processes a new message without reset.

## Reproduction Assessment

Yes at source level: current main has concrete restart-continuation, shutdown-drain, and stuck-recovery paths, and the recovery contract still permits `aborted` with `released=0`; this read-only review did not execute a live gateway restart.

## Solution Assessment

No, the proposed generic startup timeout is not the best fix; recovery should clear the exact stale run or lane owner in the existing lifecycle boundary and prove the next turn dispatches.
