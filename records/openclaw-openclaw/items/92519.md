## Best Possible Solution

The durable end state is marker-based restart recovery on current main: restart-interrupted active sessions are marked, startup-orphaned running rows are recovered or failed, and the linked PR is either closed as superseded by that implementation or revised into the canonical recovery path.

## Reproduction Assessment

Yes at source level, but not as a current-main failing live repro. The reported bad state maps to persisted `running` main-session rows with no live process owner after restart, and current tests now exercise the startup recovery shape for those rows.

## Solution Assessment

Yes for the current-main direction, but no for the open PR's marker-clearing approach. The maintainable fix is to mark and recover or fail the stale session owner, not to clear busy/recovery state without proving the lane is dispatchable.
