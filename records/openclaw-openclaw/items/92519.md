## Label Justifications

- `P1`: The reported restart bug can leave a real user's channel session unable to process messages until manual reset.
- `impact:crash-loop`: The failure is restart-related and produces an indefinite availability outage for the affected session after the gateway comes back.
- `impact:message-loss`: Messages sent after the restart are not processed while the session remains wedged busy.
- `impact:session-state`: The root behavior is stale run/session recovery state leaving the main session marked running without an active owner.

## Summary

Current main now resolves the central permanent-busy-after-restart failure: the merged restart recovery work persists active main-session recovery state before restart abort cleanup and startup recovery marks/resumes or fails orphaned running main sessions instead of leaving them indefinitely busy.

## What This Changes

The issue asks the gateway to auto-recover sessions left busy after a mid-turn restart without requiring `/new` or `/reset`.

## Best Possible Solution

Keep the merged restart-recovery marker/startup recovery path, ship it in the next release, and handle the still-open linked cleanup PR on its own review.

## Solution Assessment

Yes. The merged owner-boundary fix is better than clearing a busy flag because it preserves recovery state before abort cleanup and lets startup recovery resume or explicitly fail the interrupted main session.
