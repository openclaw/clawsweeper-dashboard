## Summary

This is a valid restart-triggered session-lifecycle bug, and current main still permits the reported `aborted`/`released=0` recovery shape. The open fix for the broader recovery defect may help, but it does not yet prove that a post-restart session becomes dispatchable, so this issue remains useful as focused regression coverage.

## What This Changes

Investigate and fix sessions that remain permanently busy after a gateway restart interrupts an active turn.

## Best Possible Solution

Identify the exact pre-restart owner, release only that stale run or lane generation, and verify that a queued or subsequent turn completes after an actual gateway restart.

## Reproduction Assessment

Yes at source level: current main can classify recovery as aborted with `released=0` without verifying that the session lane is dispatchable. A real restart reproduction is still needed to identify which owner remains stale.

## Solution Assessment

No—the proposed blanket startup busy-flag clear is unsafe because process-local run state should already disappear on restart; the best fix must identify and release the canonical stale lane or run owner and verify recovery.
