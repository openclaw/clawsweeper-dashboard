## Summary

Keep this PR open. The branch now preserves native Android Talk as the default and makes realtime relay opt-in, but it still has lifecycle defects around disabling/failure handling and the contributor proof does not show the changed runtime path on a real device.

## What This Changes

Adds an opt-in Android Talk realtime Gateway relay mode that streams mic PCM through `talk.session.*`, plays relay audio locally, bridges realtime tool calls, and documents the config.

## Best Possible Solution

Keep native Android Talk as the default, land realtime relay as an explicit opt-in only after lifecycle guards, focused tests, docs, and redacted real-device relay proof are in place.

## Reproduction Assessment

Yes, source-reproducible: the PR branch shows the async native start race and relay failure state issue directly in `TalkModeManager.kt`. I did not run a live Android device in this read-only review.

## Solution Assessment

No. The opt-in realtime relay direction is reasonable, but the implementation needs lifecycle fixes and stronger real-device proof before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Abort native start after Talk is disabled:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:494-496`
  - body: `start()` now waits for `talk.config` before entering the native path. If the user toggles Talk off while that request is pending, `stop()` can finish first and these lines still create a `SpeechRecognizer` because they only check `_isListening`; re-check `_isEnabled`/`stopRequested` before starting native recognition so the microphone cannot start after Talk is off.
  - confidence: 0.88
- **[P2] Keep relay failures from looking active:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:660`
  - body: On `appendAudio` or capture setup failure, the code sets a failure status and immediately calls `stopRealtimeRelay()`. That cleanup calls `stopRealtimePlayback()`, which changes the status back to `Listening` while `_isEnabled` remains true and no relay/capture job is active, so Talk appears live but is idle. Preserve a failed/off state or schedule a restart after cleanup.
  - confidence: 0.84
