## Summary

Keep open: this PR is a plausible Android Talk migration, but the current patch unconditionally replaces the documented native Talk path, has relay lifecycle/UI state bugs, and does not yet include after-fix proof that shows the realtime relay path running on a device.

## What This Changes

The PR changes Android Talk Mode from the native SpeechRecognizer/chat/talk.speak pipeline to Gateway relay realtime sessions with mic streaming, realtime playback, tool-result bridging, and a changelog entry.

## Best Possible Solution

Keep the documented native Android Talk path as the default unless maintainers explicitly choose a realtime default; add a capability-gated realtime path with legacy fallback, lifecycle tests, docs, and redacted real-device relay proof.

## Reproduction Assessment

Yes, source-reproducible: Android Talk without a configured realtime voice provider will now call talk.session.create for realtime/gateway-relay, and Gateway resolves that through the realtime provider registry. I did not run a live Android device in this read-only review.

## Solution Assessment

No. Unconditionally replacing the documented native Android Talk pipeline is not the narrowest maintainable fix; the safer direction is a gated realtime path with fallback and explicit docs/product approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve native Talk fallback:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:550-552`
  - body: This unconditionally switches Android Talk to the realtime relay session path. Current docs define native Android Talk as local speech recognition, Gateway chat, and talk.speak TTS, and Gateway will reject this new path when no realtime voice provider is registered/configured, so existing Android Talk setups can stop working after upgrade. Gate this on realtime capability/configuration or keep the legacy native path as a fallback.
  - confidence: 0.91
- **[P2] Close the relay if Talk stops while connecting:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:562-566`
  - body: If the user turns Talk Mode off while talk.session.create is still pending, stopRealtimeRelay() sees no session to close. When the request later returns, these lines still mark the disabled manager as listening and start capture, leaving a relay session open. Re-check _isEnabled/stopRequested after the create response and close the new session instead of activating it.
  - confidence: 0.86
- **[P2] Clear speaking state after realtime output ends:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:683-684`
  - body: Realtime audio sets _isSpeaking to true, but the final assistant transcript path only changes the status text to Listening. TalkOrbOverlay chooses the Speaking phase whenever isSpeaking is true, so the UI can remain stuck in Speaking after the response. Clear the realtime speaking state when queued output has actually finished.
  - confidence: 0.78
