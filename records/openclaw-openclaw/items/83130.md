## Summary

Keep this PR open, but do not merge it as-is. The realtime relay direction matches existing Gateway Talk contracts, but source review found relay lifecycle and playback cancellation bugs that can leave Android Talk Mode stuck or speaking after stop, and the external real-device proof is not yet strong enough for the contributor proof gate.

## What This Changes

The branch replaces Android Talk Mode's legacy SpeechRecognizer/chat/TTS loop with realtime Gateway relay session creation, mic PCM streaming, realtime audio playback, and tool-result bridging, plus a changelog entry.

## Best Possible Solution

Land the Android realtime relay migration only after relay session cleanup, event scoping, realtime playback cancellation, focused regression coverage, docs alignment, and redacted real-device runtime proof are in place.

## Reproduction Assessment

Yes, by source inspection for the stated gap: current main starts Android Talk Mode with SpeechRecognizer and sends finalized transcripts through `chat.send` before TTS playback. I did not run a live Android device session in this read-only review.

## Solution Assessment

No, not yet. Using the documented `talk.session`/`talk.client` relay contracts is the right direction, but this implementation needs lifecycle and cancellation fixes before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Tear down the relay when appendAudio fails:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:609-614`
  - body: When `talk.session.appendAudio` throws, this branch only logs, updates the status, and breaks out of the capture loop. It leaves `realtimeSessionId` set and never closes the Gateway relay session, so the next `start()` returns early and a transient append/network failure can leave Talk Mode stuck until a full toggle cleanup happens.
  - confidence: 0.91
- **[P2] Handle capture setup failures inside the job:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:579-580`
  - body: This setup runs inside the launched capture job after `startRealtimeRelay()` has already stored `realtimeSessionId`. If 24 kHz capture is unavailable or `AudioRecord` setup/start throws, the outer start try/catch will not run, leaving Android marked as listening with an unclosed relay session; catch setup failures in the job and clear/close the relay before returning.
  - confidence: 0.87
- **[P2] Ignore relay events when no session is active:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:635-637`
  - body: After `stopRealtimeRelay()` clears `realtimeSessionId`, this guard no longer rejects delayed `talk.event` frames. A late audio frame from the closed relay can therefore create or write an `AudioTrack` while Talk Mode is already off; require an active session id and a matching event session before processing relay events.
  - confidence: 0.88
- **[P2] Stop realtime playback through the normal stop path:** `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt:702-703`
  - body: Realtime audio is written through its own `AudioTrack`, but the existing stop path only advances the TTS playback generation and cancels TTS playback. Pressing stop or disabling playback can leave the realtime track alive or allow later chunks to be written; make realtime playback participate in `stopSpeaking()`/`stopTts()` cancellation or gate writes with the same token.
  - confidence: 0.86
