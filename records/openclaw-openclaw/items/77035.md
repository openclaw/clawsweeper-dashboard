## Summary

Keep open: the operator diagnostic is useful, but the patch currently regresses the Gateway pricing-cache lazy-load boundary by importing the heavy refresh module from server methods. That should be fixed before merge.

## What This Changes

The PR adds a read-scoped `diagnostics.pricing` Gateway RPC, advertises and documents it, exposes the pricing cache TTL constant, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the diagnostic, but source cache metadata from a lightweight cache-state/config module with the real TTL and add an import-boundary regression test so server-method diagnostics do not load `model-pricing-cache.js`.

## Reproduction Assessment

Yes for the review finding: the supplied diff statically imports `model-pricing-cache.js` from `diagnostics.ts`, while current main deliberately dynamic-imports that module only after pricing is enabled and scheduled services activate. No live Gateway run is needed to establish this source-level regression.

## Solution Assessment

No, not as written. Exposing aggregate pricing cache metadata is a reasonable read-scoped diagnostic, but the narrow maintainable implementation should keep the refresh/fetch module lazy by moving the TTL constant or metadata into the lightweight state side.
