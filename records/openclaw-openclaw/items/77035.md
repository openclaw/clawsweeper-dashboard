## Summary

Keep open: the new diagnostic RPC is plausible and current main does not already expose it, but the patch has a blocking lazy-loading regression in the way it sources the pricing TTL.

## What This Changes

The PR adds a `diagnostics.pricing` Gateway RPC, advertises/classifies it under operator read scope, documents it, updates the changelog, and adds diagnostics tests for empty/populated pricing cache metadata.

## Best Possible Solution

Keep the additive `diagnostics.pricing` RPC, but source TTL metadata from a lightweight cache-state/constants module so the model-pricing refresh runtime stays lazy.

## Reproduction Assessment

Yes for the review finding: source inspection shows `model-pricing-cache.js` is currently dynamically imported by scheduled services, while the PR would statically import it from the eagerly loaded diagnostics handler. For the requested feature, current main source shows the RPC is absent.

## Solution Assessment

No. The RPC direction is reasonable, but importing the TTL from the heavy pricing runtime is not the narrowest maintainable solution; move that constant to a lightweight leaf boundary and keep the existing docs/scope/list updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid importing the pricing runtime from diagnostics:** `src/gateway/server-methods/diagnostics.ts:6`
  - body: The new handler imports `GATEWAY_MODEL_PRICING_CACHE_TTL_MS` from `../model-pricing-cache.js`. `server-methods.ts` loads diagnostics handlers eagerly, while `server-runtime-services.ts` deliberately lazy-imports `./model-pricing-cache.js` and has tests proving pricing bootstrap is not imported when disabled or inert. Put the TTL in `model-pricing-cache-state.ts` or a small leaf constants module so this diagnostic does not make the pricing runtime eager.
