## Summary

Keep this PR open for maintainer review: current main still lacks `diagnostics.pricing`, but the proposed patch needs rework because it would make the diagnostics handler statically import the heavy pricing refresh module that current main keeps on a lazy startup path.

## What This Changes

This PR adds a read-scoped `diagnostics.pricing` Gateway RPC, advertises and documents it, exposes pricing cache TTL metadata, and adds diagnostics tests plus a changelog entry.

## Best Possible Solution

Expose the diagnostic through the existing Gateway method/list/scope/docs pattern, but move TTL/cache metadata into a lightweight shared module used by both diagnostics and the refresh scheduler so startup lazy-loading stays intact.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection does show the feature gap on current main: only `diagnostics.stability` is registered and advertised, while `diagnostics.pricing` is absent.

## Solution Assessment

No. The RPC shape is reasonable, but importing the refresh module just to read the TTL is not the narrowest maintainable implementation; a lightweight shared constant or state helper would preserve the lazy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep pricing cache on its lazy import path:** `src/gateway/server-methods/diagnostics.ts:6`
  - body: `diagnostics.ts` is loaded with `coreGatewayHandlers`, so importing `../model-pricing-cache.js` here makes the heavy pricing refresh module part of the normal Gateway handler load path. Current main deliberately starts pricing refresh with `await import("./model-pricing-cache.js")`, and the gateway guide calls out keeping startup/manual RPC paths from materializing plugin runtime. Move the TTL constant into a lightweight state/constants module used by both diagnostics and the scheduler.
  - confidence: 0.87
