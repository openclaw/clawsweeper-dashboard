## Summary

Keep this PR open: the new diagnostic is useful, but the current diff crosses the gateway pricing cache lazy-loading boundary and should be fixed before merge.

## What This Changes

The PR adds `diagnostics.pricing` to gateway handlers, read-scope/discovery lists, protocol docs, changelog, and diagnostics tests, and exposes the pricing cache TTL for the response.

## Best Possible Solution

Keep the new operator-read diagnostic, but move the TTL source to a lightweight state/constants module so `server-methods` does not statically import pricing refresh code, then rerun the targeted diagnostics test and build gate.

## Reproduction Assessment

Yes. By source inspection, current main lazy-loads `model-pricing-cache.js` from `server-runtime-services.ts`, while the PR statically imports that same module from `diagnostics.ts`.

## Solution Assessment

No. The diagnostic payload and read-scope placement are reasonable, but sourcing the TTL from the refresh module is not the narrowest maintainable solution because it defeats the existing lazy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep pricing cache behind the lazy boundary:** `src/gateway/server-methods/diagnostics.ts:6`
  - body: `diagnostics.ts` is loaded through `coreGatewayHandlers` on every gateway startup, so importing the TTL from `../model-pricing-cache.js` makes the pricing refresh module a static dependency. Current main intentionally loads that module with `await import("./model-pricing-cache.js")` only after pricing is enabled; this change pulls in the model catalog/plugin metadata pricing code even for disabled/on-demand paths and can make the dynamic import ineffective. Move the TTL constant to `model-pricing-cache-state.ts` or a small constants module and import only that lightweight surface here.
  - confidence: 0.87
