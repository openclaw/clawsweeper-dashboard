## Summary

Keep open. Current main still has only root-scoped plugin config, strict agent entries still reject the requested plugin override shape, and related threads either cover narrower plugin-specific isolation or were closed without replacing this generic config/API design request.

## What This Changes

The issue requests per-agent plugin configuration overrides so multi-agent deployments can give each agent different plugin settings such as memory project credentials, plugin allowlists, or user-specific runtime config.

## Best Possible Solution

Define a documented, backward-compatible agent-scoped plugin effective-config contract, including precedence, validation, SecretRef-safe merge semantics, lifecycle/reload behavior, and SDK/runtime exposure while preserving root defaults.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence in the current behavior: agent entries are strict without plugins, and runtime plugin config resolution reads only the top-level plugins.entries map.

## Solution Assessment

Unclear. The proposed deep-merge shape is plausible, but maintainers need to choose whether the durable solution is generic agent-level config, plugin-owned per-agent maps, agent plugin allow/deny scoping, or a combination with explicit precedence and secret handling.
