## Summary

Keep open: current main and v2026.6.10 still have no agent-level plugin config surface, and plugin config is still resolved only from global `plugins.entries.<id>.config`; the remaining work is a security-sensitive product/API design decision, not conservative cleanup.

## What This Changes

Request per-agent plugin configuration overrides so multi-agent installs can give different agents different plugin credentials, memory stores, and plugin-owned settings.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin config contract with precedence, SecretRef/redaction rules, schema validation, lifecycle semantics, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 both reject agent-level plugin config by schema shape and resolve plugin config only from global `plugins.entries[id].config`.

## Solution Assessment

Unclear. The requested deep-merge model is plausible, but the best fix needs maintainer decisions on precedence, SecretRef behavior, plugin lifecycle, validation, and whether generic per-agent config or plugin-specific scopes should own each use case.
