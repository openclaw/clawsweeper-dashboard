## Summary

Keep open: current main and v2026.6.8 still keep plugin configuration under global `plugins.entries.<id>.config`, while the issue and follow-up comments describe a real multi-agent credential, memory, and plugin-state isolation gap. This should remain a maintainer product/security design item, not conservative cleanup.

## What This Changes

Request per-agent plugin configuration overrides so agents in one gateway can use different plugin settings, API keys, memory stores, or account/project routing.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin config contract with precedence, SecretRef/redaction rules, schema validation, lifecycle semantics, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's strict agent schema omits `plugins`, plugin config resolution reads only global `plugins.entries[id].config`, and v2026.6.8 has the same shape.

## Solution Assessment

Unclear. The requested deep-merge model is plausible, but the best fix needs maintainer decisions on precedence, SecretRef behavior, plugin lifecycle, validation, and whether generic per-agent config or plugin-specific scopes should own each use case.
