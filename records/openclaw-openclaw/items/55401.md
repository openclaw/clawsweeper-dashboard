## Summary

Keep open. Current main still has only root-scoped plugin configuration, strict agent entries do not accept a plugin override map, and the plugin config runtime has no agent dimension. The related per-agent scoping discussion is useful context, but it covers allow/deny lifecycle gating rather than per-agent config payload overrides.

## What This Changes

Request support for agent-scoped plugin configuration overrides that deep-merge over global `plugins.entries.<id>.config` in multi-agent deployments.

## Best Possible Solution

Define a first-class agent-scoped plugin override contract in core, validate it against plugin config schemas, merge it deterministically with global plugin config at request time, expose the effective config through documented SDK/runtime seams, and update docs plus focused config/plugin-runtime tests.

## Reproduction Assessment

Not applicable as a feature request, but the config-contract check is high confidence. Current main would reject the proposed `agents.list[].plugins` key through the strict agent schema and still resolves plugin config only from `plugins.entries.<id>.config`.

## Solution Assessment

Unclear. The proposed deep-merge direction is plausible, but the durable fix needs maintainer design because it affects public config, plugin SDK behavior, and per-agent secret-bearing credentials.
