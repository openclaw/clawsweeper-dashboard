## Label Justifications

- `P2`: This is a concrete multi-agent configuration gap with production examples, but it requires feature/API design rather than an urgent regression fix.
- `impact:session-state`: The request is about preventing plugin-owned memory, tool context, and agent state from being shared or mis-associated across agents.
- `impact:security`: Per-agent plugin scoping affects credential exposure and plugin/tool authorization boundaries in shared-gateway deployments.
- `impact:auth-provider`: The issue explicitly asks for per-agent plugin credentials, API keys, backend account routing, and billing separation.

## Summary

Keep open: current main and v2026.5.22 still expose plugin configuration only at the top-level plugin entry layer, and the remaining request is a broad security-sensitive agent-scoped plugin API that is not covered by the narrower memory-wiki or web-search trackers.

## What This Changes

Request adding per-agent plugin configuration overrides under agent definitions so multi-agent installs can isolate plugin credentials, storage, capabilities, and billing while inheriting global plugin defaults.

## Best Possible Solution

Define a backward-compatible agent-scoped effective plugin config contract covering schema, validation, SecretRef-safe merging, SDK/runtime exposure, hook/tool filtering, reload behavior, and docs while preserving root plugin defaults.

## Reproduction Assessment

Not applicable as a broken-behavior bug, but yes as a source-level feature gap. Current main and v2026.5.22 have strict agent entries with no plugins field, and plugin config is resolved from top-level plugins.entries only.

## Solution Assessment

Unclear. The proposed deep-merge shape is plausible, but maintainers need to choose the durable public contract across schema, validation, SDK/runtime behavior, lifecycle, hook/tool filtering, and SecretRef handling.
