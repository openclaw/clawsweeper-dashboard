## Summary

Keep open: current main and latest release still keep plugin configuration under global `plugins.entries.<id>.config`, while this issue describes a real multi-agent credential, memory, and plugin-state isolation gap that needs maintainer product/security design rather than cleanup.

## What This Changes

Add a per-agent plugin configuration override surface so different agents in one gateway can use different plugin settings and credentials.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin config contract with precedence, SecretRef/redaction rules, schema validation, lifecycle semantics, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's strict agent schema omits `plugins`, plugin config resolution reads only global `plugins.entries[id].config`, and `v2026.6.10` has the same shape.

## Solution Assessment

Unclear. The requested deep-merge model is plausible, but the best fix needs maintainer decisions on precedence, SecretRef behavior, plugin lifecycle, validation, and whether generic per-agent config or plugin-specific scopes should own each use case.
