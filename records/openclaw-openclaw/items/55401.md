## Summary

Keep open: current main and v2026.6.8 still expose plugin configuration as a global `plugins.entries.<id>.config` surface, while the issue describes a real multi-agent credential and memory/plugin-state isolation gap. This is not conservative cleanup because the remaining work needs maintainer product and security design, not stale closure.

## What This Changes

Request to add per-agent plugin configuration overrides so multi-agent setups can give individual agents different plugin settings such as Mem0 API keys and user IDs.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin config contract with precedence, SecretRef handling, schema validation, lifecycle semantics, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's strict AgentEntrySchema omits plugins and the runtime config helper reads only global plugins.entries[id].config, so the reported agents.list[].plugins shape remains unsupported.

## Solution Assessment

Unclear. The proposed deep merge is plausible, but the best fix needs maintainer decisions on precedence, plugin lifecycle, SecretRef handling, schema validation, and when plugin-owned per-agent config is enough.
