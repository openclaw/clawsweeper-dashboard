## Summary

Keep open. Current main still has only top-level plugin config, strict agent entries still reject an agent-level `plugins` field, and the related items either cover narrower memory-wiki/plugin-scoping work or were closed without replacing this generic config/API design request.

## What This Changes

Request per-agent plugin configuration overrides so multi-agent setups can merge agent-specific plugin config with global plugin config.

## Best Possible Solution

Define a documented agent-scoped effective-plugin-config contract, including precedence, validation, SecretRef-safe merge semantics, lifecycle/reload behavior, and SDK/runtime exposure while preserving root defaults.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence in the current behavior: agent entries are strict without `plugins`, and runtime plugin config resolution reads only the top-level `plugins.entries` map.

## Solution Assessment

Unclear. The proposed deep-merge shape is plausible, but maintainers need to choose between a generic core override contract, plugin-owned per-agent maps, plugin allow/deny scoping, or a combination with explicit secret and lifecycle semantics.
