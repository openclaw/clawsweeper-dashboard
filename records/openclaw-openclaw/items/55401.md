## Summary

Keep open: current main and v2026.6.8 still only expose global plugin config, while the issue and follow-up comments describe a real multi-agent credential, memory, and plugin-state isolation gap. This needs maintainer product and security design before implementation, so it is not safe cleanup closure.

## What This Changes

Request support for per-agent plugin configuration overrides so different agents in one gateway can use different plugin credentials, state, or settings.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin config and lifecycle contract with clear precedence, SecretRef/security handling, schema validation, docs, and focused two-agent regression coverage; keep plugin-owned per-agent maps or templates as the preferred path when they solve a specific plugin safely.

## Reproduction Assessment

Yes, at source level. Current main's strict AgentEntrySchema and AgentConfig omit plugins, and plugin config is resolved from global plugins.entries[id].config, so the reported agents.list[].plugins shape remains unsupported.

## Solution Assessment

Unclear. The proposed deep merge is plausible, but the best fix needs maintainer decisions on precedence, plugin lifecycle, SecretRef handling, schema validation, and when plugin-owned per-agent config is enough.
