## Summary

Keep open: current main still exposes plugin configuration only at the root plugin entry layer, and the discussion includes production multi-agent cases that are broader than the narrower memory-wiki tracker.

## What This Changes

The issue requests per-agent plugin configuration overrides so multi-agent installs can use different plugin settings, credentials, account routing, and scoping per agent.

## Best Possible Solution

Define a backward-compatible agent-scoped plugin effective-config contract covering precedence, validation, SecretRef-safe merging, lifecycle and reload behavior, SDK/runtime exposure, hook/tool filtering, and docs while preserving root plugin defaults.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/API request. Source inspection gives high confidence in the current limitation: strict agent entries do not accept plugins, and plugin config resolution reads only the top-level plugins.entries map.

## Solution Assessment

Unclear. The proposed deep-merge shape is plausible, but maintainers need to choose the durable contract across schema, validation, SDK/runtime behavior, lifecycle, hook/tool filtering, and SecretRef handling.
