## Summary

Keep open. Current main and the latest release still expose plugin configuration only at the root `plugins.entries.<id>.config` level, while strict agent entries do not accept an `agents.list[].plugins` override; related open items cover narrower memory-wiki or per-feature cases rather than replacing this generic contract request.

## What This Changes

The issue requests per-agent plugin configuration overrides so multi-agent deployments can give different agents different plugin credentials, memory stores, and plugin behavior.

## Best Possible Solution

Define a documented, backward-compatible agent-scoped plugin effective-config contract with precedence, validation, SecretRef-safe merge semantics, lifecycle/reload behavior, and SDK/runtime exposure while preserving root defaults.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/API request. Source inspection gives high confidence in the current limitation: agent entries are strict without `plugins`, and plugin config resolution reads only the top-level `plugins.entries` map.

## Solution Assessment

Unclear. The proposed deep-merge shape is plausible, but maintainers need to choose the durable contract across config schema, SDK/runtime behavior, plugin lifecycle scoping, and secret handling.
