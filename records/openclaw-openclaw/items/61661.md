- `P2`: This is a focused Control UI session restore bug with limited blast radius and an active two-file fix path.
- `impact:session-state`: The issue is about stale persisted session keys causing the UI to restore the wrong session after reload.

## Summary

Keep open. Current main still restores scoped Control UI session keys from local settings without filtering historical routed direct/group session keys, and the linked fix candidate remains open rather than merged.

## What This Changes

Control UI should fall back to the main session on startup when persisted local state contains historical routed direct/group session keys.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/78741 with a storage-layer sanitizer that resets persisted routed direct/group session keys to main while preserving explicit session selection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed `openclaw.control.settings.v1` / `sessionsByGateway` with a routed key such as `agent:main:feishu:direct:ou_xxx`, call `loadSettings()`, and the value flows into startup settings before gateway defaults preserve it as user-selected.

## Solution Assessment

Yes. The storage restore boundary is the right place to sanitize historical routed keys; this issue should remain open until the linked PR lands or a replacement covers the routed key contract and explicit-selection behavior.
