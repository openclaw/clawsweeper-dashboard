## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Run diagnostics before provider projection:** `src/agents/tool-schema-projection.ts:135`
  - body: `projectRuntimeToolInputSchema` later searches the returned schema for dynamic-schema violations, but this change stores only the provider-normalized tree. An `anyOf` branch containing `$dynamicRef` can collapse before diagnostics, so doctor and Codex dynamic-tool quarantine no longer see the unsupported raw branch. Keep the raw parsed schema for diagnostics, then return the provider-projected schema after the checks.
  - confidence: 0.9
- **[P2] Preserve non-null union alternatives:** `src/agents/tool-schema-projection.ts:188-193`
  - body: `normalizeSchemaComposition` falls back to the first string variant whenever a composition contains one, so a runtime-supported shape like `{string | number | null}` becomes only `{type: "string"}`. That misrepresents number inputs to providers and plugin SDK consumers; drop null-only branches without discarding other non-null alternatives.
  - confidence: 0.86
- **[P2] Keep non-null union coercion ordered:** `packages/llm-core/src/validation.ts:207-212`
  - body: The new exact-match pre-pass returns the original value for any matching union branch before trying ordered coercion. For a schema like `anyOf: [{type: "number"}, {type: "string"}]`, a model-emitted numeric string now stays a string instead of following the existing first-branch numeric coercion. Limit the shortcut to explicit null preservation.
  - confidence: 0.87
