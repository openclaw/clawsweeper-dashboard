Keep the provider-facing projection idea, but land only a branch that checks raw schema diagnostics first, preserves non-null union semantics, keeps null-specific validation behavior narrow, and has current-head CI/mergeability proof.

No. Provider projection is the right layer, but this implementation normalizes too early and too broadly; the safer shape is raw diagnostics first plus narrower provider projection.

Full review comments:

- **[P2] Run diagnostics before provider projection:** `src/agents/tool-schema-projection.ts:135`
  - body: This now stores only `normalizeProviderToolInputSchema(parsed)` in the projection, and `projectRuntimeToolInputSchema` later searches that normalized tree for dynamic-schema violations. If an `anyOf` branch containing `$dynamicRef` is collapsed to a plain string branch, doctor and Codex dynamic-tool quarantine no longer see the unsupported raw branch. Keep the raw parsed schema for diagnostics, then return a provider-projected schema after the checks.
  - confidence: 0.9
- **[P2] Preserve non-null union alternatives:** `src/agents/tool-schema-projection.ts:188-193`
  - body: `normalizeSchemaComposition` falls back to the first string variant whenever a composition contains one, so the cron `threadId` shape `{string | number | null}` becomes only `{type: "string"}`. That misrepresents a runtime-supported number input to providers and plugin SDK consumers; drop null-only wrappers without discarding other non-null branches.
  - confidence: 0.86
- **[P2] Keep non-null union coercion ordered:** `packages/llm-core/src/validation.ts:207-212`
  - body: The new exact-match pre-pass returns the original value for any matching union branch before trying ordered coercion. For a plain JSON schema like `anyOf: [{type: "number"}, {type: "string"}]`, a model-emitted string number now stays a string instead of being coerced through the first numeric branch, which regresses the existing validation behavior. Limit the shortcut to explicit null preservation.
  - confidence: 0.87
