## Summary

Keep open: current main still has the reported false-default veto, the parser still emits `replyToCurrent: false` for ordinary text, and the related fix PRs are closed unmerged. The bug remains narrow and actionable, with one important repair constraint around command replies that intentionally clear threading.

## What This Changes

The issue asks for implicit reply threading to keep assigning `replyToId` when a queued or normalized reply payload carries the parser-default `replyToCurrent: false` value.

## Best Possible Solution

Update the shared reply-threading contract so parser-default `replyToCurrent: false` does not suppress implicit current-message threading, while moving intentional non-threading behavior onto an explicit policy or metadata path. Add focused tests for default-false payloads, streaming/block delivery behavior, and command replies that must remain non-threaded.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a payload like `{ text: "hello", replyToCurrent: false }` passed through `applyReplyThreading` with `replyToMode: "all"` and `currentMessageId: "42"`; line 41 takes the false branch before assigning `replyToId`.

## Solution Assessment

Yes, a narrow reply-threading fix is the right direction, but the safest implementation is not a blind removal of every false check. It should separate parser-default false from an explicit command/policy opt-out and verify the followup queue's current-message target path.
