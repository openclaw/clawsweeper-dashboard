## Summary

Keep open: current `main` still treats parser-default `replyToCurrent: false` as a veto before implicit `replyToId` assignment, and followup delivery still does not pass the queued message id into the threading helper. The open related PR covers only the missing-id handoff, while the false-default fix attempts are closed unmerged.

## What This Changes

The issue reports that default `replyToCurrent: false` prevents implicit queued/followup replies from receiving `replyToId`.

## Best Possible Solution

Separate parser-default false from intentional non-threading, forward the queued current message id through followup delivery, and add focused regression coverage while preserving the command reply opt-out contract.

## Reproduction Assessment

Yes, at source level. On current `main`, `applyReplyThreading` can receive `{ text: "hello", replyToCurrent: false }`, `replyToMode: "all"`, and `currentMessageId: "42"`, but the helper short-circuits before assigning `replyToId`; the followup drain also omits the queued message id from the helper call.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared reply-threading and followup handoff path, with regression tests that distinguish parser-default false from intentional non-threading rather than adding channel-specific workarounds.
