## Summary

Keep open: current `main` still treats `replyToCurrent: false` as a veto before implicit `replyToId` assignment, and the followup drain still omits the current message id. The linked false-default fix PRs are closed unmerged, while the open followup PR covers only the missing-id handoff.

## What This Changes

The issue requests that implicit reply threading still assign `replyToId` when payloads carry parser-default `replyToCurrent: false`, especially in queued/followup replies.

## Best Possible Solution

Separate parser-default false from intentional non-threading, forward the queued current message id through followup delivery, and add focused regression coverage while preserving the command reply opt-out contract.

## Reproduction Assessment

Yes, at source level. On current `main`, `applyReplyThreading` can be called with `{ text: "hello", replyToCurrent: false }`, `replyToMode: "all"`, and `currentMessageId: "42"`, but the helper short-circuits before assigning `replyToId`; the followup drain also never passes a current message id into that helper.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared reply-threading and followup handoff path, with regression tests that distinguish parser-default false from intentional non-threading rather than adding channel-specific workarounds.
