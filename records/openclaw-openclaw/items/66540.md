## Summary

Keep open: current `main` still treats `replyToCurrent: false` as a veto before implicit `replyToId` assignment, and the issue-linked fix PRs are closed unmerged.

## What This Changes

The issue asks for implicit reply threading to survive payloads whose `replyToCurrent: false` value came from the no-tag directive default.

## Best Possible Solution

Update the shared reply-threading contract so parser-default false does not suppress implicit current-message threading, add a regression test for that payload shape, and preserve intentional non-threading through the existing policy path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `applyReplyThreading({ payloads: [{ text: "hello", replyToCurrent: false }], replyToMode: "all", currentMessageId: "42" })`; current main leaves `replyToId` unset because line 41 short-circuits the implicit assignment.

## Solution Assessment

Yes. A narrow shared reply-threading fix plus regression coverage is the right direction, provided it distinguishes parser-default false from intentional non-threading command policy.
