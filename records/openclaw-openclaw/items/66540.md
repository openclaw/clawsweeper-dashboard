## Summary

Keep open: current `main` still treats `replyToCurrent: false` as a veto before implicit `replyToId` assignment, and the issue-linked fix PRs are closed unmerged.

## What This Changes

The issue asks for queued or normalized reply payloads with parser-default `replyToCurrent: false` to keep implicit reply threading instead of losing `replyToId`.

## Best Possible Solution

Update the shared reply-threading contract so parser-default false does not suppress implicit current-message threading, while preserving intentional non-threading through an explicit policy or metadata path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `applyReplyThreading({ payloads: [{ text: "hello", replyToCurrent: false }], replyToMode: "all", currentMessageId: "42" })`; current line 41 leaves `replyToId` unset.

## Solution Assessment

Yes. A narrow reply-threading repair is the right direction, but it should separate parser-default false from intentional non-threading rather than blindly removing every false check.
