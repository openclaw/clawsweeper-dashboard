## Summary

Keep open: current main and the latest release still contain the false-default veto, and the followup delivery path still omits the queued message id. The related open PR at https://github.com/openclaw/openclaw/pull/63637 covers only the current-message handoff, not the `replyToCurrent: false` veto.

## What This Changes

The issue asks OpenClaw to preserve implicit `replyToId` threading for queued or followup replies when `replyToCurrent: false` came from the parser default rather than an explicit opt-out.

## Best Possible Solution

Fix the shared threading helper so parser-default false does not block implicit threading, forward the queued current message id through followup delivery, and preserve command reply opt-out behavior with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A payload like `{ text: "Hello", replyToCurrent: false }` with `replyToMode: "all"` and `currentMessageId: "42"` short-circuits before `replyToId` assignment, and the followup path still never passes the queued message id.

## Solution Assessment

Yes. The maintainable fix belongs in the shared reply-threading helper plus the followup current-message handoff, while preserving the existing command reply opt-out tests rather than adding channel-specific workarounds.
