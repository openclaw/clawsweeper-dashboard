## Summary

Keep open: current main still treats `replyToCurrent: false` as a veto before implicit `replyToId` assignment, and the followup delivery path still lacks the queued current-message handoff. The related open PR only covers the handoff portion, while the false-default fix attempts are closed unmerged.

## What This Changes

The issue reports that parser-default `replyToCurrent: false` prevents implicit `replyToId` assignment for queued or followup replies.

## Best Possible Solution

Distinguish parser-default false from intentional non-threading, forward the queued current message id through followup delivery, and add focused regression coverage while preserving command reply opt-out behavior.

## Reproduction Assessment

Yes, at source level. On current main, a payload shaped as `{ text: "Hello", replyToCurrent: false }` with `replyToMode: "all"` and `currentMessageId: "42"` short-circuits before `replyToId` assignment, and the followup drain also omits the queued message id from the helper call.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared reply-threading helper plus followup handoff, with tests that separate default false from intentional command no-threading rather than adding channel-specific workarounds.
