## Summary

Keep open: current `main` still treats `replyToCurrent: false` as a veto before implicit `replyToId` assignment, while ordinary directive parsing can still produce that false value and all issue-linked fix PRs are closed unmerged.

## What This Changes

The issue asks for queued or normalized reply payloads with parser-default `replyToCurrent: false` to keep implicit reply threading instead of losing `replyToId`.

## Best Possible Solution

Update the shared reply-threading contract so parser-default false does not suppress implicit current-message threading, while moving intentional non-threading behavior onto an explicit policy or metadata path with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `applyReplyThreading({ payloads: [{ text: "hello", replyToCurrent: false }], replyToMode: "all", currentMessageId: "42" })`; current line 41 leaves `replyToId` unset.

## Solution Assessment

Yes, a narrow reply-threading repair is the right direction, but it should not be a blind removal of every false check. The safer fix separates parser-default false from explicit command/policy opt-out and verifies the queued/followup target path.
