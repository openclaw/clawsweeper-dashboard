## Summary

Keep open: current main still contains the reported false-default veto and the followup delivery path still does not provide the queued current message id, so the central reply-threading bug remains actionable.

## What This Changes

The issue requests that implicit reply threading continue to set `replyToId` when a queued or followup payload carries parser-default `replyToCurrent: false`.

## Best Possible Solution

Distinguish parser-default false from intentional non-threading, forward the queued current message id through followup delivery, and add focused regression coverage for both paths.

## Reproduction Assessment

Yes, at source level. On current main, a payload like `{ text: "Hello", replyToCurrent: false }` with `replyToMode: "all"` and `currentMessageId: "42"` short-circuits before `replyToId` assignment, and the followup path still omits the queued message id.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared reply-threading helper plus the followup current-message handoff, with tests that preserve command reply opt-out behavior instead of adding channel-specific workarounds.

Fix https://github.com/openclaw/openclaw/issues/66540 on current main. The bug is that `resolveReplyThreadingForPayload` treats `payload.replyToCurrent === false` as an implicit-threading veto even though `parseInlineDirectives` returns false by default when no reply tag is present, and followup delivery also needs the queued current message id to reach `applyReplyThreading`. Implement the smallest reply-threading/followup fix that distinguishes parser-default false from intentional non-threading, preserves command reply opt-out behavior from `commands-core.ts`, and preserves `replyThreading.implicitCurrentMessage: "deny"`. Check the state of https://github.com/openclaw/openclaw/pull/63637 first; if it has landed, build on it, and if it is still open, either avoid duplicate handoff changes or make the replacement clearly cover the same handoff. Add focused regression tests for `applyReplyThreading` with `replyToCurrent: false`, followup delivery with a current message id, and command/send-policy opt-out behavior. Do not add a new config option, channel-specific workaround, or broad reply-routing refactor. Add a single-line `CHANGELOG.md` fix entry if this remains a user-visible behavior fix at landing time, without thanking bot or maintainer accounts.
