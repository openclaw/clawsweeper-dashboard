## Summary

Keep this PR open. Current main still does not implement offline APNs chat-reply alerts, and the proposed branch is an active but incomplete implementation candidate with blocking payload/session/privacy issues before it can be safely wired or merged.

## What This Changes

The PR adds a new Gateway `chat-apns-notify` helper and tests for sending APNs alerts with chat reply text to offline main-session iOS requester devices, while deferring all lifecycle call-site wiring to a follow-up.

## Best Possible Solution

Complete or supersede the PR with an end-to-end Gateway+iOS implementation: add a dedicated chat-reply APNs payload kind and iOS route, make notification content privacy-safe or explicitly configurable, use current canonical main-session resolution, wire both agent-event and direct `chat.send` completion paths with requester device/connection tracking, and cover direct, relay, skip, and cleanup cases.

## Reproduction Assessment

Yes. The issues are statically reproducible from the PR diff and current main: the helper is not imported by any chat lifecycle path, its default main-session check rejects canonical Gateway keys, and its send path uses the existing `push.test` APNs payload builder.

## Solution Assessment

No. The branch is a useful seed, but the narrow maintainable solution needs a dedicated chat notification contract plus full call-site wiring and iOS routing before the feature can work safely.

Full review comments:

- **[P2] Send chat replies with a chat-specific APNs kind:** `src/gateway/chat-apns-notify.ts:73-77`
  - body: `sendApnsAlert` builds the current `push.test` payload kind, so a wired chat reply would be indistinguishable from diagnostic test pushes and could not be routed as a chat notification. Add a dedicated chat-reply payload/send helper before using this from chat completion paths.
  - confidence: 0.95
- **[P2] Compare against canonical main session keys:** `src/gateway/chat-apns-notify.ts:35-39`
  - body: The default guard compares the request session to raw `cfg.session.mainKey`, but current Gateway chat paths use canonical keys such as `global` or `agent:<id>:<mainKey>`. Without resolving the canonical main session key, valid main-session replies can return early and never notify.
  - confidence: 0.9
- **[P2] Avoid exposing reply text in APNs payloads by default:** `src/gateway/chat-apns-notify.ts:57-60`
  - body: The helper turns assistant reply text into the APNs alert body. That content can traverse APNs or the relay and appear on lock screens; use a generic body or explicit opt-in before sending chat transcript content in push payloads.
  - confidence: 0.82
- **[P2] Wire the helper before merging the feature:** `src/gateway/chat-apns-notify.ts:24-34`
  - body: The new helper is never called by `server-chat` or direct `chat.send`, so merging this PR would not actually send offline chat reply alerts. Add the lifecycle integration in the same feature change or keep this as unmerged preparatory work.
  - confidence: 0.88
