## Summary

Keep open: current main has some model-resolution readiness caching, but the OpenAI-compatible transport still builds per-call clients/fetch adapters and the guarded fetch path closes request dispatchers instead of maintaining a provider-level pool. The requested pooling, cache TTL, and warmup behavior are real feature/design work rather than a conservative cleanup close.

## What This Changes

Request HTTP connection pooling, model-resolution caching, and provider warmup to reduce 7-8s startup stages for custom OpenAI-compatible provider agent runs.

## Best Possible Solution

Design and implement measured provider transport/model-resolution reuse in core while preserving auth freshness, SSRF/DNS-pinning, proxy, timeout, and workspace/plugin invalidation guarantees.

## Reproduction Assessment

No high-confidence live reproduction is available from the report alone. Source inspection confirms current main lacks a long-lived provider HTTP pool in the OpenAI-compatible guarded fetch path, but the reported 7-8s timing still needs the reporter's endpoint/config or an equivalent live fixture.

## Solution Assessment

Unclear as proposed. Connection pooling may be part of the right fix, but a configurable model-resolution TTL and gateway warmup need maintainer design against the current auth, SSRF, proxy, and provider lifecycle contracts.
