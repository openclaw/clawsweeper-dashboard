## Summary

Keep open: current main still performs model resolution on each embedded run and constructs guarded provider HTTP dispatchers per request; the user reports are credible, but the requested pooling/cache/warmup design crosses provider transport, auth freshness, startup behavior, and network-security boundaries.

## What This Changes

The issue requests shared provider HTTP connection pooling, model-resolution/auth metadata caching, and gateway warmup to reduce repeated per-run provider startup latency.

## Best Possible Solution

Design a bounded provider-transport reuse layer keyed by provider base URL and resolved request policy, prove it with a controlled OpenAI-compatible benchmark, and split TTL caching, auth freshness, warmup, and session-resource-loader work into explicit maintainer-approved follow-ups.

## Reproduction Assessment

No: source inspection confirms the repeated current-main resolution and per-request dispatcher lifecycle, and multiple users provided timing logs, but there is not yet a controlled current-main reproduction endpoint/config that proves the 7-8s delay is specifically connection setup.

## Solution Assessment

Unclear: HTTP pooling is plausible, but bundling it with configurable resolved-model/auth TTL caching and gateway warmup is not yet the narrowest maintainable plan; focused dispatcher reuse plus instrumentation is the safer first design step.
