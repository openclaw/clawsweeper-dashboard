## Summary

Keep open: current `main` still resolves the model path on every embedded run and the provider HTTP transport does not expose a cross-run connection/cache design, but the requested pooling/cache/warmup plan needs maintainer product and network-security review before implementation.

## What This Changes

The issue asks for reusable provider HTTP connections, model-resolution caching, and gateway provider warmup to reduce repeated custom OpenAI-compatible provider startup latency.

## Best Possible Solution

Design a bounded provider-transport reuse layer, backed by benchmarks and tests, that preserves SSRF pinning/proxy/TLS policy and only then decide whether model metadata TTL caching or gateway warmup should become user-facing behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully: source inspection confirms repeated per-run model resolution and per-request dispatcher/client creation, but the reported 7-8s current-main latency needs the reporter's custom provider or an equivalent controlled slow endpoint to reproduce.

## Solution Assessment

Is this the best way to solve the issue? Unclear: HTTP pooling is plausible, but coupling it with model metadata/auth TTL caching and gateway warmup is a product/security design choice; a safer first step is focused instrumentation plus dispatcher reuse keyed by provider request policy.
