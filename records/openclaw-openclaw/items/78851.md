## Summary

Keep open: current main still runs model resolution on every embedded agent run and provider HTTP fetches build per-request guarded dispatchers, while the additional MiniMax/Codex and active-memory reports show this remains an active latency problem rather than stale cleanup. The proposed pooling, metadata caching, and gateway warmup plan needs maintainer provider-runtime and network-security design before implementation.

## What This Changes

The issue requests provider HTTP connection pooling plus model-resolution caching and startup warmup to reduce per-run latency for custom OpenAI-compatible providers and embedded active-memory runs.

## Best Possible Solution

Define a bounded provider transport reuse design with benchmarks and invariant tests for SSRF/proxy/TLS/timeout cleanup, then decide separately whether model/auth TTL caching or active-memory session reuse belongs in the product contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: source inspection confirms repeated per-run model resolution and per-request dispatcher lifecycle, but the reported wall-clock delay still needs a controlled slow OpenAI-compatible provider, Codex setup, or reporter logs/config to reproduce against current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: HTTP pooling is plausible, but bundling it with model/auth TTL caching and gateway warmup is a security and product decision; a safer first step is instrumented dispatcher reuse keyed to provider request policy, with active-memory/session-resource work tracked separately.
