## Summary

Keep open: current main still resolves the model path during each embedded run and builds guarded provider transport per request, while the requested cross-run pooling/cache/warmup design is not merged and needs provider-runtime security review.

## What This Changes

Request cross-run provider HTTP connection pooling, model/auth metadata caching, provider warmup, and timeout/config controls to reduce per-run model-resolution latency.

## Best Possible Solution

Design and benchmark a bounded provider-transport reuse layer, add substage instrumentation for discovery/auth/transport setup, and only add TTL caching or configurable discovery timeouts after preserving the existing SSRF/proxy/TLS contracts.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction from this review: source inspection confirms the repeated model-resolution and per-request guarded-dispatcher paths, but the reported seconds-long delay needs the reporters' providers or a controlled slow OpenAI-compatible endpoint to isolate.

## Solution Assessment

Is this the best way to solve the issue? Unclear: HTTP pooling is plausible, but combining it with model metadata TTL caching, configurable discovery timeout, and gateway warmup needs a maintainer design that preserves network security boundaries.
