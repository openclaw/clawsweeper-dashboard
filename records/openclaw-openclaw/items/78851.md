## Summary

Keep open: current main still has the reported per-run model-resolution and per-request provider dispatcher shape, and the requested pooling/cache/warmup behavior needs provider-runtime plus network-security design rather than conservative cleanup closure.

## What This Changes

Request HTTP connection reuse, model-resolution caching, and provider warmup to reduce repeated per-agent-run startup latency for custom OpenAI-compatible providers.

## Best Possible Solution

Design a bounded provider-transport reuse layer with benchmark proof and safety tests around SSRF pinning, proxy/TLS policy, dispatcher cleanup, and any model/auth TTL cache semantics.

## Reproduction Assessment

Unclear for the exact latency: current source shows repeated per-run model resolution and per-request guarded dispatcher creation, but the 7-8 second delay still needs a reproducible custom/Codex provider setup or controlled slow endpoint. The missing pooling/cache behavior is source-reproducible from the inspected paths.

## Solution Assessment

Unclear as proposed: HTTP pooling is plausible, but combining it with resolved-model/auth TTL caching and gateway warmup is not yet the narrowest proven fix. A safer path is instrumentation plus a scoped dispatcher-reuse design keyed by the existing provider request policy.
