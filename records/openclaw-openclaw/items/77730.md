## Summary

Keep open: current main still declares the intended file-transfer node-host commands and builds the node handshake from the plugin registry, but there is no merged proof that a Windows node host now advertises those plugin commands in the live connection. The open exec-approval PR and the closed commandless-node diagnostics issue are adjacent, not fixes for this file-transfer plugin registration path.

## What This Changes

The issue reports that the bundled file-transfer plugin’s node-host commands are missing from the live Windows node handshake, causing dir.list, file.fetch, dir.fetch, and file.write invocations to be rejected as unsupported.

## Best Possible Solution

Prove the Windows node-host plugin registry contents at live connect time, then fix the generic node-host plugin activation or bundled plugin resolution path without weakening file-transfer authorization policy.

## Reproduction Assessment

Partially source-reproducible: current source confirms a live node whose declared commands omit dir.list will be rejected even when gateway.nodes.allowCommands includes it. I did not run the required Windows node-host live setup, so the exact drop point is not reproduced.

## Solution Assessment

Unclear. The expected behavior is the existing contract, but the best fix depends on live proof of whether Windows node-host plugin activation, bundled plugin resolution, registry registration, or gateway reconciliation drops the commands.
