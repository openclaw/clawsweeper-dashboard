## Summary

Keep open: current main still has the intended file-transfer command declaration and node-host handshake path, but I did not find merged proof that Windows node hosts now advertise those plugin commands in the live connection. The merged pending-reapproval diagnostics and open metadata-hardening PR are useful adjacent work, but neither proves this specific live-handshake failure is fixed.

## What This Changes

The issue reports that file-transfer plugin node-host commands are missing from the Windows node host live handshake, causing gateway file-transfer tools to fail as unsupported despite config enabling them.

## Best Possible Solution

Prove the current Windows node-host declaration boundary first, then fix the generic node-host plugin activation or bundled plugin resolution path without weakening node command approval or file-transfer path policy.

## Reproduction Assessment

Source-reproducible in part: current source confirms that a node whose live commands omit dir.list will be rejected even when gateway allowCommands includes it. No live Windows current-main run has proven where the file-transfer declarations are dropped.

## Solution Assessment

Unclear. Pending-reapproval diagnostics and metadata hardening are useful adjacent work, but this issue still needs proof of whether the Windows node host fails plugin activation, bundled plugin resolution, runtime registry registration, or command reconciliation.
