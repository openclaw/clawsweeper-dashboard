## Summary

Keep open: current main still preserves the default/manual stop-timeout stale-task path, and https://github.com/openclaw/openclaw/pull/70056 is an open, unmerged closing PR for this issue. ## What This Changes

The issue asks for channel stop timeouts to mark the account stopped, clear stale task/abort bookkeeping, and allow later starts or health recovery.

## Best Possible Solution

Keep this open until the linked closing PR or an equivalent guarded gateway lifecycle fix lands, preserving the recovery-stop behavior while resolving the default timeout stale-task state with regression coverage for delayed old-task settlement.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic test shape where a never-settling account task times out after 5 seconds, a later start is skipped by store.tasks.has(id), and the runtime remains running true; I did not execute tests because this review is read-only.

## Solution Assessment

No for the minimal map-delete fix by itself. The maintainable fix should clear or quarantine stale lifecycle bookkeeping only with task-identity or generation guards so old callbacks cannot overwrite a replacement channel.
