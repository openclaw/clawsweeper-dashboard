## What This Changes

The issue asks for channel stop timeouts to mark the channel stopped, clear stale task/abort bookkeeping, and allow subsequent starts or health-monitor recovery.

## Best Possible Solution

Keep this open until the paired PR or an equivalent guarded gateway lifecycle fix lands with regression coverage for restart after timeout and stale old-task callback handling.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic test shape where a never-settling account task times out after 5 seconds, a later start is skipped by store.tasks.has(id), and the runtime remains running true.

## Solution Assessment

No for the originally suggested map cleanup by itself. The maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire contract so a timed-out old lifecycle cannot corrupt a replacement.
