## Summary

Keep open: current main still has a source-reproducible stale task stop-timeout path, and the issue is paired with an open closing PR that has not merged.

## What This Changes

The issue reports that a channel stop timeout can leave stale task and abort entries while runtime still appears running, blocking restart and health recovery.

## Best Possible Solution

Review and land or replace the paired gateway lifecycle fix with stale-task cleanup, task-owned callback fencing, focused regression coverage, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level: a never-settling channel task times out in `stopChannel`, remains in `store.tasks`, and a later `startChannel` reaches the `store.tasks.has(id)` early return. Existing focused tests still encode that blocked-start shape.

## Solution Assessment

No for closing now or applying the direct cleanup alone. The maintainable fix needs stale task cleanup plus lifecycle identity or generation fencing, or an explicit maintainer decision to accept a force-retire recovery contract.
