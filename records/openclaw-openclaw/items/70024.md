## Label Justifications

- `P1`: The issue describes real channel workflows becoming permanently dead until operator intervention, with Telegram and Slack reports in the related cluster.
- `impact:message-loss`: A wedged inbound channel can silently stop receiving or delivering user messages while appearing healthy.
- `impact:session-state`: The runtime snapshot and task/abort store can drift from the actual channel lifecycle after a stop timeout.
- `impact:crash-loop`: The reported failure is a channel availability hang requiring manual gateway or channel restart to recover.

## Summary

Keep open: current main still has a source-reproducible stale-task stop-timeout path, and the linked closing PR is open and unmerged. The merged recovery-stop work improves the runtime state for non-manual restarts, but it still waits for the old task to settle before a replacement can start.

## What This Changes

The issue asks for stopChannel timeout handling to clear stale task/abort entries, mark the channel stopped, and allow manual or health-monitor starts to recover.

## Best Possible Solution

Ship a guarded gateway lifecycle recovery contract that recovers from never-settling stop timeouts while fencing old task callbacks from replacement lifecycles.

## Reproduction Assessment

Yes, at source level. A channel task that ignores abort can make `stopChannel` time out, leave `store.tasks` registered, and make the next `startChannel` return without starting a replacement.

## Solution Assessment

No for the original direct cleanup by itself. The maintainable fix needs task identity or generation fencing so old timed-out lifecycle callbacks cannot clobber a replacement channel.
