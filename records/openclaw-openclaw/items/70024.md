## Summary

Keep open: current main still has a source-reproducible stale-task stop-timeout path, and the paired closing PR is open and unmerged. The merged recovery-stop work partially improved the runtime state, but it still does not recover from a never-settling task without a guarded lifecycle policy.

## What This Changes

The issue reports that channel stop timeouts leave stale task/abort entries and runtime state that block later channel starts and health-monitor recovery.

## Best Possible Solution

Land a guarded gateway lifecycle fix that recovers from never-settling stop timeouts without old task callbacks corrupting replacement lifecycles, with regression tests and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic path where a timed-out stop leaves `store.tasks`, the health monitor calls `startChannel`, and the start guard no-ops until the old task settles.

## Solution Assessment

No for the original direct cleanup by itself. The maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire contract so old lifecycle callbacks cannot corrupt a replacement.
