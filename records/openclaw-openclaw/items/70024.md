## Label Justifications

- `P1`: A timed-out channel stop can leave real channel workflows silently unable to receive new messages until operator intervention.
- `impact:message-loss`: The reported wedge causes inbound channel messages to be ignored or never processed while the gateway appears healthy.
- `impact:session-state`: The bug is rooted in stale runtime/task state that no longer reflects the actual channel lifecycle.

## Summary

Keep open: current main and the latest release still have a source-reproducible stop-timeout path where stale channel task bookkeeping can block recovery, and an open closing PR is already paired with this issue for maintainer review.

## What This Changes

The issue asks OpenClaw to recover cleanly when a channel stop times out by avoiding stale task/abort entries and false running state after timeout.

## Best Possible Solution

Consolidate the open stop-timeout work and land a guarded gateway lifecycle recovery fix that can retire stale bookkeeping without letting old callbacks or provider transports corrupt the replacement lifecycle.

## Reproduction Assessment

Yes, at source level: a channel task that ignores abort can make `stopChannel` time out, leave `store.tasks` registered, and make the next `startChannel` return without starting a replacement. I did not rerun the live Telegram or Slack production scenarios.

## Solution Assessment

No for the original direct cleanup alone. The maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire contract so old lifecycle callbacks and provider transports cannot corrupt a replacement.
