## Summary

Keep open: current main only fixed the non-manual recovery-stop path, while the default `stopChannel` timeout path still preserves `running: true` and leaves the stale task entry that blocks a later manual restart. The issue also has an open, unmerged closing PR, so it should stay open until that PR or an equivalent guarded fix lands.

## What This Changes

The issue requests cleanup of channel stop-timeout state so stale task/abort entries do not leave an account permanently unstartable with a misleading running snapshot.

## Best Possible Solution

Land a guarded gateway lifecycle fix that preserves the recovery-stop behavior from https://github.com/openclaw/openclaw/pull/77686, resolves the default timeout stale-task state, and includes regression coverage for delayed stale task settlement.

## Reproduction Assessment

Yes. Current source plus the existing `server-channels` test fixture show a never-settling task timing out after 5 seconds, a later start being skipped, and the runtime remaining `running: true`; I did not execute tests because this pass is read-only.

## Solution Assessment

No for the minimal cleanup alone. The maintainable fix should clear or replace stale lifecycle bookkeeping only with task-identity or generation guards so old task callbacks cannot overwrite a replacement channel.
