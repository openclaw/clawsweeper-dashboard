## What This Changes

The issue reports that `stopChannel` timeout can leave stale task/abort entries and false runtime state, blocking later channel recovery.

## Best Possible Solution

Land the linked PR or an equivalent guarded gateway lifecycle fix with task identity/generation fencing, regression coverage for never-settling timeouts, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic path where a never-settling account task times out, the timeout branch returns before cleanup, and a later start is skipped by `store.tasks.has(id)`.

## Solution Assessment

No for the suggested direct cleanup by itself. The maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire contract so an old timed-out lifecycle cannot corrupt a replacement.
