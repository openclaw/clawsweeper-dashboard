## Summary

Close as implemented: current main and the latest stable release already handle recovery stop timeouts by marking recovery stops not running/restart-pending, preserving the duplicate-task guard while allowing restart after the old task settles.

## What This Changes

The issue reports that a channel stop abort-timeout can leave stale task and abort entries while reporting the runtime as running, blocking restart and hiding the dead channel from health recovery.

## Best Possible Solution

Keep the shipped recovery-stop implementation and close the older linked workaround PRs as superseded by the merged gateway lifecycle fix.

## Reproduction Assessment

No current-main failing path remains for the central recovery-stop bug. Source and tests now show recovery stops set running:false/restartPending:true and restart after the old task settles, while manual stops intentionally keep the duplicate-task guard.

## Solution Assessment

Yes, the shipped solution is the maintainable fix: it separates user-initiated manual stops from health/recovery stops instead of blindly deleting stale task state and risking duplicate channel tasks.
