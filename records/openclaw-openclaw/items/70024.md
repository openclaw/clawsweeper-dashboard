## Summary

Keep open: current main still has a source-reproducible stale-task stop-timeout path, and the paired implementation PR remains open and unmerged. The issue should stay open until that PR or an equivalent guarded lifecycle fix lands.

## What This Changes

The issue reports that channel stop timeouts leave stale task and abort entries, causing future starts and health-monitor recovery to no-op.

## Best Possible Solution

Land the paired PR or an equivalent guarded gateway lifecycle fix with task identity or generation fencing, regression coverage for never-settling timeouts, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic path where a never-settling account task times out, the timeout branch returns before cleanup, and a later start is skipped by `store.tasks.has(id)`.

## Solution Assessment

No for the original direct cleanup by itself. The maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire contract so an old timed-out lifecycle cannot corrupt a replacement.
