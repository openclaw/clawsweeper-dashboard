Keep open: current main still has a source-reproducible stop-timeout stale-task path, and https://github.com/openclaw/openclaw/pull/70056 remains an open, unmerged closing PR paired with this issue. ## What This Changes

The issue asks OpenClaw to make timed-out channel stops clear stale task bookkeeping and report stoppage so later starts and health recovery can work.

## Best Possible Solution

Keep this open until the paired PR or an equivalent guarded gateway lifecycle fix lands with regression coverage for restart after timeout and stale old-task callback handling.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic path where a never-settling account task times out after 5 seconds, a later start is skipped by store.tasks.has(id), and the focused test still pins that blocked restart shape.

## Solution Assessment

No for the original suggested map cleanup by itself. The maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire contract so a timed-out old lifecycle cannot corrupt a replacement.
