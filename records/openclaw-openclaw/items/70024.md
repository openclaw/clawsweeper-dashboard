Keep open: current main still exposes the stop-timeout stale-task path, and a same-author implementation PR with closing syntax is still open. The cluster has newer production confirmations, but the linked direct cleanup needs lifecycle-safety review rather than issue cleanup.

## What This Changes

The issue reports that `stopChannel` timeout handling can leave stale channel task/abort state and prevent later restarts.

## Best Possible Solution

Review and revise https://github.com/openclaw/openclaw/pull/70056, or replace it with a guarded gateway lifecycle fix informed by https://github.com/openclaw/openclaw/issues/71412 that clears the recovery deadlock without stale callbacks corrupting a newer task.

## Reproduction Assessment

Yes at source level, but not from a live gateway run in this review. Current main still has the timeout early return, the task-map start guard, and tests that pin the blocked manual restart behavior.

## Solution Assessment

No for the suggested direct cleanup by itself. The narrow maintainable fix needs to free or retire the stale lifecycle while fencing old task-owned callbacks and preserving the current recovery-stop semantics.
