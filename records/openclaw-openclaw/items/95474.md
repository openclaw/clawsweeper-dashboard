## Summary

## What This Changes

The issue reports that Codex synthetic missing_tool_result local tool failures are treated as unclassified provider failures and consume configured cross-provider model fallbacks.

## Best Possible Solution

Land one canonical fix that treats Codex missing_tool_result as a local non-provider tool-execution failure while preserving explicit provider-error fallback behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction at the fallback boundary: missing_tool_result is unclassified on current main, and unrecognized candidate errors advance to later configured fallback models. I did not run the long live Codex shell-hang scenario.

## Solution Assessment

Yes: the narrow maintainable fix is to extend the existing non-provider runtime coordination guard, as the linked PR does, with tests that explicit provider metadata still wins. A new competing fix PR should not be queued while that linked PR is open.
