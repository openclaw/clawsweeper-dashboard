## Summary

Keep open: current main still has a source-visible path where a synthetic Codex missing_tool_result becomes a local prompt/tool failure and can still advance configured model fallbacks as an unknown candidate failure; no linked open PR now owns this narrower fix.

## What This Changes

Request to classify Codex synthetic missing_tool_result and similar local native-tool execution failures as non-failover-eligible so configured provider fallbacks do not switch models after a hung shell command.

## Best Possible Solution

Add a narrow local-tool or non-provider failure classification that surfaces or retries missing_tool_result on the same model without consuming configured model fallback candidates, while leaving true provider failures and the broader Codex completion work on their existing paths.

## Reproduction Assessment

Yes at source level: current main can synthesize missing_tool_result as a prompt/local tool error and then route the thrown error through prompt or outer fallback paths. I did not run the long-hanging shell-command scenario live in this read-only review.

## Solution Assessment

Yes: a narrow non-failover classification for local native-tool failures is the maintainable fix. The optional per-tool timeout is useful follow-up, not required to stop provider jumping.
