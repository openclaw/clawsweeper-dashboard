## Summary

Keep open: current main still has a source-visible path where a synthetic Codex `missing_tool_result` becomes a local prompt/tool error and the outer model-fallback loop can continue to the next provider for an unclassified non-abort error. No linked or searched PR currently owns this narrow classification fix.

## What This Changes

Classify synthetic Codex `missing_tool_result` and other local native tool-execution failures as non-failover-eligible so model fallback does not switch providers for a local shell failure.

## Best Possible Solution

Add a narrow local-tool/non-provider failure classification that stops `missing_tool_result` from consuming fallback candidates, while leaving broader Codex completion hangs tracked separately.

## Reproduction Assessment

Yes at source level: current main can synthesize `missing_tool_result` as a prompt/local tool error and then let the outer fallback loop continue to the next configured model. I did not run the long-hanging shell command live in this read-only review.

## Solution Assessment

Yes, the narrow non-failover classification is the best fix shape; the optional per-tool timeout is useful follow-up but not required to stop provider jumping.
