## Summary

Keep open: current main and v2026.6.9 still synthesize `missing_tool_result` but do not classify it as a local non-provider failure at the model-fallback boundary, and open closing PRs already own the repair path.

## What This Changes

The issue asks OpenClaw to treat Codex `missing_tool_result` local tool-execution failures as non-failover-eligible so cross-provider model fallbacks are not consumed.

## Best Possible Solution

Land one narrow linked PR, preferably the proof-positive boundary fix in https://github.com/openclaw/openclaw/pull/95543 if maintainer review agrees, then close this issue as implemented.

## Reproduction Assessment

Yes at source level: current main can synthesize a Codex `missing_tool_result` prompt/local-tool failure, and `runWithModelFallback` still lets unrecognized thrown errors advance to later model candidates. I did not run the slow live hanging-shell scenario in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow local-tool/non-provider classification at the existing failover boundary, with tests proving true provider failures remain fallback-eligible. A per-tool timeout may be useful follow-up, but it is not required to stop provider jumping.
