## Summary

Keep open: current main still has a source-visible path where a synthetic Codex `missing_tool_result` is treated as an unclassified prompt/tool failure and can consume the next configured provider fallback candidate. The related open drain PR may reduce false missing results, but it does not address the fallback classification for a real local tool-execution failure.

## What This Changes

The issue requests that Codex local tool-execution failures such as `missing_tool_result` be treated as non-failover-eligible so they do not switch the session to another provider/model.

## Best Possible Solution

Add a narrow local-tool/non-provider failure classification that prevents `missing_tool_result` and equivalent native tool lifecycle failures from consuming model fallback candidates, while leaving the broader Codex completion-stall policy tracked separately.

## Reproduction Assessment

Yes at source level: current main can synthesize `missing_tool_result` as a prompt/local tool error and then let the fallback loop advance to the next configured model. I did not run the long-hanging shell command live in this read-only review.

## Solution Assessment

Yes: the narrow non-failover classification is the best fix shape; a per-tool timeout may be useful follow-up, but it is not required to stop cross-provider model jumping.
