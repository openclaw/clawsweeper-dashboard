## Summary

Keep open: current main still has a source-visible path where a synthetic Codex missing_tool_result becomes a local prompt error and the outer model-fallback loop can record the primary model as candidate_failed and continue to the next configured provider. This is narrower than the broader Codex turn-completion issue and is not already fixed by the synthetic tool-result invariant work.

## What This Changes

The issue requests that Codex local tool-execution failures such as missing_tool_result be treated as non-failover-eligible instead of triggering cross-provider model fallback.

## Best Possible Solution

Add a narrow non-provider/local-tool failure classification that stops missing_tool_result from consuming model fallback candidates, while leaving true provider failures and the broader Codex completion work on their existing paths.

## Reproduction Assessment

Yes at source level: current main can synthesize missing_tool_result as a local prompt error, throw it, and then let the outer fallback loop continue to the next candidate when another model is configured. I did not run the long-hanging shell-command scenario live in this read-only review.

## Solution Assessment

Yes, the proposed non-failover classification is the right direction, but the fix should cover both prompt-stage handling and the outer model-fallback loop rather than only adding a result-classifier case. The optional per-tool timeout is useful follow-up, not required to fix the provider jumping.

Fix the Codex/local-tool missing_tool_result fallback bug reported in https://github.com/openclaw/openclaw/issues/95474. On current main, a missing native Codex bash result can be synthesized as missing_tool_result, thrown as a local prompt/runtime error, and then consumed by the outer model fallback loop as candidate_failed, advancing agents.defaults.model.fallbacks to another provider. Add a narrow non-provider/local-tool failure classification or equivalent control-flow guard so missing_tool_result, denied/timed-out/orphaned local tool execution failures surface or retry on the same model without consuming cross-provider model candidates. Do not add config or env knobs, do not undo the synthetic tool.result invariant from https://github.com/openclaw/openclaw/pull/88946, and keep the broader Codex turn-completion/hang work in https://github.com/openclaw/openclaw/issues/88312 separate.
