## Summary

Keep open: current main still has a source-visible path where Codex `missing_tool_result` is produced as a local native-tool failure, but the central failover guard only recognizes other local coordination errors, so the issue should stay open until one linked fix PR lands.

## What This Changes

The issue asks OpenClaw to treat Codex `missing_tool_result` local tool-execution failures as non-failover-eligible so configured model fallbacks do not switch providers for a hung local command.

## Best Possible Solution

Land one canonical central non-provider guard for Codex missing local tool results, with tests preserving provider-error fallback precedence; https://github.com/openclaw/openclaw/pull/95543 currently looks like the strongest linked candidate.

## Reproduction Assessment

Yes at source level: current main can synthesize `missing_tool_result` as a Codex prompt/local tool error, and the fallback loop can still continue unclassified errors to later configured candidates. I did not rerun the slow live hanging-shell scenario.

## Solution Assessment

Yes: the maintainable fix is a narrow central non-provider/local-tool classification at the model-fallback boundary, with provider metadata precedence tests. A per-tool timeout would be useful follow-up, not the necessary fix for provider jumping.
