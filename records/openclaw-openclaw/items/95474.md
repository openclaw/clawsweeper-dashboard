## Summary

Keep open: current main and the latest release still leave the synthetic Codex `missing_tool_result` outside the local non-provider fallback guard, while open PRs now own candidate fixes. This issue should stay open until one canonical fix, preferably the proof-positive central guard path, lands and closes it.

## What This Changes

The issue asks OpenClaw to treat Codex `missing_tool_result` local tool-execution failures as non-failover-eligible so configured provider/model fallbacks do not switch providers for a local command failure.

## Best Possible Solution

Land one central non-provider guard for Codex `missing_tool_result` local failures, preserving true provider-error fallback, then close this issue after the merged PR reaches main.

## Reproduction Assessment

Yes at source level: current main synthesizes `missing_tool_result`, does not include it in the local non-provider guard, and the fallback loop continues unknown errors to later candidates. I did not run a live hung-shell Codex OAuth scenario.

## Solution Assessment

Yes, the best fix shape is a narrow central local/non-provider classification that prevents candidate consumption while preserving real provider fallback. The open proof-positive PR https://github.com/openclaw/openclaw/pull/95543 appears to be the strongest current candidate, but it has not merged yet.
