## Label Justifications

- `P3`: The issue describes misleading update UX for container deployments, a low-urgency operator experience problem rather than an outage.
- `impact:other`: The concrete impact is deployment-specific update guidance that can waste operator time but does not fit the more specific impact labels.

## Summary

Keep open: current main and the latest release still expose local `openclaw update` guidance and the Control UI update action without a container update policy, while https://github.com/openclaw/openclaw/pull/94310 remains the broad open implementation candidate and https://github.com/openclaw/openclaw/pull/95887 covers only the direct CLI permission-failure slice.

## What This Changes

Request containerized-environment detection so OpenClaw suppresses or replaces non-durable local update suggestions and UI update actions with image redeploy guidance.

## Best Possible Solution

Resolve this by landing one maintainer-approved container update policy that keeps CLI status, startup logs/cache, gateway snapshots/events, Control UI, and direct update failure guidance aligned.

## Reproduction Assessment

Yes, source-level: current main can cache, broadcast, log, and render package update availability without applying the shared container detector. I did not run a live Docker or Kubernetes gateway/UI reproduction in this read-only review.

## Solution Assessment

Unclear until the linked PR policy is settled. The shared update producer is the right technical layer for snapshots and async UI events, but maintainers still need to decide explicit auto-update and detector false-positive behavior.
