## Summary

Keep open. Current main still has the load-time session-file repair hook, but that hook only repairs malformed JSONL, empty assistant error turns, and blank user text; it does not persist the existing tool-use/tool-result pairing repair, so the restart-surviving corruption gap remains actionable.

## What This Changes

The issue asks for session-file load repair that rewrites orphaned, duplicate, missing, or displaced tool-result/tool-call pairings on disk after interruption-related corruption.

## Best Possible Solution

Extend the existing session-file repair path to persist the same conservative pairing invariants already enforced during replay: drop truly orphaned tool results, move displaced matching results next to their assistant tool-call turn, synthesize only storage-safe missing results, keep backups, and document the on-disk behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is a valid session JSONL file with a header plus an orphaned or displaced `toolResult`; current file repair parses and keeps that entry unchanged unless one of the existing malformed-line, empty-assistant, or blank-user branches fires.

## Solution Assessment

Yes. Repair-on-load is the narrowest maintainable direction because run and compaction already call `repairSessionFileIfNeeded`; adding a new opt-in config should be a maintainer product choice, not a prerequisite for fixing the persisted corruption loop.
