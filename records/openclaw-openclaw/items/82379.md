## Summary

Keep open: the docs clarification is useful, but the PR currently documents the wrong complete Lobster env-var contract. The pinned runtime exposes `LOBSTER_ARGS_JSON` plus normalized `LOBSTER_ARG_*` values, and I found no runtime support for `LOBSTER_FINAL_STATUS`; a maintainer follow-up has already asked for that drift to be fixed before merge.

## What This Changes

The PR adds a Lobster docs subsection that lists injected workflow step env vars, warns against unsupported output-env patterns, and points readers to stdin, template, and condition-based prior-step output access.

## Best Possible Solution

Update the docs to match `@clawdbot/lobster@2026.4.6`: document `LOBSTER_ARGS_JSON` and normalized `LOBSTER_ARG_<NAME>`, keep the warning against env-based step outputs, and do not document `LOBSTER_FINAL_STATUS` unless the runtime adds it.

## Reproduction Assessment

Yes. The mismatch is source-reproducible by comparing the PR’s “complete” env-var table with the pinned `@clawdbot/lobster@2026.4.6` `mergeEnv` implementation and package-wide env-name search.

## Solution Assessment

No. A targeted docs clarification is the right solution, but this PR should first align the table with the pinned runtime instead of documenting an unsupported final-status variable and omitting `LOBSTER_ARGS_JSON`.
