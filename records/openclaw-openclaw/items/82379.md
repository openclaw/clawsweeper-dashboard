## Summary

Keep open: the docs-only direction is useful and matches the linked open docs issue, but the latest PR head still documents an inaccurate complete Lobster env contract. The pinned runtime injects `LOBSTER_ARGS_JSON` plus normalized `LOBSTER_ARG_*`, and I found no `LOBSTER_FINAL_STATUS` injection; a maintainer follow-up already asked for the same correction.

## What This Changes

The PR adds a Lobster docs subsection that enumerates injected workflow step environment variables, warns against unsupported output-env patterns, and points readers to supported prior-step output access.

## Best Possible Solution

Update the docs to match the pinned Lobster runtime contract, keep the step-output guidance docs-only, and avoid adding or documenting unsupported env-based step output behavior.

## Reproduction Assessment

Yes. The mismatch is source-reproducible by comparing the PR table to `@clawdbot/lobster@2026.4.6`, where `mergeEnv` injects `LOBSTER_ARGS_JSON` and normalized `LOBSTER_ARG_*` but no `LOBSTER_FINAL_STATUS`.

## Solution Assessment

No. A targeted docs clarification is the right approach, but this PR should first align the table with the pinned runtime instead of documenting an unsupported final-status variable and omitting `LOBSTER_ARGS_JSON`.
