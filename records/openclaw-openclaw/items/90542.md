## Summary

Keep open: the patch is not merge-ready because it repurposes the shipped `allow` mode even though the existing `deny` + `allowIn` path already provides scoped allowlists, leaves generated config UI metadata stale, and lacks inspectable real behavior proof.

## What This Changes

The PR changes mention-pattern policy so explicit provider `mentionPatterns.mode: "allow"` requires `allowIn`, updates five channel config UI hint files, and adds resolver-focused tests.

## Best Possible Solution

Keep shipped `allow` semantics broad and use `mode: "deny"` plus `allowIn` for scoped allowlists, or make any breaking contract change explicit with maintainer approval, docs/generated metadata, migration guidance, and real transport proof.

## Reproduction Assessment

Yes. Source inspection shows current main enables explicit `allow` mode unless `denyIn` matches; that reproduces the behavior this PR changes, but it is also the shipped documented contract.

## Solution Assessment

No. The existing `mode: "deny"` plus `allowIn` path already implements scoped allowlists without breaking `allow`, so changing the shipped meaning of `allow` is not the narrowest maintainable fix.
