## Summary

Keep this PR open, but it is not merge-ready: the patch changes the shipped `mentionPatterns.mode: "allow"` contract, leaves generated/public config metadata out of sync, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes provider mention-pattern resolution so explicit `mode: "allow"` requires an `allowIn` match, updates five channel config hint strings, and adds resolver-focused tests.

## Best Possible Solution

Preserve broad `allow` semantics and use `mode: "deny"` plus `allowIn` for scoped allowlists, unless maintainers explicitly accept a breaking contract with regenerated metadata, docs, upgrade guidance, and real transport proof.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.6` enable explicit allow mode unless `denyIn` matches; that reproduces the behavior this PR changes, but it is the documented released contract.

## Solution Assessment

No. The existing `mode: "deny"` plus `allowIn` path already implements scoped allowlists without breaking allow mode, so changing the shipped meaning of allow is not the narrowest maintainable fix.
