## Summary

Keep open: current main and v2026.6.6 still define `mode: "allow"` as broad while `mode: "deny"` plus `allowIn` is the scoped allowlist path, so this PR needs maintainer config-contract approval, generated metadata/docs alignment, and inspectable real transport proof before merge.

## What This Changes

The PR changes provider mention-pattern policy so explicit `mode: "allow"` requires `allowIn`, updates Discord/Matrix/Slack/Telegram/WhatsApp config UI hints, and adds resolver tests.

## Best Possible Solution

Keep shipped `allow` semantics broad and use `mode: "deny"` plus `allowIn` for scoped allowlists, or make any breaking contract change explicit with maintainer approval, regenerated metadata/docs, upgrade guidance, and real transport proof.

## Reproduction Assessment

Yes. Source inspection shows current main enables explicit `allow` mode unless `denyIn` matches; that reproduces the behavior this PR changes, but it is also the shipped documented contract.

## Solution Assessment

No. The existing `mode: "deny"` plus `allowIn` path already implements scoped allowlists without breaking `allow`, so changing the shipped meaning of `allow` is not the narrowest maintainable fix.
