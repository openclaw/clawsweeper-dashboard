## Summary

Keep open: the PR addresses real CI/i18n failures, but the publish-token changes broaden cross-repo tokens to every step in two jobs, so it should not merge as-is.

## What This Changes

The PR grants `actions: read` to the scheduled live/E2E reusable workflow caller, skips docs/website publish syncs when cross-repo tokens are absent, and adds i18n keys for two Quick Settings security labels.

## Best Possible Solution

Land a revised patch that keeps the scheduled caller permission fix and i18n regeneration, but detects missing publish tokens with a non-secret output/flag and passes each token only to the checkout, clone, or push step that needs it.

## Reproduction Assessment

Yes for the source-level CI failure: current main has a reusable-workflow permission mismatch and token-backed publish jobs that do not handle absent secrets. I did not run the expensive scheduled workflow or a live tokenless publish skip.

## Solution Assessment

No as written. Adding `actions: read` is the right narrow fix, but the publish-skip implementation should keep the token-presence check separate from the secret value to avoid job-wide token exposure.
