## Summary

Keep open: this is a maintainer-authored draft with the protected `maintainer` label, and current `main` does not yet include the ACP ledger feature. The latest head fixes the earlier prompt-write completeness direction, but the byte cap still measures a different JSON payload than the file writer persists.

## What This Changes

The PR adds an in-memory/file-backed ACP event ledger, wires ACP server and translator session replay through it, and updates ACP tests, docs, and changelog.

## Best Possible Solution

Keep the ledger design, but make the persisted byte budget authoritative for the exact JSON payload written to disk before merge.

## Reproduction Assessment

Yes for the remaining review finding: source inspection shows `trimLedger` measures compact JSON while the pinned writer persists pretty JSON. I did not execute a failing test in this read-only pass.

## Solution Assessment

No, not yet. Ledger replay is the right direction, but the current patch should use the same serializer for measurement and persistence, or otherwise enforce the cap after formatting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Measure the persisted ledger bytes:** `src/acp/event-ledger.ts:254`
  - body: `trimLedger` decides the store is under `maxSerializedBytes` using compact `JSON.stringify(state.store)`, but `writeJsonAtomic` writes two-space pretty JSON. With many retained events, the on-disk `event-ledger.json` can exceed the advertised cap and every append still rewrites that larger file; measure the same formatted payload you persist or write the compact payload you measured.
