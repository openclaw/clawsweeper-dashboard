## Summary

Keep open: this is a draft maintainer PR with the protected `maintainer` label, current `main` does not have the ledger feature, and the PR head fixes the earlier session-key lookup gap, but the patch still has P2 durability and storage-bound issues before merge.

## What This Changes

The PR adds an ACP event ledger with file-backed persistence, uses it to replay bridge-created sessions in `loadSession`, and updates ACP tests, docs, and changelog.

## Best Possible Solution

Keep the ledger approach, but make complete replay contingent on durable bounded writes, with byte-budget truncation and regression coverage for write failure and large payload retention.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: after `startSession` persists a complete session, a failed prompt/update ledger write can leave later replay marked complete with missing events. A large recorded `rawOutput` also demonstrates that count-only caps do not bound the serialized ledger file.

## Solution Assessment

No. The session-key replay lookup is now the right direction, but the current patch should not merge until ledger write failures force incomplete replay or transcript fallback and stored payloads are byte-bounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Mark replay incomplete when ledger writes fail:** `src/acp/translator.ts:1502-1503`
  - body: `newSession` starts the ledger as `complete: true`, but this catch only logs a failed prompt write. If the first prompt write fails after the start record was persisted, a later `loadSession` can see a complete ledger and skip transcript fallback while replaying history with no user prompt. Please flip the ledger/session to incomplete, or otherwise suppress complete replay, after any record failure.
  - confidence: 0.82
- **[P2] Cap ledger storage by serialized bytes:** `src/acp/event-ledger.ts:8-9`
  - body: The retention budget is only 200 sessions by 5,000 events, while recorded `SessionUpdate`s include raw tool output. A few large tool results can make `event-ledger.json` extremely large, and every append rewrites that JSON file. Add a byte budget or truncation path that marks replay incomplete when payloads are trimmed.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain change, but the new persisted ledger needs a byte budget to avoid local resource-exhaustion risk.

Concerns:

- **[medium] Unbounded local ledger growth:** `src/acp/event-ledger.ts:8`
  - body: The file-backed ledger persists full ACP `SessionUpdate` payloads, including raw tool output, under a count-only cap; large outputs can grow the local state file and repeated atomic rewrites enough to exhaust disk or CPU.
