## Summary

Keep open: this is a draft maintainer PR with a protected `maintainer` label, and the source review found a blocking ledger correctness issue around failed sends being recorded as replayable history.

## What This Changes

The PR adds ACP event-ledger storage and replay for `loadSession`, wires a file-backed ledger into the ACP server, and updates ACP docs, tests, and changelog.

## Best Possible Solution

Record prompt events only after a Gateway send is accepted, or mark failed sends as non-replayable, then add a regression test for rejected sends before continuing the draft maintainer review.

## Reproduction Assessment

Yes from source inspection: the changed code records the prompt before `gateway.request("chat.send")`, and the failure path does not remove or invalidate that ledger entry. I did not run a live ACP repro in this read-only review.

## Solution Assessment

No: the ledger direction fits the requested load-session improvement, but the current write point is not the safest implementation because failed sends become replayable history. The narrower fix is to ledger prompts only after accepted sends or explicitly mark failed runs incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Record prompts only after Gateway accepts the send:** `src/acp/translator.ts:947`
  - body: `recordUserPrompt` is queued before `chat.send` has succeeded. If the send rejects, the catch path clears and rejects the prompt but the ledger can remain complete with a `user_message_chunk`, so the next `loadSession` replays a prompt the Gateway never accepted. Move the ledger write after an accepted send, or mark the run/session incomplete on send failure, and cover the rejected-send case.
  - confidence: 0.86
