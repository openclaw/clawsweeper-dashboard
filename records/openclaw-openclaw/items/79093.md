## Summary

Keep open: this is a draft maintainer PR with a protected label, and source review found a remaining P2 correctness bug after the rejected-send prompt fix.

## What This Changes

This PR adds ACP event-ledger storage and replay for `loadSession`, wires file-backed ledger persistence into the ACP server, and updates ACP docs, tests, and changelog.

## Best Possible Solution

Keep the ledger feature, but make replay addressable by both the original ACP session id and the Gateway session key returned from session listing, with regression coverage for list-then-load.

## Reproduction Assessment

Yes, from source inspection: create a default ACP session, then after restart load the row returned by `listSessions`; the listed `sessionId` is the Gateway key (`acp:<uuid>`), while the ledger was stored under the original UUID.

## Solution Assessment

No. The ledger direction fits the feature and the rejected-send prompt case is fixed, but the current lookup is too narrow for the session id that ACP clients receive from `listSessions`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle listed session ids when replaying ledger history:** `src/acp/translator.ts:671`
  - body: `newSession` stores the ledger under the generated UUID while its Gateway key is `acp:<uuid>`, but `listSessions` exposes the Gateway key as the ACP `sessionId`. After a restart, a client that loads the row returned by `listSessions` calls `loadSession` with `acp:<uuid>`, so this lookup misses the complete ledger and falls back to `sessions.get`, losing the tool/session metadata replay this PR is meant to preserve. Please support lookup by the listed Gateway session key or store an alias, and cover the list-then-load path.
  - confidence: 0.87
