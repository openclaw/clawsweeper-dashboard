## Summary

Keep open for maintainer decision. The runtime auth-snapshot half is already fixed, and the latest release no longer registers `google-antigravity` as a bundled provider, but current main still has Antigravity aliases/model/docs leftovers that make support-versus-retirement unresolved rather than safely closeable.

## What This Changes

The issue reports that `google-antigravity` stopped working because the upstream Antigravity client version is deprecated and token refresh leaves stale runtime credentials until gateway restart.

## Best Possible Solution

Pick one consistent boundary: either externalize or restore Antigravity through a provider-owned plugin with current Cloud Code Assist headers and auth-refresh behavior, or remove the remaining aliases/docs/tests and direct users to `google-gemini-cli` or other supported providers.

## Reproduction Assessment

Yes, source-level: current main no longer declares `google-antigravity` in the Google plugin manifest and tests direct login as unknown, while docs/live tests still reference it. I did not run a live Cloud Code Assist OAuth smoke for the reported upstream version rejection.

## Solution Assessment

No, not as filed. Bumping `DEFAULT_ANTIGRAVITY_VERSION` targets an old dependency transport that current main no longer registers, while the generic auth cache bug is already covered; the maintainable path needs an owner decision to externalize/restore or fully retire Antigravity surfaces.
