## Summary

Keep open. Current main has changed since the earlier review, but it still does not restore the reported google-antigravity behavior: OpenClaw still exposes google-antigravity as a Google provider alias while the pinned pi-ai 0.71.1 package no longer registers the google-gemini-cli/Antigravity transport at all, and the runtime auth snapshot path can still return an active in-memory snapshot before checking refreshed auth-profiles.json state. Existing auth-store tests cover some stale-overwrite and internal-save cases, but they do not fully close the external refresh/restart problem described in the report.

## What This Changes

The issue asks to restore google-antigravity by handling the deprecated Antigravity user-agent/version path and by invalidating stale runtime auth snapshots after token refresh.

## Best Possible Solution

Keep the issue open until OpenClaw has an explicit, maintained Antigravity strategy and a generic auth-store snapshot refresh path. The desired end state is that google-antigravity either uses a provider-owned current Cloud Code Assist transport or is deliberately deprecated, and refreshed auth-profiles.json credentials can become active without a gateway restart.

## Reproduction Assessment

Yes for a source-level reproduction: current main still has the google-antigravity alias, no registered pi-ai google-gemini-cli/Antigravity transport in the pinned dependency, and an active runtime snapshot path that can precede disk reload. A live end-to-end reproduction still needs maintainer OAuth credentials for the sandbox endpoint.

## Solution Assessment

No current-main solution is complete. The maintainable fix should be a deliberate Google-provider transport/version decision plus a generic auth snapshot reload or invalidation mechanism, not a hidden provider-specific startup migration.
