## Summary

Keep open. Current main does not provide a complete fix: OpenClaw still exposes google-antigravity as an intended Google provider path, but the current pi-ai dependency no longer ships the old google-gemini-cli/Antigravity transport, and OpenClaw has not added a replacement transport for models that still resolve to api "google-gemini-cli". The auth snapshot concern is also only partially covered by internal save paths; active runtime snapshots still take precedence over a refreshed on-disk auth-profiles.json load.

## What This Changes

The issue reports that the google-antigravity provider uses an obsolete Antigravity transport/version and that runtime auth snapshots can keep stale failed credentials after token refresh.

## Best Possible Solution

Maintain this as an open provider/auth bug until maintainers choose whether google-antigravity remains supported, then either add a provider-owned Cloud Code Assist/Antigravity transport with an accepted version strategy or retire the surface with docs/config cleanup. Separately, runtime auth snapshots need a generic reload or invalidation path so refreshed credentials can become active without a gateway restart.

## Reproduction Assessment

Yes for a source-level reproduction: current main still resolves Antigravity models onto api "google-gemini-cli", but neither pi-ai 0.71.1 nor OpenClaw's boundary/provider stream paths currently register a stream implementation for that API. The stale-auth path is also visible because active runtime snapshots are returned before persisted auth-profiles.json is reloaded.

## Solution Assessment

No current-main solution is complete. The best fix is not a hidden startup migration; it needs an explicit Google-provider transport/version decision plus a generic auth snapshot refresh contract that is safe for OAuth and secrets runtime behavior.
