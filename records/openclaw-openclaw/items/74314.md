## Summary

Keep open. The report remains actionable against current main: the old upstream Antigravity transport/version path changed under OpenClaw, but the current Google plugin still leaves google-antigravity compatibility hooks without a provider-owned stream implementation, and the runtime auth snapshot path can still prefer an in-memory store before reloading refreshed on-disk auth state. The existing ClawSweeper comment correctly identified the same two unresolved areas, so there is no safe close reason yet.

## What This Changes

The issue asks OpenClaw to restore google-antigravity by using an accepted Antigravity client version and to recover from refreshed OAuth credentials without requiring a gateway restart.

## Best Possible Solution

Maintain a clear supported state for this provider path: either restore a provider-owned Cloud Code Assist/Antigravity transport with accepted client metadata and tests, or retire/redirect remaining google-antigravity compatibility surfaces with docs and diagnostics. Separately, auth profile runtime snapshots should observe or invalidate fresher persisted OAuth credentials so successful refreshes recover without a gateway restart.

## Reproduction Assessment

Yes for a source-level reproduction: current main still has google-antigravity compatibility hooks without a matching stream transport, and ensureAuthProfileStoreWithoutExternalProfiles can return an active runtime snapshot before reading refreshed auth-profiles.json. A live reproduction would need maintainer OAuth credentials and current Google sandbox access.

## Solution Assessment

No current-main solution is complete. The requested pi-ai constant bump is no longer the narrow fix because pi-ai 0.71.1 removed the old Antigravity transport surface; the maintainable path needs an explicit restore-or-retire decision plus auth-cache invalidation semantics.
