## Summary

Keep open. Current main still has residual `google-antigravity` aliases, tests, and live-testing docs, but the pinned `pi-ai` dependency no longer ships the old Antigravity transport, so this needs a maintainer support-versus-retire decision rather than cleanup closure.

## What This Changes

The issue reports that `google-antigravity` fails after the upstream Antigravity API rejected `pi-ai`'s `1.21.9` user agent and that refreshed OAuth credentials do not clear stale runtime auth snapshots.

## Best Possible Solution

Maintainers should choose one durable path: restore a supported, tested Antigravity/Cloud Code Assist transport with auth-refresh recovery, or remove the residual `google-antigravity` aliases, live docs, and tests while guiding users to `google` or `google-gemini-cli`.

## Reproduction Assessment

Yes for source-level reproduction: current main still exposes Antigravity aliases/live-test paths and resolves some Antigravity models to `api: "google-gemini-cli"`, while the pinned `pi-ai` package no longer includes the old transport. A live endpoint repro would require Google OAuth credentials.

## Solution Assessment

No. The suggested `DEFAULT_ANTIGRAVITY_VERSION` bump targets `pi-ai` 0.70.x code that is absent from current main's dependency, so the maintainable fix is a support-or-retire decision rather than a one-line version bump.
