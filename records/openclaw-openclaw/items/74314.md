## Summary

Keep open. Current main still pins @mariozechner/pi-ai 0.70.6, whose Antigravity transport defaults to user-agent version 1.21.9, and the active runtime auth snapshot path still returns an in-memory snapshot before reloading auth-profiles.json from disk.

## What This Changes

The issue reports google-antigravity failures from a deprecated Antigravity user-agent version and stale runtime auth snapshots after OAuth credential refresh.

## Best Possible Solution

Keep this open until maintainers choose a provider-owned Antigravity version strategy and add a generic runtime auth snapshot reload or invalidation path so refreshed auth-profiles.json state can become active without restarting the gateway.

## Reproduction Assessment

Yes for the source-level reproduction: current main still routes google-antigravity through the pinned pi-ai Cloud Code Assist transport with the 1.21.9 Antigravity default, and the stale snapshot path is visible by activating a runtime auth snapshot before changing auth-profiles.json externally. A live end-to-end sandbox reproduction still needs maintainer OAuth credentials.

## Solution Assessment

No current-main solution is complete. The narrow maintainable fix is a Google-provider/dependency decision for the Antigravity user-agent plus a generic auth snapshot reload or invalidation mechanism, not a hidden provider-specific startup migration.
