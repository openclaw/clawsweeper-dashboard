## Summary

Keep open. Current main still has the source-level conditions described in the report: the bundled pi-ai dependency is pinned to 0.70.6 with the upstream Antigravity default user-agent version still at 1.21.9, google-antigravity remains an active Cloud Code Assist-style route, and runtime auth snapshots can still take precedence over externally refreshed auth-profiles.json state.

## What This Changes

The issue requests restoring google-antigravity by updating or configuring the Antigravity client version and clearing stale runtime credential snapshots after successful token refresh.

## Best Possible Solution

Keep this open until maintainers choose a provider-owned Antigravity version strategy and add a generic runtime auth snapshot refresh or invalidation path that lets refreshed auth-profiles.json state become active without a gateway restart.

## Reproduction Assessment

Yes for the local code path: current main still pins pi-ai 0.70.6 with the 1.21.9 Antigravity default, and an active runtime auth snapshot is returned before disk state. A full live reproduction of the sandbox rejection still needs maintainer OAuth credentials.

## Solution Assessment

No current-main solution is complete. The narrow maintainable direction is a Google-provider/dependency strategy for the Antigravity user-agent plus a generic auth snapshot reload or invalidation mechanism, not a hidden provider-specific startup migration.
