## Summary

Keep open. Current main appears to have addressed part of the auth snapshot refresh path, but `google-antigravity` still sits in a legacy/hook-only compatibility state while docs and tests continue to describe an Antigravity Cloud Code Assist bridge, so this needs a maintainer support-or-retire decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to restore `google-antigravity` by avoiding the deprecated Antigravity client version and by clearing stale in-memory credential snapshots after OAuth refresh.

## Best Possible Solution

Maintainers should choose one durable direction: restore a supported and tested Antigravity/Cloud Code Assist transport with auth-refresh recovery, or retire the remaining google-antigravity aliases, docs, live-test references, and compatibility examples.

## Reproduction Assessment

Yes for source-level reproduction: current main still exposes `google-antigravity` in hook aliases and live-test docs while lacking a clear current Antigravity stream/version implementation in the Google plugin. A live endpoint reproduction would require Google OAuth credentials.

## Solution Assessment

No. The issue’s requested `DEFAULT_ANTIGRAVITY_VERSION` bump targets an older pi-ai surface, while current main has a different Google plugin and auth shape; the maintainable solution is a maintainer decision to restore or retire the compatibility surface.
