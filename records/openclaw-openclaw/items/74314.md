## Summary

Keep this open. Current main still has conflicting google-antigravity state: compatibility/model paths remain visible, but there is no current Antigravity/Cloud Code Assist stream implementation in the Google plugin or the generic transport seam, and the auth runtime snapshot path can still prefer an in-memory store over fresher persisted credentials.

## What This Changes

The issue requests restoring google-antigravity by updating the deprecated Antigravity client behavior and clearing stale runtime auth snapshots after successful token refresh.

## Best Possible Solution

Maintainers should choose one explicit direction: either restore a supported Antigravity/Cloud Code Assist transport with current client-version behavior and a generic auth snapshot refresh/invalidation rule, or retire/redirect the remaining google-antigravity docs, tests, aliases, and model compatibility paths to google-gemini-cli or other supported providers.

## Reproduction Assessment

Yes for a source-level reproduction: current main still exposes google-antigravity compatibility surfaces while lacking a stream implementation, and runtime auth store resolution can prefer a stale in-memory snapshot. A full live reproduction would require Google OAuth credentials and the Antigravity sandbox.

## Solution Assessment

Unclear. The issue's DEFAULT_ANTIGRAVITY_VERSION bump is not the best standalone fix on current main because the old upstream Antigravity surface is no longer the only problem; the maintainable fix needs a provider support/retirement decision plus a generic auth snapshot freshness rule.
