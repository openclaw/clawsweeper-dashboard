## Summary

Keep open. Current main still has unresolved google-antigravity drift: compatibility/docs/test surfaces remain, the pinned pi-ai package no longer ships the old Antigravity transport, and the auth runtime can still prefer an active in-memory snapshot over refreshed on-disk credentials.

## What This Changes

The issue reports two google-antigravity failures: an obsolete Antigravity client version rejected by the sandbox endpoint and stale in-memory auth snapshots after token refresh.

## Best Possible Solution

Maintainers should either restore a supported Antigravity/Cloud Code Assist transport with current client-version handling and tested auth snapshot refresh semantics, or retire google-antigravity from aliases, docs, tests, and config examples.

## Reproduction Assessment

Yes for source-level reproduction: current main still exposes google-antigravity compatibility paths while lacking the old upstream transport, and runtime auth resolution can prefer an in-memory snapshot over refreshed disk state. A live OAuth smoke would still be needed to validate any restored support path.

## Solution Assessment

Unclear. The issue's version-bump workaround is no longer sufficient by itself because the pinned upstream package removed the old provider surface; the maintainable fix needs a support-or-retire decision plus a scoped auth snapshot reload rule if support remains.
