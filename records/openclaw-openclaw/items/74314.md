## Summary

Keep open. Current main still leaves a real product gap: `google-antigravity` remains documented and exercised as an Antigravity OAuth bridge, but the pinned upstream package no longer ships the old Antigravity/Gemini CLI stream provider, and OpenClaw only has partial compatibility/model-hook wiring plus unresolved auth snapshot recovery semantics.

## What This Changes

The issue asks OpenClaw to restore `google-antigravity` by moving off the deprecated Antigravity client version and by recovering from refreshed OAuth credentials without a gateway restart.

## Best Possible Solution

Maintainers should explicitly choose one direction: restore a supported Antigravity/Cloud Code Assist transport with current client-version behavior, auth/profile ownership, and snapshot invalidation semantics, or retire `google-antigravity` from docs, tests, config compatibility, and live-smoke guidance while steering users to `google` or `google-gemini-cli`.

## Reproduction Assessment

Yes for a source-level reproduction: current main still exposes and documents `google-antigravity` surfaces while the pinned upstream package no longer provides the old stream implementation, and runtime snapshots can still outrank externally refreshed disk credentials. A full live reproduction requires Google OAuth credentials that were not available in this environment.

## Solution Assessment

No. The issue's proposed `DEFAULT_ANTIGRAVITY_VERSION` bump is no longer sufficient because that upstream provider file is gone from the current pin; the maintainable fix is either an owned OpenClaw/plugin transport plus auth-cache rule or an intentional retirement/migration cleanup.
