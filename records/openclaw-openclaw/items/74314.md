## Summary

Keep open. Current main has partly hardened the generic auth-store snapshot path, but the old Antigravity transport/version bump targeted by the report is no longer present in the pinned pi-ai dependency, while OpenClaw still carries Antigravity alias/template compatibility that needs an explicit support-versus-retire decision.

## What This Changes

The issue reports that google-antigravity rejects the hardcoded Antigravity API version and keeps using stale in-memory credentials after token refresh.

## Best Possible Solution

Maintainers should choose one durable path: restore Antigravity through a supported plugin-owned Cloud Code Assist transport with auth-refresh recovery, or formally retire the remaining aliases/templates and point users to ClawHub/community plugins.

## Reproduction Assessment

Yes for source-level reproduction: current main no longer has the old Antigravity stream provider in the pinned pi-ai dependency, still carries google-antigravity alias/template compatibility, and still serves runtime snapshots before disk reads. Live endpoint reproduction would require valid Google Cloud Code Assist OAuth credentials.

## Solution Assessment

No. The requested DEFAULT_ANTIGRAVITY_VERSION bump targets @mariozechner/pi-ai 0.70.x, while current main and the latest release no longer ship that provider file; the maintainable fix is a support-or-retire decision plus any needed auth snapshot follow-through.
