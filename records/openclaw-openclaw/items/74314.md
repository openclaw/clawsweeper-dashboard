## Summary

Keep open. Current main still exposes google-antigravity compatibility paths while the active Google provider does not provide a google-gemini-cli/Antigravity stream, and the auth runtime snapshot path can still prefer an active in-memory snapshot over refreshed disk credentials.

## What This Changes

The issue reports that google-antigravity stopped working because the upstream Antigravity client version is deprecated and refreshed OAuth credentials do not replace a stale in-memory auth snapshot until gateway restart.

## Best Possible Solution

Maintainers should either restore an owned Antigravity/Cloud Code Assist transport with current client-version behavior and a tested runtime snapshot refresh rule, or finish the retirement by removing/redirecting aliases, docs, tests, and model examples.

## Reproduction Assessment

Yes for a source-level reproduction. Current main still documents and resolves google-antigravity compatibility refs, but the stream factories exclude the google-gemini-cli/Antigravity API path, and runtime auth snapshots can remain ahead of disk reloads.

## Solution Assessment

Unclear. The issue's version-bump workaround is no longer a complete fix for current main; the maintainable path needs an explicit support-or-retire decision plus a generic auth snapshot invalidation/reload rule if support remains.
