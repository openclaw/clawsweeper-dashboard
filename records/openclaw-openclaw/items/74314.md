## Summary

Keep open. Current main has partially improved the runtime auth-refresh/cache path, but the central google-antigravity transport problem is still unresolved: OpenClaw still exposes antigravity model/config/test/docs surfaces while the pinned pi-ai package no longer ships the old `google-gemini-cli` Cloud Code Assist provider and OpenClaw has no replacement stream for that API.

## What This Changes

The issue reports google-antigravity failures from a deprecated Antigravity User-Agent/version and stale runtime credential snapshots after OAuth token refresh.

## Best Possible Solution

Maintainers should either restore a current Google plugin-owned Antigravity/Cloud Code Assist transport with OAuth refresh coverage, or deliberately retire google-antigravity by removing/redirecting aliases, examples, docs, and tests.

## Reproduction Assessment

Yes for source-level reproduction: current tests and provider metadata still route google-antigravity models to `api: "google-gemini-cli"`, but current pi-ai 0.71.1 no longer ships that provider and OpenClaw does not implement a replacement stream. Live reproduction of the exact sandbox response still needs credentials.

## Solution Assessment

No, the issue's version-bump workaround is not sufficient on current main because the upstream provider containing the hardcoded Antigravity version has been removed; the maintainable path needs a support-or-retire decision first.
