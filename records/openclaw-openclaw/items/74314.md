## Summary

Keep open. Current main has moved from the originally reported pi-ai 0.70.6 state, but the google-antigravity path is still not demonstrably fixed: the bundled Google plugin still exposes google-antigravity as a hook alias without a google-gemini-cli stream implementation, the generic transport path still excludes google-gemini-cli, and the auth-store runtime snapshot path can still return an active in-memory snapshot before reloading persisted auth state.

## What This Changes

The issue requests fixing google-antigravity after Antigravity rejected the old client version and making refreshed OAuth credentials recover without a gateway restart.

## Best Possible Solution

Keep this open until maintainers choose the Google/Antigravity ownership strategy: either restore or own a supported Cloud Code Assist/Antigravity transport with an accepted client version, or intentionally retire/redirect google-antigravity refs, and add a generic auth snapshot refresh or invalidation path for externally refreshed auth-profiles.json credentials.

## Reproduction Assessment

Yes for a source-level reproduction: current main still leaves google-antigravity on a hook-alias/google-gemini-cli path without a current OpenClaw stream implementation, and the auth store still prefers active runtime snapshots before disk reload. A live end-to-end reproduction still needs maintainer Google OAuth credentials.

## Solution Assessment

No current-main solution is complete. The narrow maintainable fix needs maintainer direction because it crosses an upstream provider transport decision and auth-sensitive runtime credential cache behavior.
