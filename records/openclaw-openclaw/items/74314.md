## Summary

Keep open. Current main does not provide a complete working Google Antigravity path: the pinned Pi package no longer registers the old `google-gemini-cli` API transport, while OpenClaw still has Antigravity compatibility hooks and docs/tests around that provider, and the runtime auth snapshot path can still return an in-memory store before reloading refreshed disk credentials.

## What This Changes

The issue requests restoring the `google-antigravity` provider by updating or configuring the Antigravity client version and invalidating stale runtime auth snapshots after successful OAuth refresh.

## Best Possible Solution

Keep this open until maintainers choose the product direction for Antigravity. If it remains supported, restore a provider-owned transport/version strategy and add a generic runtime auth snapshot invalidation or reload path so refreshed credentials take effect without a gateway restart; if it is no longer supported, update docs, diagnostics, and migration guidance to remove the live provider surface cleanly.

## Reproduction Assessment

Yes for a source-level reproduction: current main still exposes Antigravity hooks/docs while the pinned Pi 0.71.1 package no longer registers the `google-gemini-cli` API, and runtime auth snapshots still precede disk reload. A live end-to-end reproduction remains credentials-dependent.

## Solution Assessment

No current-main solution is complete. The safest fix is not a hidden startup migration; it needs a provider-owned Antigravity decision plus a generic auth snapshot refresh mechanism that preserves the existing credential-store contracts.
