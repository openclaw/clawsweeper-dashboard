## Summary

Keep open. Current main has not resolved the reported provider breakage: the deprecated upstream Antigravity transport is gone from the pinned pi-ai package, OpenClaw still exposes google-antigravity compatibility paths without a google-gemini-cli stream implementation, and runtime auth snapshots can still win over fresher persisted credentials.

## What This Changes

The issue asks OpenClaw to restore google-antigravity by using a supported Antigravity client version and invalidating stale runtime auth snapshots after OAuth token refresh.

## Best Possible Solution

Maintainers should choose the product direction first: either own or restore a supported Antigravity/Cloud Code Assist transport with current client-version behavior and auth snapshot invalidation, or intentionally retire/redirect google-antigravity surfaces, tests, and docs if that transport is no longer supportable.

## Reproduction Assessment

Yes for a source-level reproduction: current main still exposes google-antigravity compatibility paths while lacking a current google-gemini-cli/Antigravity stream implementation, and the auth store can return an active runtime snapshot before reading refreshed disk state. A full end-to-end reproduction still requires maintainer Google OAuth credentials.

## Solution Assessment

Unclear. The issue's original version-bump suggestion is no longer sufficient by itself because current pi-ai removed the old Antigravity provider surface; the maintainable fix needs an explicit OpenClaw ownership or retirement decision plus a generic auth snapshot refresh/invalidation rule.
