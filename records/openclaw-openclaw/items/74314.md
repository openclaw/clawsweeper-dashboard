## Summary

Keep open. Current main still has unresolved google-antigravity product/implementation drift: the old upstream Antigravity transport is gone from the pinned pi-ai package, OpenClaw still has compatibility/docs/test surfaces for google-antigravity, and the auth snapshot path can still prefer an active runtime snapshot over externally refreshed disk credentials.

## What This Changes

Requests fixing google-antigravity after upstream rejects the old Antigravity client version and stale runtime auth snapshots keep credentials failed until gateway restart.

## Best Possible Solution

Maintainers should choose one direction: restore an owned Antigravity/Cloud Code Assist transport with current client-version behavior and a scoped auth snapshot refresh rule, or intentionally retire google-antigravity by removing/redirecting aliases, docs, tests, and model examples.

## Reproduction Assessment

Yes for a source-level reproduction: current main can still resolve/expose google-antigravity compatibility surfaces while lacking the upstream or OpenClaw google-gemini-cli stream implementation needed to serve them. The runtime auth store also prefers active in-memory snapshots before disk, which matches the reported stale external-refresh behavior.

## Solution Assessment

Unclear. The issue's version-bump workaround is no longer sufficient because the pinned upstream package removed the old Antigravity provider surface; the maintainable fix needs an explicit support-or-retire decision plus a generic, tested auth snapshot invalidation/reload rule if support remains.
