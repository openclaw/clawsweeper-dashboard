## Summary

Keep open. The auth-cache half appears hardened on current main, but the remaining google-antigravity support boundary is unresolved: OpenClaw still advertises Antigravity live coverage while the pinned pi-ai dependency no longer ships the old Antigravity/Gemini CLI transport that the report’s version bump targeted.

## What This Changes

The issue reports that google-antigravity fails because the old Antigravity API version is rejected and because stale runtime auth snapshots survive OAuth token refreshes.

## Best Possible Solution

Maintainers should choose one durable direction: restore a tested Antigravity/Cloud Code Assist transport with auth-refresh recovery, or intentionally retire the google-antigravity surface across docs, tests, examples, and model routing.

## Reproduction Assessment

Yes for source-level reproduction of the remaining problem: current main still documents and routes google-antigravity as a support surface, while pi-ai 0.71.1 no longer contains the old Antigravity/Gemini CLI provider implementation. I did not run a live OAuth smoke against Google endpoints.

## Solution Assessment

No. The requested DEFAULT_ANTIGRAVITY_VERSION bump targets pi-ai 0.70.x code that is gone from current main, so the narrow maintainable fix requires a support-or-retire decision rather than a mechanical constant update.
