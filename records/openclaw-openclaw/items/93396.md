## Summary

Keep open. Current main and v2026.6.9 still leave the reported memory-embedding provider routing unresolved, and multiple open PRs are candidate fixes rather than merged implementation.

## What This Changes

The issue reports that `models.providers.openai` with a local OpenAI-compatible Ollama `baseURL` can still route memory embeddings to the official OpenAI API instead of the configured endpoint.

## Best Possible Solution

Review and converge the open fix PRs into one coordinated provider-routing fix that covers memory-core and gateway embedding paths, keeps the config contract explicit, adds regression coverage, and proves a redacted local endpoint call.

## Reproduction Assessment

Yes, source-reproducible: current main returns the direct `openai` memory adapter before configured provider hints and ignores `baseURL` in the resolver/client paths. I did not run a live local Ollama endpoint.

## Solution Assessment

Unclear until maintainers choose the config contract. The best fix is not a one-off alias patch; it should normalize provider endpoint routing across resolver, direct OpenAI memory, OpenAI-compatible client, gateway, tests, and proof.
