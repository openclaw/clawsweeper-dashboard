## Summary

Keep open. Current main has supported canonical local-embedding paths, but source inspection still leaves the reported noncanonical `baseURL` plus `openai` fallback unresolved and it needs a maintainer config-contract/security decision.

## What This Changes

The issue asks OpenClaw to keep local Ollama/OpenAI-compatible embedding calls on the configured custom endpoint instead of falling back to the official OpenAI API when the provider id is `openai`.

## Best Possible Solution

Keep one canonical local-embedding path and add validation, doctor, or diagnostics so noncanonical `baseURL` cannot silently reach the official OpenAI fallback unless maintainers deliberately accept a scoped alias.

## Reproduction Assessment

Unclear for a live current-main repro. Source inspection shows the fallback if the raw `baseURL` shape reaches the memory client, but I did not run a live Ollama gateway and strict schema validation may reject it earlier.

## Solution Assessment

Unclear. Broadly accepting `baseURL` conflicts with the canonical-config policy, so maintainers should choose a scoped alias, doctor/validation, or documentation path before repair.
