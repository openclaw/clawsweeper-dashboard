## Summary

Keep open. Current main and the latest release have native Ollama and explicit `openai-compatible` memory embedding paths, but the reported `models.providers.openai.baseURL` shape can still miss the configured local endpoint and fall back through the OpenAI memory adapter/default endpoint path.

## What This Changes

The issue reports that an OpenAI-compatible local Ollama config under the `openai` provider key can cause memory embedding calls to use the official OpenAI API instead of the configured local endpoint.

## Best Possible Solution

Maintainers should choose the config contract, then either validate or migrate the misleading `openai` plus `baseURL` shape, or guide users to native `ollama` or explicit `openai-compatible` memory config without silently routing local embeddings to api.openai.com.

## Reproduction Assessment

Yes for a source-level reproduction: current code resolves `memorySearch.provider: "openai"` to the OpenAI memory adapter and the remote client ignores `models.providers.openai.baseURL`. A live local Ollama run is still missing.

## Solution Assessment

Unclear until maintainers choose the contract. The safest existing solution is native `ollama` or explicit `openai-compatible`/`memorySearch.remote.baseUrl`, while broad `baseURL` aliasing under `openai` needs security and upgrade guardrails.
