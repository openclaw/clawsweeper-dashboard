## Summary

Keep open. Current main and the latest release still ignore `models.providers.openai.baseURL` in the generic memory embedding client while the OpenAI embedding provider falls back to `https://api.openai.com/v1`; no canonical open duplicate now owns the exact remaining config-contract/security decision.

## What This Changes

The issue reports that a local Ollama-compatible endpoint configured as `models.providers.openai.baseURL` is ignored for memory embeddings, which can make OpenClaw demand or use official OpenAI credentials.

## Best Possible Solution

Keep memory embedding endpoint resolution explicit: native Ollama or `openai-compatible` memory config should use canonical `baseUrl`, and noncanonical `baseURL` should get a maintainer-approved migration, validation warning, or tightly scoped alias.

## Reproduction Assessment

Yes by source inspection for the reported raw shape: `baseURL` is ignored by the generic memory remote client while `openai` defaults to `https://api.openai.com/v1`. A live Ollama gateway reproduction is still missing, and strict config parsing may reject some raw configs earlier.

## Solution Assessment

Unclear. A broad alias is not clearly best because OpenClaw policy prefers canonical config plus doctor migration; the maintainable fix should be a maintainer-approved validation, doctor, docs, or narrowly scoped compatibility path.
