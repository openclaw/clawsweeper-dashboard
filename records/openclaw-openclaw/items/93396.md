## Summary

Keep open. Current main and the latest release still have a source-reproducible path where the reported uppercase `baseURL` is ignored for memory embeddings, and open fix PRs already target this issue but have not merged.

## What This Changes

The issue reports that a local Ollama/OpenAI-compatible endpoint configured under `models.providers.openai.baseURL` can cause memory embeddings to use the native OpenAI default path instead of the local endpoint.

## Best Possible Solution

Keep this issue as the canonical tracker until one coordinated PR lands endpoint/key resolution with regression tests and redacted local-endpoint proof, then close through the merged PR.

## Reproduction Assessment

Yes, source-reproducible: current source reads `baseUrl` but not `baseURL` in the memory embedding endpoint paths, and the `openai` direct adapter still wins before generic routing. I did not run a live local Ollama endpoint.

## Solution Assessment

Unclear until maintainers choose the config contract. The safest fix is a coordinated route through provider selection, endpoint normalization, auth/SSRF handling, docs or doctor guidance, and regression coverage rather than another overlapping one-sided patch.
