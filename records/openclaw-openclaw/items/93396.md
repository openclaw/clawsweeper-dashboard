## Summary

Keep open: current main and v2026.6.9 still do not fully honor the reported local/OpenAI-compatible memory-embedding route, and the linked fixes are open candidate PRs rather than merged implementation.

## What This Changes

The issue reports that using `models.providers.openai.baseURL` for a local Ollama/OpenAI-compatible endpoint can make memory embeddings fall back toward the native OpenAI API instead of the configured endpoint.

## Best Possible Solution

Land one coordinated memory-embedding endpoint-normalization fix that covers direct OpenAI, OpenAI-compatible, and provider-owned Ollama paths with regression tests, docs/doctor direction for the config spelling, and redacted real endpoint proof.

## Reproduction Assessment

Yes, source-reproducible: current main still prefers the direct `openai` memory adapter and the relevant resolver/client paths read canonical `baseUrl` but not the reported `baseURL`. I did not run a live local Ollama endpoint.

## Solution Assessment

No current implementation is ready to close this. The best fix needs a coordinated endpoint/config contract across memory provider selection, OpenAI-compatible client creation, direct OpenAI remote client behavior, gateway embeddings, docs or doctor behavior, tests, and real endpoint proof.
