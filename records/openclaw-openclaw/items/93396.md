## Summary

Keep open. Current `main` still matches the reported failure path for the `models.providers.openai.baseURL` memory-embedding case, and the candidate fixes are still open rather than merged.

## What This Changes

The issue reports that `models.providers.openai` with a local Ollama OpenAI-compatible `baseURL` can route memory embedding calls to the official OpenAI endpoint instead of the configured local endpoint.

## Best Possible Solution

Land one coordinated memory-embedding provider-routing fix that chooses the config contract explicitly, covers direct OpenAI and generic OpenAI-compatible runtime paths together, and adds focused regression coverage plus endpoint/credential-routing proof.

## Reproduction Assessment

Yes, source-reproducible: current `main` still returns the direct `openai` memory adapter first and the relevant endpoint-resolution paths read `baseUrl` but not the reported `baseURL` spelling. I did not run a live local Ollama endpoint in this read-only review.

## Solution Assessment

Unclear until maintainers choose the config contract. The narrow maintainable fix should route the direct `openai` memory path through configured endpoint/provider resolution, but accepting uppercase `baseURL` at runtime needs deliberate product/security approval because repository policy favors canonical runtime config.
