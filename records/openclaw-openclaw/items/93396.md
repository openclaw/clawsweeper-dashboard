## Summary

Keep open. Current main and v2026.6.6 support native Ollama and explicit OpenAI-compatible memory embedding paths, but the reported `models.providers.openai.baseURL` shape can still route memory embeddings through the OpenAI adapter/default endpoint rather than the configured local service.

## What This Changes

The issue asks OpenClaw to avoid using the official OpenAI embedding endpoint when a locally hosted OpenAI-compatible/Ollama base URL is configured under the `openai` provider key.

## Best Possible Solution

Maintainers should choose the config contract, then either migrate/validate `baseURL` and `openai` custom-base-url memory routing or document and doctor users onto native `ollama` or explicit `openai-compatible` memory config.

## Reproduction Assessment

Source-reproducible but not live-reproduced: current code shows `memorySearch.provider=openai` resolves the OpenAI memory adapter and ignores `models.providers.openai.baseURL`; no live Ollama run was performed.

## Solution Assessment

Unclear until maintainers choose the contract; the safest existing path is native `ollama` or explicit `openai-compatible`/`memorySearch.remote.baseUrl`, while broad `baseURL` or `openai` aliasing needs guardrails.
