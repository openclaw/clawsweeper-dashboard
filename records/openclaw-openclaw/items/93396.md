## Label Justifications

- `P2`: This is a normal-priority provider/config routing bug with limited blast radius to local Ollama or OpenAI-compatible memory embedding setups.
- `impact:auth-provider`: The report is about provider identity, endpoint selection, and OpenAI credential demand when local Ollama is intended.
- `impact:security`: If a real OpenAI credential is available, the fallback path can route memory embedding content to an unintended external service.

## Summary

Keep open. Current main and v2026.6.6 support local Ollama memory embeddings through native Ollama or explicit OpenAI-compatible memory config, but the reported `models.providers.openai.baseURL` shape can still miss the canonical `baseUrl` path and fall back toward the official OpenAI embedding default.

## What This Changes

The issue asks OpenClaw to keep local Ollama/OpenAI-compatible embedding calls on the configured endpoint instead of falling back to official OpenAI when the provider key is `openai`.

## Best Possible Solution

Keep one canonical local-embedding path: use native Ollama or explicit `openai-compatible` memory config, and add validation, doctor, or diagnostics so noncanonical `openai.baseURL` cannot silently reach the official OpenAI fallback.

## Reproduction Assessment

Yes by source inspection, with a caveat: the memory embedding client ignores `baseURL` and falls back to the OpenAI default when `baseUrl` is absent, but I did not run a live Ollama gateway and strict config parsing may intercept some raw configs earlier.

## Solution Assessment

Unclear. Treating `openai` plus `baseURL` as a local Ollama setup is not the best permanent shape; maintainers should choose between targeted diagnostics/migration and a deliberately scoped compatibility alias.
