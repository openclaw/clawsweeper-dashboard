## Summary

Keep open. Current `main` and the latest inspected release still do not honor the reported `models.providers.openai.baseURL` memory-embedding path, and the related fix PRs are still open or closed unmerged rather than a merged implementation.

## What This Changes

The issue reports that an OpenAI-compatible local Ollama endpoint configured under `models.providers.openai.baseURL` can make memory embeddings fall back to the real OpenAI endpoint.

## Best Possible Solution

Land one coordinated endpoint-normalization path for memory embeddings: decide the `baseURL` contract, update direct OpenAI and generic OpenAI-compatible runtime/client resolution together, add regression coverage, and prove a redacted local endpoint call.

## Reproduction Assessment

Yes, source-reproducible: current `main` still returns the direct `openai` memory adapter first and does not honor `models.providers.openai.baseURL` in the relevant endpoint resolution paths. I did not run a live local Ollama endpoint.

## Solution Assessment

No. The best fix needs one coordinated endpoint-normalization path through memory provider selection, OpenAI-compatible client creation, direct OpenAI remote client behavior, docs/schema or doctor direction, regression tests, and live proof.
