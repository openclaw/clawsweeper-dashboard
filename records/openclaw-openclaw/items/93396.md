## Summary

Keep open. Current main and the latest release still only honor canonical `baseUrl` in the relevant memory-embedding paths, while this issue is already paired with multiple open candidate PRs that need maintainer provider/security review rather than cleanup closure.

## What This Changes

Use the configured local OpenAI-compatible/Ollama endpoint for memory embeddings instead of falling back to official OpenAI when the provider is named `openai` and configured with `baseURL`.

## Best Possible Solution

Maintainers should choose or consolidate one fix that preserves canonical `baseUrl` and endpoint/credential guardrails, then close this issue after the selected PR lands.

## Reproduction Assessment

Yes, source-reproducible: current main resolves `memorySearch.provider=openai` through the direct OpenAI memory adapter and the relevant paths read `baseUrl`, not the reported `baseURL`. A live Ollama embedding roundtrip is still missing.

## Solution Assessment

Unclear until maintainers choose the config contract. The safer fix should combine endpoint-routing repair with canonical config or doctor behavior rather than silently adding a broad runtime alias.
