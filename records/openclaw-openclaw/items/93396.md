## Summary

Keep open. Current `main` and v2026.6.9 still have the reported memory-embedding routing gap, and the linked fix PRs are open candidates rather than merged implementations.

## What This Changes

The issue asks OpenClaw to keep memory embeddings on the configured local OpenAI-compatible endpoint instead of falling back to the official OpenAI API when the provider key is `openai`.

## Best Possible Solution

Land one coordinated provider-routing and endpoint-normalization fix that covers direct OpenAI memory, generic OpenAI-compatible embeddings, gateway `/v1/embeddings`, docs/schema or doctor direction, regression tests, and redacted local endpoint proof.

## Reproduction Assessment

Yes, source-reproducible: current `main` still returns the direct `openai` memory adapter first and the relevant endpoint clients do not honor `models.providers.openai.baseURL`. I did not run a live local endpoint.

## Solution Assessment

Unclear as stated: the bug is valid, but the best fix needs maintainer agreement on the `baseUrl`/`baseURL` contract and one shared routing path rather than a partial alias-only change.
