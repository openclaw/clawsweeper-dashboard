## Label Justifications

- `P2`: This is a normal provider/config bug with limited blast radius: local embeddings can route to the wrong provider endpoint but the core embedding capability exists.
- `impact:auth-provider`: The report is about provider routing and OpenAI API-key resolution when a local OpenAI-compatible endpoint is configured.
- `impact:session-state`: The affected operation is memory embeddings/search, which can block or misroute memory-related agent state behavior.

## Summary

The issue should stay open because current main has the right canonical embedding paths for local OpenAI-compatible endpoints, but the reported config uses `baseURL`, which current schema/docs do not read and which can source-reproduce the fallback to the official OpenAI base URL for memory embeddings. A narrow doctor/config repair is safer than adding a runtime alias or overloading the `openai` provider key further.

## What This Changes

The issue requests that local Ollama/OpenAI-compatible configuration under the `openai` provider key not fall back to the official OpenAI API for memory embeddings.

## Best Possible Solution

Add a doctor/config diagnostic that rewrites or warns on `models.providers.*.baseURL` to canonical `baseUrl`, and clarify the documented local embedding paths for Ollama/OpenAI-compatible memory search.

## Reproduction Assessment

Yes, source inspection gives a medium-confidence reproduction path: `baseURL` is ignored, memory search can default to `openai`, and the embedding client falls back to the official OpenAI base URL without canonical `baseUrl` or `memorySearch.remote.baseUrl`. I did not establish a live Ollama failing run.

## Solution Assessment

No, overloading `provider: "openai"` for local Ollama embeddings is not the best long-term fix. The maintainable fix is to keep runtime on canonical `baseUrl`, add doctor/config repair for the mis-cased key, and steer users to explicit `openai-compatible` or `ollama` memory provider configuration.
