## Summary

Keep open. Current main and latest release still show the `openai` memory embedding path can bypass the reported custom endpoint shape, and two open PRs use closing syntax for this issue, so the right cleanup is to review and land or reject a coordinated fix rather than close the issue now.

## What This Changes

The issue asks OpenClaw to route memory embeddings for a custom OpenAI-compatible/Ollama endpoint configured under `models.providers.openai` to the configured local endpoint instead of the official OpenAI API.

## Best Possible Solution

Land one coordinated provider-endpoint fix, or explicitly document/doctor users onto the canonical `baseUrl`/native Ollama path, with direct OpenAI, generic OpenAI-compatible, memory-core, gateway embeddings, tests, and live endpoint proof aligned.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: current main returns the direct `openai` memory adapter before configured provider routing and only reads canonical `baseUrl` in the relevant endpoint paths. I did not run a live Ollama endpoint.

## Solution Assessment

Unclear until maintainers choose the config contract. The safest fix is one coordinated normalization path or a documented doctor migration to canonical `baseUrl`, not isolated runtime aliases in only some embedding paths.
