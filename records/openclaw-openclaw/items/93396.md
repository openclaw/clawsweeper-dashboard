## Summary

Keep open. Current main and the latest release still leave the reported memory-embedding provider path unresolved, and two open PRs already use closing syntax for this issue, so the issue remains necessary until maintainers pick and land a canonical repair.

## What This Changes

The issue asks OpenClaw to route memory embeddings through the configured local OpenAI-compatible/Ollama endpoint instead of falling back to the official OpenAI API when the provider key is `openai`.

## Best Possible Solution

Land one maintainer-approved fix that keeps canonical config ownership clear while ensuring memory embeddings never ignore a configured local provider endpoint and fall through to the official OpenAI API.

## Reproduction Assessment

Yes, there is a source-level reproduction path: configure memory embeddings through provider `openai` with only `models.providers.openai.baseURL`, then the current remote client ignores that key and can use the default OpenAI base URL. I did not establish a live Ollama endpoint repro in this read-only pass.

## Solution Assessment

Unclear pending maintainer choice. The fix should address the direct `openai` adapter short-circuit and base-URL resolution without creating an unowned long-lived malformed-config alias.
