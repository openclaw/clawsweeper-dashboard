## Summary

Keep open. Current main and v2026.6.9 still have the reported memory-embedding endpoint-routing gap, and open linked PRs are candidate fixes rather than merged implementation.

## What This Changes

The issue reports that `models.providers.openai.baseURL` for a local Ollama/OpenAI-compatible endpoint can make memory embeddings fall back to the official OpenAI endpoint instead of the configured local endpoint.

## Best Possible Solution

Land one coordinated fix that resolves the `baseUrl`/`baseURL` contract across memory provider selection and embedding client creation, with regression tests and redacted local endpoint proof.

## Reproduction Assessment

Yes, at source level: current main still returns the direct `openai` memory adapter first and the relevant resolver/client paths read `baseUrl` but not `baseURL`. I did not run a live local Ollama endpoint.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The best implementation should be one coordinated routing and endpoint-normalization fix rather than several overlapping one-sided PRs.
