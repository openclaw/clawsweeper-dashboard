## Summary

Keep this PR open: the awaited `agent_end` direction is useful, but the patch currently changes channel-backed CLI runs too, so slow observation hooks can delay reply delivery even though the PR says persistent channel-backed paths should remain fire-and-forget.

## What This Changes

The PR adds an awaited `agent_end` helper, uses it in local CLI and no-channel Codex app-server terminal paths, preserves fire-and-forget Codex channel paths, and updates docs/tests/SDK exports plus a static unused-file allowlist.

## Best Possible Solution

Keep the awaited drain for true one-shot local runs, but mirror the Codex split in the CLI runner so channel-backed CLI attempts continue to fire `agent_end` without blocking reply delivery.

## Reproduction Assessment

Yes from source inspection: current main drops the `agent_end` promise in the shared helper, and local CLI plus no-channel Codex app-server paths call that helper immediately before cleanup/return. The PR regression is also source-reproducible because channel-backed CLI calls pass channel metadata into `runCliAgent`, but the PR awaits all CLI terminal hooks unconditionally.

## Solution Assessment

No, not as currently patched: the awaited helper is a good fit for true one-shot drain points, but the CLI runner needs the same local-vs-channel split that the PR already applies to Codex app-server attempts.
