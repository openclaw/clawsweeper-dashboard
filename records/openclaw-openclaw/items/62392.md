## Summary

Keep open: the missing-MIME image attachment bug is still present on current main and this PR contains useful resolver/test work, but the branch is not mergeable as-is because it bypasses the newer attachment payload-store flow and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds extension-based MIME fallback for missing-type Control UI chat attachments, updates send serialization, adds focused tests, and adds a changelog entry.

## Best Possible Solution

Rebase the missing-MIME image fallback onto current main's payload-store flow, preserve focused regression tests, and add real Control UI proof that an empty-MIME image is attached and sent.

## Reproduction Assessment

Yes at source level: current main stores empty-type selected/dropped files as application/octet-stream and sends attachments based on the data URL MIME. I did not run a live browser drag/drop session in this read-only review.

## Solution Assessment

Yes in direction, but not as currently rebased: resolving MIME from common image extensions only when browser MIME is missing is narrow and maintainable, but the implementation must preserve registerChatAttachmentPayload and getChatAttachmentDataUrl/getChatAttachmentPreviewUrl.
