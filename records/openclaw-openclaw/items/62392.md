## Summary

Keep open: current main still has the empty-MIME image attachment gap, and this PR remains a useful fix path, but the branch predates current main's side payload-store contract and still lacks real after-fix behavior proof.

## What This Changes

The PR adds extension-based MIME resolution for empty-type Control UI chat images, updates attachment send serialization, and adds focused UI/controller tests plus a changelog entry.

## Best Possible Solution

Rebase the missing-MIME image fallback onto current main's attachment-payload-store flow, keep focused regression tests, and add real Control UI proof that an empty-MIME image is attached and sent.

## Reproduction Assessment

Yes at source level: current main accepts empty-type selected/dropped files, records them as `application/octet-stream`, and serializes from the data URL MIME rather than a filename-derived image MIME. I did not run a live browser drag/drop reproduction in this read-only review.

## Solution Assessment

Yes in direction, but not as currently rebased: resolving MIME from common image extensions only when the browser MIME is missing is narrow and maintainable, but the implementation must use `registerChatAttachmentPayload`, `getChatAttachmentDataUrl`, and `getChatAttachmentPreviewUrl` on current main.
