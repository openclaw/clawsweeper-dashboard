## Summary

Keep open: the reported Control UI attachment bug still exists on current main, and this PR is a plausible narrow fix, but the branch is stale against main's side payload-store contract and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds an extension-based MIME resolver for selected/dropped Control UI chat attachments, carries the resolved MIME through send serialization, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rebase the missing-MIME image fallback onto current main's side payload-store flow, keep the focused regression tests, and add real Control UI proof that an empty-MIME image is attached and sent.

## Reproduction Assessment

Yes at source level: current main accepts an empty-type file into the attachment flow but records it as `application/octet-stream`, and send serialization does not recover `image/png` from metadata or filename. I did not run a live browser drag/drop reproduction in this read-only review.

## Solution Assessment

Yes with adaptation: resolving MIME from common image extensions only when the browser MIME is missing is a narrow maintainable fix. The implementation should be rebased through `registerChatAttachmentPayload`, `getChatAttachmentPreviewUrl`, and `getChatAttachmentDataUrl` rather than restoring direct data URLs in chat state.
