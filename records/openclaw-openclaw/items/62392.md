## What This Changes

The PR adds a Control UI chat attachment MIME resolver for empty-MIME image files, uses the resolved MIME during selected/dropped attachment handling and send serialization, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rebase the missing-MIME image fallback onto current main while preserving `registerChatAttachmentPayload`, `getChatAttachmentPreviewUrl`, send serialization, focused regression tests, and contributor credit.

## Reproduction Assessment

Yes at source level: current main has no MIME resolver, stores empty `File.type` attachments as `application/octet-stream`, and send serialization does not recover `image/png` from attachment metadata or filename. I did not run a live browser drag/drop reproduction in this read-only review.

## Solution Assessment

Yes with adaptation: the resolver-and-serialization approach is narrow and maintainable, but it must be rebased onto current main's side payload-store contract instead of restoring direct data URLs in chat state.
