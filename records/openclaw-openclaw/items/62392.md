## Summary

Keep open: this is a plausible focused Control UI bug fix, but the current PR branch is not safe to merge because it predates current main's attachment payload-store contract and still lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds extension-based MIME fallback for empty-type Control UI chat attachments, adjusts send serialization, adds focused UI/controller tests, and updates the changelog.

## Best Possible Solution

Rebase the missing-MIME fallback onto current main's payload-store helpers, keep focused regression coverage for empty-type image files, and add real Control UI proof that an empty-MIME image attaches and sends.

## Reproduction Assessment

Yes, at source level: current main accepts empty-type selected/dropped non-video files but stores them as `application/octet-stream`, while send serialization depends on the data URL MIME. I did not run a live browser drag/drop session in this read-only review.

## Solution Assessment

Yes in direction, but no as currently rebased: extension fallback only for missing MIME types is narrow, but the implementation must preserve `registerChatAttachmentPayload` and `getChatAttachmentDataUrl`/`getChatAttachmentPreviewUrl`.
