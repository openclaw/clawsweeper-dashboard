## Summary

Keep this PR open, but it is not merge-ready: the watchdog condition now treats the real `runner_entered` start callback as progress, which can disable the 60s pre-execution guard before the runner has advanced.

## What This Changes

The PR changes the cron isolated-agent watchdog to clear the pre-execution timeout for any known phase and updates timer regression coverage for setup and fallback phase progress.

## Best Possible Solution

Preserve the cron watchdog fix, but do not count `runner_entered` alone as progress; update the regression test to match the real `onExecutionStarted({ phase: "runner_entered" })` callback shape.

## Reproduction Assessment

Yes. Source inspection shows the real isolated cron path sends `phase: "runner_entered"` through `onExecutionStarted`, and the PR predicate would clear the guard for that phase before later progress.

## Solution Assessment

No. The PR is a plausible watchdog-layer fix, but as written it is too broad; the safer repair is to exclude `runner_entered` alone or use the targeted `before_agent_reply` emission path with proof.
