## Summary

Keep open: this PR targets a real isolated cron watchdog bug, but the widened predicate also treats `runner_entered` as progress, which can bypass the 60-second pre-execution guard for runs that start and then emit no further phase. The supplied proof is focused fake-timer Vitest output, not real scheduled-cron runtime proof.

## What This Changes

The PR changes cron's isolated-agent watchdog to clear the pre-execution timer for any known phase and updates timer regression expectations for setup and fallback phase progress.

## Best Possible Solution

Preserve the watchdog-layer fix, but keep `runner_entered`-only starts under the pre-execution guard; later setup phases can count as progress once regression coverage and real cron proof cover that contract.

## Reproduction Assessment

Yes. Source inspection shows the real isolated cron path sends `phase: "runner_entered"` through `onExecutionStarted`, and the PR predicate would clear the guard for that phase before later progress.

## Solution Assessment

No. The watchdog-layer fix is plausible, but as written it is too broad; the safer repair is to exclude `runner_entered` alone while allowing later setup phases to count as progress.
