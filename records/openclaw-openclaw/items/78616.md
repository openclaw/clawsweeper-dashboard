## Summary

Close as duplicate: the same Discord default-account startup-order bug is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/77429, including the same live symptoms, local patch shape, and validation context.

## What This Changes

Discord should start the configured default/primary account before specialist accounts while preserving the existing multi-account stagger.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/77429 as the canonical tracker and fix the Discord plugin there by ordering startup-delay candidates with the resolved default account first, without hard-coded account aliases.

## Reproduction Assessment

Yes, by source inspection: sorted account ids feed directly into resolveDiscordStartupDelayMs, so accounts like billy, farber, lumbergh, main produce a delayed main account unless startup ordering promotes the configured default. I did not run a live Discord restart in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the canonical issue: use the existing Discord default-account resolver inside the startup-delay ordering, preserve the current filtering and stagger, and add a regression test.
