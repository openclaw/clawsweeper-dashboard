## Summary

Keep open: current main still computes Discord startup delay from the sorted enabled account list, so a configured default/front-desk account can be delayed behind alphabetically earlier specialist accounts. The closed duplicate-token issue covers a separate same-token runtime hazard and does not solve distinct-token startup priority.

## What This Changes

The issue requests that Discord multi-account startup place the configured default/front-desk account first while preserving the existing stagger for remaining accounts.

## Best Possible Solution

Honor the resolved Discord default account first in the startup candidate order, keep the 10 second stagger for the remaining enabled/configured accounts, and add regression coverage where defaultAccount is not alphabetically first.

## Reproduction Assessment

Yes: source inspection gives a high-confidence unit-level path by configuring enabled Discord accounts such as billy, farber, and main with channels.discord.defaultAccount set to main; listDiscordAccountIds sorts them alphabetically, so main receives a nonzero delay. I did not run a live gateway restart in this read-only review.

## Solution Assessment

Yes: using the resolved configured default account as the first startup candidate is the narrowest maintainable fix. It avoids hard-coded operator account names and does not require a new config option.
