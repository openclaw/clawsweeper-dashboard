## Summary

Keep open: current main still computes Discord startup delay from the sorted enabled account list, so a configured default account such as `main` can be staggered behind alphabetically earlier specialist accounts. The duplicate-token work from https://github.com/openclaw/openclaw/issues/78118 is present but covers a different runtime hazard.

## What This Changes

Discord multi-account startup should give the configured default account the zero-delay slot before applying the existing stagger to specialist accounts.

## Best Possible Solution

Honor the resolved Discord default account first in the startup candidate order, keep the existing 10s stagger for the remaining enabled/configured accounts, and add regression coverage where `defaultAccount` is not alphabetically first.

## Reproduction Assessment

Yes: current source gives a high-confidence unit-level reproduction by configuring enabled Discord accounts such as `billy`, `farber`, and `main` with `channels.discord.defaultAccount: "main"`; the delay helper uses the sorted account list, so `main` receives a nonzero delay. I did not run a live gateway restart.

## Solution Assessment

No: the current implementation is not the best solution because it preserves the stagger but ignores the existing default-account contract. The narrow maintainable fix is to derive priority from the resolved/default configured account, not hard-coded operator-specific ids.
