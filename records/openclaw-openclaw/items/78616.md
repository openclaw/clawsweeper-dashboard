## Summary

Keep open: current main still calculates Discord startup delay from the sorted Discord account list, so a configured default account such as `main` can be delayed behind earlier-sorted specialist accounts. The duplicate-token fix from https://github.com/openclaw/openclaw/issues/78118 is present, but it addresses a different same-token runtime hazard rather than distinct-account startup priority.

## What This Changes

Discord multi-account startup should start the configured primary/default account before other enabled Discord accounts while preserving the existing stagger.

## Best Possible Solution

Honor the resolved Discord default account first in the startup candidate order, keep the existing 10s stagger for the remaining enabled/configured accounts, and add regression coverage where `defaultAccount` is not alphabetically first.

## Reproduction Assessment

Yes: current source gives a high-confidence unit-level reproduction by configuring enabled Discord accounts such as `billy`, `farber`, and `main` with `channels.discord.defaultAccount: "main"`; the delay helper uses the sorted list, so `main` receives a nonzero delay. I did not run a live gateway restart.

## Solution Assessment

No: the current implementation is not the best solution because it preserves the stagger but ignores the existing default-account contract. The narrow maintainable fix is to derive priority from the resolved configured default account, not hard-coded operator account ids.
