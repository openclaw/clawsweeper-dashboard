## Summary

Keep this open for maintainer decomposition: release v2026.6.5 already hides delivery mirrors from model replay and normalizes volatile delivery IDs for loop detection, but loop detection remains disabled by default and the secure intervention policy is unresolved.

## What This Changes

Add structural safeguards against repeated outbound message-tool loops and provide a secure way to stop affected sessions.

## Best Possible Solution

Preserve the shipped transcript filtering and volatile-result normalization, then define a generic default loop-safety policy with explicit legitimate-repeat semantics and design cross-session intervention through the existing authenticated gateway/session authorization boundary rather than a product-specific default allowlist.

## Reproduction Assessment

Partly yes: current source confirms that rolling-history loop blocking is opt-in and therefore permits repeated sends when disabled, while the incident gives a concrete sequence. The exact kimi-code feedback loop has not been independently reproduced on current main, and delivery-mirror replay is already filtered.

## Solution Assessment

No, not as a single coordinated patch. Two structural asks are already shipped; the remaining work needs separate decisions about default loop policy, legitimate repeated messages, connector/provider-specific residuals, and secure cross-session abort authority.
