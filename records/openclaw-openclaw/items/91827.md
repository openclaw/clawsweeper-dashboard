## Summary

Close: current main and v2026.6.6 already address the central runaway loop by filtering delivery-mirror rows out of model replay and making message-send loop detection ignore volatile delivery IDs; the downstream orchestrator allowlist belongs in operator config, not OpenClaw core defaults.

## What This Changes

The issue asks OpenClaw to prevent Kimi-code message loops with content-based duplicate suppression, model-hidden delivery mirrors, a duplicate-call guard, and downstream orchestrator stop/abort access.

## Best Possible Solution

Keep the shipped replay filter and volatile-ID-aware loop hashing as the core fix, and use existing `tools.agentToAgent` and session visibility config for downstream orchestrator authority.

## Reproduction Assessment

No current-main failing reproduction remains from source inspection: delivery mirrors are filtered out of model replay, and regression tests cover repeated same-content sends with changing delivery IDs. I did not run a live Discord/Kimi replay because the shipped source and focused tests cover the two implicated seams.

## Solution Assessment

Yes for the central issue: the shipped implementation fixes the replay and loop-hash layers without gateway-wide duplicate suppression that could discard intentional repeated sends. The downstream `ag-manudis` allowlist request is better handled as operator config, not a core default.
