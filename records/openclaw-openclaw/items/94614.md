Full review comments:

- **[P2] Route the interval through every Feishu send path:** `extensions/feishu/src/reply-dispatcher.ts:241`
  - body: `sendIntervalMs` is added as an outbound rate-limiting config, but this PR only reads it in the reply/comment dispatcher loops. The message-tool outbound adapter still calls comment delivery, text/card sends, media sends, and native cards directly, while streaming-card updates/closes also bypass the new interval, so users can configure the option and still burst into Feishu rate limits. Put the delay behind a shared Feishu outbound scheduler used by all send surfaces, or narrow the config/issue claim to text chunks only.
  - confidence: 0.9
- **[P3] Clamp the configured interval before using setTimeout:** `extensions/feishu/src/reply-dispatcher.ts:482`
  - body: The schema accepts any non-negative integer and the value is passed straight to `setTimeout`. OpenClaw already has safe timer helpers because oversized delays overflow Node's timer range; a typo like `3000000000` can collapse the intended pacing instead of waiting. Add an upper bound or clamp through the existing timer helper before arming the delay.
  - confidence: 0.82
- **[P3] Publish the new config through metadata and docs:** `extensions/feishu/src/config-schema.ts:195`
  - body: The Zod schema accepts `sendIntervalMs`, but the Feishu plugin manifest config schema and Feishu channel docs still expose only the existing message-limit and optimization fields. New config surfaces should keep runtime schema, plugin metadata, and user-facing docs aligned so setup/discovery and operators see the same contract.
  - confidence: 0.86
