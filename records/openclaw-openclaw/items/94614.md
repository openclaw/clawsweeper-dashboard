Full review comments:

- **[P2] Route the interval through every Feishu send path:** `extensions/feishu/src/reply-dispatcher.ts:241`
  - body: `sendIntervalMs` is added as outbound rate-limiting config, but this PR only reads it in reply/comment dispatcher loops. The outbound adapter, media/card sends, and streaming-card sends still call Feishu APIs without the interval, so users can configure pacing and still burst into Feishu limits. Put the delay behind a shared Feishu outbound scheduler, or narrow the config and linked behavior claim to text chunks only.
  - confidence: 0.9
- **[P3] Clamp the configured interval before scheduling:** `extensions/feishu/src/reply-dispatcher.ts:482`
  - body: The schema accepts any non-negative integer and the value is passed straight to `setTimeout`. Oversized timer values can overflow Node's timer range, so a typo can collapse the intended pacing instead of waiting. Add an upper bound or route the delay through the existing timer-safe helper before arming it.
  - confidence: 0.82
- **[P3] Publish the config in metadata and docs:** `extensions/feishu/src/config-schema.ts:195`
  - body: The Zod schema accepts `sendIntervalMs`, but the Feishu plugin manifest config schema and Feishu channel docs still expose only the existing message-limit and optimization fields. New config surfaces should keep runtime schema, plugin metadata, and user-facing docs aligned so setup/discovery and operators see the same contract.
  - confidence: 0.86
