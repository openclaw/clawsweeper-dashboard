## Label Justifications

- `P1`: The report describes a current Mattermost channel workflow where generated text blocks disappear from the visible conversation during tool-heavy turns.
- `impact:message-loss`: Earlier streamed text segments are overwritten in the Mattermost post, leaving only the later content visible to the user.

## Summary

Keep open. Current main and the latest release still suppress Mattermost core block replies on the normal inbound path and route partial text through a single editable draft post, while the overlapping Mattermost draft-boundary PR at https://github.com/openclaw/openclaw/pull/75256 is still open and does not explicitly close this issue.

## What This Changes

The issue asks Mattermost to preserve separate streamed text blocks instead of overwriting one editable post during block streaming and block-style preview streaming.

## Best Possible Solution

Honor explicit Mattermost block streaming when preview streaming is off, and make the supported Mattermost `streaming.mode: "block"` path preserve text-block boundaries with focused tests, either by folding the fix into https://github.com/openclaw/openclaw/pull/75256 or replacing it with a narrower branch.

## Reproduction Assessment

Yes, source-level: Mattermost inbound dispatch currently forces `disableBlockStreaming: true`, while the draft stream edits an existing `streamPostId` and the monitor never splits it at text/tool boundaries. I did not run a live Mattermost server in this read-only review.

## Solution Assessment

Unclear as a landing path because the best code change should be coordinated with the already-open Mattermost draft-boundary PR. The maintainable fix should keep the existing config surface and add regression coverage for both explicit block replies and Mattermost block preview boundaries.
