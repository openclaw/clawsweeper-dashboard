## Label Justifications

- `P1`: The issue reports a broken Mattermost channel workflow where tool-heavy replies overwrite visible intermediate content for real users.
- `impact:message-loss`: Earlier Mattermost text/tool blocks are overwritten or cleared, leaving only the final block visible in the channel history.

## Summary

Keep open. Current main still routes normal Mattermost inbound replies through a single editable draft and hard-disables block streaming, while the open clean linked PR at https://github.com/openclaw/openclaw/pull/87449 is the canonical closing fix with supplied proof.

## What This Changes

The issue asks Mattermost to preserve separate visible text blocks during block-style streaming instead of repeatedly editing one post and leaving only the final segment visible.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/87449; once it merges, close this issue as implemented with the merge commit and release provenance.

## Reproduction Assessment

Yes. Source inspection on current main shows the normal Mattermost inbound path hard-disables block streaming and never calls forceNewMessage at text/tool boundaries, while the draft stream edits the existing post once streamPostId is set; I did not rerun a live Mattermost server scenario.

## Solution Assessment

Yes. The best current path is the linked PR's bounded Mattermost fix using existing streaming/blockStreaming contracts and the local draft-stream primitive, rather than adding another config surface or opening a duplicate repair PR.
