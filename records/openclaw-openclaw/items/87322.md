## Label Justifications

- `P1`: The issue describes a broken Mattermost channel workflow where completed assistant text blocks can disappear from visible channel history for real users.
- `impact:message-loss`: The reported and source-supported failure mode drops earlier visible reply blocks by overwriting a single Mattermost post instead of preserving separate block messages.

## Summary

Keep open. Current main still has a source-level mismatch: Mattermost accepts and documents explicit `blockStreaming`, but the normal inbound path forces core block streaming off and continues using the editable draft-preview path, while the overlapping Mattermost draft-boundary PR is still open and does not close this issue.

## What This Changes

The issue asks for Mattermost `blockStreaming: true` to preserve each completed text block as a separate Mattermost post instead of overwriting one edited post.

## Best Possible Solution

Land one Mattermost-local fix that lets explicit `blockStreaming: true` bypass or disable draft-preview delivery for normal inbound replies and route completed blocks as normal new Mattermost posts, coordinated with the open draft-boundary PR if that remains the preferred landing path.

## Reproduction Assessment

Yes, at source level. Current main accepts the Mattermost `blockStreaming` config, but the normal inbound path passes `disableBlockStreaming: true`, which core resolves to block streaming off, while the draft stream edit path reuses one post id.

## Solution Assessment

Yes for the blockStreaming contract. The narrow maintainable fix is to honor explicit Mattermost block streaming by using normal block delivery rather than the editable draft-preview sink, while coordinating with the separate draft-boundary mode work.
