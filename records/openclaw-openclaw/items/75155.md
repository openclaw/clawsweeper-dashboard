## Summary

Keep open. Current main still has the native macOS Talk-to-WebChat transcript gap: Talk sends speech through `chat.send` outside the SwiftUI view model, while the native chat transport does not subscribe to or map `session.message` transcript events; the browser Control UI already has that reload path.

## What This Changes

Request to render macOS Talk Mode user voice transcripts as user turns in the WebChat/Control UI thread, matching typed input.

## Best Possible Solution

Add native SwiftUI WebChat support for active-session transcript updates by subscribing the macOS transport to `sessions.messages.subscribe`, mapping `session.message` into the shared chat transport/view model, and merging or refreshing the active session so Talk-created user turns appear like typed input.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: send a Talk Mode transcript to the active native WebChat session; Talk writes through `chat.send`, while native SwiftUI has no subscription or event mapping for the accepted `session.message` user transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to bridge the existing gateway transcript-event contract into native WebChat; a `talk.echoTranscript` option would turn a UI consistency bug into a setting instead of fixing the shared session contract.
