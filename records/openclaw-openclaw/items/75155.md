## Summary

Keep open. Current `main` still has a native macOS WebChat gap: Talk Mode sends recognized speech through `chat.send`, but the Swift transport/view model has no `session.message` event path, while the browser Control UI already reloads history from that event.

## What This Changes

Render Talk Mode speech transcripts as user messages in the WebChat/Control UI thread, matching typed input behavior.

## Best Possible Solution

Add native SwiftUI WebChat support for transcript updates by subscribing to the active session's message events, mapping `session.message` into the Swift transport, refreshing or merging active-session user turns, and cleaning Talk prompt display so users see the spoken transcript rather than model-facing instructions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: send a Talk transcript to an active native WebChat session; Talk calls `chat.send` directly, while the native Swift transport has no `sessions.messages.subscribe` or `session.message` mapping to show the accepted user turn promptly.

## Solution Assessment

Yes. The narrow maintainable fix is a native transcript-event bridge plus Talk display cleanup; a `talk.echoTranscript` setting would hide a shared chat consistency bug instead of aligning voice input with typed input.
