## Summary

Keep open. Current main still has a native macOS WebChat gap: Talk Mode submits the recognized transcript through the shared gateway `chat.send` path, but the SwiftUI WebChat transport/view model does not subscribe to or handle the gateway `session.message` transcript event that browser Control UI already uses to refresh externally added turns.

## What This Changes

The issue requests that Talk Mode voice transcripts render as user messages in the macOS WebChat/Control UI thread instead of only sending them to the model.

## Best Possible Solution

Add native macOS SwiftUI WebChat support for active-session transcript updates: subscribe/unsubscribe through `sessions.messages.subscribe`, map `session.message` into the Swift transport, and have the view model refresh or merge visible user turns for the active session. The displayed row should show the spoken transcript rather than the Talk prompt wrapper, while leaving the browser Control UI event behavior intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: start native macOS WebChat on the active/main session, send a Talk Mode transcript, and the Talk path writes through `chat.send` while native SwiftUI lacks the `session.message` subscription/mapping needed to surface that externally submitted user turn.

## Solution Assessment

Yes. The narrow maintainable fix is a native session-message bridge plus display cleanup for Talk transcripts; adding a `talk.echoTranscript` opt-in would mask a consistency bug between voice and typed input instead of fixing the shared transcript contract.
