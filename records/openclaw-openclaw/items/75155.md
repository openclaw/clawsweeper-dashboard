## Summary

Keep open. Current main still has the native macOS WebChat gap: Talk Mode submits recognized speech through `chat.send`, but the SwiftUI WebChat transport/view model does not subscribe to or map `session.message` transcript events, so externally added user turns can stay invisible until a later history refresh path runs.

## What This Changes

The issue requests that user transcripts sent by macOS Talk Mode render in the WebChat/Control UI chat history like typed user messages.

## Best Possible Solution

Add native macOS SwiftUI WebChat support for active-session transcript updates by subscribing to `sessions.messages.subscribe`, mapping `session.message` into the Swift transport, and refreshing or merging the active session's user turn with Talk-specific display cleanup. Keep the existing browser Control UI event behavior intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: send a Talk Mode transcript to the active native WebChat session; Talk writes through `chat.send`, while native SwiftUI does not subscribe to or handle the gateway `session.message` event that carries the accepted user transcript.

## Solution Assessment

Yes. The narrow maintainable fix is a native transcript-event bridge plus display cleanup for Talk transcripts; a `talk.echoTranscript` opt-in would paper over a consistency bug between voice and typed input instead of fixing the shared transcript contract.
