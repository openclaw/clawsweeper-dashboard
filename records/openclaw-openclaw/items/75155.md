## Summary

Keep open. Current main still has a native macOS WebChat gap: Talk Mode submits the spoken transcript through the shared gateway `chat.send` path, while typed input only appears because the SwiftUI view model appends its own local user row. The gateway and browser Control UI already have `session.message` transcript events, but the macOS transport neither subscribes to those per-session events nor maps them into a transport event, so the native thread can miss externally submitted Talk user turns.

## What This Changes

The issue asks for Talk Mode speech transcripts submitted from macOS to appear as user messages in the native WebChat/Control UI history like typed input.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for active-session transcript updates: subscribe the macOS transport to the current session's message events, map `session.message` into a typed transport event, and update the view model to merge or refresh the visible user message without disrupting active runs. The rendered row should show the spoken transcript rather than the Talk prompt wrapper, while keeping the browser Control UI contract intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: open native macOS WebChat on the active/main session, submit a Talk Mode transcript, and the Talk path uses direct `chat.send` while the native transport lacks the `session.message` subscription/mapping needed to surface that externally submitted user turn immediately.

## Solution Assessment

Yes. The narrow maintainable fix is a native event/subscription bridge plus display cleanup for Talk transcripts; adding a `talk.echoTranscript` opt-in would hide a consistency bug instead of making voice input behave like typed input.
