## Summary

Keep open. Current main still appears to have the native macOS WebChat gap: Talk Mode submits the spoken transcript through direct `chat.send`, the backend and browser Control UI have a `session.message` transcript-update path, but the native SwiftUI WebChat transport/view model still lacks an active-session `session.message` subscription/mapping/merge path for externally submitted user turns.

## What This Changes

Talk Mode voice transcripts should render as visible user messages in the macOS WebChat/Control UI chat history, matching typed input behavior.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for external transcript updates: subscribe the active session to gateway `session.message` updates, map them through the shared chat transport, refresh or merge history without disrupting active runs, and display the spoken transcript rather than the model-facing Talk prompt wrapper. Keep the browser Control UI behavior intact and add focused native/shared chat regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: open native macOS WebChat for the active session, submit a Talk Mode transcript, and the Talk path uses direct `chat.send` while native WebChat has no `session.message` path comparable to browser Control UI. I did not run the macOS app live in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is native event/subscription handling plus Talk transcript display sanitation; adding an opt-in `talk.echoTranscript` setting would mask a consistency bug instead of making voice input behave like typed input.
