## Summary

Keep open. Current main still has a native macOS WebChat gap for Talk Mode: the spoken turn is sent through `chat.send` and the Gateway emits transcript updates, but the SwiftUI chat transport/view model does not subscribe to or map `session.message`, while typed input appears because it is locally appended before send.

## What This Changes

The issue requests that Talk Mode speech transcripts render as user messages in the WebChat/Control UI thread, matching typed input behavior.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for active-session transcript updates: subscribe the Mac transport to current-session `session.message` events, map them into a typed chat transport event, and have the view model merge or refresh the visible message list without disrupting active runs. The user row should show the spoken transcript rather than the Talk prompt wrapper, while browser Control UI behavior stays intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: open native macOS WebChat on the active session, submit a Talk Mode transcript, and the Talk path uses direct `chat.send` while native WebChat lacks the `session.message` subscription/mapping needed to surface that externally submitted user turn.

## Solution Assessment

Yes. The narrow maintainable fix is a native event/subscription bridge plus display sanitation for Talk transcripts; a `talk.echoTranscript` opt-in setting would mask a consistency bug instead of making voice input behave like typed input.
