## Summary

Keep open. Current main still does not give native macOS WebChat the active-session transcript update path that browser Control UI uses: Talk Mode submits through `chat.send` outside the typed-input view model path, while the native SwiftUI transport ignores `session.message`. The remaining work is narrow and source-backed.

## What This Changes

Show Talk Mode speech transcripts as user messages in WebChat/Control UI chat history, matching typed WebChat input.

## Best Possible Solution

Add native macOS SwiftUI WebChat support for active-session transcript updates: subscribe/unsubscribe through the gateway session-message API, map `session.message` into the Swift transport, refresh or merge active-session user turns in the view model, and display the spoken transcript rather than the Talk prompt wrapper.

## Reproduction Assessment

Yes. The source-level reproduction path is high-confidence: use native macOS WebChat on the active session, send a Talk Mode transcript, and the Talk path writes through `chat.send` while native SwiftUI lacks the `session.message` bridge that would surface the externally added user turn like typed input.

## Solution Assessment

Yes. The narrow maintainable fix is a native session-message bridge plus Talk transcript display cleanup; a `talk.echoTranscript` opt-in would hide a consistency bug between voice and typed input instead of fixing the shared transcript contract.
