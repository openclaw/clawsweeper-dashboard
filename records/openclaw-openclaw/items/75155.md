## Summary

Keep open. Current main still has a macOS native WebChat gap for Talk Mode: Talk sends an external `chat.send` turn through the shared gateway connection, while the SwiftUI chat transport/view model does not subscribe to or map the gateway `session.message` transcript event that browser Control UI already handles. The fix is narrow and source-local, so this is a good repair candidate rather than cleanup-close material.

## What This Changes

Render macOS Talk Mode speech transcripts as visible user messages in the WebChat/Control UI chat history alongside assistant replies.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for active-session transcript updates: subscribe the Mac transport to the current session's message events, map `session.message` into a typed transport event, and have the view model refresh or merge the visible user message without disrupting active runs. The displayed user row should show the spoken transcript rather than the `TalkPromptBuilder` model-facing wrapper, while browser Control UI behavior stays intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: open native macOS WebChat on the active/main session, submit a Talk Mode transcript, and the Talk path uses direct `chat.send` while native WebChat lacks the `session.message` subscription/mapping needed to surface that externally submitted user turn.

## Solution Assessment

Yes. The best fix is a native event/subscription bridge plus display sanitation for Talk transcripts; adding an opt-in `talk.echoTranscript` setting would mask a consistency bug instead of making voice input behave like typed input.
