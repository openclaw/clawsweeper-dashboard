## Summary

Keep open. The PR targets a real gateway timeout regression, but it only changes the generic fallback in `callGatewayLeastPrivilege`; the linked CLI media-send path still reaches gateway-executed plugin dispatch with an explicit 10000ms timeout, so the reported large-file send can still fail.

## What This Changes

The PR changes `src/gateway/call.ts` so gateway calls without an explicit timeout resolve the shared preauth handshake default of 15000ms instead of falling back to 10000ms.

## Best Possible Solution

Align gateway-executed message-action calls with the shared preauth timeout default while preserving explicit caller overrides, simplify the redundant ternary in the PR, update the now-failing timeout tests, and add focused regression coverage for plugin-dispatched CLI media sends.

## Reproduction Assessment

Yes. The linked report gives a concrete CLI command and failure output, and current main shows a deterministic static path where `messageCommand` reaches gateway-executed plugin dispatch with an explicit `timeoutMs: 10_000`.

## Solution Assessment

No. The PR changes only the fallback used when no timeout is passed to `callGatewayLeastPrivilege`, but the reported message-action route passes an explicit 10000ms timeout before reaching that fallback.
