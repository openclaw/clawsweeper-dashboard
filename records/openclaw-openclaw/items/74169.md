## Summary

Keep open: the PR targets a real timeout mismatch, but it only changes the generic no-timeout fallback while the reported CLI media-send path already passes an explicit 10000 ms message-action timeout, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes `resolveGatewayCallTimeout` so missing gateway call timeouts resolve the shared 15000 ms handshake default instead of falling back to 10000 ms.

## Best Possible Solution

Align gateway-executed message-action calls with the shared 15000 ms timeout policy while preserving explicit caller overrides, update focused gateway/message-action regression coverage, and require redacted real-send proof before merge.

## Reproduction Assessment

Yes by source inspection. The reported CLI command enters `messageCommand` without a timeout override, `resolveGatewayActionOptions` fills `10_000`, and gateway-mode plugin sends pass that explicit timeout to the `message.action` RPC.

## Solution Assessment

No. The PR raises the fallback for calls with no timeout, but the reported path already supplies `timeoutMs: 10_000`, so the fix needs to cover the message-action resolver or stop passing that hard-coded default.
