## Summary

Keep open. The PR addresses a real gateway timeout bug, but it only changes the generic gateway fallback while the linked CLI media-send route still passes an explicit 10000 ms timeout through the message-action runner, and the PR has no after-fix real behavior proof.

## What This Changes

The PR changes `src/gateway/call.ts` so gateway calls without an explicit timeout resolve the shared 15000 ms preauth handshake default instead of falling back to 10000 ms.

## Best Possible Solution

Align gateway-executed message-action calls with the shared preauth default while preserving explicit caller overrides, then add focused regression coverage for the CLI media-send timeout path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the linked `openclaw message send --channel wecom --media ...` command enters CLI message dispatch without a timeout override, then gateway-executed plugin dispatch supplies `timeoutMs: 10_000`. I did not run a live WeCom upload in this read-only review.

## Solution Assessment

No. The PR changes the fallback used only when `callGatewayLeastPrivilege` receives no timeout, but the reported route already passes an explicit 10000 ms timeout from the message-action runner.
