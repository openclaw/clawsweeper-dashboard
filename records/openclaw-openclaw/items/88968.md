## What This Changes

The PR changes auto-reply memory-flush failure handling so flush errors are logged while the main reply continues, adds reset-on-exhaustion behavior, and clears memory-flush state during session reset paths.

## Best Possible Solution

Rebind the active ReplyOperation whenever memory-flush exhaustion rotates the session, keep the log-and-continue behavior, and merge only after the cleanup policy and live proof are acceptable.

- **[P1] Rebind the reply operation after rotating the session:** `src/auto-reply/reply/agent-runner.ts:1590-1595`
  - body: This reset updates `activeSessionEntry` and `followupRun`, but it never calls `replyOperation.updateSessionId()` for the newly generated session. The existing memory-flush paths call `updateSessionId` whenever they adopt a new session id, so this new reset path needs the same rebind before continuing.
