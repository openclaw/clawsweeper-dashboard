## Summary

Needs changes before merge: the core recovery path is a plausible narrow fix for the memory-flush death loop, but the branch still expands an existing user-facing config setting without updating the matching help/docs contract, and maintainers still need to accept the fail-open session/message-delivery tradeoff.

## What This Changes

The PR changes memory-flush failures from terminal visible reply failures into log-and-continue handling, rotates exhausted sessions while preserving the old transcript, clears memory-flush state on reset, and adds regression tests plus an opt-in degraded notice.

## Best Possible Solution

Land the bounded fail-open/rotation fix after the notifyUser help/docs contract is aligned and maintainers accept the session-state and message-delivery semantics.

Yes. Current main is source-reproducible: when `runMemoryFlushIfNeeded` reports visible memory-flush error payloads, `runReplyAgent` calls `replyOperation.fail("run_failed")` and returns those payloads before the normal reply turn.

## Solution Assessment

Not fully yet. The runtime fix is an acceptable mitigation for the wedge, but the expanded `notifyUser` behavior needs docs/help alignment and maintainer acceptance of the fail-open session-rotation semantics before it is the best mergeable solution.
