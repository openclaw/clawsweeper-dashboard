## Summary

Keep open for maintainer review: the patch is a coherent candidate fix with strong proof, but it intentionally changes memoryFlush from terminal error delivery to fail-open reply delivery, rotates session state on exhaustion, broadens an existing notice setting, and overlaps the same pre-run maintenance pipeline as another open PR.

## What This Changes

The PR changes auto-reply memoryFlush failures to log and continue, rotate exhausted sessions without deleting old transcripts, clear stale flush state on reset, and update notifyUser docs/help text.

## Best Possible Solution

Merge only after maintainers accept the fail-open recovery semantics and reconcile the open pre-run ordering PR, keeping the regression tests and docs that pin non-terminal delivery, non-destructive rotation, and broadened notifyUser behavior.

## Reproduction Assessment

Yes for the code path: current main still calls replyOperation.fail("run_failed") and returns before the main reply when memoryFlush produces visible error payloads. I did not run a live provider-crash repro, but the PR body includes redacted Gateway logs showing all 12 after-fix messages completing under repeated mock flush 500s.

## Solution Assessment

Unclear as a product decision, but the implementation is a coherent candidate fix: it logs maintenance errors, rotates only after tracked exhaustion, preserves transcripts, and continues the main reply. The unresolved question is whether maintainers want those permanent fail-open/session-rotation/notice semantics and how to reconcile them with https://github.com/openclaw/openclaw/pull/84792.
