## Summary

Keep open for maintainer review: the patch appears to fix the current `main` abort path with sufficient real behavior proof and no line-level correctness finding, but it deliberately changes message-delivery, session-state, and existing `notifyUser` semantics while overlapping a conflicting pre-run maintenance PR.

## What This Changes

This PR changes auto-reply memoryFlush failures to log-and-continue, rotate exhausted sessions non-destructively, clear stale memoryFlush reset state, and document the broadened `compaction.notifyUser` notice behavior.

## Best Possible Solution

Merge only after maintainers accept the fail-open recovery semantics and reconcile the open pre-run ordering PR, keeping the regression tests and docs that pin non-terminal delivery, non-destructive rotation, and broadened `notifyUser` behavior.

## Reproduction Assessment

Yes for the code path: current `main` still calls `replyOperation.fail("run_failed")` and returns before the main reply when memoryFlush produces visible error payloads. I did not run a live reproduction in this read-only pass, but the PR body includes redacted Gateway logs showing the after-fix path delivering all 12 replies under repeated mock flush 500s.

## Solution Assessment

Unclear as a product decision, but the implementation is a coherent candidate fix: logging memoryFlush maintenance errors and rotating after exhaustion addresses the observed reply death loop. The maintainer choice is whether fail-open delivery, non-destructive rotation, broadened `notifyUser`, and reconciliation with https://github.com/openclaw/openclaw/pull/84792 are the desired permanent semantics.
