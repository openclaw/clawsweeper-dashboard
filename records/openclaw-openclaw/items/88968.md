## Summary

Keep open: current main and the latest release still abort the user reply when memory-flush maintenance emits visible error payloads, while this branch removes that terminal path and adds bounded exhaustion recovery. I found no blocking line-level correctness finding, but merge still needs maintainer acceptance of the session-state, message-delivery, and expanded `notifyUser` semantics.

## What This Changes

The PR changes auto-reply memory-flush failure handling to log and continue, rotate exhausted sessions while preserving transcripts, clear stale flush state on resets, update focused tests, and document the expanded compaction notice behavior.

## Best Possible Solution

Land or revise this branch after maintainers accept the fail-open reply behavior, session-rotation recovery, and expanded `notifyUser` semantics; keep broader compaction-order and configurable-mode work in separate canonical items.

## Reproduction Assessment

Yes. Source inspection of current main shows visible memory-flush error payloads still call `replyOperation.fail("run_failed")` before the main reply, and the current test suite explicitly expects that old terminal reply path.

## Solution Assessment

Yes for the central bug: keeping memory-flush failures out of `replyOperation.fail()` and rotating only after exhaustion is a bounded fix in the owner path. Maintainers still need to accept the session-state and notice-semantics tradeoff before merge.
