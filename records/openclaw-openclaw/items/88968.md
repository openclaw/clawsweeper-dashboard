## Summary

Keep open for maintainer review: the patch appears correct and well-proven, but it intentionally changes memory-flush failure delivery, active session recovery, and an existing notice setting in compatibility-sensitive runtime code.

## What This Changes

The PR changes auto-reply memoryFlush failures to log and continue, rotate exhausted sessions while preserving transcripts, clear stale flush metadata on resets, and update `notifyUser` docs/help.

## Best Possible Solution

Land this PR or an equivalent maintainer-approved recovery after explicitly accepting the fail-open rotation/notice semantics and reconciling the ordering overlap with https://github.com/openclaw/openclaw/pull/84792.

Yes. Current main and the latest release still show visible memory-flush errors calling `replyOperation.fail` before the normal reply path; the PR body also supplies redacted Gateway/mock-provider logs where repeated flush failures no longer abort replies.

## Solution Assessment

Mostly yes. The patch is a bounded fix for the observed abort loop and follows existing reset/rebind patterns, but fail-open delivery, automatic rotation, and expanded notice behavior need maintainer acceptance.
