## Summary

Keep this PR open: the user-visible memory-flush death-loop fix is useful, but the new exhaustion-reset path still leaves the active `ReplyOperation` bound to the pre-reset session id, and the supplied proof is indirect rather than live Gateway/channel proof of a reply delivered after a failed flush.

## What This Changes

The PR changes auto-reply memory-flush handling so flush error payloads are logged instead of returned, rotates and clears session state on flush exhaustion, and adds focused regression tests.

## Best Possible Solution

Keep the log-and-continue direction, but rebind the active `ReplyOperation` whenever exhaustion rotation creates a new session id, add focused restart-recovery coverage for that path, and require redacted live Gateway/channel proof before merge.
