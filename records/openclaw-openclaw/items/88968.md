## Summary

Keep open for maintainer review: current main still aborts the user reply when memory flush produces visible errors, and this PR is a bounded, proof-backed fix, but it deliberately changes fail-open reply delivery, active session rotation, and `notifyUser` semantics.

## What This Changes

The PR changes auto-reply memory-flush failures to log and continue, rotate exhausted sessions while preserving transcripts, clear stale flush state on resets, and update docs/help for the expanded degraded notice.

## Best Possible Solution

Merge this PR or an equivalent bounded fix after maintainers explicitly accept the fail-open reply behavior, non-destructive session rotation, and expanded `notifyUser` notice semantics.

## Reproduction Assessment

Yes, source-reproducible: current main routes visible memory-flush errors into `replyOperation.fail("run_failed")` before the normal reply run, and the linked issue provides incident logs for the same failure class. I did not run a live provider-crash reproduction in this read-only pass.

## Solution Assessment

Unclear as a product policy, but acceptable as a bounded code fix: it touches the owner path, preserves transcripts, and adds focused tests/docs. The remaining question is whether maintainers want fail-open reply delivery and automatic session rotation as the default recovery behavior.
