## Summary

Keep this issue open for the fallback-approval decision. Current main already covers model attribution through `messages.responsePrefix`, but it still automatically invokes eligible fallback candidates without an approval contract, and no newer viable issue or PR owns that remaining capability.

**Likely related people**
- Peter Steinberger — high confidence; introduced the model fallback chain.
- Sebastian — high confidence; introduced the response-prefix template variables used for attribution.

## What This Changes

Add optional user approval before switching to a fallback model and expose the selected provider/model in outbound messages.

## Best Possible Solution

Keep model attribution on `messages.responsePrefix`, and if fallback approval is accepted, add one typed fallback-decision seam with explicit unattended defaults and portable channel presentation rather than channel-specific prompts.

## Reproduction Assessment

Yes for the feature gap: current-main source shows automatic candidate execution with no approval hook or accepted approval config, while attribution is already verifiable through response-prefix variables.

## Solution Assessment

No, the original combined two-flag proposal is not the best final shape. Attribution should reuse the existing generic template, while approval needs a product-approved cross-caller decision contract with defined timeout and unattended behavior.
