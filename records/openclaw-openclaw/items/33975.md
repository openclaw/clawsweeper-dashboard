## Summary

Keep open: current main covers much of the model-attribution request through response-prefix templates and fallback notices, but current main and the latest release still have no opt-in approval gate before invoking the next fallback model.

## What This Changes

The issue requests optional config for approving model fallback before switching and visible provider/model attribution in responses.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, use existing `messages.responsePrefix` templates for general attribution, and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection gives high confidence that attribution exists through response-prefix templates while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. The narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
