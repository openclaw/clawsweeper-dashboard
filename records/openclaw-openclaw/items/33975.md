## Summary

Keep this issue open because current main only covers the model-attribution half through `messages.responsePrefix`; the requested approval gate before executing a fallback model remains absent and requires an explicit core product decision.

## What This Changes

Request an opt-in approval gate before model fallback and visible model attribution on responses.

## Best Possible Solution

Keep attribution on the existing `messages.responsePrefix` template path, and only add fallback approval after maintainers define a transport-neutral core decision contract with explicit timeout, cancellation, headless, and reliability semantics.

## Reproduction Assessment

Not applicable as a failing bug, but the current behavior is source-reproducible with high confidence: fallback candidates execute automatically and no approval configuration or hook exists.

## Solution Assessment

No, not as one bundled implementation. Model attribution already belongs on the generic response-prefix path; the fallback gate should be treated separately and designed around the shared runtime’s interactive and non-interactive callers.
