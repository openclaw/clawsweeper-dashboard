## Summary

Current main covers the attribution half through response-prefix templates, but fallback candidates still run without a user-approval boundary. The narrower confirmation report was closed in favor of this canonical issue, so the remaining product/API decision should stay open.

## What This Changes

Add optional user approval before switching to a fallback model and expose the responding provider/model in outbound messages.

## Best Possible Solution

Keep attribution on `messages.responsePrefix`, and design one generic opt-in fallback-transition approval contract with explicit routing, timeout, cancellation, and resume semantics that transport plugins can render without owning model policy.

## Reproduction Assessment

Yes, at source level. Current main advances through fallback candidates without a pre-attempt approval callback, while the attribution half can be exercised through documented response-prefix variables.

## Solution Assessment

No, not as the two proposed parallel settings. Attribution should reuse the existing template system, while fallback approval needs a deliberate generic orchestration and interaction contract.
