## Summary

Keep this issue open. Current main and the latest release already cover the model-attribution half through `messages.responsePrefix` template variables, but fallback approval before executing the next model candidate is still missing and remains a real product/config decision.

## What This Changes

Request an opt-in fallback approval mode before model fallback plus optional model attribution in outbound messages.

## Best Possible Solution

Keep attribution on `messages.responsePrefix`; if fallback approval is accepted, design one default-off fallback-decision contract across fallback callers with explicit unattended and timeout behavior.

## Reproduction Assessment

Yes for the feature gap. Current-main source shows the strict model schema lacks approval fields and `runWithModelFallback` runs eligible candidates without an approval hook; this is not a bug reproduction.

## Solution Assessment

No, the original combined two-flag proposal is not the best final shape. Attribution should use the existing response-prefix template path, while fallback approval needs a maintainer-approved cross-caller decision contract.
