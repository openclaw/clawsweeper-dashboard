## Summary

Keep open: current main covers most of the message-attribution request through `messages.responsePrefix` templates, but it still has no config field or runtime approval gate before model fallback, and the latest substantive discussion confirms that approval is the remaining core gap.

## What This Changes

This issue requests opt-in confirmation before switching to a fallback model and optional model/provider attribution in user-visible replies.

## Best Possible Solution

Keep this as the canonical core design tracker for opt-in per-agent fallback approval; use `messages.responsePrefix` for reply attribution and leave usage-footer-specific display to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high confidence that response-prefix attribution exists while fallback approval has no schema field or runtime hook.

## Solution Assessment

Unclear as an implementation path. The attribution portion already has a maintainable prefix mechanism, but fallback approval needs maintainer decisions on schema, prompt surface, timeout, denial, and proceed/fail semantics before a narrow patch is appropriate.
