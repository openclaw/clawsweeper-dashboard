## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` template variables and fallback notices, but it still has no opt-in approval gate before the fallback runner attempts the next model; the closed duplicate at https://github.com/openclaw/openclaw/issues/41294 points back here as the canonical fallback-approval tracker.

## What This Changes

The issue requests opt-in fallback approval before model failover plus optional provider/model attribution in outbound messages.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, preserve default automatic fallback behavior, use existing response-prefix templates for general reply attribution, and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection gives high confidence that attribution exists through response-prefix templates while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. The narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
