## Summary

Keep open. Current main covers much of the model-attribution need through `messages.responsePrefix` templates, but the requested approval gate before executing a fallback model is still absent from schema, runtime types, fallback execution, and docs; the related duplicate was closed in favor of this issue, and the latest human follow-up confirms that approval remains the gap.

## What This Changes

The issue requests opt-in user approval before model fallback and optional visible provider/model attribution on responses.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval; use `messages.responsePrefix` with `{modelFull}` for immediate reply attribution and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that attribution has a current `messages.responsePrefix` path, while fallback approval has no config field, runtime type, or approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public config and cross-channel approval semantics. The maintainable direction is a narrow opt-in fallback approval policy, while attribution should continue using the existing response-prefix/template surfaces unless maintainers choose a separate footer UI path.
