## Summary

Keep open: current main covers the model-attribution half through `messages.responsePrefix` template variables and shipped fallback notices, but the requested opt-in approval gate before model fallback is still missing and remains a product/config decision.

## What This Changes

Request an opt-in fallback approval policy before model failover plus optional provider/model attribution in outbound messages.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, use existing `messages.responsePrefix` templates for message attribution, and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source and docs inspection give high confidence that attribution exists through response-prefix templates while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. The narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
