## Summary

Keep open. Current main and v2026.5.19 cover model attribution through response-prefix template variables and fallback notices, but they still lack an opt-in approval gate before the fallback runner attempts the next model; the duplicate fallback-confirmation report was already closed to this issue as the canonical tracker.

## What This Changes

The issue requests an opt-in confirmation step before switching from a primary model to a fallback model, plus optional provider/model attribution in replies.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, use existing `messages.responsePrefix` templates for reply attribution, and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source and docs inspection give high confidence that attribution exists through response-prefix templates while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. The narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
