## Summary

Keep open: current main partially covers model visibility through `messages.responsePrefix` template variables, but it still has no fallback-approval config or approval pause before fallback execution.

## What This Changes

The issue asks for opt-in user approval before switching to a fallback model and optional model/provider attribution in outgoing replies.

## Best Possible Solution

Keep this as the product/design tracker for a core per-agent fallback approval policy, while treating prefix-based model display as already mostly covered by `messages.responsePrefix`.

## Reproduction Assessment

Not applicable. This is an enhancement request; source inspection shows the current schema and fallback loop lack the requested approval gate, while message model variables are already documented.

## Solution Assessment

Unclear as an implementation path. The attribution side has a maintainable existing prefix mechanism, but fallback approval needs maintainer decisions on config shape, prompt surface, timeout, and fail/proceed semantics before a narrow patch is appropriate.
