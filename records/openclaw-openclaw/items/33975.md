## Summary

Keep open: current main covers model attribution through `messages.responsePrefix` templates and fallback notices, but it still lacks an opt-in approval boundary before executing a fallback model candidate.

## What This Changes

Request an opt-in fallback approval mode plus visible model attribution in outbound messages.

## Best Possible Solution

Keep attribution on `messages.responsePrefix`, and design one generic opt-in fallback-transition approval contract with explicit routing, timeout, cancellation, and resume semantics.

## Reproduction Assessment

Yes, at source level. Current main supports model attribution through response-prefix variables, but the fallback loop has no approval config or pre-attempt approval hook before calling the fallback candidate.

## Solution Assessment

No, not as the original two parallel settings. Attribution should reuse `messages.responsePrefix`, while fallback approval needs a deliberate generic orchestration contract rather than an ad hoc config key.
