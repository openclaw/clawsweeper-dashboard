## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` template variables, but the requested opt-in approval gate before fallback execution is still absent and needs maintainer product/config review.

## What This Changes

The issue requests opt-in user approval before model fallback plus optional model attribution in outgoing messages.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, while using `messages.responsePrefix` for current model attribution and `https://github.com/openclaw/openclaw/issues/65477` for usage-footer placement.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high confidence that attribution exists through `messages.responsePrefix`, while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main or `v2026.5.12`.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. If accepted, the narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
