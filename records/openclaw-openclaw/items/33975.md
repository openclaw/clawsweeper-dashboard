## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` template variables, but the fallback-approval gate is still absent and this issue is the canonical tracker for that remaining product/config decision.

## What This Changes

Request an opt-in fallback confirmation policy before switching models, plus optional provider/model attribution in user-visible responses.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, while using `messages.responsePrefix` for current attribution and https://github.com/openclaw/openclaw/issues/65477 for usage-footer placement.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high confidence that attribution exists through `messages.responsePrefix`, while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main or `v2026.5.12`.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. If accepted, the narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
