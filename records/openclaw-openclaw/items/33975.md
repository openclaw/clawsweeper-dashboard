## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` template variables, but the fallback-approval gate is still absent and the duplicate fallback-confirmation report already points here as the canonical tracker.

## What This Changes

The issue requests an opt-in fallback approval mode before model failover plus optional model attribution in delivered messages.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, while using existing `messages.responsePrefix` templates for attribution and leaving usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high confidence that attribution exists through `messages.responsePrefix`, while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. If accepted, the narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
