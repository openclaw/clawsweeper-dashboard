## Summary

Current main partially satisfies the visibility request through configurable response-prefix templates, but it still has no opt-in approval gate before executing a fallback model. The latest discussion explicitly confirms that split, so closing this would lose a real remaining product decision.

## What This Changes

The issue requests opt-in fallback confirmation before switching models and optional model attribution in outbound assistant messages.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either add a documented opt-in fallback approval policy in the agent model config and fallback runtime, or explicitly decide that fallback approval belongs outside core orchestration.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested fallback approval config and runtime pause, while model attribution is already partly available through responsePrefix templates.

## Solution Assessment

Unclear. The attribution half is already best served by messages.responsePrefix templates, but the fallback approval half needs maintainer design before a narrow implementation can be called the best solution.
