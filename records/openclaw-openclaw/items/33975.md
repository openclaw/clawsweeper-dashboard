## Summary

Keep open. Current main and v2026.5.19 cover model visibility through `messages.responsePrefix` template variables, but there is still no config field, runtime type, or fallback-loop approval gate before `runWithModelFallback` advances to the next candidate; the duplicate confirmation report already points here as the canonical tracker.

## What This Changes

Requests opt-in user approval before model fallback plus optional provider/model attribution on assistant responses.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, use existing `messages.responsePrefix` templates for attribution, and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high confidence that attribution exists through `messages.responsePrefix`, while fallback approval has no config field, runtime type, or fallback-loop hook on current main or v2026.5.19.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. If accepted, the narrow maintainable path is a default-off per-agent fallback approval policy that reuses existing approval delivery where appropriate.
