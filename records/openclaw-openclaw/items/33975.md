## Summary

Keep open. Current main covers the message attribution half through `messages.responsePrefix` template variables, but the opt-in approval gate before running a fallback model is still absent, and the related interactive fallback request is a closed satellite rather than an implementation.

## What This Changes

The issue requests optional user approval before switching to a fallback model and optional model/provider attribution in outbound replies.

## Best Possible Solution

Keep this as the canonical fallback-approval tracker. Treat model attribution as satisfied by `messages.responsePrefix` templates, then design an opt-in per-agent fallback approval policy in the model config/schema and wire `runWithModelFallback` to a maintainer-approved approval surface with docs and tests for approve, deny, timeout, cancellation, and channel delivery.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence source-level verification path: proposed approval keys are rejected by the strict model schema and the fallback runner/test suite still proves automatic fallback execution.

## Solution Assessment

Unclear. A per-agent opt-in flag is likely the right boundary, but the exact config shape and approval lifecycle need maintainer product review before this is safe to implement mechanically.
