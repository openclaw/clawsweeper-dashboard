## Summary

Keep open. Current main covers the model-attribution half through response-prefix template variables and UI message metadata, but the remaining fallback-approval behavior still has no config field, runtime type, or pause point before fallback execution; the duplicate fallback-confirmation report was already closed to this issue as the canonical tracker.

## What This Changes

The issue requests opt-in approval before switching to fallback models plus visible model attribution in responses.

## Best Possible Solution

Keep this as the canonical core tracker for opt-in per-agent fallback approval, use existing `messages.responsePrefix` templates for reply attribution, and leave usage-footer placement to https://github.com/openclaw/openclaw/issues/65477.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection gives high confidence that attribution exists through response-prefix templates while fallback approval has no config field, runtime type, or fallback-loop approval hook on current main.

## Solution Assessment

Unclear until maintainers decide the public API and cross-channel approval semantics. The narrow maintainable path is an opt-in per-agent fallback approval policy that preserves default automatic fallback behavior and reuses existing approval delivery where appropriate.
