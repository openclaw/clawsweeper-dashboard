## Label Justifications

## Summary

Keep open. Current main and the latest release still only support outbound WhatsApp `react` plus approval-specific reaction handling; they do not expose the generic `message reactions` query action, store/search historical WhatsApp reactions, or provide a public reaction hook/event path. The related implementation PRs are closed unmerged, and the open hook issue is narrower complementary work rather than a replacement for this broader tracker.

## What This Changes

Request support for reading/querying WhatsApp message reactions and surfacing reaction feedback to agents or plugins.

## Best Possible Solution

Define one canonical WhatsApp reaction read model, then implement live add/remove events and any supported historical query inside the WhatsApp plugin through generic channel/plugin contracts, with SQLite-backed persistence only if historical retrieval is intentionally supported.

## Reproduction Assessment

Yes, at source level. Current main lacks a WhatsApp `reactions` action, a general reaction event listener, and a public reaction hook; no live WhatsApp account was exercised in this read-only review.

## Solution Assessment

Unclear at final contract detail. A channel-owned in-process implementation is the right boundary, but maintainers still need to choose the public query, hook, persistence, and privacy-gating contract before a safe fix PR can be queued.
