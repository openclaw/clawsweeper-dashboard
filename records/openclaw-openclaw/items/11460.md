## Summary

Keep open: current main still supports outbound WhatsApp `react` only; generic reaction querying, historical reaction retrieval/storage, and inbound reaction event routing remain unimplemented. The related open PRs are useful implementation candidates, but https://github.com/openclaw/openclaw/pull/46881 and https://github.com/openclaw/openclaw/pull/48570 are still unmerged, and https://github.com/openclaw/openclaw/issues/79996 is a complementary hook request rather than a replacement for this broader issue.

## What This Changes

Request WhatsApp support for querying message reactions, historical reaction retrieval, and agent-accessible reaction events.

## Best Possible Solution

Implement a WhatsApp-owned reaction read path that advertises and handles `reactions`, persists or retrieves historical reaction data where feasible, routes inbound add/remove reaction events through approved access-control and hook/session pipelines, reconciles the open PRs, and adds focused tests plus docs.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp advertises and dispatches only `react`/`poll`, the gateway returns the unsupported-channel path for unhandled `reactions`, and the inbound contract has no reaction event listener or metadata fields.

## Solution Assessment

No current-main implementation solves the issue. The maintainable path needs maintainer scope selection for query versus persistence versus hook/session events, then a single WhatsApp plugin implementation that reconciles the overlapping open PRs.
