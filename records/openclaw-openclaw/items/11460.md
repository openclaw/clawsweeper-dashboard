## Summary

Keep open: current main still supports outbound WhatsApp `react` only, while the requested reaction query/listing, historical retrieval or persistence, and inbound reaction exposure remain unimplemented. The related PRs and narrower hook issue are useful implementation input, but they are open or closed unmerged and do not supersede this broader tracker.

## What This Changes

Add WhatsApp support for querying message reactions, retrieving or storing historical reactions, and surfacing reaction events to agents or plugins.

## Best Possible Solution

Keep this as the broad WhatsApp reaction-read tracker until the official WhatsApp plugin supports `reactions` query/list behavior, persists or retrieves historical reactions where feasible, and emits privacy-gated inbound reaction events with focused tests, docs, and redacted live proof.

## Reproduction Assessment

Yes for the reported unsupported query path by source inspection: WhatsApp advertises and dispatches `react`, not `reactions`, and the shared runner throws the unsupported-channel error when no plugin handles the action. I did not exercise a live WhatsApp account.

## Solution Assessment

Unclear until maintainers choose the public API and privacy model. The safest direction is a WhatsApp-owned design that reconciles generic reaction querying, historical storage or retrieval, inbound event delivery, and the overlapping open implementation PRs.
