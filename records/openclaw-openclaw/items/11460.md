## Summary

Keep open: current main still supports WhatsApp outbound `react` only, while the requested reaction listing, historical query/storage, and inbound reaction event surfaces remain missing and related PRs are still unmerged implementation candidates.

## What This Changes

The issue requests WhatsApp support for reading/querying message reactions, including historical reactions and inbound reaction visibility.

## Best Possible Solution

Keep this open until the bundled WhatsApp plugin implements a coherent reaction read path: advertise and handle `reactions`, persist or retrieve historical reactions where feasible, surface inbound add/remove events through normal routing and access control, and cover it with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message reactions` is a generic action, but the WhatsApp plugin only advertises and dispatches `react`, so a WhatsApp `reactions` request reaches the standard unsupported-action path.

## Solution Assessment

No. Current main is not the solution because it only sends/removes reactions; the open PRs are useful partial candidates, but maintainers still need to decide the historical query/storage scope before the request is complete.
