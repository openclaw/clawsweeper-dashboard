## Label Justifications

- `P2`: This is a meaningful WhatsApp channel capability gap with limited blast radius and no current emergency regression signal.
- `impact:message-loss`: Inbound WhatsApp reaction events are currently not surfaced or queryable, so a class of channel feedback messages is effectively suppressed.

## Summary

Keep open: current main and the latest release still only support outbound WhatsApp `react`; they do not advertise or handle the generic `reactions` list action, store/query historical WhatsApp reactions, or surface ordinary inbound reaction events. https://github.com/openclaw/openclaw/issues/79996 is useful related hook work, but it is narrower than this issue's CLI/history/query request.

## What This Changes

Request to add WhatsApp support for listing, storing, and querying message reactions, plus surfacing inbound reaction events for agent feedback.

## Best Possible Solution

Keep this open until the WhatsApp plugin has a complete reaction read path: advertise and handle `reactions`, persist or retrieve historical reactions where feasible, surface inbound add/remove events through the access-controlled plugin/agent pipeline, and cover it with focused WhatsApp tests and docs.

## Reproduction Assessment

Yes, at source level: current main still rejects WhatsApp `reactions` through action discovery and dispatch, while docs only claim WhatsApp outbound `react`. I did not run a live WhatsApp account for this read-only review.

## Solution Assessment

Unclear. The issue identifies the right gap, but maintainers still need to choose the durable product shape across CLI listing, historical persistence, and hook/agent event delivery; https://github.com/openclaw/openclaw/issues/79996 covers only the narrower hook path.
