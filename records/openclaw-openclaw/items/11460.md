## Summary

Keep open. Current main still exposes WhatsApp outbound `react` only and has no WhatsApp implementation for listing reactions, storing/querying historical reaction data, or routing inbound reaction events; the related PRs are unmerged partial implementations rather than shipped fixes.

## What This Changes

The issue requests WhatsApp support for querying/listing message reactions, surfacing searchable reaction events, and retrieving historical reactions for agent feedback.

## Best Possible Solution

Implement a WhatsApp-owned read path that advertises and handles `message reactions`, persists or retrieves reaction data where feasible, routes inbound add/remove reaction events through the normal access-control pipeline, and updates focused WhatsApp tests plus docs.

## Reproduction Assessment

Yes for the unsupported query path: the CLI `reactions` command dispatches the generic `reactions` action, while the WhatsApp plugin only supports `react`, so current main reaches the unsupported-action response. The inbound and historical gaps are also source-verifiable because no WhatsApp reaction listener, metadata fields, or storage/query path exists.

## Solution Assessment

No current solution is present on main. The safest direction is to consolidate the open PR work into a WhatsApp plugin implementation that covers both inbound reaction events and the generic query/historical-read contract instead of merging a partial event-only path as the whole fix.
