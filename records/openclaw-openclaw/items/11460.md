## Summary

Keep open: current main still limits WhatsApp to outbound `react`, lacks WhatsApp `reactions` query support and inbound reaction storage/listeners, and the related PRs remain unmerged partial implementation candidates.

## What This Changes

Request support for querying WhatsApp message reactions, searchable reaction events, and historical reaction retrieval.

## Best Possible Solution

Keep this open until the WhatsApp plugin has a coherent reaction read path: advertise and handle `reactions`, persist or retrieve historical reactions where feasible, surface inbound add/remove events through the normal routing and access-control pipeline, and cover the behavior with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message reactions` is a generic action, but the WhatsApp plugin only supports `react`, so a WhatsApp `reactions` request reaches the standard unsupported-action path on current main.

## Solution Assessment

No. Current main is not a solution because it only sends/removes reactions; the open PRs are useful partial candidates, but maintainers still need to decide the historical query/storage scope before the issue is complete.
