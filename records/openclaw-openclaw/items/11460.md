## Summary

Keep open: current main still supports WhatsApp outbound `react` only, while generic reaction querying, historical reaction retrieval/storage, and inbound reaction event routing remain unimplemented. The related open pull requests are useful implementation candidates, but they are unmerged and do not supersede this broader issue.

## What This Changes

The issue requests WhatsApp support for querying, storing, and surfacing message reactions so agents can use reaction feedback.

## Best Possible Solution

Implement a WhatsApp-owned reaction read path that advertises and handles `reactions`, persists or retrieves historical reaction data where feasible, routes inbound add/remove reaction events through approved access-control and hook/session pipelines, reconciles the open pull requests, and adds focused tests plus docs.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: WhatsApp advertises and dispatches only `react`/`poll`, so `openclaw message reactions --channel whatsapp` reaches the generic unsupported-channel path, and inbound reaction metadata has no listener or storage contract.

## Solution Assessment

No current-main implementation solves the issue. The maintainable fix needs maintainer scope selection for query versus persistence versus hook/session events, then a single WhatsApp plugin implementation that reconciles the overlapping open pull requests.
