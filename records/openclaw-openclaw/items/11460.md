## Summary

Keep open: current main still only implements outbound WhatsApp `react`; it does not advertise or handle the generic `reactions` query action, and related PRs are open or closed unmerged rather than shipped fixes.

## What This Changes

Add a WhatsApp reaction read path so agents and CLI users can query, receive, and use message reactions.

## Best Possible Solution

Keep this open until the bundled WhatsApp plugin has a coherent read path: advertise and handle `reactions`, persist or retrieve historical reactions where feasible, surface inbound add/remove events through the normal pipeline, and cover it with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message reactions` is a recognized generic action, but the WhatsApp plugin only supports `react`, so a WhatsApp `reactions` request reaches the standard unsupported-action path.

## Solution Assessment

No. Current main is not a complete solution because it only sends/removes reactions; the maintainable fix belongs primarily in the WhatsApp plugin, with core/SDK changes only if a generic missing seam is proven.
