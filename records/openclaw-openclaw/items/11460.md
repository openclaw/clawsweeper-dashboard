## Label Justifications

- `P2`: This is a bounded improvement for a major messaging channel with meaningful user value but no urgent outage or regression.

## Summary

Current main has a narrow, shipped-capable inbound WhatsApp reaction path for thumbs-based approvals, but it still does not expose arbitrary reaction events to agents/plugins or support the generic `message reactions` query and historical retrieval requested here; the issue remains useful as the umbrella for the unresolved read/query behavior.

## What This Changes

Add general WhatsApp support for observing message reactions and retrieving reactions associated with current or historical messages.

## Best Possible Solution

Keep one official WhatsApp socket listener, expose arbitrary add/remove reactions through a typed generic plugin/channel event with normal access controls, and add historical `reactions` lookup only through a canonical SQLite-backed store with explicit retention, identity, and message-key semantics.

## Reproduction Assessment

Yes, at source level: current main explicitly excludes WhatsApp from the generic `reactions` action and limits inbound reaction handling to approval targets. This review did not exercise a live WhatsApp account, so transport proof is supplied by the adjacent merged approval implementation rather than a new live run.

## Solution Assessment

No single alternative in the issue is the complete best fix. The maintainable direction separates typed live event delivery from optional historical persistence while sharing the official channel-owned socket listener.
