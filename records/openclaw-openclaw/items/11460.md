## Summary

Keep open: current main still supports outbound WhatsApp `react` only, while generic `reactions` querying, inbound reaction event routing, and historical reaction storage/query semantics remain absent; the two related implementation PRs are still open and need reconciliation.

## What This Changes

Add WhatsApp support for querying message reactions, receiving reaction events, and retrieving historical reaction feedback for agents.

## Best Possible Solution

Implement a WhatsApp-owned reaction read path that advertises and handles `reactions`, persists or retrieves historical reaction data where feasible, routes inbound add/remove reaction events through access control, and adds focused tests plus docs.

## Reproduction Assessment

Yes from source inspection: the generic runner has an unsupported-action path, while the WhatsApp plugin only advertises and dispatches `react`, not `reactions`. Current source also lacks a WhatsApp `messages.reaction` listener or reaction metadata in the inbound message contract.

## Solution Assessment

No current-main implementation solves it. The maintainable fix needs a WhatsApp plugin reaction read/event contract plus a maintainer choice on historical retrieval versus local persistence, likely reconciling https://github.com/openclaw/openclaw/pull/46881 and https://github.com/openclaw/openclaw/pull/48570.
