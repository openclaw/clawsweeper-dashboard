## Summary

Keep open: current main still only supports outbound WhatsApp `react`; generic reaction querying, inbound reaction event routing, and historical reaction storage/query remain unimplemented. The related PRs are useful implementation input, but https://github.com/openclaw/openclaw/pull/46881 and https://github.com/openclaw/openclaw/pull/48570 are still open/unmerged, and https://github.com/openclaw/openclaw/issues/79996 is a complementary hook API request rather than a full replacement for this issue.

## What This Changes

Add WhatsApp support for querying message reactions, surfacing reaction events, and making historical reaction feedback available to agents.

## Best Possible Solution

Implement a WhatsApp-owned reaction read path that advertises and handles `reactions`, persists or retrieves historical reaction data where feasible, routes inbound add/remove reaction events through the approved access-control and hook/session pipeline, and reconciles the open PRs with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection shows WhatsApp advertises and dispatches only `react`/`poll`, the gateway returns the standard unsupported-channel error for unhandled `reactions`, and the inbound monitor/type contract has no reaction event path.

## Solution Assessment

No current-main implementation solves it. The maintainable solution needs maintainer scope selection for query versus persistence versus hook/session events, then a single WhatsApp plugin implementation that reconciles https://github.com/openclaw/openclaw/pull/46881 and https://github.com/openclaw/openclaw/pull/48570.
