## Summary

Keep open. Current main still supports outbound WhatsApp `react` only: the WhatsApp plugin does not advertise or dispatch the generic `reactions` action, does not expose reaction metadata on inbound messages, and does not attach a WhatsApp reaction-event listener. The related implementation PRs remain open or closed unmerged, so they are useful context but not shipped fixes.

## What This Changes

The issue requests WhatsApp support for reading/querying message reactions, including historical lookup and inbound reaction events.

## Best Possible Solution

Keep this open until the bundled WhatsApp plugin has a complete read path: advertise and handle `message reactions` for WhatsApp, persist or retrieve reaction data where feasible, surface inbound add/remove reaction events through the normal routing and access-control pipeline, and add focused WhatsApp tests plus docs. Review #46881 and #48570 as implementation inputs rather than treating them as shipped fixes.

## Reproduction Assessment

Yes for the query path: `openclaw message reactions --channel whatsapp` reaches the generic `reactions` action, but the current WhatsApp plugin only supports `react`, so the runner returns the unsupported-action behavior. The inbound-event gap is also source-verifiable because no WhatsApp `messages.reaction` listener or reaction metadata fields exist in current main.

## Solution Assessment

No current solution is present on main. The existing outbound `react` implementation is useful but does not solve reaction listing, historical lookup, or inbound reaction-event exposure; the safer path is a WhatsApp-owned implementation with tests and docs, informed by the open PRs.
