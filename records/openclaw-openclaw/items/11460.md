## Summary

Keep open: current main still supports WhatsApp outbound `react` only; generic reaction listing, historical reaction storage/querying, and inbound reaction-event routing remain unimplemented. The related pull requests at https://github.com/openclaw/openclaw/pull/46881 and https://github.com/openclaw/openclaw/pull/48570 are useful implementation candidates, but both are still open and reviewed as incomplete rather than shipped fixes.

## What This Changes

Add WhatsApp support for querying message reactions, surfacing reaction events, and retrieving historical reaction feedback for agents.

## Best Possible Solution

Keep this as the broad WhatsApp reaction-read tracker until a WhatsApp-owned implementation advertises and handles `reactions`, persists or retrieves historical reaction data where feasible, routes inbound add/remove events through approved privacy and access-control paths, and reconciles the overlapping open PRs and narrower hook request at https://github.com/openclaw/openclaw/issues/79996.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp advertises and dispatches only `react`/`poll`, so `openclaw message reactions --channel whatsapp` reaches the generic unsupported-channel path and there is no inbound reaction listener or storage contract.

## Solution Assessment

No current-main implementation solves this issue. The maintainable path needs maintainer scope selection for query versus persistence versus hook/session events, then a single WhatsApp plugin implementation that reconciles https://github.com/openclaw/openclaw/pull/46881, https://github.com/openclaw/openclaw/pull/48570, and the narrower hook discussion.
