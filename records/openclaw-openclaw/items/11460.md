## Label Justifications

- `P2`: The missing WhatsApp reaction read/event path is a normal-priority channel capability gap with limited blast radius.
- `impact:message-loss`: WhatsApp reaction feedback is not exposed through query or general inbound event paths, so reaction signals can be unavailable to agents and plugins.

## Summary

Keep open. Current main and v2026.6.5 still expose WhatsApp outbound `react` and approval-reaction handling, but not the generic `message reactions` query action, historical reaction retrieval, or a general inbound reaction event path for agents/plugins.

## What This Changes

The issue requests WhatsApp support for reading/querying message reactions, including historical reaction retrieval and agent-accessible reaction events.

## Best Possible Solution

Add a privacy-gated WhatsApp reaction read/event contract in the WhatsApp plugin: advertise and handle `reactions` only when it can return real data, preserve approval-reaction behavior, route inbound add/remove events through access control, and document whether historical reactions are queried from WhatsApp, persisted locally, or unsupported.

## Reproduction Assessment

Yes, at source level. Current main does not advertise or dispatch WhatsApp `reactions`, and the production inbound monitor has no general reaction listener beyond approval-specific handling.

## Solution Assessment

No, current main is not the best end state because outbound `react` and approval reactions do not solve querying or general inbound reaction visibility. The maintainable fix needs a WhatsApp-owned read/event contract with privacy and persistence decisions made explicitly.
