## Summary

Keep open: current main still implements WhatsApp outbound and ack `react` only; generic `reactions` query support, inbound reaction event routing, and historical storage/query semantics remain absent while related PRs are still open or unmerged.

## What This Changes

Request support for WhatsApp message-reaction querying, inbound reaction event access, and historical reaction retrieval.

## Best Possible Solution

Implement a WhatsApp plugin read path that advertises and handles `reactions`, records or retrieves historical reaction data, and routes inbound add/remove reaction events through access control with focused tests and docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: the CLI sends the generic `reactions` action, but the WhatsApp plugin only advertises and dispatches `react`, so a WhatsApp `reactions` request reaches the standard unsupported-action path.

## Solution Assessment

Is this the best way to solve the issue? No current-main implementation solves it; the maintainable direction is a WhatsApp-owned reaction read/event contract with a maintainer decision on historical retrieval versus local persistence.
