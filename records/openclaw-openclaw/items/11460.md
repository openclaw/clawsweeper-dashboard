## Summary

Keep open: current main still supports outbound WhatsApp `react` only; the requested reaction listing/query path, historical retrieval or persistence, and inbound reaction exposure are not implemented. The related open PRs and narrower hook issue are useful inputs, but they are not merged fixes and do not cover the full request.

## What This Changes

Add WhatsApp support for querying message reactions, retrieving or storing historical reactions, and surfacing inbound reaction feedback to agents or plugins.

## Best Possible Solution

Keep this as the broad WhatsApp reaction-read tracker until the official WhatsApp plugin supports `reactions` query/list behavior, persists or retrieves historical reactions where feasible, and emits privacy-gated inbound reaction events with focused tests, docs, and redacted live proof.

## Reproduction Assessment

Yes for the reported unsupported query path by source inspection: the CLI can dispatch `reactions`, but WhatsApp advertises and handles only `react`, and the shared runner throws the unsupported-channel error when no plugin handles the action. I did not exercise a live WhatsApp account.

## Solution Assessment

Unclear until maintainers choose the API and privacy model. The maintainable direction is a WhatsApp-owned design that reconciles generic reaction querying, historical storage or retrieval, inbound event delivery, and the overlapping open implementation PRs.
