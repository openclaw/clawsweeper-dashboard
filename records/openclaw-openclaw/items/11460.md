## Label Justifications

- `P2`: This is a normal-priority WhatsApp channel capability gap with limited blast radius but clear user value for reaction feedback.
- `impact:message-loss`: WhatsApp reaction feedback is currently not exposed through the generic query path or a general inbound event path, so agents and plugins can miss those message signals.

## Summary

Keep open. Current main and v2026.6.5 still support WhatsApp outbound/status/approval reactions, but not the generic `message reactions` query action, historical reaction retrieval, or a general inbound reaction event surface; the open hook request at https://github.com/openclaw/openclaw/issues/79996 is related but narrower than this issue.

## What This Changes

Request support for querying, storing, or surfacing WhatsApp message reactions so agents can use reaction feedback.

## Best Possible Solution

Add a WhatsApp-owned, privacy-gated reaction read/event contract: advertise and handle `message reactions` only when it can return real data, preserve approval-reaction behavior, route inbound add/remove events through access control, and document whether historical reactions are queried, persisted, or unsupported.

## Reproduction Assessment

Yes, at source level. Current main does not advertise or dispatch WhatsApp `reactions`, and the production inbound monitor has no general reaction listener beyond approval-specific handling.

## Solution Assessment

No, current main is not the best end state because outbound `react` and approval reactions do not solve querying or general inbound reaction visibility. The maintainable fix needs an explicit WhatsApp read/event contract with privacy and persistence decisions settled first.
