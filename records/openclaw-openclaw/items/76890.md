## Summary

Keep open: current main still has a source-backed path from memory-core session-start warming into full session transcript indexing on the Gateway process, and the linked fix PR #76978 is open with closing syntax.

## What This Changes

The issue reports memory-core session transcript indexing on new-session startup monopolizing the Gateway event loop for 30+ seconds, with disabling the memory-core plugin as the only effective workaround.

## Best Possible Solution

Land or revise #76978 so memory-core session sync yields, defers, targets, or offloads heavy transcript indexing while preserving current session source semantics and adding focused regression coverage.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a clear path from session-start memory warming into full session-file enumeration and transcript indexing on the Gateway process without a session-loop macrotask yield.

## Solution Assessment

Yes: the narrow maintainable fix is in memory-core session sync responsiveness, matching the existing Gateway sessions.list yield precedent. A blanket config gate or disabling the plugin would remove intended session-memory behavior rather than preserving asynchronous indexing.
