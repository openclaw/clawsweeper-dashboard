## Summary

Keep open: current main still has a source-backed path for memory-core session-start/full-reindex transcript indexing on the Gateway process without a batch macrotask yield, and #76978 is an open unmerged closing PR rather than an implemented fix.

## What This Changes

The issue reports that memory-core session-start indexing can monopolize the Gateway event loop for 30+ seconds and cause WebSocket/TUI timeouts.

## Best Possible Solution

Land or revise #76978, or an equivalent memory-core fix, so session transcript indexing yields, defers, targets, or offloads heavy work while preserving existing session source semantics.

## Reproduction Assessment

No live high-confidence reproduction was run for the 30+ second stall. Source inspection on current main gives a credible path from session-start warming into full transcript indexing without a session-loop macrotask yield.

## Solution Assessment

Yes: the narrow maintainable fix belongs in memory-core session sync responsiveness, not in a new config knob or provider policy. The linked PR’s yield-based direction matches the existing Gateway sessions.list precedent and should be reviewed or landed.
