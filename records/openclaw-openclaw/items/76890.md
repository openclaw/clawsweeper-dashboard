## Summary

Keep open: the report is recent and concrete, and current main still has a source-backed path from memory-core session-start warming into full session-corpus indexing on the Gateway process.

## What This Changes

The issue asks memory-core to stop blocking the Gateway event loop during new-session/session-start memory indexing, including cases where memory search is expected to be disabled.

## Best Possible Solution

Make memory-core session-start and session ingestion work yield, defer, target, or offload heavy transcript indexing so Gateway RPC and health checks remain responsive without changing plugin slot semantics or adding a new setting.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show a plausible current-main path from session-start memory warming into full session-file enumeration and indexing on the Gateway process.

## Solution Assessment

Partly yes: fixing the responsiveness inside memory-core session sync is the narrow maintainable direction. A blanket memorySearch.enabled gate needs care because current tests intentionally allow dreaming session ingestion when that feature is enabled.
