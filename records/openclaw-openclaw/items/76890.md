## Summary

Keep open: current main still has a source-backed path from memory-core session-start warming into bulk session transcript indexing, and open PR #76978 already targets this issue with closing syntax.

## What This Changes

The issue reports that memory-core session-start transcript indexing can monopolize the Gateway event loop for 30+ seconds when a new session starts with session memory enabled.

## Best Possible Solution

Land or revise #76978 so memory-core session sync yields, defers, targets, or offloads heavy transcript indexing while preserving existing session source semantics and regression coverage.

## Reproduction Assessment

No: I did not run a live Gateway/TUI corpus reproduction, so the exact 30+ second duration is not independently measured. Source inspection on current main shows session-start warming can enter full session transcript indexing without a session-loop macrotask yield.

## Solution Assessment

Yes: the narrow maintainable direction is to fix memory-core session sync responsiveness, as #76978 attempts, rather than relying on disabling the memory-core plugin or adding a new config workaround.
