## Summary

Keep open: current main still has a source-backed session-start path into memory-core full session transcript indexing, and open PR #76978 explicitly closes this issue but has not merged.

## What This Changes

The issue reports that memory-core session-start indexing can block the Gateway event loop for 30+ seconds on large session corpora.

## Best Possible Solution

Land or revise #76978 so memory-core session indexing yields, defers, or offloads enough work to keep Gateway RPC responsive while preserving existing session source semantics.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current path from session-start warming into full session-file enumeration and indexing, and the reporter supplied concrete liveness logs and a plugin-disable workaround.

## Solution Assessment

Yes: yielding during session indexing, as proposed by #76978, is the narrow maintainable fix direction. A blanket memorySearch.enabled gate would need care because current behavior intentionally preserves some session ingestion paths.
