## Summary

Keep open: the report has concrete Gateway liveness evidence, and current source still exposes a session-start memory sync path that can enter full session-corpus reindexing without a proven nonblocking boundary.

## What This Changes

Request that memory-core stop blocking Gateway new-session handling with heavy session indexing, especially around session-start sync and disabled memory search settings.

## Best Possible Solution

Make memory-core session-start work bounded and nonblocking by avoiding full session-corpus reindex on the Gateway event loop, respecting disabled memorySearch consistently, and adding regression coverage for first-session responsiveness.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show a plausible current-main path from session-start memory warming into full session-file reindexing and CPU-heavy transcript processing.

## Solution Assessment

Yes, the requested repair direction is the narrow maintainable one: fix memory-core/session sync behavior and add yielding or deferral rather than asking users to disable the whole memory plugin.

## Repair Work Prompt

Fix the memory-core Gateway event-loop stall reported in https://github.com/openclaw/openclaw/issues/76890. Keep the fix inside memory-core/session sync unless a generic helper is strictly needed. Prevent session-start or disabled-memorySearch paths from doing full session-corpus indexing on the Gateway event loop; defer, chunk with setImmediate-style yielding, target only the active/dirty session, or move heavy work off-thread as appropriate. Preserve existing memory search behavior, plugin slot disablement, QMD behavior, and session indexing semantics for explicit full reindex commands. Add focused regression coverage showing session-start sync does not wait on or synchronously process a full session corpus, and that memorySearch.enabled=false does not start the reported memory-core indexing path. Add a user-facing CHANGELOG.md fix entry. Do not add a new config option unless maintainers explicitly ask.
