## What This Changes

The issue asks for memory-core session-start indexing to stop blocking the Gateway event loop and to honor memory search disablement without requiring the whole plugin to be disabled.

## Best Possible Solution

Make memory-core session-start work nonblocking by avoiding full session-corpus reindex on the Gateway event loop, respecting memorySearch.enabled consistently, and adding regression coverage for first-session startup responsiveness.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76890. Current main still lets memory-core session-start warming call sync({ reason: "session-start" }); full reindex can then include all session transcript files because needsFullReindex bypasses the session-start skip in shouldSyncSessionsForReindex(). Implement a narrow fix so creating/opening a new session cannot block the Gateway event loop with full session-corpus indexing. Keep the fix in memory-core/session-sync or the active memory runtime boundary; do not add a new config option, do not disable memory by default, and do not change plugin slot semantics. Ensure agents.defaults.memorySearch.enabled=false prevents memory-core indexing/listener work for that agent, and make any remaining session-start indexing targeted, deferred, or chunked with event-loop yields. Add/update regression tests for the session-start full-reindex gate and disabled-memory behavior. Add a CHANGELOG.md fix entry because this is user-facing. Validate with targeted memory tests first, then the changed gate in Testbox if promoted.
