## Summary

Keep open: current main still has a source-backed memory-core path from session-start warming into full session transcript indexing without an explicit session-indexing batch yield, and the linked open PR #76978 is the active closing implementation.

## What This Changes

The issue asks OpenClaw to prevent memory-core session transcript indexing from blocking the Gateway event loop during new-session startup.

## Best Possible Solution

Land or revise #76978, or an equivalent memory-core fix, so session transcript sync yields, defers, targets, or offloads heavy indexing while preserving existing session-source semantics and adding focused regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection on current main gives a clear path from session-start warming into session-file enumeration and transcript parsing without the proposed session-indexing batch yield.

## Solution Assessment

Yes: the narrow maintainable fix is in memory-core session sync responsiveness, matching the existing sessions.list yield pattern; a worker thread or new config option would be broader than needed for the reported failure.
