## Summary

Keep open: current main still has the source-backed memory-core session-start path into full session transcript indexing without a session-loop macrotask yield, and linked PR #76978 is an open closing PR rather than a merged fix.

## What This Changes

The issue reports that memory-core session-start indexing can block the Gateway event loop for 30+ seconds when opening a new session with session memory enabled.

## Best Possible Solution

Review and land or revise #76978 so memory-core session sync yields, defers, targets, or offloads heavy transcript indexing while preserving existing session source semantics and adding focused regression coverage.

## Reproduction Assessment

No live reproduction was run. Source inspection on current main gives a clear path from session-start memory warming into full session-file enumeration and indexing on the Gateway process without a session-loop macrotask yield.

## Solution Assessment

Yes: the narrow maintainable fix belongs in memory-core session sync responsiveness, not a new config option or a broad Gateway change. The already-linked PR targets that boundary by adding yields during session indexing.
