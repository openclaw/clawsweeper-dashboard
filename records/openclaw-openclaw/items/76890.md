## Summary

Keep open: the report is concrete and current-main source still shows an active memory-core session-start path that can enter full session-corpus reindexing on the Gateway event loop when the index metadata requires a full rebuild.

## What This Changes

The issue asks memory-core to stop blocking the Gateway event loop during brand-new session startup when session memory indexing is enabled or partly disabled.

## Best Possible Solution

Keep the fix inside memory-core/session sync: avoid full session-corpus reindexing on session-start, yield or offload heavy transcript parsing/indexing, and add regression coverage for Gateway responsiveness and disabled-memory boundaries.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source inspection does show a clear current path from session-start warming into full session-file reindexing when the memory index needs a rebuild.

## Solution Assessment

Yes, the requested direction is the right maintainable fix: repair memory-core's existing async indexing path instead of adding a new user-facing setting or moving plugin-specific behavior into core.
