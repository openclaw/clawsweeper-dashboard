## What This Changes

The issue reports that `sessions_history` and the dashboard session viewer expose transcript-only OpenClaw assistant rows, producing duplicate assistant messages.

## Best Possible Solution

Land the linked fix, or an equivalent narrow shared filter, so both `sessions_history` and dashboard history projection exclude transcript-only OpenClaw assistant rows while preserving persisted transcripts and real assistant messages.

## Reproduction Assessment

Yes, source-reproducible: a `chat.history` response containing an assistant row with `provider: openclaw` and model `delivery-mirror` or `gateway-injected` reaches `sessions_history` because only tool roles are stripped. I did not run a live dashboard repro in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to apply a shared transcript-only OpenClaw assistant predicate in the history output/projection path. Because a closing PR is already open, the right next step is to review, adjust, and land or close that PR rather than close this issue now.
