## Summary

Keep open: current main and the latest release still let transcript-only OpenClaw assistant rows reach `sessions_history`, and an open linked fix PR now owns the implementation path. The comments add useful confirmation from macOS 2026.5.20 and operator impact; the narrow bug remains real until the linked PR lands or is replaced.

## What This Changes

Filter `delivery-mirror` and `gateway-injected` OpenClaw assistant rows out of `sessions_history` and the dashboard history view so internal transcript mirrors do not appear as duplicate assistant messages.

## Best Possible Solution

Land or replace the linked fix so the shared history projection filters `provider: openclaw` assistant rows with `delivery-mirror` or `gateway-injected` before sanitization and byte capping, with regression coverage for both default and `includeTools: true` calls.

## Reproduction Assessment

Yes, source-reproducible: a `chat.history` result containing assistant rows with `provider: openclaw` and `model: delivery-mirror` or `gateway-injected` will pass through because current `sessions_history` only strips tool roles. I did not run a live dashboard repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to filter transcript-only OpenClaw assistant rows in the history tool/projection path while preserving stored transcripts and existing delivery behavior; broader knobs like cron relevance filters are separate feature work.
