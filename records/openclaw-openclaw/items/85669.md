- `impact:session-state`: The reported behavior is about session transcript/history rows drifting from the user-visible conversation by showing transcript-only duplicate assistant entries.

## Summary

Keep open: current main still has a clear source path where transcript-only OpenClaw assistant rows can reach `sessions_history` and dashboard history projection. Replay filters these rows, and append-time delivery-mirror dedupe exists, but neither covers already-returned `delivery-mirror`/`gateway-injected` history rows.

## What This Changes

Filter transcript-only OpenClaw assistant messages from session history surfaces so duplicate assistant turns do not appear in the dashboard or `sessions_history` output.

## Best Possible Solution

Add a shared transcript-only OpenClaw assistant predicate to the history projection/tool path, filter raw `delivery-mirror` and `gateway-injected` rows from returned history, and cover both dashboard `chat.history` projection and `sessions_history` with regressions.

## Reproduction Assessment

Yes, source-reproducible: a `chat.history` result containing an assistant row with `provider: openclaw` and `model: delivery-mirror` or `gateway-injected` will pass through `sessions_history` because only tool roles are stripped. I did not run a live dashboard repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to filter transcript-only OpenClaw assistant rows in the shared history projection/tool output while preserving persisted transcripts and Telegram delivery behavior.
