## Summary

Keep this issue open: the reporter found their fixed 180s wall-clock boundary was caused by an external HTTP client read timeout, but current main still misclassifies a plain client-disconnect AbortError as a timeout and the open linked PR is the active closing path.

## What This Changes

The issue reports embedded agent runs aborting around 165-180 seconds despite higher configured timeouts, later narrowed to an external HTTP client timeout plus misleading timeout classification on client disconnect.

## Best Possible Solution

Land the linked fix PR that distinguishes intentional TimeoutError abort signals from plain client-disconnect AbortError, or close that PR with a maintainer rationale if the current classification is accepted.

## Reproduction Assessment

Yes from source and reporter evidence, but not from a live run in this review. Current main can route a client-disconnect AbortError through isTimeoutError, whose text patterns match "This operation was aborted" as timeout-like.

## Solution Assessment

Yes, the linked PR is the narrowest maintainable path I found. It changes only abort-signal timeout classification, leaving general provider/failover timeout heuristics and config surfaces unchanged.
