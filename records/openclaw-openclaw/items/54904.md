## Summary

Keep open: the PR addresses a real Feishu webhookPath bug, but the current branch compares a raw incoming path against a normalized configured path, so valid callbacks with the same route plus a trailing slash can be rejected before signature handling after upgrade.

## What This Changes

The PR adds an early Feishu webhook request-path gate before rate limiting/body/signature processing and adds monitor transport tests for wrong paths, configured paths, crafted request targets, config normalization, and malformed targets.

## Best Possible Solution

Land the Feishu-owned early path gate after normalizing safe incoming paths, while preserving rejection for malformed, network-path, and dot-segment bypass targets and adding a trailing-slash regression test.

## Reproduction Assessment

Yes. Source inspection of the PR head shows that with configured /feishu/events, a request to /feishu/events/ returns 404 before signature handling because getWebhookRequestPath returns the raw pathname and the handler compares it to the normalized configured path.

## Solution Assessment

No. Enforcing Feishu routing before body and signature work is the right layer, but the safe origin-form request pathname should be normalized before comparison so same-route trailing-slash callback URLs are not broken.
