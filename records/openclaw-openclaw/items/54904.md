## Summary

Keep this PR open: current main and the latest shipped release still read Feishu `webhookPath` without enforcing it, while the PR adds the missing early route check with focused regression coverage. The patch looks functionally correct, but it intentionally tightens upgrade behavior for existing Feishu webhook deployments, so maintainer review should accept that compatibility and message-delivery risk before merge.

## What This Changes

The branch normalizes Feishu webhook paths, rejects malformed or non-matching request targets before webhook processing, and adds `monitorWebhook` regression tests.

## Best Possible Solution

Land strict Feishu `webhookPath` enforcement with maintainer-accepted upgrade guidance for operators to align Feishu callback and proxy paths.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.5.28` read `webhookPath` but never compare the incoming request target before dispatch. I did not run a live current-main repro because this review is read-only.

## Solution Assessment

Yes, strict origin-form path matching before rate limits, body parsing, signature validation, and dispatch is the right Feishu plugin boundary fix. The remaining decision is whether maintainers accept the upgrade behavior; importing the helper through `webhook-ingress` instead of the deprecated alias would be a minor cleanup, not a correctness blocker.
