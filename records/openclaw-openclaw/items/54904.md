## Summary

Keep open: this is a real Feishu webhook bug fix with good proof, but the current diff still rejects safe trailing-slash callback paths after normalizing the configured path, so it needs a narrow repair before merge.

## What This Changes

The PR adds Feishu webhook request-path enforcement before rate limiting, body reading, signature validation, and dispatch, plus Feishu monitor transport regression tests.

## Best Possible Solution

Keep the fail-closed Feishu path gate, but compare the configured and incoming origin-form paths through the same trailing-slash normalization after rejecting malformed, network-path, and dot-segment-bypass targets, with regression coverage for those cases.

## Reproduction Assessment

Yes. Current main source shows `monitorWebhook` reads `webhookPath` but never compares the incoming request URL before dispatch; the PR-head trailing-slash regression is also source-reproducible because the configured path is normalized while the incoming path is returned raw.

## Solution Assessment

No, not as currently written. The owner-boundary fix is right, but the best fix is to normalize the safe extracted incoming pathname the same way as the configured path while preserving the malformed/network-path/dot-segment rejection tests.
