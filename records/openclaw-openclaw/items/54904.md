## Summary

Keep this PR open: current main still does not enforce the configured Feishu webhook path, and the PR is a focused security/behavior fix with sufficient copied live-output proof and focused tests. The remaining maintainer decision is upgrade safety because deployments with mismatched Feishu webhook URLs or reverse-proxy paths will now fail closed with 404.

## What This Changes

The PR normalizes and enforces Feishu `webhookPath` before request guards, signature verification, body parsing, and dispatch, and adds transport tests for matching, mismatch, normalized, network-path, and malformed request targets.

## Best Possible Solution

Land the fail-closed path guard after maintainer acceptance of the compatibility impact, with release-note or squash-message context that Feishu webhook URLs must match the configured normalized path.

## Reproduction Assessment

Yes from source inspection. Current main reads `webhookPath` but never compares `req.url` before signature verification and dispatch, so a signed request to another path on the same listener can still reach the Feishu dispatcher.

## Solution Assessment

Yes, this is the narrowest maintainable fix for the reported bug: enforce the existing Feishu-owned route path at the transport entry point before body parsing, signature checks, rate limiting, and dispatch. The main alternative would be migrating Feishu to the shared plugin HTTP route registry, but that is a broader refactor than this security bug requires.
