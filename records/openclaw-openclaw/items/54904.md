## Label Justifications

- `P2`: This is a bounded Feishu channel security and routing fix with normal priority and limited blast radius.
- `merge-risk: 🚨 compatibility`: The PR intentionally changes shipped Feishu webhook behavior from accepting any signed path on the listener to rejecting non-configured paths before signature validation.
- `merge-risk: 🚨 message-delivery`: A Feishu callback path or proxy rewrite mismatch will now stop inbound webhook events before dispatch, which can present as message loss.

## Summary

Keep open: this PR fixes a real Feishu webhook routing bug that is still present on current main, but it intentionally turns `webhookPath` into a fail-closed delivery boundary and should get maintainer sign-off for upgrade behavior before merge.

## What This Changes

The PR normalizes Feishu `webhookPath`, rejects malformed or non-matching webhook request targets before signature verification, and adds Feishu transport regression tests.

## Best Possible Solution

Merge after maintainers explicitly accept strict normalized Feishu `webhookPath` as the upgrade behavior; otherwise require a documented migration or opt-in compatibility design first.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.1 create the Feishu webhook server without comparing `req.url` to the configured `webhookPath`, so signed requests on other paths can still reach the dispatch pipeline. I did not run a live Feishu server in this read-only review.

## Solution Assessment

Yes, with maintainer sign-off on the upgrade behavior: enforcing the normalized configured path before signature verification is the narrow fix for the documented route setting. Preserving arbitrary-path dispatch would keep `webhookPath` ineffective, while moving Feishu to the shared plugin route pipeline would be a broader refactor than this bug requires.
