## Summary

Keep open: the PR is still needed because current main and the latest release do not enforce Feishu `webhookPath`, but the current branch can reject valid trailing-slash webhook deliveries because it normalizes the configured path and compares it to a raw request pathname.

## What This Changes

The PR adds early Feishu webhook request-path validation before signature processing and adds monitor transport regression coverage.

## Best Possible Solution

Land the Feishu-owned path gate after raw-target safety checks and request/config path normalization share the same no-trailing-slash webhook contract, with a trailing-slash request regression test.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.8` shows `monitorWebhook` reads `webhookPath` but never checks `req.url` before signature validation and dispatch.

## Solution Assessment

No, not as currently written. The owner boundary and early raw-target rejection are right, but the incoming path should be normalized with the same SDK webhook path contract as the configured path after crafted targets are rejected.
