## Summary

Keep open: current main still does not enforce Feishu `webhookPath`, and this PR is a focused, correct fix with sufficient real terminal proof; the main maintainer-visible concern is the intentional fail-closed compatibility/message-delivery behavior for misconfigured Feishu callback URLs.

## What This Changes

This PR adds an early Feishu webhook request-path gate, normalizes the configured `webhookPath`, rejects malformed or mismatched paths before signature/dispatch handling, and adds targeted HTTP regression tests.

## Best Possible Solution

Land the Feishu-owned path gate after maintainer acceptance of the fail-closed upgrade behavior, keeping the existing `webhookPath` config surface and focused regression tests.

## Reproduction Assessment

Yes. Current main reads `account.config.webhookPath` but never compares `req.url` before rate limiting, body reading, signature validation, and `eventDispatcher.invoke`, so the arbitrary-path behavior is source-reproducible.

## Solution Assessment

Yes. The PR fixes the bug in the Feishu transport owner, reuses the existing SDK path normalizer for configured paths, avoids URL canonicalization for crafted request targets, and adds regression tests for the central bypass cases.
