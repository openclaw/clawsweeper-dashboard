## Summary

Keep open: the PR is a useful Feishu ingress-boundary fix, but the latest diff is not merge-ready because it normalizes the configured webhook path while comparing the incoming request path raw, which can reject existing trailing-slash webhook URLs after upgrade.

## What This Changes

The PR adds early Feishu webhook request-path validation, normalizes the configured `webhookPath`, and adds monitor transport tests for wrong paths, crafted targets, config normalization, and malformed request targets.

## Best Possible Solution

Land the Feishu-owned path gate after raw-target safety checks and request/config path normalization share the same no-trailing-slash contract, with a trailing-slash request regression test.

## Reproduction Assessment

Yes. Source inspection on current main shows `monitorWebhook` never checks the incoming request path before signed payloads can reach dispatch; the PR tests also encode the failing wrong-path behavior.

## Solution Assessment

No, not as currently written. The owner boundary and early rejection are right, but the request path should be normalized with the same webhook path contract as the configured path after rejecting crafted request targets.
