## Summary

Current `main` and the latest release still lack Feishu path enforcement, while the PR head adds the right early guard and regression coverage. The remaining blockers are merge/readiness issues rather than a reason to close: missing real behavior proof, a dirty/conflicting branch, and one small SDK subpath cleanup.

## What This Changes

The PR normalizes Feishu webhook paths, rejects requests whose path does not match the configured `webhookPath` before the webhook pipeline, and adds focused transport tests for wrong-path, configured-path, normalized-path, and malformed-target cases.

## Best Possible Solution

Update the branch, switch the new import to `openclaw/plugin-sdk/webhook-ingress`, add redacted real curl/gateway/log proof, and land the strict Feishu path guard with maintainers accepting the documented-route compatibility change.

## Reproduction Assessment

Yes, at source level. Current main shows `monitorWebhook` never compares `req.url` or URL pathname to `webhookPath` before body parsing, signature validation, challenge handling, and dispatch; I did not run a live request because this review is read-only.

## Solution Assessment

Yes for the code direction. Early Feishu-owned path normalization and mismatch rejection before the existing webhook pipeline is the narrow maintainable fix, but merge should wait for proof, conflict resolution, and the SDK subpath cleanup.
