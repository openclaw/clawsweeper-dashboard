## Summary

Keep this PR open: current main and the latest release still do not enforce the Feishu webhook path, and the branch is a narrow, source-backed fix for that bug. It still needs real behavior proof and maintainer acceptance of the stricter fail-closed upgrade behavior before merge.

## What This Changes

The PR changes the Feishu webhook transport to normalize and enforce the configured webhook path before parsing, signature checks, rate limiting, and dispatch, with focused transport tests for accepted and rejected request targets.

## Best Possible Solution

Land a Feishu-local early ingress guard after real behavior proof is added and maintainers accept the stricter upgrade behavior for existing webhook deployments.

## Reproduction Assessment

Yes from source, but not from a live run in this review. Current main and v2026.5.28 read webhookPath without checking req.url before dispatcher invocation, so a correctly signed request on another path can reach the handler path.

## Solution Assessment

Yes, this is the best code direction: enforce the Feishu-owned route boundary before pre-auth body work and dispatch, while preserving normalized legacy path spelling. A broader shared-route refactor would be larger than needed; the remaining blocker is proof and upgrade-risk acceptance, not a different fix layer.
