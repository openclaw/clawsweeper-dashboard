## Summary

The PR is still needed because current main and the latest release do not enforce the Feishu webhook request path, but it should not merge yet because the current patch compares the incoming safe path verbatim against a normalized configured path and can reject valid trailing-slash callbacks.

## What This Changes

The PR adds a Feishu webhook path gate before rate limiting, body parsing, signature checks, and dispatch, plus monitor transport tests for wrong paths, network-path targets, malformed targets, dot segments, and configured-path normalization.

## Best Possible Solution

Land the Feishu-owned early path gate after normalizing safe incoming origin-form paths before comparison and adding trailing-slash coverage while preserving malformed, network-path, and dot-segment rejection.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.6.10 never compare the incoming request path to `webhookPath` before body/signature handling, and PR head would compare `/feishu/events/` unequal to normalized `/feishu/events`.

## Solution Assessment

No. Enforcing Feishu routing before body and signature work is the right layer, but the branch should normalize safe incoming origin-form paths before comparison so same-route trailing-slash callbacks are not broken.
