## Summary

Keep open: this is the active Feishu-owned fix lane, but PR head still compares a raw incoming request path to a normalized configured path, so valid trailing-slash webhook URLs can be rejected after merge.

## What This Changes

The branch adds early Feishu webhook request-path enforcement, normalizes the configured webhookPath, and adds monitor transport regression tests for wrong paths, crafted targets, malformed targets, and configured-path normalization.

## Best Possible Solution

Land the Feishu path gate after the stripped origin-form request path and configured webhookPath share the same normalizeWebhookPath contract, while preserving the explicit rejection of absolute-form, network-path, and dot-segment bypass targets.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 do not enforce webhookPath, and PR-head source shows a concrete trailing-slash regression path where config `/expected-feishu-hook/` normalizes but request `/expected-feishu-hook/` remains raw.

## Solution Assessment

No, not yet. The owner boundary and early path gate are the right fix shape, but the incoming origin-form pathname needs the same SDK normalization as the configured path after crafted targets are rejected.
