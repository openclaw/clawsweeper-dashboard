## Summary

Keep open: the PR fixes the right Feishu webhook boundary, but the latest head still compares a normalized configured path to a raw incoming path, so matching trailing-slash provider URLs can start returning 404 after merge.

## What This Changes

## Best Possible Solution

Land the Feishu-owned path gate after raw-target safety checks and request/config path normalization share the same no-trailing-slash contract, with a trailing-slash request regression test.

## Solution Assessment

No, not as currently written. The owner boundary and early rejection are right, but the request path should be normalized with the same webhook path contract as the configured path after rejecting crafted request targets.
