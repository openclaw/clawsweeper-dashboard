## Summary

This PR should stay open: current main still lacks a first-party `/restore` command, but the branch has correctness and transcript-confidentiality blockers plus missing after-fix real behavior proof.

## What This Changes

The PR adds a built-in `/restore [number]` command, session archive listing/restoration helpers, `sessionKey` transcript headers, docs, and restore-focused tests.

## Best Possible Solution

Revise the restore design to use the current reset-hook seam, restore into `resetResult.entry.sessionFile`, fail closed or explicitly define legacy archive ownership, and prove the real list/restore flow before merge.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR diff gives a high-confidence path for the blockers: stale reset-hook import, recomputed restore target instead of the stored reset entry path, and fail-open archive ownership filtering.

## Solution Assessment

No. A restore command may be useful, but this implementation is not the narrow maintainable solution until reset owns the target transcript path and archive ownership has a safe policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Import reset hooks from the current seam:** `src/auto-reply/reply/commands-restore.ts:4`
  - body: Current main exports and uses `emitResetCommandHooks` from `commands-reset-hooks.ts`, while this new handler imports it from `commands-core.js`. After rebasing, the restore handler will fail to compile or load until it uses the current reset-hook module.
  - confidence: 0.9
- **[P1] Restore into the reset entry session file:** `src/gateway/session-restore.ts:248-252`
  - body: `performGatewaySessionReset` persists the active transcript path on `resetResult.entry.sessionFile`. Recomputing the destination from the new session id can move the archive to a file the session store will not read, so `/restore` can report success while the next turn uses the empty reset transcript.
  - confidence: 0.94
- **[P1] Fail closed on unreadable or unowned archives:** `src/gateway/session-restore.ts:198-202`
  - body: `readSessionKeyFromHeader` returns `null` for missing keys, parse failures, and read errors, and this branch treats `null` as visible to any requested `sessionKey`. That can list and restore another channel's archived transcript instead of filtering by ownership.
  - confidence: 0.93
- **[P2] Filter archives before enforcing the scan ceiling:** `src/gateway/session-restore.ts:193`
  - body: The scan ceiling is applied before session-key filtering, so newer archives from other channels can consume the whole window and make valid current-channel archives disappear. Bound total work if needed, but collect matches after ownership filtering.
  - confidence: 0.86
- **[P2] Avoid firing reset hooks twice:** `src/auto-reply/reply/commands-restore.ts:103-116`
  - body: The handler emits reset command hooks and then calls `performSessionRestore`, whose `performGatewaySessionReset` path triggers the internal command hook again. Hook consumers can see duplicate side effects for one `/restore` invocation.
  - confidence: 0.84

## Security Review

Status: needs_attention
