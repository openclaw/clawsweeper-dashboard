## Solution Assessment

No. A restore command is useful, but this branch is not the safest maintainable solution until archive ownership fails closed, restore uses the canonical reset transcript path, and the broader archive/snapshot policy is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail closed when archive ownership is unknown:** `src/gateway/session-restore.ts:198-201`
  - body: `readSessionKeyFromHeader` collapses missing keys, parse failures, truncated headers, and read errors to `null`, and this branch treats `null` as a match for every session key. In a multi-channel setup, `/restore` can list or restore another channel's keyless or malformed archived transcript, so unknown ownership needs to be skipped or handled through an explicit, safe legacy migration path.
  - confidence: 0.93
- **[P1] Restore into the reset entry's session file:** `src/gateway/session-restore.ts:246-252`
  - body: After `performGatewaySessionReset` returns, the session store already points at `resetResult.entry.sessionFile`, but this code derives a separate path from the new session id and topic id. When reset preserves or rewrites the transcript file differently, `/restore` can move the archive into a file the next turn will not read, making the command report success while the session starts from the reset-created transcript.
  - confidence: 0.88
- **[P2] Avoid running reset hooks twice:** `src/auto-reply/reply/commands-restore.ts:103-113`
  - body: `emitResetCommandHooks` triggers the command reset hook path here, then `performSessionRestore` immediately calls `performGatewaySessionReset`, which triggers the gateway reset hook path for the same action. Hook users can see duplicate side effects, and the fire-and-forget `before_reset` reader can race the archive move during restore.
  - confidence: 0.82
