## Summary

Keep open. The PR still adds a useful built-in restore command, but the current branch has blocking build, session-file, and transcript-confidentiality defects before it is safe to merge.

## What This Changes

This PR adds a `/restore [number]` chat command, archive listing/restoration helpers, transcript `sessionKey` headers, docs, and restore-focused tests.

## Best Possible Solution

Revise the branch so restore fails closed on archive ownership, restores into `resetResult.entry.sessionFile`, uses one reset/hook owner, scans until enough matching archives are found, and adds the required changelog plus focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the PR imports a symbol current main does not export, keyless/unreadable archives pass the sessionKey filter, and preserved custom `sessionFile` paths are ignored during restore.

## Solution Assessment

No. A built-in restore command is reasonable, but this implementation is not the best merge path until it reuses the reset service's persisted transcript path and defines a fail-closed archive ownership contract.

  - confidence: 0.94

  - body: `readSessionKeyFromHeader` returns `null` for missing keys, parse failures, truncation, and read errors, and this condition treats `null` as visible to any sessionKey.   - confidence: 0.93

  - confidence: 0.86

  - body: This handler emits reset command hooks, then `performSessionRestore` calls `performGatewaySessionReset`, which emits the command hook for the reset path again. Hook consumers can see duplicate side effects; restore should route through one reset owner.
  - confidence: 0.84
