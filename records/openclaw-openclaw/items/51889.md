## Summary

Keep open. Current main still has no `/restore` command, and this PR is a real implementation candidate, but the branch has blocking correctness and security issues around the restored transcript destination, archive ownership, and filtered archive listing. The latest checks are green, but they do not cover these source-level gaps.

## What This Changes

The branch adds a built-in `/restore [number]` command, archived transcript listing/restoration helpers, `sessionKey` transcript headers, docs, and focused tests for restoring archived session transcripts.

## Best Possible Solution

Keep the feature in core only if maintainers want a built-in recovery command, but revise the branch so restore moves the archive into the exact `resetResult.entry.sessionFile`, enforces explicit archive ownership before listing or restoring, applies bounded scanning after filtering, lets one reset path own hook emission, and adds tests plus the required docs/changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create a reset entry with a preserved custom `sessionFile` and the PR moves the archive to a different synthesized path, or create keyless/corrupt/long-header archives from another sessionKey and the listing filter admits them.

## Solution Assessment

No. A `/restore` command is a reasonable feature direction, but this implementation is not the best fix until it reuses the reset service's persisted transcript path and defines a safe archive ownership contract for legacy files.

  - confidence: 0.92
