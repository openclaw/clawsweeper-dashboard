## Summary

Keep open: current main still lacks the proposed `/restore` command, but this active feature PR has correctness and transcript-confidentiality blockers against current main plus no real after-fix behavior proof.

## What This Changes

The PR adds a built-in `/restore [number]` command, archive listing/restoration helpers, `sessionKey` transcript headers, docs, and restore-focused tests.

## Best Possible Solution

Revise the branch to use the current reset-hook seam, restore into `resetResult.entry.sessionFile`, define a fail-closed or explicitly approved legacy archive-ownership policy, filter before applying scan limits, and add real behavior proof.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: the new handler imports a seam removed from current main, the restore helper recomputes a transcript path instead of using the reset result, and the archive filter treats keyless or unreadable headers as visible to any session key.

## Solution Assessment

No. A built-in restore command may be a good direction, but this implementation is not the narrow maintainable solution until reset owns the target path and archive ownership fails closed or has an explicit legacy policy.

Full review comments:

- **[P1] Import reset hooks from the current seam:** `src/auto-reply/reply/commands-restore.ts:4`
  - body: Current main no longer re-exports `emitResetCommandHooks` from `commands-core.ts`; the hook seam is `commands-reset-hooks.ts`. After rebasing, this new handler imports a non-exported symbol and will fail to compile or load.
  - confidence: 0.9
- **[P1] Restore into the reset entry's session file:** `src/gateway/session-restore.ts:248-252`
  - body: `performGatewaySessionReset` persists the active transcript path in `resetResult.entry.sessionFile` and can preserve a custom path. Recomputing a path from `newSessionId` can move the archive to a file the session store will not read, so restore can report success while the next turn uses the fresh reset transcript.
  - confidence: 0.94
- **[P1] Fail closed on unknown archive ownership:** `src/gateway/session-restore.ts:198-202`
  - body: `readSessionKeyFromHeader` returns `null` for missing keys, parse failures, and read errors, and this branch treats `null` as visible to every requested `sessionKey`. That can list and restore another channel's archived transcript.
  - confidence: 0.93
- **[P2] Scan until enough matching archives are found:** `src/gateway/session-restore.ts:193`
  - body: The scan cap is applied before session-key filtering, so newer archives from other channels can consume the whole window and make valid current-channel archives disappear. Bound total work if needed, but collect matches after ownership filtering.
  - confidence: 0.86
- **[P2] Use one reset hook owner:** `src/auto-reply/reply/commands-restore.ts:104-113`
  - body: The handler emits reset command hooks and then calls `performSessionRestore`, which calls `performGatewaySessionReset` and triggers the internal command hook again. Hook consumers can see duplicate side effects for one `/restore` invocation.
  - confidence: 0.84
