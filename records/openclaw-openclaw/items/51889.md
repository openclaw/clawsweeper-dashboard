## Summary

Keep open. Current main still lacks a built-in `/restore` command, but this PR is not merge-ready because the latest head still crosses the archived transcript ownership boundary, restores into a recomputed path instead of the reset service's persisted transcript path, and needs current-main rebase cleanup plus changelog coverage.

## What This Changes

The PR adds a built-in `/restore [number]` command that lists and restores archived session transcripts, threads `sessionKey` into transcript headers, documents the command, and adds restore-focused tests.

## Best Possible Solution

Revise the branch so restore fails closed on archive ownership, restores into `resetResult.entry.sessionFile`, uses one reset/hook owner, scans until enough matching archives are found, rebases the hook import onto current main, and adds the required changelog plus focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a keyless or unreadable archive from another channel passes the null-sessionKey filter, and an active session with a preserved custom `sessionFile` will have its restored archive moved to a different synthesized path.

## Solution Assessment

No. A built-in restore command may be reasonable, but this implementation is not the narrowest maintainable solution until it reuses the reset service's persisted path and defines a fail-closed archive ownership contract.
