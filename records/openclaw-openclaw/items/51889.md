## Summary

Keep this PR open because current main still has no `/restore` command, but the PR is not mergeable as-is: the branch has a broken reset-hook import and restore/session-key boundary bugs that can break the feature or expose archives across channels.

## What This Changes

Adds a `/restore` slash command that lists archived session transcripts, restores a selected archive after a session reset, threads `sessionKey` into transcript headers, and documents/tests the flow.

## Best Possible Solution

Land the restore feature only after the branch compiles, restores into the reset entry's persisted session file, and enforces session-key privacy for archive listing/restoration with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source review gives a high-confidence path for the PR blockers: the new handler imports a non-exported reset hook helper, restore ignores the reset entry's persisted `sessionFile`, and keyless archives are accepted for any `sessionKey`.

## Solution Assessment

No. The proposed direction is useful, but the current implementation should import the hook helper from the reset-hook module, restore to `resetResult.entry.sessionFile`, and reject or explicitly gate archives without a matching session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Import reset hooks from the reset hook module:** `src/auto-reply/reply/commands-restore.ts:4`
  - body: `commands-core.js` on current main does not export `emitResetCommandHooks`; the helper is exported from `commands-reset-hooks.js` and the runtime barrel. As written, adding this handler will fail typecheck/module loading before `/restore` can run.
  - confidence: 0.95
- **[P1] Restore into the reset entry's session file:** `src/gateway/session-restore.ts:248-252`
  - body: After reset, `performGatewaySessionReset` may preserve `resetResult.entry.sessionFile` from the current session entry. Recomputing the target from `newSessionId` can move the archive into a different file while the store still points at the fresh reset transcript, so the next turn will not see the restored history.
  - confidence: 0.91
- **[P1] Reject archives without a matching session key:** `src/gateway/session-restore.ts:201`
  - body: When `readSessionKeyFromHeader` returns `null`, this condition lets the archive match every requested `sessionKey`. Current main-created transcripts and reset headers are keyless, so an authorized user in one channel can be shown and restore another channel's archived transcript.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete session privacy concern by treating keyless archive headers as matches for every channel/session key.

Concerns:

- **[high] Keyless archives can cross session boundaries:** `src/gateway/session-restore.ts:201`
  - body: `listRecentArchives` accepts `null` header keys for any requested `sessionKey`; because existing archives can be keyless, `/restore` can list and restore transcripts from unrelated channels. This exposes persisted conversation history across session boundaries.
  - confidence: 0.9
