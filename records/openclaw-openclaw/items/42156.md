## Label Justifications

- `P2`: This is a reproducible, focused Telegram interaction bug with visible user impact but limited channel-specific blast radius.
- `impact:message-loss`: A queued callback can expire before acknowledgement, leaving the user’s inline-button action effectively suppressed.

## Summary

Current `main` still acknowledges Telegram callback queries inside the callback handler after per-chat/topic sequentialization, so a busy agent turn can outlive Telegram’s acknowledgement window. Multiple focused PRs attempted the same narrow fix; the latest was closed unmerged because it lacked real Telegram proof, not because current `main` solved the bug.

## What This Changes

Ensure Telegram callback queries are acknowledged before per-chat sequentialization so inline-button presses do not time out behind long agent turns.

## Best Possible Solution

Restore the narrow early-acknowledgement patch from the latest attempt: acknowledge valid callback queries before `sequentialize`, mark successful acknowledgement so the downstream handler does not repeat it, keep all callback action processing ordered and authorized, and prove the spinner clears in a real Telegram session.

## Reproduction Assessment

Yes at source level: current `main` installs per-chat/topic sequentialization before the callback handler, and that handler performs `answerCallbackQuery`, so a long-running same-key turn can delay acknowledgement past Telegram’s deadline. This read-only review did not run a live Telegram reproduction.

## Solution Assessment

Yes, the best fix is the narrow middleware-order change from the latest PR: move only the empty acknowledgement ahead of sequentialization while preserving downstream authorization, dedupe, and ordered callback processing. It still requires real Telegram proof before landing.
