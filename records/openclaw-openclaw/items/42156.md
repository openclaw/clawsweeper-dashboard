## Label Justifications

- `P2`: This is a normal-priority Telegram channel bug with a clear source-level cause and limited blast radius.
- `impact:message-loss`: Queued callback acknowledgements can expire before OpenClaw answers, making inline-button interactions appear lost to the Telegram user.

## Summary

Keep open: current main and v2026.5.22 still acknowledge ordinary Telegram callback queries inside the downstream callback handler after `sequentialize`, so a callback can still wait behind an active same-chat/topic agent turn. The shipped approval-lane fix only covers approval payloads, and the open callback-toast work tracks different UX text behavior rather than this timing bug.

## What This Changes

Telegram inline-button callback acknowledgements should clear before callback work is queued behind the Telegram sequentialize lane for an active chat/topic turn.

## Best Possible Solution

Add a pre-sequentialize callback acknowledgement path after accepted-update gating, keep callback routing/action handling in the ordered handler, and prove it with focused tests plus redacted live Telegram callback proof.

## Reproduction Assessment

Yes, at source level: ordinary callback data maps to the normal chat/topic sequential key, grammY waits on same-key middleware chains, and the current acknowledgement call runs inside the downstream callback handler. I did not run a live Telegram client repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to move only callback acknowledgement timing before `sequentialize` after accepted-update gating. Callback routing, payload handling, and user-visible action execution should stay in the existing ordered handler.
