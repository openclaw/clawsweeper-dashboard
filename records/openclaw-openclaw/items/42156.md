## Summary

Current main and the latest release still acknowledge regular Telegram callback queries only inside the queued callback handler, after the per-chat/topic sequentialize middleware, so the timeout path remains a valid narrow Telegram plugin bug.

## What This Changes

Acknowledge Telegram inline-button callback queries immediately when they arrive instead of waiting behind an active sequentialized agent turn.

## Best Possible Solution

Move only the callback acknowledgement into Telegram middleware after accepted-update dedupe and before sequentialize, while leaving callback action routing ordered and covered by regression tests.

## Reproduction Assessment

Yes, at source level. Current main queues regular callback-query handling behind the same chat/topic sequential key as an active turn, and the only acknowledgement call is inside that queued handler; I did not run a live Telegram client reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to acknowledge accepted callback queries before sequentialize, keep the actual callback action in the existing ordered handler, and incorporate the logging/rate-limit concerns from the closed prior PR.
