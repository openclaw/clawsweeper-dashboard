## Summary

Keep open. Current main and the latest release still acknowledge ordinary Telegram callback queries only inside the downstream callback handler after `sequentialize`, while the prior timing-fix PRs were closed unmerged and the open related callback-ack PR addresses acknowledgement text rather than this queueing path.

## What This Changes

Telegram inline button callbacks should be acknowledged before same-chat/topic sequentialization so button spinners clear while another agent turn is running.

## Best Possible Solution

Add an accepted-update, pre-sequentialize acknowledgement path for callback queries, keep callback action routing ordered in the existing handler, and prove it with focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level. A same-chat/topic callback reaches `answerCallbackQuery` only after `sequentialize` calls the downstream callback handler, and `@grammyjs/runner` waits for the previous same-key task before calling `next()`.

## Solution Assessment

Yes. The narrow maintainable fix is to acknowledge accepted callback updates before `sequentialize` while leaving routing, authorization, dedupe, sanitized logging, and throttling behavior intact.

## Repair Work Prompt

Fix Telegram callback query acknowledgement timing for https://github.com/openclaw/openclaw/issues/42156. Current main accepts updates, then registers `sequentialize(getTelegramSequentialKey)`, and only later calls `answerCallbackQuery` inside `bot.on("callback_query")`; ordinary callbacks can therefore miss Telegram's acknowledgement window while a same-chat/topic turn is queued. Add a pre-sequentialize acknowledgement path for accepted callback updates so `answerCallbackQuery` fires immediately, while leaving callback action routing, authorization, native/plugin/model callback handling, and callback data semantics ordered in the existing handler. Do not add new config. Reuse `withTelegramApiErrorLogging`/redaction and the existing Telegram API throttler. Avoid consuming `shouldSkipHandlerDispatch` from early acknowledgement code unless the tracker exposes a non-consuming accepted-update check; preserve update dedupe/watermark behavior and skip duplicate/replayed updates. Add focused regression coverage proving acknowledgement happens before a blocked same-key sequentialized handler, non-callback updates pass through, errors do not block `next()`, duplicate acknowledgement is not emitted, and ordinary callbacks still route normally. Add a `CHANGELOG.md` fix entry because this is user-visible Telegram behavior.
