## Label Justifications

- `P2`: This is a normal-priority Telegram channel bug with a clear source-level cause and limited blast radius.
- `impact:message-loss`: Queued callback acknowledgements can expire before OpenClaw answers, making inline-button interactions appear lost to Telegram users.

## Summary

Keep open: current main and the latest release still register Telegram `sequentialize(getTelegramSequentialKey)` before the callback handler, and ordinary callback acknowledgements are still sent inside that downstream handler. The closed exact-fix PRs were not merged, while the open callback-toast PR tracks separate feedback text UX rather than this timing bug.

## What This Changes

The issue asks for Telegram inline callback queries to be acknowledged before grammY sequentialization can queue them behind an active same-chat/topic agent turn.

## Best Possible Solution

Add a pre-sequentialize callback acknowledgement path after accepted-update gating, leave callback action routing in the ordered handler, and cover both the timing path and existing callback behavior with focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level: ordinary callback data maps to the normal chat/topic sequential key, grammY waits on same-key middleware chains, and current main acknowledges callbacks only inside the downstream callback handler. I did not run a live Telegram client reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to move only callback acknowledgement timing before sequentialize after accepted-update gating. Callback routing, payload handling, and user-visible action execution should stay in the existing ordered handler.
