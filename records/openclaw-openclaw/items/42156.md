## Label Justifications

- `P2`: This is a normal-priority Telegram channel bug with a clear source-level cause and limited blast radius.
- `impact:message-loss`: Queued callback acknowledgements can expire before OpenClaw answers, making inline-button interactions appear lost to the Telegram user.

## Summary

Keep open: current main and the latest release still answer ordinary Telegram callback queries inside the callback handler after `sequentialize`, so inline-button acknowledgements can still wait behind an active same-chat/topic agent turn. The shipped approval-lane work only covers approval payloads, and the open callback toast PR tracks different feedback text UX.

## What This Changes

Fix Telegram inline-button callback acknowledgement timing so `answerCallbackQuery` clears the button before same-chat/topic sequentialization can delay the handler.

## Best Possible Solution

Add a pre-sequentialize acknowledgement path after accepted-update gating, keep callback action routing in the ordered handler, and prove it with focused tests plus redacted live Telegram callback proof.

## Reproduction Assessment

Yes, at source level: ordinary callback data maps to the normal chat/topic sequential key, grammY waits on same-key middleware chains, and the current acknowledgement call runs inside the downstream callback handler. I did not run a live Telegram client repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to move only callback acknowledgement timing before `sequentialize` after accepted-update gating. Callback routing, payload handling, and user-visible action execution should stay in the existing ordered handler.
