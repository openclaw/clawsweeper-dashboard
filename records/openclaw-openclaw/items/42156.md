## Summary

Keep open: current main and the latest release still acknowledge ordinary Telegram callback queries inside the downstream callback handler after `sequentialize`, so inline-button callbacks can still wait behind an active same-chat/topic turn. The merged approval-lane fix only covers approval payloads, and the open callback-toast work is a different UX/config request.

## What This Changes

The issue reports Telegram inline-button callback acknowledgements timing out because `answerCallbackQuery` waits behind same chat/topic `sequentialize` when an agent turn is already active.

## Best Possible Solution

Add an accepted-update-gated pre-sequentialize callback acknowledgement path, keep callback routing/action handling in the ordered handler, and prove it with focused tests plus redacted live Telegram callback proof.

## Reproduction Assessment

Yes, at source level: ordinary callback data maps to the normal chat/topic sequential key, grammY waits on same-key middleware chains, and the current acknowledgement call runs inside the downstream callback handler. I did not run a live Telegram client repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to move only callback acknowledgement timing before `sequentialize` after accepted-update gating. Callback routing, payload handling, and user-visible action execution should stay in the existing ordered handler.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/42156 by moving Telegram callback-query acknowledgement timing before `sequentialize` while keeping callback action routing ordered. The observable bug is that ordinary inline-button callbacks still call `answerCallbackQuery` inside `bot.on("callback_query")`, downstream of `botRuntime.sequentialize(getTelegramSequentialKey)`, so same-chat/topic callbacks can time out behind an active agent turn. Add a pre-sequentialize acknowledgement path after accepted-update gating/dedupe, make the acknowledgement non-blocking or safely error-logged, remove or avoid duplicate handler-level acknowledgement, and preserve authorization, accepted-update skipping, callback payload routing, and normal chat/topic ordering for the actual action. Do not add a new config option, do not change approval callback lane behavior from https://github.com/openclaw/openclaw/pull/64979, and do not solve the toast-text request tracked by https://github.com/openclaw/openclaw/issues/76622 / https://github.com/openclaw/openclaw/pull/76975. Add focused regression coverage showing a callback acknowledgement can run before a busy same-key sequentialized turn, plus existing callback route tests still pass. Add a changelog fix entry if the PR is user-visible. Validation should include the focused Telegram Vitest files, oxfmt for touched files, changed checks in Testbox/Crabbox when promoted, and redacted live Telegram proof showing an inline-button spinner clears while a same-chat/topic agent turn is busy.
