## Label Justifications

- `P2`: This is a normal-priority Telegram channel bug with a clear source-level cause and limited blast radius.
- `impact:message-loss`: Queued callback acknowledgements can expire before OpenClaw answers, making inline-button interactions appear lost to the Telegram user.

## Summary

Keep open: current main and v2026.5.22 still answer ordinary Telegram callback queries inside the downstream callback handler after `sequentialize`, so non-approval inline button presses can still wait behind an active same-chat/topic agent turn. The merged approval-lane fix only covers approval payloads, and the open callback-toast work tracks different acknowledgement text behavior rather than this timing bug.

## What This Changes

The issue requests moving Telegram callback query acknowledgement ahead of grammY `sequentialize` so inline button presses clear promptly even while same-chat/topic agent turns are busy.

## Best Possible Solution

Add a pre-sequentialize callback acknowledgement path after accepted-update gating, keep callback routing/action handling in the ordered handler, and prove it with focused tests plus redacted live Telegram callback proof.

## Reproduction Assessment

Yes, at source level: ordinary callback data maps to the normal chat/topic sequential key, grammY waits on same-key middleware chains, and the current acknowledgement call runs inside the downstream callback handler. I did not run a live Telegram client repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to move only callback acknowledgement timing before `sequentialize` after accepted-update gating. Callback routing, payload handling, and user-visible action execution should stay in the existing ordered handler.

## Repair Work Prompt

Fix the Telegram callback acknowledgement timing bug from https://github.com/openclaw/openclaw/issues/42156. Current main registers `sequentialize(getTelegramSequentialKey)` before callback handlers, and ordinary non-approval callback queries use the normal chat/topic key, so `answerCallbackQuery` inside `bot.on("callback_query")` can run too late while an agent turn is busy. Add a pre-sequentialize acknowledgement path after accepted-update gating so Telegram callback spinners clear immediately, but keep callback routing, authorization, payload handling, and agent/action execution in the existing ordered handler. Preserve the approval-specific lane from https://github.com/openclaw/openclaw/pull/64979 and do not solve the separate callback acknowledgement text issue from https://github.com/openclaw/openclaw/issues/76622. Add focused regression coverage for ordinary callbacks queued behind a busy same-chat/topic lane, update any stale handler-level acknowledgement assertions, add a single `CHANGELOG.md` fix entry because this is user-visible, and include redacted live Telegram or equivalent bot-to-bot proof in the PR.
