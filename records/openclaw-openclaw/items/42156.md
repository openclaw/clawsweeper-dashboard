## Summary

Current main still answers Telegram callback queries inside the `callback_query` handler, which is registered after the per-chat/topic `sequentialize` middleware, so the reported timeout path remains source-reproducible and should stay open.

## What This Changes

Telegram inline-button callback acknowledgements should be sent before same-chat/topic update serialization so a long-running agent turn cannot push them past Telegram's answer window.

## Best Possible Solution

Add a small Telegram plugin fix that acknowledges callback queries after accepted-update dedupe but before `sequentialize`, leaves callback routing ordered, and covers the middleware order with a regression test.

## Reproduction Assessment

Yes, from source inspection. Current main queues regular callback-query handling behind the same `sequentialize` key as the active chat/topic turn, and the only `answerCallbackQuery` call is inside that queued handler; I did not run a live Telegram turn in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to move only the acknowledgement into a pre-sequentialize middleware while keeping callback action processing in the existing ordered handler. The earlier closed PRs point in the right direction, but a replacement should address the logging and abuse concerns raised in review.

## Repair Work Prompt

Fix openclaw/openclaw#42156 as a narrow Telegram bug fix. Current main registers `sequentialize(getTelegramSequentialKey)` before `registerTelegramHandlers`, while `answerCallbackQuery` is awaited inside `extensions/telegram/src/bot-handlers.runtime.ts`, so callback queries in the same chat/topic can queue behind a long-running turn. Move the acknowledgement into middleware in `extensions/telegram/src/bot-core.ts` after accepted-update dedupe and before `botRuntime.sequentialize(getTelegramSequentialKey)`, keep callback action routing and authorization in `bot-handlers.runtime.ts`, and remove the redundant handler-level ack. Preserve runtime error logging via `withTelegramApiErrorLogging`, do not add config or product UX options, and account for the abuse concern raised on closed PR #42516 by not bypassing existing dedupe/throttling unnecessarily. Add focused regression coverage in `extensions/telegram/src/bot.create-telegram-bot.test.ts` proving callbacks are answered before the sequentializer, non-callback updates still pass through, and ack failures do not block `next()`. Update `CHANGELOG.md` because this is user-facing.
