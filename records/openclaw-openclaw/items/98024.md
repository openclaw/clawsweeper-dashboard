## Summary

Keep open: the patch makes the later payload smoke check bundle-aware, but the earlier post-core missing-payload collector still treats valid bundle install directories without `package.json` as missing and can send them into repair/disable handling before the new smoke branch runs.

## What This Changes

The PR updates post-update plugin payload smoke validation to accept bundle manifests without `package.json` and persists bundle format metadata in plugin install records.

## Best Possible Solution

Use one bundle-aware payload classification path for both `collectMissingPluginInstallPayloads` and `runPluginPayloadSmokeCheck`, preserving native package precedence and covering explicit plus manifestless bundle records with update-command regression proof.

## Reproduction Assessment

Yes. Source inspection shows `collectMissingPluginInstallPayloads` still reports `missing-package-json` for tracked install directories without `package.json`, and `runPostCorePluginUpdate` calls it before the patched smoke check.

## Solution Assessment

No. This is the right general boundary, but the fix is one-sided; the best fix must make both update-time payload gates bundle-aware while preserving package validation for true npm/code-plugin payloads.
