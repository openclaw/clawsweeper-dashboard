## Summary

This should stay open: current main has the right Ollama vision-capability plumbing in principle, but source inspection shows the `/api/show` request still uses a body key that does not match the current Ollama API contract, which explains why a cloud model can report `vision` to Ollama while OpenClaw falls back to text-only metadata.

## What This Changes

The issue reports that `ollama/qwen3-vl:235b-cloud` is rejected by the image tool and `openclaw infer image describe` as text-only despite Ollama reporting a vision capability and direct `/api/chat` image calls working.

## Best Possible Solution

Update Ollama `/api/show` discovery to use the official request shape, add regression coverage for a cloud vision model returning `capabilities: ["vision", ...]`, and keep the fix inside the Ollama plugin/provider discovery path rather than adding model-specific heuristics or new config.

## Reproduction Assessment

Yes, at source level: current main can still lose `vision` metadata because it posts `{ name: ... }` to `/api/show` while the Ollama contract documents `{ model: ... }`; if that lookup returns no capabilities, the existing code falls back to text-only input. I did not perform a live qwen3-vl cloud run because it requires a signed-in Ollama setup.

## Solution Assessment

Yes. The narrow maintainable fix is to correct the Ollama `/api/show` request and tests, not to special-case `qwen3-vl` or relax the image gate for unknown model metadata.
