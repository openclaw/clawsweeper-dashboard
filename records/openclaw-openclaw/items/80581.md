## Summary

Current main already implements the requested Ollama vision behavior: Ollama `/api/show` capabilities are read into the runtime catalog, `vision` maps to `input: ["text", "image"]`, exact `:cloud` refs can be probed when absent from `/api/tags`, and the Ollama plugin registers an image media-understanding provider.

## What This Changes

The issue requests that Ollama cloud vision models such as `qwen3-vl:235b-cloud` be recognized as image-capable for image inference and the image tool.

## Best Possible Solution

Keep the provider-owned Ollama discovery and media-understanding implementation, and ask for a new config-specific report only if an exact current-main setup still resolves a `/api/show` `vision` model as text-only.

## Reproduction Assessment

Yes from source inspection, but not from a live Ollama cloud setup in this read-only review. The reported failure path is a model resolving without `image` input; current code maps `/api/show` `vision` capability and manually configured image input into the image-capable model contract.

## Solution Assessment

Yes. The maintainable fix is provider-owned Ollama discovery plus media-understanding registration, not a core special case for one model id.
