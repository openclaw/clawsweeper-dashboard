## Summary

Keep this PR open: the narrow `code_mode_exec` shell-policy mapping is still not present on current `main` or the latest release, but the branch is draft/dirty and still lacks accessible exact-head proof that the patched Codex runtime emits `code_mode_exec` and OpenClaw denies it through policy.

## What This Changes

The PR maps Codex Code Mode `code_mode_exec` hook payloads into OpenClaw’s `exec` shell-policy path, persists a native-hook config fingerprint on Codex app-server bindings, and updates focused regression tests.

## Best Possible Solution

Land the narrow alias and binding-fingerprint fix only after the branch is clean and the PR includes accessible redacted exact-head runtime proof; keep the missing-delivery fail-closed guard as the separate path tracked by https://github.com/openclaw/openclaw/pull/90805.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible for the OpenClaw side: current main only aliases `exec_command` to `exec`, so a delivered `code_mode_exec` payload would not enter the shell-policy path. The exact patched Codex emitter is described by local probes but not independently reproduced from accessible proof.

## Solution Assessment

Is this the best way to solve the issue? Yes for the delivered-payload case: normalizing the Codex tool name at the native hook relay boundary is narrower than adding a parallel shell-policy path. It is not a substitute for the missing-delivery fail-closed work tracked by https://github.com/openclaw/openclaw/pull/90805.
