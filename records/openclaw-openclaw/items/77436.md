## Summary

Keep open: the reported Docker regression is plausible on current main because container runtime path resolution can still be steered to host-style `/Users/...` paths through Docker-loaded environment/config, matching the observed `EACCES: permission denied, mkdir '/Users'` failure. No linked maintainer fix or duplicate was present in the provided discussion.

## What This Changes

The issue reports that fresh Docker setup on releases after 2026.04.29 starts the gateway but fails the first Ollama-backed conversation with `EACCES` while trying to create `/Users`.

## Best Possible Solution

Keep the Docker runtime isolated to container-local OpenClaw home/state/config/workspace paths, then add a regression test proving host `/Users/...` overrides in repo `.env` cannot drive container path creation.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a credible reproduction path: Docker loads `.env`, `OPENCLAW_HOME` outranks `HOME`, and workspace/session setup can then try to create a `/Users/...` path inside the non-root container.

## Solution Assessment

Yes, the narrow maintainable fix should be in the Docker setup/compose environment boundary, not in the Ollama provider. The Docker services should explicitly pin or scrub OpenClaw path overrides for container execution and cover that with `src/docker-setup.e2e.test.ts`.
