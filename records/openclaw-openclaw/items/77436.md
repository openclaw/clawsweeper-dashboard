## Summary

Keep this open: current main still has a source-reproducible Docker setup regression where host-side macOS workspace paths can be persisted into container runtime config and then fail on first agent workspace creation.

## What This Changes

The issue reports that a fresh Docker setup works on 2026.04.29 but later versions start the gateway and then fail the first conversation with `EACCES: permission denied, mkdir '/Users'`.

## Best Possible Solution

Keep Docker host bind-mount sources as host paths, but ensure onboarding and gateway runtime inside the container use `/home/node/.openclaw` and `/home/node/.openclaw/workspace` when persisting and resolving OpenClaw config.

## Reproduction Assessment

Yes by source inspection: on macOS, Docker setup can write `/Users/.../.openclaw/workspace` into `.env`, the container imports it during onboarding, and the first reply path tries to create that `/Users` path inside Linux Docker. I did not perform a live run because this review must leave the checkout and host state untouched.

## Solution Assessment

Yes: the narrow fix should be in the Docker setup/Compose boundary, separating host mount variables from container-visible runtime config paths. Provider-specific Ollama or MiniMax changes would not address the `mkdir '/Users'` failure source.

## Repair Work Prompt

Fix the Docker setup regression reported in https://github.com/openclaw/openclaw/issues/77436: a fresh macOS `./scripts/docker/setup.sh` run can persist host `/Users/...` workspace/config paths into container runtime config, causing first agent reply to fail with `EACCES: permission denied, mkdir '/Users'`. Keep the fix scoped to the Docker setup/Compose/onboarding boundary. Host bind-mount sources may remain host paths, but container runtime/onboarding should see `/home/node/.openclaw` and `/home/node/.openclaw/workspace` for OpenClaw state/workspace resolution. Do not change Ollama/MiniMax/provider behavior, do not add a new config option, and do not broaden Docker socket or secret exposure. Update or add regression coverage in `src/docker-setup.e2e.test.ts` proving host `OPENCLAW_WORKSPACE_DIR` is not persisted as the runtime workspace inside the gateway container. Update `docs/install/docker.md` only if user-facing env semantics change, and add a single-line user-facing `CHANGELOG.md` fix entry.
