## Summary

Keep this open: current main still has the Docker setup path-boundary regression, and open PR #77446 is already linked with closing syntax as the implementation candidate.

## What This Changes

Fresh Docker setup on macOS can persist host `/Users/...` config/workspace paths into the container, causing the first agent reply to fail with `EACCES: permission denied, mkdir '/Users'`.

## Best Possible Solution

Review and land PR #77446 or an equivalent Docker Compose/setup boundary fix that keeps host bind-mount sources as host paths while pinning in-container config and workspace runtime paths to `/home/node/.openclaw`.

## Reproduction Assessment

Yes by source inspection: current main can carry host `/Users/...` path variables from setup into the container and then try to create that workspace on the first reply path. I did not run a live Docker/Ollama Cloud reproduction in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is at the Docker setup/Compose boundary, separating host mount variables from container-visible runtime config paths. Provider-specific Ollama or MiniMax changes would not address the `mkdir '/Users'` failure source.
