## Summary

Keep this open: current main still has the source-reproducible Docker setup path leak, and an open closing PR is already paired with this issue but has not merged into the reviewed checkout.

## What This Changes

The report asks to fix a Docker setup regression after 2026.04.29 where fresh macOS container onboarding can persist host `/Users/...` workspace paths and then fail inside Linux Docker.

## Best Possible Solution

Land or close the paired PR after review; the desired end state is a Docker boundary where host mount sources stay host paths but in-container config and workspace resolution use `/home/node/.openclaw`.

## Reproduction Assessment

Yes. Source inspection shows `scripts/docker/setup.sh` can write macOS `/Users/...` paths into `.env`, Compose imports them into the container runtime, onboarding persists the workspace, and the reply path tries to create that path.

## Solution Assessment

Yes. The narrow maintainable fix is to separate host bind-mount variables from container-visible runtime paths in Docker Compose/setup; provider-specific Ollama or MiniMax changes would not address the `mkdir '/Users'` failure source.
