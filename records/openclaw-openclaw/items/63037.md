## Summary

Keep open: the underlying WhatsApp hang is valid on current main, but this PR is still incomplete because it leaves the controller-based Gateway startup path unbounded, can abandon a just-created socket when the direct monitor timeout fires, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds an optional timeout parameter to `waitForWaConnection`, timeout-focused session tests, and a 60-second timeout at the legacy `monitorWebInbox` startup wait.

## Best Possible Solution

Land a focused WhatsApp startup-timeout fix that keeps QR login waits unbounded unless that flow supplies its own deadline, applies an initial connection deadline to both direct and controller-based monitor startup, explicitly closes sockets abandoned by timeout before retry, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `waitForWaConnection` with an EventEmitter-backed socket and never emit `connection.update` with `open` or `close`; current main leaves that promise pending, including before the controller startup watchdog is installed.

## Solution Assessment

No. The helper-level opt-in timeout is a useful primitive, but this PR is not the best complete fix until the controller startup path receives the same deadline and timeout failures close the created socket before retrying.
