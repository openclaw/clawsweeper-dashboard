## Summary

Keep open. The PR addresses a real WhatsApp startup hang, but the patch is still partial: controller startup can still wait forever, the monitor timeout path can abandon a pre-listener socket, and the PR still lacks real WhatsApp runtime proof.

## What This Changes

The branch adds an optional `timeoutMs` parameter to `waitForWaConnection`, applies a 60s timeout in `monitorWebInbox`, and adds unit coverage for timeout cleanup.

## Best Possible Solution

Add a bounded connection wait at every reconnectable WhatsApp startup path, preserve QR login's TTL-controlled wait, and close sockets abandoned before listener teardown exists.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and current monitor/controller startup paths await it before listener ownership is active.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR must opt in controller startup and close the monitor socket on timeout before it is the narrow maintainable fix.

## Review Findings

  - body: Because `timeoutMs` now defaults to `0`, reconnectable production callers have to opt in. `WhatsAppConnectionController.openConnection` still awaits unbounded `waitForWaConnection(sock)` before listener and watchdog ownership, so gateway startup can still hang in the no-terminal-event state.
  - confidence: 0.91
- **[P2] Close the monitor socket on timeout:** `extensions/whatsapp/src/inbound/monitor.ts:53`

  - confidence: 0.9
