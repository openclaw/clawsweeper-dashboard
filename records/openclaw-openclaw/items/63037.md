## Summary

Keep open. The WhatsApp connection hang is source-reproducible on current main, but this PR is incomplete: it only opts the inbound monitor into a deadline, leaves controller startup unbounded, can abandon a socket when the monitor timeout fires, and still lacks real WhatsApp runtime proof.

## What This Changes

The PR adds an optional timeout parameter to WhatsApp `waitForWaConnection`, applies a 60s timeout in `monitorWebInbox`, and adds unit tests for timeout cleanup.

## Best Possible Solution

Add a bounded connection wait for reconnectable startup paths, keep QR login governed by its login TTL, and close any socket abandoned by a pre-listener timeout before retrying.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and both monitor and controller startup paths await it before reconnect ownership is active.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR must opt in all reconnectable startup waits and close timed-out sockets before it is the narrow maintainable fix.
