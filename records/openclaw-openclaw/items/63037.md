## Summary

Keep open: the hang is source-reproducible and current main still has no deadline in the central wait path, but this PR is incomplete against the current WhatsApp controller lifecycle and still needs contributor real behavior proof before merge.

## What This Changes

This PR adds an optional timeout to `waitForWaConnection`, applies 60 seconds in direct WhatsApp monitor startup, and adds session unit tests for timeout cleanup.

## Best Possible Solution

Port the deadline into the current connection-controller startup path using the existing WhatsApp socket timing contract, keep QR login unbounded unless it passes its own deadline, close sockets on timeout failures, and cover the controller retry path with focused tests.

## Reproduction Assessment

Yes, source-level: an EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves `waitForWaConnection` pending, and controller startup awaits that helper before listener/watchdog ownership. I did not run a live WhatsApp failure path.

## Solution Assessment

No. The helper-level opt-in timeout is useful, but the branch is not complete until the controller startup path receives a deadline and timeout failures close the created socket before retrying.
