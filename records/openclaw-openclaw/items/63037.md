## Summary

Keep open. The underlying WhatsApp pre-open hang is source-reproducible, but this PR is not merge-ready because it misses the current controller-owned startup path, bypasses configured timing in the direct monitor path, lacks real WhatsApp runtime proof, and currently conflicts with main.

## What This Changes

The PR adds an optional timeout parameter to `waitForWaConnection`, applies a fixed 60-second wait in `monitorWebInbox`, and adds unit tests for timeout cleanup.

## Best Possible Solution

Rebase onto current main and make `WhatsAppConnectionController` own a configured bounded pre-open wait with socket teardown and focused tests, then add redacted live WhatsApp runtime proof.

## Reproduction Assessment

Yes, source-level: an EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and current controller startup awaits it before reconnect/watchdog ownership is active. I did not run a live WhatsApp server reproduction during this read-only review.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR needs to move the bounded wait into the current controller startup path, use configured timing, and close timed-out sockets rather than adding a fixed wait to the older monitor call.
