## Summary

Keep open: this targets a real source-level WhatsApp hang, but the current PR is not ready to merge because it only applies the timeout to the direct monitor path, leaves the controller startup path unbounded, can orphan sockets on timeout, and still lacks real behavior proof.

## What This Changes

The PR adds an optional timeout to `waitForWaConnection`, applies a 60s timeout in the direct WhatsApp inbox monitor, and adds helper-level timeout tests.

## Best Possible Solution

Port the timeout into the controller-owned startup path with explicit socket cleanup on timeout, keep QR login unbounded unless that flow passes its own deadline, and add focused regression coverage for timeout, cleanup, and retry behavior.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves `waitForWaConnection` pending, and current main awaits that helper during controller startup before listener/timer ownership.

## Solution Assessment

No. The helper-level timeout is useful, but this PR is incomplete because it protects only the direct monitor call, not the controller startup path, and it does not close sockets abandoned by a timeout.
