## Summary

Keep open: the bug is source-reproducible, but the branch is incomplete for current main because one startup path remains unbounded, the new monitor timeout can abandon a socket, and external real behavior proof is still missing.

## What This Changes

The PR adds an optional timeout parameter to WhatsApp `waitForWaConnection`, applies a 60s wait in `monitorWebInbox`, and adds unit tests for timeout cleanup.

## Best Possible Solution

Bound WhatsApp connection waits where reconnect policy can recover, keep QR login unbounded or separately bounded, and close sockets on timeout before retrying.

## Reproduction Assessment

Yes, source-level. On current main, an EventEmitter-backed socket that never emits `connection.update` with `open` or `close` keeps `waitForWaConnection` pending, and both controller and monitor startup await it.

## Solution Assessment

No. The helper timeout and tests are useful, but a maintainable fix must cover controller startup and close sockets abandoned by the direct monitor timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass the timeout through controller startup:** `extensions/whatsapp/src/session.ts:194`
  - body: Because `timeoutMs` now defaults to no deadline, production callers that should fail fast must opt in. `WhatsAppConnectionController.openConnection` still awaits `waitForWaConnection(sock)` before listener/watchdog ownership, so gateway startup can still hang indefinitely.
  - confidence: 0.91
- **[P2] Close sockets on monitor timeout failure:** `extensions/whatsapp/src/inbound/monitor.ts:53`
  - body: If `waitForWaConnection(sock, 60_000)` rejects here, `monitorWebInbox` exits before `attachWebInboxToSocket` returns the listener whose close method owns socket teardown. Close the created socket in this failure path before rethrowing to avoid orphaned Baileys sockets across retries.
  - confidence: 0.9
