## Summary

Keep open. ## What This Changes

The PR adds an optional timeout to `waitForWaConnection`, passes `60_000` at one WhatsApp monitor startup wait, and adds unit tests for timeout cleanup.

## Best Possible Solution

Rebase onto main and make the WhatsApp controller own a configured bounded pre-open wait with socket teardown on timeout, while preserving QR/login waits and proving the behavior with redacted live runtime output.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and controller startup awaits that helper before reconnect ownership is active.

## Solution Assessment

No. The helper timeout and unit tests are useful, but the PR needs to move the bounded wait into the controller-owned startup path, honor configured timing, and close timed-out sockets before retry or failure.
