## Summary

Keep this PR open, but it is not merge-ready: the timeout primitive is useful, yet the patch misses the current controller startup path, can leak a direct-monitor socket on timeout, and still lacks real after-fix WhatsApp runtime proof.

## What This Changes

The PR adds an optional `timeoutMs` parameter to WhatsApp `waitForWaConnection`, applies a 60-second timeout in `monitorWebInbox`, and adds unit tests for timeout cleanup.

## Best Possible Solution

Port the timeout into the current controller startup and direct monitor paths, route timeout failures through reconnect policy, close any socket abandoned before listener ownership, keep QR login unbounded unless it gets its own deadline, and add focused regression coverage plus redacted live proof.

## Reproduction Assessment

Yes, source-level. A socket EventEmitter that never emits `connection.update` with `open` or `close` leaves `waitForWaConnection` pending, and current controller/direct-monitor startup awaits that helper before the relevant reconnect or listener ownership exists.

## Solution Assessment

No. The helper-level timeout is a useful part of the fix, but this patch leaves the current controller startup caller unbounded and does not close the direct monitor socket when the new timeout rejects before listener ownership exists.
