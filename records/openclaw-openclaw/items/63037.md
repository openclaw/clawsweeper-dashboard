## What This Changes

The branch adds optional `timeoutMs` cleanup logic to `waitForWaConnection`, tests the timeout path, and passes a 60s timeout at the legacy `monitorWebInbox` wait.

## Best Possible Solution

Rebase onto current main, implement a bounded WhatsApp startup wait inside `WhatsAppConnectionController.openConnection` with socket teardown on timeout, preserve QR/login waits, and attach redacted live WhatsApp runtime proof.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and controller startup awaits that helper before reconnect ownership is active.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR is incomplete until controller startup gets a bounded wait, timed-out sockets are closed before retry or failure, and real behavior proof is provided.
