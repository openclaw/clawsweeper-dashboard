## Summary

Keep open. The PR is aimed at a real WhatsApp availability bug, but current main has moved gateway startup through `WhatsAppConnectionController`, and this branch only adds a bounded wait to the older monitor call while leaving the controller pre-open wait unbounded and unproven in a real WhatsApp run.

## What This Changes

The PR adds an optional `timeoutMs` parameter to `waitForWaConnection`, hard-codes a 60s wait in one WhatsApp monitor call, and adds unit tests for timeout cleanup.

## Best Possible Solution

Rebase onto current main and make the WhatsApp controller own a configured bounded pre-open wait with socket teardown on timeout, while preserving QR/login behavior and proving the startup path with redacted live runtime evidence.

## Reproduction Assessment

Yes, source-level: an EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and current controller startup awaits it before reconnect/watchdog ownership is active. I did not run a live WhatsApp server reproduction during this read-only review.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR needs to move the bounded wait into the current controller startup path, use configured timing, and close timed-out sockets rather than adding a fixed wait to the older monitor call.
