## Summary

Keep open: this is a valid WhatsApp liveness bug, but the branch still misses current main's controller startup path and can leave a newly created socket unowned when the new direct-monitor timeout fires; it also lacks external real behavior proof.

## What This Changes

The PR adds an optional timeout to WhatsApp Web connection waiting, applies a 60-second timeout to direct monitor startup, and adds timeout-focused session tests.

## Best Possible Solution

Port the timeout into the current WhatsApp connection-controller startup path with explicit socket cleanup on timeout, keep QR login unbounded unless that flow passes its own deadline, and add focused regression coverage for timeout, cleanup, and controller retry behavior.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits connection.update with open or close leaves current waitForWaConnection pending, and current controller startup awaits that helper before watchdog/listener ownership; no live WhatsApp failure path was run in this read-only review.

## Solution Assessment

No. The helper-level opt-in timeout is a useful primitive, but the patch is not the narrow complete fix until the controller startup path receives the deadline and timeout failures close the created socket before retrying.
