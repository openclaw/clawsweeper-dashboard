## Summary

Keep this PR open, but do not merge it as-is. The reported WhatsApp hang is valid, yet the branch leaves the primary Gateway controller startup path waiting without a deadline and makes the legacy direct monitor timeout before any returned listener owns and closes the new socket.

## What This Changes

The branch adds an optional timeout parameter to `waitForWaConnection`, adds timeout-focused session tests, and passes a 60-second timeout at the legacy `monitorWebInbox` startup wait.

## Best Possible Solution

Land a focused WhatsApp startup-timeout fix that keeps QR login waits unbounded unless the login layer supplies its own deadline, applies an initial connection deadline to both the controller-based Gateway startup path and the legacy direct monitor path, closes any just-created socket on timeout before retry, and adds the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `waitForWaConnection` with an EventEmitter-backed socket and never emit `connection.update` with `open` or `close`; current main leaves that promise pending, and the controller startup path awaits it before installing later watchdogs.

## Solution Assessment

No. The helper-level opt-in timeout is a useful primitive, but this PR is not the best complete fix until the controller startup path receives the same deadline, the direct monitor closes the abandoned socket on timeout, and the user-facing changelog requirement is satisfied.
