## Summary

This PR should stay open with required changes. Current main still has an unbounded `waitForWaConnection` await in the WhatsApp startup paths, and the PR adds a useful opt-in timeout primitive, but it does not cover the controller-based Gateway path and leaves the direct monitor timeout path without socket cleanup.

## What This Changes

The PR adds an optional `timeoutMs` argument to `waitForWaConnection`, adds timeout/listener-cleanup tests, and passes a 60-second timeout at the legacy `monitorWebInbox` callsite.

## Best Possible Solution

Land a focused WhatsApp startup-timeout fix on current main: keep QR/login waits unbounded unless the login layer supplies its own deadline, apply a configured initial connection deadline to both controller-based Gateway startup and legacy direct monitor startup, and close any just-created socket before retrying after timeout.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `waitForWaConnection` with an EventEmitter-backed socket and never emit `connection.update` with `open` or `close`; current main leaves that promise pending, and both startup paths await it before later watchdog ownership exists.

## Solution Assessment

No. The helper-level opt-in timeout is a good primitive, but this PR is not the best complete fix until it is plumbed into the controller startup path and the direct monitor timeout path closes the socket it created.
