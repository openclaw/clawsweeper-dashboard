## Summary

Keep open. Current main still has an unbounded WhatsApp connection waiter and the PR is a useful direction, but the branch is partial: it only opts in one monitor wait, leaves controller startup unbounded, can abandon a socket on monitor timeout, lacks real WhatsApp runtime proof, and is now conflicting with current main.

## What This Changes

This PR adds an optional timeout parameter to `waitForWaConnection`, uses a 60s timeout in `monitorWebInbox`, and adds unit tests for timer and listener cleanup.

## Best Possible Solution

Add bounded waits to all reconnectable opening paths, preserve QR login's existing wait behavior, close sockets abandoned by pre-listener timeouts, rebase onto current main, and provide redacted real WhatsApp runtime proof.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and both monitor/controller startup paths await that helper before reconnect ownership is active.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR is incomplete until controller startup opts into a deadline and timed-out monitor sockets are closed before retry.
