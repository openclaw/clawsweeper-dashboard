## Summary

Keep open: the reported hang is source-reproducible and the helper timeout is a useful direction, but this branch is not merge-ready because it misses the current controller startup path, can abandon a socket on the direct monitor timeout path, and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds an optional timeout parameter to `waitForWaConnection`, applies a 60-second timeout in the direct WhatsApp inbox monitor, and adds timeout cleanup tests.

## Best Possible Solution

Port the timeout into the current controller startup path, route timeout failures through the appropriate reconnect policy, explicitly close any socket created before the timeout, keep QR login unbounded unless that flow passes its own deadline, and add focused regression coverage plus redacted live proof.

## Reproduction Assessment

Yes, source-level. In current main, an EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves `waitForWaConnection` pending, and the controller awaits that helper before listener/watchdog ownership; no live WhatsApp failure path was run in this read-only review.

## Solution Assessment

No. The helper-level timeout is useful, but this patch misses the current controller startup caller and does not close the direct monitor socket when the new timeout rejects before listener ownership exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply the timeout to controller startup:** `extensions/whatsapp/src/session.ts:194`
  - body: Because the new parameter defaults to `0`, every caller that is not updated still waits forever. Current main's Gateway path goes through `WhatsAppConnectionController.openConnection`, which still awaits `waitForWaConnection(sock)` with no deadline before listener and watchdog ownership exists, so the main startup path remains able to hang.
  - confidence: 0.91
- **[P2] Close sockets abandoned by the direct monitor timeout:** `extensions/whatsapp/src/inbound/monitor.ts:53`
  - body: `waitForWaConnection(sock, 60_000)` can reject before `attachWebInboxToSocket` returns the listener whose close handler owns `closeInboundMonitorSocket`. That leaves the newly created Baileys socket unowned while retries can create more sockets, so close the socket in a `catch` before rethrowing.
  - confidence: 0.9
