## Summary

Keep open. The bug is source-reproducible and the helper timeout is useful, but this PR is still incomplete: reconnectable controller startup remains unbounded, monitor timeout failures can abandon a socket before listener teardown owns it, and the external PR still lacks real WhatsApp runtime proof.

## What This Changes

The PR adds an optional timeout to WhatsApp `waitForWaConnection`, applies a 60s timeout in `monitorWebInbox`, and adds unit coverage for timeout cleanup.

## Best Possible Solution

Add a bounded connection wait for reconnectable startup paths, preserve QR login's existing login TTL behavior, and close any socket abandoned by a pre-listener timeout before retrying.

## Reproduction Assessment

Yes, source-level. An EventEmitter-backed socket that never emits `connection.update` with `open` or `close` leaves current `waitForWaConnection` pending, and both monitor and controller startup paths await it before reconnect ownership is active.

## Solution Assessment

No. The helper timeout and tests are useful, but the PR must opt in all reconnectable startup waits and close timed-out sockets before it is the narrow maintainable fix.
