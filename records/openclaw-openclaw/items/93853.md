## Summary

Keep open. The PR targets a real memory-embedding routing bug, but the patch only changes config-time adapter/model selection; the actual memory indexing and gateway create paths still receive `provider: "openai"` and resolve the direct OpenAI memory adapter before configured aliases, so the claimed Ollama adapter routing is not actually proven or fixed.

## What This Changes

The PR changes memory-search config resolution so direct providers with `api` or `baseUrl` hints can consult generic embedding provider resolution before using the direct adapter, and adds tests for `openai`/`ollama` routing.

## Best Possible Solution

Carry one resolved embedding adapter identity through config resolution, memory-core provider creation, and the gateway embedding endpoint while preserving the user-visible provider id for auth/endpoint lookup, then add an integration test and real runtime proof for that path.

Yes, from source inspection. Current main resolves memory config and runtime creation through direct provider ids before configured aliases, but I did not run a live Ollama endpoint reproduction in this read-only review.
