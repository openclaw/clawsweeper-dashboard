## Summary

This should stay open: current main still accepts invalid debug proxy numeric options through Commander parsing and can pass a non-number into the runtime instead of failing at the CLI boundary.

## What This Changes

The issue requests strict validation for debug proxy numeric CLI options such as `proxy sessions --limit`, `proxy validate --timeout-ms`, and `proxy start/run --port`.

## Best Possible Solution

Keep the issue open and fix numeric validation at the proxy CLI boundary, using strict positive-integer parsing for limits/timeouts and the shared port parser for bind ports.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: invalid `--limit` makes the parser return `undefined`, Commander converts that to an empty string for the required option, and the runtime binds it into SQLite `LIMIT ?`. I did not run the CLI because this read-only checkout lacks installed dependencies, but the dependency and source path match the report.

## Solution Assessment

Yes, the best fix is to reject invalid values in `src/cli/proxy-cli.ts` before runtime dispatch. The store and proxy validation runtime should not need defensive compatibility branches for malformed CLI input once the command parser is strict.
