## Summary

Keep open. Current main still lacks a `web_fetch` private-network config opt-in, so this PR addresses a real gap, but it is not merge-ready because it drops current IPv6 ULA SSRF policy/cache handling and exposes a security-sensitive fetch capability without public docs, changelog coverage, or maintainer threat-model approval.

## What This Changes

The branch adds `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork`, threads it into `web_fetch` SSRF policy/cache handling, regenerates config metadata, and adds private/loopback SSRF tests.

## Best Possible Solution

Land or replace this with one canonical `tools.web.fetch.ssrfPolicy` implementation that preserves RFC2544 and IPv6 ULA behavior, partitions cache entries by every SSRF policy flag, updates schema/types/generated metadata, documents the private-network risk boundary, adds the required changelog entry, and resolves its relationship to the broader private-network web tools requests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current-main source inspection shows strict `web_fetch` config rejects the new private-network key while the lower SSRF guard already supports it, and the supplied PR diff shows the rebase regression against current IPv6 ULA policy handling.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The nested dangerous flag is plausible, but the patch must preserve existing SSRF flags/cache keys and needs docs, changelog, and maintainer security approval before it can be the maintainable solution.
