## Summary

Keep open. Current main still lacks a web_fetch private-network config opt-in, so the PR addresses a real gap, but the branch should not merge as-is because it drops current main's IPv6 ULA SSRF policy/cache handling and exposes a security-sensitive network boundary without public docs, changelog, or maintainer threat-model signoff. The provided discussion also shows overlapping open work in #39630 and #53469, so maintainers should choose one canonical API/scope before landing this or a replacement.

## What This Changes

The PR adds `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork`, threads it into web_fetch SSRF policy assembly and cache keys, regenerates config metadata, and adds private/loopback SSRF regression tests.

## Best Possible Solution

Land or replace this with one canonical `tools.web.fetch.ssrfPolicy` implementation that preserves RFC2544 and IPv6 ULA behavior, partitions cache entries by every SSRF policy flag, updates schema/types/generated config metadata, documents the private-network risk boundary, adds the required changelog entry, and resolves how this relates to the broader private-network web tools requests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows strict web_fetch config rejects the new private-network key while the lower SSRF guard already supports it, and the supplied PR diff shows the rebasing regression against the current IPv6 ULA policy path.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The nested dangerous flag is plausible, but the patch must preserve existing SSRF flags/cache keys and needs docs, changelog, and maintainer security approval; the canonical scope should be chosen alongside #39630 and #53469.
