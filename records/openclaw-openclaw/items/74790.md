## Summary

Keep open. The requested private-network opt-in is a real gap on current main, but this PR should not merge as written because it regresses the existing IPv6 ULA SSRF policy/cache partitioning and exposes a security-sensitive web_fetch network bypass without public operator documentation, changelog coverage, or maintainer threat-model approval; it also overlaps with the still-open #39604/#39630/#53469 cluster.

## What This Changes

The PR adds a `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` config flag, threads it into web_fetch SSRF policy assembly and cache keys, regenerates config metadata, and adds SSRF regression tests for private and loopback destinations.

## Best Possible Solution

Land or replace this with one canonical `tools.web.fetch.ssrfPolicy` implementation that preserves RFC2544 and IPv6 ULA behavior, partitions cache entries by every SSRF policy flag, updates schema/types/generated config metadata, adds web_fetch docs and a changelog entry, and documents the private-network threat boundary before enabling the operator opt-in.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current-main source inspection shows strict web_fetch config rejects the new private-network policy key while the underlying SSRF guard already supports it; the supplied PR diff also shows the rebasing regression against current IPv6 ULA policy handling.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The `ssrfPolicy` location and explicit dangerous flag are plausible, but the patch must preserve existing SSRF flags/cache keys and needs public docs, changelog, and maintainer security approval before merge.
