## Summary

Keep open. The requested capability is real on current main, but this PR is not ready to merge because it would drop the existing IPv6 ULA SSRF policy/cache handling and it introduces a security-sensitive private/loopback fetch opt-in without public docs, changelog, or maintainer threat-model approval. It also overlaps active private-network access work in #39630 and #53469.

## What This Changes

The PR adds a `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` config flag, threads it into web_fetch SSRF policy assembly and cache keys, regenerates config metadata, and adds private/loopback SSRF regression tests.

## Best Possible Solution

Land or replace this with one canonical `tools.web.fetch.ssrfPolicy` implementation that preserves RFC2544 and IPv6 ULA behavior, partitions cache entries by every SSRF policy flag, updates schema/types/generated config metadata, adds web_fetch docs and a changelog entry, and documents the private-network threat boundary before enabling the operator opt-in.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current-main source inspection shows strict web_fetch config rejects the new private-network policy key while the underlying SSRF guard already supports it; the supplied PR diff also shows the rebase regression against current IPv6 ULA policy handling.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The `ssrfPolicy` location and explicit dangerous flag are plausible, but the patch must preserve existing SSRF flags/cache keys and needs public docs, changelog, and maintainer security approval before merge.
