## Summary

Keep open: current main does not implement the requested web_fetch private-network config path, and maintainer discussion says this nested API shape is useful but still not merge-ready because it changes a security-sensitive SSRF boundary and drops current policy behavior.

## What This Changes

The PR adds `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork`, forwards it into web_fetch SSRF policy/cache handling, regenerates config metadata, and adds private/loopback SSRF tests.

## Best Possible Solution

Land one canonical nested `tools.web.fetch.ssrfPolicy` implementation after security approval, preserving RFC2544/IPv6 ULA behavior, blocking metadata endpoints, and updating docs, changelog, generated config metadata, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows strict web_fetch config rejects the new private-network key while the lower SSRF guard already supports the dangerous policy flag.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The nested config shape is plausible, but the patch must preserve existing SSRF policy flags/cache keys, block metadata endpoints, and carry docs/changelog plus maintainer security approval.
