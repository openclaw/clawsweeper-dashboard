## Summary

Keep open. Current main still lacks the requested web_fetch private-network config surface, but this PR should not merge as written because it is stale against the current SSRF policy surface, drops the shipped IPv6 ULA opt-in path, and exposes a security-sensitive network-access expansion without public docs/changelog or maintainer threat-model signoff.

## What This Changes

The PR adds `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` to web_fetch runtime config, generated config metadata, and SSRF tests so explicitly configured fetches can reach private and loopback targets with separate cache keys.

## Best Possible Solution

Land or replace this with one canonical implementation under `tools.web.fetch.ssrfPolicy` that preserves RFC2544 and IPv6 ULA exceptions, partitions cache entries by every SSRF policy flag, updates schema/types/generated metadata, adds public web_fetch docs and a changelog entry, and receives maintainer security approval for the private-network threat boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows strict web_fetch config rejects `dangerouslyAllowPrivateNetwork` while the shared SSRF guard already supports it, and the PR patch shows the proposed helper drops the current IPv6 ULA policy path.

## Solution Assessment

No, not as written. The `ssrfPolicy.dangerouslyAllowPrivateNetwork` location is plausible, but the branch must be rebased to preserve current SSRF flags/cache keys and should add public docs, changelog, and maintainer security signoff before merge.
