## Summary

Keep open: the PR addresses a real web_fetch config gap and has maintainer signal as the preferred API shape, but it is not merge-ready because it drops current IPv6 ULA SSRF policy/cache behavior, lacks public docs and changelog coverage, and still needs explicit security/product approval for the private-network trust boundary.

## What This Changes

The branch adds a default-off `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` config key, threads it into web_fetch SSRF policy/cache handling, regenerates config metadata, and adds SSRF tests for private and loopback fetches.

## Best Possible Solution

Land or replace this with one canonical `tools.web.fetch.ssrfPolicy` implementation that preserves all existing policy flags, partitions cache by every flag, blocks metadata endpoints, updates generated config plus public docs and changelog, and has maintainer approval for the dangerous opt-in.

## Reproduction Assessment

Yes. Source inspection shows current main rejects the new strict config key while the lower SSRF guard can accept the policy, and the PR diff shows a clear rebase regression against current IPv6 ULA handling.

## Solution Assessment

No, not as written. The nested `ssrfPolicy` shape is plausible and maintainer-preferred, but the implementation needs to preserve shipped policy flags/cache keys, document the operator risk boundary, add changelog coverage, and settle metadata endpoint behavior before merge.
