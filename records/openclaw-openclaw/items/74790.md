## Summary

Keep open. Current `main` still lacks a supported `web_fetch` private-network opt-in, but this PR is stale against the shipped IPv6 ULA SSRF policy and changes a security-sensitive network boundary that needs maintainer approval, docs, and changelog coverage before merge.

## What This Changes

The PR adds a `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` config flag for `web_fetch`, threads it into SSRF policy/cache handling, regenerates config metadata, and adds private/loopback SSRF tests.

## Best Possible Solution

Land or replace this with one canonical `tools.web.fetch.ssrfPolicy` design that preserves RFC2544 and IPv6 ULA behavior, partitions cache entries by every effective SSRF flag, documents the private-network risk boundary, and aligns with the broader private-network web tools plan.

## Reproduction Assessment

Yes. Source inspection on current `main` shows strict config rejects the new private-network key while the lower SSRF guard already supports the bypass flag, and the PR diff shows the stale rebase gap against current IPv6 ULA policy handling.

## Solution Assessment

No, not as written. The nested dangerous flag is a plausible shape, but the patch must preserve existing SSRF flags/cache keys and needs docs, changelog, and maintainer security approval before it is mergeable.
