## Summary

Keep open. Current main still has only the inherited service-environment marker guard for package updates, not the requested check that walks the caller's PPID ancestry and compares it with the active gateway PID. The existing ancestor-walk code is relevant precedent but is private to stale-gateway cleanup, so the reported silent-restart footgun remains a valid narrow updater safety issue.

## What This Changes

Request that `openclaw update` detect when it is running under the active gateway process ancestry and fail clearly before a supervised restart can silently no-op.

## Best Possible Solution

Add a conservative preflight to the package-update path that reads the active managed gateway runtime PID, walks the caller's ancestry, and refuses restartful updates from inside that process tree with a clear external-shell or `--no-restart` instruction. Reuse or extract the existing bounded ancestor-walk logic rather than adding a second ad hoc parser, and keep automatic sudo/systemd restart helpers out of this issue.

## Reproduction Assessment

Yes. The issue gives concrete systemd reproduction steps, and current source inspection shows the only early refusal depends on inherited service env markers rather than the active gateway PID appearing in the caller's ancestor chain.

## Solution Assessment

Yes for the defensive Option A direction. A loud early refusal before package replacement or restart verification is the narrowest maintainable fix; auto-deferred systemd restarts should stay with the separate system-level restart design surface.

## Repair Work Prompt

Fix `openclaw update` so restartful package updates fail loudly when invoked from inside the active managed gateway process tree. Implement the defensive Option A only: detect the active gateway runtime PID from managed service state, walk the current process's parent chain, and if the gateway PID is an ancestor, abort before package replacement/restart work with a clear message telling the operator to run `openclaw update` from an external shell or use `--no-restart` plus an external manual restart. Prefer extracting/reusing the bounded ancestry logic from `src/infra/restart-stale-pids.ts`; do not add automatic sudo/systemd restart helpers and do not broaden #75692. Add focused tests for env-marker guard preservation, direct parent gateway PID, Linux transitive ancestor via mocked `/proc/<pid>/status`, no false positive when PID is unrelated, and `--no-restart` behavior. Update user-facing docs for the new guard and add a single-line changelog fix entry thanking @islandpreneur007. Keep changes scoped to updater/restart helper surfaces.
