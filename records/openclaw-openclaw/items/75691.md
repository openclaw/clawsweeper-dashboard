## Summary

Keep open. Current main still only blocks package updates when the caller inherited the gateway service marker environment; it does not walk the caller's PPID ancestry and compare it to the active gateway runtime PID, so the reported system-level systemd/in-lane update footgun remains. The issue is already marked In Progress and has an open related PR, so this should route to that active review rather than a duplicate ClawSweeper repair job.

## What This Changes

Request that `openclaw update` detect when it is invoked from inside the active gateway process tree and fail clearly before a restartful package update silently leaves the old gateway process running.

## Best Possible Solution

Add a conservative updater preflight that reads the active managed gateway runtime PID, walks the caller's bounded ancestor chain, and refuses restartful package updates from inside that process tree with clear external-shell and `--no-restart` guidance. Reuse or extract the existing ancestor-walk logic rather than adding a second parser, and keep detached sudo/systemd restart helpers scoped to the separate system-level restart design work.

## Reproduction Assessment

Yes. The issue gives concrete systemd reproduction steps, and source inspection on current main shows no active-gateway-PID ancestry check in the package-update path; this read-only review did not run a live systemd repro.

## Solution Assessment

Yes for the defensive refusal direction. The narrow maintainable fix is to fail before package replacement or restart verification when the caller is under the live gateway; the related PR should be adjusted or reviewed against that exact PPID ancestry requirement if it only covers service env markers.
