## Summary

Keep open. Current main still only blocks package updates when gateway service marker environment variables are present; it does not compare the active Gateway runtime PID with the caller's ancestor process chain, and an open closing PR already targets that missing guard.

## What This Changes

Request that `openclaw update` detect when it is launched from inside the active Gateway process tree and abort before the package-update restart phase silently fails.

## Best Possible Solution

Add a conservative updater preflight that reads the active managed Gateway runtime PID, walks the caller ancestry, and aborts before package replacement with clear external-shell and `--no-restart` guidance.

## Reproduction Assessment

Yes. The report gives concrete systemd and agent-exec steps, a maintainer confirmed the behavior, and current source confirms the package-update path lacks a runtime-PID ancestry comparison.

## Solution Assessment

Yes. The defensive refusal approach is the narrowest maintainable fix; the best route is to review and land or close the active linked PRs rather than create another branch.
