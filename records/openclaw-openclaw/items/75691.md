## Summary

Keep open. Current main still lacks the requested active-gateway-PID ancestry guard; it only blocks package updates when gateway service marker environment is inherited, and this issue is already paired with active fix PRs.

## What This Changes

The issue requests a PPID-ancestry guard so `openclaw update` refuses or safely handles restartful package updates launched from inside the active gateway process tree.

## Best Possible Solution

Add a conservative updater preflight that reads the active managed gateway runtime PID, walks the caller ancestry, and fails before package replacement with clear external-shell and `--no-restart` guidance.

## Reproduction Assessment

Yes. The issue gives concrete systemd/in-lane steps, and current source confirms there is no runtime-PID ancestry comparison in the package-update path.

## Solution Assessment

Yes, the defensive refusal approach is the narrowest maintainable fix. The current main implementation is incomplete, so the best route is to review and land or close the active linked PRs rather than create another repair lane.
