## Summary

Keep open. Current main has a package-update guard for direct service-marker inheritance and restart verification for managed services, but it does not implement the requested PPID-ancestry check for an updater launched from a gateway-owned child process, especially when custom/system-level systemd setup lacks OpenClaw service markers.

## What This Changes

Request `openclaw update` to detect when the active gateway PID is in the caller's parent-process ancestry and fail clearly instead of reporting success while the old gateway process keeps running.

## Best Possible Solution

Add the conservative Option A guard before package update or restart work: resolve the active local gateway PID where possible, compare it with a best-effort self/ancestor PID set, and fail with clear external-shell or `--no-restart` guidance. Keep sudo/system-level restart automation separate under #75692.

## Reproduction Assessment

Yes. The issue gives a concrete systemd plus agent-exec path, and source inspection shows current main has no PPID-ancestry preflight before update/restart work when service markers are absent.

## Solution Assessment

Yes. A fail-fast ancestry guard is the narrowest maintainable fix for this silent-success mode; auto-deferred sudo/systemctl restart is broader and should stay separate from this issue.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/75691 with the conservative Option A guard. Before `openclaw update` performs package-manager update work or restart work, detect whether the active local gateway PID is in the caller's self/ancestor PID set. If so, fail clearly and do not install, doctor, sync plugins, refresh service env, or restart. The message should explain that `openclaw update` is running inside the gateway process tree, name the gateway PID when known, advise running from an external shell, and mention `--no-restart` plus a manual external restart as the fallback. Reuse or factor the existing best-effort PID ancestry logic from `src/infra/restart-stale-pids.ts` rather than duplicating fragile parsing. Cover marker-present and marker-absent cases; do not implement detached sudo/systemctl restart automation here, and do not resolve #75692 in this PR. Update docs and `CHANGELOG.md` because this is user-facing CLI behavior, with proper non-bot attribution.
