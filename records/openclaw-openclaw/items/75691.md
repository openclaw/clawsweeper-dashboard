## Summary

Keep open. Current main has the inherited service-environment guard from #75729, but it still does not compare the active Gateway runtime PID with the caller's self/ancestor PID set, and open PR #75819 is the active closing implementation candidate.

## What This Changes

Teach `openclaw update` to detect when it is invoked from a process whose ancestor chain includes the active Gateway PID and fail or defer restart instead of silently leaving the old Gateway process running.

## Best Possible Solution

Add a conservative updater preflight that reads the active managed Gateway runtime PID, walks the caller self/ancestor PID set, and aborts with external-shell or `--no-restart` guidance before stopping the service or replacing packages.

## Reproduction Assessment

Yes. The report gives concrete systemd and agent-exec steps, and current source inspection still shows no active-runtime-PID versus caller-ancestry comparison in the updater.

## Solution Assessment

Yes. A defensive ancestry preflight is the narrowest maintainable fix; the existing service-marker guard is useful but insufficient for gateway-owned children without inherited marker env.
