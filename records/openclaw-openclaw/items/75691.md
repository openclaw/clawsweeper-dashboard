## Summary

Keep open. Current `main` has an environment-marker guard for package updates launched from the gateway service, but I did not find the requested PPID-ancestry check against the active gateway PID; a gateway-owned child process without those marker env vars can still reach the update path and the restart handling does not make this context explicit.

## What This Changes

The issue requests that `openclaw update` detect when it is invoked from a process whose PPID ancestry includes the active gateway process, so supervised restarts do not silently fail from gateway-owned subprocesses.

## Best Possible Solution

Add a conservative preflight to `openclaw update` that detects when the caller is inside the active gateway process tree and fails loudly before a restartful package update can report success from that context. Prefer the defensive error path over an automatic sudo/systemd helper, and keep broader system-level restart support tracked separately by #75692.

## Reproduction Assessment

Yes. The issue provides concrete systemd steps, and source inspection shows the existing guard depends on inherited service env markers rather than checking PPID ancestry against the active gateway PID.

## Solution Assessment

Yes for the conservative direction. Option A, implemented as an early explicit refusal with targeted tests and docs, is the narrowest maintainable fix; auto-deferred sudo/systemd restart belongs to the broader #75692 design surface.
