## Summary

Keep open. Current main has an environment-marker block and a managed-service stop/restart flow for package updates, but it still does not walk the caller PPID ancestry or compare it with the active gateway runtime PID, so the specific in-gateway-process-tree guard requested here remains unresolved. The issue is also marked In Progress with related PR #75729 active, so it should route to that review instead of being closed or queued as duplicate repair work.

## What This Changes

Request that `openclaw update` detect when it is invoked from inside the active gateway process ancestry and avoid a misleading restartful update flow with clear operator guidance.

## Best Possible Solution

Add a conservative updater preflight before service stop or package replacement: read the active managed gateway runtime PID, walk the caller's bounded ancestor chain, and refuse restartful package updates from inside that tree with explicit external-shell and manual-restart guidance. Reuse or extract the existing ancestor-walk logic, add focused updater tests for direct and transitive ancestry, and keep detached sudo/systemd restart policy separate from this guard.

## Reproduction Assessment

Yes. The issue provides concrete systemd reproduction steps and current source inspection shows the exact missing guard; this read-only review did not run a live systemd reproduction.

## Solution Assessment

Yes for the defensive PPID-ancestry refusal direction. It is narrower and safer than adding detached restart helpers here, because it prevents the misleading in-lane updater path before package replacement or restart handling begins.
