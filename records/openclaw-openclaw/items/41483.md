## Summary

Keep open. Current main still resolves explicit `--agent --to` runs through the agent main-session path, and the issue is now paired with open PR #73403, which directly targets this bug and carries the earlier closed source work from #41557.

## What This Changes

Request that `openclaw agent --agent ... --channel ... --to ...` derive distinct recipient-scoped session keys instead of reusing `agent:<id>:main`.

## Best Possible Solution

Review and land #73403, or an equivalent focused fix, after it preserves explicit `--session-key` and `--session-id` precedence, scopes only explicit deliverable channel targets per recipient, keeps non-channel/global behavior compatible, and handles session-key length/redaction concerns.

## Reproduction Assessment

Yes. A focused reproduction is the resolver path used by the CLI: `agentViaGatewayCommand` calls `resolveSessionKeyForRequest` without `channel`, and `resolveSessionKeyForRequest({ agentId, to })` prefers the agent main session key on current main.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: thread normalized channel data into session resolution and derive per-channel recipient keys only for explicit `--agent --channel --to` runs while preserving existing explicit session selectors.
