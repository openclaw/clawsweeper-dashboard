## Summary

Keep open: current main and v2026.6.6 still resolve explicit `--agent --to` direct targets through the agent main-session path, while https://github.com/openclaw/openclaw/pull/73403 remains the unmerged closing implementation candidate.

## What This Changes

The issue requests per-recipient session routing for `openclaw agent --agent ... --channel ... --to ...` instead of collapsing every recipient into the agent main session.

## Best Possible Solution

Land a focused core CLI/gateway session-routing fix that derives bounded per-recipient agent session keys for explicit deliverable channel targets while preserving `--session-key` and `--session-id` precedence.

## Reproduction Assessment

Yes, source-level: current main resolves `--agent` to the agent main session before `--to` can derive a separate direct-recipient key, and neither gateway nor embedded paths pass `channel` into the resolver. I did not run a live Chatwoot/WhatsApp scenario in this read-only review.

## Solution Assessment

Yes for the issue direction, but it is not implemented on main yet. The best fix is the focused linked PR or a narrow replacement that threads channel context into session resolution, preserves explicit key/id precedence, and addresses bounds/redaction concerns.
