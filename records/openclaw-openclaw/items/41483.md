## What This Changes

The issue reports that `openclaw agent --agent ... --channel ... --to ...` ignores the explicit target and maps distinct recipients to `agent:<id>:main`.

## Best Possible Solution

Keep this issue open as the canonical bug until PR 73403 lands or maintainers replace it with an equivalent focused fix that preserves session-key and session-id precedence.

## Reproduction Assessment

Yes. Source inspection shows the reported CLI shape flows through `agent-command.ts` or `agent-via-gateway.ts` into `resolveSessionKeyForRequest`, where `agentId` creates an explicit main-session key before `to` can derive a per-recipient key; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes in direction. The narrow maintainable solution is to fix the resolver/caller boundary so explicit channel recipients get isolated agent session keys while preserving existing `--session-key` and `--session-id` precedence.
