## Summary

Keep this issue open: current main and v2026.6.10 still do not feed `--channel` into agent CLI session resolution, so `--agent --channel --to` can still collapse onto the agent main session; the linked fix PR is open and conflicting rather than merged.

## What This Changes

The issue requests that `openclaw agent --agent ... --channel ... --to ...` route each explicit recipient to an isolated agent session instead of `agent:<id>:main`.

## Best Possible Solution

Refresh, review, and land the linked fix PR or an equivalent focused replacement that derives isolated per-recipient session keys for explicit channel targets while preserving explicit selector precedence and settling the recipient-key privacy contract.

## Reproduction Assessment

Yes, from source inspection. Current main and `v2026.6.10` resolve `--agent --channel --to` without passing channel into the session resolver, so explicit agent selection can win before recipient isolation is possible.

## Solution Assessment

Yes in direction, but not finished. The right repair boundary is the shared resolver plus Gateway and embedded callers; the merge-ready shape still needs current-main conflict resolution and maintainer acceptance of the recipient-key privacy/compatibility contract.
