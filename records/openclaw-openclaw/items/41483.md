## Label Justifications

- `P1`: The reported behavior can mix multiple customer conversations into one agent session and break a real multi-user bridge workflow.
- `impact:session-state`: All explicit recipients can share one persisted agent session, causing transcript and context state drift across users.
- `impact:security`: Cross-recipient context mixing can expose one user's conversation history or model-inferred private data to another user.
- `impact:message-loss`: The report says wrong responses may be sent to the wrong customer because routing collapses distinct targets into one session.

## Summary

Keep this open: current main and the latest release still route explicit `--agent --to` direct targets through the agent main-session path, and the linked open fix PR remains the active implementation candidate rather than a merged fix.

## What This Changes

Ensure `openclaw agent --agent ... --channel ... --to ...` routes each explicit recipient to an isolated agent session instead of the agent main session.

## Best Possible Solution

Land a focused core CLI/gateway session-routing fix, likely via the linked PR or a narrow replacement, that derives bounded per-recipient agent session keys only for explicit deliverable channel targets while preserving `--session-key` and `--session-id` precedence.

## Reproduction Assessment

Yes, source-level: current main derives the agent main session when `--agent` is set and the resolver has no channel input, so `--to` cannot create per-channel recipient sessions. I did not run a live Chatwoot/WhatsApp scenario.

## Solution Assessment

Yes for the issue direction, but not yet complete in main. The best fix is the focused session-routing change represented by the linked PR or a replacement that also handles precedence, bounds, and redaction concerns.
