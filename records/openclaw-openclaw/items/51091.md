## Summary

Keep open. The latest branch no longer implements the original `messages.suppressToolErrors` request and instead narrows heartbeat warning suppression in a way that regresses current main's documented and tested heartbeat contract; it also lacks after-fix real behavior proof.

## What This Changes

The PR branch changes agent tool-error warning policy so run-level `suppressToolErrorWarnings` no longer hides unresolved mutating failures, then updates tests, docs, config help, generated hashes, and the changelog around that narrower policy.

## Best Possible Solution

Preserve the current `messages.suppressToolErrors` safety boundary and current heartbeat suppression contract unless maintainers deliberately choose a new documented normal-chat tool-error delivery policy.

## Reproduction Assessment

Yes, source-level. Current main source and tests define the existing suppression behavior, and the PR head visibly moves the heartbeat suppression check behind the mutating branch while dropping the current error-reply guard; I did not run tests because this review is read-only.

## Solution Assessment

No. The current branch is not the best way to solve the issue because it regresses established heartbeat behavior and does not settle the broader normal-chat safety policy that would be needed to hide unresolved mutating failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve heartbeat suppression for mutating failures:** `src/agents/pi-embedded-runner/run/payloads.ts:156-158`
  - body: Moving `suppressToolErrorWarnings` below the mutating-tool branch means heartbeat runs configured with `agents.defaults.heartbeat.suppressToolErrorWarnings: true` still emit write/edit/message warnings. Current docs, tests, and the merged heartbeat flag establish that this run-level option suppresses tool warning payloads during heartbeat runs, so keep this check before the mutating return unless maintainers intentionally change the contract.
  - confidence: 0.91
- **[P2] Keep the error-reply guard for mutating warnings:** `src/agents/pi-embedded-runner/run/payloads.ts:152`
  - body: The PR head computes mutating warnings as `!hasUserFacingFailureAcknowledgement`, dropping current main's `!hasUserFacingErrorReply` guard. When the assistant already produced a user-facing error reply, this can append an extra `⚠️ Edit failed` payload; current tests cover suppressing that duplicate, so the branch needs to preserve the guard when it is rebased.
  - confidence: 0.86
