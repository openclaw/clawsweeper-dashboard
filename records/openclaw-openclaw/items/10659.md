## Label Justifications

- `P1`: The issue describes real agent workflows exposing raw API keys and remains a high-priority security/auth-provider design gap.
- `impact:security`: The requested behavior is about preventing credentials from reaching agent-readable files, tool output, transcripts, and prompt-injection paths.

## Summary

Keep open: current main and the latest release include substantial SecretRef, audit, redaction, and plugin-hook hardening, but they still do not provide the native masked-secret boundary requested for arbitrary agent-executed commands and agent-readable plaintext files.

## What This Changes

The issue requests a native masked-secrets boundary so agents can use credentials without seeing raw API keys, including placeholder substitution, output redaction, command blocking, and a mask-list config.

## Best Possible Solution

Keep this issue open until maintainers ship or explicitly supersede a native credential-isolation or masked-secret design with clear security ownership, migration guidance, and plugin/API boundaries.

## Reproduction Assessment

Yes, at source level: current docs explicitly state plaintext credentials remain agent-readable when stored in files the agent can inspect, and source search shows no requested placeholder or mask-list contract.

## Solution Assessment

No single proposed mechanism is proven as the best final design yet; SecretRefs and plugin hooks are strong mitigations, but the remaining boundary needs maintainer/security product direction.
