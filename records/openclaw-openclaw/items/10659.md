## Summary

Keep open: current main and v2026.5.22 include useful SecretRef, audit, model-marker, redaction, and host-env hardening, but they still do not provide the requested native masked-secret boundary for arbitrary agent-executed commands or agent-readable secret files. The linked user-isolation repo and credential-vault plugin are useful mitigations, and the open roadmap/RFC items are related design paths, but none supersedes this concrete security request yet.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without being able to read raw secret values from files, environment variables, command output, transcripts, or prompt context.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, source-reproducible: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. I did not run a live agent leak repro because this review is read-only.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape. The security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
