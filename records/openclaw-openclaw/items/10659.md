## Summary

Keep open: current main and latest release have meaningful SecretRef, audit, redaction, command-scoped resolution, host-env hardening, and plugin-hook mitigations, but not the requested native masked-secret boundary for arbitrary agent commands or otherwise-readable secret files. The linked user-isolation guide and `openclaw-credential-vault` plugin are useful mitigations, not superseding core implementation.

## What This Changes

Request a native masked-secrets system so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, blocked secret-file reads, and config support.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary aligned with SecretRef, provider auth, exec behavior, and plugin hooks.

## Reproduction Assessment

Yes, source-reproducible: current main has no `{{secret:NAME}}` parser, no `secrets.mask` or `maskedSecrets` schema, and docs explicitly say SecretRefs are not a process-isolation boundary.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape; the security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
