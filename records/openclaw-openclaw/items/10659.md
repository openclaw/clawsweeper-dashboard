## Label Justifications

- `P1`: The reporter describes repeated real credential disclosure and a prompt-injection path affecting high-power agent workflows.
- `impact:security`: The request concerns preventing untrusted agent actions and prompts from accessing or exfiltrating raw secrets.
- `impact:auth-provider`: The affected material includes provider API keys and other authentication credentials consumed by OpenClaw and agent tools.

## Summary

The protected security label requires explicit maintainer handling, and current `main` still provides secret resolution and redaction rather than a complete isolation boundary for host-capable agents. The open roadmap and credential-provider RFC are relevant architecture work, but neither has implemented or superseded this concrete requirement.

## What This Changes

Add a native credential boundary that lets agents use API keys without being able to read or leak the raw values.

## Best Possible Solution

Define one security-owned, capability-scoped credential-provider boundary that injects values only at the consuming operation, keeps raw values out of agent-readable files, environments, prompts, and transcripts, and supports external vaults through a narrow plugin contract.

## Reproduction Assessment

Yes, at source level: a host-capable agent can read credentials placed in accessible files or inherited environment variables, and current SecretRef resolution does not create an OS or process isolation boundary.

## Solution Assessment

No, the proposed placeholder syntax and command blacklist are not a complete best solution because equivalent reads and exfiltration paths remain; capability-scoped dispatch-time injection plus comprehensive redaction and a credential-provider contract is safer.
