## Summary

Keep open: current main and the latest release have meaningful SecretRef, runtime snapshot, host-env sanitization, redaction, and plugin-hook mitigations, but they still do not provide the requested native masked-secret boundary for arbitrary agent-executed commands or readable secret files.

## What This Changes

The issue requests a native masked-secrets mechanism so agents can use credentials without seeing raw API keys, including placeholder substitution, output redaction, and policy controls for readable env files or variables.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape. The security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
