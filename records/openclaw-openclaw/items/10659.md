## Summary

Keep open. Current main has meaningful partial hardening through SecretRefs, active runtime snapshots, command-scoped resolution, host exec environment sanitization, transcript/log redaction, and plugin-hook mitigations, but it still does not provide the native masked-secret boundary requested here.

## What This Changes

Request a native masked-secrets system that lets agents use credentials without exposing raw API keys to agent-readable files, environment variables, command output, transcripts, or chat history.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a secops-owned credential boundary for agent-executed commands and persisted output, with raw values kept out of agent-readable files, env, context, logs, and transcripts.

## Reproduction Assessment

Not applicable; this is a feature/security-boundary request rather than a broken existing contract. Source and docs inspection gives high confidence current main still lacks the requested native masked-secret boundary.

## Solution Assessment

No for an automatic implementation. The direction is valid, and existing SecretRef/redaction/plugin work is useful partial hardening, but the maintainable solution needs secops/product agreement on the credential trust boundary and config/API shape before code lands.
