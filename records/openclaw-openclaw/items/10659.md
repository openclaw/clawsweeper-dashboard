## Label Justifications

- `P1`: The issue describes real credential exposure to agents and prompt-injection risk across command and provider-auth workflows, with current main still lacking the requested boundary.
- `impact:security`: The request is about preventing raw API keys and tokens from being read, printed, logged, or injected into model-visible context.
- `impact:auth-provider`: The affected surface includes provider API keys, channel tokens, SecretRef resolution, auth profiles, and future credential-provider routing.

## Summary

Keep open: current main and v2026.5.20 have substantial SecretRef, runtime-snapshot, env-sanitization, redaction, and plugin-hook mitigations, but they still do not provide the native masked-secret boundary requested for arbitrary agent-executed commands or readable secret files.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, command/file protections, and a mask-list config contract.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape. The security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
