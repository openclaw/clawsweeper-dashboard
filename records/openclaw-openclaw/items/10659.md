## Summary

Keep open: current main has meaningful SecretRef, runtime snapshot, environment sanitization, and redaction hardening, but it still lacks the native raw-secret isolation boundary requested for agent-executed commands.

## What This Changes

Request a native masked-secrets system that lets agents use configured credentials through substitution or injection and redaction without exposing raw API keys to the agent.

## Best Possible Solution

Ship or explicitly supersede a native credential boundary that keeps raw values out of agent-readable files, env, and context, performs dispatch-time credential injection, defines exact redaction semantics, and documents the plugin/API contract for external credential providers.

## Reproduction Assessment

Yes, for the missing requested surface: current-main source inspection shows no {{secret:NAME}} placeholder support, no secrets.mask config field, and no general policy preventing agents from reading otherwise-readable env files or variables. I did not run a live exploit because this is a feature/security-boundary request and the absence is clear from schema, docs, and exec code.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally valid, but placeholder parsing, output redaction, deny-list behavior, and credential brokering need secops/product agreement on the public trust boundary before implementation.
