## Summary

Keep open: current main has useful SecretRef, audit, redaction, command-scoped resolution, and plugin-hook mitigations, but it still does not provide the native masked-secret boundary requested for arbitrary agent commands and otherwise-readable secret files.

## What This Changes

Request a native masked-secrets system so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, and policy around secret file/env access.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary that keeps raw values out of agent-readable files, env, context, outputs, transcripts, and logs.

## Reproduction Assessment

Yes, source-reproducible: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. I did not run a live secret-printing repro because this review is read-only and should not expose credentials.

## Solution Assessment

Unclear for the exact placeholder and blocked-command design: the security direction is valid, but the maintainable fix needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
