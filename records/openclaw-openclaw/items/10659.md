## Summary

Keep open. Current main has meaningful SecretRef, runtime snapshot, host exec environment sanitization, redaction, and plugin-hook hardening, but it still does not provide the native masked-secret boundary requested for arbitrary agent-executed commands; the linked workaround, credential-vault plugin, open roadmap/RFC, and closed implementation PRs are adjacent mitigations rather than a shipped core answer.

## What This Changes

The issue requests a native masked-secrets system that lets agents use configured API keys without seeing raw values, including placeholder substitution, output redaction, command blocking, and mask-list config.

## Best Possible Solution

Keep this open until maintainers either ship or explicitly supersede a native credential-use boundary for agent-executed commands, with clear policy for dispatch-time credential injection, raw-value isolation, output redaction, and plugin/provider integration.

## Reproduction Assessment

Yes for the current gap, by source inspection rather than a live failing test: current main has no {{secret:NAME}} parser, no maskedSecrets/secrets.mask schema, and no exec-time substitution boundary for arbitrary commands. The requested behavior is a new security capability, not an already-defined contract regression.

## Solution Assessment

Unclear for the exact proposed shape. The security direction is valid, but the maintainable path needs a maintainer-approved trust model that coordinates SecretRef, credential providers, command/tool injection, redaction, provider auth, and upgrade compatibility.
