## Summary

Keep this issue open: current main has useful SecretRef, snapshot, redaction, and plugin-hook hardening, but it still does not provide the native masked-secret boundary requested for arbitrary agent-executed commands and readable local credentials.

## What This Changes

Request a masked-secrets system that lets agents use credentials without seeing raw API keys, with placeholder substitution, output redaction, command blocking, and new secret-mask configuration.

## Best Possible Solution

Keep the issue open for a maintainer-sponsored security design that either ships a native masked-secret or credential-provider boundary, or explicitly supersedes it with a documented core/plugin API direction and migration story.

## Reproduction Assessment

Yes, source and docs give a high-confidence reproduction basis: current docs explicitly state readable plaintext credentials remain agent-readable, and current code lacks the requested placeholder/mask config surface. I did not run a live shell disclosure test because this review is read-only and must not touch local secrets.

## Solution Assessment

Unclear as an implementation, because the issue proposes a valid security direction but the best permanent boundary needs maintainer choice between core SecretRef evolution, a credential-provider API, and plugin-owned mitigations. Closing now would confuse partial hardening with the stronger native guarantee requested here.
