## Summary

Keep open: current main and v2026.5.22 have substantial SecretRef, runtime snapshot, redaction, model-marker, host-env hardening, and plugin-hook mitigations, but they still explicitly do not provide the requested native boundary that lets agents use arbitrary credentials while preventing access to raw values in readable files, env, or command output.

## What This Changes

Request a native masked-secrets system so agents can use API keys through references or credential injection without seeing raw secret values.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary aligned with SecretRef, provider auth, exec semantics, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape. The security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
