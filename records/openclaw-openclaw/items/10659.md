## Label Justifications

- `P1`: The issue describes real credential exposure and prompt-injection risk across agent command and auth-provider workflows.
- `impact:security`: The request is about preventing raw API keys and tokens from appearing in agent-readable files, environments, command output, transcripts, or prompt context.
- `impact:auth-provider`: The affected surface includes SecretRef resolution, provider credentials, API keys, channel tokens, and future credential-provider routing.

## Summary

Keep open: current main has meaningful SecretRef, audit, redaction, command-secret, and plugin-hook mitigations, and the discussion links useful external mitigations like jmkritt/openclaw-secrets-hardening and openclaw-credential-vault, but OpenClaw still lacks the native credential-use boundary requested for arbitrary agent-executed commands and readable secret files.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without reading raw credential values.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a native credential-use boundary for agent-executed commands that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no maskedSecrets or secrets.mask schema, and no exec-time substitution boundary for arbitrary commands. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact proposed placeholder and blocked-command shape. The security direction is valid, but the maintainable path needs a secops-approved trust model rather than a command blacklist or a second credential system that drifts from SecretRef/provider auth.
