## Summary

Keep open. Current main and the latest release provide important SecretRef, redaction, and plugin-hook mitigations, and the discussion links useful external workarounds, but OpenClaw still does not provide the native masked-secret boundary requested here and the docs explicitly warn that plaintext credentials in agent-readable files remain readable by the agent.

## What This Changes

Request a native masked-secrets system that lets agents use credentials through references or substitution while preventing raw API keys from reaching agent-readable files, environment output, command results, or chat history.

## Best Possible Solution

Define an owner-approved credential boundary for agent-executed commands, combining core SecretRef or credential-provider APIs, exact output redaction, plugin/provider hooks, docs, and migration guidance before implementing the native behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: if a plaintext credential remains in an agent-readable file or environment, the current docs and exec path show the agent can inspect it. I did not run a live secret leak because this review is read-only and must not expose real credentials.

## Solution Assessment

Unclear. The requested `{{secret:NAME}}` plus command-blocking design is one plausible shape, but maintainers should choose the permanent security boundary across core SecretRefs, credential-provider/proxy APIs, plugins, migrations, and docs before implementation.
