## Summary

Keep open: current main has SecretRefs, runtime snapshots, redaction, and plugin hooks, but it still lacks the native masked-secret boundary requested here. The maintainer-associated discussion already kept this open, and the linked user-isolation repo plus credential-vault plugin are useful mitigations rather than a core implementation.

## What This Changes

The issue requests native masked secrets so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, command blocking, and a mask-list config.

## Best Possible Solution

Keep this open until a secops-backed native credential-provider or masked-secret design is shipped or explicitly superseded by the broader open roadmap/RFC work, with dispatch-time injection, exact redaction, and clear agent-readable env/file boundaries.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main has no `{{secret:NAME}}`, `secrets.mask`, or native blocked-command surface, and host exec still derives its environment from the Gateway process after generic sanitization. No test was run because this review is read-only.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally useful, but the maintainable solution needs a security/product design for the credential boundary rather than only adding placeholder parsing or command deny rules.
