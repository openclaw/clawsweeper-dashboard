## Summary

Keep open. Current main has real SecretRef, runtime snapshot, env sanitization, and redaction hardening, but it still lacks the native masked-secret boundary requested for arbitrary agent-executed commands; the linked workarounds, closed PRs, open roadmap, and credential-provider RFC are partial or adjacent rather than a shipped core answer.

## What This Changes

Request a native masked-secrets system that lets agents use credentials without seeing raw values, including placeholder substitution, output redaction, command blocking, and config.

## Best Possible Solution

Keep this open until maintainers either ship or explicitly supersede a native credential-use boundary for agent-executed commands, with clear policy for dispatch-time credential injection, raw-value isolation, output redaction, and plugin/provider integration.

## Reproduction Assessment

Yes for the current gap, by source inspection rather than a live failing test: current main has no {{secret:NAME}} parser, no maskedSecrets/secrets.mask schema, and no exec-time substitution/redaction boundary for arbitrary commands. The requested behavior is a new security capability, not an already-defined contract regression.

## Solution Assessment

Unclear for the exact proposed shape. The security direction is valid, but the best maintainable path needs a maintainer-approved trust model that separates static SecretRefs, credential-provider lifecycle, command/tool injection, and upgrade compatibility.
