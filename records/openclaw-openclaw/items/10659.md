## Summary

Keep open. Current main has meaningful SecretRef, runtime snapshot, env sanitization, and redaction hardening, but it still does not provide the native masked-secret boundary requested for arbitrary agent-executed commands; the external workaround, credential-vault plugin, closed implementation PRs, open security roadmap, and credential-provider RFC are adjacent rather than a shipped core answer.

## What This Changes

The issue requests a native masked-secret system so agents can use API keys without seeing raw secret values.

## Best Possible Solution

Keep this open until maintainers either ship or explicitly supersede a native credential-use boundary for agent-executed commands, with clear policy for dispatch-time credential injection, raw-value isolation, output redaction, and plugin/provider integration.

## Reproduction Assessment

Yes for the current gap, by source inspection rather than a live failing test: current main has no {{secret:NAME}} parser, no maskedSecrets/secrets.mask schema, and no exec-time substitution boundary for arbitrary commands. The requested behavior is a new security capability, not an already-defined contract regression.

## Solution Assessment

Unclear for the exact proposed shape. The security direction is valid, but the maintainable path needs a maintainer-approved trust model that coordinates SecretRef, credential-provider lifecycle, command/tool injection, redaction, and upgrade compatibility.
