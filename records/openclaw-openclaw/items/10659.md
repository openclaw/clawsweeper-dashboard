## Summary

Keep open: current main has meaningful SecretRef, runtime snapshot, redaction, and plugin-hook hardening, but it still lacks the native masked-secret boundary requested here. The linked user-isolation repo and `openclaw-credential-vault` plugin are useful mitigations, and the recent maintainer-associated discussion already treated them as mitigations rather than closure evidence.

## What This Changes

Add a native masked-secrets system so agents can use API keys without seeing raw values, including secret references, runtime substitution, output redaction, blocked direct reads, and config support.

## Best Possible Solution

Keep this open until a secops-backed native credential-provider or masked-secret design ships or is explicitly superseded, with dispatch-time injection, exact redaction semantics, and clear agent-readable env/file boundaries.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main has no `{{secret:NAME}}`, `secrets.mask`, or native blocked-command surface, and host exec still builds from the Gateway process environment after generic sanitization.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally useful, but the maintainable solution needs a security/product design for the credential boundary rather than only placeholder parsing, output redaction, or command deny rules.
