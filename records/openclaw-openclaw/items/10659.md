## Summary

Keep open. Current main has meaningful SecretRef, runtime-snapshot, command-scoped resolution, and redaction hardening, but it still does not implement the native masked-secret boundary requested here, and the live issue context points to maintainer/security/product follow-up rather than cleanup closure.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without reading raw credentials, including secret references, runtime substitution, output redaction, blocked secret-reading commands, and mask-list config.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a native credential-boundary design covering command execution, output/transcript redaction, provider/channel auth, and plugin credential-provider integration.

## Reproduction Assessment

Not applicable; this is a feature/security-boundary request rather than a broken existing contract. Source, docs, and related-item inspection give high confidence that current main still lacks the requested native masked-secret boundary.

## Solution Assessment

No for automatic implementation. Existing SecretRefs, runtime snapshots, redaction, and plugin-hook mitigations are useful, but the maintainable solution needs a secops/product decision on the credential trust boundary before code lands.

Status: needs_attention

Summary: This issue is security-sensitive because it asks OpenClaw to define how credentials are isolated from agents, tools, provider auth, and persisted output.
