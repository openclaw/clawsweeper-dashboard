## Summary

Keep open: current main and the latest release have useful SecretRef, runtime snapshot, redaction, and host exec environment hardening, but they still do not provide the native masked-secret trust boundary requested here. The discussion includes working mitigations such as jmkritt/openclaw-secrets-hardening and openclaw-credential-vault, plus open roadmap/RFC follow-up, but no merged core implementation.

## What This Changes

Request a masked-secrets system that lets agent-executed commands use secrets through references while preventing raw value readback and redacting leaks.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a native credential boundary that keeps raw values out of agent-readable storage, environment, and context while defining dispatch-time injection, exact redaction semantics, and plugin/API integration.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a security architecture feature request. Source and docs inspection on current main and `v2026.5.12` show the requested `{{secret:NAME}}`, `secrets.mask`, and `maskedSecrets` surfaces are absent.

## Solution Assessment

No for a narrow automatic fix. The direction is valid, but placeholder syntax, command-denial policy, exact redaction semantics, credential-provider integration, and process trust boundaries need secops/product agreement first.
