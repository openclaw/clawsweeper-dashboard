## Summary

Keep open: current main has useful SecretRef, runtime snapshot, redaction, and plugin-hook hardening, but it still lacks the native masked-secret boundary requested here. The linked user-isolation repo and `openclaw-credential-vault` plugin are useful mitigations, not a core implementation.

## What This Changes

Request a native masked-secrets system so agents can use API keys through references, runtime substitution, output redaction, and command policy without being able to read raw credentials.

## Best Possible Solution

Keep this open until a secops-backed native credential-provider or masked-secret design ships or is explicitly superseded, with dispatch-time injection, exact redaction rules, and clear agent-readable env/file boundaries.

## Reproduction Assessment

Yes, from source inspection rather than a live failing test: current main has no `{{secret:NAME}}` placeholder support, no `secrets.mask` config surface, and no general blocked-command policy for otherwise-readable env files or variables.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally valid, but the maintainable solution needs secops/product design for the credential boundary rather than only placeholder parsing, output redaction, or deny-list rules.
