## Summary

Keep open. Current main and v2026.5.12 have useful SecretRef, runtime snapshot, host-env sanitization, redaction, and plugin-hook mitigations, but they still do not provide the native masked-secret boundary requested for arbitrary agent-executed commands and raw credential isolation.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without seeing raw credential values.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a secops-owned credential boundary that keeps raw values out of agent-readable files, env, context, logs, and transcripts while defining the core/plugin API shape.

## Reproduction Assessment

Not applicable; this is a feature/security-boundary request rather than a broken existing contract. Source and docs inspection gives high confidence current main and v2026.5.12 still lack the requested native masked-secret boundary.

## Solution Assessment

No for automatic implementation. The direction is valid, but the maintainable solution needs secops/product agreement on the credential trust boundary and API/config shape before code lands.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it asks OpenClaw to define how credentials are isolated from agents, tools, provider auth, logs, and transcripts.

Concerns:

- **[high] Define the credential trust boundary first:**
  - body: A native masked-secret design must decide whether credentials are isolated by a separate process/proxy, credential-provider API, command-scoped injection, sandboxing, redaction, or a combination; that choice affects auth, providers, tool execution, transcripts, and user expectations.
  - confidence: 0.95
- **[medium] Do not treat redaction as isolation:** `src/logging/redact.ts:199`
  - body: Current pattern-based redaction is useful defense in depth, but it cannot prevent an agent from reading plaintext files or environment values that the process can access.
  - confidence: 0.9

  - body: Current host exec starts from process.env and sanitizes dangerous keys, but arbitrary user API key variables may still be inherited unless a broader product/security boundary is designed.
  - confidence: 0.88
