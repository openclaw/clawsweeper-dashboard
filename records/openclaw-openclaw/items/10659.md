## Summary

Keep open: current main has meaningful SecretRef, runtime snapshot, host-env sanitization, redaction, and plugin-hook mitigations, but it still lacks the native masked-secret boundary requested here; maintainer-associated and ClawSweeper comments also kept this security-sensitive feature open.

## What This Changes

Request native masked-secrets support so agents can use configured API keys through references, output redaction, and command/file safeguards without seeing raw values.

## Best Possible Solution

Keep this open until secops/product either ships or explicitly supersedes a native credential boundary with dispatch-time injection, exact redaction semantics, and clear agent-readable env/file guarantees.

## Reproduction Assessment

Yes, for the missing requested surface: source inspection shows no {{secret:NAME}} placeholder support, no secrets.mask config surface, and no general policy preventing agents from reading otherwise-readable env files or variables. I did not run a live exploit because the current-main absence is clear from the schema, docs, and exec path.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally valid, but placeholder parsing, output redaction, and deny-lists alone would not be the strongest maintainable boundary without secops/product decisions on credential providers, proxying, or process isolation.
