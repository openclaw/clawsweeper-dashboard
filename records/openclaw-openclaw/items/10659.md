## Summary

Keep open: current main and v2026.5.22 have meaningful SecretRef, redaction, host-env hardening, and plugin-hook mitigations, but they still do not provide the requested native boundary where agents can use credentials for arbitrary commands without being able to read raw values.

## What This Changes

Request a native masked-secrets system for API key references, runtime substitution, output redaction, blocked raw-secret access, and related configuration.

## Best Possible Solution

Keep this open for security/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary aligned with SecretRef, AuthProfileStore, provider auth, exec semantics, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape. The security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
