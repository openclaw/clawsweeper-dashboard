## Summary

Keep this open: current main and the latest release have strong SecretRef and redaction mitigations, but they still do not provide the native masked-secret boundary requested for arbitrary agent-executed commands and agent-readable plaintext files.

## What This Changes

The issue requests native masked secrets so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, command blocking, and a mask-list config contract.

## Best Possible Solution

Keep this open until maintainers either ship a native credential-isolation/masked-secret design or explicitly supersede it with a documented credential-provider/plugin/API contract and migration guidance.

## Reproduction Assessment

Yes, at source level: current docs explicitly state plaintext credentials remain agent-readable when stored in files the agent can inspect. I did not run a live agent command because this is a read-only review and the source/docs establish the remaining behavior.

## Solution Assessment

No single proposed mechanism is proven as the best final design yet. The safer path is maintainer/security review of a credential-isolation or credential-provider boundary rather than closing on the current opt-in SecretRef implementation.
