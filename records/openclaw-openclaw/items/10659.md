## Summary

Keep open: current main has partial SecretRef, runtime snapshot, host-env sanitization, and redaction hardening, but it still lacks the native raw-secret isolation boundary requested here.

## What This Changes

Request a masked-secrets system that lets agents use API keys without seeing raw values, with placeholder substitution, output redaction, blocked secret-reading commands, and config-driven mask lists.

## Best Possible Solution

Keep this open until secops/product ships or explicitly supersedes a native credential boundary with dispatch-time credential injection, exact redaction semantics, and clear guarantees about agent-readable env and files.

## Reproduction Assessment

Yes, for the missing requested surface: source inspection shows no {{secret:NAME}} placeholder support, no secrets.mask config surface, and no general policy preventing agents from reading otherwise-readable env files or variables. I did not run a live exploit because the current-main absence is clear from schema, docs, and exec code.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally valid, but placeholder parsing, output redaction, and command deny-lists need a secops/product decision on the credential trust boundary before implementation.
