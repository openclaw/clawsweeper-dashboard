- `impact:security`: The request is about preventing raw API keys and tokens from being read, printed, logged, or injected into model-visible context.

## Summary

Keep open: current main and v2026.6.5 include useful SecretRef, audit, redaction, command-scoped resolution, host-env hardening, and plugin-hook mitigations, but they still do not provide the native masked-secret boundary requested for arbitrary agent commands or otherwise-readable secret files.

## What This Changes

The issue requests a native masked-secret system that lets agents use credentials without being able to read raw API keys from files, environment variables, command output, logs, transcripts, or chat context.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary that keeps raw values out of agent-readable files/env/context, injects credentials only at dispatch or proxy time, redacts exact output, and documents the plugin/SDK contract.

## Reproduction Assessment

Yes, source-reproducible: current main has no `{{secret:NAME}}` parser, no `secrets.mask` or `maskedSecrets` schema, and docs explicitly say SecretRefs are not a process-isolation boundary; no live secret-printing repro was run because the review is read-only and should not expose credentials.

## Solution Assessment

Unclear for the exact placeholder and blocked-command design: the security direction is valid, but the maintainable fix needs a secops-approved trust model rather than a command blacklist or parallel credential abstraction that drifts from SecretRef and provider auth.
