## Summary

Keep open: current main has useful SecretRef, redaction, command-secret, and plugin-hook mitigations, but it still does not provide the native masked-secret boundary requested for arbitrary agent-executed commands and agent-readable credential files.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without seeing raw values, including substitution, redaction, command blocking, and a mask-list config surface.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a secops-approved credential-use boundary that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and the docs state SecretRefs are not a process-isolation boundary. This is a new security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact placeholder and blocked-command design. The security direction is valid, but the maintainable solution should be a secops-approved trust model rather than a second credential system or command blacklist that drifts from SecretRef/provider auth.
