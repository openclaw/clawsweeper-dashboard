## Summary

This remains a valid security/product issue: current main has substantial SecretRef, audit, migration, snapshot, and redaction hardening, but it still does not establish a native boundary that lets host-executed tools use credentials while preventing a host-capable agent from reading raw values. The broader security roadmap and credential-provider RFC remain open; the Linux isolation guide and credential-vault plugin are useful mitigations, while both prior core PR attempts closed unmerged.

## What This Changes

Requests a native masked-secret system that lets agents invoke credentialed operations without reading or leaking raw API keys.

## Best Possible Solution

Define one security-owned credential-provider contract that injects capability-scoped credentials only at the execution boundary, keeps raw values out of agent-readable files, environment, prompts, and transcripts, and applies complete output redaction with a documented plugin integration seam.

## Reproduction Assessment

Yes at the source level: a host-capable agent can request direct reads of otherwise accessible environment variables or plaintext configuration and receive command output. A live test with real credentials would be unsafe and is unnecessary to establish this exposure class.

## Solution Assessment

No, the original placeholder substitution, output replacement, and blocked-command proposal is not the best complete security boundary because equivalent read and exfiltration paths remain. Capability-scoped injection owned by the process or service executing the authorized operation is the safer direction.
