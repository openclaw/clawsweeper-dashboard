## Summary

Current main has substantial SecretRef and redaction hardening, but not the requested native masked-secret boundary for arbitrary agent-executed commands or otherwise-readable secret files. The linked user-isolation guide, credential-vault plugin, closed unmerged PRs, and open credential-provider RFC are useful context rather than closure evidence.

## What This Changes

The issue requests a native masked-secret system that lets agents use API keys without seeing raw values, including placeholder substitution, output redaction, command blocking, and a mask-list config surface.

## Best Possible Solution

Keep this open for secops/product design until OpenClaw ships or explicitly supersedes a native credential-use boundary that keeps raw values out of agent-readable files, env, prompts, transcripts, and arbitrary command output while aligning with SecretRef and plugin/provider contracts.

## Reproduction Assessment

Yes. Source inspection reproduces the missing behavior: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly state SecretRefs are not a process-isolation boundary for agent-readable files.

## Solution Assessment

Unclear for the exact proposed shape. The security direction is valid, but the best implementation needs a secops-approved trust model instead of only adding placeholder substitution and command blacklists that could drift from SecretRef, provider auth, and plugin hooks.
