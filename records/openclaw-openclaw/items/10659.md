## Summary

Keep open. Current main has meaningful partial hardening through SecretRefs, eager runtime snapshots, command-scoped resolution, host exec environment sanitization, transcript/log redaction, and plugin-hook mitigations, but it still does not provide the native masked-secret boundary requested here. The linked user-isolation guide and credential-vault plugin are useful current mitigations, not a complete core/security design.

## What This Changes

The issue requests native masked secrets so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, blocked secret-reading commands, and a mask-list config surface.

## Best Possible Solution

Keep this open until maintainers either ship or explicitly supersede a native credential boundary for agent-executed commands and process output, with raw values kept out of agent-readable files/env/context and a secops-owned plugin/SDK story for external credential providers.

## Reproduction Assessment

Not applicable; this is a feature/security-boundary request rather than a broken existing contract. Source and docs inspection gives high confidence current main still lacks the requested native boundary.

## Solution Assessment

No for automatic implementation. Existing SecretRef, redaction, and plugin-hook work is useful partial hardening, but the maintainable solution needs secops/product agreement on the credential trust boundary before code lands.
