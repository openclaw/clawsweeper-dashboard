## Label Justifications

- `P1`: Users report raw credentials exposed through agent-controlled execution, creating an urgent security gap that affects real workflows.
- `impact:security`: The issue concerns credential confidentiality and the trust boundary between agents and secret material.
- `impact:auth-provider`: The requested design affects provider API keys and credential resolution across models, tools, channels, and plugins.

## Summary

Current main improves secret references, runtime snapshots, command-scoped resolution, and redaction, but it still lacks a general boundary that lets arbitrary agent-executed tools use credentials without exposing raw values. The open credential-provider RFC is related but does not by itself supply process isolation, so this security feature still needs an explicit maintainer product decision.

## What This Changes

Add a native masked-secret system that injects credentials only at execution time, redacts leaked values, and prevents agents from reading raw keys.

## Best Possible Solution

Adopt one maintainer-approved credential dispatch design that keeps raw values outside agent-readable files, environment, prompts, and transcripts; combine a real process or capability boundary with exact output redaction and a documented integration point for SecretRef and credential-provider plugins.

## Reproduction Assessment

Yes, at source level. Host exec accepts agent-controlled commands and environment data, while current SecretRef/runtime snapshot handling does not provide arbitrary command substitution or isolate otherwise readable credentials from the executing agent.

## Solution Assessment

No, the proposed placeholder, mask-list, and command-blocking sketch is not sufficient as the final security design. A maintainable solution needs a real dispatch or process boundary, exact redaction, and a deliberate relationship to the existing SecretRef system and https://github.com/openclaw/openclaw/issues/59165.

## Risks / Open Questions

- A plugin or in-process credential-provider abstraction alone does not protect credentials from an agent that can execute code in the same process or read the resulting subprocess environment.
- Command blocklists such as rejecting `env`, `echo`, or `cat` are bypass-prone and should not be treated as the primary security boundary.
- Adding another config and placeholder system without reconciling SecretRef and the credential-provider RFC could create competing credential sources and unclear upgrade behavior.
