## Summary

Keep open: current main has meaningful SecretRef, runtime snapshot, host exec environment, and redaction hardening, plus external workaround/plugin ideas, but it still lacks the native masked-secret boundary requested for agent-executed commands and raw credential isolation.

## What This Changes

The issue requests native masked secrets so agents can use API credentials without reading raw values, including placeholder substitution, output redaction, blocked secret-reading commands, and a mask config.

## Best Possible Solution

Ship or explicitly supersede a native credential boundary that keeps raw values out of agent-readable files, env, and context, performs dispatch-time credential injection, defines exact redaction semantics, and documents the plugin/API contract for external providers.

## Reproduction Assessment

Yes, for the missing requested surface: current-main source inspection shows no {{secret:NAME}} support, no secrets.mask or maskedSecrets config contract, and no general policy preventing agents from reading otherwise-readable env files or variables. I did not run a live exploit because the absence is clear from schema, docs, and exec code.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally valid, but placeholder parsing, output redaction, command denial, credential brokering, and plugin/provider integration need secops and product agreement on the trust boundary before implementation.
