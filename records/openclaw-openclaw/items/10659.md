## Summary

Keep this open. Current main has substantial SecretRef, snapshot, redaction, and hook hardening, but it still lacks the native masked-secret boundary requested for arbitrary agent-executed commands; the maintainer-associated keep-open comment and active credential-provider RFC make this security/product review work, not cleanup-close material.

## What This Changes

Request native masked-secret handling so agents can use configured credentials without reading raw API keys, including substitution, redaction, and command/env access controls.

## Best Possible Solution

Keep this open under secops/product review until OpenClaw ships or explicitly supersedes a native masked-secret or credential-provider design for agent-executed commands and process output, with https://github.com/openclaw/openclaw/issues/59165 and https://github.com/openclaw/openclaw/issues/11829 as related design context.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a security feature request for a new masked-secret boundary. Source inspection gives high confidence that current main lacks the requested native `{{secret:NAME}}`/mask-list behavior.

## Solution Assessment

Unclear that the exact proposed syntax and command-blocking design is the best final shape. The maintainable path is a secops-reviewed credential-provider or masked-secret boundary that keeps raw values out of agent-readable files/env/context and documents the upgrade path.
