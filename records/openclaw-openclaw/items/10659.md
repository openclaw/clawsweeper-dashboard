## Summary

Keep open: current main and the latest shipped tag contain useful SecretRef, runtime snapshot, exec environment, plugin hook, and redaction hardening, but they still do not provide the native masked-secret boundary requested here. The discussion also has a recent maintainer-associated keep-open review, active roadmap/RFC follow-up, and external mitigations rather than a merged core implementation.

## What This Changes

The issue requests a native masked-secrets system that lets agents use configured API keys without being able to read or leak raw values.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a core credential boundary that keeps raw values out of agent-readable storage, environment, and context while defining dispatch-time injection, exact redaction semantics, and plugin/API integration.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/security architecture request. Source inspection on current main and `v2026.5.12` shows the requested `{{secret:NAME}}`, `secrets.mask`, and `maskedSecrets` surfaces are absent.

## Solution Assessment

No for a narrow automatic fix. The direction is valid, but placeholder syntax, command-denial policy, exact redaction semantics, credential-provider integration, and process/trust-boundary expectations need secops and product agreement first.
