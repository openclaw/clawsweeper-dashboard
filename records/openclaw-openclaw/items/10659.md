## Summary

Keep open. Current main has meaningful partial hardening through SecretRefs, runtime snapshots, command-scoped resolution, host exec env sanitization, and redaction, but it still does not provide the native masked-secret boundary requested here: placeholder substitution for arbitrary agent commands, a configured mask list, exact output masking for configured secrets, or a policy that prevents agents from reading otherwise-readable secret files or env values.

## What This Changes

The issue requests a native masked-secrets system that lets agents use API credentials without seeing raw values, including placeholder substitution, output redaction, blocked leakage commands, and config-driven masking.

## Best Possible Solution

Keep this open until maintainers either ship or explicitly supersede a native credential boundary for agent-executed commands and process output, with raw values kept out of agent-readable files/env/context and a clear secops-owned plugin/SDK story for external credential providers.

## Reproduction Assessment

Not applicable; this is a feature/security-boundary request rather than a broken existing contract. Source and docs inspection gives high confidence current main still lacks the requested native boundary.

## Solution Assessment

No for automatic implementation. Existing SecretRef and redaction work is a partial mitigation, but the maintainable solution needs secops/product agreement on the trust boundary before code lands.
