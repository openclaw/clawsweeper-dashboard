## Summary

Keep this issue open: current main has substantial SecretRef, runtime-resolution, hook, and redaction hardening, but it still does not provide the requested boundary preventing an unsandboxed agent from reading same-user credentials. The open credential-provider RFC is related but does not yet prove complete ownership of agent-exec isolation and output persistence.

## What This Changes

Add a native credential-use boundary so agents can invoke authenticated providers and tools without receiving or exposing raw secret values.

## Best Possible Solution

Define one security-owner-approved credential-provider contract covering provider calls and agent tools: inject narrowly scoped credentials only at the owning operation boundary, keep raw values out of model-readable files and context, and redact output before logs, transcripts, persistence, and delivery.

## Reproduction Assessment

Yes, at source and documented-contract level: an unsandboxed host agent can execute commands against readable same-user files and inherited environment, allowing raw credentials to be returned through tool output. A live credential was deliberately not used during this read-only review.

## Solution Assessment

No, not exactly as proposed: dispatch-time scoped credential injection is the right direction, but command-name blocking, placeholder substitution, and output filtering alone are bypassable defenses. The durable solution is an opaque credential-provider boundary with sandboxing and redaction as complementary layers.
