## Summary

Keep open: current main has useful SecretRef, runtime snapshot, host exec environment, and redaction hardening, but it still does not provide the native masked-secret boundary requested here; recent discussion, the security roadmap, and the credential-provider RFC all point to maintainer follow-up rather than cleanup closure.

## What This Changes

Add a native masked-secrets system so agents can use configured API keys without seeing raw values, including placeholder substitution, output redaction, blocked secret reads, and config.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a core credential boundary that keeps raw values out of agent-readable storage, environment, and context while defining dispatch-time injection, exact redaction semantics, and plugin/API integration.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/security architecture request. Source inspection on current main shows the requested `{{secret:NAME}}`, `secrets.mask`, and `maskedSecrets` surfaces are absent.

## Solution Assessment

No, not as a narrow automatic fix. The direction is valid, but placeholder syntax, command denial policy, exact redaction semantics, and credential-provider integration need secops and product agreement first.
