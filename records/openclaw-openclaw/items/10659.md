## Summary

Keep this issue open. Current main has meaningful SecretRef, runtime snapshot, env sanitization, redaction, and plugin-hook mitigation surfaces, but it still does not provide the native masked-secret boundary requested here for arbitrary agent-executed commands and process output. The issue discussion also contains a recent maintainer-associated keep-open review, and the linked external hardening repo and credential-vault plugin are useful mitigations rather than closure evidence.

## What This Changes

Add native masked-secrets support so agents can use API keys through references, substitution, and redaction without being able to read raw credential values.

## Best Possible Solution

Keep this open until OpenClaw ships or explicitly supersedes a security-owned credential boundary for agent-executed commands and process output. The end state should keep raw values out of agent-readable files/env/context, inject credentials only at dispatch time, provide exact configured-value redaction at tool/log/transcript/message sinks, and document how plugin/vault integrations fit into the supported model.

## Reproduction Assessment

Yes for the missing native boundary. A dummy-secret source-level check is enough: current main has no requested placeholder/config surface, docs describe eager snapshot resolution, and exec still runs with an inherited sanitized environment rather than a per-secret brokered substitution path.

## Solution Assessment

No current workaround is the complete solution. User isolation and openclaw-credential-vault are valuable mitigations, but the maintainable fix needs a core/security-owned credential boundary rather than only pattern redaction, plugin hooks, or command-blocking policy.
