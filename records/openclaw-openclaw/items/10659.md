## Summary

Keep open. Current main and v2026.5.12 have SecretRef, runtime snapshot, redaction, command-scoped resolution, and host-env hardening, plus useful external/plugin mitigations, but they still do not provide the native arbitrary-command masked-secret boundary requested here.

## What This Changes

Requests a native masked-secrets system so agents can use API credentials without seeing raw values, including placeholder substitution, output redaction, blocked env/file reads, and a `secrets.mask` config surface.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a secops-owned credential boundary that keeps raw values out of agent-readable files, env, context, logs, and transcripts while defining the core/plugin API shape.

## Reproduction Assessment

Not applicable; this is a feature/security-boundary request rather than a broken existing contract. Source and release inspection give high confidence current main and v2026.5.12 still lack the requested native masked-secret boundary.

## Solution Assessment

No for automatic implementation. The direction is valid, but the maintainable solution needs secops/product agreement on the credential trust boundary and API/config shape before code lands.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it asks OpenClaw to define how credentials are isolated from agents, tools, providers, logs, and transcripts.

Concerns:

- **[high] Define the credential trust boundary first:**
  - body: A native masked-secret design must decide whether credentials are isolated by a separate process/proxy, credential-provider API, command-scoped injection, sandboxing, redaction, or a combination; that choice affects auth, providers, tool execution, transcripts, and user expectations.
