## Summary

Keep open. Current main and shipped releases have substantial SecretRef, audit, migration, and redaction hardening, but they do not implement the native masked-secret/access boundary requested here; the docs still say plaintext credentials in agent-readable files remain agent-readable, and the proposed `{{secret:NAME}}`/`secrets.mask` surface is absent.

## What This Changes

Request to add a masked-secrets system so agents can use credentials without reading raw API keys, including placeholder substitution, output redaction, blocked secret-read commands, and config-driven mask lists.

## Best Possible Solution

Keep this issue open for security-owner product review until OpenClaw ships or explicitly supersedes a native masked-secret or credential-provider design for agent-executed commands, credential storage, and process output.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction basis: the docs state plaintext credentials in agent-readable files remain readable, and the strict secrets schema has no native mask-list contract. I did not run a live secret-read because this review was read-only and should not create or expose credentials.

## Solution Assessment

No, current main is not yet the best complete solution. SecretRefs, audit/apply flows, and redaction are valuable partial hardening, but the durable fix needs a maintainer security decision on whether the boundary is SecretRef migration, exec/read policy, an external proxy, a CredentialProvider API, or some combination.
