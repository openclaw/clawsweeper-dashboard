## Label Justifications

- `P1`: The issue describes real credential exposure to agents and prompt-injection risk across command and provider-auth workflows, with current main still lacking the requested boundary.
- `impact:security`: The request is about preventing raw API keys and tokens from being read, printed, logged, or injected into model-visible context.
- `impact:auth-provider`: The affected surface includes provider API keys, channel tokens, SecretRef resolution, auth profiles, and possible future credential-provider routing.

## Summary

Keep open: current main has substantial SecretRef, runtime snapshot, env-sanitization, redaction, and plugin-hook mitigations, but it still lacks the native masked-secret boundary requested for arbitrary agent-executed commands and readable secret files.

## What This Changes

Request a masked-secrets system that lets agents use API keys through references without seeing raw values, with runtime substitution, output redaction, command blocking, and a new secrets mask config.

## Best Possible Solution

Keep this open until secops/product maintainers ship or explicitly supersede a native credential-use boundary that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no {{secret:NAME}} parser, no secrets.mask or maskedSecrets schema, and docs explicitly say SecretRefs are not a process-isolation boundary. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact placeholder and blocked-command shape. The security direction is valid, but the maintainable solution needs a secops-approved trust model rather than a command blacklist or parallel credential system that drifts from SecretRef/provider auth.
