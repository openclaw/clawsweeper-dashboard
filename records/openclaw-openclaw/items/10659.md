## Label Justifications

- `P1`: The issue describes real credential exposure to agents and prompt-injection risk across command and auth-provider workflows, even though the final fix requires security/product design.
- `impact:security`: The request is about preventing raw API keys and tokens from appearing in agent-readable files, environments, command output, transcripts, or prompt context.
- `impact:auth-provider`: The affected surface includes SecretRef resolution, provider credentials, API keys, channel tokens, and future credential-provider routing.

## Summary

Keep open: current main has strong SecretRef, audit, doctor warning, redaction, and command-scoped mitigation surfaces, but it still does not provide the native masked-secret boundary requested for arbitrary agent-executed commands. The linked user-isolation workaround, credential-vault plugin, closed implementation attempts, open roadmap, and credential-provider RFC are useful context rather than closure evidence.

## What This Changes

Request a native masked-secret system so agents can use API keys without seeing raw values, with placeholder substitution, output redaction, command/file-read protections, and configuration for masked secrets.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a native credential-use boundary for agent-executed commands that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no `{{secret:NAME}}` parser, no `maskedSecrets` or `secrets.mask` schema, and no exec-time substitution boundary for arbitrary commands. This is a new security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact proposed placeholder and blocked-command shape. The security direction is valid, but the maintainable path needs a secops-approved trust model rather than a command blacklist or a second credential system that drifts from SecretRef/provider auth.
