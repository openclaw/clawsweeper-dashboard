## Summary

Keep open: current main has meaningful SecretRef, runtime snapshot, redaction, and plugin-hook hardening, but the native masked-secret boundary requested here is still missing; recent maintainer and ClawSweeper comments also kept it open.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without seeing raw values, including placeholder substitution, output redaction, blocked readback commands, and a secrets mask config surface.

## Best Possible Solution

Keep this open until secops/product either ships or explicitly supersedes a native credential-provider or masked-secret design with dispatch-time injection, exact redaction rules, and clear agent-readable env/file boundaries.

## Reproduction Assessment

Yes, from source inspection rather than a live exploit: current main has no {{secret:NAME}} placeholder support, no secrets.mask config surface, and no general command policy that prevents reading otherwise-readable env files or variables.

## Solution Assessment

No, not as a narrow automatic fix. The proposal is directionally valid, but the maintainable solution needs secops/product design for the credential boundary rather than only placeholder parsing, output redaction, or deny-list rules.
