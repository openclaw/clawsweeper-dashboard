## Summary

Keep open: current main has SecretRef snapshots, audit, command-scoped resolution, and redaction mitigations, but it still lacks the native masked-secret boundary requested for arbitrary agent-executed commands. The linked user-isolation workaround, credential-vault plugin, closed implementation attempts, open roadmap, and credential-provider RFC are useful context rather than closure evidence.

## What This Changes

The issue requests a native masked-secrets system so agents can use API keys without being able to read raw secret values.

## Best Possible Solution

Keep this open until maintainers ship or explicitly supersede a security-owned credential-use boundary for agent-executed commands that keeps raw values out of agent-readable files, environments, prompt context, transcripts, and command output while aligning with SecretRef, provider auth, and plugin hooks.

## Reproduction Assessment

Yes, by source inspection: current main has no `{{secret:NAME}}` parser, no `maskedSecrets` or `secrets.mask` schema, and no general exec-time substitution boundary for arbitrary commands. This is a missing security capability rather than a regression in an already-defined contract.

## Solution Assessment

Unclear for the exact proposed placeholder and blocked-command shape. The security direction is valid, but the maintainable path needs a secops-approved trust model rather than a command blacklist or a second credential system that drifts from SecretRef/provider auth.
