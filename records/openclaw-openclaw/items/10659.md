## Summary

Keep this issue open: current main has substantial SecretRef, audit, doctor, marker, and redaction hardening, but it still does not provide the native masked-secret/process boundary requested here, and the linked workaround/plugin efforts remain mitigations rather than a core product decision.

## What This Changes

Request a native masked-secrets system so agents can use credentials without seeing raw API keys, with placeholder substitution, output redaction, command/file-read controls, and a mask-list config surface.

## Best Possible Solution

Keep this open for a maintainer/security decision on the permanent boundary: either native core credential isolation/masked-secret semantics, a credential-provider or proxy-style API that satisfies the same guarantee, or an explicit decision that plugin/workaround mitigations are the supported path.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction basis: current docs explicitly say plaintext credentials remain agent-readable in readable files, current SecretRefs are opt-in and not a process-isolation boundary, and the schema lacks the requested placeholder/mask config. I did not run a live shell disclosure test because this review is read-only and must not touch local secrets.

## Solution Assessment

Unclear as an implementation: the security direction is valid, but the exact proposal mixes placeholders, redaction, and command blocking, while the best durable boundary may be process isolation, credential-provider/proxy APIs, or a narrower core SecretRef evolution. Maintainers should choose the security/product shape before an autonomous fix PR.
