## Summary

Keep open. Current main still reproduces the reported policy mismatch, and open PR #57192 already uses closing syntax for this issue, so the issue should stay open until that PR is reviewed, merged, or explicitly rejected. The remaining decision is a privacy-sensitive product/security policy choice: default-allow macOS `screen.record`, or keep it supported but gated by explicit `gateway.nodes.allowCommands`.

## What This Changes

The issue asks OpenClaw to allow macOS node `screen.record` when the node declares screen capability and Screen Recording permission instead of rejecting it through the gateway platform allowlist.

## Best Possible Solution

Resolve this through PR #57192 or an equivalent maintainer-owned decision. If macOS screen recording should be callable by default, update the gateway policy, regression tests, security/audit messaging, and docs together; if explicit opt-in is intended, keep the current policy and make the macOS capability/docs/CLI guidance clearly explain that `screen.record` is supported but gated by `gateway.nodes.allowCommands`.

## Reproduction Assessment

Yes. Current source and existing regression tests give a high-confidence reproduction path: a macOS node can declare `screen.record`, but the resolved macOS allowlist excludes it and `node.invoke` emits the reported allowlist error.

## Solution Assessment

Unclear. Allowing `screen.record` by default is a narrow technical fix and PR #57192 is the active candidate, but it changes the default for a privacy-sensitive command; the safer alternative is to keep explicit opt-in and align the docs if maintainers prefer that policy.
