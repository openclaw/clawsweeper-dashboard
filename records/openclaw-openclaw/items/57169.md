## Summary

Keep open. Current main and the latest release still gate macOS `screen.record` by default while the macOS node declares and implements it, and https://github.com/openclaw/openclaw/pull/57192 remains open with closing syntax for this issue. The remaining decision is privacy-sensitive policy work, because `screen.record` is still classified as a dangerous command and setup/audit/docs continue to preserve explicit opt-in semantics.

## What This Changes

Request that macOS nodes which declare `screen.record` and have Screen Recording permission can run it without the default gateway allowlist rejection.

## Best Possible Solution

Resolve this through the open PR or an equivalent maintainer-owned policy update: either default-allow macOS `screen.record` consistently across gateway policy, setup defaults, audit/docs/tests, and real macOS proof, or keep explicit opt-in and clarify the macOS capability messaging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main path: macOS nodes declare and implement `screen.record`, but the default gateway allowlist excludes it and `node.invoke` emits the reported allowlist error. I did not run a live macOS companion recording in this read-only review.

## Solution Assessment

Unclear. Default-allowing `screen.record` may be the desired product outcome, but the safe implementation must update the dangerous-command policy, setup deny defaults, audit behavior, docs, and tests together rather than only adding a platform default.
