## Summary

Keep open. Current main still keeps macOS `screen.record` out of the default gateway allowlist while the macOS node declares and implements the command, and an open PR already uses closing syntax for this issue. The remaining choice is a privacy-sensitive product/security policy decision rather than cleanup-safe closure.

## What This Changes

The issue requests aligning macOS node screen-recording capability with gateway command policy so `screen.record` is not rejected after permissions are granted.

## Best Possible Solution

Resolve this through the open PR or an equivalent maintainer-owned policy update: either default-allow macOS `screen.record` with coordinated dangerous-command, wizard, audit, docs, and tests changes, or keep explicit opt-in and clarify the macOS capability/docs messaging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: macOS nodes declare `screen.record`, but the gateway allowlist excludes it by default and `node.invoke` emits the reported allowlist error when the declared command is not allowlisted.

## Solution Assessment

Unclear. Default-allowing `screen.record` is the active proposed fix, but it changes a privacy-sensitive default; the maintainable path must decide the policy and update gateway policy, setup defaults, audit behavior, docs, and tests together.
