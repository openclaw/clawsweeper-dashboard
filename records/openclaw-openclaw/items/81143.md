## Summary

Keep open: current main still has the reported auth/status path that can re-enter plugin external-auth/provider resolution, repeat the Codex bootstrap-only local-OAuth no-op, and synchronously fingerprint plugin manifests before returning cached metadata. The issue is recent, has an open paired draft PR, and the related broader sync-FS reports do not fully own this idle auth/provider trigger.

## What This Changes

Reduce idle gateway latency and CPU by avoiding repeated plugin manifest/provider/auth scans during auth-status and provider resolution.

## Best Possible Solution

Carry a gateway-owned plugin metadata snapshot or lookup table through auth/provider resolution, or narrowly memoize unchanged external-auth no-op decisions, while preserving fresh default plugin discovery.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection shows the repeated path, and the issue has fresh reporter evidence using `openclaw status --timeout 1000 --json` on 2026.5.7, but a current-main runtime profile is still missing.

## Solution Assessment

No for broad module-level TTL caches as the default solution. A narrower flow-owned metadata handoff or auth/status coalescing fix is more consistent with the plugin metadata freshness boundary.
