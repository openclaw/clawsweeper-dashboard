## Summary

Keep open: current main still has the auth-status path that can re-enter external auth/provider plugin resolution and repeat the Codex bootstrap-only no-op decision, while the paired draft PR remains open and has not landed. The issue has useful reporter and independent reproduction context, but the final fix shape needs maintainer review because broad plugin metadata TTL caches conflict with the existing freshness boundary.

## What This Changes

Avoid repeated idle gateway auth/provider/plugin resolution that scans plugin manifests and re-logs unchanged Codex bootstrap-only local OAuth decisions.

## Best Possible Solution

Make unchanged auth-status and idle provider resolution cheap by reusing a gateway-owned plugin metadata snapshot/lookup table or a narrow auth no-op state guard while preserving fresh default discovery and manifest registry behavior.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection does show the repeated path from `models.authStatus` through `ensureAuthProfileStore`, plugin external-auth resolution, provider hook resolution, and the Codex bootstrap-only no-op log.

## Solution Assessment

Unclear for the final implementation: the issue direction is valid, but the linked PR’s broad 30-second plugin metadata caches are not the safest fix under the current plugin freshness boundary. A flow-owned snapshot or narrow auth/status invalidation guard is the more maintainable direction.
