## Summary

Keep open: current main still has a source-visible path from gateway auth status into external auth/provider resolution and synchronous plugin manifest metadata work, and the linked draft fix is still open with review blockers. This is not implemented on main and should remain paired with https://github.com/openclaw/openclaw/pull/81168 while maintainers decide the cache-boundary-safe fix shape.

## What This Changes

The issue reports gateway latency and idle CPU caused by repeated plugin manifest and registry scans during auth/provider resolution.

## Best Possible Solution

Keep the issue open and fix the hot path by reusing a caller-owned gateway/plugin metadata snapshot or narrowly memoizing unchanged auth-status no-op decisions, without adding default persistent discovery or manifest-registry caches.

## Reproduction Assessment

No live current-main reproduction was run in this read-only pass. Source inspection does show the gateway auth-status path re-entering external auth/provider resolution, and the reporter plus an independent commenter provided 2026.5.7 latency/syscall evidence for the same scan family.

## Solution Assessment

No for the currently linked draft PR as written: module-level TTL caches for discovery and manifest registries conflict with the plugin freshness boundary. A narrower flow-owned snapshot, lookup table, manifest registry handoff, or auth-status state-change guard is the safer maintainable direction.
