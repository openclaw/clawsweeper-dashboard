## Summary

Keep open: current main still has a source-visible path where gateway auth-status can re-enter external auth/provider resolution and the Codex bootstrap-only no-op, and the same-author draft PR at https://github.com/openclaw/openclaw/pull/81168 is still open rather than merged. ## What This Changes

The issue reports idle gateway auth/status polling repeatedly resolving external auth/provider plugins and re-statting plugin manifests, causing latency and idle CPU.

## Best Possible Solution

Fix the gateway polling cost by reusing a gateway-owned plugin metadata snapshot or narrowly coalescing unchanged auth-status work, while preserving default plugin metadata freshness and adding focused regression plus live latency proof.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the reported auth-status to external-auth/provider/plugin-resolution path, and the discussion includes two real deployment reports on v2026.5.7-era builds.

## Solution Assessment

No for the broad module-level TTL cache path in the linked draft PR; the current plugin boundary points toward flow-owned snapshot reuse or narrow auth/status coalescing. Keeping this issue open while that design is reviewed is the maintainable path.
