## Summary

Keep open: current main still has the source path where gateway auth-status can re-enter external auth/provider/plugin resolution and repeat the Codex bootstrap-only local-OAuth no-op, and the same reporter has an open draft PR that is not merged and was reviewed as needing proof/redesign.

## What This Changes

The issue asks OpenClaw to avoid repeated idle gateway auth/provider/plugin manifest work that causes latency and CPU load, especially around `openai-codex` bootstrap-only auth status.

## Best Possible Solution

Keep the issue open and fix current main by carrying an explicit plugin metadata snapshot/registry through the auth/provider flow or memoizing unchanged auth-status no-ops, without adding broad wall-clock plugin metadata caches.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the relevant current-main path when `models.authStatus` or another caller invokes `ensureAuthProfileStore` for a config-scoped Codex profile with existing local OAuth material.

## Solution Assessment

Yes for the underlying direction of making unchanged auth/provider polling cheap, but not via the linked broad 30-second metadata TTL cache. The maintainable path is explicit snapshot/registry reuse or narrow auth-status no-op invalidation under the current plugin cache boundary.
