## Summary

Keep open: current main still has the auth/status path that can re-enter plugin external-auth/provider resolution and repeat the Codex bootstrap-only local-OAuth decision. The related stale non-bootstrapOnly external-CLI fix does not cover this Codex bootstrap-only no-op case, and the broader plugin-discovery reports do not fully own the idle auth-poll trigger.

## What This Changes

Request to reduce idle gateway latency and CPU by avoiding repeated plugin manifest/provider scans and repeated bootstrap-only auth resolution during auth/provider polling.

## Best Possible Solution

Preserve Codex local-OAuth precedence while making unchanged auth-status and idle auth polls cheap through narrow invalidation/coalescing or state-change logging, with focused regression coverage.

## Reproduction Assessment

No live idle-gateway reproduction was run. Source inspection does show the repeated path when `models.authStatus` or another caller invokes `ensureAuthProfileStore` for a config-scoped Codex profile with existing local OAuth material.

## Solution Assessment

Yes for a narrow auth-path repair. Broad manifest metadata TTLs are not the best first fix because plugin metadata freshness is an explicit boundary, but memoizing unchanged auth no-ops or adding clear invalidation in the auth/status path is maintainable.
