## Summary

Keep open: current main still has a source-visible auth/status path that can re-enter plugin external-auth/provider resolution and repeat the Codex bootstrap-only local-OAuth decision. The broad metadata-rescan family overlaps with https://github.com/openclaw/openclaw/issues/78461, but this issue has a distinct idle auth/provider trigger and an open same-author draft PR at https://github.com/openclaw/openclaw/pull/81168, so it should not be closed during cleanup.

## What This Changes

Report requests reducing idle gateway auth/provider polling that repeatedly resolves plugin metadata/manifests and re-logs the Codex bootstrap-only local-OAuth decision.

## Best Possible Solution

Preserve Codex local-OAuth precedence while making unchanged auth/status polling cheap through flow-owned metadata reuse or narrow no-op memoization, then close this issue after the linked PR or a replacement fix lands on main.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does show the repeated path when `models.authStatus` or another caller invokes `ensureAuthProfileStore` for a config-scoped Codex profile with existing local OAuth material.

## Solution Assessment

No for the broad cache-window PR as currently reviewed, because persistent discovery and manifest-registry caches conflict with the plugin metadata freshness boundary. A narrower auth/status repair that reuses a flow-owned snapshot or memoizes unchanged no-op decisions is the better direction.
