## Summary

Keep open: current main still has a source-visible path from gateway auth status into external auth/provider plugin resolution, and a same-author draft PR already targets this report. The broader plugin metadata performance tracker remains related, but this issue preserves the idle auth/provider trigger and should stay open until a fix lands or the paired PR is resolved.

## What This Changes

Request to reduce idle gateway latency and CPU by avoiding repeated plugin manifest/provider registry work during auth/provider resolution, especially Codex bootstrap-only auth polling.

## Best Possible Solution

Reuse a gateway-owned `PluginMetadataSnapshot`/`PluginLookUpTable` or narrowly memoize unchanged auth-status no-op work so idle auth/provider polling is cheap while preserving manifest freshness; review, replace, or close https://github.com/openclaw/openclaw/pull/81168 as the paired implementation path.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection confirms the current-main auth-status path can re-enter external auth/provider/plugin registry resolution, and the issue includes concrete 2026.5.7 traces plus an independent non-Discord reproduction.

## Solution Assessment

Unclear for the currently linked cache-window PR: broad wall-clock metadata caches conflict with the plugin freshness boundary. The maintainable direction is to carry explicit gateway-owned metadata through the auth/provider flow or make unchanged auth-status decisions cheap with clear invalidation.
