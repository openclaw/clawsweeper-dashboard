## Summary

Keep open: current main still has a source-visible auth/status path that can re-enter external-auth/provider resolution and plugin metadata scanning, and the issue now has both the original 2026.5.7 gateway trace and an independent reproduction comment. The linked draft PR is still open and has already been reviewed as needing redesign/proof, so cleanup should not close this issue yet.

## What This Changes

The issue asks OpenClaw to make idle gateway auth/provider polling avoid repeated plugin manifest, registry, and bootstrap-only auth-profile resolution work.

## Best Possible Solution

Thread or reuse a gateway/auth-owned metadata snapshot or memoized unchanged auth-status decision through this path, coordinate any broader plugin metadata work with https://github.com/openclaw/openclaw/issues/78461, and avoid persistent discovery/manifest caches unless the plugin-boundary owners approve that policy change.

## Reproduction Assessment

No live current-main reproduction was run in this review. Source inspection plus the reporter’s 2026.5.7 strace and the independent 48.870s status trace show a concrete path through `models.authStatus`/external auth into provider and plugin metadata resolution.

## Solution Assessment

Unclear for the proposed broad cache direction. A narrower caller-owned snapshot or state-change/no-op guard is more consistent with the current plugin metadata freshness contract than the linked PR’s module-level 30-second discovery and manifest registry caches.
