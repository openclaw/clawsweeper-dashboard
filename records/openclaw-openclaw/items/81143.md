## Summary

Keep open: current main still has source paths where auth/status reads can re-enter external auth provider resolution and plugin discovery/manifest work, and the issue is actively paired with an open draft PR rather than resolved on main.

## What This Changes

The issue asks OpenClaw to stop idle gateway auth/provider status polling from repeatedly scanning plugin manifests and registry metadata, reducing latency and idle CPU.

## Best Possible Solution

Keep this open until a focused fix lands that makes idle auth/status/provider resolution reuse prepared auth/plugin facts or a clearly owned short-lived snapshot without violating plugin freshness guarantees.

## Reproduction Assessment

No current-main live reproduction was run. Source inspection shows the repeated auth/status to plugin-resolution path, and the issue comments provide two release-environment reproductions of the same slow status/plugin-metadata shape.

## Solution Assessment

Unclear for the linked broad cache-window approach. A narrower auth/status invalidation or prepared-snapshot reuse path is safer unless maintainers explicitly accept discovery and manifest registry TTL semantics.
