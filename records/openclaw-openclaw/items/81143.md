## Summary

Keep open: current main still has the repeated auth/status to external-auth/provider path and the manifest metadata scanner still fingerprints plugin manifests synchronously before cache hits. The issue is also paired with an open same-author draft PR, so closing it before that work is reviewed or superseded would be premature.

## What This Changes

The issue requests that idle gateway auth/provider resolution stop repeatedly scanning plugin manifests and repeating unchanged bootstrap-only auth decisions.

## Best Possible Solution

Keep this as the canonical tracker while maintainers review the paired draft PR or replace it with a narrower auth/status coalescing fix that preserves plugin metadata freshness contracts.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection plus the reporter and independent non-Discord trace give a medium-confidence reproduction path through `models.authStatus` or another auth-store caller that re-enters external-auth and plugin/provider resolution.

## Solution Assessment

Unclear. Fixing the repeated work is the right direction, but broad default cache windows need owner review because the plugin boundary guidance prefers explicit snapshot ownership over persistent discovery or manifest registry caches.
