## Label Justifications

- `P2`: DeepSeek V4 spend accounting is a user-visible regression with a limited provider and usage-summary blast radius.
- `impact:data-loss`: Derived session spend data is silently corrupted to zero despite nonzero token usage and known pricing.
- `impact:session-state`: The affected path reads persisted transcript usage/cost entries that feed session and Control UI spend summaries.

## Summary

Keep open: current main and the latest release still preserve a recorded `cost.total: 0` for known-priced DeepSeek V4 turns with nonzero tokens, and multiple live PRs explicitly reference this issue as their fix target.

## What This Changes

The issue asks OpenClaw to estimate DeepSeek V4 usage cost from token counts when the API records zero cost so Control UI Spend no longer shows false zero spend.

## Best Possible Solution

Land one canonical fix that recomputes known-priced, nonzero-token recorded-zero transcript costs, refreshes stale cached summaries, and covers both Spend summaries and session logs while preserving unknown-pricing behavior.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: a DeepSeek V4 transcript entry with nonzero usage and `usage.cost.total: 0` parses as a real zero and skips flat-pricing estimation. I did not run a live DeepSeek API call in this read-only review.

## Solution Assessment

Yes, the maintainable fix belongs in the shared session-cost aggregation/log path rather than the Control UI, with guards for known pricing and nonzero tokens. The open PRs are candidate implementations, so a new repair lane would duplicate existing work.
