## Summary

Keep open: current main and v2026.6.10 still preserve recorded `cost.total: 0` for known-priced, nonzero DeepSeek V4 usage, and the linked fix PRs are still open rather than merged.

## What This Changes

The issue asks OpenClaw to estimate DeepSeek V4 spend from token usage when the API/transcript records a zero cost despite known positive pricing.

## Best Possible Solution

Land one canonical fix that recomputes nonzero known-priced zero-cost transcript entries, refreshes stale usage-cost caches, and preserves unknown-pricing plus true zero-token behavior.

## Reproduction Assessment

Yes for a source-level reproduction: a transcript assistant entry with nonzero usage, DeepSeek provider/model, and `cost.total: 0` is parsed as a real zero and skips flat-pricing estimation. I did not run a live DeepSeek API call in this read-only review.

## Solution Assessment

Yes for the core fix shape: treat recorded zero as unknown only for nonzero usage with known positive pricing, clear stale breakdowns, and bump the usage-cost cache version. The broader PR choice should also verify whether all-zero DeepSeek model config rows need catalog hydration.
