## Label Justifications

- `P2`: Incorrect DeepSeek V4 spend accounting is a user-visible regression with a limited provider and usage-summary blast radius.
- `impact:data-loss`: Derived session spend data is silently corrupted to zero despite nonzero token usage and known pricing.
- `impact:session-state`: The affected path reads persisted session transcript usage/cost entries that feed Control UI spend summaries.

## Summary

Keep open: current `main` and latest release still preserve a recorded `cost.total: 0` for nonzero usage when the model has known flat pricing, and the DeepSeek V4 catalog entries do have positive pricing. I did not find a merged fix or canonical duplicate that already owns this exact zero-cost-with-known-pricing path.

## What This Changes

The issue requests that Control UI spend estimate DeepSeek V4 costs from token usage when transcripts contain nonzero usage but a recorded zero cost.

## Best Possible Solution

Ensure nonzero token usage with known positive pricing is estimated instead of preserved as a fabricated zero, while preserving true unknown-pricing and genuinely zero-token behavior.

## Reproduction Assessment

Yes for a source-level reproduction: a transcript entry with nonzero usage, DeepSeek provider/model, and `cost.total: 0` is parsed as a real zero and skips flat-pricing estimation. I did not run a live DeepSeek API call in this read-only review.

## Solution Assessment

Unclear as a final fix location: the summary bug is clear, but a repair should first check whether externalized DeepSeek runtime model resolution is dropping `model.cost`. The maintained end state should avoid DeepSeek-specific special casing unless the provider boundary proves it is needed.
