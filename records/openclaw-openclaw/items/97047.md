## Label Justifications

- `P2`: This is a user-visible cost-accounting regression with a limited DeepSeek V4 and usage-summary blast radius.
- `impact:data-loss`: Derived session spend data is silently corrupted to zero despite nonzero token usage and known positive pricing.
- `impact:session-state`: The affected path reads persisted session transcript usage/cost entries that feed session summaries and Control UI spend.
- `maturity:stable`: The affected `usage.cost` path maps to Gateway runtime `Gateway RPC APIs and Events` / `Usage and memory APIs`, and Gateway runtime is scored M4 Stable.

## Summary

Keep open: current main and latest release still accept a recorded zero `usage.cost.total` as real for known-priced DeepSeek V4 usage, while two open same-repo PRs already reference this issue as candidate fixes and neither has merged yet.

## What This Changes

The issue asks OpenClaw to estimate DeepSeek V4 spend from nonzero token usage when the API/transcript records `usage.cost.total: 0` despite known positive model pricing.

## Best Possible Solution

Review and land one canonical fix PR that treats known-priced nonzero-usage zero totals as stale, recomputes through existing pricing, preserves unknown-pricing behavior, and invalidates stale cached summaries.

## Reproduction Assessment

Yes, source-level: a transcript entry with nonzero usage, DeepSeek provider/model, and `cost.total: 0` is parsed as a real zero and skips flat-pricing estimation on current main. I did not run a live DeepSeek API call in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to normalize the zero-total case in the existing cost aggregation/pricing path with a regression test and cache refresh. The only open design check is whether the provider boundary should also stop writing fabricated zero cost metadata.
