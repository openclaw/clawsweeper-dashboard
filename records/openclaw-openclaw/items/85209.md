## Summary

Keep open for maintainer review. The PR is focused and now supplies terminal runtime proof for the formatter output, but it widens the public text-chunking SDK contract by allowing callable style openers, so that API direction needs explicit maintainer acceptance before merge.

## What This Changes

The PR preserves fenced-code language metadata through Markdown IR/rendering/chunking and updates Telegram HTML formatting/tests to emit `<pre><code class="language-...">` for language fences.

## Best Possible Solution

Land this shape after maintainers accept the additive text-chunking SDK contract; the linked issue should close through the merge, with optional Telegram Desktop proof if maintainers want UI confirmation.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram uses a static `<pre><code>` marker and the Markdown IR discards fence `token.info`, so a language fence cannot produce `class="language-bash"`; I did not run tests because this review is read-only.

## Solution Assessment

Yes, with maintainer API approval. Carrying fence-language metadata through the shared Markdown IR and render-aware chunking path is cleaner than parsing Markdown again in Telegram send code, but callable style openers need acceptance as a public SDK contract.
