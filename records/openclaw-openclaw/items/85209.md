## What This Changes

The PR carries fenced code block language metadata through Markdown IR/chunking and renders Telegram code blocks with `class="language-..."` plus a formatter regression test.

## Best Possible Solution

Carry a sanitized first fence-language token through Markdown IR while keeping the exported marker contract source-compatible, then prove the Telegram delivery path with redacted live output.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram code blocks render with a static `<pre><code>` opener and `markdown-it` fence `token.info` is not carried into Markdown IR; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The user-facing behavior matches Telegram's documented HTML shape, but the current patch should preserve the exported `RenderStyleMarker.open` string contract or get explicit maintainer approval before changing the plugin SDK surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep RenderStyleMarker.open string-compatible:** `src/markdown/render.ts:4`
  - body: `RenderStyleMarker` is re-exported from `openclaw/plugin-sdk/text-chunking`, so widening `open` from `string` to `string | ((span) => string)` can break plugin code that reads or forwards `marker.open` as a string after an SDK upgrade. Keep the existing string property stable and add a separate span-aware opener, or make this an explicitly approved public API change with compatibility coverage.
  - confidence: 0.86
