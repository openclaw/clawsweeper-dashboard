## Summary

Keep open: this PR targets a real Android handshake race, but the proposed ordering is not mergeable because it leaves onConnected callbacks seeing a connected state before currentConnection is available, fails the Android third-party unit-test lane, and lacks after-fix real behavior proof.

## What This Changes

The PR moves Android GatewaySession.currentConnection assignment until after conn.connect() and adds a Robolectric test for sendNodeEvent during a delayed connect handshake.

## Best Possible Solution

Keep the Android-side handshake guard, but publish connection readiness before onConnected-dependent sends can run or move onConnected until after currentConnection is ready; handle notification retry/replay separately if the linked issue still requires guaranteed delivery during reconnect.

## Reproduction Assessment

Yes. Source inspection shows current main exposes currentConnection before the connect RPC completes while the gateway requires the first request to be connect; I did not run a live Pixel reconnect scenario.

## Solution Assessment

No. Moving currentConnection after conn.connect() is directionally related, but conn.connect() fires onConnected before returning, so the PR creates a ready-callback gap and still does not prove the notification path is delivered after reconnect.

- **[P1] Keep the connection ready before firing onConnected:** `apps/android/app/src/main/java/ai/openclaw/app/gateway/GatewaySession.kt:903`

Set readiness before `onConnected` can publish events, or delay the callback until after the connection is installed.
