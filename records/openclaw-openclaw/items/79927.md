## Summary

Keep open: the guard addresses the invalid first-frame symptom, but it still drops reconnect notifications because the notification sink ignores a false send result and no queue or reconciliation path is added; the supplied proof is unit/mock based.

## What This Changes

The PR adds an Android GatewaySession handshake-complete flag, gates node.event send methods on it, and adds a MockWebServer regression test for pre-ACK sendNodeEvent calls.

## Best Possible Solution

Gate node events until authenticated connect completion, queue bounded notification changes during reconnect, flush or reconcile active notifications after readiness, and prove the path with redacted real reconnect evidence.

## Reproduction Assessment

Yes from source inspection: current main exposes currentConnection before conn.connect() completes, and notification forwarding can call sendNodeEvent in that window while the Gateway rejects non-connect first requests. I did not run a live Pixel/Gateway reconnect scenario in this read-only review.

## Solution Assessment

No: the handshake readiness guard is necessary, but by itself it converts the race into a false return that notification forwarding ignores. The narrow maintainable fix also needs bounded notification replay or active-notification reconciliation after reconnect.
