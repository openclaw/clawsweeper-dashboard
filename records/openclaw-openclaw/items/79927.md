## Summary

Keep open: the PR blocks the invalid first-frame symptom, but it still drops notification events that arrive before the handshake completes and the real behavior proof is mock-only. The linked notification-loss bug is not fixed well enough for merge as-is.

## What This Changes

This PR adds an Android `GatewaySession.Connection.isHandshakeComplete` flag, gates `sendNodeEvent` and `sendNodeEventDetailed` on it, and adds a unit test for sending before the gateway connect ACK.

## Best Possible Solution

Keep the handshake readiness guard, then preserve notification events across reconnect with bounded retry/queueing or active-notification reconciliation after handshake, covered by focused Android tests and redacted real reconnect proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main publishes `currentConnection` before the connect ACK, `sendNodeEvent` can send `node.event` as the first frame, and the gateway rejects that first frame with the reported handshake error.

## Solution Assessment

No. The handshake guard is useful but partial, because returning `false` to a notification sink that ignores the result still drops the reported notification event instead of delivering or reconciling it after the handshake completes.
