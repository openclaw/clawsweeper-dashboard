## Summary

Keep open: the handshake guard is useful but still drops notification events during reconnect because the notification path ignores a `false` send result and has no bounded replay or reconciliation path.

## What This Changes

The PR adds an Android `Connection.isHandshakeComplete` flag, gates `sendNodeEvent` and `sendNodeEventDetailed` on it, and adds a MockWebServer regression test.

## Best Possible Solution

Gate node events until authenticated connect completion, queue bounded notification changes during reconnect, flush/reconcile active notifications after readiness, and prove it with redacted real reconnect evidence.

## Reproduction Assessment

Yes from source inspection: current main exposes `currentConnection` before `conn.connect()` completes, and notification forwarding can send `node.event` during that window while the Gateway rejects non-connect first frames. I did not run a live Pixel/Gateway reconnect scenario.

## Solution Assessment

No: the handshake readiness guard is necessary, but by itself it turns the race into a `false` return that notification forwarding ignores. The narrow maintainable fix also needs bounded notification replay or reconciliation after reconnect.
