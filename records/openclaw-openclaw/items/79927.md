## Summary

Keep open: the PR addresses the pre-handshake protocol violation, but it still does not solve the linked notification-loss behavior because pre-handshake notification sends now return false and the notification path still has no retry or replay.

## What This Changes

The PR adds an Android `Connection.isHandshakeComplete` guard to `sendNodeEvent` and `sendNodeEventDetailed`, plus a Robolectric/MockWebServer regression test for pre-handshake sends.

## Best Possible Solution

Use an Android-side fix that gates node events until the authenticated connect handshake is complete, queues bounded notification events during reconnect, flushes/reconciles after readiness, and proves the real reconnect path with redacted runtime evidence.

## Reproduction Assessment

Yes from source inspection: current main publishes `currentConnection` before `conn.connect()` completes, and the Gateway rejects non-connect first frames. I did not run a live Pixel/Gateway reconnect scenario.

## Solution Assessment

No: the handshake readiness guard is necessary, but by itself it converts the race into a `false` return that the notification path ignores. The narrow maintainable fix also needs bounded notification retry/reconciliation after reconnect.
