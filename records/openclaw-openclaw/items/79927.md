## Summary

Keep open: the PR fixes the invalid pre-handshake frame path, but it still returns `false` to a notification forwarding path that ignores the result, so the reported reconnect-window notification loss remains. The supplied proof is also MockWebServer/Robolectric-style proof rather than inspectable real Android/Gateway reconnect evidence.

## What This Changes

The PR gates Android `sendNodeEvent` and `sendNodeEventDetailed` behind a new `Connection.isHandshakeComplete` flag and adds a unit test for sending before the connect ACK.

## Best Possible Solution

Keep the handshake readiness guard, then preserve notification events across reconnect with bounded queue/retry or active-notification reconciliation after handshake, covered by focused Android tests and redacted real reconnect proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main publishes `currentConnection` before the connect ACK, `sendNodeEvent` can send `node.event` in that window, and the gateway rejects a non-`connect` first frame. I did not establish a live Pixel reconnect reproduction in this read-only review.

## Solution Assessment

No: the handshake guard is useful but partial, because returning `false` to a notification sink that ignores the result still drops the reported event. The narrower maintainable fix should deliver or reconcile the notification after the handshake completes.
