## Summary

Keep open: the PR fixes the pre-handshake send attempt, but the reported user problem is notification loss during reconnect and this branch still drops notification events that arrive in that window.

## What This Changes

The PR adds an Android GatewaySession handshake-complete flag, gates node.event sends on it, and adds a MockWebServer regression test for pre-handshake sendNodeEvent.

## Best Possible Solution

Keep the handshake readiness guard, add bounded notification queuing or active-notification reconciliation across reconnect, and require redacted real reconnect proof before merge.

## Reproduction Assessment

Yes from source inspection: current main stores currentConnection before conn.connect() completes, the notification sink can send notifications.changed in that window, and the Gateway rejects non-connect first frames. I did not run a live Android reconnect scenario.

## Solution Assessment

No: the handshake guard is a necessary part of the fix, but by itself it converts the race into a false return that the notification path ignores, so the notification-loss bug remains.
