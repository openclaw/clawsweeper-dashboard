## Summary

Keep open: the PR addresses the pre-handshake `node.event` race, but it still leaves the linked notification-loss bug unresolved because notification sends that return `false` are not queued, retried, or reconciled after reconnect.

## What This Changes

The PR gates Android `sendNodeEvent` and `sendNodeEventDetailed` on a new handshake-complete flag and adds a MockWebServer unit test for pre-handshake sends.

## Best Possible Solution

Keep the handshake readiness guard, then preserve notification events across reconnect with bounded queue/retry or active-notification reconciliation after handshake, covered by focused tests and redacted real reconnect proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main publishes `currentConnection` before the connect ACK, notification forwarding can emit during that window, and the gateway rejects `node.event` as the first frame. I did not run a live Pixel reconnect scenario in this read-only review.

## Solution Assessment

No: the handshake guard is useful but partial, because returning `false` to a notification sink that ignores the result still drops the reported event. The maintainable fix should deliver or reconcile the notification after the handshake completes.
