## Summary

Keep open: this targets a real Android gateway race, but the patch is incomplete and currently breaks an existing connected-session test path; after-fix real Android/Gateway reconnect proof is also missing.

## What This Changes

The PR moves Android `GatewaySession.currentConnection` publication until after `conn.connect()` completes and adds a Robolectric test for pre-handshake `sendNodeEvent` returning false.

## Best Possible Solution

Restructure Android connection readiness so a handshaked connection is visible before connected callbacks run, then add bounded notification queue/replay and active-notification reconciliation after reconnect.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the original race: Android publishes `currentConnection` before the connect RPC completes, while the Gateway rejects non-`connect` first frames. The PR head also has CI proof of a connected-session regression, though I did not run a live Pixel reconnect scenario.

## Solution Assessment

No: moving the assignment after `conn.connect()` avoids pre-handshake frames, but it publishes the connection after `onConnected` can already run and still drops notifications instead of replaying them after reconnect.

## Review Findings

The linked bug is notification loss, so add bounded queue/retry and flush/reconcile after the ready connection is visible.
  - confidence: 0.9
