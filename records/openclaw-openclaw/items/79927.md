## Summary

Keep open: the PR identifies the Android WebSocket handshake race, but the guard-only patch still drops notification events during reconnect because the notification sink ignores a false send result and there is no replay or reconciliation path.

## What This Changes

The PR adds an Android GatewaySession handshake-completion flag, gates node.event sends on it, and adds a MockWebServer regression test for pre-handshake sendNodeEvent returning false.

## Best Possible Solution

Gate node events until authenticated connect completion and preserve reconnect-window notification changes with bounded queueing, flush, and active-notification reconciliation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main exposes currentConnection before the connect RPC completes, and notification forwarding can call sendNodeEvent in that window while the Gateway requires connect as the first frame. I did not run a live Pixel/Gateway reconnect scenario.

## Solution Assessment

No: the handshake readiness guard is necessary, but it converts the race into a false return that notification forwarding ignores. The maintainable fix also needs bounded notification replay or reconciliation after reconnect.
