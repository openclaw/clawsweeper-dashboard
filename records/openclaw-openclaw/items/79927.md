## Summary

Keep open: this PR prevents one invalid pre-handshake send path, but the linked Android notification-loss bug remains because notification forwarding ignores failed sends and has no queue, replay, or reconnect reconciliation.

## What This Changes

The PR adds an Android GatewaySession handshake-complete flag, gates node.event sends on it, and adds a MockWebServer regression test for pre-handshake sendNodeEvent.

## Best Possible Solution

Keep the handshake readiness guard, add bounded notification queue/replay or active-notification reconciliation in Android, and prove it with redacted real reconnect evidence.

## Reproduction Assessment

Yes, from source inspection: current main publishes currentConnection before conn.connect() completes, notification forwarding can call sendNodeEvent in that window, and the Gateway rejects non-connect first frames. I did not run a live Android reconnect scenario.

## Solution Assessment

No: the guard is necessary, but by itself it turns the race into a false return that the notification path ignores, so the reported notification loss remains.
