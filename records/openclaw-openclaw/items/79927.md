## Summary

Keep open: this PR prevents an early `node.event` from becoming the first WebSocket request, but it still drops notification events that arrive during the handshake window and it lacks contributor-supplied real Android/Gateway reconnect proof.

## What This Changes

The PR moves Android `GatewaySession.currentConnection` assignment until after `conn.connect()` and adds a Robolectric regression test for `sendNodeEvent` before handshake completion.

## Best Possible Solution

Use an Android-side fix that publishes the node connection only after the connect RPC completes, queues bounded notification events during reconnect, flushes after the ready connection is visible, and proves the path with redacted real reconnect evidence.

## Reproduction Assessment

Yes from source inspection: current main publishes `currentConnection` before `conn.connect()` completes, the Gateway rejects non-connect first frames, and the Android notification sink does not retry failed sends. I did not run a live Pixel/Gateway reconnect scenario.

## Solution Assessment

No: moving `currentConnection` after `conn.connect()` is a necessary readiness fix, but by itself it converts the early send into a dropped `false` result. The maintainable fix also needs bounded notification replay after reconnect.
