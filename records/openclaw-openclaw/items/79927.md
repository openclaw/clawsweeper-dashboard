## Summary

Keep open: the PR adds a useful handshake readiness guard, but it still drops notification events that arrive during the reconnect window because the Android notification sink ignores failed sends and has no queue, replay, or reconciliation path. The supplied proof is MockWebServer/Robolectric test proof, not inspectable real Android/Gateway reconnect proof.

## What This Changes

The PR gates Android `node.event` sends on a new `Connection.isHandshakeComplete` flag and adds a MockWebServer unit test for pre-handshake sends.

## Best Possible Solution

Keep the handshake readiness guard, then add bounded notification queue/retry plus active-notification reconciliation after reconnect, covered by focused tests and redacted real Android/Gateway proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main exposes `currentConnection` before the connect ACK, notification forwarding can call `sendNodeEvent` in that window, and the Gateway rejects non-`connect` first frames. I did not run a live Pixel reconnect scenario in this read-only review.

## Solution Assessment

No: the PR's guard is a useful partial fix, but returning `false` to a sink that ignores the result still loses the reported notification. The safer maintainable solution preserves or reconciles notifications after handshake completion.
