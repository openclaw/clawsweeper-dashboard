## Summary

Keep this PR open for changes: current main does expose the pre-handshake `node.event` race, but this patch only turns pre-handshake sends into `false` returns that Android notification forwarding ignores, so the linked notification-loss bug remains. The supplied proof is still MockWebServer/unit-test text rather than inspectable real reconnect evidence.

## What This Changes

The PR adds a handshake-complete flag to Android `GatewaySession`, gates `node.event` sends on that flag, and adds a MockWebServer regression test for pre-handshake sends.

## Best Possible Solution

Keep the handshake readiness guard, then preserve notification events with bounded queue/retry plus active-notification reconciliation after reconnect, covered by focused tests and redacted real Android/Gateway reconnect proof.

## Reproduction Assessment

Yes, from source inspection: current main assigns `currentConnection` before the connect ACK, Android notification forwarding can call `sendNodeEvent` in that window, and the gateway rejects a non-`connect` first frame. I did not run a live Android device reconnect scenario in this read-only review.

## Solution Assessment

No: the guard is a useful part, but by returning `false` to a notification sink that ignores the result, this PR still loses notifications during the reported reconnect window. The safer fix preserves or reconciles those events after handshake completion.
