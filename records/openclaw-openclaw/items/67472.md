## Summary

This PR should stay open with changes before merge. Current main still has the reported backup race, but the proposed `tar.Pack` implementation has correctness blockers: it can swallow Pack errors without completing the underlying Pack job, and it can publish archives whose manifest does not match archive-time omissions. The overlap with #72251 also needs coordination before landing competing backup-race behavior.

## What This Changes

The PR replaces backup creation's `tar.c` archiving call with a `tar.Pack` helper that records missing paths as skipped and adds a Pack mock to backup test support.

## Best Possible Solution

Revise or replace the patch with a backup archiving path that either avoids feeding known-volatile session/log files to tar or handles archiver errors in a way that actually drains/completes the tar job. The shipped behavior should keep result, manifest, and `--verify` semantics consistent, add a focused disappearing-session regression test, include the user-facing changelog entry, and reconcile the remaining volatile-file work with #72251.

## Reproduction Assessment

Yes. #67417 gives a concrete reproduction path: include live session transcripts in a backup plan, delete one before tar archives it, and current main can fail through `tar.c`. For this PR's patch, the dependency-level reproduction is to emit ENOENT from `tar.Pack` during stat/open and observe that returning from the Pack error listener does not complete the queued Pack job.

## Solution Assessment

No. `tar.Pack` may be a useful lower-level seam, but this implementation is not the best solution because it swallows Pack errors without completing jobs and appends skipped paths after the archive manifest has already been written. A safer solution should filter or retry volatile paths while preserving manifest and verification integrity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Complete skipped Pack jobs before resolving:** `src/infra/backup-create.ts:155-160`
  - body: When node-tar fails a Pack stat/open/read with ENOENT, Pack emits `error` but does not remove the queued `PackJob`; `JOBDONE` only runs on entry `end`. Returning from this handler leaves the queue non-empty and the write stream may never close, so `backup create` can hang instead of succeeding with a skipped file.
  - confidence: 0.88
- **[P2] Keep skipped assets out of the manifest:** `src/infra/backup-create.ts:415-422`
  - body: The manifest is written before archive-time skips are discovered, then `result.skipped` is updated after archiving. If an entire planned asset is skipped, `--verify` can fail the archive it just created because `manifest.assets` still declares missing payload.
  - confidence: 0.78
- **[P3] Separate Pack errors from stream errors in the mock:** `src/commands/backup.test-support.ts:37-40`
  - body: `MockPack` invokes Pack error listeners and then emits the same error on the output stream. That makes an ENOENT skip reject through `stream.on("error")` even though the production helper relies on Pack-level file errors being handled separately.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/infra/backup-create.ts:399`
  - body: This is a user-facing backup fix, but the diff only changes source and test support. OpenClaw policy requires a `CHANGELOG.md` entry for `fix` changes so the shipped behavior is visible in release notes.
  - confidence: 0.9
