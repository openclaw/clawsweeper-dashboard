## Summary

Keep this PR open for maintainer follow-up: it targets a real backup race, but the proposed Pack-based implementation still has blocking correctness issues and the later merged volatile-file work does not prove the deleted-before-lstat ENOENT case is fully gone.

## What This Changes

The branch replaces backup creation's `tar.c` call with a `tar.Pack` helper that records not-found paths as skipped and extends the backup test tar mock.

## Best Possible Solution

Rework or replace this PR with a backup archiving fix that handles expected per-entry ENOENT without hanging, keeps manifest and verification semantics consistent, and adds focused regression coverage plus real backup proof.

## Reproduction Assessment

Yes. The linked issue gives a concrete race path, and source plus node-tar contract inspection show that files can still disappear before tar's per-entry `lstat` completes; I did not run a live race in this read-only review.

## Solution Assessment

No. The PR targets the right failure mode, but swallowing Pack errors without draining jobs can hang the archive and appending skips after manifest creation leaves verification and restore metadata inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Complete skipped Pack jobs before resolving:** `src/infra/backup-create.ts:154-160`
  - body: When node-tar emits a Pack error for an ENOENT/ENOTDIR entry, returning from this listener does not remove the queued Pack job or decrement the job count. The output stream may never close, so `backup create` can hang instead of succeeding with a skipped path.
  - confidence: 0.88
- **[P2] Keep skipped payloads out of the manifest:** `src/infra/backup-create.ts:412-419`
  - body: The manifest is written before archive-time skips are discovered, then `result.skipped` is updated afterward. That can leave the archive manifest declaring payloads that were omitted, which breaks `--verify` semantics and makes restore metadata misleading.
  - confidence: 0.8
- **[P3] Separate Pack errors from stream errors in the mock:** `src/commands/backup.test-support.ts:37-40`
  - body: The mock invokes Pack error listeners and then emits the same error on the output stream. That makes the intended ENOENT skip reject through `stream.on("error")`, so the test helper cannot exercise the real production behavior this PR depends on.
  - confidence: 0.82
