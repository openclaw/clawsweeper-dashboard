## Summary

Keep open. Current main still has the backup archiving race, but this PR is not ready to merge because the Pack error handler can leave node-tar jobs unfinished, the archive manifest is written before archive-time skips are known, and the test mock cannot exercise the intended skip behavior. The overlapping open #72251 work also needs maintainer reconciliation before choosing the final live-backup strategy.

## What This Changes

The PR replaces backup creation's `tar.c` call with a `tar.Pack` helper that records missing paths as skipped and adds a Pack-capable tar mock for backup tests.

## Best Possible Solution

Land a backup archiving design that handles volatile session/log files without hanging and keeps CLI summary, embedded manifest, and `--verify` semantics consistent. The maintainable end state is likely either proactive volatile-path filtering plus targeted retry behavior, or a lower-level tar path that explicitly completes skipped jobs and rewrites or excludes manifest assets before publishing.

## Reproduction Assessment

Yes. #67417 gives a concrete race: plan live session files, delete one before tar archives it, and current main still passes the planned paths to `tar.c`. For this PR's patch, node-tar 7.5.13 source gives a dependency-level reproduction where an ENOENT emitted from Pack/WriteEntry is handled by the listener but the queued job never reaches `JOBDONE`.

## Solution Assessment

No. `tar.Pack` can be a useful lower-level seam, but this implementation is not the best solution because it swallows Pack errors without completing failed jobs and records skips after the archive manifest is already written. A safer alternative should filter known volatile paths and/or use a tested retry/skip path that preserves manifest verification integrity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Complete skipped Pack jobs before resolving:** `src/infra/backup-create.ts:155-161`
  - body: When node-tar fails a Pack stat/open/read with ENOENT, it emits `error` but only removes queued jobs through `JOBDONE` on entry `end`. Returning from this handler records the path but leaves the Pack queue/job state unfinished, so backup creation can hang waiting for the output stream `close` instead of succeeding.
  - confidence: 0.88
- **[P2] Keep archive-time skips out of the manifest:** `src/infra/backup-create.ts:415-421`
  - body: The manifest is written before `skippedPaths` are discovered, then the result is updated after archiving. If an entire planned asset disappears, the archive manifest still declares that payload, so `openclaw backup create --verify` can fail the archive it just created and non-verified archives have misleading manifests.
  - confidence: 0.78
- **[P3] Separate Pack errors from stream errors in the mock:** `src/commands/backup.test-support.ts:41-45`
  - body: `MockPack` calls Pack error listeners and then emits the same error on the output stream. That makes a handled ENOENT reject via `stream.on("error")`, so the test support cannot write a passing regression test for the intended skip path.
  - confidence: 0.82
- **[P3] Add the backup fix to the changelog:** `src/infra/backup-create.ts:399-413`
  - body: This is a user-facing `openclaw backup create` fix, but the PR does not add a `CHANGELOG.md` entry. OpenClaw policy requires release-note coverage for user-facing `fix` changes.
  - confidence: 0.9
