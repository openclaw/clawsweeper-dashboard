## What This Changes

## Best Possible Solution

Rebase or replace the branch with an archive fix that treats expected ENOENT/ENOTDIR as non-fatal while completing tar work, preserving current volatile-file filtering, keeping manifest/verify metadata coherent, and proving the deleted-session-file race in a real backup run.

## Reproduction Assessment

Yes, source-reproducible. Current main passes source roots to node-tar, node-tar stats queued paths before invoking the filter, and session cleanup can remove transcript files while backup walks the state tree; I did not run a live race in this read-only pass.

## Solution Assessment

No. The PR targets the right failure mode, but the current Pack error listener can leave tar work unresolved and the skipped-file result is recorded after the archive manifest is written.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Complete skipped Pack jobs before resolving:** `src/infra/backup-create.ts:155-161`
  - body: When Pack emits ENOENT/ENOTDIR during stat, this listener records the path and returns. node-tar leaves that failed job in its queue and only closes once the queue is empty, so backup create can hang instead of completing with a skipped entry.
  - confidence: 0.9
- **[P2] Keep manifest metadata aligned with skipped paths:** `src/infra/backup-create.ts:413-420`
  - body: The archive manifest is written before archive-time skips are known, then result.skipped is amended afterward. If a planned payload is omitted during packing, the embedded manifest can still declare it present or omit the skip record, so verify/restore tooling can see an inconsistent archive.
  - confidence: 0.84
- **[P3] Model Pack skips without failing the stream:** `src/commands/backup.test-support.ts:41-47`
  - body: The mock invokes Pack error listeners and then emits the same error on the output stream. A mocked ENOENT skip would still reject through stream.on("error"), so tests cannot cover the successful skip behavior this helper relies on.
  - confidence: 0.82
