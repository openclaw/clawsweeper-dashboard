## Summary

Keep open. Current main still has the reported archive-time race, and the proposed Pack-based patch needs revision because swallowing node-tar Pack errors does not complete the failed internal job, so the exact ENOENT path can hang instead of being skipped. The work also overlaps the still-open backup volatility work in #72251.

## What This Changes

The PR replaces backup creation's `tar.c` call with a `tar.Pack` plus write-stream helper that records ENOENT/ENOTDIR paths as skipped, and adds a Pack mock to backup test support.

## Best Possible Solution

Revise or replace the patch with a backup archiving design that either avoids feeding known-volatile session/log files to tar or uses an archiver path that can truly complete/drop failed file jobs. The final implementation should keep result, manifest, and verify semantics consistent, add a focused disappearing-session regression test, add the required changelog entry, and coordinate explicitly with #72251 so duplicate retry/filter logic does not land twice.

## Reproduction Assessment

Yes. #67417 provides a concrete race: build a backup plan that includes session transcripts, delete one between planning and tar archiving, and current main can fail because it still hands planned roots to `tar.c`. For this PR's patch, node-tar source gives a high-confidence focused check: emit ENOENT from Pack during stat/open and observe that simply returning from the Pack error listener does not complete the Pack job.

## Solution Assessment

No. `tar.Pack` may be a useful lower-level seam, but this implementation is not the narrowest correct solution because it swallows Pack errors without advancing node-tar's queue and leaves manifest/verify behavior inconsistent for whole-asset omissions. A safer fix should either preclude volatile paths before archiving or use a controlled archiver/retry path that preserves archive completion and manifest integrity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Do not swallow Pack errors without completing the job:** `src/infra/backup-create.ts:155-161`
  - body: `tar.Pack` does not treat an emitted file error as a completed skipped entry: stat/readdir/WriteEntry failures emit `error` but leave the PackJob in the queue, and the output stream only closes when the queue drains. Returning from this ENOENT handler records the path but leaves the promise waiting on `stream.close`, so the reported race can hang `backup create` instead of producing a backup.
  - confidence: 0.88
- **[P2] Keep the manifest in sync with archive-time skips:** `src/infra/backup-create.ts:397-414`
  - body: The manifest is written before `createTarArchiveWithMissingSkip` discovers skipped paths. If a whole planned asset disappears during packing, the archive omits that payload while `manifest.assets` still declares it, so `openclaw backup create --verify` can publish an archive and then fail verification for its own manifest. Rebuild the manifest or remove omitted assets before publishing.
  - confidence: 0.76
- **[P3] Model Pack errors separately from stream errors:** `src/commands/backup.test-support.ts:41-47`
  - body: The mock calls Pack error listeners and then emits the same error on the output stream. That is not the behavior the production helper relies on, and it prevents a regression test from proving that an ENOENT file error resolves with a skipped entry instead of rejecting through `stream.on('error')`. Keep Pack-level file errors distinct from output stream write failures.
  - confidence: 0.82
