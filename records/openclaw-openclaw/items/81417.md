## Summary

Keep open. The large-window default is a plausible direction, but this patch introduces a blocking threshold regression for mid-sized context windows and still lacks external real behavior proof.

## What This Changes

The PR makes memory-core's default memory flush soft threshold scale with the resolved model context window and plumbs that window through the memory flush plan resolver.

## Best Possible Solution

Use a shared positive-threshold calculation that scales only when enough post-reserve headroom remains, preserves mid-sized model flush behavior, updates the public memory-flush docs, and includes redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with the PR's 70% soft threshold and the default 20000 reserve, 32000-token and 64000-token windows produce a non-positive gate threshold, so the current gate returns null instead of flushing.

## Solution Assessment

No. Scaling `softThresholdTokens` against the whole context window before subtracting `reserveTokensFloor` is not the safest fix; the default should be derived through a positive-threshold helper or gated to windows where the trigger remains positive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep mid-sized windows from disabling memory flush:** `extensions/memory-core/src/flush-plan.ts:44`
  - body: The new default uses 70% of the whole context window as `softThresholdTokens`, but the runtime gate subtracts both reserve and soft threshold and then skips when the result is <= 0. With the default 20000 reserve, 32000-token and 64000-token models now get a non-positive trigger and token-based memory flush/preflight compaction never run, while the old 4000 default still left a positive trigger. Please gate the scaled default to large windows or compute it from available post-reserve headroom so mid-sized models keep flushing.
  - confidence: 0.93
