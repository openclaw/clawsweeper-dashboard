## Summary

Keep this PR open: the window-aware memory flush default may address a real long-context pain point, but the diff currently feeds that larger threshold into preflight compaction before the memory-flush phase, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes memory-core so the default memory-flush soft threshold can scale with the resolved model context window and threads that window through the memory flush resolver.

## Best Possible Solution

Keep the window-aware flush idea, but separate the memory-flush trigger from preflight compaction, document the new runtime default, and require real long-session or log-based proof before merge.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection confirms current main uses a fixed 4000 default and that the PR would lower the threshold used by both memory flush and preflight compaction for large-window runs.

## Solution Assessment

No: the general direction may be useful, but applying the scaled threshold to preflight compaction is not the best fix because that phase runs before memory flush. The safer solution is to make the memory-flush default window-aware without moving the preflight compaction gate earlier unless owners explicitly want that behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep scaled flush defaults out of preflight compaction:** `src/auto-reply/reply/agent-runner-memory.ts:523`
  - body: `runPreflightCompactionIfNeeded` runs before `runMemoryFlushIfNeeded`, and its token gate uses the same `contextWindow - reserve - softThreshold` formula. Passing the new window-scaled soft threshold here means a 1M-token stale/unknown session can compact around 280k tokens before the memory-flush phase runs, which defeats the goal of flushing durable memory before compaction. Keep this preflight gate on the existing conservative threshold or split the flush and compaction thresholds.
  - confidence: 0.88
- **[P3] Update the documented memory-flush default:** `extensions/memory-core/src/flush-plan.ts:155-156`
  - body: This change makes the effective default depend on the resolved model context window when runtime callers provide it, but the user docs still say `softThresholdTokens` defaults to `4000`. Please update the compaction docs/config help so operators understand why large-window models flush much earlier than the documented fixed default.
  - confidence: 0.77
