## Summary

This PR is still useful and should stay open, but it has one merge-blocking correctness issue: the new live final broadcast can point at the final TTS media while `chat.history` keeps the earlier block media supplement.

## What This Changes

The PR changes Gateway WebChat chat delivery so agent-run media final payloads can populate final `chat` broadcasts and adds focused handler plus running-gateway regression tests.

## Best Possible Solution

Keep the PR’s final-media broadcast path, but make the single assistant-media history entry converge on the final media when a final arrives after a block, with regression coverage for the live event and `chat.history` attachment identity.

## Reproduction Assessment

Yes, from source and PR proof rather than a local run: current main lacks a final broadcast path for ordinary agent-run TTS/media finals, and the PR’s running-gateway test exercises that gap. I did not run tests because this review is read-only.

## Solution Assessment

Not yet. Broadcasting the final media is the right direction, but the block-then-final path should keep the persisted history entry aligned with the final broadcast before merge.
