## Summary

Keep open: the feature is plausible, but this branch is not merge-ready because it changes existing Telegram behavior by default, sends visible reactions before the current authorization/mention gates, lacks the advertised runtime config schema, and still has no successful real Telegram proof.

## What This Changes

The PR adds a Telegram presence indicator with received and working reactions, a typing loop, message-processor hooks, per-account config types, an SDK config export, and unit tests.

## Best Possible Solution

Make presence explicitly opt-in or add a narrow tested migration, route visible reactions and typing through the existing authorization-aware Telegram feedback path, complete schema/docs/tests, and then attach redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows missing presence config defaults to enabled behavior and onMessageReceived runs before handleInboundMessageLike. I did not run live Telegram proof, and the Mantis run did not produce one.

## Solution Assessment

No. The proposed UX may be useful, but this implementation should preserve existing account behavior, reuse the authorization-aware Telegram feedback path, and complete the config contract before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve existing no-presence configs:** `extensions/telegram/src/presence-indicator.ts:117-124`
  - body: When `channels.telegram.presence` is absent, this factory still enables the default reactions and typing loop. Existing Telegram installs would gain visible bot activity on upgrade without opting in; make absence preserve current behavior or add an explicit, tested migration/default policy.
  - confidence: 0.91
- **[P1] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2377`
  - body: The received hook runs before `handleInboundMessageLike`, so it can react to messages that current Telegram handling later drops for disabled groups/topics, unauthorized DMs, allowlists, or mention gating. Move visible presence behind the existing accepted-message/ack gate.
  - confidence: 0.93
- **[P2] Add presence config to the runtime schema:** `src/config/types.telegram.ts:223-227`
  - body: The PR adds `channels.telegram.presence` only to the TypeScript config type. The Telegram Zod schema is strict and has no `presence` property, so users who try the advertised config will get an unknown-key validation error and generated metadata will not expose it.
  - confidence: 0.9
- **[P2] Honor reaction null for working reactions:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: The docs say `reaction: null` disables reactions entirely, but `workingEmoji` still falls back to the default unless `workingReaction` is also null. That lets `onProcessingStart` set and later clear a working reaction despite the advertised disable switch.
  - confidence: 0.86
