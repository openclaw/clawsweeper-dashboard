## Summary

Keep open: source review confirms blocking correctness and privacy issues in the proposed Telegram presence path, and the external PR still lacks real Telegram after-fix proof.

## What This Changes

The PR adds Telegram typing and reaction presence hooks plus per-account presence config for inbound Telegram message lifecycle status.

## Best Possible Solution

Fold presence into the existing authorized Telegram ack/status reaction pipeline, complete the config schema and docs, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: the diff shows the new received hook before `handleInboundMessageLike`, while current source and tests gate reactions after authorization and mention checks. I did not run tests or live Telegram proof because this review is read-only.

## Solution Assessment

No; the feature direction is plausible, but the current patch should share the existing authorized ack/status reaction gates and add the missing schema/docs before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2364-2375`
  - body: The new received hook runs before `handleInboundMessageLike`, so it visibly reacts to messages that current Telegram handling can later drop for disabled groups/topics, allowlists, DM authorization, or mention gating. Move the presence reaction behind the same message-context/ack gate so ignored chats do not get an acknowledgement.
  - confidence: 0.92
- **[P2] Add the presence config to the runtime schema:** `src/config/types.telegram.ts:223`
  - body: The PR advertises `channels.telegram[.accounts.*].presence` and adds only the TypeScript type, but Telegram config validation is strict and `TelegramAccountSchemaBase` has no matching key. Users who configure this will hit an unknown-key validation error until the schema and generated metadata/docs are updated.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions too:** `extensions/telegram/src/presence-indicator.ts:118-121`
  - body: With `presence: { reaction: null }`, `receivedEmoji` is disabled but `workingEmoji` still defaults to the working emoji, so processing still sets and clears a reaction. That contradicts the stated contract that `reaction: null` disables reactions entirely.
  - confidence: 0.86
