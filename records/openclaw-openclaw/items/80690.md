## Summary

Keep open and do not merge as-is; source inspection confirms blocking Telegram authorization, settings, config-contract, and duplicated lifecycle issues, and the external PR still lacks real Telegram behavior proof.

## What This Changes

The PR adds a Telegram presence indicator with received/working reactions, a typing loop, per-account presence config types, and unit coverage around the new lifecycle.

## Best Possible Solution

Route any Telegram presence UX through the existing authorized ack/status-reaction and typing-mode pipeline, or resolve the broader product direction in https://github.com/openclaw/openclaw/issues/6946, then add aligned schema/docs/tests and live Telegram proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the new hooks run before Telegram authorization, ignore the existing typing-mode contract, and advertise config that the strict runtime schema does not accept. I did not run live Telegram proof because this sweep is read-only and no contributor artifact is present.

## Solution Assessment

No; the intended UX is plausible, but this patch should not add a second default-on reaction/typing owner outside the existing authorized ack/status/typing pipeline. The maintainable path is to reuse those gates or make an explicit product decision for the canonical Telegram waiting-indicator issue first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Honor the existing typing-mode setting:** `extensions/telegram/src/presence-indicator.ts:118-119`
  - body: `agents.defaults.typingMode: "never"` is documented as disabling typing indicators entirely, but this new loop defaults `typingEnabled` to true and is only disabled by `presence.typing: false`. Existing users who intentionally disabled typing would see Telegram typing again after upgrade; thread this through the current typing-mode decision or reuse `context.sendTyping()` instead of adding an independent loop.
  - confidence: 0.91
- **[P1] Keep one Telegram reaction lifecycle owner:** `src/config/types.telegram.ts:223`
  - body: OpenClaw already has Telegram `ackReaction` plus `messages.statusReactions.enabled`, both gated by `ackReactionScope`. This adds a separate default-on `presence` controller that can race the existing status controller and clear or overwrite its reaction state, so extend/reuse the existing controller or make this an explicit replacement with migration/docs rather than a parallel lifecycle.
  - confidence: 0.84
- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2364-2375`
  - body: The new received hook runs before `handleInboundMessageLike`, so ignored groups, disabled topics, unauthorized DMs, and unmentioned group messages can still get a visible bot reaction. Move this behind the same context/authorization and ack gates that current reactions use.
  - confidence: 0.93
- **[P2] Do not advertise unvalidated presence config:** `src/config/types.telegram.ts:223`
  - body: This only adds `presence` to the TypeScript config type, but `TelegramAccountSchemaBase` is strict and has no matching schema entry or generated metadata. Users who try the documented `channels.telegram[.accounts.*].presence` key will get an unknown-key validation error unless the runtime schema/help/docs are updated or the new config is removed.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions too:** `extensions/telegram/src/presence-indicator.ts:118-121`
  - body: The config contract says `reaction: null` disables reactions entirely, but `workingEmoji` still falls back to the default unless `workingReaction` is also null. `presence: { reaction: null }` will still set and clear the working reaction, contradicting the advertised behavior.
  - confidence: 0.87
