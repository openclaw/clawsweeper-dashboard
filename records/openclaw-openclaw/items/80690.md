## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve existing Telegram presence defaults:** `extensions/telegram/src/bot-core.ts:583-588`
  - body: With `telegramCfg.presence` absent, the new indicator still defaults typing and both reactions on, so every existing Telegram account changes visible behavior on upgrade. Make this opt-in for existing configs, or add an explicit migration/setup default that only applies to new accounts.
  - confidence: 0.85
- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2375`
  - body: The received hook runs before `handleInboundMessageLike`, so it can react to messages later dropped for disabled groups/topics, unauthorized DMs, allowlists, or mention gating. Move this behind the existing accepted-message gate so ignored chats stay quiet.
  - confidence: 0.92
- **[P2] Add presence config to the runtime schema:** `src/config/types.telegram.ts:223-227`
  - body: The PR adds `channels.telegram.presence` only to the TypeScript config type. Telegram config validation is strict and has no `presence` property, so users who try the advertised config will get an unknown-key validation error and generated metadata will not expose it.
  - confidence: 0.9
- **[P2] Honor reaction null for working reactions:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: `presence.reaction: null` disables the received emoji but `workingEmoji` still falls back to the default unless `workingReaction` is also null. That contradicts the advertised contract that `reaction: null` disables reactions entirely.
  - confidence: 0.86
