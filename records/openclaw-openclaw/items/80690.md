## Summary

Keep open: the Telegram UX idea is plausible, but the current patch has blocking correctness issues, a privacy-sensitive pre-authorization reaction path, and no real Telegram behavior proof.

## What This Changes

The PR adds Telegram received/working reactions, a typing-loop presence indicator, presence config types, and unit/processor tests around inbound message handling.

## Best Possible Solution

Fold presence into the existing authorized ack/status reaction pipeline, complete schema/metadata/docs coverage for the config, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the new received hook runs before handleInboundMessageLike, while current main gates drops and reactions inside the authorized message-context path. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No; the UX should reuse the existing authorized ack/status reaction path and add matching schema/docs/tests instead of adding a pre-handler side effect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2364-2375`
  - body: The new received hook runs before handleInboundMessageLike, so it can react to messages that current Telegram handling later drops for disabled groups/topics, DM authorization, allowlists, or mention gating. That leaks a visible acknowledgement into chats OpenClaw is configured to ignore; move the hook behind the existing authorized message-context/ack gate or share that decision.
  - confidence: 0.92
- **[P2] Add presence to the runtime schema:** `src/config/types.telegram.ts:227`
  - body: This adds channels.telegram presence only to the TypeScript config type, but Telegram config validation is strict and TelegramAccountSchemaBase has no presence property. Users who try the advertised config will get an unknown-key validation error, and generated channel metadata will not expose it.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions too:** `extensions/telegram/src/presence-indicator.ts:118-121`
  - body: With presence: { reaction: null }, receivedEmoji is disabled but workingEmoji still falls back to the default, so onProcessingStart continues to set and later clear a reaction. That contradicts the stated config contract that reaction:null disables reactions entirely.
  - confidence: 0.86
