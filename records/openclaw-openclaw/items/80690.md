## Summary

Keep open. The Telegram presence UX is plausible, but this PR is not merge-ready because it adds a competing default-on reaction/typing path, sends visible reactions before existing authorization and mention gates, lacks the runtime schema for the advertised config, and the Mantis artifact I inspected does not show the target Telegram lifecycle.

## What This Changes

Adds a Telegram presence indicator with received and working reactions, a typing loop, message-processor hooks, per-account config types, and unit coverage.

## Best Possible Solution

Fold any accepted Telegram presence UX into the existing authorized ack/status-reaction and typing pipeline, preserve current user settings by default, add schema/docs/tests for any new config, and attach redacted live Telegram proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR sends onMessageReceived before handleInboundMessageLike and defaults a new reaction path outside the existing gates. I did not run live Telegram because this is a read-only review, and the available Mantis artifact did not show the target behavior.

## Solution Assessment

No. The desired visible feedback should be implemented through the existing authorized ack/status-reaction and typing controls, with schema/docs and proof, rather than a parallel default-on lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route presence through existing reaction controls:** `extensions/telegram/src/presence-indicator.ts:113-119`
  - body: With no presence config, the new indicator defaults typing and reactions on for every Telegram account. That bypasses existing user choices in messages.ackReaction, ackReactionScope, statusReactions, removeAckAfterReply, and typingMode, creating a competing source of truth and changing upgrade behavior; integrate this with the existing controls or make any new behavior explicitly opt-in with a tested migration.
  - confidence: 0.88
- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2364-2375`
  - body: The received hook runs before handleInboundMessageLike, so it can visibly react to messages that current Telegram handling later drops for disabled groups/topics, unauthorized DMs, allowlists, or mention gating. Move this behind the existing accepted-message/ack gate so ignored chats stay quiet.
  - confidence: 0.93
- **[P2] Add presence config to the runtime schema:** `src/config/types.telegram.ts:223-227`
  - body: The PR adds channels.telegram.presence only to the TypeScript config type. Telegram config validation is strict and TelegramAccountSchemaBase has no presence property, so users trying the advertised config will hit an unknown-key validation error and generated metadata will not expose it.
  - confidence: 0.9
- **[P2] Honor reaction null for working reactions:** `extensions/telegram/src/presence-indicator.ts:114-119`
  - body: The PR documents presence.reaction: null as disabling reactions entirely, but workingEmoji still falls back to the default unless workingReaction is also null. onProcessingStart can still set and later clear a reaction, contradicting the advertised config contract.
  - confidence: 0.86
