## Summary

Keep this PR open: the feature is plausible, but the patch currently introduces visible Telegram reactions before the existing authorization and mention gates, the advertised config key is not accepted by the strict config schema, and the PR still lacks real Telegram behavior proof.

## What This Changes

The PR adds Telegram presence indicator hooks, a new presence-indicator module, tests, and Telegram presence config types for received/working reactions plus typing.

## Best Possible Solution

Fold presence into the existing authorized ack/status reaction pipeline, complete the config schema/docs/tests, and require a live Telegram proof artifact before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the new presence hook runs before handleInboundMessageLike, while current main only reacts after authorization and mention/body resolution. I did not run tests or live Telegram proof because this sweep is read-only.

## Solution Assessment

No; the intended UX is reasonable, but the current patch should reuse the existing authorized ack/status reaction gates and add the matching config schema/docs before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2364-2375`
  - body: The new received hook runs before `handleInboundMessageLike`, so it will react to messages that current Telegram handling later drops for group/topic disablement, DM authorization, allowlists, or mention gating. That leaks a visible bot acknowledgement into chats OpenClaw is configured to ignore; move this behind the existing message-context/ack gate or make it share the same authorization decision.
  - confidence: 0.92
- **[P2] Add the presence config to the runtime schema:** `src/config/types.telegram.ts:223`
  - body: This adds `channels.telegram[.accounts.*].presence` only to the TypeScript config type, but Telegram config validation is strict and `TelegramAccountSchemaBase` has no `presence` property. Users who try the advertised config will get an unknown-key validation error, and generated channel metadata/docs will not expose it.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions too:** `extensions/telegram/src/presence-indicator.ts:118-121`
  - body: With `presence: { reaction: null }`, `receivedEmoji` is disabled but `workingEmoji` still falls back to the default, so `onProcessingStart` continues to set and later clear a reaction. That contradicts the config contract saying `reaction: null` disables reactions entirely; either derive `workingEmoji` from the disabled reaction state or rename/document the knobs differently.
  - confidence: 0.86
