  - body: `onMessageReceived` is called before `handleInboundMessageLike` reaches `authorizeInboundMessage`, so a DM or group update that current main would skip can still make the bot call `setMessageReaction`.   - confidence: 0.91

  - body: This type advertises `channels.telegram.presence`, but `TelegramAccountSchemaBase` remains strict and does not accept that key. Users cannot configure the feature as described, and generated metadata/help/docs will not know about it until the schema and docs surfaces are updated together.
  - confidence: 0.93
- **[P1] Make existing accounts opt in to presence calls:** `extensions/telegram/src/presence-indicator.ts:112-118`
  - body: When `presence` is absent, the new indicator still defaults typing and both reaction phases on, so every existing Telegram account would start making extra Bot API calls after upgrade. Keep existing accounts at current behavior unless maintainers explicitly approve and document the default-on migration.
  - confidence: 0.88
- **[P1] Do not clear reactions outside the existing owner:** `extensions/telegram/src/presence-indicator.ts:274-276`
  - body: `onProcessingEnd` clears the inbound message reaction independently of the existing ack/status reaction controller. That can erase the current default ack reaction or configured status reaction even when `removeAckAfterReply` is false, so the presence lifecycle needs to coordinate with the existing reaction owner instead of issuing its own unconditional clear.
  - confidence: 0.89
