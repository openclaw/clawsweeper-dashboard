## Summary

Keep open: the Telegram presence UX is plausible, but the PR still has blocking correctness and privacy issues, incomplete config validation, and no required real Telegram proof.

## What This Changes

Adds Telegram presence indicator hooks, tests, and a per-account presence config type for received/working reactions plus typing.

## Best Possible Solution

Fold presence into the existing authorized ack/status reaction pipeline, complete schema/metadata/docs/tests for the config, preserve opt-out semantics, and require redacted live Telegram proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the received hook runs before handleInboundMessageLike, while current main only reacts after authorization/body/mention gates. I did not run tests or live Telegram proof in this read-only sweep.

## Solution Assessment

No; the direct pre-dispatch lifecycle is not the best fix because it bypasses existing authorization and ack/status reaction gates. The safer direction is to reuse that authorized pipeline and finish the config contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2377`
  - body: The received hook runs before `handleInboundMessageLike`, so messages later dropped by group/topic disablement, DM authorization, allowlists, or mention gating still get a visible reaction. Move the presence lifecycle behind the same context/ack decision so ignored chats stay ignored.
  - confidence: 0.92
- **[P2] Add presence to the runtime config schema:** `src/config/types.telegram.ts:227`
  - body: This advertises `channels.telegram.presence` in the TypeScript config type, but `TelegramAccountSchemaBase` is strict and has no `presence` key. Users who configure the new option will hit unknown-key validation and generated metadata/docs will not expose it.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: With `presence: { reaction: null }`, `receivedEmoji` is disabled but `workingEmoji` still defaults to the working emoji, so processing still sets and clears a reaction. That contradicts the stated config contract that `reaction: null` disables reactions entirely.
  - confidence: 0.86
