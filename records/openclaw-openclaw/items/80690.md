## Summary

Keep open. The Telegram UX idea is plausible, but the branch still has blocking correctness, config-contract, security/privacy, and real-behavior-proof issues, and current main has not superseded the PR.

## What This Changes

Adds Telegram presence hooks and config for received/working reactions plus a typing loop, with unit tests and config type exports.

## Best Possible Solution

Fold presence into the existing authorized ack/status reaction pipeline, add the strict schema/metadata/docs for the config, settle default behavior, and require live Telegram proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection of the branch shows onMessageReceived runs before handleInboundMessageLike, while current main gates ack/status reactions after authorization and mention resolution. I did not run live Telegram proof in this read-only sweep.

## Solution Assessment

No; the feature should reuse the current message-context authorization and ack/status reaction gates, with schema/docs/proof, instead of adding a pre-dispatch reaction lifecycle.
