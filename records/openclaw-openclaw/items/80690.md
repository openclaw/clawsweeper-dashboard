Full review comments:

- **[P1] Move received presence behind Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2377`
  - body: This hook runs before `handleInboundMessageLike` checks `shouldSkipUpdate` and `authorizeInboundMessage`, so OpenClaw can visibly react to blocked senders, unconfigured channel posts, or ambient group messages that current main would later drop. Fire the received indicator only after the existing inbound gate decides the message is allowed and in scope.
  - confidence: 0.93
- **[P1] Wire the presence option into the runtime schema:** `src/config/types.telegram.ts:227`
  - body: The PR adds `presence` to the TypeScript config contract, but `TelegramAccountSchemaBase` is strict and does not include that object. Users who try opt-outs like `presence.typing: false` or `presence.reaction: null` will have invalid config, so schema, generated metadata/docs, and validation coverage need to land with the type export.
  - confidence: 0.9
- **[P1] Preserve the existing reaction lifecycle:** `extensions/telegram/src/bot-message.ts:195-198`
  - body: Current Telegram already has scoped ack/status reactions and dispatch finalization for the inbound message reaction slot; this new `finally` cleanup clears that same slot after dispatch returns. That can race or overwrite configured `messages.statusReactions` and ack-removal behavior, so fold presence into the existing controller instead of running a parallel lifecycle.
  - confidence: 0.86
- **[P1] Do not enable presence for existing accounts by default:** `extensions/telegram/src/presence-indicator.ts:117-124`
  - body: With `deps.config ?? {}`, missing `presence` enables both reactions and typing for existing Telegram accounts that never opted in. That silently changes visible behavior and adds new Telegram API calls on upgrade; keep old behavior as the default unless maintainers explicitly approve and document the compatibility change.
  - confidence: 0.86
