## Summary

Keep open: the Telegram presence UX is useful, but the current PR introduces blocking correctness and privacy issues, leaves the advertised config unsupported by the runtime schema, and lacks required real Telegram behavior proof.

## What This Changes

This PR adds Telegram received/working reactions, a typing-loop presence indicator, related processor hooks, tests, and a new `channels.telegram[.accounts[*]].presence` TypeScript config type.

## Best Possible Solution

Fold presence into the existing authorized ack/status reaction pipeline, add the matching runtime schema/docs/tests, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows the received reaction fires before `handleInboundMessageLike`, while current main only reacts after authorization and mention/body resolution. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No; the UX direction is reasonable, but this patch should reuse the existing authorized ack/status reaction gates and complete the config schema/docs before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2377`
  - body: The new received hook runs before `handleInboundMessageLike`, so it can react to messages that current Telegram handling later drops for group/topic disablement, DM authorization, allowlists, or mention gating. That leaks a visible bot acknowledgement into chats OpenClaw is configured to ignore; move this behind the existing message-context/ack gate or make it share the same authorization decision.
  - confidence: 0.92
- **[P2] Add presence to the Telegram config schema:** `src/config/types.telegram.ts:223-227`
  - body: This adds `channels.telegram[.accounts.*].presence` only to the TypeScript config type, but Telegram config validation is strict and `TelegramAccountSchemaBase` has no `presence` property. Users who try the advertised config will get an unknown-key validation error, and generated channel metadata/docs will not expose it.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: With `presence: { reaction: null }`, `receivedEmoji` is disabled but `workingEmoji` still falls back to the default, so `onProcessingStart` continues to set and later clear a reaction. That contradicts the config contract saying `reaction: null` disables reactions entirely; either derive `workingEmoji` from the disabled reaction state or rename/document the knobs differently.
  - confidence: 0.86
