## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Gate presence reactions behind inbound authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2369-2376`
  - body: This fires the received reaction before `handleInboundMessageLike` runs the Telegram group skip and DM authorization checks. A mention-skipped group message, disabled channel post, or unauthorized DM can therefore still get a bot reaction even though the normal pipeline would ignore it, which leaks bot presence across the Telegram authorization boundary.
  - confidence: 0.95
- **[P1] Do not enable presence by default for existing accounts:** `extensions/telegram/src/presence-indicator.ts:117-124`
  - body: With `deps.config` absent, the new presence indicator enables typing and both default reactions. `bot-core` passes `telegramCfg.presence`, so every existing Telegram account without the new key would start sending reactions/typing after upgrade, even if the operator never opted into this new behavior or had reaction-related settings disabled.
  - confidence: 0.9
- **[P2] Wire the presence config into the runtime schema:** `src/config/types.telegram.ts:227`
  - body: This adds a public TypeScript config type and SDK export, but `TelegramAccountSchemaBase` is still strict and has no `presence` field. Users cannot actually set `presence.typing: false` or `presence.reaction: null`; validation will reject the config before the new opt-out can reach the code.
  - confidence: 0.93
