## Summary

Keep this PR open, but do not merge it as-is: the feature is plausible, yet the patch visibly reacts before Telegram authorization/mention gates, advertises config that the strict runtime schema will reject, leaves `reaction: null` semantically inconsistent, and has no real Telegram proof.

## What This Changes

The PR adds Telegram received/working reaction and typing presence hooks, per-account presence config types, and unit tests for the new presence lifecycle.

## Best Possible Solution

Revise the PR so presence shares the authorized Telegram ack/status reaction pipeline, complete schema/metadata/docs for the new config, align disable semantics, and attach redacted live Telegram proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head and current main shows the new received hook runs before existing Telegram authorization, mention, and ack gates. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No; the intended UX is reasonable, but this implementation should reuse the existing authorized ack/status reaction path and complete the runtime config contract before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2377`
  - body: The new received hook runs before `handleInboundMessageLike`, so OpenClaw visibly reacts to messages that current Telegram handling may later drop for disabled groups/topics, DM authorization, allowlists, or mention gating. Move this behind the existing message-context/ack gate or make it share the same authorization decision.
  - confidence: 0.92
- **[P2] Add presence to the runtime schema:** `src/config/types.telegram.ts:227`
  - body: This adds `channels.telegram[.accounts.*].presence` only to the TypeScript config type, but Telegram config validation is strict and `TelegramAccountSchemaBase` has no matching `presence` property. Users who try the advertised config will hit an unknown-key validation error and generated channel metadata will not expose it.
  - confidence: 0.9
- **[P2] Honor reaction null for all reaction phases:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: With `presence: { reaction: null }`, `receivedEmoji` is disabled but `workingEmoji` still falls back to the default, so processing can still set and later clear a reaction. That contradicts the documented contract that `reaction: null` disables reactions entirely.
  - confidence: 0.86
