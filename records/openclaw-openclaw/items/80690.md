## Summary

Keep open; this PR is not merge-ready because the implementation changes existing Telegram accounts to visible default-on presence, sends received reactions before existing authorization gates, advertises config that the strict runtime schema does not accept, and still lacks real Telegram proof.

## What This Changes

The PR adds a Telegram presence lifecycle with received and working reactions, a typing loop, per-account config types, an SDK config export, and unit tests.

## Best Possible Solution

Make Telegram presence opt-in or add an explicit tested migration, route visible reactions and typing only after existing acceptance gates, complete schema/docs/metadata/tests, and then attach redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows absent presence config defaults to enabled behavior, and `onMessageReceived` runs before `handleInboundMessageLike`. I did not run live Telegram proof in this read-only review, and the Mantis proof comment reports skipped captures.

## Solution Assessment

No. The proposed UX may be useful, but the implementation should preserve existing account behavior, reuse the authorization-aware Telegram feedback path, complete the config contract, and prove the real Telegram behavior before it is maintainable.
