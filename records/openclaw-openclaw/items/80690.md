## Summary

Keep open. The Telegram presence UX is useful, but this branch is not merge-ready: it sends visible reactions before Telegram authorization/mention gates, advertises config that the runtime schema does not accept, mishandles the documented reaction-disable knob, and still lacks real Telegram proof.

## What This Changes

Adds a Telegram presence indicator module with received/working reactions, a typing loop, message-processor hooks, config type exports, and unit coverage.

## Best Possible Solution

Rework presence through the existing authorized ack/status reaction path, add the runtime schema/docs/tests for any new config, preserve typing-mode semantics, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR sends onMessageReceived before handleInboundMessageLike, while current main gates reactions after authorization and mention/body resolution. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The intended UX is reasonable, but the current implementation should share the existing authorized ack/status reaction pipeline and complete the config schema/docs/tests before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2375`
  - body: The received hook runs before handleInboundMessageLike, so it can react to messages that current Telegram handling later drops for disabled groups/topics, unauthorized DMs, allowlists, or mention gating. That leaks visible bot activity into chats OpenClaw is configured to ignore; move this behind the existing accepted-message/ack gate.
  - confidence: 0.92
- **[P2] Add presence config to the runtime schema:** `src/config/types.telegram.ts:223-227`
  - body: The PR adds channels.telegram.presence only to the TypeScript config type. Telegram config validation is strict and TelegramAccountSchemaBase has no presence property, so users who try the advertised config will get an unknown-key validation error and generated metadata will not expose it.
  - confidence: 0.9
- **[P2] Honor reaction null for working reactions:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: With presence.reaction set to null, receivedEmoji is disabled but workingEmoji still falls back to the default unless workingReaction is also null. That contradicts the advertised contract that reaction:null disables reactions entirely, because onProcessingStart can still set and later clear a reaction.
  - confidence: 0.86
