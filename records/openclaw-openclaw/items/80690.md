## Summary

Keep this PR open, but it is not merge-ready: the feature is plausible, the branch has blocking Telegram authorization/config findings, a concrete privacy concern, and no required real Telegram proof.

## What This Changes

The PR adds Telegram received/working reactions, a processing typing loop, presence config types/exports, tests, and hooks in the Telegram handler and message processor.

## Best Possible Solution

Fold presence into the existing authorized Telegram ack/status reaction pipeline, complete schema/docs/tests, coordinate with the narrower waiting-reaction work, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR sends presence before Telegram authorization gates, while current main gates ack/status reactions after context and mention resolution. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No; the desired UX is reasonable, but this patch bypasses existing Telegram authorization and config contracts. The safer solution is to reuse the current authorized status-reaction pipeline and add the missing schema/docs/proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2364-2375`
  - body: The new received hook runs before `handleInboundMessageLike`, so it can react to messages that current Telegram handling later drops for disabled groups/topics, DM policy, allowlists, or mention gating. That leaks visible bot activity into chats OpenClaw is configured to ignore; move this behind the existing message-context/ack gate or share the same authorization decision.
  - confidence: 0.92
- **[P2] Add the presence config to the runtime schema:** `src/config/types.telegram.ts:223`
  - body: This adds `channels.telegram[.accounts.*].presence` only to the TypeScript config type, but `TelegramAccountSchemaBase` is strict and has no matching property. Users who try the advertised config will get an unknown-key validation error, and generated metadata will not expose it.
  - confidence: 0.9
- **[P2] Honor presence reaction opt-outs consistently:** `extensions/telegram/src/presence-indicator.ts:113-121`
  - body: `presence: { reaction: null }` disables the received emoji but still falls back to the default working emoji unless `workingReaction` is also null, so processing still sets and clears a reaction. That contradicts the advertised contract that `reaction: null` disables reactions entirely; derive both states from one disabled reaction state or document separate knobs.
  - confidence: 0.87
