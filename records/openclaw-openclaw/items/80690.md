## Summary

Keep open: the proposed Telegram presence UX is useful, but this PR currently reacts before Telegram authorization/mention gates, adds a type-only config key that strict runtime validation will reject, and lacks required real Telegram proof.

## What This Changes

The PR adds Telegram received/working reactions plus typing-loop presence hooks, tests, and a new per-account `presence` config type.

## Best Possible Solution

Implement presence through the existing authorized ack/status reaction pipeline, add the matching strict schema/docs/generated metadata and regression tests, and require a redacted live Telegram proof artifact before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR hook runs before `handleInboundMessageLike`, while current main sends ack/status reactions only after authorization and mention/body resolution. The config rejection is also source-reproducible because the strict Telegram runtime schema lacks `presence`.

## Solution Assessment

No; the intended UX is reasonable, but this patch should reuse the existing authorized ack/status reaction gate and complete the runtime config contract before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate presence after Telegram authorization:** `extensions/telegram/src/bot-handlers.runtime.ts:2367-2377`
  - body: The new received hook runs before `handleInboundMessageLike`, so it can visibly react to messages later dropped for group/topic disablement, DM authorization, allowlists, or mention gating. Move the presence acknowledgement behind the same authorization and ack/status gate used by current reactions.
  - confidence: 0.92
- **[P2] Add presence to the runtime schema:** `src/config/types.telegram.ts:223-227`
  - body: This exposes `channels.telegram[.accounts.*].presence` only in the TypeScript config type, but Telegram config validation is strict and `TelegramAccountSchemaBase` has no matching property. Users who configure the advertised key will hit an unknown-key validation error and generated metadata will not expose it.
  - confidence: 0.9
- **[P2] Make reaction null disable working reactions too:** `extensions/telegram/src/presence-indicator.ts:119-124`
  - body: With `presence: { reaction: null }`, `receivedEmoji` is disabled but `workingEmoji` still falls back to the default, so processing still sets and clears a reaction. That contradicts the config contract that `reaction: null` disables reactions entirely.
  - confidence: 0.86
