## Summary

Keep open: current main and the latest release still allow equivalent cron jobs to be appended or imported as separate enabled rows, while related PRs only cover narrower or rejected duplicate-detection shapes.

## What This Changes

The issue asks OpenClaw update and cron migration paths to avoid silently duplicating enabled cron jobs, or at least warn operators about duplicates that can change agent/session routing.

## Best Possible Solution

Add maintainer-approved cron duplicate audit and repair semantics that preserve valid independent jobs, detect provenance-backed update/migration duplicates, and warn or clean existing rows through doctor before changing runtime rejection or upsert behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main accepts repeated equivalent `cron.add` calls as distinct jobs and legacy migration merges only matching ids/jobIds. I did not run a full `openclaw update` replay in this read-only review.

## Solution Assessment

Unclear until maintainers choose semantics. Core cron state and doctor migration are the right boundary, but a naive schedule-identity dedupe was already rejected as unsafe for valid independent jobs.
