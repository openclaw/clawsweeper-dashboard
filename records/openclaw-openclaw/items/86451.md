- `impact:message-loss`: The reported failure produces duplicate scheduled outbound messages and repeated cron deliveries.

## Summary

Keep open: current main and v2026.6.8 still deduplicate legacy cron migration only by id/jobId, while cron.add still creates a fresh id and appends a new enabled job. The related open PR covers name-collision rejection only, is not merged, and does not solve the reported update/JSON-to-SQLite equivalent-job duplication path.

## What This Changes

The issue requests that OpenClaw update/doctor cron migration avoid appending duplicate enabled cron entries, either by upserting equivalent jobs or warning operators with duplicate IDs.

## Best Possible Solution

Define the cron duplicate identity and upgrade policy, then implement a doctor/update duplicate audit or reconciliation path with focused migration tests and real update proof.

## Reproduction Assessment

Unclear for the full end-to-end openclaw update flow. Source inspection gives a focused doctor JSON-to-SQLite migration reproduction path because equivalent legacy jobs with different ids are imported beside existing SQLite jobs.

## Solution Assessment

No, the current implementation is not the best solution because it only avoids matching id/jobId imports and leaves equivalent duplicate jobs untouched. A blanket add upsert is also not automatically safe without maintainer-approved cron identity semantics.
