## Summary

Keep open: current main still has no generic cron duplicate audit or upsert path, and the only duplicate-pruning implementation I found is scoped to memory-core's managed dreaming cron jobs. The report is still plausible, but the exact update-triggered producer remains unproven, so this needs maintainer judgment rather than conservative cleanup closure.

## What This Changes

The issue asks OpenClaw update/restart flows to avoid silently duplicating equivalent cron jobs by upserting existing jobs or warning operators about duplicates.

## Best Possible Solution

Add a conservative duplicate audit or warning for equivalent enabled cron jobs during doctor/update/startup, and use owner-specific upsert only for managed cron producers with an explicit identity contract.

## Reproduction Assessment

No high-confidence end-to-end reproduction of the update-triggered duplication is available. Source inspection does show repeated equivalent cron.add calls create distinct enabled jobs, but the update component that re-created them is still not proven.

## Solution Assessment

Unclear: a blanket cron.add upsert would solve part of the symptom but is probably not the best first fix because it changes the creation contract. A duplicate audit/warning plus owner-specific managed-job reconciliation is the safer maintainable direction after the producer is confirmed.
