## Summary

Keep open: current main has a scoped memory-core managed-cron reconciliation path, but the broader cron service still appends new jobs without a duplicate audit or upsert contract, so the reported update-triggered duplicate cron problem is not fully resolved.

## What This Changes

The issue asks OpenClaw update/cron handling to avoid silently creating duplicate enabled cron jobs with matching name, schedule, and payload, or at least warn operators with duplicate IDs.

## Best Possible Solution

Add conservative duplicate detection and operator warnings for equivalent enabled cron jobs during doctor/update/startup, then add owner-specific upsert only for managed cron producers with a clear identity contract.

## Reproduction Assessment

Unclear: source inspection shows repeated cron.add calls can create duplicate persisted jobs, but I did not establish the update-triggered path that re-creates existing user jobs on current main.

## Solution Assessment

Unclear: generic upsert would address the symptom but changes the cron.add contract; a duplicate warning/audit plus owner-specific managed-job upsert is the safer maintainable direction.
