## Summary

Keep open: current main supports the generic per-CLI-backend environment workaround described in the issue, but it does not implement the requested first-class per-agent identity binding or doctor registry validation. This is a new config and trust-model feature, not a cleanup close candidate.

## What This Changes

Requests a first-class per-agent identity binding that sets `GIT_COMMITTER_*` for CLI backend spawns and adds doctor registry validation.

## Best Possible Solution

Decide the core-versus-plugin boundary first; if accepted, add a typed config/schema/help/docs contract, derive spawn env from the selected agent identity, and add explicit doctor validation while preserving `agents.defaults.cliBackends.<id>.env` for generic overrides.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a reported regression. Source inspection gives high confidence that current main only supports generic backend env injection and display-only agent identity fields.

## Solution Assessment

Unclear until maintainers decide the product boundary. The proposed direction is plausible, but it is broader than a narrow fix because it adds top-level config, runtime policy, schema/help/docs, and remote doctor validation semantics.
