## Summary

Keep open: current main still has the auto-reply heartbeat stale-display path, and the later agent-command heartbeat fix did not cover this persistence boundary. The duplicate WhatsApp report was correctly closed back to this canonical issue, while the direct fix PR remains closed unmerged.

## What This Changes

Heartbeat turns with a per-turn model override can persist their runtime model/provider into the shared session entry, making UI/history/status surfaces display the heartbeat model as the active session model.

## Best Possible Solution

Add heartbeat-aware runtime-model preservation to the auto-reply session usage persistence boundary, covering both persistence branches and both runner call sites while leaving normal turn persistence and usage accounting intact.

## Reproduction Assessment

Yes, by source inspection. A heartbeat with a different `heartbeat.model` reaches auto-reply with that transient provider/model, `persistRunSessionUsage` forwards it to `persistSessionUsageUpdate`, and `resolveSessionModelRef` then displays the persisted runtime fields.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or `preserveRuntimeModel` through `persistRunSessionUsage` and `persistSessionUsageUpdate`, preserving `model`/`modelProvider` only for heartbeat turns instead of masking polluted display data later.
