## Summary

Keep open: current main has a partial heartbeat-preservation fix in the agent-command session-store path, but the later auto-reply usage persistence path still has no heartbeat preservation signal and can overwrite the shared session model/provider fields that Gateway display resolution reads. The direct fix PR is closed unmerged, so this remains a valid narrow bug.

## What This Changes

Heartbeat turns should not persist their per-turn model/provider override as the shared session's current model displayed by chat history and Control UI.

## Best Possible Solution

Extend the existing heartbeat runtime-model preservation semantics to persistRunSessionUsage/persistSessionUsageUpdate and pass it from both heartbeat call sites, while keeping normal turns and usage/cost accounting behavior unchanged.

## Reproduction Assessment

Yes, source-reproducible. A heartbeat run with a different heartbeat model can reach persistRunSessionUsage with the heartbeat modelUsed/providerUsed, and persistSessionUsageUpdate then writes those fields into the session entry that resolveSessionModelRef and session rows display.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing preserveRuntimeModel/isHeartbeat concept at the auto-reply session usage persistence boundary; the cost-accounting and CLI binding questions should stay separate unless maintainers intentionally expand scope.
