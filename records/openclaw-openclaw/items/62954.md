## Summary

Keep open: current main still has the auto-reply stale-display path. The merged heartbeat preservation work covers the agent-command session-store path, but `persistSessionUsageUpdate` still has no heartbeat/preserve discriminator and both auto-reply runners still pass the heartbeat run model into persistence; the direct fix PR is closed unmerged.

## What This Changes

The issue requests that heartbeat turns using `agents.defaults.heartbeat.model` stop overwriting a shared session's persisted `model` and `modelProvider` display fields.

## Best Possible Solution

Add a heartbeat-aware preservation mode at the auto-reply session usage persistence boundary, covering both persistence branches and both runner call sites while preserving normal turns and existing usage accounting semantics.

## Reproduction Assessment

Yes, by source inspection. A heartbeat with a different `heartbeat.model` reaches auto-reply with that transient provider/model, `persistRunSessionUsage` forwards those values into `persistSessionUsageUpdate`, and `resolveSessionModelRef` then displays the persisted runtime fields.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or `preserveRuntimeModel` through auto-reply session usage persistence and preserve only `model`/`modelProvider` for heartbeat turns, rather than masking polluted display data later.
