## Summary

Keep open: current main still has the reported heartbeat stale-display path because heartbeat runs can persist their just-used runtime model/provider into the shared session entry that display resolution later trusts. The direct fix PR #62987 is closed unmerged, and the newer docs/changelog claim for #75452 does not match the inspected persistence code.

## What This Changes

The issue asks OpenClaw to stop heartbeat turns with a per-run heartbeat model from overwriting the session's persisted display model/provider.

## Best Possible Solution

Add heartbeat-aware preservation at the session usage persistence boundary so heartbeat turns keep the existing shared-session runtime identity while normal turns continue updating usage and runtime metadata.

## Reproduction Assessment

Yes, by source inspection: persist a heartbeat run with a different provider/model into an existing session entry, then resolve that entry through resolveSessionModelRef; current main will surface the heartbeat runtime fields. I did not run a mutating test because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The narrowest maintainable repair is to pass an isHeartbeat or preserve-runtime-model flag into persistSessionUsageUpdate and preserve the persisted runtime identity for heartbeat turns only.
