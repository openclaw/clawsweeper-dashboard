## Summary

Keep open: current main still has the auto-reply heartbeat stale-display path because post-run usage persistence can write the heartbeat runtime model/provider into the shared session entry, and the direct fix PR #62987 is closed unmerged.

## What This Changes

The issue asks for heartbeat turns using `agents.defaults.heartbeat.model` to avoid overwriting the session entry's persisted `model`/`modelProvider` display fields.

## Best Possible Solution

Add heartbeat-aware preservation at the auto-reply session usage persistence boundary so heartbeat turns keep the existing shared-session runtime identity while normal turns continue updating model, usage, context, and cost metadata.

## Reproduction Assessment

Yes. By source inspection, run an auto-reply heartbeat with a different heartbeat model/provider, let `persistRunSessionUsage` persist `modelUsed`/`providerUsed`, then resolve the session through `resolveSessionModelRef`; the heartbeat runtime pair is the displayed model.

## Solution Assessment

Yes. The narrow maintainable fix is to pass `isHeartbeat` or a `preserveRuntimeModel` flag through `persistRunSessionUsage`/`persistSessionUsageUpdate` and preserve `entry.model`/`entry.modelProvider` only for heartbeat turns; cost accounting and CLI binding questions should stay separate unless maintainers expand scope.
