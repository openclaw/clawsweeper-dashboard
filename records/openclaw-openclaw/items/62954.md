## Summary

Keep open: current main still has the reported stale-display path in the auto-reply persistence layer. The newer heartbeat preservation fix covers `updateSessionStoreAfterAgentRun`, but `persistSessionUsageUpdate` still has no heartbeat-preservation signal and both auto-reply runners still pass the heartbeat run's model/provider into that later persistence step.

## What This Changes

Heartbeat turns should not persist their per-turn `model`/`modelProvider` as the shared session's current display model.

## Best Possible Solution

Add heartbeat-aware runtime-model preservation at `persistSessionUsageUpdate`/`persistRunSessionUsage`, cover both persistence branches, and keep the cost and CLI-binding questions separate unless maintainers explicitly expand scope.

## Reproduction Assessment

Yes, by source inspection: persist a heartbeat run with a different `modelUsed`/`providerUsed` into an existing session entry, then resolve the row through `resolveSessionModelRef`/`buildGatewaySessionRow`; current main will surface the heartbeat model. I did not run a mutating test because this review was constrained to read-only inspection.

## Solution Assessment

Yes. The narrowest maintainable repair is to pass an `isHeartbeat` or `preserveRuntimeModel` signal into `persistSessionUsageUpdate` and preserve the stored runtime model fields only for heartbeat turns.
