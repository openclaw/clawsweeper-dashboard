## Summary

Keep open: current main still has the heartbeat stale-display path because heartbeat turns can pass their runtime model/provider into session usage persistence, which still writes those values into the shared session entry consumed by session display surfaces.

## What This Changes

Heartbeat turns using `agents.defaults.heartbeat.model` should not overwrite the session's persisted `model` and `modelProvider` display fields.

## Best Possible Solution

Add a heartbeat-aware preservation mode at the session usage persistence boundary so heartbeat turns keep the existing persisted runtime model identity while normal turns continue updating model, usage, context, cost, and prompt metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run a heartbeat with a different heartbeat model, let `persistRunSessionUsage` pass that model/provider into `persistSessionUsageUpdate`, then resolve the session row through `resolveSessionModelRef` or session listing.

## Solution Assessment

Yes. The narrow maintainable fix is to pass `isHeartbeat` or a `preserveRuntimeModel` flag through the auto-reply persistence path and preserve only `entry.model`/`entry.modelProvider` for heartbeat turns; cost accounting and CLI binding behavior should stay separate unless maintainers expand scope.
