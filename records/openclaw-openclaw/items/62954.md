## Summary

Keep open: current main still has the source-level heartbeat stale-display path, and this issue is the canonical open tracker for the duplicate reports. The adjacent agent-command path has a preserve-runtime-model mode, but the auto-reply persistence boundary still lacks the same heartbeat discriminator.

## What This Changes

The issue reports that heartbeat turns persist their per-turn model and provider into shared session state, causing UI/status/history surfaces to display the heartbeat model as the current session model.

## Best Possible Solution

Add heartbeat-aware runtime-model preservation at the auto-reply session usage persistence boundary, covering both persistence branches and both runner call sites, while leaving normal turns and usage accounting behavior intact.

## Reproduction Assessment

Yes, by source inspection. A heartbeat with a different heartbeat.model reaches auto-reply with that transient provider/model, persistRunSessionUsage forwards it to persistSessionUsageUpdate, and resolveSessionModelRef then displays the persisted runtime fields.

## Solution Assessment

Yes. The narrow maintainable fix is to thread isHeartbeat or preserveRuntimeModel through persistRunSessionUsage and persistSessionUsageUpdate, preserving model/modelProvider only for heartbeat turns instead of masking polluted data later in status/history display.
