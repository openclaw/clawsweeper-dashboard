## Summary

Keep open: current main still has the reported auto-reply heartbeat stale-display path. The direct fix PR at https://github.com/openclaw/openclaw/pull/62987 is closed unmerged, and the newer duplicate report at https://github.com/openclaw/openclaw/issues/74217 was closed back to this canonical issue.

## What This Changes

The issue reports that heartbeat turns using a heartbeat-specific model persist that transient model into the session entry fields used by UI/status display.

## Best Possible Solution

Add heartbeat-aware runtime-model preservation at the auto-reply session usage persistence boundary and keep display resolution unchanged.

## Reproduction Assessment

Yes, by source inspection: a heartbeat with `heartbeat.model` reaches auto-reply with a transient provider/model, `persistRunSessionUsage` forwards those values without heartbeat state, and `resolveSessionModelRef` displays the persisted runtime fields. I did not run a live heartbeat session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or `preserveRuntimeModel` through `persistRunSessionUsage` and `persistSessionUsageUpdate`, preserving runtime model identity only for heartbeat turns instead of masking polluted display data later.
