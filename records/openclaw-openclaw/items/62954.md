## Summary

Keep open: current main still has the auto-reply heartbeat stale-display path, because heartbeat turns can persist their runtime provider/model into the shared session entry that UI/history/status resolvers prefer.

## What This Changes

The issue asks OpenClaw to keep heartbeat per-turn model overrides from overwriting a session's persisted `model` and `modelProvider` display fields.

## Best Possible Solution

Add heartbeat-aware preservation at the auto-reply session usage persistence boundary so heartbeat turns preserve the existing shared-session runtime model/provider/context identity while normal turns keep updating runtime and usage metadata.

## Reproduction Assessment

Yes, by source inspection. Run an auto-reply heartbeat with `agents.defaults.heartbeat.model` set to a different provider/model, let `persistRunSessionUsage` persist `modelUsed`/`providerUsed`, then resolve the session through `resolveSessionModelRef`; current main will display the heartbeat runtime pair.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or `preserveRuntimeModel` through `persistRunSessionUsage` and `persistSessionUsageUpdate`, preserving runtime model fields only for heartbeat turns instead of masking polluted display data later.
