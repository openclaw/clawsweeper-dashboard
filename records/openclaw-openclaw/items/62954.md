## Summary

Keep open: current main still has the auto-reply heartbeat stale-display path, the direct fix PR is closed unmerged, and the newer duplicate report was closed back to this canonical issue.

## What This Changes

Heartbeat turns should not persist their per-turn heartbeat model/provider into session runtime model fields used by UI, history, and status displays.

## Best Possible Solution

Add heartbeat-aware runtime-model preservation at the auto-reply session usage persistence boundary while leaving display resolution and normal non-heartbeat persistence unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: heartbeat runs can resolve a distinct `heartbeat.model`, auto-reply persists that runtime provider/model without heartbeat state, and display resolvers prefer the persisted runtime fields.

## Solution Assessment

Yes. The narrow maintainable fix is to thread `isHeartbeat` or `preserveRuntimeModel` through `persistRunSessionUsage` and `persistSessionUsageUpdate`, preserving model/provider only for heartbeat turns instead of masking polluted display data later.
