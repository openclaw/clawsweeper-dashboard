## What This Changes

The PR catches poisoned plugin tool descriptor getters, returns accepted tools through a scoped snapshot proxy, reuses snapshots for descriptor caching, and adds a poisoned-descriptor regression test.

## Best Possible Solution

Land the runtime-boundary guard with release-note or PR-body context that invalid plugin tool descriptors are skipped with diagnostics, once maintainers accept that compatibility tradeoff.

## Reproduction Assessment

Yes, source-reproducible: current main directly reads runtime plugin tool descriptor getters during wrapping, resolution, and cache capture, so a throwing getter can abort sibling tool discovery. I did not execute the regression locally because this was a read-only review.

## Solution Assessment

Yes: guarding descriptor reads at the plugin runtime boundary and reusing validated snapshots is narrower than adding downstream try/catch handling at every caller. The compatibility impact is limited to invalid or unreadable plugin descriptors and still needs maintainer acceptance.
