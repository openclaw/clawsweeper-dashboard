## Summary

Keep open for maintainer review: current main still rejects approval-backed symlink cwd paths by design, and this PR proposes a new security-sensitive config option with correctness, supply-chain, and real-behavior-proof blockers.

## What This Changes

The PR adds a `tools.exec.allowSymlinkPath` config flag through config/schema and node-host approval hardening, adds a symlinked-cwd unit test, and changes fs-safe/pnpm workspace lock state.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that displays both requested and canonical cwd, preserves identity revalidation, updates schema/docs/tests, and omits dependency churn.

## Reproduction Assessment

Yes for current behavior: source and regression tests on current main show approval-backed `system.run` rejects direct symlink cwd paths and mutable symlink path components. This is a feature/config opt-in request rather than a broken existing contract.

## Solution Assessment

No, not as submitted. The direction may be viable after maintainer security/product approval, but the branch mixes unrelated dependency policy changes, does not bind the symlink context into the approval decision, and has a type-unsafe `agentId` call in the prepare path.
