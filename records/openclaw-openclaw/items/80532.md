## What This Changes

The PR adds a `tools.exec.allowSymlinkPath` config option through config/schema and node-host approval hardening, adds one symlink-cwd test, and also changes fs-safe/pnpm workspace state.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that displays requested and canonical cwd, preserves canonical identity revalidation, updates docs/schema/tests, and excludes unrelated dependency changes.

Yes for the current behavior: current-main source and tests show approval-backed `system.run` rejects direct symlink cwd paths and mutable symlink path components. This is a new opt-in feature/config request rather than a broken existing contract.

## Solution Assessment

No, not as submitted. The direction may be viable after maintainer security review, but this branch mixes unrelated workspace dependency policy changes and does not bind the requested-versus-canonical cwd context into the approval plan.
