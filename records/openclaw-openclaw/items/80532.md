## Summary

Keep open: this PR is a plausible implementation path for https://github.com/openclaw/openclaw/issues/77096, but it relaxes security-sensitive exec approval hardening, lacks real behavior proof, and mixes in unrelated dependency/workspace policy changes.

## What This Changes

The branch adds `tools.exec.allowSymlinkPath` through config/schema and node-host approval hardening, adds a symlink-cwd unit test, and changes fs-safe plus pnpm workspace lock state.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that shows requested and canonical cwd in approval UX, pins canonical identity, keeps fail-closed revalidation, updates docs/schema/tests, and excludes unrelated dependency policy changes.

## Reproduction Assessment

Yes for the current behavior: source and tests on current main show approval-backed `system.run` denies direct symlink cwd paths and mutable symlink path components. This is a feature/config opt-in rather than a broken existing contract.

## Solution Assessment

No, not as submitted. The core direction is plausible only after maintainer threat-model approval, but this branch mixes in unrelated supply-chain policy changes and does not bind requested-versus-canonical cwd into the approval plan/UX.
