Keep open. The PR is an active external feature/config proposal, but it is not merge-ready because it relaxes security-sensitive exec approval symlink hardening, includes unrelated workspace dependency policy churn, and still lacks real behavior proof.

## What This Changes

The PR adds a `tools.exec.allowSymlinkPath` option and threads it through config and node-host exec approval hardening so approval-backed exec can run from symlinked cwd paths when opted in.

## Best Possible Solution

Land only if maintainers accept a default-off opt-in that preserves canonical binding, exposes requested and canonical cwd clearly in the approval contract and docs, adds focused coverage, and removes unrelated dependency policy changes.

## Reproduction Assessment

Yes for the current behavior: current-main source and tests show approval-backed `system.run` denies direct symlink cwd paths and symlink parent components. This is a feature/config opt-in request, not a broken existing contract.

## Solution Assessment

No, not as submitted. The branch relaxes a security hardening guard without carrying requested-versus-canonical cwd through the approval contract and mixes in unrelated supply-chain policy changes.
