## Summary

Keep open: this PR targets a real open exec-approval feature request, but it is not ready as submitted because it relaxes security-sensitive symlink cwd hardening, includes unrelated dependency/install policy changes, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `tools.exec.allowSymlinkPath` config option through config/schema and node-host approval hardening, adds a symlinked-cwd unit test, and also changes fs-safe/pnpm workspace lock state.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that shows requested and canonical cwd in approval UX, keeps canonical identity revalidation, updates docs/schema/tests, and excludes unrelated dependency policy changes.

## Reproduction Assessment

Yes for the current behavior: origin/main source and tests show approval-backed `system.run` denies direct symlink cwd paths and mutable symlink path components. This PR is a feature/config opt-in rather than a regression fix.

## Solution Assessment

No, not as submitted. The core direction is plausible only after maintainer threat-model approval, but this branch mixes in unrelated supply-chain policy changes and does not update the approval UX/docs around the requested-versus-canonical cwd contract.
