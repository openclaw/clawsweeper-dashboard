## Summary

Keep open. The PR targets a real current-main gap: Telegram plugin commands pass only `sessionKey`, while the Codex plugin requires `ctx.sessionFile` for commands such as `/codex bind`. The proposed implementation should not merge as-is because it hand-rolls session-file path construction instead of using the shared session-file resolver, and it is missing the required changelog entry for a user-facing Telegram fix.

## What This Changes

The PR adds Telegram plugin-command session metadata resolution before `executePluginCommand` and expands the Telegram session metadata test to cover `/codex` in forum topics.

## Best Possible Solution

Land a narrow Telegram plugin-command fix that resolves and persists the active topic session file through the shared session-file helper, passes `sessionId` and an absolute normalized `sessionFile` into plugin command handlers, covers Telegram forum-topic `/codex bind` plus stale/relative session metadata cases, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Current main can be traced statically: Telegram plugin command execution omits `sessionFile`, and the Codex `bind` handler fails when `ctx.sessionFile` is missing; a focused runtime reproduction would be an authorized Telegram forum topic `/codex bind --cwd <dir>` invocation.

## Solution Assessment

No. Passing `sessionId` and `sessionFile` is the right direction, but the PR should reuse or expose the existing `resolveAndPersistSessionFile`/`resolveSessionFilePath` flow instead of duplicating path and persistence semantics inside the Telegram plugin.
