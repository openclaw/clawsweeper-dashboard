## Summary

Keep open. The PR addresses a real Telegram plugin-command gap, but the latest head still hand-rolls transcript path resolution instead of using the shared session-file resolver and still lacks the required user-facing changelog entry.

## What This Changes

The PR resolves and persists Telegram plugin-command session metadata, passes sessionId/sessionFile into plugin command handlers for topic invocations, and adds a regression test for topic-scoped `/codex` command routing.

## Best Possible Solution

Land a narrow Telegram plugin-command fix that resolves and persists the active topic transcript through the shared session-file resolver, passes normalized sessionId/sessionFile to plugin handlers, preserves extension boundary rules through an SDK-facing helper/export if needed, covers stale/relative topic metadata cases, and records the Telegram fix in CHANGELOG.md.

## Reproduction Assessment

Yes. Current main can be reproduced statically: Telegram plugin commands omit sessionFile, while `/codex bind` requires ctx.sessionFile; a live reproduction would be an authorized Telegram forum topic `/codex bind --cwd <dir>` invocation.

## Solution Assessment

No. Passing sessionId and sessionFile is the right direction, but this patch should reuse or expose the shared resolver/persist path instead of duplicating path construction and persistence semantics inside the Telegram plugin.
