## Summary

Keep open. The latest force-pushed PR head still addresses a real Telegram plugin-command gap, but it should not merge as-is because it hand-rolls session-file path resolution instead of using the shared resolver semantics and it lacks the required user-facing changelog entry.

## What This Changes

The PR adds Telegram native-command session metadata resolution before plugin command execution, passes sessionId/sessionFile into plugin command handlers, and extends the Telegram session metadata test for topic-scoped `/codex` command routing.

## Best Possible Solution

Land a narrow Telegram plugin-command fix that resolves and persists the active topic transcript through shared session-file resolver semantics, exposes any needed helper through the plugin SDK facade instead of importing core internals from the plugin, covers stale/relative topic metadata cases, and records the Telegram user-facing fix in CHANGELOG.md.

## Reproduction Assessment

Yes. Current main can be reproduced statically: Telegram plugin commands do not pass sessionFile, while `/codex bind` requires ctx.sessionFile; a live reproduction would be an authorized Telegram forum topic `/codex bind --cwd <dir>` invocation.

## Solution Assessment

No. Passing sessionId/sessionFile is the right direction, but this PR should reuse or expose the shared resolver/persist path instead of duplicating transcript path construction and persistence semantics inside the Telegram plugin.
