## Summary

Keep open: this maintainer-labeled PR is not cleanup-closeable, and the patch has a docs correctness blocker because the new migration flow tells operators to stop the Gateway before running a live `channels status --probe` check.

## What This Changes

Adds a BlueBubbles removal announcement/migration docs and introduces `openclaw channels status --channel <name>` filtering through CLI, Gateway protocol, tests, changelog, and the Swift protocol model.

## Best Possible Solution

Land a reviewed announcement plus `--channel` filtering only after the migration workflow matches the actual probe contract: either keep a single iMessage-enabled Gateway running for the probe or implement a true local/offline probe path before documenting it.

## Reproduction Assessment

Yes for the review blocker: source and docs show that `channels status --probe` only performs live probes through a reachable Gateway, while the PR documents running it after `openclaw gateway stop`. No live iMessage account is needed to prove that mismatch.

## Solution Assessment

No as written. The announcement direction and `--channel` filter are useful, but the stopped-Gateway probe flow is not the narrowest correct solution unless the command gains a real local probe mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep the Gateway running for the probe:** `docs/channels/imessage-from-bluebubbles.md:192-197`
  - body: The new cutover flow stops the Gateway and then tells users to expect `imessage.privateApi.available` from `channels status --probe --channel imessage`. However, `channels status --probe` only runs live probes through a reachable Gateway and falls back to config-only summaries when the Gateway is unreachable, so this command cannot prove the private API bridge in the documented state. Update the workflow to run a single iMessage-enabled Gateway for the probe, or add a real local probe path before documenting it.
  - confidence: 0.91
- **[P3] Do not ignore unknown channel filters in fallback:** `src/commands/channels/status.ts:210`
  - body: When `--channel` is misspelled, `normalizeChannelId()` returns `null`; the fallback filters then treat that as no requested channel and return all configured channels. The Gateway handler now returns `INVALID_REQUEST` for an unknown channel, but the CLI catch path can relabel that as `Gateway not reachable` and show unfiltered config-only output. Validate the channel before falling back, or preserve the invalid-channel error instead of silently dropping the filter.
  - confidence: 0.82
