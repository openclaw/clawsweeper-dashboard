## Summary

Keep open: the PR addresses a real source-level bug, but the current branch still has a blocking precedence issue where a broad model-map wildcard can win before an exact configured provider-model runtime policy. It also still lacks after-fix real behavior proof beyond targeted tests.

## What This Changes

The PR adds provider-wildcard matching to model runtime policy resolution and tests the resolver plus OpenAI harness selection for `agents.defaults.models["provider/*"].agentRuntime`.

## Best Possible Solution

Keep the wildcard fallback, but make exact model policy across agent/default model maps and provider model entries win before wildcard fallback, then add a mixed provider-model regression and real runtime-selection proof.

## Reproduction Assessment

Yes, at source level: current main normalizes `vllm/qwen-local` to `qwen-local`, then `modelKeyMatches` only accepts exact key/model equality, so `agents.defaults.models["vllm/*"].agentRuntime` is skipped. I did not run tests because this read-only review forbids commands that can create artifacts.

## Solution Assessment

No, not yet; wildcard runtime fallback is the right direction, but the implementation should preserve exact provider-model runtime policy before returning a broad wildcard policy. A focused provider-model precedence test would make the intended contract explicit.
