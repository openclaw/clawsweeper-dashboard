## Summary

Keep open: current main and v2026.6.11 still only partially support readable session labels, and the remaining work is split across open PRs for account-name fallback and UI formatting rather than implemented as one shared cross-surface display-name contract.

## What This Changes

The issue requests readable display names instead of raw channel peer IDs in Control UI session lists, CLI/status session tables, and the sessions_list tool output.

## Best Possible Solution

Define one shared session display-label precedence contract for Gateway row/search output and all operator display surfaces, preserving raw keys for copy/debug and reusing existing account or label metadata before adding new config.

## Reproduction Assessment

Yes, at source level: current main still has raw-key-first CLI/status rows and Gateway/UI display fallbacks that do not resolve direct peer IDs to actual contact names. I did not run a live Feishu, Discord, or Telegram setup in this read-only review.

## Solution Assessment

No, adding a broad new sessionLabelFormat config is not yet proven to be the best fix. The safer direction is a canonical shared display-label contract reused by Gateway, CLI/status, ACP, and Control UI, with channel/plugin-owned name resolution feeding that metadata.
