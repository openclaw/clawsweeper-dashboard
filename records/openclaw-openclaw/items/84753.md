## Summary

Keep open: current main has partial display-name plumbing, but the requested operator-facing behavior is still incomplete in CLI/status listings and still needs a cross-channel display-label policy before implementation.

## What This Changes

The issue requests human-readable session labels in Control UI, CLI status/session listings, and ACP sessions output instead of raw channel peer IDs.

## Best Possible Solution

Define one generic persisted session display-label contract that all list surfaces consume, keep raw keys available for disambiguation/copy targets, and leave channel-specific name resolution to the owning channel or plugin.

## Reproduction Assessment

Yes. Source inspection shows `openclaw sessions` and `openclaw status` still build rows around raw `key` values while omitting `label` and `displayName` from the primary table data.

## Solution Assessment

Unclear. The proposed `sessionLabelFormat` config may not be the best shape; the safer direction is a canonical display-label metadata contract reused by Gateway, CLI/status, ACP, and UI with raw-key fallback.
