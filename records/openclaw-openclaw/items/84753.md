## Summary

Keep open: current main has partial label/display-name plumbing, but the requested cross-channel operator-facing behavior is not consistently implemented across CLI status, local sessions output, and channel-derived display-label policy.

## What This Changes

Request to show human-readable channel display names instead of opaque peer IDs in Control UI, CLI status/sessions, and sessions_list, possibly through a configurable sessionLabelFormat.

## Best Possible Solution

Define a generic sanitized session display-label contract that Gateway, Control UI, CLI status, local sessions output, and sessions_list all consume, with raw peer IDs preserved as fallback and channel-owned name resolution feeding the metadata.

## Reproduction Assessment

Yes at source level: current main still has CLI status and local sessions paths that project/render raw keys while Gateway/UI can consume label/displayName when present. I did not run a live channel session.

## Solution Assessment

Unclear: sessionLabelFormat may be useful, but the safer first design is a shared persisted display-label contract with sanitization, resolver precedence, and raw-ID fallback decided by maintainers.
