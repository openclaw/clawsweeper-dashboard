## Summary

Keep open: current main still has only partial label/displayName plumbing, while the requested cross-surface session display-name behavior remains incomplete and split across related open issues and PRs.

## What This Changes

The issue requests readable channel peer names in Control UI, CLI/status, and ACP session listings instead of raw Feishu/Discord/Telegram/WhatsApp IDs.

## Best Possible Solution

Define one shared session display-label precedence contract for Gateway rows/search and all operator display surfaces, preserving raw keys for copy/debug and reusing existing account/label metadata before adding new config.

## Reproduction Assessment

Yes, at source level: current main still has raw-key-first CLI/status rows and Gateway/UI display fallbacks that do not resolve direct peer IDs to actual contact names. I did not run a live Feishu/Discord/Telegram setup in this read-only review.

## Solution Assessment

No, the issue should not be solved by adding a broad new label-format config before maintainers choose the cross-channel precedence and privacy policy. The safer path is a canonical shared display-label contract reused by Gateway, CLI/status, ACP, and Control UI, with channel/plugin-owned name resolution feeding that metadata.
