## Summary

Keep open: current main has useful session label/display-name primitives, but the requested operator-facing behavior is still incomplete across local CLI/status surfaces and still needs a cross-channel policy for how display names are resolved, persisted, sanitized, and exposed.

## What This Changes

The issue requests human-readable display names, rather than raw channel peer IDs, in session listings across the Control UI, CLI status output, and ACP session listing surfaces.

## Best Possible Solution

Define a generic persisted session display-label contract that all listing surfaces consume consistently, with raw peer IDs preserved as resolvable fallback and per-channel name resolution kept in the owning channel/plugin.

## Reproduction Assessment

Yes, at source level: local sessions/status paths still project and render raw keys while only some Gateway/UI paths consume label/displayName/origin metadata. I did not run a live Feishu, Discord, Telegram, or WhatsApp session, so this is not runtime-reproduced.

## Solution Assessment

Unclear: the requested sessionLabelFormat option may be useful, but the maintainable first step is a shared persisted display-label contract with sanitized rendering and raw-key fallback. Channel plugins should own how names are resolved rather than having core perform broad live lookups during listing.
