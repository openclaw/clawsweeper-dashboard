## Summary

Keep open: current main and the latest release have partial display-name plumbing, but the requested operator-facing behavior is still incomplete across CLI/status session listings and still needs a cross-channel display-label policy.

## What This Changes

Request to show human-readable channel user display names instead of opaque peer IDs in Control UI, `openclaw status`, and `sessions_list` output.

## Best Possible Solution

Define one generic persisted session display-label contract that all list surfaces consume, keep raw keys available as fallback and copy target, and leave channel-specific name resolution to the owning channel or plugin.

## Reproduction Assessment

Yes, source-level: current main can produce/display labels through the gateway and UI when metadata exists, but `openclaw sessions` and `openclaw status` build rows without label/displayName and render raw keys. No live channel tenant was exercised.

## Solution Assessment

Unclear whether the proposed `sessionLabelFormat` option is the best shape. A safer maintainable direction is a canonical display-label metadata contract reused by gateway, CLI/status, ACP, and UI with raw-key fallback.
