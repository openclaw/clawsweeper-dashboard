## What This Changes

The issue requests stripping leading `<session-recap>` internal context blocks before user messages are rendered in the Control UI and TUI.

## Best Possible Solution

Land a narrow shared-stripper fix that removes only leading standalone `<session-recap>` and `<session_recap>` blocks while preserving mid-message literal recap text across Control UI and TUI.

## Reproduction Assessment

Yes. Source inspection shows a leading `<session-recap>` block does not match the current fast-path regex, so both shared UI and TUI stripping helpers return it unchanged; I did not run a live UI/TUI reproduction in this read-only review.

## Solution Assessment

Yes. The best fix belongs in the shared inbound metadata stripper because both Control UI and TUI already rely on that boundary; it should be leading-only so user-authored mid-message recap text stays visible.
