## Summary

Keep this issue open. Current main still has no `session-recap` handling in the shared inbound metadata sanitizer, and the reported Control UI and TUI paths still render user text through that sanitizer. The issue is also paired with open PR #41013 using closing syntax, so the right cleanup is to review/land that PR or an equivalent replacement before closing this issue.

## What This Changes

The issue requests stripping leading `<session-recap>` / `<session_recap>` internal context blocks from user-visible Control UI and TUI message rendering.

## Best Possible Solution

Implement the fix in the shared inbound metadata sanitizer rather than separately in each UI: strip only complete leading `<session-recap>` / `<session_recap>` blocks in both exported strip functions, preserve mid-message and malformed literals, and cover the shared helper plus Control UI/TUI consumers as needed.

## Reproduction Assessment

Yes. A leading `<session-recap>...</session-recap>` text lacks any `SENTINEL_FAST_RE` match, so current `stripInboundMetadata()` and `stripLeadingInboundMetadata()` return it unchanged before parsing; the Control UI and TUI still call those functions for visible user messages.

## Solution Assessment

Yes. The narrow maintainable solution is to add session-recap recognition to `src/auto-reply/reply/strip-inbound-meta.ts` and keep UI surfaces on the shared sanitizer; open PR #41013 is already the linked implementation candidate.
