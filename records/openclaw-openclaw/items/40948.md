## Summary

Keep open: current main and the latest release still lack `<session-recap>` handling in the shared display sanitizer, so the reported Control UI/TUI leak remains source-reproducible; an open linked fork PR already claims the fix path but has not landed upstream.

## What This Changes

The issue asks OpenClaw to strip leading `<session-recap>...</session-recap>` agent context blocks from user-visible Control UI and TUI message rendering.

## Best Possible Solution

Land the linked fork PR or an equivalent narrow shared-sanitizer fix that strips complete leading recap blocks while preserving active-memory blocks, normal inbound metadata, malformed literal text, and mid-message literal recap tags.

## Reproduction Assessment

Yes, at source level: a leading `<session-recap>...</session-recap>` block contains none of the current fast-path sentinel strings, so both stripper entry points return it unchanged and the Control UI/TUI callers render it.

## Solution Assessment

Yes, the shared inbound metadata stripper is the right boundary; the fix should add narrow leading recap-block stripping with regression tests for leading, empty, chained, malformed, and mid-message cases.
