## Summary

Keep open: the latest PR head fixes the watchdog ordering from the earlier review, but still leaves two blocking behavior bugs in the timeout strategy. The embedded TUI fallback still resolves to the 48-hour default on normal configs, and the OAuth timeout races a non-cancelable upstream login promise that can leave the local callback server alive.

## What This Changes

The PR adds TUI watchdog abort wiring, embedded TUI timeout handling, an OpenAI Codex OAuth timeout wrapper with logging, and a changelog entry for the onboarding hatch hang.

## Best Possible Solution

Land a narrower fix that makes onboarding hatch recover promptly without changing normal local TUI timeout contracts, and make OAuth timeout cancellation close or settle the upstream callback flow instead of only racing it.

## Reproduction Assessment

Yes for the PR defects: fake-timer/unit coverage can show the default embedded timeout remains 48 hours with no config override and that a timed-out OAuth race leaves the upstream callback flow unsettled. The linked regression has concrete user steps, but I did not live-run the Ubuntu 26/OpenAI Cloud hatch path in this read-only pass.

## Solution Assessment

No. The watchdog abort direction is useful, but the latest timeout implementation does not deliver the claimed 5-minute default and the OAuth wrapper is not cancellation-safe against the upstream dependency contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Cancel the OAuth flow when the timeout fires:** `src/plugins/provider-openai-codex-oauth.ts:34`
  - body: `loginOpenAICodex` does not accept an AbortSignal and closes its localhost callback server only in its own `finally` after the browser/manual race settles. Racing it with `withTimeout` can reject to the caller while the upstream server is still waiting, so the CLI can remain alive or leave port 1455 occupied after the timeout. Make the timeout settle the upstream flow, for example through the manual-input path, or use an abortable dependency API before returning the timeout error.
  - confidence: 0.92
- **[P2] Apply a real five-minute hatch timeout:** `src/tui/embedded-backend.ts:185`
  - body: When `opts.timeoutMs` is omitted, `configuredTimeout` is the normal agent default, which is 48 hours when `agents.defaults.timeoutSeconds` is unset. `Math.max(MIN_TUI_TIMEOUT_MS, configuredTimeout)` therefore keeps the default install path at 48 hours, so the onboarding hatch path still lacks the promised 5-minute fallback. Scope an explicit 5-minute timeout to the hatch launch or distinguish an unset config default from a user-configured longer timeout.
  - confidence: 0.96
