## Summary

Keep open. The PR targets a real onboarding freeze, but the patch is not correct as written: the watchdog abort path no-ops, the 5-minute fallback changes all local embedded TUI timeout semantics, and the OAuth timeout wrapper can leave a ref'd timer alive after fast completion.

## What This Changes

The PR adds a 5-minute embedded TUI timeout, wires the streaming watchdog to call `abortActive`, wraps OpenAI Codex OAuth login in a 5-minute timeout with extra logging, and adds a changelog entry.

## Best Possible Solution

Revise the PR so the watchdog aborts the captured run before UI cleanup, timeout defaulting preserves the existing config contract or is scoped to onboarding, and OAuth timeout cleanup is covered by focused fake-timer tests.

## Reproduction Assessment

Yes for the PR-introduced defects: unit tests can show `abortActive()` sees no active run after the watchdog clears state, and fake timers can show a successful OAuth login leaves the 5-minute timeout pending. The original hatch hang has concrete #76132 steps, but was not live-reproduced in this read-only pass.

## Solution Assessment

No. The proposed direction is useful, but the current implementation is broader than the onboarding bug and misses the core abort behavior; a narrower run-id abort plus config-aware timeout handling is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Abort the run before clearing the active id:** `src/tui/tui-event-handlers.ts:137-139`
  - body: The new watchdog branch calls `abortActive()` after this callback has already set `state.activeChatRunId = null`. `abortActive()` returns early when there is no active id, so the PR's main stuck-run recovery never reaches `client.abortChat` and the backend run can keep hanging. Preserve the captured `runId` and abort before clearing state, or add an abort-by-run-id path.
  - confidence: 0.95
- **[P2] Preserve configured TUI run timeouts:** `src/tui/embedded-backend.ts:185`
  - body: Defaulting every embedded TUI send to 5 minutes changes normal local TUI behavior whenever the CLI did not pass `--timeout-ms`. The documented contract is that omitted timeouts fall back to `agents.defaults.timeoutSeconds`, so long local chats or configured longer agent budgets will now be cut off unexpectedly. Scope the shorter budget to onboarding hatch or resolve the existing config default instead.
  - confidence: 0.9
- **[P2] Clear the OAuth timeout when login settles:** `src/plugins/provider-openai-codex-oauth.ts:24-31`
  - body: The new `withTimeout` races against a `setTimeout`, but it never clears or unrefs that timer when `loginOpenAICodex` succeeds or fails before 5 minutes. Successful auth can therefore keep the CLI process alive until the timeout fires, replacing the indefinite hang with a fixed 5-minute linger. Store the timer handle and clear it in a `finally` path.
  - confidence: 0.88
