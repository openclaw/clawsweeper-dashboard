## Summary

Keep this issue open. Current main still emits the TUI pairing-required hint without the concrete `openclaw devices approve <requestId>` command, and the linked implementation PR is open and unmerged, so closure should wait for that PR to land or be closed.

## What This Changes

The issue asks for the pairing-required TUI error message to include the concrete `openclaw devices approve <request-id>` command after `openclaw devices list`.

## Best Possible Solution

Land a focused TUI copy and test update that names both `openclaw devices list` and `openclaw devices approve <requestId>`, preferably substituting the sanitized request ID when the gateway close reason includes one.

## Reproduction Assessment

Yes. The helper `resolveGatewayDisconnectState("gateway closed (1008): pairing required")` deterministically returns a hint that lacks `openclaw devices approve <requestId>`, and the current unit test only asserts the list command.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/tui/tui.ts` plus `src/tui/tui.test.ts`, with the existing pairing close-reason parser reused if exact request-id guidance is added.
