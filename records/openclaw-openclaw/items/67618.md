## Summary

Keep open. Current main still emits the reported TUI pairing-required hint without the concrete `openclaw devices approve <requestId>` command, and the provided GitHub context shows open PR #73208 already targets the same issue with closure linkage, so the issue should wait for that PR to be reviewed and merged or closed.

## What This Changes

Request to improve the TUI pairing-required disconnect message so it tells users to run `openclaw devices approve <requestId>` after listing pending requests.

## Best Possible Solution

Review and land or close PR #73208. The desired end state is for the TUI disconnect hint to match the existing devices CLI/docs flow by naming both `openclaw devices list` and `openclaw devices approve <requestId>`, preferably using the existing pairing-required parser when the gateway close reason includes a sanitized request ID.

## Reproduction Assessment

Yes. Inspecting current main is enough: `resolveGatewayDisconnectState("gateway closed (1008): pairing required")` returns a `pairingHint` containing `openclaw devices list` but no `openclaw devices approve <requestId>`, and the existing unit test only asserts the list command.

## Solution Assessment

Yes. The narrowest maintainable fix is a TUI copy/test change, with an optional refinement to reuse `readConnectPairingRequiredMessage` for exact request IDs; because an open PR already targets that boundary, normal PR review is the right next action.
