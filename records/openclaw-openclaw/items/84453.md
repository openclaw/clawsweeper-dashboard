## What This Changes

The PR replaces the package dist import scanner's reverse context check with a forward state machine and adds Vitest coverage for static, dynamic, and re-export relative import forms.

## Best Possible Solution

Keep the forward-scanner approach, but narrow keyword handling to syntactic import/re-export forms and add real package-dist validation proof before merge.

## Reproduction Assessment

## Solution Assessment

No, not yet. The direction is maintainable, but the current state machine needs tighter syntax guards for contextual `import` and `from` tokens before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip import.meta before entering import state:** `scripts/lib/package-dist-imports.mjs:86-87`
  - body: Here any standalone `import` token enters import state. Valid ESM such as `import.meta.resolve("./not-a-dist-file.js")` then treats the following relative string as a dist import edge even though no import exists, causing package checks to fail on valid output. Please skip `import.` / `import.meta` or require a real static/dynamic import shape before capturing.
  - confidence: 0.9
- **[P2] Do not treat exported `from` bindings as re-exports:** `scripts/lib/package-dist-imports.mjs:98-99`
  - body: While `export-pending`, this promotes any standalone `from` token to a pending specifier. `export const from = "./not-imported.js";` is valid ESM, but this scanner will report the string as a missing dist import; reset on export declarations or only accept `from` after export-list/star forms.
  - confidence: 0.88
