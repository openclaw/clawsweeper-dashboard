## Summary

Keep open. Current main still emits `multiple_gateways` solely from the count of reachable probe targets, so the reported SSH tunnel plus configured direct URL path can still warn about multiple gateways even when both targets identify the same logical gateway.

## What This Changes

The issue asks `openclaw gateway probe` to avoid or clarify the `multiple_gateways` warning when an SSH tunnel target and `gateway.remote.url` reach the same gateway instance.

## Best Possible Solution

Classify reachable targets by a conservative non-null gateway self identity, suppress or downgrade the warning when all reachable targets share that identity, preserve `multiple_gateways` for distinct or unknown identities, and update focused tests and user-visible docs/changelog copy.

## Reproduction Assessment

Yes, at source level. Construct two reachable `GatewayStatusProbedTarget` entries with the same non-null `self`; current `buildGatewayStatusWarnings` still emits `multiple_gateways` because it only checks `reachable.length > 1`.

## Solution Assessment

Yes. Identity-aware warning classification is the narrowest maintainable fix because it keeps the useful direct-path probe while avoiding a false multi-gateway warning; skipping `configRemote` when a tunnel is active would hide useful reachability data.
