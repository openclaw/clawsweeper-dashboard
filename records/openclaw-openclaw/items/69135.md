## Summary

Keep open. Current main still emits `multiple_gateways` from the count of reachable probe targets, so an SSH tunnel target plus a configured remote URL to the same logical gateway can still produce the reported false positive.

## What This Changes

The issue asks `openclaw gateway probe` to avoid or clarify the `multiple_gateways` warning when an SSH tunnel and `gateway.remote.url` are two transports to the same gateway.

## Best Possible Solution

Classify reachable targets by a conservative non-null gateway self identity, suppress or downgrade the warning when all reachable targets share that identity, preserve `multiple_gateways` for distinct or unknown/mixed identities, and update focused tests, docs, and changelog copy.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: pass two reachable `GatewayStatusProbedTarget` entries with the same non-null `self` into `buildGatewayStatusWarnings`, and current main still emits `multiple_gateways` from `reachable.length > 1`. The reporter's SSH tunnel plus direct `gateway.remote.url` setup is the live equivalent.

## Solution Assessment

Yes. Identity-aware warning classification is the narrowest maintainable fix because it keeps the useful direct-path probe while avoiding a false multi-gateway warning; skipping the configured remote target would hide useful reachability information.
