## Summary

Keep open. Current main still emits `multiple_gateways` from the number of reachable probe transports alone, while the SSH tunnel path can also keep the configured remote URL as a second target and the collected gateway self identity is not used to distinguish one gateway from two.

## What This Changes

The issue asks `openclaw gateway probe` to stop reporting multiple gateways when an SSH tunnel target and `gateway.remote.url` are two transports to the same gateway.

## Best Possible Solution

Use the existing probe presence/self data to group reachable targets by confirmed gateway identity before emitting `multiple_gateways`, and add focused regression coverage for same-identity tunnel plus configured-remote probes.

## Reproduction Assessment

Yes, source-level. Current main builds both the SSH tunnel target and configured remote target when their URLs differ, then warns on `reachable.length > 1` without checking the already-collected gateway self identity.

## Solution Assessment

Yes, a narrow fix in the gateway-status warning path is the best direction. Dedupe only confirmed same-identity reachable probes, while preserving the existing warning for distinct identities or unknown identities where OpenClaw cannot prove they are the same gateway.
