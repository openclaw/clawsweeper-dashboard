## Summary

Keep open. Current main still matches the reported bug: gateway probe warns from the count of reachable probe targets, while the SSH tunnel and configured remote URL path can keep two successful transports to the same logical gateway. The prior ClawSweeper comment reached the same conclusion, and no current-main change appears to have added identity-aware warning classification.

## What This Changes

Change `openclaw gateway probe` so SSH tunnel and configured remote URL successes to the same gateway identity are not reported as multiple distinct gateways.

## Best Possible Solution

Fix the gateway-status warning path by grouping reachable targets with a conservative non-null gateway self identity, suppressing or downgrading `multiple_gateways` when all reachable targets share that identity, and preserving the warning for distinct or unknown/mixed identities. Add focused tests for same-identity and distinct-identity probes, then update warning docs and changelog copy.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: pass two reachable `GatewayStatusProbedTarget` entries with the same non-null `self` into `buildGatewayStatusWarnings`, and current main still emits `multiple_gateways` from `reachable.length > 1`. The reporter's SSH tunnel plus direct `gateway.remote.url` setup is the live equivalent.

## Solution Assessment

Yes. Identity-aware warning classification is the narrowest maintainable solution because it keeps the useful direct-path probe while avoiding a false multi-gateway warning; skipping the configured remote target would hide useful reachability information.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/69135. `openclaw gateway probe` currently emits `multiple_gateways` whenever more than one target is reachable, which creates a false positive when an SSH tunnel target and `gateway.remote.url` both reach the same logical gateway. Keep probing both transports; do not remove the configured remote target just because a tunnel exists. Update `buildGatewayStatusWarnings` to classify reachable targets by a conservative non-null gateway self identity from `GatewayStatusProbedTarget.self` and suppress or downgrade `multiple_gateways` when all reachable targets prove the same identity. Preserve the existing warning for distinct identities and for unknown/mixed identity cases where suppressing could hide real multiple gateways. Add focused coverage for same-identity suppression and distinct-identity warning, preferably in `src/commands/gateway-status/output.test.ts`, and add an integration-style check if useful in `src/commands/gateway-status.test.ts` for SSH tunnel plus configured remote. Update user-facing warning docs and `CHANGELOG.md`; credit the reporter as `Thanks @sk8man121`. Do not change gateway startup, SSH tunnel security policy, auth resolution, or target probing order beyond what is needed for warning classification.
