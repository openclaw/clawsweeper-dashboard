## Summary

Keep open: the PR addresses a real Mattermost preview-overwrite bug, but the current patch is not merge-ready because block splitting can still race pending draft lifecycle work, per-account streaming overrides can discard root defaults, and the public docs/changelog are not aligned.

## What This Changes

The PR adds Mattermost `streaming.mode` and `streaming.toolPreview` config, forwards sanitized tool args through tool-start events, wires block-mode preview boundary splitting, and adds focused Mattermost tests.

## Best Possible Solution

Land a Mattermost-local block preview mode that freezes only materialized preview posts, preserves nested streaming defaults across account overrides, and keeps schema, generated metadata, docs, and changelog aligned.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: a Mattermost turn with reasoning, tool, and partial updates repeatedly calls `draftStream.update(...)` on the same post; the PR regression can be reproduced by queuing a draft update and signaling a boundary before the prior preview flushes or finishes creating.

## Solution Assessment

No, not as currently implemented. The `block` mode direction is narrow and maintainable, but the split operation must be lifecycle-aware and the new public config surface needs merge, docs, generated metadata, and changelog follow-through.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wait for the preview post before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` treats content as split-ready as soon as it was queued with `draftStream.update()`. Because the draft loop is throttled and can have pending text or an in-flight create/update, `forceNewMessage()` can clear unsent text or race the pending create so the next phase still edits the old post. Make the boundary split flush or wait for the prior draft lifecycle before resetting the post id.
  - confidence: 0.9
- **[P2] Deep-merge streaming account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118`
  - body: Mattermost account merging still deep-merges only `commands`. With this PR, an account override like `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, silently falling back to partial mode. Add `streaming` to the nested merge keys and cover mixed root/account overrides.
  - confidence: 0.87
- **[P2] Align Mattermost streaming docs with the schema:** `extensions/mattermost/src/config-schema-core.ts:127`
  - body: The new schema accepts object-form `streaming.mode` values `partial` and `block`, plus `streaming.toolPreview`, but the checked-in Mattermost and streaming docs still advertise scalar `streaming: "partial"` and `off`/`progress` modes. Users following those docs will get rejected or stripped config unless the docs are updated or compatibility is added.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `extensions/mattermost/src/types.ts:39-49`
  - body: This PR changes user-facing Mattermost behavior and adds public config options, but it does not update `CHANGELOG.md`. Add a single-line entry under the active version with appropriate non-bot contributor attribution.
  - confidence: 0.8
