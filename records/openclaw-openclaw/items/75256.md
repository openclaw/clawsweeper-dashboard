## What This Changes

The PR adds Mattermost draft-preview boundary splitting for block mode, Mattermost-specific streaming/toolPreview config handling, raw tool-argument preview rendering, docs/changelog updates, generated metadata, and focused tests.

## Best Possible Solution

Rebase onto current main and implement Mattermost block preview splitting through the shared `streaming.mode` and `streaming.preview` contract, with accurate docs and real Mattermost proof.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: Mattermost updates one draft preview post for thinking, partial reply, and tool phases, and the local reset API is not called at turn boundaries.

## Solution Assessment

No. Boundary splitting is the right Mattermost-local direction, but this head should preserve the current shared streaming config, remove false preview-off guidance, and either redact raw tool args or document them as a sensitive opt-in before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the shared streaming config shape:** `extensions/mattermost/src/config-schema-core.ts:89`
  - body: Current main already accepts the shared Mattermost `streaming` contract, including scalar values like `"off"`/`"progress"` and nested `preview`/`progress`/`block` objects.   - confidence: 0.86
- **[P2] Remove the unusable preview-off workaround:** `docs/channels/mattermost.md:287`
  - body: The docs now tell users to disable Mattermost preview streaming by using only block streaming, but this PR head still creates a draft preview stream for every inbound turn and passes `disableBlockStreaming: true` into dispatch. That setting disables block replies here, not the draft preview path, so either wire a real preview-off behavior or remove the workaround claim.
  - confidence: 0.9
- **[P2] Do not call raw tool args sanitized:** `docs/channels/mattermost.md:291`
  - body: `toolPreview: "args"` is documented as a sanitized one-line summary, but `summarizeMattermostToolArgs()` renders raw command/path/input/text/JSON values and only trims or truncates them. Operators could enable it expecting redaction, so either redact in code or document it as raw sensitive opt-in.
  - confidence: 0.88
