## Summary

Keep open. The Mattermost boundary-splitting fix is relevant and source-reproducible, but this PR is incorrect as-is because it conflicts with current main's shared streaming contract, lacks real Mattermost proof, and documents raw tool arguments as sanitized.

## What This Changes

The PR adds Mattermost draft-preview block-mode boundary splitting, Mattermost-specific streaming/tool-preview config handling, raw tool-argument preview rendering, docs/changelog/generated metadata, and focused tests.

## Best Possible Solution

Rebase onto current main and implement Mattermost boundary splitting through the existing shared streaming.mode contract while preserving off/progress/preview settings, fixing the tool-args wording or redaction, and adding real Mattermost proof.

## Reproduction Assessment

Yes. Current-main source shows the Mattermost monitor keeps updating one draft stream across partial, reasoning, and tool phases, while the draft stream only starts a new post when forceNewMessage is called and the monitor does not call it.

## Solution Assessment

No. Boundary splitting is the right Mattermost-local direction, but this PR should preserve the current shared streaming config shape, remove the false preview-off guidance, and either redact raw tool args or document them as a sensitive opt-in before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the shared streaming config shape:** `extensions/mattermost/src/config-schema-core.ts:89-104`
  - body: Current main already accepts the shared Mattermost `streaming` contract, including scalar `"off"`/`"progress"` values and nested `preview`/`progress`/`block` objects. This branch narrows the schema to `{ mode?: "partial" | "block", toolPreview?: ... }`, so a resolved merge would reject supported config and drop shared controls.
  - confidence: 0.86
- **[P2] Remove the unusable preview-off workaround:** `docs/channels/mattermost.md:287`
  - body: The docs tell users to disable Mattermost preview streaming by using only block streaming or omitting the streaming object, but this PR head still creates a draft preview stream for every inbound turn and passes `disableBlockStreaming: true` into dispatch. That setting disables block reply delivery here, not the draft preview path.
  - confidence: 0.9
- **[P2] Do not call raw tool args sanitized:** `docs/channels/mattermost.md:291`
  - body: `toolPreview: "args"` is documented as a sanitized one-line summary, but `summarizeMattermostToolArgs()` renders raw command/path/input/text/JSON values and only trims or truncates them. Operators could enable it expecting redaction, so either redact in code or document it as a sensitive opt-in.
  - confidence: 0.88
