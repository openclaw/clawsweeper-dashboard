## Summary

Keep open: current main still has the Mattermost single-post overwrite behavior and the PR targets a real bug, but the proposed patch should not merge yet because boundary splits can race pending preview creation, nested streaming config overrides are not merged, and public docs/changelog are out of sync.

## What This Changes

The PR adds Mattermost `streaming.mode` and `streaming.toolPreview` config, propagates sanitized tool args into tool-start callbacks, wires draft-preview boundary splitting into Mattermost, updates generated metadata, and adds focused tests.

## Best Possible Solution

Land a Mattermost-local block preview mode that freezes only materialized preview posts at phase boundaries, preserves nested streaming defaults across per-account overrides, and keeps schema, generated metadata, public docs, tests, and changelog in sync.

## Reproduction Assessment

Yes. Current main can be reproduced by tracing a tool-using Mattermost turn through the preview hooks: each phase calls `draftStream.update(...)` on the same post, and a delayed Mattermost create/edit test can reproduce the PR regression by signaling a boundary before the previous preview is materialized.

## Solution Assessment

No, not as currently implemented. The `block` mode is the right narrow direction, but the split operation must be lifecycle-aware and the public config surface needs merge, docs, and changelog fixes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the preview post before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` treats content as split once it has merely been enqueued. If the first create is still pending, `forceNewMessage()` can clear pending text or be overwritten when the in-flight create later stores `streamPostId`, so the next phase can still edit the old post. Wait for or explicitly account for the prior draft update/create before splitting.
  - confidence: 0.89
- **[P2] Deep-merge streaming account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118`
  - body: Mattermost account merging still deep-merges only `commands`. An account override like `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, silently falling back to partial mode. Add `streaming` to the nested merge keys and cover mixed root/account overrides.
  - confidence: 0.86
- **[P2] Align Mattermost streaming docs with the schema:** `extensions/mattermost/src/config-schema-core.ts:127`
  - body: The schema now accepts object-form `streaming.mode` values `partial` and `block`, but the public streaming docs still advertise scalar modes including `off` and `progress` for Mattermost. Users following those docs will get rejected or stripped config unless the docs are updated or compatibility is added.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `extensions/mattermost/src/types.ts:22`
  - body: This adds user-facing Mattermost streaming config and an opt-in tool-args preview, but `CHANGELOG.md` has no active entry for it. Add a single-line entry under the active version with appropriate non-bot contributor attribution.
  - confidence: 0.8
