## Summary

Keep open. Current main still has the reported Mattermost overwrite behavior, but this PR is not merge-ready because the proposed boundary split can discard or race queued draft content, and the new public config surface is not fully aligned across merging, docs, and changelog.

## What This Changes

This PR adds Mattermost `streaming.mode` and `streaming.toolPreview` config/schema/account resolution, propagates sanitized tool args into tool-start callbacks, wires draft-preview boundary splitting into the Mattermost monitor, and updates focused tests plus generated config metadata.

## Best Possible Solution

Land a Mattermost-local block preview mode that freezes only materialized preview posts at phase boundaries, preserves nested streaming defaults across account overrides, and keeps schema, generated metadata, public docs, changelog, and changed-gate validation in sync.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: a Mattermost turn with reasoning/tool/partial updates calls `draftStream.update(...)` repeatedly and never splits the preview post. The PR regression can be covered with a focused test that delays draft create/update, signals a boundary before the prior preview flushes, and verifies no queued or in-flight preview content is lost.

## Solution Assessment

No, not as currently implemented. The `block` mode direction is narrow and maintainable, but the split operation must wait for or otherwise account for pending and in-flight draft lifecycle state, and the new public config surface needs merge, docs, and changelog fixes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the preview post before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` treats content as split-ready as soon as it was queued with `draftStream.update()`. Because the draft loop is throttled and can have an in-flight create/update, `forceNewMessage()` can clear pending text before it is posted or race with a pending create so the next phase still edits the old post. Make the boundary split await or generation-guard the prior draft lifecycle before resetting the stream.
  - confidence: 0.89
- **[P2] Deep-merge streaming account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118`
  - body: Mattermost account merging still deep-merges only `commands`. With this PR, an account override such as `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, silently falling back to partial mode. Add `streaming` to the nested merge keys and cover mixed root/account overrides.
  - confidence: 0.86
- **[P2] Align Mattermost streaming docs with the schema:** `extensions/mattermost/src/config-schema-core.ts:127`
  - body: The new schema accepts object-form `streaming.mode` values `partial` and `block`, but the checked-in Mattermost and streaming docs still advertise scalar `streaming: "partial"` plus `off` and `progress` modes. Users following those docs will get rejected or stripped config unless the docs are updated or compatibility is added.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `extensions/mattermost/src/types.ts:39-49`
  - body: This PR changes user-facing Mattermost behavior and adds public config, but `CHANGELOG.md` has no Mattermost entry on the PR head. Add a single-line entry under the active section with non-bot contributor attribution.
  - confidence: 0.8
