## Summary

Keep open. The PR targets a real Mattermost preview overwrite problem reported in #75252 and overlaps the schema/docs confusion in #71569, but it is not merge-ready: the proposed split can run before the draft lifecycle has materialized the prior preview, nested account overrides can drop channel-wide streaming fields, and the public docs/changelog are not aligned.

## What This Changes

The PR adds Mattermost `streaming.mode` and `streaming.toolPreview` config/schema/account resolution, propagates sanitized tool args into tool-start callbacks, wires draft-preview boundary splitting into the Mattermost monitor, and updates focused tests plus generated config metadata.

## Best Possible Solution

Land a Mattermost-local block preview mode that freezes only materialized preview posts at phase boundaries, preserves nested streaming defaults across account overrides, and keeps schema, generated metadata, public docs, and changelog in sync.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: a Mattermost turn with reasoning/tool/partial updates calls `draftStream.update(...)` repeatedly and never splits the preview post. The PR regression also has a focused unit-test path by delaying draft create/update and signaling a boundary before the prior preview flushes.

## Solution Assessment

No, not as currently implemented. The `block` mode direction is narrow and maintainable, but the split operation must wait for or otherwise account for pending/in-flight draft lifecycle state, and the new public config surface needs merge, docs, and changelog fixes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the preview post before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` treats content as split-ready as soon as it was queued with `draftStream.update()`. Because the draft loop is throttled and can have an in-flight create/update, `forceNewMessage()` can clear pending text before it is posted or race with a pending create so the next phase still edits the old post. Make the boundary split await/flush the prior draft lifecycle, or cover this explicitly in the stream lifecycle.
  - confidence: 0.89
- **[P2] Deep-merge streaming account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118`
  - body: Mattermost account merging still deep-merges only `commands`. With this PR, an account override such as `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, silently falling back to partial mode. Add `streaming` to the nested merge keys and test mixed root/account overrides.
  - confidence: 0.86
- **[P2] Align Mattermost streaming docs with the schema:** `extensions/mattermost/src/config-schema-core.ts:127`
  - body: The new schema accepts object-form `streaming.mode` values `partial` and `block`, but the checked-in Mattermost/streaming docs still advertise scalar `streaming: "partial"` plus `off` and `progress` modes. Users following those docs will get rejected or stripped config unless the docs are updated or compatibility is added.
  - confidence: 0.83
