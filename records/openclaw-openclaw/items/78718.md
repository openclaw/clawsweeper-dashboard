## Summary

Keep this PR open, but do not merge it as-is: the proposed precedence change conflicts with current docs/tests, the patch has a concrete cadence bug, related heartbeat gates remain on the old policy, and the external real-behavior proof gate is still missing.

## What This Changes

The PR changes heartbeat agent selection so agents without per-agent heartbeat config inherit `agents.defaults.heartbeat` even when another agent has an explicit heartbeat block.

## Best Possible Solution

If maintainers accept standard override semantics, centralize effective heartbeat resolution and update scheduler, summary, system-prompt guidance, ACP relay, docs, changelog, and regression coverage together.

## Reproduction Assessment

Yes, at source level: current main's explicit-only gates in `src/infra/heartbeat-summary.ts` and `src/infra/heartbeat-runner.ts` exclude a default-inheriting agent once another agent has a heartbeat block. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction may be acceptable after maintainer agreement, but this patch is not the best implementation because cadence parsing, cross-surface policy alignment, docs/changelog, and real behavior proof are incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

  - body: The new explicit-agent branch returns `Boolean(agentEntry.heartbeat?.every)`, so `every: "0m"` is treated as enabled and `heartbeat: {}` becomes disabled instead of inheriting defaults. Resolve the effective heartbeat config and parse the interval before deciding enabled state.

  - confidence: 0.88
- **[P3] Document and changelog the accepted precedence:** `src/infra/heartbeat-runner.ts:285-290`
  - body: The patch changes user-facing heartbeat precedence while the public guide still says any per-agent heartbeat block restricts heartbeats to those agents, and repo policy expects a changelog entry for user-facing fixes. Update both if this semantic change is accepted.
  - confidence: 0.9
