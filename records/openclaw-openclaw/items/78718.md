## Summary

Keep open: the PR targets a real source-level mixed heartbeat-default gap, but the current patch is not mergeable because it mis-handles explicit cadence values, leaves other heartbeat policy gates on the old semantics, conflicts with current docs without docs/changelog updates, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes heartbeat agent selection so per-agent heartbeat config wins, agents without overrides fall back to `agents.defaults.heartbeat`, and tests cover mixed defaults, explicit opt-out, and empty heartbeat objects.

## Best Possible Solution

If maintainers accept standard override semantics, centralize heartbeat enablement around one effective merged heartbeat config and update scheduler, summary, system-prompt, ACP relay, docs, changelog, and regression coverage together.

## Reproduction Assessment

Yes, at source level: current main's explicit-only gates in `heartbeat-summary.ts` and `heartbeat-runner.ts` exclude a default-inheriting agent as soon as any other agent has a heartbeat block. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction may be acceptable after maintainer agreement, but the current patch is not the best fix because cadence parsing, cross-surface policy alignment, docs/changelog, and real behavior proof are incomplete.
