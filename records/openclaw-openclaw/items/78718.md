## Summary

Keep open: current main still uses and documents the explicit-only heartbeat behavior, while this PR changes the heartbeat precedence contract across runtime surfaces. The latest patch looks coherent and has after-fix proof, but the remaining decision is maintainer acceptance of the changed config semantics plus targeted validation, not conservative cleanup closure.

## What This Changes

The PR changes heartbeat enablement so per-agent heartbeat blocks merge over `agents.defaults.heartbeat`, with shared logic used by the scheduler, system prompt, ACP spawn, docs, tests, and changelog.

## Best Possible Solution

Adopt one centralized heartbeat precedence contract, align docs/tests/runtime surfaces with it, and close the linked duplicate reports only after the accepted implementation merges.

## Reproduction Assessment

Yes by source inspection: the mixed defaults-plus-explicit config follows explicit-only gates in `heartbeat-summary.ts`, `heartbeat-runner.ts`, `heartbeat-system-prompt.ts`, and `acp-spawn.ts`, and the existing test expects `main` to be disabled. I did not execute the runtime in this read-only review.

## Solution Assessment

Yes, if maintainers accept the new semantics. Centralizing enablement in `isHeartbeatEnabledForAgent` and reusing it across the scheduler, system prompt, and ACP spawn paths is the narrow maintainable shape; the open question is the product/config precedence contract.
