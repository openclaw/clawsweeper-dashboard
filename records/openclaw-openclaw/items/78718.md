## Summary

This PR should stay open but not merge as-is: current main still has the source-level heartbeat selection behavior described by the linked issue, while the patch leaves heartbeat enablement inconsistent across runtime surfaces and lacks after-fix real behavior proof.

## What This Changes

The PR changes heartbeat enablement and scheduler selection so per-agent heartbeat config wins and agents without overrides inherit `agents.defaults.heartbeat`, with focused tests for mixed default and explicit configs.

## Best Possible Solution

If maintainers accept the new inheritance semantics, centralize heartbeat enablement around the effective merged heartbeat config and update scheduler, summary, system-prompt, ACP relay, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes, at source level: current main's explicit-only heartbeat gate excludes the default agent in a mixed defaults-plus-explicit config before defaults are consulted. I did not run tests because this was a read-only review.

## Solution Assessment

No. The proposed direction may be reasonable after maintainer agreement, but this implementation uses a truthiness cadence check, leaves other heartbeat gates on the old policy, and does not update docs/changelog for the behavior change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse explicit heartbeat cadence before enabling it:** `src/infra/heartbeat-summary.ts:34`
  - body: `Boolean(agentEntry.heartbeat?.every)` makes `every: "0m"` enabled because non-empty strings are truthy, while `{}` or partial heartbeat objects can be disabled instead of falling through to defaults. Resolve the effective heartbeat config and parse the interval before deciding enabled state.
  - confidence: 0.96

  - body: This scheduler path starts treating inherited-default agents as enabled, but `heartbeat-system-prompt` and `acp-spawn` still use the old explicit-only policy. In the mixed config, runtime heartbeat behavior can disagree across scheduler, prompt, and ACP relay surfaces.
  - confidence: 0.88
- **[P3] Document the new heartbeat precedence:** `src/infra/heartbeat-runner.ts:285-290`
  - body: The patch changes user-facing heartbeat precedence, but the public guide still says that if any agent has a heartbeat block only those agents run heartbeats, and repo policy also expects a changelog entry for user-facing fixes. Update both with the accepted semantics.
  - confidence: 0.9
