## Summary

Keep open. The PR is aimed at a real heartbeat precedence bug, but the latest head still has blocking compatibility and diagnostic gaps before it is safe to merge.

## What This Changes

The PR changes heartbeat enablement so per-agent heartbeat config merges with agent defaults across scheduler, summary, system prompt, ACP spawn, docs, tests, and changelog.

## Best Possible Solution

Adopt one shared heartbeat precedence resolver that preserves existing explicit-only/no-default behavior unless maintainers approve a migration, then use that policy consistently in scheduler, summary/status, system prompt, ACP spawn, doctor diagnostics, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the mixed defaults-plus-explicit config follows explicit-only branches in `isHeartbeatEnabledForAgent` and `resolveHeartbeatAgents`, and existing tests still encode related explicit-only behavior.

## Solution Assessment

No as-is. Centralizing heartbeat enablement is the right direction, but the patch needs compatibility-preserving no-default semantics, doctor alignment, and a maintainer decision on `heartbeat: {}` before it is the narrow maintainable fix.
