## Summary

Keep open: the PR targets a real heartbeat precedence problem and includes sufficient after-fix proof for the reported mixed-defaults case, but the latest patch leaves the no-defaults/explicit-agent case inconsistent with an unchanged focused test and expands default-agent heartbeat behavior beyond the original report. GitHub also reports the branch as merge-conflicting, so the remaining path needs maintainer judgment on the config contract before merge.

## What This Changes

The PR changes heartbeat enablement so per-agent heartbeat config merges over `agents.defaults.heartbeat`, reuses `isHeartbeatEnabledForAgent` across scheduler, system-prompt, and ACP spawn paths, and updates docs, tests, and changelog.

## Best Possible Solution

Centralize heartbeat enablement, but explicitly decide and encode the no-defaults plus explicit-agent precedence in runtime, docs, and tests before merging.

## Reproduction Assessment

Yes by source inspection: current main follows explicit-only gates for the reported mixed defaults-plus-explicit config. The PR-side blocker is also source-reproducible because the PR head leaves a system-prompt test expecting `undefined` while the new helper returns enabled for that no-defaults shape.

## Solution Assessment

No as submitted. Centralizing the policy is the right direction, but the fallback logic must either preserve explicit-only behavior when no defaults exist or update the contract, docs, and tests intentionally.
