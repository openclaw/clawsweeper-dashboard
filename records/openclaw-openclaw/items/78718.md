## Summary

Keep open; the reported behavior is source-traceable, but this PR is not mergeable as-is because it changes documented heartbeat selection semantics, lacks real behavior proof, and introduces a blocking cadence/partial-config bug.

## What This Changes

The PR changes heartbeat agent selection to fall back from per-agent heartbeat config to `agents.defaults.heartbeat`, and updates focused heartbeat tests for mixed default and explicit agent configs.

## Best Possible Solution

If maintainers accept the new inheritance semantics, centralize heartbeat enablement around the effective merged heartbeat config, parse cadence instead of testing truthiness, and update runner, summary, system-prompt, ACP relay, docs, changelog, and focused tests together.

## Reproduction Assessment

Yes, at source level: current main's `hasExplicitHeartbeatAgents` gate excludes `main` in the mixed default-plus-explicit config before defaults are consulted. I did not run tests because this was a read-only review.

## Solution Assessment

No. The proposed direction may be reasonable if maintainers want to change the documented semantics, but this implementation uses a truthiness check for cadence and does not update the other heartbeat policy gates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse explicit heartbeat cadence before enabling it:** `src/infra/heartbeat-summary.ts:34`
  - body: `Boolean(agentEntry.heartbeat?.every)` makes `every: "0m"` enabled because non-empty strings are truthy, while `{}` or partial heartbeat objects become disabled even when defaults provide the cadence. That contradicts the PR's own expected cases and leaves summary/status behavior wrong; resolve the effective heartbeat config and parse the interval instead.
  - confidence: 0.96
- **[P2] Align the remaining heartbeat gates:** `src/infra/heartbeat-runner.ts:282-290`
  - body: This scheduler path would start treating inherited-default agents as enabled, but `heartbeat-system-prompt` and `acp-spawn` still use the old explicit-only gate. In the mixed config, an inherited-default default agent can run without matching heartbeat prompt/bootstrap handling, and ACP heartbeat status can still report disabled.
  - confidence: 0.87
