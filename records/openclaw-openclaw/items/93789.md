## Summary

Keep open: the cooldown-precheck direction is useful, but the embedded-runner change is too broad because it removes lane locking even when no outer model-fallback summary owns the terminal failure. The supplied proof is also unit-test/mock proof, not real fallback behavior proof.

## What This Changes

The PR conditionally avoids lane locking during model cooldown precheck and changes two embedded-runner suspension paths to persist provider failure state without passing a lane id.

## Best Possible Solution

Make lane-lock deferral explicit: only the outer model-fallback loop should suppress immediate lane locking, while direct or no-fallback embedded runs should preserve terminal lane suspension, with runtime proof for fallback-success and all-exhausted cases.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 pass a concrete lane id into suspendSession in the cooldown-precheck and embedded-runner rate-limit/billing paths. I did not run a live multi-provider repro in this read-only review.

## Solution Assessment

No. The cooldown-precheck change is the right shape, but the embedded-runner change should defer lane locking only when an outer model-fallback owner will handle terminal exhaustion; forcing laneId undefined in all embedded-runner paths is too broad.
