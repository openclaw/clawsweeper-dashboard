## Summary

Keep open: current main still has a source-visible path where an `ok` subagent wait becomes `completed successfully` and empty assistant output falls back to raw tool/toolResult text, which matches the reported tail-of-exec announce failure.

## What This Changes

The issue asks OpenClaw to flag subagent runs that complete with no synthesized text/yield and zero output tokens, and to stop using the last exec output as the announced result.

## Best Possible Solution

Add a narrow subagent completion guard that treats terminal no-assistant/no-yield zero-output runs as failed or incomplete, emits an explicit no-result payload, and preserves intentional fallback behavior only where it is still safe and documented.

## Reproduction Assessment

Yes for the current-main source path: empty assistant history plus an `ok` wait can still announce raw tool output as a successful result. The exact OpenAI model/cap loop was not live-run, so the provider trigger remains unverified.

## Solution Assessment

Yes, a targeted runtime/announce fix is the right direction; the optional max-tool-iterations guard can be a follow-up if needed, but the immediate bug is the success classification and result fallback.
