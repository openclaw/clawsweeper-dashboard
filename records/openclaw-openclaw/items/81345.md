## Summary

Keep open: the report matches current main source paths where a subagent terminal event can still be treated as ok while completion capture can fall through to the latest tool result when no assistant/yield text exists.

## What This Changes

The issue asks OpenClaw to flag zero-text or zero-token subagent runs as incomplete or failed and to stop using the last exec/tool output as the announced result.

## Best Possible Solution

Derive subagent terminal outcome from explicit yield/final assistant text, run liveness/replayInvalid, and usage/text absence, and make announce result capture assistant/yield-only instead of raw tool-result fallback.

## Reproduction Assessment

Yes, at source level: a transcript with only assistant tool calls plus toolResult text follows the current readSubagentOutput path to latestRawText, while registry completion still records ok. I did not run the live gpt-5.5-pro scenario in this read-only review.

## Solution Assessment

Yes: fixing the subagent lifecycle/result-capture contract is the narrow maintainable path. The optional max-tool-iterations guard should stay separate unless maintainers want a broader runtime policy change.
