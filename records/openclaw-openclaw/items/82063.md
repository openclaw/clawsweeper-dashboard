## Summary

Keep open: this maintainer-labeled gateway refactor is plausible, but the current branch still has a blocking dispatch regression where extra handlers that override core methods become duplicate registry entries instead of preserving current-main override precedence.

## What This Changes

This PR adds a gateway method descriptor registry, rewires request dispatch and WebSocket handling to use it, derives advertised method lists from descriptors, and updates plugin/channel registry typing plus focused tests.

## Best Possible Solution

Keep the descriptor-registry direction only if it preserves legacy dispatch precedence: extra handlers must be able to replace core handlers without duplicate-registration throws, with regression coverage around that contract.

## Reproduction Assessment

Yes for the review finding: current main has a clear source path where `extraHandlers` override core handlers, and the PR source path registers both descriptors then rejects duplicates. I did not execute the test suite because this review is read-only.

## Solution Assessment

No as currently implemented; the registry abstraction is a maintainable direction, but the construction path needs to preserve current dispatch precedence by replacing or filtering colliding descriptors before registration.
