## Summary

Keep open: current main still handles Synology Chat inbound webhooks as text-only, while the requested image, CSV, and PDF attachment path needs Synology contract confirmation plus file-ingestion security decisions before implementation.

## What This Changes

Request inbound attachment support for Synology Chat messages, including images, CSV files, and PDFs.

## Best Possible Solution

Add plugin-local inbound attachment parsing and safe staging for allowed images, CSVs, and PDFs, then pass media facts into the existing inbound media context with focused docs and tests.

## Reproduction Assessment

Yes, by source inspection: an attachment-only Synology webhook without text is rejected before delivery, and the downstream Synology context has no media fields. I did not run a live Synology NAS attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right because shared inbound media context already exists, but the requested minor extension is not sufficient without confirming Synology's attachment contract and file-ingestion safety policy.
