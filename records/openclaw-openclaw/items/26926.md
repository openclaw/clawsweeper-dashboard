## Summary

Keep this issue open: current main still supports only text-bearing inbound Synology webhooks and URL-based outbound files, while no merged or open canonical item adds inbound attachments. The implementation boundary is clear, but feasibility and security depend on obtaining a real or officially documented Synology attachment event and download contract.

## What This Changes

Support inbound images, CSV files, and PDFs in the bundled Synology Chat channel.

## Best Possible Solution

Capture a redacted real Synology attachment webhook event or obtain an authoritative API contract, then add bounded plugin-owned parsing and safe media staging through the existing inbound media context; if Synology emits no attachment metadata, document that platform limitation rather than inventing unsupported fields.

## Reproduction Assessment

Yes, at source level with high confidence: current main requires inbound text and has no attachment fields or media context. A live Synology attachment event is still needed to reproduce the external payload and determine whether implementation is feasible.

## Solution Assessment

No, the proposed “minor code extension” is not yet the best substantiated solution. The narrow maintainable path is plugin-owned support based on a verified Synology event/download contract, with explicit security and resource limits.
