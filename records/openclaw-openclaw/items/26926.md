## Summary

Keep open. Current main still treats Synology Chat inbound webhooks as text-only: attachment-only payloads are rejected before delivery, and the Synology inbound turn never passes media facts into the shared channel context.

## What This Changes

Add Synology Chat inbound support for images, CSV files, and PDFs so private-network agents can receive attachments.

## Best Possible Solution

Implement plugin-owned inbound attachment parsing and safe staging in `extensions/synology-chat`, then feed supported media into the existing channel turn media facts with focused docs and tests.

## Reproduction Assessment

Yes for the current missing behavior by source inspection: an attachment-only payload with token/user but no text returns null in `parsePayload`, and no later Synology path passes media facts to `buildContext`. This is a feature gap rather than a broken documented contract.

## Solution Assessment

Yes on direction: keep the work inside the Synology Chat plugin and use the existing channel media context fields. Confirm the Synology attachment contract and media safety limits before coding; add a core seam only if the confirmed contract exposes a reusable gap.
