## Summary

Keep open. Current main still rejects attachment-only Synology webhooks before delivery and does not pass media into the channel turn context; recent maintainer discussion also explicitly kept this open for follow-up.

## What This Changes

The issue requests inbound Synology Chat support for images, CSV files, and PDFs.

## Best Possible Solution

Implement plugin-owned inbound attachment parsing and safe staging in `extensions/synology-chat`, then feed supported files into the existing channel turn media facts with docs and regression tests.

## Reproduction Assessment

Yes for the feature gap by source inspection: an attachment-only webhook with token and user but no text is rejected in `parsePayload`, and no later Synology path carries media into `buildContext`. This is not a broken documented contract; it is a missing feature.

## Solution Assessment

Yes on direction, but not as an automated fix yet. The maintainable path is to keep Synology-specific parsing/download behavior inside the plugin and reuse the shared inbound media context after confirming Synology's payload and media-safety rules.
