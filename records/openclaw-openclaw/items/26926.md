## Summary

Keep open. Current main and the latest release still support Synology Chat file/media only on the outbound URL-delivery path; inbound webhook payloads, parsing, delivery, turn context, docs, and tests remain text-only, and no attachment-specific duplicate or merged fix was found.

## What This Changes

The issue requests inbound image, CSV, and PDF attachment support for Synology Chat.

## Best Possible Solution

Implement plugin-owned inbound attachment parsing and safe staging for images, CSVs, and PDFs, then pass media facts into the shared channel turn context with Synology docs and focused webhook/context tests.

## Reproduction Assessment

Yes, by source inspection: an attachment-only Synology webhook with token and user but no text is rejected by parsePayload, and the downstream delivery/context path has no media fields. This proves the requested capability is absent on current main, though it is a feature gap rather than a broken documented contract.

## Solution Assessment

Yes on direction, but not as an automatic cleanup close or narrow repair yet. The maintainable solution is to keep Synology-specific parsing/download/staging inside the plugin, reuse the shared inbound media context, and first confirm Synology's attachment contract plus file-safety rules.
