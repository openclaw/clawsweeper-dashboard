## Label Justifications

- `P2`: This is a concrete bundled-channel feature gap with daily user impact for Synology file workflows, but it is not an emergency regression or core runtime outage.
- `impact:message-loss`: Attachment-bearing Synology messages cannot currently reach the agent as media because the inbound webhook path accepts only text.

## Summary

The Synology Chat attachment gap is still present on current main: inbound webhooks require text, the Synology delivery/context path has no media fields, and existing media support is outbound URL delivery only. The implementation shape is clear, but accepting files from a NAS webhook needs maintainer decisions on Synology's attachment contract and file-ingestion safety before automation should write it.

## What This Changes

Request support for inbound Synology Chat image, CSV, and PDF attachments so private-network users can send files to agents through the bundled Synology Chat plugin.

## Best Possible Solution

Implement Synology-owned inbound attachment parsing and safe staging for accepted image, CSV, and PDF files, then pass media facts into the existing channel media context with docs and focused regression tests.

## Reproduction Assessment

Yes, by source inspection: an attachment-only Synology webhook without text is rejected before delivery, and the downstream Synology message/context path has no media fields. I did not run a live Synology attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right and can use the existing channel media context seam, but the proposed minor extension is not sufficient without confirming Synology's attachment contract and file-safety policy.
