## Label Justifications

- `P2`: This is a valid, channel-scoped capability gap with meaningful daily usability impact but no core-runtime outage.
- `impact:message-loss`: Images and documents sent through Synology Chat are omitted from the inbound agent message and therefore cannot be processed.

## Summary

Current main and v2026.6.5 still accept only text for inbound Synology Chat webhooks, while no related issue or PR owns attachment support; the request remains valid but needs maintainer product and security decisions before implementation.

## What This Changes

Add inbound Synology Chat support for images, CSV files, and PDFs so attachment content reaches agent context.

## Best Possible Solution

Confirm and document Synology's real attachment event and retrieval contract, then implement attachment-only and text-plus-attachment messages inside the Synology plugin using shared safe media storage and canonical inbound media context fields.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 require non-empty text and carry no inbound attachment metadata, so attachment-only messages cannot reach the agent. A live Synology payload example is still needed to prove the exact upstream fields.

## Solution Assessment

The plugin-local direction is the narrowest maintainable solution because shared media helpers already exist, but the exact implementation must follow a verified Synology contract and a maintainer-approved untrusted-file policy.
