## Summary

Keep this open. Current main still treats Synology Chat inbound webhooks as text-only: payload typing, parsing, delivery, context construction, docs, and tests do not cover inbound image, CSV, or PDF attachments, while the existing media support is outbound URL delivery only. The existing maintainer/Codex comment remains accurate on the current checkout.

## What This Changes

Add inbound image, CSV, and PDF attachment support to the bundled Synology Chat channel plugin.

## Best Possible Solution

Implement this inside the Synology Chat plugin: parse Synology attachment payloads, accept attachment-only messages when they contain supported files, safely stage images/CSVs/PDFs into the existing inbound media context fields, preserve current auth/rate-limit/204 ACK behavior, and update Synology docs and focused tests.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence code-level reproduction: an attachment-only webhook payload without text would fail parsePayload's token/user/text requirement and return the missing-required-fields path before any agent delivery.

## Solution Assessment

Yes. Plugin-owned inbound attachment handling is the narrowest maintainable path because shared channel turn media fields already exist; a new core seam should only be added if Synology's confirmed API contract exposes a reusable gap.
