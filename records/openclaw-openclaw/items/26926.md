## Label Justifications

- `P2`: This is a concrete bundled-channel feature gap with daily user impact for Synology file workflows, but it is not an emergency regression or core runtime outage.
- `impact:message-loss`: Attachment-bearing Synology messages cannot currently reach the agent as media because the inbound webhook path accepts only text.

## Summary

Keep open: current main and the latest release still accept only text on Synology Chat inbound webhooks; media support exists for outbound file URL delivery, and the generic inbound media context is available once the plugin parses and stages attachments.

## What This Changes

Add inbound Synology Chat support for images, CSV files, and PDFs so users can send attachments to agents from private NAS-hosted chat.

## Best Possible Solution

Add plugin-local inbound attachment parsing and safe staging for allowed image, CSV, and PDF files, feed media facts into the existing channel inbound media context, and document/test the behavior after confirming Synology's attachment contract and file-safety policy.

## Reproduction Assessment

Yes, by source inspection: an attachment-only Synology webhook without text is rejected before delivery, and the downstream Synology message/context path has no media fields. I did not run a live Synology NAS attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right because the shared inbound media context already exists, but the requested minor extension is not sufficient without confirming Synology's attachment contract and file-ingestion safety policy.
