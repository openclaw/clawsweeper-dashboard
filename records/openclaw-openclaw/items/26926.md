## Summary

Keep open: current main and v2026.6.10 still treat Synology Chat inbound webhooks as required-text messages, while media support is outbound URL delivery only. The remaining work is real, but inbound file ingestion needs Synology attachment-contract confirmation plus a deliberate safe-download/staging policy before a repair PR should be queued.

## What This Changes

Request to add Synology Chat inbound support for images, CSV files, and PDFs so private-network channel users can send attachments to agents.

## Best Possible Solution

Confirm Synology's attachment webhook/retrieval contract, then implement plugin-local bounded inbound media parsing, safe retrieval/staging, generic media-context population, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: an attachment-only Synology webhook cannot pass the required-text parser, and the delivery/context path has no media fields. A real NAS attachment webhook payload is still missing, so the external API shape is not fully reproduced.

## Solution Assessment

Unclear until Synology's attachment event and retrieval contract are confirmed. If that contract supports the feature, plugin-local bounded retrieval/staging plus generic media-context population is the narrowest maintainable direction.
