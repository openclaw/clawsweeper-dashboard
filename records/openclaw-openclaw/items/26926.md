## Summary

Keep open. Current main still treats Synology Chat inbound webhooks as text-only; outbound URL-based media support and prior JSON/alias compatibility work do not deliver inbound images, CSVs, or PDFs to the agent context, and the remaining file-ingestion behavior needs maintainer product/safety decisions.

## What This Changes

Request inbound Synology Chat support for images, CSV files, and PDFs instead of text-only agent control.

## Best Possible Solution

Implement Synology-owned inbound attachment parsing and safe staging for accepted image, CSV, and PDF files, then pass media facts into the existing channel media context with docs and focused regression tests.

## Reproduction Assessment

Yes, by source inspection: a valid Synology webhook without text is rejected before delivery, and the downstream Synology message/context path has no media fields. I did not run a live Synology attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right and can use the existing channel media context seam, but the issue's proposed minor extension is not sufficient without confirming Synology's attachment contract and file-safety policy.
