## Summary

Keep open. Current main still handles Synology Chat inbound webhooks as text-only: the parser requires text, delivery/context construction passes no media, docs only describe outbound file URLs, and the recent maintainer/Codex comments remain accurate for the current SHA.

## What This Changes

The issue requests inbound Synology Chat support for image, CSV, and PDF attachments.

## Best Possible Solution

Implement this in the Synology Chat plugin: parse confirmed attachment payloads, accept supported attachment-only messages, safely stage images/CSVs/PDFs into existing inbound media facts, preserve current auth/rate-limit/204 ACK behavior, and update docs plus focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an attachment-only webhook with token and user fields but no text is rejected by parsePayload, and there is no later media path into dispatchSynologyChatInboundTurn.

## Solution Assessment

Yes. Plugin-owned inbound attachment handling is the narrowest maintainable solution because shared channel turn media fields already exist; a new core seam should only be added if the confirmed Synology contract exposes a reusable gap.
