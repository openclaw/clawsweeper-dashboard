## Summary

Keep open. Current main still treats Synology Chat inbound webhooks as text-only, while the existing outbound media path and the merged JSON/alias webhook compatibility work do not provide inbound image, CSV, or PDF attachment handoff; implementing file ingestion also needs maintainer product and safety decisions.

## What This Changes

The issue requests Synology Chat inbound attachment support for images, CSV files, and PDFs.

## Best Possible Solution

Implement Synology-owned inbound attachment parsing and safe staging for accepted image, CSV, and PDF files, then pass media facts into the existing channel turn media path with docs and focused regression tests.

## Reproduction Assessment

Yes, by source inspection: a webhook with a valid token and user but no text is rejected before delivery, and the downstream Synology message/context path has no media field. I did not exercise a live Synology attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right, but the issue's proposed minor extension is not enough without confirming Synology's attachment contract and file-safety policy.
