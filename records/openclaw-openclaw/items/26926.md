## Summary

Keep open: current main still treats Synology Chat inbound webhooks as text-only, and attachment support needs confirmed Synology payload fields plus explicit file-ingestion safety decisions before implementation.

## What This Changes

The issue requests image, CSV, and PDF attachment support for Synology Chat inbound messages.

## Best Possible Solution

Add plugin-local inbound attachment parsing and safe staging for allowed images, CSVs, and PDFs, then pass media facts into the existing inbound media context with focused docs and tests.

## Reproduction Assessment

Yes, by source inspection: parsePayload rejects payloads without text, and the authorized Synology delivery/context path has no media fields. I did not run a live Synology NAS attachment webhook.

## Solution Assessment

Partly. Keeping the work plugin-local is the right direction because the shared inbound media context already exists, but the requested minor extension is not sufficient without confirming Synology's attachment contract and file-ingestion safety policy.
