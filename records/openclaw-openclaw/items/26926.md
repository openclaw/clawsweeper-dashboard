## Summary

Keep open: current main and the latest release still treat Synology Chat inbound webhooks as required-text messages, while the requested images, CSVs, and PDFs need an inbound attachment contract and safe download/staging policy before implementation.

## What This Changes

The issue requests inbound Synology Chat attachment support for images, CSV files, and PDFs instead of the current text-only webhook path.

## Best Possible Solution

Confirm Synology's attachment webhook and retrieval contract, then implement plugin-local bounded inbound media parsing/download/staging that feeds existing generic inbound media context with docs and focused tests.

## Reproduction Assessment

Yes, at source level: an attachment-only webhook cannot pass the required-text parser, and the Synology inbound delivery path has no media fields. A real NAS attachment webhook payload is still missing, so the external API shape is not fully reproduced.

## Solution Assessment

Unclear until Synology's attachment event and retrieval contract are confirmed. If that contract supports the feature, plugin-local bounded download/staging plus generic media-context population is the narrowest maintainable direction.
