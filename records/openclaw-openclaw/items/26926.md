## Summary

Keep open: current main still rejects Synology Chat inbound attachment-only webhooks before delivery, while docs and tests cover text inbound plus outbound URL media. The requested capability remains valid, but it needs maintainer/security decisions around Synology's attachment contract and safe file ingestion before implementation.

## What This Changes

The issue requests inbound image, CSV, and PDF attachment support for the Synology Chat channel/plugin.

## Best Possible Solution

Add plugin-local inbound attachment parsing and safe staging for allowed image, CSV, and PDF files, feed media facts into the existing channel inbound media context, and document/test the behavior after confirming Synology's attachment contract and file-safety policy.

## Reproduction Assessment

Yes, by source inspection: an attachment-only Synology webhook without text is rejected by parsePayload before delivery, and the downstream Synology event/context path has no media fields. I did not run a live Synology NAS attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right because the shared inbound media context already exists, but the requested minor extension is not sufficient without confirming Synology's attachment contract and file-ingestion safety policy.
