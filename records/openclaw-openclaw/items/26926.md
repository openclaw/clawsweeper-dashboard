## Summary

Keep open: current main and the latest release still handle Synology Chat inbound webhooks as text-only, while the requested image/CSV/PDF attachment path has no parser, context, docs, or test support. The open related Synology PR is adjacent HEAD/ACK webhook work and does not own attachment support.

## What This Changes

The issue requests inbound Synology Chat attachment support for images, CSV files, and PDFs instead of text-only webhook handling.

## Best Possible Solution

Confirm Synology's attachment event and retrieval API, then implement bounded inbound media staging inside the Synology Chat plugin and populate the existing core inbound media fields with focused tests and docs.

## Reproduction Assessment

Yes, at source level: an attachment-only webhook cannot pass the required-text parser, and the Synology delivery/context path has no inbound media fields. A real NAS attachment webhook payload was not provided, so the external payload shape remains unverified.

## Solution Assessment

Unclear until Synology's attachment event and retrieval contract are confirmed. If that contract supports the feature, plugin-local bounded download/staging plus existing core media-context population is the narrowest maintainable solution.
