## Label Justifications

- `P2`: This is a concrete bundled-channel feature gap with daily user impact for Synology file workflows, but it is not an emergency regression or core runtime outage.
- `impact:message-loss`: Attachment-bearing Synology messages cannot currently reach the agent as media because the inbound webhook path accepts only text.

## Summary

Keep open: current main still only accepts text on the Synology Chat inbound webhook path, while attachment handling would add network/file-ingestion behavior that needs maintainer product and safety decisions before automation writes it.

## What This Changes

Add inbound image, CSV, and PDF attachment support to the Synology Chat plugin.

## Best Possible Solution

Add plugin-local inbound attachment parsing and safe staging for allowed image, CSV, and PDF files, feed the resulting media facts into the existing channel context, and document/test the behavior.

## Reproduction Assessment

Yes, by source inspection: an attachment-only Synology webhook without text is rejected before delivery, and the downstream Synology message/context path has no media fields. I did not run a live Synology attachment webhook.

## Solution Assessment

Partly. The plugin-local direction is right and can use the existing channel media context seam, but the proposed minor extension is not sufficient without confirming Synology's attachment contract and file-safety policy.
