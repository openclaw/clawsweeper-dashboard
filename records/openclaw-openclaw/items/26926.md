## Summary

Keep open. Current main still has only outbound Synology Chat media-by-URL support; inbound webhook parsing requires text and the delivered turn context has no media facts, so the requested image/CSV/PDF inbound attachment support remains unimplemented.

## What This Changes

Request inbound image, CSV, and PDF attachment support for Synology Chat so private-network agents can receive more than text messages.

## Best Possible Solution

Implement plugin-owned inbound attachment parsing and safe staging in extensions/synology-chat, then pass supported files through the existing channel turn media context with docs and focused regression tests.

## Reproduction Assessment

Yes by source inspection for the feature gap: an attachment-only webhook with token and user but no text is rejected by parsePayload, and the Synology inbound turn never passes media into buildContext. This is a missing feature, not a broken documented contract.

## Solution Assessment

Yes on direction, but not as an automated fix yet. The maintainable path is to keep Synology-specific parsing/download behavior inside the plugin and reuse the shared inbound media context after confirming Synology's payload and media-safety rules.
