## Summary

Keep open: current main still lacks a provider-level or per-candidate retry-count config, and the live related implementation PRs remain open and need maintainer API-scope selection before this can close.

## What This Changes

Add a configurable retry budget for model provider API calls so fallback chains can avoid waiting through hardcoded provider SDK retries before trying the next candidate.

## Best Possible Solution

Choose one supported model-provider retry contract, then land a rebased implementation with schema, docs, focused tests, and real configured-provider proof while preserving current guarded fetch, timeout, Retry-After, and failover behavior.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers choose the public retry scope. Provider-level retry config is plausible, but it should be designed with the existing fallback loop, SDK Retry-After cap, guarded transport, and related per-candidate retry requests in mind.
