## Summary

Keep this open. Current main has the adjacent Retry-After wait cap from #68474 and provider request timeouts, but it still does not expose a typed per-provider or per-candidate model API retry-count budget, and the strict config schemas would reject the proposed fields.

## What This Changes

Add a configurable retry-count budget for model provider attempts so fallback chains can advance quickly when a provider returns rate-limit or overloaded responses.

## Best Possible Solution

Add a typed provider request retry budget under the model-provider request config, document the transport-specific behavior, and plumb it only into model API transports that actually honor SDK retry counts. Direct fetch transports should either remain single-shot or explicitly document that the retry budget is a no-op for them.

## Reproduction Assessment

Yes for the missing configuration path: static inspection of current main shows the proposed config keys are absent from strict schemas and are not forwarded into OpenAI/Azure SDK construction. I did not live-reproduce the multi-provider 429/529 delay.

## Solution Assessment

Yes, a typed provider-scoped request retry budget is the maintainable direction; it is better than relying on timeouts or the current global Retry-After wait cap because it directly controls SDK retry attempts before model fallback can advance.
