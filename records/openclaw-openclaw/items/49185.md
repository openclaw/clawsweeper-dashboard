## Summary

This should stay open as a product/config enhancement: current main has a shipped mitigation for long SDK `Retry-After` sleeps, but it does not expose the requested per-provider or per-candidate retry ceiling.

## What This Changes

Request to add a configurable model-provider retry limit so fallback chains can move to the next candidate faster during rate-limit or overload responses.

## Best Possible Solution

Decide one supported retry contract for model providers, then either expose and document a provider/model retry-count setting threaded into each SDK transport or explicitly document the existing retry-after cap as the intended boundary.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested max-retry config surface; live provider proof would only quantify the delay.

## Solution Assessment

No. The current main mitigation is useful for long `Retry-After` sleeps, but it is not the requested configurable per-provider or per-candidate retry ceiling.
