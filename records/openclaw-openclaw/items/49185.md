## Summary

Keep this open. Current main has the adjacent SDK Retry-After wait cap from #68474 and provider request timeouts, but it still does not expose a typed per-provider or per-candidate model API retry-count budget; the current strict schemas would reject the proposed config fields.

## What This Changes

Add a configurable per-provider or per-candidate retry count for model API calls so fallback chains advance faster on rate-limit or overloaded responses.

## Best Possible Solution

Add a typed provider request retry budget in an approved model-provider config location, plumb it only into SDK transports that honor retry counts, and document/direct-test transports where the setting is unsupported or intentionally a no-op.

## Reproduction Assessment

Yes for the missing configuration path: current main has strict schemas with no provider or candidate retry budget, and SDK construction only forwards timeout. I did not live-reproduce the reported multi-provider 429/529 delay.

## Solution Assessment

Unclear until maintainers choose the public config shape; a provider-scoped request retry budget appears to be the narrowest maintainable direction. Timeouts and the existing Retry-After cap are useful mitigations but do not directly control SDK retry attempts before fallback advances.
