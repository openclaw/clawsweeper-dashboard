## Summary

Keep open: this PR targets a real DeepSeek/Foundry tool-call bug and includes sufficient after-fix runtime-path proof, but the patch should not merge as-is because recovered DSML tool calls are emitted before the existing text filter emits surrounding visible text, which can reorder assistant content around the tool call.

## What This Changes

The PR adds DeepSeek DSML text-to-toolCall recovery in `processOpenAICompletionsStream` plus regression tests for observed, split, and malformed DSML payloads.

## Best Possible Solution

Land a narrowly gated DSML recovery path that interleaves filtered visible text and recovered tool calls in stream order, while preserving native `choiceDelta.tool_calls` behavior and covering pure, split, malformed, and mixed visible-text DSML cases.

## Reproduction Assessment

Yes. The linked issue supplies a live Foundry CLI payload, and current main source confirms that only native `choiceDelta.tool_calls` become executable while DSML text is filtered; I did not rerun credentialed Foundry in this read-only review.

## Solution Assessment

No, not as currently ordered. The recovery direction is the right narrow fix, but it should preserve existing stream content order around recovered DSML blocks before merge.
