## Summary

Keep open. Current main still appears to have the stale `lastAssistant` fallback path that this PR is trying to fix, and the paired bug remains open, but the PR's broad aborted-run early return would regress timeout notices and compaction bookkeeping before merge.

## What This Changes

The PR changes the embedded runner to omit `lastAssistant` for aborted empty attempts and changes auto-reply delivery to skip block-reply flushing plus payload delivery whenever run metadata is marked aborted.

## Best Possible Solution

Narrow the fix to suppress stale `lastAssistant` fallback only for externally interrupted runs with no current-attempt assistant output, while preserving timeout error delivery, usage persistence, block/payload delivery semantics that are still valid, compaction accounting, rotated transcript refresh, and post-compaction context injection. Add regression coverage for the interrupt replay case plus timeout and compaction safeguards.

## Reproduction Assessment

Yes. There is a high-confidence source-based reproduction path: configure `messages.queue.mode` to `interrupt`, send a second message while a run is active, and the aborted attempt can have empty `assistantTexts` while `lastAssistant` still resolves from prior session history and feeds the payload fallback.

## Solution Assessment

No. Guarding the stale `lastAssistant` fallback is the right direction, but treating every `runResult.meta?.aborted` as a reason to skip reply delivery is too broad because timeout and compaction paths also use aborted metadata for intentional, stateful outcomes.
