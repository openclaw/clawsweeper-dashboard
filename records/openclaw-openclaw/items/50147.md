## Summary

Keep open for maintainer/contributor follow-up: the stale `lastAssistant` guard targets a real current-main bug path, but the blanket aborted-run early return would drop legitimate timeout output and skip compaction bookkeeping, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR guards the embedded runner’s `lastAssistant` fallback on aborted empty attempts and skips auto-reply flush plus payload delivery for aborted runs.

## Best Possible Solution

Keep the narrow stale `lastAssistant` suppression, but make any silent aborted-run exit specific to superseded interrupt runs with no new output while preserving timeout/error payload delivery, compaction accounting, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows interrupt mode aborts active work, an aborted empty attempt can retain an older `lastAssistant`, and payload delivery can still turn that fallback into a user-visible reply; I did not run a live multi-channel repro in this read-only review.

## Solution Assessment

No. The `lastAssistant` guard is the right direction, but treating every `meta.aborted` result as undeliverable is too broad because timeout and compaction paths intentionally carry useful outcomes on aborted metadata.

Overall correctness: patch is incorrect
