## Summary

Keep open: the diagnostic change is useful but not merge-ready because the new no-provider warning is unreachable and selected-image zero-output paths where every attempt is skipped still produce no warning. Current main has a generic failed-decision warning, so the repair should fit around that instead of adding duplicate or dead logging.

## What This Changes

The PR adds image-only warning logs in `runCapability` when selected image attachments produce zero media-understanding outputs.

## Best Possible Solution

Land a narrow diagnostics update that warns once for selected-image zero-output outcomes across failed attempts, all-skipped attempts, and no resolved providers, using the existing logger with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, by source inspection. The no-provider case returns before the new block can run, and the all-skipped case leaves `failedAttempts.length === 0` with `allAttempts.length > 0`, so neither warning branch fires.

## Solution Assessment

No. The intended direction is narrow, but the patch should move the no-provider warning to the early return path, add an all-skipped diagnostic branch, and use current logger/test/changelog patterns.

## Review Findings

  - body: When no media-understanding providers resolve, `runCapability` returns at the earlier `resolvedEntries.length === 0` guard, so this `allAttempts.length === 0` branch is unreachable. The no-provider warning needs to run at that guard or it will never diagnose the path it names.

  - confidence: 0.9
