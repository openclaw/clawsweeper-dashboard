## Summary

Keep open. The underlying diagnostic need still exists on current main, but the proposed patch is not correct as-is: one warning is placed after the no-provider early return, the all-skipped zero-output path still stays quiet, and the new diagnostics bypass the existing OpenClaw warning logger.

## What This Changes

The PR adds warning diagnostics in `src/media-understanding/runner.ts` when image capability processing selects attachments but produces no outputs.

## Best Possible Solution

Implement diagnostics at the actual runner decision points: warn before the no-provider early return, warn when selected image attempts all skip or return empty output, preserve useful failed-attempt context, use `logWarn`, and add focused regression tests without changing successful image routing or model selection.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: no resolved image providers return before the PR’s new block, and all-empty or skipped provider attempts produce a skipped aggregate decision with no warning-level log.

## Solution Assessment

No. The requested direction is narrow and maintainable, but this patch places one warning after an early return, misses the all-skipped path, and uses `console.warn` instead of the existing OpenClaw logger path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

  - body: The new diagnostics call `console.warn` directly while existing media-understanding warnings use `logWarn`, which also reaches OpenClaw logging. Operators relying on normal gateway logs may miss these messages.
