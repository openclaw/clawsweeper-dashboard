## Summary

Keep open: current main still has a Telegram progress-mode path where a shorter dispatcher final is sent as the durable visible reply without the transcript-aware longer-final selection used by normal preview finalization.

## What This Changes

Reported Telegram progress-mode replies can show a shortened final message while the full Codex final answer remains stored in the session transcript.

## Best Possible Solution

Make Telegram progress-mode final delivery reuse the same transcript-aware full-final selection/fallback used by normal preview finalization, then prove the visible Telegram reply, delivery mirror, and native transcript final agree.

## Reproduction Assessment

Yes at source level: in Telegram progress mode, a shorter final payload passed to deliverProgressModeFinalAnswer is sent durably without resolving the longer transcript final candidate. I did not run a live Telegram current-main reproduction.

## Solution Assessment

Yes for the narrow repair direction: progress-mode final delivery should reuse transcript-aware final selection before durable send. Changing streaming defaults or removing the GPT brevity guard broadly would be less targeted and may conflict with existing tests.
