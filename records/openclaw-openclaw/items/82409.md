## Summary

Keep open: the report maps to a real current Telegram progress-mode final-delivery path, and current main does not yet prove that a shorter dispatcher final is reconciled with the full Codex transcript final before visible Telegram delivery.

## What This Changes

Ensure Telegram visible final replies deliver the full Codex final answer from the session/transcript instead of a shorter progress preview or dispatcher payload.

## Best Possible Solution

Unify progress-mode final delivery with the transcript-aware full-final selection used by normal Telegram preview finalization, then prove that visible Telegram output, delivery mirror, and native transcript text agree.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was run, but source inspection gives a clear reproduction shape: a Telegram progress-mode turn whose dispatcher final payload is shorter than the transcript final will send the shorter payload through `deliverProgressModeFinalAnswer`.

## Solution Assessment

Is this the best way to solve the issue? Yes for the proven first boundary: make progress-mode final delivery use the same full-final candidate resolution and regression coverage as normal Telegram preview finalization, while adding live instrumentation if a lower-layer Telegram-visible mismatch remains.
