## Summary

Keep this PR open. Current main still has no top-level `auth` reload rule, so `auth.profiles.*` changes fall through to the restart branch; the proposed patch is a narrow, plausible fix with regression coverage, and the discussion includes a second-provider confirmation on `auth.profiles.microsoft:<email>`.

## What This Changes

The PR adds a top-level `auth` no-op rule to the gateway reload planner and tests OAuth auth-profile refreshes plus mixed `gateway.auth.*` restart batches.

## Best Possible Solution

Land this PR or an equivalent narrow change so OAuth auth-profile refresh writes do not restart the gateway, while preserving the existing restart behavior for `gateway.auth.*` and keeping the regression tests with the reload planner.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `auth.profiles.<provider>:<email>` matches no reload rule and therefore takes the unmatched-path restart branch in `buildGatewayReloadPlan()`.

## Solution Assessment

Yes, with a scope caveat. Adding an `auth` no-op rule is the smallest implementation shown for the reported `auth.profiles.*` restart, and the test for a mixed `gateway.auth.token` batch protects the important gateway-auth restart behavior.
