## Summary

Keep open for maintainer review: the PR targets a real delivery concern and has after-fix live logs, but the latest diff appears to bypass the current active-requester queue contract by starting the requester-agent handoff after any active wake failure.

## What This Changes

The PR removes the active-requester early return in subagent announcement delivery so failed embedded wake attempts fall through to `callGateway`, and updates the focused test expectation.

## Best Possible Solution

Decide whether active-but-non-consuming requesters should remain on the queue fallback contract or get a new narrowly defined direct handoff state; then align code, tests, and docs around that single contract.

## Reproduction Assessment

Yes at source level for the branch condition: an active requester with a failed embedded wake returns before the later `callGateway` block. I did not establish a full current-main end-to-end drop because current dispatch then queues fallback.

## Solution Assessment

No, not as written: removing the guard falls through for all active wake failures, while current docs/tests route those cases through dispatch queue fallback to avoid a second visible reply path. A safer fix would first distinguish the specific non-consuming state that should bypass queue fallback, if maintainers want that behavior.
