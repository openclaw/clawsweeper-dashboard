## Summary

Keep open: the linked bug is credible, but this PR is not mergeable as written because it lacks real behavior proof, introduces type-contract failures, and reintroduces a direct raw-send fallback that current main explicitly removed from subagent completion delivery.

## What This Changes

The PR adds a `sendMessage` fallback to subagent announce delivery, falls through to `callGateway` after active-requester wake failures, and updates tests for Telegram/Slack completion delivery paths.

## Best Possible Solution

Fix the active-but-non-consuming requester case by reaching the requester-agent handoff or queue/retry path without raw direct sends, keep result types aligned, add focused regression coverage, and provide redacted real behavior proof from a real setup.

## Reproduction Assessment

Yes, source-level. A focused seam can set `expectsCompletionMessage`, return an active requester session, make `queueEmbeddedPiMessage` return false, and observe current main returning undelivered before the `callGateway` handoff below runs.

## Solution Assessment

No as written. Falling through to the requester-agent handoff may be the right direction, but the raw `sendMessage` fallback violates the current delivery contract and the new code does not satisfy the existing TypeScript result/event types.
