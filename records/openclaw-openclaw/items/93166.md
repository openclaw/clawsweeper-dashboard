## Summary

Keep open: current main and v2026.6.8 still have the source-reproducible retry path, and the only linked fix PR was closed unmerged after maintainer feedback that its restore-on-silent approach violates the finalization contract.

## What This Changes

The issue reports that a `before_agent_finalize` retry can make a real first response visible to the retry pass, then drop delivery when the retry produces `NO_REPLY`.

## Best Possible Solution

Repair retry/transcript ordering in the embedded runner so the revision pass does not see a rejected pre-revision answer as already finalized, while keeping `NO_REPLY` authoritative when it is the revised final outcome.

## Reproduction Assessment

Yes at source level, but not as an executed live repro in this pass. The runner can honor a revise decision with a hidden continuation, and the exact `NO_REPLY` path can resolve the retry as silent with no deliverable payload.

## Solution Assessment

No, the linked restore-on-silent PR is not the best solution because maintainer feedback says it can violate the finalization contract. The safer fix is to repair retry-visible transcript ordering while preserving the revision pass as the final decision.
