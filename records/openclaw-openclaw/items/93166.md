## Summary

Keep open: current main and the latest release still have a source-reproducible path where `before_agent_finalize` runs after assistant transcript persistence, suppresses the first terminal delivery, and only suppresses the next user-message persistence before retry; I found no canonical duplicate or merged fix.

## What This Changes

The issue reports that `before_agent_finalize` revise/retry can commit the first assistant answer before the retry, then deliver a later silent response and drop the visible reply.

## Best Possible Solution

Make revise attempts replay-safe in the core run loop by keeping the pre-revision assistant answer out of retry-visible history or rolling it back before retry, while preserving the side-effect guard and silent-reply contract.

## Reproduction Assessment

No live repro was run, so we do not have a high-confidence executed reproduction. The source-level path is clear: the hook gate runs after assistant transcript write, suppresses first delivery, and the retry loop only suppresses the next user-message persistence.

## Solution Assessment

Unclear exact patch shape until implementation, but the best fix belongs in the core run/subscription transcript ordering rather than a channel-specific fallback. A safe fix should add regression coverage for revise plus group silent-reply behavior.
