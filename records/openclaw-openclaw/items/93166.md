## Summary

Keep open: current `main` and the latest release still show the source path where a `before_agent_finalize` revision suppresses first delivery, retries, and then resolves an empty/silent revision as the terminal payload. The only linked fix PR was closed unmerged after a maintainer explained that restoring the suppressed answer violates the finalization contract, so the remaining work needs a different retry/transcript-ordering fix.

## What This Changes

The issue reports that `before_agent_finalize` retry handling can suppress a deliverable first response and then deliver nothing when the retry pass resolves to a silent group-chat reply.

## Best Possible Solution

Repair the core retry/transcript ordering so an honored `before_agent_finalize` revision does not make the retry pass see a finalized copy of the rejected first answer, while preserving the revision pass as authoritative and adding a silent group-chat regression test.

## Reproduction Assessment

No executed high-confidence repro was run in this read-only review. The source path is clear enough to reproduce in a focused harness: first attempt returns visible text and a finalize revision, the retry pass returns `NO_REPLY`, the payload builder strips it, and terminal delivery resolves to silence.

## Solution Assessment

No, the closed restore-after-silent-revision PR is not the best fix because maintainer review identified transcript duplication and finalization-contract risk. The better solution is to fix retry/transcript ordering or retry context so the revision pass remains authoritative without seeing a finalized copy of the rejected first answer.
