## Label Justifications

- `P2`: This is a bounded but real core agent delivery bug that can silently suppress a response in finalize-hook retry flows.
- `impact:message-loss`: The reported failure drops the assistant response that should have been delivered to the channel.
- `impact:session-state`: The failure depends on transcript/session state being updated before the finalize-hook retry is complete.

## Summary

Keep open: current main and latest release still have the finalize-hook retry path where a suppressed first answer can be followed by a silent retry pass that becomes the terminal payload. The direct fix candidate was closed unmerged because restoring the pre-revision answer violates the finalization contract, so this still needs a retry/transcript-ordering repair.

## What This Changes

The issue asks OpenClaw to stop dropping a deliverable first response when `before_agent_finalize` requests a retry and the retry pass returns a group-chat silent token.

## Best Possible Solution

Repair the core retry/transcript ordering so a finalize-hook retry does not make the retry pass see the rejected first answer as finalized history, while keeping the revision pass authoritative and adding a silent group-chat regression test.

## Reproduction Assessment

Yes, source-reproducible: a focused harness can make attempt one return visible text plus a finalize revision, then make attempt two return `NO_REPLY` with `allowEmptyAssistantReplyAsSilent`. I did not execute that harness in this read-only review.

## Solution Assessment

No, the closed restore-after-silent PR is not the best solution because maintainer feedback identified finalization-contract and transcript-duplication problems. The best path is a retry/transcript-ordering fix that prevents the retry pass from seeing the rejected answer as finalized context.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/93166 on current main. The observable bug is that an honored `before_agent_finalize` revise decision can suppress the first deliverable answer, retry after that answer has entered transcript context, and then deliver silence when the retry pass returns `NO_REPLY` in a group-chat silent-reply flow. Preserve the finalization contract from https://github.com/openclaw/openclaw/pull/93480: do not restore a rejected pre-revision answer after a silent/empty revision, and avoid transcript duplication. Prefer repairing retry/transcript ordering so the retry pass does not see the rejected first answer as finalized history. Add focused regression coverage for first attempt visible text plus finalize revision followed by a `NO_REPLY` retry with `allowEmptyAssistantReplyAsSilent`; also keep the real revised-answer path covered. Do not edit `CHANGELOG.md`, add config, change plugin SDK shape, or reopen WebChat early-streaming behavior from https://github.com/openclaw/openclaw/pull/92382 unless the fix truly requires it.
