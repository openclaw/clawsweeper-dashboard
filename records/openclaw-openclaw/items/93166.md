## Summary

Keep open: current main and the latest release still have a source-reproducible path where the finalize hook runs after the assistant candidate is in session state, then retries with only duplicate user-message persistence suppressed. I found no merged or open PR that closes this specific retry/history/drop behavior, and the closest related PR was closed after a maintainer rejected changing the hook safety contract.

## What This Changes

The issue reports that `before_agent_finalize` revise/retry can commit the first response into session history, then deliver nothing when the second pass emits the silent token.

## Best Possible Solution

Make finalize-revision retries replay-safe in the core embedded run loop while preserving the maintainer-confirmed safety contract that rejected finalize-hook drafts are not shown as accepted replies.

## Reproduction Assessment

Yes at source level, not as an executed live repro. Current main still runs the hook from the terminal-delivery gate after assistant session state exists, then retries with only user-message persistence suppressed.

## Solution Assessment

Yes, the right fix belongs in the core embedded run-loop/finalize gate rather than a plugin workaround. The repair should avoid the closed PR's rejected early-preview approach and instead make the retry prompt/history or candidate persistence replay-safe.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/93166 by making `before_agent_finalize` revise/retry safe in the embedded agent run loop. Current main runs the hook from the terminal-delivery gate after the assistant candidate is already in session state and then retries with only `suppressNextUserMessagePersistence`; a group-chat retry can therefore see the first answer and produce `NO_REPLY`, causing no delivered response. Preserve the safety contract from closed https://github.com/openclaw/openclaw/pull/92382: do not make rejected finalize-hook drafts visible as accepted replies. Prefer a core run-loop/session-history fix that prevents the first candidate from influencing the retry, or otherwise makes the retry/delivery decision deterministic without adding config. Add focused regression coverage for a finalize hook that requests revise and a second pass that would otherwise see the first assistant answer and suppress delivery. Also check whether the secondary hot-reload hook cleanup note needs a separate issue rather than bundling it. Do not change `CHANGELOG.md`. Validate with targeted tests through `node scripts/run-vitest.mjs` for the touched embedded runner/subscription tests and `git diff --check`.
