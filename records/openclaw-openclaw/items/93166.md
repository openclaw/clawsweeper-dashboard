## Summary

Keep open: current main and the latest release still contain the finalize-hook retry path where the first answer is suppressed, the run continues, and a silent retry pass becomes the terminal payload. The directly linked restore-based PR was closed unmerged after maintainer feedback because that approach violated the finalization contract, so the remaining fix should target retry/transcript ordering instead.

## What This Changes

The issue requests fixing core `before_agent_finalize` retry behavior that can commit a first answer into session context before a revision pass, causing group-chat silent-token handling to drop the deliverable response.

## Best Possible Solution

Repair the core retry/transcript ordering so an honored finalize-hook revision does not make the retry pass see the rejected first answer as finalized history, while keeping the revision pass authoritative and adding a silent group-chat regression test.

## Reproduction Assessment

Yes at source level, but not as an executed live repro in this read-only review. A focused harness can model a visible first answer plus finalize revision, then a second pass returning `NO_REPLY` with `allowEmptyAssistantReplyAsSilent`, which current payload selection resolves to silence.

## Solution Assessment

Yes for keeping the issue open and fixing retry/transcript ordering. The closed restore-after-silent PR is not the best fix because maintainer feedback identified finalization-contract and transcript-duplication problems.
