## What This Changes

The issue reports a Telegram direct-message SQLite ingress event hot-looping on `reply session initialization conflicted`, blocking later DM updates behind the pending row.

## Best Possible Solution

Review and land a fix that both absorbs legitimate reply-session stale-snapshot races and backs off Telegram spooled replay conflicts without dropping retryable messages, then close this issue once direct-message coverage is confirmed.

## Reproduction Assessment

Yes at source level: current `main` throws the reported reply-session initialization conflict after one stale-snapshot retry, and Telegram releases unclassified spooled failures for retry. I did not run a live Telegram DM reproduction in this read-only review.

## Solution Assessment

Yes, the best shape is to handle the race at the reply-session owner boundary and add Telegram-owned backoff for retryable spooled conflicts. The open candidate PR appears to target that shape, but this issue should remain open until it lands or maintainers confirm separate DM coverage.
