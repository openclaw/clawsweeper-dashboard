The issue reports a Telegram direct-message ingress update stuck as a pending SQLite `channel_ingress_events` row and retried in a tight loop after `reply session initialization conflicted` failures.

## Reproduction Assessment

Yes at source level: current main retries one stale reply-session initialization snapshot and then throws, while Telegram releases unclassified spooled failures back to pending. I did not run a live Telegram DM reproduction in this read-only review.

## Solution Assessment

Yes, the best fix shape is to handle legitimate reply-session stale-snapshot races at the session owner boundary and add targeted Telegram spool backoff for retryable conflicts. Dead-lettering this conflict by default would risk dropping a legitimate optimistic-concurrency retry.
