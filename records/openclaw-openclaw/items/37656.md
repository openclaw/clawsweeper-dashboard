## Summary

Keep open: current main still lacks this feature, and the latest PR head still has a security-sensitive gap because exec-time workspace `.env` loading bypasses the existing untrusted-workspace dotenv blocklist. The external PR also still lacks after-fix real behavior proof from a real gateway/agent exec setup.

## What This Changes

The branch adds request-time workspace `.env` loading for gateway-host exec, filters host-env overrides, updates exec/MS Teams tests, and adds a changelog entry.

## Best Possible Solution

Keep the feature scoped to gateway-host exec, but route workspace `.env` parsing through the existing workspace dotenv filtering contract before injection and add regression coverage for blocked keys plus redacted real behavior proof.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection shows current main lacks per-request workspace `.env` exec loading, so there is no broken documented behavior to reproduce as a bug.

## Solution Assessment

No, not yet. The gateway-only direction is plausible, but the PR should reuse the existing workspace dotenv blocklist before merging repo-controlled `.env` values into exec env and still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Apply workspace dotenv blocklist before injecting exec env:** `src/agents/bash-tools.exec.workspace-env.ts:50`
  - body: `loadWorkspaceDotEnvForExec` returns every parsed key that is not already in `baseEnv`, and `createExecTool` then treats those values as inherited env after only host override filtering. Reuse/export the existing workspace dotenv filtering before returning `workspaceEnv` and add blocked-key coverage.
