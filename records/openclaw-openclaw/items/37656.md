## Summary

Keep this PR open, but it is not mergeable yet: the current diff still lets exec-time workspace `.env` loading bypass the established untrusted workspace dotenv denylist, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds request-time workspace `.env` loading for gateway-host exec, related exec tests, an async MS Teams status-line test fix, and a changelog entry.

## Best Possible Solution

Keep the gateway-only exec-time loading shape, but route parsed workspace `.env` entries through the same workspace dotenv denylist used by current startup loading before they become host exec env.

## Reproduction Assessment

Not applicable to the feature request as a user bug. The blocking security/correctness issue is source-reproducible from PR head because parsed workspace `.env` keys are injected before the established workspace dotenv denylist is applied.

## Solution Assessment

No, not yet. Gateway-only request-time loading is a plausible solution, but it should share the current untrusted workspace `.env` filtering contract and then be backed by redacted real behavior proof.
