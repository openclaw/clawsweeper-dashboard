## Summary

Keep this PR open. Current main still does not load a selected agent workspace `.env` into gateway exec at request time, so the PR is not obsolete, but the current patch has blocking review/security concerns around workspace `.env` policy reuse and test isolation.

## What This Changes

The PR adds a gateway-only workspace `.env` loader for exec, merges accepted values into the host exec environment after workdir resolution, adds gateway/node coverage, updates one exec test, and adjusts an MS Teams setup-surface async assertion.

## Best Possible Solution

Land a revised implementation that loads the resolved gateway workspace `.env` at exec request time, skips node and sandbox hosts, shares the existing workspace dotenv key policy with the new non-mutating loader, preserves non-override semantics including Windows casing, uses isolated temp-workspace tests, and updates the environment docs when the behavior ships.

## Reproduction Assessment

Yes. A focused check can create a temp workspace with `.env`, run gateway exec with that workspace as cwd, and assert the var is absent on current main but present after the intended change; the source inspection shows current main only passes the workspace as `cwd`.

## Solution Assessment

No, not in the current form. The PR targets the right boundary, but the safer solution is to share the existing workspace dotenv blocklist and isolate the regression test before this host exec environment change lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Apply the workspace dotenv blocklist:** `src/agents/bash-tools.exec.workspace-env.ts:35-42`
  - body: Current startup workspace `.env` loading blocks credentials and OpenClaw runtime-control keys, but this new loader parses every key and relies only on host override diagnostics later. Keys such as `OPENAI_API_KEY` and `OPENCLAW_GATEWAY_TOKEN` are not host override-blocked, so a repo `.env` can inject them into gateway exec child environments; reuse or export the workspace dotenv policy before returning injected vars.
  - confidence: 0.86
- **[P2] Use a temp workspace for the dotenv injection test:** `src/agents/bash-tools.test.ts:821-837`
  - body: This test writes `.env` into `process.cwd()`, which is the repo root under Vitest. Other dotenv/config-loading tests also read the CWD `.env`, so parallel workers can observe `GH_CONFIG_DIR` and fail nondeterministically; create an isolated temp workspace and pass that as `cwd` instead.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff changes host exec environment sourcing and currently bypasses the existing workspace dotenv blocklist.

Concerns:

- **[high] Workspace dotenv policy bypass:** `src/agents/bash-tools.exec.workspace-env.ts:35`
  - body: `loadWorkspaceDotEnvForExec` parses workspace `.env` entries without applying the existing workspace dotenv blocklist from `src/infra/dotenv.ts`. Host override diagnostics block many command-hijack keys, but not all workspace-blocked credential/runtime keys, so repo-controlled `.env` values can enter gateway exec child environments where current workspace dotenv loading would drop them.
