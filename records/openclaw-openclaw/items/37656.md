## Summary

Keep open. The branch still has a blocking security regression: its exec-time loader returns parsed workspace `.env` keys before applying the existing untrusted workspace dotenv denylist, so runtime-control and endpoint-routing keys can flow into gateway-host exec despite current docs/tests forbidding that; it also still lacks contributor real-behavior proof.

## What This Changes

The branch adds gateway-host exec-time workspace `.env` loading from the resolved workdir, filters it through host override diagnostics, adds focused exec tests, updates one MS Teams async status assertion, and adds a changelog entry.

## Best Possible Solution

Keep the gateway-only request-time loading shape, but factor or export the existing workspace dotenv filter for the exec loader and require redacted live gateway/agent exec proof before merge.

## Reproduction Assessment

Not applicable as a user-bug reproduction path for this feature request. The blocking PR defect is source-reproducible from PR head: parsed `.env` keys are injected after only non-override filtering and before the existing workspace dotenv denylist is applied.

## Solution Assessment

No, not yet. Gateway-only request-time loading is a plausible direction, but the implementation must share the current untrusted workspace `.env` filtering contract and include redacted live behavior proof before it is safe to merge.
