## Summary

Keep open. Current main still does not implement exec-time per-agent workspace `.env` loading, and this PR targets the right boundary, but the latest head still has a host exec environment security issue plus a test isolation bug that should block merge.

## What This Changes

The PR adds an async gateway-only workspace `.env` loader for exec, merges accepted values into the host exec environment after workdir resolution, adds gateway/node coverage and one host exec dotenv test, and updates an MS Teams setup-surface async assertion.

## Best Possible Solution

Land a revised implementation on this PR or a maintainer replacement that reads the resolved gateway agent workspace `.env` at exec request time, skips node and sandbox hosts, shares the existing workspace dotenv key policy through a non-mutating helper, preserves non-override semantics including Windows casing, uses isolated temp-workspace tests, and updates the environment docs and changelog when shipped.

## Reproduction Assessment

Yes. A focused check can create a temp workspace with `.env`, run gateway exec with that workspace as cwd, and assert the var is absent on current main but present after the intended change; the security finding can be reproduced on the PR by putting `OPENAI_API_KEY` or `OPENCLAW_GATEWAY_TOKEN` in that `.env` and observing that the new loader does not apply the existing workspace dotenv blocklist.

## Solution Assessment

No, not in the current form. The PR targets the right exec boundary, but the safer maintainable solution is to reuse the existing workspace dotenv policy before merging values into host exec env and to isolate the regression test from the repo root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Apply the workspace dotenv blocklist:** `src/agents/bash-tools.exec.workspace-env.ts:35-42`
  - body: Current startup workspace `.env` loading blocks credential and OpenClaw runtime-control keys, but this new loader parses every key and only skips already-defined vars. Host override diagnostics later block command-hijack keys such as `NPM_CONFIG_*`, but not workspace-blocked keys like `OPENAI_API_KEY` or `OPENCLAW_GATEWAY_TOKEN`, so repo-controlled `.env` values can enter gateway exec child environments. Reuse or export the workspace dotenv policy before returning injected vars.
  - confidence: 0.86
- **[P2] Use a temp workspace for the dotenv injection test:** `src/agents/bash-tools.test.ts:821-837`
  - body: This test writes `.env` into `process.cwd()`, which is the repo root under Vitest. Other dotenv/config-loading tests also read the CWD `.env`, so parallel workers can observe `GH_CONFIG_DIR` and fail nondeterministically; create an isolated temp workspace and pass that as `cwd` instead.
  - confidence: 0.93
