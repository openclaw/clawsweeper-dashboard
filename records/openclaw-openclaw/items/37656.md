Full review comments:

- **[P1] Reuse the workspace dotenv safety policy:** `src/agents/bash-tools.exec.workspace-env.ts:31-51`
  - body: The new loader parses every `.env` entry and only skips keys already present in `baseEnv`; the later host override filter does not cover current main's workspace-dotenv blocklist for `OPENCLAW_*`, provider credentials, or endpoint-routing keys. That lets a repo-local `.env` inject runtime-control variables into host exec children even though startup dotenv loading now blocks them, so this should share a pure workspace-dotenv parser/policy and add regression coverage.
  - confidence: 0.9
- **[P2] Move the .env test into a temp workdir:** `src/agents/bash-tools.test.ts:600-619`
  - body: The test writes `process.cwd()/.env`, which is the repo root in normal Vitest runs. Parallel dotenv tests can observe the temporary `GH_CONFIG_DIR`, and an interrupted run can leave the checkout dirty; create an isolated temp directory and pass it as the exec cwd instead.
  - confidence: 0.92
- **[P2] Add release notes for the new env source:** `src/agents/bash-tools.exec.ts:365-373`
  - body: This is a user-facing `feat` that changes exec environment precedence, but the PR does not update `CHANGELOG.md`. Add a single-line entry describing the gateway-host exec workspace `.env` behavior and scope.
  - confidence: 0.88
- **[P3] Document exec-time workspace .env precedence:** `src/agents/bash-tools.exec.ts:365-373`
  - body: The environment docs still say OpenClaw loads the process env, CWD `.env`, global `.env`, config env, and optional shell import. If this feature lands, those docs need to explain the gateway-host exec-time workspace `.env` source and its non-override/blocklist behavior.
  - confidence: 0.82
