## Summary

Keep open: the feature is still useful and is not implemented on current main, but the PR should not merge yet because the new exec-time workspace `.env` path bypasses the existing untrusted-workspace dotenv blocklist and the added regression test mutates the repository root `.env`.

## What This Changes

The PR adds gateway-host exec-time loading of a resolved workspace `.env`, merges accepted vars into the exec environment, adds host-selection coverage, and adjusts an MS Teams setup test for an async status resolver.

## Best Possible Solution

Land a version that keeps exec-time workspace `.env` loading for gateway host exec but shares the existing workspace dotenv trust policy, uses isolated test workdirs, and includes the required changelog entry.

## Reproduction Assessment

Yes, by source inspection for the PR findings: PR head reads `<workdir>/.env`, accepts parsed keys not already in `process.env`, and merges them into gateway host exec env without the current workspace dotenv blocklist. The test-isolation issue is also clear from the added `process.cwd()/.env` write path.

## Solution Assessment

No. Loading from the resolved gateway workdir is the right shape, but duplicating dotenv parsing/filtering is not the best implementation; the safer path is to reuse or export the existing workspace dotenv parser/blocklist and keep tests in a temp directory.
