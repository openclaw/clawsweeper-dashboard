## Summary

Keep this PR open: the latest update fixes the prior test/changelog blockers, but the patch still bypasses the existing workspace `.env` trust-boundary filter for gateway-host exec, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds request-time workspace `.env` loading for gateway-host exec, filters some host-env override keys, adds focused exec tests, updates one MS Teams async status test, and adds a changelog entry.

## Best Possible Solution

Keep the feature scoped to gateway-host exec, reuse the shared workspace dotenv filtering policy before injection, add regression coverage for blocked keys, and require redacted live proof of an allowed workspace var reaching exec while blocked keys stay out.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection shows current main lacks per-request workspace `.env` exec loading, so there is no broken documented behavior to reproduce as a bug.

## Solution Assessment

No, not yet. The gateway-only direction is plausible, but the PR's custom loader must preserve the existing workspace `.env` blocklist before it is a maintainable or secure solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Apply the workspace dotenv blocklist before injecting exec env:** `src/agents/bash-tools.exec.workspace-env.ts:51`
  - body: `loadWorkspaceDotEnvForExec` accepts every parsed key that is not already in `baseEnv`, bypassing the existing workspace `.env` denylist in `src/infra/dotenv.ts`. A repo-controlled `.env` can therefore inject `OPENCLAW_*`, provider credential, endpoint, or proxy/TLS keys into gateway-host exec even though the current security docs say those workspace keys are blocked. Reuse/export the existing workspace dotenv filtering before returning injected values and add regression coverage.
  - confidence: 0.9
