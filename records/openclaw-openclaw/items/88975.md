## What This Changes

The PR adds QQ Bot media-tag guards and regression tests so `file://` rich-media tag contents are skipped before outbound deliver, legacy `sendText`, and streaming media parsing queue a file.

## Best Possible Solution

Land the narrow guard only after maintainer/security approval and redacted QQ Bot-capable proof that blocked tags are consumed while normal absolute local paths and URLs still deliver.

## Reproduction Assessment

Yes, at source level: current main strips lowercase `file://` in the media decoder and then queues decoded media paths for sending. I did not establish a live QQ Bot runtime reproduction, so this is not a real-environment proof.

## Solution Assessment

Yes, the guard location is the right narrow boundary because raw tag text must be checked before path normalization and blocked streaming tags still need to be consumed. The safer merge path is to pair this with real QQ Bot-capable proof and explicit maintainer acceptance of the compatibility break.
