## What This Changes

The PR direct-sends text-only block replies through the shared auto-reply handler when block streaming is off, then suppresses duplicate plain final assistant text after directly sent text chunks.

## Best Possible Solution

Preserve ordered Discord/tool delivery through a maintainer-approved shared or channel-specific policy, update docs if `blockStreaming=false` is no longer final-only, and land only after redacted live channel proof shows correct ordering and no duplicate or lost final text.

## Reproduction Assessment

Yes, at source level. Current main flushes before tool execution, but `createBlockReplyDeliveryHandler` does not send text-only blocks when `blockStreamingEnabled` is false, so text can be deferred until the final payload after tool results.

## Solution Assessment

Not yet. The shared handler is a plausible owner boundary, but the PR silently changes the documented final-only `blockStreaming=false` contract across channels; a safer solution needs maintainer acceptance, docs if the contract changes, and live transport proof.
