## Summary

Keep open: this draft protected-label PR is active, and the latest head still has blocking source-level issues around lint startup, repair-mode plugin checks, and the public health SDK surface.

## What This Changes

The PR adds structured doctor health checks, `doctor --lint`, post-repair validation, a health SDK subpath, docs, and focused tests.

## Best Possible Solution

Keep the structured doctor direction, but defer or scope plugin loading so lint can always emit its own structured failures, make repair-mode plugin checks load at a safe point, and keep test-only registry mutators out of the public SDK.

## Reproduction Assessment

Yes. Source inspection of the PR head and current-main startup path shows the lint plugin preload runs before `runDoctorLintCli`, and the repair path registers only core checks.

## Solution Assessment

No. The health-check shape may be reasonable, but the current patch does not yet preserve lint recovery behavior, repair-mode plugin participation, or a clean public SDK boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Defer plugin loading until lint can report config findings:** `src/cli/command-catalog.ts:265`
  - body: This policy preloads all plugins in the preAction path for any `doctor --lint` invocation. Because plugin loading resolves `getRuntimeConfig()` and throws on plugin load errors, an invalid config or stale plugin path can abort before `runDoctorLintCli` reaches its structured config-snapshot branch, so `doctor --lint --json` cannot reliably produce the documented findings/exit contract.
  - confidence: 0.88
- **[P2] Register plugin health checks before structured repairs:** `src/flows/doctor-health-contributions.ts:229-231`
  - body: `doctor --fix` does not preload plugins, and the new repair contribution only calls `registerCoreHealthChecks()` before `runDoctorHealthRepairs()`. That means plugin-contributed `repair()` checks never enter the registry during repair mode, despite the docs and PR body saying plugins can participate in `doctor --fix` validation/repair.
  - confidence: 0.84
- **[P2] Keep test registry mutators out of the SDK:** `src/plugin-sdk/health.ts:29-33`
  - body: The new public `openclaw/plugin-sdk/health` subpath exports `clearHealthChecksForTest` and `resetCoreHealthChecksForTest`. Those helpers let production plugin code clear or reset doctor health registration, so they should stay internal/test-only while the public SDK exposes only stable registration/types.
  - confidence: 0.82
