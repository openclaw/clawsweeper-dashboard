## Summary

Keep open: this protected draft still needs the maintainer diagnostics-surface/API decision, and PR head violates its read-only lint contract because one default lint check re-enters observing config reads.

## What This Changes

The branch adds a structured doctor health-check contract, `openclaw doctor --lint`, scoped repair validation, a `plugin-sdk/health` export, docs, and focused tests.

## Best Possible Solution

Land only one maintainer-approved diagnostics surface, with every lint check using a no-observe config read path and the SDK/docs aligned to that chosen health-check boundary.

## Reproduction Assessment

Yes. Source inspection of PR head shows the default `doctor --lint` registry includes `core/doctor/final-config-validation`, and that check calls `readConfigFileSnapshot()` without `{ observe: false }` while config IO defaults observation on.

## Solution Assessment

No, not as-is. The health-check direction is coherent, but the patch must preserve the read-only lint invariant and still needs maintainer direction on `doctor --lint` versus the standalone lint draft.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep final lint validation read-only:** `src/flows/doctor-core-checks.ts:170`
  - body: `doctor --lint` is documented as read-only and the CLI's first config snapshot uses `{ observe: false }`, but the default `core/doctor/final-config-validation` check calls `readConfigFileSnapshot()` with observation enabled. Because observed reads can write config health/audit state, a plain default lint run can still touch state. Pass a no-observe snapshot path through this check, or reuse the lint snapshot from context, and add a default-lint no-state-write regression test.
  - confidence: 0.91
