## What This Changes

The PR adds a structured doctor health-check contract, a read-only `doctor --lint` mode with selection and JSON output, structured repair validation, docs, and focused tests.

## Best Possible Solution

Keep the PR open for maintainer review, fix the lint CLI contract issues, and attach viewable real behavior proof before considering merge.

## Reproduction Assessment

Yes. Source inspection shows fatal `doctor --lint` errors flow through the default wrapper as exit 1, and invalid `--severity-min` values fall back to `info`; both can be reproduced with a rejecting lint runner or bad severity argument on the PR branch.

## Solution Assessment

No. The structured doctor direction may be reasonable, but the CLI contract needs to reject invalid thresholds and reserve exit 2 for runtime failures before this is the maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return exit 2 for lint runtime failures:** `src/cli/program/register.maintenance.ts:35`
  - body: The new docs promise exit code 2 when `doctor --lint` fails before it can emit findings, but this action uses the default `runCommandWithRuntime` error path, which exits 1. That makes fatal runtime failures indistinguishable from normal lint findings for CI/preflight callers, and the new test currently locks in the wrong behavior.
  - confidence: 0.9
- **[P2] Reject invalid severity thresholds:** `src/commands/doctor-lint.ts:40`
  - body: Unsupported `--severity-min` values currently parse to null and then fall back to `info`, so a typo such as `--severity-min warnng` runs the broadest lint threshold instead of failing the command. Since this flag is intended for CI gates, invalid input should be reported as a command error rather than silently changing the gate.
  - confidence: 0.86
