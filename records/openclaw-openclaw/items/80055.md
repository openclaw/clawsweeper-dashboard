## Summary

Keep open: this is a draft feature PR with a protected `maintainer` label, and current main does not already include doctor lint/detect support. Source review found a blocking correctness issue: the new lint path can write config-health state despite being documented as read-only.

## What This Changes

The PR adds a structured doctor health-check contract, `doctor --lint` with JSON/filter options, structured repair validation for migrated checks, a `plugin-sdk/health` subpath, docs, and focused tests.

## Best Possible Solution

Keep the detect/repair contract only if maintainers accept doctor-core lint and the SDK health boundary, and make lint use a config snapshot path that suppresses observation/audit writes.

## Reproduction Assessment

Yes, source-reproducible for the review finding: `runDoctorLintCli` calls `readConfigFileSnapshot()`, whose current finalizer observes reads and can write config-health state. I did not execute the command because this review must keep the checkout and environment read-only.

## Solution Assessment

No as-is. The implementation is a coherent direction for structured health checks, but the lint path must first honor its no-write contract and the SDK/core boundary still needs maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep doctor lint from writing config health state:** `src/commands/doctor-lint.ts:44`
  - body: `doctor --lint` is documented as writing no config/state, but this path enters `readConfigFileSnapshot()`. That reader finalizes by observing the config and can write config-health state for a valid config when the fingerprint changes, so a CI/preflight lint run can mutate `OPENCLAW_HOME` before emitting findings. Use a no-observe snapshot path or add a suppress-observation option for lint.
  - confidence: 0.86
