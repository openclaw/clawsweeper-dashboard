## Summary

This PR should stay open: it is a draft protected-label feature PR, and the current head still violates the documented read-only `doctor --lint` contract by using the normal config snapshot reader that can write config-health state.

## What This Changes

The PR adds a structured doctor health-check contract, `doctor --lint` with JSON/filter options, structured repair validation, migrated core checks, a `plugin-sdk/health` subpath, docs, and focused tests.

## Best Possible Solution

Keep the detect/repair contract only if maintainers accept doctor-core lint and the SDK health boundary, and make lint use a config snapshot path that suppresses observation/audit writes.

## Reproduction Assessment

Yes. The review finding is source-reproducible: `runDoctorLintCli` calls `readConfigFileSnapshot()`, whose current finalizer observes reads and can write config-health state; I did not execute the command because this review must keep the checkout and environment read-only.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Keep doctor lint from writing config health state:** `src/commands/doctor-lint.ts:44`
