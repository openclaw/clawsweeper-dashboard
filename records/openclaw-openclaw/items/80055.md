## Summary

Keep open: this protected draft still needs maintainer direction on the doctor health-check API and the `doctor --lint` versus standalone lint surface, and the current PR head still has a blocking read-only lint regression.

## What This Changes

The PR adds a doctor health-check registry and contract, `openclaw doctor --lint`, structured repair validation, a `plugin-sdk/health` export, docs, and focused tests.

## Best Possible Solution

Choose one maintainer-approved diagnostics surface, then land it with every lint check using a no-observe config read path and the SDK/docs aligned to that boundary.

## Reproduction Assessment

Yes. Source inspection of PR head shows the default lint registry includes `core/doctor/final-config-validation`, and that check calls `readConfigFileSnapshot()` without the no-observe option while config IO observes by default.

## Solution Assessment

No, not as-is. The health-check direction is coherent, but the patch needs the read-only lint fix and maintainer direction on whether the diagnostics surface belongs under `doctor` or as the standalone lint draft.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep final lint validation read-only:** `src/flows/doctor-core-checks.ts:170`
  - body: `doctor --lint` is documented as read-only and the lint CLI's first config snapshot uses `{ observe: false }`, but the default `core/doctor/final-config-validation` check calls `readConfigFileSnapshot()` with observation enabled.
