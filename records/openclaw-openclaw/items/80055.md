## Summary

Keep open: current main does not have the proposed doctor health-check contract or `doctor --lint` surface, and this draft still has a blocking correctness issue where the default lint path can observe/write config health state despite being documented as read-only.

## What This Changes

The PR adds a doctor health-check registry and contract, `openclaw doctor --lint` selectors and JSON output, structured repair validation, a `plugin-sdk/health` export, docs, and focused tests.

## Best Possible Solution

Choose the maintainer-approved diagnostics surface, then land it only after every lint check preserves no-observe config reads and the SDK/docs/tests match that boundary.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR head shows the default lint registry includes final config validation, and that check calls `readConfigFileSnapshot()` without disabling observation while config IO observes by default.

## Solution Assessment

No, not as-is. The health-check direction is coherent, but the patch needs the read-only lint fix and maintainer direction on whether this belongs under `doctor` or the standalone lint draft before it is a maintainable landing path.

## Review Findings

Overall confidence: 0.9
