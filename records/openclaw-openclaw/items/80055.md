Keep this PR open for maintainer review: it is a draft protected by the `maintainer` label, asks maintainers to choose the diagnostics/API shape, and the current patch still violates the documented read-only lint contract through the final config validation check.

The branch adds a structured doctor health-check contract, `openclaw doctor --lint`, scoped repair validation, a `plugin-sdk/health` export, docs, and focused tests.

## Best Possible Solution

Land only one maintainer-approved diagnostics surface, with all lint checks using a no-observe config read path and SDK/docs aligned to the chosen health-check boundary.

## Reproduction Assessment

Yes. Source inspection of the PR head shows default `doctor --lint` registers the final config validation check, and that check calls `readConfigFileSnapshot()` without `{ observe: false }` while config IO defaults observation on.

## Solution Assessment

No, not as-is. The health-check direction is coherent, but the patch must preserve the read-only lint invariant and still needs maintainer direction on doctor-integrated lint versus the standalone lint draft.
