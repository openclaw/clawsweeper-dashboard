## Summary

Keep open: current main and v2026.6.8 still show a native/shared Chat UI gap for Gateway managed image blocks, while browser Control UI has a separate authenticated fetch path for the same `/api/chat/media/outgoing/...` URLs.

## What This Changes

Reported regression where Mac native WebChat can copy but cannot open or download Gateway-managed `MEDIA:` images that browser Control UI handles.

## Best Possible Solution

Implement native managed-image support in shared OpenClawChatUI and the macOS transport: decode image URL fields, fetch with authenticated Gateway context, render the image, and provide Open/Download actions with live macOS proof.

## Reproduction Assessment

Yes at source level: Gateway emits managed image URL blocks and browser Control UI has authenticated handling, while native shared Chat UI does not decode or render those URL-backed image blocks. A live macOS app repro is still needed for runtime confidence.

## Solution Assessment

Yes in direction, but not yet automation-ready: the narrow maintainable fix belongs in shared native Chat UI plus macOS authenticated media fetching, not by weakening the Gateway media route.
