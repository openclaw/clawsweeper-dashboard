## Summary

Keep open: current main and v2026.6.9 still do not carry Gateway-managed image `url`/`openUrl` through the native SwiftUI chat model or renderer, and the linked merged macOS PR fixed a different WKWebView file-input path.

## What This Changes

The issue reports that native macOS WebChat image attachment Open and Download actions do nothing for Gateway-hosted `MEDIA:` images while the browser Control UI can open the same images.

## Best Possible Solution

Implement native managed-image support in shared OpenClawChatUI and the macOS transport boundary, carrying `url`/`openUrl` metadata and using an authenticated fetch/open/download flow with live macOS proof.

## Reproduction Assessment

Yes at source level, but not by a live macOS run: Gateway emits `type: "image"` blocks with relative `url`/`openUrl`, while native SwiftUI chat ignores those fields and only renders `file`/`attachment` rows.

## Solution Assessment

Yes directionally, but not automation-ready: the fix should live at the shared native Chat UI/macOS transport boundary and reuse Gateway auth rather than raw-opening relative paths.
