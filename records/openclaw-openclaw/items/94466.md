## Summary

Keep open: current main and v2026.6.8 still have Gateway managed-image blocks with relative `url`/`openUrl` fields, while the native SwiftUI chat model/render path does not preserve or render those image fields, so the macOS WebChat media-action regression is not implemented or superseded.

## What This Changes

The issue requests restoring native macOS WebChat Open/Download behavior for `MEDIA:` images served through Gateway managed media URLs.

## Best Possible Solution

Add native SwiftUI Chat UI support for managed `type: "image"` blocks, including authenticated preview/open/download/copy behavior for Gateway relative media URLs, without weakening the existing Gateway media auth contract.

## Reproduction Assessment

No live high-confidence packaged macOS reproduction was performed. Source inspection does show the native SwiftUI chat path lacks the URL-bearing managed-image fields and ignores `type: "image"` blocks while Gateway emits exactly those blocks.

## Solution Assessment

Yes in direction: the fix belongs in the native Chat UI/transport boundary that consumes Gateway managed media, not in relaxing the Gateway route or browser Control UI policy. The exact implementation should preserve owner/admin media auth and add focused Swift coverage.
