## Summary

Keep open: current main and v2026.6.8 still emit Gateway managed-image URL/openUrl blocks, but the native SwiftUI chat model and renderer do not carry or act on those image URL fields; the browser Control UI has a separate authenticated blob-fetch path.

## What This Changes

Issue reports native macOS WebChat Open/Download actions failing for Gateway-hosted MEDIA images while browser Control UI works.

## Best Possible Solution

Implement native managed-image support in shared OpenClawChatUI and the macOS transport boundary, with authenticated fetch/open/download handling and live macOS proof against /api/chat/media/outgoing/... URLs.

## Reproduction Assessment

Yes at source level: current main still emits managed image blocks with relative url/openUrl fields, while native SwiftUI chat does not decode or render URL-aware image actions. A live macOS smoke is still needed before marking it fully reproduced.

## Solution Assessment

Yes in direction, but not yet automation-ready: the maintainable fix belongs in the native shared Chat UI plus macOS transport/auth boundary, not in the Gateway server route alone.
