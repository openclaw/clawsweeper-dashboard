## Summary

Keep open: current main and latest release still leave the native SwiftUI WebChat without managed image URL fields or image Open/Download handling, while the browser Control UI has a separate auth-aware managed-image fetch path.

## What This Changes

The issue reports native macOS WebChat image attachment Open/Download actions doing nothing for Gateway-hosted MEDIA images while browser Control UI succeeds.

## Best Possible Solution

Add native managed-image support in shared OpenClawChatUI and the macOS transport boundary, with authenticated fetch/open/download handling and live macOS proof for `/api/chat/media/outgoing/...` images.

## Reproduction Assessment

Yes at source level: Gateway emits managed image blocks with relative `url`/`openUrl`, while the native SwiftUI chat model and renderer do not expose or handle those fields. A live macOS app reproduction is still missing for runtime confidence.

## Solution Assessment

Yes in direction, but not automation-ready: the maintainable fix should live in shared native ChatUI plus the macOS transport boundary and reuse Gateway auth instead of bypassing or raw-opening protected media URLs.
