## Summary

Keep open: current main and the latest release still do not carry Gateway managed-image `url`/`openUrl` fields through the native SwiftUI chat model or renderer, while browser Control UI has a separate authenticated blob-fetch path.

## What This Changes

The issue reports that native macOS WebChat cannot open or download Gateway-hosted MEDIA images while browser Control UI can.

## Best Possible Solution

Implement native WebChat managed-image support in shared OpenClawChatUI plus the macOS transport boundary, with authenticated fetch/open/download handling and live macOS proof against `/api/chat/media/outgoing/...`.

## Reproduction Assessment

Yes at source level: Gateway emits managed image blocks with relative media URLs, while the native shared Chat UI lacks URL-aware image fields or actions. A live macOS app reproduction is still needed for high-confidence runtime proof.

## Solution Assessment

Yes in direction, but not yet enough for automation: the narrow fix belongs in the native shared Chat UI/macOS transport path while preserving the owner/admin Gateway media-byte boundary.
