## Summary

Keep open: current main and v2026.6.9 still expose Gateway-managed image URLs for browser Control UI, but shared native Chat UI does not decode or render the managed image URL/openUrl fields needed for native macOS Open/Download actions.

## What This Changes

The issue requests native macOS WebChat Open and Download support for Gateway-managed MEDIA image URLs, matching the browser Control UI behavior.

## Best Possible Solution

Add authenticated managed-image rendering, Open, and Download handling at the shared native Chat UI/macOS transport boundary, then prove it with live macOS native WebChat media actions.

## Reproduction Assessment

Yes at source level: Gateway emits managed image blocks with relative `url`/`openUrl`, while the shared native Chat UI model and renderer do not decode or render those image action fields. A live macOS app repro is still missing.

## Solution Assessment

Unclear for implementation details: the right direction is a native authenticated media action path, but maintainers need to choose the Open/Download design so it does not bypass Gateway owner auth.
