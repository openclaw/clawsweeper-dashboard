## Summary

Keep open: current main and the latest release still expose Gateway managed-image `url`/`openUrl` blocks, but the native shared SwiftUI Chat UI does not decode or render those URL fields as image open/download actions. The merged adjacent macOS WKUIDelegate fix only covers embedded Control UI and Canvas file inputs, so it does not implement this native WebChat path.

## What This Changes

The issue reports that native macOS WebChat can copy Gateway-hosted MEDIA images but its Open and Download actions do not respond, while browser Control UI can open the same image URLs.

## Best Possible Solution

Implement native WebChat managed-image support in shared OpenClawChatUI plus the macOS transport boundary, with authenticated fetch/open/download handling for `/api/chat/media/outgoing/...` and live packaged macOS proof.

## Reproduction Assessment

Yes at source level: Gateway emits managed image blocks with relative media URLs, while the native shared Chat UI lacks URL-aware image fields or actions. A live macOS app reproduction is still needed for high-confidence runtime proof.

## Solution Assessment

Yes in direction, but not yet enough for automation: the narrow fix belongs in the native shared Chat UI/macOS transport path while preserving the owner-authenticated Gateway media boundary. Maintainers still need to choose the URL/action/auth shape before implementation.
