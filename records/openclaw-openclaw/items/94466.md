## Summary

Keep open: current main and the latest release still lack native SwiftUI WebChat handling for Gateway-managed image `url`/`openUrl`, while the browser Control UI has a separate authenticated blob fetch path. The merged macOS PR at https://github.com/openclaw/openclaw/pull/94612 fixed WKWebView file inputs, not native WebChat image Open/Download actions.

## What This Changes

Restore native macOS WebChat Open/Download behavior for Gateway-managed `MEDIA:` images that already work in browser Control UI.

## Best Possible Solution

Add authenticated native managed-image preview/Open/Download support in shared `OpenClawChatUI` and the macOS transport boundary, then prove it with a live macOS WebChat MEDIA-image smoke.

## Reproduction Assessment

Yes at source level: Gateway emits managed `type: "image"` blocks with `url`/`openUrl`, while native SwiftUI chat still does not decode those fields or render image actions. I did not run a live macOS app reproduction in this read-only review.

## Solution Assessment

Yes in direction, but it needs maintainer design: the fix should live at the shared native Chat UI/macOS transport boundary and reuse Gateway auth rather than raw-opening relative paths.
