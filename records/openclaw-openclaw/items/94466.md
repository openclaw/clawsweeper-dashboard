## Summary

Keep open: current main and v2026.6.8 still have a gap between Gateway-managed image blocks and the native SwiftUI chat renderer, so the reported native macOS WebChat media failure is not implemented or superseded.

## What This Changes

The issue reports that native macOS WebChat can copy MEDIA-delivered Gateway-hosted images but cannot open or download them, while browser Control UI works.

## Best Possible Solution

Implement native WebChat managed-image support in the shared OpenClawChatUI/macOS transport boundary, preserving Gateway auth rules and proving Open/Download against `/api/chat/media/outgoing/...`.

## Reproduction Assessment

Yes at source level: Gateway emits managed `type: "image"` blocks with relative media URLs, while the native shared Chat UI model/renderer lacks a corresponding URL-aware image action path. I did not run a live macOS app reproduction.

## Solution Assessment

Yes, the best fix appears to belong in shared native Chat UI plus macOS transport/auth handling, not in the server endpoint alone, because browser Control UI and Gateway media serving already have explicit managed-image paths.
