## Summary

Keep open: current main and v2026.6.9 still do not carry Gateway-managed image `url`/`openUrl` through the native SwiftUI WebChat model or renderer, while the browser Control UI has a separate authenticated managed-image fetch path. The merged macOS PR at https://github.com/openclaw/openclaw/pull/94612 fixed WKWebView file inputs, not native WebChat image Open/Download actions.

## What This Changes

The issue asks for native macOS WebChat Open/Download actions to work for Gateway-managed MEDIA images served from `/api/chat/media/outgoing/`, matching the browser Control UI behavior.

## Best Possible Solution

Implement native managed-image support in shared OpenClawChatUI and the macOS transport boundary, with authenticated fetch/open/download handling and live macOS proof against `/api/chat/media/outgoing/...`.

## Reproduction Assessment

Yes at source level: Gateway emits managed image blocks with relative `url`/`openUrl`, while the native SwiftUI chat model and renderer do not expose those fields or image actions. A live macOS reproduction is still missing for runtime confidence.

## Solution Assessment

Yes in direction, but not automation-ready: the fix should live at the shared native Chat UI/macOS transport boundary and reuse Gateway auth rather than bypassing auth or raw-opening relative paths.
