## Summary

Keep open: current main still emits managed image blocks with relative `url`/`openUrl`, but the shared native SwiftUI Chat UI model and renderer do not carry or act on those fields. The merged macOS work at https://github.com/openclaw/openclaw/pull/94612 fixed embedded Control UI file inputs, not native WebChat image Open/Download actions.

## What This Changes

The issue asks OpenClaw to make macOS native WebChat Open in Browser and Download work for Gateway-managed `MEDIA:` images served from `/api/chat/media/outgoing/...`.

## Best Possible Solution

Add native managed-image support at the shared OpenClawChatUI/macOS transport boundary, preserving Gateway owner auth and proving Open and Download in a real macOS app run.

## Reproduction Assessment

Yes at source level: Gateway emits managed `type: "image"` blocks with relative `url`/`openUrl`, while native SwiftUI Chat UI currently has no fields or actions for those URLs. A live macOS reproduction is still missing for runtime confidence.

## Solution Assessment

Yes in direction, but the exact implementation needs maintainer judgment: the fix should be native and auth-aware, not a raw relative URL handoff that loses Gateway owner authorization.
