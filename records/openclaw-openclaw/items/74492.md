## Summary

The patch is focused and appears to address a real current-main gap, but this PR has the protected `maintainer` label and changes a security-sensitive subprocess environment boundary, so it should be reviewed and validated by a maintainer rather than closed or routed to automated cleanup.

## What This Changes

The PR adds `CLOUDSDK_PYTHON` to the workspace dotenv blocklist and changes Gmail gcloud subprocesses to always pass a resolved Python interpreter path or an undefined override, with unit coverage for both cases.

## Best Possible Solution

Have a maintainer review and validate the narrow hardening, then land it if CI passes: workspace dotenv should block `CLOUDSDK_PYTHON`, and Gmail gcloud subprocesses should receive only a controlled interpreter path or have the variable stripped.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `CLOUDSDK_PYTHON` before the Gmail gcloud helper runs on current main, and `gcloudEnv()` returns `undefined`; a workspace `.env` can also currently supply the key because the blocklist omits it.

## Solution Assessment

Yes, pending maintainer validation. The PR uses the narrowest maintainable boundary I found: block the untrusted workspace dotenv key and pass an explicit child-env override, relying on the existing `resolveCommandEnv()` undefined-stripping contract.
