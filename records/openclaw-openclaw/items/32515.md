## What This Changes

The issue requests an Anthropic onboarding flow where `openclaw onboard` opens a browser, signs the user in with Anthropic, detects account type, and configures credentials automatically.

## Best Possible Solution

Ship an optional Anthropic provider-owned browser auth method only after the supported Anthropic auth contract and callback/token-storage security model are clear, while preserving API-key, Claude CLI, and setup-token routes.

## Risks / Open Questions

- A browser sign-in flow would introduce callback state, token storage, refresh, profile/account detection, and secret-handling behavior that needs explicit OAuth security review before implementation.
- Anthropic’s current docs distinguish direct API auth from Workload Identity Federation and local `ant auth login` browser OAuth, so OpenClaw needs a product decision about whether to integrate `ant`/Claude CLI, a WIF-style service-account path, or a new provider-owned user OAuth flow. ([platform.claude.com](https://platform.claude.com/docs/en/manage-claude/authentication?utm_source=openai))
- Replacing the whole Anthropic setup flow could regress users who rely on API keys, Claude CLI reuse, setup-token profiles, or long-lived gateway hosts.
