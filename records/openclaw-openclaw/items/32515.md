## Summary

Keep open: current main still has improved Anthropic auth through Claude CLI, setup-token, and API-key paths, but it does not provide the requested one-click Anthropic browser OAuth flow or automatic account-type setup. The related closed OAuth regression was about runtime use of existing tokens, not this onboarding feature.

## What This Changes

The issue requests an Anthropic onboarding flow that opens a browser, signs in with Anthropic, detects account type, and configures credentials automatically.

## Best Possible Solution

Confirm Anthropic's supported auth contract, then add any approved sign-in as an Anthropic-provider-owned auth method with docs while preserving API-key, Claude CLI, and setup-token compatibility.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source inspection on current main shows no Anthropic browser OAuth flow to reproduce, only Claude CLI, setup-token, and API-key setup paths.

## Solution Assessment

Unclear as a direct implementation request because the full browser flow depends on Anthropic-supported endpoint, scope, refresh, and account-type semantics. The maintainable direction is provider-owned Anthropic auth work, with pasted-token prefix detection as a narrower follow-up only if maintainers want it.
