## Summary

Keep open: current main and the latest release still expose Anthropic Claude CLI, setup-token, and API-key auth paths, but not the requested one-click browser OAuth/account-type onboarding flow. The related closed OAuth regression only covered runtime handling of existing tokens, so it does not supersede this feature request.

## What This Changes

The issue requests a one-click Anthropic browser OAuth onboarding flow that detects account type and configures authentication automatically.

## Best Possible Solution

Confirm Anthropic’s supported auth contract, then implement any approved sign-in as an Anthropic-provider-owned auth method with docs and focused tests while preserving existing API-key, Claude CLI, setup-token, and token-profile paths.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing behavior report. Source inspection on current main and the latest release shows no Anthropic browser OAuth onboarding path to reproduce, only Claude CLI, setup-token, and API-key setup paths.

## Solution Assessment

Unclear as a direct implementation request because the full browser flow depends on Anthropic-supported endpoint, scope, refresh, and account-type semantics. The maintainable direction is provider-owned Anthropic auth work, with pasted-token prefix detection as a narrower follow-up only if maintainers choose that UX.
