## Summary

Keep open: current main still lacks the requested one-click Anthropic browser OAuth/account-type onboarding; the code and docs expose Claude CLI, setup-token, and API-key paths, while the related closed OAuth regression only covered runtime handling of existing tokens.

## What This Changes

The issue requests a one-click Anthropic sign-in flow that opens a browser, detects Pro/API account type or token shape, and configures OpenClaw automatically.

## Best Possible Solution

Confirm Anthropic’s supported auth contract, then implement any approved sign-in as an Anthropic-provider-owned auth method with docs and focused tests while preserving API-key, Claude CLI, and setup-token compatibility.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source inspection on current main shows no Anthropic browser OAuth flow to reproduce, only Claude CLI, setup-token, and API-key setup paths.

## Solution Assessment

Unclear as a direct implementation request because the full browser flow depends on Anthropic-supported endpoint, scope, refresh, and account-type semantics. The maintainable direction is provider-owned Anthropic auth work, with pasted-token prefix detection as a narrower follow-up only if maintainers choose that UX.
