## Summary

Keep open: current main and v2026.6.8 still expose explicit Anthropic Claude CLI, setup-token, and API-key onboarding choices, not the requested one-click browser OAuth flow. The linked open PR overlaps with first-run friction, but it is conflicting, proof/security-blocked, and only addresses Claude subscription CLI onboarding rather than the full direct Anthropic OAuth product decision.

## What This Changes

Request a one-click Anthropic browser OAuth onboarding flow that detects account/credential type and configures OpenClaw without manual API-key or token setup.

## Best Possible Solution

Define an Anthropic-owned onboarding contract that keeps API-key, setup-token, Claude CLI, and any browser OAuth scope/token/billing choices explicit, then land a focused provider change only after product and security approval.

## Reproduction Assessment

Not applicable: this is a feature request for a new authentication experience, not a reproducible bug in an existing contract. Source inspection confirms the requested browser OAuth onboarding path is not exposed as a current provider auth choice.

## Solution Assessment

No: replacing the entire auth setup with automatic browser OAuth is not the best path until maintainers approve the auth, billing, and token-storage contract. A safer solution keeps the existing explicit choices and adds any Anthropic OAuth path as an approved, well-scoped provider-owned flow.
