## Summary

Keep open: current main still has the reported retry gap. Restart continuations are durably queued and immediately drained, but the only automatic session-delivery recovery is a one-shot startup pass, so a transient send failure after that pass remains pending until another recovery trigger.

## What This Changes

The issue asks for restart-sentinel continuation delivery to retry automatically after transient Telegram sendMessage failures instead of staying pending in the session-delivery queue.

## Best Possible Solution

Add a generic session-delivery backoff retry scheduler for restart-continuation entries, preserving idempotency and max-retry movement to failed/ while keeping optional diagnostics as follow-up scope.

## Reproduction Assessment

Yes. A focused test can enqueue a restart continuation, make the first routed delivery throw, verify retryCount/lastError remain pending, advance past backoff, and show current main performs no second attempt without another startup recovery call.

## Solution Assessment

Yes. The narrow maintainable fix is in the core session-delivery recovery scheduling layer; Telegram-specific retry logic would address the wrong boundary.
