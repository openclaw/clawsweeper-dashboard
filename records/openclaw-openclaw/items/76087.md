## Summary

Keep open: current main and latest release still run restart-continuation session-delivery recovery as a one-shot startup timer, and the linked closing PR at https://github.com/openclaw/openclaw/pull/90490 is open rather than merged.

## What This Changes

The issue reports that restart-sentinel continuation delivery can remain pending after a transient Telegram sendMessage failure because session-delivery recovery is only run around gateway startup.

## Best Possible Solution

Merge a narrow lifecycle-owned retry path that reuses the existing session-delivery backoff, max-retry, and idempotency contracts, then close the issue with the merged commit or shipped release proof.

## Reproduction Assessment

Yes at source level: current main queues and immediately drains restart continuations, but only schedules one delayed recovery pass, so a failed retry can remain pending until another startup. I did not run a live Telegram outage reproduction in this read-only review.

## Solution Assessment

Yes, the right fix shape is to add a lifecycle-owned retry scheduler or one-shot wake that calls the existing recovery contract after backoff. The issue itself should stay open while the linked closing PR is reviewed and either landed or replaced.
