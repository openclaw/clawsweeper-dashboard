## Summary

Keep open: current main still has the reported retry gap. Restart continuations are durably queued and immediately drained, but a failed drain only records retry metadata; the normal runtime recovery is a one-shot startup pass with a cutoff that does not cover newly enqueued restart-continuation entries.

## What This Changes

The issue requests durable backoff retry/recovery for restart-sentinel continuation entries left in `session-delivery-queue` after transient Telegram send failures.

## Best Possible Solution

Add a generic session-delivery backoff retry loop or one-shot retry scheduler for restart-continuation entries, preserving idempotency and moving exhausted entries to `failed/` with clear diagnostics.

## Reproduction Assessment

Yes. A focused test can enqueue a restart continuation, make the first routed delivery throw, verify `retryCount`/`lastError` remains pending, advance past the backoff, and show current main has no automatic second attempt without another gateway startup.

## Solution Assessment

Yes. The narrow maintainable fix is core-generic session-delivery retry scheduling; Telegram-specific retry logic or broad outbound queue refactoring would solve the wrong layer.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76087. Current main persists restart-sentinel continuation entries in `session-delivery-queue` and drains them immediately; if `deliverQueuedSessionDelivery` throws after a transient channel send failure, `failSessionDelivery` leaves retry metadata but normal runtime has only a startup-only recovery pass. Add generic session-delivery recovery scheduling so pending restart-continuation entries are retried after `isSessionDeliveryEligibleForRetry` backoff up to the existing max retry limit and exhausted entries move to `failed/`. Prefer a lightweight managed timer loop in gateway scheduled services or a one-shot retry scheduled after a failed continuation drain. Keep the fix core-generic and channel-agnostic: no Telegram-specific ids, defaults, or transport policy, and avoid broad outbound-delivery refactors. Add focused tests proving a failed restart continuation is retried after backoff without another gateway startup and that max-retry entries are handled. Add a `CHANGELOG.md` fix entry because this is user-facing delivery reliability.
