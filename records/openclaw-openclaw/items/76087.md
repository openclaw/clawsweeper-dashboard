## What This Changes

The issue asks for restart-sentinel session-delivery continuations to retry after transient Telegram sendMessage failures without needing another gateway restart.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90490 after its lifecycle concerns are resolved, or replace it with a narrow gateway-owned retry fix that preserves backoff, max-retry, idempotency, and shutdown ordering.

## Reproduction Assessment

Yes, at source level. Current main persists failed session-delivery entries with retry metadata and only schedules one startup recovery pass, so a transient post-drain failure can remain pending without a later wake.

## Solution Assessment

Yes directionally: retrying through the existing session-delivery recovery/backoff path is the right owner boundary. The linked PR still needs the maintainer-requested lifecycle and durable-handoff concerns resolved before it is the best mergeable fix.
