Keep open: current main still has the reported retry gap for restart-continuation session deliveries after a transient transport failure.

## What This Changes

The issue requests durable backoff retry scheduling for restart-sentinel continuation entries that remain in the session-delivery queue after a failed Telegram send.

## Best Possible Solution

Add a core-generic retry scheduler or periodic recovery loop for restart-continuation session-delivery entries, respecting existing backoff, max retries, and idempotency.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76087. Current main durably enqueues restart-sentinel continuations but only performs an immediate drain plus one startup recovery pass, so transient send failures can leave entries pending in `session-delivery-queue/` until the next gateway restart. Add a core-generic retry mechanism for restart-continuation session deliveries: either a lightweight periodic recovery loop or a one-shot retry scheduled from the failed drain/backoff deadline. Respect `isSessionDeliveryEligibleForRetry`, the existing max retry behavior, `entriesInProgress`/drain guards, and idempotency keys so duplicate continuation delivery is avoided. Keep the fix in gateway/session-delivery code, not Telegram-specific transport code. Add focused tests proving a failed restart continuation is retried automatically after backoff without another gateway startup, and that max-retry entries still move to failed. Add a user-facing changelog entry because this fixes gateway restart reliability.
