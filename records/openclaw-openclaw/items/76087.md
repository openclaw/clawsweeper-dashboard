## Label Justifications

- `P2`: This is a normal-priority gateway reliability bug with a narrow restart-continuation delivery surface.
- `impact:message-loss`: A transient send failure can leave the post-restart continuation pending so the user never receives the expected channel message.

## Summary

Keep open: current main and the latest release still recover restart-continuation session-delivery entries with a one-shot startup timer only, so a transient delivery failure can leave the continuation pending until another gateway startup.

## What This Changes

The issue asks for restart-continuation session-delivery entries to keep retrying after transient Telegram sendMessage failures instead of remaining pending indefinitely.

## Best Possible Solution

Add a lifecycle-owned bounded retry path for restart-continuation session-delivery entries that reuses the existing backoff and max-retry queue contract without adding a new config surface.

## Reproduction Assessment

Yes at source level: the immediate drain can persist a failed queue entry, while current main only schedules one delayed startup recovery pass. No live Telegram repro was run because this review was read-only.

## Solution Assessment

Yes: the narrowest maintainable solution is a bounded scheduler or one-shot retry that calls the existing session-delivery recovery after backoff and honors the current max-retry behavior.
