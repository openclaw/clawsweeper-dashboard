## Summary

Keep open: current main still has the reported restart-continuation retry gap. The continuation is queued and immediately drained, but gateway scheduled services only trigger session-delivery recovery once after startup, so a transient send failure can stay pending until another explicit recovery/startup path runs.

## What This Changes

The issue requests durable automatic retry for restart-sentinel continuation entries after transient Telegram sendMessage failures.

## Best Possible Solution

Add a generic session-delivery retry scheduler for restart-continuation entries that respects existing backoff, idempotency, max retries, and failed-entry handling without adding Telegram-specific retry policy.

## Reproduction Assessment

Yes, source-reproducible. Enqueue a restart continuation, make immediate routed delivery throw, observe retry metadata persisted, advance past backoff, and current main has no automatic second recovery call without another startup or explicit recovery trigger.

## Solution Assessment

Yes. The narrow maintainable fix belongs in core session-delivery recovery scheduling; Telegram-specific retry logic would fix the wrong layer and leave other restart-continuation routes with the same durability gap.
