## Summary

Keep open: current main and v2026.6.6 still perform only one delayed startup recovery for restart-continuation session deliveries, and the linked fix PR is open with maintainer changes requested rather than merged.

## What This Changes

The issue asks for durable retry of restart-sentinel session-delivery entries after transient Telegram sendMessage failures so continuations are delivered without another gateway restart.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/90490, or a narrow replacement, with lifecycle-safe restart-continuation retry proof and close this issue only after that fix merges.

## Reproduction Assessment

Yes, at source level. Current main can persist a failed restart-continuation queue entry with retry metadata, but the gateway only schedules one delayed startup recovery pass, so no normal-operation retry is guaranteed after backoff.

## Solution Assessment

Yes for the issue direction, but not yet for the linked PR as submitted. The right end state is a lifecycle-owned retry that reuses the existing queue backoff/max-retry contract; the open PR needs the maintainer-requested shutdown and durable-handoff safety before it is landable.
