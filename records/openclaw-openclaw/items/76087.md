## Summary

Keep open: current main still has the reported retry gap. Restart continuations are durably queued and immediately drained, but session-delivery recovery is only scheduled as a one-shot startup pass, so a transient send failure can remain pending until another explicit recovery trigger.

## What This Changes

The issue requests durable retry scheduling for restart-continuation entries after transient Telegram sendMessage failures leave session-delivery queue items pending.

## Best Possible Solution

Add a generic session-delivery retry scheduler for restart-continuation entries that respects existing backoff, idempotency, max retries, and failed-entry handling without adding Telegram-specific policy.

## Reproduction Assessment

Yes, source-reproducible. The path is to enqueue a restart continuation, make the immediate routed delivery throw, observe retry metadata persisted, advance past backoff, and see that current main has no automatic second recovery call without another startup/recovery trigger.

## Solution Assessment

Yes. The narrow maintainable fix belongs in core session-delivery recovery scheduling; Telegram-specific retry logic would fix the wrong layer and leave other restart-continuation routes with the same durability gap.

## Repair Work Prompt

Fix the restart-sentinel session-delivery retry gap from https://github.com/openclaw/openclaw/issues/76087. On current main, a restart continuation is queued and immediately drained; if routed delivery throws, the entry remains pending with retry metadata but no automatic later recovery runs for that newly failed entry. Add a scoped, generic retry scheduler for restart-continuation session-delivery entries that respects existing backoff, idempotency, max retry handling, and failed-entry movement. Keep the fix in core session-delivery/restart-continuation scheduling; do not add Telegram-specific retry policy, config knobs, or status/diagnostic UI unless required by the narrow fix. Add regression coverage proving a failed immediate restart-continuation drain is retried automatically after backoff without requiring another gateway startup, and that duplicate delivery is avoided when drains overlap. Add a user-facing changelog fix entry if behavior changes. Coordinate conceptually with #70780, #71429, #75539, and adjacent open durable-delivery PRs, but do not broaden the PR to solve unrelated outbound queue or IPv6 transport issues.
