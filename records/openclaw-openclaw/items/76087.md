## Summary

Keep open: current main still has the reported retry gap. Restart continuations are durably queued and immediately drained, but failed delivery only records retry metadata; the runtime recovery path is still a one-shot startup pass with a cutoff, not an ongoing backoff retry scheduler.

## What This Changes

The issue requests durable retry recovery for restart-sentinel continuation entries after transient Telegram `sendMessage` failures.

## Best Possible Solution

Add a core/gateway session-delivery retry scheduler for restart-continuation entries that respects existing backoff, max retries, idempotency, and failed-entry handling; treat status visibility as a separate follow-up unless it falls out naturally.

## Reproduction Assessment

Yes. A focused fake-timer test can enqueue a restart continuation, make the first routed delivery throw, confirm `retryCount` and `lastError` remain pending, advance past the first backoff, and show current main makes no automatic second attempt without another gateway startup.

## Solution Assessment

Yes. The best fix is a generic session-delivery recovery loop or one-shot scheduler in the restart-continuation/runtime layer; Telegram-specific retry logic would not cover the core queue gap and broad outbound-queue refactoring is not necessary for this report.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76087. Current main queues restart-sentinel continuations in `src/gateway/server-restart-sentinel.ts` and drains once immediately; if routed delivery throws, the entry remains in `session-delivery-queue/` with retry metadata and no normal-runtime follow-up retry. Add a small core/gateway retry scheduler for restart-continuation session deliveries, either periodic or one-shot from failed drain/startup recovery. It must respect `isSessionDeliveryEligibleForRetry`, the existing max retry limit, `drainInProgress`/`entriesInProgress`, idempotency keys/message IDs, and move exhausted entries to `failed/`. Avoid Telegram-specific retry logic and avoid broad outbound delivery queue refactors unless a tiny shared helper is clearly needed. Add focused tests proving a failed immediate continuation delivery is retried automatically after backoff without another gateway startup, entries newer than the startup cutoff are not permanently excluded from the runtime retry path, and overlapping immediate drain/recovery does not duplicate delivery.
