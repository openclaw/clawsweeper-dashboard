## Summary

Keep open: current main still has the reported retry gap. Restart continuations are durably queued and immediately drained, but session-delivery recovery is only scheduled as a one-shot startup pass, so a transient send failure can remain pending until another explicit recovery trigger.

## What This Changes

The issue requests durable retry scheduling for restart-sentinel session-delivery entries after transient Telegram sendMessage failures.

## Best Possible Solution

Add a generic session-delivery retry scheduler for restart-continuation entries that respects existing backoff, idempotency, max retries, and failed-entry handling without adding Telegram-specific retry policy.

## Reproduction Assessment

Yes, source-reproducible. The path is to enqueue a restart continuation, make the immediate routed delivery throw, observe retry metadata persisted, advance past backoff, and see that current main has no automatic second recovery call without another startup/recovery trigger.

## Solution Assessment

Yes. The narrow maintainable fix belongs in core session-delivery recovery scheduling; Telegram-specific retry logic would fix the wrong layer and leave other restart-continuation routes with the same durability gap.
