## Summary

Keep open: current main and the latest release still use one startup-only restart-continuation session-delivery recovery pass, and the active closing PR remains open with maintainer changes requested.

## What This Changes

The issue reports that restart-sentinel session-delivery entries can remain pending after a transient Telegram sendMessage failure because recovery only runs once at startup.

## Best Possible Solution

Resolve the lifecycle concerns on https://github.com/openclaw/openclaw/pull/90490 or replace it with a narrow gateway-owned recovery fix that retries queued restart continuations using the existing backoff, max-retry, and idempotency contracts.

## Reproduction Assessment

Yes, at source level: current main persists failed session-delivery entries with retry metadata and only schedules one startup recovery timer. I did not run a live Telegram failure in this read-only review.

## Solution Assessment

Yes, directionally: a gateway lifecycle-owned retry path that reuses the session-delivery backoff contract is the right layer. The active PR still needs the maintainer-requested shutdown and durable-handoff concerns resolved before it is the best mergeable patch.
