## Summary

Keep open: current main and the latest release still run restart-continuation session-delivery recovery as a one-shot startup timer. Failed delivery attempts persist retry metadata and backoff state, but no scheduler invokes recovery again during normal gateway operation after the immediate drain fails.

## What This Changes

Retry pending restart-sentinel session-delivery entries after transient Telegram sendMessage failures instead of leaving them pending until another gateway startup.

## Best Possible Solution

Add a generic session-delivery retry scheduler for restart-continuation entries that respects existing backoff, idempotency, max retries, and failed-entry handling without adding Telegram-specific retry policy.

## Reproduction Assessment

Yes, at source level. Enqueue a restart continuation, make the immediate routed delivery throw, observe retry metadata persisted, then advance past backoff; current main has no automatic second recovery call without another startup or explicit helper invocation.

## Solution Assessment

Yes. The narrow maintainable fix belongs in core session-delivery recovery scheduling, while Telegram-specific send retry logic would leave other restart-continuation routes with the same durability gap.
