## Summary

Keep open: current main still has the reported retry gap for restart-continuation session-delivery entries. The queue records transient failures and backoff state, but normal gateway runtime only wires a one-shot startup recovery, so a failed immediate drain can remain pending until another explicit recovery trigger.

## What This Changes

Restart continuations should retry automatically after transient session-delivery failures instead of staying pending until another gateway startup.

## Best Possible Solution

Add a generic session-delivery retry scheduler or one-shot reschedule path for restart-continuation entries that respects existing backoff, max retries, permanent-failure handling, and idempotency; keep diagnostics as follow-up scope unless needed for regression coverage.

## Reproduction Assessment

Yes, from source inspection. A focused current-main test can enqueue a restart continuation, make the immediate routed delivery throw, advance past backoff, and observe that no second attempt occurs unless recovery is called again.

## Solution Assessment

Yes. The narrow fix belongs in the generic session-delivery recovery scheduling layer; Telegram-specific retry handling would be the wrong boundary for a durable restart-continuation queue.
