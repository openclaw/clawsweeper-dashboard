## Summary

Keep open: current main and the latest release still use a one-shot startup recovery for restart-continuation session deliveries, and the open linked PR at https://github.com/openclaw/openclaw/pull/90490 is the active fix path for this issue.

## What This Changes

The issue reports that restart-sentinel continuation delivery can remain pending after a transient Telegram sendMessage failure because no periodic recovery runs after the startup drain.

## Best Possible Solution

Keep this issue open and review or land the linked PR if maintainers accept its lifecycle-owned periodic retry loop and normal checks remain green.

## Reproduction Assessment

Yes at source level: current main schedules one delayed startup recovery, while failed session-delivery entries remain pending with retry metadata and need a later recovery pass. I did not run a live Telegram outage reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is to reuse the existing session-delivery recovery, backoff, max-retry, and idempotency contracts from the gateway lifecycle owner. The linked PR appears to follow that shape, so opening another fix lane would duplicate the active work.
