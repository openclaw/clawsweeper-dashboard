## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76087. Current main durably queues restart-sentinel continuations and immediately drains the queued entry, but if that delivery fails transiently the JSON remains in `session-delivery-queue/` with retry metadata and no automatic follow-up retry until another gateway startup. Implement a generic session-delivery retry scheduler for restart-continuation entries, not Telegram-specific retry logic. Prefer a lightweight gateway runtime loop or one-shot timer that respects `isSessionDeliveryEligibleForRetry`, `MAX_SESSION_DELIVERY_RETRIES`, existing `entriesInProgress`/drain-key protections, and idempotency keys. Avoid broad outbound queue refactors and do not change plugin APIs unless strictly necessary. Add focused regression tests showing: first delivery failure persists retry metadata, automatic retry happens after backoff without another startup, exhausted entries move to `failed/`, and overlapping immediate drain/recovery cannot double-deliver. Add a user-facing `CHANGELOG.md` fix entry if runtime behavior changes. Be aware open PR #73812 touches restart-continuation delivery visibility/deferral but does not solve this scheduler gap; rebase or adjust around it if it lands first.
