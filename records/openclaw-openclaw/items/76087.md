## Summary

Keep open: current main and the latest release still schedule restart-continuation session-delivery recovery as a single startup timer, while the active closing PR is open and unmerged.

## What This Changes

Restart-sentinel continuation delivery should retry pending session-delivery entries after a transient Telegram sendMessage failure without requiring another gateway restart.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/90490 so restart-continuation session-delivery entries get lifecycle-owned retry recovery using the existing backoff, max-retry, and idempotency contract.

## Reproduction Assessment

Yes at source level: current main can persist a failed queued session-delivery entry and only schedules a single 1,250 ms recovery pass. I did not run a live Telegram failure in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is a lifecycle-owned retry path that reuses the existing session-delivery recovery contract rather than adding Telegram-specific delivery logic.
