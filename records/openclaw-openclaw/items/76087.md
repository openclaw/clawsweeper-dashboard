## Summary

Keep open: current main and the latest release still recover restart-continuation session-delivery entries with a one-shot startup timer only. Failed entries are persisted with retry/backoff metadata, but no normal-operation scheduler wakes them again after the startup pass or immediate drain fails.

## What This Changes

The issue reports that a transient Telegram `sendMessage` failure can leave a restart-sentinel continuation queued without another automatic retry until the gateway restarts.

## Best Possible Solution

Add a lifecycle-owned periodic or one-shot retry wake for restart-continuation session-delivery entries that respects existing backoff, max retries, shutdown, and startup cutoff semantics without adding Telegram-specific policy.

## Reproduction Assessment

Yes from source inspection, though not from a live Telegram run: force `deliverQueuedSessionDelivery` to fail, observe retry metadata persisted, then note no scheduler invokes recovery again after the startup one-shot and immediate drain complete.

## Solution Assessment

Yes: the narrow maintainable fix is to schedule existing session-delivery recovery after backoff expiry or on a bounded interval, using the existing queue/backoff/max-retry contract rather than adding channel-specific retry behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76087. Current `main` queues restart-sentinel continuations and has backoff-aware session-delivery recovery, but `activateGatewayScheduledServices` only invokes restart-continuation recovery once after startup. Add a lifecycle-owned retry mechanism so failed restart-continuation session-delivery entries are retried after their existing backoff expires during normal gateway operation, up to the existing max retry limit, without hardcoding Telegram or adding config. Preserve startup cutoff semantics for crash recovery; if the fix uses a periodic loop, make it shutdown-safe and bounded, and if it uses one-shot timers, reschedule after failed drain/recovery. Add regression tests with fake timers for transient failure followed by retry success and max-retry/failed behavior where appropriate. Do not edit `CHANGELOG.md`.
