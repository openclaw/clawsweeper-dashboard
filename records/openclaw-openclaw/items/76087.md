## Summary

Keep open: current main still has the reported retry gap. Restart continuations are durably queued and immediately drained, but session-delivery recovery is only scheduled as a one-shot startup pass, so a transient send failure can remain pending until another gateway startup or manual recovery trigger.

## What This Changes

The issue asks OpenClaw to automatically retry restart-sentinel session-delivery continuations after transient Telegram sendMessage failures without requiring another gateway restart.

## Best Possible Solution

Add a channel-agnostic session-delivery retry scheduler for restart-continuation entries that respects existing backoff, idempotency, max retries, and failed-entry handling.

## Reproduction Assessment

Yes, source-reproducible. Enqueue a restart continuation, make the immediate routed delivery throw, observe retry metadata persisted, advance past backoff, and current main has no automatic second recovery call without another startup or explicit recovery trigger.

## Solution Assessment

Yes. The narrow maintainable fix belongs in generic session-delivery recovery scheduling; Telegram-specific retry logic would fix the wrong layer and leave other restart-continuation routes with the same durability gap.

Restart-sentinel continuation delivery can fail during the immediate session-delivery drain, persist retry metadata, and then remain pending until another gateway startup because scheduled session-delivery recovery is a one-shot startup pass. Implement a generic session-delivery retry scheduler or one-shot reschedule path for restart-continuation entries that retries after the existing backoff and moves entries to failed after the existing max retry limit.
