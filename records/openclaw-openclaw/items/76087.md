## Summary

Keep open: current main still has the reported retry gap. Restart continuations are queued and immediately drained, but the only automatic session-delivery recovery is a one-shot startup pass, so a transient send failure can leave the entry pending until another recovery trigger.

## What This Changes

Restart-sentinel continuation delivery should retry pending session-delivery queue entries after transient Telegram send failures without requiring another gateway startup.

## Best Possible Solution

Add a generic session-delivery backoff retry scheduler for restart-continuation entries, preserving idempotency and max-retry movement to failed/ while keeping diagnostics as follow-up scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enqueue a restart continuation, make the first routed delivery throw, verify retry metadata remains pending, then advance past backoff and observe no second attempt unless recovery is invoked again.

## Solution Assessment

Yes. The narrow maintainable fix is in the core session-delivery recovery scheduling layer; Telegram-specific retry logic would address the wrong boundary.
