## Summary

Keep open: current main still has the reported retry gap. Restart continuations are queued and drained once, but a transient delivery failure only records retry metadata; normal gateway runtime starts a single delayed recovery pass rather than a continuing backoff retry path.

## What This Changes

The issue requests durable automatic retry for restart-sentinel session-delivery queue entries after transient Telegram `sendMessage` failures.

## Best Possible Solution

Add a generic session-delivery retry scheduler or one-shot backoff rescheduler for restart-continuation entries, preserving idempotency and moving exhausted entries to `failed/` with clear diagnostics.

## Reproduction Assessment

Yes. A focused test can enqueue a restart continuation, make the first routed delivery throw, verify `retryCount` and `lastError` remain pending, advance beyond backoff, and show current main does not automatically make a second attempt without another gateway startup.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the core session-delivery recovery path; Telegram-specific retries or a broad outbound queue rewrite would solve the wrong layer for this report.
