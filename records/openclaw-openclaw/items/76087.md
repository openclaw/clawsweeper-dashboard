## Summary

Keep open: current main and the latest release still run restart-continuation session-delivery recovery as a one-shot startup timer, and the open linked fix PR is the right path to review instead of closing this issue now.

## What This Changes

The issue reports restart-sentinel session-delivery entries remaining pending after a transient Telegram sendMessage failure because no periodic retry wakes the queue after startup.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90490 with a lifecycle-safe periodic/backoff retry for restart-continuation session deliveries, then close this issue after the fix is merged.

## Reproduction Assessment

Yes, at source level: current main persists failed session-delivery entries with retry metadata and backoff, but only schedules restart-continuation recovery once after gateway startup. I did not run a live Telegram failure scenario in this read-only review.

## Solution Assessment

Yes, the best path is to keep this issue open until https://github.com/openclaw/openclaw/pull/90490 either lands or is replaced. Current main still has only the one-shot startup recovery, while that PR adds the periodic retry and stop handle but still has changes requested.
