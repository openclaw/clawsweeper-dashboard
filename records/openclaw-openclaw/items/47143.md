## What This Changes

Fix memory-flush deduplication so a persisted never-flushed 0/0 compaction-counter state does not suppress the first flush.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/83015; if rejected, keep the replacement fix narrowly focused on disambiguating real cycle-0 flushes from never-flushed 0/0 rows with regression tests.

## Reproduction Assessment

Yes, source-level. On current main, `shouldRunMemoryFlush` reaches a helper that treats `compactionCount ?? 0` equal to `memoryFlushCompactionCount: 0` as already flushed, and the adjacent unit test still expects that behavior.

## Solution Assessment

The best fix is to disambiguate never-flushed 0/0 rows from real cycle-0 flushes, likely using `memoryFlushAt`, while preserving once-per-cycle dedupe. The issue body's Option A alone is not sufficient because the current helper already returns false only for non-number flush counts.
