## Summary

Keep open: current main and the latest release still treat the reported 0/0 memory-flush counter state as already flushed, and the open linked fix PR is the active implementation path.

## What This Changes

The issue reports that `memoryFlush` is skipped for persisted sessions where `compactionCount` and `memoryFlushCompactionCount` are both `0`, even though no flush has run.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83015 or a narrow equivalent that distinguishes real cycle-zero flushes from legacy or never-flushed 0/0 rows, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Current main's helper returns true for a numeric `0` matching the defaulted compaction count of `0`, and the runtime uses that helper to skip memory flush; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The linked PR's `memoryFlushAt` discriminator is the narrowest visible fix because it preserves real cycle-zero flush deduplication while letting legacy or never-flushed 0/0 rows proceed.
