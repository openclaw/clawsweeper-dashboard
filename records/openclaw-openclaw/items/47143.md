## Summary

Keep open. Current main and the latest release still skip the reported persisted 0/0 memory-flush counter state, and the persistence path shows the repair needs to distinguish legacy/no-flush zero rows from real cycle-0 flush records.

## What This Changes

The issue reports that memoryFlush is skipped when an above-threshold session has both compactionCount and memoryFlushCompactionCount set to 0.

## Best Possible Solution

Use memoryFlushAt or an explicit session-state sentinel in the helper and persistence path so legacy/no-flush 0/0 rows can flush while actual cycle-0 flushes remain deduped once per cycle.

## Reproduction Assessment

Yes, at source level. An above-threshold entry with compactionCount: 0 and memoryFlushCompactionCount: 0 reaches hasAlreadyFlushedForCurrentCompaction, returns true, and makes shouldRunMemoryFlush skip the flush.

## Solution Assessment

No patch is currently under review. The simple compactionCount === 0 special case from prior closed attempts is not the best fix because it would lose valid cycle-0 dedup; the repair should distinguish recorded flush state from legacy/no-flush state.

The narrow bug is that shouldRunMemoryFlush treats an above-threshold persisted session entry with compactionCount: 0 and memoryFlushCompactionCount: 0 as already flushed, even when there is no evidence a memory flush has actually run. Preserve the once-per-compaction-cycle duplicate guard and do not implement the broader pre-request context-budget feature work. Add focused regression tests for reported 0/0 without memoryFlushAt, real cycle-0 flush with memoryFlushAt, missing marker, and post-compaction counts.
