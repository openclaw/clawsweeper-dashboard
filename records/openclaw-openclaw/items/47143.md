## Summary

Keep open. Current main and the latest release still treat the reported zero-counter memory-flush state as already flushed, and https://github.com/openclaw/openclaw/pull/83015 is an open closing PR for the narrow fix.

## What This Changes

The issue reports that `memoryFlush` is skipped for never-compacted sessions when `compactionCount` and `memoryFlushCompactionCount` are both `0`.

## Best Possible Solution

Review, validate, and land https://github.com/openclaw/openclaw/pull/83015, or replace it with an equivalent narrow `memoryFlushAt`-aware guard that preserves real cycle-0 deduplication.

## Reproduction Assessment

Yes, at source level. An above-threshold entry with `compactionCount: 0` and `memoryFlushCompactionCount: 0` reaches the equality helper, returns true for already flushed, and suppresses the memory-flush path.

## Solution Assessment

Yes for the linked fix direction. Using `memoryFlushAt` only to disambiguate the zero-counter case is narrower than treating every cycle-0 row as unflushed, because it preserves dedupe for real cycle-0 flush records.
