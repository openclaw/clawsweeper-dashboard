## Summary

Keep open. Current main and the latest release still treat the reported persisted 0/0 memory-flush counter state as already flushed, and the focused tests continue to encode that behavior. The remaining fix needs a counter-state compatibility decision because real cycle-0 flush records can also store `memoryFlushCompactionCount: 0`.

## What This Changes

The issue requests a fix so memoryFlush can run for never-compacted sessions instead of being skipped when `compactionCount` and `memoryFlushCompactionCount` are both 0.

## Best Possible Solution

Define a backward-compatible session metadata contract that distinguishes never-flushed legacy 0/0 rows from real cycle-0 flush records, likely using `memoryFlushAt` or an explicit sentinel, then update the helper, runner persistence, focused tests, docs, and changelog while coordinating with the #51421 counter-semantics work.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with fresh total tokens above threshold and both counters at 0, `shouldRunMemoryFlush` reaches `hasAlreadyFlushedForCurrentCompaction`, which returns true and skips memoryFlush.

## Solution Assessment

No current fix is present. The issue’s Option A is effectively the current numeric check and does not fix 0/0; the safer solution is an explicit state contract that preserves real cycle-0 flush deduplication.
