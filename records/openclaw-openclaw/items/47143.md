## Summary

Keep open. Current main still treats the reported persisted 0/0 memory-flush counter state as already flushed, and the focused tests still encode that behavior; #51421 is adjacent counter-semantics work but does not by itself close this issue.

## What This Changes

The issue asks OpenClaw to run the first memory flush for never-compacted sessions instead of skipping when both `compactionCount` and `memoryFlushCompactionCount` are 0.

## Best Possible Solution

Define a backward-compatible memory-flush state contract that treats legacy or never-flushed 0/0 rows as not flushed while preserving deduplication for real cycle-0 flush records, then update focused tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: above-threshold `shouldRunMemoryFlush` reaches `hasAlreadyFlushedForCurrentCompaction`, which returns true for the reported 0/0 state and skips the flush.

## Solution Assessment

No current fix is present. The narrow maintainable solution is to distinguish never-flushed persisted rows from recorded cycle-0 flushes instead of adding a broad pre-request budget change or a blanket `compactionCount === 0` bypass.
