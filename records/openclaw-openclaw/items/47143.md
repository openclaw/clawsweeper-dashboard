## Summary

Keep open. Current main still treats the reported 0/0 memory-flush counter state as already flushed, and the focused tests still encode that behavior; the fix needs to preserve real cycle-0 flush deduplication while distinguishing never-flushed persisted rows.

## What This Changes

The issue requests that memoryFlush run for high-token sessions whose `compactionCount` and `memoryFlushCompactionCount` are both 0 instead of skipping them as already flushed.

## Best Possible Solution

Define a backward-compatible memory-flush state contract that treats 0/0 without a recorded flush marker as never flushed, preserves cycle-0 duplicate-flush prevention for recorded flushes, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with fresh total tokens above threshold and both counters at 0, `shouldRunMemoryFlush` reaches `hasAlreadyFlushedForCurrentCompaction`, which returns true and skips memoryFlush.

## Solution Assessment

No current fix is present. The safest fix is not a blind 0/0 special case, but an explicit state distinction, likely using `memoryFlushAt` or a future sentinel, while coordinating with the #51421 counter-semantics work.
