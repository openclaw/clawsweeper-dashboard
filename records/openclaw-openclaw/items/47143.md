## Summary

Current main still contains the reported 0/0 false-positive guard, and an independent user confirmed the resulting skipped flushes. The focused closing PR remains open, so this issue should stay open until that implementation merges or is rejected; separate preflight token-estimation concerns do not invalidate this narrower bug.

## What This Changes

Fix memory-flush deduplication so a never-flushed session with zero compactions remains eligible for its first flush.

## Best Possible Solution

Merge a focused repair that uses the existing flush timestamp to distinguish never-flushed zero rows from genuine cycle-zero flushes, while preserving equality-based deduplication for later cycles.

## Reproduction Assessment

Yes from current-main source: an otherwise eligible entry with both counters at zero makes hasAlreadyFlushedForCurrentCompaction return true. This read-only review did not execute tests, but the existing test suite encodes the same predicate and a second user reported the behavior in a real session.

## Solution Assessment

Yes in direction. Using existing memoryFlushAt state to disambiguate a genuine cycle-zero flush is narrower and safer than disabling deduplication for every zero-compaction session.
