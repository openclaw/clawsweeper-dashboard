## Summary

Keep open. Current main and the latest release still use the equality-only memory-flush guard, the focused test still encodes the faulty 0/0 state, and https://github.com/openclaw/openclaw/pull/83015 is an open closing PR for the narrow fix path.

## What This Changes

The issue reports that `memoryFlush` is skipped for sessions whose `compactionCount` and `memoryFlushCompactionCount` are both `0` because the dedupe helper treats that state as already flushed.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/83015 or an equivalent `memoryFlushAt`-aware fix while keeping broader pre-request context-budget work separate.

## Reproduction Assessment

Yes, at source level. An above-threshold entry with `compactionCount: 0` and `memoryFlushCompactionCount: 0` reaches the equality helper, returns true for already flushed, and suppresses the memory-flush path.

## Solution Assessment

Yes for the `memoryFlushAt`-aware direction. It is narrower than treating every cycle-0 row as unflushed because it preserves dedupe for real cycle-0 flush records while allowing never-flushed legacy rows through.
