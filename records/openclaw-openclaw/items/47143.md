## Summary

Keep open. Current main and latest release still skip the reported 0/0 memory-flush counter state, and the focused tests still encode that skip; the prior direct fix PRs are closed unmerged and #51421 is adjacent counter-semantics work rather than a complete fix for this issue.

## What This Changes

The issue reports that a persisted memory-flush counter state with both compaction counters at 0 is treated as already flushed, preventing the first automatic memory flush.

## Best Possible Solution

Define a backward-compatible memory-flush state contract that treats legacy or never-flushed 0/0 rows as not flushed while preserving deduplication for real cycle-0 flush records.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an above-threshold entry with `compactionCount: 0` and `memoryFlushCompactionCount: 0` reaches `hasAlreadyFlushedForCurrentCompaction`, returns true, and skips the flush.

## Solution Assessment

No current fix is present. The narrow maintainable solution is to distinguish never-flushed persisted rows from recorded cycle-0 flushes, while keeping broader pre-request context-budget work separate.
