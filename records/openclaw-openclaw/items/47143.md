## What This Changes

The issue asks OpenClaw to let memoryFlush run for over-threshold sessions whose `compactionCount` and `memoryFlushCompactionCount` are both `0` when no flush has actually run.

## Best Possible Solution

Land a narrow fix that distinguishes legacy or never-flushed 0/0 rows from recorded cycle-0 flushes while preserving the documented once-per-compaction-cycle guard.

## Reproduction Assessment

Yes, from source inspection: current main's helper returns true for the reported 0/0 counter state, and `shouldRunMemoryFlush` then skips the flush. I did not run tests because this review is read-only and local test runners may write artifacts.

## Solution Assessment

Yes, the best fix is a narrow state-contract disambiguation such as the linked PR's `memoryFlushAt` guard, because it allows legacy or never-flushed 0/0 rows through while preserving real cycle-0 flush deduplication.
