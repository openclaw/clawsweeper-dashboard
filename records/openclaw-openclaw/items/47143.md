## Summary

Keep open. Current main and the latest release still treat the reported persisted 0/0 memory-flush counter state as already flushed, and the surrounding code shows why a naive 0-special-case fix would break duplicate-flush prevention for real cycle-0 flushes.

## What This Changes

The issue requests that memoryFlush run for never-flushed sessions whose compaction and flush counters are both 0 instead of treating that state as already flushed.

## Best Possible Solution

Define a backward-compatible memory-flush state contract that distinguishes legacy or never-flushed 0/0 rows from recorded cycle-0 flushes, then update the helper, persistence path, and focused regression tests.

## Reproduction Assessment

Yes, at source level. An above-threshold entry with `totalTokensFresh: true`, `compactionCount: 0`, and `memoryFlushCompactionCount: 0` reaches the equality helper, returns true, and causes `shouldRunMemoryFlush` to skip.

## Solution Assessment

No patch is currently under review. The narrow maintainable fix should use `memoryFlushAt` or an explicit sentinel/state distinction rather than treating every cycle-0 row as unflushed.
