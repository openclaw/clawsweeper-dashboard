## Summary

Keep open: current main and v2026.6.6 still support explicit flat model fallback lists, not provider-family wildcard/auto fallback expansion or grouped fallback families. The request is useful, but it adds public model-routing and config semantics that need maintainer product design.

## What This Changes

The issue requests provider-family model fallback syntax such as wildcard/auto model refs or ordered fallback groups so fallback stays within a user's intended provider before crossing providers.

## Best Possible Solution

Design an additive provider-family fallback contract that preserves explicit flat fallbacks, defines migration/doctor behavior, and updates schema, docs, CLI, runtime candidate resolution, and focused fallback tests.

## Reproduction Assessment

Yes, from source inspection: current main only accepts `primary` plus flat `fallbacks`, and the runtime candidate chain consumes only that flat list. I did not run a live provider failure because the missing behavior is a config/runtime capability gap rather than a single provider outage.

## Solution Assessment

Unclear as proposed: wildcard/auto and `fallbackFamilies` are both plausible, but the best permanent shape needs maintainer choices on config contract, fallback order, provider ownership, and upgrade handling.
