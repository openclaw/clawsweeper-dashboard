## Summary

Keep open. Current main has explicit fallback chains, provider wildcard visibility, and OpenAI Codex legacy/default cleanup, but it still does not implement dynamic provider-family auto fallback; that would change model eligibility, ordering, quality, and cost policy.

## What This Changes

Request provider-scoped auto/wildcard model fallback so refs like `openai-codex/auto` or `openai-codex/*` try models within that provider before cross-provider fallbacks.

## Best Possible Solution

Define a generic provider-scoped fallback contract that expands `provider/*` or `provider/auto` through the configured or discovered catalog with deterministic priority and safety rules, then update fallback runtime, validation, CLI/docs, and tests together.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for new model-selection behavior. Source inspection confirms current fallback walks configured candidates, and the issue does not include a current-main config/log for the alleged cross-provider fallback path.

## Solution Assessment

Unclear as stated: provider-scoped auto fallback is plausible, but the maintainable fix needs a deterministic eligibility and ordering contract first. Current exact fallback lists and visibility wildcards should remain distinct until that policy is defined.
