## Summary

Keep open. Current main has explicit ordered model fallback and provider wildcard entries for model visibility, but it still does not implement provider-scoped `auto` or wildcard expansion as a fallback candidate source before cross-provider fallback.

## What This Changes

The issue requests provider-scoped automatic model fallback using selectors such as `openai-codex/auto` or `openai-codex/*`.

## Best Possible Solution

Define a generic provider-scoped fallback contract, keep explicit `fallbacks` as the deterministic operator-controlled path, and document/test any `provider/auto` or `provider/*` behavior with auth, capability, and cost guardrails.

## Reproduction Assessment

Yes, at source level. Current main parses `openai-codex/auto` as a literal model id and builds fallback candidates only from the requested model plus explicit fallback strings.

## Solution Assessment

Unclear as proposed. The user problem is valid, but a literal wildcard or `auto` suffix is not enough without maintainer-defined ordering, eligibility, and cost/tier rules.
