## Summary

Keep open. Current main has explicit fallback chains, model-not-found failover for configured candidates, and specific OpenAI Codex forward-compat entries, but it still does not implement provider-scoped `auto` or wildcard fallback expansion for refs such as `openai-codex/auto` or `openai-codex/*`. The remaining request needs maintainer/API semantics before it is safe to implement.

## What This Changes

Request provider-scoped `auto` or wildcard model refs that try an ordered provider-family model pool before falling back to cross-provider router models.

## Best Possible Solution

Define a generic provider-scoped fallback contract before implementation: keep explicit `fallbacks` as the predictable operator-controlled path, and if maintainers want `provider/auto` or `provider/*`, expose an ordered provider/plugin candidate API with capability, auth, and cost/tier filtering, then wire that into `runWithModelFallback` and the config/docs/tests.

## Reproduction Assessment

Yes for the missing feature at source level: `openai-codex/auto` parses as provider `openai-codex` with literal model `auto`, and the fallback runner only walks explicit configured fallback refs. I did not run live provider calls because this is a read-only review, but the code path is direct.

## Solution Assessment

Unclear as proposed. The user problem is real, but a literal wildcard/auto suffix is not yet the safest maintainable solution until maintainers decide ordering, provider ownership, eligibility, and cost semantics; explicit fallback lists remain the current workaround.
