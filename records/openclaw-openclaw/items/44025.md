## Summary

Keep open. Current main supports exact configured fallback chains and provider-wide model visibility, but it does not implement provider-family auto fallback from refs like `openai-codex/auto` or `openai-codex/*`; that would be a new config and selection-policy feature.

## What This Changes

Request provider-scoped automatic model fallback using refs like `openai-codex/auto` or `openai-codex/*`, with optional priority lists, before falling through to cross-provider routers.

## Best Possible Solution

Add an explicit, documented provider-family fallback contract that expands to an ordered, auth-backed model catalog during fallback-candidate construction, with schema/docs/tests and clear opt-in semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main only builds fallback candidates from the requested model plus explicit fallback lists, so the requested provider-family auto fallback behavior is not present today.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation needs maintainer decisions around catalog ordering, failover-worthy errors, provider cost/subscription boundaries, and how it should interact with existing `provider/*` visibility semantics.
