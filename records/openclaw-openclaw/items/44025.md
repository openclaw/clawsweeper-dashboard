## Summary

Keep open. Current main supports explicit ordered model fallbacks and OpenAI Codex forward-compat rows for known GPT-5.4/GPT-5.5 models, but it still does not implement provider-scoped `auto` or wildcard expansion before cross-provider fallback; the remaining behavior needs maintainer semantics for ordering, auth eligibility, capabilities, and cost/tier guardrails.

## What This Changes

The issue requests provider-scoped `auto` or wildcard model selection that falls back within a provider family before cross-provider fallback.

## Best Possible Solution

Define a generic provider-scoped fallback contract in core, keep explicit `fallbacks` as the deterministic operator-controlled path, and document/test any `provider/auto` or `provider/*` behavior with capability, auth, and cost/tier filtering.

## Reproduction Assessment

Yes, at source level. `openai-codex/auto` is parsed as a literal `auto` model under `openai-codex`, and the fallback planner only appends explicit configured fallback refs.

## Solution Assessment

Unclear as proposed. The user problem is valid, but a literal wildcard/auto suffix is not clearly the safest contract until maintainers define ordering, eligibility filters, and guardrails; explicit fallback lists remain the current workaround.
