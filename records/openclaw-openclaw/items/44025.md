## Summary

Keep open. Current main still supports explicit ordered model fallbacks and provider wildcard allowlists, but it does not implement provider-family wildcard or `auto` fallback expansion; adding that would change model eligibility, ordering, and cost/quality policy.

## What This Changes

The issue requests provider-scoped wildcard or `auto` model fallback selection so OpenClaw can try models within the intended provider family before falling back to cross-provider routers.

## Best Possible Solution

Define a generic provider-scoped fallback contract that expands `provider/*` or `provider/auto` through a deterministic configured/discovered catalog order, then update runtime fallback, validation, CLI/docs, and regression tests together.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for new model-selection behavior. Source inspection confirms current fallback uses explicit configured candidates, and the issue lacks a current-main config/log that demonstrates the claimed cross-provider fallback path.

## Solution Assessment

Unclear as stated. The direction is plausible, but the maintainable solution needs a provider-scoped ordering and eligibility contract before overloading `provider/*` or adding `provider/auto` runtime semantics.
