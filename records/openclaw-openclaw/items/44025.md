## Summary

Keep open: current main and the latest release support explicit fallback chains, `openrouter:auto`, and `provider/*` catalog visibility, but they still do not implement provider-scoped `auto` or wildcard fallback expansion. This remains a product/API decision about user-facing model config and routing policy, not conservative cleanup.

## What This Changes

The issue requests provider-scoped `auto` or wildcard model fallback semantics so users can stay within a selected provider family when configured model IDs rotate.

## Best Possible Solution

Define a provider-scoped fallback contract first, then align parsing, schema/help, runtime candidate expansion, docs, and regression coverage while preserving explicit fallback chains and strict user selections.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config semantics request. Source inspection gives high confidence that current main lacks generic `provider/auto` or `provider/*` fallback candidate expansion.

## Solution Assessment

Unclear until maintainers decide the contract. `provider/auto` and `provider/*` are plausible syntax, but provider-local ordering, eligibility, and cost/subscription policy need product/API approval first.
