## Summary

Keep open. Current main has explicit configured model fallback chains and provider-wildcard allowlists for model visibility, but it does not implement provider-scoped `auto` or wildcard expansion as a runtime fallback candidate source.

## What This Changes

The issue requests provider-scoped automatic model fallback using model refs such as `openai-codex/auto` or `openai-codex/*`, with optional provider-priority lists.

## Best Possible Solution

Define a backward-compatible model fallback policy that can expand provider-scoped dynamic candidates only from an approved source, with docs and tests covering allowlists, explicit user overrides, auth/profile rotation, and cost-control expectations.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for new model-routing semantics. Source and docs inspection do show current main only supports explicit configured fallback lists, not provider-scoped auto expansion.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers should choose the policy boundary before implementation because wildcard/auto fallback interacts with dynamic catalogs, allowlists, provider auth, and strict session overrides.
