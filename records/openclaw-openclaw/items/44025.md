## Summary

Keep open. Current main has explicit model fallback chains and provider wildcard allowlists, but it does not implement dynamic provider-family fallback selection, and adding that would change runtime model selection and cost/quality policy.

## What This Changes

Request provider-local automatic fallback selection via `provider/auto` or `provider/*` model config strings before falling back to cross-provider routers.

## Best Possible Solution

Define a generic provider-scoped fallback contract that expands `provider/*` or `provider/auto` through the configured/discovered catalog with deterministic priority and safety rules, then update fallback runtime, validation, CLI, tests, and docs together.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for new model-selection behavior. Source inspection confirms current fallback uses explicit configured candidates, but the issue does not provide a current-main config/log that demonstrates the reported cross-provider fallback path.

## Solution Assessment

The direction is plausible but not yet the maintainable solution as stated. Current `provider/*` support is an allowlist/catalog-visibility feature, so runtime fallback should reuse that contract only after maintainers define ordering, eligibility, and cost/quality semantics.
