## Summary

Keeping this open: current main and the latest release still only support flat configured model fallback lists; provider wildcard support is for model visibility, not fallback execution, so wildcard/auto provider-family fallback needs maintainer product design.

## What This Changes

Request provider-family wildcard or auto model fallback selection, with optional ordered provider priority lists, so model rotation stays within the intended provider before crossing to generic routers.

## Best Possible Solution

Choose one canonical model-fallback contract, then update schema/types, config normalization, runtime candidate ordering, public docs, and focused fallback-chain tests around that contract.

## Reproduction Assessment

Not a bug reproduction. Source inspection gives a high-confidence way to verify the missing behavior: current schema, config parsing, runtime fallback resolution, docs, and v2026.6.6 all expose only flat fallback lists.

## Solution Assessment

Unclear as proposed. The direction fits the provider roadmap, but maintainers should first choose between implicit provider-family auto expansion, explicit fallbackFamilies ordering, or a related dynamic-catalog design so fallback behavior stays predictable.
