## Summary

Keep open: current main and v2026.6.8 still only support explicit flat model fallback lists, while wildcard/auto provider-family fallback and `fallbackFamilies` remain absent. The related model-router hook discussions do not cover failure-driven provider-family retry semantics, so this still needs maintainer product/config direction.

## What This Changes

The issue requests provider-family model fallback syntax such as `provider/auto`, `provider/*`, or ordered fallback families so OpenClaw stays within an intended provider/subscription tier before crossing to generic routers.

## Best Possible Solution

Maintainers should choose one canonical model-family fallback contract, then update schema/types, config normalization, runtime candidate construction, docs, and fallback tests around that contract.

## Reproduction Assessment

Yes for source-level verification: current main and v2026.6.8 only support flat `fallbacks`, and searches found no `fallbackFamilies` implementation. This is a feature request rather than a failing-bug reproduction.

## Solution Assessment

Unclear as proposed: the goal is valid, but wildcard/auto expansion and explicit fallback families are different public contracts. The best fix needs maintainer selection of one predictable config and provider-routing model before implementation.
