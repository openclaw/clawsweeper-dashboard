## Summary

Keep open. Current main supports explicit ordered model fallbacks and provider wildcard allowlists for model visibility, but it still does not expand `provider/*` or `provider/auto` into provider-local fallback candidates; implementing that needs a model eligibility and ordering decision.

## What This Changes

Request provider-scoped wildcard or auto model fallback so refs such as `openai-codex/auto` or `openai-codex/*` try a provider-local model pool before cross-provider fallbacks.

## Best Possible Solution

Define a generic provider-scoped fallback contract that expands `provider/*` or `provider/auto` through a deterministic configured or discovered catalog order, then update runtime fallback, validation, CLI/docs, and regression tests together.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for new model-selection behavior. Source inspection confirms current main uses explicit configured fallback candidates, and the issue lacks a config/log that demonstrates the claimed silent cross-provider fallback path.

## Solution Assessment

Unclear as stated. The direction is plausible, but the maintainable solution needs a provider-scoped ordering and eligibility contract before overloading `provider/*` or adding `provider/auto` runtime semantics.
