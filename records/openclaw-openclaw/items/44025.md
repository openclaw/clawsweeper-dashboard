## Summary

Keep open: current main and v2026.6.5 support explicit fallback chains and provider-scoped wildcard catalog visibility, but runtime fallback still does not expand a provider pool or choose a provider-local default automatically; the newer `fallbackFamilies` discussion adds a viable alternative but reinforces that maintainers must choose the public routing contract.

## What This Changes

Requests provider-scoped automatic or wildcard model fallback selection that exhausts the user’s chosen provider family before crossing providers.

## Best Possible Solution

Choose one canonical provider-local fallback contract under the current `openai` provider, then align schema, config normalization, candidate expansion, observability, documentation, and regression coverage while retaining flat explicit fallbacks.

## Reproduction Assessment

No as a failing-bug reproduction because this is a new routing capability; yes at source level, because current main demonstrably builds fallback candidates only from explicit model references.

## Solution Assessment

Unclear until maintainers choose the contract: `provider/auto`, runtime `provider/*`, catalog-default selection, and grouped fallback families are plausible but materially different policies.
