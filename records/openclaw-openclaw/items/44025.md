## Summary

Keeping this open: current main and the latest release still only support flat model fallback lists plus narrow alias/visibility behavior, while the requested provider-family wildcard or auto selection would add new provider-routing and config semantics that need maintainer product design.

## What This Changes

Request provider-family model fallback selection such as `openai-codex/auto` or `provider/*`, with optional ordered family/group fallback before cross-provider routers.

## Best Possible Solution

Design one canonical model-fallback contract, likely either provider-family `auto` expansion or explicit fallback groups, then update schema/types, normalization, docs, doctor guidance if needed, and fallback-chain tests.

## Reproduction Assessment

Not a bug reproduction. Source inspection confirms current behavior is flat fallback lists only; no provider-family wildcard or auto fallback path exists on main or `v2026.6.6`.

## Solution Assessment

Unclear as proposed. The direction fits the model-provider roadmap, but maintainers still need to choose between implicit provider `auto` expansion and explicit `fallbackFamilies` ordering so costs and provider boundaries stay predictable.
