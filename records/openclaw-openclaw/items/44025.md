## Summary

Keep open: current main and the latest release still expose only explicit flat model fallback lists, while provider-family wildcard/auto selection and the commenter-proposed `fallbackFamilies` shape remain absent. The existing `before_model_resolve` hook is useful adjacent context, but it only preselects a provider/model before a turn and does not define failure-driven provider-family retry semantics.

## What This Changes

The issue requests provider-family model fallback semantics such as `openai-codex/auto`, wildcards, or ordered fallback families so failures stay inside the intended provider tier before crossing to router models.

## Best Possible Solution

Keep this issue as the provider-family fallback design tracker until maintainers choose a canonical contract, then update schema/types, config normalization, fallback candidate construction, docs, and focused fallback tests around that contract.

## Reproduction Assessment

Yes from source-level verification: current main and `v2026.6.10` only support flat explicit fallback lists, while provider-family wildcard/auto fallback and `fallbackFamilies` are absent. This is a feature request rather than a failing-bug reproduction.

## Solution Assessment

Unclear as proposed: the goal is valid, but wildcard/auto expansion and explicit fallback families are different public contracts. The best fix needs maintainer selection of one predictable config and provider-routing model before implementation.
