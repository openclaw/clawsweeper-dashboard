## Summary

Keep open. Current main supports explicit ordered model fallback lists and OpenAI Codex forward-compat rows for known models, but it still does not provide provider-scoped `auto` or wildcard expansion before cross-provider fallback; the remaining work needs maintainer semantics for ordering, auth eligibility, capability matching, and cost/tier behavior.

## What This Changes

Request provider-scoped auto/wildcard model refs such as `openai-codex/auto` or `openai-codex/*` that try ordered models within that provider before falling back across providers.

## Best Possible Solution

Define a generic provider-scoped fallback contract, keep explicit `fallbacks` as the predictable operator-controlled path, and document/test any `provider/auto` or `provider/*` behavior with capability, auth, and cost/tier filtering.

## Reproduction Assessment

Yes, at source level for the missing capability: `openai-codex/auto` parses as a literal `auto` model under `openai-codex`, and the fallback planner only appends explicit configured fallback refs. I did not run live provider calls because this review was read-only.

## Solution Assessment

Unclear as proposed. The user problem is valid, but a literal wildcard/auto suffix is not clearly the safest contract until maintainers define provider ordering, eligibility filters, and cost/tier guardrails; explicit fallback lists remain the current workaround.
