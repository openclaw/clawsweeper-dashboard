## Summary

Keep open. Current main supports explicit ordered fallback chains and provider wildcard visibility, but it does not expand `provider/auto` or `provider/*` into provider-local fallback candidates; that needs a product/API decision for model eligibility, ordering, and cost behavior.

## What This Changes

Add provider-scoped `provider/auto` or `provider/*` model config semantics that try models inside the selected provider family before cross-provider fallbacks.

## Best Possible Solution

Define a generic provider-scoped fallback contract that deterministically expands `provider/auto` or `provider/*` through configured or discovered provider catalog order, then align runtime fallback, validation, CLI/docs, and regression tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for new model-selection/config semantics. Source inspection confirms current main only walks explicit fallback refs, and the report lacks current-main config/log evidence for the claimed silent cross-provider fallback path.

## Solution Assessment

Unclear as stated. The direction is plausible, but overloading `provider/*` or adding `provider/auto` should wait for an explicit provider-scoped ordering, eligibility, and cost/quality policy.
