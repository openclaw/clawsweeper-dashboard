## Summary

Keep open: the credential exposure is a real Gmail watcher bug, but this PR is not merge-ready because it calls gogcli file-secret flags that are absent from current gogcli source/docs and the latest gogcli release, while also leaving early temporary-file cleanup gaps.

## What This Changes

The PR changes Gmail watcher startup to write push and hook tokens to temporary files, pass file paths to `gog gmail watch serve`, and add unit coverage for file-backed argv and file permissions.

## Best Possible Solution

Ship or require a compatible gogcli contract first, then land a cleanup-safe OpenClaw handoff that covers both Gateway auto-start and the documented foreground Gmail runner.

## Reproduction Assessment

Yes. Current main builds the Gmail watcher command with raw `cfg.pushToken` and `cfg.hookToken` in `buildGogWatchServeArgs`, so a configured watcher exposes both values through the child process argv.

## Solution Assessment

No. The mitigation is directionally right, but this PR invokes unsupported gogcli flags, leaves temporary secret files behind on some setup failures, and does not cover the foreground Gmail runner sibling path.
