## Summary

Keep open: the shared schema-derived direction is useful, but latest head still misses explicit-false semantics in the CLI picker for absent flags, and the PR still needs real toggle/write proof plus maintainer approval for the new config-editing surface.

## What This Changes

The PR adds a schema-derived experimental-flag registry exposed through `openclaw experimental`, `/experimental`, TUI handlers, native command registration, config schema/docs, and focused tests.

## Best Possible Solution

Keep the shared schema-derived registry, make the CLI picker persist explicit false when false is a meaningful opt-out, then merge only after maintainer approval and real toggle/write proof.

## Reproduction Assessment

Yes. By source inspection, PR head's CLI selection helper treats an absent, unselected flag as no delta, while current main treats undefined `tools.experimental.planTool` as eligible for strict-agentic auto-enable and docs require explicit false to opt out.

## Solution Assessment

No, not as-is. The schema-derived shared surface is the right shape, but the CLI picker still misses the absent-false opt-out case and the broader config-editing surface needs maintainer approval and real behavior proof.
