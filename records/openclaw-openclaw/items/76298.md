## Summary

Keep open: the shared schema-derived direction responds to maintainer feedback, but the interactive CLI picker still fails the explicit-false opt-out case and the PR lacks real toggle/write proof plus maintainer acceptance for the new config-editing surface.

## What This Changes

This PR adds a schema-derived experimental flag registry exposed through `openclaw experimental`, `/experimental`, TUI handlers, command gating, docs, and tests.

## Best Possible Solution

Keep the shared schema-derived registry, make both interactive and direct writes persist explicit `false` when absence has runtime meaning, then require maintainer approval and redacted real toggle/write proof before merge.

## Reproduction Assessment

Yes. Source inspection shows the CLI picker path treats an absent unchecked flag as no change, while current main treats undefined `tools.experimental.planTool` as eligible for strict-agentic auto-enable and explicit `false` as the opt-out.

## Solution Assessment

No, not as-is. The shared schema-derived approach is maintainable, but the interactive picker must preserve explicit false semantics and the new config-editing surface needs maintainer acceptance plus real behavior proof.
