## Summary

Keep open: the approach is plausible for the reported Telegram behavior, but the branch still needs real Telegram proof, retains a small test-coverage regression, and introduces a public plugin-command result flag whose documented channel scope is not yet clear.

## What This Changes

The PR adds a `suppressReply` plugin-command result flag, makes Telegram native commands skip the empty fallback when it is set, and adds a regression test for that path.

## Best Possible Solution

Land a documented silent command-result contract with Telegram suppression, retained direct-message context coverage, and real Telegram proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes from source inspection: current main converts an undefined plugin command result to `{}` and the Telegram native command path replaces non-renderable results with the empty-response fallback. I did not run a live Telegram repro because this was a read-only review.

## Solution Assessment

Mostly yes: an explicit result flag is a narrow way to distinguish out-of-band delivery from an accidental empty result. The safer completion is to document or scope the public SDK contract, keep the existing context regression test, and attach real Telegram proof before merge.
