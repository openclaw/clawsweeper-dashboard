## Summary

Keep this PR open: it targets the right Telegram fallback path, but it exposes `suppressReply` on the public plugin command result contract while only the Telegram native-command path honors it. The supplied proof is unit-test terminal output only, and Telegram maintainer guidance calls for real Telegram proof for this kind of visible behavior change.

## What This Changes

The PR adds a `suppressReply` plugin command result flag, skips the Telegram native empty-response fallback when it is set, and adds a focused unit test.

## Best Possible Solution

Define the silent command result as a documented plugin SDK contract and either handle it in shared plugin command plumbing plus native adapters, or keep it scoped to Telegram-only channel data if only Telegram should honor it.

## Reproduction Assessment

Yes. Current main is source-reproducible: undefined or empty plugin command results normalize to an empty payload, and Telegram native commands map non-renderable results to the empty-response fallback; an existing test asserts that fallback.

## Solution Assessment

No, not as currently scoped. A silent command result is plausible, but adding it to the public `PluginCommandResult` type requires consistent handling or explicit Telegram scoping before merge.
