## Summary

Keep open: current main still lacks a silent plugin-command result, this PR remains the canonical implementation path for the linked Telegram fallback issue, and Mantis now supplies real Telegram proof. The remaining concerns are narrow cleanup: preserve the direct-message binding test and document the new public SDK flag/scope.

## What This Changes

The PR adds a `suppressReply` plugin-command result flag, skips Telegram’s empty fallback when that flag is returned, and adds a regression test for the suppressed-reply path.

## Best Possible Solution

Land a documented silent command-result contract with Telegram suppression, retained direct-message binding coverage, and the attached Mantis Telegram proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: undefined command results become `{}` and Telegram falls back on non-renderable payloads. The Mantis baseline also shows the duplicate fallback in native Telegram Desktop, though I did not run a fresh local live repro.

## Solution Assessment

Mostly yes. An explicit `suppressReply` flag is a narrow way to distinguish out-of-band delivery from an accidental empty result, but the PR should document the SDK scope and keep the existing direct-message context coverage.
