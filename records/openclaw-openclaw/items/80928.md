## Best Possible Solution

Land the fallback suppression after maintainers approve the SDK contract shape or narrow it to a Telegram-specific result contract, and add the suppression test without removing the existing DM binding test.

## Reproduction Assessment

Yes. The linked bug gives a clear plugin-command path, current source falls back when the Telegram native plugin result has no renderable content, and Mantis before/after Telegram Desktop proof shows the duplicate fallback on main but not on the candidate.

## Solution Assessment

Unclear until maintainer API-scope approval. Suppressing the fallback is the right behavior, but a Telegram-only top-level `PluginCommandResult` field may be broader than the narrowest maintainable contract, and the deleted DM regression coverage should be restored.
