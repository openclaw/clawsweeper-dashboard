## Summary

Keep open: the PR is a plausible narrow fix for the Telegram fallback path, but it still needs real Telegram behavior proof or an explicit proof override, plus small SDK/docs and test-coverage follow-up before merge.

## What This Changes

Adds a `suppressReply` plugin-command result flag, makes Telegram native commands skip the empty-response fallback when that flag is set, and adds a regression test for the suppressed path.

## Best Possible Solution

Land a documented silent command-result contract with Telegram suppression, retained direct-message binding coverage, and real Telegram proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, from source inspection: current main turns an undefined command result into `{}` and Telegram native commands map non-renderable results to the empty fallback. I did not run a live Telegram reproduction during this read-only review.

## Solution Assessment

Mostly yes: an explicit result flag is a narrow way to distinguish out-of-band delivery from an accidental empty result. The maintainable completion is to document the public SDK scope, preserve existing binding-context coverage, and attach real Telegram proof or a maintainer override.
