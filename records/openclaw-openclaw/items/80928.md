## Summary

Keep open: this PR addresses a concrete Telegram plugin-command duplicate-fallback problem with source evidence and Mantis before/after Telegram proof, but it still needs maintainer acceptance of the new public command-result contract plus small docs/test cleanup before merge.

## What This Changes

The PR adds a `suppressReply` plugin-command result flag, makes Telegram native plugin commands clean up and skip the empty fallback when it is set, and adds regression coverage for that path.

## Best Possible Solution

Land a small, documented no-reply command-result contract that preserves Telegram binding coverage, clearly states channel support in SDK docs, and keeps the Mantis Telegram proof attached to the canonical PR.

## Reproduction Assessment

Yes. The failing path is high-confidence from source and current tests, and Mantis captured the baseline Telegram Desktop fallback on the PR base; I did not run the same live probe on current `main` SHA `49e288823cda6efeef12b0ebfca3777fe36fbe12`.

## Solution Assessment

Mostly yes. An explicit command-result flag is a narrow way to distinguish out-of-band delivery from an accidental empty result, but the public SDK scope must be documented and existing direct-message coverage should be preserved.
