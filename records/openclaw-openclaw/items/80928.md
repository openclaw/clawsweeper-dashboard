## Summary

Keep this PR open: current main still reproduces the Telegram fallback problem, the branch contains the canonical useful fix, and the later Mantis Telegram Desktop proof is sufficient real-behavior evidence. The remaining issues are small review items around preserving existing binding coverage and documenting the new public command-result flag before maintainers decide whether to land the SDK contract.

## What This Changes

The PR adds a `suppressReply` plugin-command result flag and makes Telegram native command delivery skip the empty fallback when that flag is returned.

## Best Possible Solution

Land a small documented command-result contract, keep the existing Telegram binding coverage, retain the suppression regression test, and use the Mantis proof as the real Telegram verification record.

## Reproduction Assessment

Yes. Current main source and tests show undefined command results become `{}` and Telegram sends the empty-response fallback, and Mantis captured the same fallback in native Telegram Desktop before the candidate fix.

## Solution Assessment

Mostly yes. An explicit `suppressReply` result is a narrow fix for out-of-band Telegram delivery, but the public SDK docs and retained binding coverage should be completed before maintainers land the contract.
