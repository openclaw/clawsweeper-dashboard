## Summary

Keep open. The PR targets a real source-reproducible Telegram native-command gap, but it introduces a new public plugin-command result flag and the supplied proof is only a mocked/unit test run, with no live Telegram behavior evidence.

## What This Changes

The PR adds a `suppressReply` flag to `PluginCommandResult`, handles it in the Telegram native-command fallback path, and adds a Telegram native-command regression test.

## Best Possible Solution

Land a documented SDK-level silent command result and Telegram native-command suppression path, with existing command-context tests retained and real Telegram proof attached.

## Reproduction Assessment

Yes from source inspection: `executePluginCommand` returns `{}` for an undefined handler result, and Telegram native commands currently replace non-renderable command results with the empty-response fallback. I did not run a live Telegram repro because this review was read-only.

## Solution Assessment

Mostly yes: an explicit command-result flag is a narrow way to distinguish deliberate out-of-band delivery from an accidental empty result. The safer completion is to document the SDK contract, keep the removed context coverage, and require real Telegram proof before merge.
