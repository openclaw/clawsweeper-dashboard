## Summary

Keep this PR open: the change is still useful on current main, the Mantis Telegram Desktop proof now covers the visible before/after behavior, and the only remaining issue I found is a low-priority public SDK documentation/scope gap.

## What This Changes

Adds a `suppressReply` plugin-command result flag, makes Telegram native plugin commands skip the empty fallback when it is set, and adds a regression test for that path.

## Best Possible Solution

Land the narrow suppression contract once its SDK semantics are documented, keep the Telegram regression coverage and Mantis proof attached, and close the linked issue after merge.

## Reproduction Assessment

Yes. Mantis reproduced the current-main symptom in Telegram Desktop, and source inspection shows undefined plugin-command results become `{}` before the Telegram adapter sends the empty-response fallback.

## Solution Assessment

Mostly yes. An explicit `suppressReply` result is a narrow maintainable way to distinguish plugin-owned delivery from an accidental empty result, but the public SDK docs should define the flag and its channel scope before landing.
