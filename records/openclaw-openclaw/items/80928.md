## Summary

Keep open: the source path is real and the patch is plausible, but the latest diff still needs real Telegram proof or an explicit proof override, plus small SDK docs/test coverage fixes before merge.

## What This Changes

The PR adds a `suppressReply` flag to plugin command results, makes Telegram native commands skip the empty fallback when it is set, and adds a regression test.

## Best Possible Solution

Land a documented `suppressReply` command-result contract that Telegram honors while preserving existing binding coverage, with real Telegram before/after proof or an explicit proof override.

## Reproduction Assessment

Yes from source and test inspection: current main normalizes undefined plugin command results to `{}` and Telegram native commands convert non-renderable results to the fallback. I did not run a live Telegram repro during this read-only review.

## Solution Assessment

Yes with caveats: an explicit result flag is a narrow way to distinguish out-of-band delivery from an accidental empty result. The maintainable completion is to document the public SDK scope, keep binding-context coverage, and attach real Telegram proof or a maintainer override.
