## Summary

This PR should stay open: the code direction is plausible and focused, but it still needs real Telegram proof or an explicit maintainer override, and the public command-result contract needs a little more cleanup before merge.

## What This Changes

The branch adds a `suppressReply` plugin-command result flag, makes Telegram native command handling skip its empty-response fallback when that flag is set, and adds a regression test for that path.

## Best Possible Solution

Land a small documented command-result suppression contract after maintainer approval, preserve the existing Telegram context coverage, and attach native Telegram proof showing one intended direct-send reply and no fallback.

## Reproduction Assessment

Yes, source-reproducible: current main normalizes undefined plugin command results to `{}` and Telegram maps non-renderable native command results to the empty-response fallback. I did not run live Telegram in this read-only review.

## Solution Assessment

Mostly yes: an explicit result flag is a narrow way to distinguish out-of-band delivery from accidental empty output. The safer completion is to document the SDK contract, keep existing context regression coverage, and require real Telegram proof or maintainer override.
