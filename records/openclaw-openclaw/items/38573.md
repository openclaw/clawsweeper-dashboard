## Summary

Current main and v2026.6.6 still force thread-bound cron completions through direct outbound delivery, so the requested main-session handoff remains unavailable. The rule protects reliable delivery, making this a valid feature request that needs a maintainer decision on silent agent outcomes, fallback behavior, and the configuration boundary.

## What This Changes

Allow isolated text-only cron completions targeting topics or forum threads to opt into main-session announce and agent-turn processing instead of mandatory direct delivery.

## Best Possible Solution

Define one per-job text-completion routing enum, retain direct delivery as the default, preserve topic identity, and specify atomic success and fallback behavior for silent, failed, and partially delivered main-agent turns.

## Reproduction Assessment

Yes. Current main deterministically selects direct delivery whenever the resolved cron target has a thread ID, proving with high confidence that the requested opt-in route is unavailable.

## Solution Assessment

No final solution is established yet. The opt-in direction is reasonable, but the proposed global and per-job alternatives should be consolidated into one canonical field after maintainers define delivery success and fallback semantics.

## Risks / Open Questions

- An announce-triggered main-agent turn may intentionally return no visible reply, so treating that path as delivery can silently lose scheduled output.
- Falling back to direct delivery after a main-agent turn partially sends or invokes a message tool could duplicate the cron result.
- A persisted routing option would affect cron schemas, storage codecs, CLI and gateway inputs, documentation, and upgrade expectations.
