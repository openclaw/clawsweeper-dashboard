## Summary

Keep open: current main still forces resolved thread/topic cron deliveries through direct outbound delivery and exposes no supported route selector or fallback policy, so this remains a valid feature/API request that needs maintainer design.

## What This Changes

The issue requests an opt-in cron delivery route for text-only topic/forum targets that uses the main-session announce/agent-turn flow instead of forced direct outbound send.

## Best Possible Solution

Add a backward-compatible explicit route selector for text-only isolated cron completions targeting thread/topic destinations, keep structured payloads and the default topic behavior on direct delivery, and update protocol, CLI/tool/docs, and tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows any resolved threadId forces direct delivery, and the public cron contracts have no supported opt-in route selector. A live Telegram/forum cron run is not needed to verify the missing API surface.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is reasonable, but the maintainable fix should be an additive public contract with explicit fallback semantics rather than changing the existing direct-delivery default.
