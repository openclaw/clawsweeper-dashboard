- `impact:session-state`: The request is about whether topic-bound cron output participates in the main session turn flow so users can continue chatting with the result.

## Summary

Keep open: current main and the latest release still force cron completions with a resolved thread/topic id through direct outbound delivery, and the requested opt-in announce/main-agent route needs an additive cron contract plus maintainer decisions on naming, scope, and fallback semantics.

## What This Changes

The issue requests an opt-in cron delivery mode so text-only topic/forum cron completions can use the main-session announce/agent-turn flow instead of always being sent directly.

## Best Possible Solution

Add a documented additive selector for text-only isolated cron completion routing, keep direct outbound delivery as the default, keep structured payloads direct, and define failure fallback semantics before implementation.

## Reproduction Assessment

Yes for the source-level limitation: a Telegram `:topic:` target becomes `threadId`, and dispatch sends any non-null `threadId` through direct delivery while the cron type and gateway schema have no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves current defaults instead of removing the threadId direct-delivery guard outright.
