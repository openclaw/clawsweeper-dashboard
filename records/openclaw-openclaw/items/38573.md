## Summary

Keep open: current main still forces threaded/topic isolated cron completions through direct outbound delivery, and the cron API has no opt-in route selector for announce/main-agent-turn delivery. The report is a valid feature/API gap that needs maintainer route and fallback design, not cleanup closure.

## What This Changes

Request an opt-in per-job or config option that lets text-only threaded/topic isolated cron completions use the main-session announce/agent-turn path instead of forced direct send.

## Best Possible Solution

Add an additive, default-preserving route selector for text-only isolated cron completions to thread/topic targets, keep structured payloads direct, and document/test explicit fallback semantics across protocol, CLI, agent tool, docs, and channel behavior.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main shows any resolved threadId selects the direct branch, and focused tests assert threaded Telegram outbound delivery. This is a feature/API gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the public option shape and fallback policy. The maintainable direction is an additive, default-preserving selector rather than changing the existing topic default globally.
