## Summary

Keep open. Current main still forces thread/topic cron completions through direct outbound delivery, the public cron schemas expose no opt-in main-agent-turn route, and the merged target-awareness PR only makes delivered content available to later turns rather than restoring a pre-send main-agent rewrite/handoff path.

## What This Changes

Request an explicit config or per-cron option so text-only topic/forum cron completions can opt into a main-session announce/agent-turn flow instead of always using direct delivery.

## Best Possible Solution

Define one explicit job-level text-completion routing enum for thread/topic cron delivery, default it to current direct delivery, keep structured payloads on direct delivery, and specify delivery status plus fallback behavior before implementation.

## Reproduction Assessment

Yes. Source inspection on current main shows dispatchCronDelivery uses direct delivery whenever resolvedDelivery.threadId is present, and the cron type/protocol/config surfaces expose no opt-in main-agent-turn selector.

## Solution Assessment

Unclear. The opt-in direction is plausible, but maintainers still need to choose the durable API shape and silent, failed, partial, and fallback semantics before this becomes a safe implementation task.
