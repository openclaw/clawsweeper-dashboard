## Summary

Keep open: current main and the latest release still force resolved thread/topic cron completions through direct delivery, and the public cron contract has no selector for the requested announce/main-agent-turn route. This is a valid feature/config request that needs maintainer API and fallback-policy judgment before implementation.

## What This Changes

Request adding an explicit per-job or config selector so text-only topic/forum cron completions can opt into announce/main-agent-turn delivery instead of forced direct delivery.

## Best Possible Solution

Add a documented, default-preserving selector for text-only isolated cron completions to thread/topic targets, keep structured payloads direct, and define explicit failure fallback semantics across protocol, CLI/tool, runtime, docs, and tests.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main and v2026.5.12 shows any resolved threadId selects the direct-delivery branch, and the exposed cron type/protocol/tool surfaces have no opt-in route selector. I did not run a live cron delivery flow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public option shape and fallback policy. The maintainable direction is an additive selector that preserves current direct-send defaults rather than changing topic/forum behavior globally.
