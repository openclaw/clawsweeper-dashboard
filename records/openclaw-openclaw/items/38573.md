## Label Justifications

- `P2`: This is a normal-priority enhancement for a specific cron/topic workflow with real user impact but limited blast radius.
- `impact:session-state`: The request is about whether topic-bound cron output participates in the main session turn flow so users can keep conversational context around the result.

## Summary

Keep open: current main still forces any resolved thread/topic cron delivery down the direct outbound path, and the cron type, Gateway schema, and docs expose no opt-in selector for main-session announce/agent-turn routing. The remaining work is an additive public cron/API behavior decision, not stale cleanup.

## What This Changes

Requests an additive config or cron-job option that lets text-only topic/forum cron completions opt into main-session announce/agent-turn routing while keeping direct delivery as the default.

## Best Possible Solution

Add a backward-compatible, documented selector for text-only isolated cron completion routing, keep structured payloads and current direct topic delivery as the default, and define failure fallback semantics before implementation.

## Reproduction Assessment

Yes, at source level: Telegram :topic targets resolve to threadId, dispatch sends any non-null threadId through direct delivery, and the cron type/schema/docs expose no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves current defaults instead of removing the threadId direct-delivery guard outright.
