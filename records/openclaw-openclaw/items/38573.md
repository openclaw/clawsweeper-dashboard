## Summary

Keep open: current main still routes any threaded/topic isolated cron completion through direct outbound delivery, and the cron job, protocol, CLI, tool, UI, and config surfaces do not expose an opt-in route selector for the requested main-agent-turn behavior.

## What This Changes

Add an opt-in config or per-cron delivery route for text-only topic/forum isolated cron completions to use an announce/main-agent-turn route instead of forced direct delivery.

## Best Possible Solution

Add an additive, default-preserving route selector for text-only isolated cron completions to threaded/topic targets, keep structured payloads direct, and document/test explicit failure fallback behavior across runtime, protocol, CLI/tool, UI, and docs.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main shows any resolved threadId selects the direct branch, and the delivery schemas/surfaces expose no opt-in route selector. I did not run a live Telegram cron flow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public option shape and fallback policy. The maintainable direction is an additive selector that preserves current direct-send defaults rather than changing topic/forum behavior globally.
