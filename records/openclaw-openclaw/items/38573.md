## Summary

Keep open: current main and the latest release still force thread/topic cron completion delivery through direct outbound send, and the remaining work is an additive product/API decision around the option name, scope, and fallback behavior.

## What This Changes

The issue requests an opt-in cron delivery mode that lets text-only thread/topic cron completions use main-session announce/agent-turn handling instead of forced direct delivery.

## Best Possible Solution

Add a documented additive selector for text-only isolated cron completion delivery, keep direct outbound delivery as the default, keep structured payloads direct, and define explicit fallback semantics before implementation.

## Reproduction Assessment

Yes for the current limitation: a Telegram :topic: target resolves to threadId, and current dispatch sends any non-null threadId through direct delivery while the cron type and gateway schema have no accepted opt-in selector. I did not run a live Telegram cron delivery flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves the current default and updates cron types, gateway schema, docs, and focused delivery tests together.
