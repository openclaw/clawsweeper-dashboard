## Summary

Keep open: current main still routes any isolated cron delivery with a resolved `threadId` through the direct outbound path, and cron's public CLI/tool/protocol surfaces still lack an opt-in route selector for main-session announce/agent-turn delivery. The linked subagent announce fix is adjacent but did not add this cron delivery option.

## What This Changes

Allow isolated cron deliveries to topic/forum targets to opt into main-session announce/agent-turn flow instead of always direct-sending text completions.

## Best Possible Solution

Add an additive, default-preserving route selector for text-only isolated cron completions to thread/topic targets, keep structured payloads direct, and update protocol, CLI, agent tool, UI, docs, and regression tests with explicit fallback semantics.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows any resolved `threadId` selects direct delivery, and tests assert Telegram topic direct delivery. This is a feature/API gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the API shape and fallback policy. The maintainable direction is an additive, default-preserving selector rather than changing the topic default globally.
