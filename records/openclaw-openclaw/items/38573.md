## Summary

Keep open. Current main and the latest release still force thread/topic isolated cron completions through direct outbound delivery, and no cron job or config route selector exists for opting into a main-session announce/agent-turn flow.

## What This Changes

The issue requests an opt-in cron delivery mode so text-only topic/forum isolated cron completions can route through the main-session announce/agent-turn path instead of always direct-sending to the thread.

## Best Possible Solution

Define one explicit job-level text-completion routing enum, keep direct topic delivery as the default, preserve topic identity, and specify silent, failed, partial, and fallback outcomes before implementation.

## Reproduction Assessment

Yes for the reported limitation. Source inspection shows current main deterministically selects direct delivery for any resolved cron target with threadId, and the cron schemas expose no opt-in main-agent-turn route.

## Solution Assessment

Unclear. The requested opt-in direction is plausible, but maintainers still need to choose the durable public API and fallback semantics before this is the best implementation path.

## Risks / Open Questions

- An announce-triggered main-agent turn can intentionally produce no visible reply, so opt-in delivery semantics need to define whether that is delivered, failed, or fallback-eligible.
- Fallback from failed or partial main-agent delivery to direct delivery could duplicate scheduled output if the agent already sent through a message tool or channel adapter.
- A persisted routing option would touch cron schema/types, config or job storage, CLI/Gateway validation, docs, delivery traces, and upgrade expectations.
