## Summary

Keep open. Current main and the latest release still force thread/topic isolated cron completions through direct outbound delivery, while the requested opt-in main-session announce/agent-turn route is absent and would add public cron routing plus fallback semantics that need maintainer product/API review.

## What This Changes

Request to add an opt-in cron delivery mode so topic/forum targets can use the main-session announce and agent-turn path instead of always direct-sending text completions.

## Best Possible Solution

Define one explicit job-level text-completion routing enum, keep direct topic delivery as the default, preserve topic identity, and specify delivery status plus fallback behavior for silent, failed, and partially delivered main-agent turns.

## Reproduction Assessment

Yes. Source inspection reproduces the limitation with high confidence because current main deterministically selects direct delivery for any resolved cron target with threadId, and no current cron config or job field opts into a main-session announce-agent-turn route.

## Solution Assessment

Unclear. The opt-in direction is reasonable, but maintainers still need to choose the durable API shape and silent, failed, partial, and fallback semantics before this becomes a safe implementation task.

## Risks / Open Questions

- An announce-triggered main-agent turn can intentionally produce no visible reply, so treating that route as delivery without explicit semantics could silently drop scheduled output.

- A persisted routing option would touch cron config/schema, storage codecs, CLI/Gateway input, docs, delivery status, and upgrade expectations.
