## Summary

Keep open: current main still routes any text-only cron delivery with a resolved thread/topic id through direct outbound delivery, and the public cron contract has no supported opt-in route selector or fallback policy for announce/main-agent-turn delivery.

## What This Changes

The issue requests a per-job or global option that lets text-only isolated cron completions targeting thread/topic destinations use the main-session announce/agent-turn flow instead of forced direct delivery.

## Best Possible Solution

Add a backward-compatible explicit route selector for text-only isolated cron completions to thread/topic targets, keep structured payloads and default topic delivery on direct send, and document/test the selected fallback semantics.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows any resolved threadId forces the direct delivery path, and the public cron type/schema/CLI/tool surfaces have no opt-in route selector. A live Telegram forum run is not needed to verify the missing API surface.

## Solution Assessment

Unclear until maintainers choose the API shape and fallback policy. The requested direction is reasonable, but the maintainable fix should be an additive public contract instead of changing the existing direct-delivery default.
