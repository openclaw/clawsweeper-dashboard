## Summary

Keep open: current main still forces any resolved thread/topic isolated cron completion through the direct outbound delivery path, and the public cron API does not expose an opt-in announce/main-agent-turn route selector. This is a valid feature/API gap that needs maintainer route and fallback design, not shipped or duplicate cleanup.

## What This Changes

Request an opt-in config or per-job cron option that lets text-only topic/forum isolated cron completions use announce/main-agent-turn routing while preserving direct delivery by default.

## Best Possible Solution

Add an additive, default-preserving route selector for text-only isolated cron completions to thread/topic targets, keep structured payloads direct, and document/test explicit fallback semantics across protocol, CLI, agent tool, UI, and docs surfaces.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main shows any resolved threadId selects the direct branch, and a focused test asserts threaded Telegram outbound delivery. This is a feature/API gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the public option shape and fallback policy. The maintainable direction is an additive, default-preserving selector rather than changing the existing topic default globally.
