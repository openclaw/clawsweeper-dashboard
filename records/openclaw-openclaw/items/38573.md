## Summary

Keep open: current main still routes resolved thread/topic isolated cron fallback delivery through the direct outbound path and exposes no public selector for announce/main-agent-turn routing, so this remains a feature/API gap that needs maintainer design.

## What This Changes

Request an opt-in config or per-cron route selector so text-only isolated cron completions to thread/topic targets can use main-session announce/agent-turn behavior instead of forced direct outbound delivery.

## Best Possible Solution

Add a backward-compatible explicit route selector for text-only isolated cron completions to thread/topic targets, keep direct delivery as the default, leave structured payloads direct, and document/test failure fallback semantics across protocol, CLI, tool, docs, and UI surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/API request. Source inspection gives high confidence that current main still selects direct delivery when `resolvedDelivery.threadId` is set and public cron surfaces lack an opt-in route selector.

## Solution Assessment

Unclear until maintainers choose the public option shape and fallback policy. The maintainable direction is an additive, default-preserving route selector rather than changing topic/thread delivery globally.
