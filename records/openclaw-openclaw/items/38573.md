- `impact:session-state`: The request is about whether topic-bound cron output participates in the main session turn flow so users can continue chatting with the result.

## Summary

Keep open: current main and v2026.5.12 still force topic/thread cron completions through direct outbound delivery, and the requested opt-in announce/main-agent route needs an additive cron contract plus maintainer decisions on naming, scope, and fallback behavior.

## What This Changes

Add a config or per-cron option so text-only topic/forum cron completions can opt into main-session announce/agent-turn routing instead of mandatory direct delivery.

## Best Possible Solution

Add a documented additive selector for text-only isolated cron completion routing, preserve direct outbound delivery as the default, keep structured payloads direct, and define failure fallback semantics before implementation.

## Reproduction Assessment

Yes, at source level: a Telegram :topic: target becomes threadId, and dispatch sends any non-null threadId through direct delivery while the cron type and gateway schema have no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves current defaults instead of removing the threadId direct-delivery guard outright.
