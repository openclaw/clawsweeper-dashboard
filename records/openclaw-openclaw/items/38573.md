## Summary

Keep open: current main and v2026.6.5 still force threaded cron completions through direct outbound delivery, while the public cron schema has no opt-in route through the target main-session agent turn. The request remains a valid core product/configuration decision rather than conservative cleanup.

## What This Changes

Allow text-only isolated cron completions targeting topics or threads to opt into main-session announce/agent-turn processing instead of forced direct delivery.

## Best Possible Solution

Add one narrowly scoped per-job routing field for text-only announce completions, retain direct delivery as the default, keep structured payloads direct, and define a single deterministic failure outcome with regression coverage for threaded delivery, silence, and duplicate prevention.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 deterministically choose direct delivery whenever the resolved cron destination has a thread ID. A live Telegram run is unnecessary to establish that routing decision, though it would be appropriate proof for a future PR.

## Solution Assessment

Unclear. An opt-in route is appropriate, but the proposed three-level configuration and multiple fallback modes are broader than necessary; maintainers should first choose one per-job contract and deterministic failure behavior.
