## Summary

Keep open: current main still forces resolved thread/topic cron deliveries through direct outbound delivery, and neither the cron type, Gateway schema, CLI/docs, nor the latest release expose an opt-in text-only route through the main-session announce/agent-turn path. This remains a product/API decision rather than cleanup.

## What This Changes

The issue requests a backward-compatible cron option that lets topic/forum text completions use main-session announce/agent-turn routing instead of always using direct delivery.

## Best Possible Solution

Add a documented, backward-compatible selector for text-only isolated cron completion routing, keep current direct topic delivery and structured payload handling as defaults, and define failure fallback semantics before implementation.

## Reproduction Assessment

Yes, at source level: topic targets resolve to threadId, dispatch sends any non-null threadId through direct delivery, and the cron type/schema/docs expose no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public selector and fallback policy. The narrow maintainable direction is an additive selector that preserves current defaults instead of removing the threadId direct-delivery guard outright.
