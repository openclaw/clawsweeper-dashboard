## Summary

Keep open: current main and the latest release still have no public cron selector for routing text-only isolated cron completions from thread/topic targets through a main-agent turn, and the remaining decision is product/API shape plus fallback semantics. The related subagent announce issue was closed by later work, but that did not add this cron delivery option.

## What This Changes

Request an additive cron delivery option so text-only isolated cron completions for thread/topic targets can opt into main-session agent-turn handling instead of forced direct outbound delivery.

## Best Possible Solution

Add a documented additive selector for text-only isolated cron completion delivery, keep direct outbound delivery as the default, keep structured payloads direct, and define explicit failure/fallback semantics before implementation.

## Reproduction Assessment

Yes for the current limitation: a Telegram :topic: target resolves to threadId, and current dispatch sends any non-null threadId through direct delivery while the cron schema has no accepted opt-in selector. I did not run a live Telegram cron delivery flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves the current default and updates cron types, gateway schema, docs, and focused delivery tests together.
