## Summary

Keep open: current main and v2026.5.12 still force thread/topic isolated cron completions through direct delivery, and the cron contract still has no opt-in route selector for main-session announce/agent-turn handling. The related subagent issue was fixed, but that work does not cover this cron-specific selector.

## What This Changes

Add an opt-in cron delivery mode or config so text-only thread/topic completions can use main-session announce/agent-turn routing instead of forced direct delivery.

## Best Possible Solution

Add an additive, documented selector for text-only isolated cron completions to thread/topic targets, preserve direct delivery as the default, keep structured payloads direct, and define explicit fallback semantics.

## Reproduction Assessment

Yes for the current limitation from source: a Telegram :topic: target resolves to threadId, and dispatchCronDelivery sends any non-null threadId through direct delivery. I did not run a live Telegram cron delivery flow.

## Solution Assessment

Unclear until maintainers choose the public option shape and fallback policy. The narrow maintainable direction is an additive selector that preserves the current direct-send default rather than changing topic/forum behavior globally.
