## Summary

Keep open: current main still forces cron deliveries with a resolved thread/topic id through direct outbound delivery and does not expose a supported route selector or fallback policy for text-only topic/forum deliveries.

## What This Changes

The issue requests a backward-compatible cron delivery option that lets text-only thread/topic targets use the announce/main-agent-turn route instead of the current forced direct-send path.

## Best Possible Solution

Add an explicit, backward-compatible public route selector for text-only isolated cron completions targeting thread/topic destinations, with documented fallback semantics and aligned type, protocol, CLI/tool, docs, and regression-test coverage.

## Reproduction Assessment

Yes for the current limitation: source inspection shows any resolved threadId forces direct delivery, and the public cron type/protocol/CLI/tool surfaces have no supported opt-in route selector. A live Telegram forum cron run is not needed to verify the missing API surface.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is reasonable, but the maintainable fix should be an additive opt-in selector with explicit fallback behavior rather than changing the existing direct-delivery default.
