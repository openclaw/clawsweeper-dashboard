## Summary

Keep open: current main and the latest release still force resolved thread/topic cron deliveries through the direct outbound path, and there is no cron type, Gateway schema, CLI/docs, or config selector for opting text-only topic completions into the main-session announce/agent-turn flow.

## What This Changes

The issue requests an opt-in cron delivery mode that lets text-only thread/topic completions use the main-session announce/agent-turn path instead of forced direct delivery.

## Best Possible Solution

Add a backward-compatible, documented selector for text-only isolated cron completion routing, keep structured payloads and current direct topic delivery as the default, and define failure fallback semantics before implementation.

## Reproduction Assessment

Yes, at source level: topic targets resolve to a threadId, dispatch sends any non-null threadId through direct delivery, and the cron type/schema/docs expose no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves current defaults instead of removing the threadId direct-delivery guard outright.
