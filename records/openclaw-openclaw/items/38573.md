## Summary

Keep open: current main and v2026.5.12 still force topic/thread cron completion delivery through direct outbound send, and adding an opt-in announce/main-agent route is an additive product/API decision rather than an implemented cleanup candidate.

## What This Changes

Requests a configurable cron delivery mode so text-only isolated cron completions for topic/forum thread targets can opt into announce/main-agent-turn handling instead of always using direct delivery.

## Best Possible Solution

Add a documented additive selector for text-only isolated cron completion delivery, keep direct outbound delivery as the default, keep structured payloads direct, and define fallback semantics before implementation.

## Reproduction Assessment

Yes, for the source-level limitation: a Telegram :topic target becomes threadId, and dispatch sends any non-null threadId through direct delivery while the cron type and gateway schema have no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves the current default instead of removing the threadId direct-delivery guard outright.
