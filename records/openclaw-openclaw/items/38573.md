## Summary

Keep open: current main and latest release still have no cron delivery selector for topic/forum text completions, and the remaining work is an additive public cron/API behavior decision rather than a stale or already-implemented cleanup item.

## What This Changes

The issue requests an opt-in per-job or config mode so topic/forum isolated cron completions can route through a main-session announce/agent-turn flow instead of always using direct outbound delivery.

## Best Possible Solution

Add a backward-compatible, documented selector for text-only isolated cron completion routing, keep structured payloads and current direct topic delivery as the default, and define failure fallback and trust-boundary semantics before implementation.

## Reproduction Assessment

Yes, at source level: Telegram :topic: targets become threadId, and dispatch sends any non-null threadId through direct delivery while the cron type and Gateway schema have no opt-in selector. I did not run a live Telegram cron flow.

## Solution Assessment

Unclear until maintainers choose the public API shape and fallback policy. The narrow maintainable direction is an additive selector that preserves current defaults instead of removing the threadId direct-delivery guard outright.
