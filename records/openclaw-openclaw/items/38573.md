## Summary

Keep open. Current main and v2026.6.10 still force thread/topic cron completions through direct outbound delivery, and the merged target-awareness work only makes delivered content available to later turns; the requested opt-in main-agent rewrite/handoff route remains a public cron routing feature needing maintainer product/API choice.

## What This Changes

Request to add an opt-in cron delivery mode/config so text-only isolated cron results for thread/topic targets can go through main-session announce/agent-turn instead of forced direct send.

## Best Possible Solution

Define one explicit job-level text-completion routing enum for thread/topic cron delivery, default it to direct delivery, preserve target session/thread identity, and specify delivery status plus fallback behavior for silent, failed, and partially delivered main-agent turns.

## Reproduction Assessment

Yes. Source inspection on current main shows dispatchCronDelivery uses direct delivery whenever resolvedDelivery.threadId is present, and the cron type/schema/config surfaces expose no opt-in main-session announce-agent-turn selector.

## Solution Assessment

Unclear. The opt-in direction is plausible, but maintainers still need to choose the durable API shape and silent, failed, partial, and fallback semantics before this becomes a safe implementation task.
