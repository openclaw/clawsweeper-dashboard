## Summary

Keep open: the reported secondary-agent model lookup bug is well supported, but this PR is not merge-ready because inherited catalog parsing can mutate local provider request config before duplicate inherited models are filtered, letting default-agent auth/header metadata override a secondary agent's local setup.

## What This Changes

The PR adds default-agent generated plugin model catalog inheritance for secondary-agent model discovery, threads the inherited agent dir through cached discovery, fingerprints inherited catalogs, and adds ModelRegistry tests.

## Best Possible Solution

Keep the inheritance behavior, but make inherited catalog reads side-effect-free or apply inherited request metadata only for models that are actually inherited while preserving local root and local plugin provider config as authoritative.

## Reproduction Assessment

Yes. Current-main source shows plugin catalog rows are only loaded from the active agent dir, and the PR supplies live before/after terminal proof for a secondary Google/Gemini agent with a model-less local catalog.

## Solution Assessment

No, not yet. The inheritance direction is plausible and focused, but the implementation must avoid mutating local provider request config while merely inspecting inherited catalogs.
