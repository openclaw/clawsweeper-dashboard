## Summary

Keep open. Current main has adjacent provider cooldown/model fallback and a narrow subagent orphan-recovery tombstone, but it still lacks the requested gateway/session-level circuit breaker that persists consecutive unhealthy-session failures, marks the session unhealthy, suppresses same-session retries or resumes after a threshold, and exposes operator recovery controls.

## What This Changes

Request a gateway/runtime session-level circuit breaker that bounds retries on unhealthy persistent sessions after consecutive failures, emits visible unhealthy-session state, and optionally rotates or resumes safely.

## Best Possible Solution

Define a durable session-health contract in the gateway/session lifecycle: record consecutive failure signatures and retry exposure per session, reset on successful model output or explicit operator action, suppress automatic same-session retry/resume when tripped, surface the state through sessions RPC/status/UI/logs, and integrate with provider cooldown/model fallback without changing strict user-selected model behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; this is primarily a feature request and the linked observed retry storm does not include a runnable trace here. Source inspection does give high-confidence evidence that current main lacks the requested persistent session breaker state.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose scope. A session-level breaker is directionally sound, but the safest path is to reuse lifecycle/fallback telemetry and review or narrow #42933 rather than adding an isolated retry wrapper.
