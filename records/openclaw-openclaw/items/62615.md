## Summary

Keep open. Current main and the latest release have targeted safeguards for idle timeouts, post-compaction tool loops, tool-call repetition, and terminal failover suspension, but they still do not provide the requested first-class session-health breaker with failure receipts, unhealthy-session state, recovery actions, and configurable policy.

## What This Changes

The issue requests a gateway/runtime session circuit breaker that stops retrying unhealthy sessions after repeated failures and surfaces clear recovery state.

## Best Possible Solution

Keep this issue open as the canonical session-level breaker discussion and have maintainers define the session-health receipt, thresholds, defaults, recovery actions, and UI/log/API surfaces before implementation.

## Reproduction Assessment

Yes for the missing capability by source inspection: current main and the latest release have narrower guards but no broad first-class unhealthy-session breaker. No live retry storm was reproduced in this read-only review.

## Solution Assessment

Unclear. A counter beside the idle-timeout breaker is plausible, but the best solution needs maintainer direction on durable receipt fields, thresholds, defaults, UI/log surfacing, and resume versus fresh-session recovery semantics.
