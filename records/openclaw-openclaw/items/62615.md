## Summary

Keep open. Current main and the latest release contain narrower safeguards for idle timeouts, post-compaction tool loops, tool-call repetition, and session suspension, but they still do not provide the requested first-class gateway/session-health breaker with failure receipts, unhealthy-session state, configurable policy, and recovery actions.

## What This Changes

The issue requests a gateway/runtime session-level circuit breaker that bounds repeated unhealthy-session failures and surfaces operator-visible recovery state.

## Best Possible Solution

Define one owner-approved core session-health path with structured failure receipts, bounded retry admission, persistence or cooldown semantics, and clear resume/start-fresh recovery before implementation.

## Reproduction Assessment

Yes for the source-level gap: current main lacks the requested broad session-health breaker and only has narrower guards. No live retry-storm workflow was reproduced in this read-only review.

## Solution Assessment

Unclear. A counter beside the idle-timeout breaker is plausible, but the best fix needs maintainer direction on receipt fields, thresholds, defaults, persistence, UI/log surfacing, and resume versus fresh-session recovery semantics.
