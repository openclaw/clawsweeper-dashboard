## Summary

Keep open: current main still preserves the core remote-node approval/token boundary reported here. Node device-token scopes can still be empty by design, `nodes approve` still uses the separate `node.pair.*` request store, node-role token rotation remains admin-gated, and the companion-side evidence in https://github.com/openclaw/openclaw-windows-node/issues/710 confirms the workflow is still confusing and partly unresolved.

## What This Changes

The issue reports remote node registration gaps where node tokens get empty scopes, node approval request IDs are not accepted, and node-role token rotation is denied.

## Best Possible Solution

Keep this issue open until maintainers define the remote-node approval contract across device role approval, node-surface approval, node-role token rotation, CLI guidance, and Windows Companion UX, then land the chosen gateway/client/docs path with security-focused proof.

## Reproduction Assessment

Yes, at source level: current main shows empty node device-token scopes, separate device-pairing and node-pairing request stores, admin-gated node-role token rotation, and empty effective node surfaces before node-pairing approval. I did not run a fresh live Windows Companion remote wss:// reproduction.

## Solution Assessment

Unclear. The current implementation is coherent as a security boundary, but the reported user workflow needs a maintainer decision on whether device role approval should guide or bridge node-surface approval, how `nodes approve` should recover from mismatched IDs, and who may rotate node-role tokens.
