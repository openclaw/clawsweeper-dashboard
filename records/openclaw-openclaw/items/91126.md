## Summary

Keep open: current main and v2026.6.8 still leave the reported remote-node behavior as an active auth/security product boundary, not a completed cleanup. Node device tokens can intentionally have empty scopes, `openclaw nodes approve` targets the separate `node.pair.*` store, and node-role device-token rotation remains admin-gated.

## What This Changes

The issue asks for remote WebSocket node registration to produce usable node capability/token state, expose or approve node-specific pairing requests, and allow node-role token rotation where appropriate.

## Best Possible Solution

Have maintainers define the remote-node approval contract across device pairing, node-surface approval, and node-role token rotation, then update gateway, CLI, docs, and Windows Companion validation around that contract.

## Reproduction Assessment

Yes for the source-level symptoms: current main shows empty node token scopes, a separate `node.pair.*` approval namespace, admin-gated node-role rotation, and empty effective node surfaces before node-pairing approval. A full live Windows Companion remote wss:// reproduction was not run in this read-only review.

## Solution Assessment

Unclear; the narrowest safe solution depends on maintainer direction for whether device role=node approval should imply node capability approval, whether `nodes approve` should bridge device-pairing requests, and who may rotate node-role tokens.
