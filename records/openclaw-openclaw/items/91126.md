## Summary

Keep open: current main explains several reported symptoms as documented security boundaries, but the fresh remote Windows node workflow still needs maintainer security/product direction; the merged pending-reapproval diagnostics help discovery but do not settle the node scope or approval contract.

## What This Changes

The issue asks OpenClaw to fix remote node registration where node tokens have empty scopes, `openclaw nodes approve` cannot approve the observed request ID, and node-role device-token rotation is denied.

## Best Possible Solution

Define the remote node pairing contract explicitly: either keep empty node token scopes and admin-gated rotation as the security boundary with clearer Windows/client guidance, or add a narrowly tested approval path for node capabilities and token scopes without bypassing node command authorization.

## Reproduction Assessment

Partly. Current source clearly reproduces the reported empty node-scope possibility, `nodes approve` versus device-pair request mismatch, and non-admin node-role rotation denial, but the full live Windows Companion workflow was not reproduced in this review.

## Solution Assessment

Unclear. The narrowest maintainable solution depends on whether maintainers want device-paired WS nodes to remain separate from `node.pair.*` approval or to add an explicit bridge/migration for Windows node capability approval.
