## Summary

Keep open: current main and the latest release still preserve the split that produces the reported fresh remote-node state: node device tokens intentionally carry empty operator scopes, first-time node declarations remain non-effective until separate node-surface approval, and the narrower `unknown requestId` work covers only one symptom.

## What This Changes

The issue asks OpenClaw to make remote node registration complete for fresh Windows/Linux node clients by resolving empty node capabilities, node approval request matching, and node-role token rotation behavior.

## Best Possible Solution

Define the remote-node approval contract across device role approval, node-surface approval, node-role rotation, CLI recovery guidance, and companion UX, then land a security-reviewed gateway/client/docs change with fresh-pair live proof.

## Reproduction Assessment

Yes, at source level: current main still leaves first-time node declarations non-effective until separate node approval, keeps node role tokens scoped to `[]`, and returns `unknown requestId` from the node-pairing API when the request is not in that store. Recent comments add live v2026.6.9 Linux fresh-pair evidence, but I did not run a live remote-node repro myself.

## Solution Assessment

Unclear. The current implementation is internally consistent as a security boundary, but the user workflow needs a maintainer decision on whether device role approval should bridge into node-surface approval, how `nodes approve` should recover from mismatched IDs, and who may rotate node-role tokens.
