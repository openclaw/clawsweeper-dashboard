## Summary

Keep open: current main still preserves the security split behind the reported symptoms, and the latest v2026.6.9 Linux reproduction shows the fresh remote-node approval path still strands nodes with pending approval and empty effective caps/commands. Empty node token scopes are expected by the protocol, but the approval/list/rename/rotation recovery path is still actionable.

## What This Changes

The issue reports remote node registration failures where node tokens/surfaces end up empty and node-specific approval or rotation commands do not complete correctly.

## Best Possible Solution

Keep empty-scoped node tokens and explicit node-surface approval, but make fresh remote-node approval operator-visible and completable so approved nodes move from pending approval to approved capabilities without weakening the trust boundary.

## Reproduction Assessment

Yes, at source level: first-time node surfaces are declared but not effective until node pairing approval, and the latest shipped Linux report shows that approval transition still failing for fresh remote nodes. I did not run a live remote-node setup in this read-only review.

## Solution Assessment

Unclear: the current implementation protects the intended security boundary, but the user workflow still needs a maintainer decision on how device role approval, node-surface approval, operator visibility, and node-role rotation should fit together.
