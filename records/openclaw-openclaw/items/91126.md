## Summary

Keep open: current main still shows the reported remote-node approval boundary as unresolved product/security work, not conservative cleanup. The reporter's Windows Companion evidence matches source-level behavior: node device tokens can remain empty-scoped, `nodes approve` uses a separate `node.pair.*` store, and node-role device-token rotation is admin-gated.

## What This Changes

The issue asks OpenClaw to make remote node registration assign usable node authorization, expose node pairing requests through the expected approval API, and allow safe node-role token rotation.

## Best Possible Solution

Have maintainers define the remote-node approval contract across device pairing, node-surface approval, and node-role token rotation, then update gateway, CLI, docs, and companion validation around that contract.

## Reproduction Assessment

Yes for the source-level symptoms: current main shows empty node device-token scopes, separate device-pairing and node-pairing request stores, admin-gated node-role token rotation, and empty effective node surfaces before node-pairing approval. I did not run a fresh live Windows Companion remote wss:// reproduction.

## Solution Assessment

Unclear; the narrowest safe fix depends on maintainer direction for whether device role=node approval should imply node-surface approval, whether `nodes approve` should bridge device-pairing requests, and who may rotate node-role tokens.
