## Summary

Keep open: current main and v2026.6.9 still keep first-time node capabilities non-effective until separate node-pairing approval, while the reported Windows/Linux flows and public node docs still lead users through device approval as if that makes the node usable. The merged diagnostics and Android wait-state PRs help users see the pending state, and https://github.com/openclaw/openclaw/pull/94452 covers one CLI wording subset, but the approval-boundary and fresh remote-node usability problem remains.

## What This Changes

The issue asks OpenClaw to make remote node registration produce usable approved node capabilities, working node approval request handling, and node-token rotation/recovery instead of empty scopes/capabilities and `unknown requestId` failures.

## Best Possible Solution

Have maintainers choose the node approval model, then align gateway behavior, CLI/UI diagnostics, companion onboarding docs, and regression tests around either explicit node-surface approval or a narrow trusted bootstrap path that preserves high-risk command gates.

## Reproduction Assessment

Yes, at source level: a first-time node connect records declared caps/commands as pending but registers empty effective caps/commands, and `node.pair.approve` still returns `unknown requestId` when the pending ID is absent or claimed. I did not run a fresh cross-host Windows or Linux node setup live in this read-only review.

## Solution Assessment

Unclear until maintainers choose the approval boundary. Device approval alone would match parts of the user-facing docs, but explicit node-surface approval is the current security model and needs better recovery UX if it stays.
