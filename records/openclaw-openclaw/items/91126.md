## Summary

Keep open: current main still preserves the security split behind the reported symptoms, and fresh v2026.6.9 Linux evidence shows the issue remains after the diagnostics and Android wait-state PRs landed.

## What This Changes

The issue reports that remote node registration leaves node tokens with empty scopes, cannot approve node requests through `openclaw nodes approve`, and denies node-role token rotation.

## Best Possible Solution

Keep this issue open until maintainers define the remote-node approval contract across device role approval, node-surface approval, node-role token rotation, CLI recovery guidance, and companion/node-host UX, then land that path with security-focused tests and real node-host proof.

## Reproduction Assessment

Yes. The issue comments reproduce the symptom on Windows Companion v2026.6.1 and a fresh Linux v2026.6.9 node, and current source shows first-time node connects get empty effective caps/commands until a separate node-pairing approval exists.

## Solution Assessment

Unclear. The current implementation is coherent as a security boundary, but the user workflow needs a maintainer decision on whether device role approval should imply node-surface approval or whether `node.pair.*` should become the explicit recovery path for fresh WS nodes.
