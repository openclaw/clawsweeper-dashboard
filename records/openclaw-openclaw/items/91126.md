## Summary

Keep open for maintainer review: current main still leaves the reported remote-node behavior in a security-sensitive boundary between device pairing, node-surface approval, token scopes, and node-role rotation, and the same zero-capability symptom is only partially covered by a broader open node issue rather than fully resolved here.

## What This Changes

The issue reports that remote Windows node registration produces empty node token scopes, mismatched node approval request IDs, and denied node-role token rotation.

## Best Possible Solution

Resolve this through a maintainer-approved node/device pairing contract: either document and recover the required device plus node capability approvals, or change gateway token/rotation semantics with focused security tests while preserving high-risk command gates.

## Reproduction Assessment

Yes for the source-level symptoms: current code can mint empty node token scopes for scope-less node requests, nodes approve reads the separate node.pair store, and unapproved node surfaces register empty effective capabilities. No live current-main Windows remote reproduction was run here.

## Solution Assessment

Unclear: directly assigning node scopes or making nodes approve accept device-pairing IDs would change a security boundary. The safer solution is a maintainer-chosen pairing contract with gateway and Windows proof.
