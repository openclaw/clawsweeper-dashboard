## Summary

Keep open: this is a protected maintainer-labeled draft that changes public presentation SDK/channel contracts, and source review found a concrete Telegram budget bug in the proposed adapter.

## What This Changes

This PR adds channel-advertised presentation limits, adapts presentation blocks before renderer execution, declares limits for Slack/Discord/Telegram/Feishu/Matrix, and updates SDK docs/tests around legacy interactive APIs.

## Best Possible Solution

Keep the centralized adapter direction, but make budget accounting match each renderer's native output and land only after maintainer review of the public SDK contract and channel limit values.

## Reproduction Assessment

Yes, source-reproducible: a Telegram presentation with a 100-option select plus one button passes the adapter because the select consumes one slot, while Telegram renders every option as an inline button.

## Solution Assessment

No as currently patched: central adaptation is the right architectural place, but the select budget must match renderer output before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count select options against native action budgets:** `src/channels/plugins/outbound/presentation-limits.ts:337`
  - body: The adapter reserves only one action/row for each select, but Telegram renders every select option as an inline button and this PR declares Telegram `maxActions: 100`. A 100-option select plus one button can pass adaptation and still produce 101 native buttons, so the new limit pass can send an over-budget Telegram payload. Count rendered options against the shared action/row budget or keep Telegram selects out of this native path.
  - confidence: 0.88
