## Summary

Keep open: the presentation API direction is useful, but this protected-label PR is not merge-ready because two Telegram control-rendering correctness issues remain and the real behavior proof is still mock-only.

## What This Changes

The PR adds portable message-presentation capability limits, deprecates legacy interactive reply APIs, moves approval payloads to `presentation`, updates Slack/Telegram renderers, docs, SDK API baseline, changelog, and tests.

## Best Possible Solution

Land the presentation contract after Telegram override handling is non-null-aware, Telegram action budgeting matches native inline-button output, and redacted live channel proof demonstrates the visible controls.

## Reproduction Assessment

Yes, source-reproducible on the PR head: nullish `channelData.telegram.buttons` suppresses generated presentation controls, and a max-size Telegram select plus a sibling button can exceed the rendered inline-button budget. I did not run live Telegram or Slack proof in this read-only review.

## Solution Assessment

No as currently patched: central adaptation is the right layer, but the Telegram renderer contracts need to match nullish overrides and select option fan-out before this public API ships.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Treat Telegram button overrides as non-null only:** `extensions/telegram/src/outbound-adapter.ts:215-218`
  - body: `renderPresentation` treats any `telegram.buttons` key as an explicit override, even when the value is `null` or `undefined`. That skips generating buttons from `presentation`, and delivery later strips `presentation`, so a payload with an optional nullish buttons field loses all Telegram controls.
  - confidence: 0.9
- **[P2] Count Telegram select options against action budget:** `src/channels/plugins/outbound/presentation-limits.ts:337`
  - body: The adapter spends one action for a select, but Telegram renders each select option as its own inline button. With `maxActions` and `maxOptions` both at 100, a 100-option select plus another button passes adaptation and renders 101 inline buttons, so the advertised limit can still be exceeded.
  - confidence: 0.88
