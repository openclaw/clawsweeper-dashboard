## Summary

Keep open. Current main and latest v2026.6.6 still have the Tailscale parse behavior this replacement targets, and the patch is a narrow repair with focused tests; the remaining concern is maintainer acceptance of the intentional fail-closed behavior for malformed Funnel status output.

## What This Changes

The branch changes Tailscale helper parsing so noisy valid JSON is accepted for whois and Funnel status while malformed whois stays on the short error TTL path and malformed Funnel status is not treated as confirmed disabled state.

## Best Possible Solution

Land the replacement only if maintainers accept the intentional fail-closed behavior for malformed preserve-funnel status; otherwise return a typed status result so callers can distinguish exec failure, malformed JSON, and a confirmed missing route.

## Reproduction Assessment

Yes for source-level reproduction: mocked malformed stdout reaches current main's raw `ensureFunnel` parse and current main's catch-all Funnel route helper. I did not run a live Tailscale daemon repro because this is a read-only review.

## Solution Assessment

Yes, with one maintainer-facing caveat: accepting noisy valid JSON while preserving malformed parse failures is the narrowest fix for the earlier branch's empty-state bug. The preserve-funnel fail-closed behavior is the upgrade-safety decision to confirm before merge.
