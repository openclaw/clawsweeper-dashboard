## Summary

This PR addresses a real safe-bin profile modeling gap and now has relevant after-fix proof, but it should not merge yet: the new filter protects denied-option prefixes only, leaving prefixes of allowed value-taking options able to gain exact no-value semantics and make OpenClaw validate a different argv interpretation than the real binary.

## What This Changes

The PR adds a configurable `knownLongFlags` array to safe-bin profiles, merges it with long flags derived from allowed and denied options, filters strict prefixes of denied flags, and updates operator docs, generated config hashes, and policy tests.

## Best Possible Solution

Filter or reject user-provided entries that are strict prefixes of every auto-derived canonical long flag, not only denied flags, then add regression coverage proving value-taking, denied, ambiguous, and independent boolean options retain matching validator and binary semantics.

## Reproduction Assessment

Yes. With `allowedValueFlags: ["--output"]` and `knownLongFlags: ["--out"]`, the compiler retains both and exact-known precedence makes `--out` a no-value flag instead of resolving it to value-taking `--output`.

## Solution Assessment

No in its current form. Exposing independent boolean long-option metadata is a reasonable narrow solution, but its compiler must preserve the value semantics of all derived canonical options, not only denial precedence.
