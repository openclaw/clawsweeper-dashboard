## Summary

Keep open. The PR is aimed at a real Telegram delivery bug, but the current diff still has a merge-blocking multi-continuation delivery-state gap and the required real Telegram proof is still missing.

## What This Changes

The branch changes Telegram partial-stream final handling to treat failed or suppressed follow-up chunks as delivery failures, repair preview-finalized text from transcript-backed final text, and add focused regression tests.

## Best Possible Solution

Make streamed-final continuation delivery atomic from the fallback and delivered-state perspective, add later-continuation failure coverage, then attach redacted live Telegram proof.

## Reproduction Assessment

Yes at source level: a partial-stream final split into at least three chunks can have an early continuation mark the shared delivery state before a later continuation fails. I did not run a live Telegram reproduction, and that remains the proof gap.

## Solution Assessment

No: the direction is right, but the current implementation still lets continuation sends mark final delivery early. The safer fix is to make continuation delivery atomic from the shared delivered-state perspective before adding live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep continuation sends from marking final delivery early:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:367-370`
  - body: The patch delays `params.markDelivered()`, but each continuation still goes through `sendPayload`, and that callback marks the shared delivery state on any successful send. If a final splits into three chunks and the first continuation succeeds but the next one fails or is suppressed, the fallback path sees the reply as already delivered and can leave a truncated final visible. Make continuation sends avoid marking final delivery until all non-empty chunks succeed, and add that multi-continuation regression.
  - confidence: 0.87
