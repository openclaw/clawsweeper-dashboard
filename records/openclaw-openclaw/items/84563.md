## Summary

Keep open: the fix is focused, but it is not merge-ready because the delivery-state bug still survives multi-continuation finals and the external real Telegram proof gate is still unmet.

## What This Changes

The PR changes Telegram partial-stream final delivery to treat failed or suppressed continuation sends as errors, repair preview-finalized text from the transcript, and add targeted regression tests.

## Best Possible Solution

Make streamed-final continuation delivery atomic from the fallback/delivered-state perspective, add regression coverage for later continuation failure after an earlier continuation succeeds, then attach redacted live Telegram proof.

## Reproduction Assessment

Yes at source level: a partial-stream final split into at least three chunks can have the first continuation mark the shared delivery state before a later continuation fails. I did not run a live Telegram reproduction, and that remains the required proof gap.

## Solution Assessment

No: the proposed direction is right, but it only fixes the lane-local delivered mark and first-failing-continuation cases. The maintainable fix needs to stop continuation sends from marking final delivery early and prove the real Telegram fallback path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep continuation sends from marking final delivery early:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:367-370`
  - body: The patch delays `params.markDelivered()`, but each continuation still goes through `sendPayload`, and `sendPayload` marks the shared delivery state on any successful send. If a final splits into three chunks and the first continuation succeeds but the next one fails or is suppressed, `deliveryState.snapshot().delivered` is already true, so the fallback path is skipped and the user still sees a truncated final. Make continuation delivery avoid marking the final delivered until all non-empty chunks succeed, and add that multi-continuation regression.
  - confidence: 0.87
