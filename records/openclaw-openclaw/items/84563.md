## Summary

Keep open; the PR addresses a real Telegram partial-streaming message-loss path, but it still can mark a preview-finalized final as delivered when a follow-up send returns false, and the external PR has no real Telegram proof yet.

## What This Changes

The PR changes Telegram partial-stream final delivery handling and stuck-session diagnostics to avoid silent truncation and premature queued-work aborts.

## Best Possible Solution

Keep the one-preview Telegram design, but only mark delivery after every continuation chunk actually reports delivered and back it with regression coverage plus redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main marks Telegram streamed finals delivered before sending continuation chunks, and the PR's new test models the thrown-failure path. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet; moving markDelivered after successful follow-up work is the right direction, but the patch still ignores sendPayload returning false and lacks required real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Respect failed follow-up sends before marking delivered:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:367-369`
  - body: `params.sendPayload` returns `false` when delivery is skipped or canceled without throwing, but this loop discards that result and then calls `markDelivered()`. A continuation chunk that is not delivered can still suppress the fallback and leave the user with a truncated final, so treat any false follow-up as undelivered and cover it with a regression test.
