## Summary

Keep open. The Telegram change targets a real source-level partial-stream failure, but the PR lacks required after-fix real Telegram proof and introduces a diagnostic recovery regression that can leave queued embedded-run stalls unrecovered.

## What This Changes

This PR changes Telegram partial-stream finalization to delay delivered-state marking until follow-up chunks send, repairs truncated preview-finalized text from the session transcript, adds regression tests, and changes diagnostic active-abort eligibility for queued embedded work.

## Best Possible Solution

Keep the Telegram delivery-state and transcript-repair direction, fix or split out the diagnostic recovery regression, and require redacted live Telegram partial-stream proof before merge.

## Reproduction Assessment

Yes, at source level: current main marks a streamed final delivered before sending follow-up chunks, and the PR adds focused tests for that failure. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The Telegram fix direction is maintainable, but the diagnostic recovery guard needs a narrower recovery-safe shape or should be split, and the PR still needs real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve recovery for truly stalled queued embedded runs:** `src/logging/diagnostic.ts:490`
  - body: With this guard, a processing session with `queueDepth > 0` and terminal embedded-run progress is never active-abort eligible. That classification is `recoveryEligible: false`, so the heartbeat will no longer call `recoverStuckSession` even after the abort threshold; a truly stuck active run can keep the queued backlog stalled indefinitely. Please keep a bounded grace/progress check or another recovery path instead of suppressing all queued processing sessions.
  - confidence: 0.87
