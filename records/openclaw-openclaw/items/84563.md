## Summary

Keep this PR open: the Telegram delivery-state fix is source-backed and the current code-review pass did not find a new blocking patch defect, but this external transport PR still lacks after-fix real Telegram proof and includes an availability-sensitive diagnostics recovery policy change.

## What This Changes

The branch changes Telegram partial-stream final delivery to wait for successful continuation chunks, repair truncated preview-finalized text from transcript data, add regression tests, and suppress active abort recovery while queued embedded diagnostic work remains.

## Best Possible Solution

Land the Telegram delivery-state fix only after redacted live Telegram proof confirms the partial-stream final fallback or repair path, and either split the diagnostics abort-policy change or have a maintainer explicitly accept that recovery tradeoff.

## Reproduction Assessment

I did not run tests or a live Telegram scenario because this review is read-only and the remaining gate is real behavior proof.

## Solution Assessment

Mostly yes for the Telegram fix: moving markDelivered after successful continuation sends matches the one-preview streaming design. The diagnostics change is less clearly coupled, so the safer path is maintainer acceptance with proof or splitting that policy change into a separate review.
