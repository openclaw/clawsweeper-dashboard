## Summary

## What This Changes

The PR changes Telegram partial-stream final handling, adds transcript-backed repair for truncated preview finals, updates regression tests, and changes queued embedded-run diagnostic abort eligibility.

## Best Possible Solution

Keep final-delivery accounting separate from per-continuation sends until every Telegram continuation chunk succeeds, add a multi-continuation failure regression, then attach redacted real Telegram proof; split or explicitly accept the diagnostics recovery-policy change.

## Reproduction Assessment

Yes at source level: current main marks Telegram preview finals delivered before continuation chunks, and the PR path can still mark delivered via sendPayload after an intermediate continuation. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not yet: moving the lane-level markDelivered later is directionally right, but continuation sends still use a helper that marks deliveryState on each successful send. The safer fix is to avoid final delivered state until the entire continuation chain succeeds and then prove it in real Telegram.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep continuation sends out of delivered state until all finish:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:367`
  - body: `params.sendPayload` is the Telegram helper that marks `deliveryState` delivered whenever a send succeeds. With a final split into three chunks, the first follow-up can mark the turn delivered, a later follow-up can fail and be swallowed through `onError`, and the fallback gate then sees `delivered=true` and skips the visible error fallback. Please send continuation chunks through a path that does not mark the final delivered until all chunks complete, and add a regression where the second continuation fails.
  - confidence: 0.88
