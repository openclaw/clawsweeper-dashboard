## Summary

Keep open: current main supports literal and config-env-substituted MCP HTTP header strings, but it does not support SecretRef object values for `mcp.servers.*.headers.*`, and no related issue or PR owns that exact remaining surface.

## What This Changes

The issue requests SecretRef-backed values for remote MCP HTTP headers so authenticated MCP tokens do not need to be stored literally in `openclaw.json`.

## Best Possible Solution

Extend the canonical SecretInput surface for `mcp.servers.*.headers.*` only after maintainer approval, covering schema/types, SecretRef registry/audit/apply, runtime snapshot resolution before MCP client creation, docs/tests, and preserving literal string headers.

## Reproduction Assessment

Yes from source inspection: the schema and transport coercion accept only scalar MCP header values, and object SecretRefs are outside the current supported SecretRef credential matrix. I did not run the CLI because this review is read-only.

## Solution Assessment

Likely yes in direction, but it needs maintainer design review because adding a new SecretRef credential surface affects config validation, active-surface resolution, audit/apply docs, and runtime startup behavior.
