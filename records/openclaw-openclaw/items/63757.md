## Summary

Keep open. Current main still receives Telegram reaction updates and enqueues text system events, but it does not expose a stable structured internal or plugin hook payload for downstream feedback integrations; adjacent wake/semantics issues and PRs do not replace this hook-contract gap.

## What This Changes

Expose Telegram message reactions as a stable structured event or hook for downstream feedback and automation integrations.

## Best Possible Solution

Add an additive observation-only structured reaction hook/API, preferably generic across channels but emitted by Telegram after existing authorization and reactionNotifications filtering, with typed payloads, SDK/docs coverage, and focused tests.

## Reproduction Assessment

Yes, source-level. An authorized Telegram message_reaction reaches the current handler and can enqueue a system event, while the internal and plugin hook catalogs expose no structured reaction event to subscribe to.

## Solution Assessment

Yes directionally, but not as an ad hoc Telegram-only local patch. The maintainable solution is an additive typed hook contract whose payload and cross-channel scope are settled before Telegram emission is wired.
