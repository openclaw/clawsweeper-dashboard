## Summary

Keep open. Current main still receives Telegram reaction updates and turns them into text system events, but the internal hook and public plugin hook catalogs still have no stable structured reaction event for downstream feedback integrations.

## What This Changes

Expose Telegram message reactions as a stable structured internal/plugin hook for downstream feedback integrations instead of only text system events.

## Best Possible Solution

Define an additive observation-only structured reaction hook/API, preferably generic across channels, then emit it from Telegram only after the existing authorization and reactionNotifications filters pass, with SDK/docs/tests updated and reaction-wake behavior kept separate.

## Reproduction Assessment

Yes, source-level. An authorized Telegram message_reaction reaches the current handler and can enqueue a system event, while current internal and plugin hook catalogs expose no structured reaction event to subscribe to.

## Solution Assessment

Unclear until maintainers choose the public API shape. A typed additive reaction hook is the maintainable direction; an ad hoc Telegram-only local patch would risk drifting from the broader reaction-hook contract.
