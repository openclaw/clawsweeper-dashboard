## Summary

Keep open. Current main still receives Telegram `message_reaction` updates and turns them into text system events, but it does not expose a stable structured internal or plugin hook payload that downstream feedback integrations can consume; related reaction-wake and broad reaction-semantics items are adjacent, not replacements for this hook contract gap.

## What This Changes

Expose Telegram message reactions as a stable structured downstream hook/event for feedback and automation integrations.

## Best Possible Solution

Add an additive observation-only reaction event seam, preferably generic across channels but emitted by Telegram after existing authorization and `reactionNotifications` filtering, with typed payloads, SDK/docs coverage, and focused regression tests.

## Reproduction Assessment

Yes, source-level. An authorized Telegram `message_reaction` with `reactionNotifications: "all"` reaches the handler and enqueues text system events, while the internal and plugin hook catalogs expose no structured reaction event to subscribe to.

## Solution Assessment

Yes directionally, but not as an ad hoc Telegram-only local patch. The maintainable path is an additive typed hook contract whose payload and cross-channel scope are settled before wiring Telegram emission.
