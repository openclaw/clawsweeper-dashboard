## Summary

Keep open. Current main and the latest release still turn accepted Telegram reactions into text system events, while the internal and public plugin hook surfaces have no stable structured reaction event for downstream feedback integrations.

## What This Changes

Request a stable structured Telegram reaction hook or internal event so downstream managed hooks and plugins can consume reaction metadata without patching core.

## Best Possible Solution

Define an additive observation-only structured reaction hook/API, preferably generic across channels, then emit it from Telegram only after existing authorization and reactionNotifications filters pass, with SDK/docs/tests and live Telegram proof.

## Reproduction Assessment

Yes, source-level. An authorized Telegram message_reaction reaches the current handler and can enqueue a system event, while current internal and plugin hook catalogs expose no structured reaction event to subscribe to.

## Solution Assessment

Unclear until maintainers choose the public API shape. A typed additive reaction hook is the maintainable direction; an ad hoc Telegram-only local patch would risk drifting from the broader reaction-hook contract.
