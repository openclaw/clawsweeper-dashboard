## Summary

Keep open. Current main still receives Telegram reaction updates and turns added reactions into human-readable system-event text, but the internal hook catalog, plugin hook catalog, typed message hook payloads, docs, and tests still do not expose a stable structured reaction event that downstream feedback automation can consume.

## What This Changes

The issue requests a stable structured Telegram reaction hook/event that downstream managed hooks or plugins can consume without patching core.

## Best Possible Solution

Add an additive typed reaction event seam after the existing Telegram authorization and notification filtering. Prefer a cross-channel `message_reaction` plugin hook and/or `message:reaction` internal hook carrying channel/account/session context, target message id, emoji, add/remove operation, sender metadata, authorization outcome, and bounded raw metadata; keep feedback storage itself in a hook or plugin built on that seam.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: drive the Telegram `message_reaction` handler with `reactionNotifications: "all"` and it enqueues text system events, while the plugin/internal hook catalogs have no reaction event to subscribe to.

## Solution Assessment

Yes directionally, but the exact API should be designed as an additive stable hook contract rather than a Telegram-only local patch. The current text-only system-event path is not a maintainable downstream feedback interface.
