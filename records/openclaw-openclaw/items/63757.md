## Summary

Keep open. Current main still handles Telegram `message_reaction` updates by enqueueing text system events, but neither the internal hook surface nor the typed plugin hook catalog exposes a stable structured reaction event for downstream feedback automation; related reaction-wake and Discord hook items are adjacent, not a complete Telegram replacement.

## What This Changes

Expose Telegram reaction updates as a stable structured internal or plugin hook carrying message, emoji, session/account, and sender metadata for downstream feedback automation.

## Best Possible Solution

Add an additive observation-only reaction event seam, preferably cross-channel but emitted by Telegram only after existing authorization and `reactionNotifications` filtering, with typed payloads, docs, SDK exports, and focused tests.

## Reproduction Assessment

Yes, source-level. Driving an authorized Telegram `message_reaction` with `reactionNotifications: "all"` reaches the handler and enqueues text system events, while current internal and plugin hook catalogs expose no structured reaction event to subscribe to.

## Solution Assessment

Yes directionally, but not as an ad hoc local Telegram patch. The maintainable solution is an additive typed hook contract whose payload and cross-channel scope are settled before wiring Telegram emission.
