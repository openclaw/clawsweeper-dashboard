## Summary

Keep open. Current main still handles Telegram reaction updates only by enqueuing human-readable system-event text, while the internal hook and plugin hook catalogs do not expose a typed reaction event for downstream feedback automation. Solving this would expand a public hook/plugin API surface, so it needs maintainer API direction rather than conservative cleanup closure.

## What This Changes

Expose Telegram message reactions as a stable structured hook/event for downstream feedback and automation integrations.

## Best Possible Solution

Add an additive typed reaction event seam, preferably cross-channel, after existing Telegram authorization and notification filtering; keep feedback storage and reward logic in hooks/plugins built on that seam.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: drive the Telegram `message_reaction` handler with `reactionNotifications: "all"` and it enqueues text system events while the internal/plugin hook catalogs have no reaction event to subscribe to.

## Solution Assessment

Yes directionally, but the best solution is an additive stable hook contract rather than the local Telegram-only core patch described as the workaround. The contract shape should be decided before implementation because third-party plugins can consume this surface.
