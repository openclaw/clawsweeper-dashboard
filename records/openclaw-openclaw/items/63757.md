## Summary

Keep this issue open. Current main still handles Telegram `message_reaction` updates by authorizing/filtering them and enqueueing human-readable system-event text; the plugin hook catalog, message hook payloads, internal hook types, docs, and tests still do not expose a stable structured reaction hook for downstream feedback automation.

## What This Changes

The issue requests a stable structured Telegram message-reaction hook/event for downstream managed hooks or plugins instead of only text system events.

## Best Possible Solution

Add an additive typed reaction event seam after existing channel authorization and notification filtering. Prefer a cross-channel `message_reaction` plugin hook and/or `message:reaction` internal hook with channel/account/session context, target message id, emoji, add/remove operation, sender metadata, authorization result, and bounded raw metadata; keep feedback storage itself in a hook/plugin built on that seam.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: drive the existing Telegram `message_reaction` handler with `reactionNotifications: "all"` and it enqueues text system events, while the plugin/internal hook catalogs have no reaction event to subscribe to.

## Solution Assessment

No, the current text-only system-event path is not the best long-term solution for downstream feedback. The maintainable fix is an additive typed hook contract emitted from the existing authorized reaction path, rather than requiring consumers to parse prompt text or patch installed Telegram code.
