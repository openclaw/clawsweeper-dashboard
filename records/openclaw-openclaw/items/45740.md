## Summary

Keep open: current main and v2026.5.22 still require gh-issues worker prompts to include GitHub issue body text, and `sessions_spawn` still has no caller-supplied per-spawn child tool policy. The prior prompt-only fix was closed unmerged after review found a fence bypass and notification/tool-policy regression, so the remaining runtime hardening decision is still real.

## What This Changes

Harden gh-issues so untrusted GitHub issue/comment text is not injected into tool-enabled child-agent prompts without an enforced untrusted-data and child capability boundary.

## Best Possible Solution

Keep this open until maintainers choose a runtime-backed untrusted-content handoff for gh-issues plus an explicit child capability policy for unattended workers, or land a canonical security architecture that demonstrably covers this workflow with regression coverage for hostile issue/comment content.
