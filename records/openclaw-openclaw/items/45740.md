## Summary

Keep open: current main and v2026.6.6 still have the gh-issues workflow ask spawned workers to include raw issue body text, while sessions_spawn exposes only a raw task plus opt-in sandboxing rather than a runtime untrusted-content or per-spawn capability boundary. The earlier direct PR was closed as bypassable, and the current open prompt-guidance PR is useful defense-in-depth but not the runtime fix requested here.

## What This Changes

The issue requests hardening for gh-issues so untrusted GitHub issue bodies and review comments are not injected directly into spawned child-agent prompts.

## Best Possible Solution

Keep this as the canonical security-hardening item and choose a runtime or workflow boundary that wraps or summarizes GitHub issue and review-comment content as untrusted data before child task construction, with explicit child sandbox/tool-policy behavior that preserves notifications.

## Reproduction Assessment

Yes, source-reproducible: current main still asks gh-issues workers to include issue body text, and sessions_spawn accepts a raw task string without a per-call untrusted-content boundary. I did not run a live crafted issue through the workflow.

## Solution Assessment

No, the prior template-only fence is not the best complete solution because it was reviewed as bypassable and conflicted with notifications. The better path is a maintainer-approved runtime/workflow boundary plus explicit capability or sandbox policy.
