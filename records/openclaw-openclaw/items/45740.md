## Summary

Keep open. Current main and the latest shipped release still leave the gh-issues workflow depending on GitHub issue/review text becoming child prompt material, while the related tool-scoping work is open and only addresses part of the hardening story.

## What This Changes

The issue requests hardening gh-issues so untrusted GitHub issue and review bodies are not injected into tool-enabled sub-agent prompts without a runtime-backed data boundary and child capability policy.

## Best Possible Solution

Keep this open until gh-issues uses an approved runtime-backed untrusted-content handoff and a child capability policy that narrows tool-enabled unattended workers without breaking notifications.

## Reproduction Assessment

Yes for source-level reproduction: current main still requires gh-issues worker prompts to include issue body text, and sessions_spawn lacks a workflow-supplied untrusted-content or per-spawn tool-scope handoff. I did not execute a runtime exploit after the model receives that content.

## Solution Assessment

No, this is not solved by a SKILL-only fence: the prior template-only PR was closed for delimiter-bypass and notification/tool-policy regressions. The maintainable direction is runtime/API support for untrusted content plus explicit child capability policy.
