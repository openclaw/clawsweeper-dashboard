## Summary

Keep open: current main and v2026.6.10 still leave `gh-issues` requiring GitHub issue body text in spawned worker prompts, while the complete runtime/content-boundary and child capability decision has not landed. The related open PRs are useful partial mitigations, but they do not fully replace this gh-issues-specific security hardening tracker.

## What This Changes

The issue requests hardening for `gh-issues` so untrusted GitHub issue bodies and review-comment text are not injected into tool-enabled sub-agent prompts as trusted instructions.

## Best Possible Solution

Choose and land a maintainer-approved gh-issues untrusted-content boundary that keeps issue/comment text data-shaped and pairs it with sandbox or per-spawn child tool policy where needed; keep prompt warnings as defense-in-depth.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.10 shows `gh-issues` still requires worker prompts to include issue body text, while no complete gh-issues-specific runtime content boundary is present.

## Solution Assessment

Unclear for the final shape. The requested runtime boundary is the right layer, but maintainers still need to choose whether the durable solution is structured wrapping, summarization, sandbox requirements, per-spawn tool policy, or a combination.
