## Summary

Keep open: current main and the latest release still leave the gh-issues worker task as prompt text that includes the raw issue body, while sessions_spawn exposes only a raw task plus sandbox inheritance/requirement and no per-spawn untrusted-content or tool-scope boundary. The linked prompt-guidance PR is useful defense-in-depth, but it does not settle the runtime hardening requested here.

## What This Changes

The issue asks for hardened handling of untrusted GitHub issue and review-comment content before gh-issues spawns tool-enabled sub-agents.

## Best Possible Solution

Choose a maintainer-approved runtime boundary for gh-issues subagent task construction, then pair it with explicit child capability or sandbox policy for unattended runs while keeping prompt-only skill guidance as defense-in-depth.

## Reproduction Assessment

Yes, as a source-level reproduction: current main and v2026.6.6 still have gh-issues ask workers to include the issue body, and sessions_spawn accepts a raw task without a per-call tool-scope or untrusted-data boundary. I did not run a live prompt-injection exploit PoC.

## Solution Assessment

No complete fix is present yet. A SKILL.md warning is only a partial mitigation; the best solution needs a runtime wrapping/sanitization and child-capability decision that maintainers approve.
