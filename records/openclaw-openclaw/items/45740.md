## Summary

Keep open. Current main still passes raw GitHub issue bodies and review-comment JSON into `gh-issues` child-agent task prompts, while the available runtime controls are opt-in sandboxing and config-level subagent tool policy rather than a first-class untrusted-content boundary for this workflow.

## What This Changes

The issue asks for `gh-issues` to stop passing raw GitHub issue and review-comment bodies directly into child-agent prompts, especially in unattended cron mode.

## Best Possible Solution

Define a maintainer-approved untrusted external content boundary for `gh-issues` before child task construction, then pair it with an explicit unattended subagent sandbox/tool-scope policy and regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: `skills/gh-issues/SKILL.md` still substitutes `{body}` and `{json_array_of_actionable_comments}` into child task prompt strings. I did not establish a runtime PoC showing a security-boundary bypass after the model receives that content.

## Solution Assessment

Unclear because no merged fix exists. Prompt warnings would be a useful defensive layer, but the narrow maintainable direction is structured untrusted-data handling plus a maintainer decision on sandbox and tool scope for unattended child agents.
