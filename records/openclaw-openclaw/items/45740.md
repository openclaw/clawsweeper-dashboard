## Summary

Keep open: current main and the latest shipped tag still leave `gh-issues` dependent on GitHub issue text becoming child-task prompt material, while the related `sessions_spawn` tool-policy work is still open and only covers part of the requested hardening.

## What This Changes

The issue asks OpenClaw to stop passing untrusted GitHub issue and review-comment bodies directly into tool-enabled sub-agent prompts and to add runtime-backed isolation or capability controls.

## Best Possible Solution

Keep this open until gh-issues uses a runtime-backed untrusted-content handoff and an approved child capability policy, coordinated with the open `toolsAllow` PR and the broader per-spawn policy tracker.

## Reproduction Assessment

Yes, for a source-level reproduction: current main and `v2026.5.19` still require gh-issues worker prompts to include issue body text, and `sessions_spawn` still lacks a merged per-spawn tool-policy entry point. I did not run a live exploit after the model receives that content.

## Solution Assessment

No, a SKILL-only fence is not the complete fix because the discussion already identified delimiter bypass and notification/tool-policy conflicts. The maintainable direction is runtime/API support for untrusted content plus explicit child capability policy.
