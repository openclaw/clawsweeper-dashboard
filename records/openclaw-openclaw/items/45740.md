## Summary

Keep open. Current main still passes raw GitHub issue bodies and review-comment JSON into `gh-issues` child-agent prompts, while the available runtime controls are opt-in sandboxing and config-level subagent tool policy rather than a first-class untrusted-content boundary for this workflow.

## What This Changes

The issue requests hardening `gh-issues` so untrusted GitHub issue bodies and review comments are not injected raw into tool-enabled child-agent prompts.

## Best Possible Solution

Define a maintainer-approved untrusted external content boundary before `gh-issues` child task construction, then pair it with explicit unattended subagent sandbox/tool-scope policy and regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes. Static source inspection on current main is high confidence: `skills/gh-issues/SKILL.md` still substitutes `{body}` and `{json_array_of_actionable_comments}` into child task prompt strings. I did not establish a runtime PoC showing a security-boundary bypass after the model receives that content.

## Solution Assessment

Unclear because no merged fix exists. Prompt warnings would be a useful defensive layer, but the maintainable direction is structured untrusted-data handling plus a maintainer decision on sandbox and tool scope for unattended child agents.
