## Summary

Keep open: current main and the latest release still route gh-issues worker briefing through task prompt text that must include the GitHub issue body, while sessions_spawn still has no caller-supplied per-spawn child tool policy. The closed prompt-only PRs are useful context but do not resolve the runtime untrusted-content and capability-boundary decision.

## What This Changes

The issue requests hardening gh-issues so untrusted GitHub issue and review-comment text is not handed to tool-enabled subagents as ordinary prompt instructions.

## Best Possible Solution

Keep this open until maintainers choose a runtime-backed untrusted-content handoff for gh-issues plus an explicit child capability policy for unattended workers, with regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still require gh-issues worker prompts to include issue body text, and sessions_spawn still lacks a caller-supplied child tool-policy field. I did not run a live exploit after model ingestion.

## Solution Assessment

No, a SKILL-only warning is not the best complete solution because prior prompt-only attempts were bypassable and could break notifications. The maintainable direction is runtime/API support for treating external content as data plus an explicit child capability policy.
