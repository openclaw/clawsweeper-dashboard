## Summary

Keep open. Current main and the latest release still require gh-issues workers to receive GitHub issue body text in the spawned worker prompt path, while sessions_spawn still lacks a caller-supplied untrusted-content or per-spawn tool-policy handoff that would make this a completed runtime fix.

## What This Changes

The issue asks OpenClaw to prevent untrusted GitHub issue bodies and review comments from being injected directly into gh-issues child-agent prompts without runtime-backed isolation or capability scoping.

## Best Possible Solution

Keep this open until maintainers choose and implement a runtime-backed untrusted-content handoff for gh-issues, with explicit child capability policy for unattended workers and tests covering hostile issue/comment content.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.20 still require gh-issues worker prompts to include issue body text, and sessions_spawn has no workflow-supplied untrusted-content or per-spawn tool-scope parameter. I did not run a live exploit after model ingestion.

## Solution Assessment

No, a SKILL-only fence is not the best complete solution because it remains prompt guidance and prior discussion identified delimiter-bypass and notification/tool-policy tradeoffs. The maintainable fix is runtime/API support for untrusted content plus an explicit child capability policy.
