## Summary

Keep open: current main and latest release still leave the gh-issues worker prompt carrying GitHub issue body text, while the runtime lacks a complete untrusted-content boundary or per-spawn child tool policy. The related open PRs are useful partial paths, but they are not merged and do not fully replace this gh-issues-specific security hardening tracker.

## What This Changes

The issue requests hardening so untrusted GitHub issue bodies and review comments are not passed as raw prompt instructions to tool-enabled gh-issues subagents.

## Best Possible Solution

Choose a maintainer-approved runtime boundary for gh-issues task construction, then pair it with child capability or sandbox policy for unattended runs while treating SKILL.md warnings as defense-in-depth.

## Reproduction Assessment

Yes, as a source-level reproduction: current main still requires gh-issues workers to include issue bodies in prompt text, and sessions_spawn has no per-call untrusted-content or child tool-scope boundary. I did not run a live prompt-injection exploit PoC.

## Solution Assessment

No complete fix is present yet. Prompt-only guidance and the open toolsAllow implementation are useful pieces, but the best fix still needs a maintainer-approved runtime wrapping/sanitization and child-capability decision for this workflow.
