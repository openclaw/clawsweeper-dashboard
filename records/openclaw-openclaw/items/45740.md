## Summary

Keep open. Current main still leaves the gh-issues workflow dependent on issue/review text becoming worker prompt material, while the runtime does not yet expose a merged untrusted-data plus child-capability handoff; the SKILL-only fix attempts were closed as bypassable or incomplete, and the related toolsAllow work is still an open implementation candidate rather than shipped behavior.

## What This Changes

The issue asks OpenClaw to stop passing untrusted GitHub issue and review-comment text directly into gh-issues sub-agent prompts without a runtime-backed untrusted-data or capability boundary.

## Best Possible Solution

Define a runtime-backed untrusted-data handoff for gh-issues, coordinate it with the canonical per-spawn tool-policy work in https://github.com/openclaw/openclaw/issues/15032 and https://github.com/openclaw/openclaw/pull/78441, and add hostile-content regression coverage; prompt-template fences should remain defense-in-depth, not the whole fix.

## Reproduction Assessment

Yes for source-level reproduction: current main still has gh-issues require issue body text in spawned worker prompts and the sessions_spawn contract lacks a workflow-supplied untrusted-content or per-spawn tool-scope handoff. I did not reproduce a runtime exploit or concrete boundary bypass after the model receives that content.

## Solution Assessment

No for a SKILL-only fence as the complete solution: both target-repo prompt-template attempts were closed with delimiter/CDATA bypasses or notification/tool-policy regressions. The maintainable direction is runtime/API support for untrusted content plus explicit child capability policy.
