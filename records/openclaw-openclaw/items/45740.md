## Summary

Keep open: current main and the latest release still leave gh-issues worker tasking dependent on GitHub issue body text in spawned worker prompts, and the runtime still lacks a workflow-supplied child tool-policy contract for this path. The linked prompt-only attempts are useful context but do not resolve the runtime untrusted-content and capability-boundary decision.

## What This Changes

The issue requests hardening so untrusted GitHub issue bodies and review comments are not directly treated as child-agent prompt instructions in the gh-issues workflow.

## Best Possible Solution

Keep this open until maintainers choose a runtime-backed untrusted-content handoff for gh-issues plus an explicit child capability policy for unattended workers, with regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still require gh-issues worker prompts to include issue body text, and sessions_spawn still lacks a caller-supplied child tool-policy field. I did not run a live exploit after model ingestion.

## Solution Assessment

No, a SKILL-only warning is not the best complete solution because the prior prompt-only attempt was bypassable and conflicted with notifications. The maintainable direction is runtime/API support for treating external content as data plus an explicit child capability policy.
