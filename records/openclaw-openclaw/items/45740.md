## Summary

Keep open. Current main still embeds untrusted GitHub issue bodies and review-comment JSON directly into `gh-issues` child-agent task prompts, while the runtime controls available today are opt-in sandboxing, config-level subagent tool policy, and reusable untrusted-content helpers that are not wired into this skill. The prior target-repo PR was closed unmerged, and the linked fork PR is only a partial prompt-warning layer rather than the runtime boundary described in the issue.

## What This Changes

Request hardening for `gh-issues` so untrusted GitHub issue bodies and review comments are not injected raw into tool-enabled sub-agent prompts.

## Best Possible Solution

Define a maintainer-approved runtime boundary for automation skills that ingest external GitHub content: sanitize, escape, summarize, or pass issue/comment text as untrusted data before child task construction, then decide whether unattended `gh-issues` runs should require sandboxed target agents, named tool profiles, or per-spawn capability constraints. Add regression coverage for hostile issue/comment content and any `sessions_spawn` policy changes.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: `skills/gh-issues/SKILL.md` still substitutes `{body}` and `{json_array_of_actionable_comments}` into child task prompt strings. There is not a high-confidence runtime PoC showing the model crosses a security boundary after receiving that content.

## Solution Assessment

Unclear because no merged fix exists. The prompt-warning/fence approach in the closed target PR and linked fork PR is a useful first layer, but the more maintainable solution is structured untrusted-data handling plus an explicit sandbox/tool-scope policy decision for unattended subagents.
