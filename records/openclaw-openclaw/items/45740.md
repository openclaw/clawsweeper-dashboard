## Summary

Keep open: current main and the latest release still have gh-issues instruct workers to include issue body text in spawned prompts, while sessions_spawn still accepts a plain task string with only opt-in sandboxing and no runtime untrusted-content or per-spawn capability boundary. The related direct PR was closed as bypassable, and the open prompt-guidance PR is useful defense-in-depth but not the runtime/workflow fix requested here.

## What This Changes

The issue reports that the gh-issues skill passes untrusted GitHub issue and review-comment text into spawned sub-agent prompts without a runtime isolation or capability boundary.

## Best Possible Solution

Keep this as the canonical security-hardening item and choose a maintainer-approved workflow/runtime boundary for untrusted GitHub bodies and review comments before child task construction.

## Reproduction Assessment

Yes, from source inspection: gh-issues still tells the workflow to include issue bodies/review comments in worker prompts, and sessions_spawn accepts the delegated work as a plain task string that becomes child prompt content. I did not execute a live crafted issue through the automation.

## Solution Assessment

No, the prior template-only fence is not the best complete solution because it was reviewed as bypassable and conflicted with notifications. The better path is a maintainer-approved runtime or workflow boundary plus explicit capability or sandbox policy.
