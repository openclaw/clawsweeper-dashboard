## Summary

Keep open: current main and the latest release still have the gh-issues workflow ask workers to include raw issue bodies in spawned prompts, while sessions_spawn only exposes a raw task string plus optional sandboxing. The related closed PR was explicitly abandoned as bypassable, and the open prompt-guidance PR is not a complete runtime fix.

## What This Changes

The issue requests hardening so the gh-issues skill does not pass untrusted GitHub issue bodies and review comments directly into spawned sub-agent prompts.

## Best Possible Solution

Keep this issue as the canonical maintainer/security decision item and implement a runtime or workflow boundary that wraps or summarizes GitHub issue and review-comment content as untrusted data before child task prompt construction, with explicit sandbox/tool-policy behavior that preserves notifications.

## Reproduction Assessment

Yes, source-reproducible: current main still asks gh-issues workers to include issue body text, and sessions_spawn accepts a raw task string without a per-call untrusted-content boundary. I did not run a live crafted issue through the workflow.

## Solution Assessment

No, the prior template-only fix is not the best complete solution because it was reviewed as bypassable and conflicted with notifications. The better path is a maintainer-approved runtime/workflow boundary plus explicit capability or sandbox policy.
