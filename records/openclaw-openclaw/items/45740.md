## Summary

Keep open. Current main and the latest shipped tag still leave gh-issues dependent on GitHub issue/review text becoming child-task prompt material, while the shipped sessions_spawn contract still lacks a merged workflow-supplied child tool policy; the closed SKILL-only attempt and the open toolsAllow PR are useful context, not a completed fix.

## What This Changes

The issue asks OpenClaw to harden gh-issues so untrusted GitHub issue bodies and review comments are not passed directly into tool-enabled sub-agent prompts without runtime isolation or capability constraints.

## Best Possible Solution

Keep this open until gh-issues uses a runtime-backed untrusted-content handoff and an approved child capability policy, coordinated with the open sessions_spawn toolsAllow work and the broader per-spawn policy tracker.

## Reproduction Assessment

Yes for source-level reproduction: current main still has gh-issues require issue body text in spawned worker prompts, and sessions_spawn lacks a workflow-supplied untrusted-content or per-spawn tool-scope handoff. I did not reproduce a runtime exploit after the model receives that content.

## Solution Assessment

No for a SKILL-only fence as the complete solution: the prior template-only PR was closed with delimiter-bypass and notification/tool-policy regressions. The maintainable direction is runtime/API support for untrusted content plus explicit child capability policy.
