## Summary

Keep open. Current `main` and the latest release still place raw GitHub issue bodies and review-comment JSON into `gh-issues` child-agent prompts, while the runtime only has coarse sandboxing/inherited tool policy rather than a first-class untrusted-content boundary for this workflow.

## What This Changes

The issue asks for hardening the `gh-issues` skill so untrusted GitHub issue and review-comment content is not injected directly into tool-enabled child-agent prompts.

## Best Possible Solution

Define a maintainer-approved untrusted external content boundary before `gh-issues` child task construction, then pair it with explicit unattended child sandbox/tool-scope policy and regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes, source inspection is high confidence: current `main` still substitutes `{body}` and `{json_array_of_actionable_comments}` directly into child task prompt strings. I did not establish a runtime PoC showing a security-boundary bypass after the model receives that content.

## Solution Assessment

Unclear because there is no merged fix and prompt-only patches have already shown bypass/regression problems. The safest direction is runtime-backed untrusted-data handling plus a maintainer decision on sandbox and per-spawn tool scope, not only another SKILL.md fence.
