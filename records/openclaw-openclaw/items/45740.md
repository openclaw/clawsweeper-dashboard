## Summary

Keep open. Current main removed the old long template, but the `gh-issues` workflow still tells workers to include the issue body in the spawned worker prompt, and the runtime/tool-policy work requested by the discussion is not implemented.

## What This Changes

Harden the `gh-issues` automation path so untrusted GitHub issue and review text is not injected into tool-enabled child-agent prompts without a runtime boundary.

## Best Possible Solution

Define a runtime-backed untrusted-data handoff for `gh-issues`, then pair it with explicit child sandbox/tool-scope policy and hostile-content regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main still requires the issue body to be included in the worker prompt, and v2026.5.12 shows the older direct body/comment interpolation. I did not establish a runtime exploit showing a boundary bypass after the model receives that content.

## Solution Assessment

Unclear until maintainers choose the runtime/API contract. A SKILL-only fence is not the best complete fix because the issue discussion already identifies fence bypass and notification/tool-policy tradeoffs; the maintainable path is runtime untrusted-data handling plus explicit child capability policy.
