## Summary

Keep open: current main and v2026.5.22 still make GitHub issue body text part of gh-issues worker prompt material, and the shipped `sessions_spawn` schema still lacks a merged per-spawn child tool policy. The prior prompt-only PRs were closed unmerged after concrete fence-bypass and notification/tool-policy concerns, while the open toolsAllow PR would only cover one layer of the remaining runtime hardening.

## What This Changes

The issue requests hardening for the gh-issues skill so untrusted GitHub issue and review comment bodies are not injected directly into tool-enabled sub-agent prompts.

## Best Possible Solution

Keep this open until maintainers choose and land a runtime-backed untrusted-content handoff for gh-issues plus an explicit child capability policy for unattended workers, with regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still fetch GitHub issue bodies, require gh-issues worker prompts to include body text, and pass delegated task text into the child prompt. I did not run a live model exploit.

## Solution Assessment

No for a SKILL-only warning: prior prompt-only attempts were bypassable and could break notifications. The maintainable direction is runtime/API support for treating external content as data plus an explicit child capability policy.
