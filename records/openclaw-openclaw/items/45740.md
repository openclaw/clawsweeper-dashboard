## Summary

Keep open: current main and v2026.5.22 still instruct `gh-issues` workers to include issue body text in the subagent task, and main still has no merged per-spawn tool allow/deny API. The prior prompt-only fix at https://github.com/openclaw/openclaw/pull/45748 closed unmerged after concrete bypass and notification-regression concerns, while https://github.com/openclaw/openclaw/pull/78441 only covers part of the child tool-policy layer.

## What This Changes

The issue requests hardening `gh-issues` automation so untrusted GitHub issue and review text is not passed into spawned subagent prompts without isolation, sanitization, or child capability controls.

## Best Possible Solution

Keep this open until maintainers land a runtime-backed untrusted-content handoff for `gh-issues` and an explicit child capability policy for unattended workers, likely composing with the open toolsAllow work rather than relying on prompt-template warnings.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still tell `gh-issues` worker prompts to include issue body text, and native subagent startup appends delegated task text into the child prompt. I did not run a live exploit.

## Solution Assessment

No for the earlier SKILL-only path: the closed PR showed why fences alone and an unsupported `toolAllowList` are not enough. The safer solution is maintainer-approved runtime/API support for untrusted content and child capability boundaries.
