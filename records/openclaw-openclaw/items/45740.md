## Summary

Keep open. Current main still has `gh-issues` carrying issue body text into spawned worker prompts, while `sessions_spawn` exposes sandboxing and inherited/configured tool policy rather than a workflow-specific untrusted-data and child-capability handoff; the prior target-repo PR was closed as incomplete and bypassable.

## What This Changes

Harden `gh-issues` so untrusted GitHub issue and review-comment bodies are not passed into tool-enabled sub-agent prompts without runtime isolation or sanitization.

## Best Possible Solution

Define a runtime-backed untrusted-data handoff for `gh-issues`, align child sandbox/tool-scope policy with the per-spawn restriction work in https://github.com/openclaw/openclaw/issues/15032, and add hostile-content regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main still requires issue body text in spawned worker prompts and lacks a workflow-supplied untrusted-content or tool-scope handoff. I did not reproduce a runtime exploit or boundary bypass after the model receives that content.

## Solution Assessment

No for a SKILL-only fence as the complete fix: the closed target-repo PR showed delimiter bypass and notification/tool-policy conflicts. The maintainable direction is a runtime/API contract for untrusted content plus explicit child capability policy.
