## Summary

Keep open: current main and v2026.6.6 still leave the gh-issues worker task as prompt text containing the issue body, while sessions_spawn has no runtime untrusted-content field or per-call child tool policy. The linked prompt-guidance and toolsAllow work are useful partial paths, but neither has landed as the complete runtime hardening requested here.

## What This Changes

Harden gh-issues automation so untrusted GitHub issue/comment content is handled as data and spawned child agents can be constrained for unattended processing.

## Best Possible Solution

Choose a maintainer-approved runtime boundary for gh-issues task construction, then pair it with child capability or sandbox policy for unattended runs while treating SKILL.md warnings as defense-in-depth.

## Reproduction Assessment

Yes, as a source-level reproduction: current main and v2026.6.6 still require gh-issues workers to include issue bodies in prompt text, and sessions_spawn has no per-call untrusted-content or child tool-scope boundary. I did not run a live prompt-injection exploit PoC.

## Solution Assessment

No complete fix is present yet. Prompt-only warnings and an open per-spawn toolsAllow PR are useful pieces, but the best fix needs a maintainer-approved runtime wrapping/sanitization and child-capability decision for this workflow.
