## Summary

Keep open: current main and v2026.5.22 still require gh-issues worker prompts to include issue body text, and current sessions_spawn still lacks a merged per-spawn tool policy. The prior prompt-only fix was closed unmerged for concrete bypass and notification-regression concerns, while the open toolsAllow PR only covers one layer of the remaining runtime hardening.

## What This Changes

The issue requests runtime hardening so gh-issues does not pass untrusted GitHub issue/review text into spawned worker prompts as trusted prompt content and can constrain child tools.

## Best Possible Solution

Keep this issue open until maintainers choose and land a runtime-backed untrusted-content handoff for gh-issues plus an explicit child capability policy for unattended workers, likely composing with the open toolsAllow work rather than relying on SKILL-only warnings.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still require gh-issues worker prompts to include issue body text, and native subagent startup places delegated task text into the child prompt. I did not run a live model exploit.

## Solution Assessment

No for a SKILL-only warning: the prior prompt-only attempt was bypassable and could break notifications. The maintainable direction is runtime/API support for treating external content as data plus an explicit child capability policy.
