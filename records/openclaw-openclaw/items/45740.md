## Summary

The issue remains necessary: current main still routes GitHub issue bodies into unattended, tool-capable worker prompts without requiring the existing external-content wrapper or sandbox enforcement, while the linked open PR adds only advisory prompt text. The report is best treated as security hardening rather than a confirmed vulnerability because no authorization, policy, or sandbox boundary bypass is demonstrated.

## What This Changes

Harden the unattended gh-issues workflow so untrusted issue and review-comment text is isolated from instructions and processed by constrained sub-agents.

## Best Possible Solution

Add a runtime-supported untrusted-data boundary and least-capability child-execution contract, then require unattended gh-issues workers to use both while preserving completion notifications.

## Reproduction Assessment

Yes at the source level: current main fetches issue bodies and directs the orchestrator to include them in spawned worker prompts, including confirmation-free cron mode. No live security-boundary bypass has been demonstrated.

## Solution Assessment

No, a static prompt warning alone is not the best solution; the durable fix needs runtime-supported data isolation plus an explicit constrained child-execution policy that preserves required workflow actions.
