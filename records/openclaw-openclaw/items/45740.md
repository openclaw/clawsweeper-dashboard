## Summary

Keep open: current main and the latest release still make gh-issues worker briefing include GitHub issue body text, and the spawn/runtime path still needs a maintainer-approved contract for untrusted-content handoff plus child capability scoping. The closed prompt-only PR and open related policy PRs are useful context, but they do not yet solve this issue on main.

## What This Changes

The issue requests hardening for gh-issues handling of untrusted GitHub issue and review-comment text passed into spawned sub-agent prompts.

## Best Possible Solution

Keep this open until maintainers choose a runtime-backed untrusted-content handoff for gh-issues and an explicit child capability policy for unattended workers, with regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still require gh-issues worker prompts to include issue body text, and sessions_spawn still lacks a caller-supplied child tool-policy field. I did not run a live exploit after model ingestion.

## Solution Assessment

No, a SKILL-only warning is not the best complete solution because prior prompt-only attempts were bypassable and could break notifications. The maintainable direction is runtime/API support for treating external content as data plus an explicit child capability policy.
