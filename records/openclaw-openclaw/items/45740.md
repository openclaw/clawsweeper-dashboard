## Summary

Keep open: current main and v2026.5.22 still route GitHub issue body text into gh-issues worker prompt material, and the merged `sessions_spawn` schema still has no per-spawn child tool policy. The linked SKILL-only fix was closed unmerged because reviewers found concrete fence-bypass and notification-regression problems, while the open `toolsAllow` PR only addresses one layer of the remaining runtime hardening.

## What This Changes

The issue asks OpenClaw to harden gh-issues automation so untrusted GitHub issue bodies and review comments are treated as data, not directly trusted sub-agent prompt instructions.

## Best Possible Solution

Keep this open until maintainers choose and land a runtime-backed untrusted-content handoff for gh-issues plus an explicit child capability policy for unattended workers, with regression coverage for hostile issue/comment content.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.22 still fetch GitHub issue bodies, require gh-issues worker prompts to include body text, and place delegated task text into the child prompt. I did not run a live model exploit.

## Solution Assessment

No for a SKILL-only warning: prior prompt-only attempts were bypassable and could break notifications. The maintainable direction is runtime/API support for treating external content as data plus an explicit child capability policy.
