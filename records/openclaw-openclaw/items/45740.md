## Summary

Keep open. Current main still leaves `gh-issues` carrying issue body text into spawned worker prompts, while `sessions_spawn` provides `sandbox` plus inherited/configured tool policy rather than a workflow-specific untrusted-data and child-capability handoff; the prior target-repo PR attempts were closed as incomplete or bypassable.

## What This Changes

The issue requests hardening for `gh-issues` so untrusted GitHub issue bodies and review comments are not injected directly into tool-enabled child prompts.

## Best Possible Solution

Define a runtime-backed untrusted-data handoff for `gh-issues`, then pair it with explicit child sandbox/tool-scope policy and hostile-content regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current `main` still requires issue body text in spawned worker prompts, and `v2026.5.12` shows the older direct body/comment interpolation. I did not reproduce a current-main runtime exploit or boundary bypass after the model receives that content.

## Solution Assessment

No for a SKILL-only fence as the complete fix: prior target PRs showed delimiter bypasses and notification/tool-policy conflicts. The maintainable solution is a runtime/API contract for untrusted content plus explicit child capability policy.
