## Summary

Keep open: current main and v2026.6.10 still leave `gh-issues` worker tasks requiring issue body text in spawned prompts, while the runtime content-boundary and child capability decisions remain unmerged. The linked PRs are useful partial mitigations, but none fully replaces this issue as the gh-issues hardening tracker.

## What This Changes

The issue asks OpenClaw to harden the gh-issues skill so untrusted GitHub issue and review-comment content is not injected directly into spawned sub-agent prompts.

## Best Possible Solution

Choose and land a maintainer-approved gh-issues untrusted-content boundary that keeps issue/comment text data-shaped and pairs it with sandbox or per-spawn child tool policy where needed; keep prompt warnings as defense-in-depth.

## Reproduction Assessment

Yes, at source level. Current main still requires `gh-issues` worker prompts to include issue body text, and the shipped `sessions_spawn` tool schema still lacks a caller-selected per-spawn child tool restriction field.

## Solution Assessment

Unclear for the final shape. The issue correctly points at runtime/content-boundary hardening rather than a template-only warning, but maintainers still need to choose the durable model for structured content, summarization, sandboxing, or per-spawn policy.
