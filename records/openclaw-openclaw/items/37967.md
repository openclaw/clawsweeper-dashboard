## Summary

Keep open: current main and v2026.6.8 provide adjacent Feishu sender/topic-scoped sessions, DM-only dynamic agents, and explicit Feishu thread-bound subagent delivery, but they do not implement automatic persistent group-sender subagents with isolated workspaces, skills, config, and memory. The KinthAI comments add useful production constraints around lifecycle, warm pools, cost tracking, and shared/private memory, which reinforces that this needs maintainer product and security design before code.

## What This Changes

The issue requests Feishu group-chat routing that automatically creates or reuses one isolated subagent per sender or role, with per-user skills, config, memory, and admin controls.

## Best Possible Solution

Define a generic, administrator-authorized identity-to-agent binding and lifecycle contract first, then implement optional Feishu group-sender provisioning with quotas, SQLite-backed state, memory isolation, deterministic routing, and security/audit coverage.

## Reproduction Assessment

Yes, at source level. Current main shows sender/topic-scoped Feishu group sessions and DM-only dynamic agent creation, while automatic persistent group-sender agent provisioning with isolated workspace, skills, config, and memory is absent.

## Solution Assessment

No, not as proposed. A Feishu-only implicit subagent factory is too broad for a first implementation; the safer path is a generic authorized identity-to-agent binding and lifecycle contract, then an optional Feishu implementation.
