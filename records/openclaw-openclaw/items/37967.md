## Summary

Keep open: current main still has adjacent Feishu sender/topic-scoped group sessions, DM-only dynamic agents, and explicit Feishu subagent thread bindings, but not the requested automatic persistent group-sender subagents with isolated workspaces, skills, config, memory, lifecycle, and quotas.

## What This Changes

Request Feishu group-chat routing that dynamically provisions each sender as an isolated subagent with its own workspace, skills, configuration, memory, and lifecycle.

## Best Possible Solution

Define a maintainer-approved identity-to-agent binding and lifecycle contract first, then implement optional Feishu group-sender provisioning with quotas, SQLite-backed state, memory isolation, deterministic routing, and security/audit coverage.

## Reproduction Assessment

Yes, at source level. Current main gates dynamic agent creation to non-group Feishu messages and persists only direct-peer dynamic bindings; group routes can be sender/topic scoped, but they do not create persistent per-sender subagents with isolated workspace, skills, config, and memory.

## Solution Assessment

No, not as an implicit one-off Feishu subagent factory. The safer path is a generic administrator-authorized identity-to-agent binding and lifecycle contract, followed by an optional Feishu implementation using existing group session and conversation-binding seams.
