## Summary

Keep open. Current main has sender-scoped Feishu group sessions, DM-only dynamic agent creation, and explicit subagent/thread binding, but it still does not automatically provision each Feishu group sender or role as an isolated `__mc_*` subagent with separate skills, config, memory, lifecycle, quotas, and admin controls; the KinthAI comments are useful prior art, not a merged OpenClaw implementation.

## What This Changes

The issue requests Feishu group-chat routing that dynamically creates per-sender or per-role subagents with isolated skills, config, memory, lifecycle, and admin controls.

## Best Possible Solution

Design an opt-in Feishu plugin mode, adding generic plugin SDK seams only if needed, that maps sender or role identity to bounded isolated agent profiles with memory namespaces, cleanup/quotas, admin controls, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug reproduction; source inspection gives a high-confidence feature-gap check because group messages route to group peer ids while dynamic agent creation is DM-only and Feishu subagent binding requires explicit spawning.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Per-sender subagents may solve isolation, but the maintainable path should be opt-in and designed around Feishu identity, memory boundaries, lifecycle, quotas, and admin authority.
