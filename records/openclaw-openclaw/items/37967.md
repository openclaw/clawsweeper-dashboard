## Summary

Keep open: current main and v2026.5.22 still only provide adjacent Feishu group sender/topic session scoping, DM-only dynamic agent creation, and explicit thread-bound subagent hooks, not automatic Feishu group per-sender/per-role agents with isolated skills, config, memory, lifecycle, quotas, routing, and admin controls. The KinthAI comments are useful external prior art for memory tiers, lifecycle, warm pools, and cost tracking, but they are not an OpenClaw implementation.

## What This Changes

Request automatic Feishu group-chat subagents per sender or role, with isolated skills, configuration, memory, routing, lifecycle, quotas, and admin controls.

## Best Possible Solution

Design an opt-in Feishu plugin mode that maps verified sender or role identity to bounded agent profiles with explicit memory namespaces, lifecycle cleanup, quotas, admin authorization, tests, docs, and redacted live Feishu group proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks automatic Feishu group per-sender or per-role subagent provisioning despite adjacent session scoping, DM dynamic agents, and manual thread-bound subagent support.

## Solution Assessment

Unclear. Per-sender or per-role subagents are plausible, but the maintainable path needs an opt-in Feishu identity, memory, lifecycle, quota, and admin-authority design rather than a narrow routing patch.
