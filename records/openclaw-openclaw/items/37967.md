## Summary

Keep open. Current main has sender-scoped Feishu group sessions, DM-only dynamic agent creation, and explicit thread-bound subagent sessions, but it still does not automatically provision each Feishu group sender or role as an isolated subagent with separate skills, config, memory, lifecycle, quotas, and admin controls; the KinthAI comments are useful implementation prior art, not a merged OpenClaw implementation.

## What This Changes

Request dynamic per-sender or per-role Feishu group-chat subagents with isolated skills, config, memory, lifecycle, and admin controls.

## Best Possible Solution

Design an opt-in Feishu plugin mode, adding generic plugin SDK seams only if needed, that maps sender or role identity to bounded isolated agent profiles with memory namespaces, cleanup/quotas, admin controls, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source inspection shows current main routes Feishu groups to sender/topic peer ids and supports DM dynamic agents plus explicit subagent thread binding, but does not auto-create isolated group-sender subagents.

## Solution Assessment

Unclear. Per-sender subagents may be a good direction, but the maintainable solution should be opt-in and designed around Feishu identity, memory boundaries, lifecycle, quotas, and admin authority rather than a narrow routing-only patch.
