## Summary

Keep open for product and security review: current main still lacks automatic persistent per-sender Feishu group agents, while the requested design crosses routing, authorization, memory, agent provisioning, and lifecycle boundaries that require an explicit maintainer decision.

## What This Changes

Add automatic per-sender or per-role Feishu group agents with isolated workspaces, skills, configuration, memory, lifecycle management, and reply routing.

## Best Possible Solution

Keep sender-scoped group sessions for context isolation, then design persistent per-user agents as an operator-approved generic provisioning and binding contract with explicit identity mapping, access profiles, quotas, cleanup, auditing, and non-conflicting delivery semantics.

## Reproduction Assessment

Yes, at source level: current main demonstrably offers sender/topic session isolation, DM-only dynamic agents, and explicit direct/topic subagent binding, but not automatic persistent per-sender group agents.

## Solution Assessment

No, not as proposed; a Feishu-owned automatic agent hierarchy duplicates existing routing and binding responsibilities and lacks settled authorization, lifecycle, storage, quota, and delivery contracts.

## Risks / Open Questions

- Creating persistent agents and rewriting configuration from inbound group identities needs explicit authorization, quotas, cleanup, and operator-approval semantics.
- A main-bot relay or orchestrator could compete with current route resolution and conversation bindings, causing duplicate, suppressed, or misdirected replies.
- Sender or role identity is not an OpenClaw authorization boundary; granting distinct skills, tools, models, workspace access, or memory requires a defined trust and sandbox policy.
- The requested hot configuration, role mapping, lifecycle, cost controls, shared/private memory tiers, and admin commands make this broad multi-area product work rather than one bounded Feishu fix.
