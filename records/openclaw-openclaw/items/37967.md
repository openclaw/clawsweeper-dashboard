## Summary

Keep open for maintainer product and security review: current main has Feishu sender/topic session isolation and DM-only dynamic agents, but still does not provide automatic persistent per-sender Feishu group subagents with separate skills, config, memory, lifecycle, and delivery semantics.

## What This Changes

The issue requests automatic Feishu group-chat routing to user-specific subagents with isolated workspace, skills, configuration, memory, and admin-managed lifecycle.

## Best Possible Solution

Design a generic, operator-approved provisioning and binding contract for persistent per-sender agents, then implement the Feishu plugin path on top of it with explicit trust, quota, lifecycle, memory, and delivery semantics.

## Reproduction Assessment

Yes, source-level: current main can be checked by following Feishu group dispatch through groupSessionScope and the !isGroup dynamic-agent guard. This proves group senders get session routing only, not automatic persistent per-sender agents.

## Solution Assessment

No, not as proposed yet. Automatic Feishu-owned agent hierarchies need a maintainer-approved generic provisioning, trust, lifecycle, and delivery design before implementation should be scoped.

## Risks / Open Questions

- Creating durable agents from inbound group identities needs an explicit authorization model, access profile mapping, audit trail, quota, and cleanup policy.
- A relay/orchestrator path could compete with existing Feishu bindings, topic sessions, broadcast routing, and completion delivery, causing duplicate, suppressed, or misdirected replies.
- Sender-scoped session isolation does not grant separate tool, workspace, credential, model, or host permissions; treating it as a security boundary would conflict with the documented gateway trust model.
- The requested admin commands, hot configuration, shared/private memory tiers, cost controls, and lifecycle behavior make this broad multi-area design work rather than a narrow Feishu bug fix.
