## Summary

Keep open. Current main has adjacent Feishu sender-scoped group routing, DM-only dynamic agent creation, and explicit Feishu subagent/thread binding, but it does not automatically provision and route each Feishu group sender or role to an isolated `__mc_*` subagent with separate workspace, skills/config, memory, lifecycle, quotas, and admin controls. KinthAI's comments are useful external prior art, not a merged OpenClaw implementation or canonical replacement.

## What This Changes

The issue requests dynamic Feishu group-chat subagents per sender or role, with isolated skills, configuration, memory, and admin controls.

## Best Possible Solution

Design an opt-in Feishu plugin mode, adding generic plugin SDK seams only if needed, that maps sender or role identity to bounded isolated agent profiles with memory namespaces, cleanup/quotas, admin controls, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug reproduction; source inspection gives a high-confidence feature-gap check because group messages route to group peer ids while dynamic agent creation is DM-only and subagent binding requires explicit spawning.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Per-sender subagents may solve isolation, but the maintainable path should be opt-in and designed around Feishu identity, memory boundaries, lifecycle, quotas, and admin authority.
