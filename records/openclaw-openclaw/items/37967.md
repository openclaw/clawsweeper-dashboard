## Summary

Keep open. Current main has adjacent Feishu sender/topic-scoped group sessions, DM-only dynamic agent creation, and thread-bound subagent hooks, but it still does not provide automatic Feishu group per-sender or per-role subagents with isolated skills, config, memory, lifecycle, quotas, and admin controls; the KinthAI comments are useful design evidence rather than a superseding implementation.

## What This Changes

The issue requests Feishu group-chat routing that dynamically provisions one isolated subagent per sender or role, with separate skills, config, memory, lifecycle, and admin controls.

## Best Possible Solution

Design an opt-in Feishu plugin mode that maps verified sender or role identity to bounded isolated agent profiles with explicit memory namespaces, lifecycle cleanup, quotas, admin authorization, regression tests, and redacted live Feishu group proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main has adjacent group session scoping, DM dynamic agents, and thread-bound subagent hooks, but no automatic group-sender or role subagent provisioning to reproduce as broken existing behavior.

## Solution Assessment

Unclear. Per-sender subagents may be a good direction, but the maintainable solution needs an opt-in design for Feishu identity, memory boundaries, lifecycle, quotas, and admin authority rather than a routing-only patch.
