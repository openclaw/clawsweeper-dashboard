## Summary

Current main already addresses the original shared-context concern with Feishu group sessions scoped by sender or topic-plus-sender, and it can bind explicitly spawned ACP/subagent runs to Feishu conversations. The remaining request—automatically provisioning a persistent agent, workspace, skills, configuration, and memory for every group sender—is still missing and needs an explicit product and security decision rather than conservative cleanup.

## What This Changes

Add automatic per-sender Feishu group-chat agents with isolated skills, configuration, workspaces, memory, lifecycle, and reply routing.

## Best Possible Solution

Keep sender-scoped Feishu sessions as the isolation baseline, then design a generic, administrator-authorized identity-to-agent binding contract with explicit lifecycle, quotas, canonical SQLite persistence, shared-versus-private context rules, and plugin-facing routing hooks before enabling optional Feishu auto-provisioning.

## Reproduction Assessment

Yes at the source level. Current main demonstrably supports per-sender group sessions and conversation-bound subagents, while automatic persistent agent creation for each group sender is absent; this is a capability gap rather than a failing existing contract.

## Solution Assessment

No. Feishu-only implicit provisioning combines identity, authorization, persistence, skills, lifecycle, and message ownership; a generic authorized binding contract followed by an optional Feishu implementation is narrower and safer.

## Risks / Open Questions

- Automatically creating durable agents for untrusted group senders could allocate workspaces, credentials, skills, model access, and compute without an explicit administrator authorization boundary.
- Persistent identity-to-agent mappings need quotas, expiration, cleanup, rename/deletion handling, recovery, and canonical SQLite ownership; an in-memory or file-backed lifecycle would not meet repository state policy.
- Mention, reply, topic, broadcast, and ambiguous-message cases need one deterministic ownership contract or replies can be routed to the wrong personalized agent.
- Hot per-agent skill and configuration changes introduce a new public configuration and permission surface with upgrade and audit implications.
