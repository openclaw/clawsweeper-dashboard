## Summary

Keep open. Current main has adjacent Feishu pieces: sender-scoped group session keys, DM-only dynamic agent creation, and explicit Feishu subagent/thread bindings, but it still does not automatically provision and route every Feishu group sender or role to an isolated `__mc_*` subagent with separate workspace, skills/config, memory, lifecycle, quotas, and admin controls. The KinthAI comments are useful prior art for memory isolation, lifecycle, warm pools, routing, and cost attribution, but they are not a merged OpenClaw implementation or canonical replacement.

## What This Changes

Request dynamic Feishu group-chat routing that creates per-sender or per-role subagents with isolated skills, config, memory, and admin-managed lifecycle.

## Best Possible Solution

Design an opt-in Feishu plugin mode or plugin/API extension that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory namespaces, routes replies back to the originating group/topic, and includes admin controls, lifecycle cleanup, quota/cost tracking, and isolation tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug reproduction, but source inspection gives a high-confidence feature-gap check: Feishu group messages resolve to sender-scoped group peer ids while dynamic agent creation is guarded by `!isGroup` and direct peer bindings.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Automatic subagent creation for every group sender may solve isolation, but the maintainable path should be opt-in and designed around lifecycle, memory, quotas, admin authority, and Feishu group security policy.
