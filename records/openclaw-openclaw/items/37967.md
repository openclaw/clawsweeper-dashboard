## Summary

Keep this open. Current main has adjacent Feishu capabilities, including sender-scoped group session keys, DM-only dynamic agent creation, and explicit Feishu conversation binding for ACP/subagent runs, but it still does not automatically create or route each Feishu group sender or role to an isolated `__mc_*` subagent with separate workspace, skills/config, memory, lifecycle, and admin controls. The KinthAI comments are useful prior-art for memory isolation, lifecycle, warm pools, routing, and cost attribution, but they are not a merged OpenClaw implementation or a canonical replacement.

## What This Changes

The issue requests Feishu group chat support that dynamically maps each sender or role to an isolated per-user subagent with its own skills, config, memory, workspace, routing, and lifecycle controls.

## Best Possible Solution

Design an opt-in Feishu plugin mode or plugin/API extension that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory namespaces, routes replies back to the originating group/topic, and includes admin controls, lifecycle cleanup, quota/cost tracking, and isolation tests. If maintainers want this outside the bundled Feishu plugin, first add the stable Feishu routing/provisioning seam needed for an external plugin to own the behavior.

## Reproduction Assessment

Yes. A Feishu group message with `groupSessionScope: "group_sender"` or `"group_topic_sender"` currently resolves to a sender-scoped group peer route, while dynamic agent creation is guarded by `!isGroup`, so the requested automatic per-user subagent creation is not exercised.

## Solution Assessment

Unclear. The user problem is real, but automatically spawning a subagent for every group sender is not yet proven as the narrowest maintainable OpenClaw solution; an opt-in Feishu design with explicit lifecycle, memory, admin, and security policy is safer than a broad default behavior change.
