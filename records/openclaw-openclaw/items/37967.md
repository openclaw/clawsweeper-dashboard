## Summary

Keep open: current main and v2026.6.10 provide adjacent Feishu sender/topic-scoped group sessions, DM-only dynamic agents, and explicit Feishu subagent thread bindings, but they still do not implement automatic persistent group-sender subagents with isolated workspaces, skills, config, memory, lifecycle, and quotas. The KinthAI comments are useful production evidence for lifecycle, warm pools, cost tracking, and shared/private memory, which reinforces that this needs maintainer product and security design before code.

## What This Changes

The issue requests Feishu group-chat routing that creates persistent per-sender subagents with isolated skills, configuration, memory, and lifecycle management.

## Best Possible Solution

Define a maintainer-approved identity-to-agent binding and lifecycle contract first, then implement optional Feishu group-sender provisioning with quotas, SQLite-backed state, memory isolation, deterministic routing, and security/audit coverage.

## Reproduction Assessment

Yes, at source level. Current main shows sender/topic-scoped Feishu group sessions and DM-only dynamic agent creation, while automatic persistent group-sender agent provisioning with isolated workspace, skills, config, and memory is absent.

## Solution Assessment

No, not as a one-off implicit Feishu subagent factory. The safer path is a generic administrator-authorized identity-to-agent binding and lifecycle contract, followed by an optional Feishu implementation that uses existing group session and conversation-binding seams.
