## Summary

Keep open. Current main has adjacent Feishu sender-scoped group session routing, DM/comment dynamic agent creation, and explicit Feishu subagent thread bindings, but it still does not automatically provision and route every Feishu group sender or role to an isolated `__mc_*` subagent with separate workspace, skills/config, memory, lifecycle, quotas, and admin controls. The KinthAI comments are useful prior art for memory isolation, lifecycle, warm pools, routing, and cost attribution, but they are not a merged OpenClaw implementation or a canonical replacement.

## What This Changes

Request automatic Feishu group-chat routing that creates per-sender or per-role subagents with isolated skills, config, memory, and admin-controlled lifecycle.

## Best Possible Solution

Design an opt-in Feishu plugin mode, plus plugin SDK seams only if needed, that maps sender or role identity to bounded isolated agent profiles with memory namespaces, cleanup/quotas, admin controls, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug reproduction; source inspection gives a high-confidence feature-gap check because group messages route to group peer ids while dynamic agent creation is DM/direct-only and subagent hooks require explicit thread spawning.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Per-sender subagents may solve isolation, but the maintainable path should be opt-in and designed around Feishu identity, memory boundaries, lifecycle, quotas, and admin authority.
