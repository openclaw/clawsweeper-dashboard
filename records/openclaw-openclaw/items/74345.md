Current main still contains the ACP task-maintenance close paths, the exact reported warning strings, and the synchronous session-store scan/write surfaces, with no source, test, changelog, or release evidence proving the idle 480s event-loop saturation fingerprint is fixed. The related reports are useful context but track different or broader gateway stall/process-leak shapes rather than superseding this ACP maintenance loop.

## What This Changes

The issue reports idle gateway event-loop saturation, repeated ACP session-close maintenance failures, long session-write-lock holds, embedded model timeouts, and transport flapping after upgrading within the 2026.4.27 post-tag commit range.

## Best Possible Solution

Make ACP task-maintenance cleanup bounded and idempotent: avoid reattempting the same failing orphan close every sweep without backoff, yield or cap work during ACP cleanup, avoid broad synchronous session-store scans in the hot maintenance loop where possible, safely clear or quarantine stale metadata when the backend is unavailable, and add regression coverage for repeated close failures plus large session-store scans.
