## Summary

Keep open. Current main has adjacent Gateway and ACP cleanup hardening, but I did not find source, test, release, or live-run proof that the exact idle ACP maintenance close loop with 480s event-loop saturation is fixed. The related closed reports cover different Gateway stall fingerprints, while the open ACP orphan-process report is adjacent but not a full duplicate of this task-maintenance/session-lock failure shape.

## What This Changes

Investigate and fix a 2026.4.27 Gateway regression where ACP task maintenance appears to saturate the event loop, leak ACP sessions, hold session write locks, and stall Telegram/Discord transports at idle.

## Best Possible Solution

Root-cause the current-main ACP task-maintenance path and make cleanup bounded, idempotent, and recoverable: add per-session close failure backoff/quarantine or timeouts where appropriate, avoid broad synchronous session-store scans in hot maintenance sweeps, preserve active persistent bindings, and add regression coverage plus a live idle Gateway validation with stale/orphaned ACP metadata.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this read-only review did not run a live systemd Gateway, but the matching maintenance and session-store paths remain present.

## Solution Assessment

No patch is proposed here, and closing is not the best way to solve the issue. The narrowest maintainable path is a targeted ACP task-maintenance/session-metadata fix after live reproduction or a bisect confirms whether the close loop, store scan, or lock contention is the current-main cause.
