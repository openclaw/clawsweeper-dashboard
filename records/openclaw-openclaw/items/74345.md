## Summary

Keep open. Current main has adjacent Gateway responsiveness and ACP stale-session cleanup fixes, but the exact task-registry ACP close-failure loop and idle 480-second event-loop saturation fingerprint are not proven fixed, and the relevant warning/source paths still remain.

## What This Changes

The issue reports a 2026.4.27 regression where idle Gateway task maintenance loops on ACP session close failures, holds session-write locks for minutes, saturates the event loop, and stalls Telegram, Discord, and embedded model traffic.

## Best Possible Solution

Verify current-main idle Gateway behavior with stale/orphaned ACP metadata and large session stores, then either close with shipped proof or add bounded ACP cleanup failure handling, per-session backoff/quarantine, and yielding around all-store ACP scans.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong log fingerprint. For current main, reproduction is source-reproducible only because the warning and synchronous cleanup paths remain, but no live Gateway run was established.

## Solution Assessment

No patch is proposed, and closing is not the best way to solve it yet. The narrowest maintainable path is a maintainer-led current-main live reproduction or bisect, followed by a targeted task-maintenance/ACP cleanup fix if the fingerprint remains.
