## Summary

Keep open. The report is fresh, concrete, and current main still contains the ACP task-maintenance close/list/session-store paths that match the reported fingerprint; current main has related cleanup hardening, but I did not find high-confidence source, test, live-run, changelog, or release proof that the idle 480s event-loop saturation and repeated ACP close loop are resolved. The related closed reports cover broader or different gateway stall shapes, and the open ACP orphan-process report does not supersede this exact maintenance-loop fingerprint.

## What This Changes

Investigate and fix a 2026.4.27 Gateway regression where task-registry ACP cleanup appears to saturate the event loop, leak ACP sessions, hold session write locks, and stall Telegram/Discord transports at idle.

## Best Possible Solution

Root-cause the current-main ACP task-maintenance path and make cleanup bounded, idempotent, and recoverable: add per-session close failure backoff/quarantine or timeouts where appropriate, avoid broad synchronous session-store scans in hot maintenance sweeps, preserve active persistent bindings, and add regression coverage plus a live idle Gateway validation with stale/orphaned ACP metadata.

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this read-only review did not run a live systemd Gateway, but the current source still contains the plausible maintenance path.

## Solution Assessment

No patch is proposed here, and closing is not the best outcome. The narrowest maintainable path is a targeted ACP task-maintenance/session-metadata fix after live reproduction or a bisect confirms whether the close loop, session-store scan, or lock contention is the active current-main cause.
