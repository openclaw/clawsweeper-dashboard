## Summary

Keep open. Current main has adjacent event-loop mitigations for large session stores and transcript-heavy session listing, but the ACP task-maintenance path reported here still lists ACP metadata by scanning session stores, still retries close attempts during maintenance, and still logs the exact close-failure warnings without backoff/quarantine evidence. The closed related reports cover broader `sessions.list`/`models.list`/tool-construction stall shapes, not this idle ACP maintenance-loop fingerprint.

## What This Changes

The issue reports idle gateway event-loop saturation, ACP session close failures, long session-write-lock holds, embedded model timeouts, and transport flapping after upgrading to a 2026.4.27 build.

## Best Possible Solution

Make ACP task-maintenance cleanup bounded and idempotent: add backoff or quarantine for repeated close failures, cap/yield orphan cleanup work, avoid broad synchronous session-store scans in the hot maintenance sweep where possible, safely clear stale ACP metadata when backend closure is unrecoverable, and add regression coverage for repeated close failures plus large session-store scans.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this review did not run a live gateway service, and adjacent fixes do not prove this ACP-specific path is resolved.

## Solution Assessment

No patch is proposed here. Keeping this open is the best current disposition because current main still exposes the ACP maintenance failure loop, while the related closed reports solve different gateway stall shapes.
