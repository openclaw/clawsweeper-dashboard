## Summary

Keep open. Current main contains adjacent Gateway responsiveness and ACP lifecycle fixes, but source inspection still shows the ACP maintenance close-failure loop and synchronous session-store scan paths, and there is no live or shipped proof that this exact idle 480-second saturation fingerprint is gone.

## What This Changes

The issue reports an OpenClaw 2026.4.27 regression where idle Gateway task maintenance loops on ACP session close failures, holds session-write locks for minutes, saturates the event loop, and stalls Telegram/Discord/model traffic.

## Best Possible Solution

Verify current-main idle Gateway behavior with stale/orphaned ACP metadata and large session stores, then either close with shipped proof or add bounded ACP cleanup failure handling, backoff/quarantine, and yielding around the all-store scan.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong log fingerprint. For current main, reproduction is source-reproducible only because the warning and synchronous cleanup paths remain, but no live Gateway run was established.

## Solution Assessment

No patch is proposed, and closing is not the best solution yet. The narrowest maintainable path is a maintainer-led current-main live reproduction or bisect, followed by a targeted task-maintenance/ACP cleanup fix if the fingerprint remains.
