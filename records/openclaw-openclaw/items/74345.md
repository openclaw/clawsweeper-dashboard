## Summary

Keep open. Current main still contains the reported ACP task-maintenance close paths, exact warning strings, and synchronous session-store scan/write surfaces, and I found no current source, docs, tests, release note, or related merged PR proving the idle event-loop saturation fingerprint is fixed. The cross-referenced reports are relevant, but they track either broader gateway stalls or distinct ACP process-orphan symptoms rather than a canonical replacement for this ACP maintenance loop.

## What This Changes

The issue reports idle gateway event-loop saturation, repeated ACP session-close maintenance failures, long session-write-lock holds, embedded model timeouts, and transport flapping after upgrading within the 2026.4.27 post-tag commit range.

## Best Possible Solution

Make ACP task-maintenance cleanup bounded and idempotent: avoid reattempting the same failing orphan close every sweep without backoff, yield or cap work during ACP cleanup, avoid broad synchronous session-store scans in the hot maintenance loop where possible, clear or quarantine stale metadata safely when the backend is unavailable, and add regression coverage for repeated close failures plus large session-store scans.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this review did not run a live gateway service, but the current source still contains the plausible maintenance path.

## Solution Assessment

No patch is proposed here, so the best current disposition is to keep this open for maintainer-led current-main reproduction or bisect. A narrow code fix should target ACP task maintenance/session metadata once the failing close or scan loop is confirmed.
