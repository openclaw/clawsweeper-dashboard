The report is fresh, concrete, production-impacting, and still maps directly to current-main ACP task-maintenance/session-store code paths; I did not find a shipped or current-main fix that proves the reported idle 480s event-loop saturation plus repeated ACP session-close loop is resolved.

## What This Changes

The issue reports that a 2026.4.27 gateway build at commit 9d8de70 leaks ACP sessions during task-registry maintenance and pegs the Node event loop at idle, causing embedded model timeouts and channel transport stalls.

## Best Possible Solution

Root-cause the current-main ACP maintenance path and make cleanup bounded, idempotent, and recoverable: avoid unbounded synchronous session-store work in maintenance sweeps, add per-session close failure backoff/quarantine or bounded retries where appropriate, preserve active persistent bindings, and cover stale/orphaned ACP metadata with regression tests plus a live idle gateway validation.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this read-only review did not run a live systemd gateway, but the current source still contains the plausible maintenance path and warning strings.

## Solution Assessment

No patch is proposed, and closing is not the best way to solve this issue. The narrowest maintainable path is a targeted ACP task-maintenance/session-metadata fix after live reproduction or a bisect confirms whether close-loop retries, session-store scans, or lock contention are the active current-main cause.
