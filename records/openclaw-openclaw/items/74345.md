## Summary

Keep open. Current main still contains the ACP task-maintenance close loop, the exact reported warning paths, serial orphaned ACP cleanup, and synchronous session-store scans; newer mainline fixes reduce adjacent `sessions.list`, `models.list`, plugin-tool, and transcript stalls, but they do not prove this idle ACP maintenance/session-lock fingerprint is fixed.

## What This Changes

The issue reports idle gateway event-loop saturation, ACP session close failures, long session-write-lock holds, embedded model timeouts, and Telegram/Discord transport flapping after upgrading to a 2026.4.27 build.

## Best Possible Solution

Make ACP task-maintenance cleanup bounded and idempotent: record backoff or quarantine for repeated close failures, cap/yield orphan cleanup work, avoid broad synchronous ACP metadata scans in the hot maintenance sweep where possible, safely clear stale metadata when backend closure is unrecoverable, and add regression coverage for repeated close failures plus large session-store scans.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this read-only review did not run a live gateway service, and source/history do not prove the ACP-specific path is resolved.

## Solution Assessment

No patch is proposed here. Keeping this open is the best current disposition because current main still exposes the ACP maintenance failure loop while the related closed reports solve different gateway stall shapes.
