## Summary

Keep open. The report is fresh, concrete, and still maps to current-main ACP task-maintenance/session-store paths; later provider/runtime hot-path fixes explain nearby stall reports, but I did not find source, test, shipped-release, or live-run proof that this exact ACP close-failure loop plus multi-minute event-loop saturation is fixed.

## What This Changes

The issue reports a Gateway regression where ACP task maintenance repeatedly fails to close terminal/orphaned sessions, session locks are held for minutes, and the Node event loop saturates after a 2026.4.27 build.

## Best Possible Solution

Root-cause the current-main ACP task-maintenance path and make cleanup bounded and recoverable: add per-session close failure backoff or quarantine where appropriate, avoid unbounded synchronous session-store work in hot sweeps, preserve active persistent bindings, and add regression coverage plus a live idle Gateway validation with stale/orphaned ACP metadata.

## Reproduction Assessment

Yes for the originally reported bad build: the issue provides exact checkout/build/restart steps and a strong five-signal log fingerprint. Current-main reproduction is unclear because this read-only review did not run a live systemd Gateway, but the matching maintenance paths and warning strings still exist.

## Solution Assessment

No patch is proposed here, and closing is not the best way to solve the issue. The narrowest maintainable next step is a targeted live reproduction or bisect followed by an ACP task-maintenance/session-store fix tied to the confirmed current-main root cause.
