## Summary

Keep open. The PR is unsafe as written because it removes the Discord gateway emitter's post-dispose `error` listener, while current `main` intentionally keeps that listener to suppress late gateway errors that can arrive during or after shutdown.

## What This Changes

The PR adds a call to remove the Discord gateway supervisor's `error` listener during `dispose()`.

## Best Possible Solution

Preserve the invariant that late Discord gateway `error` events are swallowed or logged after shutdown. If retained supervisor state is a real memory issue, redesign disposal to swap the full handler for a minimal bounded late-error sink and update the existing gateway-supervisor and provider cleanup tests.

## Reproduction Assessment

Yes. The current `gateway-supervisor.test.ts` path creates a supervisor, calls `dispose()`, then emits `error` and expects no throw; applying the PR's detachment would make that same EventEmitter path throw when no other `error` listener is present.

## Solution Assessment

No. Directly removing the listener is not the best solution because it fixes retained closure state by regressing Node's `error` event contract and the shipped Discord shutdown behavior; a replacement sink is the safer direction.
