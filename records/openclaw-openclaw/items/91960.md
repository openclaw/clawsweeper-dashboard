## Summary

Current main already has durable internal lifecycle support for disabling a completion announcement, but that control is absent from the public `sessions_spawn` schema and documentation, so callers still cannot reliably request deterministic suppression. The issue remains useful; the narrow per-spawn API direction is supported by the code, while a new global configuration default still needs separate justification.

## What This Changes

Requests a deterministic spawn-time or configuration control that prevents a sub-agent completion from being announced into its parent context regardless of the child model’s final output.

## Best Possible Solution

Expose the existing persisted per-run completion-suppression policy in the `sessions_spawn` schema, give it explicit native and ACP semantics, add schema-to-lifecycle coverage, and document it; defer a global default unless a separate product case demonstrates that per-spawn control is insufficient.

## Reproduction Assessment

Yes, at source level: current main deterministically honors `expectsCompletionMessage: false` internally, but the public `sessions_spawn` schema cannot express it and the docs expose only model-output token suppression. This establishes the reported public-contract gap without relying on a live model run.

## Solution Assessment

Partly yes: exposing the existing per-spawn lifecycle control is the narrowest maintainable solution and avoids a duplicate announce path. The proposed global configuration default is not yet the best solution because it adds broad policy surface without evidence that per-spawn control is insufficient.

- The public argument name and description must make clear that suppression disables all completion delivery, including nested parent injection, rather than only direct user-facing channel delivery.

- A global `agents.defaults.subagents` option would enlarge the configuration surface and silently affect every spawn; the current evidence only clearly supports a per-spawn control.
