## Label Justifications

- `impact:session-state`: The reported behavior injects unwanted child completion content into the parent transcript and working context.

## Summary

Keep open: current main and v2026.6.6 still expose only sentinel-token suppression publicly, while the deterministic no-announce state remains internal. The issue remains useful, but it needs maintainer API/product judgment on the public per-spawn policy and cross-runtime semantics.

## What This Changes

The issue requests a deterministic spawn-time or configuration-controlled way to suppress sub-agent completion announcements to the parent context, independent of child output tokens.

## Best Possible Solution

Add one typed per-spawn completion-announcement policy backed by the existing durable run state, document native and ACP semantics, preserve announce as the default, and defer any global config default until a separate use case proves it necessary.

## Reproduction Assessment

Yes. Source inspection on current main shows public `sessions_spawn` callers still rely on exact child-output tokens, while the deterministic `expectsCompletionMessage === false` path is internal and absent from the tool schema.

## Solution Assessment

Yes for a per-spawn policy; no for adding both a parallel boolean and an unproven global config default. The narrowest maintainable solution is a typed public option that reuses the registry’s existing durable no-completion-message state.
