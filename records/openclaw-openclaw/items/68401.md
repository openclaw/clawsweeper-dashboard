## Summary

Keep open. Current main has fixed the narrower Discord listener-timeout path, but the broader request remains unimplemented: same-session turns and compaction still serialize, compacting runs reject steering, and busy inbound messages fall back to later followup handling instead of being processed during compaction.

## What This Changes

The issue requests that Telegram/Discord-style inbound messages be accepted or processed with minimal delay while session compaction is running.

## Best Possible Solution

Treat this as a core session-runtime design change: either make compaction copy-on-write/background-safe, or add an explicit bounded busy-session acknowledgment/offload contract that channel plugins can use while preserving transcript order and followup delivery.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: send a same-session inbound message while compaction is active; current main rejects steering for compacting runs, serializes normal runs through the session lane, and enqueues busy messages as followups. I did not run a live Telegram or Discord reproduction because this review was read-only.

## Solution Assessment

No. Current main is not the best solution to the requested behavior because it mainly hardens channel timeout handoff and followup queuing; the maintainable fix needs a deliberate nonblocking compaction or busy-session/offload design across core and channels.
