## Summary

Keep open. Current main still serializes same-session turns and compaction through the session lane, rejects steering into compacting runs, and queues busy inbound messages as followups instead of processing or acknowledging them during compaction; the related closed Discord item covers a narrower timeout/drop symptom, not this broader nonblocking-compaction request.

## What This Changes

Allow inbound channel messages to be processed, offloaded, or promptly acknowledged while session compaction is running instead of waiting for compaction to finish.

## Best Possible Solution

Define a core session-runtime contract for nonblocking compaction or bounded busy-session acknowledgement/offload, then add channel-facing coverage for inbound messages during compaction without corrupting transcript or followup ordering.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to send a same-session Telegram or Discord message while compaction is active and observe that steering returns false, normal execution stays behind the same session lane, and the message is queued as a later followup.

## Solution Assessment

No. Current main has useful queue and timeout hardening, but a config-only or channel-only patch would not safely solve the requested behavior without addressing session lanes, write locking, active-run steering, and transcript rotation together.
