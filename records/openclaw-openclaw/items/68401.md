## Summary

Keep open. Current main fixes the narrower Discord listener-timeout symptom from #73204, but it still does not implement the broader request to accept or process normal same-session channel messages while compaction is running: compaction and normal turns still serialize through session lanes, direct compaction still holds the session write lock during provider compaction, compacting runs reject steering, and busy-session messages are queued as followups rather than processed independently.

## What This Changes

Request to allow inbound Telegram and other channel messages to be accepted or processed with minimal delay while session compaction is running.

## Best Possible Solution

Treat this as a core session-runtime design change: make compaction nonblocking through copy-on-write/background compaction, or add a bounded busy-session acknowledgment/offload path, then cover Telegram/Discord-style inbound messages during compaction without corrupting transcript order or followup delivery.

## Reproduction Assessment

Yes, at the source-path level: an inbound same-session message during compaction follows the active-run steering/followup path, but queueEmbeddedPiMessage rejects compacting runs and direct compaction holds the write lock while activeSession.compact runs. I did not run a live Telegram or Discord reproduction because this review was read-only.

## Solution Assessment

No, current main is not yet the best solution to the requested behavior. The Discord listener-timeout fix is valuable and shipped, but the broader maintainable solution needs a core nonblocking compaction or explicit busy-session/offload contract rather than only channel queue timeout hardening.
