## Summary

Keep open. Current `main` still serializes normal same-session turns and queued compaction through the session lane, rejects steering while an active run is compacting, and queues busy-session followups instead of processing or explicitly acknowledging Telegram-style inbound messages during compaction. The related Discord listener-timeout report at https://github.com/openclaw/openclaw/issues/73204 was fixed with an async Discord handoff, and https://github.com/openclaw/openclaw/issues/53008 overlaps, but neither supersedes this broader channel-message/nonblocking-compaction request.

## What This Changes

Request to let OpenClaw keep accepting or responding to Telegram-style inbound messages while session compaction is running instead of batching them until compaction finishes.

## Best Possible Solution

Define an explicit session-runtime contract for nonblocking compaction or bounded busy-session acknowledgement/offload, then cover Telegram and Discord inbound messages during compaction with transcript-order and rotation regression tests.

## Reproduction Assessment

Yes, source-level. Send a same-session inbound message while compaction is active: the session lane remains occupied, steering returns `compacting`, and the message is stored as a later followup rather than processed during compaction.

## Solution Assessment

No, not as a config-only switch or local queue tweak. The maintainable fix needs an explicit nonblocking compaction or busy-session contract that preserves write-lock, transcript rotation, active-run, and per-channel acknowledgement semantics.
