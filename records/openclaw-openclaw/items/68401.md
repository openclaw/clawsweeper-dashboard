## Summary

Close as duplicate: the same remaining nonblocking-compaction/session-lane work is already tracked by the older open canonical issue at https://github.com/openclaw/openclaw/issues/53008. Current main still serializes normal same-session work and compaction through the session lane, rejects steering while compacting, and the related closed Discord listener-timeout report only fixed a narrower drop symptom.

## What This Changes

Request that OpenClaw continue accepting, processing, or acknowledging same-session channel messages while session compaction is running instead of queuing them behind compaction.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/53008 as the canonical implementation thread and solve it with a maintainer-approved nonblocking or bounded compaction scheduling design that preserves transcript ordering, rotation safety, and channel acknowledgments.

## Reproduction Assessment

Yes, source-level. Trigger same-session compaction or preflight compaction, then send another same-session channel message; current main routes normal runs and queued compaction through the same session lane and rejects steering while the run is compacting.

## Solution Assessment

Yes for cleanup. The best triage path is to close this duplicate and keep the older open canonical issue at https://github.com/openclaw/openclaw/issues/53008 as the design and implementation thread; a config-only switch would be unsafe without a transcript-locking and channel-acknowledgment contract.
