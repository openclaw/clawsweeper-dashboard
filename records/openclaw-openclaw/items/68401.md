## Summary

Keep open. Current main still serializes same-session runs and compaction through the session lane, rejects active-run steering while compacting, and queues busy inbound messages as later followups rather than processing or explicitly acknowledging them during compaction; the closed Discord timeout item only covers a narrower loss/timeout symptom.

## What This Changes

The issue requests nonblocking or acknowledged processing for inbound channel messages while a session compaction is active.

## Best Possible Solution

Design a core session-runtime contract for nonblocking compaction or explicit busy-session acknowledgement/offload, then cover Telegram/Discord-style inbound messages arriving during compaction without corrupting transcript order or followup delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to send a same-session inbound message while compaction is active and observe that steering returns false for compacting runs, the normal run path stays on the same session lane, and the message is enqueued as a later followup rather than processed during compaction.

## Solution Assessment

No. Current main is not the best solution to the requested behavior; it has useful queue and timeout hardening, but the requested behavior needs an explicit nonblocking compaction or busy-session design rather than a narrow timeout or config-only patch.
