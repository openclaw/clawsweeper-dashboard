## Summary

Keep open. Current main still serializes normal same-session turns and queued compaction on the same session lane, rejects low-latency steering while an active run reports compaction, and queues busy-session followups for later instead of processing or explicitly acknowledging Telegram-style inbound messages during compaction.

## What This Changes

Request processing or promptly acknowledging incoming channel messages while session compaction is in progress instead of delaying them until compaction completes.

## Best Possible Solution

Treat this as a core session-runtime design change: implement nonblocking compaction through a safe copy-on-write/background contract or add a bounded busy-session acknowledgment/offload path, then cover Telegram and Discord-style inbound messages without corrupting transcript order.

## Reproduction Assessment

Yes, source-level. Send a same-session inbound channel message while compaction is active: the active run reports `compacting`, steering returns that failure reason, and normal same-session work remains serialized behind the session lane.

## Solution Assessment

No current implementation solves the broader request. A config-only switch would be risky without an explicit transcript, lock, rotation, and per-channel acknowledgment contract.
