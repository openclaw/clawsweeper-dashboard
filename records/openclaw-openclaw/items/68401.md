## Summary

Keep open. Current main fixes the narrower Discord listener-timeout/drop symptom, but the broader request is still not implemented: same-session work and compaction still share the session lane, compacting runs reject steering, and normal inbound followups are queued instead of processed or acknowledged during compaction.

## What This Changes

The issue requests processing or acknowledging inbound channel messages while session compaction is running, instead of blocking them behind compaction.

## Best Possible Solution

Design this as a core session-runtime change: either safe copy-on-write/background compaction or a bounded busy-session acknowledgment/offload path with transcript-ordering and channel-delivery tests.

## Reproduction Assessment

Yes, source-level. Send a same-session inbound channel message while compaction is active: steering reports `compacting`, the normal run path remains serialized on the session lane, and busy followups are queued rather than processed independently.

## Solution Assessment

No current implementation fully solves the issue. A config-only switch would be risky without an explicit transcript locking/rotation contract and channel acknowledgment semantics.
