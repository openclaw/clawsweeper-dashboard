## Summary

Keep open. Current main still serializes same-session runs and compaction through the session lane, rejects steering into compacting runs, and falls back to later followup queueing rather than processing or explicitly acknowledging inbound Telegram-style messages during compaction; the closed Discord issue only fixed the narrower listener-timeout/drop symptom.

## What This Changes

Allow channel messages to be processed, offloaded, or promptly acknowledged while session compaction is running.

## Best Possible Solution

Design a core session-runtime contract for nonblocking compaction or bounded busy-session acknowledgement/offload, then add Telegram/Discord-style coverage for inbound messages arriving during compaction without corrupting transcript order.

## Reproduction Assessment

Yes. A high-confidence source-level path is to send a same-session inbound message while compaction is active: the same session lane remains occupied, steering returns reason 'compacting', and the message is queued as a later followup rather than processed during compaction.

## Solution Assessment

No. Current main has useful queueing, status, and Discord timeout hardening, but it is not the requested solution; the maintainable fix needs an explicit nonblocking compaction or busy-session contract rather than a config-only switch.
