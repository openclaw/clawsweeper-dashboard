## Summary

Keep open: the Slack report matches a current and shipped source path where a subagent completion wake can be rejected with `source_reply_delivery_mode_mismatch`, and the existing related issues do not fully cover this Slack-specific failure mode.

## What This Changes

Ensure Slack requester sessions can receive or recover subagent completion wakes instead of failing with `source_reply_delivery_mode_mismatch` while Slack remains healthy.

## Best Possible Solution

Keep this as a Slack-specific subagent completion-delivery bug and decide how completion wakes should reconcile the active run's source-reply mode with message-tool-only completion policy before adding focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main and `v2026.5.22` have a deterministic path where a completion wake requests `message_tool_only` and the active run rejects it with the reported mismatch if the original run lacks that mode. I did not run a live Slack reproduction because this review is read-only and the issue omits the sanitized session files needed for an end-to-end replay.

## Solution Assessment

Unclear. A fix is needed, but the best fix needs maintainer judgment because loosening the mismatch guard could conflict with the configured message-tool-only visible-reply contract; preserving the active run mode while surfacing a terminal recovery message looks safer than a broad fallback.
