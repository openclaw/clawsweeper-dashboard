## Summary

Keep this open. Current main has several related latency optimizations and the Telegram getMe probe is wired to the configured proxy, but I did not find measured current-main proof that the reporter's 45-53s pre-stream local-backend case is fixed, and the SOCKS timeout still needs a focused live probe to distinguish proxy bypass from event-loop starvation or transport behavior.

## What This Changes

Investigate slow embedded-agent pre-stream startup on a local OpenAI-compatible backend and Telegram getMe timeouts despite configured SOCKS proxy.

## Best Possible Solution

Split the work into measured current-main embedded-runner latency tracking and a focused Telegram SOCKS getMe probe check, then land narrow fixes with regression coverage for whichever path is confirmed.

## Reproduction Assessment

Yes for the embedded-runner latency surface: the reporter supplied concrete commands and trace logs for 2026.4.29. The Telegram SOCKS bypass is unclear on current main because source threads the proxy into getMe and undici 8.1 supports `socks5:`, so it needs a live SOCKS-backed probe reproduction.

## Solution Assessment

Not applicable because there is no proposed patch. The safest path is not a broad automated fix, but measured stage-specific optimization plus a separate transport/probe fix only if live verification confirms a real bypass or unsupported proxy scheme.
