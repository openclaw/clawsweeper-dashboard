## Summary

Keep this open. Current main still performs the same per-turn embedded startup and prep sequence that the reporter timed before `stream-ready`; the newer plugin-tool timing diagnostics help narrow one slow stage but do not remove or cache the reported work. The Telegram proxy symptom also should not be collapsed into the latency duplicate yet: source passes `account.config.proxy` into the getMe probe, but the documented `socks5://` configuration and timeout log need a focused SOCKS/probe reproduction before this can be called fixed or user setup.

## What This Changes

The issue asks OpenClaw to reduce embedded-agent pre-stream latency for local OpenAI-compatible runs and to make Telegram getMe health/probe calls consistently honor `channels.telegram.proxy`.

## Best Possible Solution

Track the embedded-runner latency with the #75725 cluster and add measured warm-turn caching, lazy preparation, or stage-specific optimization behind clear invalidation boundaries. Split or separately verify the Telegram SOCKS getMe path with focused diagnostics and regression coverage so a proxy transport bug is not hidden inside the performance report.

## Reproduction Assessment

Yes for the embedded-runner latency surface: the reporter supplied concrete trace timings and current main still has the traced startup/prep stages. The Telegram proxy bypass is plausible but not yet high-confidence because the code passes `proxyUrl` into the probe path, so it needs a SOCKS-backed getMe reproduction or more specific diagnostics.

## Solution Assessment

Not applicable because there is no proposed patch. The narrowest maintainable path is to measure and optimize the shared embedded-runner prep path, then handle the Telegram proxy behavior as a separate verified transport/logging fix if the SOCKS probe reproduction confirms it.
