## Summary

Keep open. Current main still performs the same per-turn embedded-agent startup and prep work before `stream-ready`, and the closely related Telegram latency report remains open. The Telegram proxy portion is also not closeable: source shows the main probe/startup paths try to pass `account.config.proxy` into Telegram fetch resolution, but the reporter's `getMe` timeout needs targeted live verification instead of being dismissed from source inspection alone.

## What This Changes

The issue reports slow embedded-agent startup/prep before streaming on a fast local OpenAI-compatible backend and a possible Telegram `getMe` health/probe path that does not honor `channels.telegram.proxy`.

## Best Possible Solution

Keep this open and split the work into two maintainable tracks: use #75725 as the canonical embedded-runner latency cluster unless maintainers want this richer local-backend trace attached there, and add a focused Telegram proxy repro that identifies the exact `getMe` caller and transport policy before changing proxy code.

## Reproduction Assessment

Partially yes. The embedded latency has a high-confidence reproduction path through repeated Telegram or isolated embedded turns with trace logging or the RTT harness; the Telegram proxy bypass is plausible from the reported curl comparison but unclear from current source because the main probe/startup paths do pass `account.config.proxy`.

## Solution Assessment

Unclear. The best fix is unlikely to be a Telegram-specific shortcut; the latency side needs measured shared-runner warm-path or caching work with strict invalidation, while the proxy side needs a targeted transport repro before any code change.
