## What This Changes

This PR serializes gateway WebSocket message handling per connection, rejects duplicate client installation on the same socket, and cleans stale node-registry state when a node registration is superseded.

## Best Possible Solution

Land the bounded Gateway fix after redacted real Gateway proof or a focused WebSocket harness shows duplicate pre-auth connect frames leave one active client/node state without regressing reconnect or invoke cleanup.

## Reproduction Assessment

Yes, from source inspection rather than an executed live repro. Current main starts each socket message handler independently and performs async auth/pairing before installing the client, which gives two valid pre-auth connect frames a clear race window.

## Solution Assessment

Yes, the per-socket queue plus duplicate setClient guard is a narrow maintainable fix for the race. The remaining gap is proof: the PR should demonstrate the real Gateway duplicate-connect path or a focused harness before merge.
