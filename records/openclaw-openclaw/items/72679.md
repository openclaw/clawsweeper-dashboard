## Summary

Keep open: the root Nextcloud Talk reaction bug is real, but this PR is not merge-ready because it mishandles shared `react` semantics, lacks real behavior proof for the changed path, and overlaps the more complete open implementation at https://github.com/openclaw/openclaw/pull/72348.

## What This Changes

The PR adds a Nextcloud Talk `react` channel-message action adapter, registers it on the channel plugin, and adds unit tests for validation and dispatch.

## Best Possible Solution

Land one canonical Nextcloud Talk action implementation that registers a plugin-owned `react` adapter, follows the shared reaction contract, rejects or implements unsupported removal, covers sender/request shape, and then closes https://github.com/openclaw/openclaw/issues/70110.

## Reproduction Assessment

Yes. Source inspection on current main shows Nextcloud Talk advertises reactions and has a guarded sender, but lacks `actions.handleAction`, so `react` dispatch returns no handler before it can send.

## Solution Assessment

No. The adapter seam is the right direction, but this patch is not the best merge form because it ignores `remove`, misses the documented message-id alias/current numeric path, advertises actions without configured accounts, and overlaps https://github.com/openclaw/openclaw/pull/72348.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject removals before calling the add sender:** `extensions/nextcloud-talk/src/channel.adapters.ts:78-83`
  - body: The shared `react` action accepts `remove`, but this handler never checks it and always calls the POST/add sender. A caller passing `remove: true` would add or refresh the reaction instead of removing it, so either implement the DELETE path or reject removal before sending.
  - confidence: 0.9
- **[P2] Use the shared reaction message-id resolver:** `extensions/nextcloud-talk/src/channel.adapters.ts:74`
  - body: This only accepts string `messageId` and string `toolContext.currentMessageId`, so the documented `message_id` alias and numeric current-message fallback are ignored. Use `resolveReactionMessageId` and stringify the non-null result before sending.
  - confidence: 0.87
- **[P3] Scope discovery to configured accounts:** `extensions/nextcloud-talk/src/channel.adapters.ts:59`
  - body: `describeMessageTool` advertises `react` without checking `cfg` or the selected account. That can expose Nextcloud Talk reactions when no enabled account has both a base URL and bot secret; mirror the configured-account checks used by other channel action adapters.
  - confidence: 0.8
