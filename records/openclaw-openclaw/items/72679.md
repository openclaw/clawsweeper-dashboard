## Summary

Keep open. The PR targets a real current-main gap, but it is not landable as written: it wires the right channel-action seam while missing shared message-id alias handling, ignoring reaction removal semantics, advertising actions for unconfigured accounts, and omitting required user-facing changelog coverage. The overlapping open implementation in #72348 also means maintainers should consolidate rather than close or auto-replace this PR.

## What This Changes

This PR adds a Nextcloud Talk `react` channel message-action adapter, registers it on the plugin, and adds unit tests that mock the existing reaction sender.

## Best Possible Solution

Land one consolidated Nextcloud Talk message-action implementation that registers `react`, scopes discovery to configured accounts, resolves message IDs through the shared helper, implements or explicitly rejects removal semantics, covers adapter and HTTP sender request shape, and adds the required changelog entry before closing #70110.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: Nextcloud Talk advertises reactions and has `sendReactionNextcloudTalk`, but no `actions.handleAction` is registered, so plugin message-action dispatch cannot reach the sender.

## Solution Assessment

No. Registering a channel action adapter is the right seam, but this patch is not the best form because it bypasses existing shared helpers, ignores removal requests documented by both OpenClaw and Nextcloud Talk, advertises unconfigured accounts, and overlaps #72348.
