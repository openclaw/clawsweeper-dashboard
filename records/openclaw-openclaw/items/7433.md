## Summary

This issue is still necessary: current main and latest release still leave the Baileys retry and socket-level group metadata hooks at their defaults, while only the reconnect-time group metadata fallback has landed.

## What This Changes

The issue asks to wire Baileys WhatsApp retry, group metadata cache, group event, participant event, and LID-mapping hooks to improve group message reliability.

## Best Possible Solution

Land a focused WhatsApp plugin patch that reuses existing metadata and identity state, supplies Baileys with bounded retry content before sends, and adds regression coverage for the socket options and event handlers.

## Reproduction Assessment

Yes, at source level with high confidence: current main omits the Baileys callbacks/events, and Baileys rc13 directly uses `getMessage` during retry-receipt handling. I did not run a live WhatsApp retransmission failure in this read-only review.

## Solution Assessment

Yes on the dependency seams, but not by copying the issue's parallel maps verbatim. The best fix should route Baileys callbacks through OpenClaw's existing bounded caches and auth-backed LID mapping state.
