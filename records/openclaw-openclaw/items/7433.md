## Summary

This issue remains necessary: current main and v2026.6.5 leave Baileys rc13 retry and socket-level group metadata callbacks at their no-op defaults. The merged reconnect recovery addresses one failure mode, but not retransmission after retry receipts; the requested implementation should be narrowed to reuse existing metadata and LID state rather than copying the proposed parallel maps.

## What This Changes

Improve WhatsApp group reliability by wiring Baileys message-retry, group metadata cache, group update, and identity-mapping behavior into the existing plugin runtime.

## Best Possible Solution

Provide Baileys rc13 with a bounded canonical source of outgoing message content before send, connect `cachedGroupMetadata` to the existing plugin cache, and let relevant events update existing metadata and auth-backed identity state without introducing parallel stores.

## Reproduction Assessment

Yes at source level with high confidence: current main omits the callbacks, and Baileys rc13 directly invokes `getMessage` during retry-receipt processing. This read-only review did not execute a live WhatsApp retransmission failure.

## Solution Assessment

No, not exactly as proposed: the dependency seams are correct, but the best implementation should reuse OpenClaw's existing metadata cache and Baileys' auth-backed LID mappings instead of adding parallel in-memory maps.
