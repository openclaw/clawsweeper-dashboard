## Summary

Current main still lacks Baileys’ recommended `getMessage` and `cachedGroupMetadata` socket hooks, while the merged reconnect cache covers only part of the requested reliability behavior. This remains a focused WhatsApp plugin repair; the newer case where messages reach inbound logs but do not trigger an agent may be a separate routing problem.

## What This Changes

Add Baileys retry-message retrieval, socket-level group metadata caching, proactive group cache updates, and durable LID mapping handling for WhatsApp group reliability.

## Best Possible Solution

Implement a plugin-local lifecycle owner that supplies Baileys `getMessage` and full `cachedGroupMetadata`, refreshes metadata from supported events, stores only bounded retry content in canonical SQLite state, and preserves the current mapping-store path for LID resolution.

## Reproduction Assessment

Yes at source level: current main’s socket options and event registrations can be inspected directly, and the dependency-recommended hooks are absent. A live forced retransmission or decryption retry would strengthen eventual PR proof but is not required to establish the implementation gap.

## Solution Assessment

Yes for `getMessage` and `cachedGroupMetadata`, provided the implementation follows Baileys rc13 and OpenClaw’s lifecycle and SQLite state contracts. The issue’s unbounded Maps and reliance on `lid-mapping.update` should not be copied literally.
