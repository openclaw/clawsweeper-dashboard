## Summary

Keep this issue open, but narrow the repair to the remaining group-metadata lifecycle gap. Current main still does not connect its existing bounded group metadata cache to Baileys or update it from group events; however, Baileys rc13 now enables a bounded recent-message retry cache by default and persists discovered LID mappings through its auth-backed signal store, so parallel OpenClaw message and LID stores would duplicate dependency behavior.

## What This Changes

Improve WhatsApp group reliability by supporting message retransmission, socket-level group metadata caching, proactive group metadata updates, and durable LID resolution.

## Best Possible Solution

Pass the existing bounded group metadata cache into Baileys and maintain it through dependency-supported group events with symmetric listener cleanup, while relying on Baileys’ existing retry and auth-backed LID stores unless focused proof demonstrates a remaining gap.

## Reproduction Assessment

Yes, at source level: current main has an existing group metadata cache but neither passes it to Baileys nor maintains it from the declared group events. This read-only review did not run a live WhatsApp group scenario.

## Solution Assessment

No, not exactly as originally proposed: separate message and LID maps now duplicate Baileys rc13 behavior. The best repair is the narrower missing group-cache integration, with additional retry or LID work only if focused proof exposes a dependency gap.
