## Repair Work Prompt

Fix the WhatsApp/Baileys reliability gaps tracked by https://github.com/openclaw/openclaw/issues/7433. Keep the change scoped to the WhatsApp plugin. Wire Baileys `getMessage` with a bounded, privacy-conscious retry-content store that records outgoing `WAMessageContent` before `sendMessage` awaits the acknowledgment and can serve retry requests by message key. Pass a socket-level `cachedGroupMetadata` callback backed by the existing bounded group metadata cache. Add proactive handlers for `groups.upsert`, `groups.update`, `group-participants.update`, and `lid-mapping.update` where they can safely refresh existing metadata/LID state without broad persistence. Extend the Baileys test mock as needed. Add regression tests for socket option construction, retry-content lookup, group metadata cache callback behavior, group/participant updates, and LID update handling. Add a single user-facing CHANGELOG.md fix entry crediting @octopuslabs-fl and, if appropriate for the retry-content detail, @m13v. Do not add a core/SDK API, config knob, dependency upgrade, or persistent message-content database unless the existing plugin boundary cannot support the fix.
