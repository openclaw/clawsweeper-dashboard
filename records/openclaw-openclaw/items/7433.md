## Summary

Keep open. Current main still has the partial WhatsApp reconnect/group metadata fallback from https://github.com/openclaw/openclaw/pull/73732, but it still does not wire Baileys `getMessage` or `cachedGroupMetadata` into socket construction and does not listen for the requested group or LID update events.

## What This Changes

The issue requests WhatsApp/Baileys retry, group metadata, participant, and LID cache hooks to improve group message reliability.

## Best Possible Solution

Land a focused WhatsApp plugin patch that passes Baileys `getMessage` and `cachedGroupMetadata`, keeps group/LID metadata fresh from Baileys events, and retains retryable message content only in a bounded privacy-preserving cache.

## Reproduction Assessment

Yes, at source level: current main can be checked against the pinned Baileys v7.0.0-rc11 source/types and still lacks the requested socket options and event handlers. I did not run a live WhatsApp high-volume group retry scenario.

## Solution Assessment

Yes. A plugin-local repair in the WhatsApp session and inbound monitor paths is the narrowest maintainable solution; avoid core API changes, new config, or durable message storage unless explicitly approved.
