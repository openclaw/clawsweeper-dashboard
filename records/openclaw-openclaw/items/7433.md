## Summary

This issue is still necessary: current `main` and release v2026.6.5 retain only the partial reconnect-time group cache and still omit Baileys retry and socket-level group metadata callbacks. The repair should exclude a parallel LID map because Baileys rc13 already persists those mappings through its signal repository.

## What This Changes

Improve WhatsApp group reliability by wiring Baileys message-retry and group-metadata cache behavior and keeping group state current.

## Best Possible Solution

Use one canonical full group-metadata cache for socket and inbound behavior, add bounded pre-send message-content retention for Baileys retransmission, and continue using Baileys' existing LID repository rather than duplicating identity state.

## Reproduction Assessment

Yes at source level with high confidence: current `main` omits callbacks that pinned Baileys rc13 explicitly uses for retry and group sending, while tests cover only the partial reconnect cache and existing LID repository. ## Solution Assessment

No, not exactly as proposed: `getMessage` and a socket-connected full group cache are appropriate, but a separate LID map would duplicate Baileys rc13's built-in persistence; participant events should update the cache only when complete metadata invariants can be preserved.
