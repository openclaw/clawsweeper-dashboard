## What This Changes

The PR branch changes credential keychain readers and also modifies embedded runner prompt construction plus two plugin contract assertions.

## Best Possible Solution

Land a focused gateway/auth patch that moves hot macOS keychain reads to an awaitable non-blocking path, preserves credential expiry semantics and contract tests, and keeps user prompt data in user/runtime context.

No. The async direction is appropriate, but the latest diff keeps synchronous keychain reads in the public call path and adds unrelated prompt and contract-test changes; a focused non-blocking credential path is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Move Codex keychain lookup off the event loop:** `src/agents/cli-credentials.ts:251-253`
  - body: The final diff still runs execSyncImpl for macOS Codex keychain reads, and public readCodexCliCredentials calls this sync helper. That leaves the gateway event loop blocked during the exact keychain lookup this PR is meant to move off-thread.
  - confidence: 0.96
- **[P1] Keep user prompts out of extraSystemPrompt:** `src/agents/pi-embedded-runner/run.ts:365-368`
  - body: This appends raw params.prompt to extraSystemPrompt, which current prompt assembly places in the model system prompt. Labeling it untrusted in text does not preserve user-role isolation.
  - confidence: 0.94
- **[P2] Do not churn sessions with per-turn system prompt text:** `src/agents/pi-embedded-runner/run.ts:371-373`
  - body: Appending the current prompt to extraSystemPrompt changes that field every turn; CLI session reuse hashes that text and invalidates on system-prompt hash mismatch, so normal chat flows can repeatedly lose reusable sessions.
  - confidence: 0.88
- **[P2] Preserve last_refresh anchored keychain expiry:** `src/agents/cli-credentials.ts:265`
  - body: The new Date.now() + 1h fallback makes non-JWT keychain tokens look freshly valid on every read. The previous last_refresh anchor avoided a sliding expiry window for stale credentials.
  - confidence: 0.87
- **[P2] Keep provider compat IDs an exact set:** `src/plugins/contracts/loader.contract.test.ts:84`
  - body: Switching this exact equality to arrayContaining lets extra provider IDs enter the compat list without failing the contract test, weakening the guard against unintended plugin-scope expansion.
  - confidence: 0.88
- **[P2] Restore exact registry coverage assertions:** `src/plugins/contracts/registry.contract.test.ts:21-22`
  - body: The subset assertion allows runtime registry entries to disappear while the manifest list still contains them, so the contract no longer verifies that registry coverage matches every bundled capability.
  - confidence: 0.89
