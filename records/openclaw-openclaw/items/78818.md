## What This Changes

The branch changes CLI credential keychain readers, adds an embedded-runner last-message prompt anchor, and relaxes two plugin contract test assertions.

## Best Possible Solution

Land a focused auth/gateway fix that moves proven macOS keychain reads off the event loop while preserving sync caller contracts, credential expiry semantics, prompt authority boundaries, and exact plugin contract tests.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection confirms both current main and the PR head still route macOS keychain reads through synchronous `execSync` calls, matching the reported event-loop warning path.

## Solution Assessment

No. The async direction is plausible, but this PR head keeps the blocking sync keychain read and adds unrelated prompt/security and contract-test changes; a narrower auth/gateway revision is the maintainable fix.

- **[P1] Move keychain lookup off the event loop:** `src/agents/cli-credentials.ts:251-253`
  - body: The latest head still calls `execSyncImpl` for `security find-generic-password` inside the public credential path, so the gateway-capable macOS keychain read remains a blocking event-loop operation and the central bug is not fixed.
  - confidence: 0.96

  - body: This appends raw `params.prompt` into `extraSystemPrompt`, which current prompt assembly places in the model system prompt section. Marking the text untrusted does not preserve user-role authority isolation.
  - confidence: 0.94
- **[P2] Keep per-turn prompt text out of session hashes:** `src/agents/pi-embedded-runner/run.ts:371-373`
  - body: Appending the current user message to `extraSystemPrompt` changes that field every turn, and CLI session reuse hashes this prompt material. Normal chat flows can repeatedly invalidate reusable sessions as `system-prompt` mismatches.
  - confidence: 0.88
- **[P2] Preserve last_refresh anchored keychain expiry:** `src/agents/cli-credentials.ts:265`
  - body: The fallback `Date.now() + 60 * 60 * 1000` makes non-JWT keychain credentials look freshly valid on every read. The current `last_refresh` anchor avoids a sliding expiry window for stale credentials.
  - confidence: 0.87
- **[P2] Keep provider compat contract as an exact set check:** `src/plugins/contracts/loader.contract.test.ts:84`
  - body: Switching this assertion to `arrayContaining` allows extra provider IDs in the compatibility list without failing the contract, weakening the guard against unintended bundled provider-scope expansion.
  - confidence: 0.88
- **[P2] Restore exact registry coverage assertions:** `src/plugins/contracts/registry.contract.test.ts:21-22`
  - body: The subset assertion allows runtime registry entries to disappear while manifest IDs remain, so the contract no longer verifies that registry coverage matches every bundled capability.
  - confidence: 0.89
