## Summary

Keep open: the latest PR head is not mergeable and does not fix the reported gateway blocking. It leaves synchronous macOS keychain calls in the credential path, has a stale `execAsync` reference after removing that option from the type, and includes unrelated prompt-authority and plugin contract-test changes.

## What This Changes

The PR changes CLI credential keychain reads, injects the current user prompt into `extraSystemPrompt`, and loosens two plugin contract assertions.

## Best Possible Solution

Land a focused gateway/auth fix that moves hot keychain credential reads onto an awaitable non-blocking path without changing prompt authority or plugin contract coverage, then add regression coverage and fresh macOS proof.

No high-confidence live reproduction was run here. Source inspection on current main confirms the synchronous macOS `security find-generic-password` path that can block the Node event loop, and the linked issue plus PR body provide matching liveness output.

No. The async direction is appropriate, but this PR’s latest head removed the async implementation from the live call path and added unrelated prompt and plugin-test changes.

  - body: The final head removed `execAsync` from these option types, but this call still passes `options?.execAsync`. That should fail TypeScript before the branch can build or run tests.

  - confidence: 0.96
- **[P1] Keep user prompts out of extraSystemPrompt:** `src/agents/pi-embedded-runner/run.ts:365-368`
  - body: This copies raw user input into `extraSystemPrompt`, which is appended to the system prompt by the current prompt builder. Marking it untrusted in text does not preserve user-role isolation.
  - confidence: 0.94
- **[P2] Preserve stable keychain token expiry:** `src/agents/cli-credentials.ts:262`
  - body: Replacing the `last_refresh`-anchored fallback with `Date.now() + 60 * 60 * 1000` makes non-JWT keychain tokens look freshly valid on every read, which can prevent stale credential detection from ever expiring them.
  - confidence: 0.87
- **[P2] Keep provider compat IDs an exact set:** `src/plugins/contracts/loader.contract.test.ts:84`
  - body: Changing this exact equality to `arrayContaining` lets extra provider IDs enter the compat list without failing the contract test, weakening the guard against unintended plugin-scope expansion.
  - confidence: 0.88
- **[P2] Restore exact registry coverage assertions:** `src/plugins/contracts/registry.contract.test.ts:21-23`
  - body: The new subset assertion allows runtime registry entries to disappear while the manifest list still contains them, so the contract no longer verifies the registry covers every bundled capability.
  - confidence: 0.89
