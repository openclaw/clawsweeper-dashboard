## Summary

Keep open: current main still forwards a caller-supplied Webhooks `run_task.childSessionKey` into TaskFlow without a route-session-tree guard, and the issue is already paired with an open closing PR that targets the fix.

## What This Changes

The issue asks the Webhooks plugin to reject `run_task.childSessionKey` unless it belongs to the route's bound session tree.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/79041, then close this issue after the merged Webhooks guard is on main.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a route-bound managed flow, call `run_task` with an unrelated `childSessionKey`, then `cancel_flow` reaches cancellation through the stored child session key. I did not execute a live runtime exploit in this read-only review.

## Solution Assessment

Yes, the best fix is a Webhooks-boundary guard that accepts the bound session or recorded spawned/parented child sessions before calling `runTask`. The open linked PR already follows that direction, so this issue should stay paired with that review path.
