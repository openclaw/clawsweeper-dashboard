## Label Justifications

- `P2`: This is a real bundled-plugin security boundary bug with a limited configured-webhook blast radius and an open focused fix path.
- `impact:security`: The issue is about a webhook route exceeding its configured session authority by supplying a foreign child session key.
- `impact:session-state`: The reported path can associate and cancel task work against an unrelated OpenClaw session key.

## Summary

Keep open: current main and the latest release still forward Webhooks `run_task.childSessionKey` into TaskFlow without a route-session-tree guard, and the linked fix at https://github.com/openclaw/openclaw/pull/79041 is open but currently conflicting.

## What This Changes

The issue asks for Webhooks `run_task` to reject caller-supplied `childSessionKey` values outside the configured route session tree.

## Best Possible Solution

Land a narrow Webhooks-owned guard that accepts the route session or recorded child sessions only, with focused handler/plugin-route regression tests and docs for external automation identifiers.

## Reproduction Assessment

Yes. The current source path shows Webhooks forwarding `childSessionKey` unchecked, TaskFlow persisting it on a managed child task, and cancellation later targeting that session key; I did not run a live repro in this read-only review.

## Solution Assessment

Yes. A Webhooks-layer route-session-tree guard is the best owner boundary because Webhooks owns the external input, while the generic TaskFlow runtime still needs to serve trusted plugin callers.
