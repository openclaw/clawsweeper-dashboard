## Summary

Keep open: current main and v2026.6.8 still forward Webhooks `run_task.childSessionKey` into TaskFlow without a route-session-tree guard, and the open linked PR remains the focused fix path rather than a merged implementation.

## What This Changes

The issue asks for Webhooks `run_task.childSessionKey` to be rejected unless it identifies the route-bound session or a recorded child of that session.

## Best Possible Solution

Review, repair or rebase, and land https://github.com/openclaw/openclaw/pull/79041 or an equivalent Webhooks-owned route-session-tree guard, then close this issue after merge.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts `run_task.childSessionKey`, forwards it to TaskFlow, persists it on the task, and later uses it as the ACP/subagent cancellation target.

## Solution Assessment

Yes in direction. Enforcing the guard at the Webhooks route boundary is the narrow owner-boundary fix because Webhooks owns the external input while generic TaskFlow remains available to trusted plugin callers.
