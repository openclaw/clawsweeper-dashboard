## Label Justifications

- `P2`: The issue reports a real bundled-plugin security boundary bug with a bounded configured-Webhooks blast radius and an open focused fix path.
- `impact:session-state`: The reported path can associate and later cancel managed child task work against an unrelated OpenClaw session key.
- `impact:security`: A Webhooks route can exceed its configured session authority by supplying a foreign child session key.

## Summary

Keep open: current main and v2026.6.6 still forward Webhooks `run_task.childSessionKey` into TaskFlow without a route-session-tree guard, and the linked fix remains open and conflicting.

## What This Changes

The issue asks for Webhooks `run_task.childSessionKey` to be rejected unless it names the route-bound session or a recorded child in that route session tree.

## Best Possible Solution

Refresh and review https://github.com/openclaw/openclaw/pull/79041, or land an equivalent Webhooks-layer guard with focused handler/plugin-route regression coverage and docs for `runId`/`sourceId` as external identifiers.

## Reproduction Assessment

Yes, source-reproducible: current main forwards Webhooks `childSessionKey` unchecked into TaskFlow, TaskFlow persists it on a managed child task, and `cancel_flow` later cancels ACP/subagent sessions via that key. I did not run a live HTTP repro in this read-only pass.

## Solution Assessment

Yes. A Webhooks-layer route-session-tree guard is the narrow owner-boundary fix because Webhooks owns the external input while generic TaskFlow still needs to serve trusted plugin callers.
