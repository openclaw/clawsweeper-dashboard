## Label Justifications

- `P2`: The issue reports a bounded bundled-plugin security/session-state bug with an open focused fix path, not an emergency core-runtime outage.
- `impact:security`: A webhook route trusted for one configured session can currently submit a child session key for a different session boundary.
- `impact:session-state`: The reported path can attach managed child task state to a session outside the route's intended session tree.

## Summary

Keep open: current main and v2026.6.10 still forward Webhooks `run_task.childSessionKey` without a route-session-tree guard, and the open linked PR remains the candidate fix rather than a merged implementation.

## What This Changes

The issue requests rejecting Webhooks `run_task.childSessionKey` values that are outside the route-bound session tree.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/79041, or replace it with an equivalent Webhooks-owned route-session-tree guard, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows the Webhooks handler accepts `run_task.childSessionKey`, forwards it into bound TaskFlow task creation, and task cancellation later uses the stored child key as the ACP/subagent session target.

## Solution Assessment

Yes in direction. Enforcing the guard at the Webhooks route boundary is the narrow owner-boundary fix because Webhooks owns the external input while generic TaskFlow remains available to trusted plugin callers.
