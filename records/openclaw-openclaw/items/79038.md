## Summary

Keep open: current main still has the Webhooks/TaskFlow session-scope gap, and https://github.com/openclaw/openclaw/pull/79041 is an open closing PR for the same bug. The issue should stay paired with that implementation path until a fix merges.

## What This Changes

The issue requests a Webhooks `run_task.childSessionKey` guard that only allows the route session or child sessions recorded under that route session.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/79041 with a Webhooks-side session-tree guard, regression coverage for raw and registered routes, and the required docs/changelog follow-through, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a route-bound managed flow, call `run_task` with an unrelated `childSessionKey`, and then cancellation can act on the stored child session key. I did not execute a live current-main exploit path in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to validate `childSessionKey` at the Webhooks boundary before calling `runTask`, using the session-tree visibility model while leaving external automation identifiers on `runId` or `sourceId`.
