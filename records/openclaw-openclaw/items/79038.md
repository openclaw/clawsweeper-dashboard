## Summary

Keep open: current main and v2026.6.6 still forward Webhooks `run_task.childSessionKey` into TaskFlow without a route-session-tree guard, and https://github.com/openclaw/openclaw/pull/79041 is the open closing PR for the focused fix.

## What This Changes

The issue asks the bundled Webhooks plugin to reject `run_task.childSessionKey` values outside the route-bound session tree.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/79041 or an equivalent Webhooks-owned route-session-tree guard, then close this issue after merge.

## Reproduction Assessment

Yes. Current source shows Webhooks forwarding `childSessionKey` unchecked, TaskFlow persisting it on a managed child task, and cancellation later targeting that session key; I did not run a live repro in this read-only review.

## Solution Assessment

Yes. Enforcing the guard at the Webhooks route boundary is the narrow owner-boundary fix because Webhooks owns the external input while generic TaskFlow still serves trusted plugin callers.
