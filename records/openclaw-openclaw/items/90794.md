## Summary

Keep open: the PR fixes the right core outbound path, but the patch introduces a blocking resource-bound regression by using an effectively unbounded media size fallback for buffer-only sends.

## What This Changes

The PR stages top-level buffer-only `message.send` attachments into outbound media and wires the staged path through `media`/`mediaUrl`/`mediaUrls` before channel delivery.

## Best Possible Solution

Keep the buffer-to-outbound-media approach, but preserve the existing default media byte cap when mediaMaxMb is unset and add focused oversized-buffer coverage before merge.

## Reproduction Assessment

Yes by source inspection: current `message.send` builds deliverable media from media/path/URL hints while ordinary send skips attachment hydration, so a top-level buffer without an explicit media path never becomes a media URL.

## Solution Assessment

No as written: materializing buffer-only sends before dispatch is the right boundary, but the implementation must preserve the default media byte cap instead of falling back to `Number.MAX_SAFE_INTEGER`.
