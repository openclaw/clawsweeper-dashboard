## Summary

Keep open: the PR targets a real remaining Telegram subagent forum-topic gap, but the branch is currently conflicting and parts of the implementation now land in the wrong owner boundary after current main moved thread-bound subagent binding into core.

## What This Changes

The PR adds a Telegram per-kind automatic thread-spawn capability, Telegram subagent lifecycle hooks, Gateway progress mirroring, cron delivery-route persistence, docs, prompt snapshots, and regression tests.

## Best Possible Solution

Rebase onto current main, keep the explicit Telegram subagent spawn-capability seam, route binding through the core session-binding adapter, move progress mirroring behind a channel/plugin streaming seam, and remove the changelog edit.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.8 keep Telegram current-placement while thread/session schema is gated on automatic child-placement support, even though the Telegram adapter can create child forum topics.

## Solution Assessment

No. The explicit capability is a plausible direction, but routing through deprecated subagent_spawning and adding a Gateway-owned Telegram mirror are not the best owner boundaries after the merged core thread-binding refactor.
