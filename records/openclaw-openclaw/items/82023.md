## Summary

Keep open: the feature direction addresses a real Telegram thread-bound subagent gap and the supplied live proof is credible, but the new gateway mirror bypasses existing verbose and Telegram streaming controls, so the PR is not merge-ready.

## What This Changes

The PR adds Telegram subagent lifecycle hooks, per-kind automatic thread-spawn capability, docs/tests, and a gateway event mirror for thread-bound Telegram subagent and cron session progress.

## Best Possible Solution

Keep the per-kind Telegram subagent binding direction, but route topic progress through existing channel streaming and verbose-policy controls before resolving conflicts and landing.

## Reproduction Assessment

Yes, at source level: current main keeps Telegram top-level placement as current and registers no Telegram subagent lifecycle hooks, while the PR supplies live after-fix logs. The blocking review findings are also source-reproducible from the added gateway mirror.

## Solution Assessment

No, not as currently implemented: the subagent binding seam is plausible, but the added mirror bypasses established Telegram streaming and verbose-output controls. A safer fix should reuse or align with the existing channel delivery policy before sending topic progress.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor verbose before mirroring command output:** `src/gateway/agent-event-channel-mirror.ts:152-163`
  - body: `formatAgentEventForChannelMirror` formats every `command_output` event into a durable Telegram message, but the mirror is registered globally and never checks the run or session verbose level. The documented behavior only forwards tool output at `/verbose full`, so default topic-bound runs can expose command stdout/stderr to Telegram even when the user left verbose off.
  - confidence: 0.91
- **[P2] Coalesce assistant deltas before sending topic messages:** `src/gateway/agent-event-channel-mirror.ts:95-101`
  - body: `textFromData` returns raw `data.delta`, and the caller sends each formatted event as a new Telegram message. Agent assistant events can be token-sized deltas, while Telegram streaming is expected to coalesce/edit a single preview, so long runs can backlog and spam topic messages instead of producing stable transcript parity.
  - confidence: 0.84
