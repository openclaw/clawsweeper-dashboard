## Summary

Keep open: the Telegram thread-bound subagent capability is useful and has strong live proof, but the branch is dirty and still introduces deprecated hook-based binding plus Telegram-specific Gateway delivery that conflict with current ownership boundaries.

## What This Changes

The PR adds Telegram forum-topic binding for spawned subagents, per-kind automatic thread-spawn capability exposure, Telegram topic progress mirroring, cron delivery-route persistence, docs, snapshots, and regression tests.

## Best Possible Solution

Rebase onto current main, keep the useful explicit Telegram spawn-capability seam, route child-topic binding through the existing core session-binding adapter, move progress mirroring behind a generic channel/plugin-owned streaming contract, and drop the changelog edit.

## Reproduction Assessment

Yes. Source inspection shows current main can create Telegram child-topic bindings through the session-binding adapter, but sessions_spawn thread availability is hidden because Telegram's default top-level placement is current and no explicit spawn capability exists.

## Solution Assessment

No. The missing capability is a good target, but this branch is not the best merge shape because it reintroduces deprecated hook-based binding and adds Telegram-specific Gateway delivery instead of using core binding plus a channel-owned streaming seam.
