## Summary

Keep open. Current main and latest release still preserve the stale Android TLS pin mismatch path after a reverse-proxy certificate rotation, and the duplicate report was already closed into this canonical issue.

## What This Changes

The issue asks Android to recover when a reverse-proxy TLS certificate rotates without requiring users to clear all app data and redo onboarding.

## Best Possible Solution

Add a fail-closed Android stale-pin recovery or reset flow that shows old and new fingerprints and requires explicit user confirmation before replacing a stored gateway TLS pin.

## Reproduction Assessment

Yes, from source inspection: save a TLS fingerprint for a non-loopback Android WSS endpoint, then reconnect or rescan after the same stable endpoint presents a different leaf certificate. Current main reuses the stored pin with `allowTOFU = false` and reaches the reported mismatch throw.

## Solution Assessment

Unclear. A user-confirmed pin replacement or reset flow is likely the safest direction, while disabling pinning or accepting changed certificates silently would not be a maintainable security fix.
