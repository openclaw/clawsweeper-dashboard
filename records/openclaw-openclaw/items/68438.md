## Summary

Keep open. Current main and the latest release still preserve the Android stale TLS pin path after reverse-proxy certificate rotation, and the duplicate report is already closed into this canonical issue.

## What This Changes

Handle Android gateway TLS certificate rotation behind reverse proxies without requiring users to clear app data and repeat onboarding.

## Best Possible Solution

Add a fail-closed Android stale-pin recovery or reset flow that shows the old and new fingerprints and requires explicit user confirmation before replacing a stored gateway TLS pin.

## Reproduction Assessment

Yes, from source inspection: save a TLS fingerprint for a non-loopback Android WSS endpoint, then reconnect or rescan after the same stable endpoint presents a different leaf certificate. Current main reuses the stored pin with `allowTOFU = false` and reaches the reported mismatch throw.

## Solution Assessment

Unclear. A user-confirmed pin replacement or reset flow is likely the safest direction, while disabling pinning or accepting changed certificates silently would not be a maintainable security fix.
