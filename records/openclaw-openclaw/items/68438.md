## Summary

Keep open. Current main and v2026.5.12 still preserve the reported Android stale TLS pin path: stored pins are reused with `allowTOFU = false`, and a changed leaf certificate reaches the reported `gateway TLS fingerprint mismatch`; the duplicate report is already closed into this canonical issue.

## What This Changes

The issue asks Android to recover from reverse-proxy TLS certificate rotation without requiring users to clear app data and redo onboarding.

## Best Possible Solution

Add a fail-closed Android stale-pin recovery or reset flow that shows the old and new fingerprints and requires explicit user confirmation before replacing a stored gateway TLS pin.

## Reproduction Assessment

Yes, from source inspection: save a TLS fingerprint for a non-loopback Android `wss://` endpoint, then reconnect or rescan after the same stable endpoint presents a different leaf certificate. Current main reuses the stored pin with `allowTOFU = false` and reaches the reported mismatch throw.

## Solution Assessment

Unclear. A user-confirmed pin replacement or reset flow appears safest, while disabling pinning or accepting changed certificates silently would not be a maintainable security fix.
