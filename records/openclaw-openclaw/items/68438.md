## Summary

Keep open. Current main still persists Android gateway TLS pins by endpoint and rejects a later certificate with the reported mismatch error, while the trust prompt only runs before a stored pin exists. The report is plausible and security-sensitive, so cleanup closure is not appropriate without maintainer security/product judgment.

## What This Changes

The issue requests an Android recovery path for reverse-proxy TLS certificate renewal so users are not forced to clear app data after a stored gateway certificate fingerprint changes.

## Best Possible Solution

Add an Android certificate-change recovery flow that remains fail-closed by default, shows the old and new SHA-256 fingerprints with a clear trust warning, and only replaces the stored pin after explicit user confirmation or another maintainer-approved verification policy. Cover manual reverse-proxy `wss://` endpoints and discovered endpoints with focused Android tests, and keep unauthenticated TXT fingerprints from overriding stored trust.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: store a fingerprint for a manual non-loopback `wss://` endpoint, reconnect after the server presents a different certificate, and the current TLS trust manager throws the reported mismatch while the first-use prompt is skipped because `expectedFingerprint` is already set. I did not run a live Android reverse-proxy renewal test in this read-only review.

## Solution Assessment

Unclear. A deliberate replace-pin prompt is a maintainable direction, but disabling pinning whenever setup codes omit `tlsFingerprint` or adding a broad “disable TLS fingerprint check” switch would be a security regression unless maintainers explicitly approve that trust model.
