## Summary

Keep open. Current main and the latest release still keep Android's stored TLS pin for the same gateway stable ID and fail closed on a renewed reverse-proxy certificate, while the related newer report was closed as a duplicate of this canonical issue.

## What This Changes

The issue requests Android recovery from a stale gateway TLS fingerprint after a reverse proxy certificate rotates.

## Best Possible Solution

Add an explicit user-confirmed stale-pin recovery path on Android, such as a mismatch prompt or reset saved fingerprint action, while preserving fail-closed pinning and never trusting advertised fingerprints silently.

## Reproduction Assessment

Yes from source inspection, though not from a live Android run. Save a `gateway.tls.<stableId>` pin, reconnect the same endpoint after it presents a different certificate, and Android continues using the stored pin until `GatewayTls.kt` throws the reported mismatch.

## Solution Assessment

No for the unsafe variants in the report, such as disabling pinning or silently accepting a new fingerprint. The maintainable fix should require deliberate user confirmation or an explicit reset before replacing the stored pin.
