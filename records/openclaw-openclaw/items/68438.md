## Summary

Keep open. Current main still rejects Android WSS reconnects when a stored gateway TLS leaf fingerprint changes, and the only trust prompt is the first-time path before a pin exists; changing recovery behavior is security-sensitive and needs maintainer product/security review.

## What This Changes

Request Android recovery for reverse-proxy TLS certificate rotation without requiring users to clear app data and redo onboarding.

## Best Possible Solution

Add a fail-closed Android stale-certificate recovery flow that shows old and new fingerprints, requires deliberate user trust before replacing the stored pin, and covers manual `wss://` plus discovered secure endpoints without trusting unauthenticated TXT fingerprints.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: save a TLS fingerprint for a non-loopback Android `wss://` endpoint, then reconnect after the server presents a different certificate; current code skips the first-time prompt and throws the reported mismatch during TLS validation.

## Solution Assessment

Unclear. The requested recovery direction is valid, but the safest solution is a maintainer-approved explicit pin-replacement prompt, not disabling pinning or accepting new fingerprints silently.
