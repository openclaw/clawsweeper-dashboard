## Summary

Keep open. Current main still pins the Android gateway TLS leaf fingerprint per endpoint and rejects changed certificates, while the only trust prompt is the first-time path before a stored pin exists. The report is plausible, user-impacting, and security-sensitive, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

Android node should recover from reverse-proxy TLS certificate renewal without requiring app-data reset, while preserving the gateway TLS pinning trust model.

## Best Possible Solution

Add an Android certificate-change recovery flow that remains fail-closed by default but lets the user deliberately replace a stale pin after seeing the old and new fingerprints plus a clear trust warning. Cover manual `wss://` reverse-proxy endpoints and discovered endpoints with focused Android tests, and avoid trusting unauthenticated TXT fingerprints or silently disabling pins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: save a trusted fingerprint for a manual non-loopback `wss://` endpoint, then reconnect when the server presents a different certificate; current validation throws the mismatch error and the first-time trust prompt is skipped because `expectedFingerprint` is already set.

## Solution Assessment

Unclear. The requested recovery direction is valid, but the safest fix is a deliberate certificate-change prompt or maintainer-approved trust policy, not wholesale pin disabling when setup codes omit `tlsFingerprint`.
