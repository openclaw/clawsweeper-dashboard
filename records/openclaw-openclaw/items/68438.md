## Summary

Keep open. Current main and the latest release still fail closed when Android reconnects to a stored WSS gateway pin after the server leaf certificate changes, and changing stale-pin recovery semantics needs maintainer product/security review.

## What This Changes

The issue asks Android to recover from renewed reverse-proxy TLS certificates without requiring users to clear app data and redo onboarding.

## Best Possible Solution

Add a fail-closed Android stale-pin recovery path that shows old and new fingerprints and requires explicit user confirmation before replacing a stored pin.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store a TLS fingerprint for a non-loopback Android WSS endpoint, then reconnect after the endpoint presents a different leaf certificate; current code reuses the stored pin and throws the reported mismatch.

## Solution Assessment

Unclear. The requested recovery direction is valid, but the safest maintainable fix is an explicit, maintainer-approved pin-replacement flow rather than disabling pinning or accepting changed fingerprints silently.
