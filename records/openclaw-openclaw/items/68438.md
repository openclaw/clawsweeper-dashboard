## Summary

Keep open. Current main and v2026.5.7 still reuse the stored Android TLS pin for the same gateway endpoint and fail closed on renewed leaf certificates; the remaining fix needs product/security review because stale-pin replacement can weaken impersonation protection.

## What This Changes

The issue requests an Android recovery path for stale gateway TLS fingerprints after a reverse-proxy certificate renewal.

## Best Possible Solution

Add a fail-closed Android recovery path that shows old and new fingerprints and requires explicit user confirmation before replacing or clearing a stored pin.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store a TLS fingerprint for a non-loopback WSS endpoint, then reconnect or rescan setup code after the same endpoint presents a different leaf certificate; Android keeps using the stored pin until `GatewayTls.kt` throws the reported mismatch.

## Solution Assessment

Unclear. The recovery direction is valid, but the best maintainable fix needs explicit maintainer-approved stale-pin replacement or reset semantics rather than silent pin updates or a pinning-disable option.
