## Summary

Keep open. Current main and v2026.5.7 still fail closed on an Android stored TLS fingerprint after a same-endpoint certificate rotation, and the recovery policy is security-sensitive rather than a cleanup-close candidate.

## What This Changes

The issue requests an Android recovery path for TLS fingerprint changes after a reverse-proxy certificate renewal.

## Best Possible Solution

Add a fail-closed Android pin recovery or reset flow that shows old and new fingerprints and requires explicit user confirmation before replacing a stored pin.

## Reproduction Assessment

Yes, by source inspection: save a TLS fingerprint for a non-loopback WSS endpoint, then reconnect or rescan setup after the same stable endpoint presents a different leaf certificate; Android reuses the stored pin and throws the reported mismatch.

## Solution Assessment

Unclear. The requested recovery direction is valid, but the maintainable solution should be an explicit pin replacement/reset flow rather than disabling pinning or silently trusting a renewed certificate.
