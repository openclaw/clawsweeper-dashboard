## Summary

Keep open. Current main and the latest release still keep Android on the stored TLS pin mismatch path after a reverse-proxy certificate rotates, and the remaining fix needs product/security approval because pin replacement must remain fail-closed.

## What This Changes

The issue requests an Android recovery path for renewed reverse-proxy TLS certificates so users do not have to clear app data after a stored gateway fingerprint becomes stale.

## Best Possible Solution

Add a fail-closed Android stale-pin recovery or reset flow that shows the old and new fingerprints and requires explicit user confirmation before replacing a stored gateway TLS pin.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: save a TLS fingerprint for a non-loopback Android WSS endpoint, then reconnect or rescan after the same stable endpoint presents a different leaf certificate; current main reuses the stored pin and throws the reported mismatch.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A user-confirmed pin replacement or reset flow is likely the safest direction, while disabling pinning or accepting changed certificates silently would not be a maintainable security fix.
