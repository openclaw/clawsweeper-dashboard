## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` HTML without preserving fence-language metadata, and the related WebChat copy-button issue is a separate UI surface.

## What This Changes

Add Telegram HTML output support for fenced-code language identifiers such as `class="language-bash"` so Telegram can show native code-block copy controls.

## Best Possible Solution

Carry sanitized optional fence-language metadata through Telegram rendering and render-aware chunking, then cover fenced, unfenced, escaped, file-reference, and chunk-limit cases with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows `markdownToTelegramHtml("```js\nconst x = 1;\n```")` follows the static `code_block` marker, and the existing formatter test asserts output without `class="language-js"`.

## Solution Assessment

Yes. Preserving sanitized fence-language metadata through the shared Markdown IR/render-aware chunking path, or an equivalently narrow Telegram-local renderer if maintainers avoid shared type expansion, is the narrow maintainable fix; WebChat copy-button work does not reach Telegram outbound messages.
