## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` without preserving the fence language, and the related WebChat copy-button work is a separate renderer path.

## What This Changes

Preserve fenced-code language identifiers in Telegram HTML code blocks so Telegram can use native code-block affordances.

## Best Possible Solution

Carry optional fenced-code language metadata through the markdown render/chunking path, or an equivalently narrow Telegram renderer, and cover it with Telegram formatter and chunking regressions plus a user-facing changelog entry.

## Reproduction Assessment

Yes for the current missing behavior. Source inspection shows `markdownToTelegramHtml("```js\nconst x = 1;\n```")` still reaches a static Telegram code-block marker, and the existing formatter test expects no `language-js` class.

## Solution Assessment

Yes. The requested Telegram HTML shape is the narrow maintainable direction, provided the implementation carries sanitized fence metadata through rendering and chunking rather than interpolating raw fence info directly.
