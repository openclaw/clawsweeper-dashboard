## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` HTML without preserving the fence language, while the related WebChat copy-button work is a separate renderer path.

## What This Changes

Add Telegram fenced-code language classes to outbound HTML so Telegram clients can use native code-block copy support.

## Best Possible Solution

Carry sanitized optional fence-language metadata through Telegram rendering and chunking, then cover fenced, unfenced, escaped, file-reference, and chunk-limit cases with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: `markdownToTelegramHtml("```js\nconst x = 1;\n```")` follows the static `code_block` marker, and the formatter test still asserts output without `class="language-js"`.

## Solution Assessment

Yes. Preserving sanitized fence language metadata through shared Markdown IR/render-aware chunking, or an equivalently narrow Telegram-local renderer if maintainers avoid SDK surface expansion, is the narrow maintainable fix; WebChat's renderer does not reach Telegram outbound messages.
