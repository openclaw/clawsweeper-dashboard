## Summary

Keep open. Current main and the latest release still render Telegram fenced code as static `<pre><code>` and the shared Markdown IR drops fence language metadata before Telegram rendering, so the requested Telegram-native copy support is not implemented; the related WebChat copy-button issue is a separate UI path.

## What This Changes

The issue requests preserving fenced-code language identifiers in Telegram HTML code blocks so Telegram can show native copy controls.

## Best Possible Solution

Add optional fenced-code language metadata to the shared Markdown rendering path, or a narrow equivalent, and update Telegram formatter/chunking tests to emit sanitized `language-X` classes only for fenced blocks while preserving existing escaping and fallback behavior.

## Reproduction Assessment

Yes, at source level: current main has a Telegram formatter test for a `js` fence that expects no class, and the shared IR drops `token.info` before rendering. I did not run tests or a live Telegram probe because this was a read-only review.

## Solution Assessment

Yes, preserving fence language through the Markdown IR/rendering contract and rendering a sanitized Telegram `class="language-X"` is the narrowest durable direction. A Telegram-only fence-aware renderer is a fallback if maintainers do not want to widen the shared IR metadata.
