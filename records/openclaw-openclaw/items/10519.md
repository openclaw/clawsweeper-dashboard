## Summary

Keep open. Current main and v2026.5.12 still render Telegram fenced code as static `<pre><code>` and discard markdown fence language metadata before Telegram HTML output; the related WebChat copy-button item is a separate UI renderer path.

## What This Changes

The issue requests preserving fenced-code language identifiers in Telegram HTML code blocks so Telegram can show native code-copy affordances.

## Best Possible Solution

Add sanitized optional fenced-code language metadata to the Markdown IR/render-aware chunking path, then assert Telegram `class="language-X"` output in formatter and chunking tests.

## Reproduction Assessment

Yes, at source level: current main has a Telegram formatter test for a `js` fence that still expects no class, and `src/markdown/ir.ts` discards fence metadata before rendering. I did not run Vitest or live Telegram in this read-only review.

## Solution Assessment

Yes. Preserving a sanitized first fence-language token through the shared Markdown rendering/chunking contract is the narrowest maintainable path; a Telegram-local renderer is a fallback if maintainers reject widening the IR metadata.
