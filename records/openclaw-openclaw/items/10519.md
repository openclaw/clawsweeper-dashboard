## Summary

Keep open. Current main still drops fenced-code language metadata before Telegram rendering, the shipped v2026.5.12 source still emits static Telegram code-block markers, and the related WebChat copy-button issue is a separate UI path rather than a Telegram Bot API HTML fix.

## What This Changes

Request preserving markdown fenced-code language identifiers in Telegram HTML code blocks to enable Telegram native copy button support.

## Best Possible Solution

Add sanitized optional fenced-code language metadata to the shared Markdown IR/render-aware chunking path, allow only safe Telegram `code class="language-X"` HTML, and cover Telegram formatter plus chunking regressions.

## Reproduction Assessment

Yes, at source level: current main has a Telegram formatter test for a `js` fence that expects no class, and `src/markdown/ir.ts` drops fence metadata before rendering. I did not run Vitest or live Telegram in this read-only review.

## Solution Assessment

Yes. Preserving a sanitized first fence-language token through the shared Markdown rendering/chunking contract is the narrowest maintainable path; a Telegram-local renderer is only a fallback if maintainers reject widening the IR metadata.
