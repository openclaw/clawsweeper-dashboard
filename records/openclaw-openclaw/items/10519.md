## Summary

Keep open. Current main and the latest release still discard fenced-code language metadata before Telegram rendering and chunking; the related WebChat copy/language-class work is a separate renderer path and does not affect Telegram Bot API HTML output.

## What This Changes

Request Telegram HTML rendering preserve fenced-code language identifiers as `<pre><code class="language-X">...` so Telegram can show its native code-block copy UI.

## Best Possible Solution

Preserve optional fenced-code language metadata through the shared Markdown IR/chunking path, or an equivalent narrowly scoped Telegram renderer, and emit a sanitized `language-X` class only when a fence language is present.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: `markdownToTelegramHtml("```js\nconst x = 1;\n```")` is still covered by a test expecting a plain `<pre><code>` block without `class="language-js"`.

## Solution Assessment

Yes, the requested Telegram HTML output is the right user-facing direction. The maintainable fix should carry language metadata through rendering/chunking rather than only replacing the static marker string.
