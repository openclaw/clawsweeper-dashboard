## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` without carrying the fence language, while the WebChat copy-button work is a separate UI renderer path and does not satisfy Telegram Bot API HTML output for this issue.

## What This Changes

Request to preserve Telegram Markdown fence language metadata and emit `class="language-X"` on outbound `<pre><code>` blocks.

## Best Possible Solution

Preserve optional fenced-code language metadata through the shared markdown IR/render-aware chunking path, or an equivalently narrow Telegram renderer, and add regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes for the current missing behavior. Source inspection shows `markdownToTelegramHtml("```js\nconst x = 1;\n```")` still reaches a static Telegram code-block marker, and the existing formatter test expects no `language-js` class.

## Solution Assessment

Yes. The requested Telegram HTML shape is the right direction, provided the implementation carries sanitized fence metadata through rendering and chunking rather than interpolating raw fence info directly.
