## Summary

Keep this issue open. Current main and the latest release still render Telegram fenced code blocks as plain `<pre><code>` without preserving the fence language, while the separate WebChat copy-button/language-class path is UI-only and does not affect Telegram Bot API HTML output.

## What This Changes

Request preservation of fenced-code language identifiers in Telegram HTML code blocks as `class="language-X"` so Telegram can show its native copy affordance.

## Best Possible Solution

Preserve fenced-code language metadata for Telegram HTML rendering and chunking, preferably by extending the internal Markdown IR with optional code-block language metadata and teaching the Telegram renderer to emit a sanitized `language-X` class only when present.

## Reproduction Assessment

Yes. The current formatter source and its existing unit test provide a high-confidence reproduction path: `markdownToTelegramHtml("```js\nconst x = 1;\n```")` is still expected to return a `<pre><code>` block without `class="language-js"`.

## Solution Assessment

Yes, the requested output is the right user-facing fix, but the implementation should not only replace the static marker string; it needs to carry the fence info through the IR or an equivalent Telegram-specific renderer so chunking preserves the class too.
