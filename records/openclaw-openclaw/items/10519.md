## Summary

Keep open. Current main and v2026.5.12 still discard fenced-code language metadata before Telegram HTML rendering; the WebChat copy-button implementation is a separate UI renderer path, and no linked PR is closing this issue.

## What This Changes

The issue requests preserving fenced code-block language identifiers in Telegram HTML as `class="language-X"` so Telegram can show native copy controls.

## Best Possible Solution

Preserve a sanitized first fence-language token through Telegram-safe Markdown rendering/chunking and emit `code class="language-X"` only for safe fenced blocks, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main still has a Telegram formatter test for a `js` fence that expects no class, and the shared Markdown IR drops fence metadata before Telegram rendering. I did not run Vitest or live Telegram in this read-only review.

## Solution Assessment

Yes. Preserving and sanitizing the first fence info token through the shared Markdown rendering/chunking contract is the cleanest path; a Telegram-local fence-aware renderer is acceptable only if maintainers want to avoid widening the IR shape.
