## Summary

Keep open. Current main and the latest tagged release still render Telegram fenced code blocks as plain `<pre><code>` without preserving the fence language, and the related WebChat language-class/copy-button path is a separate UI renderer rather than Telegram Bot API output.

## What This Changes

The issue requests preserving fenced Markdown language identifiers in Telegram HTML code blocks so Telegram can show native copy controls.

## Best Possible Solution

Preserve a sanitized first fence-language token through the shared Markdown IR/render-aware chunking path and emit `class="language-X"` only for safe Telegram fenced code blocks while leaving unlabeled or invalid fences unchanged.

## Reproduction Assessment

Yes, at source level: current main still renders a `js` fence as `<pre><code>` without a class and drops `markdown-it` `token.info` before Telegram rendering. I did not run Vitest or a live Telegram send in this read-only review.

## Solution Assessment

Yes. Carrying a sanitized fence-language field through the existing Markdown IR and render-aware chunking contract is the narrowest maintainable fix; a Telegram-local fence-aware renderer is only a fallback if maintainers reject widening the IR shape.
