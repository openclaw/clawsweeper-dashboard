## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks with a static `<pre><code>` marker, and the shared Markdown IR still drops the fence language before Telegram rendering; the related WebChat copy-button work is a separate UI path.

## What This Changes

Request Telegram Markdown rendering to preserve fenced-code language identifiers as `class="language-*"` on HTML code blocks for native Telegram copy support.

## Best Possible Solution

Carry sanitized optional fence-language metadata through the Telegram render/chunk path and emit `class="language-X"` for fenced blocks while preserving existing escaping, file-reference wrapping, and chunk-limit behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a ` ```js ` fence is still tested to render as `<pre><code>...` without a language class, and outbound Telegram delivery uses that same markdown chunker.

## Solution Assessment

Yes. Preserving a sanitized first fence-info token in the Telegram rendering/chunking path is the narrow maintainable direction; the WebChat renderer is a different surface and does not solve Telegram output.
