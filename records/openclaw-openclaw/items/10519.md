## Summary

Keep open: current main and the latest release still drop fenced-code language metadata before Telegram HTML rendering, and the related WebChat copy-button work is a separate renderer path that does not fix Telegram Bot API output.

## What This Changes

Add Telegram HTML language classes for Markdown fenced code blocks so Telegram clients can use their native code-block copy UI.

## Best Possible Solution

Teach Telegram markdown rendering to retain a sanitized fence language and emit `class="language-X"` through normal and chunked send paths, with regression coverage around escaping, file references, and chunk limits.

## Reproduction Assessment

Yes, source-based: `markdownToTelegramHtml("```js\n...\n```")` is still asserted to render without a language class, and outbound Telegram delivery uses that renderer. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Yes directionally: preserving sanitized fence metadata is the narrow maintainable solution. If maintainers do not want to widen MarkdownIR, a narrowly scoped Telegram fence-aware renderer would be safer than adding a config option.
