## Summary

Keep open. Current main still renders Telegram fenced code blocks as static `<pre><code>` and drops the fence language before Telegram HTML rendering, so the requested native Telegram copy-button support is not implemented.

## What This Changes

The issue requests preserving fenced-code language identifiers in Telegram HTML code blocks for native Telegram copy support.

## Best Possible Solution

Add sanitized optional fenced-code language metadata to the Markdown rendering/chunking path, or a narrow Telegram-local equivalent, and update Telegram formatter/chunking tests to emit `class="language-X"` only for fenced blocks.

## Reproduction Assessment

Yes, at source level: current main has a Telegram formatter test for a `js` fence that expects no class, and the shared IR drops `token.info` before rendering. I did not run tests or live Telegram because this was a read-only review.

## Solution Assessment

Yes. Preserving sanitized fence language metadata through the shared Markdown IR/render-aware chunking contract is the most durable direction; a Telegram-only fence-aware renderer is the fallback if maintainers do not want to widen the IR metadata.
