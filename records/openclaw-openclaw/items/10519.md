## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` blocks, the existing formatter test asserts the no-language-class output, and the related WebChat copy-button work is a separate UI renderer path.

## What This Changes

Request adds Telegram HTML language classes from fenced Markdown code blocks, e.g. `class="language-bash"`, so Telegram can expose native copy affordances.

## Best Possible Solution

Carry sanitized optional fence-language metadata through the Telegram render/chunk path, emit `class="language-X"` for fenced code blocks, and cover fenced, unfenced, escaped, file-reference, and chunk-limit cases.

## Reproduction Assessment

Yes. Source inspection on current main shows a ` ```js ` fence is tested to render as `<pre><code>...` without a language class, and Telegram outbound delivery uses that same markdown chunker.

## Solution Assessment

Yes. Preserving sanitized fence-language metadata in the Telegram rendering/chunking path is the narrow maintainable fix; the WebChat copy-button renderer is a different surface and does not change Telegram Bot API HTML output.
