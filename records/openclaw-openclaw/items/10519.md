## Summary

Keep open. Current main still drops fenced-code language metadata before Telegram HTML rendering, and the latest shipped release shows the same behavior; the related WebChat copy-button work is a separate UI renderer path and does not satisfy Telegram Bot API HTML output.

## What This Changes

Add Telegram HTML output support for preserving fenced-code language identifiers as `class="language-X"` on `<code>` inside `<pre>` blocks.

## Best Possible Solution

Preserve a sanitized first fence-language token through Markdown rendering/chunking and emit only safe Telegram `code class="language-X"` attributes, with focused Telegram formatter and chunking regression coverage.

## Reproduction Assessment

Yes, at source level: current main still has a Telegram formatter test for a `js` fence that expects no class, and the shared Markdown IR drops fence metadata before Telegram rendering. I did not run Vitest or live Telegram in this read-only review.

## Solution Assessment

Yes. Preserving sanitized fence-language metadata through the shared Markdown render/chunking contract is the narrowest maintainable solution; a Telegram-local fence-aware renderer is a fallback if maintainers do not want to widen the IR shape.
