## Summary

Keep open. Current `main` and the latest tagged release still render Telegram fenced code blocks as plain `<pre><code>` without preserving the fence language, and the closed related PR attempts were not merged into `main`.

## What This Changes

Requested Telegram Markdown fenced code blocks to preserve their fence language as a Bot API-compatible `class="language-X"` attribute.

## Best Possible Solution

Preserve a sanitized fenced-code language through the existing Markdown IR/render-aware chunking path and emit it in Telegram HTML while keeping unlabeled or invalid fences unchanged.

## Reproduction Assessment

Yes. At source level, current `main` still renders a `js` fence as `<pre><code>` without a class and drops `markdown-it` `token.info` before Telegram rendering; I did not run Vitest or a live Telegram send in this read-only review.

## Solution Assessment

Yes. Carrying a sanitized fence-language field through the existing Markdown IR and render-aware chunking contract is the narrowest maintainable fix; a Telegram-local fence-aware renderer is only a fallback if maintainers reject widening the IR shape.
