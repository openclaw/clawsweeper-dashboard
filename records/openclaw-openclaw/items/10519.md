## Summary

Keep open. Current main still drops fenced-code language metadata before Telegram HTML rendering, and the latest release still contains the same static `<pre><code>` output path; the closed WebChat copy-button item is a separate UI renderer path.

## What This Changes

Add Telegram HTML code-block language classes such as `class="language-bash"` when markdown fenced code blocks include a language identifier.

## Best Possible Solution

Preserve optional fenced-code language metadata through the shared markdown IR/chunking path, or an equivalent narrowly scoped Telegram renderer, and emit a sanitized `language-X` class only when a fence language is present.

## Reproduction Assessment

Yes for the current missing behavior. Source inspection shows `markdownToTelegramHtml("```js\nconst x = 1;\n```")` still flows through a static Telegram code-block marker, and the existing test expects no `language-js` class.

## Solution Assessment

Yes. The requested Telegram HTML output is the right user-facing direction, but the maintainable fix should carry sanitized fence metadata through rendering and chunking rather than interpolating unsanitized raw input.
