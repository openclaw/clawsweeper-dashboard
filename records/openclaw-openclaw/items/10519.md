## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as plain `<pre><code>` while discarding the Markdown fence language, and the three related fix PRs are closed unmerged.

## What This Changes

The issue requests Telegram fenced code blocks preserve Markdown fence language metadata as Bot API HTML `language-*` classes for native copy-button support.

## Best Possible Solution

Preserve a sanitized first fence-language token through the shared Markdown IR/render-aware chunking path and emit Telegram `class="language-X"` only for safe fenced code blocks, leaving unlabeled or invalid fences unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows a `js` fence is still expected to render as plain `<pre><code>`, and `markdown-it` fence `token.info` is discarded before Telegram rendering.

## Solution Assessment

Yes. The maintainable fix is to carry sanitized fence-language metadata through the existing Markdown IR and render-aware chunking contract; string-grepping Markdown in Telegram send code would be a more fragile workaround.
