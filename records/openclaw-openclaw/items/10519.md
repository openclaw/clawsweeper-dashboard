## Summary

Current main still drops Telegram fenced-code language metadata before rendering, and the latest release has the same static `<pre><code>` output. The related WebChat copy-button issue is a separate UI renderer path, so this remains valid Telegram work.

## What This Changes

Request Telegram outbound Markdown rendering to include a sanitized `language-X` class on fenced code blocks so Telegram can show native copy support.

## Best Possible Solution

Carry a sanitized optional fence language through the Telegram render/chunk path and emit `class="language-X"` only for fenced code blocks with a valid language identifier.

## Reproduction Assessment

Yes. Source inspection on current main shows a `js` fence is still tested to render as `<pre><code>...` without a language class, and Telegram outbound delivery uses that same markdown chunker.

## Solution Assessment

Yes, directionally. Preserving sanitized fence metadata in the Telegram rendering path is the narrow maintainable solution, but the shared-IR versus Telegram-specific implementation choice needs maintainer judgment.
