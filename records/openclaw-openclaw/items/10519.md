## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` blocks without preserving the fence language, while Telegram’s documented HTML format supports `class="language-..."` on preformatted code.

## What This Changes

Request to preserve fenced-code language identifiers in Telegram HTML as `class="language-X"` so Telegram clients can expose native code-block copy behavior.

## Best Possible Solution

Carry sanitized optional fence-language metadata through Telegram rendering and render-aware chunking, emit `class="language-X"` for fenced blocks, and cover fenced, unfenced, escaped, file-reference, and chunk-limit cases with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows a ` ```js ` fence becomes a `code_block` span with a static `<pre><code>` marker, and the existing Telegram formatter test asserts the no-language-class output.

## Solution Assessment

Yes. Preserving sanitized fence-language metadata through the Telegram render/chunk path is the narrow maintainable fix; the related WebChat copy-button work is a separate UI renderer path and does not affect Telegram Bot API HTML output.
