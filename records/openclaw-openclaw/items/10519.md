## Summary

Keep this open. Current main and the latest release still render Telegram fenced code blocks as plain `<pre><code>` and drop the fence language before Telegram rendering; the related WebChat copy-button work is a separate UI renderer path and does not satisfy Telegram output.

## What This Changes

The issue asks Telegram rendering to preserve fenced-code language metadata as `class="language-*"` on HTML code blocks.

## Best Possible Solution

Carry a sanitized optional fence language through Markdown IR/chunking and have Telegram render `<pre><code class="language-X">` for fenced blocks while preserving existing escaping, file-reference wrapping, and chunk-limit behavior.

## Reproduction Assessment

A bug-style reproduction is not applicable because this is an enhancement, but source inspection gives a high-confidence current-main path: a ```js fence becomes a generic `code_block` span and the Telegram formatter test still expects no language class.

## Solution Assessment

Yes. The maintainable fix is to preserve optional fence language metadata through the existing Markdown rendering pipeline; a Telegram-specific reparse should be a fallback only if maintainers reject adding that metadata to the shared IR.
