## Summary

Keep open. Current main and the latest shipped release still emit static Telegram `<pre><code>` blocks and discard fenced-code language metadata before rendering; the related WebChat and broad command-block issues do not supersede this narrower Telegram request.

## What This Changes

The issue asks OpenClaw to preserve Markdown fence language identifiers in Telegram HTML as `class="language-X"` so Telegram can show native code-block copy support.

## Best Possible Solution

Preserve a sanitized first fence-language token through Markdown rendering and chunking, then emit safe Telegram `code class="language-X"` attributes with focused formatter, outbound, and chunking coverage.

## Reproduction Assessment

Yes, at source level: current main still tests a `js` fence as `<pre><code>` and the shared Markdown IR drops `token.info` before Telegram rendering. I did not run Vitest or live Telegram because this review is read-only.

## Solution Assessment

Yes. Carrying sanitized fence-language metadata through the shared Markdown render/chunking path is the narrowest maintainable fix; a Telegram-local fence-aware renderer is a fallback only if maintainers reject widening the IR shape.
