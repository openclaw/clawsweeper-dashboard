## Summary

Keep open. Current main and v2026.5.3 still render Telegram fenced code blocks as static `<pre><code>` blocks without preserving the fence language, and the related WebChat copy-button work is a separate renderer path.

## What This Changes

Request Telegram HTML rendering to preserve Markdown fence languages as `<pre><code class="language-X">...` so Telegram can show native copy support.

## Best Possible Solution

Carry sanitized fenced-code language metadata through Telegram rendering and chunking, then cover fenced, unfenced, escaping, file-reference wrapping, and chunk-limit behavior with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: `markdownToTelegramHtml("```js\nconst x = 1;\n```")` follows the static `code_block` marker and the current formatter test expects no language class.

## Solution Assessment

Yes. Preserving sanitized fence language metadata through shared IR/render-aware chunking, or an equivalently narrow Telegram renderer, is the maintainable fix; the implementation should not interpolate raw `token.info`.
