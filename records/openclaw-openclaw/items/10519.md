## Summary

Keep open. Current main and v2026.5.12 still render Telegram fenced code as static `<pre><code>` and drop markdown-it fence language metadata before Telegram HTML output; the related WebChat copy-button item is a separate UI renderer path.

## What This Changes

The issue requests preserving fenced Markdown language identifiers in Telegram HTML output so Telegram can show its native code-block copy affordance.

## Best Possible Solution

Add sanitized optional fenced-code language metadata to the Markdown IR/render-aware chunking path, or a narrow Telegram-local equivalent, then assert `class="language-X"` in Telegram formatter and chunking tests.

## Reproduction Assessment

Yes, at source level: current main has a Telegram formatter test for a `js` fence that still expects no class, and the shared Markdown IR drops fence metadata before rendering. I did not run Vitest or live Telegram in this read-only review.

## Solution Assessment

Yes. Preserving a sanitized first fence-language token through the shared Markdown rendering/chunking contract is the narrowest maintainable solution; a Telegram-local renderer is a fallback if maintainers do not want to widen the IR metadata contract.
