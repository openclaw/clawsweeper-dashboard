## Summary

Keep this open: current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` without preserving the fence language, and the related WebChat copy-button work is a separate UI renderer path.

## What This Changes

The issue asks OpenClaw to preserve fenced-code language identifiers when rendering Telegram HTML so code blocks can use Telegram's native copy affordance.

## Best Possible Solution

Carry a sanitized optional fence language through the Telegram render/chunk path and emit `class="language-X"` only for fenced code blocks with a valid language identifier.

## Reproduction Assessment

Yes. Source inspection on current main shows `markdownToTelegramHtml("```js\n...\n```")` is still expected to render as `<pre><code>...` without a language class, and outbound Telegram delivery uses that renderer.

## Solution Assessment

Yes, directionally. Preserving sanitized fence metadata in the Telegram rendering path is the narrow maintainable direction, but the shared-IR versus Telegram-specific implementation choice needs maintainer judgment.
