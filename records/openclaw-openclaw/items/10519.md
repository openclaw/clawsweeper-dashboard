## Summary

Keep open. Current main and the latest release still drop fenced-code language metadata before Telegram HTML rendering, and the related WebChat copy-button work is a separate UI renderer path rather than a Telegram Bot API output fix.

## What This Changes

The issue requests preserving Markdown fence language metadata in Telegram HTML code blocks as `class="language-X"` for native Telegram copy-button support.

## Best Possible Solution

Add sanitized optional fenced-code language metadata to the Markdown IR/render-aware chunking path, or a narrow Telegram-local equivalent, then update Telegram formatter/chunking tests to emit `class="language-X"` only for fenced blocks.

## Reproduction Assessment

Yes, at source level: current main has a Telegram formatter test for a `js` fence that expects no class, and the shared IR drops `token.info` before rendering. I did not run tests or live Telegram because this review was read-only.

## Solution Assessment

Yes. Preserving sanitized fence language metadata through the shared Markdown IR/render-aware chunking contract is the most maintainable path; a Telegram-only fence-aware renderer is a fallback if maintainers do not want to extend IR metadata.
