## Summary

Keep open. Current main and the latest shipped release still drop Markdown fence language metadata before Telegram HTML rendering, so Telegram fenced code blocks cannot emit the requested native language class.

## What This Changes

Preserve Markdown fenced-code language identifiers in Telegram HTML output so fenced blocks can render as `<pre><code class="language-X">...` when a language is present.

## Best Possible Solution

Add sanitized optional fence-language metadata to the Markdown IR/render-aware chunking path, or a narrow Telegram-local equivalent, and render `class="language-X"` only for fenced code blocks that provide a language.

## Reproduction Assessment

Yes, at source level. Current main has a Telegram formatter test for a `js` fence that expects no class, and the shared Markdown IR drops `token.info` before the Telegram renderer sees it.

## Solution Assessment

Yes, with a narrower implementation than the sketch in the issue. Preserving sanitized fence language metadata through the shared Markdown rendering/chunking contract is the most maintainable path; a Telegram-only fence-aware renderer is the fallback if maintainers do not want to widen the IR metadata.
