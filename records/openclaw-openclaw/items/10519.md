## Summary

Keep open: current main and the latest release still drop fenced-code language metadata before Telegram HTML rendering, and no duplicate issue or PR appears to own the remaining Telegram-specific work.

## What This Changes

Request to preserve the markdown fence language in Telegram HTML code blocks so Bot API output can include `class="language-X"`.

## Best Possible Solution

Teach Telegram markdown rendering to carry a sanitized fence language through the render/chunk path and emit `class="language-X"` only for fenced code blocks with a valid language identifier.

## Reproduction Assessment

Yes, source-based: `markdownToTelegramHtml("```js\n...\n```")` is still asserted to render as `<pre><code>...` without a language class, and outbound Telegram markdown delivery uses that renderer. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Yes directionally: preserving sanitized fence metadata is the narrow maintainable behavior. The unresolved design choice is whether to extend the shared Markdown IR or add a Telegram-local fence-aware rendering path.
