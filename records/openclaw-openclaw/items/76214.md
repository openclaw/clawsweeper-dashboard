## Summary

Keep open. Current main still lacks a TUI mouse-hover cursor affordance: the terminal TUI is wired around keyboard/input-sequence components, and related mouse groundwork in #51128 is still open and does not resolve hover cursor styling.

## What This Changes

Request cursor/pointer feedback when hovering interactive elements in the terminal TUI.

## Best Possible Solution

Decide whether terminal TUI mouse hover should be supported at all; if yes, build it as a capability-aware mouse/hover abstraction after or alongside #51128, otherwise clarify the TUI as keyboard-first and route browser cursor issues to the Control UI CSS surface.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction path: current main’s TUI selectors and root input wiring handle keyboard/input sequences but expose no hover or pointer cursor path.

## Solution Assessment

No. The suggested CSS-style cursor:pointer fix is not the best terminal TUI solution; the maintainable path is either a proper pi-tui/OpenClaw mouse capability seam or an explicit docs/product clarification.
