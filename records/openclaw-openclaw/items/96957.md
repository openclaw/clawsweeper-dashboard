## Summary

Keep open: the PR has useful intent, but it is not merge-ready because it only prefers `chatAvatarUrl` when already populated and still falls back to the protected `/avatar/<agentId>` image route when it is null. The linked issue remains open, and the older overlapping PR is not a safe superseding close target because it is still open and proof/refresh-limited.

## What This Changes

The PR reorders the Control UI Settings assistant avatar fallback so `chatAvatarUrl` is preferred before the assistant agent avatar route, and adds tests for that priority plus route fallback behavior.

## Best Possible Solution

Keep the `chatAvatarUrl` priority, but make the missing-`chatAvatarUrl` local-avatar path use an authenticated blob/renderable URL or safe fallback instead of a direct protected avatar route.

## Reproduction Assessment

Yes at source level: current main can pass `/avatar/<agentId>` directly to Quick Settings for local assistant avatars while gateway tests and docs show that route requires bearer auth. I did not run the full Tailscale browser flow.

## Solution Assessment

No. Prioritizing `chatAvatarUrl` is useful but incomplete because the PR still falls back to the same protected route when that URL is unavailable; the safer existing direction is bearer-authenticated fetch plus blob URL or fallback.
