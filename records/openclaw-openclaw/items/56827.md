## Summary

Keep open. Current main and the latest release still contain the default doctor session-store read path and do not classify stale `sessions.json.*.tmp` sidecars for cleanup; the open linked fix PR owns the remaining work and should close this issue only after merge.

## What This Changes

The issue reports that `openclaw doctor` can fail on long-lived installs with a large `sessions.json` plus many stale `sessions.json.*.tmp` files.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85967 if its session-state cleanup policy is acceptable, then let that merge close this issue; keep the broader SQLite-backed store direction in https://github.com/openclaw/openclaw/issues/55334.

## Reproduction Assessment

Yes, at source level and through the linked PR's supplied harness. Current main still takes the default clone/cache-heavy doctor load path, and the linked PR body shows a Linux Node 22 heap-capped reproduction where that path OOMs while the read-only load succeeds.

## Solution Assessment

Yes, the linked PR is the narrowest current path: use existing `loadSessionStore` options for read-only doctor scans and add age-gated stale temp cleanup. A SQLite-backed session store is a broader follow-up, not required to fix this issue.
