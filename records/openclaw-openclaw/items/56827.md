## Summary

Keep open. Current main still has the source-backed failure shape: `doctor` materializes the monolithic session store, while cleanup and disk-budget maintenance do not detect or remove stale session-store temp sidecars; the adjacent recovery PR at https://github.com/openclaw/openclaw/pull/79910 only covers corrupt-store recovery for https://github.com/openclaw/openclaw/issues/77783.

## What This Changes

The issue asks for `openclaw doctor` and session maintenance to tolerate large session stores and clean up accumulated stale `sessions.json.*.tmp` sidecars.

## Best Possible Solution

Add conservative doctor/session-cleanup handling for stale session-store temp sidecars, including legacy `sessions.json.<uuid>.tmp` and current fs-safe temp names, before disk-budget eviction of real session history; leave SQLite migration to a separate architecture discussion.
