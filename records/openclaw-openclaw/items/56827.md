## Summary

Keep open. Current main and v2026.5.18 still have the source-backed failure shape: `openclaw doctor` materializes the monolithic session store, while session cleanup and disk-budget paths still do not detect or remove stale `sessions.json.*.tmp` sidecars; https://github.com/openclaw/openclaw/pull/47635 is related cleanup work but remains open, and https://github.com/openclaw/openclaw/pull/79910 covers corrupt-store recovery rather than this large-store doctor path.

## What This Changes

The issue reports `openclaw doctor` failing on a long-lived installation with a large `sessions.json` and many stale `sessions.json.*.tmp` files consuming substantial disk space.

## Best Possible Solution

Add conservative stale session-store temp sidecar handling to explicit doctor/session cleanup paths before disk-budget eviction of real session history, while leaving broader session-store schema work to https://github.com/openclaw/openclaw/issues/55334 and corrupt-store recovery to https://github.com/openclaw/openclaw/pull/79910.

## Reproduction Assessment

No high-confidence runtime reproduction of the exact V8 failure was established. The source path is reproducible by inspection: doctor still loads and enumerates the full `sessions.json`, and current cleanup/disk-budget code still excludes stale store temp sidecars.

## Solution Assessment

Yes for a focused first repair. The maintainable path is explicit doctor/session-cleanup handling for known stale store temp sidecars, not a SQLite migration or a duplicate corrupt-store recovery implementation.
