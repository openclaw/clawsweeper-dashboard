## Summary

Keep open. Current main has improved load-time session pruning, but the reported failure mode is still not resolved: `doctor` still reaches session integrity by loading and parsing the whole `sessions.json`, and the session cleanup paths still do not recognize stale `sessions.json.<uuid>.tmp` atomic-write leftovers. The issue is recent, concrete, and the stale label was removed after the previous ClawSweeper review, so this should stay actionable.

## What This Changes

The issue requests that `openclaw doctor` complete on oversized long-lived session stores and that stale `sessions.json.<uuid>.tmp` files be detected or cleaned instead of accumulating.

## Best Possible Solution

Harden the existing doctor/session maintenance path first: make doctor non-fatal and cheap enough around oversized session stores, report and safely clean stale same-directory session-store temp files, and include those artifacts in `sessions cleanup`/disk-budget maintenance. Treat a SQLite-backed session metadata store as a separate design follow-up rather than the first repair.

## Reproduction Assessment

Unclear. The report gives concrete store sizes and current main still follows the heavy load path, but this read-only review did not create a large fixture or reproduce the exact V8 stack.

## Solution Assessment

Yes for a narrow repair PR, not for an immediate storage redesign. The best first step is targeted doctor/session-cleanup hardening for oversized stores and stale atomic temp files; SQLite would be broader product work.
