## Summary

Keep open. Current main still has the source-backed failure shape: doctor reads and materializes the monolithic session store, while session cleanup/disk-budget maintenance does not detect or remove stale `sessions.json.*.tmp` files.

## What This Changes

Make `openclaw doctor` and session maintenance tolerate large session stores and stale `sessions.json.*.tmp` files without crashing or accumulating disk bloat.

## Best Possible Solution

Add a focused doctor/session-cleanup repair that reports and safely removes stale same-directory session-store temp sidecars, and make disk-budget cleanup handle those files before evicting real session entries.

## Reproduction Assessment

No. The report gives concrete store sizes and current source confirms the full-store read plus missing temp cleanup path, but I did not build a 1.2 GB fixture or reproduce the exact V8 deserialization crash.

## Solution Assessment

Yes for a focused first repair. A narrow doctor/session-maintenance fix is the maintainable next step; a SQLite metadata-store migration would be a larger product and architecture decision.
