## Summary

Keep open. Current main still has the reported unresolved surfaces: doctor reaches session state integrity by loading and parsing the full monolithic store, and the inspected doctor/session cleanup paths do not detect or remove stale `sessions.json.<uuid>.tmp` atomic-write leftovers. The stale label should not drive closure because the report is recent, concrete, and still maps to live code.

## What This Changes

Make `openclaw doctor` complete reliably on long-lived installations with oversized session stores and detect or safely clean stale `sessions.json.*.tmp` files.

## Best Possible Solution

Harden the existing diagnostics and maintenance path first: make doctor non-fatal and cheap enough around oversized stores, report stale same-directory session-store temp files, and add safe cleanup through doctor/session maintenance. Treat a SQLite session metadata store as a separate design follow-up rather than the first required repair.

## Repair Work Prompt

Fix the long-lived session-store doctor failure reported in https://github.com/openclaw/openclaw/issues/56827. Current main still parses the entire `sessions.json` during doctor state integrity and does not detect or clean stale `sessions.json.<uuid>.tmp` files left by atomic writes. Implement a narrow repair: add a safe helper for identifying same-directory stale session-store temp artifacts matching the writer pattern, report them in `openclaw doctor`, and include conservative cleanup through doctor/session maintenance or disk-budget cleanup without touching active temp files.
