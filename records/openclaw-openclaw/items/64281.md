## What This Changes

The PR adds `sessions_delegate` and `sessions_delegate_batch` core agent tools, registers them in tool catalogs/display config, and adds delegate-specific subagent lifecycle cleanup/output capture code.

## Best Possible Solution

Rebase onto current main, settle the synchronous delegation contract, integrate output capture with the current subagent lifecycle before cleanup can race, extend transcript redaction, and add docs/tests plus redacted real-run proof.

## Solution Assessment

No. A synchronous delegate primitive may be the right direction, but this patch is not the safest implementation until it rebases and fixes the security, lifecycle, dependency, and proof gaps.

- **[P2] Import TypeBox from the declared package:** `src/agents/tools/sessions-delegate-tool.ts:1`
  - body: This file imports `Type` from `@sinclair/typebox`, but current main declares and consistently imports `typebox` for schemas. Unless the PR adds an approved dependency change, clean installs can fail module resolution for the new tool.
  - confidence: 0.94
- **[P1] Redact delegate attachment content before persistence:** `src/agents/tools/sessions-delegate-tool.ts:51-56`
  - body: `sessions_delegate` accepts inline `attachments[].content`, but transcript sanitization only redacts attachment payloads for `sessions_spawn`. Persisted delegate tool inputs can therefore keep raw attachment text or secrets in session history.
  - confidence: 0.95
- **[P1] Register the output gate before child completion can race:** `src/agents/tools/sessions-delegate-tool.ts:191`
  - body: The cleanup gate is created only after `spawnSubagentDirect` returns, but that call registers the run and can continue through hooks/lifecycle events before returning. A fast child can complete cleanup before a gate exists, so the child session can be deleted before delegate output is read.
  - confidence: 0.9
- **[P2] Release the output gate when output reading throws:** `src/agents/tools/sessions-delegate-tool.ts:220-221`
  - body: `signalOutputCaptured(runId)` runs only after `readChildOutput` succeeds. If the history fallback throws, cleanup waits for the full gate timeout even though the child already finished; wrap the output read in `try/finally`.
  - confidence: 0.84
