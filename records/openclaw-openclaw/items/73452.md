## Summary

Keep this PR open, but it is not merge-ready. Current main still has the Mattermost draft-preview hardcode the PR targets, yet the proposed one-line change can turn block replies back on while the draft-preview path still streams partial updates into a preview post, creating a likely double-delivery path rather than honoring a single configured streaming mode.

## What This Changes

The PR changes Mattermost's draft-preview dispatch path to derive `replyOptions.disableBlockStreaming` from `account.blockStreaming` instead of always disabling block streaming there.

## Best Possible Solution

Implement Mattermost streaming control as an explicit delivery-mode choice: honor the documented `channels.mattermost.streaming` preview setting, keep `blockStreaming` as block-reply enablement, and ensure draft preview and block replies cannot both own the same turn. Add a focused Mattermost monitor assertion for the chosen dispatch options and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence code reproduction is available by mocking or inspecting the draft-preview `dispatchReplyFromConfig` call: current main always passes `disableBlockStreaming: true`, while the PR would pass `false` for `account.blockStreaming: true` with `onPartialReply` still registered.

## Solution Assessment

No. The proposed one-line mapping is correct for non-preview reply paths, but it is not the best fix for the draft-preview path because it can enable block replies and live draft edits simultaneously; the safer fix is to gate Mattermost preview streaming from the documented streaming config and suppress one delivery path when the other is active.
