## Summary

Keep open. Current main still exposes separate scoped model-selection mechanisms rather than one owner-safe bulk retarget command, and the requested behavior crosses defaults, stored sessions, cron jobs, subagents, channel-bound sessions, fallback policy, and auth-profile pins. This is valid product work, but it needs maintainer scope and safety decisions before implementation.

## What This Changes

Request for one command to move the main agent, subagents, cron jobs, other tasks, and the current Telegram context to another configured provider/model during provider rate limits.

## Best Possible Solution

Design an owner-only bulk model migration flow under the existing model/session command surface that validates the target provider/model, previews affected records, supports explicit scopes, and preserves intentional user/job/subagent overrides unless the operator opts into replacing them.

## Reproduction Assessment

Not applicable as a feature request. The verification path is source and docs inspection on current main, which shows scoped model controls but no single bulk model retarget command or API.

## Solution Assessment

Unclear as proposed. A dry-run-capable owner command with explicit scopes is the maintainable direction; an unqualified `/moveall` that silently rewrites every model reference would be too risky.
