## Summary

Keep open. Current main still intentionally sends ordinary MP3-style Feishu attachments as files unless `audioAsVoice`/`asVoice` is requested, so this PR is not obsolete; the branch changes a documented default and still needs maintainer policy review, docs/lockfile alignment, and real behavior proof before merge.

## What This Changes

The PR broadens Feishu outbound media routing so common audio extensions and audio MIME types upload as `file_type: "opus"`/`msg_type: "audio"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Maintainers should decide the Feishu audio contract; if direct playable bubbles for common formats are desired, land a rebased narrow change that updates routing, duration strategy, docs, package metadata, lockfile, and dependency contract coverage together.

Yes, from source inspection rather than a live Feishu run. Current main routes `song.mp3` with `audio/mpeg` as `stream`/`file`, and the colocated test asserts that behavior unless `audioAsVoice` is set.

Unclear as-is. The direction is plausible because the PR cites live Feishu API validation and an external plugin, but it is not the best merge shape until maintainers choose the default-audio policy and the branch adds docs, lockfile, dependency-contract, and proof alignment.
