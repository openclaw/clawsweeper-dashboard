## Summary

Keep open for maintainer review. Current main still routes ordinary MP3/WAV/M4A-style Feishu audio as file attachments unless explicit voice delivery requests transcoding, so this PR is not obsolete; the author also addressed the prior duration-helper review feedback and supplied useful live Feishu API plus community-plugin context. The branch still needs a Feishu default-audio policy decision and merge-prep updates before it is landable.

## What This Changes

The PR broadens Feishu outbound audio routing so common audio extensions and audio MIME types upload as `file_type: "opus"` and send as `msg_type: "audio"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu audio contract explicitly. If direct playable bubbles for common formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, and lockfile together; otherwise keep the shipped explicit voice-transcode path and supersede this branch with that policy stated.

Yes. Current main has both source and a focused Feishu media test showing `song.mp3` with `audio/mpeg` is sent as `file_type: "stream"` and `msg_type: "file"` unless explicit voice intent is set.
