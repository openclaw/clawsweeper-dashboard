## Summary

Keep open. Current main still preserves the explicit Feishu voice policy, where ordinary MP3-style attachments remain files unless `audioAsVoice`/`asVoice` requests voice delivery, so this PR is a real behavior change that needs maintainer product review plus package/docs/lockfile cleanup before merge.

## What This Changes

The PR broadens Feishu outbound audio routing so common audio extensions and audio MIME types send as native audio messages, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu audio contract explicitly; if direct playable bubbles for common formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, lockfile, and changelog together.

## Reproduction Assessment

Yes. Source and test evidence on current main show `song.mp3` with `audio/mpeg` and no `audioAsVoice` uploads as `stream` and sends as `msg_type: "file"`.

## Solution Assessment

Unclear as-is. The PR direction is plausible and backed by useful Feishu/API/community-plugin context, but it is not the best merge shape until maintainers choose the default-audio policy and require docs, changelog, lockfile, package metadata, and rebase alignment.
