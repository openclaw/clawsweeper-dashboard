## Summary

Keep open. Current main still keeps ordinary MP3/WAV/M4A-style Feishu media as file attachments unless explicit voice intent requests transcoding, while this PR proposes changing that default to direct playable audio bubbles. The PR has useful supporting context from live Feishu API testing and the community @m1heng-clawd/feishu implementation, but it is not merge-ready until maintainers choose the Feishu default-audio policy and the branch aligns docs, lockfile, and current package metadata.

## What This Changes

The PR broadens Feishu outbound audio routing to treat common audio filenames and audio MIME types as `file_type: "opus"` / `msg_type: "audio"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu audio contract explicitly. If direct playable bubbles for common audio formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, lockfile, and changelog together; if not, keep the shipped explicit `audioAsVoice`/`asVoice` transcode path and close or supersede this PR with that policy stated.

## Reproduction Assessment

Yes. Current main has a focused source-and-test reproduction: sending `song.mp3` with `audio/mpeg` through `sendMediaFeishu()` without `audioAsVoice` uploads as `file_type: "stream"` and sends `msg_type: "file"`.

## Solution Assessment

Unclear as-is. The PR direction is plausible and backed by useful Feishu/API/community-plugin context, but it is not the best merge shape until maintainers choose the default-audio policy and require docs, lockfile, package metadata, and rebase alignment.
