## Label Justifications

- `P2`: The remaining issue is a real Signal voice-note delivery gap with a bounded channel/media fix, but it is limited to configured Signal audio/mention-gating workflows.
- `impact:message-loss`: Signal group voice notes that would mention the bot can be skipped before the transcript is available, so the intended user message never reaches the agent.

## Summary

Keep open: current main and v2026.6.1 still have Signal mention gating return before attachment fetch or audio preflight, while Telegram and Discord already call the shared audio preflight helper for this path.

## What This Changes

The issue asks for Signal voice-note audio preflight so inbound audio can be transcribed before the agent and before mention gating, with optional discussion of a configurable external converter hook.

## Best Possible Solution

Add Signal preflight parity with the existing transcribeFirstAudio path before mention gating, carry the transcript into dispatch with MediaTranscribedIndexes, and leave arbitrary shell-hook configuration out of the first fix.

## Reproduction Assessment

Yes, at source level: Signal checks mentionDecision.shouldSkip and returns before fetching attachments, while an existing test confirms an audio-only requireMention group message records only <media:audio> and captures no dispatch context. I did not perform a live Signal transcription run in this read-only review.

## Solution Assessment

Yes for the narrowed fix: reusing the existing shared preflight helper in the Signal plugin is the maintainable path because Telegram and Discord already use that shape. The requested arbitrary shell-command hook is not the best first fix because it creates a new command-execution/config surface.
