## Summary

Keep open: current main and the latest release still lack Signal audio preflight parity, while the existing Signal work only fixes metadata classification. The external command-hook flavor is useful context but needs maintainer product and security scoping before implementation.

## What This Changes

The issue requests a Signal/channel audio-preflight hook so Signal voice notes can be transcribed before reaching the agent, with either built-in STT parity or a configurable external converter command.

## Best Possible Solution

Add Signal parity by reusing the existing shared transcribeFirstAudio path before Signal mention gating, with focused Signal tests and docs, and split any arbitrary external command hook into a separate security-reviewed product decision.

## Reproduction Assessment

Yes, at source level: Signal mention-gated audio-only group messages can return before attachments are fetched or transcribed, and the current test records only a <media:audio> placeholder. I did not run a live Signal plus STT scenario in this read-only review.

## Solution Assessment

Yes for the narrowed parity fix: reuse the shared audio-preflight helper in the Signal-owned path before mention gating. No for bundling the arbitrary shell-command hook into the first repair, because that adds a new command-execution/config surface requiring maintainer security and product design.
