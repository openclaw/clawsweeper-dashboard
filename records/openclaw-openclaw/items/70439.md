## Summary

Keep open: current main and v2026.6.10 still lack Signal audio preflight parity, while the existing Signal path records `<media:audio>` placeholders before attachment fetch/transcription when mention-gated messages are skipped. The shared STT path exists, but the requested Signal hook and especially the external command flavor need maintainer product/security scoping.

## What This Changes

Request to add Signal audio preflight transcription, either through built-in shared STT/whisper parity or a configurable external preflight command.

## Best Possible Solution

Implement Signal-owned preflight with the shared `transcribeFirstAudio` path before mention gating, reusing `tools.media.audio`, and treat any arbitrary external command hook as a separate security-approved product design.

## Reproduction Assessment

Yes, at source level: Signal mention-gated audio-only group messages return before attachments are fetched or transcribed, and the current test records only a `<media:audio>` placeholder. I did not run a live Signal plus STT scenario in this read-only review.

## Solution Assessment

Partly: the narrowed Signal parity fix should reuse the shared audio-preflight helper before mention gating. The external `preflight_cmd` flavor is not yet the best first fix because it adds command-execution and config surface that needs maintainer security/product approval.
