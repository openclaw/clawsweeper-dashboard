## Summary

Keep open: current main and the latest release still lack Signal audio preflight parity, while Telegram and Discord use the shared transcription helper before mention-gated processing. The remaining core issue is narrow if scoped to Signal reusing the existing helper, but the requested arbitrary command hook needs human security/product scoping first.

## What This Changes

The issue requests a Signal audio-preflight path so voice notes can be transcribed before the agent sees only an audio placeholder.

## Best Possible Solution

Add Signal parity by fetching eligible audio and invoking the existing shared `transcribeFirstAudio` path before mention gating, then carry the transcript and transcribed media index into dispatch; defer arbitrary shell-command hooks to a separate security-reviewed design.

## Reproduction Assessment

Yes, at source level: Signal checks mention skip before fetching attachments or running audio preflight, and an existing test confirms audio-only skipped group messages record only `<media:audio>` with no dispatch context. I did not run a live Signal transcription scenario in this read-only review.

## Solution Assessment

Yes for the narrowed Signal parity fix: reusing the existing shared helper is the maintainable path because Telegram and Discord already use that shape. No for the arbitrary shell-command hook as a first fix, because it adds a command-execution/config surface that needs separate design review.
