Keep open: current main and v2026.6.8 still do not implement Signal audio preflight parity, while sibling channels use the shared transcription helper before mention gating. ## What This Changes

The issue asks for Signal voice-note audio preflight/transcription parity, either through built-in shared STT handling or a configurable external preflight command hook.

## Best Possible Solution

Add Signal parity by reusing the existing shared `transcribeFirstAudio` path before Signal mention gating, and defer arbitrary shell-command hooks to a separate security-reviewed product design.

## Reproduction Assessment

Yes, at source level: Signal can skip mention-gated audio-only group messages before attachment fetch or transcription, and the existing test captures the `<media:audio>` placeholder path. I did not run a live Signal plus STT scenario in this read-only review.

## Solution Assessment

Yes for the narrowed parity fix: reusing the shared helper before Signal mention gating is the maintainable path. No for bundling the arbitrary shell-command hook into the first fix, because that adds command execution/config surface needing separate security review.
