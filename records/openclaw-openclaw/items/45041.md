## Summary

Keep open: current main has structural assistant-text sanitizers and outbound plugin hooks, but it still lacks the requested global or per-agent `outputFilters` config contract. This remains a privacy-sensitive product/API decision, especially with the narrower related sanitizer PR still open.

## What This Changes

The issue requests an opt-in global/per-agent output filter that strips configured reasoning or meta-planning patterns from assistant messages before channel delivery.

## Best Possible Solution

Maintainers should choose the privacy/product boundary first: either bless a ClawHub/plugin-based `message_sending` filter and add any missing per-agent context, or add a guarded opt-in core config with schema, docs, and false-positive regression tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection on current main confirms the requested `outputFilters` surface is absent.

## Solution Assessment

Unclear. A raw regex filter is flexible but privacy-sensitive and over-stripping-prone; a maintainer-approved sanitizer/config contract or plugin/API path with false-positive coverage is safer.
