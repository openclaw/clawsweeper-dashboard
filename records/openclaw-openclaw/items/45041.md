## Summary

Current main has a canonical delivery sanitizer for structured reasoning tags, tool-call contamination, and internal trace lines, but it does not implement the requested arbitrary per-agent output filters or cover the issue’s ordinary-language meta-planning examples. The directly related implementation PR closed unmerged, and the open reasoning-field issue addresses provider-structured outputs rather than this broader destructive filtering policy, so meaningful product work remains and this issue should stay open for explicit maintainer security and UX judgment.

## What This Changes

Add an optional per-agent regex filter that strips configured reasoning or meta-planning patterns from assistant messages before channel delivery.

## Best Possible Solution

Keep provider reasoning structurally separated and extend the single canonical delivery sanitizer only for high-confidence leak shapes with strip-and-preserve regressions; if maintainers approve advanced user filtering, design a constrained output-policy surface with safe pattern limits, per-agent scope, diagnostics, and explicit empty-message behavior instead of accepting arbitrary regex flags directly.

## Reproduction Assessment

Partly yes: the linked incidents and current source establish that structured and plain-text internal material can reach user-facing text, but the exact five-agent Discord scenario has not been reproduced against current main. The requested general regex behavior is demonstrably absent from current source and config.

## Solution Assessment

No, the raw regex-array proposal is not yet the best solution because it creates destructive message-delivery and performance risks alongside the existing canonical sanitizer. The narrower maintainable path is structural provider parsing plus conservative shared sanitizer rules, unless maintainers explicitly approve and constrain a new output-policy feature.
