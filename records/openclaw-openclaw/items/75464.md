## Summary

Keep this PR open but blocked: current `main` still has the RTT preflight bugs, while the PR head still exposes Convex broker secrets to package install lifecycle code and can leak a lease when acquired payload validation throws.

## What This Changes

The PR defaults `pnpm rtt` to the current checkout and adds Convex-backed Telegram QA credential handling for package RTT runs, with a new credential helper, Docker harness changes, tests, and a changelog entry.

## Best Possible Solution

Keep the RTT wrapper fix, but split package installation from credential-bearing runtime execution and ensure acquired Convex leases are released on every validation or child-process failure path before landing.

## Reproduction Assessment

Yes. Current `main` source still rejects Convex-only RTT credentials before Docker and defaults `pnpm rtt` to an old non-repo harness root; the PR blockers are also source-reproducible from the Docker env flow and acquire/release control flow.

## Solution Assessment

No. The PR addresses the original preflight failures, but its current implementation is not the safest maintainable solution because it crosses the package-install credential boundary and does not release leases on all acquire validation failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Convex secrets out of the install phase:** `scripts/e2e/npm-telegram-rtt-docker.sh:153-156`
  - body: The PR appends `OPENCLAW_QA_CONVEX_SECRET_CI` and `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER` to the same `docker_env` used for the container that later runs `npm install -g "$install_source"`. That exposes broker secrets to package install lifecycle code for the package under test; split install from the credential-bearing runtime phase like the existing live lane.
  - confidence: 0.9
- **[P2] Release leases when RTT payload validation fails:** `scripts/e2e/npm-telegram-rtt-credentials.ts:347-349`
  - body: After a broker acquire succeeds, `parseTelegramPayload(response.payload)` runs before the lease is returned to `runCommand` and before its release `finally` is active. A malformed Telegram payload can therefore leave the pooled credential leased until TTL.
  - confidence: 0.87
