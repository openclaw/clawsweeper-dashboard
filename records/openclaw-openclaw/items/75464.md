## Summary

Keep open. Current main still has the RTT wrapper regressions this PR targets, but the proposed Convex preflight is incomplete because it accepts any Convex role secret instead of the selected role's required secret, allowing a CI or role=ci run to pass preflight and fail later in the Docker QA lease path.

## What This Changes

The PR makes the RTT wrapper accept Convex credential env, defaults the harness root to the current checkout, and adds regression coverage for those behaviors.

## Best Possible Solution

Align the RTT wrapper preflight with the existing npm Telegram Docker and QA lease contract: normalize source and role the same way, require the secret for the selected role, keep the current-checkout harness-root default, and cover the role-specific Convex cases in the focused RTT tests.

## Reproduction Assessment

Yes. Current main has a static reproduction: Convex-only env reaches `assertRequiredEnv()` before Docker and fails on raw Telegram vars, and `parseArgs()` defaults the documented `pnpm rtt` invocation away from the repo checkout. The PR's remaining defect reproduces with Convex role `ci` plus only the maintainer secret: the new preflight passes while the downstream lease requires the CI secret.

## Solution Assessment

No as written. The direction is the right narrow wrapper fix, but the Convex preflight should resolve the selected role and require that role's secret before Docker starts; otherwise it preserves a delayed failure path the wrapper is meant to catch.
