## Summary

Keep open. Current main still has the original RTT wrapper regressions, and this PR is directionally useful, but the current PR head has a blocking secret-forwarding issue plus a lease-cleanup bug that should be fixed before merge.

## What This Changes

The PR defaults `pnpm rtt` to the current checkout and adds Convex-backed credential handling for package RTT runs, including a new Docker credential helper, tests, and a changelog entry.

## Best Possible Solution

Keep the RTT wrapper fix, but split package installation from credential-bearing runtime execution like the existing live lane and ensure acquired Convex leases are released on every validation or child-process failure path.

## Reproduction Assessment

Yes. The original bugs reproduce from current-main preflight code, and the PR blockers reproduce by static shell/TypeScript control flow without live credentials: Convex secrets are in the install container env, and payload validation can throw before the lease reaches the release `finally`.

## Solution Assessment

No. The PR solves the two reported preflight failures, but the safest maintainable solution should reuse the live lane’s install-then-runtime boundary and tighten lease cleanup semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Convex secrets out of the install phase:** `scripts/e2e/npm-telegram-rtt-docker.sh:153-156`
  - body: The PR appends `OPENCLAW_QA_CONVEX_SECRET_CI` and `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER` to `docker_env`, and that same env array is passed to the container before the heredoc runs `npm install -g "$install_source"`. This exposes broker secrets to package install lifecycle code for the package under test; split the install and credential-bearing runtime phases like the existing live lane.
  - confidence: 0.88
- **[P2] Release leases when RTT payload validation fails:** `scripts/e2e/npm-telegram-rtt-credentials.ts:347-349`
  - body: After a broker acquire succeeds, `parseTelegramPayload(response.payload)` runs while building the return object. If the broker returns malformed Telegram payload data, the function throws before returning a lease, so `runCommand` never reaches its `finally` release path and the pooled credential remains leased until TTL.
  - confidence: 0.86
