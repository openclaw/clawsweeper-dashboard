## Summary

Keep open. The latest PR head addresses the earlier wrapper-only Convex blocker, but static review found a new secret-forwarding regression and a lease cleanup gap in the RTT credential helper; current main still contains the original RTT wrapper issues, so the branch remains useful but is not merge-ready.

## What This Changes

The PR defaults the RTT CLI harness root to the current checkout, adds RTT Convex credential acquisition and release support, forwards Convex env through the RTT Docker wrapper, and adds changelog plus focused tests.

## Best Possible Solution

Keep the PR open and revise the RTT Docker path so package installation happens before any credential-bearing runtime env is forwarded, then make the new RTT Convex helper match the shared QA lease cleanup semantics while preserving raw-env operation and the current-checkout harness-root default.

## Reproduction Assessment

Yes. The secret exposure is reproducible by following the static shell path: Convex secrets are added to `docker_env`, the same array is passed to the container, and that container runs `npm install -g "$install_source"`; the lease leak follows from a successful acquire response with an invalid Telegram payload at the parse call.

## Solution Assessment

No. The PR is directionally right, but the best fix should mirror the existing live lane's install-then-secret-runtime split and release acquired leases on payload validation failure instead of adding a separate helper with weaker cleanup behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Convex secrets out of the install phase:** `scripts/e2e/npm-telegram-rtt-docker.sh:153-156`
  - body: The PR appends `OPENCLAW_QA_CONVEX_SECRET_CI` and `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER` to `docker_env`, and that same env array is passed to the container before the heredoc runs `npm install -g "$install_source"`. That exposes the Convex broker secrets to package install lifecycle code for the package under test; split the install and credential-bearing runtime phases like the existing live lane.
  - confidence: 0.88
- **[P2] Release leases when RTT credential payload validation fails:** `scripts/e2e/npm-telegram-rtt-credentials.ts:347-349`
  - body: After a broker acquire succeeds, `parseTelegramPayload(response.payload)` runs while building the return object. If the broker returns a malformed Telegram payload, the function throws before returning a lease, so `runCommand` never reaches its `finally` release path and the pooled credential remains leased until TTL. Release the lease on payload validation failure, matching the shared QA lease runtime.
  - confidence: 0.86
