## Summary

Keep open. Current main still has the RTT wrapper defects, but this PR only fully fixes the default harness-root issue; its Convex credential change is incomplete because the actual RTT Docker harness still rejects Convex-only env before Docker starts.

## What This Changes

The PR makes the RTT wrapper accept Convex credential env, defaults the harness root to the current checkout, and adds regression tests for both behaviors.

## Best Possible Solution

Update the RTT implementation so the wrapper, tests, and invoked Docker harness share one credential contract: Convex-only env should either acquire usable Telegram credentials before npm-telegram-rtt-config.mjs runs, or the RTT command should consistently reject/document raw-env-only operation.

## Reproduction Assessment

Yes. Current main reproduces the wrapper failure with Convex-only env, and the PR still leaves a high-confidence static reproduction because npm-telegram-rtt-docker.sh exits on missing raw Telegram env before Docker starts.

## Solution Assessment

No. The harness-root default change is the right narrow fix, but accepting Convex only in assertRequiredEnv is not sufficient while runHarness still calls a shell harness that requires raw Telegram credentials.
