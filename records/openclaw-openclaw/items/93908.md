## Summary

Current main and v2026.6.11 still have the upgrade-sensitive compatibility-host gap: discovery skips external plugins when the resolved host version is stale or unknown, while post-core convergence still forces the running `VERSION` and the current-process update path only pins downgrade targets. Keep this open because https://github.com/openclaw/openclaw/pull/95384 is the active closing PR, and the newer Slack/Linux systemd reproduction shows the broader unknown-host variant still needs maintainer review.

## What This Changes

The issue reports an external plugin discovery regression where OpenClaw 2026.6.x installs can skip Feishu or Slack because plugin API compatibility is evaluated against a stale or undetermined host version.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/95384 so post-core plugin compatibility checks use the version that the restarted gateway will actually run, with coverage for stale and unknown host-version variants.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and report evidence: current discovery skips when the resolved compatibility host is stale or unknown, and the issue now includes live Feishu and Slack install reports plus an env override that restores plugin loading.

## Solution Assessment

Is this the best way to solve the issue? Likely yes: fixing the post-core compatibility host handoff is narrower than changing Feishu or Slack metadata or relaxing plugin API compatibility checks, but the open PR still needs maintainer review for upgrade safety.
