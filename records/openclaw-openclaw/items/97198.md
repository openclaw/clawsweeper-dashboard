## Summary

Keep this PR open. The Feishu owner boundary is right, but the new stale-revision guard can still stop recovery after the retry task writes its own unknown result, and the contributor has only provided deterministic local/mocked proof rather than real Feishu restart/reload evidence.

## What This Changes

The PR changes Feishu monitor startup to schedule detached bot identity recovery after aborted unknown startup, guard stale recovery with identity revisions, and add regression coverage.

## Best Possible Solution

Keep the fix in the Feishu plugin, but make staleRevision suppress only newer lifecycle writes, add a failed-then-success regression test, and require redacted Feishu reload proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main can return an empty identity for an aborted Feishu probe, apply it as unknown, and skip recovery because the same signal is aborted. PR head also has a source-reproducible retry defect when an unknown retry precedes a later successful retry.

## Solution Assessment

No, not yet. The owner boundary is right, but the staleRevision check is too broad and can suppress this recovery task's own later success; live Feishu proof is also still absent.
