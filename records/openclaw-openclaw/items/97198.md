## Summary

Keep open. The PR targets a real Feishu message-loss bug, but this patch is not merge-ready because it detaches identity recovery from an already-aborted lifecycle, allowing a stale stopped monitor to retry and write global bot identity later; it also lacks real Feishu reload/restart proof.

## What This Changes

The PR starts Feishu bot identity recovery when startup leaves an unknown bot open_id after an aborted signal, passes no abort signal to that retry, and adds a mocked lifecycle regression test.

## Best Possible Solution

Rework the Feishu monitor to distinguish abort-driven probe cancellation from active unknown identity, keep retries tied to an active lifecycle, preserve replacement identity, and then attach redacted Feishu reload proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main can return an empty identity for an aborted Feishu probe, apply it, and skip recovery because the same signal is aborted. I did not run a live Feishu tenant reload.

## Solution Assessment

No. This is a plausible mitigation, but the best fix should keep abort-driven startup cancellation separate from active unknown identity instead of launching an uncancelable stale retry.
