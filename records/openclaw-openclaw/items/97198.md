## Summary

Keep this PR open, but do not treat it as merge-ready: it targets the right Feishu monitor lifecycle bug, yet the new revision guard can still suppress this recovery task's own later successful retry, and the PR still has only local/mocked proof.

## What This Changes

This PR changes Feishu monitor startup to launch detached bot identity recovery after aborted unknown identity, adds a captured identity revision guard, exports the revision reader, and adds regression tests.

## Best Possible Solution

Keep the fix in the Feishu plugin, but make the revision guard distinguish newer lifecycle writes from this retry task's own failed writes, add failed-then-success regression coverage, and require real Feishu reload proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main can return an empty identity for an aborted Feishu probe, apply it as unknown, and skip recovery because the same signal is aborted. PR head also has a source-reproducible retry defect when an unknown retry precedes a later successful retry.

## Solution Assessment

No, not yet. The owner boundary is right, but the staleRevision check is too broad and can suppress this recovery task's own later success; live Feishu proof is also still absent.
