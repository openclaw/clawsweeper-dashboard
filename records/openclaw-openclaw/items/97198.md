## Summary

Keep open. The PR targets a valid Feishu message-loss race, but the patch starts identity recovery without lifecycle cancellation after an aborted startup signal, so a stale stopped monitor can keep retrying and later write identity state; it also lacks real Feishu reload/restart proof.

## What This Changes

The PR changes Feishu account startup so unknown bot identity always schedules background recovery, passes no abort signal when the startup signal is already aborted, and adds a mocked lifecycle regression test.

## Best Possible Solution

Rework the Feishu monitor to distinguish abort-driven probe cancellation from active unknown identity, keep retries tied to an active lifecycle, preserve replacement identity, and attach redacted Feishu reload proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main can return an empty identity for an aborted Feishu probe, apply it, and skip recovery because the same signal is aborted. I did not run a live Feishu tenant reload.

## Solution Assessment

No. This is a plausible mitigation, but the best fix should keep abort-driven startup cancellation separate from active unknown identity instead of launching an uncancelable stale retry.
