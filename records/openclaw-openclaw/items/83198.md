## What This Changes

The PR adds a doctor health-check registry/lint flow and converts several read-only doctor diagnostics into structured detect findings with docs, tests, package export metadata, and config snapshot read options.

## Best Possible Solution

Land the base health-check contract first, then keep this slice focused on read-only diagnostics that reuse non-observing config snapshots and prove a real `doctor --lint` run.

## Reproduction Assessment

Yes, for the PR blocker: source inspection shows the default lint check set reaches `finalConfigValidationCheck.detect()`, which calls `readConfigFileSnapshot()` without `observe: false`. That path can write config health state, contrary to the lint contract.

## Solution Assessment

No, not as written. The migration direction is reasonable, but the final validation check needs to preserve read-only snapshot behavior and the stacked base contract needs maintainer resolution before this follow-up is merge-ready.
