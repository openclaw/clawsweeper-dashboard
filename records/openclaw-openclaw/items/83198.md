## Summary

Keep open: this PR still registers transitional health checks whose `detect()` returns no findings, so `doctor --lint --only` can false-clean auth, session, plugin, gateway, and automation-adjacent doctor areas that still rely on the legacy doctor path.

## What This Changes

The branch adds a doctor health conversion plan, maps legacy doctor contributions to health-check IDs, and converts several read-only doctor notes into structured `doctor --lint` detections while preserving legacy `doctor --fix`.

## Best Possible Solution

Register only health checks that perform real detection, keep transitional target inventory in the conversion plan, and add coverage proving unconverted doctor targets cannot be selected as clean lint checks.

## Reproduction Assessment

Yes by source inspection: PR head registers placeholder `HealthCheck` objects whose `detect()` always returns `[]`, and the current lint runner treats selected registered IDs as complete results. A direct runtime reproduction would be `doctor --lint --only core/doctor/auth-profiles/keychain` or another placeholder ID on the PR head.

## Solution Assessment

No. The real converted detectors are the right direction, but registering inventory-only placeholders as runtime lint checks is not the best migration path; unconverted targets should remain non-runtime until they have detectors.
