## Summary

Keep open. Current main has the heartbeat run-model override path, but the `session_status` tool only receives session/config context and resolves the displayed model from the session entry/defaults, so the active heartbeat override is still not represented for `sessionKey="current"`.

## What This Changes

The issue asks for `session_status({"sessionKey":"current"})` inside a heartbeat turn to report the active `heartbeat.model` override instead of the agent/session default model.

## Best Possible Solution

Thread the selected current-run model into the `session_status` tool context for semantic-current calls, without changing explicit other-session lookups or persisting heartbeat model state.

## Reproduction Assessment

Yes, at source level: configure a heartbeat with `heartbeat.model`, let it call `session_status({"sessionKey":"current"})`, and the current code path has no way for the tool to see the transient heartbeat-selected provider/model. I did not run the live heartbeat in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to pass the selected runtime model through the existing tool-construction/run context for the current session only. Updating session persistence would be the wrong direction because related reports show heartbeat model state must not leak into normal session status.
