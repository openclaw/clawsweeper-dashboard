## Summary

Keep open. Current main still has a source-backed mismatch: heartbeat turns can execute with `heartbeat.model`, but `session_status` only derives its displayed model from session/config state and does not receive the transient model selected for the active heartbeat run.

## What This Changes

Make `session_status({"sessionKey":"current"})` report the active heartbeat run model when a heartbeat uses `heartbeat.model` as a per-turn override.

## Best Possible Solution

Pass the selected active-run provider/model into the status tool context and use it only for semantic-current readback during that run, without persisting heartbeat model state.

## Reproduction Assessment

Yes at source level: configure a heartbeat with `heartbeat.model`, let the heartbeat call `session_status({"sessionKey":"current"})`, and current main has no field for the tool to read the transient selected heartbeat model. I did not live-fire a heartbeat in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to thread the selected runtime model through existing tool construction for current-session status only. Persisting the heartbeat model into session state would be the wrong direction because related reports show that state must not leak after the heartbeat turn.
