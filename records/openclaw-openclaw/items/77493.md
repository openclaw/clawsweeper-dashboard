## Summary

Keep open. Current main still selects `heartbeat.model` for heartbeat execution but does not thread that transient runtime model into `session_status`, and #77553 is an open closing PR that should be reviewed or landed before closing this issue.

## What This Changes

The issue asks for `session_status({"sessionKey":"current"})` during heartbeat runs to report the active `heartbeat.model` override rather than the session/default model.

## Best Possible Solution

Land a narrow fix that exposes the selected current-run model to `session_status` only for semantic-current lookups, while leaving explicit other-session status reads and persisted session model state unchanged.

## Reproduction Assessment

Yes, at source level: configure a heartbeat with `heartbeat.model`, let that heartbeat call `session_status({"sessionKey":"current"})`, and current main has no field for the tool to receive the selected heartbeat provider/model. I did not run a live heartbeat turn in this read-only review.

## Solution Assessment

Yes. Threading the selected runtime model through the existing tool context for current-session status is the narrow maintainable direction; updating session persistence would conflict with the related heartbeat model leakage boundary.
