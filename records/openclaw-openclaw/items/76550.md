## Summary

Keep open: current main still has the reported doctor recovery gap. The stale-plugin repair removes missing plugin ids from allow/entries and dangling channel-related config, but it does not scan or mutate `plugins.slots.*`, while explicit missing memory slots still fail config validation hard.

## What This Changes

The issue asks `doctor --fix` to repair or warn about stale `plugins.slots.*` references when a configured plugin is missing instead of leaving the config invalid after allow/entries cleanup.

## Best Possible Solution

Extend doctor stale-plugin cleanup to handle selected slot references atomically with allow/entries cleanup, using existing default slot reset semantics and adding regression coverage plus docs/changelog updates.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an explicit `plugins.slots.memory` value pointing at a missing plugin fails validation, and current doctor stale cleanup cannot remove or warn on that slot reference. I did not execute the CLI because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is in doctor stale-plugin cleanup, not by weakening config validation or adding a new setting. It should treat slot refs as part of the same stale plugin repair unit and cover the sibling state reported in #76551.
