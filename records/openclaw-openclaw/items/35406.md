## Summary

Current main does not yet implement the requested binary-document guard in the `read` tool, and an open draft PR already tracks the narrow fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that the `read` tool avoid returning raw binary output for `.docx`, `.pdf`, `.pptx`, `.xlsx`, and similar document formats.

## Best Possible Solution

Review, land, or replace the existing draft PR with a narrow read-wrapper fallback that blocks known binary document formats while leaving full document extraction to the existing PDF/document-extract surfaces.

## Reproduction Assessment

Yes, from source inspection: current main has a clear path where non-image documents can pass through the read wrapper to the upstream UTF-8 text path. I did not live-run a fixture under this read-only review.

## Solution Assessment

Yes, the safe fallback guard is the narrowest maintainable direction for this issue; automatic Office/PDF parsing should remain a separate document extraction capability rather than being folded into `read` by default.
