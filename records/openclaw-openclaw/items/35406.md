## Summary

Keep open. Current main still appears source-reproducible for the reported read-tool behavior: OpenClaw wraps the upstream read tool but does not add a binary-document guard, and the linked draft PR that explicitly closes this issue is still open.

## What This Changes

The issue requests safer `read` tool handling for binary document formats so `.docx`, `.pdf`, `.pptx`, and `.xlsx` do not return raw binary text.

## Best Possible Solution

Land or replace PR #57600 with a small read-wrapper guard for known binary document formats, regression coverage for host and sandbox read paths, and keep full document parsing as separate PDF/document-tool work.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Current main delegates non-image reads to upstream `@mariozechner/pi-coding-agent@0.71.1`, whose non-image branch UTF-8-decodes file bytes, so a DOCX/OOXML ZIP payload can surface as `PK...` text.

## Solution Assessment

Yes for the safe-fallback direction. A targeted guard in the OpenClaw read wrapper is narrower than automatic Office parsing, while PDF analysis can continue to live in the dedicated `pdf`/document extraction surface.
