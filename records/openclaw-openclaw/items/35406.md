## Summary

Keep open. Current main still lacks a `read`-path binary document fallback, and the open PR that uses closing syntax for this issue has not merged.

## What This Changes

Request that the filesystem `read` tool avoid raw UTF-8 output for binary document formats such as DOCX, PDF, PPTX, and XLSX, either by warning or extracting text.

## Best Possible Solution

Land or replace a bounded read-wrapper guard for known binary document formats, while routing richer parser-backed extraction through the document-extract/plugin path with focused security review.

## Reproduction Assessment

Yes. Source inspection shows current main delegates non-image reads to `@mariozechner/pi-coding-agent@0.73.1`, whose non-image branch UTF-8-decodes file bytes, so OOXML ZIP payloads can still surface as `PK...` text.

## Solution Assessment

Yes for the safe-fallback direction, but current main does not implement it. The narrowest maintainable fix is bounded MIME or extension handling in the read wrapper, with document parsers treated as separate extraction work.
