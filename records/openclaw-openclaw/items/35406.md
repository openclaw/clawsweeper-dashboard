## Summary

Keep open. Current main still lacks a `read` fallback for PDF/Office-style binary documents, and the open PR that uses closing syntax for this issue has not merged and still needs maintainer handling.

## What This Changes

Prevent `read` from returning raw UTF-8-decoded binary for PDF/Office documents by warning or routing supported formats through document extraction.

## Best Possible Solution

Land or replace a focused, bounded `read` wrapper guard for known binary document formats, while keeping richer Office/PDF parsing behind the document-extract plugin path with explicit limits and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows non-image `read` calls delegate to the upstream reader, and the upstream reader UTF-8-decodes non-image buffers; this establishes the reported behavior without a live tool run.

## Solution Assessment

Yes for the safe-fallback direction, but current main does not implement it. The narrow maintainable path is a bounded MIME/extension guard in `read`, with parser-backed extraction handled through the existing document-extract surface.
