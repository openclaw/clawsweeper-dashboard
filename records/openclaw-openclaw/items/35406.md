## Summary

Keep open: current main still lacks a generic `read` guard or parser path for PDF/Office binary documents, and an open linked PR explicitly closes this issue but has not merged.

## What This Changes

Prevent the `read` tool from returning raw binary text for PDF and Office-style documents, either by warning or extracting readable text.

## Best Possible Solution

Keep this as the canonical issue until maintainers land one bounded document-read path with limits, regression coverage, docs for any public contract change, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection plus the pinned upstream read-tool source show non-image files are read as UTF-8 text, while current OpenClaw only normalizes image results and has no document guard; I did not run a live binary read in this read-only review.

## Solution Assessment

Unclear for the final implementation. Keeping this issue open and tied to the existing PRs is the right path while maintainers decide whether the canonical fix is a bounded warning fallback, document-extractor-backed parsing, or a narrower format-specific path.
