## Summary

Keep open. Current main still routes `read` through the upstream text/image reader with no PDF/Office fallback or extraction branch, and an open draft PR already uses closing syntax for this issue while a related DOCX extraction PR needs maintainer/security boundary decisions.

## What This Changes

Request the `read` tool to avoid returning raw binary-looking output for PDF and Office document formats by warning or extracting text.

## Best Possible Solution

Land or close one maintainer-approved document-read path that prevents raw binary prompt output while preserving read limits; parser-backed extraction should go through the documented document-extract/plugin contract.

## Reproduction Assessment

Yes. Source inspection plus the pinned upstream package gives a high-confidence path: non-image `read` results can still be decoded as UTF-8, so PDF and Office documents can produce binary-looking text.

## Solution Assessment

Unclear as a final product choice. A safe fallback is the narrowest first step, while parser-backed extraction needs maintainer agreement on limits and the document-extract/plugin boundary.
