## Label Justifications

- `P2`: The issue affects a common agent tool path and can waste context, but it is not an outage or urgent regression.
- `impact:session-state`: Raw binary document bytes can pollute agent-visible context and degrade the transcript/state used for reasoning.

## Summary

Keep open: current main and v2026.5.22 still do not guard filesystem `read` for PDF/Office documents, and the issue has open linked PRs for the fallback and DOCX extraction directions that need maintainer review before the tracking issue closes.

## What This Changes

The issue asks OpenClaw to stop the filesystem read tool from returning raw binary/Office document bytes and instead warn or extract readable document text.

## Best Possible Solution

Keep this issue as the canonical tracker, land one bounded read-path guard to stop raw binary token waste, and route parser-backed extraction through the shared document-extractor boundary only after maintainers approve that scope.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates non-image reads to the upstream read tool, and the locked upstream implementation converts non-image buffers to UTF-8 text.

## Solution Assessment

Unclear as one implementation direction. A bounded warning fallback is the safest first step, while parser-backed extraction should use the shared document-extractor boundary rather than a parallel core parser.
