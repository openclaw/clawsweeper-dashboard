## Summary

Keep open. Current main still lacks a PDF/Office preflight in the generic read wrapper, and open draft PR #57600 explicitly closes this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that the generic read tool avoid returning garbled raw binary content for PDF and Office document formats, preferably by returning a concise fallback or using a document parser.

## Best Possible Solution

Land PR #57600 or a maintainer-owned equivalent that adds a small read-wrapper preflight for PDF and Office documents, reuses shared MIME detection where possible, covers host and sandbox paths, and keeps richer parsing in the PDF/document-extract surfaces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: read a `.pdf`, `.docx`, `.xlsx`, or `.pptx` through the generic `read` tool and current main delegates to the upstream non-image path, which decodes the buffer as UTF-8.

## Solution Assessment

Yes. The safe-fallback guard requested in the issue and active PR is the narrowest maintainable fix for generic `read`; automatic parsing should stay in the existing `pdf` tool or future document-extractor tooling.
