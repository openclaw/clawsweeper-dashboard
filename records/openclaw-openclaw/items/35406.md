## Summary

Keep open: current main and v2026.6.10 still send non-image `read` files through text decoding, so PDF/Office bytes can still enter tool output; the only open closing-reference PR is a conflicting fork PR, while same-repo attempts are closed unmerged.

## What This Changes

Prevent the `read` tool from returning raw PDF and Office-document bytes as text, or add native extraction for those formats.

## Best Possible Solution

Land or replace the linked work with a narrow read-path guard that prevents raw binary document bytes from entering context, points PDFs toward the existing PDF tool where appropriate, and leaves Office extraction to a maintainer-approved document-extractor/plugin design.

## Reproduction Assessment

Yes, at source level. Current `read` only detects image MIME before non-image files are decoded as text, so a DOCX ZIP/PDF buffer can become `read` tool text output; I did not run a live binary read in this read-only review.

## Solution Assessment

Unclear. A safe fallback is likely the narrowest first fix, but automatic document parsing and PDF routing need maintainer decisions about ownership, security limits, and shared extractor boundaries.
