## Summary

Keep open: current main and the latest release still do not add a binary-document guard or extraction path to the filesystem `read` tool. The related open PRs are implementation candidates, but they have not landed, and the existing PDF tool/gateway attachment handling do not solve this specific `read(path="*.docx")` behavior.

## What This Changes

The issue asks for `read` to avoid returning raw binary output for document formats such as `.docx`, `.pdf`, `.pptx`, and `.xlsx`, either by warning or by extracting readable text.

## Best Possible Solution

Keep this as the canonical issue until maintainers either land a bounded `read` fallback/extraction implementation or deliberately close the linked implementation PRs and choose a different document-handling boundary.

## Reproduction Assessment

Yes, by source inspection. Current main still routes non-image files through the upstream `read` UTF-8 text path, and the pinned upstream implementation only special-cases images before decoding buffers as text.

## Solution Assessment

Unclear for the implementation, but keeping the issue open is the right triage action. The safest product direction is a bounded read fallback first, with parser-backed extraction only through an approved document-extractor boundary and size/security limits.
