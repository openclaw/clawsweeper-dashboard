## Summary

Keep open. Current main still lacks a `read` fallback or extraction path for PDF/Office-style binary documents, and the open closing PR at https://github.com/openclaw/openclaw/pull/57600 is still unmerged, draft, and conflicting while the broader DOCX extraction PR at https://github.com/openclaw/openclaw/pull/57612 also remains open and conflicting.

## What This Changes

Request to make the `read` tool avoid raw UTF-8 output for binary document formats by returning a safe warning or extracting readable text.

## Best Possible Solution

Keep this issue open until a bounded safe-fallback guard or maintainer-approved document extraction path lands; richer parser-backed extraction should go through the document-extract/plugin seam with limits and docs.

## Reproduction Assessment

Yes. Source inspection on current main plus the pinned upstream read implementation gives a high-confidence path: non-image `read` results are decoded as UTF-8, so PDF/Office documents can still produce garbled text.

## Solution Assessment

Yes for the safe-fallback direction, but it is not implemented on current main. The maintainable path is bounded MIME/extension detection in `read`, with parser-backed extraction handled through document-extract or explicit document tools after security review.
