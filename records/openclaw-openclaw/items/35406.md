## Summary

Keep open: current main and v2026.6.8 still only detect image files before non-image reads fall through to UTF-8 decoding, so Office/PDF bytes can still become transcript-visible text. The in-repo fix attempts were closed unmerged, and the remaining open fork PR is conflicting, so there is no safe implemented or canonical merged fix yet.

## What This Changes

The issue requests a safer read-tool behavior for binary document formats, either by returning a clear fallback message or by routing/parsing supported document text.

## Best Possible Solution

Add a bounded core read-tool guard that prevents raw binary document bytes from entering the transcript, points PDFs to the existing pdf tool where available, and leaves parser-backed Office extraction to an explicitly approved document-extractor path.

## Reproduction Assessment

Yes. Source inspection shows non-image files flow through image-only detection and then UTF-8 text decoding, so a DOCX ZIP/PDF buffer can become read-tool text output; I did not run a live reproduction in this read-only review.

## Solution Assessment

Unclear. The safe fallback direction is likely the narrowest maintainable first step, but maintainers still need to choose the detection and document-tool routing contract before implementation.
