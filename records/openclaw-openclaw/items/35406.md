## Summary

Keep open: current main still lacks a generic `read` fallback or parser path for PDF/Office documents, and the linked implementation PRs are still open and need maintainer boundary/proof decisions before this canonical report can close.

## What This Changes

The issue asks for `read` to avoid returning raw binary-looking output for `.docx`, `.pdf`, `.pptx`, and `.xlsx`, either by returning a clear fallback message or by extracting document text.

## Best Possible Solution

Keep this as the canonical issue until maintainers land one bounded document-read path with regression coverage, real behavior proof, and a clear document-extractor boundary decision.

## Reproduction Assessment

Yes, by source inspection. Current main only wraps generic `read` with adaptive paging and image normalization, while PDF/Office document handling is absent from `read` or routed through separate `input_file`/PDF surfaces.

## Solution Assessment

Unclear for the final implementation. A lightweight fallback is plausible, but parser-backed extraction should be coordinated with the existing document-extractor contract and the open PRs before the issue is considered solved.
