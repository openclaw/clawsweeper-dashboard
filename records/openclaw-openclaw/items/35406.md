## Summary

Keep open: current main still has no PDF/Office document fallback in the read wrapper, and the linked open implementation PRs are the right place to resolve the issue once their proof, resource-limit, and product-boundary questions are settled.

## What This Changes

The issue asks OpenClaw to stop the `read` tool from emitting raw binary output for PDF and Office document formats, either with a clear fallback message or document extraction.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land a bounded read fallback or document-extractor-backed path, then close it with the merged PR and release/main provenance.

## Reproduction Assessment

Yes, by source inspection. The pinned upstream read tool treats non-image files as UTF-8 text, and current main adds only image normalization/sanitization around that path rather than a document fallback.

## Solution Assessment

Unclear as a final product path. A bounded fallback is the safest narrow fix, but parser-backed extraction should go through the established document-extractor boundary and be coordinated with the open document-handling PRs.
