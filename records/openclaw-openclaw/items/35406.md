## Summary

Keep open. Current main still lacks a `read` fallback or extraction path for PDF/Office-style binary documents, and an open draft PR already uses closing syntax for this issue, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Request a safe fallback or parser-backed path so `read` does not UTF-8 decode binary document formats such as PDF, DOCX, PPTX, and XLSX into token-wasting garbage text.

## Best Possible Solution

Keep this issue open until a bounded safe-fallback guard or maintainer-approved document extraction path lands; richer parser-backed extraction should go through explicit document tooling or plugin contracts with limits and docs.

## Reproduction Assessment

Yes. Source inspection on current main plus the pinned upstream read implementation gives a high-confidence path: non-image `read` results are decoded as UTF-8, so PDF/Office documents can still produce garbled text.

## Solution Assessment

Yes for the safe-fallback direction, but it is not implemented on current main. The maintainable path is bounded MIME/extension detection in `read`, with parser-backed extraction handled through document-extract or explicit document tools after security review.
