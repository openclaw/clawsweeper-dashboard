## Summary

Keep open. Current main still lacks a `read` fallback for PDF/Office-style binary documents, and the open PR that uses closing syntax for this issue is still unmerged, draft, and conflicting.

## What This Changes

The issue asks for the `read` tool to avoid returning raw binary content for PDF and Office-style document files, either by warning or by extracting text.

## Best Possible Solution

Land or replace a bounded safe-fallback guard in `read` for known binary document formats, while keeping richer parser-backed extraction behind the document-extract plugin or explicit document tools with limits and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows non-image `read` calls delegate to the upstream reader, and the upstream reader UTF-8-decodes non-image buffers.

## Solution Assessment

Yes for the safe-fallback direction, but it is not implemented on current main. The narrow maintainable path is bounded MIME/extension detection in `read`, with parser-backed extraction handled through the document-extract/plugin path after security review.
