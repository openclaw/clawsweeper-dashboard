## Label Justifications

- `P2`: The issue is a source-reproducible, bounded tool defect that can waste substantial context but does not make the core runtime unavailable.
- `impact:session-state`: Garbled binary text is inserted into the active transcript, consuming context and degrading reasoning in later turns.

## Summary

Current main still decodes every non-image file buffer as UTF-8, so binary Office documents can enter the transcript as garbled text. This issue should remain open because the source-reproducible problem remains and the open draft PR at https://github.com/openclaw/openclaw/pull/57600 explicitly owns the proposed fix.

## What This Changes

Prevent the read tool from returning raw binary document data as text, using a safe rejection or supported text-extraction path.

## Best Possible Solution

Add a content-aware unsupported-binary guard to the current shared session reader, retain existing image behavior, and direct supported PDF or document workflows to the dedicated document tools instead of building a second parser stack into generic read.

## Reproduction Assessment

Yes, at source level with high confidence: give `read` a DOCX or another non-image binary file, and the current implementation decodes its buffer as UTF-8 because only image MIME types receive special handling. No live test was run because this review was required to remain read-only.

## Solution Assessment

The recommended safe fallback is the narrowest maintainable solution, but the existing PR is not the best current implementation because it targets superseded files and relies primarily on a format list. The fix should move into the current shared reader and use content-aware binary detection while preserving dedicated document extraction.
