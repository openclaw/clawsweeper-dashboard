## Summary

Keep open: current main still lets non-image `read` files fall through to text decoding, so PDF/Office bytes can still pollute the tool result. The existing PDF tool is adjacent but does not make `read` safe for binary document formats, the same-repo guard/extraction PRs are closed unmerged, and the open fork PR that references this issue is conflicting rather than a safe canonical fix.

## What This Changes

The issue requests that the `read` tool avoid returning raw binary Office/PDF bytes and instead provide a safe fallback message or document text extraction.

## Best Possible Solution

Add a bounded core `read` guard that prevents raw binary document bytes from entering context, points PDFs toward the existing PDF path where appropriate, and leaves broader Office extraction to a maintainer-approved document extractor or plugin design.

## Reproduction Assessment

Yes, at source level. Current main only detects images before non-image files are decoded as text, so a DOCX ZIP/PDF buffer can become `read` tool text output; I did not run a live binary read in this read-only review.

## Solution Assessment

Unclear. A safe fallback is likely the narrowest first fix, but automatic document parsing and PDF routing need maintainer decisions about ownership, security limits, and shared extractor boundaries.
