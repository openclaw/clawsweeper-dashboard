## Summary

Keep open: current main and the latest shipped release still lack a `read`-tool fallback or extraction branch for PDF/Office document formats, and the linked implementation PRs remain open candidates rather than merged fixes.

## What This Changes

The issue requests that `read` avoid returning raw binary output for PDF and Office-style document formats, either by warning or by extracting readable text.

## Best Possible Solution

Keep this as the canonical issue until maintainers land one bounded document-read path with size limits, regression coverage, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows `createOpenClawReadTool` delegates non-image reads to upstream, and upstream `pi-coding-agent` v0.74.1 decodes non-image buffers as UTF-8; I did not run a live binary read because this review is read-only.

## Solution Assessment

Unclear for the final implementation. A warning fallback is the narrowest way to stop token waste, while native extraction should be coordinated with the existing document-extractor/plugin boundary and bounded read limits.
