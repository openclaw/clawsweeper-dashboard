## Summary

Keep open: current main still has no `read`-tool guard or parser path for PDF/Office binary documents, and there are open linked implementation PRs that should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for the filesystem `read` tool to avoid returning raw binary output for PDF and Office-style document files, either by warning or by extracting text.

## Best Possible Solution

Keep this as the canonical issue until maintainers land one bounded document-read path with limits, regression coverage, docs for any public contract change, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection plus the pinned upstream read-tool source show non-image files are read as UTF-8 text, while current OpenClaw only normalizes image results and has no document guard; I did not run a live binary read because this review is read-only.

## Solution Assessment

Unclear for the final implementation. Keeping this issue open and tied to the existing PRs is the right path while maintainers decide whether the canonical fix is a bounded warning fallback, document-extractor-backed parsing, or a narrower format-specific path.
