## Summary

Keep open: current main still lacks a read-path guard or extraction branch for binary Office documents, and open implementation candidates are already linked but not merged.

## What This Changes

Request that `read` avoid returning raw binary output for Office/PDF documents, either by a clear fallback message or text extraction.

## Best Possible Solution

Ship a bounded, documented `read` behavior that prevents binary token waste, with parser extraction added only through an approved document-extraction boundary and size limits.

## Reproduction Assessment

Yes, by source inspection. Current main’s `read` wrapper only performs adaptive text paging plus image normalization/sanitization, so binary document handling remains source-reproducible without a live file-read run.

## Solution Assessment

Unclear. The safe fallback direction is plausible, but the current open PRs still need maintainer review for file-size bounds, document-parser ownership, and real behavior proof.
