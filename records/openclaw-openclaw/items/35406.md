## Summary

Keep open. Current main still has no PDF/Office fallback or extraction branch in the `read` wrapper, the pinned upstream read implementation still decodes non-image buffers as UTF-8, and https://github.com/openclaw/openclaw/pull/57600 remains an open draft PR with closing syntax for this issue.

## What This Changes

The issue requests that the `read` tool avoid returning raw binary data for document formats such as PDF, DOCX, PPTX, and XLSX, either by returning a safe fallback message or extracting text.

## Best Possible Solution

Land one bounded, maintainer-approved path: either a safe document fallback in `read` or parser-backed extraction through the document-extract/plugin boundary, with limits, tests, and docs aligned.

## Reproduction Assessment

Yes. Source inspection of current main plus the pinned upstream package gives a high-confidence path: non-image `read` results can still be decoded as UTF-8, so PDF and Office documents can produce garbled text.

## Solution Assessment

Yes for the safe-fallback direction, but it is not implemented on current main. Parser-backed extraction should stay bounded and go through the maintainer-approved document extraction/plugin contract rather than ad hoc raw binary reads.
