## Summary

Keep open: current main and v2026.6.5 still decode every non-image filesystem read as UTF-8, so binary Office/PDF data can enter model context; the linked draft PR tracks the needed guard but requires a bounded current-main implementation and maintainer direction on rejection versus extraction.

## What This Changes

Prevent filesystem reads of binary Office and PDF documents from returning garbled, token-heavy output by rejecting them with actionable guidance or extracting readable text.

## Best Possible Solution

Integrate bounded binary detection into the canonical session read backend, route PDFs to the existing `pdf` tool, and return actionable guidance for unsupported Office formats without rereading the entire file or duplicating document parsers.

## Reproduction Assessment

Yes, from current source with high confidence: non-image reads reach `buffer.toString("utf-8")`, so binary Office and PDF bytes can become model-visible text. This read-only review did not execute a live binary file, but the failing path is direct and unambiguous.

## Solution Assessment

No, the linked PR is not yet the best implementation because it rereads the full file for detection. The safer solution is bounded detection at the canonical read backend, with PDFs directed to the existing dedicated tool and unsupported Office formats rejected clearly.
