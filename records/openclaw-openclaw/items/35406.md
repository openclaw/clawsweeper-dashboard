## Summary

Keep open. Current main still lacks a PDF/Office document guard in the generic read tool, and the active implementation path is the open draft PR #57600, whose body explicitly links this issue for automatic closure after merge.

## What This Changes

The issue requests that the read tool stop returning raw binary output for PDF and Office document formats, preferably by returning a concise fallback or using a parser.

## Best Possible Solution

Land PR #57600 or a maintainer-owned equivalent that adds a small preflight in `src/agents/pi-tools.read.ts` before adaptive paging, reuses shared MIME detection for PDF and Office formats, covers host and sandbox read paths, returns a concise fallback pointing PDFs to the `pdf` tool and Office files to parser/conversion tooling, and adds regression coverage for `.pdf`, `.docx`, `.xlsx`, and `.pptx`.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: read a `.docx`, `.pdf`, `.xlsx`, or `.pptx` through the generic `read` tool and current main delegates to the upstream non-image path, which decodes the file buffer as UTF-8.

## Solution Assessment

Yes. The issue's safe-fallback option is the narrowest maintainable default for generic `read`; automatic parsing should remain in the PDF tool or future document-parser tooling rather than making every read call parse binary documents.
