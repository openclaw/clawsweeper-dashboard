## Summary

Keep open: current main and v2026.6.8 still only detect images before non-image reads fall through to text decoding, so Office/PDF bytes can still become read-tool output. Related same-repo fix attempts are closed unmerged, and the open cross-repo closing PR is conflicting rather than a safe canonical fix.

## What This Changes

The issue asks OpenClaw to stop the generic read tool from returning garbled raw content for binary document formats and either warn or route users toward document extraction.

## Best Possible Solution

Add a bounded core read-tool guard that prevents raw binary document bytes from entering the transcript, points PDFs to the existing pdf tool when appropriate, and leaves Office extraction to a maintainer-approved shared document-extractor path.

## Reproduction Assessment

Yes, at source level. Current main only detects image MIME before non-image files are decoded as text, so a DOCX ZIP/PDF buffer can become read-tool text output; I did not run a live binary read in this read-only review.

## Solution Assessment

Unclear. The safe fallback is likely the narrowest first step, but maintainers still need to choose the detection and routing contract before implementation.
