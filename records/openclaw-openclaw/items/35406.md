## Summary

Keep open: current main and v2026.5.22 still route filesystem `read` through an image-only wrapper and the upstream non-image UTF-8 read path. The issue also has open linked implementation candidates, so it should remain open until maintainers land or close https://github.com/openclaw/openclaw/pull/57600 and decide how https://github.com/openclaw/openclaw/pull/57612 fits the parser-backed direction.

## What This Changes

The issue requests that `read` avoid returning garbled raw binary content for PDF and Office document formats, either by returning a safe warning or by extracting document text.

## Best Possible Solution

Keep this issue as the canonical tracking thread, land one bounded read-path guard to stop raw binary token waste, and route parser-backed extraction through the shared document-extractor boundary when maintainers approve that scope.

## Reproduction Assessment

Yes, source-reproducible. Current main delegates non-image reads to the upstream read tool, and the locked upstream implementation converts non-image buffers to UTF-8 text.

## Solution Assessment

Unclear as one implementation direction. A bounded warning fallback is the safest first step, while parser-backed extraction should use the shared document-extractor boundary rather than a parallel core parser.
