## Summary

Keep open. Current main still lacks a `read`-path fallback for PDF/Office binary documents, and the open implementation PR that uses closing syntax for this issue has not merged.

## What This Changes

Request that `read` avoid returning raw binary document bytes for PDF and Office formats, either by returning a safe fallback message or by extracting readable text.

## Best Possible Solution

Land or replace a focused, bounded read-wrapper guard for known binary document formats, while keeping richer Office parsing behind the document-extract/plugin path with separate limits and tests.

## Reproduction Assessment

Yes. Source inspection shows current main delegates non-image reads to the upstream read tool, and that dependency UTF-8-decodes non-image buffers; I did not run a live tool invocation because dependencies are unavailable, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes for the safe-fallback direction, but current main does not implement it. The narrowest maintainable fix is a bounded MIME/extension guard in the read wrapper, with parser-backed extraction handled through the document-extract surface.
