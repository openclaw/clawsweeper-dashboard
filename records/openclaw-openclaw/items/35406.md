## Summary

Keep open: current main and v2026.6.10 still route non-image `read` files through text decoding, so PDF/Office bytes can still enter the tool result; related same-repo PRs are closed unmerged and the open fork PR is conflicting rather than a safe canonical fix.

## What This Changes

The issue asks OpenClaw to stop the `read` tool from returning raw binary output for document formats such as PDF, DOCX, PPTX, and XLSX, either by warning or extracting text.

## Best Possible Solution

Add a bounded core `read` guard that prevents raw binary document bytes from entering context, points PDFs toward the existing PDF path where appropriate, and leaves Office extraction to a maintainer-approved shared document-extractor/plugin design.

## Reproduction Assessment

Yes, at source level. Current main only detects images before non-image files are decoded as text, so a DOCX ZIP/PDF buffer can become `read` tool text output; I did not run a live binary read in this read-only review.

## Solution Assessment

Unclear. A safe fallback is likely the narrowest first fix, but automatic document parsing and PDF routing need maintainer decisions about ownership, security limits, and shared extractor boundaries.
