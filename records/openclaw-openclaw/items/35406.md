## Summary

Keep open: current main and the latest release still decode non-image file buffers as UTF-8 in the read tool, so Office/PDF binaries can still enter the transcript as garbled text. The same-repo guard PRs were closed unmerged, and the remaining related open issues cover Telegram attachment extraction or generic upload persistence rather than this generic read-tool fallback.

## What This Changes

The issue requests that the read tool avoid returning raw binary output for Office/PDF document formats, either by warning or by extracting text.

## Best Possible Solution

A bounded core read-tool guard should prevent raw document bytes from entering the transcript and direct PDFs to the existing pdf tool or document extraction path where appropriate.

## Reproduction Assessment

Yes. Source inspection shows non-image files flow through UTF-8 decoding, so a DOCX ZIP/PDF buffer can become text output; I did not run a live reproduction in this read-only review.

## Solution Assessment

Unclear. The safe fallback is likely the narrowest maintainable direction, but maintainers still need to settle the detection/routing contract instead of landing an obsolete PR patch.
