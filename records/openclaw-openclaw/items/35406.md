## Label Justifications

- `P2`: The issue affects a common agent read-tool workflow and can waste context, but it is not an outage or urgent regression.
- `impact:session-state`: Raw binary document bytes can pollute the agent-visible transcript/context and degrade subsequent reasoning.

## Summary

Keep open: current main still routes non-image PDF/Office reads through the generic upstream read path, the latest release has the same wrapper shape, and the open closing PR remains unmerged while maintainers decide the fallback-versus-extraction boundary.

## What This Changes

Request to prevent the `read` tool from returning raw binary output for PDF and Office document formats, either with a safe fallback message or document text extraction.

## Best Possible Solution

Land one bounded document-read path that prevents raw binary bytes from entering context, preserves read size caps, and coordinates parser-backed extraction through the existing document-extractor/PDF tool boundary.

## Reproduction Assessment

Yes, source-reproducible. Current main's wrapper does not add a document guard, sandbox MIME detection only returns images, and the locked upstream read tool converts non-image buffers to UTF-8 text.

## Solution Assessment

Unclear as one implementation direction. A safe fallback is a reasonable first step, while parser-backed extraction should use the shared document-extractor boundary with bounded resource handling rather than a parallel core parser.
