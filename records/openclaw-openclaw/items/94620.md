## Summary

Keep this issue open: current main and the latest release still contain the Telegram fallback path that can turn a code-less EADDRNOTAVAIL fetch failure into the misleading pinned-IP fallback log, and the open candidate PRs have not merged yet.

## What This Changes

The issue asks Telegram transport to recognize local EADDRNOTAVAIL socket allocation failures, log an accurate local diagnostic, and avoid IP-rotation fallback for that errno class.

## Best Possible Solution

Land one complete Telegram fix that carries the original fetch/probe error into fallback decisions, skips IP rotation for EADDRNOTAVAIL, preserves valid IPv4 and remote-reachability recovery, and adds focused fetch/probe regression coverage.

## Reproduction Assessment

Yes, at source level: current main still falls back for code-less `fetch failed` envelopes and the probe path drops the original fetch error before forced fallback. I did not run a live socket-exhaustion repro.

## Solution Assessment

Yes, keeping this issue open is the right triage outcome until a complete PR lands. The best fix should cover both direct fetch classification and the probe/forceFallback boundary before this issue closes.
