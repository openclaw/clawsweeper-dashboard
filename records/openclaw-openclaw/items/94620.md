## Summary

This issue is still necessary: current main and the latest release still route the reported Telegram fetch-fallback shape into IP rotation, and an open linked PR with closing syntax should be reviewed before the issue is resolved.

## What This Changes

The issue requests accurate Telegram handling for `EADDRNOTAVAIL`: log it as a local socket allocation failure and skip remote Telegram API IP-rotation retries for that errno.

## Best Possible Solution

Land one narrow Telegram plugin fix that fail-fast logs `EADDRNOTAVAIL` as local socket allocation failure while preserving fallback for recoverable remote or address-family errors, then close this issue from the merged PR.

## Reproduction Assessment

Yes. A focused source reproduction is to pass a `TypeError("fetch failed")` with a numeric `errno=99` cause through `shouldRetryTelegramTransportFallback`; current main collects no code and therefore treats the envelope as retryable.

## Solution Assessment

Yes. The best fix is the narrow classifier/logging change in the Telegram fetch boundary, with tests proving `EADDRNOTAVAIL` skips IP rotation while existing remote-error fallback behavior remains intact.
