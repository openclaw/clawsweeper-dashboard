## Summary

Keep open: current main and the latest release still contain the Telegram fallback classifier, pinned-IP warning, and probe-forced fallback path that match the reported problem; open linked PRs are candidate fixes, not merged fixes.

## What This Changes

The issue asks Telegram fetch fallback to classify EADDRNOTAVAIL as a local socket allocation failure, skip remote IP rotation for that errno, and log an accurate diagnostic.

## Best Possible Solution

Land one complete Telegram transport fix that carries EADDRNOTAVAIL through both direct fetch classification and probe-forced fallback decisions while preserving valid timeout and IPv4 recovery fallback.

## Reproduction Assessment

Yes, at source level: current main can route code-less fetch failed envelopes and probe-forced fallback into the pinned-IP warning with codes=none. I did not perform a live ephemeral-port exhaustion repro.

## Solution Assessment

Yes, the requested direction is the right narrow fix if it covers every fallback boundary. The best implementation should gate both direct fetch retry and forced probe promotion on the original EADDRNOTAVAIL/local-socket decision.
