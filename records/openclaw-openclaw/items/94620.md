## Summary

Keep open: current main and the latest release still contain the Telegram fallback paths that can produce the misleading pinned-IP fallback log, and https://github.com/openclaw/openclaw/pull/95221 is an open closing candidate rather than a merged fix.

## What This Changes

Request to make Telegram transport treat EADDRNOTAVAIL as a local socket allocation failure, log an accurate diagnostic, and skip remote-IP fallback retries.

## Best Possible Solution

Land one complete Telegram transport fix that carries EADDRNOTAVAIL through both direct fetch classification and probe-forced fallback decisions while preserving valid timeout and IPv4 recovery fallback.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can route a code-less `fetch failed` or probe-forced fallback into the pinned-IP warning with `codes=none`. I did not perform a live ephemeral-port exhaustion repro.

## Solution Assessment

No shipped or current-main solution is present yet. The best fix is a narrow Telegram transport change that gates every automatic or forced IP-rotation path on the original EADDRNOTAVAIL error.
