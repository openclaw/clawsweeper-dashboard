## Summary

Keep open. Current main mitigates silent empty turns with retries and a generic fallback, but it still does not preserve the provider error detail for Qianfan-style top-level error chunks or the `error_msg`/`error_code` variants discussed in the issue.

## What This Changes

The issue asks OpenClaw to surface Baidu Qianfan API error details instead of recording an error stop with null error text and empty output.

## Best Possible Solution

Add shared OpenAI-compatible provider-error normalization for no-choice streamed error chunks and Baidu-style field aliases, with focused Qianfan/OpenAI-compatible regression tests and no bundled-provider ID branch in core.

## Reproduction Assessment

Yes, at source level: a no-choice OpenAI-compatible stream chunk containing `error_msg`/`error_code` is skipped by the current loop, and the shared formatter does not normalize those aliases. I did not run tests because this review is read-only and the checkout must remain clean.

## Solution Assessment

Yes. The best fix is a narrow shared OpenAI-compatible error-normalization path plus regression fixtures; a Qianfan-only core special case or raw-response logging would be less maintainable.
