## Summary

Keep open. Current main still has a credible silent Qianfan error path: Qianfan is routed through the shared OpenAI-compatible stream, Baidu-style `error_msg`/`error_code` payloads are not normalized, and streamed objects without `choices[0]` are skipped instead of becoming a visible `errorMessage`.

## What This Changes

The issue asks OpenClaw to surface Baidu Qianfan API error details for `qianfan/qianfan-code-latest` instead of recording an error turn with null error data and empty content.

## Best Possible Solution

Add shared OpenAI-compatible provider-error normalization that recognizes Baidu/Qianfan top-level and nested error fields, converts streamed error-only chunks into a sanitized assistant `errorMessage`, preserves usage-only chunks, and adds focused regression coverage for Qianfan-style HTTP and SSE payloads.

## Reproduction Assessment

Yes. The issue includes a concrete transcript, and current-main source inspection gives a deterministic unit reproduction: feed a Qianfan-style top-level error chunk without `choices` through `processOpenAICompletionsStream` or a Baidu `error_msg`/`error_code` payload through the shared formatter.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared OpenAI-compatible transport/error-normalization layer with Qianfan regression fixtures, because Qianfan is manifest-routed through `openai-completions` and does not need bundled-provider IDs or Qianfan-specific policy in core.
