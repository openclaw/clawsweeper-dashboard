## Summary

Keep open. Current main improves generic empty-error handling, but it still does not prove the Qianfan-specific error detail loss is fixed: Qianfan uses the shared OpenAI-compatible transport, Baidu-style error fields are not normalized cleanly, and no-choice streamed chunks are still skipped before top-level error payloads can become visible error text.

## What This Changes

The issue asks OpenClaw to surface Baidu Qianfan API error details instead of recording an errored assistant turn with null error data and empty content.

## Best Possible Solution

Add shared OpenAI-compatible provider error normalization for Baidu/Qianfan error_msg, error_message, msg, and numeric error_code variants, and handle error-only streamed chunks before the no-choice continue path. Keep provider-specific branching out of core unless the plugin API needs a generic extension point.

## Reproduction Assessment

Yes. A focused fixture can feed a Baidu-style error_msg/error_code payload through provider error formatting, and a no-choices streamed error object through processOpenAICompletionsStream, to show the missing visible error detail on current main.

## Solution Assessment

Yes. The maintainable fix is in the shared OpenAI-compatible transport/error-normalization layer with Qianfan regression coverage; raw response logging or hard-coded Qianfan branches in unrelated core flow would be weaker.
