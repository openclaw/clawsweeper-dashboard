## Summary

Keep open. Current main still does not prove the Qianfan silent-error path is fixed: Qianfan is routed through the shared OpenAI-compatible completions transport, and the current parser still has gaps for the Baidu-style fields and error-only streamed chunks discussed in the issue.

## What This Changes

The issue requests that Baidu Qianfan API errors be parsed and surfaced to users instead of producing an empty assistant response.

## Best Possible Solution

Add a narrow shared OpenAI-compatible provider-error normalization path that recognizes documented Qianfan errors plus the Baidu-style variants discussed here, turns streamed error-only chunks into sanitized errorMessage output, and covers the behavior with focused regression tests without adding bundled-provider IDs to core logic.

## Reproduction Assessment

Yes for the current source-level gap: a unit fixture with a Baidu-style error_msg/error_code payload or a streamed no-choices error object still exercises a path that does not become visible error text. The exact live Qianfan payload should be captured if available, but it is not required to see the parser gap.

## Solution Assessment

Yes. The best fix is a shared OpenAI-compatible transport/error-normalization change with Qianfan regression coverage; a Qianfan-only core branch or raw response logging would be less maintainable and could risk leaking provider payload details.
