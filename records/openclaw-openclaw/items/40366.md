## Summary

Keep this open for maintainer follow-up. Current main still lacks the central command-delivery and Slack no-final fallback behavior, while the proposed branch is useful but not mergeable as-is because the new fallback paths introduce JSON/metadata regressions and the branch carries broad unrelated churn.

## What This Changes

The PR adds normalized-empty command delivery fallback handling, Slack preview/no-final keepalive and delayed status fallback behavior, related tests, a turn-source delivery guard, and broad formatting-only changes across Control UI and adjacent tests.

## Best Possible Solution

Land a narrower delivery reliability fix that keeps intentional silence intact, sends a user-visible fallback only after final deliverable payload selection, preserves JSON output and returned metadata consistency, adds focused command and Slack regressions, and coordinates any core overlap with #45290 before merging.

## Reproduction Assessment

Yes. The source-level reproduction is clear: a non-empty command payload such as whitespace can produce zero deliverable outbound payloads on current main, and a Slack preview partial that returns no final currently reaches the no-delivery cleanup path without an in-thread fallback.

## Solution Assessment

No, not as-is. The proposed direction is maintainable, but the branch should first move JSON emission after fallback selection or otherwise emit the final payload set, return `resultMeta` from every new branch, and remove unrelated churn or split it out.

## Review Findings

  - body: In `--json --deliver` normalized-empty runs, the function has already written the JSON envelope from the original `normalizedPayloads` before this fallback branch delivers and returns `fallbackPayloadsJson`. Machine consumers will see an empty/old payload list on stdout even though the fallback text was sent, so JSON emission needs to happen after the final payload set is known.
  - confidence: 0.88

  - body: These new fallback/silent returns use `result.meta` instead of the merged `resultMeta`, so callers lose `resultMetaOverrides` such as embedded gateway-fallback provenance. The existing JSON/return contract expects those overrides to be preserved, so these branches should return `resultMeta`.
  - confidence: 0.86
