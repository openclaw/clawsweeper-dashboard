## Summary

This should stay open: current `main` still has a source-level path where gateway startup resolves layered stage roots but then installs the full bundled runtime dependency plan into the final writable root when that root is not already materialized, matching the reported 2026.4.29 behavior and conflicting with the documented layered-stage contract.

## What This Changes

The issue reports that layered `OPENCLAW_PLUGIN_STAGE_DIR` reinstalls the full bundled runtime dependency set into the final writable root even when an earlier read-only layer already contains the complete hashed dependency tree.

## Best Possible Solution

Align gateway startup and doctor repair with the layered stage-dir contract: resolve dependencies from all search roots, do no install when the full plan is already satisfied by earlier layers, install only truly missing dependencies into the final writable root, and keep the full-plan manifest/validation behavior where it is needed to avoid package-manager pruning.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is the existing layered-stage fixture shape: populate the earlier search root, leave the final writable root unmaterialized, and call `ensureBundledPluginRuntimeDeps`; current main passes the full plan as `missingSpecs` instead of skipping or installing only absent deps.

## Solution Assessment

No for the current implementation; it is not the narrowest maintainable solution because it treats final-root materialization as the only convergence check. The requested direction is the right fix if implemented with care around full-plan manifests and partial-missing installs.
