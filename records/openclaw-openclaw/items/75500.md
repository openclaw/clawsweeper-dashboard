## Summary

Current main now avoids the reported full reinstall when a compatible materialized bundled runtime-deps root already exists in the staged root set. The proof is on main after v2026.4.29, so this is fixed on current main but not yet in a tagged release I could identify.

## What This Changes

The issue requests that layered `OPENCLAW_PLUGIN_STAGE_DIR` reuse bundled plugin runtime dependencies from earlier read-only roots and avoid reinstalling the full dependency set into the final writable root when those deps are already present.

## Best Possible Solution

Keep the current main implementation and ship it in the next release: compatible materialized runtime-deps layers should be reused before any package-manager repair, while the final root still owns the current manifest and runtime mirror state.

## Reproduction Assessment

Yes. The current source-level reproduction is the existing staged-root fixture: create a compatible earlier runtime-deps root with the generated manifest, leave the selected root empty, and assert the installer is not called while `node_modules` is linked into the selected root.

## Solution Assessment

Yes. The current fix is narrow for this regression because it adds a reuse check before installation without changing plugin discovery, package-manager repair, or bundled plugin activation semantics.
