## Summary

Keep open. Current main has part of the facade/source fallback work and a changelog entry, but it still lacks this PR head's guarded source public-artifact require-to-Jiti fallback; the PR remains a valid implementation candidate, and I found no blocking correctness or security finding in the diff.

## What This Changes

The PR adds a guarded Jiti fallback for bundled source public-artifact loads that fail on emitted-style `.js` imports, keeps channel entry load profiling explicit, and adds focused loader/runtime-deps regression coverage.

## Best Possible Solution

Have a maintainer land this PR or an equivalent rebased patch after normal validation, keeping the guarded public-artifact Jiti fallback and regression tests; do not close the PR as implemented until that missing fallback is on main.

## Reproduction Assessment

Yes. The current-main source path is traceable without a live run: source public artifacts are loaded through `sourceArtifactRequire`, and `extensions/anthropic/provider-policy-api.ts` imports a sibling via a `.js` specifier; the PR adds a focused test that models the resulting module-not-found path.

## Solution Assessment

Yes. The PR keeps the source `require` fast path, narrows fallback to module-not-found errors whose message references a `.js` specifier, and preserves syntax errors on the original path, which is the narrowest maintainable fix I found.
