## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads, but it does not bridge a proxy-injected shared-secret header into the browser WebSocket connect path; the prior discussion correctly treats this as a Gateway auth design item.

## What This Changes

The issue requests accepting a reverse-proxy-supplied Gateway token from HTTP headers for Control UI bootstrap and WebSocket authentication.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve current URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, docs, and browser-hardening tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection of current main shows the requested feature is absent: HTTP bootstrap reads can use bearer auth, but browser WebSocket shared auth is resolved from connectParams.auth or trusted-proxy identity mode.

## Solution Assessment

Unclear as an implementation plan. The direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, rate-limit, device, and revocation rules before code is queued.
