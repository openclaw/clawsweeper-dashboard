## Summary

Keep open. Current main and the latest release still separate bearer-authenticated Control UI HTTP reads from WebSocket connect-frame authentication, so a proxy-injected Authorization or X-OpenClaw-Token header cannot bootstrap the browser WebSocket session; the remaining work is a security/product decision, not stale cleanup.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided Authorization or X-OpenClaw-Token headers instead of URL/manual token entry.

## Best Possible Solution

Define one hardened reverse-proxy bootstrap contract, preferably using a purpose-specific trusted header or same-origin exchange to mint a narrowly scoped short-lived browser credential while preserving existing fragment/manual-token and trusted-proxy identity modes.

## Reproduction Assessment

Yes, at source level with high confidence: bearer headers authorize protected Control UI HTTP reads, but the browser WebSocket independently requires credentials in its connect frame and cannot read the proxy-injected request header.

## Solution Assessment

No final implementation direction is approved yet. A purpose-specific short-lived credential exchange or carefully gated upgrade-header path is safer than exposing a long-lived proxy bearer token to browser state or broadly accepting forwarded Authorization headers.

## Risks / Open Questions

- Trusting a generic forwarded Authorization header could confuse an upstream OIDC bearer token with an OpenClaw gateway secret and consume shared-secret rate-limit attempts.
- Returning or persisting a proxy's long-lived gateway secret in browser-readable bootstrap data would widen the credential exposure boundary.
- Authenticating directly during WebSocket upgrade must preserve current origin checks, trusted-proxy source validation, pre-auth connection limits, and shared-secret rate limiting.
