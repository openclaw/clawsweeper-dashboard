## Summary

Keep open. Current main still supports bearer headers for Control UI HTTP reads, but the browser WebSocket shared-secret path is authenticated from the connect frame and there is no bootstrap/session credential that carries a proxy-injected Authorization header into that path; this remains a Gateway auth design change.

## What This Changes

Request Control UI support for bootstrapping gateway shared-secret auth from proxy-provided HTTP headers instead of only URL/manual token state.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve existing URL/manual/Tailscale/trusted-proxy flows, and add focused Gateway, UI, and docs coverage.

## Reproduction Assessment

Yes, from source inspection. Configure token/password gateway auth behind a proxy that injects Authorization for the HTTP config request, leave browser token/password state empty, and the HTTP read can authenticate while WebSocket shared auth is resolved only from connectParams.auth.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the WebSocket-upgrade versus short-lived-session model and associated proxy, origin, device, and rate-limit rules before implementation.
