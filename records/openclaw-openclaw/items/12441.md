## Summary

Keep open. Current main still accepts bearer auth for Control UI HTTP/bootstrap reads, but it does not bridge a reverse-proxy-injected shared-secret header into the browser WebSocket auth path; the remaining work changes the Gateway admin auth boundary and needs maintainer/secops design.

## What This Changes

Allow Control UI shared-secret auth to bootstrap from proxy-provided Authorization or X-OpenClaw-Token headers instead of requiring URL/manual browser state.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and add focused Gateway, UI, and docs coverage.

## Reproduction Assessment

Yes, from source inspection. Configure token/password gateway auth behind a proxy that injects Authorization for the HTTP config request, leave browser token/password state empty, and the HTTP read can authenticate while WebSocket shared auth is resolved only from connectParams.auth.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the WebSocket-upgrade versus short-lived-session model and the associated proxy, origin, device, and rate-limit rules before code is queued.
