## Summary

Keep open. Current main and v2026.5.22 still split Control UI HTTP bearer reads from WebSocket shared-secret auth, and carrying proxy-provided headers into the admin WebSocket path remains a Gateway/security design decision.

## What This Changes

The issue requests Control UI reverse-proxy shared-secret bootstrap from Authorization or X-OpenClaw-Token headers instead of only URL/manual token entry and connect-frame credentials.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen Gateway/UI/docs/security path with focused tests.

## Reproduction Assessment

Yes, source-level. Current main clearly shows HTTP bearer reads are supported while shared-secret WebSocket auth still comes only from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the header versus short-lived-session model and the proxy, origin, revocation, device, and rate-limit rules before implementation.
