## Summary

Keep open. Current main supports bearer auth for Control UI HTTP reads, but shared-secret Control UI WebSocket auth still comes from `connect.params.auth`; proxy-injected `Authorization` or `X-OpenClaw-Token` is not bridged into the browser WebSocket path and remains a Gateway admin auth design item.

## What This Changes

The issue requests allowing Control UI authentication to bootstrap from proxy-provided HTTP headers instead of only URL/manual browser token state.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Gateway admin Control UI, preserve existing URL/manual/trusted-proxy flows, and cover it with Gateway, UI bootstrap, WebSocket auth, and docs tests.

## Reproduction Assessment

Yes. Configure token or password gateway auth behind a proxy that injects Authorization for the page/config request, leave browser token/password state empty, and current source shows the HTTP read can use the header while WebSocket shared auth is resolved from `connectParams.auth` only.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the safest maintainable solution depends on a maintainer/secops choice between honoring trusted upgrade headers directly or issuing a short-lived same-origin browser credential after header validation.
