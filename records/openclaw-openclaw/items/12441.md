## Summary

Keep open. Current main and v2026.6.5 still support bearer auth for Control UI HTTP reads, but shared-secret Control UI WebSocket auth still comes from connect-frame credentials or identity-header modes, so reverse-proxy header bootstrap remains unresolved.

## What This Changes

The issue requests a Control UI auth flow where a reverse proxy can provide a gateway token via Authorization or X-OpenClaw-Token headers instead of requiring URL/manual token bootstrap.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap contract in Gateway auth, then update Control UI, docs, and tests while preserving URL/manual/Tailscale/trusted-proxy paths.

## Reproduction Assessment

Yes. Source inspection shows HTTP bearer reads are supported while shared-secret WebSocket auth still comes only from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the header versus short-lived-session model and the proxy, origin, revocation, device, and rate-limit rules before implementation.
