## Summary

Keep open. Current main still accepts bearer credentials for Control UI HTTP reads but does not bridge a proxy-injected Authorization or X-OpenClaw-Token header into browser WebSocket shared-secret auth, so the existing maintainer-side discussion remains consistent with the code and this needs explicit Gateway auth design.

## What This Changes

Request Control UI support for reverse-proxy-supplied gateway credentials from Authorization or X-OpenClaw-Token headers instead of token-in-URL or manual UI entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve the current URL/manual/Tailscale/trusted-proxy flows, and cover the chosen model with focused Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested feature: HTTP bearer auth can load Control UI read endpoints, but browser WebSocket shared auth is resolved from connectParams.auth and existing identity-header modes only.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, device, and rate-limit rules before implementation.
