## Summary

Keep open. Current main still lacks a shared-secret reverse-proxy header bootstrap for Control UI WebSocket auth; existing code supports bearer auth for HTTP Control UI reads and separate trusted-proxy identity auth, so the remaining work is a Gateway/security design decision.

## What This Changes

Allow Control UI behind a reverse proxy to bootstrap Gateway shared-secret auth from Authorization or X-OpenClaw-Token headers instead of relying on URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives high confidence that the requested feature is absent: HTTP reads can use bearer auth, but browser WebSocket shared auth is still resolved from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, device, revocation, and rate-limit rules before implementation.
