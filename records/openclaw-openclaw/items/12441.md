## Label Justifications

- `P2`: This is a valid, bounded Control UI improvement with limited blast radius, but it requires deliberate security design rather than urgent regression repair.
- `impact:security`: The request changes the browser and Gateway trust boundary and handling of a long-lived shared secret behind reverse proxies.
- `impact:auth-provider`: The issue concerns how gateway credentials are sourced and routed into Control UI WebSocket authentication.

## Summary

The issue is still necessary: current main accepts bearer credentials for protected Control UI HTTP reads, but the browser cannot inherit a reverse-proxy-injected request header and still authenticates the WebSocket through explicit connect-frame credentials. No linked or searchable canonical implementation currently owns the gap, and the existing device bootstrap token serves pairing rather than this reverse-proxy flow.

## What This Changes

Add a secure Control UI reverse-proxy bootstrap path that can derive WebSocket gateway authentication from proxy-supplied HTTP headers without putting long-lived tokens in URLs.

## Best Possible Solution

Issue a short-lived, scoped, same-origin browser credential only after existing Control UI HTTP authentication, origin, trusted-proxy, and rate-limit checks, then redeem it through the established WebSocket connect-auth path while preserving current fragment/manual and trusted-proxy identity modes.

## Reproduction Assessment

Yes at source level: an authenticated HTTP bootstrap request can use a bearer token, but the returned payload contains no WebSocket credential and the browser WebSocket authenticates from connect-frame state that cannot read a proxy-injected page-request header.

## Solution Assessment

The requested direction is valid, but accepting the long-lived header value directly into browser state is not the best fix; a short-lived scoped bootstrap credential is the narrower and safer design.
