## Summary

Current main still does not carry a reverse-proxy-injected shared Gateway secret from the Control UI HTTP request into the browser WebSocket connect path. HTTP bearer reads, trusted-proxy identity auth, and setup-code bootstrap tokens are separate mechanisms, so the requested behavior remains useful but needs an explicit maintainer security/product decision before implementation.

## What This Changes

Allow Control UI to bootstrap Gateway authentication from a reverse-proxy-injected bearer or dedicated token header without putting the long-lived token in the URL or manual UI state.

## Best Possible Solution

Add one explicit trusted-reverse-proxy bootstrap contract that validates the injected shared secret server-side and issues a short-lived, same-origin credential for the existing WebSocket connect-auth path, while preserving manual/fragment token flows and trusted-proxy identity auth.

## Reproduction Assessment

Yes. Current source provides a high-confidence reproduction: a proxy-injected bearer can authorize the HTTP bootstrap read, but that credential is neither returned to the browser nor available when the browser constructs its WebSocket connect frame.

## Solution Assessment

The requested outcome is valid, but direct header reuse or token reflection is not the best implementation. A server-validated, short-lived browser/WebSocket credential using the existing connect-auth boundary is narrower and safer.

## Risks / Open Questions

- A header-bootstrap implementation could make arbitrary client-supplied headers authoritative unless it is restricted to explicitly trusted proxies and retains origin, forwarded-client, and rate-limit checks.
- Reflecting the reusable Gateway secret into HTML, JSON, URLs, logs, or persistent JavaScript storage would recreate or worsen the credential-exposure concern; the browser handoff should be short-lived and narrowly scoped.
- The existing trusted-proxy identity mode and setup-code/device bootstrap tokens have different trust contracts, so a new shared-secret bridge must not silently conflate or bypass those modes.
