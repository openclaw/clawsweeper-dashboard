## Summary

Keep open. Current main still has HTTP bearer auth for Control UI reads, but no supported bridge from a proxy-injected header into browser WebSocket shared-secret auth, and prior discussion already identifies this as a security-sensitive Gateway auth design item.

## What This Changes

Request Control UI authentication to bootstrap a gateway token from proxy-supplied Authorization/X-OpenClaw-Token headers instead of requiring URL or manual UI token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve current URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, and docs tests.

## Reproduction Assessment

Yes, from source inspection. A proxy can authenticate the HTTP config request with Authorization, but a browser with empty token/password state still opens a WebSocket whose shared auth is resolved only from connectParams.auth.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, device, and rate-limit rules before implementation.
