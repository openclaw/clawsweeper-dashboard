## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads, but it does not bridge proxy-injected shared-secret headers into the browser WebSocket auth path; this is a Gateway admin auth feature that needs maintainer/secops design.

## What This Changes

Request Control UI to bootstrap gateway shared-secret auth from proxy-injected Authorization or X-OpenClaw-Token headers instead of only URL/manual UI credentials.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve URL/manual/trusted-proxy flows, and add focused Gateway, UI, and docs coverage.

## Reproduction Assessment

Yes, from source inspection. Configure token/password gateway auth behind a proxy that injects Authorization for the HTTP config request, leave browser token/password state empty, and the HTTP read can authenticate while WebSocket shared auth is resolved from connectParams.auth only.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the WebSocket-upgrade versus short-lived-session model and the associated proxy/origin/rate-limit rules before code is queued.
