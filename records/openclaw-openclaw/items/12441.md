## Label Justifications

- `P2`: This is a normal-priority Gateway auth improvement with meaningful but bounded Control UI impact, not an urgent regression or outage.
- `impact:security`: The requested behavior changes how a shared Gateway secret crosses a reverse proxy into a browser-admin session.
- `impact:auth-provider`: The issue concerns credential routing between Gateway HTTP bearer auth and the Control UI WebSocket authentication path.

## Summary

Keep open. Current main and v2026.6.6 still support bearer tokens for Control UI HTTP reads, but the browser WebSocket shared-secret path is still driven by connect-frame auth and does not bridge a reverse-proxy-provided Authorization or X-OpenClaw-Token header into the Control UI session.

## What This Changes

Request to let Control UI bootstrap Gateway shared-secret auth from reverse-proxy-provided HTTP headers instead of URL fragments or manual UI entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for Gateway auth, either validating a trusted upgrade header after existing security checks or issuing a short-lived same-origin browser session credential, while preserving current URL-fragment/manual token and identity-bearing proxy flows.

## Reproduction Assessment

Yes at source level, though this is a feature request rather than broken existing behavior. Current main accepts bearer auth for Control UI HTTP reads but does not carry proxy-injected headers into WebSocket shared-secret auth.

## Solution Assessment

Unclear until maintainers choose the trust model. Header bootstrap is directionally plausible, but the safest permanent shape may be a short-lived same-origin session credential or a purpose-specific trusted-header path rather than blindly treating every Authorization bearer value as a Gateway token.
