## Label Justifications

- `impact:security`: The request changes the trust boundary and lifetime of a shared Gateway credential crossing from a reverse proxy into a browser session.
- `impact:auth-provider`: The missing behavior is credential routing between Gateway HTTP authorization and the Control UI WebSocket authentication path.

## Summary

Current main still separates bearer-authenticated Control UI HTTP reads from browser WebSocket connect-frame authentication, so a proxy-injected shared secret cannot bootstrap the requested session. The feature remains useful, but the closed direct-header proposal exposed unresolved trust-boundary and credential-scope questions that require maintainer security/product judgment.

## What This Changes

Allow a trusted reverse proxy to bootstrap Control UI WebSocket authentication from an HTTP authorization header without exposing the long-lived gateway token in the browser URL.

## Best Possible Solution

Define a single explicit trusted-proxy exchange that validates a purpose-specific proxy credential after existing source, origin, and rate-limit checks, then issues a short-lived WebSocket-scoped browser credential while preserving manual/fragment token, device, and trusted-proxy identity flows.

## Reproduction Assessment

Yes, at source level with high confidence: bearer authorization can admit the Control UI bootstrap HTTP read, but the bootstrap response has no credential bridge and the browser WebSocket authenticates from connect-frame state instead.

## Solution Assessment

No, the suggested direct header fallback is not yet the best maintainable solution because the closed implementation coupled it to device-auth bypass and could conflate unrelated bearer credentials; a purpose-specific short-lived exchange is safer.
