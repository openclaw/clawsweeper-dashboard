## Summary

Keep open. Current main authenticates Control UI HTTP reads from bearer headers, but the browser WebSocket still receives credentials only through its connect frame; no current implementation or canonical related item bridges a reverse-proxy-injected shared secret into that path. This remains a valid security-sensitive product decision.

Likely related people
- @steipete — high confidence from the Control UI bootstrap and Gateway authentication feature history.
- @vincentkoc — medium confidence from recent work on the same HTTP and WebSocket authentication paths.
- @ngutman — medium confidence from adjacent bootstrap-token precedence work.

## What This Changes

Allow reverse-proxy-provided bearer or dedicated token headers to bootstrap Control UI WebSocket authentication without putting a gateway token in the URL.

## Best Possible Solution

After explicit security approval, issue a narrowly scoped, short-lived same-origin bootstrap credential following validated proxy HTTP authentication and consume it through the existing WebSocket connect-frame machinery, while retaining current manual, fragment, and trusted-proxy identity flows.

## Reproduction Assessment

Yes at source level: current main has no path from a proxy-injected initial-page header into the browser's WebSocket connect credentials. A live reverse-proxy setup was not needed to establish the missing capability, though it will be required to validate an implementation.

## Solution Assessment

No approved implementation exists yet. Directly trusting forwarded Authorization headers is harder to secure than a validated short-lived same-origin bootstrap credential integrated with the existing connect-frame authentication path.
