## Summary

Keep open. Current main and v2026.6.10 still accept bearer auth for Control UI HTTP reads, but Control UI WebSocket shared-secret auth is still driven by connect-frame credentials or identity-bearing proxy modes; adding proxy-injected header bootstrap remains a security-sensitive Gateway auth product decision.

## What This Changes

Allow Control UI behind a reverse proxy to bootstrap shared-secret auth from `Authorization` or `X-OpenClaw-Token` headers instead of relying on URL/manual token handoff.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preferably either a trusted same-origin bootstrap endpoint that issues a short-lived browser session credential or a tightly scoped opt-in header path for the WebSocket upgrade.

## Reproduction Assessment

Yes, at source level. Current main accepts bearer auth for Control UI HTTP reads, but WebSocket shared-secret auth consumes only connect-frame auth and has no header-to-session bridge for proxy-injected credentials.

## Solution Assessment

Unclear until maintainers choose the trust model. Direct header bootstrap is plausible, but a short-lived same-origin session credential or purpose-specific opt-in header may be safer than treating forwarded `Authorization` as a Gateway token.
