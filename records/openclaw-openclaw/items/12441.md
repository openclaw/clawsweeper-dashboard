## Label Justifications

- `impact:security`: The request changes where a privileged Gateway credential is accepted and how authentication crosses the proxy-to-browser trust boundary.
- `impact:auth-provider`: The issue concerns the source and routing of the credential used to authorize the Control UI Gateway connection.

## Summary

Current main still lacks a bridge from HTTP header authentication into the browser WebSocket connect credential, the public docs describe only fragment/manual shared-secret auth or identity-aware proxy auth, and the prior direct header-token implementation was closed unmerged after exposing unresolved coupling and header semantics. This remains a valid security-sensitive feature that needs an explicit maintainer design decision rather than cleanup closure.

## What This Changes

Allow Control UI to authenticate its Gateway WebSocket from a reverse-proxy-injected bearer or dedicated token header without requiring token-in-URL or manual entry.

## Best Possible Solution

Define one explicit trusted-proxy bootstrap mode that validates the direct proxy and same-origin request, then exchanges successful header authentication for a short-lived, scoped WebSocket session credential while preserving existing fragment/manual shared-secret and trusted-proxy identity modes.

## Reproduction Assessment

Yes, at source level with high confidence: HTTP bearer authorization ends in the HTTP handler, the bootstrap payload has no session credential, and the browser WebSocket authenticates through its connect frame. A live nginx or Caddy setup would demonstrate the UX but is not necessary to establish the missing capability.

## Solution Assessment

No, directly copying the long-lived header token into existing UI token state is not the best solution. A short-lived, scoped credential exchange behind explicit trusted-proxy configuration better preserves the current browser and Gateway security boundary.

## Risks / Open Questions

- A naive bootstrap field would expose the long-lived Gateway token to browser JavaScript and any compromised same-origin content.
- Trusting forwarded Authorization headers without an explicit proxy allowlist and direct-connection rejection could permit credential spoofing or confuse upstream OIDC bearer tokens with OpenClaw tokens.
- A new header/session/config contract requires clear expiry, scope, origin binding, rate-limit behavior, and upgrade documentation; otherwise existing proxy and authentication setups could behave unpredictably.
