## Summary

Keep open. Current main and v2026.5.22 still do not implement shared-secret reverse-proxy header bootstrap for Control UI WebSocket auth, and the remaining work is a Gateway/security trust-boundary design choice rather than a strict bug fix.

## What This Changes

Request Control UI support for bootstrapping gateway auth from reverse-proxy injected Authorization or X-OpenClaw-Token headers instead of URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen Gateway/UI/docs/security path with focused tests.

## Reproduction Assessment

Yes, source-level. Current main clearly shows HTTP bearer reads are supported while shared-secret WebSocket auth still comes only from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the header versus short-lived-session model and the proxy, origin, revocation, device, and rate-limit rules before implementation.

- Accepting proxy-provided Authorization as a gateway shared-secret input could confuse upstream OAuth/OIDC bearer credentials with OpenClaw gateway tokens unless header ownership and proxy requirements are explicit.
- The WebSocket-upgrade-header versus short-lived browser-session choice affects origin checks, trusted proxy requirements, rate limiting, device identity, credential lifetime, and revocation behavior.
- Existing trusted-proxy identity auth covers some SSO deployments, but it does not settle whether shared-secret header bootstrap should also be supported.
