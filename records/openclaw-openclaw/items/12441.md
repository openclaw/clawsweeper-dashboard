## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads, but shared-secret WebSocket auth is still sourced from browser connect-frame credentials or the separate trusted-proxy identity mode, so the requested reverse-proxy header bootstrap remains a real Gateway auth design item.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided Authorization or X-OpenClaw-Token headers instead of URL/manual token input only.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen model with Gateway, UI, docs, and security tests.

## Reproduction Assessment

Yes, source-level. Current main clearly shows HTTP bearer reads are supported while shared-secret WebSocket auth still comes only from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, revocation, device, and rate-limit rules before implementation.
