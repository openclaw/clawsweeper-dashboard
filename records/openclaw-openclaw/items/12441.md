## Summary

Keep open. Current main still has bearer auth for Control UI HTTP reads, but it does not turn a proxy-injected Authorization or X-OpenClaw-Token header into the browser WebSocket shared-secret auth path, and the existing comments correctly identify this as a Gateway auth design item.

## What This Changes

The request is for Control UI to bootstrap gateway auth from a reverse-proxy-provided Authorization Bearer or X-OpenClaw-Token header instead of requiring URL or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested feature: HTTP bearer auth can load Control UI read endpoints, but browser WebSocket shared auth is still resolved from connectParams.auth and identity-header modes only.

## Solution Assessment

Unclear as an implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, device, revocation, and rate-limit rules before implementation.
