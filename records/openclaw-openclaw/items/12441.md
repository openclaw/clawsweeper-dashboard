## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads but does not bridge a proxy-injected Authorization or X-OpenClaw-Token header into browser WebSocket shared-secret authentication, and the existing discussion correctly treats this as a Gateway admin auth design decision.

## What This Changes

Request header-based Control UI authentication bootstrap so reverse proxies can inject a bearer token without putting the gateway token in the URL or requiring manual UI settings.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve the existing URL/manual/Tailscale/trusted-proxy flows, and cover the chosen model with focused Gateway, UI, and docs tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested feature: HTTP bearer auth can load Control UI read endpoints, but browser WebSocket shared auth is resolved only from connectParams.auth.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, device, and rate-limit rules before implementation.
