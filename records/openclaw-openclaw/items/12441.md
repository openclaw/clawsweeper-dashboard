## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads, but shared-secret Control UI WebSocket auth is resolved from browser connect-frame credentials only; adding proxy-injected header bootstrap changes the Gateway admin auth trust boundary.

## What This Changes

Allow Control UI to bootstrap shared-secret Gateway auth from proxy-provided Authorization or X-OpenClaw-Token headers instead of URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Gateway admin Control UI, preserve URL/manual/trusted-proxy flows, and cover it with Gateway, UI bootstrap, WebSocket auth, and docs tests.

## Reproduction Assessment

Yes from source inspection: configure token/password gateway auth behind a proxy that injects Authorization for the page/config request, leave browser token/password state empty, and the HTTP read path can use the header while WebSocket shared auth is resolved from connectParams.auth only.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose between validating the header on the WebSocket upgrade and issuing a short-lived same-origin browser credential from a bootstrap endpoint before code is queued.
