## Summary

Keep open. Current main accepts bearer credentials for authenticated Control UI HTTP reads, but it still does not bridge a proxy-injected Authorization or X-OpenClaw-Token header into the browser WebSocket shared-secret auth path; the prior issue comments are consistent with the current code and this remains a Gateway auth design item.

## What This Changes

The issue requests Control UI shared-secret auth bootstrap from reverse-proxy HTTP headers instead of URL fragments or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested feature: HTTP bearer auth can load Control UI read endpoints, but browser WebSocket shared auth is resolved from connectParams.auth and existing identity-header modes only.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, device, and rate-limit rules before implementation.
