## Summary

Keep open. Current main still supports bearer credentials for Control UI HTTP reads, but shared-secret Control UI WebSocket auth is still sourced from the browser connect frame or separate identity-header modes, so the requested reverse-proxy header bootstrap remains a Gateway/security design item.

## What This Changes

The issue requests Control UI bootstrap authentication from proxy-provided Authorization or X-OpenClaw-Token headers instead of URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen model with Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives high confidence that the requested feature is absent: HTTP reads can use bearer auth, but browser WebSocket shared auth is resolved from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, device, revocation, and rate-limit rules before implementation.
