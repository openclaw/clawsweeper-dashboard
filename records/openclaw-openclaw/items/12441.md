## Label Justifications

- `P2`: This is a normal-priority Gateway auth improvement with meaningful but bounded Control UI impact, not an urgent regression or outage.
- `impact:security`: The requested behavior changes how a shared Gateway secret crosses a reverse proxy into a browser-admin session.
- `impact:auth-provider`: The missing behavior is credential routing between Gateway HTTP authorization and the Control UI WebSocket authentication path.

## Summary

Keep open. Current main and v2026.6.6 still separate bearer-authenticated Control UI HTTP reads from WebSocket shared-secret auth, so the requested reverse-proxy header bootstrap is not implemented and still needs a Gateway auth/security product decision.

## What This Changes

Request to let Control UI bootstrap shared-secret gateway auth from reverse-proxy-provided `Authorization: Bearer ...` or `X-OpenClaw-Token` headers instead of URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for Gateway auth, either a trusted WebSocket-upgrade header path or a same-origin bootstrap endpoint that issues a short-lived browser session credential, while keeping current URL-fragment/manual token and identity-bearing proxy paths compatible.

## Reproduction Assessment

Yes at source level, though this is a feature request rather than a broken existing contract. Current main accepts bearer auth for Control UI HTTP reads but does not carry a proxy-injected header into the WebSocket shared-secret connect path.

## Solution Assessment

Unclear until maintainers choose the security model. A direct upgrade-header path is plausible, but a bootstrap endpoint with a short-lived browser credential may be safer than treating every trusted-proxy bearer header as a Gateway token.
