## Label Justifications

- `P2`: The request is a normal-priority Gateway/Control UI auth improvement with limited blast radius but important security design implications.
- `impact:security`: The issue changes how proxy-provided headers can authenticate an admin Control UI surface.
- `impact:auth-provider`: The issue concerns token/auth routing between HTTP bootstrap, WebSocket connect auth, and trusted-proxy identity modes.

## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads, but shared-secret WebSocket auth still comes only from the browser connect frame or the separate trusted-proxy identity mode, so the requested reverse-proxy token-header bootstrap remains a security-sensitive feature gap.

## What This Changes

Request to let Control UI bootstrap or WebSocket authentication consume a reverse-proxy-provided gateway token from Authorization or X-OpenClaw-Token headers instead of relying on URL or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen model with Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes as a source-level feature gap. Current main clearly shows HTTP bearer reads are supported while shared-secret WebSocket auth still comes only from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, revocation, device, and rate-limit rules before implementation.
