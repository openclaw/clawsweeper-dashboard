## Summary

Keep open. Current main still permits bearer authentication for protected Control UI HTTP reads, but it neither carries a proxy-injected shared secret into the browser nor returns a short-lived credential for the WebSocket connect flow; this remains a security-sensitive product-design decision rather than an implemented capability.

Likely related people: @steipete introduced and substantially maintained the bootstrap/auth surface (high confidence); @vincentkoc recently maintained the consolidated current-main Gateway and UI auth paths (medium confidence).

## What This Changes

Requests reverse-proxy header bootstrap for Control UI gateway authentication without placing the gateway token in the browser URL.

## Best Possible Solution

Add one explicit same-origin proxy bootstrap contract, preferably exchanging an already-authorized HTTP request for a short-lived, audience-bound WebSocket credential while preserving fragment/manual token and trusted-proxy identity paths.

## Reproduction Assessment

Yes, at source level with high confidence: an injected Authorization header can authorize an HTTP bootstrap read, but the response carries no credential and the browser WebSocket authenticates from connect-frame state. A real reverse-proxy setup would be useful implementation proof but is not needed to establish the current gap.

## Solution Assessment

No single suggested implementation is yet proven best. A short-lived same-origin credential exchange is the narrowest security-preserving direction; direct upgrade-header authentication would require a separately approved trusted-proxy contract.
