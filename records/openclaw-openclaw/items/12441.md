## Label Justifications

## Summary

Current main supports bearer authentication for protected Control UI HTTP reads but does not carry a proxy-injected shared secret into the browser WebSocket handshake. The request remains useful and vision-aligned, but it needs an explicit security/product decision because credential exchange, proxy trust, origin binding, expiry, replay protection, and rate limiting are permanent Gateway authentication contracts.

## What This Changes

Allow Control UI to bootstrap Gateway authentication from a reverse-proxy-injected Authorization or dedicated token header without placing the long-lived token in the URL.

## Best Possible Solution

Add one canonical same-origin bootstrap exchange that validates the proxy-provided header through existing Gateway auth and returns an origin-bound, short-lived browser credential for the WebSocket connect frame, while preserving current manual/fragment credentials and trusted-proxy identity behavior.

## Reproduction Assessment

Yes, at source level. A bearer header can authorize protected Control UI HTTP reads, but the browser WebSocket handshake still obtains credentials from connect-frame state and the bootstrap response contains no proxy-derived session credential.

## Solution Assessment

No single implementation proposed in the issue is already the proven best fix. Reusing the canonical HTTP authorization path to mint a narrowly scoped, short-lived WebSocket credential is safer and more maintainable than placing the long-lived token in browser state or creating a parallel auth mode.

## Risks / Open Questions

- A direct header-to-WebSocket shared-secret bridge could expose the long-lived Gateway credential to browser state or replay unless the design defines trusted proxy boundaries, same-origin enforcement, expiry, one-time or replay behavior, and shared rate limiting.
- Accepting credentials on the WebSocket upgrade may behave differently across reverse proxies and browser deployments; the chosen contract needs live nginx or Caddy proof in addition to focused tests.
- The existing trusted-proxy identity mode is a separate authorization model, so a new shared-secret bootstrap must not silently blur its identity, scope, or fallback semantics.
