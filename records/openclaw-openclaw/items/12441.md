## Summary

Keep open. Current main accepts bearer tokens for Control UI HTTP reads, but the shared-secret WebSocket path still authenticates from the browser connect frame only; the requested reverse-proxy header bootstrap is a security-sensitive Gateway auth feature that needs maintainer/secops design.

## What This Changes

Request Control UI support for reverse-proxy-injected Authorization or X-OpenClaw-Token headers as a gateway-token bootstrap source instead of token-in-URL or manual entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Gateway admin Control UI, preserve existing URL/manual/trusted-proxy flows, and cover the chosen path with Gateway, UI bootstrap, WebSocket auth, and docs updates.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure token/password gateway auth behind a proxy that injects Authorization for the page/config request, leave browser token/password state empty, and the HTTP read can use the header while WebSocket shared auth is resolved from connectParams.auth only.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an implementation plan. The requested direction is valid, but maintainers should choose between WebSocket-upgrade header validation and a same-origin bootstrap endpoint that issues a short-lived browser credential before code is queued.
