- `impact:security`: The requested behavior changes how a shared Gateway secret crosses a reverse proxy into a browser admin session.
- `impact:auth-provider`: The issue concerns credential routing between Gateway HTTP bearer auth and the Control UI WebSocket authentication path.

## Summary

Keep open. Current main and v2026.6.8 still accept bearer auth for Control UI HTTP reads, but the browser WebSocket shared-secret path is still driven by connect-frame credentials and does not bridge a proxy-injected Authorization or X-OpenClaw-Token header into the Control UI session.

## What This Changes

Allow Control UI reverse-proxy deployments to bootstrap shared gateway auth from an Authorization bearer or X-OpenClaw-Token header instead of relying on URL/manual token state.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for Gateway auth, either validating a trusted upgrade header after the existing origin/proxy/rate-limit checks or issuing a short-lived same-origin browser session credential, while preserving current URL-fragment/manual token and identity-bearing proxy flows.

## Reproduction Assessment

Yes, at source level. Current main accepts bearer auth for Control UI HTTP reads, but the WebSocket shared-secret path only consumes connect-frame auth and has no bootstrap session bridge for proxy-injected headers.

## Solution Assessment

Unclear until maintainers choose the trust model. Header bootstrap is plausible, but the safest permanent shape may be a purpose-specific trusted header or short-lived same-origin session credential rather than treating every forwarded Authorization bearer as a Gateway token.
