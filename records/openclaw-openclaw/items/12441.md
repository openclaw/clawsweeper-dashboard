## Summary

Keep open. Current main and v2026.6.6 still split Control UI auth: HTTP bootstrap/read requests can use bearer auth, but browser WebSocket shared-secret auth still comes only from the connect-frame token/password path, so the requested reverse-proxy header bootstrap remains unresolved and security-sensitive.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided Authorization or token headers instead of relying on URL/query/hash or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, either validating a purpose-built header during the WebSocket upgrade after proxy/origin/rate-limit checks or issuing a short-lived same-origin browser session credential from an authenticated bootstrap endpoint, with docs and focused Gateway/UI tests.

## Reproduction Assessment

Yes, at source level with high confidence. Current main accepts bearer auth for Control UI HTTP reads but does not bridge a reverse-proxy-provided Authorization or X-OpenClaw-Token header into the browser WebSocket connect-frame auth path.

## Solution Assessment

Unclear until maintainers choose the security model. A narrow, explicit Gateway auth design is better than reviving the closed one-file upgrade-header PR without settling proxy source, origin, rate-limit, device identity, and session lifetime semantics.
