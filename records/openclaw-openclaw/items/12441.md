## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads and separate trusted-proxy identity auth, but it does not bridge a proxy-injected shared-secret header into the browser WebSocket auth path; the prior maintainer and ClawSweeper comments match the current source.

## What This Changes

Request Control UI shared-secret authentication bootstrap from reverse-proxy HTTP headers such as Authorization or X-OpenClaw-Token instead of requiring URL/manual token state for WebSocket connect auth.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen model with focused Gateway, UI, and docs tests.

## Reproduction Assessment

Yes, from source inspection. A proxy can authenticate the HTTP config request with Authorization, but a browser with empty token/password state still opens a WebSocket whose shared auth is resolved only from connectParams.auth.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, device, and rate-limit rules before implementation.
