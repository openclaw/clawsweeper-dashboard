## Summary

Keep open. Current main still supports bearer headers for Control UI HTTP reads and a separate trusted-proxy identity mode, but it does not bridge a proxy-injected shared-secret header into browser WebSocket auth; the remaining work is a Gateway/Control UI auth trust-boundary design item.

## What This Changes

Request Control UI to bootstrap gateway shared-secret auth from proxy-provided `Authorization` or `X-OpenClaw-Token` headers instead of only URL/UI-supplied WebSocket credentials.

## Best Possible Solution

Define one supported reverse-proxy shared-secret bootstrap flow in Gateway auth, preserve existing URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, and docs tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested feature: HTTP bearer auth can load the config endpoint, but browser WebSocket shared auth is resolved only from `connectParams.auth`.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose the upgrade-header versus short-lived-session model and the required proxy, origin, device, and rate-limit rules before implementation.
