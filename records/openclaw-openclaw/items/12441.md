- `impact:auth-provider`: The affected surface is Gateway auth routing between HTTP bearer auth, WebSocket connect auth, trusted-proxy identity, Tailscale identity, and browser token state.
- `maturity:stable`: The issue affects `gateway-runtime`, which is scored M4 Stable and includes hosted web, device auth, and WebSocket connection categories.

## Summary

Keep open. Current main and v2026.6.10 still split bearer auth for Control UI HTTP reads from WebSocket shared-secret auth, so proxy-injected header bootstrap remains a security-sensitive Gateway feature decision rather than implemented cleanup.

## What This Changes

Request Control UI bootstrap authentication from proxy-injected Authorization or OpenClaw token headers instead of requiring URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model, preferably either a trusted same-origin bootstrap endpoint that issues a short-lived browser session credential or a tightly scoped opt-in header path for the WebSocket upgrade.

## Reproduction Assessment

Yes, at source level. Current main accepts bearer auth for Control UI HTTP reads, but WebSocket shared-secret auth consumes only connect-frame auth and has no header-to-session bridge for proxy-injected credentials.

## Solution Assessment

Unclear until maintainers choose the trust model. Direct header bootstrap is plausible, but a short-lived same-origin session credential or purpose-specific opt-in header may be safer than treating forwarded `Authorization` as a Gateway token.
