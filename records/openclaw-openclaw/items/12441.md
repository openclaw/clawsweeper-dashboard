## Summary

Keep open. Current main supports bearer auth for Control UI HTTP reads, but the shared-secret WebSocket path still authenticates from the client connect frame only; accepting proxy-injected headers is a Gateway admin auth design change that needs maintainer/secops choice.

## What This Changes

Request to let Control UI bootstrap shared-secret gateway auth from proxy-supplied Authorization or X-OpenClaw-Token headers instead of requiring URL/manual UI token state.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Gateway admin Control UI, preserve existing URL/manual/trusted-proxy flows, and cover the chosen path with Gateway, UI bootstrap, WebSocket auth, and docs updates.

## Reproduction Assessment

Yes. Configure token/password gateway auth behind a proxy that injects Authorization for the page/config request, leave browser token/password state empty, and current source shows the HTTP read can use the header while WebSocket shared auth is resolved from connectParams.auth only.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers should choose between WebSocket-upgrade header validation and a same-origin bootstrap endpoint that issues a short-lived browser credential before code is queued.
