- `impact:security`: The request changes how a shared Gateway secret crosses a reverse proxy into a browser admin session.

## Summary

Keep open. Current main and v2026.6.10 still accept bearer auth for Control UI HTTP reads, but WebSocket shared-secret auth is still driven by `connect.params.auth` or identity-bearing proxy modes, with no header-to-session bridge for a proxy-injected shared secret.

## What This Changes

Request Control UI support for bootstrapping gateway auth from reverse-proxy-provided `Authorization` or `X-OpenClaw-Token` headers instead of URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth: either validate a trusted header during the WebSocket upgrade after the existing origin/proxy/rate-limit checks, or validate it on a same-origin bootstrap endpoint and issue a short-lived browser session credential.

## Reproduction Assessment

Yes, at source level. Current main accepts bearer auth for Control UI HTTP reads, but the WebSocket shared-secret path consumes only connect-frame auth and has no bootstrap session bridge for proxy-injected headers.

## Solution Assessment

Unclear until maintainers choose the trust model. Header bootstrap is plausible, but a short-lived same-origin session credential or purpose-specific trusted header may be safer than directly treating forwarded `Authorization` bearer tokens as Gateway tokens.
