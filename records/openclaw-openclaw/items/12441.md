## Summary

Keep open. Current main still supports bearer-header auth for Control UI HTTP reads, but it does not bridge a proxy-injected Authorization or X-OpenClaw-Token header into the browser WebSocket shared-secret connect path. The existing issue discussion correctly identifies this as a valid but security-sensitive Gateway admin auth design item.

## What This Changes

Allow the Control UI to bootstrap gateway shared-secret authentication from proxy-injected HTTP headers instead of requiring URL tokens or manual UI token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Gateway admin Control UI: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after existing origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve URL-fragment/manual token flows and the existing trusted-proxy identity mode, then document and test the chosen path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure token/password gateway auth behind a proxy that injects Authorization for the page/config request, leave browser token/password state empty, and observe that the WebSocket shared auth is resolved from connectParams.auth rather than the injected header.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but maintainers/secops need to choose the trust-boundary model before a code change is the best solution.
