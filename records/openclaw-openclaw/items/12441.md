## Summary

Keep open. Current main supports bearer auth for Control UI HTTP bootstrap reads, but it still does not turn a proxy-injected Authorization or X-OpenClaw-Token header into the browser WebSocket shared-secret auth path, so the requested reverse-proxy bootstrap feature remains unimplemented and needs Gateway auth design review.

## What This Changes

The issue requests Control UI bootstrap authentication from reverse-proxy-provided Authorization or X-OpenClaw-Token headers instead of URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested feature: HTTP reads can use bearer auth, but browser WebSocket shared auth is still resolved from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as an implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, device, revocation, and rate-limit rules before implementation.
