## Summary

Keep open. Current main and v2026.5.20 still support bearer auth for Control UI HTTP reads, but browser WebSocket shared-secret auth still comes from the connect frame or identity-header modes; the requested proxy-header bootstrap remains a security-sensitive feature decision.

## What This Changes

The issue requests Control UI authentication bootstrap from proxy-provided Authorization or OpenClaw token headers instead of requiring URL/manual token state for browser WebSocket auth.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model, preserve URL/manual/Tailscale/trusted-proxy flows, and cover the chosen Gateway/UI/docs/security path with focused tests.

## Reproduction Assessment

Yes, source-level. Current main clearly shows HTTP bearer reads are supported while shared-secret WebSocket auth still comes only from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as a final implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, revocation, device, and rate-limit rules before implementation.
