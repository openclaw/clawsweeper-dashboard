## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads and separate trusted-proxy identity auth, but it does not bridge a proxy-injected Authorization or X-OpenClaw-Token shared-secret header into the browser WebSocket connect path, so this remains a Gateway auth design item.

## What This Changes

Request Control UI reverse-proxy bootstrap support that accepts a proxy-injected shared-secret header without relying on URL tokens or manual UI entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, preserve URL/manual/Tailscale/trusted-proxy flows, and cover it with focused Gateway, UI, docs, and security tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives high confidence that the requested feature is missing: HTTP reads can use bearer auth, but browser WebSocket shared auth is still resolved from connect-frame credentials or identity-header modes.

## Solution Assessment

Unclear as an implementation plan. The request is valid, but maintainers should choose the WebSocket-upgrade-header versus short-lived-session model and the required proxy, origin, device, revocation, and rate-limit rules before implementation.
