## Label Justifications

- `impact:security`: The requested behavior changes how a shared Gateway secret crosses a reverse proxy into a browser admin session.
- `impact:auth-provider`: The issue concerns credential routing between Gateway HTTP bearer auth and the Control UI WebSocket authentication path.

## Summary

Keep open. Current main and the latest release still support bearer tokens for Control UI HTTP reads, but the browser WebSocket shared-secret path remains driven by connect-frame auth and does not bridge reverse-proxy-provided Authorization or X-OpenClaw-Token headers into the Control UI session.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided Authorization or token headers instead of relying on URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth, either validating a trusted upgrade header after existing origin, proxy, and rate-limit checks or issuing a short-lived same-origin browser session credential for the WebSocket, while preserving current URL-fragment/manual token and identity-bearing proxy flows.

## Reproduction Assessment

Yes at source level, though this is a feature request rather than broken existing behavior. Current main accepts bearer auth for Control UI HTTP reads but does not carry proxy-injected headers into WebSocket shared-secret auth.

## Solution Assessment

Unclear until maintainers choose the trust model. Header bootstrap is directionally plausible, but the safest permanent shape may be a short-lived same-origin session credential or a purpose-specific trusted-header path rather than blindly treating every Authorization bearer value as a Gateway token.
