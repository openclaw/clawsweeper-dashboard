## Summary

Keep open: the Android fix direction is useful, but the current PR is not merge-ready because it reopens non-loopback cleartext gateway parsing, can lose the one pairing retry if lifecycle cancellation happens during the delay, and lacks inspectable after-fix proof.

## What This Changes

The branch adds Android transport URL parsing/use, changes pairing-required auto-retry to a single-edge retry, adds focused tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves loopback-only cleartext policy, keeps transport/display semantics only where connection behavior requires them, and schedules one pairing retry without losing the pending edge, with tests and redacted live Android proof.

## Reproduction Assessment

Yes for the pairing churn: current main loops every six seconds while pairingRequired remains true on a started lifecycle. The original HTTP-vs-WebSocket transport failure is supported by the PR discussion, but I did not establish a separate current-main runtime failure because the dial path already builds ws/wss from parsed host, port, and TLS state.

## Solution Assessment

No, not as currently proposed. The patch should retain current main's loopback-only cleartext policy and consume the pairing retry edge only after the delayed retry actually runs.
