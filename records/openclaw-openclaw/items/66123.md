## Summary

Keep open: current main still has source-reproducible Android pairing retry churn, but this PR is not mergeable as-is because it conflicts with current cleartext gateway policy, can lose the pending retry across lifecycle cancellation, and lacks real behavior proof.

## What This Changes

The PR adds an Android transport URL, switches UI/onboarding endpoint handling to it, changes pairing auto-retry to a single edge-triggered retry, adds focused tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves loopback-only cleartext policy, keeps transport/display semantics only where connection behavior requires them, schedules one pairing retry without losing pending edges, and includes focused tests plus redacted live Android evidence.

## Reproduction Assessment

Yes for the pairing churn by source inspection: current main's PairingAutoRetryEffect loops every six seconds while pairingRequired stays true on a started lifecycle. The transport claim has contributor live-testing context but no inspectable runtime artifact, and the current dial path already builds ws/wss from parsed host, port, and TLS state.

## Solution Assessment

No, not as currently proposed. The patch should retain current main's loopback-only cleartext policy and update the pairing retry edge only after the delayed retry actually runs.
