## Summary

Keep this PR open. Current main still has a reproducible Android pairing retry loop, but the PR head is dirty against main and has blocking retry lifecycle behavior; it also needs a rebase that preserves the loopback-only cleartext gateway policy added after the branch diverged.

## What This Changes

The PR adds Android gateway endpoint `transportUrl` handling, switches selected onboarding/connect UI flows to ws/wss URLs, changes pairing auto-retry toward one retry per pairing-required edge, adds resolver/retry tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves current loopback-only cleartext security, keeps transport and display semantics explicit only where they affect the actual connection path, consumes a pairing-required retry edge only after `onRetry()` runs or keeps it pending across lifecycle cancellation, and adds focused lifecycle coverage plus the required changelog entry.

## Reproduction Assessment

Yes for the pairing churn: current main loops every six seconds while `pairingRequired` remains true on a started lifecycle. The claimed current-main HTTP-vs-WebSocket transport failure is less clear from source because the actual dial path already builds `ws`/`wss` from host, port, and TLS state rather than using `displayUrl` directly.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the patch should be rebased onto current main, preserve the Android cleartext security hardening, and keep the retry edge pending until the scheduled retry actually executes.
