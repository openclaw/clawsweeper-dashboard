## Summary

Keep open. The PR targets a real Android pairing problem still present on current main, but it is not mergeable as-is because the branch conflicts with current loopback-only cleartext policy, can still lose the one-shot pairing retry edge, and lacks inspectable after-fix Android runtime proof.

## What This Changes

The PR adds Android gateway transport/display URL separation, switches onboarding/connect surfaces toward transport URLs, limits pairing auto-retries, adds focused Android tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves loopback-only cleartext policy, keeps transport semantics only where connection behavior requires them, schedules one pairing retry without losing pending edges, and includes focused tests plus redacted live Android evidence.

## Reproduction Assessment

Partially yes. Source inspection gives a high-confidence path for pairing churn because current main retries every six seconds while pairing remains required; the transport-route failure has contributor live-test context but no inspectable runtime artifact.

## Solution Assessment

No, not as currently proposed. The branch should be rebased to retain loopback-only cleartext parsing and should consume the pairing retry edge only after the delayed retry actually runs.
