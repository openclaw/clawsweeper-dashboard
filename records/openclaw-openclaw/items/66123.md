## Summary

Keep open: this is a plausible Android bug fix, but the current PR head is not safe to merge because it conflicts with current main's loopback-only cleartext gateway policy and can lose the one pending pairing retry across lifecycle cancellation. The PR also lacks concrete after-fix real behavior proof beyond a live-testing claim.

## What This Changes

The PR adds Android `transportUrl` parsing and use in onboarding/connect flows, changes pairing auto-retry from a loop to a one-edge retry, adds focused tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves loopback-only cleartext parsing, uses explicit transport semantics only where they affect connection behavior, and keeps one pairing retry pending until it actually runs, with lifecycle coverage and redacted live Android proof.

## Reproduction Assessment

Yes for the pairing churn: current main loops every six seconds while `pairingRequired` remains true on a started lifecycle. The original HTTP-vs-WebSocket transport failure is supported by the PR's live-testing notes, but it is less directly reproducible from current source because the dial URL is built from host, port, and TLS state.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the patch must retain current main's loopback-only cleartext security policy and avoid consuming the pairing retry edge before the delayed retry actually runs.
