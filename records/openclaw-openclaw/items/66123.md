## Summary

This PR should stay open but is not merge-ready: the Android gateway direction is useful, yet the current head lacks inspectable real-device proof, conflicts with current main, reopens broader cleartext gateway acceptance, and can still lose the one-shot pairing retry after lifecycle cancellation.

## What This Changes

The PR adds an Android gateway display/transport URL split, switches onboarding/connect UI paths toward WebSocket transport URLs, limits pairing auto-retries, adds focused Android tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves loopback-only cleartext and unsupported-scheme validation, keeps transport semantics only where connection behavior requires them, schedules one pairing retry without losing pending lifecycle edges, and includes focused tests plus redacted Android proof.

## Reproduction Assessment

Partially yes. Current main source gives a high-confidence reproduction path for pairing churn because it retries every six seconds while pairing remains required; the transport-route failure has contributor live-test context but no inspectable artifact.

## Solution Assessment

No, not as currently proposed. The branch should be rebased to retain current gateway security validation and should consume the pairing retry edge only after the delayed retry actually runs or otherwise preserve it across cancellation.
