## Summary

Keep this PR open. The Android pairing churn is still visible on current main, but the proposed branch should not merge as-is because it conflicts with the current loopback-only cleartext gateway policy and can lose the one-shot retry if lifecycle cancellation happens before the delayed retry fires.

## What This Changes

The PR adds Android gateway endpoint transport URL handling, switches selected onboarding/connect URL surfaces toward `ws`/`wss`, changes pairing auto-retry to a one-shot edge, adds resolver/retry tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that keeps cleartext Android gateway input loopback-only, preserves TLS and explicit ports through typed endpoint state instead of relying on reparsed presentation URLs, replaces the unbounded pairing loop with a lifecycle-safe one-shot retry edge, and updates the docs plus changelog for the shipped behavior.

## Reproduction Assessment

Yes for the pairing churn: current main's `PairingAutoRetryEffect` loops every six seconds while pairing remains required on a started lifecycle. The transport failure is source-verified rather than live-reproduced: current main's actual dial path already builds `ws`/`wss` from host, port, and TLS params instead of dialing `displayUrl` directly.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the patch must preserve the current fail-closed cleartext policy and consume retry state only after the scheduled retry actually executes.
