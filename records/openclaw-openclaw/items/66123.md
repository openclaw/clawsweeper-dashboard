## Summary

Keep this PR open, but it is not mergeable as-is: the branch still conflicts with current main's loopback-only cleartext Android gateway policy, can lose the intended one-shot pairing retry if the app backgrounds during the delay, and lacks inspectable after-fix Android runtime proof.

## What This Changes

The PR adds Android gateway transport/display URL separation, switches onboarding/connect endpoint handling toward transport URLs, limits pairing auto-retry churn, adds focused Android tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves loopback-only cleartext policy, keeps transport/display semantics only where connection behavior requires them, schedules one pairing retry without losing pending edges, and includes focused tests plus redacted live Android evidence.

## Reproduction Assessment

Yes for the pairing churn by source inspection: current main's PairingAutoRetryEffect loops every six seconds while pairingRequired stays true on a started lifecycle. The transport route problem has contributor live-test context but no inspectable runtime artifact in the PR.

## Solution Assessment

No, not as currently proposed. The patch should retain current main's loopback-only cleartext policy and consume the pairing retry edge only after the delayed retry actually runs.
