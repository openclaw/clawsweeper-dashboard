## Summary

Keep this PR open, but do not merge it as-is. The pairing churn problem is still real on current main, while the branch is dirty against current main and has blocking regressions: it would restore private-LAN cleartext gateway acceptance that current main intentionally removed, and its one-shot retry can be consumed before `onRetry()` if the lifecycle cancels during the delay. The original transport failure is less certain on current main because the live dial path already builds `ws`/`wss` from endpoint host, port, and TLS state.

## What This Changes

The PR adds Android gateway endpoint `transportUrl` handling, switches selected onboarding/connect endpoint displays and stored URLs to `ws`/`wss`, changes pairing auto-retry toward one retry per pairing-required edge, adds resolver/retry tests, and updates mobile pairing docs.

## Best Possible Solution

Land a rebased Android fix that preserves the current loopback-only cleartext gateway policy, keeps transport and display semantics explicit only where they affect the actual connection path, consumes a pairing-required retry edge only after `onRetry()` runs or keeps it pending across lifecycle cancellation, and adds focused lifecycle coverage plus the required changelog entry.

## Reproduction Assessment

Yes for the pairing churn: current main loops every six seconds while `pairingRequired` remains true on a started lifecycle. The claimed current-main HTTP-vs-WebSocket transport failure is less clear from source because the actual dial path already builds `ws`/`wss` from host, port, and TLS state rather than using `displayUrl` directly.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the patch must be rebased onto current main, preserve Android's loopback-only cleartext security hardening, and keep the retry edge pending until the scheduled retry actually executes.
