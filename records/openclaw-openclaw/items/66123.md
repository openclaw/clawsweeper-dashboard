## Summary

Keep this PR open, but do not merge it as-is. Current main still has the repeated Android pairing retry loop and does not contain the PR's separate `transportUrl` resolver contract, so the work is not obsolete; the PR also still has a concrete lifecycle cancellation bug in its proposed edge-triggered retry behavior.

## What This Changes

The PR adds Android `transportUrl` handling for gateway endpoints, switches setup/manual Android UI flows to use ws/wss transport URLs, changes pairing auto-retry toward one retry per pairing-required edge, adds resolver/retry unit coverage, and updates mobile pairing docs.

## Best Possible Solution

Land a revised Android fix that keeps explicit ws/wss transport semantics and display semantics separate, preserves the existing cleartext/TLS policy, and changes pairing retry state so a pending pairing-required edge is consumed only after the scheduled retry actually runs or remains pending across lifecycle cancellation. Include focused resolver tests, retry lifecycle coverage, and the matching docs clarification.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction for repeated pairing churn: when `pairingRequired` stays true and lifecycle is started, `PairingAutoRetryEffect` loops forever and calls `onRetry()` every six seconds. The PR-specific cancellation bug is also reproducible by letting pairing become required, backgrounding before the delayed retry fires, and resuming with `enabled` still true.

## Solution Assessment

No, not as currently proposed. The PR's direction is narrow and appropriate, but the retry implementation should keep the pending edge until `onRetry()` has actually executed so lifecycle cancellation does not silently drop the one intended retry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Consume the pairing edge only after retry runs:** `apps/android/app/src/main/java/ai/openclaw/app/ui/GatewayPairingRetry.kt:49`
  - body: The new retry state marks the pairing-required edge as seen before the delayed retry executes. If the app backgrounds during the six-second delay, Compose cancels the coroutine; on resume `enabled` is still true but the edge has already been consumed, so no retry is scheduled. Move the edge consumption until after `onRetry()` runs, or keep a pending edge across lifecycle cancellation.
  - confidence: 0.86
