## Label Justifications

- `P2`: This is a normal-priority cross-provider message delivery bug with a clear source path and limited blast radius.
- `impact:message-loss`: The reported failure prevents a cross-provider `message` tool send from reaching the target channel when the leaked account id is invalid there.
- `impact:auth-provider`: The failure is caused by forwarding a source channel account identity into a different provider's account resolution path.

## Summary

Keep open. Current main still resolves `accountId` from the explicit tool parameter or the source session `defaultAccountId` before sending, then passes that value into the selected target channel runtime; the linked fix PR remains open and unmerged, so the issue still needs the targeted message-runner fix or that PR landed.

## What This Changes

The issue asks OpenClaw core to avoid forwarding a source session `defaultAccountId` to a different target channel provider during cross-provider `message` tool sends.

## Best Possible Solution

Land a focused core change in `src/infra/outbound/message-action-runner.ts` that keeps explicit `accountId` values but omits the source session default when the resolved target channel provider differs, with regression coverage for cross-provider sends.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a bound source session with `defaultAccountId` and an allowed cross-provider target will select the source default and forward it to the target runtime. I did not run a live Telegram-to-weixin setup in this read-only review.

## Solution Assessment

Yes, the narrow best fix is in the core message action runner: preserve explicit user-supplied `accountId`, but do not reuse the source session default for a different target provider. Plugin-side defensive validation is useful hardening, but it does not address the core leak at the boundary.
