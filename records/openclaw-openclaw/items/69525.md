## Summary

Keep open. Current main still copies the source session default accountId into the message action params before the allowed cross-provider path returns, and the only matching fix is an open, conflicting PR rather than merged code.

## What This Changes

The issue reports that cross-provider message sends forward the source session default accountId to a different target channel plugin.

## Best Possible Solution

Land a focused current-main core fix in `runMessageAction` that preserves explicit accountIds and same-provider defaults while omitting only the implicit source default for allowed cross-provider sends.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction` with `defaultAccountId`, `tools.message.crossContext.allowAcrossProviders: true`, a source `currentChannelProvider`, and a different target channel, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
