## Summary

Keep open. Current main still resolves an implicit source-session `defaultAccountId` before cross-provider policy enforcement and forwards that account id into target plugin dispatch; the linked fix path remains the open, unmerged https://github.com/openclaw/openclaw/pull/69529.

## What This Changes

The issue reports that cross-provider `message` sends pass the source channel session account id to the target channel plugin instead of letting the target provider resolve its own account.

## Best Possible Solution

Land or replace the focused outbound runner fix so implicit source defaults are preserved for same-provider sends but omitted for allowed cross-provider sends, while explicit account ids continue to pass through unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction` with a source `defaultAccountId`, `allowAcrossProviders` enabled, and a different target channel provider, then assert the target plugin dispatch does not receive the implicit source account id.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
