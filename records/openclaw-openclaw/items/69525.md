## Summary

Keep open. Current main still propagates the source session default accountId into allowed cross-provider sends, and the open targeted fix PR remains unmerged/conflicting rather than shipped.

## What This Changes

The issue reports that cross-provider message sends inherit the source session default accountId and pass it to the target channel plugin.

## Best Possible Solution

Land or replace the linked PR with a focused runner fix that preserves explicit accountIds and same-provider defaults while omitting only implicit source defaults for allowed cross-provider sends.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call runMessageAction with defaultAccountId, allowAcrossProviders enabled, a source currentChannelProvider, and a different target channel, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side account validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
