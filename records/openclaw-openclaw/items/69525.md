## Summary

Keep open. Current main still derives the outbound `accountId` from `input.defaultAccountId` before the cross-provider policy check, the allowed cross-provider path does not clear it, and dispatch forwards it to the target plugin; the related fix PR at https://github.com/openclaw/openclaw/pull/69529 remains open, conflicting, and unmerged.

## What This Changes

The issue requests that cross-provider message sends stop passing an implicit source-session accountId to a different target channel plugin.

## Best Possible Solution

Land a focused core regression fix in `runMessageAction` that distinguishes explicit accountId from implicit source defaultAccountId and omits only the implicit source default for allowed cross-provider sends.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence focused reproduction path: call `runMessageAction` with `defaultAccountId`, `tools.message.crossContext.allowAcrossProviders: true`, a source `currentChannelProvider`, and a different target channel, then assert the target plugin receives no implicit source accountId; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side account validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
