## Summary

Keep open. Current main still derives `accountId` from `input.defaultAccountId` before cross-provider policy runs, the allowed cross-provider path does not clear it, and plugin dispatch forwards that value; the related fix PR at https://github.com/openclaw/openclaw/pull/69529 remains open and unmerged.

## What This Changes

The issue asks OpenClaw core to stop passing an implicit source-session accountId into a different target provider during allowed cross-provider message sends.

## Best Possible Solution

Land a focused core regression fix in `runMessageAction` that distinguishes explicit accountId from implicit source defaultAccountId and omits only the implicit source default for allowed cross-provider sends.

## Reproduction Assessment

Yes. A focused unit scenario can call `runMessageAction` with `defaultAccountId`, `tools.message.crossContext.allowAcrossProviders: true`, a source `currentChannelProvider`, and a different target channel, then assert the target plugin receives `accountId` as undefined unless the tool explicitly supplied it; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side account validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
