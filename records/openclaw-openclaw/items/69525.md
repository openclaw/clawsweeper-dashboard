## Summary

Keep open. Current main still resolves the outbound message accountId from the source session default before cross-provider policy runs, the allowed cross-provider path returns without clearing it, and plugin dispatch forwards that same accountId to the target channel handler. Related PR #69529 targets this bug but remains open and bundles additional unrelated changes.

## What This Changes

The issue asks OpenClaw core to stop forwarding an implicit source-channel defaultAccountId to a different target channel provider during allowed cross-provider message sends.

## Best Possible Solution

Land a narrow core fix in runMessageAction that preserves explicit accountId values, keeps same-provider default and agent-binding behavior, and suppresses only the implicit source defaultAccountId when allowAcrossProviders is enabled and the current provider differs from the resolved target channel. Add a focused regression around plugin dispatch so the target plugin receives undefined for the implicit cross-provider default.

## Reproduction Assessment

Yes. A focused unit scenario can call runMessageAction with defaultAccountId, tools.message.crossContext.allowAcrossProviders: true, a source currentChannelProvider, and a different target channel, then assert the target plugin handleAction context does not receive the source default.

## Solution Assessment

Yes. The narrow core fix is the best direction because plugin-side account validation is only defensive; the shared outbound runner should not inject a source-provider account id into an unrelated target-provider action.
