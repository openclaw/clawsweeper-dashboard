## Summary

Keep open. Current main still resolves an implicit outbound accountId from the source session default before cross-provider policy runs, and the allowed cross-provider path forwards that value to the target plugin unchanged; open PR #69529 targets the bug but is still unmerged and bundles unrelated fixes.

## What This Changes

The issue asks OpenClaw core to stop passing the source session's implicit accountId to a different target provider during cross-provider message sends.

## Best Possible Solution

Land a narrow core regression fix in runMessageAction that distinguishes explicit accountId from implicit defaultAccountId and omits only the implicit source default when the resolved target channel differs from currentChannelProvider under allowAcrossProviders.

## Reproduction Assessment

Yes. A focused unit scenario can call runMessageAction with defaultAccountId, tools.message.crossContext.allowAcrossProviders: true, a source currentChannelProvider, and a different target channel, then assert the target plugin handleAction context receives accountId undefined unless the tool explicitly supplied accountId.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side account validation is a useful defense but should not be required to compensate for core injecting a source-provider account id into a different provider.
