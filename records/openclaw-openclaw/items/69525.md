## Summary

Keep open. Current main still assigns the source session default accountId before cross-provider policy enforcement, and the allowed cross-provider path does not clear it before plugin dispatch; the only exact fix path is still the open, conflicting https://github.com/openclaw/openclaw/pull/69529.

## What This Changes

The issue reports that the message tool passes an implicit source-session defaultAccountId to a different target provider during allowed cross-provider sends, causing the target plugin to resolve a non-existent account.

## Best Possible Solution

Land a focused current-main runner fix that preserves explicit accountIds and same-provider defaults, uses the effective message cross-context policy, and omits only the implicit source default for allowed cross-provider sends.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call runMessageAction with defaultAccountId, allowAcrossProviders enabled, a source currentChannelProvider, and a different target channel, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
