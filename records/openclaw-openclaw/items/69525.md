## Summary

Keep open. Current main still resolves the outbound account id from the source session default before the cross-provider policy path and forwards it to the target channel plugin; the related fix PR is open and unmerged.

## What This Changes

Prevent the message tool from passing an implicit source-session accountId to a different target provider during allowed cross-provider sends.

## Best Possible Solution

Land a focused current-main core fix in `runMessageAction` that distinguishes explicit accountId from the implicit source default and omits only that implicit default for allowed cross-provider sends.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runMessageAction` with `defaultAccountId`, `tools.message.crossContext.allowAcrossProviders: true`, a source `currentChannelProvider`, and a different target channel, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

Yes. Fixing the shared outbound runner is the narrow maintainable solution; plugin-side validation is useful defense-in-depth but should not be required to compensate for core injecting a source-provider account id into another provider.
