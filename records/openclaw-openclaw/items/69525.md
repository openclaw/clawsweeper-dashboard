## Label Justifications

- `P2`: This is a valid cross-provider messaging bug with a bounded outbound-runner fix and limited blast radius.
- `impact:message-loss`: The reported leak causes allowed cross-provider sends to fail before delivery to the target channel.
- `impact:auth-provider`: The failure is caused by provider/account resolution receiving the wrong source session account id.

## Summary

Keep open. Current main and the latest release still select the source `defaultAccountId` before cross-provider enforcement and forward the selected account id to the target plugin context; the linked fix PR remains open and unmerged.

## What This Changes

The issue reports that cross-provider message sends can pass the source session default accountId into the target channel plugin, causing the target provider to resolve the wrong account.

## Best Possible Solution

Land or replace the linked focused outbound runner fix so implicit source defaults are omitted for allowed cross-provider sends, explicit account ids and same-provider defaults are preserved, and regression coverage proves target plugin context receives no source default.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction` with a source `defaultAccountId`, effective cross-provider policy enabled, and a different target provider, then assert the target plugin dispatch does not receive the implicit source account id.

## Solution Assessment

Yes for the core direction. The maintainable fix belongs in the shared outbound runner using the same effective cross-context policy as enforcement, not as the primary responsibility of every target plugin.
