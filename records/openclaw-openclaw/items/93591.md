## What This Changes

## Best Possible Solution

Fix the runner/watchdog phase contract so no-hook progress is distinguishable from an unhandled `before_agent_reply` hook, then prove the linked cron scenario with focused tests and redacted real cron output.

## Reproduction Assessment

Yes, source-reproducible: the PR head emits `before_agent_reply` and then a pre-execution phase, while the current watchdog re-arms on that exact transition. Existing regression coverage asserts that sequence times out at `runtime-plugins`.

## Solution Assessment

No. Moving `before_agent_reply` outside `hasHooks` is only a plausible fix; the safer fix must preserve the unhandled-hook re-arm behavior without making the no-hook path look unhandled.
