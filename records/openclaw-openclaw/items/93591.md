## Summary

Keep open: this PR is a focused attempt to fix the open cron watchdog bug, but the current branch is not merge-ready because it emits `before_agent_reply` and then immediately follows with a pre-execution phase that the watchdog already treats as an unhandled-hook re-arm. The PR also has test-only verification and still needs real cron behavior proof from a real setup.

## What This Changes

The branch changes cron-triggered embedded and CLI agent runners to emit `before_agent_reply` even when no `before_agent_reply` hook is registered, while keeping hook execution guarded by `hasHooks`.

## Best Possible Solution

Keep the linked issue open and replace this with a watchdog/phase-contract fix that proves the no-hook cron path no longer re-enters `waiting_for_execution` while preserving the prior unhandled-hook re-arm behavior.

## Reproduction Assessment

Yes, source-reproducible: the PR head emits `before_agent_reply` and then a pre-execution phase, while the current watchdog re-arms exactly on that transition. The existing `before_agent_reply` followed by `runtime_plugins` regression test demonstrates the resulting timeout path without running a live cron job in this read-only review.

## Solution Assessment

No. Moving the phase notification outside `hasHooks` is only a plausible fix; the best fix must update the runner/watchdog contract so no-hook progress is distinguishable from an unhandled `before_agent_reply` hook, with focused regression coverage and real cron proof.
