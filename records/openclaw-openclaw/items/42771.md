## Summary

Keep open. Current main still uses the hidden Playwright cache path, so the intended Docker behavior is not implemented, but this PR is not merge-ready because the new default path is not writable by the runtime user, the new Chromium symlink is wrong for linux-x64 Playwright installs, and the browser install step deletes a BuildKit apt cache mount.

## What This Changes

The PR sets Docker `PLAYWRIGHT_BROWSERS_PATH` to `/opt/playwright`, pre-creates that directory, installs optional Chromium there with a `/usr/local/bin/chromium` symlink, and updates Docker/browser docs to describe the new path.

## Best Possible Solution

Land a corrected Docker/browser fix that makes `/opt/playwright` the stable default, creates it with `node:node` ownership before runtime installs, recursively restores ownership after optional root-run preinstall, preserves BuildKit apt caches, creates an architecture-correct Chromium launcher only after the browser exists, updates all Docker/browser persistence docs, and adds focused Dockerfile assertions.

## Reproduction Assessment

Yes. Apply the PR head and build without `OPENCLAW_INSTALL_BROWSER`; `/opt/playwright` is created as root-owned while the image later runs as `node`, and Playwright writes registry metadata under `PLAYWRIGHT_BROWSERS_PATH`. For browser-preinstalled linux-x64 builds, Playwright 1.59.1 uses `chrome-linux64/chrome`, while the PR symlink targets `chrome-linux/chrome`.

## Solution Assessment

No. Moving the Docker browser path to `/opt/playwright` is a reasonable direction, but this exact patch is not the narrow maintainable fix until ownership, symlink target selection, cache preservation, and docs coverage are corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Create /opt/playwright with node ownership:** `Dockerfile:164-165`
  - body: The default build path creates `/opt/playwright` as `root:root` with mode `775`, and no later `chown` runs when `OPENCLAW_INSTALL_BROWSER` is unset. Since the image runs as `node`, documented runtime installs into the new default browser path can fail when Playwright writes its registry metadata there.
  - confidence: 0.94
- **[P1] Use Playwright's x64 executable path for the symlink:** `Dockerfile:176`
  - body: This symlink hardcodes `chrome-linux/chrome`, but Playwright 1.59.1 uses `chrome-linux64/chrome` on linux-x64 and `chrome-linux/chrome` only on linux-arm64. On the common x64 Docker image, `ln -s` can still succeed with a dangling source, leaving `chromium` in PATH but unusable.
  - confidence: 0.9
- **[P2] Stop deleting apt lists from the cache mount:** `Dockerfile:175`
  - body: This `RUN` mounts `/var/lib/apt` as a BuildKit cache, so deleting `/var/lib/apt/lists/*` clears the shared apt metadata cache for later builds without reducing the final image layer. Remove the cleanup from this cache-mounted step.
  - confidence: 0.88
