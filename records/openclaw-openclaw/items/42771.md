## Summary

Keep open: the Docker path change is still useful and current main has not implemented it, but this PR head still has source-level blockers around runtime ownership, BuildKit cache preservation, and the current browser-control docs surface.

## What This Changes

The PR changes Docker Playwright browser storage to `/opt/playwright`, adds a `/usr/local/bin/chromium` symlink, and updates Docker/browser persistence docs.

## Best Possible Solution

Land a corrected Docker/browser update that consistently uses `/opt/playwright`, keeps it writable for `node` in both build modes, preserves BuildKit apt caches, updates current docs, and adds focused Dockerfile assertions.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the default build creates `/opt/playwright` as root-owned before the final `USER node`, and the browser-enabled build deletes a cache-mounted apt metadata directory; I did not run Docker in this read-only review.

## Solution Assessment

No. Moving Docker browser storage to `/opt/playwright` is a maintainable direction, but this patch needs ownership, cache-preservation, current-docs, and focused test fixes before it is the narrow safe solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Assign `/opt/playwright` to node during setup:** `Dockerfile:164-165`
  - body: The default build path creates `$PLAYWRIGHT_BROWSERS_PATH` as root with mode `775`, and no later `chown` runs when `OPENCLAW_INSTALL_BROWSER` is unset. Since the image later runs as `node`, documented runtime installs into the new default path can fail with `EACCES`.
  - confidence: 0.94
- **[P2] Leave the BuildKit apt lists cache intact:** `Dockerfile:175`
  - body: This step mounts `/var/lib/apt` as a BuildKit cache, so deleting `/var/lib/apt/lists/*` clears cached apt metadata for later builds while not reducing the committed image layer. Remove the cleanup from this cache-mounted step.
  - confidence: 0.89
- **[P3] Update the current browser-control docs:** `docs/tools/browser.md:414-415`
  - body: The PR updates the old browser page on its branch, but current main's Docker Playwright instructions live in `docs/tools/browser-control.md` and still mention `/home/node/.cache/ms-playwright`. After rebase, users would get conflicting persistence paths unless that current page is updated too.
  - confidence: 0.78
