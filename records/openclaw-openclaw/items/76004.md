## Summary

Keep open. The optimization targets a real hot path, but the proposed cache key omits model-derived data even though the transport hook contract receives `model`, so a provider hook can receive stale params; the PR also lacks the required changelog coverage.

## What This Changes

The PR adds a WeakMap-backed per-config cache around `resolvePreparedExtraParams` in `src/agents/pi-embedded-runner/extra-params.ts`.

## Best Possible Solution

Keep the cache optimization, but key it by every hook input that can affect output, add a regression test for model-varying transport hooks, and add the required changelog entry before merge.

## Reproduction Assessment

Yes for the PR-introduced bug: a focused unit test can install a fake `extraParamsForTransport` hook that returns different patches for two `model.api` values under the same config/provider/modelId, then assert the second call is not served from the first cache entry.

## Solution Assessment

No as proposed. Per-config memoization is a reasonable direction, but the key must include stable model-derived fields such as API/baseUrl/compat when hook output can depend on `context.model`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Include model-derived inputs in the cache key:** `src/agents/pi-embedded-runner/extra-params.ts:159-169`

Please include stable model-derived fields that the hook contract exposes, or avoid caching calls where the model can vary.
