## Summary

Keep open. The extra-params memoization is a reasonable optimization, but the PR still replaces several request-time plugin loader paths with active-registry-only probes, so cold paths and custom-env/plugin-root paths can silently lose providers, tools, channel delivery, web providers, and middleware.

## What This Changes

This PR memoizes `resolvePreparedExtraParams` and rewires provider, tool, channel, capability, memory, migration, web-provider, and middleware helpers to reuse loaded runtime registries.

## Best Possible Solution

Keep the extra-param cache, but make registry reuse a loader-compatible fast path with env/config/workspace/runtimeOptions checks and preserve existing load fallbacks for each runtime surface.

## Reproduction Assessment

Yes. A source-level reproduction is to call `ensureRuntimePluginsLoaded`, `resolvePluginProviders`, `resolvePluginTools`, or `resolvePluginWebProviders` on the PR head without a compatible active registry and observe that the configured plugin registry is not loaded; a second reproduction is to call the new active-registry helper with an explicit env after loading a registry under a different env.

## Solution Assessment

No. Memoizing prepared extra params per stable config can be the narrow fix, but replacing loader calls with unchecked global active-registry reads is not the narrowest maintainable solution.
