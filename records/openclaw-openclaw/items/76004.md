## Summary

Keep open. The latest PR head fixes the earlier cache-key and changelog concerns, but it also broadens the change into runtime plugin registry loading and now drops cold/on-demand plugin loading paths while ignoring explicit env/plugin-root boundaries.

## What This Changes

The PR memoizes prepared extra params and rewires request-time provider, tool, channel, capability, memory, migration, and web-provider helpers to reuse the loaded runtime plugin registry, with tests and a changelog entry.

## Best Possible Solution

Keep the cache optimization, but make registry reuse a compatibility-checked fast path that falls back to the existing loaders and respects config, workspace, explicit env, and runtime options.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call ensureRuntimePluginsLoaded or resolvePluginProviders on the PR head with no active registry and observe that no runtime registry load occurs; a second one is to set an active registry under process env and call provider resolution with a distinct env.

## Solution Assessment

No as proposed. Per-config memoization is a reasonable narrow optimization, but replacing runtime loaders with active-registry reads changes cold-start, custom-env, and fallback behavior instead of preserving the old contract behind a fast path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve runtime plugin loading on registry misses:** `src/agents/runtime-plugins.ts:42-45`
  - body: `ensureRuntimePluginsLoaded` now only probes the active registry and ignores a miss. In cold embedded, dispatch-from-config, or subagent paths this no longer loads the configured runtime plugins, and the existing PR-head test still expects the loader call. Please keep the active-registry reuse as a fast path, then fall back to `resolveRuntimePluginRegistry` with the prior config/workspace/runtimeOptions when no compatible registry is loaded.
  - confidence: 0.93
- **[P2] Keep provider hook resolution from returning empty on cold paths:** `src/plugins/providers.runtime.ts:312-318`
  - body: Runtime provider lookup now calls `getLoadedRuntimePluginRegistry` and returns `[]` when there is no active registry. `resolvePreparedExtraParams` and other provider hook paths depend on this resolver to load the requested provider plugin, so non-startup or cache-miss flows silently lose provider hooks. Preserve the active-registry fast path but keep the existing runtime loader fallback.
  - confidence: 0.9
- **[P2] Respect explicit env before reusing active registries:** `src/plugins/active-runtime-registry.ts:78`
  - body: `getLoadedRuntimePluginRegistry` accepts `env` but discards it. Several callers document explicit env as selecting independent plugin roots, so a call with a custom `OPENCLAW_HOME` can reuse the process active registry loaded from a different root. Include env/root compatibility in the reuse check or decline reuse when `env` differs from `process.env`.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff touches plugin runtime code-execution paths and currently ignores explicit env/plugin-root boundaries when reusing an active registry.

Concerns:
