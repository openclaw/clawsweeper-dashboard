## Summary

Keep open. The latest head still expands the optimization into broad runtime plugin registry reuse, and several changed call sites now probe active registries without the current-main loader fallback or full compatibility checks.

## What This Changes

The PR memoizes prepared extra params and rewires provider, tool, channel, capability, memory, migration, and web-provider helpers to reuse startup or active plugin registries.

## Best Possible Solution

Keep the extra-param cache, but make registry reuse a compatibility-checked fast path that falls back to the existing loaders and preserves config, env, workspace, runtimeOptions, and channel-surface behavior.

## Reproduction Assessment

Yes. A source-level reproduction is to call ensureRuntimePluginsLoaded or resolvePluginProviders on the PR head without a compatible active registry and observe that no runtime registry load occurs; explicit-env reuse can be checked by loading an active registry under process env and resolving with a distinct env.

## Solution Assessment

No as proposed. Per-config memoization is a reasonable narrow optimization, but replacing loader calls with active-registry reads changes cold-start, custom-env, and fallback behavior instead of preserving the old contract behind a fast path.
