## Label Justifications

- `P1`: The issue reports a broken LINE channel workflow where uploaded audio files can be unreadable to the agent for real users.
- `impact:message-loss`: The reported attachment is either empty or unclassified and therefore missed by transcription, effectively losing the user’s audio message for the agent turn.

## Summary

Keep open: current main still drops LINE `FileMessage.fileName` before media storage, so uploaded audio files can remain unclassified when MIME sniffing is inconclusive. https://github.com/openclaw/openclaw/pull/96403 is the open candidate for the filename/audio-detection half, while the separate 0-byte LINE stream claim still needs live LINE/API evidence.

## What This Changes

The issue asks for LINE file-upload audio attachments to download non-empty, retain filename metadata, and enter the existing audio transcription pipeline like LINE voice messages.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/96403 for the filename handoff, then keep this issue open for live LINE evidence on the empty-stream behavior or split that into a narrower follow-up.

## Reproduction Assessment

No for the whole issue: source inspection proves the metadata/audio-detection miss on current main, but there is not a high-confidence live reproduction of the reported 0-byte LINE download stream. A live LINE upload or captured HTTP/status evidence is still needed for that half.

## Solution Assessment

Yes for the filename half: using LINE’s existing `fileName` and the existing `saveMediaStream` `originalFilename` seam is the narrowest maintainable fix. The empty-stream half should not change endpoints or stream handling until live LINE/API evidence shows the failure mode.
