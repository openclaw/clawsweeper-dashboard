## Label Justifications

- `P2`: LINE file-uploaded audio is channel-specific and blocks a normal inbound media workflow without making core runtime unusable.
- `impact:message-loss`: The issue reports audio file content being unavailable to the agent because it is saved empty or classified as a non-audio document.

## Summary

Keep open: current main downloads LINE `file` messages but still drops original file metadata before the shared audio classifier, and the reported 0-byte behavior conflicts with the official content endpoint contract enough that a maintainer or LINE live proof should confirm the correct retrieval path before repair.

## What This Changes

The issue requests support for LINE file-uploaded audio attachments so they download with content and route through audio transcription like LINE voice messages.

## Best Possible Solution

Keep the work in the LINE plugin: verify file-message content retrieval against a real LINE upload, then pass `FileMessage.fileName` or a trusted MIME hint into the existing media-store filename seam so uploaded audio is classified and transcribed.

## Reproduction Assessment

Yes at source level for the audio-detection half: current LINE code maps `file` messages to `<media:document>` and forwards only saved path/content type, while shared media selection transcribes only MIME/extension-classified audio. The 0-byte download portion is not fully reproduced because it needs a live LINE file upload or stronger API proof.

## Solution Assessment

No for the full suggested fix as written: filename-extension routing is a good narrow piece, but the download path should first be proven against LINE's current content endpoint behavior. A core media-understanding change would be the wrong layer because the media store already supports filename hints.
