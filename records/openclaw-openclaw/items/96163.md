## Summary

## What This Changes

The issue asks OpenClaw to download LINE file-upload audio correctly and route those uploaded files through the existing audio transcription pipeline.

## Best Possible Solution

Review and land a focused filename-metadata fix, then keep this issue open until live LINE evidence identifies whether a separate empty-stream download change is needed.

## Reproduction Assessment

Partially: source inspection reproduces the filename/audio-detection failure because current main drops LINE `FileMessage.fileName` before `saveMediaStream`. No high-confidence live LINE reproduction exists yet for the separate 0-byte content stream claim.

## Solution Assessment

Yes for the filename half: using LINE's existing `fileName` field and OpenClaw's existing `saveMediaStream` `originalFilename` contract is the narrowest maintainable fix. Endpoint or stream-handling changes should wait for live LINE/API evidence.
