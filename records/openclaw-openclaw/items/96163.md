## Label Justifications

- `P1`: The issue reports a broken LINE channel workflow where uploaded audio files can become unreadable to the agent for real users.
- `impact:message-loss`: The reported attachment becomes a 0-byte or unclassified media item and is missed by transcription, effectively losing the user's audio message for the agent turn.

## Summary

Keep open: current main still drops LINE `FileMessage.fileName` before media storage, so the audio-detection half remains unfixed until a PR lands; https://github.com/openclaw/openclaw/pull/96403 is the best current candidate, while the reported 0-byte stream still needs live LINE/API proof.

## What This Changes

The issue asks for LINE file-upload audio attachments to download with usable metadata and be routed through the existing audio transcription pipeline.

## Best Possible Solution

Land the focused filename handoff from https://github.com/openclaw/openclaw/pull/96403 if maintainers accept it, then keep this issue open for a live LINE file-upload check of the empty-stream behavior.

## Reproduction Assessment

No for the whole issue: source inspection proves the filename/audio-detection miss, but I do not have a high-confidence live reproduction of LINE returning a 0-byte stream for file uploads.

## Solution Assessment

Yes for the metadata half: forwarding LINE `fileName` through the existing `saveMediaStream` `originalFilename` seam is the narrowest owner-boundary fix. The empty-stream half should wait for live LINE/API evidence before changing endpoints or retry behavior.
