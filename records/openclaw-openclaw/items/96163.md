## Summary

Keep open: current main and the latest release still drop LINE `FileMessage.fileName` before media storage/classification, and the open related PRs only cover the filename/audio-detection half while the reported 0-byte LINE stream still needs live proof.

## What This Changes

The issue requests that LINE file-upload audio attachments download with usable content and be classified/transcribed like LINE `audio` voice messages.

## Best Possible Solution

Use the existing media-store `originalFilename` seam for LINE `FileMessage.fileName` preservation, land one canonical partial fix for filename-backed audio classification, and keep the 0-byte stream investigation tied to live LINE proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection on current main reproduces the missing filename/audio-classification path, but the separate 0-byte LINE `file` stream still needs a live LINE upload or captured API/status proof.

## Solution Assessment

Is this the best way to solve the issue? Yes for the audio-detection half: preserving `FileMessage.fileName` through the LINE plugin into the existing `saveMediaStream` `originalFilename` parameter is the narrowest maintainable fix; endpoint or stream changes should wait for live LINE evidence.
