## Label Justifications

- `P1`: The issue reports a broken LINE channel workflow where audio file uploads become unreadable to the agent for real users.
- `impact:message-loss`: The reported attachment arrives as a 0-byte or unclassified media item and is silently missed by transcription, effectively losing the user’s audio message for the agent turn.

## Summary

Keep open: current main downloads LINE `file` messages, but it still drops LINE `FileMessage` filename metadata before media storage/classification, so uploaded audio files that land as unknown or `.bin` attachments are not selected for transcription; no merged related item owns this exact remaining behavior.

## What This Changes

The issue requests that LINE file-upload audio attachments download correctly and enter the existing audio transcription pipeline like LINE voice-message audio.

## Best Possible Solution

Keep the fix in the LINE plugin: preserve LINE file metadata through download/context, use existing media-store filename hints for MIME/extension detection, and verify the empty-stream case with LINE API evidence before adding any alternate download path.

## Reproduction Assessment

No: there is not a high-confidence live reproduction of the 0-byte LINE download in this review. Source inspection does reproduce the metadata/audio-detection miss because current main discards LINE `fileName` before `saveMediaStream` and audio classification.

## Solution Assessment

Yes: a bounded LINE plugin fix is the best path, using the existing media-store filename hint contract instead of adding config or broad media-core fallback behavior. The download-empty portion should be investigated with HTTP/status evidence before changing endpoints.
