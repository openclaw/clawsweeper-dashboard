## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route structured file paths through the media loader:** `extensions/mattermost/src/mattermost/send.ts:552-557`
  - body: The `att.filePath` branch reads `opts.mediaReadFile` directly and uploads those bytes. That skips the shared `loadOutboundMediaFromUrl` path that applies `mediaLocalRoots`, size limits, MIME/host-read checks, and sandbox/local path policy, so structured attachment paths can bypass the safety boundary.
  - confidence: 0.91
- **[P2] Fail local filePath sends instead of falling back to text success:** `extensions/mattermost/src/channel.ts:279-282`
  - body: Mapping `params.path` and `params.filePath` into `mediaUrl` sends them through the existing mediaUrl fallback. If the local media load fails while the message has text, `send.ts` keeps posting text with no `file_ids`, so the reported false-success behavior remains for failed local file reads.
  - confidence: 0.86
- **[P2] Handle every advertised structured attachment alias:** `extensions/mattermost/src/channel.ts:299-318`
  - body: The shared send schema accepts `attachments[].mediaUrl`, `fileUrl`, `url`, and `name`, but this mapper only forwards `media`, `path`/`filePath`, `buffer`, and `filename`. When multiple attachment sources are present, those advertised aliases can be omitted from `opts.attachments` and silently dropped.
  - confidence: 0.82
