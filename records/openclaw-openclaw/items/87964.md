## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route attachment file paths through the media loader:** `extensions/mattermost/src/mattermost/send.ts:556`
  - body: This branch calls `mediaReadFile(att.filePath)` directly for structured attachments. That skips the shared loader's localRoots, size, MIME/sniffing, and host-read checks, so a Mattermost send can cross the local media boundary differently from the existing media path; load the path through `loadOutboundMediaFromUrl` with the existing media options instead.
  - confidence: 0.9
- **[P2] Handle all advertised structured attachment aliases:** `extensions/mattermost/src/channel.ts:299-314`
  - body: The message tool schema accepts structured `mediaUrl`, `fileUrl`, `url`, and `name`, but the new adapter only reads `media`, `path`/`filePath`, `buffer`, and `filename`. Those advertised attachment entries can still be ignored while the text post returns success, so normalize them or fail loudly.
  - confidence: 0.84
- **[P2] Fail filePath sends instead of falling back to text-only success:** `extensions/mattermost/src/channel.ts:279-282`
  - body: Mapping top-level `path`/`filePath` into `mediaUrl` means a local upload failure still uses the existing mediaUrl catch path and can send the text with ok:true and no file. Since these aliases are supposed to fix false-success attachment sends, track local path aliases separately or surface the upload error instead of falling back.
  - confidence: 0.78
