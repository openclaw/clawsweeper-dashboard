## Summary

Keep open. Current main still has the reported no-progress overflow recovery shape: aggregate prompt pressure can route to `compact_only`, and the outer embedded runner retries after `compacted: true` without checking that the next selected prompt fits or materially shrank.

## What This Changes

The issue asks OpenClaw to stop embedded/Paperclip agent auto-compaction from retrying identical oversized prompts and instead shrink/summarize/truncate history or fail once with an actionable overflow error.

## Best Possible Solution

Add an overflow-recovery progress gate: after compaction, require the selected prompt estimate to fit the same budget or require a material token shrink before retrying. If there is no progress, stop the loop and return an actionable context-overflow error; keep transcript rotation as an opt-in mitigation rather than changing broad retention defaults in this narrow fix.

## Reproduction Assessment

Yes. The issue provides concrete repeated precheck diagnostics, and the existing overflow-loop harness can model a `compact_only` precheck where compaction reports success but the next attempt still overflows.

## Solution Assessment

No. Current main is not the best solution because it treats `compacted: true` as enough progress; the safer narrow repair is to gate retries on post-compaction budget or shrink evidence.
