## Summary

Keep open. Current main still has the no-progress overflow retry shape: aggregate transcript overflow routes to compaction, a `compacted` result is treated as enough to retry, and the existing loop test still expects three compaction retries before the final context-overflow failure.

## What This Changes

The issue reports embedded/Paperclip agent auto-compaction retrying oversized context-overflow prompts without reducing the selected prompt window.

## Best Possible Solution

Add a focused overflow-recovery progress gate that stops retrying after compaction unless the retry prompt is below budget or has materially shrunk, keeping broader transcript-retention policy separate.

## Reproduction Assessment

Yes from source and test inspection. The issue has concrete repeated diagnostics, and current main's overflow-loop harness can model repeated over-budget precheck failures after `compact_only` compaction without the private session file.

## Solution Assessment

Yes. The narrow maintainable repair is a post-compaction budget or shrink check with a clear no-progress overflow error, not a new retention policy or configuration knob.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75740. Current main retries overflow recovery whenever `contextEngine.compact()` returns `compacted`, even when the next precheck would be the same oversized `compact_only` prompt. Add a focused progress gate in the embedded runner overflow-recovery path so a compaction retry proceeds only when the post-compaction prompt is under budget or has materially shrunk; otherwise return one clear context-overflow/no-progress error instead of retrying identical oversized prompts. Prefer using reliable `tokensAfter` and/or a fresh precheck estimate from the active or rotated transcript; handle engines that omit `tokensAfter` conservatively. Preserve timeout compaction and tool-result truncation behavior, do not add a config option, and do not broaden session retention/rotation policy. Add regression coverage for repeated `compact_only` over-budget precheck failures with no reducible tool results, including absent or non-improving `tokensAfter`. Add a user-facing CHANGELOG.md fix entry if the code change ships.
