## Summary

Keep open. Current main still has a plausible no-progress overflow-recovery path: the precheck can route aggregate history pressure to `compact_only`, and the outer loop treats any `compacted: true` result as enough to retry without proving the next selected prompt is smaller or within budget.

## What This Changes

Report that a long-running embedded/Paperclip agent session retried auto-compaction three times after precheck overflow while the estimated prompt stayed at about 873k tokens against a roughly 242k-token budget.

## Best Possible Solution

Add an overflow-recovery progress check: after compaction, prove the next selected prompt fits or materially shrank, or stop retrying and surface a clear actionable overflow error. Keep transcript rotation as a configurable mitigation rather than changing broad default retention policy in this narrow fix.

## Reproduction Assessment

Yes. The reporter provided repeated precheck diagnostics with identical token estimates, and the existing overflow-loop unit harness already models repeated compaction retries before failure; a targeted regression can mock `compact_only` plus no post-compaction budget progress.

## Solution Assessment

No. The current retry-after-`compacted` path is not the best solution because it assumes progress; the safer narrow fix is to carry precheck metrics or re-estimate after compaction and gate retries on budget/progress evidence.
