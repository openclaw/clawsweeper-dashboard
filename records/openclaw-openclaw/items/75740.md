## Summary

Keep open. Current main still has a plausible embedded-runner recovery gap for aggregate transcript growth: the precheck routes no-reducible-tool-result overflows to compaction, the outer loop retries after any reported compaction, and existing tests explicitly allow three successful compactions followed by the same overflow without checking that the selected prompt actually shrank.

## What This Changes

The issue reports that embedded/Paperclip agent auto-compaction retries identical oversized prompts instead of reducing a long accumulated session history or failing once with a clearer recovery path.

## Best Possible Solution

Add a convergence guard and recovery path for overflow compaction: after a forced compaction, verify the selected prompt estimate or active transcript actually shrank below budget, rotate/truncate/summarize accumulated history when policy permits, and otherwise return one actionable blocked error instead of retrying identical precheck overflows.

## Reproduction Assessment

Yes. The issue has a concrete log sequence, and current tests already model the outer loop's three compaction retries; a focused regression can synthesize many medium-sized session messages with no oversized tool results and assert that retries stop or the prompt estimate decreases.

## Solution Assessment

No current behavior is not the best way to solve it. Retrying after any `compactResult.compacted` value is too weak; the maintainable fix should prove compaction convergence against the actual next prompt window and make the non-converging path explicit.
