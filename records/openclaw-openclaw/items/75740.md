## Summary

Keep open. The earlier ClawSweeper diagnosis still matches current main: overflow recovery treats any successful compaction result as progress, then retries the same prompt path without proving the selected prompt now fits or has materially shrunk.

## What This Changes

The issue reports that embedded/Paperclip agent overflow recovery retries auto-compaction three times without reducing an 873k-token selected prompt window.

## Best Possible Solution

Add a focused overflow-recovery progress gate that stops retrying after compaction unless the next prompt is below budget or has materially shrunk, while leaving broader transcript-retention policy separate.

## Reproduction Assessment

Yes, at source level. The existing overflow-loop harness models repeated over-budget failures after successful compaction, and current run.ts still continues after any compacted result without a shrink or budget check; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is a post-compaction budget or material-shrink check with a clear no-progress overflow error, not a broad retention-policy rewrite.
