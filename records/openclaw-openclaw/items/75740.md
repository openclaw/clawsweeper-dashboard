## Summary

Keep open. Current main still has a plausible no-progress overflow recovery path: aggregate prompt pressure can be routed to `compact_only`, and the outer runner retries after any `compacted: true` result without proving that the next prompt fits or materially shrank.

## What This Changes

Embedded/Paperclip agent overflow recovery should avoid retrying identical oversized prompts when auto-compaction reports success without reducing the selected prompt below budget.

## Best Possible Solution

Add an overflow-recovery progress gate: after compaction, compare the next selected prompt estimate against the same budget or require a material shrink before retrying; if there is no progress, stop the retry loop and return an actionable overflow error. Keep transcript rotation as an opt-in mitigation rather than changing broad retention policy in this narrow fix.

## Reproduction Assessment

Yes. The issue includes concrete repeated precheck diagnostics, and the existing overflow-loop unit harness can model a `compact_only` precheck where compaction reports success but the retry remains over budget.

## Solution Assessment

No. The current implementation is not the best way to solve it because it treats `compacted: true` as sufficient progress; the safer narrow fix is to gate retries on post-compaction budget/progress evidence.
