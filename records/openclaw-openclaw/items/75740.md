## Summary

Keep open. Current main still has a plausible no-progress overflow-recovery path: the precheck can route aggregate history pressure to `compact_only`, and the outer loop treats any `compacted: true` result as success without proving the next selected prompt is smaller or under budget.

## What This Changes

Prevent embedded/Paperclip auto-compaction from retrying identical oversized prompts when accumulated session history remains far over the model budget after compaction.

## Best Possible Solution

Add an overflow-recovery progress check: after compaction, prove the next selected prompt fits or materially shrank, or stop retrying and surface a clear actionable overflow error. Keep transcript rotation as a configurable mitigation rather than making a broad default-policy change in this fix.

## Reproduction Assessment

Yes. The reporter provided concrete repeated precheck diagnostics, and the existing overflow-loop unit harness already models repeated compaction retries before failure; a targeted test can reproduce a `compact_only` precheck where compaction reports success but the post-compaction prompt remains over budget.

## Solution Assessment

No. The current retry-after-`compacted` path is not the best fix because it assumes progress; the safer narrow solution is to carry precheck metrics through the attempt result and gate retries on post-compaction budget/progress evidence.
