## Summary

Keep open. Current main still has the no-progress overflow retry shape: aggregate transcript overflow routes to compaction, the attempt path logs the reported precheck metrics, and the retry loop treats `compacted: true` as enough progress without proving the next prompt fits or shrank.

## What This Changes

Embedded/Paperclip agent overflow recovery should shrink the selected prompt or fail clearly instead of retrying identical over-budget prompts.

## Best Possible Solution

Add a focused overflow-recovery progress gate that stops retrying after compaction unless the retry prompt is below budget or has materially shrunk, leaving broader session-retention policy separate.

## Reproduction Assessment

Yes. The issue includes concrete repeated diagnostics, and the existing overflow-loop harness can model repeated over-budget precheck failures after `compact_only` compaction without the private session file.

## Solution Assessment

Yes. The narrow maintainable fix is a post-compaction budget or shrink check with a clear no-progress overflow error, not a new retention policy or config knob.
