## Summary

Keep open. Current main still has the reported no-progress overflow recovery shape: aggregate prompt pressure can route to `compact_only`, and the runner retries after `compacted: true` without proving the next selected prompt shrank or fits the same budget.

## What This Changes

The issue asks for embedded/Paperclip auto-compaction to reduce an oversized accumulated prompt, rotate/truncate history, or fail once with an actionable no-progress overflow error instead of retrying identical oversized prompts.

## Best Possible Solution

Add a narrow overflow-recovery progress gate: after compaction, require a below-budget estimate or material shrink before retrying, and return a clear overflow error when recovery makes no progress; leave broad transcript-retention policy as a separate product decision.

## Reproduction Assessment

Yes. The report provides concrete repeated diagnostics, and the existing overflow-loop unit harness can model `compact_only` plus `compacted: true` with repeated over-budget precheck failures.

## Solution Assessment

No. Current main is not the best solution because it treats `compacted: true` as sufficient progress; the safer fix is a post-compaction budget/progress gate rather than changing broad session retention defaults.
