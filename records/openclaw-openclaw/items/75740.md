## Summary

Keep open. Current `main` still has the reported no-progress retry shape: overflow recovery accepts a successful compaction call as progress, then retries without proving the selected prompt now fits or has materially shrunk.

## What This Changes

The issue reports that embedded/Paperclip agent auto-compaction retried after a precheck overflow while the selected prompt stayed at about 873k estimated tokens against a 242k budget.

## Best Possible Solution

Add a focused overflow-recovery progress gate that stops retrying after compaction unless the retry prompt is below budget or has materially shrunk, while leaving broader session-retention policy separate.

## Reproduction Assessment

Yes. The issue includes concrete repeated diagnostics, and the existing overflow-loop unit harness can model repeated over-budget precheck failures after `compact_only` compaction without the private session file.

## Solution Assessment

No. Current `main` treats `compacted: true` as sufficient progress; the safer maintainable fix is a post-compaction budget or shrink check with a clear no-progress overflow error.

Current `main` can route aggregate prompt overflow to `compact_only`, run compaction, and retry up to three times without proving the selected prompt is now within budget or materially smaller. Add a focused progress gate in the embedded runner overflow-recovery path so no-progress compaction stops with a clear actionable context-overflow error instead of retrying identical oversized prompts.
