## Summary

Keep open: current main and the latest release still do not expose `openclaw skills uninstall`, while source inspection shows stale ClawHub lockfile entries can still drive reinstall on `skills update --all`. The closed original request now points here as the canonical issue, and the open Gateway RPC PR is adjacent but unmerged and does not provide the requested CLI surface.

## What This Changes

Add a narrow `openclaw skills uninstall <slug>` command that removes a workspace ClawHub skill directory and its `.clawhub/lock.json` entry.

## Best Possible Solution

Land one narrow CLI lifecycle implementation that safely removes only the selected skill directory and matching lockfile entry while sharing backend semantics with any eventual Gateway RPC.

## Reproduction Assessment

Yes. Source inspection shows manual directory deletion leaves `.clawhub/lock.json`, and `skills update --all` reads that lockfile and force-installs tracked slugs; I did not run a mutating repro because this review is read-only.

## Solution Assessment

Yes. A narrow CLI command at the existing skills lifecycle boundary is the best fit, using `resolveWorkspaceSkillInstallDir` and `untrackClawHubSkill` rather than the older dashboard/bundled-skill approach.
