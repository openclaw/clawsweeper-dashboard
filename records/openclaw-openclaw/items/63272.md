## Summary

Keep open: current main and the latest release still do not provide `openclaw skills uninstall`, while source inspection confirms stale `.clawhub/lock.json` entries can drive `skills update --all` to reinstall a manually deleted ClawHub skill. The related open Gateway RPC PR is useful adjacent work, but it is not merged and does not cover the requested CLI command.

## What This Changes

Request a narrow `openclaw skills uninstall <slug>` CLI that removes a ClawHub-installed workspace skill directory and its `.clawhub/lock.json` entry.

## Best Possible Solution

Add a single-slug CLI uninstall path for ClawHub-tracked workspace/shared skills that safe-resolves the skill directory, removes the lockfile entry, documents the behavior, and keeps dashboard/Gateway expansion separate unless the existing RPC PR lands first.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: manually removing a tracked skill directory leaves `.clawhub/lock.json`, and `updateSkillsFromClawHub({ workspaceDir })` iterates lockfile slugs and force-installs them again. I did not run a live mutating repro because this review is read-only.

## Solution Assessment

Yes, the narrow CLI/workspace-ClawHub scope is the best starting point; it avoids the stale dashboard/bundled-skill path from the prior PR while using existing safe path and lockfile helpers. The implementation should stay conservative on untracked manual skill directories.
