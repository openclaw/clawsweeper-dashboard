## Summary

Keep open: current main and v2026.6.8 still lack the CLI uninstall surface, and source inspection confirms stale ClawHub lockfile entries can reinstall manually deleted skills during `skills update --all`. The older feature request is closed in favor of this issue, while the open Gateway RPC PR is adjacent and does not provide the requested CLI command.

## What This Changes

Add a narrow `openclaw skills uninstall <slug>` command that removes a workspace ClawHub skill directory and its `.clawhub/lock.json` entry.

## Best Possible Solution

Land one focused CLI lifecycle implementation that safely removes only the selected workspace ClawHub skill directory and matching lockfile entry, sharing backend semantics with any future Gateway RPC.

## Reproduction Assessment

Yes for the source-level lockfile problem: current install writes `.clawhub/lock.json`, and current `skills update --all` reads that lockfile and force-reinstalls tracked slugs. I did not run a mutating repro because this review is read-only.

## Solution Assessment

Yes. A narrow `openclaw skills uninstall <slug>` command at the existing skills lifecycle boundary is the best fit; the older dashboard/bundled-skill approach is broader and riskier.
