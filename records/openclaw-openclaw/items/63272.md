## Summary

Keep open: current main and latest release still lack `openclaw skills uninstall`, and source inspection confirms a ClawHub lockfile entry can cause a manually deleted skill to be reinstalled by `skills update --all`. The closed older request points here, while the open Gateway RPC PR is adjacent, conflicting, and does not implement the requested CLI surface.

## What This Changes

Add a narrow `openclaw skills uninstall <slug>` command that removes a ClawHub-installed workspace skill and its `.clawhub/lock.json` tracking entry.

## Best Possible Solution

Land one focused CLI lifecycle implementation that safely removes the selected workspace ClawHub skill directory and matching lockfile entry, sharing backend semantics with any eventual Gateway RPC.

## Reproduction Assessment

Yes. Source inspection shows ClawHub install writes a workspace lockfile and `skills update --all` uses that lockfile to force-reinstall tracked slugs, while current main has no CLI path that removes both the directory and lock entry.

## Solution Assessment

Yes, with a conservative boundary. A narrow `openclaw skills uninstall <slug>` command at the existing skills lifecycle layer is the maintainable solution for the CLI workflow; dashboard/bundled-skill deletion and Gateway-only RPC work are broader or different surfaces.
