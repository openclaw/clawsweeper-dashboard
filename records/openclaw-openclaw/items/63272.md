## Summary

Keep open: current main and v2026.6.1 expose `skills install`, `skills update`, and `skills verify`, but no `skills uninstall` command, and the current update path can reinstall a manually deleted ClawHub skill from a stale lockfile. The related Gateway RPC PR at https://github.com/openclaw/openclaw/pull/69319 is useful adjacent work, but it is unmerged and does not provide the requested CLI surface.

## What This Changes

Add an `openclaw skills uninstall <slug>` CLI flow that removes a workspace ClawHub skill directory and its `.clawhub/lock.json` entry.

## Best Possible Solution

Land one narrow CLI-backed skill lifecycle uninstall implementation that safely resolves the skill path, removes only the selected managed skill directory, deletes the matching lockfile entry, documents the command, and reuses any merged Gateway/backend helper rather than duplicating semantics.

## Reproduction Assessment

Yes, from source inspection: current main reads slugs from `.clawhub/lock.json` for `skills update --all` and force-reinstalls each tracked slug, so manual directory deletion does not clear the reinstall path. I did not run a mutating repro because this review is read-only.

## Solution Assessment

Yes, the proposed narrow CLI command is the right layer for the reported user workflow, provided it shares lifecycle/path-safety semantics with any merged Gateway helper. The older dashboard/bundled-skill approach is broader and riskier than the workspace ClawHub CLI scope.
