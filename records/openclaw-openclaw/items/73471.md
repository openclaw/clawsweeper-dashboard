## Summary

Keep open. Current main still classifies orphan primary transcripts by absence from the active `sessions.json` map, so valid older bare `.jsonl` chat transcripts can still be offered for `.deleted.<timestamp>` archival; the interactive-confirmation and archive-discovery mitigations reduce impact but do not fix the classifier described here.

## What This Changes

The issue asks doctor to stop treating valid historical primary session transcripts as orphan files and proposes broader hardening for doctor repair recommendations.

## Best Possible Solution

Define a real orphan-transcript predicate in the doctor state-integrity path and add regression coverage that preserves valid historical primary transcripts absent from `sessions.json`. Prompt wording, restore support, corrupt-store diagnostics, and doctor recommendation CI hardening should follow as explicitly scoped work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: create a sessions directory with one current `sessions.json` entry plus an older valid `<sessionId>.jsonl` absent from the store; current `noteStateIntegrity` will classify the older file as orphan because it is missing from `referencedTranscriptPaths`.

## Solution Assessment

No. The current interactive confirmation and archive-discovery mitigations are safer than silent repair, but they do not solve the definition bug; the narrow durable solution is to change the classifier and tests first.
