## Label Justifications

- `P2`: The issue describes a normal-priority but serious doctor/session-history safety bug with limited scope and no evidence of a current widespread crash or outage.
- `impact:data-loss`: doctor --fix can rename valid historical transcript files so they disappear from chat history and memory discovery surfaces.
- `impact:session-state`: The bug is in session transcript lifecycle classification and can drift the visible session history from stored transcript state.

## Summary

Keep open. Current main still treats any primary transcript file not referenced by sessions.json as an orphan archive candidate, while sessions.json remains a latest-pointer store rather than a complete transcript registry; related fixes cover trajectory suffixes or memory visibility fallback, not the doctor classifier and archive prompt that can hide historical chat history.

## What This Changes

The issue asks OpenClaw to stop doctor --fix from classifying historical primary session transcripts as orphaned disposable files, improve recovery/prompting, and harden doctor recommendation tests.

## Best Possible Solution

Define the canonical transcript lifecycle contract first, then change doctor to classify only genuinely invalid or deleted-agent transcript files as archive candidates, add recovery or clearly scoped tombstone semantics, and cover state-mutating doctor recommendations with semantic preservation tests.

## Reproduction Assessment

Yes by source inspection, though I did not mutate a live checkout to reproduce it. Current main still builds the orphan set from sessions.json references and any unreferenced primary .jsonl file, matching the reporter's minimal path.

## Solution Assessment

Unclear as a one-shot fix because the core bug is real, but the right repair requires a maintainer decision on historical transcript lifecycle and recovery semantics before changing doctor --fix behavior.
