## Summary

Keep open: the PR is focused gateway work, but current evidence shows it does not fix the linked Control UI switch-back path and it adds an unbounded HTTP/SSE in-flight payload. Real behavior proof is still mock-only.

## What This Changes

The PR threads an in-flight run resolver from gateway runtime state into the HTTP session-history JSON/SSE endpoint and adds a new gateway test file for that response field.

## Best Possible Solution

Fix the browser Control UI reload path to consume the existing bounded `chat.history.inFlightRun` contract, then keep HTTP/SSE parity only if that response is bounded and proven in a real reconnect flow.

## Reproduction Assessment

Yes, from source inspection: Control UI session switching clears local stream/run state and reloads through JSON-RPC `chat.history`, while the current UI result type and loader do not adopt `inFlightRun`. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No. This is plausible HTTP API parity, but not the best fix for the reported Control UI bug unless the UI consumes `chat.history.inFlightRun` and the new HTTP/SSE snapshot follows the same bounding contract.
