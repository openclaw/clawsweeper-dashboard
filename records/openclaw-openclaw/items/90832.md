## Summary

Keep open: the PR has a useful HTTP history parity idea, but it is not merge-ready because it does not fix the browser Control UI path described by the linked issue and it forwards unbounded in-flight text into HTTP JSON/SSE responses.

## What This Changes

The PR threads an optional in-flight run resolver from gateway runtime state into the HTTP session-history handler and adds tests for the new JSON/SSE response field.

## Best Possible Solution

Fix the browser Control UI reload path to consume and adopt the existing `chat.history.inFlightRun` contract, and keep the HTTP/SSE extension only if it is bounded and covered by an active-run test.

## Reproduction Assessment

Yes from source inspection for the review findings: the browser UI reload path calls JSON-RPC `chat.history` and does not consume `inFlightRun`, while this PR changes only the HTTP handler. The unbounded HTTP payload issue is also source-reproducible from the direct spread of the resolver snapshot into JSON/SSE responses.

## Solution Assessment

No: this is a plausible HTTP API parity change, but not the best fix for the stated Control UI bug as written. The safer path is to adopt `chat.history.inFlightRun` in the Control UI and reuse the existing in-flight payload bounding for HTTP parity.
