## Summary

Keep open. Current main still routes `sessions.list` through per-row model/cost/thinking enrichment that matches the reporter's hot-path evidence, and the linked timeout PR is only a call-site workaround rather than a fix for the underlying list latency.

## What This Changes

The issue asks OpenClaw to make `sessions.list` responsive for roughly 100-200 sessions by reducing repeated per-row enrichment and improving timing visibility.

## Best Possible Solution

Add request-scoped memoization or batching for repeated provider/model/thinking/cost/context enrichment in the existing `sessions.list` row builder, preserving the current row schema and leaving timeout or lean-list API changes separate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; the production logs and current source make the hot path credible, but I did not run current main against a matching large multi-agent session store to reproduce the 16-21s wall time.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow repair direction: optimize or memoize the existing row-enrichment path before changing the public session row contract or relying on longer timeouts.
