## Summary

Keep open. Current main and the latest release still provide local backup archive creation and verification only; native restore/import/list/cloud/encryption/scheduled backup behavior and component-aware cron/session restore semantics remain unresolved, despite useful community workarounds linked in the discussion.

## What This Changes

Request to add built-in backup, export, import, restore, list, cloud storage, encryption, retention, and scheduling workflows for config, cron jobs, sessions, workspace state, and related OpenClaw data.

## Best Possible Solution

Keep this as the native backup lifecycle umbrella: build on create/verify archives, define safe restore/import semantics and component modeling for cron and sessions, then split cloud, encryption, scheduling, and docs into reviewed follow-ups.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main verification path is high-confidence because source, docs, tests, changelog, and the v2026.5.3 tag all show create/verify-only backup support with no native restore/import/list/cloud/encryption/scheduled lifecycle.

## Solution Assessment

No, not as one autonomous monolithic implementation. The maintainable path is to settle archive format, restore safety, secret policy, and component semantics first, then land smaller reviewed pieces for restore, cron merge, session-history handling, cloud/encryption, and scheduling.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested native restore/import and cloud-backup lifecycle is security-sensitive and needs maintainer design before implementation.

Concerns:

- **[medium] Define restore archive safety before native import:**
  - body: A native restore command would consume archives and write into OpenClaw state, credentials-adjacent files, cron stores, session stores/transcripts, pairing data, and workspaces. Core needs path containment, symlink handling, permissions, dry-run, conflict, rollback, overwrite, and partial-failure rules before merge.
  - confidence: 0.92
- **[medium] Define secret and cloud-backup policy:**
  - body: The request includes credentials-adjacent state plus cloud storage, retention, and encryption. Maintainers should define what is excluded, how keys are managed, whether credentials are restorable, and how docs prevent plaintext cloud artifacts.
  - confidence: 0.86
