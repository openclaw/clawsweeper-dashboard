## Summary

Keep open. Current main provides first-party local backup archive creation and verification, plus migration pre-apply backup support, but it still does not implement the requested native restore/import/export/list/cloud/encryption/scheduled-backup lifecycle or component-aware cron/session restore behavior. Community tools such as Keep My Claw and openclaw-workspace-sync are useful workarounds, and narrower follow-ups exist, but the maintainer-side discussion explicitly keeps the native umbrella open.

## What This Changes

Request native backup and restore utilities for OpenClaw config, cron jobs, session history, workspace files, cloud storage, encryption, scheduling, and disaster recovery.

## Best Possible Solution

Keep this as the native backup lifecycle umbrella: build on shipped create/verify archive support, define a safe restore/import contract with dry-run validation and explicit component modeling, decide secret/cloud/encryption policy, then split cron merge, session-history export/import, cloud storage, scheduling, and docs into reviewed follow-ups while documenting community tools as optional workarounds.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main verification path is high-confidence because source, docs, tests, changelog, and issue discussion all show create/verify only with no native restore/import/list/cloud/encryption/scheduled-backup lifecycle.

## Solution Assessment

Unclear as a single implementation because the requested workflow is broad and security-sensitive. The maintainable direction is to settle archive format and restore safety first, then land smaller reviewed pieces for restore, cloud/encryption, scheduling, cron merge, and session-history semantics.

## Risks / Open Questions

- Native restore/import would write into config, credentials-adjacent files, cron stores, sessions/transcripts, pairing state, and workspaces, so path containment, symlink handling, permissions, rollback, conflict, and partial-failure semantics need explicit design.
- Cloud backup and encryption choices remain product/security decisions, including key management, plaintext artifact avoidance, provider scope, and what credential material may be included or restored.
- The umbrella overlaps narrower open work such as backup restore, per-agent backup/clone, configurable exclusion rules, and profile export/import, so maintainers should decide which item is canonical before implementation.
- Session history and cron job merge semantics are not modeled by the current archive manifest, so a restore command cannot be safely added as a thin extractor over the existing tarball format.
