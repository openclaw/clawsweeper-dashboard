## Summary

Keep open: current main still serializes ACPX startup plugin loading before gateway sidecars, and an open fix PR with closing syntax is the active path rather than a reason to close this issue before merge.

## What This Changes

Decouple ACPX-independent gateway sidecars so channels, heartbeat, and cron can start without waiting for ACPX/runtime-post-bind readiness.

## Best Possible Solution

Review and land a maintained fix such as https://github.com/openclaw/openclaw/pull/79756 if it safely starts ACPX-independent sidecars before runtime-post-bind and resolves the heartbeat/cron expectations, then close this issue on merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main awaits startup plugin loading before sidecars and gates heartbeat on `startupSidecarsReady`; the Windows cold-boot timestamps were not live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the open PR is reviewed against the full issue scope, because starting channels early is plausible but heartbeat, cron, and readiness semantics may need explicit lifecycle ownership decisions.
