## Summary

Keep open: current main still serializes runtime plugin loading before sidecar startup, heartbeat/cron still wait on sidecars-ready, and the open closing PR is unmerged with an acknowledged channel-registry ordering blocker.

## What This Changes

Request to start Discord, browser, heartbeat, and cron independently of slow ACPX/runtime-post-bind readiness during Gateway startup.

## Best Possible Solution

Land a dependency-aware startup path that makes channel runtime registry availability happen before channel startup, while moving heartbeat and cron off slow ACPX readiness without weakening ACPX readiness contracts.

## Reproduction Assessment

Yes for source-level reproduction: current main awaits runtime plugin loading before sidecars and gates scheduled services on startupSidecarsReady. I did not run the Windows 11 cold-boot timing scenario.

## Solution Assessment

No: current main has not solved the issue, and the linked PR's one-file reorder is not the best fix because it can run startChannels before the channel registry is pinned. A startup dependency seam or channel-runtime pin is the safer direction.
