## Summary

Keep open: current main and v2026.5.7 have already improved the 2026.4.15 startup path by starting channels before plugin services and making ACPX lazy by default, but the requested dependency-driven parallel sidecar startup is not fully implemented because heartbeat and cron still activate only after startup sidecars finish and ACPX remains a default startup plugin service.

## What This Changes

The issue requests dependency-driven gateway startup so Discord, browser, heartbeat, and cron are not serialized behind ACPX readiness or registration.

## Best Possible Solution

Implement a small startup dependency model that lets channel transports, heartbeat, and other ACPX-independent services start without waiting for ACPX, while ACP-specific consumers subscribe to ACPX readiness/degraded state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows heartbeat and cron still cannot activate until startup sidecars, including plugin services, complete; I did not live-reproduce the Windows cold-boot timestamps, and Discord is partly improved in current main because channels start before plugin services.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the requested dependency-driven startup model is the right end state, but it needs maintainer design judgment rather than a narrow one-file bug fix or the ACPX readiness-only change in https://github.com/openclaw/openclaw/pull/79624.
