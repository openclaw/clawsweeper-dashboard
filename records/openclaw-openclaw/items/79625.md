## Summary

Keep open: current main improves part of the path by starting channel sidecars before plugin services, but it still does not implement dependency-driven parallel startup; heartbeat and cron remain gated on sidecars-ready, and ACPX is still an onStartup plugin service participating in startup orchestration.

## What This Changes

Request to decouple Discord, heartbeat, and cron startup from ACPX readiness so ACPX-independent sidecars start in parallel instead of waiting behind ACPX.

## Best Possible Solution

Add a small dependency-aware startup lifecycle that lets ACPX-independent channel transports and schedulers start without waiting for ACPX probe work, while preserving explicit readiness/degraded signals for ACP consumers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection on current main shows heartbeat and cron still cannot activate until sidecars-ready, including awaited startup plugin services, but the Discord portion is partly improved and I did not run a Windows cold-boot timing repro.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the requested dependency-driven startup model is the right target shape, but it needs maintainer lifecycle design because current code intentionally gates scheduled services until startup sidecars finish.
