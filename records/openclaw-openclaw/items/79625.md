## Summary

Keep open: current main still lets ACPX-related startup work gate the sidecar-ready path, and the open closing PR has not merged. The existing discussion correctly separates this from ACPX readiness semantics in https://github.com/openclaw/openclaw/pull/79624.

## What This Changes

Request decouples gateway sidecars such as Discord, heartbeat, and cron from ACPX readiness so ACPX startup latency does not delay channel availability.

## Best Possible Solution

Keep this issue as the canonical tracker for a dependency-aware gateway startup fix where ACPX-independent channels, heartbeat, and cron can start without waiting for ACPX, while ACPX-dependent consumers wait on explicit ACPX readiness.

## Reproduction Assessment

No live high-confidence reproduction was run here. Source inspection does show the relevant current-main ordering: startup plugins load before sidecars, and heartbeat/cron wait for the sidecars-ready path.

## Solution Assessment

Yes for the requested direction, but the implementation should be dependency-aware rather than a broad early `Promise.all()` before channel runtime registry state exists. The open PR should be reviewed or replaced with that narrower startup-boundary fix.
