## Summary

Keep open: current main still awaits startup plugin runtime work before entering the sidecar phase, and heartbeat/cron still wait for sidecars-ready. The linked closing PR is still open and its own discussion acknowledges a registry-order blocker, so the issue should remain open while that design is reviewed.

## What This Changes

The issue requests dependency-driven Gateway startup so Discord, heartbeat, cron, and other sidecars do not wait for ACPX readiness when they have no ACPX dependency.

## Best Possible Solution

Land a dependency-aware startup fix that pins or loads channel runtime registry state before channel start, runs slow ACPX-dependent work on its own readiness path, and separately decouples heartbeat/cron from ACPX where safe.

## Reproduction Assessment

Yes for source-level reproduction: current main still awaits `plugins.runtime-post-bind` before sidecars and gates scheduled-service activation on `startupSidecarsReady`. I did not reproduce the Windows cold-boot timing trace live.

## Solution Assessment

No, closing now is not the best path. The requested direction is valid, but the linked PR's early channel start happens before the channel registry is pinned, so the maintainable fix needs a dependency-aware startup seam rather than the current reorder.
