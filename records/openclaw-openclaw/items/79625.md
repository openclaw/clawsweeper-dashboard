## Label Justifications

- `P1`: The issue reports a real startup sequencing problem that can delay live channel and scheduler availability for users during cold gateway boot.
- `impact:message-loss`: Discord/channel startup is delayed behind ACPX startup work, so inbound channel handling can be unavailable during the startup window.
- `impact:crash-loop`: The affected behavior is gateway process startup availability and readiness latency, with sidecars blocked behind slow startup work.

## Summary

Keep open: current main still loads startup plugin runtime before starting the sidecar phase, and heartbeat activation still waits for sidecars-ready. The linked implementation PR remains open and its own discussion identifies a channel-registry ordering blocker, so this needs maintainer design review rather than cleanup closure.

## What This Changes

The issue requests dependency-driven gateway sidecar startup so Discord, heartbeat, and cron are not serialized behind ACPX runtime readiness.

## Best Possible Solution

Land or replace the linked PR with a dependency-aware startup design that pins or loads channel registry state before channel start, keeps ACPX-dependent work on its own readiness path, and revalidates heartbeat and current lazy cron behavior separately.

## Reproduction Assessment

Yes from source inspection: current main awaits plugins.runtime-post-bind before starting the sidecar phase that starts channels, and heartbeat activation waits for sidecars-ready. I did not run a live Windows cold-start timing repro in this read-only review.

## Solution Assessment

No, closing now is not the best way to solve this. The requested direction remains valid, but the current linked PR needs a dependency-aware registry/startup design rather than a simple reorder that can run before channel registry pinning.
