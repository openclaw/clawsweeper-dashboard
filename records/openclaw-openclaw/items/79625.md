## Summary

Keep open: current main still starts sidecars only after `plugins.runtime-post-bind`, heartbeat still waits for `startupSidecarsReady`, cron remains post-ready maintenance, and the closing PR is still open with an acknowledged registry-order blocker.

## What This Changes

Request decoupling Discord/channel, heartbeat, and cron startup from slow ACPX/runtime plugin readiness so independent sidecars can start in parallel.

## Best Possible Solution

Add a dependency-aware startup path that makes channel runtime availability, heartbeat, and cron independent from slow ACPX readiness while preserving explicit readiness contracts for ACPX consumers.

## Reproduction Assessment

Yes for source-level reproduction: current main awaits runtime plugin loading before sidecars and gates heartbeat on `startupSidecarsReady`. I did not run the Windows cold-boot timing path from the report.

## Solution Assessment

No, closing now is not the best way to solve the issue: current main still has the coupling, and the open closing PR has a known registry-order flaw. The maintainable direction is a channel-runtime readiness or dependency seam before slow ACPX-dependent work.
