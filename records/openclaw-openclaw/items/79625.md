## Summary

Keep open: current main still serializes startup plugin runtime work before sidecars, and scheduled-service activation still waits for sidecars-ready. The open closing PR tracks the issue but its own discussion acknowledges a registry-order blocker, so the issue should stay open for maintainer design review.

## What This Changes

The issue requests dependency-aware gateway startup so Discord/channel sidecars, heartbeat, and cron are not serialized behind ACPX readiness.

## Best Possible Solution

Introduce a dependency-aware startup path that makes channel registry readiness available before channel startup, keeps ACPX-dependent work on an ACPX readiness path, and preserves heartbeat/cron semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main awaits `plugins.runtime-post-bind` before sidecars and gates scheduled-service activation on `startupSidecarsReady`. I did not run the Windows cold-boot trace live.

## Solution Assessment

No, closing now is not the best way to solve the issue. The requested direction is valid, but the existing closing PR needs a dependency-aware startup seam rather than a one-file reorder before registry pinning.
