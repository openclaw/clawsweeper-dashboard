## Summary

Keep open: current main still serializes startup plugin runtime loading before channel sidecars and scheduled-service activation, and the open closing PR remains unmerged with its registry-order blocker acknowledged by the author.

## What This Changes

Request dependency-driven startup so Discord, heartbeat, and cron can start without waiting for ACPX runtime readiness.

## Best Possible Solution

Rework or replace the linked PR with a dependency-aware startup path that makes channel registry availability precede channel startup while keeping ACPX consumers on ACPX readiness.

## Reproduction Assessment

Yes for source-level reproduction: current main awaits plugins.runtime-post-bind before channel sidecars and gates heartbeat activation on startupSidecarsReady. I did not run the Windows cold-boot timing sequence.

## Solution Assessment

No for the currently proposed fix path: the linked PR moves channel start too early, before the channel registry is pinned. The safer solution is an explicit dependency-ready startup seam or a split channel-runtime load before slow ACPX-dependent work.
