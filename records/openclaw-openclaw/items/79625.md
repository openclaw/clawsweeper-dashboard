## Summary

Keep open: current main and the latest release still load startup plugin runtime before the sidecar phase that starts channels, and the open linked PR remains a candidate but has a registry-ordering blocker that needs maintainer startup-design review.

## What This Changes

The issue requests decoupling Discord/channel, heartbeat, and cron sidecar startup from ACPX readiness so independent services can start before ACPX finishes initialization.

## Best Possible Solution

Land or replace the linked PR with a dependency-aware startup design that pins channel registry state before channel start, keeps ACPX-dependent work on its own readiness path, and validates heartbeat plus lazy-cron behavior explicitly.

## Reproduction Assessment

Yes at source level: current `main` still awaits startup plugin runtime work before the sidecar phase that starts channels, and tests explicitly preserve that order. I did not run a live cold-start repro because this review is read-only and the source path is sufficient to prove the current sequencing.

## Solution Assessment

No: the open linked PR is a plausible implementation candidate, but its own discussion and prior review identify a registry-ordering blocker. The best fix is a dependency-aware startup split or registry-ready seam rather than moving `startChannels()` ahead of the registry pin.
