## Label Justifications

- `P1`: The issue reports a startup sequencing problem that can delay live channel and scheduler availability during cold gateway boot.
- `impact:message-loss`: Discord/channel startup can be delayed behind ACPX startup, leaving inbound channel handling unavailable during the startup window.
- `impact:crash-loop`: The affected behavior is gateway process startup availability and readiness latency, with sidecars blocked behind slow startup work.

## Summary

Keep open: current main still awaits the runtime startup plugin phase before channel sidecars start, and the linked closing PR remains open with an unresolved channel-registry ordering blocker.

## What This Changes

The issue requests dependency-driven gateway startup so Discord, heartbeat, cron, and other non-ACPX sidecars are not delayed by ACPX readiness.

## Best Possible Solution

Land or replace the linked PR with a dependency-aware startup design that pins channel registry state before channel start, keeps ACPX-dependent work on its own readiness path, and separately validates heartbeat and current lazy cron behavior.

## Reproduction Assessment

Yes. Source inspection shows current main awaits `plugins.runtime-post-bind` before the sidecar phase that starts channels, and an adjacent test explicitly preserves that ordering.

## Solution Assessment

No. The active linked PR is a plausible candidate but not the best fix as reviewed, because it can run before channel registry pinning; maintainers need a dependency-aware startup seam.
