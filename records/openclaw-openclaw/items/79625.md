## Summary

Keep open: current main and v2026.6.10 still await startup plugin runtime loading before the sidecar phase that starts channels, while the linked candidate PR remains open with a documented channel-registry ordering blocker.

## What This Changes

The issue requests dependency-driven gateway sidecar startup so Discord, heartbeat, and cron are not serialized behind ACPX readiness.

## Best Possible Solution

Land or replace the linked PR with a dependency-aware startup design that pins channel registry state before channel start, keeps ACPX-dependent work on its own readiness path, and validates heartbeat plus current lazy cron behavior.
