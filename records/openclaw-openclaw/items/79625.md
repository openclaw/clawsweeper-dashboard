## Summary

Keep open: current main and the latest shipped release still serialize sidecars and scheduled services behind startup plugin runtime loading, and the linked closing PR is still open with an acknowledged registry-order blocker.

## What This Changes

Request decoupling gateway sidecar startup so Discord, heartbeat, and cron can start independently from slow ACPX runtime readiness.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/79756 with a dependency-aware startup path that primes channel runtime availability before channel start while preserving ACPX readiness for ACP consumers.

## Reproduction Assessment

Yes for source-level reproduction: current main awaits `plugins.runtime-post-bind` before the sidecar phase, and the focused startup test asserts sidecars wait for startup plugin attachment. I did not run the Windows cold-boot timing path.

## Solution Assessment

No: current main still has the reported coupling, and the open PR direction is useful but currently blocked by channel registry ordering. The safer solution is a channel-runtime registry seam or dependency-aware startup split before slow ACPX-dependent work.
