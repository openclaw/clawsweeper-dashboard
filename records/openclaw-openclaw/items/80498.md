## Summary

Keep open: current main still has a source-reproducible premature-output path for assistant text on non-`sessions_yield` tool-use turns, and the duplicate-announcement side remains plausible but not fully covered by current tests. The open fix candidate at https://github.com/openclaw/openclaw/pull/81621 targets the same bug family, so this issue should stay open until that work is reviewed, landed, or replaced.

## What This Changes

The issue asks OpenClaw to stop treating tool-use progress text as terminal subagent output and to dedupe terminal announcements per child session/run.

## Best Possible Solution

Land or replace the targeted agents-runtime fix so output selection waits for a later assistant message without tool calls and terminal delivery is idempotent by the child-run announce identity.

## Reproduction Assessment

Yes for the premature-output half by source inspection: current main selects assistant text from a non-`sessions_yield` tool-use turn and has a unit fixture expecting that behavior. No for the duplicate-delivery half without a focused failing test or live run, although docs and prior dedupe history support the expected one-completion contract.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: keep it inside subagent announce output/lifecycle code, reuse the existing announce-id machinery, and add regression tests for tool-boundary continuation and terminal idempotency.
