## Label Justifications

- `P1`: Premature or duplicated subagent completion announcements can make active parent workflows proceed on stale or repeated child results.
- `impact:session-state`: The bug lets parent/child run completion state drift from the child run's actual terminal state.
- `impact:message-loss`: The reported selection and duplicate-delivery paths can replace, suppress, or repeat the intended terminal child message.

## Summary

Keep open: current main and v2026.6.8 still have a source-reproducible path for mixed assistant/tool-call text to become subagent completion output, and the duplicate-announcement half is still owned by open, conflicting repair PRs rather than a merged fix.

## What This Changes

The issue requests fixing subagent completion announcements so tool-use progress text is not treated as final child output and the same terminal child result is not announced more than once for the same child session/run.

## Best Possible Solution

Land a refreshed focused fix, or reviewed replacement PRs, that treats mixed text/tool-call turns as progress until a later non-tool final assistant response and adds bounded announce idempotency at the delivery boundary.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the selector returns assistant text from a mixed text/tool-call turn, the adjacent test expects that behavior, and the selected text is persisted into completion state.

## Solution Assessment

Yes for the requested direction. The maintainable fix belongs in the existing subagent output-selection and announcement-delivery/idempotency boundaries, but the open PR branches need refresh or replacement before this can close.
