## Summary

Keep open. The Zalo HTTP-error bug is real and the fix is in the right plugin-local API client, but this patch is not merge-ready because it puts untrusted upstream response text into the thrown error message and still lacks real behavior proof.

## What This Changes

The branch adds a non-2xx response guard to the Zalo API client, adds an `httpStatus` field to `ZaloApiError`, and changes polling-timeout detection to ignore HTTP-status errors.

## Best Possible Solution

Land a Zalo-local `response.ok` guard that preserves structured API errors, keeps HTTP status separate from app `errorCode`, uses stable sanitized error messages, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes at source level. Current main reaches `response.json()` before any HTTP-status guard, so a fake Zalo fetcher returning a non-JSON 502 body would reproduce the reported SyntaxError path; no tests were run because this review is read-only.

## Solution Assessment

No, not as submitted. The plugin-local guard and `httpStatus` split are the right shape, but the final patch should keep remote HTTP body text out of `Error.message` and add proof/tests for the new error paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep remote HTTP bodies out of Error.message:** `extensions/zalo/src/api.ts:146`
  - body: This new branch uses the upstream response body as the `ZaloApiError` message. Existing Zalo paths pass `error.stack`/`error.message` to runtime logs and return `err.description ?? err.message` from probes, so a proxy/WAF/API HTML body or control-character payload can now reach operator-visible output. Use a stable local message such as `Zalo API HTTP error: ${method} (${response.status})` and keep any optional body snippet bounded, redacted, and not part of the default message.
  - confidence: 0.87
- **[P2] Preserve JSON error descriptions without requiring error_code:** `extensions/zalo/src/api.ts:135`
  - body: Current main throws a `ZaloApiError` for any parsed `{ ok: false, description }` response even when `error_code` is absent because the type marks that field optional. The new non-OK branch only preserves structured JSON when `error_code !== undefined`; a non-2xx JSON error with only `description` falls through to a raw JSON-string message and `description: "HTTP <status>"`, so `probeZalo` loses the API description it used to surface. Treat parsed `ok === false`/`description` as structured too, while still recording `httpStatus`.
  - confidence: 0.76
