## Summary

Keep open: current main still has the Zalo HTTP-error parsing bug, and this PR is directionally useful, but the current branch is blocked by missing real behavior proof and two concrete security-sensitive error-body handling issues introduced by the diff.

## What This Changes

The PR adds an HTTP `response.ok` guard to the Zalo API client, preserves structured non-2xx JSON errors, adds `httpStatus` to `ZaloApiError`, and avoids treating HTTP 408 as a Zalo polling timeout.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body handling that preserves structured Zalo JSON errors, keeps HTTP status separate from API error codes, uses stable sanitized messages, and adds focused regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main calls `response.json()` before any HTTP status guard in `callZaloApi`, so a mocked fetch returning `new Response('<html>', { status: 502 })` would throw a JSON parse error instead of `ZaloApiError`. I did not run tests because this is a read-only review.

## Solution Assessment

No. The pre-JSON `response.ok` guard and `httpStatus` split are the right direction, but the branch should use a capped body read and keep remote body snippets out of `Error.message`, with sanitized details stored separately if needed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path uses `await response.text()` and truncates only after the full body is buffered. A large upstream or proxy error response can still create avoidable memory and event-loop pressure, so read only a capped number of bytes from `response.body` and cancel once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:148`
  - body: For non-JSON HTTP errors, the truncated remote response body becomes the `ZaloApiError` message. Zalo polling/startup logging formats `error.stack` or `error.message`, and probe returns `err.description ?? err.message`, so upstream HTML or control characters can be logged or surfaced; use a stable local message and sanitized bounded details separately.
  - confidence: 0.87
