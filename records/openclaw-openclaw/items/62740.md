## Summary

Keep open: current main and the latest release still parse Zalo HTTP responses as JSON before checking HTTP status, but this PR is not merge-ready because the new non-OK branch reads and surfaces untrusted error bodies unsafely and still lacks real behavior proof.

## What This Changes

This PR adds a non-OK HTTP branch to the Zalo API client, preserves structured Zalo JSON errors, adds `httpStatus` to `ZaloApiError`, and narrows polling-timeout detection to app-level error code 408.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body reading, stable sanitized local error messages, preserved structured Zalo JSON details, focused regression coverage, and redacted after-fix runtime proof.

## Reproduction Assessment

Yes, by source inspection. A fetch returning a non-OK HTML/plain-text response reaches `response.json()` on current main before any `response.ok` guard, so the reported SyntaxError path is clear.

## Solution Assessment

No. The status guard and `httpStatus` split are the right direction, but the PR should bound the body read, avoid remote body text in `Error.message`, add focused coverage, and include real after-fix proof before merge.
