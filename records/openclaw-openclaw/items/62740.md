## Summary

Keep open; the PR addresses a real current-main Zalo error path, but the latest diff still introduces unsafe remote error-body handling and lacks required after-fix real behavior proof.

## What This Changes

The branch adds a Zalo `response.ok` guard before JSON parsing, preserves structured Zalo JSON errors on non-2xx responses, and separates HTTP status from API `error_code` for polling timeout checks.

## Best Possible Solution

Revise the Zalo API client to guard non-2xx responses before JSON parsing, preserve structured JSON errors, keep HTTP status separate from API `error_code`, use bounded sanitized non-JSON body handling, and add focused tests for those paths.

## Reproduction Assessment

Yes. Source inspection shows a mocked `ZaloFetch` returning `new Response('<html>', { status: 502 })` reaches `response.json()` on current main before any HTTP-status guard.

## Solution Assessment

No. The `response.ok` guard and `httpStatus` split are the right direction, but the PR should also cap the body read and keep remote body snippets out of standard `Error.message` and logs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK branch calls `response.text()` and truncates only after the whole upstream body is buffered. A large HTML/proxy error response can still create avoidable memory and event-loop pressure, so read a capped number of bytes from `response.body` and cancel once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:145-146`
  - body: For non-JSON HTTP errors, the remote body snippet becomes the `ZaloApiError` message. Polling/startup logging and send/probe error formatting consume `error.stack`/`error.message`, so upstream HTML, reflected data, or control characters can reach logs or user-visible errors; use a stable local message and store sanitized bounded detail separately.
  - confidence: 0.86
