## What This Changes

The branch adds a non-OK HTTP branch to `callZaloApi`, preserves structured Zalo JSON error fields when possible, truncates non-JSON bodies, and adds `httpStatus` to `ZaloApiError`.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body reads, stable sanitized error messages, preserved structured Zalo JSON details, focused regression coverage, and redacted after-fix runtime proof.

## Reproduction Assessment

Yes. A focused source repro is to inject a fetcher that returns `new Response('<html>', { status: 502 })` into `callZaloApi`/`getMe`; current main reaches `response.json()` before any HTTP status guard and would throw a parse error.

## Solution Assessment

No. The status guard and `httpStatus` split are the right direction, but the best fix must bound the body read, avoid remote body text in `Error.message`, add focused regression tests, and include real after-fix proof.
