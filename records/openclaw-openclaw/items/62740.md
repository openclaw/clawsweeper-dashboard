## Summary

Keep open: the underlying Zalo bug is valid, but this PR is not mergeable because the non-2xx path still buffers remote error bodies and exposes body text through Error.message/log surfaces, and the PR does not include after-fix real behavior proof.

## What This Changes

The branch adds a non-2xx guard in Zalo `callZaloApi`, preserves structured JSON error payloads, adds `httpStatus` to `ZaloApiError`, and separates HTTP 408 from app-level polling timeout 408.

## Best Possible Solution

Revise the Zalo API client to guard non-2xx responses before JSON parsing, preserve structured JSON errors, keep HTTP status separate from API `error_code`, and handle non-JSON bodies with bounded reads plus stable sanitized error messages and focused tests.

## Reproduction Assessment

Yes. Source inspection shows a mocked `ZaloFetch` returning `new Response('<html>', { status: 502 })` would reach `response.json()` on current main before any HTTP-status guard, so the failing path is high-confidence even though I did not run a test.

## Solution Assessment

No. The `response.ok` guard and `httpStatus` split are the right direction, but the current PR should use a bounded body read and keep remote body snippets out of standard `Error.message`/logs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path calls `response.text()` and only truncates after the full body is buffered. A large HTML/error response from Zalo or an intermediary can still create avoidable memory and event-loop pressure, so read only a capped number of bytes from `response.body` and cancel the reader once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:145-146`
  - body: For non-JSON HTTP errors, the remote response body becomes the `ZaloApiError` message. Zalo polling/startup logging and send helpers consume `error.stack`/`error.message`, so upstream HTML, reflected data, or control characters can be written to logs or user-visible errors. Use a stable local message and store only sanitized bounded details separately if needed.
  - confidence: 0.86
