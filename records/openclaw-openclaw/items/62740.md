## Summary

Keep open. The PR addresses a real current-main gap in Zalo HTTP error handling, but it should not merge as-is because the new non-2xx branch reads the entire remote error body before truncation and can place remote body text into Error.message, which existing Zalo logging and send paths can surface.

## What This Changes

The PR adds a `response.ok` branch in `extensions/zalo/src/api.ts`, parses non-2xx JSON error bodies, adds `httpStatus` to `ZaloApiError`, truncates non-JSON body text, and excludes HTTP-status errors from `isPollingTimeout`.

## Best Possible Solution

Keep the PR open and revise the Zalo API client so non-2xx responses are handled before JSON parsing, structured Zalo JSON errors are preserved, HTTP status is distinguished from app `error_code`, and any non-JSON body handling uses a bounded read with stable sanitized error messages. Add focused tests for HTML/plaintext 5xx, non-2xx Zalo JSON, and HTTP 408 versus app-level polling timeout.

## Reproduction Assessment

Yes. Current main can be reproduced with a mocked `ZaloFetch` returning `new Response('<html>', { status: 502 })`, which reaches `response.json()` in `callZaloApi`; the PR-introduced concerns can be reproduced with a very large non-JSON error body and a body containing CR/LF/control text.

## Solution Assessment

No. The `response.ok` guard and `httpStatus` split are the right direction, but the safest maintainable solution is to cap the error-body read and keep remote body snippets out of standard `Error.message`/logs unless sanitized and deliberately bounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path calls `response.text()` and only truncates afterward. A large HTML/error response from Zalo or an intermediary will still be fully buffered before the 200-character limit is applied, so this can stall or exhaust memory on the channel runtime. Read only a capped number of bytes from `response.body` and cancel the reader when the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:144-146`
  - body: For non-JSON HTTP errors, the remote response body becomes the `ZaloApiError` message. Zalo polling/startup logging formats `error.stack`/`error.message`, and send helpers return formatted error messages, so upstream HTML, reflected data, or control characters can be written to logs or user-visible errors. Use a stable local message and store only a sanitized bounded snippet separately if needed.
  - confidence: 0.86
