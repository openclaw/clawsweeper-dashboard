## Summary

Keep open. Current main still has the reported Zalo JSON-parse failure path, but this PR is not merge-ready because the new non-OK branch fully buffers remote error bodies and can put remote text into standard error/log surfaces.

## What This Changes

The PR adds a non-2xx guard in `callZaloApi`, preserves structured Zalo JSON errors, adds `httpStatus` to `ZaloApiError`, truncates non-JSON body text, and distinguishes HTTP 408 from app-level polling timeout 408.

## Best Possible Solution

Revise the Zalo API client so non-2xx responses are handled before JSON parsing, structured Zalo JSON errors are preserved, HTTP status stays distinct from API `error_code`, and non-JSON body details use a bounded read with stable sanitized error messages plus focused tests.

## Reproduction Assessment

Yes. A mocked `ZaloFetch` returning `new Response('<html>', { status: 502 })` against current main reaches `response.json()` in `callZaloApi`; the PR-introduced risks reproduce with a very large non-JSON body and a body containing control characters.

## Solution Assessment

No. The `response.ok` guard and `httpStatus` split are the right direction, but the safest maintainable fix caps body reading and avoids putting remote body snippets directly in standard `Error.message`/logs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path calls `response.text()` and only truncates after the full body is buffered. A large HTML/error response from Zalo or an intermediary can still create avoidable memory and event-loop pressure, so read only a capped number of bytes from `response.body` and cancel the reader once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:144-146`
  - body: For non-JSON HTTP errors, the remote response body becomes the `ZaloApiError` message. Zalo polling/startup logging and send helpers consume `error.stack`/`error.message`, so upstream HTML, reflected data, or control characters can be written to logs or user-visible errors. Use a stable local message and store only sanitized bounded details separately if needed.
  - confidence: 0.86
