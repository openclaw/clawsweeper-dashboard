## Summary

Keep open: this PR targets a real shipped Zalo error-handling bug, but the branch still has merge-blocking error-body handling issues and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a non-OK response guard in the Zalo API client, preserves structured JSON error details, and separates HTTP status from app-level polling timeout codes.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body handling, stable sanitized error messages, preserved structured Zalo JSON details, focused regression coverage, and redacted after-fix runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.12 shows callZaloApi awaits response.json() before any response.ok guard, so a non-JSON HTTP 5xx or gateway body can fail as a JSON parse error instead of ZaloApiError.

## Solution Assessment

No. The response.ok guard and httpStatus split are the right direction, but the branch should cap the error-body read and keep remote body snippets out of Error.message before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Limit the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK branch calls await response.text() and only truncates after the full body has been buffered. A large upstream or proxy error response can still create avoidable memory and event-loop pressure, so read only a capped snippet and cancel the stream once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:148`
  - body: For non-JSON HTTP errors, the truncated remote response body becomes the ZaloApiError message. Existing Zalo polling/startup logging and probe fallback consume error.stack or error.message, so upstream HTML or control characters can be logged or shown; use a stable local message with bounded sanitized details stored separately.
  - confidence: 0.87
