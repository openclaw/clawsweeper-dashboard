## Summary

This PR should stay open because it addresses a real Zalo API error-handling bug on current main, but the branch still has blocking remote error-body handling issues and lacks required after-fix real behavior proof.

## What This Changes

The branch adds a non-OK HTTP response path to the Zalo API client, preserves structured Zalo JSON errors, and separates HTTP status from Zalo polling timeout error codes.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body handling, stable sanitized error messages, preserved structured Zalo JSON details, focused regression coverage, and redacted after-fix runtime proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main calls response.json() before any response.ok guard in callZaloApi, so a non-JSON HTTP 502 response would throw a JSON parse error instead of ZaloApiError. I did not run tests because this is a read-only review.

## Solution Assessment

No. The response.ok guard and httpStatus split are the right direction, but the branch should cap the body read and keep remote body snippets out of Error.message before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path calls await response.text() and only truncates after the full body has been buffered. A large upstream or proxy error response can still create avoidable memory and event-loop pressure, so read only a capped amount from response.body and cancel once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:148`
  - body: For non-JSON HTTP errors, the truncated remote response body becomes the ZaloApiError message. Zalo polling/startup logging can emit error.stack or error.message, and probe returns err.description ?? err.message, so upstream HTML or control characters can be logged or surfaced; use a stable local message with sanitized bounded details stored separately.
  - confidence: 0.87
