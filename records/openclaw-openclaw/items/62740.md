## Summary

Keep open: the PR targets a real current-main Zalo bug, but it is not merge-ready because after-fix real behavior proof is missing and the new HTTP error path still needs security hardening around remote response bodies.

## What This Changes

The PR changes `extensions/zalo/src/api.ts` to check `response.ok` before JSON parsing, preserve structured Zalo JSON errors, and separate HTTP status from application polling timeout codes.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body handling, stable sanitized error messages, preserved structured Zalo JSON details, focused regression coverage, and redacted after-fix runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main calls `response.json()` before checking `response.ok`, so a non-JSON HTTP error response can throw a JSON parse error instead of `ZaloApiError`.

## Solution Assessment

No. The `response.ok` guard and `httpStatus` split are the right direction, but the branch should cap the body read and keep remote body snippets out of `Error.message` before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path calls `await response.text()` and only truncates after the full body has been buffered. A large upstream or proxy error response can still create avoidable memory and event-loop pressure, so use a capped stream read such as `readResponseTextLimited` and cancel once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:148`
  - body: For non-JSON HTTP errors, the truncated remote response body becomes the `ZaloApiError` message. Zalo polling/startup logging can emit `error.stack` or `error.message`, and probe falls back to `err.message`, so upstream HTML or control characters can be logged or surfaced; use a stable local message with sanitized bounded details stored separately.
  - confidence: 0.87
