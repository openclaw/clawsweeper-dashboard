## Summary

## What This Changes

The PR adds a `response.ok` guard in the Zalo API client before JSON parsing and separates HTTP status from app-level Zalo error codes.

## Best Possible Solution

Update the Zalo API client with bounded non-2xx body reading, stable sanitized error messages, preserved structured Zalo JSON details, focused regression coverage, and redacted after-fix runtime proof.

## Reproduction Assessment

Yes. Current main and v2026.5.18 both parse `response.json()` before any HTTP status guard in `extensions/zalo/src/api.ts`, so a non-JSON HTTP error body can throw a JSON parse error instead of `ZaloApiError`.

## Solution Assessment

No. The status guard and `httpStatus` split are the right direction, but the PR should bound the body read, avoid remote body text in `Error.message`, and include real after-fix proof before merge.
