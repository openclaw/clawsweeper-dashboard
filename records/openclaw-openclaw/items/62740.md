## Summary

Keep open: current main still parses Zalo API responses as JSON before checking HTTP status, but this branch should not merge as-is because it buffers remote error bodies, puts remote body text in error/log paths, and lacks real behavior proof.

## What This Changes

The PR adds a `response.ok` guard to Zalo `callZaloApi`, preserves non-2xx JSON error payloads, introduces `httpStatus` on `ZaloApiError`, and changes polling-timeout detection to ignore HTTP-status 408.

## Best Possible Solution

Update the Zalo API client with a pre-JSON non-2xx guard that preserves structured Zalo JSON errors, keeps HTTP status separate from API `error_code`, reads only bounded body snippets, uses stable sanitized error messages, and adds focused API tests plus real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A mocked `ZaloFetch` returning `new Response('<html>', { status: 502 })` would hit `response.json()` on current main before any HTTP-status guard, producing a JSON parse error rather than a typed Zalo API error.

## Solution Assessment

No. The `response.ok` guard and `httpStatus` split are the right direction, but the PR should use bounded reads and stable sanitized messages instead of `response.text()` plus remote body text as `Error.message`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the HTTP error body read:** `extensions/zalo/src/api.ts:129`
  - body: The new non-OK path calls `response.text()` and truncates only after the full body is buffered. A large upstream or proxy error response can still create avoidable memory and event-loop pressure, so read only a capped number of bytes from `response.body` and cancel the reader once the cap is reached.
  - confidence: 0.9
- **[P2] Keep remote body text out of Error.message:** `extensions/zalo/src/api.ts:148`
  - body: For non-JSON HTTP errors, the remote response body becomes the `ZaloApiError` message. Zalo polling/startup logging and send/probe error formatting consume `error.stack` or `error.message`, so upstream HTML, reflected data, or control characters can be logged or surfaced; use a stable local message and store only sanitized bounded details separately if needed.
  - confidence: 0.86
