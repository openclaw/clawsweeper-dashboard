## Best Possible Solution

Keep the QMD and URL-response compatibility direction, but require URL-backed generated-image downloads to use a bounded response reader and prove oversized responses fail safely before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap URL image downloads before buffering:** `src/image-generation/image-assets.ts:226`
  - body: This new URL path fetches a provider-controlled response through the guarded HTTP wrapper, but then immediately calls `response.arrayBuffer()`. The guard preserves SSRF/proxy/timeout policy, not body size, so a buggy or malicious compatible provider can stream an arbitrarily large image response and make OpenClaw buffer it before later asset handling has a chance to reject it. Please read the response with an explicit max-byte cap, ideally reusing the existing media limit helpers, and add an oversized-response regression test.
  - confidence: 0.9
