## Summary

Keep this PR open: it fixes real Windows QMD and OpenAI-compatible image response gaps, but the new URL-backed image path still buffers provider-controlled response bodies without a size cap, so it needs a focused repair before merge.

## What This Changes

The PR updates Windows QMD `.cmd` shim and `@tobilu/qmd` fallback resolution, then adds guarded async parsing for OpenAI-compatible image responses that return `data[].url`.

## Best Possible Solution

Keep the PR's ownership direction, but read URL-backed image responses through an existing bounded media reader or equivalent max-byte stream guard while preserving guarded fetch policy and release handling.

## Reproduction Assessment

Yes, at source level. Current main only parses `b64_json` image entries and only resolves `%dp0`-relative `.cmd` shim targets, while the PR body supplies after-fix Windows resolver output and local URL-backed image generation output.

## Solution Assessment

No, not yet. The proposed boundaries are maintainable, but URL fallback should use bounded response reading before buffering provider-controlled bytes.
