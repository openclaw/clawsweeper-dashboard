## Summary

Keep open: the PR is a focused fix with useful proof, but the new provider-controlled image URL path reads the whole response body into memory before any media-size cap can apply, so it needs a small security/availability hardening before merge.

## What This Changes

The branch updates Windows QMD `.cmd` shim and scoped package fallback resolution, then adds guarded async parsing for OpenAI-compatible image responses that return `data[].url`.

## Best Possible Solution

Keep the QMD resolver/package fix and URL response support, but make URL image downloads use a bounded guarded reader or propagated generated-media cap before merge.

## Reproduction Assessment

Yes, at source level: current main only parses `b64_json` image entries and only resolves `%~dp0`-relative `.cmd` shim targets, while the PR body supplies after-fix Windows resolver output and local image generation output. I did not run a Windows or live-provider repro in this read-only review.

## Solution Assessment

No, not quite: the high-level fix direction is right, but the URL branch should read through a bounded guarded path instead of `response.arrayBuffer()` before merge.
