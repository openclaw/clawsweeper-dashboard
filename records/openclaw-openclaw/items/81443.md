## Summary

Keep this PR open: the QMD Windows resolver changes look aligned with the existing strict no-shell boundary, and the contributor supplied useful Windows and image-generation proof, but the new provider-controlled image URL path still buffers the whole response before any size cap.

## What This Changes

The PR teaches Windows QMD shim resolution about quoted absolute entrypoints, switches QMD package fallback to `@tobilu/qmd`, and adds OpenAI-compatible `data[].url` image response parsing through guarded provider HTTP.

## Best Possible Solution

Keep the QMD and URL-response direction, but require URL-backed generated-image downloads to use a bounded response reader before buffering and to prove oversized responses fail safely.

## Solution Assessment

No, not yet: the QMD resolver and `data[].url` support are the right direction, but the URL download implementation should use a bounded response read before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap URL image downloads before buffering:** `src/image-generation/image-assets.ts:226`
  - body: This new URL path fetches a provider-controlled response through the guarded HTTP wrapper, but then immediately calls `response.arrayBuffer()`. The guard preserves SSRF/proxy/timeout policy, not body size, so a buggy or malicious compatible provider can stream an arbitrarily large image response and make OpenClaw buffer it before later asset handling has a chance to reject it. Please read the response with an explicit max-byte cap, ideally reusing the existing media limit helpers, and add an oversized-response regression test.
  - confidence: 0.9
