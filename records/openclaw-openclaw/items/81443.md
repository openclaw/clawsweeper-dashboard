## Summary

Keep this PR open: the QMD and OpenAI-compatible URL-response directions are well grounded, but the new URL image path still buffers a provider-controlled response body without a size cap before producing the asset.

## What This Changes

The PR updates Windows QMD `.cmd` shim and `@tobilu/qmd` fallback resolution, then adds guarded async parsing for OpenAI-compatible image responses that return `data[].url`.

## Best Possible Solution

Keep the PR’s ownership direction, but read URL-backed image responses through an existing bounded media reader or equivalent max-byte stream guard while preserving guarded fetch policy and release handling.

Yes, at source level. ## Solution Assessment

No, not yet. The proposed boundaries are maintainable, but the URL fallback should use a bounded response reader before buffering provider-controlled bytes.
