## Summary

Keep open. Current main has partial installer diagnostics and a few hard-coded retries, but the requested network-aware retry policy, clearer per-error guidance, and cleanup of masked config reads are still incomplete.

## What This Changes

Request to add structured installer error handling, clearer npm failure guidance, and intelligent retries for transient install/config failures.

## Best Possible Solution

Add a small installer-owned error classification layer around the existing log-capture helpers, retry only known transient npm/network failures with bounded backoff, keep permission/disk/version errors fail-fast with actionable guidance, and replace remaining masked config reads with explicit best-effort warnings.

## Reproduction Assessment

Not applicable as a feature/installer-hardening request. Source inspection does show the requested behavior is not fully present: current retries are special-case or broad, and Docker/Podman config reads still suppress failures.

## Solution Assessment

No proposed patch is present, and the sample approach should not be copied as-is because an eval-based generic retry wrapper would be too broad for installer scripts. The maintainable direction is to extend the existing npm log-capture helpers and config-get wrappers with targeted classification and tests.

Implement a narrow installer hardening PR for https://github.com/openclaw/openclaw/issues/42592. In `scripts/install.sh`, extend the existing npm log-capture path instead of adding an eval-based generic wrapper: classify npm error codes from the captured log, retry only transient registry/network failures such as ECONNREFUSED, ETIMEDOUT, EAI_AGAIN, ECONNRESET with bounded backoff, and keep EACCES, ENOSPC, E404/ETARGET, EBADENGINE, EEXIST, and native build-tool failures fail-fast or routed to the existing repair paths with clearer guidance. Preserve macOS Bash 3.2 compatibility and current gum/non-gum behavior. Add equivalent or consciously scoped Windows handling in `scripts/install.ps1` if feasible.
