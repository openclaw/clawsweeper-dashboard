## Summary

This should stay open: current main has partial npm failure diagnostics and a few targeted retries, but it does not implement the requested structured retry/error policy across installer surfaces. The remaining question is product scope, especially whether installer code should auto-repair permission failures or only print actionable guidance.

## What This Changes

The issue requests structured installer error classification, clearer npm diagnostics, and smarter retries for transient install failures.

## Best Possible Solution

Define a narrow maintainer-approved installer error policy, then add focused helpers and tests for transient npm/network retry and per-code guidance without broad auto-repair or blanket removal of optional probe fallbacks.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is an enhancement request. Source inspection does confirm current main only has partial diagnostics/retries and not the requested structured retry policy.

## Solution Assessment

The direction is plausible, but the proposed broad Bash `eval` retry wrapper and blanket `|| true` replacement are not the best final shape. A safer solution is a narrow, tested installer policy around known npm/config probes and explicit maintainer-approved retry/repair cases.
