## Summary

Keep this PR open: the fix direction matches the linked Discord DM read bug, but the branch builds the new read authorization fallback from raw merged `allowFrom`, which does not preserve the effective Discord DM allowlist precedence for legacy/account-scoped config and can authorize the wrong peer on named accounts. Real Discord after-fix proof is also missing.

## What This Changes

The PR adds a Discord read-target fallback that inspects fetched one-to-one DM channel metadata and allows `readMessages` when the recipient matches `allowFrom`, with mocked runtime coverage.

## Best Possible Solution

Land a narrow Discord read-guard fix that uses the effective account DM allowlist helper, preserves one-to-one DM fail-closed checks, covers canonical and legacy/account precedence, and includes redacted real Discord proof.

## Reproduction Assessment

Yes. The linked issue and current source give a high-confidence path: `readMessages` calls `assertReadTargetAllowed`, and current main's no-guild fallback checks only guild channel ids, so a fetched one-to-one DM recipient is not considered; I did not run a live Discord DM.

## Solution Assessment

No. The proposed shape is close, but the implementation should authorize against the effective Discord account DM allowlist rather than raw merged `allowFrom`, so it preserves legacy/account precedence and avoids wrong-peer authorization.

- **[P1] Use the effective DM allowlist for reads:** `extensions/discord/src/actions/runtime.messaging.shared.ts:221-225`
