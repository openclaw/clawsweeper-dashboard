## What This Changes

The PR adds a Discord read-target fallback that permits one-to-one DM reads when fetched DM recipient metadata matches `channels.discord.allowFrom`, with mocked regression tests.

## Best Possible Solution

Land one narrow Discord plugin read-guard fix that reuses the existing effective DM ingress authorization path, rejects disabled/group/malformed DM cases, preserves guild/thread behavior, and includes redacted live Discord proof before merge.

## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P1] Honor disabled DM policy before reads:** `extensions/discord/src/actions/runtime.messaging.shared.ts:375-383`
  - body: The new fallback returns as soon as the fetched recipient matches `allowFrom`, regardless of the effective `dmPolicy` or `dm.enabled` setting. With `dmPolicy: "disabled"` or `dm.enabled: false`, a stale allowFrom entry would still authorize private DM history reads, so route this through the existing DM ingress policy resolver before returning.
  - confidence: 0.92
- **[P1] Use the effective DM allowlist for reads:** `extensions/discord/src/actions/runtime.messaging.shared.ts:218-222`
  - body: `mergeDiscordAccountConfig` can leave inherited root `allowFrom` on a named account even when that account has legacy `dm.allowFrom`; current docs and tests say the account legacy allowlist wins. Build this new DM read fallback from `resolveDiscordAccountAllowFrom({ cfg: params.cfg, accountId })` or the existing DM ingress resolver so it does not authorize the wrong peer/account.
  - confidence: 0.9
