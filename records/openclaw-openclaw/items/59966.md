## Summary

Keep open. Current `main` and the latest release still have the Synology Chat webhook-level 120-second `Promise.race` watchdog before the configured agent timeout path, and the related open PR only addresses timer cleanup rather than the hardcoded timeout behavior.

## What This Changes

The issue asks the Synology Chat webhook/channel path to respect configured `agents.defaults.timeoutSeconds` instead of enforcing a hardcoded 120-second response timeout.

## Best Possible Solution

Remove the Synology webhook-level 120-second watchdog or derive it from the resolved runtime timeout, while preserving request-body/security guards and adding a regression that proves slow valid replies past 120 seconds are not preempted.

## Reproduction Assessment

Yes, by source inspection. A focused fake-timer test can configure a higher agent timeout, keep the Synology `deliver` promise pending past 120 seconds, and observe the current handler rejecting with `Agent response timeout (120s)` before the configured runtime path can finish.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned in the Synology Chat webhook path: remove or configure the outer response watchdog and let the existing channel turn and agent timeout path control runtime duration.
