## Summary

Keep open. Current main and the latest release still race Synology Chat webhook delivery against a hardcoded 120s timeout, so a longer configured `agents.defaults.timeoutSeconds` can still be preempted; the related open PR only addresses timer cleanup and does not change this behavior.

## What This Changes

Make Synology Chat channel delivery respect configured agent runtime timeouts instead of aborting after a hardcoded 120 seconds.

## Best Possible Solution

Fix the Synology Chat plugin so webhook delivery no longer imposes an independent 120s cap, while preserving immediate webhook ACK behavior and adding focused regression coverage for delivery that exceeds 120 seconds under a larger configured agent timeout.

## Reproduction Assessment

Yes. A focused test can configure a delivery promise that remains pending past 120 seconds, advance fake timers, and observe current `processAuthorizedSynologyWebhook` reject with `Agent response timeout (120s)` before the configured agent runtime budget can complete.

## Solution Assessment

Yes. The narrowest maintainable direction is plugin-owned: remove the outer watchdog or derive it from the resolved runtime timeout, rather than changing core timeout defaults or channel contracts.
