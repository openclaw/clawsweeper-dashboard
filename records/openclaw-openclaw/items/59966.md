## Summary

Keep open. Current main and the latest release still have the Synology Chat webhook-level `Agent response timeout (120s)` race, so a longer configured agent runtime can still be preempted before `channel.turn.run` finishes.

## What This Changes

The issue asks the Synology Chat channel timeout path to honor configured agent runtime instead of failing after a hardcoded 120 seconds.

## Best Possible Solution

Fix this in the Synology Chat plugin by removing the independent 120s watchdog, or deriving it from the resolved runtime timeout, while preserving immediate webhook ACK behavior and adding focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. A focused test can configure `deliver` to remain pending past 120 seconds, advance fake timers, and observe the current handler reject with `Agent response timeout (120s)` before a larger configured agent timeout could complete.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned: let `channel.turn.run` own the agent runtime budget instead of changing core timeout defaults or channel contracts.
