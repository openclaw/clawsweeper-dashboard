## Summary

Keep open. Current main and the latest release still have a Synology Chat webhook-level `Agent response timeout (120s)` race that can preempt a longer configured agent runtime, and the related open PR only clears timer handles without changing that behavior.

## What This Changes

The issue asks Synology Chat webhook delivery to honor configured agent timeout settings instead of failing after a hardcoded 120 seconds.

## Best Possible Solution

Fix this inside the Synology Chat plugin by removing the independent 120s watchdog, or deriving it from the resolved runtime timeout, while preserving immediate webhook ACK behavior and adding regression coverage for delivery that exceeds 120 seconds.

## Reproduction Assessment

Yes. A focused test can configure `deliver` to remain pending past 120 seconds, advance fake timers, and observe the current handler reject with `Agent response timeout (120s)` before a larger configured agent timeout could complete.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned: let `channel.turn.run` own the agent runtime budget instead of changing core timeout defaults or channel contracts.
