## Summary

Keep open. Current `main` and the latest release still have a Synology Chat webhook-level 120-second `Promise.race` watchdog that can reject before the configured agent timeout path completes, and the related open PR only addresses timer cleanup rather than honoring `agents.defaults.timeoutSeconds`.

## What This Changes

The issue requests that Synology Chat channel delivery respect the configured agent timeout instead of failing after a hardcoded 120 seconds.

## Best Possible Solution

Remove the Synology webhook-level 120-second watchdog or derive it from the resolved runtime timeout, while preserving existing security/request guards and adding a regression that proves valid slow turns past 120 seconds are not preempted.

## Reproduction Assessment

Yes, by source inspection. Configure `agents.defaults.timeoutSeconds` above 120 seconds, keep the Synology `deliver` promise pending past 120 seconds, and the current handler will reject with `Agent response timeout (120s)` before the configured runtime path can finish.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned in the Synology Chat webhook path: remove or configure the outer response watchdog and let the existing channel turn and agent timeout path control runtime duration.

## Repair Work Prompt

Fix the Synology Chat webhook timeout bug reported at https://github.com/openclaw/openclaw/issues/59966. Current `extensions/synology-chat/src/webhook-handler.ts` races `params.deliver(...)` against a hardcoded 120-second `setTimeout`, which can preempt `agents.defaults.timeoutSeconds`. Remove that outer watchdog or derive it from the resolved runtime timeout without adding a new config option. Preserve existing request-body timeout, invalid-token throttling, in-flight guard, DM policy, username-resolution, and reply-delivery behavior. Add focused regression coverage in `extensions/synology-chat/src/webhook-handler.test.ts`, preferably with fake timers, proving a valid slow `deliver` that resolves after 120 seconds is not converted into `Agent response timeout (120s)`. Coordinate with https://github.com/openclaw/openclaw/pull/39745 only as timer-cleanup context; that PR intentionally preserves behavior and is not the functional fix. Add a changelog entry because this is user-visible channel behavior. Validate with `pnpm test extensions/synology-chat/src/webhook-handler.test.ts` and `pnpm check:changed`.
