## Summary

Keep open. Current main and v2026.5.22 still contain the Synology Chat webhook-level 120-second watchdog, which can preempt the documented configurable agent runtime timeout before a long reply finishes.

## What This Changes

The issue asks for Synology Chat inbound reply handling to respect the configured agent timeout instead of a hardcoded 120-second channel watchdog.

## Best Possible Solution

Fix this in the Synology Chat plugin by removing the outer delivery watchdog or deriving it from the resolved agent runtime timeout, with regression coverage for long-running replies.

## Reproduction Assessment

Yes for a focused source-level or fake-timer reproduction, though not a live Synology run. With `agents.defaults.timeoutSeconds` above 120 seconds and Synology `deliver` still pending, `processAuthorizedSynologyWebhook` rejects at 120 seconds first.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned in the Synology webhook path; the related timer-cleanup PR is not sufficient because it preserves the hardcoded behavior.
