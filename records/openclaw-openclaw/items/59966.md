## Summary

Keep open. Current main still has the Synology Chat webhook-level 120-second Promise.race watchdog, and the related open PR only clears timer handles without changing that timeout behavior.

## What This Changes

The issue asks Synology Chat webhook delivery to respect the configured agent timeoutSeconds setting instead of using a hardcoded 120-second watchdog.

## Best Possible Solution

Remove the plugin-local 120-second agent-response watchdog or derive it from the resolved runtime timeout, with regression coverage proving slow valid Synology replies past 120 seconds are not preempted.

## Reproduction Assessment

Yes, by source inspection. A focused fake-timer test can leave the Synology deliver promise pending beyond 120 seconds and observe the current handler hitting the hardcoded timeout before a higher configured agent runtime could finish.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned: remove or configure the Synology webhook watchdog and let the existing channel turn and agent timeout path control the runtime.
