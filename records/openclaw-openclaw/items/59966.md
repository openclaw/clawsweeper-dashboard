## Summary

Keep open. Current main and v2026.5.19 still contain the Synology Chat webhook-level 120-second watchdog, while the documented agent runtime timeout can be configured higher; the related timer-cleanup PR is open but preserves the hardcoded behavior, and the earlier PR that removed it was closed unmerged.

## What This Changes

The issue requests that Synology Chat/channel delivery respect configured `agents.defaults.timeoutSeconds` instead of a hardcoded 120-second watchdog.

## Best Possible Solution

Fix this in the Synology Chat plugin by removing the outer delivery watchdog or deriving it from the resolved agent runtime timeout, then add focused regression coverage without changing the core timeout contract.

## Reproduction Assessment

Yes, by source inspection rather than a live Synology run. With `agents.defaults.timeoutSeconds` above 120 seconds and Synology `deliver` still pending, `processAuthorizedSynologyWebhook` rejects at 120 seconds before the configured timeout path can finish.

## Solution Assessment

Yes. The requested direction is correct, but the narrowest maintainable fix is plugin-owned in the Synology webhook path; the related timer-cleanup PR alone is not sufficient because it preserves the hardcoded behavior.
