## Summary

Keep open. Current main and latest release still contain the Synology Chat webhook-level 120-second watchdog, so a configured `agents.defaults.timeoutSeconds` above 120 seconds can still be preempted before the normal agent timeout path finishes.

## What This Changes

The issue asks for Synology Chat channel delivery to honor the configured agent timeout instead of a hardcoded 120-second watchdog.

## Best Possible Solution

Remove the Synology webhook-level 120-second delivery watchdog, or derive any remaining watchdog from the resolved runtime timeout, and add focused regression coverage proving slow valid turns past 120 seconds are not preempted.

## Reproduction Assessment

Yes, by source inspection. Configure `agents.defaults.timeoutSeconds` above 120 seconds, let the Synology `deliver` promise remain pending past 120 seconds, and the current handler rejects with `Agent response timeout (120s)` before the configured timeout path can finish.

## Solution Assessment

Yes. The narrowest maintainable solution is plugin-owned in `extensions/synology-chat/src/webhook-handler.ts`: remove or configure the outer delivery race and let the shared channel turn and agent timeout path control runtime duration.
