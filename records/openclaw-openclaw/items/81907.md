## Summary

Keep open: current main and the latest release still trim WebChat text before `chat.send`, and this PR is the active focused fix candidate with sufficient real send-path proof. I found no blocking correctness or security finding, but maintainers should decide whether optimistic transcript display preservation must be part of the same fix.

## What This Changes

The PR preserves raw WebChat message text for normal, queued, steered, and controller `chat.send` paths while keeping blank, command, history, and dedup checks on trimmed text.

## Best Possible Solution

Preserve one canonical raw user message through `chat.send` and, if maintainers consider it part of this bug, the immediate WebChat user transcript display; keep trimmed text only for empty-message, command, history, and dedup decisions.

## Reproduction Assessment

Yes. Current main trims the WebChat draft before `chat.send` in the host and controller send paths, which source-reproduces the linked issue's whitespace-loss behavior.

## Solution Assessment

Mostly yes. Preserving raw text at the send funnels while keeping trimmed validation and command detection is the narrow maintainable fix for the outbound payload; transcript rendering can be included only if maintainers want the broader display contract in this PR.
