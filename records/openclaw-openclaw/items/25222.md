## Summary

Current main still accepts same-session follow-ups through steer or follow-up queue paths without returning a channel-visible acknowledgement, and the closed unmerged implementation PR plus duplicate issue do not supersede this canonical product request. The feature remains necessary, but its cross-channel defaults, portable actions, and reason-code contract require maintainer product judgment before implementation.

## What This Changes

Add an immediate channel-visible busy acknowledgement, optional stop/new-session guidance, continued status signaling, and stable busy-reason metadata for messages arriving during an active same-session run.

## Best Possible Solution

Define one typed active-run acknowledgement result at the core queue/runner boundary that preserves the queued turn, carries stable reason metadata and portable optional actions, and is rendered by generic dispatch plus channel adapters without channel-owned product policy.

## Reproduction Assessment

Yes at source level: active same-session follow-ups that are accepted for steering or follow-up queueing return undefined, so no explicit acknowledgement payload is available to channel dispatch. A live cross-channel recording would still be useful when an implementation exists, but it is not needed to establish the current source gap.

## Solution Assessment

No, the issue’s suggested per-channel SessionBusyError mapping is not the best owner boundary. The maintainable solution is a generic typed queue/runner result that preserves queue semantics and lets transports render portable presentation and actions.
