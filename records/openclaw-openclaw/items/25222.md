## Summary

Keep this issue open. Current main still preserves active-run locking and queued/steered followups, but it does not return the requested immediate channel-visible busy acknowledgement, cancel/new-session guidance, or stable busy-reason/config contract; the related implementation PR is closed unmerged, and the overlapping config request was closed into this canonical tracker.

## What This Changes

Add a scripted session-busy/status reply, with optional cancel or new-session guidance, when a user sends same-session input during a long-running active model response.

## Best Possible Solution

Keep this as the canonical tracker and design one generic busy-ack contract across queue policy, runner return payloads, dispatch, config/docs, and channel-safe tests while preserving steering, followup, collect, interrupt, /stop, and /new semantics.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then returns undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
