## Summary

Keep this open. Current main still treats same-session active-run contention as queue/steer/followup work and, for the followup enqueue path, returns no outbound reply, so the requested immediate busy acknowledgement, cancel/new-session guidance, and stable busy reason are not implemented. The recent discussion already points to #39117 as related unmerged work, so this should remain the canonical UX tracker until that design is reviewed and landed or replaced.

## What This Changes

The issue requests an immediate user-facing busy/status reply and optional cancel/new-session guidance when messages arrive for a locked session during a long-running model or tool turn.

## Best Possible Solution

Implement a generic active-session busy-reply contract in the queue/runner/dispatch boundary: configurable channel-safe status text, optional /stop and /new guidance, stable reason metadata, and focused tests for queue policy, runReplyAgent return payloads, dispatch delivery, and at least one channel path. Review #39117 as the implementation candidate before creating duplicate repair work.

## Reproduction Assessment

Yes. The code-level path is high-confidence: with an active run and followup/collect/steer fallback behavior, the queue policy returns enqueue-followup, runReplyAgent returns undefined, and dispatch sends no reply; this matches the issue’s slow-model same-session reproduction.

## Solution Assessment

Unclear for the open PR path because #39117 is not merged and was not available in the local checkout. The maintainable direction is a generic queue/dispatch contract rather than channel-specific bridge patches, with safeguards around queued-message draining and cancel semantics.
