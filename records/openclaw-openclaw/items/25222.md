## Summary

Keep open: current main still does not provide the requested scripted busy acknowledgement, because active same-session input can be steered or queued and then return no outbound reply payload. The overlapping implementation PR is closed unmerged, and the narrower busy-ack issue was closed as a duplicate of this canonical request.

## What This Changes

Request adding an immediate channel-visible busy/status acknowledgement with optional cancel/new-session guidance when a message arrives for an already active locked session.

## Best Possible Solution

Design a generic active-session busy acknowledgement contract across queue policy, runner return payloads, dispatch, config/docs, and channel tests while preserving steering, followup, collect, interrupt, /stop, and /new semantics.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then return undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
