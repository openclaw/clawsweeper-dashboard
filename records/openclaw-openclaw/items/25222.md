## Summary

Keep open. Current main still has the silent active-session enqueue path for same-session messages, and the related implementation work is open/unmerged rather than shipped or a canonical replacement.

## What This Changes

Request a configurable immediate busy/status reply with optional cancel or new-session guidance when a chat message targets a locked long-running session.

## Best Possible Solution

Keep this as the canonical UX request until maintainers land a generic busy-ack contract across queue policy, runner return payloads, dispatch, channel-safe configuration, and optional actions.

## Reproduction Assessment

Yes, from source inspection. Active same-session followup/steer/queue messages resolve to enqueue-followup, runReplyAgent returns undefined, and dispatch maps undefined to no outbound replies; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation path. The right boundary appears to be a generic queue/runner/dispatch contract, but defaults, config fields, busy reason metadata, and relationship to #39117 need maintainer design.
