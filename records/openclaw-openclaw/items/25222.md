## Summary

Keep open: current main and v2026.5.20 still steer or enqueue same-session busy input without a scripted channel-visible busy acknowledgement, cancel/new-session guidance, or stable busy-reason payload. The related queued-receipt PR is closed unmerged and the narrower configurable-busy-ack issue was closed as duplicate of this issue, so this remains the canonical product/API request.

## What This Changes

Request to add a scripted session-busy acknowledgement, optional cancel/new-session guidance, heartbeat/status behavior, and busy reason metadata when same-session input arrives during an active model/tool run.

## Best Possible Solution

Design one generic active-session busy acknowledgement/status contract across queue policy, runner return payloads, dispatch, config/docs, diagnostics, and representative channel tests while preserving existing queue modes and command semantics.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then return undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config/status shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
