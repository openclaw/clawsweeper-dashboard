## Summary

Keep open. Current main still queues or steers active same-session messages without a generic channel-visible busy/status acknowledgement, and the related implementation PR remains open with unresolved design/security review concerns.

## What This Changes

Request an immediate user-visible busy/status reply, optional cancel or new-session guidance, heartbeat behavior, and busy-reason metadata when a message arrives for an already locked active session.

## Best Possible Solution

Define and land a generic busy-ack contract across queue policy, runReplyAgent return payloads, dispatch, config/docs, and focused tests, while reconciling or replacing the open related PR rather than closing the canonical tracker early.

## Reproduction Assessment

Yes, from source inspection. Active same-session messages can resolve to enqueue-followup, runReplyAgent returns undefined on that path, and dispatch maps undefined to no outbound replies; I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation path. The likely boundary is a generic queue/runner/dispatch contract, but defaults, config shape, busy reason metadata, and the relationship to https://github.com/openclaw/openclaw/pull/39117 need maintainer design.
