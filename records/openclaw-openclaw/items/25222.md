## Summary

Keep open: current main still lacks the requested scripted busy acknowledgement, optional cancel/new-session guidance, and stable busy reason payload. Source review shows active same-session input can be steered or enqueued and then return no outbound reply payload, while related implementation work is closed or narrower rather than a shipped replacement.

## What This Changes

The issue requests a fast channel-visible busy/status reply with optional cancel or new-session guidance when a user messages an already-running session.

## Best Possible Solution

Design a generic active-session busy acknowledgement contract across queue policy, runner return payloads, dispatch, config/docs, diagnostics, and representative channel tests while preserving steering, followup, collect, interrupt, /stop, and /new semantics.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
