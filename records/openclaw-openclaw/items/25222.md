## Summary

Keep open: current main and the latest release still enqueue active same-session followups without a dedicated channel-visible busy/status reply, cancel/new-session guidance, or stable busy-reason contract; related work is closed, unmerged, or only adjacent.

## What This Changes

Request an immediate scripted session-busy reply, optional cancel/new-session actions, typing/status heartbeat, and busy reason metadata when a message arrives for a locked active session.

## Best Possible Solution

Keep this issue as the canonical cross-channel busy-session UX tracker and implement one generic active-session busy contract in queue policy, runReplyAgent, and dispatch with channel-safe text/actions, reason metadata, and focused queue/runner/dispatch/channel tests.
