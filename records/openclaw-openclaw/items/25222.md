## Label Justifications

- `P2`: Silent active-session contention is a normal-priority cross-channel UX gap that affects real long-running conversations without making the core runtime unusable.
- `impact:session-state`: The request concerns status and control while an existing session run remains active and subsequent input is queued or steered.
- `impact:message-loss`: A follow-up that produces no immediate acknowledgement can appear ignored or lost even though it remains queued for later processing.

## Summary

Current main still queues or steers same-session follow-ups without a dedicated channel-visible busy acknowledgement, while the prior implementation PR closed unmerged and the overlapping issue was closed in favor of this canonical request; the remaining work needs a maintainer product decision.

## What This Changes

Requests a channel-visible busy acknowledgement, optional cancel/new-session actions, heartbeat continuity, and stable busy-reason metadata when a message arrives during an active same-session run.

## Best Possible Solution

Introduce one generic active-run acknowledgement result at the queue/runner boundary, with stable reason metadata and portable optional actions that dispatch can render channel-safely while preserving the queued turn and existing queue modes.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence path: send a normal same-session follow-up while an active run uses followup or collect handling, and the runner enqueues it, returns undefined, and dispatch emits no dedicated busy acknowledgement.

## Solution Assessment

Partly. Handling contention in the generic queue/runner/dispatch seam is the narrowest maintainable direction, but raising an error for normal queue control flow and mapping it separately in each channel would duplicate policy; maintainers still need to choose timing, defaults, reason vocabulary, and portable actions.

## Risks / Open Questions

- The acknowledgement must not start a second model run, consume the queued message, or cause duplicate delivery when the original run ends.
- Acknowledgement timing and wording could become noisy across steer, followup, collect, and interrupt modes unless maintainers define when it appears and whether it is configurable.
- Cancel and new-session guidance must use portable typed actions and preserve existing `/stop`, `/new`, authorization, and channel callback semantics rather than making channels infer commands from text.
- A stable busy-reason vocabulary requires an owner for lifecycle transitions such as model wait, tool execution, retry, compaction, and queue wait; otherwise displayed reasons can become stale or misleading.
