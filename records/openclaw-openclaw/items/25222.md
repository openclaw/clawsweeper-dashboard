## Summary

Keep this issue open. Current main partially improved active-run contention by keeping the typing/status cue alive for queued followups, but it still lacks the requested immediate scripted busy acknowledgement, cancel/new-session guidance, and stable busy reason payload; the related implementation PR is closed unmerged and the overlapping configurable-ack issue was closed in favor of this tracker.

## What This Changes

The issue requests a configurable cross-channel busy/status reply with optional cancel or new-session guidance when a user sends input to a locked session during a long-running model response.

## Best Possible Solution

Keep this as the canonical tracker and design one generic busy-ack contract across queue policy, runner return payloads, dispatch, config/docs, and channel-safe tests.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then returns undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, fallback, /stop, /new, collect, followup, steer, and heartbeat semantics.
