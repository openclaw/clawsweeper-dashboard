## Summary

Keep open: current main and the latest release still steer or enqueue busy-session messages without a scripted channel-visible acknowledgement, cancel/new-session guidance, or stable busy reason payload. Related work is either closed, unmerged, or narrower, so this remains the canonical cross-channel UX/product contract.

## What This Changes

Add an immediate user-facing busy/status acknowledgement with optional cancel or new-session guidance when a message arrives for a locked or busy session.

## Best Possible Solution

Design one generic active-session busy acknowledgement contract across queue policy, runner return payloads, dispatch, config/docs, diagnostics, and representative channel tests while preserving steering, followup, collect, interrupt, /stop, and /new semantics.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then return undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
