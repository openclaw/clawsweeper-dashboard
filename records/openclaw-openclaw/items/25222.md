## Label Justifications

- `P2`: Silent same-session queuing is a recurring user-facing reliability and ergonomics problem with limited blast radius, not an emergency outage.

- `impact:message-loss`: Accepted follow-up messages currently receive no acknowledgement and can appear lost while waiting behind the active run.

## Summary

Current main still silently queues same-session follow-ups: it preserves typing while blocked, but returns no channel-visible acknowledgement. The closed implementation PR did not merge, and the overlapping issue was closed in favor of this canonical request, so the feature remains necessary and needs a maintainer product decision before implementation.

## What This Changes

Requests an immediate channel-visible busy acknowledgement, optional cancel/new-session guidance, heartbeat status, and stable busy reasons when a message arrives during an active session run.

## Best Possible Solution

Add a typed generic active-run queue result carrying prepared status facts, preserve the queued turn, and render one conservative channel-safe acknowledgement through portable presentation actions; add configuration only if maintainers reject a safe universal default.

## Reproduction Assessment

Yes at source level. An active same-session follow-up resolves to enqueue-followup, the runner returns undefined, and dispatch emits zero immediate replies; no live channel run was performed during this read-only review.

## Solution Assessment

No, not exactly as proposed. A channel-mapped SessionBusyError would leak core session policy into transports; the narrower maintainable solution is a typed queue-admission result rendered by the existing generic dispatch and portable-action layers.
