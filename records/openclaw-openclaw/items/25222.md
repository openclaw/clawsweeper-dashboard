## Summary

Keep open: current main and latest release still enqueue active same-session followups without an immediate channel-visible busy/status reply, and related work either closed unmerged or covers adjacent session/delivery races.

## What This Changes

Requested an immediate scripted busy/status reply with optional cancel/new-session guidance when a user sends another message to a locked long-running session.

## Best Possible Solution

Implement one generic active-session busy/status contract across queue policy, runner return payloads, and dispatch, with channel-safe text/actions, stable reason metadata, and focused queue/runner/dispatch/channel coverage.

## Reproduction Assessment

Yes, at source level: active same-session followups resolve to enqueue-followup, runReplyAgent returns undefined, and dispatch converts that to no channel-visible replies. I did not run a live chat reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the UX and config contract. The maintainable direction is a generic cross-channel busy/status contract, not a Telegram-only bridge patch or another isolated delivery workaround.
