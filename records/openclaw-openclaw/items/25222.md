## Summary

Keep open. Current main still preserves the active same-session followup path as enqueue plus typing, not an immediate channel-visible busy/status acknowledgement with cancel/new-session guidance or stable busy reason metadata.

## What This Changes

Requests an immediate channel-visible busy/status reply with optional cancel/new-session guidance when a message arrives for an active locked session.

## Best Possible Solution

Keep this issue as the canonical tracker and design one generic busy-ack contract across queue policy, runner return payloads, dispatch, config/docs, and channel-safe tests, folding in the useful config ideas from the duplicate issue without reviving unrelated PR risks.

## Reproduction Assessment

Yes, at source level. Active same-session followups resolve to enqueue-followup, runReplyAgent returns undefined, and dispatch turns undefined into no outbound replies; I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation path. The likely maintainable boundary is a generic queue/runner/dispatch contract, but defaults, config shape, cancel guidance, and stable busy reason metadata need maintainer design before a repair PR is safe.
