## Label Justifications

- `P2`: Silent active-session contention is a normal-priority cross-channel UX gap that affects real long-running conversations without making the core runtime unusable.
- `impact:session-state`: The request concerns user-visible status and control while an existing session run remains active and later input is queued or steered.
- `impact:message-loss`: A follow-up can appear ignored or lost because current queued handling emits no dedicated busy acknowledgement.

## Summary

Keep this open: current main still has a source-reproducible active-session contention path that queues or steers the follow-up without a dedicated channel-visible busy acknowledgement, cancel/new guidance, or stable busy-reason contract.

## What This Changes

Request an immediate session-busy status reply, optional cancel/new-session actions, typing/status continuity, and busy-reason metadata when a same-session message arrives during an active long-running response.

## Best Possible Solution

Add one generic active-run acknowledgement contract at the queue/runner/dispatch boundary, with stable reason metadata and portable optional actions, while preserving existing queue modes and queued-message delivery.

## Reproduction Assessment

Yes, source-reproducible: send a normal same-session follow-up while an active run uses followup or collect handling, and current source enqueues it, returns undefined, and dispatch emits no dedicated busy acknowledgement. I did not run a live chat reproduction in this read-only pass.

## Solution Assessment

Partly: handling contention in the generic queue/runner/dispatch seam is the right maintainable direction, but raising a normal queue state as a per-channel error would duplicate policy. Maintainers still need to choose timing, defaults, reason semantics, and portable actions.
