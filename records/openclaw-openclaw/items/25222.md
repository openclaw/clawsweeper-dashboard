## Summary

Current main and the latest release still lack a channel-visible busy/status acknowledgement when same-session follow-ups are steered or queued behind an active run; the related implementation PR closed unmerged and the narrower duplicate issue was closed in favor of this issue, so the request remains necessary.

## What This Changes

Request an immediate session-busy acknowledgement with optional wait/cancel/new-session guidance and stable busy-reason metadata when a user messages a locked active session.

## Best Possible Solution

Define one generic typed active-run acknowledgement result at the queue/runner boundary, preserve queued-turn semantics, carry stable reason metadata and portable optional actions, and render it through generic dispatch plus channel adapters without channel-owned product policy.

## Reproduction Assessment

Yes, at source level: active same-session follow-ups that are accepted for steering or queued follow-up return undefined, leaving dispatch without a busy acknowledgement payload. I did not run a live cross-channel repro because this read-only review cannot execute artifact-creating tests or live channel flows.

## Solution Assessment

No, the issue's per-channel SessionBusyError mapping is not the best owner boundary by itself. The maintainable solution is a generic typed queue/runner result that preserves queue semantics and lets transports render portable presentation and actions.
