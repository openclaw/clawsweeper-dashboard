## Summary

Current main still does not implement the requested busy acknowledgement: active same-session input is steered or enqueued without a scripted outbound status reply, and this issue remains the canonical tracker after overlapping work was closed or left unmerged.

## What This Changes

Add an immediate channel-visible busy/session status reply with optional cancel/new-session guidance, heartbeat behavior, and busy-reason metadata when same-session input arrives during a long-running active response.

## Best Possible Solution

Design a generic active-session busy acknowledgement contract across queue policy, runner return payloads, dispatch, config/docs, and channel-safe tests while preserving steering, followup, collect, interrupt, /stop, and /new semantics.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then return undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
