## Summary

Keep this issue open. Current main still steers or queues same-session input while an active run is in progress, but it does not produce the requested immediate channel-visible busy acknowledgement, cancel/new-session guidance, or stable busy-reason/config contract.

## What This Changes

Request an immediate busy/status response, optional cancel/new-session guidance, heartbeat behavior, and reason metadata when a message arrives for an active locked session.

## Best Possible Solution

Design one generic active-session busy acknowledgement contract across queue policy, runner return payloads, dispatch, config/docs, and channel-safe tests while preserving steering, followup, collect, interrupt, /stop, and /new semantics.

## Reproduction Assessment

Yes, at source level: active same-session input can be steered or enqueued and then return undefined, which dispatch maps to no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel product/API contract. The maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, queued delivery, steering, /stop, /new, and channel heartbeat semantics.
