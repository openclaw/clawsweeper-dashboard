## Summary

Keep open: current main and v2026.6.10 still enqueue active same-session followups without returning a channel-visible busy/status reply; the duplicate busy-ack issue was closed into this tracker, the related implementation PR is closed unmerged, and adjacent open PRs target different session or delivery races.

## What This Changes

The issue requests a generic busy-session acknowledgement with optional cancel/new-session guidance and reason metadata when a message arrives during an active long-running session.

## Best Possible Solution

Implement one generic active-session busy/status contract in queue policy, runner return, and dispatch, with channel-safe text/actions, stable reason metadata, and focused tests.

## Reproduction Assessment

Yes, at source level: active same-session followups resolve to enqueue-followup, runReplyAgent returns undefined, and dispatch converts that to no channel-visible replies. I did not run a live chat reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the UX and config contract. The maintainable direction is a generic cross-channel busy/status contract, not a Telegram-only bridge patch or another isolated delivery workaround.
