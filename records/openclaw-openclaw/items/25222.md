## Summary

Keep open. Current main still enqueues active same-session messages without an immediate channel-visible busy acknowledgement, and the related implementation work remains open/unmerged rather than shipped or superseding this request.

## What This Changes

Add a fast session-busy/status reply with optional cancel/new-session guidance when a user sends another message to a locked long-running session.

## Best Possible Solution

Keep this as the canonical UX request and implement a generic active-session busy-reply contract around queue policy, runner return payloads, dispatch, and channel-safe config, while reconciling #39117 and #50600.

## Reproduction Assessment

Yes, from source inspection. Active follow-up/queue/steer messages resolve to `enqueue-followup`, `runReplyAgent` returns `undefined`, and dispatch maps that to no replies; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation path. A generic queue/runner/dispatch contract is the right boundary, but the exact config shape, default text, busy reason metadata, and relationship to #39117 need maintainer design.
