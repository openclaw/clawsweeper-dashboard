## Summary

Current main still does not implement the requested channel-visible busy acknowledgement: active same-session input is steered or enqueued and returns no final reply payload, while related attempts were closed without replacing this canonical feature request.

## What This Changes

Request an immediate scripted busy/status reply, optional cancel or new-session guidance, heartbeat/status continuity, and busy reason metadata when a message arrives for a locked session.

## Best Possible Solution

Keep this issue as the canonical tracker and design one generic busy-ack contract across queue policy, runner return payloads, dispatch, config/docs, and focused channel-safe tests.

## Reproduction Assessment

Yes, at source level: active same-session turns currently steer or enqueue and return undefined, and dispatch converts that into no outbound final reply. I did not run a live chat-channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the cross-channel contract. The narrow maintainable direction is a generic runner/dispatch payload and config shape that preserves locking, fallback, /stop, /new, collect, followup, steer, and heartbeat semantics.
