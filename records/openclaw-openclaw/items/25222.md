## Summary

Keep this open. Current main still queues same-session active-run messages without an immediate channel-visible busy acknowledgement, cancel/new-session guidance, or stable busy reason, and the related implementation work in #39117 remains open rather than shipped.

## What This Changes

Request an immediate scripted busy/status reply, optional cancel/new-session actions, typing heartbeat, and busy-reason metadata when a user messages a locked session during a long-running response.

## Best Possible Solution

Keep this as the canonical UX tracker and review #39117 against a generic queue/runner/dispatch busy-reply contract with channel-safe text, optional action guidance, stable reason metadata, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows active same-session followup/collect/steer paths enqueue work, return undefined, and dispatch no reply, matching the slow-model followup scenario described in the issue.

## Solution Assessment

Unclear for the open PR path because #39117 is unmerged and needs design review. The maintainable direction is a generic queue/runner/dispatch contract rather than channel-specific bridge patches.
