- `P2`: This is a normal-priority cross-channel UX improvement for long-running active sessions with bounded but real user impact.
- `impact:message-loss`: A same-session follow-up can appear ignored or lost because current queued handling emits no dedicated busy acknowledgement.

## Summary

Keep open: current main and v2026.6.8 still enqueue active-session follow-ups without a dedicated channel-visible busy acknowledgement, cancel/new guidance, or stable busy-reason contract, and the overlapping PRs/issues are either closed, adjacent, or not a safe replacement.

## What This Changes

Request an immediate scripted session-busy reply with optional wait/cancel/new-session actions and busy reason/status handling when a user sends another message to a locked active session.

## Best Possible Solution

Define one generic active-session busy acknowledgement contract at the queue/runner/dispatch boundary, with portable typed actions and stable reason metadata, then validate it through focused queue, runner, dispatch, and at least one channel-path test.

## Reproduction Assessment

Yes, source-reproducible: send a same-session follow-up while an active run is using followup or collect handling, and current source enqueues it, returns undefined, and dispatch emits no dedicated busy acknowledgement. I did not run a live chat reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the product contract. The generic queue/runner/dispatch seam is the maintainable place, but timing, defaults, reason semantics, and portable actions need maintainer direction before implementation.
