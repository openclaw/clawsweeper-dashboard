## Summary

Keep open. Current main still builds the full Control UI in the Dockerfile, so the intended install-smoke speedup is not implemented, but this PR should not merge as-is because the root Dockerfile image is now reused by package/update and Bun-global smoke lanes that need real Control UI assets.

## What This Changes

The PR adds an opt-in Docker build argument that writes a minimal Control UI placeholder instead of running `pnpm ui:build`, then passes that argument from install-smoke Docker image builds.

## Best Possible Solution

Rebase the idea onto current main, keep the build-stage `ARG` so the Docker skip works, and pass the skip only to a fast-only Docker startup smoke image while preserving a real Control UI bundle for full install-smoke, package, update, Bun-global, release, and package-acceptance paths.

## Reproduction Assessment

Yes. Source inspection is enough: current main routes the full root Dockerfile image into package/update and Bun-global smoke scripts that copy `/app/dist` and pack it, while package checks require real Control UI asset payloads.

## Solution Assessment

No. The Docker `ARG` approach is viable, but the workflow scoping in this PR is not the narrowest maintainable fix after current main split the fast and full install-smoke paths.
