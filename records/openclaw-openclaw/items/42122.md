## Summary

Keep open. The CI speedup is still potentially useful and current main still runs `pnpm ui:build` unconditionally in the Dockerfile, but this PR is not merge-ready because its skip argument would be unsafe if applied to the root image now reused by full install-smoke package/update and Bun-global lanes.

## What This Changes

The PR adds an opt-in Docker build argument to skip the Control UI build during install-smoke Docker image builds and writes a minimal placeholder `dist/control-ui/index.html` instead.

## Best Possible Solution

Keep the optimization but rebase it onto the current fast/full install-smoke split, limiting the placeholder UI image to fast-only Docker startup coverage while preserving real Control UI assets for package, update, Bun-global, release, and package-acceptance paths.

## Reproduction Assessment

Yes. Source inspection is enough: the current full install-smoke workflow reuses the root Docker image for package/update and Bun-global package smokes, and package validators require real `dist/control-ui/assets/` entries that the PR placeholder would not provide.

## Solution Assessment

No. The Docker `ARG` approach is viable after the PR’s scoping fix, but passing it to a root image reused by full package/update lanes is not the narrowest maintainable solution on current main.
