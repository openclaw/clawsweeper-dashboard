## Summary

Keep open. The PR has a useful smoke-build speedup and the Docker ARG scoping issue was addressed in `5b0fc196c`, but it is still not merge-ready on current `main` because the placeholder Control UI image would be reused by package/update and Bun-global smoke paths that require real Control UI assets.

## What This Changes

The branch adds an opt-in Docker build arg that writes a minimal Control UI placeholder instead of running `pnpm ui:build`, and passes it from install-smoke Docker image builds.

## Best Possible Solution

Keep the optimization, but rebase it onto the current fast/full install-smoke split and limit the placeholder UI build to fast-only Docker startup coverage while preserving full UI assets for package, update, Bun-global, release, and package-acceptance paths.

## Reproduction Assessment

Yes. Source inspection is enough: current `main` reuses the root Dockerfile image for package/update and Bun-global paths, those paths copy `/app/dist`, and tarball validation requires `dist/control-ui/assets/` entries that the PR placeholder would not create.

## Solution Assessment

No. The Docker ARG mechanism is viable after the PR's build-stage scoping fix, but passing the skip arg to a reusable full root image is broader than the maintainable boundary on current `main`.
