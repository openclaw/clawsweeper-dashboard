## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement the smoke-only Control UI build skip, but the submitted branch is stale against the split fast/full install-smoke workflow and needs tighter scoping so placeholder Control UI output cannot leak into package, update, or release-adjacent smoke paths.

## What This Changes

The PR adds an opt-in Docker build argument that writes a minimal `dist/control-ui/index.html` placeholder instead of running `pnpm ui:build`, and passes that argument from the install-smoke root and extension Docker image builds.

## Best Possible Solution

Rebase the idea onto current main, keep the Docker build argument re-declared inside the build stage, pass it only to the fast root Docker smoke image that checks CLI/container startup, and keep a real Control UI bundle in any image reused for package, update, Bun-global, release, or package-acceptance validation.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing optimization: the Dockerfile still runs `pnpm ui:build` unconditionally, and the install-smoke build commands do not pass `OPENCLAW_SKIP_UI_BUILD`. The package-surface risk is also traceable from the full workflow and scripts that copy `/app/dist` out of the root image before packing.

## Solution Assessment

No. The basic Docker ARG approach is viable, and the PR discussion already resolved the original ARG scoping bug, but the submitted workflow scoping is no longer the best current fix because full install-smoke lanes now reuse the root image for package/update paths that require real Control UI assets.
