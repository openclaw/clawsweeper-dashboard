## Summary

Keep open. The optimization is useful and the earlier Docker ARG scoping bug was addressed, but the PR is stale against the current fast/full install-smoke split and would be unsafe if the placeholder UI is used for the full root image that feeds package, update, and Bun-global smoke paths.

## What This Changes

The PR adds an opt-in Docker build argument that replaces the Docker Control UI build with a placeholder and passes it from install-smoke Docker image builds.

## Best Possible Solution

Keep the Dockerfile opt-in skip, but rebase it onto the current fast/full workflow and pass it only to the standalone fast Docker startup image while preserving real Control UI assets for full/package images.

## Reproduction Assessment

Yes. Source inspection is enough: the current full install-smoke workflow reuses the root Docker image for package/update and Bun-global smokes, while package validation requires real dist/control-ui/assets entries that the PR placeholder would omit.

## Solution Assessment

No. The Docker ARG approach is viable after the scoping fix, but passing it to root images that can feed package/update paths is not the narrowest maintainable solution on current main.
