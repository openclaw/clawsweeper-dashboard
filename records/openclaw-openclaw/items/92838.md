## Summary

Keep open: current main and the latest release still send the reported OpenAI OAuth client ID in the browser OAuth path, and OpenClaw docs define ChatGPT/Codex OAuth as supported. I could not independently reproduce the browser `invalid client id` page because the authorize URL was Cloudflare-challenged from this environment, and the same client ID still received a device-code response, so the next step needs maintainer-owned live OAuth verification rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to replace a revoked built-in OpenAI OAuth client ID so fresh ChatGPT Plus/Pro logins can authenticate again.

## Best Possible Solution

Maintainers should verify the browser OAuth failure with a real fresh ChatGPT account, then either update both OpenAI OAuth flows to a maintainer-owned valid client or document the supported fallback if OpenAI keeps the current client valid for device-code only.

## Reproduction Assessment

No high-confidence failing reproduction yet: source proves current main sends the reported client ID in the browser OAuth flow, but the live authorize probe hit Cloudflare and the device-code usercode endpoint returned HTTP 200. A real browser/fresh-account login is still needed to prove `invalid client id` on current main.

## Solution Assessment

Unclear: replacing the client ID is a plausible fix only if maintainers confirm revocation and can provide/own a replacement OAuth application. The safer path is to validate the real browser flow first and update browser plus device-code behavior together if the client is actually revoked for the supported flow.
