## Summary

Keep open. Current main still has the active non-streaming Pi steering gap this PR targets, but the PR is not merge-ready because omitted/default Pi steering would inherit interrupt semantics, the changelog is missing, and the forward-compat Pi API still needs maintainer validation.

## What This Changes

The PR removes the streaming-only embedded Pi steering gate, adds per-message Pi tool-call steering behavior, classifies reply steering as inject versus interrupt, and expands steering tests.

## Best Possible Solution

Land one coordinated active-run steering fix that defaults normal steering to inject, uses interrupt only for stop-like text, includes caller coverage and a changelog entry, and validates the final Pi API contract.

Yes. Current main is source-reproducible through `runReplyAgent` requiring streaming and `queueEmbeddedPiMessage` rejecting non-streaming handles; the PR defect is source-reproducible by following subagent steering callers that omit the new option into the session-wide interrupt default.

## Solution Assessment

No, not as-is. The direction is right, but the narrow maintainable fix should set omitted/default Pi tool-call steering behavior to inject on each steer or make every caller explicit, then leave only stop-like reply text on the interrupt path.
