## Summary

Keep this PR open, but do not merge it as-is. Current main still has the active/non-streaming steering gap, and the PR moves in the right direction, but the proposed session-wide Pi default would make existing normal steering callers inherit interrupt behavior; it also lacks the required changelog coverage for a user-facing reply fix.

## What This Changes

The PR changes embedded Pi reply steering to attempt delivery during active non-streaming runs, adds a per-message Pi tool-call steering behavior option, and expands Pi run-registry and auto-reply steering tests.

## Best Possible Solution

Land a narrow steering fix after omitted/default Pi tool-call steering behavior is normal `inject`, stop-like text is the only `interrupt` path, all non-auto-reply steering callers are explicit or covered by a safe default, release notes are updated, and the final Pi API contract is validated.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible through `runReplyAgent` requiring `isStreaming` and `queueEmbeddedPiMessage` rejecting non-streaming active handles; the PR defect is source-reproducible by following subagent steering callers that omit `toolCallSteeringBehavior` into the PR's session-wide interrupt default.

## Solution Assessment

No, not as-is. Steering active non-streaming Pi runs is the right direction, but defaulting the whole session to interrupt is broader than the intended stop-like path; default omitted behavior to `inject` or make every caller explicit before merge.
