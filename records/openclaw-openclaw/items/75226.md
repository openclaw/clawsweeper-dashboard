## Summary

Keep open. Current main still has the active non-streaming steering gap, and this PR is a useful direction, but it is not mergeable as-is because normal steering callers that omit the new Pi option can inherit interrupt semantics.

## What This Changes

The PR removes streaming-only Pi steering gates, adds a per-message `toolCallSteeringBehavior` option, classifies reply steering as inject or interrupt, and expands active-run steering tests.

## Best Possible Solution

Land one coordinated active-run steering fix that steers active non-streaming Pi runs, defaults normal steering to inject on every message, reserves interrupt for stop-like text, includes caller coverage, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Current main is source-reproducible by following `runReplyAgent` through the `isStreaming` gate and `queueEmbeddedPiMessage` through the `not_streaming` rejection; the PR defect is reproducible by tracing subagent steering callers that omit the new option into the session-wide interrupt default.

## Solution Assessment

No, not as-is. The direction is narrow and maintainable, but the PR should assign `options?.toolCallSteeringBehavior ?? "inject"` before each Pi `steer()` call, or make every normal caller explicit, before relying on interrupt only for abort-like messages.
