## Summary

Keep open. Current main still has the active-run steering gap this PR is trying to address, but the proposed patch is not merge-ready because normal steering callers that omit the new option would inherit a session-wide interrupt default once the paired Pi API exists.

## What This Changes

The PR changes embedded Pi reply steering to attempt delivery during active non-streaming runs, adds a per-message Pi steering behavior option, and expands Pi run-registry and auto-reply steering tests.

## Best Possible Solution

Land a narrow steering fix only after omitted/default Pi steering behavior is treated as normal inject behavior, stop-like text is the only interrupt path, non-auto-reply steering callers are covered, the changelog is updated, and the final Pi API contract is validated.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible through runReplyAgent and queueEmbeddedPiMessage requiring streaming, and the PR defect is source-reproducible by following subagent announce steering calls that omit toolCallSteeringBehavior into the PR's session-wide interrupt default.

## Solution Assessment

No, not as-is. Steering active non-streaming Pi runs is the right direction, but defaulting the session-wide Pi behavior to interrupt is broader than the intended stop-like path; default omitted behavior to inject or make every caller explicit.
