## Summary

Keep open: current main still has a source-reproducible Codex cron tool-hydration gap, and the active fix PRs are still open rather than landed, so no conservative close reason applies.

## What This Changes

The issue reports that isolated cron agent turns can omit explicitly requested local tools such as `exec` despite `payload.toolsAllow`, causing scheduled commands to be skipped or reported healthy without running.

## Best Possible Solution

Land a focused current-main fix that keeps narrow Codex native tool surfaces fail-closed while either hydrating explicitly requested OpenClaw local tools or failing cron preflight before model dispatch with a missing-required-tool diagnostic.

## Reproduction Assessment

Yes at source level: current main forwards cron `toolsAllow`, then the Codex path excludes `exec`/file tools before final allowlist filtering while disabling native shell/file mode for restricted allowlists. The issue comments add live Linux/macOS release repros, including 2026.5.20, but I did not run a live cron job in this read-only review.

## Solution Assessment

No, current main is not the best end state because it preserves the fail-closed native-shell rule but leaves no usable `exec` path or loud preflight for explicit cron requests. The safer solution is a maintainer-approved dynamic fallback or required-tool preflight with focused regression coverage and runtime proof.
