## Summary

Keep open: current main still has a source-reproducible Codex isolated-cron tool-surface gap, and the active fix PRs are still open rather than landed. The issue discussion also adds independent Linux/macOS live evidence, including a 2026.5.20 report where `exec`, `read`, `write`, and `edit` were absent despite `toolsAllow`.

## What This Changes

The issue reports that isolated cron `agentTurn` runs with `toolsAllow` including `exec` can start without executable local tools and still appear successful.

## Best Possible Solution

Keep this issue open as the canonical tracker until a focused current-main fix either hydrates explicitly requested known local tools for Codex isolated cron runs or fails preflight with a clear missing-required-tool diagnostic, with regression coverage and runtime proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main forwards cron `toolsAllow`, then the Codex path filters out local app-server-owned tools before final allowlist filtering while native shell/file mode is disabled for restricted allowlists.

## Solution Assessment

Is this the best way to solve the issue? No for current main: the fail-closed native-surface rule is reasonable, but it currently leaves explicit `exec` cron requests without a hydrated local-tool fallback or a loud preflight failure.
