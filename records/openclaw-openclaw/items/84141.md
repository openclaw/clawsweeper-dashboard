## Summary

Keep open: current main and the reported v2026.5.18 release still have a source-reproducible Codex cron tool-hydration gap where `toolsAllow: ["exec"]` is forwarded but can be applied after `exec` has already been removed from the Codex dynamic tool set.

## What This Changes

The issue reports that isolated Codex cron agent turns with `payload.toolsAllow` including `exec` can start without an executable tool and skip scheduled commands.

## Best Possible Solution

Keep Codex native shell fail-closed for narrow allowlists, then either hydrate an explicit OpenClaw `exec`/`process` tool path for cron requests or fail cron preflight before model invocation with a missing-required-tool diagnostic.

## Reproduction Assessment

Yes at source level: current main forwards cron `toolsAllow`, then the Codex app-server excludes `exec` before allowlist filtering and disables native shell for restricted allowlists. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security posture: dynamic OpenClaw `exec` hydration and loud cron preflight failure are both plausible, but the current silent model-turn path is not the right end state.
