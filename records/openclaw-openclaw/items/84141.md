## Summary

Keep open: current main still has a source-reproducible Codex cron tool-hydration gap where an explicit cron toolsAllow request for exec can lose OpenClaw exec before allowlist filtering, while restricted allowlists also disable Codex native shell fallback.

## What This Changes

The issue reports that isolated Codex cron agent turns in OpenClaw 2026.5.18 can start without exec even when payload.toolsAllow explicitly requests it.

## Best Possible Solution

Keep Codex native code mode fail-closed for restricted allowlists, but add an explicit path that either hydrates OpenClaw exec/process when cron requested them or fails cron preflight before model invocation with a missing-required-tool diagnostic.

## Reproduction Assessment

Yes, at source level: a focused Codex app-server path with toolsAllow: ["exec"] can filter OpenClaw exec out before runtime allowlist filtering while native shell is disabled for the same restricted allowlist. A live cron run was not executed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security boundary. The safest direction is either explicit OpenClaw exec hydration for requested cron tools or a loud preflight failure, without widening Codex native shell for unrelated restricted allowlists.
