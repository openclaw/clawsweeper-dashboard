## Summary

Keep open: current main still has a source-reproducible Codex cron tool-hydration gap. The cron runner forwards `toolsAllow`, but the Codex app-server path excludes OpenClaw `exec` before applying the allowlist and disables Codex native shell for restricted allowlists, so an isolated cron job that explicitly asks for `exec` can still start without it.

## What This Changes

The issue reports that isolated Codex cron agent turns on OpenClaw 2026.5.18 can omit the requested `exec` tool despite `payload.toolsAllow` including `exec`, causing scheduled command jobs to be skipped or misclassified.

## Best Possible Solution

Keep Codex native code mode fail-closed for restricted allowlists, but add an explicit path that either hydrates existing OpenClaw `exec`/`process` dynamic tools when cron requested them or fails cron preflight before model invocation with a missing-required-tool diagnostic.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a focused Codex app-server test with `toolsAllow: ["exec"]` should show `exec` removed before allowlist filtering while native shell is disabled for the same restricted allowlist; a live cron run was not executed here.

## Solution Assessment

Is this the best way to solve the issue? No for current main: the implementation preserves a valid fail-closed native Codex rule but leaves no executable fallback for an explicit cron `exec` request; the safer fix is dynamic OpenClaw exec hydration or a loud preflight failure.
