## Label Justifications

- `P1`: The report describes a recent 2026.5.18 regression that can prevent isolated cron command-wrapper jobs from running their scheduled commands.
- `impact:security`: The issue concerns command-execution tool exposure and fail-closed behavior in isolated Codex cron sessions.
- `impact:message-loss`: The reported cron runs can skip intended command execution and produce misleading delivered or healthy-looking results.

## Summary

Keep open: current main still has a source-reproducible Codex isolated-cron path where `toolsAllow: ["exec"]` reaches the run, but OpenClaw `exec` is removed from Codex dynamic tools before the allowlist is applied and Codex native shell is disabled for restricted allowlists. Adjacent issue https://github.com/openclaw/openclaw/issues/60841 and PR https://github.com/openclaw/openclaw/pull/60842 cover restrictive core-tool policy re-exposure, but they do not fully cover this Codex app-server default-exclude/native-shell gap.

## What This Changes

Fix isolated Codex cron agent turns so an explicit cron `toolsAllow` request for `exec` either exposes an executable tool or fails loudly before model invocation.

## Best Possible Solution

Preserve the current fail-closed Codex native-shell rule for unrelated restricted allowlists, while adding a narrow Codex cron path that either hydrates the existing OpenClaw `exec`/follow-up tool when cron explicitly requests it or fails cron preflight with a clear missing-required-tool diagnostic before model invocation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: cron forwards `toolsAllow`, Codex removes OpenClaw `exec` before the allowlist filter, and restricted allowlists disable native shell fallback; a live cron run was not executed here.

## Solution Assessment

Is this the best way to solve the issue? Current main is not sufficient. The best fix shape needs maintainer approval because it must choose between explicit OpenClaw `exec` hydration and fail-loud preflight while preserving command-execution safety boundaries.
