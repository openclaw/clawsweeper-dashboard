## Summary

Keep open: current main still has a source-reproducible Codex cron tool-surface gap, and a live commenter confirmed the same missing local tools on v2026.5.20. The likely fix path is already represented by open PRs, but it needs maintainer review because it changes command/file tool exposure in unattended cron runs.

## What This Changes

Fix isolated Codex cron agent turns so finite `payload.toolsAllow` entries such as `exec`, `read`, `write`, and `edit` are available to the run or fail before model dispatch.

## Best Possible Solution

Land a bounded fix that treats cron `toolsAllow` as a required pre-dispatch capability contract: hydrate only the scoped allowed dynamic fallbacks or fail the cron run loudly before model invocation, with regression coverage for deny and sandbox boundaries.

## Reproduction Assessment

Yes at source level, with additional reporter live runs. Current main forwards `toolsAllow`, then the Codex app-server path filters out local execution tools and disables native Code Mode for finite allowlists such as `['exec']`.

## Solution Assessment

No merged solution is present yet. The best path is a scoped dynamic fallback or fail-closed preflight with explicit security tests, and the open candidate PRs need maintainer selection/review.
