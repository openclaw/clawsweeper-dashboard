## Summary

Keep open: current main still has a source-reproducible Codex cron tool-hydration gap. The cron path forwards `toolsAllow`, but the Codex app-server removes OpenClaw `exec` before applying the allowlist and also disables the native Codex tool surface for restricted allowlists, matching the Linux and macOS live reports where isolated cron jobs could not run commands despite requesting `exec`.

## What This Changes

The issue reports a 2026.5.18 regression where isolated cron `agentTurn` jobs that request `exec` through `toolsAllow` can start without command-execution tools and may be marked successful despite not running the command.

## Best Possible Solution

Keep Codex native code mode fail-closed for restricted allowlists, but add a narrow cron/runtime path that either hydrates the requested OpenClaw command-execution tool after full discovery or fails the cron run before model invocation with a clear missing-required-tool diagnostic.

## Reproduction Assessment

Yes at source level: current main forwards cron `toolsAllow`, then the Codex app-server excludes OpenClaw `exec` before allowlist filtering and disables native Codex code mode for `toolsAllow: ["exec"]`. I did not run a live cron job in this read-only review, but the two reported live environments line up with that path.

## Solution Assessment

No final fix is present yet. The best direction is explicit Codex-aware `exec` hydration or a loud preflight failure for cron-required tools, and https://github.com/openclaw/openclaw/pull/60842 is adjacent but not sufficient because it does not touch the Codex app-server filtering path.
