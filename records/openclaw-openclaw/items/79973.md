## Summary

Keep open: current main still drops ACP runtime events and errors emitted after the turn-timeout gate closes, and the only exact fix PR is closed unmerged. The remaining question is a real post-timeout event contract choice because forwarding all late events through `onEvent` can create visible output after timeout.

## What This Changes

The issue reports that ACP runtime events, including late error/done events, are drained but not delivered or surfaced after the turn-timeout gate closes.

## Best Possible Solution

Define the ACP post-timeout stream contract, then update `consumeAcpTurnStream` and focused manager/dispatch coverage so late error or terminal diagnostics remain observable while post-timeout visible output stays suppressed.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: close `eventGate` during a timed-out turn, then have the runtime iterator yield an `error`, `done`, or other event afterward; current main skips it before delivery or error conversion. I did not run a live ACP timeout because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. Capturing late errors is the right direction, but delivering every late `onEvent` is not the best narrow fix because ACP dispatch can render those events visibly after timeout.
