## Summary

Keep open: current main still drops ACP runtime events after the timeout gate closes, including error events, and I did not find a merged fix or canonical duplicate. The remaining decision is the timeout contract: whether late non-output events should still be delivered or a cancellation event should be synthesized.

## What This Changes

The issue asks for ACP turn-timeout stream handling to keep late runtime events and errors observable instead of silently dropping them after `eventGate` closes.

## Best Possible Solution

Define the ACP timeout contract, then add focused regression coverage and update the turn-stream gate so late terminal/error observability is preserved without allowing late output to mutate completed user-facing deliveries.

## Reproduction Assessment

Yes, from source inspection: a runtime iterator that emits `error` or `done` after `eventGate.open` becomes false will hit the top-level `continue` in `consumeAcpTurnStream` and bypass event/error handling. I did not run a live test because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A narrow fix should avoid late output projection while preserving late terminal/error visibility, but the issue's deliver-then-cancel versus synthetic-cancel choice is a product/API decision.
