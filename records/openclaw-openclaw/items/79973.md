## Summary

Keep open: current main and the latest release still have the reported ACP timeout behavior, and the only related fix PR was closed unmerged after review found its late-event forwarding could leak visible output after timeout.

## What This Changes

The issue reports that `consumeAcpTurnStream` silently drops ACP runtime events and errors emitted after the turn-timeout gate closes.

## Best Possible Solution

Define the ACP post-timeout contract, then preserve late error or terminal diagnostics without delivering late visible output, with focused manager and dispatch/projector regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: close `eventGate` through the timeout handler, then have `runtime.runTurn()` yield an `error` or other event; current code skips it before delivery or error surfacing.

## Solution Assessment

Unclear. The issue is valid, but the proposed deliver-then-cancel shape is unsafe as a blanket `onEvent` forwarder because ACP dispatch projects visible events into replies.
