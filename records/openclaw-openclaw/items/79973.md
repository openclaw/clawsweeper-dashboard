## Summary

Keep open: current main still drops ACP runtime events after the timeout gate closes, including runtime `error` events, and the remaining question is the post-timeout ACP event contract rather than whether the code path exists.

## What This Changes

The issue asks OpenClaw to define and fix ACP turn-timeout stream handling so events, especially errors, emitted after the timeout gate closes are not silently discarded.

## Best Possible Solution

Define the ACP post-timeout stream contract, then update `consumeAcpTurnStream`, manager timeout handling, and targeted tests so late errors remain observable without delivering late user-visible output after timeout.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: close `eventGate.open` during a timed-out turn, then have `runtime.runTurn()` yield an `error` or other event afterward; current main skips it before delivery or error conversion. I did not run an executable test because this review is read-only and the local environment lacks `pnpm`.

## Solution Assessment

Unclear until maintainers choose the ACP post-timeout event contract. A safe solution should preserve the timeout boundary for output while still observing or reporting late terminal/error information with regression coverage.
