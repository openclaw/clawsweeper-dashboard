## Summary

Keep open: current main still has the reported ACP timeout behavior, and source inspection shows late stream events are skipped before `onEvent` or error handling can observe them. The reporter also raises a real contract choice, so this should get maintainer direction before implementation.

## What This Changes

The issue requests that ACP runtime events and errors emitted after a turn timeout remain observable instead of being dropped by `consumeAcpTurnStream`.

## Best Possible Solution

Define the ACP post-timeout stream contract, then update the manager and targeted tests so late errors remain observable without leaking late output after timeout.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: close the gate during a timed-out turn, then have the runtime iterator yield an `error` or event afterward; current code skips it before delivery or error conversion. I did not execute that test because this sweep is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. A safe fix should likely preserve the timeout boundary for user-visible output while still observing or logging late terminal/error information, with regression coverage for the chosen behavior.
