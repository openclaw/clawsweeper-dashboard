## Summary

Close as ClawHub/plugin work: current OpenClaw policy and existing plugin APIs already cover optional context-engine, gateway method/HTTP route, and CLI surfaces, while this draft adds a large native core feature without real behavior proof and with security blockers.

## What This Changes

The PR adds a built-in ContextMesh context engine, a new `src/contextmesh` coordinator/worker/store subsystem, core gateway RPC methods/events, and `openclaw contextmesh` CLI commands.

## Best Possible Solution

Publish ContextMesh as an external ClawHub plugin using the existing context-engine, gateway method/HTTP route, and CLI registration APIs; open a narrow API-gap discussion only if a plugin implementation proves one is missing.

Not applicable; this is a new optional feature PR, not a reported bug. Source review and the PR diff are enough to assess the product boundary and merge risks.

## Solution Assessment

No. A native core integration is not the best path because OpenClaw already documents plugin-owned context engines, RPC/HTTP routes, and CLI registration, while core policy sends optional capabilities to ClawHub/plugin distribution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require authorization before issuing worker tokens:** `src/contextmesh/coordinator.ts:309-338`
  - body: The coordinator's approve endpoint can be called over its own HTTP server, approves the pending pairing, and returns the worker device token without going through Gateway operator authorization. Anyone who can reach that coordinator port can mint a ContextMesh worker token, so this approval path needs authenticated operator/pairing authorization or should be removed from the standalone unauthenticated server.
  - confidence: 0.9
- **[P2] Classify the new gateway methods:** `src/gateway/server-methods-list.ts:142-146`
  - body: These `contextmesh.*` methods are added to the advertised core method list, but the diff does not add them to `METHOD_SCOPE_GROUPS` or another classifier. The existing `core gateway method classification` test checks every `coreGatewayHandlers` key and will fail once these handlers are merged; the methods also need an explicit least-privilege operator scope.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces coordinator endpoints that can issue worker device tokens and read host files without a clear Gateway-authenticated authorization boundary.

Concerns:

- **[high] Unauthenticated worker token issuance:** `src/contextmesh/coordinator.ts:309`
  - body: `/contextmesh/workers/approve` approves a pending device pairing and returns `deviceToken` from the standalone coordinator HTTP server, bypassing the existing Gateway `device.pair.approve` caller-scope checks.
  - confidence: 0.9
- **[medium] Coordinator-side arbitrary file reads:** `src/contextmesh/coordinator.ts:351`
  - body: The job submission endpoint accepts a `file` path and reads it from the coordinator host. Without an authenticated and constrained file-selection model, a caller that can reach the coordinator can make it ingest local files.
  - confidence: 0.82
