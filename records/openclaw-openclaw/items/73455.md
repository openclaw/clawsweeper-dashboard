## Summary

Keep open. Current main still lacks the requested Matrix participation/freshness behavior, but this PR is not mergeable as-is because the submitted patch has correctness, plugin-boundary, privacy, and supply-chain review blockers.

## What This Changes

The PR adds Matrix participation/freshness config, state, latest-visible tracking, handler gates, tests, and generated metadata, while also carrying release workflow/docs, updater, changelog, and live-test changes.

## Best Possible Solution

Land a focused Matrix-only implementation on current main: make participation and freshness explicit opt-in, keep Matrix production code behind SDK/runtime seams, preserve command and /steer exemptions, fix revise/auto freshness semantics and the AI parsing contract, document the privacy model, and split release workflow/updater/live-test changes into separate owner-reviewed PRs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has no Matrix participation/freshness schema or handler gate, while the PR diff shows the default-on AI gates and the pre-draft holdback path that suppresses revise/auto before final delivery.

## Solution Assessment

No. The feature direction is plausible, and it relates to #45184 and the broader freshness discussion in #62206, but this branch is too broad and has correctness/security blockers; the safer path is a focused Matrix-only PR with explicit opt-in controls and SDK-boundary imports.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Default the Matrix AI gates to explicit opt-in:** `extensions/matrix/src/matrix/monitor/index.ts:246-257`
  - body: `participationEnabled` and `freshnessEnabled` resolve to true unless explicitly set false, with AI-backed defaults and zero thresholds. Existing Matrix rooms can therefore start sending room text/context to the configured model without adding the new config blocks. Default these controls off, or require an explicit documented migration/approval path.
  - confidence: 0.88
- **[P2] Let revise and auto pass the holdback gate:** `extensions/matrix/src/matrix/monitor/handler.ts:2360-2370`
  - body: When the holdback sees newer room activity, every mode except `send-as-is` is suppressed before drafting. That makes `revise` and `auto` skip the freshness resolver entirely, contrary to their documented semantics. Restrict this early return to `freshnessMode === "suppress"`.
  - confidence: 0.87
- **[P2] Use plugin SDK runtime imports:** `extensions/matrix/src/matrix/monitor/handler.ts:11-17`
  - body: The Matrix plugin production handler deep-imports core `src/**` modules for simple completion and reply generation even though plugin SDK subpaths already expose those helpers. Route this through `openclaw/plugin-sdk/*` before merge to preserve the bundled-plugin boundary.
  - confidence: 0.9
- **[P2] Fix the AI participation schema prompt:** `extensions/matrix/src/matrix/monitor/participation-policy.ts:314`
  - body: The schema string given to the model has malformed optional array field names for `includeAgentIds?` and `excludeAgentIds?`. Because this prompt asks for structured JSON, the malformed keys can reduce or corrupt model output for targeted participation decisions.
  - confidence: 0.78
- **[P2] Remove the deterministic ai-first fallback:** `extensions/matrix/src/matrix/monitor/participation-policy.ts:378-382`
  - body: `resolveParticipationDecision` can parse raw `text` deterministically when `currentDirective` is absent. That leaves a silent fallback path despite the PR contract saying ai-first must not silently fall back to deterministic behavior. Remove the raw-text fallback and require callers to pass an already resolved directive.
  - confidence: 0.83
