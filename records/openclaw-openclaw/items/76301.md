## Summary

Close as not_actionable_in_repo: current main intentionally keeps OpenRouter DeepSeek on provider-managed cache-TTL behavior rather than Anthropic `cache_control` marker injection, and OpenRouter documents DeepSeek prompt caching as automatic with no additional configuration. ([openrouter.ai](https://openrouter.ai/docs/features/prompt-caching))

## What This Changes

The issue requests applying OpenRouter `cache_control` breakpoints to all cache-eligible OpenRouter models, including DeepSeek, instead of only Anthropic model refs.

## Best Possible Solution

Keep the Anthropic-only marker wrapper and documented DeepSeek automatic cache-TTL path; if DeepSeek cache reads still miss, collect repeated same-prefix OpenRouter usage metrics for a new targeted report. ([openrouter.ai](https://openrouter.ai/docs/features/prompt-caching))

## Reproduction Assessment

Partly. Code and test inspection reproduces the absence of `cache_control` markers for non-Anthropic OpenRouter models, but the report does not provide a high-confidence reproduction that DeepSeek cache reads fail via `usage.prompt_tokens_details.cached_tokens`.

## Solution Assessment

Yes. The narrowest maintainable outcome is no source change here, because current OpenClaw behavior matches its docs and OpenRouter’s DeepSeek automatic caching contract. ([openrouter.ai](https://openrouter.ai/docs/features/prompt-caching))
