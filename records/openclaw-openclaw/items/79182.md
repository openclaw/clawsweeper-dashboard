## Summary

Keep open: current main does not implement `--message-file` for `openclaw message send`; the CLI and docs still expose only inline `--message` text for the message body. This is a narrow core CLI feature request rather than a broken existing contract.

## What This Changes

Request adding `openclaw message send --message-file <path>` to read UTF-8 message text from a file while preserving existing routing flags and rejecting simultaneous `--message`.

## Best Possible Solution

Add the flag at the CLI/message command boundary, read UTF-8 into the existing `message` parameter, reject `--message` plus `--message-file`, reuse outbound routing and chunking, and cover it in docs, changelog, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection provide high-confidence evidence that current main lacks a file-backed message body flag.

## Solution Assessment

Yes. A narrow CLI-side file input that hydrates the existing message path is the maintainable direction; it should avoid changing channel delivery policy beyond reusing current chunking behavior.
