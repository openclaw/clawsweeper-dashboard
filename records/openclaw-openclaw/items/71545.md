## Summary

Keep open: current main still lacks streamable-http MCP stale-session recovery, and the PR is useful, but the latest diff still treats any streamable-http tool error whose message contains a stale-session phrase as transport session loss. That can retry application-level JSON-RPC tool errors after reconnecting, so the PR needs a narrow fix before merge.

## What This Changes

The PR changes the bundled MCP runtime to rebuild lost streamable-http sessions, retry the original tool call once, add streamable-http regression harness coverage, and add a changelog entry.

## Best Possible Solution

Keep the PR's streamable-http-only reconnect design, but narrow stale-session detection to SDK transport-level HTTP session failures before retrying, add a regression test that a tool-level JSON-RPC error containing the same phrase is not retried, then rebase and run the targeted MCP runtime test plus the changed gate.

## Reproduction Assessment

Yes. Configure a stateful streamable-http MCP server, make one successful tool call, reset or redeploy the server-side transport so the cached Mcp-Session-Id is invalid, then call the tool again; current main keeps the stale cached client and has no reconnect path.

## Solution Assessment

No, not quite yet. Rebuilding only streamable-http sessions and retrying once is the right shape, but the current detector should distinguish SDK StreamableHTTPError/non-OK POST session validation failures from ordinary MCP tool JSON-RPC errors before it is merge-safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Only retry transport-level stale-session errors:** `src/agents/pi-bundle-mcp-runtime.ts:167-173`
  - body: `isStreamableHttpSessionLostError` matches only message substrings, so a streamable-http tool that throws an ordinary JSON-RPC error like `Session not found` is treated as a lost transport session. The runtime will reconnect and retry the same tool call, which can duplicate side effects for a non-idempotent tool error. Please narrow this to SDK streamable-http POST/HTTP session failures, or otherwise exclude `McpError` tool responses, before retrying.
