## Summary

Keep open: current main still has a source-reproducible Telegram direct fallback path, the latest release contains the same dispatch-error fallback shape, and the related repair/proof PRs were closed unmerged while fresh comments report reachable generic error surfaces on newer builds.

## What This Changes

The issue asks OpenClaw to stop showing spurious generic Telegram failure replies after successful Codex-backed turns and to handle Codex app-server client/terminal failures more precisely in large direct sessions.

## Best Possible Solution

Keep this issue as the canonical maintainer thread, prove the Telegram late-dispatch-after-final path with a live or deterministic QA scenario, suppress duplicate generic fallback only when final delivery is already delivered/queued/suppressed, and route Codex terminal/client-close recovery separately when the app-server lifecycle evidence points outside Telegram.

## Reproduction Assessment

Partially: source inspection gives a deterministic synthetic path where a Telegram DM dispatchError still sends a generic fallback even if delivery evidence exists, but the full large Codex direct-session sequence does not have a high-confidence current-main live repro artifact.

## Solution Assessment

No: current main is only a partial mitigation because Codex retries are narrowly gated and the Telegram dispatch fallback still treats any dispatchError as user-visible fallback-worthy; maintainers should pick the exact live repro boundary before another fix PR.
