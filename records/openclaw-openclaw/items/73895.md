## Summary

Keep open: current main still lacks the requested Gateway v2 routing/config lineage, timing, fallback, policy, gateway-version, decision-trace, and reasoning-token metadata on the relevant Gateway response surfaces. This is a public instrumentation/protocol feature with security-sensitive config lineage, so it needs Gateway/protocol owner design before implementation.

## What This Changes

Request the Gateway to add an additive v2 metadata envelope containing routing/config lineage, route decisions, latency metrics, gateway/policy identifiers, fallback chain, decision trace, and token usage including reasoning tokens for live benchmark smoke compatibility.

## Best Possible Solution

Define an additive, typed, redacted Gateway v2 metadata envelope at the protocol and HTTP boundaries, document its compatibility and privacy contract, then wire it through the OpenAI-compatible, OpenResponses, and live protocol response paths with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. Source inspection is high confidence because the current response schemas and builders do not contain the requested metadata fields.

## Solution Assessment

Unclear because no patch exists yet. Emitting from the routing layer is plausible, but the maintainable path needs owner approval for schema shape, redaction, versioning, and reasoning-token semantics before implementation.
