## Summary

Keep open. Current main still lacks the requested Gateway v2 routing/config lineage, timing, decision trace, gateway version, and reasoning-token metadata across the inspected Gateway response surfaces, and exposing those fields needs explicit schema and redaction decisions before implementation.

## What This Changes

Requests a Gateway v2 metadata envelope that emits routing/config lineage, route policy, timing, gateway version, decision trace, and token usage details for live benchmark smoke validation.

## Best Possible Solution

Define a typed, additive, redacted Gateway metadata envelope at the protocol and HTTP response boundaries, document omitted-field semantics and versioning, then wire it consistently through the approved benchmark-facing live surfaces with compatibility tests and provider-specific usage normalization.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. We do have a high-confidence verification path from current-main source inspection because the relevant schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. The suggested routing-layer envelope is directionally maintainable, but the safest next step is maintainer approval of schema shape, redaction policy, target surfaces, versioning, and reasoning-token semantics.
