## Summary

Keep open. Current main still does not emit the requested Gateway v2 routing/config lineage, policy, timing, decision-trace, gateway-version, or reasoning-token metadata on the inspected Gateway protocol, OpenAI-compatible Chat Completions, OpenResponses, or agent acknowledgement surfaces; exposing those fields also needs explicit Gateway/security owner review.

## What This Changes

The issue requests a backward-compatible Gateway v2 metadata envelope with routing/config lineage, policy identifiers, timing metrics, gateway version, decision trace, fallback chain, and token usage including reasoning tokens for live benchmark smoke validation.

## Best Possible Solution

Define a typed, additive, redacted Gateway metadata envelope at the protocol and HTTP response boundaries, document omitted-field semantics and versioning, then wire it through the approved benchmark-facing live surfaces with compatibility tests and provider-specific usage normalization.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. We do have a high-confidence verification path from source inspection because the relevant schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. Emitting from the routing layer is directionally maintainable, but the safest path is owner-approved schema shape, redaction policy, target surfaces, versioning, and reasoning-token semantics before implementation.
