## Summary

Keep open: current main still lacks the requested Gateway v2 routing/config/timing metadata, and adding it would expand public Gateway wire schemas with security-sensitive lineage data that needs owner design first.

## What This Changes

Request Gateway v2 responses to include routing/config lineage, fallback, policy, gateway version, decision trace, timing, and reasoning-token usage metadata for benchmark smoke validation.

## Best Possible Solution

Define an additive, typed, redacted, versioned Gateway metadata envelope at the protocol and HTTP response boundaries, then wire it through live response surfaces with docs and compatibility tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. Source inspection gives a high-confidence verification path because current response builders and schemas do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. Emitting from the routing layer is plausible, but the maintainable path is to approve schema shape, redaction, versioning, and reasoning-token semantics before wiring responses.
