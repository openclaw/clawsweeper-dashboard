## Summary

Keep open. Current main still does not emit the requested Gateway v2 routing/config lineage, policy, timing, trace, or reasoning-token metadata, and the requested surface crosses public Gateway response schemas plus sensitive config/trace data that needs owner-approved shape and redaction before implementation.

## What This Changes

Request Gateway v2 responses include full routing/config lineage, policy, timing, trace, and token metadata so the external live Gateway smoke harness can run without stub-only mode.

## Best Possible Solution

Define an additive, typed, redacted Gateway metadata envelope at the protocol and HTTP boundaries, document versioning and omitted-field semantics, then wire it through the approved live benchmark-facing surfaces with compatibility tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. Source inspection gives a high-confidence verification path because the relevant schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. Emitting from the routing layer is directionally maintainable, but the safe first step is owner approval of schema shape, redaction policy, target surfaces, versioning, and reasoning-token semantics.
