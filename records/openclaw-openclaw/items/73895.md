## Summary

Keep open: current main still does not expose the requested Gateway v2 routing/config lineage, fallback/policy, timing, gateway-version, decision-trace, or reasoning-token metadata on the Gateway response surfaces. This is a public protocol/instrumentation feature with security-sensitive config lineage, so it needs Gateway/protocol owner design before implementation.

## What This Changes

The issue requests Gateway v2 response metadata for routing/config lineage, fallback/policy identity, gateway version, timing, and token usage including reasoning tokens.

## Best Possible Solution

Define an additive, typed, redacted Gateway v2 metadata envelope at the protocol and HTTP boundaries, document its privacy and compatibility contract, then wire it through OpenAI-compatible chat, OpenResponses, and WebSocket/live response paths with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation/protocol feature request. Source inspection is high-confidence because the current schemas, builders, and usage types do not contain the requested metadata fields or equivalents.

## Solution Assessment

Unclear because no patch exists yet. Emitting from the routing layer is plausible, but the maintainable solution needs owner approval for schema shape, redaction, versioning, and reasoning-token semantics before implementation.
