## Summary

Keep open. Current main still does not emit the requested promotion-grade Gateway v2 metadata on the inspected WebSocket protocol, OpenAI-compatible Chat Completions, OpenResponses, or agent acknowledgement surfaces, and the requested config lineage / decision trace fields need Gateway owner and security review before implementation.

## What This Changes

The issue requests Gateway v2 response metadata for routing, configuration lineage, policy/versioning, timing, decision trace, gateway version, and token usage including reasoning tokens.

## Best Possible Solution

Define a typed, additive, redacted Gateway metadata envelope for the approved benchmark-facing surfaces, document omitted-field semantics and versioning, then wire it through the routing/agent response path with compatibility tests and provider-specific usage normalization for reasoning-token accounting.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. We do have a high-confidence verification path from source inspection because the relevant schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. Emitting metadata from the routing layer is directionally maintainable, but the safest path is to first approve the schema shape, redaction policy, versioning, target surfaces, and reasoning-token semantics.
