## Summary

Keep open. Current main still does not expose the requested Gateway v2 routing/config metadata envelope on the inspected WebSocket, agent RPC, OpenAI-compatible Chat Completions, OpenResponses, or shared usage surfaces; the request is also security-sensitive enough to need maintainer schema and redaction decisions before implementation.

## What This Changes

Request Gateway responses to include routing/config lineage, policy, timing, gateway version, decision trace, and detailed token usage fields so live benchmark smoke validation can run without stub-only mode.

## Best Possible Solution

Define an additive, typed, redacted Gateway v2 metadata envelope at the protocol boundary, then emit it consistently only on approved live benchmark-facing surfaces with docs and tests covering compatibility, omitted fields, timing semantics, and provider-specific usage details.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. We do have a high-confidence verification path from source inspection because the current schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. The suggested routing-layer envelope is directionally maintainable, but the safest solution needs maintainer approval on schema shape, redaction, versioning, target surfaces, and reasoning-token semantics first.
