## Summary

Keep open. Current main still does not emit the requested Gateway v2 routing/config lineage, policy, timing, decision-trace, gateway-version, or reasoning-token metadata on the inspected Gateway protocol, OpenAI-compatible Chat Completions, OpenResponses, or agent acknowledgement surfaces; the requested fields also need maintainer-approved redaction and versioning before they are safe to expose.

## What This Changes

Request Gateway v2 response metadata that reports routing/config lineage, policy/version, decision trace, timing, gateway version, and token usage including reasoning tokens so the benchmark smoke can pass against live Gateway output.

## Best Possible Solution

Define a typed, additive, redacted Gateway metadata envelope at the protocol and HTTP response boundaries, document omitted-field semantics and versioning, then wire it through the approved benchmark-facing live surfaces with compatibility tests and provider-specific usage normalization.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. We do have a high-confidence verification path from source inspection because the relevant schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. The routing-layer envelope is directionally maintainable, but the safest next step is maintainer approval of schema shape, redaction policy, target surfaces, versioning, and reasoning-token semantics.
