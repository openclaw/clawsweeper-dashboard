## Summary

Keep open: current main still does not emit the requested Gateway v2 routing/config/timing metadata, and the change would expand public Gateway response/protocol surfaces with sensitive lineage data that needs owner design first.

## What This Changes

Request Gateway response metadata for config lineage, routing decisions, timings, gateway version, and token usage including reasoning tokens so a live benchmark smoke can pass without stubs.

## Best Possible Solution

Define an additive, typed, redacted, versioned Gateway metadata envelope at the protocol and HTTP response boundaries, then wire it through live responses with docs and compatibility tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an instrumentation feature request. Source inspection gives a high-confidence verification path because current Gateway response builders and schemas do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. Emitting from the routing layer is plausible, but the maintainable path is to approve schema shape, redaction, versioning, and reasoning-token semantics before wiring responses.
