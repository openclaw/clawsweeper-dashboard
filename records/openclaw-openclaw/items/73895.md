## Summary

Keep open: current main still does not emit the requested Gateway v2 routing/config lineage, timing, fallback, policy, gateway-version, decision-trace, or reasoning-token metadata on the Gateway response surfaces, and this is a security-sensitive public protocol feature rather than a narrow bug fix.

## What This Changes

Add additive Gateway v2 response metadata covering routing/config lineage, timing, fallback/policy data, gateway version, decision trace, and reasoning-token usage so benchmark smoke can run against live Gateway output.

## Best Possible Solution

Define an additive, typed, redacted Gateway metadata envelope at the protocol and HTTP boundaries, document its compatibility/privacy contract, then wire it through OpenAI-compatible, OpenResponses, and WebSocket response paths with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation/protocol feature request. Source inspection is high confidence because current main lacks the requested metadata fields in the response schemas and builders.

## Solution Assessment

Unclear because no implementation exists yet. Emitting from the routing layer is plausible, but the maintainable path needs owner approval for schema shape, redaction, versioning, and reasoning-token semantics.
