## Summary

Keep open. Current main still has deterministic skill source precedence, eligibility filtering, alphabetical prompt ordering, prompt limits, strict per-skill config fields, and no `openclaw skills priority` surface; the related preferences and truncation threads are useful context but do not supersede this specific ordering contract request.

## What This Changes

The issue requests user-configurable skill priority or ordering so overlapping skills can be selected predictably.

## Best Possible Solution

Define one first-class skills ordering contract in the existing config schema/docs, apply it after eligibility and per-agent filters but before prompt limits, preserve deterministic fallback ordering, and wire it through CLI inspection, snapshots, runtime materialization, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is source-verifiable because current main has alphabetical ordering, strict skill config without priority, and no CLI/docs command for priority.

## Solution Assessment

Unclear until maintainer product/API review. The requested behavior is useful, but a separate `~/.openclaw/skills-config/priority.json` is probably less maintainable than extending the existing skills config contract so it composes with allowlists, eligibility, prompt limits, and snapshots.
