## Summary

Keep open. Current main and v2026.5.22 still use fixed skill-source precedence plus alphabetical prompt ordering, strict skills config fields, and the existing `openclaw skills` command set; the closed priority PR and related truncation/preference issues are useful context but do not supersede this specific ordering-contract request.

## What This Changes

The issue requests user-configurable priority or ordering for skills so overlapping eligible skills can be selected deterministically.

## Best Possible Solution

Define one first-class skills ordering contract in the existing config schema and docs, apply it after eligibility and per-agent filters but before prompt limits, keep non-priority fallback ordering deterministic, and wire the same semantics through snapshots, CLI inspection, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is source-verifiable because current main and v2026.5.22 have alphabetical ordering, strict skills config without priority, and no `openclaw skills priority` command.

## Solution Assessment

Unclear until maintainer product/API review. User-controlled ordering is useful, but the proposed separate `~/.openclaw/skills-config/priority.json` is probably less maintainable than extending the existing skills config contract so it composes with allowlists, prompt limits, and snapshots.
