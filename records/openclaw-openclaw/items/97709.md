## Summary

Keep open: the narrow MSTeams hardening is useful and uses the right shared bounded-reader helper, but the Graph attachment flow is still incomplete because the main message JSON read remains unbounded and the new collection cap failure is swallowed as an empty result.

## What This Changes

Replaces direct `response.json()` calls with `readProviderJsonResponse` in MSTeams Bot Framework attachment info and Graph hosted-content collection fetches.

## Best Possible Solution

Use the shared bounded JSON reader consistently for all MSTeams attachment success JSON reads, keep bounded-reader overflows diagnostically visible, and add focused regression coverage around oversized Graph message and hosted-content responses.

## Reproduction Assessment

Yes, source-level. Current main has direct `response.json()` reads in the affected MSTeams attachment paths, and the PR body provides terminal output showing bounded cancellation for the two changed paths; I did not run local tests because this review is read-only.

## Solution Assessment

No as submitted. Reusing `readProviderJsonResponse` is the right fix shape, but the best fix should cover the main Graph message JSON read in the same attachment flow and make cap overflows visible rather than silently returning empty hosted content.
