## Summary

Keep this open. Current main still exposes cron and background-task delivery through coarse status enums and raw error text, while the related open items overlap only parts of the requested delivery-reason taxonomy and do not supersede this broader cron/messaging normalization request.

## What This Changes

Request to add structured normalized delivery failure reason codes across cron run history, messaging/background task delivery, status cards, and JSON output.

## Best Possible Solution

Define an additive delivery reason field shared by cron run history and background task delivery while preserving existing status fields for compatibility. Adopt stable reason codes through Gateway protocol schemas, persistence, CLI/UI rendering, docs, and focused tests for config/routing failures, unavailable channels, attempted-send failures, rate limits, skips, and fallback delivery.

## Reproduction Assessment

Yes. The gap is reproducible by static contract inspection on current main: cron, Gateway protocol, UI, and task delivery types still expose only coarse delivery statuses plus raw deliveryError text, and current tests assert those coarse outcomes.

## Solution Assessment

Unclear. The requested direction is sound, but the exact taxonomy and cross-surface rollout should be designed additively with maintainer review rather than bolted onto one cron path, because this touches public protocol and UI/CLI behavior.
