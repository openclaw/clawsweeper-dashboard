## Summary

Keep open. Current main still exposes cron and background task delivery through coarse status enums plus raw error text, and the linked #49190 and #74095 are adjacent open trackers rather than a full replacement for this cross-surface delivery-reason request.

## What This Changes

Request structured, normalized delivery failure reason codes across cron runs, messaging/background task delivery, status cards, and JSON output.

## Best Possible Solution

Define an additive shared delivery reason field while preserving existing status fields, then carry it through cron persistence, Gateway protocol, plugin hooks, task delivery, CLI/UI rendering, docs, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the gap: cron, Gateway protocol, Control UI, plugin hooks, and task delivery expose only coarse delivery statuses plus raw deliveryError text, and current tests assert those coarse outcomes.

## Solution Assessment

Unclear. The requested direction is sound, but choosing reason-code names and semantics is a public protocol/UI/product decision that should be designed additively with #49190 and #74095 rather than fixed in one cron path.
