## Summary

Keep open. Current main still exposes coarse cron/task delivery states plus raw error text across persistence, Gateway protocol, Control UI, plugin hooks, and task delivery; the related cron-warning PR only covers execution-vs-delivery status, not normalized delivery failure reasons.

## What This Changes

Request structured delivery failure reason codes across cron runs, messaging delivery, status cards, and JSON output.

## Best Possible Solution

Define an additive normalized delivery-reason contract while preserving existing status/error fields, then carry it through cron persistence, Gateway protocol, plugin hooks, task delivery, CLI/UI rendering, docs, and focused regression tests.

## Reproduction Assessment

Yes for the current gap: source inspection shows current main lacks normalized delivery reason codes across cron, Gateway protocol, Control UI, plugin hooks, and task delivery. This is source-level verification of a feature gap, not a runtime crash reproduction.

## Solution Assessment

Unclear as an implementation plan. The direction is sound, but maintainers need to choose the reason-code names, additive protocol/storage shape, UI semantics, and relationship to https://github.com/openclaw/openclaw/issues/49190 and https://github.com/openclaw/openclaw/issues/74095 before a narrow implementation is safe.
