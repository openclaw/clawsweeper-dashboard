## Summary

Keep open. Current main still exposes coarse cron/task delivery states plus raw or free-form error text, and the newer CLI deliveryStatus object only partially overlaps the requested normalized cross-surface reason taxonomy.

## What This Changes

The issue requests structured delivery failure reason codes across cron runs, messaging delivery, UI status cards, and JSON output.

## Best Possible Solution

Define an additive normalized delivery-reason contract that preserves existing status/error fields, then carry it through cron persistence, Gateway protocol, plugin hooks, task delivery, CLI/UI rendering, docs, and focused regression tests.

## Reproduction Assessment

Yes for verifying the current feature gap: source inspection shows current main lacks normalized delivery reason codes across cron, Gateway protocol, Control UI, plugin hooks, and task delivery. This is source-level verification of an API/product gap, not a runtime crash reproduction.

## Solution Assessment

Unclear as an implementation plan. The direction is sound, but maintainers need to choose reason-code names, additive storage/protocol shape, UI semantics, and coordination with https://github.com/openclaw/openclaw/issues/49190 and https://github.com/openclaw/openclaw/issues/74095 before a narrow implementation is safe.
