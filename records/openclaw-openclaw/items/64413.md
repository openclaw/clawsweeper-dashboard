## Summary

Keep open. Current main has partial CLI delivery-status reason work, but cron runs, Gateway cron protocol, Control UI, plugin hooks, and background task delivery still expose coarse delivery status enums plus raw error text, so the cross-surface normalized delivery-reason request remains valid and is not fully covered by the adjacent open trackers.

## What This Changes

The issue requests structured, normalized delivery failure reason codes across cron runs, messaging delivery, JSON output, and operator UI surfaces.

## Best Possible Solution

Define an additive shared delivery reason field while preserving existing status/error fields, then carry it through cron persistence, Gateway protocol, plugin hooks, task delivery, CLI/UI rendering, docs, and focused regression tests.

## Reproduction Assessment

Yes for the current gap: source inspection shows cron, Gateway cron protocol, Control UI, plugin hooks, and task delivery still expose coarse delivery statuses plus raw deliveryError/error text. This is not a runtime crash reproduction; it is a high-confidence source-level verification that the requested normalized reason field is absent across the relevant surfaces.

## Solution Assessment

Unclear as an implementation plan. The direction is sound, but the durable reason-code names, persistence shape, protocol compatibility, and relationship to https://github.com/openclaw/openclaw/issues/49190 and https://github.com/openclaw/openclaw/issues/74095 need maintainer agreement before a narrow fix PR is safe.
