## Summary

Keep open. Current main still exposes cron/task/message delivery through separate coarse status enums and raw error text; the related open trackers cover adjacent cron warning and task taxonomy work but do not replace this cross-surface delivery-reason request.

## What This Changes

The issue requests normalized, structured delivery failure reason codes across cron runs, messaging delivery, UI/status cards, and JSON output.

## Best Possible Solution

Define an additive shared delivery reason field while preserving existing status fields, then carry it through cron persistence, Gateway protocol, plugin/task delivery, CLI/UI rendering, docs, and focused regression tests.

Not applicable as a strict bug reproduction because this is a feature/API taxonomy request. Source inspection on current main gives high-confidence verification that the requested normalized reason model is still missing from cron, task, message delivery, protocol, and UI surfaces.

## Solution Assessment

Unclear. The direction is sound, but the best fix requires maintainer agreement on reason-code names, persistence shape, protocol compatibility, and how it composes with the adjacent cron-warning and task-taxonomy issues.
