## Summary

Current main still has the reported Slack interaction routing gap: block-action events carry `threadTs` in the payload but enqueue against the base Slack session key, and the callback path still only enqueues a system event rather than proving an end-to-end thread wake. The issue should stay open while the overlapping wake PR and the closed unmerged thread-key fix are reconciled.

## What This Changes

The issue requests reliable Slack interactive button handling that wakes the correct agent session and replies in the originating Slack thread, including thread-aware `block_actions` session-key resolution.

## Best Possible Solution

Keep the issue open until Slack block actions carry `threadTs` into session-key resolution and the wake path produces one agent turn for the matching thread session, with tests covering threaded DM/channel clicks and no public Plugin SDK expansion unless multiple plugins need the seam.

## Reproduction Assessment

Yes. The issue has a concrete Socket Mode flow, and current main provides a deterministic source-level reproduction: a payload with `container.thread_ts` still resolves the system event against the base session key because the resolver never receives `threadTs`.

## Solution Assessment

Unclear. A narrow Slack-owned thread-key fix is necessary, but the maintainable path should be coordinated with #67751's broader proactive wake work so button clicks are not handled by two competing dispatch paths.
