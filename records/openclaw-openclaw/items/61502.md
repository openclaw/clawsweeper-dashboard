## Label Justifications

- `P2`: The issue describes a real Slack workflow break with limited channel-specific blast radius, now fixed in the shipped Slack interaction path.
- `impact:session-state`: The failure routed button-click events to the wrong base/thread session key, causing session state drift for Slack thread interactions.
- `impact:message-loss`: The visible Slack click could be acknowledged while the agent never processed or replied to the originating thread.

## Summary

Current main and the v2026.5.19 release now cover the reported Slack interaction failure: block-action clicks carry `threadTs` into session resolution, enqueue with Slack delivery context, and request an immediate targeted heartbeat so the queued event can continue the visible thread conversation.

## What This Changes

Slack block-action clicks should route to the originating Slack thread session, wake that session, and produce a visible continuation instead of being queued passively.

## Best Possible Solution

Keep the shipped Slack interaction wake/thread-routing fix, and let broader wrong-thread reports continue under their narrower follow-up items if fresh latest-main evidence appears.

## Reproduction Assessment

No current-main failing reproduction remains. The historical Socket Mode repro was concrete, and current main now carries `threadTs` into session resolution and requests a heartbeat after enqueue, with regression coverage.

## Solution Assessment

Yes. The landed fix is the narrow Slack-owned path change: preserve plugin/approval early returns while adding thread-aware routing, delivery context, and immediate wake for generic reply-button events.
