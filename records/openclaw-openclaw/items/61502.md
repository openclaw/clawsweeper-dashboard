## Summary

Current main implements the reported Slack interaction path: block actions now carry `threadTs` into session resolution, enqueue Slack delivery context, and request an immediate heartbeat for the resolved session. The fix landed in merged PR https://github.com/openclaw/openclaw/pull/79836 and is included in pre-release v2026.5.10-beta.1.

## What This Changes

Slack interactive button/select clicks should wake the correct target session and reply in the originating Slack thread.

## Best Possible Solution

Keep the merged Slack interaction wake/thread-routing implementation and handle older overlapping Slack interaction PRs as superseded in their own reviews.

## Reproduction Assessment

No, not against current main: the original Socket Mode flow was source-reproducible before the fix, but current code now passes `threadTs`, stores Slack delivery context, and requests a heartbeat wake for the resolved session.

## Solution Assessment

Yes. The merged implementation is the narrow Slack-owned path: it keeps interactions as system events, resolves the target thread session, and wakes that session without adding a new user-facing mode or config option.
