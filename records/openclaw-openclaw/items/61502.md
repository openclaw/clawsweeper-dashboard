- `impact:message-loss`: The issue was about Slack button-click events being received by the gateway but not delivered to the agent turn.

## Summary

Current main and the v2026.5.18 release already contain the Slack block-action routing fix: interaction clicks now carry `threadTs`, enqueue Slack delivery context, and request an immediate heartbeat for the resolved session. The merged fix explicitly covered this issue in the release changelog, so this can close as implemented.

## What This Changes

Request that Slack interactive button/select clicks reliably wake the correct Slack session and reply in the originating thread.

## Best Possible Solution

Keep the shipped Slack-owned implementation from https://github.com/openclaw/openclaw/pull/79836; any remaining Slack button failures should be reported with current-version live proof as narrower follow-ups.

## Reproduction Assessment

No current-main failing reproduction remains. The historical Slack Socket Mode flow was concrete, and current source plus tests show the former `threadTs`/wake gap has been fixed.

## Solution Assessment

Yes. The merged Slack monitor fix is the narrow maintainable path: it keeps the event in Slack-owned code, resolves the thread session, carries delivery context, and wakes only after a queued event.
