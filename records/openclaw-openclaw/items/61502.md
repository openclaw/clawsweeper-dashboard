- `impact:message-loss`: The reported failure dropped or suppressed the visible continuation after a Slack button click.
- `impact:session-state`: The root cause routed clicks to the base session instead of the active thread-scoped session.

## Summary

Close as implemented: Saicheg's follow-up repro identified the missing Slack `threadTs`/wake path, and current main plus `v2026.5.18` now carry `threadTs`, preserve Slack delivery context, and request the target session wake for interactive reply clicks.

## What This Changes

Slack interactive reply button/select clicks should wake the correct Slack session and continue in the originating thread.

## Best Possible Solution

Keep the shipped Slack monitor implementation as the canonical fix, and use narrower follow-up issues only for fresh latest-main Slack wrong-thread logs or modal-lifecycle wake gaps.

## Reproduction Assessment

Yes for the historical bug: the reporter supplied a concrete Socket Mode flow, and the pre-fix source path omitted `threadTs` and did not wake the resolved session. Current main no longer matches that failing path.

## Solution Assessment

Yes. The merged Slack-owned fix is the narrow maintainable solution because it preserves the existing interaction event path while carrying the thread target and waking only after a new event is queued.
