## Summary

Current main still has the reported Slack interaction routing gap: block-action payloads preserve `threadTs`, but the session-key resolver is still called without it and the system-event path still queues passively instead of proving a wake into the target thread session.

## What This Changes

The issue requests reliable Slack interactive button handling that routes clicks to the correct agent session/thread and produces a follow-up turn in the originating Slack thread.

## Best Possible Solution

Slack block actions should resolve the same thread session key as normal inbound Slack thread messages and trigger exactly one wake/turn for that session, with focused threaded DM/channel coverage.

## Reproduction Assessment

Yes. The provided Socket Mode flow is concrete, and current source gives a deterministic reproduction: `container.thread_ts` is parsed but not passed into session-key resolution, so queued events can target the base session instead of the thread session.

## Solution Assessment

Unclear. A narrow Slack-owned thread-key and wake fix is necessary, but maintainers should reconcile #61374 and #67751 so there is one canonical interaction dispatch path.
