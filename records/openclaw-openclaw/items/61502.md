## Label Justifications

- `P2`: The item concerns a Slack-specific interaction workflow with a real but channel-scoped user impact.
- `impact:message-loss`: Before the fix, Slack could receive and acknowledge a button click while OpenClaw failed to deliver that event into the agent turn.
- `impact:session-state`: The failure mode was a session/thread routing mismatch between the clicked Slack thread and the OpenClaw session key.

## Summary

Current main and the v2026.5.18 release now implement the Slack interaction wake/thread-routing path described here, with the merged Slack fix carrying threadTs into session resolution, preserving Slack delivery context, and requesting an immediate heartbeat wake after a queued interaction.

## What This Changes

Slack interactive button clicks should wake the correct Slack session/thread and continue the originating Slack conversation.

## Best Possible Solution

Keep the shipped Slack-owned implementation from https://github.com/openclaw/openclaw/pull/79836; broader intermittent wrong-thread reports should continue in https://github.com/openclaw/openclaw/issues/75969 with fresh logs if they still reproduce.

## Reproduction Assessment

Yes. The historical repro was clear from Saicheg's Slack Socket Mode report and the old source path; current main no longer has that missing threadTs/no-wake path because the merged fix threads it through and wakes the target session.

## Solution Assessment

Yes. The shipped fix is the narrow owner-boundary solution: keep Slack block-action ingestion in the Slack plugin, reuse system-event queues, add delivery context, and request a targeted heartbeat wake only after enqueue succeeds.
