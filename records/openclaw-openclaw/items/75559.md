## Summary

Keep open. Current main still misses the snake_case Discord thread metadata fallback, and the PR is a narrow, correct fix for that behavior; the remaining work is maintainer review, rebase onto current main, and exact-head CI rather than conservative cleanup.

## What This Changes

The PR updates Discord channel-access helpers to read `owner_id` and `parent_id` from direct and `rawData` snake_case payload fields, adds focused regression tests, and adds an Unreleased changelog entry.

## Best Possible Solution

Keep the normalization in `extensions/discord/src/monitor/channel-access.ts`, preserve the focused helper tests and changelog entry, rebase onto current main while keeping the current unexported `DiscordChannelInfoSafe` shape, then land after exact-head CI is green.

## Reproduction Assessment

Yes. On current main, a thread-like object with `{ owner_id: "bot-id", parent_id: "parent-id" }` or `{ rawData: { owner_id: "bot-id", parent_id: "parent-id" } }` will return undefined owner/parent metadata because the helper reads only camelCase properties.

## Solution Assessment

Yes. Centralizing the alias fallback in `channel-access.ts` is the narrowest maintainable fix because the same helper feeds preflight mention gating, fetched channel context, inbound job normalization, and message-channel info.
