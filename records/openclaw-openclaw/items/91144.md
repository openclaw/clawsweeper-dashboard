## Best Possible Solution

Fix the Windows Scheduled Task launch proof so install/start/restart distinguish the managed task or generated script process from an unrelated foreground gateway listener, then prove the flow on native Windows.

## Repair Work Prompt

Fix the Windows native gateway Scheduled Task flow reported at https://github.com/openclaw/openclaw/issues/91144. Current main's `src/daemon/schtasks.ts` can treat an already-running foreground gateway listener on port 18789 as launch evidence after `schtasks /Run`, which can skip the fallback launcher and leave no managed gateway after the foreground window closes. Keep the fix inside the Windows daemon/Scheduled Task ownership boundary: make launch verification prove the managed task script/process or a post-run managed process started, not merely that any foreground gateway listener exists. Add focused regression coverage in `src/daemon/schtasks.startup-fallback.test.ts` for installing/starting while a foreground gateway listener is already present and `schtasks` remains not-yet-run. Do not add config flags, change docs unless behavior or diagnostics change, or rewrite unrelated daemon service code. Validate with the focused daemon tests and, when available, a native Windows/Parallels smoke that runs a foreground gateway, installs/starts the service, closes the foreground process, and confirms `openclaw gateway status --deep --require-rpc` remains healthy.
