## Summary

Keep this PR open: current main still lacks a session-level model-error circuit breaker, but this branch has blocking correctness and design issues around cron error classification, interactive reply coverage, and isolated heartbeat session ownership.

## What This Changes

The PR adds a configurable agent/session circuit-breaker module, schema and session-state fields, docs, tests, and pause/reset/alert handling for heartbeat and isolated cron paths.

## Best Possible Solution

Keep a PR or replacement open until OpenClaw has a documented core session breaker with generated config/schema alignment, persisted per-session state, structured model-failure signals across interactive replies, heartbeat, cron, and auto-resume paths, and correct ownership semantics for isolated versus base sessions.

## Reproduction Assessment

Yes. Source inspection of current main shows no session-level `circuitBreaker` config/state/enforcement, and the PR-specific defects are statically reproducible from the proposed diff and current heartbeat/cron/reply code paths.

## Solution Assessment

No. The proposed direction is useful, but the current patch is not the best maintainable solution until it uses structured model-error metadata and consistently applies the breaker across all session-owning run paths with correct isolated-session ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Count cron error payloads before clearing breaker state:** `src/cron/isolated-agent/run.ts:787-794`
  - body: This clears the breaker whenever `finalRunResult.meta?.error` is absent, but cron can still return fatal timeout or error payloads that are converted into job errors later. Those failures would reset the accumulated counter, so repeated timeouts never trip the breaker.
  - confidence: 0.86
- **[P2] Wire the interactive reply path or narrow the design:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:81-87`
  - body: The design lists `getReplyFromConfig()` as a required integration point, but the patch only wires heartbeat and isolated cron. Normal inbound replies can hit the same repeated model-error loop and will bypass pause/reset/alert entirely.
  - confidence: 0.82
- **[P2] Avoid resetting the main session for isolated heartbeat failures:** `src/infra/heartbeat-runner.ts:746-754`
  - body: When `heartbeat.isolatedSession` is enabled, the model call runs against the synthetic heartbeat session, but this block records and executes actions against the base `entry` and `sessionKey`. A breaker `reset` can rotate the user's real conversation because only an isolated heartbeat run failed.
  - confidence: 0.76
- **[P2] Use structured error state instead of warning-prefix matching:** `src/infra/heartbeat-runner.ts:742-744`
  - body: Heartbeat error detection depends on `replyPayload.text` starting with the warning glyph. Current reply normalization can prepend a configured response prefix, and `ReplyPayload` already carries `isError`, so prefixed model-error replies can miss the breaker while warning-style text can be counted incorrectly.
  - confidence: 0.72
