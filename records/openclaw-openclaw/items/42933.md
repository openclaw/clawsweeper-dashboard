## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Record isolated heartbeat failures on the isolated session:** `src/infra/heartbeat-runner.ts:747-753`
  - body: When `heartbeat.isolatedSession` is enabled, the model run uses `runSessionKey`, but this breaker path records errors and executes reset/pause against the preflight `entry` and base `sessionKey`. A failed ephemeral heartbeat can therefore reset or pause the user's real conversation instead of the isolated heartbeat session.
  - confidence: 0.93
- **[P1] Count fatal timeout payloads before clearing breaker state:** `src/cron/isolated-agent/run.ts:779-794`
  - body: This branch only records `meta.error`; if the embedded runner returns a fatal timeout as `payloads[].isError` without `meta.error`, cron later reports the run as an error but the breaker state has already been cleared. Repeated timeout failures can then never reach the configured threshold.
  - confidence: 0.91
- **[P2] Wire the breaker into normal replies or narrow the design:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:81-87`
  - body: The added design doc lists `getReplyFromConfig()` as a required integration point, and the linked feature includes normal messages, but this PR only wires heartbeat and cron. Normal inbound/auto-reply sessions would still bypass the session breaker unless this path is implemented or the feature scope is explicitly narrowed.
  - confidence: 0.84
- **[P2] Clear heartbeat dedupe fields during breaker resets:** `src/agents/circuit-breaker/state.ts:126-142`
  - body: `resetSession` is documented as `/new`-equivalent, but it leaves `lastHeartbeatText` and `lastHeartbeatSentAt` in place. After a reset action, duplicate suppression can drop the first successful heartbeat if it repeats the previous text, even though the session was supposed to start fresh.
  - confidence: 0.86
