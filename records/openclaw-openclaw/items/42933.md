Keep open. The branch is a plausible implementation candidate for the open session-health breaker work, but current main does not contain this configurable pause/reset/alert feature, and the PR still has blocking correctness issues plus no after-fix real behavior proof.

## What This Changes

The PR adds a session circuit-breaker module, config/schema fields, session `cb*` state, heartbeat and cron integration, design docs, and unit tests for pause/reset/alert behavior after consecutive model errors.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/62615 to decide the durable session-health contract, then either revise this branch or replace it with a smaller implementation that integrates with current session suspension/failover state, uses structured failure signals, handles isolated heartbeat sessions correctly, and includes redacted runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Record isolated heartbeat failures on the isolated session:** `src/infra/heartbeat-runner.ts:747-753`
  - body: When `heartbeat.isolatedSession` is enabled, the model run uses `runSessionKey` (`<session>:heartbeat`), but this breaker path still records errors and executes reset/pause against the preflight `entry` and `sessionKey`, which are the user's main session. A failed ephemeral heartbeat can therefore reset the real conversation or pause future heartbeats based on the wrong state.
  - confidence: 0.93
- **[P1] Count fatal timeout payloads before clearing cron breaker state:** `src/cron/isolated-agent/run.ts:779-794`
  - body: This branch only records `meta.error`; if `runEmbeddedPiAgent` returns a fatal timeout as an `isError` payload without `meta.error`, cron later reports the run as an error but the breaker state has already been cleared here. That misses repeated model timeouts, which are a primary target of the feature.
  - confidence: 0.9
- **[P2] Wire the breaker into the normal reply path or narrow the design:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:87-91`
  - body: The added design doc lists `getReplyFromConfig()` as an integration point, and the linked feature asks to stop repeated failures on messages as well as heartbeat/cron retries, but this PR only wires heartbeat and cron. Normal inbound/auto-reply sessions will keep bypassing the new session breaker unless this path is implemented or the scope is explicitly narrowed.
  - confidence: 0.82
- **[P2] Clear heartbeat dedupe fields when resetting a session:** `src/agents/circuit-breaker/state.ts:126-142`
  - body: `resetSession` is described as `/new`-equivalent, but it leaves `lastHeartbeatText` and `lastHeartbeatSentAt` on the entry. Heartbeat duplicate suppression can then drop the first successful heartbeat after a breaker reset if it repeats the previous text, even though the session was supposed to start fresh.
  - confidence: 0.86
