## Summary

Keep open. The PR is a plausible implementation attempt for session-level circuit breaking, but the current branch is incomplete for the advertised session-level behavior, has concrete heartbeat/cron correctness blockers, lacks real behavior proof, and overlaps the broader open product discussion in https://github.com/openclaw/openclaw/issues/62615.

## What This Changes

The PR adds a configurable agent circuit breaker with persisted session `cb*` fields and pause/reset/alert handling for heartbeat and isolated cron runs.

## Best Possible Solution

Define the durable session-health contract under the canonical gateway/session lifecycle work, then either salvage this PR after a rebase and blocker fixes or replace it with a narrower implementation tied to https://github.com/openclaw/openclaw/issues/62615.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR blockers by source inspection: the branch omits the normal reply integration, records isolated heartbeat failures on the base session, and clears cron breaker state for timeout payloads without `meta.error`. No live fleet reproduction was run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet. A session-level breaker is directionally reasonable, but this implementation needs structured error metadata across reply paths, correct isolated-session ownership, public config/docs alignment, and maintainer scope decisions before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire the breaker into normal reply runs:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:87-91`
  - body: The design lists `getReplyFromConfig()` as one of the required integration points, but the patch never changes that path. Normal interactive/auto-reply model failures therefore never call the breaker check or recorder, so the advertised session-level breaker only protects heartbeat/cron runs and leaves user-triggered stuck sessions retrying.
  - confidence: 0.9
- **[P1] Record isolated heartbeat state on the run session:** `src/infra/heartbeat-runner.ts:747-753`
  - body: When `heartbeat.isolatedSession` is enabled, the model turn runs under `runSessionKey` (`<base>:heartbeat`), but this code records errors and executes reset/pause against the base `entry` and `sessionKey`. A breaker reset can rotate the user's real conversation for an ephemeral heartbeat failure, and pause state is stored on the wrong session.
  - confidence: 0.92
- **[P1] Count timeout payloads before clearing cron breaker state:** `src/cron/isolated-agent/run.ts:779-794`
  - body: This branch treats only `meta.error` as a non-throwing model failure and clears breaker state otherwise. `runEmbeddedPiAgent` can return a fatal timeout as an `isError` payload without `meta.error`, after which cron still returns an error, so repeated timeout failures never reach the breaker threshold.
  - confidence: 0.86
- **[P2] Avoid emoji-prefix model error detection:** `src/infra/heartbeat-runner.ts:745-746`
  - body: Classifying heartbeat model failures with `replyText.startsWith("⚠️")` is brittle because reply normalization can prepend a configured response prefix, producing `<prefix> ⚠️ ...` for real fallback errors. Those deployments will miss breaker increments, while any legitimate reply that starts with that glyph can be counted as a model failure.
  - confidence: 0.82
