## Summary

Keep open: current main does not implement the requested session-level breaker, and the PR still needs maintainer product and ownership review because the proposed patch has correctness gaps across cron, heartbeat, and interactive reply semantics.

## What This Changes

The PR adds an agent circuit-breaker config/state module, `cb*` session fields, schema/docs/tests, and heartbeat/cron pause/reset/alert integration.

## Best Possible Solution

Define a structured model-failure signal and session-ownership contract, then implement pause/reset/alert consistently across heartbeat, cron, auto-resume, and interactive replies with generated config/docs alignment.

## Reproduction Assessment

Yes for review purposes. Source inspection shows current main lacks the requested session-level breaker, and the PR-specific defects are statically reproducible from the proposed diff and current heartbeat, cron, and reply outcome code paths.

## Solution Assessment

No. The direction is useful, but this patch is not the best maintainable solution until it uses structured error metadata, covers interactive reply paths, and applies breaker state to the correct session owner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Count cron fatal payloads before clearing breaker state:** `src/cron/isolated-agent/run.ts:791`
  - body: This clears the breaker whenever `meta.error` is absent, but cron still treats fatal payloads, failure signals, and denial signals as run errors later in `resolveCronPayloadOutcome`. Those failures would reset the accumulated counter, so repeated fatal payload-only cron failures never trip the breaker.
  - confidence: 0.86
- **[P2] Record isolated heartbeat failures on the isolated session:** `src/infra/heartbeat-runner.ts:750-752`
  - body: When `heartbeat.isolatedSession` is enabled, the model run uses a synthetic `:heartbeat` session, but this action block still mutates and executes against the base entry and session key. A breaker `reset` or `pause` can therefore affect the user's real conversation because only the isolated heartbeat run failed.
  - confidence: 0.78
- **[P2] Wire the interactive reply path or narrow the design:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:81-87`
  - body: The design lists `getReplyFromConfig()` as a required integration point, but the patch only wires heartbeat and isolated cron. Normal inbound replies can hit the same repeated model-error loop and will bypass pause/reset/alert entirely.
  - confidence: 0.82
