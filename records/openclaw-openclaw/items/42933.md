## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Record isolated heartbeat failures on the isolated session:** `src/infra/heartbeat-runner.ts:747-753`
  - body: When `heartbeat.isolatedSession` is enabled, the model run uses `runSessionKey` (`<session>:heartbeat`), but this branch records errors and executes reset/pause against the preflight `entry` and `sessionKey`, which are the user's main session. A failed ephemeral heartbeat can therefore reset the real conversation or pause future heartbeats based on the wrong state.
  - confidence: 0.93
- **[P1] Count fatal timeout payloads before clearing cron state:** `src/cron/isolated-agent/run.ts:792-794`
  - body: This branch clears the breaker whenever `meta.error` is absent, but the embedded runner can return a fatal `isError` payload without `meta.error`. Cron later treats that run as an error, so repeated timeout/no-response payload failures can keep clearing the breaker instead of reaching the threshold.
  - confidence: 0.9
- **[P2] Wire the breaker into normal replies or narrow the design:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:83-87`
  - body: The added design lists `getReplyFromConfig()` as an integration point, but the branch only wires heartbeat and cron. Normal inbound/auto-reply sessions will keep bypassing the new session breaker unless this path is implemented or the feature scope is explicitly narrowed.
  - confidence: 0.82
- **[P2] Clear heartbeat dedupe fields when resetting a session:** `src/agents/circuit-breaker/state.ts:126-142`
  - body: `resetSession` is documented as `/new`-equivalent, but it leaves `lastHeartbeatText` and `lastHeartbeatSentAt` on the entry. Heartbeat duplicate suppression can then drop the first successful heartbeat after a breaker reset if it repeats the previous text, even though the session was supposed to start fresh.
  - confidence: 0.86
