## Summary

Keep open: current main still does not provide a configurable session-level circuit breaker, and this PR is a plausible implementation path, but the proposed patch is not merge-ready because it misses important run paths and can apply breaker state to the wrong session owner.

## What This Changes

The PR adds a configurable agent/session circuit-breaker module, config/schema/session-state fields, docs, tests, and pause/reset/alert handling for heartbeat and isolated cron paths.

## Best Possible Solution

Land a revised core resilience design only after it consistently tracks structured model-failure outcomes across interactive replies, heartbeat, cron, and auto-resume paths, with clear session ownership and generated config/docs alignment.

## Reproduction Assessment

Yes for review purposes. Source inspection shows current main lacks the requested session-level breaker, and the PR-specific defects are statically reproducible from the proposed diff plus current heartbeat, cron, and reply outcome code paths.

## Solution Assessment

No. The direction is useful, but this patch is not the best maintainable solution until it uses structured error metadata, covers interactive reply paths, and applies breaker state to the correct session owner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Count cron fatal payloads before clearing breaker state:** `src/cron/isolated-agent/run.ts:791`
  - body: This clears the breaker whenever `meta.error` is absent, but cron still treats `payloads[].isError`, fatal failure signals, and denial signals as run errors later in `resolveCronPayloadOutcome`. Those failures would reset the accumulated counter, so repeated fatal payload-only cron failures never trip the breaker.
  - confidence: 0.86
- **[P2] Record isolated heartbeat failures on the isolated session:** `src/infra/heartbeat-runner.ts:750-752`
  - body: When `heartbeat.isolatedSession` is enabled, the model run uses a synthetic `:heartbeat` session, but this action block still mutates and executes against the base `entry` and `sessionKey`. A breaker `reset` or `pause` can therefore affect the user's real conversation because only the isolated heartbeat run failed.
  - confidence: 0.78
- **[P2] Wire the interactive reply path or narrow the design:** `docs/superpowers/specs/2026-03-11-session-circuit-breaker-design.md:81-87`
  - body: The design lists `getReplyFromConfig()` as a required integration point, but the patch only wires heartbeat and isolated cron. Normal inbound replies can hit the same repeated model-error loop and will bypass pause/reset/alert entirely.
  - confidence: 0.82
