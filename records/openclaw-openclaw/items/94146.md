## Label Justifications

- `P2`: This is a normal-priority skills runtime bug with limited blast radius: affected skills can miss configured env/API key injection when restored from snapshots.
- `impact:session-state`: The issue is about `SkillSnapshot` losing the resolved skill identity needed for later runtime behavior.
- `impact:auth-provider`: The observable failure is that configured skill API keys or env overrides under `skills.entries.<skillKey>` are not injected at runtime.

## Summary

This is still necessary: current main and the latest release both keep snapshot env override lookup keyed by `skill.name`, while the documented/configured skill key path uses `metadata.openclaw.skillKey`. The issue should stay open because two live PRs already reference it with closing links and are the right place to finish the fix.

## What This Changes

The issue requests preserving the resolved skill key in skill snapshots so snapshot-based env/API key overrides can resolve `skills.entries.<skillKey>` when `name !== skillKey`.

## Best Possible Solution

Land one focused fix that persists optional `skillKey` in skill/session snapshots, uses `skill.skillKey ?? skill.name` for snapshot config lookup, and covers both `name !== skillKey` and older snapshots without `skillKey`.

## Reproduction Assessment

Yes, from source inspection: docs/config paths use `skillKey`, but current snapshots only carry `name` and snapshot env overrides look up `skills.entries` by `skill.name`. A focused regression test with `name: "Pretty Weather"` and `skillKey: "weather"` should fail on current main without needing a new product decision.

## Solution Assessment

Yes, the proposed repair shape is the narrow maintainable fix: carry the already-resolved key into snapshots and fall back to `name` for older snapshots. Because open PRs already target this issue, the best path is to review and land one of them rather than opening a separate repair lane now.
