## Label Justifications

- `P2`: This is a normal-priority skills runtime bug with limited blast radius to skills configured under a source-specific `skillKey`.
- `impact:session-state`: The bug is caused by persisted skill snapshot state losing metadata needed for later runtime behavior.
- `impact:auth-provider`: The bug can prevent configured skill API keys or env credentials under `skills.entries.<skillKey>` from being injected at runtime.

## Summary

Keep open: current `main` still drops the resolved skill key before snapshot env injection, and live GitHub state shows open linked closing PRs, with https://github.com/openclaw/openclaw/pull/94163 currently the stronger reviewed fix path.

## What This Changes

The issue requests preserving the resolved `skillKey` in skill snapshots so snapshot-based env/API-key overrides can resolve `skills.entries.<skillKey>` instead of only the display name.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/94163 or an equivalent narrow fix, then close this issue and redirect or close the narrower duplicate PRs as superseded.

## Reproduction Assessment

Yes, from source inspection: a snapshot for a skill whose display `name` differs from `metadata.openclaw.skillKey` is built without `skillKey` and later resolves config by display name. I did not run tests because this review is read-only.

## Solution Assessment

Yes: persisting an optional resolved `skillKey`, using `skillKey ?? name` for snapshot env lookup, and refreshing or safely handling older persisted snapshots is the narrow owner-boundary fix.
