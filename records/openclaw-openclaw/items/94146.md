## Label Justifications

- `P2`: This is a normal-priority skills runtime bug with limited blast radius to skills configured by a source-specific `skillKey`.
- `impact:session-state`: The issue is caused by persisted skill snapshot state losing metadata needed for later runtime behavior.
- `impact:auth-provider`: The issue can prevent configured skill API keys or env credentials under `skills.entries.<skillKey>` from being injected at runtime.

## Summary

Keep open: current main and the latest release still drop the resolved skill key before snapshot env injection, and open linked PRs already use closing syntax for this bug.

## What This Changes

The issue reports that `SkillSnapshot.skills[]` omits the resolved `skillKey`, so snapshot-based env/API-key overrides look up `skills.entries` by display name instead of `metadata.openclaw.skillKey`.

## Best Possible Solution

Land one canonical linked fix that stores optional `skillKey`, uses it for snapshot env lookup, handles existing snapshots safely, and then close this issue.

## Reproduction Assessment

Yes. Source inspection shows current main builds snapshots without the resolved `skillKey` and then resolves snapshot env config by display `name`; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Persisting an optional resolved `skillKey` in new snapshots and using `skillKey ?? name` for snapshot env lookup is the narrow owner-boundary fix, with upgrade-safe handling for older snapshots.
