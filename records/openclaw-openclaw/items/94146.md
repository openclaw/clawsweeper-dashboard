- `impact:auth-provider`: The issue affects `skills.entries` API-key/env and SecretRef-style credential resolution for skill runtime setup.

## Summary

Keep open: current main still builds snapshots without the resolved skill key and the snapshot env override path still resolves config by `skill.name`, while multiple open PRs already reference this issue as their fix target.

## What This Changes

The issue requests preserving resolved skill keys in `SkillSnapshot` so snapshot-based skill env/API-key overrides can resolve `skills.entries.<skillKey>` entries.

## Best Possible Solution

Land one canonical fix that persists optional `skillKey` on runtime and persisted skill snapshots, populates it with `resolveSkillKey()`, uses `skill.skillKey ?? skill.name` for snapshot env lookup, and covers `name !== skillKey` with regression tests.

## Reproduction Assessment

Yes from source inspection: construct a snapshot whose skill name differs from its configured `skillKey`, store the API key under `skills.entries.<skillKey>`, and current main resolves config by `skill.name`, leaving the env var unset. I did not run tests because this review is read-only, but the failing path is direct in the current source.

## Solution Assessment

Yes: preserving the resolved key in the snapshot and falling back to `skill.name` for older snapshots is the narrow maintainable fix. The open PR that also updates the persisted session snapshot type and adds producer/consumer regression tests looks like the best candidate to review first.
