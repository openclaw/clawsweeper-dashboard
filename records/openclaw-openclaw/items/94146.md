## Label Justifications

- `P2`: This is a normal-priority skills runtime bug with limited blast radius to skills whose configured key differs from their display name.
- `impact:session-state`: The issue is caused by persisted/lightweight skill snapshot state losing metadata needed for later runtime behavior.
- `impact:auth-provider`: The bug can prevent configured skill API keys or env credentials under `skills.entries.<skillKey>` from being injected at runtime.

## Summary

Keep open: current `main` and the latest release still lose the resolved skill key before snapshot env injection, and two open linked PRs use closing syntax for this issue, so the issue should stay open until one fix lands.

## What This Changes

The issue asks for `SkillSnapshot.skills[]` to preserve the resolved `skillKey` so snapshot env overrides can resolve `skills.entries.<skillKey>` instead of only the display name.

## Best Possible Solution

Land one narrow linked fix that adds optional `skillKey` to runtime and persisted snapshot shapes, populates it with `resolveSkillKey()`, uses `skill.skillKey ?? skill.name` for snapshot env lookup, and covers producer, consumer, and older-snapshot fallback behavior.

## Reproduction Assessment

Yes, from source inspection: a snapshot for a skill whose display `name` differs from `metadata.openclaw.skillKey` cannot resolve `skills.entries.<skillKey>` because the snapshot path only has and uses `skill.name`. I did not run tests because this cleanup review is read-only.

## Solution Assessment

Yes. The requested shape is the narrow maintainable fix: preserve an optional resolved key in snapshots and keep the older `skill.name` fallback for existing snapshots while adding focused regression coverage.
