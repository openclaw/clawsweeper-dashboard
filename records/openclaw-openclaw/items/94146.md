## Label Justifications

- `P2`: This is a normal-priority skills runtime bug with limited blast radius to skills configured by a source-specific `skillKey`.
- `impact:auth-provider`: The issue can prevent configured skill API keys or env credentials under `skills.entries.<skillKey>` from being injected at runtime.
- `impact:session-state`: The issue is caused by lightweight skill snapshot state losing metadata needed for later runtime behavior.

## Summary

Keep open: current `main` still loses the resolved skill key before snapshot env injection, and two open linked PRs use closing syntax for this issue, so the issue should remain open until one fix merges.

## What This Changes

The issue reports that `SkillSnapshot.skills[]` stores only the display name, causing snapshot env override lookup to miss config stored under `skills.entries.<skillKey>`.

## Best Possible Solution

Land one narrow linked fix that preserves `skillKey` in snapshots, uses it for snapshot env override lookup, keeps older snapshots safe, and covers producer, consumer, and reusable-snapshot paths with regression tests.

## Reproduction Assessment

Yes from source: a snapshot for a skill whose display name differs from `metadata.openclaw.skillKey` cannot resolve `skills.entries.<skillKey>` because the snapshot path has only `skill.name`. I did not run tests because this cleanup review is read-only.

## Solution Assessment

Yes. The maintainable fix is to persist an optional resolved `skillKey` in new snapshots and use it for snapshot env override lookup while handling older reusable snapshots safely.
