## Summary

Keep open: current main and the latest release still build skill snapshots without a resolved skill key and still resolve snapshot env overrides by `skill.name`, while two open PRs are candidate fixes that use closing syntax for this issue. The issue should remain open until one canonical fix merges, especially because the richer candidate has active discussion about existing persisted snapshots that lack `skillKey`.

## What This Changes

The issue reports that `SkillSnapshot.skills[]` does not retain the resolved skill key, so snapshot-based env override lookup can miss `skills.entries.<skillKey>` config entries.

## Best Possible Solution

Land one canonical fix that persists `skillKey`, uses it for snapshot env override lookup, and refreshes or hydrates older persisted snapshots that lack it; then close duplicate fix PRs after merge.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the path: current main writes snapshots without `skillKey`, then `applySkillEnvOverridesFromSnapshot()` resolves config by display name while docs and loading/status paths use the resolved skill key.

## Solution Assessment

Yes directionally. The maintainable fix is the suggested snapshot shape and lookup change, with one added upgrade safeguard for existing persisted snapshots that predate `skillKey`.
