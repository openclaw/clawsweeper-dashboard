- `impact:session-state`: The issue is caused by persisted skill snapshot state losing metadata needed for later runtime behavior.
- `impact:auth-provider`: The issue can prevent configured skill API keys or env credentials under `skills.entries.<skillKey>` from being injected at runtime.

## Summary

Keep open: current main still drops the resolved skill key before snapshot env injection, and two open linked PRs use closing syntax for this issue, so the issue should remain open until one fix merges.

## What This Changes

The issue reports that skill snapshots persist only the display name, causing snapshot-based env/API-key overrides configured under `skills.entries.<skillKey>` to be missed.

## Best Possible Solution

Land one narrow linked fix that persists optional `skillKey`, uses it for snapshot env override lookup, refreshes or safely handles older snapshots, and includes focused producer/consumer/upgrade coverage.

## Reproduction Assessment

Yes from source inspection: a snapshot for a skill whose display `name` differs from `metadata.openclaw.skillKey` looks up `skills.entries` by display name on current main and misses the configured env/API key. I did not run tests because this review is read-only.

## Solution Assessment

Yes: persisting an optional resolved `skillKey` in new snapshots and using it for snapshot env lookup is the narrow owner-boundary fix. The open work is to land one linked candidate with upgrade-path proof, not to create a separate implementation lane.
