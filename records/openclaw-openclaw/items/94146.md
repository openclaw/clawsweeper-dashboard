## Label Justifications

- `P2`: This is a normal-priority skills runtime bug with limited blast radius to snapshot-backed skills configured by source-specific `skillKey`.
- `impact:session-state`: The bug is caused by persisted skill snapshot state losing metadata needed for later runtime behavior.
- `impact:auth-provider`: The bug can prevent configured skill API keys or SecretRef-backed env credentials under `skills.entries.<skillKey>` from being injected at runtime.

## Summary

Keep open: current main still drops the resolved skill key before snapshot env injection, and the two linked open PRs are candidate fixes rather than a reason to close the issue before one merges.

## What This Changes

The issue reports that skill snapshots drop `metadata.openclaw.skillKey`, so runtime snapshot env overrides miss configuration stored under `skills.entries.<skillKey>`.

## Best Possible Solution

Land the broader canonical fix or an equivalent patch that persists optional `skillKey`, uses `skillKey ?? name` for snapshot env lookup, handles old persisted snapshots safely, and includes focused producer/consumer/upgrade coverage.

## Reproduction Assessment

Yes. Source inspection is enough: current main builds snapshots without the resolved key and then resolves snapshot env config by display `name`, so config under `skills.entries.<skillKey>` is missed when the display name differs.

## Solution Assessment

Yes. Persisting an optional resolved `skillKey` in snapshots, using `skillKey ?? name` at lookup time, and refreshing or safely handling old persisted snapshots is the narrow maintainable fix.
