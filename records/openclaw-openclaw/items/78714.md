## Summary

Keep open: current main and the latest release still pass plain `resolveFetch(opts.fetch)` into provider usage hooks, and an open PR with closing syntax is already the active fix candidate for this exact proxy-aware usage path.

## What This Changes

The issue reports that Codex/OpenAI provider usage snapshots ignore configured HTTP(S)_PROXY env vars and return `fetch failed` when direct access to `chatgpt.com` is blocked.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/93943`, or an equally narrow replacement, so explicit fetch injection still wins and provider usage otherwise uses the existing env proxy helper before global fetch.

## Reproduction Assessment

Yes, from source: current main still resolves provider-usage fetch with plain `resolveFetch(opts.fetch)`, and the OpenAI/Codex hook passes that fetch into the `chatgpt.com` usage request. I did not run a live proxy-only network scenario in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is at the provider-usage loader boundary: preserve explicit fetch injection, then use `resolveProxyFetchFromEnv(env)`, then fall back to global fetch.
