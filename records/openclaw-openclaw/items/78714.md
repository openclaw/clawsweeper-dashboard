## Summary

Keep open: current `main` still resolves provider-usage fetches with plain `resolveFetch(opts.fetch)`, while the OpenAI/Codex usage hook passes that fetch into the `chatgpt.com` usage request. The linked repair PR at https://github.com/openclaw/openclaw/pull/78753 was closed unmerged, and the latest release still has the same loader behavior.

## What This Changes

The issue requests that Codex/OpenAI provider usage snapshots honor `HTTP_PROXY`/`HTTPS_PROXY` env proxy settings instead of returning `fetch failed` when direct access to `chatgpt.com` is blocked.

## Best Possible Solution

Land a focused loader fix that preserves explicit injected fetch precedence, uses `resolveProxyFetchFromEnv(env)` when no explicit fetch is supplied, and adds regression coverage for env proxy selection.

## Reproduction Assessment

Yes, from source: the current loader still chooses plain `resolveFetch(opts.fetch)`, and the OpenAI/Codex usage hook passes that fetch into the `chatgpt.com/backend-api/wham/usage` request. I did not run a live proxy-only network scenario.

## Solution Assessment

Yes, the best fix is the narrow loader-boundary repair: reuse the existing env proxy fetch helper when no explicit fetch is supplied, instead of adding a new config option or duplicating proxy logic in the Codex hook.
