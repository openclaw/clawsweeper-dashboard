## Summary

Keep open: current main still has the source-backed Codex provider-usage proxy gap, and an open linked PR already targets the same issue with closing syntax, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Codex provider usage loading honor HTTP_PROXY/HTTPS_PROXY environment proxy settings instead of returning fetch failed when direct access to chatgpt.com is blocked.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78753 or an equivalent narrow patch that prefers the existing env-proxy fetch helper only when no explicit fetch override is supplied, with regression coverage and redacted real proxy proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main passes plain resolveFetch(opts.fetch) into the Codex WHAM usage request even when proxy env settings are available. I did not establish a live network environment where direct fetch fails but proxy fetch succeeds.

## Solution Assessment

Is this the best way to solve the issue? Yes: reusing resolveProxyFetchFromEnv(env) when no explicit fetch override is provided is the narrow maintainable fix and preserves explicit-fetch precedence.
