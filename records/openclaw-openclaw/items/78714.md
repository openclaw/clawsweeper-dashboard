## Summary

Keep open: this is still a valid source-backed bug on current main, and the exact closing PR is open but unmerged. The loader still selects plain fetch for provider usage, while the Codex usage hook sends that fetch function to the ChatGPT WHAM usage endpoint.

## What This Changes

Codex provider usage should honor configured HTTP(S) proxy environment variables when loading usage windows instead of returning `fetch failed` from direct fetch.

## Best Possible Solution

Review and land the linked narrow provider-usage fix after it has adequate regression coverage, redacted real proxy usage proof, and green validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level, yes: invoke Codex provider usage with proxy env set and direct `chatgpt.com` egress blocked, and the loader still passes a plain fetch unless a fetch override is injected. I did not perform that live network setup in this review.

## Solution Assessment

Is this the best way to solve the issue? Yes: preferring an explicit fetch override, then an env-proxy fetch, then the normal fetch fallback inside `loadProviderUsageSummary` is the narrowest maintainable fix and keeps Codex endpoint parsing in the provider usage hook.
