## Summary

Keep open: current main still has a source-backed gap where provider usage fetch selection ignores the explicit env-proxy helper, and the Codex usage plugin sends that fetch function to the WHAM usage endpoint. The report is narrow and actionable, but I did not run a live blocked-direct-network proxy repro.

## What This Changes

The issue asks Codex provider usage snapshots in status and Control UI surfaces to honor `HTTP_PROXY`/`HTTPS_PROXY` env routing instead of returning `fetch failed` behind direct-network blocks.

## Best Possible Solution

Make provider usage loading choose a proxy-aware fetch from the supplied env when no explicit fetch override is provided, then add focused regression coverage and a changelog fix entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? We have a source-backed repro path and detailed reporter logs: invoke Codex provider usage with proxy env set and direct `chatgpt.com` egress blocked, then observe the loader still selects plain fetch unless an explicit fetch is injected. I did not establish a live current-main failure in this environment, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Is this the best way to solve the issue? Yes: preferring `opts.fetch`, then `resolveProxyFetchFromEnv(env)`, then normal fetch inside the provider-usage loader is the narrowest maintainable fix and preserves provider-owned Codex usage logic.
