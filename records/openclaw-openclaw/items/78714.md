## Summary

Keep open: current main and the latest release still choose plain fetch for provider-usage loading, and an open PR already tracks the exact proxy-aware repair with closing syntax. This issue should remain open until that PR lands or maintainers choose a different fix path.

## What This Changes

The issue reports that Codex/OpenAI provider usage fetches ignore HTTP(S)_PROXY env settings and fail with plain fetch in proxied networks.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93943, or an equally narrow replacement, so provider-usage loading preserves explicit fetch precedence and otherwise uses the existing env proxy helper before falling back to global fetch.

## Reproduction Assessment

Yes from source: current main still resolves provider-usage fetch with plain resolveFetch(opts.fetch), and the OpenAI/Codex hook passes that fetch into the chatgpt.com usage request. I did not run a live proxy-only network scenario in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is at the provider-usage loader boundary: preserve explicit fetch injection, then use resolveProxyFetchFromEnv(env), then fall back to global fetch; adding a new config knob or duplicating proxy logic in the Codex hook would be broader than needed.
