## Summary

Keep open: current main still has the reported source-level gap, and an open linked PR already references this issue with a proposed fix. The issue should stay open until that PR lands or is replaced, because provider usage still resolves plain fetch before handing the fetch function to the Codex WHAM usage request.

## What This Changes

Codex provider usage should honor HTTP_PROXY/HTTPS_PROXY proxy settings when fetching ChatGPT WHAM quota data for status surfaces.

## Best Possible Solution

Land the narrow loader fix through the linked PR or an equivalent replacement, with regression coverage and redacted real proxy usage proof, then close this issue through the merged fix.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main selects resolveFetch(opts.fetch) despite env proxy settings, then Codex usage calls the WHAM endpoint with that fetchFn. I did not run a live blocked-direct/proxy-enabled network repro.

## Solution Assessment

Yes, using the existing resolveProxyFetchFromEnv(env) only when no explicit fetch override is provided is the narrow maintainable fix. The current best path is to review and land the linked PR or an equivalent patch after proof and validation.
