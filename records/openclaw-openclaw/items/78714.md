## Summary

Keep open: current main and latest release still resolve provider-usage fetches with plain `resolveFetch(opts.fetch)`, while the OpenAI/Codex usage hook passes that fetch directly to the ChatGPT usage request. The linked repair PR was closed unmerged, so no merged or released implementation owns the remaining bug.

## What This Changes

Codex/OpenAI provider usage requests should honor configured HTTP(S) proxy environment settings instead of failing with direct `fetch failed` errors.

## Best Possible Solution

Land a focused provider-usage loader fix that preserves explicit `opts.fetch` precedence, falls back to `resolveProxyFetchFromEnv(env)` when no explicit fetch is supplied, and adds regression coverage without introducing a new config surface.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a plain fetch in `loadProviderUsageSummary()` and passes it into the Codex usage request. I did not run the live proxy-only network scenario.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing env proxy fetch helper at the provider-usage loader boundary, preserving explicit injected fetch behavior and avoiding duplicate Codex-specific proxy logic.
