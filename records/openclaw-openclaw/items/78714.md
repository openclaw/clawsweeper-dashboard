## Summary

Keep open: current main and the latest release still resolve provider-usage fetches with plain `resolveFetch(opts.fetch)`, while the OpenAI/Codex usage hook passes that fetch function directly to the `chatgpt.com` usage request. The exact linked repair PR was closed unmerged, so the central reported bug still needs a focused fix.

## What This Changes

The issue reports that Codex/OpenAI provider usage snapshots ignore configured HTTP_PROXY/HTTPS_PROXY env vars and return `fetch failed` when direct Node fetch cannot reach `chatgpt.com`.

## Best Possible Solution

Land a focused provider-usage loader fix that preserves explicit injected fetch precedence, falls back to `resolveProxyFetchFromEnv(env)` when no explicit fetch is supplied, and covers the behavior with regression tests.

## Reproduction Assessment

Yes from source: the current loader still chooses plain `resolveFetch(opts.fetch)`, and the OpenAI/Codex usage hook passes that fetch into the `chatgpt.com/backend-api/wham/usage` request. I did not run the live proxy-only network scenario.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing env proxy fetch helper at the provider-usage loader boundary, without adding a new config option or duplicating proxy logic in the Codex hook.
