## Summary

Keep open: current main still has the source-level proxy gap, and an open linked PR already references this issue with closing syntax. Closing now would preempt the normal review-and-merge path for the fix.

## What This Changes

Make Codex provider usage loading respect proxy environment variables when no explicit fetch override is supplied.

## Best Possible Solution

Review and land the linked proxy-aware provider-usage PR after it has adequate regression coverage and real proxied Codex usage proof.

## Reproduction Assessment

No, not as a live current-main reproduction in this environment. We do have a strong source-level path: the current loader supplies plain `resolveFetch(opts.fetch)` to the Codex usage hook, and the reporter provided logs showing direct fetch fails while the proxy-aware helper works.

## Solution Assessment

Yes: selecting an explicit fetch first, then an env-proxy-aware fetch, then global fetch in `loadProviderUsageSummary` is the narrow maintainable fix and keeps Codex usage fetching in the provider-owned hook.
