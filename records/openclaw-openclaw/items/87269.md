## Summary

Keep open: current main and the latest release still send successful WhatsApp audio preflight transcripts to the agent as plain transcript text in `Body` and `BodyForAgent`, so the reported marker/body-context behavior remains unresolved. The narrow code path is clear, but the exact agent-facing wrapper is still a maintainer product decision rather than an automatic cleanup close.

## What This Changes

The issue requests that WhatsApp preflight audio transcription preserve explicit audio context in the agent-facing body instead of replacing `<media:audio>` with plain transcript text.

## Best Possible Solution

Maintainers should choose the WhatsApp preflight transcript body contract, then update the plugin and focused tests so agent-facing text carries explicit audio context while preserving raw command bodies, media metadata, and duplicate-STT prevention.

## Reproduction Assessment

Yes for the marker-loss part: current-main source and tests show WhatsApp replaces `Body` and `BodyForAgent` with plain transcript text after preflight STT. No live WhatsApp run was performed, and the outbound voice-reply symptom remains only partly proven from source because media metadata is preserved.

## Solution Assessment

Unclear: preserving explicit audio context is the right direction, but maintainers need to choose the intended body representation before automation changes channel behavior.
