## Summary

Keep open: current main and the latest release still send WhatsApp preflight audio transcripts to the agent as plain transcript text in `Body`/`BodyForAgent`, so the reported marker/body-context behavior is not implemented away. The remaining decision is the intended WhatsApp transcript wrapper, because current media metadata and raw command fields are preserved and the voice-reply symptom is only partly proven from source.

## What This Changes

The issue asks WhatsApp preflight audio transcription to preserve explicit audio context in the agent-facing body instead of replacing `<media:audio>` with only transcript text.

## Best Possible Solution

Maintainers should choose the WhatsApp preflight transcript body contract, then update the plugin and focused tests so agent-facing text carries explicit audio context while preserving media metadata, raw/command bodies, and duplicate-transcription prevention.

## Reproduction Assessment

Yes for the marker-loss part: current-main source and tests show WhatsApp replaces `Body`/`BodyForAgent` with plain transcript text after preflight STT. No live WhatsApp run was performed, and the outbound voice-reply symptom still needs runtime confirmation.

## Solution Assessment

Unclear: preserving explicit audio context is the right direction, but maintainers need to choose the intended body representation before automation changes channel behavior.
