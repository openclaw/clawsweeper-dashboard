## Summary

Keep open: current main still gives WhatsApp agents the plain preflight transcript as `Body`/`BodyForAgent`, while the report asks for visible audio context to be preserved. The marker-loss part is source-reproducible, but the stated voice-reply cause needs maintainer clarification because outbound voice replies are gated by `messages.tts.auto`, not only `tools.media.audio.enabled`.

## What This Changes

The issue reports that WhatsApp preflight STT replaces an inbound audio body with only the transcript, dropping the audio marker needed for voice-reply behavior.

## Best Possible Solution

Clarify the intended preflight transcript rendering contract, then update WhatsApp so agent-facing text carries explicit audio context while preserving `MediaType`, `MediaPaths`, `Transcript`, and `MediaTranscribedIndexes`.

## Reproduction Assessment

Yes for the marker-loss part: current-main source and tests show WhatsApp replaces `Body`/`BodyForAgent` with plain transcript text after preflight STT. No live WhatsApp run was performed, and the text-vs-voice reply symptom still needs runtime confirmation.

## Solution Assessment

Unclear: preserving audio context is a plausible fix, but maintainers should choose whether WhatsApp should use the shared `[Audio]` block, Telegram's untrusted transcript wrapper, or another channel-owned representation before a repair PR changes behavior.
