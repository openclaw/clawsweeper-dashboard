## Summary

Keep open: current main and the latest release still deliver successful WhatsApp audio preflight transcripts to the agent as plain transcript text in `Body` and `BodyForAgent`, so the reported marker/context loss remains. The code path is source-reproducible, but the exact WhatsApp body wrapper needs maintainer product judgment before repair automation should change prompt-facing behavior.

## What This Changes

The issue requests that WhatsApp preflight audio transcription preserve explicit audio context in the agent-facing body instead of replacing the media marker with plain transcript text.

## Best Possible Solution

Maintainers should choose the WhatsApp preflight transcript body contract, then update the plugin and focused tests so agent-facing text carries explicit audio context while preserving media metadata, raw/command bodies, and duplicate-transcription prevention.

## Reproduction Assessment

Yes for the marker-loss part: current source and tests show WhatsApp replaces `Body` and `BodyForAgent` with plain transcript text after preflight STT. No live WhatsApp run was performed, so the outbound voice-reply symptom remains only partly proven.

## Solution Assessment

Unclear: preserving explicit audio context is the right direction, but maintainers need to choose the intended body representation before automation changes channel behavior.
