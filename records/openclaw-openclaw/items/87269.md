## Summary

Keep open: current main and the latest release still make WhatsApp preflight STT set `Body`/`BodyForAgent` to the plain transcript, so the reported agent-facing audio-context loss is not implemented away. Media metadata and raw command fields are preserved now, which weakens the stated voice-reply root cause but leaves the body rendering contract unresolved.

## What This Changes

The issue asks WhatsApp preflight STT to preserve explicit audio context in the agent-facing body instead of replacing `<media:audio>` with only transcript text.

## Best Possible Solution

Maintainers should choose the WhatsApp preflight transcript body contract, then align the plugin and tests so agent-facing text carries explicit audio context while media metadata and duplicate-transcription prevention stay intact.

## Reproduction Assessment

Yes for the marker-loss part: current-main source and tests show WhatsApp replaces `Body`/`BodyForAgent` with plain transcript text after preflight STT. No live WhatsApp run was performed, and the outbound voice-reply symptom still needs runtime confirmation.

## Solution Assessment

Unclear: preserving explicit audio context is the right direction, but maintainers need to choose the intended body representation before automation edits channel behavior.
