## Summary

Keep open: current main and v2026.6.9 still make WhatsApp preflight audio success set `Body` and `BodyForAgent` to plain transcript text, so the reported agent-facing marker/context behavior remains unresolved. The exact wrapper is still a maintainer product choice because media metadata and raw command bodies are preserved, and source proves marker loss more strongly than the downstream voice-reply symptom.

## What This Changes

The issue asks WhatsApp audio preflight transcription to preserve an explicit audio marker or `[Audio]` block in the agent-facing body instead of replacing it with plain transcript text.

## Best Possible Solution

Maintainers should choose one WhatsApp preflight transcript body contract, then update the plugin and focused tests so agent-facing text carries explicit audio context while preserving raw command bodies, transcript fields, media metadata, and duplicate-STT prevention.

## Reproduction Assessment

Yes for the marker-loss part: current-main source and tests show WhatsApp replaces `Body` and `BodyForAgent` with plain transcript text after preflight STT. No live WhatsApp run was performed, so the downstream text-versus-audio reply symptom remains source-inferred only.

## Solution Assessment

Unclear: preserving explicit audio context is the right direction, but maintainers still need to choose the intended WhatsApp body representation before automation changes channel behavior.
