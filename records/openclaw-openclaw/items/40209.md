## Summary

Keep open: the queue-key mismatch is a real source-level bug, but this branch's blanket lowercasing would break the current Signal group session-key preservation contract and still lacks real behavior proof.

## What This Changes

The PR lowercases system-event session keys in `requireSessionKey()` and adds unit tests for mixed-case Slack channel enqueue/peek and deduplication.

## Best Possible Solution

Normalize system-event queue keys with the shared session-key canonicalizer that preserves opaque Signal IDs, add Slack and Signal regression coverage, then merge only after redacted real plugin/channel proof shows delivery.

## Reproduction Assessment

Yes at source level: current main can store a system event under one session-key spelling while heartbeat peeks with a canonicalized key. I did not run a live Hermes, Slack, or Signal flow in this read-only review.

## Solution Assessment

No as written: normalizing the system-event boundary is the right direction, but unconditional lowercasing conflicts with current Signal group ID preservation. Use the shared canonicalizer or equivalent preservation logic instead.

- **[P1] Preserve Signal group session IDs:** `src/infra/system-events.ts:52`
