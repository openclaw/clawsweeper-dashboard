## Summary

Keep open: the reported queue-key mismatch is still plausible on current main, but this patch is not mergeable as written because it lowercases provider-owned opaque Signal group IDs and still lacks after-fix real behavior proof.

## What This Changes

The PR changes the system-events queue to lowercase session keys and adds unit coverage for mixed-case Slack channel lookup and deduplication.

## Best Possible Solution

Normalize system-event queue keys with the shared session-key canonicalizer that preserves opaque Signal IDs, add Slack/mixed-case and Signal group regression coverage, and merge only after redacted real plugin/channel proof confirms delivery.

## Reproduction Assessment

Yes at source level: current main can store a system event under a mixed-case queue key while the heartbeat path peeks with a canonicalized lowercase Slack key. I did not run a live Hermes, Slack, or Signal flow in this read-only review.

## Solution Assessment

No as written: normalizing at the system-event boundary is the right direction, but unconditional `.toLowerCase()` conflicts with the current shared Signal group ID preservation contract. Use the shared canonicalizer or equivalent preservation logic instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve Signal group session IDs:** `src/infra/system-events.ts:52`
  - body: This lowercases the full `sessionKey`, but current main intentionally preserves `signal:group:<id>` because those IDs are opaque. Use the shared session-key canonicalizer and add a Signal group regression case.
