## Summary

Keep open: the queue-key mismatch is source-reproducible and the PR addresses the Slack-style case mismatch, but the proposed unconditional lowercasing conflicts with OpenClaw's Signal group session-key contract and the PR still lacks real plugin/channel proof.

## What This Changes

The PR lowercases system-event session keys in the in-memory queue and adds tests for mixed-case channel lookup and deduplication.

## Best Possible Solution

Normalize system-event keys with the shared session-key canonicalizer that preserves Signal group IDs, add Slack/mixed-case and Signal group regression coverage, then merge after redacted real plugin/channel proof confirms delivery.

## Reproduction Assessment

Yes at source level: current main stores system events under `requireSessionKey()`'s trimmed input while heartbeat resolves and peeks by a canonical session key. I did not run a live Hermes, Slack, or Signal flow in this read-only review.

## Solution Assessment

No as written: normalizing queue keys at the shared boundary is the right shape, but unconditional `.toLowerCase()` violates the current Signal group ID preservation contract. Use the shared session-key canonicalizer or equivalent preservation logic instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve opaque Signal group session IDs:** `src/infra/system-events.ts:52`
  - body: This lowercases the entire session key, but current session-key canonicalization intentionally preserves `signal:group:<id>` because Signal group IDs are opaque/base64-like. A system event for `agent:main:signal:group:VWAT...` would now be stored under a lowercased ID while heartbeat/session-store helpers keep the original ID, so Signal group events can miss the queue. Use the shared session-key canonicalizer and add a Signal group regression case.
