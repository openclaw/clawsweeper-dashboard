## Summary

Keep open. Current main still has the reported casing mismatch: the system-event queue stores caller-provided session-key casing, while the heartbeat forced-session path canonicalizes agent session keys to lowercase before peeking or draining queued events. The PR remains a focused implementation for the linked channel-session heartbeat bug, with one maintainer contract check around whether lowercasing should apply to every system-event key or only agent-scoped keys.

## What This Changes

The PR lowercases `requireSessionKey()` in the in-memory system-events queue and adds regression tests for mixed-case Slack channel session lookup plus cross-case deduplication.

## Best Possible Solution

Land this PR or an equivalent patch that normalizes system-event queue keys to the same canonical form used by heartbeat session resolution, while keeping the mixed-case channel lookup and cross-case dedupe regression tests. If maintainers want non-agent system-event keys to remain case-sensitive, scope the normalization to parseable agent session keys and document that contract.

## Reproduction Assessment

Yes. The deterministic source-level reproduction is to enqueue under `agent:main:slack:channel:C0XX` and peek or drain under `agent:main:slack:channel:c0xx`; current main addresses different `Map` keys, while the PR's added tests encode the expected cross-case lookup.

## Solution Assessment

Yes, with one contract check. Normalizing in `requireSessionKey()` is the narrowest shared point for enqueue, peek, drain, consume, and has-events; a safer variant would normalize only parseable agent-scoped keys if maintainers require case-sensitive custom queues.
