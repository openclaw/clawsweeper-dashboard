## Summary

Keep open: current main still exposes mutable, sequential plugin hooks without hook-chain integrity state, host-owned modification audit/provenance, or a per-hook permission allowlist. The live thread already treats this as security-sensitive plugin-runtime design, and https://github.com/openclaw/openclaw/issues/18571 is a narrower outbound-verification request rather than a replacement.

## What This Changes

Request adds tamper-evident hook-chain integrity, modification auditing, and per-hook permission or manifest enforcement for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, distinguish native in-process trust from isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a security hardening and feature-design request. Source inspection on current main gives high-confidence feature-gap proof that integrity state, audit/provenance records, and per-hook permission allowlists are still missing.

## Solution Assessment

No for checksum-only hook-chain integrity as a standalone fix. The maintainable solution needs explicit trust-boundary, audit/redaction, config, manifest, and compatibility decisions before implementation.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive hook-runtime design, and native plugin trust, audit redaction, and permission boundaries need maintainer decisions before implementation.

Concerns:

- **[medium] Define the native plugin trust boundary before checksums:** `docs/plugins/architecture.md:444`
  - body: Native plugins run in-process with Gateway-level trust, so checksums alone may not defend against a malicious loaded plugin unless the protected boundary and isolated-plugin behavior are specified.
  - confidence: 0.9
- **[medium] Specify provenance and audit semantics before enforcement:** `src/plugins/hooks.ts:570`
  - body: Current modifying hooks can rewrite messages or hook results sequentially without a host-owned modification ledger, so enforcement needs explicit per-handler audit semantics instead of only a terminal block decision.
  - confidence: 0.88
- **[medium] Redact hook audit content before logging:** `docs/plugins/hooks.md:304`
  - body: Hook audit records could capture prompts, transcripts, tool output, or credential-adjacent data unless the schema is bounded and redacted before storage or logs.
  - confidence: 0.82

  - body: The current registry policy only gates broad prompt and conversation hook categories, so a new per-hook allowlist needs compatibility, diagnostics, and docs to avoid silently breaking existing plugins.
  - confidence: 0.78
