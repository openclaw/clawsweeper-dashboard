## Summary

Keep open: current main still has mutable, sequential plugin hooks without host-owned integrity/audit state or a per-hook permission allowlist, and the issue thread already kept this as security-sensitive plugin-runtime design work rather than a stale cleanup item.

## What This Changes

The issue requests hook-chain integrity checks, audit logging, and per-hook permission controls for mutable plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics, explicitly separate in-process native plugin trust from any isolated IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable as a feature/security hardening request rather than broken documented behavior. Source inspection does confirm the current mutable hook chain and the absence of the requested integrity, audit, and per-hook allowlist policy on current main.

## Solution Assessment

No: a narrow checksum-only patch would not be the best solution because native plugins currently share the Gateway process trust boundary. Maintainers need to decide trust, audit, enforcement, and compatibility semantics first.
