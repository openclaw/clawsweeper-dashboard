## Summary

The requested hardening remains materially unimplemented on current main, and the existing maintainer-thread direction still applies: this needs an explicit plugin trust-model and compatibility decision rather than closure or an automated patch.

## What This Changes

Add tamper-evident hook-chain provenance, per-handler modification auditing, and hook registration permissions for plugin hooks.

## Best Possible Solution

Define an isolation-aware plugin threat model, then add host-owned provenance and redacted audit events plus a backwards-compatible named-hook capability policy at the plugin-runtime boundary.

## Reproduction Assessment

Yes at the source level: current main clearly executes modifying hooks sequentially and accepts returned mutations without a general integrity ledger. This demonstrates the requested gap, but it is intentional extension behavior rather than a broken existing contract.

## Solution Assessment

No, cryptographic checksums alone are not the best solution for trusted in-process plugins. Meaningful protection must be anchored outside the plugin privilege boundary and paired with explicit permissions and privacy-safe audit semantics.
