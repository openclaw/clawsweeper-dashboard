## Summary

Keep open: current main still lacks host-owned hook-chain integrity state, per-handler modification audit/provenance, and a per-hook permission allowlist. The live thread already kept this for security-sensitive plugin-runtime design, and the related outbound-verification issue is narrower rather than a superseding implementation.

## What This Changes

Request hook-chain integrity checks, modification audit logging, and per-hook permission policy for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, distinguish native in-process behavior from isolated-plugin behavior, then add a backwards-compatible per-hook permission/config contract with redaction, docs, and focused tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a security hardening and plugin API design request. Source inspection gives high-confidence proof that current main still lacks hook-chain integrity state, audit/provenance records, and per-hook permission allowlists.

## Solution Assessment

No for checksum-only hook-chain integrity as a standalone fix. The maintainable path needs explicit trust-boundary, audit/redaction, config, manifest, and compatibility decisions before implementation.

## Security Review

Status: needs_attention

Summary: This is security-sensitive hook-runtime design; native-plugin trust, audit redaction, and permission-boundary decisions need maintainer review.

Concerns:

- **[medium] Define the native plugin trust boundary before checksums:** `docs/plugins/architecture.md:444`
  - body: Native plugins run in-process with Gateway-level trust, so checksums alone may not defend against a malicious loaded plugin unless the protected boundary and isolated-plugin behavior are specified.
  - confidence: 0.9
- **[medium] Specify provenance and audit semantics before enforcement:** `src/plugins/hooks.ts:570`
  - body: Current modifying hooks can rewrite hook results or messages sequentially without a host-owned modification ledger, so enforcement needs explicit per-handler audit semantics instead of only a terminal block decision.
  - confidence: 0.88
- **[medium] Redact hook audit content before logging:** `docs/plugins/hooks.md:229`
  - body: Hook audit records could capture prompts, transcripts, tool output, file paths, or credential-adjacent data unless the schema is bounded and redacted before storage or logs.
  - confidence: 0.82
- **[medium] Design per-hook policy migration carefully:** `src/plugins/registry.ts:2322`
  - body: The current registry policy only gates broad prompt and conversation hook categories, so a new per-hook allowlist needs compatibility, diagnostics, and docs to avoid silently breaking existing plugins.
  - confidence: 0.8
