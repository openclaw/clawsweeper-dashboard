## Summary

Keep open. Current main still documents and implements in-process, sequential, mutation-capable plugin hooks without hook-chain integrity state, per-handler mutation audit/provenance, or per-hook registration policy; the existing thread also keeps this routed as security-sensitive plugin-runtime design, and #18571 is narrower outbound-hook work rather than a replacement.

## What This Changes

Request hook-chain integrity protection, audit logging, and per-hook plugin permission policy to detect or prevent tampering across mutable plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and redacted per-handler mutation audit semantics in the plugin runtime, separate in-process native-plugin behavior from any isolated IPC path, then add a backwards-compatible per-hook registration policy with schema, docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by registering ordered modifying hooks such as before_tool_call, message_sending, tool_result_persist, or before_message_write and observing that current main passes merged or rewritten results sequentially without checksum, provenance, or mutation-audit validation.

## Solution Assessment

Unclear. The requested checksum approach is not obviously sufficient for in-process native plugins; the safer route is a maintained trust-boundary design followed by provenance/audit and per-hook policy that match that model.
