## Summary

Current main still intentionally runs modifying plugin hooks sequentially and permits mutations to reach agent and persisted message state; it has added coarse prompt/conversation consent and timeout controls, but not the requested per-handler provenance, tamper-evident audit records, or per-hook capability policy. This should remain open because it is security-aligned core/plugin-runtime design, and the related outbound-verification issue is narrower rather than canonical for this work.

## What This Changes

Add tamper-evident hook processing, per-handler mutation auditing, and per-hook registration permissions to the plugin runtime.

## Best Possible Solution

Define host-owned provenance and redacted modification-audit semantics at an enforceable process boundary, then evolve the existing category-level consent controls into a documented, compatibility-planned per-hook capability policy.

## Reproduction Assessment

Yes at the source-contract level: current code intentionally executes modifying hooks sequentially, and tests confirm their mutations can reach persisted session data. This demonstrates the requested hardening gap, not a violation of an existing integrity guarantee.

## Solution Assessment

No, the checksum-first proposal is not sufficient as written because native plugins share the Gateway process trust boundary. The maintainable solution must first define isolation ownership, audit/redaction semantics, and a backward-planned capability contract.
