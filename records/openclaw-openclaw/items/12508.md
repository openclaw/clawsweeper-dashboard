## Summary

Keep open. Current main and v2026.5.12 still have mutable sequential plugin hook chains without host-owned integrity state, per-handler audit semantics, or a per-hook permission allowlist, and the existing discussion keeps this in core plugin-runtime security design rather than treating the narrower outbound-verification item as a replacement.

## What This Changes

Request hook-chain integrity, modification audit logging, and hook permission controls to detect or constrain plugin message tampering.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract in the plugin runtime that distinguishes in-process native plugins from any isolated-plugin boundary, then implement it compatibly with docs and focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main shows mutable sequential hook execution, transcript-adjacent hook mutation, and broad hook policy fields without integrity/audit/per-hook allowlist controls. This is a feature and API-contract gap, not a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum patch. The maintainable solution needs a trust-boundary, audit, and permission contract that handles in-process plugins, future isolation, compatibility, and sensitive-data redaction together.
