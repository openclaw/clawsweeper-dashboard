## Summary

Keep open. Current main and v2026.5.20 still have mutable sequential plugin hooks without hook-chain integrity/provenance state, per-handler modification audit semantics, or a per-hook permission allowlist, and the prior thread review correctly treats this as security-sensitive plugin-runtime design work rather than stale cleanup.

## What This Changes

The issue requests host-owned integrity checks, audit logging, and permission controls for plugin hook chains so one plugin cannot silently tamper with messages or hook context seen by later handlers.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract that separates in-process plugins from any isolated-plugin boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.20 show sequential mutable hook execution, transcript-adjacent message mutation, and broad hook policy fields without chain integrity, per-handler audit, or per-hook allowlist controls. This is a feature/API-contract gap, not a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum-only patch. The maintainable solution needs a trust-boundary, audit redaction, permission, and compatibility contract that handles in-process plugins and any future isolation boundary together.
