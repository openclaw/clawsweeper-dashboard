## Summary

Keep open: current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler audit/provenance records, or a per-hook permission allowlist, and the discussion already treats this as broader security-sensitive plugin-runtime design rather than a duplicate of the narrower outbound verification issue.

## What This Changes

The issue requests hook-chain integrity checks, hook modification auditing, and per-hook permission controls for plugin hook execution.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, separate native in-process trust from any isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/security hardening request rather than a broken documented behavior. Source inspection does confirm current main still lacks the requested integrity state, audit/provenance records, and per-hook permission allowlist.

## Solution Assessment

No: a narrow checksum-only patch is not the best solution because native plugins currently share the Gateway process trust boundary. Maintainers should decide trust, audit, enforcement, and compatibility semantics before implementation.
