## Summary

Keep open. Current main and v2026.5.19 still expose mutable sequential plugin hooks without host-owned chain integrity/provenance state, per-handler modification audit semantics, or a per-hook permission allowlist; the related outbound-verification request is narrower and does not supersede this security-sensitive plugin-runtime design item.

## What This Changes

The issue requests hook-chain integrity protection, tamper/modification audit logging, and per-hook permission controls for plugin hook execution.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract that explicitly separates in-process native plugins from any isolated-plugin IPC boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.19 show mutable sequential hook execution, transcript-adjacent hook mutation, and broad hook policy fields without integrity, audit, or per-hook allowlist controls. This is a feature and API-contract gap, not a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum-only patch. The maintainable solution needs a trust-boundary, audit redaction, permission, and compatibility contract that handles in-process plugins and future isolation together.
