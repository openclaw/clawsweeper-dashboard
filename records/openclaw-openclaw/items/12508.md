## Summary

Keep open: current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler audit/provenance records, or a per-hook permission allowlist, and the existing discussion treats this as security-sensitive plugin-runtime design work rather than something already covered by https://github.com/openclaw/openclaw/issues/18571.

## What This Changes

The issue requests adding hook-chain integrity checks, per-handler audit/provenance records, and per-hook permission controls for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics, explicitly separate native in-process trust from isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/security hardening request rather than broken documented behavior. Source inspection does confirm current main still lacks the requested integrity state, audit/provenance records, and per-hook permission allowlist.

## Solution Assessment

No: a narrow checksum-only patch would not be the best way to solve this because native plugins currently share the Gateway process trust boundary. Maintainers should decide trust, audit, enforcement, and compatibility semantics before implementation.
