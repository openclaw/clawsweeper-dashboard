## Summary

Keep open. Current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist, and the related outbound-verification issue is narrower rather than a replacement.

## What This Changes

Request to add hook-chain integrity verification, audit logging, and per-hook permission policy for plugin hooks that can mutate messages, tool calls, and session-adjacent state.

## Best Possible Solution

Define a host-owned hook provenance and mutation-audit contract, explicitly separate in-process native plugin behavior from any future isolated-plugin IPC boundary, then add a backwards-compatible per-hook permission policy with docs and focused runtime tests.

## Reproduction Assessment

Yes at source level: current main clearly shows ordered mutable hook execution and the absence of integrity/audit/per-hook allowlist state. This is a reproducible capability gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear as proposed. The desired security outcome is valid, but cryptographic checksums alone are not necessarily the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and whether this belongs in current in-process hooks or a future isolated plugin IPC path.
