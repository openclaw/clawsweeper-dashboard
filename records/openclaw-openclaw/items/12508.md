## Summary

Keep open: current main still exposes mutable, sequential plugin hooks without a host-owned integrity/audit chain or per-hook permission allowlist, and the existing maintainer-thread review kept this as security-sensitive plugin-runtime design work.

## What This Changes

Request adding hook-chain integrity verification, tamper detection, audit logging, and per-hook permission controls for plugin hook execution.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics, distinguish in-process native plugins from any isolated IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable as a feature/security hardening request rather than a broken documented behavior. Source inspection does confirm the current mutable hook chain and missing integrity/audit policy on current main.

## Solution Assessment

No. A narrow checksum-only patch would not be the best solution because native plugins currently share the Gateway process trust boundary; maintainers need to define trust, audit, and compatibility semantics first.
