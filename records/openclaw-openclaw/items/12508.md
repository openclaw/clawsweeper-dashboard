## Summary

Keep open. Current main and the latest release still expose mutable sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist; related open issues cover narrower outbound verification, sandboxing, or hook permission slices rather than replacing this broader security-design request.

## What This Changes

The issue requests hook-chain integrity, tamper detection, audit logging, and per-hook permission controls for mutable plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and mutation-audit semantics, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Yes, at source level: current main shows ordered mutable hook execution and no hook-chain integrity, per-handler mutation audit, or per-hook allowlist state. This is a reproducible capability gap rather than broken behavior under an existing contract.

## Solution Assessment

Unclear as proposed: the security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the protected boundary, audit semantics, and plugin compatibility path.
