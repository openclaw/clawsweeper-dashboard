## Summary

Keep open. Current main and the latest release still document and implement mutable sequential plugin hooks without hook-chain integrity/provenance state, per-handler modification audit semantics, or a per-hook permission allowlist; prior review also kept this for security-sensitive plugin-runtime design rather than stale cleanup.

## What This Changes

The issue requests host-owned integrity, audit logging, and per-hook permission controls to detect or constrain plugin hook message tampering.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract that separates in-process plugins from any isolated-plugin boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.22 show sequential mutable hook execution, transcript-adjacent message mutation, and broad hook policy fields without chain integrity, per-handler audit, or per-hook allowlist controls. This is a feature/API-contract gap, not a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum-only patch. The maintainable solution needs a trust-boundary, audit redaction, permission, and compatibility contract that handles in-process plugins and any future isolation boundary together.
