## Summary

Keep open. Current main and the latest release still have mutable sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist; https://github.com/openclaw/openclaw/issues/18571 is narrower outbound verification work and does not replace this broader security design item.

## What This Changes

The issue requests host-owned integrity checks, mutation audit logs, and per-hook permission controls for mutable plugin hook chains.

## Best Possible Solution

Define a host-owned hook provenance and mutation-audit contract, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with focused docs and tests.

## Reproduction Assessment

Yes at source level: current main shows ordered mutable hook execution and no hook-chain integrity, per-handler mutation audit, or per-hook allowlist state. This is a reproducible capability gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear as proposed: the security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and relationship to isolated plugin IPC.
