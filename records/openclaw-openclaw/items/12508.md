## Summary

Keep open: current main and v2026.5.12 still lack host-owned hook-chain integrity/provenance, modification audit semantics, and per-hook permission policy. Prior thread reviews and the related narrower outbound-verification issue support treating this as security-sensitive plugin-runtime design, not cleanup.

## What This Changes

The issue requests hook-chain integrity protection, audit logging, and per-hook permissions for mutable plugin hooks.

## Best Possible Solution

Define a host-owned hook provenance and redacted modification-audit contract, distinguish native in-process behavior from any isolated-plugin IPC boundary, then add a backwards-compatible per-hook permission policy with focused runtime, registry, config, docs, and transcript/message tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a security hardening/API feature request. Source and release inspection give high-confidence proof that current main and v2026.5.12 still lack hook-chain integrity state, per-handler audit/provenance records, and per-hook permission allowlists.

## Solution Assessment

No for checksum-only integrity as a standalone implementation. The maintainable solution needs maintainer decisions on native-plugin trust boundaries, isolated-plugin enforcement, audit redaction, manifest/config policy, and migration compatibility before code is written.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive plugin-runtime design work and needs maintainer trust-boundary, audit-redaction, and compatibility decisions before implementation.

Concerns:

- **[medium] Define the native plugin trust boundary before checksums:** `docs/plugins/architecture.md:444`
  - body: Native plugins run in-process with Gateway-level capabilities, so checksums alone may not defend against a malicious loaded plugin unless maintainers specify what boundary is protected and how isolated-plugin IPC should differ.
  - confidence: 0.9
- **[medium] Specify provenance and audit semantics before enforcement:** `src/plugins/hooks.ts:570`
  - body: Current modifying hooks can rewrite hook results or messages sequentially without a host-owned modification ledger, so enforcement needs explicit per-handler audit semantics rather than only a final checksum decision.
  - confidence: 0.88
- **[medium] Redact hook audit content before storage or logs:** `docs/plugins/hooks.md:253`
  - body: Hook audit records could capture prompts, transcripts, tool output, file paths, or credential-adjacent data unless the schema is bounded and redacted before persistence or logging.
  - confidence: 0.82
- **[medium] Plan per-hook policy migration carefully:** `src/config/types.plugins.ts:3`
  - body: Current policy only gates broad prompt and conversation hook categories, so a new per-hook allowlist needs defaults, diagnostics, docs, and compatibility tests to avoid silently breaking existing plugins.
  - confidence: 0.8
