## Summary

Keep open: current main still has mutable, sequential plugin hooks without hook-chain integrity state, per-handler audit/provenance records, or a per-hook permission allowlist, and the existing discussion already kept this as security-sensitive plugin-runtime design work.

## What This Changes

Request hook-chain integrity, audit logging, and per-hook permission hardening for plugin hooks so message tampering between sequential handlers can be detected or constrained.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics, explicitly separate in-process native plugin trust from any isolated IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/security hardening request rather than broken documented behavior. Source inspection does confirm the current mutable sequential hook chain and the absence of the requested integrity, audit, and per-hook allowlist policy on current main.

## Solution Assessment

No: a narrow checksum-only patch would not be the best way to solve this because native plugins currently share the Gateway process trust boundary. Maintainers should decide trust, audit, enforcement, and compatibility semantics before implementation.
