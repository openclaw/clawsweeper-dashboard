## Summary

Keep open for maintainer security and product design. Current main attributes some terminal hook decisions to plugins, but modifying hooks still execute sequentially without a general per-handler mutation audit contract or per-hook authorization policy; the narrower outbound-verification work does not supersede this request.

## What This Changes

Add tamper-evident plugin hook execution, per-handler modification auditing, and per-hook registration permissions.

## Best Possible Solution

Define one host-owned hook provenance and authorization contract, specify its guarantees separately for in-process and isolated plugins, and introduce compatibility-safe enforcement with diagnostics, documentation, and focused security tests.

## Reproduction Assessment

Yes at the source level: current main demonstrably runs modifying hooks sequentially and lacks a general per-hook authorization or per-handler mutation-audit contract. This confirms the architectural condition, not a separate sandbox escape.

## Solution Assessment

No, the checksum-first proposal is not yet the best solution because native plugins share the host process and could bypass an in-process checksum mechanism. The maintainable direction is host-owned provenance and authorization tied to a real isolation boundary where available.
