- `impact:security`: The request concerns plugin trust boundaries, tamper detection, hook permission gating, and auditability of security-relevant hook mutations.

## Summary

Keep open. Current main and the latest release still expose ordered mutable plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist; the related outbound verification issue is narrower and does not supersede this broader plugin-runtime security design.

## What This Changes

Request hook-chain integrity verification, mutation audit logging, and per-hook plugin permissions for OpenClaw plugin hooks.

## Best Possible Solution

Define a host-owned hook provenance and mutation-audit contract, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with focused docs and tests.

## Reproduction Assessment

Yes at source level: current main shows ordered mutable hook execution and no hook-chain integrity, mutation-audit, or per-hook allowlist state. This is a reproducible capability gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear as proposed: the security outcome is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and relationship to isolated plugin IPC.
