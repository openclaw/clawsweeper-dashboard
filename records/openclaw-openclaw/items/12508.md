## Summary

Keep open. Current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook permission allowlist, and the related outbound-verification issue does not cover this broader trust-boundary design.

## What This Changes

Request hook-chain integrity protection against message/context tampering, including per-handler audit records and hook permission policy.

## Best Possible Solution

Define a host-owned hook provenance and audit contract, explicitly separate in-process native plugins from any future isolated-plugin IPC boundary, then add backwards-compatible permission policy with docs and focused tests.

## Reproduction Assessment

Yes. Current source shows mutable, sequential hook execution and current policy lacks the requested hook-chain integrity state, per-handler audit records, and per-hook allowlist.

## Solution Assessment

No. The exact checksum proposal is only a plausible direction; the best solution first needs a maintainer trust-boundary decision and a compatible provenance/audit contract.
