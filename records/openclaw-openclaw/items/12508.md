## Summary

Keep open. Current main and v2026.5.12 still expose mutable, sequential plugin hook chains without host-owned integrity state, per-handler modification audit records, or a per-hook permission allowlist; the existing discussion also kept this for core plugin-runtime security design, and https://github.com/openclaw/openclaw/issues/18571 is a narrower outbound-verification request rather than a replacement.

## What This Changes

The issue requests hook-chain integrity checks, per-handler audit logging, and per-hook permission policy to protect plugin hook message/context mutation from silent tampering.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, explicitly distinguish in-process native plugins from any future isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Yes, at source level: current main shows mutable sequential hook execution and the config schema only exposes broad hook policy booleans/timeouts, not integrity/audit or per-hook allowlists. This is a feature/API contract gap rather than a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum patch. The safer maintainable direction is a host-owned provenance/audit and permission contract that accounts for in-process native plugins, isolated-plugin IPC, compatibility, and plugin privacy boundaries.
