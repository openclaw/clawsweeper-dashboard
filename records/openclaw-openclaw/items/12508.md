## Summary

Keep open. Current main and v2026.6.8 still expose mutable sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist; the prior thread reviews also kept this for maintainer security design, and https://github.com/openclaw/openclaw/issues/18571 is narrower outbound verification work rather than a replacement.

## What This Changes

Request adds hook-chain integrity checks, per-handler modification audit logging, and per-hook permission policy for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and mutation-audit semantics, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Yes at source level: current main shows ordered mutable hook execution and no hook-chain integrity, per-handler mutation audit, or per-hook allowlist state. This is a reproducible capability gap rather than broken existing documented behavior.

## Solution Assessment

Unclear as proposed: the security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and relationship to isolated plugin IPC.
