## Label Justifications

- `P2`: This is meaningful plugin-runtime security architecture work, but it is a feature/design request rather than an urgent outage or regression.
- `impact:security`: The request concerns plugin trust boundaries, tamper detection, hook permission gating, and auditability of security-relevant mutations.

## Summary

Keep open. Current main and the latest release still expose mutable, sequential plugin hooks with only coarse hook policy, and related issues cover narrower sandbox, profile, outbound-verification, or state-permission pieces rather than replacing this broader hook-chain integrity/security design work.

## What This Changes

Request to add hook-chain tamper detection, mutation audit logging, and per-hook plugin permission declarations for mutable plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and redacted mutation-audit semantics, separate in-process versus isolated-plugin threat models, then add a backwards-compatible per-hook permission policy through manifest/config/docs/tests.

## Reproduction Assessment

Yes, at source level: current main shows ordered mutable hook execution and no hook-chain integrity, per-handler mutation audit, or per-hook allowlist state. This is a reproducible security capability gap rather than broken behavior under an existing contract.

## Solution Assessment

Unclear as proposed: the security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and relationship to isolated plugin IPC.
