Keep open. Current main still exposes mutable sequential plugin hooks without hook-chain integrity/provenance state, per-handler modification audit semantics, or a per-hook permission allowlist, and the prior issue discussion already kept this for maintainer security-design work rather than treating the narrower outbound-verification issue as a replacement.

## What This Changes

The issue requests hook-chain integrity checks, modification audit records, and per-hook permission controls for plugin hooks.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract that separates in-process plugins from any isolated-plugin boundary, then implement it compatibly with docs and focused runtime tests.

- Checksum-style hook integrity may not provide meaningful protection against in-process native plugins unless maintainers define the protected boundary and threat model.

- The narrower outbound verification work at https://github.com/openclaw/openclaw/issues/18571 may inform part of the design but does not cover inbound, tool, session, or transcript-adjacent hook mutation.
