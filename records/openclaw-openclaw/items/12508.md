## Summary

Keep open. Current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook registration allowlist, and the existing issue thread already routes this as security-sensitive plugin-runtime design rather than cleanup closure.

## What This Changes

Add tamper-evident integrity checks, audit logging, and per-hook permission controls to the plugin hook chain.

## Best Possible Solution

Define host-owned hook provenance and redacted per-handler mutation audit semantics in the plugin runtime, explicitly separate in-process native plugins from any isolated IPC path, then add a backwards-compatible per-hook registration policy with schema, docs, and tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by registering ordered modifying hooks such as before_tool_call, message_sending, tool_result_persist, or before_message_write and observing that current main passes rewritten results through sequential handlers without checksum, provenance, or mutation-audit verification.

## Solution Assessment

Unclear whether the issue's checksum proposal is the best standalone fix because native plugins currently share the Gateway process trust boundary. The safer path is maintainer-defined security semantics first, followed by provenance/audit and per-hook policy that match that model.
