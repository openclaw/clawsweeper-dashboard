## Summary

Keep open: current main and the latest shipped release still expose mutable sequential plugin hooks without hook-chain integrity state, host-owned modification audit/provenance, or a per-hook permission allowlist. The live thread already kept this open for security-sensitive plugin-runtime design, and the related outbound-verification issue is narrower rather than a superseding implementation.

## What This Changes

The issue requests hook-chain integrity protection, audit/provenance logging, and per-hook permission controls for mutable plugin hook execution.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, distinguish native in-process trust from isolated-plugin behavior, then add a backwards-compatible per-hook permission policy with redaction, docs, and focused tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a security hardening and feature-design request. Source and release inspection give high-confidence proof that integrity state, audit/provenance records, and per-hook permission allowlists are still missing on current main and v2026.5.12.

## Solution Assessment

No for checksum-only hook-chain integrity as a standalone fix. The maintainable path needs explicit trust-boundary, audit/redaction, config, manifest, and compatibility decisions before implementation.
