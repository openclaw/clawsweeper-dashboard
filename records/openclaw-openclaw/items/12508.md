## Summary

Keep open. Current main still exposes mutable sequential plugin hooks without hook-chain integrity/provenance state, per-handler audit semantics, or per-hook allowlists, and the related outbound-verification issue is narrower rather than a replacement.

## What This Changes

Request host-enforced hook-chain integrity/provenance, per-handler audit logging, and per-hook permission declarations for plugin hooks.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract that separates in-process native plugins from any isolated-plugin boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main shows sequential mutable hook execution, transcript-adjacent message mutation, and broad hook policy fields without chain integrity, per-handler audit, or per-hook allowlist controls. This is a feature and API-contract gap, not a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum-only patch. The maintainable solution needs a trust-boundary, audit redaction, permission, and compatibility contract that handles in-process plugins and any future isolation boundary together.
