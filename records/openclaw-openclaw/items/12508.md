## Summary

Keep open. Current main and the latest release still have mutable sequential plugin hooks without hook-chain integrity/provenance state, per-handler modification audit semantics, or a per-hook permission allowlist; https://github.com/openclaw/openclaw/issues/18571 remains a narrower outbound-verification request and does not supersede this broader plugin-runtime security design.

## What This Changes

The issue requests hook-chain integrity verification, hook modification audit logging, and per-hook permission controls for plugin hooks.

## Best Possible Solution

Define a host-owned hook provenance, redacted audit, and permission contract that distinguishes in-process native plugins from any isolated IPC boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 show sequential mutable hook execution, transcript-adjacent message mutation, and hook config without integrity/audit/per-hook allowlist state. This is a feature/API-contract gap rather than a failing existing-behavior bug.

## Solution Assessment

No, not as a narrow checksum-only patch. The maintainable solution needs a trust-boundary, redacted audit, permission, and compatibility contract that handles in-process plugins and any future isolation boundary together.
