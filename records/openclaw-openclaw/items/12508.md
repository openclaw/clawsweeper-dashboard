## Summary

Current main and v2026.6.5 still expose sequential, mutable plugin hooks without hook-chain provenance, per-handler modification records, or a per-hook registration allowlist. The request remains relevant, but checksums cannot alone defend against native plugins sharing core's process trust boundary, so this requires maintainer security and compatibility design rather than cleanup closure.

## What This Changes

Add tamper-evident hook-chain handling, per-hook registration permissions, and audit records for plugin modifications.

## Best Possible Solution

Adopt a host-owned plugin security design that distinguishes trusted in-process plugins from isolated plugins, records bounded and redacted per-handler provenance at the hook runner, and introduces explicit hook capabilities through a backwards-safe public plugin contract with upgrade diagnostics and focused security tests.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 clearly show prioritized sequential mutation, category-level hook policy, and no integrity or provenance state. This reproduces the described capability gap, although it is a feature and security-hardening request rather than broken documented behavior.

## Solution Assessment

No, the checksum proposal alone is not the best solution because native plugins share the host process and can tamper with verification state. The maintainable path is a host-owned provenance and authorization contract tied to a clearly defined isolation boundary, with auditing designed for privacy and upgrade safety.

- A checksum computed and verified by the same unsandboxed process does not prevent a malicious native plugin from altering both data and verification state; the protected boundary must be defined first.
