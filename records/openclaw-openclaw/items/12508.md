## Summary

Keep open. Current main and the latest release still expose mutable, sequential plugin hook paths with coarse prompt/conversation gates only; the remaining work is a security/product design decision, not a stale or already-implemented request.

## What This Changes

The issue requests hook-chain integrity checks, tamper/audit logging, and per-hook permission policy for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and redacted mutation-audit semantics, distinguish in-process from isolated-plugin trust boundaries, then add a compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Yes, at source level: current main shows ordered mutable hook execution and no hook-chain integrity, per-handler mutation audit, or per-hook allowlist state. This is a reproducible security capability gap rather than broken behavior under an existing contract.

## Solution Assessment

Unclear as proposed: the security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and relationship to isolated plugin IPC.
