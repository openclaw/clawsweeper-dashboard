## Summary

Keep open. Current main and the latest release still have mutable sequential plugin hooks without hook-chain integrity/provenance state, per-handler modification audit semantics, or a per-hook allowlist, and the issue remains security-sensitive design work rather than cleanup-close material.

## What This Changes

The issue requests tamper-evident integrity checks, audit logging, and per-hook permission policy for mutable sequential plugin hooks.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract that distinguishes in-process native plugins from any isolated-plugin boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.22 show sequential mutable hook execution, transcript-adjacent message mutation, and broad hook policy fields without chain integrity, per-handler audit, or per-hook allowlist controls. This is a feature/API-contract gap rather than a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum-only patch. The maintainable solution needs a trust-boundary, audit redaction, permission, and compatibility contract that handles in-process plugins and any future isolation boundary together.
