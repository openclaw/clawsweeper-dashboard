## Summary

Current main still intentionally supports ordered, mutable plugin hooks without a hook-wide provenance chain, per-handler mutation audit records, or a generic hook-registration allowlist. This remains necessary security architecture work, but native plugins currently share the Gateway process trust boundary, so checksum-first protection would be incomplete and the issue needs maintainer decisions on isolation, audit-data handling, authorization defaults, and plugin compatibility.

## What This Changes

Add host-owned hook provenance, mutation auditing, integrity verification, and hook-specific plugin authorization across the plugin hook chain.

## Best Possible Solution

Define one host-owned provenance and redacted audit model, extend existing permission contracts with an explicit compatibility path, and apply cryptographic verification only across a real process or IPC isolation boundary.

## Reproduction Assessment

Yes at source level: current main clearly executes modifying hooks sequentially and lacks the requested provenance and audit chain. The security severity depends on the intended plugin trust boundary because native plugins are already trusted in-process code.

## Solution Assessment

No, not as a checksum-first implementation; effective protection requires an explicit isolation boundary, while auditing and authorization should build on existing host policy contracts rather than create a parallel system.
