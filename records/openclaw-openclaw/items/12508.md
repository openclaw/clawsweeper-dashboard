## Summary

Keep open: current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit/provenance, or a per-hook permission allowlist, and the issue discussion already treats this as security-sensitive plugin-runtime design rather than cleanup closure.

## What This Changes

Add hook-chain tamper detection, modification audit logging, and per-hook permission controls for plugin hook execution.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, separate native in-process trust from any isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable: this is a security hardening and feature-design request rather than broken documented behavior. Source inspection on current main does confirm the requested integrity state, audit/provenance records, and per-hook permission allowlist are still missing.

## Solution Assessment

No: a narrow checksum-only patch is not the best solution because native plugins currently share the Gateway process trust boundary. Maintainers should decide trust, audit, enforcement, redaction, and compatibility semantics before implementation.

## Risks / Open Questions

- Checksum-style hook integrity may not protect against a malicious native plugin unless the protected boundary is defined separately for in-process and isolated plugin execution.
- A per-hook permission or manifest allowlist could break existing plugins without migration semantics, diagnostics, documentation, and compatibility tests.
- Audit logging for hook modifications can expose sensitive prompt, transcript, or tool data unless the log schema and redaction rules are designed first.
