## Summary

Keep open for maintainer security and product design: current main adds coarse prompt/conversation hook controls, but it still lacks host-owned per-handler provenance, modification audit records, and a per-hook capability policy, and the narrower outbound-verification issue does not cover this broader trust-boundary work.

## What This Changes

Requests tamper-evident hook-chain provenance, modification auditing, and per-hook plugin permissions.

## Best Possible Solution

Define host-owned hook provenance and privacy-safe audit semantics around a clear isolation boundary, then introduce a backwards-compatible per-hook capability policy with diagnostics, migration, documentation, and focused runtime tests.

## Reproduction Assessment

Yes at source level: current main clearly demonstrates sequential mutable hooks and category-level policy without per-handler integrity records. This is a capability and security-design request, not a failing runtime bug that needs an execution repro.

## Solution Assessment

No, not in its checksum-first form: an in-process plugin can act with core-equivalent authority, so cryptographic wrapping alone does not establish the claimed protection. The maintainable approach starts with a defined host or process boundary and then adds provenance, auditing, and compatible permissions.
