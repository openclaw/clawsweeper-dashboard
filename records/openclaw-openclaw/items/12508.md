## Summary

Keep open. Current main still exposes mutable sequential plugin hook execution without host-owned integrity/provenance state, per-handler modification audit semantics, or a per-hook permission allowlist, and the thread already records security-design review that this broader trust-boundary problem is not superseded by the narrower outbound-verification request.

## What This Changes

The issue requests hook-chain integrity protection, audit logging, and per-hook permission policy so plugins cannot silently tamper with messages or context as hook handlers run.

## Best Possible Solution

Define a host-owned hook provenance, audit, and permission contract in the plugin runtime that explicitly distinguishes in-process native plugins from any isolated-plugin IPC boundary, then implement it compatibly with docs and focused runtime tests.

## Reproduction Assessment

Yes, at source level: current main shows mutable sequential hook execution, transcript-adjacent hook mutation, and broad hook policy fields without integrity, audit, or per-hook allowlist controls. This is a feature and API-contract gap, not a failing documented behavior test.

## Solution Assessment

No, not as a narrow checksum patch. The maintainable solution needs a trust-boundary, audit, redaction, and permission contract that handles in-process plugins, future isolation, compatibility, and sensitive-data handling together.

## Risks / Open Questions

- Checksum-style hook integrity may not meaningfully protect against in-process native plugins unless maintainers define the protected boundary and expected threat model.
- Per-hook allowlists and fail-closed integrity checks could break existing plugins without migration rules, diagnostics, docs, and compatibility tests.
- Audit logging for hook modifications can expose prompts, transcripts, tool outputs, file paths, or credential-adjacent data unless the log contract is redacted and permission-aware.
- The narrower outbound verification request at https://github.com/openclaw/openclaw/issues/18571 may inform part of the design but does not cover inbound/tool/session hook mutation or per-handler provenance.
