## Summary

Current main still intentionally permits sequential message/context modification by trusted code plugins and has only coarse prompt/conversation hook gates; it does not provide hook-chain provenance, per-handler mutation auditing, or individual-hook authorization. The request therefore remains necessary, but the proposed checksum approach cannot establish integrity against arbitrary code in the same Gateway process, so this should stay open for a maintainer-owned security and compatibility design rather than an automated fix.

## What This Changes

Add tamper-evident hook-chain processing, per-handler modification auditing, and per-hook plugin permissions.

## Best Possible Solution

Define trusted in-process and isolated plugin modes first, then add host-owned provenance at the isolation boundary, privacy-aware structured mutation audit records, and a versioned per-hook permission contract with upgrade diagnostics and compatibility tests.

## Reproduction Assessment

Yes at the source level: current main deliberately chains modifying hook results and lacks per-handler provenance or individual-hook authorization. This confirms the requested capability gap, not a regression against an existing documented integrity guarantee.

## Solution Assessment

No, the checksum-first proposal is not by itself the best solution because native plugins share the Gateway trust domain; isolation, provenance, authorization, audit privacy, and compatibility semantics need one coherent design.
