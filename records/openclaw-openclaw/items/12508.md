## Summary

Keep open: current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit/provenance, or a per-hook permission allowlist, and the existing discussion correctly treats this as security-sensitive plugin-runtime design rather than cleanup closure.

## What This Changes

The issue requests hook-chain integrity checks, audit logging, and per-hook permission policy to protect sequential plugin hooks from silent message tampering.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, distinguish native in-process trust from isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable: this is a security hardening and feature-design request rather than broken documented behavior. Source inspection on current main confirms the requested integrity state, audit/provenance records, and per-hook permission allowlist are still missing.

## Solution Assessment

No for a checksum-only hook chain as a standalone fix. The maintainable solution needs maintainer trust-boundary, audit/redaction, config, manifest, and compatibility decisions before implementation.
