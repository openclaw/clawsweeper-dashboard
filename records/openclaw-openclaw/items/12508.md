## Summary

Keep open: current main and the latest shipped release still expose mutable sequential plugin hooks without host-owned integrity/provenance state, per-handler modification audit records, or per-hook permission allowlists, and the issue thread already treats this as security-sensitive plugin-runtime design rather than a narrow bug fix.

## What This Changes

Request host-enforced integrity verification, audit logging, and per-hook permissions for mutable plugin hook chains.

## Best Possible Solution

Define host-owned hook provenance and redacted modification-audit semantics, distinguish native in-process trust from isolated-plugin behavior, then add a backwards-compatible per-hook permission policy with focused runtime, registry, config, docs, and transcript/message tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a security hardening and feature-design request. Source and release inspection give high-confidence proof that current main and v2026.5.12 still lack hook-chain integrity state, per-handler audit/provenance records, and per-hook permission allowlists.

## Solution Assessment

No for checksum-only integrity as a standalone implementation. The maintainable solution needs maintainer decisions on native-plugin trust boundaries, isolated-plugin enforcement, audit redaction, manifest/config policy, and migration compatibility before code is written.

- **[medium] Define the native plugin trust boundary before checksums:** `docs/plugins/architecture.md:444`
  - body: Native plugins run in-process with Gateway-level capabilities, so checksums alone may not defend against a malicious loaded plugin unless maintainers specify what boundary is protected and how isolated-plugin IPC should differ.
  - confidence: 0.9
- **[medium] Specify provenance and audit semantics before enforcement:** `src/plugins/hooks.ts:570`
  - body: Current modifying hooks can rewrite hook results or messages sequentially without a host-owned modification ledger, so enforcement needs explicit per-handler audit semantics rather than only a final checksum decision.
  - confidence: 0.88

  - confidence: 0.82

  - body: Current policy only gates broad prompt and conversation hook categories, so a new per-hook allowlist needs defaults, diagnostics, docs, and compatibility tests to avoid silently breaking existing plugins.
  - confidence: 0.8
