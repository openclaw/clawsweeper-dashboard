## Summary

Keep open. Current main still has ordered mutable plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist; related outbound-verification and sandbox/profile items are adjacent or narrower, not replacements.

## What This Changes

The issue requests tamper-evident integrity checks, audit logging, and per-hook permission policy for the plugin hook chain.

## Best Possible Solution

Define a host-owned hook provenance and mutation-audit contract, explicitly separate in-process native plugins from any future isolated-plugin IPC boundary, then add a backwards-compatible per-hook permission policy with docs and focused runtime tests.

## Reproduction Assessment

Yes at source level: current main clearly shows ordered mutable hook execution and the absence of integrity/audit/per-hook allowlist state. This is a reproducible capability gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear as proposed: the security outcome is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the hook trust boundary, audit semantics, compatibility plan, and relationship to future isolated plugin IPC.
