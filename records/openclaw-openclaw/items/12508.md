## Summary

Keep open: current main still has mutable sequential plugin hooks without hook-chain integrity state, host-owned modification audit/provenance, or a per-hook permission allowlist, and the live discussion already treats this as security-sensitive plugin-runtime design rather than cleanup closure.

## What This Changes

The issue requests cryptographic hook-chain integrity checks, tamper/modification audit logging, and per-hook permission policy for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, distinguish native in-process trust from isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction: this is a security hardening and feature-design request. Source inspection on current main gives high-confidence feature-gap proof that integrity state, audit/provenance records, and per-hook permission allowlists are still missing.

## Solution Assessment

No for checksum-only hook-chain integrity as a standalone fix. The maintainable solution needs explicit trust-boundary, audit/redaction, config, manifest, and compatibility decisions before implementation.
