## Summary

Keep open: current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit/provenance, or a per-hook permission allowlist, and the discussion already treats this as security-sensitive plugin-runtime design rather than cleanup closure.

## What This Changes

The issue requests hook-chain integrity checks, audit/provenance records, and permission controls for plugin hook message mutation.

## Best Possible Solution

Define host-owned hook provenance and modification-audit semantics, separate native in-process trust from isolated-plugin IPC behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Not applicable: this is a security hardening and feature-design request rather than broken documented behavior. Source inspection on current main confirms the requested integrity state, audit/provenance records, and per-hook permission allowlist are still missing.

## Solution Assessment

No: a narrow checksum-only hook chain is not the best standalone solution because native plugins currently share the Gateway process trust boundary. Maintainers should decide trust, audit, enforcement, redaction, and compatibility semantics before implementation.
