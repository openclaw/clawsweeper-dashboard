## Summary

Keep open. Current main and the latest release still expose mutable sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook permission allowlist; the related open issues cover narrower or prerequisite security surfaces rather than replacing this tracker.

## What This Changes

The issue requests hook-chain integrity checks, mutation audit logging, and hook permission controls for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and mutation-audit semantics, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests.

## Reproduction Assessment

Yes, at source level. Current main shows ordered mutable hook execution and no hook-chain integrity state, per-handler mutation audit, or per-hook allowlist; this is a reproducible capability gap rather than a broken existing contract.

## Solution Assessment

Unclear as proposed. The security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the trust boundary, audit semantics, compatibility plan, and relationship to isolated plugin IPC.
