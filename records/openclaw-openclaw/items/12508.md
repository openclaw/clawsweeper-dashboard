## Summary

Keep open: current main and the latest release still lack hook-chain integrity/provenance, per-handler modification audit semantics, and per-hook permission allowlists. Prior thread reviews and the related outbound verification issue support treating this as security-sensitive plugin-runtime design, not conservative cleanup.

## What This Changes

The issue requests cryptographic hook-chain integrity checks, modification audit logging, and per-hook registration permissions for plugin hooks.

## Best Possible Solution

Define a host-owned, redacted hook provenance and modification-audit contract, distinguish native in-process behavior from isolated-plugin enforcement, then add backwards-compatible per-hook policy with focused runtime, registry, config, docs, and transcript/message tests.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a security hardening and plugin API feature request. Source inspection on current main and v2026.5.12 gives high-confidence proof that hook-chain integrity state, per-handler audit/provenance records, and per-hook allowlists are still absent.

## Solution Assessment

No for checksum-only integrity as a standalone implementation. The maintainable solution needs maintainer decisions on native-plugin trust boundaries, isolated-plugin enforcement, audit redaction, manifest/config policy, and migration compatibility before code is written.

## Security Review

Status: needs_attention

Summary: This is security-sensitive plugin-runtime design work and needs maintainer trust-boundary, audit-redaction, and compatibility decisions before implementation.

Concerns:

- **[medium] Define the native plugin trust boundary before checksums:** `docs/plugins/architecture.md:444`
  - body: Native plugins run in-process with Gateway-level capabilities, so checksums alone may not defend against a malicious loaded plugin unless maintainers specify what boundary is protected and how isolated-plugin IPC should differ.
  - confidence: 0.9
- **[medium] Specify provenance and audit semantics before enforcement:** `src/plugins/hooks.ts:577`
  - body: Current modifying hooks can rewrite results or messages sequentially without a host-owned modification ledger, so enforcement needs explicit per-handler audit semantics rather than only a final checksum decision.
  - confidence: 0.88
- **[medium] Redact hook audit content before storage or logs:** `docs/plugins/hooks.md:229`
  - body: Hook audit records could capture prompts, transcripts, tool output, file paths, or credential-adjacent data unless the schema is bounded and redacted before persistence or logging.
  - confidence: 0.82
- **[medium] Plan per-hook policy migration carefully:** `src/config/types.plugins.ts:3`
  - body: Current policy only gates broad prompt and conversation hook categories, so a new per-hook allowlist needs defaults, diagnostics, docs, and compatibility tests to avoid silently breaking existing plugins.
  - confidence: 0.8
