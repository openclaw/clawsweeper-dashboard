## Summary

Keep open. Current main and v2026.6.8 still expose ordered mutable plugin hooks without host-owned hook-chain integrity, per-handler mutation audit records, or a per-hook permission allowlist; the related issues cover narrower outbound verification, sandbox, security-profile, and permission slices rather than replacing this core plugin-runtime security design.

## What This Changes

The issue requests hook-chain integrity protection, mutation audit logging, and per-hook permission policy to reduce silent plugin message tampering.

## Best Possible Solution

Define host-owned hook provenance and mutation-audit semantics, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/registry tests.

## Reproduction Assessment

Yes, at source level. Current main shows ordered mutable hook execution and no hook-chain integrity, mutation-audit, or per-hook allowlist state; this is a reproducible capability gap rather than broken existing documented behavior.

## Solution Assessment

Unclear as proposed. The security goal is valid, but cryptographic checksums alone are not established as the best fix until maintainers choose the plugin trust boundary, audit semantics, compatibility plan, and isolated-plugin relationship.
