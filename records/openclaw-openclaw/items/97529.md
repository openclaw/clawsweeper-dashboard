## Summary

Keep open: current main and the latest release still hard-delete the previous role-ordering-reset transcript after replaying only a bounded tail, while normal reset/rollover paths archive the previous transcript. The report is source-backed, narrowly scoped, and still needs a fix rather than cleanup.

## What This Changes

The issue requests preserving the previous transcript during role-ordering-conflict auto-reset instead of deleting it after replaying only recent user/assistant messages.

## Best Possible Solution

Reuse the existing reset archive lifecycle so role-ordering auto-reset renames the previous transcript to a recoverable `.jsonl.reset.<timestamp>` artifact while the new session keeps the safe replay tail.

## Reproduction Assessment

Yes, at source level. Calling `persistSessionResetLifecycle` with `cleanupPreviousTranscript: true` reaches `cleanupPreviousResetTranscripts`, and current tests assert the previous transcript is missing after that path; I did not run a live provider end-to-end repro.

## Solution Assessment

Yes, the narrow maintainable fix is to replace the role-ordering cleanup unlink with the same reset archive contract already used by `/new`, `/reset`, and lifecycle rollover. A broader rename/never-delete policy belongs to the separate archive-contract issue, not this focused defect.

## Repair Work Prompt

Fix the role-ordering-conflict auto-reset transcript deletion reported in https://github.com/openclaw/openclaw/issues/97529. Current main routes `resetSessionAfterRoleOrderingConflict` through `resetReplyRunSession` with `cleanupTranscripts: true`; `persistSessionResetLifecycle` replays a bounded tail and then `cleanupPreviousResetTranscripts` unlinks the previous transcript. Replace that hard delete with the existing reset archive contract so the previous transcript is renamed/preserved as a `.jsonl.reset.<timestamp>` artifact while the new session still receives the safe replay tail. Keep the corrupt-session recovery intent: the archived transcript must not remain the live transcript. Do not broaden this into the archive naming/never-delete product decision from https://github.com/openclaw/openclaw/issues/56609, do not add config, and do not change retention semantics beyond this path. Update regression tests that currently expect the old transcript path to be missing so they assert one reset archive exists and the new transcript contains only the intended replay tail; include coverage for a transcript longer than the replay cap.
