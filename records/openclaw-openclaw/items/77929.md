## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require bearer auth before revoking handoffs:** `src/gateway/server-methods/talk.ts:753`
  - body: `talk.handoff.join` and the turn methods require the bearer token, but revoke validates only `id` and immediately revokes. Because handoff ids appear in room URLs/metadata, a write-scoped client that learns the id can close another managed room; require the token or an owner/admin authorization path before revoking.
  - confidence: 0.88
- **[P2] Bind managed-room sessions before accepting control:** `src/gateway/server-methods/talk-session.ts:395-406`
  - body: Managed-room records created through `talk.session.create` store the handoff token but no creator connection, then use that stored token for any caller with the session id. Since the session id is the handoff id used in room URLs, enforce creator ownership or require the bearer token before controlling the room.
  - confidence: 0.82
- **[P2] Normalize provider transports before responding:** `src/gateway/server-methods/talk.ts:941-945`
  - body: The new protocol advertises canonical `webrtc` and `provider-websocket`, but this handler returns provider `createBrowserSession()` output unchanged. A provider returning legacy `webrtc-sdp` or `json-pcm-websocket` can produce a successful response that violates the advertised schema; normalize a copied session before comparing and responding.
  - confidence: 0.84
- **[P2] Broadcast managed-room closes through unified close:** `src/gateway/server-methods/talk-session.ts:489`
  - body: `revokeTalkHandoff` returns `session.closed` events and the active client id, but `talk.session.close` ignores them for managed-room sessions. Closing through the unified API leaves the joined room client uninformed while the room is gone; broadcast the returned events like `talk.handoff.revoke`.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: Managed-room revoke/control paths can act on handoff/session ids without bearer-token or connection-owner authorization.

Concerns:

- **[medium] Id-only handoff revocation:** `src/gateway/server-methods/talk.ts:753`
  - body: `talk.handoff.revoke` accepts only a handoff id and revokes immediately. Room ids are exposed through room URLs/metadata while join and turn operations require the bearer token, so a write-scoped client that learns the id can deny service to another managed room.
  - confidence: 0.88
- **[medium] Managed-room control is not connection-bound:** `src/gateway/server-methods/talk-session.ts:395`
  - body: Unified managed-room sessions store the handoff token but no creator connection id, then use that stored token for any caller with the session id. Because the session id is the handoff id, the unified control path bypasses the bearer-token boundary used by `talk.handoff.*` turn methods.
  - confidence: 0.82
