## Summary

Keep open: this draft XL feature PR has a protected maintainer label and still has two source-reproducible blockers in the latest head, including an authorization gap in handoff revocation and a protocol compatibility gap for browser realtime transports.

## What This Changes

The PR adds a shared Talk session runtime and Gateway-managed `talk.session.*` API across realtime relay, transcription relay, managed-room handoff, VoiceClaw, Voice Call, Google Meet, native apps, UI, docs, SDK exports, and changelog.

## Best Possible Solution

Keep the shared Talk controller direction, but require bearer/admin authorization for handoff revocation, normalize legacy provider transports before responses, add focused regression coverage, and complete maintainer review with a live media smoke before merge.

## Reproduction Assessment

Not applicable for the feature as a whole. The two blocking review findings are source-reproducible from the latest PR head: revoke is id-only, and browser provider sessions are returned without transport normalization.

## Solution Assessment

No, not yet. The proposed architecture is plausible, but the current implementation leaves a narrow authorization gap and a protocol compatibility gap that should be fixed before maintainer review treats it as merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require bearer auth before revoking handoffs:** `src/gateway/server-methods/talk.ts:1444`
  - body: `talk.handoff.join` and all turn methods require the bearer token, but revoke validates only `id` and immediately calls `revokeTalkHandoff`. Since the handoff id is exposed in room metadata/URLs, any authenticated client that learns it can close another managed room; require the token or an admin/owner authorization path before revoking.
  - confidence: 0.86
- **[P2] Normalize provider transports before returning sessions:** `src/gateway/server-methods/talk.ts:1636`
  - body: The new protocol/type surface only accepts canonical `webrtc` and `provider-websocket`, but this handler returns provider `createBrowserSession()` output unchanged. A provider or mixed-version plugin that still returns `webrtc-sdp` or `json-pcm-websocket` can get a successful response that violates the advertised schema; normalize a copied session before comparing and responding.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The new handoff revoke RPC invalidates rooms by id without the bearer-token or admin/owner check used by the rest of the handoff flow.

Concerns:

- **[medium] Id-only handoff revocation:** `src/gateway/server-methods/talk.ts:1444`

  - confidence: 0.86
