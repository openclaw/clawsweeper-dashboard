## Summary

Keep open: current main still reads a direct plugin tool object's `name` during registration, while this PR adds a narrow diagnostic-and-skip guard plus regression coverage. Because the PR is draft, maintainer-labeled, and overlaps a broader open hardening PR, the remaining action is maintainer review and normal CI rather than cleanup closure or automated repair.

## What This Changes

The PR catches unreadable direct plugin tool-name getters during `api.registerTool` registration, emits a plugin diagnostic, skips that malformed registration, and tests that later healthy tools still register.

## Best Possible Solution

Land this narrow registration-time guard after maintainer review and CI if it remains separate, or fold the same direct-registration behavior into the broader plugin-tool hardening PR at https://github.com/openclaw/openclaw/pull/89463.

## Reproduction Assessment

Yes, from source inspection: a direct plugin tool object with a throwing `name` getter reaches `src/plugins/registry.ts:626` before any guard and can abort the registration call. I did not execute the test in this read-only review.

## Solution Assessment

Yes: guarding the earliest direct-registration name read is the narrowest maintainable fix for this path while keeping manifest contract validation intact. The returned-tool and descriptor-cache getter cases are separate surfaces covered by https://github.com/openclaw/openclaw/pull/89463.
