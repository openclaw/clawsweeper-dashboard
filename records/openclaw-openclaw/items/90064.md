## Summary

Keep open: current main still has the unguarded direct tool-name read, and this draft maintainer-labeled PR is a focused registry/test fix with no blocking findings. It should continue through normal maintainer review and CI rather than cleanup-close.

## What This Changes

The PR changes plugin registry tool registration so an unreadable direct tool `name` becomes a plugin diagnostic and only that malformed tool is skipped, with a regression test for a later healthy tool.

## Best Possible Solution

Land the narrow registration-time isolation after maintainer review and CI, preserving manifest contract validation and the existing plugin API.

## Reproduction Assessment

Yes. Current main evaluates a plugin tool object's `name` getter during direct registration before guarded validation, so a throwing getter can abort that registration call; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The patch guards the earliest direct-registration name read, records a plugin diagnostic, and returns before the malformed tool enters registry state while healthy later registrations continue; broader request-time unreadable returned-tool hardening belongs to https://github.com/openclaw/openclaw/pull/89463 rather than widening this PR.
