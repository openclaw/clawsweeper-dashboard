## Summary

Keep open. Current main and the latest release still read a direct plugin tool object's `name` during registration, so the requested resilience fix is not already implemented; the maintainer-labeled draft patch appears correct but still needs maintainer review, normal CI or changed-gate proof, and sequencing against the broader related hardening PR.

## What This Changes

The PR catches unreadable direct plugin tool names during `api.registerTool`, emits a diagnostic, skips only that malformed registration, and adds loader regression coverage.

## Best Possible Solution

Land or consciously fold in the narrow registration-time isolation after maintainer review and CI, preserving the regression that a malformed direct tool name cannot block later healthy registrations.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 directly read `tool.name` during direct tool registration, so a throwing getter can abort that registration call; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Guarding the earliest direct-registration name read is the narrow owner-boundary fix for this PR, while broader runtime tool-resolution and descriptor-cache getter hardening belongs in the related broader PR.
