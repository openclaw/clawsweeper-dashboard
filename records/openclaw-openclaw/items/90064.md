I found no line-level correctness defect, but the PR still needs normal maintainer review and broad CI/Testbox validation before merge.

## What This Changes

The PR wraps direct plugin tool-name reads during registration in a diagnostic-producing guard and adds a loader regression that preserves a later healthy tool.

## Best Possible Solution

Land the narrow registration-time isolation fix after maintainer review and CI/Testbox confirmation, keeping manifest contract validation unchanged.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main directly reads `tool.name` during direct plugin tool registration, so a throwing getter can abort the registration call before later plugin code runs.

## Solution Assessment

Is this the best way to solve the issue? Yes: guarding the direct registration-time `tool.name` read is the narrowest owner-boundary fix for this PR's bug, while runtime factory unreadable-tool hardening remains separate in related PRs such as https://github.com/openclaw/openclaw/pull/89463.
