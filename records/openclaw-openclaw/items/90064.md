## Summary

Keep open: current main still reads a direct plugin tool object's `name` during registration, so the PR's narrow guard and regression test address a real remaining plugin-registration failure path; it is also a protected draft maintainer PR, so cleanup should not close it.

## What This Changes

The branch wraps direct plugin tool-name reads in `registerTool`, emits a diagnostic when the name is unreadable, skips that one registration, and adds a loader regression proving a later healthy tool survives.

## Best Possible Solution

Land the narrow registration-time isolation after maintainer review and CI, preserving manifest contract validation and plugin diagnostics without adding a new plugin API or config surface.

## Reproduction Assessment

Yes from source inspection: current main synchronously evaluates a direct plugin tool object's `name` getter during `api.registerTool`, so a throwing getter can abort registration before later healthy calls run. The PR body also reports a focused before/after regression test, but this read-only review did not execute tests.

## Solution Assessment

Yes: guarding the direct registration-time name read is the narrowest owner-boundary fix, and it keeps the existing manifest validation path for readable tool names intact. Loader-level catch-all handling would be broader and would still lose later registrations inside the same plugin `register(api)` function.
