## What This Changes

The PR wraps direct plugin tool `name` reads during `api.registerTool` registration, emits a plugin diagnostic for unreadable names, skips only that malformed registration, and adds loader regression coverage.

## Best Possible Solution

Land or revise the narrow registration-time isolation after maintainer review and CI, while leaving broader runtime tool-resolution and descriptor-cache getter hardening to the related plugin-tool PR.

## Reproduction Assessment

Yes. Source inspection shows current main evaluates a direct plugin tool object's `name` getter during registration before guarded validation, so a throwing getter can abort the registration call; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Guarding the earliest direct-registration name read isolates the malformed registration, records a diagnostic, and preserves the existing manifest contract path; the broader unreadable returned-tool/cache cases belong in the related plugin-tool hardening PR.
