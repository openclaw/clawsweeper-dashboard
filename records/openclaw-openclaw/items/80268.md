## Summary

Keep open: current main and the latest release still expose Chrome MCP existing-session status as transport readiness only, and an open linked PR explicitly closes this issue but has not merged yet.

## What This Changes

Expose a separate browser status signal for Chrome MCP existing-session page-tool readiness instead of reporting only transport-handshake fields.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80280 once it satisfies maintainer review, while keeping the deeper page-tool timeout investigation in https://github.com/openclaw/openclaw/issues/80036 separate.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has a test path where transport is true and isReachable is false, yet status still reports cdpReady without a page-readiness field. I did not run a live macOS Chrome session.

## Solution Assessment

Yes, a separate additive pageReady field is the narrowest maintainable direction because it preserves the existing transport-oriented cdpReady contract while exposing page-tool reachability separately.
