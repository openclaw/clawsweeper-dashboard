## Summary

Keep open: current main and the latest release still derive Chrome MCP existing-session status from transport readiness only, while a page-level reachability helper already exists and the current tests codify the gap. The broader page-tool timeout problem is tracked at https://github.com/openclaw/openclaw/issues/80036, but this narrower status-contract work remains independently actionable.

## What This Changes

The issue requests a separate page-tool readiness signal in `openclaw browser status` for Chrome MCP existing-session profiles.

## Best Possible Solution

Add an additive page-tool readiness field to the browser status contract, derive it from a bounded page probe after Chrome MCP transport succeeds, keep managed CDP behavior compatible, and update CLI/doctor/docs/tests around the new signal.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the existing test exercises transport-up/page-reachability-down and current main still returns `running: true` and `cdpReady: true` without any page-readiness field. I did not run a live macOS Chrome session.

## Solution Assessment

Yes, a separate additive readiness field is the narrowest maintainable fix for the status contract because it avoids overloading the existing transport-oriented `cdpReady` field. The page probe must be bounded and skipped when transport is down so normal status calls do not become slow or hang-prone.
