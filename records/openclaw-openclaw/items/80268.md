## Summary

Keep open: current main still exposes Chrome MCP existing-session status as transport readiness only, and an open linked PR with closing syntax is the right fix path but has not merged yet.

## What This Changes

Request that `openclaw browser status` expose a page-tool readiness signal for Chrome MCP existing-session profiles instead of reporting only transport readiness.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/80280 that adds a bounded, status-safe `pageReady` signal and leaves the broader page-tool timeout root cause to https://github.com/openclaw/openclaw/issues/80036.

## Reproduction Assessment

Yes, at source level: current main has a test fixture where transport is available and page reachability is false, yet status still reports `running: true` and `cdpReady: true` with no page-readiness field. I did not run the live macOS Chrome workflow.

## Solution Assessment

Yes for the direction: an additive `pageReady` field is the narrow fix because it avoids changing the existing transport-oriented `cdpReady` meaning. The implementation should use a bounded, status-safe page probe before the linked PR merges.
