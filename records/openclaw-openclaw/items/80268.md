## What This Changes

Add a separate browser status signal for Chrome MCP existing-session page-tool readiness instead of relying only on `cdpReady` and `cdpHttp` transport readiness.

## Best Possible Solution

Keep this issue open while https://github.com/openclaw/openclaw/pull/80280 is reviewed; the desired end state is an additive, bounded page-readiness status signal, with the broader page-tool timeout root cause left to https://github.com/openclaw/openclaw/issues/80036.

Yes, at source level: current main maps Chrome MCP status to transport readiness, while the existing test explicitly keeps `cdpReady: true` when page reachability is false. I did not run a live macOS Chrome session.

## Solution Assessment
