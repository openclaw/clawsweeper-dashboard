## Summary

Keep open: current main still exposes Chrome MCP existing-session status from transport readiness only, with no page-tool readiness field, and the linked closing PR at https://github.com/openclaw/openclaw/pull/80280 is still open rather than merged.

## What This Changes

The issue requests a separate page-tool readiness signal in `openclaw browser status` for Chrome MCP existing-session profiles.

## Best Possible Solution

Land the linked PR or an equivalent replacement that adds a bounded `pageReady` status signal, keeps transport status distinct, and updates the browser status tests and any affected CLI/doctor docs.

## Reproduction Assessment

Yes at source level: current main has a test where transport is available but `isReachable` is false, and status still returns `running: true`/`cdpReady: true` with no page-readiness field. I did not perform a live macOS Chrome reproduction.

## Solution Assessment

Yes, the narrow maintainable direction is an additive readiness field derived from a bounded page probe after transport succeeds. The current open PR is the active implementation path, so the issue should remain open until that work merges or is replaced.
