## Summary

Keep this open. Current main still has the same actionable embedded/default-session gap: compaction summaries remain part of the legacy context, while the fresh embedded prompt is submitted plainly; only the CLI reseed path has an explicit `next_user_message` marker that separates prior history from the active ask.

## What This Changes

The issue reports post-compaction topic drift where a new user prompt can be answered as if the older summarized topic is still active.

## Best Possible Solution

Add a generic embedded/default-session context contract that explicitly treats the post-compaction incoming prompt as the active user request and compaction summaries as prior context, with deterministic regression tests and a user-facing changelog entry.

## Reproduction Assessment

No high-confidence live reproduction was established because the wrong-answer symptom depends on model behavior. Source inspection does show the current default embedded path lacks the explicit latest-ask priority marker that the CLI reseed path already uses.

## Solution Assessment

Yes for the keep-open direction: a generic prompt/context assembly contract is narrower and more maintainable than keyword-based topic-switch detection or channel/provider-specific behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/41082 by adding a generic embedded/default-session prompt assembly guarantee: after compaction, the incoming prompt for the current turn must be clearly identified as the active user request while compaction summaries and prior transcript remain prior context. Keep the fix generic: no provider/channel IDs, no keyword topic-switch heuristics, and no new config option. Prefer the existing context-engine/harness or embedded prompt assembly seams over plugin-specific behavior, and avoid duplicating transcript persistence. Add deterministic regression coverage for a legacy embedded run with a `compactionSummary` plus a fresh current prompt, and preserve the existing CLI reseed `next_user_message` behavior. Add a single-line `CHANGELOG.md` entry because the behavior is user-facing. Stop and hand back if the implementation would require a broader context-engine API/product decision.
