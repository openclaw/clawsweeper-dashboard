## Summary

Keep open: the PR targets a real source-reproducible Telegram bug, but the current patch still suppresses media-download feedback too broadly and does not match the session activation decision used by inbound context. The supplied proof is only mocked-handler Vitest output, so it also still needs real Telegram or equivalent bot-to-bot proof before merge.

## What This Changes

The branch gates Telegram media-fetch error replies on `isGroup && requireMention`, adds a focused regression test, and updates the changelog.

## Best Possible Solution

Compute or carry the same effective mention/session decision into the media-fetch error path and suppress only unaddressed mention-only group failures while preserving feedback for DMs, mentioned group messages, reply-to-bot messages, and activation overrides.

## Reproduction Assessment

Yes by source inspection: current main resolves Telegram media and can send the failed-download reply before the downstream no-mention `requireMention` gate runs. I did not run a live Telegram/WSL scenario, so this is source-reproducible rather than live reproduced.

## Solution Assessment

No: the current PR gates on group-level `requireMention` instead of the actual inbound mention/session decision. A narrower maintainable fix should reuse or precompute that canonical decision for non-audio media failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor session activation in the reply gate:** `extensions/telegram/src/bot-handlers.runtime.ts:2411-2420`
  - body: The value threaded into `processInboundMessage` is computed from config/group policy only, but the downstream context also applies `resolveGroupActivation(...)` before `requireMention`. A session switched to mention-only can still post the failed-download reply, while an always-on session can lose useful media-failure feedback.
  - confidence: 0.88
- **[P2] Preserve feedback for addressed media failures:** `extensions/telegram/src/bot-handlers.runtime.ts:1467-1469`
  - body: Returning on every `isGroup && requireMention` media-fetch failure suppresses the chat error even for messages that explicitly mention the bot or reply to a bot message. Those addressed messages are supposed to be handled, so users get no visible failure feedback when the requested media download fails.
  - confidence: 0.87
