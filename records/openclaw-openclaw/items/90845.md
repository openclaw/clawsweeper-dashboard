## What This Changes

The PR replaces the iMessage RPC client's `readline` stdout reader with UTF-8 buffering that splits only on LF and adds regression coverage for raw U+2028/U+2029 plus batched LF responses.

## Best Possible Solution

Land a small plugin-local LF-only framing fix in the shared iMessage RPC client with regression tests and maintainer signoff; live `imsg` proof is a useful final risk reducer, not a reason to move the fix elsewhere.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main uses `readline`, and a focused Node probe shows it splits one valid JSON response containing raw U+2028/U+2029 into unparseable fragments. The PR body also includes before/after terminal output through the actual client over a spawned pipe.

## Solution Assessment

Yes, this is the best bounded fix I found: LF-only framing belongs in the iMessage RPC client owner path and preserves the existing parser, pending-request handling, config, and transport API. A bridge encoder change would be complementary, but OpenClaw should not depend on upstream escaping to parse valid JSON strings.
