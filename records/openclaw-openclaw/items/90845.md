## What This Changes

The PR replaces the iMessage RPC client's readline stdout reader with LF-only UTF-8 buffering and adds regression tests for U+2028/U+2029, batched LF responses, and stale-child stdout.

## Best Possible Solution

Land the focused LF-only reader after maintainer acceptance of the remaining live-`imsg` proof gap, and let the linked issue close when this PR merges.

## Reproduction Assessment

Yes. Current main uses `node:readline` for `imsg rpc --json` stdout, Node's readline source treats U+2028/U+2029 as line endings, and a focused Node probe shows valid whole-buffer JSON splitting into unparseable fragments.

## Solution Assessment

Yes. LF-only buffering is the narrow owner-local fix because the documented `imsg rpc` contract is one JSON object per stdout line; changing the imsg encoder or adding skip-forward recovery would be broader and would not fix current emitted payloads in OpenClaw's reader.
