## Summary

Keep open: current main still lacks session-store artifact recovery, but the latest PR head has blocking correctness and private-store safety issues, and the external real-behavior proof is still not convincing for the current head.

## What This Changes

The PR adds session-store recovery from `.bak` and `.tmp` artifacts in `loadSessionStore`, focused recovery tests, a small validation helper, and an Unreleased changelog entry.

## Best Possible Solution

Keep recovery at the session-loader boundary, but trigger it only for failed or zero-byte primary reads, validate regular non-linked backup/tmp candidates, match the real fs-safe temp artifacts, and self-heal through an atomic 0600 replacement path with passing regression tests.

## Reproduction Assessment

Yes. Current main is source-reproducible: a zero-byte or malformed `sessions.json` is caught by `loadSessionStore` and returns an empty store without attempting sibling recovery.

## Solution Assessment

No. The PR targets the right boundary, but the current diff still recovers after valid empty loads, leaves stale fs-safe test fixtures, reads `.bak` without link validation, and rewrites recovered private state non-atomically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery on failed primary loads:** `src/config/sessions/store-load.ts:175`
  - body: This still runs recovery when the primary file is a valid empty `{}` because the condition includes `Object.keys(store).length === 0`. That resurrects stale `sessions.json.bak` data after an intentional reset, and the added valid-empty test would fail; keep a separate failed/zero-byte primary-read flag and remove the entry-count gate.
  - confidence: 0.97
- **[P2] Reject linked backup artifacts before reading:** `src/config/sessions/store-load.ts:181`
  - body: The `.bak` path is read directly with `readFileSync`, unlike the tmp scan's `lstatSync` guard. A symlinked or hardlinked `sessions.json.bak` in the store directory can be imported as session state before any regular-file check.
  - confidence: 0.88
- **[P2] Align fs-safe tmp fixtures with the matcher:** `src/config/sessions/store-load.recovery.test.ts:87-88`
  - body: The updated matcher accepts `.fs-safe-replace.<pid>.<uuid>.tmp`, but this test creates `.sessions.json...tmp`. It will return an empty store instead of the expected two entries, so update the fixtures to the real fs-safe prefix and cover the fresh/stale choice separately.
  - confidence: 0.96
- **[P2] Rewrite recovered stores atomically:** `src/config/sessions/store-load.ts:250`
  - body: This still uses `fs.writeFileSync` for self-heal, so an interrupted recovery can truncate `sessions.json`, and POSIX mode is not tightened on an existing file. Use the existing atomic 0600 replacement path, or its sync equivalent, before updating cache metadata.
  - confidence: 0.92
