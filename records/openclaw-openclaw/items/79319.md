## Summary

Keep open: the PR targets a real session-store loss path, but the current head still has correctness blockers in recovery gating, artifact validation, tmp staleness handling, and test coverage, with security-sensitive recovery semantics still unresolved.

## What This Changes

The PR changes session-store loading to recover corrupt stores from `.bak` and `.tmp` artifacts, self-heal the primary file, add recovery tests and a validation helper, and update the changelog.

## Best Possible Solution

Keep recovery at the session-loader boundary, but only after corrupt or zero-byte existing primary reads; validate backup and tmp artifacts before parsing, reject fresh in-flight temps, align tests with real temp names, and provide current-head runtime proof.

## Reproduction Assessment

Yes, by source inspection. A zero-byte or malformed `sessions.json` on current `main` is swallowed by `loadSessionStore`, which returns the initialized empty store without consulting recoverable backup or temp artifacts.

## Solution Assessment

No. The PR is aimed at the right boundary, but this implementation still recovers from cases it should not, reads unvalidated backups, accepts fresh temp files, and has tests that miss the intended fs-safe branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery to failed primary reads:** `src/config/sessions/store-load.ts:176`
  - body: The condition includes `Object.keys(store).length === 0`, so a missing primary or valid empty `{}` still restores stale `.bak` sessions. That contradicts reset/missing-store behavior and the added tests; only corrupt or zero-byte existing primary reads should enter recovery.
  - confidence: 0.97
- **[P2] Validate backup artifacts before reading them:** `src/config/sessions/store-load.ts:182`
  - body: `sessions.json.bak` is parsed with `readFileSync` before any regular-file, symlink, or hardlink validation. A linked backup in the session directory can therefore be imported as trusted private session state.
  - confidence: 0.9
- **[P2] Skip fresh tmp artifacts during recovery:** `src/config/sessions/store-load.ts:218-221`
  - body: The scan accepts every matching `.tmp` file and only sorts by mtime; it never enforces a stale threshold before parsing. If another writer has a fresh temp in progress after a transient primary read failure, the loader can import an in-flight artifact.
  - confidence: 0.86
- **[P2] Make fs-safe tmp tests exercise the fs-safe matcher:** `src/config/sessions/store-load.recovery.test.ts:87-88`
  - body: These fixtures use `.sessions.json.12345.abcdef-1234.tmp`, but the matcher accepts `.fs-safe-replace.<pid>.<uuid>.tmp`. The fs-safe tests currently miss the intended branch, so they can pass or fail for the wrong reason once recovery logic changes.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff introduces private session-store recovery from sibling artifacts without sufficient link validation or race protection.

Concerns:

- **[medium] Backup recovery follows linked artifacts:** `src/config/sessions/store-load.ts:182`
  - body: The backup file is read before `lstatSync`/hardlink checks, so a symlinked or hardlinked `sessions.json.bak` can be parsed as trusted session state.
