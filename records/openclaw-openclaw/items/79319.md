## Summary

Keep open: this is a plausible focused bug fix, but the PR head still restores stale artifacts over valid empty or missing stores, reads backup artifacts before validation, and has tests that miss the actual fs-safe temp naming contract.

## What This Changes

The PR adds load-time session-store recovery from `.bak` and temp artifacts, synchronous self-healing writeback, recovery tests, and a changelog entry.

## Best Possible Solution

Keep recovery at the session-loader boundary, but only after an existing primary store is proven corrupt, and validate every recovery artifact before parsing or trusting it.

## Reproduction Assessment

Yes. By source inspection, current main returns an empty store after zero-byte or malformed primary reads, while the PR’s recovery path and its own new tests expose the remaining bad cases.

## Solution Assessment

No. The loader boundary is the right place, but this PR head is not yet the best fix because it recovers outside corrupt-primary cases and does not apply the same trust checks to backup artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery to corrupt existing primary stores:** `src/config/sessions/store-load.ts:176`
  - body: The gate still includes `Object.keys(store).length === 0`, and `ENOENT` also sets `primaryReadFailed`, so a missing primary or a valid empty `{}` can restore stale `.bak` state. The new tests for missing primary and valid empty store expect the opposite, so this should be split into an explicit existing-corrupt/zero-byte/non-object primary failure signal.
  - confidence: 0.98
- **[P2] Validate backup artifacts before parsing:** `src/config/sessions/store-load.ts:182-183`
  - body: `sessions.json.bak` is read and parsed before any lstat, regular-file, symlink, or hardlink validation. Since this imports private session metadata as trusted recovery state, apply the same candidate validation before reading the backup file.
  - confidence: 0.93
- **[P2] Reject fresh temp recovery candidates:** `src/config/sessions/store-load.ts:218-221`
  - body: Temp candidates are accepted solely by name and file shape, then sorted oldest-first. If the only matching artifact is from an active writer, a transient primary read failure can import an in-flight temp despite the PR’s stated `<10s` freshness mitigation.
  - confidence: 0.88
- **[P2] Use the actual fs-safe temp prefix in recovery tests:** `src/config/sessions/store-load.recovery.test.ts:87-88`
  - body: These fixtures use `.sessions.json.<pid>.<uuid>.tmp`, but the pinned fs-safe atomic helper creates `.fs-safe-replace.<pid>.<uuid>.tmp` and the loader regex matches that prefix. The advertised fs-safe recovery tests currently miss the intended branch and should use the real temp pattern.
  - confidence: 0.96
