## Summary

Keep open: the PR targets a real session-store durability gap, but the latest head still has merge-blocking recovery and security issues in the loader and its new tests do not exercise the fs-safe temp path correctly.

## What This Changes

The PR adds session-store recovery from `.bak` and `.tmp` artifacts in `loadSessionStore`, self-heals the primary store, adds recovery tests and a validation helper, and updates the changelog.

## Best Possible Solution

Keep recovery at the session-loader boundary, but only after a failed primary read of an existing corrupt or zero-byte store; validate all recovery artifacts before parsing, reject fresh temp files, use fs-safe sync atomic writeback, and align tests with real temp names.

## Reproduction Assessment

Yes, by source inspection. On current main, a zero-byte or malformed `sessions.json` falls through to an empty store after read/parse failure, so recoverable sibling artifacts are ignored.

## Solution Assessment

No. The PR is aimed at the right boundary, but this implementation still recovers in cases it should not, parses unvalidated backups, accepts fresh temp artifacts, and has tests that miss the actual fs-safe temp pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery to failed primary reads:** `src/config/sessions/store-load.ts:176`
  - body: The gate still includes `Object.keys(store).length === 0`, so a missing primary or a valid empty `{}` can restore stale `.bak` sessions. That contradicts the reset/missing-store behavior and the added tests; recovery should only run after an existing primary read/parse failure.
  - confidence: 0.97
- **[P2] Validate backups before parsing:** `src/config/sessions/store-load.ts:182-183`
  - body: `sessions.json.bak` is parsed before any `lstatSync`, regular-file, symlink, or hardlink validation. A linked backup in the session directory can therefore be imported as trusted private session state.
  - confidence: 0.91
- **[P2] Reject fresh temp artifacts:** `src/config/sessions/store-load.ts:218-221`
  - body: The temp scan accepts every matching `.tmp` file and only sorts by mtime, so a fresh in-flight writer artifact can be imported after a transient primary read failure. Enforce a staleness threshold before parsing temp recovery candidates.
  - confidence: 0.88
- **[P2] Use real fs-safe temp names in tests:** `src/config/sessions/store-load.recovery.test.ts:87-88`
  - body: These fs-safe fixtures use `.sessions.json.12345.abcdef-1234.tmp`, but the matcher accepts `.fs-safe-replace.<pid>.<uuid>.tmp`. The tests miss the intended branch and the fs-safe recovery cases fail or pass for the wrong reason.
  - confidence: 0.96

## Security Review
