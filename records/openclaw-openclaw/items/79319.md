## Summary

This PR should stay open: current main still has a source-reproducible corrupt-session-store data-loss path, but the current PR head does not fix it safely or consistently enough to merge.

## What This Changes

The PR adds best-effort session-store recovery from `.bak` and `.tmp` sibling artifacts, a validation helper, recovery tests, and an Unreleased changelog entry.

## Best Possible Solution

Keep the recovery feature, but trigger it only after failed or zero-byte primary loads, validate regular sibling candidates matching the actual writer artifacts, choose a valid recovery source, and self-heal through the existing atomic 0600 replacement path.

## Reproduction Assessment

Yes. Current main is source-reproducible: a zero-byte or malformed `sessions.json` falls through `loadSessionStore` and returns `{}` with no recovery attempt.

## Solution Assessment

No. The PR targets the right loader boundary, but the current diff recovers after valid empty loads, misses the real fs-safe temp prefix, follows linked tmp artifacts, and rewrites recovered data in place.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P2] Gate recovery on failed primary loads:** `src/config/sessions/store-load.ts:170`
  - body: This condition also matches a valid `{}` session store, so a stale `.bak` file can resurrect old sessions after an intentional reset or clean empty state. Track whether the primary read failed or was zero-byte instead of using the parsed entry count.
  - confidence: 0.97
- **[P2] Match the real fs-safe temp prefix:** `src/config/sessions/store-load.ts:43`
  - body: Normal session writes call `writeTextAtomic` without a `tempPrefix`, and the pinned fs-safe default creates `.fs-safe-replace.<pid>.<uuid>.tmp`. This matcher only accepts `sessions.json...tmp` and `.sessions.json...tmp`, so it ignores the residual writer artifact the proof claims to recover from.
  - confidence: 0.98
- **[P2] Reject linked recovery artifacts before reading:** `src/config/sessions/store-load.ts:194-200`
  - body: The tmp scan records `mtime` with `statSync` and later reads the path directly, so a matching symlink is followed before any regular-file validation. Use `lstatSync` and reject symlinks or hardlinked candidates before parsing private session data.
  - confidence: 0.94
- **[P2] Rewrite recovered stores atomically:** `src/config/sessions/store-load.ts:225`
  - body: `writeFileSync` overwrites the corrupt store in place, so a crash during self-heal can leave another truncated store; on existing files, `mode` also does not reliably tighten permissions. Reuse the existing atomic 0600 replacement path used by normal session saves.
  - confidence: 0.9
