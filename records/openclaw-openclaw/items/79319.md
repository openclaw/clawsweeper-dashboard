## Summary

Keep open: the PR targets a real session-store corruption gap, but it is not merge-ready because it misses the actual current fs-safe temp filename shape, its new tests share state across cases, and the supplied proof is test-only.

## What This Changes

The PR adds load-time session-store recovery from `.bak` and residual `.tmp` files, self-heals `sessions.json`, adds recovery tests, and updates the changelog.

## Best Possible Solution

Keep the recovery in the session-store loader, but align it with the current writer's real temp-file contract, isolate each regression test case, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main returns `{}` after zero-byte or malformed `sessions.json` parse failures, and the PR defects are visible from the proposed matcher, current writer path, and pinned fs-safe source.

## Solution Assessment

No. The direction is appropriate, but the implementation misses the actual `.fs-safe-replace.<pid>.<uuid>.tmp` artifacts and the tests are order-dependent; derive recovery from the current writer contract and use fresh per-test state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Match the current writer temp-file names:** `src/config/sessions/store-load.ts:56-59`
  - body: The recovery scanner never considers the temp files produced by the current session-store writer. `saveSessionStore` calls `writeTextAtomic`, which delegates to `replaceFileAtomic` without a temp prefix; the pinned fs-safe implementation leaves crash residues named `.fs-safe-replace.<pid>.<uuid>.tmp`, not `.sessions.json.<pid>.<uuid>.tmp`. A crash before rename would still be skipped and the loader would fall back to `{}`.
  - confidence: 0.93
- **[P2] Isolate each recovery test case:** `src/config/sessions/store-load.recovery.test.ts:43-46`
  - body: The suite reuses one `tmpDir` and `storePath` for all tests and leaves `sessions.json.bak` plus valid tmp files behind. After the `.bak` tests run, cases that expect tmp recovery or no candidates will still see stale candidates and recover the wrong store, so the focused suite is order-dependent. Use a fresh temp dir per test or cleanup in `afterEach`.
  - confidence: 0.95
