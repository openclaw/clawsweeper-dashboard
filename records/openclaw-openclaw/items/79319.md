## Summary

Keep open: the recovery direction is useful for the linked session-store corruption bug, but this PR still has correctness blockers, a concrete symlink-safety concern, and insufficient current-head real behavior proof.

## What This Changes

The PR adds load-time session-store recovery from `.bak` and `.tmp` files, recovery tests, a validation helper, and a changelog entry.

## Best Possible Solution

Keep a narrow load-time recovery path that runs only after failed, zero-byte, or non-object primary loads, validates regular sibling artifacts matching the actual writer names, chooses the freshest valid candidate, and self-heals through atomic 0600 replacement.

## Reproduction Assessment

Yes. Current main is source-reproducible: a zero-byte or malformed `sessions.json` makes `loadSessionStore` catch the read/parse failure and return `{}` without any recovery path.

## Solution Assessment

No. Load-time recovery is the right narrow area, but this diff recovers from valid empty stores, misses the actual `.fs-safe-replace` temp prefix, follows linked candidates, and rewrites the healed store non-atomically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery on failed primary reads:** `src/config/sessions/store-load.ts:170`
  - body: This condition also matches a valid `{}` store, so a stale `.bak` can revive old sessions after an intentional reset. The new valid-empty-store test expects no recovery here, but the implementation will recover anyway.
  - confidence: 0.98
- **[P2] Match the actual fs-safe temp prefix:** `src/config/sessions/store-load.ts:43`
  - body: Current session writes call `writeTextAtomic` without a `tempPrefix`, and pinned `@openclaw/fs-safe` creates `.fs-safe-replace.<pid>.<uuid>.tmp`. This matcher only accepts names derived from `sessions.json`, so real residual writer artifacts are ignored.
  - confidence: 0.98
- **[P2] Reject linked recovery artifacts before reading:** `src/config/sessions/store-load.ts:194-200`
  - body: The tmp scan filters by name, then uses `statSync` and `readFileSync`, both of which follow symlinks. A matching sibling link can import another readable JSON file as session data; validate regular single-link siblings before parsing.
  - confidence: 0.91
- **[P2] Rewrite recovered stores atomically:** `src/config/sessions/store-load.ts:225`
  - body: `writeFileSync` overwrites the corrupt store in place, and `mode` does not tighten permissions on an existing file. A crash during self-heal can leave another truncated store; use the existing atomic 0600 replacement path.
  - confidence: 0.9
