## Summary

Keep open: current main still lacks session-store recovery, but this PR is not mergeable because the recovery trigger, temp artifact matching, linked-file hardening, self-heal write path, and real behavior proof are still not reliable.

## What This Changes

The branch adds load-time session-store recovery from backup/tmp artifacts, a recovery test file, a validation helper, and a changelog entry.

## Best Possible Solution

Keep a narrow session-store recovery change that only runs after a failed or zero-byte primary read, matches current fs-safe and supported legacy artifacts, rejects linked or non-regular candidates before reading, and self-heals through the existing atomic 0600 write path.

## Reproduction Assessment

Yes. Current main is source-reproducible by making `sessions.json` malformed or zero bytes with a valid recovery artifact nearby; the load path catches the read/parse failure and returns an empty store without scanning artifacts.

## Solution Assessment

No. Load-time recovery is a reasonable narrow direction, but this patch still recovers valid empty stores, misses the real fs-safe temp names, follows linked candidates, and rewrites the healed store outside the existing atomic/private write path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery on failed primary reads:** `src/config/sessions/store-load.ts:167`
  - body: When `sessions.json` is valid `{}`, `store` is empty and this branch still reads backup/tmp artifacts, so an intentional reset can resurrect stale sessions and the new valid-empty-store test fails. Track primary read failure or zero-byte state instead of using entry count.
  - confidence: 0.98
- **[P2] Match fs-safe replace temp names:** `src/config/sessions/store-load.ts:43`
  - body: Current session writes call `writeTextAtomic` without a custom `tempPrefix`, and pinned fs-safe creates `.fs-safe-replace.<pid>.<uuid>.tmp`. This matcher only accepts `sessions.json...` and `.sessions.json...`, so interrupted current writers are still ignored.
  - confidence: 0.98
- **[P2] Reject linked candidates before reading:** `src/config/sessions/store-load.ts:196-202`
  - body: The scan selects names, then uses `statSync`/`readFileSync`, which follow symlinks, despite the PR body claiming `lstatSync` hardening. A matching symlink can import another readable JSON file as the session store; require `lstatSync` regular single-link siblings before reading.
  - confidence: 0.9
- **[P2] Use atomic private writes for self-heal:** `src/config/sessions/store-load.ts:222`
  - body: `fs.writeFileSync(..., { mode: 0o600 })` does not chmod an existing corrupt file and can leave a partial rewrite if the process dies mid-write. Self-heal through the existing atomic write helper or chmod after an atomic replacement.
  - confidence: 0.9
