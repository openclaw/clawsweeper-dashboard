## Summary

Keep open: the PR is aimed at a real session-store recovery gap, but the current patch still recovers from valid empty stores, misses the real fs-safe temp artifact, follows linked tmp candidates, and lacks sufficient real behavior proof.

## What This Changes

The branch adds load-time session store recovery from `.bak` and selected `.tmp` files, self-heals `sessions.json`, adds recovery tests, and updates the changelog.

## Best Possible Solution

Keep the PR open for contributor changes: recover only after a failed or zero-byte read, match `.fs-safe-replace.<pid>.<uuid>.tmp`, reject linked/non-regular artifacts, then add redacted real gateway or filesystem proof.

## Reproduction Assessment

Yes, source-reproducible. Current main loads an empty store after failed reads without consulting `.bak` or tmp artifacts, and current writes use fs-safe default temp names that the PR still does not match.

## Solution Assessment

No. Load-time recovery is a narrow maintainable direction, but this patch needs the correct recovery trigger, the actual dependency artifact pattern, linked-file hardening, and convincing after-fix proof before it is a good solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P2] Gate recovery on a failed read, not an empty valid store:** `src/config/sessions/store-load.ts:170`
  - body: When `sessions.json` is valid `{}`, `store` is empty and this branch still reads `sessions.json.bak`, so stale sessions can be resurrected. The added test at lines 94-99 expects the opposite, so the recovery trigger needs to distinguish a successful empty store from an empty or malformed file.
  - confidence: 0.98
- **[P2] Match fs-safe's actual temp prefix:** `src/config/sessions/store-load.ts:43`
  - body: Current session writes call `writeTextAtomic` without `tempPrefix`, and pinned fs-safe writes `.fs-safe-replace.<pid>.<uuid>.tmp`. This matcher only accepts store-name prefixes, so interrupted current writers are still ignored.
  - confidence: 0.97
- **[P2] Reject linked recovery artifacts before reading:** `src/config/sessions/store-load.ts:194-200`
  - body: The recovery scan follows candidate paths with `statSync`/`readFileSync`, so a symlink named like a recovery tmp can make the gateway ingest another readable JSON object as the session store. Please lstat and accept only the intended regular sibling artifacts before reading.
  - confidence: 0.84
