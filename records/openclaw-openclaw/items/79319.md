## Summary

Keep this PR open: the session-store recovery problem is plausible and source-reproducible on current main, but the current patch still has correctness and private-state safety blockers plus insufficient current-head behavior proof.

## What This Changes

The PR modifies `loadSessionStore` to recover from `.bak` and `.tmp` artifacts, adds focused recovery tests and a validation helper, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep recovery at the session-loader boundary, but only recover from corrupt or zero-byte existing primary stores, validate backup/tmp artifacts before reading, ignore fresh in-flight temps, align fixtures with real fs-safe temp names, and prove the behavior on the PR head.

## Reproduction Assessment

Yes, by source inspection. A zero-byte or malformed `sessions.json` on current main is caught by `loadSessionStore`, which then returns the initialized empty store without attempting backup or tmp recovery.

## Solution Assessment

No. The PR targets the right boundary, but the current diff still recovers in cases it should not, follows backup links before validation, accepts fresh tmp artifacts, and has mismatched fs-safe tmp tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate recovery to corrupt existing primary stores:** `src/config/sessions/store-load.ts:176`
  - body: This condition still runs recovery when the primary file is a valid empty `{}`, and it also recovers from `.bak` when the primary file is missing. That resurrects stale sessions after an intentional reset and contradicts the new missing-primary test; track an actual corrupt/zero-byte existing primary read instead of using entry count as a trigger.
  - confidence: 0.97
- **[P2] Reject linked backup artifacts before reading:** `src/config/sessions/store-load.ts:182`
  - body: `sessions.json.bak` is parsed with `readFileSync` before any regular-file, symlink, or hardlink validation. A linked backup artifact in the store directory can be imported as trusted session state, unlike the guarded tmp scan.
  - confidence: 0.9
- **[P2] Ignore fresh temp candidates during recovery:** `src/config/sessions/store-load.ts:218-221`
  - body: The scan accepts every matching `.tmp` file and only sorts by mtime; it never checks that a temp file is stale before parsing it. If the main read fails while another writer has a fresh temp in progress, the loader can import an in-flight artifact instead of waiting or falling back safely.
  - confidence: 0.86
- **[P2] Align fs-safe tmp fixtures with the matcher:** `src/config/sessions/store-load.recovery.test.ts:87-88`
  - body: These tests create `.sessions.json.12345.abcdef-1234.tmp`, but the matcher accepts `.fs-safe-replace.<pid>.<uuid>.tmp`. The fs-safe cases will not exercise the intended recovery branch, so update the fixtures to the real prefix and keep the stale/fresh choice covered separately.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff introduces private session-store recovery paths that can import linked backups and fresh in-flight tmp files before sufficient artifact validation.

Concerns:

- **[medium] Backup recovery follows linked artifacts:** `src/config/sessions/store-load.ts:182`
  - body: `sessions.json.bak` is read directly before `lstatSync`/hardlink checks, so a symlinked or hardlinked backup in the session store directory can be parsed as trusted private session state.
  - confidence: 0.9

  - body: The tmp scan accepts any matching temp file without a staleness threshold, which can import an actively written artifact after a transient primary read failure.
  - confidence: 0.84
