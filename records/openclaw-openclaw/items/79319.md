## Summary

Keep open: the PR targets a real session-store data-loss bug, but the current head still has blocking loader, security, and test issues. The runtime proof in the PR body is sufficient; the remaining work is code repair.

## What This Changes

The PR adds load-time session-store recovery from `.bak` and `.tmp` artifacts, self-heals `sessions.json`, adds recovery tests and a validation helper, and updates the changelog.

## Best Possible Solution

Keep recovery at the session-loader boundary, but only after an existing primary store fails to parse or read as corrupt; validate regular non-linked recovery files with real session entries, ignore fresh temps, use the existing sync fs-safe atomic replace path, and align tests with actual temp names.

## Reproduction Assessment

Yes, by source inspection. On current main, a zero-byte or malformed `sessions.json` falls through to an empty in-memory store, so recoverable sibling artifacts are ignored.

## Solution Assessment

No. The PR is aimed at the right loader boundary, but this head still recovers in cases it should not, parses backups before validation, accepts fresh temps, and has fs-safe temp tests that miss the implemented pattern.

  - body: The gate still includes `Object.keys(store).length === 0`, and missing-file reads also set `primaryReadFailed`, so a missing primary or valid empty `{}` can restore stale `.bak` sessions.
