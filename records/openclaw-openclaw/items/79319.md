## Summary

Keep open: the PR targets a plausible session-store recovery gap, but the current head still resurrects backups for valid empty stores, misses the actual fs-safe temp filename, follows linked recovery candidates, uses non-atomic self-heal writes, and lacks sufficient real behavior proof.

## What This Changes

The branch modifies session-store loading to recover from `.bak` and `.tmp` artifacts, adds recovery tests and a validation helper, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep this PR open for a corrected load-time recovery implementation that triggers only on failed or zero-byte reads, matches actual fs-safe artifacts, rejects non-regular candidates, and uses atomic private self-heal writes.

## Reproduction Assessment

Yes, source-reproducible. Current main returns an empty session store after unreadable or malformed `sessions.json`, and the PR's own valid-empty-store test shows a high-confidence failing path in the proposed implementation.

## Solution Assessment

No. Load-time recovery is a reasonable narrow direction, but this patch needs the correct recovery trigger, actual dependency artifact pattern, linked-file hardening, atomic/private writeback, and real setup proof before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P2] Gate recovery on failed reads only:** `src/config/sessions/store-load.ts:170`
  - body: When `sessions.json` is valid `{}`, `store` is empty and this branch still reads `sessions.json.bak`, so an intentional empty/reset store can resurrect stale sessions. Track whether the primary read failed or was zero bytes instead of using entry count; the added valid-empty-store test already expects that behavior.
  - confidence: 0.98
- **[P2] Match the fs-safe temp filenames:** `src/config/sessions/store-load.ts:43`
  - body: Current session writes call `writeTextAtomic` without `tempPrefix`, and pinned fs-safe creates `.fs-safe-replace.<pid>.<uuid>.tmp`. This matcher only accepts store-name prefixes, so interrupted current writers are still ignored by the recovery path.
  - confidence: 0.98
- **[P2] Reject linked recovery artifacts before reading:** `src/config/sessions/store-load.ts:194-200`
  - body: The recovery scan filters names but then uses `statSync` and `readFileSync`, which follow symlinks. A symlink named like a recovery tmp can make the gateway ingest another readable JSON object as session data; use `lstat` and accept only intended regular sibling artifacts before reading.
  - confidence: 0.86
- **[P2] Preserve private mode during self-heal writes:** `src/config/sessions/store-load.ts:225`
  - body: `fs.writeFileSync` does not apply the `mode` option to an existing corrupted or zero-byte file, so the recovered store can keep a looser mode and the new 0600 assertion can fail. Self-heal through the existing atomic/private write path or explicitly chmod after replacement.
  - confidence: 0.9
