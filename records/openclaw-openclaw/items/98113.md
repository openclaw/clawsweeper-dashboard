## Summary

Keep this PR open, but it is not merge-ready: the fix is aimed at the right OpenAI plugin OAuth boundary, yet the current head still fails TypeScript checks at the cloned `Response` body and the posted terminal proof does not show the revised exchange/refresh path running after the change.

## What This Changes

The PR bounds OpenAI Codex OAuth token response reads in the OpenAI plugin runtime and adds Vitest coverage for oversized, malformed, and failed token responses.

## Best Possible Solution

Keep the bounded read inside `postTokenForm` before `release()`, convert the bounded bytes to a DOM-compatible `BodyInit`, and add redacted terminal/log proof from the actual exchange and refresh paths.

## Reproduction Assessment

Yes, by source inspection: current main buffers the guarded OAuth token endpoint response with raw `arrayBuffer()` before callers can apply bounded JSON/text helpers. The PR's introduced build failure is also reproduced by current CI at the changed `new Response(responseBody, ...)` line.

## Solution Assessment

No, not as submitted. The boundary is right, but the implementation must pass BodyInit-compatible bytes to `Response` and the contributor proof should show the revised OAuth exchange/refresh path, not only helper-level behavior.
