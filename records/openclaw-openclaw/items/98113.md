## Summary

Keep open: this is useful OpenAI OAuth availability hardening, but it is not merge-ready because failed exchange/refresh diagnostics still use the 16 MiB response cap and the contributor proof does not visibly exercise both final oversized exchange and refresh paths.

## What This Changes

The PR bounds OpenAI OAuth token response reads in the OpenAI plugin and adds tests for oversized and malformed token responses.

## Best Possible Solution

Keep the pre-release bounded read in `postTokenForm`, switch failed exchange and refresh diagnostics to a small error-body cap, add an oversized failed-response regression test, and require redacted path-level terminal/log proof.

## Reproduction Assessment

Yes. Current main source-reproduces the issue because `postTokenForm` buffers the token endpoint body with raw `response.arrayBuffer()` before downstream parsing can apply a size cap.

## Solution Assessment

No, not as submitted. The fetch-boundary cap is the right fix location, but failed-response diagnostic reads should follow the existing small error-body cap pattern and be proven on both exchange and refresh paths.
