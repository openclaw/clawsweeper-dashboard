## Summary

Keep open: the intent is right, but the patch does not fix the OOM path because `postTokenForm` still buffers the token response with `arrayBuffer()` before the new bounded readers run.

## What This Changes

The PR imports shared provider HTTP bounded readers, uses them for OpenAI Codex OAuth token exchange/refresh success and error bodies, and adds OAuth-flow tests for oversized and malformed responses.

## Best Possible Solution

Apply the bounded JSON/text reads to the live guarded response before `release()`, remove the unbounded `arrayBuffer()` clone, and keep regression coverage that fails if raw token-response buffering is used.

## Reproduction Assessment

Yes from source inspection: the token response is consumed with `response.arrayBuffer()` before the new bounded readers can run, so a streaming oversized response can still be buffered first. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is to move the bounded read into the guarded-fetch lifetime, not to parse a cloned response after an unbounded buffer has already been created.
