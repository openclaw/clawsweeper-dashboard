## Summary

Keep this PR open: current main still lacks config-path suggestions, and the open PR appears to be a useful narrow implementation. The remaining concern is a non-blocking scope question around whether suggestions should use the full runtime config-path inventory instead of only static `FIELD_HELP` keys.

## What This Changes

This PR adds a Levenshtein-based config path suggestion helper with unit tests and wires `openclaw config get/unset` not-found errors to print `Did you mean` hints.

## Best Possible Solution

Land a config-path suggestion feature after deciding the candidate source: either use the same shared runtime schema/UI-hint inventory that powers config schema lookup, or explicitly scope the helper, docs, and tests to static core config keys.

## Reproduction Assessment

Yes for the observable PR behavior. Current main's get/unset not-found branches have a direct source-level path that emits only `Config path not found`, while the PR adds hints for close static paths such as `gateway.prot`.

## Solution Assessment

Unclear. The get/unset integration is narrow and maintainable, but `FIELD_HELP` alone is not the complete current config-path inventory if the feature is meant to cover channel/plugin paths like the PR body describes.

## Review Findings
