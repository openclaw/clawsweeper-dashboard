## Label Justifications

## Summary

The scheduler fix is still necessary and the production change is coherent: current `main` can infer a missed slot from the updated cron expression while comparing it with pre-update run history, and the proposed timestamp survives the canonical SQLite state codec. The PR should remain open because its claimed real behavior proof is a static print-only script, while its tests seed the new timestamp directly instead of demonstrating the full `cron.update` → persistence → fresh startup path already shown by the prior attempt.

## What This Changes

The PR adds a persisted cron schedule-activation timestamp, uses it to exclude pre-update slots from both startup catch-up paths, and adds direct-state regression tests plus a static proof script.

## Best Possible Solution

Keep the activation timestamp at the centralized update boundary, reuse the established restart-catchup suite to drive a real `CronService.update` through SQLite and a fresh service start, preserve legacy and genuine missed-slot behavior, and remove the static proof script.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: update a monthly cron expression, retain the old `lastRunAtMs`, then restart before the new next run so catch-up computes a pre-update slot from the new expression. The current PR does not execute that full sequence in its own proof or tests.

## Solution Assessment

Yes for the production direction: one persisted activation timestamp at the existing mutation owner is narrower than changing general catch-up policy. The PR's validation shape is not the best implementation evidence; it should use the established service-level restart test from https://github.com/openclaw/openclaw/pull/92250.
