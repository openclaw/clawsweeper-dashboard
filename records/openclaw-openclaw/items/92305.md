## Summary

The scheduler fix is directionally sound and matches the reporter’s updated evidence, but this branch is not merge-ready because its claimed real behavior proof is a print-only script that never executes the changed code. The durable replay boundary also warrants an integrated public update, SQLite reload, and fresh-service restart demonstration before merge.

Likely related people: @tyler6204 for earlier scheduler reliability work, @steipete for extensive history across the central cron files, and @vincentkoc for recent adjacent cron and upgrade-path work.

## What This Changes

The PR adds a persisted cron schedule-activation timestamp and uses it in both startup catch-up classifiers to ignore inferred slots that predate a schedule or enabled-state update.

## Best Possible Solution

Keep the schedule-activation guard, move the regression into the existing service restart-catch-up surface using CronService.update plus a fresh SQLite-backed service instance, and remove the print-only proof helper.

## Reproduction Assessment

Yes. Current source and the reporter’s second-restart timing provide a high-confidence reproduction model for stale-slot inference, although this branch’s supplied script does not execute that model.

## Solution Assessment

Yes in design direction: recording when the active schedule changed is narrower and safer than weakening normal catch-up or adding another persistence authority, but the submitted proof and test integration are not the best merge-ready form.
