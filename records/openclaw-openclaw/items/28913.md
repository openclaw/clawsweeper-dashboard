## Summary

Keep open: current main hardens memory-lancedb against OpenClaw transport/envelope metadata, but I could not verify current-main or v2026.6.5 coverage for the exact EXTERNAL_UNTRUSTED_CONTENT wrapper-filtering behavior requested here. The issue remains a valid security/product decision rather than a cleanup close.

## What This Changes

Add source-based filtering for memory-lancedb autoCapture so wrapped external untrusted content is not stored as long-term memory.

## Best Possible Solution

Keep this open for maintainer security/product review and resolve it through the memory plugin’s capture policy, preferably aligned with the broader provenance/trust-tagging work rather than adding an isolated brittle config surface by default.

## Reproduction Assessment

Yes, from source inspection: current main emits EXTERNAL_UNTRUSTED_CONTENT wrappers and autoCapture stores sanitized user text after trigger matching, but memory-lancedb has no exact wrapper-marker or captureFilter gate. I did not run a live LanceDB flow because this review is read-only and the source path is enough for triage.

## Solution Assessment

Unclear: the proposed captureFilter is a plausible mitigation, but the best durable fix may be default wrapper rejection or typed provenance shared with related memory-security issues. Because it adds config/policy surface, this should not be treated as a strict autonomous bug fix.
