## Summary

Keep open. Current main still has no `captureFilter` or wrapper/provenance-aware auto-capture guard in `memory-lancedb`, and the related open provenance/cron architecture issues are broader rather than canonical replacements; because this changes persistent-memory security policy and config, it needs maintainer/security design.

## What This Changes

Add source/wrapper-based filtering to `memory-lancedb` auto-capture so content inside `EXTERNAL_UNTRUSTED_CONTENT` blocks is not stored as long-term memory.

## Best Possible Solution

Add a maintainer-approved `memory-lancedb` auto-capture policy that reuses shared external-content/provenance semantics, with config/schema/UI hints, docs, and regression tests preserving normal direct-user preference capture.

## Reproduction Assessment

Yes, at source level. A compact role=user turn containing `EXTERNAL_UNTRUSTED_CONTENT` text with a broad trigger such as `always` reaches `extractUserTextContent` and `shouldCapture` with no wrapper/provenance filter, while the requested `captureFilter` key is absent from strict config parsing.

## Solution Assessment

Unclear as a final design. A wrapper exclusion targets the right memory-poisoning surface, but the durable solution should use maintainer-approved shared boundary/provenance helpers rather than ad hoc marker-name parsing.
