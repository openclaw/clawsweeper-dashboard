## Summary

Keep open: current main and v2026.6.5 still lack any Memory LanceDB `autoCapture` wrapper/provenance filter, and the related open memory-trust issues are broader design threads rather than a merged or canonical replacement for this narrower capture-policy gap.

## What This Changes

Request source-aware filtering for Memory LanceDB autoCapture so wrapped untrusted external content is not silently stored as long-term memory.

## Best Possible Solution

Define a maintainer-approved Memory LanceDB capture trust policy, then align runtime filtering, config/manifest/docs, and regression coverage so wrapped or provenance-marked untrusted content is not auto-stored unintentionally.

## Reproduction Assessment

Yes, source-level. With `autoCapture` enabled, wrapped external content containing a trigger such as `always` can reach the existing `shouldCapture` and store path because no wrapper/provenance guard or accepted `captureFilter` setting exists.

## Solution Assessment

Unclear. `captureFilter.excludeWrapped` is a plausible narrow solution, but a default skip for wrapped external content or a broader provenance-aware persistence rule may be safer and needs maintainer security/product judgment first.
