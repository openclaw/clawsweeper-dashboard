- `impact:session-state`: Automatically captured untrusted text can become durable memory that affects later sessions after the original external context is gone.
- `impact:security`: The issue targets prompt-injection and memory-poisoning content crossing a trust boundary into long-term memory.

## Summary

Keep open: current main and v2026.6.10 still lack a source/provenance gate before memory-lancedb autoCapture stores trigger-matching user text, and the related broader memory-trust issues overlap but do not replace this narrower capture-time gap.

## What This Changes

Request adds source-based filtering for memory-lancedb autoCapture so EXTERNAL_UNTRUSTED_CONTENT-wrapped content is excluded from durable memory capture.

## Best Possible Solution

Define a host-authored provenance/trust fact for memory auto-capture and make untrusted external content ineligible by secure default, with any override as explicit maintainer-approved policy.

## Reproduction Assessment

Yes, at source level: current main extracts user text, sanitizes it, runs shouldCapture, and stores trigger-matching content without a source/provenance gate. I did not run a live poisoning scenario because the checkout is read-only and the remaining blocker is security/product policy.

## Solution Assessment

No, not exactly as proposed. Excluding untrusted auto-capture is the right goal, but a wrapper-name config is weaker than a typed host-authored provenance/trust signal with a secure default chosen by maintainers.
