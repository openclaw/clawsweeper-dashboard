## Label Justifications

- `P2`: The issue identifies a real, bounded persistent-memory security gap that needs design work but is not an emergency core-runtime failure.
- `impact:security`: Attacker-controlled external content can cross a trust boundary into durable memory without a structural source check.
- `impact:session-state`: Automatically captured content can influence future sessions after the original untrusted context is gone.

## Summary

Current main still lacks a structural trust check for externally wrapped content before memory-lancedb auto-capture. The issue should remain open for a maintainer security/product decision because the proposed opt-in wrapper list is narrower than the broader provenance work but is not the best durable contract.

## What This Changes

Add source-aware filtering so memory-lancedb automatic capture skips content inside untrusted external-content wrappers.

## Best Possible Solution

Introduce a stable plugin-visible provenance or trust fact and reject untrusted external content by default at the pre-storage auto-capture boundary, coordinating the contract with the broader persistent-state provenance design and adding an override only if maintainers explicitly require one.

## Reproduction Assessment

Yes at source level. A user-role message containing a benign capture trigger inside EXTERNAL_UNTRUSTED_CONTENT passes the current sanitizer because that wrapper is not recognized, then reaches embedding and storage.

## Solution Assessment

No. Excluding untrusted auto-capture is the correct goal, but a typed provenance/trust signal with a secure default is more durable than an operator-maintained list of wrapper names.
