## Summary

Keep open: current main and v2026.6.10 still lack a source/provenance gate before memory-lancedb autoCapture stores trigger-matching external text, and the broader trust/provenance issues overlap without replacing this narrower autoCapture gap.

## What This Changes

Request source/provenance filtering so memory-lancedb autoCapture skips EXTERNAL_UNTRUSTED_CONTENT-wrapped external content before trigger-based storage.

## Best Possible Solution

Define a typed, host-authored source/provenance trust fact and make untrusted external content ineligible for automatic memory capture by default, with any operator override explicit and reviewed.

## Reproduction Assessment

Yes, at source level: current main extracts user text, sanitizes metadata, runs shouldCapture, and stores trigger-matching content without a source/provenance gate. I did not run a live poisoning scenario because the checkout is read-only and the remaining question is security/product policy.

## Solution Assessment

No, not exactly as proposed. Excluding untrusted auto-capture is the right goal, but a raw wrapper-name config is weaker than a typed provenance/trust signal with a secure default chosen by maintainers.
