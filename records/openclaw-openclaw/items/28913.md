## Label Justifications

- `P2`: The issue describes a real persistent-memory security hardening gap with bounded scope, but it needs design work rather than urgent runtime repair.
- `impact:security`: Externally sourced content can cross a trust boundary into durable memory without a structural source check.
- `impact:session-state`: Automatically captured untrusted text can influence later sessions after the original external context is gone.

## Summary

Keep open: current main and v2026.6.8 still lack a source/provenance gate before memory-lancedb auto-capture stores trigger-matching user text, and the remaining choice is security/product policy rather than cleanup closure.

## What This Changes

The issue requests a source-aware autoCapture filter that skips content wrapped as external untrusted input before memory capture triggers run.

## Best Possible Solution

Define a typed plugin-visible source/provenance trust fact and make untrusted external content ineligible for automatic memory capture by default, with any operator override as explicit reviewed policy.

## Reproduction Assessment

Yes, at source level: current main extracts user text, sanitizes it, runs shouldCapture, and stores trigger-matching content without an external-content or provenance gate. I did not run a live LanceDB poisoning scenario because this is a read-only cleanup review and the remaining question is policy/design.

## Solution Assessment

No, not exactly as proposed. Excluding untrusted auto-capture is the right goal, but a wrapper-name config is weaker than a typed provenance/trust signal with a secure default chosen by maintainers.
