## Label Justifications

- `P2`: The issue describes a real persistent-memory security hardening gap with bounded scope, but it needs design work rather than urgent runtime repair.
- `impact:security`: Externally sourced content can cross a trust boundary into durable memory without a structural source check.
- `impact:session-state`: Automatically captured untrusted text can influence later sessions after the original external context is gone.

## Summary

Keep open: current main and the latest release still lack a source/provenance gate before memory-lancedb auto-capture stores trigger-matching text, and the remaining work needs maintainer security/product direction rather than cleanup closure.

## What This Changes

The issue requests a memory auto-capture filter that can skip untrusted wrapped source content before trigger-based memory capture.

## Best Possible Solution

Define a typed plugin-visible source/provenance trust fact and make untrusted external content ineligible for automatic memory capture by default, with any operator override as explicit reviewed policy.

## Reproduction Assessment

Yes, at source level: current main extracts user text, sanitizes it, runs shouldCapture, and stores trigger-matching content without an external-content/provenance gate. I did not run a live poisoning scenario because the source path is sufficient and the remaining question is security/product policy.

## Solution Assessment

No, not as proposed. Excluding untrusted auto-capture is the right goal, but a wrapper-name config is weaker than a typed provenance/trust signal with a secure default chosen by maintainers.
