## Summary

Keep open. Current main still lacks the requested source- or wrapper-aware autoCapture filter: memory-lancedb does not accept a `captureFilter` config key, the manifest/docs expose no such setting, and the `agent_end` auto-capture path still feeds extracted user text directly into broad trigger matching. Related #29442 and #48519 are broader security/provenance proposals, not a replacement for this narrow memory plugin option.

## What This Changes

Add a memory-lancedb `captureFilter.excludeWrapped` option so autoCapture can skip turns containing configured untrusted wrapper tags such as `EXTERNAL_UNTRUSTED_CONTENT`.

## Best Possible Solution

Implement a backward-compatible memory-lancedb `captureFilter.excludeWrapped` setting in the plugin config parser, manifest schema/UI hints, docs, and auto-capture hook, with focused tests proving wrapped untrusted trigger text is skipped while ordinary direct preference capture still works. Keep the implementation on plugin-owned code and documented SDK security helpers rather than importing core internals directly.

## Reproduction Assessment

Yes. A source-level reproduction is clear: parsing a config with `captureFilter.excludeWrapped` would hit the unknown-key guard, and an `agent_end` message containing wrapped trigger text currently reaches `shouldCapture` without a wrapper-aware filter.

## Solution Assessment

Yes, the requested opt-in filter is a narrow maintainable fix for this specific memory poisoning path. The safest implementation should reuse or add a public security-runtime helper for wrapper detection instead of ad hoc parsing in the plugin.
