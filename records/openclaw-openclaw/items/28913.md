## Label Justifications

- `P2`: The issue identifies a real persistent-memory security gap with bounded scope, but it needs design work rather than emergency runtime repair.
- `impact:security`: Externally sourced content can cross a trust boundary into durable memory without a structural source check.
- `impact:session-state`: Automatically captured untrusted text can influence future sessions after the original external context is gone.

## Summary

Keep open: current main and v2026.6.6 still lack a source/provenance gate before memory-lancedb auto-capture stores trigger-matching text, but the proposed wrapper-name config needs maintainer security/product direction before implementation.

## What This Changes

Request adds a memory-lancedb autoCapture filter/config option to skip content wrapped as untrusted external content before trigger-based capture.

## Best Possible Solution

Define a plugin-visible source/provenance trust fact and make untrusted external content ineligible for automatic memory capture by default, adding any operator override only as explicit reviewed policy.

## Reproduction Assessment

Yes, at source level: a user text block containing EXTERNAL_UNTRUSTED_CONTENT markers and a benign trigger such as “always” reaches sanitizeForMemoryCapture and shouldCapture, and the plugin has no wrapper/provenance gate before db.store. I did not run a live poisoning scenario because this is a feature/security design request.

## Solution Assessment

No, not as proposed. Excluding untrusted auto-capture is the right goal, but a wrapper-name config is weaker than a typed provenance/trust signal with a secure default chosen by maintainers.
