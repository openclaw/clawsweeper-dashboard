## Label Justifications

- `P2`: This is a meaningful persistent-memory security hardening gap with bounded scope, but it needs design work rather than urgent runtime repair.
- `impact:session-state`: Automatically captured untrusted text can become durable memory that affects later sessions after the original external context is gone.
- `impact:security`: The issue targets prompt-injection and memory-poisoning content crossing a trust boundary into long-term memory.

## Summary

Keep open: current main and v2026.6.8 still lack a source/provenance gate before memory-lancedb autoCapture stores trigger-matching text, and the related broader memory-trust issues do not fully replace this narrower autoCapture gap.

## What This Changes

Requested behavior: add source/tag-based filtering so memory autoCapture can avoid persisting untrusted external content.

## Best Possible Solution

Define a typed, host-authored source/provenance trust fact and make untrusted external content ineligible for automatic memory capture by default, with any operator override as explicit reviewed policy.

## Reproduction Assessment

Yes, at source level: current main extracts user text, sanitizes metadata, runs shouldCapture, and stores trigger-matching content without a source/provenance gate. I did not run a live poisoning scenario because the source path is sufficient and the remaining question is security/product policy.

## Solution Assessment

No, not exactly as proposed. Excluding untrusted auto-capture is the right goal, but a wrapper-name config is weaker than a typed provenance/trust signal with a secure default chosen by maintainers.
