## Summary

Keep open: current main still has a source-reproducible false positive in the gateway service audit for shell-wrapped LaunchAgent commands, and the reported behavior matches the active status/audit path.

## What This Changes

The issue requests that `gateway status` recognize macOS LaunchAgent shell wrappers that ultimately execute the OpenClaw CLI with the `gateway --port` subcommand.

## Best Possible Solution

Teach the service audit to derive the effective gateway command from supported shell-wrapped LaunchAgent argv before deciding that the gateway subcommand is missing, with regression coverage for the reported wrapper form.

## Reproduction Assessment

Yes from source inspection: a `/bin/zsh -lc` LaunchAgent wrapper keeps `gateway --port` inside one shell payload string, while current main only checks for an argv element exactly equal to `gateway`. I did not run a live macOS LaunchAgent in this read-only pass.

## Solution Assessment

Yes: a narrow command-normalization fix in the service audit/status path is the best route. It should avoid broad substring matching and add focused tests for shell wrappers that ultimately `exec` the OpenClaw gateway command.
