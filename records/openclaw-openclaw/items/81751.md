## Summary

Current main still appears to have the reported false positive: the service audit only accepts `gateway` as a standalone argv element, while the reported LaunchAgent puts the real `openclaw ... gateway --port ...` command inside a `/bin/zsh -lc` string. The `gateway status` path feeds the parsed LaunchAgent command into that audit, and related searches found only closed platform-specific false-positive reports, not a canonical open fix for this macOS wrapper case.

## What This Changes

The issue requests that `gateway status` recognize shell-wrapped macOS LaunchAgent commands that execute `openclaw gateway` and stop reporting a missing gateway subcommand.

## Best Possible Solution

Add a focused service-audit parser path for macOS shell wrappers that can recognize an embedded `openclaw ... gateway` invocation for diagnostics, with regression coverage for the reported `/bin/zsh -lc` shape and no startup-time migration.

## Reproduction Assessment

Yes, at source level: the reported LaunchAgent shape makes `hasGatewaySubcommand()` return false because `gateway` is inside the shell command string instead of a standalone argv element. I did not establish a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to improve service-audit command recognition and tests for shell-wrapped LaunchAgent commands. It should not broaden service installation policy or rewrite custom LaunchAgents during status checks.
