## Label Justifications

- `P1`: The issue reports a current release aborting active agent turns at the default recovery threshold, causing abrupt user-visible turn loss.
- `impact:message-loss`: The abort terminates the active turn mid-flow and can leave the user without the expected final response until they manually resume.
- `impact:session-state`: The diagnostic session state can remain pinned to `embedded_run:started` despite ongoing live-session work, leading recovery to end the wrong active state.

## Summary

Keep open: current main still has the reported Claude live-session liveness gap, so native/tool-heavy stdout can flow while diagnostic progress remains stuck at `embedded_run:started` until recovery aborts the active embedded run. Open PR #84550 appears to target the right live-session repair, but it is still open and not cleanly ready as the canonical fix.

## What This Changes

The issue asks OpenClaw to count active Claude live/native tool-call activity as diagnostic progress so long tool-heavy embedded runs are not aborted as stalled.

## Best Possible Solution

Land or replace the focused Claude live-session progress fix, with throttled diagnostic progress for real native stream/tool activity and regression coverage that still preserves recovery for genuinely silent hangs.

## Reproduction Assessment

Yes at source level: current main has the same diagnostic path where Claude live native stream events can avoid `run.progress`, then stale active embedded work becomes abort-eligible after the configured threshold; I did not run a live Windows/Claude CLI reproduction.

## Solution Assessment

Yes directionally: emitting throttled live-session diagnostic progress for real raw stdout/native tool or thinking activity is the narrow fix, but #84550 still needs review cleanup and maintainer acceptance before it can resolve the issue.
