## Summary

Keep open: current main still aborts media buffer saves when the post-write `FileHandle.sync()` rejects before rename, and this PR is a narrow code/test/changelog fix for the linked Windows media upload failures. I found no blocking review findings; maintainer action is review, validation, and landing one canonical fix.

## What This Changes

The PR catches `EPERM`/`EACCES` from media temp-file `FileHandle.sync()`, adds regression tests for those sync failures, and adds a Windows Office-file upload changelog entry.

## Best Possible Solution

Land one consolidated media-store fsync access-error fix with regression coverage and changelog, then close #76844 and supersede the sibling Windows fsync PRs.

## Reproduction Assessment

Yes, source-reproducible: current main writes the temp file, opens it read-only, awaits `handle.sync()`, and only renames afterward; an injected `EPERM`/`EACCES` at sync propagates into WebChat's `MediaOffloadError`. I did not run Windows live in this read-only pass.

## Solution Assessment

Yes: catching only access-style sync failures on the best-effort post-write flush preserves other storage errors and directly covers the reported Windows failure class. The added tests cover both `EPERM` and `EACCES`.
