## Summary

Keep open: current main still aborts media saves when `FileHandle.sync()` rejects before rename, and this PR addresses the Windows access-error path with focused code, regression tests, and a changelog entry; I found no blocking review findings.

## What This Changes

The PR catches `EPERM`/`EACCES` from media temp-file `FileHandle.sync()`, adds media-store regression tests for those Windows fsync failures, and adds a changelog entry for Windows Office-file uploads.

## Best Possible Solution

Land one consolidated media-store fsync access-error fix with regression coverage and changelog, then close the linked Windows media issues and sibling PRs as superseded.

## Reproduction Assessment

Yes, source-reproducible: current main awaits `handle.sync()` before rename and WebChat wraps that rejection as the reported `MediaOffloadError`. The linked Windows reports plus injected `EPERM`/`EACCES` tests give a high-confidence path, though I did not run Windows live in this read-only pass.

## Solution Assessment

Yes: the PR handles only access-style sync failures on this best-effort flush path, rethrows other storage errors, and adds targeted regression coverage. It is more complete than the sibling EPERM-only alternatives because it also covers `EACCES`.
