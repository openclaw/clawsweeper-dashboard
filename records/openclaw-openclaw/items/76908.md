## Summary

Current main already implements the central Windows `.docx`/media upload EPERM fix with regression coverage and changelog provenance, so this PR is now redundant. The branch also added EACCES hardening, but the canonical reported failure and shipped user problem were the EPERM fsync path now handled on main.

## What This Changes

This PR changes media temp-file fsync handling to ignore Windows access errors, adds regression tests for EPERM/EACCES sync failures, and adds a changelog entry for the Windows Office-file upload fix.

## Best Possible Solution

Keep the main-branch EPERM media fsync implementation as the canonical fix, and open a narrower follow-up only if an EACCES-specific Windows media failure is confirmed.

## Reproduction Assessment

Yes. The pre-fix source path wrote the temp file, opened it read-only, awaited `handle.sync()`, and only renamed afterward, so an injected EPERM sync failure aborts the save; current main now has a targeted regression test for that path.

## Solution Assessment

Yes for the reported EPERM bug. The current main helper keeps the durability flush best-effort only for the reported Windows access error and preserves other storage failures, which is the narrow maintainable fix for #76844.
