## Summary

Keep open. Current source does not prove the read tool itself loses an 8.7KB result from model context, but it does show a plausible hard 8,000-character live/tool-output event truncation path that can make a medium read result look incomplete.

## What This Changes

The issue reports that the `read` tool started omitting part of a 365-line, 8,737-byte `SKILL.md` after the 2026.5.18 update.

## Best Possible Solution

First pin the failing surface with a focused repro; if it is the live event/display path, align the 8K sanitizer with the configured tool-result cap or add explicit full-output retrieval metadata rather than leaving medium read results silently clipped.

## Reproduction Assessment

No, not for the full agent-context claim. Source inspection gives a medium-confidence reproduction for live/display output truncation because `sanitizeToolResult` clips text over 8,000 characters before tool result events are emitted.

## Solution Assessment

Unclear. The narrow fix would be to align the hard-coded live sanitizer with `toolResultMaxChars`, but maintainers should first decide whether this belongs in that cap path or in the broader full-message reader tracked by https://github.com/openclaw/openclaw/issues/84651.
