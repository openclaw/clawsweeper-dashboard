## Summary

Keep open: the PR fixes a real unbounded successful provider JSON read on current main and has sufficient terminal proof, with no concrete line-level defect found. The remaining blocker is compatibility-sensitive because the changed helper is exported through the public plugin SDK and now rejects successful JSON bodies over 16 MiB by default.

## What This Changes

The PR routes `readProviderJsonResponse` through `readResponseWithLimit` with a 16 MiB default cap and adds regression coverage for under-cap JSON parsing and oversized streamed JSON bodies.

## Best Possible Solution

Land the shared-helper cap after maintainers confirm the 16 MiB default is acceptable for public plugin SDK callers, or adjust and document the exposed helper contract before merge.

## Reproduction Assessment

Yes. Current main and `v2026.6.8` still call `await response.json()` for successful provider JSON bodies, and the PR body includes a negative-control stream harness showing the old path reads the full 64 MiB body.

## Solution Assessment

Yes, this is the right owner boundary because object/array helpers and plugin SDK callers flow through `readProviderJsonResponse`, while the binary sibling already uses the same bounded reader. The only unresolved choice is the maintainer-owned public SDK default cap.
