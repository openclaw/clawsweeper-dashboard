## Summary

Keep open: this is a focused, still-needed fix for an unbounded successful provider JSON read on current main, and the patch looks mechanically correct, but merging it changes the default behavior of a public plugin SDK helper and needs maintainer acceptance of that compatibility tradeoff.

## What This Changes

The PR adds a 16 MiB default byte cap to `readProviderJsonResponse`, parses bounded bytes with `JSON.parse`, and adds streaming JSON regression coverage.

## Best Possible Solution

Land the bounded JSON reader after maintainers explicitly accept the 16 MiB public-helper default, or adjust the SDK contract/default and focused compatibility coverage before merge.

## Reproduction Assessment

Yes. Current main and `v2026.6.8` still read successful provider JSON bodies through unbounded `response.json()`, and the PR body includes a negative-control stream harness showing the old path reads the full 64 MiB body.

## Solution Assessment

Yes, with one maintainer-owned compatibility choice. The shared provider HTTP helper is the right owner boundary because current callers already funnel through it and the binary sibling uses the same bounded reader; the remaining decision is the exported SDK default cap.
