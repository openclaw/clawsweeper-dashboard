## Summary

Keep open: current main and the latest release still use unbounded `response.json()` for provider JSON success bodies, and this PR is a focused fix with adequate terminal proof. There are no blocking code findings, but maintainers should intentionally accept the new default cap because the helper is exported through the plugin SDK.

## What This Changes

The PR routes `readProviderJsonResponse` through the shared bounded response reader with a 16 MiB default cap and adds focused regression coverage.

## Best Possible Solution

Land the shared-helper cap after maintainers confirm the 16 MiB default is the intended plugin SDK contract, preserving the focused provider HTTP regression coverage.

## Reproduction Assessment

Yes. Current main and `v2026.6.8` still call unbounded `response.json()` for provider JSON success bodies, and the PR body includes a negative-control stream harness showing the old path reads the full 64 MiB body.

## Solution Assessment

Yes, this is the right owner boundary because shared provider JSON callers flow through `readProviderJsonResponse` and the binary sibling already uses `readResponseWithLimit`. The remaining question is the maintainer-owned SDK default cap choice.
