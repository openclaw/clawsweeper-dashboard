## Summary

Keep open. Current main and the latest release still parse successful provider JSON bodies with the unbounded Fetch JSON path, and this PR is a focused, well-proven fix; the remaining blocker is maintainer acceptance of the new 16 MiB default on an exported plugin SDK helper.

## What This Changes

The PR routes `readProviderJsonResponse` through the shared bounded response reader with a 16 MiB default cap and adds streamed JSON regression coverage.

## Best Possible Solution

Land the bounded JSON reader once maintainers intentionally accept the 16 MiB exported-helper default, or adjust the SDK contract and compatibility coverage before merge.

## Reproduction Assessment

Yes, by source and PR proof. Current main and `v2026.6.9` call the unbounded Fetch JSON path, and the PR body includes terminal output from a negative-control stream harness showing the old path reads the full 64 MiB body.

## Solution Assessment

Yes, with one maintainer-owned compatibility choice. The shared provider HTTP helper is the right owner boundary because callers already funnel through it and the binary sibling uses the same bounded reader; maintainers should confirm the exported 16 MiB default cap.
