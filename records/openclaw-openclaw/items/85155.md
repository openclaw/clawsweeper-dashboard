## Summary

This PR is still necessary: current main still tells the model to prefer aliases, while the linked report is about exact provider/model requests being rewritten across providers. The patch aligns the model-facing prompt with the existing resolver behavior that only resolves aliases for unqualified model strings; I found no blocking code findings, but maintainers should explicitly accept the provider-routing prompt wording before merge.

## What This Changes

The PR changes the Model Aliases system-prompt sentence and its focused test so exact provider/model overrides are preserved and aliases are described as shortcuts for unqualified model requests.

## Best Possible Solution

Land the narrow prompt/test change if maintainers agree that exact provider/model strings are the canonical routing contract and aliases are only user-facing shortcuts for unqualified requests.

## Reproduction Assessment

Yes for the prompt-generation symptom: current main source emits the old alias-preference sentence when modelAliasLines is present, and the linked report describes provider substitution after that guidance. I did not rerun a full live provider-swap loop in this read-only review.

## Solution Assessment

Yes, this is the narrowest maintainable fix for the reported prompt ambiguity because the existing resolvers already preserve qualified provider/model refs and only resolve aliases for unqualified strings. Broader options like removing aliases entirely or filtering multi-provider overlaps would be larger policy changes.
