## Summary

Close as implemented: the shipped media image timeout config already raises both the explicit image tool provider timeout and the Codex dynamic image-tool watchdog; provider-level timeouts remain a lower-level HTTP guard, not the supported outer tool deadline knob.

## What This Changes

The issue requests an image-tool timeout knob or reuse of provider `timeoutSeconds` for slow local vision models.

## Best Possible Solution

Use the shipped `tools.media.image.timeoutSeconds` or matching `tools.media.image.models[].timeoutSeconds` setting for the image tool, while keeping `models.providers.*.timeoutSeconds` as the provider HTTP request guard.

## Reproduction Assessment

Not applicable as a feature/config request. I did not run the private LiteLLM model, but source, tests, docs, and release checks verify that the supported image-tool timeout configuration exists and is shipped.

## Solution Assessment

Yes. The existing media-tool timeout contract is the maintainable solution; adding `agents.defaults.image.timeoutSeconds` or making provider HTTP timeout silently control the outer tool watchdog would create overlapping sources of truth.
