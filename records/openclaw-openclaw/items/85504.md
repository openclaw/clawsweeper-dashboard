## Summary

The requested image-tool timeout is already supported on current main and in the latest release through the media-understanding image timeout settings, so this can close with guidance to use that shipped config path rather than provider-level model timeouts.

## What This Changes

The issue requests a configurable timeout for the image tool, or reuse of provider timeout settings, for slow local vision models.

## Best Possible Solution

Keep the shipped media-specific timeout contract and direct slow local vision setups to use tools.media.image.timeoutSeconds or per-image-model timeoutSeconds, while leaving models.providers.*.timeoutSeconds scoped to normal provider model calls.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request, and current source plus docs show the supported timeout path already exists.

## Solution Assessment

Yes. The current media-specific timeout is the maintainable boundary because the explicit image tool is a media-understanding path, while provider-level timeoutSeconds remains the normal model-call HTTP guard.
