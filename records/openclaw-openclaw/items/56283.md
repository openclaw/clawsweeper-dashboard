## Summary

Keep open: the requested plain-text Markdown policy is not implemented on current main, but this branch is incorrect as-is because its default auto-strip path can corrupt renderer/lightweight channels, ignores account-scoped config, and lacks real channel proof.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires outbound delivery to strip Markdown after plain-text sanitization for channels detected as non-Markdown-capable.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips only for true plain-text delivery, preserves renderer-owned/lightweight-format channels, honors account-scoped config, and includes focused tests plus redacted real-channel proof.

## Reproduction Assessment

Yes for source-level PR review. The PR head routes default outbound text through `stripMarkdown()` based on channel capability metadata, and the WhatsApp, Matrix, account-scope, fenced-code, and underscore-regression paths are visible from the diff plus current-main source; no live channel delivery proof was established.

## Solution Assessment

No, not as-is. The direction is useful, but the branch needs a current-main rebase, capability-aware policy, account-scoped config resolution, safer stripping semantics, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` falls through to strip by default, but WhatsApp send converts Markdown into WhatsApp markers before sending. The shared strip pass would erase intended bold, italic, and code formatting before that renderer can preserve it.
  - confidence: 0.93
- **[P1] Honor renderer-owned plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto predicate depends on Markdown capability metadata, but Matrix uses `chunkerMode: "markdown"` and renders Markdown downstream while its package metadata does not declare `markdownCapable`. A rebased version can strip links, code, and emphasis before Matrix HTML rendering.
  - confidence: 0.9
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:371`
  - body: `resolveChannelMarkdownConfig()` only reads `channels[channel].markdown`, while existing Markdown table resolution checks `accounts.<id>.markdown` first. Multi-account configs would silently ignore account-specific strip overrides.
  - confidence: 0.88
- **[P1] Avoid mutating identifiers in the default strip path:** `src/infra/outbound/deliver.ts:349`
  - body: This branch makes `stripMarkdown()` a default outbound step, but its single-underscore regex predates current main's word-boundary fix and removes underscores from tokens like `foo_bar_baz`, mutating copyable identifiers and paths.
  - confidence: 0.9
- **[P2] Consume complete fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex accepts only lowercase info strings and can leave suffixes such as `++` from `c++` fences, while also producing extra blank lines in mid-message blocks. Plain-text recipients can receive visibly corrupted code snippets.
  - confidence: 0.86
