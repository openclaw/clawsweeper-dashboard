## Summary

Keep open: current main still lacks the shared `markdown.strip` config and outbound Markdown-stripping policy, and the open tracking issue points to this PR or an equivalent implementation as the active path. The branch is not mergeable as-is because it can regress renderer-owned channels, ignores account-scoped config, and lacks real behavior proof from a plain-text channel.

## What This Changes

The PR adds a `markdown.strip` option, expands `stripMarkdown()`, wires outbound delivery to strip Markdown for non-Markdown-capable channels, and adds tests plus a changelog entry.

## Best Possible Solution

Land a rebased or replacement implementation that strips Markdown only for true plain-text delivery, preserves channel-owned renderers and lightweight formatting, honors account-scoped config precedence, and includes focused tests plus redacted real-channel proof.

## Reproduction Assessment

Yes, for source-level PR review: the diff routes default outbound text through `stripMarkdown()` based on broad channel metadata while current WhatsApp and Matrix paths rely on lightweight or downstream Markdown rendering. I did not establish live channel delivery proof in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the implementation needs capability-aware policy, account-scoped config resolution, safer stripping semantics, a current-main rebase, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve channel-owned Markdown renderers during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown()` defaults to stripping whenever `markdownCapable` metadata is absent. That would downgrade channels such as WhatsApp, which owns lightweight markers after sanitization, and Matrix, which uses a markdown chunker/renderer without manifest metadata.
  - confidence: 0.92
- **[P2] Resolve strip config through account scope:** `src/infra/outbound/deliver.ts:366-371`
  - body: The new helper only reads `cfg.channels[channel].markdown`, while existing Markdown config resolution checks `channels.<channel>.accounts.<id>.markdown` first. Multi-account users would have their account-specific strip overrides ignored.
  - confidence: 0.88
- **[P1] Keep identifiers intact in default stripping:** `src/shared/text/strip-markdown.ts:12`
  - body: The branch version of `stripMarkdown()` removes single-underscore spans without word-boundary guards, so applying it by default can turn identifiers like `foo_bar_baz` or path segments into corrupted text. Current main already fixed this helper for TTS/LINE usage, so this branch would reintroduce that behavior on a broader path.
  - confidence: 0.9
- **[P2] Consume full fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase language tags, so fences such as `c++` or `c#` can leak tag remnants, and mid-message blocks can retain an extra blank line around the closing fence. This is visible output corruption once the helper runs in outbound delivery.
  - confidence: 0.86
