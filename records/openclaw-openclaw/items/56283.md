## Summary

Keep open: current main still lacks the requested shared plain-text Markdown policy, and this PR remains a plausible implementation path, but the current branch is not mergeable because source review shows channel-renderer regressions, account-scope gaps, and missing real behavior proof.

## What This Changes

The branch adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires shared outbound delivery to strip Markdown on channels classified as non-Markdown-capable, with tests and a changelog entry.

## Best Possible Solution

Land a rebased or replacement implementation that strips Markdown only for true plain-text delivery, preserves channel-owned renderers and lightweight formatting, honors account-scoped config precedence, and includes focused tests plus redacted real-channel proof.

## Reproduction Assessment

Yes, for source-level PR review: the diff routes default outbound text through `stripMarkdown()` based on broad channel metadata, while current WhatsApp and Matrix paths rely on lightweight or downstream Markdown rendering. I did not establish live channel delivery proof in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the implementation needs a capability-aware policy, account-scoped config resolution, safer stripping semantics, a current-main rebase, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` falls through to auto-strip because WhatsApp is not marked Markdown-capable, but current delivery converts HTML and Markdown into WhatsApp markers before send. The new post-sanitize strip removes those markers by default, so WhatsApp loses intentional formatting.
  - confidence: 0.93
- **[P1] Honor Markdown-rendering plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto predicate only checks channel metadata. Matrix uses `chunkerMode: "markdown"` and renders Markdown downstream, but its manifest does not advertise `markdownCapable`, so auto-strip downgrades links, code, and emphasis before Matrix can render them.
  - confidence: 0.89
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:366-371`
  - body: The new lookup only reads `channels[channel].markdown`, while existing Markdown table resolution checks `accounts.<id>.markdown` first. Multi-account configs would silently ignore account-specific strip overrides.
  - confidence: 0.88
- **[P1] Avoid identifier mutation in the default strip path:** `src/infra/outbound/deliver.ts:345-349`
  - body: The branch helper still strips single-underscore spans inside tokens like `foo_bar_baz`, and this new default delivery call applies it to iMessage/SMS/IRC text. That can corrupt copyable identifiers and paths in outbound messages.
  - confidence: 0.9
- **[P2] Consume complete fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase info strings and leaves the closing-fence newline outside the match. Common fences such as `c++` or `c#` can leak remnants, and mid-message blocks can gain an extra blank line.
  - confidence: 0.9
