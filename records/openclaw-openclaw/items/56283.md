## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve adapter-owned formatting before stripping:** `src/infra/outbound/deliver.ts:344-349`
  - body: This runs `stripMarkdown()` after channel sanitizers have created lightweight formatting markers and before current adapters can render their own formatting. On channels such as WhatsApp, Signal, and current iMessage, auto mode can silently downgrade bold/italic/code or typed-run formatting, so the strip decision needs to preserve adapter-owned formatting paths by default.
  - confidence: 0.93
- **[P1] Preserve renderer-owned Markdown plugin channels:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `auto` strips whenever `isMarkdownCapableMessageChannel()` is false, but current plugin adapters can own Markdown rendering/chunking without complete `markdownCapable` metadata, as Matrix and Mattermost do with `chunkerMode: "markdown"`. Default stripping would downgrade links/code/bold before those adapters render them.
  - confidence: 0.9
- **[P2] Resolve strip settings from account config:** `src/infra/outbound/deliver.ts:367-371`
  - body: `resolveChannelMarkdownConfig()` only reads `cfg.channels[channel].markdown`, while current Markdown config resolution supports `channels.<channel>.accounts.<id>.markdown`. Multi-account users who set different strip behavior per account would have those overrides ignored during delivery.
  - confidence: 0.91
- **[P1] Preserve underscore-safe text when stripping by default:** `src/shared/text/strip-markdown.ts:12`
  - body: The branch broadens `stripMarkdown()` to ordinary outbound iMessage/SMS/IRC text, but it is stale against current main's underscore-safe helper. A rebase must preserve the current behavior that leaves identifiers and URL paths like `foo_bar_baz` intact before auto-stripping is safe.
  - confidence: 0.9
- **[P2] Consume complete fenced-code fences:** `src/shared/text/strip-markdown.ts:9`
  - body: The added fenced-code regex only accepts `[a-z]*` info strings and does not consume the newline after the closing fence, so common fences like ```c++ and mid-message blocks can leak or add extra blank lines in delivered text. The regex and tests should cover full info strings plus adjacent and mid-message fences.
  - confidence: 0.86
- **[P3] Drop the stale generated schema diff:** `src/config/schema.base.generated.ts:1269-1277`
  - body: The PR still changes `src/config/schema.base.generated.ts`, but current main no longer has that generated file and the `unsupportedToolSchemaKeywords` field already lives in the current Zod schema. Keeping this stale generated change makes the branch conflict and mixes unrelated config churn into the formatting feature.
  - confidence: 0.82
