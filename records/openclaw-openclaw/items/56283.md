  - body: `shouldStripMarkdown("whatsapp")` falls through to strip by default, but the existing sanitizer and WhatsApp sender intentionally emit markers such as `*bold*`, `_italic_`, and backtick code for the channel renderer. Running `stripMarkdown()` after that erases formatting the channel owns.
  - confidence: 0.93

  - body: The auto predicate relies on channel markdown metadata alone, so plugin channels that own Markdown rendering through their outbound adapter but lack `markdownCapable` metadata are downgraded before their formatter can handle links, code, and emphasis.
  - confidence: 0.88
- **[P2] Resolve strip config with account scope:** `src/infra/outbound/deliver.ts:371`
  - body: `resolveChannelMarkdownConfig()` only reads `channels[channel].markdown`, while the existing Markdown config contract checks `channels[channel].accounts[accountId].markdown` first. Multi-account operators would have account-specific `strip` overrides ignored.
  - confidence: 0.89
- **[P1] Preserve underscore-safe stripping before broad use:** `src/shared/text/strip-markdown.ts:17`
  - body: This branch puts `stripMarkdown()` on a default outbound path, but its stale single-underscore regex predates current main's identifier-safe behavior and can alter text such as `foo_bar_baz` or URL paths. A rebased implementation must keep the current boundary fix.
  - confidence: 0.9
- **[P2] Consume complete fenced-code fences:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex accepts only lowercase language tags and leaves the newline after the closing fence outside the match, so tags like `c++` can leak suffix text and mid-message blocks can gain extra blank lines in delivered plain text.
  - confidence: 0.86
