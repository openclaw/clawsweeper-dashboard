## Summary

Keep open: the feature is still useful and not implemented on current main, but this external PR is not mergeable as-is because the default auto-strip policy can regress existing renderer-owned channels, account-scoped config is ignored, and no inspectable real behavior proof is present.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires outbound delivery to strip Markdown after plain-text sanitization for channels detected as non-Markdown-capable.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips only for true plain-text delivery, preserves renderer-owned/lightweight-format channels, honors account-scoped config, and includes focused tests plus redacted real-channel proof.

## Reproduction Assessment

Yes for source-level PR review. The PR diff routes default outbound text through `stripMarkdown()` based on capability metadata, and current-main WhatsApp, Matrix, account-scope, fenced-code, and underscore-safety paths make the regressions inspectable without a live channel send.

## Solution Assessment

No, not as-is. The direction is useful, but the patch needs a current-main rebase or replacement, a capability-aware policy, account-scoped config resolution, safer stripping semantics, and real behavior proof before it is the maintainable fix.

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` falls through to strip by default, but current WhatsApp delivery converts Markdown into WhatsApp markers before sending. The shared strip pass would erase intended bold, italic, code, and table formatting before that renderer can preserve it.
  - confidence: 0.93
- **[P1] Honor renderer-owned plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto predicate depends on Markdown capability metadata, but current Matrix outbound uses `chunkerMode: "markdown"` while its package metadata does not declare `markdownCapable`. This can strip links, code, and emphasis before Matrix's renderer handles them.
  - confidence: 0.9
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:371`
  - body: `resolveChannelMarkdownConfig()` only reads `channels[channel].markdown`, while existing Markdown config resolution checks `accounts.<id>.markdown` first. Multi-account configs would silently ignore account-specific strip overrides.
  - confidence: 0.88
- **[P1] Avoid mutating identifiers in the default strip path:** `src/infra/outbound/deliver.ts:349`
  - body: This branch makes `stripMarkdown()` a default outbound step, but the branch's single-underscore regex predates current main's word-boundary fix and can remove underscores from tokens like `foo_bar_baz`. A rebased implementation must preserve the current identifier-safety behavior.
  - confidence: 0.9
- **[P2] Consume complete fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex accepts only lowercase info strings and can leave suffixes such as `++` from `c++` fences, while also producing extra blank lines in mid-message blocks. Plain-text recipients can receive visibly corrupted code snippets.
  - confidence: 0.86
