## Summary

Keep open: current main still lacks the proposed shared outbound Markdown stripping and markdown.strip config surface, while this PR is not merge-ready because the auto-strip policy would regress existing channel renderers, misses account-scoped config, and has no real behavior proof from the contributor.

## What This Changes

The branch adds a markdown.strip config field, expands stripMarkdown, and wires automatic outbound Markdown stripping into shared delivery for channels detected as non-Markdown-capable.

## Best Possible Solution

Land a rebased, capability-aware outbound plain-text policy that preserves channel-owned renderers, honors account-scoped config, and proves real iMessage/plain-text behavior after the fix.

## Reproduction Assessment

Yes for source-level review: current main lacks the shared strip hook/config, and the PR diff routes sanitizer output through stripMarkdown based on a broad channel-capability check. I did not run live channel delivery in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this implementation should not merge until capability detection, lightweight-format channels, account-scoped config, and real behavior proof are handled together.

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` defaults to true because WhatsApp is not marked Markdown-capable, but existing WhatsApp delivery intentionally uses `*bold*`, `_italic_`, and backticks before `markdownToWhatsApp()`. The new post-sanitize strip pass would remove intended WhatsApp formatting by default.
  - confidence: 0.91
- **[P1] Honor Markdown-rendering plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto path only checks channel metadata before stripping. Matrix currently uses `chunkerMode: "markdown"` and renders Markdown into Matrix HTML while its metadata lacks `markdownCapable`, so this would strip links, code, and emphasis before Matrix can produce formatted content.
  - confidence: 0.88
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:369-371`
  - body: The new config lookup only reads `channels[channel].markdown`, but existing Markdown config resolves `accounts.<id>.markdown` before channel-level settings. Multi-account users would not be able to set different strip behavior per account even though the config shape supports account-scoped markdown settings.
  - confidence: 0.88
- **[P1] Avoid mutating identifiers in the default strip path:** `src/infra/outbound/deliver.ts:343-346`
  - body: Wiring `stripMarkdown()` into default delivery exposes the PR-head underscore emphasis regex to iMessage/SMS/IRC text, so normal copyable identifiers like `foo_bar_baz` can be delivered as `foobarbaz`. Either use the current-main boundary-aware helper or add regression coverage before making this automatic.
  - confidence: 0.9
- **[P2] Consume fenced code blocks without leaking markers:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase info strings and leaves the newline after the closing fence outside the match. Common fences like ```c++ can leak text, and mid-message fenced blocks can gain an extra blank line in plain-text output.
  - confidence: 0.9
