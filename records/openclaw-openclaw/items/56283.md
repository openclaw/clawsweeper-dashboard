Overall confidence: 0.93

Full review comments:

- **[P1] Preserve adapter-owned formatting before stripping:** `src/infra/outbound/deliver.ts:344-349`
  - body: This runs `stripMarkdown()` after `sanitizeForPlainText()` has created lightweight markers and before current adapters can render their own formatting. On channels such as WhatsApp, Signal, and current iMessage, auto mode can silently downgrade bold/italic/code or typed-run formatting, so the strip decision needs to preserve adapter-owned formatting paths by default.
  - confidence: 0.93
- **[P1] Preserve renderer-owned Markdown plugin channels:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `auto` strips whenever `isMarkdownCapableMessageChannel()` is false, but current plugin adapters can own Markdown rendering/chunking without complete `markdownCapable` metadata, as Matrix does with `chunkerMode: "markdown"`. Default stripping would downgrade links/code/bold before those adapters render them, so the policy needs adapter or metadata coverage before merge.
  - confidence: 0.88
- **[P2] Resolve strip settings from account config:** `src/infra/outbound/deliver.ts:367-371`
  - body: `resolveChannelMarkdownConfig()` only reads `cfg.channels[channel].markdown`, while current Markdown config resolution supports `channels.<channel>.accounts.<id>.markdown`. Multi-account users who set different strip behavior per account would have those overrides ignored during delivery.
  - confidence: 0.91
- **[P1] Preserve underscore-safe text when stripping by default:** `src/infra/outbound/deliver.ts:344-349`
  - body: The branch broadens `stripMarkdown()` to ordinary outbound iMessage/SMS/IRC text, but it is stale against current main's underscore-safe helper. A rebase must preserve the current behavior that leaves identifiers and URL paths like `foo_bar_baz` intact before auto-stripping is enabled.
  - confidence: 0.86
- **[P2] Consume complete fenced-code fences:** `src/shared/text/strip-markdown.ts:9`
  - body: The new fenced-code regex only accepts `[a-z]*` as the info string and does not consume the newline after the closing fence, so languages like `c++` or `c#` can leak remnants and mid-message fences can leave extra blank lines. Since this helper is being moved onto a delivery path, the fence parser needs to handle the whole info string and surrounding newlines.
  - confidence: 0.9
- **[P3] Drop the unrelated generated schema field:** `src/config/schema.base.generated.ts:1269-1277`
  - body: The regenerated schema adds `unsupportedToolSchemaKeywords`, which is unrelated to Markdown stripping and the generated file no longer exists on current main. Leaving this in the branch makes conflict resolution harder and can accidentally ship another feature's schema change.
  - confidence: 0.78
