## Summary

Keep this PR open for maintainer review: current main still lacks the proposed shared Markdown-stripping config and delivery hook, but the branch has blocking correctness issues around channel-owned formatting, account-scoped config, and the shared `stripMarkdown()` helper.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires automatic outbound Markdown stripping into delivery for auto-detected plain-text channels.

## Best Possible Solution

Land a rebased, capability-aware outbound text strategy that preserves channel-owned renderers, honors account-scoped markdown config, and covers iMessage/BlueBubbles, WhatsApp, Matrix, IRC, and explicit overrides with focused regression tests.

## Reproduction Assessment

Yes. Source inspection plus a pure regex probe reproduces the helper bugs, and the WhatsApp/Matrix regressions follow from the PR's auto-strip path against current channel formatting code.

## Solution Assessment

No. The feature direction is useful, but broad shared auto-strip is not the best fix until capability detection, account-scoped config, and channel-owned Markdown/lightweight formatting contracts are handled together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` defaults to true because WhatsApp is not marked Markdown-capable, but the existing sanitizer and WhatsApp sender intentionally use `*bold*`, `_italic_`, and backticks as WhatsApp formatting. Running the new strip pass after sanitization removes intentional rich text from WhatsApp outbound messages.
  - confidence: 0.91
- **[P1] Honor Markdown renderers before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto path only checks channel metadata before stripping. Matrix currently accepts Markdown and renders it into Matrix HTML in the sender while its plugin metadata lacks `markdownCapable`, so this strips links, code, and emphasis before Matrix can produce `formatted_body`.
  - confidence: 0.87
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:371`
  - body: The delivery lookup only reads channel-level `markdown`, but existing markdown config supports `accounts.<id>.markdown` and the table-mode resolver already honors account scope. Multi-account users would not be able to set different strip behavior per account.
  - confidence: 0.88
- **[P1] Avoid stripping underscores inside identifiers:** `src/shared/text/strip-markdown.ts:17`
  - body: The PR's underscore emphasis regex removes single-underscore pairs inside normal tokens, so default stripping can mutate copyable text like `foo_bar_baz` into `foobarbaz` for iMessage/SMS/IRC delivery.
  - confidence: 0.95
- **[P2] Consume full fenced-code blocks:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase info strings and leaves the newline after the closing fence outside the match. Common fences like ```c++ leak `++`, and mid-message code blocks gain an extra blank line.
  - confidence: 0.94
