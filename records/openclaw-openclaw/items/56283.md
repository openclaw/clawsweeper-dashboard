## Summary

Keep open: the feature direction is useful and current main still lacks the proposed shared `markdown.strip` policy, but the PR is not mergeable as-is because it can regress channel-owned formatting, account-scoped config, and copyable plain-text identifiers, and it still lacks real behavior proof.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown`, and wires automatic Markdown stripping into shared outbound delivery for channels detected as non-Markdown-capable.

## Best Possible Solution

Land a rebased or replacement implementation that preserves channel-owned renderers, follows account-scoped markdown config precedence, keeps current `stripMarkdown` safety fixes, and proves real iMessage/plain-text behavior after the change.

## Reproduction Assessment

Yes for source-level review: current main lacks the shared strip hook/config, and the PR diff routes sanitizer output through `stripMarkdown` based on a broad capability check. I did not run live channel delivery in this read-only review.

## Solution Assessment

No. The direction is useful, but this implementation should not merge until channel renderer preservation, account-scoped config, stale helper behavior, and real behavior proof are handled together.

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` defaults to true because WhatsApp is not marked Markdown-capable, but current WhatsApp delivery intentionally uses `*bold*`, `_italic_`, and backticks before `markdownToWhatsApp()`.   - confidence: 0.92

  - body: The auto path only checks channel metadata before stripping. Matrix declares `chunkerMode: "markdown"` and renders Markdown into Matrix HTML, while its manifest lacks `markdownCapable`, so this would strip links, code, and emphasis before Matrix can format them.
  - confidence: 0.9
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:369-371`
  - body: The new config lookup reads only `channels[channel].markdown`, but existing Markdown config resolution checks `accounts.<id>.markdown` before channel-level settings. Multi-account users would silently lose per-account `markdown.strip` overrides even though the config shape supports account-scoped markdown settings.
  - confidence: 0.88
- **[P1] Keep boundary-safe underscore stripping:** `src/shared/text/strip-markdown.ts:18`
  - body: The PR-head `stripMarkdown()` underscore regex removes single-underscore pairs inside normal tokens, so default auto-strip can turn copyable identifiers like `foo_bar_baz` into `foobarbaz`. Rebase on current main's boundary-aware helper or add equivalent coverage before making this automatic.
  - confidence: 0.91
- **[P2] Consume fenced code blocks without leaking markers:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase info strings and leaves the newline after the closing fence outside the match. Common fences like ```c++ leak text, and mid-message fenced blocks can gain an extra blank line in plain-text output.
