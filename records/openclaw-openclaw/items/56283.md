## Summary

Keep open: current main does not yet provide this PR's configurable delivery-time Markdown stripping, and the linked iMessage feature issue remains open, but this PR is not merge-ready. The head is dirty against main and the diff still has blocking outbound regressions around channel capability detection, account-scoped config, fenced-code parsing, and plain identifier corruption.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires automatic delivery-time Markdown stripping after HTML sanitization for outbound channels detected as non-Markdown-capable, with tests and a changelog entry.

## Best Possible Solution

Rebase onto current main and land a capability-aware outbound text strategy that preserves Markdown-capable and lightweight-markup channels, respects per-account `markdown` config, and keeps narrow fenced-code handling either in the shared helper or the already-related focused PR. The end state should have regression tests for iMessage/BlueBubbles, WhatsApp, Matrix, account overrides, fenced code info strings, and plain identifiers.

## Reproduction Assessment

Yes. The problematic paths are reproducible from the PR diff without live services: the exact regex corrupts fenced `c++` blocks and underscore identifiers, and the WhatsApp regression follows directly from `sanitizeForPlainText()` output plus `shouldStripMarkdown("whatsapp")` returning true under the PR's logic.

## Solution Assessment

No. The requested direction is useful, but this implementation is not the safest maintainable fix because it uses a broad auto-strip default before the channel capability and account-config contracts are complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve lightweight-markup channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` returns true because WhatsApp metadata is not marked Markdown-capable, but the existing plain-text sanitizer deliberately emits WhatsApp/Signal-style markers such as `*bold*`, `_italic_`, and backticks. The new pass then removes that formatting from every WhatsApp outbound message under the default `auto` mode, so rich text that currently survives delivery is silently downgraded.
  - confidence: 0.91
- **[P1] Do not auto-strip Markdown-capable plugin channels:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The default check depends on channel metadata being marked `markdownCapable`, but Matrix's bundled channel currently uses markdown chunking/rendering while its package metadata lacks that flag. With this PR, Matrix messages sent through the shared delivery path can have links, code, and emphasis stripped before the Matrix formatter sees them.
  - confidence: 0.84
- **[P2] Resolve markdown.strip from account config:** `src/infra/outbound/deliver.ts:371`
  - body: This lookup only reads `cfg.channels[channel].markdown`, while channel configs already support `accounts.<id>.markdown` and the existing table-mode resolver honors account scope. Multi-account users cannot set different strip behavior per account even though the new schema surface is nested under the same `markdown` config object.
  - confidence: 0.88
- **[P2] Consume arbitrary fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase letters after the opening fence. A common fence such as ```c++ is only partially matched, leaving `++` in the delivered plain text; use an info-string pattern that consumes everything up to the newline before preserving the code body.
  - confidence: 0.93
- **[P1] Avoid corrupting plain identifiers during auto-strip:** `src/infra/outbound/deliver.ts:349`
  - body: This new default call applies `stripMarkdown()` to ordinary outbound text on non-Markdown channels. On the PR head, that helper removes single underscores inside normal tokens, so messages containing `foo_bar_baz` or URL/path segments can be delivered as changed text that users cannot copy correctly.
  - confidence: 0.9
