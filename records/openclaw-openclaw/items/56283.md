Full review comments:

- **[P1] Preserve lightweight-format channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` falls through to strip by default, but the existing sanitizer intentionally emits WhatsApp/Signal-style markers such as `*bold*`, `_italic_`, and backtick code. Running `stripMarkdown()` after that erases formatting the channel renderer owns.
  - confidence: 0.93
- **[P1] Honor renderer-owned plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto predicate relies on channel markdown metadata, but plugin channels such as Matrix can own Markdown rendering through their outbound adapter while lacking `markdownCapable` metadata. That downgrades links, code, and emphasis before the channel formatter can handle them.
  - confidence: 0.88
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:371`
  - body: `resolveChannelMarkdownConfig()` only reads `channels[channel].markdown`, while existing Markdown config resolution checks `channels[channel].accounts[accountId].markdown` first. Multi-account operators would have account-specific `strip` overrides ignored.
  - confidence: 0.89
- **[P1] Preserve underscore-safe stripping before broad use:** `src/shared/text/strip-markdown.ts:17`
  - body: This branch puts `stripMarkdown()` on a default outbound path, but its stale single-underscore regex predates current main's identifier-safe behavior and can alter text such as `foo_bar_baz` or URL paths. A rebased implementation must keep the current word-boundary fix.
  - confidence: 0.9
- **[P2] Consume complete fenced-code fences:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex accepts only lowercase language tags and leaves the newline after the closing fence outside the match, so tags like `c++` can leak suffix text and mid-message blocks can gain extra blank lines in delivered plain text.
  - confidence: 0.86
