## Summary

Keep this PR open. Current main still does not implement the requested general outbound Markdown stripping, but this branch is not merge-ready because its shared auto-strip pass can regress WhatsApp and Matrix formatting, ignores account-scoped markdown config, and introduces concrete text-corruption bugs in the expanded Markdown stripper.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown()`, wires delivery-time Markdown stripping after HTML sanitization for auto-detected plain-text channels, and adds tests plus a changelog entry.

## Best Possible Solution

Rebase onto current main and land a capability-aware outbound text strategy that preserves Markdown-capable and lightweight-format channels, resolves `markdown.strip` with the same account-scoped precedence as existing markdown table config, and hardens `stripMarkdown()` with focused tests for identifiers, fenced code, WhatsApp, Matrix, BlueBubbles/iMessage, and explicit overrides.

## Reproduction Assessment

Yes. The PR's regex bugs reproduce with a pure local probe, and the WhatsApp/Matrix regressions follow from current-main channel metadata plus the existing sanitizer and formatter call paths without needing live channel credentials.

## Solution Assessment

No. The feature direction is valid, but this implementation is not the best maintainable fix because it applies a broad shared auto-strip before the channel capability and account-config contracts are complete; a safer path is a capability-aware resolver plus channel-specific preservation/conversion where needed.

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` defaults to true because WhatsApp is not marked `markdownCapable`, but the existing sanitizer and WhatsApp sender deliberately use `*bold*`, `_italic_`, and backticks as WhatsApp formatting. Running the new strip pass after sanitization removes intentional rich text from WhatsApp outbound messages.
  - confidence: 0.91
- **[P1] Honor plugin Markdown renderers before stripping:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto path treats channels without `markdownCapable` metadata as plain text, but Matrix currently renders Markdown into Matrix HTML in its sender while its manifest lacks that flag. This strips links, code, and emphasis before the Matrix formatter can produce `formatted_body`.
  - confidence: 0.86
- **[P2] Resolve markdown.strip from account config:** `src/infra/outbound/deliver.ts:371`
  - body: The new delivery lookup only reads `cfg.channels[channel].markdown`, while existing markdown config supports `accounts.<id>.markdown` and the table-mode resolver already honors account scope. Multi-account users would not be able to override stripping per account even though the option lives under the same markdown config object.
  - confidence: 0.88
- **[P1] Preserve plain identifiers during stripping:** `src/infra/outbound/deliver.ts:349`
  - body: Applying `stripMarkdown()` by default to non-Markdown channels changes ordinary text such as `foo_bar_baz` into `foobarbaz` with the PR's underscore emphasis regex. That can corrupt identifiers, paths, and copyable command text in outbound iMessage/SMS/IRC messages.
  - confidence: 0.9
- **[P2] Consume full fenced-code blocks:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase language tags and does not consume the newline after the closing fence, so common blocks such as ```c++ leak `++` into output and mid-message fenced blocks gain an extra blank line. This is visible in the new outbound plain-text path.
  - confidence: 0.92
