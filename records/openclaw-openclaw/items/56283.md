## Summary

Keep open: the requested outbound Markdown stripping is still not implemented on current main, but this PR is not mergeable as-is because it lacks real behavior proof and still has source-level regressions in channel policy, config precedence, and Markdown stripping semantics.

## What This Changes

The branch adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires outbound delivery to strip Markdown after HTML sanitization for channels treated as non-Markdown-capable.

## Best Possible Solution

Land a rebased or replacement implementation that strips Markdown only for true plain-text delivery, preserves renderer-owned and lightweight-format channels, honors account-scoped config precedence, and includes focused tests plus redacted real-channel proof.

## Reproduction Assessment

Yes for source-level PR review: the PR head routes default outbound text through `stripMarkdown()` based on broad channel capability metadata, and the WhatsApp, Matrix, account-config, and helper-regression paths are visible in source. I did not establish live channel delivery proof in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the implementation needs capability-aware policy, account-scoped config resolution, a current-main rebase, safer stripping semantics, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` falls through to auto-strip, but WhatsApp delivery converts Markdown into WhatsApp markers before sending. The new shared strip pass would remove those markers by default, so WhatsApp loses intentional formatting.
  - confidence: 0.93
- **[P1] Honor Markdown-rendering plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto predicate relies on channel capability metadata, but renderer-backed plugin channels such as Matrix consume Markdown downstream and are not protected by this PR. Links, code, and emphasis can be stripped before the plugin renderer sees them.
  - confidence: 0.9
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:370-371`
  - body: The new config lookup only reads `channels[channel].markdown`, while existing Markdown table resolution checks `accounts.<id>.markdown` first. Multi-account configs would silently ignore account-specific strip overrides.
  - confidence: 0.88
- **[P1] Avoid identifier mutation in the default strip path:** `src/shared/text/strip-markdown.ts:17`
  - body: This branch's `stripMarkdown()` removes single-underscore pairs inside normal tokens like `foo_bar_baz`. Because the PR makes the helper a default outbound step, it can mutate copyable identifiers and paths unless rebased onto current main's underscore-safe behavior.
  - confidence: 0.9
- **[P2] Consume complete fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase info strings and leaves the closing-fence newline outside the match. Common fences such as `c++` can leak remnants, and mid-message blocks can gain an extra blank line.
  - confidence: 0.86
