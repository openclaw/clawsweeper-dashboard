## Summary

Keep open: the central behavior is still useful and not implemented on current main, but this PR is not merge-ready because it lacks real behavior proof and the current diff would regress channel-owned Markdown/rendering paths.

## What This Changes

The PR adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires outbound delivery to strip Markdown after plain-text sanitization for channels treated as non-Markdown-capable.

## Best Possible Solution

Land a rebased or replacement implementation that strips Markdown only for true plain-text delivery, preserves channel-owned renderers and lightweight formatting, honors account-scoped config precedence, and includes focused tests plus redacted real-channel proof.

## Reproduction Assessment

Yes for source-level PR review: the changed predicate and delivery hook route default outbound text through `stripMarkdown()` based on broad channel metadata, and the account-config and renderer-regression paths are visible in source. I did not establish live channel delivery proof in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the implementation needs capability-aware policy, account-scoped config resolution, a current-main rebase, safer stripping semantics, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve lightweight-format channels during auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` falls through to auto-strip because WhatsApp is not marked Markdown-capable in this branch, but current delivery converts HTML and Markdown into WhatsApp markers before send. The new post-sanitize strip removes those markers by default, so WhatsApp loses intentional formatting.
  - confidence: 0.93
- **[P1] Honor Markdown-rendering plugin channels before auto-strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto predicate only checks the hard-coded channel list in this branch. Matrix uses `chunkerMode: "markdown"` and renders Markdown downstream, so auto-strip can downgrade links, code, and emphasis before Matrix can render them.
  - confidence: 0.9
- **[P2] Resolve markdown.strip with account scope:** `src/infra/outbound/deliver.ts:370-371`
  - body: The new lookup only reads `channels[channel].markdown`, while existing Markdown table resolution checks `accounts.<id>.markdown` first. Multi-account configs would silently ignore account-specific strip overrides.
  - confidence: 0.88
- **[P1] Avoid identifier mutation in the default strip path:** `src/shared/text/strip-markdown.ts:17`
  - body: Applying this branch's `stripMarkdown()` by default on non-Markdown channels can alter legitimate tokens like `foo_bar_baz` because the single-underscore regex is not word-boundary-safe. Rebase onto current main's underscore-preservation behavior before making the helper a default delivery step.
  - confidence: 0.9
- **[P2] Consume complete fenced-code info strings:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase info strings and leaves the closing-fence newline outside the match. Common fences such as `c++` can leak remnants, and mid-message blocks can gain an extra blank line.
  - confidence: 0.86
