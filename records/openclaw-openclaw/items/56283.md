## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve lightweight-format channels during auto strip:** `src/infra/outbound/sanitize-text.ts:86`
  - body: `shouldStripMarkdown("whatsapp")` defaults to true, but `sanitizeForPlainText()` intentionally turns HTML into WhatsApp/Signal-style `*bold*`, `_italic_`, and backtick markers before this pass. Running `stripMarkdown()` after that silently removes formatting that those channels can render, so auto mode needs to exclude or explicitly convert lightweight-markup channels instead of stripping them.
  - confidence: 0.94
- **[P1] Preserve renderer-owned Markdown plugin channels:** `src/infra/outbound/sanitize-text.ts:86`
  - body: The auto decision strips any channel not marked Markdown-capable, but several plugin adapters own Markdown rendering/chunking paths and do not all advertise `markdownCapable`. Default stripping would downgrade their links/code/bold before the adapter can render them, so the decision needs to use the outbound adapter/renderer contract or explicit metadata coverage before merge.
  - confidence: 0.86
- **[P2] Resolve strip settings from account config:** `src/infra/outbound/deliver.ts:370-371`
  - body: `resolveChannelMarkdownConfig()` only reads `cfg.channels[channel].markdown`, while current Markdown settings can live under `channels.<channel>.accounts.<id>.markdown`. Multi-account users who set different strip behavior per account would have those overrides ignored during delivery.
  - confidence: 0.91
- **[P1] Preserve underscore-safe text when stripping by default:** `src/infra/outbound/deliver.ts:345-349`
  - body: This patch makes `stripMarkdown()` run on ordinary outbound iMessage/SMS/IRC text, but the branch's helper can strip single-underscore spans inside identifiers such as `foo_bar_baz` or URL paths. Because users often copy commands and paths from these channels, the rebase needs to preserve current main's underscore-safe regex before broad auto-stripping is enabled.
  - confidence: 0.88
- **[P2] Consume complete fenced-code fences:** `src/shared/text/strip-markdown.ts:9`
  - body: The new fenced-code regex only accepts `[a-z]*` as the info string and does not consume the newline after the closing fence, so languages like `c++`/`c#` can leak remnants and mid-message fences can leave extra blank lines. Since the helper is now on a user-visible delivery path, the fence parser needs to handle the whole info string and surrounding newlines.
  - confidence: 0.9
- **[P3] Drop the unrelated generated schema field:** `src/config/schema.base.generated.ts:1269-1277`
  - body: The regenerated base schema adds `unsupportedToolSchemaKeywords`, which is unrelated to Markdown stripping and is not described by the PR. Leaving unrelated generated config surface in this branch makes review and conflict resolution harder and can accidentally ship another feature's schema change.
  - confidence: 0.78
