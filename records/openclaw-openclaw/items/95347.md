## Summary

## What This Changes

The PR derives QMD-backed `memory_search` deadlines from manager-owned budgets, adds optional memory manager timeout/close hooks, bounds fallback and one-shot cleanup waits, and adds memory-core regression tests.

## Best Possible Solution

Land the focused QMD timeout fix after maintainers accept manager-owned search budgets and the additive SDK hook, with the close-timeout accounting tightened and subprocess cancellation left to the separate QMD cancellation PR.

## Reproduction Assessment

Yes. Current main and v2026.6.9 still wrap `memory_search` in a fixed 15s deadline while docs expose `memory.qmd.limits.timeoutMs` as the QMD search timeout, and the linked issue includes real QMD timing evidence beyond 15s.

## Solution Assessment

Mostly yes. Manager-owned whole-search budgeting is the right boundary because QMD owns collection/search/update/embed timing, but maintainers still need to accept the additive SDK hook and the close helper should use one cleanup deadline.
