## Summary

The PR changes memory-core `memory_search` timeout handling so QMD managers can report a longer whole-search budget, bounds fallback and one-shot cleanup waits, updates the memory-host manager interface, and adds focused tests.

## Best Possible Solution

Make the search deadline derive from the same effective QMD mode used by the actual search, then land only after maintainers accept the longer configured QMD wait and additive manager hooks.

- The optional `MemorySearchManager.getSearchTimeoutMs()` hook and `close(timeoutMs?)` parameter expand the memory-host manager interface and need maintainer acceptance as the lasting contract shape.
- The current budget calculation can diverge from the actual active-memory QMD `qmdSearchModeOverride`, so that user-visible sibling path may still time out under a budget for the wrong mode.
- QMD subprocess cancellation remains separate in https://github.com/openclaw/openclaw/pull/93394, so this PR should not be treated as closing orphaned-process cleanup work.
