## Summary

Keep this PR open: current main and the latest release still use the fixed 15s `memory_search` guard, while this branch provides a plausible QMD-specific budget fix with sufficient real behavior proof. ## What This Changes

This PR derives QMD-backed `memory_search` deadlines from memory-manager budgets, bounds fallback and one-shot cleanup waits, and adds memory-core/SDK regression coverage.

## Best Possible Solution

Land the focused QMD timeout fix after maintainers accept manager-owned search budgets and the additive SDK hook, with the close-timeout accounting tightened or explicitly accepted and subprocess cancellation left to https://github.com/openclaw/openclaw/pull/93394.

Yes. Current main and v2026.6.9 still wrap `memory_search` in a fixed 15s deadline while QMD config/docs expose a longer search timeout, and the linked issue plus PR body include real QMD timing evidence beyond 15s.

## Solution Assessment

Mostly yes. Manager-owned whole-search budgeting is the right boundary because QMD owns collection/search/update/embed timing, but maintainers still need to accept the additive SDK hook and the close helper should use one cleanup deadline.
