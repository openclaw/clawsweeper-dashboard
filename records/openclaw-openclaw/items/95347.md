## Summary

Keep open: the latest head fixes the prior close-timeout accounting and has strong proof, but it still budgets active-memory QMD searches from the manager's configured mode before the per-call override is known, so one supported QMD path can still get the wrong deadline.

## What This Changes

This PR derives `memory_search` deadlines from QMD manager whole-search budgets, adds timeout-aware memory manager hooks, bounds QMD one-shot cleanup, and expands memory-core regression coverage.

## Best Possible Solution

Repair the mode-aware QMD deadline accounting, then have maintainers explicitly accept the additive memory-host hooks and longer configured QMD wait behavior before merge.

## Reproduction Assessment

Yes. Source inspection shows current main still wraps `memory_search` in a fixed 15s deadline while QMD docs and config expose a longer `memory.qmd.limits.timeoutMs` budget, and the PR body includes terminal proof of a delayed QMD-backed search completing after the old guard.

## Solution Assessment

No. The direction is plausible, but it is not the best fix until the whole-search deadline is derived from the effective per-call QMD mode and the memory-host SDK/availability tradeoffs are accepted explicitly.
