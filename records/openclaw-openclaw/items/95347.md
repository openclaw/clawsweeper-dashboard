## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Budget QMD searches from the effective mode:** `extensions/memory-core/src/tools.ts:549-552`
  - body: `activeMemoryDeadline` is computed from `activeMemory.manager.getSearchTimeoutMs()` before `qmdSearchModeOverride` is resolved and passed to `manager.search(...)`. Since `QmdMemoryManager.getSearchTimeoutMs()` budgets from `this.qmd.searchMode`, active-memory sessions can run under a different QMD command than the one used for the deadline, leaving raised QMD timeouts under-budgeted for that path. Please compute the budget from the same effective mode used by the actual search, or conservatively cover override paths.
  - confidence: 0.86
- **[P2] Share the close cleanup deadline across update waits:** `extensions/memory-core/src/memory/qmd-manager.ts:1705-1706`
  - body: `close(5000)` creates a fresh timeout for `pendingUpdate` and then another for `queuedForcedUpdate`. Because a forced sync can be queued while a pending update exists, one-shot CLI cleanup can still wait about 10 seconds even though `closeMemoryManagers` passes a 5 second cleanup grace period. Use one total deadline or pass the remaining budget to the second wait.
  - confidence: 0.82

## Risks / Open Questions

- Active-memory sessions can still use a QMD effective search mode that differs from the manager's configured mode, so raised QMD timeouts can remain under-budgeted for that user-visible path.
- One-shot CLI cleanup passes a 5s timeout, but QMD close can spend that timeout on both pending and queued forced updates before returning.
- Configured QMD searches can intentionally keep an interactive `memory_search` call running far longer than the former 15 second guard when users raise `memory.qmd.limits.timeoutMs`.
- The optional `MemorySearchManager.getSearchTimeoutMs()` hook and `close(timeoutMs?)` parameter expand the memory-host manager interface and need maintainer acceptance as the lasting contract shape.
- QMD subprocess cancellation remains separate in https://github.com/openclaw/openclaw/pull/93394, so this PR should not be treated as closing orphaned-process cleanup work.
