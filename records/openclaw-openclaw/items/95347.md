## Summary

Keep open: the requested timeout fix is still useful and the PR now includes real QMD proof, but current head is incorrect because it calls `getSearchTimeoutMs` as an unbound method, which can throw before QMD search runs. The same line is already failing CI lint, and merge still needs maintainer acceptance of the SDK surface and longer interactive wait.

## What This Changes

The PR changes memory-core `memory_search` to use QMD whole-search deadlines, bounds one-shot QMD cleanup, forwards timeout options through cached/fallback managers, and updates SDK types plus regression tests.

## Best Possible Solution

Bind or call `getSearchTimeoutMs` through the manager, rerun latest-head delayed-QMD proof and focused CI, then have maintainers explicitly accept or revise the SDK timeout/close surface and long-wait QMD behavior.

## Reproduction Assessment

Yes. Source inspection shows current main still wraps `memory_search` in a fixed 15s deadline while QMD config/docs expose a longer `memory.qmd.limits.timeoutMs` budget.

## Solution Assessment

No as submitted. The intended QMD deadline boundary is plausible, but current head invokes the timeout method unbound and can fail before search; after repair, maintainers still need to accept the SDK and availability tradeoffs.
