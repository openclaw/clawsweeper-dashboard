## Summary

Keep open: the branch addresses a real QMD memory_search timeout problem and includes terminal proof, but the current patch still computes part of the active-memory budget from a different QMD mode than the command it executes and can spend the close timeout twice during queued QMD cleanup.

## What This Changes

The PR changes memory-core QMD search deadline handling, adds an optional memory-host manager timeout/close contract, bounds one-shot cleanup, and adds related memory-core and script tests.

## Best Possible Solution

Fix the two timeout accounting bugs on the branch, keep the API additions explicitly optional and documented by focused tests, then have maintainers decide whether the longer QMD search budget is the desired product contract.

## Reproduction Assessment

Yes from source: current shipped and main behavior still route memory_search through a fixed 15s wrapper while QMD manager search and cleanup can exceed or ignore the intended budget on the affected paths.

## Solution Assessment

No: the proposed direction is plausible and well-proven for the main QMD case, but the implementation is not yet the best fix because effective-mode budgeting and total cleanup deadline accounting are still wrong.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Budget the active-memory search from the effective QMD mode:** `extensions/memory-core/src/tools.ts:549-552`
  - body: `activeMemoryDeadline` is computed before `qmdSearchModeOverride` is resolved, but that override is what `manager.search()` actually uses. If a user has a default QMD mode of `query` with active-memory overriding to `search` or the reverse, the patched budget can still use the wrong timeout and either keep the old 15s cap for a slow `query` search or allow too much time for a `search` call. Please compute the deadline from the effective QMD mode passed to the manager.
  - confidence: 0.9
- **[P1] Use one total deadline for queued QMD cleanup:** `extensions/memory-core/src/memory/qmd-manager.ts:1705-1706`
  - body: `close(timeoutMs)` races `pendingUpdate` and `queuedForcedUpdate` separately with the full timeout, so a pending update followed by a queued forced update can take about 2x the requested close cap. The one-shot cleanup path still needs one shared budget across all awaited cleanup work, and a test should assert total elapsed time rather than only that `close(5000)` was called.
  - confidence: 0.88
- **[P3] Isolate the resolver test's Git safe-directory config:** `test/scripts/resolve-openclaw-ref.test.ts:12-14`
  - body: This test adds `/tmp` to the user's global Git `safe.directory` list, which permanently changes the runner or developer account outside the test process and is unrelated to the memory timeout fix. Please isolate the setting with a test-local `GIT_CONFIG_GLOBAL` or repo-scoped config instead of mutating global Git configuration.
  - confidence: 0.8
