## Summary

Keep this PR open: it addresses a real QMD-backed `memory_search` timeout bug with sufficient terminal proof, but the current head still computes the QMD whole-search deadline from the manager's configured mode before applying the active-memory effective mode override, and the SDK/longer-wait tradeoff still needs maintainer acceptance.

## What This Changes

The PR lets memory-core derive QMD `memory_search` deadlines from a manager-reported whole-search budget, bounds fallback and one-shot cleanup waits, extends the memory-host manager interface, and adds focused tests.

## Best Possible Solution

Derive the QMD whole-search deadline from the same effective search mode passed to the actual search, then land only after maintainers accept the longer configured QMD wait and additive manager hooks.

## Reproduction Assessment

Yes. Current main and `v2026.6.9` still use a fixed 15 second `memory_search` wrapper while QMD config and docs expose a longer search timeout, and the PR body includes real QMD-backed proof that a configured search can complete after the old guard.

## Solution Assessment

No, not yet. Manager-owned QMD budgeting is a plausible owner boundary for the main bug, but the implementation should use the effective search mode that the actual QMD search receives and needs maintainer acceptance for the SDK and availability tradeoff.

Full review comments:

- **[P2] Budget QMD searches from the effective mode:** `extensions/memory-core/src/tools.ts:549-552`
  - body: `activeMemoryDeadline` is created from `activeMemory.manager.getSearchTimeoutMs()` before `qmdSearchModeOverride` is resolved and passed to `manager.search(...)`. `QmdMemoryManager.getSearchTimeoutMs()` budgets from `this.qmd.searchMode`, so active-memory sessions that override QMD to `search` or `vsearch` while the manager is configured for another mode can still run under the wrong whole-search budget and time out despite the configured longer timeout. Please compute the deadline from the same effective QMD mode used by the search, or conservatively cover override paths.
