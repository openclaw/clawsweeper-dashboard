## Summary

Keep this PR open: it is a useful candidate with sufficient real QMD proof, but the latest head still computes the active-memory QMD deadline before resolving the effective search-mode override, so some calls can be budgeted for a different QMD lifecycle than they actually run.

## What This Changes

The PR changes memory-core `memory_search` deadlines to use manager-provided QMD whole-search budgets, adds optional memory-host `getSearchTimeoutMs` and `close(timeoutMs)` hooks, bounds one-shot QMD cleanup, and adds focused regression tests.

## Best Possible Solution

Derive the deadline from the same effective QMD mode used by the search, keep the bounded fallback and cleanup behavior, then have maintainers explicitly accept the additive SDK hooks and longer wait semantics.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 still wrap `memory_search` in a fixed 15s deadline while QMD config/docs expose a longer `memory.qmd.limits.timeoutMs` budget; the PR body adds terminal proof of a delayed QMD search completing after 15s on the branch.

## Solution Assessment

No. The direction is plausible, but it is not the best fix until the deadline is derived from the effective QMD mode used for the call and maintainers accept the SDK/availability contract shape.
