## What This Changes

The issue reports that QQBot ignores `QQBOT_DATA_DIR` and still reports/uses the default `.openclaw/qqbot` data path on Windows.

## Best Possible Solution

Keep the issue as the canonical tracker and resolve it by landing a safe linked PR or making a maintainer decision to replace the `QQBOT_DATA_DIR` guidance with the supported path mechanism.

## Reproduction Assessment

Yes, from source inspection: `runDiagnostics()` reports `getQQBotDataDir()` and tells affected Windows users to set `QQBOT_DATA_DIR`, but the path helper never reads that variable. I did not run a live Windows Gateway reproduction.

## Solution Assessment

Unclear until maintainer product review: centralizing the override in the QQBot path helper is the plausible layer, but adding or honoring this env surface needs invalid-value handling, docs/tests, and upgrade compatibility proof.
