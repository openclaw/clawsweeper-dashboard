## Summary

Keep open: current main still recommends `QQBOT_DATA_DIR` in QQBot diagnostics but `getQQBotDataPath()`/`getQQBotDataDir()` ignore it, so the Windows data-directory override bug remains valid and narrow.

## What This Changes

The issue requests that QQBot use `QQBOT_DATA_DIR` as its data directory instead of always storing state under the user home directory.

## Best Possible Solution

Implement the override in the QQBot-owned storage helper, preserve the default and lazy-creation behavior, keep media paths under the existing media root, and align diagnostics, docs, metadata, tests, and changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `QQBOT_DATA_DIR`, then `runDiagnostics()` and QQBot persistence still call helpers that always resolve under `getHomeDir()/.openclaw/qqbot`.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned: centralize `QQBOT_DATA_DIR` handling in `getQQBotDataPath()`/`getQQBotDataDir()` and update the adjacent public/test surfaces that already imply the variable exists.
