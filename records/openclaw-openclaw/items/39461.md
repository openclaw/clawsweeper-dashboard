## Label Justifications

- `P2`: This is a reproducible, bounded QQBot diagnostic/configuration mismatch that misleads affected Windows users without breaking the core runtime.
- `impact:other`: Windows users with non-ASCII or spaced home paths receive remediation guidance that current QQBot code does not implement.

## Summary

Current main still emits the reported ineffective remediation: QQBot diagnostics tells Windows users to set `QQBOT_DATA_DIR`, but no current QQBot runtime code reads that variable. The linked closing PR remains open, so this issue should stay open; however, since active QQBot state now uses plugin SQLite and the old directory is retained only for legacy imports/remaps, the proposed new state-root environment variable is no longer clearly the best fix.

## What This Changes

Make QQBot honor `QQBOT_DATA_DIR` so Windows users can move its reported data directory away from a home path containing Chinese characters or spaces.

## Best Possible Solution

Keep active QQBot state in plugin SQLite and change the Windows diagnostic to recommend an already-supported canonical home/state setting, or remove the warning if the legacy path no longer affects the failing tools; resolve the open PR rather than adding a parallel QQBot-specific state root.

## Reproduction Assessment

Yes. Current main deterministically recommends `QQBOT_DATA_DIR`, while a repository-wide source search shows no code that reads it; no live Windows Gateway run was performed in this read-only review.

## Solution Assessment

No. Adding a QQBot-specific state-root variable would duplicate the canonical SQLite/`OPENCLAW_STATE_DIR` ownership now present on main; a focused diagnostic correction is the narrower maintainable solution unless maintainers explicitly approve a separate legacy-artifact override.
