## Summary

Keep open. Current main still ignores `QQBOT_DATA_DIR` in the QQBot persistent-storage helper while diagnostics recommends that env var, and the affected helper is used for real QQBot state such as sessions, known users, refs, and credential backups. Exact public issue/PR search found no PR implementing the variable, so this remains a narrow valid bug.

## What This Changes

The issue requests that the QQBot plugin honor `QQBOT_DATA_DIR` for its persistent data directory instead of always using `~/.openclaw/qqbot`.

## Best Possible Solution

Implement the env override once in the QQBot persistent-storage path helper, using a trimmed non-empty `QQBOT_DATA_DIR` as the root and preserving existing subpath joining plus the default `~/.openclaw/qqbot` behavior. Then align diagnostics, tests, docs, plugin metadata, and changelog so the documented workaround is actually supported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: set `QQBOT_DATA_DIR`, then `runDiagnostics()` and QQBot persistence still route through `getQQBotDataPath()`, which never reads that env var.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-owned: centralize the env handling in `getQQBotDataPath()`/`getQQBotDataDir()` and update the diagnostic/docs/test surfaces that already reference the same concept.
