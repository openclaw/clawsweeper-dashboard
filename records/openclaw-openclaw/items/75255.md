## Summary

Keep open. Current main has a bundled Xiaomi provider, but it still only wires catalog/auth and Xiaomi TTS; there is no Xiaomi-owned config or request wrapper that appends MiMo's native `web_search` tool. The issue is concrete, provider-scoped, and fits the existing plugin seam rather than needing core architecture work.

## What This Changes

Add Xiaomi MiMo provider support for the native `web_search` tool by exposing plugin config and appending the Xiaomi native tool object to chat-completions requests when enabled.

## Best Possible Solution

Keep the feature Xiaomi-owned: add a validated `plugins.entries.xiaomi.config.webSearch` surface, inject the native tool through the existing provider wrapper path only for supported Xiaomi chat models and the official endpoint behavior, preserve ordinary function tools without duplicates, and document activation plus billing/privacy caveats.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for an absence check: current main has no Xiaomi `webSearch` config schema, no Xiaomi request wrapper that can append `{ type: "web_search" }`, and no Xiaomi docs/tests for native search. Live API verification was not run, but the reporter supplied a tested request and response shape.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable path is a Xiaomi provider-plugin implementation using the existing `wrapStreamFn`-style request rewrite seam, not a core special case or a separate ClawHub plugin.
