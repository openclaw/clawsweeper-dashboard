## Summary

Close this PR as superseded: it addresses the right WhatsApp inbound-audio TTS bug, but the open maintainer PR at https://github.com/openclaw/openclaw/pull/95596 owns the same root cause with an operation-local marker instead of persisting transient delivery state in `SessionEntry`.

## What This Changes

The branch adds a persisted `SessionEntry.steeredInboundAudio` flag, sets it when active steer mode accepts an inbound-audio follow-up, reads and clears it during final dispatch, reserves the key, and adds regression tests.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/pull/95596 as the canonical fix path, keep https://github.com/openclaw/openclaw/issues/76831 open until that or an equivalent operation-local fix lands, and avoid adding a durable `SessionEntry` marker for this transient run fact.

## Reproduction Assessment

Yes. Source inspection on current main shows `currentInboundAudio` is prepared for the steered audio inbound, the active steer queue call does not carry it, and final TTS dispatch still gates from the original text context; I did not run a live WhatsApp timing repro.

## Solution Assessment

No. The durable `SessionEntry` marker is a plausible fix but not the best state boundary; the operation-local marker in https://github.com/openclaw/openclaw/pull/95596 is narrower and avoids stale persisted session state.
