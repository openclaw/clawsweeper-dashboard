## Summary

Keep open: the PR is a useful candidate, but it clears the one-shot audio marker at dispatch start, so a later steered text inbound can erase the audio fact before the original active run delivers its final reply; real WhatsApp/TTS proof is also still absent.

## What This Changes

The PR adds a persisted `SessionEntry.steeredInboundAudio` marker, sets it when an active steer-mode turn receives audio, reads it during default dispatch TTS gating, clears it after reading, reserves the session slot key, and adds regression coverage.

## Best Possible Solution

Keep the focused marker approach only if the marker is cleared at the actual outbound TTS consumption boundary, with coverage for an audio steer followed by another text steer before final delivery and redacted live WhatsApp/TTS proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main prepares `currentInboundAudio`, active steering queues only prompt text, and final TTS dispatch gates on the original context unless a durable marker survives until reply delivery.

## Solution Assessment

No. The persisted marker is a plausible narrow fix, but this patch clears it before the outbound reply actually consumes it; moving the clear to the real TTS consumption boundary is the safer implementation.
