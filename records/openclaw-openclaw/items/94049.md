## Summary

Keep open: the reported WhatsApp/TTS bug is real, but this branch should not merge as-is because it persists a turn-local audio fact in durable session state; the open maintainer PR shows a safer operation-local shape, but it is not merged yet.

## What This Changes

The PR adds a persisted `SessionEntry.steeredInboundAudio` marker, sets it when active steer-mode accepts an inbound audio turn, reads and clears it during dispatch, reserves the key, and adds regression tests.

## Best Possible Solution

Use an operation-local marker set only after accepted steering and read during the same reply operation; merge or adapt the maintainer PR, then require redacted WhatsApp audio-to-TTS proof before landing.

## Reproduction Assessment

Yes. Source inspection shows current main records `currentInboundAudio` before steering, but the active steer queue call drops that fact and final TTS still gates from the original text context; no live WhatsApp run was performed in this read-only review.

## Solution Assessment

No. The PR is a plausible fix for the bug, but not the best shape because the inbound-audio fact belongs to the current reply operation rather than durable `SessionEntry` state; the open operation-local maintainer PR is safer.
