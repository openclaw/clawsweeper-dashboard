## Summary

Keep open: the PR targets the right WhatsApp/TTS bug, but the proposed session marker is only set on the local runner entry while the later dispatch path reloads the session store, so the reported active-steer case can still dispatch as text. The marker also needs a consume/clear lifecycle before it is safe to use as dispatch state.

## What This Changes

The PR adds a `SessionEntry.steeredInboundAudio` flag, sets it when an audio inbound is steered into an active run, reads it during default reply dispatch, reserves the key from plugin slots, and adds a focused unit test.

## Best Possible Solution

Carry the current-run inbound-audio fact through the same session-store/accessor path final dispatch actually reads, and consume or clear it after the affected reply so later text turns do not inherit voice behavior.

## Reproduction Assessment

Yes for a high-confidence source-level path: current main prepares `currentInboundAudio`, active steering sends only prompt text, and final TTS dispatch gates on `inboundAudio`. I did not run a live WhatsApp timing repro in this read-only review.

## Solution Assessment

No. The PR is aimed at the right boundary, but setting a local `SessionEntry` flag without persisting and consuming it through the dispatch-visible state path is not the safest or sufficient fix.
