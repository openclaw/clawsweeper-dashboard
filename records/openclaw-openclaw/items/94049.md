## What This Changes

The PR adds a durable `SessionEntry.steeredInboundAudio` marker, sets it when active steer mode accepts audio, uses it in final TTS dispatch, clears it afterward, reserves the slot key, and adds regression tests.

## Best Possible Solution

Land the operation-local active-reply fix from https://github.com/openclaw/openclaw/pull/95596 or an equivalent non-persistent handoff, then close the canonical issue and this alternate branch.

## Reproduction Assessment

Yes. Source inspection shows `currentInboundAudio` is computed for the later audio inbound, dropped by the active steer queue call, and final TTS still gates from the original text context; I did not run a live WhatsApp timing repro.

## Solution Assessment

No. The PR fixes the observed path, but persisting a one-shot operation fact in `SessionEntry` is not the best layer; the operation-local handoff in https://github.com/openclaw/openclaw/pull/95596 is the safer shape.
