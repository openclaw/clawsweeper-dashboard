## Summary

Keep open. Current main still has the Feishu same-key starvation bug and this PR targets the right queue boundary, but the latest head also makes the generic Feishu queue default time out for existing no-option callers, which can make drive comment notices look successfully processed before their handler finishes.

## What This Changes

Adds a five-minute Feishu sequential-queue timeout/release guard, logs message timeout evictions, adds queue regression tests, and records the fix in the changelog.

## Best Possible Solution

Keep the Feishu message receive timeout release guard, but preserve legacy unbounded semantics for other sequential-queue callers unless they explicitly handle timeout completion.

## Reproduction Assessment

Yes. Current main can be source-reproduced by holding the first same-key Feishu queue task promise; the second same-key task remains chained behind it until the first settles. The PR blocker is also source-reproducible because the unchanged drive comment handler awaits a no-option queue call and then records the event as processed.

## Solution Assessment

No as submitted. The timeout release guard is the right narrow Feishu message fix, but the generic helper should not make timeout completion indistinguishable from success for existing callers that record completion after `enqueue` returns.
