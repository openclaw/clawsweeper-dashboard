## Summary

Keep open. Current main still resolves bare `session_status` model overrides with the active provider before allowlist validation, so this focused PR still addresses a real source-reproducible bug; it needs contributor-owned real behavior proof and a branch refresh before merge.

## What This Changes

The PR changes `session_status` model override resolution to infer a unique configured provider for bare non-alias model names before falling back to the current provider, with a regression test for switching from `openai-codex` to `deepseek`.

## Best Possible Solution

Refresh the branch onto current main, keep `session_status` aligned with the configured-provider inference behavior used by shared model selection, retain focused regression coverage, and merge after redacted real behavior proof plus targeted validation.

## Reproduction Assessment

Yes. Source inspection of current main gives a high-confidence path: a bare `session_status` override is parsed with `defaultProvider: currentProvider`, so a configured `deepseek/deepseek-chat` selection can become `openai-codex/deepseek-chat` and fail allowlist validation.

## Solution Assessment

Yes in direction. Inferring a unique configured provider before falling back to the active provider is a narrow maintainable fix for this path, with final merge gated on branch refresh, real behavior proof, and focused validation.
