## Summary

Keep open. Current main still has the reported `session_status` resolver gap, and the PR is a narrow functional fix with regression coverage; the only concrete merge blocker I found is the missing required changelog entry for a user-facing fix.

## What This Changes

The PR adds configured-provider inference for bare `session_status(model=...)` overrides and a regression test for switching from an `openai-codex` session to an allowlisted DeepSeek model.

## Best Possible Solution

Land the behavior fix with a regression test and changelog, ideally by routing `session_status` model overrides through the same allowlist-aware resolver semantics used by `sessions.patch` if that stays small and preserves current errors.

## Reproduction Assessment

Yes. The current code path is deterministic: with current provider `openai-codex`, an allowlist containing `deepseek/deepseek-chat`, and raw input `deepseek-chat`, `session_status` currently resolves using `defaultProvider: currentProvider` before checking the allowlist.

## Solution Assessment

Mostly yes. The PR fixes the observable bug narrowly, but the most maintainable form would reuse the shared `resolveAllowedModelRef` semantics rather than duplicating the inference step in `resolveModelOverride`.
