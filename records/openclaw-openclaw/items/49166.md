## Summary

Keep open: this PR targets a real remaining `session_status` model-switch bug on current main, and its approach matches the shared allowlist inference already added for other model-switch paths, but the external PR still needs real behavior proof before merge.

## What This Changes

The PR changes `session_status` model override resolution to infer a unique provider from configured models for bare model names and adds a regression test for switching from `openai-codex` to `deepseek`.

## Best Possible Solution

Wire `session_status` model override resolution to the same configured-allowlist provider inference used by the shared model-switch resolver, add focused current-main regression coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main resolves a bare `session_status` model override with `defaultProvider: currentProvider`, then rejects the resulting wrong provider/model key against the allowlist. I did not run the repro because the checkout is read-only for this review.

## Solution Assessment

Yes in direction: inferring a unique provider from configured models before falling back to the current provider is the narrow fix and matches the merged shared resolver behavior. The current branch should be refreshed against main and ideally reuse the existing shared resolver shape instead of adding a second divergent path.
