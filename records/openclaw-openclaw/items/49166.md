## Summary

Keep this PR open. Current main still resolves bare `session_status(model=...)` overrides through the current session provider before allowlist validation, while the shared model-selection resolver already has the configured-provider inference this PR is trying to bring to the tool path.

## What This Changes

The PR adds configured-provider inference for bare `session_status(model=...)` overrides and adds a regression test for switching from an `openai-codex` session to `deepseek/deepseek-chat`.

## Best Possible Solution

Keep the behavior fix, but wire `resolveModelOverride` through the shared allowlist-aware model resolver if feasible, and retain the regression test proving a bare cross-provider model selection stores the inferred provider/model pair.

## Reproduction Assessment

Yes. The current source path is deterministic: with current provider `openai-codex`, allowlist entry `deepseek/deepseek-chat`, and raw input `deepseek-chat`, `session_status` resolves against `currentProvider` before checking the allowlist.

## Solution Assessment

Mostly yes for behavior, but not clearly the best maintainability shape. The PR fixes the observable bug, while the safer long-term implementation is to reuse the existing shared `resolveAllowedModelRef` semantics rather than copy the inference step into `session_status`.
