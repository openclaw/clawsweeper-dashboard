## Summary

Keep open: current main still has a real Codex-runtime gap where OpenClaw-owned background exec exits enqueue `exec-event` wakes, but Codex-native `commandExecution` completions are only projected as native item/tool observations and are not bridged into `notifyOnExit` semantics.

## What This Changes

The issue requests that Codex-native exec completions honor OpenClaw `tools.exec.notifyOnExit` and `notifyOnExitEmptySuccess`, or that the unsupported boundary be made explicit.

## Best Possible Solution

Choose the Codex runtime contract, then either bridge native `commandExecution` completions into the existing OpenClaw exec-event wake path with regression coverage, or explicitly document and validate that `notifyOnExit` excludes Codex-native exec.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: OpenClaw exec sessions enqueue `exec-event` wakes, Codex-native completions are projected without that wake, and the reporter supplied a controlled Codex-vs-Default runtime comparison.

## Solution Assessment

Unclear until maintainers choose the contract; a Codex-side bridge is plausible and testable, but documenting an intentional OpenClaw-exec-only boundary would be the lower-risk alternative if native Codex exec should stay separate.
