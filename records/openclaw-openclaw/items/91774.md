## Label Justifications

- `P1`: The report affects real Codex-backed Telegram workflows where long-running command completion can leave the agent silent until the user manually wakes the session.
- `impact:session-state`: The affected behavior is a missing internal continuation/wake for the originating agent session after command completion.
- `impact:message-loss`: When the completion wake is missing, the agent may never inspect or relay the completed command result to the user-visible channel.

## Summary

Keep open. Current main still wires `notifyOnExit` wakes to OpenClaw-owned exec/process sessions, while Codex-native command execution is projected as app-server item progress without a bridge into the OpenClaw system-event/heartbeat wake path.

## What This Changes

The issue requests that Codex-backed native command execution honor OpenClaw `tools.exec.notifyOnExit` and `notifyOnExitEmptySuccess` completion wake semantics, or that the boundary be documented if it is intentionally out of scope.

## Best Possible Solution

Choose and document one Codex runtime contract, then either bridge native Codex command/process completions into the existing OpenClaw exec wake path without adding another config knob, or clearly document and validate that `notifyOnExit` only covers OpenClaw-owned exec/process tools.

## Reproduction Assessment

Yes by source and reporter evidence: the same background command/config reportedly wakes on the Default runtime, while current main only has completion-wake code on OpenClaw-owned exec/process paths and not on Codex-native command notifications.

## Solution Assessment

Unclear until maintainers choose the runtime boundary; the best fix is either a Codex-native completion bridge to the existing wake path or explicit docs/config validation that the feature is out of scope for native Codex execution.
