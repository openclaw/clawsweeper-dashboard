## Label Justifications

- `P1`: Codex-backed sessions can miss completion wakes for real long-running shell work, leaving users to manually poll or inject system events.
- `impact:message-loss`: The reported failure suppresses the completion-driven continuation that would otherwise decide whether to send a Telegram follow-up.
- `impact:session-state`: The originating agent session remains unaware that native exec completed because the OpenClaw system-event wake is not bridged.

## Summary

Keeping this issue open because current main still has separate OpenClaw exec wake and Codex native exec surfaces: the configured OpenClaw `notifyOnExit` path is wired to OpenClaw-owned exec/process completion, while Codex app-server native shell execution can complete outside that path and no source or related PR shows a bridge or documentation fix yet.

## What This Changes

The issue requests that Codex app-server native exec completions honor OpenClaw `tools.exec.notifyOnExit` and `notifyOnExitEmptySuccess` wake semantics, or that the runtime boundary be documented clearly.

## Best Possible Solution

Maintainers should decide the Codex native exec contract, then either bridge Codex native completion into the existing OpenClaw system-event wake path or explicitly document and validate that `notifyOnExit` only applies to OpenClaw-owned `exec`/`process`.

## Reproduction Assessment

Yes from source and reporter A/B evidence, although I did not run a live Telegram/Codex app-server repro; current code shows Codex-native exec completion is outside the OpenClaw `ProcessSession` notify path.

## Solution Assessment

Unclear until maintainers choose the runtime contract; the source-supported fixes are either a Codex-side completion bridge into OpenClaw wakes or explicit documentation/config validation for the exclusion.
