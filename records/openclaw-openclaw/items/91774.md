## Label Justifications

- `P1`: The issue affects real Codex-backed Telegram workflows where long-running command completion can leave the agent silent until the user manually wakes the session.
- `impact:session-state`: The reported failure is a missing internal continuation for the originating Codex-backed session after command completion.
- `impact:message-loss`: Without a completion wake, the agent may never inspect or relay the completed command result to the user-visible channel.

## Summary

Keep open. Current main and the latest release still wire `notifyOnExit` through OpenClaw-owned `ProcessSession` exec only; Codex native `commandExecution` completion is observed inside the Codex app-server turn and is not bridged into OpenClaw system-event or heartbeat wake semantics.

## What This Changes

Request that Codex app-server native exec honor OpenClaw `tools.exec.notifyOnExit` and `notifyOnExitEmptySuccess` completion wake semantics, or make that boundary explicit.

## Best Possible Solution

Define the Codex native exec completion contract, then either bridge Codex `commandExecution` completion into the existing system-event/heartbeat wake path with tests or document/diagnose that `notifyOnExit` is OpenClaw exec-only.

## Reproduction Assessment

Yes, by source and reporter evidence. The reporter isolated the same config and command to Codex app-server versus OpenClaw Default, and current main still shows wake code only on OpenClaw-owned exec sessions while Codex-native command completion stays inside Codex app-server notifications.

## Solution Assessment

Unclear until maintainers choose the contract. A Codex-native completion bridge is the clean runtime fix if the config is meant to be cross-runtime; otherwise docs and config diagnostics should explicitly scope `notifyOnExit` to OpenClaw exec/process tools.
