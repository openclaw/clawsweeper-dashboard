## Summary

Current main still has a real cross-runtime contract gap: OpenClaw completion wake is attached only to its own background process sessions, while Codex-native command executions are projected through a separate plugin event path. The requested behavior should remain open for a maintainer decision because current docs do not clearly promise that global exec settings govern Codex-native tools, and no canonical issue or viable open PR owns this exact integration.

## What This Changes

Extend OpenClaw completion-wake semantics to Codex-native background commands, or explicitly document and validate that the exec settings apply only to OpenClaw-owned exec sessions.

## Best Possible Solution

Add or reuse a generic runtime completion-to-session-event seam, then have the Codex plugin invoke it only for native executions that outlive their originating turn; keep process ownership in Codex and align docs and focused tests with the chosen configuration contract.

## Reproduction Assessment

Yes, from source with high confidence. Current main ties automatic completion wake to OpenClaw ProcessSession exits, while Codex-native commandExecution events stay in the plugin projector; the reporter's controlled Codex-versus-default live comparison matches that boundary.

## Solution Assessment

No final solution is established yet. Registering Codex commands as OpenClaw process sessions would duplicate ownership; the narrowest maintainable direction is a generic completion-event seam after maintainers confirm that `tools.exec` settings should cover native runtime tools.

## Risks / Open Questions

- The public contract is ambiguous: the documentation says “backgrounded exec sessions,” but global `tools.exec` configuration can reasonably appear to govern the native command tool exposed in a Codex-backed agent.
- A bridge must distinguish a genuinely detached native execution from an ordinary command that completes during the active Codex turn; otherwise it could inject duplicate or spurious continuations.
- The Codex protocol supplies command lifecycle events, but the implementation still needs a stable way to retain originating OpenClaw session and delivery context after the active turn ends.
