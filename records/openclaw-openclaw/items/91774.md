## Label Justifications

- `P1`: The report affects real Codex-backed Telegram workflows where long-running command completion can leave the agent silent until the user manually wakes the session.
- `impact:session-state`: The affected behavior is a missing internal continuation for the originating Codex-backed session after command completion.
- `impact:message-loss`: The missing completion wake can prevent the agent from inspecting or relaying the completed command result to the user-visible channel.

## Summary

Keep open. Current main still implements notify-on-exit only for OpenClaw `ProcessSession`-backed exec, while Codex-native command execution is projected as app-server item progress without a bridge into OpenClaw system-event or heartbeat wake semantics. The open on-exit cron PR is adjacent, not a merged fix for this config contract.

## What This Changes

Codex-backed native exec should either honor OpenClaw `tools.exec.notifyOnExit` completion wakes or clearly document that the setting only applies to OpenClaw exec/process tools.

## Best Possible Solution

Choose one Codex runtime contract, then either bridge Codex native command/process completion into the existing OpenClaw wake path or update docs/config diagnostics to make the unsupported boundary explicit.

## Reproduction Assessment

Yes by source and reporter evidence. The reporter isolated the same config and command to Codex app-server versus OpenClaw Default, and current main still shows wake code only on OpenClaw-owned exec sessions while Codex-native command completion is projected as app-server progress.

## Solution Assessment

Unclear until maintainers choose the contract. A Codex-native completion bridge is the clean runtime fix if the config is meant to be cross-runtime; otherwise docs and validation should explicitly scope `notifyOnExit` to OpenClaw exec/process tools.
