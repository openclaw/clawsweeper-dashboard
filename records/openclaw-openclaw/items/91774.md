## Summary

Keep this open: current main still documents `tools.exec.notifyOnExit` as an exec completion wake contract, but the Codex app-server native exec path only projects Codex `commandExecution` progress and does not enqueue OpenClaw system events or heartbeat wakes when native commands finish.

## What This Changes

The issue asks OpenClaw to make `tools.exec.notifyOnExit` and quiet-success wake behavior work for Codex app-server/native exec in Telegram direct sessions, or clearly define that the setting is OpenClaw-exec-only.

## Best Possible Solution

Maintainers should define the exec-notification contract for Codex native execution, then either bridge Codex `commandExecution` completion into the existing OpenClaw wake path or update docs/schema so users are not promised unsupported behavior.

## Reproduction Assessment

Yes for source-level reproduction: the reporter supplied a runtime comparison and current main shows the OpenClaw exec wake path is not connected to Codex native `commandExecution` completions.

## Solution Assessment

Unclear until maintainers choose the contract; the best fix is not just adding a local callback, because Codex native exec ownership and OpenClaw channel wake delivery need an explicit boundary.
