## Summary

Keep open: the linked diagnostic bug is valid and the PR targets the right area, but it still needs real behavior proof and one correctness fix in the shell-option scanner before merge.

## What This Changes

The PR updates the daemon service audit to treat POSIX shell `-c`/`-lc` LaunchAgent wrappers containing `gateway` as valid and adds three service-audit regression tests.

## Best Possible Solution

Keep the narrow audit fix, but detect only a real shell `-c` command argument and add regression coverage for both the reported wrapper and near-miss positional/rcfile cases.

## Reproduction Assessment

Yes from source inspection: current main only accepts a standalone `gateway` argv token while the reported LaunchAgent wrapper puts `gateway --port` inside the `/bin/zsh -lc` payload. I did not run a live macOS LaunchAgent, so this is source-reproducible rather than live reproduced.

## Solution Assessment

No: the PR targets the right helper, but its scanner can treat non-command arguments as shell payloads. The safer fix is to identify the first actual `-c` option and inspect only that immediate command string.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Limit shell scanning to the command argument:** `src/daemon/service-audit.ts:110-113`
  - body: After a shell sees `-c`, only the next argv element is the inline command; later elements are `$0` or positional parameters. This loop keeps scanning later flags, so `['/bin/sh','-c','exec node /opt/openclaw/dist/index.js start','--config','/tmp/gateway.json']` would return true even though the inline command never invokes the `gateway` subcommand. Restrict detection to a real `-c` option and inspect only its immediate command string.
  - confidence: 0.84
