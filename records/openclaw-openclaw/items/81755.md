## Summary

Keep open: the PR addresses a real, source-reproducible gateway status false positive, but the proposed shell-string regex is too broad and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR extends `src/daemon/service-audit.ts` to recognize shell-wrapped `-c`/`-lc` gateway commands and adds three regression tests in `src/daemon/service-audit.test.ts`.

## Best Possible Solution

Token-bound or parse the shell inline command so only an actual `gateway` subcommand counts, add a path-only negative regression, and require redacted real `gateway status` proof before merge.

## Reproduction Assessment

Yes from source inspection: current `main` reads the LaunchAgent command into `auditGatewayServiceConfig`, but `hasGatewaySubcommand` only accepts an argv element exactly equal to `gateway`. I did not run a live macOS LaunchAgent flow in this read-only review.

## Solution Assessment

No. The repair direction is right, but matching `/\bgateway\b/` across the whole shell string is not the narrowest maintainable solution because path-only occurrences can satisfy it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require gateway to be a shell token:** `src/daemon/service-audit.ts:113`
  - body: This regex matches `gateway` anywhere with word boundaries, including path components like `/tmp/gateway/index.js start` or `/opt/openclaw-gateway/dist/index.js start`. Those commands still lack the `gateway` subcommand, but the audit would stop reporting `gateway-command-missing`; please parse or token-bound the inline command and add a negative regression for path-only matches.
