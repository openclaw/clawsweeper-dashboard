## Summary

Keep open: this targets a valid current-main gateway diagnostic false positive, but the PR needs real behavior proof and the inline command scan is broad enough to hide malformed shell-wrapped services.

## What This Changes

The PR updates `src/daemon/service-audit.ts` to recognize POSIX shell `-c`/`-lc` inline commands containing `gateway`, with three regression tests in `src/daemon/service-audit.test.ts`.

## Best Possible Solution

Tighten shell-wrapper recognition to detect `gateway` as a shell token/subcommand, add a regression for path-only matches, and include redacted real `gateway status` proof from the reported LaunchAgent layout.

## Reproduction Assessment

Yes from source inspection: current main only checks for an exact `gateway` ProgramArguments element, while the linked macOS LaunchAgent puts `gateway --port` inside a `/bin/zsh -lc` inline command string. I did not run the live macOS flow in this read-only review.

## Solution Assessment

No. The repair direction is right, but accepting `/\bgateway\b/` over the whole shell string is too broad; the maintainable fix should parse or token-bound the inline command so path components do not count as the subcommand.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require gateway to be a shell token:** `src/daemon/service-audit.ts:113`
  - body: This regex matches `gateway` anywhere with word boundaries, including path components like `/opt/openclaw-gateway/dist/index.js start` or `/tmp/gateway/index.js start`. Those commands still lack the `gateway` subcommand, but the audit would stop reporting `gateway-command-missing`, so malformed shell-wrapped services could avoid the repair path. Please parse/token-bound the inline command and add a negative regression for a path-only match.
  - confidence: 0.86
