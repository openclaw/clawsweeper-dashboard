The issue reports that iMessage Full Disk Access denial after a Node/Homebrew upgrade can flood gateway logs and make the gateway appear frozen.

## Best Possible Solution

Review and land the canonical iMessage fix in https://github.com/openclaw/openclaw/pull/80403, reconciling overlap with https://github.com/openclaw/openclaw/pull/72898, then close this issue after the fix merges.

Yes. Source inspection gives a high-confidence path: feed current `IMessageRpcClient` the upstream `imsg` Full Disk Access banner on stdout and it logs one parse error per non-empty line; I did not run the live macOS TCC denial scenario.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local stdout classification, one structured permission error, fail-fast behavior, and fatal doctor/probe surfacing, which is the direction of the open linked PR.
