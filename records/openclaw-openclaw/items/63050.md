## Summary

Keep open: the validTypes diagnostic improvement is narrow and useful, but the PR still adds an argv-based provider-token path that needs maintainer security/product approval or a safer non-argv replacement before merge. The real CLI proof also predates the current head.

## What This Changes

The PR adds a `models auth paste-token --token <value>` CLI path, validates noninteractive token input, surfaces valid auth-profile types in invalid-type warnings, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the `validTypes` diagnostic improvement, but route noninteractive token setup through stdin or another non-argv secret-safe input path unless maintainers explicitly approve argv-token support with security proof.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.5.22 use `readPastedSecret` for prompt/stdin token input, while the PR adds a direct `--token` option that skips the prompt path.

## Solution Assessment

No. The `validTypes` warning is a good narrow fix, but accepting provider tokens through argv is not the safest way to solve noninteractive setup when stdin already avoids that exposure.
