## Summary

Keep this PR open, but it is not merge-ready: the validTypes/debugging piece is useful, while the new argv-carried provider-token path duplicates the existing piped-stdin path and creates credential-exposure risk that needs maintainer judgment.

## What This Changes

Adds a `models auth paste-token --token <value>` CLI option, validates and persists noninteractive token input, surfaces valid auth-profile types in invalid-type warnings, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the useful validTypes warning, but use stdin or a token-file/SecretRef-style input for scripted setup and require latest-head real CLI proof before merge.

## Reproduction Assessment

Partly: current main lacks a `--token` flag, but source shows `paste-token` already reads piped stdin in non-TTY mode. The stronger claim that no noninteractive token path exists is not reproduced from source.

## Solution Assessment

No: adding argv-carried provider tokens is not the safest scripted setup path when the command already has stdin plumbing. The narrower maintainable fix is documentation/tests for stdin, or an explicit token-file/stdin surface, plus the validTypes warning.
