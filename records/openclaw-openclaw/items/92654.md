This PR should stay open because it is the active implementation for the linked Dashboard webchat reply feature, but the exact head is not merge-ready: the production Dashboard caller still does not pass the new reply props, while the added context-menu handler suppresses native right-click before it knows Reply can open.

## What This Changes

The branch adds transient Control UI chat reply state, a right-click Reply context menu, a reply preview, quoted Markdown send prepending, and related chat CSS.

## Best Possible Solution

Wire replyTarget, onSetReply, and onClearReply through the real app-render Dashboard caller, preserve native right-click behavior when no custom menu opens, keep the current accessibility CSS baseline, add focused tests, and refresh exact-head browser proof.

## Reproduction Assessment

Yes for the PR defects: exact-head source shows the production caller omits the required reply props while the handler suppresses native right-click before returning, and the CSS regression is visible by comparing the PR head with current main.

## Solution Assessment

No. A UI-local quoted Markdown reply flow is a plausible first step, but this implementation is not the best fix until it uses the real Dashboard state path, preserves fallback right-click behavior, removes unrelated CSS reversions, and proves the exact head.

Full review comments:

- **[P1] Wire Reply through the production chat caller:** `ui/src/ui/views/chat.ts:1654-1657`
  - body: `handleChatContextMenu` suppresses the native right-click menu before returning when `onSetReply` is absent, and the exact PR head's production `renderChat` call still passes no reply props. In the real Dashboard path, right-clicking a bubble would lose the browser menu and never show Reply.
  - confidence: 0.97
- **[P2] Drop unrelated accessibility CSS reversions:** `ui/src/styles/chat/layout.css:887`
  - body: This line and the related hunks lower current-main 12px chat UI text back to 11px/10px and change Talk controls from `:focus-visible` to `:focus`. Those changes are unrelated to Reply and would undo the current accessibility baseline, so keep only the reply-menu and preview styles here.
  - confidence: 0.93
- **[P2] Add Escape handling for the active reply preview:** `ui/src/ui/views/chat.ts:2203`
  - body: The PR body says Escape cancels an active reply, but the preview only clears through the dismiss button and the added keydown listener only targets the transient context menu before a reply target is selected. After wiring the feature, keyboard users would not be able to cancel the active reply with Escape.
  - confidence: 0.86
- **[P3] Keep Escape dismissal registered until the menu closes:** `ui/src/ui/views/chat.ts:1697`
  - body: The context-menu keydown listener is registered with `{ once: true }`, so any first key after the menu opens, even a non-Escape key, removes the listener while leaving the menu open. Register it until dismissal so Escape reliably closes the custom menu.
  - confidence: 0.78
