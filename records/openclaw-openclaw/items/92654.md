Overall confidence: 0.94

Full review comments:

- **[P1] Wire Reply through the production caller:** `ui/src/ui/views/chat.ts:1654-1657`
  - body: `handleChatContextMenu` prevents the native context menu and then returns when `onSetReply` is absent, while the only production `renderChat` caller in `app-render.ts` still passes no reply props. In the Dashboard path, right-clicking a bubble would lose the browser menu and never show Reply.
  - confidence: 0.97
- **[P2] Drop the unrelated accessibility CSS reversions:** `ui/src/styles/chat/layout.css:887`
  - body: This line and the related hunks lower current-main 12px chat UI text back to 11px/10px and change Talk controls from `:focus-visible` to `:focus`. Those changes are unrelated to Reply and would undo the current accessibility baseline, so keep only the reply-menu and preview styles here.
  - confidence: 0.93
- **[P2] Add Escape handling for the active reply preview:** `ui/src/ui/views/chat.ts:2203`
  - body: The PR body says Escape cancels an active reply, but the preview only clears through the dismiss button and the added keydown listener only targets the transient context menu before a reply target is selected. After wiring the feature, keyboard users would not be able to cancel the active reply with Escape.
  - confidence: 0.86
- **[P3] Keep Escape dismissal registered until the menu closes:** `ui/src/ui/views/chat.ts:1697`
  - body: The context-menu keydown listener is registered with `{ once: true }`, so any first key after the menu opens, even a non-Escape key, removes the listener while leaving the menu open. Register it until dismissal so Escape reliably closes the custom menu.
  - confidence: 0.78
