## Summary

Keep open. Current main has narrowed some `sessions_spawn` runtime-specific behavior, but it still does not provide the requested per-agent allow/deny contract for individual model-visible tool schema parameters; that remaining work is a public config and schema-semantics decision.

## What This Changes

The issue requests per-agent filtering of individual tool schema parameters, such as `sessions_spawn.allowedParams` or `deniedParams`, so subagents see slimmer tool definitions.

## Best Possible Solution

Define a generic schema-visibility layer after effective tool selection and before provider schema normalization, then document and test the per-agent contract across `sessions_spawn` and at least one generic or plugin-owned tool.

## Reproduction Assessment

Yes, at source level. Current main still exposes broad `sessions_spawn` schema properties and the strict per-agent/global tool config has no parameter-filter fields, so the requested contract is absent without needing a live model run.

## Solution Assessment

Unclear. `allowedParams` or `deniedParams` is plausible, but maintainers need to choose naming, precedence, required-field behavior, and plugin/generic schema semantics before implementation.
