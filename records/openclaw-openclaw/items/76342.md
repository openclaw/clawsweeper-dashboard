## Summary

Keep open: the PR targets a valid Dream Diary timeout failure, but the current patch still accepts partial timeout override strings and leaves the new operator env knob out of stable docs, so it needs a focused repair before merge.

## What This Changes

The PR raises the memory-core Dream Diary narrative wait default from 60s to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates the focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land a bounded 240s default with strict whole-string positive-integer env parsing, focused override regression tests, and stable docs for the env knob.

## Reproduction Assessment

Yes, at source level: current main passes a hard-coded 60_000ms into `waitForRun` and returns without appending when that wait times out, while #76333 supplies repeated multi-workspace timeout logs. I did not live-run that host-scale setup in this read-only review.

## Solution Assessment

No, not yet: raising the bounded default and adding an env override is the right narrow shape, but the current parser accepts partial values and the operator knob is not in stable docs.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.9

A changelog entry helps upgrades, but later debugging still requires source or PR history.
  - confidence: 0.86
