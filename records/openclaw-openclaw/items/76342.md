## Summary

Keep open. Current main still hardcodes the Dream Diary narrative wait at 60 seconds, and this PR is a narrow candidate fix for the linked timeout report, but it still needs the user-facing docs/changelog follow-through before it is ready to land.

## What This Changes

The PR raises the memory-core Dream Diary narrative subagent timeout from 60s to 240s, adds an `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` env override, and updates the existing timeout assertion.

## Best Possible Solution

Land the timeout increase with explicit release notes and operator-facing documentation, then decide separately whether the existing dreaming execution config should expose the same timeout through the plugin schema.

## Reproduction Assessment

Yes, at source level: current main hardcodes `NARRATIVE_TIMEOUT_MS` to 60 seconds and returns without writing a diary entry when `waitForRun` reports `timeout`. I did not live-reproduce the multi-workspace/model-latency timing from the linked issue.

## Solution Assessment

Mostly yes: raising the default directly addresses the hardcoded budget that the linked issue shows is too small. The env override is reasonable, but it should be documented and release-noted before merge.
