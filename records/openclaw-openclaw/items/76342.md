## Summary

Keep open: the PR targets a real memory-core Dream Diary timeout path, but latest head still accepts partial env override values and lacks after-fix real behavior proof before merge.

## What This Changes

The PR raises the memory-core Dream Diary narrative wait default to 240000 ms, adds an `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` env override, updates one focused timeout assertion, and adds a changelog fix entry.

## Best Possible Solution

Land a bounded 240s default with strict whole-string positive-integer env parsing, focused override regression tests, stable docs for the env knob, and contributor-provided real behavior proof.

## Reproduction Assessment

Yes at source level, but not by a live run in this review: current main passes a hardcoded 60000 ms timeout to `waitForRun`, and #76333/#77094 provide repeated field logs of narrative timeout while dreaming phase outputs still completed.

## Solution Assessment

No, not yet: raising the bounded default and adding an env override is the right narrow shape, but the current parser can turn plausible invalid values into tiny timeouts and the operator knob is not in stable docs.

## Review Findings

  - body: `Number.parseInt` accepts numeric prefixes, so operator inputs like `240_000`, `240s`, or `1e5` become 240 ms, 240 ms, or 1 ms instead of falling back to the 240 s default. That can recreate the Dream Diary timeout loss this PR is meant to fix; require the trimmed whole string to be a positive integer and cover valid and invalid override cases.
  - confidence: 0.9

  - confidence: 0.84
