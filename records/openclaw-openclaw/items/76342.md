## Summary

Keep open. The PR still targets a real current-main bug, but it needs maintainer timeout-contract judgment, a small parser hardening fix, and after-fix proof from an actual Dream Diary runtime path before merge.

## What This Changes

The branch raises the memory-core Dream Diary narrative timeout from 60s to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates the focused timeout assertion, and adds changelog text.

## Best Possible Solution

Land or replace this with one focused memory-core fix that gives Dream Diary narrative runs enough bounded time, strictly validates any override, and updates tests plus operator-facing docs or changelog for the chosen contract.

## Reproduction Assessment

Yes, source-backed: current main and `v2026.5.12` still pass a fixed 60s budget to the Dream Diary narrative subagent, and the linked issue includes repeated real timeout logs from multi-workspace setups. I did not run a live 15-workspace gateway reproduction in this read-only review.

## Solution Assessment

Unclear. The PR touches the right wait path, but the env-only override and 240s default are a timeout-policy decision, and the override parser should reject mixed numeric strings before this lands.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Reject partially numeric timeout overrides:** `extensions/memory-core/src/dreaming-narrative.ts:109`
  - body: `Number.parseInt` accepts mixed strings such as `1m` as `1`, so a malformed `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS=1m` would turn the bounded wait into 1 ms and make narratives time out immediately. Parse the full trimmed value as a positive integer and cover a mixed-string case.
  - confidence: 0.86
