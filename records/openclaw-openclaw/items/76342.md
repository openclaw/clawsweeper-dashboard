## Summary

Keep open: current main still uses the hardcoded 60s Dream Diary narrative wait, and the field report tied to this PR is source-reproducible. The runtime change looks useful, but the new operator env var still needs stable docs before merge.

## What This Changes

The branch raises memory-core Dream Diary narrative wait default from 60s to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates the focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land the runtime timeout/default change after adding stable docs for the env var; leave a schema-level timeout knob and late-output recovery for separate maintainer decisions.

## Reproduction Assessment

Yes, at source level: current main waits only 60s and returns on timeout before reading/appending narrative text, while #76333 provides repeated field logs from a multi-workspace host. I did not live-run that host-scale scenario.

## Solution Assessment

Mostly yes: raising the bounded default directly addresses the exhausted wait budget, and the env override is a narrow operator escape hatch. The safer merge path is to document the new knob in stable Dreaming/config docs before landing.
