## Summary

Keep this PR open: current main still has the 60s Dream Diary narrative timeout, and the PR is a narrow source-level fix for the linked timeout reports. The remaining issue is that the new operator env override is not documented in stable Dreaming/config references, so users would need source or PR history to discover it later.

## What This Changes

Raises the memory-core Dream Diary narrative timeout from 60s to 240s, adds an `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` env override, updates the focused timeout test, and adds a changelog entry.

## Best Possible Solution

Land the runtime timeout/default change after stable Dreaming and memory configuration docs document the env var, millisecond unit, 240000 default, and fallback for invalid or non-positive values, while leaving a schema-level timeout key and late-output handling for separate maintainer decisions.

## Reproduction Assessment

Yes, at source level: current main waits only 60s and returns before appending diary text on timeout, while #76333 and #77094 provide repeated field logs from multi-workspace or slow-prep hosts. I did not live-run that host-scale scenario.

## Solution Assessment

Mostly yes: raising the bounded default directly addresses the exhausted wait budget, and the env override is a narrow operator escape hatch. The safer landing path is to add durable docs for the new knob before merge.
