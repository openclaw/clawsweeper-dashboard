## Summary

Keep open: current main still uses the 60s Dream Diary narrative wait, and this PR addresses a source-reproducible timeout bug, but the new operator env override still lacks durable Dreaming/config documentation after the latest update only added changelog coverage.

## What This Changes

The PR raises the memory-core Dream Diary narrative timeout default to 240s, adds an `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` env override, updates the focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land the runtime timeout/default change once stable Dreaming and config docs document the env var, its millisecond unit, default, and invalid-value fallback; leave a schema-level timeout key and late-output handling for separate maintainer decisions.

## Reproduction Assessment

Yes, at source level: current main waits only 60s and returns without appending diary text on timeout, while #76333 provides repeated field logs from a multi-workspace host. I did not live-run that host-scale scenario.

## Solution Assessment

Mostly yes: raising the bounded default directly addresses the exhausted wait budget, and the env override is a narrow operator escape hatch. The remaining fix is stable docs coverage for the new knob before merge.
