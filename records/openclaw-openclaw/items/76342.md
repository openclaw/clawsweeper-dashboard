## Summary

Keep open: current main still hardcodes the 60s Dream Diary narrative timeout, and the PR addresses a real source-level failure path, but the current patch still accepts malformed timeout env values as tiny partial integers and does not document the new operator override in stable docs.

## What This Changes

The PR raises the memory-core Dream Diary narrative timeout default from 60s to 240s, adds an `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` env override, updates the focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land a rebased version after strict positive-integer env parsing, invalid override coverage, and public docs for the env var; leave schema-level config and late-output recovery for separate follow-ups.

## Reproduction Assessment

Yes, at source level: current main waits only 60s and returns without appending diary text on timeout, while the linked reports provide repeated field logs from multi-workspace hosts. I did not live-run that host-scale setup in this read-only review.

## Solution Assessment

No, not yet: raising the bounded default and adding an env override is the right narrow shape, but the override should reject malformed partial values and be documented in stable operator docs before merge.

The changelog helps during upgrade, but later timeout debugging would still require source or PR history.
