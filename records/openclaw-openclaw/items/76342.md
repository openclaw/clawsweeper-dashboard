## Summary

Keep open: current main still hardcodes the 60s Dream Diary narrative timeout, and this PR is a narrow fix for the linked timeout reports. The remaining issue is that the new operator env override is still documented only in changelog/source, not in the Dreaming docs or memory config reference.

## What This Changes

This PR raises the memory-core Dream Diary narrative wait default from 60s to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates the focused timeout assertion, and adds a changelog fix entry.

## Best Possible Solution

Land the runtime change after adding concise docs for the host-level env override, while leaving any schema config knob or late-output recovery changes to separate maintainer decisions.

## Reproduction Assessment

Yes, at source level: current main waits only 60s and returns on `status=timeout`, while the linked reports provide repeated timeout logs that match this path. I did not live-run the host-scale scenario.

## Solution Assessment

Mostly yes: raising the bounded default and adding a host env override directly addresses the exhausted wait budget with a small extension-owned change. The safer merge path is to document the new operator knob before landing.
