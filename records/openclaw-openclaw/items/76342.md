## Summary

Keep open: current main still has the hardcoded 60s Dream Diary narrative wait, so the PR addresses a valid bug, but the current PR head still has a concrete parser regression and lacks stable docs for the new operator override.

## What This Changes

The PR raises the memory-core Dream Diary narrative timeout default from 60s to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates one focused test assertion, and adds a changelog entry.

## Best Possible Solution

Land a bounded 240s default with strict whole-string positive-integer env parsing, focused override tests, and stable docs for the env knob; avoid adding a broader config-schema field unless maintainers request that product surface.

## Reproduction Assessment

Yes, at source level: current main hardcodes `60_000` into `waitForRun` and exits before the diary append path on timeout, while the linked reports provide repeated host logs for that same status. I did not live-run the host-scale workspace setup in this read-only review.

## Solution Assessment

No, not yet: raising the bounded default and adding an override is the right narrow shape, but the parser accepts partial values and the operator-facing env knob is not documented in stable docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject partial timeout override values:** `extensions/memory-core/src/dreaming-narrative.ts:108`
  - body: `Number.parseInt` accepts numeric prefixes, so likely operator inputs such as `240_000`, `240s`, or `1e5` become 240ms, 240ms, or 1ms instead of falling back to the 240s default. That can recreate the narrative timeout loss this PR is meant to fix; parse the trimmed whole string as a positive integer and cover valid/invalid override cases.
  - confidence: 0.9
- **[P3] Document the timeout override:** `extensions/memory-core/src/dreaming-narrative.ts:99-101`
  - body: `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` becomes an operator-facing knob, but the Dreaming and memory config docs still only list `enabled`, `frequency`, and `model`, with no millisecond unit, 240s default, or invalid-value fallback. The changelog helps upgrades, but stable docs are still needed for later troubleshooting.
  - confidence: 0.86
