## Summary

Keep open: current `main` still has the 60s timeout, and this PR addresses a real source-reproducible Dream Diary failure, but the proposed patch is not merge-ready because the env override accepts partial values and the new operator knob is not in stable docs.

## What This Changes

The PR raises the memory-core Dream Diary narrative timeout default to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates one focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land the timeout/default fix after strict whole-string positive-integer env parsing, regression coverage for invalid override strings, and stable operator docs for the env var’s unit, default, and fallback behavior.

## Reproduction Assessment

Yes, at source level: current `main` waits only 60s and returns without appending diary text on timeout, while the linked issue provides repeated field logs from a multi-workspace host. I did not live-run that host-scale setup in this read-only review.

## Solution Assessment

Not yet: the bounded default increase and env override are the right narrow shape, but the override should parse strict positive millisecond integers and be documented outside the changelog before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Reject partial timeout env values:** `extensions/memory-core/src/dreaming-narrative.ts:109`
  - body: `Number.parseInt` accepts prefixes, so values like `240_000`, `240s`, or `1e5` shrink the narrative wait budget instead of falling back to the 240s default. That malformed operator setting can reintroduce the timeout failure this PR is trying to fix; parse the whole trimmed string as a strict positive integer and cover partial/invalid values.
  - confidence: 0.86
- **[P3] Document the timeout override:** `extensions/memory-core/src/dreaming-narrative.ts:100`

A changelog entry helps during upgrade, but later timeout debugging would still require source or PR history.
  - confidence: 0.9
