## Summary

Keep open: the fix direction matches the source-level Dream Diary timeout bug, but this PR still accepts partial env override values as tiny millisecond budgets and leaves the operator override out of stable docs.

## What This Changes

The PR raises the memory-core Dream Diary narrative wait default from 60s to 240s, adds `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS`, updates the focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land a bounded 240s default with strict whole-string positive-integer env parsing, focused override regression tests, and stable docs for the env knob.

## Reproduction Assessment

Yes, at source level: current main hardcodes 60_000ms into `waitForRun` and returns before diary append on timeout, while #76333 and #77094 provide repeated field logs from multi-workspace runs. I did not live-run that host-scale setup in this read-only review.

## Solution Assessment

No, not yet: raising the bounded default and adding an env override is the right narrow shape, but the parser accepts partial values and the operator knob is not in stable docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Reject partial timeout override values:** `extensions/memory-core/src/dreaming-narrative.ts:108`
  - body: `Number.parseInt` accepts numeric prefixes, so likely operator inputs such as `240_000`, `240s`, or `1e5` become 240ms, 240ms, or 1ms instead of falling back to the 240s default. That can recreate the narrative timeout loss this PR is meant to fix; parse the trimmed whole string as a positive integer and cover valid/invalid override cases.
  - confidence: 0.9

  - body: `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` is now an operator-facing knob, but the Dreaming and memory config docs still only list `enabled`, `frequency`, and `model`, with no millisecond unit, 240s default, or invalid-value fallback. The changelog helps upgrades, but stable docs are still needed for later troubleshooting.
  - confidence: 0.86
