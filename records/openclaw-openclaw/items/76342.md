## What This Changes

The PR raises memory-core Dream Diary narrative wait budget from 60s to 240s, adds an `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS` env override, updates the focused timeout assertion, and adds a changelog entry.

## Best Possible Solution

Land the timeout increase and env override after adding stable operator docs, while leaving any schema-level config knob for a separate maintainer decision.

## Reproduction Assessment

Yes, at source level: current main hardcodes a 60s wait and returns before appending when `waitForRun` reports timeout, while #76333 supplies repeated field logs from a 15-workspace host. I did not live-run that host-scale scenario.

## Solution Assessment

Mostly yes: a bounded 240s default plus a positive env override is a narrow fix for the wait-budget failure and avoids expanding the config schema. The safer merge path is to document the override in the stable Dreaming and memory config docs, not only changelog/source comments.
