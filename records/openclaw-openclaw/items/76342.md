Keep open. The PR targets a real memory-core timeout path that is still present on current main, but it needs contributor real-runtime proof and maintainer choice between the env-only timeout policy and the open config-schema alternative before merge.

## What This Changes

## Best Possible Solution

Choose one supported memory-core timeout contract, land a focused implementation with strict integer parsing, docs/tests/changelog as needed, and require redacted real Dream Diary runtime proof before merge.

## Reproduction Assessment

## Solution Assessment

Unclear. The PR touches the right wait path, but the env-only 240s policy competes with an open config-schema alternative and the parser should reject mixed numeric strings first.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Reject partially numeric timeout overrides:** `extensions/memory-core/src/dreaming-narrative.ts:109`
  - body: `Number.parseInt` accepts mixed strings such as `1m` as `1`, so a malformed `OPENCLAW_DREAMING_NARRATIVE_TIMEOUT_MS=1m` would turn the intended bounded wait into 1 ms and make narratives time out immediately. Parse the full trimmed value as a positive integer and cover a mixed-string case.
  - confidence: 0.84
