## Summary

Keep open: current main and the latest v2026.5.20 release still resolve remote bundle-MCP HTTP auth as static configured headers only. The MCP SDK has a plausible client_credentials authProvider primitive, but adding it is a new security-sensitive config/auth surface that needs maintainer design and coordination with related OAuth and SecretRef work.

## What This Changes

Request adding optional OAuth 2.1 client_credentials configuration for remote bundle-MCP HTTP servers instead of relying on static Authorization headers.

## Best Possible Solution

Define an approved mcp.servers OAuth client_credentials schema, wire it through the existing resolver to SDK authProvider while preserving redirect/header scrubbing, and document/test SecretRef, redaction, refresh, and static-header compatibility.

## Reproduction Assessment

Yes for the reported limitation by source inspection: remote MCP HTTP config resolves only static headers, and transport construction never passes an SDK authProvider. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the security and config shape. The SDK ClientCredentialsProvider path is plausible, but SecretRef support, redaction, token refresh, and coordination with the interactive OAuth PR need owner decisions.
