## Summary

Keep open: current main still resolves remote MCP auth as static config headers only, and the adjacent open remote-MCP OAuth PR is for interactive authorization-code OAuth rather than this issue's non-interactive client_credentials path.

## What This Changes

Add OAuth 2.1 client_credentials support for remote MCP servers in `mcp.servers` config instead of relying on static bearer headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth client_credentials contract, wire it through the existing resolver to the SDK `authProvider` path, and document/test static-header compatibility, SecretRef/redaction, token refresh, and interaction with interactive OAuth.

## Reproduction Assessment

Yes for the current limitation by source inspection: remote MCP HTTP config resolves only static headers, and transport construction never passes an SDK `authProvider`. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the config and security shape. The SDK `ClientCredentialsProvider` path is plausible, but SecretRef support, redaction, token endpoint auth method scope, and coordination with https://github.com/openclaw/openclaw/pull/62134 need owner decisions.
