## Summary

Keep open: current main still has only static MCP HTTP headers, while client_credentials would add a new security-sensitive auth/config capability that needs maintainer design rather than cleanup closure.

## What This Changes

Request adds OAuth 2.1 client_credentials support for remote bundle-mcp HTTP servers via a new config block and SDK authProvider wiring.

## Best Possible Solution

Define a typed MCP OAuth client-credentials contract with SecretInput/SecretRef, redaction, auth-method scope, docs, and regression tests, then wire it through bundle-mcp HTTP resolution to the SDK authProvider path.

## Reproduction Assessment

Yes for the current limitation by source inspection: the HTTP resolver returns only url/transportType/headers, and streamable HTTP only passes requestInit headers. There is no oauth field or SDK authProvider wiring on current main.

## Solution Assessment

Unclear until maintainers approve the config shape. The SDK authProvider direction is plausible, but SecretRef support, mutual-exclusion behavior, and supported client authentication methods need owner decisions first.
