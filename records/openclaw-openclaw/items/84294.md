## Summary

Keep open: current main still only supports static `headers` for remote MCP HTTP auth, and the related open OAuth PR covers interactive authorization-code OAuth rather than this issue's non-interactive `client_credentials` config path.

## What This Changes

The issue requests an additive `mcp.servers.*.oauth` client_credentials config surface for remote MCP servers and SDK authProvider wiring instead of static bearer headers.

## Best Possible Solution

Define one owner-approved MCP HTTP auth contract, wire client_credentials through the resolver to the SDK authProvider path, and document/test compatibility with static headers, SecretRef/redaction, token refresh, and the existing interactive OAuth work.

## Reproduction Assessment

Yes for the current limitation by source inspection: remote MCP HTTP config resolves only static headers, and the transport construction never passes an SDK authProvider. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the config and security shape. The SDK `ClientCredentialsProvider` path is plausible, but the durable solution needs SecretRef/redaction, token endpoint auth, token refresh, and coordination with the interactive OAuth PR.
