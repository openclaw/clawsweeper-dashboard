## Summary

This issue should stay open because current main still exposes only static configured headers for remote bundle-MCP HTTP auth and has no OAuth client_credentials config or authProvider wiring. The request is source-reproducible, but it is a security-sensitive auth/config feature that needs maintainer product and security review before implementation.

## What This Changes

The issue requests an additive OAuth 2.1 client_credentials config path for remote MCP HTTP servers so OpenClaw can obtain and refresh access tokens instead of relying on static Authorization headers.

## Best Possible Solution

Define one owner-approved remote MCP OAuth config contract that covers client_credentials without conflicting with interactive OAuth, wire it through the existing HTTP resolver to the SDK authProvider, and document/test static-header compatibility plus secret redaction.

## Reproduction Assessment

Yes for the current limitation by source inspection: remote HTTP MCP config resolves static headers only, and streamable/SSE transport construction never supplies an SDK authProvider. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the config and security shape. The SDK ClientCredentialsProvider route is plausible, but the durable solution should coordinate with the existing interactive OAuth PR and OpenClaw secret/redaction patterns.
