## Summary

Keep open: current main and v2026.5.18 still model remote bundle-mcp HTTP auth as static headers only. The requested client_credentials support would add a new auth/config/secret-handling surface, and the existing open browser-OAuth PR is related but does not cover this machine-to-machine flow.

## What This Changes

The issue requests an optional OAuth client_credentials configuration path for HTTP and streamable bundle-mcp servers so OpenClaw can obtain and refresh access tokens instead of relying on static bearer headers.

## Best Possible Solution

Define an owner-approved remote MCP OAuth client_credentials contract, coordinate it with the browser-OAuth work, wire it through the existing transport resolver to the SDK authProvider path, and document/test static-header compatibility plus secret handling.

## Reproduction Assessment

Yes for the source-level limitation: current main parses only url/transport/headers for remote MCP HTTP config and constructs transports without authProvider. I did not run a live OAuth MCP server, but the current code and docs make the missing config path clear.

## Solution Assessment

Unclear until maintainers approve the config and security model. Using the SDK authProvider and ClientCredentialsProvider is plausible, but the best solution must settle SecretRef/redaction, client auth method scope, mutual exclusion with static Authorization, and coordination with the open browser-OAuth PR.
