## Summary

Keep open: current main still only supports static remote MCP HTTP headers, while the related OAuth PR targets interactive authorization-code OAuth and does not supersede this client_credentials request. ## What This Changes

Request adding an OAuth client_credentials configuration path for remote MCP servers instead of requiring static Authorization headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth client_credentials contract, wire it through the existing resolver to the SDK authProvider path, and document/test SecretRef, redaction, static-header compatibility, token refresh, and interaction with interactive OAuth.

## Reproduction Assessment

Yes, by source inspection: origin/main's remote MCP resolver and transport only carry static headers into requestInit and never pass an SDK authProvider. This proves the limitation, although it is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the config and security shape. The SDK ClientCredentialsProvider path is plausible, but SecretRef support, token endpoint auth method policy, redaction, and coordination with the interactive OAuth PR need owner decisions.
