## Summary

Keep this open: current main and the latest shipped release still only support static remote-MCP HTTP headers, while the adjacent open OAuth PR covers interactive authorization-code OAuth and does not satisfy this client_credentials request.

## What This Changes

The issue requests an additive remote MCP config surface for OAuth client_credentials and SDK authProvider wiring so operators can use rotating per-client tokens instead of static bearer headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth client_credentials config contract, wire it through the existing resolver to the SDK authProvider path, and document/test compatibility with static headers, SecretRef/redaction, token refresh, and the interactive OAuth work.

## Reproduction Assessment

Yes. Source inspection shows remote MCP HTTP config resolves only static headers, and streamable HTTP transport construction never passes an SDK authProvider; this is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the config and security shape. The SDK ClientCredentialsProvider path is plausible, but SecretRef support, token endpoint auth method scope, redaction, and coordination with the existing OAuth PR need owner decisions.
