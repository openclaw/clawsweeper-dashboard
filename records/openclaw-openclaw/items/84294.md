## Summary

Keep open: current main still wires remote MCP HTTP auth through static configured headers only, and the related open OAuth PR targets interactive authorization-code OAuth rather than this issue's non-interactive client_credentials flow.

## What This Changes

The issue requests an optional OAuth client_credentials config path for remote bundle-mcp HTTP servers instead of relying on static Authorization bearer headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth client_credentials config contract, wire it through the existing resolver to the SDK authProvider path, and document/test static-header compatibility, SecretRef/redaction, token refresh, and interaction with interactive OAuth work.

## Reproduction Assessment

Yes for the current limitation by source inspection: remote MCP HTTP config resolves only static headers, and transport construction never passes an SDK authProvider. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the config and security shape. The SDK ClientCredentialsProvider path is plausible, but SecretRef support, token endpoint auth method scope, token storage/redaction, and coordination with existing OAuth work need owner decisions.
