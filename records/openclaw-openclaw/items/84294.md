## Summary

This should stay open: current main still has only static header support for remote MCP HTTP auth, and the related open OAuth PR covers interactive browser authorization rather than this issue's non-interactive client_credentials path.

## What This Changes

The issue requests an additive OAuth 2.1 client_credentials config surface for remote bundle-mcp HTTP servers so OpenClaw can rotate per-client access tokens instead of using static bearer headers.

## Best Possible Solution

Define an owner-approved remote MCP OAuth client_credentials contract, wire it to the SDK authProvider path while preserving static-header compatibility and redirect scrubbing, then document and test schema, redaction, SecretRef, and token refresh behavior.

## Reproduction Assessment

Yes, by source inspection: current main resolves remote MCP HTTP config to static headers only and constructs transports without an SDK authProvider. I did not run a live OAuth MCP server, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear until maintainers approve the security and config shape. The SDK ClientCredentialsProvider path is plausible, but the final design should decide SecretRef/redaction, token endpoint auth methods, static-header precedence, and coordination with the interactive OAuth PR.
