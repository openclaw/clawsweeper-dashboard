## Summary

This should stay open: current main and the latest release have shipped interactive MCP OAuth, but they still do not expose a client_credentials config/client-secret path for remote MCP servers.

## What This Changes

Request support for OAuth 2.1 client_credentials authentication in remote bundle-mcp HTTP server config instead of static bearer headers only.

## Best Possible Solution

Define one owner-approved MCP OAuth config contract that supports client_credentials securely, reuses the shipped authProvider transport path, and documents/tests its interaction with static headers, interactive OAuth, and Codex projection.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.6 expose interactive MCP OAuth and static headers, but no client_credentials/clientSecret config path. This is a missing feature rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the config/security contract. The SDK ClientCredentialsProvider path is plausible, but the best fix needs an owner-approved shape for secrets, token endpoint auth methods, and coexistence with shipped interactive OAuth.
