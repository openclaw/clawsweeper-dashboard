## Summary

This should stay open: current main and the latest release still support remote bundle-mcp HTTP auth only through static configured headers, while OAuth client-credentials support would add a new security-sensitive MCP auth/config capability that needs maintainer design review.

## What This Changes

The issue requests an additive OAuth 2.1 client-credentials configuration path for remote bundle MCP HTTP servers that currently use static configured headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth client-credentials contract, wire it through the existing resolver to the SDK authProvider path, and document/test static-header compatibility, secret handling, token refresh, and auth-method behavior.

## Reproduction Assessment

Yes for the current limitation by source inspection: the HTTP config resolver returns only url/transportType/headers, and the streamable HTTP transport passes static requestInit headers with no oauth field or authProvider. I did not run a live OAuth MCP server repro.

## Solution Assessment

Unclear pending maintainer approval. The proposed SDK ClientCredentialsProvider direction is plausible, but the best fix needs an approved schema, secret-handling policy, and coordination with the existing remote MCP OAuth/dynamic-header discussions.
