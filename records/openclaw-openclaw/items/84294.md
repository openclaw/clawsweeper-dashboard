## Summary

This should stay open: current main still supports remote bundle-mcp HTTP auth only through static configured headers, and the adjacent open remote-MCP OAuth PR covers interactive authorization-code OAuth rather than this issue's non-interactive client_credentials path.

## What This Changes

Add an optional remote MCP OAuth client_credentials configuration path so streamable-HTTP servers can use SDK-managed token acquisition/refresh instead of static bearer headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth client_credentials config contract, wire it through the existing resolver to the SDK authProvider path, and document/test static-header compatibility, SecretRef/redaction, token refresh, and interaction with the interactive OAuth work.

## Reproduction Assessment

Yes for the current limitation by source inspection: remote MCP HTTP config resolves only static headers, and streamable HTTP transport construction never passes an SDK authProvider. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the config and security shape. The SDK ClientCredentialsProvider path is plausible, but SecretRef support, token endpoint auth method scope, token storage/redaction, and coordination with the existing OAuth PR need owner decisions.
