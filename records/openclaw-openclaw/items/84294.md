## Summary

This should stay open: current main still supports remote bundle-mcp HTTP auth only through static configured headers, while the requested OAuth client-credentials path would add a real auth/config capability that needs maintainer design review.

## What This Changes

Add OAuth 2.1 client_credentials support to remote bundle-mcp HTTP server configuration instead of requiring static bearer headers.

## Best Possible Solution

Define an owner-approved MCP HTTP OAuth config contract, wire it through the existing bundle-mcp transport resolver to the SDK authProvider path, and document/test static-header compatibility and secret handling.

## Reproduction Assessment

Yes for the current limitation by source inspection: the HTTP config resolver returns only url/transportType/headers, and the streamable HTTP transport only passes requestInit headers. There is no oauth field or SDK authProvider wiring on current main.

## Solution Assessment

Unclear until maintainers approve the config shape; the proposed SDK direction is plausible, but SecretRef support, mutual-exclusion behavior, and client auth method scope need owner decisions before implementation.
