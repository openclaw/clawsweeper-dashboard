## Summary

Keep this open: current main still has only static header auth for remote MCP HTTP transports, and the adjacent open OAuth PR covers interactive authorization-code OAuth rather than this issue's non-interactive client_credentials path.

## What This Changes

Add OAuth 2.1 client_credentials configuration and transport wiring for remote MCP servers instead of relying on static Authorization headers.

## Best Possible Solution

Define an owner-approved remote MCP client_credentials config contract, wire it through the existing resolver to SDK authProvider, and document/test compatibility, redaction, token refresh, and interaction with the interactive OAuth work.

## Reproduction Assessment

Yes for the current limitation by source inspection: remote MCP HTTP config resolves only static headers, and the transport constructor never passes an SDK authProvider. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the auth/config shape. The SDK ClientCredentialsProvider path is plausible, but SecretRef/redaction, token endpoint auth methods, oauth-vs-headers behavior, and coordination with interactive OAuth need owner decisions.
