## Summary

Keep open: the PR addresses a real Telegram observability gap, but the current head can log a topic `threadId` even after Telegram accepted a threadless fallback retry.

## What This Changes

The branch adds info-level Telegram outbound success logging for text/media sends and tests that message bodies, captions, and media locations are not logged.

## Best Possible Solution

Update the PR so the success logger receives the effective params from the successful Telegram attempt and add focused fallback coverage that proves threadless retries do not get journaled as topic successes.

## Reproduction Assessment

Yes, source-reproducible. Current main reaches `sendMessageTelegram()` without the requested success log, and the PR defect follows directly from `withTelegramThreadFallback` stripping `message_thread_id` while the new log still reads the original `threadParams`.

## Solution Assessment

No, not yet. Adding the log in `extensions/telegram/src/send.ts` is the right boundary, but it needs to log the effective successful request instead of the original thread parameters after fallback.
