## Summary

Keep open: the PR targets the right Telegram send boundary, but the new success log can report a thread ID after the existing threadless retry path has actually sent without one, which makes the journal evidence misleading.

## What This Changes

The PR adds info-level Telegram outbound success logging for text/media sends and tests that body, caption, and media locations are not logged.

## Best Possible Solution

Land a send-level success log that records only identifiers from the effective successful Telegram request, omitting or explicitly marking the thread when fallback removed it.

## Reproduction Assessment

Yes, source-reproducible. Current main’s durable Telegram path reaches sendMessageTelegram without an outbound success log, and the PR finding is visible from the thread fallback code that strips `message_thread_id` before a successful retry.

## Solution Assessment

No, not yet. Adding the log in `extensions/telegram/src/send.ts` is the right boundary, but it needs to log the effective successful request instead of the original thread parameters after fallback.

## Review Findings
