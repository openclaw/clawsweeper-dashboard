## Summary

Keep open. The linked bug is valid and the PR takes a reasonable diagnostic direction, but current main does not already implement it and the proposed branch still has narrow merge blockers: missing required changelog coverage and warning text that can fire on CLI-backed cron runs where the recommended knobs are not applied.

## What This Changes

The PR adds a WARN log during isolated cron `agentTurn` preparation when neither `lightContext` nor a non-empty `toolsAllow` is set, and updates one cron skill-filter fixture to set `lightContext: true`.

## Best Possible Solution

Land the diagnostic as a narrow cron improvement after adding the required release note and making the warning accurate for the actual runner path: either scope it to embedded providers or use provider-neutral wording with focused coverage for both embedded and CLI-backed cron runs.

## Reproduction Assessment

Yes, at code-path level. The linked issue gives concrete cron-job settings that trigger the timeout, and current main shows `lightContext` and `toolsAllow` are meaningful for embedded isolated cron runs, though the wall-clock timeout depends on machine and workspace size.

## Solution Assessment

Yes for the general approach, but not yet for the exact branch. An operator-visible warning is a low-risk diagnostic for the silent timeout class, but the branch should add changelog coverage and avoid recommending embedded-runner-only knobs to CLI-backed cron jobs.
