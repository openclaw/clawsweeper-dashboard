## Summary

Keep open: current main still computes and sends only the suffix delta to the Feishu CardKit text-update endpoint, while Feishu's CardKit streaming docs describe passing full text and letting the platform calculate the increment. ([open.feishu.cn](https://open.feishu.cn/document/cardkit-v1/card-element/content?lang=zh-CN&utm_source=openai))

## What This Changes

The issue reports that Feishu streaming-card updates send only newly appended text to CardKit instead of the full accumulated text expected by the platform.

## Best Possible Solution

Reconcile the prior suffix-delta fix with the official CardKit contract by sending full accumulated content to the streaming text endpoint, adding focused regression coverage, and proving https://github.com/openclaw/openclaw/issues/82417 does not regress.

## Reproduction Assessment

Yes at source level: current main computes suffix-only content and the focused tests assert suffix payloads like " small" and "!" for CardKit updates. I did not run a live Feishu client reproduction in this read-only review.

## Solution Assessment

No, the current suffix-delta path does not match the official CardKit streaming text contract. The maintainable fix should be narrow, but because a prior merged issue reported the opposite symptom, maintainers should require live proof before reverting the behavior.
