## What This Changes

The PR changes health/status session summary code and tests to keep the freshest row per `sessionId` and adds a changelog entry for the user-facing command output fix.

## Best Possible Solution

Land a rebased, narrow PR that keeps the freshest-by-`sessionId` health/status dedupe, resolves the test conflicts cleanly, preserves the changelog entry, and ensures the aggregate status session list is deduped at the right scope.

## Reproduction Assessment

Yes. By source inspection, a current-main session store with two non-reserved entries sharing the same `sessionId` will produce two health/status rows because both paths map raw entries before sorting and never group by session ID.

## Solution Assessment

No, not as submitted. The newest-first dedupe direction is maintainable, but the current PR head must remove committed conflict markers and should dedupe the top-level status aggregate after rows from all store paths are combined if the goal is to stop inflated status counts.
