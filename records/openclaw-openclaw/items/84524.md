## Summary

Keep this PR open for review: current `main` still has the parseInt-based `--limit` behavior reported by the linked issue, and the proposed fix is focused, but the supplied proof does not exercise the real OpenClaw CLI command path after the patch.

## What This Changes

The branch replaces parseInt-based `--limit` parsers in `plugins search` and `skills search` with strict positive-integer validation that throws Commander `InvalidArgumentError`.

## Best Possible Solution

Keep validation at the CLI option boundary, add focused command-level coverage for both search commands, and require real terminal proof of invalid and valid CLI invocations before landing.

## Reproduction Assessment

Yes, the current-main source path is clear: both search commands parse `--limit` with `Number.parseInt`, and a Node probe confirms the partial-number coercion. I did not rerun the full CLI in this read-only review.

## Solution Assessment

Mostly yes: CLI option parsing is the right boundary for rejecting invalid values before ClawHub calls. The patch should be backed by real CLI proof and regression coverage, and maintainers should accept the stricter treatment of previously coerced inputs.
