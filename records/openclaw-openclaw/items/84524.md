Keep open: the patch targets a real current-main CLI parser bug, but it still needs real `openclaw` CLI proof and maintainer acceptance of the stricter search-limit compatibility change before merge.

## What This Changes

The PR replaces `Number.parseInt` parsing for `plugins search --limit` and `skills search --limit` with Commander `InvalidArgumentError` validation for strict positive integers.

## Best Possible Solution

Keep validation at the Commander option boundary, add focused command-level coverage for both search commands, and land only after real CLI proof shows invalid and valid limits behaving as intended.

## Reproduction Assessment

Yes, at source level: current main parses both search `--limit` options with `Number.parseInt`, and the linked issue includes concrete CLI examples showing `1abc` being accepted. I did not run the full CLI because this read-only checkout lacks installed dependencies.

## Solution Assessment

Mostly yes: the CLI option parser is the right boundary to reject invalid values before ClawHub calls. The maintainable finish is to pair this with command-level tests and real CLI proof, and to confirm whether the stricter rejection of non-canonical integer spellings is intended.
