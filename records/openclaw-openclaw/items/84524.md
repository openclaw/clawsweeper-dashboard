## What This Changes

The PR changes `plugins search` and `skills search` to parse `--limit` with `Number()` plus `InvalidArgumentError` validation instead of `Number.parseInt`.

## Best Possible Solution

Keep validation at the Commander option boundary, add focused command-level coverage for both search commands, and land only after real CLI proof shows invalid and valid limits behaving as intended.

## Reproduction Assessment

Yes, at source level: current main parses both search `--limit` options with `Number.parseInt`, and the linked report includes concrete CLI examples showing `1abc` being accepted. I did not establish a live current-main CLI run in this read-only review.

## Solution Assessment

Mostly yes: the CLI option parser is the right boundary to reject invalid values before ClawHub calls. The maintainable finish is to add command-level tests, supply real CLI proof, and confirm the stricter rejection of non-canonical integer spellings is intended.
