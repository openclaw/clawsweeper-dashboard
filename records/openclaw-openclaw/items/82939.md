## What This Changes

The issue reports Azure OpenAI Responses prompt-cache read drops even when tracked cache inputs stay stable, likely because OpenAI-compatible tool payloads preserve caller order while cache diagnostics treat tool names as a sorted set.

## Best Possible Solution

Stabilize OpenAI-compatible transport tool serialization with focused regression coverage, then close the issue only after the linked fix merges or maintainers choose a different implementation path.

## Reproduction Assessment

Yes, from source inspection and the reporter's logs: current main hashes tool names as an order-insensitive diagnostic set while serializing OpenAI-compatible tools in caller order. I did not run a live Azure session, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes: sorting at the OpenAI-compatible payload serialization boundary is the narrowest maintainable fix because it removes the untracked request-byte drift without changing provider selection, prompt text, tool execution, or cache-key policy.
