## Summary

Keep open: current main still has a plausible LM Studio provider/model-selection bug where suffix-bearing model IDs can reach preload and inference unchanged, and no merged fix or exact duplicate owns this quant-suffix retry/load problem.

## What This Changes

The issue asks OpenClaw to stop appending GGUF quant suffixes to LM Studio retry/load model identifiers and avoid counting those failed attempts as assistant errors.

## Best Possible Solution

Confirm the LM Studio version-specific identifier contract, then fix the provider/model-selection boundary so retry/preload never invents a suffix while explicit quant-suffixed refs that LM Studio truly advertises remain valid.

## Reproduction Assessment

No high-confidence live reproduction was run here. The report includes concrete real-session logs, and current source shows suffix-bearing model IDs pass unchanged into preload and chat-completions requests, but the exact suffix-append path still needs LM Studio live confirmation.

## Solution Assessment

Unclear as written. Stripping all quant suffixes is not the best general fix because OpenClaw intentionally preserves explicit LM Studio quant-suffixed model refs; the safer solution is to prevent retry/discovery from mutating a clean logical id while preserving valid explicit suffix ids.
