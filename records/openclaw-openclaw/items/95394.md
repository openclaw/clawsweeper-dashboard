## Summary

Keep open: current main and the latest release still carry the LM Studio provider path that can send a variant-flavored model id through preload and chat, and https://github.com/openclaw/openclaw/pull/95401 is the active candidate fix but has not merged.

## What This Changes

The issue reports that LM Studio retry/load handling can send quant-suffixed variant identifiers to native load and chat requests, causing false assistant-turn errors and polluted model/session analytics.

## Best Possible Solution

Land or replace the focused LM Studio provider fix that resolves variant identifiers to the server-advertised canonical load/chat key while preserving suffixed ids when LM Studio advertises them as canonical.

## Reproduction Assessment

Unclear for a full live reproduction in this review: the reporter provided concrete logs, and current source shows the suffixed id can reach load/chat unchanged once selected. I did not run the reporter's exact LM Studio Desktop 0.4.17-beta+3 setup, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, provider-local canonicalization against LM Studio `key`/`variants` data is the best fix shape. It avoids the unsafe alternative of blanket suffix stripping while addressing the retry/load path that produced the false error.
