## Summary

Keep open. The linked latency bug is real and the PR is moving in a useful direction, but the current head still regresses recent-context recall and bypasses shared embedded-run provider/transcript setup, so it should not merge as-is.

## What This Changes

The PR moves Active Memory recall to an eager memory-only embedded path, adds required memory-tool and QMD fallback instrumentation, filters runtime tools, and includes Discord startup responsiveness changes.

## Best Possible Solution

Keep the targeted tool filtering, query sanitization, and QMD fallback improvements, but feed eager hooks the real prompt history, share normal embedded-run setup for memory-only recall, and add the required changelog entry before CI/live validation.

## Reproduction Assessment

Yes. The linked bug has direct-chat timeout steps and logs, and the PR regressions are reproducible by source inspection: eager hooks receive an empty message list and memory-only runs leave the shared setup path early.

## Solution Assessment

No. Starting recall earlier and bounding memory-only work is a reasonable direction, but this implementation is not the narrowest maintainable fix because it trades away documented query context and provider compatibility.

  - body: Active Memory is now registered as an eager `agent_turn_prepare` hook, but the eager call is started with `messages: []`.
