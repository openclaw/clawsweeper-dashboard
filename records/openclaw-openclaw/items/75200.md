## Summary

Keep open. The PR targets a real Active Memory latency bug, but the current head still has correctness blockers in the new eager hook path and memory-only embedded runner path, so it should not merge as-is.

## What This Changes

The PR changes Active Memory recall to an eager memory-only embedded path, adds memory-tool/QMD fallback instrumentation, filters runtime tools, and includes Discord startup responsiveness changes.

## Best Possible Solution

Keep the latency-focused recall improvements only after the eager hook receives real prompt history, memory-only runs share the normal provider/transcript setup, and the user-facing change is recorded in the active changelog.

## Reproduction Assessment

Yes. The linked bug has concrete direct-chat timeout steps and logs, and the PR regressions are reproducible by inspecting PR head: eager hooks receive empty messages and memory-only runs branch away before shared setup.

## Solution Assessment

No. Starting recall earlier and bounding memory-only work is a reasonable direction, but this implementation is not yet the narrowest maintainable fix because it drops documented query context and provider compatibility setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass real history to eager turn-prepare hooks:** `src/agents/pi-embedded-runner/run.ts:692`
  - body: Active Memory is registered as an eager `agent_turn_prepare` hook, but the eager call is started with `messages: []`. That makes the default `recent` mode lose the conversation tail and also breaks `full` mode, even though the hook still builds its query from `event.messages`.
  - confidence: 0.9
- **[P2] Share embedded setup with memory-only runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1512-1521`
  - body: The new `memory-only` branch returns before the normal embedded attempt setup applies transcript policy, provider extra params, and live stream compatibility wrappers. Active Memory can inherit the current session model, so providers that rely on those wrappers can regress only for recall runs; keep the tool filtering but factor the shared setup before prompting.
  - confidence: 0.86
