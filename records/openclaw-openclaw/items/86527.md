## Summary

Keep this PR open for maintainer review: the patch addresses a source-reproducible voice-call webhook bug and I found no blocking correctness defect, but it intentionally changes the trusted-proxy predicate that gates forwarded host/proto headers.

## What This Changes

The branch adds voice-call webhook proxy IP normalization plus Twilio and Plivo regression tests so IPv4-mapped remote addresses match equivalent plain IPv4 trustedProxyIPs before forwarded headers are used.

## Best Possible Solution

Voice-call webhook security should accept equivalent plain IPv4 and IPv4-mapped trusted proxy addresses while preserving the existing forwarded-header opt-in gates and provider regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main compares trustedProxyIPs to remoteIP with literal includes, so trustedProxyIPs containing 127.0.0.1 misses remoteAddress ::ffff:127.0.0.1. I did not run a live provider callback in this read-only review.

## Solution Assessment

Yes, normalizing both configured proxy IPs and the request remote IP before the existing fromTrustedProxy gate is the narrow maintainable fix and matches the sibling media-stream path. The remaining decision is maintainer acceptance of the security-boundary equivalence.
