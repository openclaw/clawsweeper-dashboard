## Summary

Keep open. This is a useful Discord bug-fix PR from a member, but the current guard can turn mixed targeted-plus-broadcast final replies into fresh Discord creates without `allowed_mentions` suppression, so it needs a narrow delivery-safety fix before merge.

## What This Changes

The PR routes Discord live-preview final replies containing rewritten or literal user/role mentions through fresh-message delivery and adds regression tests for alias, literal, plain-handle, and broadcast-only cases.

## Best Possible Solution

Constrain the fresh-create fallback so targeted user/role mentions can ping without ever escalating `@everyone` or `@here`, and cover the mixed case in the Discord process tests.

## Reproduction Assessment

Yes, at source level. Current main edits eligible live-preview finals in place, while the normal create path is the one that applies `mentionAliases`; I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The edit-to-create fallback is the right shape for targeted mention pings, but it needs explicit broadcast suppression or constrained allowed-mentions handling before it is the safest merge path.
