## Summary

Keep open: current main still has the source-level failure shape reported here. The shared SSRF pinned lookup keeps mixed A/AAAA results and rotates through them, provider transport uses that pinned dispatcher path, and EADDRNOTAVAIL is still absent from the benign uncaught-exception network lists.

## What This Changes

Report that Gateway provider calls can crash on IPv6-unreachable hosts because SSRF DNS pinning rotates from IPv4 to AAAA records and EADDRNOTAVAIL is not treated as benign.

## Best Possible Solution

Fix the shared SSRF/provider transport path so pinned dispatchers do not select unreachable AAAA records in IPv4-working environments, classify EADDRNOTAVAIL consistently, and add focused regression coverage without provider-specific host exceptions.

## Reproduction Assessment

Yes, from source inspection: a mixed A/AAAA pinned lookup can return IPv6 on a later connection, provider traffic uses that lookup path, and EADDRNOTAVAIL is not currently classified benign. I did not run the full Gateway on an IPv6-unroutable host.

## Solution Assessment

Partly: adding EADDRNOTAVAIL to the benign list is necessary but not sufficient. The maintainable fix should also address pinned lookup family selection while preserving SSRF DNS pinning and private-network checks.
