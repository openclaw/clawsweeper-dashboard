## Summary

Keep open: current main still has the source-level failure shape reported here, with pinned SSRF lookup retaining AAAA records and rotating through them while the benign uncaught-exception network lists still omit EADDRNOTAVAIL.

## What This Changes

The issue reports a gateway crash when SSRF-pinned provider traffic to api.anthropic.com rotates from IPv4 to IPv6 on hosts where DNS returns AAAA records but global IPv6 routing is unavailable.

## Best Possible Solution

Fix this in the shared SSRF/provider transport path by preventing pinned lookup from selecting unreachable AAAA records in IPv4-working environments, classifying EADDRNOTAVAIL consistently, and adding focused regression coverage without adding provider-specific host exceptions.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a mixed A/AAAA pinned lookup can return IPv6 on a later provider request, and EADDRNOTAVAIL is not classified benign. I did not run the full gateway on an IPv6-unroutable WSL2 host.

## Solution Assessment

No single suggested one-line regex change is sufficient because the pinned lookup can still hand undici an unreachable IPv6 address. The narrow maintainable fix should address both family selection in the pinned dispatcher path and EADDRNOTAVAIL classification while preserving SSRF safeguards.
