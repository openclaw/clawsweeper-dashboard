## Summary

Keep open: current main still has the reported failure shape in the shared SSRF/provider transport path, with pinned lookup retaining A and AAAA answers, rotating through them, and leaving EADDRNOTAVAIL out of both transient and benign uncaught network classifications.

## What This Changes

The issue reports Gateway crashes on IPv6-unreachable hosts when SSRF DNS pinning can return an AAAA address for provider traffic and the resulting EADDRNOTAVAIL is not handled as non-fatal.

## Best Possible Solution

Fix the shared SSRF/provider transport path so pinned dispatchers do not select unreachable AAAA records in IPv4-working environments, classify EADDRNOTAVAIL consistently, and add focused regression coverage without provider-specific host exceptions.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: a mixed A/AAAA pinned lookup can return IPv6 on a later provider request, and EADDRNOTAVAIL is not classified benign or transient. I did not run the full Gateway crash on an IPv6-unroutable WSL2 host.

## Solution Assessment

No single one-line benign-regex change is enough because the pinned lookup can still hand undici an unreachable IPv6 address. The narrow maintainable fix should address pinned family selection or fallback plus EADDRNOTAVAIL classification while preserving SSRF safeguards.
