## Summary

Keep open: current main still has the changed-connection node approval rejection path for chat-sourced no-device replays, and https://github.com/openclaw/openclaw/pull/79781 is an open closing PR for this issue. Closing before that PR merges would be premature.

## What This Changes

The issue requests reliable Telegram/native chat approval handling for macOS `host=node` exec commands when approvals are submitted from the same chat but the approved replay uses a different backend connection.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79781, then close this issue as implemented with commit and release provenance after merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core failure mode from source and reporter logs: current main rejects changed-connection no-device node approval replays, and the WeCom comment shows the same `APPROVAL_CLIENT_MISMATCH` after `connId` changes; I did not run a live Telegram/WeCom macOS-node setup in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes, the linked PR direction is the narrowest maintainable fix seen here because it carries existing turn-source facts into the approved replay and preserves node, command, session, agent, and chat target bindings instead of relaxing the connection check globally.
