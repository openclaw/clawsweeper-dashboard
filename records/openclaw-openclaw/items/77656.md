## Summary

Keep open: current main still has a credible source-level node-approval failure path after Telegram resolves the approval, and the related reports do not cover this exact post-approval node forwarding mismatch.

## What This Changes

Investigate Telegram approvals for macOS node-hosted exec commands that appear approved but fail before the approved node command runs.

## Best Possible Solution

Preserve the anti-replay guard while giving the post-approval node.invoke path a stable trusted requester identity or explicit approval-forwarding context, then add regression coverage for two-phase node approvals across separate backend gateway calls.

## Reproduction Assessment

Yes, at source level: a node approval registered by one backend gateway connection can later be forwarded through node.invoke on another backend connection with no device id, which the current guard rejects. I did not run the live macOS Telegram scenario in this read-only review.

## Solution Assessment

No, closing is not the best path because current main still appears vulnerable to the reported node-only failure mode. The narrow maintainable fix is in the node approval requester/forwarding identity path, not in Telegram approval parsing alone.
