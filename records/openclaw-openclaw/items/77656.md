## Summary

Keep open: current main still has the source-level changed-connection rejection for chat-channel host=node approval replay, and the open linked PR is the active closing implementation candidate.

## What This Changes

Telegram and other chat-channel approvals for host=node exec commands should approve and replay reliably after transient backend connection changes instead of failing as unknown, expired, denied, or client-mismatched.

## Best Possible Solution

Land or replace the open chat-channel replay fix with reviewed code that preserves node, command, session, and stable chat-source bindings, then close this issue as implemented after merge.

Do we have a high-confidence way to reproduce the issue? ## Solution Assessment

Is this the best way to solve the issue? Yes for the active direction: carrying stable chat-source facts into approved replay and validating them is narrower than relaxing the connection check globally.
