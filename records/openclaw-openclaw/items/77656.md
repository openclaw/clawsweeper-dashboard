## Summary

Keep open: the issue has concrete Telegram and WeCom evidence, current main still lacks the stable chat-source replay binding needed for changed chat-agent connections, and an open PR already references this issue with closing syntax.

## What This Changes

The issue asks for Telegram and other chat-channel `host=node` exec approvals to replay reliably after same-chat approval, with clear routing/error behavior when approval cannot be applied.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79781 so chat-sourced node exec approvals bind to stable source metadata instead of transient WebSocket connection ids.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the WeCom comment identifies changed-connection replay hitting `APPROVAL_CLIENT_MISMATCH`, and current main still has that rejection path when no stable device/bridgeable requester binding applies; I did not run a live Telegram/WeCom macOS-node setup.

## Solution Assessment

Is this the best way to solve the issue? Yes for the active direction: carrying stable chat-source bindings into approved node invokes is narrower and safer than relaxing the replay client check globally.
