## Summary

Keep open: the issue remains valid on current main because chat-channel `host=node` approval replay still lacks the stable chat-source binding needed after transient backend connection changes, and the closing implementation is still an open PR.

## What This Changes

The issue asks for Telegram and other chat-channel `host=node` exec approvals to approve and replay reliably for macOS node commands, or to clearly report why approval cannot be routed.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79781 so chat-sourced node exec approval replay carries stable source bindings without weakening the existing client, node, command, session, and expiry checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the WeCom follow-up identifies `APPROVAL_CLIENT_MISMATCH` after a changed `connId`, and current main still lacks the chat-source replay metadata needed for that path; I did not run a live Telegram/WeCom macOS-node setup.

## Solution Assessment

Is this the best way to solve the issue? Yes for the active direction: carrying stable chat-source facts through approved node invokes is narrower than relaxing the approval client guard globally, and the linked PR is the right implementation candidate to review.
