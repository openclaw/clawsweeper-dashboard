Keep open: current main still has the changed-connection node approval rejection path for chat-sourced host=node approvals, and an open maintainer PR with closing syntax is the active fix candidate.

## What This Changes

The issue reports Telegram and WeCom host=node exec approvals failing after the user approves because the approved node.invoke replay is rejected as an invalid client.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/79781 with stable chat-source approval replay bindings, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: current main still rejects changed-connection no-device node approval replays without a matching stable device/backend bridge, and the WeCom follow-up shows that failure mode with APPROVAL_CLIENT_MISMATCH. I did not run a live Telegram/WeCom macOS-node setup.

## Solution Assessment

Yes, the active direction is the narrow maintainable fix: carry existing chat-source facts into approved node invoke replay and validate them, instead of relaxing the client check globally. The issue should stay open until the linked implementation merges or is superseded.
