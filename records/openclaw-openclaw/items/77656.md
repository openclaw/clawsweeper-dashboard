Keep open: this has concrete Telegram and WeCom evidence for chat-channel `host=node` approval replay failing after transient backend connection changes, and the linked closing PR is still open.

## What This Changes

The issue reports that Telegram chat approvals for `host=node` exec commands can be accepted by the user but fail before execution with denied, expired, or client-mismatch approval errors while gateway-hosted approvals work.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79781, preserving stable chat-source approval bindings for node exec replay across transient backend connections.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the WeCom logs identify changed-`connId` replay hitting `APPROVAL_CLIENT_MISMATCH`, and current main still has that rejection path for chat-channel node approval replay; I did not run a live Telegram/WeCom macOS-node setup.

## Solution Assessment

Is this the best way to solve the issue? Yes for the active direction: carrying stable chat-source bindings through approved node invoke is narrower than relaxing the client check globally, though the open PR still needs normal PR review and merge handling.
