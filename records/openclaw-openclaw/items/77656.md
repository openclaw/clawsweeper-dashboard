## Summary

Keep open: the issue has concrete Telegram and WeCom evidence for chat-channel `host=node` approval replay failing after backend connection changes, and the open closing PR is still the active implementation candidate.

## What This Changes

The issue reports that chat-channel approvals for `host=node` exec commands can be accepted by the user but replayed on a different backend connection and rejected before the node command runs.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/79781 so approved chat-sourced node exec replays use stable source bindings without weakening device, node, command, session, or changed-target checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the WeCom evidence names the changed-`connId` replay failure, and current main still has the corresponding no-stable-device rejection path for changed chat-channel replay clients; I did not run a live Telegram/WeCom macOS-node setup.

## Solution Assessment

Is this the best way to solve the issue? Yes for the active direction: carrying stable chat-source replay metadata is narrower than relaxing the client check globally and keeps the approval binding model intact.
