## Summary

Keep open: this is a plausible fix for a real stale node-host update failure, but the patch has a blocking wiring gap because the new detail code never reaches the node-host exit handler through `GatewayClient`, and the supplied proof does not show the actual exit/restart path.

## What This Changes

The PR adds a `CLIENT_VERSION_MISMATCH` gateway handshake detail, rejects local node clients with mismatched release versions, and teaches the node-host reconnect handler to exit for that detail code.

## Best Possible Solution

Wire `CLIENT_VERSION_MISMATCH` into the `GatewayClient` reconnect-pause predicate, add coverage for the actual gateway rejection to node-host exit path, and require redacted real logs or terminal proof showing the local process exits or supervisor restarts while remote skew remains allowed.

## Reproduction Assessment

Yes from source inspection: a `CLIENT_VERSION_MISMATCH` handshake error sets the pending detail, but `GatewayClient.shouldPauseReconnectAfterAuthFailure` returns false for that detail, so `onReconnectPaused` is not invoked. I did not run a live repro because this review is read-only.

## Solution Assessment

No: the gateway-side version check is directionally appropriate, but the implementation misses the client pause wiring required for the node host to exit. The safer path is to add that wiring plus focused coverage and real restart proof.
