## Summary

Keep open: the PR targets a real stale local node-host failure and now wires the mismatch detail into the node-host exit path, but it should not merge as-is because the loopback-only locality gate also matches SSH-tunneled remote nodes that the PR says must remain exempt.

## What This Changes

The PR adds a `CLIENT_VERSION_MISMATCH` gateway handshake detail, rejects local node clients with mismatched release versions, and makes node-host exit for that detail so a supervisor restarts it.

## Best Possible Solution

Keep the stale local node-host restart behavior, but gate it on a reliable same-install or local-supervisor signal that excludes SSH tunnels and remote nodes, with regression coverage for local, direct remote, and SSH-tunneled remote cases.

## Reproduction Assessment

Yes for the review finding from source and docs: an SSH-tunneled node arrives at the gateway as `127.0.0.1`, so the new loopback gate would reject a mismatched released node and the node-host exit set would terminate it. I did not run the live SSH tunnel scenario in this read-only review.

## Solution Assessment

No: restarting stale same-install local nodes is a good direction, but loopback address alone is not the right boundary because SSH tunnels also present as loopback. The safer solution needs an explicit same-install/local-supervisor signal or a maintainer-approved locality contract.
