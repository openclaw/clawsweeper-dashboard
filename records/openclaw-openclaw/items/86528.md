## Summary

Keep open: the stale node-host restart direction is useful, and the author's follow-up fixed the reconnect-pause path, but the patch still uses loopback locality as the same-install boundary. Current docs support remote node hosts over SSH tunnels to `127.0.0.1`, so mismatched tunneled nodes can still be rejected and forced to exit.

## What This Changes

The PR adds a `CLIENT_VERSION_MISMATCH` connect detail, rejects mismatched released local node clients, and makes the node host exit on that detail so a supervisor can restart it.

## Best Possible Solution

Keep the stale same-install local node restart behavior, but gate it on a reliable same-install/local-supervisor signal that excludes SSH-tunneled and remote nodes, with regression coverage for both paths.

## Reproduction Assessment

Yes for the review finding from source and docs: the documented SSH-tunneled node flow connects to `127.0.0.1`, and current main treats no-forwarded-header loopback sockets as local. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No: restarting stale same-install local nodes is a good direction, but loopback address alone is not the right boundary because SSH tunnels also present as loopback. The safer solution needs an explicit same-install/local-supervisor signal or a maintainer-approved locality contract.
