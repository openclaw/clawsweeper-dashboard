## Summary

Keep open: the latest push fixes the reconnect-pause path and supplies useful real behavior proof, but the new fail-closed guard still treats loopback/no-forwarded-header nodes as same-install local. That can maroon SSH-tunneled or subordinate nodes during version skew, so the PR needs maintainer contract work before merge.

## What This Changes

The PR adds a gateway/node-host version-mismatch handshake detail that rejects released local node clients with stale versions and makes the node host pause and exit so a supervisor can restart it, with gateway and node-host regression tests.

## Best Possible Solution

Keep the stale local-node recovery idea, but gate the forced restart on a stronger same-install local-node signal or an explicitly approved policy that preserves a maintenance path for tunneled and subordinate nodes.

## Reproduction Assessment

Yes for the merge blocker at source level: `isLocalDirectRequest` treats loopback/no-forwarded-header connections as local, and the remote-mode SSH tunnel docs say the gateway sees tunneled traffic as `127.0.0.1`. I did not run a live SSH-tunnel E2E in this read-only review.

## Solution Assessment

No: the stale local-node restart path is useful, and the latest push fixed the reconnect pause, but loopback alone is not a safe same-install signal. The safer fix is to preserve remote/tunneled skew tolerance unless the node can prove it shares the gateway install.

- **[P1] Don't treat loopback-only nodes as safe to restart:** `src/gateway/server/ws-connection/message-handler.ts:1595`
  - body: This guard uses `isLocalClient` as proof that the node host shares the gateway install, but current `isLocalDirectRequest` returns true for any loopback socket with no forwarded headers. The docs for macOS remote mode say SSH tunnels make the gateway see remote node traffic as `127.0.0.1`, so an older remote/subordinate node reached through such a tunnel would now receive `CLIENT_VERSION_MISMATCH`, exit, and restart against the same old install forever, recreating the marooned-node failure tracked at https://github.com/openclaw/openclaw/issues/83736. Gate this on a stronger same-install signal, or keep tunneled/remote nodes on the tolerated path.
