## Summary

Keep open. Current main still exposes `message:received` without `agentId` or `workspaceDir`, while the shared dispatcher has already resolved both values before emitting the internal hook. This is a narrow, source-backed hook-contract gap rather than a stale support report or duplicate cleanup candidate.

## What This Changes

Request that `message:received` hook events expose `workspaceDir` and/or `agentId` so WAL protocol hooks can write per-agent session state instead of the main workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` hook context, populate them from already-resolved routed session data at every internal emitter, update docs and focused tests, and keep WAL-specific file-writing behavior outside core.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while `dispatchReplyFromConfig` has both before emission. The exact WAL artifact symptom is plausible but was not directly reproduced in this read-only review because the reported dist hook file is not in source.

## Solution Assessment

Yes. An additive hook-context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while giving workspace-scoped hooks the route data they need; moving WAL behavior into core would cross the plugin boundary.
