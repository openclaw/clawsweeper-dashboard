## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context even though `dispatchReplyFromConfig` resolves both before hook emission, so the reported per-agent WAL hook failure remains a valid hook API bug.

## What This Changes

The issue requests route identity on `message:received` hook events so WAL/session-state hook data can be written to the correct per-agent workspace.

## Best Possible Solution

Add optional route identity to the internal inbound message hook context at route-aware emission sites, update focused docs/tests/changelog, and keep WAL memory-writing policy outside core.

## Reproduction Assessment

Yes. Source inspection on current main shows `dispatchReplyFromConfig` computes `sessionAgentId` and `workspaceDir`, but the `message:received` type, mapper, docs, and dispatcher payload omit those fields.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while exposing route data already available at the main emission point.
