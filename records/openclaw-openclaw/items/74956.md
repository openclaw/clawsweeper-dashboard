## Summary

Keep open. Current main still omits route identity from the `message:received` hook context even though the shared dispatcher already has the resolved `agentId` and `workspaceDir`, so the reported per-agent WAL hook failure remains a valid hook API bug.

## What This Changes

The issue asks for the inbound `message:received` hook context to expose `workspaceDir` and/or `agentId`, or for the WAL protocol hook to use an event that already exposes those fields.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the canonical/internal inbound message hook context at route-aware emission sites, update public hooks docs and focused regression tests, and keep WAL memory-writing policy outside core.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the root cause: the dispatcher knows `sessionAgentId` and `workspaceDir`, but the `message:received` hook type, canonical mapper, docs, and tests do not expose those fields.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while exposing route data that is already available at the main emission point.
