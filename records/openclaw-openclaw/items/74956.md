## Summary

Keep open. Current main still omits route identity from the internal `message:received` hook context even though the dispatcher has already resolved `sessionAgentId` and `workspaceDir`, so the reported per-agent WAL failure remains a valid hook API bug with a narrow repair path and no fixing PR found.

## What This Changes

The issue reports that `message:received` hook events lack `workspaceDir` and `agentId`, causing per-agent WAL state writes to fall back to the main workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the route-aware internal inbound message hook context, update the mapper/docs/regression tests, and leave WAL memory-writing policy outside core.

## Reproduction Assessment

Yes. Source inspection on current main shows `dispatchReplyFromConfig` resolves `sessionAgentId` and `workspaceDir` before creating the internal `message:received` event, while the hook type, mapper, docs, and tests omit those fields.

## Solution Assessment

Yes. Additive route-field enrichment in the existing hook context is the narrowest maintainable fix because it preserves current hook consumers and avoids duplicating workspace/session discovery inside WAL hooks.
