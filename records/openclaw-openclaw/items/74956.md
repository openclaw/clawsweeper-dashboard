## Summary

Keep open. Current main still omits explicit `agentId` and `workspaceDir` from the internal `message:received` hook context even though the main dispatcher already resolves both values before emitting the event, and the adjacent open hook-policy PR does not change the handler protocol.

## What This Changes

The issue requests that `message:received` hook events expose agent/workspace routing data so per-agent hook state can be written to the correct workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal inbound message hook context at route-aware emitters, update docs and regression tests, and keep WAL-specific file-writing behavior outside core.

## Reproduction Assessment

Yes. Current source and docs reproduce the root cause: the dispatcher resolves the agent/workspace before hook emission, but the emitted `message:received` context, mapper, docs, and tests omit those fields.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while exposing route data that is already known at the emission points.
