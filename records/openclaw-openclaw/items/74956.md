## Summary

Keep open. Current main still exposes `message:received` without `agentId` or `workspaceDir`, while the main dispatcher resolves both values before emitting the hook; adjacent PR #38670 is hook policy work and does not supersede this handler-context gap.

## What This Changes

Add route/workspace identity to the internal `message:received` hook context so workspace-scoped hooks can target per-agent state.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal/canonical inbound message hook contract wherever emitters already have route data, with docs and focused tests updated while WAL file-writing remains plugin-owned.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while the main dispatcher resolves both before emitting the hook. The exact WAL write symptom was not run in this read-only review because the cited dist hook is not in source.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers, avoids moving memory behavior into core, and gives workspace-scoped hooks data already available at emission time.
