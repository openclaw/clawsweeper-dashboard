## Summary

Keep open. Current main still drops the already-known agent/workspace routing data from the internal `message:received` hook context, so the reported hook-contract gap remains valid and narrow enough for a focused fix.

## What This Changes

The issue asks for `message:received` hook events to expose per-agent routing context such as `agentId` and/or `workspaceDir` so hook handlers can write per-agent state instead of defaulting to the main workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` context, populate them from already-resolved session routing data at the shared dispatcher and relevant channel ingest emitters, update docs and focused tests, and keep WAL-specific file-writing behavior in the hook or plugin layer.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir` while the shared dispatcher has both before emitting. The exact WAL artifact write path was not directly exercised because the reported dist hook file is absent from this checkout.

## Solution Assessment

Yes. An additive hook-contract enrichment is the narrowest maintainable fix; moving WAL or active-memory write policy into core would cross the plugin/hook ownership boundary.
