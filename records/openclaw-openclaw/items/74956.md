## Summary

Keep open. Current main still exposes `message:received` without `agentId` or `workspaceDir`, while the main dispatcher and route-specific channel paths already resolve the missing routing data before emitting the hook. The earlier ClawSweeper assessment remains current; related PR #38670 is adjacent hook policy work, not an exact handler-context fix.

## What This Changes

Add route/workspace identity to the internal `message:received` hook context so workspace-scoped hooks can target per-agent state instead of defaulting to the main workspace.

## Best Possible Solution

The maintainable end state is an additive internal hook contract where `message:received` carries optional `agentId` and `workspaceDir` whenever the emitter has already resolved them, with docs and tests updated while WAL or active-memory file-writing behavior stays plugin-owned.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while the main dispatcher resolves both before emitting the hook. The exact WAL write symptom was not run because this was a read-only review and the cited dist hook is not in source.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers, avoids moving memory behavior into core, and gives workspace-scoped hooks the routing data already available at emission time.
