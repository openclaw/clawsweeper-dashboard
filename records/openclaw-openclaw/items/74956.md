## Summary

Keep open. Current main still has the reported internal hook contract gap: `message:received` carries inbound message metadata, but not the already-resolved agent/workspace route data that workspace-scoped hook handlers need. The related open hook-policy PR is adjacent but explicitly does not change the hook handler protocol, so it does not supersede this bug.

## What This Changes

The issue requests that `message:received` hook context include agent/workspace routing data, such as `workspaceDir` or `agentId`, so per-agent WAL hook handlers write state to the correct workspace.

## Best Possible Solution

Add an additive internal hook contract for `message:received` that carries optional route identity and workspace path wherever the dispatcher or channel emitter already has those values. Keep WAL or memory file-writing plugin-owned, and update docs, mapper tests, dispatcher tests, and channel-specific emitter tests so third-party hooks can rely on the new fields without breaking existing consumers.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while the shared dispatcher resolves both before emitting the hook. The exact WAL artifact behavior was not live-run because this was a read-only review and that generated dist hook is not in source.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers, avoids moving WAL behavior into core, and uses route data already available at emission sites.
