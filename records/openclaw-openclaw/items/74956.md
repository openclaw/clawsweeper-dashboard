## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context, while the shared dispatcher has already resolved both values before emitting the hook; the prior ClawSweeper keep-open comment remains accurate against the newer main SHA.

## What This Changes

The issue requests routed-agent context on `message:received` hooks so WAL-style per-agent state writes can target the correct workspace instead of the default workspace.

## Best Possible Solution

Expose optional routed-agent fields on the internal `message:received` context, populate them from already-resolved session or route data at every internal emitter, update docs/tests/changelog, and keep WAL-specific file-writing policy outside core.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir` while `dispatchReplyFromConfig` has both values before hook emission. The exact WAL artifact symptom is plausible but not directly reproducible from this checkout because the reported dist hook file is absent.

## Solution Assessment

Yes, an additive hook-contract enrichment is the narrowest maintainable fix. Rebuilding WAL behavior in core or relying on hook authors to parse undocumented session keys would be less stable than exposing the routed context directly.
