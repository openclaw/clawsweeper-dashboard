## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context, while the shared reply dispatcher resolves both before emitting the hook. The existing ClawSweeper comment remains accurate against `7340c0322fc3f7b1279729663f3ae37bd4bf978f`, and the adjacent related reports do not supersede this routed-context gap.

## What This Changes

The issue requests that `message:received` hook events expose routed-agent context, such as `agentId` and `workspaceDir`, so WAL-style per-agent state writes can target the correct agent workspace.

## Best Possible Solution

Expose optional routed-agent fields on the internal `message:received` context, populate them from already-resolved session or route data wherever available, document the additive fields, and keep WAL-specific file-writing policy outside core.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir` while `dispatchReplyFromConfig` already has both values before emission. The exact WAL artifact symptom is plausible but not directly reproducible from this checkout because the reported dist hook file is absent.

## Solution Assessment

Yes. An additive hook-contract enrichment is the narrowest maintainable fix because it exposes generic routed-agent context without moving WAL-specific behavior into core or requiring hook authors to parse undocumented session keys.
