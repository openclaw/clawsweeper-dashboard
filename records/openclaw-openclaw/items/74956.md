## Summary

Keep open. Current `main` still omits `agentId` and `workspaceDir` from the internal `message:received` hook context even though the dispatcher has both values before emitting the event; related PR #38670 overlaps hook policy but explicitly does not change the hook handler protocol.

## What This Changes

The issue requests per-agent routing identity on `message:received` hook events so WAL/session-state hook data can be written to the receiving agent workspace instead of the main workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the canonical and internal inbound message hook context at route-aware emitters, update docs and focused tests, and keep WAL-specific file-writing behavior outside core.

## Reproduction Assessment

Yes. Current source and docs reproduce the gap: the dispatch path resolves the agent/workspace before hook emission, while the type, mapper, docs, and mapper tests omit those fields from `message:received`.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while exposing route data that already exists at the main emission point.
