## Summary

Keep open. Current `main` still exposes `message:received` without `agentId` or `workspaceDir`, while the shared dispatcher has already resolved both the session agent and workspace before emitting the internal hook. This is a narrow hook-contract gap with a clear repair path, not a stale report or duplicate cleanup candidate.

## What This Changes

The issue requests that `message:received` hook events expose routing context such as `workspaceDir` or `agentId` so workspace-scoped hooks can write per-agent state correctly.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` hook context, populate them from already-resolved routed session data at every internal emitter, update docs and focused tests, and keep WAL-specific file-writing behavior outside core.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while `dispatchReplyFromConfig` has the resolved workspace before emission. The exact WAL write-location symptom was not directly reproduced because this is a read-only review and the cited dist hook is not in source.

## Solution Assessment

Yes. An additive hook-context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while giving workspace-scoped hooks the route data they need; moving WAL behavior into core would cross the plugin boundary.
