## Summary

Keep open. Current main still exposes `message:received` without `agentId` or `workspaceDir`, even though the shared dispatcher has already resolved both before emitting the internal hook. The prior ClawSweeper comment remains accurate against current `main`, and this looks like a narrow hook-contract gap rather than a duplicate or stale support report.

## What This Changes

The issue asks for the internal `message:received` hook context to include enough routed-agent context, such as `agentId` and `workspaceDir`, so per-agent hook state is written to the correct workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` hook context, populate them from already-resolved routed session data at each emitter, update docs/tests, and keep WAL-specific file-writing policy outside core.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir` while `dispatchReplyFromConfig` already has both values. The exact WAL artifact symptom is plausible but not directly reproducible from this checkout because the reported dist hook file is absent.

## Solution Assessment

Yes. An additive hook-contract enrichment is the narrowest maintainable fix; rebuilding WAL behavior in core would cross the wrong boundary.
