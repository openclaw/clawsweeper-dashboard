## Summary

Keep open. Current main still lacks explicit `agentId` and `workspaceDir` on the internal `message:received` hook context even though the dispatcher has both values before emitting the event, so the reported hook API gap remains valid.

## What This Changes

The issue requests route identity on `message:received` hook events so per-agent hook handlers can write to the correct workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the canonical/internal inbound message hook context where route-aware emitters already know them, update docs and focused tests, and keep WAL or memory file-writing behavior outside core.

## Reproduction Assessment

Yes. Current source and docs reproduce the root cause: the dispatcher resolves the agent/workspace before hook emission, but the hook type, mapper, docs, and tests omit those fields from `message:received`.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while exposing route data that is already available at the main emission point.
