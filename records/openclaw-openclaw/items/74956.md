## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context even though the dispatcher has already resolved both values before emitting the hook, so the reported per-agent WAL hook failure remains valid.

## What This Changes

The issue requests route identity on `message:received` hook events so WAL/session-state hook data can be written to the correct per-agent workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal inbound message hook context at the route-aware emission path, update focused tests/docs/changelog, and leave WAL memory-writing policy outside core.

## Reproduction Assessment

Yes. Source inspection on current main shows `dispatchReplyFromConfig` computes `sessionAgentId` and `workspaceDir`, but the `message:received` type, mapper, docs, and emitted payload omit those fields.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers while exposing route data that is already available at the main emission point.
