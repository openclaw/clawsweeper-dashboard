## Summary

Keep open. Current main still exposes internal `message:received` without `agentId` or `workspaceDir`, even though the main dispatcher resolves both before emitting the hook; open PR #38670 is hook policy work and does not supersede this handler-context gap.

## What This Changes

The issue reports that internal `message:received` hook events lack route identity fields, preventing workspace-scoped hooks such as `wal-protocol` from writing per-agent state.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the canonical/internal inbound message hook context wherever emitters already have route data, then update docs and focused tests while keeping WAL file-writing plugin-owned.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while the main dispatcher resolves both values before emitting. The exact WAL write symptom was not run in this read-only review because the cited dist hook is not present in source.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers, avoids moving memory behavior into core, and exposes route data already available at emission time.
