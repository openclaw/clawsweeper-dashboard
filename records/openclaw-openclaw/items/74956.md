## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context, even though the shared dispatcher resolves both before emitting the hook; related PR #38670 is policy work and does not supersede this handler-context gap.

## What This Changes

The issue requests enriching `message:received` hook context so WAL-style hooks can write per-agent state to the correct workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the canonical/internal inbound message hook contract where the emitter already knows the route, update docs and focused tests, and keep WAL file-writing behavior plugin-owned.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while the main dispatcher resolves both before emitting the hook. The exact WAL file-write symptom was not executed because this was a read-only review and the cited dist hook is not in source.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable fix because it preserves existing hook consumers and passes through route data already available at emission time.
