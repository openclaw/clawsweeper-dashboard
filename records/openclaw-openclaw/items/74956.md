## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context even though the main dispatcher resolves both before emitting the event; the related hook-policy PRs do not cover this handler-context gap.

## What This Changes

Expose per-agent route context on `message:received` hooks so WAL-style hooks can write to the correct agent workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the canonical and internal inbound message hook contract, populate them wherever the emitter already knows the route, and keep WAL file-writing behavior plugin-owned.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir`, while the dispatcher resolves both before emitting the hook. The exact WAL file write was not run in this read-only review because the cited dist hook is not in source.

## Solution Assessment

Yes. Additive context enrichment is the narrowest maintainable solution because it preserves existing hook consumers, avoids moving memory behavior into core, and passes through route data already available at emission time.
