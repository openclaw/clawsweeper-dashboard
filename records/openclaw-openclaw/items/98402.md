## Summary

Keep this PR open: the parser hardening addresses a real channel ingress queue crash path, but the current patch can still turn corrupt pending payload rows into invisible claimed rows and still conflates valid JSON null with parse failure.

## What This Changes

The PR guards channel ingress queue JSON decoding and adds SQLite-backed corrupt-row tests for pending, claimed, and completed duplicate-detection paths.

## Best Possible Solution

Keep the shared parser hardening, but use a tagged parse result or non-null failure sentinel and validate or explicitly tombstone corrupt pending payloads before claimNext and claim mutate queue state.

## Reproduction Assessment

Yes. Source inspection on the PR head shows claimNext and claim update pending rows before proving claimedRecord can decode the payload, so a corrupt payload can become an invisible claimed row; I did not run tests because this review is read-only.

## Solution Assessment

No. The parser guard is the right shared chokepoint for list and duplicate-detection crashes, but this implementation needs a non-null failure sentinel and pre-claim validation or tombstoning before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate payloads before claiming rows:** `src/channels/message/ingress-queue.ts:584-599`
  - body: This still lets claimNext update a selected pending row to `claimed` before checking whether `claimedRecord()` can decode the payload; direct `claim()` has the same post-update decode shape. With a corrupt `payload_json`, the caller gets null while the row is hidden from listPending/listClaims and queue-local stale recovery, so decode or tombstone corrupt rows before the status update.
  - confidence: 0.94
- **[P2] Distinguish parse failures from JSON null:** `src/channels/message/ingress-queue.ts:221-225`
  - body: `JSON.parse("null")` is valid, but this helper uses null for both a valid value and a parse failure. That makes legitimate null payloads look corrupt, drops explicit null metadata, and currently fails lint on the redundant `unknown | null` return type; use a tagged parse result or another non-null failure sentinel.
  - confidence: 0.9
