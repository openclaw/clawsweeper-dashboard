## Summary

Keep open: the updated branch fixes the earlier parse sentinel and pre-claim mutation gaps, but claimNext can still stall behind a corrupt first pending row when no deriveLaneKey callback is supplied, and the live lint job has a one-line test failure.

## What This Changes

The PR hardens channel ingress queue JSON parsing, skips or omits corrupt row JSON, validates payloads before claiming, and adds SQLite-backed corrupt-row regression tests.

## Best Possible Solution

Keep the shared parser hardening, but make claimNext scan past or explicitly tombstone corrupt pending rows regardless of deriveLaneKey, then remove the lint failure and keep focused queue tests green.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: insert a corrupt pending row sorted before a valid row, then call claimNext() without deriveLaneKey; the PR filters the bad row after a one-row SQL limit and returns null.

## Solution Assessment

No. The parser guard and pre-claim validation are the right owner-boundary fix, but claimNext should scan or tombstone corrupt rows without requiring deriveLaneKey and the lint blocker should be removed before merge.
