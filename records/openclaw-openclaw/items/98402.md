## Label Justifications

- `P2`: This is a focused channel ingress queue resilience fix with limited blast radius but real message processing impact.
- `merge-risk: 🚨 compatibility`: The PR changes the queue API semantics for valid JSON null payloads and metadata by treating the same value as parse failure.

## Summary

Keep open: the parser hardening is useful, but the PR still can mark corrupt pending payload rows as claimed before decode validation and it conflates valid JSON null with parse failure.

## What This Changes

The PR wraps channel ingress queue JSON parsing, skips or omits corrupted persisted JSON rows in listings and duplicate detection, and adds SQLite-backed regression tests.

## Best Possible Solution

Preserve the shared parser hardening, but use a tagged parse result and validate or explicitly fail/tombstone corrupt pending rows before claimNext or claim updates status.

## Reproduction Assessment

Yes from source inspection: on the PR branch, claimNext and claim can update corrupt pending payload rows before proving the payload decodes. I did not run tests because this review is read-only.

## Solution Assessment

No; the parser chokepoint is reasonable, but the fix is not complete until claim mutation validates or tombstones corrupt payloads and parse failure is distinct from valid JSON null.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate payloads before claiming rows:** `src/channels/message/ingress-queue.ts:559-568`
  - body: This still lets claimNext select a raw pending row before proving baseRecord can decode it, then updates the row to claimed and only afterward returns null from claimedRecord. A corrupt payload can become invisible to both listPending and listClaims, and recoverStaleClaims depends on listClaims, so decode or tombstone the row before the status update in both claimNext and direct claim.
  - confidence: 0.94
- **[P2] Keep JSON null distinct from parse failure:** `src/channels/message/ingress-queue.ts:221-225`
  - body: JSON.parse("null") is valid, but the new helper returns null for both a valid JSON null and a parse failure. That makes legitimate null payloads look corrupt during enqueue/list/claim and drops explicit null metadata, so use a tagged parse result or separate failure sentinel.
  - confidence: 0.86
