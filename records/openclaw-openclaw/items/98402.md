## Summary

Keep open: the corrupted-row hardening is useful, but the PR is not merge-ready because corrupted pending payloads can still be moved to claimed state before decode validation and then disappear from recovery/listing paths.

## What This Changes

The PR changes the shared channel ingress queue JSON parsing path so corrupted persisted queue JSON is skipped or omitted instead of throwing through list and duplicate-detection operations.

## Best Possible Solution

Keep the shared parser hardening, but validate or tombstone corrupt pending payloads before claimNext and claim mutate row status, and use a parse result shape that preserves valid JSON null.

## Reproduction Assessment

Yes from source inspection: on the PR branch, claimNext and claim update pending rows before proving claimedRecord can decode the payload, so a corrupt payload can become an invisible claimed row. I did not run the branch because this review is read-only.

## Solution Assessment

No; the parser guard is the right chokepoint for listings and duplicate metadata, but claim paths need pre-update validation or an explicit failed/tombstoned state before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate payloads before claiming rows:** `src/channels/message/ingress-queue.ts:559-568`
  - body: This still selects raw pending rows and flips the selected row to `claimed` before checking whether `claimedRecord()` can decode the payload. With a corrupt `payload_json`, the method returns `null` after the update, leaving the bad row out of `listPending()` and also out of `listClaims()` because claimed rows with corrupt payloads are filtered, so `recoverStaleClaims()` cannot release it later. Decode or tombstone the row before the status update in both `claimNext()` and direct `claim()`.
  - confidence: 0.94
- **[P2] Distinguish JSON null from parse failure:** `src/channels/message/ingress-queue.ts:221-225`
  - body: `JSON.parse("null")` is valid, but the new helper returns `null` for both a valid JSON null and a parse failure. That makes a legitimate pending payload of `null` look corrupt during enqueue/list/claim, and similarly drops explicit null metadata. Use a tagged parse result or a separate failure sentinel so valid JSON values are preserved.
  - confidence: 0.86
