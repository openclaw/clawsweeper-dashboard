## Summary

Keep open: the branch fixes the direct claim-before-parse problem, but claimNext still caps the no-deriveLaneKey query at one row before filtering corrupt payloads, so a corrupt earliest pending row can make the queue report no work while valid later rows remain.

## What This Changes

The branch wraps channel ingress queue JSON parsing in tagged results, skips or omits corrupt row JSON, validates payloads before claims, and adds SQLite-backed corrupt-row regression tests.

## Best Possible Solution

Keep the parser hardening, but make claimNext scan a bounded batch or explicitly tombstone corrupt pending rows before returning null, with no-deriveLaneKey regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: insert a corrupt pending row sorted before a valid row, then call claimNext() without deriveLaneKey; PR head filters the bad row after a one-row SQL limit and returns null.

## Solution Assessment

No. The parser guard and pre-claim validation are the right owner-boundary fix, but claimNext still needs to scan or tombstone corrupt rows without requiring deriveLaneKey before the PR is the best fix.
