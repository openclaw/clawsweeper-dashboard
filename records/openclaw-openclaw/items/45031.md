## Label Justifications

- `P1`: The issue describes a live install-time security policy gap for untrusted third-party skills installed or updated through native ClawHub flows.
- `impact:security`: The requested behavior concerns supply-chain scanning, malicious skill detection, install blocking, and policy enforcement for untrusted code.

## Summary

Keep open: current main and the latest shipped release still leave native ClawHub skill install/update outside the local scanner and `before_install` policy path, while the requested default scanner/policy behavior needs maintainer security and product decisions. The AgentShield and ClawGuard comments are useful ecosystem context, but optional scanners do not replace an OpenClaw-owned install/update policy boundary.

## What This Changes

Request built-in security scanning and policy enforcement for skill installation, updates, loading, and ClawHub publishing/upload flows.

## Best Possible Solution

Define a maintained, pluggable install/update security policy that covers native ClawHub installs and updates, composes with the existing built-in scanner and `before_install` hook, and documents which publish/upload checks remain ClawHub service ownership.

## Reproduction Assessment

Yes, as a source-verifiable coverage gap: native ClawHub install/update reaches `installExtractedSkillRoot` with `scan: false`, while dependency and upload installs can enter scanner-backed paths. This is a security feature gap rather than a broken documented command with a simple live failure repro.

## Solution Assessment

Unclear. Hard-coding AgentShield into core is likely not the narrowest maintainable answer; a generic scanner/policy contract with deliberate defaults and room for ClawHub-published scanner implementations is safer.
