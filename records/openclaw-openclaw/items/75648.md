## Summary

Keep open: current main implements the provider timeout knob and much of the 429/rate-limit classification path, but the explicit client-visible Retry-After passthrough requested by the issue is still not present in the OpenAI-compatible error response contract.

## What This Changes

Request to make embedded-run provider timeouts configurable and preserve upstream 429/Retry-After semantics instead of collapsing them to generic timeouts.

## Best Possible Solution

Keep the existing provider-level timeout owner and add a narrow, documented Retry-After propagation path for OpenAI-compatible HTTP/SDK clients instead of introducing a parallel gateway-specific timeout key.

## Reproduction Assessment

Source-level yes for the remaining gap: current OpenAI-compatible error mapping does not carry Retry-After, but no live Anthropic/vision burst was run against current main.

## Solution Assessment

Partially: current main chose the right timeout owner with `models.providers.<id>.timeoutSeconds`, but the remaining client-visible Retry-After contract needs maintainer API judgment and focused proof.
