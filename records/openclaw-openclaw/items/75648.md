## Summary

Keep open. Current main covers the configurable timeout and progress-aware idle watchdog parts, but the provider metadata ask remains unresolved because OpenAI-compatible Gateway handlers still collapse embedded-run provider failures into generic internal errors and the failover contract has no Retry-After field.

## What This Changes

Make embedded-run provider timeouts configurable and surface upstream 429/Retry-After metadata for OpenAI-compatible Gateway clients.

## Best Possible Solution

Preserve the existing configurable/progress-aware timeout behavior, then add an additive provider-error metadata path that maps known provider status and Retry-After values onto OpenAI-compatible HTTP responses while leaving durable queueing to #49251.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused Gateway test can make the embedded agent path throw a rate-limit `FailoverError` and show `/v1/chat/completions` and `/v1/responses` currently return generic internal-error responses. The exact production timing pattern still needs live provider logs or an upstream stub.

## Solution Assessment

Yes for the narrowed repair: preserving provider metadata at the agent-to-HTTP boundary is the smallest maintainable fix. Changing timeout policy again or adding durable queueing would broaden this beyond the unresolved ask.

## Repair Work Prompt

Fix the remaining provider error metadata gap for https://github.com/openclaw/openclaw/issues/75648. Current main already supports `models.providers.*.timeoutSeconds` and a progress-aware embedded-run idle timeout, so do not rework timeout policy and do not implement durable queueing here. Add an additive Retry-After metadata path to provider/failover errors where upstream HTTP responses or SDK errors expose `retry-after-ms` or `retry-after`; preserve existing status/reason/provider/model/session/lane fields. Map known provider failures, especially 429 with Retry-After, through `/v1/chat/completions` and `/v1/responses` so non-streaming callers get the provider-appropriate HTTP status, OpenAI-compatible error body, and `Retry-After` header when available. For streaming responses, surface structured failure information as far as the protocol permits without breaking already-started SSE responses. Keep raw provider text sanitized and do not leak credentials. Add focused tests for failover metadata, provider transport retry-after extraction, and both OpenAI-compatible Gateway endpoints. Add a CHANGELOG.md fix entry because this changes user-facing runtime behavior.
