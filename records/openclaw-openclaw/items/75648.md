## Summary

Keep open: current main has the timeout configurability and progress-aware idle watchdog, but the provider Retry-After value still has no generic metadata path through failover and OpenAI-compatible gateway errors.

## What This Changes

The issue asks OpenClaw to make embedded-run provider timeouts tunable and to preserve provider 429/Retry-After details in client-visible errors.

## Best Possible Solution

Keep the shipped timeout controls and add generic provider retry-timing propagation through failover metadata and OpenAI-compatible HTTP/SSE error mapping.

## Reproduction Assessment

Yes, at source level: construct a provider rate-limit failure with status 429 and Retry-After at the embedded command boundary, then inspect the current failover mapper and endpoints, which have no retry timing field to propagate. The production throttling sequence itself was not rerun.

## Solution Assessment

Yes for the narrowed repair: extend existing failover metadata and `resolveOpenAiCompatError`, set non-streaming `Retry-After`, and include streaming-safe retry metadata. Changing timeout defaults or implementing durable queue/backoff policy would be broader than this issue’s remaining gap.
