## Summary

Keep open. Current main already has configurable whole-run and provider-scoped timeout paths, plus a chunk-resetting streaming idle watchdog, so the hard 60-second timeout claim is only partly current; however the structured provider 429/Retry-After propagation gap remains real on the embedded-run and OpenAI-compatible HTTP surfaces.

## What This Changes

The issue requests configurable embedded-run upstream provider timeouts and structured propagation of provider rate-limit details such as HTTP 429 and Retry-After, with streaming-aware timeout reset as a stretch goal.

## Best Possible Solution

Keep the existing timeout knobs, add an additive sanitized provider-error metadata path that preserves known status/reason/retry-after timing from provider responses through FailoverError or equivalent result metadata, and map that onto the OpenAI-compatible HTTP status/body/header surfaces without implementing durable queued retry behavior here.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused handler test can make the embedded agent path throw or return a rate-limit FailoverError and show `/v1/chat/completions` and `/v1/responses` currently return generic internal-error payloads without Retry-After. The exact live Anthropic throttling sequence is not high-confidence without controlled upstream logs or a stub provider.

## Solution Assessment

Yes for the narrowed repair direction; adding structured provider error metadata is the smallest maintainable fix for downstream backoff. Adding a new embedded-run-only timeout knob would duplicate existing `agents.defaults.timeoutSeconds` and `models.providers.<id>.timeoutSeconds` behavior unless a distinct missing boundary is later proven.
