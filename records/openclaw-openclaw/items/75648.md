## Summary

Keep open. Current main already has provider-scoped request timeout plumbing and a streaming idle watchdog that resets on chunks, so the hard 60-second timeout claim is only partly current. The remaining client-facing gap is real: provider 429/Retry-After data is used internally for SDK retry behavior, but the OpenAI-compatible HTTP surfaces still collapse agent failures into generic internal errors without structured retry metadata.

## What This Changes

Request to make embedded-run provider-call timeouts configurable and preserve provider 429/Retry-After details instead of collapsing them into generic timeout or surface-error results.

## Best Possible Solution

Add an additive provider-error metadata path that preserves known `status`, `reason`, and `retryAfterMs`/`retryAfterSeconds` from provider responses through FailoverError or equivalent agent result metadata, then map that onto the OpenAI-compatible HTTP response status/body/header without implementing durable queued retry behavior here.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused handler test can make the embedded agent path throw or return a rate-limit FailoverError and show `/v1/chat/completions` and `/v1/responses` currently return generic internal-error payloads. The exact Anthropic 47-call timing sequence is not high-confidence without live provider logs or a controlled upstream stub.

## Solution Assessment

Yes on the narrowed direction: keeping the existing timeout knobs and adding structured error/header propagation is the maintainable fix for downstream clients. Durable pause-and-replay should stay with #49251 rather than being folded into this issue.
