## Summary

Keep open. Current main and the latest release already cover the configurable provider timeout path and much of the 429/rate-limit classification, but source inspection does not show provider Retry-After being preserved as a structured embedded-run error field, and the active related Google 429 report means this should not be swept as fully implemented.

## What This Changes

Issue asks for configurable embedded-run upstream timeouts and preservation of provider 429/Retry-After details instead of collapsing them into generic timeout errors.

## Best Possible Solution

Keep the existing provider timeout knob as the timeout-tuning path, and decide whether embedded-run failover errors should carry provider Retry-After/status through the existing error/protocol surface instead of adding a duplicate gateway timeout knob.

## Reproduction Assessment

No high-confidence live repro was established. Source inspection does show the timeout portion is implemented, while provider Retry-After is parsed for SDK retry bypass but not exposed as a structured embedded-run failover error field.

## Solution Assessment

No, not as originally shaped. A new `gateway.embeddedRun.upstreamTimeoutMs` would duplicate the existing `models.providers.<id>.timeoutSeconds` path; the narrower remaining solution is a maintainer-approved structured Retry-After/status propagation path if clients should depend on it.
