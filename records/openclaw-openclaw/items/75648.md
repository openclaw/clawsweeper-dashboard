## Summary

Keep open. Current main already covers the configurable timeout and progress-aware idle timeout pieces, but the remaining provider metadata ask is still valid because OpenAI-compatible HTTP responses flatten provider failures instead of preserving status or Retry-After.

## What This Changes

The issue asks for configurable embedded-run provider timeouts and structured upstream 429/Retry-After metadata instead of generic timeout/internal-error surfaces.

## Best Possible Solution

Keep the existing timeout behavior and add a sanitized, additive provider-error metadata path from embedded-run failover through the OpenAI-compatible HTTP endpoints.

## Reproduction Assessment

Yes for the remaining source-level issue: a focused test can make the embedded agent path throw a rate-limit `FailoverError` and assert `/v1/chat/completions` and `/v1/responses` currently return generic internal-error payloads. The production provider-throttling sequence itself would still need live logs or a provider stub to reproduce end to end.

## Solution Assessment

Yes for the narrowed repair: preserving provider status/reason/Retry-After at the failover-to-HTTP boundary is the smallest maintainable change. Queueing delayed work remains better tracked by #49251 rather than folded into this fix.

## Repair Work Prompt

Fix the provider error metadata gap reported at https://github.com/openclaw/openclaw/issues/75648. Current main already supports `models.providers.<id>.timeoutSeconds` and progress-aware idle timeouts, so do not add a separate `gateway.embeddedRun` timeout unless the existing provider config cannot satisfy a test. Add a sanitized additive path that preserves provider failure metadata, especially 429/status/reason and Retry-After, from provider SDK errors through `FailoverError`/`describeFailoverError` into `/v1/chat/completions` and `/v1/responses`. For non-stream HTTP responses, map known provider status and `Retry-After` onto the HTTP status/header and body error code/type without exposing raw provider bodies. For streaming responses, use the endpoint's existing compatible failure event/chunk shape while including safe structured metadata where the schema permits. Add focused tests for `FailoverError`, provider transport Retry-After parsing/propagation if needed, OpenAI-compatible chat, and OpenResponses. Update `CHANGELOG.md` for the user-facing fix. Do not implement durable prompt queueing or broad rate-limit scheduling here; keep that with #49251.
