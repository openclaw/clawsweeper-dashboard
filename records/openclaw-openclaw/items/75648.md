## Summary

Keep open. Current main already makes the embedded model timeout configurable and progress-aware, but the provider error metadata request remains valid because OpenAI-compatible HTTP endpoints still flatten provider failures into generic internal errors without preserving upstream status or Retry-After.

## What This Changes

The issue requests configurable embedded-run provider timeouts, structured propagation of upstream 429/Retry-After metadata, and a progress-aware timeout reset for slow streams.

## Best Possible Solution

Keep the current timeout controls and progress-aware watchdog, then add an additive provider-error metadata path that maps known provider status/reason/Retry-After values onto `/v1/chat/completions` and `/v1/responses` responses.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused Gateway test can make the embedded agent path reject with a rate-limit `FailoverError` and show the OpenAI-compatible endpoints currently return generic internal-error responses without `Retry-After`. The production 47-call sequence itself is not reproduced without live provider logs or a stub.

## Solution Assessment

Yes for the narrowed repair: preserving provider metadata at the agent-to-HTTP boundary is the smallest maintainable fix. Raising timeouts and resetting streaming idle timers are already covered; durable queueing belongs in #49251.
