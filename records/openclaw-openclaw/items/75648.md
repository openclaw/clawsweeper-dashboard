## Summary

Keep open. Current main has the timeout configurability, progress-aware idle watchdog, and basic OpenAI-compatible status/error mapping, but the remaining Retry-After metadata still has no source field to travel through FailoverError and no endpoint path that sets a provider-derived Retry-After for `/v1/chat/completions` or `/v1/responses`.

## What This Changes

The issue asks for configurable embedded-run provider timeouts and preservation of provider 429/Retry-After metadata for OpenAI-compatible gateway callers.

## Best Possible Solution

Keep the shipped timeout controls and status mapping, then add additive provider retry-timing propagation through failover metadata and OpenAI-compatible HTTP/SSE error mapping.

## Reproduction Assessment

Yes for the remaining source-level gap: inject or construct a provider `rate_limit` failure with HTTP 429 and Retry-After at the embedded command boundary, then inspect the current mapper/endpoints, which have no retry timing field to propagate. The original production throttling sequence was not reproduced end to end.

## Solution Assessment

Yes for the narrowed repair: extend the existing failover metadata and `resolveOpenAiCompatError` path, then set non-streaming HTTP `Retry-After` and include streaming-safe retry metadata. Changing timeout defaults or adding provider-specific gateway policy would be broader than needed.
