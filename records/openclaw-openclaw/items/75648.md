## Summary

Keep open. Current main covers timeout configurability and progress-aware model idle handling, but the provider error metadata part remains valid because OpenAI-compatible HTTP endpoints still flatten provider failures to generic internal errors and there is no Retry-After field to propagate.

## What This Changes

Request to make embedded-run provider timeouts tunable and preserve upstream rate-limit metadata such as HTTP 429 and Retry-After for OpenAI-compatible clients.

## Best Possible Solution

Add an additive provider-error metadata path from provider transport and FailoverError into OpenAI-compatible HTTP responses, while keeping the existing timeout config and idle-timeout behavior.

## Reproduction Assessment

Yes for the remaining source-level gap: mock the embedded agent command to throw a rate-limit FailoverError and assert `/v1/chat/completions` and `/v1/responses` currently return generic internal-error responses without Retry-After. The reported 47-call production timing sequence still needs live logs or an upstream stub to verify end to end.

## Solution Assessment

Yes. The narrow maintainable fix is metadata propagation and endpoint mapping; timeout tuning and stream progress handling are already present, and durable queueing belongs with #49251.
