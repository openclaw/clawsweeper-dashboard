## Summary

Keep open. Current main already covers much of the timeout request through provider request timeout configuration and a per-chunk streaming idle watchdog, but the remaining HTTP compatibility gap is real: embedded-run provider failures can carry status/reason internally while `/v1/chat/completions` and `/v1/responses` still collapse them into generic internal errors without provider Retry-After metadata.

## What This Changes

The issue requests configurable embedded-run upstream timeouts, structured propagation of provider 429/Retry-After details, and streaming-aware timeout behavior for long-running provider calls.

## Best Possible Solution

Keep the existing provider timeout and streaming idle-timeout behavior, then add an additive provider-error metadata path that preserves known status/reason/retry-after values through the embedded-run failure boundary and maps them onto the OpenAI-compatible HTTP status, body, and `Retry-After` header. Durable queueing or pause/resume behavior should remain under #49251.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused handler test can make the embedded agent path throw or return a provider rate-limit failure and show the OpenAI-compatible endpoints currently return generic internal-error responses. The reported 47-call production timing sequence is not high-confidence without live logs or a deterministic upstream stub.

## Solution Assessment

Yes, the narrowed direction is the best maintainable fix: preserve provider error metadata at the agent-to-HTTP boundary while leaving existing timeout configuration and streaming idle behavior intact. Queueing, delayed retry scheduling, and session unblocking are separate product behaviors and should not be folded into this repair.
