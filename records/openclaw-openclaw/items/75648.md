## Summary

Keep open. Current main already covers the configurable timeout and progress-aware idle watchdog parts, but the upstream provider status/Retry-After metadata still does not reach OpenAI-compatible HTTP clients; both `/v1/chat/completions` and `/v1/responses` flatten ordinary agent failures to generic internal errors.

## What This Changes

Request configurable embedded-run upstream timeouts plus structured upstream 429/Retry-After propagation for downstream clients.

## Best Possible Solution

Keep the current timeout controls and progress-aware watchdog, then add an additive provider-error metadata path that maps known provider status/reason/Retry-After values onto `/v1/chat/completions` and `/v1/responses` responses.

## Reproduction Assessment

Yes for the remaining source-level gap: mock the embedded command path to reject with a rate-limit `FailoverError` carrying status 429 and retry timing, then verify the OpenAI-compatible endpoints currently return generic internal-error responses. The live production throttling sequence still needs provider logs or a stubbed provider to reproduce end to end.

## Solution Assessment

Yes for the narrowed repair: preserving provider metadata at the agent-to-HTTP boundary is the smallest maintainable fix. The timeout and streaming-idle asks are already covered; durable queueing/backoff policy belongs in the separate queueing follow-up.

## Repair Work Prompt

Fix the remaining provider metadata gap for https://github.com/openclaw/openclaw/issues/75648. Current main already supports `models.providers.<id>.timeoutSeconds` and progress-aware model idle timeouts; do not redesign those paths and do not implement durable prompt queueing. Add an additive way for provider failures to carry upstream HTTP status and Retry-After/retryAfterMs through `FailoverError`/`describeFailoverError`/`coerceToFailoverError` or the nearest existing metadata seam, then map known rate-limit/overload/auth/billing provider failures onto `/v1/chat/completions` and `/v1/responses`: non-streaming responses should use the appropriate HTTP status and `Retry-After` header when available; streaming responses should preserve the metadata in the SSE error/failure payload because headers have already been sent. Keep provider error messages sanitized; do not leak raw secrets or full upstream bodies. Add focused tests for `src/agents/failover-error.test.ts`, `src/agents/provider-transport-fetch.test.ts` if needed, `src/gateway/openai-http.test.ts`, and `src/gateway/openresponses-http.test.ts`. Add a user-facing CHANGELOG.md fix entry for the OpenAI-compatible Gateway behavior.
