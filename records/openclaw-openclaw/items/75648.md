## Summary

Keep open. Current main covers the configurable provider timeout and progress-aware idle watchdog portions, but provider 429/Retry-After metadata still does not reach the OpenAI-compatible HTTP surfaces because the gateway catch paths flatten embedded agent failures to generic internal errors.

## What This Changes

The issue requests configurable embedded-run provider timeouts and preservation of provider 429/Retry-After metadata for OpenAI-compatible clients, with progress-aware idle timeout reset as a stretch.

## Best Possible Solution

Keep the current timeout controls and add generic failover retry metadata propagation plus OpenAI-compatible HTTP/SSE error mapping for known provider status and retry timing.

## Reproduction Assessment

Yes for the remaining source-level gap: make `agentCommandFromIngress` reject with a `rate_limit` `FailoverError` carrying status 429, then inspect the current `/v1/chat/completions` and `/v1/responses` catch paths, which return generic internal errors. Full production throttling still needs provider logs or a stubbed provider to prove end to end.

## Solution Assessment

Yes for the narrowed repair: an additive metadata path through `FailoverError`/`describeFailoverError` and endpoint-specific OpenAI-compatible mapping is the maintainable fix. Changing timeout defaults or adding provider-specific gateway policy would be broader than needed.
