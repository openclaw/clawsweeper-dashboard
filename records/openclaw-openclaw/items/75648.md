## Summary

Keep open. Current main implements configurable provider request timeouts and progress-aware idle timeout behavior, but the remaining provider 429/Retry-After ask is still valid because the OpenAI-compatible HTTP endpoints flatten embedded agent failures into generic internal errors.

## What This Changes

Request is to make embedded-run provider timeout behavior tunable and preserve upstream provider 429/Retry-After metadata for OpenAI-compatible clients.

## Best Possible Solution

Keep the current timeout controls and add an additive provider-error metadata path that maps known embedded failover status/reason/Retry-After values onto `/v1/chat/completions` and `/v1/responses` error responses.

## Reproduction Assessment

Yes for the remaining source-level gap: mock `agentCommandFromIngress` to reject with a rate-limit `FailoverError` and verify the OpenAI-compatible endpoints currently return generic internal errors. The original provider-throttling scenario still needs provider logs or a stubbed provider for full end-to-end reproduction.

## Solution Assessment

Yes for the narrowed repair: preserving provider metadata at the agent-to-HTTP boundary is the smallest maintainable fix. The timeout override and streaming-idle parts are already covered on current main.

Fix the remaining provider-error metadata gap for https://github.com/openclaw/openclaw/issues/75648. Current main already supports `models.providers.<id>.timeoutSeconds` and progress-aware embedded idle timeout; do not redesign timeout defaults or fallback policy. Add an additive metadata path, likely starting with `FailoverError`/`describeFailoverError`, so provider status 429 and usable Retry-After timing can be preserved from embedded provider failures and mapped onto `/v1/chat/completions` and `/v1/responses` responses. Keep the change generic, provider-agnostic, and backwards-compatible. Add focused tests that mock `agentCommandFromIngress` or the nearest embedded command seam to reject with a rate-limit `FailoverError` carrying status and retry timing, then assert both non-streaming OpenAI-compatible endpoints return a structured rate-limit error with HTTP 429 and `Retry-After` when timing is available; cover streaming behavior only if the existing endpoint contract has a safe SSE error shape. Also keep existing generic internal-error behavior for unknown errors. Add a single-line `CHANGELOG.md` fix entry under the active version, crediting the human reporter if appropriate. Do not change bundled provider IDs, provider default timeout values, or unrelated gateway auth rate-limit behavior.
