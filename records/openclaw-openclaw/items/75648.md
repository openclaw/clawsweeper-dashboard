## Summary

Keep open. Current main already has configurable whole-run and provider-scoped timeout paths, and the model idle watchdog resets after stream chunks, but the requested structured upstream rate-limit signal is still not carried through the embedded-run/OpenAI-compatible gateway surface.

## What This Changes

The issue requests a configurable embedded-run upstream provider timeout plus structured propagation of provider rate-limit details such as HTTP 429 and Retry-After.

## Best Possible Solution

Keep the existing `agents.defaults.timeoutSeconds` and `models.providers.<id>.timeoutSeconds` knobs, then add a narrow structured error path that preserves upstream rate-limit status and sanitized retry timing when a provider response exists, including the OpenAI-compatible HTTP surfaces.

## Reproduction Assessment

Partially. There is a high-confidence source-level reproduction for the remaining structured-error gap: current main parses Retry-After internally but has no FailoverError/result/gateway field to return it; the exact 60s production timing was not live-reproduced.

## Solution Assessment

No, not as a new embedded-run-only timeout knob. The timeout side is already best handled by the existing provider-scoped and whole-run timeout config; the missing fix is additive propagation of rate-limit metadata plus docs/tests for the existing knobs.

## Repair Work Prompt

Fix the remaining structured rate-limit propagation gap for https://github.com/openclaw/openclaw/issues/75648. Current main already has `agents.defaults.timeoutSeconds`, `models.providers.<id>.timeoutSeconds`, and a chunk-resetting idle watchdog, so do not add a new duplicate embedded-run timeout knob unless source review proves a separate missing boundary. Instead, preserve sanitized upstream provider rate-limit metadata when a provider HTTP response exists: at minimum status 429 and retry-after timing from `retry-after` / `retry-after-ms` should be available to embedded-run callers and the OpenAI-compatible `/v1/chat/completions` surface. Avoid leaking arbitrary provider headers or raw secret-bearing payloads. Likely implementation areas are provider transport response/error capture, `FailoverError` or equivalent embedded-run metadata, embedded run payload/meta propagation, and `src/gateway/openai-http.ts`/possibly `src/gateway/openresponses-http.ts` HTTP error mapping. Add targeted tests for 429 with Retry-After, 429 without Retry-After, timeout with no upstream response, and the existing provider timeout/idle reset behavior. Add a user-facing changelog entry if the HTTP/API behavior changes.
