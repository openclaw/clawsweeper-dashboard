## Summary

Keep open. Current main already addresses the configurable timeout and streaming idle-timeout parts, but the provider error metadata ask remains valid because OpenAI-compatible HTTP handlers still turn embedded-run failures into generic internal errors instead of preserving provider 429/Retry-After details.

## What This Changes

Make embedded-run upstream provider timeouts configurable and preserve provider rate-limit metadata such as HTTP 429 and Retry-After through OpenAI-compatible gateway errors.

## Best Possible Solution

Keep the existing `models.providers.*.timeoutSeconds` and idle-stream watchdog behavior, then add an additive provider-error metadata path that preserves known status, reason, and Retry-After values through failover normalization and maps them onto OpenAI-compatible HTTP status, body, and headers where protocol timing permits.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused gateway handler test can make the embedded agent path reject with a provider rate-limit failure and show that `/v1/chat/completions` and `/v1/responses` currently return generic internal-error responses. The live upstream throttling sequence itself is not high-confidence without provider logs or a deterministic upstream stub.

## Solution Assessment

Yes, the narrowest maintainable fix is to preserve provider error metadata at the agent-to-HTTP boundary while leaving existing timeout configuration, streaming idle behavior, and separate queueing work unchanged.
