## Summary

Keep open. Current main already makes the timeout side configurable and progress-aware, but the provider error metadata request remains valid because OpenAI-compatible HTTP endpoints still collapse embedded-run provider failures into generic internal errors without preserving provider status or Retry-After.

## What This Changes

Request configurable embedded-run provider timeouts and structured propagation of provider 429/Retry-After metadata for long or throttled upstream calls.

## Best Possible Solution

Keep the existing provider timeout and streaming idle-timeout behavior, then add an additive provider-error metadata path that maps known provider status/reason/Retry-After values onto `/v1/chat/completions` and `/v1/responses` HTTP status, body, and `Retry-After` header.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused Gateway test can make the embedded agent path throw a rate-limit `FailoverError` and show the OpenAI-compatible endpoints currently return generic internal-error responses. The 47-call production timing sequence remains unverified without live provider logs or an upstream stub.

## Solution Assessment

Yes for the narrowed repair: preserving provider metadata at the agent-to-HTTP boundary is the smallest maintainable fix. Raising timeouts and resetting streaming idle timers are already covered, while durable queueing belongs in #49251.

## Repair Work Prompt

Fix the remaining provider error metadata gap for https://github.com/openclaw/openclaw/issues/75648. Preserve the existing `models.providers.*.timeoutSeconds` and streaming idle-timeout semantics; do not implement queueing, delayed retry scheduling, or broad session-unblocking behavior from #49251/#75665. Add an additive metadata path that can carry provider status/reason and Retry-After values through embedded-run failover errors without exposing raw provider secrets or unbounded error bodies, then map that metadata onto `/v1/chat/completions` and `/v1/responses` HTTP status, response body, and `Retry-After` header for both non-streaming and streaming failure paths where the protocol allows it. Update focused tests for provider transport/failover metadata and the OpenAI-compatible HTTP endpoints, update user-facing docs if the HTTP error contract changes, and add a single-line changelog entry because this is a user-facing fix.
