## Summary

Keep open. Current main covers the configurable provider timeout and progress-aware idle watchdog parts, but the OpenAI-compatible gateway handlers still flatten provider failover/status metadata to generic internal errors, so the 429/Retry-After propagation ask remains.

## What This Changes

Request configurable embedded-run upstream timeouts and structured provider rate-limit/Retry-After propagation for OpenAI-compatible gateway responses.

## Best Possible Solution

Keep the existing timeout controls and add additive failover retry metadata plus OpenAI-compatible endpoint mapping that returns provider status and Retry-After when available.

## Reproduction Assessment

Yes for the remaining source-level gap: stub `agentCommandFromIngress` to reject with a `rate_limit` `FailoverError` carrying status 429 and the current catch paths still return generic internal errors. The original production throttling sequence would still need provider logs or a stubbed provider for full end-to-end proof.

## Solution Assessment

Yes for the narrowed repair: extend generic failover metadata and map known provider status/retry timing at `/v1/chat/completions` and `/v1/responses`. Changing timeout defaults or adding provider-specific gateway policy would be broader than needed.
