## Summary

Keep open. Current main covers the configurable provider timeout and progress-aware idle watchdog pieces, but provider rate-limit metadata is still flattened before OpenAI-compatible HTTP clients can see it.

## What This Changes

The issue asks OpenClaw to make embedded-run provider timeouts tunable and preserve upstream 429/Retry-After details for OpenAI-compatible HTTP clients.

## Best Possible Solution

Keep `models.providers.<id>.timeoutSeconds` as the timeout knob, then add additive failover retry metadata and map known provider status/retry timing onto `/v1/chat/completions` and `/v1/responses`.

## Reproduction Assessment

Yes for the remaining source-level gap: mock the embedded agent command path to reject with a rate-limit `FailoverError` carrying status 429 and retry timing, then verify both OpenAI-compatible endpoints currently return generic internal-error responses. The reported production throttling pattern still needs live provider logs or a stub provider to reproduce end to end.

## Solution Assessment

Yes for the narrowed repair. The smallest maintainable fix is to preserve provider status/retry timing at the agent-to-HTTP boundary; adding another timeout config key would duplicate the timeout surface already present on model providers.
