## Summary

Keep open. Current main addresses the configurable timeout and progress-aware watchdog parts, but the OpenAI-compatible HTTP surfaces still collapse embedded-run provider failures into generic internal errors and there is no Retry-After metadata path on `FailoverError`.

## What This Changes

The issue asks OpenClaw to make embedded-run provider timeouts configurable, preserve upstream 429/Retry-After metadata, and reset streaming idle timeouts on progress.

## Best Possible Solution

Keep the current timeout controls, then add an additive provider-error metadata path that maps known status/code/Retry-After values onto OpenAI-compatible chat and Responses error outputs.

## Reproduction Assessment

Yes for the remaining source-level gap: a focused Gateway test can make `agentCommandFromIngress` reject with a rate-limit `FailoverError` and show `/v1/chat/completions` and `/v1/responses` currently return generic internal errors without Retry-After. The production 47-call provider throttling sequence still needs live logs or a provider stub to reproduce end to end.

## Solution Assessment

Yes for a narrowed fix: preserving provider metadata at the embedded-run-to-HTTP boundary is smaller and more maintainable than changing timeout policy again. Durable queueing/backpressure is related but belongs in the separate queueing tracker.

## Repair Work Prompt

Fix the remaining provider error metadata gap for https://github.com/openclaw/openclaw/issues/75648. Current main already supports `models.providers.<id>.timeoutSeconds` for provider HTTP request timeouts and the embedded-run idle watchdog, and `streamWithIdleTimeout` is progress-aware; do not redesign those parts. Add an additive metadata path, likely on `FailoverError` / `describeFailoverError` / `coerceToFailoverError`, that can carry upstream HTTP status, provider code/reason, and Retry-After or retry-after-ms when available. Map that metadata in `/v1/chat/completions` and `/v1/responses` non-stream error responses so provider rate limits return a structured 429 and `Retry-After` header instead of generic 500 `api_error`. For streaming OpenAI-compatible responses, preserve the existing SSE contract but include the structured provider error in the SSE failure payload where headers/status cannot be changed after flush. Add focused tests that reject the embedded command path with a rate-limit `FailoverError` carrying retry metadata and assert the OpenAI-compatible endpoints preserve it. Keep #49251-style durable queueing out of scope. Add a user-facing changelog entry if runtime behavior changes.
