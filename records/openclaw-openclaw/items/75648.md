## Summary

Keep open. Current main covers the configurable provider timeout and progress-aware idle watchdog portions, but provider 429/Retry-After metadata still does not reach `/v1/chat/completions` or `/v1/responses` because those handlers flatten embedded agent failures to generic internal errors.

## What This Changes

The issue requests configurable embedded-run upstream timeouts and preservation of provider 429/Retry-After metadata for OpenAI-compatible clients.

## Best Possible Solution

Keep the current timeout controls and add generic failover metadata propagation plus OpenAI-compatible HTTP error mapping for known provider status and retry timing.

## Reproduction Assessment

Yes for the remaining source-level gap: inject `agentCommandFromIngress` with a `rate_limit` `FailoverError` carrying status 429 and inspect the current catch paths, which return generic internal errors. The original production throttling sequence still needs provider logs or a stubbed provider for full end-to-end proof.

## Solution Assessment

Yes for the narrowed repair: an additive metadata path through `FailoverError`/`describeFailoverError` and endpoint-specific mapping is the maintainable fix. Changing timeout defaults or adding provider-specific gateway policy would be broader than needed.

Current main already supports `models.providers.<id>.timeoutSeconds` and progress-aware embedded idle timeout; do not redesign timeout defaults, provider defaults, or fallback policy.
