## Summary

Keep open: current main has removed the hard 60s-only timeout shape and added provider-level timeout plus Retry-After handling, but the requested downstream structured Retry-After/error contract is not fully implemented or settled.

## What This Changes

The issue requests configurable embedded-run upstream timeouts and structured propagation of provider 429/Retry-After details instead of collapsing them into generic timeouts.

## Best Possible Solution

Keep `models.providers.<id>.timeoutSeconds` as the timeout override, document it clearly for embedded-run operators, and have maintainers define one structured provider error/Retry-After contract before code changes.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection shows the old hard 60s timeout ask is now addressed by provider timeout config and streaming resets, while structured Retry-After propagation remains missing from `FailoverError` and timed-out surface errors can avoid the structured error path.

## Solution Assessment

No, the original new gateway/env timeout knob is not the best solution because current main already has a supported provider timeout surface; the remaining best fix is a maintainer-approved structured provider error contract.
