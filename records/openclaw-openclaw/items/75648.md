## Summary

Keep open. Current main already handles the timeout configurability and progress-aware idle watchdog parts, but the provider 429/Retry-After metadata is still lost at the OpenAI-compatible Gateway HTTP boundary.

## What This Changes

The issue requests a configurable embedded-run upstream timeout, structured surfacing of provider 429/Retry-After metadata, and progress-aware timeout handling for slow streams.

## Best Possible Solution

Keep the current timeout controls and add an additive provider-error metadata path that maps known provider status/reason/Retry-After values onto `/v1/chat/completions` and `/v1/responses` responses without exposing raw provider bodies or secrets.

## Reproduction Assessment

Yes for the remaining source-level gap: mock the embedded agent path to reject with a rate-limit `FailoverError` and verify the OpenAI-compatible endpoints currently return generic internal-error responses without `Retry-After`. The original production throttling sequence still needs live provider logs or a stub to replay exactly.

## Solution Assessment

Yes for a narrowed repair: preserving provider metadata at the agent-to-HTTP boundary is the smallest maintainable fix. Timeout configurability and streaming idle reset already exist, while durable queueing belongs in #49251.

## Repair Work Prompt

Fix the remaining provider metadata gap for openclaw/openclaw#75648. Current main already supports `models.providers.*.timeoutSeconds` and progress-aware model idle timeouts; do not redesign those timeout paths or queueing. Add an additive way to preserve provider HTTP status/reason and Retry-After metadata through failover errors where it is available, then map rate-limit/overload/auth/billing provider failures onto `/v1/chat/completions` and `/v1/responses` responses. Non-streaming HTTP should use the appropriate status and set `Retry-After` when known; streaming responses should surface structured error metadata in the compatible stream shape as far as headers already sent allow. Avoid leaking raw provider bodies, request headers, credentials, or prompt content. Add focused tests for `FailoverError` metadata, provider transport retry-after extraction where needed, OpenAI chat completions, and OpenResponses non-stream/stream error mapping. Add a user-facing changelog entry; docs only if the public error contract changes enough to document.
