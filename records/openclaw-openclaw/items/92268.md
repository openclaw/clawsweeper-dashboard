## Summary

The issue remains necessary: OpenClaw v2026.6.5 and current main unconditionally deny outbound network for native Codex workspace-write turns, while the Codex app-server protocol independently supports workspace-write network access. The linked earlier report was closed without a merged fix, and the prior proposed PR was closed unmerged, so neither resolves this current-release reproduction.

## What This Changes

Requests a supported, documented opt-in for outbound network access in native Codex app-server workspace-write mode without requiring danger-full-access.

## Best Possible Solution

Add one disabled-by-default Codex plugin option that maps directly to Codex's supported workspaceWrite.networkAccess field for native app-server turns, with a clearly documented ownership boundary, schema/help metadata, and focused tests proving both default denial and explicit opt-in behavior; avoid requiring danger-full-access or reading raw Codex config as a competing source of truth.

## Reproduction Assessment

Yes, at high confidence from source and live evidence: current main always sends networkAccess false for native workspace-write turns, and the reporter reproduced both DNS and raw TCP denial on v2026.6.5. I did not independently execute the live probe, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes in overall direction: using Codex's existing independent workspace-write network field is the narrowest supported alternative to danger-full-access. The final config name, scope, and behavior under active OpenClaw sandboxing still require explicit maintainer product/security judgment.
