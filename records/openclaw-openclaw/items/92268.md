## Label Justifications

- `impact:security`: The issue concerns the outbound-network authority granted to sandboxed native command execution.

## Summary

This remains necessary: current main and v2026.6.6 explicitly send a workspace-write turn policy with outbound network disabled, while upstream Codex supports workspace-write with network access. No supported OpenClaw configuration exposes that upstream capability, but the opt-in changes a deliberate security boundary and needs maintainer product approval.

## What This Changes

Add a supported, documented opt-in for outbound network access during Codex app-server workspace-write turns without granting danger-full-access.

## Best Possible Solution

Add one Codex-plugin-owned, false-by-default per-agent egress setting that maps to Codex's per-turn workspace-write networkAccess field, with containment, precedence, fresh-install, upgrade, and native network tests plus operator documentation.

## Reproduction Assessment

Yes at source level: OpenClaw sends an explicit workspace-write turn policy with networkAccess false, upstream Codex enforces that field, and the reporter's DNS and raw-TCP failures match this path. A fresh live current-main execution was not performed during this read-only review.

## Solution Assessment

Yes directionally, but the maintainable solution is a single supported OpenClaw Codex-plugin policy surface rather than relying on raw Codex config that the app-server turn request overrides.
