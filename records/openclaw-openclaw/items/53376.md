## Summary

Keep open. Current main still defaults every unrecognized non-empty Feishu target to `user_id` and forwards that derived type into outbound send paths, while the linked implementation PR remains open with closing syntax for this issue. The direct Feishu send call does pass `receive_id_type`, so the remaining work is target/type validation and live confirmation rather than closing as already implemented.

## What This Changes

The issue asks OpenClaw to fix Feishu outbound message sends that fail with a 400 `receive_id_type is required` response in WebSocket mode.

## Best Possible Solution

Review and either land or replace PR #53787 as the canonical fix. The desired end state is a narrow Feishu plugin change that validates receive-id type derivation before direct send, media send, reply fallback, and streaming startup paths call Feishu, while preserving documented and intentionally supported target formats.

## Reproduction Assessment

Unclear. The issue provides concrete user steps, and static inspection shows a focused path where unrecognized non-empty targets still resolve to `user_id`, but the exact Feishu 400 requires a live Feishu app and target value to reproduce with high confidence.

## Solution Assessment

Yes, keeping this open is the best current path because current main has not landed a guard and an open PR already references this issue with closing syntax. The narrow maintainable fix should target Feishu send-target validation rather than moving `receive_id_type` out of the SDK `params` object, which the SDK contract already supports.
