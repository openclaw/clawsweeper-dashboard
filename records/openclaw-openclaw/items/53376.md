## Summary

Keep open. Current main still falls through to `user_id` for unrecognized Feishu targets, and the open unmerged PR #53787 already references this issue as the implementation path.

## What This Changes

The issue asks for Feishu WebSocket message sends to stop failing with Feishu/Lark 400 `receive_id_type is required` errors.

## Best Possible Solution

Land or replace #53787 with an equivalent narrow Feishu plugin patch that validates target/type derivation before direct sends, media sends, reply fallbacks, and streaming startup while preserving documented `oc_` chat IDs, `ou_` open IDs, and explicitly supported bare user IDs.

No full live reproduction was established. The issue gives concrete setup and an exact Feishu error, and source inspection shows current main still has the permissive receive-id type fallback, but the vendor request was not replayed against a configured Feishu app.

## Solution Assessment
