## Summary

Keep open. Current main still resolves every unrecognized non-empty Feishu target to `user_id`, and the open PR #53787 already references this issue with closing syntax as the implementation path. The send call itself does include `receive_id_type`, so this should remain a targeted Feishu target/type validation fix rather than be closed as implemented.

## What This Changes

Fix Feishu outbound message sending so the plugin sends a valid `receive_id_type` and does not hit Feishu 400 validation errors in WebSocket-mode message delivery.

## Best Possible Solution

Review and either land #53787 or replace it with an equivalent narrow Feishu plugin patch that validates target/type derivation before direct send, media send, reply fallback, and streaming startup paths reach the Feishu API while preserving documented `oc_` chat IDs, `ou_` open IDs, and explicitly supported bare user IDs.
