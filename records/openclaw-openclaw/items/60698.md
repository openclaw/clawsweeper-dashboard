## Summary

Keep open. The Bedrock cost-attribution feature is useful and current main still lacks it, but this PR is not mergeable as written because it injects raw unconstrained metadata, reverses the current provider-wrapper precedence contract, mixes in unrelated Feishu media changes, and lacks live Bedrock/CloudTrail proof.

## What This Changes

The PR adds automatic Bedrock `requestMetadata` with per-agent id/name in the embedded runner and also adds Feishu outbound media MIME-sniffing fallback tests/code.

## Best Possible Solution

Land a narrow Bedrock-owned implementation that injects validated agent metadata without changing wrapper precedence, adds focused coverage/docs, and leaves arbitrary configured metadata to https://github.com/openclaw/openclaw/issues/57360.

## Reproduction Assessment

Yes. Source inspection shows current main lacks any `requestMetadata` path, and the PR defects are source-reproducible from raw unconstrained agent strings plus the AWS metadata contract and the changed wrapper order.

## Solution Assessment

No. The direction is useful, but this branch should not merge until metadata is validated or normalized, wrapper precedence is preserved, unrelated Feishu changes are removed, and live Bedrock proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Sanitize Bedrock metadata before injection:** `src/agents/pi-embedded-runner/extra-params.ts:282-284`
  - body: `agentId` and `agentName` come from config fields that are plain strings, but Bedrock only accepts a limited key/value pattern and length for `requestMetadata`. A valid OpenClaw agent name such as `Support 🤖` or `O'Reilly` would make every Bedrock stream fail with an AWS validation error, so normalize, reject, or omit invalid metadata before forwarding it.
  - confidence: 0.92
- **[P2] Preserve provider wrapper precedence:** `src/agents/pi-embedded-runner/extra-params.ts:510`
  - body: This moves the extra-param wrapper inside provider wrapping, reversing the current contract that explicit runtime params are applied outside provider defaults. Since the extra-param wrapper spreads `...options` after `streamParams`, provider wrapper defaults can override explicit config values instead of the other way around.
  - confidence: 0.86
