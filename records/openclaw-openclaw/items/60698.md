## Summary

Keep open: the Bedrock direction is useful and not implemented on current main, but this PR is not mergeable as-is because raw metadata can be rejected by AWS, the wrapper ordering regresses explicit runtime params, real Bedrock proof is missing, and unrelated Feishu media changes should be split.

## What This Changes

The PR adds automatic Bedrock requestMetadata with agent id/name in the embedded runner and also adds unrelated Feishu outbound media MIME-sniffing fallback code and tests.

## Best Possible Solution

Land a narrow Bedrock-owned implementation that validates or omits invalid metadata, preserves existing wrapper precedence, includes focused tests/docs, and leaves arbitrary configured metadata to https://github.com/openclaw/openclaw/issues/57360.

## Reproduction Assessment

Yes. Source inspection shows current main lacks a requestMetadata path, and the PR defects are source-reproducible from unconstrained agent strings plus the AWS requestMetadata contract and the changed wrapper order.

## Solution Assessment

No. The requested direction is useful, but this branch should not merge until metadata is validated or normalized, wrapper precedence is preserved, unrelated Feishu changes are split, and live Bedrock proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Sanitize Bedrock metadata before injection:** `src/agents/pi-embedded-runner/extra-params.ts:280-281`
  - body: agentId and agentName come from plain config strings, but Bedrock requestMetadata only accepts a limited key/value pattern and length. A valid OpenClaw agent name such as `Support 🤖` or `O'Reilly` would make Bedrock streams fail with validation errors, so normalize, reject, or omit invalid metadata before forwarding it.
  - confidence: 0.92
- **[P2] Preserve provider wrapper precedence:** `src/agents/pi-embedded-runner/extra-params.ts:508`
  - body: Moving the extra-param wrapper before provider wrapping reverses the current contract that explicit runtime params are applied outside provider defaults. Because the extra-param wrapper spreads caller options after streamParams, provider-added defaults can now override explicit config values instead of the other way around.
  - confidence: 0.86
