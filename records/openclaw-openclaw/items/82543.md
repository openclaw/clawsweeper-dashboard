## Label Justifications

- `P1`: A real user reports a bundled channel workflow hanging indefinitely after re-login, leaving Zalo Personal unable to start cleanly or fail visibly.
- `impact:message-loss`: A stuck Zalo Personal listener can prevent inbound and outbound message handling for that account.
- `impact:crash-loop`: The issue is about a channel startup hang with no timeout or terminal error path.

## Summary

Keep open: current main and v2026.6.8 still show a plausible zalouser startup-readiness gap after restored credentials. Session restore itself is timeout-bounded, but listener startup still calls `zca-js` without awaiting `connected` or timing out a listener that never reaches `connected`, `error`, or `closed`.

## What This Changes

The issue reports that the Zalo Personal plugin hangs indefinitely during gateway startup after logout and QR re-login instead of connecting or failing with a bounded error.

## Best Possible Solution

Add a plugin-local bounded readiness wait around the zca-js listener `connected`/`error`/`closed` events, then keep the account task long-running only after listener startup is known to be ready.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a medium-confidence path: saved-session restore is bounded, but listener startup returns after `start()` without a `connected` readiness wait and the monitor can wait indefinitely if no terminal event arrives.

## Solution Assessment

Unclear until live Zalo proof confirms the endpoint behavior, but the narrowest maintainable fix is in the zalouser listener startup path. A generic gateway `start-account` timeout would be broader and could break passive channels that are correctly long-running after connection.
