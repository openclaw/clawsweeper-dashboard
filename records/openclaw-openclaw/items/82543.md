## Label Justifications

- `impact:crash-loop`: The reported failure is a channel startup hang with liveness warnings rather than a clean startup failure.
- `impact:message-loss`: A stuck Zalo Personal provider cannot receive or deliver channel messages for the affected account.

## Summary

Keep open: current main and the latest release still have a plausible Zalo Personal startup-readiness gap after saved-session restore. The code starts the `zca-js` listener and handles later `error` or `closed` events, but it does not wait for or time out an initial `connected` event, so a listener that never opens or fails can leave the account task pending without a useful error.

## What This Changes

The issue reports that the Zalo Personal plugin can hang indefinitely during provider startup after logout, QR re-login, and gateway restart.

## Best Possible Solution

Add plugin-local Zalo Personal listener readiness handling around `zca-js` startup, using `connected`/`error`/`closed` plus a startup timeout, while preserving the long-running passive monitor after a successful connection signal.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection gives a medium-confidence path: after saved-session restore, a mocked `api.listener.start()` that emits no `connected`, `error`, or `closed` event leaves `monitorZalouserProvider` pending without surfacing a failure.

## Solution Assessment

Unclear until live Zalo proof confirms the endpoint behavior, but the narrowest maintainable repair is plugin-local listener readiness/error handling. Changing generic gateway `startAccount` semantics would be broader than the reported Zalo Personal failure.
