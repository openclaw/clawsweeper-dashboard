## Summary

Keep open: this is a fresh, detailed report against the bundled Zalo Personal plugin, and current main still has a plausible plugin-local startup-readiness gap after credential restore. The related cookie persistence fix helps the same re-login surface, but it does not add a bounded listener connected/error outcome for a `zca-js` listener that never reaches a terminal event.

## What This Changes

The issue asks for the Zalo Personal provider to either complete startup or fail with a timeout/error after logout, QR re-login, and gateway restart.

## Best Possible Solution

Add plugin-local listener readiness/error handling around `zca-js` startup, using the dependency's `connected`, `error`, and `closed` events with a bounded startup timeout while preserving the passive long-running account lifecycle after connection.

## Reproduction Assessment

No live high-confidence reproduction is available from this read-only review. Source inspection gives a credible mocked path: after saved-session restore, make `api.listener.start()` produce no `connected`, `error`, or `closed` event and `monitorZalouserProvider` remains pending.

## Solution Assessment

Yes, the likely best fix boundary is plugin-local: wait for or time out Zalo listener readiness using the dependency's existing events. A gateway-wide `startAccount` liveness change would be broader than the reported plugin failure.
