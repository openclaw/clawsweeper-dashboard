## Label Justifications

- `P1`: A real user reports a bundled channel workflow hanging indefinitely after re-login, leaving Zalo Personal unable to start cleanly or fail visibly.
- `impact:crash-loop`: The issue is about a channel startup hang with no timeout or terminal error path.
- `impact:message-loss`: A stuck Zalo Personal listener means inbound and outbound channel message handling for that account cannot reliably run.

## Summary

Keep open: current main and the latest release still have a plausible Zalo Personal startup-readiness gap after restored login credentials. The plugin bounds saved-session restore, but listener startup still returns after calling `zca-js` without waiting for a `connected` signal or timing out a listener that never reaches `connected`, `error`, or `closed`.

## What This Changes

Report requests Zalo Personal startup to connect or fail with a timeout/error after QR re-login instead of hanging in the `start-account` phase.

## Best Possible Solution

Add a plugin-local bounded readiness wait around `zca-js` listener `connected`/`error`/`closed` events, then keep the passive account task long-running after connection.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: saved-session restore succeeds, `api.listener.start()` neither emits `connected`, `error`, nor `closed`, and `monitorZalouserProvider` waits indefinitely.

## Solution Assessment

Unclear until live Zalo proof confirms the endpoint behavior, but the narrowest maintainable fix is in the `zalouser` plugin listener startup path. A generic gateway `start-account` timeout would be broader and could break passive channels that are correctly long-running after connection.
