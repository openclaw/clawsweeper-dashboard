## Summary

Keep open: this is a fresh, detailed report against the bundled Zalo Personal plugin, and current main still has a source-visible startup-readiness gap after credentials restore. The code can time out session restore, but once `zca-js` listener startup is invoked it only reacts to `error` or `closed`; a listener that never reaches connected/error/closed can leave the account task pending with no clearer failure.

## What This Changes

The issue reports `@openclaw/zalouser` hanging indefinitely in the `channels.zalouser.start-account` phase after logout, QR re-login, and gateway restart.

## Best Possible Solution

Add plugin-local Zalo Personal listener readiness/error handling around `zca-js` startup, preserving the long-running passive account lifecycle after a successful connection signal.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction. Source inspection gives a medium-confidence path: after saved-session restore, if the `zca-js` listener never emits `connected`, `error`, or `closed`, `monitorZalouserProvider` can remain pending without surfacing a failure.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-local listener readiness and timeout handling using the dependency's listener events. Changing generic gateway liveness would be broader than the reported Zalo Personal startup failure.
