## Summary

Keep open: this is a fresh, detailed report against the bundled Zalo Personal plugin, and current main still has a plausible plugin-local startup-readiness gap after restoring credentials. The existing code times out session restore, but once `zca-js` listener startup is invoked it only reacts to later `error` or `closed` events; there is no bounded readiness check for a listener that never connects or emits a terminal event.

## What This Changes

The issue asks for `@openclaw/zalouser` to stop hanging silently after logout, QR re-login, and gateway restart, and instead either connect or report a timeout/error.

## Best Possible Solution

Add plugin-local Zalo Personal listener readiness/error handling around `zca-js` startup, preserving the long-running passive account lifecycle after a successful connection signal.

## Reproduction Assessment

Yes, source inspection gives a medium-confidence reproduction path: after saved-session restore, mock `api.listener.start()` so no `connected`, `error`, or `closed` event fires, and `monitorZalouserProvider` will stay pending without surfacing a failure. I did not reproduce against a live Zalo account.

## Solution Assessment

Unclear until a live Zalo repro confirms the endpoint behavior, but the narrow maintainable fix is likely in the `zalouser` plugin: wait for or time out listener readiness using the dependency's `connected`/`error`/`closed` events. Changing generic gateway `startAccount` liveness would be broader than the reported plugin failure.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/82543 in the `@openclaw/zalouser` plugin. The reported behavior is: after `openclaw channels logout --channel zalouser`, QR re-login, and gateway restart, logs stop after `[default] starting zalouser provider (...)` and diagnostics remain at `channels.zalouser.start-account` instead of either connecting or reporting a timeout/error. Current code restores the saved session with a timeout, but `startZaloListener` calls `api.listener.start({ retryOnClose: false })` and returns without waiting for the upstream `zca-js` `connected` event or a bounded startup failure. Implement a plugin-local readiness guard around listener startup using the `zca-js` 2.1.2 listener events (`connected`, `error`, `closed`), so a listener that never reaches connected/error/closed fails with a clear error and lets the existing channel supervisor handle it. Preserve passive long-running channel behavior after readiness succeeds; do not change generic gateway `startAccount` lifecycle semantics, do not add a config option unless maintainers ask for one, and do not use real Zalo credentials in tests. Add focused mocked regression tests for a listener that never emits readiness and for the successful connected path staying alive until abort. Include a user-visible changelog entry if this becomes a landing PR.
