## Summary

Keep open: current main has fixed the generic startup-trace handoff that could pin diagnostics to a long-lived `start-account` phase, but the Zalo Personal plugin still has no bounded listener-readiness outcome after `zca-js` listener startup. A restored session can still remain pending if the listener never emits `connected`, `error`, or `closed`.

## What This Changes

The issue reports that the bundled Zalo Personal plugin can hang after logout, QR re-login, and gateway restart instead of connecting or failing with an error.

## Best Possible Solution

Add plugin-local Zalo Personal listener readiness/error handling around `zca-js` startup, preserving the long-running passive account lifecycle after a successful `connected` signal.

## Reproduction Assessment

Yes for a source-level path: after saved-session restore, a listener that starts but never emits `connected`, `error`, or `closed` leaves `monitorZalouserProvider` pending until abort. I did not establish a live Zalo-account reproduction.

## Solution Assessment

Yes, the best narrow fix is in the `zalouser` plugin: wait for or time out listener readiness using the dependency's `connected`/`error`/`closed` events. Changing generic gateway liveness would not prove the Zalo transport became usable.
