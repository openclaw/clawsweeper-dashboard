## Summary

Keep open: this is a fresh, detailed Zalo Personal startup report and current main still has a plausible plugin-local listener readiness gap after restored-session login. The report has no linked open fix PR, no duplicate found by live GitHub keyword searches, and the existing code still starts `zca-js` without waiting for `connected` or enforcing a startup timeout.

## What This Changes

The issue requests that Zalo Personal startup after logout and QR re-login either connect successfully or fail with a bounded timeout/error instead of hanging silently.

## Best Possible Solution

Add a plugin-local listener readiness wait/timeout around `zca-js` `connected`/`error`/`closed` events, preserving the long-running passive account task after a successful connection signal.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. There is a medium-confidence source reproduction path: after saved-session restore, use a mocked `zca-js` listener whose `start()` never emits `connected`, `error`, or `closed`, and the Zalo monitor remains pending without surfacing a startup failure.

## Solution Assessment

Yes, the narrowest maintainable direction appears to be a Zalo Personal plugin fix that treats listener startup as ready only after `connected` or fails on timeout/error/closed. A generic gateway `startAccount` timeout would conflict with passive channel monitors that are intentionally long-lived.
