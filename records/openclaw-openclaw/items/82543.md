## Summary

Keep open: current main still has a plausible Zalo Personal startup-readiness gap after re-login, and no duplicate or merged follow-up appears to own the remaining listener hang. The code restores saved credentials with a timeout, but listener startup can return without any bounded wait for `connected`, `error`, or `closed`, leaving the account task pending with the same last log reported by the user.

## What This Changes

Fix Zalo Personal startup after logout and QR re-login so gateway restart either connects or surfaces a timeout/error instead of hanging at `channels.zalouser.start-account`.

## Best Possible Solution

Add plugin-local Zalo listener readiness/error handling around `zca-js` startup, using `connected`/`error`/`closed` events plus a bounded startup timeout while preserving the long-running account task after readiness.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: restore a saved Zalo session, mock `api.listener.start()` so it never emits `connected`, `error`, or `closed`, and `monitorZalouserProvider` remains pending without surfacing a failure.

## Solution Assessment

Unclear until a live Zalo repro confirms the endpoint behavior, but the best maintainable fix is likely narrow and plugin-local: wait for or time out listener readiness using the dependency events rather than changing generic gateway liveness semantics.
