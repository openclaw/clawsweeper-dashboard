## Summary

Keep open: current main and v2026.6.10 still leave a plausible Zalouser startup-readiness gap after restored credentials; the listener startup path does not await or time out zca-js connection readiness before the account task becomes a long-lived wait.

## What This Changes

The issue reports that the Zalouser plugin can hang indefinitely while starting an account after logout and QR re-login, instead of connecting or failing with a timeout.

## Best Possible Solution

Zalouser should fail or mark startup unhealthy within a bounded plugin-local readiness window when zca-js never reaches connected, while preserving long-lived listener tasks after successful connection.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a medium-confidence path: current main still starts the zca-js listener without awaiting connected readiness, and the monitor can remain pending if no error or close event arrives.

## Solution Assessment

No, the current implementation is not the best end state because it distinguishes error/closed but not never-connected startup. The narrow maintainable fix is a Zalouser-owned readiness timeout around zca-js listener startup, not a generic gateway start-account timeout.
