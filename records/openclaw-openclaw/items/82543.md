## Summary

Keep open: this is a fresh, detailed report against the bundled Zalo Personal plugin, and current main still has a source-visible startup readiness gap after restoring credentials. The plugin has a login restore timeout, but listener startup can remain pending if `zca-js` never emits `connected`, `error`, or `closed`.

## What This Changes

The issue reports that `@openclaw/zalouser` hangs indefinitely during `channels.zalouser.start-account` after logout, QR re-login, and gateway restart.

## Best Possible Solution

Add plugin-local Zalo Personal listener readiness handling around `zca-js` startup, with a bounded timeout or terminal error path before treating the account as successfully started.

## Reproduction Assessment

Yes, by source inspection with medium confidence: restore a saved Zalouser session, have the mocked listener start without emitting `connected`, `error`, or `closed`, and `monitorZalouserProvider` remains pending. I did not reproduce it against a live Zalo account.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-local listener readiness/error handling using the dependency's existing events. A generic gateway `startAccount` timeout would be broader and could break intentionally long-running passive channel tasks.
