## Summary

Keep open: current main still has a plausible Zalo Personal startup-readiness gap after restored-session login. The related cookie-persistence fix helps stale credentials, but the listener startup path still does not wait for a `connected` event or enforce a bounded startup timeout when `zca-js` never reaches `connected`, `error`, or `closed`.

## What This Changes

The issue reports that the bundled Zalo Personal plugin hangs at provider startup after logout, QR re-login, and gateway restart.

## Best Possible Solution

Add plugin-local listener startup readiness handling around `zca-js`, failing with a clear timeout/error if the listener never reaches `connected` while preserving the long-running account lifecycle after a successful connection.

## Reproduction Assessment

No high-confidence live reproduction is available from this read-only review. Source inspection gives a medium-confidence mocked path: restore saved credentials, have `api.listener.start()` never emit `connected`, `error`, or `closed`, and the current monitor remains pending.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-local listener readiness/error handling using the dependency's `connected`, `error`, and `closed` events. A generic gateway `startAccount` liveness change would be broader than the reported Zalo Personal listener failure.
