## Summary

Keep open: the owned steering-write change addresses the reported fence race and the supplied proof is strong, but the branch is not merge-ready because it removes default retry resilience from existing embedded OpenAI requests and gives users a connection-retry diagnosis that takeover detection does not establish.

## What This Changes

The PR wraps embedded steering writes in the owned transcript fence, adds takeover phase diagnostics and resend handling, changes embedded OpenAI retry defaults, forwards request-level retry options through native transports, and adds focused plus real-setup regression coverage.

## Best Possible Solution

Retain the owned steering-write fence, takeover phase diagnostics, and explicit/configured retry forwarding; leave `maxRetries` unset for unconfigured requests so the existing SDK default remains, and use neutral resend guidance based only on the verified session takeover.

No in its current form. Moving steering under the owned transcript-write fence is the right root boundary, but the implicit zero-retry policy is a broader availability change and the reply text asserts a cause the error does not prove.
