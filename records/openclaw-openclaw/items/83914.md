## Label Justifications

- `P2`: A single malformed or dependency-failing diff card can break remaining viewer cards, but the blast radius is limited to the optional diffs plugin viewer surface.

## Summary

Keep open: current main and v2026.5.18 still hydrate all diff cards through one uncaught per-card loop, so a thrown `FileDiff` construction or hydration error can abort the remaining cards and trip the page-level error path.

## What This Changes

The issue requests per-card failure isolation for the diffs viewer hydration loop so one bad card does not prevent later cards from hydrating.

## Best Possible Solution

Add per-card error isolation around `FileDiff` construction, hydration, controller registration, and state application while keeping preload/global failures fatal, then cover the behavior with a focused DOM regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: once a parsed card reaches the hydration loop, any throw before the loop advances exits `hydrateViewer()` and sends control to the page-level error handler. I did not execute a browser repro in this read-only review.

## Solution Assessment

Yes, the recommended per-card try/catch is the narrow maintainable fix. Failed cards should not enter `controllers`, later cards should continue, and a regression test should mock the first `FileDiff.hydrate()` call throwing while the second card still hydrates.
