## Summary

Keep this open. Current main still does not provide the requested cross-platform argument-level exec allowlist contract: non-Windows matching ignores `argPattern`, public editing surfaces still expose path/name patterns only, and the requested `argsPattern` field is not accepted by the protocol schema.

## What This Changes

Request allowlist entries that can constrain both the executable and argv so one trusted interpreter script invocation can be allowed without allowing the entire interpreter.

## Best Possible Solution

Define and document a secure cross-platform argv-level allowlist contract, then implement it consistently in runtime matching, persistence, protocol schema, CLI, Control UI, docs, and regression tests while preserving existing path-only entries.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but the missing behavior is high-confidence from source: on non-Windows a path-matched entry bypasses `argPattern`, and `argsPattern` is rejected by the schema.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed `argsPattern` glob spelling; the capability is valid, but the best path is to reconcile it with existing regex-shaped `argPattern` metadata and define secure argv parsing semantics first.
