## Summary

Keep this open. Current main still only has partial internal `argPattern` handling: non-Windows allowlist matches ignore it, public docs and editors still describe executable-pattern allowlists, and the adjacent closed reports do not cover this cross-platform interpreter/script argv authorization request.

## What This Changes

Add argument-level exec allowlist matching so operators can authorize one specific interpreter or script invocation without allowlisting the entire binary.

## Best Possible Solution

Define a public, cross-platform argv-level exec allowlist contract and implement it consistently across runtime matching, allow-always persistence, protocol schema, CLI, Control UI, docs, and regression tests, while preserving existing path-only allowlist behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a pure feature request, but the missing capability is high-confidence from source and docs: non-Windows matching ignores `argPattern`, and public editing/docs surfaces expose executable patterns only.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the proposed `argsPattern` glob spelling; the capability is valid, but maintainers should reconcile it with the existing regex-shaped `argPattern` metadata and define secure argv parsing semantics first.
