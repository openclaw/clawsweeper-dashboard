## Summary

Keep this open. Current main still lacks the requested cross-platform argv-level exec allowlist contract: non-Windows matching ignores `argPattern`, the proposed `argsPattern` field is not accepted by the protocol schema, and public editing/docs surfaces still describe path or command-name patterns.

## What This Changes

The issue requests argument-level exec allowlist matching so a restricted agent can allow one interpreter invocation without allowing arbitrary interpreter execution.

## Best Possible Solution

Define and document a secure argv-level allowlist contract, then implement it consistently across runtime matching, persistence, protocol schema, CLI, Control UI, docs, and regression tests while preserving existing path-only entries.

## Reproduction Assessment

Not applicable as a feature request. The missing behavior is high-confidence from source inspection: non-Windows path matches bypass argv checks, and the requested `argsPattern` spelling is not part of the accepted schema.

## Solution Assessment

Unclear for the exact proposed `argsPattern` glob field. The capability is valid, but the safer maintainable path is to reconcile it with the existing regex-shaped `argPattern` metadata and specify secure argv semantics first.
