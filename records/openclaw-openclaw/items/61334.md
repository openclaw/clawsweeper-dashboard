## Summary

Keep open: the original 4x/second claim was corrected, but the refined scheduler-policy request remains true on current main and v2026.5.19. Current cron behavior intentionally caps far-future wakes to one minute and logs timer arms at debug level, while adjacent tests and comments preserve one-minute recovery/watchdog semantics that need maintainer policy review before a safe change.

## What This Changes

The issue asks OpenClaw to stop polling cron every 60 seconds for far-future jobs and instead sleep until the next scheduled job, reducing timer-arm log noise and wakeups.

## Best Possible Solution

Define a cron timer policy that separates long normal sleeps from short recovery/watchdog rechecks, then align routine timer-arm debug logging with that policy.

## Reproduction Assessment

Yes. Source inspection shows far-future cron wakes still clamp to 60_000 ms and emit `cron: timer armed`; the reporter already narrowed the original claim to this once-per-minute behavior.

## Solution Assessment

Unclear. Wake-on-next-job scheduling could reduce noise, but current comments and tests intentionally preserve one-minute recovery and watchdog semantics, so maintainers should choose the target policy before implementation.
