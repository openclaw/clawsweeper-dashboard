## Summary

Keep open: current main still re-arms cron at most every 60 seconds for far-future jobs and logs each arm, while tests and comments show that cadence is intentional recovery behavior rather than an already-fixed bug.

## What This Changes

The issue asks cron scheduling to avoid fixed once-per-minute timer re-arms when the next job is much farther away.

## Best Possible Solution

Keep the issue open and choose an explicit scheduler policy: either use longer/dynamic sleeps for normal future jobs while preserving short watchdog and maintenance rechecks, or keep the current cadence and reduce the visibility of `cron: timer armed` in user-facing logs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: a job with `nextRunAtMs` more than 60 seconds in the future still gets clamped to `60_000` and emits `cron: timer armed`.

## Solution Assessment

Unclear. Wake-on-next-job is plausible for log and wakeup reduction, but the current implementation and tests intentionally preserve one-minute recovery behavior, so maintainers should choose the target policy before implementation.
