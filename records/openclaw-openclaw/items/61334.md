## Summary

Keep open: the original 4x/second claim was corrected, but the refined request remains true on current main and latest release: cron still caps normal far-future wakes to 60 seconds and logs the timer arm at debug level. Existing tests and comments protect that one-minute recovery/watchdog behavior, so this needs a maintainer scheduler/logging policy decision rather than cleanup closure.

## What This Changes

The issue requests changing cron scheduling/logging from fixed one-minute timer polling to a wake-on-next-job policy for quiet long-idle cron setups.

## Best Possible Solution

Define a cron timer policy that separates long normal sleeps from short recovery/watchdog rechecks, then align routine timer-arm logging with that policy.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.18 shows far-future timer waits still clamp to 60_000 ms and log `cron: timer armed`; the reporter already corrected the original 4x/second claim to once per minute.

## Solution Assessment

Unclear. Wake-on-next-job scheduling could reduce noise, but current comments and tests intentionally preserve one-minute recovery and watchdog semantics, so maintainers should choose the target policy before implementation.
