## Summary

Keep open: the original 4x/second claim was corrected by the reporter, but the refined concern is still present on current main and the latest release. The one-minute cron wake/log cadence is intentional enough to be covered by tests and comments, so this needs a maintainer scheduler/logging policy decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw cron to avoid fixed 60-second polling and routine `cron: timer armed` log noise when the next scheduled job is much farther in the future.

## Best Possible Solution

Define a cron timer policy that separates long normal sleeps from short recovery/watchdog rechecks, then align routine timer-arm logging with that policy.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.19` shows far-future timer waits still clamp to `60_000` ms and log `cron: timer armed`, and existing tests assert that cadence.

## Solution Assessment

Unclear. Wake-on-next-job scheduling could reduce noise, but current comments and tests intentionally preserve one-minute recovery and watchdog semantics, so maintainers should choose the target policy before implementation.
