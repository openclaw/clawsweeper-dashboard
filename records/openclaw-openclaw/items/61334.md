## Summary

Keep open: current main still clamps cron timer waits to one minute and emits the reported `cron: timer armed` debug line, while regression tests intentionally protect the one-minute cadence for far-future jobs, maintenance rechecks, and active-run watchdog behavior. The refined report is a valid scheduler/logging policy request, not a cleanup-close candidate.

## What This Changes

The issue asks for cron scheduling to sleep until the next due job instead of polling and logging a timer re-arm every minute.

## Best Possible Solution

Define a cron timer policy that separates long normal sleeps from short recovery/watchdog rechecks, then adjust routine timer-arm logging to match that policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the refined behavior: far-future jobs still clamp to `60_000` ms and log `cron: timer armed`. The original 4x/second claim was corrected by the reporter and is not the remaining issue.

## Solution Assessment

Unclear. Wake-on-next-job scheduling could reduce noise, but current comments and tests intentionally preserve one-minute recovery and watchdog semantics, so maintainers should choose the target policy before implementation.
