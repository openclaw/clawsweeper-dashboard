## Summary

Keep open: current main still clamps cron timer arms to a one-minute maximum and logs each normal arm, while nearby comments and tests treat that cadence as intentional recovery/watchdog behavior. The remaining work is a scheduler/logging policy decision rather than something already implemented or safe to auto-close.

## What This Changes

The issue requests reducing cron scheduler wakeups/log noise by sleeping until the next scheduled job instead of re-arming on a fixed one-minute cadence.

## Best Possible Solution

Choose an explicit cron timer policy: either separate long normal sleeps from short watchdog/maintenance rechecks, or keep the current cadence and make routine arm logs less visible in user-facing logs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: a daily or far-future job still gets clamped to a 60-second timer and emits `cron: timer armed` on each arm.

## Solution Assessment

Unclear. Wake-on-next-job scheduling is plausible for reducing wakeups and log noise, but current code comments and tests intentionally preserve one-minute recovery behavior, so maintainers should choose the target policy first.
