## Summary

Keep open: current main still intentionally caps cron timer arms to 60 seconds and logs each normal arm, while nearby comments and tests protect that cadence for recovery/watchdog behavior. This needs a maintainer scheduler/logging policy decision rather than cleanup closure.

## What This Changes

The issue asks for cron to avoid routine one-minute timer arms/logs when the next scheduled job is much farther away.

## Best Possible Solution

Define an explicit timer policy: split long normal sleeps from short watchdog/maintenance rechecks, or keep the cadence and make routine timer-arm debug logs less visible in user-facing logs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: far-future jobs still get clamped to a 60-second timer arm, and the timer arm path still emits `cron: timer armed`.

## Solution Assessment

Unclear. Wake-on-next-job scheduling is plausible, but current comments and tests intentionally preserve one-minute recovery behavior, so maintainers should choose the target policy first.
