## Summary

Keep open: current main still clamps cron timer sleeps to 60 seconds and logs each arm, so the refined once-per-minute polling complaint remains present. The cap is intentional in code and tests for drift, clock-jump, and watchdog recovery, so this needs maintainer policy rather than a conservative cleanup close.

## What This Changes

The issue asks for the cron scheduler to sleep until the next scheduled job instead of re-arming and logging once per minute when all jobs are much later.

## Best Possible Solution

Keep the issue open and decide the scheduler policy explicitly: either use longer/dynamic sleeps for normal future jobs while preserving short watchdog and maintenance rechecks, or keep the current scheduler cadence and reduce the visibility of `cron: timer armed` in user-facing logs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: a job with `nextRunAtMs` more than 60 seconds in the future still gets `delayMs = 60_000` and emits `cron: timer armed` from `armTimer`.

## Solution Assessment

Unclear. Wake-on-next-job is a plausible fix for noise and wakeups, but current code and tests intentionally use a 60-second cap for recovery behavior, so maintainers should choose the target policy before implementation.
