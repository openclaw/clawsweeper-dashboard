## Summary

Keep open: current main still implements the one-minute cron timer clamp and emits the reported debug log, while tests and comments explicitly protect that cadence for recovery and watchdog behavior. The refined request is a scheduler/logging policy change, not a cleanup-close candidate.

## What This Changes

The issue asks cron scheduling to sleep until the next due job instead of polling and logging timer arms every 60 seconds for far-future jobs.

## Best Possible Solution

Define an explicit cron timer policy that separates long normal sleeps from short watchdog or maintenance rechecks, then adjust routine timer-arm logging to match that policy.

## Reproduction Assessment

Yes. Source inspection on current main shows far-future jobs still clamp to `60_000` ms and log `cron: timer armed`, with regression tests asserting the same cadence.

## Solution Assessment

Unclear. Wake-on-next-job scheduling is plausible, but current comments and tests intentionally preserve one-minute recovery and watchdog semantics, so maintainers should choose the target policy first.
