## Summary

Keep open. Current main and the latest release have useful partial primitives for isolated cron runs, direct command jobs, light context, timeouts, model/fallback selection, and tool allow-lists, but they still do not implement the requested first-class scoped/script-limited mode with admin script registry, hard token/spend admission, daily ceilings, dry-run cost preview, explicit persistence policy, or planned-versus-actual audit records.

## What This Changes

The issue requests a scoped/script-limited agent mode for predictable, cost-controlled cron-style execution.

## Best Possible Solution

Compose the existing bounded command-job and isolated-agent primitives into one additive scoped-run contract, adding only an approved admin script registry, a shared budget admission gate with auditable decisions, explicit persistence semantics, and planned-versus-actual run records.

## Reproduction Assessment

Not applicable; this requests a new execution and policy capability rather than reporting a failure of an established contract. Current-source inspection provides high-confidence evidence of which requested components exist and which remain absent.

## Solution Assessment

No, not as a separate parallel agent runtime. The maintainable approach is to reuse current cron command jobs, isolated light-context runs, and tool policies, then design the missing registry, budget, persistence, and audit contracts at their existing owner boundaries.
