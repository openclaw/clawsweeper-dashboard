## Label Justifications

- `P2`: This is a real operator-facing feature request with meaningful safety and cost impact, but it needs design work rather than an urgent regression fix.
- `impact:security`: The requested mode includes whitelisted unattended script execution and therefore changes exec approval, sandboxing, and audit boundaries.
- `impact:session-state`: The request explicitly targets persistent context accumulation and asks for fresh/minimal context per scheduled job.
- `impact:auth-provider`: The requested token/spend ceilings and fallback behavior would affect model/provider dispatch and metered-provider routing.

## Summary

Keep open: current main still provides useful partial cron controls for isolated runs, light context, timeouts, model selection, and tool allow-lists, but it does not implement the requested first-class scoped/script-limited mode with admin scripts, dry-run cost previews, per-job token/spend ceilings, daily budget aborts, or planned-vs-actual run logs. The live discussion and related open trackers support keeping this as a broader product and security design item.

## What This Changes

Request a scoped/script-limited agent mode for predictable cron-first automation with whitelisted scripts, cost/token ceilings, fresh context, dry-run planning, and structured run transparency.

## Best Possible Solution

Design one native scoped automation mode across cron/hook protocol, CLI/config/docs, script policy, budget admission, dry-run planning, and run accounting, coordinating narrower trackers such as https://github.com/openclaw/openclaw/issues/18160, https://github.com/openclaw/openclaw/issues/42475, and https://github.com/openclaw/openclaw/issues/64721.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks a first-class scoped/script-limited mode rather than violating an existing contract.

## Solution Assessment

Unclear until maintainers choose the product and security shape. Existing isolated cron sessions, light context, per-job toolsAllow, timeouts, model/fallback selection, and post-run cost telemetry solve only parts of the requested mode.
