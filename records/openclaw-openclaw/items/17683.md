## Summary

Keep open: current main and the latest release still provide only partial cron controls, not the requested first-class scoped/script-limited mode with admin scripts, dry-run cost preview, per-job token/spend ceilings, daily budget aborts, or planned-vs-actual run logs. Prior review comments and the related open trackers support keeping this as a broader product and security design item.

## What This Changes

The issue requests a scoped/script-limited agent mode for predictable cron execution with script allow-lists, token/spend ceilings, minimal context, and transparent run accounting.

## Best Possible Solution

Design one native scoped automation mode across cron/hook protocol, CLI/config/docs, script policy, budget admission, dry-run planning, and run accounting, coordinating the narrower direct-exec, gateway-budget, and cron-resource-field trackers.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and release inspection give high confidence that current main lacks a first-class scoped/script-limited mode rather than violating an existing contract.

## Solution Assessment

Unclear until maintainers choose the product and security shape. Existing isolated cron sessions, lightContext, per-job toolsAllow, timeouts, model/fallback selection, and post-run cost telemetry solve only parts of the requested mode.
