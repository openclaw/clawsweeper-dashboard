## Summary

Keep open. Current main has the visibility filter and prior dotted archive handling, but the shared transcript-hit helper still treats a QMD-normalized archived Markdown basename as an ordinary `.md` stem, so the later session-key lookup can fail and drop the hit; the broader related report remains open, while this narrower path-mapping bug is actionable on its own.

## What This Changes

The issue reports that `memory_search(corpus="sessions")` can drop QMD archived session hits when QMD normalizes `<session-id>.jsonl.reset.<timestamp>.md` into a hyphenated Markdown path before visibility filtering.

## Best Possible Solution

Normalize QMD-slugged reset/deleted transcript basenames in the shared transcript-hit helper, then cover the memory-core and memory-wiki visibility callers without changing visibility defaults.

## Reproduction Assessment

Yes, at source level. A focused repro can feed `filterMemorySearchHitsBySessionVisibility()` a `source: "sessions"` hit with `qmd/sessions-main/<id>-jsonl-reset-<timestamp>.md` and a store entry for `<id>.jsonl`; current helper logic derives the wrong stem and the filter drops the hit.

## Solution Assessment

Yes. The narrow maintainable fix is to canonicalize QMD-normalized archived transcript names in `src/plugin-sdk/session-transcript-hit.ts` so existing visibility policy decides the result, rather than changing QMD export defaults or session visibility semantics.
