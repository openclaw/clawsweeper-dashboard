## Summary

Keep open. Current main still treats assistant turns that contain message-tool call/result content as tool output in the Control UI rendering path, and the existing delivery-mirror transcript path does not provide a UI fallback when the readable assistant bubble is absent.

## What This Changes

The issue requests Control UI/WebChat to render successful `message(action="send")` replies as normal assistant bubbles while keeping the underlying tool call/result available for debugging.

## Best Possible Solution

Add a narrow, tested Control UI projection for successful `message(action="send")` tool sends that emits one readable assistant bubble only when an equivalent assistant or delivery-mirror message is not already present.

## Reproduction Assessment

Yes, at source level. A history entry containing `message` tool call/result content is normalized as `toolResult` and rendered through the tool-message path rather than projected as an assistant message bubble.

## Solution Assessment

Yes. The narrowest maintainable fix is a UI transcript projection for successful message-tool sends, with dedupe against existing assistant and delivery-mirror messages rather than changing delivery routing or message-tool policy.
