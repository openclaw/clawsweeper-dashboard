## Summary

Keep this PR open for changes: the proposed dispatch-idle wait still has a P1 followup lifecycle regression and the external PR has no real after-fix behavior proof.

## What This Changes

The PR adds an internal dispatch-idle waiter to reply options and uses it to delay queued followups before execution and after followup block delivery.

## Best Possible Solution

Add an abort-aware, turn-scoped delivery barrier that preserves transcript ordering after followup lifecycle registration, then prove the queued followup flow in a real affected channel.

## Reproduction Assessment

Yes, source-reproducible: hold the PR's dispatch-idle waiter pending after queue drain enters the runner, then send /stop; the followup is active in the drain but not yet registered as a cancelable reply operation.

## Solution Assessment

No. Waiting for delivery ordering is useful, but this patch parks before lifecycle registration; the safer fix is an abort-aware registration point or turn-scoped delivery boundary.
