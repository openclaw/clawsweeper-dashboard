## Summary

Keep open for maintainer review: the underlying reply-ordering problem is real, but this branch still lacks after-fix proof and the patch can let a queued followup send after `/stop` because the idle wait happens before abort-visible registration.

## What This Changes

The PR adds an internal `waitForDispatchIdle` reply option, gates queued followup startup and followup delivery on dispatcher-idle waits, and adds helper tests.

## Best Possible Solution

Use an abort-aware, turn-scoped delivery barrier that registers queued followups before any idle wait, and coordinate with https://github.com/openclaw/openclaw/pull/83722 instead of merging competing delivery barriers.

## Reproduction Assessment

Yes. Source-reproducible: hold the PR's dispatch-idle waiter pending after queue drain enters the runner but before `createReplyOperation`, then issue `/stop`; the parked followup is not yet abort-visible.

## Solution Assessment

No. Waiting for dispatcher ordering may be useful, but this implementation waits before lifecycle registration; the safer solution is an abort-aware turn-scoped boundary after registration or convergence with the active dispatcher-delivery fix.
