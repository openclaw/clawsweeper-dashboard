## What This Changes

The PR adds a `waitForDispatchIdle` reply option, wraps queued followup startup and followup block delivery in dispatcher-idle waits, and adds small helper tests.

## Best Possible Solution

Land a bounded, abort-aware followup ordering barrier that registers queued followups before any wait and reuses the dispatcher delivery barrier already merged for block replies.

## Reproduction Assessment

Yes. Source-reproducible: hold the PR's dispatch-idle waiter pending before the followup creates `ReplyOperation`, then issue `/stop`; the current abort path only sees registered reply runs and embedded sessions.

## Solution Assessment

No. Waiting for dispatch ordering may be useful, but this implementation waits before lifecycle registration and should be redesigned around an abort-aware, registered followup boundary coordinated with the merged delivery barrier.
