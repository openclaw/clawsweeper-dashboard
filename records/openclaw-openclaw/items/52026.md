## Summary

Keep this PR open. Current main still lacks the requested followup dispatch-idle barrier, and the related #65761 replacement was closed unmerged, but this patch should not merge as-is because the new pre-run wait can let `/stop` miss a queued followup that has already entered the runner but has not registered a reply operation or embedded run yet.

## What This Changes

The PR adds a `waitForDispatchIdle` reply option, helper coverage for followup dispatch-idle waits, and followup-runner calls that wait before starting queued followup work and after dispatcher-backed followup delivery.

## Best Possible Solution

Land a turn-scoped, abort-aware dispatch barrier that keeps parked followups visible to the stop path until they either start or are cancelled, and add regression coverage for dispatch ordering, `/stop` during the wait window, and multiple queued followups.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to enqueue a followup, keep the prior dispatcher delivery pending so the PR's new wait parks the followup, then issue `/stop`; the current stop path clears queues and active runs but cannot cancel that parked pre-operation waiter.

## Solution Assessment

No. The intended direction is valid, but this implementation places the wait outside the cancellation/operation lifecycle; the safer fix is to make the barrier abort-aware and registered with the same stop semantics as active followup work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cancel followups parked behind dispatch-idle waits:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait runs before the followup creates a reply operation or starts `runEmbeddedPiAgent`. `scheduleFollowupDrain()` has already handed this item to `run()`, so `/stop` can clear queued items and abort active embedded runs without cancelling this parked followup; once dispatch drains, it can still start and reply after the stop request. Make the idle wait abort-aware or keep the dequeued followup registered with the stop path until it either starts or is cancelled.
  - confidence: 0.9
