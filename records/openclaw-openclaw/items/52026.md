## Summary

Keep open: the branch is still a plausible reply-ordering fix attempt, but the current patch has a P1 lifecycle regression and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a dispatcher-idle wait around queued auto-reply followup execution and delivery, threads a `waitForDispatchIdle` callback through reply options, and adds small helper tests.

## Best Possible Solution

Use a turn-scoped dispatch barrier or awaitable delivery contract that preserves transcript ordering while registering an abort-aware followup lifecycle before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the PR's `waitForDispatchIdle` pending after `drainNextQueueItem` enters the runner, then send `/stop`; the parked followup has not registered a reply operation or embedded run yet.

## Solution Assessment

No. Waiting for delivery ordering is a useful direction, but this implementation parks before lifecycle registration; the wait should be behind an abort-aware registration point or replaced by an awaitable delivery boundary.
