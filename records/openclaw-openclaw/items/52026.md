## Summary

Keep this PR open for maintainer/contributor follow-up. The dispatch-idle direction is plausible, but this patch parks a followup before it is registered with the reply abort lifecycle, the branch is currently conflicting, and the external contributor has not supplied real after-fix behavior proof.

## What This Changes

The PR adds an internal dispatch-idle waiter to auto-reply followup delivery, passes it through reply options, and adds helper tests for the idle barrier.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier that waits for prior delivery before followup transcript reads while registering the followup with an abort-aware `ReplyOperation` before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the new idle wait pending after `drainNextQueueItem` starts, then issue `/stop`; the parked followup has no `ReplyOperation` or embedded run yet, so the abort path can miss it.

## Solution Assessment

No. The dispatcher-idle direction is useful, but the wait must move under a registered abort-aware operation or otherwise observe the stop lifecycle before parking the followup.
