## What This Changes

The PR adds a `waitForDispatchIdle` reply option, wraps queued followup startup and followup block delivery around dispatcher-idle waits, and adds helper tests for those waits.

## Best Possible Solution

Use an abort-aware, turn-scoped delivery barrier that registers queued followups before any idle wait, and converge with the active same-channel dispatcher-delivery fix instead of adding a competing wait.

## Reproduction Assessment

Yes. Source-reproducible: hold the PR's dispatch-idle waiter pending after queue drain enters the runner but before `createReplyOperation`, then issue `/stop`; the parked followup is not yet abort-visible.

## Solution Assessment

No. Waiting for dispatcher ordering may be useful, but this implementation waits before lifecycle registration; the safer solution is an abort-aware turn-scoped boundary after registration or an awaited delivery contract at the send site.
