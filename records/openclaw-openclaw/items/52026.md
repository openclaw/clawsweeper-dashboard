## Summary

Keep open: the PR targets a real reply-ordering hazard, but the current branch is not safe to merge because it waits before followup lifecycle registration and still lacks required real behavior proof. Current main also does not already contain the PR's dispatch-idle helper surface.

## What This Changes

The PR threads a dispatch-idle waiter through reply options, waits before queued followup execution and after followup block delivery, and adds small helper coverage.

## Best Possible Solution

Use a turn-scoped dispatch barrier or awaitable delivery contract that preserves transcript ordering while registering an abort-aware followup lifecycle before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the PR's waitForDispatchIdle pending after drainNextQueueItem enters the runner, then send /stop; the followup is already handed to the runner but has not registered a reply operation or embedded run.

## Solution Assessment

No. Waiting for delivery ordering is useful, but this implementation parks before lifecycle registration; the wait should move behind an abort-aware registration point or be replaced by a turn-scoped awaitable delivery boundary.
