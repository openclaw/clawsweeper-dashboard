## Summary

Keep open. Current main still lacks this PR's dispatch-idle barrier, but the PR should not merge as-is because its pre-run wait can park a followup outside the `/stop` cancellation and reply-operation lifecycle.

## What This Changes

The PR adds a `waitForDispatchIdle` reply option, followup dispatch-idle helpers/tests, and followup-runner waits before starting followups and after dispatcher-backed followup delivery.

## Best Possible Solution

Land a turn-scoped, abort-aware dispatch barrier that keeps parked followups visible to the stop path until they either start or are cancelled.

## Reproduction Assessment

Yes. A source-level reproduction is to enqueue a followup, keep the prior dispatcher delivery pending so the PR parks the followup, then issue `/stop`; that parked waiter is not registered as active work.

## Solution Assessment

No. The intended ordering barrier is valid, but this implementation places it before `createReplyOperation`; the safer fix is to register or abort-scope the wait under the same cancellation lifecycle as active followup work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Cancel followups parked behind dispatch-idle waits:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait runs before the followup creates a reply operation or starts `runEmbeddedPiAgent`. `scheduleFollowupDrain()` has already handed this item to `run()`, so `/stop` can clear queued items and abort active embedded runs without cancelling this parked followup; once dispatch drains, it can still start and reply after the stop request. Make the idle wait abort-aware or keep the dequeued followup registered with the stop path until it either starts or is cancelled.
  - confidence: 0.9
