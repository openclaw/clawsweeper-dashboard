## Summary

Keep open: the PR targets a real reply-ordering problem, but the current patch introduces an abort lifecycle gap by waiting before the followup is registered, and the external real behavior proof gate is still missing.

## What This Changes

The PR adds a dispatch-idle waiter to reply options and uses it to delay queued followups before execution and after followup block delivery.

## Best Possible Solution

Add an abort-aware, turn-scoped delivery barrier that preserves transcript ordering while creating the followup ReplyOperation before any potentially long wait, then prove the flow in a real channel.

## Reproduction Assessment

Yes, source-reproducible: hold the PR's dispatch-idle waiter pending after the queue drain enters the runner, then send /stop; the followup is active in the drain but not yet registered as a reply operation or embedded run.

## Solution Assessment

No. Waiting for delivery ordering is useful, but this patch parks before lifecycle registration; the wait should move behind an abort-aware registration point or be replaced by a turn-scoped delivery boundary.
