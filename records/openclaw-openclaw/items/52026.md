## Summary

Keep open, but this branch is not merge-ready: source inspection shows the new wait can park an already-dequeued followup before reply operation registration, so `/stop` cannot cancel it, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds a followup dispatch-idle helper and internal reply option so queued followups wait around asynchronous dispatcher delivery before continuing.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier or awaitable delivery contract that preserves transcript ordering while registering an abort-aware followup lifecycle before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the PR's new idle wait pending after queue drain enters the runner, then issue `/stop`; the parked followup has no registered reply operation or embedded run for the abort path to cancel.

## Solution Assessment

No. Waiting for delivery ordering is a useful direction, but this branch waits before abort-aware lifecycle registration and should be reworked under the existing reply operation/run cancellation contract or a narrower turn-scoped delivery awaiter.
