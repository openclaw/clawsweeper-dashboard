## Summary

Keep open: current main has not implemented an equivalent idle barrier, but this PR should not merge as-is because it waits before followups are registered with the reply-operation/abort lifecycle, letting stopped followups start after dispatcher idle.

## What This Changes

The PR adds a dispatcher-idle waiter to reply options, uses it to gate queued followup startup and followup block delivery, and adds helper tests.

## Best Possible Solution

A current-main fix should preserve turn-scoped dispatch backpressure before transcript reads and keep every waiting followup cancellable through `/stop` and fast-abort before it can start.

## Reproduction Assessment

Yes. By source inspection, hold the dispatcher idle wait, let `scheduleFollowupDrain()` enter the followup runner, then issue `/stop`; the waiter is before `createReplyOperation`, so the abort path cannot cancel it.

## Solution Assessment

No. The ordering barrier is a valid direction, but this implementation parks dequeued followups outside the registered abort lifecycle; the safer fix is an abort-aware wait or moving the wait under the current reply operation.
