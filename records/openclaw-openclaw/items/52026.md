## Summary

Keep open: the PR is a plausible fix attempt, but it is not safe to merge as-is because the added idle wait can park a dequeued followup before abort-aware lifecycle registration, and the PR still lacks real after-fix behavior proof. Current main also does not already contain this waiter surface, while the related followup/block-delivery attempt at https://github.com/openclaw/openclaw/pull/65761 is closed unmerged rather than a canonical merged replacement.

## What This Changes

The PR threads a dispatcher-idle callback through reply options, waits before queued followup runs and after followup block delivery, and adds small helper coverage.

## Best Possible Solution

Use a turn-scoped dispatch barrier or awaitable delivery contract that preserves transcript ordering while registering an abort-aware followup lifecycle before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the PR's `waitForDispatchIdle` pending after `drainNextQueueItem` enters the runner, then send `/stop`; the parked followup has already been handed to the runner but has not registered a reply operation or embedded run.

## Solution Assessment

No. Waiting for delivery ordering is a useful direction, but this implementation parks before lifecycle registration; the wait should move behind an abort-aware registration point or be replaced by a turn-scoped awaitable delivery boundary.
