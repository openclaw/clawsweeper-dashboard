## Summary

Keep open, but this PR is not merge-ready: the proposed idle wait addresses a real reply-ordering area, yet it parks queued followups before reply lifecycle registration, so abort/stop handling can miss already-dequeued work. The PR also still lacks real behavior proof and overlaps active delivery-order work tracked separately.

## What This Changes

The PR adds a `waitForDispatchIdle` reply option, wraps queued followup work and followup block delivery around dispatcher-idle waits, and adds helper tests for that behavior.

## Best Possible Solution

Use an abort-aware, turn-scoped delivery barrier that registers queued followups before any idle wait, and converge the shared dispatcher-delivery ordering work with the active canonical delivery-order fix.

## Reproduction Assessment

Yes, source-reproducible: hold the PR's dispatch-idle waiter pending after queue drain enters the runner, then send `/stop`; the followup is active in the drain but has not yet created a reply operation or embedded run for abort to target.

## Solution Assessment

No. Waiting for dispatch ordering is a useful direction, but this implementation waits before lifecycle registration; the safer fix is an abort-aware turn-scoped delivery boundary after followup registration, or an awaited dispatcher-delivery contract at the send site.
