## Summary

Keep open, but not merge-ready: this PR targets a real reply-ordering hazard, but the diff parks queued followups before abort-visible lifecycle registration and still lacks after-fix real behavior proof. It also overlaps active delivery-order work in https://github.com/openclaw/openclaw/issues/32868 and https://github.com/openclaw/openclaw/pull/83722.

## What This Changes

The PR adds a `waitForDispatchIdle` reply option, wraps queued followup startup and followup block delivery around dispatcher-idle waits, and adds helper tests for those waits.

## Best Possible Solution

Use an abort-aware, turn-scoped delivery barrier that registers queued followups before any idle wait, and coordinate with https://github.com/openclaw/openclaw/pull/83722 instead of merging competing delivery barriers.

## Reproduction Assessment

Yes. Source-reproducible: hold the PR's dispatch-idle waiter pending after queue drain enters the runner but before `createReplyOperation`, then issue `/stop`; the parked followup is not yet abort-visible.

## Solution Assessment

No. Waiting for dispatcher ordering may be useful, but this implementation waits before lifecycle registration; the safer solution is an abort-aware turn-scoped boundary after registration or an awaited delivery contract at the send site.
