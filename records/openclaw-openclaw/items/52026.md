## Summary

Keep open. The dispatcher-idle direction addresses a real ordering hazard, but this PR still parks a queued followup before it has a `ReplyOperation` or embedded-run handle, so `/stop` can miss it and the followup can resume after dispatch drains.

## What This Changes

The branch adds an internal dispatch-idle waiter through reply options, uses it before and after followup block delivery, and adds helper tests.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier that waits for prior delivery before followup transcript reads while registering the followup with an abort-aware operation before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the new idle wait pending after the queue runner starts, then issue `/stop`; because no reply operation or embedded run exists yet, the stopped followup can later resume.

## Solution Assessment

No. The dispatcher-idle direction is useful, but the pre-run wait needs to move under a registered `ReplyOperation` or become explicitly abort-aware before parking the followup.
