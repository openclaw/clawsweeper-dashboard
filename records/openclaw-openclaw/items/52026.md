## Summary

Keep this PR open for maintainer/contributor follow-up: it targets a real reply-ordering bug, but the current patch is not mergeable as-is because the idle wait is tied to the wrong dispatcher path and can also park a followup outside the abort lifecycle.

## What This Changes

The PR adds a dispatcher-idle waiter through reply dispatch, waits before queued followup execution and after followup block delivery, and adds helper tests.

## Best Possible Solution

Rework this against current main as a turn-scoped, abort-aware dispatch barrier that follows the active run's final delivery into queued followup transcript reads and preserves stop/restart lifecycle handling.

## Reproduction Assessment

Yes, by source inspection. Current main still lacks a followup dispatch-idle barrier, and the PR-introduced failure follows from the new wait running before lifecycle registration while queue drain and stop handling only know about queued items or registered runs.

## Solution Assessment

No. Waiting for dispatcher idle is the right direction, but this patch waits on the wrong dispatcher boundary and parks followups before they are abort-visible; the safer fix is a turn-scoped barrier registered inside the reply-operation lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait on the active turn dispatcher:** `src/auto-reply/reply/dispatch-from-config.ts:555`
  - body: The waiter passed to followups is bound to this `dispatchReplyFromConfig` call's dispatcher, but queued followup drains can use the callback cached by the later message that enqueued the followup. In that case the followup waits on an idle newer/empty dispatcher while the previous turn's final reply is still in flight, so the transcript-ordering race remains. Carry the active turn's dispatch barrier into the queue drain instead.
  - confidence: 0.82
- **[P1] Register followups before dispatch-idle waits:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait happens before this followup creates a `ReplyOperation` or starts the embedded run. Because the queue drain awaits the item before shifting it, `/stop` can clear queued entries and abort registered runs without seeing this parked followup; once dispatch drains, it can still start and reply after the stop request. Move the wait under a registered or explicitly abort-aware operation.
  - confidence: 0.9
