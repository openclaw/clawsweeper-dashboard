## Summary

Keep open: the proposed ordering fix is plausible, but current PR head still has a blocking abort-lifecycle regression and lacks real behavior proof. Current main does not have this followup dispatch-idle barrier, and the broader same-channel delivery problem remains tracked in https://github.com/openclaw/openclaw/issues/32868.

## What This Changes

The branch passes a dispatcher-idle waiter into followup handling, waits before followup work and after followup block sends, and adds helper tests for that wait behavior.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier that waits for prior delivery before followup transcript reads while registering an abort-aware followup lifecycle before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the new idle wait pending after queue drain enters the runner, then issue `/stop`; the parked followup has no registered reply operation or embedded run for the abort path to cancel.

## Solution Assessment

No. Waiting for dispatch idle is a useful direction, but this implementation waits outside the registered abort lifecycle and should be moved under an abort-aware operation or replaced with a narrower awaitable delivery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

- **[P1] Register followups before waiting for dispatch idle:** `src/auto-reply/reply/followup-runner.ts:140`
