## Summary

Keep open. The PR targets a real followup ordering bug that is not implemented on current main, but the branch is not merge-ready because it waits before an abortable `ReplyOperation` exists and lacks real after-fix behavior proof.

## What This Changes

The branch adds an internal dispatch-idle waiter through reply options, waits around queued followup delivery, and adds helper tests for the idle barrier.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier that waits for prior delivery before followup transcript reads while registering the followup with an abort-aware operation before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the new idle wait pending after `drainNextQueueItem` starts, then issue `/stop`; the parked followup has no `ReplyOperation` or embedded run yet, so the abort path can miss it.

## Solution Assessment

No. The dispatcher-idle direction is useful, but the pre-run wait needs to move under a registered `ReplyOperation` or become explicitly abort-aware before parking the followup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Register followups before waiting for dispatch idle:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait happens before this followup creates a `ReplyOperation` or starts the embedded run. Because `drainNextQueueItem` awaits the runner before shifting the item, `/stop` can clear queued entries and abort registered runs without seeing this parked followup; once dispatch drains, it can still start and reply after the stop request.
