## Summary

Keep open. The ordering race is still real on current main and this PR is useful source work, but the submitted head parks a dequeued followup behind the new dispatcher-idle wait before the followup is registered with the abort lifecycle.

## What This Changes

The PR adds an internal `waitForDispatchIdle` reply option, a dispatcher-idle gate in `dispatch-from-config`, helper coverage, and followup-runner waits before starting and after dispatcher-backed followup sends.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier that waits for prior delivery before followup transcript reads while registering the followup with `ReplyOperation` or an abort-aware wait before it can be parked.

## Reproduction Assessment

Yes by source inspection. Current main still fire-and-forgets same-channel block delivery, and the PR-introduced defect is reproducible by holding `waitForDispatchIdle` pending after `scheduleFollowupDrain()` enters the runner, then issuing `/stop` before any `ReplyOperation` or embedded run exists.

## Solution Assessment

No. The turn-scoped dispatcher-idle direction is good, but the first wait must be under the current reply-operation and abort lifecycle or be explicitly abort-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Register parked followups before dispatch-idle waits:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait executes before the followup creates any `ReplyOperation` or starts `runEmbeddedPiAgent`. Since `scheduleFollowupDrain()` has already entered `run(next)`, `/stop` can clear queued items and abort active runs without seeing this parked followup; after dispatch drains it can still start and reply. Move the wait under a registered/abort-aware operation before parking the followup.
  - confidence: 0.9
