## Summary

Keep open: current main still lacks the PR's dispatch-idle followup behavior and the related replacement is closed unmerged, but this patch should not merge because the new pre-run wait can survive `/stop` and send a followup after cancellation.

## What This Changes

The PR adds a `waitForDispatchIdle` reply option, wraps `dispatch-from-config` with a dispatcher idle gate, and makes queued followups wait before starting and after dispatcher-backed delivery.

## Best Possible Solution

Land a current-main, turn-scoped dispatch barrier that blocks followup transcript reads until prior delivery drains while keeping the waiting followup under the reply-operation and abort lifecycle.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: keep the dispatcher wait pending, let `scheduleFollowupDrain()` enter the runner, then issue `/stop`; the waiter is before `createReplyOperation`, so abort cannot see it. I did not run live tests in this read-only sweep.

## Solution Assessment

No. The intended ordering barrier is valid, but this implementation parks the dequeued followup outside the cancellation lifecycle; the safer fix is an abort-aware or registered wait.

## Review Findings

- **[P1] Cancel followups parked behind dispatch-idle waits:** `src/auto-reply/reply/followup-runner.ts:140`
