## Summary

Keep this PR open but require changes: current main still lacks the proposed dispatcher-idle barrier and the ordering problem is real, but the PR parks queued followups before they are visible to the abort/reply-operation lifecycle and it has no real behavior proof.

## What This Changes

The PR adds an internal dispatcher-idle wait option, wires it through reply dispatch, gates queued followups before and after dispatcher-backed sends, and adds helper tests.

## Best Possible Solution

Land a rebased, turn-scoped dispatch barrier that waits for prior delivery before followup transcript reads while registering the followup with ReplyOperation or an abort-aware wait before it can be parked.

## Reproduction Assessment

Yes, by source inspection. Current main still fire-and-forgets same-channel block delivery, and the PR-introduced defect is reproducible by holding the new idle wait pending after the queue runner starts, then issuing `/stop` before any ReplyOperation or embedded run exists.

## Solution Assessment

No. The dispatcher-idle direction is useful, but the first wait needs to be inside the registered reply-operation/abort lifecycle or explicitly abort-aware so stopped followups cannot resume and reply later.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Register followups before waiting for dispatch idle:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait happens before this followup creates a ReplyOperation or starts the embedded run. Because `drainNextQueueItem` awaits `run(next)` before shifting, `/stop` can clear queued entries and abort registered runs without seeing this parked followup; once dispatch drains, it can still start and reply after the stop request. Move the wait under a registered or abort-aware operation before parking the followup.
  - confidence: 0.9
