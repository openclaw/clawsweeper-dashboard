## Summary

Keep open. The ordering problem is plausible, but this patch parks a followup before it registers an abort-aware reply operation, so `/stop` can miss the active followup; the external PR also still lacks real behavior proof.

## What This Changes

The branch wraps auto-reply dispatch with a `waitForDispatchIdle` callback, adds followup idle helper/tests, and passes the option through `GetReplyOptions`.

## Best Possible Solution

Rework this as a turn-scoped dispatch barrier or awaitable delivery contract that preserves transcript ordering while registering an abort-aware followup lifecycle before any long wait.

## Reproduction Assessment

Yes, by source inspection. Hold the PR's `waitForDispatchIdle` pending after `drainNextQueueItem` calls the runner, then issue `/stop`; the active local followup has not registered a reply operation or embedded run yet.

## Solution Assessment

No. Waiting for delivery ordering is a useful direction, but the wait belongs behind an abort-aware lifecycle registration or an awaitable delivery boundary that `/stop` can observe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Register followups before waiting for dispatch idle:** `src/auto-reply/reply/followup-runner.ts:140`
