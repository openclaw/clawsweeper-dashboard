## Summary

Keep this PR open: it addresses a real current-main gateway routing bug, but PR head still has a blocking same-channel mixed-route issue and lacks after-fix real behavior proof before merge.

## What This Changes

The PR changes gateway agent delivery-context precedence so current request routes can override stale session routes, propagates `currentChannelId`, adds gateway regression tests, and adds a changelog entry.

## Best Possible Solution

Amend this PR or replace it with a narrow branch that atomically replaces same-channel target routes: when `to` changes, clear `accountId` and `threadId` unless the request supplies them or a channel resolver derives a validated pair, then add the omitted-account regression and rebase.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence reproduction path: a same-channel `deliver: true` request changes `to` while omitting `accountId`, causing the stored route to combine the new target with the previous accountId.

## Solution Assessment

No. The PR targets the right gateway layer and fixes part of the stale-route problem, but the same-channel replacement path is not the best solution until route fields are updated atomically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not carry stale accountId onto a new target:** `src/gateway/server-methods/agent.ts:946`
  - body: When `requestTargetDiffers` is true, this branch persists `requestDeliveryHint.to` but falls back to `sessionDelivery.accountId`. A same-channel `deliver: true` request that changes `to` while omitting `accountId` leaves the session with a mixed `{to, accountId}` route, so later implicit delivery can use credentials from the previous target.
