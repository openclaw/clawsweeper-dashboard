## Summary

Keep this PR open: it targets a real gateway routing bug, but the current head still has a blocking route-safety issue and lacks after-fix real behavior proof.

## What This Changes

The PR changes gateway agent delivery-context precedence, forwards the resolved target into run context, adds route regression tests, and adds a changelog entry.

## Best Possible Solution

Amend this PR or replace it with a narrow gateway fix that updates coupled route fields atomically when `to` changes, adds the missing regression, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection of PR head shows a same-channel `deliver: true` request can change `to` while omitting `accountId`, and the branch falls back to `sessionDelivery.accountId`; I did not run a live gateway/channel flow in this read-only review.

## Solution Assessment

No. The gateway handler is the right layer, but target replacement should clear or validate coupled `accountId` and `threadId` fields instead of independently falling back to stale session values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not carry stale accountId onto a new target:** `src/gateway/server-methods/agent.ts:946`
