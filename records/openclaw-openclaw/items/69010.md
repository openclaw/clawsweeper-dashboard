## Summary

Keep open: this PR targets a real gateway routing bug and current main still lacks the PR's session-persistence and `currentChannelId` pieces, but PR head still persists a new same-channel target with a stale `accountId` and lacks real behavior proof for the latest head.

## What This Changes

The PR changes gateway agent delivery-context merging, forwards the resolved target into run context, adds gateway regression tests, and adds a changelog entry.

## Best Possible Solution

Amend this PR or replace it with a narrow gateway fix that updates coupled same-channel route fields atomically: when `to` changes, clear `accountId` and `threadId` unless the request supplies them or a channel resolver validates the pair, then add the missing regression and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection of PR head: a same-channel `deliver: true` request can change `to` while omitting `accountId`, and line 946 falls back to `sessionDelivery.accountId`. I did not run a live gateway/channel flow in this read-only review.

## Solution Assessment

No. The PR targets the right gateway layer and covers part of the stale-route bug, but target replacement should update coupled route fields atomically or validate the route pair before persisting it.

- **[P1] Do not carry stale accountId onto a new target:** `src/gateway/server-methods/agent.ts:946`
