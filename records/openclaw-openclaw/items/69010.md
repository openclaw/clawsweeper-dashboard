## Label Justifications

- `P1`: The PR targets an active multi-channel gateway misdelivery bug and currently has a P1 route-safety defect before merge.
- `merge-risk: 🚨 message-delivery`: The diff changes gateway route precedence and can currently persist a mixed route that sends responses to the wrong destination.
- `merge-risk: 🚨 session-state`: The problematic state is stored in session `deliveryContext` and `last*` fields, so a bad merge can poison later implicit sends.
- `merge-risk: 🚨 security-boundary`: A stale `accountId` can select the wrong channel account or credential for a new target, which is a concrete sensitive-delivery boundary.

## Summary

Keep open: current main still lacks the route-precedence and `currentChannelId` fix, but this PR head should not merge because same-channel target replacement can persist a stale `accountId` and the PR still lacks real behavior proof.

## What This Changes

The PR changes gateway agent delivery-context precedence, forwards resolved delivery targets into run context, adds route regression tests, and updates the changelog.

## Best Possible Solution

Amend the gateway handler so `to`, `accountId`, and `threadId` are updated as coupled route fields, add a regression for changing `to` without `accountId`, then attach redacted real gateway/channel proof before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows a same-channel `deliver: true` request can change `to` while omitting `accountId`, and the branch falls back to `sessionDelivery.accountId`; I did not run a live gateway/channel flow in this read-only review.

## Solution Assessment

No. The gateway handler is the right layer, but target replacement should clear or validate coupled `accountId` and `threadId` fields instead of independently falling back to stale session values.
