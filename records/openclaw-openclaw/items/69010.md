## Summary

Keep open. The PR addresses a real gateway routing gap on current main, but the head patch still has a concrete same-channel route replacement bug: a new target can be persisted with the old session accountId, which can misroute later implicit deliveries. The Aisle security comment raised the same concern, so this should not merge until that path is made atomic.

## What This Changes

The PR changes gateway agent delivery-context precedence, forwards the resolved delivery target into runContext.currentChannelId, adds gateway agent regression tests, and adds a changelog fix entry.

## Best Possible Solution

Keep the PR open and revise the same-channel replacement path so coupled route fields are updated atomically: when a target changes, do not carry accountId or threadId from the previous target unless the request supplies them or a channel resolver validates and derives them. Add a regression for changing to without accountId, then rebase and rerun the targeted gateway tests plus the changed gate.

## Reproduction Assessment

Yes. The original bug has a high-confidence reproduction shape from #68378 and the PR’s stale Feishu deliveryContext plus current Discord request regression test; the remaining review finding is reproducible by a same-channel deliver=true request that changes to while omitting accountId.

## Solution Assessment

No. The PR’s direction is the right narrow gateway layer, but the current implementation is not the best fix until target/account/thread replacement is atomic for same-channel target changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not carry stale accountId onto a new target:** `src/gateway/server-methods/agent.ts:946`
  - body: When requestTargetDiffers is true, this branch persists requestDeliveryHint.to but falls back to sessionDelivery.accountId. A deliver=true request that changes to on the same channel without accountId leaves the session with a mixed {to, accountId} route, so later implicit delivery can use credentials from the previous target. Clear coupled fields or resolve them atomically when the target changes.
  - confidence: 0.88
