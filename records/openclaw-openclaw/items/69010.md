## Summary

Keep open: the PR addresses a real gateway route-precedence bug, but the latest head still has a P1 same-channel route-safety defect and lacks after-fix real gateway/channel behavior proof.

## What This Changes

This PR changes gateway agent delivery-context precedence and canonical target comparison, forwards `resolvedTo` into `runContext.currentChannelId`, adds gateway regression tests, and adds a changelog entry.

## Best Possible Solution

Amend this PR or replace it with a narrow branch that atomically replaces same-channel target routes: when `to` changes, clear `accountId` and `threadId` unless the request supplies them or a channel resolver validates the pair, then add the omitted-account regression and real gateway/channel proof.

## Reproduction Assessment

Yes by source inspection of PR head: a same-channel `deliver: true` request can change `to` while omitting `accountId`, causing the stored route to combine the new target with the previous accountId. I did not run the gateway/channel flow live in this read-only review.

## Solution Assessment

No. The PR targets the right gateway layer and fixes part of the stale-route bug, but same-channel target replacement should update coupled route fields atomically before merge.

- **[P1] Do not carry stale accountId onto a new target:** `src/gateway/server-methods/agent.ts:946`
