## Summary

Keep open: this PR targets a real gateway route-precedence bug that is still not implemented on current main, but the latest head can persist a new same-channel target with a stale accountId and still lacks after-fix real behavior proof.

## What This Changes

The PR changes gateway agent delivery-context precedence to favor concrete current request routes over stale session routes, forwards the resolved target into `runContext.currentChannelId`, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Amend this PR or replace it with a narrow branch that atomically replaces same-channel target routes: when `to` changes, clear `accountId` and `threadId` unless the request supplies them or a channel resolver validates the pair, then add the omitted-account regression and real gateway/channel proof.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: a same-channel `deliver: true` request can change `to` while omitting `accountId`, causing the stored route to combine the new target with the previous accountId.

## Solution Assessment

No. The PR targets the right gateway layer and fixes part of the stale-route bug, but the same-channel target replacement path should update coupled route fields atomically before merge.
