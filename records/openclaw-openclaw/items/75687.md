## Summary

Keep open. The report is fresh, detailed, and corroborated by a second Windows setup, while current main still has source-level gaps around concurrent channel/account startup and the exact Bonjour self-probe crash signature; related closed reports and 2026.4.30 changelog mitigations cover narrower stall classes but do not prove this Windows Telegram profile is fixed.

## What This Changes

The issue reports Windows regressions across OpenClaw 2026.4.24 through 2026.4.29 that make Telegram unusable through event-loop saturation during plugin startup, Bonjour ciao crashes, polling stalls, slow model fallback, and a Windows ESM loader path failure.

## Best Possible Solution

Keep this open until a maintainer or focused validation run reproduces a package candidate newer than v2026.4.29 on Windows with the reported Telegram/plugin profile, then split or land targeted fixes for the confirmed boundary: channel/account startup backpressure, runtime-deps startup pressure, Bonjour self-probe recovery, model fallback timing, or Windows ESM path handling.

## Reproduction Assessment

Unclear for current main. The issue provides a strong reproduction path for shipped v2026.4.29 and another Windows/Node 24 confirmation, but this read-only review did not run a Windows package candidate and current main contains unshipped mitigations that need live proof.

## Solution Assessment

Unclear. Staggering or deferring Telegram account startup is plausible, but the evidence also implicates runtime-deps startup work and Bonjour ciao handling, so the narrow maintainable fix should be chosen after Windows package-candidate reproduction.
