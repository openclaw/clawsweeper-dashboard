## Summary

Keep open. Current main has adjacent post-v2026.4.29 startup/runtime-deps, Telegram, and Bonjour mitigations, but it still starts all accounts for a channel concurrently, still starts channels with up to eight workers, and does not classify the exact Bonjour self-probe throw as a recoverable ciao process error; there is no current-main Windows package-candidate proof that the central report is fixed.

## What This Changes

The issue reports Windows regressions in OpenClaw 2026.4.24 through 2026.4.29 that make Telegram unusable through startup event-loop saturation, polling/send stalls, Bonjour ciao crashes, slow model fallback, and Windows ESM loading failures.

## Best Possible Solution

Verify a current Windows package candidate with the reported Telegram/plugin profile, then split the smallest targeted fixes for Telegram startup backpressure and Bonjour self-probe recovery while keeping broader runtime-deps and embedded-run latency work tied to their canonical threads.

## Reproduction Assessment

Yes for shipped v2026.4.29; unclear for current main. The issue gives concrete reproduction steps, logs, and a second Windows confirmation, but this read-only review did not run a current-main Windows package candidate.

## Solution Assessment

Unclear. Staggering or deferring Telegram startup is plausible, but the remaining fix boundary should be chosen after current-main Windows proof because the logs also implicate packaged plugin startup/runtime-deps and Bonjour ciao recovery.
