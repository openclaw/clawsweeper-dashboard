## Label Justifications

- `P2`: This is a real compatibility/product gap with production examples, but it is limited to reasoning-capable model paths and needs design work rather than an emergency runtime patch.
- `impact:message-loss`: The reported `content:null` and reasoning-only output shapes can leave users with no visible assistant reply or trigger fallback instead of delivering the useful answer.
- `impact:security`: Reasoning-field ingestion and reporting could expose hidden provider reasoning unless the behavior is opt-in and redacted.
- `impact:auth-provider`: The requested solution involves provider/model reasoning payload policy, model-profile fields, and fallback/routing behavior.

## Summary

Keep open: current main has several partial mitigations for reasoning-only responses and OpenRouter reasoning payloads, but the requested reasoning-aware benchmark/reporting schema, extraction-source telemetry, redaction policy, and explicit opt-in runtime/profile contract are still not present.

## What This Changes

Request support for native reasoning-model outputs by recording and optionally extracting `message.reasoning` separately from visible `message.content`, enforcing visible final answers, and reporting split benchmark compatibility scores.

## Best Possible Solution

Add a redacted reporting-first measurement layer with extraction-source fields and compatibility counters, then introduce explicit provider-profile/runtime opt-ins for reasoning-field extraction and visible-content enforcement while keeping normal visible-answer handling conservative.

## Reproduction Assessment

Source-reproducible for the core behavior: current parsing preserves provider reasoning as thinking and visible extraction ignores thinking, so `content:null` reasoning-only turns remain non-visible unless a retry or explicit policy intervenes. The exact benchmark score split was not rerun because the referenced local artifacts are not in the repository.

## Solution Assessment

Unclear as a complete implementation because there is no patch and the requested behavior spans reporting schema, redaction, provider profiles, and runtime routing. The safest direction is to add reporting-only instrumentation and redacted fixtures before enabling opt-in runtime extraction from reasoning fields.
