## Summary

Keep open. Current main has adjacent runtime support for reasoning-field ingestion, reasoning-only fallback, and vLLM/Nemotron request shaping, but it still does not implement the requested benchmark/report schema, extraction-source counters, separate reasoning-aware versus visible-content scores, or first-class profile policy knobs. The reporter also agreed this should remain separate from the narrower #74017 scorer bug.

## What This Changes

Add reasoning-aware response handling and reporting for native reasoning models, including content-vs-reasoning extraction sources, visible-content compatibility scoring, profile policy knobs, and optional visible final-answer enforcement.

## Best Possible Solution

Keep this as a maintainer design item. A good first slice would be reporting-only instrumentation for content/reasoning presence and extraction source, with counts and separate visible-content versus reasoning-aware versus strict-content score breakdowns, while defaulting away from raw reasoning retention until redaction policy is documented.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/reporting request. There is a high-confidence current-main verification path for the missing surfaces through exact field search and code/docs inspection, while the benchmark percentages require reporter fixtures or regenerated data.

## Solution Assessment

Yes, keeping this open for design is the best current handling. A narrow automated fix would be unsafe because the remaining work crosses benchmark/report schema, model/profile contracts, runtime routing policy, and reasoning-data retention rules.
