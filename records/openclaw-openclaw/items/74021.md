## Summary

Keep open. Current main has adjacent reasoning-field ingestion, vLLM/Nemotron request shaping, and reasoning-only fallback handling, but it still lacks the requested benchmark/report schema, extraction-source counters, separate reasoning-aware versus visible-content scores, and first-class profile/routing policy. The issue discussion also confirms this is intentionally broader than the narrower scorer bug in #74017.

## What This Changes

Request to make OpenClaw and `oc llm` explicitly handle native reasoning-model outputs by recording reasoning fields, distinguishing visible-content compatibility from reasoning-aware quality, and adding opt-in profile/reporting policy.

## Best Possible Solution

Keep this as a maintainer design item. The safest first slice is reporting-only instrumentation for content/reasoning presence, extraction source, and separate visible-content versus reasoning-aware versus strict-content score breakdowns, with raw reasoning disabled by default until redaction and retention policy is documented.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/reporting request. There is a high-confidence current-main verification path via targeted field search and code/docs inspection, while the benchmark percentages require the reporter’s fixtures or regenerated data.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current handling. The requested direction is plausible, but the implementation should start with measurement and policy-safe reporting before runtime routing/profile behavior relies on reasoning fields.
