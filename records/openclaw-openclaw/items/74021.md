## Label Justifications

- `P2`: This is a real compatibility and reporting gap for reasoning-capable model paths, but it needs product/security design rather than an emergency runtime patch.
- `impact:message-loss`: Reasoning-only `content:null` responses can leave users with no visible assistant reply or trigger fallback instead of delivering the useful answer.

- `impact:auth-provider`: The requested solution involves provider/model reasoning payload policy, profile fields, and routing/fallback behavior.

## Summary

Keep open: current main has important partial mitigations for reasoning-only turns, OpenRouter reasoning payloads, and legacy MiMo final-answer promotion, but it still lacks the requested reporting schema, extraction-source telemetry, redacted reasoning-field policy, and explicit opt-in runtime/profile contract.

## What This Changes

The issue requests opt-in reasoning-field extraction, visible-content enforcement, and benchmark/reporting telemetry for native reasoning model outputs.

## Best Possible Solution

Add redacted reporting-first instrumentation and fixtures for extraction source and compatibility counters, then design explicit provider-profile/runtime opt-ins for reasoning-field extraction and visible-content enforcement.

## Reproduction Assessment

Yes at source level for the core compatibility gap: current main preserves provider reasoning as thinking while visible extraction ignores it and retry/fallback handles reasoning-only turns. The exact benchmark score split and latest OpenRouter models were not live-reproduced because the artifacts and credentials are external.

## Solution Assessment

Unclear as a complete implementation because there is no patch and the request spans reporting schema, redaction, provider profiles, and runtime routing. The safest direction is reporting-only instrumentation with redacted fixtures before any opt-in runtime extraction from reasoning fields.
