## Summary

Keep open. Current main has adjacent reasoning-field stream parsing, vLLM/Nemotron request shaping, and reasoning-only fallback handling, but it still does not implement the requested benchmark/report schema, extraction-source counters, separate reasoning-aware versus visible-content scores, or first-class model/profile routing policy. The discussion also confirms this is intentionally broader than the narrower scorer bug in #74017.

## What This Changes

Request reasoning-aware benchmark/reporting and runtime/profile controls that distinguish visible-content compatibility from reasoning-field final-answer extraction for native reasoning models.

## Best Possible Solution

Keep this as a maintainer design item. The safest first slice is reporting-only instrumentation for content/reasoning presence, extraction source, and split visible-content versus reasoning-aware versus strict-content scores, with raw reasoning disabled by default until redaction and retention policy are documented.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/reporting request. There is a high-confidence current-main verification path through targeted search and code inspection, while the exact benchmark percentages require the reporter's redacted fixtures or regenerated data.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current handling. A narrow automated patch would be premature until the report schema, redaction defaults, and runtime/profile routing semantics are decided.
