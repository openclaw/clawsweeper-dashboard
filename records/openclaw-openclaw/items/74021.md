## Summary

Keep open. Current main has adjacent reasoning-field ingestion, reasoning-only fallback classification, and vLLM/Nemotron request shaping, but it still does not provide the requested benchmark/report schema, extraction-source counters, separate reasoning-aware versus visible-content scoring, or first-class profile/routing policy knobs. The issue discussion also confirms this should remain broader than the narrower #74017 scorer bug.

## What This Changes

The issue requests reasoning-aware handling and reporting for native reasoning model outputs, including opt-in extraction from `message.reasoning`, visible-content compatibility scoring, extraction-source metadata, and profile/runtime policy controls.

## Best Possible Solution

Keep this as a maintainer design item. A safe first slice would add reporting-only metadata and counters for content/reasoning presence plus separate visible-content, strict-content, and reasoning-aware score breakdowns, while leaving raw reasoning capture off by default until retention/redaction rules are documented.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/reporting request. There is a high-confidence source-verification path showing the requested fields and scoring modes are missing, while the exact benchmark percentages require the reporter's local fixtures or regenerated data.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current handling. The safer implementation path is staged reporting metadata first, then explicit profile/runtime policy once the privacy and routing semantics are settled.
