## Label Justifications

- `P2`: The issue describes a bounded but user-visible cron reliability problem that can create false alerts and suppress normal delivery.
- `impact:session-state`: The persisted isolated-run terminal state and consecutive error accounting can disagree with the agent's completed work.
- `impact:message-loss`: A false error state can produce `not-delivered` output and a misleading failure notification instead of the intended result.

## Summary

Current release and main already improve recovered tool-warning handling, but this report still contains a distinct typed LLM-abort path and a concrete diagnostic-context gap, so it is not fully implemented or safely reducible to the older recovered-payload issues.

## What This Changes

Correct isolated cron false failures after successful completion and expose the actual tool or model failure context in run diagnostics.

## Best Possible Solution

Define and test the canonical terminal-outcome boundary: keep typed aborts and timeouts fatal, mark only explicitly recovered tool failures successful, and carry structured failure kind, tool name, code, and original message into cron diagnostics.

## Reproduction Assessment

Partly yes. Source inspection gives a high-confidence path for lossy failure-signal diagnostics and shows how typed run errors become cron failures, but the report does not include the raw terminal metadata needed to reproduce Mode B exactly on current main.

## Solution Assessment

No, not as proposed. Treating every `stopReason: stop` run as successful could mask a genuine mid-run provider abort; the better fix is structured terminal-state precedence plus preserved causal diagnostics.
