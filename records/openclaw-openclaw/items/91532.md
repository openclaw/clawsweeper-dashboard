## Label Justifications

- `P2`: This is a normal-priority cron reliability bug with limited blast radius but real operator impact for isolated scheduled runs.
- `impact:session-state`: The reported failure is a mismatch between the isolated agent session terminal success state and the persisted cron run status, diagnostics, and error counter.
- `impact:message-loss`: The false error status suppresses delivery of the final assistant result and sends a misleading failure notification instead.

## Summary

Keep open: current main and v2026.6.6 still have cron isolated-agent paths where generic or non-`⚠️ 🛠️` error payloads can beat terminal assistant-visible success, and the latest live repro points to both classifier and diagnostics/finalization propagation rather than a fully implemented fix.

## What This Changes

The issue asks OpenClaw to stop marking successful isolated cron agent turns as failed when recoverable tool-level errors occur mid-run, and to improve the diagnostics behind cron failure notifications.

## Best Possible Solution

Land or replace the focused cron isolated-agent fix so terminal successful completion can recover eligible tool errors across both payload classification and diagnostic finalization, while keeping typed fatal signals, real run-level errors, denial markers, and unrecovered trailing errors fatal.

## Reproduction Assessment

Yes, at source level: current main still has an all-error-payload plus terminal assistant text case that resolves to the last error, and finalization persists that fatal classifier result as cron error status. I did not run a private live cron repro in this read-only review.

## Solution Assessment

Unclear. The resolver-level PR is directionally right, but the latest repro shows the best fix also needs maintainer-approved handling for diagnostics/finalization so recovered terminal assistant output is not overwritten by tool and agent-run error diagnostics.
