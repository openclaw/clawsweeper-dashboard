## Summary

Keep open: current main still contains source-backed paths where isolated cron finalization can treat recovered tool/run diagnostics as fatal, record an error, and suppress the final assistant output. The related open fixes and sibling reports overlap, but none is a merged or clean canonical solution for the broader failure described here.

## What This Changes

The issue requests that isolated cron agentTurn runs preserve successful final assistant completion instead of letting recovered tool-level errors drive cron error status, diagnostics, delivery failure, and consecutive-error state.

## Best Possible Solution

Define one cron finalization contract that separates recovered tool diagnostics, true fatal run failures, final assistant output, and delivery/finalization failure phases, then land one reviewed fix that preserves that contract across the classifier, diagnostics, and run-state write path.

## Reproduction Assessment

Yes, at source level: current main still maps recovered/fatal classifier output into cron error state, and tests preserve an all-error-payload case where final assistant text does not win. The live issue comments add concrete scheduled-run evidence, but I did not run a live cron daemon repro in this read-only review.

## Solution Assessment

Unclear as a final implementation: the issue is valid, but the best fix needs maintainer scoping across classifier recovery, diagnostics, delivery dispatch, and true fatal error preservation. The existing open PRs are partial candidates rather than a merged canonical answer.
