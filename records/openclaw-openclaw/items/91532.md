## Label Justifications

- `P2`: The issue affects cron status, failure notifications, and monitoring trust for isolated agent jobs, but it is bounded to cron/session delivery behavior.
- `impact:session-state`: The reported bug persists the wrong cron run status, diagnostics, and consecutive error state after a session that completed successfully.
- `impact:message-loss`: The latest repro says the successful final assistant text was not delivered and a failure notification was sent instead.

## Summary

Keep open: current main and v2026.6.6 still contain the recovered tool-error classifier and diagnostics paths that can report an isolated cron run as failed after terminal assistant success. The related fix PR at https://github.com/openclaw/openclaw/pull/84156 is open, conflicting, and only owns part of the problem, while this issue now has newer live repro evidence for the separate `agent-run` diagnostic latch.

## What This Changes

The issue asks OpenClaw to stop marking successful isolated cron `agentTurn` runs as failed because of recovered tool-level errors, generic agent-run diagnostics, or misleading failure delivery state.

## Best Possible Solution

Define the isolated cron recovered-error status contract, then land https://github.com/openclaw/openclaw/pull/84156 or a follow-up that also covers `agent-run` diagnostics/meta-error latching while preserving true fatal run and delivery failures.

## Reproduction Assessment

Yes for the source-level recovered tool-error path: current tests and helper logic still show all-error payloads plus terminal assistant text selecting the last error. The exact private long-running Windows/opencode and June 13 live trajectories were not locally replayed.

## Solution Assessment

Unclear as a final product contract. A resolver-level fix is necessary, but the newest repro suggests the diagnostics/meta-error path also needs scoping so recovered tool errors do not stay fatal while true aborts, denials, timeouts, and delivery failures remain failures.
