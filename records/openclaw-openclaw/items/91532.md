## Summary

Keep open: current main has partial recovered-tool-warning handling, but the latest live repro uses an apply-patch warning shape and duplicated agent-run diagnostics that are not clearly solved by main or by the open related PR.

## What This Changes

The issue asks OpenClaw to stop marking successful isolated cron agent runs as failed when recoverable tool-level errors or post-run diagnostics remain after a successful terminal assistant result.

## Best Possible Solution

Define the recovered-tool-error status contract for isolated cron runs, then update the cron classifier and diagnostics/finalization path so recovered tool failures remain warnings while true run-level fatal signals still persist as errors.

## Reproduction Assessment

Yes from source for the remaining recovered-tool shape: current main still keeps generic all-error payloads fatal despite terminal assistant-visible text, and apply-patch warnings are outside the recognized `⚠️ 🛠️` recovery prefix. I did not run a private live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the exact status contract. The open related PR is directionally relevant for payload recovery, but the latest live repro indicates diagnostics/finalization also need to stop latching recovered tool failures into delivery and failure-notification state.
