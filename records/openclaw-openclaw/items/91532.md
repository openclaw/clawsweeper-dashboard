## Best Possible Solution

Keep this issue open while maintainers reconcile the open resolver PR with the newer diagnostic and delivery-state evidence, then land one scoped cron status/diagnostics contract that preserves true run-level failures.

Yes for a source-level subset, but not for the exact private live trajectories. Current main still has a helper test where all-error payloads plus terminal assistant text resolve to the last error, and the finalizer maps that classifier result into persisted cron status.

## Solution Assessment

Unclear. The narrow resolver fix is the right layer for recovered tool errors, but a blanket `stopReason: "stop"` override is not the best fix because docs and code intentionally keep true run-level model/provider failures fatal.
