## Summary

Keep open. Current main still hashes Claude token-shaped credentials and token auth-profile token material into the CLI auth epoch, so token rotation or a transient token-shaped parse can still reset same-version CLI sessions; open PR #74493 directly targets this issue and should be reviewed before closure.

## What This Changes

The issue requests making Claude CLI auth-epoch hashing identity-based so rotating OAuth/token material does not reset live sessions mid-conversation.

## Best Possible Solution

Review and land #74493 or an equivalent narrow patch that makes Claude token-shaped epochs identity-based, removes raw token material from token auth-profile epoch hashing where appropriate, bumps the auth-epoch version, and preserves regression coverage for token rotation stability plus real identity changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to have readClaudeCliCredentialsCached return a Claude CLI credential with type "token", change only credential.token, and compare resolveCliAuthEpoch; current main changes the epoch, and same-version epoch mismatches reset sessions with reason "auth-epoch".

## Solution Assessment

Yes. Identity-only hashing at the auth-epoch layer is the narrow maintainable fix for the reset loop; parser hardening can remain a separate defense-in-depth follow-up after the main token-epoch contract is corrected.
