## Summary

Keep open. Current main and the latest tagged release still hash Claude token-shaped credential material into the CLI auth epoch, and the linked open PR #74493 is the active implementation path for this bug; this issue should close only after that PR, or an equivalent fix, lands.

## What This Changes

Make Claude CLI auth-epoch hashing stable across OAuth token rotation so active CLI sessions are not reset mid-conversation.

## Best Possible Solution

Review and land #74493, or an equivalent narrow patch, so Claude token-shaped local credentials and token auth-profile credentials use stable identity fields for the auth epoch, the auth-epoch version is bumped for migration safety, and regression tests prove token rotation stability plus real identity-change invalidation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to stub readClaudeCliCredentialsCached to return a Claude CLI credential with type "token", change only credential.token, and compare resolveCliAuthEpoch; current main changes the epoch and same-version session reuse then resets with reason "auth-epoch".

## Solution Assessment

Yes. Identity-only hashing at the auth-epoch layer is the narrow maintainable fix for the reset loop; parser hardening for partial keychain reads can remain a defense-in-depth follow-up after the epoch contract is corrected.
