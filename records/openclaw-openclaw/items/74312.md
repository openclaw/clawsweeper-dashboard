## Summary

Keep open. Current main still contains the reported auth-epoch behavior: Claude token-shaped credentials and token auth-profile credentials include rotating token material in the epoch input, and same-version epoch mismatches still invalidate CLI session reuse. PR #74493 is an open, unmerged implementation with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to keep Claude CLI auth epochs stable across OAuth token rotation or transient token-shaped keychain reads so live conversations are not reset unless the underlying identity changes.

## Best Possible Solution

Review and land PR #74493, or an equivalent narrow patch, so Claude token-shaped epochs and token auth-profile epochs are keyed on stable identity fields, the auth-epoch version is bumped for migration safety, and regression tests cover token rotation stability plus real identity changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to stub `readClaudeCliCredentialsCached` with a Claude `type: "token"` credential, change only `credential.token`, and compare `resolveCliAuthEpoch`; current main changes the epoch, and `resolveCliSessionReuse` then resets same-version bindings with `reason=auth-epoch`.

## Solution Assessment

Yes. Identity-only hashing at the auth-epoch layer plus an epoch-version bump is the narrow maintainable fix for the observed reset loop; parser hardening for partial keychain reads can remain a defense-in-depth follow-up.
