## Summary

Keep open: current main still reaches direct `net.connect()` for debug-proxy CONNECT requests, and this draft maintainer PR is the active fix path, but the branch still has a concrete test-fixture hygiene issue around generated CA key material.

## What This Changes

The PR adds a managed-proxy CONNECT guard and diagnostics override to the debug proxy, plus regression coverage, docs, and changelog notes.

## Best Possible Solution

Keep the CONNECT guard, docs, and changelog direction, but move the regression fixture to a cleaned OS temp directory before maintainer approval.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, set `OPENCLAW_PROXY_ACTIVE=1`, send an HTTP CONNECT request to the debug proxy, and the handler reaches `net.connect()` without a managed-proxy guard.

## Solution Assessment

No, not quite as submitted. The production guard is at the narrow maintainable boundary before `net.connect()`, but the new test fixture should use `os.tmpdir()`/`mkdtemp` so generated CA key material is never written under the checkout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep generated CA files out of the checkout:** `src/proxy-capture/proxy-server.managed-proxy.test.ts:7`
  - body: Even with `afterEach` cleanup, `startDebugProxyServer()` calls `ensureDebugProxyCa()` before the rejection path, and this repo-relative `certDir` causes `root-ca.pem` and `root-ca-key.pem` to be created under the checkout while the test runs. If the test worker is interrupted or crashes, generated CA private-key material is left in the working tree. Please create the fixture under `os.tmpdir()` with `mkdtemp` and remove that root in cleanup.
  - confidence: 0.88
