## Summary

Keep open: current main still runs MiniMax music generation as one direct POST and the tests only cover immediate audio output, but the proposed polling endpoint needs MiniMax contract or live proof before this is a safe automated repair.

## What This Changes

Request MiniMax music generation to use short async task creation plus status polling instead of one long blocking HTTP request for full-length tracks.

## Best Possible Solution

Move MiniMax music to a documented short-request/poll/download flow if MiniMax confirms that contract; otherwise keep using the documented endpoint and fix timeout behavior without inventing an unsupported API.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection confirms the blocking MiniMax music request path, but the exact timeout failure was not reproduced against MiniMax and the tool-level default timeout is currently 300000ms.

## Solution Assessment

Unclear until the MiniMax music polling contract is confirmed. The existing MiniMax video polling pattern is the right shape if the endpoint exists, but dependency-backed provider behavior should not be implemented from the issue text alone.
