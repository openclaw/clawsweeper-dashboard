## Summary

Keep open: current main still uses a single MiniMax music-generation POST and has no polling coverage, while the proposed `/v1/query/music_generation` contract still needs vendor confirmation or live proof before a safe fix PR.

## What This Changes

The issue requests changing the MiniMax music-generation provider from one blocking `/v1/music_generation` request to a short-request async task polling flow.

## Best Possible Solution

Confirm the MiniMax music async contract with official docs or a redacted live run, then implement provider-local poll/download handling with operation-deadline helpers and focused tests; otherwise keep the provider aligned to the documented synchronous contract.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection confirms the single blocking MiniMax music request path, but the timeout failure was not reproduced against MiniMax and the tool-level default timeout is currently 300000ms.

## Solution Assessment

Unclear until the MiniMax music polling contract is confirmed. The existing MiniMax video polling implementation is the right local pattern if the endpoint exists, but provider behavior should not be changed from the issue text alone.
