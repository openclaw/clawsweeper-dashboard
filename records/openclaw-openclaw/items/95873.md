## Summary

Keep open: current main still has the source-visible attribution gap on timer-built isolated cron timeout/cancel outcomes, and the active closing PR is open but not merged.

## What This Changes

The issue asks cron run logs to preserve already-known session, provider, and model metadata when an isolated agent run fails after that identity has been resolved.

## Best Possible Solution

Merge a focused fix that carries existing execution metadata into timer-built timeout/cancel outcomes before cron run-log persistence, while leaving true pre-runner failures unattributed when no run identity exists.

## Reproduction Assessment

Yes. Current-main source shows `runCronIsolatedAgentTurn` can report session/provider/model through execution callbacks, while `executeJobCoreWithTimeout` returns timer-owned timeout/cancel outcomes without those fields and the run-log writer persists only the returned metadata.

## Solution Assessment

Yes, the active fix direction is the right layer because the timer owns the synthetic race outcomes. Recovering metadata later in the SQLite writer would be too late because the event never receives it.
