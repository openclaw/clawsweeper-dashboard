## Summary

Keep open: current main still has a source-proven path where the cron timeout wrapper knows the active session/provider/model through the watchdog but returns an error outcome without those fields, so the finished cron log row can be un-attributed as reported.

## What This Changes

The issue reports that `cron_run_logs` can lose `session_id`, provider, and model attribution when an isolated cron agent run times out after execution has already resolved runtime metadata.

## Best Possible Solution

Preserve the watchdog's already-known execution metadata on timeout outcomes before `emitJobFinished`, then add focused regression coverage proving failed cron timeout logs retain session, provider, and model.

## Reproduction Assessment

Yes from source inspection, not from a live run in this read-only pass. The timeout wrapper returns an error outcome without copying the watchdog's known session/provider/model fields, while the run-log path only persists those fields when they are present on the result.

## Solution Assessment

Yes, the narrowest maintainable fix is to carry already-collected watchdog execution metadata into the timeout outcome. Changing the run-log schema or folding in the separate Codex missing-tool-result bug would be the wrong layer for this issue.

## Repair Work Prompt

Fix the cron timeout observability bug reported in https://github.com/openclaw/openclaw/issues/95873. In current main, `executeJobCoreWithTimeout` reads `watchdog.activeExecution()` on timeout but returns an error `CronCoreRunOutcome` without `sessionId`, `sessionKey`, `provider`, or `model`, so `emitJobFinished` and `appendCronRunLog` can persist blank cron_run_logs metadata. Preserve already-known active execution metadata on timeout outcomes without changing the `cron_run_logs` schema, without adding config, and without changing non-timeout success/error behavior. Add focused regression coverage for an isolated cron agent timeout after execution metadata is reported, proving the finished event or persisted run-log entry includes session id, provider, and model. Keep the fix within cron service/isolated-agent test boundaries and do not touch CHANGELOG.md.
