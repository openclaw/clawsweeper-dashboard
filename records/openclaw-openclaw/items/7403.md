## Summary

Keep open: current main and v2026.6.10 still lack an enforceable Private/Secret/Incognito session contract; existing context-injection controls can reduce bootstrap context but do not prevent durable session metadata, JSONL transcript creation, memory/compaction/plugin persistence, or UI/channel enforcement.

## What This Changes

Request to add a Private Mode for Mission Control and sessions that suppresses workspace context and prevents private session memory/history from touching durable workspace or session storage.

## Best Possible Solution

Keep this as the canonical tracker and design one additive private/ephemeral session contract across Gateway protocol, SessionEntry/session store/transcripts, bootstrap/memory/compaction/plugins, channel triggers, and Control UI, with tests proving no private-context reads or durable history writes.

## Reproduction Assessment

Yes by source inspection: current session schemas and creation paths have no private/no-persist field and still create durable session entries plus JSONL transcript headers. I did not run a live private-mode session because there is no private/session-persist flag to exercise on current main.

## Solution Assessment

Yes for triage: keeping this open is the best current outcome because prompt-only, config-only, or UI-only changes would falsely promise privacy without a cross-layer enforcement contract. The actual implementation needs maintainer-approved security and product design before code work.
