## Summary

Keep open: current main still emits raw `llm_output` once per embedded attempt with aggregate usage. The existing `model_call_started` / `model_call_ended` hooks provide sanitized per-call metadata, but not the requested per-call output/usage contract, and the overlapping hook PRs still need API and security review.

## What This Changes

The issue requests per-LLM-call plugin observability for agentic loops, either by per-iteration `llm_output` events or a new `llm_call` hook carrying per-call usage and run/session/model metadata.

## Best Possible Solution

Ship one additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility and gates raw conversation data through the existing trusted-plugin access policy.

## Reproduction Assessment

Yes, at source level. Current main records assistant-message usage on each `message_end`, accumulates it into `usageTotals`, and emits one `llm_output` later with `attemptUsage = getUsageTotals()`; I did not run a live multi-call LLM loop in this read-only review.

## Solution Assessment

Unclear. The observability gap is real, but changing existing `llm_output` semantics in place could break aggregate consumers, so an additive gated hook or explicit aggregate marker needs maintainer API judgment.
