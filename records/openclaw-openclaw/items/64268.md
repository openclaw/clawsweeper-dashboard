## Summary

Keep open: current main still records per-assistant-message usage but emits raw `llm_output` once at the embedded-attempt boundary with aggregate usage. The shipped `model_call_started` / `model_call_ended` hooks cover sanitized per-call telemetry, and related context/timing issues are closed, but the requested per-call raw output/usage contract still needs a public plugin API and access-policy decision.

## What This Changes

Expose per-LLM-call plugin output and usage events in agentic loops instead of only one aggregate `llm_output` at the end of a turn.

## Best Possible Solution

Ship an additive, documented, access-gated per-call observability contract that preserves aggregate `llm_output` compatibility and keeps sanitized `model_call_*` telemetry as the safe default.

## Reproduction Assessment

Yes. Source inspection on current main shows per-assistant-message usage is recorded and accumulated, while raw `llm_output` still dispatches once later with aggregate `getUsageTotals()` data.

## Solution Assessment

Unclear. The observability gap is real, but changing existing `llm_output` semantics in-place is not the safest path; maintainers should approve an additive, gated per-call contract that preserves aggregate consumers.
