## Summary

Keep open: current main still dispatches raw `llm_output` once per embedded attempt with aggregate usage. The related context-window and latency requests are covered by current metadata/sanitized model-call hooks, but the requested per-call raw usage/output contract remains an API and security-boundary decision with an open implementation PR still under review.

## What This Changes

Request additive plugin observability so each LLM call in an agentic turn exposes per-call usage/output metadata instead of only one aggregate `llm_output` at turn end.

## Best Possible Solution

Ship one maintainer-approved, documented additive per-call observability contract that preserves aggregate `llm_output` compatibility, keeps sanitized `model_call_*` telemetry as the default path, and gates any raw payload access through the plugin conversation-access policy.

## Reproduction Assessment

Yes. Source inspection shows per-assistant-message usage is recorded during `message_end`, while the raw `llm_output` hook is still dispatched once later with aggregate `getUsageTotals()` data.

## Solution Assessment

Unclear as a product/API choice. The gap is real, but changing `llm_output` semantics in place is not the safest fix; an additive, gated per-call contract is the maintainable direction.
