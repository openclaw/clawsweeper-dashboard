## Summary

Keep open: current main still records per-assistant-message usage into aggregate attempt totals and emits one raw `llm_output` at the attempt boundary, while the related implementation PR remains open with API-contract review blockers.

## What This Changes

The issue requests per-LLM-call plugin observability for agentic loops instead of only an aggregate end-of-turn `llm_output` event.

## Best Possible Solution

Ship an additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility, existing context metadata, and the raw conversation access gate.

## Reproduction Assessment

Yes. Source inspection on current main shows each assistant `message_end` can record and commit usage, while the only raw `llm_output` dispatch occurs later with aggregate `getUsageTotals()` data.

## Solution Assessment

Unclear. The requested direction is valid, but the safest solution needs maintainer API-contract judgment because the open PR’s approach changes existing aggregate hook semantics and metadata compatibility.
