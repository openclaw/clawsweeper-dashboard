## Summary

Keep open: current main still accumulates assistant usage snapshots and emits one attempt-level `llm_output`; there is no `llm_call` hook or per-call raw output payload, and the related implementation PR #45990 remains open.

## What This Changes

The issue requests per-LLM-call observability from agentic loops instead of only one aggregate `llm_output` event per turn.

## Best Possible Solution

Ship an additive, documented per-call observability contract in the embedded runner/subscription path that preserves the raw conversation access gate and clearly separates per-call data from aggregate turn totals.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: multiple assistant `message_end` events can each commit usage into `usageTotals`, while `runEmbeddedAttempt` later emits a single `llm_output` using `getUsageTotals()` rather than per-call usage.

## Solution Assessment

Unclear. Current main is not a solution; #45990 is the open implementation candidate, but maintainers still need to choose the public hook semantics and compatibility story before this should be considered resolved.
