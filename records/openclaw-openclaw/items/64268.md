## Summary

Keep open: current main still records per-assistant-message usage into accumulated attempt totals and emits one attempt-level `llm_output`; there is no `llm_call` hook or per-call raw output event, and the related open implementation PR at https://github.com/openclaw/openclaw/pull/45990 still needs contract review.

## What This Changes

The issue requests per-LLM-call observability for agentic loops, either by emitting `llm_output` per iteration with aggregate marking or by adding a new `llm_call` hook.

## Best Possible Solution

Ship an additive, documented per-call observability contract in the embedded runner/subscription path that preserves the raw conversation access gate and clearly separates per-call data from aggregate turn totals.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: each assistant `message_end` can commit usage into `usageTotals`, while `runEmbeddedAttempt` later emits one `llm_output` using `getUsageTotals()` rather than per-call usage.

## Solution Assessment

Unclear. Current main is not a solution, and the related open PR is a plausible implementation candidate, but maintainers still need to settle the public hook semantics and compatibility story before this is resolved.
