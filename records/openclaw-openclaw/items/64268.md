## Summary

Keep open. Current main still emits `llm_output` only once at embedded-attempt completion with accumulated usage, while per-assistant-message handling records usage without emitting a per-call raw output hook; the related PR #45990 is open and unmerged, so this is not shipped behavior yet.

## What This Changes

Add per-LLM-call observability for agentic loops by emitting per-iteration `llm_output` or a new `llm_call` hook with call-specific usage, provider/model, run/session/agent identifiers, and ideally tool-call data.

## Best Possible Solution

Ship an additive, documented per-call observability contract in the embedded runner/subscription path that preserves or clearly labels existing aggregate semantics, exposes normalized per-call usage with provider/model/run/session/agent identifiers, and includes focused multi-round tests plus plugin hook docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is available by inspection and focused test harnessing: emit multiple assistant `message_end` events with usage, then observe that subscription code accumulates usage while `runEmbeddedAttempt` emits one final `llm_output` from `getUsageTotals()`.

## Solution Assessment

Unclear until maintainers choose the public API shape. A new `llm_call` hook or explicitly marked per-call `llm_output` events would be maintainable if existing aggregate semantics and the raw conversation-access gate are preserved; current main is not sufficient, and #45990 needs revision before it is the best fix.
