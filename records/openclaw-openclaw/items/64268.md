## Summary

Keep open: current main still records per-assistant-message usage into aggregate attempt totals and emits one raw `llm_output` at the attempt boundary. The related implementation PR is still open, conflicting, and has API-contract review blockers.

## What This Changes

The issue requests per-LLM-call `llm_output` or equivalent hook events during agentic loops instead of only one aggregate event per turn.

## Best Possible Solution

Ship an additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility, existing context metadata, and the raw conversation access gate.

## Reproduction Assessment

Yes. Source inspection on current main shows each assistant `message_end` can record and commit usage, while the only raw `llm_output` dispatch occurs later with aggregate `getUsageTotals()` data.

## Solution Assessment

Unclear. The requested direction is valid, but the safest solution needs maintainer API-contract judgment because the open PR changes existing aggregate hook semantics and metadata compatibility.
