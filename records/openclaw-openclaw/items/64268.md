## Summary

Keep open: current main still records per-assistant-message usage but dispatches raw `llm_output` only once from the embedded attempt boundary with aggregate usage. Related work has solved context metadata and sanitized per-call timing, but the requested per-call raw output/usage contract remains an API decision with open implementation PRs still needing review.

## What This Changes

Request an additive plugin hook/API that exposes per-LLM-call output and usage metadata during agentic loops instead of only the final aggregate.

## Best Possible Solution

Ship an additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility, context metadata, and raw conversation access gating.

## Reproduction Assessment

Yes. Source inspection on current main shows each assistant `message_end` can record and commit usage, while the only raw `llm_output` dispatch occurs later with aggregate `getUsageTotals()` data.

## Solution Assessment

Unclear. The requested observability gap is real, but the narrowest maintainable fix needs maintainer API-contract judgment because changing existing `llm_output` semantics could break aggregate consumers.
