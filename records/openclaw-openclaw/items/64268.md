## Summary

Keep open: current main still records and aggregates per-assistant usage, then emits a single `llm_output` at embedded-attempt completion. The related implementation PR #45990 is open and unmerged, so no shipped or current-main behavior satisfies the per-call hook request yet.

## What This Changes

The issue requests per-LLM-call observability for agentic loops, including per-call usage plus provider, model, run, session, and agent identifiers, either via per-iteration `llm_output` events or a new `llm_call` hook.

## Best Possible Solution

Ship an additive, documented per-call observability contract in the embedded runner/subscription path that preserves or clearly labels existing aggregate semantics, exposes normalized per-call usage with provider/model/run/session/agent identifiers, and adds focused multi-round tests plus plugin hook docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is available by inspection and focused harnessing: emit multiple assistant `message_end` events with usage, observe `commitAssistantUsage()` accumulating totals, then observe `runEmbeddedAttempt` emitting one final `llm_output` from `getUsageTotals()`.

## Solution Assessment

Unclear until maintainers choose the public contract shape. A new `llm_call` hook or explicitly marked per-call `llm_output` events would be maintainable if the existing aggregate behavior and raw conversation-access gate are preserved; current main does not solve the issue.
