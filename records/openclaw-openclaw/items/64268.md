## Summary

Keep open: current main still exposes only aggregate raw `llm_output` for the embedded attempt, while the related implementation PR remains open, conflicting, and has API-contract blockers.

## What This Changes

Add per-LLM-call raw output observability so plugins can see usage and metadata for each agentic loop iteration instead of only final turn totals.

## Best Possible Solution

Ship an additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility, context metadata, and raw conversation access gating.

## Reproduction Assessment

Yes. Source inspection on current main shows each assistant `message_end` can record and commit usage, while the only raw `llm_output` dispatch occurs later with aggregate `getUsageTotals()` data.

## Solution Assessment

Unclear. The requested observability gap is real, but the safest fix needs maintainer API-contract judgment because the open PR changes existing aggregate hook semantics and metadata compatibility.
