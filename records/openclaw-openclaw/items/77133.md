## Summary

Close as superseded: #77154 is the open maintainer replacement for the same trajectory flush problem, explicitly says it replaces this PR, and covers the gaps in this smaller patch.

## What This Changes

This PR adds a queued writer setImmediate yield, lowers the trajectory runtime file cap from 50 MiB to 10 MiB, and raises the agent cleanup step timeout from 10s to 30s.

## Best Possible Solution

Close this PR in favor of #77154, then review and land that bounded recorder/writer fix while leaving broader sessions.list latency follow-up tracked separately.

## Reproduction Assessment

Yes, at source level: current main has a single queued writer chain with no yield/queue cap and cleanup timeout handling that leaves the flush promise running after 10s. I did not run a live 50 MiB gateway benchmark in this read-only review.

## Solution Assessment

No, this smaller patch is not the best fix. #77154 is safer because it bounds payload shaping before stringify, caps pending queued bytes, records truncation, updates docs/changelog/tests, and keeps export compatibility for existing 50 MiB sidecars.
