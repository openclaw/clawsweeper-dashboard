## Summary

Keep open: the WebChat auto-TTS bug looks plausible from source, but this patch should not merge because it encodes TypeScript sources as UTF-16LE/binary, changes final-mode streaming semantics, skips media normalization too broadly, and lacks after-fix real behavior proof.

## What This Changes

The PR changes speech-core auto-TTS to trust generated local audio and allow block replies in final mode, and changes reply dispatch to skip media normalization for trusted local media payloads.

## Best Possible Solution

Keep the WebChat trust fix narrow: preserve trustedLocalMedia for the final synthetic TTS payload, keep final-mode streaming to one post-stream audio reply, and scope any normalizer change so non-WebChat channels still stage media correctly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live WebChat/MiniMax reproduction was run, but source inspection shows WebChat requires trustedLocalMedia and current auto-TTS output on main does not set it.

## Solution Assessment

Is this the best way to solve the issue? No; the patch targets the right area but should not bypass final-mode block gating or global media normalization, and it must keep the source files UTF-8.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the TypeScript sources UTF-8:** `extensions/speech-core/src/tts.ts:1`
  - body: The branch rewrites both changed `.ts` files as UTF-16LE, so GitHub shows binary patches and reviewers cannot inspect the actual source diff. Please re-save these files as normal UTF-8 text before this can be reviewed or merged.
  - confidence: 0.95
- **[P2] Do not synthesize final-mode TTS for every streamed block:** `extensions/speech-core/src/tts.ts:1762`
  - body: In `mode: "final"`, streamed block replies are accumulated and synthesized once after streaming completes. Letting `kind: "block"` through this gate makes auto-TTS run per block, which can fragment or duplicate audio instead of producing the documented single final audio reply.
  - confidence: 0.87
- **[P2] Do not bypass media normalization for every trusted payload:** `src/auto-reply/reply/dispatch-from-config.ts:595-596`
  - body: `trustedLocalMedia` is a WebChat embedding trust signal, but this skip runs in the shared dispatch path for all channels. Auto-TTS local files would avoid `createReplyMediaPathNormalizer`, so non-WebChat deliveries can miss the normal staging/normalization path.
  - confidence: 0.84
- **[P2] Carry the trust flag into the synthetic final TTS payload:** `src/auto-reply/reply/dispatch-from-config.ts:1660-1664`
  - body: The final-mode streamed reply path builds a fresh `ttsOnlyPayload` from `ttsSyntheticReply` but does not copy `trustedLocalMedia`. That means the intended WebChat final audio path still loses the trust flag before WebChat embedding checks it.
  - confidence: 0.8
