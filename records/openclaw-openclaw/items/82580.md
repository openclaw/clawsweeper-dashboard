## Summary

Keep open: the WebChat auto-TTS trust problem is source-reproducible, but this PR still has blocking correctness and security review findings and lacks after-fix real behavior proof; the narrower trustedLocalMedia portion is also already tracked by https://github.com/openclaw/openclaw/pull/81203.

## What This Changes

The branch changes speech-core auto-TTS to trust generated local audio and allow block replies in final mode, and changes shared reply dispatch to skip media normalization when trustedLocalMedia is true.

## Best Possible Solution

Consolidate on a narrow WebChat auto-TTS fix that preserves trustedLocalMedia on the final synthetic TTS payload, keeps final-mode streaming to one post-stream audio reply, preserves channel media normalization, repairs the mojibake, and includes focused regression coverage plus redacted real WebChat proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live WebChat/MiniMax run was performed, but source inspection shows WebChat requires trustedLocalMedia and current auto-TTS final-stream payload construction does not carry that flag into the WebChat embedding path.

## Solution Assessment

Is this the best way to solve the issue? No; the patch touches the right area but broadens final-mode block synthesis, bypasses shared media normalization, corrupts source text, and still drops the trust flag on the final synthetic payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restore the TTS max-length interpolation:** `extensions/speech-core/src/tts.ts:559`
  - body: The PR corrupts the prompt hint from `≤${maxLength}` to `鈮?{maxLength}`, so agents no longer receive the actual max length in the auto-TTS instruction. Please repair the string and any other mojibake introduced by the branch.
  - confidence: 0.93
- **[P2] Keep final-mode TTS out of streamed block delivery:** `extensions/speech-core/src/tts.ts:1762`
  - body: In `mode: "final"`, streamed blocks are accumulated and synthesized once after streaming completes. Allowing `kind: "block"` through this gate makes TTS run during each block and can fragment or duplicate audio instead of producing the intended single final audio reply.
  - confidence: 0.88
- **[P2] Do not bypass shared media normalization for trusted payloads:** `src/auto-reply/reply/dispatch-from-config.ts:595-596`
  - body: `trustedLocalMedia` is a WebChat embedding trust signal, but this early return runs in the shared dispatch path for every channel. Trusted TTS files would skip the normal staging and blocking behavior that non-WebChat deliveries still need.
  - confidence: 0.86
- **[P2] Propagate trust on the synthetic final TTS payload:** `src/auto-reply/reply/dispatch-from-config.ts:1660-1664`
  - body: The WebChat final-stream path builds a fresh `ttsOnlyPayload` from the TTS result but does not copy `trustedLocalMedia`. That means the intended final audio path can still reach WebChat as untrusted even after speech-core sets the flag.
  - confidence: 0.82
