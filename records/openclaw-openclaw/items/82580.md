## Summary

Keep open: the reported WebChat auto-TTS gap is plausible from source, but the current patch is not mergeable because it corrupts TTS prompt text, changes final-mode streaming into per-block TTS, bypasses shared media normalization too broadly, and still lacks real WebChat proof.

## What This Changes

The PR changes speech-core auto-TTS to set `trustedLocalMedia` and allow block replies in final mode, and changes shared reply dispatch to skip media normalization for trusted local media.

## Best Possible Solution

Preserve final-mode one-shot TTS, carry `trustedLocalMedia` through the synthetic final TTS payload for WebChat, and keep channel media staging for other deliveries with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live WebChat/MiniMax run was performed, but source inspection shows the clear failing path: WebChat rejects local audio without `trustedLocalMedia`, and current final synthetic TTS payloads do not carry it.

## Solution Assessment

Is this the best way to solve the issue? No; the patch touches the right surface, but the safer fix is to preserve trust on the final synthetic TTS payload without allowing per-block final-mode TTS or bypassing shared media normalization globally.
