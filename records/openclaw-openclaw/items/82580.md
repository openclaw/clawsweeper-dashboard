## Summary

Keep open: the reported WebChat auto-TTS gap appears valid from source, but this PR is not mergeable because it corrupts source encoding, changes final-mode TTS into per-block synthesis, broadens a local-media trust bypass, and lacks after-fix real behavior proof.

## What This Changes

This PR changes speech-core TTS payload handling to allow final-mode TTS on block replies and add `trustedLocalMedia`, and changes auto-reply dispatch to skip media path normalization for trusted local media payloads.

## Best Possible Solution

Fix WebChat auto-TTS by preserving final-mode batching, carrying a narrowly scoped trusted marker through the TTS-only final payload or WebChat embed path, constraining local-media bypasses to OpenClaw-owned TTS temp output, and adding targeted regression coverage plus real WebChat proof.

## Reproduction Assessment

Yes at source level: current main calls auto-TTS with `kind: "block"` during streamed block delivery while final-mode TTS rejects non-final kinds, and WebChat refuses local audio without `trustedLocalMedia`. A live WebChat reproduction still needs artifact-backed proof.

## Solution Assessment

No. The PR should not allow final-mode TTS on every block or skip media normalization globally; the narrower fix is to preserve the final accumulated TTS path, propagate trust only for OpenClaw-generated TTS media, and keep local-media safety checks intact.

Constrain the bypass to OpenClaw-owned TTS temp outputs or preserve the flag after safe normalization.
  - confidence: 0.82
