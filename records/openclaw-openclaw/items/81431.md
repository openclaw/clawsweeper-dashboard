## What This Changes

The branch adds optional `filter`, `sort`, `field_names`, and `view_id` parameters to `feishu_bitable_list_records`, using `appTableRecord.search` for filter/sort requests and `list` for simple queries.

## Best Possible Solution

Land one corrected Feishu Bitable update that matches the chosen Lark list/search contracts, includes focused forwarding tests, reconciles the overlapping PRs, and adds redacted real behavior proof.

## Reproduction Assessment

Yes for the review findings: source inspection against the locked Lark SDK types and CI logs reproduce the TypeScript failures. The broader feature still lacks a live Feishu setup proof path.

## Solution Assessment

No. Passing supported server-side parameters through the Feishu plugin is the right narrow direction, but this patch needs the registered tool param type fixed and the filter schema aligned with either `list` string filters or `search` structured filters.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Declare the new list-record params in the executor type:** `extensions/feishu/src/bitable.ts:716-719`
  - body: The registered tool generic still only includes the old pagination params, so the new `params.filter`, `params.sort`, `params.field_names`, and `params.view_id` reads fail `tsgo:prod` and `check:test-types` with TS2339. Add the new fields to this type before the branch can compile.
  - confidence: 0.97
- **[P1] Match the search filter schema before switching endpoints:** `extensions/feishu/src/bitable.ts:215`
  - body: When `filter` is provided this calls `appTableRecord.search` and forwards the string expression as `data.filter`, but the locked Lark SDK defines search filters as structured `{ conjunction, conditions, children }` objects. Calls following the advertised string syntax fail typecheck and would not form a valid search request; either keep string filters on `list` or expose/translate the structured search shape.
  - confidence: 0.95
