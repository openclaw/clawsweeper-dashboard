## Summary

Keep open: the Feishu Bitable feature is not implemented on current main, but this PR is not merge-ready because it exposes `filter` as a string while routing filtered calls to the Lark SDK search endpoint, whose locked contract expects a structured filter object; it also lacks required real behavior proof from a real Feishu setup.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `view_id` parameters to `feishu_bitable_list_records`, using the Lark SDK search endpoint for filtered or sorted requests.

## Best Possible Solution

Land one corrected Feishu plugin update that either keeps string filters on the list API or exposes the search API's structured filter contract, adds focused forwarding coverage, and includes redacted real Feishu behavior proof.

## Reproduction Assessment

Yes for the review finding: source inspection of the PR branch against the locked Lark SDK types shows the new string `filter` schema is routed to `appTableRecord.search`, whose `data.filter` contract is structured. No real Feishu setup proof was provided for the intended new behavior.

## Solution Assessment

No. Exposing server-side Feishu Bitable query parameters is the right narrow direction, but the current implementation mixes the list API's string filter syntax with the search API's structured filter body and needs correction before merge.
