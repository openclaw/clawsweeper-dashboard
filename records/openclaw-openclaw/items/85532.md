## Summary

Keep open: current main still has the reported diagnostic path where a fresh item-level terminal-looking Codex progress reason can make an old processing session eligible for active embedded-run abort, even though the run may have only just produced app-server progress.

## What This Changes

The issue reports that diagnostic stuck-session recovery can abort active Codex embedded runs after item-level terminal-looking app-server progress events, leaving direct/group task outcomes ambiguous.

## Best Possible Solution

Gate active recovery on durable run/session terminal evidence or a genuinely stale no-progress invariant, and keep item-level terminal-looking Codex notifications from bypassing the normal progress-age threshold.

## Reproduction Assessment

Yes at source level: current code classifies queued embedded runs behind item-level terminal-looking progress as stalled and requests `allowActiveAbort` based on processing-session age, and existing tests assert that behavior. I did not run a live Codex app-server replay.

## Solution Assessment

Yes: the narrowest maintainable fix is to stop treating item/span completion as enough evidence to abort active embedded work, while preserving aborts for durable terminal run state or stale no-progress conditions. Broader raw-assistant default semantics should remain with the existing semantic hub.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/85532 by narrowing diagnostic stuck-session active aborts for embedded runs. In `src/logging/diagnostic-session-attention.ts` / `src/logging/diagnostic.ts`, do not let item/span-level terminal-looking Codex app-server progress such as `rawResponseItem/completed`, `response.completed`, or `output_item.done` make an active embedded run abortable solely because the processing session age crossed `diagnostics.stuckSessionAbortMs`; require durable run/session terminal evidence or require `lastProgressAgeMs` to be stale at the abort threshold. Preserve existing recovery for true no-progress embedded/model/tool stalls and idle queued stale lanes. Update the regression that currently expects recovery immediately after a fresh terminal progress event, add a case where later non-terminal progress disarms the terminal classification, and keep a case proving genuinely stale terminal/no-progress work still recovers. Do not change Codex app-server default raw-assistant semantics or add a new config option. Add a changelog entry only if maintainers require one for this user-visible runtime fix.
