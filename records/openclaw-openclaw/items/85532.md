## What This Changes

The issue asks OpenClaw to stop diagnostic recovery from aborting active embedded runs solely because the latest Codex app-server progress event looks terminal at an item level.

## Best Possible Solution

Review and land, revise, or replace https://github.com/openclaw/openclaw/pull/85623 so active abort requires durable run/session terminal state or a true no-progress invariant, with regression coverage for recent and stale terminal-looking progress.

## Reproduction Assessment

Yes. Current source and tests reproduce the core path: `rawResponseItem/completed` behind queued embedded work is classified as stalled and the heartbeat test expects active abort recovery after the abort threshold.

## Solution Assessment

Yes for the issue direction. The narrow maintainable solution is to keep item-level terminal progress from triggering active embedded-run abort unless the run/session is durably terminal or genuinely stale, while preserving recovery for truly orphaned runs.
