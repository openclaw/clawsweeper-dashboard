## Summary

Keep open: current main still has the reported diagnostic path, and the linked diagnostic-side PR is open rather than merged. Source and tests show item-level Codex terminal-looking progress can classify a queued active embedded run as stalled and then request active abort recovery once the session-age threshold is reached.

## What This Changes

The issue asks OpenClaw to stop treating item-level Codex app-server terminal notifications as enough to abort an active embedded run and to preserve an unambiguous recovery outcome.

## Best Possible Solution

Review, revise, or land the diagnostic-side fix so active embedded-run aborts require durable run/session terminal state or a true no-progress invariant, while preserving recovery for genuinely orphaned runs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence because current tests explicitly feed `rawResponseItem/completed` into a queued embedded run and expect active abort recovery after the abort threshold; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because item-level terminal progress is being used as enough evidence for active embedded-run abort. The narrow fix should separate item/span completion from run/session completion and only abort when active work is truly orphaned or stale.
