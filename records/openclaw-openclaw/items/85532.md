## Summary

Keep open: current main and the latest release still contain the diagnostic path that treats queued embedded runs with item-level terminal-looking progress as stalled and can then abort the active embedded run. The issue is paired with the open closing PR https://github.com/openclaw/openclaw/pull/85623, so it should stay open until that repair is reviewed and merged or rejected.

## What This Changes

The issue reports OpenClaw diagnostic recovery aborting active embedded Codex runs when app-server item-level completion progress is treated as terminal stalled work.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85623, or a maintainer-approved equivalent, so fresh item-level terminal progress cannot trigger active embedded-run abort while truly stale/orphaned runs still recover.

## Reproduction Assessment

Yes, at source level: current main still classifies queued embedded runs with recent rawResponseItem/completed progress as stalled, and focused tests encode that behavior. I did not run a live gateway reproduction because this review is read-only and test execution could write artifacts.

## Solution Assessment

Yes for the narrow repair direction: the maintainable path is to stop fresh item-level terminal notifications from authorizing active embedded-run abort unless progress is actually stale or the run/session is durably terminal. The linked PR should be reviewed rather than starting a parallel fix.
