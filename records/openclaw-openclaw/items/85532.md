## Summary

Keep open: current `main` and the shipped `v2026.5.20` code still classify queued active embedded runs as stalled when the latest progress reason is an item-level terminal-looking Codex notification, then request active abort recovery after the session-age threshold.

## What This Changes

The issue reports that diagnostic recovery can abort active embedded Codex runs after terminal-looking app-server item progress even though task-level work may still be active.

## Best Possible Solution

Gate active embedded-run aborts on durable run/session terminal state or a true no-progress invariant, and add regression coverage for fresh item-level terminal progress followed by later non-terminal assistant/tool/browser progress with queued work.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current tests already exercise `rawResponseItem/completed` behind queued embedded work and expect active abort recovery after the abort threshold. I did not run a live gateway reproduction because this was a read-only review.

## Solution Assessment

Yes for the first repair: the narrowest maintainable fix is to stop item-level terminal notifications from authorizing active embedded-run abort unless the run/session is durably terminal or progress is actually stale. A separate `system_aborted` UI/API state can be considered after the abort boundary is corrected.
