## Summary

Keep open: current main still classifies queued active embedded runs as stalled when the last progress reason is an item-level terminal-looking Codex notification, then permits active embedded-run abort by session age. The issue is source-reproducible and has overlapping open diagnostic PRs, so it should stay open until a narrow fix lands or a maintainer chooses the canonical PR path.

## What This Changes

The issue asks OpenClaw to stop diagnostic recovery from aborting active embedded Codex runs solely because the latest app-server progress event looks terminal at the item level.

## Best Possible Solution

Land or replace a narrow diagnostic fix that separates item-level terminal progress from durable embedded-run terminal state, with regression coverage for terminal-looking raw item progress followed by later non-terminal progress and queued work.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main has a test that feeds rawResponseItem/completed behind queued embedded work and expects active-abort recovery after the abort threshold; I did not run live gateway proof in this read-only review.

## Solution Assessment

Yes for the first repair. The narrow maintainable fix is to stop item-level terminal notifications from authorizing active embedded-run abort unless the run/session is durably terminal or progress is truly stale; a distinct system_aborted state can be a follow-up if needed.
