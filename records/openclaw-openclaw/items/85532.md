## What This Changes

OpenClaw should not abort active embedded Codex runs solely because the latest app-server progress event looks terminal at item level while the run may still be producing useful work.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/85623, with coverage that recent item-level terminal progress does not trigger abort while truly stale/orphaned embedded runs still recover safely.

## Reproduction Assessment

Yes, at source level: current tests exercise a queued embedded run with rawResponseItem/completed and expect stalled classification plus active abort recovery. I did not run a live gateway reproduction because this was a read-only review.

## Solution Assessment

Yes for the narrow bug boundary: stop item-level terminal notifications from authorizing active embedded-run abort unless the run is truly stale or durably terminal. The open linked PR is the active candidate, but it still needs its own review and proof before this issue can close.
