## Summary

Keep open: current main and the latest release still classify queued embedded runs as stalled on item-level terminal-looking progress and can then run active embedded-run abort recovery. The issue is now paired with an open closing PR, so the issue should stay open until that repair is merged or rejected.

## What This Changes

Prevent diagnostic stuck-session recovery from aborting active embedded Codex direct/group runs based only on item-level terminal-looking app-server progress.

## Best Possible Solution

Review and land the narrow diagnostic repair, likely through https://github.com/openclaw/openclaw/pull/85623 after proof and test updates, so item-level terminal progress does not trigger active abort unless the run is durably terminal or truly stale.

## Reproduction Assessment

Yes, at source level: current main still classifies queued embedded runs with `rawResponseItem/completed` as stalled and the focused heartbeat test expects active abort recovery. I did not run a live gateway reproduction because this was a read-only review.

## Solution Assessment

Yes for the first repair direction: the maintainable fix is to stop item-level terminal notifications from authorizing active embedded-run abort unless progress is actually stale or the run/session is durably terminal. The open linked PR should be reviewed rather than starting a parallel repair.
