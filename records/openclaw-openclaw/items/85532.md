## Label Justifications

- `P1`: The issue describes core diagnostic recovery interrupting active channel work and leaving direct/group task outcomes uncertain for real users.
- `impact:session-state`: The affected code is the session diagnostic classifier and stuck-session recovery state machine for embedded runs.
- `impact:message-loss`: The reported failure can abort or obscure a final channel response before it is delivered to the user.

## Summary

Keep open: current main still classifies queued embedded Codex runs with item-level terminal-looking progress as stalled and eligible for active abort after the abort threshold. The related open PR at https://github.com/openclaw/openclaw/pull/85571 is a plausible fix path but is not merged and should be reviewed for whether faster recovery with the same abort action satisfies the reported failure.

## What This Changes

The issue asks OpenClaw to stop diagnostic recovery from aborting active embedded Codex runs based only on item-level terminal progress and to preserve clear system-aborted outcome state.

## Best Possible Solution

Keep the issue open until the diagnostic classifier and recovery path distinguish item-level terminal progress from durable run/session terminal state, preserve recovery for truly orphaned embedded runs, and add regression coverage for alternating terminal and non-terminal progress.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main has a test using codex_app_server:notification:rawResponseItem/completed behind queued embedded work that expects active abort recovery after the abort threshold; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes for the bug boundary: the maintainable fix is to gate active embedded-run abort on durable run/session terminal state or a true orphan/no-progress invariant. A faster recovery path alone needs review because it may keep the same abort outcome that the issue reports as harmful.
