## Summary

Keep open: current main and the latest release still contain the source path that can classify a queued embedded run as stalled solely from terminal-looking item progress and then allow active abort recovery. The issue is already paired with open repair PRs, including a closing PR, so it should remain open until maintainers land or reject that repair path.

## What This Changes

Prevent diagnostic recovery from aborting active embedded Codex runs solely because a terminal-looking app-server item progress event appears while queued work exists.

## Best Possible Solution

Review and land the narrow diagnostic repair through https://github.com/openclaw/openclaw/pull/85623, or explicitly choose the companion recovery-semantics path in https://github.com/openclaw/openclaw/pull/85571, so item-level terminal progress cannot by itself authorize active abort of live embedded work.

## Reproduction Assessment

Yes, at source level: current main still classifies queued embedded runs with `rawResponseItem/completed` as stalled and the focused heartbeat test expects active abort recovery after the abort threshold. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

Yes for the narrow first repair: the maintainable fix is to require actual stale progress or durable terminal run state before item-level terminal notifications can lead to active abort. Because an open closing PR already implements that shape, the issue should track that PR rather than spawning a parallel fix.
