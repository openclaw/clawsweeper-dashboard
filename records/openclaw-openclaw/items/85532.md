## Summary

Keep open: current main and v2026.5.22 still classify queued embedded runs with item-level terminal-looking progress as stalled and can run active embedded-run abort recovery. The issue has an open closing PR, so it should stay open until that repair is reviewed, merged, or rejected.

## What This Changes

Diagnostic recovery should not abort active embedded Codex direct/group runs solely because the latest app-server progress event looks terminal while queued work exists.

## Best Possible Solution

Review and land the canonical diagnostic repair, likely through https://github.com/openclaw/openclaw/pull/85623 after proof and test updates, so item-level terminal progress does not authorize active abort unless the embedded run is durably terminal or actually stale.

## Reproduction Assessment

Yes, at source level: current main still classifies queued embedded runs with `rawResponseItem/completed` as stalled, and the focused heartbeat test expects active abort recovery. I did not run a live gateway reproduction because this was a read-only review.

## Solution Assessment

Yes for the narrow repair direction: the maintainable fix is to stop item-level terminal notifications from authorizing active embedded-run abort unless progress is actually stale or the run/session is durably terminal. The open linked PR should be reviewed instead of starting a parallel repair.
