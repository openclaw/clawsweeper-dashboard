## Summary

Keep open: current main and the latest release still contain the diagnostic path that treats item-level terminal-looking Codex progress as a stalled embedded run and can request active abort recovery. The linked open PR https://github.com/openclaw/openclaw/pull/85623 is the current fix candidate, so this issue should stay open until that PR or an equivalent repair lands.

## What This Changes

The issue reports diagnostic recovery aborting active embedded Codex runs after terminal-looking app-server item progress while queued channel work is present.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85623, or an equivalent fix, so active embedded-run aborts require stale progress or durable run/session terminal evidence before this issue is closed.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current classifier and heartbeat tests already encode `rawResponseItem/completed` behind queued embedded work leading to stalled classification and active abort recovery after the threshold.

## Solution Assessment

Yes for the narrow repair direction. The maintainable fix is to stop item-level terminal notifications from authorizing active embedded-run abort unless progress is actually stale or the run/session is durably terminal; a separate `system_aborted` state can be evaluated after the abort boundary is fixed.
