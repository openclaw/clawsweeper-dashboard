## Summary

Keep open. Current main still lacks contact-backed WhatsApp directory ingestion: peer listing is config-backed from `allowFrom`, the WhatsApp socket paths do not subscribe to Baileys contact-list events, and the generic outbound resolver can only resolve names if the plugin supplies matching directory entries.

## What This Changes

The issue asks the WhatsApp plugin to persist Baileys contact-list data, expose saved phone contacts through `openclaw directory peers`, and allow outbound messaging by contact name.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin as an opt-in, account-scoped local contact directory fed by verified Baileys contact sync events or an equivalent sync path. Directory entries should expose stable ids, display names/handles, and raw metadata while preserving existing outbound authorization checks and updating the WhatsApp and directory docs once the behavior exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, `openclaw directory peers list --channel whatsapp --query Emily` can only search entries derived from `allowFrom`, so a saved phone contact named Emily cannot appear unless the user manually configured that number.

## Solution Assessment

Unclear as an automated change, but the requested direction is the right product boundary. The safest maintainable solution is narrower than the old message-store PR: opt-in contact persistence plus directory/name-resolution integration in the WhatsApp plugin, with privacy review first.
