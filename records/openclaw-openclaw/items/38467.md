## Summary

Keep open. Current main still has a source-confirmed WhatsApp directory feature gap: peers are config-backed from `allowFrom`, there is no contact-event store, and implementing local phone-contact persistence needs maintainer privacy/product direction.

## What This Changes

Expose WhatsApp phone contacts from Baileys contact events through the directory and outbound target resolver so users can address contacts by name.

## Best Possible Solution

Add an opt-in, WhatsApp-plugin-owned contact store that consumes Baileys contact events, exposes named contacts through the existing directory adapter, and lets the generic resolver handle name-based sends.

## Reproduction Assessment

Yes, as a source-confirmed feature gap: current main lists WhatsApp peers from `allowFrom`, has no `contacts.upsert` listener/store, and Baileys exposes the contact event the issue names. This is not a broken documented contract, but the missing path is clear.

## Solution Assessment

Unclear. Using Baileys contact events is plausible, but the best maintainable fix needs a maintainer decision on opt-in storage, privacy wording, retention, and lifecycle placement inside the WhatsApp plugin.
