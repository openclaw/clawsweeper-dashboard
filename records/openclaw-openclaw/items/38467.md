## Summary

Keep open. Current main still has no WhatsApp contact ingestion or persisted name-bearing peer directory, and the requested behavior is a new privacy-sensitive WhatsApp plugin feature rather than an existing broken contract.

## What This Changes

The issue requests persisting WhatsApp phone contacts from Baileys contact events, exposing them through the directory API, and using them for name-based outbound resolution.

## Best Possible Solution

Add an opt-in, WhatsApp-plugin-owned contact store that consumes Baileys contact events, exposes account-scoped name-bearing directory entries, and reuses the generic resolver with clear ambiguity and authorization rules.

## Reproduction Assessment

Yes, as a source-confirmed feature gap rather than a broken existing contract: current main returns WhatsApp peers from `allowFrom`, has no `contacts.upsert` ingestion, and can only resolve names when directory entries already provide `name` or `handle`. I did not run a live WhatsApp session.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best fix needs maintainer decisions around opt-in contact persistence, retention, ambiguity handling, and whether trusted relay behavior should depend on contact-directory data or explicit aliases.
