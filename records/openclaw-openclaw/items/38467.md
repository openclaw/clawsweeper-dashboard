## Summary

Close as ClawHub/plugin-surface work: current main intentionally keeps WhatsApp directory peers config-backed, and a prior opt-in WhatsApp contact/message-store PR with `contacts.upsert` was closed unmerged as outside the project vision with `wacli` named as the alternative.

## What This Changes

The issue requests persisting WhatsApp contacts received from Baileys `contacts.upsert`, exposing them through directory peer lookup, and using them for name-based message resolution.

## Best Possible Solution

Keep the bundled WhatsApp channel focused on safe explicit and allowlisted messaging; users needing contact/history resolution should use or publish a ClawHub skill/plugin such as wacli, and propose a narrow plugin API only if current surfaces are insufficient.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms current main lacks the requested feature: WhatsApp peers are `allowFrom`-backed and there is no `contacts.upsert` persistence path.

## Solution Assessment

Yes for cleanup. Closing as ClawHub/out-of-core is the best current path because a similar opt-in contact/message store PR was already declined as outside `VISION.md`, and wacli covers third-party WhatsApp search/send without expanding bundled channel defaults.
