## Summary

Keep open. Current main still exposes WhatsApp peers only from configured `allowFrom` entries, and the requested contact persistence/name-resolution behavior is a new privacy-sensitive WhatsApp plugin capability rather than an already-implemented fix.

## What This Changes

Expose WhatsApp phone contacts received from Baileys `contacts.upsert` through directory peer lookup and name-based outbound message resolution.

## Best Possible Solution

Add an opt-in, plugin-owned WhatsApp contact store that consumes Baileys contact events, exposes contacts as directory entries with names/handles, and reuses the existing generic target resolver without adding WhatsApp-specific behavior to core.

## Reproduction Assessment

Yes, as a source-confirmed feature gap rather than a broken existing contract. Current main wires WhatsApp peers to `allowFrom`, has no `contacts.upsert` handler, and already has generic directory-name matching that would work once WhatsApp supplies named entries.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best implementation needs maintainer choices around opt-in storage, privacy wording, and whether contact ingestion belongs in socket setup or monitor lifecycle.
