## Summary

Keep open. Current main still has the generic directory/name-resolution plumbing, but the bundled WhatsApp plugin only lists config-backed allowlist peers and has no Baileys contact-list ingestion or persistence path, so saved phone contacts cannot be resolved by name.

## What This Changes

Request opt-in persistence and directory exposure of WhatsApp phone contacts from Baileys contact events so agents can resolve saved contact names for outbound messages.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin as opt-in, account-scoped local contact storage feeding directory entries with stable id/name/handle/raw metadata, while preserving existing allowFrom/outbound authorization checks and updating WhatsApp plus directory docs.

## Reproduction Assessment

Yes, at source level. On current main, `openclaw directory peers list --channel whatsapp --query Emily` can only search peers derived from `allowFrom`, so a saved phone contact named Emily cannot appear unless the user manually added that number to config.

## Solution Assessment

Unclear as an automated change. The requested direction belongs in the WhatsApp plugin, but contact persistence needs a maintainer privacy/product decision before code should be queued.
