Keep open for maintainer direction: current main still lacks a WhatsApp contact-backed directory or `contacts.upsert` persistence, but the requested name-resolution feature would persist a user's phone contacts and likely needs an opt-in config/privacy policy rather than a narrow bug-fix cleanup.

## What This Changes

Request to persist WhatsApp contacts from Baileys `contacts.upsert` and expose them through directory/name-based WhatsApp target resolution.

## Best Possible Solution

Choose the WhatsApp plugin privacy/API boundary first; if accepted, add an opt-in account-scoped contact directory that consumes Baileys contacts, exposes safe directory entries, documents retention/deletion, and enforces allowlist plus ambiguity rules before name-based sends.

Not applicable as a feature request, but there is a high-confidence source verification path. Current main lists WhatsApp peers from `allowFrom`, has no `contacts.upsert` handler/store, and Baileys exposes the requested contact events and contact fields.

## Solution Assessment

Unclear until maintainers decide the privacy and authorization policy. Consuming Baileys contact events is technically plausible through the WhatsApp plugin directory surface, but the durable solution needs opt-in storage, retention/deletion behavior, and ambiguity-safe target resolution.
