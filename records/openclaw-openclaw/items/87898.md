## Summary

Keep this PR open, but it still needs changes before merge. The branch fixed the earlier broad suffix heuristic by reusing the known-marker contract, but the new skip is wired into the generic config-secret audit loop, so it can suppress plaintext findings outside the reported model-provider apiKey case.

## What This Changes

The PR updates secrets audit handling so known non-secret API-key marker values in `openclaw.json` do not produce `PLAINTEXT_FOUND`, and adds regression tests for known markers, arbitrary uppercase values, and real key-like strings.

## Best Possible Solution

Keep the known-marker approach, but constrain the exemption to string values on the intended `models.providers.*.apiKey` target and prove other secret fields still warn.

## Reproduction Assessment

Yes. Source inspection shows current main flags plaintext config secret targets, while the PR branch adds a generic marker skip before adding `PLAINTEXT_FOUND`; the existing tests and docs define `secrets audit --check` as the plaintext-finding path.

## Solution Assessment

No. Reusing `isNonSecretApiKeyMarker` is the right direction, but applying it to every config secret target is broader than the reported model-provider `apiKey` false positive.
