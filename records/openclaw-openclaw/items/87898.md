## What This Changes

The PR adds a suffix-based helper in `src/secrets/audit.ts` that skips `PLAINTEXT_FOUND` for audited config secret values that look like uppercase environment variable names.

## Best Possible Solution

Align the audit skip with the existing SecretInput and known-marker contracts, preserve warnings for literal all-caps credentials, and add focused regression coverage for both valid env refs and literal credential strings.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current audit warns after `resolveSecretInputRef` fails, while the proposed skip would bypass that warning for suffix-matching strings. I did not run the CLI because this review is read-only.

## Solution Assessment

No. The patch is broader than the established SecretInput contract; the safer solution is to reuse the existing SecretRef/known-marker logic or make any new bare-env syntax an explicit, tested product decision.

Full review comments:

- **[P1] Do not suppress literal all-caps secret values:** `src/secrets/audit.ts:214-216`
  - body: The new skip treats any uppercase value ending in `_KEY`, `_TOKEN`, `_SECRET`, or `_PASSWORD` as safe, but `openclaw.json` only treats SecretRef objects and `$VAR`/`${VAR}` strings as env refs.
