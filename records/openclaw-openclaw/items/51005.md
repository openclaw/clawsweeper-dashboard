## Summary

Keep open. Current main still rejects the reported Brave LLM Context freshness and date filters in the active runtime, while Brave's own API docs support `freshness`; the PR is useful source work but is not mergeable as-is because it targets stale provider code and leaves required release polish unfinished.

## What This Changes

The PR unblocks Brave `llm-context` `freshness`, `date_after`, and `date_before`, converts date filters into Brave `freshness` ranges, updates web-search docs, and replaces rejection tests with endpoint-parameter assertions.

## Best Possible Solution

Land a narrow Brave plugin fix in the active runtime: allow documented LLM Context freshness/date filters, keep `ui_lang` unsupported unless a separate API decision changes that, handle one-sided date ranges deliberately, update both Brave/web docs, add focused Brave extension tests, and include the required changelog entry.

## Reproduction Assessment

Yes. With Brave configured as `mode: "llm-context"`, current main deterministically returns `unsupported_freshness` or `unsupported_date_filter` before any Brave network call; this is visible directly in `executeBraveSearch`. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The intended fix direction is right, but this branch edits stale provider code instead of the active runtime on current main and needs date-range/changelog cleanup before it is the narrowest maintainable solution.
