## Summary

Keep open. Current main still leaves the reported Playwright CDP assertion shape outside the browser plugin’s registered unhandled-rejection classifier, and the related broad trackers/fix PRs are closed rather than providing an open canonical fix.

## What This Changes

The issue asks OpenClaw to prevent a Playwright CDP `Assertion error` from `CRSession._onMessage` / `CRConnection._onMessage` from crashing the entire Gateway process.

## Best Possible Solution

Add a browser-owned stability boundary that recognizes direct and wrapped Playwright CDP assertion rejections with `CRSession._onMessage` / `CRConnection._onMessage` stack markers, degrades or resets browser control where feasible, and preserves the existing fatal fallback for unknown errors.

## Reproduction Assessment

Yes for a high-confidence unit-style reproduction: after the browser runtime handler is registered, synthesize an unhandled rejection with `Assertion error` plus `CRSession._onMessage` or `CRConnection._onMessage` stack markers and verify current main does not classify it as handled. A live Chrome reproduction is less deterministic because the unexpected CDP message is timing-dependent.

## Solution Assessment

Yes for the near-term repair: extending the existing browser plugin `registerUnhandledRejectionHandler` seam is the narrowest maintainable fix. Browser subprocess isolation and crash-state persistence are larger architecture topics and should not block this focused crash-class guard.
