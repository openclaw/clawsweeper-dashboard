## Summary

Keep open. Current main still attaches browser profiles through Playwright CDP, the browser plugin only registers a dialog-race unhandled-rejection classifier, and an unclassified CRSession assertion still reaches the process-level fatal unhandled-rejection fallback; the related broader reports and attempted fix PR are closed rather than owning the remaining work.

## What This Changes

The issue asks OpenClaw to keep Playwright CDP assertion failures from crashing the Gateway process.

## Best Possible Solution

Add a browser-plugin-owned, narrowly matched classifier for the CRSession._onMessage unmatched-response assertion class with positive and negative regression coverage; treat browser subprocess isolation as a separate architecture discussion.

## Reproduction Assessment

No high-confidence live reproduction path was established here. The source path is clear enough to keep open: Playwright CDP is still used, Playwright v1.60.0 still has the CRSession._onMessage assertion, and OpenClaw still exits for unclassified unhandled rejections.

## Solution Assessment

Yes, a narrow browser-owned rejection classifier is the best first repair for the concrete crash signature. A gateway-wide Playwright swallow or browser subprocess redesign is broader and riskier than needed for this specific assertion path.
