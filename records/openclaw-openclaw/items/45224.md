## Summary

Keep open. Current main still only suppresses the Playwright dialog-race unhandled rejection; the reported CDP assertion shape remains outside the browser plugin classifier and would still reach the Gateway fatal unhandled-rejection path.

## What This Changes

The issue requests graceful handling or isolation of Playwright CDP assertion failures so browser-control errors do not terminate the Gateway and unrelated sessions.

## Best Possible Solution

Add a browser-owned stability boundary that recognizes direct and wrapped Playwright CDP `Assertion error` rejections with `CRSession._onMessage` or `CRConnection._onMessage` stack markers, then degrades or resets browser control where feasible.

## Reproduction Assessment

Yes. A high-confidence unit-style reproduction can register the browser runtime handler, synthesize an unhandled `Assertion error` with `CRSession._onMessage` or `CRConnection._onMessage` stack markers, and verify current main leaves it unhandled; the live Chrome trigger is less deterministic.

## Solution Assessment

Yes for the near-term repair. Extending the existing browser plugin `registerUnhandledRejectionHandler` seam is the narrowest maintainable fix, while subprocess isolation and crash-state persistence are larger architecture topics.
