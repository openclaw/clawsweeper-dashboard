## Summary

Keep open: current main still has the reported failure shape. The browser plugin only registers a narrow Playwright dialog-race unhandled-rejection handler, Playwright 1.59.1 still asserts in the CDP `_onMessage` path, and the process-level fallback still exits on generic unhandled `Error: Assertion error`.

## What This Changes

The issue asks OpenClaw to prevent Playwright CDP assertion errors from crashing the Gateway when browser control is connected.

## Best Possible Solution

Add a precise browser-owned error boundary for known Playwright CDP assertion stacks, with regression tests proving the Gateway continues while unrelated assertions remain fatal.

## Reproduction Assessment

Yes, as a source-level reproduction: an unhandled `Error: Assertion error` with a Playwright `CRSession._onMessage` stack is not consumed by the current browser handler and therefore reaches the fatal global fallback. A live Chrome trigger was not run in this read-only pass.

## Solution Assessment

Yes, a narrow browser plugin handler/classifier is the best first fix. A broad process-level Playwright swallow or child-process isolation would be riskier and larger than needed for this concrete stack.
