## Summary

Keep open: current main still has the source-plausible failure shape for raw/remote CDP profiles, because browser CDP still goes through Playwright, the browser plugin only suppresses a narrow dialog-race rejection, and generic unclassified unhandled rejections still exit the Gateway. The newer `existing-session` Chrome MCP path reduces exposure for user-profile attach, but it does not implement the requested crash isolation or prove this CRSession assertion class is fixed.

## What This Changes

The issue reports that a Playwright CDP assertion from browser session handling can escape as an unhandled rejection, crash the Gateway, and terminate unrelated active sessions.

## Best Possible Solution

Keep the issue open until maintainers choose the browser-runtime boundary: narrow Playwright assertion classification with focused tests and live CDP proof, or a larger browser/CDP isolation design that prevents browser failures from taking down unrelated Gateway sessions.

## Reproduction Assessment

No, not as a live current-main repro. The reported Playwright 1.58.2 assertion path, current raw-CDP Playwright path, and current fatal fallback are source-backed, but the exact Playwright 1.60.0 CDP trigger still needs live reproduction.

Unclear. A narrow assertion classifier may be an acceptable mitigation, but the discussion also asks for browser/CDP isolation and crash recovery, which is a maintainer product/architecture decision rather than a safe automatic bug patch.
