## Summary

Keep open. Current main and the latest shipped release still attach browser profiles through Playwright CDP, the browser-owned rejection handler only suppresses dialog-race errors, and an unclassified CRSession assertion still falls through to the process-level fatal unhandled-rejection path; the related broader issues and attempted fix PR are closed rather than owning the remaining work.

## What This Changes

The issue asks OpenClaw Gateway to gracefully handle Playwright CDP CRSession._onMessage assertion errors so browser/CDP failures do not terminate the whole Gateway process.

## Best Possible Solution

Add a browser-plugin-owned, narrowly matched handler for the CRSession._onMessage unmatched-response assertion class with positive and negative regression coverage; treat broader browser subprocess isolation as a separate architecture discussion.

## Reproduction Assessment

No high-confidence live reproduction path was established here. The current source path remains clear: Playwright CDP is still used, Playwright v1.60.0 still has the CRSession._onMessage assertion, and OpenClaw still exits for unclassified unhandled rejections.

## Solution Assessment

Yes, a narrow browser-owned rejection classifier is the best first repair for the concrete crash signature. Browser subprocess isolation may be useful later, but it is broader than needed for this specific assertion path.
