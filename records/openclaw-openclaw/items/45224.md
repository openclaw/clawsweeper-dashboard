## Summary

Current main and the latest release still run Playwright CDP in the Gateway process and intentionally terminate on an unknown unhandled rejection; Playwright 1.60.0 retains the same `CRSession._onMessage` assertion, so the reported crash class is not proven fixed. The issue should remain open, but the exact offending CDP response is still needed before choosing a safe narrow repair.

## What This Changes

Prevent an unexpected Playwright CDP response from crashing the Gateway and terminating unrelated in-flight sessions.

## Best Possible Solution

Capture the offending Playwright 1.60 CDP response and implement recovery at the browser plugin or upstream session boundary—evicting or reconnecting the affected browser connection—while retaining the global fatal fallback for genuinely unknown rejections.

## Reproduction Assessment

No, not at high confidence. The reported stack maps to a current source-level crash path and the upstream assertion remains, but the missing CDP response prevents a deterministic current-main reproduction.

## Solution Assessment

No, the suggested broad global Playwright swallow is not the best fix because it could hide genuine protocol corruption. A narrow browser-owned recovery path based on a reproduced response and connection state is safer.
