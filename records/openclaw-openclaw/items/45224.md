## Summary

Keep open: current main and v2026.6.8 still connect browser profiles through Playwright CDP, only suppress a narrow dialog-race rejection, and send unclassified Playwright assertion rejections to the fatal Gateway handler; the exact CRSession trigger still needs live repro and a maintainer policy choice.

## What This Changes

The issue asks OpenClaw to prevent Playwright CRSession._onMessage CDP assertion errors from crashing the whole Gateway and losing unrelated in-flight session work.

## Best Possible Solution

Keep the issue open until maintainers choose a browser/CDP crash-isolation policy, then add a narrow recovery path that degrades browser control without swallowing unknown process errors and proves it with a live CDP repro.

## Reproduction Assessment

No, not as a high-confidence live repro. The reported stack maps to current source and Playwright 1.60.0 still has the asserted CRSession._onMessage path, but the exact CDP message trigger has not been reproduced on current main.

## Solution Assessment

Unclear. The issue's broad global-catch suggestion is risky; the safer direction is a scoped browser/CDP recoverability boundary or explicit isolation design with live proof.
