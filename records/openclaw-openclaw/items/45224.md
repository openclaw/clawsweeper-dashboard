## Summary

Keep open. Current main still leaves the reported Playwright CDP assertion shape outside the browser plugin’s registered unhandled-rejection classifier, and the related broad trackers/fix PRs are closed rather than providing an open canonical fix.

## What This Changes

The issue requests graceful isolation or handling for Playwright CDP assertion failures so browser-control errors do not terminate unrelated Gateway sessions and cron jobs.

## Best Possible Solution

Add a browser-owned stability boundary that recognizes direct and wrapped Playwright CDP assertion rejections with `CRSession._onMessage` / `CRConnection._onMessage` stack markers, degrades or resets browser control where feasible, and preserves the existing fatal fallback for unknown errors.

## Reproduction Assessment

Yes. A high-confidence unit-style reproduction can register the browser runtime handler, synthesize an unhandled `Assertion error` with `CRSession._onMessage` or `CRConnection._onMessage` stack markers, and verify current main leaves it unhandled; the live Chrome trigger is less deterministic.

## Solution Assessment

Yes for the near-term repair. Extending the existing browser plugin `registerUnhandledRejectionHandler` seam is the narrowest maintainable fix, while subprocess isolation and crash-state persistence are larger architecture topics that should not block this focused crash-class guard.
