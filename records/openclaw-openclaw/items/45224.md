## Summary

Keep open. Current main still leaves this Playwright CDP assertion crash class unhandled: the Gateway fatal fallback exits on unclassified unhandled rejections, and the browser plugin’s registered handler only consumes the narrower Playwright dialog-race shape, not `Assertion error` rejections with `CRSession._onMessage` or `CRConnection._onMessage` stack markers.

## What This Changes

The issue requests that Gateway-owned browser/CDP failures be isolated or handled gracefully so a Playwright assertion from Chrome control does not terminate unrelated Gateway sessions and cron jobs.

## Best Possible Solution

Add a browser-owned stability boundary for Playwright CDP assertion rejections: classify direct and wrapped `Assertion error` failures with `CRSession._onMessage` or `CRConnection._onMessage` stack markers, log/degrade browser control, clear cached Playwright CDP state where feasible, and preserve the existing fatal behavior for unknown unhandled rejections.

## Reproduction Assessment

Yes for a high-confidence unit-style reproduction: synthesize an unhandled rejection with `Assertion error` plus `CRSession._onMessage` / `CRConnection._onMessage` stack markers after the browser runtime handler is registered and verify current main leaves it unhandled. A live Chrome reproduction is less deterministic because it depends on receiving an unexpected CDP message while connected.

## Solution Assessment

Yes. The best near-term fix is a narrow browser-plugin classifier and recovery path using the existing `registerUnhandledRejectionHandler` seam; subprocess isolation and crash-state persistence are larger architecture topics that should not block this focused repair.
