## Summary

Keep open. Current main still leaves the reported Playwright CDP assertion shape outside the browser plugin’s registered unhandled-rejection classifier, so it can still fall through to the Gateway fatal unhandled-rejection path.

## What This Changes

The issue requests that Playwright CDP assertion failures from browser control be isolated or handled gracefully so they do not terminate unrelated Gateway sessions and cron jobs.

## Best Possible Solution

Add a browser-owned stability boundary that recognizes direct and wrapped Playwright CDP assertion rejections with `CRSession._onMessage` or `CRConnection._onMessage` stack markers, degrades or resets browser control where feasible, and keeps unknown unhandled rejections fatal.

## Reproduction Assessment

Yes. A high-confidence unit-style reproduction can register the browser runtime handler, synthesize an `Assertion error` rejection with `CRSession._onMessage` or `CRConnection._onMessage` in the stack, and verify current main leaves it unhandled.

## Solution Assessment

Yes for the near-term repair. Extending the existing browser plugin `registerUnhandledRejectionHandler` seam is narrower and more maintainable than a broad global swallow or immediate browser subprocess isolation.
