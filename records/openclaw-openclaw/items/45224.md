## Summary

Keep open. Current main still only suppresses a narrow Playwright dialog-close race, while a generic Playwright CDP `Assertion error` with `CRSession._onMessage` or `CRConnection._onMessage` stack markers would fall through to the Gateway fatal unhandled-rejection path.

## What This Changes

The issue asks the Gateway browser-control path to tolerate Playwright CDP assertion rejections from `CRSession._onMessage` / `CRConnection._onMessage` without crashing unrelated sessions.

## Best Possible Solution

Add a browser-owned stability guard that recognizes direct and wrapped Playwright CDP assertion rejections with `CRSession._onMessage` / `CRConnection._onMessage` stack markers, degrades or resets browser control where appropriate, and leaves unrelated unhandled rejections fatal.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can register the browser runtime handler, synthesize an `Assertion error` carrying `CRSession._onMessage` or `CRConnection._onMessage` stack text, and verify current main leaves it unhandled; the live Chrome trigger is less deterministic.

## Solution Assessment

Yes for a near-term repair. Extending the existing browser plugin `registerUnhandledRejectionHandler` seam is narrower and safer than adding a process-wide catch-all, while subprocess isolation is a larger architecture follow-up.
