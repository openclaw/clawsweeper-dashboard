## Summary

Keep open: the underlying provider-error formatting bug is still real on current main, but this PR is not mergeable as written because it keeps upstream request IDs in chat-visible copy and lacks viewable after-fix real behavior proof.

## What This Changes

The PR adds generic OpenAI-compatible internal-error detection to assistant error formatting, sanitization, and failover classification, with tests expecting friendly copy that includes the support request ID.

## Best Possible Solution

Port a formatter-level fix onto current main so structured and plain-text provider internal errors produce generic chat-safe copy without raw request IDs, while preserving failover classification and adding focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A reported provider `server_error` payload can flow through `formatAssistantErrorText` into the shared raw formatter and then into Feishu post text without a channel-safe fallback.

## Solution Assessment

No, not as written. The formatter seam is the right boundary, but the user-facing copy should omit provider request IDs and the patch needs to be adapted to the current shared formatter layout.
