## Summary

Keep open: the problem is real and useful, but current main does not fully implement the PR's intended plain-text rewrite, and the PR as written is stale against the shared formatter layout and exposes raw provider request IDs in chat copy.

## What This Changes

The PR adds generic plain-text provider internal-error detection, rewrites helper output with the upstream support request ID, and classifies the message as a timeout/failover error.

## Best Possible Solution

Rebase the useful plain-text detection into `src/shared/assistant-error-format.ts` and the sanitizer path, keep generic provider internal errors failoverable, omit or hash raw request IDs, add focused formatter/sanitizer/failover tests plus a changelog entry, and leave broader Feishu retry behavior to #45050.

## Reproduction Assessment

Yes, by source inspection. The supplied plain-text provider message is already classified as timeout on current main, but `formatAssistantErrorText` and `sanitizeUserFacingText` do not have a generic plain-text rewrite path and can still return provider-shaped text.

## Solution Assessment

No. The detection is useful, but this PR should not merge as-is because it targets stale helper structure and displays raw provider request IDs; the safer solution is a rebased shared formatter/sanitizer change that omits or hashes the ID.
