## Summary

Keep open. Current main already covers the transient failover classification for the reported provider phrase, but it still does not implement the PR's main user-visible formatter/sanitizer behavior, and the PR is stale against the current shared formatter layout while intentionally surfacing raw provider request IDs.

## What This Changes

The PR detects plain-text OpenAI-compatible internal errors, rewrites them to friendly copy with support request-id handling, and classifies them as transient failover errors.

## Best Possible Solution

Rebase the useful behavior into the current shared formatter/sanitizer structure, keep the already-shipped timeout classification, add focused coverage for formatter, raw formatter, and sanitizer paths, and omit or hash provider request IDs unless maintainers explicitly approve a user-visible exception.

## Reproduction Assessment

Yes. The supplied sample plain-text error with no HTTP prefix is enough: current main's raw formatter falls through to trimmed text, `formatAssistantErrorText` only delegates on HTTP/API-shaped input, and the sanitizer lacks a matching branch; tests were not run because this review is read-only.

## Solution Assessment

No. The intended normalization is useful, but this patch is not the best current fix because it targets the old helper layout and exposes raw provider request IDs; the safer route is a narrow rebase into shared formatting with explicit request-id policy.

  - body: The new formatter returns the upstream support request ID in the user-facing message. Current main treats provider request IDs as diagnostic-only hash/omit values, and the related Feishu report expects raw IDs to stay in logs, so this should omit or hash the ID before displaying the copy.
