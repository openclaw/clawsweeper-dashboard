## Summary

Keep open: the PR targets a valid provider-error leak path, but the current branch is stale and its chat-facing fallback intentionally appends raw upstream request IDs, which conflicts with the repository's diagnostics privacy boundary.

## What This Changes

The PR adds generic plain-text provider internal-error detection, user-facing fallback formatting, failover classification, and focused formatter/classifier tests.

## Best Possible Solution

Rebase or replace the patch in the shared formatter/sanitizer path so generic provider internal errors return generic chat-safe copy without raw request IDs, while preserving failover classification and focused regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main can parse a provider `server_error` payload into provider message text and then push the formatted error as a channel reply; the related Feishu issue provides the observed leak path.

## Solution Assessment

No. The shared formatter direction is right, but this patch should not merge while it appends raw request IDs to chat-facing copy and remains stale against current main's formatter/test layout.
