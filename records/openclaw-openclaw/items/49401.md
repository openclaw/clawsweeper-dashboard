## Summary

Keep open: the PR targets a real formatter gap still present on current main, but it should not merge as-is because it exposes raw upstream request IDs in chat-facing copy and still lacks inspectable after-fix proof.

## What This Changes

The PR adds generic OpenAI-style internal-error detection in assistant error helpers, rewrites the displayed message while preserving the support request ID, and adds formatter/failover tests.

## Best Possible Solution

Add shared formatter/sanitizer coverage for generic provider internal errors that returns generic chat-safe copy without raw request IDs, while preserving request IDs only in hashed diagnostics/logs and keeping failover classification behavior.

## Reproduction Assessment

Yes, by source inspection. A reported Codex `server_error` payload can flow through the shared formatter into `formatAssistantErrorText`, then through `buildEmbeddedRunPayloads` and Feishu send without a channel-safe generic fallback on current main.

## Solution Assessment

No. The shared formatter direction is the right narrow seam, but this PR’s user-facing fallback should not include raw upstream request IDs and should be rebased into the current shared formatter layout.
