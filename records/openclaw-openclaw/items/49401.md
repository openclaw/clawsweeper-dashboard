## Summary

Keep this PR open. It is a plausible focused fix for a real remaining gap, but it is not merge-ready: current main already classifies the reported phrase as transient and sanitizes JSON-shaped Codex provider errors, while plain no-status provider text can still fall through; this branch is dirty against current main and exposes raw provider request IDs in chat-facing copy, which conflicts with current diagnostics/request-id policy.

## What This Changes

The PR adds generic plain-text provider internal-error detection, rewrites those errors to friendly user-facing copy with the upstream request ID, extends sanitizer/failover classification paths, and adds focused helper tests.

## Best Possible Solution

Rebase the useful detection into the current shared formatter and sanitizer, keep generic internal provider errors classified as transient, return channel-safe copy without raw request IDs unless maintainers explicitly approve an exception, add focused formatter/sanitizer/failover tests plus the required changelog entry, and coordinate the remaining Feishu behavior through #45050.

## Reproduction Assessment

Yes. The supplied plain-text sample with no HTTP prefix is enough to trace: current main classifies it as timeout, but raw formatting and sanitizer paths lack a matching rewrite and can return provider-shaped text; tests were not run because this review is read-only.

## Solution Assessment

No. The intended normalization is useful, but this PR is not the best current fix because it targets stale helper structure and displays raw provider request IDs; the safer path is a narrow rebase into the shared formatting surface with explicit request-id omission or hashing.
