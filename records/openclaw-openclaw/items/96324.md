## Summary

This PR still targets a real Google plugin hardening gap on current main, so it should not close as obsolete, but the current head is not merge-ready: it has source-level OAuth breakage, duplicate imports, risky 1 MiB caps for base64 image/audio JSON payloads, and no contributor real-behavior proof.

## What This Changes

The PR replaces 11 Google plugin `response.json()` success reads with `readResponseWithLimit` plus `JSON.parse` under 1 MiB caps across embedding batch, image, media, OAuth, and speech code paths.

## Best Possible Solution

Land the Google bounded-read hardening only after repairing the OAuth source defects, reusing the shared provider JSON helper or a justified wrapper, choosing per-surface caps that fit media-bearing responses, and adding redacted runtime proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the current PR head shows duplicate imports, undefined `data`/`errorPayload` uses, and media-bearing JSON capped at 1 MiB. The original unbounded-read risk is also source-reproducible on current main, but I did not run a live oversized-response harness.

## Solution Assessment

No. Bounded reads are the right direction, but this PR is not the best mergeable fix until it repairs OAuth parsing, avoids duplicated bounded-JSON logic, and uses cap values that fit Google media payloads.
