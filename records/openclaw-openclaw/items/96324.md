## Summary

Keep open: the work is still useful because current main has the targeted unbounded Google JSON reads, but the current PR head is not merge-ready because it discards parsed OAuth project payloads, duplicates the existing bounded JSON helper pattern, and still lacks after-fix real behavior proof.

## What This Changes

The PR replaces Google plugin JSON response reads with `readResponseWithLimit` plus `JSON.parse` under 1 MiB caps across embedding batch, image generation, media understanding, OAuth, and speech paths.

## Best Possible Solution

Use the existing provider JSON reader or a narrow shared Google wrapper, preserve the original OAuth payload variables, and add focused oversized-stream proof before merge.

## Reproduction Assessment

Yes, at source level: current main has unbounded `response.json()` success-body reads on the targeted Google plugin paths, and the current PR head plainly drops OAuth parse results before reading `data` and `errorPayload`. I did not run a live oversized Google endpoint repro in this read-only review.

## Solution Assessment

No: the bounded-read direction is appropriate, but this is not the best mergeable fix while it reimplements the existing bounded JSON helper at each call site, drops OAuth payload assignments, and lacks runtime proof. A narrower helper-based repair with proof would be safer.
