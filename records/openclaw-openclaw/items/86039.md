## What This Changes

The issue reports that disabled bundled channel/plugin config is ignored during bundled setup discovery, causing missing generated module warnings during doctor/status paths.

## Best Possible Solution

Create the owner-approved process-stable setup discovery snapshot that separates loaded, installable, bundled-display, and status ownership, then have doctor/status honor explicit disabled config before generated setup modules load.

## Reproduction Assessment

Yes, at source level: current main still has status/setup discovery calling a config-less bundled setup fallback that can load generated modules for explicitly disabled bundled channels. I did not run the Windows doctor scenario because this review is read-only.

## Solution Assessment

No for the originally suggested direct root-list filter: the latest closed fix PR confirms the bug but says the one-sided filter is unsafe. The better solution is the setup discovery snapshot refactor that preserves separate ownership for loaded adapters, installable entries, bundled display metadata, and status generation.
