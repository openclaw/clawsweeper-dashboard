## Summary

Keep open: current main and the latest release still use config-less bundled setup fallback discovery, and the two linked fix PRs are closed unmerged after review confirmed the bug but rejected the narrow filter as unsafe.

## What This Changes

The issue reports noisy bundled channel setup warnings when explicitly disabled bundled channels still get enumerated and their generated setup modules are missing.

## Best Possible Solution

Build the owner-approved setup discovery snapshot that separates loaded, installable, bundled-display, and status ownership, then make doctor/status honor explicit disabled config before any generated setup module is loaded.

## Reproduction Assessment

Yes, at source level: current main still has channel status calling config-less setup fallback discovery, which broadly enumerates bundled setup ids before generated setup module loading. I did not run the reported Windows doctor scenario because this review is read-only.

## Solution Assessment

No for the originally suggested direct root-list filter: the latest closed fix PR confirms the bug but says the one-sided filter is unsafe. The better solution is the setup discovery snapshot refactor that preserves separate ownership for loaded adapters, installable entries, bundled display metadata, and status generation.
