## Summary

Keep open: current main and the latest release still have a config-less bundled setup fallback path that can load disabled bundled setup entries, and the latest linked fix PR was closed unmerged after maintainer review called for a broader setup discovery snapshot instead of another narrow filter.

## What This Changes

The issue asks OpenClaw to skip explicitly disabled bundled channel/plugin setup entries before generated setup modules are loaded, so doctor/status stop warninging about missing modules for disabled channels.

## Best Possible Solution

Build the owner-approved process-stable setup discovery snapshot that separates loaded, installable, bundled-display, and status ownership, then make doctor/status honor explicit disabled config before generated setup modules are loaded.

## Reproduction Assessment

Yes, at source level: collectChannelStatus still calls config-less setup discovery, which can call listBundledChannelSetupPlugins and load generated setup entries for disabled bundled channels. I did not run the Windows doctor scenario in this read-only review.

## Solution Assessment

No for the originally suggested direct root-list filter: maintainer review of the latest candidate PR says the bug is real but needs a setup discovery snapshot that preserves separate ownership buckets.
