## Summary

Keep open: current main and the latest release still use config-less bundled setup fallback discovery, and the two linked fix PRs were closed unmerged after maintainer review confirmed the bug but rejected the narrow filter as unsafe.

## What This Changes

The issue requests that bundled channel setup discovery honor explicit `channels.<id>.enabled=false` and `plugins.entries.<id>.enabled=false` before loading generated setup modules.

## Best Possible Solution

Create a process-stable setup discovery snapshot that separates loaded, installable, bundled-display, and status ownership, then have doctor/status honor explicit disabled config before any generated setup module is loaded.

## Reproduction Assessment

Yes, at source level: current main still calls config-less setup fallback discovery and then broadly enumerates bundled setup ids before loading setup entries. I did not run the reported Windows doctor scenario in this read-only pass.

## Solution Assessment

No for the originally suggested direct root-list filter: maintainer review on the closed fix PR found that approach too one-sided. The safer solution is the setup discovery snapshot refactor described in the maintainer comment.
