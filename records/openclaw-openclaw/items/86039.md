## Summary

Keep open, but narrow the scope: current main still has a config-unaware bundled setup fallback that can load every bundled setup entry, while the doctor-specific channel-doctor path already skips disabled channels/plugins. The remaining useful work is to make setup/status fallback enumeration config-aware and add a regression test for disabled config plus a missing generated setup module.

## What This Changes

The issue asks OpenClaw to suppress bundled channel setup warnings for explicitly disabled channels/plugins when generated modules are missing.

## Best Possible Solution

Add a config-aware bundled setup-plugin enumeration path for setup/status callers, reuse the existing disabled-config filtering helper, and cover the missing setup-entry module case without changing plugin API contracts.

## Reproduction Assessment

Yes, at source level for the generic setup/status fallback: `collectChannelStatus()` can default into `listChannelSetupPlugins()`, which falls back to unfiltered bundled setup loading. The exact `openclaw doctor` command path is less certain because channel-doctor filtering already skips disabled entries.

## Solution Assessment

Yes, with narrowed scope: reusing the existing config-aware bundled setup-feature filter for generic setup enumeration is the narrowest maintainable fix. A runtime-specific doctor rewrite is not the right first move unless a live repro shows a separate doctor path.
