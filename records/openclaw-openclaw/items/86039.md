## Summary

Keep open: current main and the latest release still have config-less bundled setup fallback discovery, and the linked fix PRs were closed unmerged after maintainer review confirmed the bug but rejected the narrow filter shape as unsafe.

## What This Changes

The issue asks for explicitly disabled bundled channels/plugins to be skipped before setup fallback discovery tries to load generated bundled setup modules.

## Best Possible Solution

Implement the owner-approved setup discovery snapshot that separates loaded, installable, bundled-display, and status ownership, then have doctor/status honor explicit disabled config before any generated setup module is loaded.

## Reproduction Assessment

Yes, at source level: current main still has status/setup discovery calling a config-less bundled setup fallback that can load generated setup modules for explicitly disabled bundled channels. I did not run the Windows doctor scenario because this review is read-only.

## Solution Assessment

No for the originally suggested direct root-list filter: the latest closed fix PR confirms the bug but says that one-sided shape is unsafe. The better solution is the setup discovery snapshot refactor that preserves separate ownership for loaded adapters, installable entries, bundled display metadata, and status generation.
