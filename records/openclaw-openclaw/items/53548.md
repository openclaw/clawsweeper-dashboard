## Summary

Keep open. Current main and the latest release still reject `mode="session"` without `thread: true` in both native subagent and ACP spawn paths, and the active related PR only covers a narrower hub-delegated ACP contract rather than the full decoupling request.

## What This Changes

The issue asks OpenClaw to allow persistent `mode="session"` sessions without requiring channel thread binding, with parent-driven follow-up via `sessions_send`.

## Best Possible Solution

Define an explicit non-thread persistent-session contract, likely aligned with delegated or affinity-style ownership, then update ACP/native spawn, `sessions_send`, listing/close lifecycle, docs, and focused tests before closing this issue.

## Reproduction Assessment

Yes, source-level: current main returns before session creation for both native and ACP `mode="session"` calls when `thread: true` is absent. I did not run a live channel repro because the rejection is explicit in source and covered by existing tests.

## Solution Assessment

Unclear as proposed. Decoupling persistence from threads is a useful direction, but silently relaxing `mode="session"` is not the best fix until maintainers choose the ownership, label, delivery, and lifecycle semantics.
