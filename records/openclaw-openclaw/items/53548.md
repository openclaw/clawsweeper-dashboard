## Summary

Keep open. Current main still rejects unthreaded persistent native and ACP `sessions_spawn` calls, tests and docs still encode the `thread=true` requirement, and open PR #39372 is an implementation candidate rather than shipped behavior.

## What This Changes

Allow `sessions_spawn` with `mode:"session"` to create persistent native and ACP child sessions without requiring `thread:true`, while preserving optional thread-bound routing when requested.

## Best Possible Solution

Review and harden PR #39372 or a replacement so native and ACP persistent sessions can exist without channel thread binding, return stable continuation metadata for `sessions_send`, preserve existing `thread:true` behavior, avoid unintended ACP inline delivery for headless sessions, and update tests plus public docs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call native or ACP `sessions_spawn` with `mode:"session"` and no `thread:true`; current main returns explicit thread-required errors before hook or gateway startup, and native tests assert that behavior.

## Solution Assessment

Yes, decoupling session persistence from channel thread binding is the narrow maintainable direction if maintainers accept the product behavior. The safe fix is broader than removing guards: it should also cover routing, lifecycle records, continuation metadata, tests, and docs.
