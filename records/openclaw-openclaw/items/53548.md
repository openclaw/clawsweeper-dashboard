## Summary

Keep open. Current main and latest release still reject `mode="session"` without `thread: true` in both native subagent and ACP spawn paths, and the active related PR only addresses a narrower `delegate: true` hub-worker contract rather than this broader decoupling request.

## What This Changes

Request to allow persistent `sessions_spawn`/ACP `mode="session"` sessions without requiring a channel thread binding.

## Best Possible Solution

Define an explicit non-thread persistent-session contract, then update ACP/native spawn, follow-up targeting, lifecycle cleanup, docs, and focused tests before removing the current fail-closed guards.

## Reproduction Assessment

Yes, source-level: current main returns before session creation for both native and ACP `mode="session"` calls when `thread: true` is absent. Existing tests also assert the native rejection path; I did not run live channel proof in this read-only review.

## Solution Assessment

Unclear as proposed. Decoupling persistence from threads is a useful direction, but silently relaxing `mode="session"` is not the best fix until maintainers choose ownership, label, delivery, close, and lifecycle semantics.
