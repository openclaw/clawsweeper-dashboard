## Label Justifications

- `P2`: The request addresses a real orchestration limitation with limited blast radius, but it is a feature/product-contract change rather than an urgent regression.
- `impact:session-state`: The issue is about whether persistent child sessions can exist and remain reachable without a bound channel thread.
- `impact:message-loss`: The requested behavior changes how child-session output and follow-up messages are routed when no channel thread binding exists.

## Summary

Keep open: current main and v2026.5.19 still reject unthreaded native and ACP `mode="session"` spawns, and the linked implementation at https://github.com/openclaw/openclaw/pull/39372 remains an open, conflicting product-contract change rather than a merged fix.

## What This Changes

The issue asks OpenClaw to allow persistent native and ACP `sessions_spawn` sessions without requiring channel thread binding.

## Best Possible Solution

Land a rebased, maintainer-approved update to https://github.com/openclaw/openclaw/pull/39372 or a replacement that defines headless native/ACP session delivery, updates schema/docs/tests, and proves non-threaded plus thread-bound behavior.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.19 shows native and ACP guards rejecting `mode="session"` without `thread=true`, and existing spawn-hook coverage asserts the native rejection path.

## Solution Assessment

Unclear as a direct guard removal. The maintainable solution needs a defined headless native/ACP delivery contract, schema/docs updates, and real behavior proof across a non-threaded channel and a supported thread-bound channel.
