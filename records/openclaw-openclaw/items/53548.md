## Summary

Keep open. Current main still rejects unthreaded persistent `sessions_spawn` in both native subagent and ACP paths, tests and public docs still encode the `thread=true` requirement, and open PR #39372 remains an implementation candidate rather than shipped behavior.

## What This Changes

The issue requests allowing persistent `sessions_spawn` sessions to run without requiring channel thread binding, so parent agents can continue them with `sessions_send` on channels without native thread support.

## Best Possible Solution

Review and harden PR #39372 or a replacement so native and ACP persistent sessions can exist without thread binding, while preserving existing `thread:true` behavior, returning stable continuation metadata for `sessions_send`, preventing unintended ACP inline delivery for headless sessions, and updating tests plus public docs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call native or ACP `sessions_spawn` with `mode:"session"` and no `thread:true`; current main returns the explicit thread-required errors, and tests assert that behavior.

## Solution Assessment

Yes, decoupling session persistence from channel thread binding is the narrow maintainable direction if maintainers accept the product behavior. Removing the guards alone is incomplete; the fix should also cover native/ACP delivery routing, lifecycle records, docs, and regression tests.
