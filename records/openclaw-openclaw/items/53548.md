## Summary

Keep open: current main and v2026.6.10 still couple persistent `mode="session"` to `thread: true` in both native and ACP spawn paths, while related work covers narrower channel, affinity, or thread-follow-up slices rather than the broad core contract requested here.

## What This Changes

The issue requests allowing `sessions_spawn` persistent session mode without requiring channel thread binding, for both native subagents and ACP sessions.

## Best Possible Solution

Define a first-class non-thread persistent-session contract, then update native spawn, ACP spawn, tool schema, follow-up targeting, lifecycle cleanup, docs, and focused tests around that contract.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.10` still reject both native and ACP `sessions_spawn` calls with `mode="session"` unless `thread: true` is set, and tests/docs preserve that contract.

## Solution Assessment

Unclear. Decoupling persistence from thread binding is plausible, but the best fix needs an explicit non-thread ownership, delivery, cleanup, and ACP lifecycle contract rather than direct guard removal.
