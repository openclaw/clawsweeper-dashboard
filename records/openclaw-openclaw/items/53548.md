## Summary

Keep open. Current main and the latest release still reject `mode="session"` without `thread: true` in both native subagent and ACP `sessions_spawn` paths, while the active related PR explicitly covers only a narrower hub-delegated ACP worker contract.

## What This Changes

The issue requests allowing persistent `sessions_spawn` session mode without requiring channel thread binding.

## Best Possible Solution

Define a first-class non-thread persistent-session contract, then update native spawn, ACP spawn, follow-up targeting, lifecycle cleanup, docs, and focused tests around that contract.

## Reproduction Assessment

Yes. Source inspection shows both native and ACP `sessions_spawn` return before creating a persistent session when `mode="session"` is requested without `thread: true`; I did not run a live channel scenario in this read-only review.

## Solution Assessment

Unclear. Decoupling persistence from threads is a useful direction, but current code and the active related PR show that follow-up, visibility, labels, and lifecycle semantics need maintainer design before changing the fail-closed guards.
