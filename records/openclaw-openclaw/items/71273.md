## Summary

Keep open. The Kimi Coding repeat-tool-call failure remains plausible on current main, but the proposed stopReason root cause is contradicted by the current wrapper guard and regression coverage; the missing blocker is a redacted trajectory or replay fixture that proves which layer repeats the prior read/exec calls.

## What This Changes

The issue requests a fix for Kimi Code via the `kimi-coding` provider repeatedly issuing identical tool calls after tool results instead of producing a final answer.

## Best Possible Solution

Keep the issue open until a redacted Kimi Coding trajectory or focused replay fixture proves the repeated tool-result-to-tool-call cycle. Then add regression coverage and land the narrowest provider-owned fix in `extensions/kimi-coding/stream.ts`, or a generic dispatch/replay fix only if the replay shows old tool calls being re-executed without new structured tool calls.

## Reproduction Assessment

Unclear: the issue gives concrete manual steps and a useful log shape, but there is no attached redacted trajectory or replay fixture to reproduce it against current main. I did verify the narrower negative path that normal final assistant text is not rewritten to `toolUse`.

## Solution Assessment

No: the proposed stopReason guard is not the best proven fix because current main already only rewrites after parsing tagged Kimi tool-call markup. The safer path is transcript-first regression coverage, then a provider-owned or generic loop-dispatch fix based on that replay.
