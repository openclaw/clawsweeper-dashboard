## Summary

Keep open. Current main still contains the Kimi Coding provider and no targeted merged fix for the reported repeated tool-call loop; the issue is recent and has concrete log evidence, while the proposed stop-reason root cause is contradicted by current source, tests, and the closed unmerged paired PR review.

## What This Changes

The issue requests a fix for Kimi Code via the `kimi` provider repeatedly reissuing identical tool calls after tool results instead of producing a final `stop` answer.

## Best Possible Solution

Keep this open until maintainers capture a redacted Kimi Coding trajectory or replay fixture for the post-tool-result loop, then fix the owning Kimi provider stream/replay path, or a generic loop guard only if the fixture proves the problem is not provider-specific, with focused regression coverage.

## Reproduction Assessment

Unclear: the issue provides concrete steps and repeated-log evidence, but this review found no current replay fixture or live transcript proving the post-tool-result message shape. Current Kimi stream tests show plain final assistant text remains `stop`, so the proposed root-cause reproduction is not high-confidence.

## Solution Assessment

No: the stop-reason-only fix from the paired PR is not the best proven solution because current main already gates the rewrite on parsed tagged tool calls, and adding a `role` gate can regress role-less tagged Kimi tool-call messages. A transcript-first provider or replay regression is the safer path.
