## Summary

Keep open. Current main still leaves the reported Kimi Coding repeat-tool-call failure plausible, but the proposed root cause is contradicted by source and regression coverage: normal final assistant text is left as `stop`, and the Kimi wrapper only flips `stopReason` after parsing tagged tool-call markup.

## What This Changes

The issue reports that the Kimi Coding provider repeats identical tool calls after tool results instead of producing a final assistant answer.

## Best Possible Solution

Keep this open until a redacted Kimi Coding trajectory or focused replay fixture proves whether the repeat originates in the Kimi stream wrapper, replay history, model output, or generic tool dispatch; then land the narrowest provider-owned or generic loop fix with regression coverage.

## Reproduction Assessment

Unclear: the issue gives concrete manual steps and a useful log pattern, but there is no checked-in replay fixture or redacted current-main trajectory. Source inspection reproduces the negative check that ordinary final assistant text is not rewritten to `toolUse`.

## Solution Assessment

No: the proposed stopReason guard from the paired closed PR is not the best proven fix because current main already leaves normal final text unchanged. The safer path is transcript-first regression coverage, then a Kimi wrapper or generic dispatch fix based on that replay.
