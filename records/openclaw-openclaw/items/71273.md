## Summary

Keep open. Current main still has the Kimi Coding provider path, but the reported one-line root cause is not supported by the current source or tests: the stop-reason rewrite only runs after tagged Kimi tool-call markup is parsed, and normal final assistant text is preserved. The repeated-call report remains plausible and recent, but it needs a raw redacted transcript or replay fixture before a safe repair can be made.

## What This Changes

The issue reports that the Kimi Coding provider repeats identical tool calls after tool results instead of producing a final assistant response.

## Best Possible Solution

Keep the issue open until there is a redacted Kimi Coding trajectory/replay that shows tool results followed by repeated identical tool calls, then add a provider-owned regression test and fix the Kimi stream or replay path only if the fixture proves the bug lives there; otherwise fix the generic tool-dispatch loop layer with the same fixture.
