## Summary

Keep open. Current main still has the Kimi Coding provider path and no landed regression fixture or live proof for this exact infinite-loop report; the proposed stop-reason root cause is contradicted by current source and tests, but repeated Kimi tool calls remain plausible because the generic loop guard is opt-in by default.

## What This Changes

The issue reports that the Kimi Coding provider can repeat identical tool calls indefinitely after receiving tool results instead of returning a final assistant answer.

## Best Possible Solution

Keep the issue open until a redacted Kimi Coding trajectory or replay fixture shows the repeated tool-result-to-tool-call cycle, then add a provider-owned regression test in `extensions/kimi-coding/stream.test.ts` if the stream wrapper is at fault, or a generic agent-loop regression if the replay proves repeated calls reach dispatch correctly.

## Reproduction Assessment

Unclear: the report gives concrete manual steps and a useful log shape, but this review found no checked-in replay fixture or live Kimi credential proof. Current source and tests verify normal final assistant text remains `stop`, so the reported infinite loop is not reproduced from the claimed root cause.

## Solution Assessment

No: the suggested stop-reason guard is not the best proven fix because current main already avoids rewriting normal final text unless tagged Kimi tool-call markup was parsed. The safer path is transcript-first regression coverage before changing provider or generic loop behavior.
