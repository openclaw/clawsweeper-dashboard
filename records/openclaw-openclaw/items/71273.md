## Summary

Keep open. Current main still leaves the reported Kimi Coding repeat-tool-call failure plausible, but the issue's proposed stopReason root cause is contradicted by source and tests: normal assistant text is left unchanged, and the stop-to-toolUse rewrite only runs after tagged Kimi tool-call markup is parsed. The remaining blocker is a redacted Kimi trajectory or replay fixture showing whether the repeat is in the Kimi stream wrapper, replay history, or generic tool-dispatch loop.

## What This Changes

The issue reports that the bundled Kimi Coding provider can repeat identical tool calls indefinitely after tool results instead of producing a final assistant answer.

## Best Possible Solution

Keep this open until a redacted Kimi Coding trajectory or focused replay fixture proves the repeated tool-result-to-tool-call cycle. Then land the narrowest fix in `extensions/kimi-coding/stream.ts` if the wrapper emits incorrect assistant messages, or in the generic agent loop/dispatch path if replay shows previous tool calls are being re-executed without new structured tool calls.

## Reproduction Assessment

Unclear: the issue gives concrete manual steps and a useful log shape, but there is no checked-in replay fixture or redacted trajectory to reproduce it against current main. Current source and tests do reproduce the negative check that normal final assistant text is not rewritten to `toolUse`.

## Solution Assessment

No: the proposed stopReason guard is not the best proven solution because current main already only rewrites after parsing tagged Kimi tool-call markup. The safer path is transcript-first regression coverage, then a provider-owned or generic loop-dispatch fix based on that replay.
