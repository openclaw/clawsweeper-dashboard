## Summary

Keep open: this focused PR mitigates two named local TUI commands, but it is not merge-ready because other advertised shared commands can still fall through to the embedded model path and the supplied proof is only static/unit-level.

## What This Changes

The PR adds `/status` and `/compact` cases to `src/tui/tui-command-handlers.ts` so local mode emits an unsupported-command system message while gateway mode still forwards the raw command.

## Best Possible Solution

Make local TUI help, autocomplete, and dispatch agree by handling, hiding, or explicitly rejecting every advertised shared command before embedded send, then prove it with the local PTY path.

## Reproduction Assessment

Yes from source: slash input enters `handleCommand`, local discovery appends shared commands, unhandled names call `sendMessage(raw)`, and the embedded backend passes message text into `agentCommandFromIngress`. I did not run the local PTY lane in this read-only review.

## Solution Assessment

No. The two-command switch case is a plausible mitigation, but the best fix is a local command boundary that covers every advertised unsupported shared command before embedded send.
