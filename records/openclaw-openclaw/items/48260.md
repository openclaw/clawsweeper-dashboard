## Summary

Keep this PR open: current main still lacks the Active Time overview card, but the current branch has a P2 metric correctness bug and no viewable real behavior proof, so it is not ready to merge or close as obsolete.

## What This Changes

This PR adds localized Active Time labels and a Usage Overview card, clips usage insight durations to selected day/hour filters, and adds focused usage metrics/rendering tests.

## Best Possible Solution

Keep the UI-only Active Time direction, but align filtered active-time and throughput semantics or hide throughput when totals are not sliced, then attach real after-fix UI proof before merge.

## Reproduction Assessment

Yes for the blocking review finding: source inspection of the PR head shows selected-hour totals use full matching sessions while duration is clipped by selectedHours. I did not run the UI because this was a read-only review.

## Solution Assessment

No. The Active Time card is a reasonable narrow UI feature, but the current implementation clips only the denominator used by Throughput; the safer fix is to keep numerator and denominator on the same filtered basis or suppress throughput for that case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep filtered throughput aligned with active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing selectedHours into buildUsageInsightStats clips durationSumMs to the selected hour, but displayTotals for hour filters is still computed from full matching sessions. A session that only overlaps one selected hour can therefore show full-session tokens/cost over only the clipped active time, inflating the Throughput card as users narrow the filter.
