## Summary

Keep this PR open: current main still does not surface a separate Active Time card, so the branch contains unique user-facing UI work, but the latest head still has a source-proven throughput mismatch when hour filters clip active duration while totals remain whole-session, and the external contributor has not attached inspectable real UI proof.

## What This Changes

This PR adds an Active Time card to the Control UI Usage Overview, clips duration stats for selected day/hour filters, and updates focused tests plus generated Control UI i18n strings/metadata.

## Best Possible Solution

Land a narrow Active Time overview card after making throughput use a consistent numerator and denominator for filtered windows, then attach redacted real UI proof and rerun focused Usage UI, metrics, and i18n validation.

## Reproduction Assessment

Yes for the blocking PR finding: PR-head source shows selected-hour totals still use whole-session usage while active duration is clipped to the selected window. The added selected-hour test also asserts 120 tokens over one clipped hour, preserving the mismatch.

## Solution Assessment

No. The Active Time card direction is reasonable, but the patch should either align filtered throughput totals with the clipped active-time window or suppress throughput when totals and duration are not sliced on the same basis.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Align filtered throughput with active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing selected day/hour filters into `buildUsageInsightStats` clips active duration, but selected-hour `displayTotals` still sums each matching session's full usage totals. A partially overlapping session can therefore show all tokens and cost over only the clipped active minutes, inflating Throughput as users narrow the hour filter.
