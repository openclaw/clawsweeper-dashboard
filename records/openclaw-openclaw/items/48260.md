## Summary

This PR should stay open because the latest branch still introduces a metrics consistency bug for selected-hour filters and still lacks inspectable real behavior proof. The Active Time card itself is a reasonable UI feature, but the current implementation can display throughput calculated from full-session tokens over clipped active minutes, which is misleading when users narrow the time range.

## What This Changes

The PR adds an Active Time summary card to the Control UI Usage Overview, related duration metrics coverage, localized usage overview strings, and generated i18n metadata updates.

## Best Possible Solution

Land a narrow Active Time card after aligning filtered throughput with the same time-window basis, or suppressing throughput when totals and active duration are not comparable, and attach redacted real UI proof.

## Reproduction Assessment

Yes for the blocking PR finding: PR-head source and the added selected-hour test show full-session totals divided by clipped active time for selected-hour filters. The user-visible Active Time feature itself is a new UI capability rather than a bug reproduction.

## Solution Assessment

No. The Active Time card direction is maintainable, but the patch should keep throughput totals and active duration on the same filtered basis or avoid showing throughput for clipped windows where the numerator cannot be clipped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Align filtered throughput with active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing selected day/hour filters into `buildUsageInsightStats` clips active duration, but selected-hour `displayTotals` still sums each matching session's full usage totals. A partially overlapping session can therefore show all tokens and cost over only the clipped active minutes, inflating Throughput as users narrow the hour filter.
  - confidence: 0.9
