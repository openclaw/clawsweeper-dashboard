## Summary

Keep this PR open: current main still lacks the Active Time card, but the latest branch still has a source-level throughput semantics bug and lacks inspectable real behavior proof before merge.

## What This Changes

The PR adds a localized Active Time card to the Control UI Usage Overview, clips duration stats to selected day/hour filters, adds focused metrics/render tests, and refreshes generated locale metadata.

## Best Possible Solution

Land a narrow Active Time card after aligning throughput with the same filtered-duration semantics or hiding throughput under partial time filters, with generated i18n and inspectable UI proof kept in sync.

## Reproduction Assessment

Yes, from source inspection: filtering a two-hour session to one selected hour clips duration to 60 minutes while displayTotals can still carry the full 120 tokens, and the added test expects that inflated 2 tok/min path.

## Solution Assessment

No. The Active Time card direction is reasonable, but the implementation should compute matching in-range token/cost numerators or suppress throughput when the denominator is clipped and the numerator is not.
