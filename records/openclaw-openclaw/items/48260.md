## Summary

Keep open: current main still lacks the Active Time card, and the latest PR head remains a plausible feature but has a blocking metric-semantics bug plus missing contributor real-behavior proof.

## What This Changes

This PR adds a localized Active Time card to the Control UI Usage Overview and passes selected day/hour filters into usage insight duration calculations, with focused UI metrics tests and generated locale metadata.

## Best Possible Solution

Land a narrow localized Active Time card after aligning filtered duration-based throughput with the same token/cost window, keeping generated locale metadata consistent and requiring redacted UI proof.

## Reproduction Assessment

Yes for the blocking review finding: PR-head source shows selected-hour totals still use whole-session usage while the new duration calculation is clipped to the selected window. The PR's own test asserts whole-session tokens over a clipped one-hour duration.

## Solution Assessment

No. The Active Time card direction is reasonable, but the patch should either align the throughput numerator with the clipped active-time window or suppress throughput when totals are not sliced to the same basis.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87
