## Summary

Keep open. Current main still lacks the Active Time card, but this branch should not merge as-is because filtered throughput can be inflated and the external PR has no viewable real behavior proof.

## What This Changes

This PR adds an Active Time card to Usage Overview, adds localized UI strings, and changes usage insight duration calculations/tests to clip active time by selected day/hour filters.

## Best Possible Solution

Keep the UI-only Active Time direction, but align filtered active-time and throughput semantics or hide throughput when totals are not sliced, then attach real after-fix UI proof before merge.

## Reproduction Assessment

Yes for the blocking review finding: source inspection gives a concrete path with one multi-hour session, a selected hour filter, full-session displayTotals, and clipped durationSumMs. I did not run the UI because this review was read-only.

## Solution Assessment

No. The narrow Active Time card is reasonable, but the current implementation clips only the denominator used by throughput under hour filters; the safer fix is to keep numerator and denominator on the same filtered basis or suppress throughput for that case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep filtered throughput aligned with active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing selectedHours into buildUsageInsightStats clips durationSumMs to the selected hour, but displayTotals for hour filters is still computed from the full matching sessions. A 09:30-11:30 session filtered to hour 10 can therefore show full-session tokens/cost over only one hour of active time, inflating the Throughput card as users narrow the filter.
