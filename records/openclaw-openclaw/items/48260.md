## Summary

Keep open: current main still lacks a separate Active Time overview card, so the PR has useful unmerged UI work, but the current branch still has a blocking metric-consistency issue and lacks inspectable real behavior proof.

## What This Changes

This PR adds localized Active Time summary UI and tests, and updates usage insight stats to clip active duration to selected day/hour filters.

## Best Possible Solution

Land a narrow Active Time card after making throughput, active time, and displayed totals use the same filtered range semantics, with visible proof from the Usage page after the fix.

## Reproduction Assessment

Yes. Source inspection and the added test show the current PR can report 120 full-session tokens over only 60 clipped minutes when an hour filter is active, inflating Throughput while Active Time claims the selected range.

## Solution Assessment

No. The UI feature direction is reasonable, but the patch should either compute token/cost totals for the same clipped window or avoid showing throughput when duration is clipped without matching totals.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align throughput totals with clipped active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing day/hour filters here clips `durationSumMs`, but `displayTotals` is still built from whole sessions for hour filters and selected-session filters. A session that only overlaps part of the selected hour can contribute all tokens/cost while only the overlapping minutes are used as the denominator, inflating the Throughput card for the filtered range.
