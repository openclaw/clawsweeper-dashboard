## Summary

Keep this PR open. The feature remains useful and the latest branch appears to have addressed the earlier generated i18n metadata concern, but the latest diff still leaves a blocking throughput math bug: active duration is clipped to the selected day/hour window while token and cost totals can still come from the full matching sessions.

## What This Changes

The PR adds an Active Time card to the Control UI Usage Overview using existing duration stats, clips active duration to selected day/hour filters, adds focused usage rendering/metrics tests, and updates locale bundles plus generated i18n metadata.

## Best Possible Solution

Land the Active Time card as a narrow Control UI improvement, but first make throughput use the same filtered scope as the active-duration denominator, or hide throughput whenever token/cost totals cannot be sliced to that scope. Keep the generated i18n metadata in sync with the English source strings.

## Reproduction Assessment

Yes. A session spanning 09:30-11:30 with hour 10 selected exercises the bug: the PR clips active duration to 60 minutes, but displayTotals still includes the full session tokens/cost, so throughput is inflated.

## Solution Assessment

No. The UI direction is reasonable, but this patch is not the best merge path until throughput numerators and duration denominators are aligned or throughput is suppressed for unsupported slices.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep throughput totals aligned with clipped duration:** `ui/src/ui/views/usage-metrics.ts:655-658`
  - body: When hour filters are active, `displayTotals` is still built from each matching session's full token and cost totals, but this call now clips `durationSumMs` to the selected day/hour window. Throughput then divides full-session totals by a partial-session duration, so the card can spike as users narrow the filter.
