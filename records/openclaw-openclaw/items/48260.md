## Summary

Keep open: current main still does not surface a separate Active Time card, so this PR contains useful feature work, but the latest head is not merge-ready because selected-hour throughput can still divide full-session totals by clipped active duration and contributor proof is missing.

## What This Changes

This PR adds a Control UI Usage Overview Active Time card, clips insight duration by selected day/hour filters, adds focused usage metrics/render tests, and regenerates usage i18n strings and metadata.

## Best Possible Solution

Land a narrow Active Time card using existing duration stats after aligning filtered totals and duration to the same window and attaching redacted real UI proof.

## Reproduction Assessment

Yes for the blocking PR finding by source inspection: selected-hour totals are still full-session totals while the PR clips active duration by the selected filter window. The added test also asserts the inflated 120-token-over-60-minute case.

## Solution Assessment

No. The Active Time card direction is reasonable, but the patch should compute filtered totals on the same clipped basis as active duration, or suppress throughput when the numerator and denominator are not aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Align filtered throughput with active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing selected day/hour filters into `buildUsageInsightStats` clips active duration, but selected-hour `displayTotals` still sums each matching session's full usage totals. A partially overlapping session can therefore show all tokens and cost over only the clipped active minutes, inflating Throughput as users narrow the hour filter.
  - confidence: 0.9
