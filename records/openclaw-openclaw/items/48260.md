## Summary

The central feature is still not on current main, and the latest branch still has a metric-semantics blocker: selected-hour active time is clipped, but throughput still uses whole-session totals. The PR also lacks inspectable real behavior proof, so it should stay open for contributor follow-up rather than close as cleanup.

## What This Changes

The PR adds a localized Active Time summary card to the Control UI Usage Overview, clips usage insight durations to selected day/hour filters, updates focused usage overview tests, and refreshes generated locale metadata.

## Best Possible Solution

Land a narrow localized Active Time card after making duration-based throughput use a consistent numerator/denominator for selected day/hour filters, then validate the focused UI metrics/rendering path and attach redacted real UI proof.

## Reproduction Assessment

Yes for the blocking review finding: PR-head source shows selected-hour totals still use whole-session usage while the new duration calculation is clipped to the selected window. The PR test at ui/src/ui/views/usage-metrics.test.ts:371-374 asserts all 120 tokens over one clipped hour.

## Solution Assessment

No. The Active Time card direction is reasonable, but the patch should either align the throughput numerator with the clipped active-time window or suppress throughput when totals are not sliced to the same basis.
