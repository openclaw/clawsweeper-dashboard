## Summary

Keep open. Current main still lacks the Active Time card, and the latest PR head still has a blocking metrics consistency issue plus missing inspectable real behavior proof.

## What This Changes

This PR adds an Active Time card to the Control UI Usage Overview, adds focused usage metrics/render tests, and updates Control UI i18n strings and metadata.

## Best Possible Solution

Land a narrow Active Time card after aligning throughput totals with the same clipped time window, or suppressing throughput for sliced filters, and attach redacted real UI proof.

## Reproduction Assessment

Yes for the blocking PR finding from source inspection: a partially overlapping selected-hour session can still contribute full tokens/cost while duration is clipped to the selected hour. I did not run a live browser repro in this read-only review.

## Solution Assessment

No. The Active Time card direction is reasonable, but the current patch should not compute throughput from full-session totals over a clipped active-time denominator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Align filtered throughput with active time:** `ui/src/ui/views/usage.ts:325-329`
  - body: Passing selected day/hour filters into `buildUsageInsightStats` clips active duration, but selected-hour `displayTotals` still sums each matching session's full usage totals. A partially overlapping session can therefore show all tokens and cost over only the clipped active minutes, inflating Throughput as users narrow the hour filter.
