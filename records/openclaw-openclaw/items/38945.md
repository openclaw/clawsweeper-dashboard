## Summary

Keep open: the Unicode tokenization bug is still valid, but this PR is stale against current main because it edits legacy `src/memory` files while active memory search now lives in `memory-core`, and a direct port would regress current CJK MMR behavior.

## What This Changes

The PR updates legacy MMR and hybrid FTS tokenizer helpers and tests to NFC-normalize input and include Unicode letters, marks, numbers, and underscores.

## Best Possible Solution

Land a narrow `memory-core` repair that adds NFC plus Unicode mark/letter/number token support in all active FTS paths while preserving CJK MMR unigram/bigram behavior and adding focused regressions plus a changelog entry.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: current active regexes still omit `\p{M}`/NFC in FTS paths and keep MMR ASCII-only outside the CJK special case. I did not run the full app or test suite because this review was read-only.

## Solution Assessment

No, not as currently proposed. The tokenizer direction is right, but the patch must move from deleted legacy `src/memory` files into active `extensions/memory-core` files and preserve current CJK token behavior.
