## Summary

Keep open: the underlying Unicode memory-search bug is still source-reproducible in the active memory-core implementation, but this PR cannot merge as-is because it edits removed legacy files and lacks contributor real behavior proof.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizers/tests to add NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC handling into active `extensions/memory-core` FTS, search-planner, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real memory-search proof.

## Reproduction Assessment

Yes, at source level. Current memory-core FTS/search-planner tokenization still excludes combining marks and NFC normalization, and active MMR only handles ASCII plus CJK-family characters; I did not run a live `openclaw memory` flow in this read-only review.

## Solution Assessment

No. The Unicode/NFC direction is right, but this branch edits inactive legacy files and a straight MMR regex port would not preserve current CJK token granularity.
