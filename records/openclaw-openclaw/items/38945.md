## Summary

Keep open. The contribution targets a real Unicode memory-search bug, but the PR is stale against current main: user-facing memory search now lives in `extensions/memory-core`, while the PR changes legacy `src/memory` files that are no longer present on main. Current main still has the underlying Unicode/NFC gaps in the active MMR, FTS, and shared search-planner tokenizers, so this should be updated or replaced rather than closed.

## What This Changes

This PR updates legacy `src/memory` MMR and FTS tokenizer helpers and tests to NFC-normalize input and tokenize Unicode letters, marks, numbers, and underscores.

## Best Possible Solution

Land a narrow `extensions/memory-core` fix that NFC-normalizes before tokenization, includes Unicode letters/marks/numbers for non-CJK scripts, preserves existing CJK unigram/bigram MMR tokens, updates both FTS tokenizer paths including `manager-search.ts`, and adds focused regressions for accented Latin, combining-mark scripts, and NFC/NFD equivalence.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current active regexes still fragment accented Latin, NFD text, and combining-mark scripts, and the PR's original stale files are absent from main.

## Solution Assessment

No, not as currently proposed. The tokenizer approach is directionally right, but the best fix now is to port it into `extensions/memory-core` and the shared search planner while preserving current CJK behavior.
