## Summary

Keep open: the reported Unicode memory-search bug still appears valid on current main, but this PR cannot merge as written because the active runtime moved to `memory-core` and needs a careful port that preserves current CJK token behavior plus real behavior proof.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizers/tests to use Unicode letter/mark/number token patterns and NFC normalization.

## Best Possible Solution

Port Unicode mark and NFC handling into `extensions/memory-core` FTS query building, manager search token planning, and MMR tokenization while preserving CJK unigrams/bigrams; add focused regressions and redacted live memory-search proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS/search-planner patterns omit combining marks and NFC normalization, and the active MMR tokenizer still degrades non-CJK Unicode such as Finnish diacritics and Russian text. I did not run an end-to-end memory search in this read-only review.

## Solution Assessment

No. The Unicode/NFC direction is right, but this PR edits removed legacy `src/memory` files and a direct port would regress current `memory-core` CJK unigram/bigram MMR behavior.
