## Summary

Keep open: the underlying Unicode memory-search bug still appears valid in the active memory-core path, but this branch targets obsolete legacy files and would not preserve the current CJK MMR token contract.

## What This Changes

The PR changes legacy `src/memory` FTS/MMR tokenizer helpers and tests to add NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC tokenization into active `memory-core` FTS, MMR, and search-planner paths while preserving CJK unigram/bigram behavior, then add focused regression tests and real behavior proof.

## Reproduction Assessment

Yes, at source level: current memory-core FTS tokenizers exclude combining marks/NFC normalization, and MMR tokenizes only ASCII plus CJK-family characters. I did not run a full `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but the patch must move from legacy `src/memory` files into active memory-core paths and preserve the existing CJK granularity contract.
