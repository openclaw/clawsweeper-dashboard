## Summary

Keep open: the Unicode/NFC bug still appears valid on current main, but this branch is not merge-ready because it edits stale `src/memory` files after the memory engine moved to `extensions/memory-core`, and the MMR tokenizer approach conflicts with current CJK unigram/bigram behavior.

## What This Changes

The PR updates legacy memory MMR and FTS tokenizers under `src/memory` plus tests to add Unicode mark matching and NFC normalization.

## Best Possible Solution

Port mark/NFC support into active memory-core FTS/search-planning and MMR tokenization while preserving CJK unigrams/bigrams, then add regressions and redacted runtime memory-search proof before merge.

## Reproduction Assessment

Yes, at source level. Active memory-core FTS regexes omit `\p{M}`/NFC and active MMR still only accepts ASCII plus CJK-specific tokens; I did not run a persisted memory-search flow under this read-only review.

## Solution Assessment

No. The Unicode/NFC direction is right, but the submitted patch targets stale files and would need a memory-core port that preserves current CJK MMR behavior.
