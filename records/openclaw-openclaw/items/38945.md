## Summary

Keep open: the Unicode/NFC bug is still source-reproducible on current main, but this PR edits legacy `src/memory` files that are no longer the active runtime path, so it would not fix `memory-core` as merged.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizers/tests to use Unicode letter/mark/number matching with NFC normalization.

## Best Possible Solution

Replace or update the branch with a `memory-core` port that covers FTS query building, search planning, and MMR tokenization while preserving CJK unigrams/bigrams, with focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main's active `memory-core` FTS/search-planner regexes omit combining marks and NFC normalization, and active MMR only tokenizes ASCII plus CJK unigrams/bigrams; I did not run a live `openclaw memory` flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this PR targets legacy files absent from current main; the maintainable fix belongs in `extensions/memory-core` and must preserve existing CJK token granularity.
