## Summary

Keep open. The Unicode/NFC bug is credible, but this PR is stale against current main: it edits removed legacy `src/memory` files while active memory search now runs through `extensions/memory-core`, and it still needs inspectable real behavior proof before merge.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizer regexes/tests to add Unicode letter/mark/number matching with NFC normalization.

## Best Possible Solution

Port Unicode mark and NFC handling into `memory-core` FTS query building, search planning, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current `memory-core` FTS/search-planner tokenizers omit combining marks and NFC normalization, and active MMR tokenization still degrades non-CJK Unicode such as Finnish diacritics.

## Solution Assessment

No. The Unicode/NFC direction is right, but the PR edits legacy files that are no longer the active runtime and its direct regex approach would collapse current CJK MMR token granularity if ported unchanged.
