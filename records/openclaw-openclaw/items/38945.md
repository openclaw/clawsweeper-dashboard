## Summary

Keep open: the reported Unicode tokenizer bug remains source-reproducible on current main, but this PR is not correct as-is because it edits legacy `src/memory` files that no longer power `openclaw memory`. It needs to be retargeted to active `memory-core` paths and backed by real behavior proof before merge.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizer helpers/tests to use Unicode letter/mark/number matching with NFC normalization.

## Best Possible Solution

Port Unicode/NFC token handling into active `memory-core` FTS query building, search planning, and MMR/shared tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current `memory-core` FTS/search-planner regexes omit combining marks and NFC normalization, and active MMR still drops non-CJK Unicode text while preserving CJK via a separate unigram/bigram contract; I did not run a live CLI memory flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but the patch edits legacy files absent from current main and must be retargeted to `extensions/memory-core` without collapsing current CJK token granularity.
