## Summary

Keep open: the Unicode memory tokenizer bug is valid and source-reproducible, but this PR is not mergeable as-is because it patches legacy `src/memory` files instead of active `memory-core` paths, would regress established CJK MMR token granularity if ported directly, and lacks required real behavior proof.

## What This Changes

The PR updates legacy memory MMR and FTS tokenizer helpers/tests to use Unicode-aware token regexes and NFC normalization.

## Best Possible Solution

Port Unicode/NFC tokenization into active `extensions/memory-core` FTS, MMR, and shared search-planner paths while preserving CJK unigram/bigram behavior, then attach redacted real behavior proof before merge.

## Reproduction Assessment

Yes, as source-level current-main proof: active `memory-core` token regexes still omit combining marks/NFC and a read-only Node probe shows Thai/Hindi FTS fragmentation plus accented Latin/NFD MMR token loss. I did not run the full `openclaw memory` command.

## Solution Assessment

No. The tokenizer direction is right, but the patch targets legacy `src/memory` files; the maintainable fix belongs in `extensions/memory-core` and must preserve existing CJK MMR token granularity.
