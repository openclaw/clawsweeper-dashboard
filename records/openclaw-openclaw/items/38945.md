## Summary

Keep open: the reported Unicode tokenizer bug is real and useful, but this branch edits legacy `src/memory` files that are no longer the active memory implementation, and it still lacks contributor-supplied real behavior proof.

## What This Changes

The PR updates legacy memory MMR and FTS tokenizer helpers/tests to use Unicode-aware token regexes and NFC normalization.

## Best Possible Solution

Port Unicode/NFC tokenization into active `extensions/memory-core` FTS, MMR, and shared search-planner paths while preserving CJK unigram/bigram behavior, then attach redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current active `memory-core` regexes still omit combining marks/NFC, and a read-only Node probe shows Thai/Hindi FTS fragmentation plus accented Latin/NFD MMR token loss. I did not run a full `openclaw memory` live flow.

## Solution Assessment

No. The tokenizer direction is right, but this PR targets legacy deleted paths; the maintainable fix belongs in `extensions/memory-core` and must preserve current CJK MMR granularity.
