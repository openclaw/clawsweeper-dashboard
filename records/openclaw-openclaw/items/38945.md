## Summary

Keep open: the Unicode tokenizer bug is still source-reproducible on current main, but this PR edits legacy files that are no longer used by `openclaw memory`, lacks inspectable real-behavior proof, and a literal MMR port would regress current CJK token granularity.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizers/tests to add NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC handling into active `extensions/memory-core` FTS, search-planner, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real memory-search proof.

## Reproduction Assessment

Yes, at source level. Current memory-core regex paths still split combining-mark scripts and active MMR still drops or fragments non-Latin non-CJK text; the read-only probe demonstrates those tokenizer failures without running a full live memory flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this branch edits legacy files absent from current main; the maintainable fix is a memory-core port that also updates `manager-search` and preserves current CJK tokenizer behavior.
