## Summary

Keep open: the Unicode/NFC bug is still credible on current main, but this branch edits legacy `src/memory` files that no longer carry the active memory runtime, so the PR is not mergeable as the fix path.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizers/tests to use NFC normalization and Unicode letter/mark/number token matching.

## Best Possible Solution

Replace or update this branch with a `memory-core` port that adds Unicode/NFC handling to FTS, search planning, and MMR while preserving CJK unigrams/bigrams, plus focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main's active `memory-core` regex paths still omit NFC/combining marks for FTS/search planning, and active MMR still only extracts ASCII outside the CJK path; I did not run a live `openclaw memory` flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this PR edits legacy files absent from current main; the maintainable fix belongs in active `extensions/memory-core` and must preserve CJK tokenizer behavior.
