## Summary

Keep open: the bug is still valid, but this PR now targets legacy `src/memory` files that no longer exist on current main, while active memory search is served by `memory-core`. It also still needs real behavior proof before merge.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizer helpers and tests to add Unicode-aware matching, combining marks, and NFC normalization.

## Best Possible Solution

Port Unicode/NFC tokenization into active `extensions/memory-core` FTS, MMR, and shared search-planner paths while preserving CJK unigram/bigram behavior.

## Reproduction Assessment

Yes, at source level: active `memory-core` still fragments combining-mark scripts in FTS tokenization and only handles ASCII plus CJK-family tokens in MMR. I did not run the full `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but the PR changes legacy files; the maintainable fix belongs in active `memory-core` code and must preserve existing CJK granularity.
