## Summary

Keep open: the reported Unicode memory-tokenization bug is credible, but this PR still targets removed legacy `src/memory` files, leaves active `memory-core` FTS/search-planner tokenization unchanged, and lacks contributor-supplied real behavior proof.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizers/tests to add NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC handling into active `extensions/memory-core` FTS, search planning, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real memory-search proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS/search-planner tokenization still excludes combining marks and NFC normalization, and active MMR only handles ASCII plus CJK-family characters. I did not run a live `openclaw memory` flow in this read-only review.

## Solution Assessment

No. The Unicode/NFC direction is right, but this branch targets inactive legacy files and a literal MMR regex port would not preserve current CJK token granularity.
