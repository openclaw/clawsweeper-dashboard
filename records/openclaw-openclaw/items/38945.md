## Summary

Keep open: the Unicode tokenizer bug still exists in the active `memory-core` implementation, but this PR now patches legacy `src/memory` files that current `main` no longer uses and still lacks contributor-visible real behavior proof.

## What This Changes

The PR updates legacy memory FTS and MMR tokenizer helpers plus tests to add NFC normalization and Unicode letter/mark/number token matching.

## Best Possible Solution

Port Unicode/NFC tokenization into the active `memory-core` FTS, MMR, and shared search-planner paths while preserving CJK unigram/bigram behavior, then add focused regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS tokenizers still omit combining marks/NFC normalization, and MMR only tokenizes ASCII plus CJK-family characters. I did not run the full `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but this PR changes legacy `src/memory` files and would also replace the current CJK MMR granularity instead of porting the fix into the active `memory-core` paths.
