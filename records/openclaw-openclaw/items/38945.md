## Summary

Keep open: the bug still appears valid, but this PR is not merge-ready because it edits removed legacy memory files instead of the active memory-core implementation, would lose current CJK token granularity if ported verbatim, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes legacy memory tokenizer helpers and tests to add Unicode-aware matching, combining marks, and NFC normalization for MMR and FTS tokenization.

## Best Possible Solution

Port Unicode/NFC tokenization into active `extensions/memory-core` FTS, MMR, and shared search-planner paths while preserving existing CJK unigram/bigram behavior, then attach redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main still routes memory to `memory-core`, and the active regexes still omit combining marks/NFC while MMR only handles ASCII plus CJK-family tokens. I did not run the full `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but this PR changes legacy paths; the maintainable fix belongs in active `memory-core` code and must preserve CJK unigram/bigram matching.
