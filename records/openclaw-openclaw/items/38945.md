## Summary

Keep open: the bug is valid and the tokenizer direction is useful, but this PR edits legacy `src/memory` files that are absent from current `main`; the active `memory-core` implementation still needs the Unicode/NFC port and the port must preserve existing CJK token granularity.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizers/tests to add NFC normalization and Unicode letter/mark/number token matching.

## Best Possible Solution

Port Unicode/NFC tokenization into active `extensions/memory-core` FTS, search-planner fallback, and MMR paths while preserving CJK unigrams/bigrams, then add focused regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS/search-planner tokenizers exclude combining marks and NFC normalization, and MMR only handles ASCII plus CJK-family tokens. I did not run a full live `openclaw memory` CLI flow in this read-only review.

## Solution Assessment

No. The tokenizer direction is right, but the current PR edits inactive legacy files and would also lose the active CJK MMR token-granularity contract if ported literally.
