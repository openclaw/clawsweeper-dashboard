## Summary

Keep open: the reported Unicode memory-search bug is still valid, but this branch cannot merge as-is because current main serves memory from `memory-core` while the PR edits legacy `src/memory` files and would also flatten current CJK MMR token granularity.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizers/tests to use NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC tokenization into `extensions/memory-core` FTS, search-planner fallback, and MMR while preserving CJK unigram/bigram behavior, then add focused regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS/search-planner tokenizers exclude combining marks and NFC normalization, and MMR still drops non-ASCII non-CJK letters from similarity tokens. I did not run a full live `openclaw memory` CLI flow in this read-only review.

## Solution Assessment

No. The Unicode tokenizer direction is right, but this PR edits inactive legacy files and the MMR change would not preserve current CJK token granularity if ported literally.
