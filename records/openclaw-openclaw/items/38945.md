## Summary

Keep open for https://github.com/openclaw/openclaw/pull/38945: the Unicode tokenizer bug is still source-reproducible on current main, but this PR changes legacy `src/memory` paths that no longer serve active memory search, lacks contributor real behavior proof, and is currently conflicting.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizer helpers/tests to use Unicode-aware token regexes and NFC normalization.

## Best Possible Solution

Port Unicode/NFC tokenization into active `memory-core` FTS, shared search planning, and MMR while preserving CJK unigram/bigram behavior, then attach redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` token regexes still omit combining marks/NFC normalization, and a read-only Node probe reproduces Thai/Hindi FTS fragmentation plus accented/NFD MMR token loss. I did not run the full `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but the PR changes legacy `src/memory` files; the maintainable fix belongs in active `extensions/memory-core` paths and must keep existing CJK MMR granularity.
