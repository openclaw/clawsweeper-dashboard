## Summary

Keep this PR open, but it is not mergeable as-is: the Unicode tokenizer bug is still visible in current `memory-core`, while the branch edits legacy `src/memory` files that no longer drive `openclaw memory` and would also need to preserve the current CJK MMR token contract.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizer helpers and tests to add NFC normalization and Unicode letter/mark/number token matching.

## Best Possible Solution

Port Unicode/NFC tokenization into active `memory-core` FTS, search-planner fallback, and MMR paths while preserving CJK unigram/bigram behavior, then add focused regression tests and real behavior proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS/search-planner tokenizers exclude combining marks and NFC normalization, and MMR still tokenizes only ASCII plus CJK-family characters. I did not run a full live `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but the patch needs to move from legacy `src/memory` files into active `extensions/memory-core` paths and preserve the current CJK token granularity contract.
