## Summary

Keep open: the underlying Unicode memory-tokenization bug is source-reproducible, but this PR is not correct as-is because it edits removed legacy `src/memory` files instead of the active `memory-core` plugin and a literal MMR port would drop current CJK token granularity.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizers/tests to add NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC handling into active `extensions/memory-core` FTS, search-planner, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real memory-search proof.

## Reproduction Assessment

Yes, at source level. Current active memory-core FTS/search tokenizers still exclude combining marks and NFC normalization, and active MMR only handles ASCII plus CJK-family characters; I confirmed the tokenizer failure with a read-only regex probe rather than a full live `openclaw memory` flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this branch edits inactive legacy files, and the MMR change must be adapted to preserve existing CJK unigrams and adjacent bigrams.
