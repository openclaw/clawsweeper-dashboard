## Summary

Keep open. The Unicode tokenizer bug is valid, but this branch edits legacy `src/memory` files that current main no longer uses for `openclaw memory`; the fix needs to be ported to active `memory-core` and backed by real behavior proof before merge.

## What This Changes

The branch updates legacy `src/memory` MMR and FTS tokenizers/tests to use NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC handling into `extensions/memory-core` FTS, search planning, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real memory-search proof.

## Reproduction Assessment

Yes, at source level: active `memory-core` still excludes combining marks and NFC normalization in FTS/search planning, and MMR still only tokenizes ASCII plus CJK-family characters. I did not run a live `openclaw memory` flow in this read-only review.

## Solution Assessment

No. The Unicode/NFC direction is right, but this branch targets inactive legacy files and its MMR regex would not preserve current CJK token granularity if ported literally.
