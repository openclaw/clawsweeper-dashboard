## Summary

Keep open: this is a valid memory Unicode bug, but the PR is stale against current main because it edits removed legacy `src/memory` files instead of the active `memory-core` plugin, and a direct port would also need to preserve current CJK MMR behavior.

## What This Changes

The PR updates legacy memory MMR and FTS tokenizers plus tests to NFC-normalize input and tokenize Unicode letters, combining marks, numbers, and underscores.

## Best Possible Solution

Land a narrow `memory-core` fix that normalizes before tokenization, supports Unicode letters/marks/numbers consistently in FTS and fallback token paths, preserves CJK MMR unigram/bigram handling, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction. Current `memory-core` regexes still fragment combining-mark scripts in FTS paths and drop non-CJK Unicode word tokens in MMR, and the read-only Node probe demonstrates those failures on current main logic.

## Solution Assessment

No, not as currently proposed. The tokenizer direction is correct, but the patch must move to `extensions/memory-core` and preserve the active CJK MMR token model instead of editing removed legacy files.
