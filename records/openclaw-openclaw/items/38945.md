## Summary

Keep open: the Unicode/NFC bug is still valid, but this branch edits legacy `src/memory` files that are absent from current main. The active `memory-core` tokenizer/search paths still need a targeted port, and the PR is missing inspectable real behavior proof.

## What This Changes

The PR updates legacy `src/memory` MMR and FTS tokenizers/tests to use Unicode letter/mark/number matching with NFC normalization.

## Best Possible Solution

Port the Unicode/NFC token handling into `extensions/memory-core` FTS query building, search planning, and MMR/shared tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main's active `memory-core` FTS/search-planner regexes omit combining marks and NFC normalization, and active MMR only tokenizes ASCII plus CJK unigrams/bigrams; I did not run a live `openclaw memory` flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this PR targets legacy files absent from current main; the maintainable fix belongs in `extensions/memory-core` and must preserve existing CJK token granularity.
