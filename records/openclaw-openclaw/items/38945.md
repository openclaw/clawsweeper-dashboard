## Summary

Keep open: the Unicode memory-search bug is valid, but this PR is not mergeable as-is because it edits legacy `src/memory` files that no longer drive current `openclaw memory`, misses active `memory-core` paths, and lacks inspectable after-fix real behavior proof.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizers/tests to add NFC normalization and Unicode letter/mark/number matching.

## Best Possible Solution

Port Unicode/NFC handling into active `extensions/memory-core` FTS, search-planner, and MMR tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real memory-search proof.

## Reproduction Assessment

Yes, at source level. Current memory-core regex paths still split combining-mark scripts and active MMR drops or fragments non-CJK Unicode terms; I did not run a full live `openclaw memory` flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this branch edits legacy files absent from current main; the maintainable fix is a memory-core port that also updates `manager-search` and preserves CJK tokenizer behavior.
