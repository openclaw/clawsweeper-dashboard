## Summary

Keep open: the Unicode tokenizer bug is source-reproducible on current main, but this PR targets legacy `src/memory` files that are no longer the active memory runtime, and it still lacks after-fix real behavior proof.

## What This Changes

The PR changes legacy `src/memory` MMR and FTS tokenizers/tests to use Unicode letter/mark/number matching with NFC normalization.

## Best Possible Solution

Port Unicode/NFC token handling into active `memory-core` FTS query building, search planning, and MMR/shared tokenization while preserving CJK unigrams/bigrams, then add focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main's active `memory-core` FTS/search-planner regexes omit combining marks and NFC normalization, and active MMR still drops or fragments non-CJK Unicode text; I did not run a live `openclaw memory` flow.

## Solution Assessment

No. The Unicode/NFC direction is right, but this PR edits legacy `src/memory` files absent from current main and its MMR regex would collapse CJK sequences that active `memory-core` intentionally tokenizes as unigrams and adjacent bigrams.
