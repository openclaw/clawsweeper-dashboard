## Summary

Keep open: the Unicode tokenizer bug is valid, but this PR cannot merge as-is because current main serves memory through `memory-core`, the branch changes legacy `src/memory` files, and external real behavior proof is still missing.

## What This Changes

The branch updates legacy `src/memory` FTS and MMR tokenizer helpers and tests to add NFC normalization plus Unicode letter/mark/number token matching.

## Best Possible Solution

Port Unicode/NFC tokenization into `extensions/memory-core` FTS, MMR, and shared search-planner paths, preserve existing CJK unigram/bigram behavior, add focused regression tests, and include redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current `memory-core` FTS tokenizers still omit combining marks/NFC normalization, and MMR only handles ASCII plus CJK-family tokens. I did not run the full `openclaw memory` CLI flow.

## Solution Assessment

No. The tokenizer direction is right, but the PR changes legacy files and would replace current CJK MMR granularity instead of porting the fix into the active `memory-core` paths.
