## What This Changes

The PR changes `refreshCostUsageCache` to checkpoint usage-cost cache writes every 256 files or 5 seconds, skip no-op rewrites, and adds a regression test plus a proof script.

## Best Possible Solution

Land the checkpointing change after maintainer review, keeping the existing cache format, lock, atomic writer, and refresh semantics intact.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path because the stale-file loop writes the full cache after each scanned file, and the PR proof script provides a concrete runtime path for exercising the same production refresh/load code.

## Solution Assessment

Yes. Batching writes inside `refreshCostUsageCache` is the narrow maintainable fix because it leaves the derived-cache format, freshness checks, lock, and atomic writer in place while reducing repeated serialization.
