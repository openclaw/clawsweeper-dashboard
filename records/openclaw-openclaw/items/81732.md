## Summary

Keep open. The PR targets the right macOS LaunchAgent stop/restart postcondition, but the new guard can still skip verification for normal managed installs because the port resolver ignores the LaunchAgent command environment where installs store `OPENCLAW_GATEWAY_PORT`.

## What This Changes

The PR adds macOS LaunchAgent stop/restart port-release checks, focused launchd regression tests, and a changelog entry for the gateway stop bug.

## Best Possible Solution

Keep the postcondition approach, but resolve the effective LaunchAgent port from program arguments, the stored LaunchAgent command environment, and then caller env before adding env-file-only regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: normal installs store `OPENCLAW_GATEWAY_PORT` in the LaunchAgent environment, but the new guard skips cleanup and inspection when the invoking shell lacks that env var and the command has no `--port` argument.

## Solution Assessment

No. The added postcondition is the right fix shape, but it must use the stored LaunchAgent command environment before deciding there is no port to verify.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use the service environment when resolving the LaunchAgent port:** `src/daemon/launchd.ts:606`
  - body: The new stop/restart guard returns early whenever `resolveLaunchAgentGatewayPort` yields `null`, but that resolver only checks `ProgramArguments` and the caller's env. Normal managed installs write `OPENCLAW_GATEWAY_PORT` into the LaunchAgent environment, so a shell without that env var and no `--port` arg will skip `cleanStaleGatewayProcessesSync`/`inspectPortUsage` and can still report success while the configured port remains held. Please read `command.environment.OPENCLAW_GATEWAY_PORT` before falling back or skipping, and add a regression where the port only comes from the plist/env wrapper.
  - confidence: 0.9
