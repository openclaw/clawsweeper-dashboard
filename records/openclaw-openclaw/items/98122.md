## Label Justifications

- `P1`: The report describes a current shipped Telegram workflow where group messages are not received at all under the default long-polling setup.
- `impact:message-loss`: The affected symptom is missed Telegram group inbound messages while DMs still arrive.

## Summary

Current main still contains a Telegram isolated polling ingress that starts a separate grammY runner from the main Telegram transport, so the reported single-token dual `getUpdates` conflict remains plausible and is not implemented or superseded by the related polling-stall/spool issues.

## What This Changes

The issue asks OpenClaw to avoid conflicting Telegram long-polling consumers by having group/channel updates arrive through a single polling connection or by otherwise preventing the isolated ingress from competing with the main grammY poller.

## Best Possible Solution

Keep this issue open and route it to a Telegram polling design fix that preserves isolated-ingress reliability while enforcing one active `getUpdates` consumer per bot token or making the main poller the single source for all Telegram update types.

## Reproduction Assessment

No high-confidence live reproduction was performed. Source inspection shows current main can start both the main grammY long poller and the isolated polling runner with the same token, matching the reported Telegram `getUpdates` conflict class.

## Solution Assessment

Unclear. A single-poller design matches the Telegram Bot API constraint, but the best fix must preserve the isolated-ingress reason for existence instead of simply deleting the worker path.
