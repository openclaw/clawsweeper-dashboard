## Summary

Keep open: current main still runs Active Memory recall as a foreground `before_prompt_build` hook, so a slow enabled recall can delay the first main reply until the configured hook/recall budget. Recent timeout and fail-open work bounds the delay, but it does not change the blocking semantics reported here.

## What This Changes

Request to prevent enabled Active Memory from delaying chat replies by its full recall timeout, potentially through non-blocking recall, shorter or auto-tuned timeout behavior, or fail-open first-output handling.

## Best Possible Solution

Add an explicit Active Memory foreground latency policy, such as a short first-output budget with async or next-turn recall fallback, while preserving clear docs for when memory may affect the current reply.

## Reproduction Assessment

Yes, source-reproducible. Enable `active-memory` for an eligible direct chat, use a slow or non-returning recall model with `timeoutMs` around 5000, and send a simple message; current main awaits the recall hook before the main prompt can proceed.

## Solution Assessment

Unclear. Lowering defaults would not help users who configure a 5000 ms foreground recall budget, while non-blocking or next-turn recall changes when memory can influence the current reply and needs a maintainer product decision.
