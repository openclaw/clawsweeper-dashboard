## Summary

Keep open. Current main still routes eligible Active Memory turns through a foreground `before_prompt_build` hook that awaits recall before prompt construction can continue; recent timeout, cache, circuit-breaker, and hook-budget mitigations reduce worst cases but do not resolve the reported first-reply delay. Related #75020 has already been closed as a duplicate pointing here, so this should remain an active canonical tracker rather than be swept closed.

## What This Changes

The issue asks for Active Memory to stop delaying visible chat replies by its full foreground recall timeout, potentially through non-blocking recall, tuned defaults, or fail-open behavior.

## Best Possible Solution

Keep this as the canonical Active Memory foreground-latency tracker. The desired end state should preserve the current hard-deadline, partial-summary, cache, circuit-breaker, and diagnostics safeguards while making normal chat responsiveness explicit through a non-blocking/background recall mode, a stricter fail-open foreground budget, or measured defaults/docs that avoid surprising first-token delays.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable Active Memory for an eligible direct or webchat session and send a user turn; current source shows prompt construction awaits the Active Memory `before_prompt_build` recall path until success, timeout, or circuit-breaker skip.

## Solution Assessment

Unclear. The issue lists several possible fixes, and current main’s timeout mitigations are useful but incomplete; maintainers should choose the product shape before changing foreground recall behavior.
