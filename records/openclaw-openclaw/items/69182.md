## Summary

Keep open: current main still runs Active Memory as a foreground `before_prompt_build` hook, so an enabled slow recall pass can delay the first visible reply until the configured timeout even though recent work bounds and fails open the hook.

## What This Changes

Request to prevent the `active-memory` plugin from blocking chat replies for its full `timeoutMs` budget, potentially by making recall non-blocking, reducing/tuning the timeout, or failing open earlier.

## Best Possible Solution

Add an explicit Active Memory foreground latency policy, such as a short first-output budget with async or next-turn recall fallback, and document how it differs from the full recall/model timeout.

## Reproduction Assessment

Yes. Enable `active-memory` for an eligible direct chat, use a slow or non-returning recall model with `timeoutMs` around 5000, and send a simple message; current main awaits recall until completion or timeout before the main reply can start.

## Solution Assessment

Unclear. Lowering the default timeout would not solve users who already configure 5000ms, while making recall non-blocking changes Active Memory semantics and needs maintainer product direction.
