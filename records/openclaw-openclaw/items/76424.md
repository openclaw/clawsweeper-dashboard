## Summary

Keep open: the report matches current main on source inspection, because overflow compaction can adopt a rotated transcript and then retry with the same prompt, transcript prompt, and external message id. Existing inbound and queue dedupe guards do not cover this internal embedded-run retry path, and the related replay/fence PRs are still open or adjacent rather than a merged fix.

## What This Changes

Channel-backed context-overflow retries should be idempotent for the current Telegram message and avoid delivering stale output after newer messages arrive.

## Best Possible Solution

Add a channel-aware overflow-retry invariant: after successful compaction, continue the in-flight turn without re-appending the already-recorded external message id, and fence any final delivery that has been superseded by a newer channel message.

## Reproduction Assessment

Yes, source-level: the issue provides concrete Telegram logs and current main shows the compaction retry loop reuses the original prompt/currentMessageId before another `activeSession.prompt(...)`. I did not run a live Telegram reproduction in this read-only pass.

## Solution Assessment

No for current main: relying on inbound or queued-message dedupe is not enough because the duplicate is created inside the embedded retry loop. The narrow maintainable fix is to make compaction retry a continuation/idempotent path for channel-backed prompts, with a regression test for external message id reuse.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76424. On current main, channel-backed context-overflow compaction retry in `src/agents/pi-embedded-runner/run.ts` can adopt a rotated transcript and then rerun `runEmbeddedAttemptWithBackend` with the original `prompt`, `transcriptPrompt`, and `currentMessageId`, so the same external Telegram message can be appended again. Implement a narrow idempotent retry path for channel-backed user prompts: after successful overflow/timeout/precheck compaction, if the external current message was already submitted in the active transcript, retry as a continuation/runtime-only prompt instead of re-submitting the visible transcript prompt. Add regression coverage that forces overflow, successful compaction, retry, and asserts the same external message id is not recorded twice; include stale-final suppression only if needed to satisfy the reported newer-message case, using existing run/queue primitives rather than a broad new policy. Do not change Telegram webhook inbound dedupe, queue mode semantics, provider defaults, or unrelated replay cleanup. Add a single-line `CHANGELOG.md` Fixes entry if production behavior changes.
