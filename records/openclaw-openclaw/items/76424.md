## Summary

Keep open: current main still has a normal overflow-compaction retry path that adopts the compacted transcript and then reuses the original prompt, transcript prompt, and current message id; the existing inbound dedupe only protects transport ingress, and the related replay/fencing PRs are still open adjacent work rather than a merged fix for this specific path.

## What This Changes

Ensure Telegram/channel-backed context-overflow compaction retries do not append the same inbound message/currentMessageId twice or deliver stale output after newer messages arrive.

## Best Possible Solution

Add a generic channel-backed compaction-retry invariant: after overflow compaction, continue from the compacted transcript or otherwise suppress a duplicate currentMessageId/current prompt append, and fence final delivery when the run has been superseded by newer external channel input.

## Reproduction Assessment

No high-confidence end-to-end reproduction was executed. Source inspection gives a focused reproduction path: trigger assistantError overflow in a channel-backed run, let auto-compaction succeed, and observe the retry reusing the same prompt/currentMessageId against the compacted session.

## Solution Assessment

No, current main's plain prompt retry is not the best solution for externally delivered channel turns. The narrow maintainable fix is to make the retry idempotent at the compaction/retry boundary and add a stale-output fence where newer channel messages supersede the old run.
