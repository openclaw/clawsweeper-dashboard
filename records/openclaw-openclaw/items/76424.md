## Summary

This issue should stay open because the reported failure is still a valid current-main source risk and an open PR explicitly references it with closing syntax. Current `main` has inbound-message dedupe and Telegram supersession fences, but the overflow retry loop still re-enters `runEmbeddedAttemptWithBackend` with the original prompt, transcript prompt, and `currentMessageId` after ordinary assistant-error compaction unless the separate mid-turn preflight path set a continuation prompt.

## What This Changes

The issue requests that Telegram channel-backed context-overflow retries avoid re-appending the same external message id and suppress stale replies after newer messages arrive.

## Best Possible Solution

Land #76820 or an equivalent runner-level idempotency fix with regression coverage that proves assistant-error compaction retries continue from the compacted transcript without re-submitting the same external message id.

## Reproduction Assessment

Yes, by source inspection: trigger an assistant-error context overflow in a channel-backed run with `currentMessageId`, let compaction succeed, and the next attempt still receives the original prompt and `currentMessageId` unless only the mid-turn preflight path applied a continuation override. The issue also has two independent 2026.5.2 Telegram reports with repeated external message ids.

## Solution Assessment

Yes, the best narrow fix is at the embedded runner retry boundary: after successful compaction for an already-submitted channel message, retry with a continuation prompt or otherwise suppress re-submission of the same external message id. A broader Telegram-only delivery fence is useful defense in depth, but it does not prevent the duplicate transcript user turn.
