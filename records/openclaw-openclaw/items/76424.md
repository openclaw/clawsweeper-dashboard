## Summary

Keep open: current main still reuses the original prompt and currentMessageId on ordinary assistantError auto-compaction retries, and open PR #76820 explicitly fixes this with closing syntax; close only after that PR or an equivalent fix lands.

## What This Changes

Ensure Telegram/channel compaction retries do not re-persist the same external currentMessageId or deliver stale replies after context overflow.

## Best Possible Solution

Land a narrow runner fix that converts same-currentMessageId compaction retries into continuation prompts and suppresses duplicate user persistence, then use any remaining stale-output cases to drive the broader generation-fence work.

## Reproduction Assessment

Yes, by source-level reproduction: force runEmbeddedPiAgent to return a context-overflow error on the first attempt with currentMessageId set, make compaction succeed, and observe current main retries with the original prompt/currentMessageId unless the recovery was mid-turn precheck. I did not rerun a live Telegram session in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrowest maintainable fix for the duplicate transcript entry because it reuses the existing continuation-prompt and one-user-message suppression seam. A run-generation fence is a likely follow-up for late stale deliveries, not a blocker to fixing this replay bug.
