## Summary

Keep open: current main still has a plausible source-level path for the reported channel-backed compaction retry replay, and the related open items do not conclusively cover this exact Telegram/currentMessageId idempotency failure.

## What This Changes

Prevent Telegram/channel-backed context-overflow compaction retries from re-appending the same external inbound message and from delivering stale output after newer messages.

## Best Possible Solution

Add a core embedded-runner regression and guard so channel-backed overflow compaction retries continue from the compacted transcript without persisting the same external currentMessageId as a new user turn, then fence any stale outbound delivery from superseded runs through the existing reply operation mechanisms.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was run, but the source path is clear enough for a focused regression: force overflow compaction on a channel-backed run with currentMessageId X and assert the retry does not submit the original prompt/currentMessageId as a fresh user turn.

## Solution Assessment

Is this the best way to solve the issue? Yes: a targeted embedded-runner retry idempotency guard is narrower than Telegram transport dedupe because the duplicate is generated inside the active retry loop after inbound dedupe has already accepted the message.

In current main, a channel-backed embedded run that hits context overflow can compact, adopt a successor transcript, then retry with the original prompt/transcriptPrompt/currentMessageId and append the same external inbound message again.
