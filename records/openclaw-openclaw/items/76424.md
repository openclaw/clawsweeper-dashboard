Ensure Telegram/channel-backed context-overflow compaction retries do not re-append the same external inbound message or deliver stale output after newer messages.

## Best Possible Solution

Land a regression-tested retry fix that treats unchanged channel `currentMessageId` retries as continuation/suppressed-user-persistence retries, then split any remaining stale-output fencing into a focused follow-up if needed.
