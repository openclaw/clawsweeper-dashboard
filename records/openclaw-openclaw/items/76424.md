## Summary

Keep open: current main still has the normal overflow-compaction retry shape that can reuse the original channel prompt and `currentMessageId`, and the open linked PR is an unmerged implementation candidate rather than a reason to close the issue early.

## What This Changes

The issue requests that Telegram channel-backed context-overflow retries avoid re-appending the same external inbound message and avoid delivering stale output after newer messages arrive.

## Best Possible Solution

Review and land PR #76820 or an equivalent core runner fix that makes channel-backed compaction retries idempotent for the same external message id, with follow-up fencing if late stale delivery remains possible.

## Reproduction Assessment

Yes from source inspection and independent production traces: trigger overflow compaction in a channel-backed run while `currentMessageId` is unchanged, and current main retries with the same prompt metadata. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix belongs in the embedded runner retry boundary, not Telegram-specific delivery code, because the reused `transcriptPrompt` and `currentMessageId` are passed from core retry state. If duplicate retry is fixed but stale late output remains, add a focused generation fence as a separate guard.
