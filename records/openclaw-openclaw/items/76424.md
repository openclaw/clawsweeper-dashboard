## Summary

Keep open: current main still lacks a non-mid-turn unchanged-currentMessageId guard for overflow compaction retry, and an open closing PR explicitly targets this issue, so closing before merge would be premature.

## What This Changes

Telegram channel-backed context-overflow retry should not re-append the same inbound currentMessageId or deliver stale output after newer channel messages arrive.

## Best Possible Solution

Review and land #76820 or an equivalent core runner fix that switches unchanged currentMessageId compaction retries to a continuation prompt and locks the channel-idempotency regression coverage before closing this issue.

## Reproduction Assessment

Yes. The issue has two independent Telegram incidents on 2026.5.2, and current-main source inspection shows the retry loop can reuse the original prompt and currentMessageId after compaction outside the mid-turn precheck path.

## Solution Assessment

Yes. The narrow maintainable fix is to make compaction retry idempotent for an unchanged channel currentMessageId; broader run-generation fencing should be added only if review shows the retry guard does not cover stale delivery.
