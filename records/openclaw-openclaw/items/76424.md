## Summary

Keep open: current main still has the source path described in the report, where overflow recovery adopts the compacted transcript but retries through the normal prompt submission path with the original prompt, transcript prompt, and current message id. The existing inbound/queue dedupe and compaction duplicate-pruning helpers cover adjacent duplicate-delivery cases, but they do not make this in-run compaction retry idempotent for the same external Telegram message.

## What This Changes

The issue requests idempotent Telegram/channel context-overflow retry so the same external inbound message is not re-appended and stale old-turn output is not delivered after newer messages arrive.

## Best Possible Solution

Add a core channel-aware retry guard in the embedded overflow recovery path: after compaction, retry from the current transcript or suppress/fail external delivery when the same external currentMessageId has already been recorded, with regression coverage for newer-message stale output suppression.

## Reproduction Assessment

Yes from source inspection: an assistant-error context-overflow retry compacts, keeps params.currentMessageId and params.transcriptPrompt, then calls activeSession.prompt again on the compacted transcript. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the reported direction is the right narrow fix boundary: make the compaction retry idempotent for channel-backed currentMessageId and add a stale-output fence for superseded channel turns. A broad Telegram-only workaround would miss other channel-backed sessions that use the same embedded runner path.

## Repair Work Prompt

Fix the channel-backed context-overflow retry replay bug reported in https://github.com/openclaw/openclaw/issues/76424. In src/agents/pi-embedded-runner/run.ts and run/attempt.ts, make overflow/auto-compaction retry idempotent when params.currentMessageId identifies an external channel message already submitted to the active transcript. After compaction, do not re-submit the original visible transcriptPrompt/currentMessageId as a fresh user turn; use a continuation prompt or otherwise preserve in-flight state without duplicating the external message. Add a guard/fence so stale output from a superseded channel run is not delivered after newer external message ids have arrived. Keep the fix generic for channel-backed sessions, not Telegram-id-specific. Add regression coverage around assistant-error overflow compaction retry with currentMessageId, and cover the stale newer-message case if it can be isolated cheaply. Do not change queue mode semantics, provider fallback policy, dependency pins, or plugin-specific Telegram routing unless a small generic seam requires it. Add a CHANGELOG.md fix entry because this is user-facing.
