## Summary

Keep open: the current runner still has a source-visible path where overflow recovery compacts, adopts the new transcript, and retries the original channel prompt/current message metadata; the two issue reports provide matching direct and group-topic Telegram evidence on v2026.5.2.

## What This Changes

The issue asks for channel-backed context-overflow retries to be idempotent so the same external Telegram message is not appended again or answered stale after newer messages arrive.

## Best Possible Solution

Add a core embedded-runner guard so channel-backed overflow compaction retries continue from the compacted transcript or suppress retry prompt persistence for the already-recorded external message, with regression coverage for duplicate message ids.

## Reproduction Assessment

Yes, at source level: the issue and follow-up give two concrete v2026.5.2 Telegram reproductions, and current main still resubmits the original prompt/currentMessageId after overflow compaction. I did not perform a live Telegram run.

## Solution Assessment

Yes: the narrowest maintainable fix is in the embedded overflow retry path, not the Telegram plugin, because the retry loop owns the post-compaction resubmission. A broader generation fence can remain a follow-up if duplicate prompt idempotency is insufficient.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76424. In the embedded PI run loop, make channel-backed context-overflow auto-compaction retries idempotent for the current external message: after compaction succeeds, do not append/persist the same inbound prompt/currentMessageId into the compacted transcript as a fresh user turn. Prefer a generic core runner solution that either retries from the compacted transcript with a continuation prompt or reuses the existing suppress-next-user-message-persistence pattern for compaction retries after the original user message has already been persisted. Keep Telegram-specific ids out of core; use generic sessionKey/messageProvider/currentMessageId fields. Add regression coverage in the overflow compaction runner tests that simulates a channel-backed attempt with currentMessageId, forces assistantError context overflow, compacts, retries, and asserts the retry does not resubmit the original user prompt as a fresh transcript write. If stale final delivery after a newer message requires a broad run-generation policy change, stop at the duplicate-replay fix and mention the follow-up rather than adding a new product knob. Add a CHANGELOG.md fix entry for the user-facing channel behavior.
