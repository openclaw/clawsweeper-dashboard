## Summary

Keep open: the report is current, source-backed, and paired with an open unmerged fix PR that uses closing syntax. Current main still retries after overflow compaction by carrying the same prompt, transcript prompt, and current message id into the next attempt unless the path is specifically marked as mid-turn precheck recovery.

## What This Changes

The issue requests idempotent Telegram channel retry behavior after context-overflow auto-compaction so the same external message id is not replayed as a fresh user turn and stale replies are not delivered after newer messages.

## Best Possible Solution

Review and land PR #76820 or an equivalent narrow run-loop fix, then close this issue after the fix is merged and covered by regression tests for unchanged channel currentMessageId retries.

## Reproduction Assessment

Yes. The report has two independent 2026.5.2 Telegram reproductions, and current main still has the source path that reuses the same prompt/transcriptPrompt/currentMessageId after compaction; I did not run a live Telegram session in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to make compaction retries idempotent for an unchanged channel currentMessageId and cover it in the overflow-compaction run-loop tests. This issue should stay open until the linked PR or equivalent fix merges.
