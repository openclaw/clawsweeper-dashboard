## Summary

Keep open: current main still lacks idempotent channel retry handling after ordinary context-overflow compaction, and the linked #76820 fix PR is open and unmerged. The report also has two concrete Telegram reproductions covering both direct chats and group topics.

## What This Changes

Ensure Telegram/channel-backed context-overflow compaction retries do not append the same external message id as a new user turn or deliver stale output after newer messages.

## Best Possible Solution

Review, revise if needed, and land #76820 or an equivalent narrow fix that makes channel-backed compaction retries idempotent for unchanged external message ids, with regression coverage for both duplicate transcript appends and stale delivery behavior.

## Reproduction Assessment

Yes, there is a high-confidence source and report-level reproduction path: a Telegram direct or topic session hits context-overflow auto-compaction, then current main retries with the same currentMessageId/transcriptPrompt unless it is the mid-turn precheck branch. I did not live-run Telegram in this read-only review.

## Solution Assessment

Yes, making compaction retry continue from the compacted transcript when the channel currentMessageId is unchanged is the narrowest maintainable first fix. Add stale-output suppression only if the duplicate-message regression test still permits late delivery after newer external messages.
