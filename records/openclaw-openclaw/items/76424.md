## Summary

Keep open: the report describes a current, source-backed channel-session bug, and current main still retries overflow compaction with the same inbound prompt and external message id instead of making the retry idempotent.

## What This Changes

Prevent Telegram/channel-backed context-overflow compaction retries from re-appending the same inbound external message and delivering superseded stale replies.

## Best Possible Solution

Add an embedded-runner idempotency guard for channel-backed compaction retries, then cover it with a regression test that asserts one external message id produces one persisted user turn and superseded output is fenced.

## Reproduction Assessment

Yes at source level: a focused test can mock overflow, successful compaction, and retry, then assert current main re-enters the attempt with the same prompt/currentMessageId and no duplicate-persistence suppression. I did not run a live Telegram replay because the needed session artifacts are redacted.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the generic embedded retry/channel idempotency path rather than adding a Telegram-only workaround. Existing transcript-rotation dedupe is only a partial content-based cleanup and is not the right contract for external message ids.

Expected behavior: after successful overflow or timeout compaction, retrying an already-persisted channel inbound turn must not append the same user prompt/currentMessageId as a fresh user message, and late final output from a superseded channel turn should be suppressed or fenced if a newer external message has arrived.
