## Label Justifications

- `P1`: The report describes duplicate agent replies after a real iMessage gateway restart, which is a broken active channel workflow.
- `impact:message-loss`: The iMessage delivery path can reprocess already-handled inbound rows after restart and produce duplicate replies.
- `impact:session-state`: The root state mismatch is between a persisted catch-up cursor and process-local dedupe across gateway restarts.

## Summary

Keep open: current main still has the reported restart-boundary gap for opt-in iMessage catch-up. Catch-up persists its cursor after replay passes, but live `watch.subscribe` traffic still goes through the normal dispatch path and only relies on process-local inbound dedupe, so a gateway restart can forget already-handled live rows while the durable catch-up cursor lags.

## What This Changes

The issue asks OpenClaw to prevent iMessage catch-up from replaying inbound rows that were already handled live before a gateway restart.

## Best Possible Solution

Make iMessage live handling advance a durable monotonic high-water mark, or add an equivalent persistent provider-message dedupe keyed by stable iMessage identifiers, while preserving existing catch-up retry and failure-hold behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: persist an iMessage catch-up cursor behind a live-handled inbound row, restart the gateway so in-memory inbound dedupe resets, and catch-up will query from the stale cursor and re-feed that row.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable fix is to make the live path update durable restart-safe state, or to add a persistent stable-message dedupe, rather than relying on process-local overlap absorption.

## Repair Work Prompt

Fix the iMessage catch-up restart replay bug reported in https://github.com/openclaw/openclaw/issues/85363. Current main persists the catch-up cursor only from `performIMessageCatchup`, while live `watch.subscribe` traffic flows through `handleMessageNow` and relies on process-local inbound dedupe, so a restart can replay already-handled rows when the cursor lags. Implement a narrow restart-safe fix in the iMessage plugin: either advance the per-account catch-up cursor monotonically from the live path after rows are safely handled/skipped, or add an equivalent persistent provider-message dedupe keyed by stable iMessage identifiers and route scope. Preserve existing catch-up failure-hold, give-up, parser-watermark, and perRunLimit truncation semantics; do not advance past a held failed row or cap-truncated valid rows. Add focused regression coverage for the stale cursor plus reset inbound-dedupe scenario and update docs/changelog if user-visible behavior or configuration text changes.
