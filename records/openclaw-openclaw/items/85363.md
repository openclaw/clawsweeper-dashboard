## Label Justifications

- `P1`: The report describes duplicate agent replies after a real iMessage gateway restart, which is a broken channel workflow for active users.
- `impact:message-loss`: The affected channel delivery path can replay and duplicate inbound message handling after restart.
- `impact:session-state`: The root state involved is the persisted iMessage catch-up cursor versus process-local dedupe across gateway restarts.

## Summary

Keep open: current main still has the reported restart replay gap. The iMessage catch-up cursor is persisted by the catch-up pass, while live-handled messages only hit the normal inbound dispatch path and the overlap protection is process-local, so the narrower follow-up remains useful even with https://github.com/openclaw/openclaw/issues/62761 still open.

## What This Changes

The issue requests restart-safe iMessage catch-up so live-handled rows are not replayed after a gateway restart when the persisted cursor lags.

## Best Possible Solution

Make iMessage live handling update a durable monotonic high-water mark, or add an equivalent persistent provider-message dedupe keyed by stable iMessage identifiers, while preserving existing catch-up retry and failure-hold semantics.

## Reproduction Assessment

Yes from source inspection, though not from a live run: set the persisted iMessage catch-up cursor behind a live-handled inbound row, restart the gateway, and catch-up will query from the stale cursor after the in-memory inbound dedupe has been reset.

## Solution Assessment

Yes: the report’s proposed directions are appropriate, with the cleanest narrow path likely sharing a monotonic cursor merge helper between catch-up and the live path or using the existing persistent-dedupe style for stable iMessage IDs.
