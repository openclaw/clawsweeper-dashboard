## Label Justifications

- `P1`: The issue describes duplicate agent replies after a real iMessage gateway restart, which is a broken active channel workflow.
- `impact:message-loss`: The affected channel delivery path can replay already-handled inbound messages and produce duplicate replies after restart.
- `impact:session-state`: The root state is the persisted iMessage catch-up cursor falling behind process-local live handling state across restarts.

## Summary

Keep open: current main still has the restart replay gap because the persisted iMessage catch-up cursor is only advanced by the catch-up pass, while live-handled rows go through the normal dispatch path without a durable cursor update. The issue is also linked to an open, unmerged fix PR, so it should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to prevent iMessage startup catch-up from replaying inbound rows that were already handled live before a gateway restart.

## Best Possible Solution

Land a safe iMessage fix that advances a durable monotonic cursor from live-handled rows, or an equivalent persistent dedupe, while preserving catch-up retry holds and coalesced-message high-water behavior.

## Reproduction Assessment

Yes, from source inspection: leave the persisted iMessage catch-up cursor behind a live-handled inbound row, restart the gateway so the in-memory dedupe is empty, and startup catch-up can query and dispatch that row again. I did not perform a live macOS Messages restart in this read-only sweep.

## Solution Assessment

Yes: the requested direction is the right fix boundary, with the cleanest path being a monotonic durable cursor update from successful live handling or an equivalent persistent provider-message dedupe. The open PR is the current implementation candidate but still needs review/proof before it can close the issue.
