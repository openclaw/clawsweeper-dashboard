## Summary

Keep open. Current main does not already implement `agent.run.status`, and the PR is a useful direction, but the current head still misses the promised metadata on the normal embedded terminal path. The author addressed Greptile's initial `toolCallCount` gap with a counter, but that counter, model, and usage are only attached to the synthetic fallback lifecycle event that normal runs skip.

## What This Changes

The PR adds an `agent.run.status` gateway broadcast event, registers it for discovery and read-scoped subscribers, and wires start/terminal status broadcasts through gateway chat lifecycle handling with model, usage, duration, exit reason, and tool-call metadata.

## Best Possible Solution

Keep the new gateway event concept, but move run-status metadata assembly to the real lifecycle source or a shared lifecycle payload builder so normal started and terminal events consistently carry model, startedAt, duration, usage, tool-count, and exit status; then cover it with focused gateway and embedded-agent tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: normal embedded runs emit real lifecycle terminal events from `handleAgentEnd`, `agent-command.ts` marks `lifecycleEnded = true`, and the PR's enriched synthetic terminal event is skipped.

## Solution Assessment

No. The broadcast surface is reasonable, but enriching only the fallback synthetic lifecycle event is not the narrow maintainable fix because it misses the common terminal path and leaves started-event model data unpopulated.
