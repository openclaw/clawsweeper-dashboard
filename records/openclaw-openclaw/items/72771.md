## Summary

Keep open. The event is useful, but the PR head still misses promised metadata on the normal embedded-run path because model, usage, and tool-count data are added only to a fallback lifecycle event that ordinary terminal lifecycle events skip; the started payload also reads a model field that upstream start events do not emit, and the feature is missing a required changelog entry.

## What This Changes

The PR adds an `agent.run.status` gateway event, registers it for gateway discovery and read-scoped broadcast delivery, and emits started/terminal run-status payloads from gateway chat lifecycle handling.

## Best Possible Solution

Keep the new gateway event concept, but assemble run-status metadata at the real lifecycle source or in a shared lifecycle payload helper so started and terminal events consistently carry model, startedAt, duration, usage, tool-call count, and exit status across normal, error, interrupted, and fallback paths; then add focused tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: normal embedded runs emit real lifecycle terminal events from `handleAgentEnd`, `agent-command.ts` marks `lifecycleEnded = true`, and the PR's enriched fallback terminal event is skipped.

## Solution Assessment

No. Broadcasting a read-scoped run-status event is a reasonable direction, but enriching only the fallback synthetic lifecycle event is not the narrow maintainable fix because it misses the common terminal path and leaves started-event model data unpopulated.
