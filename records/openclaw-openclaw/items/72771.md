## Summary

Keep open: the proposed event surface is useful, but the PR still misses the normal embedded lifecycle path, so the promised start and completion metadata would often be absent.

## What This Changes

The PR adds an `agent.run.status` gateway event, registers it for read-scoped subscribers, and emits started and terminal agent run status payloads from gateway chat handling.

## Best Possible Solution

Keep the new event, but build the run-status metadata at the real lifecycle source or a shared lifecycle payload builder, then cover started, completed, failed, interrupted, usage, and tool-count cases with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: normal embedded runs emit lifecycle `end`/`error` through `handleAgentEnd`, `agentCommandInternal` marks those as terminal, and the PR's enriched fallback event is skipped; `handleAgentStart` also sends no model in the callback that the gateway observes.

## Solution Assessment

No. The event and scope registration are reasonable, but enriching only the synthetic fallback terminal event is not the narrow maintainable fix because it misses the common terminal path and leaves started model data unpopulated.
