## Summary

Keep open: current main still has no `agent.run.status` gateway event, so the PR is not obsolete, but the proposed patch has correctness blockers and no after-fix real behavior proof from the external contributor.

## What This Changes

The PR adds an `agent.run.status` gateway event with started and terminal run metadata, registering it in gateway event lists and broadcasting it from agent lifecycle handling.

## Best Possible Solution

Build the run-status payload from the authoritative lifecycle source or a shared lifecycle payload builder, then cover start, completion, failure, interruption, model, usage, and tool-count behavior with focused gateway and agent tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: normal embedded lifecycle start/end events do not carry the PR-promised metadata, and the enriched fallback event is skipped once real lifecycle end/error arrives.

## Solution Assessment

No. The event surface is plausible, but enriching only the fallback lifecycle event and deriving interruption only from chat abort state is not the narrow maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move terminal metadata onto the real lifecycle path:** `src/agents/agent-command.ts:1020-1022`
  - body: Normal embedded runs emit lifecycle `end` or `error` before this fallback block runs, which sets `lifecycleEnded` and skips the only path where this PR attaches model, usage, and tool-call count. Completion `agent.run.status` events will usually miss the promised terminal metadata.
  - confidence: 0.9
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1018-1027`
  - body: The started-status payload reads `evt.data.model`, but the normal embedded lifecycle start event does not include a model. The new event contract promises a start payload with `model`, so common started events will omit it.
  - confidence: 0.84
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:651-652`
  - body: Terminal lifecycle events can carry `aborted: true` or `stopReason: "aborted"`, but this mapping only checks chat abort bookkeeping. Non-chat aborted runs can therefore be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.8
