## Summary

Keep open: the feature is not on current main and the PR still has merge-blocking behavior gaps in the normal embedded lifecycle path, plus missing real behavior proof from the external contributor.

## What This Changes

The PR adds an `agent.run.status` gateway event and wires started/terminal run metadata through gateway and agent lifecycle code.

## Best Possible Solution

Keep the event concept only if the run-status payload is built from the real lifecycle source or a shared lifecycle payload builder, with tests covering start, completion, failure, interruption, usage, model, and tool-count fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: normal embedded runs emit real lifecycle start/end events without the PR's promised model, usage, and tool-count metadata, and that terminal callback prevents the enriched fallback event from running.

## Solution Assessment

No. The event surface is reasonable, but enriching only the fallback lifecycle event and deriving interruption only from chat abort state is not the narrow maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Populate terminal metadata on the real lifecycle path:** `src/agents/agent-command.ts:1017-1019`
  - body: Normal embedded runs emit lifecycle `end` or `error` before this fallback block runs, which sets `lifecycleEnded` and skips the only path where this PR attaches model, usage, and tool-call count. Completion `agent.run.status` events will usually miss the promised terminal metadata.
  - confidence: 0.9
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1015-1027`
  - body: The started-status payload reads `evt.data.model`, but the normal embedded lifecycle start event does not include a model. The new event contract promises a start payload with `model`, so common started events will omit it.
  - confidence: 0.84
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:648-649`
  - body: Terminal lifecycle events can carry `aborted: true` or `stopReason: "aborted"`, but this mapping only checks chat abort bookkeeping. Non-chat aborted runs can therefore be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.8
