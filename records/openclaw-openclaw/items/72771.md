## Summary

Keep open: the new gateway event is a useful direction, but the branch still misses promised metadata on normal embedded runs because model, usage, and tool-count enrichment is attached to a fallback terminal event that is usually skipped; started events also cannot populate model from the current lifecycle source.

## What This Changes

The PR adds an `agent.run.status` gateway broadcast event, registers it for discovery and read-scoped subscribers, and emits started and terminal agent run payloads from gateway chat lifecycle handling.

## Best Possible Solution

Keep the new event, but assemble run-status metadata at the real lifecycle source or a shared lifecycle payload builder so started and terminal events consistently carry model, timestamps, duration, usage, tool-count, and exit status, then add focused tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current `handleAgentEnd` emits the real terminal callback that sets `lifecycleEnded = true`, so the PR's enriched fallback event is skipped in normal embedded runs; current `handleAgentStart` callback also omits model.

## Solution Assessment

No. The event surface is reasonable, but enriching only the fallback synthetic terminal event is not the narrow maintainable fix because it misses the common terminal path and leaves started model data unpopulated.
