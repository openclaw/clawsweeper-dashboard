## Summary

Keep open. The gateway event is a useful direction, but the current PR still misses the common terminal lifecycle path: normal embedded runs set `lifecycleEnded` from the real lifecycle `end`/`error` event, so the PR’s enriched fallback event is skipped and the promised completion metadata is usually absent.

## What This Changes

The PR adds an `agent.run.status` gateway broadcast event, registers it for read-scoped subscribers, and emits started/completed/failed/interrupted status from gateway agent lifecycle handling.

## Best Possible Solution

Keep the new gateway event concept, but build the run-status payload from the real lifecycle source or a shared lifecycle payload builder so started and terminal events consistently carry model, timing, usage, and tool-count data; then add focused gateway/agent tests and a changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is enough: normal embedded runs emit lifecycle `end`/`error` from `handleAgentEnd`, `agentCommandInternal` sets `lifecycleEnded = true`, and the PR’s enriched synthetic terminal event is skipped.

## Solution Assessment

No. The broadcast surface is reasonable, but enriching only the fallback synthetic lifecycle event is not the narrow maintainable fix because it misses the common terminal path and leaves started-event model data unpopulated.
