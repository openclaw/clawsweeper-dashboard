## Summary

Keep open. Current main does not have the new gateway event, and the PR branch still misses the normal lifecycle path because its enriched terminal metadata is added only to the fallback synthetic lifecycle event, which normal embedded runs skip after the real lifecycle end/error event fires. The branch also lacks the required changelog entry for a new user-facing gateway event surface.

## What This Changes

The PR adds an `agent.run.status` gateway broadcast event, registers it for read-scoped WebSocket discovery, and derives started/terminal payloads from agent lifecycle events with model, usage, duration, exit reason, and tool-call metadata.

## Best Possible Solution

Keep the new event concept, but assemble run-status metadata at the real lifecycle source or a shared lifecycle payload builder so normal start/end/error events consistently carry model, startedAt, duration, usage, tool-count, and exit state; then cover the gateway broadcast and embedded lifecycle paths with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: normal embedded runs emit real lifecycle terminal events from `handleAgentEnd`, `agent-command.ts` then sets `lifecycleEnded = true`, and the PR's enriched fallback terminal block is skipped.

## Solution Assessment

No. The broadcast surface is reasonable, but enriching only the fallback synthetic lifecycle event is not the narrow maintainable fix because it misses the common terminal path and leaves started-event model data unpopulated.
