## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the resolved event agent id:** `src/gateway/server-chat.ts:655`
  - body: The new status payload derives `agentId` by reparsing `sessionKey`, but this handler already computes `sessionAgentId` from the chat link or event. For unscoped or legacy session keys with a non-default agent, subscribers will see the status event attributed to `main` instead of the run's real agent.
  - confidence: 0.86
- **[P2] Move terminal metadata onto the real lifecycle path:** `src/agents/agent-command.ts:1019-1022`
  - body: The completion metadata is attached to a fallback terminal emit path, while normal embedded runs complete through the lifecycle callback/helper path. Completed and failed `agent.run.status` events will commonly miss the promised model, usage, and tool-call count fields.
  - confidence: 0.9
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:651-652`
  - body: The status mapping only checks chat abort bookkeeping. Terminal lifecycle events can carry `aborted: true`, so non-chat aborted runs can be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.84
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1018-1027`
  - body: The started payload reads `evt.data.model`, but the normal embedded lifecycle start callback forwards only `phase: "start"`. The event contract promises `model` on start, so common started events will omit it.
  - confidence: 0.82
