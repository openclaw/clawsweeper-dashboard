## Summary

Keep open: the feature is not on current main, but the PR still has correctness blockers in lifecycle metadata propagation and lacks after-fix real behavior proof from the contributor.

## What This Changes

The PR adds an `agent.run.status` gateway event for agent run start and terminal status with model, timing, usage, tool-count, and exit metadata.

## Best Possible Solution

Keep the new event concept, but build the run-status payload from the real lifecycle source or a shared lifecycle payload builder so started and terminal events consistently carry model, timing, usage, tool-count, exit status, and abort status, then add focused tests and contributor proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: normal embedded runs emit real lifecycle start/end callbacks that omit the PR's promised metadata, and those callbacks cause the enriched fallback emit to be skipped.

## Solution Assessment

No. The event surface is reasonable, but enriching only the fallback terminal event and reading model from lifecycle data that is normally absent is not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Populate terminal metadata on the real lifecycle path:** `src/agents/agent-command.ts:1017-1019`
  - body: Normal embedded runs already emit lifecycle `end`/`error` before this fallback block runs, which sets `lifecycleEnded` and skips the only place this PR attaches model, usage, and tool-call count. Completion `agent.run.status` events will therefore usually miss the promised metadata and duration.
  - confidence: 0.9
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1015-1027`
  - body: The started-status payload reads `evt.data.model`, but the upstream lifecycle start callback only sends `{ phase: "start" }`. Normal started events will omit the model promised by the new event contract.
  - confidence: 0.84
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:648-649`
  - body: Terminal lifecycle events can carry `aborted: true` or `stopReason: "aborted"`, but the new status mapping only checks chat abort bookkeeping. Non-chat aborted runs can be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.8
