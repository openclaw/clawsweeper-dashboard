## Summary

Keep open: this is a meaningful Gateway observability feature, but it is not merge-ready because the proposed public event duplicates existing run lifecycle surfaces, has identity and correlation defects, lacks real behavior proof, and is currently merge-conflicting.

## What This Changes

The PR adds a new `agent.run.status` Gateway event, emits started and terminal run-status payloads from `server-chat`, and adds model, usage, and tool-call-count metadata to synthetic agent lifecycle end events.

## Best Possible Solution

If maintainers want this event, define it as an intentional extension of the existing lifecycle contract with correct `sessionAgentId`, `runId`, SDK/docs/tests, and real Gateway-client proof; otherwise keep run observability on the existing `agent` plus `agent.wait` path and leave durable status design in the related issue.

## Reproduction Assessment

Yes for the patch defects by source inspection: a `global` run linked to a non-main agent would be emitted as `main`, and the new status payload has no `runId`. I did not run live Gateway proof because this review is read-only.

## Solution Assessment

No: this is only a plausible slice, not the best settled contract. The safer path is to extend the existing lifecycle/SDK/`agent.wait` contract or explicitly define the new event with correct identity, correlation, tests, docs, and real proof.
