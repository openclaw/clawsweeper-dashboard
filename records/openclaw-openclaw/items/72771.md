## Summary

Keep this PR open, but it is not ready to merge: current main does not already provide `agent.run.status`, while the PR still has source-visible lifecycle metadata gaps and lacks required real behavior proof.

## What This Changes

The PR adds a new `agent.run.status` gateway event contract and maps agent lifecycle start/end/error events into started, completed, failed, or interrupted broadcasts.

## Best Possible Solution

Add the event only after enriching the authoritative lifecycle source, defining the public payload contract, and covering started/completed/failed/interrupted cases with focused gateway and agent tests.

## Reproduction Assessment

Yes. Source inspection against the PR diff and current lifecycle code shows normal embedded start/end/error paths do not carry the promised metadata, and the new terminal metadata is attached only to a fallback path skipped after normal lifecycle completion.

## Solution Assessment

No. The event surface may be useful, but the implementation should build from the authoritative lifecycle payload or a shared payload builder instead of relying on fallback-only metadata and chat-only abort bookkeeping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move terminal metadata onto the real lifecycle path:** `src/agents/agent-command.ts:1020-1022`
  - body: Normal embedded runs emit lifecycle `end` or `error` before this fallback block runs, which sets `lifecycleEnded` and skips the only path where the PR attaches model, usage, and tool-call count. Completion and failure `agent.run.status` events will usually miss the promised terminal metadata.
  - confidence: 0.9
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1018-1027`
  - body: The started-status payload reads `evt.data.model`, but the normal embedded lifecycle start callback forwards only `phase: "start"`. The new event contract promises a start payload with `model`, so common started events will omit it.
  - confidence: 0.84
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:651-652`
  - body: Terminal lifecycle events can carry abort state, but this mapping only checks chat abort bookkeeping. Non-chat aborted runs can therefore be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.8
