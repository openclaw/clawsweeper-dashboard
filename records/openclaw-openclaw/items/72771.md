## Summary

This PR should stay open: current main does not already provide the new gateway event, but the proposed implementation still misses promised metadata on normal lifecycle paths and lacks external real behavior proof before merge.

## What This Changes

The branch adds a new `agent.run.status` gateway event, registers it for broadcast/listing, and maps agent lifecycle start/end/error events into started/completed/failed/interrupted payloads.

## Best Possible Solution

Build the run-status payload from the authoritative lifecycle source or a shared lifecycle payload builder, then cover the public event contract with focused gateway and agent tests before merging.

## Reproduction Assessment

Yes. Source inspection against the PR head shows normal embedded start/end/error lifecycle events do not carry the promised metadata, and the new terminal metadata is added only in the fallback block skipped after real lifecycle completion.

## Solution Assessment

No. The event surface may be useful, but the implementation should enrich the authoritative lifecycle path or shared payload builder instead of relying on fallback-only metadata and chat-only interruption detection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Move terminal metadata onto the real lifecycle path:** `src/agents/agent-command.ts:1020-1022`
  - body: Normal embedded runs emit lifecycle `end` or `error` before this fallback block runs, which sets `lifecycleEnded` and skips the only path where the PR attaches model, usage, and tool-call count. Completion and failure `agent.run.status` events will usually miss the promised terminal metadata.
  - confidence: 0.9
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1018-1027`
  - body: The started-status payload reads `evt.data.model`, but the normal embedded lifecycle start event forwards only `phase: "start"` through `onAgentEvent`. The new event contract promises a start payload with `model`, so common started events will omit it.
  - confidence: 0.84
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:651-652`
  - body: Terminal lifecycle events can carry abort state, but this mapping only checks chat abort bookkeeping. Non-chat aborted runs can therefore be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.8
