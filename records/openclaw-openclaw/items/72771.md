  - body: This advertises a new public `agent.run.status` event even though clients already have raw `agent` lifecycle frames, SDK-normalized `run.*` events, `agent.wait`, and usage RPCs.   - body: For selected-agent `global` runs, deriving `agentId` from `sessionKey` resolves to the default agent, while the existing lifecycle path keeps `chatLink?.agentId ?? evt.agentId`. Status subscribers can receive frames for a non-default run labeled as `main`.
  - confidence: 0.88
- **[P1] Keep terminal status aligned with lifecycle data:** `src/gateway/server-chat.ts:651-652`
  - body: The new status calculation only checks the chat abort marker, so lifecycle `end` events carrying `aborted`, timeout attribution, or cancellation stop reasons can still broadcast `completed`. Existing `agent.wait`, session lifecycle persistence, and SDK normalization classify those same payloads as timeout, cancelled, killed, or failed.
  - confidence: 0.87
- **[P2] Include runId in status events:** `src/gateway/events.ts:13-16`
  - body: The new status payload includes session and timestamps but no `runId`, so clients cannot reliably pair started and terminal frames when a session has retries, reused idempotency keys, or overlapping run activity. Existing lifecycle and SDK events already carry run identity.
  - confidence: 0.86
