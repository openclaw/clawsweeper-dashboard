## Summary

Keep open. Current main does not yet expose `agent.run.status`, and the PR is useful, but the proposed metadata propagation misses the normal terminal lifecycle path and lacks the required changelog entry for a new public gateway event.

## What This Changes

The PR adds an `agent.run.status` gateway broadcast event, registers it for discovery and read-scoped subscribers, and attempts to emit started/completed/failed/interrupted run status with timing, model, usage, tool-call count, and exit reason metadata.

## Best Possible Solution

Keep the gateway event idea, but move metadata assembly to the real lifecycle source or a shared lifecycle payload builder so started and terminal events carry consistent model, timing, usage, and tool-count fields; add focused gateway/agent tests and a changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is enough: normal embedded runs emit lifecycle `end`/`error` from `handleAgentEnd`, `agentCommandInternal` marks `lifecycleEnded = true`, and the PRâ€™s enriched synthetic terminal event is skipped.

## Solution Assessment

No. The broadcast surface is a reasonable direction, but enriching only the fallback synthetic lifecycle event is not the narrow maintainable fix because it misses the common terminal path and leaves started-event model data unpopulated.
