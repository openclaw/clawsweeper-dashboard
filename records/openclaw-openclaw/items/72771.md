- **[P1] Use one canonical run lifecycle contract:** `src/gateway/server-methods-list.ts:183`
  - body: This adds and advertises a new public `agent.run.status` event even though current clients already have raw `agent` lifecycle frames, SDK-normalized `run.*` events, and `agent.wait` for terminal completion.   - confidence: 0.9
- **[P1] Preserve selected-global agent ids:** `src/gateway/server-chat.ts:656`
  - body: For selected-agent `global` runs, deriving the event agent id from `sessionKey` resolves to the default agent, while the existing lifecycle path keeps the active run's `chatLink?.agentId ?? evt.agentId`. Subscribers can receive status for a non-default run labeled as `main`.
  - confidence: 0.88
- **[P1] Keep terminal status aligned with lifecycle data:** `src/gateway/server-chat.ts:651-652`
  - body: The new status calculation only checks the chat abort marker, so a lifecycle `end` carrying `aborted`, timeout attribution, or cancellation stop reasons can still broadcast `completed`. Existing `agent.wait` and SDK normalization classify those same lifecycle payloads as timeout, cancelled, or failed, so this event can contradict the canonical terminal result.
  - confidence: 0.87
- **[P2] Include runId in status events:** `src/gateway/events.ts:13-16`
  - body: The new status payload includes session and timestamps but no `runId`, so clients cannot reliably pair started and terminal frames when a session has retries, reused idempotency keys, or overlapping runs. Existing lifecycle and SDK events already carry run identity.
  - confidence: 0.86
