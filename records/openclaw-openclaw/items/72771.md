## Summary

Keep this PR open: current main does not already provide `agent.run.status`, and the branch still has merge-blocking correctness, proof, and conflict issues around a new subscriber-visible gateway event contract.

## What This Changes

The PR adds a new gateway `agent.run.status` event contract, registers it for subscribers, and maps agent lifecycle events into started, completed, failed, or interrupted status payloads.

## Best Possible Solution

Define the public status payload contract, enrich the authoritative lifecycle source for start/end/error/abort metadata, then cover started, completed, failed, and interrupted broadcasts with focused tests and real subscriber proof.

## Reproduction Assessment

Yes for the PR defects. Source inspection shows normal embedded lifecycle start/end/error paths do not carry the metadata the new event promises, and the new terminal metadata is attached only to a fallback path skipped after normal lifecycle completion.

## Solution Assessment

No. The event surface may be useful, but the implementation should build from the authoritative lifecycle payload or a shared payload builder instead of relying on fallback-only metadata and chat-only abort bookkeeping.

Full review comments:

- **[P2] Move terminal metadata onto the real lifecycle path:** `src/agents/agent-command.ts:1000-1019`
  - body: Normal embedded runs set `lifecycleEnded` when lifecycle `end` or `error` arrives, so this fallback block is skipped on the common path. Because the PR adds `model`, `usage`, and `toolCallCount` only to this fallback emit, completed and failed status events will usually miss the promised terminal metadata.
  - confidence: 0.9
- **[P3] Populate model on started status events:** `src/gateway/server-chat.ts:1016-1027`
  - body: The started payload reads `evt.data.model`, but the normal embedded lifecycle start callback forwards only `{ phase: "start" }`. That means common `started` events omit the `model` field promised by the new event contract.
  - confidence: 0.84
- **[P2] Derive interrupted status from lifecycle abort data:** `src/gateway/server-chat.ts:651-652`
  - body: The mapping checks chat abort bookkeeping rather than lifecycle abort data, so aborted runs outside that chat bookkeeping path can be broadcast as `completed` instead of the promised `interrupted` status.
  - confidence: 0.8
