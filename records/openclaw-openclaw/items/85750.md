## Summary

Keep open: current main and the latest release still have authenticated avatar-route behavior that can render protected `/avatar/<agentId>` paths directly on some Control UI surfaces, and two open PRs are already carrying the repair discussion instead of this being cleanup-closeable.

## What This Changes

The issue reports that authenticated Control UI webchat sessions over Tailscale fetch `/avatar/main` without usable bearer auth, producing 401 responses and a missing assistant avatar.

## Best Possible Solution

Use the linked PRs to converge on one safe fix that renders Control UI avatars through authenticated blob URLs or safe fallbacks, avoiding reusable gateway tokens in image URLs.

## Reproduction Assessment

No live high-confidence Tailscale reproduction was established here. The source, docs, tests, and reporter curl/browser evidence make the auth-gated avatar route failure source-reproducible enough to keep open.

## Solution Assessment

Unclear. The safe direction is to keep bearer-authenticated fetches and blob rendering, but the same-author backend query-token PR needs maintainer security review before it can be treated as the right fix.
