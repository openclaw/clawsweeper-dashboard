## Summary

Keep open: current main and v2026.6.9 still have a Quick Settings path that can hand `/avatar/<agentId>` directly to an image renderer while the gateway intentionally rejects unauthenticated avatar reads. The remaining focused fix is tracked by https://github.com/openclaw/openclaw/pull/83235, which is still open and needs maintainer/security review before this issue should close.

## What This Changes

The issue reports authenticated Control UI sessions receiving 401s for `/avatar/<agentId>` avatar image requests over Tailscale instead of rendering via the documented bearer-fetch/blob path.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers refresh, review, and land https://github.com/openclaw/openclaw/pull/83235 or an equivalent bearer/blob or short-lived-ticket fix that preserves avatar route auth.

## Reproduction Assessment

No for the exact Tailscale browser session, because this read-only review did not run a live Tailscale setup. Yes at source level: current main can pass `/avatar/<agentId>` directly into Quick Settings while the gateway rejects unauthenticated avatar reads.

## Solution Assessment

Yes for the implementation direction, but the open fix still needs maintainer/security review: the best shape is to keep bearer-authenticated fetch plus blob/fallback rendering, not reusable query-token avatar auth.
