## Summary

Keep open: current main and v2026.6.8 already cover the main webchat bearer-fetch/blob path, but current main still has a direct protected local avatar route path in Quick Settings and an open candidate PR is tracking that remaining behavior. This is auth-boundary-sensitive and not safe to close as implemented yet.

## What This Changes

Authenticated Control UI sessions over Tailscale should render configured local avatars without unauthenticated `/avatar/<agentId>` 401s by using bearer-authenticated avatar fetches and blob rendering.

## Best Possible Solution

Keep this issue open as the canonical tracker until maintainers refresh and land https://github.com/openclaw/openclaw/pull/83235 or choose a scoped-ticket/blob-based alternative that avoids reusable query-token avatar auth.

## Reproduction Assessment

No for the exact Tailscale webchat browser flow; this review did not run a live Tailscale repro. Yes at source level for the remaining protected local avatar preview path: current main can still hand `/avatar/<agentId>` directly to Quick Settings while the route rejects unauthenticated image loads.

## Solution Assessment

Unclear as a product/security decision, but the safest implementation direction is clear: keep the existing bearer-fetch/blob contract or use a short-lived scoped ticket, not reusable query-token avatar auth.
