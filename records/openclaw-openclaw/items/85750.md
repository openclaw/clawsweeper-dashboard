## Summary

Keep open: current main and v2026.6.9 still contain a Control UI Quick Settings path that can render a protected `/avatar/<agentId>` route directly while the gateway intentionally rejects unauthenticated avatar reads. The focused remaining repair is already tracked by https://github.com/openclaw/openclaw/pull/83235, which is open but needs refresh plus maintainer/security review.

## What This Changes

The issue requests that authenticated Control UI webchat sessions fetch `/avatar/<agentId>` with the gateway bearer token so avatars do not 401 when accessed through Tailscale.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers refresh and review https://github.com/openclaw/openclaw/pull/83235 or choose an equivalent scoped bearer/blob or short-lived-ticket fix that preserves avatar route auth.

## Reproduction Assessment

No for the exact Tailscale webchat browser flow, because this read-only review did not run a live Tailscale session. Yes at source level for the remaining path: current main can still pass `/avatar/<agentId>` directly into Quick Settings while the gateway rejects unauthenticated avatar reads.

## Solution Assessment

Yes for the implementation direction, but not ready to close: reusing the existing bearer-authenticated blob/fallback path is narrower and safer than query-token image URLs. Maintainers still need to refresh and review the open candidate PR or choose an equivalent security-preserving shape.
