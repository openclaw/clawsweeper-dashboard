## Summary

Keep open: current main and v2026.6.9 still have a Quick Settings path that can hand `/avatar/<agentId>` directly to an image renderer while the gateway intentionally rejects unauthenticated avatar reads. The focused remaining fix is tracked by https://github.com/openclaw/openclaw/pull/83235, which is still open and needs maintainer/security review before this issue should close.

## What This Changes

The issue reports authenticated Control UI sessions showing 401s for local agent avatar requests because some UI paths do not use the documented bearer-fetch/blob avatar flow.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers refresh, review, and land https://github.com/openclaw/openclaw/pull/83235 or choose another scoped bearer/blob or short-lived-ticket fix that preserves avatar route auth.

## Reproduction Assessment

No for the exact Tailscale webchat browser flow, because this read-only review did not run a live Tailscale session. Yes at source level for the remaining path: current main can still pass `/avatar/<agentId>` directly into Quick Settings while the gateway rejects unauthenticated avatar reads.

## Solution Assessment

Unclear until maintainers choose the final security shape, but the safest implementation direction is clear: keep bearer-authenticated fetch plus blob rendering, or use a short-lived scoped ticket if a native image URL is required. Reusable query-token avatar auth should stay out of the default path.
