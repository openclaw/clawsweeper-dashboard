## What This Changes

The issue reports that authenticated Control UI webchat over Tailscale fails to render `/avatar/main` because the avatar request returns 401 despite a valid gateway session.

## Best Possible Solution

Keep the avatar route on the documented bearer-authenticated fetch/blob contract, or add a short-lived scoped avatar ticket only if a native image URL path is truly required, then close this issue after the chosen PR lands with redacted Tailscale proof.

## Reproduction Assessment

No, not for the exact Tailscale webchat failure. Source and tests prove the protected avatar route and intended bearer/blob path, and current main still has a related direct protected-route UI path, but I did not establish a live Tailscale reproduction.

## Solution Assessment

Unclear: forwarding bearer auth into blob-backed avatar rendering is the right contract, but current main already does that for the main chat fetch path. The open query-token fallback is not the best direction unless maintainers explicitly accept the credential-exposure tradeoff.
