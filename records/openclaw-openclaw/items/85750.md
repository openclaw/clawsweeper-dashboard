## Summary

Keep open: the issue describes a real authenticated Control UI avatar failure, current main and the latest tag still have direct protected avatar URL paths, and two open related PRs are already competing over the safe fix direction.

## What This Changes

The issue reports that authenticated Control UI webchat sessions over Tailscale can still request `/avatar/main` without bearer auth, returning 401 and hiding the assistant avatar.

## Best Possible Solution

Keep the avatar route protected and remove direct rendered `/avatar/<agentId>` URLs from Control UI surfaces by reusing bearer-authenticated fetch/blob rendering or short-lived scoped tickets, then land the safe related PR path with live Tailscale proof.

## Reproduction Assessment

Partially: source and tests show the avatar route is auth-gated and that the intended fetch path uses bearer auth, while current main still has direct protected avatar URL rendering paths. I did not establish a live Tailscale browser reproduction.

## Solution Assessment

Unclear: the requested bearer-forwarding direction matches docs, but the same-author PR's query-token fallback is not the best safe fix. The safer path is to finish bearer/blob or scoped-ticket rendering across every Control UI avatar surface.
