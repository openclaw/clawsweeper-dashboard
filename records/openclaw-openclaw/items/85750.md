## Summary

Keep open: current main and the latest release already use bearer-authenticated fetch/blob rendering for the main webchat avatar path, but a related direct protected-avatar rendering path still exists and is covered by an open PR; the broader Tailscale media/auth question is also still open, so this is not safe cleanup closure.

## What This Changes

The issue reports authenticated Control UI sessions over Tailscale failing to render agent avatars because `/avatar/main` returns 401 unless called with a bearer token.

## Best Possible Solution

Keep this open until maintainers decide whether the focused Quick Settings PR or the broader Tailscale media/auth issue is the canonical path, and preserve bearer/blob rendering or scoped tickets rather than reusable query-token avatar auth.

## Reproduction Assessment

No live high-confidence Tailscale reproduction was run. Source inspection makes the related protected-image path reproducible: the gateway rejects unauthenticated `/avatar/main`, the webchat refresh helper sends Authorization headers, and Quick Settings can still render `/avatar/main` directly before a blob URL exists.

## Solution Assessment

Unclear. Bearer-authenticated fetch plus blob rendering is the documented contract and is already present for the main webchat path, but the remaining direct-avatar and Tailscale media auth cases need a coordinated maintainer/security decision.
