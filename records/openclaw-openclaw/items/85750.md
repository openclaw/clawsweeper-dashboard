## Summary

Keep open: current main and v2026.6.8 cover the main webchat avatar fetch with bearer-authenticated blob rendering, but current main still has a Quick Settings path that can hand a protected `/avatar/<agentId>` route directly to an image preview, and https://github.com/openclaw/openclaw/pull/83235 is the open candidate for that remaining path.

## What This Changes

The issue asks for authenticated Control UI avatar rendering over Tailscale so `/avatar/<agentId>` does not 401 for logged-in webchat sessions.

## Best Possible Solution

Refresh and review https://github.com/openclaw/openclaw/pull/83235 or an equivalent UI-side fix so Quick Settings never renders protected local avatar routes directly, with redacted live proof and no reusable query-token avatar auth.

## Reproduction Assessment

No live high-confidence Tailscale reproduction was run. Source inspection shows the remaining protected-image path: the gateway rejects unauthenticated `/avatar/main`, webchat fetches with Authorization, and Quick Settings can still render `/avatar/main` directly for local avatars.

## Solution Assessment

Unclear for the whole issue until maintainers finish the security/product call. The documented bearer-fetch plus blob-rendering path is the safest current shape, and the open Quick Settings PR appears to be the narrow remaining implementation path once refreshed and proven.
