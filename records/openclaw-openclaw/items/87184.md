## Summary

Keep open for review, but the patch should not merge as written: current and shipped OpenClaw behavior treats MiniMax `current_*_usage_count` values as remaining quota, and this PR would reintroduce the false exhaustion path reported by users.

## What This Changes

The PR moves four MiniMax `current_*_usage_count` fields from remaining-count handling to used-count handling and updates MiniMax usage tests to expect that semantic change.

## Best Possible Solution

Preserve the shipped remaining-count handling and either close this PR or rework it only around new redacted live evidence for a specific endpoint whose semantics differ.

## Reproduction Assessment

Yes for the introduced regression from source: with total 1500 and `current_interval_usage_count` 1473, moving the key to `USED_KEYS` computes 98.2% used instead of roughly 1.8% used. I did not run a live MiniMax call in this read-only review.

## Solution Assessment

No. The narrow maintainable path is to keep the shipped remaining-count interpretation for `/v1/token_plan/remains`; only add endpoint-specific handling if new redacted live evidence proves a different endpoint returns used counts.
