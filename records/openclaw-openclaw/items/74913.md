## Summary

Keep this PR open. The linked cron transcript bug is valid and current main still lacks direct-delivery mirroring, but the patch is not merge-ready because it can mirror the full transcript text during partial bestEffort sends that cron intentionally marks undelivered for retry, and it is missing the required user-facing changelog entry.

## What This Changes

The branch adds direct cron delivery transcript mirroring into the agent-scoped destination session using the filtered outbound payload projection, plus regression coverage for custom-session and media-only silent-token deliveries.

## Best Possible Solution

Keep the cron-owned mirror approach, but make transcript appending respect delivery success semantics: append only after a fully successful direct delivery, or mirror only the payload subset that actually sent, then add the active-version changelog entry and keep #74286/#75529 as separate follow-up surfaces.

## Reproduction Assessment

Yes. The linked bug has concrete custom-session cron CLI steps, and current main shows direct cron delivery does not pass a mirror. The PR-specific regression is reproducible from existing bestEffort semantics: partial sends call onError while still returning results, and cron then marks the run undelivered for retry.

## Solution Assessment

No as-is. The selected cron dispatch surface is the right small place to fix #74743, but the mirror must be gated on full success or successful-payload knowledge so transcript state cannot diverge from what the user actually received.

Overall confidence: 0.88

Full review comments:

- **[P2] Gate mirroring on fully successful bestEffort sends:** `src/cron/isolated-agent/delivery-dispatch.ts:617-623`
  - body: This passes a transcript mirror into `deliverOutboundPayloads` even when `deliveryBestEffort` is true. A partial send can therefore write the full mirrored text with the run idempotency key before the retry, leaving the destination transcript out of sync and blocking correction on the later successful attempt.
