## Summary

Keep open. The linked bug is real and this PR targets the right cron delivery surface, but the current head still mirrors transcript content before cron can confirm a fully successful best-effort delivery, builds the mirror with a different normalization context than the actual send path, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds transcript mirroring for direct cron deliveries into the resolved agent-scoped destination session, with regression coverage for custom-session and media/silent-token delivery projections.

## Best Possible Solution

Keep the narrow cron-owned fix, but derive transcript mirror content from the same context-aware outbound payload plan used for delivery and append only after all direct delivery payloads succeed. Add focused regression coverage for partial bestEffort delivery and context-sensitive payload projection, then add the active-version changelog entry before landing.

## Reproduction Assessment

Yes. The linked bug has concrete CLI reproduction steps, and current main shows the direct cron delivery path has no mirror object. The PR-specific blockers are reproducible from existing bestEffort partial-replay coverage and by comparing the new context-free mirror projection with deliverOutboundPayloadsCore's context-aware plan.

## Solution Assessment

No as-is. The chosen cron delivery seam is appropriate, but the implementation should mirror after full success and use the same delivery projection context; otherwise it can persist transcript content that was not actually delivered or cannot be corrected on retry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate transcript mirroring on full delivery success:** `src/cron/isolated-agent/delivery-dispatch.ts:617-623`
  - body: This passes a mirror into deliverOutboundPayloads even when deliveryBestEffort is true. That helper appends the mirror as soon as any payload returns a result, while cron's existing hadPartialFailure path marks the run not delivered and retries. A partial send can therefore persist the full mirrored transcript with the run idempotency key before the later retry, leaving the destination session out of sync and preventing the successful retry from correcting it.
  - confidence: 0.9
- **[P2] Build the mirror from the delivery plan's context:** `src/cron/isolated-agent/delivery-dispatch.ts:590-592`
  - body: The new mirrorProjection uses createOutboundPayloadPlan(payloadsForDelivery) without the cfg, sessionKey, surface, conversationType, and channel directive options that deliverOutboundPayloadsCore supplies for the actual send. For context-sensitive cases such as Telegram markdown-image extraction or session-specific silent-reply policy, the transcript mirror can diverge from the payload that was delivered.
  - confidence: 0.83
