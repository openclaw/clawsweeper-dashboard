## Summary

Keep open. The branch targets a real current-main cron transcript gap, but it is not merge-ready because the new mirror can write a full transcript entry during a partial best-effort delivery that cron intentionally treats as undelivered and retries; it also omits the required changelog entry.

## What This Changes

The PR adds direct cron delivery transcript mirroring for destination session keys and regression coverage for `session:<id>` delivery plus filtered media/silent payload projection.

## Best Possible Solution

Keep the cron-owned direct-delivery mirror, but make transcript appends respect delivery success semantics by appending only after a fully successful direct delivery or by passing only the payload subset proven to have sent; then add the active-version changelog entry and land with focused cron delivery tests.

## Reproduction Assessment

Yes. The linked bug has concrete CLI steps for `openclaw cron run` with `--session session:<id>`, and current main source shows direct cron delivery sends externally without a transcript mirror. The PR-specific regression is reproducible at the unit seam by simulating best-effort `onError` plus one successful result.

## Solution Assessment

No as-is. The destination-session mirror is the right narrow area, but passing the mirror into `deliverOutboundPayloads` before cron knows the send fully succeeded is not the safest implementation; gate the mirror after full success or make the outbound helper success-subset aware.
