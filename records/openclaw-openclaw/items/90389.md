## Summary

Keep open: current main still keeps Mattermost slash account state in a module-local Map, while this PR moves that state to a Symbol.for globalThis singleton and adds focused reload coverage for the dual-loader 503 regression. ## What This Changes

The PR changes Mattermost slash-command account state from a module-local Map to a Symbol.for globalThis-backed singleton and adds regression tests for state sharing across module reloads.

## Best Possible Solution

Land this PR or an equivalent Mattermost plugin fix after normal maintainer and CI review, then let the linked bug close from the merged fix.

## Reproduction Assessment

Yes, at source level: current main registers the route through the bundled-entry loader while monitor activation writes module-local accountStates, and the linked issue shows the resulting post-activation 503. I did not run a live Mattermost gateway in this read-only review.

## Solution Assessment

Yes: a process-global Symbol.for-backed Map is the narrowest maintainable fix for duplicate module instances and matches existing globalThis singleton patterns used for similar loader-boundary state. Forcing a single loader instance would be a broader plugin loader refactor for the same bug.
