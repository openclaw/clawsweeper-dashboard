Keep open: current main still does not show a QQBot replay/idempotency guard on the implicated queue path, and the reporter’s corrected logs leave a real unresolved question about stable message-id redelivery versus distinct-ID outbound retry behavior.

## What This Changes

The issue reports repeated QQBot outbound replies from one inbound QQ interaction, with the reporter later retracting the WebChat polling theory and asking whether QQBot needs message-id dedupe or outbound idempotency.

## Best Possible Solution

Add QQBot-owned idempotency at the boundary proven by live evidence: an enqueue/preflight message-id replay guard for stable redeliveries, or outbound send idempotency/retry handling if QQ is retrying visible replies with distinct inbound IDs.
