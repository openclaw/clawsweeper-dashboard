## Summary

Keep open: the report is fresh and plausible, but current `main` does not provide enough evidence for a safe cleanup close. The `chat.history` path is a read-only transcript projection, while outbound `message_sending` hooks and inbound dedupe live in the dispatch path, so the actual QQBot repeat-send path still needs a focused reproduction or regression test.

## What This Changes

The issue asks OpenClaw to prevent QQBot replies from being resent when WebChat polling/history interacts with a `message_sending` hook.

## Best Possible Solution

Add a focused regression that reproduces the QQBot/WebChat polling resend path on current main, then fix the dispatch boundary that re-enters channel delivery; avoid adding `isReplay` hook API unless the failing path proves plugins need that new contract.

## Reproduction Assessment

No: the reporter supplied a concrete live path, but I did not reproduce it on current `main`, and the inspected `chat.history` handler does not call `message_sending` directly. The best current source check narrows the suspect area to dispatch/re-delivery rather than history projection itself.

## Solution Assessment

Unclear: skipping hooks in `chat.history` is not the narrow current-main fix because `chat.history` appears read-only. A maintainable fix should first prove which QQBot/WebChat path is re-dispatching the already-sent reply, then add dedupe or routing guards there.
