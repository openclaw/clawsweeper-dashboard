## Summary

Keep open: the duplicate-send report is recent and plausible, but current `main` does not confirm the original WebChat replay mechanism, and the updated comments now point to either QQBot queue/admission dedupe or QQ platform outbound retry behavior that needs targeted reproduction.

## What This Changes

Prevent QQBot from sending the same assistant reply multiple times when duplicate or retried QQ events occur with `message_sending` hooks enabled.

## Best Possible Solution

Reproduce on current `main` with QQBot plus a mutating `message_sending` hook, then fix the proven boundary: queue/admission dedupe for same-id redelivery, or outbound idempotency/retry handling if QQ retries sends under different inbound ids.

## Reproduction Assessment

No. The issue has concrete user logs, but the original WebChat replay path conflicts with current source, exact same-`MessageSid` replays should be skipped by generic inbound dedupe, and the revised different-id QQ retry hypothesis still needs a live or fixture-backed reproduction.

## Solution Assessment

Unclear. Queue-level `messageId` dedupe would be narrow for same-id redelivery, but the revised evidence may require outbound idempotency or acknowledgement handling instead, so the boundary should be proven before a repair PR is queued.
