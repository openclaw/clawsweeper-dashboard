## Summary

Keep open: the original WebChat-history mechanism has been retracted and current source agrees it is read-only, but the QQBot duplicate-send symptom remains plausible and needs targeted QQBot reproduction to identify whether the fix belongs at inbound message-id dedupe or outbound API idempotency.

## What This Changes

Prevent QQBot from sending the same assistant reply multiple times when QQ gateway/API retry or redelivery behavior repeats a message flow.

## Best Possible Solution

Fix the QQBot-owned boundary that reproduction proves: add a scoped inbound idempotency guard before queue processing if duplicates share stable message IDs, or add outbound idempotency/ack handling if QQ API retry behavior is responsible.

## Reproduction Assessment

No, not at high confidence. Source inspection shows that two QQBot DISPATCH payloads with the same stable message id would enqueue and process twice on current main, but the reporter's revised logs also mention different incoming message ids and no live QQBot reproduction was established.

## Solution Assessment

Unclear. Skipping WebChat hooks is not the right fix; QQBot inbound message-id dedupe is likely narrow if replayed events share ids, while outbound idempotency is the safer target if the QQ API is retrying sends after slow acknowledgement.
