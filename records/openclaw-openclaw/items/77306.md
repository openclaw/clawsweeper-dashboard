## Summary

Keep open: the original WebChat-history mechanism was retracted and current source supports that correction, but the QQBot duplicate-send symptom remains plausible and needs a focused QQBot reproduction before choosing queue-level dedupe versus outbound idempotency.

## What This Changes

The issue requests that QQBot send each assistant reply only once when QQ or OpenClaw observes repeated delivery events.

## Best Possible Solution

Capture raw QQ gateway event IDs, QQ send request/response IDs, and `onMessageSent` timestamps for one duplicate, then add bounded idempotency at the QQBot boundary proven to re-enter delivery.

## Reproduction Assessment

No. Source shows a duplicated raw QQ event would be enqueued and processed again, but the live failure has not been reproduced and the revised logs do not prove one stable duplicate input path.

## Solution Assessment

Unclear. Queue-level `messageId` dedupe is narrow if QQ redelivers the same event, but sender-level idempotency or acknowledgement handling is safer if the repeat happens after outbound submission.
