## Summary

Keep open: current main still has a plausible QQBot duplicate-reply path, but the latest reporter logs leave the exact fix boundary unclear between same-messageId enqueue dedupe, documented self-echo suppression, and outbound idempotency.

## What This Changes

The issue asks OpenClaw to stop QQBot from sending the same assistant reply multiple times after one inbound QQ interaction.

## Best Possible Solution

Add a QQBot-owned idempotency guard at the earliest stable boundary, backed by a regression test; if live logs confirm distinct platform retries, extend the outbound send path with QQ-compatible idempotency instead of changing WebChat/history hooks.

## Reproduction Assessment

Unclear for the full live QQ symptom. Source inspection shows duplicate QQ events with the same peer/messageId would be enqueued and processed again, and current code contradicts the documented self-echo guard, but the reporter's later distinct-ID logs still need live confirmation.

## Solution Assessment

Unclear. Skipping message_sending for WebChat is not the right fix because WebChat history/list handlers are read-only; the likely fix belongs in QQBot queue/self-echo/outbound idempotency after checking QQ retry semantics.
