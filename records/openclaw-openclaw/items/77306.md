Keep open: current main disproves the original WebChat replay theory and has core same-message-id inbound dedupe, but the updated QQBot duplicate-send symptom still needs a QQ-owned idempotency or live trace decision before a safe fix can be chosen.

## What This Changes

The issue reports repeated QQBot replies for one user interaction and asks whether QQBot should dedupe stable inbound message IDs or add outbound idempotency handling.

## Best Possible Solution

Keep the WebChat paths unchanged, then add a QQBot-owned preflight replay guard or outbound idempotency fix only after a live trace confirms whether duplicates reuse stable message IDs or arrive as distinct QQ events for the same semantic send.
