## What This Changes

QQBot should avoid repeatedly sending the same assistant reply when WebChat history polling and a content-mutating `message_sending` hook are active.

## Best Possible Solution

Reproduce on current `main` with QQBot and a mutating `message_sending` hook, then fix the proven boundary: skip hooks for non-delivery replay if such a replay exists, or dedupe QQBot inbound events by stable message id before dispatch.

No, not yet. The issue provides concrete logs, but current source inspection shows `sessions.list` and `chat.history` are read-only paths that do not invoke `message_sending`, and no live QQBot reproduction was established.
