## Summary

Keep open: this is a useful implementation attempt for the paired-device naming request, but the patch is not merge-ready because the new rename RPC bypasses the existing per-device management authorization guard and stores operator labels in the reconnect-refreshable client `displayName` field.

## What This Changes

The PR adds `openclaw devices rename --device <id> --name <name>`, a `device.pair.rename` gateway RPC, protocol schemas/types, and regenerated Swift protocol models.

## Best Possible Solution

Rework the feature around a durable operator-owned paired-device label with explicit label-over-client-displayName-over-deviceId precedence, then apply the existing device-management authz/audit pattern and cover reconnect preservation plus cross-device denial in tests.

## Reproduction Assessment

Yes, at source level: current code refreshes `displayName` from reconnect metadata and sibling device-management methods enforce target-device authz, while the PR writes `displayName` from a new handler that does not receive `client` or `context`. I did not run a live gateway flow because this review is read-only.

## Solution Assessment

No. The CLI/RPC direction is plausible, but the implementation is not the best fix because it reuses client-owned metadata and omits the existing device-management authorization boundary; a durable operator label with sibling authz is the safer owner-boundary solution.
