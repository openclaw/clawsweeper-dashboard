- **[P2] Render operator labels in the Web UI too:** `src/cli/devices-cli.runtime.ts:761`
  - body: The PR only updates the CLI table to prefer `operatorLabel`, but the Control UI Devices card still types paired devices without `operatorLabel` and renders `normalizeOptionalString(device.displayName) || device.deviceId`. After a rename, the Web UI side of the linked feature still shows the stale device-reported name or raw id, so add the field to the UI device type/render path and cover it in `nodes.devices.test.ts`, or stop closing the broader Web UI issue.
  - confidence: 0.92
- **[P2] Preserve operator labels during reapproval:** `src/infra/device-pairing.ts:95`
  - body: Adding `operatorLabel` as persisted paired-device state is not enough because `buildApprovedPairedDevice` recreates the record on later scope/role reapproval and only preserves `createdAtMs`/last-seen fields from `existing`. A user can rename a device, approve a later scope upgrade, and silently lose that saved label; carry `params.existing?.operatorLabel` into the rebuilt record and add a lifecycle regression test.
  - confidence: 0.9
- **[P2] Sanitize the rename label before gateway logging:** `src/gateway/server-methods/devices.ts:685`
  - body: `displayName` is only validated as non-empty text, then `trimmed` is written directly into a gateway log message. A label containing newlines or terminal/control sequences can forge or corrupt log output; omit the label from this log or pass it through the same terminal/log sanitization used by the CLI.
  - confidence: 0.87
