## Summary

## What This Changes

The PR adds an `openclaw devices rename` CLI command, a `device.pair.rename` Gateway RPC, protocol validation/types, and Swift protocol bindings that write the requested name to paired-device `displayName`.

## Best Possible Solution

Revise this into an additive operator-owned paired-device label workflow with explicit label-over-client-displayName-over-deviceId precedence, existing device-management authorization, and real CLI/Gateway proof against a paired device.

## Reproduction Assessment

Yes, at source level: the proposed handler lacks `client`/`context` authorization checks and writes labels into `displayName`, while current reconnect handling refreshes that same field from client metadata. I did not run tests because this review is read-only.

## Solution Assessment

No. The maintainable fix is a separate operator-owned label field with explicit display precedence and the existing device-management authorization boundary, not a direct write into client-controlled `displayName`.
