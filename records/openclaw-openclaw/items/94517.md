## Summary

Keep open: the feature is useful and tied to the canonical paired-device naming issue, but this patch is not merge-ready because it stores user labels in reconnect-refreshable metadata and adds a paired-device mutation RPC without the existing self/admin authorization boundary. The previously suggested replacement PR is not live-accessible, so this PR should be reviewed and repaired on its own merits rather than closed as superseded.

## What This Changes

The PR adds a new `openclaw devices rename` CLI command, a `device.pair.rename` gateway RPC, protocol schema/type exports, and Swift protocol bindings for `DevicePairRenameParams`.

## Best Possible Solution

Implement a separate operator-owned paired-device label with explicit label-over-client-displayName display precedence, gate rename with the same self/admin authorization model used by existing paired-device mutations, and document the new CLI/RPC surface.

## Reproduction Assessment

Yes, at source level: current main lacks the requested rename path, and the PR's proposed storage/authz behavior can be checked directly against current reconnect and sibling mutation code.

## Solution Assessment

No. Reusing client `displayName` is the wrong durable label boundary, and the handler needs the existing paired-device management authorization checks before this can be a safe fix.
