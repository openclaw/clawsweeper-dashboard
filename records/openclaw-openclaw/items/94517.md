## Summary

Keep open: the feature is useful and targets the canonical paired-device naming issue, but this patch is not merge-ready because the new mutating RPC lacks the existing device self/admin authorization boundary and stores the operator label in reconnect-refreshable `displayName`.

## What This Changes

The PR adds `openclaw devices rename`, a `device.pair.rename` Gateway RPC/schema/Swift binding, and a handler that persists the requested name through `updatePairedDeviceMetadata({ displayName })`.

## Best Possible Solution

Add a durable operator-owned paired-device label with explicit label-over-client-displayName-over-deviceId precedence, reuse the existing device-management authorization boundary, update CLI/UI/protocol/docs/tests, and provide redacted real CLI/Gateway proof.

## Reproduction Assessment

Yes, at source level for the review findings: current code shows reconnect refresh writes `displayName`, and sibling device mutations enforce self/admin authorization that the new handler omits. I did not run a live paired-device rename flow in this read-only review.

## Solution Assessment

No. The maintainable fix is a separate operator-owned label field plus existing device-management authz; writing into refreshable `displayName` is only a plausible shortcut and breaks the requested durability boundary.
