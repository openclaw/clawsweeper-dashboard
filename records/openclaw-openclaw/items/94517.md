## Summary

Keep open: the PR targets the right paired-device naming problem, but the current patch is not merge-ready because the new mutation RPC bypasses the sibling device-management authorization boundary, stores the user label in reconnect-refreshable metadata, and lacks real rename/list proof against an actual paired device.

## What This Changes

Adds an `openclaw devices rename` CLI command, a `device.pair.rename` Gateway RPC, gateway-protocol schema/type exports, and a generated Swift protocol model.

## Best Possible Solution

Revise this into a durable operator-owned paired-device label workflow with label-over-client-displayName-over-deviceId precedence, existing self/admin authorization checks, focused coverage, and real CLI/Gateway proof.

## Reproduction Assessment

Yes, at source level: the proposed handler lacks `client`/`context` authorization checks and writes the label into a metadata field that current reconnect paths refresh from client input. I did not run tests because this review is read-only.

## Solution Assessment

No. The maintainable solution is a separate operator-owned label field with explicit display precedence and the existing device-management authorization boundary, not a direct write into client-controlled `displayName`.
