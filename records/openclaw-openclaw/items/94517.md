## Summary

## What This Changes

The PR adds `openclaw devices rename`, a `device.pair.rename` Gateway RPC, protocol schema/type exports, and Swift protocol binding support for paired-device display names.

## Best Possible Solution

Revise this into an additive operator-owned paired-device label workflow, separate from client `displayName`, protected by existing device-management authorization and proven through a real CLI/Gateway paired-device run.

## Reproduction Assessment

Yes, at source level: the changed handler lacks `client`/`context` authorization checks and writes the label into `displayName`, while current reconnect paths refresh that same field from client metadata. I did not run tests because this review is read-only.

## Solution Assessment

No. The maintainable fix is a separate operator-owned label field with explicit display precedence and the existing device-management authorization boundary, not a direct write into client-controlled `displayName`.
