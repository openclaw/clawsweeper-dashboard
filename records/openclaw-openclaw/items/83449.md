## Summary

Keep open: this is a recent, concrete regression report with repeated MiniMax per-message auth timings, and current main still runs the generic embedded-run auth bootstrap and runtime-auth preparation on each non-plugin-harness run; the related Feishu timeout cluster does not prove this MiniMax auth-specific delay is fixed.

## What This Changes

The issue reports that OpenClaw 5.x spends 30-50s in the MiniMax auth stage on every Feishu/Lark message on Windows, regressing from sub-second auth in 4.23.

## Best Possible Solution

Profile MiniMax auth startup on current main and v2026.5.12, then hoist or cache valid provider auth/runtime resolution across embedded runs through the provider/auth seams so refresh only happens on expiry or auth failure.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection maps the reported auth stage to initializeAuthProfile and prepareRuntimeAuthForModel, but the 30-50s delay needs the reported Windows + MiniMax CN + Feishu/Lark setup to reproduce directly.

## Solution Assessment

Unclear. A narrow auth/runtime hoisting or cache fix is plausible, but maintainers need profiling first to distinguish profile lookup, provider runtime lookup, credential formatting, and external MiniMax network latency.
