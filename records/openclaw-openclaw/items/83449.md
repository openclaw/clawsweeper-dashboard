## Summary

Keep open: this is a recent, concrete regression report with repeated per-message MiniMax auth-stage timings, and current main still runs generic embedded-run auth bootstrap for MiniMax on each run; the broader Feishu timeout cluster does not by itself prove this auth-specific delay is fixed.

## What This Changes

The issue requests fixing a 5.x regression where MiniMax auth initialization takes 30-50 seconds on every Feishu/Lark message instead of staying under one second as in 4.23.

## Best Possible Solution

Profile the MiniMax auth startup stage on current main and v2026.5.12, then hoist or cache valid MiniMax provider auth/runtime resolution across embedded runs so refresh only happens on expiry or an auth failure.

## Reproduction Assessment

No high-confidence current-main reproduction was established. The reporter supplied concrete 5.7/5.12 logs and current source maps the auth stage to initializeAuthProfile, but reproducing the 30-50s delay needs the reported Windows + MiniMax CN + Feishu/Lark setup.

## Solution Assessment

Unclear. A narrow auth/runtime caching or hoisting fix is plausible, but the current evidence does not yet isolate whether MiniMax profile lookup, provider runtime lookup, credential formatting, or external network behavior dominates the auth stage.
