- `impact:auth-provider`: The central reported bottleneck is MiniMax provider auth resolution for both OAuth and API-key modes.

## Summary

Keep open: this is a same-day, concrete regression report with repeated MiniMax auth-stage timings and an attached log/config snapshot; current main still routes MiniMax requests through the embedded-run auth bootstrap on each run, and the related Feishu timeout cluster does not prove this auth-specific delay is fixed.

## What This Changes

The issue asks OpenClaw to stop spending 30-50 seconds in the MiniMax auth stage on each Feishu/Lark message after upgrading from 4.23 to 5.x.

## Best Possible Solution

Profile MiniMax auth startup on current main and v2026.5.12 in the reported Windows Feishu/Lark setup, then hoist or cache valid MiniMax auth/runtime resolution across embedded runs only after the auth hotspot is proven.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. The reporter supplied concrete logs and the auth-stage path maps to current source, but reproducing the delay requires the reported Windows 11 + MiniMax CN + Feishu/Lark websocket setup.

## Solution Assessment

Unclear. Auth/runtime caching or hoisting is plausible, but the current evidence must first isolate whether the delay comes from auth profile lookup, external CLI/OAuth refresh handling, provider runtime lookup, Feishu startup pressure, or external network behavior.
