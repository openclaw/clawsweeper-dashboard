## Summary

Keep open. The PR fixes the real SecretRef auth gap on the macOS browser-proxy path, but the new keychain subprocess still has no timeout and can hang endpoint resolution instead of degrading to nil/fallback.

## What This Changes

The PR adds a macOS GatewayEndpointStore SecretRef resolver for local gateway auth token/password values, threads it through localConfig, and adds focused Swift tests.

## Best Possible Solution

Land the SecretInputResolver approach after bounding the keychain process, preserving the local-only SecretRef behavior and adding the required changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: provide gateway.auth.token or gateway.auth.password as a dictionary and the String casts in local auth resolution return nil before MacNodeBrowserProxy builds its auth headers.

## Solution Assessment

Not yet. The resolver seam, source/provider guard, and tests are the right direction, but the subprocess needs a deadline/termination path before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Bound the keychain lookup time:** `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:146`
  - body: `securityFindGenericPassword` waits on `/usr/bin/security` without any deadline. Because the live resolver and the sync `localConfig()` browser-proxy path depend on this runner, a locked or stalled keychain command can hang endpoint resolution instead of returning nil/falling back; add a timeout that terminates the process and returns nil.
  - confidence: 0.84
