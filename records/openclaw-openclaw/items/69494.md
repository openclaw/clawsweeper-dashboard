## Summary

Keep open: this is a real source-reproducible macOS browser-proxy auth bug and the PR is useful, but the current patch still leaves an unbounded keychain subprocess wait and lacks contributor real behavior proof.

## What This Changes

The PR changes macOS GatewayEndpointStore to resolve local gateway auth token/password SecretRef dictionaries through a keychain-backed resolver and adds Swift tests for helper and localConfig paths.

## Best Possible Solution

Land a bounded, tested macOS app SecretRef resolver for local gateway auth, keeping remote auth semantics unchanged unless a separate design expands that surface.

## Reproduction Assessment

Yes, by source inspection. Current main casts local gateway.auth.token and gateway.auth.password to strings while MacNodeBrowserProxy consumes localConfig, so structured SecretRefs resolve to nil before auth headers are built.

## Solution Assessment

No, not yet. The resolver seam and tests match the narrow fix direction, but the keychain subprocess needs a timeout/termination path and the contributor still needs redacted after-fix runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound the keychain subprocess wait:** `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:146`
  - body: securityFindGenericPassword launches /usr/bin/security and waits with no timeout. If the keychain helper stalls or the keychain is locked, both the async resolver awaiting it and the synchronous localConfig browser-proxy path can hang instead of falling back to nil, so add a bounded timeout that terminates the process and returns nil.
  - confidence: 0.84
