## Summary

Keep open: current main still has the reported macOS SecretRef auth gap, and the PR now addresses the central token/password path, but it still needs a bounded keychain subprocess wait plus acceptable real behavior proof before merge.

## What This Changes

The PR adds a macOS GatewayEndpointStore SecretInputResolver for local keychain-backed gateway auth SecretRefs and adds Swift tests for helper and localConfig paths.

## Best Possible Solution

Land a bounded, tested app-side SecretRef resolver for local macOS gateway auth after contributor supplies redacted real behavior proof, while keeping remote auth semantics string-only unless separately designed.

## Reproduction Assessment

Yes, by source inspection. Current main casts local `gateway.auth.token` and `gateway.auth.password` as strings, while the browser proxy consumes `GatewayEndpointStore.localConfig()`, so a SecretRef dictionary resolves to nil on that path.

## Solution Assessment

No, not yet. The resolver seam and tests are the right narrow direction, but the keychain subprocess should have a timeout/termination path and the contributor still needs to show after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Bound the keychain subprocess wait:** `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:146`
  - body: `securityFindGenericPassword` waits indefinitely for `/usr/bin/security`. If the keychain is locked or the helper hangs, both the async resolver and sync `localConfig()` browser-proxy path never reach the intended nil fallback, so gateway endpoint/browser auth resolution can stall instead of failing closed. Add a timeout that terminates the process and returns nil.
  - confidence: 0.84
