## Summary

Keep this PR open: the underlying macOS browser-proxy SecretRef auth bug is valid and current main still has the string-only auth path, but the branch still needs redacted real runtime proof and a bounded keychain subprocess before merge.

## What This Changes

The PR adds a macOS `SecretInputResolver` in `GatewayEndpointStore` so local `gateway.auth.token` and `gateway.auth.password` SecretRef dictionaries can resolve through `/usr/bin/security` for desktop browser proxy auth.

## Best Possible Solution

Land a bounded, tested app-side SecretRef resolver for local macOS gateway auth, with remote auth semantics kept string-only unless a separate design expands that surface.

## Reproduction Assessment

Yes, by source inspection. Current main casts local gateway auth token/password values as strings while the browser proxy consumes `GatewayEndpointStore.localConfig()`, so a SecretRef dictionary resolves to nil before auth headers are built.

## Solution Assessment

No, not yet. The resolver seam and focused tests match the narrow maintainable fix shape, but the keychain subprocess needs a timeout/termination path and the contributor still needs redacted after-fix runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Bound the keychain subprocess wait:** `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:146`
  - body: `securityFindGenericPassword` waits indefinitely for `/usr/bin/security`. If the keychain is locked or the helper stalls, both the async resolver and sync `localConfig()` browser-proxy path can hang instead of falling back to nil, so add a timeout that terminates the process and returns nil.
  - confidence: 0.84
