## What This Changes

This PR changes macOS GatewayEndpointStore so local gateway token/password values can be resolved from keychain-backed SecretRef dictionaries, and adds Swift tests for the resolver and localConfig paths.

## Best Possible Solution

Rebase onto current main, keep the focused local SecretRef resolver, bound the keychain lookup, make unresolved parsed SecretRefs fail closed or explicitly document/test any launchd compatibility fallback, and require redacted real macOS browser-proxy proof before merge.

## Reproduction Assessment

Yes from source inspection: current main's macOS localConfig path casts local gateway.auth.token/password to String, so a SecretRef dictionary cannot become a browser-proxy auth header. I did not run a live macOS/keychain reproduction in this read-only review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Bound the keychain subprocess before browser proxy auth:** `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:146`
  - body: `securityFindGenericPassword` still waits forever for `/usr/bin/security`. Because `MacNodeBrowserProxy.defaultEndpoint()` calls the sync `GatewayEndpointStore.localConfig()` path, a locked or stuck keychain command can block every browser-proxy request before the normal request timeout can apply; add a timeout/termination path or move this endpoint resolution off the sync request path.
  - confidence: 0.88
- **[P1] Stop launchd fallback after unresolved SecretRefs:** `apps/macos/Sources/OpenClaw/GatewayEndpointStore.swift:314-318`
  - body: When a parsed keychain SecretRef returns nil, this password path continues into the launchd fallback, and the token sync path repeats the same pattern below. That can silently authenticate with a stale launchd credential even though the user explicitly configured gateway.auth.* as a SecretRef; track parsed-unresolved separately and fail closed or make this fallback an explicit tested compatibility decision.
  - confidence: 0.8
