## Summary

Current main already implements the Telegram control-lane behavior requested here: read-only status commands and targeted `/stop@bot` updates are routed away from the blocked per-topic lane, including isolated ingress spool replay. The full fix is current-main-only for the targeted stop case; the reported v2026.5.12 image predates that merged PR.

## What This Changes

The issue asks for Telegram `/stop` and `/status` recovery commands to bypass blocked per-topic FIFO processing so they can inspect or cancel stuck turns.

## Best Possible Solution

Keep the current main control-lane implementation and ship it in the next release while leaving broader stuck-session/runtime recovery work to the existing canonical reports.

## Reproduction Assessment

Yes, source-level: current regression tests model both a busy Telegram topic lane and spooled `/status` plus `/stop@bot` updates bypassing that lane. I did not run tests because this was a read-only review.

## Solution Assessment

Yes. Handling the recovery commands in Telegram lane selection before normal dispatch is the narrowest maintainable fix because it preserves FIFO ordering for ordinary topic messages while letting recovery commands reach existing status/abort handling.
