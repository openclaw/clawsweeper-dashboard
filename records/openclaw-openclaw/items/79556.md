## Summary

Keep open: the group-id part is not correct because the added `GroupId` payload field is not part of the consumed message context, and current main still resolves group policy from `resolveGroupSessionKey(ctx)`. The PR also lacks real Feishu behavior proof, while the overlapping group-context fix at https://github.com/openclaw/openclaw/pull/79483 is the better path for that half of the bug.

## What This Changes

The PR adds Feishu inbound `CommandSource: "text"` for slash-looking messages, adds a group `GroupId`, and updates two Feishu context assertions.

## Best Possible Solution

Keep the useful `CommandSource` direction only if focused tests and real Feishu proof support it, and fix group identity through the canonical resolver/session path using the trusted chat target, likely coordinated with https://github.com/openclaw/openclaw/pull/79483.

## Reproduction Assessment

Yes at source level for the group-id bug: current Feishu group turns keep `From` shaped as the sender while shared group resolution and tool policy derive group identity from that resolver output. The slash-command report has runtime logs, but I did not establish a live Feishu workspace reproduction.

## Solution Assessment

No. Setting `CommandSource` may be part of the slash-command fix, but adding `GroupId` to the Feishu payload is not the best group-id fix because current main does not consume that field in the resolver or tool-policy path.
