## Summary

Keep open. Current main only partially mitigates the reported Telegram startup slow-path: it limits startup probes to two slots, but `startAccount` still waits on `getMe` with a 15s startup budget and the probe still retries within that budget, so the slow-api/event-loop report is not already fixed or safely duplicate.

## What This Changes

Harden Telegram multi-account startup so slow or intermittent `api.telegram.org/getMe` calls do not stall the Gateway event loop or block other sessions.

## Best Possible Solution

Implement a Telegram-owned startup slow-path that bounds transient `getMe` latency per account, preserves 401 invalid-token detection, and leaves broad Gateway scheduling work to https://github.com/openclaw/openclaw/issues/78861.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows current main still executes startup `getMe` with a 15s budget behind a two-slot limiter, and the issue provides detailed logs, but reproducing the event-loop starvation requires a Windows/multi-bot slow Telegram setup.

## Solution Assessment

Unclear on the exact timeout and backoff values, but keeping this open is the right triage outcome. The existing two-slot limiter is only partial; a narrow Telegram startup slow-path is preferable to closing this as implemented or folding it into a broad Gateway architecture issue.
