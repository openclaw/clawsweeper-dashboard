## Summary

Keep open: current main and the latest release still emit `plugins.metadata.scan` spans for plugin metadata memo hits. The open closing PR targets the exact behavior, but it has not merged and is currently dirty/conflicting, while the related concurrency PR explicitly leaves cache-hit span noise as separate work.

## What This Changes

The issue reports that plugin metadata cache hits still create full `plugins.metadata.scan` diagnostics timeline spans and flood hot-path diagnostics.

## Best Possible Solution

Rebase and land the linked narrow fix, or an equivalent patch, so cache misses keep real scan spans while memo hits return the cached snapshot without timeline span creation.

## Reproduction Assessment

Yes. Source inspection shows repeated `loadPluginMetadataSnapshot` calls can hit the process memo, and current `main` still wraps memo hits in `measureDiagnosticsTimelineSpanSync("plugins.metadata.scan", ...)`; I did not run a live gateway profile in this read-only review.

## Solution Assessment

Yes. The best fix is narrow: return the memoized snapshot directly on cache hits while preserving `plugins.metadata.scan` instrumentation for real scan misses and adding a timeline-count regression test.
