## Summary

Keep open: the exact macOS EPERM propagation is not high-confidence reproduced on current `main` because the central device-auth store now catches read errors, but explicit-token diagnostic paths still reach cached device-auth lookup and merit a narrow robustness fix with regression coverage.

## What This Changes

The issue reports that macOS CLI diagnostics fail on `device-auth.json` access even when the gateway is healthy and explicit token auth is supplied.

## Best Possible Solution

Harden stored-device-auth lookup so diagnostics treat unreadable or throwing cached auth as absent while preserving documented cached-auth reuse and trusted retry behavior.

## Reproduction Assessment

Unclear for the exact EPERM failure: current source shows explicit-token diagnostics still enter the cached device-auth lookup, but the central store helper catches ordinary read errors. A regression test should simulate `loadDeviceAuthToken()` throwing to cover the reported failure mode.

## Solution Assessment

Yes for a narrow robustness fix: catch stored-auth load failures at the GatewayClient/probe boundary and continue with explicit token/password auth. Do not remove cached device-auth reuse wholesale because docs and tests expect it for local read-scope diagnostics.
