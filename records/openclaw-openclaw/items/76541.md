## Summary

Close as implemented: current `main` and shipped `v2026.5.2` no longer propagate unreadable `device-auth.json` as a probe/status failure, and status JSON now treats post-connect diagnostic failures as degraded reachability instead of an unreachable gateway.

## What This Changes

The issue asks for macOS local CLI diagnostics to avoid failing when `device-auth.json` cannot be opened and to keep a healthy loopback Gateway reported as reachable.

## Best Possible Solution

Keep the shipped `v2026.5.2` behavior; reporters still seeing `EPERM` should retest on `v2026.5.2` or newer and file a fresh report with current `gateway probe --json` and `status --json` output.

## Reproduction Assessment

No current-main failing reproduction is available from source inspection: the reported old-version path is clear, but current source treats the unreadable cache as absent and falls back to token/password auth.

## Solution Assessment

Yes. The shipped implementation is the narrow maintainable fix: keep cached paired auth when usable, but make diagnostics tolerate restricted device-auth reads and report post-connect detail failures as degraded reachability.
