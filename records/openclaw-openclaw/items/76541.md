## Summary

Current main already prevents the reported EPERM from surfacing through the CLI diagnostic path: probes treat unreadable cached device auth as absent, skip device identity when appropriate, and keep explicit token/password probe paths non-mutating.

## What This Changes

The issue asks for macOS CLI gateway diagnostics to stop failing with `EPERM` on `identity/device-auth.json` when the gateway is healthy or explicit token auth is provided.

## Best Possible Solution

Keep the current main behavior: diagnostic probes should treat inaccessible cached device auth as absent, preserve explicit token/password auth, and avoid mutating first-time device identity state.

## Reproduction Assessment

No. The reported current-main failure path is not reproducible from source inspection because device-auth read failures are caught and `probeGateway` passes no device identity when cached auth cannot be safely reused.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it changes the diagnostic auth fallback behavior without adding a new option, widening auth scope, or changing gateway runtime behavior.
