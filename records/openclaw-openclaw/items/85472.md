- `merge-risk: 🚨 session-state`: The diff can duplicate or mis-associate persisted WebChat user turns in session JSONL and chat.history.

## Summary

Keep this PR open but do not merge as-is: the WebChat persistence bug is real and the latest body now includes useful live-output proof, but the patch still puts a disk write inside the shared transcript-update notifier, which can collide with SessionManager-owned agent turns and bypass transcript write hooks.

## What This Changes

The branch appends WebChat user messages to the session JSONL transcript before broadcasting the existing transcript update event and adds fixture-level regression tests for parentId chaining.

## Best Possible Solution

Make WebChat user persistence ownership-aware: only the no-SessionManager path should write from the gateway, and any gateway-owned user write must preserve hook, provenance, redaction, idempotency, and parentId semantics.

## Reproduction Assessment

Yes for a source-level path: current main's no-agent WebChat branch emits the user transcript update while only the assistant path appends to JSONL, and the PR-introduced duplicate/hook risks are clear from the surrounding SessionManager ownership code. I did not run a live WebChat flow in this read-only review.

## Solution Assessment

No. The patch fixes the missing durable user row by writing inside a shared notifier, but the safer solution is a runtime-owner-aware persistence path that keeps a single transcript writer and preserves SessionManager guard semantics.
