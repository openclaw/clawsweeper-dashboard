## Summary

Keep open. The narrowed PR removed the unrelated CLI logging and uses the parentId-aware append helper, but the write is still placed in the shared live-update helper where it overlaps with SessionManager-owned agent persistence and bypasses the guarded transcript-write hook contract; the supplied proof is still fixture/test-only rather than a real WebChat reconnect/history run.

## What This Changes

The PR persists WebChat user turns to session JSONL by calling the existing parentId-aware transcript append helper before broadcasting the transcript update, and adds focused transcript persistence tests.

## Best Possible Solution

Persist user turns only on paths that do not already have a SessionManager owner, and make any gateway-owned transcript write honor the same hook, provenance, redaction, and single-writer contract as the existing guarded SessionManager path.

## Reproduction Assessment

Yes for a source-level path: current main has a no-agent WebChat branch where the gateway emits the user transcript update and appends the gateway assistant entry without an obvious user-row disk writer. I did not run a live WebChat reconnect proof, and the PR-introduced duplicate/hook-bypass issues are source-provable from the changed append site and SessionManager contract.

## Solution Assessment

No. The parentId append helper is the right primitive for gateway-owned writes, but using it inside the shared live-update helper is not the best fix because agent-backed sends already have a guarded SessionManager writer.
