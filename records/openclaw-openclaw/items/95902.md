## Summary

Keep open: this is a focused gateway status diagnostic fix, but the current patch is overbroad in local-log scoping, can duplicate the same error in an existing status path, and still lacks after-fix real behavior proof from a real gateway status run.

## What This Changes

The PR widens gateway status gathering to read the last gateway log error when an RPC probe fails, prints that error beside failed probe output, and adds ENOSPC log matching plus focused tests.

## Best Possible Solution

Scope the log read/rendering to local managed RPC failures where the listener is relevant, keep Last gateway error rendered once, and add redacted after-fix terminal output for the real `openclaw gateway status --deep` path.

## Reproduction Assessment

Yes, source-level: current main only gathers `lastError` for a running service when the port is not busy, so a busy listener with `rpc.ok=false` cannot show the log error. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The layer is plausible, but the implementation should be narrowed to local managed probe failures and avoid duplicate rendering before it is the best fix.

Full review comments:

- **[P2] Keep local log errors scoped to local probes:** `src/cli/daemon-cli/status.gather.ts:711`
  - body: The widened condition now reads the local managed service logs whenever `rpc.ok === false`. For `--url` or remote-mode probes, that failure can be for a different gateway target while the local service is merely running, so the output can attach a stale local `Last gateway error` to an auth/scope/remote failure. Gate this to local managed probe failures, consistent with the existing `probeUrlOverride` checks around other local-only diagnostics.
  - confidence: 0.86
- **[P3] Avoid printing the last gateway error twice:** `src/cli/daemon-cli/status.print.ts:277-279`
  - body: In the existing service-running/port-not-listening case, `lastError` is already set and the existing later branch prints `Last gateway error:` with log paths. The new RPC-failure print runs first for the same failed probe, so that path will emit the same last error twice. Guard the new print to the busy-listener RPC-failure case or move the rendering to a single shared location.
  - confidence: 0.88
