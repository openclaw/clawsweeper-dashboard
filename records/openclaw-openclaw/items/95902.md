## Summary

Keep this PR open: the underlying diagnostic bug is valid, but the current patch can misattribute local gateway logs to explicit URL probes and can print the same last-error line twice in the existing not-listening path. The PR also lacks after-fix real behavior proof; the body has before-fix field evidence and tests only.

## What This Changes

The PR changes gateway status diagnostics to read and print the latest gateway error when a running service fails the RPC status probe, and prioritizes ENOSPC log lines.

## Best Possible Solution

Land a narrow gateway-status diagnostic fix that reads local gateway logs only for local managed probes, prints `Last gateway error` once, covers explicit-url and not-listening cases, and includes redacted after-fix terminal proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection shows current main only reads the last gateway error for a running service when the daemon port is not busy, matching the reported missing diagnostic for a busy port with a failed RPC probe.

## Solution Assessment

No, not as written. Broadening the last-error read for failed local RPC probes is the right layer, but the fix should guard explicit URL probes and avoid duplicating the existing not-listening output path.
