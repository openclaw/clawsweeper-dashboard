## Summary

Keep open: current main and v2026.6.6 still assign fixed daily and session ingestion scores before averaging them into light-sleep and REM confidence, so the source-level reproduction remains valid. The repair needs maintainer scoring-policy approval because changing the heuristic affects memory promotion, pruning, and existing persisted scores.

## What This Changes

The issue asks for daily-memory and session-corpus ingestion scores to become content-dependent instead of fixed at 0.62 and 0.58.

## Best Possible Solution

Define an owner-approved content-aware ingestion scoring policy, decide the persisted-entry backfill stance, and update dreaming-phase tests to require a meaningful score distribution.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.6 still assign fixed 0.62 and 0.58 ingestion scores and then average signal scores, so ingestion-only entries cannot differentiate by content. I did not run a live multi-week workspace replay.

## Solution Assessment

Unclear: replacing the constants is likely the right direction, but the best formula and migration/backfill policy need maintainer product judgment because they change memory ranking behavior.
