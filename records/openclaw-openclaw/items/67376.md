## Summary

Keep open. The linked security issue still has remaining full-scope loopback work, and this PR is a plausible implementation path, but the current branch is conflicting, lacks real behavior proof, and introduces blocking token-lifecycle and channel-scope issues.

## What This Changes

The PR replaces MCP loopback header-derived scope with a per-backend token registry, removes generated scope env/headers, invalidates scoped tool-cache entries, updates tests, and changes bundled plugin public-surface loader fallback behavior.

## Best Possible Solution

Land a rebased, security-reviewed MCP loopback fix that derives the full session/account/channel/owner tuple from server-side token identity, removes emitted and accepted scope headers, fixes token lifecycle, and keeps unrelated loader behavior separate unless explicitly approved.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a valid loopback bearer can still vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values flow into McpLoopbackToolCache and resolveGatewayScopedTools.

## Solution Assessment

No, not as currently proposed. Token-bound full-tuple scope is the right direction, but this branch needs a rebase, lifecycle fixes, actual messageChannel binding, removal or explicit review of the loader fallback, and contributor real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: clearActiveMcpLoopbackRuntime() clears the active runtime and token table unconditionally. Since closeMcpLoopbackServer() drops activeMcpLoopbackServer before awaiting close, an older close callback can wipe tokens for a freshly started server and make valid new clients receive 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:124-126`
  - body: The new token revocation runs after awaiting context.preparedBackend.cleanup. If cleanup throws, unregisterMcpLoopbackToken is skipped and the bearer remains registered after the backend run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: prepareCliRunContext registers the loopback token before later async setup work, but the catch only covers prepareCliBundleMcpConfig. A later setup error leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into token scope:** `src/agents/cli-runner/prepare.ts:152`
  - body: The registered token scope uses params.messageProvider, while current main passes params.messageChannel ?? params.messageProvider to bundled MCP clients. Runs with a distinct messageChannel would resolve tools under the wrong channel after this change.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch hardens a real MCP loopback auth boundary, but its token lifecycle, server-clear behavior, channel binding, and unrelated loader fallback need security review before merge.

Concerns:

- **[medium] Bearer tokens can outlive backend lifecycle:** `src/agents/cli-runner.ts:124`
  - body: The token registry can retain authorization material when setup fails after registration or when backend cleanup throws before token revocation runs.
  - confidence: 0.86
- **[medium] Old server shutdown can revoke fresh tokens:** `src/gateway/mcp-http.loopback-runtime.ts:30`
  - body: The unconditional runtime clear can erase a newer server's token table when an older server close callback runs later, making authorization state depend on shutdown timing.
  - confidence: 0.9
- **[medium] Token scope can use the wrong channel:** `src/agents/cli-runner/prepare.ts:152`
  - body: Binding messageProvider instead of the actual messageChannel means the server-side token scope may not match the channel-specific scope current main passes to bundled MCP clients.
  - confidence: 0.82
- **[low] Plugin loader fallback changes code-loading behavior:** `src/plugins/public-surface-loader.ts:115`
  - body: The PR changes bundled plugin public-surface loading to fall back to jiti for selected source-artifact resolution errors; that is unrelated to the MCP token fix and should be split or explicitly reviewed.
  - confidence: 0.78
