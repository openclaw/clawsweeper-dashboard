## Summary

Keep this PR open. Current main only shipped the narrower owner-vs-non-owner bearer mitigation; it still emits and trusts mutable session/account/channel scope headers, so this PR’s central token-bound scope work is not obsolete. The PR is not merge-ready as-is because the proposed token registry has lifecycle/revocation bugs and includes an unrelated plugin public-surface loader fallback in a security-sensitive patch.

## What This Changes

This PR binds MCP loopback session/account/channel/owner scope to per-backend registered bearer tokens, removes the generated scope headers/env vars, adds cache invalidation and regression tests, and also changes bundled plugin public-surface loader fallback behavior.

## Best Possible Solution

Preserve the shipped owner/non-owner bearer mitigation, then land a narrow MCP loopback security patch that binds the full scope tuple to per-backend bearer tokens, stops emitting/reading scope headers and env vars, invalidates cache entries on revocation, handles runtime shutdown and cleanup failures safely, and keeps unrelated plugin loader changes out of this fix unless they are separately reviewed.

## Reproduction Assessment

Yes. The current-main source and tests provide a high-confidence reproduction path: a valid non-owner loopback bearer can still send `x-session-key`, `x-openclaw-account-id`, and `x-openclaw-message-channel`, and those values flow into `resolveGatewayScopedTools`; the original issue supplies the raw HTTP shape for exercising the same path.

## Solution Assessment

No, not as proposed. Binding scope to the token is the right direction, but this patch needs lifecycle fixes and should split or justify the plugin loader fallback before it is the narrowest maintainable solution for the security issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` clears the global runtime and token table unconditionally. Because `closeMcpLoopbackServer()` can drop the active server before the old `close()` callback runs, a concurrent `ensureMcpLoopbackServer()` can create fresh tokens that the old close then wipes, causing valid new backends to receive 401s.
  - confidence: 0.9
- **[P2] Revoke the token even when cleanup fails:** `src/agents/cli-runner.ts:381`
  - body: `runPreparedCliAgent` unregisters the loopback token after awaiting `context.preparedBackend.cleanup?.()`. If backend cleanup throws, revocation is skipped and the bearer token remains registered after the run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:148`
  - body: `prepareCliRunContext` registers a token before later async setup work. The narrow cleanup only covers the MCP config preparation path, so a later failure such as docs-path resolution leaves a valid token in the registry even though no run context is returned.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch is security-hardening in intent, but it introduces token lifecycle risks and mixes in an unrelated plugin code-loading change that needs security review before merge.

Concerns:

- **[medium] Bearer tokens can survive failed cleanup paths:** `src/agents/cli-runner.ts:381`
  - body: The proposed registry can retain tokens when backend cleanup or later prepare steps fail. That keeps authorization material live longer than the backend lifecycle and grows the per-request token scan set.
  - confidence: 0.86
- **[medium] Runtime shutdown can revoke unrelated fresh tokens:** `src/gateway/mcp-http.loopback-runtime.ts:30`
  - body: The unconditional global clear in the proposed runtime can erase a newer server's token table when an older server close completes later, turning valid fresh MCP clients into unauthorized callers.
  - confidence: 0.9
- **[low] Unrelated plugin loader fallback changes code execution behavior:** `src/plugins/public-surface-loader.ts:116`
  - body: The PR also changes `public-surface-loader` to fall back from native source require to Jiti for selected module-resolution errors. That is a plugin code-loading path unrelated to the MCP token fix and should be split or separately reviewed and tested.
  - confidence: 0.78
