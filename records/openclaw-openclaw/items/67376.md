## Summary

Keep open. Current main still accepts session, account, and message-channel scope from mutable MCP loopback headers, but this PR needs a careful security rebase because it is conflicting, has unresolved token lifecycle bugs, includes an unrelated plugin loader fallback, and lacks after-fix real behavior proof.

## What This Changes

The PR binds MCP loopback scope to per-backend bearer tokens, removes generated scope headers/env fields, invalidates scoped cache entries on token revocation, updates tests/changelog, and adds a bundled plugin public-surface loader fallback.

## Best Possible Solution

Land a reviewed, rebased MCP loopback security patch that binds the full session/account/channel/owner scope tuple to token identity, handles token revocation robustly, removes emitted mutable scope headers, and keeps plugin-loader changes separate.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a valid loopback bearer can still vary `x-session-key`, `x-openclaw-account-id`, or `x-openclaw-message-channel`, and those values reach `resolveGatewayScopedTools`. I did not run a live exploit in this read-only review.

## Solution Assessment

No, not as proposed. Token-bound scope is the right direction, but this branch needs a rebase, lifecycle fixes, `messageChannel` handling, removal or separate review of the plugin loader fallback, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` clears the active runtime and the full registered-token table without checking that the closing server is still the active server. A concurrent restart can register fresh tokens and then have them wiped by the old close callback, causing valid clients to receive 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:122-126`
  - body: The new revocation runs after `await context.preparedBackend.cleanup?.()`. If backend cleanup throws, `unregisterMcpLoopbackToken` is skipped and the bearer remains registered after the run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:145-153`
  - body: `prepareCliRunContext` registers the loopback token before later async setup work, but only catches failures from `prepareCliBundleMcpConfig`. Any later setup error leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into token scope:** `src/agents/cli-runner/prepare.ts:151`
  - body: The token scope records `params.messageProvider`, but run params can carry a distinct `messageChannel` and current main already prefers `messageChannel ?? messageProvider` for loopback headers. Runs where the backend provider differs from the source channel would resolve tools under the wrong channel.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR hardens a real auth boundary, but the proposed token registry lifecycle and unrelated plugin loader fallback need security review before merge.

Concerns:

- **[medium] Bearer tokens can outlive backend lifecycle:** `src/agents/cli-runner.ts:122`
  - body: The token registry can retain authorization material when setup fails after registration or when backend cleanup throws before token revocation runs.
  - confidence: 0.86
- **[medium] Old server shutdown can revoke fresh tokens:** `src/gateway/mcp-http.loopback-runtime.ts:30`
  - body: The unconditional runtime clear can erase a newer server's token table when an older server close callback runs later, making authorization state depend on shutdown timing.
  - confidence: 0.9
- **[low] Plugin loader fallback changes code-loading behavior:** `src/plugins/public-surface-loader.ts:112`
  - body: The PR changes bundled plugin public-surface loading to fall back to the module loader for selected resolution errors, which is unrelated to the MCP token fix and should be split or separately reviewed.
  - confidence: 0.78
