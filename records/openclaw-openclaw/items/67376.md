## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Guard runtime clearing by the closed server:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `closeMcpLoopbackServer()` clears `activeMcpLoopbackServer` before awaiting close, so a concurrent ensure can start a new loopback server and then the old close callback clears the new server's runtime and registered tokens. Keep the owner-token guard or otherwise bind clearing to the server instance being closed.
  - confidence: 0.93
- **[P2] Revoke the token even when backend cleanup fails:** `src/agents/cli-runner.ts:124-126`
  - body: The new revocation runs after `await context.preparedBackend.cleanup?.()`, so a cleanup exception skips `unregisterMcpLoopbackToken` and leaves the bearer registered past run teardown. Put revocation in its own finally path that cannot be bypassed by cleanup failures.
  - confidence: 0.9
- **[P2] Revoke registered tokens on all prepare failures:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: The token is registered before later preparation work, but only `prepareCliBundleMcpConfig` failures revoke it. If later preparation throws before the context is returned, the bearer remains in the process token map without a backend cleanup owner.
  - confidence: 0.88
