## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Bind every current loopback context field to the token:** `src/gateway/mcp-http.loopback-runtime.ts:7-12`
  - body: The new `McpLoopbackScope` only records session, account, message provider, and owner status, but current main also carries channel/thread/message/audio, inbound-event, and reply-delivery context through the same loopback headers. A rebased fix that leaves those fields header-controlled would preserve a spoofable tool context under a valid bearer.
  - confidence: 0.91
- **[P1] Guard runtime clearing by the server instance:** `src/gateway/mcp-http.ts:117`
  - body: `clearActiveMcpLoopbackRuntime()` now wipes the global runtime and registered token table from any close callback. If an old loopback server is still closing while a new one starts, the old close can clear fresh tokens and make valid backends start receiving 401s.
  - confidence: 0.9
- **[P2] Revoke the token even when backend cleanup fails:** `src/agents/cli-runner.ts:124-126`
  - body: Token revocation runs after `await context.preparedBackend.cleanup?.()`, so a cleanup exception skips `unregisterMcpLoopbackToken` and leaves the bearer token registered past run teardown. Put revocation in its own finally path so cleanup failures cannot preserve stale authorization.
  - confidence: 0.87
- **[P2] Clean up registered tokens on all prepare errors:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: The token is registered before preparation continues, but the catch only covers `prepareCliBundleMcpConfig`; later preparation failures can leave a token registered before any backend is launched. Wrap all post-registration preparation in a cleanup guard or register after the remaining throwing work is complete.
  - confidence: 0.84
