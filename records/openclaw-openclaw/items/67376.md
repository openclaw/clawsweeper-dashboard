## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Guard runtime clearing by server instance:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` now clears the active runtime and every registered token unconditionally. Since `closeMcpLoopbackServer()` drops `activeMcpLoopbackServer` before awaiting the old close, a concurrent `ensureMcpLoopbackServer()` can start a fresh server whose tokens are then wiped by the old close callback, producing 401s for valid new clients.
  - confidence: 0.93
- **[P2] Always revoke after cleanup failures:** `src/agents/cli-runner.ts:124-126`
  - body: The token revocation runs after `await context.preparedBackend.cleanup?.()`, so any cleanup rejection skips `unregisterMcpLoopbackToken` and leaves the scoped bearer registered after run teardown. Put token revocation in its own `finally` path so cleanup errors cannot preserve authorization state.
  - confidence: 0.9
- **[P2] Revoke tokens on all prepare failures:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: The new catch only unregisters the token when `prepareCliBundleMcpConfig` rejects. Later preparation work can still throw before the returned cleanup is installed, so the issued token remains live until process shutdown; wrap all post-registration preparation in cleanup-on-error handling.
  - confidence: 0.88
