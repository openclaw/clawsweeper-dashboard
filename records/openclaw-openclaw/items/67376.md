## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Guard runtime clearing by server identity:** `src/gateway/mcp-http.ts:117`
  - body: `clearActiveMcpLoopbackRuntime()` now clears the active runtime and token table from any close callback. An older server can still be closing while a new loopback server starts, so the old close can wipe fresh tokens and make valid backends receive 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup fails:** `src/agents/cli-runner.ts:124-126`
  - body: Token revocation runs after `await context.preparedBackend.cleanup?.()`, so any cleanup exception skips `unregisterMcpLoopbackToken` and leaves a bearer token registered past run teardown. Put revocation in its own `finally` path.
  - confidence: 0.87
- **[P2] Clean up registered tokens on all prepare errors:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: The token is registered before prepare continues, but this catch only covers `prepareCliBundleMcpConfig`; later work such as docs-path resolution can throw before launch and leave the token registered until server shutdown.
  - confidence: 0.84
- **[P2] Keep plugin loader fallback out of this security fix:** `src/plugins/public-surface-loader.ts:115-121`
  - body: This security PR also changes plugin public-surface loading to fall back from source-artifact require to jiti on unresolved relative imports. That loader behavior is unrelated to MCP scope binding and should be removed or justified separately to avoid masking loader contract changes in an auth fix.
  - confidence: 0.8
