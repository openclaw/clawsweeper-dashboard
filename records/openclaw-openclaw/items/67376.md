## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Guard runtime clearing to the closing server:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` now clears the active runtime and the whole token table unconditionally. Because `closeMcpLoopbackServer()` drops `activeMcpLoopbackServer` before awaiting the old close callback, a concurrent new server can be started and then have its fresh token table wiped by the old server closing, causing valid new backends to get 401s.
  - confidence: 0.93
- **[P2] Revoke tokens even when backend cleanup fails:** `src/agents/cli-runner.ts:126`
  - body: The new revocation runs after `await context.preparedBackend.cleanup?.()`. If cleanup throws, the scoped bearer is never unregistered, leaving stale authorization in the process token map after the run tears down.
  - confidence: 0.9
- **[P2] Unregister prepare tokens on all later failures:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: The token is registered before the rest of preparation completes, but only `prepareCliBundleMcpConfig` failures revoke it. If later preparation work throws before the returned cleanup is installed, the token stays registered until server shutdown.
  - confidence: 0.88
