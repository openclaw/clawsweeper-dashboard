## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Guard runtime clearing to the closing server:** `src/gateway/mcp-http.loopback-runtime.ts:30-33`
  - body: `clearActiveMcpLoopbackRuntime()` now clears the active runtime and all registered tokens unconditionally. Because `closeMcpLoopbackServer()` drops `activeMcpLoopbackServer` before awaiting close, a concurrent `ensureMcpLoopbackServer()` can start a new server whose tokens are then wiped by the old server's close callback, causing valid fresh backends to get 401s.
  - confidence: 0.95
- **[P2] Revoke tokens even when cleanup fails:** `src/agents/cli-runner.ts:124-126`
  - body: The new token revocation runs after `await context.preparedBackend.cleanup?.()`. If cleanup throws, the `finally` exits before `unregisterMcpLoopbackToken()` and leaves a bearer token registered after run teardown.
  - confidence: 0.92
- **[P2] Revoke tokens on all prepare failures:** `src/agents/cli-runner/prepare.ts:148-156`
  - body: The token is registered before later preparation work, but only `prepareCliBundleMcpConfig()` failures unregister it. If docs-path, skills, auth, or other later prepare steps throw before cleanup is returned, the token remains registered in the process token map.
  - confidence: 0.88
- **[P2] Remove the unrelated plugin-loader fallback:** `src/plugins/public-surface-loader.ts:115-123`
  - body: This gateway security PR also changes plugin public-surface loading to swallow selected module-resolution errors and fall back to Jiti. That is outside the MCP loopback fix and can mask broken source artifacts or change plugin loading behavior without owner-reviewed proof, so it should be removed or split into a tested plugin-loader PR.
  - confidence: 0.82
