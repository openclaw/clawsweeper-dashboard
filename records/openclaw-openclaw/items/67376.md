## Summary

Keep open. Current main still has the remaining session/account/channel scope-spoofing path, and this PR is the right general direction, but the current patch has blocking token-lifecycle/channel-binding issues and lacks contributor real behavior proof.

## What This Changes

The PR replaces MCP loopback header-derived scope with a per-backend token registry, removes generated scope headers/env vars, invalidates scoped tool-cache entries, updates tests/changelog, and changes bundled plugin public-surface loader fallback behavior.

## Best Possible Solution

Land a rebased, security-reviewed MCP loopback fix that resolves the full session/account/channel/owner tuple from server-side token identity, removes emitted and accepted scope headers, invalidates scoped caches on revocation, and keeps unrelated loader behavior separate unless explicitly approved.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: use a valid loopback bearer and vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values flow into McpLoopbackToolCache and resolveGatewayScopedTools.

## Solution Assessment

No, not as currently proposed. Token-bound full-tuple scope is the maintainable direction, but this patch needs lifecycle fixes, actual messageChannel binding, separation or explicit review of the loader fallback, and contributor real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` clears the active runtime and token table unconditionally. Since `closeMcpLoopbackServer()` drops `activeMcpLoopbackServer` before awaiting close, an older close callback can wipe tokens for a freshly started server and make valid new clients receive 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:124-126`
  - body: The new token revocation runs after awaiting `context.preparedBackend.cleanup`. If cleanup throws, `unregisterMcpLoopbackToken` is skipped and the bearer remains registered after the backend run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: `prepareCliRunContext` registers the loopback token before later async setup work, but the catch only covers `prepareCliBundleMcpConfig`. A later setup error leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into token scope:** `src/agents/cli-runner/prepare.ts:152`
  - body: The registered token scope uses `params.messageProvider`, while current main passes `params.messageChannel ?? params.messageProvider` to bundled MCP clients. Runs with a distinct messageChannel would resolve tools under the wrong channel after this change.
  - confidence: 0.82
