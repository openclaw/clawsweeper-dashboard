## Summary

Keep open. This PR addresses a real, still-open MCP loopback auth-boundary issue, but the branch is dirty against current main, has blocking token lifecycle and runtime-clear findings, and lacks after-fix real behavior proof.

## What This Changes

The branch binds loopback MCP scope to a per-backend bearer-token registry, removes generated scope headers/env vars, adds cache invalidation/tests, and also changes bundled plugin public-surface loader fallback behavior.

## Best Possible Solution

Land a rebased, reviewed MCP loopback security patch that binds the full session/account/channel/owner tuple to token identity, fixes token registration and revocation lifecycle, and keeps plugin-loader fallback changes separate unless explicitly reviewed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: on current main, a valid loopback bearer can vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel and those values flow into resolveGatewayScopedTools. No live exploit was run in this read-only review.

## Solution Assessment

No, not as currently proposed. Token-bound scope is the right direction, but this branch needs a rebase, lifecycle fixes, actual messageChannel binding, separation or explicit review of the loader fallback, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: clearActiveMcpLoopbackRuntime() clears the active runtime and token table unconditionally. Because closeMcpLoopbackServer() drops activeMcpLoopbackServer before awaiting close, a concurrent ensure can start a new server and then the old close callback can wipe the new server's tokens, causing valid fresh clients to get 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:124-126`
  - body: The token revocation runs after awaiting context.preparedBackend.cleanup. If cleanup throws, unregisterMcpLoopbackToken is skipped and the bearer remains registered after the run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: prepareCliRunContext registers the loopback token before later async setup work, but only catches prepareCliBundleMcpConfig failures. A later setup error, such as docs path resolution, leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into token scope:** `src/agents/cli-runner/prepare.ts:152`
  - body: The registered token scope uses params.messageProvider, while current main already prefers params.messageChannel over params.messageProvider for loopback scope env. Runs with a distinct messageChannel would resolve tools under the wrong channel after this change.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR hardens a real auth boundary, but the proposed token registry lifecycle, server-clear behavior, channel binding, and unrelated loader fallback need security review before merge.

Concerns:

- **[medium] Bearer tokens can outlive backend lifecycle:** `src/agents/cli-runner.ts:124`
  - body: The token registry can retain authorization material when setup fails after registration or when backend cleanup throws before token revocation runs.
  - confidence: 0.86
- **[medium] Old server shutdown can revoke fresh tokens:** `src/gateway/mcp-http.loopback-runtime.ts:30`
  - body: The unconditional runtime clear can erase a newer server's token table when an older server close callback runs later, making authorization state depend on shutdown timing.
  - confidence: 0.9
- **[medium] Token scope can use the wrong channel:** `src/agents/cli-runner/prepare.ts:152`
  - body: Binding messageProvider instead of the actual messageChannel means the server-side token scope may not match the channel-specific scope current main passes to bundled MCP clients.
  - confidence: 0.82
- **[low] Plugin loader fallback changes code-loading behavior:** `src/plugins/public-surface-loader.ts:112`
  - body: The PR changes bundled plugin public-surface loading to fall back to the module loader for selected resolution errors, which is unrelated to the MCP token fix and should be split or explicitly reviewed.
  - confidence: 0.78
