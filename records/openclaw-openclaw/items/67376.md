Full review comments:

- **[P1] Bind the full current loopback context:** `src/gateway/mcp-http.loopback-runtime.ts:7-12`
  - body: This scoped-token type only carries session/account/provider/owner, but current main also carries channel, thread, message, audio, inbound-event, and reply-delivery context through the same loopback surface. A rebased fix must bind every current context field to the token identity or it will leave part of the spoofable security boundary unresolved.
  - confidence: 0.92
- **[P1] Guard runtime clearing by the server instance:** `src/gateway/mcp-http.ts:117`
  - body: `clearActiveMcpLoopbackRuntime()` now clears the active runtime and token table from any close callback. Because an old server can still be closing while a new loopback server starts, the old close can wipe fresh tokens and make valid backends receive 401s; keep the owner/server identity guard when clearing global runtime state.
  - confidence: 0.9
- **[P2] Revoke the token even when backend cleanup fails:** `src/agents/cli-runner.ts:124-126`
  - body: The new revocation runs after `await context.preparedBackend.cleanup?.()`, so any cleanup exception skips `unregisterMcpLoopbackToken` and leaves the bearer token registered past run teardown. Put token revocation in its own `finally` so cleanup failures cannot preserve stale authorization.
  - confidence: 0.87
- **[P2] Clean up registered tokens on all prepare errors:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: The token is registered before prepare continues, but the catch only covers `prepareCliBundleMcpConfig`; later work such as docs-path resolution can throw before the backend launches and leave the token registered until server shutdown. Wrap all post-registration preparation in a cleanup guard or register after the remaining throwing work is complete.
  - confidence: 0.84
