## Summary

Keep open: the linked security issue remains valid on current main and this PR contains the right token-bound-scope direction, but the branch is conflicting, lacks real behavior proof, and still has security-sensitive lifecycle blockers.

## What This Changes

This PR binds loopback MCP scope to registered per-backend bearer tokens, removes generated scope headers/env values, adds scoped cache invalidation/tests, and changes bundled plugin public-surface loading fallback behavior.

## Best Possible Solution

Land a reviewed, rebased MCP loopback security patch that binds the full session/account/channel/owner scope tuple to token identity, fixes token lifecycle cleanup, and keeps unrelated plugin-loader behavior separate unless explicitly reviewed.

## Reproduction Assessment

Yes. Source inspection on current main shows a valid loopback bearer can still vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values flow into resolveGatewayScopedTools through McpLoopbackToolCache; no live exploit was run in this read-only review.

## Solution Assessment

No, not as proposed. Token-bound scope is the right direction, but this branch needs a rebase, lifecycle fixes, messageChannel handling, separation or explicit review of the plugin-loader fallback, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: clearActiveMcpLoopbackRuntime() clears the global active runtime and registered-token table unconditionally. A concurrent restart can register fresh tokens and then have an older server close callback wipe them, causing valid new clients to get 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:124-126`
  - body: The new token revocation runs after awaiting preparedBackend.cleanup. If cleanup throws, unregisterMcpLoopbackToken is skipped and the bearer remains registered after the run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: prepareCliRunContext registers the loopback token before later async setup work, but only catches prepareCliBundleMcpConfig failures. Any later setup error leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into token scope:** `src/agents/cli-runner/prepare.ts:152`
  - body: The registered token scope uses params.messageProvider, but run params can carry a distinct messageChannel and current main already prefers messageChannel over messageProvider for loopback scope. Those runs would resolve tools under the wrong channel.
  - confidence: 0.82
