## What This Changes

The branch replaces MCP loopback header-derived scope with a per-backend token registry, removes generated scope headers/env, adds token/cache regression tests, and tweaks bundled plugin public-surface fallback loading.

## Best Possible Solution

Land a rebased, security-reviewed fix that binds the full current loopback scope tuple to token identity, preserves inbound-event and message-channel semantics, removes mutable scope headers, fixes token lifecycle, and keeps unrelated loader changes separate.

Full review comments:

- **[P1] Bind inbound-event scope to the token:** `src/gateway/mcp-http.loopback-runtime.ts:8-14`
  - body: Current main threads `x-openclaw-inbound-event-kind` into loopback tool resolution so room-event runs get the narrower message-only context. The new token-bound scope omits `inboundEventKind`, so merging this shape would turn those requests into an undefined inbound-event context and can expose broader loopback tools unless the token registry records and returns that field too.
  - confidence: 0.9
- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` clears the active runtime and token table unconditionally. Since `closeMcpLoopbackServer()` drops `activeMcpLoopbackServer` before awaiting close, an older close callback can wipe tokens for a freshly started server and make valid new clients receive 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:122-126`
  - body: The new token revocation runs after awaiting `context.preparedBackend.cleanup`. If cleanup throws, `unregisterMcpLoopbackToken` is skipped and the bearer remains registered after the backend run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:173-177`
  - body: `prepareCliRunContext` registers the loopback token before later async setup work, but the catch only covers `prepareCliBundleMcpConfig`. A later setup error leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into token scope:** `src/agents/cli-runner/prepare.ts:152`
  - body: The registered token scope uses `params.messageProvider`, while current main passes `params.messageChannel ?? params.messageProvider` to bundled MCP clients. Runs with a distinct messageChannel would resolve tools under the wrong channel after this change.
  - confidence: 0.82
