## Summary

Keep open. Current main has already mitigated owner-header escalation with owner/non-owner bearers, but it still accepts session/account/channel scope from mutable headers; this PR targets real remaining work, but the proposed patch has token lifecycle bugs and an unrelated plugin loader change that need maintainer/security review before merge.

## What This Changes

This PR adds a loopback MCP token registry for request scope, removes generated scope headers/env vars, invalidates scoped tool-cache entries on token revocation, updates tests/changelog, and changes the bundled plugin public-surface loader fallback.

## Best Possible Solution

Land a narrow MCP loopback security patch that binds the full session/account/channel/owner scope tuple to per-backend tokens, preserves current owner-token mitigation, handles all revocation/shutdown failure paths, and keeps plugin loader behavior in a separate reviewed change.

## Reproduction Assessment

Yes. Current main source and tests give a high-confidence reproduction for the remaining mutable session/account/channel scope path; the owner-header escalation part is already mitigated by the merged owner/non-owner bearer split.

## Solution Assessment

No, not as proposed. Token-bound scope is the right direction, but this branch needs lifecycle fixes, `messageChannel` handling, and removal or separate review of the plugin loader fallback before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Guard runtime clears by server identity:** `src/gateway/mcp-http.loopback-runtime.ts:30-32`
  - body: `clearActiveMcpLoopbackRuntime()` clears the global runtime and registered token table unconditionally. Because `closeMcpLoopbackServer()` drops the active server before the old close callback runs, a concurrent `ensureMcpLoopbackServer()` can create fresh tokens that the old close then wipes, causing valid new backends to receive 401s.
  - confidence: 0.9
- **[P2] Revoke tokens even when cleanup throws:** `src/agents/cli-runner.ts:122-126`
  - body: The new revocation happens after `await context.preparedBackend.cleanup?.()`. If backend cleanup throws, token revocation is skipped and the bearer remains registered after the run has ended.
  - confidence: 0.88
- **[P2] Unregister prepared tokens on setup failure:** `src/agents/cli-runner/prepare.ts:145-153`
  - body: `prepareCliRunContext` registers the loopback token before later async setup work, but only catches failures from `prepareCliBundleMcpConfig`. Any later setup error leaves a valid token in the registry even though no prepared context is returned.
  - confidence: 0.84
- **[P2] Bind the actual message channel into the token scope:** `src/agents/cli-runner/prepare.ts:151`
  - body: The token scope records `params.messageProvider`, but current run params can carry a distinct `messageChannel` and current main already prefers `messageChannel ?? messageProvider` for the loopback channel header. Runs where the backend provider differs from the source channel would resolve tools under the wrong channel.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR hardens an auth boundary, but the proposed registry lifecycle and unrelated plugin loader fallback need security review before merge.

Concerns:

- **[medium] Bearer tokens can outlive backend lifecycle:** `src/agents/cli-runner.ts:122`
  - body: The token registry can retain authorization material when backend cleanup throws or when setup fails after registration, keeping credentials active longer than intended.
  - confidence: 0.86
- **[medium] Old server shutdown can revoke fresh tokens:** `src/gateway/mcp-http.loopback-runtime.ts:30`
  - body: The unconditional runtime clear can erase a newer server's token table when an older server close callback runs later, breaking valid clients and making auth state depend on close timing.
  - confidence: 0.9
- **[low] Plugin loader fallback changes code-loading behavior:** `src/plugins/public-surface-loader.ts:112`
  - body: The PR changes a bundled plugin public-surface loading path to fall back to Jiti for selected module-resolution errors, which is unrelated to the MCP token fix and should be split or separately reviewed.
  - confidence: 0.78
