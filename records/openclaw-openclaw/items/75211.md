## Summary

Keep open. Current main supports documented normal WhatsApp route bindings through `resolveAgentRoute` when the sender is expressed as `match.peer`, so the title overstates that path, but WhatsApp still lacks the configured-binding provider hooks and inbound conversation-binding integration used by other channels. Because unsupported configured bindings are skipped and this is a privilege-boundary report, it needs maintainer/security review before implementation.

## What This Changes

The issue requests WhatsApp binding support or diagnostics so inbound WhatsApp messages route to the configured restricted agent instead of falling through to the default agent.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: preserve and test normal `match.peer` route bindings, add validation or doctor diagnostics for WhatsApp sender rules placed in `accountId`, and implement WhatsApp configured ACP/conversation binding hooks plus inbound readiness only if maintainers want persistent WhatsApp conversation binding support.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction path for the missing configured-binding provider, but the report's normal route example uses `accountId` where current docs require `match.peer` for sender routing.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address configured ACP/conversation bindings but would not fix the shown accountId-as-sender config; the safer solution is a maintainer-chosen combination of diagnostics, docs, and explicit WhatsApp configured-binding support.
