## Summary

Keep open. Current main supports documented WhatsApp sender routing through `match.peer`, but the security-sensitive silent-fallback concern remains for mis-scoped `accountId` rules and unsupported configured ACP/conversation binding hooks.

## What This Changes

The issue asks WhatsApp bindings to route inbound senders correctly or warn when a configured binding is unsupported instead of falling through to the default agent.

## Best Possible Solution

Make WhatsApp binding semantics explicit with diagnostics for unsupported or mis-scoped rules, and add configured ACP/conversation binding support only if maintainers want that surface for WhatsApp.

## Reproduction Assessment

Yes for the static routing path: a WhatsApp sender phone number placed in `match.accountId` will not match an inbound message whose actual `accountId` is the login account, so it can fall through to the default agent. Source inspection also confirms the missing configured-binding hook path, though live WhatsApp delivery was not run.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown `accountId`-as-sender route config, so the safer fix is a maintainer-chosen combination of diagnostics, docs, and optional configured-binding support.
