## Summary

Keep open. Current main supports documented WhatsApp sender routing through `match.peer`, but the security-sensitive gap remains because `accountId` is account identity rather than sender identity and WhatsApp still lacks the configured-binding hooks used by ACP/conversation binding paths.

## What This Changes

The issue asks for WhatsApp route/configured bindings to be enforced or diagnosed instead of silently routing restricted senders to the default agent.

## Best Possible Solution

Make WhatsApp binding behavior explicit: preserve documented `match.peer` sender routing, add diagnostics for accountId-as-sender or unsupported configured bindings, and add WhatsApp configured ACP/conversation binding hooks if maintainers want that persistent binding surface.

## Reproduction Assessment

Yes for the static routing path: a WhatsApp binding with a phone number in `match.accountId` misses inbound messages whose account id is `default` or a named account. Source search also reproduces the missing configured-binding provider path; live WhatsApp delivery was not run.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown accountId-as-sender route config, so the safer path is maintainer-chosen diagnostics/docs plus explicit configured-binding support if desired.
