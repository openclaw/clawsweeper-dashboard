## Summary

Keep open. Current main does support documented normal WhatsApp route bindings through the shared route resolver, so the broad title overstates that path, but the security-relevant configured ACP/conversation binding gap remains: WhatsApp has no configured-binding provider hooks and its inbound path does not apply configured/runtime conversation binding route resolution like supported channels.

## What This Changes

The issue asks for WhatsApp bindings to route inbound WhatsApp conversations to the intended restricted agent, or to warn when a configured WhatsApp binding shape is unsupported.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: preserve and test normal peer/account route bindings, add config validation or doctor diagnostics for WhatsApp sender rules placed in accountId, and implement WhatsApp configured ACP/conversation binding hooks plus inbound readiness only if maintainers want persistent WhatsApp ACP binding support.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction path for the missing configured-binding provider and inbound integration, but the report's normal route example uses accountId where current docs require match.peer for sender routing.

## Solution Assessment

Unclear. Adding only compileConfiguredBinding and matchInboundConversation would address configured ACP/conversation bindings but would not fix the shown accountId-as-sender config; the safest solution is a maintainer-chosen combination of diagnostics, docs, and explicit WhatsApp configured-binding support.
