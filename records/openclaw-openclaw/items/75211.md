## Summary

Keep open. Current main supports documented WhatsApp route bindings through `match.peer`, but WhatsApp still lacks configured-binding provider hooks and diagnostics for the reported `accountId`-as-sender misconfiguration, so this security-sensitive routing boundary needs maintainer review.

## What This Changes

The issue asks WhatsApp routing to honor or warn about bindings that operators expect to route restricted senders away from the default agent.

## Best Possible Solution

Make the WhatsApp binding contract explicit: preserve documented `match.peer` routing, warn or doctor-fix unsupported/mis-scoped WhatsApp binding rules, and add configured ACP/conversation binding support only if maintainers want that persistent binding surface for WhatsApp.

## Reproduction Assessment

Yes. Source inspection and existing route tests reproduce the static path: a phone number in `match.accountId` is treated as a WhatsApp account selector, while inbound sender identity is passed as `peer.id`, so the example can fall through to the default agent; the missing configured-binding provider path is also reproducible by targeted search.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address the ACP/configured conversation-binding gap but not the shown route binding that uses `accountId` as the sender, so the safer solution needs maintainer-chosen diagnostics/docs plus optional provider support.
