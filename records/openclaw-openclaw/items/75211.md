## Summary

Keep open. Current main does honor normal WhatsApp route bindings when sender routing is configured as `match.peer`, but WhatsApp still does not expose the configured-binding provider hooks used by ACP/conversation bindings, and unsupported configured-binding providers are skipped without a WhatsApp-specific diagnostic. Because the report describes a privilege-boundary assumption, this should stay with maintainers for security/product review rather than being swept closed.

## What This Changes

The issue asks for WhatsApp bindings to route inbound messages to the configured target agent, or at least warn when WhatsApp binding configuration is unsupported or mis-scoped.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: keep the existing documented `match.peer` route-binding behavior, add validation or doctor diagnostics for sender rules placed in `accountId`, and only add WhatsApp configured ACP/conversation binding hooks if maintainers decide persistent WhatsApp conversation bindings are supported.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path for the missing configured-binding provider hooks, but the issue’s route-binding example uses `accountId` where current docs require `match.peer` for sender routing.

## Solution Assessment

Unclear. Implementing only `compileConfiguredBinding` and `matchInboundConversation` would help configured ACP/conversation bindings, but it would not fix the shown accountId-as-sender configuration; the maintainable fix likely combines diagnostics, docs, and a maintainer-approved WhatsApp configured-binding contract.
