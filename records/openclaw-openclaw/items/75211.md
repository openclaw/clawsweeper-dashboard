## Summary

Keep open: current main still has the WhatsApp route behavior described in the report, no WhatsApp configured-binding provider hooks, and no runtime diagnostic for phone-shaped `match.accountId` rules. The only related fix PR was closed unmerged, so there is no merged implementation to treat as superseding or fixed.

## What This Changes

The issue asks for WhatsApp bindings to either honor sender-specific route/configured-binding semantics or warn when a phone-number-shaped `match.accountId` rule is likely being used as a sender match.

## Best Possible Solution

Choose a maintainer-approved fix: add a clear WhatsApp `accountId`-vs-`peer.id` diagnostic, and only add provider parity if WhatsApp inbound routing is also wired through the configured-binding route contract with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: WhatsApp inbound passes the connected account as `accountId`, the sender as `peer.id`, and current main has no WhatsApp provider hooks or diagnostic for the reported misconfiguration.

## Solution Assessment

Unclear as a final product direction. A diagnostic is the narrowest low-risk fix for the reported `accountId` misuse, while configured-binding provider parity should only land if inbound routing is completed and tested.
