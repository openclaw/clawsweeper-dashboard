## Summary

Keep open. Current main documents and supports WhatsApp sender routing through `match.peer`, but the reported `accountId`-as-sender shape can still fall through to the default agent and WhatsApp still lacks configured-binding provider hooks or diagnostics for that gap.

## What This Changes

The issue asks for WhatsApp bindings support or warnings so mis-scoped route rules do not silently send restricted contacts to the default agent.

## Best Possible Solution

Make the WhatsApp binding contract explicit by preserving documented `match.peer` routing, warning or doctor-reporting likely sender-in-`accountId` mistakes, and adding configured-binding hooks only if maintainers want WhatsApp to support that persistent binding surface.

## Reproduction Assessment

Yes. The static path is high-confidence: WhatsApp passes the sender as `peer.id`, while `accountId` is treated as the channel account selector, and configured conversation bindings are skipped for channels without provider hooks.

## Solution Assessment

Unclear. Implementing only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown route binding that puts a sender phone in `accountId`, so the best fix likely needs maintainer-chosen diagnostics plus any desired provider support.
