## Summary

Keep open. Current main does not support closing the report as already fixed: documented WhatsApp peer/account route bindings are wired through the generic route resolver, but WhatsApp still lacks the separate configured-conversation binding adapter and inbound configured/runtime binding resolution used by other channel plugins, so the security-relevant boundary needs maintainer review rather than cleanup closure.

## What This Changes

The issue asks WhatsApp to honor route/binding configuration consistently or warn when unsupported binding rules would fall through to the default agent.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: preserve and test documented `match.peer` sender routing, add validation or doctor diagnostics for sender values placed in `match.accountId` if maintainers want that guardrail, and add WhatsApp configured-binding provider plus inbound configured/runtime binding route resolution only if WhatsApp should support persistent ACP/conversation bindings.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path for the narrower missing configured-binding integration, but the exact report example uses `match.accountId` where current docs require `match.peer` for a WhatsApp sender split.

## Solution Assessment

Unclear. The requested hook implementation is only part of a correct fix; the maintainable solution needs maintainer/security choice between diagnostics for mis-scoped route rules and full WhatsApp configured-conversation binding support.
