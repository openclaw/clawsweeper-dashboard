## Summary

Keep open. Current main implements documented WhatsApp route bindings through `match.peer`, so the reported `accountId` example is using the account selector for a sender phone number, but the silent default-agent fallback and missing WhatsApp configured-binding provider hooks are still a security-sensitive operator-expectation gap.

## What This Changes

The issue asks WhatsApp to honor sender-specific bindings or warn when unsupported/mis-specified bindings would silently fall through to the default agent.

## Best Possible Solution

Make WhatsApp binding semantics explicit: preserve documented `match.peer` routing, add a warning or doctor/docs path for sender values placed in `accountId`, and add WhatsApp configured ACP binding hooks only if maintainers want WhatsApp in that contract.

## Reproduction Assessment

Yes. Source inspection reproduces the static path: WhatsApp puts sender identity in `peer.id` and the linked account in `accountId`, while the resolver tests show a nonmatching account selector falls through to the default route.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address the ACP configured-binding gap but not the shown route binding, so the maintainable fix needs maintainer-chosen diagnostics/docs plus optional provider support.
