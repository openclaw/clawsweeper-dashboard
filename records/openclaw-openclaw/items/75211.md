## Summary

Keep open: current main still lacks the WhatsApp configured-binding provider hooks and the runtime diagnostic for phone-shaped `match.accountId` values, and the issue is already paired with open closing PR #76434.

## What This Changes

The issue requests WhatsApp routing bindings to either honor the channel configured-binding contract or warn operators when a sender phone is placed in `match.accountId` and would fall through to the default agent.

## Best Possible Solution

Review and land or replace #76434 with a narrow fix that preserves documented `match.peer` sender routing, adds a clear diagnostic for sender values in `match.accountId`, and adds WhatsApp configured-binding parity if maintainers accept that contract for WhatsApp.

## Reproduction Assessment

Yes. Source inspection reproduces the static path: WhatsApp sender identity is supplied as `peer.id`, a sender phone in `match.accountId` is treated as a connected-account selector, and WhatsApp currently has no configured-binding provider hooks.

## Solution Assessment

Likely yes, but it belongs in PR review. A diagnostic plus provider parity is safer than silently auto-promoting `accountId` into `peer.id`, because account ids can legitimately be phone-shaped.
