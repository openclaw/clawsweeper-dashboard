## Summary

Keep open: current main still lacks WhatsApp configured-binding provider hooks and an accountId-vs-peer diagnostic, and the only related fix PR was closed unmerged. Because the remaining work affects security-sensitive routing semantics, it should stay in maintainer review rather than be swept closed.

## What This Changes

The issue asks OpenClaw to stop silently falling back for WhatsApp route bindings by adding WhatsApp configured-binding provider support or a warning for unsupported or misconfigured bindings.

## Best Possible Solution

Choose a narrow maintainer-approved fix: add a clear WhatsApp accountId-vs-peer diagnostic, and only add configured-binding provider parity if WhatsApp inbound routing is wired through that contract with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: WhatsApp sender identity is routed as peer.id, a sender phone in match.accountId is treated as a connected-account selector, and WhatsApp currently has no configured-binding provider hooks.

## Solution Assessment

Unclear as a final product direction. The reported accountId-as-sender shape is not the documented route binding shape, so a diagnostic is the safest narrow fix unless maintainers also want to complete WhatsApp configured-binding provider support.
