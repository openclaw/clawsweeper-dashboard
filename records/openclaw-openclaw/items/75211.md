## Summary

Keep open. Current main supports documented WhatsApp DM routing through `match.peer`, but the reported accountId-as-sender configuration still falls through silently and WhatsApp still lacks the configured-binding provider hooks used by ACP/conversation binding paths, so this security-sensitive routing ambiguity needs maintainer review.

## What This Changes

The issue requests WhatsApp bindings to honor sender-based routing rules or warn when WhatsApp binding rules cannot be enforced.

## Best Possible Solution

Make WhatsApp binding semantics explicit: keep peer-based DM routing, add diagnostics for unsupported or accountId-as-sender rules, and add WhatsApp configured-binding hooks only if persistent ACP/conversation binding should be supported for this channel.

## Reproduction Assessment

Yes for the static routing path: a WhatsApp binding that puts the sender phone in `match.accountId` will not be considered when inbound `msg.accountId` is `default`, then the resolver falls through to the default agent. Source inspection also reproduces the missing configured-binding provider path, but live WhatsApp delivery was not exercised.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown accountId-as-sender route config, so the safer maintainer path is diagnostics/docs plus explicit WhatsApp configured-binding support if that surface is intended.
