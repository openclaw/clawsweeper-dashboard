## Summary

Keep open. Current main does not support the title’s broad claim for normal WhatsApp route bindings: inbound WhatsApp messages are routed through the generic `resolveAgentRoute` path with channel, account, and peer data, and the docs show sender routing via `match.peer`. The remaining concern is narrower but real: WhatsApp still has no configured-binding provider hooks and its inbound path does not apply configured/runtime conversation binding route resolution like Telegram, Discord, or LINE, so this should stay with maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make WhatsApp inbound binding behavior enforceable or diagnosable so messages do not fall back to a more privileged default agent when operators expect restricted-agent routing.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: keep and test normal sender routing through `match.peer`, add config validation or doctor diagnostics for `accountId`-as-sender mistakes if maintainers want that guardrail, and add WhatsApp configured-binding provider hooks plus inbound configured/runtime binding route resolution only if WhatsApp should support persistent ACP/conversation bindings.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path for the missing WhatsApp configured-binding provider/inbound integration, but the exact reported route example uses `accountId` as a sender selector where current docs and routing require `match.peer`.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown normal route-binding config; the maintainable fix needs maintainer choice between docs/diagnostics for mis-scoped route rules and full WhatsApp configured-binding support.
