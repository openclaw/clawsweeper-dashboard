## Summary

Close as implemented on current main: WhatsApp now exposes the configured-binding hooks, inbound dispatch rewrites matching configured ACP routes and fails closed when the target cannot be readied, and current docs show sender-specific WhatsApp matching through `match.peer` rather than `accountId`. The fix is current-main-only relative to `v2026.6.6`.

## What This Changes

The issue reports WhatsApp binding rules being ignored, especially for sender-specific routing, causing inbound messages to fall back to the default agent.

## Best Possible Solution

Keep the current main implementation and close this report; if maintainers still want an extra accountId-versus-peer diagnostic, track that as a separate narrow enhancement.

## Reproduction Assessment

No current-main failing path was found. The historical failure is source-backed by the previously missing WhatsApp configured-binding adapter, and current tests now exercise the fixed direct/group materialization and no-local-fallback behavior.

## Solution Assessment

Yes. The landed fix is at the WhatsApp plugin/channel boundary, uses the existing configured-binding registry instead of core special-casing, and the docs clarify that sender-specific WhatsApp routing uses `match.peer` while `accountId` selects the connected account.
