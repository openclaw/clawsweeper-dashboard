## Summary

Keep open. Current main does not support the broad claim that WhatsApp ignores all normal route bindings: WhatsApp inbound calls the generic route resolver, and the docs say sender-specific WhatsApp routing should use `match.peer`, not a phone number in `match.accountId`. But the security-relevant remaining gap is real: the WhatsApp plugin still has no configured-binding provider hooks, and its inbound path does not apply the configured/runtime binding route integration used by comparable channels.

## What This Changes

The issue requests WhatsApp binding support or diagnostics so inbound WhatsApp conversations do not silently fall back to the default agent when operators expect restricted-agent routing.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: preserve documented sender routing through `match.peer`, warn or validate phone-like/mis-scoped `accountId` rules where possible, and add WhatsApp configured-binding provider plus inbound configured/runtime binding resolution only if maintainers intend WhatsApp to support persistent ACP/conversation bindings.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction path for the missing WhatsApp configured-binding provider and absent inbound configured-binding route step, but the report's concrete `accountId` phone-number example is not the documented way to route one sender.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address the configured-binding gap, but the safer fix also needs maintainer judgment on diagnostics and docs so operators do not confuse channel account routing with sender routing.
