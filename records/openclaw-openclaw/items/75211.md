## Summary

Keep open. Current main handles documented WhatsApp sender routing through `match.peer`, but WhatsApp still lacks configured-binding provider hooks and inbound conversation-binding integration, so the security-sensitive silent-fallback concern needs maintainer review rather than cleanup closure.

## What This Changes

The issue asks for WhatsApp route/configured bindings to be enforced or diagnosed instead of silently falling back to the default agent when unsupported binding shapes are configured.

## Best Possible Solution

Make the WhatsApp binding contract explicit: preserve documented `match.peer` routing, add diagnostics for unsupported or mis-scoped WhatsApp binding rules, and add configured ACP/conversation binding support only if maintainers want that persistent binding surface for WhatsApp.

## Reproduction Assessment

Yes for the static routing/config path: a WhatsApp binding with a phone number in `match.accountId` will miss an inbound message whose actual account id is `default` and fall through the resolver tiers. Source inspection also reproduces the missing configured-binding provider path, but live WhatsApp delivery was not exercised.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown accountId-as-sender config, so the safer solution is a maintainer-chosen combination of diagnostics, docs, and explicit WhatsApp configured-binding support.
