## Summary

Keep open: current main still has the source-visible WhatsApp binding gap and #76434 is an open, unmerged PR with closing syntax for this issue. The documented sender route shape uses `match.peer`, while phone-shaped `match.accountId` remains a connected-account selector that can fall through without a targeted runtime warning.

## What This Changes

The issue requests WhatsApp routing to honor or warn on sender-specific binding rules instead of silently falling back to the default agent.

## Best Possible Solution

Review and land or replace #76434 with a narrow fix that preserves documented `match.peer` routing, adds a clear diagnostic for phone-shaped WhatsApp sender values in `match.accountId`, and adds WhatsApp configured-binding parity if maintainers accept that channel contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: WhatsApp supplies sender identity as `peer.id`, `match.accountId` is evaluated as a connected-account selector, and the WhatsApp plugin currently lacks configured-binding provider hooks.

## Solution Assessment

Likely yes, but it belongs in PR review. A diagnostic plus provider parity is narrower and safer than auto-promoting `match.accountId` into `match.peer.id`, because account ids can legitimately be phone-shaped.
