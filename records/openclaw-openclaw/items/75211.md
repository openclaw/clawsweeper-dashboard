## Summary

Keep open. Current main supports documented WhatsApp route bindings through `match.peer`, but the reported sender-in-`accountId` example can still fall through and WhatsApp still lacks configured-binding provider hooks or diagnostics, so this security-sensitive routing expectation needs maintainer handling.

## What This Changes

The issue asks for WhatsApp to honor expected route bindings for inbound senders or warn when configured WhatsApp bindings are unsupported or mis-scoped.

## Best Possible Solution

Make WhatsApp binding semantics explicit: preserve documented `match.peer` routing, add diagnostics or doctor/docs for sender values placed in `accountId`, and add configured ACP binding support only if maintainers want WhatsApp in that contract.

## Reproduction Assessment

Yes. Source inspection reproduces the static path: WhatsApp passes sender identity as `peer.id` and the linked account as `accountId`, while the resolver tests show a nonmatching account selector falls through to the default route.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address the configured ACP binding gap but not the shown route binding, so the maintainable fix needs maintainer-chosen diagnostics/docs plus optional provider support.
