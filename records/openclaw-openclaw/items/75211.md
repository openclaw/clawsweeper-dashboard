## Summary

Keep open. Current main does not support the issue's broad claim for normal WhatsApp sender route bindings: WhatsApp inbound routing does call the generic route resolver, and documented sender rules use `match.peer`, not `match.accountId`. The remaining concern is still real and security-relevant: WhatsApp has no configured-binding provider hooks or inbound configured/runtime conversation-binding route pass, so maintainers should decide the exact contract before cleanup closure.

## What This Changes

The issue asks for WhatsApp inbound route bindings to be honored for restricted-agent routing, or for unsupported WhatsApp binding shapes to warn instead of silently falling through to the default agent.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: keep documented sender routing on `match.peer`, add validation or doctor diagnostics for `accountId`-as-sender mistakes if maintainers want to prevent false security assumptions, and add WhatsApp configured-binding plus runtime conversation-binding support only if WhatsApp should support persistent ACP conversation bindings.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence explanation for the shown config falling through and for missing WhatsApp ACP configured-binding hooks, but the exact live WhatsApp session reproduction was not exercised here.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would not fix the shown `type: "route"` example; the safer solution is a maintainer-chosen combination of docs/diagnostics for route config and optional full WhatsApp configured-binding support.
