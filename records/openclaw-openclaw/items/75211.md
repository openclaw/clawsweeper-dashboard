## Summary

Keep open: current main still lacks WhatsApp configured-binding provider hooks and runtime diagnostics for sender-phone values placed in `match.accountId`, and #76434 is an open unmerged PR that explicitly closes this issue.

## What This Changes

The issue asks WhatsApp routing to enforce or warn on inbound sender bindings so restricted contacts do not fall through to a more privileged default agent.

## Best Possible Solution

Review and land or replace #76434 with a narrow fix that keeps documented `match.peer` routing, adds clear diagnostics for sender values in `match.accountId`, and adds WhatsApp configured-binding parity if maintainers accept that contract for WhatsApp.

## Reproduction Assessment

Yes. Source inspection reproduces the static path: WhatsApp sender identity is routed as `peer.id`, a sender phone in `match.accountId` is treated as a connected-account selector, and WhatsApp currently has no configured-binding provider hooks.

## Solution Assessment

Likely yes, but it belongs in PR review. A diagnostic plus provider parity is safer than silently auto-promoting `accountId` into `peer.id`, because account ids can legitimately be phone-shaped.
