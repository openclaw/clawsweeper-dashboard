## Summary

Keep open. Current main supports documented WhatsApp route bindings via `match.peer`, but WhatsApp still lacks configured-binding provider hooks and a sender phone placed in `match.accountId` can fall through to the default agent without a targeted diagnostic, so the security-sensitive routing expectation still needs maintainer handling.

## What This Changes

The issue asks for WhatsApp bindings not to fail silently when operators try to route WhatsApp senders, either by adding configured-binding hooks or by warning on unsupported or mis-scoped bindings.

## Best Possible Solution

Make WhatsApp binding semantics explicit: keep sender routing on `match.peer`, add validation/doctor or runtime diagnostics for sender-looking values in `match.accountId`, and decide whether WhatsApp should gain configured ACP binding provider support through a reviewed narrow change or the existing broad PR.

## Reproduction Assessment

Yes. Source inspection plus existing resolver tests reproduce the static failure path: WhatsApp passes the sender as `peer.id`, while the reported sender value in `accountId` is treated as an account selector and can fall through to the default route.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address persistent ACP binding support, but the reported `type: "route"` example also needs clearer route-binding diagnostics or docs so operators do not mistake `accountId` for sender identity.
