## Summary

Keep open. Current main supports documented WhatsApp route bindings through `match.peer`, but the reported sender-in-`accountId` shape still falls through and WhatsApp still lacks configured ACP binding hooks or diagnostics; related PR #56866 is open and only addresses the ACP side.

## What This Changes

The issue requests WhatsApp binding support or diagnostics so sender-specific bindings do not silently route to the default agent.

## Best Possible Solution

Make WhatsApp binding semantics explicit: keep documented `match.peer` routing, add diagnostics or docs for sender values placed in `accountId`, and review a narrowed ACP provider hook implementation for persistent `type: "acp"` bindings.

## Reproduction Assessment

Yes. Source inspection reproduces the static path: WhatsApp uses sender identity as `peer.id` and the linked account as `accountId`, while resolver coverage shows a nonmatching account selector falls back to the default route.

## Solution Assessment

Unclear. Adding only configured-binding hooks would help persistent ACP bindings, but the reported `type: "route"` example needs `match.peer` diagnostics/docs or validation so operators do not mistake a fallback for an enforced security boundary.
