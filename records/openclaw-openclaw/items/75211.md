## Summary

Keep open. Current main supports documented WhatsApp route bindings through `match.peer`, but the reported `accountId`-as-sender configuration can still silently fall through to the default agent and WhatsApp still lacks configured-binding provider hooks or diagnostics for that confusion.

## What This Changes

The issue asks WhatsApp bindings to enforce sender-scoped routing or warn when WhatsApp binding rules are unsupported or mis-scoped.

## Best Possible Solution

Make the WhatsApp binding contract explicit: preserve `match.peer` routing, add validation or doctor warnings for sender phones placed in `match.accountId`, and add WhatsApp configured ACP binding hooks only if maintainers want that persistent binding surface.

## Reproduction Assessment

Yes. The current source path shows direct sender identity is matched as `peer.id` while `accountId` is the linked WhatsApp account, and existing route tests cover the default fallback for account mismatch.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address the ACP/configured-binding gap but would not fix the shown route binding that uses `accountId` as the sender; diagnostics plus docs and possibly provider hooks is safer.
