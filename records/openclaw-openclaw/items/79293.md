## Summary

Keep open: current main still lacks a generic accepted/unknown delivery status that would let a channel distinguish Tencent-side acceptance from user-visible Weixin delivery, and the open identity-less receipt PR does not cover this valid-context local-messageId long-message/chunking case.

## What This Changes

The issue asks OpenClaw and the external Weixin plugin to stop reporting proactive/direct Weixin sends as successful when long or rapid messages are missing or replaced by a Weixin fallback.

## Best Possible Solution

Keep this as the canonical Weixin ambiguous-delivery report: fix context-token, text/rate limits, and fail-closed behavior in the external plugin, and extend OpenClaw’s delivery-status contract only if the plugin cannot faithfully express Tencent acceptance semantics today.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the generic status-mapping weakness: current main treats a non-empty plugin result as sent/delivered without a distinct accepted or unknown state.

## Solution Assessment

Unclear: a plugin-side fail-closed and limits fix may be enough if Tencent exposes reliable failure signals, but OpenClaw currently lacks an accepted/unknown status for genuinely ambiguous downstream responses.
