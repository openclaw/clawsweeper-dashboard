## Summary

Keep open: current main still cannot distinguish a Weixin plugin result that only proves local/platform acceptance from actual user-visible delivery, and this issue adds a current valid-context-token long-message/chunking failure mode that the older closed reports do not fully cover.

## What This Changes

The issue asks OpenClaw and the Weixin plugin path not to report proactive Weixin sends as definitively delivered when the downstream service may only have accepted, dropped, or fallback-handled the send.

## Best Possible Solution

Keep this as the canonical current report, have the Weixin plugin fail closed when context is missing or ambiguous and enforce/document text and rate limits, then add a generic accepted/unknown delivery seam only if the existing receipt contract cannot represent Tencent semantics.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the status-mapping concern: non-empty send results can become successful direct delivery and cron delivered state.

## Solution Assessment

Unclear: a plugin-side fix may be enough if Tencent exposes reliable failure or acceptance signals, but genuinely ambiguous downstream responses may need a core SDK/status extension rather than more local cron heuristics.
