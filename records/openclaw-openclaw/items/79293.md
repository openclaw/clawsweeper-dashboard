## Summary

Keep open: current main has a generic empty-receipt accounting fix, but it still does not solve Weixin's final user-visible delivery contract. The latest external Weixin plugin still returns a locally generated client ID after `sendmessage` accepts the HTTP response, and recent comments report active production drops from isolated cron/message-tool sends.

## What This Changes

The issue asks OpenClaw and the external Weixin plugin to stop reporting final success for proactive/direct sends when only local client IDs or ambiguous iLink acceptance exist, and to expose honest delivery limits and status.

## Best Possible Solution

Keep this as the canonical Weixin delivery-status thread and define a core/plugin contract where Weixin can report failed, accepted/unknown, or pending instead of final delivered when user-visible receipt is not confirmed.

## Reproduction Assessment

No high-confidence current-main live reproduction was run here. Source inspection and the latest comments support the false-confidence path, but proving the user-visible Weixin fallback still needs live iLink/Weixin validation.

## Solution Assessment

Unclear as a concrete implementation: fixing empty receipts on main was only a partial slice. The maintainable solution is a maintainer-approved core/plugin delivery-status contract plus external-plugin behavior and live proof.
