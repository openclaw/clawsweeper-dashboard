## Summary

Keep open: current main still turns successful plugin send returns into sent/delivered cron state, while this report adds current valid-context-token long-message and chunking failures that are not fully covered by the earlier closed context-token reports.

## What This Changes

The issue requests that Weixin proactive/direct sends fail or report unknown/accepted instead of delivered when context tokens, long text/chunks, or Tencent responses do not prove user-visible delivery.

## Best Possible Solution

Keep this as the canonical current report, have the Weixin plugin map missing or ambiguous Tencent delivery to failure or unknown, and add a generic accepted/unknown receipt contract only if existing deliveryAttempted/unknown state cannot represent the transport semantics.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the Weixin plugin and credentials are external. Source inspection does reproduce the status-mapping risk: non-empty send results become sent/delivered, while unknown is only used when delivered is omitted.

## Solution Assessment

Unclear: a plugin-side fail-closed or unknown mapping is the narrowest fix if Tencent exposes reliable signals, but genuinely ambiguous 200 + {} responses likely need a small core SDK/status contract extension.
