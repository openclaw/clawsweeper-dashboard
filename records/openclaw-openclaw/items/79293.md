## Summary

Keep open: the current repo still treats a non-empty plugin send result as successful delivery, while the reported Weixin failure needs a maintainer decision across the external plugin boundary and core delivery-status semantics.

## What This Changes

The issue requests safer Weixin proactive-send semantics so OpenClaw does not report final delivery when the user sees no message, 请稍后再试, or missing chunks.

## Best Possible Solution

Define the delivery-state boundary explicitly: keep Weixin transport limits and context-token enforcement in @tencent-weixin/openclaw-weixin, and add any missing generic core/plugin status API only if plugins need a first-class way to report accepted or unknown outcomes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the core status mapping: a plugin result with any identity can become send success and cron delivered; live Weixin user-visible failure still needs the external plugin and Tencent runtime to confirm end to end.

## Solution Assessment

Is this the best way to solve the issue? Unclear: fail-closing missing context tokens and adding plugin-owned text/rate limits are narrow, but changing delivered versus accepted/unknown semantics should be designed as a generic plugin contract rather than a Weixin-only core branch.
