## Summary

Keep open: current main still treats returned outbound send results as successful delivery evidence, while the report describes a credible current Weixin path where plugin success is not the same as user-visible delivery.

## What This Changes

The issue reports that `openclaw-weixin` proactive/direct sends can return success and a local message id even when long or rapid sends are missing or surface `请稍后再试` to the user.

## Best Possible Solution

Keep this as the canonical current report, have the Weixin plugin fail closed and document/enforce transport limits, and extend the generic delivery contract only if the existing receipt surface cannot represent Tencent's ambiguous acceptance semantics.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the core status-mapping part: non-empty outbound results become successful/delivered state.

## Solution Assessment

Unclear: the narrowest maintainable fix may be plugin-side if Tencent exposes reliable failure or limit signals. If responses are inherently ambiguous, OpenClaw likely needs a generic accepted/unknown delivery-status seam rather than a Weixin-specific core special case.
