## Summary

Keep open: current main and the current Weixin plugin still treat a non-empty local result/messageId as sent, while this report covers valid-context downstream fallback and chunk loss that the open empty-receipt PR does not address.

## What This Changes

Report and fix Weixin proactive/direct sends that return OpenClaw success and a local messageId while the user sees no message, 请稍后再试, or missing chunks.

## Best Possible Solution

Keep this as the canonical Weixin ambiguous-delivery report: fix context-token, payload/rate limits, and fail-closed behavior in the external plugin, and extend OpenClaw’s delivery-status contract only if the plugin cannot faithfully express Tencent acceptance semantics today.

## Reproduction Assessment

No high-confidence live reproduction is available from this checkout because Weixin credentials and the external service are absent. Source and package inspection do reproduce the contract weakness: a generated local client id is enough for OpenClaw to record a sent result, with no accepted/unknown distinction.

## Solution Assessment

Unclear: a plugin-side fail-closed, size-limit, and throttling fix may be enough if Tencent exposes reliable failure signals. If downstream responses are genuinely ambiguous, the narrow maintainable core fix is a versioned delivery-status/receipt contract extension rather than another Weixin-specific core branch.
