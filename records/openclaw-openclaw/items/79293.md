## Summary

Keep open: current main fixed the generic empty-receipt false-success slice, but it still has no Weixin-specific or generic confirmed/accepted/pending delivery contract for valid-looking adapter results that later fail user-visible delivery.

## What This Changes

The issue asks OpenClaw and the external Weixin plugin to stop treating ambiguous proactive Weixin sends as final delivery and to define context-token, size/rate, retry, and delivery-status semantics.

## Best Possible Solution

Keep this as the canonical Weixin proactive-delivery thread, then define the smallest core delivery-state/API seam and plugin-owned Weixin context-token, limit, retry, or pending-queue behavior with live Weixin proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully: the source path still shows coarse delivery semantics and the reporter supplied production logs, but a fresh live Weixin/iLink run is still needed for the exact `请稍后再试` and missing-chunk behavior.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a single implementation path; the best fix likely needs a maintainer decision on core delivery states plus plugin-owned Weixin fail-closed, pending, rate-limit, or retry semantics.
