## Summary

Keep open: current main still maps any non-empty plugin send result to successful delivery, while the report describes a credible Weixin case where that result is not user-visible and the external plugin is not part of this repo.

## What This Changes

The issue asks Weixin proactive/direct sends to fail or report weaker status when context, size, rate, or downstream acceptance is ambiguous.

## Best Possible Solution

Keep this as the canonical current report, have the Weixin plugin owner fail closed without context and enforce/document transport limits, and add a generic accepted/unknown delivery seam only if the existing receipt contract cannot represent Tencent's response semantics.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the reported status-mapping problem: core and cron treat non-empty plugin send results as delivered.

## Solution Assessment

Unclear: a plugin-side fix may be enough if Tencent exposes reliable failure/acceptance signals, but a core SDK/status extension may be needed for genuinely ambiguous downstream responses.
