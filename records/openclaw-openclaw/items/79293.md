## Summary

Keep open: the specific valid-context Weixin false-success path is not implemented on current main, and the open generic fix PR only addresses identity-less/empty outbound results, not a plugin returning a local client message id after an ambiguous Tencent `sendmessage` response.

## What This Changes

The issue asks OpenClaw and the external Weixin plugin to avoid reporting ambiguous proactive/direct Weixin sends as final success and to document or enforce safe text/rate limits.

## Best Possible Solution

Keep this as the canonical Weixin ambiguous-delivery status report: fix context-token, body-level send validation, text/rate limits, and fail-closed behavior in the external plugin, and extend the generic OpenClaw delivery-status contract only if the plugin cannot faithfully express Tencent acceptance semantics today.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because it requires the external Weixin service and credentials. Source inspection does show the status-mapping weakness: current main has no accepted/unknown result state, while the current external plugin returns a generated client id after HTTP 2xx without proving user-visible delivery.

## Solution Assessment

Unclear: plugin-side fail-closed response parsing and conservative limits may be enough if Tencent exposes reliable failure signals, but current OpenClaw types do not provide a non-final accepted/unknown state for genuinely ambiguous downstream responses.
