## Summary

Keep open: current main still treats non-empty channel send results as successful delivery, while the report adds a current valid-context Weixin failure mode that is not fully covered by the older closed Weixin reports.

## What This Changes

The issue asks OpenClaw and the external Weixin plugin to stop reporting proactive Weixin sends as final success when long or rapid sends may show 请稍后再试, drop chunks, or only have ambiguous downstream acceptance.

## Best Possible Solution

Keep this as the canonical current report, have the Weixin plugin fail closed or report ambiguity for unsafe proactive sends, and add a generic accepted/unknown delivery seam only if existing plugin hooks cannot represent Tencent's response semantics.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the status-mapping concern: current main turns non-empty plugin send results into sent/delivered outcomes.

## Solution Assessment

Unclear: a plugin-side fix is likely best if Tencent exposes reliable failure or ambiguity signals, but a small core SDK/status extension may be needed if the existing result contract cannot represent accepted-but-unconfirmed delivery.
