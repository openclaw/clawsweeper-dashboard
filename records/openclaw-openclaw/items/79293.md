## Summary

Keep open: current main still cannot represent the reported ambiguous Weixin success state through the outbound send result, and cron still records non-empty successful sends as delivered. Earlier related reports are closed, and no newer canonical item or PR owns this valid-context-token long-message/chunking failure mode.

## What This Changes

The issue reports that openclaw-weixin proactive/direct sends with a valid contextToken can return success and a local messageId while long or rapid sends are missing or show 请稍后再试, and asks for truthful delivery status plus transport limits.

## Best Possible Solution

Keep this as the canonical current report: have the Weixin plugin fail closed without usable context, enforce or document payload and rate limits, and extend OpenClaw's generic receipt/status contract only if the existing lifecycle cannot represent ambiguous downstream acceptance.

## Reproduction Assessment

No, there is no high-confidence live reproduction path from this checkout because the external Weixin plugin, account, and Tencent credentials are absent. Source inspection does reproduce the core status-mapping concern: non-empty plugin send results still become sent/delivered in current main.

## Solution Assessment

Unclear: a plugin-side fail-closed and transport-limit fix may be enough if Tencent exposes reliable error or acceptance signals. If Tencent only returns ambiguous 200 + local/client ids, the safer maintainable path is a generic accepted/unknown receipt/status extension rather than more Weixin-specific core policy.
