## Summary

Keep open: current main and v2026.5.28 still convert plugin-returned delivery results into `sent`/success, while the latest external Weixin plugin still returns a generated client id after an HTTP 2xx from `sendmessage`. The remaining work needs Weixin plugin owner input and likely a maintainer decision on whether OpenClaw needs a generic accepted/unknown delivery contract.

## What This Changes

The issue reports Weixin proactive/direct sends returning success and local message ids while the user sees missing chunks or the generic Weixin fallback message.

## Best Possible Solution

Define the durable-send contract for ambiguous Weixin sends, then implement Weixin-owned context-token, size/rate, downstream-response, and user-visible fallback handling in the external plugin or through a generic SDK seam if current contracts are insufficient.

## Reproduction Assessment

No live high-confidence reproduction was run against Weixin. Source inspection does show the reported status-mapping risk: a plugin return with an identity is recorded as sent/succeeded without a downstream-visible state.

## Solution Assessment

Unclear. The maintainable direction is likely not a one-off OpenClaw core special case, but either a Weixin plugin fix using existing receipt/failure semantics or a maintainer-approved generic accepted/unknown contract if the SDK lacks one.
