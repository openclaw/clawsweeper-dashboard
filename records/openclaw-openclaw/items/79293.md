## Summary

Keep open: current main still lacks a Weixin-specific or generic delivery contract that can distinguish user-visible delivery from ambiguous Tencent/iLink acceptance, and related later items were closed back to this issue as the canonical thread.

## What This Changes

The issue asks OpenClaw and the external Weixin plugin to stop reporting proactive Weixin sends as final success when the user sees missing chunks or `请稍后再试`.

## Best Possible Solution

Keep this as the canonical Weixin proactive-delivery thread, then have maintainers choose the generic SDK/core delivery-state seam and the plugin-owned fail-closed, limit, retry, or pending-queue behavior with live Weixin proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows current main still has coarse delivered/sent accounting, and the reporter provided production steps and logs, but I did not establish a fresh live Weixin/iLink reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an implementation path: the problem is valid, but maintainers still need to choose whether the best fix is plugin-owned fail-closed behavior, generic accepted/unknown statuses, or a pending recipient-window queue.
