## Summary

Keep open: current main still lacks a generic way for a channel plugin to report ambiguous accepted/unknown delivery, while this issue describes a valid-token Weixin path where a local message id can still be user-visible failure. The open generic fix PR only covers identity-less/empty outbound receipts, not this local-messageId long-message and chunking case.

## What This Changes

The issue asks OpenClaw and openclaw-weixin to stop reporting ambiguous proactive/direct Weixin sends as final success and to expose safe limits or status for long and chunked messages.

## Best Possible Solution

Keep this as the canonical Weixin ambiguous-delivery report: fix context-token, text/rate limits, and fail-closed behavior in the external plugin, and extend the OpenClaw delivery-status contract only if the plugin cannot faithfully express Tencent acceptance semantics today.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the generic status-mapping weakness: current main treats a non-empty plugin result as sent/delivered without a distinct accepted or unknown state.

## Solution Assessment

Unclear: a plugin-side fail-closed and limits fix may be enough if Tencent exposes reliable failure signals, but current OpenClaw types do not offer an accepted/unknown status for genuinely ambiguous downstream responses.
