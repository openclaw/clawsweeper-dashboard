## Summary

Keep open: current main still cannot represent an ambiguous Weixin send as accepted/unknown when a plugin returns a non-empty local message id, and the open identity-less receipt fix does not cover this valid-token long-message/chunking case.

## What This Changes

The issue asks OpenClaw and the external Weixin channel plugin not to report ambiguous proactive sends as final success when users see fallback text or missing chunks.

## Best Possible Solution

Keep this as the canonical Weixin ambiguous-delivery report: fix token, text/rate limit, and fail-closed behavior in the external plugin, and extend the OpenClaw delivery-status contract only if plugin-owned Tencent acceptance semantics cannot be expressed today.

## Reproduction Assessment

No live high-confidence reproduction is available from this checkout because the external Weixin plugin and credentials are absent. Source inspection does reproduce the generic status-contract gap: current main treats a non-empty plugin result as sent/delivered without a distinct accepted or unknown state.

## Solution Assessment

Unclear: plugin-side fail-closed handling and documented size/rate limits may be enough if Tencent exposes reliable failure signals, but current OpenClaw types do not offer an accepted/unknown outcome for genuinely ambiguous downstream responses.
