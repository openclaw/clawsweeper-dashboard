## Summary

Keep open: the PR targets a valid middleware bug, but the patch is not merge-ready because the coercer now accepts invalid middleware `details` by dropping them, which breaks the existing fail-closed contract. The supplied proof is also still a copied Node probe rather than the patched OpenClaw runner or a live Codex/message path.

## What This Changes

The PR adds private tool-result middleware coercion for nested `toolResult`/`tool_result`/function-like content blocks into bounded text plus focused regression tests.

## Best Possible Solution

Land a bounded content-block coercion that salvages recognized nested tool-result blocks while preserving the existing fail-closed behavior for invalid middleware `details`, backed by patched-runner and Codex dynamic-tool proof.

## Reproduction Assessment

Yes. From source, current main rejects non-`text`/`image` nested `toolResult` content and returns the middleware failure result; I did not execute the live Codex/channel flow.

## Solution Assessment

No, not as written. The content coercion direction is narrow and appropriate, but the current implementation also weakens invalid-`details` fail-closed handling and needs patched-code proof before merge.

## Review Findings

- **[P2] Keep invalid middleware details fail-closed:** `src/agents/harness/tool-result-middleware.ts:222-225`
