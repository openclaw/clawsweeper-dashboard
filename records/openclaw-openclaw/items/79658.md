## Summary

Keep open: the PR is a plausible feature extension with real behavior proof, but the current diff has a concrete allowlist bug because MIME-level `text/plain` also admits existing `.log` files, beyond the advertised TXT/JSON/YAML/YML scope.

## What This Changes

This PR expands host-read local media sends to validated TXT, JSON, YAML, and YML documents, adds YAML MIME mappings, updates media/outbound tests, and adds a changelog entry.

## Best Possible Solution

Keep the validated-text direction, but make the host-read text allowlist extension-scoped so only the intended formats are admitted before maintainer policy review.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature PR. Source inspection shows current main rejects host-local `.txt` attachments and only permits CSV/Markdown through the validated-text host-read path.

## Solution Assessment

No, not as currently written: the approach is narrow, but MIME-level `text/plain` allowlisting also allows `.log`. The safer solution is an extension-scoped text-format allowlist for `.txt`, `.json`, `.yaml`, `.yml`, and the existing CSV/Markdown formats.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restrict text/plain host reads to .txt:** `src/media/web-media.ts:138`
  - body: Adding `text/plain` to the MIME-level host-read alias also accepts any extension that `mimeTypeFromFilePath` maps to `text/plain`; current main already maps `.log` that way. That silently expands host-read local sends beyond the advertised TXT/JSON/YAML/YML formats, so a text `.log` file would be sendable through this path. Gate the fallback by extension or keep a separate allowed text-extension set.
  - confidence: 0.88
