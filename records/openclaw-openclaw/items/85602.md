## Summary

Keep open: the diagnosis is useful, but this patch does not clear the compaction trigger it is meant to avoid. Returning `contextTokens` still leaves a fresh token snapshot above the compaction budget, so the claude-cli stall path can remain active, and the supplied proof is unit-test output rather than real claude-cli behavior.

## What This Changes

The PR changes `deriveSessionTotalTokens` to cap prompt-token snapshots at `contextTokens` and updates usage/session tests for that clamp.

## Best Possible Solution

Stop impossible claude-cli token snapshots from being treated as fresh compaction input, preferably by using the last real assistant sub-call usage or marking impossible totals stale/ignored for CLI compaction, with a regression test through the compaction gate and redacted real-behavior proof.

## Reproduction Assessment

Yes for source-level reproduction: with `totalTokensFresh=true` and `totalTokens=contextTokens`, current CLI compaction still sees a token count above `promptBudgetBeforeReserve`. I did not rerun the reporter's live claude-cli setup.

## Solution Assessment

No. Clamping to the full context size is not the best fix because the compaction gate reserves tokens below that size; the fix should avoid using the inflated snapshot as a fresh gate input.
