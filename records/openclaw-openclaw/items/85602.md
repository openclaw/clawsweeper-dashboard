## Summary

Keep open: the linked bug is valid, but the current clamp leaves a fresh context-sized token snapshot that still exceeds the compaction reserve budget, so the same compaction/stall path can still fire; proof remains unit-only.

## What This Changes

The branch adds a `contextTokens` clamp in `deriveSessionTotalTokens` and updates usage/session tests to expect the clamped value.

## Best Possible Solution

Stop inflated claude-cli result usage from becoming a fresh compaction input, preferably by using a valid last-assistant-call snapshot or marking impossible totals stale/ignored for CLI compaction, then cover the gate and add redacted live proof.

## Reproduction Assessment

Yes for source-level reproduction: a fresh `sessionEntry.totalTokens` equal to `contextTokens` still exceeds `promptBudgetBeforeReserve` when reserve tokens are positive, so `runCliTurnCompactionLifecycle` enters compaction even if the estimated prompt fits. I did not rerun a live claude-cli tool-heavy session.

## Solution Assessment

No. Clamping to the full context window is not sufficient because the compaction gate reserves tokens below that size; the fix should keep impossible claude-cli usage out of fresh compaction input or use a valid last-call snapshot.
