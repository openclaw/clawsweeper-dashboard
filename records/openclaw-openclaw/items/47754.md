## Summary

Keep this PR open: the reported snippet behavior is still source-reproducible in the active memory-core search path, but the branch edits a retired `src/memory` file and lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/memory/manager-search.ts` to return SQLite FTS5 `snippet()` output with clamped token count and ellipsis markers for keyword search instead of truncating selected row text.

## Best Possible Solution

Retarget the FTS5 snippet projection to the active memory-core keyword MATCH path, keep full `row.text` for scoring/fallbacks, add a deep-match regression test, and attach real CLI/search proof.

## Reproduction Assessment

Yes, from source inspection: a long FTS row with the query term after the first 700 characters would still be truncated by the active `row.text` snippet path. I did not run a live repro in this read-only review.

## Solution Assessment

No. FTS5 `snippet()` with a 1-64 token clamp is a reasonable direction, but this patch targets a retired path; the maintainable fix belongs in `extensions/memory-core/src/memory/manager-search.ts` while preserving full text for ranking.
