## Summary

Current main and the latest release still truncate keyword-search results from the beginning of the matched chunk, so this PR’s central fix remains necessary. The submitted branch is now conflict-dirty and predates the memory-core move and later FTS fallback/tokenizer work, so it needs a narrow current-main refresh plus real behavior proof rather than cleanup closure.

## What This Changes

The PR changes builtin FTS keyword results to use SQLite `snippet()` around matched terms instead of truncating each matched chunk from its beginning.

## Best Possible Solution

Implement match-centered excerpts in the current memory-core keyword MATCH path, retain a deterministic match-centered fallback for LIKE-only paths, preserve full chunk text for ranking, and cover deep matches across current tokenizer/hybrid behavior.

## Reproduction Assessment

Yes, from current source with high confidence: index a chunk longer than 700 characters with the query term after character 700, then run keyword or hybrid memory search; the current result truncates the chunk before the matching term.

## Solution Assessment

Yes in direction, but not as the stale branch stands: FTS5 `snippet()` is the narrowest match-aware solution for successful MATCH queries, while the current LIKE fallback paths need equivalent handling without losing ranking input.
