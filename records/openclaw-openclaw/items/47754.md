## Summary

Keep this PR open, but it should not merge as written: the branch applies the FTS5 snippet idea to the retired `src/memory` search file while current `main` runs keyword search from `extensions/memory-core`, where the exact late-in-row truncation path still exists. The PR also lacks after-fix real behavior proof, so it needs contributor or maintainer follow-up rather than cleanup closure.

## What This Changes

The branch modifies `src/memory/manager-search.ts` to return SQLite FTS5 `snippet()` output with a clamped ellipsis-marked token window instead of truncated row text.

## Best Possible Solution

Retarget the snippet projection to the active memory-core keyword MATCH path, keep full `row.text` available for fallback ranking, and add a focused deep-match regression test plus real CLI/search proof.

## Reproduction Assessment

Yes, from source inspection: the active MATCH path selects `text` and returns `truncateUtf16Safe(row.text, params.snippetMaxChars)`, so a match after the visible prefix of a large FTS row can be hidden. I did not run a live repro because this review is read-only.

## Solution Assessment

No. FTS5 `snippet()` is a reasonable direction, but this patch applies it to a retired path; the maintainable fix belongs in `extensions/memory-core/src/memory/manager-search.ts` with full text retained separately for scoring.
