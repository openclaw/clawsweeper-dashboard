## Summary

Keep open: the FTS5-centered keyword snippet behavior is still useful and not on current main, but this branch cannot merge as-is because it edits the retired memory-search path and lacks after-fix real behavior proof.

## What This Changes

The PR changes keyword memory search to return SQLite FTS5 `snippet()` output with ellipsis markers and a clamped token count instead of truncating the FTS row text from the beginning.

## Best Possible Solution

Port the `snippet()` extraction to `extensions/memory-core/src/memory/manager-search.ts`, preserve full text for fallback scoring where needed, clamp snippet tokens to SQLite's limit, and add a regression test for a deep keyword match inside one FTS row.

## Reproduction Assessment

Yes, from source inspection. The active keyword MATCH query still selects `text` and returns `truncateUtf16Safe(row.text, ...)`, so a match deep inside one long FTS row can be hidden from the visible snippet.

## Solution Assessment

No. The `snippet()` approach is directionally right, but this PR edits a retired path; the maintainable fix is to port it to the active memory-core implementation with focused coverage.
