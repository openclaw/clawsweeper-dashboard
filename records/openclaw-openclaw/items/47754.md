## Summary

Keep open: the SQLite FTS5 `snippet()` direction is valid, but this PR edits the legacy `src/memory` helper while current main routes user-facing memory search through `memory-core`, where MATCH results still truncate from the start of `row.text`.

## What This Changes

The PR changes legacy memory keyword search to return SQLite FTS5 `snippet()` output with clamped token count and visible ellipsis markers instead of truncating the full row text prefix.

## Best Possible Solution

Port the snippet behavior into `extensions/memory-core/src/memory/manager-search.ts`, using SQLite FTS5 `snippet()` only for MATCH-backed rows, clamping tokens to 1..64, keeping visible omission markers, preserving fallback search paths, and adding regression coverage plus changelog.

## Reproduction Assessment

Yes. The linked bug gives a concrete CLI workflow, and current main source confirms keyword results still return `truncateUtf16Safe(row.text, 700)`, so an indexed chunk with a unique term after the prefix window reproduces the hidden-snippet behavior.

## Solution Assessment

No. The SQL approach is the right narrow fix, but this PR applies it to an obsolete path; the maintainable solution is the same behavior in the active memory-core helper with tests for deep matches and fallback behavior.
