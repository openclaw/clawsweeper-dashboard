## Summary

Keep open: the reported memory-search snippet bug still exists on current main, and this PR is directionally useful but edits the obsolete legacy memory path instead of the active memory-core plugin path.

## What This Changes

The PR changes legacy `src/memory/manager-search.ts` to return SQLite FTS5 `snippet()` output with clamped token count and visible ellipsis markers for keyword search results.

## Best Possible Solution

Port the snippet behavior into `extensions/memory-core/src/memory/manager-search.ts`, use FTS5 `snippet()` only for MATCH-backed rows with 1..64 token clamping and visible omission markers, preserve fallback paths, and add regression coverage plus changelog.

## Reproduction Assessment

Yes. #47730 gives a concrete CLI workflow, and current main confirms keyword results still truncate `row.text` from the start; an indexed chunk with a unique term after the first 700 characters reproduces the hidden-match snippet.

## Solution Assessment

No. SQLite `snippet()` is the right narrow mechanism for MATCH-backed keyword rows, but this PR applies it to an obsolete path; the maintainable fix belongs in memory-core with tests for deep matches and fallback behavior.
