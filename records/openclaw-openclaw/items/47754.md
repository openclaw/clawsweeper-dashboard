## Summary

Keep open: the PR is a plausible fix direction, but the branch edits retired `src/memory/manager-search.ts` while current `main` runs the memory search implementation from `extensions/memory-core/src/memory/manager-search.ts`. The active keyword path still truncates the selected FTS row text, so the useful change needs to be retargeted and proven before merge.

## What This Changes

The PR changes keyword memory search to return SQLite FTS5 `snippet()` output with a clamped ellipsis-marked token window instead of truncating the selected FTS row text.

## Best Possible Solution

Port the snippet extraction to `extensions/memory-core/src/memory/manager-search.ts`, keep `row.text` available for fallback scoring, clamp snippet tokens to SQLite’s 64-token limit, and add a focused deep-match regression test.

## Reproduction Assessment

Yes, from source inspection. The active MATCH path selects `text` and returns `truncateUtf16Safe(row.text, params.snippetMaxChars)`, so a match late in a configured large FTS row can be hidden from the visible snippet.

## Solution Assessment

No. The FTS5 `snippet()` approach is directionally right, but this PR modifies a retired path; the maintainable fix is a memory-core patch that preserves full text for ranking and covers the behavior.
