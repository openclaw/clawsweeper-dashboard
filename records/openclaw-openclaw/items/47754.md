## Summary

Keep open. The specific FTS-centered keyword snippet change is not on current main, and current source still truncates matched chunk text from the beginning; however, this PR is stale against the memory-core plugin move and needs real behavior proof before merge.

## What This Changes

The PR changes keyword memory search to select SQLite FTS5 `snippet()` output, clamp snippet tokens to 64, use ellipsis markers, and return that snippet instead of truncating full chunk text.

## Best Possible Solution

Rebase the change onto `extensions/memory-core/src/memory/manager-search.ts`, keep it within SQLite FTS5's 64-token `snippet()` contract, and add focused proof that a deep keyword match appears in the returned snippet.

## Reproduction Assessment

Yes. Source inspection shows default chunks can exceed the 700-character snippet limit and current keyword search still truncates `row.text` from the beginning, so a match later in the chunk can be omitted without running the CLI flow.

## Solution Assessment

No, not as-is. FTS5 `snippet()` is a good narrow primitive for the successful keyword MATCH path, but the patch targets the pre-plugin path and needs current-file coverage plus real behavior proof.
