## Summary

Keep open. The FTS5 snippet approach is useful source work, but the PR currently edits the removed legacy memory helper while current main routes memory search through memory-core, where MATCH-backed keyword rows still select full text and truncate from the beginning.

## What This Changes

The PR changes legacy `src/memory/manager-search.ts` keyword search to return SQLite FTS5 `snippet()` output with clamped token count and visible ellipsis markers instead of truncating the row text prefix.

## Best Possible Solution

Port the snippet behavior into `extensions/memory-core/src/memory/manager-search.ts`, using SQLite FTS5 `snippet()` only for MATCH-backed rows, clamping token count to the 1..64 contract, keeping visible omission markers, preserving fallback search/scoring behavior, and adding a regression test for a deep match beyond the old prefix window.

## Reproduction Assessment

Yes. The linked bug gives a concrete memory CLI workflow, and current main source confirms MATCH-backed keyword results still return `truncateUtf16Safe(row.text, 700)`, so an indexed chunk with a unique term after the first 700 characters reproduces the hidden-snippet behavior.

## Solution Assessment

No. The PR's SQLite snippet direction is the right narrow fix, but applying it to `src/memory/manager-search.ts` no longer solves current main; the maintainable fix belongs in the active memory-core helper with fallback-preserving tests.
