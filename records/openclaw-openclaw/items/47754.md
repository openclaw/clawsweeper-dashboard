## Summary

Keep open: the FTS5 snippet direction is useful, and current main still does not use `snippet()` for active keyword MATCH results, but this PR is not mergeable as written because it targets the retired `src/memory` path and lacks contributor-supplied real behavior proof.

## What This Changes

The PR changes keyword memory search to return SQLite FTS5 `snippet()` output with clamped ellipsis-marked token windows instead of truncating the selected FTS row text.

## Best Possible Solution

Port the `snippet()` extraction to `extensions/memory-core/src/memory/manager-search.ts`, keep full text available for fallback ranking, clamp the token count to SQLite's limit, and add a regression test for a deep keyword match inside one FTS row.

## Reproduction Assessment

Yes, from source inspection. The active keyword MATCH path still selects `text` and returns `truncateUtf16Safe(row.text, ...)`, so a match deep inside one long FTS row can be hidden from the visible snippet.

## Solution Assessment

No. The `snippet()` approach is directionally right, but this PR edits a retired path; the maintainable fix is to retarget the active memory-core implementation with focused coverage and proof.
