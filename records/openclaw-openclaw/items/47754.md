## Summary

Keep open: the reported memory keyword snippet problem is source-reproducible on the active path, but this PR edits the retired core file and lacks required after-fix real behavior proof.

## What This Changes

The PR changes keyword memory search to return SQLite FTS5 `snippet()` output instead of prefix-truncating matched chunk text.

## Best Possible Solution

Port the snippet projection to `extensions/memory-core/src/memory/manager-search.ts`, retain full text for ranking, add a deep keyword-match regression test, and include redacted real CLI/search proof.

## Reproduction Assessment

Yes, from source inspection: insert or index an FTS row whose match occurs after `snippetMaxChars`, then the active MATCH path still returns `truncateUtf16Safe(row.text, params.snippetMaxChars)` from the row start. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. FTS5 `snippet()` is a plausible direction under the SQLite contract, but this branch targets a retired path; the maintainable fix must retarget active memory-core and preserve full text separately for scoring. ([sqlite.org](https://www.sqlite.org/fts5.html?utm_source=openai))
