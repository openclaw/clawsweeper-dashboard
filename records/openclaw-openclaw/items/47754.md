## Summary

Keep open: this is a focused bug-fix PR for memory keyword snippets, and current builtin keyword search still has a source-reproducible prefix-truncation gap. The branch should not merge as-is because it targets the retired `src/memory` path, is conflicting, and lacks contributor real behavior proof.

## What This Changes

The PR changes memory keyword search SQL to return SQLite FTS5 `snippet()` output from `src/memory/manager-search.ts` instead of truncating the returned row text from the beginning.

## Best Possible Solution

Port the snippet projection to `extensions/memory-core/src/memory/manager-search.ts`, keep full text for ranking, add a deep keyword-match regression test, and include redacted real CLI/search output as proof.

## Reproduction Assessment

Yes, from source inspection: insert or index an FTS row whose match occurs after the first `SNIPPET_MAX_CHARS` characters, then the active MATCH path returns `truncateUtf16Safe(row.text, params.snippetMaxChars)` from the row start. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. FTS5 `snippet()` is a plausible fix direction, but this branch changes a retired path; the maintainable solution must target active memory-core and keep full row text available separately for scoring.
