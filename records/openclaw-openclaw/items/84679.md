## Summary

Keep open: the PR targets the right P1 heartbeat bug, but it is not merge-ready because the new target-none bypass can clear unrelated durable pending delivery state in a shared session, and the external PR has no after-fix real behavior proof.

## What This Changes

The branch adds a heartbeat-only getReply option, passes it for `target: "none"` heartbeats, clears pending-final-delivery fields before a fresh heartbeat, and adds fast-path regression coverage.

## Best Possible Solution

Keep the target-none heartbeat recovery, but make pending delivery ownership explicit so internal heartbeats cannot erase visible durable delivery state.

## Reproduction Assessment

Yes. Source inspection shows the PR's skip flag clears pending-final-delivery state for any `target: "none"` heartbeat, while current main uses those fields for durable final-delivery recovery; I did not run a live Docker/GLM repro.

## Solution Assessment

No. Bypassing stale replay is the right direction for heartbeat-owned target-none state, but clearing any pending final delivery based only on the current heartbeat target is not the safest maintainable fix.
