## Summary

Keep open: current main does not implement this feature, and the contribution addresses a real observability need, but it is not merge-ready because the PR exports raw internal session keys into outbound provider headers by default and overlaps open #54985's separate gateway-context header contract.

## What This Changes

The PR adds optional `diagnostics.sessionTracing` config/schema types plus a stream wrapper that injects configurable session-id and session-name headers into embedded LLM requests.

## Best Possible Solution

Settle on one privacy-safe request-correlation contract that uses non-sensitive run/model-call identifiers or explicit operator-provided values, preserves trusted `traceparent` behavior, documents the config and privacy implications, adds changelog coverage, and resolves the overlap with #54985 before merge.

## Reproduction Assessment

Not applicable as a bug report; this is a feature PR. The P2 finding has a high-confidence source-level reproduction: enable `diagnostics.sessionTracing` without `sessionName` and `resolveSessionTracingHeaders` returns raw `sessionKey` values, which the PR test also asserts.

## Solution Assessment

No, not as-is. The core need is valid, but the current approach should avoid raw session-key defaults and be reconciled with the existing diagnostics model-call surface and #54985's gateway header proposal.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Avoid exporting raw session keys by default:** `src/agents/observability-session-headers.ts:39-40`
  - body: When tracing is enabled and `sessionName` is omitted, this sends the raw internal `sessionKey` as `x-session-name`; when `runId` is absent, line 36 also sends it as `x-session-id`.   - confidence: 0.78
