## Summary

Keep open. The request-correlation feature is useful, but this PR still sends raw internal session keys across the provider boundary by default, imports an obsolete Pi package namespace, lacks docs for the new privacy-sensitive config, and still has no inspectable real behavior proof.

## What This Changes

The PR adds an opt-in `diagnostics.sessionTracing` config block, schema/types/tests, and a stream wrapper that injects configurable session-id/name headers into outgoing LLM provider requests.

## Best Possible Solution

Define one documented, privacy-safe LLM request-correlation contract that is explicit opt-in or trusted-target scoped, uses bounded non-sensitive identifiers by default, and aligns generic config with provider-owned hooks.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. Yes for the blocking findings: PR-head source inspection shows raw session-key defaults, an obsolete Pi namespace import, and missing docs for the new config surface.

## Solution Assessment

No, not as-is. The request-correlation direction is useful, but the implementation should avoid raw session-key defaults, use the current package namespace, document the privacy/config contract, and coordinate with the overlapping provider metadata proposals.
