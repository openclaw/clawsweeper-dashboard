## Summary

Keep open: this feature PR is not implemented on current main and still has blocking privacy/API, documentation, namespace, and real-behavior-proof gaps before merge.

## What This Changes

Adds optional `diagnostics.sessionTracing` config/schema support plus a stream wrapper that injects configurable session-id and session-name headers into embedded LLM provider requests.

## Best Possible Solution

Define one opt-in or trusted-target request-correlation contract using non-sensitive run/model-call identifiers or explicit operator values, document the privacy implications, and reconcile the overlapping Gateway header PR before merge.

## Reproduction Assessment

Not applicable as a current-main bug; this is a feature PR. The blocking privacy finding is source-reproducible because PR head returns raw `sessionKey` values in outbound headers when tracing is enabled without explicit safe values.

## Solution Assessment

No, not as-is. The request-correlation goal is useful, but raw session-key defaults, missing public docs, stale package namespace, and overlap with the open Gateway header proposal make this the wrong merge shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid exporting raw session keys by default:** `src/agents/observability-session-headers.ts:36-40`
  - body: When tracing is enabled and `sessionName` is omitted, this sends the raw internal `sessionKey` as `x-session-name`; when `runId` is absent, it also sends it as `x-session-id`. Current session keys can include channel, account, peer kind, and peer id components, while the existing OTEL privacy contract avoids exporting session keys. Require explicit non-sensitive values or derive bounded/hash identifiers before sending these headers out.
  - confidence: 0.9
- **[P3] Document the public tracing config before merge:** `src/config/zod-schema.ts:310-323`
  - body: This adds a user-facing `diagnostics.sessionTracing` config key, but the configuration reference, OpenTelemetry/privacy docs, and schema help/labels do not explain what data leaves the process or how operators should choose safe values. Add those docs before exposing the config surface.
  - confidence: 0.8
- **[P2] Use the current Pi package namespace:** `src/agents/observability-session-headers.ts:1`
  - body: Current main depends on and imports `@earendil-works/pi-agent-core`, but this new helper imports `@mariozechner/pi-agent-core`. After rebasing onto current main, this added file should use the current package namespace or it will not match the repository's Pi runtime surface.
  - confidence: 0.88
