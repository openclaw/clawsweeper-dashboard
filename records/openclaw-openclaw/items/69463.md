## Summary

Keep open, but this PR is not merge-ready: the request-correlation direction is useful, yet the current branch lacks inspectable real provider proof, uses a stale Pi package namespace against current main, and needs maintainer privacy/API judgment before exporting OpenClaw session identifiers as provider-bound headers.

## What This Changes

The PR adds a `diagnostics.sessionTracing` config block, schemas/types/tests, and an embedded-runner stream wrapper that injects configurable session-id and session-name headers into outbound LLM provider requests.

## Best Possible Solution

Define one documented, opt-in or trusted-target LLM request-correlation contract with bounded non-sensitive identifiers by default, coordinated with https://github.com/openclaw/openclaw/issues/63901, then land it with docs, focused tests, and redacted downstream provider proof.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. The PR defects are source-reproducible from PR-head code and current-main package, session-key, telemetry, and docs contracts.

## Solution Assessment

No, not as-is. The request-correlation direction is useful, but the implementation should avoid raw session-key defaults, use the current Pi package namespace, document the privacy/config contract, and coordinate with the overlapping provider metadata proposals.
