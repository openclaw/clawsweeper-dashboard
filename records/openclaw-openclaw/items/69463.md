## Summary

Keep open: the request-correlation feature is useful, but the branch still sends raw internal session keys by default, imports the stale Pi package namespace, lacks docs for the new privacy-sensitive config surface, and has no inspectable real behavior proof.

## What This Changes

The PR adds an opt-in `diagnostics.sessionTracing` config block, schema/types/tests, and a stream wrapper that injects configurable session-id/name headers into outgoing LLM provider requests.

## Best Possible Solution

Define one documented, privacy-safe LLM request-correlation contract that is explicit opt-in or trusted-target scoped, uses bounded non-sensitive identifiers by default, and aligns generic config with provider-owned header hooks.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. Yes for the blocking findings: PR-head source inspection shows raw session-key defaults, a stale Pi namespace import, and missing docs for the new config surface.

## Solution Assessment

No, not as-is. The request-correlation direction is useful, but the implementation should avoid raw session-key defaults, use the current package namespace, document the privacy/config contract, and coordinate with the overlapping gateway-header proposals.

## Review Findings

Overall correctness: patch is incorrect
