## Summary

Keep open: the request is a useful observability feature, but current main does not implement this config surface and the PR still needs privacy/API review, real behavior proof, and rebase fixes before it is mergeable.

## What This Changes

The PR adds optional `diagnostics.sessionTracing` config/schema/types and a stream wrapper that injects configurable session-id and session-name headers into embedded LLM provider requests.

## Best Possible Solution

Define one privacy-safe, documented LLM request-correlation contract using explicit operator-provided values or bounded non-sensitive identifiers, and reconcile it with the related Gateway header proposal before merging.

## Reproduction Assessment

Not applicable as a current-main bug; this is a feature PR. Yes for the blocking review findings: source inspection of the PR diff shows raw `sessionKey` defaults and the stale Pi package namespace.

## Solution Assessment

No, not as-is. The request-correlation goal is valid, but the PR should not export raw session keys by default, should document the privacy/config contract, should use the current Pi namespace, and should be reconciled with the overlapping Gateway header proposal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid exporting raw session keys by default:** `src/agents/observability-session-headers.ts:36-40`
  - body: When tracing is enabled and `sessionName` is omitted, this sends the raw internal `sessionKey` as `x-session-name`; when `runId` is absent, it also sends it as `x-session-id`.   - confidence: 0.8
