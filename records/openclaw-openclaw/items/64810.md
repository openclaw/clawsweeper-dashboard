## Summary

Keep open. Current main still only skips heartbeat when the resolved session lane has queued work, while the active reply-run registry is not consulted; the issue is also paired with open, unmerged PR #64963.

## What This Changes

Prevent heartbeat or async system-event turns from preempting in-progress replies in Telegram topic sessions.

## Best Possible Solution

Land or cherry-pick #64963 with the active reply-run heartbeat guard, its regression test, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A focused reproduction is to seed a Telegram-backed session, enqueue a system event for that session, register an active reply run for the same session key, mock all lane queue sizes as zero, and call runHeartbeatOnce; current main has no guard for that state.

## Solution Assessment

Yes. Reusing resolveActiveReplyRunSessionId and returning the existing requests-in-flight skip is the narrowest maintainable fix because the wake layer already retries that busy reason.
