## Summary

Keep open. Current main still defers heartbeat for busy command/session lanes but does not consult the active reply-run registry, and the issue is paired with the open unmerged fix at https://github.com/openclaw/openclaw/pull/64963.

## What This Changes

The issue asks OpenClaw to prevent heartbeat and async system-event turns from preempting or swallowing active user-visible Telegram topic replies.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64963, or an equivalent active reply-run heartbeat guard with regression coverage, while tracking broader async targeting separately.

## Reproduction Assessment

Yes, from source. Seed a Telegram-backed session, enqueue a system event for that session, register an active reply run for the same session key, mock all lane queues as idle, and run runHeartbeatOnce; current main has no guard for that active reply-run state.

## Solution Assessment

Yes. Reusing resolveActiveReplyRunSessionId after heartbeat preflight resolves the canonical session key and returning the existing requests-in-flight skip is the narrowest maintainable fix for this preemption window.
