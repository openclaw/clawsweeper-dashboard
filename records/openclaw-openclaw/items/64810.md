## Label Justifications

- `P1`: The original report affected active Telegram topic conversations by losing or suppressing user-visible replies during in-flight work.
- `impact:message-loss`: The issue is about heartbeat or system-event turns causing a normal reply to disappear from the user's perspective.
- `impact:session-state`: The failure depended on session-lane and reply-run lifecycle state drifting apart during an active turn.

## Summary

Close as implemented: current main and the shipped v2026.5.16-beta.4+ line now skip heartbeat runs when the resolved session has an active reply run, with regression coverage for the idle-queue active-reply race; later stale replay/default-target reports remain separate.

## What This Changes

Prevent heartbeat or async system-event turns from preempting in-progress user replies in Telegram topic sessions.

## Best Possible Solution

Keep the shipped active-reply heartbeat guard and track any remaining stale replay or delivery-context leakage under the narrower open reports.

## Reproduction Assessment

Yes. The source-level reproduction is an idle resolved session lane with an active reply run; current main has a regression test for that exact state and now returns requests-in-flight instead of starting a heartbeat reply.

## Solution Assessment

Yes. The shipped guard uses the existing reply-run registry and retryable heartbeat skip reason before the lane queue check, which is the narrow maintainable fix for the active-reply race.
