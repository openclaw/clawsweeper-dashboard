## Label Justifications

- `P1`: The reported bug could interrupt active user-facing Telegram replies for affected deployments before the shipped fix.
- `impact:message-loss`: The issue is about an in-progress user-visible reply being swallowed or replaced by heartbeat/system output.
- `impact:session-state`: The failure involved heartbeat/system turns colliding with active session and reply-run lifecycle state.

## Summary

Current main and the latest stable release now implement the active reply-run heartbeat guard that this issue asked for; the newer stale replay/default-target reports are adjacent but already have separate open tracking issues.

## What This Changes

The issue reports that heartbeat or async system-event turns can preempt in-progress Telegram topic replies and leave the original user-visible answer lost.

## Best Possible Solution

Keep the shipped `v2026.5.18` active reply-run defer behavior, and use the narrower follow-up issues for stale replay or cross-target delivery leaks that are not the original active-reply preemption race.

## Reproduction Assessment

Yes. The source-level reproduction is an idle resolved session lane with an active reply run for the same session key; current main now has a regression test for that shape and skips instead of invoking the heartbeat reply path.

## Solution Assessment

Yes. The shipped fix checks the canonical reply-run registry immediately after heartbeat preflight resolves the session key and reuses the existing retryable `requests-in-flight` skip, which is the narrow maintainable solution for this race.
