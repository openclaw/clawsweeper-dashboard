## Summary

Keep open. Current main still has the queue-depth heartbeat guard but does not consult the active reply-run registry, and the issue is paired with an open unmerged closing PR that targets the exact narrow race.

## What This Changes

The issue reports Telegram topic replies being preempted or swallowed when heartbeat or async system-event turns run in the same active session.

## Best Possible Solution

Review, refresh or replace, and land https://github.com/openclaw/openclaw/pull/64963 for the active reply-run heartbeat guard, while keeping broader stale replay and delivery-context leaks on their narrower open reports.

## Reproduction Assessment

Yes, as a high-confidence source-level path: seed a Telegram-backed session, enqueue a system event for that session, register an active reply run for the same session key, force all queue sizes to zero, and call `runHeartbeatOnce`. Current main lacks the active reply-run guard that would skip that state.

## Solution Assessment

Yes for the narrow active-reply race: using `resolveActiveReplyRunSessionId(sessionKey)` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the smallest maintainable fix. The newer stale-output replay comments should stay separate unless the landed patch explicitly covers them.
