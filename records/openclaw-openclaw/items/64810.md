## Summary

Keep open. Current main still checks only the main command lane and resolved session-lane queue depth before running heartbeat, while the active reply-run registry exists but is not consulted; the issue also has an open, unmerged implementation candidate in #64963.

## What This Changes

The issue requests that Telegram topic heartbeat or async system-event turns avoid preempting in-progress user replies, or reliably resume/replay interrupted replies.

## Best Possible Solution

Review and land #64963, or apply the same focused change on current main: after heartbeat preflight resolves sessionKey, skip with requests-in-flight when resolveActiveReplyRunSessionId(sessionKey) is truthy, and keep the regression test for an active reply operation with otherwise idle queues. Broader async completion routing should stay tracked separately.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to seed a Telegram-backed session, enqueue a system event for that session, create a running reply operation for the same sessionKey, keep queue sizes at zero, and call runHeartbeatOnce; current main has no active-run guard for that case.

## Solution Assessment

Yes. The narrowest maintainable fix is the one proposed by #64963: reuse the existing reply-run registry and existing requests-in-flight retry path rather than adding a new routing lane or broader delivery policy in this bug fix.
