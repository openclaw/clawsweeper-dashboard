## Summary

Keep open. Current main still only defers heartbeat when the resolved session lane has queued or active command-lane work; it does not consult the active reply-run registry, and an open unmerged closing PR already targets this exact race.

## What This Changes

Prevent heartbeat or async system-event turns from preempting in-progress replies in Telegram topic sessions.

## Best Possible Solution

Land or supersede #64963 with the active reply-run heartbeat guard, regression test, and required user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is to seed a Telegram-backed session, enqueue a system event for that session, register a live ReplyOperation for the same session key, mock all lane queues as idle, and call runHeartbeatOnce; current main has no active reply-run guard for that state.

## Solution Assessment

Yes. Reusing resolveActiveReplyRunSessionId after heartbeat preflight resolves sessionKey and returning the existing requests-in-flight skip is the narrowest maintainable fix because heartbeat-wake already retries that busy reason.
