## Summary

Keep open. Current main still has the heartbeat runner’s lane-depth-only deferral, while the active reply-run registry seam exists but is not consulted; an open, unmerged closing PR already targets the exact gap.

## What This Changes

The issue reports Telegram topic replies being preempted by heartbeat or async system-event turns before the original user-visible reply completes.

## Best Possible Solution

Land or cherry-pick #64963, or an equivalent active reply-run heartbeat guard, with its regression test and a user-facing changelog entry; keep broader async completion routing work separate.

## Reproduction Assessment

Yes. A focused source-level reproduction is to seed a Telegram-backed session, enqueue a system event for that session, register an active reply run for the same session key, force all lane queue sizes to zero, and call runHeartbeatOnce; current main has no guard for that state.

## Solution Assessment

Yes. Reusing resolveActiveReplyRunSessionId after heartbeat preflight and returning the existing requests-in-flight skip is the narrowest maintainable fix because the wake layer already retries that busy reason.
