## Summary

Keep open. Current main still lacks the active reply-run heartbeat guard, the issue is paired with the open closing PR at https://github.com/openclaw/openclaw/pull/64963, and fresh v2026.5.7 comments describe adjacent stale-output replay and delivery-context leakage that should not be swept as fixed yet.

## What This Changes

Heartbeat and async system-event wakes should not preempt, swallow, or replay in-progress Telegram topic replies in the wrong session or target.

## Best Possible Solution

Land or cherry-pick https://github.com/openclaw/openclaw/pull/64963 for the active-reply race, then split or link any remaining stale-output replay and delivery-context leak work to a narrower follow-up if it persists.

## Reproduction Assessment

Yes, as a high-confidence source-level path: seed a Telegram-backed session, enqueue a system event for that session, register an active reply run for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main has no active reply-run guard for that state.

## Solution Assessment

Yes for the central race: reusing `resolveActiveReplyRunSessionId` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the narrowest maintainable fix. The newer stale replay and default-target leak reports should stay separate if they remain after that guard lands.
