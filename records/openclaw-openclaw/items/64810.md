## Summary

Keep open. Current main still lacks the active reply-run guard described by the report, and the issue is already paired with open, unmerged closing PR #64963 rather than a landed fix.

## What This Changes

Report asks OpenClaw to prevent heartbeat and async system-event turns from preempting or swallowing in-progress Telegram topic replies.

## Best Possible Solution

Review and land #64963, or apply the same focused current-main change: after heartbeat preflight resolves sessionKey, skip with requests-in-flight when resolveActiveReplyRunSessionId(sessionKey) is truthy, keep the regression test for active reply runs with idle queues, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. A focused reproduction is to seed a Telegram-backed session, enqueue a system event for that session, register a running reply operation for the same sessionKey, mock queue sizes as zero, and call runHeartbeatOnce; current main has no active-run guard for that state.

## Solution Assessment

Yes. The narrow fix proposed by #64963 is the best current direction because it reuses the existing reply-run registry and requests-in-flight heartbeat retry path instead of adding a new routing lane or broad delivery-policy change.
