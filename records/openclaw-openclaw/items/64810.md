## Summary

Keep open. Current main still lacks the active reply-run guard in `runHeartbeatOnce`, and an open unmerged closing PR targets the exact narrow race while newer comments describe adjacent stale-replay cases that are not proof of a merged fix.

## What This Changes

Prevent heartbeat, system-event, or async-completion turns in Telegram topic sessions from preempting and swallowing an in-progress user reply.

## Best Possible Solution

Land or replace the linked active reply-run heartbeat guard, then keep broader stale replay/default-target leakage on the narrower related reports unless the landed fix explicitly covers them.

## Reproduction Assessment

Yes, at source level: seed a Telegram-backed session, enqueue a system event for that session, register a running reply operation for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main lacks the guard that would skip that state.

## Solution Assessment

Yes for the narrow active-reply race: checking `resolveActiveReplyRunSessionId(sessionKey)` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the smallest maintainable fix. The newer stale-replay/default-target cases should stay separate unless the final patch covers them directly.
