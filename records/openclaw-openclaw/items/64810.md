## Label Justifications

- `P1`: The reported bug could interrupt live Telegram topic work and make user-visible replies disappear during normal agent operation.
- `impact:session-state`: The failure mode mixed heartbeat/system-event turns into the active conversation lane and changed which turn owned the session state.
- `impact:message-loss`: The user-visible effect was a swallowed or unreplayed in-progress reply in a messaging channel.

## Summary

The central heartbeat/system-event preemption bug is fixed on current main and shipped in v2026.6.6: heartbeat/event wakes now defer while same-agent reply runs are active, retry through the wake layer, and stale heartbeat pending-final replay is guarded separately. Later comments describe adjacent Telegram/session-lane symptoms, but those are covered by narrower open follow-ups rather than keeping this shipped preemption report open.

## What This Changes

The issue asks OpenClaw to prevent heartbeat, async completion, and restart/system-event turns from interrupting or swallowing in-progress Telegram topic replies.

## Best Possible Solution

Keep the v2026.6.6 heartbeat/event deferral and handle any remaining activation, visible-text delivery, or pending-delivery TTL symptoms in their narrower open reports.

## Reproduction Assessment

Yes. The historical race is source-reproducible from the heartbeat runner path, and the shipped regression tests model the drained queue plus active reply-run window without needing live Telegram credentials.

## Solution Assessment

Yes. The shipped solution defers background/event heartbeat work at the shared heartbeat runner boundary and lets the existing wake retry path reschedule, which is narrower than adding Telegram-specific delivery policy.
