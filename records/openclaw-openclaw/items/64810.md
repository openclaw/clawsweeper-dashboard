## Summary

Close as implemented: current main and v2026.6.6 now defer heartbeat runs around active reply work and suppress stale pending-final heartbeat replay; later broader activation and delivery reports are tracked by separate open issues.

## What This Changes

The issue requests that Telegram topic heartbeat/system-event turns not preempt, swallow, or replay in-progress user-visible replies.

## Best Possible Solution

Keep the v2026.6.6 heartbeat/replay implementation and let the separate activation, steer-queueing, and bare-text delivery issues carry any remaining broader behavior.

## Reproduction Assessment

Yes for the original pre-fix race: the issue discussion and regression tests model an active reply run with drained queues while heartbeat/system wake traffic arrives. I did not run a live Telegram repro during this read-only sweep because current main now has the guard and replay suppression in source and tests.

## Solution Assessment

Yes. The best fix is in the core heartbeat/reply boundary: defer background heartbeat turns while reply work is active and prevent heartbeat runs from replaying non-ack pending final text through the wrong delivery target.
