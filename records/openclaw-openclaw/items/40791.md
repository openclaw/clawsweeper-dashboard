## Summary

Keep open. Current main still lacks the proposed external rescue watchdog, but this PR has concrete state/race bugs and security-sensitive incident handling concerns, and it overlaps unresolved rescue architecture work in #46502 and #44441.

## What This Changes

The PR adds an opt-in shell rescue watchdog, a Claude Code incident runner, systemd user timer templates, setup documentation, an auth-monitoring docs link, and test-only TypeScript inference annotations.

## Best Possible Solution

Maintainers should first pick the canonical rescue model, then either harden this external script path with locking, state safety, incident permissions, focused script coverage, and clearer sensitive-data docs, or carry only the useful docs/setup pieces into the core-service or docs-only approach.

## Reproduction Assessment

Yes for the patch defects. The fail-recover-fail cooldown path, same-second overlapping invocation path, and long-runner overlap path follow directly from the proposed scripts/rescue-watchdog.sh control flow; this review did not execute the PR scripts because the task is read-only.

## Solution Assessment

No. An external watchdog may be useful, but this PR is not the best mergeable form until the recovery architecture and security posture are chosen and the state/concurrency defects are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: When the watchdog sees a healthy cycle, it exits without clearing the previous fingerprint and timestamp. A fail -> recover -> fail sequence inside OPENCLAW_RESCUE_COOLDOWN_SEC can suppress the fresh outage and skip incident capture, runner execution, and notification.
  - confidence: 0.86
- **[P2] Use unique incident directories per run:** `scripts/rescue-watchdog.sh:196-198`
  - body: incident_id has only second-level granularity and mkdir -p reuses an existing directory. Two overlapping timer or manual runs in the same second can write into the same bundle, producing mixed or partial incident artifacts instead of independent captures.
  - confidence: 0.88
- **[P2] Persist suppression state before launching the runner:** `scripts/rescue-watchdog.sh:250`
  - body: state.env is saved only after RUNNER_CMD returns. During a long rescue run, another watchdog invocation sees no updated fingerprint and can launch a duplicate runner or notification cycle for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff adds opt-in command execution and diagnostic capture that need state-file and incident-data hardening before merge.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:51`
  - body: load_state sources state.env from an environment-configurable state directory without regular-file, ownership, permission, or symlink checks. Replacing that file would turn the next watchdog run into shell execution under the operator account.
  - confidence: 0.84
- **[medium] Document sensitive incident bundle handling:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs list health snapshots, status output, gateway log tails, git state, and runner output in the incident bundle, and the runner adds that directory to Claude context. Operators need explicit permissions, retention, and sensitive-data guidance before enabling this flow.
  - confidence: 0.82
