## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep the Claude rescue runner opt-in:** `scripts/systemd/openclaw-rescue-watchdog.service:9`
  - body: The setup guide frames the runner as optional and tells operators to edit the copied service before setting it, but the shipped systemd template enables the Claude runner by default. Anyone who follows the timer setup gets command execution and workspace/incident handoff on the first detected failure without making that explicit choice, so keep this line commented or split the runner-enabled template from the base watchdog service.
  - confidence: 0.88
- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: A healthy run exits before touching `state.env`, so the previous failure fingerprint remains active. In a fail, recover, then fail-again sequence within the cooldown window, the second outage is treated as a duplicate and skips incident capture, runner execution, and notification even though recovery happened in between.
  - confidence: 0.9
- **[P2] Use collision-free incident directories:** `scripts/rescue-watchdog.sh:196-198`
  - body: Incident IDs only include seconds and `mkdir -p` reuses the directory. Two overlapping watchdog invocations in the same second can write the same bundle, mix artifacts, or fail under `set -e`; include a unique suffix and create with `mkdir` so each run owns its incident directory.
  - confidence: 0.86
- **[P2] Persist suppression before launching the runner:** `scripts/rescue-watchdog.sh:250`
  - body: The cooldown state is written only after the runner completes, so a long rescue run leaves the current fingerprint invisible to another timer/manual invocation. Persist the fingerprint before invoking the runner, or take a lock, so overlapping runs do not start duplicate remediation and notifications for the same incident.
  - confidence: 0.87
