Keep open. This is a real feature proposal, but current main does not implement it or have a merged successor, and the PR still has source-proven script defects, security hardening gaps, missing real behavior proof, and overlapping open watchdog work that needs maintainer direction.

## What This Changes

The PR adds optional external rescue-watchdog and Claude runner scripts, systemd user timer templates, rescue-watchdog docs, and test-only type annotation sync changes.

## Best Possible Solution

Choose one maintainer-approved watchdog model, then land a hardened implementation with safe state handling, collision-free incident bundles, immediate overlap suppression, redaction/retention docs, and real behavior proof.

Yes for the PR defects by source inspection: stale suppression, same-second incident collision, and duplicate long-runner behavior follow from the shell script at the PR head. The broader watchdog capability is a feature request, so runtime reproduction is not the deciding question.

## Solution Assessment

No, not as-is. The external script approach may be useful, but the current branch needs script hardening, security documentation, real proof, and maintainer selection among the open core-service and CLI watchdog alternatives.

Full review comments:

  - body: When `incident_reason` is empty, the script exits before loading or clearing `state.env`, so a fail -> recover -> fail sequence inside `OPENCLAW_RESCUE_COOLDOWN_SEC` can be treated as a duplicate and skip the fresh incident capture, runner, and notification.
  - confidence: 0.86

  - body: `incident_id` uses only second-level timestamp precision and `mkdir -p` reuses existing paths, so overlapping timer/manual runs in the same second can write mixed or partial artifacts into one incident bundle instead of separate captures.
  - confidence: 0.88
- **[P2] Persist suppression state before running rescues:** `scripts/rescue-watchdog.sh:250`
  - body: The latest fingerprint is saved only after the runner and post-health probe complete, so another watchdog invocation during a long rescue run can start a duplicate incident capture, notification, and remediation attempt for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds opt-in command execution and diagnostic capture, and the proposed state-file and incident-bundle handling need hardening before merge.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:49`
  - body: `load_state` sources `state.env` from an environment-configurable state directory without regular-file, owner, permission, or symlink checks; replacing that file would cause shell code execution under the operator account on the next watchdog run.
  - confidence: 0.84
- **[medium] Document sensitive incident bundle handling:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs describe bundles containing health/status snapshots, gateway log tails, git state, diff stats, and runner output, and the runner adds that directory to Claude context; operators need explicit permissions, retention, and redaction guidance before enabling this flow.
  - confidence: 0.82
