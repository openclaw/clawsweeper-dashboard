## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:181-183`
  - body: After an incident, a successful probe exits here before loading or clearing state.env. A fail -> recover -> fail sequence inside the cooldown window can still match LAST_FINGERPRINT and skip a fresh incident capture, notification, and runner invocation.
  - confidence: 0.86
- **[P2] Use collision-free incident directories:** `scripts/rescue-watchdog.sh:202-204`
  - body: incident_id has only second-level precision and mkdir -p reuses an existing path. Overlapping timer/manual runs in the same second can write mixed or partial artifacts into one incident bundle instead of two independent captures.
  - confidence: 0.88
- **[P2] Persist suppression state before running rescues:** `scripts/rescue-watchdog.sh:246`
  - body: The new fingerprint is saved only after the runner and post-health probe finish. During a slow rescue command, another invocation still sees old state and can start duplicate capture, notification, and remediation for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds opt-in command execution and diagnostic capture, and this pass found concrete state-file and incident-data hardening gaps.

  - body: load_state sources state.env from an environment-configurable state directory without regular-file, ownership, permission, or symlink checks. Replacing that file would execute shell code under the operator account on the next watchdog run.

  - confidence: 0.82
