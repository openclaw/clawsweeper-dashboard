## Summary

Keep open. Current main still lacks this rescue watchdog surface, but this PR is not mergeable as-is because the shell watchdog has source-reproducible state/concurrency bugs, security-sensitive state and incident-bundle handling, missing real behavior proof, and overlapping watchdog designs that need maintainer direction.

## What This Changes

This PR adds optional external rescue watchdog and Claude runner scripts, systemd user timer templates, rescue watchdog docs, and test-only TypeScript annotation changes.

## Best Possible Solution

Pick one canonical rescue model first; if external scripts remain desired, harden state, concurrency, and incident privacy, add focused scripted coverage and real behavior proof, and split unrelated test typing churn out of the feature branch.

## Reproduction Assessment

Yes for the PR defects by source inspection. The fail-recover-fail cooldown path, same-second incident directory collision, and long-runner overlap follow directly from `scripts/rescue-watchdog.sh`; no live script execution was attempted in this read-only review.

## Solution Assessment

No. The external watchdog idea may be useful, but this branch is not the best mergeable form until maintainers choose the rescue architecture and security posture and the concrete script defects are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: When `incident_reason` is empty, the script exits before loading or clearing `state.env`, so a fail -> recover -> fail sequence inside the cooldown window can be treated as a duplicate and skip the fresh incident capture, runner, and notification.
  - confidence: 0.86
- **[P2] Use collision-free incident directories:** `scripts/rescue-watchdog.sh:196-198`
  - body: `incident_id` uses only second-level timestamp precision and `mkdir -p` reuses existing paths, so overlapping timer/manual runs in the same second can write mixed or partial artifacts into one incident bundle.
  - confidence: 0.88
- **[P2] Persist suppression state before running rescues:** `scripts/rescue-watchdog.sh:250`
  - body: The latest fingerprint is saved only after the runner and post-health probe complete, so another watchdog invocation during a long rescue run can start a duplicate incident capture, notification, and remediation attempt for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds opt-in command execution and diagnostic capture, and this pass found concrete state-file and incident-data hardening gaps.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:49`
  - body: `load_state` sources `state.env` from an environment-configurable state directory without regular-file, owner, permission, or symlink checks. Replacing that file would cause shell code execution under the operator account on the next watchdog run.
  - confidence: 0.84
- **[medium] Document sensitive incident bundle handling:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs describe bundles containing health/status snapshots, gateway log tails, git state, diff stats, and runner output, and the runner adds that directory to Claude context. Operators need explicit permissions, retention, and redaction guidance before enabling this flow.
  - confidence: 0.82
