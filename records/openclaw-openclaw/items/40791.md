## Summary

Keep open. Current main still does not implement the external rescue watchdog, and the proposed patch has concrete state/concurrency bugs plus security hardening questions that need maintainer direction alongside the open rescue alternatives #46502 and #44441.

## What This Changes

This PR adds an optional external shell rescue watchdog, a Claude Code incident runner, systemd user timer templates, rescue-watchdog docs, and test-only TypeScript annotations.

## Best Possible Solution

Choose one canonical rescue model first; if an external watchdog remains desired, harden its state, concurrency, and incident-bundle handling, add focused script coverage, and keep the docs aligned with that chosen model.

## Reproduction Assessment

Yes for the patch defects: the fail-recover-fail cooldown path, same-second incident overlap, and long-runner overlap follow directly from `scripts/rescue-watchdog.sh` control flow. I did not execute the PR scripts because this review was read-only.

## Solution Assessment

No. The external watchdog may be useful, but this PR is not the best mergeable form until maintainers choose the rescue architecture/security posture and the concrete script defects are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: When the probe becomes healthy, the script exits without clearing the previous fingerprint and timestamp. A fail -> recover -> fail sequence inside the cooldown can suppress the fresh outage and skip capture, runner execution, and notification.
  - confidence: 0.86
- **[P2] Use unique incident directories per run:** `scripts/rescue-watchdog.sh:196-198`
  - body: `incident_id` has only second-level granularity and `mkdir -p` reuses an existing directory. Two overlapping invocations in the same second can write into the same bundle and leave mixed or partial artifacts.
  - confidence: 0.88
- **[P2] Persist suppression state before launching the runner:** `scripts/rescue-watchdog.sh:250`
  - body: `state.env` is saved only after `RUNNER_CMD` returns. During a long rescue run, another invocation sees no updated fingerprint and can launch a duplicate runner or notification cycle for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds opt-in command execution and diagnostic capture without enough state-file and incident-data hardening.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:51`
  - body: `load_state` sources `state.env` from an environment-configurable state directory without regular-file, ownership, permission, or symlink checks. Replacing that file would turn the next watchdog run into shell execution under the operator account.
  - confidence: 0.84
- **[medium] Document sensitive incident bundle handling:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs list health/status snapshots, gateway log tails, git state, diffs, and runner output, and the runner adds the incident directory to Claude context. Operators need explicit permissions, retention, and sensitive-data guidance before enabling this flow.
  - confidence: 0.82
