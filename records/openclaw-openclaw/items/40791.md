## Summary

Keep open. Current main still lacks the external rescue watchdog surface, and the PR is not merge-ready because the proposed shell watchdog has reproducible state/concurrency defects plus security hardening gaps around sourced state and incident bundles. The related open rescue work in #46502 and #44441 also means maintainers should choose the recovery architecture before polishing this branch.

## What This Changes

The PR adds an opt-in shell rescue watchdog, a Claude Code incident runner, systemd user timer templates, setup documentation, an auth-monitoring docs link, and several test-only TypeScript inference annotations.

## Best Possible Solution

Maintainers should first pick the canonical rescue model. If the external watchdog remains desired, keep it explicitly opt-in, harden state and incident handling, fix concurrency and cooldown semantics, add focused script coverage, document sensitive bundle handling, and split unrelated test typing changes; otherwise, preserve only useful setup/docs ideas in the core-service path from #46502 or the docs-only path from #44441.

## Reproduction Assessment

Yes for the patch defects. The fail-recover-fail cooldown path, same-second overlap path, and long-runner overlap path follow directly from the proposed scripts/rescue-watchdog.sh control flow; this review did not execute the PR scripts because the task was read-only.

## Solution Assessment

No. The proposed external watchdog may be useful, but this PR is not the best mergeable form until maintainers choose between the external script, the core rescue cron/service work in #46502, and the operator-docs approach in #44441, and until the state/concurrency/security issues are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: When the watchdog sees a healthy cycle, it exits without clearing the previous fingerprint and timestamp. A fail -> recover -> fail sequence inside OPENCLAW_RESCUE_COOLDOWN_SEC can therefore suppress the fresh outage and skip incident capture, runner execution, and notification.
  - confidence: 0.86
- **[P2] Use unique incident directories per run:** `scripts/rescue-watchdog.sh:196-198`
  - body: incident_id has only second-level granularity and mkdir -p reuses an existing directory. Two overlapping timer or manual runs in the same second can write into the same bundle, producing mixed or partial incident artifacts instead of independent captures.
  - confidence: 0.88
- **[P2] Persist suppression state before launching the runner:** `scripts/rescue-watchdog.sh:250`
  - body: state.env is saved only after RUNNER_CMD returns. During a long rescue run, another watchdog invocation sees no updated fingerprint and can launch a duplicate runner or notification cycle for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds opt-in command execution and diagnostic capture, and it needs state-file and incident-data hardening before merge.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:51`
  - body: load_state sources state.env from an environment-configurable state directory without regular-file, ownership, permission, or symlink checks. Replacing that file would turn the next watchdog run into shell execution under the operator account.
  - confidence: 0.84
- **[medium] Document sensitive incident bundle handling:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs list health snapshots, status output, gateway log tails, git state, and runner output in the incident bundle, and the runner adds that directory to Claude context. Operators need explicit permissions, retention, and sensitive-data guidance before enabling this flow.
  - confidence: 0.82
