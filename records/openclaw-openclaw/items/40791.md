## Summary

Keep open: current main does not implement the external rescue watchdog, and the PR still has blocking script state/concurrency defects plus security hardening concerns while overlapping rescue designs remain open.

## What This Changes

This PR adds optional external rescue watchdog and Claude runner scripts, systemd user timer templates, rescue watchdog docs, and test-only TypeScript annotation changes.

## Best Possible Solution

Pick one canonical rescue model first; if this external watchdog remains desired, harden state, concurrency, and sensitive-bundle handling, add focused scripted coverage, and split unrelated test typing changes.

## Reproduction Assessment

Yes for the patch defects by source inspection. The fail-recover-fail cooldown path, same-second incident directory collision, and long-runner overlap follow directly from the proposed shell control flow; I did not execute the scripts because this review was read-only.

## Solution Assessment

No. The external watchdog may be useful, but this PR is not the best mergeable form until maintainers choose the rescue architecture and security posture across the overlapping rescue PRs and the concrete script defects are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clear suppression state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: When `incident_reason` is empty, the script exits without updating `state.env`, so a fail -> recover -> fail sequence inside `OPENCLAW_RESCUE_COOLDOWN_SEC` can suppress the fresh outage and skip incident capture, runner execution, and notification.
  - confidence: 0.86
- **[P2] Use unique incident directories per run:** `scripts/rescue-watchdog.sh:196-198`
  - body: `incident_id` has only second-level granularity and `mkdir -p` reuses an existing directory. Two overlapping timer/manual runs in the same second can write into the same bundle and leave mixed or partial incident artifacts.
  - confidence: 0.88
- **[P2] Persist suppression state before launching the runner:** `scripts/rescue-watchdog.sh:250`
  - body: `state.env` is saved only after `RUNNER_CMD` returns. During a long rescue run, another watchdog invocation sees no updated fingerprint and can launch a duplicate runner or notification cycle for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds opt-in command execution and diagnostic capture without enough state-file and incident-data hardening.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:39`
  - body: `load_state` sources `state.env` from an environment-configurable state directory without regular-file, ownership, permission, or symlink checks. Replacing that file would turn the next watchdog run into shell execution under the operator account.
  - confidence: 0.84
- **[medium] Document sensitive incident bundle handling:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs describe bundles containing health/status snapshots, gateway log tails, git state, diffs, and runner output, and the runner adds that directory to Claude context. Operators need explicit permissions, retention, and sensitive-data guidance before enabling this flow.
  - confidence: 0.82
