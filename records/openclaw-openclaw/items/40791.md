  - body: When no incident is detected, the script exits before loading or clearing prior state, so a fail, recover, fail sequence inside the cooldown can be skipped as a duplicate even though it is a fresh outage.
  - confidence: 0.86
- **[P2] Use collision-free incident directories:** `scripts/rescue-watchdog.sh:196-198`
  - body: The incident id only has second-level precision and mkdir -p reuses an existing path, so overlapping timer/manual runs in the same second can mix artifacts or leave partial bundles instead of independent captures.
  - confidence: 0.88
- **[P2] Persist suppression state before running rescues:** `scripts/rescue-watchdog.sh:250`
  - body: The fingerprint is saved only after the runner and post-health probe finish, so another invocation during a long rescue can start a duplicate capture, notification, and remediation attempt for the same outage.
  - confidence: 0.89
- **[P2] Parse watchdog state without sourcing shell code:** `scripts/rescue-watchdog.sh:49-51`
  - body: load_state sources state.env from an environment-configurable state directory without regular-file, owner, permission, or symlink checks, so replacing that file could execute shell code under the operator account on the next watchdog run.
  - confidence: 0.84

  - body: The docs list health/status snapshots, gateway logs, git state, diff stats, and runner output, then the runner adds the incident directory to Claude context; operators need clear permissions, retention, and redaction guidance before enabling this flow.
