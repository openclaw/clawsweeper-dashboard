## Label Justifications

- `P1`: The issue misroutes or blocks outgoing Telegram messages in real multi-bot setups on current main and the latest release.
- `impact:message-loss`: The reported failure is misdelivered Telegram outbound messages through the wrong bot or failed sends when accountId is omitted.
- `impact:auth-provider`: The broken path is account token selection for Telegram bot identities and default account routing.

## Summary

Keep open: current main and v2026.5.18 still route token-gated Telegram message actions through the raw token resolver, so omitted accountId does not honor the configured Telegram default account despite the existing documented resolver semantics.

## What This Changes

Make outgoing Telegram message actions use the configured default bot account when no explicit account is supplied.

## Best Possible Solution

Route every token-gated Telegram action through resolveTelegramAccount for omitted accountId and add regression coverage for defaultAccount, while preserving explicit unknown-account fail-closed token behavior.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: configure channels.telegram.defaultAccount = "work" with channels.telegram.accounts.work.botToken and no top-level/default/env token, then call handleTelegramAction or openclaw message send without accountId. The current source resolves the token as the implicit default before the defaultAccount resolver can run.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse resolveTelegramAccount in extensions/telegram/src/action-runtime.ts for token-gated actions, not to require --account or add another local fallback.
