## Label Justifications

- `P1`: The issue can block or misroute outgoing Telegram messages in real multi-bot setups on current main and the latest release.
- `impact:message-loss`: The reported failure is outbound Telegram delivery through the wrong bot or failed delivery when accountId is omitted.
- `impact:auth-provider`: The broken path is Telegram account token selection and default account routing for bot identities.

## Summary

Keep open: current main and the latest release still have the narrower bug confirmed in prior discussion, where token-gated Telegram message actions bypass the default-account resolver when accountId is omitted. The linked replacement PRs that targeted this path are closed unmerged, and no open canonical fix appears to own this exact action-runtime token-selection work.

## What This Changes

The issue asks OpenClaw to honor the configured default Telegram account for outgoing Telegram messages when no explicit account is passed.

## Best Possible Solution

Unify token-gated Telegram message actions on resolveTelegramAccount for omitted accountId, while preserving explicit unknown-account fail-closed behavior and the existing direct-send resolver semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure channels.telegram.defaultAccount = "work" with channels.telegram.accounts.work.botToken and no top-level/default/env token, then invoke a Telegram message action without accountId; action-runtime still resolves the raw implicit-default token path.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse resolveTelegramAccount in extensions/telegram/src/action-runtime.ts for token-gated actions instead of requiring --account or adding another local fallback.
