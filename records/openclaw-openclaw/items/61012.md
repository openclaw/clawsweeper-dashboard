## Summary

Keep open: current main still has the Telegram message-action token path that bypasses the default-account resolver, and the linked repair PRs were closed unmerged. The prior discussion usefully narrowed this from a client-cache claim to the omitted-account `openclaw message send` / action-runtime bug.

## What This Changes

The issue asks for omitted-account Telegram outgoing actions to honor the configured default Telegram account in multi-bot setups.

## Best Possible Solution

Land a narrow Telegram plugin repair that resolves token-gated message actions through the existing account resolver and locks the omitted-account defaultAccount case with focused regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction. Configure `channels.telegram.defaultAccount = "work"`, `channels.telegram.accounts.work.botToken`, no top-level/default token or env token, then call `handleTelegramAction` or `openclaw message send --channel telegram` without `accountId`; current source resolves the token as implicit `default`.

## Solution Assessment

Yes. The best fix is to align `extensions/telegram/src/action-runtime.ts` with `resolveTelegramAccount` rather than adding another special-case fallback or documenting `--account` as required.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/61012 narrowly in the Telegram plugin. In `extensions/telegram/src/action-runtime.ts`, stop resolving token-gated message actions with raw `resolveTelegramToken(cfg, { accountId })` when `accountId` is omitted; use the existing `resolveTelegramAccount` semantics so `channels.telegram.defaultAccount` and configured account tokens are honored. Preserve explicit account behavior, especially unknown explicit account IDs in multi-account configs refusing channel-level fallback. Add focused regression coverage in `extensions/telegram/src/action-runtime.test.ts` for omitted `accountId` plus named `channels.telegram.defaultAccount` across token-gated actions, and keep existing `accounts.ts`, `token.ts`, `send.ts`, and `src/commands/message.ts` behavior green. Add a user-facing changelog fix entry if the repair changes runtime behavior. Do not broaden this into duplicate-token validation, startup-warning redesign, or Telegram client/cache refactors unless a test proves that is required for the central bug.
