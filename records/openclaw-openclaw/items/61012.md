## Summary

Keep open. Current main still has the narrow Telegram message-action bug: omitted-account outgoing actions can resolve tokens through the implicit `default` account instead of `channels.telegram.defaultAccount`, while the lower-level send/account resolver already has the intended fallback.

## What This Changes

The issue requests that Telegram outgoing message actions without an explicit `--account` honor the configured default Telegram account and document or validate multi-bot routing edge cases.

## Best Possible Solution

Land a narrow Telegram plugin repair that resolves omitted-account token-gated message actions through `resolveTelegramAccount`, forwards the resolved account ID and token into the action helpers, and adds focused action-runtime regression coverage for named `defaultAccount`, explicit accounts, and unknown-account safeguards.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `channels.telegram.defaultAccount = "work"`, `channels.telegram.accounts.work.botToken`, no top-level/default token, then `handleTelegramAction` or `openclaw message send --channel telegram` without `accountId`; current code resolves the token as implicit `default` before the send helper fallback can run.

## Solution Assessment

Yes. The best fix is to align `extensions/telegram/src/action-runtime.ts` with the existing `resolveTelegramAccount` semantics already used by direct sends, rather than adding a separate default-account special case or weakening token fallback guards.
