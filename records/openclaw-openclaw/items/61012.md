## Summary

Keep open: current main still has the narrow Telegram message-action bug where omitted-account outgoing actions bypass the default-account resolver and resolve tokens as the implicit `default` account.

## What This Changes

Make Telegram outgoing message actions without an explicit account use `channels.telegram.defaultAccount` and improve multi-bot routing diagnostics.

## Best Possible Solution

Land a narrow Telegram plugin repair that resolves omitted-account token-gated message actions through `resolveTelegramAccount`, forwards the resolved account ID and token, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Configure `channels.telegram.defaultAccount = "work"`, `channels.telegram.accounts.work.botToken`, no top-level/default token or env token, then run `handleTelegramAction` or `openclaw message send --channel telegram` without `accountId`; current source resolves the token as implicit `default` before the send helper fallback can run.

## Solution Assessment

Yes. The narrowest maintainable fix is to align `extensions/telegram/src/action-runtime.ts` with the existing `resolveTelegramAccount` semantics rather than adding a separate special case or weakening token fallback guards.

## Repair Work Prompt

Fix the Telegram default-account message-action bug for this issue. Current main resolves token-gated actions in `extensions/telegram/src/action-runtime.ts` with `resolveTelegramToken(cfg, { accountId })`; when `accountId` is omitted this selects the implicit `default` account instead of `channels.telegram.defaultAccount`. Update the action runtime to resolve a `ResolvedTelegramAccount` through `resolveTelegramAccount` for omitted and explicit accounts, pass the resolved `account.accountId` and token into send, poll, react, delete, edit, sticker, and forum-topic helpers, and preserve the existing unknown explicit-account no-fallback safety from `token.ts`. Add focused regression coverage in `extensions/telegram/src/action-runtime.test.ts` for a named `defaultAccount` with only `accounts.work.botToken` and no top-level/env token across token-gated actions, plus explicit-account and unknown-account safeguards. Do not change Telegram client caching, incoming routing, plugin APIs, or the account resolver contract. Add a user-facing `CHANGELOG.md` fix entry.
