## Label Justifications

- `P1`: The issue can misroute or block outgoing Telegram message actions in real multi-bot setups on current main and the latest release.
- `impact:message-loss`: The reported failure concerns outbound Telegram messages going through the wrong bot or failing when accountId is omitted.
- `impact:auth-provider`: The broken path is Telegram account token selection for default bot identity routing.

## Summary

Keep open: current main and the latest v2026.5.19 release still route Telegram message-action tokens through the raw token resolver, so omitted `accountId` does not reliably honor `channels.telegram.defaultAccount`. The closed replacement PR at https://github.com/openclaw/openclaw/pull/61144 identified the same narrow action-runtime fix and tests, but it was not merged.

## What This Changes

The issue asks for omitted-account Telegram outbound message actions to honor the configured default Telegram account instead of falling back to the implicit default bot token.

## Best Possible Solution

Route every token-gated Telegram action through `resolveTelegramAccount` for omitted `accountId`, add action-runtime regression coverage for `defaultAccount`, and keep broader warning or validation requests as separate follow-ups.

## Reproduction Assessment

Yes. Configure `channels.telegram.defaultAccount = "work"` with only `channels.telegram.accounts.work.botToken`, then call `handleTelegramAction` or `openclaw message send` without `accountId`; current source resolves the token as the implicit default before the defaultAccount resolver can run.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse `resolveTelegramAccount` in `extensions/telegram/src/action-runtime.ts` for token-gated actions, not to require `--account` or add another local fallback.
