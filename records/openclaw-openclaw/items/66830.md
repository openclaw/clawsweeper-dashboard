## Summary

Keep open. Current main still computes both preflight compaction and memoryFlush thresholds from `reserveTokensFloor`, while the public config contract and Pi settings path treat `reserveTokens` as the effective headroom; the focused fix path remains the open replacement PR at https://github.com/openclaw/openclaw/pull/74010.

## What This Changes

The issue asks OpenClaw to make preflight compaction and memoryFlush thresholds honor configured `compaction.reserveTokens` when it is above `reserveTokensFloor`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74010 or an equivalent narrow patch that resolves one effective compaction reserve and feeds it consistently into preflight compaction and memoryFlush gates with focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction. Configure `reserveTokens` above `reserveTokensFloor`; current main still subtracts only the floor in both gate formulas while the config and Pi settings contract make the higher reserve meaningful.

## Solution Assessment

Yes. The narrow maintainable fix is to centralize effective reserve resolution and pass that value into both gates without changing the public `MemoryFlushPlan` plugin surface.
