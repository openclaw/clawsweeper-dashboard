## What This Changes

The issue asks for memoryFlush and preflight compaction threshold gates to use the effective `compaction.reserveTokens` budget instead of treating `reserveTokensFloor` as the whole reserve.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/74010 or an equivalent narrow patch that centralizes effective reserve calculation and feeds it into both gates, while leaving adjacent compaction issues to their own threads.

## Reproduction Assessment

Yes. Source inspection on current main shows both threshold gates still subtract `reserveTokensFloor`, while the config contract defines `reserveTokens` as the effective headroom knob.

## Solution Assessment

Yes. The maintainable fix is to resolve the effective Pi compaction reserve once from the config/floor semantics and pass that value into preflight and memoryFlush gates without changing the public `MemoryFlushPlan` shape.
