## Summary

Keep open. Current main still computes both preflight compaction and memoryFlush token thresholds from `reserveTokensFloor`, while the adjacent Pi settings path honors the configured `reserveTokens`; the focused replacement PR #74010 remains the right path before this issue closes.

## What This Changes

The issue asks OpenClaw to make memoryFlush and preflight compaction thresholds honor `agents.defaults.compaction.reserveTokens` above `reserveTokensFloor`.

## Best Possible Solution

Land PR #74010 or an equivalent narrow patch that resolves one effective compaction reserve and uses it consistently in preflight compaction and memoryFlush gates, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. This has a high-confidence source reproduction on current main: configure `reserveTokens` above `reserveTokensFloor`, then inspect the two gate formulas and see they still subtract only the floor.

## Solution Assessment

Yes. The narrow maintainable fix is to centralize effective reserve resolution, preserve the existing public `MemoryFlushPlan` contract where possible, and thread the resolved value into both gate call sites.
