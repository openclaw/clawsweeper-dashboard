## Summary

Keep open. Current main still computes the preflight compaction and memoryFlush token thresholds from `reserveTokensFloor`, while the config contract and Pi settings path treat `reserveTokens` as the effective headroom; the focused fix path remains the open PR at https://github.com/openclaw/openclaw/pull/74010.

## What This Changes

The issue asks for preflight compaction and memoryFlush gates to respect `agents.defaults.compaction.reserveTokens` when it is above `reserveTokensFloor`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74010 or an equivalent narrow patch that resolves one effective compaction reserve and uses it consistently in preflight compaction and memoryFlush gates with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: configure `reserveTokens` above `reserveTokensFloor`, and both gate formulas still subtract only the floor while the config contract and Pi settings path treat `reserveTokens` as effective headroom.

## Solution Assessment

Yes. The best fix direction is the narrow one in https://github.com/openclaw/openclaw/pull/74010: centralize effective reserve resolution and pass that value into both gates without changing the public `MemoryFlushPlan` contract.
