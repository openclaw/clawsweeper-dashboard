## Summary

Keep open. Current main still has the reported floor-only threshold behavior in both preflight compaction and memoryFlush, and the focused replacement PR is still the appropriate path to resolve it before this issue closes.

## What This Changes

The issue asks for preflight compaction and memoryFlush thresholds to use the effective `agents.defaults.compaction.reserveTokens` budget instead of only `reserveTokensFloor`.

## Best Possible Solution

Land PR #74010 or an equivalent narrow core patch that resolves one effective compaction reserve from `reserveTokens`, `reserveTokensFloor`, and the small-context cap, then uses that value consistently in preflight compaction and memoryFlush gates with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. This is reproducible by source inspection on current main: set `reserveTokens` above `reserveTokensFloor`, then both gate formulas still subtract only the floor while the Pi settings path treats `reserveTokens` as the effective reserve target.

## Solution Assessment

Yes. The best fix direction is to centralize effective reserve resolution and thread that value into both gates; changing broader compaction policy is unnecessary for the reported bug.
