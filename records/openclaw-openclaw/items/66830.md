## Summary

Keep this open: current main still has the reported reserve-token asymmetry, and the active fix path is the open, conflicting replacement PR at https://github.com/openclaw/openclaw/pull/74010 rather than an already-landed implementation.

## What This Changes

The issue reports that memoryFlush and preflight compaction thresholds ignore configured `agents.defaults.compaction.reserveTokens` and use only `reserveTokensFloor`.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/74010, or an equivalent narrow patch that computes the effective reserve once and feeds it to both compaction gates without expanding the public config surface.

## Reproduction Assessment

Yes. Source inspection on current main shows both threshold gates still subtract `reserveTokensFloor`, while the config contract defines `reserveTokens` as the effective headroom knob.

## Solution Assessment

Yes. The best fix direction is the narrow effective-reserve helper approach already proposed by the replacement PR, with focused regression coverage for `reserveTokens > reserveTokensFloor`.
