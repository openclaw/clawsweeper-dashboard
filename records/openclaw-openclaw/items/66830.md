## Summary

Keep open: current `main` still computes the memoryFlush and preflight compaction thresholds from `reserveTokensFloor`, while the config contract and Pi settings path distinguish `reserveTokens` as the user-facing headroom with floor enforcement. The active fix path is the open replacement PR at https://github.com/openclaw/openclaw/pull/74010, so this issue should stay open until that PR lands or is explicitly replaced.

## What This Changes

The issue asks for memoryFlush and preflight compaction threshold math to honor `agents.defaults.compaction.reserveTokens` when it is above `reserveTokensFloor`.

## Best Possible Solution

Land a narrow fix that resolves the effective compaction reserve once, threads it into both preflight and memoryFlush gates, and adds regression coverage for `reserveTokens > reserveTokensFloor`.

## Reproduction Assessment

Yes, source-level. On current `main`, a config with `reserveTokens` above `reserveTokensFloor` still reaches both gate formulas through `reserveTokensFloor`, so the threshold remains `contextWindow - floor - softThreshold` instead of using the configured reserve.

## Solution Assessment

Yes in direction, but not yet landed. The maintainable fix is the helper-based effective-reserve approach reflected in https://github.com/openclaw/openclaw/pull/74010, with focused tests around the two gate call sites.
