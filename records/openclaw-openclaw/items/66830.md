## Summary

Keep open. Current main still computes both preflight compaction and memoryFlush token thresholds from `reserveTokensFloor`, while the public config contract distinguishes `reserveTokens` as the user-facing reserve headroom; the active focused path is still PR #74010.

## What This Changes

Make preflight compaction and memoryFlush thresholds honor configured `agents.defaults.compaction.reserveTokens` above `reserveTokensFloor`.

## Best Possible Solution

Land PR #74010 or an equivalent narrow core patch that resolves one effective compaction reserve and uses it consistently in preflight compaction and memoryFlush gates with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main is enough: set `reserveTokens` above `reserveTokensFloor`, and both gate formulas still subtract only the floor while the Pi settings path treats the higher reserve as effective.

## Solution Assessment

Yes. The best fix direction is to centralize effective reserve resolution and pass that value into both gates; broad compaction policy changes are not needed for this bug.
