## Summary

Keep open. Current main still computes the preflight compaction and memoryFlush token gates from `reserveTokensFloor`, while the config contract and Pi settings path treat `reserveTokens` as the effective headroom; the focused fix path is the open replacement PR at https://github.com/openclaw/openclaw/pull/74010.

## What This Changes

The issue asks OpenClaw to use the effective compaction reserve in preflight compaction and memoryFlush threshold gates instead of only `reserveTokensFloor`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74010 or an equivalent narrow patch that resolves one effective compaction reserve and uses it consistently in both token gates with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: configure `reserveTokens` above `reserveTokensFloor`, and both gate formulas still subtract only the floor while the config contract and Pi settings path treat `reserveTokens` as effective headroom.

## Solution Assessment

Yes. Centralizing effective reserve calculation in the Pi settings surface and passing that value into both gates is the narrow maintainable fix; the open replacement PR follows that direction but still needs refresh/review before landing.
