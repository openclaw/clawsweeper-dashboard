## Summary

Keep open. Current main and the latest tagged release still compute the preflight compaction and memoryFlush thresholds from `reserveTokensFloor`, while the focused replacement PR #74010 remains open and unmerged.

## What This Changes

The issue asks for preflight compaction and memoryFlush gates to honor the effective `agents.defaults.compaction.reserveTokens` budget above `reserveTokensFloor`.

## Best Possible Solution

Land #74010 or an equivalent narrow core patch that centralizes effective reserve resolution, uses that resolved reserve in both preflight compaction and memoryFlush threshold gates, and keeps focused regression coverage for `reserveTokens > reserveTokensFloor` plus small-context floor-capping behavior.

## Reproduction Assessment

Yes. The issue is reproducible by source inspection on current main: configure `reserveTokens` above `reserveTokensFloor`, and both gate paths still subtract only the floor value, so the threshold does not move with the user-facing reserve setting.

## Solution Assessment

Yes. The best fix direction is a narrow shared effective-reserve helper threaded into the two gate call sites, because the existing Pi settings path already defines the intended max-of-config-and-floor semantics.
