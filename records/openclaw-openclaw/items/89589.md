## Summary

Current main fixes the reported update-check permission regression by removing the update-check JSON write path and persisting that state in the shared SQLite state database, whose open/write path hardens the managed state directory and database files.

## What This Changes

The issue reports update-check JSON writes relaxing a hardened OpenClaw state directory from 0700 to group-writable modes under a permissive umask.

## Best Possible Solution

Keep the SQLite-backed update-check state implementation on main; use the separate open state-directory hardening threads for broader mkdir permission cases outside this update-check JSON path.

## Reproduction Assessment

Yes for the affected release/source path: the old update-check code wrote `update-check.json` through the umask-derived JSON writer, while current main no longer uses that path. I did not run a live filesystem repro because the source path and migration are clear and the checkout must stay read-only.

## Solution Assessment

Yes. Moving update-check state into the shared SQLite state database is the maintainable fix because it removes the app-state JSON write and reuses the managed state permission hardening path.
