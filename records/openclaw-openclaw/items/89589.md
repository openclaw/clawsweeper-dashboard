## Label Justifications

- `P2`: This was a scoped local security hardening regression where normal update checks could undo the security-audit chmod recommendation.
- `impact:security`: The reported state directory can contain sensitive OpenClaw state and was becoming group-writable again on local Unix setups.
- `maturity:stable`: The issue affects Linux Gateway host state/update behavior, and linux-gateway-host is scored M4 Stable in qa/maturity-scores.yaml.

## Summary

Close: current main removes the update-check JSON write path that could relax the state directory and now persists update-check state through the shared SQLite state database with private state-file handling. The fix is present on current main, but no release tag in this checkout contains the fix commit yet.

## What This Changes

The issue asks OpenClaw to stop update-check state writes from chmodding the state root back to a group-writable umask-derived mode after it was hardened to 0700.

## Best Possible Solution

Keep the current SQLite-backed update-check implementation and handle broader state-directory mkdir hardening through the separately linked open work.

## Reproduction Assessment

Yes. The old update-check path is source-reproducible from the fix diff: it wrote update-check.json with the JSON writer, while current main no longer creates that file for update-check state.

## Solution Assessment

Yes. Moving update-check persistence into the shared SQLite state DB removes the reported atomic JSON write path and uses the existing private state database permission path instead of adding another JSON-directory chmod special case.
