## Best Possible Solution

Use the configured positive inbound debounce value for forwarded bursts, fall back to 80 ms only when inbound debounce is disabled, cover both paths, and add redacted real Telegram proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main hardcodes the forwarded Telegram lane to 80 ms while the existing inbound debounce config accepts other positive values; the remaining PR defect for 1-79 ms values is also source-reproducible.

## Solution Assessment

No. The fix is in the right Telegram owner boundary, but the implementation should use every configured positive debounce value and reserve the 80 ms fallback for the disabled 0 case.
