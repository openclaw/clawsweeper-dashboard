## Summary

Keep this PR open: the targeted Telegram bug fix is useful and not superseded, but the patch still ignores configured positive debounce windows below 80 ms and has no after-fix real Telegram behavior proof.

## What This Changes

The PR changes Telegram forwarded-burst debounce resolution to use the inbound debounce setting with an 80 ms floor and updates the forwarded debounce cancellation regression test.

## Best Possible Solution

Use the configured positive inbound debounce value for forwarded bursts, fall back to 80 ms only when inbound debouncing is disabled, cover both paths, and add redacted real Telegram proof before merge.

## Reproduction Assessment

Yes. Source inspection shows the changed resolver still floors every forwarded entry with `Math.max(debounceMs, 80)`, while the config schema accepts positive debounce windows below 80 ms.

## Solution Assessment

No. The narrow maintainable fix is to use the configured positive debounce value and reserve the 80 ms fallback for the disabled `0` case, then prove the Telegram behavior in a real setup.
