## Summary

Keep open: the PR addresses the right onboarding default and now has terminal proof, but the repair rule is too broad and would rewrite legitimate explicit small custom-provider context windows; it also still needs the required changelog entry.

## What This Changes

The PR changes non-Azure custom-provider onboarding to use a 200,000-token context window, repairs existing low context-window entries, updates focused tests, and adjusts one memory-core test typing pattern.

## Best Possible Solution

Land a narrower onboarding fix that defaults new non-Azure custom providers to 200,000 tokens, heals known generated bad defaults, preserves intentional explicit small model limits, and includes focused regression coverage plus a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Current main writes a 4,000-token contextWindow for new non-Azure custom-provider models while the default reserve floor is 20,000 and docs list 200,000 as the omitted custom-provider default; I verified the path by source inspection.

## Solution Assessment

No. The new default is the right direction, but using the reserve floor as the repair threshold is too broad because current runtime supports explicit small context windows by capping reserve; only known bad generated defaults should be healed.
