## Summary

Keep open. The reporter clarified this is the macOS Talk Mode voice overlay, and current main plus the latest release still render the generic Talk orb without any assistant identity/avatar fetch or render state, even though the gateway already exposes resolved avatar metadata.

## What This Changes

Request the macOS Talk Mode overlay to render the configured assistant avatar within the existing voice-state visuals, with fallback to the default orb.

## Best Possible Solution

Add a small macOS Talk Mode identity path that fetches resolved assistant identity, stores render-safe avatar state, and displays it under the existing voice-state motion without parsing config directly in Swift.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure ui.assistant.avatar and open macOS Talk Mode; current main still constructs TalkOrbView and the macOS Talk runtime never requests agent.identity.get/avatar state.

## Solution Assessment

Yes. Reusing the gateway's agent.identity.get/avatar resolution is the narrowest maintainable direction because it preserves identity precedence, avatar validation, and source redaction while keeping the default orb fallback.
