## Summary

Keep this open. Current main still renders the macOS Talk Mode overlay as the generic `TalkOrbView` and only carries `ui.seamColor` through the Talk config path, while the existing gateway identity API already resolves assistant avatar metadata that the macOS overlay does not consume.

## What This Changes

Request the macOS Talk Mode overlay to render the configured assistant avatar when available, while preserving the existing animated voice-state orb visuals and fallback behavior.

## Best Possible Solution

Add a narrow macOS Talk Mode identity layer that asks the gateway for the active session or default agent identity, stores only render-safe avatar state needed by the overlay, renders a circular cropped avatar inside the existing rings/arcs when available, and falls back to the current orb on missing identity, unsupported avatar type, load failure, or auth failure.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure `ui.assistant.avatar`, open macOS Talk Mode, and current main has no code path that fetches identity or renders an avatar in the overlay.

## Solution Assessment

Yes. Reusing the existing `agent.identity.get` result and avatar route is the narrowest maintainable fix; parsing `ui.assistant.avatar` directly in Swift would duplicate gateway identity precedence and validation.
