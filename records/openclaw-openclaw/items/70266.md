## Summary

Keep this open: current main and v2026.6.8 still render the native macOS Talk Mode overlay as the generic Circle-based TalkOrbView, while existing assistant-avatar resolution is available elsewhere but not bridged into the native overlay.

## What This Changes

Request to have the native macOS Talk Mode overlay render the configured assistant avatar while preserving the orb fallback and voice-state animation.

## Best Possible Solution

Route an already-resolved safe assistant avatar into TalkOverlayView and render it as a circular center layer inside the existing pulse/ring visuals, with the current orb when no avatar resolves.

## Reproduction Assessment

Yes. Source inspection shows TalkOverlayView always builds TalkOrbView, and the macOS talk.config path only carries ui.seamColor, so a configured assistant avatar has no native overlay rendering path on current main.

## Solution Assessment

Yes directionally, but it needs maintainer UX approval. Reusing the existing safe avatar resolver and preserving the current rings/fallback is the narrow maintainable approach; the open decision is how the macOS app should receive, cache, and render the image source.
