## Summary

Keep this open: current main and the latest release still render the native macOS Talk Mode overlay as the generic orb, and the already-closed Control UI identity work does not cover this native overlay surface.

## What This Changes

Request macOS Talk Mode to render the configured assistant avatar in the native overlay while preserving the existing orb animation and fallback behavior.

## Best Possible Solution

Route a resolved safe assistant avatar into TalkOverlayView and render it as a circular center layer inside the existing pulse/ring states, with the current orb when no avatar resolves.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current TalkOverlayView always builds TalkOrbView and talk.config only exposes ui.seamColor, so setting ui.assistant.avatar has no native overlay path. I did not run a live macOS visual repro in this read-only review.

## Solution Assessment

Yes directionally: reusing the existing resolved assistant avatar and preserving the current rings/fallback is the narrow maintainable shape. The remaining maintainer decision is how the macOS app should receive and cache the safe resolved image.
