## Summary

Keep this open. Current main still renders the macOS Talk Mode overlay through the generic orb path and has no macOS identity/avatar fetch or render state, while the gateway already exposes resolved assistant avatar metadata and no canonical PR appears to own this exact overlay request.

## What This Changes

Request the macOS Talk Mode overlay to render the configured assistant avatar, with the existing animated orb as fallback.

## Best Possible Solution

Add a small macOS Talk Mode identity path that calls `agent.identity.get`, normalizes a render-safe avatar URL/image state, displays it clipped inside the existing overlay motion, and falls back to `TalkOrbView` when unavailable or unsupported.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure `ui.assistant.avatar` and open macOS Talk Mode; current main's overlay only constructs `TalkOrbView`, and the Talk Mode config/runtime paths never request `agent.identity.get` or store avatar state.

## Solution Assessment

Yes. Reusing the gateway's `agent.identity.get` result and avatar route is the narrowest maintainable direction because it preserves identity precedence and avatar validation instead of parsing `ui.assistant.avatar` directly in Swift.
