## Summary

Keep this open. Current main still renders the macOS Talk Mode overlay through the generic `TalkOrbView` path and does not fetch or store assistant avatar identity for that surface, while the gateway already exposes resolved assistant avatar metadata.

## What This Changes

Request the macOS Talk Mode overlay to render the configured assistant avatar when available while preserving the existing animated voice-state visuals and fallback behavior.

## Best Possible Solution

Add a narrow macOS Talk Mode identity layer that asks the gateway for resolved assistant identity, stores only render-safe avatar state for the overlay, renders a circular clipped avatar inside the existing rings/arcs when available, and falls back to the current orb on missing identity or load failure.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `ui.assistant.avatar`, open macOS Talk Mode, and current main has no overlay, config, or runtime path that fetches identity or renders an avatar.

## Solution Assessment

Yes. Reusing the existing `agent.identity.get` result and avatar route is the narrowest maintainable direction; parsing `ui.assistant.avatar` directly in Swift would duplicate gateway identity precedence and validation.
