## Summary

Keep open. Current main and v2026.5.18 still route the macOS Talk Mode overlay through the native orb/accent-color path, while the existing assistant-avatar identity work remains separate in the Control UI/gateway identity surface.

## What This Changes

Request that macOS Talk Mode render the configured assistant avatar in the voice overlay, while preserving the current animated orb fallback when no avatar is available.

## Best Possible Solution

Add an additive native Talk Mode identity path that reuses resolved gateway assistant identity or an equivalent validated talk.config payload, renders a safe circular avatar inside the existing motion cues, and keeps the current orb fallback.

## Reproduction Assessment

Yes for the missing-capability claim: source inspection on current main and v2026.5.18 shows assistant avatar data cannot reach TalkOverlayView because the native path only carries seamColorHex into the orb renderer. This was not live-reproduced visually on macOS.

## Solution Assessment

Yes as a product direction, but not as automated cleanup. The maintainable fix should reuse the resolved assistant identity/avatar contract and preserve the existing animated orb fallback rather than adding a separate raw-config parser in Swift.
