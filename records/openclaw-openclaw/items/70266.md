## Summary

Keep open. Current main still renders the macOS Talk Mode overlay through the generic orb path, while assistant/avatar identity is resolved in gateway and Control UI code but is not consumed by the macOS overlay.

## What This Changes

Request macOS Talk Mode render the configured assistant avatar inside the existing voice overlay visuals with fallback to the default orb.

## Best Possible Solution

Add a small macOS Talk Mode identity path that reuses agent.identity.get, renders a safe circular avatar inside the existing overlay motion, and falls back to TalkOrbView when unavailable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure ui.assistant.avatar and open macOS Talk Mode; the current overlay only builds TalkOrbView and the macOS Talk runtime never fetches assistant identity/avatar state.

## Solution Assessment

Yes. Reusing the gateway's agent.identity.get result and avatar route is the narrowest maintainable direction because it preserves existing identity precedence, validation, and avatar-source metadata.
