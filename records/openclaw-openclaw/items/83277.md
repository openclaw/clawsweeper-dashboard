## Summary

Keep open: current main still has the exact generic web-login failure path when no QR-capable channel plugin is loaded, while WhatsApp is now an official external plugin and `wacli` is documented as a separate skill/tool rather than the normal channel runtime.

## What This Changes

Report that the Windows dashboard WhatsApp QR flow returns `web login provider is not available` after configuring `channels.whatsapp.enabled`, despite `wacli` being installed and authenticated.

## Best Possible Solution

Keep WhatsApp as an external plugin, but make the dashboard/web-login path detect a configured missing official WhatsApp plugin and surface the existing install or `doctor --fix` guidance.

## Reproduction Assessment

Yes for the reported error path from source inspection: configured WhatsApp without a loaded QR-capable plugin makes `web.login.start` return the exact generic provider-unavailable error. I did not run the Windows scheduled-task gateway flow.

## Solution Assessment

No, treating `wacli` as the channel provider would be the wrong fix. The narrow maintainable fix is to reuse the existing official-external-plugin repair hint in the web-login/dashboard path.
