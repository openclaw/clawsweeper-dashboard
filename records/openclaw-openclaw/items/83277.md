## Summary

Keep open: current main still has the reported generic web-login failure when no loaded QR-capable channel plugin is present, and WhatsApp is an official external plugin with existing install/doctor guidance that this path does not surface.

## What This Changes

Fix the WhatsApp dashboard Show QR path so a configured but missing WhatsApp channel plugin does not return the generic `web login provider is not available` error.

## Best Possible Solution

Gateway web-login should detect a configured official external WhatsApp channel with no loaded owner plugin and return the existing install or `doctor --fix` hint before falling back to the generic provider-unavailable error.

Yes for source-level reproduction: with no loaded channel plugin advertising `web.login.start` or `web.login.wait`, current main returns the exact `web login provider is not available` error. I did not run the Windows scheduled-task gateway flow live.

## Solution Assessment

Yes for the repair direction: reusing the existing official-external-plugin repair hint is the narrow maintainable fix. Treating `wacli` as the channel provider would be the wrong solution because `wacli` is a separate skill/tool surface.

## Repair Work Prompt

Fix the current-main WhatsApp dashboard/web-login missing-plugin path for https://github.com/openclaw/openclaw/issues/83277. When `web.login.start` or `web.login.wait` cannot find a loaded QR-capable provider, inspect configured official external channels and, for configured WhatsApp with no owner plugin, return an actionable INVALID_REQUEST message using the existing `resolveMissingOfficialExternalChannelPluginRepairHint()` guidance (`openclaw plugins install clawhub:@openclaw/whatsapp` or `openclaw doctor --fix`) instead of only `web login provider is not available`. Keep WhatsApp external; do not make `wacli` a channel runtime dependency, do not install packages from the gateway RPC, and do not eagerly load plugin runtime. Add focused regression coverage for missing WhatsApp plugin plus the existing loaded-provider behavior; add a concise changelog fix entry if the repair branch is user-facing.
