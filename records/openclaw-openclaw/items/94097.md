## What This Changes

The PR adds an early return in `runHeartbeatOnce` to suppress raw heartbeat text when visibleReplies resolves to `message_tool` and `heartbeat_respond` was not called.

## Best Possible Solution

Use one heartbeat delivery decision that treats ordinary raw final text as private whenever response-tool delivery is active, preserves marked runtime/status notices, and adds focused tests plus redacted live channel proof.

## Reproduction Assessment

Yes. Source inspection shows current main sends raw `replyPayload` through `sendDurableMessageBatch` after a response-tool heartbeat when no `heartbeatToolResponse` is present, and the PR guard can be evaluated against the existing metadata and Codex/default tests.

## Solution Assessment

No. The fix is in the right function but should key off the active response-tool delivery contract and preserve marked runtime/status notices instead of adding a visibleReplies-only early return.
