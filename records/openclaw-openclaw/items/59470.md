## Summary

Keep open. Current main still has the reported plugin gateway timeout path: return-only plugin gateway handlers are awaited but their returned payload is not converted into an RPC response, and there is no linked closing PR for this issue.

## What This Changes

The issue reports that plugin gateway methods can complete successfully while `openclaw gateway call` times out instead of returning the method result.

## Best Possible Solution

Add a backwards-compatible result adapter in gateway method dispatch so returned JSON-compatible payloads produce successful RPC responses while existing `respond(...)` handlers keep working.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: register a plugin gateway handler that returns a payload without calling `respond(...)`; current `handleGatewayRequest` awaits it but sends no response frame.

## Solution Assessment

Yes. The narrow maintainable fix is in gateway method dispatch and SDK typing, with regression coverage for returned payloads and double-response prevention rather than increasing timeout budgets.

## Repair Work Prompt

Fix the plugin gateway method return-value timeout in OpenClaw. Current main awaits plugin `registerGatewayMethod` handlers but only sends an RPC response when the handler calls `respond(...)`; handlers that return a JSON-compatible payload complete successfully but leave `openclaw gateway call` waiting until timeout. Add a backwards-compatible adapter in the gateway method dispatch path so a handler return value sends `respond(true, payload)` only if the handler did not already respond. Preserve existing callback-style `respond(...)` behavior, error handling, authorization, rate limiting, and gateway startup unavailable behavior. Update public SDK types/docs if the return-value contract becomes supported. Add focused regression coverage for a return-only plugin handler and for a callback-style handler that must not double-respond. Do not change gateway auth/scopes or only increase timeout budgets.
