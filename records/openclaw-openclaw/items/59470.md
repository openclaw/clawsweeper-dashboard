## Summary

Keep open. Current `main` and the latest release still invoke plugin gateway handlers without adapting a returned value into a Gateway RPC response, so a return-style plugin method can complete its work while `openclaw gateway call` waits until timeout.

## What This Changes

The issue asks for `openclaw gateway call` to return the result of plugin gateway methods that complete by returning a payload.

## Best Possible Solution

Add a backward-compatible return-value adapter in the gateway method dispatch path, update the SDK type/docs if needed, and cover both return-style handlers and existing `respond(...)` handlers with regression tests.

## Reproduction Assessment

Yes, from source. Register a plugin gateway method that returns a payload without calling `respond(...)`, then invoke it through `openclaw gateway call`; the dispatcher awaits the handler but never sends a response frame.

## Solution Assessment

Yes. The maintainable fix is a narrow gateway dispatch/SDK contract adapter, not a longer CLI timeout or a Discord-specific workaround.

In current main, plugin gateway handlers that return a payload without calling `respond(...)` can complete successfully while `openclaw gateway call` times out because `handleGatewayRequest` ignores the handler return value. Add a backward-compatible adapter in the gateway method dispatch path so a JSON-compatible returned value sends a successful RPC payload when the handler has not already responded.
