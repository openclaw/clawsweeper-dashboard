## Summary

Keep open. Current `main` still has a plausible source-level path for this timeout: plugin gateway handlers are stored and invoked, but the RPC response is only sent when the handler calls `respond(...)`; a handler that completes side effects and returns a value can finish successfully while the CLI waits until its timeout. Repo fixtures also register gateway methods with async return values, so the reporter’s expectation is not isolated.

## What This Changes

The issue reports that `openclaw gateway call` invokes plugin gateway methods but times out instead of returning the method result.

## Best Possible Solution

Add a backwards-compatible plugin gateway handler result adapter: preserve the existing `respond(...)` callback path, track whether a response was sent, and for plugin gateway handlers that return a JSON-compatible value without responding, send that value as the successful RPC payload. Update the public handler type/docs and add a focused regression test for a plugin method that returns a value through `openclaw gateway call`; avoid changing gateway auth, scope classification, or timeout behavior.

## Reproduction Assessment

Yes. A plugin gateway method that completes work and returns a value without calling `respond(...)` should reproduce the timeout on current `main`; source inspection shows the server awaits the handler but never sends a response from its return value.

## Solution Assessment

Yes. A narrow compatibility fix in the gateway method dispatch/SDK contract is the maintainable direction because it preserves existing callback handlers while making return-value handlers, including existing repo fixtures, behave like normal RPC methods.
