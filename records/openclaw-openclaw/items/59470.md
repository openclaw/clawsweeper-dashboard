## Summary

Keep open. Current main and the latest release still await plugin gateway handlers without adapting returned values into RPC responses, so a return-style plugin method can complete its side effect while `openclaw gateway call` waits until its timeout.

## What This Changes

The issue asks for `openclaw gateway call` to return a successful result when a plugin gateway method completes by returning a value.

## Best Possible Solution

Add a backward-compatible gateway dispatch adapter that preserves explicit `respond(...)` handlers but sends a successful RPC payload when a plugin handler returns a JSON-compatible value without already responding.

## Reproduction Assessment

Yes, from source. Register a plugin gateway method that returns a payload without calling `respond(...)`, then invoke it through `openclaw gateway call`; current dispatch awaits the handler but never emits a response frame.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway method dispatch and SDK contract, not in increasing the CLI timeout or changing the Discord-specific plugin behavior.

In `src/gateway/server-methods.ts`, adapt plugin gateway handler return values into successful RPC responses when the handler returns a JSON-compatible value and did not already call `respond(...)`; preserve all existing callback-style `respond(...)` behavior and accepted/final multi-response flows.
