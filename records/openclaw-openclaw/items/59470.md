## Summary

Keep open. Current main still awaits plugin gateway handlers without emitting a response for returned values, and the latest release tag shows the same return-dropping path, so the reported timeout remains a narrow source-backed bug.

## What This Changes

The issue reports that `openclaw gateway call` times out for plugin gateway methods whose handlers complete successfully but return a value instead of sending an RPC response.

## Best Possible Solution

Add a backward-compatible gateway method result adapter that preserves existing `respond(...)` handlers while sending a successful RPC payload when a plugin handler returns a JSON-compatible value without responding.

## Reproduction Assessment

Yes. Register a plugin gateway method that returns a payload without calling `respond(...)`, then call it through `openclaw gateway call`; current source awaits the handler but never emits a response frame.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway method dispatch/SDK contract, not in increasing gateway timeouts, changing Discord behavior, or broadening auth/scopes.

Preserve existing callback-style handlers, do not double-respond when `respond(...)` was already called, and do not change gateway timeout defaults, auth scopes, Discord behavior, or unrelated plugin loading.
