Keep open. Current main still completes plugin Gateway RPCs only when a handler calls `respond(...)`, while repo fixtures use return-style plugin methods, so the reported `openclaw gateway call` timeout remains a valid source-reproducible runtime bug.

## What This Changes

The issue reports that return-style plugin gateway methods can complete their side effects but `openclaw gateway call` still times out because no RPC response is delivered.

## Best Possible Solution

Add a backward-compatible return-value adapter in gateway method dispatch, align the SDK type/docs, and cover both return-style and existing `respond(...)` handlers with regression tests.

## Reproduction Assessment

Yes, from source. Register a plugin gateway method that returns a payload without calling `respond(...)`; `handleGatewayRequest` awaits it but no response frame is emitted, so the client waits until timeout.

Yes. The maintainable fix is a narrow dispatcher/SDK contract adapter, not a longer CLI timeout or a Discord-specific workaround.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/59470. On current main, `openclaw gateway call` can time out when a plugin gateway method returns a payload but does not call `respond(...)`: `handleGatewayRequest` awaits the handler inside `withPluginRuntimeGatewayRequestScope(...)` and never adapts the return value into a response frame. Implement a backward-compatible adapter in the gateway method dispatch/handler contract so a non-undefined returned value sends a success payload only when the handler has not already responded. Preserve existing `respond(...)` handlers, error propagation, request scope behavior, and accepted/final or multi-response flows. Update `GatewayRequestHandler` types and public plugin SDK docs if needed. Add focused regression coverage for return-style plugin methods and double-response prevention. Do not lengthen CLI timeouts or add Discord-specific behavior. Include a changelog entry if the fix changes user-visible plugin Gateway behavior.
