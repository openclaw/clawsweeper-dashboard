- `P2`: The bug breaks plugin gateway RPC reliability and operator trust in `openclaw gateway call`, but the blast radius is limited to plugin gateway methods rather than the whole gateway runtime.

## Summary

Keep open. Current `main` still has a source-backed timeout path for return-only plugin gateway handlers: the dispatcher awaits the handler but never turns its returned payload into a Gateway RPC response, and the live issue has no closing PR or exact canonical duplicate.

## What This Changes

The issue requests that `openclaw gateway call` return a plugin gateway method result when the method executes successfully instead of timing out.

## Best Possible Solution

Add a backward-compatible return-value adapter in the gateway dispatch path, preserve explicit `respond(...)` handlers, update the SDK type/docs contract, and cover both return-style and callback-style plugin methods with regression tests.

## Reproduction Assessment

Yes, from source. Register a plugin gateway method that returns a payload without calling `respond(...)`; `handleGatewayRequest` awaits the handler but emits no response frame, so `callGateway` can wait until its 10000ms timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adapt returned plugin handler values into a Gateway RPC success response while preserving existing callback-style handlers that already call `respond(...)`.

Fix the source-reproducible plugin gateway RPC timeout from https://github.com/openclaw/openclaw/issues/59470. A plugin handler registered through `api.registerGatewayMethod(...)` that returns a JSON-serializable payload without calling `respond(...)` currently runs to completion but leaves `openclaw gateway call` waiting until the 10000ms timeout. Keep the fix bounded to the gateway method dispatch/SDK contract: adapt non-void returned handler values into a normal Gateway RPC success response, preserve handlers that explicitly call `respond(...)`, and prevent double responses.
