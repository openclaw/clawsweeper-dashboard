## Summary

Keep open. Current `main` and the latest tagged release still await plugin gateway handlers without adapting returned payloads into RPC responses, so a handler that completes by returning a value can leave `openclaw gateway call` waiting until its timeout.

## What This Changes

The issue reports that `openclaw gateway call` times out when a plugin gateway handler completes work but returns a result instead of producing an RPC response.

## Best Possible Solution

Add a backward-compatible result adapter in the gateway method dispatch path: preserve existing `respond(...)` handlers, but send a successful RPC payload when a plugin handler returns a JSON-compatible value without already responding.

## Reproduction Assessment

Yes, from source. Register a plugin gateway method that returns a payload without calling `respond(...)`, then invoke it through `openclaw gateway call`; current dispatch awaits the handler but never emits a response frame.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway method dispatch/SDK contract, not in increasing the CLI timeout or changing the Discord-specific plugin behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/59470 by making plugin gateway method return values produce RPC responses. Keep the change narrow: adapt the result in the gateway method dispatch path so handlers that call `respond(...)` keep working unchanged, handlers that return a JSON-compatible value without responding send `ok: true` with that payload, and thrown/rejected handler errors continue to flow through the existing request failure path. Do not solve this by lengthening gateway timeouts or adding a new config option. Update the public handler type/docs if the return-value contract becomes explicit, add focused regression coverage for a handler returning `{ ok: true }` without calling `respond`, and add a changelog entry because the CLI/plugin behavior is user-visible. Be careful not to double-send responses for handlers that already call `respond(...)` or for accepted/final response patterns.
