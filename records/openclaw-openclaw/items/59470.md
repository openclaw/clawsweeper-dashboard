## Summary

Keep open. Current main and the latest release still have the source-backed timeout path: plugin gateway handlers are awaited, but a returned payload is not converted into a Gateway RPC response.

## What This Changes

The issue reports that return-style plugin gateway methods can complete successfully while `openclaw gateway call` still times out waiting for a response.

## Best Possible Solution

Add a backward-compatible return-value adapter in the gateway dispatch path, align the SDK type/docs with the supported contract, and cover both return-style and existing `respond(...)` handlers with regression tests.

## Reproduction Assessment

Yes, from source. Register a plugin gateway method that returns a payload without calling `respond(...)`; `handleGatewayRequest` awaits the handler but no response frame is sent, so the CLI waits until timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adapt returned plugin handler values into a Gateway RPC success response while preserving callback-style handlers that already call `respond(...)`.
