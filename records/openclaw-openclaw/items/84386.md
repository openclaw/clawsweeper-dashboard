## Summary

Keep open: current main still has a Codex account/status active-profile path that can choose stale `lastGood` ahead of an explicit Codex auth order, and the current-build report is not covered by the closed refresh-token-reused fix or the broader primary-first policy request.

## What This Changes

The issue asks OpenClaw to make a fresh Codex OAuth profile effective after re-login and explicit auth-order selection instead of continuing to choose stale `lastGood` default state.

## Best Possible Solution

Make explicit Codex auth order authoritative over stale provider `lastGood` in the affected active-profile/status path, or clear provider `lastGood` when setting a conflicting explicit order, with regression coverage for both explicit-order recovery and current failover behavior without explicit order.

## Reproduction Assessment

Yes at source level: construct Codex auth state with order `[fresh, default]` and `lastGood.openai-codex = default`; the current Codex active-profile helper can return `lastGood` before the first ordered profile. I did not exercise a live OAuth login in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make explicit auth order override stale `lastGood` for the Codex active-profile/status path, or clear provider `lastGood` when setting a conflicting explicit order; a new primary-first policy mode belongs to the separate feature discussion.
