## Summary

Keep open: current main has fixed the refresh-token-reused fallback loop from https://github.com/openclaw/openclaw/issues/79021, but it still has an order/status path where Codex account resolution can prefer stale `lastGood` over an explicit fresh Codex auth order. The reporter supplied a current 2026.5.18 reproduction, and the remaining source path is narrow enough to fix.

## What This Changes

The issue reports that Codex OAuth relogin creates a fresh named profile but stale `lastGood.openai-codex` can keep `openai-codex:default` selected after `models auth order set` puts the fresh profile first.

## Best Possible Solution

Make explicit Codex auth order the source of truth across Codex account/status and runtime-facing selection, using `lastGood` only as a fallback when there is no live account signal and no explicit ordered profile to prefer.

## Reproduction Assessment

Yes, at source level: with an explicit Codex order `[fresh, default]` and `lastGood.openai-codex=default`, `extensions/codex/src/command-account.ts` can still select `default` when the live account response does not identify the fresh profile. I did not run a live OAuth reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to align Codex account/status active-profile resolution with the existing order-first runtime contract, and only fall back to `lastGood` when explicit order and live account evidence do not identify a candidate.
