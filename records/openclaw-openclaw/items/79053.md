## Summary

Keep open: current main still has the reported active-requester dead-end, and the issue is paired with open fix PR https://github.com/openclaw/openclaw/pull/79059 using closing syntax. The newest issue comments add v2026.5.7 Telegram and Discord reproductions, so cleanup should wait for the linked PR to merge or for a replacement fix to land.

## What This Changes

The issue reports dropped subagent completion announcements when the parent requester session is active but idle or no longer consuming embedded Pi messages, especially after `sessions_yield`.

## Best Possible Solution

Review and land the narrow fallthrough fix in https://github.com/openclaw/openclaw/pull/79059 if CI and maintainer review agree, then track durable queueing or wrong-session routing as separate follow-ups if they remain.

## Reproduction Assessment

Yes, at source level. Set `expectsCompletionMessage` true, return an active requester session id, make `queueEmbeddedPiMessageWithOutcome` return `queued: false`, and current main returns undelivered before the later `callGateway` handoff can run.

## Solution Assessment

Yes for the central bug: letting the failed active-requester wake fall through to the existing requester-agent handoff is the narrowest contract-preserving direction. A raw `sendMessage` fallback would conflict with the current no-raw-child-output delivery contract, while durable queueing should be handled as a separate design if retry exhaustion still drops completions.
