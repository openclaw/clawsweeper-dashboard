## Summary

Keep open: current main still has the active-requester early return before the requester-agent handoff, and an open contributor PR with closing syntax is the active fix path. Recent comments also provide v2026.5.7 Telegram and Discord repro evidence, so this should not be swept as stale or already fixed.

## What This Changes

The issue reports dropped subagent completion announcements when an idle or yielded requester session cannot accept a queued embedded-run message.

## Best Possible Solution

Review and land the narrow fallthrough fix in https://github.com/openclaw/openclaw/pull/79059, then track any remaining durable-queue or wrong-session delivery behavior separately if it still reproduces.

## Reproduction Assessment

Yes, at source level: set `expectsCompletionMessage` true, provide an active requester session id, make embedded-run queueing fail with `not_streaming` or `no_active_run`, and current main returns before the `callGateway` handoff. The issue comments add live v2026.5.7 Telegram and Discord evidence, but I did not run a live push-channel repro in this read-only pass.

## Solution Assessment

Yes for the revised linked PR direction: removing the dead-end fallthrough is narrower than the original `sendMessage` fallback and preserves the requester-agent delivery contract from https://github.com/openclaw/openclaw/pull/78700. The original raw `sendMessage` fallback is not the best direction because it would bypass the requester-agent rewrite path.
