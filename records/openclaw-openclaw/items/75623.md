## Summary

Keep open. Current main still leaves a plausible generic bundled-channel runtime-dependency resolution gap: `json5` is staged and mirrored as intended, but the Slack/Telegram channel-entry load path can still bypass runtime-deps preparation and does not visibly consume the staged Jiti alias map used by the main plugin loader.

## What This Changes

Slack and Telegram bundled channel providers should resolve `json5` from the staged plugin-runtime-deps root when launched by the Gateway after repair and restart.

## Best Possible Solution

Fix the generic bundled channel runtime-dependency loading path so staged root dependencies and registered Jiti aliases are honored consistently for bundled channel entry/runtime loads, then cover Slack and Telegram through a packaged staged-deps smoke instead of adding channel-specific `json5` workarounds.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The reporter provided exact packaged startup steps plus direct-import controls, and current source inspection identifies the loader path that can diverge from direct Node resolution.

## Solution Assessment

Is this the best way to solve the issue? Yes. A generic bundled runtime-deps loader fix is the narrow maintainable path; inlining or special-casing `json5` for Slack/Telegram would mask the shared staged-dependency resolution bug.
