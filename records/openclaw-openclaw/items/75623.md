## Summary

Keep open. The report is a fresh, concrete packaged-runtime regression with multiple confirmations, and current `main` still shows the bundled channel entry/runtime paths opting out of runtime-deps preparation while the main plugin loader is the path that registers staged Jiti aliases. The closed unmerged #75666 is useful related context, but it did not land and does not make this issue obsolete.

## What This Changes

Slack and Telegram provider startup should resolve `json5` from the matching staged `plugin-runtime-deps/<version-hash>/node_modules` tree instead of crashing after Gateway startup.

## Best Possible Solution

Fix the generic bundled channel runtime-deps load path so mirrored root chunks used by Slack and Telegram resolve staged dependencies consistently, then cover it with a packaged staged-deps smoke. Avoid Slack- or Telegram-only `json5`/`grammy` workarounds unless the generic loader boundary is proven insufficient.

## Reproduction Assessment

Yes. The issue and comments give a high-confidence packaged reproduction path on 2026.4.29 with Slack/Telegram enabled, and current source inspection matches the reported `installRuntimeDeps: false` channel-entry boundary. Live reproduction was not run because this was a read-only review of the checked-out repository.

## Solution Assessment

Yes, the right direction is a narrow generic runtime-deps loader fix at the bundled channel entry/runtime boundary. Adding `json5` or `grammy` directly to unrelated core startup paths would only mask the shared staged-dependency resolution problem.

## Repair Work Prompt

Fix the Slack/Telegram packaged runtime-deps crash reported at https://github.com/openclaw/openclaw/issues/75623. Current main still has bundled channel entry/runtime loads passing `installRuntimeDeps: false`, while staged root chunks such as `dist/frontmatter-*.js` can import root mirrored deps like `json5`. Implement a generic bundled channel/runtime-deps fix, not a Slack- or Telegram-specific workaround: either make the mirrored dist root reliably resolve staged `node_modules` for native require/ESM/Jiti, or route channel-entry/runtime loading through the same staged root preparation and runtime alias contract used by the main plugin loader.
