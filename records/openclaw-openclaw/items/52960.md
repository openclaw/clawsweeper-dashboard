## Summary

Keep open. Current main now has a useful but coarse composer run-status chip, but it still does not provide the requested compact progress model with latest meaningful action, elapsed or last-activity timing, subagent/tool/finalizing phases, delivery-delay state, or reconnect-during-run history.

## What This Changes

The issue requests a compact Control UI run-progress surface for long and multi-agent tasks, including phase, latest action, timing, and reconnect visibility.

## Best Possible Solution

Define an additive active-run activity contract and render a compact Control UI status strip that consumes explicit stream, tool, subagent, delivery, and reconnect signals without becoming a raw event log.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main only has a coarse three-state chip and not the requested active-run progress payload or renderer.

## Solution Assessment

Unclear until maintainers settle the UX/API boundary. The requested direction is sound, but the maintainable path is a shared activity contract plus compact UI rather than brittle client-side inference.
