## Summary

Keep open. Current main still has chat streaming, live tool cards, queued follow-ups, Stop/abort, compaction/fallback indicators, and websocket reconnect recovery behavior, but it does not expose the requested compact active-run progress model with phase, latest meaningful action, last activity, elapsed time, wait target, subagent summary, delivery delay, or reconnect-during-run visibility. The supplied discussion correctly treats merged PR #31192 as unrelated delivery-queue retry work rather than this Control UI feature.

## What This Changes

Request a compact Control UI active-run progress display for long or multi-agent tasks, including phase, latest action, last activity, elapsed time, waiting state, and reconnect visibility.

## Best Possible Solution

Define a small Gateway/Control UI active-run progress contract and render a low-noise strip near the thread or composer. It should derive phases from chat, agent lifecycle, tool, subagent, delivery, and reconnect signals; include focused Gateway and Control UI tests; and update the public Control UI docs once behavior lands.

## Reproduction Assessment

Yes, at source level. Any active Control UI chat run can only surface the existing busy, stream, queue, tool-card, fallback, and compaction UI because the state, protocol, controller, and render paths have no fields or props for the requested compact run-progress model.

## Solution Assessment

Yes, the requested direction is the right class of fix, but the implementation should be a narrow shared run-progress contract plus a compact UI, not raw event-log rendering or unrelated delivery retry changes.
