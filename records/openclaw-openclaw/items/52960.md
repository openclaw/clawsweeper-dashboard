## Summary

Keep open. Current main has a coarse composer run-status chip and session active-run flags, but it still lacks the requested compact progress model with latest meaningful activity, elapsed/last-activity timing, subagent/tool/finalizing phases, delivery-delay state, and reconnect-during-run context.

## What This Changes

The issue requests a compact Control UI run-progress display for long or multi-agent tasks, including phase, latest activity, timing, and reconnect/session context.

## Best Possible Solution

Define an additive active-run activity contract, then render a compact Control UI status strip from explicit stream, tool, subagent, delivery, and reconnect signals without turning it into a raw event log.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main only has coarse run/session status fields and not the requested active-run progress payload or renderer.

## Solution Assessment

Unclear until maintainers settle the UX/API boundary. The requested direction is sound, but the maintainable path is a shared activity contract plus compact UI rather than brittle client-side inference.
