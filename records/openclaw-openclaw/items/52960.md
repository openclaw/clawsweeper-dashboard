## Summary

Keep open: current main and v2026.6.8 still only provide coarse Control UI chat run status plus session-row lifecycle/tool activity signals, not the requested compact active-run progress strip with phase/latest activity/elapsed/reconnect context. The related gateway activity issue is the backend/API dependency, but this issue still owns the Control UI presentation scope.

## What This Changes

The issue requests a compact Control UI run-progress/status surface for long or multi-agent runs.

## Best Possible Solution

Define or reuse a canonical gateway session activity/attention contract, then render a compact Control UI status strip near the composer or thread without raw logs or message-text inference.

## Reproduction Assessment

Yes, at source level: current main shows only coarse active/terminal composer status, tool-scoped activity entries, and partial session-row lifecycle fields. This is a feature gap rather than a failing documented command or test reproduction.

## Solution Assessment

No, a UI-only implementation is not the best path. The maintainable solution should coordinate with the gateway/session activity design so the UI does not infer subagent/tool wait state locally.
