## Summary

Keep open. Current main has a narrow composer run-status chip, but it only covers In progress/Done/Interrupted and does not expose the requested active-run progress model: latest meaningful action, last-activity/elapsed timing, subagent/tool/finalizing states, or reconnect-during-run history.

## What This Changes

Add a compact, low-noise Control UI run-progress surface for long or multi-agent tasks with phase, latest action, timing, wait-state, and reconnect visibility.

## Best Possible Solution

Define an additive active-run activity contract and render a compact Control UI status strip that consumes explicit stream, tool, subagent, delivery, and reconnect signals without becoming a raw event log.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Static inspection gives a high-confidence gap: current main has a coarse three-state composer chip and internal busy/session fields, but not the requested active-run progress payload or renderer.

## Solution Assessment

Unclear until maintainers settle the UX/API boundary. The requested direction is reasonable, but the maintainable path is likely a shared activity contract plus compact UI, coordinated with https://github.com/openclaw/openclaw/issues/39127 and the narrower status-chip work.
