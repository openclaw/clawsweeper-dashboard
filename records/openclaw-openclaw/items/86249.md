## Summary

Keep open. Current main and the latest release still let REM build candidate truths from the whole short-term lookback store and rank by accumulated recall, instead of consuming the light-staged keys, so the reported memory-core handoff bug remains source-supported and has no canonical merged fix yet.

## What This Changes

The issue asks memory-core dreaming to make REM consume the candidates staged by light sleep so deep promotion can receive combined light/rem boosts for current material.

## Best Possible Solution

Make the full light -> REM -> deep sweep carry the light-staged candidate keys into REM candidate truth selection, while preserving a deliberate fallback for standalone REM preview/trigger paths.

## Reproduction Assessment

Yes at source level: current main reads all REM lookback entries, ranks them without checking light-staged keys, and only records `remHits` for that independent selection. A live multi-day workspace reproduction was not run in this read-only review.

## Solution Assessment

Yes: the maintainable repair is to thread same-sweep light-staged keys or equivalent phase-signal state into REM selection, with focused tests for old high-recall entries competing against fresh light-staged entries. The fix should avoid adding user-facing config and should keep direct REM preview behavior intentional.
