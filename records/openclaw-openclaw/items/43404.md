## Summary

Keep open: the PR targets a real reminder-guard bug, but the current patch can over-suppress warnings for non-canonical session keys and still lacks after-fix real behavior proof.

## What This Changes

The PR preserves `successfulCronAdds` across compaction, expands reminder-guard cron matching to enabled isolated same-agent jobs, adds guard tests, and changes one doctor E2E fixture `git add` command.

## Best Possible Solution

Land an updated version or narrow replacement that preserves cron-add evidence across compaction and matches isolated crons only with trusted same-agent session evidence, with regression tests and real Telegram or equivalent runtime proof.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.12 reset `successfulCronAdds` during compaction and only match existing cron rows by `sessionKey`, so the reported enabled isolated cron row cannot suppress the note after counter reset.

## Solution Assessment

No, not as currently proposed. The compaction-counter and isolated-cron direction is narrow, but the caller should pass `agentId` only after `parseAgentSessionKey(sessionKey)` succeeds, and the external PR still needs after-fix real behavior proof.
