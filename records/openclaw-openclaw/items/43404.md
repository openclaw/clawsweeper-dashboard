## Summary

Keep open. Current main still has the underlying reminder-guard bugs, and this PR is a useful narrow fix, but the latest branch is not mergeable as-is because the runner call adds an unresolved resolver reference and still derives a default `main` agent id from missing or non-agent session keys. ## What This Changes

The branch expands the unscheduled-reminder guard to match enabled isolated cron jobs by `agentId`, preserves `successfulCronAdds` across compaction retry, and adds reminder-guard regression coverage.

## Best Possible Solution

Merge this contribution or a narrow replacement after the runner derives an agent id only from trusted session identity, the resolver reference is imported or replaced with an existing scoped helper, the compaction counter remains preserved, and focused regression tests plus a changelog entry land with it.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original issue: isolated cron jobs cannot match the guard, and compaction clears the successful cron-add counter. The PR regression is also reproducible from the diff by invoking `runReplyAgent` without a reliable `sessionKey` while an enabled isolated `main` cron exists.

## Solution Assessment

No. The matcher and counter direction is the right narrow approach, but the current PR is not the best mergeable solution until it fixes the missing resolver import and prevents absent or malformed session keys from becoming a default `main` agent match.
