## Summary

Keep open: the PR targets a real source-reproducible reminder-guard bug, but the current branch still has a correctness blocker in the caller-side agent-id fallback and lacks required real behavior proof from an external contributor.

## What This Changes

The PR preserves `successfulCronAdds` across compaction retries, extends reminder-guard cron matching to isolated same-agent jobs, adds focused guard tests, and tweaks one doctor E2E git fixture command.

## Best Possible Solution

Update this PR or a narrow replacement to preserve cron-add evidence across compaction, derive `agentId` only from canonical parsed session keys, add caller-level regression coverage, and attach redacted real reminder proof before merge.

## Reproduction Assessment

Yes, at source level. Current main resets `successfulCronAdds` during compaction and only matches existing cron rows by `sessionKey`, so an enabled isolated cron without a session key cannot suppress the note after the counter is lost.

## Solution Assessment

No, not as currently proposed. Preserving the counter and adding isolated-cron matching are the right direction, but the caller should pass `agentId` only when the session key parses as canonical so missing or malformed sessions remain fail-closed.
