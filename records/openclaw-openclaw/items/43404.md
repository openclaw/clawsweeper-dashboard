## Summary

## What This Changes

The PR preserves `successfulCronAdds` across compaction retries, adds isolated-cron matching by `agentId` in the reminder guard with tests, and narrows one doctor E2E fixture `git add` command.

## Best Possible Solution

Land an updated version or narrow replacement that preserves cron-add evidence across compaction and matches isolated crons only when same-agent evidence comes from a canonical session key, with focused regression tests, conflict resolution, and redacted runtime proof.

## Reproduction Assessment

Yes, at source level. Current main resets `successfulCronAdds` during compaction and checks existing cron coverage by `sessionKey` only, so an enabled isolated cron row cannot suppress the note after the counter is lost.

## Solution Assessment

No, not as currently proposed. The fix direction is narrow, but the caller should only pass `agentId` when the session key parses as canonical so missing or malformed sessions remain fail-closed.
