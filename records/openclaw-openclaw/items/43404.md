## Summary

## What This Changes

The PR preserves `successfulCronAdds` across compaction retry, adds isolated same-agent cron matching to the reminder guard with tests, and narrows one doctor E2E fixture `git add` command.

## Best Possible Solution

Land an updated version or narrow replacement that preserves cron-add evidence across compaction and matches isolated crons only when same-agent evidence comes from a canonical session key, with focused regression tests and real Telegram or equivalent runtime proof.

## Reproduction Assessment

Yes, at source level. Current main resets `successfulCronAdds` during compaction and only matches existing cron rows by `sessionKey`, so an enabled isolated cron row cannot suppress the note after the counter is lost.

## Solution Assessment

No, not as currently proposed. The fix direction is narrow, but the caller should pass `agentId` only from canonical session-key evidence so missing or malformed sessions remain fail-closed.
