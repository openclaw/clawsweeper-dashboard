## Summary

Keep open: the branch addresses a plausible NFS-backed SQLite crash mode, but it is not ready because the proposed toggle only disables database-file mmap while WAL shared memory remains active, the new config/SDK contract artifacts are incomplete, and the external PR still lacks after-fix real NFS proof.

## What This Changes

The PR adds an `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap` toggle, applies `PRAGMA mmap_size = 0` at task and memory SQLite opens, and exports a new `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that handles WAL shared memory before first database access, then align config/schema docs, SDK metadata/baselines, tests, and redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: source inspection and SQLite's WAL contract show the PR only disables database-file mmap while WAL shared memory remains in use. The original NFS SIGBUS crash was not live-reproduced here and has no after-fix real-environment proof.

## Solution Assessment

No. A `mmap_size=0` toggle is too narrow for the stated WAL/NFS crash mode; the safer solution is an explicit WAL/locking storage policy plus generated contract updates.
