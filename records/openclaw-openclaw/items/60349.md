## Summary

Keep this PR open, but it is not merge-ready: the proposed mmap toggle does not cover SQLite WAL shared-memory mapping, lacks after-fix NFS runtime proof, adds unsettled public config and SDK surfaces, and is currently merge-conflicting with main's centralized WAL setup.

## What This Changes

The PR adds an `OPENCLAW_SQLITE_MMAP`/`storage.sqliteMmap` toggle, applies `PRAGMA mmap_size = 0` at task and memory SQLite opens, and exposes a new `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode in shared WAL setup, align the public config and SDK contract artifacts, and require redacted real NFS runtime proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head plus SQLite's WAL contract shows WAL shared memory remains active while the helper only applies `mmap_size=0`. No for the original production SIGBUS crash: I did not live-reproduce an NFS-backed gateway.

## Solution Assessment

No. The current PR is too narrow for the stated WAL/NFS failure mode; the maintainable solution belongs in shared SQLite WAL policy with explicit concurrency semantics and aligned public config/API artifacts.
