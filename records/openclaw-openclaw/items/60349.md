## Summary

Keep open: the PR targets a real storage pain point, but the latest head is still a partial NFS mitigation because it only toggles database-file mmap while the affected stores remain in WAL mode, and it also lacks required generated contract updates and after-fix real behavior proof.

## What This Changes

The PR adds an env/config-controlled SQLite `mmap_size` toggle, applies it to task and memory SQLite open paths, and exposes a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that covers WAL shared-memory and locking semantics across all affected stores, then align config schema/docs, SDK artifacts, tests, and redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection shows current affected stores still enter WAL mode while the PR head only applies `PRAGMA mmap_size = 0`. A live NFS SIGBUS reproduction was not run or provided.

## Solution Assessment

No. The current patch is a partial database-file mmap toggle; the maintainable fix needs an explicit storage mode that handles WAL shared memory/locking and keeps generated config and SDK contracts in sync.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Cover WAL shared memory in the NFS storage mode:** `src/infra/node-sqlite.ts:41-44`
  - body: This helper only sets `PRAGMA mmap_size = 0`, but the affected stores still enable WAL through `configureSqliteWalMaintenance`. SQLite's WAL wal-index is separate mmap-backed shared memory and does not work over network filesystems, so `OPENCLAW_SQLITE_MMAP=0` can still leave the reported `-shm` SIGBUS path exposed. The fix needs to change the WAL/locking policy or avoid presenting this as NFS-safe.
  - confidence: 0.91
- **[P2] Regenerate the config docs baseline:** `src/config/zod-schema.ts:959-963`
  - body: Adding `storage.sqliteMmap` to `OpenClawSchema` without updating the generated config baseline leaves the config docs/check surface stale. Current main wires `config:docs:check` to the generated baseline hash, so the new public key needs the matching generated artifact update.
  - confidence: 0.84
- **[P2] Update the SDK API contract for the new subpath:** `src/plugin-sdk/infra-sqlite.ts:1`
  - body: `openclaw/plugin-sdk/infra-sqlite` is a new public SDK subpath, but the PR does not update the SDK API baseline or subpath documentation. That leaves the published plugin SDK contract out of sync with the new package export.
  - confidence: 0.86
