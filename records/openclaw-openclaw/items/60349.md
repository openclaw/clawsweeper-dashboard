## Summary

Keep open: current main does not implement the requested NFS mitigation, and the PR remains merge-blocked because `PRAGMA mmap_size=0` does not remove the WAL `-shm` mmap path described by the report; generated public config/SDK artifacts and changelog are also missing.

## What This Changes

The PR adds env/config controls to apply `PRAGMA mmap_size = 0` at task and memory SQLite open sites and exposes a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define and document an NFS-safe SQLite storage mode that handles WAL shared-memory/locking for the task registry, task-flow registry, memory sync store, and QMD manager, then align config schema, SDK artifacts, tests, docs, and changelog.

## Reproduction Assessment

Yes for the review blockers: the PR code path only sets `mmap_size` and the affected databases still enter WAL mode. A live NFS SIGBUS was not reproduced because this was a read-only review.

## Solution Assessment

No. `mmap_size=0` is a partial database-file mmap toggle, while the reported crash path is WAL shared memory; the safer fix needs an explicit storage mode with documented locking/WAL semantics.
