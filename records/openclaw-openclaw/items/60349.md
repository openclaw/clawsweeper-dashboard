## Summary

Keep open: this is a plausible storage reliability fix, but the current PR is not merge-ready because the mmap toggle does not make WAL shared memory NFS-safe, generated config/SDK contract updates are incomplete, and the external contributor has not provided after-fix real NFS proof.

## What This Changes

The PR adds an `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap` toggle that runs `PRAGMA mmap_size = 0` on task and memory SQLite connections and exposes an `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that handles WAL shared memory before first database access, then align config/schema docs, SDK metadata, tests, and redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: source inspection and SQLite's official WAL contract show the patch only disables database-file mmap while WAL shared memory remains in use. The original SIGBUS crash was not live-reproduced here and has no after-fix real-environment proof.

## Solution Assessment

No. A `mmap_size=0` toggle is too narrow for the stated WAL/NFS crash mode; a safer solution is an explicit WAL/locking storage policy plus generated config and SDK contract updates.
