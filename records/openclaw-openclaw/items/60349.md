## Summary

Keep open: the PR still has blocking correctness and contract issues. Disabling `PRAGMA mmap_size` does not cover SQLite WAL shared-memory on NFS, and the new public config/SDK surfaces are not fully aligned with the generated contract checks or real behavior proof gate.

## What This Changes

The branch adds `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap`, applies a SQLite mmap pragma to task and memory database opens, and exposes a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that handles WAL shared-memory and locking before first database access, then align config schema/docs, SDK baselines, tests, and redacted real behavior proof.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the affected stores still use WAL while the PR only applies `PRAGMA mmap_size = 0`, and SQLite's docs distinguish database-file mmap from WAL shared memory. No live NFS SIGBUS reproduction or after-fix validation was provided.

## Solution Assessment

No. The current patch is a partial database-file mmap toggle; the maintainable solution needs an explicit storage/WAL mode and generated contract updates rather than presenting this as NFS-safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Cover WAL shared memory before calling this NFS-safe:** `src/infra/node-sqlite.ts:41-44`
  - body: This helper only disables mapped reads of the main database file, while the affected connections still enter WAL via `configureSqliteWalMaintenance`. SQLite uses a separate mmap-backed `-shm` WAL index and documents that WAL does not work over network filesystems, so this flag can still leave the reported SIGBUS path exposed. Please change the WAL/locking storage policy before first WAL access or narrow the feature to what it actually mitigates.
