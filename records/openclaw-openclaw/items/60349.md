## Summary

Keep this PR open, but it is not merge-ready: the branch proposes a useful mitigation for NFS-backed SQLite storage, yet the current patch only disables database-file mmap while the affected stores still use WAL shared memory, and it also leaves generated config/SDK contract artifacts out of sync.

## What This Changes

The branch adds an `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap` control, applies `PRAGMA mmap_size = 0` at task and memory SQLite open sites, and exposes a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that covers WAL shared-memory and locking semantics across all current SQLite stores, then align config schema, docs, SDK artifacts, regression tests, and redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR only sets `mmap_size=0` while current affected stores still enter WAL mode. A live NFS SIGBUS reproduction was not performed in this read-only review.

## Solution Assessment

No. The proposed `mmap_size=0` toggle is a partial database-file mmap mitigation, but the reported crash path involves WAL shared memory and needs an explicit storage-mode decision plus generated artifact updates.
