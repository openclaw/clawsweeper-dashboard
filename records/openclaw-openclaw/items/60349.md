## Summary

Keep open: the PR targets a plausible NFS-backed SQLite crash, but the current patch is not safe to merge because `mmap_size=0` leaves WAL `-shm` shared-memory mapping in use, public config/SDK artifacts are not aligned, and after-fix real NFS proof is missing.

## What This Changes

Adds `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap` to run `PRAGMA mmap_size=0` at task and memory SQLite opens and exposes an `infra-sqlite` Plugin SDK helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that handles WAL shared-memory/locking before first database access, then align config docs, SDK contract metadata, tests, and redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: source inspection plus SQLite's WAL docs show the patch leaves WAL shared-memory in use. No for the original production SIGBUS crash: I did not live-reproduce an NFS-backed gateway and the PR has no after-fix proof.

## Solution Assessment

No. `mmap_size=0` is too narrow for the stated WAL/NFS crash mode; the maintainable fix is an explicit storage policy that covers WAL shared memory and documents the concurrency constraints.
