## Summary

Keep open: the PR targets a real availability concern, but the current patch is not safe to merge because `PRAGMA mmap_size = 0` does not cover SQLite WAL shared-memory, and the new public config/SDK surfaces and real NFS proof are still incomplete.

## What This Changes

The PR adds `OPENCLAW_SQLITE_MMAP` and `storage.sqliteMmap` controls that run `PRAGMA mmap_size = 0` in task and memory SQLite open paths, plus a new `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode in the shared WAL setup, document its concurrency limits, align config and SDK artifacts, and require redacted NFS runtime proof before merge.

## Reproduction Assessment

Yes for the review blocker: source inspection plus SQLite's WAL docs show the PR leaves WAL shared-memory active. No for the original production crash: I did not reproduce a live NFS-backed gateway and the PR has no after-fix runtime proof.

## Solution Assessment

No. `mmap_size=0` is too narrow for the stated WAL/NFS crash mode; the maintainable fix is an explicit storage policy in the shared WAL setup with aligned config/docs/tests and real NFS proof.
