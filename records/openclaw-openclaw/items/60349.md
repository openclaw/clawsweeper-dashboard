## Summary

Keep open: the NFS/SIGBUS problem is plausible, but this patch is not mergeable because it only toggles database-file mmap while WAL shared memory remains in use, generated public contracts are stale, and no after-fix real NFS proof was provided.

## What This Changes

The branch adds an `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap` toggle, applies `PRAGMA mmap_size = 0` at task and memory SQLite opens, and exposes a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that handles WAL shared memory and locking before first database access, then align config/schema docs, SDK baselines, tests, and redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the patch only applies `PRAGMA mmap_size = 0` while affected stores still use WAL, and SQLite documents WAL shared-memory as incompatible with network filesystems. No live after-fix NFS SIGBUS reproduction was provided.

## Solution Assessment

No. A database-file mmap toggle is too narrow for the stated WAL/NFS crash mode; the maintainable fix is an explicit storage/WAL policy plus generated contract updates.
