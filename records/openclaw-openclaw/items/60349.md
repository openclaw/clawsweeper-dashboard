## Summary

Keep open: current main does not already implement this feature, and the PR is not mergeable as-is. The branch’s main mitigation only applies `PRAGMA mmap_size = 0`, while the affected databases remain in WAL mode and SQLite’s WAL shared-memory file remains a separate mmap-backed surface; the branch also leaves generated config and SDK contract artifacts stale.

## What This Changes

The branch adds `OPENCLAW_SQLITE_MMAP` and `storage.sqliteMmap` controls, applies a SQLite mmap PRAGMA helper to task and memory SQLite connection sites, and exposes that helper through a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define a documented SQLite storage mode for NFS-backed deployments that actually addresses WAL shared-memory behavior across the task registry, task-flow registry, memory sync store, and QMD manager, then update config schema artifacts, SDK contract artifacts if a public subpath remains necessary, tests, docs, and changelog together.

## Reproduction Assessment

Yes for the review blockers: current main and the PR both leave the affected connections in WAL mode, and SQLite’s official docs show `mmap_size` does not remove the WAL `-shm` mmap surface. A live NFS SIGBUS was not reproduced because this was a read-only review.

## Solution Assessment

No. The proposed `mmap_size` control is a partial database-file mmap toggle, not the narrowest reliable fix for the reported WAL/NFS crash mode; a safer approach needs an explicit NFS-safe SQLite mode and regenerated public artifacts.
