## Summary

This PR should stay open because it addresses a plausible availability problem, but the proposed mitigation is still incomplete for the stated WAL/NFS crash mode. The branch adds a public storage knob while only disabling SQLite main-database mmap; WAL shared-memory semantics, generated/public contract artifacts, real NFS proof, and the current dirty merge state still need maintainer handling before merge.

## What This Changes

The PR adds an `OPENCLAW_SQLITE_MMAP`/`storage.sqliteMmap` switch, bridges config into the environment, applies `PRAGMA mmap_size = 0` at task and memory SQLite opens, and exposes a new `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Define one supported NFS-safe SQLite storage mode in the shared database-open/WAL setup path, with explicit locking and concurrency semantics, aligned config/docs/API artifacts, and redacted proof from an NFS-backed OpenClaw state directory before merge.

## Solution Assessment

No. The proposed toggle is a useful mitigation direction, but the best solution is a named storage mode that owns WAL shared memory, locking, concurrency, generated public artifacts, and real NFS proof together.
