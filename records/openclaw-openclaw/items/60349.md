## Summary

Keep open: the PR is a useful report and partial mitigation, but the latest patch is not mergeable because it only toggles SQLite database-file mmap while the affected stores still use WAL shared memory, generated config/SDK contract artifacts are missing, and the external PR has no real after-fix behavior proof.

## What This Changes

The PR adds `OPENCLAW_SQLITE_MMAP` and `storage.sqliteMmap` controls, applies `PRAGMA mmap_size = 0` at task and memory SQLite open sites, and exposes a new `openclaw/plugin-sdk/infra-sqlite` subpath.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that covers WAL shared-memory and locking semantics across all current SQLite stores, then align config schema/docs, SDK artifacts, tests, and redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection shows current affected stores still enter WAL mode while the PR only applies `PRAGMA mmap_size = 0`. A live NFS SIGBUS reproduction was not performed in this read-only review.

## Solution Assessment

No. The current PR is a partial database-file mmap toggle, but the reported crash path involves WAL shared memory, so the maintainable fix needs an explicit NFS-safe storage-mode policy plus generated contract updates.
