## Summary

Keep this PR open, but it is not merge-ready: the NFS availability concern is plausible, yet the patch only disables database-file mmap while WAL shared-memory remains active, adds public config and SDK surfaces that need contract alignment, lacks real NFS runtime proof, and is currently conflicting with main.

## What This Changes

This PR adds `OPENCLAW_SQLITE_MMAP` and `storage.sqliteMmap` controls, applies `PRAGMA mmap_size = 0` to task and memory SQLite opens, and exposes a new plugin SDK SQLite helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode in shared WAL setup, align the public config and SDK contracts if they remain, and require redacted real NFS runtime proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection plus SQLite's WAL contract show the patch leaves WAL shared memory active while only applying `mmap_size=0`. No for the original production SIGBUS crash: I did not live-reproduce an NFS-backed gateway.

## Solution Assessment

No. The current PR is too narrow for the stated WAL/NFS failure mode; the maintainable solution belongs in shared SQLite storage policy with explicit WAL, locking, and concurrency semantics.
