## Summary

Keep open: the NFS-backed SQLite crash report is plausible and current main does not already provide this toggle, but this PR is not merge-ready because it only disables database-file mmap while WAL shared-memory remains active, the public config/SDK surfaces are incomplete, and there is no after-fix real NFS proof.

## What This Changes

The PR adds `OPENCLAW_SQLITE_MMAP` and `storage.sqliteMmap` controls, applies `PRAGMA mmap_size = 0` at task and memory SQLite opens, and exposes a new `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Add an explicit NFS-safe SQLite storage mode in the shared WAL setup that applies mmap, WAL/shared-memory, and locking policy before first database access, then align config/SDK docs and artifacts with tests plus redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head plus SQLite’s WAL contract shows WAL shared memory remains active while the helper only applies `mmap_size=0`. No for the original production SIGBUS crash: I did not live-reproduce an NFS-backed gateway and the PR has no after-fix runtime proof.

## Solution Assessment

No. The current PR is too narrow for the stated WAL/NFS failure mode; the maintainable solution belongs in the shared SQLite WAL policy with explicit concurrency semantics and aligned public config/API artifacts.
