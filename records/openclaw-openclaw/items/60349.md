## Summary

Keep open: current main does not provide this NFS mitigation, but the PR is not merge-ready because the proposed toggle does not address WAL shared-memory mmap, generated config/SDK contract surfaces are incomplete, and after-fix real NFS proof is missing.

## What This Changes

The PR adds an `OPENCLAW_SQLITE_MMAP` / `storage.sqliteMmap` toggle that runs `PRAGMA mmap_size=0` at task and memory SQLite open sites and exposes an `infra-sqlite` SDK helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode that handles WAL shared-memory/locking before first database access, then align schema docs, SDK metadata, tests, and redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: source inspection plus SQLite's WAL docs show the patch leaves WAL shared-memory in use. No for the original production SIGBUS crash: I did not live-reproduce an NFS-backed gateway and the PR has no after-fix proof.

## Solution Assessment

No. `mmap_size=0` is too narrow for the stated WAL/NFS crash mode; the maintainable fix is an explicit storage policy that covers WAL shared memory and documents the concurrency constraints.
