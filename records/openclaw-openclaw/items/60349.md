## Summary

Keep open: the PR targets a plausible NFS-backed SQLite crash, but the patch is not merge-ready because `mmap_size=0` only disables main database file mmap while the changed WAL paths still use the `-shm` shared-memory mapping, the public config/SDK artifacts are incomplete, the branch conflicts with current `main`, and there is no after-fix NFS proof.

## What This Changes

The branch adds `OPENCLAW_SQLITE_MMAP` and `storage.sqliteMmap` controls, applies a SQLite mmap pragma at task and memory database opens, and exposes an `openclaw/plugin-sdk/infra-sqlite` helper.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage policy that covers WAL shared memory and locking before first access, keep the default compatible, update config/SDK docs and baselines, and require redacted real NFS proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection plus SQLite docs show the patch leaves WAL shared memory active. No for the original production SIGBUS crash: I did not establish a live NFS reproduction and the PR has no after-fix proof.

## Solution Assessment

No. The current implementation is too narrow for the stated WAL/NFS failure mode; the maintainable solution is an explicit storage mode that owns WAL, locking, documentation, and upgrade behavior together.
