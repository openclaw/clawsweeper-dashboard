## Summary

This PR should stay open because it targets a plausible NFS-backed availability problem, but the current patch does not yet prove or fully implement an NFS-safe SQLite mode. Current main has since centralized WAL setup, while the PR only disables main-database mmap and leaves WAL shared-memory behavior as a maintainer storage-policy decision.

## What This Changes

The PR adds `OPENCLAW_SQLITE_MMAP`, a `storage.sqliteMmap` config bridge, a new plugin SDK SQLite helper subpath, and calls to disable SQLite mmap at task registry and memory database opens.

## Best Possible Solution

Define an explicit NFS-safe SQLite storage mode in shared SQLite open/setup code, covering WAL, shared memory, locking, concurrency, config/docs artifacts, and SDK exposure only if that public seam is really needed.

## Reproduction Assessment

Yes for the review blockers: source inspection plus SQLite's WAL contract show the patch does not cover WAL shared memory. No for the original production SIGBUS crash: there is no posted real NFS-backed reproduction or after-fix runtime proof.

## Solution Assessment

No. A partial `mmap_size` toggle is not the best fix for an NFS-backed SQLite support promise; a named storage mode should own WAL, locking, concurrency, schema/docs, and upgrade semantics together.
