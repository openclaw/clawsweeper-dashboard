## Summary

Keep open. Current main still has source-backed gaps matching this report: request-scoped narrative fallback writes the first raw snippet, terminal/no-text narrative paths can exit before writing `DREAMS.md`, corpus ingestion only partially filters routing wrappers, and fallback counts are not surfaced in `memory status`.

## What This Changes

The issue asks memory-core Dreaming to produce real Dream Diary entries instead of raw request-scoped fallback snippets, filter routing metadata from the recall corpus, and expose fallback diagnostics.

## Best Possible Solution

Keep this issue open as the current v2026.5.7 Dreaming fallback and diary tracker, land or replace the focused fallback/denoise PRs, and handle the public execution config shape in the related config issue.

## Reproduction Assessment

No high-confidence full cron reproduction was run in this read-only review. Source inspection does reproduce the narrowed failure modes: request-scoped fallback writes the first snippet verbatim, and terminal/no-text narrative runs return before a diary append.

## Solution Assessment

Unclear. Adding direct `dreaming.execution.model` is not clearly the narrowest fix because current main already has `dreaming.model` and `dreaming.execution.defaults.model`; the safer path is to harden fallback writing, filtering, and diagnostics while coordinating the config contract separately.
