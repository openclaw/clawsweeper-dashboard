## Summary

Keep open. Current main partially addresses the reported model configuration and gateway subagent-binding path, but source inspection still shows meaningful remaining gaps: request-scoped narrative fallback writes a raw snippet into `DREAMS.md`, literal routing metadata can still enter the session corpus when provenance is missing, and `memory status` has no fallback-count visibility.

## What This Changes

The issue requests fixing memory-core Dream Diary generation so managed dreaming writes real diary entries, filters routing metadata from session-corpus input, and reports narrative fallback use in memory status.

## Best Possible Solution

Keep the documented `dreaming.model` / `dreaming.execution.defaults.model` path, then make fallback diary writes safe, drop literal OpenClaw routing metadata before session-corpus ingestion, and add a clear fallback diagnostic to memory status.

## Reproduction Assessment

Unclear for a full current-main cron reproduction because I did not run the gateway-managed cron path. Source inspection does provide a concrete reproduction path for the remaining fallback/raw-snippet, routing-metadata ingestion, and missing-status-diagnostic gaps.

## Solution Assessment

Unclear. Adding the reporter’s direct `dreaming.execution.model` key alone is not the best fix because current main already has documented single-model paths; the durable fix should address fallback output safety, corpus hygiene, and fallback observability while leaving the broader fallback-chain contract to https://github.com/openclaw/openclaw/issues/70616.
