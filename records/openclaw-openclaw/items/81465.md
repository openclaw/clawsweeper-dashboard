## Summary

Current main and the latest release now address the central Dream Diary failure: narrative reads settle before fallback, raw snippets are no longer written as fallback diary prose, the dreaming model config is accepted, and session-corpus ingestion has archive/cron/heartbeat filtering coverage. Any remaining edge cases are better handled in newer narrower reports rather than this older umbrella.

## What This Changes

The issue asks memory-core dreaming to stop falling back or skipping real DREAMS.md diary entries, keep routing/session metadata out of the dreaming corpus, and make fallback behavior visible to operators.

## Best Possible Solution

Keep the v2026.6.6 Dream Diary retry and ingestion hardening, and track any remaining rare fallback/readback variants in the newer focused reports instead of this broad umbrella.

## Reproduction Assessment

Yes at source level for the historical failure: the older code could take the no-text/fallback path after a successful narrative run, and newer related reports supplied live timing/readback evidence. I did not run a live cron reproduction in this read-only review.

## Solution Assessment

Yes for cleanup: current main and v2026.6.6 use a bounded settled read before fallback, keep fallback from leaking raw snippets, and accept the documented dreaming model config. A separate exact fallback counter in `memory status` is not central enough to keep this broad fixed report open.
