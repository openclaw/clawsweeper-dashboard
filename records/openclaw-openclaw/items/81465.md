- `impact:data-loss`: Generated Dream Diary entries can be absent or skipped even when phase reports and cron runs succeed.

## Summary

Keep open. Current main and v2026.5.18 still have source-backed paths for the reported Dream Diary failure: request-scoped fallback writes a raw snippet, timeout/no-text paths return without a diary append, literal routing text is not fully filtered from session ingestion, and `memory status` does not expose narrative fallback counts.

## What This Changes

The issue requests reliable Dream Diary narrative generation, denoised session-corpus ingestion, and fallback visibility in `openclaw memory status`.

## Best Possible Solution

Keep this as a canonical Dream Diary fallback/denoise/status tracker, land or replace the focused fallback and ingestion PRs after review, and coordinate the public config/status contract with the related Dreaming execution and observability issues.

## Reproduction Assessment

Yes, at source level. Current main still has the raw request-scoped fallback write, terminal/no-text no-diary returns, partial routing-metadata filtering, and missing fallback-count status visibility; I did not run a live multi-day cron scenario.

## Solution Assessment

Unclear. The fallback and ingestion bugs are concrete, but the best final shape needs maintainer agreement on whether to use deterministic fallback prose, transcript/run-result extraction, direct model execution, and what `memory status` should expose.
