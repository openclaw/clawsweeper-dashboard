## Summary

Keep open. Current main still has source-backed gaps for this report: request-scoped Dream Diary fallback writes the first raw snippet, terminal/no-text narrative paths can return before writing `DREAMS.md`, ingestion filtering is still partial for literal routing metadata, and `memory status` does not expose narrative fallback counts; the config-model portion is partially covered but the direct execution/fallback contract remains tracked separately.

## What This Changes

The issue asks OpenClaw to make Dreaming write real `DREAMS.md` diary entries instead of raw or missing fallback output, filter routing metadata from the memory corpus, and surface narrative fallback activity in `memory status`.

## Best Possible Solution

Keep this as the canonical current-release Dream Diary fallback/no-diary tracker, land or replace focused fallback and corpus-denoise PRs after review, and resolve the public Dreaming execution/fallback config contract through the related config issue.

## Reproduction Assessment

Yes, source-reproducible on current main: the raw fallback write, return-before-diary terminal paths, partial ingestion filtering, and missing status visibility are all visible in the inspected code and tests. I did not run a live cron/gateway reproduction.

## Solution Assessment

Unclear as a single implementation path. The narrow maintainable fix is to harden diary fallback writes, no-text/timeout handling, corpus filtering, and diagnostics while leaving the public execution/fallback config shape to https://github.com/openclaw/openclaw/issues/70616.
