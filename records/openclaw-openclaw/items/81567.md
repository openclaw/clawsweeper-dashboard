## Summary

Keep this open: current main still treats `openai/gpt-4o` as a selectable model, but the strict-agentic plan-only retry contract remains scoped to GPT-5-family OpenAI/Codex runs, so the reporter's added real-world GPT-4o evidence needs a maintainer support-boundary decision rather than cleanup closure.

## What This Changes

The issue requests investigation and a support decision for `openai/gpt-4o` agent sessions that return one plan-only text response and stop instead of continuing the tool-use loop.

## Best Possible Solution

Decide the GPT-4o support boundary: either extend strict-agentic/incomplete-turn coverage to GPT-4o with focused tests and live proof, or document and surface that GPT-4o is not supported for reliable multi-step agent workflows.

## Reproduction Assessment

No high-confidence live reproduction on current main. Source inspection does confirm that GPT-4o remains on the default contract and therefore will not receive the strict-agentic plan-only retry guard that would address the reported stop-after-plan path.

## Solution Assessment

Unclear. The code path to broaden the guard is identifiable, but current docs and tests deliberately scope strict-agentic behavior to GPT-5-family OpenAI/Codex runs, so maintainers should choose the support stance before implementation.
