## Summary

Keep open: current main still exposes GPT-4o as an OpenAI model but deliberately scopes strict-agentic planning-only retry and blocked-exit behavior to GPT-5-family OpenAI/Codex runs, so this needs a maintainer support-boundary decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to investigate GPT-4o plan-only terminal agent sessions and either enforce multi-step tool-use behavior or document the model boundary.

## Best Possible Solution

Decide the GPT-4o support boundary: either extend strict-agentic/incomplete-turn handling to GPT-4o with focused tests and live proof, or document and surface that GPT-4o is not supported for long multi-step agentic workflows.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. Source inspection does show that if GPT-4o returns a stop/planning-only assistant turn, current main will not route it through the strict-agentic planning-only retry and blocked-exit path.

## Solution Assessment

Unclear. Extending strict-agentic or adding a tool-choice nudge could be viable, but current docs and tests intentionally scope the stronger contract to GPT-5-family OpenAI/Codex models, so maintainers should choose the support policy first.
