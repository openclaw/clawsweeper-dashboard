## Summary

Keep this open: current main has strong GPT-5 strict-agentic handling, but the same guard is intentionally not active for `openai/gpt-4o`, so the reported GPT-4o plan-only terminal path still needs a maintainer support-boundary decision.

## What This Changes

The issue requests investigation/support for GPT-4o OpenAI agent turns that return plan-only text and end without tool calls.

## Best Possible Solution

Decide the GPT-4o support boundary: either extend incomplete-turn/strict-agentic coverage to specific GPT-4o OpenAI runs with focused tests and live proof, or document GPT-5.5/Codex as the supported OpenAI agent path and warn that GPT-4o may stop after plan-only text.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows current main will not apply the plan-only retry guard to GPT-4o, but the exact GPT-4o response pattern still needs a live run or transcript on current main.

## Solution Assessment

Unclear. A `tool_choice` nudge or broader strict-agentic allowlist is plausible, but the docs and tests currently scope the stronger contract to GPT-5-family OpenAI runs, so maintainers should first decide whether older GPT-4.x models are supported for this agentic behavior.
