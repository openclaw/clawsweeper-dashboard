## Summary

Keep open. Current main and the latest release still leave `openai/gpt-4o` outside the strict-agentic OpenAI execution contract, while the report has concrete production evidence that GPT-4o can terminate multi-step sessions with terminal planning text and no tools.

## What This Changes

The issue asks OpenClaw to investigate and resolve GPT-4o agent sessions ending after one text response instead of continuing multi-step tool use.

## Best Possible Solution

Maintain one GPT-4o policy path: either add focused GPT-4o strict-agentic recovery with live file-read/write proof, or update model/docs guidance to steer multi-step agents away from GPT-4o.

## Reproduction Assessment

Partly yes: the source-level support gap is high-confidence because current main and v2026.6.8 keep GPT-4o on the default contract. The exact live GPT-4o plan-text terminal behavior still needs a fresh real-provider CLI repro.

Unclear as an implementation path: enabling GPT-4o strict-agentic recovery is plausible, but maintainers first need to decide whether GPT-4o should be supported for multi-step agent workflows or explicitly documented as unsuitable.
