## Summary

Keep open: current main still makes `openai/gpt-4o` selectable, but the strict-agentic planning-only retry and blocked-exit contract is scoped to GPT-5-family OpenAI/Codex runs, so this needs an explicit maintainer support-boundary decision rather than cleanup closure.

## What This Changes

The request is to prevent `openai/gpt-4o` agent sessions from ending after one planning-text response, or to document that GPT-4o is not supported for multi-step agentic workflows.

## Best Possible Solution

Choose the GPT-4o support boundary: either extend strict-agentic/incomplete-turn handling to GPT-4o with focused tests and live proof, or clearly document and surface that GPT-4o is not supported for long multi-step agentic workflows.

## Reproduction Assessment

No high-confidence fresh live reproduction was run on current main. Source inspection does show that if GPT-4o returns a `stop` planning-only assistant turn, current main keeps it on the default contract and will not route it through the strict-agentic retry/blocked path.

## Solution Assessment

Unclear. Extending strict-agentic handling or request shaping to GPT-4o may be viable, but current docs and tests deliberately scope the stronger contract to GPT-5-family OpenAI/Codex runs, so maintainers should decide the support policy before implementation.
