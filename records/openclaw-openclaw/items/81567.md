## Summary

Keep open: current main still treats `openai/gpt-4o` as a default-contract OpenAI model, so the strict plan-only retry and blocked-exit behavior that exists for GPT-5-family runs does not apply to the reported GPT-4o path. The related strict-agentic work is adjacent but does not make a maintainer support-boundary decision for GPT-4o.

## What This Changes

The issue asks OpenClaw to address GPT-4o agent sessions that return one planning-text response and exit instead of continuing through tool calls.

## Best Possible Solution

Maintainers should decide the GPT-4o support boundary, then either add focused GPT-4o coverage to the strict/incomplete-turn path with live proof and regression tests, or document and surface that GPT-4o is not supported for reliable multi-step agentic workflows.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show the reported failure remains plausible on current main because `openai/gpt-4o` resolves to the default contract and does not enter the strict plan-only retry path.

## Solution Assessment

Unclear. A `tool_choice` nudge or adding GPT-4o to the strict-agentic allowlist might solve the symptom, but the current docs and tests make this a support-policy decision before it is a narrow code fix.
