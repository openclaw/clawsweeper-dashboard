## Summary

Keep this PR open. The goal is useful and current main still lacks selected-session bare-model inference for this exact gateway path, but the proposed post-processing normalizer introduces provider-routing regressions that should block merge.

## What This Changes

The PR adds a gateway session model-ref normalization helper and a focused regression test file so bare session model overrides can infer a unique configured provider.

## Best Possible Solution

Implement selected-session bare-model inference in the shared selected-model resolver flow, preserving explicit `providerOverride`, slash-containing wrapper model ids, runtime-only sessions, ambiguous configured matches, and non-default configured providers, with focused gateway regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with `providerOverride: "openrouter"` and `modelOverride: "anthropic/claude-haiku-4.5"`, the PR's slash branch reparses the model as provider `anthropic`; with a non-OpenAI default provider and bare `modelOverride: "gpt-5.4"`, the PR skips configured-provider inference.

## Solution Assessment

No. The requested behavior is valid, but a gateway-only post-processing normalizer is not the best fix because it partially bypasses existing selected-model semantics; extending the shared resolver path is safer.

Full review comments:

  - body: `normalizedModel.includes("/")` is handled before `entry.providerOverride`, so a wrapper selection such as `providerOverride: "openrouter"` with `modelOverride: "anthropic/claude-haiku-4.5"` is rewritten to provider `anthropic`. That regresses the current wrapper-provider contract and routes the session through the wrong provider.
  - confidence: 0.91
- **[P2] Do not let stored runtime providers suppress inference:** `src/gateway/session-utils.ts:1065-1078`
  - body: When an existing session has `entry.modelProvider`, this branch returns that stored runtime provider before the inferred configured provider can be used. Sessions that have already run once can keep routing a bare selected override through stale/default runtime metadata instead of the unique configured provider this feature is trying to select.
  - confidence: 0.86

  - body: The new inference only applies when `params.ref.provider` is empty or equal to `DEFAULT_PROVIDER`. If the configured default provider is `anthropic` and `gpt-5.4` uniquely belongs to `openai-codex`, the resolver leaves the session on `anthropic/gpt-5.4`, so the feature fails for non-OpenAI defaults.
  - confidence: 0.84
