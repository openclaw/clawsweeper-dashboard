## Summary

Keep open. Current main partially reduced the release-era bootstrap caps, but the reported local-model prompt-pressure problem is not resolved: ordinary main sessions still inject the broad bootstrap set, no `bootstrapTier` config exists, generated templates remain large, and doctor only warns near configured truncation budgets rather than model-aware tool-dispatch risk.

## What This Changes

The issue requests reducing or tiering default workspace bootstrap context, trimming generated bootstrap templates, or adding model-aware doctor guidance so small and mid local models keep reliable structured tool dispatch.

## Best Possible Solution

Define one canonical local-model prompt-pressure strategy: a supported bootstrap profile or tier with maintainer-approved defaults, schema/docs/tests, hook-aware filtering, and model-aware doctor remediation, while keeping richer bootstrap context available as an explicit opt-in for capable models.

## Reproduction Assessment

Yes. The issue provides concrete setup steps, an `openclaw agent` command, systemPromptReport measurements, and session JSONL checks for real `toolCall`/`toolResult` events; current main still has the broad ordinary-session bootstrap path needed to validate the same failure shape.

## Solution Assessment

Unclear. Lowering `bootstrapMaxChars` is a narrow mitigation, but the best maintainable fix likely needs coordinated tier/default/doctor/template behavior, and the linked open PRs already cover plausible implementation paths that need maintainer selection.
