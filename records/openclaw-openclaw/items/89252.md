## Summary

Keep open. Current main and the latest release still have the Telegram command-menu paths this issue is preserving: custom commands are appended after native/plugin commands before trimming, and an empty command list still clears Telegram scopes and caches the empty hash.

## What This Changes

The issue reports Telegram `customCommands` and native bot commands disappearing or not being registered after gateway restart, especially when command sync sees an empty command list or stale hash state.

## Best Possible Solution

Land a maintainer-approved Telegram command-menu fix that preserves operator customCommands under trimming and distinguishes explicit native-command disablement from accidental empty command resolution, with focused tests and live Telegram proof.

## Reproduction Assessment

Partial yes. Source inspection on current main shows the reported empty-list clear/cache decision path and supported customCommands surface, but I did not establish a live Telegram restart that makes runtime command resolution return an empty list.

## Solution Assessment

Unclear. A guard that only clears on explicit disablement is plausible and matches the issue's suggested direction, but the final fix should also address customCommands trimming priority and preserve intentional operator cleanup semantics.
