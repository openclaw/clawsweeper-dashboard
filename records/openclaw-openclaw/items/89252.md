Keep open. Current main and v2026.6.8 still have the two remaining behaviors this issue preserves: customCommands are ordered behind native/plugin commands before cap trimming, and an empty Telegram menu sync still clears scopes before caching the empty hash.

The issue reports Telegram customCommands not reliably appearing after gateway restart and asks for an empty-list sync guard equivalent to the closed unmerged Telegram fix PRs.

## Best Possible Solution

Land a maintainer-approved Telegram command-menu fix that preserves operator customCommands under cap pressure and only clears Telegram menus when native command disablement is explicit, with focused regression tests and live Telegram proof.

## Reproduction Assessment

Partially. Source inspection shows current main can drop tail commands after ordering customCommands behind native/plugin entries and still clears/caches an empty payload if the runtime resolves no commands; a live Telegram restart proof for the exact race is still missing.

## Solution Assessment

Unclear. A custom-first ordering change is likely the narrow fix for cap trimming, but the empty-list clearing guard changes Telegram startup semantics and needs maintainer agreement plus live proof.
