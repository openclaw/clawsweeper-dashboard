## Summary

Keep open. Current main and the latest release still have the central Telegram command-menu risk: `customCommands` are appended after native/plugin commands before cap trimming, and the empty-list sync path still clears scopes before caching the empty hash; the related open PR covers only the custom-priority part and is not merged.

## What This Changes

The issue asks OpenClaw to reliably register Telegram `channels.telegram.customCommands` on gateway restart and avoid wiping or skipping the bot command menu when command resolution unexpectedly becomes empty.

## Best Possible Solution

Maintainers should settle the Telegram menu priority and fail-safe clearing semantics, then land a refreshed narrow fix that preserves configured `customCommands` under cap pressure and only clears an empty menu when that state is intentional, with regression coverage and real Telegram proof.

## Reproduction Assessment

Partially. Source inspection gives a current-main reproduction for the overflow path where custom commands are appended after native/plugin entries and trimmed from the tail, but the exact empty-list restart race still needs current-main config, logs, `getMyCommands` output, or live Telegram proof.

## Solution Assessment

Unclear. Custom-first ordering looks like the narrow repair for cap-related omission, but the empty-list guard changes startup clearing semantics and needs maintainer agreement around explicit native-command disable behavior.
