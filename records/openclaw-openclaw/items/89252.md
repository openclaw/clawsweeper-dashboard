## Summary

Keep open. Current main and v2026.6.8 still preserve the reported risk: `customCommands` are ordered after native/plugin entries before menu trimming, and an empty `commandsToRegister` payload still clears Telegram command scopes before caching the empty hash.

## What This Changes

The issue asks OpenClaw to reliably register Telegram `customCommands` on gateway restart and avoid treating an unexpected empty command list as an intentional menu clear.

## Best Possible Solution

Land a maintainer-approved Telegram command-menu fix that preserves operator `customCommands` under cap pressure and only clears Telegram menus when native command disablement is explicit, with focused regression tests and live Telegram proof.

## Reproduction Assessment

Partially. Source inspection shows current main can drop tail commands after ordering `customCommands` behind native/plugin entries under overflow and still clears/caches an empty payload if the runtime resolves no commands; a live Telegram restart proof for the exact race is still missing.

## Solution Assessment

Unclear. A custom-first ordering change is likely the narrow fix for cap trimming, but the empty-list clearing guard changes Telegram startup semantics and needs maintainer agreement plus live proof.
