## Summary

Keep open. Current main still has the reported plugin-hook media gap, but this branch should not merge as-is because it edits the pre-split hook type surface and changes plugin-visible media authority before sandbox staging.

## What This Changes

The PR adds media path, URL, and MIME-type arrays to the `before_agent_reply` plugin hook context and wires them from inbound message media metadata.

## Best Possible Solution

Land a rebased additive hook API in `src/plugins/hook-types.ts`, preferably with defensive-copy media fields or a structured attachment shape, plus focused hook wiring tests, public docs, changelog, and explicit raw-versus-staged media semantics.

## Reproduction Assessment

Yes. On current main, a focused test can pass inbound `MediaPaths`/`MediaUrls` into `getReplyFromConfig` and assert the `runBeforeAgentReply` context omits them, then observe that `{ handled: true }` returns before `stageSandboxMedia`.

## Solution Assessment

No. The additive direction is reasonable, but the current PR edits the stale type file and exposes pre-staging media references without a documented security/API decision; the safer path is a rebased hook-contract change with documented, tested semantics.
