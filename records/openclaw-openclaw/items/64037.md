## Summary

Keep open, but do not merge as-is: the feature gap is real, while the branch targets a stale hook type surface, exposes raw pre-staging media references, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds optional `mediaPaths`, `mediaUrls`, and `mediaTypes` to the `before_agent_reply` plugin hook context and populates them from inbound `MsgContext` media metadata.

## Best Possible Solution

Define a maintainer-approved structured or staged inbound-media hook contract in `src/plugins/hook-types.ts`, wire it through `getReplyFromConfig`, add focused hook tests/docs, and require redacted real plugin proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main passes only `cleanedBody` plus agent/session/channel fields to `before_agent_reply`, and a handled hook returns before media staging. This is a capability gap rather than a broken documented bug contract.

## Solution Assessment

No. The feature direction is reasonable, but the current diff updates a stale type surface and forwards pre-staging raw media references; a staged or structured media contract is the safer maintainable path.
