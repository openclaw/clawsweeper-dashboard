## Summary

Keep open: the plugin media-access gap is plausible from current source, but this PR is stale against the live hook type surface and exposes raw inbound media references before staging policy runs. It needs maintainer API/security judgment, a rebase, and contributor real behavior proof before merge.

## What This Changes

This PR adds `mediaPaths`, `mediaUrls`, and `mediaTypes` to the `before_agent_reply` hook context and populates them from inbound `MsgContext` media metadata.

## Best Possible Solution

Define a maintainer-approved structured or staged inbound-media hook contract in `src/plugins/hook-types.ts`, add focused `before_agent_reply` tests and plugin docs, and require redacted real plugin proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main passes only cleaned text plus agent/session/channel context to `before_agent_reply`, and a handled hook returns before media staging. This is a capability gap rather than a broken documented bug contract.

## Solution Assessment

No. The feature direction is reasonable, but the current diff targets the old type location and forwards pre-staging raw paths/URLs; the safer solution is a staged or structured media contract with tests and docs.
