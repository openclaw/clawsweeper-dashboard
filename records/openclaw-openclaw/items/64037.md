## Summary

Keep open. The requested hook-media gap still exists on current main, but this PR is not merge-ready because it updates the stale hook type surface, omits the required public API changelog/docs/test follow-through, and needs maintainer review for raw media path and URL exposure before staging.

## What This Changes

This PR adds media path, URL, and MIME-type arrays to the `before_agent_reply` plugin hook context and wires them from inbound message media metadata.

## Best Possible Solution

Land a rebased additive hook API in `src/plugins/hook-types.ts` with defensive-copy media fields or a structured attachment shape, explicit raw-versus-staged media semantics, focused `before_agent_reply` wiring tests, public plugin docs, and a valid changelog entry.

## Reproduction Assessment

Yes. A focused current-main test can pass `MediaPaths`/`MediaUrls` into `getReplyFromConfig`, assert the `runBeforeAgentReply` context omits them, and verify a handled hook returns before `stageSandboxMedia`.

## Solution Assessment

No. The additive direction is reasonable, but this patch targets the old type location and exposes pre-staging media references without settled API/security documentation; the safer fix is a rebased hook-contract change with explicit media semantics.
