## Summary

Keep open: the capability gap is real, but this PR is not ready because it updates a stale hook type surface, exposes raw media references before sandbox staging, lacks after-fix real behavior proof, and still needs a maintainer API/security decision.

## What This Changes

The PR adds `mediaPaths`, `mediaUrls`, and `mediaTypes` to the `before_agent_reply` hook context and populates them from inbound `MsgContext` media metadata.

## Best Possible Solution

Define a maintainer-approved structured or staged inbound-media hook contract in `src/plugins/hook-types.ts`, wire it through `getReplyFromConfig`, add focused hook tests/docs, and require redacted real plugin proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main passes only `cleanedBody` plus agent/session/channel fields to `before_agent_reply`, and a handled hook returns before media staging. This proves a capability gap, not a broken documented bug contract.

## Solution Assessment

No. The feature direction is reasonable, but the current diff updates an obsolete type location and forwards pre-staging raw media references; a staged or structured media contract is the safer maintainable path.
