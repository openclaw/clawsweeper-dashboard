## Summary

Keep open. The feature direction is plausible, but this PR is not merge-ready: it targets the stale hook type location after the current-main hook contract split, exposes raw media references before staging policy runs, lacks after-fix real plugin proof, and needs maintainer API/security judgment.

## What This Changes

The PR adds optional `mediaPaths`, `mediaUrls`, and `mediaTypes` fields to the `before_agent_reply` plugin hook context and populates them from inbound `MsgContext` media metadata.

## Best Possible Solution

Define a maintainer-approved structured or staged inbound-media hook contract in `src/plugins/hook-types.ts`, wire it through `getReplyFromConfig`, add focused hook tests/docs, and require redacted real plugin proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main passes only cleaned text plus agent/session/channel context to `before_agent_reply`, and a handled hook returns before media staging. This is a capability gap rather than a broken documented bug contract.

## Solution Assessment

No. The feature direction is reasonable, but the current diff updates the stale type surface and forwards pre-staging raw media references; a staged or structured media contract is the safer maintainable path.
