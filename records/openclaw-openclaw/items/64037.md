## Summary

Keep open: the feature gap is real, but this PR should not merge as-is because it edits a stale hook type location, exposes pre-staging media references, and lacks required real behavior proof.

## What This Changes

This PR adds `mediaPaths`, `mediaUrls`, and `mediaTypes` to the `before_agent_reply` hook context and populates them from inbound `MsgContext` media metadata.

## Best Possible Solution

Define a rebased hook API with explicit staged or structured inbound-media semantics, focused `before_agent_reply` tests, plugin docs, and maintainer-approved security boundaries.

## Reproduction Assessment

Yes, from source inspection: current main's `before_agent_reply` call passes only `cleanedBody` plus agent/session/channel context, and a handled result returns before media staging. This is a capability gap rather than a broken documented contract.

## Solution Assessment

No. The feature direction is plausible, but the current diff targets a stale type location and exposes pre-staging raw paths and URLs; the safer path is a staged or structured media contract in `src/plugins/hook-types.ts` with tests and docs.
