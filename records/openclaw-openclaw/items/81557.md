## Summary

Close as superseded: the adapter-only patch does not reach the final Pi tool-result error flag, while the open companion PR tracks the bridge-level fix that Pi actually consumes.

## What This Changes

The branch adds `isError: true` to adapter-wrapped tool execution error results and extends adapter tests to assert that flag.

## Best Possible Solution

Focus remaining work on the Pi `tool_result` bridge tracked by https://github.com/openclaw/openclaw/pull/81564, preserving status-error payload compatibility while returning the error override that Pi consumes.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main wraps caught tool exceptions into `status:"error"` payloads, and the pinned Pi loop finalizes successful tool returns with `isError:false` unless `afterToolCall` supplies an override.

## Solution Assessment

No. Adding `isError` to the adapter return object is not the best fix because Pi drops that property when finalizing successful tool returns; the maintainable fix is to return the error override from the embedded Pi bridge.
