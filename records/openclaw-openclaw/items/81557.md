## Summary

Close as superseded: a newer open PR tracks the same linked bug at the Pi tool-result bridge, and source/dependency inspection shows this adapter-only patch does not change the final model-visible `toolResult.isError` flag.

## What This Changes

The PR changes the Pi tool-definition adapter so caught tool execution exceptions return a JSON error payload with an added `isError: true` property and adds adapter regression assertions.

## Best Possible Solution

Use the bridge-level fix tracked in https://github.com/openclaw/openclaw/pull/81564, with redacted real behavior proof, rather than merging this adapter-only patch.

## Reproduction Assessment

Yes, at source level: caught tool exceptions become `details.status:"error"` payloads, while Pi finalizes successful tool returns with `isError:false` unless the bridge returns an `afterToolCall` override. I did not run a live Minimax/OpenClaw replay in this read-only review.

## Solution Assessment

No. This adapter-only change marks an intermediate return object, but the maintainable fix must set or preserve the final Pi `isError` override in the tool-result bridge that writes transcript/model-visible tool results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Set the final Pi error flag, not only the adapter result:** `src/agents/pi-tool-definition-adapter.ts:160`
  - body: `pi-agent-core` ignores `result.isError` from a successful tool `execute` return and initializes the finalized call with `isError: false`; only thrown tools or an `afterToolCall` override change the transcript/model-visible flag. Since this adapter catches the exception and returns a payload, adding `isError` here does not fix the reported final `toolResult.isError=false`; the bridge should return an `isError` override when the result status is `error` or `timeout`.
  - confidence: 0.92
