## Summary

Keep open. Current main does not implement this archive API/UI, and the PR still has useful direction, but the branch has blocking type and transcript-read issues, lacks real behavior proof, changes memory retention semantics, and is dirty against newer Sessions UI work.

## What This Changes

The PR adds session archiving metadata, backend filtering, archive/unarchive UI controls, memory-backed archive summaries, inbound auto-unarchive, Swift protocol fields, and a Telegram `/set_topic_name` command.

## Best Possible Solution

Split or rebase the work around current Sessions UI, settle the archive retention contract with https://github.com/openclaw/openclaw/issues/56609 and the row-action UX with https://github.com/openclaw/openclaw/issues/69271, then add bounded summaries and real UI/Telegram proof.

## Reproduction Assessment

Yes for the patch blockers: PR-head source inspection shows a `null` write to a `number | undefined` field and a full synchronous transcript scan before the first await. I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The direction is plausible, but the patch needs a typed archive/unarchive contract, bounded or async summaries, retention review, a rebase against current Sessions UI, and real UI/Telegram proof before merge.

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear archivedAt through the typed patch contract:** `src/auto-reply/reply/session.ts:445`
  - body: `SessionEntry.archivedAt` is declared as `number | undefined`, but this branch assigns `null` when inbound activity should unarchive a session. Clear the field through the `sessions.patch` contract that accepts `archivedAt: null`, or delete it in a typed helper, so the persisted session shape stays valid under strict types.
  - confidence: 0.93
- **[P2] Bound archive summary transcript reads:** `src/sessions/archive-summary.ts:94`
  - body: `writeSessionArchiveSummary` calls `readSessionMessages`, which synchronously scans the selected transcript before the first await. Even though the caller schedules it with `setImmediate`, archiving a large session can still monopolize the gateway event loop; use a bounded recent scan or an async reader.
  - confidence: 0.86
