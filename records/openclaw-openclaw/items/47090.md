## Summary

Keep open. Current main does not implement the PR's persistent archive/unarchive API, but the PR still has source-reproducible blockers, no real behavior proof, and an unresolved retention/security decision around writing chat excerpts into workspace memory.

## What This Changes

The PR adds persistent session archiving with UI actions and filters, summary files under `memory/`, inbound auto-unarchive behavior, Swift protocol fields, and a Telegram `/set_topic_name` command.

## Best Possible Solution

Split and rebase the work: land a typed, bounded archive/unarchive contract only after the retention location is decided, preserve internal exact-key lookups, and handle `/set_topic_name` separately with live Telegram proof.

## Reproduction Assessment

Yes for the patch blockers by source inspection: PR head shows an invalid `null` assignment to a `number | undefined` session field, default filtering that hides archived rows from internal lookup callers, and a synchronous full-transcript archive summary read. I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The feature direction is plausible, but the patch needs a typed archive/unarchive path, internal lookup preservation, bounded transcript reads, a retention decision, and real UI/Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear archivedAt through the typed patch path:** `src/auto-reply/reply/session.ts:445`
  - body: `SessionEntry.archivedAt` is `number | undefined`, but this writes `null` into a `SessionEntry` under strict TypeScript before it is merged back into the store. Clear the archive flag through the existing `sessions.patch` null contract or a typed helper that deletes the property.
  - confidence: 0.93
- **[P2] Include archived rows for exact session metadata lookups:** `src/gateway/session-utils.ts:893-894`
  - body: This default filter removes archived sessions from all `sessions.list` calls unless callers know to set `includeArchived`. ACP still does an exact-key search without that flag, so archived sessions can lose snapshot/model metadata even though the session exists.
  - confidence: 0.86
- **[P2] Bound archive-summary transcript reads:** `src/sessions/archive-summary.ts:94`
  - body: `readSessionMessages` uses a synchronous full-transcript read before the first await in this helper. Even inside `setImmediate`, archiving a large session can block the gateway event loop; use a bounded recent reader or an async transcript path.
  - confidence: 0.86
