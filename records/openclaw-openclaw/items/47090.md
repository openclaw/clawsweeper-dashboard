## Summary

Keep open. Current main has not absorbed the durable `archivedAt` archive/unarchive API, and the PR still has correctness, retention, and real-behavior-proof blockers that need human review rather than conservative cleanup.

## What This Changes

The PR adds durable session archiving with `archivedAt`, Control UI archive filters/actions, archive summary files under `memory/`, inbound auto-unarchive, Swift protocol fields, and a Telegram `/set_topic_name` command.

## Best Possible Solution

Rebase or split the work around current Sessions UI, settle the archive retention/location contract with the linked issues, use a typed archive/unarchive API with bounded async summaries, then add focused tests and redacted live UI/Telegram proof.

## Reproduction Assessment

Yes for the patch blockers: PR-head source inspection shows a `number | undefined` archive field assigned `null`, and archive summary generation performs a synchronous transcript scan before yielding. I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The feature direction is plausible, but the patch needs a typed archive/unarchive contract, bounded or async summaries, retention review, a rebase against current Sessions UI, and real UI/Telegram proof before merge.

  - body: `SessionEntry.archivedAt` is declared as `number | undefined`, but inbound auto-unarchive assigns `null` directly to a session entry.
