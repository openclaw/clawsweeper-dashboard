## Summary

Keep open. The PR still contains concrete patch blockers, lacks required real behavior proof, and changes session archive retention/recall semantics that need maintainer product/security review before merge.

## What This Changes

The PR adds durable session archiving with `archivedAt`, Control UI archive filters/actions, archive summary files under `memory/`, inbound auto-unarchive, Swift protocol fields, and a Telegram `/set_topic_name` command.

## Best Possible Solution

Split and rebase the work: settle the archive retention/location contract, implement a typed bounded archive/unarchive API with focused tests, and handle the Telegram topic command separately with live proof.

## Reproduction Assessment

Yes for the patch blockers: PR-head source inspection shows a `number | undefined` archive field assigned `null`, and archive summary generation synchronously scans transcripts before yielding. I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The feature direction is plausible, but the patch needs a typed archive/unarchive contract, bounded or async summaries, retention review, a rebase against current Sessions UI, and real UI/Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Clear archivedAt through the patch contract:** `src/auto-reply/reply/session.ts:445`
  - body: `SessionEntry.archivedAt` is declared as `number | undefined`, but inbound auto-unarchive assigns `null` directly to a session entry. Clear the field through the `sessions.patch` contract that already treats `archivedAt: null` as delete, or use a typed helper, so persisted session records keep the declared shape.

  - confidence: 0.86
