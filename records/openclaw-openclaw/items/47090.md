## Summary

Keep this PR open but do not merge it as-is. The feature is not fully implemented on current main, but the branch has a type-breaking auto-unarchive path, a synchronous archive-summary transcript scan, missing real behavior proof, and product/security questions around writing chat excerpts into agent memory.

## What This Changes

The PR adds `archivedAt` session metadata, Gateway/UI archive filters and archive/unarchive controls, archive-summary files under agent `memory/`, auto-unarchive on inbound activity, and a Telegram `/set_topic_name` command.

## Best Possible Solution

Rebase and likely split the work against current main, keep the archive contract type-safe, use async or bounded transcript summarization, document/confirm the archive-to-memory retention behavior, and add real UI plus Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings from source inspection: the PR assigns `null` to a `number | undefined` field and calls a synchronous transcript reader from the archive summary path. I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The direction is plausible, but the implementation needs type-contract cleanup, non-blocking transcript reading, current-main UI integration, and explicit proof/review for the new memory-retention behavior.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Make auto-unarchive type-safe:** `src/auto-reply/reply/session.ts:445`
  - body: `SessionEntry.archivedAt` is declared as `number | undefined`, but this branch assigns `null` so the inbound auto-unarchive path will fail strict TypeScript checks.   - confidence: 0.93

  - body: `readSessionMessages` scans the transcript with synchronous file I/O. Scheduling the call with `setImmediate` avoids blocking the patch response, but once it runs it still blocks the gateway event loop on large sessions; use the async reader or a bounded recent scan for archive summaries.
  - confidence: 0.86
