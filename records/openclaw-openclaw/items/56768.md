## Summary

Keep open for now: the status/dashboard token-auth usability gap is still present on current main, but this PR is not merge-ready because it would print bearer-token fragments in normal `gateway status` output and is now dirty against current main’s newer token-safe dashboard behavior.

## What This Changes

The PR adds shared Control UI access URL resolution for `openclaw gateway status`, including token-fragment handling, SecretRef hints, JSON redaction, tests, and a changelog entry.

## Best Possible Solution

Redesign the PR so `gateway status` remains token-free in normal human output, uses the current gateway auth resolver, avoids SecretRef work when Control UI output is disabled, and offers safe guidance or an explicit delivery path for token-auth users.

## Reproduction Assessment

Yes, source-reproducible: current main prints `gateway status` Dashboard from a bare Control UI URL while `openclaw dashboard` has token-aware browser/clipboard delivery. I did not run a live browser/token-auth flow in this read-only review.

## Solution Assessment

No. Sharing link construction is a reasonable direction, but printing tokenized URLs in normal status output is not the safest match for current dashboard behavior; a token-free status hint or explicit delivery path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid printing bearer tokens in status output:** `src/cli/daemon-cli/status.print.ts:190`
  - body: This makes the normal human `openclaw gateway status` Dashboard line use `controlUiAccessUrl`, which can include `#token=...`. Current `openclaw dashboard` deliberately logs only the bare URL and uses the tokenized URL for browser/clipboard delivery, so this would put gateway bearer tokens into terminal output that users often copy into logs or support issues; keep status output token-free or add an explicit protected delivery path.
  - confidence: 0.93
- **[P2] Skip token resolution when the dashboard is disabled:** `src/cli/daemon-cli/status.gather.ts:333-337`
  - body: `gatherDaemonStatus` resolves Control UI access tokens unconditionally, even though `printDaemonStatus` will print `Dashboard: disabled` when `gateway.controlUi.enabled === false`. Since resolving this URL can touch SecretRef providers, disabled Control UI deployments can pay latency or see provider failures for a URL that will not be used; guard the resolver behind the daemon config’s Control UI enabled state.
  - confidence: 0.86
- **[P2] Use the canonical gateway token resolver:** `src/commands/control-ui-access-url.ts:40-45`
  - body: This new helper reimplements gateway token resolution with the older configured-secret fallback path while current main has `resolveGatewayAuthToken` for source/SecretRef/env-fallback precedence. Reintroducing a parallel auth resolver creates drift exactly on the auth-provider surface this PR is trying to align; build the access URL from the canonical resolver instead.
  - confidence: 0.82
