## Summary

Keep this PR open, but do not merge the current head: it addresses a real dashboard/status URL parity gap, while still introducing token-log regressions, dropping TLS-aware dashboard links, resolving token data for disabled Control UI, and lacking contributor real-behavior proof.

## What This Changes

The PR extracts a shared Control UI access URL resolver, uses it from `openclaw dashboard` and `openclaw gateway status`, adds token-fragment status URLs with JSON redaction and SecretRef hints, adds focused tests, and updates the changelog.

## Best Possible Solution

Rework the shared URL path against current main so status can surface a maintainer-approved authenticated access path without leaking secrets, while preserving TLS links, disabled-UI guards, SecretRef guidance, JSON redaction, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main prints a bare status dashboard URL while `openclaw dashboard` builds a tokenized URL for browser/clipboard delivery. The PR-introduced regressions are also source-reproducible from the PR head; I did not run a live gateway.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this implementation is not the best fix until it keeps dashboard logs token-free, preserves TLS and disabled-dashboard behavior, and resolves the human status token-output policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Log only the clean dashboard URL:** `src/commands/dashboard.ts:25`
  - body: Current main gives clipboard/browser the tokenized URL but logs only the clean Control UI URL, and the regression test forbids both the token value and `#token=` in logs. Logging `access.dashboardUrl` here can expose the gateway token in terminal output or captured logs.
  - confidence: 0.94
- **[P1] Keep tokens out of SSH hints:** `src/commands/dashboard.ts:49-53`
  - body: The SSH fallback hint is written to runtime logs when browser launch fails. Passing the resolved token into the hint can print a `#token=` URL, reversing the current token-free fallback behavior.
  - confidence: 0.9
- **[P2] Preserve TLS in access URLs:** `src/commands/control-ui-access-url.ts:39-44`
  - body: The new resolver calls `resolveControlUiLinks` without forwarding the gateway TLS flag. TLS-enabled gateways can therefore get `http://` dashboard access URLs even though the gateway serves HTTPS/WSS.
  - confidence: 0.9
- **[P2] Skip token resolution when Control UI is disabled:** `src/cli/daemon-cli/status.gather.ts:333-337`
  - body: `gateway status` already reports disabled Control UI as unavailable, but this code resolves dashboard token data unconditionally. That can invoke SecretRef providers and add latency or hidden provider failures for a URL status should not use.
  - confidence: 0.84
