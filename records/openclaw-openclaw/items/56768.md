## Summary

Keep open. This is a useful candidate for the dashboard/status token-parity problem linked from #45471, and the earlier runtime-port plus SecretRef-hint review comments have been addressed, but the current PR head is not merge-ready because it regresses the existing no-token-in-dashboard-logs contract, drops TLS-aware dashboard links, resolves access URLs even when Control UI is disabled, and needs explicit maintainer/secops judgment for token-bearing human status output.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `openclaw gateway status` to emit token-aware dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard` to use that resolver, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a revised version that centralizes Control UI access URL generation while preserving clean dashboard logs, carrying TLS and disabled-Control-UI state through the resolver, redacting JSON output, and adding contributor credit. The maintainers should explicitly decide whether human status output may show token fragments at all before that behavior ships.

## Reproduction Assessment

Yes. A plain `gateway.auth.token` makes the PR helper return `access.dashboardUrl` with `#token=...`, and the PR then logs that value from `dashboardCommand`; current main already has a regression test that rejects any token fragment in dashboard logs.

## Solution Assessment

No. The shared-resolver direction is reasonable, but this implementation is not the narrowest maintainable fix until it preserves the current token-log, TLS, and disabled-dashboard contracts and gets maintainer/secops agreement on human status token output.
