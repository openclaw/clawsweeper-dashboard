## Summary

Keep open. Current main has not implemented status/dashboard token URL parity, but the PR should not merge as-is because it leaks tokenized dashboard URLs into dashboard logs, drops TLS link state, resolves token data when Control UI is disabled, and needs maintainer/security approval for printing shared-secret URLs in human status output.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `openclaw gateway status` to emit token-aware dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard`, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rework the parity fix so `gateway status` uses approved tokenized-link behavior only on safe surfaces, keeps JSON redacted, preserves TLS and disabled-Control-UI behavior, and adds focused regression coverage plus contributor changelog credit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current dashboard logs only `links.httpUrl` and tests forbid token fragments, while the PR diff logs `access.dashboardUrl`; current status also preserves TLS and disabled UI state that the PR resolver path bypasses.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this implementation is not the narrow maintainable fix until it preserves token-log, TLS, and disabled-dashboard contracts and gets approval for human status token output.
