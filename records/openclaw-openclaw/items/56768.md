## Summary

This PR should stay open because the status/dashboard parity request is still useful, but the current branch would regress token-free dashboard logs, SSH fallback hints, TLS-aware links, and disabled-Control-UI behavior.

## What This Changes

The PR extracts shared Control UI access URL resolution for `openclaw dashboard` and `openclaw gateway status`, adds token-fragment handling and SecretRef hints, redacts status JSON output, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rework the status/dashboard URL sharing so status can surface an authenticated access path without logging secrets, while preserving TLS-aware links, disabled-dashboard guards, SecretRef guidance, JSON redaction, and focused regression tests.

## Reproduction Assessment

Yes for the PR regressions by source inspection: PR head logs tokenized dashboard URLs, passes token data into a logged SSH fallback hint, omits TLS state in the shared resolver, and resolves token data even when Control UI is disabled. I did not run a live gateway, so the original user symptom remains source-reproducible rather than locally reproduced here.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this implementation is not the best fix until it keeps terminal logs token-free, preserves TLS and disabled-dashboard behavior, and has explicit approval for any human status output that may contain shared-secret material.
