## Summary

Keep open: the branch targets a real status/dashboard token-auth parity gap, but the current diff would reintroduce token leaks in dashboard logs and SSH hints, drop TLS-aware dashboard links, resolve secrets even when Control UI is disabled, and still lacks real CLI behavior proof.

## What This Changes

The PR adds a shared Control UI access URL resolver for `openclaw dashboard` and `openclaw gateway status`, emits optional `#token=...` status URLs, redacts JSON output, adds SecretRef hints/tests, and updates the changelog.

## Best Possible Solution

Refactor onto current `resolveGatewayAuthToken` and `resolveControlUiLinks` paths so status can surface an auth-capable dashboard path without logging secrets, while preserving TLS, disabled-Control-UI, SecretRef, JSON redaction, and focused tests.

## Reproduction Assessment

Yes, by source inspection: current `gateway status` prints only `links.httpUrl` while `openclaw dashboard` can deliver a tokenized URL to browser/clipboard. The PR-introduced regressions are also source-reproducible from the diff and current guardrail tests.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this branch is not the best fix until it preserves current token-log, TLS, and disabled-UI guardrails and gets explicit maintainer approval for token-bearing status output.
