## Summary

Keep this PR open, but it is not currently safe to merge: the branch reintroduces token exposure in dashboard logs, loses TLS-aware links, resolves token data for disabled Control UI, and still lacks real after-fix behavior proof.

## What This Changes

This PR extracts dashboard access URL resolution into a shared helper, wires it into `gateway status`, redacts token fragments in JSON output, adds SecretRef hints/tests, and updates the changelog.

## Best Possible Solution

Rework the shared URL path so status can surface an authenticated access path without logging secrets, while preserving TLS-aware links, disabled-dashboard guards, SecretRef guidance, JSON redaction, and focused regression tests.

## Reproduction Assessment

Yes, for the PR-introduced regressions by source inspection: the diff logs `access.dashboardUrl`, forwards token data into a logged SSH hint, omits `tlsEnabled`, and resolves access data without checking whether Control UI is disabled. I did not run a live gateway, so the original user flow remains source-proven rather than locally reproduced.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this implementation is not the best fix until it keeps terminal logs token-free, preserves TLS and disabled-dashboard behavior, and resolves the policy question for human status output containing shared-secret material.

  - body: The shared resolver calls `resolveControlUiLinks` without the gateway TLS flag. TLS-enabled gateways can therefore get `http://` dashboard URLs even though the gateway serves HTTPS/WSS.
  - confidence: 0.9

  - body: `gateway status` already treats disabled Control UI as unavailable, but this code resolves dashboard access data unconditionally. That can invoke SecretRef/token providers and add avoidable latency or provider errors for a URL status should not show or use.
  - confidence: 0.84
