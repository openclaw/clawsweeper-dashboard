## Summary

Keep open: the status/dashboard token-parity problem is still not implemented on current main, and the linked issue was closed as superseded by this PR, but this branch is not mergeable as-is because it reintroduces token logging, drops TLS-aware links, resolves token data when Control UI is disabled, lacks real behavior proof, and needs explicit maintainer/secops judgment for printing token-bearing URLs in human status output.

## What This Changes

The PR adds shared Control UI access URL resolution for `openclaw dashboard` and `openclaw gateway status`, with token-fragment handling, SecretRef hints, JSON redaction, tests, and a changelog entry.

## Best Possible Solution

Rework the status URL parity on current main with an approved token-delivery policy, preserving token-free dashboard logs and SSH hints, TLS-aware links, disabled-Control-UI guards, JSON redaction, SecretRef guidance, focused tests, and real CLI proof.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the PR regressions: tokenized URLs are logged at PR head, TLS state is not passed into the shared resolver, and status token resolution runs before any disabled-Control-UI guard.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this implementation is not the best fix until it preserves the current security/TLS/disabled-dashboard contracts and gets explicit approval for human status output that may contain token fragments.
