## Summary

Keep open. Current main has not implemented the requested token-aware `gateway status` URL, and the PR is useful source work, but the current head would regress token secrecy, TLS-aware dashboard links, and disabled-Control-UI status behavior.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `openclaw gateway status` to emit token-aware dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard`, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the URL-parity idea, but rework it on current main so tokenized URLs appear only on approved surfaces, JSON stays redacted, dashboard logs remain clean, TLS and disabled-Control-UI behavior are preserved, and focused tests cover each branch.

## Reproduction Assessment

Yes. With a plain or env-backed gateway token, the PR resolver returns a `#token=` URL and `dashboardCommand` logs it; with `gateway.tls.enabled=true`, the PR resolver still omits `tlsEnabled` and returns an `http://` dashboard URL.

## Solution Assessment

No. A shared resolver is a reasonable direction, but this implementation is not the narrow maintainable fix until it preserves the existing token-log, TLS-link, and disabled-dashboard contracts and gets explicit approval for human status token output.
