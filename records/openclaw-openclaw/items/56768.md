Full review comments:

- **[P1] Keep token fragments out of dashboard logs:** `src/commands/dashboard.ts:25`
  - body: `dashboardUrl` includes `#token=...` when the helper resolves a local token, but this line logs that full URL. Current main has explicit CVE regression coverage that the tokenized URL only goes to clipboard/browser and never appears in `runtime.log`, so this would leak the gateway bearer token into terminal/log capture.
  - confidence: 0.94
- **[P1] Pass TLS state into the access URL resolver:** `src/commands/control-ui-access-url.ts:39-44`
  - body: The new helper calls `resolveControlUiLinks` without `tlsEnabled`, so TLS-enabled gateways build `http://` dashboard access URLs even though current status/dashboard paths pass TLS state and probe with `wss://`. This breaks existing TLS setups by printing or opening the wrong scheme.
  - confidence: 0.9
- **[P2] Preserve the dashboard readiness gate:** `src/commands/dashboard.ts:20`
  - body: The PR head resolves and opens/copies the dashboard URL directly, but current main first calls `ensureGatewayReadyForOperation` and skips copy/open when the gateway is not ready. Replaying this branch without restoring that gate would bring back dead dashboard launches and stale token delivery.
  - confidence: 0.86
- **[P2] Skip token resolution when Control UI is disabled:** `src/cli/daemon-cli/status.gather.ts:333-337`
  - body: `gatherDaemonStatus` resolves the Control UI access URL unconditionally before the print layer checks `gateway.controlUi.enabled`. For disabled Control UI deployments this still performs SecretRef/token resolution, which can hit external secret providers or add failures for a URL that will not be rendered.
  - confidence: 0.82
