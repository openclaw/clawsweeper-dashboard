## Summary

Keep open. Current main still prints a non-tokenized dashboard URL from `gateway status`, so this PR is not obsolete, but the branch is not mergeable as-is because it reintroduces token logging in `openclaw dashboard`, drops the current TLS link behavior, and resolves token/SecretRef data even when Control UI is disabled.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `openclaw gateway status` to emit token-aware dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard` to use that resolver, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rework this on current main using the existing gateway token and Control UI link helpers, preserving clean dashboard logs, TLS-aware links, and disabled-Control-UI behavior, then add focused coverage for status parity, JSON redaction, TLS links, disabled UI, and no token fragments in dashboard logs. A maintainer/secops decision should explicitly cover whether human status output may include token fragments.

## Reproduction Assessment

Yes. For the PR defects, a plain `gateway.auth.token` makes the proposed helper return `access.dashboardUrl` with `#token=...`, and the PR then logs that value from `dashboardCommand`; the current dashboard regression test is the guardrail that would catch this.

## Solution Assessment

No. The shared-resolver direction is reasonable, but this implementation is not the best fix until it preserves the current token-log, TLS, and disabled-dashboard contracts and gets maintainer/secops agreement on human status token output.
