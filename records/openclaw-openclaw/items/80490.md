## Summary

Keep open. Current main has clearer plugin-packaging wording for source-only TypeScript packages, but the Gateway invalid-config startup path still appends the generic `openclaw doctor --fix` repair hint, while doctor’s plugin repair paths reinstall or clean plugin registry state rather than building third-party package source.

## What This Changes

The issue asks OpenClaw either to make `doctor --fix` build unbuilt TypeScript plugin packages or to stop telling users that doctor can repair that packaging failure.

## Best Possible Solution

Keep packaged-plugin runtime ownership with the plugin publisher, and make Gateway startup guidance suppress or replace the generic doctor repair hint when validation already identifies an unbuilt package as a publisher-side packaging issue.

## Reproduction Assessment

Yes, source-reproducible. The source path shows a source-only global plugin is excluded with a packaging diagnostic, the configured memory slot then becomes `plugin not found`, and Gateway startup still appends the generic doctor repair hint.

## Solution Assessment

Yes, the best narrow fix is to improve the startup/validation guidance rather than auto-build third-party package source in doctor. Auto-building would expand doctor into package build execution and should need explicit maintainer approval if pursued.
