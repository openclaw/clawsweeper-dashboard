## Summary

Keep open. Current main still emits the source-only plugin packaging diagnostic and then gateway startup appends the generic `openclaw doctor --fix` hint, while doctor repairs managed plugin installs by reinstalling/updating rather than building third-party TypeScript source.

## What This Changes

The issue asks doctor to auto-build source-only TypeScript plugin packages or stop directing those plugin-packaging errors to `doctor --fix`.

## Best Possible Solution

Keep packaged-plugin runtime artifacts as publisher-owned and add a narrow gateway/config-formatting fix that suppresses or replaces the generic doctor hint when validation already identifies an unbuilt plugin package as a packaging issue.

## Reproduction Assessment

Yes, source-reproducible. Current main shows a source-only installed plugin is diagnosed as a packaging issue, while gateway invalid-config formatting still appends the generic `doctor --fix` hint whenever `includeDoctorHint` is enabled.

## Solution Assessment

Yes, the best narrow solution is to improve startup/validation guidance rather than auto-build third-party plugin source. Auto-building would expand doctor into arbitrary package build execution and should be separate maintainer-approved work if pursued.

Fix https://github.com/openclaw/openclaw/issues/80490 by removing the misleading generic `openclaw doctor --fix` guidance for installed plugin package diagnostics that say the package requires compiled runtime output and is a plugin packaging issue. Keep the fix narrow: do not teach doctor to run third-party `npm run build`, `tsdown`, `tsc`, or other package build scripts. In the gateway invalid-config formatting path, suppress or replace the generic recovery hint when validation issues include the source-only plugin packaging diagnostic, while preserving the doctor hint for ordinary invalid configs. Add focused regression coverage for both branches and add a single user-facing changelog fix entry if this lane owns changelog updates. Do not change plugin loading policy, package install trust, or bundled/external plugin ownership semantics.
