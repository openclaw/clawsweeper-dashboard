## Summary

Keep open. Current main still appends the generic `doctor --fix` hint to gateway invalid-config startup errors, while source-only package diagnostics remain publisher packaging issues and the linked fix PR was closed unmerged.

## What This Changes

The issue requests that unbuilt TypeScript plugin packages either be built by `doctor --fix` or stop being accompanied by misleading doctor repair guidance.

## Best Possible Solution

Update gateway invalid-config formatting to suppress or replace the generic doctor hint when validation already identifies a plugin packaging issue, while keeping packaged runtime output owned by the plugin publisher.

## Reproduction Assessment

Yes, source-reproducible. Current main still combines the source-only package diagnostic, the explicit missing memory slot issue, and the unconditional gateway `doctor --fix` suffix for invalid startup config snapshots.

## Solution Assessment

Yes: suppressing or replacing the generic startup hint for packaging diagnostics is the narrowest maintainable fix. Auto-building third-party plugin packages in doctor is a broader and riskier execution-policy change.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80490 by making Gateway startup stop appending the generic `Run "openclaw doctor --fix"` recovery hint when the invalid config issues already include a plugin packaging diagnostic such as `plugin packaging issue, not a local config problem` or `requires compiled runtime output for TypeScript entry`. Keep the normal doctor hint for legacy config, stale plugin state, and other repairable invalid-config failures. Do not teach doctor to run `npm run build`, `tsdown`, `tsc`, or any third-party package build code. Add focused regression coverage for packaging-diagnostic invalid snapshots and non-packaging invalid snapshots. Add a changelog entry if the user-facing startup guidance changes.
