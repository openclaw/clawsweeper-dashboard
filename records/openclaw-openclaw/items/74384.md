## Summary

Keep open. Current main still has a source-level path where restrictive `tools.profile: "coding"` filtering excludes separately registered non-optional plugin tools whose names are not in the coding profile, matching the report that only `memory_search` and `memory_get` survive.

## What This Changes

The issue asks OpenClaw to expose third-party plugin tools registered through `api.registerTool()` after the 4.26 SDK/tool-profile changes.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level path: with `tools.profile: "coding"`, separately registered required plugin tools whose names are not in the coding profile are filtered, while `memory_search` and `memory_get` survive because they are coding-profile memory tools. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the best behavior depends on maintainer policy. The narrow choices are either to exempt required plugin tools from profile-only filtering while still honoring deny lists, or to document and diagnose that restrictive profiles require `full`, `alsoAllow`, plugin id, or explicit tool entries.
