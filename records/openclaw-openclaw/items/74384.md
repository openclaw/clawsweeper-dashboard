## Summary

Keep open. Current main still has a source-level path where required `api.registerTool()` plugin tools are filtered down to names that match the active restrictive tool profile, which matches the reporter seeing only `memory_search` and `memory_get` under the coding profile.

## What This Changes

The issue requests restoring exposure of all required tools registered by an external plugin after the 4.26 plugin SDK/tool-policy changes.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level path: a plugin whose `contracts.tools` includes `memory_search`, `memory_get`, and additional required tools under `tools.profile: "coding"` is selected through the two profile-matching names, then its other registered tools are filtered out. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the correct behavior is a product/security contract decision. The maintainable fix is either to preserve required plugin tools under profile filtering while honoring deny lists, or to document and diagnose that restrictive profiles require explicit `alsoAllow` entries for plugin-owned tools.
