## Summary

Keep open. Current main still has the credible core mismatch described in the prior review: required plugin tools are loaded, but restrictive tool profiles can filter them out before the model sees them unless the user explicitly grants the tool name or plugin id at the profile stage.

## What This Changes

The issue reports that a third-party plugin’s required `api.registerTool()` tools stopped being model-visible after the 4.26 upgrade, with only `memory_search` and `memory_get` exposed.

## Best Possible Solution

Define the contract between required plugin tools and restrictive tool profiles, then align implementation, docs, diagnostics, and tests. The maintainable end state is either an explicit maintainer-approved opt-in path for memory-like plugins, such as `tools.alsoAllow` by plugin id/tool name, or a narrower trusted capability/profile policy that does not silently expose arbitrary third-party side-effecting tools.

## Reproduction Assessment

Yes for the core mechanism: a synthetic plugin that registers `memory_search`, `memory_get`, and another required tool under `tools.profile: "coding"` should show the extra required tool being filtered by the profile pipeline. The exact external MemOS package and Windows environment were not exercised here.

## Solution Assessment

Unclear because there is no patch and the remaining decision is product/security policy. Exposing every required third-party tool through restrictive profiles is riskier than a documented profile-stage opt-in or a maintainer-approved memory-plugin policy.
