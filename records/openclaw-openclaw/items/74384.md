## Summary

Keep open. Current main still makes the reported symptom source-reproducible: required third-party tools registered with `api.registerTool()` are selected through the active profile allowlist, and the `coding` profile only overlaps the reporter's MemOS tool set on `memory_search` and `memory_get`.

## What This Changes

The issue requests that third-party plugin tools registered via `api.registerTool()` remain exposed to the agent after the 4.26 plugin SDK and tool-profile changes.

## Best Possible Solution

Define the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, from source inspection. A plugin that declares and registers `memory_search`, `memory_get`, and additional required tools under `tools.profile: "coding"` will only keep entries matching the profile allowlist or an explicit plugin/tool grant.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice affects tool-access policy. `tools.profile: "full"` or explicit allow entries are useful workarounds, but they do not resolve whether required third-party tools should bypass restrictive profile semantics.
