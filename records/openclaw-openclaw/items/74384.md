## Summary

Keep open. Current main still routes plugin-registered tools through restrictive profile allowlists, so a `coding` profile can select only `memory_search` and `memory_get` while dropping the reported additional MemOS tools unless the operator uses `full` or explicit allow entries; the remaining work is a plugin-tool contract decision, not cleanup closure.

## What This Changes

The issue reports third-party `api.registerTool()` tools becoming unavailable after the 4.26 upgrade, with only `memory_search` and `memory_get` exposed to the agent.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, changelog, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level reproduction path: enable a plugin that declares required `memory_search`, `memory_get`, and additional required registered tools under `tools.profile: "coding"`; only names matching the coding allowlist or explicit plugin allow entries survive. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the best behavior depends on maintainer policy. The narrow implementation should either exempt required plugin tools from profile-only filtering while still honoring denies, or document and diagnose that restrictive profiles require `full` or `alsoAllow` for plugin-owned tools.
