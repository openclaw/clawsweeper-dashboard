## Summary

Keep open. Current main still has a real plugin-tool exposure/migration problem to resolve: `registerTool()` is supported, but current resolver policy depends on manifest `contracts.tools` and active tool allowlists, while the reported MemOS package registers many tools without a matching manifest contract in the inspected v1.0.10 package.

## What This Changes

The issue reports that a third-party MemOS plugin using `api.registerTool()` exposes only `memory_search` and `memory_get` after the 4.26 upgrade instead of all registered tools.

## Best Possible Solution

Define the 4.26+ plugin-tool migration contract for external plugins, then align OpenClaw diagnostics/docs and any compatibility behavior with that contract while preserving manifest-first tool discovery.

## Reproduction Assessment

Yes, at source level: current main rejects or filters plugin tools unless they are declared in `contracts.tools` and selected by the active tool policy, and the inspected MemOS v1.0.10 package registers the missing tools without a manifest `contracts.tools` list. I did not run the plugin live on Windows.

## Solution Assessment

Unclear because there is no proposed patch and the right outcome is a maintainer contract decision. The narrow paths are either external package migration to declare all tools plus explicit profile guidance, or an OpenClaw compatibility/diagnostic change if maintainers want older `registerTool()`-only plugins to degrade more clearly.
