## Summary

Keep open. Current main still supports the reporter's core diagnosis: required third-party `api.registerTool()` tools can be filtered by restrictive tool profiles, and the visible `memory_search`/`memory_get` pair matches the built-in `coding` profile rather than proving all MemOS tools are exposed.

## What This Changes

The issue asks OpenClaw to restore exposure of third-party plugin `api.registerTool()` tools after the 4.26 upgrade so MemOS required tools beyond `memory_search` and `memory_get` are available to the agent.

## Best Possible Solution

Define the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics/docs, and regression coverage around that contract.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enable a plugin that declares/registers `memory_search`, `memory_get`, and additional required tools under `tools.profile: "coding"`; current resolver and policy code can preserve only entries that match the profile/allowlist.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` may be too broad; an explicit approved contract with diagnostics and tests is safer.
