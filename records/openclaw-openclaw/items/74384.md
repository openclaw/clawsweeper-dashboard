## Summary

Keep open. Current main still has a source-level path where a coding-profile tool allowlist exposes only plugin tools whose names match the profile, which explains the reported `memory_search`/`memory_get` survival while leaving the broader third-party plugin tool contract unresolved.

## What This Changes

The issue asks OpenClaw to restore or explain exposure of third-party plugin tools registered with `api.registerTool()` after the 4.26 upgrade.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, source-level: enable a plugin whose `contracts.tools` includes `memory_search`, `memory_get`, and additional registered tools under `tools.profile: "coding"`; only names matching the coding allowlist are selected. I did not establish a live MemOS Windows run.

## Solution Assessment

Unclear because there is no patch and the right behavior depends on maintainer policy. The narrow choices are to exempt required plugin tools from profile-only filtering while honoring denylists, or document and diagnose that restrictive profiles require `full`, plugin id, or explicit `alsoAllow` entries.
