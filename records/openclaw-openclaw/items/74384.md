## Summary

Keep open. Current main still has a credible plugin SDK/tool-policy mismatch: required third-party `api.registerTool()` tools are discovered and filtered through restrictive profile allowlists, so `tools.profile: "coding"` can leave only names that also match the built-in coding profile, such as `memory_search` and `memory_get`.

## What This Changes

The issue asks OpenClaw 4.26 to expose all required `api.registerTool()` tools from the MemOS plugin to the agent instead of only the built-in memory pair.

## Best Possible Solution

Define the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics/docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, source-reproducible for the core mechanism: enable a plugin that declares/registers `memory_search`, `memory_get`, and additional required tools under `tools.profile: "coding"`, then inspect the resolved tool names. Source inspection shows non-profile plugin tools are filtered unless the plugin id, tool name, or `group:plugins` is explicitly allowed.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` may be too broad; explicit opt-in, targeted diagnostics/docs, or an approved trusted memory-plugin/profile contract would be safer.
