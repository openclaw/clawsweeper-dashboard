## Summary

Keep open. Current main still has a credible contract mismatch: required third-party `registerTool()` tools can be loaded successfully, then filtered out by the final restrictive `tools.profile: "coding"` policy unless the operator explicitly allowlists the plugin, tool, or `group:plugins`. The already-linked #73748 conversation-hook workaround does not cover this later tool-profile filtering stage.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party memory plugin still registers several tools with `api.registerTool()` but only `memory_search` and `memory_get` remain visible to the agent.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable outcome should either document and warn for explicit opt-in via plugin id/tool/group `alsoAllow`, or add a maintainer-approved trusted memory-plugin/profile grant rather than silently exposing all required third-party tools.

## Reproduction Assessment

Yes for the core mechanism: register an enabled plugin with required `memory_search`, `memory_get`, and one additional required tool, set `tools.profile: "coding"`, and inspect the final tool names after `applyToolPolicyPipeline`; the extra plugin tool is filtered unless explicitly allowlisted. The exact MemOS plugin was not live-run.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` is probably too broad; an explicit opt-in, plugin-id/group allowlist, diagnostic, or trusted memory-plugin contract is safer.
