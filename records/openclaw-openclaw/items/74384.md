## Summary

Keep open. Current main still has a credible plugin SDK/tool-policy mismatch: required third-party `api.registerTool()` tools can be withheld from the agent when `tools.profile: "coding"` creates an allowlist that only happens to match `memory_search` and `memory_get`.

## What This Changes

The issue reports that a third-party MemOS plugin using `api.registerTool()` exposes only the built-in memory pair after upgrading to OpenClaw 4.26.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, diagnostics, and docs around either explicit opt-in or a maintainer-approved trusted memory-plugin/profile grant.

## Reproduction Assessment

Yes for the core mechanism: enable a plugin that declares `memory_search`, `memory_get`, and another required tool, set `tools.profile: "coding"`, and inspect resolved tool names. Source inspection shows the extra plugin tool is filtered unless the tool, plugin id, `*`, or `group:plugins` is allowed.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is product/security-sensitive. Explicit opt-in plus clearer diagnostics/docs is safer than automatically exposing all required third-party tools under `coding`, unless maintainers want a trusted memory-plugin exception.
