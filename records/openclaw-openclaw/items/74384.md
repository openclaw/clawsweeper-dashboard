## Summary

Keep open. Current main still has a credible OpenClaw contract/policy mismatch: required third-party `registerTool()` tools can be materialized successfully, then removed by the final restrictive `tools.profile: "coding"` policy before the model sees them. The linked #73748 workaround covers conversation hook access, not this profile-stage tool exposure behavior.

## What This Changes

The issue asks OpenClaw to restore agent visibility for multiple third-party tools registered with `api.registerTool()` after upgrading to 4.26, where only `memory_search` and `memory_get` remain exposed.

## Best Possible Solution

Define the contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable outcome should either add a maintainer-approved explicit opt-in for memory-like plugin tool groups or make the plugin-id/tool `alsoAllow` path clear and safe for operators.

## Reproduction Assessment

Yes for the core mechanism: register a synthetic enabled plugin with required `memory_search`, `memory_get`, and one extra required tool, set `tools.profile: "coding"`, and inspect the final assembled tool names after `applyToolPolicyPipeline`; the extra plugin tool is filtered by the profile allowlist. The exact external MemOS plugin was not live-run in this read-only review.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Exposing all required third-party tools through `coding` would be risky; an explicit documented opt-in, plugin-id/group allowlist, or maintainer-approved memory-plugin profile contract is safer.
