## Summary

Keep open. Current main still has a credible SDK/tool-policy contract mismatch: required third-party tools registered through `api.registerTool()` are materialized, but a restrictive `tools.profile: "coding"` is applied afterward and only grants the built-in memory pair plus `bundle-mcp`, so additional memory-plugin tools can disappear unless explicitly allowlisted.

## What This Changes

The issue asks OpenClaw to restore or clarify exposure of third-party plugin tools registered with `api.registerTool()` after the 4.26 upgrade, where only `memory_search` and `memory_get` remain agent-visible for the reported memory plugin.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. The durable result should either document explicit opt-in through plugin id/tool/group `alsoAllow`, or add a maintainer-approved trusted memory-plugin/profile grant instead of silently exposing all required third-party tools.

## Reproduction Assessment

Yes for the core mechanism: enable a plugin with required `memory_search`, `memory_get`, and one additional required tool, set `tools.profile: "coding"`, then inspect the final names after `applyToolPolicyPipeline`; the extra plugin tool is filtered unless the plugin, tool, or `group:plugins` is explicitly allowed. The exact MemOS package was not live-run.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` is likely too broad; an explicit opt-in, targeted diagnostic/docs fix, or trusted memory-plugin contract is safer.
