## Summary

Keep open. Current main still has a credible plugin SDK/tool-profile mismatch: restrictive profiles feed plugin tool discovery and policy filtering, so required third-party `api.registerTool()` tools that are not explicitly allowed can be hidden even though plugin docs describe required tools as always available.

## What This Changes

The issue asks OpenClaw to restore visibility for all required `api.registerTool()` tools from the MemOS plugin after the 4.26 plugin SDK/tool-policy changes.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and regression tests around explicit opt-in or a maintainer-approved trusted plugin/profile grant.

## Reproduction Assessment

Yes for the core mechanism: enable a plugin that registers `memory_search`, `memory_get`, and another required tool, set `tools.profile: "coding"`, and inspect the resolved tool names. Current source shows the extra plugin tool is skipped unless the tool, plugin id, or `group:plugins` is explicitly allowed.

## Solution Assessment

Unclear because there is no proposed patch and the narrowest correct policy has not been chosen. Blanket exposure of all required third-party tools under restrictive profiles may widen the tool surface; explicit opt-in, stronger diagnostics/docs, or a trusted memory-plugin contract looks safer.
