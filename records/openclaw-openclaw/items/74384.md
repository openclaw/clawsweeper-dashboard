## Summary

Keep open. Current main still supports the reported failure mode: plugin tools can be successfully materialized, then the final tool-policy profile stage can filter them back to the built-in coding-profile memory pair unless the operator explicitly allowlists the plugin/tools.

## What This Changes

The issue reports that a third-party MemOS plugin registers several tools with `api.registerTool()`, but after the 4.26 upgrade only `memory_search` and `memory_get` are exposed to the agent.

## Best Possible Solution

Define the intended contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A maintainable endpoint is either an explicit plugin/tool allowlist path that operators can understand, or a maintainer-approved memory-plugin profile contract that does not silently widen third-party write/share/publish tools.

## Reproduction Assessment

Yes for the core mechanism: a synthetic plugin with `memory_search`, `memory_get`, and another required tool under `tools.profile: "coding"` should show the extra tool materialized, then filtered out by `applyToolPolicyPipeline`. The exact MemOS package was not live-run because this review is read-only.

## Solution Assessment

Unclear because there is no proposed patch and the remaining question is policy. Exposing all required third-party tools through restrictive profiles is not automatically the safest fix; an explicit documented opt-in or maintainer-approved memory-like plugin contract is safer.
