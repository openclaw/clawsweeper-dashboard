## Summary

Keep open. Current main still makes the reported plugin-tool exposure failure source-reproducible under restrictive tool profiles: the coding profile admits `memory_search` and `memory_get`, but plugin discovery and final filtering still drop other third-party plugin tool names unless the profile is `full` or an explicit plugin/tool allow entry is present.

## What This Changes

The issue reports that a third-party plugin’s `api.registerTool()` tools stopped being exposed after the 4.26 upgrade, leaving only `memory_search` and `memory_get` visible to the agent.

## Best Possible Solution

Settle one contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, changelog, and regression coverage with that contract.

## Reproduction Assessment

Yes, source-level. Enable a plugin whose manifest declares required `memory_search`, `memory_get`, and additional required registered tools under `tools.profile: "coding"`; current main selects and preserves only allowlist-matching names unless the plugin/tool is explicitly allowed or the profile is `full`.

## Solution Assessment

Unclear because there is no proposed patch and the safer behavior is a maintainer policy choice. The narrow options are to exempt required plugin tools from profile-only filtering while still honoring deny lists, or explicitly document and diagnose that restrictive profiles require `full`/`alsoAllow` for plugin-owned tools.
