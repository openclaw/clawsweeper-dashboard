## Summary

Keep open. Current main still leaves the reported third-party plugin tool exposure behavior source-reproducible: registered plugin tools are gathered, but restrictive tool-profile allowlists can filter required plugin tools unless they match the profile names, plugin allow entries, wildcards, or plugin group allowances.

## What This Changes

The issue requests that third-party plugin tools registered with `api.registerTool()` remain exposed to agents after the 4.26 plugin SDK/tooling changes.

## Best Possible Solution

Define the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a plugin with required `memory_search`, `memory_get`, and additional registered tools under `tools.profile: "coding"` will keep only entries matching the profile or an explicit plugin allow entry.

## Solution Assessment

Unclear. There is no proposed patch, and the remaining decision is whether required third-party plugin tools should bypass restrictive profile semantics or instead require explicit profile/allowlist configuration.
