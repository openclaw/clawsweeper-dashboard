## Summary

Keep open. Current main still has a credible OpenClaw core contract/policy mismatch: third-party `registerTool()` tools are materialized, but the coding profile can filter the final model-visible list back to the two built-in memory tool names unless the operator explicitly allowlists the plugin/tool. The related hooks workaround addresses conversation access, not this profile-stage tool exposure behavior.

## What This Changes

The issue asks OpenClaw to keep third-party plugin tools registered with `api.registerTool()` model-visible on 4.26 instead of reducing the plugin to `memory_search` and `memory_get`.

## Best Possible Solution

Define the contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable fix should either add a maintainer-approved profile-stage opt-in for memory-like plugins or make the plugin-id/tool allowlist path explicit and safe for operators.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core mechanism: register a synthetic plugin with `memory_search`, `memory_get`, and another required tool, enable `tools.profile: "coding"`, and inspect the final assembled tool names after `applyToolPolicyPipeline`. The external MemOS setup was not live-run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch and the open question is policy. Exposing all required third-party tools through restrictive profiles would be risky; an explicit, documented opt-in or maintainer-approved memory-plugin profile contract is safer.
