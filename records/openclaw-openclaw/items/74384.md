## Summary

Keep open. Current main still has a credible SDK contract versus restrictive profile mismatch: required third-party plugin tools can register successfully, but `tools.profile: "coding"` applies a final allowlist that only includes the built-in memory pair, so extra memory-like tools remain filtered unless the operator adds an explicit allow/profile exception.

## What This Changes

The issue requests that all required tools registered by the MemOS plugin via `api.registerTool()` remain visible to the agent after the 4.26 plugin SDK/tool-policy changes.

## Best Possible Solution

Define the contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable outcome should either document the explicit `tools.alsoAllow`/plugin-id path for memory-like plugins or add a maintainer-approved safe opt-in/profile extension for active memory-plugin tool groups.

## Reproduction Assessment

Yes for the core mechanism: enable a synthetic plugin that registers `memory_search`, `memory_get`, and one extra required tool, set `tools.profile: "coding"`, and inspect the final tool list after `applyToolPolicyPipeline`. The extra required plugin tool is materialized first and then filtered by the profile allowlist.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Exposing every required third-party plugin tool through `coding` would be too broad; an explicit documented opt-in, plugin-id/group allowlist, or memory-plugin-specific profile contract is safer.
