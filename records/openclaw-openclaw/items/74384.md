## Summary

Keep open. Current main still leaves the reported `api.registerTool()` exposure failure source-reproducible under restrictive tool profiles, and the docs still describe required plugin tools as always available, so this needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue reports that a third-party plugin's `api.registerTool()` tools disappeared after the 4.26 upgrade except for `memory_search` and `memory_get`.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level reproduction path: enable a plugin with required `memory_search`, `memory_get`, and additional required registered tools under `tools.profile: "coding"`; only names that match the coding allowlist or an explicit plugin allow entry survive. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the best behavior depends on maintainer policy. The narrow implementation would either exempt required plugin tools from profile-only filtering while still honoring deny lists, or explicitly document and diagnose that restrictive profiles require `full`/`alsoAllow` for plugin-owned tools.
