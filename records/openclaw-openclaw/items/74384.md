## Summary

Keep open. Current main still has a credible contract mismatch: non-optional `registerTool()` tools are materialized, but the final restrictive `tools.profile: "coding"` stage only grants the built-in memory tool names, so additional MemOS-style required tools remain filtered unless explicitly allowed.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party MemOS plugin registering multiple tools via `api.registerTool()` exposes only `memory_search` and `memory_get` to the agent.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable outcome should either document and warn for explicit opt-in via plugin id, tool name, or `group:plugins` in `tools.alsoAllow`, or add a narrow maintainer-approved trusted memory-plugin/profile grant rather than exposing every required third-party tool under `coding`.

## Reproduction Assessment

Yes for the core mechanism: register an enabled plugin with required `memory_search`, `memory_get`, and one additional required tool, set `tools.profile: "coding"`, and inspect final tool names after `applyToolPolicyPipeline`; the extra plugin tool is filtered unless explicitly allowed. The exact MemOS package was not live-run.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` is likely too broad; an explicit opt-in, diagnostic, or trusted memory-slot contract is safer.
