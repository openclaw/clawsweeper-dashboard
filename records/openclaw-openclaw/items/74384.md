## Summary

Keep open. Current main still has a source-level path where `tools.profile: "coding"` narrows plugin tool discovery to profile-listed tool names, so required third-party `api.registerTool()` tools like the reported MemOS write/share/timeline tools can be skipped unless explicitly allowlisted.

## What This Changes

The issue reports that a MemOS plugin using `api.registerTool()` exposes only `memory_search` and `memory_get` after upgrading to OpenClaw 4.26.

## Best Possible Solution

Define the supported SDK/tool-profile contract for required third-party plugin tools, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level reproduction: under `tools.profile: "coding"`, registered plugin tools whose names are absent from the coding allowlist can be skipped, while `memory_search` and `memory_get` survive because they are coding-profile memory tools. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the best behavior requires maintainer policy. The narrow choices are to exempt required plugin tools from profile-only filtering while still honoring deny lists, or to document and diagnose the explicit `alsoAllow` requirement.
