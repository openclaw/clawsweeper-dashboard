## Summary

Keep open. Current `main` still has a credible plugin SDK/tool-policy mismatch: once `tools.profile: "coding"` contributes an allowlist, required third-party plugin tools are filtered unless the plugin id, tool name, or `group:plugins` is explicitly allowed.

## What This Changes

Report that after the 4.26 upgrade only `memory_search` and `memory_get` from the MemOS plugin are exposed while other required `api.registerTool()` tools disappear from the agent.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive profiles, then align code, docs, diagnostics, and regression tests around explicit opt-in or a maintainer-approved trusted memory-plugin grant.

## Reproduction Assessment

Yes. A memory-like plugin that registers `memory_search`, `memory_get`, and another required tool under `tools.profile: "coding"` will expose the built-in memory pair but filter the extra plugin tool unless the plugin, tool, or `group:plugins` is allowed.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` is likely too broad; explicit opt-in, targeted diagnostics/docs, or a trusted memory-plugin contract is safer.
