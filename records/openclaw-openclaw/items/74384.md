## Summary

Keep open. Current main still has an actionable SDK/tool-policy mismatch to resolve: the reported MemOS package is not declaring `contracts.tools`, which current OpenClaw rejects, and restrictive `tools.profile: "coding"` also narrows discovered plugin tools to explicitly allowed names such as `memory_search` and `memory_get`.

## What This Changes

Investigate why a third-party MemOS plugin’s `api.registerTool()` tools disappeared from the agent tool list after upgrading to OpenClaw 4.26.

## Best Possible Solution

Define the supported 4.26 contract for third-party required plugin tools, then align diagnostics/docs and regression coverage so missing `contracts.tools` and restrictive-profile filtering both produce clear, intentional behavior.

## Reproduction Assessment

Yes, source-reproducible for the core mechanisms: the published MemOS v1.0.10 package registers tools without `contracts.tools`, which current registry code rejects, and current profile policy can further filter required plugin tools under `coding`. I did not establish a live Windows/MemOS runtime reproduction in this read-only review.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. The safest path is not blanket exposure of all third-party required tools, but a maintainer decision on manifest migration, diagnostics, and whether explicit `tools.alsoAllow`, plugin-id allowlisting, or profile semantics should govern these tools.
