## Summary

Keep open. Current main still shows a credible plugin SDK/tool-policy mismatch: restrictive profiles feed plugin tool discovery and the final policy filter, so required third-party tools outside the active profile can remain hidden unless explicitly allowlisted, while the plugin docs still describe required `api.registerTool(...)` tools as always available.

## What This Changes

The issue asks OpenClaw to restore `api.registerTool(...)` visibility for the MemOS plugin after 4.26 so all required tools appear to the agent, not only `memory_search` and `memory_get`.

## Best Possible Solution

Define and document the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage without silently widening untrusted plugin tools beyond that contract.

## Reproduction Assessment

Yes, source-reproducible for the core mechanism: use an enabled plugin with required tools named both inside and outside the coding profile, then resolve tools under `tools.profile: "coding"`. Source inspection shows non-profile plugin tools are filtered unless the tool name, plugin id, or `group:plugins` is explicitly allowed.

## Solution Assessment

Unclear because there is no proposed patch and the remaining fix is policy-sensitive. Blanket exposure of all required third-party tools under restrictive profiles may be too broad; an explicit maintainer-approved contract plus diagnostics/docs would be safer.
