## Summary

Keep open. Current main still leaves a credible plugin-tool/profile policy mismatch: required third-party `api.registerTool()` tools are gated by restrictive tool-profile allowlists unless the tool name, plugin id, `*`, or `group:plugins` is explicitly allowed.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party memory plugin's required `api.registerTool()` tools no longer reach the agent except for the built-in memory pair.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and regression tests around either explicit opt-in or a narrowly approved trusted memory-plugin/profile grant.

## Reproduction Assessment

Yes for the core mechanism: enable a plugin that declares/registers required tools such as `memory_search`, `memory_get`, and `memory_share`, set `tools.profile: "coding"`, and inspect resolved tool names. Source inspection shows non-built-in plugin tools are filtered unless explicitly allowed by name, plugin id, `*`, or `group:plugins`.

## Solution Assessment

Unclear because there is no proposed patch and the policy choice is security-sensitive. Blanket exposure of all required third-party tools under restrictive profiles is risky; explicit opt-in, better diagnostics/docs, or a maintainer-approved trusted memory-plugin contract would be safer.
