## Summary

Keep open. Current main still source-reproduces the reported partial plugin-tool exposure under `tools.profile: "coding"`: profile allowlists are threaded into plugin materialization, non-matching required plugin tools are skipped, and docs still describe required `api.registerTool()` tools as always available. The merged descriptor-cache work in #76079 shipped separately and does not settle this policy contract.

## What This Changes

The issue asks OpenClaw to expose all required tools registered by a third-party MemOS plugin via `api.registerTool()` after only `memory_search` and `memory_get` remain available on 4.26.

## Best Possible Solution

Define the supported required-plugin-tool contract under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, at source level: enable a plugin whose required `contracts.tools` include `memory_search`, `memory_get`, and additional required names under `tools.profile: "coding"`; only names matching the coding allowlist or an explicit plugin/tool allow entry are selected. I did not live-run the external MemOS plugin.

## Solution Assessment

Unclear because there is no proposed patch and the correct behavior needs maintainer policy. The narrow implementation would either keep non-optional plugin tools visible under profile filtering while still honoring deny lists, or explicitly document and diagnose that restrictive profiles require `full`, `alsoAllow`, `tools.allow`, or a plugin-id allow entry.
