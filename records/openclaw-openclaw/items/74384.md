## Summary

Keep open. Current main still has a credible core contract mismatch: third-party `registerTool()` tools are materialized, but the active `coding` tool profile can filter the final model-visible list down to the built-in memory pair unless the operator explicitly allowlists the plugin or tool names.

## What This Changes

The issue reports that third-party plugin tools registered with `api.registerTool()` are no longer exposed to the agent after upgrading to OpenClaw 4.26.

## Best Possible Solution

Define the contract between required plugin tools and restrictive profiles, then align code, docs, diagnostics, and regression tests. A durable fix should use an explicit, documented opt-in such as plugin-id/tool `alsoAllow`, or a maintainer-approved memory-plugin profile contract, rather than exposing all third-party required tools automatically.

## Reproduction Assessment

Yes for the core mechanism: register a synthetic enabled plugin with `memory_search`, `memory_get`, and another required tool, set `tools.profile: "coding"`, and inspect the final tool list after `applyToolPolicyPipeline`. The external MemOS install was not live-run.

## Solution Assessment

Unclear because there is no proposed patch. Exposing every required third-party tool through restrictive profiles would be too broad; a narrower explicit opt-in or documented memory-plugin profile rule is safer.
