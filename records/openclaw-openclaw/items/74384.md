## Summary

Keep open. Current main still source-reproduces the reported mechanism for local installs using the default `coding` profile: required third-party plugin tools whose names are outside the profile allowlist are not exposed unless the user switches to `full` or explicitly allows the plugin/tool, and the remaining choice is a tool-policy contract decision.

## What This Changes

The issue reports that a third-party plugin’s `api.registerTool()` tools are mostly hidden from the agent after upgrading to OpenClaw 4.26.

## Best Possible Solution

Define the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes. The current source path shows that a plugin declaring required `memory_search`, `memory_get`, and additional registered tools under `tools.profile: "coding"` will only expose names that match the coding allowlist or an explicit plugin/tool allow entry.

## Solution Assessment

Unclear. There is no proposed patch, and the narrow code change depends on whether maintainers want required third-party plugin tools to bypass restrictive profile semantics or require `full`/`alsoAllow` configuration.
