## Summary

Keep open. Current main still makes the reported symptom source-reproducible for restrictive profiles: plugin tools registered with `api.registerTool()` are resolved through profile-derived allowlists and then filtered again by the final tool policy, so the `coding` profile naturally leaves `memory_search` and `memory_get` while dropping MemOS-specific required tools unless the user switches to `full` or explicitly allows them.

## What This Changes

Expose all MemOS tools registered with `api.registerTool()` after the OpenClaw 4.26 upgrade, rather than only `memory_search` and `memory_get`.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. A required plugin tool whose name is outside the active restrictive profile remains filtered unless the profile is `full`, the plugin/tool is explicitly allowed, or policy semantics change.

## Solution Assessment

Unclear. `tools.profile: "full"` and explicit allow entries are available workarounds, but deciding whether required third-party tools should bypass restrictive profile semantics is a maintainer product/security decision.
