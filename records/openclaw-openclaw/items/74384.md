## Summary

Keep open. Current main still has a credible core contract/policy mismatch: non-optional `registerTool()` results can be materialized, but the final tool-policy pipeline applies `tools.profile: "coding"` as a restrictive allowlist that only includes the built-in `memory_search` and `memory_get` memory tools unless the operator explicitly widens the profile-stage allowlist. The related #73748 hook-access workaround addresses conversation hooks, not this model-visible tool filtering path.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party MemOS plugin's `api.registerTool()` tools beyond `memory_search` and `memory_get` are registered but not exposed to the agent.

## Best Possible Solution

Define the contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and regression tests. A maintainable end state should either document the explicit plugin/tool `alsoAllow` path for restrictive profiles or add a maintainer-approved, narrowly scoped profile-stage opt-in for memory-like plugins.

## Reproduction Assessment

Yes for the core mechanism: register a synthetic plugin with `memory_search`, `memory_get`, and one additional required tool, enable `tools.profile: "coding"`, and inspect the final assembled tool names after `applyToolPolicyPipeline`. The specific MemOS package was not executed during this read-only review.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Exposing all required third-party tools through restrictive profiles would be risky; an explicit documented opt-in or approved memory-plugin profile contract is safer.
