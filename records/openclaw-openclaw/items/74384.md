## Summary

Keep open. Current main still source-reproduces the reported third-party `api.registerTool()` exposure problem under restrictive tool profiles, and the remaining question is the supported contract for required plugin tools rather than cleanup closure.

## What This Changes

The issue reports that after upgrading to 4.26, a third-party MemOS plugin registers several tools with `api.registerTool()` but only `memory_search` and `memory_get` are exposed to the agent.

## Best Possible Solution

Define the required-plugin-tool contract under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level reproduction path: enable a plugin whose manifest declares `memory_search`, `memory_get`, and additional required registered tools under `tools.profile: "coding"`; current main selects and materializes only the names matching the coding allowlist. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the desired behavior is a maintainer contract decision. The narrow implementation should either exempt required plugin tools from profile-only filtering while still honoring deny lists, or explicitly document and diagnose that restrictive profiles require `full`, `alsoAllow`, or plugin-id allow entries for plugin-owned tools.
