## Summary

Keep open. Current main still has a source-level path where `tools.profile: "coding"` can select a plugin through `memory_search`/`memory_get` while filtering other required registered tools, and the plugin docs still describe required `api.registerTool()` tools as always available, so this needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to restore exposure of all required third-party plugin tools registered via `api.registerTool()` after the 4.26 plugin SDK and tool-policy changes.

## Best Possible Solution

Define the supported contract for required third-party plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage with that contract.

## Reproduction Assessment

Yes, as a source-level reproduction path: enable a plugin with required `memory_search`, `memory_get`, and additional required registered tools under `tools.profile: "coding"`; only names matching the coding allowlist or explicit plugin/tool grants survive. I did not run the external MemOS plugin live.

## Solution Assessment

Unclear because there is no proposed patch and the best behavior depends on maintainer policy. The narrow options are to exempt required plugin tools from profile-only filtering while preserving deny lists, or explicitly document and diagnose that restrictive profiles require `full` or explicit `alsoAllow` entries for plugin-owned tools.
