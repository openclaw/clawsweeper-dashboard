## Summary

Keep open. Current main still has a credible plugin SDK/tool-policy mismatch: restrictive tool profiles can prevent required third-party `registerTool()` entries from reaching the agent unless the plugin id, tool name, or `group:plugins` is explicitly allowed.

## What This Changes

The issue asks OpenClaw to restore or clarify exposure of third-party plugin tools registered with `api.registerTool()` after the 4.26 upgrade, where only `memory_search` and `memory_get` remain agent-visible for the reported memory plugin.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests around either explicit opt-in or a maintainer-approved trusted memory-plugin/profile grant.

## Reproduction Assessment

Yes for the core mechanism: enable a plugin that registers `memory_search`, `memory_get`, and another required tool, set `tools.profile: "coding"`, and inspect the resolved/final tool names. The extra plugin tool is filtered unless the plugin, tool, or `group:plugins` is explicitly allowed.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` is likely too broad; explicit opt-in, targeted diagnostics/docs, or a trusted memory-plugin contract is safer.
