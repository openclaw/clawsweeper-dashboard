## Summary

Keep open. Current main still supports `api.registerTool()`, but restrictive tool profiles can prevent required third-party plugin tools from reaching the agent unless the tool name, plugin id, or `group:plugins` is explicitly allowed; the docs still describe required plugin tools as always available.

## What This Changes

The issue asks OpenClaw to restore exposure of third-party plugin tools registered with `api.registerTool()` after the 4.26 upgrade.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and regression tests around either explicit opt-in or a maintainer-approved trusted plugin/profile grant.

## Reproduction Assessment

Yes for the core mechanism: enable a plugin that declares `memory_search`, `memory_get`, and another required tool, set `tools.profile: "coding"`, and inspect the resolved tool names. Source inspection shows the extra plugin tool is filtered unless the allowlist explicitly includes its name, the plugin id, or `group:plugins`.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. Blanket exposure of all required third-party tools under `coding` is likely too broad; explicit opt-in, targeted diagnostics/docs, or a trusted memory-plugin contract is safer.
