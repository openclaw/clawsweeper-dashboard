## Summary

Keep open. Current main still has a credible OpenClaw contract mismatch: required third-party plugin tools can be registered and materialized, then removed by the final restrictive `tools.profile: "coding"` policy before the model sees them. The linked #73748 workaround addresses conversation hook access, not this profile-stage tool exposure behavior.

## What This Changes

The issue asks OpenClaw to restore visibility of external MemOS plugin tools registered with `api.registerTool()` after OpenClaw 4.26, where only `memory_search` and `memory_get` remain visible to the agent.

## Best Possible Solution

Define the supported contract between required plugin tools and restrictive tool profiles, then align code, diagnostics, docs, and regression tests. A durable outcome should either require an explicit profile-stage opt-in such as plugin-id/tool `alsoAllow`, or introduce a maintainer-approved memory-plugin/profile contract that does not silently widen third-party write/share/publish capabilities.

## Reproduction Assessment

Yes for the core mechanism: register an enabled synthetic plugin with required `memory_search`, `memory_get`, and one extra required tool, set `tools.profile: "coding"`, and inspect the final tool names after `applyToolPolicyPipeline`; the extra plugin tool is filtered by the profile allowlist. The exact external MemOS plugin was not live-run.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is security/product policy. Exposing all required third-party tools through `coding` is not obviously the safest fix; explicit opt-in plus clearer diagnostics/docs may be the narrower maintainable direction.
