## Summary

Keep open. Current main still makes the reported symptom source-reproducible: required third-party plugin tools registered with `api.registerTool()` are resolved and then filtered through restrictive profile allowlists, so a `coding` profile can leave only names that overlap built-in memory tools such as `memory_search` and `memory_get`.

## What This Changes

Expose all required tools registered by a third-party plugin via `api.registerTool()` after the 4.26 plugin SDK changes, not only `memory_search` and `memory_get`.

## Best Possible Solution

Define the supported contract for required plugin tools under restrictive profiles, then align resolver behavior, diagnostics, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes. The current source path reproduces the mechanism: a plugin with required `memory_search`, `memory_get`, and additional registered tools under `tools.profile: "coding"` will only keep entries that match the coding allowlist or an explicit plugin allow entry.

## Solution Assessment

Unclear because there is no proposed patch and the remaining choice is policy-sensitive. The `full` profile wildcard is a useful workaround, but it does not settle whether required third-party tools should bypass restrictive profile semantics.
