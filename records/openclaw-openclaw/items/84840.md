## Summary

Keep open for review: the PR targets a real Mistral prompt-cache compat gap, but the shared patch now opts OpenRouter and hinted custom routes into `prompt_cache_key` without proof that those non-native routes accept that field. The supplied proof is targeted Vitest output, so this also needs real transport proof or maintainer override before merge.

## What This Changes

The PR adds `supportsPromptCacheKey: true` to the Mistral transport compat patch and extends Mistral compat tests for native, provider-family, and custom-route resolution.

## Best Possible Solution

Keep the Mistral-native prompt-cache opt-in, but split or gate it so only `api.mistral.ai` and other documented routes advertise `supportsPromptCacheKey`, while broader Mistral-route compat keeps the existing request-shape fixes.

## Reproduction Assessment

Yes from source inspection, though not from a live provider run: current main's Mistral patch lacks `supportsPromptCacheKey`, and the completions transport only injects `prompt_cache_key` when that compat flag is true.

## Solution Assessment

No as written: adding the flag to the shared Mistral patch fixes the native gap but also opts unproven OpenRouter/custom routes into the field; a route-scoped compat split is the narrower maintainable fix.
