## Summary

Keep this PR open: it targets a real Mistral prompt-cache compatibility gap, but the latest diff still opts `provider: "mistral"` custom/proxy base URLs into `prompt_cache_key`, so existing proxy setups can receive a new unproven request field.

## What This Changes

The PR adds `supportsPromptCacheKey` to Mistral transport compat, makes hinted routes opt out, and extends the Mistral compat tests.

## Best Possible Solution

Enable `supportsPromptCacheKey` for default/native Mistral API routes, keep custom/proxy or model-hinted routes opted out unless explicitly configured, and prove the corrected route matrix with redacted transport output.

## Reproduction Assessment

Yes from source inspection: current main lacks `supportsPromptCacheKey` in the Mistral compat patch, and the completions transport only injects `prompt_cache_key` when that flag is true. The remaining PR bug is also source-reproducible because `knownProviderFamily === "mistral"` still wins for custom base URLs.

## Solution Assessment

No as written: adding the flag is the right native-Mistral direction, but treating provider-family custom base URLs as direct is not the safest compatibility boundary. The narrower fix is native/default Mistral only, with hinted or proxy routes staying opted out unless explicitly configured.
