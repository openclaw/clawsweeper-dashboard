## Summary

Keep this open. Current main still has direct file-backed `node:sqlite` use across the reported memory path and adjacent core stores, with no merged driver selection, lock probe, or fallback implementation for the ARM64 container `SQLITE_BUSY` failure mode.

## What This Changes

Request to make OpenClaw's file-backed SQLite storage configurable or resilient when `node:sqlite` fails with `SQLITE_BUSY` on ARM64 container hosts.

## Best Possible Solution

Add a shared SQLite storage seam or runtime file-lock probe used by memory, task registry, Task Flow, plugin state, and proxy capture, with a documented fallback policy and focused regression coverage for first-statement `SQLITE_BUSY` failures.

## Reproduction Assessment

Unclear. The reporter provided a concrete ARM64 container one-liner, and current main still runs the same file-backed `DatabaseSync` paths, but this review did not have that ARM64/Kubernetes environment for live reproduction.

## Solution Assessment

Unclear. A memory-only `better-sqlite3` fallback would miss other affected file-backed stores; the more maintainable direction is a shared driver/probe policy before adding a native dependency or in-memory persistence fallback.
