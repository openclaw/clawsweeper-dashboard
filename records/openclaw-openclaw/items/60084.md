## Summary

Keep open. Current main still routes the affected durable stores through file-backed `node:sqlite` `DatabaseSync(path)` with no driver selector, lock-health probe, or supported fallback, and the live issue has no linked closing pull request or release evidence.

## What This Changes

The issue requests a supported fallback or configurable SQLite driver for ARM64 container environments where file-backed `node:sqlite` fails with `SQLITE_BUSY`.

## Best Possible Solution

Add a shared, documented SQLite storage selection and health-probe layer for durable stores, with maintainer-approved fallback semantics and ARM64 container validation.

## Reproduction Assessment

Yes, the issue provides a concrete ARM64 container command that exercises file-backed `node:sqlite`, and current main still uses that same dependency path for durable stores. I did not reproduce it on the review host because this host is Linux x64, where the minimal file-backed command succeeds.

## Solution Assessment

Unclear. A lock-health probe plus a supported fallback/storage policy looks safer than a direct `better-sqlite3` file-backed swap, because the report says file-backed `better-sqlite3` also hits the same lock failure.
