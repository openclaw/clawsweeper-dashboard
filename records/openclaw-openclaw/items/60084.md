## Summary

Keep open. Current main still routes the affected durable stores through file-backed `node:sqlite` `DatabaseSync(path)` with no driver selector, lock probe, or supported fallback, so the reported ARM64 container `SQLITE_BUSY` failure is not resolved.

## What This Changes

Request support for ARM64 container environments where file-backed SQLite operations fail by adding a supported SQLite fallback, driver selection, or lock-probe path.

## Best Possible Solution

Add a shared, documented SQLite storage selection and health-probe layer that handles persistent file-backed lock failure across durable stores with maintainer-approved fallback semantics and ARM64 container validation.

## Reproduction Assessment

Yes: run the issue's minimal file-backed `node:sqlite` `DatabaseSync('/tmp/test.sqlite')` repro inside an ARM64 Kubernetes/container environment matching the report. I did not live-reproduce it here, but current main still sends the affected durable stores through the same file-backed `DatabaseSync(path)` path.

## Solution Assessment

Unclear. A lock probe plus a supported fallback/configuration policy looks safer than a naive `better-sqlite3` file-backed driver swap, because the report says file-backed `better-sqlite3` also hit the same lock failure.
