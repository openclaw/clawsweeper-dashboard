## Summary

Keep open. Current main still opens file-backed `node:sqlite` databases directly for memory-core, QMD, task and Task Flow registries, plugin state, and proxy capture with no driver selection, lock probe, or durable fallback; adjacent open issues cover missing `node:sqlite` and memory file-search fallback, but not this ARM64/container `SQLITE_BUSY` locking failure.

## What This Changes

The issue asks OpenClaw to survive ARM64 container SQLite file-lock failures by making file-backed SQLite storage configurable or adding a fallback for memory, indexing, task registry, and related SQLite-backed stores.

## Best Possible Solution

Add a maintainer-approved SQLite storage selection and health-probe layer that detects persistent file-backed `SQLITE_BUSY` on startup, routes affected durable stores to a supported fallback, and documents the ARM64/container behavior while preserving permissions, WAL cleanup, and migration semantics.

## Reproduction Assessment

Yes, the issue gives a minimal file-backed `node:sqlite` repro inside an ARM64 Kubernetes container, while `:memory:` succeeds. This review could not run that architecture, but current source still reaches the same `new DatabaseSync(path)` plus first-`exec` paths across the reported OpenClaw surfaces.

## Solution Assessment

Unclear. A naive `better-sqlite3` file-backed fallback is not the best proven fix because the report says that path also hits the same lock failure; the safer direction is a lock probe plus explicit supported storage fallback or configuration chosen by maintainers.
