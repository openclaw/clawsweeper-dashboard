## Summary

Keep open. Current main still routes memory, QMD, task registry, plugin state, and proxy capture storage through file-backed `node:sqlite` without a supported backend selector, lock-health probe, or fallback, and live GitHub state shows no closing PR or superseding canonical item.

## What This Changes

Request a configurable or fallback SQLite storage layer because file-backed `node:sqlite` reports `SQLITE_BUSY` on ARM64 Kubernetes/container hosts.

## Best Possible Solution

Add a shared, documented durable-SQLite selection and lock-health layer that covers all file-backed stores, with maintainer-approved fallback semantics and ARM64 container proof.

## Reproduction Assessment

Yes for the affected environment: the reporter provides exact ARM64 container `node:sqlite` commands, and current source still opens the same file-backed `DatabaseSync(path)` stores. I did not live-reproduce it on an ARM64 Kubernetes host during this read-only review.

## Solution Assessment

Unclear. A shared lock-health probe plus maintainer-approved fallback/storage policy is safer than directly swapping to `better-sqlite3`, because the report says file-backed `better-sqlite3` hit the same lock failure.
