## Summary

Keep open. Current main still routes memory, QMD, task registry, Task Flow registry, plugin state, and proxy capture storage through file-backed `node:sqlite` without driver selection, lock probing, or a durable fallback for the ARM64 container `SQLITE_BUSY` failure described here.

## What This Changes

The issue reports ARM64 container file-backed SQLite failures with `SQLITE_BUSY` and asks for a configurable or fallback SQLite storage path.

## Best Possible Solution

Introduce a shared, maintainer-approved SQLite storage selection and health-probe layer that detects persistent file-backed lock failure, routes durable stores to a supported fallback or explicit configuration, and documents ARM64 container validation without weakening WAL, permissions, or migration semantics.

## Reproduction Assessment

Yes, for the affected ARM64 container runtime. The report gives a minimal file-backed `node:sqlite` repro and current main still exercises the same `new DatabaseSync(path)` plus first-`exec`/query path across the reported stores.

## Solution Assessment

Unclear. A naive `better-sqlite3` file-backed fallback is not proven because the report says it also hit the same lock failure; a lock probe plus an explicit supported fallback/configuration policy is the safer direction.
