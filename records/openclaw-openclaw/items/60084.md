## Summary

Keep open. Current main still routes memory, QMD, task registry, Task Flow registry, plugin state, and proxy capture storage through file-backed `node:sqlite` without a driver selector, lock probe, or supported fallback for the ARM64 container `SQLITE_BUSY` failure described here.

## What This Changes

The issue asks OpenClaw to handle ARM64 container environments where file-backed SQLite operations fail with `SQLITE_BUSY`, through a fallback driver, configuration, or runtime lock probe.

## Best Possible Solution

Add a shared, documented SQLite storage selection and health-probe layer that safely handles persistent file-backed lock failure across durable stores, with maintainer-approved fallback semantics and ARM64 container validation.

## Reproduction Assessment

Yes for the reporter's ARM64 container setup: the issue gives a minimal file-backed `node:sqlite` repro, and current main still sends the affected durable stores through `DatabaseSync(path)` plus immediate SQL statements. I did not reproduce it live because this checkout is not the reported ARM64 Kubernetes/container environment.

## Solution Assessment

Unclear. The requested direction is reasonable, but the safest fix is not a naive `better-sqlite3` file-backed fallback because the report says that also hit the lock failure; a lock probe plus an explicit supported fallback/configuration policy is the maintainable direction.
