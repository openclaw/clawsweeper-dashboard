## Summary

Keep open because current main and the latest release still depend on file-backed `node:sqlite` for the affected stores, with no driver selection, lock probe, or durable fallback for the reported ARM64 container failure.

## What This Changes

Request a configurable or fallback SQLite storage path for ARM64 container environments where file-backed SQLite operations return `SQLITE_BUSY`.

## Best Possible Solution

Introduce a shared, maintainer-approved SQLite storage selection and health-probe layer that handles persistent file-backed lock failure consistently across memory, tasks, plugin state, and proxy capture, with ARM64 container validation and documentation.

## Reproduction Assessment

Yes, on an affected ARM64 container runtime: the issue gives a minimal file-backed `node:sqlite` command that should fail before OpenClaw-specific code is involved. Current main still exercises the same `new DatabaseSync(path)` plus first-SQL-statement path across the affected stores.

## Solution Assessment

Unclear. A naive `better-sqlite3` file-backed fallback is not proven because the reporter says it also failed; the safer direction is a shared lock probe plus an explicit maintainer-approved backend/configuration policy.
