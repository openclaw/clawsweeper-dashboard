## Summary

Keep open. Current main and the latest release still use file-backed `node:sqlite` directly across the reported memory and registry surfaces, and no related issue or PR currently supersedes this ARM64/container `SQLITE_BUSY` locking failure.

## What This Changes

Request configurable SQLite storage or a durable fallback for ARM64 container file-backed SQLite `SQLITE_BUSY` failures affecting memory search and registry stores.

## Best Possible Solution

Add a maintainer-approved SQLite health-probe and storage fallback strategy that detects persistent file-backed lock failures, preserves durable store semantics, and documents the supported ARM64/container behavior.

## Reproduction Assessment

Yes. The issue gives a concrete ARM64 container repro where file-backed `node:sqlite` fails and `:memory:` works, and current main still reaches the same `new DatabaseSync(path)` plus first-`exec` paths across the affected stores.

## Solution Assessment

Unclear. The requested direction is valid, but a plain `better-sqlite3` file-backed fallback is not proven because the report says it also fails; the safer solution is a maintainer-chosen lock probe plus supported durable fallback or explicit storage selection.
