## Summary

Keep open. Current main still routes memory and task-registry SQLite through file-backed `node:sqlite` `DatabaseSync` handles with no alternate driver, lock probe, or configured fallback, so the ARM64/container `SQLITE_BUSY` report maps to a real current code path.

## What This Changes

Request a configurable or fallback SQLite driver/locking probe so ARM64 container file-backed SQLite failures do not break memory search and task registries.

## Best Possible Solution

Add a shared SQLite storage seam and runtime lock probe used by memory and task registries, then choose a maintainer-approved fallback or diagnostic/degradation path that preserves durable state without hardcoding a new dependency blindly.

## Reproduction Assessment

Yes, the issue provides a focused single-process `node:sqlite` command that should reproduce on the affected ARM64/Kubernetes host. This review verified OpenClaw uses the same file-backed `DatabaseSync` path, but did not run live ARM64 container proof.

## Solution Assessment

Unclear. A `better-sqlite3` fallback is plausible, but the best maintainable fix should cover memory, task registry, and task-flow SQLite through one generic seam/probe and needs maintainer choice on native dependency policy.
