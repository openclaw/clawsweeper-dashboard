## Summary

Keep open. Current main still routes memory, QMD, task registry, plugin state, and proxy capture storage through file-backed `node:sqlite` without a supported backend selector, lock-health probe, or fallback, and the live issue has no linked closing PR or superseding canonical item.

## What This Changes

The issue requests a supported SQLite fallback or configurable driver path for ARM64 container environments where file-backed SQLite operations fail with `SQLITE_BUSY`.

## Best Possible Solution

Add a shared, documented SQLite storage selection and health-probe layer for durable stores, with maintainer-approved fallback semantics and ARM64 container validation.

## Reproduction Assessment

Yes for the affected environment: the issue gives an exact ARM64 container `node:sqlite` file-backed repro, and current main still uses the same file-backed `DatabaseSync(path)` dependency path for durable stores. I did not live-reproduce it on this x64 review host.

## Solution Assessment

Unclear. A shared lock-health probe plus maintainer-approved fallback/storage policy is safer than directly swapping to `better-sqlite3`, because the report says file-backed `better-sqlite3` also hit the same lock failure.
