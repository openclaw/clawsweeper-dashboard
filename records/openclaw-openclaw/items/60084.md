## Summary

Keep open. Current main still routes memory, QMD, task registry, plugin state, and proxy capture storage through file-backed `node:sqlite` without a supported backend selector, lock-health probe, or fallback, and there is no linked closing PR or superseding canonical item.

## What This Changes

The issue requests a configurable SQLite driver, runtime lock probe, or fallback for ARM64 container environments where file-backed SQLite operations fail with `SQLITE_BUSY`.

## Best Possible Solution

Add a shared, documented SQLite storage selection and lock-health layer for durable stores, with maintainer-approved fallback semantics and ARM64 container validation.

## Reproduction Assessment

Yes for the affected environment: the issue gives exact ARM64 container commands, and current main still uses the same file-backed `DatabaseSync(path)` dependency path for durable stores. I did not live-reproduce it on an ARM64 Kubernetes/container host during this read-only review.

## Solution Assessment

Unclear. A shared lock-health probe plus maintainer-approved fallback/storage policy is safer than directly swapping to `better-sqlite3`, because the report says file-backed `better-sqlite3` also hit the same lock failure.
