## Summary

Keep open. Current main has partial restart-continuation and durable task-ledger pieces, but the requested general resume/report flow for arbitrary reboot or process-killing work is still a new core feature; background exec/process state is still documented as memory-only across restarts.

## What This Changes

Request durable task context plus automatic post-restart verification and reporting when reboots or OpenClaw service restarts interrupt automation.

## Best Possible Solution

Define a core-owned continuation contract that reuses the restart sentinel, task ledger, and Task Flow where appropriate, with explicit rules for expected-outcome verification and user notification after restart.

## Reproduction Assessment

Not applicable for a feature request. Source and docs show partial restart/task persistence surfaces, but there is no existing contract that arbitrary reboot or process-killing commands must proactively verify and report after restart.

## Solution Assessment

Unclear. The proposed local durable pending-state idea fits existing sentinels and task stores, but maintainers should first decide the product boundary and security model for automatic post-restart execution and notification.
