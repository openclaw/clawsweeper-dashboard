## Summary

Keep open. Current main and v2026.4.27 already ship an important gateway-restart continuation path, including durable restart sentinels, routed post-restart notices, and one-shot agent continuations. The remaining request is broader: generic task checkpointing before host reboots or arbitrary process-killing commands, expected-outcome verification after boot, and proactive final delivery for normal interactive work still need explicit product and security design.

## What This Changes

The issue requests durable task state, expected-result checks, and proactive user reporting after OpenClaw restarts, service restarts, or host reboots interrupt an automation task.

## Best Possible Solution

Extend the shipped restart sentinel, session-delivery queue, Task Flow, and task ledger into a first-class durable continuation contract for authorized restart-causing workflows: persist route, expected checks, restart/reboot kind, notification policy, and continuation authority, then run bounded verification after startup and deliver a final result to the original channel.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is code and docs inspection: gateway restarts have a durable continuation path, but normal interactive chat and arbitrary host reboot/process-kill workflows are not generally task-backed checkpoint flows.

## Solution Assessment

No, the shipped gateway restart continuation is a strong partial solution but not the full requested generic mechanism. The safer direction is to extend the existing sentinel/task/flow infrastructure with explicit authorization and verification semantics rather than add an unrelated persistence database.
