## Summary

Keep open. Current main has important restart-continuation pieces, but the remaining request is broader than the shipped/current gateway restart path: generic checkpointing, expected-outcome verification, and proactive reporting after host reboots or arbitrary process-killing commands still need maintainer product and security design.

## What This Changes

Persist task context before restart-causing work and automatically verify and report the result after OpenClaw comes back online.

## Best Possible Solution

Extend the existing restart sentinel, session-delivery queue, task ledger, and Task Flow infrastructure into an authorized durable continuation contract that records route, restart kind, expected checks, notification policy, and bounded verification semantics.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection show current main has partial gateway restart/update continuations and bounded interrupted-session recovery, but not a generic task checkpoint and verification mechanism for all restart-causing work.

## Solution Assessment

No. The existing restart sentinel and delivery queue are the right foundation, but the full request should be solved by extending that contract with explicit authorization and verification semantics rather than adding an unrelated persistence store.
