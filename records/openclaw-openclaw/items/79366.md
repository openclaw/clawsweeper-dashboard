## Summary

Keep open: current main still has no pending MCP tool tracking in the Claude live-session path. The parser records raw lines and resolves on `result`, while the watchdog abort path logs a generic no-output timeout without naming any in-flight tool; related open items cover resume recovery and trajectory parity, but not this specific post-mortem diagnostic.

## What This Changes

Track Claude CLI stream-json MCP tool-use starts/results in the live session and emit diagnostic logs for long-pending tools before or when the no-output watchdog kills the turn.

## Best Possible Solution

Add per-turn pending-tool state in `src/agents/cli-runner/claude-live-session.ts`, clear it on result/session close, include a bounded pending-tool summary in watchdog/abort logs, and cover it with focused stream-json tests.

## Reproduction Assessment

Yes, for the missing diagnostic on current main: source inspection shows no pending-tool map or tool-use lifecycle handling in `src/agents/cli-runner/claude-live-session.ts`, and the no-output timer aborts with a generic timeout message. I did not run a live hanging MCP tool.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it only records already-present stream-json metadata and stays diagnostic-only. The implementation should verify the current Claude event shape and redact or omit sensitive tool arguments.
