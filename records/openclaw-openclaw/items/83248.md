## Summary

Keep open: this is a focused Telegram bug-fix PR and the source path supports the reported debounce bypass problem, but the external PR only provides synthetic Vitest proof while Telegram behavior changes require real Telegram proof before merge.

## What This Changes

The PR adds generic inbound debounce cancellation and uses it in Telegram ingress so `stop` and targeted `/stop` cancel same-key pending buffers and bypass inbound debounce, with focused Telegram regression tests.

## Best Possible Solution

Land the narrow debounce-cancellation fix after real Telegram proof shows `stop` and bot-addressed `/stop` execute immediately and the earlier same-chat buffered prompt does not run.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main path: a debounced Telegram message creates a same-key buffer, `/stop` is non-debounceable but waits for `flushKey`, and plain `stop` is an abort trigger that the current debounce policy does not exclude.

## Solution Assessment

Yes, the proposed code shape is narrow and maintainable: add explicit generic debounce cancellation, then use the existing abort primitive in Telegram ingress to cancel only same-key pending state before running the stop control.
