## Summary

Keep open. The PR targets a real Telegram debounce bug, but the current patch cancels pending buffers before `/stop` or bare abort text has passed command authorization, and the posted proof is only a middleware harness rather than real Telegram behavior.

## What This Changes

The branch adds `cancelKey` to the generic inbound debouncer, uses Telegram ingress to cancel same-chat default and forwarded debounce buffers for stop-like messages, bypasses debounce for the abort turn, and adds focused Telegram middleware tests.

## Best Possible Solution

Move the buffer cancellation behind the same authorization decision used by `/stop` and bare abort handlers, then land with focused regression coverage plus redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: Telegram same-key inbound text enters the debounce buffer, while stop-like text is handled later as a command/abort turn.

## Solution Assessment

No as written. The cancellation hook is the right general shape, but it needs to be authorization-safe before it discards buffered user input, and it still needs real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Defer cancellation until stop is authorized:** `extensions/telegram/src/bot-handlers.runtime.ts:1757-1768`
  - body: This cancels the pending Telegram debounce lanes for any text that `isAbortRequestText` recognizes, before `/stop` or bare abort text reaches command authorization. In configs where a sender may chat but is not in `commands.allowFrom`, an unauthorized `/stop` would still delete that sender's pending debounced prompt even though the command handler later rejects the abort; keep the pending buffer unless the abort is actually authorized.
  - confidence: 0.87
