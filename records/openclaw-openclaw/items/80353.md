## Summary

Keep open. The discussion narrowed the report to prompt assembly of `openclaw.runtime-context`, and current `main` still has source-level evidence that Discord untrusted context can be wrapped into hidden runtime custom messages while the LLM-boundary normalizer keeps current post-user runtime-context custom messages; that contract needs maintainer/security judgment rather than cleanup closure.

## What This Changes

The issue requests that Discord runtime-context records containing external-untrusted envelope markers not surface as raw agent-visible prompt input or duplicated inbound message content.

## Best Possible Solution

Define a single prompt-assembly contract for channel untrusted context: preserve the security boundary, avoid duplicate raw text in model input, add regression coverage for Discord `UntrustedContext` plus session `custom_message` ordering, and document whether wrappers are model-facing or internal-only.

## Reproduction Assessment

No high-confidence live reproduction. The reporter supplied Discord message IDs and a JSONL byte-walk, and current source makes the path plausible, but I did not inspect that session file or run a live Discord channel session.

## Solution Assessment

Unclear. Stripping all runtime-context custom messages at the LLM boundary is too blunt because it may drop intentional untrusted context; a safer direction is an explicit contract that moves current-turn runtime context into typed or system context while keeping transcripts and UI hidden.
