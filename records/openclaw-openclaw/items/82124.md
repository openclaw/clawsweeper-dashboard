## Summary

Keep open: current main and latest release still lack a public `config unset` size-drop override, while the lower config writer already has the narrow primitive the issue wants to expose. The remaining blocker is maintainer product/security judgment about adding a public destructive-write CLI flag and its exact name/scope.

## What This Changes

Request to add a narrow opt-in flag to `openclaw config unset` so intentional large deletes can bypass only the size-drop guard.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads `allowConfigSizeDrop` only after the target path exists, keeps `unsetPaths`, schema validation, rejected artifacts, and hard guards such as `gateway-mode-removed`, and covers Commander, fast-route, CLI tests, writer-boundary tests, and docs.

## Reproduction Assessment

Yes, source-level: current `config unset` can delete an existing path and record `unsetPaths`, but neither the Commander command nor the fast route parser can pass `allowConfigSizeDrop`, while the writer blocks size drops unless that option is true. I did not run the destructive-write repro because this review is read-only.

## Solution Assessment

Mostly yes technically, pending maintainer approval: reusing the existing writer primitive is narrower than a broad `--force`, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
