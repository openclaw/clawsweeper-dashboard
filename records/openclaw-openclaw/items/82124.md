## Summary

Keep open. Current main and v2026.6.6 still lack a public `config unset` flag for intentionally allowing large size drops, while the lower config writer already has the narrower `allowConfigSizeDrop` primitive; this remains a valid feature request that needs maintainer CLI/safety-policy approval before implementation.

## What This Changes

The issue requests a narrow `openclaw config unset` opt-in flag to allow intentional large config deletions to bypass only the size-drop guard while preserving other config write protections.

## Best Possible Solution

Approve the public flag name and policy, then implement a narrow `config unset` flag that sets `allowConfigSizeDrop` only for explicit successful unsets while preserving missing-path behavior, schema validation, `unsetPaths`, and gateway-mode blocking.

## Reproduction Assessment

Yes, by source inspection rather than a live run in this read-only review. `config unset` can remove an existing path but never passes `allowConfigSizeDrop`, while the writer rejects `size-drop:*` unless that option is true.

## Solution Assessment

Unclear pending maintainer product decision. The narrow flag is the best apparent implementation shape because it reuses the existing lower-layer guard option, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
