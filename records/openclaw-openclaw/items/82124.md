## Summary

Keep open. Current main and the latest release still do not expose a public `config unset` size-drop override, while the underlying config writer already has the narrow `allowConfigSizeDrop` primitive; the remaining decision is whether maintainers want this destructive-write affordance on the public CLI.

## What This Changes

The issue requests a narrow `openclaw config unset` opt-in flag that allows intentional large config deletions to bypass only the size-drop write guard.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads `allowConfigSizeDrop` only for existing-path unsets, while preserving schema validation, rejected artifacts, `unsetPaths`, and hard guards such as `gateway-mode-removed`.

## Reproduction Assessment

Yes, source-level. Current `config unset` deletes an existing path and records `unsetPaths`, but neither the command options nor the route parser can pass `allowConfigSizeDrop`, while the writer blocks size drops unless that option is true.

## Solution Assessment

Unclear pending maintainer product decision. The narrow flag is the best apparent implementation shape because it reuses the existing writer primitive, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
