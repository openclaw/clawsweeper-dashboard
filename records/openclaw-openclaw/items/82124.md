## Summary

Keep open. Current main and the latest release still lack a public `config unset` size-drop override, while the lower config writer already has the narrow `allowConfigSizeDrop` primitive; this remains a valid feature request that needs maintainer CLI/safety-policy judgment.

## What This Changes

Request an explicit `openclaw config unset` flag that lets intentional large deletions bypass only the size-drop guard while preserving other config-write protections.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads the existing `allowConfigSizeDrop` write option only for explicit unsets, with focused tests for default rejection, flagged acceptance, missing-path behavior, schema validation, and gateway-mode protection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `config unset` can remove an existing path but never passes `allowConfigSizeDrop`, while the writer rejects `size-drop:*` unless that option is true.

## Solution Assessment

Unclear pending maintainer product decision. The narrow flag is the best apparent implementation shape because it reuses the existing lower-layer guard option, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
