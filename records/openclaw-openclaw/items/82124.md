## Summary

Keep open: current main and the latest release still do not expose a `config unset` size-drop opt-in, while the shared writer already has the narrow primitive the issue wants to reuse. The remaining blocker is maintainer approval of a public destructive-write CLI flag and its exact safety wording.

## What This Changes

The issue requests a public `openclaw config unset` opt-in that bypasses only size-drop rejection for intentional large deletes.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads `allowConfigSizeDrop` only after the target path exists, keeps `unsetPaths`, schema validation, rejected artifacts, and hard guards such as `gateway-mode-removed`, and updates Commander, fast-route, CLI tests, writer-boundary tests, and docs.

## Reproduction Assessment

Yes, source-level. Current `config unset` can delete an existing path and record `unsetPaths`, but neither the command options nor the route parser can pass `allowConfigSizeDrop`, while the writer blocks `size-drop:*` unless that option is true.

## Solution Assessment

Mostly yes technically, pending maintainer approval. Reusing the existing writer primitive is narrower than a broad `--force`, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
