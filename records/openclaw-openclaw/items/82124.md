## Summary

Keep open. Current main still has the reported product gap: `config unset` exposes only a path argument and does not forward the existing narrow `allowConfigSizeDrop` write option, so this is a valid feature request that needs maintainer decision on the public CLI affordance.

## What This Changes

Request adding a public `openclaw config unset` opt-in flag to bypass only the `size-drop:*` config write guard for intentional large deletions.

## Best Possible Solution

Add a narrow `config unset` opt-in flag that sets `allowConfigSizeDrop` only for intentional large deletes while preserving missing-path behavior, schema validation, `unsetPaths`, rejected artifacts, and hard destructive guards.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: `config unset` can remove a large node but never passes `allowConfigSizeDrop`, while the write guard rejects drops below half the original byte size unless that internal option is set.

## Solution Assessment

Unclear pending maintainer product decision. The proposed narrow flag matches the existing internal write option, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
