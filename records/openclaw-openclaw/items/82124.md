## Summary

Keep open: current `main` still lacks a public `openclaw config unset` flag for intentional size-drop deletes. The lower-level writer already has the narrow `allowConfigSizeDrop` primitive, and related issues/PRs cover different size-drop cases, so this remains a valid but maintainer-owned CLI/product decision.

## What This Changes

The issue requests a narrow `openclaw config unset` opt-in that bypasses only the `size-drop:*` config write guard for intentional large config deletions.

## Best Possible Solution

Keep the default size-drop rejection, and if maintainers approve the affordance, expose a narrow unset-only flag that passes `allowConfigSizeDrop: true` while preserving missing-path handling, schema validation, `unsetPaths`, and hard guards such as `gateway-mode-removed`.

## Reproduction Assessment

Yes at source level. Current `config unset` writes through `replaceConfigFile` with `unsetPaths`, while the shared writer rejects size drops unless `allowConfigSizeDrop` is supplied, and the public unset command has no such option.

## Solution Assessment

Unclear until maintainers choose the public UX. The narrow writer option already exists and is the right lower-level mechanism if this CLI feature is accepted, but adding a new destructive-write flag is a product decision.
