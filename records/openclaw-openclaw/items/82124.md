## Summary

Keep open: current main and the latest release still do not expose a `config unset` opt-in for intentional large size drops, while the lower config writer already has the narrow `allowConfigSizeDrop` primitive. The remaining blocker is maintainer product/security approval for a public destructive-write CLI affordance and its exact semantics.

## What This Changes

Add an explicit `openclaw config unset` flag that bypasses only the `size-drop:*` guard for intentional large deletes while preserving other config write protections.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads `allowConfigSizeDrop` only after the target path exists, keeps `unsetPaths`, schema validation, rejected artifacts, and hard guards such as `gateway-mode-removed`, and covers Commander, fast-route, CLI tests, writer-boundary tests, and docs.

## Reproduction Assessment

Yes, source-level: current `config unset` can delete an existing path and record `unsetPaths`, but neither the command options nor the route parser can pass `allowConfigSizeDrop`, while the writer blocks size drops unless that option is true. I did not run the destructive-write repro because this review is read-only.

## Solution Assessment

Yes technically, pending maintainer policy: reusing the existing writer primitive is narrower than a broad force path, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
