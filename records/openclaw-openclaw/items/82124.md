## Summary

Keep open. Current main and the latest release still expose `config unset` only with dry-run related flags, while the lower config writer already has the narrow `allowConfigSizeDrop` primitive; the remaining blocker is maintainer approval for adding this public destructive-write affordance.

## What This Changes

The issue requests a narrow `openclaw config unset` opt-in flag for intentional large deletes that bypasses only the `size-drop:*` guard.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads `allowConfigSizeDrop` only after the target path exists, while preserving schema validation, rejected artifacts, `unsetPaths`, and hard guards such as `gateway-mode-removed`.

## Reproduction Assessment

Yes, source-level. Current `config unset` can delete an existing path and record `unsetPaths`, but neither the command options nor the route parser can pass `allowConfigSizeDrop`, while the writer blocks size drops unless that option is true.

## Solution Assessment

Unclear pending maintainer product decision. The narrow flag is the best apparent implementation shape because it reuses the existing writer primitive, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
