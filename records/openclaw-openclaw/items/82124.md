## Summary

Keep open: current main and v2026.6.10 still do not expose a `config unset` flag that forwards the existing size-drop bypass, so the requested public CLI affordance remains unresolved and needs maintainer product/security approval.

## What This Changes

The issue requests a narrow public `openclaw config unset` opt-in that allows intentional large deletes to bypass only the size-drop write guard.

## Best Possible Solution

Approve and add a narrow `config unset` flag that forwards `allowConfigSizeDrop` only after the target path exists, while keeping `unsetPaths`, schema validation, rejected artifacts, and hard guards such as `gateway-mode-removed`.

## Reproduction Assessment

Yes, source-level: current `config unset` records deletion intent through `unsetPaths`, but neither the command options nor fast-route parser can pass `allowConfigSizeDrop`, while the writer still blocks size drops unless that option is true.

## Solution Assessment

Mostly yes technically, pending maintainer approval: reusing the existing writer primitive is narrower than a broad force path, but maintainers should choose the public flag name and confirm this safety affordance belongs on `config unset`.
