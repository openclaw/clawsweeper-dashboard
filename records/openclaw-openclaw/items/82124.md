## Summary

Keep open. Current main still has the reported CLI gap: `config unset` accepts only a path and never forwards the existing narrow `allowConfigSizeDrop` write option, so this remains a valid feature request that needs maintainer CLI-policy judgment.

## What This Changes

The issue requests a narrow `config unset` flag for intentional large deletes that bypasses only the size-drop guard.

## Best Possible Solution

Expose a narrow `config unset` opt-in that forwards `allowConfigSizeDrop: true` only for intentional deletes while preserving `unsetPaths`, missing-path behavior, schema validation, rejected artifacts, and hard destructive guards.

## Reproduction Assessment

Yes, source-level. `config unset` can remove a large node but never passes `allowConfigSizeDrop`, while the shared writer rejects drops below half the original byte size unless that internal option is set.

## Solution Assessment

Unclear pending maintainer product decision. The proposed narrow flag matches the existing lower-layer option, but maintainers should choose the public flag name and confirm this destructive-write affordance belongs on the CLI.
