## Summary

Keep open: current main and v2026.6.10 still expose `config unset` only with dry-run-related flags, while the lower writer already has the narrow primitive this request wants to use. The remaining work is a maintainer product decision about adding a public destructive-write CLI affordance and its exact safety semantics.

## What This Changes

The issue requests a narrow `openclaw config unset` opt-in that bypasses only the config writer size-drop guard for intentional large deletes while preserving other destructive-write protections.

## Best Possible Solution

Add a maintainer-approved `config unset` flag that threads `allowConfigSizeDrop` only after the target path exists, keeps `unsetPaths`, schema validation, rejected artifacts, and hard guards such as `gateway-mode-removed`, and covers Commander, fast-route, CLI tests, writer-boundary tests, and docs.

## Reproduction Assessment

Yes, source-level: current `config unset` can delete an existing path and record `unsetPaths`, but neither the command options nor the route parser can pass `allowConfigSizeDrop`, while the writer blocks size drops unless that option is true. I did not run the destructive-write repro because this review is read-only.

## Solution Assessment

Mostly yes technically, pending maintainer approval: reusing the existing writer primitive is narrower than a broad `--force`, but maintainers should choose the public flag name and confirm this escape hatch belongs on `config unset`.
