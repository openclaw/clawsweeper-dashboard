## Summary

Keep open: the PR is aimed at the right WhatsApp plugin boundary, but the new `getMessage` backing store only records `messages.upsert` traffic and does not reliably provide OpenClaw-sent message content for Baileys resend callbacks. The contributor proof is also mocked/unit output only, so the live WhatsApp delivery path is not merge-ready.

## What This Changes

The PR adds optional Baileys `getMessage` and `cachedGroupMetadata` callbacks to WhatsApp socket creation and wires in-memory retry/group metadata caches plus group/LID event listeners through the WhatsApp monitors.

## Best Possible Solution

Land a WhatsApp-plugin-local fix that backs `getMessage` with bounded in-memory sent-message content from the send path, keeps the group metadata cache wiring, adds focused tests for callbacks/listeners/cache invalidation, and includes redacted live WhatsApp proof before merge.

## Reproduction Assessment

Yes at source level: current main omits the requested Baileys callbacks and group/LID event handlers, and Baileys v7.0.0-rc13 exposes and uses those hooks. I did not run a live WhatsApp retransmission or decryption failure in this read-only review.

## Solution Assessment

No: the owner boundary is right, but the `getMessage` store is populated from `messages.upsert` rather than the OpenClaw send path, so it cannot reliably satisfy Baileys resend callbacks for sent messages after reconnect/cache miss. The safer fix is to cache bounded sent `WAMessage` content where `sendTrackedMessage` receives the send result, then prove the live path.
