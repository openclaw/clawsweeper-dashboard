## Summary

Keep open. The branch addresses a real builtin-backend listener race, but the current diff still advertises all-backend coverage while qmd can return a wrapper without constructing the listener-owning `MemoryIndexManager`, so early archive events can still be missed for healthy qmd deployments.

## What This Changes

The PR adds a gateway sidecar that preloads session-source memory managers at startup, extends the gateway memory startup tests, and adds a changelog entry.

## Best Possible Solution

Keep the builtin eager preload direction, then make listener arming backend-aware for qmd or narrow the code and release note to the backend it actually covers.

## Reproduction Assessment

Yes, at source level. Current `main` emits archive updates before any guaranteed memory listener exists, and the PR body reports repeated builtin cold-start reproductions; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The sidecar is a narrow maintainable fix for builtin memory, but the qmd path can return a qmd/Fallback wrapper without instantiating `MemoryIndexManager`, so the advertised all-backend solution is incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Arm the qmd listener before marking preload ready:** `src/gateway/server-startup-memory.ts:157-160`
  - body: With `memory.backend=qmd`, this call can return the healthy qmd/Fallback wrapper. The builtin `MemoryIndexManager` is only created inside that wrapper's fallback factory, so `ensureSessionListener()` has not run and early reset/archive events can still miss the memory listener even though the preload logs as armed. Either explicitly arm the listener-owning manager for qmd or narrow the sidecar and changelog to builtin-only coverage.
  - confidence: 0.88
