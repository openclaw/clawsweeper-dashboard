## Summary

Keep open. This PR still has a unique builtin-backend startup-listener fix that current main does not provide; current main has later session startup catch-up behavior, but it does not eagerly arm the listener at gateway startup before the first memory manager load. The branch needs maintainer/contributor rebase and scope cleanup rather than conservative cleanup-close.

## What This Changes

The branch adds a nonblocking gateway startup sidecar that preloads builtin sessions-enabled `MemoryIndexManager` instances, skips qmd, adds focused gateway memory tests, and adds a `CHANGELOG.md` entry.

## Best Possible Solution

Land a rebased builtin-backend preload if maintainers accept builtin-only scope, while keeping qmd listener arming as a separate reviewed memory-backend design and leaving release-note wording to PR body or squash/release generation.

## Reproduction Assessment

Yes, from source and PR proof: with builtin memory, sessions source enabled, and `/reset` before the first memory manager load, current main has no gateway-startup listener to receive the archive event. I did not rerun the live container repro, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Mostly yes for the builtin backend: preloading the non-transient manager at gateway startup is the narrow owner-path fix for arming the existing listener. It is not the whole cross-backend solution because qmd is deliberately skipped and needs maintainer acceptance or a follow-up design.
