## Summary

Keep open: the scoped builtin-backend approach is useful, but the PR still leaves a startup race because the listener preload is scheduled in a nested `setImmediate` and sidecars can be marked ready before the manager is constructed.

## What This Changes

The PR adds a builtin-backend gateway sidecar that preloads non-transient sessions-enabled memory managers, adds regression coverage, and documents the scoped fix in the changelog.

## Best Possible Solution

Make gateway readiness wait until the builtin sessions listener preload has completed, or keep the relevant session methods gated until it has, while leaving qmd listener ownership for a separate design PR.

## Reproduction Assessment

Yes, at source level. Current main emits archive updates into an unqueued process-local listener set, while the sessions listener subscribes only after a non-transient `MemoryIndexManager` is constructed.

## Solution Assessment

No. The helper is scoped correctly for builtin memory, but the post-attach scheduling should not expose `/reset` or `/new` before that helper has actually armed the listener.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Await the listener preload before marking sidecars ready:** `src/gateway/server-startup-post-attach.ts:557`
  - body: This schedules the preload for a later `setImmediate` and then lets `startGatewaySidecars` return. Post-attach readiness removes the `sessions.send` startup gate as soon as that return happens, so a first `/reset` or `/new` can still be accepted before `startGatewayMemorySessionListeners` imports the module and constructs the manager; because transcript updates are unqueued, that archive emit is still lost. Await the preload in this measured sidecar, or keep session methods gated until the listener is armed.
  - confidence: 0.86
