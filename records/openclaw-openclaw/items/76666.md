## Summary

Keep open. The PR addresses a real source-visible startup race for builtin memory, but the proposed implementation still claims qmd coverage while qmd returns a wrapper without instantiating the builtin listener owner.

## What This Changes

The branch adds a gateway sidecar that preloads session-source memory managers at startup, tests the helper, and adds a changelog entry.

## Best Possible Solution

Keep the eager preload direction for builtin memory, then either make listener arming backend-aware for qmd or narrow the public claim before merge.

## Reproduction Assessment

Yes, at source level. Current main only subscribes when a non-transient MemoryIndexManager is instantiated, and the PR body reports repeated builtin cold-start reproductions; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The new sidecar is a narrow maintainable approach for builtin, but the qmd path still returns a wrapper without constructing the listener-owning MemoryIndexManager, so the advertised backend coverage is incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preload the builtin listener when qmd is active:** `src/gateway/server-startup-memory.ts:158-161`
  - body: This treats any successful `getActiveMemorySearchManager` result as listener-armed, but with the qmd backend that call returns the cached qmd/FallbackMemoryManager wrapper. The builtin `MemoryIndexManager` only gets constructed inside the wrapper's `fallbackFactory`, so `ensureSessionListener()` still has not run when qmd is healthy and reset/archive `sessionTranscriptUpdate` events can still be dropped before the first fallback/search path. Either explicitly arm the listener owner for qmd too, or narrow the PR and changelog to builtin-only coverage.
  - confidence: 0.88
