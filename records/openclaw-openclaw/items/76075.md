## Summary

Keep open: the bug is real and the parser/runner seam is the right place, but the latest PR head still puts parsed tool arguments on result events while current clients read them from start events.

## What This Changes

This PR adds Claude stream-json tool-use parsing to the CLI JSONL parser, threads a tool-event callback through live and one-shot CLI runner paths, forwards those events through `emitAgentEvent({ stream: "tool" })`, and adds parser tests plus a changelog entry.

## Best Possible Solution

Keep the callback and runner wiring, but emit the existing sanitized tool-event contract so parsed args are available where current web and native consumers read them, with regression coverage for the parser and at least one downstream consumer.

## Reproduction Assessment

Yes. A Claude stream-json fixture with `content_block_start`, `input_json_delta`, and `content_block_stop` reproduces the current-main drop by source inspection, and the PR fixture demonstrates the remaining start-without-args payload shape.

## Solution Assessment

No. The boundary is maintainable, and the raw-delta security issue appears fixed, but the event payload still does not match the current cross-client consumer contract.
