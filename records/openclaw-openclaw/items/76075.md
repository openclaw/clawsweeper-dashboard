## Summary

Keep open: current main still has the reported Claude CLI tool-stream gap, and this PR is useful, but the latest head still emits `phase: "result"` from Claude's `content_block_stop`, which closes the pending tool state immediately after `start` instead of representing actual tool completion.

## What This Changes

The PR adds Claude stream-json `tool_use` parsing to the CLI JSONL parser, forwards those tool events through live and one-shot CLI runner paths, adds parser coverage, and updates the changelog.

## Best Possible Solution

Keep the parser callback and runner wiring, but emit sanitized `start` when the tool call input is complete and defer or omit `result` until a real completion signal or explicit turn-end cleanup path exists.

## Reproduction Assessment

Yes. A Claude stream-json fixture with `content_block_start`, `input_json_delta`, and `content_block_stop` reproduces the main-branch drop because the parser has no tool callback, and the PR's added fixture demonstrates the current back-to-back start/result behavior.

## Solution Assessment

No. The PR uses the right parser and runner boundary, but treating `content_block_stop` as tool completion is not the narrowest correct lifecycle mapping for existing Control UI, Android, and Swift consumers.
