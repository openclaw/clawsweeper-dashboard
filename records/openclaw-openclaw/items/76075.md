## Summary

Keep this PR open: it fixes the main parser/runner gap, but the latest head still synthesizes a `phase: "result"` at `content_block_stop`, which marks the end of tool-call input rather than the actual tool result, so native clients can clear the pending tool activity immediately.

## What This Changes

The PR adds Claude stream-json tool-use parsing to the CLI JSONL parser, forwards those events through live and one-shot CLI runner paths, adds parser tests, and updates the changelog.

## Best Possible Solution

Keep the parser callback and runner wiring, but make the emitted `tool` stream reflect the real lifecycle: sanitized args on start when the tool call is complete, and result only when the matching tool result or another defensible terminal signal arrives.

## Reproduction Assessment

Yes. A Claude stream-json fixture with `content_block_start`, `input_json_delta`, and `content_block_stop` reproduces both the current-main drop and the PR's immediate start/result sequence; the native consumer code shows why that sequence clears pending activity too early.

## Solution Assessment

No. The approach is close, and the raw-delta security issue is addressed, but synthesizing a result from `content_block_stop` is not the narrowest correct lifecycle mapping.
