## Summary

Keep open. The linked bug is valid, but this patch would forward raw Claude tool arguments and introduces `delta`/`end` tool phases that do not match the existing downstream tool-stream contract.

## What This Changes

The PR adds a Claude CLI JSONL tool-use callback, threads it through one-shot and live CLI runners, emits tool events, and adds parser tests plus a changelog entry.

## Best Possible Solution

Keep the parser callback seam, but emit the existing sanitized `stream: "tool"` start/update/result contract and cover the parser-to-runner-to-gateway/UI path with focused regression tests.

## Reproduction Assessment

Yes. A Claude stream-json fixture with `content_block_start`, `input_json_delta`, and `content_block_stop` proves current main drops tool activity, and source inspection shows the proposed payload would not match downstream consumers.

## Solution Assessment

No. The callback wiring is the right boundary, but the event data should use the established sanitized `start`/`update`/`result` shape rather than introducing raw `delta`/`end` payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Sanitize tool arguments before emitting them:** `src/agents/cli-output.ts:468`
  - body: This emits Claude's parsed tool input as `args`. The established pi path redacts tool arguments before `emitAgentEvent`, while gateway fanout only strips `result` and `partialResult`, so sensitive argument fields can reach WS/session subscribers.
  - confidence: 0.86
- **[P2] Emit the existing tool stream phases:** `src/agents/cli-output.ts:447-468`
  - body: The new parser emits `delta` and `end`, but current consumers handle tool progress through `start`, `update`, and `result`; Control UI reads args only on `start` and output only on `update`/`result`, so this will not mirror the pi-embedded contract.
  - confidence: 0.83
