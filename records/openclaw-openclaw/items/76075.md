## Summary

Keep open for normal PR review. Current main still drops Claude CLI `tool_use` stream-json records, and this PR targets the right parser/runner bridge with sanitized `start`/`update`/`result` tool events and focused parser coverage; I did not find a discrete diff-level correctness bug, but exact-head CI was not green at review time.

## What This Changes

The branch adds Claude stream-json `tool_use` parsing and `onToolEvent` forwarding for one-shot and live Claude CLI runners, updates parser tests and the changelog, and adjusts root package dist extension exclusions.

## Best Possible Solution

Merge this parser-level event bridge after exact-head CI is green or the current CI failures are proven unrelated, preserving the existing `stream: "tool"` consumer contract and sanitized payload boundary.

## Reproduction Assessment

Yes. Feed current main’s `createCliJsonlStreamingParser` a Claude stream-json `content_block_start` with `content_block.type: "tool_use"`, an `input_json_delta`, and `content_block_stop`; there is no `onToolEvent` hook or runner forwarding path, so no `stream: "tool"` event can be produced.

## Solution Assessment

Yes, with the CI caveat. An optional parser callback wired into both one-shot and live Claude CLI runner paths is the narrow maintainable fix because downstream consumers already handle the generic `stream: "tool"` contract once emitted.
