## Summary

Keep open: the reported Claude CLI tool-event gap is valid and the PR chooses the right parser/runner boundary, but current head still exposes raw streamed tool arguments and does not deliver parsed args where existing tool-event consumers read them.

## What This Changes

The PR adds Claude CLI stream-json tool-use parsing, wires parser tool events through one-shot and live CLI runners to `emitAgentEvent({ stream: "tool" })`, adds parser tests, and updates the changelog.

## Best Possible Solution

Keep the parser callback seam, accumulate Claude tool argument JSON internally, and emit only sanitized `stream: "tool"` payloads that match the established start/update/result consumer contract, with parser plus gateway/UI/native-facing coverage.

## Reproduction Assessment

Yes. A Claude stream-json fixture with `content_block_start`, `input_json_delta`, and `content_block_stop` reproduces the current-main drop, and source inspection verifies the PR head still mismatches the existing consumer contract.

## Solution Assessment

No. The boundary is maintainable, but the payload shape should avoid raw `partialJson` fanout and place sanitized argument data on the consumer-visible start path or deliberately update every downstream consumer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not stream raw argument deltas:** `src/agents/cli-output.ts:448`
  - body: This still forwards Claude's raw `input_json_delta.partial_json` as `data.partialJson` on every tool update. Gateway non-full tool fanout only strips `result` and `partialResult`, so secrets embedded in tool arguments can reach WebSocket/session subscribers before the final args are sanitized. Accumulate the JSON internally and avoid broadcasting raw argument chunks.
  - confidence: 0.91
- **[P2] Put parsed args on the consumer-visible start payload:** `src/agents/cli-output.ts:470`
  - body: The parsed args are emitted only on `phase: "result"`, but current Control UI, Android, Swift, macOS activity, and ACP consumers populate tool details from `phase: "start"` and treat `result` as completion. Claude CLI tool cards/status will still lack command/path details unless the sanitized args are delivered on the established start contract or all consumers are updated together.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch still introduces a raw streamed tool-argument field that can bypass existing gateway tool-payload redaction.

Concerns:

  - confidence: 0.91
