## Summary

Keep open: current main and v2026.6.8 still have only the `tools.fs.workspaceOnly` boolean, memory-triggered writes still go through the workspace-only file-write path, and the linked implementation PR is open rather than merged.

## What This Changes

The issue requests an explicit trusted workspace alias or memory-root config so symlinked shared memory paths can be written without disabling workspace-only path-alias protection.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/90986 after product/security approval or choose a narrower memory-root/filesystem-policy design, then close this issue after the accepted implementation merges.

## Reproduction Assessment

## Solution Assessment

Unclear as a final product choice. A narrow exact-realpath allowlist is plausible, but maintainers still need to approve whether this config shape and host-only filesystem boundary are the best long-term contract.
