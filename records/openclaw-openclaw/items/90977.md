- `P2`: This is a normal-priority filesystem/config improvement with real memory-flush impact but no crash loop, outage, or emergency security bypass.
- `impact:security`: The request changes how `tools.fs.workspaceOnly` treats symlinked paths and would create an explicit filesystem-boundary exception.

## Summary

Keep open: current main and v2026.6.6 still have only the binary `tools.fs.workspaceOnly` guard, and the active linked implementation PR is open rather than merged. The remaining decision is security/product approval of a new trusted filesystem alias boundary, not conservative cleanup.

## What This Changes

Add a trusted workspace alias or memory-root configuration so file tools and memory flushes can write through an intentional symlinked `memory/` directory without disabling `tools.fs.workspaceOnly`.

## Best Possible Solution

Keep this issue open until maintainers either land https://github.com/openclaw/openclaw/pull/90986 or choose a narrower memory-root/filesystem-policy design with explicit security review.

## Reproduction Assessment

Yes. Source inspection shows memory-triggered runs force workspace-only writes and host writes canonicalize symlink parents, while current tests explicitly reject outside-workspace symlink parents.

## Solution Assessment

Unclear as a final product choice. A narrow exact-realpath alias is plausible, but the permanent config shape, operation scope, host/sandbox boundary, and audit visibility need maintainer/security approval.
