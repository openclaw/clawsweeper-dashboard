## Summary

Keep open: current main and v2026.6.1 still only expose `tools.fs.workspaceOnly` and intentionally reject symlinked workspace parents that resolve outside the workspace. This is a security-sensitive feature/design request, not an already-implemented cleanup item.

## What This Changes

The issue requests trusted workspace aliases or a memory-root policy so `memory/...` writes can target an intentional symlink outside the agent workspace without disabling path-escape protection.

## Best Possible Solution

Keep this open as a focused design item; if accepted, add a narrow exact-realpath alias or memory-root policy that preserves default escape rejection and is reconciled with broader filesystem sandboxing config.

## Reproduction Assessment

Yes, by source inspection and existing tests: a `memory` symlink that resolves outside the workspace will fail the workspace-only write path, and memory flush runs force that policy. I did not run a live repro because this review was read-only.

## Solution Assessment

Unclear; the requested alias direction is plausible and has a skill-loading precedent, but the permanent API shape needs maintainer security/product judgment. Disabling `workspaceOnly` or broadening the whole workspace is not the best fix for the stated goal.
