## Summary

Keep open: current main and v2026.6.6 still lack a trusted workspace alias or memory-root config, and the linked closing PR is open and unmerged. The remaining decision is product/security approval for a new filesystem boundary config.

## What This Changes

Add an explicit trusted workspace alias or memory-root config so memory/file writes can use intentional out-of-workspace symlink targets without disabling the default path-escape guard.

## Best Possible Solution

Keep this issue open as the canonical design/acceptance item while maintainers review https://github.com/openclaw/openclaw/pull/90986 and any broader filesystem-root alternative such as https://github.com/openclaw/openclaw/pull/52951.

## Reproduction Assessment

Yes, at source level. Current main forces memory-triggered writes into workspace-only mode, and existing tests reject writes through symlink parents that resolve outside the workspace.

## Solution Assessment

Unclear as a final product choice. A narrow exact-realpath alias is plausible and matches the requested workflow, but maintainers still need to decide whether that config shape or a broader filesystem-roots contract is the best long-term boundary.
