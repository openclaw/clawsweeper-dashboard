## Summary

Keep open: current main and v2026.6.9 still lack a trusted workspace-alias or memory-root configuration, and the open linked PR is the active implementation candidate awaiting maintainer product/security review.

## What This Changes

The issue requests a safe allowlisted workspace-alias or memory-root mechanism so file tools and memory flushes can write through intentional symlinks without disabling workspace escape protection.

## Best Possible Solution

Choose one explicit filesystem-boundary design: land the open exact-realpath alias PR after product/security approval, or replace it with a narrower memory-root or broader roots contract, then close this issue only after the accepted implementation merges.
