## Summary

Keep open: current main includes the PR #91241 budget-deferral retry fix, but that PR explicitly left Feishu transport-readiness recovery open, and source still shows startup recovery running without a channel-ready gate while Feishu lacks the reconnect drain used by sibling channels.

## What This Changes

The issue reports that pending outbound deliveries can fail and exhaust retries after a gateway restart because recovery can run before Feishu transport readiness is established.

## Best Possible Solution

Keep the merged budget-deferral fix, then add a Feishu or generic channel-owned readiness/reconnect drain through the existing delivery-queue runtime seam with explicit maintainer-approved retry semantics instead of a fixed startup sleep or blanket retry reset.

## Reproduction Assessment

Yes at source level for the remaining risk: startup recovery can still call delivery without a channel transport-ready gate, and Feishu does not appear to drain queued deliveries on WebSocket reconnect. No live Feishu restart run was performed, so the exact user-environment timing still needs live proof.

## Solution Assessment

No; PR #91241 is the right narrow fix for budget deferral, but the reporter's fixed sleep and blanket retry reset are not the best durable solution. The maintainable path is a readiness/reconnect-drain fix with an explicit retry policy.
