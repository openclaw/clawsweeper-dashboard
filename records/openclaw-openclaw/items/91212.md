## Summary

Keep open: current main and v2026.6.8 fix the budget-deferral retry accounting, but the Feishu transport-readiness/reconnect drain remains unowned and still needs maintainer policy on restart retry semantics.

## What This Changes

The issue asks OpenClaw to prevent outbound delivery recovery from failing queued Feishu messages during gateway restart before the channel transport is ready.

## Best Possible Solution

Keep the shipped budget-deferral fix, then add a Feishu-owned readiness/reconnect drain through the existing plugin SDK delivery-queue seam if maintainers confirm the retry policy.

## Reproduction Assessment

Yes at source level: current main still starts outbound recovery without a Feishu-ready drain, and Feishu has no reconnect drain while sibling channels do. No live Feishu restart was executed in this read-only review.

## Solution Assessment

No for the exact workaround shape: a fixed sleep and unconditional retry reset are brittle. The better path is a Feishu/channel-owned readiness or reconnect drain plus an explicit maintainer decision on whether restart should reset retry history.
