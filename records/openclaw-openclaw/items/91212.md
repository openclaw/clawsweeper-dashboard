## Summary

Keep open: current main and v2026.6.8 include the budget-deferral retry-accounting fix, but the remaining Feishu restart/readiness behavior is not implemented or superseded by a canonical merged fix.

## What This Changes

Ensure delivery recovery after gateway restart does not fail pending Feishu deliveries before channel readiness or exhaust retryCount for restart-window failures.

## Best Possible Solution

Keep the shipped budget-deferral fix, then prove the Feishu restart failure live and add a channel-owned readiness or reconnect-drain path rather than a fixed sleep or unconditional retry reset.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection does show startup recovery can run after channel startup handoff but before Feishu WebSocket readiness, and Feishu lacks the reconnect-drain pattern used by Telegram and WhatsApp.

## Solution Assessment

No, the local workaround is not the best final shape. A fixed sleep and unconditional retry reset are too broad; the maintainable solution is a Feishu/channel-owned readiness or reconnect drain backed by live restart proof and a clear retry policy.
