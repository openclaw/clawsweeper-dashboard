## Summary

Keep open: current main and the latest release still encode the reported delivery-recovery behavior, and Feishu does not appear to have the reconnect-triggered delivery drain that sibling channels use. The reporter's workaround is useful context, but the durable retry/reset policy and channel-readiness boundary need maintainer direction before a repair should land.

## What This Changes

The issue asks OpenClaw to prevent pending Feishu outbound deliveries from failing after gateway restart when recovery runs before WebSocket transport readiness and retry accounting persists across restarts.

## Best Possible Solution

Add a generic, channel-owned transport-recovery drain for Feishu using the existing delivery-queue runtime seam, and adjust core recovery retry accounting only after maintainers decide how budget deferral and restart-boundary retries should behave.

## Reproduction Assessment

Yes for the retry-accounting part from current source: budget-deferred entries are passed through failDelivery, which increments retryCount. The Feishu timing part is source-plausible because Feishu lacks a reconnect drain used by sibling channels, but it still needs live Feishu restart proof.

## Solution Assessment

No; the proposed local five-second sleep and unconditional retry reset are useful debugging proof, but the best fix should use a channel readiness/reconnect signal and an explicit maintainer-approved retry policy.
