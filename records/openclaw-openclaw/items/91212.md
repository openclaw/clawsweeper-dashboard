## Summary

Keep open: current main and v2026.6.6 include the budget-deferral retry-accounting fix from https://github.com/openclaw/openclaw/pull/91241, but the Feishu transport-readiness/reconnect drain and restart retry-policy question remain unresolved.

## What This Changes

The issue reports outbound delivery recovery losing queued Feishu messages after gateway restart because recovery can run before Feishu WebSocket readiness and retry accounting can be consumed across restarts.

## Best Possible Solution

Keep the shipped budget-deferral fix, then add a Feishu-owned transport-ready or reconnect-triggered drain using the existing delivery-queue runtime seam instead of a fixed startup sleep.

## Reproduction Assessment

Partly yes: current source proves the budget-deferral retry bug is fixed, and source inspection shows Feishu still lacks the reconnect drain used by sibling channels. A live Feishu gateway restart is still needed for high-confidence end-to-end reproduction of the transport-readiness race.

## Solution Assessment

No; the reporter's five-second sleep and unconditional retry reset are useful workaround evidence, but the best fix is a channel readiness/reconnect drain plus an explicit maintainer-approved retry policy.
