## Label Justifications

- `P1`: The report describes silent loss of queued channel replies after gateway restart for a real Feishu WebSocket deployment on the latest release.
- `impact:message-loss`: The affected surface is durable outbound delivery recovery for queued channel messages that users expect to be delivered after restart.

## Summary

Keep open: current main and the latest release still have the startup recovery and retry-accounting paths described here, and Feishu does not appear to drain its pending delivery queue after WebSocket reconnect the way sibling channels do. The report is specific enough to route, but the durable retry policy needs maintainer direction before an automated repair should change it.

## What This Changes

The issue reports Feishu pending outbound deliveries failing during gateway restart recovery because startup replay can run before transport readiness and retry accounting can exhaust queued messages.

## Best Possible Solution

Add a transport-ready replay path for Feishu or a generic readiness-gated recovery path, stop counting unattempted budget deferrals as delivery attempts, and cover startup plus reconnect behavior with regression tests while preserving unknown-send safety.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a clear reproduction path: channel startup hands off transport asynchronously, scheduled startup recovery drains pending entries, Feishu lacks a reconnect drain, and failed recovery attempts increment retryCount.

## Solution Assessment

Unclear pending maintainer policy. The narrow code direction is likely Feishu reconnect/transport-ready drain plus safer budget deferral, but resetting or capping retryCount across restarts needs an explicit durable-queue policy decision.
