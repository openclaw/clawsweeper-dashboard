- `P1`: The issue reports restart-triggered failure of queued Feishu replies, which can break a real channel workflow for affected users.

## Summary

Keep open: current main and v2026.6.8 fix the budget-deferral retry inflation, but the remaining Feishu gateway-restart/readiness behavior is not implemented or superseded by a canonical merged fix.

## What This Changes

The issue requests preventing outbound delivery recovery after gateway restart from failing queued Feishu messages before channel transport readiness, without consuming retry budget for unattempted entries.

## Best Possible Solution

Keep the shipped budget-deferral fix, then add a Feishu/channel-owned readiness or reconnect-drain path backed by live restart proof rather than a fixed sleep or blanket retry reset.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection shows startup delivery recovery runs independently of Feishu WebSocket lifecycle and Feishu lacks the reconnect drain used by Telegram and WhatsApp, so the reported race is source-reproducible but still needs live Feishu restart proof.

## Solution Assessment

No, the local workaround is not the best final shape. A fixed sleep and unconditional retry reset are too broad; the safer solution is a channel-owned readiness or reconnect drain with explicit retry policy.
