## Summary

Keep open: current main and v2026.6.6 include the budget-deferral retry-accounting fix from https://github.com/openclaw/openclaw/pull/91241, but the Feishu transport-readiness part remains unowned; Feishu still lacks a reconnect/readiness drain while sibling channels use the delivery-queue runtime seam.

## What This Changes

The issue reports outbound Feishu deliveries being lost after gateway restart because startup recovery can replay queued messages before the channel transport is actually ready and retry counts persist across restarts.

## Best Possible Solution

Add a Feishu-owned reconnect/readiness drain using the existing plugin SDK delivery-queue seam, keep the shipped budget-deferral fix, and decide the restart-boundary retry reset policy explicitly before changing retry semantics.

## Reproduction Assessment

Yes at source level for the remaining race shape: channel startup hands off monitor tasks before Feishu WebSocket readiness is proven, scheduled recovery can run after that handoff, and Feishu has no reconnect drain. No live Feishu restart proof was run in this review.

## Solution Assessment

No for the reporter's exact patch shape: a fixed sleep and unconditional retry reset are brittle. The maintainable fix should be a Feishu channel readiness/reconnect drain plus an explicit retry policy decision.
