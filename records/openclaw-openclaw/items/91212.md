## Summary

Keep open: current main and v2026.6.6 fix the budget-deferral retry inflation, but the central Feishu transport-readiness restart behavior remains unimplemented and is not owned by another canonical issue or PR.

## What This Changes

The issue asks for pending Feishu outbound deliveries to survive gateway restart until channel transport readiness instead of failing recovery and consuming retry budget before reconnect.

## Best Possible Solution

Keep the no-increment budget deferral behavior, then add a signal-driven Feishu or generic channel-owned reconnect drain through the existing delivery-queue runtime seam instead of a fixed startup sleep or unconditional retry reset.

## Reproduction Assessment

No live high-confidence Feishu restart reproduction was established in this read-only pass. The source path is clear: startup recovery still runs before a Feishu reconnect drain exists, while sibling channels drain after reconnect.

## Solution Assessment

No, the local five-second sleep and blanket retry reset are not the best maintainable fix. A transport-ready or reconnect-triggered drain using the existing queue seam is the safer direction, with retry reset policy decided explicitly.
