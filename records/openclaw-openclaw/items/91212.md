## Summary

Keep open: current main and v2026.6.9 ship the budget-deferral retry fix, but Feishu gateway-restart/readiness recovery is still not implemented or superseded by a merged canonical fix.

## What This Changes

The issue reports that Feishu pending outbound deliveries can fail after gateway restart because startup recovery runs before channel transport readiness and retry accounting can exhaust queued messages.

## Best Possible Solution

Keep the shipped budget-deferral fix, then add a Feishu/channel-owned readiness or reconnect-drain path backed by live restart proof rather than a fixed sleep or blanket retry reset.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection shows startup delivery recovery still runs independently of Feishu WebSocket lifecycle and Feishu lacks the reconnect drain used by other channel recovery paths, so the race remains source-reproducible but needs live Feishu restart proof.

## Solution Assessment

No, the local workaround is not the best final shape. A fixed sleep and unconditional retry reset are too broad; the safer solution is a channel-owned readiness or reconnect drain with explicit retry policy.
