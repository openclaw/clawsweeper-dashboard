## Summary

Keep open: current main and v2026.6.8 include the budget-deferral retry-accounting fix, but the Feishu transport-readiness/reconnect part remains unresolved and still needs maintainer policy plus live restart proof.

## What This Changes

The issue reports queued Feishu outbound deliveries failing after gateway restart because delivery recovery can run before the channel transport is ready, with retry counts compounding the loss.

## Best Possible Solution

Keep the budget-deferral fix as shipped, then add a maintainer-approved Feishu/channel readiness or reconnect drain path with live restart proof instead of a fixed sleep or unconditional retry reset.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection does show the remaining startup recovery versus Feishu readiness gap, and the merged regression test proves the budget-deferral sub-bug was fixed.

## Solution Assessment

No, not as proposed in the local workaround. A channel-owned readiness/reconnect drain is the maintainable direction; fixed sleeps and unconditional retry resets need policy review because they can mask transport timing and alter retry semantics.
