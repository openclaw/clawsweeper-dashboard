- `P1`: The issue reports restart-time outbound recovery failures that can silently drop queued channel replies for real users.

## Summary

Keep open: current main and v2026.6.8 include the budget-deferral retry-accounting fix, but the Feishu transport-readiness/reconnect drain part remains unresolved; Feishu still starts its WebSocket monitor without using the delivery-queue reconnect drain that sibling channels use.

## What This Changes

The issue requests that pending Feishu outbound deliveries survive gateway restarts without being failed before the Feishu WebSocket transport is ready or exhausting retry budget across restarts.

## Best Possible Solution

Add a Feishu-owned readiness/reconnect drain through the existing plugin SDK delivery-queue seam, keep the shipped budget-deferral fix, and decide restart-boundary retry reset policy explicitly before changing retry semantics.

## Reproduction Assessment

Yes at source level: startup recovery can run after channel handoff while Feishu WebSocket readiness is not proven, and Feishu has no reconnect drain. No live Feishu restart reproduction was run in this read-only pass.

## Solution Assessment

No for the suggested fixed sleep and unconditional retry reset; the maintainable solution is a Feishu-owned readiness/reconnect drain plus an explicit retry-policy decision for restart boundaries.
