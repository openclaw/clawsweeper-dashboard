## What This Changes

The issue asks delivery recovery after gateway restart to wait for Feishu transport readiness and avoid consuming retry budget for unattempted or restart-boundary retries.

## Best Possible Solution

Keep the shipped budget-deferral fix, then add a Feishu-owned or generic channel-readiness drain using the existing delivery-queue runtime seam after live Feishu restart proof and an explicit retry policy decision.

## Reproduction Assessment

No for the full issue: current source proves the old budget-deferral retry accounting is fixed and shows Feishu still lacks a reconnect drain, but the Feishu WebSocket restart failure still needs live transport reproduction. The best current reproduction target is a real Feishu WebSocket restart with pending queued deliveries.

## Solution Assessment

No: the local five-second sleep and unconditional retry reset are useful workaround evidence but not the best product fix. A channel readiness/reconnect drain plus a maintainer-approved retry policy is narrower and less likely to mask permanent failures.
