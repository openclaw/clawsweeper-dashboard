## Summary

Keep open: the room-event direction matches the Telegram visible-reply policy and has live proof, but the latest head still lets room events reach the generic Telegram fallback reply path when dispatch throws. The protected maintainer label also means this PR should stay in normal maintainer review rather than cleanup closure.

## What This Changes

The PR adds room-event turn semantics across Telegram inbound context, reply prompting, session persistence, context maps, docs, and tests so ambient group messages are context-only unless the message tool sends.

## Best Possible Solution

Keep the room-event design, but suppress Telegram fallback/error delivery for room events unless `message(action=send)` is used, then add a dispatcher-rejection regression test.

## Reproduction Assessment

Yes, at source level: make `dispatchReplyWithBufferedBlockDispatcher` reject for a `room_event`; the catch sets `dispatchError`, and the shared fallback still calls `deliverReplies`. I did not run a live Telegram failure-path repro.

## Solution Assessment

No, not yet. The room-event model is the maintainable direction, but the implementation needs a narrow fallback/error guard before it fully enforces the message-tool-only contract.
