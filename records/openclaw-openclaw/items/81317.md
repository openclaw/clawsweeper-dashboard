## Summary

Keep open: the room-event direction matches Telegram group-reply policy and the real behavior proof is sufficient, but the latest head still has a fallback path that can visibly reply to a room event without the message tool.

## What This Changes

The branch adds `room_event` turn semantics for ambient Telegram group chatter, carries those turns through prompt/context accounting, documents the behavior, and suppresses normal Telegram reactions/drafts/replies unless the agent uses the message tool.

## Best Possible Solution

Keep the room-event design, but add a room-event guard around Telegram error/fallback delivery so ambient turns remain quiet unless `message(action=send)` is used, then cover that failure path with a focused regression test.

## Reproduction Assessment

Yes, at source level: make `dispatchReplyWithBufferedBlockDispatcher` reject for a context whose `InboundTurnKind` is `room_event`; the catch sets `dispatchError`, and the shared fallback still calls `deliverReplies`. I did not run a live Telegram failure-path repro.

## Solution Assessment

No, not yet. The proposed room-event model is the right shape for the maintainer policy, but the implementation needs a narrow fallback/error guard before it fully enforces the message-tool-only contract.
