## Summary

Keep open: this is an active Telegram behavior PR with a protected maintainer label, and source review found a remaining quiet-contract gap where room-event dispatch errors can still post a visible Telegram fallback.

## What This Changes

The PR adds a new `room_event` turn kind, routes ambient always-on Telegram group messages through message-tool-only processing, preserves them as context/history, and updates prompt/session reporting, docs, and tests.

## Best Possible Solution

Keep the room-event design, but make room events message-tool-only through both success and failure paths so ambient group activity remains context-only unless the message tool sends.

## Reproduction Assessment

Yes, source-reproducible for the review finding: a room event whose dispatch throws leaves `dispatchError` set and reaches the visible Telegram fallback path. I did not run a live Telegram failure repro in this read-only review.

## Solution Assessment

No, not yet. The current PR handles normal quiet paths well, but it should also suppress room-event error fallback delivery and keep those failures internal unless the message tool already sent something visible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Suppress error fallbacks for room events:** `extensions/telegram/src/bot-message-dispatch.ts:1433-1434`
  - body: This sets room events to `message_tool_only`, but that only suppresses the normal no-reply fallback after a successful dispatch result. If the agent/provider dispatch throws, the later `dispatchError` fallback still sends a visible Telegram error message into the group, which violates the quiet-unless-message-tool contract for ambient chatter. Gate that fallback for `isRoomEvent` and keep the failure internal/logged unless a message-tool send already happened.
  - confidence: 0.86
