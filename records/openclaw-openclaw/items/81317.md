## Summary

Keep open: this active Telegram PR is protected by the `maintainer` label and still has a blocking source-level quiet-path bug. Nominal real Telegram proof is sufficient, but a `room_event` dispatch error can still send the shared visible Telegram fallback without a `message(action=send)` call.

## What This Changes

The PR adds `room_event` turn semantics across Telegram inbound context, reply prompt/session handling, context reporting, docs, and tests so ambient always-on group chatter is context-only unless the message tool sends a visible reply.

## Best Possible Solution

Keep the room-event design if maintainers approve it, but make every room-event error and fallback path internal/logged unless the agent explicitly sends through the message tool, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: make the room-event dispatch call throw before it returns a turn result, and the current PR head leaves `dispatchError` to reach the shared `deliverReplies` fallback. I did not run a live Telegram failure-path repro in this read-only review.

## Solution Assessment

No, not yet. The normal quiet path is well covered, but the implementation must suppress visible error fallbacks for room events before it fully matches the `message_tool_only` contract.
