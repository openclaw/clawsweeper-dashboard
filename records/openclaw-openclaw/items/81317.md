## Summary

Keep this PR open: the main room-event behavior is useful and has real Telegram proof, but the current head still has a P2 quietness leak on the dispatch error path. A room event can still hit the shared visible fallback reply without an explicit message-tool send.

## What This Changes

The PR adds room-event semantics for ambient Telegram group messages, treating them as context-only turns that can speak visibly only through the message tool.

## Best Possible Solution

Keep the room-event design only if every ambient room-event error and fallback path stays internal/logged unless the agent explicitly sends through the message tool.

## Reproduction Assessment

Yes, at source level: set `ctxPayload.InboundTurnKind` to `room_event` and make the reply dispatch throw before it returns a turn result; `dispatchError` then reaches the shared Telegram fallback at `extensions/telegram/src/bot-message-dispatch.ts:1659`. I did not run a live Telegram failure-path repro in this read-only review.

## Solution Assessment

No, not yet. The proposed direction matches the Telegram visible-reply policy, but the implementation needs a room-event guard around visible error/fallback delivery before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Suppress room-event error fallbacks:** `extensions/telegram/src/bot-message-dispatch.ts:1433-1434`

That lets ambient group chatter produce a visible Telegram reply without `message(action=send)`, so guard this fallback for room events and add a failure-path regression test.
