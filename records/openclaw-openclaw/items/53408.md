## Summary

Keep open: current main has several scoped repairs for malformed or empty tool-call arguments, but I could not find a high-confidence fix for the reported long-session custom Anthropic-compatible and large-argument routes.

## What This Changes

The issue asks OpenClaw to stop silently losing `exec` and `write` tool arguments after long, tool-heavy conversations, especially with large CJK or heredoc payloads through webchat/custom provider routes.

## Best Possible Solution

Keep this as the canonical long-session argument-loss investigation, add a redacted trace-backed regression for the affected provider stream sequence, and then make the narrowest transport or repair-policy change while leaving the larger staged-payload API design to the separate feature request.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection supports a plausible current-main path because the reported custom Anthropic-compatible route can bypass the scoped repair gate and still reach required-parameter validation after stream parsing.

## Solution Assessment

Unclear. A trace-backed provider or transport policy fix is safer than globally enabling malformed-argument repair for every Anthropic-compatible/custom provider, and the separate streaming/file-argument contract should not be folded into this bug fix without maintainer design.
