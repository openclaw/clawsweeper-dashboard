## Label Justifications

- `P1`: The issue reports a provider-auth regression that adds roughly 8-13s to every xAI agent turn for affected users.
- `impact:auth-provider`: The affected surface is xAI credential resolution and provider auth setup for OAuth and API-key profiles.

## Summary

Keep open: the reporter's follow-up rules out the earlier OAuth-refresh-only hypothesis by showing the same 8-10s auth-stage delay with an API-key-only xAI profile, and current main still does per-turn credential resolution plus provider runtime-auth lookup inside the measured auth stage.

## What This Changes

The issue asks OpenClaw to stop spending 8-13s in the per-turn auth stage for already-resolved xAI credentials on Windows.

## Best Possible Solution

Add per-substage timing for the auth bootstrap path, then fix the identified xAI/provider-auth hot path so warm OAuth and API-key turns avoid repeated expensive resolution while preserving providers that require runtime-auth preparation.

## Reproduction Assessment

Partly. The reporter supplied concrete Windows/npm logs and an OAuth-vs-API-key A/B on 2026.5.20, and source inspection confirms the per-turn auth stage runs credential and provider-runtime resolution, but this review did not independently run the live xAI path on current main.

## Solution Assessment

Unclear. The earlier OAuth-refresh explanation is no longer the best fit; the maintainable next step is targeted timing around getApiKeyForModel, auth-store access, and prepareProviderRuntimeAuth before changing caching or provider-runtime behavior.
