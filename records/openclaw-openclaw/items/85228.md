## Label Justifications

- `P1`: The report describes a recent provider-auth regression that adds roughly 8-13s to every affected xAI Grok agent turn.
- `impact:auth-provider`: The affected surface is provider credential resolution and auth initialization for xAI/Grok profiles.

## Summary

Keep open: the reporter narrowed the symptom from xAI OAuth refresh to a provider-auth hot path that affects both xAI OAuth and xAI static API-key profiles, and current main still awaits embedded-run auth initialization before the measured auth-stage marker. Related broader auth-stage work is useful context, but it does not yet prove this xAI/Windows latency is fixed.

## What This Changes

The issue asks OpenClaw to stop spending 8-13s in embedded-run auth on every xAI Grok turn when credentials are already resolved.

## Best Possible Solution

Add fine-grained auth substage timing for xAI OAuth and xAI API-key turns on current main, then optimize the proven hot subpath while preserving provider hook and OAuth refresh contracts.

## Reproduction Assessment

No high-confidence local current-main reproduction was established. The reporter supplied strong live v2026.5.20 traces, and current source confirms every PI-owned xAI attempt still waits for auth initialization inside the measured auth bucket.

## Solution Assessment

Unclear. A blanket skip or config flag would be risky because provider runtime auth hooks are part of the contract; the right fix needs targeted timing data and then a narrow cache, prepared-fact carry-forward, or hook-resolution fast path.
