## Label Justifications

- `P1`: The issue reports a recent provider-auth regression that adds several seconds to every xAI/Grok agent turn for affected users.
- `impact:auth-provider`: The affected surface is xAI credential resolution and provider authentication during each embedded run attempt.

## Summary

Keep open: fresh user logs plus a follow-up A/B narrow this from OAuth refresh to a broader xAI auth-stage hot path, and current main still resolves provider credentials and probes provider runtime auth on each embedded attempt without live proof that the delay is gone.

## What This Changes

The issue reports 8-13s of xAI/Grok per-turn auth-stage latency on Windows for both cached OAuth and static API-key profiles while the model request itself is fast.

## Best Possible Solution

Add focused timing around getApiKeyForModel, prepareProviderRuntimeAuth, and xAI plugin lookup on the live Windows path, then make cached OAuth and static xAI credentials avoid repeated network or expensive plugin runtime work during per-turn auth.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. The issue has strong v2026.5.20 Windows logs and current source confirms the per-attempt auth work, but I did not run a live xAI/Grok turn or isolate the 8-10s subpath.

## Solution Assessment

Unclear: near-instant cached auth is the right target, but the follow-up shows static API-key profiles are slow too, so the best fix should profile and remove repeated auth-stage work rather than patch only OAuth refresh.
