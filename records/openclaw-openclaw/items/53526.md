Keep open: current main still uses the documented daily default, and this PR contains potentially useful product work, but it is not mergeable as submitted because it changes shared-channel retention defaults, targets the pre-split reset implementation, has a blocking guard bug, and lacks contributor-provided real behavior proof.

## What This Changes

The PR changes implicit group and thread session resets to a seven-day idle policy while keeping direct sessions on daily reset, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the retention change, implement the type-aware default in src/config/sessions/reset-policy.ts, preserve explicit base/type/channel overrides, add exact group/thread no-config and unrelated-override tests, and update the user-facing session docs plus changelog.

## Reproduction Assessment

Yes, as source-level behavior: current main still falls back to the daily default for unmatched group/thread reset policy, and the PR's unrelated resetByType guard bug is traceable from the diff. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction may be acceptable after maintainer approval, but this branch is not the best mergeable solution until it moves to reset-policy.ts, scopes the default check to the current reset type, pins the 10080-minute coverage, updates docs, and includes real behavior proof.
