## Summary

Keep open. The requested behavior is not implemented on current main or the latest release, but this PR should not merge as submitted because its default-mode guard misses configurations with unrelated type overrides and the behavior change needs maintainer privacy/product acceptance.

## What This Changes

The PR changes implicit group and thread session reset defaults from daily to idle with a 10080-minute window while keeping direct/DM sessions on daily reset, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the retention change, refresh the implementation in `src/config/sessions/reset-policy.ts`, keep explicit base/type/channel overrides authoritative, add exact 10080-minute coverage for group/thread defaults, and update docs plus changelog.

## Reproduction Assessment

Yes. A source-level trace of the PR diff shows `resetByType: { dm: ... }` makes `hasExplicitReset` true for a group policy, so the proposed group/thread idle default falls back to `daily`; current main also has tests that encode the existing daily behavior.

## Solution Assessment

No. The direction may be valid, but this PR is not the best fix as submitted because it has a blocking logic bug, targets a stale helper location, and changes default shared-channel retention without maintainer acceptance.

## Review Findings

  - body: The new `impliedDefaultMode` uses `!hasExplicitReset`, but `hasExplicitReset` becomes true when any `resetByType` entry exists.   - confidence: 0.93

  - confidence: 0.82
