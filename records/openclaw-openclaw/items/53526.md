## Summary

Keep open. Current main still uses the daily implicit reset default, while this PR is not safe to merge as submitted because its explicit-reset guard defeats the intended group/thread default in common override configs and the implementation targets a helper that has since moved.

## What This Changes

The PR changes implicit group and thread session reset defaults to idle with a 10080-minute window while keeping direct/DM sessions on daily reset, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the behavior change, implement the type-aware implicit default in `src/config/sessions/reset-policy.ts`, keep explicit base/type/channel overrides authoritative, add exact `10080` coverage for no-config and unrelated-override group/thread paths, and update the user-facing session docs plus changelog.

## Reproduction Assessment

Yes. A source-level trace of the PR diff shows `resetByType: { dm: ... }` makes `hasExplicitReset` true for a group policy, so the proposed group/thread idle default falls back to `daily`; current main also has tests that encode the existing daily behavior.

## Solution Assessment

No. The requested direction may be reasonable, but this implementation is not the best fix as submitted; a safer version should target `reset-policy.ts`, scope the explicit-reset check to the current type for the implied default, pin the 10080-minute test coverage, and include docs/changelog updates after maintainer acceptance.
