## Summary

Keep open: current main still has a source-clear restart-recovery/WebChat gap. Recovery resumes through an internal agent run, the runner can return the existing incomplete-turn error payload, but the chat final path only renders buffered assistant text or reloads persisted history, so a tool-only recovered turn can still end without a visible terminal error.

## What This Changes

The issue asks OpenClaw to show a clear Control UI/WebChat terminal error when restart recovery completes with no visible assistant payload after tool activity.

## Best Possible Solution

Bridge the existing incomplete-turn error payload into WebChat final/history visibility for restart-recovery agent runs, without changing provider payload handling, enabling broad deliver:true behavior, or duplicating normal Pi assistant turns.

## Reproduction Assessment

Yes, at source level: a restart-recovery agent run can return an incomplete-turn isError payload with no assistant delta, while WebChat final/history handling can reload to durable tool rows without a visible terminal error. I did not run a live Windows/custom-provider restart recovery scenario.

## Solution Assessment

Yes. The narrow maintainable fix is to surface or persist the existing incomplete-turn error at the Gateway/WebChat boundary for recovered runs, rather than changing provider fallback logic or redesigning checkpointing.
