## Summary

Keep open: current main still has a source-clear restart-recovery/WebChat path where a recovered run can terminate with an incomplete-turn error payload while Control UI receives or reloads only a final-without-message/tool-history shape. The related WebChat rendering and transcript-tail issues are adjacent, but this narrower restart-recovery terminal-error surfacing gap is not fully owned by a merged fix or canonical duplicate.

## What This Changes

The issue requests that restart recovery make an empty terminal recovered turn visibly fail in Control UI instead of leaving only tool blocks and no assistant/error text.

## Best Possible Solution

Make restart-recovery incomplete-turn failures produce a durable or final WebChat-visible error row using the existing incomplete-turn text, without redesigning provider payload extraction or checkpointing in this issue.

## Reproduction Assessment

Yes, for a source-level reproduction path; no live restart run was performed. Current main resumes with `deliver: false`, the runner can return an `isError` incomplete-turn payload for payloads=0, and the UI can process a final-without-message by reloading persisted tool history with no visible terminal error.

## Solution Assessment

Yes. The narrow maintainable fix is to surface or persist the existing incomplete-turn error at the Gateway/WebChat boundary for recovered runs, rather than broadening this into provider-specific payload handling or full restart checkpoint redesign.
