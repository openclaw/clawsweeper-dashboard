## Summary

This looks like a valid current-main runtime/UI bug rather than a stale provider payload report: the backend can synthesize the incomplete-turn error, but the restart-recovery/WebChat path has places where that error is not guaranteed to become a durable or visible chat-body message after tool history reload.

## What This Changes

Ensure restart-recovered main sessions that terminally produce no visible assistant payload show a clear Control UI error or final state instead of only tool rows.

## Best Possible Solution

Make restart-recovery incomplete-turn failures produce a durable or final WebChat-visible error row using the existing incomplete-turn text, while preserving the no-duplicate normal Pi assistant-turn boundary.

## Reproduction Assessment

No live current-main reproduction was executed, but source inspection gives a high-confidence source reproduction path: restart recovery launches an internal `agent` run, the runner returns an error payload for `payloads=0`, and WebChat final/history handling can drop that payload when no assistant text is persisted.

## Solution Assessment

Yes, the best direction is a narrow Gateway/WebChat repair that surfaces the existing incomplete-turn error for recovered runs; a broader checkpoint/restart redesign is not required for this specific user-visible gap.
