## Summary

Keep open: current main still has a source-clear gap where restart-recovery agent runs can return the existing incomplete-turn error payload while WebChat receives or reloads an empty final/tool-only history with no visible terminal error. The closed WebChat restart-delivery issue covers normal resumed replies, not this empty terminal payload surfacing path.

## What This Changes

The issue requests visible Control UI/WebChat error surfacing when restart recovery resumes a turn that terminally produces no assistant payload after tool activity.

## Best Possible Solution

Surface or persist the existing incomplete-turn error for restart-recovery agent runs at the Gateway/WebChat boundary, without turning WebChat into a generic outbound channel or duplicating normal Pi assistant turns.

## Reproduction Assessment

Yes, at source level: a restart-recovery `agent` run can return an incomplete-turn `isError` payload with no assistant delta, while WebChat final/history handling can reload to durable tool rows without a visible terminal error. No live Windows/custom-provider restart run was performed in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to bridge the existing incomplete-turn error into WebChat final/history visibility for recovered runs, rather than changing provider payload handling, setting broad `deliver: true`, or redesigning checkpointing.
