## Summary

Keep open: current main still has a source-clear restart-recovery/WebChat path where the runner can create an incomplete-turn error payload, while the Control UI can reload durable tool history after an empty final event and leave no visible terminal error. The related WebChat long-history and transcript-tail issues are adjacent, but neither owns this narrower restart-recovery error-surfacing gap.

## What This Changes

The issue asks Control UI/WebChat to visibly surface a terminal incomplete-turn error after restart recovery resumes a tool-using main session that ends with no assistant payload.

## Best Possible Solution

Make restart-recovery incomplete-turn failures produce a durable or final WebChat-visible error row using the existing incomplete-turn text, without duplicating normal Pi assistant-turn persistence.

## Reproduction Assessment

Yes for a source-level path; no live run was performed. Restart recovery calls an internal agent run, the runner can return an `isError` incomplete-turn payload for `payloads=0`, and WebChat final/history handling can reload to durable tool rows without a visible error message.

## Solution Assessment

Yes. The narrow maintainable fix is to surface or persist the existing incomplete-turn error for recovered runs at the Gateway/WebChat boundary, not to redesign checkpointing or provider payload handling for this issue.
