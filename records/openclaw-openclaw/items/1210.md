## Summary

Keep open: current main still has mitigation layers, but not the durable session-transcript claim-check requested here. Discord media is saved to disk, then hydrated into base64 `image` blocks for the Pi prompt, and the guarded persistence path still appends user messages without rewriting those blocks to compact media references; the issue discussion also includes live Discord forensic proof from v2026.3.24.

## What This Changes

Store Discord inbound image references in session transcripts instead of raw base64 image blocks, then hydrate bounded references from disk when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content, using compact media reference metadata for new writes and bounded hydration for model context while preserving legacy inline image transcript reads.

## Reproduction Assessment

Yes, at source level: Discord stores inbound media on disk, the runner hydrates it into base64 `image` blocks, upstream Pi appends those blocks to the user message, and the OpenClaw persistence guard writes the final user message unchanged. I did not run a live Discord DM, but the issue discussion includes a concrete v2026.3.24 transcript and overflow report.

## Solution Assessment

Yes. The maintainable fix is at the transcript persistence/hydration boundary; chat-history display redaction, replay-time pruning, or media-size rejection alone does not solve durable JSONL bloat or duplicate storage.
