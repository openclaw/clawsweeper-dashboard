## Summary

Keep open: current main has replay pruning and Gateway chat-send media-field mitigations, but it still lacks a storage-level claim-check in the shared session persistence path. Discord media attachments are saved to disk and exposed through `MediaPath`/`MediaPaths`, then hydrated into current-turn image blocks before `activeSession.prompt`; the guarded append path can still persist those image blocks unchanged.

## What This Changes

Replace inline base64 image blocks in session transcripts with compact media references and hydrate them from disk when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata, hydrate only bounded current/recent references for model context, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes. A focused unit reproduction can append a user message containing `{ type: "image", data, mimeType }` through the guarded session manager and verify the persisted JSONL still contains the image block; the current Discord media-note-to-image hydration path explains how real Discord turns reach that state.

## Solution Assessment

Yes. The requested claim-check is the narrow maintainable fix because replay pruning and Gateway/UI redaction reduce later blast radius but do not remove the raw base64 from stored session transcripts.
