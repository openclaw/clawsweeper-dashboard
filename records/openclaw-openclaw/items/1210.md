## Summary

Keep open: current main still has replay-time image pruning, Gateway/media offload, and display-size protections, but it does not implement the requested storage-level claim-check for Discord/user image blocks before session JSONL persistence. The issue discussion includes a concrete March 27 reproduction on v2026.3.24, and the current source still leaves ordinary user image content to pass through the guarded append path unless a caller or plugin hook rewrites it.

## What This Changes

Replace inline base64 image blocks in session transcripts with durable media references, then hydrate bounded image data only when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata instead of raw `image.data`, hydrate only bounded recent references for model context, and keep existing inline-base64 transcripts readable.

## Reproduction Assessment

Yes. A focused unit reproduction is to append a user message containing `{ type: "image", data, mimeType }` through the guarded session manager and assert the persisted message still contains the inline data; the March 27 Discord forensics provide a practical production reproduction.

## Solution Assessment

Yes. The requested claim-check is the narrow maintainable fix for transcript bloat and double storage; replay pruning and `chat.history` redaction are useful mitigations but do not solve the on-disk JSONL storage bug.
