## Summary

Keep open: current main still has mitigation and adjacent media-reference paths, but the central session transcript persistence path does not claim-check inbound user image blocks before JSONL persistence.

## What This Changes

The issue requests replacing inline base64 image payloads in Discord/inbound session transcripts with durable media references that are hydrated only when needed for model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata, hydrate only bounded current/recent references for model input, and keep legacy inline-image transcripts readable.

## Reproduction Assessment

Yes, from source inspection. A focused reproduction is to append a user message containing `{ type: "image", data, mimeType }` through the guarded session manager and verify the persisted message still contains `data`; I did not execute it because this review is read-only.

## Solution Assessment

Yes. A storage-level claim-check in the persistence/context-hydration seam is the narrow maintainable fix; replay pruning and `chat.history` redaction reduce symptoms but do not remove raw base64 from the stored transcript path.
