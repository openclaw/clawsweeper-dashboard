## Summary

Keep open: current main still has replay-time pruning, display projection, and Gateway chat-send media offload support, but the Discord session persistence path still does not replace user image content blocks with durable media references before JSONL append.

## What This Changes

Store Discord image references in session transcripts instead of persisting inline base64 image data.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata, hydrate bounded recent references for model context, and keep old inline-image transcripts readable.

## Reproduction Assessment

Yes from source inspection. A focused reproduction is to append a user message containing `{ type: "image", data, mimeType }` through the guarded session manager and verify the persisted message is unchanged; I did not run it because this review is read-only.

## Solution Assessment

Yes. The storage-level claim-check is the narrow maintainable fix; replay pruning, display redaction, and chat-history payload caps reduce symptoms but do not remove raw base64 from the persisted transcript.

## Repair Work Prompt

Fix the remaining inline-image transcript storage bug for https://github.com/openclaw/openclaw/issues/1210. Implement a backward-compatible claim-check for inbound user image content so new session JSONL writes persist compact media reference metadata instead of raw base64 `image.data`, while preserving model-context hydration for current/recent images and keeping legacy inline-base64 transcripts readable. Start at the guarded session persistence path and Discord/Gateway media-context seams; reuse the existing media store / `media://inbound/<id>` style where appropriate. Add regression coverage proving a user message with `{ type: "image", data, mimeType }` does not write the base64 payload to the transcript, and that replay/context construction can still hydrate or safely prune the media reference. Coordinate with the display-only chat.history behavior from #43982, but do not turn this into a UI-only redaction. Do not add a config knob, change media retention policy, or weaken existing size/SSRF/media-root checks. Add a user-facing `CHANGELOG.md` fix entry unless the final patch is tests-only.
