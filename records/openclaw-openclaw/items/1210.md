## Summary

Keep open: current main still mitigates replay, display, and large Gateway attachment cases, but it does not persist inbound user image blocks as durable media references. The guarded session append path can still write `type: "image"` content with inline `data` into JSONL when images are passed to the agent prompt.

## What This Changes

The issue asks OpenClaw to persist Discord image transcript entries as media references instead of inline base64 payloads.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content using the existing media store and `media://inbound` patterns, hydrate only bounded current or recent references during model-context construction, and keep old inline-base64 transcripts readable.

## Reproduction Assessment

Yes, at source level: a channel turn with images reaches `AgentSession.prompt(..., { images })`, the dependency appends those images to user content, and OpenClaw's guarded append path does not rewrite image blocks before persistence. I did not establish a live Discord failure on current main.

## Solution Assessment

Yes. A storage-layer claim-check at session persistence plus bounded replay hydration is the narrow maintainable fix; Gateway display redaction and replay pruning reduce symptoms but do not change the durable JSONL boundary.
