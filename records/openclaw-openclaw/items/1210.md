## Summary

Keep open: current main still mitigates inline image payloads with replay-time pruning, redaction, and scoped Gateway chat-send media rewrites, but it does not implement the requested session-transcript media-reference claim-check for Discord/inbound user image blocks.

## What This Changes

Request storing image references/metadata in session transcripts instead of raw base64 image payloads, then hydrating bounded image data from disk for model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata, hydrate only bounded recent references for model context, and keep legacy inline-image transcripts readable.

## Reproduction Assessment

Yes from source inspection. A focused reproduction is to append a user message containing `{ type: "image", data, mimeType }` through the guarded session manager and verify the persisted message still contains the image data; I did not execute it because this review is read-only.

## Solution Assessment

Yes. The storage-level claim-check is the narrow maintainable fix; replay pruning, text redaction, and chat.history/chat.send projections reduce symptoms but do not remove raw base64 from the persisted Discord/inbound transcript path.
