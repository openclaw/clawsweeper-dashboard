## Summary

Keep open: current main still mitigates and displays some image payloads more safely, but it does not implement the durable session-transcript claim-check requested here for Discord inbound images.

## What This Changes

The issue requests replacing inline base64 image blocks in Discord session transcripts with compact media references that are hydrated from disk for model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata for new writes, hydrate bounded recent references for model context, and preserve legacy inline-base64 transcript reads.

## Reproduction Assessment

Yes, at source level: Discord stores inbound media, prompt assembly hydrates it into base64 image blocks, and the guarded persistence path still appends the final user message without an image claim-check rewrite. I did not run a live Discord DM, but the issue includes concrete v2026.3.24 live transcript forensics.

## Solution Assessment

Yes. The durable fix belongs at the transcript persistence and hydration boundary; replay pruning, UI display redaction, or media-size rejection alone does not solve JSONL bloat or duplicate storage.
