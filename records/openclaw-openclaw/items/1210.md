## Summary

Keep open: current main still mitigates old image replay and Gateway display bloat, but it does not implement the requested durable transcript claim-check for inbound image content. The Discord/media path can still hydrate saved media into `type: "image"` blocks for the Pi prompt, and the session guard persists user messages without rewriting those image blocks to references.

## What This Changes

The issue requests replacing inline base64 image payloads in session transcripts with durable media references and bounded hydration.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata instead of raw `image.data`, hydrate bounded recent references during context construction, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes, at source level: Discord media becomes `MediaPaths`, the prompt prelude turns those into media notes, the runner hydrates them into image blocks, and Pi persists the prompt user message. I did not run a live Discord DM against current main.

## Solution Assessment

Yes. A persistence-layer claim-check with bounded hydration is the narrow maintainable fix; display-only redaction, replay pruning, or immediate image deletion do not solve the durable JSONL storage boundary safely.
