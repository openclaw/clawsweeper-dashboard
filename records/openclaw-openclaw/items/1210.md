## Summary

Keep open: current main still lacks a durable transcript claim-check for inbound/user image blocks; replay pruning, display projection, and Gateway attachment offload reduce adjacent symptoms but do not replace persisted `image.data` in session JSONL.

## What This Changes

The issue requests storing compact media references for Discord/inbound images in session transcripts instead of inline base64, then hydrating those references for model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound/user image content: persist compact media reference/path/metadata instead of raw `image.data`, hydrate bounded references for model replay, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes, source-reproducible: append a guarded user message containing a `{ type: "image", data: ... }` block and current main will pass it through to `SessionManager.appendMessage`. The live Discord path was not rerun here, but the reporter's v2026.3.24 JSONL/log forensics match the inspected source path.

## Solution Assessment

Yes. A storage-layer claim-check is the narrowest maintainable direction because it fixes the durable transcript boundary; channel-only handling, replay pruning, or UI display redaction leave the JSONL bloat path intact.
