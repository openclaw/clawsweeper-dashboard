## Summary

Keep open: current main has Discord media saved to disk plus replay/display pruning, but the session persistence path still allows user image content blocks with inline base64 data to be appended to JSONL transcripts. The related open Gateway history PR is adjacent display cleanup, not the requested transcript storage claim-check.

## What This Changes

The issue asks OpenClaw to replace inline base64 image blocks in session transcripts with compact persisted media references and hydrate them only when building bounded model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound images: persist compact media reference/path/metadata instead of raw image data, hydrate only bounded recent references for model context, keep legacy inline-base64 transcripts readable, and keep the core seam channel-agnostic.

## Reproduction Assessment

Yes, from source inspection and the reported Discord transcript forensics, though I did not execute a live Discord DM. Discord media is saved as media paths while current images are still passed into AgentSession.prompt and the dependency persists those user image blocks unchanged unless OpenClaw rewrites them first.

## Solution Assessment

Yes, the requested claim-check is the narrowest maintainable fix for the storage bug. Replay pruning and chat-history display redaction reduce symptoms but do not stop raw image bytes from entering the durable transcript.
