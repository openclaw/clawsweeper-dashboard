## Summary

Keep open: current main still has replay-time pruning, text/display cleanup, Gateway/WebChat media offload paths, and Discord media saving, but it does not add the requested storage-layer claim-check before session JSONL persistence. Inline user image blocks can still be passed to the upstream session prompt and written through the guarded append path without rewriting `image.data`.

## What This Changes

The issue requests replacing inline base64 image payloads in session transcripts with compact media references that can be hydrated when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound/user image content: persist compact media reference and metadata instead of raw `image.data`, hydrate only bounded recent references during model-context construction, keep legacy inline-base64 transcripts readable, and coordinate UI display projection with #43982.

## Reproduction Assessment

Yes. The issue discussion gives concrete Discord DM forensics, and current-main source inspection shows the same path still hydrates images as inline base64 and passes them to upstream prompt/session persistence.

## Solution Assessment

Yes. A central persistence-layer claim-check is the narrowest maintainable direction because replay pruning and Gateway history projection reduce symptoms but do not remove duplicate inline base64 from stored transcripts.
