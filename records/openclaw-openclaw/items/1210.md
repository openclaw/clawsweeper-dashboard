## Summary

Keep open: current main has replay-time pruning, media-note hydration, and Gateway attachment offload paths, but it still does not implement the requested storage-level claim-check for session transcripts. User image blocks can still reach `SessionManager.appendMessage` without an OpenClaw persistence transform that replaces inline `image.data` with a compact media reference.

## What This Changes

Store inbound Discord/image content in session transcripts as compact media references instead of inline base64 image blocks, then hydrate bounded references during context construction.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata, hydrate only bounded recent references for model context, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes. A focused unit reproduction is to append a user message containing `{ type: "image", data, mimeType }` through the guarded session manager and verify the current persistence path writes the block unchanged; the docs and replay-prune test also confirm raw transcripts are not rewritten.

## Solution Assessment

Yes, the requested claim-check direction is the narrow maintainable fix. Replay pruning and UI redaction are useful mitigations, but they do not solve transcript storage bloat or double storage.
