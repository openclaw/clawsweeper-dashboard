## Summary

Keep open: current main has media offload, replay-time pruning, and display redaction paths, but it still lacks the requested transcript-level claim-check for inbound image blocks before session JSONL persistence.

## What This Changes

Replace inline base64 image blocks in session transcripts with durable media references and hydrate bounded image data later when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check: persist compact media reference metadata instead of raw `image.data`, hydrate only bounded recent references during context construction, and keep legacy inline-image transcripts readable.

## Reproduction Assessment

Yes. The code path is traceable on current main: Discord saves inbound media, the runner hydrates media refs into base64 `ImageContent`, and `activeSession.prompt(..., { images })` runs without a persistence transform that rewrites image blocks.

## Solution Assessment

Yes. The requested claim-check is the narrow maintainable direction; replay-time pruning and display redaction are useful mitigations but do not fix the duplicated on-disk transcript storage.
