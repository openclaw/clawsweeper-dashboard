## Summary

Keep open: current main has replay-time pruning, Gateway media offload, and UI/history redaction mitigations, but it still does not implement the requested storage-level claim-check for user image blocks in session JSONL transcripts.

## What This Changes

Store Discord and other inbound image transcript content as compact media references instead of inline base64, then hydrate bounded images when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound user image content: persist media reference metadata, hydrate only bounded recent references for model context, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes. A focused reproduction is to append or run a user turn containing `{ type: "image", data, mimeType }` through the guarded Pi session path and verify the current transcript entry still contains `image.data`; the source and existing replay-prune test show this path remains present.

## Solution Assessment

Yes. The requested claim-check is the narrow maintainable fix; replay pruning and Gateway/UI redaction reduce symptoms but do not remove duplicate base64 storage from session transcripts.
