## Summary

Keep open: current main still has mitigation layers, but not the requested durable transcript claim-check for inbound Discord images. Source inspection shows Discord attachments are saved as media paths, then hydrated into `type: "image"` base64 blocks for the Pi prompt, and the session persistence guard appends non-tool user messages without rewriting those image blocks to compact media references.

## What This Changes

The issue asks OpenClaw to persist inbound Discord image references in session transcripts instead of storing base64 image payloads inline.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata, hydrate bounded recent references during context construction, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes, at source level: Discord media is saved to disk, rendered into prompt media notes, hydrated back into base64 image blocks, and then persisted by the Pi session manager as user content. I did not run a live Discord DM in this read-only review, but the issue discussion includes prior live Discord forensic proof.

## Solution Assessment

Yes. The narrow maintainable fix is a persistence-layer claim-check with bounded hydration; chat-history display redaction, replay pruning, or immediate media deletion would not solve the durable JSONL storage boundary.
