## Summary

Keep open: current main still has replay/display/offload mitigations, but it does not replace persisted user/inbound `image.data` blocks with durable transcript media references. The exact-scope fix attempts at https://github.com/openclaw/openclaw/pull/42941 and https://github.com/openclaw/openclaw/pull/50914 are closed unmerged, and the open Gateway history PR at https://github.com/openclaw/openclaw/pull/43982 is a display projection change, not JSONL persistence.

## What This Changes

Requests storing inbound Discord/user images in session transcripts as media references instead of inline base64 payloads, then hydrating them for model context as needed.

## Best Possible Solution

Implement a backward-compatible transcript claim-check for user/inbound image blocks using the existing media store and `media://inbound`/MediaPath patterns, hydrate bounded current or recent references for model replay, and leave legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes, at source level: a user message with `content[].type === "image"` reaches the guarded `SessionManager.appendMessage` path, and current main does not rewrite that block before persistence. I did not rerun a live Discord DM, but the inspected path matches the reporter's v2026.3.24 JSONL forensics.

## Solution Assessment

Yes. A storage-layer claim-check is the narrowest maintainable fix because replay pruning, Gateway offload, and chat history redaction reduce adjacent symptoms but still leave the durable JSONL boundary unresolved.
