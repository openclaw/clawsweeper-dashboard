## Summary

Keep open: current main still allows user image content blocks with inline base64 to reach the JSONL append path, while replay/display pruning and media notes only reduce downstream symptoms. The issue has a clear storage boundary and should be fixed in a focused persistence/context-hydration change.

## What This Changes

Request to store Discord and other inbound images in session transcripts as media references instead of inline base64 payloads, then hydrate them from disk when constructing model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for image content: persist compact media reference/path/metadata instead of raw `image.data`, hydrate only bounded recent references during model-context construction, and keep legacy inline-base64 transcripts readable.

## Reproduction Assessment

Yes. A focused reproduction can append a guarded user message containing an image content block and observe that current main would pass the block through to `SessionManager.appendMessage`; the live Discord DM path was not run here.

## Solution Assessment

Yes. A generic persistence-layer claim-check is narrower and more maintainable than channel-specific Discord handling or display-only redaction, because the upstream dependency persists the final user message object unchanged.
