## Summary

Close as ClawHub scope: this PR hardcodes a community Adaptive Cards marker renderer into core iOS/macOS chat, while current OpenClaw policy and companion PR decisions route this plugin-specific capability to ClawHub or a future generic rich-message contract.

## What This Changes

Adds Swift Adaptive Card models, marker parsing, SwiftUI rendering in shared iOS/macOS chat bubbles, and a community plugin docs listing.

## Best Possible Solution

Keep Adaptive Cards in the external ClawHub/plugin path, and pursue any native client rendering through a generic MessagePresentation, A2UI, or plugin API extension that applies across clients.

## Reproduction Assessment

Yes for the PR review blockers by source inspection: the PR head decodes schema-incompatible widths, parses markers before thinking visibility filtering, and renders URL/image payloads from card JSON. There is no separate current-main bug to reproduce because native Adaptive Card marker support is not a documented iOS/macOS contract.

## Solution Assessment

No. A platform-specific core parser for a community plugin's HTML markers is not the best maintainable shape; ClawHub/plugin distribution plus a generic rich-message or A2UI contract is the safer product direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode string-or-number column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow column widths to be strings or numbers, but `Column.width` and `TableColumnDefinition.width` are `String?`. Valid weighted numeric widths can make `JSONDecoder` reject the card and fall back to marker text instead of rendering.
  - confidence: 0.95
- **[P2] Parse cards after thinking visibility filtering:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:622-623`
  - body: This parses adaptive-card markers from raw assistant text before `AssistantTextParser` applies the `includesThinking` policy. A card inside hidden `<think>` content can still be decoded and rendered.
  - confidence: 0.91
- **[P2] Honor column widths during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:108-116`
  - body: `renderColumnSet` never reads the decoded `width` value and gives every column `.frame(maxWidth: .infinity)`, so `auto`, `stretch`, and weighted layouts render as equal columns.
  - confidence: 0.92
- **[P2] Render fallback for invalid image URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-156`
  - body: When `URL(string:)` fails, the image branch renders nothing and drops any `altText`. Add the same placeholder/fallback used for load failures so malformed URLs do not silently remove card content.
  - confidence: 0.86
- **[P3] Use a width-based ImageSet grid minimum:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:239`
  - body: `renderImageSet` passes `imageMaxHeight(size)` as the adaptive grid's minimum column width. That helper returns height limits, so grid density is driven by the wrong dimension.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces native URL-opening and remote-image-fetch behavior from plugin/model-authored card JSON without a documented client policy gate.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed to SwiftUI `openURL`. Without an allowlist, confirmation, or documented policy, cards can drive arbitrary URL schemes or deep links when tapped.

  - body: Card image URLs are rendered with `AsyncImage`, so a card producer can cause native clients to request arbitrary remote URLs during chat rendering, unlike the current markdown path that flattens remote images into text.
