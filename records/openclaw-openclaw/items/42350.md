## Summary

Keep open. Current main still lacks iOS/macOS Adaptive Card marker parsing/rendering, and this PR branch still has schema, layout, visibility, and client security policy issues; the linked external Adaptive Cards plugin context is useful but does not make the branch safe to close or merge as-is.

## What This Changes

The PR adds Swift Adaptive Card models, marker parsing, inline SwiftUI rendering in shared iOS/macOS chat UI, and a community plugin docs entry for the external Adaptive Cards plugin.

## Best Possible Solution

If maintainers want native app support, land it behind a documented rich-message contract or marker format, preserve thinking visibility rules, decode schema-compatible widths, respect layout semantics, gate URL/image behavior, add focused Swift tests, and include the required changelog/docs updates.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode string-or-number widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow weighted numeric widths such as `"width": 2`, but Column and TableColumnDefinition model `width` as `String?`. Swift Codable treats numeric values as type mismatches, so otherwise valid cards can fail to parse and fall back to raw marker text.
  - confidence: 0.94
- **[P2] Parse cards after filtering thinking:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw assistant text before the existing AssistantTextParser visibility policy runs. When thinking is hidden, a card embedded in `<think>` content can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.9
- **[P2] Use width when laying out columns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: The ColumnSet renderer gives every column an infinite-width frame and never reads the decoded width value. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Render fallback for invalid image URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-156`
  - body: The image renderer only renders inside the URL parse success branch. If the URL string is malformed, the image element and its alt text disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623-640`
  - body: This wires a user-facing iOS/macOS chat feature into the assistant message view, but the PR does not update CHANGELOG.md. Repo policy requires user-facing feat/fix/perf changes to include a changelog entry before merge.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces native handling of plugin/model-supplied remote images and URL actions without explicit client policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: Action.OpenUrl values come from assistant/plugin-emitted card JSON and are passed to SwiftUI openURL. Without an allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: Card image URLs are rendered with AsyncImage, so a card producer can cause client-side requests to arbitrary remote URLs. That has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.86
