## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still lacks inline iOS/macOS Adaptive Card rendering, so the contribution is not obsolete; however, the branch introduces blocking schema/rendering correctness issues and a security-sensitive native URL/media surface that needs explicit product and security direction.

## What This Changes

The PR adds Swift Adaptive Card models, marker parsing, inline SwiftUI rendering in shared iOS/macOS chat UI, and a community plugin docs entry for the external Adaptive Cards plugin.

## Best Possible Solution

If maintainers want native app support, the mergeable end state should consume a maintained rich-message contract or a clearly documented marker format, preserve thinking visibility rules, decode schema-compatible string-or-number widths, respect column/table layout semantics, gate URL schemes and remote image fetching, and keep third-party plugin discovery aligned with ClawHub unless explicitly approved.

## Reproduction Assessment

Yes. A valid Adaptive Card using a numeric column width such as `"width": 2` exercises the String-only decode failure, and a marker inside `<think>...</think>` exercises the parse-before-visibility path; the layout and image fallback issues are directly visible from the PR diff.

## Solution Assessment

No. Native iOS/macOS rendering may be a valid direction, especially with the linked external plugin emitting markers, but this branch is not the best merge path until it aligns with the current visibility, security, schema, and presentation-contract policies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric Adaptive Card widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow weighted numeric widths such as `"width": 2`, but the PR models column and table widths as `String?`. Swift decoding treats those payloads as type mismatches, so otherwise valid cards can fail to parse instead of rendering inline.
  - confidence: 0.94
- **[P2] Filter thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623-625`
  - body: The PR parses adaptive-card markers from raw assistant text before the existing AssistantTextParser visibility policy runs. When thinking is hidden, a card embedded in `<think>` content can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.9
- **[P2] Honor Adaptive Card column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: The ColumnSet renderer gives every column an infinite-width frame and never reads the decoded width value. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Show fallback for invalid image URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-156`
  - body: The image renderer only renders inside the URL parse success branch. If the URL string is malformed, the image element and its alt text disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces native handling of plugin/model-supplied remote images and URL actions without explicit client policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: Action.OpenUrl values come from assistant/plugin-emitted card JSON and are passed to SwiftUI openURL. Without an allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: Card image URLs are rendered with AsyncImage, so a card producer can cause client-side requests to arbitrary remote URLs. That has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.86
