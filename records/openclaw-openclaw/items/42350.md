## Summary

Keep open. Current main still does not implement inline iOS/macOS Adaptive Card rendering, and the linked external plugin provides useful producer-side context, but this branch is not merge-ready because the diff still has schema, thinking-visibility, layout, changelog, and URL/image policy blockers.

## What This Changes

The PR adds Swift Adaptive Card models, marker parsing, inline SwiftUI rendering in shared iOS/macOS chat UI, and a community docs entry for the Adaptive Cards plugin.

## Best Possible Solution

If maintainers want native app support, land it through a documented rich-message or presentation contract with schema-compatible parsing, thinking filtering, URL/image policy, Swift tests, docs, and changelog coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection plus the Adaptive Cards Column docs show a valid card with `"width": 2` hits the PR's String-only width model, and a marker inside `<think>...</think>` is parsed before the current thinking filter runs.

## Solution Assessment

Is this the best way to solve the issue? No. Native rendering may be a valid direction, especially with the external Adaptive Cards plugin, but the branch should first align with the shared presentation/rich-message contract and define client URL/image policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:44`
  - body: Adaptive Cards allow weighted widths as numbers such as `"width": 2`, but this model decodes `Column.width` as `String?`. Swift Codable treats numeric values as a type mismatch, so otherwise valid ColumnSet cards can fail to parse and fall back to raw marker text.
  - confidence: 0.94
- **[P2] Filter thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw assistant text before the existing AssistantTextParser visibility policy runs. When thinking is hidden, a card embedded in `<think>` content can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.9
- **[P2] Honor column widths in layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: The ColumnSet renderer gives every column an infinite-width frame and never reads the decoded width value. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Show image fallback for invalid URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138`
  - body: The image renderer only renders inside the URL parse success branch. If the URL string is malformed, the image element and its alt text disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623-640`
  - body: This wires a user-facing iOS/macOS chat feature into the assistant message view, but the PR does not update CHANGELOG.md. Repo policy requires user-facing feat/fix/perf changes to include a changelog entry before merge.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces native handling of plugin/model-supplied remote images and URL actions without explicit client policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: Action.OpenUrl values come from assistant/plugin-emitted card JSON and are passed to SwiftUI openURL. Without an allowlist, confirmation step, or documented policy, cards can trigger arbitrary URL schemes or native deep links when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: Card image URLs are rendered with AsyncImage, so a card producer can cause client-side requests to arbitrary remote URLs. That has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.86
