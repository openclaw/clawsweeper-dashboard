## Summary

Keep this PR open. Current main still does not implement native iOS/macOS Adaptive Card rendering, and the branch has concrete schema, visibility, layout, and client URL/image policy issues that should block merge until maintainers decide the native rendering direction.

## What This Changes

The PR adds Swift Adaptive Card models, marker parsing, inline SwiftUI rendering in shared iOS/macOS chat UI, and a community plugin docs entry for `@vikrantsingh01/openclaw-adaptive-cards`.

## Best Possible Solution

If maintainers want native app support, the mergeable end state should consume a maintained rich-message contract or clearly documented marker format, preserve thinking visibility rules, decode Adaptive Cards schema-compatible string-or-number widths, respect column/table layout semantics, gate URL schemes and remote image fetching, and keep third-party plugin discovery aligned with ClawHub unless explicitly approved.

## Reproduction Assessment

Yes. A valid Adaptive Card with `"width": 2` in a ColumnSet exercises the branch's `String?` decoding failure, and a card marker inside `<think>...</think>` exercises the raw-text parse path before visibility filtering.

## Solution Assessment

No. The native rendering direction may be useful, especially with the linked external plugin emitting markers, but this branch is not the best merge path until it aligns with existing visibility, security, and semantic presentation policies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric Adaptive Card widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow weighted numeric column widths such as `"width": 2`, but this model declares column and table widths as `String?`. Swift Codable treats those valid payloads as type mismatches, so otherwise valid cards can fail to parse instead of rendering inline.
  - confidence: 0.94
- **[P2] Filter hidden thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623-625`
  - body: The branch parses adaptive-card markers from raw assistant text before the existing `AssistantTextParser` visibility policy runs. When thinking is hidden, a card embedded in `<think>` content can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.9
- **[P2] Honor Adaptive Card column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: `renderColumnSet` gives every column an infinite-width frame and never reads the decoded width value. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Render fallback for invalid image URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-156`
  - body: `renderImage` only renders inside the `URL(string:)` success branch. If the URL string cannot be parsed, the image element and its `altText` disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82
- **[P3] Encode unsupported action data explicitly:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:266-283`
  - body: `AnyCodableAction.encode(to:)` has no final fallback after the handled value types. If an unsupported value reaches the wrapper through `init(_:)`, encoding can complete without writing a value, which silently corrupts action payloads.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The diff introduces native handling of plugin/model-supplied remote images and URL actions without explicit client policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed to SwiftUI `openURL`. Without an allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: `AsyncImage` fetches card image URLs directly from the native app as soon as the card renders. A card producer can cause client-side requests to arbitrary remote URLs, which has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.86
