## Summary

Keep open. Current main still lacks iOS/macOS inline Adaptive Card rendering, but this PR has unresolved correctness findings, a maintainer changes-requested review, and native URL/image policy questions that need product/security review before merge.

## What This Changes

The PR adds Swift Codable Adaptive Card models, marker parsing, SwiftUI rendering in shared OpenClawChatUI, chat-message wiring for `<!--adaptive-card-->` payloads, and an Adaptive Cards community plugin docs entry.

## Best Possible Solution

Define the native rich-message contract and client security policy first, then land a rebased Swift implementation that preserves thinking visibility, accepts schema-valid widths, honors layout/fallback semantics, removes stale plugin-listing docs, and includes focused Swift coverage plus the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a card with numeric `"width": 2` exercises the `String?` decode failure, and a marker inside hidden `<think>` content exercises the parse-before-visibility path.

## Solution Assessment

No. Native iOS/macOS rendering may be a valid direction, but this branch is not the best merge shape until it aligns with schema compatibility, thinking visibility, URL/image policy, current docs policy, tests, and changelog requirements.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode string-or-number column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow weighted numeric widths such as `"width": 2`, but `Column.width` and `TableColumnDefinition.width` are declared as `String?`. Swift decoding treats numeric values as a type mismatch, so otherwise valid cards can fail parsing and fall back to raw marker text.
  - confidence: 0.95
- **[P2] Parse cards after filtering thinking:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw assistant text before the existing `AssistantTextParser` visibility policy runs. When thinking is hidden, a card embedded in `<think>` content can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.91
- **[P2] Honor column widths during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: The ColumnSet renderer gives every column an infinite-width frame and never reads the decoded `width` value. Cards using `auto`, `stretch`, pixel, or weighted widths render as equal columns, breaking common Adaptive Card layouts.
  - confidence: 0.92
- **[P2] Render fallback for invalid image URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-156`
  - body: The image renderer only emits a view when `URL(string:)` succeeds. If the URL string is malformed, the image element and alt text disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623-640`
  - body: This wires a user-facing iOS/macOS chat feature into assistant message rendering, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing feature changes to include a changelog entry before merge.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff needs security review because model/plugin-authored card JSON can fetch remote images and open native URL schemes without a documented client policy gate.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed to SwiftUI `openURL`. Without an allowlist, confirmation, or documented policy, cards can drive arbitrary URL schemes or deep links when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: Card image URLs are rendered with `AsyncImage`, so a card producer can cause native clients to request arbitrary remote URLs during chat rendering. That has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.87
