## Summary

Keep open for normal PR review: this is a small user-facing ClawHub diagnostics feature, but the patch has a reset-header formatting bug and the PR carries the protected `maintainer` label.

## What This Changes

The PR centralizes ClawHub HTTP error construction so 429 search/download failures include rate-limit header details and an unauthenticated sign-in hint, with focused ClawHub helper tests.

## Best Possible Solution

Keep the shared error helper, but make reset formatting header-aware and add the required changelog entry before landing.

## Reproduction Assessment

Not applicable as a feature PR rather than a reported bug. The blocking review finding is source-reproducible: a 429 carrying only `X-RateLimit-Reset: 1777820460` would be rendered as `resets in 1777820460s`.

## Solution Assessment

No. The shared helper is the right direction, but the reset value needs header-aware formatting before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Parse legacy reset headers before formatting:** `src/infra/clawhub.ts:617-619`
  - body: Live ClawHub responses expose `X-RateLimit-Reset` as an absolute Unix timestamp while `RateLimit-Reset` is relative. If a 429 only carries the legacy header, this fallback will render an epoch as seconds, so parse the legacy value as a timestamp or omit it from the relative message.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `src/infra/clawhub.ts:591-595`
  - body: This PR changes user-facing ClawHub error output and is marked as a feature, but the diff only updates implementation and tests. Repo policy requires a `CHANGELOG.md` entry for user-facing feat/fix/perf changes.
  - confidence: 0.9
