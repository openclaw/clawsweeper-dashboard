## Summary

Close as a ClawHub/plugin-promotion item: this PR adds a service-specific external channel discovery entry to OpenClaw core, while current policy routes plugin discovery, official publisher status, provenance, and security review through ClawHub. The existing DingTalk connector is already published externally, and the proposed core catalog entry is also not merge-ready because it uses a floating npm install source and links to docs for a different DingTalk plugin.

## What This Changes

The PR adds a DingTalk external channel entry to `scripts/lib/official-external-channel-catalog.json` so `openclaw configure` can discover and install `@dingtalk-real-ai/dingtalk-connector`.

## Best Possible Solution

Publish and promote the DingTalk connector through ClawHub, and open a narrow SDK/API issue only if the external plugin path exposes a concrete missing OpenClaw extension surface.

## Reproduction Assessment

Not applicable; this PR requests catalog discovery/promotion for an external DingTalk channel, not a broken existing behavior. I inspected the diff, current catalog policy tests, plugin docs, and the published package metadata instead of running a bug repro.

## Solution Assessment

No; adding a service-specific external plugin to the core official catalog is not the best default path under current ClawHub/plugin-promotion policy. If maintainers intentionally make a core catalog exception, the entry still needs an exact npm version, `expectedIntegrity`, matching docs, and real configure/install proof.
