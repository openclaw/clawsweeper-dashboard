## Summary

Keep open: current main still has an unguarded `nodes` sub-CLI plugin-registration path, so `openclaw nodes list --json` can emit plugin startup diagnostics on stdout before the JSON payload.

## What This Changes

The issue asks for `--json` CLI commands, especially `openclaw nodes list --json`, to keep stdout pure JSON by routing startup/plugin diagnostics to stderr.

## Best Possible Solution

Route the `nodes` sub-CLI plugin-registration path through the same temporary stderr guard used by main lazy plugin registration, and add regression coverage that proves `nodes list --json` stdout is parseable when plugin startup logs are emitted.

## Reproduction Assessment

Yes. The reporter provided an exact command and current source shows a pre-preAction `nodes` plugin-registration path that can write diagnostics before JSON stdout; I did not execute the Helm/container flow in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to apply the existing temporary stderr routing contract to the `nodes` registration path, not to special-case LCM output or suppress plugin logs globally.

## Repair Work Prompt

Fix the CLI JSON stdout pollution reported in https://github.com/openclaw/openclaw/issues/84293. Current main already guards the main lazy plugin-registration path in `src/cli/run-main.ts`, but `src/cli/nodes-cli/register.ts` calls `registerPluginCliCommandsFromValidatedConfig` for the `nodes` primary without temporarily routing console diagnostics to stderr, before commander preAction runs. Add the smallest shared helper or local guard that keeps plugin registration diagnostics off stdout for real pre-terminator `--json` invocations while preserving non-JSON output and pass-through `-- --json` behavior. Add regression coverage for `openclaw nodes list --json` with noisy plugin startup output; do not special-case LCM logs or change the JSON payload writer contract. Add a `CHANGELOG.md` fix entry if this becomes a user-facing fix PR.
