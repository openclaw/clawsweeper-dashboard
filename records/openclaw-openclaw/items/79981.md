## Summary

Keep this PR open but do not merge as-is: the intended mcporter/XDG bug is real, but this patch strips the qmd-scoped environment from mcporter without preserving it for the qmd MCP server that mcporter launches, and the branch also bundles unrelated provider and diagnostic fixes.

## What This Changes

The branch changes `QmdMemoryManager` to spawn mcporter with a separate env lacking qmd XDG overrides, while also including unrelated DeepSeek/OpenRouter payload and diagnostic logging changes.

## Best Possible Solution

Keep mcporter config discovery away from qmd-scoped XDG while still passing the qmd-scoped env to the qmd MCP server, preferably through an explicit mcporter config/server env approach or equivalent; split unrelated provider and diagnostic fixes into their own PRs.

## Reproduction Assessment

Yes, at source level: current main passes qmd-scoped XDG variables into mcporter, and mcporter v0.10 uses XDG config discovery. I did not run the live daemon scenario in this read-only review.

## Solution Assessment

No. The env split addresses mcporter's own config lookup, but it also removes the qmd env that mcporter's stdio `qmd mcp` child normally inherits; the safer fix must preserve both sides of that boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep qmd's env available to the MCP server:** `extensions/memory-core/src/memory/qmd-manager.ts:2018`
  - body: Changing mcporter to run with `mcporterEnv` removes `XDG_CONFIG_HOME`, `QMD_CONFIG_DIR`, and `XDG_CACHE_HOME` from the mcporter process. mcporter's stdio transport launches server commands with `process.env` plus any config `env`, so a normal `qmd mcp` entry without explicit env will no longer see the agent-scoped QMD config/cache that OpenClaw just updated; it can search the user's default qmd state instead. Keep mcporter's config discovery clean while still passing the qmd env to the qmd server, for example with an explicit mcporter config that contains the server env.
  - confidence: 0.89
