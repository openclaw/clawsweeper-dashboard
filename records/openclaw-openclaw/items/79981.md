## Summary

This PR should stay open: the memory-core env split is a plausible fix for the mcporter 0.10 XDG regression, but the latest branch still leaves the existing qmd-manager regression test asserting the old mcporter env contract and the after-fix proof is not the real `memory_search`/mcporter path.

## What This Changes

The branch adds a separate `mcporterEnv` for QMD memory searches, demotes one diagnostic stuck-session log path from warn to debug, and removes `payload.reasoning` when DeepSeek V4 thinking uses `reasoning_effort`.

## Best Possible Solution

Land a focused memory-core change that separates qmd-only XDG scoping from mcporter config discovery, updates qmd-manager regression coverage, and coordinates with the overlapping mcporter isolation PR before merging unrelated fixes.

## Reproduction Assessment

Yes, from source inspection: current `main` sends qmd-scoped XDG variables into `runMcporter`, and mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery. I did not establish a live failing gateway run in this read-only review.

## Solution Assessment

No, not as submitted. Splitting the env is the narrow likely fix, but this branch needs the stale qmd-manager test updated, direct after-fix runtime proof, and maintainer coordination with the overlapping QMD/mcporter PR.
