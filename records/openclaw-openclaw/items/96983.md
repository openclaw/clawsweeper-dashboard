## Label Justifications

- `P2`: The issue describes a normal-priority agent tool-loop reliability bug with limited but real user impact.
- `impact:session-state`: Repeated no-op mutations can keep a session in a false-progress state and affect terminal run handling.
- `impact:crash-loop`: The reported failure mode is a runaway repeated tool loop that may require manual interruption.

## Summary

Keep open: current main and the latest release still allow no-op file mutations to look like normal tool progress or unstructured tool errors, while this issue is a narrower repair than the broader circuit-breaker tracker.

## What This Changes

Treat no-op file writes and no-op edit/apply_patch attempts as terminal, machine-readable agent tool-loop failures without breaking real writes, edits, or new-file creation.

## Best Possible Solution

Add a focused no-op file-mutation result contract for built-in `write`, `edit`, and `apply_patch`, with regression tests, while leaving general retry/cooldown policy to the existing circuit-breaker tracker.

## Reproduction Assessment

Yes at source level: `write` can detect identical content but still writes and returns success, `edit` throws only an unstructured no-change error, and `apply_patch` update hunks do not compare the result before reporting a modification. I did not run a live agent-loop replay in this read-only review.

## Solution Assessment

Yes if kept narrow. The maintainable fix is to classify no-op file mutations at the built-in file-tool/result boundary, not to add a new broad retry-budget or cooldown system here.
