## Summary

Keep open: the reported Docker setup bug is still plausible on current main, but this PR no longer fixes the supported path because it patches the removed root `docker-setup.sh` while current Docker setup runs through `scripts/docker/setup.sh`. ## What This Changes

The PR adds a manual `.env` loader, quote-preserving env writes, `docker-compose.override.yml` ignore coverage, and an `OPENCLAW_INSTALL_BROWSER` build arg to the old root `docker-setup.sh`.

## Best Possible Solution

Retarget the dotenv defaults loader and quote-preserving `upsert_env` behavior to `scripts/docker/setup.sh`, update `src/docker-setup.e2e.test.ts`, and keep caller environment overrides authoritative.

## Reproduction Assessment

Yes, from source inspection rather than a live Docker run: current main still writes whitespace package values bare and does not load general root `.env` defaults. The focused path is a Docker setup run that persists `OPENCLAW_DOCKER_APT_PACKAGES` with spaces and then expects `.env` defaults to round-trip safely.

## Solution Assessment

No. The parser/writer direction is plausible after review iterations, but the implementation must target `scripts/docker/setup.sh`; changing the removed root script does not fix the supported Docker setup flow.
