## Summary

Keep open: the reported Docker `.env` handling problem is still source-reproducible in the active setup path, but this PR targets the removed root `docker-setup.sh` entrypoint and lacks after-fix real Docker setup proof.

## What This Changes

The PR adds a manual `.env` loader and quote-preserving writes to the old root `docker-setup.sh`, ignores `docker-compose.override.yml`, and passes `OPENCLAW_INSTALL_BROWSER` as a Docker build arg.

## Best Possible Solution

Retarget the dotenv defaults loader and quote-preserving `upsert_env` behavior to `scripts/docker/setup.sh`, update the Docker setup e2e coverage, and keep caller environment overrides authoritative.

## Reproduction Assessment

Yes, from source inspection rather than a live Docker run: the active setup script does not load general root `.env` defaults and still writes managed whitespace values bare. The focused reproduction path is a Docker setup run that persists `OPENCLAW_DOCKER_APT_PACKAGES` with spaces and then expects `.env` defaults to round-trip safely.

## Solution Assessment

No. The parser/writer approach is plausible after the review iterations, but the implementation must target `scripts/docker/setup.sh`; changes to the removed root script do not fix the supported Docker setup flow.
