## Summary

Keep open. The Docker `.env` handling bug is still present on current `main`, but this PR is not correct as-is because it modifies the obsolete root setup implementation while current `main` routes users through `scripts/docker/setup.sh`.

## What This Changes

The PR updates root `docker-setup.sh` to load `.env` defaults, preserve quoting when rewriting managed `.env` keys, pass `OPENCLAW_INSTALL_BROWSER` into Docker builds, and ignore `docker-compose.override.yml`.

## Best Possible Solution

Keep the root script as a wrapper and implement the dotenv-default loading, quote-preserving managed `.env` writes, and supported browser build-arg wiring in `scripts/docker/setup.sh`, with e2e coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: put `OPENCLAW_DOCKER_APT_PACKAGES="jq ripgrep"` in root `.env` and run the documented `./scripts/docker/setup.sh`; the active script does not generally load that default and still rewrites managed whitespace values as bare assignments.

## Solution Assessment

No. The PR contains useful source logic for the right bug, but the maintainable fix must target the active setup script and update the existing Docker setup e2e expectations.
