## Summary

Keep open: the Docker `.env` handling bug still appears valid on current main, but this PR targets the removed root `docker-setup.sh` entrypoint, conflicts with current main, and lacks after-fix real behavior proof.

## What This Changes

Adds a manual `.env` loader and quote-preserving `upsert_env` helper to root `docker-setup.sh`, ignores `docker-compose.override.yml`, and passes `OPENCLAW_INSTALL_BROWSER` as a Docker build arg.

## Best Possible Solution

Retarget dotenv defaults loading and quote-preserving writes to `scripts/docker/setup.sh`, add focused e2e coverage, and preserve caller environment precedence over `.env` defaults.

## Reproduction Assessment

Yes, from source inspection rather than a live Docker run: current main still writes whitespace package values bare and does not load general `.env` defaults for the active Docker setup script.

## Solution Assessment

No. The parser/writer approach is plausible, but changing root `docker-setup.sh` is not the best fix because current docs and code use `scripts/docker/setup.sh`.
