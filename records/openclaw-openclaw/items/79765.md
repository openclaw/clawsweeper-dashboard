## Summary

Keep open: the PR moves the image-generation SSRF policy wiring in the right direction, but it only updates OpenAI/OpenAI-compatible image calls while the linked bug names Google/MiniMax-style provider HTTP calls that still ignore the new request policy.

## What This Changes

The PR adds optional SSRF policy fields to the image-generation runtime/request types, passes the tool's web-fetch SSRF policy into `generateImage`, and forwards that policy through the shared provider HTTP helper for OpenAI/OpenAI-compatible image calls.

## Best Possible Solution

Finish the existing policy propagation through the generic image-generation contract and every image provider that uses provider HTTP helpers, add focused regression coverage for named providers and the shared helper, and keep default SSRF behavior strict.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, choose a Google or MiniMax image provider behind fake-IP DNS, and the provider POST path still receives no RFC2544-capable policy. I did not run a live Clash/Surge network setup in this read-only review.

## Solution Assessment

No, this is not complete yet. The requested direction is narrow and maintainable, but the PR must wire the new request policy into all current image provider HTTP call sites, not only OpenAI/OpenAI-compatible providers.
