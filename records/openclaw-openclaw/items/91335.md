## What This Changes

The PR replaces iMessage opt-in catchup with always-on inbound dedupe, local `since_rowid` downtime recovery, stale-backlog age fencing, doctor/config cleanup, docs, and regression tests.

## Best Possible Solution

## Reproduction Assessment

Yes at source level. Current main suppresses only rows at or below the startup watermark and has no iMessage inbound replay guard, so old-date rows written above that boundary can reach dispatch; the PR body and tests exercise the intended suppression and downtime replay paths.

## Solution Assessment

No, not yet. The runtime design is plausible and matches imsg's exclusive cursor contract, but retiring a public config key must also remove it from exported TypeScript types and needs maintainer acceptance for the remote catchup behavior change.
