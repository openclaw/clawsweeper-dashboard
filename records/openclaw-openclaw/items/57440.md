## Summary

Close: current main already implements the intended subagent model behavior through session-persisted provider/model overrides, and this PR's direct RPC forwarding approach conflicts with the gateway's model-override authorization guard.

## What This Changes

The PR adds `model: resolvedModel || undefined` to the `sessions_spawn` subagent `agent` gateway call and adds two tests expecting that RPC param to be forwarded.

## Best Possible Solution

Keep the current session-persisted model override implementation and handle any remaining subagent-default edge cases in the active, narrower follow-up PR.

## Reproduction Assessment

Yes, at source level. The PR sends `model` on the `agent` RPC, and the current gateway rejects requested provider/model overrides from ordinary write-scope callers without `internal.allowModelOverride`.

## Solution Assessment

No. Forwarding the resolved model as an `agent` request override is not the best fix; the maintainable path is the current main behavior that persists the resolved model on the child session before launch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid forcing model override on subagent agent calls:** `src/agents/subagent-spawn.ts:713`
  - body: Adding `model` here makes every subagent spawn look like an explicit model override, because the resolver always returns a model. The gateway rejects provider/model override fields for ordinary write-scope `agent` callers without `internal.allowModelOverride`, so normal `sessions_spawn` calls fail before the child run starts instead of using the session-persisted model.
