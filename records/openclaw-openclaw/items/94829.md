## Summary

Keep open: current main still appears to compensate for duplicate assembled tool pairs after the context engine returns messages, but the reported durable problem is legacy null-transcript rows being reselected by lossless-claw on every assembly. That boundary is not solved by the generic sanitizer alone.

## What This Changes

The issue reports repeated lossless-claw sanitizer warnings from duplicated legacy null-transcript tool-use rows being reassembled into active context.

## Best Possible Solution

Have the context-engine/lossless-claw owner decide the repair boundary, then dedupe or quarantine legacy null-transcript duplicate tool pairs before assembly and add focused regression/health coverage; keep the core sanitizer as a final guard rather than the primary fix.

## Reproduction Assessment

No high-confidence live current-main reproduction is available because the private LCM database is not in this checkout. The source path is still reproducible by inspection: context-engine output is repaired after assembly, while the reported legacy/null-transcript duplicate source is not identified or migrated by that sanitizer.

## Solution Assessment

No, the current compensating sanitizer is not the best durable fix. The maintainable solution is to stop legacy duplicate rows from being selected or persistently repair them in the lossless-claw/context-engine owner boundary, adding a core API only if the plugin cannot safely identify the legacy rows itself.
