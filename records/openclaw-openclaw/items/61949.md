## Summary

Keep open. Current main still renders model aliases with guidance to prefer aliases, while runtime only preserves an exact provider/model after the LLM passes that exact string through.

## What This Changes

Prevent the Model Aliases prompt section from encouraging LLMs to replace an explicit provider/model subagent request with another provider's equivalent model.

## Best Possible Solution

Update alias prompt rendering so exact provider/model strings are passed through verbatim, aliases are only convenience names for bare requests, and overlapping-provider cases have regression coverage.

## Reproduction Assessment

Yes for the source-level precondition: a config with `agents.defaults.models` aliases still renders a Model Aliases section that tells agents to prefer aliases. The live provider substitution itself depends on model behavior, but the issue discussion provides a credible production trace.

## Solution Assessment

Yes. The narrowest maintainable fix is prompt and alias-presentation wording plus focused tests, because the runtime resolver already preserves qualified refs once it receives them.

Current main still renders `## Model Aliases` with guidance to prefer aliases, which can encourage an LLM to replace an explicit `provider/model` subagent request with another provider's alias. Update the core agent prompt and alias presentation so a full `provider/model` string is treated as an exact routing instruction to pass through verbatim, and aliases are framed only as convenience names for bare model requests. Do not change provider fallback semantics, auth, model catalog ownership, or runtime alias resolution beyond what is needed for this prompt behavior. Add focused regression coverage for overlapping provider aliases and exact `sessions_spawn.model` pass-through.
