## Summary

Keep open. Current main and the latest tagged release still render the same Model Aliases prompt guidance that tells the agent to prefer aliases, while the alias list is built from every configured `agents.defaults.models` alias. The runtime resolver preserves a fully qualified provider/model only after the model passes that exact value to the tool, so the reported prompt-level substitution hazard remains actionable.

## What This Changes

The issue asks OpenClaw to prevent model-alias prompt guidance from causing agents to substitute a different provider when an explicit provider/model override is requested.

## Best Possible Solution

Update the core prompt and alias presentation so exact `provider/model` strings are treated as provider-routing instructions to pass through verbatim, while aliases remain a convenience only when no provider is specified. Add focused regression coverage for overlapping providers and keep the existing runtime alias resolver behavior intact.

## Reproduction Assessment

Yes for the source-level prompt condition: a config with `agents.defaults.models` aliases renders the exact Model Aliases section and all alias lines. The end-to-end provider substitution is LLM-choice-dependent, but the issue discussion provides a credible production trace of the symptom.

## Solution Assessment

Yes. The narrowest maintainable fix is in core prompt/model-alias presentation, not a provider-specific workaround, because the runtime resolver already preserves a qualified ref once it receives one.

## Repair Work Prompt

Fix the model-alias prompt behavior reported in https://github.com/openclaw/openclaw/issues/61949. Current main renders `## Model Aliases` with `Prefer aliases when specifying model overrides`, and `buildModelAliasLines` includes every `agents.defaults.models` alias. Update the prompt/model-alias presentation so when a user/task/tool text provides a full `provider/model` string, agents are instructed to pass it through verbatim for `sessions_spawn`, `/model`, `session_status`, and other model overrides, and not translate it to an alias or a different provider. Preserve user-facing alias resolution for bare aliases and configured subagent defaults. Consider filtering or labeling aliases that are fallback-only or provider-overlapping if needed, but do not change provider fallback mechanics, auth routing, or provider-specific Bedrock/Copilot behavior.
