## Summary

Keep open. Current main still tells agents to prefer model aliases and still exposes every configured `agents.defaults.models` alias in the system prompt, while exact `provider/model` refs are only protected after the LLM passes that exact string to runtime code.

## What This Changes

The issue asks OpenClaw to stop prompt-level model alias guidance from causing agents to replace an explicit provider/model with a different provider's alias.

## Best Possible Solution

Update the core prompt and alias presentation so exact `provider/model` strings are treated as routing instructions to pass through verbatim, with aliases reserved for bare user convenience and covered by overlapping-provider regression tests.

## Reproduction Assessment

Yes for the source-level precondition: a config with `agents.defaults.models` aliases renders the Model Aliases section and all aliases with guidance to prefer aliases. The reported provider swap itself depends on model behavior, but the discussion provides a credible production trace of that symptom.

## Solution Assessment

Yes. The narrowest maintainable fix is prompt/model-alias wording and focused regression coverage, because the runtime resolver already preserves qualified refs once it receives them.

## Repair Work Prompt

Fix the model-alias prompt behavior reported in https://github.com/openclaw/openclaw/issues/61949. Current main renders `## Model Aliases` with `Prefer aliases when specifying model overrides`, and `buildModelAliasLines` includes every `agents.defaults.models` alias. Update the prompt/model-alias presentation so when a user, task, or tool text provides a full `provider/model` string, agents are instructed to pass it through verbatim for `sessions_spawn`, `/model`, `session_status`, and other model overrides, and not translate it to an alias or a different provider. Preserve user-facing alias resolution for bare aliases and configured subagent defaults. Consider filtering or labeling aliases that are fallback-only or provider-overlapping if needed, but do not change provider fallback mechanics, auth routing, or provider-specific Bedrock/Copilot behavior.
