The PR prevents cron-tool from inferring LINE recipients from lowercased session-key fragments, adds a LINE recipient case guard, and adds regression tests for both paths.

## Best Possible Solution

Land the focused cron fallback refusal and LINE-owned permanent-error guard with the included regression coverage, then track stricter LINE id validation or per-peer source improvements separately if maintainers want them.

## Solution Assessment

Yes; refusing the unrecoverable LINE session-key fallback and adding a channel-owned permanent-error guard is a narrow maintainable fix. Full strict LINE id validation would be broader follow-up work because current fixtures and compatibility rely on shorter placeholder ids.
