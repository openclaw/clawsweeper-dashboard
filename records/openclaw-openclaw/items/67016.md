## Summary

Keep open; current main and v2026.5.7 still expose only partial Dreams status, while the requested last-run timestamp, run outcome, processed-item count, and diary write/workspace metadata are not surfaced in the Dreams UI.

## What This Changes

Expose Dreaming run metadata in Control UI, including last run, items processed, diary update/workspace status, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreams run summary, expose it through the doctor Dreams status surface or a sibling method, normalize it in Control UI, render it in the Dreams tab, and document the semantics.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.5.7 shows the Dreams status path carries existing counts, phase next-run data, and diary content, but not the requested run-summary fields.

## Solution Assessment

Yes, with a contract decision first. A memory-core-owned run summary surfaced through the Dreams status path is the narrow maintainable direction; relying only on raw cron state would miss diary generation and multi-workspace partial failures.
