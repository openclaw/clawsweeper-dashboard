## Summary

Keep open. Current main still exposes only partial Dreams visibility: enabled state, aggregate signal/promoted counts, phase next-run timing, and diary content are present, but last run time, run outcome/partial failure, processed trace count, and diary write/workspace status are still not exposed in the Dreams UI. The closed related reliability report #74929 is useful context, but its managed-cron delivery fix does not resolve this observability request.

## What This Changes

The issue requests Dreams UI run metadata for last run time, processed item count, diary update/workspace state, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreams run summary, expose it through the existing doctor Dreams status path or an additive sibling method, normalize it in the Control UI controller, render it in the Dreams tab, and document the resulting operator-visible status semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to inspect `doctor.memory.status`, `loadDreamingStatus`, `loadDreamDiary`, `renderDreaming`, and the memory-core dreaming run path on current main; the requested UI-visible run-summary fields are absent.

## Solution Assessment

Yes. The narrow maintainable direction is a memory-core-owned run summary surfaced through the existing Dreams status path; relying only on raw cron state would miss diary, phase, and multi-workspace partial-failure semantics.
