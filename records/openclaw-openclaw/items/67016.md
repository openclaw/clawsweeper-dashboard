## Summary

Keep open; current main still does not surface Dreaming last-run, outcome, processed-count, or diary-write metadata, and the related cron/dreaming artifact reports do not supersede this UI observability request.

## What This Changes

Request to add Dreaming UI status metadata for last run, items processed, diary updates/workspace, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreaming run summary, expose it through doctor Dreams status or an additive sibling method, normalize it in Control UI, render it in the Dreams tab, and document the semantics.

## Reproduction Assessment

Yes, source-reproducible. Current main’s doctor payload, UI controller, and Dreams view do not carry or render the requested last-run, outcome, processed-count, or diary-write fields.

## Solution Assessment

Yes. A memory-core-owned run summary surfaced through the Dreams status path is the narrow maintainable direction; using raw cron state alone would miss phase execution, diary generation, and multi-workspace partial failures.
