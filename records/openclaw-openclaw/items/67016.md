## Summary

Keep open. Current main and v2026.5.12 still expose only partial Dreams status in the Control UI; the requested last-run timestamp, run outcome, processed item count, and diary write/workspace status remain log-only or backend-only data. The related closed cron-delivery issue fixed diary execution, and the open memory-status PR targets CLI phase wording, but neither supersedes this UI run-summary request.

## What This Changes

Expose Dreaming run metadata in the Control UI, including last run, processed item count, diary write/workspace status, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreams run summary, expose it through the doctor Dreams status surface or a sibling status method, normalize it in the Control UI controller, render the requested metadata in the Dreams tab, and update focused backend/UI tests plus Dreaming docs.

## Reproduction Assessment

Yes. Source inspection of current main shows the Dreams status path carries counts, entries, and next-run phase data but not the requested last-run, outcome, processed-count, or diary-write summary fields.

## Solution Assessment

Yes, directionally, but the implementation should not derive everything from raw cron state alone. A memory-core-owned run summary is the maintainable path because diary generation and per-workspace failures can diverge from cron success.
