## Summary

Keep open; current main and v2026.5.7 still expose only partial Dreams status, while the requested last-run, run-result, processed-count, and diary-write/workspace metadata remain absent.

## What This Changes

Expose Dreaming run metadata in the Control UI, including last run, items processed, diary update/workspace status, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreaming run summary, expose it through the doctor Dreams status surface or an additive sibling method, normalize it in Control UI, render it in the Dreams tab, and document the semantics.

## Reproduction Assessment

Yes. Source inspection of current main shows the doctor payload, UI controller, Dreams view, and focused tests only carry the existing counts, phase next-run data, and diary content, not the requested run-summary fields.

## Solution Assessment

Yes, with a contract decision first. A memory-core-owned run summary surfaced through the Dreams status path is the narrow maintainable direction; relying only on raw cron state would miss diary generation and multi-workspace partial failures.
