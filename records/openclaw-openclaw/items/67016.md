## Summary

Keep open. Current main still exposes partial Dreams visibility, but not the requested last-run timestamp, run result or partial-failure state, processed-trace count, or diary update/workspace metadata; the closed diary-delivery issue was related operational context, not a fix for this UI status request.

## What This Changes

Expose Dreaming run metadata in the Control UI, including last run, processed items, diary write/workspace status, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreaming run summary, expose it through doctor.memory.status or a sibling Dreams status method, render it in the Control UI, and document the aggregation semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows the Dreams status path carries counts, phase next-run data, and diary content, but not the requested last-run, result, processed-count, or diary-write summary fields.

## Solution Assessment

Yes, directionally, but it needs a contract decision first. A memory-core-owned run summary is more maintainable than deriving the UI from raw cron state alone, because diary generation and per-workspace failures can diverge from cron success.
