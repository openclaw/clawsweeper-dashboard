## Summary

Keep open. Current main still exposes only partial Dreams visibility: enabled state, counts, next scheduled timing, phase status, and diary content are present, but last run time, run outcome, processed trace count, and diary write/workspace status are not surfaced in the Dreams UI; the related closed cron-delivery report #74929 does not cover this observability request.

## What This Changes

Expose Dreams UI run metadata for last execution, items processed, diary write/workspace status, run outcome, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreams run summary, expose it through the existing doctor Dreams status surface or an additive sibling method, normalize it in the Control UI controller, render it in the Dreams tab, and document the operator-visible semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows the `doctor.memory.status` payload, Control UI controller, and Dreams view do not carry or render the requested last-run, outcome, processed-count, or diary-write fields.

## Solution Assessment

Yes. The narrow maintainable direction is a memory-core-owned run summary surfaced through Dreams status; using raw cron state alone would miss diary generation, phase execution, and multi-workspace partial-failure semantics.
