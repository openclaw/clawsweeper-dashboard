## Summary

Keep open. Current main and v2026.5.7 still expose only partial Dreams status; the requested last-run timestamp, run outcome, processed-item count, and diary write/workspace metadata are still not surfaced in the Control UI.

## What This Changes

Request that the Dreams UI expose Dreaming run metadata: last run, items processed, diary update/workspace status, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreaming run summary, expose it through doctor.memory.status or a sibling Dreams status method, normalize it in the Control UI, render it in the Dreams tab, and document the aggregation semantics.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.5.7 shows the Dreams status path carries counts, diary content, and phase next-run data, but not the requested last-run, result, processed-count, or diary-write summary fields.

## Solution Assessment

Yes, directionally, but it needs a contract decision first. A memory-core-owned run summary is more maintainable than deriving the UI from raw cron state alone because diary generation and per-workspace failures can diverge from cron success.
