## Summary

Keep open. Current main and v2026.6.6 only partially cover inline media: structured/managed image and trusted local audio/video attachment rendering exists, but remote Markdown images and web previews remain intentionally constrained and the remaining umbrella scope needs product and security review.

## What This Changes

Request inline chat previews for generated or received images, audio/music, video, and web links instead of requiring external open commands.

## Best Possible Solution

Define a secure Control UI media-preview policy, keep structured attachment paths as the default, land narrow image fixes where appropriate, and add link/web previews only through scoped or explicitly gated fetch/proxy behavior.

## Reproduction Assessment

Yes for the concrete remaining pieces: current source and tests show remote Markdown images flatten to alt text, CSP blocks arbitrary remote image/media loads, and video chat uploads are filtered. The full umbrella still needs product-scoped acceptance criteria for which media sources and preview types should be supported.

## Solution Assessment

No, the issue as written is too broad for one direct fix. The maintainable path is to split source types and media kinds, preserve managed attachment safety paths, and require product/security approval for remote web previews.
