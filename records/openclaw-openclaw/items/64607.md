## Label Justifications

- `P2`: This is a normal-priority user-facing media UX improvement with real security scope but no evidence of an urgent regression or outage.
- `impact:security`: Remote media and web previews involve untrusted links, browser fetches, CSP, local media serving, and attachment exposure boundaries.
- `impact:message-loss`: Generated or received media can exist in the conversation while remaining unavailable or invisible inline in the chat transcript.

## Summary

Keep open. Current main and v2026.6.6 only partially cover inline media: managed/local image, audio, and video attachment rendering exists, but remote Markdown images, broad remote media fetches, and web previews remain intentionally blocked or undefined and need product/security decisions.

## What This Changes

The issue requests inline display for generated or received images, audio, video, and web previews in chat instead of requiring external open commands.

## Best Possible Solution

Define a secure Control UI media-preview matrix, land the existing focused image/media fixes where appropriate, and add remote/web previews only through authenticated, scoped, or explicitly opt-in proxy paths.

## Reproduction Assessment

Yes for concrete remaining pieces: current source and tests show remote Markdown images flatten to alt text, CSP blocks remote image/media fetches, and docs state remote images are rejected. The full umbrella still needs product-scoped acceptance criteria for supported media sources and preview types.

## Solution Assessment

No as a single direct implementation; the issue is too broad and security-sensitive for one automatic fix. The maintainable path is to split by media kind and source, preserve the existing managed attachment safety model, and make remote/web preview behavior an explicit product decision.
