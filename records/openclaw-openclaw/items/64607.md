## Summary

Keep open. Current main and v2026.6.6 only partially cover inline media: managed/local image, audio, and video attachment rendering exists, but remote Markdown images are still flattened, remote image/media fetches are blocked by CSP, video uploads are filtered, and web previews remain a product/security decision.

## What This Changes

Request inline Control UI chat previews and players for generated or received images, audio, video, and web links instead of requiring external open commands.

## Best Possible Solution

Define a secure Control UI media-preview matrix, land narrow image fixes where appropriate, and implement remote media or web previews only through authenticated, scoped, or explicitly opt-in proxy paths.

## Reproduction Assessment

Yes for concrete remaining pieces: current source and tests show remote Markdown images flatten to alt text, CSP blocks remote image/media fetches, and video chat uploads are filtered. The full umbrella still needs product-scoped acceptance criteria for supported media sources and preview types.

## Solution Assessment

No, the issue as written is too broad for one direct implementation. The maintainable path is to split media kinds and sources, preserve existing managed attachment safety paths, and require product/security approval for remote web previews.
