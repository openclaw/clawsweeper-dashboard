## Summary

Keep open. Current main and the latest release only partially cover inline media: structured/managed image and trusted local audio/video attachment paths exist, but remote Markdown images are still intentionally flattened, web previews are not implemented, and the remaining behavior needs product and security scoping rather than cleanup closure.

## What This Changes

The issue requests inline rendering and preview controls for images, audio, video, and web links in OpenClaw chat instead of requiring users to open media externally.

## Best Possible Solution

Define a secure Control UI media-preview policy, land the narrow image/tool/inbound fixes where appropriate, and add web/link previews only through authenticated, allowlisted or explicitly gated fetch/proxy paths.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence for at least the remote Markdown image gap: current main and v2026.6.6 tests assert HTTPS markdown images flatten to alt text. The broader all-media/web-preview request still needs product-scoped acceptance criteria rather than one live repro path.

## Solution Assessment

No, the issue as written is too broad for one fix. The better path is to split source types and media kinds, preserve existing managed attachment safety paths, and require product/security approval for remote web previews.
