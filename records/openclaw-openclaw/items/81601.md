## Summary

Keep open. This is a focused bug-fix PR for a real current-main ClawHub archive body timeout gap; the reader-owned cancellation approach looks correct, the updated proof is sufficient, and I found no blocking functional or security findings.

## What This Changes

The PR adds reader-owned stalled-body timeout/cancel handling for ClawHub skill, package, and ClawPack archive downloads, adds regression tests, and updates CHANGELOG.md.

## Best Possible Solution

Land this reader-owned ClawHub archive body timeout fix after CI is green or confirmed unrelated, then resolve the older overlapping PR in favor of this implementation.

## Reproduction Assessment

Yes, at source level: current main clears the fetch timeout before direct archive arrayBuffer() calls, so a headers-then-stalled body can leave the install path pending. The PR body also supplies after-fix local Node output for the same stalled stream condition.

## Solution Assessment

Yes, the reader-owned getReader() loop is the narrow maintainable fix because it cancels the active body read and keeps existing digest/temp-file behavior. It is also safer than the older overlapping approach that tried to cancel the body after arrayBuffer() had locked it.
