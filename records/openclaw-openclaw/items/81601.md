## Summary

Keep open for normal maintainer review: current main still has the ClawHub archive body-timeout gap, while this PR fixes the affected archive paths with reader-owned cancellation, regression tests, and sufficient terminal proof. I found no blocking review findings or security concerns.

## What This Changes

The PR changes ClawHub archive downloads to read response bodies through an owned stream reader with timeout cancellation, adds stalled-body tests, and updates the changelog.

## Best Possible Solution

Land this reader-owned ClawHub archive body timeout fix after maintainer review, then resolve https://github.com/openclaw/openclaw/issues/52073 and close or supersede https://github.com/openclaw/openclaw/pull/80006 in favor of this implementation.

## Reproduction Assessment

Yes, at source level: current main clears the fetch timeout before direct archive `arrayBuffer()` calls, so a headers-then-stalled response body can remain pending. The PR body also supplies after-fix terminal output showing timeout and cancellation for all three affected download paths.

## Solution Assessment

Yes, the reader-owned `getReader()` loop is the narrow maintainable fix because it cancels the active body read while preserving existing digest and temp-file behavior. It is safer than the older overlapping approach in https://github.com/openclaw/openclaw/pull/80006, which tried to cancel the response body while `arrayBuffer()` owned the stream.
