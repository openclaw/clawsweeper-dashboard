## Label Justifications

- `P1`: The PR addresses a reported Codex harness failure that can prevent normal model turns when optional plugin app inventory is unavailable.
- `merge-risk: 🚨 compatibility`: The shared cache API changes from rejecting refresh failures to resolving diagnostic snapshots, which alters how all consumers detect failure.

## Summary

This PR remains necessary because current main still rejects failed Codex app-inventory refreshes; the patch consistently converts that shared failure contract to diagnostic snapshots while preserving activation diagnostics and migration’s fail-closed behavior, but the compatibility-sensitive contract change needs explicit owner acceptance before merge.

## What This Changes

The PR changes Codex app-inventory refresh failures from rejected promises to diagnostic snapshots and updates activation, migration, regression tests, and a focused repro script for that contract.

## Best Possible Solution

Adopt the diagnostic-snapshot contract if Codex app inventory is intended to be fail-open by default, while retaining explicit `lastError` checks wherever activation diagnostics or migration eligibility must treat inventory failure specially.

## Reproduction Assessment

Yes at source level: current main propagates an `app/list` rejection through `refreshNow`, and the thread-config path must catch it to continue. The supplied direct-runtime proof demonstrates the proposed resolved-snapshot behavior, though it does not run a complete live Codex harness turn.

## Solution Assessment

Yes, subject to owner acceptance of the shared fail-open contract. The patch centralizes optional inventory degradation and explicitly preserves activation diagnostics and migration blocking instead of leaving inconsistent rejection handling across callers.
