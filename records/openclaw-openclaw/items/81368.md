## Summary

Keep open: current main still has the reported cron watchdog classifier gap, and the focused fix PRs linked to this issue were closed without merging.

## What This Changes

This issue reports that isolated cron runs still hit the 60s pre-execution watchdog after reaching `attempt_dispatch`, even when the configured job timeout is much longer.

## Best Possible Solution

Land a narrow cron timer fix that treats the existing `attempt_dispatch` phase as execution-started and locks it with focused timer regression coverage.

## Reproduction Assessment

Yes. The current source path is source-reproducible: `attempt_dispatch` is emitted by the runner and is a valid cron phase, but the watchdog classifier does not treat it as execution-started; reporter logs show the same failure across beta releases.

## Solution Assessment

Yes. Adding `attempt_dispatch` to the private execution-start classifier with a regression test is the narrowest maintainable fix for the reported false stall; richer runner-side model-call telemetry can be a separate follow-up.
