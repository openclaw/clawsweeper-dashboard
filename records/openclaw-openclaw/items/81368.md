## Summary

Keep open: current main still leaves `attempt_dispatch` out of the cron execution-start classifier, and the issue is now paired with an open focused PR that should be reviewed or landed before this issue closes.

## What This Changes

Treat `attempt_dispatch` as an execution-start milestone so isolated cron runs are governed by the configured job timeout instead of the 60s pre-execution watchdog.

## Best Possible Solution

Review and land the focused fix in https://github.com/openclaw/openclaw/pull/81871, then close this issue with the merge commit and release provenance.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction: current main accepts and emits `attempt_dispatch` but does not classify it as execution-started. The reporter also provided live beta evidence showing the watchdog failure and a matching one-line installed-bundle hotfix passing past 60s.

## Solution Assessment

Yes, the linked PR's approach is the narrowest maintainable fix for the reported false stall: add `attempt_dispatch` to the existing classifier and add a focused timer regression. A larger runner-emission redesign can be considered separately, but it is not needed to unblock this bug.
