## Summary

Keep open: current main still does not classify `attempt_dispatch` as an execution-start milestone, and an open PR explicitly closes this issue. Repository policy says an issue with an open closing PR should stay open until that PR lands or is rejected.

## What This Changes

The issue reports that isolated cron runs on beta.4/beta.6 can remain at `attempt_dispatch` past the 60s pre-execution watchdog and be falsely aborted despite a longer job timeout.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81371, or an equivalent narrow change, so `attempt_dispatch` clears the pre-execution watchdog with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main can receive `attempt_dispatch` while the cron watchdog classifier does not treat that phase as execution-started; I did not run the live cron smoke in this read-only review.

## Solution Assessment

Yes. The linked PR’s classifier addition plus focused regression test is the narrowest maintainable fix for the reported false timeout, while broader runner phase refinements can remain separate if maintainers want finer diagnostics.
