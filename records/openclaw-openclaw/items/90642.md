## Summary

Keep open: current main still promotes native media input only for images, while audio is captured and routed through transcription/media-understanding rather than normalized into model-turn audio content. The closest related open PR targets media-understanding and still needs maintainer API/product direction, so this is not implemented or safe for an autonomous fix lane.

## What This Changes

The issue requests native audio input forwarding for audio-capable multimodal models instead of always using transcription/media-understanding for audio attachments.

## Best Possible Solution

Define one normalized audio-content capability through the model/provider turn path, gate it on model input support, and preserve transcription/media-understanding fallback when the selected model cannot accept audio natively.

## Reproduction Assessment

Yes from source inspection at a capability level: Telegram audio reaches MediaPaths/MediaTypes, but native model input promotion and OpenAI-compatible conversions are image-only. No live Telegram plus llama.cpp reproduction was provided or run.

## Solution Assessment

Unclear. Native audio forwarding is a plausible desired end state, but the maintainable fix should define a shared provider/model audio-content contract and fallback policy rather than hardcoding Telegram voice messages into one transport.
