## Summary

Keep open: current main still lacks a native audio model-content path for normal agent turns, even though audio attachments can be transcribed through media understanding and some sibling surfaces mention `input_audio`. The related native audio/video PR is open and appears to target a neighboring media-understanding design, so maintainers still need to decide the core provider/content contract.

## What This Changes

Request native forwarding of Telegram audio attachments to audio-capable OpenAI-compatible/local models instead of only using transcription/media-understanding.

## Best Possible Solution

Define a generic native audio content contract across LLM core, provider capability metadata, and provider adapters, then wire channel audio attachments into model turns only when the active model declares support while preserving transcription fallback.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run: current main's model-turn content and OpenAI adapters only support text/image, while audio is routed through transcription/media-understanding. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear: native audio forwarding is a plausible and aligned direction, but the exact implementation should be a generic content/provider contract rather than a llama.cpp-specific payload shortcut.
