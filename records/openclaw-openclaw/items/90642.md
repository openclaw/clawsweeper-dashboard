## Summary

Keep open: current main and v2026.6.6 still have an image-only direct LLM content path, while audio is handled through media-understanding transcription. The issue is active, assigned, and overlaps an open native audio/video PR that is still asking for maintainer direction on the content/provider contract.

## What This Changes

Request native forwarding of inbound audio attachments to audio-capable models instead of always converting them through transcription first.

## Best Possible Solution

Define one typed native audio content contract shared by model catalogs, agent runner inputs, provider adapters, media-understanding fallback, and operator-visible raw-media policy.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 show text/image-only direct LLM content and audio.transcription media handling, so a Telegram voice note cannot flow to the selected model as native input_audio through the normal agent path today.

## Solution Assessment

No, not as a llama.cpp-only input_audio shortcut. The best solution is a generic typed modality/provider contract with explicit fallback and raw-media policy, then provider-specific payload mapping behind that contract.
