## What This Changes

Request native audio forwarding for Telegram voice/audio attachments to audio-capable OpenAI-compatible models instead of only transcription through `tools.media.audio`.

## Best Possible Solution

Define a generic native audio content/provider contract, wire channel audio attachments into model turns only for declared audio-capable models, and preserve transcription fallback for unsupported providers.

## Reproduction Assessment

Yes. Source inspection shows current model-turn content types and run options carry text/images, while Telegram/audio paths feed transcription and media-understanding context instead of native audio blocks.

## Solution Assessment

Unclear. Native audio forwarding is a plausible direction, but the maintainable fix is a generic content/provider capability contract rather than a llama.cpp-specific `input_audio` shortcut.
