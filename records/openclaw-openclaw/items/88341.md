## Label Justifications

- `P1`: The report describes a live Discord channel regression where internal tool-progress output is posted to users during active agent runs.
- `impact:message-loss`: The issue is about channel delivery correctness: internal runtime progress is being misdelivered as visible Discord messages.
- `impact:security`: The leaked progress lines can expose internal commands and local path-like context to a Discord channel, even though the report does not show credential exposure.

## Summary

Keep open: current main still has Discord paths that preserve explicit tool-progress payloads and render visible progress-draft lines, while the report describes those lines escaping as durable Discord messages during Barnacling/progress state. The issue is actionable, but I did not run a live Discord repro in this read-only review.

## What This Changes

The issue asks OpenClaw to stop Discord from posting internal tool-progress narration as visible channel messages when a Discord-bound agent is in the Barnacling progress state.

## Best Possible Solution

Keep Discord progress drafts as one transient/editable status message, but ensure any fallback or tool-delivery path that would create durable Discord posts either suppresses standalone tool progress or runs the existing internal-trace stripping before send.

## Reproduction Assessment

No live high-confidence repro was run, but source inspection gives a credible current-main path: Discord preserves `kind: "tool"` progress payloads and separately renders progress drafts with Barnacling-style labels. The issue body also provides concrete live-message evidence.

## Solution Assessment

Yes, the maintainable fix is to keep the existing Discord progress-draft feature but close the bypass where tool-progress text can become standalone durable Discord output. A fix should add focused tests around progress mode, finalization/fallback, and `kind: "tool"` delivery.
