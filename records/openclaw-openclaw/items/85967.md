## Summary

Keep open: the OOM mitigation and force-pushed temp-prefix repair are useful, but the disk-budget sweep still treats fresh session-store temp files as immediately removable, which can race an active atomic write under disk pressure.

## What This Changes

The PR changes read-only doctor session-store loads to skip cache/clones, threads a session-store tempPrefix through atomic writes, and adds stale temp cleanup/classifier tests.

## Best Possible Solution

Keep the doctor read-only load optimization and store-specific temp prefix, but gate disk-budget store-temp deletion with the same staleness protection used by the prune path before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows disk-budget cleanup matches store temps without an mtime gate while the prune path uses a five-minute cutoff, and the added disk-budget test removes a freshly created temp under pressure.

## Solution Assessment

No. The doctor load and store-specific temp naming are the right direction, but the cleanup implementation needs the same stale-only safety check in the disk-budget path before it is the best fix.
