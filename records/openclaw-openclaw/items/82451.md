## Summary

Close: the reported Telegram polling/event-loop starvation on 2026.5.7 is covered by the merged isolated-ingress and getUpdates-liveness fixes, and the v2026.5.12 release notes include the Telegram polling fix path.

## What This Changes

The issue reports Gateway stalls during `channels.telegram.start-account` with multiple Telegram bot accounts, causing delayed getMe/getUpdates calls, blocked RPC/WebSocket traffic, and dropped agent sessions on openclaw 2026.5.7.

## Best Possible Solution

Keep the shipped isolated-ingress and getUpdates-liveness implementation, and ask for a fresh report only if the symptom reproduces on v2026.5.12 or newer with updated logs.

## Reproduction Assessment

Yes for the original failure mode: the canonical related report and fix proof deliberately blocked the main event loop and showed Telegram ingress continuing through the worker-backed spool. I did not establish a failing reproduction on current main because the implicated path has already changed.

## Solution Assessment

Yes. Moving Telegram getUpdates polling into an isolated worker with durable spooling is the narrow maintainable fix for main event-loop starvation, and the startup probe limiter separately addresses multi-account getMe fanout.
