## Label Justifications

- `P2`: This is a normal-priority gateway CLI bug fix with limited blast radius but a concrete correctness blocker before merge.
- `merge-risk: 🚨 compatibility`: The new CLI flag does not consistently satisfy the documented `--port` over environment precedence contract for existing env-configured shells.
- `merge-risk: 🚨 security-boundary`: The health path can route explicit credentials supplied with `--port` to an env-selected gateway URL rather than the intended local target.

## Summary

Keep open: this is a useful fix for the linked gateway CLI bug, but the health path is still not correct when gateway environment overrides are present. The new probe path carries a port override directly, while the health path only overlays config and still lets `OPENCLAW_GATEWAY_PORT` or `OPENCLAW_GATEWAY_URL` win later.

## What This Changes

The PR adds `--port` parsing for `gateway health` and `gateway probe`, wires the selected port into local target selection, updates gateway CLI docs, and adds focused Vitest coverage.

## Best Possible Solution

Make `--port` an authoritative local-target override for both health and probe, bypassing env URL/port target selection where appropriate, and add env-precedence regression coverage before merge.

## Reproduction Assessment

Yes, at source level: current main registers no `--port` option on `gateway health` or `gateway probe`, matching the linked report that Commander rejects the option. I did not run the CLI in this read-only review.

## Solution Assessment

No. Adding the flag is the right repair direction, but the health implementation is not yet the best fix because it mutates config while later target resolution still honors environment overrides.
