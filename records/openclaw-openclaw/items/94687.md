## Summary

Keep open; current main still lacks the requested health/probe port flags, but this PR should not merge yet because `gateway health --port` only mutates config while the gateway call path still lets `OPENCLAW_GATEWAY_URL` or `OPENCLAW_GATEWAY_PORT` override target selection.

## What This Changes

The PR adds `--port` parsing and docs for `gateway health` and `gateway probe`, forwards probe's port into status target resolution, and adds focused CLI/status tests.

## Best Possible Solution

Make `--port` a real explicit local target override for both health and probe, bypassing gateway env URL/port only for that invocation and covering env precedence in tests.

## Reproduction Assessment

Yes, at source level. Current main registers health and probe without child `--port`, and the PR's remaining health bug is source-reproducible because the health path writes only config while downstream target resolution still consults gateway env overrides.

## Solution Assessment

No. The probe path carries a real override, but health needs the same explicit override semantics through the gateway call/connection layer or an env-bypass signal rather than only replacing `config.gateway.port`.
