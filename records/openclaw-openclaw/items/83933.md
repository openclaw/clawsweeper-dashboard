## Summary

Keep this PR open: current main and the latest release still route manual cron completions through scheduled-state mutation and `deleteAfterRun`, while this branch is the active, well-proven candidate fix for the linked cron data-loss/session-state issue. I did not find a discrete line-level defect in the patch, but the change intentionally redefines manual cron run state semantics and still needs maintainer acceptance before merge.

## What This Changes

This PR marks manual cron run results, preserves `deleteAfterRun` one-shot jobs, and prevents manual completions from mutating scheduled retry, backoff, alert, skip, and counter state.

## Best Possible Solution

Land this only after a maintainer accepts the intended policy that manual cron runs record observability state while scheduled executions remain the sole owner of one-shot deletion, retry, backoff, alert, skip, and catch-up scheduling state.

## Reproduction Assessment

## Solution Assessment

Yes, with one maintainer decision still required. The PR puts the manual/scheduled distinction at the shared result-application boundary and adds protocol/state support where restart/backoff logic needs it; the remaining question is whether maintainers accept the compatibility-sensitive policy change.
