## Summary

Keep open: the PR targets the right cron state path, but it still lets manual error/skipped outcomes affect failure-alert bookkeeping before the manual guard, and it has no redacted real CLI/log proof of the fixed one-shot flow.

## What This Changes

The PR adds an `isManual` option to cron result handling, passes it from manual cron run paths, and adds regression tests so manual one-shot `deleteAfterRun` jobs are preserved.

## Best Possible Solution

Keep the `isManual` seam, make manual one-shot ok/error/skipped outcomes preserve scheduled job state without alert/counter/cooldown side effects unless maintainers intentionally keep them, and merge after redacted CLI proof.

## Reproduction Assessment

Yes, by source inspection rather than live execution. Current main sends manual cron completion into `applyJobResult` without a manual flag, and successful one-shot `deleteAfterRun` jobs return the deletion path there.

## Solution Assessment

No, not yet. The `isManual` discriminator is the right seam, but the PR needs to guard manual one-shot alert/counter/cooldown mutations before stateful failure handling and then prove the CLI flow.
