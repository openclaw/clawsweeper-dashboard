## Summary

Keep open: the PR addresses a real cron state-loss bug and has useful Docker Gateway CLI proof, but it changes documented manual-run failure semantics and should update the cron docs before merge.

## What This Changes

The PR adds an `isManual` cron result option, passes it from manual run paths, and adds regression coverage so manual cron runs preserve scheduled delete/backoff/counter state while scheduled runs keep existing behavior.

## Best Possible Solution

Land the runtime fix with a small cron docs update that states manual `cron run` preserves scheduled delete/retry/backoff/counter/alert state, while scheduled runs keep the existing semantics.

## Reproduction Assessment

Yes. Current main's `applyJobResult` has no manual-run discriminator, so successful `deleteAfterRun` at-jobs return `shouldDelete=true` and manual failures update the same counters/backoff state used by scheduled runs.

## Solution Assessment

Mostly yes. Adding an explicit manual-run discriminator at the shared state-application boundary is the narrow runtime fix, but the docs need to describe the new manual-run exception before merge.
