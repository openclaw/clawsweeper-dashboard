## Summary

This PR is a plausible fix for a real cron data-loss bug, but it is not merge-ready: the patch still lets a manual success clear scheduled failure-alert cooldown state, and the supplied proof does not exercise the patched `openclaw cron run` flow against a real Gateway/CLI setup.

## What This Changes

The PR adds an `isManual` result-application option for manual cron runs, passes it from manual-run completion paths, and adds timer regression coverage for `deleteAfterRun` and counter preservation.

## Best Possible Solution

Land the manual result boundary after preserving all scheduled retry and failure-alert state for manual runs and after redacted patched Gateway/CLI proof shows manual runs preserve the job while scheduled runs still delete it.

## Reproduction Assessment

Yes, source-reproducible. Current main manual-run completion calls `applyJobResult` without a manual discriminator, and `applyJobResult` deletes successful `deleteAfterRun` at-jobs regardless of whether the run was manual.

## Solution Assessment

Mostly yes. Adding a manual discriminator at the result-application boundary is the narrow fix, but it should also preserve failure-alert cooldown state and be backed by real patched CLI/Gateway proof.
