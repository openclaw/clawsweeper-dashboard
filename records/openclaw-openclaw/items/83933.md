## Label Justifications

- `P1`: The PR addresses a destructive cron manual-run data-loss path and still has a blocking retry-state correctness issue.
- `merge-risk: 🚨 compatibility`: Changing manual `deleteAfterRun` semantics can alter existing operator workflows that used manual runs to consume one-shot jobs.
- `merge-risk: 🚨 message-delivery`: Preserving a manually fired one-shot job can cause a later scheduled delivery that existing users may not expect after manual verification.

## Summary

Keep open: the PR is a focused fix for a real cron data-loss path, but it is not merge-ready because the latest patch still lets manual at-job results mutate the scheduled retry counters and the supplied proof is unit/source-level rather than a live patched Gateway/CLI run.

## What This Changes

The PR passes an `isManual` flag from manual cron run paths into `applyJobResult`, skips `deleteAfterRun` and at-job disable/retry handling for manual runs, preserves failure-alert cooldown on manual success, and adds timer regression tests.

## Best Possible Solution

Land a bounded manual-result boundary that preserves one-shot jobs and scheduled retry state while keeping the intended failure-alert behavior explicit, with redacted live CLI/Gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main routes manual cron completion into `applyJobResult` without a manual flag, and that function deletes successful `deleteAfterRun` at-jobs solely from schedule kind, flag, and status. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

No: the skip-delete direction is the right boundary, but the latest patch still lets manual results mutate scheduled retry counters and lacks live patched CLI/Gateway proof. The safer solution is to separate or restore scheduled retry state while keeping failure-alert behavior explicit.
