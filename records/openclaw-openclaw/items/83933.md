## Summary

Keep open: the PR fixes the destructive manual `deleteAfterRun` deletion path, but the latest head still lets manual results mutate the same counters used for scheduled retry/backoff state, and the supplied proof is source/unit-level rather than a live patched Gateway/CLI flow.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual cron run paths, skips `deleteAfterRun` for manual at-jobs, and adds timer regression coverage.

## Best Possible Solution

Land a bounded manual-result boundary that skips `deleteAfterRun` and isolates scheduled retry/backoff state, while preserving any intended manual failure-alert behavior through explicit separate accounting and live CLI proof.

## Reproduction Assessment

Yes, source-reproducible: current main routes manual completion into `applyJobResult` without a manual discriminator, and successful `deleteAfterRun` at-jobs return `shouldDelete` from kind, flag, and status alone. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

No, not yet: the PR isolates deletion and at-job enable/nextRunAtMs handling, but it still mutates scheduled retry counters for manual results. The safer fix is to separate manual alert behavior from scheduled retry/backoff state and prove the real CLI/Gateway flow.
