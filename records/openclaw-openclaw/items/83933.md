## Summary

This PR still targets a real P1 cron state-loss bug, but it is not merge-ready: current head leaves a manual due one-shot path that can still lose its scheduled fire, and one cron CI shard is failing after the manual-run semantic change.

## What This Changes

The branch adds an `isManual` cron outcome flag, persists `lastRunWasManual`, updates the gateway cron schema, and expands cron tests so manual runs do not delete one-shot jobs or mutate scheduled retry counters.

## Best Possible Solution

Keep the manual-run isolation, but make the post-run maintenance path preserve the manually run job's scheduled slot, update scheduled-backoff coverage to use the timer path, and require current-head CLI/service proof for successful and error `deleteAfterRun` manual runs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a due manual at-job sets `lastRunAtMs` at or after the original slot, then the post-run maintenance recompute treats that as scheduled execution and clears the past `at` schedule. The current cron-service CI failure separately reproduces the stale recurring-backoff test path.

## Solution Assessment

No; adding an `isManual` flag at `applyJobResult` is the right direction, but the solution is incomplete because the post-run maintenance and scheduled-backoff test paths were not updated consistently. The narrower best fix is to carry the manual-run invariant through recompute and keep scheduled retry coverage on `onTimer`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manual due slots through recompute:** `src/cron/service/ops.ts:948`
  - body: Passing `isManual: true` here keeps `applyJobResult` from deleting the job, but the same manual finish path later calls `recomputeNextRunsForMaintenance(..., { recomputeExpired: true })`. For a due `at` job, the manual `lastRunAtMs` is at or after the original `nextRunAtMs`, so maintenance treats the slot as already executed and `computeNextRunAtMs` returns `undefined` for the past `at` schedule. The job can remain enabled with no future fire, so a successful manual `cron run` can still consume the scheduled one-shot.
  - confidence: 0.91
- **[P2] Move recurring backoff coverage off manual runs:** `src/cron/service/timer.ts:878`
  - body: This new manual-run guard intentionally skips recurring error backoff for `cron run`, but the existing `timer.backoff-config-readback.test.ts` still uses `run(state, job.id, "due")` to assert scheduled backoff persistence. Current CI fails with `nextRunAtMs` at the natural 1s interval instead of the configured 300s floor, so that regression should be driven through the scheduled timer path before merge.
  - confidence: 0.86
