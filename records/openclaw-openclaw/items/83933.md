## Summary

Keep this PR open: the `deleteAfterRun` preservation direction is valid, but the latest patch suppresses failure-alert/error bookkeeping for every manual cron run and still lacks real patched CLI/API proof.

## What This Changes

This PR adds an `isManual` option to cron result handling, passes it from manual cron run paths, and adds regression tests for preserving `deleteAfterRun` one-shot jobs during manual runs.

## Best Possible Solution

Preserve `deleteAfterRun` at-jobs for manual runs without deleting or disabling them, while keeping existing manual failure-alert/counter behavior unless maintainers explicitly change that contract.

## Reproduction Assessment

Yes. Current main source routes manual cron completions into `applyJobResult` without a manual flag, and the PR regression is source-reproducible because `opts.isManual` now bypasses alert bookkeeping for every manual run.

## Solution Assessment

No, not yet. The `isManual` seam is the right shape for preserving one-shot jobs, but it needs to be scoped so existing manual failure alerts and counters still work, plus real patched CLI/API proof.
