## Summary

Keep open: this is a focused fix for a real cron data-loss bug, but the latest patch still mutates failure-alert/cooldown state before the manual-run preservation branch and the supplied proof does not show an end-to-end patched cron CLI/API run.

## What This Changes

The PR adds an `isManual` result-handling option, passes it from manual cron run paths, and adds timer regression tests so manual one-shot `deleteAfterRun` jobs are preserved.

## Best Possible Solution

Keep the `isManual` seam, guard manual one-shot error/skipped bookkeeping before counters and failure-alert cooldown mutate, and merge only after redacted real cron CLI/API proof shows the scheduled job is preserved.

## Reproduction Assessment

Yes, by source inspection and the linked issue context: current main routes manual cron completions into `applyJobResult` without a manual flag, and that function deletes successful `deleteAfterRun` at-jobs. I did not run a live cron service reproduction because this review is read-only.

## Solution Assessment

No, not yet. The `isManual` discriminator is the right maintainable seam, but the fix should also suppress manual one-shot alert/counter/cooldown side effects before merge and needs real CLI/API proof.
