## Label Justifications

- `P1`: The underlying bug can silently destroy scheduled cron delivery, and the PR changes a real operator workflow around manual cron runs.
- `merge-risk: 🚨 compatibility`: Existing users who relied on manual `cron run` consuming one-shot jobs will see different post-upgrade behavior.

- `merge-risk: 🚨 session-state`: The patch changes persisted cron counters, backoff, cooldown, enabled, and nextRunAtMs state transitions for manual runs.

## Summary

Keep open for maintainer review. The patch appears to fix the destructive manual cron run path and has real Docker/CLI proof, but it intentionally changes compatibility-sensitive manual-run semantics for one-shot delivery and scheduled retry state.

## What This Changes

The PR threads an `isManual` flag through cron manual-run completion so manual runs do not consume `deleteAfterRun` jobs or mutate scheduled retry, skip, backoff, disable, and failure-alert state.

## Best Possible Solution

Land this only after a maintainer explicitly accepts the non-consuming manual-run contract as the desired cron behavior; otherwise choose an explicit warning or override path instead.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main's manual completion calls `applyJobResult` without a manual discriminator, and `applyJobResult` unconditionally deletes successful `deleteAfterRun` at-jobs. I did not run local tests because this review is read-only.

## Solution Assessment

Mostly yes: the PR fixes the implicated shared state handler and both manual call sites with focused regression coverage. The open question is not mechanics but whether maintainers accept non-consuming manual cron runs as the permanent product contract.
