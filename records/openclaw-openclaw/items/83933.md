## Label Justifications

- `P1`: The PR targets a linked P1 cron bug where manual runs can silently destroy future scheduled one-shot jobs and lose scheduled work.

## Summary

Keep open for maintainer review: the patch appears to fix the linked cron manual-run data-loss bug and I found no blocking line-level defect, but it intentionally changes compatibility-sensitive manual-run semantics and adds a protocol-visible cron state field.

## What This Changes

The PR marks cron manual completions as manual, prevents manual runs from consuming scheduled delete/retry/counter/failure-alert state, adds `lastRunWasManual` to cron state/protocol schemas, and expands cron regression coverage.

## Best Possible Solution

Land the code after a maintainer explicitly accepts the non-consuming manual-run contract and durable `lastRunWasManual` state, with a short docs/protocol note if maintainers want the behavior documented in this PR.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main routes manual completion through `applyJobResult` without a manual discriminator, while the linked issue and PR proof describe the destructive `deleteAfterRun` outcome. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes for implementation shape, with a maintainer acceptance caveat: centralizing the manual/scheduled distinction in `applyJobResult` and passing `isManual` from manual call sites is the narrow owner-boundary fix. Rejecting manual runs or adding an override flag would be a larger product change than needed for the reported data-loss path.
