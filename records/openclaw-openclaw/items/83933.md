## Label Justifications

- `P1`: The linked bug can silently destroy scheduled one-shot cron jobs during manual verification, which is a serious user-facing automation regression.
- `merge-risk: 🚨 compatibility`: Existing manual cron runs that used to consume one-shots or update retry/alert state will behave differently after merge.
- `merge-risk: 🚨 session-state`: The PR changes persisted cron job state transitions, retry counters, next-run state, and adds `lastRunWasManual`.
- `merge-risk: 🚨 message-delivery`: Manual failure and skipped runs no longer emit failure alerts or mutate alert cooldown state, changing observable notification behavior.

## Summary

Keep open: current main still has the manual-run cron deletion/state-mutation bug, and this PR is the active, well-proven fix candidate, but merge still needs maintainer acceptance of the compatibility-sensitive manual-run contract and additive persisted/protocol state.

## What This Changes

The branch adds an `isManual` cron outcome path that preserves one-shot jobs and scheduled retry/alert state for manual runs, records `lastRunWasManual` in cron state/protocol, and updates cron regression coverage.

## Best Possible Solution

Land the shared manual-run discriminator after maintainers accept and document that manual cron runs are non-consuming and that `lastRunWasManual` is part of the gateway contract.

## Reproduction Assessment

Yes. Current main routes manual completion through `applyJobResult` without a manual discriminator, and that function deletes successful `at`/`deleteAfterRun` jobs and mutates retry/alert state.

## Solution Assessment

Yes, as a fix shape: `applyJobResult` is the shared owner boundary for scheduled and manual outcomes, so adding the manual discriminator there is narrower than duplicating policy in callers. The remaining question is product acceptance of the new manual-run contract.
