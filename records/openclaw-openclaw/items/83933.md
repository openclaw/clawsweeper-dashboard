## Summary

Keep this PR open: it targets the right destructive cron behavior, but the proposed guard only covers manual ok/skipped results and leaves manual one-shot error results able to consume the future scheduled fire; the external PR also lacks real behavior proof.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual-run call sites, and adds timer regression coverage for manual versus scheduled `deleteAfterRun` behavior.

## Best Possible Solution

Treat manual cron runs as non-consuming probes for every terminal status: preserve the stored job and future scheduled fire for manual ok, skipped, and error outcomes while leaving timer-driven delete/retry/disable behavior unchanged.

## Reproduction Assessment

Yes from source inspection: current main's manual-run completion path reaches the same `applyJobResult` logic that deletes successful `at` jobs with `deleteAfterRun:true`. I did not run a live gateway scenario because this review was read-only.

## Solution Assessment

No, not yet: the proposed direction is right, but the implementation only skips ok/skipped one-shot consumption. The safer fix is to make `isManual` bypass all one-shot delete/retry/disable consumption while keeping scheduled timer runs unchanged.
