## Summary

Keep open for maintainer review: current main and the latest release still have the manual-run deletion/counter behavior, and this PR is a proof-positive, likely correct fix, but it deliberately changes compatibility-sensitive cron manual-run and persisted state semantics.

## What This Changes

This PR threads manual-run awareness through cron result application, adds an observable `lastRunWasManual` state field, and expands cron tests so manual runs no longer consume one-shot jobs or mutate scheduled retry, alert, and backoff state.

## Best Possible Solution

Land this after maintainers explicitly accept the manual-run isolation contract and additive cron state field, while keeping failed-run `deleteAfterRun` policy in its separate issue.

## Reproduction Assessment

Yes at source level. Current main and `v2026.6.8` route manual completion through `applyJobResult` without a manual discriminator, and an ok `at` job with `deleteAfterRun: true` returns `shouldDelete`.

## Solution Assessment

Yes, with a maintainer policy gate. The shared `applyJobResult` discriminator is the clean fix point because both manual completion paths use it and scheduled callers keep default behavior, but the manual-versus-scheduled state contract needs explicit acceptance.
