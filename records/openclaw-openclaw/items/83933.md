## Summary

Keep open: current main and the latest release still have the reported manual-run deletion path, and this PR remains the active candidate fix. I found no definite line-level correctness bug in the patch, but merge readiness still depends on maintainer acceptance of the changed cron semantics and durable state/protocol contract.

## What This Changes

The PR marks manual cron completions, prevents manual runs from consuming `deleteAfterRun` jobs or scheduled retry/alert/backoff state, records `lastRunWasManual`, and updates cron regression and failure-alert coverage.

## Best Possible Solution

Land this only after maintainers accept manual cron runs as non-consuming verification runs and document the scheduled-only `deleteAfterRun` contract.

## Reproduction Assessment

Yes. Source inspection on current main shows manual cron completion calls `applyJobResult` without a manual discriminator, and successful `at` jobs with `deleteAfterRun: true` return `shouldDelete` and are removed.

## Solution Assessment

Yes, pending product acceptance. The shared `applyJobResult` discriminator is the narrow owner-boundary fix; the safer remaining path is to make the `lastRunWasManual` state/protocol contract and docs explicit before merge.
