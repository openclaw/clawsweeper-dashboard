## Summary

Keep open: current main still has the manual-run/deleteAfterRun bug and this PR is the viable fix candidate; no blocking line-level defect was found, but merge remains a maintainer decision because it changes cron manual-run semantics and adds protocol-visible state.

## What This Changes

The PR marks manual cron completions with `isManual`, prevents manual runs from consuming scheduled delete/retry/counter/failure-alert state, adds `lastRunWasManual` to cron state/protocol, and expands cron regression coverage.

## Best Possible Solution

Land the shared `applyJobResult` manual-run discriminator after maintainers approve the non-consuming manual-run contract and protocol field; keep failed-run deletion semantics tracked separately.

## Reproduction Assessment

Yes at source level: current main manual completion calls `applyJobResult` without a manual discriminator, and `applyJobResult` deletes successful at/deleteAfterRun jobs while also mutating counters and alerts; I did not run a live gateway repro in this read-only pass.

## Solution Assessment

Yes, pending maintainer policy acceptance: putting the manual discriminator in shared `applyJobResult` is the clean owner-boundary fix because CLI, RPC, and tool manual completions converge there; CLI-only rejection or warning would be too narrow.
