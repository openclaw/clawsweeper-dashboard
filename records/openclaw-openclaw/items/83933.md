## Summary

Keep open. This is a focused cron bug-fix PR with source-backed root cause, regression coverage, and Docker Gateway CLI proof, but it changes manual `cron run` state semantics enough that maintainers should explicitly accept the upgrade behavior before merge.

## What This Changes

The PR adds an `isManual` option to cron result handling, passes it from manual cron run paths, and adds regression coverage so manual runs stop deleting one-shot jobs or mutating scheduled retry, alert, cooldown, and backoff state.

## Best Possible Solution

Keep manual `cron run` as a non-consuming debug path that preserves scheduled execution state, while scheduled timer runs retain deleteAfterRun, retry, failure-alert, cooldown, and backoff behavior with focused regression coverage.

## Solution Assessment

Yes, the implementation targets the right boundary by threading an explicit manual-run discriminator from `ops.ts` into the central cron result state reducer. The remaining decision is semantic, not mechanical: maintainers should confirm manual debug runs should not participate in scheduled state mutation.
