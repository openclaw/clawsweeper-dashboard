## Summary

Keep this PR open. The direct manual-delete path is addressed, but the patch still records terminal manual status on preserved one-shot `at` jobs, so startup catch-up can skip the scheduled future fire after a restart; the PR also still lacks required real behavior proof.

## What This Changes

The PR adds an `isManual` flag to cron result application, passes it from manual cron run paths, and adds regression tests for manual versus scheduled `deleteAfterRun` handling.

## Best Possible Solution

Distinguish manual-run state from scheduled one-shot consumption so both live timer execution and startup catch-up preserve the future fire, then add focused `runMissedJobs` coverage and redacted real CLI/log proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: manual completion reaches `applyJobResult`, the PR preserves the job while leaving terminal `lastStatus`, and `runMissedJobs` skips terminal `at` jobs during startup catch-up.

## Solution Assessment

No, not yet. The `isManual` boundary is the right direction, but the fix must also prevent manual terminal status from being interpreted as scheduled one-shot consumption during restart catch-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve manual one-shots through startup catch-up:** `src/cron/service/timer.ts:926`
  - body: With this guard, a successful or skipped manual `at` job now stays enabled with its original `nextRunAtMs`, but `applyJobResult` still records `lastStatus`. On the next gateway startup after that scheduled time, `runMissedJobs` calls `isRunnableJob` with `skipAtIfAlreadyRan: true`, and the terminal-status check returns `false` for ok/skipped one-shots, so the preserved scheduled fire is silently skipped. Please either keep manual results from marking the scheduled one-shot as consumed or teach the startup guard to allow this preserved future fire, with a regression that runs through catch-up.
  - confidence: 0.92
