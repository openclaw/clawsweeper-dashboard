## Summary

Keep open for maintainer review before merge: the patch appears to fix the source-reproducible manual cron deletion/counter bug, but it intentionally changes compatibility-sensitive cron scheduler state and adds a persisted/protocol state field.

## What This Changes

The PR adds manual-run awareness to cron result application, records `lastRunWasManual`, updates gateway cron state schemas, and expands cron scheduler regression coverage.

## Best Possible Solution

Merge after maintainers explicitly accept the manual-versus-scheduled cron state policy, with normal CI/merge gates passing, and let the linked issue close from the merged fix.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.8` route manual cron completion through `applyJobResult` without a manual discriminator, so successful `deleteAfterRun` at-jobs can be removed and counters are updated. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, with policy acceptance: adding the discriminator at `applyJobResult` is the right owner boundary because manual and scheduled completions converge there while scheduled timer paths keep the default behavior. The remaining question is maintainer acceptance of the changed manual-run state contract, not a concrete code defect.
