## Label Justifications

- `P1`: The PR addresses a user-facing cron workflow that can silently destroy scheduled future fires and perturb retry state.
- `merge-risk: 🚨 compatibility`: Manual `cron run` will no longer consume one-shot jobs, which changes behavior for any existing workflow that depended on the old destructive semantics.
- `merge-risk: 🚨 session-state`: The patch deliberately changes persisted cron job state updates for counters, alert cooldowns, enabled state, and nextRunAtMs during manual runs.
- `merge-risk: 🚨 message-delivery`: Preserving a one-shot job after manual execution can cause a later scheduled delivery that current behavior would have removed.

## Summary

Keep this PR open: current main still has the manual-run deletion/counter bug, and the PR targets the right cron result-application boundary with scheduled callers left on the existing path. I found no blocking patch findings, but the semantic change should still get maintainer acceptance because it changes what a manual `cron run` can consume or advance.

## What This Changes

The PR adds a manual-run discriminator to cron result handling so manual `cron run` paths preserve `deleteAfterRun` jobs, retry/backoff counters, alert cooldowns, and scheduled next-run state while scheduled runs keep the existing state transitions.

## Best Possible Solution

Land the cron runtime fix once maintainers accept the manual-run isolation semantics and the exact PR head satisfies required checks, while keeping scheduled-run deletion/backoff behavior unchanged.

## Reproduction Assessment

Yes. Current main is source-reproducible through `finishPreparedManualRun`/`skipInvalidPersistedManualRun` calling applyJobResult without a manual discriminator, after which applyJobResult deletes successful `deleteAfterRun` at-jobs and mutates counters unconditionally.

## Solution Assessment

Yes. The patch fixes the owner boundary by making applyJobResult distinguish manual from scheduled results and wiring only manual callers to that mode; rejecting manual runs or only logging deletion would not fix counter/backoff pollution.
