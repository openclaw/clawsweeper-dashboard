## Summary

Keep this PR open: the fix is aimed at the right cron result path, but latest head still mutates existing scheduled error state during manual error runs and the supplied proof remains source-level rather than a patched real cron CLI/API run.

## What This Changes

The PR adds an `isManual` option to cron result handling, passes it from manual cron run paths, and adds regression tests so `deleteAfterRun` one-shot jobs are preserved during manual runs.

## Best Possible Solution

Preserve manual one-shot jobs in `applyJobResult`, leave existing scheduled error and alert counters untouched on manual outcomes, keep scheduled `deleteAfterRun` behavior unchanged, and verify with redacted patched cron CLI/API proof.

## Reproduction Assessment

Yes, by source inspection: current main manual run paths call `applyJobResult`, and successful `at` jobs with `deleteAfterRun` return `shouldDelete=true` without knowing the run was manual. I did not run a live current-main cron service reproduction in this read-only review.

## Solution Assessment

No, not yet. The `isManual` boundary is the right place to solve the bug, but the latest patch should stop decrementing existing error counters and needs real patched cron CLI/API proof before merge.
