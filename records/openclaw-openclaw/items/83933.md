## Summary

## What This Changes

The PR adds manual-run discrimination to cron result application, persists `lastRunWasManual`, exposes it in the gateway cron schema, and updates cron tests for manual versus scheduled state transitions.

## Best Possible Solution

Land this PR, or a functionally equivalent shared result-application discriminator, after maintainers explicitly accept manual-run isolation and the API-visible scheduler state bit.

Yes. Source inspection shows current main manual completion calls `applyJobResult` without a manual discriminator, and an ok `at` job with `deleteAfterRun: true` returns `shouldDelete`; I did not run a live gateway repro during this read-only review.

## Solution Assessment

Yes, with maintainer policy acceptance. Putting the discriminator in shared result application is the narrow owner-boundary fix because both manual completion paths converge there while scheduled paths keep the default behavior; rejecting manual runs or adding an override flag would be a product alternative, not a smaller code repair.
