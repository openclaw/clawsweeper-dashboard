## Summary

Keep open: the PR is a plausible fix direction, but it still has a P1 cron-state defect where a successful manual due run can preserve the job in `applyJobResult` and then lose the scheduled slot during the final maintenance recompute. The contributor also supplied useful Docker/Gateway proof, but it does not show the latest-head successful manual `deleteAfterRun` due-run readback after this recompute path.

## What This Changes

The PR adds manual-run state to cron result handling, cron state schemas, and regression tests so manual `cron run` calls do not consume `deleteAfterRun` jobs or mutate scheduled retry, backoff, and alert counters.

## Best Possible Solution

Land only after the manual due success path preserves the job and `nextRunAtMs` through final recompute, has a regression that fails on this PR head, and has latest-head redacted CLI/Gateway proof for `cron run`, `cron show`, and `cron list`.

## Reproduction Assessment

Yes. Source inspection shows current main deletes successful manual `deleteAfterRun` at-jobs without distinguishing manual runs, and PR-head source shows the remaining successful manual due recompute failure path.

## Solution Assessment

No. The `isManual` flag is an acceptable direction, but the current implementation is incomplete because final maintenance recompute can clear the preserved due slot; the narrow fix is to preserve or exempt the just-manual-run job through that recompute and prove it with an integrated success-path test.
