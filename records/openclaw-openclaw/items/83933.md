- `P1`: The PR addresses a destructive cron workflow that can silently remove scheduled one-shot delivery state during manual verification.

## Summary

Keep this PR open for maintainer acceptance: the patch is a well-proven fix for destructive manual cron behavior, but it intentionally changes existing `cron run` semantics and adds a public/persisted cron state bit.

## What This Changes

The PR changes cron manual-run result handling so manual completions do not consume `deleteAfterRun` jobs or mutate scheduled retry, backoff, or failure-alert state.

## Best Possible Solution

Land the non-consuming manual-run behavior only after maintainers accept that manual cron runs are verification/debug invocations and capture the new scheduled-vs-manual contract in docs or release context.

## Reproduction Assessment

Yes from source inspection: current main's manual completion path reaches `applyJobResult` without a manual discriminator, and `applyJobResult` performs `deleteAfterRun` and scheduled retry/alert mutations unconditionally.

## Solution Assessment

Yes, with maintainer acceptance needed: carrying an explicit manual-run discriminator into the shared result applicator is the right owner-boundary fix, but the changed cron contract and additive state/protocol field need maintainer approval.
