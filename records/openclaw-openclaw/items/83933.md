Full review comments:

- **[P1] Guard manual errors before at-job retry handling:** `src/cron/service/timer.ts:993-995`
  - body: When `isManual` is true, the counter block above skips incrementing `consecutiveErrors`, but the at-job error branch still runs. For a fresh manual error, `consecutive` is `undefined`, so the retry check fails and the code disables the job at lines 1016-1017; with an existing count it can instead overwrite `nextRunAtMs`. That still lets a manual verification run mutate the scheduled one-shot state this PR is trying to preserve.
  - confidence: 0.94
- **[P2] Keep manual success from clearing alert cooldown:** `src/cron/service/timer.ts:963-967`
  - body: The `ok` branch now protects the counters with `!opts?.isManual`, but `lastFailureAlertAtMs` is still cleared for manual successes. That means a manual verification run can reset the scheduled failure-alert cooldown and cause the next scheduled failure to alert earlier than intended; keep the cooldown assignment inside the same scheduled-only boundary or define separate manual alert semantics with tests.
  - confidence: 0.83
