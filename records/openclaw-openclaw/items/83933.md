## Summary

Keep open: the PR is the right fix direction, but current PR head still lets a successful manual `due` run on an expired `at` job survive as a row while losing its scheduled `nextRunAtMs` during the final maintenance recompute.

## What This Changes

The PR marks manual cron run outcomes with `isManual`/`lastRunWasManual`, preserves manual-run scheduler state in `applyJobResult`, updates the Gateway cron state schema, and expands cron regression coverage.

## Best Possible Solution

Carry the manual-run invariant through `applyJobResult`, maintenance recompute, runnable checks, Gateway schema readback, and regression coverage so manual verification records an outcome without consuming or suppressing the scheduled slot.

## Reproduction Assessment

Yes, source-level. On PR head, a successful manual `due` run can set `lastRunStatus: "ok"` and then enter maintenance recompute, whose `at`-job logic returns `undefined` when the scheduled timestamp is not after `lastRunAtMs`; I did not run tests because the review was read-only.

## Solution Assessment

No. The `isManual` marker is the right direction, but the fix is incomplete until the recompute path and a full manual `due` integration test prove that manual terminal outcomes do not consume or suppress scheduled `at` slots.
