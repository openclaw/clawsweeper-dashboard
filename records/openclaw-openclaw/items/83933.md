## Summary

Keep open: this is a focused fix for the linked cron data-loss bug, but the PR head still lets manual error runs emit failure alerts and stamp cooldown before the new manual rollback branch runs, and the external PR still lacks real queued-run proof.

## What This Changes

The PR adds an `isManual` option to cron result handling, marks manual run completion paths as manual, and adds regression tests for preserving `deleteAfterRun` one-shot jobs during manual runs.

## Best Possible Solution

Keep the manual-run discriminator, suppress or explicitly accept manual error alert/cooldown effects before result alerting, then merge after redacted real `openclaw cron run` proof confirms preserved job state and later scheduled eligibility.

## Reproduction Assessment

Yes by source inspection, though not by a live run in this read-only review. Current main routes manual cron completion through `applyJobResult` without a manual discriminator, and successful one-shot `deleteAfterRun` jobs return the deletion path there.

## Solution Assessment

No, not yet. The manual-run discriminator is the right boundary, but the PR should prevent manual error alert/cooldown side effects before they happen and add real CLI/log proof before merge.
