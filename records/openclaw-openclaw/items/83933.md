## Summary

Keep this PR open: it is still the active candidate for the manual `deleteAfterRun` cron bug, but current head still has a blocking cron scheduling defect and the proof does not show the successful due one-shot path after the latest rebase.

## What This Changes

The branch adds a manual-run discriminator to cron result application, persists/exposes `lastRunWasManual`, and expands cron regression, failure-alert, restart, and one-shot tests.

## Best Possible Solution

Keep the shared `isManual` result discriminator, carry that invariant through post-run recompute/persist, and require current-head CLI or equivalent service proof for due `deleteAfterRun` success before merge.

## Reproduction Assessment

Yes. Source inspection on PR head shows a successful due `at` manual run can preserve `nextRunAtMs` in `applyJobResult`, then the later maintenance recompute can treat that same slot as already executed and clear it.

## Solution Assessment

No. The `applyJobResult` discriminator is the right layer, but the current branch does not preserve that invariant through post-run recompute; the safer fix is to skip the just-manual job in that recompute or make expired-slot recompute ignore manual terminal state.
