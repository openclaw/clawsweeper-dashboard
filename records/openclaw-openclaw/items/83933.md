## Summary

Keep open: the PR fixes a real cron manual-run bug that is still present on current main and v2026.6.8, and I found no concrete blocking patch defect; the remaining blocker is maintainer acceptance of the compatibility-sensitive manual-run/session-state contract.

## What This Changes

Threads an `isManual` flag through cron manual completion, preserves scheduled delete/retry/backoff/alert state for manual runs, adds `lastRunWasManual` to cron state/protocol schemas, and expands cron regression coverage.

## Best Possible Solution

Land this PR, or a narrow equivalent, after maintainers explicitly accept manual cron runs as non-destructive verification/debug operations that record their outcome without mutating scheduled execution state.

## Reproduction Assessment

Yes. Current main and v2026.6.8 source show manual completion uses the same `applyJobResult` deletion/counter path as scheduled runs, so a successful manual `at` job with `deleteAfterRun: true` can delete the scheduled job.

## Solution Assessment

Yes, with maintainer acceptance. The shared `applyJobResult` discriminator is the clean owner-boundary fix; rejecting manual runs or adding a CLI-only override would leave duplicated policy and would not address counter/backoff pollution across manual completion paths.
