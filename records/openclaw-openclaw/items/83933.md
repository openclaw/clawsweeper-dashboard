## Summary

Keep open: current main still has the manual `deleteAfterRun` cron bug, and this PR is the active fix candidate, but the latest head still has a blocking full-path cron state defect after manual-run recompute.

## What This Changes

The branch adds a manual-run discriminator to cron result application, persists and exposes `lastRunWasManual` in cron job state, and expands cron regression, failure-alert, restart, and one-shot coverage.

## Best Possible Solution

Keep the shared `isManual` result discriminator, but carry the manual-run invariant through maintenance recompute/persist and require current-head CLI or equivalent service proof for due `deleteAfterRun` success.

## Reproduction Assessment

Yes. Source inspection on PR head shows a successful due `at` manual run can preserve `nextRunAtMs` in `applyJobResult`, then the later maintenance recompute can treat that same slot as already executed and clear it.

## Solution Assessment

No. The `applyJobResult` discriminator is the right layer, but the current branch does not preserve that invariant through post-run recompute; the safer fix is to skip the just-manual job in that recompute or make expired-slot recompute ignore manual terminal state.
