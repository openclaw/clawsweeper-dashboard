## Summary

Keep this PR open: the reported cron data-loss bug is valid and the `isManual` seam is the right direction, but the patch currently regresses configured manual-run failure alerts, still mutates alert cooldown state on manual success, and lacks live patched CLI/Gateway proof.

## What This Changes

The PR adds an `isManual` result-application path for cron manual runs so `deleteAfterRun` at-jobs are preserved and scheduled retry counters are not updated by manual executions.

## Best Possible Solution

Land a bounded manual-result boundary that preserves `deleteAfterRun` jobs and scheduled counters without regressing failure-alert delivery/cooldown, backed by redacted live CLI/Gateway proof and focused regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main routes manual completion through `applyJobResult` without a manual flag, and that function deletes successful `deleteAfterRun` at-jobs solely from schedule kind, flag, and status.

## Solution Assessment

No, not yet. The `isManual` boundary is the right shape, but the implementation needs to preserve or deliberately redesign failure-alert behavior and cooldown state before it is the safest fix.
