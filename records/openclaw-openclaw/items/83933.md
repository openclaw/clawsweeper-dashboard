## Label Justifications

- `P1`: The PR fixes a user-facing cron bug where manual verification can silently consume scheduled one-shot work and lose future fires.
- `merge-risk: 🚨 compatibility`: Merging changes existing manual cron-run semantics from consuming scheduled state to non-consuming verification behavior.
- `merge-risk: 🚨 session-state`: The diff changes persisted cron job state and adds the gateway-visible `lastRunWasManual` field.
- `merge-risk: 🚨 message-delivery`: Preserving a one-shot job after manual execution can allow a later scheduled run to enqueue or deliver messages that current behavior would have removed.

## Summary

Keep open: the latest head appears technically sound and fixes the previous test-helper blocker, but it still needs maintainer acceptance of compatibility-sensitive cron semantics and the new persisted, gateway-visible manual-run state.

## What This Changes

The branch makes manual cron runs non-consuming by passing `isManual` into cron result application, preserving `deleteAfterRun` jobs and scheduled retry state while adding `lastRunWasManual` protocol/state coverage and cron regression tests.

## Best Possible Solution

Land this after a maintainer explicitly accepts non-consuming manual runs and the additive `lastRunWasManual` state contract, while keeping scheduled runs as the only path that consumes `deleteAfterRun` jobs.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` route manual completion through `applyJobResult` without an `isManual` discriminator, and the success `deleteAfterRun` gate removes `at` jobs.

## Solution Assessment

Mostly yes. The shared result-application boundary is the right owner boundary for manual-versus-scheduled semantics; the remaining question is maintainer acceptance of the compatibility and state-contract change.
