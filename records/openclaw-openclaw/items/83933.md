## Label Justifications

- `P1`: The linked bug can silently consume future scheduled cron fires from manual runs, which is an urgent scheduler workflow regression.
- `merge-risk: 🚨 compatibility`: The PR intentionally changes existing manual `openclaw cron run` semantics for deleteAfterRun and retry/backoff side effects.
- `merge-risk: 🚨 session-state`: The PR adds and consumes persisted cron state indicating whether the last run was manual.
- `merge-risk: 🚨 message-delivery`: Cron jobs can deliver scheduled agent/channel output, so changing deletion, retry, and alert behavior can change whether scheduled messages or failure notifications fire later.

## Summary

Keep open: the patch appears mechanically correct and well-proven, but it intentionally changes compatibility-sensitive cron manual-run semantics and adds a persisted gateway protocol state field, so maintainer acceptance remains the blocker.

## What This Changes

The branch adds an `isManual` outcome flag, records `lastRunWasManual`, gates manual cron run deletion/retry/alert state, updates gateway cron schema/types, and expands cron regression tests.

## Best Possible Solution

Have maintainers explicitly accept manual cron runs as non-consuming verification runs, then land the patch with a short docs/protocol note while keeping adjacent deleteAfterRun failed-run and due-batch issues separate.
