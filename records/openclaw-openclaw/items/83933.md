## Summary

Keep open: this is a focused candidate fix for the linked manual cron `deleteAfterRun` data-loss bug, and current main/latest release still have the bug. The patch looks technically sound, but it changes compatibility-sensitive manual cron semantics and adds protocol-visible persisted state, so it needs maintainer acceptance before merge.

## What This Changes

The PR adds manual-run isolation to cron outcome handling so manual `cron run` does not consume `deleteAfterRun` jobs or mutate scheduled retry, skip, backoff, disable, or failure-alert state.

## Best Possible Solution

Land the shared manual-run discriminator after maintainer acceptance of the non-consuming manual cron contract and the durable lastRunWasManual protocol field.

## Reproduction Assessment

Yes, source-reproducible on current main and v2026.6.11: manual completion routes through applyJobResult without a manual discriminator, and applyJobResult deletes successful at/deleteAfterRun jobs.

## Solution Assessment

Yes as a fix shape: applyJobResult owns both scheduled and manual completion state, and PR head also updates restart catch-up to distinguish manual terminal history. The remaining blocker is maintainer acceptance of the behavior and durable protocol state.
