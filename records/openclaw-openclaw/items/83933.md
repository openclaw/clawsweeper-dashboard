## Summary

Keep open: the cron state fix is useful and focused, but the latest branch still lets manual error runs emit failure-alert/cooldown side effects before the manual rollback branch runs, and the external PR still lacks real `openclaw cron run` proof.

## What This Changes

This PR adds an `isManual` flag to cron result application, marks manual run completion paths as manual, and adds regression tests so manual cron runs preserve `deleteAfterRun` one-shot jobs while scheduled runs still delete them.

## Best Possible Solution

Keep the manual-run discriminator, suppress manual error alert/cooldown side effects before they fire, and merge after redacted queued-run proof confirms one-shot job preservation and later scheduled eligibility.

## Reproduction Assessment

Yes by source inspection, not by a live run in this read-only review. Current main routes manual cron completion through `applyJobResult` without a manual discriminator, and successful one-shot `deleteAfterRun` jobs return the deletion path there.

## Solution Assessment

No. The manual-run discriminator is the right boundary, but the PR should suppress manual error alert/cooldown side effects before they happen and add real CLI proof before merge.
