## Summary

Keep open: the patch is focused and proof-positive with no blocking code finding, but it changes compatibility-sensitive cron manual-run semantics and adds a gateway-visible persisted state marker that should get maintainer acceptance before merge.

## What This Changes

The PR threads an `isManual` flag through cron manual-run completion, prevents manual runs from deleting one-shot jobs or changing scheduled retry/backoff/alert state, adds `lastRunWasManual` to cron state schemas, and expands cron regression coverage.

## Best Possible Solution

Land this after maintainer acceptance that manual cron runs are non-destructive verification and that the additive state marker is the right cron state contract; keep failed-run deletion policy in the separate related issue.

Yes. Source inspection shows current main sends manual completion through `applyJobResult` without a manual discriminator, and a successful `at` job with `deleteAfterRun: true` returns `shouldDelete`.

## Solution Assessment

Yes, with maintainer acceptance. The shared `applyJobResult` discriminator is the narrow owner-boundary fix; rejecting manual runs or adding command-specific delete checks would add surface area or duplicate scheduling rules.
