## Summary

Keep open: the follow-up commit addresses the earlier manual error disable/retry finding, but the PR still leaves a startup-catch-up path where a preserved manual one-shot can skip its scheduled fire, and the contributor has not added real behavior proof.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual cron run paths, and adds regression tests for manual versus scheduled `deleteAfterRun` handling.

## Best Possible Solution

Distinguish manual-run state from scheduled one-shot consumption so live timer and startup catch-up both preserve the future fire, then add focused restart coverage and redacted real CLI/log proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: manual completion reaches `applyJobResult`, and the PR preserves the job while leaving terminal `lastStatus` that startup catch-up skips for `at` jobs. I did not run a live gateway scenario because this review was read-only.

## Solution Assessment

No, not yet: the `isManual` guard is the right boundary, but the fix must also prevent manual terminal status from being interpreted as scheduled one-shot consumption during restart catch-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve manual one-shots through startup catch-up:** `src/cron/service/timer.ts:926`
  - body: With this guard, a successful or skipped manual `at` job now stays enabled with its original `nextRunAtMs`, but `applyJobResult` still records `lastStatus`. On the next gateway startup after that scheduled time, `runMissedJobs` calls `isRunnableJob(..., skipAtIfAlreadyRan: true)`, and the existing `lastStatus` check returns `false` for ok/skipped one-shots, so the preserved scheduled fire is silently skipped. Please either keep manual results from marking the scheduled one-shot as consumed or teach the startup guard to allow this preserved future fire, with a regression that runs through catch-up.
  - confidence: 0.92
