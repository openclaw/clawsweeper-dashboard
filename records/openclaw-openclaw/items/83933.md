## Summary

Keep open: source review shows current main can delete or disable `deleteAfterRun` one-shot cron jobs through manual result application, and this PR's code path looks like the right focused fix, but the external PR still lacks real queued `openclaw cron run` proof before merge.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual run paths, and adds regression tests so manual one-shot runs do not delete, disable, or reschedule the future job.

## Best Possible Solution

Land the manual-run discriminator after real queued-run proof confirms persisted job state, run history, and later scheduled eligibility for the same job id.

## Reproduction Assessment

Yes. Source inspection on current main shows manual cron completion reaches `applyJobResult` without a manual discriminator, so `deleteAfterRun:true` one-shot success can delete the job and skipped/ok one-shot results can disable it; I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Yes, with proof still required. The PR fixes the result-application boundary used by manual cron runs without changing scheduled timer paths, but it should not merge until real queued-run behavior is shown.
