## Summary

Keep this PR open for maintainer review: current main still has no manual-run discriminator in cron result handling, while the branch provides a focused fix with sufficient terminal/Docker proof and no discrete blocking code-review finding found.

## What This Changes

The PR marks manual cron run completion paths as manual, updates cron result application to keep manual runs from deleting `deleteAfterRun` jobs or mutating scheduled retry/backoff/alert state, and adds cron regression/failure-alert coverage.

## Best Possible Solution

Land the manual-run isolation behavior if maintainers agree that `cron run` is a non-destructive debug/verification path, keeping scheduled runs as the only path that consumes one-shot jobs and advances scheduled retry state.

## Reproduction Assessment

Yes from source inspection: current main has no manual discriminator in `applyJobResult`, and manual completion call sites pass only `preserveSchedule`, so manual runs can follow scheduled deletion and counter paths. I did not run the live repro in this read-only review.

## Solution Assessment

Yes, if maintainers accept the product contract: threading `isManual` through the existing result-application boundary is narrower than adding a new CLI flag and keeps scheduled behavior unchanged. The remaining question is whether the compatibility change is intentional for existing users.
