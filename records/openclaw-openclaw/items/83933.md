## Label Justifications

- `P1`: The PR addresses a user-facing cron bug where manual verification can silently destroy scheduled one-shot work and lose future fires.
- `merge-risk: 🚨 compatibility`: Manual cron runs will no longer have side effects that existing scripts may have relied on, including consuming one-shot jobs and resetting retry or alert state.

## Summary

Keep open: the PR targets a source-reproducible cron data-loss bug and the implementation/proof look sound, but it is not a cleanup-close candidate because it changes compatibility-sensitive manual-run semantics and adds observable cron state that needs maintainer acceptance before merge.

## What This Changes

The PR threads manual-run awareness through cron result application, adds `lastRunWasManual` cron state, and expands cron regression coverage so manual runs no longer consume one-shot jobs or mutate scheduled retry, alert, and backoff state.

## Best Possible Solution

Land this PR after maintainers explicitly accept the manual-versus-scheduled cron state contract, while keeping failed-run deletion policy in https://github.com/openclaw/openclaw/issues/88197 separate.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.8` shows manual completion calls the same success-only `deleteAfterRun` gate as scheduled completion, and the PR body includes Docker Gateway CLI proof for the after-fix path.

## Solution Assessment

Yes, with maintainer acceptance. Putting the discriminator in shared `applyJobResult` is the right owner boundary because delete, counter, alert, backoff, and restart-catch-up semantics all flow through that result application path; the remaining question is compatibility policy, not a narrower code defect.
