## Summary

Keep this PR open: the fix direction is plausible, but the current head fails type/test checks, suppresses existing manual-run failure alert behavior, and still lacks real patched Gateway/CLI proof.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual cron-run paths, and adds timer regressions so manual runs preserve `deleteAfterRun` jobs and counters.

## Best Possible Solution

Land a narrower manual-run result boundary that preserves `deleteAfterRun` at-jobs without breaking existing failure alert/skip accounting, after CI is green and redacted patched Gateway/CLI proof is posted.

## Reproduction Assessment

Yes, source-reproducible. Current main manual-run completion reaches `applyJobResult` without a manual discriminator, and `applyJobResult` deletes successful `deleteAfterRun` at-jobs regardless of how the run was triggered.

## Solution Assessment

No, not yet. The `isManual` boundary is the right direction, but this implementation is too broad because it breaks failure alert accounting and currently fails type/test checks.
