## Summary

Keep this PR open: the core manual-run fix is useful and supported by proof, but the current branch still leaves a restart catch-up hole where a preserved one-shot job can miss its scheduled fire after gateway downtime.

## What This Changes

The PR adds an `isManual` path to cron result application, passes it from manual `cron run` completion paths, and adds regression coverage for preserving one-shot jobs, counters, retry/backoff state, and failure-alert state during manual runs.

## Best Possible Solution

Keep the manual-run isolation, but add a focused restart catch-up repair so manual terminal state cannot suppress the preserved scheduled at-job slot, then have maintainers explicitly accept the manual-run compatibility contract before merge.

Yes. Current main has no manual-run discriminator in applyJobResult, and the PR discussion includes Docker CLI proof showing manual `cron run` can consume or perturb scheduled cron state before the fix.
