## Summary

Keep open: the central fix is still not merge-ready because a successful manual due run can preserve the job in `applyJobResult` and then lose the due `nextRunAtMs` in the post-run maintenance recompute; the supplied proof also does not cover that latest-head success path.

## What This Changes

The branch changes cron manual-run result handling to pass an `isManual` flag, record `lastRunWasManual`, update gateway cron state schemas, and expand cron regression coverage.

## Best Possible Solution

Fix the post-manual-run recompute boundary so manual completions never count as executed scheduled slots, then prove the latest head with a successful `cron run` due `deleteAfterRun` job plus `cron show`, `cron list`, and run-history readback.

## Reproduction Assessment

## Solution Assessment

No. The `isManual` guard is a plausible fix layer, but it is incomplete until the final manual-run recompute path also preserves the scheduled slot and the service-level success case is covered.

Full review comments:

- **[P1] Preserve successful manual due slots after recompute:** `src/cron/service/ops.ts:948`
  - body: This marks every prepared manual completion as manual, but a successful due at-job still flows into the post-run maintenance recompute. When the manual run starts at the due `nextRunAtMs`, `applyJobResult` sets `lastRunAtMs` to that same timestamp; the later `recomputeNextRunsForMaintenance(..., { recomputeExpired: true })` sees `lastRunAtMs >= nextRunAtMs` and recomputes the past at-job to no next slot. That leaves the job enabled but unscheduled, so the scheduled fire is still silently lost. Please preserve the manual target's slot through that recompute and add a `cron.run(job.id, "due")` success regression that fails on this head.
