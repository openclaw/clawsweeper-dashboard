## Summary

Keep open: current main still has the manual-run deletion and counter-mutation behavior, and this branch addresses it with strong proof, but the additive cron state/protocol surface and manual-run alert semantics need maintainer acceptance before merge.

## What This Changes

The branch marks manual cron completions, skips `deleteAfterRun` and scheduled retry/backoff/failure-alert state changes for manual runs, records `lastRunWasManual`, and expands cron regression coverage.

## Best Possible Solution

Land this after cron owners accept the `lastRunWasManual` state/protocol contract and the manual-run isolation semantics; otherwise narrow the public state surface or preserve manual alert behavior explicitly.

## Reproduction Assessment

Yes at source level: current main's manual run paths call `applyJobResult` without a manual discriminator, and that function unconditionally deletes successful `deleteAfterRun` at-jobs and mutates counters. I did not run the CLI locally because this review is read-only.

## Solution Assessment

Yes, with maintainer sign-off: carrying an explicit manual-run discriminator through `ops.ts` to `applyJobResult` is the maintainable owner-boundary fix because scheduled callers keep the default path. The open product/API choice is whether `lastRunWasManual` should be public/patchable and whether manual errors should suppress failure alerts.
