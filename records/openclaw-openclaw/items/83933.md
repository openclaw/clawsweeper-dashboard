## Summary

Keep this PR open: the at-job deleteAfterRun direction is useful and the latest Docker CLI proof is meaningful, but manual every/cron errors still flow into the generic error-backoff branch and can rewrite scheduled nextRunAtMs during a debug run.

## What This Changes

The PR adds an isManual option to cron result application, marks manual run completions as manual, and adds timer/failure-alert regression coverage for deleteAfterRun, counters, alerts, and at-job state.

## Best Possible Solution

Land a complete manual-run boundary in applyJobResult that preserves deleteAfterRun jobs and scheduled counters/backoff/cooldown/nextRunAtMs for all schedule kinds while leaving scheduled-run deletion, retries, and alerts unchanged.

Yes, source-level reproduction is high confidence: current main manual run paths call applyJobResult without a manual discriminator, and current applyJobResult deletes deleteAfterRun at-jobs on ok results while mutating retry counters. I did not run tests in this read-only review.

No, not yet. The isManual boundary is the right approach for at-jobs, but the PR still lets manual every/cron errors run the generic backoff branch and rewrite nextRunAtMs.
