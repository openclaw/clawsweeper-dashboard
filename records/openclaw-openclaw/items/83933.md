## Summary

This PR should stay open for maintainer review: the implementation coherently separates manual cron runs from scheduled timer runs, but it intentionally changes existing `deleteAfterRun`/retry/alert semantics and adds a durable cron state/protocol field.

## What This Changes

The branch marks manual cron completions, prevents them from consuming `deleteAfterRun` jobs or scheduled retry/alert/backoff state, records `lastRunWasManual`, and updates cron regression/failure-alert coverage.

## Best Possible Solution

Approve and document manual cron runs as non-consuming verification runs, with `lastRunWasManual` as the durable state/protocol marker, before landing this implementation.

## Reproduction Assessment

Yes. Current main's manual completion path calls `applyJobResult` without a manual discriminator, and `applyJobResult` deletes successful `at` jobs with `deleteAfterRun: true`; I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainers accept the product contract. The shared `applyJobResult` discriminator is the right implementation layer if manual runs are meant to be non-consuming, but that behavior and the new state/protocol field need maintainer approval before merge.
