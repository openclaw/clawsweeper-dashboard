## Label Justifications

- `P1`: The PR addresses a destructive manual cron workflow that can silently remove scheduled one-shot delivery state.
- `merge-risk: 🚨 compatibility`: Manual `cron run` semantics change for existing operators who may currently expect a force-run success to consume a `deleteAfterRun` one-shot job.
- `merge-risk: 🚨 session-state`: The patch adds persisted cron state and changes how manual outcomes affect retry, backoff, counters, alerts, and next-run state.
- `merge-risk: 🚨 message-delivery`: Preserving a one-shot after manual verification means scheduled delivery can still happen later where current behavior would remove it.

## Summary

Keep open: the patch appears correct and well proven, but it intentionally changes compatibility-sensitive cron manual-run semantics and adds an additive persisted/protocol state field, so maintainer acceptance remains the merge gate.

## What This Changes

The PR marks cron manual completions as manual so they do not consume `deleteAfterRun` one-shots or mutate scheduled retry, backoff, failure-alert, or next-run state.

## Best Possible Solution

Land the non-consuming manual-run behavior after maintainers accept it as the cron contract, with docs or release context capturing how manual runs differ from scheduled executions.

## Reproduction Assessment

Yes. Current main's manual completion path reaches `applyJobResult` without a manual discriminator, and `applyJobResult` can delete a successful at/deleteAfterRun job and mutate scheduled counters/state.

## Solution Assessment

Yes, with maintainer acceptance needed. Carrying an explicit manual-run discriminator into the shared result applicator is the clean owner-boundary fix; rejecting manual runs at only the CLI/RPC edge would miss shared completion and queued/manual paths.
