## Label Justifications

- `P1`: The linked bug can silently destroy scheduled one-shot cron jobs during manual verification, which is a serious user-facing automation regression.
- `merge-risk: 🚨 compatibility`: Existing manual cron runs that used to consume one-shots or update retry/alert state will behave differently after merge.
- `merge-risk: 🚨 session-state`: The PR changes persisted cron job state transitions, retry counters, next-run state, and adds `lastRunWasManual`.
- `merge-risk: 🚨 message-delivery`: Manual failure and skipped runs no longer emit failure alerts or mutate alert cooldown state, changing observable notification behavior.

## Summary

Keep open for maintainer review: the patch is technically coherent and well-proven, but it intentionally changes manual cron run semantics and durable cron state/protocol, so maintainers should accept and document that contract before merge.

## What This Changes

The branch adds an `isManual` cron outcome path, marks manual completions, adds `lastRunWasManual` to cron state/protocol, and updates cron regression and failure-alert tests.

## Best Possible Solution

Land the shared manual-run discriminator after maintainers accept and document that manual cron runs are non-consuming and that `lastRunWasManual` is part of the gateway contract.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.6.11` route manual completion through `applyJobResult` without a manual discriminator, and that function deletes successful `at`/`deleteAfterRun` jobs and mutates retry/alert state.

## Solution Assessment

Yes, as a fix shape: `applyJobResult` is the shared owner boundary for scheduled and manual outcomes, so adding the manual discriminator there is narrower than duplicating policy in callers. The remaining question is maintainer acceptance of the manual-run contract and durable protocol field.
