## Summary

Keep open: the branch still has a P1 correctness blocker because successful manual due one-shot runs can survive `applyJobResult` but then lose `nextRunAtMs` during the post-run maintenance recompute.

## What This Changes

The PR changes cron manual-run result handling to mark manual outcomes, avoid `deleteAfterRun` deletion, preserve scheduled counters/backoff state, and expose `lastRunWasManual` in cron state schemas.

## Best Possible Solution

Fix the post-manual recompute path so successful manual due at-jobs keep their scheduled `nextRunAtMs`, then add focused regression coverage and refreshed redacted gateway CLI proof for that scenario.

## Reproduction Assessment

Yes. Source inspection shows current main has the original manual-run deletion path, and PR-head source shows a remaining successful manual due path where maintenance recompute can still clear the scheduled at-slot.

## Solution Assessment

No. The shared `applyJobResult` discriminator is the right layer, but the implementation remains incomplete until the post-manual reload/recompute path preserves successful manual due slots too.
