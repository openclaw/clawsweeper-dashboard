## Summary

Keep open for maintainer review: the PR is a focused cron-state fix with no blocking code findings, but it intentionally changes manual `cron run` lifecycle semantics and needs an explicit compatibility decision before merge.

## What This Changes

The PR adds an `isManual` option through manual cron completion paths so manual runs preserve `deleteAfterRun` jobs and scheduled retry, alert, cooldown, and backoff state.

## Best Possible Solution

Land the central `isManual` boundary if maintainers accept that manual cron runs are verification/debug runs that must not consume scheduled lifecycle state.

## Reproduction Assessment

Yes. Current main routes manual completions into `applyJobResult` without a manual discriminator, and that function deletes successful `deleteAfterRun` at-jobs plus mutates retry counters unconditionally; I did not run tests in this read-only review.

## Solution Assessment

Yes, if maintainers accept the policy change. A central `isManual` discriminator keeps scheduled runs on the existing path while only manual callers preserve scheduled lifecycle state.
