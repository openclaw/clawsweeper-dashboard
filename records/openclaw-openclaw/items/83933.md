## Summary

Keep open. The linked cron bug is real and the `deleteAfterRun` direction is plausible, but this PR currently suppresses existing manual-run failure-alert behavior, causing relevant CI failures, and still lacks patched Gateway/CLI proof of the real `openclaw cron run` flow.

## What This Changes

The PR adds an `isManual` result option for cron manual runs, skips `deleteAfterRun`/at-job retry state changes for manual results, and adds timer regression coverage.

## Best Possible Solution

Land a narrow manual-run boundary that preserves one-shot scheduled fires without regressing configured failure alerts, then require redacted patched Gateway/CLI proof showing manual preservation and scheduled deletion behavior.

## Reproduction Assessment

Yes, source-reproducible. Current main manual-run completion calls `applyJobResult` without a manual discriminator, and `applyJobResult` deletes successful `deleteAfterRun` at-jobs regardless of whether the run was manual.

## Solution Assessment

Not yet. The `isManual` boundary is the right shape for the one-shot deletion bug, but the current patch also suppresses existing manual-run failure alerts and needs real patched Gateway/CLI proof before merge.
