## Summary

Keep open: the PR fixes the linked cron state path in the right area, but the latest head still rolls back manual error state after failure alerts can fire, and the external PR has no real after-fix behavior proof.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual run paths, and adds regression tests so manual `deleteAfterRun` one-shot runs preserve the scheduled job.

## Best Possible Solution

Keep the manual-run discriminator, move or suppress manual error alert side effects before they occur, then merge after redacted queued-run proof confirms job preservation and later scheduled eligibility.

## Reproduction Assessment

Yes by source inspection, though not by a live run in this read-only review. Current main routes manual cron completion through `applyJobResult` without a manual discriminator, and that function deletes successful `deleteAfterRun` one-shot jobs or disables non-deleted one-shot ok/skipped jobs.

## Solution Assessment

Not yet. The manual-run discriminator is the right shape for the core bug, but the manual error path must avoid alert/cooldown side effects before rollback and the PR still needs real queued-run proof.

Full review comments:

- **[P2] Guard manual errors before emitting failure alerts:** `src/cron/service/timer.ts:933`
  - body: For manual at-job errors, this branch runs after the generic error block has already incremented `consecutiveErrors` and called `maybeEmitFailureAlert`. With `failureAlert.after: 1` or an existing threshold, a manual verification failure can still send a failure notification and set `lastFailureAlertAtMs`, so the scheduled fire starts with the counter rolled back but alert cooldown state already written. Move the manual guard before the counter/alert side effects or explicitly skip failure alerts for manual-preserved one-shot results.
