## Summary

Keep this PR open: the core bug is valid and the patch is close, but the current head still lets manual one-shot error/skipped outcomes run failure-alert bookkeeping before the manual-run guard, and the submitted proof is unit/mock proof rather than a real cron CLI/API run.

## What This Changes

The PR adds an `isManual` option to cron result handling, passes it from manual cron run paths, and adds regression tests for preserving `deleteAfterRun` one-shot jobs during manual runs.

## Best Possible Solution

Keep the `isManual` seam, move or suppress manual one-shot alert/counter/cooldown bookkeeping before failure-alert side effects, and merge only after redacted `openclaw cron run`/`cron get`/`cron runs` proof shows the scheduled job is preserved.

Yes, by source inspection and the linked issue context: current main routes manual cron completions into `applyJobResult` without a manual flag, and that function deletes successful `deleteAfterRun` at-jobs. I did not run a live cron service reproduction in this read-only review.

## Solution Assessment

No, not yet. The `isManual` discriminator is the right seam, but the fix should guard manual one-shot alert/counter/cooldown mutations before failure-alert bookkeeping and needs real CLI/API proof.
