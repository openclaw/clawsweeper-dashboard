## Summary

Keep open: this PR addresses a real cron data-loss bug, but the latest diff still suppresses existing manual-run failure alerts and lets manual success reset alert cooldown state; the supplied proof is still unit/source-level rather than a live patched CLI/Gateway run.

## What This Changes

The PR adds an `isManual` path to cron result application so manual runs preserve `deleteAfterRun` at-jobs and scheduled retry counters, with regression coverage for the new state behavior.

## Best Possible Solution

Land a bounded manual-result boundary that preserves one-shot schedules and scheduled counters while keeping existing failure-alert delivery/cooldown behavior unless maintainers explicitly approve and test a changed alert contract.

## Reproduction Assessment

Yes. Source inspection on current `main` shows manual cron completion reaches `applyJobResult` without a manual flag, and that function deletes successful `deleteAfterRun` at-jobs; I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The `isManual` boundary is the right shape, but the implementation must preserve or explicitly redesign failure-alert delivery and cooldown state before it is the safest fix.
