## Summary

Keep open: current main and v2026.6.8 still have the manual cron deletion/state-mutation path, while this PR is a proof-backed candidate fix with no concrete line-level review findings. The remaining blocker is maintainer acceptance of the manual-run policy and additive persisted cron state field before merge.

## What This Changes

Adds an `isManual` cron-result path that records `lastRunWasManual`, prevents manual cron runs from deleting one-shot jobs or mutating scheduled retry/alert state, updates the gateway protocol schema, and expands cron regression coverage.

## Best Possible Solution

Land this patch or an equivalent shared-boundary fix after maintainers accept manual-run isolation and the additive cron state field, with required checks green.

## Reproduction Assessment

Yes, source-level. Current main's manual completion path calls `applyJobResult` without a manual discriminator, and that shared handler deletes successful `deleteAfterRun` at-jobs while mutating scheduled counters and alert state.

## Solution Assessment

Yes, conditional on maintainer policy acceptance. The shared `applyJobResult` discriminator is the narrowest maintainable fix shape I found; rejecting manual runs or adding a delete override would be a larger CLI/API policy change.
