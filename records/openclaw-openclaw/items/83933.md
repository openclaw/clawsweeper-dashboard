## Label Justifications

- `merge-risk: 🚨 compatibility`: Manual-run semantics and the persisted gateway cron state contract change for existing jobs and clients.

## Summary

This PR remains necessary: current main still applies scheduled deletion, retry, alert, and timing state to manual cron completions, while the branch cleanly distinguishes manual origin at the shared result boundary. The optional persisted marker is schema-aligned and covered across restart/backoff paths; the contributor also supplied real Docker Gateway CLI evidence showing one-shot and recurring jobs retain their scheduled state after manual runs.

## What This Changes

The PR marks cron results as manual or scheduled, prevents manual runs from consuming `deleteAfterRun` jobs or mutating scheduled retry/alert state, and persists manual origin for restart and backoff maintenance.

## Best Possible Solution

Land the shared execution-origin model with the optional persisted marker and its focused restart/backoff tests, while keeping scheduled-run behavior as the control path and avoiding a separate CLI-only workaround or new config switch.

## Reproduction Assessment

Yes. Current main’s call path has no manual discriminator before one-shot deletion and scheduled-state mutation, and the contributor demonstrated the before/after behavior through a real Docker Gateway CLI sequence.

## Solution Assessment

Yes. Adding execution origin at the canonical result-state boundary is narrower and more maintainable than CLI rejection, temporary flag edits, or duplicating outcome logic; the persisted marker is justified because restart and backoff maintenance read the latest terminal outcome.
