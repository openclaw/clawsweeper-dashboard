## Label Justifications

- `P1`: The linked bug can silently consume scheduled one-shot cron work and lose future fires for real users.

## Summary

Keep this PR open: the linked cron data-loss bug still exists on current main and the latest release, and this patch is a plausible owner-boundary fix with no blocking code findings from this read-only review. The remaining decision is maintainer acceptance of the compatibility-sensitive manual-run state policy.

## What This Changes

The PR makes manual `openclaw cron run` completions pass an `isManual` discriminator so they record the run outcome without consuming `deleteAfterRun` jobs or mutating scheduled retry, backoff, alert, and counter state.

## Best Possible Solution

Land this PR after maintainers accept the manual-run isolation policy and current-head checks pass; otherwise keep the linked issue as the canonical place for the alternate policy decision.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 route manual completion through `applyJobResult` without a manual discriminator, so an ok `deleteAfterRun` at-job can be removed by a manual run.

## Solution Assessment

Yes, with a policy caveat. Passing an `isManual` discriminator into the shared result-application path is the narrowest maintainable fix compared with a CLI-only workaround, but maintainers still need to accept the manual-run state contract.
