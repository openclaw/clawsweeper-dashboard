## Summary

Keep open: current `main` and the latest release still share manual and scheduled cron result handling, while this PR is the active implementation for the linked destructive manual-run bug. I found no discrete correctness defect in the patch, but merge still needs maintainer acceptance of the changed manual-run semantics and new persisted/public cron state field.

## What This Changes

The PR adds a manual-run discriminator to cron result handling, preserves scheduled one-shot/retry state during manual `cron run`, exposes `lastRunWasManual`, and expands cron regression coverage.

## Best Possible Solution

Land a maintainer-approved cron contract where manual runs are non-destructive verification/debug runs, scheduled runs keep existing delete/retry behavior, and the new persisted/public state field is accepted with exact-head proof and required checks green.

## Reproduction Assessment

Yes at source level: current main routes manual and scheduled outcomes through the same `applyJobResult` logic, so a manual ok result on a `deleteAfterRun` at-job satisfies the same deletion predicate as a scheduled result. I did not run a live current-main repro in this read-only review.

## Solution Assessment

Yes mechanically: `applyJobResult` is the common outcome boundary, and passing `isManual` from the manual-run callers is the narrow place to separate manual from scheduled state effects. The remaining question is product/compatibility acceptance of the new manual-run contract and public `lastRunWasManual` state.
