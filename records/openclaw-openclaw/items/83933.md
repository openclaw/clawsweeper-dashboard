## Summary

Keep open: this PR is still not merge-ready because the manual-run fix preserves a successful due one-shot inside `applyJobResult`, but the same completion path then runs maintenance recompute that can clear `nextRunAtMs` for an already-due successful at-job. The supplied proof is strong for adjacent manual error/skipped paths, but it still does not prove the successful manual due one-shot path that can lose the scheduled fire.

## What This Changes

The PR adds manual-run isolation to cron result handling, persists a `lastRunWasManual` cron state field in the gateway protocol, and expands cron regression tests around `deleteAfterRun`, counters, alerts, and restart catch-up.

## Best Possible Solution

Keep the shared `applyJobResult` manual discriminator, but also make post-manual maintenance preserve successful due one-shot slots and prove it with a full `cron.run(job.id, "due")` success regression plus real CLI output.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a successful manual `cron.run(job.id, "due")` on an already-due at-job sets `lastRunStatus: "ok"`, then maintenance recompute can clear `nextRunAtMs`.

## Solution Assessment

No. The shared `applyJobResult` discriminator is the right layer, but the implementation is incomplete until post-run maintenance also respects the manual due success contract.
