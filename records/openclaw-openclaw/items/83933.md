## Label Justifications

## Summary

Keep open: current main and the latest release still have the manual cron completion path sharing scheduled `deleteAfterRun`, counter, backoff, and failure-alert state. The PR is a useful, proof-backed fix with no concrete patch defect found, but it still needs maintainer acceptance for the new cron state/protocol marker and manual-run alert semantics.

## What This Changes

The branch marks manual cron result application with `isManual`, prevents manual runs from consuming `deleteAfterRun` or scheduled retry/backoff/failure-alert state, adds a `lastRunWasManual` cron state field, and expands cron regression coverage.

## Best Possible Solution

Land a cron-owner-reviewed version that keeps manual `cron run` isolated from scheduled delete/backoff/counter state while deliberately accepting `lastRunWasManual` as the cron state/protocol marker.

## Reproduction Assessment

Yes from source inspection: current main routes manual run completion through `applyJobResult` without a manual discriminator, and that function deletes successful `deleteAfterRun` at-jobs plus mutates retry counters. I did not execute tests because this review was read-only, but the PR discussion includes Docker/Gateway before-and-after proof.

## Solution Assessment

Yes, conditionally: adding a manual discriminator at `applyJobResult` and preserving scheduled state is the narrowest owner-boundary fix I found. Maintainers still need to accept the public `lastRunWasManual` marker and the choice not to emit scheduled failure alerts for manual probes.
