## Summary

Keep open for maintainer review before merge. The patch appears necessary and technically coherent because current main still routes manual cron completions through scheduled-state mutation, but the semantic change and new persisted/public state bit need explicit maintainer acceptance.

## What This Changes

The PR branch adds an `isManual` cron result path, persists/exposes `lastRunWasManual`, and updates cron tests so manual `cron run` no longer consumes one-shot jobs or mutates scheduled retry state.

## Best Possible Solution

Land the PR only after maintainers accept non-destructive manual-run semantics and the additive lastRunWasManual state/protocol contract, or revise the state exposure before merge.

## Reproduction Assessment

Yes at source level: current main routes manual run completions through applyJobResult without a manual discriminator, so deleteAfterRun and scheduled retry-state mutations are reachable. The linked issue and PR proof also provide real CLI evidence of the destructive behavior and the proposed after-fix behavior.

## Solution Assessment

Yes for the code shape, conditional on maintainer acceptance: applyJobResult is the owner boundary where delete, retry, backoff, counters, and alerts are decided. Rejecting manual runs or adding a CLI override would be a larger UX change; the remaining question is the public/persisted state contract.
