## Summary

Keep this PR open: it fixes a real cron manual-run bug, but the current patch still lets a forced manual run that completes after an at-job's scheduled timestamp make the post-run maintenance recompute treat that scheduled slot as already executed.

## What This Changes

The PR changes cron manual-run completion handling so manual runs preserve delete-after-run jobs and scheduled counters/backoff/failure-alert state, and adds `lastRunWasManual` to cron state schemas.

## Best Possible Solution

Land the manual-run isolation only after the maintenance recompute path also treats manual terminal results as non-executing for overdue scheduled at-job slots.

- **[P1] Preserve overdue at-jobs after manual runs:** `src/cron/service/timer.ts:411`
  - body: When a manual `cron run` finishes after an at-job's scheduled `nextRunAtMs`, `finishPreparedManualRun` still calls `recomputeNextRunsForMaintenance(..., { recomputeExpired: true })`. That recompute path treats `lastRunAtMs >= nextRunAtMs` as an already executed slot and recomputes/clears the one-shot schedule; because this patch writes the manual completion time to `lastRunAtMs` but only checks `lastRunWasManual` in `isRunnableJob`, an overdue scheduled fire can still be consumed by the manual run instead of remaining runnable. Please make the maintenance already-executed check ignore manual terminal runs too.
