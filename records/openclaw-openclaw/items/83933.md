## Summary

Keep open for maintainer review: the runtime patch is source-backed and has strong Docker CLI proof, but it intentionally changes manual cron run state and alert semantics, so the compatibility decision should be accepted before merge.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual run paths, and adds regression coverage for preserving `deleteAfterRun`, next-run, backoff, counters, and alert cooldown state during manual runs.

## Best Possible Solution

Land this or a narrow equivalent fix once cron owners accept manual `cron run` as a verification/debug path that must not consume scheduled delivery state, and leave the linked issue open until the fix merges.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no manual discriminator in `applyJobResult`, and manual run paths pass only `preserveSchedule`. The contributor also provided Docker Gateway CLI proof showing the changed behavior at the PR head.

## Solution Assessment

Yes, with maintainer acceptance: adding the discriminator at `applyJobResult` and setting it only from manual run call sites is the narrow runtime fix. The remaining decision is whether the broader counter/alert isolation is the intended compatibility contract.
