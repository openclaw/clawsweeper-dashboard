## Summary

Keep open: the patch targets a real destructive cron manual-run bug and the current diff looks functionally sound with strong Docker/CLI/test proof, but it intentionally changes public manual-run state semantics and should get explicit maintainer acceptance before merge.

## What This Changes

The PR threads an `isManual` flag through cron manual-run completion so `applyJobResult` records manual outcomes without deleting `deleteAfterRun` jobs or mutating scheduled retry, skip, backoff, and failure-alert state.

## Best Possible Solution

Merge only after maintainers explicitly accept the non-consuming manual cron run contract and keep the current regression coverage and proof attached to that decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main applies manual run results through `applyJobResult` without a manual discriminator, so deleteAfterRun and scheduler counters are mutated. The contributor also posted before/after Docker and CLI proof for the same tree as the current PR head.

## Solution Assessment

Yes, if maintainers accept the contract: centralizing the discriminator in `applyJobResult` is the narrowest owner-path fix for both manual success and failure completion. The safer alternative, if old behavior must remain, is an explicit opt-in consuming mode rather than unconditional manual consumption.
