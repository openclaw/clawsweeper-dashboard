## Summary

Keep open for maintainer review and possible landing: current main still has the destructive manual-run cron path, and this PR addresses it centrally with source-level coverage and Docker Gateway CLI proof. I found no blocking code-review finding in the read-only pass, but the change is compatibility-sensitive because it adds public/persisted cron state and changes manual-run alert/backoff behavior.

## What This Changes

The PR tags manual cron completions, preserves `deleteAfterRun` jobs and scheduled retry/alert counters for manual runs, and adds regression/protocol coverage around `lastRunWasManual` state.

## Best Possible Solution

Land the fix after maintainer acceptance of the additive cron state marker and green current-head cron/protocol checks, keeping scheduled runs as the only path that consumes `deleteAfterRun`, backoff, counters, and failure alerts.

## Reproduction Assessment

Yes. Current main’s manual completion path calls `applyJobResult` without a manual discriminator, and that function deletes successful `deleteAfterRun` at-jobs while mutating retry/skip state; I verified this by source inspection rather than by running the CLI.

## Solution Assessment

Yes. The PR fixes the owner boundary by tagging manual completions at `ops.ts`, centralizing scheduled-state isolation in `applyJobResult`, and teaching shared backoff/catch-up helpers about manual terminal state; rejecting `cron run` would be more disruptive and less useful for verification.
