## Summary

Keep open: the PR is directionally fixing a real cron data-loss bug, but the latest diff still lets manual one-shot error runs trigger failure-alert side effects before rolling back the error counter, and the PR still lacks real behavior proof.

## What This Changes

The PR adds an `isManual` flag to cron result application, passes it from manual cron run paths, and adds regression tests so manual runs preserve `deleteAfterRun` one-shot jobs.

## Best Possible Solution

Keep the manual-run discriminator, move or skip manual at-job error counter and failure-alert side effects before they fire, then merge after redacted queued-run proof confirms job preservation and later scheduled eligibility.

## Reproduction Assessment

Yes by source inspection, but not by a live run in this read-only review. Current main routes manual cron completion through `applyJobResult` without a manual discriminator, so a `deleteAfterRun:true` one-shot manual success can delete the job and ok/skipped manual paths can disable the future fire.

## Solution Assessment

Not yet. The manual-run discriminator is the right shape for the core bug, but the manual error branch must avoid failure-alert side effects before rollback and the PR needs real queued-run proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move the manual error guard before failure alerts:** `src/cron/service/timer.ts:933`
  - body: For manual at-job errors, this rollback happens after the generic error block has already incremented `consecutiveErrors` and called `maybeEmitFailureAlert`. If the job is at its alert threshold, a manual verification failure can still send a failure notification and set `lastFailureAlertAtMs`, so the scheduled fire inherits polluted alert/cooldown state even though this branch says it is ignoring manual error state. Skip or defer the error-counter and alert side effects for manual at-job runs before calling `maybeEmitFailureAlert`.
