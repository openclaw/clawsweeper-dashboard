## Summary

Keep open: this PR is still useful and appears technically sound, but it changes compatibility-sensitive cron manual-run semantics and persisted protocol/state surface, so the remaining action is maintainer acceptance rather than cleanup close.

## What This Changes

The PR adds manual-run isolation to cron result handling so manual runs record outcomes without consuming `deleteAfterRun` one-shots or mutating scheduled retry, backoff, and failure-alert state, plus an optional state/protocol flag and regression coverage.

## Best Possible Solution

Have maintainers explicitly adopt the non-consuming manual-run contract, land this focused timer/ops/schema fix with docs or release-note context, and keep failed-run and scheduled-repeat `deleteAfterRun` policy in their canonical items.

## Reproduction Assessment

Yes. Current main and v2026.6.10 route manual completion through `applyJobResult` without a manual discriminator, so source inspection shows a successful manual at/deleteAfterRun result can delete the job and manual errors/skips mutate scheduled counters.

## Solution Assessment

Yes, technically: `applyJobResult` is the narrow owner boundary shared by manual and scheduled completions, and the PR wires both manual callers there. The best-fix caveat is product acceptance of the new non-consuming manual-run contract and its docs/protocol surface.
