## Summary

Keep open for maintainer review: the patch appears to fix the reported cron manual-run data-loss path, but it intentionally changes manual-run side effects and adds gateway-visible persisted cron state, so this should not be auto-closed or auto-merged without owner acceptance.

## What This Changes

Threads an `isManual` completion flag through cron manual-run paths, preserves scheduled delete/retry/backoff/alert state for manual executions, adds `lastRunWasManual` to cron state/protocol schemas, and expands cron regression coverage.

## Best Possible Solution

Land the shared manual-run isolation contract after maintainer acceptance, keeping scheduled cron completions as the only path that consumes `deleteAfterRun`, retry/backoff, and failure-alert state.

## Reproduction Assessment

Yes. Source inspection of current main shows manual completion reaches `applyJobResult` without a manual discriminator, and a successful `deleteAfterRun` at-job returns `shouldDelete` while counters and alert state mutate unconditionally.

## Solution Assessment

Yes, with maintainer acceptance. The shared `applyJobResult` discriminator is the narrowest owner-boundary fix because both manual and scheduled completion already converge there; `lastRunWasManual` is justified by restart/backoff gates but needs acceptance as a new state contract.
