## Summary

This PR is a plausible fix for the destructive manual-run cron bug, but it should stay open because the manual guard still runs after failure-alert bookkeeping and the supplied proof does not exercise a real patched cron CLI/API flow.

## What This Changes

This PR adds an `isManual` option to cron result handling, passes it from manual cron run paths, and adds regression tests for preserving `deleteAfterRun` one-shot jobs during manual runs.

## Best Possible Solution

Keep the `isManual` seam, suppress manual one-shot alert/counter/cooldown side effects before they mutate state, and merge only after redacted real CLI/API proof shows the scheduled job is preserved.

## Reproduction Assessment

Yes, by source inspection and the linked report context: current main routes manual cron completions into `applyJobResult` without a manual flag, and successful `deleteAfterRun` at-jobs return `shouldDelete=true`. I did not run a live current-main cron service reproduction in this read-only review.

## Solution Assessment

No, not yet: the `isManual` seam is the right shape, but the patch needs to place manual one-shot preservation before failure-alert/cooldown bookkeeping or explicitly skip those side effects. The tests should cover that state, and the contributor still needs real patched CLI/API proof.
