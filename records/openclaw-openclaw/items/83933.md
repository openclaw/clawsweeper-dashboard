## Summary

This PR should stay open: the main manual-run fix is well targeted and well proven, but the latest patch makes `cron.run(..., "due")` manual for backoff/counter state while a current-main regression test from merged backoff work still requires that path to persist configured recurring-error backoff.

## What This Changes

The PR adds manual-run awareness to cron result application, records `lastRunWasManual`, skips delete/retry/counter/alert state changes for manual runs, updates the cron gateway schema, and expands cron regression coverage.

## Best Possible Solution

Land a manual-run isolation fix at the shared cron result boundary after reconciling due-mode recurring backoff coverage and maintainer acceptance of the compatibility-visible semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows manual cron completion reaches `applyJobResult` without a manual discriminator, and that shared function deletes successful `deleteAfterRun` at-jobs and mutates retry counters.

## Solution Assessment

No, not yet. The shared `applyJobResult` discriminator is the right boundary, but the patch must reconcile `cron.run(..., "due")` manual isolation with the current-main recurring backoff readback contract before it is the best fix.
