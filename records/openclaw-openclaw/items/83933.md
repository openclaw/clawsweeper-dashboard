## Summary

Keep open for maintainer review and merge work: this is a focused candidate fix for #83538, and source/proof show it addresses current-main manual cron state loss without blocking code or security findings. The remaining concerns are compatibility-sensitive manual-run semantics, an additive cron state/protocol field, a small docs mismatch, and currently red unrelated merge gates.

## What This Changes

The PR adds manual-run awareness to cron outcome application, preserves scheduled delete/retry/counter/backoff/alert state for manual runs, adds `lastRunWasManual` to cron state/protocol, and updates cron regression coverage.

## Best Possible Solution

Land this after maintainer acceptance of the non-consuming manual-run contract, a short docs clarification, and green merge gates; keep failed-run cleanup policy in #88197 separate.

## Reproduction Assessment

Yes, source-reproducible: current main’s manual completion path calls `applyJobResult` without a manual discriminator, and `applyJobResult` deletes ok `at`/`deleteAfterRun` jobs while mutating scheduled counters and alerts; I did not run a live gateway repro in this read-only pass.

## Solution Assessment

Yes, with maintainer policy acceptance: placing the discriminator at shared `applyJobResult` and passing it from both manual completion paths is the clean owner-boundary fix; rejecting `cron run` or special-casing only deletion would leave counters, backoff, alerts, and restart catch-up inconsistent.
