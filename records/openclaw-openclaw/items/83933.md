## Summary

Keep this PR open: it is a focused, proof-backed fix for the open manual cron-run data-loss issue, but it still needs maintainer acceptance because it changes compatibility-sensitive cron state semantics and adds a protocol-visible persisted state bit.

## What This Changes

The PR marks manual cron runs as manual, prevents them from consuming deleteAfterRun jobs or mutating scheduled retry/backoff/alert state, and exposes lastRunWasManual in cron state schemas.

## Best Possible Solution

Land the centralized manual-run isolation only after maintainers accept the manual-versus-scheduled cron contract and the latest head finishes required checks.

## Reproduction Assessment

Yes. Current main calls applyJobResult from manual completion without a manual discriminator, and that shared function returns deletion for successful deleteAfterRun at-jobs.

## Solution Assessment

Yes, pending maintainer acceptance. Centralizing the distinction in applyJobResult and passing isManual from manual-run callers is the narrowest maintainable fix; rejecting manual runs or adding an override flag would leave more user-facing policy surface.
