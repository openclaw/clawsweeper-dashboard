## Summary

Keep open: current main and v2026.6.10 still have the manual-run deletion/counter bug, while this PR is the unique open candidate fix with sufficient real behavior proof and no confirmed blocking correctness or security finding. Merge still needs maintainer acceptance because it changes compatibility-sensitive manual cron-run semantics and adds a protocol-visible persisted state bit.

## What This Changes

The PR marks manual cron runs as manual, prevents them from consuming `deleteAfterRun` jobs or mutating scheduled retry/backoff/alert state, and adds `lastRunWasManual` to cron state contracts and tests.

## Best Possible Solution

Keep the centralized manual-run isolation, but land it only after maintainers accept the manual-versus-scheduled cron state contract and clarify the docs/API wording they want to ship.

## Reproduction Assessment

Yes. Current main and v2026.6.10 call `applyJobResult` from manual completion without a manual discriminator, and that shared function returns deletion for successful `deleteAfterRun` at-jobs.

## Solution Assessment

Yes, pending maintainer acceptance. Centralizing the distinction in `applyJobResult` and passing `isManual` from manual-run callers is the narrowest maintainable fix; rejecting manual runs or adding an override flag would create more user-facing policy surface.
