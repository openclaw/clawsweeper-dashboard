## Summary

Keep open. The PR is aimed at a real cron data-loss bug, but current head still has a blocking state bug: successful manual `due` runs can preserve the job through `applyJobResult` and then lose the scheduled slot during the later maintenance recompute.

## What This Changes

The PR adds manual-run state isolation for cron result application, records and exposes `lastRunWasManual`, and expands cron regression/protocol coverage.

## Best Possible Solution

Keep the manual-run outcome recording, but make maintenance recompute ignore manual terminal at-runs or skip the just-finished manual job, then prove successful, skipped, and errored manual due runs preserve scheduled state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-head path: a successful manual due at-job can set `lastRunAtMs >= nextRunAtMs`, then maintenance recompute treats the slot as executed and clears the scheduled fire.

## Solution Assessment

No. The `isManual` discriminator is the right fix direction, but it is not yet the best complete fix because post-run maintenance recompute still ignores the manual-run marker for successful due one-shots.
