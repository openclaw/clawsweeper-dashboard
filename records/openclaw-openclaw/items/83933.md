## Summary

This PR is a focused fix for a real cron data-loss bug, but it is not merge-ready: the latest head still lets manual error runs emit failure alerts and set cooldown before the new manual rollback runs, and the external PR still lacks real behavior proof.

## What This Changes

Adds an `isManual` result-application path so manual `cron run` calls preserve `deleteAfterRun` one-shot jobs, with cron regression tests for manual and scheduled outcomes.

## Best Possible Solution

Keep the `isManual` approach, move manual error suppression before any failure-alert or cooldown side effects, then merge only after redacted queued-run proof confirms the one-shot job remains scheduled.

## Reproduction Assessment

Yes by source inspection, not by a live run. Current main routes manual cron completion through `applyJobResult` without a manual discriminator, and that function deletes successful `deleteAfterRun` one-shot jobs or disables one-shot ok/skipped jobs.

## Solution Assessment

Not yet. The manual-run discriminator is the right shape for the core bug, but the PR still needs to suppress manual error alert side effects before rollback and provide real queued-run proof.
