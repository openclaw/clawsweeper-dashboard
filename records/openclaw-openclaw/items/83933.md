## Summary

Keep this PR open: the fix direction is right, but the current patch still lets manual errored `at` jobs enter the one-shot retry/disable path, and the supplied proof does not show the patched cron CLI or Gateway flow end to end.

## What This Changes

The PR adds an `isManual` option to cron result application, passes it from manual cron run paths, and adds timer regression tests for preserving `deleteAfterRun` jobs and retry counters.

## Best Possible Solution

Preserve manual one-shot jobs across ok, skipped, and error outcomes, leave scheduled retry counters and pending schedule state untouched for manual runs, keep scheduled `deleteAfterRun` behavior unchanged, and verify with redacted patched CLI/Gateway output.

## Reproduction Assessment

Yes. Source inspection shows current main manual run completion calls `applyJobResult` without a manual discriminator, and successful `deleteAfterRun` `at` jobs return `shouldDelete=true`; I did not run a live repro in this read-only review.

## Solution Assessment

No. The `isManual` direction is maintainable, but the patch must also bypass one-shot error retry/disable state for manual runs and needs real patched cron CLI/Gateway proof before merge.
