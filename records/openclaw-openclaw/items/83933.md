## Summary

The core fix is credible and the new execution-origin field round-trips through the generic SQLite `state_json` path, but this PR intentionally changes configured alert semantics for every manual failure and skip; that broader operator-facing policy needs explicit cron ownership approval before merge.

## What This Changes

The branch marks cron completions as manual or scheduled so manual runs preserve one-shot jobs, counters, retry/backoff state, alert cooldowns, and restart catch-up behavior, with gateway-schema and regression-test updates.

## Best Possible Solution

Keep the execution-origin discriminator and its restart/backoff coverage, but merge only after cron and gateway-protocol owners explicitly approve the durable state field and decide whether configured alerts should exclude all manual failures and skips or remain independently configurable.

## Solution Assessment

Unclear. Distinguishing manual from scheduled results at `applyJobResult` is the correct owner boundary and the persistence path is coherent, but excluding every manual failure and skip from configured alerts is a product-policy expansion that needs explicit approval.
