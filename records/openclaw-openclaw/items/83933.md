## Summary

Keep open: this PR still owns the canonical manual-run cron fix, and current main plus v2026.6.8 still route manual completions through the scheduled-run `deleteAfterRun` and counter paths. The patch appears correct, but it intentionally changes compatibility-sensitive cron state semantics, so it needs maintainer acceptance rather than conservative cleanup closure.

## What This Changes

The PR adds manual-run awareness to cron result application, preserves one-shot jobs and scheduled retry/alert counters for manual runs, records `lastRunWasManual`, updates the gateway cron state schema, and expands cron regression coverage.

## Best Possible Solution

Land this branch after maintainers explicitly accept manual cron runs as non-destructive verification and align user-facing wording for the scheduled-only side effects.

## Reproduction Assessment

Yes. Source inspection of current main shows manual completion calls applyJobResult without a manual discriminator, and an ok at-job with deleteAfterRun returns shouldDelete; the PR body also supplies before/after Docker and regression-test proof.

## Solution Assessment

Yes, if maintainers accept the policy. The shared applyJobResult discriminator is the narrowest owner-boundary fix because both scheduled and manual completions already converge there; CLI-only rejection or post-hoc restoration would leave other manual callers and state counters divergent.
