## Summary

Keep open: current main still has the source-reproducible manual-run/deleteAfterRun bug and this PR is a plausible owner-boundary fix, but it changes cron run semantics and protocol-visible state while lacking current-head Gateway CLI proof after later rebases.

## What This Changes

The branch adds manual-run-aware cron result handling, persists `lastRunWasManual`, updates the cron protocol state schema, and expands cron tests so manual runs preserve scheduled delete/backoff/counter state.

## Best Possible Solution

Land a maintainer-approved manual/scheduled cron result split only after current-head Gateway CLI proof and explicit acceptance of the additive state field and changed manual-run semantics.

## Reproduction Assessment

Yes at source level: current main's `applyJobResult` deletes successful `deleteAfterRun` at-jobs without knowing whether the run was manual, and manual callers do not provide such a discriminator. I did not run a live repro in this read-only review.

## Solution Assessment

Mostly yes: threading a manual/scheduled discriminator into the shared result applier is the right owner boundary, and alternatives like rejecting manual runs or adding a new flag are broader. The remaining concern is maintainer acceptance of the compatibility/API semantics plus current-head proof.
