## Label Justifications

- `P1`: The PR fixes a high-impact cron workflow where manual verification can consume or corrupt scheduled one-shot state for real users.
- `merge-risk: 🚨 compatibility`: The patch changes manual-run semantics and adds a gateway-visible cron state field, which can affect upgrades and tooling that read cron job state.
- `merge-risk: 🚨 session-state`: The patch deliberately changes persisted cron scheduling state, counters, retry/backoff behavior, and failure-alert state for manual runs.

## Summary

Keep this PR open for normal maintainer review: current main and v2026.6.8 still route manual cron results through scheduled delete/counter state, while the PR applies the fix at the shared cron result boundary with regression coverage and passing CI. I found no blocking code-review finding, but the change intentionally alters persisted cron state semantics and needs maintainer acceptance of the compatibility/session-state risk before merge.

## What This Changes

The PR adds an `isManual` cron result discriminator, records `lastRunWasManual`, skips manual-run deletion/counter/backoff/alert mutations, updates the gateway protocol state schema, and adds cron regression coverage.

## Best Possible Solution

Land this patch or an equivalent shared-boundary fix after maintainers accept that manual cron runs are non-destructive verification runs and that the additive cron state field is acceptable for upgrades.

## Reproduction Assessment

Yes, source-level: current main's manual completion path calls `applyJobResult` without a manual discriminator, and that shared handler deletes successful `deleteAfterRun` at-jobs while mutating scheduled counters. I did not run local tests because this review was read-only, but the PR body and CI include after-fix Docker/CLI proof.

## Solution Assessment

Yes, this is the best fix shape I found: the behavior converges in `applyJobResult`, and passing `isManual` from the manual callers preserves scheduled state without adding a second CLI policy path. Rejecting manual runs or adding a delete override would be a larger product/API change rather than repairing the shared state transition.
