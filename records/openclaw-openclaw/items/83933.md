## Summary

Keep open: the PR targets a real current-main cron bug and I did not find a blocking code defect, but it still needs maintainer acceptance of the manual-run state semantics and current-head real CLI/Gateway proof before merge.

## What This Changes

The PR adds a manual-run discriminator to cron result handling so manual cron runs do not consume `deleteAfterRun` jobs or mutate scheduled retry, skip, backoff, and failure-alert state.

## Best Possible Solution

Land a scoped manual-run isolation fix after current-head live CLI/Gateway proof shows the job and counters survive manual execution, with maintainer acceptance that manual runs are verification/debugging paths rather than scheduled-state transitions.

## Reproduction Assessment

Yes for source-level reproduction: current main routes manual completion through `applyJobResult` without a manual discriminator, and that function unconditionally mutates counters and deletes successful `deleteAfterRun` at-jobs. I did not run a live current-main Gateway repro in this read-only review.

## Solution Assessment

Mostly yes: adding a manual discriminator at the shared cron result state machine and passing it only from manual callers is the narrow repair path. The broadened manual-run state contract still needs maintainer acceptance and current-head live proof before merge.
