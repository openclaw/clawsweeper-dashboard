## Summary

Keep open: this PR targets a real P1 cron bug, but the current head still appears to consume a successful manual due one-shot through the post-run maintenance recompute path, and the latest proof does not demonstrate that full path.

## What This Changes

The PR adds manual-run state to cron result application, protocol state, and regression tests so manual cron runs do not delete `deleteAfterRun` jobs or mutate scheduled retry counters.

## Best Possible Solution

Keep the shared manual-run discriminator, but make maintenance recompute treat manual terminal outcomes as non-consuming scheduled slots and prove the full latest-head CLI/Gateway success path.

## Reproduction Assessment

Yes, source-level. A manual due success can set `lastRunWasManual` while leaving an already-due `nextRunAtMs`, then the post-run `recomputeExpired` path sees `lastRunAtMs >= nextRunAtMs` and can clear that one-shot slot.

## Solution Assessment

No, not yet. The shared `applyJobResult` discriminator is the right layer, but the fix is incomplete until maintenance recompute and proof cover successful manual due completion.
