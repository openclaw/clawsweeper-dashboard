## Label Justifications

- `P1`: The PR addresses silent loss of scheduled cron jobs and future scheduled message delivery from manual cron runs.
- `merge-risk: 🚨 compatibility`: Existing operator workflows that expected manual cron runs to consume one-shot jobs or retry state will behave differently after merge.
- `merge-risk: 🚨 session-state`: The diff changes persisted cron job counters, cooldowns, enabled state, and nextRunAtMs behavior for manual runs.
- `merge-risk: 🚨 message-delivery`: Preserving deleteAfterRun jobs after manual execution can cause the future scheduled delivery to still fire where current behavior would remove it.

## Summary

Keep this PR open for maintainer review: the patch appears to fix the cron data-loss/message-loss bug without a discrete blocking code finding, but it intentionally changes compatibility-sensitive manual-run state semantics.

## What This Changes

The PR adds a manual-run discriminator to cron result handling so manual cron runs no longer consume delete-after-run jobs or mutate scheduled retry, skip, backoff, and failure-alert state.

## Best Possible Solution

Land this only if maintainers accept manual cron runs as non-consuming diagnostic executions, with scheduled timer execution remaining the only path that consumes deleteAfterRun and scheduled retry/alert state.

## Reproduction Assessment

Yes. Current main source shows applyJobResult has no manual-run discriminator, and the PR discussion includes Docker CLI proof that manual cron runs can alter the scheduled job state.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility change. Passing a narrow isManual flag from the two manual completion paths into the shared result applicator is the smallest maintainable fix while leaving scheduled paths unchanged.
