## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` params, and #74974 is open and unmerged while maintainers still need to choose the public protocol shape.

## What This Changes

Allow Paperclip's `openclaw_gateway` agent payload metadata to pass Gateway `AgentParams` validation instead of being rejected as an unexpected root property.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, keep strict rejection for unrelated root typos, and update focused tests, generated protocol artifacts, docs if needed, and the changelog.

## Reproduction Assessment

Yes. A minimal `agent` request with `message`, `sessionKey`, `idempotencyKey`, and a root-level `paperclip` object follows the reported payload and fails current `validateAgentParams` before dispatch.

## Solution Assessment

Unclear. Accepting `paperclip` is the narrowest compatibility shim and matches #74974, but a generic adapter metadata namespace may be the more maintainable protocol surface if maintainers want to avoid vendor-named root fields.
