## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` params, while https://github.com/openclaw/openclaw/pull/74974 is open and explicitly targets this issue with the broader `adapterMeta` plus compatibility-alias design.

## What This Changes

The issue asks OpenClaw Gateway to stop rejecting Paperclip's root-level `paperclip` metadata on `agent` requests.

## Best Possible Solution

Review and land one maintainer-approved additive Gateway `AgentParams` metadata contract with schema, validator tests, generated client artifacts, and changelog aligned, then close the linked Paperclip reports.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send an `agent` RPC with valid `message` and `idempotencyKey` plus a root `paperclip` property, and current `validateAgentParams` rejects it before dispatch.

## Solution Assessment

Unclear. The open `adapterMeta` plus temporary `paperclip` alias design is a strong candidate, but the generic namespace and alias lifecycle need maintainer protocol/API judgment before this issue should close.
