## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` params, and the open implementation PR that targets this issue has not landed.

## What This Changes

The issue asks OpenClaw Gateway to accept Paperclip adapter metadata on agent requests or provide a supported metadata path.

## Best Possible Solution

Review, update, and land a maintainer-approved additive Gateway `AgentParams` metadata contract with schema, validator coverage, generated client models, and changelog aligned, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus a root `paperclip` property, and current `validateAgentParams` rejects it before dispatch.

## Solution Assessment

Unclear. The open `adapterMeta` plus `paperclip` compatibility-alias design is a strong technical candidate, but the public namespace and alias lifecycle need maintainer protocol/API judgment before this issue should close.
