## Label Justifications

- `P1`: Multiple users report a broken Paperclip-to-Gateway agent workflow that prevents tasks from reaching OpenClaw agents.
- `impact:message-loss`: Gateway validation rejects Paperclip agent invocation messages before dispatch, so downstream tasks never reach the agent workflow.

## Summary

Keep open. Current main and the latest stable release still reject root-level `paperclip` on Gateway `agent` params, and the open implementation candidate at https://github.com/openclaw/openclaw/pull/74974 explicitly references this issue, so it should stay open until that PR or a superseding protocol fix lands.

## What This Changes

The issue requests an OpenClaw Gateway and Paperclip compatibility fix so `AgentParams` validation no longer rejects Paperclip's root-level metadata.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74974 or a superseding maintainer-approved additive `AgentParams` metadata contract with schema, generated client model, focused validator coverage, and changelog, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed `AgentParamsSchema` validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's generic `adapterMeta` namespace plus a temporary `paperclip` alias is more maintainable than a permanent vendor-only root field, but it is still an API/product decision.
