## Label Justifications

- `P1`: Multiple users report a production Paperclip-to-Gateway agent workflow is completely blocked by schema rejection.
- `impact:message-loss`: The issue is about Gateway agent request messages being rejected before they can dispatch to the runtime.

## Summary

Keep open. Current `main` and latest stable release still reject root-level `paperclip` on Gateway `agent` params, while the live Paperclip adapter still sends that field and the open implementation PR that closes this issue has not landed.

## What This Changes

Paperclip Gateway adapter requests need OpenClaw to tolerate or otherwise route root-level `paperclip` metadata in `agent` params so Paperclip-triggered agent runs do not fail schema validation.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74974 or a superseding additive `AgentParams` metadata contract with schema, generated client model, focused validator coverage, and changelog aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed `AgentParamsSchema` validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's generic `adapterMeta` namespace plus a temporary `paperclip` compatibility alias is the strongest current candidate and is more maintainable than repeated vendor-only root fields.
