## Label Justifications

- `P1`: Multiple users report a broken Paperclip-to-Gateway agent workflow that prevents tasks from reaching OpenClaw agents.
- `impact:message-loss`: Gateway validation rejects Paperclip agent invocation messages before dispatch, so downstream tasks never enter the agent workflow.

## Summary

Keep open. Current main and v2026.5.19 still reject root-level `paperclip` on Gateway `agent` params, and the open implementation candidate at https://github.com/openclaw/openclaw/pull/74974 targets this protocol gap, so the issue should remain open until that PR or a superseding contract lands.

## What This Changes

Paperclip's `openclaw_gateway` adapter sends root-level `paperclip` metadata in Gateway agent params, but OpenClaw's closed AgentParams schema rejects it before dispatch.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74974 or a superseding additive `AgentParams` metadata contract with schema, generated client model updates, focused validator coverage, and changelog, then close the Paperclip compatibility cluster.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed `AgentParamsSchema` validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's generic `adapterMeta` namespace plus a temporary `paperclip` alias looks more maintainable than repeated vendor-only root fields, but it is still an API/product decision.
