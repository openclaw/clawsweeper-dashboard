## Summary

Keep open. Current main and the latest stable release still reject root-level `paperclip` on Gateway `agent` params, and the open implementation PR that closes this issue has not landed.

## What This Changes

The issue requests compatibility for Paperclip's root-level `paperclip` metadata in Gateway `agent` params, either by accepting it or moving it to a supported metadata path.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74974 or a superseding maintainer-approved additive `AgentParams` metadata contract with schema, generated client model, focused validator coverage, and changelog aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed AgentParams validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's generic `adapterMeta` namespace plus temporary `paperclip` alias is the strongest current implementation candidate and is more maintainable than a permanent vendor-only root field.
