## Summary

Keep open. Current `main` and the latest stable release still reject root-level `paperclip` on Gateway `agent` params, while the linked implementation PR remains open and unmerged.

## What This Changes

The issue asks for OpenClaw Gateway to stop rejecting Paperclip adapter metadata on `agent` request params.

## Best Possible Solution

Land one maintainer-approved additive Gateway `AgentParams` metadata contract with validator coverage, generated client models, docs or changelog alignment, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus root `paperclip`, and current `validateAgentParams` rejects before dispatch.

## Solution Assessment

Unclear. The `adapterMeta` plus temporary `paperclip` alias design in the open PR is a strong candidate, but the public namespace and compatibility alias need maintainer protocol judgment before this issue should close.
