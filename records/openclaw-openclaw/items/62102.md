## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` params, and the open implementation PR is explicitly linked to this issue but has not landed.

## What This Changes

The issue reports that Paperclip's OpenClaw Gateway adapter sends a root-level `paperclip` metadata field that current `AgentParamsSchema` rejects.

## Best Possible Solution

Review and land one maintainer-approved additive Gateway `AgentParams` metadata contract with schema, validator coverage, generated client models, and changelog aligned, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus a root `paperclip` property, and current `validateAgentParams` rejects it before dispatch.

## Solution Assessment

Unclear. The open `adapterMeta` plus `paperclip` compatibility-alias design is a strong candidate, but the public namespace and alias lifecycle need maintainer protocol/API judgment before this issue should close.
