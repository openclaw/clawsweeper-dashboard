## Summary

Keep open. Current main still matches the reported Paperclip failure: Gateway `agent` params are validated against a closed schema with no `paperclip` or generic adapter metadata field, and validation rejects before agent dispatch. Related duplicate reports have been consolidated around this issue, while open implementation PRs remain unmerged and still need a maintainer decision on the public protocol shape.

## What This Changes

Paperclip's `openclaw_gateway` adapter needs its root-level run metadata accepted by OpenClaw Gateway `agent` params or moved to a documented supported metadata path.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, preserve strict rejection for unrelated root typos, add focused regression coverage and any required generated protocol artifacts, docs, and changelog entry, then close this issue and consolidate the duplicate Paperclip reports.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request containing `message`, `sessionKey`, `idempotencyKey`, and a root-level `paperclip` object follows the reported payload and fails current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. A Paperclip-specific field is the narrowest shim for the immediate break, but a generic adapter metadata namespace is likely the more maintainable core protocol surface unless maintainers explicitly approve a vendor-named exception.
