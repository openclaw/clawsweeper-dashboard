## Summary

Keep open: the branch is a plausible Discord queue fix, but the current head imports a timeout helper from a Plugin SDK subpath that does not export it, and the external PR still only supplies synthetic queue-test proof for a user-visible Discord delivery change.

## What This Changes

The PR adds a Discord inbound run watchdog that aborts timed-out queued message processing, commits replay keys, advances the per-session queue, and expands queue regression tests while preserving the shipped timeout constant as deprecated.

## Best Possible Solution

Expose a supported agent-timeout resolver through the Plugin SDK, or use another existing public seam, keep the shipped Discord timeout export as deprecated, add focused type or contract coverage, and require real Gateway/Discord proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows the keyed channel queue has no task timeout, and the PR adds a synthetic regression for a never-settling Discord run, but I did not verify a live current-main Discord/Gateway reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear: tying the watchdog to the existing agent timeout is directionally sound, but the current implementation uses an unsupported SDK import and still needs maintainer acceptance of consumed-on-timeout replay/session overlap semantics.
