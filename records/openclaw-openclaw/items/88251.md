## Summary

Keep open: the queue-stall problem is real and the PR moved in the right direction, but the current patch is not merge-ready because it removes a shipped Discord runtime export and reads `runtime.agent` from a `RuntimeEnv` that does not expose that contract. Real behavior proof is still mock-only.

## What This Changes

Adds a Discord inbound message-run watchdog that aborts a timed-out queued run, commits its replay keys, and lets later same-session Discord messages proceed, while updating Discord queue regression tests and public export guards.

## Best Possible Solution

Keep the watchdog concept, but land it only after preserving the shipped Discord export, sourcing the timeout through a typed plugin/runtime contract, documenting or accepting same-session overlap semantics, and adding redacted live Discord/Gateway proof.

## Reproduction Assessment

Yes from source inspection: current `createChannelRunQueue` has no per-task timeout and Discord awaits each queued processor before the same key can advance. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No, not yet: the watchdog direction addresses the right code path, but this implementation needs a supported timeout resolver contract and must keep the shipped Discord export before it is the best maintainable fix.
