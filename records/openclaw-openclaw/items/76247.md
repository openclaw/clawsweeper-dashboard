## Summary

Keep open. Current main and v2026.6.10 still expose accepted/runId/task tracking paths, but not the requested receiver-entry ACK taxonomy, track/hop ledger, or plugin-readable landing telemetry across the named dispatch surfaces.

## What This Changes

The issue requests native dispatch landing ACK and receiver-entry telemetry across cron, sessions_spawn, channel_inbound, gateway_agent, cli_agent, and acp_downstream dispatch paths.

## Best Possible Solution

Design one redacted, non-blocking dispatch telemetry surface owned by the shared state/plugin API boundary, using the existing state database unless maintainers explicitly approve a separate ledger or env override.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the central gap: gateway can return an accepted ACK, then skip dispatch if abort lands before the scheduled runner. I did not run a live multi-agent deployment repro.

## Solution Assessment

No, the exact sidecar dispatches.db plus OPENCLAW_DISPATCH_LEDGER_PATH design is not yet proven as the best solution. The safer direction is a maintainer-approved telemetry contract that fits OpenClaw's shared SQLite and plugin-boundary policy.
