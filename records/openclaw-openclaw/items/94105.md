## Summary

Keep open: the typed `system_event` direction is plausible, but the patch currently broadens the side-effect `NO_REPLY` bypass for every already-allowed silent turn, including existing `room_event` paths, while the PR says `room_event` behavior is unchanged.

## What This Changes

The branch adds `system_event` to inbound/gateway event contracts, lets system-originated turns end silently on explicit `NO_REPLY`, and documents/tests those semantics.

## Best Possible Solution

Keep the typed event approach only if maintainers accept the public contract, and scope the side-effect explicit-`NO_REPLY` bypass to `system_event` while preserving existing `room_event` and `user_request` behavior with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main hard-blocks direct silent `system_event` because the value does not exist, and PR-head function-level proof exercises the exact silence decision path. The unintended room_event widening is also source-reproducible from the generic helper signature and existing allowed-silent room_event caller.

## Solution Assessment

No: adding a typed event seam is a plausible owner-boundary fix, but the current incomplete-turn change is too broad because it applies the side-effect bypass without knowing the inbound event kind. The safer repair is to thread the event kind or a narrower permission into the helper and test both `system_event` and existing `room_event` behavior.
