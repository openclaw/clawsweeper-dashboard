## Summary

## What This Changes

The PR changes outbound delivery recovery so `send_attempt_started` and `unknown_after_send` entries without adapter reconciliation remain pending, updates related recovery tests, and adds Discord troubleshooting guidance.

## Best Possible Solution

Keep the no-blind-replay mitigation, but make no-reconciliation ambiguous states pending deferrals that do not consume retry budget or bypass later reconciliation, with repeated-drain regression coverage.

## Reproduction Assessment

Yes. Source inspection and the PR body's terminal proof show current recovery refuses blind replay and keeps an ambiguous entry pending, while the remaining blocker is the repeated-drain retry-budget path.

## Solution Assessment

No, not as-is. The shared recovery layer is the right place for this mitigation, but the implementation should not consume ordinary retry budget when no reconciliation-capable adapter was available.
