## Summary

Keep open for maintainer review: the patch looks like a correct narrow mitigation for ambiguous outbound delivery recovery, but it intentionally changes compatibility-sensitive message-delivery behavior by leaving no-reconciler ambiguous rows pending instead of aging them into terminal failure.

## What This Changes

The PR changes outbound delivery recovery so ambiguous send_attempt_started and unknown_after_send rows remain pending without retry-budget consumption when no reconciliation-capable adapter is available, while adapter null/unresolved reconciliation attempts still consume retry budget.

## Best Possible Solution

Merge only if maintainers accept pending no-reconciler ambiguous rows as the interim recovery policy while the broader delivery-ledger work remains tracked under https://github.com/openclaw/openclaw/issues/87544.

## Reproduction Assessment

Yes at source level: current main moves no-adapter ambiguous rows toward failed, while the PR proof shows the after-fix SQLite recovery path leaves the row pending without replay or retry-budget burn. I did not run a live transport restart reproduction in this read-only review.

## Solution Assessment

Yes for the narrow mitigation, subject to maintainer policy acceptance. It preserves the no-blind-replay invariant and separates no-adapter deferral from retryable reconciliation failures; the full durable ledger remains separate work.
