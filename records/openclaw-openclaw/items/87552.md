## Summary

Keep open for maintainer review: I found no blocking code-review defect in the current head, but the PR intentionally changes ambiguous outbound recovery from fail/consume-retry behavior to pending/no-budget deferral, which needs explicit maintainer acceptance before merge.

## What This Changes

The PR changes outbound delivery recovery so ambiguous `send_attempt_started` and `unknown_after_send` entries stay pending without retry-budget burn when no reconciliation-capable adapter is available, while adapter-null or unresolved reconciliation attempts still count against retries, with focused tests and Discord troubleshooting docs.

## Best Possible Solution

Land this only as an accepted interim recovery policy, while keeping the durable delivery-ledger and UI/API visibility work tracked separately under https://github.com/openclaw/openclaw/issues/87544.

## Reproduction Assessment

Yes, at source level: current main moves or budgets ambiguous no-reconciler rows toward failure, while the PR head and terminal proof show the after-fix queue/SQLite row remains pending with retry count unchanged. I did not run a live transport restart in this read-only review.

## Solution Assessment

The patch is an acceptable narrow mitigation, but not the final product solution. The durable end state should still add delivery receipts, reconciliation semantics, and operator/session visibility instead of relying on pending ambiguous rows alone.
