## Summary

Keep open for maintainer review: the patch looks correct for the narrow recovery mitigation, but it changes the fail-closed behavior for ambiguous outbound deliveries without a reconciler, so maintainers need to accept that policy before merge.

## What This Changes

The PR keeps `send_attempt_started` and `unknown_after_send` outbound queue rows pending without retry-budget consumption when no reconciliation-capable adapter is available, treats adapter `null` results as retryable unresolved reconciliation attempts, and updates focused tests plus Discord troubleshooting docs.

## Best Possible Solution

Merge only if maintainers accept pending no-reconciler ambiguous rows as the interim recovery policy while keeping the full delivery-ledger design tracked in https://github.com/openclaw/openclaw/issues/87544.

## Reproduction Assessment

Yes at source level: current main moves no-adapter ambiguous rows toward failure, while the PR proof shows the after-fix SQLite recovery path leaves the row pending without replay or retry-budget burn. I did not establish a live transport restart reproduction in this read-only review.

## Solution Assessment

Yes for the narrow mitigation, subject to maintainer policy acceptance. It preserves the no-blind-replay invariant and separates no-adapter deferral from retryable reconciliation failures; the full durable ledger remains separate work in https://github.com/openclaw/openclaw/issues/87544.
