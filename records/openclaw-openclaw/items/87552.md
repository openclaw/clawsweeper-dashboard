## Summary

Keep open for maintainer review: the patch is a focused outbound queue mitigation with sufficient terminal proof and no blocking code finding, but it changes message-delivery recovery policy by keeping no-adapter ambiguous rows pending instead of terminalizing them.

## What This Changes

The PR changes outbound delivery recovery so ambiguous `send_attempt_started` and `unknown_after_send` entries without reconciliation support stay pending without retry-budget consumption, while adapter-null or unresolved reconciliation attempts still count against retry budget, and updates tests plus Discord troubleshooting docs.

## Best Possible Solution

Merge the narrow deferral only after maintainers accept pending ambiguous rows as the interim recovery state, while keeping the full durable delivery-ledger work in https://github.com/openclaw/openclaw/issues/87544.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 move no-adapter ambiguous rows to failed, while PR head keeps them pending without replay or retry-budget consumption. I did not run a live transport restart reproduction in this read-only review.

## Solution Assessment

Yes for the narrow mitigation, subject to maintainer policy acceptance. It preserves the no-blind-replay invariant and separates no-adapter deferral from retryable reconciliation failures; the full durable ledger remains separate work.
