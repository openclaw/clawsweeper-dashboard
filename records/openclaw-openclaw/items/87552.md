## Summary

Keep open for maintainer review: the patch looks correct as a narrow no-blind-replay mitigation, but it intentionally changes ambiguous outbound rows with no reconciler from fail-closed to pending, so a maintainer should accept that delivery policy before merge.

## What This Changes

This branch keeps ambiguous outbound delivery entries pending without retry-budget consumption when no reconciliation-capable adapter is available, counts completed/null reconciliation attempts against retry budget, and updates focused tests plus Discord troubleshooting docs.

## Best Possible Solution

Merge only if maintainers accept pending no-reconciler ambiguous rows as the interim recovery policy while the full delivery-ledger design remains tracked in https://github.com/openclaw/openclaw/issues/87544.

## Reproduction Assessment

Yes at source level: current main moves no-adapter ambiguous rows toward failure, while the PR proof shows the after-fix SQLite recovery path leaves the row pending without replay or retry-budget burn. I did not establish a live transport restart reproduction in this read-only review.

## Solution Assessment

Yes for the narrow mitigation, subject to maintainer policy acceptance. It preserves the no-blind-replay invariant and separates no-adapter deferral from retryable reconciliation failures; the full durable ledger remains separate work in https://github.com/openclaw/openclaw/issues/87544.
