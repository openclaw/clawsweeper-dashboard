## Summary

Keep open: the PR is a useful, well-proven outbound recovery mitigation, but the current patch exempts every ambiguous entry from the max-retry terminal guard, including entries that did run real adapter reconciliation and received retryable unresolved results.

## What This Changes

The PR changes outbound delivery recovery/storage, tests, and Discord troubleshooting docs so ambiguous send states can remain pending without consuming retry budget when no adapter reconciliation ran.

## Best Possible Solution

Keep the no-adapter deferral behavior, restore the retry cap for real reconciliation attempts, and merge only after maintainers accept the pending-deferral message-delivery policy.

## Reproduction Assessment

Yes, at source level. Current main terminalizes max-retry pending entries, while PR head skips that guard for all ambiguous entries even though retryable unresolved reconciliation still increments retryCount.

## Solution Assessment

No, not as submitted. The no-adapter deferral is a narrow mitigation, but the max-retry exemption must distinguish no-reconciliation deferral from real reconciliation attempts.
