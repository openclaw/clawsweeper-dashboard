## Summary

Keep open for maintainer review: the patch looks correct and proof-sufficient for the shared SQLite recovery path, but merging intentionally changes outbound message-delivery recovery policy for ambiguous rows with no reconciler.

## What This Changes

The PR changes outbound delivery recovery so ambiguous post-send entries without reconciliation support remain pending and recoverable without retry-budget consumption, while adapter reconciliation attempts that return null still count as unresolved attempts.

## Best Possible Solution

Merge the narrow deferral only after maintainers accept pending ambiguous rows as the interim recovery state, while keeping the full durable delivery-ledger work in the canonical issue.

## Reproduction Assessment

Yes, at source level: current main moves no-adapter ambiguous rows to failed, while PR head keeps them pending without replay or retry-budget consumption. I did not run a live transport restart reproduction in this read-only review.

## Solution Assessment

Yes for the narrow mitigation, subject to maintainer policy acceptance. It preserves the no-blind-replay invariant and separates no-adapter deferral from retryable reconciliation failures; the full durable ledger remains separate work.
