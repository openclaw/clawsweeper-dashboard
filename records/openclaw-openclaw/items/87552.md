## Summary

Keep open for maintainer review: the patch looks like a correct narrow mitigation, but it intentionally changes the recovery policy for ambiguous outbound sends and should be accepted as a product/operational tradeoff before merge.

## What This Changes

The PR keeps `send_attempt_started` and `unknown_after_send` outbound queue entries pending without retry-budget burn when no unknown-send reconciler is available, while counting completed unresolved or null reconciliation attempts.

## Best Possible Solution

Land only if maintainers accept pending/no-budget deferral as the interim recovery policy, while keeping the durable delivery-ledger and UI/API visibility work in the canonical issue.

## Reproduction Assessment

Yes at source level: current main's ambiguous-state branch moves no-reconciler rows out of pending, while the PR tests and terminal proof show the after-fix queue/SQLite behavior. I did not run a live Discord or transport restart proof in this read-only review.

## Solution Assessment

The patch is an acceptable narrow mitigation, but not the complete best solution. The full fix remains a durable delivery receipt and adapter reconciliation design so pending ambiguous rows become observable and resolvable.
