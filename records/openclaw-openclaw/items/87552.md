## Summary

Keep open for maintainer review: the PR is a focused, meaningful outbound delivery recovery mitigation that is not already on main, and I found no discrete line-level defect. The remaining blocker is that it deliberately changes ambiguous no-reconciler sends from fail-closed to pending/recoverable, so maintainers need to accept that interim message-delivery policy before merge.

## What This Changes

The PR keeps ambiguous outbound delivery entries pending without retry-budget consumption when no reconciliation-capable adapter is available, while still counting completed unresolved reconciliation attempts against the retry cap.

## Best Possible Solution

Merge the narrow deferral only after maintainers accept pending ambiguous no-reconciler rows as the interim recovery state, while keeping the full delivery-ledger work in the canonical issue.

## Reproduction Assessment

Yes at source level: current main moves no-adapter ambiguous rows to failed, while PR head keeps them pending without replay or retry-budget consumption. I did not run live transport restart proof in this read-only review.

## Solution Assessment

Yes for the narrow mitigation, subject to maintainer policy acceptance. It preserves the no-blind-replay invariant and separates no-adapter deferral from retryable reconciliation failures; the full durable ledger remains separate work.
