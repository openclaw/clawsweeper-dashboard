## Summary

Keep open: current main still resolves the CLI compaction context engine before registering the built-in `legacy` engine, and an open closing PR already owns the narrow fix. The follow-up Slack boot comment appears to be the same root cause, so this should close only after the canonical fix PR merges or is abandoned.

## What This Changes

The issue reports that CLI transcript persistence resolves the default `legacy` context engine before built-in engines are registered, causing repeated registration errors and degraded session persistence.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79484 as the canonical initializer-order fix, then close this issue as implemented and dedupe the related legacy-context-engine threads.

## Reproduction Assessment

Yes, at source level: a persisted CLI session with `sessionFile` and `contextTokens` enters `runCliTurnCompactionLifecycle`, which resolves the context engine before any built-in initializer. An empty registry then produces the exact default `legacy` unregistered error; I did not run the live macOS or Slack command.

## Solution Assessment

Yes. Calling the existing idempotent context-engine initializer immediately before CLI compaction resolves the engine is the narrowest maintainable fix and matches the already-migrated subagent and embedded compaction paths.
