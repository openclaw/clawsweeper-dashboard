## Summary

Keep open: current main still resolves the default context engine in CLI compaction before registering the built-in `legacy` engine, and https://github.com/openclaw/openclaw/pull/79484 is an open closing PR for this issue, so this should not be closed before that PR merges or is abandoned.

## What This Changes

Fix gateway/CLI transcript persistence so the built-in `legacy` context engine is registered before CLI compaction/session persistence resolves the default engine.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79484 if its exact-head gates are acceptable, then close this issue as implemented and dedupe the older root-cause threads.

## Reproduction Assessment

Yes, at source level: a persisted CLI session with `sessionFile` and `contextTokens` enters `runCliTurnCompactionLifecycle`, which resolves the context engine before any built-in initializer. I did not run the live `openclaw agent` command in this read-only review.

## Solution Assessment

Yes; the narrowest maintainable fix is to call the existing idempotent initializer before `resolveContextEngine`, matching adjacent subagent and embedded compaction paths. The linked PR implements that direction, so opening a duplicate repair lane would be counterproductive.
