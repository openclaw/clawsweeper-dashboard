## Summary

Keep open: current main still has the CLI compaction path resolving the default `legacy` context engine before the built-in context-engine initializer runs, and the directly linked fix PR remains open. The later Slack-channel comment is useful because it reports the same root cause in a fatal gateway boot mode, so the issue should wait for the linked fix to land rather than be closed early.

## What This Changes

The issue reports that CLI transcript persistence fails because the default `legacy` context engine is resolved before any built-in context engine is registered.

## Best Possible Solution

Land one canonical initializer-order fix, preferably https://github.com/openclaw/openclaw/pull/79484 after review and CI, then close this issue and reconcile duplicate reports or older duplicate PRs.

## Reproduction Assessment

Yes, at source level: a persisted CLI session enters `runCliTurnCompactionLifecycle`, which resolves the default `legacy` context engine before initializing the built-in registry, and the resolver throws when that registry is empty. I did not run the live `openclaw agent` command in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to call the existing idempotent `ensureContextEnginesInitialized()` helper before CLI compaction resolves the context engine, matching adjacent embedded and subagent paths; the open linked PR implements that direction.
