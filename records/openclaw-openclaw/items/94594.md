## What This Changes

The PR adds visited-agent provenance to gateway and agent tool paths, then blocks `sessions_send` before delivery when the target agent already appears in the inter-session chain.

## Best Possible Solution

Land the normalized `InputProvenance` chain guard after real runtime proof, the duplicate error-chain cleanup, and maintainer acceptance of the fail-closed cyclic-route behavior.

## Reproduction Assessment

Yes at source level: current main has only pairwise A2A ping-pong limiting and no propagated visited-agent chain, so A -> B -> C -> A can bypass the existing counter. I did not run a live multi-agent reproduction in this read-only pass.

Yes, with conditions: carrying normalized visited-agent state in `InputProvenance` is the right owner boundary because gateway and in-process handoffs already consume that contract. The PR still needs real runtime proof, maintainer acceptance of the fail-closed policy, and a small error-message cleanup.
