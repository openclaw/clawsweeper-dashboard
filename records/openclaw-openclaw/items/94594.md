## Summary

Keep open: this is a focused candidate fix for the linked graph-aware `sessions_send` loop bug, but it still needs real multi-agent runtime proof and maintainer acceptance of the compatibility-sensitive fail-closed route policy before merge.

## What This Changes

The PR adds a normalized `visitedAgentIds` provenance chain and threads it through gateway and agent handoffs so `sessions_send` can block delivery when the target agent already appears in the chain.

## Best Possible Solution

Land the provenance-chain guard after redacted real runtime proof, duplicate-chain message cleanup, and explicit maintainer acceptance of revisiting an agent as a cycle that fails closed.

## Reproduction Assessment

Yes at source level: current `main` has only pairwise A2A ping-pong limiting and no propagated visited-agent chain, so A -> B -> C -> A can bypass the existing counter. I did not run a live multi-agent deployment in this read-only review.

## Solution Assessment

Yes, with conditions: carrying normalized visited-agent state in `InputProvenance` is the right boundary because gateway and in-process handoffs already consume that contract. The PR still needs real runtime proof, maintainer acceptance of the fail-closed behavior, and a small error-message cleanup.
