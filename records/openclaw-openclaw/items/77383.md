## Summary

Keep open: current main deliberately treats Feishu/Lark broadcast-only `@all` as not mentioning the bot, and `respondToMentionAll` is not a documented or schema-supported OpenClaw option, so this needs a Feishu plugin product/config decision rather than cleanup closure.

## What This Changes

The issue requests an opt-in Feishu behavior where group `@所有人` messages can trigger replies under mention gating via `respondToMentionAll`.

## Best Possible Solution

If maintainers want this behavior, add a documented Feishu-owned opt-in that detects broadcast mentions separately from direct bot mentions and preserves the current safe default.

## Reproduction Assessment

Yes for current-main behavior by source inspection: broadcast-only `@all` is filtered out of bot mention detection, and mention gating returns before dispatch when `requireMention` is true. No live Feishu WebSocket payload was exercised, and `respondToMentionAll` is not a current OpenClaw contract.

## Solution Assessment

No for the suggested raw-content fallback as the whole fix; it risks restoring the prior `@_all` false-positive behavior. The safer direction is an explicit opt-in config plus structured parser coverage for text/post payload variants.
