## Summary

Keep open: current main intentionally treats Feishu broadcast-only `@all`/`@_all` as not being a bot mention, and `respondToMentionAll` is not a documented or schema-supported OpenClaw Feishu option. The report is valid as a request for new plugin behavior, but it conflicts with the shipped false-positive fix from #37706 and needs maintainer product judgment.

## What This Changes

The issue requests that Feishu group `@所有人` messages be recognized so `respondToMentionAll: true` allows the bot to reply when `requireMention` is enabled.

## Best Possible Solution

If maintainers want broadcast mentions to wake Feishu bots, add a plugin-owned, documented opt-in that detects structured `@all` safely without reclassifying broadcasts as direct bot mentions, and preserve the #37706 false-positive protections.

## Reproduction Assessment

Yes for the current behavior: source and tests show broadcast-only Feishu `@all` does not set `mentionedBot`, so `requireMention` gating returns before dispatch. No for a broken current contract, because `respondToMentionAll` is not supported by the current schema or docs.

## Solution Assessment

No for the proposed raw-content fallback as written; it risks recreating the false-positive class fixed for #37706. A safer solution is an explicit Feishu plugin option with structured parser coverage for text and post payload variants.
