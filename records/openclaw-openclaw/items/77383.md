## Summary

Keep open: current main intentionally treats broadcast-only Feishu `@all` as not being a bot mention and has no `respondToMentionAll` config surface, so this is a Feishu behavior/product request rather than an already-fixed bug.

## What This Changes

The issue asks for Feishu broadcast `@all` messages to trigger replies when a `respondToMentionAll` opt-in is configured.

## Best Possible Solution

If maintainers want this behavior, add a documented Feishu plugin opt-in that detects broadcast mentions separately from direct bot mentions and gates on that separate flag only when enabled.

## Reproduction Assessment

Yes for the current observable behavior: source inspection shows broadcast-only `@all` does not set `mentionedBot`, and mention-required groups return before dispatch. No for a broken current-main contract, because `respondToMentionAll` is not documented or accepted by the current Feishu schema.

## Solution Assessment

No for the suggested raw-content fallback as the whole fix, because it could recreate the prior `@_all` false-positive path. A safer solution is a Feishu-owned opt-in with structured text/post payload coverage and no change to direct bot mention semantics.
