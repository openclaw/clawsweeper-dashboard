## Summary

Keep open. Current main does not implement the reported `respondToMentionAll` contract, and the requested behavior conflicts with the current documented Feishu policy that broadcast-only `@all` is not a bot mention, so this needs maintainer/plugin-owner direction rather than cleanup closure.

## What This Changes

Request Feishu/Lark group `@all` messages to pass mention gating when an opt-in `respondToMentionAll` policy is configured.

## Best Possible Solution

Decide whether Feishu/Lark should expose a documented opt-in broadcast-response policy; if yes, implement it as separate mention-all detection and gate logic in the owning plugin, not by folding `@all` back into `mentionedBot`.

## Reproduction Assessment

No live Feishu/Lark reproduction was performed, but source inspection gives a medium-confidence path: current main has no `respondToMentionAll` option or `mentionAll` context, and broadcast-only `@all` is filtered out before mention gating. The exact external `@larksuite/openclaw-lark` runtime path was not available in this checkout.

## Solution Assessment

No, the suggested raw-content fallback is not clearly the best fix because raw `@_all` matching caused the prior #37706 false positive. A safer solution would add an explicit opt-in broadcast policy that stays separate from direct bot mention detection.
