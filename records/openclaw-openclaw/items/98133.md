## What This Changes

The PR adds a module-local 16 MiB `maxPayload` cap to both Codex Supervisor WebSocket client constructors and adds a loopback regression test for oversized and normal frames.

## Best Possible Solution

Merge a scoped WebSocket `maxPayload` cap only after redacted real Codex app-server/supervisor proof and maintainer acceptance or adjustment of the cap value.

## Reproduction Assessment

Yes, from source: current main omits `maxPayload` on both Codex Supervisor WebSocket constructors, while `ws` 8.21.0 defaults the client limit to 100 MiB and enforces configured lower limits. I did not run a live hostile-frame Codex daemon repro in this read-only review.

## Solution Assessment

Yes, with merge-gate caveats: `ws` `maxPayload` is the right enforcement layer because it rejects oversized frames before JSON parsing, and sibling WebSocket surfaces use the same mechanism. The cap size still needs maintainer acceptance or tuning, and the body should not rely on the unrelated https://github.com/openclaw/openclaw/pull/97483 reference as supporting proof.
