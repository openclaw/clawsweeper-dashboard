The PR adds a module-local 16 MiB `maxPayload` cap to both Codex Supervisor WebSocket constructors and adds loopback regression tests for oversized and normal frames.

## Best Possible Solution

Merge a scoped `maxPayload` cap after redacted real Codex app-server/supervisor proof and maintainer acceptance or adjustment of the cutoff value.

## Reproduction Assessment

Yes, source-reproducible: current main omits `maxPayload` on both Codex Supervisor WebSocket constructors, while `ws` 8.21.0 defaults the client limit to 100 MiB and enforces configured lower limits. I did not run a live hostile-frame Codex app-server repro in this read-only review.

## Solution Assessment

Yes, with merge-gate caveats: `ws` `maxPayload` is the right layer because it rejects oversized frames before JSON parsing, and sibling WebSocket surfaces use the same mechanism. The cap value still needs maintainer acceptance or tuning, and the external proof remains test-harness only.
