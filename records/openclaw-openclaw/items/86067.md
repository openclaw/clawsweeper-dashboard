## Summary

Keep open: the central fix is useful and the proof is strong, but the branch still leaves the new prompt-turn queue uncleared when current-main teardown disposes a controller during an active prompt window, so later same-file runs can hang.

## What This Changes

The PR adds a per-session-file prompt-window queue for embedded runner locks, adds a non-prompt idle-wait release path, and extends session-lock regression coverage.

## Best Possible Solution

Keep the file-scoped queue design, but make controller disposal release and clear any active prompt turn and cover that teardown path with a focused regression before merge.

## Reproduction Assessment

Yes. Source inspection of the merge result shows releaseForPrompt() can leave activePromptSessionTurn set with no heldLock, while current-main teardown calls dispose() and dispose() returns without clearing that turn; I did not execute tests because this review is read-only.

## Solution Assessment

No, not yet. The file-scoped queue is the right shape for the reported cross-lane race, but it is incomplete until the controller teardown path releases the new queue resource too.

- **[P2] Release prompt turns during controller disposal:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:850`
  - body: After this line, the controller owns a prompt turn while `heldLock` is cleared.
