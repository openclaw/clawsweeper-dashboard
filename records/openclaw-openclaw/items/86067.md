## Summary

Keep open for maintainer review: the patch is a focused internal fix with sufficient terminal proof and no blocking correctness finding, but the new per-session-file prompt queue is availability-sensitive if a provider stream never returns or abort cleanup never reaches the turn release.

## What This Changes

The PR adds a process-global, file-scoped prompt-window queue for embedded runner session locks, separates compaction idle release from prompt release, and adds regression coverage for same-file contention and cleanup paths.

## Best Possible Solution

Land only after an embedded-runner owner accepts the per-file prompt-window serialization tradeoff; keep the same-lane fixes in https://github.com/openclaw/openclaw/pull/86584 and https://github.com/openclaw/openclaw/pull/86585 as separate review paths.

## Reproduction Assessment

Yes. The source path is clear from releaseForPrompt/reacquireAfterPrompt and the PR supplies before/after terminal proof with real controllers, but I did not rerun the harness in this read-only review.

## Solution Assessment

Yes, with a maintainer-risk caveat. Serializing only provider prompt windows while keeping compaction idle release separate is a narrow fix for the cross-lane race, but owners must accept the availability behavior for indefinitely hung provider streams.
