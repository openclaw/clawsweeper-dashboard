## Summary

Keep open for maintainer review: the source diff, focused regressions, and supplied real-output proof support the fix, and I found no blocking patch defect. The remaining question is whether maintainers accept the availability tradeoff of a process-global same-file prompt queue alongside the companion session-takeover fixes.

## What This Changes

The branch adds a per-resolved-session-file prompt-turn queue in the embedded runner, separates post-prompt idle release from real prompt release, and adds regression tests for serialization and cleanup paths.

## Best Possible Solution

Land the file-scoped guard only if maintainers accept the bounded availability tradeoff, then keep the same-lane and fork-specific fence fixes tracked in their companion PRs.

## Reproduction Assessment

Yes. The issue and PR provide a high-confidence source and harness reproduction: two real lock controllers on one session JSONL race before the patch, while the patched head serializes the second prompt window; I did not rerun it in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the availability tradeoff. Serializing only prompt windows by resolved session file is a narrow root-cause fix for the cross-lane race, and separating compaction idle release avoids over-serializing non-prompt waits.
