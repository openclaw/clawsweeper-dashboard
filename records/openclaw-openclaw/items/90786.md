## Summary

Keep open: current main and v2026.6.1 still register Google memory embeddings under the `gemini` memory adapter id, while `memory status --deep` and `--index` initialize the configured provider and can still throw `Unknown memory embedding provider: google`. The report is source-reproducible, but the right fix needs a maintainer decision because docs currently name `gemini` as the canonical memory provider id.

## What This Changes

The issue reports that Google-configured memory embedding initialization fails for `memory status --index` and `--deep` with `Unknown memory embedding provider: google` after upgrading to OpenClaw 2026.6.1.

## Best Possible Solution

Define a canonical upgrade path for `memorySearch.provider: "google"`: either migrate it to `gemini` with doctor/config tests if it was a shipped shape, or make CLI status/deep/index emit a clear documented-provider diagnostic, with regression coverage for both `--deep` and `--index`.

## Reproduction Assessment

Yes, source-reproducible: with `memorySearch.provider` set to `google`, the deep/index command path initializes the provider and `getAdapter("google")` has no matching memory adapter, so it throws. I did not run a live Google-key command because this was a read-only review.

## Solution Assessment

Unclear as requested: treating `google` as a direct memory provider alias may be user-friendly, but docs and current contracts name `gemini` as the adapter id. The maintainable fix should first choose between alias support, doctor migration, or a clearer unsupported-provider diagnostic.
