## Summary

Keep open: current main and latest release v2026.6.8 still expose the Google memory embedding adapter as `gemini` only, while two closing PRs for the `google` provider-id path remain open with unresolved compatibility/product-contract review.

## What This Changes

The issue reports that `openclaw memory status --index` and `--deep` fail for `memorySearch.provider: "google"` because the Google memory embedding adapter is registered as `gemini`.

## Best Possible Solution

Keep this issue open while maintainers choose the provider-id contract, then land one narrow linked fix that covers manifest/startup/runtime/docs/tests or add explicit migration and diagnostics to keep `gemini` canonical.

## Reproduction Assessment

Yes. Source inspection shows `memorySearch.provider: "google"` reaches id-based provider lookup while the Google memory adapter and manifest are keyed as `gemini`, matching the reported unknown-provider throw path; no live CLI run was performed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A `google` alias is plausible only if it is supported end to end through manifest/startup/runtime/docs/tests; otherwise the better fix is canonical `gemini` migration or diagnostics.
