## Summary

Keep open: current main and latest release v2026.6.8 still leave `memorySearch.provider: "google"` unresolved for memory embeddings, and the two linked closing PRs are open rather than merged. The issue should stay open while maintainers choose the permanent provider-id contract and land or close one canonical fix path.

## What This Changes

The issue reports that `openclaw memory status --index` and `--deep` fail for Google-backed memory search because `memorySearch.provider: "google"` cannot resolve to the registered Gemini memory embedding adapter.

## Best Possible Solution

Review the two linked fix PRs against the provider/config compatibility contract, then land one canonical path that makes `memorySearch.provider: "google"` work or migrates/diagnoses it clearly without creating drifting memory provider semantics.

## Reproduction Assessment

Yes. Source inspection shows `memorySearch.provider: "google"` reaches provider initialization while current main registers the Google memory adapter under `gemini` only, and the shared lookup does not map `authProviderId` back to an adapter id.

## Solution Assessment

Unclear until maintainers choose the supported contract. The shared `authProviderId` fallback PR and the plugin-local alias PR are both plausible, but they encode different long-term provider semantics.
