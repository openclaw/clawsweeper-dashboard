## Label Justifications

- `P1`: This regression blocks semantic memory indexing and deep status for users configured with Google-backed memory embeddings.
- `impact:auth-provider`: The bug is a provider-id routing mismatch between the Google auth/provider id and the Gemini memory embedding adapter.
- `impact:session-state`: The failure can leave semantic memory search paused or unavailable for affected agents.

## Summary

Keep open: current main and v2026.6.1 still preserve `memorySearch.provider: "google"` into memory-core provider initialization, while the bundled Google memory embedding adapter is registered under `gemini`, so the reported unknown-provider path remains source-reproducible.

## What This Changes

The issue reports that `openclaw memory status --index` and `--deep` fail with `Unknown memory embedding provider: google` when Google is configured for memory search after upgrading to v2026.6.1.

## Best Possible Solution

Choose and document the Google memory provider contract, then either support `google` as a tested alias for the `gemini` memory adapter or migrate/explain it through doctor with a clearer diagnostic.

## Reproduction Assessment

Yes by source inspection. With `memorySearch.provider: "google"`, current main preserves that id and passes it to memory-core provider initialization, while the bundled Google memory adapter is registered as `gemini`, matching the reported unknown-provider path.

## Solution Assessment

Unclear as a product-contract choice. Supporting `google` as an alias is a plausible narrow fix, but maintainers should first decide whether alias support or doctor migration to the documented `gemini` id is the best long-term contract.
