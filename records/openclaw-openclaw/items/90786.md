## What This Changes

The issue asks OpenClaw to stop `memory status --index` and `--deep` from failing with `Unknown memory embedding provider: google` when Google-backed memory search is configured.

## Best Possible Solution

Pick one documented memory provider-id contract, then land or replace one linked PR with an end-to-end fix covering startup, runtime, docs or diagnostics, and upgrade behavior.

## Reproduction Assessment

Yes by source inspection: `memorySearch.provider: "google"` is looked up as a provider id, while current Google memory registration and manifest ownership are keyed as `gemini`. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the contract. Supporting `google` as an alias is plausible, but the best fix must cover cold startup, runtime lookup, diagnostics/docs, and upgrade behavior; otherwise `gemini` should remain canonical with migration or diagnostic guidance.
