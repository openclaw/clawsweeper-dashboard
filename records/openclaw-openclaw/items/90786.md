## Summary

Keep open: current main and the latest release still have the `google` versus `gemini` memory embedding provider mismatch, and two open closing PRs already own competing repair paths that need maintainer direction.

## What This Changes

The issue reports that `openclaw memory status --index` and `--deep` reject `memorySearch.provider: "google"` after upgrade because the Google memory embedding adapter is registered under `gemini`.

## Best Possible Solution

Review the open closing PRs, choose one canonical provider-id contract that covers runtime lookup, startup activation, docs, diagnostics, and tests, then close this issue after that repair merges.

## Reproduction Assessment

Yes by source inspection: with `memorySearch.provider: "google"`, current main passes `google` into memory provider initialization, while the registered Google memory adapter and manifest contract are `gemini`, leading to the observed unknown-provider throw. I did not run a live Google API indexing command in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. A centralized resolver or a Google-owned alias can both address the symptom, but the best fix must also cover cold startup activation, diagnostics, docs, and upgrade compatibility.
