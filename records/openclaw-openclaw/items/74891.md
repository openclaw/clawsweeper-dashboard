## Summary

Keep this PR open. The patch is a plausible narrow direction, but source review found a blocking gap: it skips credential prompts only for unselected skills that still have missing binaries, so the linked Whisper case can still prompt for `OPENAI_API_KEY` when `openai-whisper-api` is missing only its env var because `curl` is already available.

## What This Changes

The PR records selected skills from the onboarding dependency multiselect, skips primary-env prompts for unselected bin-backed skills, and adds two onboarding skill regression tests.

## Best Possible Solution

Keep the fix in `setupSkills`, but make credential prompting match an explicit user choice: selecting only local `openai-whisper` should not prompt for the separate API skill even when that API skill is env-only. Add regression coverage for `openai-whisper-api` with `missing.bins: []` and `missing.env: ["OPENAI_API_KEY"]`, and add the required Unreleased changelog entry.

## Reproduction Assessment

Yes. A high-confidence mocked reproduction is to return missing local `openai-whisper` plus env-only `openai-whisper-api` with `curl` satisfied, select only `openai-whisper`, and observe that this PR still reaches the `Set OPENAI_API_KEY for openai-whisper-api?` confirm path.

## Solution Assessment

No. The proposed guard is narrower than the linked bug because it treats env-only primary-env skills as always promptable; the safer solution is to gate that prompt on explicit selection/configuration or add an explicit env-only skill selection step.
