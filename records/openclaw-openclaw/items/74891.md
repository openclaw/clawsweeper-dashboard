## Summary

Keep open. The PR targets a real onboarding bug from #74382, but the proposed guard only skips credential prompts for unselected skills that still have missing binaries. When `curl` is already present, `openai-whisper-api` is missing only `OPENAI_API_KEY`, so this PR can still ask for the API-backed skill's key after the user selected only the local keyless Whisper skill. The PR also lacks the required changelog entry for a user-facing production change.

## What This Changes

This PR records selected skills in the onboarding dependency multiselect, skips primary-env prompts for unselected skills with missing binaries, and adds regression tests for skipped and selected install paths.

## Best Possible Solution

Land one coherent fix for #74382 that prevents the keyless local Whisper choice from being followed by an unrelated API-key prompt. The fix should cover the curl-present env-only path, keep or improve the user-facing credential framing, add focused regression coverage, and include a one-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to model `openai-whisper-api` with `curl` satisfied and only `OPENAI_API_KEY` missing, select only local `openai-whisper`, and observe that this PR's `missing.bins.length > 0` guard still lets the API-key confirm prompt run.

## Solution Assessment

No. Tracking selected installs is useful for skipped binary installs, but it is not sufficient because env-only missing skills bypass the new guard; the safer fix is to tie credential prompts to an explicit user-visible skill choice or otherwise exclude unrelated API-backed skills in the env-only path.
