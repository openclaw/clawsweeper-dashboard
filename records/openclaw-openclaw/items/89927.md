## Label Justifications

- `P1`: The report describes a broken long-running agent workflow that can duplicate paid subagent work and wedge active sessions.
- `impact:session-state`: The central failure is concurrent lanes mutating shared session, heartbeat, worklog, and transcript-lock state.
- `impact:message-loss`: The report says subagent work can be dispatched and billed without appearing in the orchestrator worklog or producing delivered artifacts.
- `impact:crash-loop`: The reported lock contention can wedge a session until Gateway restart, which is an availability failure.

## Summary

Keep open: current main and the latest release still allow CLI Gateway timeout fallback to start a fresh `gateway-fallback-*` embedded lane without first proving the accepted Gateway run was aborted or quiesced. Older related fixes addressed direct transcript-lock reuse and same-file prompt serialization, but they do not close this accepted-run ownership question.

## What This Changes

The issue reports that Gateway timeout failover starts fallback main sessions without quiescing the parent lane, causing duplicate subagent dispatches, shared session-state writes, and wedged sessions.

## Best Possible Solution

Define a single-owner Gateway timeout fallback contract: either abort/quiesce the accepted Gateway run before embedded fallback starts, or suppress fallback unless ownership transfer is confirmed.

## Reproduction Assessment

Yes for the central source path: current main and `v2026.6.6` still allow a fresh embedded `gateway-fallback-*` run after a Gateway timeout without proving the accepted run was aborted. No live long-running wedged-session reproduction was run in this read-only review.

## Solution Assessment

Unclear as a final product choice. Aborting or quiescing the accepted Gateway run before fallback appears to be the right owner-boundary fix, but maintainers should choose whether timeout fallback cancels late-finishing Gateway work or suppresses fallback when abort cannot be confirmed.
