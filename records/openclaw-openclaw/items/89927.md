## Summary

Keep open: current main and v2026.6.8 still let a CLI Gateway final-wait timeout start a fresh `gateway-fallback-*` embedded lane, while accepted-run abort is only wired to CLI signal interrupts and maintenance expiry is not an explicit ownership transfer before fallback starts.

## What This Changes

The issue asks OpenClaw to terminate, quiesce, or otherwise single-own the parent Gateway lane before starting a `gateway-fallback-*` fallback session.

## Best Possible Solution

Define a single-owner Gateway timeout fallback contract: abort or quiesce the accepted parent run before embedded fallback starts, or suppress fallback unless parent ownership transfer is confirmed.

## Reproduction Assessment

Yes, at source level: an accepted Gateway run whose final wait times out reaches the `gateway-fallback-*` embedded fallback branch without first confirming the accepted run was aborted or quiesced. I did not run the long live duplicate-billing scenario in this read-only review.

## Solution Assessment

Unclear as a final product choice: aborting or quiescing the accepted run before fallback appears to be the right owner-boundary direction, but maintainers must choose whether timeout fallback cancels late-finishing Gateway work or suppresses fallback when abort cannot be confirmed.
