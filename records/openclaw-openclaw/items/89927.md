## Summary

Keep open: current main and v2026.6.8 still allow CLI Gateway timeout fallback to start a fresh `gateway-fallback-*` embedded lane while the accepted Gateway run is only aborted for CLI signal interrupts, so the accepted-run ownership problem remains.

## What This Changes

The issue requests a single-owner rule for CLI Gateway timeout fallback so a fresh embedded fallback lane cannot run concurrently with the original accepted Gateway lane on related session state.

## Best Possible Solution

Define and implement a single-owner Gateway timeout fallback contract: abort or quiesce the accepted Gateway run before embedded fallback starts, or suppress fallback unless ownership transfer is confirmed.

## Reproduction Assessment

Yes, for the central source path: current main creates a fresh embedded `gateway-fallback-*` session on Gateway timeout without proving the accepted Gateway run was aborted or quiesced. The full long-running duplicate-billing sequence was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear as a final product choice: aborting or quiescing the accepted run before fallback appears to be the right owner-boundary direction, but maintainers must choose whether timeout fallback cancels late-finishing Gateway work or suppresses fallback when abort cannot be confirmed.
