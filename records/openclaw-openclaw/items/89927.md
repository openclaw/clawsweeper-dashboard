## Label Justifications

- `P1`: The issue describes a current core orchestration path that can duplicate paid work, lose useful subagent output, and wedge active sessions.
- `impact:session-state`: The implicated fallback path can leave parent and fallback lanes mutating related session, heartbeat, worklog, and transcript state.
- `impact:message-loss`: The reported phantom subagent dispatches were not recorded by the active orchestrator worklog or delivered as useful artifacts.
- `impact:crash-loop`: The report includes lock contention and wedged sessions that require Gateway restart or manual recovery.

## Summary

Keep open: current main and v2026.6.9 still use a fresh `gateway-fallback-*` embedded lane after an accepted Gateway run times out, without first aborting or quiescing that accepted parent run. The related session-lock delivery PR is useful context, but it explicitly leaves session scheduling and lock ownership out of scope, so it does not replace this issue.

## What This Changes

Ensure Gateway timeout fallback sessions do not leave the accepted parent orchestrator lane active on the same shared session state.

## Best Possible Solution

Define a single-owner Gateway timeout fallback contract: abort or quiesce the accepted parent run before embedded fallback starts, or suppress fallback unless parent ownership transfer is confirmed, with regression coverage for the chosen behavior.

## Reproduction Assessment

Yes, at source level: an accepted Gateway run whose final wait times out reaches the `gateway-fallback-*` embedded fallback branch without first confirming the accepted run was aborted or quiesced. I did not run the long live duplicate-billing scenario in this read-only review.

## Solution Assessment

Unclear as a final product choice: aborting or quiescing the accepted run before fallback appears to be the right owner-boundary direction, but maintainers must choose whether timeout fallback cancels late-finishing Gateway work or suppresses fallback when abort cannot be confirmed.
