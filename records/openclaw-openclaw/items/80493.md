## Summary

Keep this PR open: it is an active external PR with the protected `maintainer` label, it targets the real Codex `/btw` transport bug tracked at https://github.com/openclaw/openclaw/issues/80469, but it is not merge-ready because the new side-thread path appears to keep Codex native code-mode tools available and the PR has no after-fix real behavior proof.

## What This Changes

The PR adds an optional agent-harness side-question hook implemented by Codex app-server side-thread forks, updates OpenAI/Codex auth ordering and fallback account display, and documents the new `/btw` and auth behavior.

## Best Possible Solution

Land a Codex-native `/btw` path that preserves Codex auth and thread context while enforcing a true no-tools side turn, then include real proof from the reported setup before maintainer review completes.

## Reproduction Assessment

Source-level yes, live proof no. Current main’s `/btw` path bypasses the Codex harness and streams through provider auth, which matches the linked Codex OAuth failure mode; I did not run a live Telegram/Codex reproduction.

## Solution Assessment

Not yet. A harness-native side thread is the right direction for the bug, but this implementation must enforce the no-tools contract and needs after-fix real behavior proof before it is the best merge path.
