## Summary

Keep open: this is a plausible fix for the linked Codex OAuth `/btw` transport bug, but the latest PR still needs real behavior proof and has blocking review findings around the no-tools contract and public SDK baseline hygiene.

## What This Changes

The PR adds an optional agent-harness side-question hook, routes Codex `/btw` and `/side` through an ephemeral Codex app-server side thread, updates OpenAI/Codex auth ordering, and adds docs, changelog, and tests.

## Best Possible Solution

Keep the native Codex transport direction, but make the side path programmatically preserve `/btw` isolation and no-tools semantics, then refresh the SDK contract artifacts and add redacted live Telegram/Codex OAuth proof.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. Source inspection on current main does show `/btw` resolving the runtime model/auth and then using the generic provider stream path, which matches the linked Codex OAuth 401 transport mismatch.

## Solution Assessment

No, not as written. Native Codex side threads are the right repair direction, but the PR still needs a programmatic no-tools guarantee and public SDK baseline follow-through before it is the narrow maintainable fix.
