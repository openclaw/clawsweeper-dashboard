## Summary

Keep this PR open for maintainer review: the implementation is a plausible fix direction for Telegram DM topics, but it retires existing user-facing `threadReplies` config and changes session-key behavior, which is compatibility- and session-state-sensitive before merge.

## What This Changes

The PR removes Telegram DM thread reply config from validation, generated metadata, UI hints, and docs, derives DM topic session splitting from `getMe.has_topics_enabled`, and adds doctor cleanup for retired keys.

## Best Possible Solution

Choose and document the permanent Telegram DM topic contract: either keep explicit overrides with capability-based automatic default behavior, or explicitly approve the config retirement with upgrade notes, doctor migration, and live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: current main keeps DM `message_thread_id` sessions flat unless explicit `threadReplies`, `requireTopic`, or topic config opts in, so a bot with DM topics and no config can mix topics. I did not run a live Telegram bot repro in this read-only review.

## Solution Assessment

No, not yet: deriving the default from `getMe.has_topics_enabled` is well supported, but deleting the existing explicit override contract is not the narrowest safe fix without maintainer approval and upgrade proof.
