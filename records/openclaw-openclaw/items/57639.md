## Summary

Keep open: current main still only has concurrent, nesting-depth, active-child, timeout, archive, and liveness controls for subagents. I did not find a schema field, runtime guard, doc entry, or test for a lifetime per-requester-session spawn cap equivalent to the requested `maxPerSession`/`maxTotal`, and the native plus ACP paths both enforce active-child counts rather than completed sequential totals.

## What This Changes

Request adds an `agents.defaults.subagents.maxPerSession` or `maxTotal` lifetime cap so a session cannot spawn unbounded sequential subagents even when concurrent and active-child limits are respected.

## Best Possible Solution

Add an optional, backward-compatible `agents.defaults.subagents.maxPerSession` lifetime budget with no cap when unset, include it in config types/schema/generated docs/help and the subagents guide, and enforce it in both native and ACP `sessions_spawn` paths by counting all child runs for the requester session rather than only active children. Keep the adjacent per-agent concurrency request separate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is enough here: current strict config/schema has no requested field, and both spawn paths gate on `countActiveRunsForSession`, so sequential completed children can continue spawning after active count drops.

## Solution Assessment

Yes, the requested optional per-session lifetime cap is the narrow maintainable direction for this issue. Lowering `maxConcurrent` or `maxChildrenPerAgent` only limits simultaneous active fan-out and does not address accumulated sequential child results.
