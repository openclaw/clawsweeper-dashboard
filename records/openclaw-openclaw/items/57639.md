## Summary

Keep this open: current main still has global concurrency, depth, and active-child controls, but no typed, validated, documented, or enforced lifetime per-session cap for sequential subagent spawns.

## What This Changes

The issue requests an optional `subagents.maxPerSession`-style limit that caps total subagent spawns over a requester session, not only concurrently active children.

## Best Possible Solution

Add an optional documented lifetime spawn budget that defaults to current behavior when unset and is enforced consistently for native and ACP `sessions_spawn` requests with clear reset/archive semantics.

## Reproduction Assessment

Yes. Source inspection shows sequential spawns can continue after prior children end because both native and ACP spawn guards only count active children, and the registry test confirms ended children no longer count.

## Solution Assessment

Yes on direction, pending maintainer naming and reset-semantics review. A positive optional `maxPerSession`-style config enforced at spawn time is the narrowest maintainable shape if unset preserves existing behavior.
