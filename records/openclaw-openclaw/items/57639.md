## Summary

Keep this open: current main still has active/concurrent/depth subagent controls, but no durable lifetime cap for sequential subagent spawns within one requester session.

## What This Changes

Add a `subagents.maxPerSession` or equivalent setting to cap the total number of subagents spawned over a session lifetime.

## Best Possible Solution

Add an optional, documented lifetime spawn budget that defaults to current behavior when unset and is enforced consistently for native and ACP `sessions_spawn` requests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: once prior children are no longer active, the current guard only sees `countActiveRunsForSession`, so another sequential spawn can proceed even after many completed children.

## Solution Assessment

Yes, with naming/default review. A positive optional `maxPerSession`-style config enforced at spawn time is the narrowest maintainable fix, provided unset preserves existing behavior.
