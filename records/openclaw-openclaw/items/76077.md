## Best Possible Solution

No high-confidence live reproduction was established. Source inspection confirms current main lacks the proposed `message send --timeout` flag while already having a generic gateway wrapper timeout, so the remaining SIGTERM/CLI-lifetime claim still needs real behavior proof.

## Solution Assessment

No: the timeout flag is a plausible part of the fix, but process signal handling in the shared gateway helper is the wrong layer without a broader maintainer-approved signal policy. A narrower CLI-owned abort path plus strict validation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep process signals out of the shared gateway call:** `src/gateway/call.ts:709-710`
  - body: Registering `SIGTERM`/`SIGINT` handlers inside `executeGatewayRequestWithScopes` changes Node's default termination behavior for every gateway request, not only `openclaw message send`. If a non-message caller catches the rejected promise, the process can keep running after a supervisor sent SIGTERM, which is the failure mode this PR is meant to prevent; scope this to the CLI boundary with an AbortController and guaranteed exit/propagation semantics.
  - confidence: 0.86
- **[P2] Reject malformed timeout values instead of falling back:** `src/cli/program/message/helpers.ts:37`
  - body: The new timeout parser uses `parseInt` and silently returns `undefined` for invalid values, so inputs like `--timeout 10x` become `10` and `--timeout abc` falls back to the default. Existing message CLI numeric options fail fast through strict parsers, so `--timeout` should join that validation path and report a clear error before dispatch.
  - confidence: 0.93
- **[P3] Remove the release-owned changelog entry:** `CHANGELOG.md:7-9`
  - body: Normal PRs should not edit `CHANGELOG.md`; release generation owns that file in this repo. Keep the release-note context in the PR body or squash message instead and drop this changelog hunk.
  - confidence: 0.89
