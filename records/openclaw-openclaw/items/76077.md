## Summary

Keep open, but not merge-ready: the timeout idea is relevant, yet the branch conflicts with current main, duplicates newer gateway timeout/signal-abort work in the wrong layer, accepts bad timeout values silently, edits release-owned changelog content, and still lacks real behavior proof.

## What This Changes

The branch adds an `openclaw message send --timeout <ms>` option, threads `timeoutMs` into gateway message actions, installs SIGTERM/SIGINT abort handlers in the gateway call helper, adds listener-cleanup coverage, and edits CLI docs/changelog.

## Best Possible Solution

Rebase and implement a narrow CLI-owned timeout/abort path on top of current gateway request timeout/signal APIs, with strict timeout validation and real `openclaw message send` timeout/SIGTERM proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the patch defects at source level: the diff installs process-wide signal handlers in a shared gateway helper, and existing message CLI tests show malformed numeric options should reject. I did not run a live `message send` hang reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. The safer fix is to use the current gateway timeout/AbortSignal seams from the CLI boundary, validate the new flag with the shared strict parser, preserve signal exit semantics, and leave changelog generation to release tooling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep signal handling out of the gateway helper:** `src/gateway/call.ts:707-708`
  - body: `executeGatewayRequestWithScopes` is shared by gateway callers, but these `process.on` listeners remove Node's default SIGINT/SIGTERM exit behavior while any gateway call is active. A caller that catches the rejected promise can keep running after Ctrl+C or supervisor SIGTERM, so the signal handling needs to live at the CLI boundary and drive the existing AbortSignal path instead. ([nodejs.org](https://nodejs.org/api/process.html))
  - confidence: 0.86
- **[P2] Validate --timeout with the shared strict parser:** `src/cli/program/message/helpers.ts:37-38`
  - body: Using `parseInt` accepts partial values like `1abc`, and invalid or non-positive values fall back to the default timeout instead of failing. Existing message CLI numeric options reject malformed values before `messageCommand` runs, so scripts need the same clear error for a bad timeout value.
  - confidence: 0.9
