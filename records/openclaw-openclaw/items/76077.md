## Summary

Keep open: the PR targets a plausible CLI/gateway timeout problem, but the proposed shared signal handling is not safe as-is because it leaves process-level SIGINT/SIGTERM listeners installed after successful gateway requests.

## What This Changes

The PR adds a `--timeout <ms>` option to `openclaw message send`, forwards it into the message gateway call, and installs SIGTERM/SIGINT handling around gateway requests.

## Best Possible Solution

Keep a scoped `message send` timeout option if maintainers want the user-facing knob, but implement signal cancellation with disposable per-request handlers and matching CLI docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes for the introduced regression: a local Node signal check demonstrates that a registered SIGINT listener swallows default termination, and the PR leaves such listeners behind after settled gateway requests. The original linked production CPU hang remains unverified against a live gateway in this read-only review.

## Solution Assessment

No. The timeout flag forwarding is directionally reasonable, but installing process-level listeners in a shared gateway helper without cleanup is not the narrowest maintainable fix; the handler must be disposed on every settle path and timeout parsing should use the existing CLI timeout helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove signal listeners when the request settles:** `src/gateway/call.ts:707-708`
  - body: These new process-level SIGTERM/SIGINT listeners are never removed after a normal gateway response, timeout, or close. In any process that performs multiple gateway calls, stale `once` listeners accumulate and later Ctrl-C/SIGTERM can be swallowed by no-op settled handlers instead of terminating the process. Please dispose both listeners in the request settlement path, after client cleanup, so they only exist while this request is active.
  - confidence: 0.89
- **[P3] Add the required docs and changelog entry:** `src/cli/program/message/register.send.ts:38-40`
  - body: This adds a new user-facing `openclaw message send --timeout` flag, but the message CLI docs and changelog are not updated. OpenClaw policy requires user-facing fix/feat/perf changes to document the changed surface, so please add the send option to `docs/cli/message.md` and a single-line `CHANGELOG.md` entry.
  - confidence: 0.82
