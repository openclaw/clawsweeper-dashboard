## Summary

Keep open: the underlying `message send` timeout/SIGTERM problem still has a useful fix target, but this PR is not merge-ready because it puts process signal handling in the shared gateway RPC helper, silently coerces malformed timeout values, edits the release-owned changelog, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds `--timeout <ms>` to `openclaw message send`, threads that value into gateway-backed message actions, adds SIGTERM/SIGINT handling in the gateway call helper, and updates tests/docs/changelog.

## Best Possible Solution

Land a refreshed command-layer fix that adds a strict `message send --timeout <ms>` flag, passes timeout and AbortSignal through the existing message/gateway path, leaves shared `callGateway` signal semantics unchanged, removes the changelog edit, and includes focused tests plus redacted real CLI proof.

Yes at source level: current main does not expose `--timeout` on `message send`, and `messageCommand` does not wire command-level signal handling into the gateway request. I did not run a live hanging Telegram/Windows process reproduction in this read-only review.

## Solution Assessment

No. The timeout flag is a plausible fix surface, but the proposed process-wide SIGTERM/SIGINT handling belongs at the `message send` command layer and should use the existing `callGateway` AbortSignal path instead of changing shared RPC behavior.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep signal handling out of the shared gateway helper:** `src/gateway/call.ts:710-711`
  - body: Installing SIGTERM/SIGINT listeners inside `executeGatewayRequestWithScopes` changes signal behavior for every `callGateway` caller. In Node, adding a SIGTERM listener prevents the default termination path, so callers that catch the rejected request can keep the process alive; current main already has a command-layer signal bridge pattern that passes an AbortSignal into `callGateway`, which is the safer layer for `message send`.
  - confidence: 0.88
- **[P2] Reject malformed timeout values instead of coercing them:** `src/cli/program/message/helpers.ts:37-38`
  - body: `parseInt` accepts strings such as `5s` as `5` and this helper silently falls back to the default for `abc`, `0`, or negative values. For a safety timeout flag, malformed input should fail like the existing strict numeric message options so operators do not unknowingly run with a near-zero or default timeout.
  - confidence: 0.84
