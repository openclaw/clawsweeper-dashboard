## Summary

Keep open: the change is directionally useful for the linked cron hang report, but the current patch is not mergeable because it leaves process signal listeners behind, accepts invalid timeout values, and omits required user-facing docs/changelog updates.

## What This Changes

The PR adds a `--timeout <ms>` option to `openclaw message send`, forwards the timeout into message gateway calls, and adds SIGTERM/SIGINT handling around gateway requests.

## Best Possible Solution

Keep the scoped message-send timeout flag if maintainers want that CLI surface, but implement cancellation with disposable per-request signal handlers, strict positive-integer timeout parsing, focused regression coverage, and matching docs/changelog.

## Reproduction Assessment

Yes for the PR-introduced regressions by static inspection: the diff adds process-level signal listeners without cleanup and parses timeout values permissively. The original production hang remains unclear because it was not live-reproduced here and current main already has a finite transport timeout on this path.

## Solution Assessment

No. The timeout forwarding is a reasonable direction, but installing process-level listeners in a shared helper without disposal and accepting invalid timeout input is not the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove signal listeners when the request settles:** `src/gateway/call.ts:707-708`
  - body: These new `SIGTERM`/`SIGINT` listeners are registered for every gateway request but are never removed after success, timeout, or rejection. In a long-lived process or any path that performs multiple gateway calls, stale listeners can accumulate and later signals can be swallowed by already-settled handlers instead of behaving normally.
  - confidence: 0.9
- **[P2] Validate the timeout flag before forwarding it:** `src/cli/program/message/helpers.ts:27`
  - body: Parsing the CLI value with `parseInt` accepts partial strings and turns invalid input into `NaN`; because `messageCommand` only checks `typeof number`, invalid values reach the gateway layer and silently fall back to the old default instead of producing a clear CLI error. Please use a strict positive-integer parse for milliseconds and cover invalid, zero, and negative values.
  - confidence: 0.86
- **[P3] Add the required docs and changelog entry:** `src/cli/program/message/register.send.ts:38-40`
  - body: This adds a new user-facing `openclaw message send --timeout` flag, but the message CLI docs and active changelog are not updated. OpenClaw policy requires user-facing fix/feature changes to document the changed surface.
  - confidence: 0.82
