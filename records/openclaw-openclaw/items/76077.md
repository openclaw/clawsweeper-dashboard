## Summary

Keep open: the reported CLI hang/termination problem is real, but this patch is not merge-ready because it adds process-wide signal handlers in shared gateway RPC, changes the effective message-action timeout default, silently coerces malformed timeout input, and still lacks after-fix real behavior proof.

## What This Changes

This PR adds `--timeout <ms>` to `openclaw message send`, forwards it into gateway-backed message actions, adds SIGTERM/SIGINT listeners in shared gateway RPC, and updates tests, docs, and changelog text.

## Best Possible Solution

Land a refreshed command-layer fix that adds strict `message send --timeout <ms>` parsing, feeds timeout and AbortSignal through the existing gateway path, leaves shared `callGateway` signal semantics unchanged, removes release-owned changelog edits, and includes focused tests plus redacted real CLI proof.

No. The timeout flag is a plausible repair surface, but signal handling should be command-owned and passed through the existing AbortSignal seam rather than installed in shared gateway RPC; malformed timeout input should fail strictly.
