## Summary

Keep this PR open: current main and the latest release still do not expose the requested `openclaw message send --timeout` behavior, but the submitted branch is conflicting, lacks real behavior proof, and introduces repairable correctness issues in timeout parsing and shared process-signal handling.

## What This Changes

The PR adds a public `openclaw message send --timeout <ms>` flag, forwards that value into message gateway dispatch, installs SIGTERM/SIGINT handling in the shared gateway call helper, and updates tests, docs, and `CHANGELOG.md`.

## Best Possible Solution

Preserve the useful timeout mitigation, but rework it onto current main with strict positive-integer validation, scoped signal/abort handling that does not change unrelated gateway callers, no normal-PR changelog edit, and redacted real terminal or log proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate timeout strictly instead of falling back:** `src/cli/program/message/helpers.ts:37-40`
  - body: `parseInt` accepts partial strings like `5abc` and decimals, while `abc`, `0`, or negative values become `undefined` and then silently use the 30s default in `messageCommand`. This makes invalid user input look accepted; use the existing strict positive-integer parser/validation path and surface an error for malformed `--timeout` values.
  - confidence: 0.91
- **[P1] Keep process signals out of shared gateway calls:** `src/gateway/call.ts:707-708`
  - body: Adding `process.on("SIGTERM"/"SIGINT")` inside `executeGatewayRequestWithScopes` changes signal handling for every gateway caller while a request is active, not only `openclaw message send`. In Node, installing a SIGINT listener suppresses the default Ctrl+C termination path, so callers that catch the rejected gateway promise can keep running after the user or supervisor tried to terminate them.
  - confidence: 0.83
- **[P3] Remove the normal-PR changelog edit:** `CHANGELOG.md:8`
  - body: `CHANGELOG.md` is release-owned in this repo, so this PR should keep the release-note context in the PR body or squash message instead of editing the unreleased changelog section directly.
  - confidence: 0.95
