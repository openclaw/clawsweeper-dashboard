## Summary

Keep open: this PR targets a valid `message send` reliability gap and current main does not already expose a user-facing timeout flag, but the patch is not mergeable as-is because it leaves process signal listeners behind, accepts invalid timeout values, and omits required docs/changelog coverage.

## What This Changes

The PR adds `openclaw message send --timeout`, forwards `timeoutMs` into the message gateway request, and adds SIGTERM/SIGINT handling around gateway calls.

## Best Possible Solution

Keep a scoped `message send` timeout, but implement it with validated positive millisecond input, disposable cancellation/signal handling, focused regression tests, and matching docs/changelog text.

## Reproduction Assessment

Yes for the PR-introduced signal regression: the diff leaves process signal listeners registered after request settlement, and a local Node check shows such listeners suppress default SIGINT termination. The production hang in #75895 is documented in the linked issue but was not live-reproduced in this read-only review.

## Solution Assessment

No. The timeout flag is directionally useful, but the shared gateway signal handler must be scoped and removed on every settle path, and timeout parsing should use the existing positive-timeout helpers rather than raw `parseInt`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove signal listeners when the request settles:** `src/gateway/call.ts:707-708`
  - body: These process-level SIGTERM/SIGINT listeners are never removed after a normal response, timeout, close, or startup failure. A later Ctrl-C/SIGTERM in the same process can be consumed by stale settled handlers instead of terminating, so the listeners need to be disposed on every request settlement path.
  - confidence: 0.9
- **[P2] Reject invalid timeout flag values:** `src/cli/program/message/helpers.ts:27`
  - body: The new flag uses `parseInt`, which accepts partial values and lets `NaN`, `0`, or negative timeouts flow into later fallback/clamping behavior. This makes `--timeout nope` or `--timeout 0` silently do the wrong thing; parse with the existing positive timeout helper and fail clearly on invalid input.
  - confidence: 0.86
- **[P3] Document the timeout flag and changelog entry:** `src/cli/program/message/register.send.ts:38-40`
  - body: This adds a user-facing `openclaw message send --timeout` option, but the message CLI docs and active changelog are not updated. Please add the option to `docs/cli/message.md` and add a single-line `CHANGELOG.md` fix entry.
  - confidence: 0.82
