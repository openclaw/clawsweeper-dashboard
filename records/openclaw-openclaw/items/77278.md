## Summary

Keep open: this is a valid new TUI/security capability request, and current main still has exec approval events plus process stdin primitives but no TUI-owned masked sudo prompt or private password injection path.

## What This Changes

Request a masked TUI sudo password prompt after exec approval, with the password piped directly to sudo without entering model-visible channels.

## Best Possible Solution

Design a TUI-owned exec approval and sudo-auth flow with masked local input, an audited host-side credential injection path, redaction invariants, focused regression tests, and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection against current main gives high confidence that the requested TUI sudo credential flow does not exist.

## Solution Assessment

Unclear as proposed because the UX direction is useful, but the safe implementation needs maintainer product/security design. The maintainable path is a native TUI approval/password surface plus a redacted credential channel, not reuse of model-visible process stdin.
