## Summary

Keep open: this is a valid new core TUI capability request, and current main still has exec approvals plus PTY/stdin primitives but no TUI-owned masked sudo password prompt or private credential path. The request is security-sensitive, so it needs maintainer product/security design rather than conservative cleanup closure.

## What This Changes

The issue requests a masked, TUI-local sudo password prompt after exec approval so sudo credentials can be passed privately to the host process without entering model-visible context.

## Best Possible Solution

Design a native TUI approval and sudo-auth flow that consumes exec approval broadcasts, prompts locally with masked input, and passes credentials only through an audited host-side execution path with redaction tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow because the TUI does not handle exec approval broadcasts and the approval/runtime schemas do not expose a private password channel.

## Solution Assessment

Unclear as proposed: the user-facing direction is reasonable, but the exact implementation needs maintainer UX and security design. The narrowest maintainable path is native TUI approval handling plus a credential-redacted sudo path with explicit invariants.
