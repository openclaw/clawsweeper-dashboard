## Summary

Keep open: this is a valid security-sensitive TUI feature request, and current main still lacks a TUI-owned exec approval surface plus a private sudo password path. The existing ClawSweeper comment's conclusion still matches the source review.

## What This Changes

Request to add a masked inline sudo password prompt in `openclaw-tui` after exec approval, piping the credential to sudo without exposing it to the model.

## Best Possible Solution

Design a native TUI approval and sudo-auth flow that consumes Gateway approval events, prompts locally with masked input, and passes credentials only through an audited host-side execution path with regression tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow.

## Solution Assessment

Unclear as proposed because the direction is useful, but the safe implementation needs maintainer product and security design. The maintainable path is to add native TUI approval handling and a password-redacted sudo credential boundary, not reuse model-visible process stdin.
