## Summary

Keep open: current main and the latest release still provide exec approval decisions and generic TUI tool rendering, but no masked TUI sudo prompt or non-model secret-to-stdin handoff. The remaining work is a security-sensitive product design, not cleanup-close material.

## What This Changes

Request a TUI-only interactive masked sudo password prompt after exec approval, with the password sent directly to sudo stdin and excluded from model context.

## Best Possible Solution

Define and implement a TUI-owned masked secret prompt, triggered only after an approved sudo exec, backed by a protocol/runtime secret channel that bypasses model-visible args/results and durable approval state.

## Reproduction Assessment

Not applicable as a feature request. I did not run a live sudo command, but source and docs give high confidence that current main lacks the requested masked sudo prompt and secure password-to-stdin handoff.

## Solution Assessment

Unclear until maintainers choose the security contract. A Cursor-like second prompt is directionally appropriate for terminal UX, but the best implementation must be a first-class secret handoff rather than putting a password into existing approval resolve or model-facing process APIs.
