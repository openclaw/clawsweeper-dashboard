## Summary

Keep open: current main has exec approvals plus PTY/stdin primitives, but not the requested TUI-owned masked sudo password prompt or private credential path. This is a useful new capability request with security and UX design work, not a cleanup-close candidate.

## What This Changes

Request an inline masked sudo password prompt in the TUI exec approval flow that sends the password directly to sudo without exposing it to the model.

## Best Possible Solution

Design a TUI-owned exec approval and sudo-auth flow with masked local input, a narrow host-side credential injection boundary, redaction invariants, regression tests, and user-facing docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow: approval events are not handled by the TUI, and existing stdin paths are normal exec/process primitives rather than a private password channel.

## Solution Assessment

Unclear as proposed because the UX goal is sound, but the safe solution needs maintainer product/security design. The narrow maintainable path is TUI-native approval handling plus a password-redacted sudo credential path, not reuse of model-visible process stdin.
