## Summary

Keep open: current main and the latest release still lack TUI approval-event handling, masked sudo password capture, and a private redacted credential-injection path. This is a valid new TUI/security capability request, not cleanup closure.

## What This Changes

Request an inline masked sudo password prompt in the TUI exec approval flow, with the password sent directly to sudo and hidden from the model.

## Best Possible Solution

Design a native TUI approval and sudo-auth flow with a private credential channel, explicit redaction/audit invariants, PTY/sudoers behavior choices, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection against current main and v2026.5.12 gives high confidence that the requested TUI sudo credential flow is absent.

## Solution Assessment

Unclear as proposed because the UX direction is plausible but the safe implementation needs maintainer product/security design. The maintainable path is a TUI-owned approval/password surface plus a redacted credential channel, not model-visible process stdin.
