## Summary

Keep open: current main has gateway exec approval events and generic PTY/stdin plumbing, but no TUI-owned masked sudo password prompt or private redacted credential-injection path. This is a security-sensitive new capability, not conservative cleanup closure.

## What This Changes

Request adding a masked TUI sudo password prompt after exec approval so sudo credentials can be sent privately to the host process.

## Best Possible Solution

Design a native TUI approval and sudo-auth flow with masked local input, a redacted host-side credential injection path, clear retry/cancel semantics, regression tests, and user-facing docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection against current main gives high confidence that the requested TUI sudo credential flow does not exist.

## Solution Assessment

Unclear as proposed. The UX direction is useful, but the maintainable solution needs maintainer product/security design around a TUI-owned local prompt and redacted credential channel rather than normal model-visible process stdin.
