## Summary

Keep open: current main still lacks TUI-owned exec approval handling and a private masked sudo credential path, so this is a new security-sensitive TUI capability rather than an implemented or stale cleanup item.

## What This Changes

Request an inline masked sudo password prompt in the TUI exec approval flow so approved sudo commands can receive a password without exposing it to the model.

## Best Possible Solution

Design a TUI-owned approval client plus masked sudo credential handoff, with a private host-side stdin path and tests/docs proving password redaction, retry, cancellation, and current approval semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence current main lacks the requested TUI sudo credential flow because the TUI does not consume exec approval broadcasts and the exec/process schemas expose no private password channel.

## Solution Assessment

Unclear as proposed because the UX direction is reasonable, but the safe implementation needs maintainer product/security design. The narrowest maintainable path is native TUI approval handling first, then a tightly audited sudo credential path with redaction invariants.
