## Summary

Keep open. Current main and v2026.6.6 do not implement an inline masked sudo password prompt for TUI exec approvals; related approval issues cover pause/recovery UI pieces but not this exact sudo password handoff. The request is plausible but security-sensitive, so it needs maintainer product/security direction before an implementation PR.

## What This Changes

Request TUI support for an inline masked sudo password prompt after exec approval, with the password sent only to sudo stdin and never to the model.

## Best Possible Solution

Define a TUI-only sudo authentication contract that keeps the password out of model context, durable Gateway state, logs, and approval records, then implement it through the existing exec approval boundary with targeted protocol, TUI, and exec runtime tests.

## Reproduction Assessment

Not applicable as a feature request. Source review gives high confidence that current main lacks the requested masked sudo prompt and password-to-stdin handoff.

## Solution Assessment

Unclear until maintainers choose the security contract. The TUI-only direction is plausible, but the implementation must avoid turning approval resolve payloads, Gateway state, or model-visible tool results into a password transport.
