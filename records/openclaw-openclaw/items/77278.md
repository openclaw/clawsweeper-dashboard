## Summary

Keep open: this is a valid new TUI capability request, not a cleanup-close candidate. Current main has exec approvals plus PTY/stdin primitives, but no TUI-owned sudo password prompt or private credential path that keeps the sudo password out of model-visible tool channels.

## What This Changes

Request adding an inline masked sudo password prompt to the TUI exec approval flow after a sudo command is approved.

## Best Possible Solution

Design a TUI-owned approval and sudo-auth flow with masked local input, a narrowly audited host-side credential injection path, explicit redaction invariants, regression tests, and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow because the TUI does not consume exec approval broadcasts and the exec/process schemas expose only normal tool-visible stdin paths.

## Solution Assessment

Unclear as proposed because the user direction is useful, but the safe implementation needs maintainer product/security design. The narrowest maintainable path is native TUI approval handling plus a password-redacted sudo credential path, not reuse of the existing model-visible process write/paste contract.
