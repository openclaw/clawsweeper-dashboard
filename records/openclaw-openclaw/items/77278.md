## Summary

Keep open. Current main and the latest release still lack a TUI masked sudo-password prompt or secure password-to-stdin handoff, and this remains a security-sensitive product decision rather than a cleanup-close candidate.

## What This Changes

The issue requests a masked interactive sudo-password prompt in the TUI exec approval flow so sudo commands can receive credentials through stdin without exposing them to the model.

## Best Possible Solution

Define a TUI-only sudo-auth contract that keeps the password out of model context, durable Gateway state, logs, transcripts, and approval records, then implement it through the existing exec approval boundary with focused protocol, TUI, exec-runtime, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence current main lacks the requested masked sudo prompt and password-to-stdin handoff.

## Solution Assessment

Unclear until maintainers choose the security contract. The TUI-only direction is plausible, but the password must bypass approval resolve payloads, durable state, logs, transcripts, and model-visible tool results.
