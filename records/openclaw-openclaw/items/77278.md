## Summary

Keep open: this is a valid new, security-sensitive TUI capability request, not a cleanup-close candidate. Current main has exec approvals, `/elevated`, PTY/stdin primitives, and approval broadcasts, but it does not have a TUI-owned sudo password prompt or a private credential path that bypasses model-visible tool contracts.

## What This Changes

The issue requests a TUI-only interactive sudo password prompt after exec approval, with masked input and a model-hidden credential path.

## Best Possible Solution

Design a TUI-owned approval and sudo-auth flow that consumes approval broadcasts, prompts locally with masked input, and passes credentials only through a narrowly audited host-side execution path with tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow: the TUI does not handle exec approval broadcasts and the exec/runtime schemas do not provide a private password channel.

## Solution Assessment

Unclear as proposed because the UX goal is sound, but the safe implementation needs maintainer product/security design. The narrowest maintainable path is native TUI approval handling first, then a password-redacted sudo credential path with explicit invariants.
