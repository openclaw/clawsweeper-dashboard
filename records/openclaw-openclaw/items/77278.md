## Summary

Keep open: this is a new security-sensitive TUI capability request, and current main has approval plus PTY/stdin primitives but no TUI-owned sudo password prompt or private credential handoff.

## What This Changes

The issue requests an inline masked sudo password prompt in the TUI exec approval flow so approved sudo commands can receive a password without exposing it to the model.

## Best Possible Solution

Design a native TUI approval client plus a narrowly audited sudo-auth handoff that masks input locally and injects credentials only through a password-redacted host execution path.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence current main lacks the requested flow because the TUI ignores exec approval events and the approval/runtime schemas do not expose a private sudo password channel.

## Solution Assessment

Unclear as proposed because the direction is useful but security-sensitive. The maintainable path is to add TUI-owned approval handling first, then a password-redacted sudo credential path with explicit logging/transcript invariants.
