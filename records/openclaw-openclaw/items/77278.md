## Summary

Keep open: this is a valid new TUI capability request, not a cleanup-close candidate. Current main has exec approvals and PTY/stdin primitives, but no TUI-owned sudo password prompt or password-safe credential path.

## What This Changes

Add an inline masked sudo password prompt to the OpenClaw TUI exec approval flow so approved sudo commands can receive credentials without exposing them to the model.

## Best Possible Solution

Design a TUI-owned approval and sudo-auth flow that consumes exec approval broadcasts, prompts locally with masked input, and passes credentials only through a narrowly audited host-side execution path with regression tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow: the TUI does not handle exec approval broadcasts and the exec/runtime schemas do not provide a private password channel.

## Solution Assessment

Unclear as proposed because the direction is useful, but the safe implementation needs maintainer product/security design. The narrowest maintainable path is to first add native TUI approval handling, then add a password-redacted sudo credential path with explicit invariants.
