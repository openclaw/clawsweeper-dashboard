## Summary

Keep open: this is a valid new TUI capability request, not a cleanup-close candidate. Current main has exec approval broadcasts and PTY/stdin primitives, but no TUI-owned sudo password prompt or private credential path.

## What This Changes

The issue requests a masked sudo password prompt in the TUI exec approval flow, with the password piped to sudo without entering model-visible context.

## Best Possible Solution

Design a TUI-owned approval and sudo-auth flow with a narrowly audited redacted credential channel into host-side exec, plus regression tests proving the password never enters model-visible or persisted data.

## Reproduction Assessment

Not applicable: this is a feature request rather than a current contract bug. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow.

## Solution Assessment

Unclear as proposed because the direction is useful but the safe implementation needs maintainer UX and security design. The narrow maintainable path is native TUI approval handling first, then a redacted sudo credential injection path with explicit invariants.

## Risks / Open Questions

- Sudo password handling must keep the password out of model context, tool arguments/results, approval records, logs, transcripts, process polling output, crash messages, and durable state.
- Sudo behavior varies across PTY prompts, sudo -S, cached credentials, askpass, retry limits, and sudoers policy, so maintainers need to define the supported path.
- The TUI currently appears able to connect as an operator client while not rendering exec approval broadcasts, so native approval UX should be designed before adding a second sudo-auth layer.
