## Summary

Keep open: this is a valid new TUI capability request, not a cleanup-close candidate. Current main has exec approvals plus PTY/stdin primitives, but it does not have TUI-owned exec approval handling or a private sudo credential path.

## What This Changes

Request a masked, TUI-local sudo password prompt after exec approval so sudo credentials can be passed directly to the process without entering model-visible context.

## Best Possible Solution

Design a TUI-owned approval and sudo-auth flow that consumes exec approval broadcasts, prompts locally with masked input, and passes credentials only through a narrowly audited host-side execution path with regression tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested TUI sudo credential flow: TUI event handling does not consume exec approval broadcasts, and the runtime/tool schemas do not expose a private password channel.

## Solution Assessment

Unclear as proposed because the direction is useful but security-sensitive. The narrow maintainable path is native TUI approval handling first, then a password-redacted sudo credential path with explicit non-leakage invariants.
