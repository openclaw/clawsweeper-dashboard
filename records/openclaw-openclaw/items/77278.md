## Summary

Keep open. Current main and v2026.6.6 still provide exec approval through `/approve` plus generic process stdin, not a TUI-side masked sudo password handoff; the request fits terminal UX work but needs maintainer security/product design before implementation.

## What This Changes

Add a TUI-only masked sudo password prompt after exec approval so approved sudo commands can receive the password without exposing it to the model.

## Best Possible Solution

Design a TUI-only secret-input flow bound to one approved exec process, with masked entry, retry/cancel/timeout handling, and proof that the password never reaches transcripts, model context, logs, approval payloads, or channel surfaces.

## Reproduction Assessment

No runtime sudo repro was executed. Source inspection gives medium-confidence reproduction of the missing behavior: current TUI approval paths show `/approve` and generic tool/process stdin only, with no masked sudo password prompt or TUI-only secret handoff.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix should not be generic command-string sudo sniffing; it needs an explicit TUI-only, approval-bound credential flow with security proof.
