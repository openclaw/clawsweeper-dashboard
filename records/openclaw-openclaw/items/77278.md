## Summary

Keep open: current main has exec approval plumbing and TUI chat/tool event rendering, but no TUI-owned masked sudo credential prompt or secret handoff path after approval.

## What This Changes

Requested a TUI-only masked sudo password prompt after exec approval so sudo commands can receive credentials through stdin without exposing them to the model.

## Best Possible Solution

Define a TUI-owned masked credential flow and explicit secret handoff contract, then wire sudo execution so the password bypasses transcript/model paths and is covered by focused protocol, exec, TUI, and redaction tests.

## Reproduction Assessment

Not applicable: this is a new TUI capability request rather than a broken documented behavior. Source inspection shows current main has approval text and decision routing, but no masked sudo password collection path.

## Solution Assessment

Unclear as an implementation plan: the requested direction is reasonable, but the safest fix needs maintainer decisions on the credential protocol, supported surfaces, PTY versus sudo -S behavior, retries, cancellation, and redaction boundaries.
