## Summary

Keep open: current main has exec approval broadcasts and normal PTY/stdin process plumbing, but no TUI-owned masked sudo password prompt or private credential-injection path. This is a security-sensitive feature request that needs maintainer product/security design before implementation.

## What This Changes

Request an interactive masked sudo password prompt in the TUI exec approval flow, with the password piped privately to sudo after command approval.

## Best Possible Solution

Design a native TUI approval and sudo-auth flow with masked local input, a redacted host-side credential injection contract, clear retry/cancel semantics, regression coverage, and user-facing docs.

## Reproduction Assessment

Not applicable as a feature request. Current main source inspection gives high confidence that the requested TUI sudo credential flow is absent, rather than a broken existing contract.

## Solution Assessment

Unclear as proposed. The UX direction is plausible, but the maintainable solution needs explicit product/security design around TUI-owned masked input and a redacted credential channel instead of normal process stdin.
