## Summary

Keep open: the latest head still has the merge blockers from the prior ClawSweeper review. The performance direction is useful, but the patch currently weakens the plugin package hardlink boundary and can add discovery work to runtime paths current main can skip.

## What This Changes

The PR threads optional shared plugin discovery through plugin registry/loading helpers, adds a scan-scoped package.json manifest cache during discovery, and exposes the parsed package manifest on PluginCandidate.

## Best Possible Solution

Keep the perf direction, but make manifest caching trust-aware, create shared discovery lazily, thread it into channel manifest loading, and add focused cache/runtime regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection of the latest head shows the cache key omits trust mode, ensurePluginRegistryLoaded eagerly creates shared discovery before active-registry returns, and channel manifest loading does not consume the threaded discovery value.

## Solution Assessment

No. The cache and discovery-sharing direction is maintainable, but this patch should first separate cache entries by trust policy and make runtime sharing lazy and complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the package-manifest cache trust-aware:** `src/plugins/discovery.ts:510-519`
  - body: This cache is keyed only by the package directory real path, but the read below depends on whether the candidate is bundled-trusted or must go through readPackageManifest with hardlink rejection. In a shared scan, the same physical package reached later through an external origin can reuse the trusted parse and skip the hardlink policy that current main applies to non-bundled package.json reads. Include the trust mode/rejectHardlinks in the key, or keep trusted and external manifest reads in separate caches.
  - confidence: 0.84
- **[P2] Keep the runtime discovery scan lazy:** `src/plugins/runtime/runtime-registry-loader.ts:142-145`
  - body: This new call runs before the active-registry checks below can return. Current main can satisfy already-loaded or scoped calls without discovering plugins again, but this head always performs a synchronous discovery first; move shared discovery behind a lazy helper and call it only on paths that actually need to pass discovery onward.
  - confidence: 0.91
- **[P2] Thread discovery into channel manifest loading:** `src/plugins/channel-presence-policy.ts:454`
  - body: The new option is accepted here but never reaches loadInstalledChannelManifestRecords, so configured-channel and onlyChannelIds flows still load channel manifest records through the old registry path. When ensurePluginRegistryLoaded passes a shared discovery into these resolvers, those paths can still pay the old scan plus the new eager scan.
  - confidence: 0.86
