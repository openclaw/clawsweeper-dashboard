## Summary

Keep open: this maintainer-labeled PR is a useful performance direction, but it is not merge-ready because the branch can add discovery work to runtime fast paths, leaves one channel-scope path unthreaded, and the new package-manifest cache should not mix trusted bundled reads with hardlink-rejecting external reads.

## What This Changes

The branch threads optional shared plugin discovery through plugin registry/loading helpers, adds a scan-scoped package.json manifest cache in discovery, and adds discovery-threading tests.

## Best Possible Solution

Keep the perf direction, but make manifest caching trust-aware, make shared runtime discovery lazy, thread it into channel manifest loading, and add focused runtime/cache regression coverage plus startup read-count proof before merge.

## Reproduction Assessment

Yes, from source inspection. PR head eagerly calls discoverOpenClawPlugins before existing active-registry return paths, leaves the channel manifest-record lookup unthreaded, and keys the new cache without the trust policy used by the hardlink boundary.

## Solution Assessment

No. The cache and discovery-sharing direction is maintainable, but this branch should first separate cache entries by trust policy and make runtime sharing lazy and complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the package-manifest cache trust-aware:** `src/plugins/discovery.ts:509-517`
  - body: This cache is keyed only by the package directory real path, but the read below depends on whether the candidate is bundled-trusted or must go through readPackageManifest with hardlink rejection. In a shared scan, bundled roots are visited before installed/global roots, so the same physical package reached through a later external origin can reuse the trusted parse and skip the hardlink policy that current main applies to non-bundled package.json reads. Include the trust mode/rejectHardlinks in the key, or keep trusted and external manifest reads in separate caches.
  - confidence: 0.84
- **[P2] Keep the runtime discovery scan lazy:** `src/plugins/runtime/runtime-registry-loader.ts:142-145`
  - body: This new call runs before the active-registry checks below can return. Current main can satisfy already-loaded or scoped calls without discovering plugins again, but this head always performs a synchronous discovery first; move shared discovery behind a lazy helper and call it only on paths that actually need to pass discovery onward.
  - confidence: 0.91
- **[P2] Thread discovery into channel manifest loading:** `src/plugins/channel-presence-policy.ts:454`
  - body: The new option is accepted here but never reaches loadInstalledChannelManifestRecords, so configured-channel and onlyChannelIds flows still load channel manifest records through the old registry path. When ensurePluginRegistryLoaded passes a shared discovery into these resolvers, those paths can still pay the old scan plus the new eager scan.
  - confidence: 0.86
