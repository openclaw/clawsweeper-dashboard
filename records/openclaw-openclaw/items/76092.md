## Summary

Keep this PR open: it is a useful gateway-runtime drift mitigation, but the current patch is still too narrow because documented local-prefix installs put managed Node under the install prefix while the PR only searches the gateway state directory. The recent discussion also correctly notes that this should not close https://github.com/openclaw/openclaw/issues/69799, which remains the broader signed-runtime/TCC boundary issue.

## What This Changes

The PR changes gateway service Node selection to prefer a validated bundled state-dir Node before falling back to the current or system Node, adds daemon runtime-path tests, adjusts a cron lint guard, and adds a changelog line.

## Best Possible Solution

Update bundled Node discovery to search the documented local-prefix managed runtime as well as the gateway state directory, keep current/system Node as fallback, and leave the broader signed-runtime/TCC issue tracked separately.

## Reproduction Assessment

Yes. Source and docs give a high-confidence reproduction path: install with a custom local prefix where managed Node lives under <prefix>/tools while the gateway state dir differs, then the PR's state-dir-only lookup cannot find that bundled runtime.

## Solution Assessment

No. The state-dir preference is a good mitigation for the default layout, but the best fix also includes the installer-prefix contract before falling back to current/system Node.
