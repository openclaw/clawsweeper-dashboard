## Summary

Keep open: the state-dir bundled Node preference is useful and has after-fix terminal proof, but the patch still misses documented local-prefix installs where the managed Node lives under the install prefix rather than the gateway state directory. The linked discussion also correctly says this should not be treated as closing the broader macOS TCC security-boundary work at https://github.com/openclaw/openclaw/issues/69799.

## What This Changes

The PR changes gateway service Node resolution to prefer supported bundled Node executables under the state directory, adds runtime-path tests, adjusts a cron delivery guard, and adds a changelog note.

## Best Possible Solution

Extend bundled Node discovery to include the active local-prefix tools root as well as the gateway state dir, keep the current fallback behavior, and leave the broader signed macOS TCC identity work tracked at https://github.com/openclaw/openclaw/issues/69799.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: use a local-prefix install where managed Node lives under <prefix>/tools while the gateway state directory is different, and the PR's state-dir-only lookup cannot find that bundled runtime.

## Solution Assessment

No, not yet. Preferring a bundled managed Node is the right mitigation, but the best fix must cover the installer prefix contract as well as the default state-dir layout before falling back to current/system Node.
