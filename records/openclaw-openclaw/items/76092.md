## Summary

Keep open: the branch addresses the default state-dir bundled Node path and includes sufficient macOS terminal proof, but the latest head still has a state-dir-only lookup that misses the documented custom-prefix installer layout.

## What This Changes

The PR adds state-dir bundled Node discovery ahead of current/system Node fallback for gateway service installs, regression tests, a changelog entry, and a small cron lint adjustment.

## Best Possible Solution

Land after bundled Node resolution covers both the state-dir and active local installer-prefix layouts, with regression coverage for custom --prefix installs plus the existing macOS and Windows cases.

## Reproduction Assessment

Yes for the source path: current main selects current/system Node without a bundled tools scan, and the related report plus PR proof show the macOS LaunchAgent drift scenario. The remaining custom-prefix gap is also source-reproducible from the documented <prefix>/tools layout and the PR's state-dir-only lookup.

## Solution Assessment

No. Preferring a validated bundled Node is the right direction, but the current implementation should also consider the active installer prefix before preserving current/system fallbacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include the installer prefix in bundled Node lookup:** `src/daemon/runtime-paths.ts:129`
  - body: This lookup only scans `resolveGatewayStateDir(...)/tools`, but the local-prefix installer documents and creates bundled Node under `<prefix>/tools/node-v...`. With a custom prefix such as `/opt/openclaw` and the default `~/.openclaw` state dir, forced gateway installs can still miss the prefix bundle and record current/system Node, leaving that documented install path exposed to the same runtime drift. Add prefix-aware lookup and a regression test for a prefix that differs from the state dir.
  - confidence: 0.88
