## Summary

Keep open. The PR targets a real TUI visibility symptom, and the prior Greptile note correctly identified a conflicting shared test, but current main shows the broader problem: the patch changes a shared visibility helper used by Gateway preview/history and Control UI paths that still intentionally suppress commentary-only assistant text.

## What This Changes

The PR branch adds a commentary-phase fallback to `extractAssistantVisibleText` so commentary-only assistant messages render when no `final_answer` text is present.

## Best Possible Solution

Keep the shared helper history/preview-safe, then fix the terminal symptom in a TUI-scoped live-stream projection path or normalize completed no-final-tag provider output into `final_answer` before shared display helpers see it.

## Reproduction Assessment

Yes. A source-level reproduction is a phased assistant message with only commentary text: current main returns no shared visible text, the TUI path therefore has nothing to render, and this PR would make that same commentary visible through every shared-helper caller.

## Solution Assessment

No. The PR solves the observable TUI symptom by changing a shared helper used by Gateway preview/history and Control UI extraction; a caller-scoped TUI fallback or provider normalization is the narrower maintainable solution.
