## Summary

Keep open: the PR targets a plausible TUI no-final/commentary visibility bug, but this branch should not merge as-is because it broadens the shared assistant visibility helper used by Gateway previews, Control UI, and transcript readers where commentary-only text is intentionally suppressed.

## What This Changes

The branch adds a commentary-phase fallback in `src/shared/chat-message-content.ts` so `extractAssistantVisibleText` returns commentary text when no `final_answer` text exists.

## Best Possible Solution

Keep shared extraction final-answer-safe and add a TUI-scoped or caller-scoped opt-in fallback, with regression coverage proving TUI no-final visibility and non-TUI commentary suppression.

## Reproduction Assessment

Yes, from source inspection: a commentary-only phased assistant message returns no shared visible text, and the TUI stream assembler reaches that helper through `extractContentFromMessage`. I did not live-run a TUI session in this read-only review.

## Solution Assessment

No. The patch targets the right symptom, but changing the shared helper is too broad; the maintainable fix is a TUI-scoped fallback or explicit caller option that preserves non-TUI commentary suppression.
