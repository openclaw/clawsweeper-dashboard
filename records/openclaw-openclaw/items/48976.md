## Summary

Keep open. The linked Feishu bug is still source-reproducible on current main, but this PR is not mergeable as-is because it bundles unrelated security-sensitive changes, removes existing owner gates for `/config` and `/debug`, has an ineffective dependency security update, and lacks after-fix real behavior proof.

## What This Changes

The PR adds Feishu quoted-message mention normalization and also changes gateway cron cleanup, command authorization, browser control auth, health status merging, device bootstrap tokens, node-host logging, package dependencies, and the changelog.

## Best Possible Solution

Narrow or replace this with a Feishu-plugin-only patch that models fetched-message `mentions`, reuses/adapts the existing mention normalizer for quoted/root/thread content, and adds focused `extensions/feishu/src/send.test.ts` coverage without unrelated security or dependency churn.

## Reproduction Assessment

Yes, source-level. A mocked `getMessageFeishu()` response whose `body.content` contains `@_user_1` plus mention metadata would still return the raw placeholder on current main because the fetched parser ignores mentions and quoted context uses that content directly.

## Solution Assessment

No. The Feishu fix direction is right, but this PR is not the best solution because it open-codes the normalization and bundles unrelated security-sensitive changes that introduce regressions.

Full review comments:

- **[P1] Restore owner gating for `/config show`:** `src/auto-reply/reply/commands-config.ts:45`
  - body: This removes the owner check after sender authorization, so any command-allowlisted non-owner on an external channel can read raw config via `/config show`, including sensitive subpaths. Current main intentionally keeps the owner gate and has regression coverage for this case.
  - confidence: 0.95
- **[P1] Restore owner gating before `/debug` commands:** `src/auto-reply/reply/commands-config.ts:215`
  - body: `/debug set`, `unset`, and `reset` mutate process-wide runtime overrides, but the PR lets any command-allowlisted non-owner reach them after `rejectUnauthorizedCommand`. Keep the existing owner check, with only a deliberately scoped internal-channel exception if needed.
  - confidence: 0.95
- **[P1] Update the lockfile with the security dependency changes:** `package.json:371`
  - body: The PR changes root dependency and override versions in `package.json`, but `pnpm-lock.yaml` is not part of the PR file list and the PR head lock still resolves the old `file-type` and `fast-xml-parser` versions. Frozen installs will not apply the intended security updates.
  - confidence: 0.92
- **[P2] Read account runtime status from `channelAccounts`:** `src/commands/health.ts:517-518`
  - body: `channels.status` stores per-account snapshots under `channelAccounts[channelId]`, not under `channels[channelId].accounts`. As written, `gwAccount` stays undefined and the health command still misses account runtime fields.
  - confidence: 0.9
- **[P2] Match cron session keys exactly before deletion:** `src/gateway/server-methods/cron.ts:207-209`
  - body: Deleting the first persisted session key that merely includes `sessionKey` or `jobId` can remove the wrong session when identifiers overlap, for example `job-1` matching `job-10`. Cleanup needs a deterministic exact key mapping before mutating the store.
  - confidence: 0.86
