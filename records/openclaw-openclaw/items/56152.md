## Summary

Keep open: the macOS `/tmp` versus `/private/tmp` false negative is credible, but this PR currently weakens the local media read boundary by allowing lexical media paths after canonicalization succeeds, and it still lacks observable after-fix Telegram proof.

## What This Changes

The PR changes local media path validation to compare lexical and canonical path variants and adds macOS `/tmp`/`/private/tmp` regression tests.

## Best Possible Solution

Revise `assertLocalMediaAllowed()` so existing files remain contained by canonical target, macOS root aliases are tolerated without symlink escapes, workspace sibling checks stay canonical, and regression coverage includes out-of-root symlink rejection.

## Reproduction Assessment

Yes, at source level: current main can reject the PR's mocked `/tmp` and `/private/tmp` alias cases, and the PR's symlink escape follows directly from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR targets the right false negative, but the maintainable fix must preserve canonical containment for existing files while adding alias tolerance and explicit symlink-escape coverage.
