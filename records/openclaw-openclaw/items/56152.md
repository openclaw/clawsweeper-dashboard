## Summary

Keep open: this PR targets a valid macOS local-media false negative, but the proposed path-variant matching weakens the existing canonical file-read boundary and needs repair before merge.

## What This Changes

The branch changes local media root validation to compare lexical and realpath variants for media paths and roots, with Vitest coverage for macOS `/tmp` and `/private/tmp` alias cases.

## Best Possible Solution

Revise the PR so successful media-path realpath remains authoritative, tolerate `/tmp` and `/private/tmp` aliasing without widening containment, and add escape regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can reject mismatched `/tmp` and `/private/tmp` aliases, while the PR regression follows from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The requested bug fix is right, but the current PR shape is not the best solution because it broadens media-path containment instead of keeping canonical media targets authoritative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve canonical media path containment:** `src/media/local-media-access.ts:89-97`
  - body: The new cross-product check accepts the lexical `path.resolve(mediaPath)` variant even when `fs.realpath(mediaPath)` resolves outside every allowed root. A symlinked path under an allowed root can therefore pass the root check lexically and be read outside the configured boundary.
  - confidence: 0.93
- **[P2] Keep workspace sibling checks canonical:** `src/media/local-media-access.ts:60-62`
  - body: `resolved` now comes from the first variant, which is always the lexical path, so the default `workspace-*` sibling restriction no longer checks the canonical target when the media file exists. A symlinked path can miss the guard that previously ran against `fs.realpath(mediaPath)`.
  - confidence: 0.86
