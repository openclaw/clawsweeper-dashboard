## Summary

Keep open: the reported macOS `/tmp` alias bug still appears valid on current main, but this PR head is not safe to merge because it weakens canonical local-media containment and still lacks required real Telegram proof.

## What This Changes

The PR changes local media validation to compare lexical and canonical path variants for configured local media roots and adds regression tests for macOS `/tmp` versus `/private/tmp` symlink aliases.

## Best Possible Solution

Revise `assertLocalMediaAllowed()` so existing files remain contained by canonical target, macOS root aliases are tolerated safely, workspace sibling checks stay canonical, and coverage includes out-of-root symlink rejection plus real Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main can reject the `/tmp` versus `/private/tmp` alias case, and the PR’s symlink escape follows directly from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR addresses the right false negative, but the maintainable fix must preserve canonical containment for existing files while adding safe root-alias tolerance and explicit escape coverage.
