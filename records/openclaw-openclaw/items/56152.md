## Summary

Keep open. Current main still does not implement the PR's `/tmp` versus `/private/tmp` local media-root alias handling, but the proposed patch weakens the `localRoots` filesystem containment boundary and should be revised before merge.

## What This Changes

The PR changes local media access validation to compare lexical and realpath variants of media paths and roots, and adds regression tests for macOS `/tmp` and `/private/tmp` symlink alias cases.

## Best Possible Solution

Revise the PR so media paths use the canonical path whenever `fs.realpath(mediaPath)` succeeds, fall back to lexical media paths only for missing/unresolvable files, still allow configured root aliases needed for `/tmp` and `/private/tmp`, preserve workspace-sibling checks against canonical variants, and add regression tests for alias acceptance plus symlink escapes.

## Reproduction Assessment

Yes. The original false negative is reproducible from current main's one-sided realpath behavior when the media path and configured root resolve through different `/tmp` aliases, and the PR regression is reproducible from the diff with a symlink under an allowed root whose realpath points outside it.

## Solution Assessment

No. The PR targets the right bug, but accepting lexical media-path variants after `fs.realpath(mediaPath)` succeeds is not the narrowest maintainable fix because it weakens containment; a safer fix should keep canonical media-path enforcement and make root alias handling more precise.
