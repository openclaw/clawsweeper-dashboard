## Summary

Keep open: the PR targets a plausible macOS `/tmp` alias false negative, but the current patch weakens a local file-read containment boundary and still lacks observable real Telegram proof.

## What This Changes

The PR changes local media root validation to compare lexical and realpath path variants and adds macOS `/tmp`/`/private/tmp` regression tests.

## Best Possible Solution

Revise `assertLocalMediaAllowed()` so existing media files are contained by canonical target, macOS root aliases are tolerated without symlink escapes, workspace sibling checks stay canonical, and tests cover both alias false negatives and out-of-root symlink rejection.

## Reproduction Assessment

Yes, at source level. Current main compares one resolved media path against one resolved root, so the PR's mocked `/tmp` and `/private/tmp` alias cases can fail; the PR regression also follows directly from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR aims at the right false negative, but the maintainable fix must preserve canonical containment for existing files while adding alias tolerance and explicit symlink-escape regression coverage.
