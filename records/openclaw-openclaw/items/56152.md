## Summary

Keep open: the PR addresses a real macOS `/tmp` versus `/private/tmp` false negative, but the current patch weakens local media containment by accepting lexical media-path variants after canonicalization succeeds.

## What This Changes

The PR changes local media root validation to compare lexical and realpath variants and adds Vitest coverage for macOS `/tmp` and `/private/tmp` alias cases.

## Best Possible Solution

Revise the PR so media paths use the canonical path whenever `fs.realpath(mediaPath)` succeeds, tolerate `/tmp` and `/private/tmp` aliasing only without widening media-path containment, keep workspace-sibling checks canonical, and add escape regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can reject mismatched lexical/canonical `/tmp` aliases, and the PR regression follows from matching lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR targets the right false negative, but accepting lexical media variants after successful realpath resolution is not the narrowest maintainable fix for a local-root boundary.

## Review Findings

  - body: The new cross-product check accepts the lexical `path.resolve(mediaPath)` variant even when `fs.realpath(mediaPath)` resolves outside every allowed root.   - confidence: 0.93

  - confidence: 0.86
