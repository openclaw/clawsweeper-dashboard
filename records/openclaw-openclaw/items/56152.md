## Summary

Keep this PR open: it targets a real local media false negative, but the current patch weakens the local media read boundary and still lacks observable after-fix proof from a real Telegram run.

## What This Changes

The PR changes local media root validation to compare lexical and realpath variants, and adds macOS `/tmp` versus `/private/tmp` regression tests.

## Best Possible Solution

Revise local media validation so existing files remain checked by canonical paths, root alias tolerance handles `/tmp` versus `/private/tmp` without symlink escapes, and regression coverage plus real Telegram proof are added.

## Reproduction Assessment

Yes, at source level: current main can reject mismatched lexical/canonical `/tmp` aliases, and the PR regression follows directly from matching lexical media variants after `fs.realpath(mediaPath)` succeeds. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR targets the right false negative, but the safer fix is to keep canonical media containment for existing files while tolerating root aliases and covering out-of-root symlink rejection.
