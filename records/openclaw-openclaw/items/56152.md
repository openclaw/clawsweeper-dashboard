## Summary

Keep this PR open. It targets a plausible macOS `/tmp` versus `/private/tmp` local media false negative, but the current diff weakens the local filesystem access boundary and still lacks observable real Telegram proof.

## What This Changes

The PR changes local media root validation to compare lexical and realpath variants for media paths and roots, and adds mocked regression tests for macOS `/tmp` and `/private/tmp` alias cases.

## Best Possible Solution

Revise local media validation so existing media files remain checked by canonical target, `/tmp` alias tolerance does not permit symlink escapes, workspace sibling checks stay canonical, and regression coverage plus real Telegram proof are added.

## Reproduction Assessment

Yes, at source level. Current main uses a single canonical-or-lexical path pair, so the mocked `/tmp` alias cases in the PR can fail, and the PR regression follows directly from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR targets the right false negative, but the maintainable fix needs to preserve canonical containment for existing files while tolerating root aliases and explicitly covering out-of-root symlink rejection.
