## Summary

Keep open: the reported macOS `/tmp` alias false negative is plausible, but this PR weakens local media containment by accepting lexical media-path variants after `realpath` succeeds, and it still needs observable after-fix proof before merge.

## What This Changes

The PR changes local media root validation to compare lexical and realpath variants and adds macOS `/tmp`/`/private/tmp` alias regression tests.

## Best Possible Solution

Revise the PR so existing media files are checked against canonical paths, root alias tolerance handles `/tmp` versus `/private/tmp` without authorizing symlink escapes, and the new tests cover both alias success and out-of-root symlink rejection.

## Reproduction Assessment

Yes, at source level: current main can reject mismatched lexical/canonical `/tmp` aliases, and the PR regression follows directly from matching lexical media variants after `fs.realpath(mediaPath)` succeeds. I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No. The PR targets the right false negative, but accepting lexical media variants after successful canonicalization is not the narrowest maintainable fix for a local-root access boundary.
