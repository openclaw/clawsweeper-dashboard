## Summary

Keep this PR open, but it is not mergeable as-is: the change addresses a real macOS `/tmp` alias false negative, yet the proposed cross-product validation weakens the existing canonical local-media containment boundary and still lacks attached real Telegram proof.

## What This Changes

The PR changes local media root validation to compare lexical and realpath variants for media paths and allowed roots, and adds mocked `/tmp`/`/private/tmp` regression tests.

## Best Possible Solution

Revise `assertLocalMediaAllowed()` so existing files remain contained by canonical target, macOS root aliases are tolerated without symlink escapes, workspace sibling checks stay canonical, and regression coverage includes out-of-root symlink rejection plus redacted Telegram proof.

## Reproduction Assessment

Yes, at source level. Current `main` can reject the mocked `/tmp` and `/private/tmp` alias case, and the PR's symlink escape follows from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR targets the right false negative, but the maintainable fix must preserve canonical containment for existing files while adding safe alias tolerance and explicit escape coverage.
