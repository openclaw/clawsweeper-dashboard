## Summary

Keep this PR open: it targets a plausible current-main local media false negative, but the proposed patch weakens canonical path containment and still lacks observable Telegram proof before merge.

## What This Changes

The PR changes local media root validation to compare lexical and realpath variants for media paths and roots, plus adds Vitest coverage for macOS `/tmp` and `/private/tmp` alias cases.

## Best Possible Solution

Revise `assertLocalMediaAllowed()` so existing files are contained by canonical target, macOS root aliases are tolerated without symlink escapes, workspace sibling checks stay canonical, and tests cover both alias false negatives and out-of-root symlink rejection.

## Reproduction Assessment

Yes, at source level. Current main compares one resolved media path against one resolved root, so the PR's mocked `/tmp` and `/private/tmp` alias cases can fail; the PR regression follows from accepting lexical media variants after `fs.realpath(mediaPath)` succeeds.

## Solution Assessment

No. The PR aims at the right false negative, but the safer fix must preserve canonical containment for existing media files while only adding alias tolerance where it does not reopen symlink escapes.
