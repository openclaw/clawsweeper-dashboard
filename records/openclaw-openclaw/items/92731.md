## Best Possible Solution

Replace the arbitrary string-key credential test with a canonical active top-level account predicate aligned with account selection and SecretInput handling, then cover inherited config-only strings, `defaultAccount`, SecretRefs, real top-level credentials, and `accounts.default`.

## Solution Assessment

No. The scanner is the right layer, but the current string-key heuristic is not the best fix; the maintainable solution is a canonical active-account predicate shared with the channel account/secret contracts.

Full review comments:

- **[P2] Ignore inherited config strings when deciding parent account activity:** `src/commands/doctor/shared/empty-allowlist-scan.ts:124-126`
  - body: This predicate still treats any non-excluded top-level string as credentials. A supported Telegram multi-account config can set inherited `apiRoot` while every named account has its own `groupAllowFrom`; the parent is only fallback config, but `apiRoot` keeps `skipGroupAllowlistCheck` off and the original false warning remains. Use explicit active-account detection and cover inherited config-only strings.
  - confidence: 0.9
- **[P2] Recognize SecretRef credentials before suppressing parent warnings:** `src/commands/doctor/shared/empty-allowlist-scan.ts:126`
  - body: The inverse case is a top-level `botToken` configured as a structured SecretRef. Telegram selection treats any configured top-level `botToken` as an implicit default account, and the schema accepts `botToken` as `SecretInput`, but this predicate only counts string values. With a SecretRef token plus named accounts that have allowlists, doctor can suppress the parent warning even though the active default account may drop group messages.
  - confidence: 0.86
