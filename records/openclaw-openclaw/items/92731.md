## Summary

Keep open: the scanner-layer fix is useful, but the branch is not merge-ready because the new top-level credential heuristic treats supported string config such as `defaultAccount` as credentials, so a documented multi-account shape can still hit the original false-positive warning.

## What This Changes

The PR adds a skip flag to doctor empty-allowlist scanning so top-level group-allowlist and plugin extra warnings are suppressed when sub-accounts appear to have effective allowlists, with regression tests.

## Best Possible Solution

Use an explicit top-level active-account test based on known account identity or credential fields, and add regression coverage for selector-only parent config such as `defaultAccount` before landing one canonical fix for the linked doctor warning bug.

## Reproduction Assessment

Yes. Source inspection shows that `defaultAccount: "work"` keeps `topLevelHasCredentials` true, so a documented multi-account config with every named account carrying `groupAllowFrom` never passes the parent skip flag.

## Solution Assessment

No. The scanner is the right layer, but the current solution uses a broad string-key heuristic instead of an explicit active-account predicate, leaving supported selector-only parent configs unfixed.
