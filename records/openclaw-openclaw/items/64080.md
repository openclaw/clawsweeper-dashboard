## Summary

Keep open. The PR targets a real Slack split-session report that is still observable in current main, but the branch is not correct as submitted: it is based on the older inline routing helper, current main now routes through `prepare-routing.ts`, and the proposed simplification leaves an unused `isRoomish` binding while changing room-session semantics that current tests explicitly assert.

## What This Changes

The PR changes Slack routing and tests so top-level channel/group messages under `replyToMode="all"` use per-message thread-scoped session keys aligned with `MessageThreadId` instead of the shared room session.

## Best Possible Solution

Land a rebased Slack-owned fix only if maintainers confirm that top-level room messages under `replyToMode="all"` should align inbound `SessionKey` with `MessageThreadId`. The implementation should live in `prepare-routing.ts`, preserve `off`/`first`/`batched`, seeded-thread, and runtime-binding behavior, update the routing tests, and add the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveSlackRoutingContext` on current main for a top-level Slack channel message with `replyToMode="all"`, `isRoomish=true`, no `thread_ts`, and no seeded mention; it resolves the inbound session to the base channel key while `resolveSlackThreadContext` sets `MessageThreadId` to the message timestamp.

## Solution Assessment

No, not as submitted. The intended behavior matches the linked report, but the patch targets an older helper shape, leaves an unused binding, omits the changelog, and needs maintainer confirmation that the current shared-room-session assertion should change for `replyToMode="all"`.
