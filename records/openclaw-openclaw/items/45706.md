## Summary

Keep open. Current main still unconditionally sends `stream_options.include_usage` for streaming OpenAI Chat Completions payloads even when model compat resolves `supportsUsageInStreaming: false`, so the Mindie-style strict-parameter 422 remains source-reproducible.

## What This Changes

The issue asks OpenClaw to stop failing against a strict Mindie/OpenAI-compatible DeepSeek backend that rejects unsupported request parameters with HTTP 422.

## Best Possible Solution

Honor the existing `compat.supportsUsageInStreaming: false` contract in the outbound Chat Completions streaming payload while preserving `stream_options.include_usage` for known or explicitly compatible endpoints.

## Reproduction Assessment

Yes for a source-level current-main path: configure an `openai-completions` model with `compat.supportsUsageInStreaming: false`, and `buildOpenAICompletionsParams()` still emits `stream_options.include_usage`. I did not run a live Mindie server in this read-only review.

## Solution Assessment

Yes. A generic compat-gated payload fix is the narrow maintainable shape, but it must preserve streamed usage for providers and local backends that explicitly support or require the field.
