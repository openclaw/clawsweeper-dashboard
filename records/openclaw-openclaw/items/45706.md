## Summary

Keep open. Current main still has an actionable OpenAI-compatible request-shaping gap for strict Mindie-style endpoints: the config surface can express a streaming-usage opt-out, but the Chat Completions payload builder still sends streamed usage options on every streaming request. The related #50374 is relevant strict-provider cleanup, but it tracks message-key sanitization rather than this request-parameter issue.

## What This Changes

The reporter wants a local Mindie/DeepSeek OpenAI-compatible model that returns HTTP 422 request-parameter validation errors to work through OpenClaw.

## Best Possible Solution

Honor the existing generic compat contract in the Chat Completions transport: omit `stream_options` when `compat.supportsUsageInStreaming` is explicitly false, keep streamed usage for known or explicitly compatible endpoints, and avoid Mindie-specific branching.

## Reproduction Assessment

Yes for the OpenClaw-side bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false` and inspect `buildOpenAICompletionsParams()`, which still includes `stream_options.include_usage`. The full remote 422 still needs a strict Mindie-style backend or its rejected-parameter response for end-to-end proof.

## Solution Assessment

Yes. A generic compat-gated payload fix is the narrowest maintainable solution; globally removing `stream_options` would regress compatible local/proxy providers, while hardcoding Mindie would put provider-specific policy into the core transport.
