## Summary

Close as implemented on current main. The remaining confirmed Mindie/OpenAI-compatible 422 path was strict request-parameter shaping around `stream_options.include_usage`; current main now honors the generic `compat.supportsUsageInStreaming` contract, while existing schema-backed flags cover the tool and strict-message-key cases discussed in the thread.

## What This Changes

The issue reports Mindie/OpenAI-compatible backend HTTP 422 failures from unsupported OpenAI request parameters during agent runs.

## Best Possible Solution

Keep the generic compat-gated OpenAI-compatible request shaping and direct strict backends to the supported `compat.supportsUsageInStreaming`, `compat.supportsTools`, and `compat.strictMessageKeys` flags instead of Mindie-specific core logic.

## Reproduction Assessment

Yes, at source level. A generic custom `openai-completions` model that does not support streaming usage now omits `stream_options`, and current tests cover that path; I did not run a live Mindie server.

## Solution Assessment

Yes. The current fix is the narrow maintainable solution because it honors existing provider/model compat metadata, avoids hardcoding Mindie, and preserves usage reporting for known or explicitly opted-in compatible backends.
