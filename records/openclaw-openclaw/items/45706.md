## Summary

Keep open. Current main still sends `stream_options.include_usage` unconditionally for `openai-completions` streaming requests, so strict Mindie-style endpoints cannot use the existing compat opt-out to avoid unsupported request fields.

## What This Changes

Fix Mindie/OpenAI-compatible request shaping so unsupported OpenAI request parameters do not cause HTTP 422 failures on strict local backends.

## Best Possible Solution

Honor the existing generic compat contract in the Chat Completions transport: omit `stream_options` when `compat.supportsUsageInStreaming` is explicitly false, keep streamed usage for known or explicitly compatible endpoints, and avoid Mindie-specific branching.

## Reproduction Assessment

Yes for the OpenClaw-side payload bug: source inspection shows `buildOpenAICompletionsParams()` still includes `stream_options.include_usage` even when the model compat path can resolve `supportsUsageInStreaming: false`. The full remote 422 still needs a strict Mindie-style backend for end-to-end confirmation.

## Solution Assessment

Yes. A generic compat-gated payload fix is the narrowest maintainable solution; globally removing `stream_options` would regress compatible local/proxy providers, while hardcoding Mindie would put provider-specific policy into the core transport.
