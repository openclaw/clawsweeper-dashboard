## Summary

Keep open. Current main still has the actionable OpenAI-compatible request-shaping gap described in the prior ClawSweeper comments: OpenClaw can store a generic streaming-usage opt-out, but the Chat Completions transport still sends `stream_options.include_usage` on every streaming request.

## What This Changes

The issue reports Mindie/OpenAI-compatible HTTP 422 failures from unsupported request parameters when using a local DeepSeek model.

## Best Possible Solution

Honor the existing generic `compat.supportsUsageInStreaming` contract in the OpenAI-compatible Chat Completions payload builder, preserving streamed usage for known or explicitly compatible endpoints without adding Mindie-specific branching.

## Reproduction Assessment

Yes for the OpenClaw-side payload bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false`, call `buildOpenAICompletionsParams()`, and current main still includes `stream_options`. End-to-end confirmation still needs a strict Mindie-style backend.

## Solution Assessment

Yes. A generic compat-gated payload fix is the narrowest maintainable solution; globally removing streamed usage would regress compatible local/proxy providers, while hardcoding Mindie would put provider-specific policy into core.
