## Summary

Keep open. Current main still has the OpenAI-compatible request-shaping gap from the prior reviews: strict message-key support has landed, but `buildOpenAICompletionsParams()` still sends `stream_options.include_usage` unconditionally, so a strict Mindie-style endpoint cannot opt out through the existing compat contract.

## What This Changes

The issue reports Mindie/OpenAI-compatible Chat Completions requests failing with HTTP 422 when OpenClaw sends parameters the backend rejects.

## Best Possible Solution

Honor `compat.supportsUsageInStreaming: false` in the OpenAI-compatible Chat Completions payload builder, while keeping streamed usage for known or explicitly compatible providers and using existing strict-message/tool compat flags for other rejected payload fields.

## Reproduction Assessment

Yes for a source-level current-main path: construct an `openai-completions` model with `compat.supportsUsageInStreaming: false` and `buildOpenAICompletionsParams()` still includes `stream_options`. I did not run a live Mindie backend in this read-only review.

## Solution Assessment

Yes. A generic compat-gated payload fix is the narrowest maintainable solution; hardcoding Mindie would put provider-specific policy into core, and globally removing streamed usage would regress compatible providers.
