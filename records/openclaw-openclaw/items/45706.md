## Summary

Keep open. Current main has fixed adjacent strict-provider cases like string-only message content and strict replay message keys, but it still sends `stream_options.include_usage` on every OpenAI Chat Completions streaming request, so the Mindie-style strict request-parameter failure remains actionable.

## What This Changes

The issue reports Mindie/OpenAI-compatible HTTP 422 errors from strict request-parameter validation when using a local DeepSeek model.

## Best Possible Solution

Add generic OpenAI-compatible request shaping that lets strict backends omit unsupported top-level streaming usage parameters while preserving streamed usage for known or explicitly compatible providers.

## Reproduction Assessment

Yes, at source level: configure an `openai-completions` model with `compat.supportsUsageInStreaming: false` and `buildOpenAICompletionsParams()` still emits `stream_options.include_usage`. I did not run a live Mindie server.

## Solution Assessment

Yes to keeping this open for a generic payload-shaping fix, but not as a blanket one-line gate. The repair must preserve usage tracking for providers/local backends that rely on `stream_options.include_usage`.
