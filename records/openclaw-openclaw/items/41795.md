## Summary

Keep open. Current main now supports Telegram Bot API transport through standard HTTP(S)/ALL proxy env vars and the managed `OPENCLAW_PROXY_URL` fallback, but it still does not implement this PR's Telegram-specific `OPENCLAW_TELEGRAM_PROXY` override. The submitted patch is also stale against the current plugin layout and has a review-flagged malformed-proxy handling gap, so it needs refresh or maintainer product decision rather than cleanup closure.

## What This Changes

The PR adds an old-layout Telegram proxy resolver with precedence from config to `OPENCLAW_TELEGRAM_PROXY` to standard HTTP(S) proxy env vars, wires it into send, polling, probe, and audit paths, and adds proxy precedence tests.

## Best Possible Solution

Retarget the useful part of the PR into `extensions/telegram` only if maintainers want the new env surface: implement a Telegram-owned resolver with precedence `channels.telegram[.accounts.<id>].proxy` > `OPENCLAW_TELEGRAM_PROXY` > existing standard env/managed proxy behavior, thread it through all Telegram API paths including chat lookup, and document/test the exact fallback and failure behavior.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature PR and stale patch review. The current-main gap is high-confidence from exact symbol search plus source inspection of the Telegram plugin proxy, send, monitor, probe, audit, and transport paths.

## Solution Assessment

No. The submitted approach is not the best current solution because it edits the old `src/telegram` layout and does not handle malformed Telegram-specific proxy values as carefully as the existing env-proxy transport; the safer path is a narrow `extensions/telegram` refresh using current transport helpers.
