## Summary

Keep this PR open for maintainer judgment. Current main already implements the standard Telegram proxy-env path in the Telegram plugin, but it does not implement the requested `OPENCLAW_TELEGRAM_PROXY` surface, and the submitted patch still targets the removed `src/telegram` layout.

## What This Changes

The PR adds a proxy resolver in `src/telegram` that prefers config proxy, then `OPENCLAW_TELEGRAM_PROXY`, then standard HTTP(S) proxy env vars across polling, send, probe, audit, and tests.

## Best Possible Solution

If maintainers want a Telegram-specific env override, implement it inside `extensions/telegram` with explicit precedence, malformed-value behavior, docs, changelog, and coverage for all Bot API paths; otherwise keep the existing standard env and managed proxy contract.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature PR review. The stale path and missing `OPENCLAW_TELEGRAM_PROXY` surface are source-verifiable on current main.

## Solution Assessment

No. The submitted approach is not the best current solution because it edits the removed Telegram layout and introduces a new public env surface without retargeting the plugin, documenting it, or matching current proxy failure semantics.
