## Summary

Keep this PR open. The standard HTTP(S)/ALL proxy part is already implemented on current main, but the PR's Telegram-specific `OPENCLAW_TELEGRAM_PROXY` env surface is not present, and the submitted patch is stale against the current Telegram plugin layout and has a malformed-proxy failure path that should block merge as-is.

## What This Changes

The PR adds an old-layout Telegram proxy resolver with precedence from config to `OPENCLAW_TELEGRAM_PROXY` to standard HTTP(S) proxy env vars, then wires it into send, polling, probe, audit, and proxy tests.

## Best Possible Solution

Retarget only the still-missing Telegram-specific env behavior into `extensions/telegram` if maintainers want that public config surface: keep config proxy precedence, preserve existing standard env and managed-proxy behavior, handle malformed values consistently, include chat lookup and startup paths, and document/changelog the new env variable.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature PR review. The current-main gap and patch staleness are high-confidence from exact symbol search, absence of `src/telegram`, and inspection of the current Telegram plugin transport paths.

## Solution Assessment

No. The submitted approach is not the best current solution because it edits the old Telegram layout and routes malformed Telegram-specific env/config proxy values through `makeProxyFetch` without the defensive fallback used by the current env-proxy transport.
