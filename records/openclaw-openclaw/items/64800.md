## Summary

Keep open. The reported provider-wrapper composition bug is still source-reproducible, but this PR is not merge-ready because it changes a stale helper body instead of the active hook-runtime helper, and the only after-fix proof is mock/unit evidence.

## What This Changes

The PR attempts to chain matching provider `wrapStreamFn` hooks and adds a provider-runtime regression test for a primary provider plus a `hookAliases` plugin.

## Best Possible Solution

Port stream-wrapper composition into `src/plugins/provider-hook-runtime.ts`, preserving provider matching, stable dedupe, and `runtimeHandle` semantics with focused regression coverage and contributor-supplied real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the active helper invokes only one provider wrapper while the runtime caller calls that helper once, so a primary provider plus matching hookAlias wrapper cannot compose there.

## Solution Assessment

No. The intended composition is the right narrow direction, but this PR edits the stale `provider-runtime.ts` helper location instead of the active `provider-hook-runtime.ts` helper used by current main.
