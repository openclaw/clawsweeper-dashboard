## Summary

The reported provider wrapper bug is valid and the author addressed the earlier provider-filtering and dedupe review comments, but this PR now patches a stale helper body. Current main routes `wrapProviderStreamFn` through `src/plugins/provider-hook-runtime.ts`, where the active implementation still invokes only one provider wrapper.

## What This Changes

The PR changes provider stream wrapping so the primary provider and matching `hookAliases` providers can chain `wrapStreamFn`, with a regression test that skips unrelated providers.

## Best Possible Solution

Port the wrapper composition into `src/plugins/provider-hook-runtime.ts`, preserve provider matching, stable dedupe, and `runtimeHandle` semantics, keep focused regression coverage, and collect real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a primary provider plus matching hookAlias provider path would resolve only one runtime-owner wrapper, while `extra-params.ts` calls `wrapProviderStreamFn` once.

## Solution Assessment

No. The intended behavior is the narrow maintainable fix, but this branch edits `src/plugins/provider-runtime.ts` instead of the active `src/plugins/provider-hook-runtime.ts` helper used on current main.
