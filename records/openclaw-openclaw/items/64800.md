## What This Changes

The PR changes provider stream-wrapper dispatch to call the primary provider wrapper first, then chain additional matching `hookAliases` providers, with a regression test for chained wrapping and unrelated-provider skipping.

## Best Possible Solution

## Solution Assessment

No as submitted: the fix direction is plausible, but the branch targets the old hook boundary and is now conflicting. The maintainable path is a current-main port with explicit ordering/filtering tests and maintainer agreement that multi-wrapper composition is the intended provider API behavior.
