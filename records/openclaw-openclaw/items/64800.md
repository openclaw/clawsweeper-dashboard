## Summary

Keep open: this targets a real provider-plugin hook gap, but the branch is not merge-ready because current main moved the wrapper owner to `provider-hook-runtime.ts`, the PR lacks real behavior proof, and the proposed extra-wrapper loop needs scoped provider resolution before it is safe.

## What This Changes

The PR changes provider stream wrapper resolution to call the primary provider `wrapStreamFn` and then chain additional matching hook-alias provider wrappers, with a regression test that skips unrelated providers.

## Best Possible Solution

Move any wrapper composition into `src/plugins/provider-hook-runtime.ts` with targeted provider/model owner resolution, explicit ordering/dedupe tests, and redacted real-plugin proof before merge.

## Reproduction Assessment

Yes from source inspection: current main resolves one provider runtime plugin for `wrapProviderStreamFn`, so a secondary hook-alias wrapper is not composed. I did not run tests because this review is read-only.

## Solution Assessment

No as currently branched: the intended provider-first composition is plausible, but the patch targets the old wrapper location and uses an unscoped provider loop that conflicts with current provider-runtime fanout safeguards.
