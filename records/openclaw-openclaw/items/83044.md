## What This Changes

The PR adds an inline finite-number guard around `parsed.httpCode` before passing `status` into provider runtime failure classification.

## Best Possible Solution

Keep the centralized parser and `inferSignalStatus` normalization as the status contract rather than duplicating a caller-local guard in the observation builder.

## Reproduction Assessment

No. Current source inspection shows `parsed.httpCode` cannot be a non-numeric string from `parseApiErrorInfo`, and the classifier ignores non-finite numeric status values before classification.

## Solution Assessment

No. The PR duplicates a guard that already exists in the shared status normalization path, and its changed comment describes a NaN misclassification path current main does not have.
