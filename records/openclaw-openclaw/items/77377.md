## Summary

Keep open: current main still contains a Feishu inbound debounce path that can merge same-sender text messages into one dispatch turn, and this fresh report is too credible to close without a current-main trace of the exact after-response cumulative behavior.

## What This Changes

Report that Feishu WebSocket p2p messages on 2026.5.2 reach the agent with previous questions prepended instead of only the latest user message.

## Best Possible Solution

Keep the issue open and verify whether Feishu p2p DMs are being batched by inbound debounce, queued followups, or session prompt assembly, then add a targeted regression that preserves true fragment batching without carrying completed prior turns into a new user message.

## Reproduction Assessment

Unclear/partial: source inspection can reproduce concatenation for rapid same-sender Feishu text events when inbound debounce is nonzero, but I do not have a current-main live trace showing cumulative messages after separate completed replies.

## Solution Assessment

Unclear: changing Feishu to always dispatch p2p text messages separately would conflict with the documented debounce feature, so the narrow fix should first identify whether this report is debounce config, queue fallback, or a session prompt assembly regression.
