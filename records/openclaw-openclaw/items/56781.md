- `impact:session-state`: The requested fallback chain affects whether compaction and summary models can keep producing session summary state during provider failures.

## Summary

Keep open: current main and v2026.6.6 still lack a compaction-specific fallback array and the related Lossless/context-engine summary-model fallback contract; this remains the canonical product/API decision point rather than a stale cleanup candidate.

## What This Changes

Request an ordered fallback model chain for explicit compaction models and Lossless Claw summary-model configuration so summary work can retry another model on provider failures.

## Best Possible Solution

Design one canonical fallback contract for explicit compaction models and context-engine summary models, with schema/docs/tests covering per-candidate auth, transcript policy re-derivation, all-fail degradation, and upgrade behavior.

## Reproduction Assessment

Yes for the source-level capability gap: current main bypasses fallback traversal when `agents.defaults.compaction.model` is set, and the plugin/context-engine LLM contract accepts one model override. I did not run a live 429, 500, or timeout scenario.

## Solution Assessment

Unclear. The requested direction is reasonable, but maintainers still need to choose whether the fallback contract belongs in compaction config, the generic plugin LLM runtime, Lossless plugin config, or a shared model-selection helper.
