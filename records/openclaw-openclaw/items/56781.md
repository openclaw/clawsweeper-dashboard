## Summary

Keep open. Current main still has the reported feature gap: core compaction accepts a single `agents.defaults.compaction.model`, the strict config schema has no compaction fallback field, and the direct compaction path resolves one provider/model before calling Pi compaction without the shared fallback runner. The Lossless Claw `summaryModel` portion remains plugin-owned/external, so the issue needs maintainer/API-shape judgment rather than cleanup closure.

## What This Changes

Request adding fallback model chains for core compaction and the Lossless Claw summary model so transient model failures can try secondary models before compaction gives up.

## Best Possible Solution

Add a documented core compaction fallback contract that reuses the existing transient-error model failover semantics and proves compaction tries secondary candidates. Keep Lossless Claw summary fallback in the owning plugin unless the plugin needs a generic, plugin-agnostic OpenClaw SDK/runtime helper.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the strict schema has no compaction fallback field, the resolver returns one compaction target, and direct compaction never enters `runWithModelFallback`; no live provider is needed to verify the missing surface.

## Solution Assessment

Unclear. The requested direction is maintainable for core compaction, but the exact public config shape and the external Lossless Claw summary-model boundary should be chosen by a maintainer or plugin owner before implementation.
