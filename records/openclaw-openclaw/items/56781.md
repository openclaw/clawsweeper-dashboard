## Summary

Keep open. Current main still lacks the requested public fallback surfaces: embedded compaction has no `compaction.fallbacks` schema/type field, explicit `compaction.model` remains exact, and Lossless still exposes only plugin-owned `summaryModel` without a fallback chain; the related implementation PRs are closed unmerged or only partially overlap.

## What This Changes

Requests ordered fallback model chains for embedded compaction and Lossless/LCM `summaryModel` so summarization can retry another model on rate limits, provider errors, or timeouts.

## Best Possible Solution

Have maintainers decide the public fallback contract, implement embedded compaction in core if approved, and keep Lossless `summaryModel` fallback in the Lossless/context-engine plugin or a generic plugin SDK seam if that seam is missing.

## Reproduction Assessment

Yes for the source-level gap: current main lacks the requested fallback fields, and explicit compaction models bypass the fallback wrapper. A live 429/timeout repro is not required to show that the requested capability is missing.

## Solution Assessment

Unclear. An explicit fallback list is plausible for embedded compaction, but the Lossless half crosses plugin/context-engine ownership and needs maintainer API direction before a repair PR should choose the contract.
