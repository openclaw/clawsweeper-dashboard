## Label Justifications

- `P2`: This is a useful resilience improvement for compaction and summary-model failures with limited blast radius and no emergency regression evidence.
- `impact:session-state`: Failed compaction can allow transcripts and context state to keep growing without a successful summary.
- `impact:auth-provider`: The requested fallback chain changes model/provider routing and credential selection when the primary summarizer is unavailable.

## Summary

Keep open. Current main and the latest release still intentionally keep explicit compaction model overrides as single exact targets, while plugin LLM completion exposes one requested model and no public fallback-list contract for Lossless-style summary calls; the only related compaction fallback PR was closed unmerged as superseded by this broader API decision.

## What This Changes

The issue requests fallback model chains for explicit compaction model overrides and Lossless/LCM `summaryModel` configuration.

## Best Possible Solution

Approve the fallback contract first, then land a narrow implementation for explicit compaction fallback plus either a generic plugin LLM fallback seam or a clearly plugin-owned Lossless `summaryModel` fallback option, with regression coverage for auth, session state, and exact-model defaults.

## Reproduction Assessment

Yes for the missing capability by source inspection: current config/schema/docs preserve a single exact compaction model, and plugin LLM completion exposes one requested model. I did not run a live provider outage reproduction in this read-only review.

## Solution Assessment

Unclear. The requested fallback behavior is reasonable, but the best fix depends on maintainer API judgment about explicit compaction fallback semantics and whether Lossless summary fallback should be plugin-owned or exposed through a generic runtime contract.
