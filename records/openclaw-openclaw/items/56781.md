## Summary

Keep open. Current main still has the reported core compaction feature gap: compaction configuration accepts one model override, the strict schema has no fallback list, and the direct compaction path resolves one provider/model before running instead of using the shared model fallback runner. The Lossless Claw part is plugin-owned/external, so it should be handled in that plugin or through a generic SDK/runtime helper if core support is needed.

## What This Changes

Request adding optional fallback model chains for core compaction and the Lossless Claw summary model configuration.

## Best Possible Solution

Add a documented core compaction fallback contract that reuses the existing transient-error model failover semantics, with focused schema/runtime tests proving compaction tries secondary candidates. Keep Lossless Claw summary fallback in the owning plugin unless that plugin needs a generic OpenClaw SDK/runtime helper.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agents.defaults.compaction.fallbacks` is absent from the strict schema/type, and the direct compaction runner resolves one model without entering `runWithModelFallback`.

## Solution Assessment

Unclear. The requested direction is sound for core compaction, but the exact config shape and the external Lossless Claw ownership boundary should be settled by a maintainer or plugin owner before an automated repair attempts it.
