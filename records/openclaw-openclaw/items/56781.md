## Label Justifications

- `P2`: This is a useful reliability improvement for compaction and summary-model outages with limited blast radius and no emergency regression evidence.
- `impact:session-state`: Failed compaction can leave transcripts and context state growing without a successful summary checkpoint.
- `impact:auth-provider`: The requested fallback chain changes model/provider routing and credential selection when a primary summarizer is unavailable.

## Summary

Keep open. Current main and the latest release now support fallback for implicit compaction targets, but they still do not provide the requested public fallback list for explicit `agents.defaults.compaction.model` or Lossless/context-engine `summaryModel` calls; the related compaction-only PR was closed as superseded by this broader API decision.

## What This Changes

The issue requests configurable fallback model chains for compaction and Lossless/context-engine summary model calls when the primary model is unavailable.

## Best Possible Solution

Approve an explicit fallback contract first, then land a focused implementation for exact compaction-model fallbacks plus either a generic context-engine/plugin LLM fallback seam or a clearly owned Lossless `summaryModel` fallback option, with auth-routing and session-state regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main accepts only one explicit compaction model, the strict schema has no fallback field, and the explicit compaction path bypasses model fallback. I did not run a live provider outage reproduction in this read-only review.

## Solution Assessment

Unclear. The requested behavior is reasonable, but the best implementation needs a maintainer decision on exact explicit-model semantics and whether summary-model fallback belongs in plugin-owned Lossless config or a generic runtime LLM fallback contract.
