## Summary

Keep open. Current main still has the reported feature gap: core compaction accepts one compaction model and resolves one runtime target, while the shared model fallback runner exists separately and is not used by the direct compaction path. The Lossless Claw summary-model portion appears plugin-owned, so the API shape and core/plugin boundary need maintainer judgment before implementation.

## What This Changes

Request adding optional fallback model chains for core compaction and Lossless Claw summary-model selection so transient primary failures try alternates before failing.

## Best Possible Solution

Add a documented core compaction fallback contract that reuses the existing transient-error model failover semantics and proves secondary candidates are attempted. Keep Lossless Claw summary fallback in the owning plugin unless a plugin-agnostic SDK/runtime helper is needed.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the strict compaction config schema has no fallback key, and direct compaction resolves and runs exactly one model target.

## Solution Assessment

Unclear. The core compaction direction is maintainable if it reuses `runWithModelFallback`, but the final config field names and Lossless Claw ownership boundary should be decided by a maintainer before an automated repair.
