## Summary

Keep open. Current main has shipped sanitized `model_call_started` / `model_call_ended` hooks, but those do not implement this PR’s raw provider payload hook contract; the PR remains the canonical design thread for that remaining capability. It is not merge-ready as proposed because the raw payload exposure needs explicit conversation-access/security gating and the terminal hook semantics still have an unmatched-event path. https://github.com/openclaw/openclaw/pull/58360

## What This Changes

The PR adds additive `before_model_call` and `after_model_call` plugin hooks around embedded runner provider invocations, with new hook event types, hook-runner methods, embedded-runner wiring, replay-payload normalization helper extraction, and focused tests.

## Best Possible Solution

Decide the product/security policy first: either keep the shipped sanitized model-call hooks as the supported API and close this PR, or rework this PR into a gated, documented raw-payload extension that builds on the current `model_call_started` / `model_call_ended` contract, requires explicit conversation access for non-bundled plugins, and has terminal-event coverage for promise, error, full iteration, and abandoned iteration paths.

## Reproduction Assessment

Yes. Code inspection against current main reproduces the remaining capability gap: `before_model_call` / `after_model_call` are absent, while the shipped call-level hooks are metadata-only; the security finding is reproducible by comparing the PR’s raw `requestMessages` hook payload to the existing conversation-access gate.

## Solution Assessment

No, not as proposed. The narrow maintainable direction is either the current sanitized hook pair or a raw-payload variant that is explicitly gated, documented, and wired at a terminal-safe stream boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Gate raw model-call hooks behind conversation access:** `src/plugins/types.ts:2018-2019`
  - body: `before_model_call` carries provider-facing `requestMessages`, but the PR adds the hook as a normal typed hook rather than a conversation-content hook. That lets a non-bundled plugin register for raw prompt/history payloads without `plugins.entries.<id>.hooks.allowConversationAccess=true`, bypassing the existing gate used for `llm_input` and `llm_output`. Add both raw model-call hooks to the conversation-access classification with tests, or avoid exposing raw payloads.
  - confidence: 0.86
- **[P2] Emit after hooks when streams are abandoned:** `src/agents/pi-embedded-runner/run/attempt.ts:1170-1196`
  - body: The after hook is wired through the returned stream’s `.result()` path, so consumers that iterate the stream and return early can skip `after_model_call` entirely. That violates the advertised per-provider-call lifecycle and leaves tracing plugins with unmatched call IDs. Wrap async iteration/finally, like the current diagnostic model-call wrapper, or document and test the one-sided contract before exposing it.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR adds a raw model-call payload surface without showing the existing conversation-content access gate, which is a concrete plugin data-access regression.

Concerns:

- **[high] Raw prompt payload hooks bypass conversation access:** `src/plugins/types.ts:2019`
  - body: The PR exposes provider request messages through a new typed hook, but current registry policy only requires `allowConversationAccess` for names in the conversation hook set. Without adding these raw hooks to that set, third-party plugins can receive prompt/history/tool-result payloads without the explicit conversation-data opt-in required for existing LLM content hooks.
  - confidence: 0.84
