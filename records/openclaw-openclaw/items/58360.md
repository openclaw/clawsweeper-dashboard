## Summary

Keep open for maintainer/security review: current main has safer sanitized per-model-call hooks, while this PR still proposes raw model payload hooks with data-access policy and lifecycle correctness blockers.

## What This Changes

The PR adds raw `before_model_call` and `after_model_call` plugin hooks around embedded-runner provider calls, refactors replay tool-call normalization, and adds focused hook-runner and payload tests.

## Best Possible Solution

Keep the current sanitized `model_call_started` / `model_call_ended` hooks as the default telemetry API; if raw payload hooks are accepted, add them behind explicit conversation-access policy with terminal-safe emission, unique call IDs, provider-bound metadata, docs, and loader/runtime tests.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the review blockers: raw payload fields are exposed through new hooks, terminal emission depends on `.result()`, and call IDs are derived from identifiers reused by retry attempts.

## Solution Assessment

No. The raw hook shape is not the best mergeable solution until maintainers decide the plugin data-access policy; the safer path is current sanitized telemetry or a gated raw-payload API with the lifecycle fixes above.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate raw model-call hooks behind conversation access:** `src/plugins/types.ts:2023`
  - body: `before_model_call` carries provider-facing `requestMessages`, and `after_model_call` can carry the full response message. Current registry policy only gates names classified as conversation hooks, so this new raw surface would let a non-bundled plugin receive conversation content without the existing explicit opt-in. Add both raw hook names to the conversation-access set with loader coverage, or keep these events sanitized.
  - confidence: 0.87
- **[P2] Emit after hooks when streams are abandoned:** `src/agents/pi-embedded-runner/run/attempt.ts:1258-1267`
  - body: `after_model_call` is fired by replacing the stream's `.result()` method. A caller that consumes the async iterator and returns early without calling `.result()` never emits the terminal event, leaving tracing plugins with unmatched call IDs. Wrap async iteration/finally like the current diagnostic model-call wrapper or document and test a one-sided contract before exposing it.
  - confidence: 0.81
- **[P2] Make model-call IDs unique across retries:** `src/agents/pi-embedded-runner/run/attempt.ts:271-272`
  - body: `buildModelCallId` combines `runId`, `sessionId`, and a counter that resets each time `runEmbeddedAttempt` is entered. The retry loop reuses the same run and session identifiers, so the first provider call in each retry can reuse the same call ID. Include an attempt-unique token or carry the counter outside the retry boundary.
  - confidence: 0.78
- **[P2] Report provider-bound model metadata:** `src/agents/pi-embedded-runner/run/attempt.ts:1180-1192`
  - body: The hook captures `provider`/`model`/`api` before delegating to inner stream wrappers. Existing wrappers such as Minimax fast mode can rewrite `model.id` before the real provider call, so the new per-call telemetry can attribute latency/errors to the wrong model. Move this observation closer to the final provider boundary or provide final metadata from the wrapper chain.
  - confidence: 0.74
