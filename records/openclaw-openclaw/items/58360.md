## Summary

Keep open: current main solves provider-call telemetry only with sanitized hooks, while this PR still proposes raw model payload hooks with unresolved access-policy, lifecycle, and API-shape questions.

## What This Changes

The PR adds `before_model_call` and `after_model_call` plugin hook contracts, hook-runner methods, embedded-runner provider-call wiring, replay-normalization extraction, and targeted tests.

## Best Possible Solution

Keep sanitized `model_call_started` and `model_call_ended` as the default telemetry path, and add raw payload hooks only through one maintainer-approved, gated, documented, lifecycle-complete API with real plugin proof.

## Reproduction Assessment

Yes for the PR review concerns by source inspection: the diff exposes raw model-call payload fields, patches `after_model_call` through `.result()`, and builds call IDs from run/session/sequence while retry attempts can reuse run and session identifiers. This is not a current-main bug reproduction because the requested hook is a new public plugin capability.

## Solution Assessment

No as submitted. The observability goal is valid, but the maintainable solution needs one approved hook contract that preserves sanitized defaults, gates raw conversation data, handles lifecycle/correlation edge cases, and coordinates with the newer overlapping preflight PR.

Full review comments:

- **[P1] Gate raw model-call hooks behind conversation access:** `src/plugins/types.ts:2003-2049`
  - body: `requestMessages` exposes provider-facing prompt/history and `responseMessage` can expose the full assistant response, but this PR does not demonstrate that the new hook names are registered through the same conversation-content access gate as other raw hooks. Add the hook names to that policy and cover non-bundled blocked/allowed registration before exposing this data.
  - confidence: 0.9
- **[P2] Emit after hooks when streams are abandoned:** `src/agents/pi-embedded-runner/run/attempt.ts:1261-1270`
  - body: `after_model_call` is still driven by patching `.result()`. If the stream is consumed or abandoned without resolving `.result()`, tracing plugins receive `before_model_call` without a matching after event; implement finalization at the stream iteration boundary or explicitly document and test the one-sided contract.
  - confidence: 0.84
- **[P2] Make model-call IDs unique across retry attempts:** `src/agents/pi-embedded-runner/run/attempt.ts:267-270`
  - body: `buildModelCallId` combines `runId`, `sessionId`, and a wrapper-local sequence, but retry attempts can reuse the same run and session while reinstalling the wrapper and resetting the counter. Include an attempt-unique token or keep the counter outside the retry boundary so downstream correlation cannot merge separate provider calls.
  - confidence: 0.82
