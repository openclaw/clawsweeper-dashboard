## Summary

Keep this open. Current main still sends non-string `choice.delta.content` straight into the OpenAI-compatible text stream, and the issue is explicitly paired with open PR #75339, so the right terminal outcome is to review and land or replace that PR before closing this issue.

## What This Changes

The issue reports repeated `[object Object]` text leaking into assistant replies, persisted sessions, and memory when Mistral thinking output returns structured content blocks.

## Best Possible Solution

Normalize structured content at the shared OpenAI-compatible stream boundary: route `thinking` blocks into hidden thinking output, route `text` blocks into visible assistant text, drop unrecognized objects without string coercion, and add regression coverage before closing after PR #75339 or an equivalent fix lands.

## Reproduction Assessment

Yes. The issue gives concrete versioned steps, and current-main source inspection gives a focused unit reproduction: feed a Mistral-style `delta.content` array with typed `thinking`/`text` blocks into `processOpenAICompletionsStream` and assert no `[object Object]` enters visible text.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared OpenAI-compatible stream parser rather than memory cleanup or a Mistral-only postprocessor, because the corruption starts in live assistant text before session persistence and memory ingestion.
