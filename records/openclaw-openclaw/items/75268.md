## Summary

Keep this open. Current main still routes truthy OpenAI-compatible `choice.delta.content` directly into visible text before reasoning parsing, and open PR #75339 already uses closing syntax for the same Mistral thinking-mode parser bug.

## What This Changes

The issue asks OpenClaw to stop leaking `[object Object]` into Mistral thinking-mode replies, session files, and memory.

## Best Possible Solution

Normalize structured `delta.content` at the shared OpenAI-compatible streaming boundary, routing thinking blocks to hidden thinking output and text blocks to visible assistant text before closing after #75339 or an equivalent replacement lands.

## Reproduction Assessment

Yes. The report gives concrete versioned steps, and source inspection gives a focused reproduction path: feed a Mistral-style structured `delta.content` array through the OpenAI-compatible completions stream and verify `[object Object]` does not enter visible text.

## Solution Assessment

Yes. The shared OpenAI-compatible stream parser is the narrow maintainable fix boundary; a Mistral-only cleanup or memory repair would leave the live assistant text corruption path in place.
