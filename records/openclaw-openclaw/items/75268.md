## Summary

Keep this open. Current main still routes truthy OpenAI-compatible `choice.delta.content` directly into visible text before reasoning normalization, and the provided GitHub context shows open PR #75339 already uses closing syntax for the same parser fix, so the right cleanup path is to review, land, or replace that PR rather than closing this report or queueing duplicate repair work.

## What This Changes

Fix Mistral thinking-mode streaming responses so structured reasoning/text content blocks do not leak as `[object Object]` into assistant replies, session files, or memory.

## Best Possible Solution

Normalize structured `delta.content` at the shared OpenAI-compatible streaming boundary, route thinking/reasoning blocks to hidden thinking output and text blocks to visible assistant text, add regression coverage for object/array content non-leakage, then close after #75339 or an equivalent replacement merges.

Yes. The issue gives concrete versioned steps, and current-main source inspection gives a focused reproduction: feed a Mistral-style structured `delta.content` array through `processOpenAICompletionsStream` and verify `[object Object]` does not enter visible text.

Yes. The shared OpenAI-compatible stream parser is the narrow maintainable fix boundary because Mistral reaches that transport through the provider manifest; a Mistral-only postprocessor or memory cleanup would leave the live text corruption path in place.
