## Label Justifications

- `P1`: Background exec completion is a core async agent workflow, and this PR changes whether completion output reaches or bypasses user-visible delivery paths.
- `merge-risk: 🚨 message-delivery`: The PR can change whether exec completion output is relayed, suppressed, or consumed for targetless WebChat sessions.
- `merge-risk: 🚨 compatibility`: Heartbeat `target: "none"` is the documented default, and the PR creates a new exception for omitted targets based on raw session channel state.

## Summary

Keep open: the PR fixes the right central symptom by making exec completion details model-visible, but the current webchat delivery override still mixes context visibility with user-delivery authorization and can bypass the documented heartbeat target-default behavior.

## What This Changes

The PR adds an exec-completion transcript marker, appends exec details to exec-event heartbeat transcript bodies, filters those transcript artifacts, and expands exec-event prompt delivery for targetless webchat sessions.

## Best Possible Solution

Make exec completion details model-visible independently of delivery authorization, keep user-relay wording tied to the canonical delivery resolver or an explicit active source-reply context, and add runner tests for default-none, explicit-none, target-last/external route, and active WebChat behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Mostly: the central model-input problem is supported by the PR's before/after live logs and by current source showing heartbeat transcript prompts hide exec details, but I did not run a local gateway repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: the transcript marker and inline exec body are the right direction, but the webchat fix should separate model context visibility from user delivery authorization instead of expanding `deliverToUser` from raw `lastChannel`.
