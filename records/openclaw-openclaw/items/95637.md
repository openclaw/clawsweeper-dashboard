## Summary

Keep open. The patch is a narrow, source-backed QQ Bot bug fix: current main and latest v2026.6.9 still require three-or-more dashes in table delimiter cells, while the PR changes only the private separator helper and focused regression coverage; I found no patch-introduced correctness or security finding, but required CI is still failing in unrelated paths and maintainer/check gating remains.

## What This Changes

The PR changes QQ Bot markdown table separator detection from three-or-more dashes to one-or-more dashes and adds regression coverage for one-, two-, and aligned-dash separators.

## Best Possible Solution

Land the narrow QQ Bot helper and regression test once maintainer review and required checks are satisfied, while keeping the broader long-reply chunking behavior in https://github.com/openclaw/openclaw/issues/95526 separate.

## Reproduction Assessment

Yes by source inspection: current main and v2026.6.9 use /^:?-{3,}:?$/, so one- and two-dash separator lines fail recognition and the pending-header state can drop earlier table content. I did not run tests because this review is read-only, but the PR body includes before/after terminal output against the exported chunker.

## Solution Assessment

Yes. The private QQ Bot separator helper is the implicated layer, and changing only the dash quantifier matches the GFM delimiter contract without adding a parser, config knob, or cross-channel policy change.
