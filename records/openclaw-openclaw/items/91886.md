## Summary

Keep open: current main and the latest release still route Feishu streaming through CardKit interactive cards, and the requested adaptive/hybrid/plain-text fallback behavior remains a product/channel decision rather than an implemented fix.

## What This Changes

The issue asks for Feishu streaming to avoid laggy or empty incremental CardKit updates by adding adaptive fallback, tuning, retry, or a final-only/hybrid delivery mode.

## Best Possible Solution

Define the supported Feishu streaming UX first, then implement one coherent path: harden CardKit fallback/retry behavior, restore stricter renderMode separation, or add a narrow documented hybrid mode with tests and live Feishu proof.

## Reproduction Assessment

Yes for the core routing claim: current source shows Feishu streaming starts a CardKit session and pushes partial/final text through card updates. No live Feishu/ZhipuAI run was used to reproduce latency or failed intermediate updates.

## Solution Assessment

Unclear as an implementation plan: streaming:false already gives final-only delivery, while adaptive fallback, interval tuning, and hybrid behavior need a maintainer decision before a narrow fix is safe.
