## Summary

Keep open: current main and the latest release still make Feishu `streaming: true` use CardKit interactive-card updates, while the requested adaptive fallback, hybrid/segment delivery, or smoother provider-specific behavior remains unsettled product/channel work.

## What This Changes

The issue requests smoother Feishu streaming or a hybrid/final-text delivery mode because `streaming: true` sends incremental CardKit card updates instead of native plain text.

## Best Possible Solution

Define one supported Feishu delivery contract: either keep CardKit-first streaming and harden fallback/retry behavior, or add a narrow opt-in segment/hybrid/final-text mode with docs, config metadata, tests, and live Feishu proof.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence path for the core routing claim: Feishu `streaming: true` starts and updates a CardKit card, but the latency and failed-update behavior were not reproduced in a live Feishu/ZhipuAI setup.

## Solution Assessment

Unclear. `streaming: false` already gives final-only plain message delivery, while adaptive fallback, debounce tuning, segment delivery, or changing the default card policy needs a maintainer product decision and live channel proof.
