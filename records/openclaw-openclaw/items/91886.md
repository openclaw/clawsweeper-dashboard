## Summary

Keep open: current main confirms Feishu streaming is intentionally CardKit-card based and documents final-only plain text via `streaming: false`, but the requested adaptive/hybrid fallback and CardKit reliability behavior remain unimplemented product/channel work.

## What This Changes

Request configurable or adaptive Feishu streaming delivery so unreliable CardKit incremental updates can fall back to plain final text while preserving websocket reception.

## Best Possible Solution

Define the supported Feishu streaming UX in one maintainer-owned path: either harden CardKit streaming and fallback behavior, or add a narrow final-only/hybrid mode with clear docs and tests.

## Reproduction Assessment

Yes for the core routing claim: current source shows Feishu streaming starts a CardKit session and queues partial/final text into card updates. No live Feishu/ZhipuAI setup was used to reproduce the latency and failed-update behavior.

## Solution Assessment

Unclear as an implementation plan: `streaming: false` already gives final-only message delivery, while adaptive fallback, interval tuning, and hybrid mode need a maintainer product choice before a narrow fix PR.
