## Summary

Keep open: current main still implements Feishu streaming through CardKit interactive cards with update calls and only offers the existing `streaming: false` final plain text workaround; the requested adaptive fallback, retry behavior, or hybrid plain-text delivery mode is not implemented and needs channel product direction.

## What This Changes

The issue requests a Feishu/Lark streaming mode that avoids empty or laggy CardKit incremental updates by falling back, retrying, batching, or using a hybrid final plain-text delivery path.

## Best Possible Solution

Decide the Feishu channel contract first, then implement a narrow channel-owned streaming policy that either preserves CardKit streaming with tested fallback/retry behavior or adds an explicit final-only/hybrid delivery mode without surprising existing `streaming` users.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: source inspection confirms Feishu streaming is CardKit-card based and can display the reported empty-content fallback, but the live ZhipuAI/Feishu failure cadence has not been reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the requested fallback, granularity, and hybrid options are plausible, but maintainers should first choose whether Feishu streaming remains CardKit-first or gains an explicit final-only/hybrid mode.
