## Summary

Keep open: current main retries cumulative incremental updates and falls back when CardKit cannot start, but a failed final card content update can still leave the user with only the generic no-visible-content warning instead of the completed answer. The optional adaptive, debounce, and hybrid ideas are broader product work, but the final-delivery loss is a narrow Feishu bug that remains necessary to fix.

## What This Changes

Request reliable Feishu streaming delivery with graceful final-text fallback, plus possible adaptive, debounce, or hybrid streaming controls.

## Best Possible Solution

Keep CardKit streaming as the existing opt-in behavior, but guarantee final-answer delivery by sending the completed response through normal Feishu text delivery whenever the streaming card did not successfully display it; evaluate tuning or hybrid modes separately only with live latency evidence.

## Reproduction Assessment

Yes, at source level with high confidence: fail the final CardKit update or replacement before any content is displayed, and close() reports no visible content so the dispatcher sends its generic warning rather than the completed answer. A live Feishu run is still needed to prove the external failure trigger and the eventual fix.

## Solution Assessment

No, the whole bundle of suggested adaptive and hybrid controls is not the best first fix. The narrow maintainable solution is final-text fallback when CardKit cannot confirm visible final content, preserving existing retries, throttle behavior, and the current `streaming: false` option.
