## Summary

Keep open. Current main has adjacent timeout and recovery work, including `gateway.handshakeTimeoutMs`, channel health-monitor restart knobs, and subagent announce timeout configuration, but it still does not expose the requested gateway-level announce, connection, or plugin-init timeout surface, and the channel startup/plugin bootstrap paths still have pre-handoff awaits without a generic configurable fast-fail policy.

## What This Changes

Request to add gateway-configurable announce, connection, and plugin initialization timeouts that log structured timeout failures and avoid indefinite gateway hangs.

## Best Possible Solution

Keep the issue open and design the timeout work by boundary: add only generic core-enforceable startup/connection fast-fail behavior under the gateway config contract, keep provider-specific connection policies in provider/channel plugins, route A2A announce timeout changes through the existing sessions_send work, and add a documented plugin-loader/SDK timeout contract only if core can enforce it consistently for third-party plugins.
