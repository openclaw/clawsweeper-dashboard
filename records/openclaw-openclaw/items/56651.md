## Summary

Keep this open. Current main partially addresses the report with `agents.defaults.subagents.announceTimeoutMs`, provider request timeouts, MCP connection timeouts, channel health monitoring, and bounded channel stop waits, but it still has a hardcoded 30s `sessions_send(timeoutSeconds=0)` announce timeout and no generic plugin-startup timeout wrapper; the exact `gateway.*TimeoutMs` shape also needs maintainer product judgment.

## What This Changes

The issue requests configurable gateway announce, provider connection, and plugin initialization timeouts so stalled startup or announce phases fail with structured diagnostics instead of hanging indefinitely.

## Best Possible Solution

Keep this open for maintainer timeout-policy review: consolidate on existing agent/provider/MCP timeout surfaces where possible, route the `sessions_send` announce remainder through the related A2A issue, and add a gateway-level lifecycle timeout only if maintainers decide the startup/channel gap needs that product surface.

## Reproduction Assessment

Yes for source-level gaps: current main still sets `announceTimeoutMs` to `30_000` for `sessions_send(timeoutSeconds=0)`, and plugin startup loading is not wrapped in a timeout. No live gateway hang reproduction was run for provider websocket or plugin validation stalls.

## Solution Assessment

No for the exact proposed `gateway.*TimeoutMs` shape yet. Current code already has agent/provider/MCP-specific timeout surfaces, so maintainers should decide whether to extend those or add bounded startup internals before adding parallel gateway config.
