## Summary

Keep open. Current main still has only adjacent mitigations: configurable WebSocket handshake timeout, channel health-monitor restart knobs, and subagent announce timeout configuration, but it does not expose the requested gateway-level announce, provider-connection, or plugin-init timeout surface, and the startup/bootstrap paths still contain pre-handoff awaits without a generic fast-fail policy.

## What This Changes

The issue requests gateway-configurable announce, connection, and plugin initialization timeouts that log structured failures and prevent indefinite gateway stalls.

## Best Possible Solution

Keep this open and design the fix by boundary: add only generic core-enforceable startup or connection fast-fail behavior to the gateway contract, leave provider-specific connection policy in channel/provider plugins, keep the sessions_send announce budget in the narrower #59339 track, and add a documented plugin-loader/SDK timeout contract only if core can enforce it consistently for third-party plugins.

## Reproduction Assessment

Yes for the missing behavior by source inspection: the current schema/type/docs omit the requested gateway keys, and channel/plugin startup paths still await pre-handoff phases without a configurable timeout. I did not run a live stalled-provider reproduction because this review is read-only.

## Solution Assessment

Unclear as proposed. Configurable timeout fields are directionally useful, but the maintainable solution should split gateway-owned startup enforcement from provider/channel-owned connection behavior and from the separate A2A announce timeout path.
