## Summary

Current `main` already fixes the reported Microsoft Teams DM misrouting path by routing `findByUserId` through a DM-specific selector that ignores channel/group conversations, prefers personal conversations, and breaks ties by `lastSeenAt`. The shipped changelog entry for `v2026.3.28` matches this behavior.

## Evidence

- **DM lookup now excludes channel/group matches:** `findPreferredDmConversationByUserId` only considers `personal` conversations as first-class candidates, skips `channel` and `groupChat`, falls back only to legacy unknown-type entries, and prefers the newest `lastSeenAt` among remaining candidates.
  - file: [extensions/msteams/src/conversation-store-helpers.ts:55](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/msteams/src/conversation-store-helpers.ts#L55)
- **Both store implementations use the DM-specific lookup for `findByUserId`:** The filesystem and memory conversation stores both wire `findByUserId` to `findPreferredDmByUserId`, so proactive sends no longer use the old first-match iteration behavior described in the issue.
  - file: [extensions/msteams/src/conversation-store-fs.ts:103](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/msteams/src/conversation-store-fs.ts#L103)
- **Companion memory-store implementation matches:** The in-memory store uses the same helper and aliases `findByUserId` to the DM-preferred lookup as well.
  - file: [extensions/msteams/src/conversation-store-memory.ts:21](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/msteams/src/conversation-store-memory.ts#L21)
- **Regression tests cover the reported scenario:** Tests now assert that user lookups do not return channel or group chats, prefer personal DMs even when channel/group entries are newer, and choose the freshest personal DM when multiple DMs exist.
  - file: [extensions/msteams/src/conversation-store-helpers.test.ts:63](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/msteams/src/conversation-store-helpers.test.ts#L63)
- **End-to-end store coverage exists for repeated upserts:** Shared store tests verify both fs and memory stores prefer the freshest personal conversation for a user who also has a group conversation.
  - file: [extensions/msteams/src/conversation-store.shared.test.ts:182](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/msteams/src/conversation-store.shared.test.ts#L182)
- **Changelog shows this shipped in `2026.3.28`:** The release notes explicitly mention Microsoft Teams proactive DMs now preferring the freshest personal conversation reference for `user:<aadObjectId>` sends when multiple stored references exist.
  - file: [CHANGELOG.md:2031](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/CHANGELOG.md#L2031)

## Risks / Open Questions

- `extensions/msteams/src/monitor-handler/message-handler.ts:462` still uses `user:{senderId}` for direct messages, so this flow still depends on the conversation-store helper and its tests remaining correct.

## Close Comment

Both store backends route `findByUserId` through that helper in `extensions/msteams/src/conversation-store-fs.ts:103` and `extensions/msteams/src/conversation-store-memory.ts:21`, and the regression coverage in `extensions/msteams/src/conversation-store-helpers.test.ts:63` plus `extensions/msteams/src/conversation-store.shared.test.ts:182` covers the multi-conversation DM case from this report. This also appears in the shipped notes for `v2026.3.28` at `CHANGELOG.md:2031`.
