## Summary

Keep open: this is a focused Feishu routing bugfix candidate, but the patch adds a refreshed-config return path in the shared dynamic-agent helper while only updating the DM message caller; the Feishu Drive comment caller still reroutes only when an agent was newly created. The external PR also has only mocked/unit-test proof and should not merge until the live Feishu behavior is shown.

## What This Changes

The PR makes Feishu dynamic-agent lookup consult `runtime.config.current()` for existing bindings, re-resolves DM routes when that refreshed config is returned, and adds helper coverage for the existing-binding case.

## Best Possible Solution

Update every Feishu dynamic-agent caller that can receive a refreshed config, especially Drive comment handling, add sibling regression coverage, and require redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, source-reproducible: on the PR head the helper returns `created:false` with a refreshed config, while `handleFeishuCommentEvent` only re-resolves inside the `created` branch. I did not run tests because this review is read-only.

## Solution Assessment

No. Rechecking `runtime.config.current()` is a reasonable fix shape, but applying refreshed-config rerouting only to DM messages is too narrow because the comment flow shares the same helper contract.
