## Summary

Keep this PR open: current main does not yet add macOS Talk realtime relay, and the PR is a useful implementation candidate, but it is not merge-ready because it can silently opt macOS into relay when `talk.realtime.transport` is unset and it lacks after-fix real macOS behavior proof.

## What This Changes

The PR adds a macOS Talk realtime relay path over `talk.session.*` Gateway RPCs, parses `talk.realtime` relay settings, and keeps STT/TTS as fallback when no relay config is detected.

## Best Possible Solution

Land the macOS relay path only after it preserves STT/TTS unless `talk.realtime.transport` is explicitly `gateway-relay`, adds focused parser/runtime coverage for that upgrade boundary, and includes redacted real macOS relay proof.

## Reproduction Assessment

Yes, from source inspection: a config with `talk.realtime.mode` set to `realtime` and no `transport` reaches the new relay descriptor because the PR accepts `transport == nil`. I did not run the Swift test suite in this read-only review.

## Solution Assessment

No; the implementation is plausible, but the config gate is too broad for the existing native-client contract. The safer fix is explicit `gateway-relay` opt-in plus macOS real behavior proof.
