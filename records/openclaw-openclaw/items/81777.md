## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve current shared-client keying:** `extensions/codex/src/app-server/shared-client.ts:22`
  - body: Current main keeps a map of shared Codex clients keyed by start options, auth profile, and agentDir so different agents/auth profiles can coexist. This branch changes the state to one entry per isolation key and closes that entry when the start/auth key changes, so the default `clientIsolation=agent` path would tear down another agent's client instead of preserving current behavior. Add the isolation key to the existing cache key rather than replacing the current key map.
  - confidence: 0.9
- **[P1] Keep agentDir in Codex bindings:** `extensions/codex/src/conversation-binding-data.ts:11`
  - body: The branch removes `agentDir` from Codex conversation binding data and from `/codex bind` start params, while current main uses it to resolve the per-agent Codex auth store and later bound-turn client. Non-main agents can therefore read/write the binding and run controls with the default agent's Codex credentials or miss the intended thread. Carry `agentDir` alongside `isolationKey` and add multi-agent coverage.
  - confidence: 0.91
- **[P2] Queue bound turns by isolation key:** `extensions/codex/src/conversation-binding.ts:194`
  - body: The isolated binding/client lookup is reached only after `handleCodexConversationInboundClaim` queues by `data.sessionFile`, so two isolated topics sharing the same transcript still block behind one queue before they reach separate clients. Include `data.isolationKey` in that queue key and cover same-file/different-isolation turns.
  - confidence: 0.88
