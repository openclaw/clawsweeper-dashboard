## Summary

Keep open. The PR is useful and not implemented on current main, but the proposed session-isolation contract is still incomplete because bound conversation turns enter the per-session-file queue before reaching the isolated binding/client path.

## What This Changes

Adds opt-in `appServer.clientIsolation=session` and related Codex, gateway, session, plugin-command, config, and test plumbing to isolate Codex app-server clients and bindings per runtime-policy session/topic.

## Best Possible Solution

Keep the opt-in session-isolation design, but make every shared queue/state path use the same runtime-policy isolation key and rebase onto current main before merge.

## Reproduction Assessment

No high-confidence local runtime reproduction was established. Source inspection confirms current main shares one Codex app-server client per agent, and the PR body supplies redacted human-run Linux gateway proof for the multi-topic Telegram scenario after enabling session isolation.

## Solution Assessment

No, not yet. Per-session client and binding isolation is a reasonable direction, but bound conversation turns still queue only by transcript file before they reach the isolated client path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Queue bound turns by isolation key:** `extensions/codex/src/conversation-binding.ts:341`
  - body: The PR resolves the isolated binding/client inside `runBoundTurn`, but `handleCodexConversationInboundClaim` still enters `enqueueBoundTurn(data.sessionFile, ...)` first. Two isolated topics that share the same transcript still serialize behind one queue, so a wedged turn can block the other before it reaches its isolated client; include `data.isolationKey` in that queue key and add same-file/different-isolation coverage.
