## Summary

Keep open: the latest head is a useful, default-compatible direction with sufficient real-environment proof, but bound Codex app-server conversation turns still serialize by transcript file only, so one wedged isolation key can block another before the separate client is used.

## What This Changes

Adds opt-in Codex `appServer.clientIsolation="session"`, propagating runtime-policy session keys through Codex app-server clients, bindings, commands, compaction, reset cleanup, config, and tests.

## Best Possible Solution

Keep the opt-in, default-compatible design, but key every bound app-server turn queue by transcript plus isolation key and add same-transcript/different-isolation coverage before merge.

## Reproduction Assessment

Yes for the review blocker: source inspection shows two app-server bindings with the same `sessionFile` but different `isolationKey` still enter the same `enqueueBoundTurn(data.sessionFile, ...)` queue. I did not run a live wedged-worker repro in this read-only review.

## Solution Assessment

No, not yet. The opt-in isolation design is maintainable, but the current patch must isolate the bound-turn queue as well as the client and binding paths before it is the right fix.
