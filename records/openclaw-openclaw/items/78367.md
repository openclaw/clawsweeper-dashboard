## Summary

Keep open: current `main` and the latest release still have compaction safeguards, counters, and post-compaction loop detection, but no per-session minimum-interval or rolling-hour compaction throttle. The two production reports make this a credible runtime reliability issue, while the requested public config keys and default cancellation behavior still need runtime owner approval.

## What This Changes

Add `agents.defaults.compaction.minIntervalSeconds` and `maxPerHour` guardrails that cancel and warn when a session compacts too frequently.

## Best Possible Solution

Design and implement an owner-approved compaction-rate guard in the runtime/config contract, with schema/help/docs updates, structured warning or lifecycle signal behavior, and focused coverage for both minimum interval and hourly cap decisions.

## Reproduction Assessment

Yes, from source for the missing guardrail: current `main` has no per-session minimum interval or rolling-hour cap in the compaction safeguard, runtime state, config schema, or docs. I did not live-reproduce the full production storm.

## Solution Assessment

Unclear as a final design: a `minIntervalSeconds` plus `maxPerHour` throttle is a plausible narrow guardrail, but the default values, cancellation semantics, and monitoring event contract need maintainer/runtime approval before implementation.
