## Summary

Keep open: current main still lacks the requested per-session compaction minimum interval or rolling hourly cap, and the newer macOS report reinforces that this is still a real runtime guardrail gap. The request adds public config keys and cancellation/event policy, so it should go through runtime owner review before implementation.

## What This Changes

Add compaction rate-limit guardrails under `agents.defaults.compaction` with `minIntervalSeconds` and `maxPerHour` to bound repeated per-session auto-compactions.

## Best Possible Solution

Add an owner-approved compaction throttle in the embedded runtime/config contract, with schema/help/docs updates, structured warning or event behavior, and regression coverage for minimum interval and hourly cap decisions.

## Reproduction Assessment

Yes from source for the missing guardrail: current main has no minimum-interval or rolling-hour state in the safeguard runtime, config schema, docs, or post-compaction guard. The production storm itself was not live-reproduced here, but two independent issue reports give concrete session counts and impact.

## Solution Assessment

Unclear as a final design: a throttle is a plausible narrow guardrail, but public defaults, cancel reason wording, and lifecycle/watchdog signaling need runtime owner approval. The maintainable path should reuse the existing compaction config/schema/help patterns and preserve current compaction behavior unless the configured session budget is exceeded.
