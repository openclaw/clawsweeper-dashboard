## Summary

Keep open: current main still lacks a per-session minimum-interval or rolling-hour compaction throttle, and the issue now has two concrete production reports. The requested public config keys and default cancellation/logging behavior are plausible but need runtime owner approval before implementation.

## What This Changes

Add `agents.defaults.compaction.minIntervalSeconds` and `maxPerHour` guardrails so repeated compactions are throttled, logged, and optionally surfaced as lifecycle events.

## Best Possible Solution

Design an owner-approved compaction-rate guard in the runtime/config contract, then implement schema/help/docs updates, structured warning or lifecycle signaling, and focused coverage for both minimum interval and rolling-hour decisions.

## Reproduction Assessment

Yes, from source for the missing guardrail: current main has no per-session minimum interval or rolling-hour cap in the safeguard hook, runtime state, config schema, docs, or focused tests. I did not live-reproduce the full production compaction storm.

## Solution Assessment

Unclear as a final design: `minIntervalSeconds` plus `maxPerHour` is a plausible guardrail, but the defaults, cancellation behavior, and monitoring event contract need maintainer/runtime approval before code work should proceed.
