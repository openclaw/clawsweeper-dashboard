## Summary

Keep open: current main still lacks a per-session compaction minimum interval or rolling hourly cap, and the 2026-05-10 production report adds corroborating evidence that this can become a host-level outage. The remaining work changes public runtime config and compaction cancellation policy, so it needs runtime/product owner approval before implementation.

## What This Changes

The issue requests per-session compaction rate-limit guardrails using minimum interval and rolling hourly caps to prevent runaway compaction storms.

## Best Possible Solution

Add an owner-approved compaction throttle in the runtime/config contract, with schema/help/docs updates, structured warning or lifecycle-event behavior, and regression coverage for both minimum interval and hourly cap decisions.

## Reproduction Assessment

Yes, source-reproducible for the missing guardrail: current main has threshold/overflow compaction paths but no per-session min-interval or rolling-hour cap, and two production reports describe storms. I did not live-reproduce the full high-traffic outage scenario.

## Solution Assessment

Unclear as a final design: a throttle is a plausible narrow guardrail, but adding public defaults and canceling compactions under load needs runtime owner approval. The maintainable path is to reuse the existing config/schema/help patterns and add deterministic tests around the chosen policy.
