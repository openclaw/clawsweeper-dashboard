## Summary

Keep open: current main still lacks a per-session compaction minimum interval or rolling hourly cap, and the new macOS/Homebrew report reinforces that this is an active runtime availability problem rather than stale cleanup.

## What This Changes

The issue requests `agents.defaults.compaction.minIntervalSeconds` and `maxPerHour` guardrails to throttle runaway compaction cycles per session.

## Best Possible Solution

Add an owner-approved per-session compaction throttle in the runtime/config contract, with schema/help/docs updates, warn-level diagnostics, optional lifecycle signaling, and regression coverage for both minimum interval and rolling-hour cap behavior.

## Reproduction Assessment

Yes, source-backed: current main lacks the requested config keys and any compaction frequency state, while two production reports describe repeated successful compactions. I did not live-reproduce a high-traffic session in this read-only review.

## Solution Assessment

Unclear as final design: a per-session throttle is the right guardrail shape, but the default values, cancellation behavior, and lifecycle event contract need runtime owner approval before implementation.
