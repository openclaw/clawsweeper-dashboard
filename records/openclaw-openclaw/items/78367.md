## Summary

Keep open: current main and the latest release still lack a durable per-session compaction minimum interval or rolling-hour cap, and this issue has multiple production reports of repeated compaction causing token, transcript, checkpoint, and availability impact. Existing guards are narrower per-run or post-tool-loop defenses, so they do not fully cover the reported cross-turn compaction storm.

## What This Changes

The issue requests new `agents.defaults.compaction.minIntervalSeconds` and `maxPerHour` guardrails to throttle repeated compactions per session.

## Best Possible Solution

Design an owner-approved per-session compaction throttle that persists enough timing state to bound cross-turn storms, then align schema, docs, warning/event behavior, and focused regression coverage.

## Reproduction Assessment

Yes for the missing guardrail from source inspection: current main has no session timestamp/window state or schema keys for the requested throttle. I did not live-reproduce the full production compaction storm.

## Solution Assessment

Unclear as a final implementation plan: `minIntervalSeconds` plus `maxPerHour` is a plausible guardrail, but maintainers need to approve the public config surface, defaults, and fail-closed behavior.
