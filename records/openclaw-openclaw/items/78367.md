## Summary

Keep open: current main still has threshold and overflow compaction safeguards, but no per-session minimum-interval or rolling-hour throttle. The new macOS/Homebrew incident comment adds concrete recent evidence that this is not only the original reporter's production case.

## What This Changes

Add per-session compaction rate-limit guardrails under `agents.defaults.compaction`, including minimum interval and rolling hourly cap behavior.

## Best Possible Solution

Add an owner-approved compaction-throttle layer with schema/help/docs updates and regression coverage, while preserving current compaction behavior until a session exceeds the configured rate budget.

## Reproduction Assessment

Yes, at source level: current main has no accepted config keys or runtime state for a minimum interval or rolling hourly compaction cap. I did not live-reproduce the high-traffic production storm, but the two issue reports make the scenario credible.

## Solution Assessment

Unclear as a final design: a throttle is a plausible narrow guardrail, but public defaults, cancellation behavior, and lifecycle event shape need runtime maintainer approval. The implementation should reuse the existing compaction config/schema/runtime patterns rather than adding ad hoc state in the hot path.
