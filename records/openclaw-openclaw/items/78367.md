## Summary

Keep open: current `main` has safeguard compaction and a narrow post-compaction tool-loop detector, but no per-session minimum-interval or rolling-hour compaction throttle, so the production storm scenario is not already covered.

## What This Changes

The issue requests configurable `agents.defaults.compaction` rate limits that cancel and warn when one session compacts too frequently.

## Best Possible Solution

Add an owner-approved compaction-throttle layer in the embedded runtime/config contract, with schema/help/docs updates and regression coverage for minimum interval and hourly cap behavior.

## Reproduction Assessment

Yes from source for the missing guardrail: current compaction paths and config schema have no min-interval or rolling-hour throttle. I did not live-reproduce the production storm, so the exact high-traffic scenario remains source-backed rather than runtime-reproduced.

## Solution Assessment

Unclear as a final design: a throttle is a plausible narrow guardrail, but adding public config defaults and canceling compactions needs maintainer/runtime policy approval. A maintainable fix should reuse existing config/schema/help patterns and preserve current compaction behavior unless the configured/session rate budget is exceeded.
