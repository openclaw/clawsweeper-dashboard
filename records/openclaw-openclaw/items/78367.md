## Summary

Keep open: current main still has no per-session compaction minimum interval or rolling-hour throttle, and the newer macOS incident comment strengthens the failure-mode evidence. Existing fixes around safeguard ownership, post-compaction identical-tool loops, and repeated overflow compaction are adjacent, but they do not implement the requested rate-limit guardrail.

## What This Changes

Add a per-session compaction rate-limit guardrail under `agents.defaults.compaction`, including minimum interval, hourly cap, warning logs, and optional lifecycle visibility.

## Best Possible Solution

Add a maintainer-approved compaction throttle or circuit-breaker layer that reuses existing runtime/config patterns, updates schema/help/docs/tests, and coordinates with the related repeated-overflow reports without over-throttling normal sessions.

## Reproduction Assessment

Yes at source level: current main has no interval or hourly compaction budget in the safeguard/runtime/config path, and two users supplied detailed production storm data. I did not run a live sustained-traffic replay of the exact 89-compaction or 26-compaction incidents.

## Solution Assessment

Unclear as final design: a throttle is a plausible guardrail, but new public defaults, cancel behavior, and lifecycle event semantics need runtime/product approval. A narrower implementation should be weighed against integrating this into the existing repeated-overflow circuit-breaker work.
