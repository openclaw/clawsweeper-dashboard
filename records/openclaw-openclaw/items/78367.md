## Summary

Keep open: current main still has safeguard compaction, session compaction counts, and a post-compaction tool-loop detector, but no per-session minimum-interval or rolling-hour throttle. The follow-up incident comment strengthens the need for a runtime guardrail, while the requested config keys and default cancellation behavior still require maintainer/runtime approval.

## What This Changes

The issue requests per-session compaction throttling with `minIntervalSeconds` and `maxPerHour` settings to bound repeated auto-compaction loops.

## Best Possible Solution

Design and implement an owner-approved compaction-rate guard in the runtime/config contract, with schema/help/docs updates, structured logging or lifecycle signal behavior, and focused regression coverage for minimum interval and hourly cap decisions.

## Reproduction Assessment

Yes, from source for the missing guardrail: current main has no per-session minimum interval or rolling-hour cap in the compaction safeguard, runtime state, or public config schema. I did not live-reproduce the full production storm.

## Solution Assessment

Unclear as a final design: the requested throttle is a plausible narrow guardrail, but the default values, cancellation reason, and monitoring event semantics need maintainer/runtime policy approval before implementation.
