## Summary

Keep open: current main still lacks a per-session minimum-interval or rolling-hour compaction throttle, and the newer production comment corroborates the reported storm pattern. The requested fix adds public config keys and default cancellation policy, so it needs runtime/product owner approval before implementation.

## What This Changes

The issue requests `agents.defaults.compaction.minIntervalSeconds` and `maxPerHour` guardrails to bound repeated compactions in a single busy session.

## Best Possible Solution

Add an owner-approved compaction throttle layer that tracks per-session compaction timing/counts, exposes documented config/schema/help entries, emits structured warnings/events, and includes focused regression coverage for minimum interval and rolling-hour caps.

## Reproduction Assessment

Yes from source for the missing guardrail: current schema, runtime state, and safeguard code have no `minIntervalSeconds` or `maxPerHour` path. I did not live-reproduce the full production storm, so the incident loop remains source-backed plus reporter-observed rather than locally reproduced.

## Solution Assessment

Unclear as a final design: a throttle is a plausible narrow guardrail, but public defaults, cancellation behavior, logging, and lifecycle-event semantics need runtime owner approval. The implementation should reuse the existing compaction config/schema/help patterns and avoid changing normal compaction behavior until a configured/session rate budget is exceeded.
