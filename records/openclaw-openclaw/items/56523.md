## Label Justifications

- `impact:data-loss`: The report is about silently missing cost telemetry when accepted model cost metadata omits cache-rate fields.

## Summary

Keep open: current main and the latest release still multiply flat cost estimates by raw `cost.cacheRead`/`cost.cacheWrite`, so omitted cache rates can make `costUsd` undefined before `diagnostics-otel` records `openclaw.cost.usd`. A linked fork PR has the right source hunk, but it is open outside upstream and includes unrelated docs churn, so it does not implement the fix on main.

## What This Changes

The issue asks OpenClaw to treat missing model cost `cacheRead` and `cacheWrite` rates as zero so cost metrics are still emitted.

## Best Possible Solution

Normalize missing flat `cacheRead` and `cacheWrite` cost rates to zero in the usage-cost path, add focused regression coverage, and close this issue only after an upstream PR merges.

## Reproduction Assessment

Yes, at source level: an accepted model cost object with `input`/`output` but omitted cache rates reaches `estimateUsageCost`, where the flat branch multiplies by `undefined` and returns no finite cost. I did not run a live OTel export in this read-only review.

## Solution Assessment

The narrow defaulting change is the right fix shape, but the linked fork branch is not the best merge vehicle as-is because it also contains unrelated docs edits and lacks upstream regression coverage.
