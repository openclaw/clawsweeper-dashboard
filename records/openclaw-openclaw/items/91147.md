## Summary

This PR addresses a real P1 native hook relay process-leak bug, but it should stay open because the forced `process.exit(exitCode)` path can terminate before provider-compatible hook stdout/stderr has flushed. That can break Codex PreToolUse and PermissionRequest deny output, so the process-lifetime fix needs one more repair before merge.

## What This Changes

The PR bounds `openclaw hooks relay` stdin and overall process lifetime and adds relay timeout tests for leaked `openclaw-hooks` processes.

## Best Possible Solution

Keep the bounded stdin read and hard deadline, but flush or await provider-compatible stdout/stderr before any forced process exit so Codex receives deny/noop decisions reliably.

## Reproduction Assessment

Yes from source and contributor proof, but I did not rerun the MRE locally. Current main awaits an unbounded stdin async iterator before timeout handling, and the PR body's open-stdin MRE shows leaked `openclaw-hooks` processes on v2026.6.1.

## Solution Assessment

No. Bounding stdin and adding a hard deadline is the right direction, but the immediate forced exit is not the safest implementation because it can drop hook decision output before Codex parses it.
