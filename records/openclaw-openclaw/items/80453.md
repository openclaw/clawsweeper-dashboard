## Summary

Keep open: the reported GH_TOKEN service-env path is plausible and the supplied real behavior proof is sufficient, but the current head still broadens trusted inherited env beyond the proven credential path by allowing any service-managed key that is not OPENCLAW_* or dangerous everywhere.

## What This Changes

The PR changes durable service env parsing, trusted gateway exec env sanitization, docs, and tests so service-managed keys like GH_TOKEN can be inherited by trusted full/off gateway exec children.

## Best Possible Solution

Keep the durable GH_TOKEN path, but derive trusted exec inheritance from a narrow explicit credential-key policy and add regressions proving command-control keys remain blocked.

## Reproduction Assessment

Yes. Source inspection of current origin/main shows GH_TOKEN is filtered out of state-dir .env and stripped from inherited exec env; I did not run a live current-main gateway repro.

## Solution Assessment

No. The service-managed metadata shape is reasonable, but accepting all non-OPENCLAW, non-everywhere-dangerous managed keys is broader than the current curated inherited-env policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Constrain service-managed inheritance to credential keys:** `src/daemon/service-managed-env.ts:84`
  - body: `readOperatorInheritedEnvAllowlist` currently permits any service-managed key that is not `OPENCLAW_*` or dangerous everywhere. Because the patch also retains override-only keys from durable env, command-control entries such as `PIP_INDEX_URL` or `TF_CLI_CONFIG_FILE` can bypass the current inherited-env policy in trusted full/off exec; please restrict this to an explicit credential allowlist and add a non-secret blocked regression.
