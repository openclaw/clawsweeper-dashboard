## Summary

Keep open. The reported managed-service `GH_TOKEN` gap is real and the supplied proof is sufficient, but the PR still lets every non-`OPENCLAW_*`, non-everywhere-dangerous service-managed key bypass the inherited-env blocklist in trusted exec, which is broader than the credential use case.

## What This Changes

The PR retains override-only keys from `~/.openclaw/.env`, derives a trusted gateway exec inherited-env allowlist from `OPENCLAW_SERVICE_MANAGED_ENV_KEYS`, and adds docs plus regression tests for `GH_TOKEN` propagation.

## Best Possible Solution

Keep the managed-service `GH_TOKEN` path, but constrain trusted inheritance to an explicit credential-key policy and add regression coverage proving noncredential override-only keys stay blocked.

## Reproduction Assessment

Yes, by source inspection. Current main drops `GH_TOKEN` from durable service env and strips it from inherited host exec env; I did not run a live current-main gateway repro in this read-only review.

## Solution Assessment

No, not as written. Reusing service-managed metadata for trusted gateway exec is a reasonable shape, but the current allowlist is broader than the proven credential use case and should be narrowed before merge.
