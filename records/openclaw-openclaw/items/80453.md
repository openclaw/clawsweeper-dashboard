## Summary

Keep this PR open: the GH_TOKEN service-env fix is useful and has real proof, but the patch broadens trusted exec inheritance beyond the demonstrated token case to arbitrary service-managed override-only env keys.

## What This Changes

The PR lets selected service-managed env keys from `~/.openclaw/.env` survive gateway install/restart and reach trusted `host=gateway`, `security=full`, `ask=off` exec children, with docs and regression tests.

## Best Possible Solution

Keep the service-managed GH_TOKEN path, but constrain trusted inheritance to an explicit credential-key policy and add regression coverage proving non-secret override-only keys stay blocked.

## Reproduction Assessment

Yes. Source inspection shows current main drops `GH_TOKEN` from durable `.env` and strips inherited `GH_TOKEN` from host exec, while the supplied macOS managed-gateway screenshot shows the patched trusted exec path authenticating `gh` via `GH_TOKEN`.

## Solution Assessment

No, not as written. Reusing service-managed metadata for the trusted posture is the right shape, but the allowed key set needs to be narrower than all non-everywhere-dangerous managed env keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Constrain durable env inheritance to intended secrets:** `src/daemon/service-managed-env.ts:83`
  - body: The new trusted allowlist admits every service-managed key that is not `OPENCLAW_*` or everywhere-dangerous. Because the `.env` parser now also accepts override-only blocked keys, variables such as toolchain, package-manager, Git, Terraform, or compiler control env can bypass the existing inherited-env allowlist when placed in `~/.openclaw/.env`. Please restrict this path to the intended credential keys and add a regression that a non-secret override-only key remains blocked.
  - confidence: 0.83
