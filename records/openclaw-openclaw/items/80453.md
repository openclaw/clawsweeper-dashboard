## Summary

Keep open: the PR addresses a real managed-service env gap and includes credible after-fix proof, but the latest head still broadens trusted exec inheritance beyond the proven credential path by allowing any service-managed key that is not `OPENCLAW_*` or everywhere-dangerous.

## What This Changes

The PR changes durable service env parsing, trusted gateway exec env inheritance, docs, and regression tests so service-managed credential keys like `GH_TOKEN` can reach trusted exec children.

## Best Possible Solution

Keep the service-managed `GH_TOKEN` path, but constrain trusted inheritance to an explicit credential-key policy and add regression coverage proving noncredential override-only keys stay blocked.

## Reproduction Assessment

Yes. Source inspection of current main shows `GH_TOKEN` is dropped from durable `.env` parsing and stripped from inherited host exec env; the PR's live screenshot shows the patched managed gateway exec path authenticating `gh` through `GH_TOKEN`.

## Solution Assessment

No. Reusing service-managed metadata for a trusted gateway exec allowlist is a reasonable shape, but the current allowlist is broader than the proven credential use case and should be narrowed before merge.
