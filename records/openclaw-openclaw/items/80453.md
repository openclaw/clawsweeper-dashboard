## Summary

This PR addresses a plausible GH_TOKEN durability gap, but it is not mergeable as written because it broadens trusted exec inheritance to every service-managed key that is not OPENCLAW_* or dangerous-everywhere. The prior ClawSweeper review identified the same security-boundary issue, and no later PR update supersedes that finding.

## What This Changes

The PR lets selected keys from ~/.openclaw/.env be recorded as managed service env keys and inherited by trusted host=gateway, security=full, ask=off exec children, with docs and tests.

## Best Possible Solution

Keep the durable GH_TOKEN path, but gate trusted inheritance through an explicit credential-key policy with regression coverage for non-credential override-only keys.

## Reproduction Assessment

Yes. Source inspection shows current main drops GH_TOKEN from ~/.openclaw/.env and strips inherited GH_TOKEN from host exec, while the inspected after-fix screenshot proves the patched macOS managed gateway path can authenticate gh via GH_TOKEN.

## Solution Assessment

No, not as written. Reusing service-managed metadata for trusted exec is the right shape, but the key set needs to be narrower than all managed keys that are not OPENCLAW_* or dangerous-everywhere.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84
