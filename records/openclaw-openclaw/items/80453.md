## Summary

Keep open: the PR addresses a real service-env gap with sufficient live proof, but the current patch broadens trusted exec inheritance beyond the proven credential use case and needs a maintainer/security policy decision before merge.

## What This Changes

The PR changes durable `.env` filtering, service-managed env metadata, trusted gateway exec env sanitization, docs, and regression tests so service-managed keys such as `GH_TOKEN` can reach trusted exec children.

## Best Possible Solution

Keep the durable `GH_TOKEN` path, but constrain trusted exec inheritance through an explicit credential-key policy and regression tests proving noncredential override-only keys stay blocked.

## Reproduction Assessment

Yes. Current main source shows `~/.openclaw/.env` filters override-only keys and host exec strips inherited `GH_TOKEN`; the inspected after-fix screenshot shows the patched macOS managed gateway trusted exec path authenticating `gh` through `GH_TOKEN`.

## Solution Assessment

No. Reusing service-managed metadata for a trusted gateway exec allowlist is a reasonable shape, but the current allowlist is broader than the credential use case and bypasses the existing curated inherited-env policy for command-control variables.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Constrain trusted service env inheritance:** `src/daemon/service-managed-env.ts:84`
  - body: `readOperatorInheritedEnvAllowlist` admits every service-managed key that is not `OPENCLAW_*` or everywhere-dangerous. Because this PR also keeps override-only blocked names from `~/.openclaw/.env`, command-control variables such as `PIP_INDEX_URL` or `TF_CLI_CONFIG_FILE` can bypass the existing inherited-env policy in trusted exec. Please restrict this path to intended credential keys and add a regression for a noncredential override-only key staying blocked.
  - confidence: 0.88
