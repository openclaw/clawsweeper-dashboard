## Summary

Keep open: the PR addresses a real current-main gap for GH_TOKEN in the managed-service trusted exec path, but the branch should not merge as-is because it broadens trusted inherited env beyond the proven credential case and bypasses the curated inherited-env policy for noncredential override-only keys.

## What This Changes

The PR changes durable service env parsing, managed service key metadata, trusted gateway exec env inheritance, docs, and tests so service-managed keys like GH_TOKEN can reach trusted exec children.

## Best Possible Solution

Keep the service-managed GH_TOKEN path, but constrain trusted inheritance to an explicit credential-key policy and add regression coverage proving noncredential override-only keys stay blocked.

## Reproduction Assessment

Yes. Source inspection of current main shows GH_TOKEN is filtered out of durable service env parsing and inherited exec env, and the PR body supplies after-fix managed-gateway proof for the intended path.

## Solution Assessment

No. Reusing service-managed metadata for trusted gateway exec is a reasonable shape, but the current allowlist is broader than the proven credential use case and should be narrowed before merge.
