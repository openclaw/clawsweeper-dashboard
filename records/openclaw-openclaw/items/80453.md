## Summary

Keep this PR open: it addresses a real managed-service env gap, but the current patch broadens trusted exec inheritance beyond the proven `GH_TOKEN` use case and needs a narrowed security policy before merge.

## What This Changes

The branch keeps override-only keys from `~/.openclaw/.env` in service-managed metadata, passes `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` into trusted gateway exec env inheritance, and documents/tests the durable env behavior.

## Best Possible Solution

Keep the durable `GH_TOKEN` path, but restrict trusted service-managed inheritance to a deliberate credential-key policy and add regression coverage proving noncredential override-only keys remain blocked.

## Reproduction Assessment

Yes. Source inspection shows current main drops `GH_TOKEN` from state-dir `.env` and inherited host exec env, and the supplied macOS screenshot shows the patched trusted exec path authenticating `gh` via `GH_TOKEN`.

## Solution Assessment

No. Reusing service-managed metadata for a trusted gateway exec allowlist is a reasonable shape, but the current implementation is broader than the proven credential use case and should be narrowed before merge.

## Review Findings

Overall correctness: patch is incorrect

  - body: This adds every service-managed key that is not `OPENCLAW_*` or everywhere-dangerous to trusted exec inheritance.
