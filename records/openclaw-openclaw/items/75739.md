## Summary

Keep open. Current main fixes the Codex OAuth-profile forwarding half, but the requested mixed-provider fallback escape is still absent and now conflicts with documented fail-closed runtime semantics.

## What This Changes

The issue asks Codex harness migration to bind existing `openai-codex` OAuth profiles for canonical `openai/*` refs and to keep non-Codex model fallbacks from being forced through Codex when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Make an explicit maintainer runtime-policy decision: keep the documented fail-closed contract and steer mixed-provider users to `auto` or dedicated agents, or design a per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes for the remaining source-level behavior. The issue gives concrete config and logs, and current source confirms forced `codex` selection happens before support checks while selected plugin harness failures are not replayed through PI.

## Solution Assessment

Unclear. Current main's OAuth forwarding is the narrow maintainable fix for the auth half, but letting non-Codex fallback candidates escape a forced runtime would contradict current docs and needs maintainer product judgment.
