## Summary

Keep this issue open. Current main addresses the Codex OAuth-profile forwarding half and now documents forced Codex runtimes as fail-closed, but the requested mixed-provider fallback escape is still not implemented and needs maintainer runtime/product judgment.

## What This Changes

The issue asks for Codex-harness migration to bind `openai/*` forced Codex runs to `openai-codex` OAuth and to keep non-Codex fallback candidates from being forced through the Codex harness.

## Best Possible Solution

Keep the OAuth forwarding path, then make an explicit maintainer call: either close this as intended fail-closed semantics with the current docs, or design a per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes. The issue includes concrete config and logs, and current source confirms the remaining path: forced `codex` selection happens before support checks and selected plugin failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for missing profile binding, but changing `fallback: "none"` to let non-Codex candidates escape the forced runtime would alter the documented runtime contract.
