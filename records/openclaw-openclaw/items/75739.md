## Summary

Keep this issue open. Current main fixes the Codex OAuth-profile forwarding half and now documents forced Codex runtime as fail-closed, but the requested mixed-provider fallback escape is still not implemented and conflicts with the documented runtime contract.

## What This Changes

The issue requests that canonical `openai/*` Codex-harness runs reuse existing `openai-codex` OAuth profiles and that non-Codex fallback candidates avoid being forced through the Codex runtime when mixed providers are configured.

## Best Possible Solution

Keep the OAuth forwarding path, then make an explicit maintainer runtime decision: either close this under the documented fail-closed semantics or design per-candidate runtime routing for mixed fallback chains.

## Reproduction Assessment

Yes. The issue includes concrete config and logs, and current main source confirms the remaining path: forced `codex` selection happens before support checks and selected plugin failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for missing profile binding, but changing `fallback: "none"` to let non-Codex candidates escape the forced runtime would alter the documented fail-closed contract.
