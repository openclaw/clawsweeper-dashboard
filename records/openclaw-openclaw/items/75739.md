## Summary

Keep open. Current main fixes the Codex OAuth profile forwarding half, but the mixed-provider fallback behavior remains intentionally fail-closed for forced Codex runtimes and needs a maintainer product decision rather than cleanup closure.

## What This Changes

The issue asks the Codex harness migration path to bind existing `openai-codex` OAuth profiles for canonical `openai/*` refs and to let non-Codex fallback candidates escape a forced Codex runtime when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Maintainers should explicitly choose the runtime policy: keep forced Codex fail-closed and steer mixed-provider chains to `auto` or per-agent configs, or design and test a new per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes for the remaining source-level behavior: current selection returns a forced `codex` harness before support checks, and selected plugin harness failures are not replayed through PI. I did not run a live gateway/OAuth reproduction.

## Solution Assessment

Unclear. The current OAuth forwarding fix is the narrow maintainable fix for the auth half, but letting non-Codex fallback candidates escape a forced runtime would change the documented fail-closed semantics and needs a maintainer product call.
