## Summary

Keep this open. Current main appears to have addressed the Codex-harness OAuth-profile forwarding half, but the requested cross-provider fallback behavior still conflicts with the documented forced-runtime contract and needs maintainer/product judgment rather than cleanup closure.

## What This Changes

The issue requests that `openai/*` models forced through the Codex harness reuse existing `openai-codex` OAuth profiles, while non-Codex fallback models continue through their normal provider runtimes instead of the Codex harness.

## Best Possible Solution

Keep the main-branch OAuth-profile forwarding behavior, then have maintainers decide whether mixed provider fallbacks under a forced Codex runtime should remain explicitly unsupported/documented or get a new per-candidate runtime policy with regression coverage.

## Reproduction Assessment

Yes for the reported release behavior: the issue includes concrete `2026.4.29` config and gateway logs, and current source confirms that a forced `codex` runtime applies before provider support checks. For the OAuth half, current main has source and regression-test proof for profile forwarding rather than a live run in this review.

## Solution Assessment

Unclear. Main's OAuth forwarding is the right narrow fix for the missing `openai-codex` profile binding, but routing non-Codex fallback candidates back through PI would be a runtime-contract change rather than a small bug fix.
