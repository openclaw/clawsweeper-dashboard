## Summary

Keep open: the OAuth-profile forwarding half is covered on current main, but the fallback-runtime half remains. Current source still selects a forced `codex` plugin harness before checking provider/model support, and https://github.com/openclaw/openclaw/pull/74341 is the active but still-open fix path.

## What This Changes

The issue asks Codex-harness migration to forward existing `openai-codex` OAuth profiles for canonical `openai/*` refs and to stop routing non-Codex fallback candidates through the forced Codex runtime.

## Best Possible Solution

Update and land the targeted forced-harness selector fix so plugin runtimes consult `supports()` per fallback candidate, preserving the documented fail-closed behavior while avoiding late Codex app-server errors for unsupported providers.

## Reproduction Assessment

Yes, source-level. With `agentRuntime.id: "codex"` and a non-Codex fallback candidate after primary failure, current main selects the forced Codex harness before support checks and rethrows non-PI harness failures instead of rerouting through PI.

## Solution Assessment

Yes. The narrow maintainable fix is the direction in https://github.com/openclaw/openclaw/pull/74341: validate forced plugin harness support per candidate and fail clearly or route only under an explicit supported policy, rather than changing model fallback enumeration or auth handling.
