## Summary

Keep this issue open. Current main fixes the Codex OAuth-profile forwarding half and now documents the forced-runtime behavior, but the requested mixed-provider fallback behavior is still not implemented and would change the explicit plugin-runtime fail-closed contract.

## What This Changes

The issue requests that canonical `openai/*` Codex harness runs bind existing `openai-codex` OAuth profiles and that non-Codex fallback models avoid being forced through the Codex runtime when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Keep the current OAuth forwarding behavior, then make an explicit maintainer call on mixed fallbacks: either preserve fail-closed forced runtimes with clear docs/status/doctor guidance, or design a per-candidate runtime policy that sends only Codex-capable candidates through Codex and non-Codex candidates through PI with targeted regression coverage.

## Reproduction Assessment

Yes. The issue includes concrete `2026.4.29` config and logs, and current source confirms the remaining path: a forced registered `codex` runtime is selected before provider-support checks and selected plugin failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for missing `openai-codex` profile binding, but allowing non-Codex fallback candidates to escape a forced Codex runtime would be a contract change rather than a small bug fix.
