## Summary

Keep this issue open. Current main addresses the Codex OAuth-profile forwarding half and documents the forced-runtime behavior, but the requested mixed-provider fallback behavior is still not implemented and would change the explicit plugin-runtime fail-closed contract.

## What This Changes

The issue asks Codex-harness migration to forward existing `openai-codex` OAuth profiles for canonical `openai/*` refs and to avoid forcing non-Codex fallback models through the Codex runtime when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Keep the current OAuth forwarding behavior, then make an explicit maintainer call on mixed fallbacks: either preserve fail-closed forced runtimes with clear docs/status/doctor guidance, or design a per-candidate runtime policy that routes only Codex-capable candidates through Codex and non-Codex candidates through PI with targeted regression coverage.

## Reproduction Assessment

Yes. The issue includes concrete `2026.4.29` config and logs, and current source confirms the remaining path: a forced registered `codex` runtime is selected before provider-support checks and selected plugin failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for the missing `openai-codex` profile binding, but letting non-Codex fallback candidates escape a forced Codex runtime would be a runtime-contract change rather than a small bug fix.
