## Summary

Keep open. The OAuth half has current-main fixes and tests, and the old whole-agent `agentRuntime.fallback` shape has been superseded, but current `main` still lets an explicit Codex runtime select the Codex harness without consulting `supports()` for the fallback provider/model; https://github.com/openclaw/openclaw/pull/74341 is the active but unmerged fix path.

## What This Changes

The issue asks OpenClaw to make the Codex harness migration bind `openai-codex` OAuth for canonical `openai/*` refs and stop routing non-Codex fallback providers through the forced Codex runtime.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74341 or an equivalent current-main fix that validates explicit Codex runtime support up front and fails clearly for unsupported providers, while keeping the merged OAuth bridge fix and provider/model-scoped runtime policy.

## Reproduction Assessment

Yes, source-reproducible. Current main returns a forced Codex harness without checking provider/model support, and the Codex harness default support set excludes the non-Codex fallback providers shown in the issue logs.

## Solution Assessment

Yes for the current contract. The narrow maintainable fix is to enforce the existing harness `supports()` contract for explicit provider/model Codex runtime and fail fast with a clear error, not to revive `agentRuntime.fallback`.
