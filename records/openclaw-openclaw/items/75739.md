## Summary

Keep open. Current main fixes the Codex OAuth-profile forwarding path and now documents forced Codex runtimes as fail-closed, but the requested mixed-provider fallback escape is still absent and would change runtime semantics.

## What This Changes

The issue asks for Codex harness migration to bind existing `openai-codex` OAuth profiles for canonical `openai/*` refs and to keep non-Codex model fallbacks from being forced through the Codex harness when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Make an explicit maintainer runtime decision: either keep the documented fail-closed contract and steer mixed-provider users to `auto` or per-agent config, or design a per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes for the remaining source-level behavior. The issue includes concrete config and logs, and current source confirms forced `codex` selection happens before support checks and selected plugin harness failures are not replayed through PI.

## Solution Assessment

Unclear. Current main's OAuth forwarding is the narrow maintainable fix for the auth half, but making `fallback: "none"` let non-Codex candidates escape a forced runtime would contradict the current docs and needs a maintainer product call.
