## Summary

Keep this issue open. Current main fixes the Codex OAuth-profile forwarding half and documents forced Codex runtimes as fail-closed, but it still does not implement the requested mixed-provider fallback escape from a forced Codex runtime.

## What This Changes

The issue requests Codex-harness `openai/*` configs to reuse `openai-codex` OAuth profiles and prevent non-Codex fallback candidates from being forced through Codex when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Keep the OAuth forwarding path, then explicitly choose the runtime contract: preserve documented fail-closed Codex-only behavior or design a per-candidate policy for mixed fallback chains.

## Reproduction Assessment

Yes. The issue includes concrete config and logs, and current main still selects a forced Codex harness before support checks and rethrows plugin failures without PI replay.

## Solution Assessment

Unclear. The OAuth fix is the narrow maintainable solution for profile binding, but mixed-provider fallback under a forced plugin runtime is a broader behavior-contract change.
