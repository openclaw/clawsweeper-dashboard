## Summary

Keep this issue open. Current main fixes the Codex OAuth-profile forwarding half and now documents forced Codex runtimes as fail-closed, but it still does not implement the requested mixed-provider fallback escape from a forced Codex runtime.

## What This Changes

The issue asks Codex harness migration to forward existing `openai-codex` OAuth profiles for canonical `openai/*` refs and to keep non-Codex fallback candidates from being forced through the Codex harness when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Keep the OAuth forwarding path, then make an explicit maintainer call: preserve the documented fail-closed Codex contract, or design a per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes. The issue provides concrete config and logs, and current source confirms the remaining path: forced `codex` selection happens before support checks and selected plugin failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for missing profile binding, but changing `fallback: "none"` to let non-Codex candidates escape a forced runtime would contradict the current documented runtime contract.
