## Summary

Keep open. Current main fixes the Codex OAuth-profile forwarding half and now documents/removes `agentRuntime.fallback`, but the requested mixed-provider fallback escape is still absent and would change the explicit-runtime fail-closed contract.

## What This Changes

The issue asks OpenClaw to preserve Codex OAuth auth when using canonical `openai/*` refs with the Codex harness and to let non-Codex fallback models escape a forced Codex runtime instead of fail-closing.

## Best Possible Solution

Maintainers should either keep the documented fail-closed contract and treat `auto` or per-agent separation as the supported mixed-provider path, or explicitly design a versioned per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes for the remaining source-level behavior: current selection code forces the Codex harness before provider support checks and does not replay selected harness failures through PI. I did not run a live OAuth gateway turn.

## Solution Assessment

Unclear. The OAuth half is already handled by current auth forwarding, but making non-Codex fallback candidates escape a forced runtime would contradict current docs and needs a maintainer runtime-policy decision.
