## Summary

Keep this issue open. Current main appears to solve the Codex OAuth-profile forwarding half and now documents forced Codex runtimes as fail-closed, but the requested mixed-provider fallback escape is not implemented and needs maintainer runtime/product judgment.

## What This Changes

The issue asks Codex harness migration to forward existing `openai-codex` OAuth profiles for canonical `openai/*` models and to keep non-Codex fallback providers on their normal PI runtime when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Keep OAuth forwarding, then make one explicit contract decision: either forced Codex remains fail-closed with the current migration guidance, or OpenClaw gains a designed per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes. The issue includes concrete config and logs, and current source confirms the remaining path: forced `codex` selection happens before support checks and selected plugin harness failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for the missing profile binding, but letting non-Codex candidates escape a forced runtime would change the documented fail-closed runtime contract.
