## Summary

Keep this issue open. Current main addresses the Codex OAuth-profile forwarding half and documents the forced-runtime fail-closed behavior, but it still does not implement the requested mixed-provider fallback escape from a forced Codex runtime.

## What This Changes

Request that Codex harness migration bind existing `openai-codex` OAuth profiles for `openai/*` models and route non-Codex fallback candidates through PI when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Keep the OAuth forwarding behavior, then make an explicit maintainer call: either preserve forced plugin runtimes as fail-closed with clear docs/status guidance, or design a per-candidate runtime policy for mixed fallback chains.

## Reproduction Assessment

Yes. The issue includes concrete `2026.4.29` config and logs, and current source confirms the remaining path: forced `codex` selection happens before support checks and selected plugin failures are not replayed through PI.

## Solution Assessment

Unclear. Main's OAuth forwarding is the narrow maintainable fix for the missing profile binding, but mixed-provider fallback under a forced plugin runtime is a broader behavior contract change.
