## Summary

Keep open. Current main fixes the Codex OAuth-profile forwarding half and documents forced Codex runtimes as fail-closed, but it still does not support a native Codex primary with non-Codex PI fallbacks; that remaining behavior needs a maintainer runtime-policy decision.

## What This Changes

The issue asks for canonical `openai/*` Codex-harness configs to reuse `openai-codex` OAuth and for non-Codex fallback models to stay on their normal provider runtime when `agentRuntime.fallback` is `"none"`.

## Best Possible Solution

Make an explicit runtime-policy choice: either keep the fail-closed contract and steer mixed-provider users to separate agents or `auto`, or design a per-candidate runtime policy for Codex primary plus PI fallbacks.

## Reproduction Assessment

Yes for source-level reproduction of the remaining routing behavior. A forced `codex` runtime selects the plugin harness before support checks and selected plugin harness failures are not replayed through PI, so non-Codex fallback candidates remain trapped in the Codex harness path.

## Solution Assessment

Unclear. The current OAuth forwarding is the narrow maintainable fix for the auth half, but making non-Codex fallbacks escape a forced Codex runtime would contradict the current fail-closed docs and revive or replace a removed config policy.
