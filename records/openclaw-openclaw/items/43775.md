## Summary

Keep this open. Current main still treats a future-stamped config as warning-only during ordinary config load and snapshot reads, while hard refusal is limited to service/process mutation paths; the earlier auto-downstamp PR is closed unmerged and the merged GPT-5.4 hardening work is adjacent rather than a config downgrade-policy fix.

## What This Changes

Request a clearer recovery policy for configs whose `meta.lastTouchedVersion` is newer than the running OpenClaw version so downgrades do not silently break Discord delivery for OpenAI-Codex/GPT-5 models.

## Best Possible Solution

Define one supported newer-config downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary runtime startup and service/process mutations, document the recovery path, and add focused proof for Discord/OpenAI-Codex final-message delivery with a future-stamped config.

## Reproduction Assessment

No for the full live Discord/OpenAI-Codex delivery failure; this review did not run a live gateway against Discord and OpenAI-Codex. Yes for the source-level boundary: current main warns and continues during ordinary load/snapshot reads while only named mutation paths block.

## Solution Assessment

Unclear. The current action-scoped guard is a useful partial mitigation for split-brain service mutations, but it is not the narrow complete fix for the reported ordinary runtime failure mode; maintainers should choose the downgrade/recovery policy before implementation.
