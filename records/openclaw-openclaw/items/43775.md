## Summary

Keep this open. Current `main` still treats a newer `meta.lastTouchedVersion` as warning-only for ordinary config load and snapshot reads, while the hard block is scoped to service/process mutation paths; the adjacent GPT-5.x delivery hardening and broader downgrade issue do not fully settle this reported foreground Discord/OpenAI-Codex failure.

## What This Changes

A newer `meta.lastTouchedVersion` in `openclaw.json` is reported to silently break Discord delivery for OpenAI Codex models, with a request for hard failure, repair/downstamp, or much clearer startup remediation.

## Best Possible Solution

Define a single newer-config downgrade/recovery policy and make foreground runtime delivery either safe or explicitly blocked, with docs/doctor/runtime coverage for future-stamped configs and Discord/OpenAI-Codex final-message delivery.

## Reproduction Assessment

No for the full live Discord/OpenAI-Codex delivery failure; that was not reproduced against current `main`. Yes for the source-level boundary: ordinary load and snapshot reads still warn and continue, while hard blocking is scoped to service-mode startup, forced port cleanup, and service mutations.

## Solution Assessment

Unclear. The current action-scoped guard and per-process warning dedupe are useful partial mitigations, but they are not a complete answer to the reported foreground runtime delivery failure; maintainers should choose the downgrade/recovery policy before implementation.
