## Summary

Keep this open. Current `main` still treats a valid future-stamped config as warning-only during ordinary config load and snapshot reads, while refusal is scoped to service/process mutation paths; adjacent GPT-5.x delivery hardening and the broader open downgrade report do not fully settle this policy boundary.

## What This Changes

The issue asks OpenClaw to prevent a config written by a newer version from silently breaking Discord/OpenAI-Codex delivery.

## Best Possible Solution

Define one supported newer-config downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary startup and service/process mutations, document it, and add focused proof for Discord/OpenAI-Codex final-message delivery with a future-stamped config.

## Reproduction Assessment

No for the full live Discord/OpenAI-Codex delivery failure; this review did not run a live gateway against Discord and OpenAI-Codex. Yes for the source-level boundary: current `main` warns and continues during ordinary load/snapshot reads while only named mutation paths block.

## Solution Assessment

Unclear. The current action-scoped guard is a useful partial mitigation for split-brain service mutations, but it is not a complete fix for the reported ordinary runtime failure mode; maintainers should choose the downgrade/recovery policy before implementation.
