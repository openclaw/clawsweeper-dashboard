## Summary

Keep this open. Current main and v2026.4.27 still treat future-stamped configs as a warning for ordinary config load and snapshot reads, while hard refusal is limited to service/process mutation paths and service-mode or forced gateway startup; the closed auto-downstamp PR was not merged, and the merged GPT-5.4 hardening PR is adjacent rather than a resolution of the downgrade/recovery policy.

## What This Changes

The issue asks OpenClaw to handle `meta.lastTouchedVersion` values newer than the running binary so OpenAI-Codex Discord delivery does not silently fail after a downgrade or split-brain install.

## Best Possible Solution

Define one supported future-config downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary runtime startup as well as service/process mutations, document the recovery path, and add focused proof for a future-stamped config with Discord/OpenAI-Codex final-message delivery.

## Reproduction Assessment

Unclear for the live Discord/OpenAI-Codex final-delivery failure because this review did not run a live gateway against Discord and OpenAI-Codex. There is a high-confidence source-level reproduction of the remaining boundary: future-stamped config warns and continues during ordinary load/snapshot reads while only named mutation paths block.

## Solution Assessment

Unclear. The current action-scoped guard is a useful partial mitigation for split-brain service mutations, but it is not the narrow complete fix for the reported ordinary runtime failure mode; maintainers should choose the downgrade policy before implementation.
