## Summary

Keep this open. Current main still only warns and continues for ordinary config loads and snapshot reads with a future-stamped config; the newer-config hard block is limited to service/process mutation paths, while the auto-downstamp PR is closed unmerged and the merged GPT-5.4 hardening work is adjacent rather than a downgrade-policy fix.

## What This Changes

The issue asks OpenClaw to prevent or clearly remediate silent Discord delivery failures when `openclaw.json` was last written by a newer OpenClaw version.

## Best Possible Solution

Define one supported newer-config downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary runtime startup and service/process mutations, document the recovery path, and add focused Discord/OpenAI-Codex delivery proof.

## Reproduction Assessment

No for the full live Discord/OpenAI-Codex delivery failure; this review did not run a live gateway against Discord and OpenAI-Codex. Yes for the source-level boundary: current main warns and continues during ordinary load/snapshot reads while only named mutation paths block.

## Solution Assessment

Unclear. The current action-scoped guard is a useful partial mitigation for split-brain service mutations, but it is not the narrow complete fix for the reported ordinary runtime failure mode; maintainers should choose the downgrade/recovery policy before implementation.
