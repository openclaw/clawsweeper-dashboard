## Summary

Keep open: current main still has the reported compaction-retry delivery race. The runner only awaits the block-reply flush before the compaction retry wait, while lifecycle resolves the retry wait before the retry-generated async flush has completed; existing tests still assert that ordering.

## What This Changes

The issue requests a retry-safe delivery barrier so replies generated during auto-compaction session rollover are flushed to messaging channels before the attempt completes.

## Best Possible Solution

Add a retry-specific post-wait `onBlockReplyFlush()` or equivalent delivery barrier in the runner path, with regression coverage proving a compaction retry cannot finish the attempt while retry-generated block replies are still draining, while preserving #59308's slow-channel compaction-wait protection.

## Reproduction Assessment

Yes. A focused unit test can keep `onBlockReplyFlush()` pending while `pendingCompactionRetry` resolves, then assert the runner currently proceeds after `waitForCompactionRetryWithAggregateTimeout()` without awaiting the retry-generated delivery barrier.

## Solution Assessment

Yes. The narrowest maintainable fix is a post-retry-wait flush/barrier in the runner path; changing general lifecycle ordering is riskier because current tests intentionally protect compaction wait from slow channel flushes.

## Repair Work Prompt

Fix the compaction-retry reply delivery race from https://github.com/openclaw/openclaw/issues/47335. Current main flushes block replies before `waitForCompactionRetryWithAggregateTimeout()` but does not await a retry-generated flush after that wait resolves, while lifecycle resolves pending compaction retry before async channel flush completion. Add a retry-specific post-wait `onBlockReplyFlush()` or equivalent delivery barrier in `src/agents/pi-embedded-runner/run/attempt.ts` so the attempt cannot finish while retry-generated block replies are still draining. Preserve #59308's intent that ordinary slow channel flushes do not block compaction wait resolution. Use #47338 as source context but make the fix fit current main. Add focused regression coverage that holds `onBlockReplyFlush` pending while compaction retry resolves and proves the runner waits for the delivery barrier. Do not broaden this into a general final-payload suppression rewrite; coordinate with #62157 if that work has landed.
