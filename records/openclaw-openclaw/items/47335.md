## Summary

Keep open. Current main still has the reported compaction-retry delivery race: the runner waits for compaction retry completion after only a pre-wait block flush, while lifecycle retry resolution can happen before the retry-generated block/channel flush finishes.

## What This Changes

The issue asks OpenClaw to deliver retry-generated replies when session compaction rolls over a turn instead of recording the reply in the transcript but losing it before channel delivery.

## Best Possible Solution

Add a retry-specific delivery barrier so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, while preserving the ordinary slow-channel ordering from https://github.com/openclaw/openclaw/pull/59308.

## Reproduction Assessment

Yes, at source level: set a pending compaction retry, make `onBlockReplyFlush` asynchronous, and `handleAgentEnd` resolves the compaction wait before that flush settles; the runner has no post-wait flush before continuing. I did not run a live high-context channel reproduction in this read-only sweep.

## Solution Assessment

Yes, a retry-specific post-wait flush or equivalent ordering guarantee is the narrowest maintainable fix. Awaiting all lifecycle/channel flushing before resolving compaction wait would be riskier because it conflicts with the intent of https://github.com/openclaw/openclaw/pull/59308.

## Repair Work Prompt

Fix the compaction-retry reply delivery race from https://github.com/openclaw/openclaw/issues/47335. Current main flushes the block reply pipeline before `waitForCompactionRetryWithAggregateTimeout()`, but retry-generated replies can still be draining when the wait resolves because `handleAgentEnd()` resolves compaction retry before async `onBlockReplyFlush` completes. Add a retry-specific ordering guarantee, likely in `src/agents/pi-embedded-runner/run/attempt.ts` after the retry wait or through an equivalent narrow barrier, so the attempt cannot finish before retry-generated block replies are delivered. Preserve the broader intent of https://github.com/openclaw/openclaw/pull/59308: ordinary slow end-of-run channel flushes should not generally block compaction completion. Add or update focused regression coverage for the retry path and keep the change out of channel-specific code. Add a `CHANGELOG.md` fix entry if the repair changes user-visible reply reliability.
