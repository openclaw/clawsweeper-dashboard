## Summary

Current main still has the source-level compaction-retry delivery race: the runner drains block replies before the retry wait, but not after it, while lifecycle code can resolve the retry wait before the async block/channel flush settles. The issue remains actionable and should stay open for a narrow fix.

## What This Changes

The issue requests a compaction-retry delivery barrier so replies generated during auto-compaction retry are delivered before the attempt can finish.

## Best Possible Solution

Add a retry-specific post-wait drain or equivalent delivery barrier so retry-generated block replies flush before attempt completion, with focused regression coverage around async `onBlockReplyFlush` ordering.

## Reproduction Assessment

Yes, at source level: set a pending compaction retry, make `onBlockReplyFlush` asynchronous, and `handleAgentEnd` can resolve the retry wait before that flush settles while the runner has no post-wait drain. I did not establish a live high-context channel reproduction in this read-only review.

## Solution Assessment

Yes. A retry-specific post-wait drain or equivalent delivery barrier is the narrowest maintainable fix; broadly awaiting all lifecycle/channel flushing before compaction wait resolution would risk regressing https://github.com/openclaw/openclaw/pull/59308.
