## Summary

Keep open: current main still has the compaction-retry delivery race. The prior ClawSweeper reviews remain accurate at bd6035d: the runner drains block replies only before the retry wait, lifecycle resolves the retry wait before awaiting the retry-generated channel flush, and current tests still protect that ordering.

## What This Changes

The issue requests that compaction-retry replies be delivered by adding a delivery barrier after the compaction retry wait resolves.

## Best Possible Solution

Add a retry-specific post-wait `onBlockReplyFlush()` or equivalent delivery barrier so a compaction retry cannot finish the attempt while retry-generated block replies are still draining.

## Reproduction Assessment

Yes. A focused unit test can keep `onBlockReplyFlush()` pending while a pending compaction retry resolves, then assert current main allows the runner to continue after `waitForCompactionRetryWithAggregateTimeout()` without awaiting that retry-generated delivery barrier.

## Solution Assessment

Yes. The narrowest maintainable direction is a runner-side post-retry-wait flush/barrier; changing general lifecycle resolution order is riskier because current tests intentionally protect slow-channel compaction wait behavior.
