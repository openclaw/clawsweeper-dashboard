## Summary

Current main still lacks the retry-specific delivery barrier, and this issue is now paired with an open closing PR that implements the narrow post-wait block-reply flush. Keep this issue open until https://github.com/openclaw/openclaw/pull/85288 lands or is replaced.

## What This Changes

The issue reports that compaction retry rollover can generate a transcript-visible reply that is not delivered to Discord, Telegram, or another channel.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85288 if it remains green; if it closes unmerged, replace it with the same narrow post-wait flush and regression test.

## Reproduction Assessment

Yes, at source level: pending retry state plus an async `onBlockReplyFlush` lets `handleAgentEnd()` resolve the compaction wait before delivery flush settles, and current main has no post-wait drain. I did not run a live high-context channel reproduction.

## Solution Assessment

Yes. The linked PR’s idempotent post-wait flush after a successful retry wait is the narrow maintainable fix, while awaiting all lifecycle/channel flushes before compaction resolution would risk undoing https://github.com/openclaw/openclaw/pull/59308.
