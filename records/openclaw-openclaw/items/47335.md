## Summary

Keep open. Current main still has the source-level compaction-retry delivery race: the runner drains before waiting for compaction retry completion, while lifecycle resolution can unblock that wait before retry-generated block/channel flushing finishes.

## What This Changes

Ensure compaction retry/session rollover drains retry-generated reply blocks before attempt cleanup so generated replies are delivered to messaging channels.

## Best Possible Solution

Land a narrow regression-backed compaction-retry delivery barrier that drains retry-generated block replies before the attempt can finish.

## Reproduction Assessment

Yes, at source level: set a pending compaction retry, make `onBlockReplyFlush` asynchronous, and `handleAgentEnd` can resolve the compaction wait before that flush settles while the runner has no post-wait drain. I did not run a live high-context channel reproduction in this read-only sweep.

## Solution Assessment

Yes. A retry-specific post-wait flush or equivalent delivery barrier is the narrow maintainable fix; awaiting every lifecycle/channel flush before resolving compaction wait would risk undoing https://github.com/openclaw/openclaw/pull/59308.
