## Summary

Current main still has the source-level compaction-retry delivery race: retry wait can resolve before the retry's async block/channel flush settles, and the runner has no post-wait delivery barrier. The issue is still actionable and should not be closed as stale or superseded.

## What This Changes

The issue asks for retry-generated block replies to be drained before an auto-compaction rollover attempt can finish and suppress final payload delivery.

## Best Possible Solution

Land a narrow regression-backed compaction-retry delivery barrier that drains retry-generated block replies before the attempt can finish, while preserving the existing slow-channel protection for ordinary compaction completion.

## Reproduction Assessment

Yes, at source level: set a pending compaction retry, make `onBlockReplyFlush` asynchronous, and `handleAgentEnd` can resolve the retry wait before that flush settles while the runner has no post-wait drain. I did not establish a live high-context channel reproduction in this read-only review.

## Solution Assessment

Yes. A retry-specific post-wait flush or equivalent delivery barrier is the narrow maintainable fix; awaiting all lifecycle/channel flushing before resolving compaction wait would risk undoing the intentional ordering from https://github.com/openclaw/openclaw/pull/59308.
