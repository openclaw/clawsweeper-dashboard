- `impact:message-loss`: The reported failure mode is a generated assistant reply that never reaches the messaging channel.
- `impact:session-state`: The transcript can record a reply that the user never received, causing channel and session state drift.

## Summary

Keep this open: current main and the latest release still have the compaction-retry delivery ordering race, and the open closing PR now owns the narrow implementation path.

## What This Changes

The issue reports generated replies being recorded in the transcript but not delivered when compaction retry session rollover resolves before the block reply pipeline drains.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85288 or an equivalent narrow post-wait delivery barrier that skips the extra drain on aggregate timeout.

## Reproduction Assessment

Yes, at source level: pending compaction retry plus an asynchronous `onBlockReplyFlush()` lets `handleAgentEnd()` resolve the retry wait before delivery flush settles, while the runner has no post-wait drain. I did not run a live high-context channel reproduction in this read-only review.

## Solution Assessment

Yes. A second idempotent drain after a successful compaction retry wait is the narrow maintainable fix, while moving all lifecycle/channel flushes before retry resolution would risk undoing https://github.com/openclaw/openclaw/pull/59308's slow-channel protection.
