## Summary

Keep open: current main still allows the compaction retry wait to resolve before the retry-generated block reply delivery flush is awaited, and the runner has no post-wait delivery barrier.

## What This Changes

The issue reports that replies generated during overflow compaction retry can be recorded in the transcript but never delivered to messaging channels.

## Best Possible Solution

Add a retry-specific delivery barrier so a compaction retry cannot let the attempt finish while retry-generated block replies are still draining, while preserving the #59308 behavior for ordinary slow channel flushes.

## Reproduction Assessment

Yes, source-level. A focused unit test can keep `onBlockReplyFlush()` pending, resolve a pending compaction retry through `handleAgentEnd()`, and assert current `runEmbeddedAttempt` can continue after `waitForCompactionRetryWithAggregateTimeout()` without awaiting the retry-generated flush.

## Solution Assessment

Yes. The narrowest maintainable fix is a runner-side post-retry-wait flush or equivalent retry-specific barrier; broadly changing lifecycle resolution order would conflict with tests that intentionally keep compaction wait independent from slow channel flushes.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/47335 as a narrow runtime reliability bug. Current main flushes `onBlockReplyFlush()` before `waitForCompactionRetryWithAggregateTimeout()` but does not await a delivery barrier after a compaction retry resolves; `handleAgentEnd()` can resolve `pendingCompactionRetry` before async channel/block flush completion. Add a retry-specific post-wait `onBlockReplyFlush()` or equivalent delivery barrier so retry-generated block replies cannot be lost before the attempt completes. Preserve #59308's intent that ordinary slow end-of-run channel flushes do not block compaction wait resolution, and avoid broad final-payload dedupe rewrites unless required for the regression. Add focused regression coverage that holds `onBlockReplyFlush()` pending while a compaction retry resolves, plus any necessary lifecycle/payload assertions. Add a one-line user-facing `CHANGELOG.md` fix entry. Do not change channel-specific behavior, compaction protocol semantics, or dependency versions.
