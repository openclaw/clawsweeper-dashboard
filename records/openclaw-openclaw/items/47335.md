- `impact:message-loss`: The reported failure mode is a generated assistant reply that never reaches the messaging channel.
- `impact:session-state`: The transcript can record a reply the user never received, causing session/channel state drift.

## Summary

Current main still has the source-level compaction-retry delivery race: the runner drains block replies before the compaction retry wait, but not after it, while lifecycle code resolves the retry wait before the async block/channel flush finishes. The closed unmerged fix PR and the related open block-streaming PR do not supersede this narrow ordering bug, so this should stay open as a focused repair candidate.

## What This Changes

The issue reports a compaction retry race where a generated reply can be recorded in the session transcript but not delivered to the messaging channel after session rollover.

## Best Possible Solution

Add a retry-specific post-wait drain or equivalent delivery barrier so retry-generated block replies flush before attempt completion, while preserving the fast compaction-wait resolution for ordinary slow channel drains.

## Reproduction Assessment

Yes, at source level: set pending compaction retry state and make `onBlockReplyFlush` asynchronous, then `handleAgentEnd()` can resolve the retry wait before the delivery flush settles while the runner has no post-wait drain. I did not establish a live high-context channel reproduction in this read-only review.

## Solution Assessment

Yes. A retry-specific post-wait drain or equivalent barrier is the narrow maintainable fix; broadly moving all lifecycle/channel flushes before compaction resolution would risk regressing the slow-channel timeout behavior fixed by https://github.com/openclaw/openclaw/pull/59308.
