- `P2`: The issue is a focused but user-visible reliability bug in the compaction retry reply-delivery path.
- `impact:message-loss`: The reported failure mode is a generated assistant reply that never reaches the messaging channel.
- `impact:session-state`: The transcript can record a reply the user never received, causing channel and session state drift.

## Summary

Current main still has the source-level compaction-retry delivery race: the runner drains block replies before the retry wait but not after it, while lifecycle code resolves the retry wait before the async block/channel flush settles. The related closed source PR is unmerged, and the open final-payload filtering PR does not replace this retry-specific delivery barrier.

## What This Changes

The issue reports generated assistant replies being recorded in the transcript but not delivered to messaging channels when compaction retry rolls the session over.

## Best Possible Solution

Add a retry-specific post-wait drain or equivalent delivery barrier so retry-generated block replies flush before attempt completion, while preserving fast compaction-wait resolution for ordinary slow channel drains.

## Reproduction Assessment

Yes, at source level: set pending compaction retry state and make `onBlockReplyFlush` asynchronous, then `handleAgentEnd()` can resolve the retry wait before delivery flush settles while the runner has no post-wait drain. I did not establish a live high-context channel reproduction in this read-only review.

## Solution Assessment

Yes. A retry-specific post-wait drain or equivalent barrier is the narrow maintainable fix; broadly moving all lifecycle/channel flushes before compaction resolution would risk regressing the slow-channel timeout behavior preserved by https://github.com/openclaw/openclaw/pull/59308.
