## Label Justifications

- `P2`: The issue is a narrow but user-visible reliability bug in the compaction retry reply-delivery path.
- `impact:message-loss`: The reported failure mode is a generated assistant reply that never reaches the messaging channel.
- `impact:session-state`: The transcript can record a reply the user never received, causing session/channel state drift.

## Summary

Current main still has the source-level compaction-retry delivery race: the runner flushes block replies before the retry wait, but not after it, while lifecycle code resolves the retry wait before the async block/channel drain settles. This remains a narrow, actionable reliability bug rather than a close candidate.

## What This Changes

The issue asks OpenClaw to ensure retry-generated replies are delivered when compaction triggers a session rollover.

## Best Possible Solution

Add a retry-specific post-wait drain or equivalent delivery barrier so retry-generated block replies flush before attempt completion, while preserving the existing fast compaction-wait resolution for ordinary slow channel drains.

## Reproduction Assessment

Yes, at source level: set pending compaction retry state and make `onBlockReplyFlush` asynchronous, then `handleAgentEnd()` can resolve the retry wait before the delivery flush settles while the runner has no post-wait drain. I did not establish a live high-context channel reproduction in this read-only review.

## Solution Assessment

Yes. A retry-specific post-wait drain or equivalent barrier is the narrow maintainable fix; broadly moving all lifecycle/channel flushes back before compaction resolution would risk undoing the slow-channel timeout fix from https://github.com/openclaw/openclaw/pull/59308.
