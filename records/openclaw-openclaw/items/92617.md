## Label Justifications

- `P1`: The paired report describes a currently broken WhatsApp plugin-update workflow that can destroy a linked session and force QR re-pairing.
- `merge-risk: 🚨 compatibility`: The PR changes shipped WhatsApp credential cleanup/update behavior in an upgrade-sensitive path and can still leave existing users with destructive cleanup.
- `merge-risk: 🚨 session-state`: The new backup/restore mode can fail to preserve active session state, delete it on backup failure, or resurrect stale session state from sibling backup directories.
- `merge-risk: 🚨 security-boundary`: WhatsApp credential directories contain sensitive authentication material, and the PR retains and restores them without a clear logout or cleanup policy.

## Summary

Keep this PR open, but it is not merge-ready: the current head adds backup/restore helpers without wiring backup creation into the real production cleanup path, still deletes active credentials when backup creation fails, and lacks a lifecycle policy for sensitive restorable backups.

## What This Changes

The PR adds a WhatsApp auth-directory backup/restore helper, exports it, calls restore at monitor startup, and updates logout/auth-store tests while leaving default logout destructive.

## Best Possible Solution

Preserve WhatsApp credentials at the actual plugin-update or reconnect cleanup boundary with fail-closed backup handling, logout invalidation/cleanup for backups, and redacted live update proof before merge.

## Reproduction Assessment

No for the original live report: I did not reproduce a real linked WhatsApp plugin update. Yes at source level for the PR blockers: current head has no production backupOnClear caller, falls back to deletion on backup failure, and restores matching sibling backups without lifecycle checks.

## Solution Assessment

No. The patch is a plausible mitigation shape, but it is not the best fix because startup restore is added without creating backups on the production cleanup path and without fail-closed or logout-invalidation semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Create backups on the actual preservation path:** `extensions/whatsapp/src/auto-reply/monitor.ts:222`
  - body: `restoreLatestBackup()` can only consume an existing `.bak` directory, but this head never passes `backupOnClear:true` from production code; `clearTerminalWebAuthState` and the logged-out setup path still call destructive `logoutWeb`. With no real backup created during the reported plugin-update cleanup, the session wipe remains.
  - confidence: 0.93
- **[P1] Do not delete credentials when backup creation fails:** `extensions/whatsapp/src/auth-store.ts:430`
  - body: `backupOnClear` is supposed to preserve a linked session, but if `fs.rename` fails this fallback deletes the active auth directory. A timestamp collision, permission failure, or cross-device rename would recreate the exact QR re-pair failure this PR is trying to prevent; leave the active directory intact or abort cleanup instead.
  - confidence: 0.9
- **[P1] Invalidate restorable backups on logout:** `extensions/whatsapp/src/auth-store.ts:379-380`
  - body: `restoreLatestBackup` restores any sibling `authDir.bak.*` when active creds are missing, but the backups live outside the active auth directory and are not invalidated by explicit logout or cleanup. A stale backup can therefore relink sensitive WhatsApp session material after an operator tries to clear the account.
  - confidence: 0.87
