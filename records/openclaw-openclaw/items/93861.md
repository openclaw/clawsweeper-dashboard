## Summary

Keep open: the PR targets a real foreground-fence bug, but this branch is not merge-ready because the active-generation guard uses whole-dispatch lifetime as freshness, breaks existing stale-suppression assertions in CI, and still lacks real channel proof for overlapping visible replies.

## What This Changes

The PR changes the foreground reply cancellation predicate so a newer visible generation only cancels an older generation after the older generation leaves the active set, and updates the matching freshness test expectation.

## Best Possible Solution

Use a delivery-phase or lazy foreground fence that prevents queued turns from truncating active delivery while preserving newer-visible stale suppression, then prove it with focused freshness tests and redacted live channel evidence.

## Reproduction Assessment

Yes at source level for the PR defect: current tests encode stale suppression after newer visible delivery, and CI on this PR head fails five freshness assertions where older output or fallback delivery becomes visible. I did not run a live channel replay in this read-only review.

## Solution Assessment

No. Guarding on `activeGenerations` uses whole dispatch lifetime as freshness; a delivery-phase or lazy fence is the narrower maintainable fix because it can avoid premature truncation without replaying stale older output.
