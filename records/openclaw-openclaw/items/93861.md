## Summary

The PR should stay open but is not merge-ready: the one-line active-generation guard solves the reported truncation shape only by treating whole dispatch lifetime as freshness, and CI shows it breaks existing stale-suppression coverage for visible-winner delivery. Current main still has the underlying foreground-fence bug, but linked lazy-fence candidates are open and this branch needs repair plus real behavior proof rather than cleanup closure.

## What This Changes

The PR changes `shouldCancelForegroundReplyDelivery` to allow older foreground replies while their generation remains active and rewrites one freshness test to expect the older final to deliver after newer settled visible delivery.

## Best Possible Solution

Use a delivery-phase or lazy foreground fence that prevents queued turns from truncating active delivery while preserving newer-visible stale suppression, coordinating with https://github.com/openclaw/openclaw/pull/91963 or https://github.com/openclaw/openclaw/pull/93977 if maintainers choose that path.

## Reproduction Assessment

Yes, at source level for the PR defect: current tests encode stale suppression after newer visible delivery, and the PR head fails those assertions under CI. I did not run live channel proof in this read-only review.

## Solution Assessment

No. Guarding on `activeGenerations` uses whole dispatch lifetime as freshness; a delivery-phase or lazy fence is the narrower maintainable fix because it can avoid premature truncation without replaying stale older output.
