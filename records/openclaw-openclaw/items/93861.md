## Summary

Keep open: this is a plausible bug-fix PR, but it is not merge-ready because the active-generation guard can let stale older replies or settled fallback deliver after a newer visible reply, and the PR head fails the existing freshness invariant in CI.

## What This Changes

This PR changes the auto-reply foreground fence predicate so a newer visible generation only cancels an older generation after it leaves the active set, and updates one freshness test expectation.

## Best Possible Solution

Use a delivery-phase or lazy foreground fence, or equivalent ordering state, that prevents active reply truncation while preserving newer-visible stale suppression, then prove it with focused freshness tests and redacted real channel evidence.

## Reproduction Assessment

Yes, at source level. Current tests encode newer-visible stale suppression, and the PR head failed five focused freshness assertions in CI; I did not run local or live channel proof in this read-only review.

## Solution Assessment

No. Using dispatch-lifetime `activeGenerations` is too broad; a delivery-phase or lazy fence is the narrower maintainable fix because it can prevent truncation without replaying stale older output.
