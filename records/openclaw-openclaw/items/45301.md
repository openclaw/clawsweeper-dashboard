## Summary

Keep open. Current main still lets Feishu wiki calls receive precision-lost numeric `space_id` values and forward them into Lark requests, while the schema and bundled skill do not warn models to keep long numeric-looking IDs quoted.

## What This Changes

The issue requests that Feishu/Lark wiki `space_id` values remain exact when passed to `feishu_wiki` create/list/move-style tools.

## Best Possible Solution

Feishu wiki IDs should be treated as opaque strings end to end, with precision-preserving parsing where raw stringified tool args are available, Feishu-local rejection of number-typed IDs, and clearer schema/skill wording.

## Reproduction Assessment

Yes. A local JSON parse probe reproduces the exact suffix mutation, and current-main source inspection shows the rounded value can be forwarded through the Feishu wiki execute path into Lark request fields.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve or reject unsafe numeric-looking IDs at the tool boundary and strengthen Feishu wiki model guidance, rather than silently accepting rounded numbers or adding a user-facing config knob.
