## Summary

Keep open. Current main still allows the Feishu wiki tool to receive already-rounded numeric `space_id` values and forward them to the Lark client, while the schema and skill guidance do not warn models clearly enough to keep long numeric-looking IDs quoted.

## What This Changes

The issue asks OpenClaw to preserve Feishu/Lark knowledge-base `space_id` values when wiki/doc tools create or move nodes.

## Best Possible Solution

Feishu wiki tools should treat wiki space IDs as opaque strings end to end, reject number-typed `space_id`/`target_space_id` inputs with precision-loss guidance, and add focused regression coverage plus schema/skill wording.

## Reproduction Assessment

Yes. A source-level probe reproduces the exact rounding pattern, and current main forwards those rounded values through the Feishu wiki execute path into Lark request fields.

## Solution Assessment

Yes. A Feishu-owned guard plus schema/skill guidance is the narrow maintainable fix; a global big-integer parser would be broader and still cannot restore digits after rounding has already happened.
