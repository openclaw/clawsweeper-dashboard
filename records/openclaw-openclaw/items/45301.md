## Summary

Keep open. Current main still has a clear precision-loss path where numeric-looking Feishu wiki `space_id` values can be rounded by JSON parsing and then forwarded into Lark wiki requests without a Feishu-local guard or stronger model guidance.

## What This Changes

Ensure Feishu/Lark wiki space IDs remain exact opaque strings when `space_id` and `target_space_id` are received and forwarded by Feishu wiki tools.

## Best Possible Solution

Feishu wiki IDs should be treated as opaque strings end to end, with Feishu-local validation that rejects number-typed unsafe IDs and schema/skill wording that tells models to keep long numeric-looking IDs quoted.

## Reproduction Assessment

Yes. A local JSON parse probe reproduces the exact reported suffix mutation, and current-main source inspection shows the rounded value can flow into Feishu wiki Lark request fields.

## Solution Assessment

Yes. The narrow maintainable repair is to fail closed or preserve exact strings at the Feishu tool boundary and strengthen model guidance, without adding a config option or changing product policy.
