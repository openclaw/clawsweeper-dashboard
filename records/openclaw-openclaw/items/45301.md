## Summary

Keep open. Current main still accepts precision-lost numeric Feishu/Lark wiki space IDs at the tool boundary and forwards them into SDK calls, so the reported regression remains a valid bundled-plugin bug.

## What This Changes

The issue asks OpenClaw to stop Feishu/Lark wiki tools from mutating long numeric-looking `space_id` values before Lark receives them.

## Best Possible Solution

Feishu wiki IDs should be treated as opaque strings end to end, with Feishu-local rejection of unsafe number-typed IDs and clearer schema/skill wording that tells models to quote long numeric-looking IDs.

## Reproduction Assessment

Yes. A focused JSON parse probe reproduces the exact suffix mutation, and current-main source inspection shows the rounded value can be forwarded through the Feishu wiki execute path into Lark request fields.

## Solution Assessment

Yes. The narrow maintainable repair is to preserve or reject unsafe numeric-looking IDs at the Feishu tool boundary and strengthen Feishu wiki model guidance, rather than adding a setting or accepting rounded numbers silently.

Fix the Feishu/Lark wiki `space_id` precision-loss bug from https://github.com/openclaw/openclaw/issues/45301. Treat Feishu wiki IDs as opaque strings: reject number-typed unsafe integer IDs in the Feishu wiki tool boundary, preserve exact string IDs wherever raw stringified tool args are available, and improve `extensions/feishu/src/wiki-schema.ts` plus `extensions/feishu/skills/feishu-wiki/SKILL.md` so models are explicitly told to pass long numeric-looking IDs as quoted strings.
