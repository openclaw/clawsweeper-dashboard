## Summary

Keep open. Current main still has a source-reproducible Feishu wiki precision-loss path where long numeric-looking `space_id` values can be rounded before the plugin forwards them to Lark, and no current close reason applies.

## What This Changes

The issue reports Feishu/Lark wiki tools altering user-provided `space_id` values when long knowledge-base IDs are emitted as unquoted JSON numbers.

## Best Possible Solution

Treat Feishu wiki IDs as opaque strings end to end: strengthen model guidance, reject unsafe number-typed `space_id` inputs at the Feishu tool boundary, and cover the behavior with a focused regression test.

## Reproduction Assessment

Yes, at source level. A focused JSON parse probe reproduces the exact suffix mutation, and current-main source shows rounded values can flow into Feishu wiki Lark request fields without validation.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local validation plus stronger schema and skill guidance; a broader shared parser change should be attempted only if it can preserve raw arguments without changing provider behavior unexpectedly.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/45301. Feishu wiki `space_id` and `target_space_id` are opaque 19+ digit IDs that must stay strings; when a model emits them as bare JSON numbers, JavaScript rounds them before Lark receives the request. Add a Feishu-local guard in `extensions/feishu/src/wiki.ts` so wiki ID fields are accepted only as strings and unsafe number-typed values fail closed with clear retry guidance telling the model/user to quote the ID. Strengthen `extensions/feishu/src/wiki-schema.ts` descriptions and `extensions/feishu/skills/feishu-wiki/SKILL.md` examples/instructions so `space_id` and `target_space_id` are explicitly described as string IDs that must not be converted to numbers. Add focused regression coverage, preferably a new `extensions/feishu/src/wiki.test.ts`, and keep existing account-routing behavior intact. Do not add a config option or broad product policy. If you touch shared tool-call parsing, preserve existing provider behavior and add focused shared parser coverage. Add a user-facing changelog fix entry.
