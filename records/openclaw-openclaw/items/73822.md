## Summary

Keep open. The SecretRef feature is useful and not implemented on current main, but this PR should not merge while it lets a SecretRef-only allowlist satisfy the public-open wildcard guard.

## What This Changes

The PR widens phone-number and sender config fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call to accept SecretRef inputs, with docs, UI hints, secret contracts, and tests.

## Best Possible Solution

Keep the SecretRef feature, but require a literal "*" for dmPolicy="open" while letting SecretRefs satisfy non-empty allowlist validation for allowlist policy.

## Reproduction Assessment

Yes, at source level. On the PR head, requireOpenAllowFrom with policy="open" and allowFrom containing only a valid SecretRef returns no issue because of params.allowFrom?.some(isSecretRef), while current main and docs require a literal "*".

## Solution Assessment

No. The feature direction is maintainable, but the current implementation is not the best fix because open-policy validation should preserve the visible wildcard opt-in.
