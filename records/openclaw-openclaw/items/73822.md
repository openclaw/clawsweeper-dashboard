## Summary

Keep this PR open: the SecretRef feature is still useful and not implemented on current main, but the current patch weakens the public-open DM guard and still lacks real gateway/secret-provider proof.

## What This Changes

The PR adds SecretRef support for phone-number and sender-identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call, with schema, secret-contract, docs, metadata, UI hints, changelog, and tests.

## Best Possible Solution

Keep the SecretRef feature direction, but require a literal "*" for dmPolicy="open", allow SecretRefs only to satisfy non-empty allowlist validation, update the tests, and add real gateway/secret-provider proof before merge.

## Reproduction Assessment

Yes. A source-level reproduction is to parse dmPolicy="open" with allowFrom containing only a structured SecretRef on the PR head; it passes there while current main and docs require a literal "*".

## Solution Assessment

No. The feature direction is maintainable, but this implementation is not the best fix because public-open authorization should remain a visible wildcard opt-in in reviewed config.
