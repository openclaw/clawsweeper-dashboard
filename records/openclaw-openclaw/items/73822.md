## Summary

Keep open. Current main does not implement the linked SecretRef phone-number feature, and the PR is useful, but the latest head still introduces a security-sensitive open-DM validation regression by treating any SecretRef as the visible public wildcard.

## What This Changes

The PR adds SecretRef acceptance, resolution contracts, UI hints, docs, changelog, and tests for phone-number and sender fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin.

## Best Possible Solution

Land the feature after preserving the visible public-open wildcard contract: SecretRefs can satisfy configured allowlist presence, but dmPolicy="open" should still require a literal "*".

## Reproduction Assessment

Yes. A schema-level check of requireOpenAllowFrom with policy="open" and allowFrom=[SecretRef] reproduces the regression on the PR head because line 636 returns without adding an issue.

## Solution Assessment

No. The feature direction is maintainable, but allowing external secret material to stand in for the public-open wildcard is broader than needed; keep that guard literal and only relax allowlist presence for SecretRefs.
