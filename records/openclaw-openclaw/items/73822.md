## Summary

Keep open. The SecretRef feature is useful and current main does not already provide it, but the latest PR head still weakens the audited public-open DM contract and also has a concrete lint failure.

## What This Changes

This PR adds SecretRef support for phone-number and sender-identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call, with schema, secret-resolution, metadata, docs, UI hint, changelog, and test updates.

## Best Possible Solution

Keep the SecretRef feature, but preserve the visible public-open contract: SecretRefs can satisfy non-empty allowlist policy, while `dmPolicy="open"` should still require a literal `"*"`; then fix the lint failure and rerun the targeted channel/secret checks.

## Reproduction Assessment

Yes. On latest head `b8899b352955c92b316424bd219e8754f0d58ab5`, source inspection and the added unit test show a SecretRef-only open-policy allowlist is accepted; GitHub check annotations reproduce the lint failure.

## Solution Assessment

No. The feature direction is maintainable, but accepting any SecretRef as the public-open wildcard is not the safest fix; the narrower solution is to limit the SecretRef escape hatch to allowlist non-empty validation and keep open-policy wildcard validation literal.
