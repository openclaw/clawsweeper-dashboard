## Summary

Keep open. The branch addresses a real gap that still exists on current main, but it is not ready to merge because it lets any SecretRef satisfy the explicit public-open DM wildcard guard, moving a security-relevant opt-in out of reviewed config.

## What This Changes

The PR adds SecretRef support for phone-number and sender-identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call, with schema, runtime secret resolution, manifest target metadata, UI hints, docs, changelog, and tests.

## Best Possible Solution

Keep the SecretRef feature direction, but preserve the auditable public-open contract: SecretRefs can satisfy non-empty allowlist policy and resolved sender entries, while dmPolicy="open" should still require a literal "*" in reviewed config.

## Reproduction Assessment

Yes. Source inspection on PR head shows requireOpenAllowFrom returns when any SecretRef is present, and the added test expects SecretRef-only open policy to pass; current main and docs provide the contrasting literal-wildcard contract.

## Solution Assessment

No. The feature direction is useful, but accepting a hidden SecretRef as the public-open wildcard is not the narrowest safe implementation; remove the SecretRef shortcut from requireOpenAllowFrom and update the test while keeping SecretRef support for sender allowlists.
