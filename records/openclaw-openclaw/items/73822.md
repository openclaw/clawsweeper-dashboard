## Summary

Keep open: current main still lacks the requested SecretRef support, and this active PR is the implementation lane, but the PR head has a blocking access-control regression and only test/CI proof for the real runtime behavior.

## What This Changes

The PR adds SecretRef acceptance, resolution metadata, UI hints, docs, and tests for phone-number and sender-identity fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin.

## Best Possible Solution

Keep the SecretRef feature direction, but require a literal `"*"` for `dmPolicy="open"`, allow SecretRefs to satisfy only non-empty allowlist validation, update the tests, and add redacted real `openclaw doctor` or gateway proof with an env SecretRef.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse `dmPolicy="open"` with `allowFrom` containing only a structured SecretRef on the PR head; it passes there while current main and docs require a literal `"*"`.

## Solution Assessment

No. The overall feature direction is maintainable, but this implementation is not the best fix because public-open authorization should remain a visible wildcard opt-in in reviewed config.
