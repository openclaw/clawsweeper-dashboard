## Summary

Keep this PR open, but it is not merge-ready: the feature direction is still valid and not on current main, while the PR currently lets a structured SecretRef satisfy the public-open DM wildcard guard and only provides test/CI proof rather than a real gateway/secret-provider run.

## What This Changes

The PR adds SecretRef support for phone-number and sender-identity fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call, with schema, secret resolution metadata, UI hints, docs, changelog, and tests.

## Best Possible Solution

Keep the SecretRef feature direction, but require a literal "*" for dmPolicy="open", allow SecretRefs only to satisfy non-empty allowlist validation, update the tests, and add redacted real gateway or doctor proof with an env SecretRef.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse dmPolicy="open" with allowFrom containing only a structured SecretRef on the PR head; it passes there while current main and docs require a literal "*".

## Solution Assessment

No. The overall feature direction is maintainable, but this implementation is not the best fix because public-open authorization should remain a visible wildcard opt-in in reviewed config.
