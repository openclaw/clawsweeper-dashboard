## Summary

Keep open. The PR addresses a real SecretRef feature gap on current main, but it is not ready because it lets any SecretRef satisfy the visible wildcard guard for `dmPolicy="open"`, moving a public-open access-control opt-in out of reviewed config.

## What This Changes

The PR widens phone-number and sender config fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call to accept SecretRef inputs, with docs, UI hints, secret contracts, runtime resolution, and tests.

## Best Possible Solution

Keep the SecretRef feature, but require a literal `"*"` for `dmPolicy="open"` while allowing SecretRefs to satisfy non-empty allowlist validation.

## Reproduction Assessment

Yes, at source level. The PR diff makes `requireOpenAllowFrom` accept any SecretRef for `dmPolicy="open"`, and the added test asserts the SecretRef-only open-policy case; I did not run tests because this review is read-only.

## Solution Assessment

No. SecretRef support is a maintainable direction for phone-number allowlists, but public-open policy should preserve the existing visible wildcard opt-in and not depend on external secret contents.
