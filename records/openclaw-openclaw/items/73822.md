## Summary

Keep open. The feature is still useful because current main only accepts plain strings/E164 values on the requested channel fields, but latest PR head is not merge-ready: it lets any structured SecretRef satisfy dmPolicy="open" validation that current source, docs, and channel tests require to be a visible literal "*".

## What This Changes

This PR adds SecretRef support for phone-number and sender-identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, updating schemas, runtime secret resolution, target registry metadata, manifests, UI hints, docs, changelog, and tests.

## Best Possible Solution

Keep the SecretRef feature on this PR branch, but preserve the auditable public-open contract: SecretRefs can satisfy non-empty allowlist policy and resolved sender entries, while dmPolicy="open" should still require a literal "*" in reviewed config.

## Reproduction Assessment

Yes. On PR head 62d18ae17386ef007d62a656a4b3a3604f621ede, source inspection shows requireOpenAllowFrom returns when any SecretRef is present, and the new test expects a SecretRef-only open policy to pass.

## Solution Assessment

No. The feature direction is useful and mostly well-scoped, but accepting a hidden SecretRef as the public-open wildcard is not the safest implementation; the narrow fix is to reject SecretRef-only dmPolicy="open" while still accepting SecretRefs for allowlist sender entries.
