## Summary

Keep open. The SecretRef feature is useful and current main does not already implement it, but this PR is not ready because it weakens the documented public-open DM access-control guard by allowing a SecretRef object to stand in for the required visible wildcard.

## What This Changes

The PR widens phone-number and sender config fields across WhatsApp, Signal, iMessage, BlueBubbles, and voice-call to accept SecretRef inputs, with docs, UI hints, secret contracts, runtime resolution, manifest support, changelog, and tests.

## Best Possible Solution

Keep the SecretRef support, but require a literal "*" for dmPolicy="open" while allowing SecretRefs to satisfy non-empty allowlist validation after the policy distinction is covered by tests.

## Reproduction Assessment

Yes, at source level. On the PR head, requireOpenAllowFrom accepts policy="open" with only a SecretRef because of params.allowFrom?.some(isSecretRef), while current main and docs require a literal "*".

## Solution Assessment

No. The feature direction is maintainable, but the open-policy implementation should preserve the visible wildcard opt-in and keep SecretRef-only acceptance limited to allowlist-style non-empty checks.
