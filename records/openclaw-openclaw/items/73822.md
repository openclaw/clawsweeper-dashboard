## Summary

Keep open. The feature is not implemented on current main and the PR is useful, but latest PR head still has a blocking access-control regression: any SecretRef can satisfy the explicit wildcard guard for public-open DM policy.

## What This Changes

The PR adds SecretRef support, secret resolution contracts, sensitive UI hints, docs, changelog, and tests for phone-number and sender-identity fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin.

## Best Possible Solution

Keep the SecretRef feature, but preserve the public-open contract: SecretRefs may count as configured allowlist entries for allowlist policy, while dmPolicy="open" should still require a literal visible "*" in reviewed config. Update the SecretRef allowFrom tests to reject SecretRef-only open policy.

## Reproduction Assessment

Yes. Latest PR head 62d18ae17386ef007d62a656a4b3a3604f621ede shows requireOpenAllowFrom returns when allowFrom contains any SecretRef, and the new test expects a SecretRef-only open policy to pass.

## Solution Assessment

No. The feature direction is appropriate, but the open-policy escape hatch is broader than the existing access-control contract; the safer narrow fix is to keep SecretRefs valid for allowlists while requiring a literal wildcard for public-open mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
