## Summary

Keep open; the linked privacy feature is valid, but this PR is not ready to merge because it weakens the documented public-open DM wildcard guard and only provides tests/CI as behavior proof.

## What This Changes

The PR widens WhatsApp, Signal, iMessage, BlueBubbles, and voice-call phone/sender config fields to accept SecretRef inputs, with secret-contract/runtime collection, UI hints, docs, generated metadata, and tests.

## Best Possible Solution

Keep the SecretRef feature, but require a literal "*" for dmPolicy="open", allow SecretRefs only for non-empty allowlist validation, update the tests, and add real gateway/secret-provider proof before merge.

## Reproduction Assessment

Yes. At source level, the PR head accepts policy="open" with allowFrom containing only a SecretRef because requireOpenAllowFrom checks params.allowFrom?.some(isSecretRef), while current main and docs require a literal "*".

## Solution Assessment

No. The feature direction is maintainable, but the current implementation is not the best fix because public-open access should remain a visible wildcard opt-in in reviewed config.
