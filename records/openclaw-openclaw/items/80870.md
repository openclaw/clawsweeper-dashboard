## Summary

Keep open: this is a focused docs fix for a real contradiction, but the proposed managed-only qualifier would incorrectly exclude raw remote CDP profiles, and the external PR has no after-fix real behavior proof.

## What This Changes

Updates three browser docs pages to qualify `browser wait --load networkidle` support as managed-profile-only and existing-session-unsupported.

## Best Possible Solution

Revise the docs to say `--load networkidle` is unsupported for existing-session/Chrome MCP profiles and remains supported for Playwright-backed managed and raw remote CDP profiles.

## Reproduction Assessment

Yes. Current main reproduces the docs contradiction by reading the unqualified `networkidle` example beside the existing-session limitation, and source inspection confirms the runtime split.

## Solution Assessment

No. The PR heads in the right direction, but `managed openclaw profile only` is not the narrowest correct boundary because raw remote CDP profiles also use the Playwright wait path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not narrow networkidle to managed profiles:** `docs/tools/browser-control.md:295`
  - body: This qualifier says `--load networkidle` is managed `openclaw` only, but source only rejects it when `usesChromeMcp` is true. Remote CDP profiles set `usesChromeMcp: false` and use the Playwright path, which forwards `loadState` to `waitForLoadState`, so Browserless/Browserbase/raw-CDP users would get incorrect docs. Please describe the unsupported case as existing-session/Chrome MCP rather than managed-only.
  - confidence: 0.9
