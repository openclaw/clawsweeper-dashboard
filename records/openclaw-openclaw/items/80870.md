## Summary

## What This Changes

The branch edits three browser docs pages to add profile-scope qualifiers around `browser wait --load networkidle`, saying managed `openclaw` profiles support it and existing-session profiles do not.

## Best Possible Solution

Follow or land https://github.com/openclaw/openclaw/pull/80678, or equivalent wording that says `networkidle` is unsupported only for existing-session/user profiles while preserving managed and remote CDP support.

## Reproduction Assessment

Yes. Source inspection reproduces the docs contradiction, and the runtime source/tests identify the unsupported case as existing-session `loadState=networkidle`; no live browser run is needed for the docs-only contradiction.

## Solution Assessment

No. This PR duplicates an already-open, proof-backed docs fix and narrows support to managed `openclaw` profiles even though remote CDP falls through to the Playwright wait path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep remote CDP in the supported scope:** `docs/tools/browser-control.md:295`
  - body: The new qualifier makes `--load networkidle` managed-only, but current code only excludes Chrome MCP/existing-session profiles. Remote CDP profiles are non-Chrome-MCP and fall through to the Playwright path, which calls `page.waitForLoadState(opts.loadState, ...)`, so this would incorrectly tell remote CDP users the command is unsupported.
