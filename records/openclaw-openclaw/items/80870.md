## Summary

Close as duplicate/superseded: the same docs contradiction is already tracked by https://github.com/openclaw/openclaw/pull/80678, which is open, narrower, and has sufficient proof, while this branch still over-narrows support by excluding raw remote CDP profiles.

## What This Changes

The PR updates three browser docs pages to describe `browser wait --load networkidle` as supported only on managed `openclaw` profiles and unsupported on existing-session profiles.

## Best Possible Solution

Keep the canonical docs-only fix in https://github.com/openclaw/openclaw/pull/80678 and phrase the boundary as unsupported for existing-session/Chrome MCP while leaving managed and raw remote CDP support intact.

## Reproduction Assessment

Yes. Reading current main reproduces the docs ambiguity, and source/tests show the unsupported case is existing-session `loadState=networkidle` while Playwright-backed managed and raw CDP paths still accept load-state waits.

## Solution Assessment

No. A docs-only clarification is the right direction, but this PR's managed-only wording is not the best boundary; the open canonical PR tracks the narrower existing-session qualifier.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Do not narrow networkidle to managed profiles:** `docs/tools/browser-control.md:295`
  - body: This labels load-state waits as managed `openclaw` only, but current runtime only rejects `networkidle` when `usesChromeMcp` is true. Remote CDP profiles set `usesChromeMcp: false` and run through the Playwright `waitForLoadState` path, so Browserless/Browserbase/raw-CDP users would get incorrect docs.
