## Summary

Keep open. The PR addresses a real docs contradiction, but its current wording over-narrows `--load networkidle` to managed profiles even though current source shows remote CDP also uses the Playwright wait path.

## What This Changes

The PR edits three browser docs pages to qualify `browser wait --load networkidle` as unsupported for existing-session profiles and supported for managed OpenClaw profiles.

## Best Possible Solution

Revise the docs to state that `--load networkidle` is unsupported for existing-session/Chrome MCP profiles and remains supported for Playwright-backed managed and remote CDP profiles.

## Reproduction Assessment

Yes. Current main reproduces the contradiction by reading the browser-control `networkidle` example beside the existing-session limitation, and source inspection confirms the profile-dependent split.

## Solution Assessment

No. The PR is aimed at the right contradiction, but `managed openclaw profile only` is not the best wording because remote CDP profiles also use the Playwright wait path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not narrow networkidle to managed profiles:** `docs/tools/browser-control.md:295`
  - body: This qualifier says `--load networkidle` is managed `openclaw` only, but source only rejects it for existing-session/Chrome MCP profiles. Remote CDP profiles set `usesChromeMcp: false` and use the Playwright path that forwards `loadState` to `waitForLoadState`, so Browserless/Browserbase/raw-CDP users would get incorrect docs. Please describe the unsupported case as existing-session/Chrome MCP rather than managed-only, and apply that boundary to the other new parentheticals too.
