## Summary

Keep open. The docs clarification is aimed at the right profile boundary, but the PR overstates it by labeling all load-state waits as managed-only even though existing-session profiles support `load` and `domcontentloaded` and reject only `networkidle`; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR edits three browser docs pages to clarify that `browser wait --load networkidle` is supported for managed `openclaw` profiles but rejected for existing-session profiles.

## Best Possible Solution

Revise the docs to say specifically that `--load networkidle` is unsupported for existing-session/user profiles while `load` and `domcontentloaded` remain valid there, then add rendered-docs or terminal proof.

## Reproduction Assessment

Yes. The docs issue is source-reproducible by reading the current docs, and the PR issue is source-reproducible from `agent.act.ts`: existing-session rejects only `networkidle` while still supporting `load` and `domcontentloaded`.

## Solution Assessment

No. A docs-only clarification is the right shape, but this patch should scope the managed-only wording to `networkidle` instead of all load-state waits; runtime behavior should remain unchanged.
