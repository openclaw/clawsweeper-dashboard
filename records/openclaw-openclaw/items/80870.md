## Summary

## What This Changes

The PR edits browser CLI/control docs to clarify `browser wait --load networkidle` as managed-profile-supported and existing-session-unsupported.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80678 or equivalent wording that says `networkidle` is unsupported only for existing-session/Chrome MCP profiles while preserving remote-CDP support.

## Reproduction Assessment

Yes. Reading current main's browser-control wait examples beside the existing-session limitations reproduces the ambiguity, and source/tests confirm the actual unsupported case is existing-session `loadState=networkidle`.

## Solution Assessment

No. A docs-only clarification is the right direction, but this patch's managed-only wording is not the best fix because remote CDP also uses the Playwright wait path; the narrower open PR is the better route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Do not narrow networkidle support to managed profiles:** `docs/tools/browser-control.md:295`
  - body: The new qualifier says load-state waits require the managed `openclaw` profile, but current source only rejects `loadState=networkidle` when `usesChromeMcp` is true. Remote CDP profiles set `usesChromeMcp: false` and go through the Playwright path, so this would incorrectly tell Browserbase/raw-CDP users their supported wait is unavailable.
