The PR appends an oom_score_adj/cgroup OOM diagnostic to Linux wrapper-active SIGKILL exec failures, normalizes numeric signal 9 to SIGKILL in the displayed reason, and adds formatter, predicate, and spawn-env tests.

## Best Possible Solution

Land the render-only diagnostic and regression tests if maintainers accept the failure-message compatibility tradeoff and the probabilistic wording.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.28 still render signal failures as a bare `Command aborted by signal ...`, while the inspected proof commit shows a real cgroup OOM kill shape and the new formatter output. I did not run a live Linux VPS repro in this read-only pass.

## Solution Assessment

Yes, this is the narrowest maintainable fix I found: the kill behavior and oom_score_adj wrapper remain owned by the existing process path, and only the exec failure formatter gets additional context gated by spawn-time env and SIGKILL. A post-exit cgroup/proc probe would be broader and less reliable.
