## Summary

Keep open. Current main still leaves source-reproducible maintenance gaps: terminal cron `lost` rows cannot reach durable recovery, and stale CLI rows with subagent-shaped child sessions can still be treated as live; PR #75036 is open but unmerged and covers only the cron subset.

## What This Changes

`tasks maintenance --apply` should repair or prune `lost` backing-session-missing records and stale running task records instead of leaving audit errors unchanged.

## Best Possible Solution

Keep this as the canonical tracker until terminal lost cron recovery and stale CLI/subagent child-session liveness are both fixed with regression coverage; handle timestamp clamping separately.

## Reproduction Assessment

Yes, at source level. A terminal `runtime: "cron"`, `status: "lost"` task with matching durable cron evidence cannot reach recovery, and a stale `runtime: "cli"` task with an `agent:...:subagent:...` child session can be kept live by an old session row.

## Solution Assessment

Partly. `tasks maintenance` is the right surface, but the maintainable path is to land or replace #75036 for terminal cron recovery and handle CLI/subagent liveness and timestamp-rewrite policy as separate reviewed changes.
