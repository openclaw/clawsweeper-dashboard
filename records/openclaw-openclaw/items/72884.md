## Summary

Keep open. Current main has fixed the narrow CLI run-identity false-liveness path, but source inspection still shows already-terminal `lost` cron rows cannot reach durable cron recovery, and the remaining stale-running cleanup is still covered by open review work.

## What This Changes

Make `openclaw tasks maintenance --apply` reconcile or prune `lost` backing-session-missing and `stale_running` task records instead of reporting no changes.

## Best Possible Solution

Finish the scoped fixes already under review: land terminal lost cron recovery via https://github.com/openclaw/openclaw/pull/75036 and stale-running reconciliation via https://github.com/openclaw/openclaw/pull/79288, then handle timestamp clamping as a separate policy decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: terminal `runtime: "cron"`, `status: "lost"` rows cannot pass the active-task guard for durable cron recovery, while the broader stale-running path is source-backed and separately covered by https://github.com/openclaw/openclaw/issues/79226.

## Solution Assessment

Partly. `tasks maintenance` is the right surface, but the maintainable path is to land the narrow open fixes and keep timestamp rewriting separate rather than queueing another broad repair from this issue.
