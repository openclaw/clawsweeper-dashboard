## Summary

Keep open. Current main still leaves the reported maintenance gap partially unresolved: terminal `lost` cron rows cannot be recovered by the durable cron path, and CLI rows with subagent-shaped child-session keys can still be treated as live from a persisted session row after the owning run context is gone.

## What This Changes

The issue requests that `tasks maintenance --apply` clean up or repair `lost` backing-session-missing and `stale_running` task records that currently remain after audit.

## Best Possible Solution

Keep this as the canonical tracker until terminal lost cron recovery and stale CLI/subagent child-session liveness are fixed with regression coverage; handle timestamp clamping as a separate reviewed policy change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: terminal `runtime: "cron"`, `status: "lost"` rows cannot reach durable cron recovery, and stale `runtime: "cli"` rows with `agent:...:subagent:...` child-session keys can be kept live by an old session row.

## Solution Assessment

Partly. `tasks maintenance` is the correct surface, but the maintainable fix should split terminal cron recovery, CLI/subagent liveness, and timestamp-rewrite policy into narrow reviewed changes.
