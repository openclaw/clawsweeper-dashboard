## Summary

Keep open. Current main still leaves part of the reported maintenance gap unresolved: durable cron recovery only runs for active cron rows, and stale CLI tasks with subagent-shaped child-session keys can still be treated as live when only a historical session row remains.

## What This Changes

`tasks maintenance --apply` is expected to repair, mark, or prune lost backing-session-missing task records and stale running records that survive supported maintenance.

## Best Possible Solution

Keep this as the canonical tracker until terminal lost cron recovery and stale CLI/subagent child-session liveness are both covered with regression tests; handle timestamp rewriting separately if maintainers want that policy.

## Reproduction Assessment

Yes. A source-level reproduction is a terminal `runtime: "cron"`, `status: "lost"` task with durable cron run evidence, plus a stale `runtime: "cli"` task whose `childSessionKey` is `agent:...:subagent:...`, whose live run context is gone, and whose session row still exists.

## Solution Assessment

No, not as blanket pruning of every audit finding. Task maintenance is the right surface, but the maintainable fix should split terminal cron recovery, CLI/subagent liveness, and timestamp rewrite policy into narrow tested changes.
