## Summary

Keep open. Current main still leaves two reported no-op classes unresolved: terminal `lost` cron rows cannot use durable cron recovery because that path is active-task-only, and stale `runtime: "cli"` rows with `agent:...:subagent:...` child-session keys can still be treated as backed by a historical session row after the live run context is gone. Open PR #75036 targets only the cron subset, so this issue remains the broader tracker.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` repair or prune `lost` task records whose backing session is missing and `stale_running` task records, rather than leaving audit errors unchanged.

## Best Possible Solution

Keep this issue as the canonical tracker until maintenance can recover terminal `lost` cron rows when durable cron evidence exists, stop treating stale CLI subagent child-session rows as live after the owning run context disappears, and then prune or cleanup-stamp terminal rows according to the existing retention model. Timestamp clamping should be split into a small policy-backed follow-up if maintainers want persistent correction.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a terminal `runtime: "cron"` task already marked `lost` with durable run evidence, plus a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present; current main leaves both paths unrepaired.

## Solution Assessment

Partially. The `tasks maintenance --apply` surface is the right place to repair the task ledger, but a blanket prune is less maintainable than recovering cron terminal state when possible, marking unrecoverable stale rows terminal, and letting the existing cleanup retention prune them.
