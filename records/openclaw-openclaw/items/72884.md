## Summary

Keep open. Current main has partial task-maintenance fixes, but the reported no-op class is still real for stale CLI rows backed only by persisted subagent child-session rows, and the related open PR only covers the terminal lost cron subset.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune `lost` backing-session-missing task records, stale running task records, and benign timestamp drift instead of reporting no changes.

## Best Possible Solution

Keep this as the canonical tracker until the cron terminal-recovery patch lands or is replaced, add a focused CLI/subagent child-session liveness fix with regression coverage, and handle any timestamp rewrite policy explicitly.

## Reproduction Assessment

Yes. A source-level reproduction is a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present; current predicates treat the persisted row as live backing. A second source-level path is a terminal `runtime: "cron"` task with durable run history, because current recovery is active-task-only.

## Solution Assessment

No, current main is not yet the full solution. The narrow maintainable path is to finish the cron subset via #75036 or equivalent, then separately fix CLI/subagent liveness without broadening session-row trust.
