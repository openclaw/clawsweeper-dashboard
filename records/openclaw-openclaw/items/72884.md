## Summary

Keep open. Current main has partial task-maintenance hardening, but it still does not fully address the reported no-op behavior for terminal `lost` cron rows or stale CLI rows backed only by persisted subagent session history.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` repair, recover, or prune `lost` backing-session-missing task records and `stale_running` rows instead of leaving audit errors unchanged.

## Best Possible Solution

Keep this issue as the canonical tracker for the operator-visible audit cleanup problem, land or replace the cron-focused #75036 work, then add a focused CLI/subagent liveness fix so a persisted child-session row is not treated as proof of a live CLI run after the owning run context is gone.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present; current main treats that stored row as backing state. A terminal `lost` cron task also cannot be recovered from durable history because cron recovery returns early for non-active tasks.

## Solution Assessment

No. Current main is only a partial solution, and the open related PR covers the cron subset rather than the full reported behavior; the maintainable path is a cron recovery change plus a separate regression-tested CLI/subagent liveness predicate change.
