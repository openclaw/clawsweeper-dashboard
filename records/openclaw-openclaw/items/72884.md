## Summary

Keep open. Current main still leaves the reported maintenance no-op partially unresolved: terminal `lost` cron rows cannot use durable cron recovery because recovery is active-task-only, and stale CLI rows with subagent-shaped child-session keys can still be treated as live when only a persisted session row remains; related PR #75036 covers only the cron subset and is not merged in the provided context.

## What This Changes

Request that `openclaw tasks maintenance --apply` repair or prune `lost` backing-session-missing task records and stale-running CLI/cron task records, with optional timestamp clamping for benign skew.

## Best Possible Solution

Keep this as the canonical tracker until terminal `lost` cron recovery and stale CLI/subagent liveness are both handled; treat timestamp rewriting as a separate maintainer policy decision.

## Reproduction Assessment

Yes. A source-level reproduction is a terminal `runtime: "cron"` task with durable run history, which current main skips because `resolveDurableCronTaskRecovery()` requires an active task; a second path is a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable path is to land or rework the cron recovery change and add a separate regression-tested CLI/subagent liveness predicate fix.
