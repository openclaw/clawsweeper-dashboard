## Summary

Keep open. Current main still leaves two reported maintenance gaps: terminal `lost` cron rows cannot be recovered from durable cron history because durable recovery is active-task-only, and stale CLI task rows with `agent:...:subagent:...` child-session keys can still be treated as live when only an old session-store row remains. The related open PR #75036 covers the terminal-lost cron slice only, so this issue should stay open as the canonical tracker until the cron fix and the separate CLI/subagent liveness fix are handled.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to clean or recover `lost` backing-session-missing records, stale `running` task records, and optionally benign `inconsistent_timestamps` warnings.

## Best Possible Solution

Keep this issue as the canonical tracker, review or replace #75036 for terminal lost cron recovery, then add a focused stale CLI/subagent liveness change so persisted child-session history is not treated as proof of a live CLI run after the owning run context is gone. Handle timestamp clamping as a separate policy decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present; current main treats that historical session entry as backing state and leaves the row running. A terminal `lost` cron row with durable run history also cannot recover because durable cron recovery is active-task-only.

## Solution Assessment

No. Current main is incomplete for the reported behavior, and #75036 is only a partial cron fix; the maintainable path is a bounded cron recovery fix plus a separate regression-tested CLI/subagent liveness predicate change.
