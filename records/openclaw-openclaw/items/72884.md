## Summary

Keep open. Current main still leaves the reported maintenance no-op partially unresolved: terminal `lost` cron rows cannot use durable cron recovery, and stale CLI rows with subagent-style child-session keys can still be treated as live because persisted session history counts as backing state.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` reconcile or prune `lost` backing-session-missing and `stale_running` task records, with optional timestamp cleanup for benign skew.

## Best Possible Solution

Keep this as the canonical tracker until current main handles both terminal `lost` cron recovery and stale CLI/subagent liveness; land or rework #75036 for the cron subset, then add a focused CLI child-session predicate fix and treat timestamp rewriting separately.

## Reproduction Assessment

Yes. A source-level reproduction is a terminal `runtime: "cron"` task with durable run history, which current main refuses to recover because recovery is active-task-only; a second path is a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present.

## Solution Assessment

No. Current main is only a partial solution, and the related open PR appears scoped to cron rows; the narrow maintainable fix is cron terminal recovery plus a separate regression-tested CLI/subagent liveness change.
