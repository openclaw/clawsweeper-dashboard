## Summary

Keep open. Current main has addressed the narrow CLI run-identity false-liveness path and has cleanup stamping/pruning for terminal task rows, but the reported maintenance no-op is not fully resolved because terminal `lost` cron recovery and stale-running reconciliation are still covered by open, unmerged follow-up PRs.

## What This Changes

Request that `openclaw tasks maintenance --apply` clean up `lost` backing-session-missing task records and stale `running` records instead of leaving audit errors unchanged.

## Best Possible Solution

Land or replace the focused terminal-lost cron recovery and stale-running reconciliation PRs, then handle timestamp normalization as a separate policy decision.

## Reproduction Assessment

Yes, by source inspection. Current main still has an active-only durable cron recovery guard for terminal `lost` cron rows, and stale-running rows can remain active when old persisted child-session evidence is treated as backing state.

## Solution Assessment

Partly. `tasks maintenance` is the correct surface, but the safest solution is to finish the two narrower open PRs rather than fold terminal cron recovery, stale-running policy, and timestamp clamping into one broad change.
