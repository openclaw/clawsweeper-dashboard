## Summary

Keep open. Current main still leaves the reported maintenance gap partially unresolved: terminal `lost` cron rows cannot be recovered from durable cron state, and stale CLI rows with subagent-shaped child-session keys can still be treated as live when only an old session row remains.

## What This Changes

`tasks maintenance --apply` should repair or prune `lost` backing-session-missing records and stale running records instead of leaving audit errors unchanged.

## Best Possible Solution

Keep this as the canonical tracker until terminal lost cron recovery and stale CLI/subagent child-session liveness are both fixed with regression coverage; handle timestamp clamping separately.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a terminal `runtime: "cron"`, `status: "lost"` task with durable cron evidence cannot reach recovery, and a stale `runtime: "cli"` task with an `agent:...:subagent:...` child session can be kept live by an old session row.

## Solution Assessment

Partly. `tasks maintenance` is the correct surface, but the maintainable fix should split terminal cron recovery, CLI/subagent liveness, and timestamp-rewrite policy into narrow reviewed changes.
