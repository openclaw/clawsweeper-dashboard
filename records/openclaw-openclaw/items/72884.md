## Summary

Keep open. Current main still leaves the reported maintenance gap unresolved: terminal `lost` cron rows cannot be recovered because durable cron recovery is gated to active tasks, and stale CLI rows with persisted subagent-shaped child session rows can still be treated as live after their run context is gone; open PR #75036 covers only the cron subset.

## What This Changes

Request that `openclaw tasks maintenance --apply` repair or prune `lost` backing-session-missing records and stale running task records instead of reporting zero changes while audit findings remain.

## Best Possible Solution

Keep this as the canonical tracker until maintenance recovers terminal lost cron rows and marks stale CLI/subagent-owned rows lost when no live run context remains; treat timestamp clamping as separate policy work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a terminal cron `lost` task cannot reach durable recovery because recovery requires an active task, and a stale CLI task with a subagent-shaped child session can stay live if only an old session row remains.

## Solution Assessment

Partly. `tasks maintenance` is the right surface, but the maintainable path should split terminal cron recovery, CLI/subagent liveness, and timestamp-clamping policy into narrow reviewed changes.
