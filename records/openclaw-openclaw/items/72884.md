## Summary

Keep open. Current main still leaves the central maintenance bug partially unresolved: terminal `lost` cron rows do not enter durable cron recovery, and stale CLI rows with subagent-style child-session keys can still be treated as live when only an old session row remains.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` reconcile or prune `lost` and `stale_running` task records with missing backing sessions, with optional timestamp warning cleanup.

## Best Possible Solution

Keep this as the canonical tracker until terminal lost cron recovery and stale CLI/subagent child-session liveness are both handled with regression coverage; treat timestamp rewriting as a separate explicit cleanup policy.

## Reproduction Assessment

Yes. A source-level reproduction is a terminal `runtime: "cron"`, `status: "lost"` task with durable cron run evidence, plus a stale `runtime: "cli"` task whose `childSessionKey` is `agent:...:subagent:...`, whose live run context is gone, and whose session row still exists.

## Solution Assessment

Partly. Task maintenance is the right surface, but the maintainable implementation should split terminal cron recovery, CLI/subagent liveness, and timestamp rewrite policy into narrow tested changes rather than generically pruning every audit finding.
