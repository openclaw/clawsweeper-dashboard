## Summary

Keep open. Current main still only partially addresses the report: terminal `lost` cron rows cannot be recovered by the active-only durable cron recovery predicate, and stale CLI rows with subagent child-session keys can still be treated as live because historical session rows count as backing state.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune `lost` backing-session-missing task rows, stale running task rows, and benign timestamp drift instead of reporting no maintenance action.

## Best Possible Solution

Keep this as the canonical tracker until the cron terminal-recovery patch lands or is replaced, add a focused CLI/subagent child-session liveness fix with regression coverage, and handle timestamp rewriting as an explicit policy choice.

## Reproduction Assessment

Yes. A source-level reproduction is a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present; a second source-level path is a terminal `runtime: "cron"` task with durable run history, because current recovery is active-task-only.

## Solution Assessment

No. Current main is only a partial solution, and the related open PR appears scoped to cron rows; the narrow maintainable path is terminal cron recovery plus a separate CLI/subagent liveness fix, with timestamp rewriting handled separately.
