## Summary

Keep open. Current main still leaves the reported maintenance no-op partially unresolved: durable cron recovery is active-task-only, and stale CLI rows with subagent-style child-session keys can still be treated as live because persisted session history counts as backing state; #75036 is open but only covers the cron subset.

## What This Changes

Request that `openclaw tasks maintenance --apply` clean up or recover `lost` backing-session-missing records and stale running task rows, with a secondary suggestion to normalize benign timestamp skew.

## Best Possible Solution

Keep this as the canonical tracker until terminal lost cron recovery and stale CLI/subagent child-session liveness are both handled, with timestamp rewriting treated as a separate policy-backed cleanup if needed.

## Reproduction Assessment

Yes. A source-level reproduction is a terminal `runtime: "cron"` task with durable cron history, plus a stale `runtime: "cli"` task whose `childSessionKey` is `agent:...:subagent:...` and whose live run context is gone while the session row remains.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable fix is to recover or clean terminal lost cron rows from durable history and make CLI liveness depend on active run context for subagent-style child sessions instead of persisted transcript existence.
