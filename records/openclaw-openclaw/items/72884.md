## Summary

Keep open. Current main still has a real maintenance gap: durable cron recovery is active-task-only, and stale CLI rows using subagent-scoped session keys can still look backed by old session-store data. The related open PR #75036 covers only the terminal lost cron slice, while the stale CLI/subagent liveness path remains separate.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` repair or prune lost backing-session-missing task records and stale running task rows, with optional normalization for benign timestamp skew.

## Best Possible Solution

Keep this issue as the canonical tracker, review or replace #75036 for terminal lost cron recovery, then add a separate focused liveness fix so stale CLI rows with subagent-scoped child-session keys are not kept alive by historical session-store entries after their live run context is gone. Treat timestamp normalization as already mostly handled unless a new raw-data case reproduces warnings on current main.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a stale `runtime: "cli"` task with no active run context, `childSessionKey: "agent:main:subagent:..."`, and a retained session-store row; current main treats that row as backing state. A terminal `lost` cron row also cannot take the durable cron recovery path because recovery exits for non-active tasks.

## Solution Assessment

No. Current main is incomplete for the reported operator problem, and #75036 is only a partial cron fix; the maintainable solution is a bounded terminal-cron recovery fix plus a separate regression-tested CLI/subagent liveness predicate change.
