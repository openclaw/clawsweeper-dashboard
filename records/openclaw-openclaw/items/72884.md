## Summary

Keep open. Current main still leaves the reported maintenance gap: terminal `lost` cron rows are excluded from durable cron recovery, and stale CLI rows with subagent-scoped child-session keys can still be treated as backed by historical session-store state. The related open PR only addresses the terminal-lost cron slice, so this issue remains useful as the canonical tracker for the broader behavior.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune `lost` task records whose backing sessions disappeared and stale-running CLI/subagent-backed records, with optional timestamp clamping for benign clock skew.

## Best Possible Solution

Keep this issue as the canonical tracker until terminal lost cron rows and stale CLI/subagent false-liveness rows are both covered by focused maintenance logic and regression tests. Treat timestamp clamping as a separate policy decision if maintainers want audit warnings to stay diagnostic.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a terminal `lost` cron task whose durable run history would otherwise recover it, plus a stale `runtime: "cli"` task with an `agent:...:subagent:...` child-session key whose historical session row remains after the live run context is gone; current main leaves those cases unrepaired.

## Solution Assessment

No. Current main is incomplete, and the related open PR is only a partial cron fix; the safer solution is to land or revise that cron change, then add a separate regression-tested CLI/subagent liveness predicate change.
