## Summary

Keep open. Current main still leaves part of the reported maintenance no-op behavior unresolved: terminal `lost` cron rows are excluded from durable cron recovery because recovery is active-task-only, and stale CLI rows with subagent child-session keys can still be treated as backed by historical session-store rows. Timestamp normalization and terminal pruning now reduce part of the report, and open PR #75036 targets only the cron subset.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune task records reported by audit as `lost` from missing backing sessions and `stale_running`, with optional timestamp cleanup.

## Best Possible Solution

Keep this issue as the canonical tracker, review or replace #75036 for terminal `lost` cron recovery, then add a focused CLI/subagent liveness fix so persisted child-session history is not treated as proof of a live CLI run after the owning run context is gone. Treat timestamp cleanup as a separate bounded policy decision if maintainers still want persistent rewrites.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a stale `runtime: "cli"` task with no active run context and an `agent:...:subagent:...` child-session row still present; current main treats the historical session entry as backing state and leaves the row running. A terminal `lost` cron task with durable run history also cannot recover because durable cron recovery returns early for non-active tasks.

## Solution Assessment

No. Current main is not the complete solution for the reported behavior, and the related open PR is only a partial cron fix; the narrow maintainable path is a cron recovery change plus a separate regression-tested CLI/subagent liveness predicate change.
