## Summary

Keep open. Current main still leaves the reported maintenance gap: durable cron recovery only runs for active cron tasks, so already-terminal `lost` cron rows are not repaired, and stale CLI rows with `agent:...:subagent:...` child-session keys can still look live because a historical session-store row counts as backing state. The related open PR #75036 addresses only the terminal-lost cron slice, not the stale CLI/subagent liveness path or the optional timestamp policy.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` repair or prune `lost` task records whose backing session is gone and `stale_running` task records that no longer have a live run context.

## Best Possible Solution

Keep this issue as the canonical tracker until the terminal lost cron recovery lands or is replaced, then add a focused stale CLI/subagent liveness fix so persisted child-session history is not treated as proof of a live CLI run after the run context is gone. Treat timestamp clamping as a separate policy follow-up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a terminal `lost` cron task with matching durable run history and a stale `runtime: "cli"` task whose only remaining backing evidence is an old `agent:...:subagent:...` child-session row, then observe current main leaves those rows unrepaired.

## Solution Assessment

No. Current main is incomplete for the reported behavior, and the open related PR is only a partial cron fix; the safer solution is a focused cron recovery change plus a separate regression-tested CLI/subagent liveness predicate change.
