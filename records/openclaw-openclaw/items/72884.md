## Summary

Keep open. Current main still has the reported maintenance gap: terminal `lost` cron rows cannot use durable cron recovery because recovery is active-task-only, and stale `runtime: "cli"` rows backed only by persisted `agent:...:subagent:...` child-session history can still be treated as live. The open related PR #75036 covers only the terminal-lost cron slice, so this issue remains useful as the broader tracker.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune `lost` task rows whose backing session is missing and `stale_running` rows that no longer have real liveness evidence.

## Best Possible Solution

Keep this issue as the canonical tracker until #75036 or an equivalent fix lands for terminal lost cron rows, then add a focused source and regression-test change for stale CLI/subagent liveness. Treat timestamp clamping as a separate maintainer policy decision rather than bundling it into the core recovery fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an already-terminal `lost` cron task with matching durable cron history and a stale `runtime: "cli"` task whose `childSessionKey` is `agent:...:subagent:...` with only a persisted session-store row; current main leaves those cases unrepaired for the reasons shown above.

## Solution Assessment

No, current main is not the best solution because it only handles active cron recovery and chat-backed CLI false liveness. The narrow maintainable path is runtime-aware terminal cron recovery/pruning plus a stricter CLI child-session liveness predicate, with timestamp normalization handled separately.
