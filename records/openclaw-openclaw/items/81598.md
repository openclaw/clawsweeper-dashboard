## Summary

Keep open: current main still has the reported message-tool TTS gap, and the linked closing PR is open and not merged. The issue is concrete, externally reproduced on the latest release, and should close only after the fix PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to apply existing TTS payload normalization to proactive `message(action="send")` delivery paths so TTS directives are stripped and audio is attached.

## Best Possible Solution

Land or replace the linked fix so TTS normalization runs before every shared message-action send branch, including gateway, prepared payload, plugin-action, and core fallback delivery.

## Reproduction Assessment

Yes. Source inspection shows `message(action="send")` reaches outbound send delivery without a TTS hook while final reply delivery does call the TTS normalizer, and a reporter confirmed the shipped v2026.5.12 Telegram path still exposes literal directives.

## Solution Assessment

Yes for the direction, but not closed yet. The maintainable fix is to reuse the existing TTS payload normalizer in the shared outbound send path; the linked PR is the implementation candidate and still needs review/rework/proof before this issue closes.
