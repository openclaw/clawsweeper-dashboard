## Summary

Keep open: current main still has a source-visible gap in the message-tool send path, and https://github.com/openclaw/openclaw/pull/81681 is an open closing PR rather than a merged fix. The issue should close only after that PR, or a corrected replacement, lands.

## What This Changes

The issue requests applying the existing TTS payload normalizer to `message(action=send)` so TTS directives are stripped and generated audio is attached on outbound message-tool sends.

## Best Possible Solution

Land a corrected shared outbound TTS normalization path that covers gateway, prepared-payload, legacy plugin-action, and core sends, with focused regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message(action="send")` flows through `message-tool.ts`, `message-action-runner.ts`, and `outbound-send-service.ts` without a TTS hook, while final replies call the TTS normalizer before delivery.

## Solution Assessment

Yes. The maintainable direction is to reuse the existing TTS payload pipeline in the shared outbound send path; adding another `visibleReplies` config mode would leave the default path fractured.
