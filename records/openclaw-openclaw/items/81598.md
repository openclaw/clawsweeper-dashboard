## Summary

Keep this open: current main still has a source-visible gap where `message(action="send")` reaches outbound send paths without invoking the existing TTS payload normalizer, while final and cron delivery do invoke it. The linked fix PR at https://github.com/openclaw/openclaw/pull/81681 closes this issue but is still open and has review feedback, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `message(action=send)` skips TTS normalization, leaving TTS directives visible and no audio attached on outbound message-tool sends.

## Best Possible Solution

Land a corrected shared outbound TTS normalization that covers gateway, prepared-payload, plugin-action, and core send branches without adding a new config option.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message(action=send)` reaches gateway or outbound send branches without the TTS gate/runtime call, while final and cron delivery call the existing TTS normalizer.

## Solution Assessment

Yes. The right direction is to reuse the existing TTS payload pipeline in the shared outbound send path; the currently linked PR needs revisions before merge because a fix must cover gateway-mode sends and default TTS mode semantics.
