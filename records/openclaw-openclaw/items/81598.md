## Label Justifications

- `P1`: The report describes a deterministic broken agent/channel workflow where configured TTS audio is not delivered and raw directives reach users across message-tool sends.
- `impact:message-loss`: The affected path drops the synthesized audio attachment for outbound visible messages and sends only literal directive text.

## Summary

Keep open: current main still has a source-visible gap where `message(action="send")` reaches outbound delivery without the TTS payload normalizer, while final replies do use that path. An open linked PR references the issue with closing syntax, so this issue should remain open until a corrected fix lands.

## What This Changes

The issue requests applying the existing TTS payload transformation to `message(action=send)` outbound paths so TTS directives are stripped and synthesized audio is attached.

## Best Possible Solution

Normalize TTS once at the shared message-action payload boundary before gateway, plugin-action, prepared-payload, and core delivery branches, treating visible message-tool sends as final visible output where appropriate and covering the behavior with focused regression proof.

## Reproduction Assessment

Yes. Source inspection shows `message(action="send")` reaches outbound send delivery without the TTS hook while final replies use the existing TTS normalizer; I did not run a live provider repro in this read-only review.

## Solution Assessment

Yes, the maintainable direction is to reuse the existing TTS payload normalizer in a shared outbound message-action path. The currently linked PR should be reviewed or revised so it covers gateway and plugin branches before this issue closes.
