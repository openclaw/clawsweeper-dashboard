## Summary

Keep open: current `main` still has the reported source-visible TTS gap on `message(action="send")`, and the linked closing PR is still open/unmerged with review findings, so this issue should remain open until a corrected fix lands.

## What This Changes

The issue reports that message-tool sends skip TTS normalization, leaving `[[tts:text]]` directives visible and no audio attached.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/81681, or an equivalent patch, that applies existing TTS normalization before all message-action send delivery branches and adds focused regression proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message(action="send")` flows through the message tool and outbound runner into `executeSendAction` without a TTS hook, while final replies call the TTS normalizer before delivery.

## Solution Assessment

Yes. The best direction is to reuse the existing TTS payload normalizer in the shared outbound message-action path; adding a new visible-replies config mode would leave existing message-tool sends broken and add unnecessary surface.
