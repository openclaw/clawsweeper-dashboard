## Summary

This should stay open: current `main` still has a source-visible gap where `message(action="send")` builds and delivers outbound payloads without invoking the existing TTS payload normalizer, while final reply delivery does invoke it. The report is recent, concrete, externally authored, unprotected by labels, and points to a narrow shared outbound fix rather than stale cleanup.

## What This Changes

Ensure proactive `message(action="send")` sends pass through auto-TTS so TTS directives are stripped and generated audio is attached on tool-only visible replies.

## Best Possible Solution

Apply the existing TTS payload normalization in the shared outbound message-action send path before channel delivery, treating visible source sends as the final visible reply equivalent and avoiding a new config option.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a `message(action="send")` call flows through `message-tool.ts` into `message-action-runner.ts` and `outbound-send-service.ts` without any TTS hook, while final replies call the TTS normalizer before delivery. I did not run the live provider path in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable solution is to reuse the existing `maybeApplyTtsToPayload`/`shouldAttemptTtsPayload` pipeline in the shared outbound payload path. Adding a new `visibleReplies` enum value would leave the default tool-only route broken and add avoidable config surface.
