## Summary

Keep this open: current main still has a source-visible TTS gap on the message-tool send path, and an open fixing PR already references this issue with closing syntax, so the issue should wait for that PR to be corrected and merged.

## What This Changes

Apply existing TTS payload normalization to `message(action=send)` so TTS directives are stripped and generated audio attaches on outbound message-tool sends.

## Best Possible Solution

Land a corrected shared outbound TTS normalization before every message-action delivery branch, including gateway, plugin-action, prepared-payload, and core fallback sends, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message(action=send)` flows through `message-tool.ts`, `message-action-runner.ts`, and outbound/gateway send branches without a TTS hook, while final replies call the TTS normalizer before delivery.

## Solution Assessment

Yes for the requested direction, but not for the current linked PR as written. The maintainable fix should normalize once before all send branches and treat visible message-tool sends as the same TTS-visible reply class, rather than adding a new config option.
