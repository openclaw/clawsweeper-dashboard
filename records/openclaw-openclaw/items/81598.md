## Summary

Keep this issue open: current main still has a source-visible TTS bypass on the message-tool send path, and the linked closing PR is open rather than merged.

## What This Changes

Ensure proactive `message(action=send)` sends run the existing TTS payload pipeline so TTS directives are stripped and synthesized audio attaches to outbound messages.

## Best Possible Solution

Review and repair or replace the linked PR so TTS normalization happens once before every message-tool send branch, including gateway, prepared-payload, plugin-action, and core fallback delivery, without adding a new config option.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `message(action=send)` flows through `message-tool.ts`, `message-action-runner.ts`, and `outbound-send-service.ts` without a TTS hook, while final replies call the TTS payload path. I did not run a live Slack, Telegram, Signal, or iMessage provider in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing TTS payload normalization in the shared outbound message-action path. The linked PR is the active implementation path, but it should be reviewed and fixed before this issue closes.
