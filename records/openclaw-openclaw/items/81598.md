## Summary

Current `main` still has a source-visible bug: message-tool sends build and deliver outbound payloads without invoking the existing TTS payload normalizer, while final reply delivery does invoke it. The issue should stay open because https://github.com/openclaw/openclaw/pull/81681 is an open linked fix PR with closing syntax, not a merged implementation.

## What This Changes

The issue requests that `message(action="send")` apply the same TTS payload transformation used by final reply delivery so tagged TTS text is stripped and converted to audio before channel delivery.

## Best Possible Solution

Review and land the linked focused fix PR, or replace it with an equivalent shared outbound-send TTS normalization patch plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `message(action="send")` flows through `message-tool.ts`, `message-action-runner.ts`, and `outbound-send-service.ts` without a TTS hook, while final replies call the TTS normalizer before delivery.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse `shouldAttemptTtsPayload` and `maybeApplyTtsToPayload` in the shared outbound send path, instead of adding a new visible-replies config mode.
