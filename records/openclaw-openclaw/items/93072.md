## Summary

## What This Changes

The PR changes `upsertSessionEntry` replacement writes to preserve existing non-empty `label` and `displayName` values when the incoming entry omits those fields, adds regression coverage, and documents the SDK/runtime contract.

## Best Possible Solution

Land this only after maintainer/API-owner acceptance that omitted title metadata in replacement upserts means preserve existing human titles, with explicit empty/null values and `patchSessionEntry({ replaceEntry: true })` remaining the clear paths.

## Reproduction Assessment

Yes. Current main’s file-store `upsertSessionEntry` clones the replacement entry and persists it, so a runtime refresh that omits `label` and `displayName` deterministically drops existing title metadata.

## Solution Assessment

Yes, with maintainer contract acceptance. The store boundary is the narrowest durable fix because all replacement upserts pass through it; the safer alternative if compatibility concerns surface is to keep the current behavior and design an explicit opt-in preservation or clear contract.
