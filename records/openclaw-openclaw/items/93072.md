## Summary

Keep open for maintainer/API-owner review: the patch is focused and source-backed, with supplied real behavior proof, but it intentionally changes exported replacement-upsert semantics for omitted session title fields.

## What This Changes

The PR preserves existing non-empty `label` and `displayName` values during `upsertSessionEntry` replacement writes when the incoming entry omits those fields, and adds focused regression tests.

## Best Possible Solution

Land after maintainer/API-owner acceptance that omitted title metadata in replacement upserts means preserve existing human titles, while explicit clear behavior stays on `patchSessionEntry({ replaceEntry: true })` or explicit empty/null title fields.

## Reproduction Assessment

Yes from source inspection: current main’s file-store `upsertSessionEntry` persists a cloned replacement entry directly, so omitted `label` and `displayName` fields can erase existing title metadata. I did not execute a local repro because this review is read-only, but the code path and added regression tests are deterministic.

## Solution Assessment

Yes, with maintainer contract acceptance: the store boundary is the narrowest durable fix because every replacement upsert passes through it. The remaining question is compatibility policy for the exported helper, not a concrete patch defect.
