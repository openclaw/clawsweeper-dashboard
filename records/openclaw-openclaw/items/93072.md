## Summary

Keep open for maintainer review: the patch appears focused and correct, while current main and the latest release still have the replacement-upsert title-loss behavior. The remaining concern is compatibility acceptance for an SDK-exposed helper and keeping release/main behavior aligned.

## What This Changes

The PR changes session-store replacement upserts to preserve existing non-empty `label` and `displayName` when incoming entries omit those fields, adds regression tests, and documents the SDK runtime helper contract.

## Best Possible Solution

Land or supersede this narrow store-boundary guard after maintainer/API-owner acceptance of the documented omitted-field contract, then keep main and the release branch aligned.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` persist the cloned replacement entry directly, so storing an entry with `label`/`displayName` and then upserting the same key with those fields omitted can erase the titles; I did not run a mutating local repro in this read-only review.

## Solution Assessment

Yes, with compatibility acceptance. The store upsert boundary is the narrowest durable fix because plugin/runtime replacement writes pass through it, while explicit clear behavior remains available through documented empty-string or patch/replace paths.
