## What This Changes

The branch preserves existing non-empty `label` and `displayName` during replacement `upsertSessionEntry` writes when incoming entries omit those fields, adds focused session-store tests, and documents the SDK helper contract.

## Best Possible Solution

Land the narrow store-boundary guard after maintainer/API-owner acceptance of the documented omitted-field contract, keeping explicit title clearing on the documented empty-string or patch/replace paths.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.8` shows `upsertSessionEntry` persists the cloned replacement entry directly, so a stored entry with titles followed by an upsert that omits title fields can lose them; I did not run a mutating local repro in this read-only review.

## Solution Assessment

Yes, with maintainer compatibility acceptance. The store upsert boundary is the narrowest durable fix because plugin/runtime replacement writes pass through it, while explicit clear behavior remains available through documented empty-string or patch/replace paths.
