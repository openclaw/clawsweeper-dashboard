## Label Justifications

- `impact:data-loss`: The request is about avoiding manual recreation or fragile external merging of persisted cron job definitions during declarative deploys.

## Summary

Close as ClawHub/plugin-scoped: the requested jobs.d workflow is useful, but current core and the latest release now keep cron schedules in SQLite, while existing plugin lifecycle and cron service APIs can support an external directory-sync plugin without adding a second core storage backend.

## What This Changes

The issue requests support for a directory-backed cron job definition store so each job can be managed as an individual JSON file for NixOS/GitOps workflows.

## Best Possible Solution

Keep SQLite as the OpenClaw core cron runtime store and move jobs.d synchronization to a ClawHub code plugin or import/export workflow with explicit conflict, deletion, symlink, and validation semantics.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs checks give high confidence that core jobs.d support is absent and that current cron persistence is SQLite-backed.

## Solution Assessment

No for adding jobs.d as a second core runtime store. The safer maintainable path is a ClawHub plugin or import/export workflow that syncs declarative files through the existing cron APIs while preserving SQLite as the source of truth.
