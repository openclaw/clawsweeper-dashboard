## Summary

Keep this issue open: current main and the latest release still only support an aggregate cron definition file plus the runtime-state sidecar, while the prior directory-store implementation PR is closed unmerged and no open canonical PR now owns the work.

## What This Changes

Add support for loading and managing cron job definitions from a directory of one JSON file per job for declarative GitOps/NixOS setups.

## Best Possible Solution

Define an additive directory-backed cron definition store that stays compatible with existing cron.store/jobs.json users, preserves jobs-state.json, and documents deterministic load, write, delete, migration, symlink, and duplicate-id behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and related-item state give high confidence that directory-backed cron jobs are still absent on current main.

## Solution Assessment

Unclear whether the exact cron.storeDir API is the best final shape. The maintainable direction is a backward-compatible directory-backed definition store, but maintainers need to settle precedence, migration, symlink, filename, and runtime-state semantics first.

## Risks / Open Questions

- The final public API is unresolved: cron.storeDir, a directory derived from cron.store, and an explicit backend mode each have different compatibility and migration implications.
