## Summary

Keep this issue open. Current main and the latest release still expose only the aggregate `jobs.json` cron definition store with a `jobs-state.json` runtime sidecar, while the related directory-store implementation remains open and unresolved at https://github.com/openclaw/openclaw/pull/41206.

## What This Changes

Support a directory-backed cron job store with one JSON file per job for declarative NixOS/GitOps management.

## Best Possible Solution

Add a backward-compatible, documented directory-backed cron definition store that coexists with `jobs.json`, preserves `jobs-state.json`, defines deterministic per-job load/write/delete semantics, and updates schema, help, doctor, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and release inspection give high confidence the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The current open PR should be reviewed, reworked, or replaced before merge so compatibility and storage semantics are settled.
