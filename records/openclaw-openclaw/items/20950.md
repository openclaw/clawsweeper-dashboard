## Summary

Keep open. Current main still supports one aggregate cron definition store plus the runtime-state sidecar, while this issue asks for directory-backed per-job definitions and the discussion identifies open PR #41206 as the active implementation candidate.

## What This Changes

The issue requests optional directory-backed cron job definitions so each job can live in its own JSON file for declarative NixOS/GitOps setups.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace #41206 with an additive directory-store mode that preserves existing cron.store users and documents deterministic per-job loading.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is high-confidence verifiable by current source and docs: config, Gateway startup, store load/save, and docs still use one aggregate definition file rather than a per-job directory.
