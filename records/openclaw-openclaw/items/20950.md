## Summary

Keep this issue open. Current main still has only the aggregate cron definition file plus the runtime-state sidecar, and the open directory-store PR remains the active but unresolved implementation path.

## What This Changes

The issue requests opt-in directory-backed cron job definitions so each scheduled job can live in its own JSON file for GitOps/NixOS workflows.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store backend plus schema, docs, doctor behavior, and focused cron tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection on current main gives high confidence that the requested directory-backed cron definition store is absent.

## Solution Assessment

Unclear whether the exact `cron.storeDir` shape is the best final API, but an additive directory-backed backend is the right implementation direction. The safer path is maintainer review or replacement of the existing open PR so compatibility, state separation, filenames, symlinks, and migration semantics are settled before merge.
