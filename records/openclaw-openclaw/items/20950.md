## Summary

Keep open. Current main and the latest release still do not provide `storeDir`, `jobs.d`, or a per-job JSON reconcile workflow, while the old implementation PR is closed unmerged and current storage policy makes this a maintainer product decision rather than a safe cleanup close.

## What This Changes

The issue requests directory-backed cron job definitions so each scheduled job can live in its own JSON file for NixOS/GitOps workflows.

## Best Possible Solution

Define a maintainer-approved declarative cron import or reconcile path that keeps SQLite as the canonical runtime store while making per-job files a deliberate, documented artifact/source with path-safety, deletion, validation, doctor, schema/help, and focused tests.

## Reproduction Assessment

Yes, at source level: current main has no `storeDir` or jobs.d config and no directory reconcile path, while docs state active cron storage is SQLite-backed. This is a missing feature rather than a failing existing behavior path.

## Solution Assessment

No for the exact steady-state runtime jobs.d store proposal. The safer maintainable direction is an explicit import/reconcile surface that keeps SQLite canonical and makes ownership, deletion, validation, and path boundaries deliberate.
