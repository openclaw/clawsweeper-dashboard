## Summary

Keep this open as the canonical tracker for declarative cron management. Current main and the latest stable release use SQLite as the cron runtime store and provide JSON inspection, but they do not provide a jobs.d directory or a safe declarative import/reconcile workflow.

## What This Changes

The issue requests a per-job cron definitions directory, such as jobs.d, for NixOS/GitOps-style declarative management.

## Best Possible Solution

Keep SQLite as the runtime store and design a declarative import or reconcile surface with explicit identity, dry-run, update, delete, conflict, and failure semantics.

## Reproduction Assessment

Yes for the missing capability by source inspection: current main and v2026.6.5 have SQLite-backed cron storage and JSON inspection commands, but no jobs.d or declarative reconcile surface. There is no failing runtime bug to reproduce because this is a feature request.

## Solution Assessment

No, not as proposed. A steady-state directory backend conflicts with the current SQLite runtime-store direction; a validated reconcile/import workflow appears safer.
