## Summary

Keep open. Current main and v2026.6.8 still use SQLite-backed cron rows keyed by one logical `cron.store` path, with no `jobs.d`, `storeDir`, or per-job declarative file reconcile workflow; the old implementation PR is closed unmerged and related issues point back here as the canonical product-design tracker.

## What This Changes

The issue requests directory-backed cron job definitions so each scheduled job can live in its own JSON file for NixOS/GitOps-style declarative management.

## Best Possible Solution

Provide a maintainer-approved transactional cron import/reconcile surface for per-job files, or an equivalent declarative workflow, while keeping SQLite as the sole runtime store and documenting dry-run, ownership, deletion, validation, and migration behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 document SQLite as canonical cron storage and expose no `jobs.d`, `storeDir`, or per-job file reconcile path. This is a missing feature rather than a failing existing behavior path.

## Solution Assessment

No for the exact steady-state `jobs.d` runtime-store proposal. The safer maintainable direction is an explicit import/reconcile surface that preserves SQLite as the runtime store and makes ownership, deletion, validation, and path boundaries deliberate.
