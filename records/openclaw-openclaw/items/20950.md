## Label Justifications

- `P2`: The request is a useful cron persistence/workflow enhancement with limited blast radius, but it is not urgent runtime breakage.
- `impact:data-loss`: The requested storage shape controls how persisted cron job definitions are loaded, edited, removed, and recovered.

## Summary

Close as ClawHub/plugin-scoped: the requested jobs.d workflow is useful, but current main and the latest release intentionally keep cron runtime state in SQLite, while plugin lifecycle hooks and cron service APIs are sufficient for an external directory-sync plugin without adding a second core storage backend.

## What This Changes

The issue requests a cron jobs.d directory mode with one JSON file per scheduled job for NixOS/GitOps-style declarative cron management.

## Best Possible Solution

Keep SQLite as OpenClaw core's cron runtime store and publish a ClawHub code plugin that syncs a configured jobs.d directory into cron through the gateway cron APIs, with explicit conflict, deletion, symlink, and validation semantics.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs checks give high confidence that core jobs.d support is absent and that current cron runtime persistence is SQLite-backed.

## Solution Assessment

No for adding jobs.d as a second core runtime store. The safer maintainable path is an external ClawHub plugin or import/export workflow that syncs declarative files through the existing cron APIs while preserving SQLite as the source of truth.
