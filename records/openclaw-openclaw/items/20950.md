## Summary

Keep this issue open. Current main and the latest release still expose the aggregate `cron.store` file contract with `jobs.json` plus `jobs-state.json`, and the directory-backed implementation remains unresolved in https://github.com/openclaw/openclaw/pull/41206.

## What This Changes

Request support for a directory-backed cron job store so each cron job can live in its own JSON file for NixOS/GitOps setups.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition store that preserves runtime state and documents the final config contract.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and live related-item state give high confidence the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The open implementation PR should be reviewed, reworked, or replaced before merge so compatibility and storage semantics are settled.
