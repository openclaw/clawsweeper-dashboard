## Summary

Keep this issue open. Current main and v2026.5.12 still expose only the aggregate `cron.store` / `jobs.json` cron definition file plus a `jobs-state.json` runtime sidecar, while the related directory-store implementation candidate remains open and needs maintainer review.

## What This Changes

The issue requests an additive directory-backed cron job store so declarative setups can manage one JSON file per scheduled job.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition store that preserves runtime state and documents the final config contract.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and live related-item state give high confidence that the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The open implementation PR should be reviewed, reworked, or replaced so compatibility and storage semantics are settled before merge.
