## Summary

Keep this issue open. Current main still uses one aggregate cron definition file plus a runtime-state sidecar, and the related directory-store implementation remains open with unresolved storage-contract and review concerns.

## What This Changes

The issue requests an opt-in directory-backed cron job store so each scheduled job can be managed as its own JSON file for NixOS/GitOps setups.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store mode that preserves the current state sidecar, deterministic load/write/delete semantics, schema/help, doctor behavior, docs, and focused cron tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source/docs inspection on current main gives high confidence the requested directory-backed cron store is absent: config, schema, docs, Gateway startup, service load/persist, and store code all use one aggregate store file.

## Solution Assessment

Unclear whether the proposed `cron.storeDir` shape is the best final API, but an additive directory-backed backend is the right implementation direction. The current open PR should be reviewed, reworked, or replaced so compatibility, id-keyed filenames, state separation, symlinks, and migration behavior are settled before merge.
