## Summary

Keep this issue open. Current main and the latest release still expose only the aggregate `jobs.json` cron definition store plus `jobs-state.json`, while the directory-backed one-file-per-job request remains unmerged and tied to an open, conflicting implementation candidate.

## What This Changes

Add a directory-backed cron job definition store so GitOps/NixOS users can manage one JSON file per job alongside or instead of the existing aggregate `jobs.json`.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition store that preserves `jobs-state.json`, documents activation and precedence, and covers load/add/update/remove/migration behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and live related-item state give high confidence that the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. Maintainers should settle config, precedence, migration, symlink, and state-preservation semantics before merging an implementation.
