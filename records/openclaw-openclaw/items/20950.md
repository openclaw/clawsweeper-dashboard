## Summary

Keep this issue open. Current main still has only the aggregate `jobs.json` cron definition store plus the `jobs-state.json` runtime sidecar, while the directory-store work remains an open implementation candidate with unresolved storage-contract concerns.

## What This Changes

The issue requests a `jobs.d`-style cron store so each scheduled job can be managed as an individual JSON file for NixOS/GitOps setups.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store backend that preserves runtime state separation, deterministic load/write/delete semantics, documented symlink behavior, schema/help, doctor behavior, docs, and focused cron tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs inspection on current main gives high confidence the requested directory-backed cron store is absent.

## Solution Assessment

Unclear whether the exact proposed `cron.storeDir` API is the best final shape, but an additive directory-backed backend is the right implementation direction. The current open PR should be reviewed, reworked, or replaced so compatibility, id-keyed filenames, state separation, symlinks, and migration behavior are settled before merge.
