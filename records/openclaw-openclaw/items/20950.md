## Summary

Keep open. Current main still has a single aggregate cron definition store plus a runtime-state sidecar, while this feature asks for directory-backed per-job cron definitions and the related context identifies open PR #41206 as the active implementation candidate.

## What This Changes

Request support for a `jobs.d`-style directory where each cron job definition can live in its own JSON file for NixOS/GitOps workflows.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace #41206 with an additive directory-store mode that preserves existing `cron.store`/`jobs.json` users and documents deterministic per-job loading.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is high-confidence verifiable from current source and docs: config, Gateway startup, store load/save, tests, and docs still use one aggregate definition file rather than a per-job directory.
