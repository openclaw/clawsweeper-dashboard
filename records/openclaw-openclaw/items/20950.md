## Summary

Keep this open as the canonical declarative cron-file tracker. Current main and the latest release still have SQLite as the active cron runtime store with no jobs.d, storeDir, or per-job file reconciliation path, while the earlier implementation PR was closed unmerged and is not a viable canonical fix.

## What This Changes

The issue requests directory-backed cron job definitions so each scheduled job can be managed as its own JSON file for NixOS/GitOps workflows.

## Best Possible Solution

Provide an explicit transactional cron import/reconcile command or Gateway API that consumes per-job files, supports dry-run and deterministic ownership/deletion rules, and leaves SQLite as the sole runtime store.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 document SQLite as canonical cron storage and expose no directory-backed source or declarative import/reconcile workflow. This is a missing feature rather than a failing existing behavior path.

## Solution Assessment

No for the original steady-state jobs.d runtime-store proposal. A transactional import/reconcile interface is the narrower maintainable direction because it avoids dual ownership while preserving the current SQLite runtime contract.
