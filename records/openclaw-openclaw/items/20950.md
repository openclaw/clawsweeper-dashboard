## Summary

Keep this issue open. Current main and the latest release still only support the aggregate `jobs.json` cron definition store plus the `jobs-state.json` sidecar, while the directory-backed one-file-per-job implementation candidate is still open and needs maintainer decisions.

## What This Changes

Request directory-backed cron job storage so each job can live in its own JSON file for NixOS/GitOps declarative setups.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition store that preserves `jobs-state.json` runtime state and documents the final storage contract.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and live related-item state give high confidence that the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. Maintainers should settle config, precedence, migration, symlink, delete/rename, and state-preservation semantics before merging an implementation.
