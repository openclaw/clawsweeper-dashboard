## Summary

Keep this issue open. Current main still only supports a single aggregate cron definition file plus a derived runtime-state sidecar, and open PR #41206 remains the active implementation candidate rather than a reason to close this tracker.

## What This Changes

The issue requests a directory-backed cron job store so each cron job can live in its own JSON file for NixOS/GitOps-style declarative management.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace #41206 with an additive directory-store mode that preserves existing jobs.json behavior and updates config schema/help, doctor/docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested directory-backed cron store path because config, schema, docs, store, and gateway startup all still point at a single store file.

## Solution Assessment

Unclear for the exact config shape, but an additive directory-backed mode is the maintainable direction. The safest next step is to review, land, rework, or replace #41206 while preserving the current file-store and jobs-state sidecar contracts.
