## Summary

Keep this issue open. Current main and the latest release still use the aggregate `jobs.json` cron definition store plus the `jobs-state.json` sidecar, while the related directory-store PR is still open, conflicting, and carrying unresolved storage-contract review concerns.

## What This Changes

Request support for an optional directory-backed cron job store with one JSON file per job for NixOS/GitOps-style declarative management.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store mode plus aligned schema/help, doctor, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and release inspection give high confidence the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` API is the best final shape, but an additive directory-backed backend is the right direction for the GitOps use case. The open implementation candidate should be reviewed, reworked, or replaced before merge so compatibility and storage semantics are settled.
