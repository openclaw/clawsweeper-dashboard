## Summary

Keep open: current main still lacks a directory-backed cron definition store, and the open implementation candidate remains unresolved with review concerns, so closing this would drop the canonical tracker.

## What This Changes

This issue requests a directory-backed cron definition store so each job can live in its own JSON file for NixOS/GitOps-style symlinking.

## Best Possible Solution

Land or replace the existing implementation candidate with a backward-compatible directory store that preserves `jobs.json` users and `jobs-state.json` runtime state, with schema/help/doctor/docs and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection on current main show no directory-backed cron store surface.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but a compatible directory-backed storage mode is the maintainable direction for the GitOps use case. The safest path is maintainer review, rework, or replacement of https://github.com/openclaw/openclaw/pull/41206.
