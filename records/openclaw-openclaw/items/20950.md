## Summary

Keep this issue open. Current main and the latest release still lack a directory-backed cron definition store, while https://github.com/openclaw/openclaw/pull/41206 remains the open implementation candidate and this issue is the canonical tracker for the GitOps/jobs.d storage request.

## What This Changes

Request directory-backed cron job definitions, with one JSON file per job for NixOS/GitOps-style declarative management alongside or instead of the aggregate jobs.json store.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition backend that documents the final config contract and preserves runtime state safely.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and live related-item state give high confidence that the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact cron.storeDir shape is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The active PR or a replacement should settle compatibility, file identity, and runtime-state semantics before merge.
