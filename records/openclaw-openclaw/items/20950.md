## Summary

Close as ClawHub/plugin-scoped: current main and v2026.6.5 have moved cron persistence to SQLite, while a jobs.d file-sync workflow can be built as an external plugin or import/export layer using existing cron APIs without adding a second core runtime store.

## What This Changes

The issue requests directory-backed cron job definitions so each job can be managed as a separate JSON file for NixOS/GitOps workflows.

## Best Possible Solution

Keep SQLite as the OpenClaw core cron runtime store and publish any jobs.d synchronization as a ClawHub code plugin or explicit import/export workflow with clear conflict, deletion, symlink, and validation semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs checks give high confidence that jobs.d support is absent from core and that current cron persistence is SQLite-backed.

## Solution Assessment

No for adding jobs.d as a second core runtime store. The narrower maintainable path is a ClawHub plugin or import/export workflow that syncs declarative files through supported cron APIs while SQLite remains the source of truth.
