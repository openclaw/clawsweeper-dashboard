## Summary

Keep this issue open. Current main and the latest release still use the aggregate `jobs.json` cron definition store plus `jobs-state.json`, while the directory-backed implementation remains unresolved at https://github.com/openclaw/openclaw/pull/41206.

## What This Changes

The issue requests directory-backed cron job definitions so declarative setups can manage one JSON file per job instead of one aggregate `jobs.json` file.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition store that preserves runtime state and documents the final config contract.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, release-tag inspection, and live related-item state give high confidence the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The open implementation PR should be reviewed, reworked, or replaced before merge so compatibility and storage semantics are settled.
