## Summary

Keep open. Current main still supports only a single aggregate cron definition store plus the derived runtime-state sidecar, while the issue asks for directory-backed per-job definitions. The provided discussion also identifies open PR #41206 as the active implementation candidate, so this should remain the tracker for maintainer review rather than be closed or sent to a duplicate repair lane.

## What This Changes

Request support for a directory-backed cron job store so each job can live in its own JSON file for NixOS/GitOps declarative management.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace #41206 with an additive directory-store mode that preserves existing cron.store/jobs.json users, keeps runtime state out of declarative job files, deterministically loads per-job JSON definitions, and updates schema/help, doctor behavior, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is high-confidence verifiable by source and docs inspection: current config/schema, Gateway startup, store load/save, and docs still use one aggregate definition file rather than a per-job directory.

## Solution Assessment

Yes for triage. Keeping this issue open while maintainers review the active implementation candidate is the narrowest maintainable path; a separate automated repair would duplicate already-open work.
