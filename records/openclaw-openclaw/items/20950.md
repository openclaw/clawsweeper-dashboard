## Summary

Keep this issue open. Current main still supports only a single aggregate cron definition file plus a derived runtime-state sidecar; the requested per-job directory store remains unimplemented, and open PR #41206 is the active implementation candidate rather than a reason to close this tracker.

## What This Changes

The issue requests an opt-in cron jobs directory where each job is stored as its own JSON file for NixOS/GitOps-style declarative management.

## Best Possible Solution

Keep this as the canonical tracker and land or rework #41206, or an equivalent additive implementation, that preserves existing jobs.json behavior while adding documented, deterministic per-job directory storage with focused store, Gateway, doctor, docs, and config-schema coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested directory-backed cron store path.

## Solution Assessment

Unclear for the exact config shape, but the maintainable direction is an additive directory-store mode rather than replacing jobs.json. The safest next step is to review, land, or replace #41206 while preserving the current file-store contract.
