## Summary

Keep this issue open. Current main still lacks directory-backed cron job definitions, while https://github.com/openclaw/openclaw/pull/41206 remains the open implementation candidate with unresolved storage-contract and review concerns.

## What This Changes

Request directory-backed cron job definitions so each job can live in its own JSON file for NixOS/GitOps symlink management.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store mode that preserves existing `jobs.json` users and the runtime-state sidecar while documenting deterministic load/write/delete, symlink, migration, schema, doctor, and test behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high confidence the requested directory-backed cron store is absent because config, schema, docs, Gateway startup, service load/persist, and store code still use one aggregate store file.

## Solution Assessment

Unclear whether the proposed `cron.storeDir` shape is the best final API, but an additive directory-backed backend is the maintainable direction. The current open PR should be reviewed, reworked, or replaced so compatibility, id-keyed filenames, state separation, symlinks, and migration behavior are settled before merge.
