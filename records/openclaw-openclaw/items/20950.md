## Summary

Keep this issue open as the canonical tracker for declarative cron management. Current main still lacks a supported jobs.d, managed-source, import/reconcile, or equivalent GitOps workflow; the earlier implementation PR closed unmerged, and a later related report was explicitly routed back here. The original directory-backed runtime design now needs reconsideration because cron persistence has moved to canonical SQLite storage.

## What This Changes

Add a declarative per-job cron definition directory so NixOS and GitOps setups can manage jobs as individual version-controlled files.

## Best Possible Solution

Keep SQLite as the sole runtime owner and define an explicit declarative import/reconcile or export interface for GitOps, including deterministic identity, validation, deletion policy, drift reporting, and runtime-state preservation.

## Reproduction Assessment

Not applicable; this issue requests a new supported declarative-management mode rather than reporting a failure of documented behavior.

## Solution Assessment

No for the original direct jobs.d runtime backend. Under the current SQLite architecture, a deliberate declarative import or reconciliation boundary is safer and more maintainable than introducing a second persistence backend.
