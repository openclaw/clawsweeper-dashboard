## Summary

Keep this issue open. Current main still exposes only flat `cron.store`/`jobs.json` definitions plus `jobs-state.json`, while https://github.com/openclaw/openclaw/pull/41206 remains the unmerged implementation candidate with open review concerns.

## What This Changes

The issue requests optional `jobs.d` or directory-backed cron job definitions so declarative setups can manage one JSON file per job.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store mode that preserves `jobs.json` users and the runtime-state sidecar.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested directory-backed cron store because config, schema, docs, store, Gateway startup, and service persistence all still use one store file.

## Solution Assessment

Unclear for the exact config, symlink, and migration shape, but an additive directory-backed mode is the maintainable direction. The safest path is maintainer review and rework of https://github.com/openclaw/openclaw/pull/41206 or an equivalent replacement that preserves current `jobs.json` and `jobs-state.json` contracts.
