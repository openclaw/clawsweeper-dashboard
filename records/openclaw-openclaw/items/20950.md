## Summary

Keep this issue open for product review. Current main no longer uses `jobs.json` as the runtime source of truth, so the proposed `jobs.d` backend is obsolete, but the underlying declarative NixOS/GitOps workflow remains unsupported; the previous implementation at https://github.com/openclaw/openclaw/pull/41206 closed unmerged. Likely related people include steipete and Feelw00 with high confidence, plus tyler6204 with medium confidence based on cron persistence history.

## What This Changes

Add declarative one-file-per-cron-job management for NixOS and GitOps deployments.

## Best Possible Solution

Keep SQLite as the sole runtime store and design a documented declarative apply/sync command that accepts a manifest or directory as an input artifact, reconciles desired definitions through the existing cron service, and preserves runtime state.

## Reproduction Assessment

Not applicable: this requests a new declarative management capability rather than reporting broken existing behavior; current SQLite and CLI/API behavior is clear from source and documentation.

## Solution Assessment

No: direct directory-backed loading at Gateway startup is no longer the best solution because runtime persistence is intentionally SQLite-only; an explicit desired-state apply/sync boundary would meet the GitOps goal without creating competing stores.
