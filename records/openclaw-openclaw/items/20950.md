## Summary

Keep this open as the canonical declarative cron-file tracker. Current main and the latest release still use SQLite-backed cron rows keyed by a single logical `cron.store` path, the prior directory-store PR is closed unmerged, and the remaining work needs maintainer product direction rather than cleanup closure.

## What This Changes

Request support for a declarative per-job cron definitions directory, such as `jobs.d` or `cron.storeDir`, for GitOps/NixOS setups.

## Best Possible Solution

Provide an explicit transactional cron import/reconcile command or Gateway API that consumes per-job files with dry-run and deterministic ownership/deletion rules while keeping SQLite as the sole runtime store.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 document SQLite as canonical cron storage and expose no `jobs.d`, `storeDir`, or per-job file reconcile path. This is a missing feature rather than a failing existing behavior path.

## Solution Assessment

No for the exact steady-state `jobs.d` runtime-store proposal. The safer maintainable direction is an explicit import/reconcile surface that preserves SQLite as the runtime store and makes ownership, deletion, validation, and path boundaries deliberate.
