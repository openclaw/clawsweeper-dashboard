## Summary

Keep open as the canonical declarative cron-file tracker. Current main and latest release still use a single logical cron store key backed by SQLite, expose no `jobs.d`/`storeDir` directory source, and the prior implementation PR is closed unmerged, so the remaining work needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests one JSON file per cron job, likely through a `jobs.d` directory or `cron.storeDir`, to support NixOS/GitOps declarative cron setup.

## Best Possible Solution

Define a maintainer-approved declarative cron import/reconcile surface, or a directory mode if maintainers choose it, that keeps SQLite as canonical runtime storage unless product direction changes and includes deterministic identity, deletion, validation, doctor/docs, and focused Gateway/store tests.

## Reproduction Assessment

Yes, at source level for the missing capability: current main and v2026.6.8 expose no `jobs.d`, `storeDir`, or per-job file reconcile path while docs state SQLite is the active cron store. This is a feature gap rather than a failing existing behavior path.

## Solution Assessment

No for the exact steady-state `jobs.d` runtime-store proposal. The safer maintainable solution is a deliberate import/reconcile contract, or an explicitly approved storage-direction change, so SQLite runtime state does not drift from file-managed definitions.
