## Summary

Keep open. Current main and v2026.6.10 still have no `storeDir`, `jobs.d`, or per-job JSON reconcile path; cron jobs are SQLite-backed under one logical `cron.store` key, and the old implementation PR is closed unmerged with correctness/proof/security blockers.

## What This Changes

Request support for directory-backed cron job definitions so GitOps/NixOS users can manage one JSON file per scheduled job.

## Best Possible Solution

Define a maintainer-approved declarative cron import or reconcile path that keeps SQLite canonical while making per-job files an explicit, validated, documented artifact/source.

## Reproduction Assessment

Yes, at source level: current main has no `storeDir`/`jobs.d` config or directory reconcile path, and the docs say active cron storage is SQLite-backed. This is a missing feature, not a failing existing behavior path.

## Solution Assessment

No for the exact steady-state runtime `jobs.d` store proposal. The safer maintainable direction is an explicit import/reconcile surface that keeps SQLite canonical and makes ownership, deletion, validation, and path boundaries deliberate.
