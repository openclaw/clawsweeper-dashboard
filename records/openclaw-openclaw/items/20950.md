## Summary

Keep this issue open. Current main still exposes only the aggregate `jobs.json` cron definition store plus the derived `jobs-state.json` runtime sidecar, and the directory-backed implementation remains unresolved at https://github.com/openclaw/openclaw/pull/41206.

## What This Changes

The issue requests support for directory-backed cron job definitions so each job can live in its own JSON file for declarative GitOps/NixOS setups.

## Best Possible Solution

Add a backward-compatible, documented directory-backed cron definition store that coexists with `jobs.json`, preserves `jobs-state.json`, defines deterministic per-job load/write/delete semantics, and updates schema, help, doctor, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and release-tag inspection give high confidence the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The open implementation PR should be reviewed, reworked, or replaced before merge so compatibility and storage semantics are settled.
