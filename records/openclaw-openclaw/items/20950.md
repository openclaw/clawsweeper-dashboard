## Summary

Keep this issue open. Current main and the latest release still use the aggregate `jobs.json` cron definition store plus a `jobs-state.json` sidecar, while the related directory-store implementation remains open with unresolved storage-contract review concerns.

## What This Changes

Request support for a directory-backed cron job store where each job can live in its own JSON file for NixOS/GitOps-style declarative management.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-store mode plus aligned schema/help, doctor, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and release inspection give high confidence the requested directory-backed cron store is absent: config, schema, docs, Gateway startup, service load/persist, and store code all use one aggregate store file.

## Solution Assessment

Unclear whether the exact `cron.storeDir` API is the best final shape, but an additive directory-backed backend is the right implementation direction for the GitOps use case. The current open PR should be reviewed, reworked, or replaced before merge so compatibility, filenames, state separation, symlinks, and migration behavior are settled.
