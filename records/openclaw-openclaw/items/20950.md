## Summary

Keep this issue open as the canonical tracker for declarative cron management, but do not implement the proposed `jobs.d` runtime backend as written: current main now makes shared SQLite the sole cron store, the prior directory-store PR closed unmerged, and the remaining work requires a maintainer product decision on a safe import/reconciliation contract.

## What This Changes

Add a declarative one-file-per-job cron workflow for NixOS and GitOps deployments.

## Best Possible Solution

Keep SQLite as the sole runtime store and design an explicit, validated, idempotent cron reconcile/import command that consumes a Git-managed directory or manifest, shows a dry-run plan, and defines stable identity, update, deletion, conflict, and failure semantics.

## Reproduction Assessment

Not applicable. Current main’s SQLite-backed behavior is documented and source-verifiable; the issue requests a new declarative management capability rather than reporting broken existing behavior.

## Solution Assessment

No, not as proposed. A steady-state directory backend conflicts with the canonical SQLite architecture; an explicit one-way reconciliation/import surface is the narrower maintainable solution.
