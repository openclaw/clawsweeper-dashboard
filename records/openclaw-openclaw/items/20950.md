## Summary

Keep this issue open: current main and the latest release still lack a directory-backed cron definition store, and the prior implementation PR is now closed unmerged, leaving a real but product-sensitive cron storage enhancement unresolved.

## What This Changes

The issue requests a cron `jobs.d` or similar directory mode where each scheduled job can be stored as a separate JSON file for declarative NixOS/GitOps management.

## Best Possible Solution

Define and document an additive directory-backed cron definition store that remains compatible with existing `cron.store` and `jobs.json` users, preserves `jobs-state.json`, and covers deterministic load, write, delete, migration, and symlink behavior with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and related-item state give high confidence that directory-backed cron jobs are still absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` API is the best final shape, but a backward-compatible directory-backed store is the maintainable direction for the GitOps use case. Maintainers need to settle precedence, migration, symlink, and runtime-state semantics before implementation.
