## Summary

Keep this issue open. Current main and v2026.5.12 still expose only the aggregate `cron.store`/`jobs.json` contract plus the `jobs-state.json` sidecar; the directory-backed implementation remains open at https://github.com/openclaw/openclaw/pull/41206 and needs maintainer product review.

## What This Changes

Request directory-backed per-cron-job JSON files for NixOS/GitOps-style declarative cron definitions alongside or instead of the current aggregate `jobs.json` store.

## Best Possible Solution

Keep this as the canonical tracker and land, rework, or replace https://github.com/openclaw/openclaw/pull/41206 with a backward-compatible directory-backed cron definition store that preserves runtime state and documents the final config contract.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, latest-release inspection, and live related-item state give high confidence the requested directory-backed cron store is absent on current main.

## Solution Assessment

Unclear whether the exact `cron.storeDir` key is the best final API, but an additive directory-backed backend is the right direction for the GitOps use case. The open implementation PR should be reviewed, reworked, or replaced so compatibility and storage semantics are settled before merge.
