## Summary

Keep this open as the canonical declarative-cron tracker. Current main still lacks a per-job directory or reconciliation workflow, but cron persistence is now canonically SQLite-backed, so the original parallel `jobs.d` runtime-store design needs a maintainer-approved replacement contract.

## What This Changes

Add declarative per-job cron definitions suitable for NixOS and GitOps instead of requiring imperative management of the canonical runtime store.

## Best Possible Solution

Provide an explicit transactional cron import/reconcile command or Gateway API that consumes per-job files, supports dry-run and deterministic ownership/deletion rules, and leaves SQLite as the sole runtime store.

## Reproduction Assessment

Yes at source level: current main documents SQLite as canonical and exposes no directory-backed source or declarative import/reconcile workflow, so the requested GitOps capability is demonstrably absent.

## Solution Assessment

No, the original steady-state `jobs.d` store is no longer the best solution because it conflicts with the canonical SQLite storage policy. An explicit one-way reconciliation interface is narrower and avoids dual ownership.
