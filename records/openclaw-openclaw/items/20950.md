## Summary

Keep open. Current main and v2026.6.10 still use SQLite-backed cron rows keyed by one logical `cron.store` path, with no `storeDir`, `jobs.d`, or per-job file reconcile workflow; the related implementation PR is closed unmerged, so this remains the canonical product-design tracker.

## What This Changes

Requests a declarative cron `jobs.d` or per-job JSON directory so NixOS/GitOps setups can manage one file per scheduled job.

## Best Possible Solution

Define a maintainer-approved declarative cron import or reconcile path that preserves SQLite as the runtime store while making per-job files an explicit artifact/source with documented validation, deletion, path-safety, doctor, schema/help, and test coverage.

## Reproduction Assessment

Yes, at source level: current main has no `storeDir`/`jobs.d` config or directory reconcile path, and the docs say active cron storage is SQLite-backed. This is a missing feature, not a failing existing behavior path.

## Solution Assessment

No for the exact steady-state runtime `jobs.d` store proposal. The safer maintainable direction is an explicit import/reconcile surface that keeps SQLite canonical and makes ownership, deletion, validation, and path boundaries deliberate.
