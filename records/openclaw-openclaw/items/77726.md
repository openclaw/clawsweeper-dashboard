## Summary

Keep open: the PR targets a real iOS/LAN onboarding problem, but the current patch weakens the setup-code security model, is missing required release notes, leaves public QR docs inconsistent, and does not include real iOS onboarding proof.

## What This Changes

The PR changes setup-code payloads from bootstrap auth to shared token/password auth, updates TypeScript and Swift auth precedence, permits iOS local-network ws:// inputs, and adds targeted tests plus pairing docs.

## Best Possible Solution

Preserve a bounded short-lived setup credential model, fix the iOS LAN ws:// and mixed-auth precedence paths with complete docs/changelog coverage, and require real iOS onboarding proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main forces manual iOS non-loopback hosts through TLS and DeepLinks rejects non-loopback ws://, while setup-code generation emits bootstrapToken only. I did not run a live iOS device flow.

## Solution Assessment

No. The local-network ws:// and password-over-bootstrap changes are narrow, but replacing short-lived bootstrap setup codes with shared gateway secrets is not the safest fix without explicit maintainer security approval.
