Keep open: current main still leaves the substantive Haiku/default-allowlist regression unresolved; the later cron diagnostic fix only made the rejection clearer.

## What This Changes

The issue asks OpenClaw to preserve lightweight Anthropic Haiku cron workflows after v2026.5.4 by clarifying valid model slugs and deciding on allowlist or migration behavior.

## Best Possible Solution

Choose and document a supported Haiku path, likely `anthropic/claude-haiku-4-5` until Anthropic publishes a newer Haiku slug, then implement the corresponding upgrade, doctor, docs, and default-allowlist policy deliberately.

Yes, source-reproducible: a cron `payload.model` not present in `agents.defaults.models` reaches `resolveAllowedModelRef` and fails before the run, while current Anthropic API-key defaults do not backfill Haiku. I did not execute a live cron job because this review is read-only.

## Solution Assessment

Unclear: the diagnostic-only fix in https://github.com/openclaw/openclaw/issues/79058 is useful but not sufficient, and adding `claude-haiku-4-7` is not supported by current repo or Anthropic docs. A maintainer should pick the migration/default/docs policy first.
