## Summary

Keep open: current main still routes all disconnected Control UI renders to the login gate, and PR #61345 remains the active open implementation candidate for preserving the dashboard during reconnects.

## What This Changes

The issue requests preserving the Control UI dashboard and prior messages during transient gateway disconnects, with active controls disabled and a reconnect path instead of replacing the app with the login gate.

## Best Possible Solution

Land PR #61345 or an equivalent Control UI change that sends only true login-required auth/config/device-auth failures to the login gate while keeping the dashboard shell visible during transient reconnect states.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: `connectGateway` sets `host.connected = false` during reconnect and close paths, and `renderApp` immediately returns the login gate for any false connected state before chat/dashboard can render.

## Solution Assessment

Yes. The best direction is to distinguish login-required failures from transient reconnect states in the Control UI and rely on the existing disconnected dashboard controls, without changing gateway protocol semantics.
