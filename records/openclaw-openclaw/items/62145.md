## Summary

Keep open. Current main still replaces the entire Control UI with the login gate on any Gateway disconnect, so the reported workflow problem remains valid. A related open PR already targets the same login-gate reconnect behavior, so this should stay open while that PR is reviewed, landed, or replaced.

## What This Changes

Request to keep prior Control UI messages readable after a Gateway disconnect by disabling active controls and offering an explicit reconnect path instead of replacing the whole UI with the login gate.

## Best Possible Solution

Review and land PR #61345 or an equivalent narrow Control UI fix that keeps the initial credential/login flow intact, preserves the cached dashboard/chat after a successful connection drops, disables mutating controls while offline, and provides a clear reconnect/login action.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a Gateway WebSocket close sets `connected` false in `app-gateway.ts`, and `renderApp` immediately short-circuits to the login gate, so prior chat content is no longer rendered.

## Solution Assessment

Yes. The maintainable solution is a UI state/rendering change that relaxes the top-level login-gate condition only after a previous successful connection and reuses existing disconnected/disabled chat controls; backend auth and protocol behavior should remain unchanged.
