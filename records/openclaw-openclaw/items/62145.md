## Summary

Keep open: current main still sends every disconnected Control UI render through the login gate, so the reported workflow remains, and PR #61345 is the active unmerged implementation candidate for the same reconnect behavior.

## What This Changes

The issue requests preserving prior Control UI messages during transient gateway disconnects instead of replacing the whole app with the login gate.

## Best Possible Solution

Review and land PR #61345, or an equivalent Control UI patch that sends only true login-required auth/config/device-auth failures to the login gate while keeping the dashboard shell visible during reconnect states.

## Reproduction Assessment

Yes. On current main, `connectGateway` sets `connected = false` during reconnect/close paths, and `renderApp` immediately returns the login gate for any disconnected state.

## Solution Assessment

Yes. The narrow maintainable direction is to distinguish login-required failures from transient reconnect states in the Control UI gate, then rely on existing disconnected-control behavior inside the dashboard.
