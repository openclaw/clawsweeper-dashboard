## Summary

Keep open: current main still swaps to the login gate whenever the Control UI is disconnected, and the related implementation candidate remains open and unmerged.

## What This Changes

The issue requests that the Control UI preserve the visible dashboard/messages during gateway disconnects, disable active controls, and offer a reconnect/login path instead of replacing the whole UI immediately.

## Best Possible Solution

Review and land PR #61345, or an equivalent Control UI change that routes only true login-required failures to the login gate while keeping the dashboard shell visible for transient reconnects.

## Reproduction Assessment

Yes. A source-level reproduction is clear: force `state.connected` false through a gateway close or initial reconnect and `renderApp` returns the login gate before rendering the dashboard; I did not run a browser/live test in this read-only pass.

## Solution Assessment

Yes, the described direction is the narrowest maintainable fix: distinguish login-required auth/config/device-auth failures from transient reconnect states in the UI gate, then rely on existing disabled-control behavior inside the dashboard.
