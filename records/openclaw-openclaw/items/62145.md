## Summary

Keep open: current main still routes every disconnected Control UI state to the login gate, and open PR #61345 remains the active implementation candidate rather than a merged fix.

## What This Changes

Request to keep prior Control UI messages visible during transient gateway disconnects by disabling active controls and offering an explicit reconnect/login path.

## Best Possible Solution

Land PR #61345 or an equivalent Control UI change that sends only true auth/config/device-auth failures to the login gate while leaving the dashboard shell and prior messages visible for transient reconnects.

## Reproduction Assessment

Yes. A source-level reproduction is clear: force `state.connected` false through `connectGateway` startup or close handling and `renderApp` returns the login gate before rendering the dashboard; no browser/live run was performed in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable direction is to distinguish login-required failures from reconnect states in the UI gate and rely on the existing disabled-control behavior inside the dashboard.
