## Summary

Keep open: current main still routes every disconnected Control UI render to the login gate, and open PR #61345 remains the active implementation candidate for the dashboard-preserving reconnect behavior.

## What This Changes

The request asks the Control UI to keep prior dashboard/chat content visible during gateway disconnects instead of replacing the whole app with the login gate.

## Best Possible Solution

Review and land PR #61345, or an equivalent Control UI change that routes only true login-required failures to the login gate while preserving the dashboard shell for reconnect states.

## Reproduction Assessment

Yes. Force `state.connected` false through `connectGateway` startup or close handling and `renderApp` returns `renderLoginGate` before the dashboard/chat branch; this is a high-confidence source-level reproduction.

## Solution Assessment

Yes. The narrow maintainable direction is to distinguish login-required auth/config/device-auth failures from transient reconnect states at the top-level gate, then rely on existing disabled-control paths inside the dashboard.
