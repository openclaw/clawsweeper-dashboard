## Summary

Keep open: current main still stamps auto-managed config metadata after CLI mutation handling, then reports an `Updated` success line, so this is a valid narrow CLI/config behavior bug distinct from https://github.com/openclaw/openclaw/issues/79856 and https://github.com/openclaw/openclaw/pull/80106.

## What This Changes

The issue requests that `openclaw config set` stop reporting success when writes to auto-managed `meta.lastTouchedVersion` and `meta.lastTouchedAt` are overwritten by config stamping.

## Best Possible Solution

Reject `config set` and shared mutation attempts for stamped auto-managed `meta.*` fields before writing, using the existing schema-help semantics and focused regression coverage to prevent false success output.

## Reproduction Assessment

Yes. A high-confidence path is to run `openclaw config set meta.lastTouchedVersion BOGUS-NOT-A-VERSION` or `openclaw config set meta.lastTouchedAt 1999-01-01T00:00:00.000Z` against a temp config and inspect the persisted file; source inspection shows the user value is overwritten during serialization.

## Solution Assessment

Yes. The narrowest maintainable fix is to block these documented auto-managed fields at the CLI/config mutation boundary, preserving `stampConfigVersion` as the system-owned writer instead of trying to make user values win.
