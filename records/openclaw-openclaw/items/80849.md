## Summary

Keep open: current main still auto-stamps `meta.lastTouchedVersion` and `meta.lastTouchedAt` after config mutation handling and still emits the generic `Updated ...` success line, while the linked fixing PR is open and unmerged.

## What This Changes

The issue asks OpenClaw to stop reporting successful `config set` writes for auto-managed `meta.lastTouchedVersion` and `meta.lastTouchedAt` values that are overwritten during config serialization.

## Best Possible Solution

Review and land the linked CLI-boundary rejection in https://github.com/openclaw/openclaw/pull/80856, keeping `stampConfigVersion` system-owned and adding focused config CLI regression coverage.

## Reproduction Assessment

Yes. A high-confidence path is to run `openclaw config set meta.lastTouchedVersion BOGUS-NOT-A-VERSION` or `openclaw config set meta.lastTouchedAt 1999-01-01T00:00:00.000Z` against a temp config and inspect the persisted file; source inspection shows the user value is overwritten during serialization.

## Solution Assessment

Yes. The narrowest maintainable fix is to reject these documented auto-managed leaf paths at the CLI/config mutation boundary rather than letting user input compete with the system version stamp.
