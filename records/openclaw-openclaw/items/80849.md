## Summary

Keep open: current main still has the false-success config path, and an open closing PR already targets the narrow fix, so this issue should wait for that PR to merge or be closed.

## What This Changes

The issue reports that `openclaw config set` accepts auto-managed `meta.lastTouchedVersion` and `meta.lastTouchedAt` writes, prints success, and then the write pipeline overwrites the requested values.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80856 or an equivalent fix that rejects stamped metadata paths before config writes and adds focused CLI regression coverage.

## Reproduction Assessment

Yes. Current-main source shows user mutations are prepared before serialization, then `stampConfigVersion` overwrites both fields, while the CLI success message is emitted after any write rather than after confirming the requested value persisted.

## Solution Assessment

Yes. Rejecting these documented auto-managed fields at the CLI/config mutation boundary is the narrowest maintainable fix; preserving user values would conflict with the writer-owned metadata contract, and warning while still printing `Updated` would leave the false-success behavior partly intact.
