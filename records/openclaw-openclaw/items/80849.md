## Summary

Keep open: current main still overwrites the requested auto-managed config metadata during serialization and then reports a successful update, and the related default-value persistence fix does not cover this final stamp override.

## What This Changes

The issue asks OpenClaw to stop reporting successful `config set` writes for `meta.lastTouchedVersion` and `meta.lastTouchedAt` when those auto-managed values are replaced before persistence.

## Best Possible Solution

Reject `config set` and equivalent shared mutation requests for stamped auto-managed `meta.*` fields before writing, with focused regression coverage proving the CLI no longer emits a false success message.

## Reproduction Assessment

Yes. A high-confidence path is to set either stamped `meta.*` field and inspect the persisted file; source inspection shows the value is replaced in `stampConfigVersion` before serialization while the CLI reports success.

## Solution Assessment

Yes. The narrowest maintainable fix is to block these documented auto-managed fields at the config mutation boundary, preserving `stampConfigVersion` as the system-owned writer instead of making user values win.
