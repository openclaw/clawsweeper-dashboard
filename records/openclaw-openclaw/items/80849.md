## Summary

Keep open: current `main` still accepts writes to the auto-managed meta fields, then overwrites them during serialization while the CLI reports success.

## What This Changes

The issue asks for `openclaw config set` on `meta.lastTouchedVersion` and `meta.lastTouchedAt` to stop reporting a successful update when those auto-managed values are discarded.

## Best Possible Solution

Reject `config set` and `config patch` attempts to mutate auto-managed meta stamp fields before writing, keep `stampConfigVersion` authoritative, and cover the behavior with focused CLI tests plus a changelog entry.

## Reproduction Assessment

Yes, at source level: running `openclaw config set meta.lastTouchedVersion ...` or `meta.lastTouchedAt ...` reaches the normal config write path, and current code overwrites those fields immediately before serialization. I did not execute the write locally because the sweep is read-only.

## Solution Assessment

Yes: the narrow maintainable fix is to block or clearly fail auto-managed meta paths at the config mutation boundary, similar to the existing managed `plugins.installs` guard, rather than letting user values reach `stampConfigVersion`.
