## Summary

Keep open: this is a useful security-hardening PR, but it has the protected maintainer label, lacks contributor real-behavior proof, and the latest head still introduces auth persistence/security blockers that should stop merge.

## What This Changes

The PR adds shared structured secret redaction to logging, config audit, trajectory, and transcript persistence, adds OpenAI Codex OAuth sidecar secret persistence, changes daemon log handling, and includes oc-path formatting churn.

## Best Possible Solution

Land a focused persistence-redaction hardening patch after owner-approved auth storage semantics, explicit migration boundaries, hardened sidecar key handling, real runtime proof, and trimmed unrelated churn.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main directly persists config audit records and session transcript messages, and trajectory sanitization lacks the PR's broader token-shape redaction; I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No, not yet. Shared redaction at persistence boundaries is the right direction, but normal load-time auth rewrites and the sidecar key-derivation model need maintainer/security review before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Move auth-store rewriting out of normal loads:** `src/agents/auth-profiles/store.ts:521-523`
  - body: Normal non-read-only auth-store loads now opt into `rewriteInlineOAuthSecrets`, so startup/status/runtime reads can create sidecar key material and rewrite `auth-profiles.json`. Keep save-time redaction, but move one-way migration to an explicit save/doctor path or owner-approved migration flow.
  - confidence: 0.82
- **[P2] Harden OAuth sidecar key derivation:** `src/agents/auth-profiles/persisted.ts:369-374`
  - body: The OAuth sidecar AES key is derived with a fast SHA-256 hash over `OPENCLAW_AUTH_PROFILE_SECRET_KEY` or generated seed material. If the env value is passphrase-like, the encrypted sidecar can be brute-forced cheaply; require random key material or use an approved password KDF before storing tokens this way.
  - confidence: 0.68
