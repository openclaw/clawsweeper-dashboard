## Summary

Keep open. The PR addresses a source-reproducible secret-persistence gap, but it carries the protected `maintainer` label, lacks required external real behavior proof, and still introduces security-sensitive auth-store storage behavior that needs maintainer review.

## What This Changes

The branch adds structured secret redaction to persisted audit, log, trajectory, and transcript sinks, broadens token-shape patterns, moves OpenAI Codex OAuth token material into encrypted sidecar files, and includes daemon log handling plus oc-path formatting changes.

## Best Possible Solution

Keep the shared persistence-boundary redaction, move OAuth sidecar migration to an explicit save/doctor or owner-approved path with hardened key handling, trim unrelated churn, and merge only after maintainer review plus real proof.

## Reproduction Assessment

Yes, source-reproducible. Current main persists config audit records and transcript messages directly, and trajectory runtime lacks the PR's broader token-shape redaction pass; I did not live-run a gateway scenario in this read-only review.

## Solution Assessment

No, not yet. Persistence-boundary redaction is the right direction, but normal auth-store loads should not rewrite credential storage and the new encryption key derivation plus proof gates need resolution first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.79

Full review comments:

- **[P2] Move auth-store rewriting out of normal loads:** `src/agents/auth-profiles/store.ts:521-523`
  - body: Normal non-read-only loads now pass `rewriteInlineOAuthSecrets`, so startup/status/runtime reads can create key material, sidecar files, and rewrite `auth-profiles.json`. Keep save-time redaction, but run this one-way migration from an explicit save/doctor path or after owner approval.
  - confidence: 0.8
- **[P2] Harden OAuth secret key derivation:** `src/agents/auth-profiles/persisted.ts:371`
  - body: The new AES-GCM key is a single SHA-256 digest of `OPENCLAW_AUTH_PROFILE_SECRET_KEY` or another seed. If that seed is password-like, the sidecar files can be brute-forced cheaply; require random key material or derive with an approved password KDF before storing tokens this way.
  - confidence: 0.62
