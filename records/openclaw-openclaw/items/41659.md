## Summary

Keep open: the configured Control UI basePath favicon bug is still source-reproducible on current main, and the PR has useful proof, but the current patch inserts configurable basePath text into HTML attributes without escaping or validation.

## What This Changes

The PR adds gateway-side rewriting of Control UI favicon and apple-touch-icon hrefs when a controlUi basePath is configured, plus focused gateway tests.

## Best Possible Solution

Port the mounted favicon/apple-touch-icon handling onto current main, escape or reject unsafe basePath text before HTML output, and preserve the existing CSP hash behavior.

## Reproduction Assessment

Yes, source-level. Current main emits root-relative icon hrefs, and basePath routing excludes `/favicon.svg` from a `/webchat` mount, so the browser request bypasses the Control UI handler.

## Solution Assessment

Yes, with changes. Rewriting only favicon and apple-touch-icon hrefs at index serving time is narrow for configured basePath, but the implementation must escape or validate basePath and retain current CSP hash handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Escape basePath before injecting favicon hrefs:** `src/gateway/control-ui.ts:244`
  - body: The helper writes normalized `gateway.controlUi.basePath` directly inside a double-quoted `href`. The schema accepts arbitrary strings and normalization only trims/slashes, so quote-bearing values can break the HTML attribute context when serving the index; escape the attribute value or reject unsafe basePath input before rewriting.
  - confidence: 0.79
