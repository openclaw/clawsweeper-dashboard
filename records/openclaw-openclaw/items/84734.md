## Summary

Keep open: current main still has a source-level path where the agent-facing cron tool auto-scopes `cron.list` by the requester agent while the CLI list path is unscoped unless `--agent` is passed, so a jobs store can be visible to `openclaw cron list --all` but empty through the agent tool. I did not find a canonical duplicate or merged fix; the exact reporter store shape, job `agentId` values, and requester session key are still needed before changing the contract.

## What This Changes

The issue asks OpenClaw to make the agent-facing cron list tool return existing jobs from `jobs.json` instead of an empty list when the CLI can see those jobs.

## Best Possible Solution

Define the cron tool visibility contract first, then either align the agent tool with CLI all-jobs behavior or make the scoped result explicit and provide a safe all-agent lookup path with regression coverage.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path where `cron(action="list")` forwards the requester agent id and the gateway filters out jobs for other/default agents while CLI list stays unfiltered. I did not run a live gateway against the reporter's private store, so the exact 43-job setup is not fully reproduced.

## Solution Assessment

Unclear: the current code and tests intentionally scope agent cron lists by session agent, so a maintainer decision is needed before treating CLI parity or an all-agent list mode as the right fix. A safe solution should preserve explicit `agentId` filtering and cover both scoped and unscoped behavior.
