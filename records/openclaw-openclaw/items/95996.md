## Best Possible Solution

Fix the classifier so `phase: "error"` remains terminal while yielded `phase: "end"` is waiting, then add redacted Discord behavior proof and get maintainer acceptance for the new public cron status and SDK/hook contract before merge.

## Reproduction Assessment

Yes. The PR-introduced blocker is source-reproducible because the new classifier returns `terminal` for the same yielded `phase: "end"` payload that the added test expects to be `yielded_waiting`; current main's cron and Discord gaps are also clear from source inspection.

## Solution Assessment

No as-is. The shared classifier is a plausible owner-boundary seam, but it must classify yielded end events as waiting while keeping error events terminal; after that, the Discord path and public cron/SDK compatibility still need proof and maintainer acceptance.
