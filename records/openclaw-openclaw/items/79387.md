## Summary

Keep open: this maintainer-labeled feature PR addresses the right iMessage gap, but the patch has blocking cursor correctness issues, a low-severity local retention concern, and no after-fix live outage proof.

## What This Changes

Adds opt-in iMessage gateway-downtime catchup with config/schema, cursor and replay helpers, monitor wiring through imsg history RPCs, longer echo-cache retention, tests, docs, generated config hashes, and a changelog entry.

## Best Possible Solution

Land an iMessage-owned catchup implementation only after it preserves retryable failures, prevents duplicate replay of live-handled messages, scopes the retention tradeoff, and includes redacted live macOS outage proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review blockers: source inspection shows a fail-then-success row sequence can advance the cursor past the failed row, and current main still documents no iMessage catchup path; I did not run a live macOS outage.

## Solution Assessment

Is this the best way to solve the issue? No. The extension-local catchup direction is right, but this patch needs cursor/dedupe fixes, a retention decision, and real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep the cursor behind retryable failures:** `extensions/imessage/src/monitor/catchup.ts:382-384`
  - body: When a dispatch returns false below `maxFailureRetries`, the loop records the failure but keeps processing later rows. If any later row succeeds, `lastSeenRowid` is saved past the failed row, so the next startup treats that failed message as pre-cursor and never retries it. Stop after the first retryable failure or compute the saved cursor from the earliest retryable failure.
  - confidence: 0.95
- **[P2] Advance the catchup cursor from live messages:** `extensions/imessage/src/monitor/monitor-provider.ts:875-891`
  - body: The new cursor only advances during the startup catchup pass. iMessage current-main does not have a persistent inbound GUID dedupe, so messages handled normally while the gateway is up remain above the cursor and will be replayed as fresh turns on the next restart before the actual downtime backlog. Update the cursor from successful live handling or add a persistent inbound replay dedupe before enabling this path.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: No supply-chain change was found, but the diff expands local iMessage sent-message retention from 2 minutes to 12 hours for all users.

Concerns:

- **[low] Scope longer echo retention to catchup:** `extensions/imessage/src/monitor/persisted-echo-cache.ts:19`

The file remains owner-only, but this is still a local sensitive-data retention expansion that should be accepted explicitly or tied to the opt-in catchup window.
