## Summary

Keep open: the underlying iMessage downtime-catchup gap is valid and the PR has real happy-path proof, but the latest head still has a cursor/high-watermark bug that can skip undispatched rows when a catchup pass is capped by `perRunLimit`.

## What This Changes

The branch adds opt-in iMessage downtime catchup with config/schema metadata, cursor persistence, replay bridge and monitor wiring, longer echo-cache retention, docs, tests, generated config metadata, and a changelog entry.

## Best Possible Solution

Clamp the catchup high-watermark to the capped/dispatched page, or suppress the raw watermark when valid rows were truncated, and add a regression assertion for the persisted cursor before maintainer review resumes.

## Reproduction Assessment

Yes. Current main docs/source show the underlying gap: iMessage only subscribes to live `watch.subscribe` events and documents no downtime catchup; the PR-introduced cap bug is source-reproducible by returning more valid rows than `perRunLimit`.

## Solution Assessment

No. The owner-local catchup direction is plausible, but the current high-watermark behavior is not the narrowest safe implementation because it can skip capped rows that were never dispatched.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not advance the cursor past capped rows:** `extensions/imessage/src/monitor/catchup-bridge.ts:218-219`
  - body: When more valid rows are fetched than `perRunLimit`, `collected` is sliced to `capped` but the returned high-watermark still reflects every raw `messages.history` row. Because `performIMessageCatchup` applies that watermark when there is no held failure, the saved cursor can jump past rows that were never dispatched and the next startup will filter them out. Limit the watermark to the capped page, or omit it when truncating valid rows, and assert `cursorAfter.lastSeenRowid` in the cap test.
  - confidence: 0.94
