## Summary

Keep open: this protected, maintainer-labeled feature PR has a cursor correctness bug where the catchup high-watermark can advance past rows truncated by `perRunLimit`, silently losing inbound messages. The PR body provides useful live happy-path proof, but merge should wait for the cap/high-watermark fix and maintainer review.

## What This Changes

The branch adds opt-in iMessage inbound catchup with config/schema metadata, cursor persistence, a replay loop, live RPC bridge wiring, longer echo-cache retention, docs, tests, and a changelog entry.

## Best Possible Solution

Fix the cap/high-watermark interaction so the cursor cannot advance past undispatched rows, add regression coverage for capped replay, then let a maintainer decide whether to ship the new opt-in config and retention semantics.

## Reproduction Assessment

Yes for the underlying gap: current main docs state that gateway-downtime iMessage catchup is missing, and the monitor source has only live `watch.subscribe` handling with no history replay. The PR-introduced cap bug is also source-reproducible by fetching more valid cross-chat rows than `perRunLimit`.

## Solution Assessment

No. The owner-local catchup direction is plausible, but the current implementation is not safe because raw high-watermarks can skip capped rows; clamp the high-watermark to the dispatched page or only advance past parser-rejected rows within the page boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not advance the cursor past capped rows:** `extensions/imessage/src/monitor/catchup-bridge.ts:215-219`
  - body: When more valid rows are fetched than `perRunLimit`, `capped` drops the later rows but the returned `highWatermarkRowid`/`highWatermarkMs` still reflect every raw row fetched across chats. `performIMessageCatchup` applies those as the cursor floor, so the next startup filters out rows that were never dispatched. Limit the returned watermark to the capped page, or omit it when truncating valid rows, and add a cursor assertion to the cap test.
  - confidence: 0.94
