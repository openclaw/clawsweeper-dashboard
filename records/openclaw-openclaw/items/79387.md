## Summary

Keep open: this PR addresses a real iMessage catchup gap and now includes plausible live macOS proof, but the patch still has cursor correctness bugs and a local data-retention concern that should block merge pending maintainer review.

## What This Changes

Adds opt-in iMessage downtime catchup with config/schema, per-account cursor and retry state, history-based replay through the monitor dispatch path, longer sent-echo retention, docs, changelog, generated metadata, and tests.

## Best Possible Solution

Land one iMessage-owned catchup design that keeps a durable cursor in sync with both startup replay and live processing, preserves retryable failures, scopes retention to opted-in recovery, and then closes https://github.com/openclaw/openclaw/issues/78649.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main documents the iMessage downtime gap and subscribes live-only, while the PR branch's fail-then-success catchup loop can advance past a retryable failed row. I did not rerun the macOS outage locally in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not yet. The extension-local opt-in direction and updated live proof are promising, but the patch needs cursor fixes, a retention-scope decision, and maintainer choice against https://github.com/openclaw/openclaw/pull/79085 before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep the cursor behind retryable failures:** `extensions/imessage/src/monitor/catchup.ts:388-390`
  - body: When a dispatch fails below `maxFailureRetries`, the loop records the failure but keeps processing later rows. If any later row succeeds, `lastSeenRowid` is saved past the failed row, so the next startup treats that failed message as pre-cursor and never retries it. Stop after the first retryable failure or compute the saved cursor from the earliest retryable failure.
  - confidence: 0.95
- **[P2] Advance the catchup cursor from live messages:** `extensions/imessage/src/monitor/monitor-provider.ts:889`
  - body: The new cursor only advances during the startup catchup pass. iMessage current-main does not have a persistent inbound GUID dedupe, so messages handled normally while the gateway is up remain above the cursor and can be replayed as fresh turns on the next restart before the actual downtime backlog. Update the cursor from successful live handling or add a persistent inbound replay dedupe before enabling this path.
  - confidence: 0.89

## Security Review
