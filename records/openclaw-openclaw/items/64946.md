## Summary

Keep open. Current main still has the false-positive validation boundary: plugin-owned fields are documented and modeled under the wrapped `plugins.entries.<id>.config` object, but loader, CLI metadata, and config-validation paths can still validate an absent `entry.config` as `{}` and emit required-property `invalid config` diagnostics. The related #70811 remains open and unmerged, so it is useful follow-up context rather than proof that this issue is fixed.

## What This Changes

The issue asks OpenClaw to stop repeatedly logging false plugin invalid-config errors for working plugins that use the documented wrapped `plugins.entries.<id>.config` shape.

## Best Possible Solution

Keep strict plugin config validation for real activation and explicit config checks, but make metadata, snapshot, preflight, and CLI metadata validation distinguish absent config from invalid config. Add regressions for strict required-field plugin schemas with both wrapped valid config and non-activating undefined-config paths.

## Reproduction Assessment

Yes. A strict plugin `configSchema` with required root fields can reproduce the source-level failure when an enabled or discovered plugin reaches loader or config validation without a materialized `entry.config`; current main validates that as `{}` and formats required-property invalid-config errors.

## Solution Assessment

No for current main. The narrow maintainable fix is context-aware validation, likely by extending or following #70811 so absent config is deferred only in non-activation/preflight paths while activation and explicit config still fail closed.
