## Summary

Keep open: current main still has loader paths that can validate omitted plugin config as `{}` against strict required schemas, and the external AgentMail plugin declares the reported `apiKey`/`inboxId` fields under the documented plugin config wrapper. The related loader PR likely overlaps, but it is still open and does not explicitly close this issue.

## What This Changes

The issue asks OpenClaw to stop emitting repeated false-positive plugin invalid-config errors for working plugins configured under `plugins.entries.<id>.config`.

## Best Possible Solution

Verify the AgentMail case against the related loader fix, then land a core-side validation change that suppresses false positives in non-activating or unmaterialized-config paths while preserving strict validation for real activation and explicit config.

## Reproduction Assessment

Yes, source inspection gives a credible reproduction path for the false-positive invalid-config shape: strict required schema plus `entry?.config` becoming `undefined` is validated as `{}`. I did not live-reproduce the AgentMail gateway loop, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear. The open related PR is a plausible narrow fix for the omitted-config validator noise, but maintainers should verify it against the AgentMail report before treating this issue as fixed because the stated wrapped-schema mismatch is not independently confirmed in current source.
