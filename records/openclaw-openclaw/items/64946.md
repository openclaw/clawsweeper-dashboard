## Summary

Keep open. Current main still has the false-positive validation boundary: plugin schemas are documented and wired for `plugins.entries.<id>.config`, but loader paths still validate `entry?.config` through a helper that coerces missing config to `{}`, so strict required schemas can emit noisy `invalid config` diagnostics before a real activation config is materialized.

## What This Changes

The issue asks OpenClaw to stop emitting repeated plugin `invalid config` diagnostics for working plugins that use the wrapped `plugins.entries.<id>.config` shape.

## Best Possible Solution

Keep strict validation for real activation and explicit config, but make non-activating preflight, snapshot, and CLI metadata paths defer validation when plugin config is absent, with regression tests for required-field plugin schemas.

## Reproduction Assessment

Yes. A strict plugin `configSchema` with required root fields can reproduce the source-level failure when a loader or CLI metadata path reaches `validatePluginConfig` with `entry?.config === undefined`, because current main validates `{}` and formats required-property errors.

## Solution Assessment

No for current main. The narrow maintainable fix is context-aware validation, likely by reviewing or extending #70811 so absent config is deferred only in non-activation paths while activation and explicit config still fail closed.
