## Summary

Keep open. Current main still contains the false-positive validation boundary: plugin config is documented and wired under the wrapped `config` object, but loader and CLI metadata paths still validate absent `entry?.config` as `{}`, and the open related PR #70811 has not merged or closed this issue.

## What This Changes

Stop plugin config validation from logging false `invalid config` errors for working plugins whose wrapped config is valid but absent from a preflight or metadata validator path.

## Best Possible Solution

Keep strict validation for real activation and explicit config, but defer validation when config is absent in non-activating metadata or preflight paths, with regression coverage for required-field plugin schemas.

## Reproduction Assessment

Yes from source inspection, not from a live agentmail package run. Current main still validates `entry?.config` through `params.value ?? {}` in loader and CLI metadata paths, so a strict required root schema can emit the reported `invalid config` when that path has not materialized the wrapped config.

## Solution Assessment

No for current main. The narrow maintainable fix is context-aware validation: defer only absent config in non-activating metadata/preflight paths while keeping activation and explicit config strict; #70811 is the active related implementation to review or extend.
