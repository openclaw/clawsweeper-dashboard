## Summary

Keep open. Current main still has a source-level false-positive path: plugin config is documented and modeled under `plugins.entries.<id>.config`, but loader and config-validation paths still validate a possibly absent `entry?.config` as `{}`, which can emit required-property `invalid config` messages for strict plugin schemas before real config is materialized. The overlapping #70811 is open and unmerged, so it is not proof that this issue is fixed.

## What This Changes

The issue asks OpenClaw to stop emitting repeated false-positive plugin invalid-config logs for working plugins configured with the wrapped `plugins.entries.<id>.config` shape.

## Best Possible Solution

Keep strict validation for real plugin activation and explicit config checks, but make metadata, snapshot, preflight, and CLI metadata validation distinguish absent config from invalid config. Add regressions for strict required-field plugin schemas with both wrapped valid config and non-activating undefined-config paths.

## Reproduction Assessment

Yes. A strict plugin `configSchema` with required root fields can reproduce the source-level failure when an enabled/discovered plugin reaches loader or config validation without a materialized `entry.config`; current main validates that as `{}` and formats required-property invalid-config errors.

## Solution Assessment

No for current main. The narrow maintainable fix is context-aware validation, likely by extending or following #70811 so absent config is deferred only in non-activation/preflight paths while activation and explicit config still fail closed.
