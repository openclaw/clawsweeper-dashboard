## Summary

Keep open. Current main still has the false-positive validation boundary: documented plugin config lives under `plugins.entries.<id>.config`, but loader paths still validate `entry?.config` through a helper that converts missing config to `{}`, so strict plugin schemas can emit noisy `invalid config` diagnostics; https://github.com/openclaw/openclaw/pull/70811 is related but remains open and needs maintainer review.

## What This Changes

Stop plugin config validation from emitting repeated false `invalid config` errors for working plugins whose config is wrapped under `plugins.entries.<id>.config`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/70811 or an equivalent context-aware loader fix that defers absent config validation only in non-activating paths while preserving strict activation and explicit-config validation.

## Reproduction Assessment

Yes, from source inspection: a plugin with strict required root fields and no `entry?.config` in a non-activating load reaches `validatePluginConfig`, which validates `{}` and can log `invalid config`. I did not live-run the external agentmail plugin in this read-only pass.

## Solution Assessment

Unclear for the open related PR until maintainer review confirms the discovery/metadata boundary is safe. The narrow maintainable direction is context-aware validation that skips only absent config in non-activating metadata/preflight paths while keeping real activation and explicit config fail-closed.
