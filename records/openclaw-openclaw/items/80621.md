## Summary

Keep open. Current main and the latest release still treat `contracts.tools` as an explicit string-array ownership contract, and a live PR with closing syntax already targets this issue, so closure should wait for maintainer API judgment and a landed fix.

## What This Changes

The issue requests that a native plugin manifest using `contracts.tools: true` allow `api.registerTool` to register tools instead of producing the missing-contract diagnostic.

## Best Possible Solution

Maintain the explicit `string[]` contract unless maintainers approve a wildcard mode; otherwise resolve this with clearer invalid-manifest diagnostics and docs, then close after the chosen fix or linked PR lands.

## Reproduction Assessment

Yes. Current main source shows `contracts.tools: true` normalizes away and `registerTool` then emits the reported missing-contract diagnostic; I did not run a live gateway flow because this review is read-only.

## Solution Assessment

Unclear. Accepting boolean `true` as wildcard ownership is not a narrow bug fix under the current docs; the maintainable path needs maintainer approval for a new manifest mode or a diagnostics/docs-only repair.
