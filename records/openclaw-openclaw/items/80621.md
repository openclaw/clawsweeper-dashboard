## Summary

Keep open. Current main still treats `contracts.tools` as an explicit string-array ownership contract, so the reported boolean value fails by source inspection, and the issue is paired with an open closing PR that has not landed.

## What This Changes

A plugin author reports that a native plugin declaring `contracts.tools: true` loads but `api.registerTool` fails with the missing-`contracts.tools` diagnostic and registers no tools.

## Best Possible Solution

Keep the issue open while the linked PR or a replacement is reviewed, and preserve explicit string-array ownership unless maintainers intentionally approve and document wildcard or boolean manifest semantics.

## Reproduction Assessment

Yes. Source inspection shows `contracts.tools: true` normalizes to no declared tool names, after which `registerTool` emits the exact reported diagnostic and skips registration.

## Solution Assessment

No. Treating boolean true as wildcard ownership is not the narrowest safe fix under the current documented contract; clearer diagnostics/docs or a maintainer-approved wildcard contract would be safer.
