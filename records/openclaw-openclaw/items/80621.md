## What This Changes

The issue requests that native plugin manifests using `contracts.tools: true` allow `api.registerTool` registrations instead of producing the missing-`contracts.tools` diagnostic.

## Best Possible Solution

Decide the public manifest contract first, then land either bounded wildcard support with docs/tests/proof or a narrower diagnostics/docs fix that keeps explicit `contracts.tools` names required.

## Reproduction Assessment

Yes. Source inspection shows `contracts.tools: true` is dropped during manifest normalization, leaving `registerTool` with no declared names and triggering the reported diagnostic.

## Solution Assessment

No. Treating boolean `true` as a catch-all wildcard is not clearly the best fix because current docs require concrete names and the active dynamic-tool work favors bounded wildcard namespaces.
