## Summary

Keep open. Current main and v2026.6.8 still drop boolean `contracts.tools: true`, and the related dynamic/wildcard and diagnostic PRs are open partial paths rather than a landed fix for this boolean/catch-all plugin API decision.

## What This Changes

Support or clarify plugin manifests that declare `contracts.tools: true` so `api.registerTool(...)` does not load the plugin while rejecting its tools with the missing-contract diagnostic.

## Best Possible Solution

Make one maintainer-approved plugin contract decision, then either implement and document bounded dynamic ownership or explicitly reject boolean `true` while improving diagnostics and docs for that invalid manifest shape.

## Reproduction Assessment

Yes, source-level. `contracts.tools: true` is normalized away, then `registerTool` observes no declared tool names and emits the exact missing-contract diagnostic before registering nothing.

## Solution Assessment

Unclear. Treating boolean `true` as catch-all is not the current documented contract; bounded wildcard ownership or diagnostics/docs-only may be safer depending on maintainer API and security direction.
