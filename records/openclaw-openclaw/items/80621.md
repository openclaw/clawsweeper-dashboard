## Summary

Keep open. Current main still treats `contracts.tools` as an explicit `string[]` of tool names, so `contracts.tools: true` is source-reproducible as an invalid manifest shape; an open PR already closes this issue and should be resolved before any cleanup close.

## What This Changes

The issue asks OpenClaw to accept a native plugin manifest with `contracts.tools: true` so `api.registerTool` succeeds and runtime inspection shows the registered tool.

## Best Possible Solution

Resolve the linked PR with maintainer-approved manifest semantics: either document and test a wildcard contract, or keep the string-array contract and improve invalid-manifest diagnostics and plugin-author docs.

## Reproduction Assessment

Yes, at source level: `contracts.tools: true` is dropped by manifest normalization, then `api.registerTool` emits the reported missing-contract diagnostic. I did not run the live Windows/npm gateway flow in this read-only review.

## Solution Assessment

No as a narrow bug fix: accepting boolean `true` as wildcard ownership is a new manifest mode under the current docs and tests. The maintainable path is a maintainer decision on wildcard semantics, or clearer diagnostics/docs if the explicit array contract stays.
