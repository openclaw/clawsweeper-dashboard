## Summary

Keep open. Current main and v2026.6.10 still drop `contracts.tools: true` before the plugin tool contract gate, and the open dynamic-contract PR is a related partial path rather than a landed decision for boolean/catch-all manifest semantics.

## What This Changes

The issue requests that a native plugin manifest using `contracts.tools: true` allow `api.registerTool(...)` to register agent tools instead of failing with the missing-contract diagnostic.

## Best Possible Solution

Make one maintainer-approved plugin manifest contract decision, then either land bounded dynamic ownership with docs/tests/proof or explicitly reject boolean `true` while improving diagnostics and docs.

## Reproduction Assessment

Yes, source-level. Current main normalizes `contracts.tools: true` away, then `registerTool` sees no declared tool names and emits the exact missing-contract diagnostic before registering nothing.

## Solution Assessment

Unclear. Treating boolean `true` as catch-all is not the current documented contract; bounded wildcard ownership or a docs/diagnostics-only rejection path may be safer depending on maintainer API and security direction.
