## Summary

Keep open. Current main still treats `contracts.tools` as an explicit string array, so the reported boolean manifest value source-reproduces the generic runtime diagnostic, and an open PR with closing syntax is already the active implementation/product-decision path.

## What This Changes

The issue asks for a native plugin manifest using `contracts.tools: true` to allow `api.registerTool` registrations instead of producing the missing-contract diagnostic.

## Best Possible Solution

Keep the issue open until https://github.com/openclaw/openclaw/pull/80623 is reviewed and either landed or closed; preserve explicit array ownership unless maintainers intentionally approve and document a wildcard/boolean manifest mode.

## Reproduction Assessment

Yes. Source inspection shows `contracts.tools: true` is a non-array value that normalizes to no declared tool names, after which `api.registerTool` emits the exact reported diagnostic.

## Solution Assessment

No, not as a straight bug fix. Accepting `true` as wildcard ownership changes the public manifest contract; the safer path is clearer invalid-manifest diagnostics/docs unless maintainers approve a documented wildcard mode.
