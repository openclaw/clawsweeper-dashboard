## Summary

## What This Changes

The issue asks for native plugin manifests using `contracts.tools: true` to allow `api.registerTool` instead of producing the missing `contracts.tools` diagnostic.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/80623 for the boolean/wildcard API decision; if that direction is rejected, keep the explicit array contract and improve invalid-manifest diagnostics plus plugin-author docs.

## Reproduction Assessment

Yes, by source inspection: `contracts.tools: true` is normalized to no declared tool names, and `api.registerTool` then emits the exact missing-contract diagnostic. I did not run the live gateway/install flow because this review is read-only.

## Solution Assessment

No, accepting boolean `true` as wildcard ownership is not the narrowest fix unless maintainers intentionally approve that manifest API. The safer path is explicit `string[]` ownership with clearer validation/docs, or a reviewed API change in the linked PR.
