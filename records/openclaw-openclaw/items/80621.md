## Summary

Keep open. Current main intentionally requires `contracts.tools` to be an array of tool names, so `contracts.tools: true` is not a valid manifest shape, but the report exposes a real docs/diagnostic gap because that invalid value is silently normalized away and later surfaces as the generic `plugin must declare contracts.tools` runtime error.

## What This Changes

The issue asks for a native plugin declaring `contracts.tools: true` to successfully register an agent tool and show it in `plugins inspect`.

## Best Possible Solution

Keep the explicit string-array manifest contract, but make invalid non-array `contracts.tools` values and the required manifest location clearer in diagnostics and plugin-author docs.

## Reproduction Assessment

Yes, by source inspection: `contracts.tools: true` normalizes to no declared tool names, and `api.registerTool` then emits the exact reported diagnostic. I did not run the live gateway flow because this was a read-only review.

## Solution Assessment

No, accepting `contracts.tools: true` is not the best fix because current docs and shipped behavior require a list of owned tool names. The narrow maintainable fix is a clearer invalid-manifest diagnostic plus docs/examples on the cited plugin API page.
