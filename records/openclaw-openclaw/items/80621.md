## Summary

Keep open. Current main still source-reproduces the reported failure, but the shipped/public manifest contract is explicit `string[]` ownership; accepting `contracts.tools: true` would be a new wildcard manifest mode, and the linked open PR uses closing syntax so this issue should wait for maintainer decision and PR resolution.

## What This Changes

The issue asks for native plugin manifests with `contracts.tools: true` to allow `api.registerTool(...)` registrations instead of producing the missing-contract diagnostic.

## Best Possible Solution

Keep `contracts.tools` as explicit `string[]` ownership unless maintainers intentionally approve and document wildcard semantics; otherwise improve invalid-manifest diagnostics and plugin-author docs, then close after the chosen PR or equivalent fix lands.

## Reproduction Assessment

Yes. Source inspection shows `contracts.tools: true` normalizes to `[]`, then `registerTool` emits the exact missing-contract diagnostic and skips registration.

## Solution Assessment

No. Treating boolean `true` as a wildcard is a public manifest contract change, so the narrower maintainable path is clearer diagnostics/docs unless maintainers explicitly choose wildcard tool ownership.
