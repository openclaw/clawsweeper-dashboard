## Summary

Keep open. Current main and the latest release still reject `contracts.tools: true`, and accepting that boolean as catch-all tool ownership remains a plugin API/security-boundary decision rather than an already-implemented fix.

## What This Changes

The issue asks for native plugins declaring `contracts.tools: true` to be able to register agent tools without the missing-contract diagnostic.

## Best Possible Solution

Make one maintainer-visible plugin API decision, then land either a documented bounded wildcard contract with resolver tests and real runtime proof, or keep explicit `contracts.tools` names and improve diagnostics/docs for invalid boolean manifests.

## Reproduction Assessment

Yes, source-level: `contracts.tools: true` is normalized as no declared names, and `registerTool` then emits the exact missing-contract diagnostic. I did not run a live Windows/npm plugin install in this read-only review.

## Solution Assessment

Unclear. Accepting `true` as catch-all ownership is not the current documented contract, and related reviews show maintainers need to choose the public plugin API/security direction before implementation.
