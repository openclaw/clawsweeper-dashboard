## Summary

Keep open: current main still source-reproduces the reported diagnostic for boolean `contracts.tools`, and an open PR references this issue with closing syntax. Closing should wait for maintainer review of whether boolean/wildcard tool ownership is an accepted manifest contract or whether this should be fixed as clearer diagnostics/docs.

## What This Changes

The issue asks OpenClaw to allow or clearly handle native plugins that declare `contracts.tools: true` before calling `api.registerTool`.

## Best Possible Solution

Keep explicit `contracts.tools` arrays unless maintainers approve a wildcard mode; otherwise fix this by improving invalid-manifest diagnostics and plugin-author docs, then resolve the linked PR accordingly.

## Reproduction Assessment

Yes, source-level: boolean `contracts.tools` is dropped by `normalizeTrimmedStringList`, leaving no declared tool names, and `registerTool` then emits the reported diagnostic. I did not run the full Windows/npm gateway flow in this read-only review.

## Solution Assessment

No for the proposed boolean-to-wildcard behavior without maintainer approval. The narrower maintainable path is clearer diagnostics/docs, or a deliberate documented wildcard manifest design if maintainers choose that direction.
