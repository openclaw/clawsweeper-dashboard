## Summary

Close as a duplicate/satellite: current main treats `contracts.tools: true` as invalid because plugin tool ownership is an exact `string[]` manifest contract, and the remaining SDK/DX/dynamic-tool questions are already tracked in the canonical plugin-tool contract thread.

## What This Changes

The issue asks for `api.registerTool` to succeed for a native plugin whose manifest uses `contracts.tools: true`, and notes related plugin SDK docs and command-registration gaps.

## Best Possible Solution

Keep the supported manifest contract as explicit tool names, and resolve any broader dynamic/wildcard ownership or failure-DX changes in the canonical plugin-tool contract thread.

## Reproduction Assessment

Yes, at source level: with `contracts.tools: true`, the manifest normalizer produces no tool names, and `registerTool` then emits the exact reported diagnostic and skips registration. I did not run a live Windows plugin install during this read-only review.

## Solution Assessment

Yes for cleanup: closing this as a duplicate is the best repository action because accepting boolean or dynamic tool ownership would change the public manifest contract already being discussed in the canonical issue and open PR.
