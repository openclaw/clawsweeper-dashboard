## Summary

Keep open. Current main still source-reproduces the reported failure for `contracts.tools: true`, but that value is outside the documented `string[]` manifest contract and an open linked PR, https://github.com/openclaw/openclaw/pull/80623, already owns a proposed API-changing fix with closing syntax.

## What This Changes

A plugin author reports that a native plugin declaring `contracts.tools: true` loads but `api.registerTool` fails, and the docs page they used does not explain the required tool-name array.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/80623 is reviewed; either land a maintainer-approved manifest mode with docs, tests, and live proof, or close that PR and clarify the explicit `contracts.tools` array contract in diagnostics and plugin-author docs.

## Reproduction Assessment

Yes. Source inspection shows `contracts.tools: true` normalizes to no declared names and then `api.registerTool` emits the exact reported diagnostic; I did not run the Windows gateway flow in this read-only review.

## Solution Assessment

No, not as currently proposed by the linked PR. Accepting boolean `true` as wildcard tool ownership is a new public manifest mode; the narrower maintainable path is clearer diagnostics/docs unless maintainers explicitly approve that API change.
