## Summary

Keep open. Current main still lacks the requested memory file-ingestion event API: the public memory capability can register runtime/search/public-artifact behavior, but the scan, watch, hash-skip, chunk, and stale-cleanup pipeline remains owned by memory-core.

## What This Changes

The issue requests a memory plugin API that lets third-party memory backends consume OpenClaw's file scanning and indexing pipeline instead of reimplementing it.

## Best Possible Solution

Add a narrow, documented memory ingestion SDK seam that reuses core scanner/watch policy and explicitly defines file access, event delivery, chunking ownership, deletion semantics, and tests for memory-core plus one external-style memory plugin.

## Reproduction Assessment

Yes, at source level: current main exposes memory runtime/public-artifact capability and file helpers, but no file scan/watch event API for memory plugins. I did not run a live repro because this is a read-only API-gap review, not a runtime bug.

## Solution Assessment

Unclear as proposed. A plugin-facing ingestion seam fits the product direction, but maintainers should choose the contract shape and security model before implementation because this becomes public SDK surface.
