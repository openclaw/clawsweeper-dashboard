## Summary

Keep this open. Current main still lacks a host-owned memory file scan/watch/change-event API for active memory plugins, and the request is a real plugin SDK gap rather than ClawHub-only plugin work because the project scope explicitly welcomes plugin API extensions when useful capabilities cannot yet be built externally.

## What This Changes

Request an additive Memory Plugin API so third-party memory plugins can receive host-managed full-scan and add/change/delete file events instead of reimplementing extraPaths scanning, watching, chunking, and stale cleanup.

## Best Possible Solution

Design an additive memory file-indexing capability for the active memory plugin: a host-owned scanner/watcher service that can deliver full-scan batches and add/change/delete events built on existing list/chunk helpers, with contract tests, docs, and a memory-core dogfood path before third-party plugins rely on it.

## Reproduction Assessment

Yes for the API gap. Source inspection of the memory plugin types, public plugin API, memory-core watcher/sync implementation, and memory-host SDK exports shows no host-managed file scan/watch event surface for third-party memory plugins today.

## Solution Assessment

Unclear at the interface level. The requested direction matches the plugin-boundary policy, but the best implementation should be a generic additive memory SDK/runtime contract rather than a LanceDB-specific workaround.
