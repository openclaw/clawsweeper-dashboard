## Summary

Keep open. Current main and v2026.5.22 still keep file scanning private to memory-core/QMD flows, while the public memory plugin capability has no scan/watch/change-event API, so the requested core design work remains necessary.

## What This Changes

The issue requests a public Memory Plugin API that exposes OpenClaw’s host-owned file scan/watch/hash/chunk/delete pipeline to third-party memory backends.

## Best Possible Solution

Design an additive host-managed memory file-ingestion contract for the active memory plugin, dogfood it in memory-core, and document/test scan, watch, chunking, extraPaths, deletion, and security semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection on current main gives high confidence that the public memory capability contract has no scanner/change-event lifecycle while memory-core owns that lifecycle privately.

## Solution Assessment

Unclear at the exact interface level. The direction fits OpenClaw's plugin API expansion guidance, but maintainers should choose payload shape, chunking ownership, permissions, security boundaries, and failure semantics before implementation.
