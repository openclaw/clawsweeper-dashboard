## Summary

Keep open. Current main still has helper exports for memory files and an active memory plugin slot, but it does not expose a host-owned scan/watch/change-event contract that third-party memory plugins can implement; the related backend-abstraction issue is closed stale and does not supersede this narrower ingestion API request.

## What This Changes

The issue requests a public memory plugin API for host-owned file scanning, watching, incremental sync, and stale cleanup events.

## Best Possible Solution

Design an additive, versioned memory file-indexing contract for the active memory plugin, backed by host-owned scan/watch lifecycle code, contract tests, docs, and a memory-core dogfood implementation.

## Reproduction Assessment

Yes for the API gap, though this is a feature request rather than a bug reproduction. Source inspection on current main shows the plugin API, memory runtime type, memory-host file helpers, and memory-core watcher/sync path do not expose a managed file scan/change event contract to third-party memory plugins.

## Solution Assessment

Unclear at the interface level. The direction fits OpenClaw's plugin-boundary policy, but maintainers should choose the public event contract and lifecycle ownership before any implementation PR is queued.
