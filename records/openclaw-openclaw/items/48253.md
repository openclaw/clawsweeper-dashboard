## Summary

Keep open. Current main still has memory helper pieces and public artifacts, but no host-owned file scan/watch/change-event API that active memory plugins can consume; the related backend-abstraction issue was closed stale and does not supersede this narrower SDK design request.

## What This Changes

Request to expose the built-in memory file scanning, watching, incremental sync, and stale cleanup pipeline as an additive Memory Plugin API for third-party memory backends.

## Best Possible Solution

Design an additive, versioned memory file-indexing contract for the active memory plugin, backed by host-owned scan/watch lifecycle code, contract tests, docs, and a memory-core dogfood implementation.

## Reproduction Assessment

Yes for the API gap, though this is a feature request rather than a bug repro. Source inspection shows the current plugin API, memory runtime type, memory-host files helpers, and memory-core watcher/sync path do not expose a managed file scan/change event contract to third-party memory plugins.

## Solution Assessment

Unclear at the interface level. The direction fits OpenClaw's plugin-boundary policy, but maintainers should choose the public event contract and lifecycle ownership before any implementation PR is queued.
