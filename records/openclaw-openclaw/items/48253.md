## Summary

Keep open. Current main has the built-in memory-core scan/watch/hash/index/stale-cleanup pipeline, but the public memory plugin runtime does not expose equivalent file scan/change ingestion hooks for third-party memory backends, and the related open memory-role PR does not own this narrower file-ingestion API gap.

## What This Changes

Expose the built-in memory file scan/watch/hash/chunk/stale-cleanup pipeline as a stable Memory Plugin API for third-party memory backends.

## Best Possible Solution

Keep this as a maintainer-reviewed plugin API design item and define a narrow, documented memory file ingestion seam that lets external memory backends reuse the scanner without moving optional backends into core.

## Reproduction Assessment

Yes for the source-level API gap, not as a runtime bug: current main shows the scanner/watch/sync pipeline inside memory-core and no public file-event ingestion hook in MemoryPluginRuntime. I did not run a live third-party plugin reproduction because this is a feature/API request rather than a broken shipped behavior report.

## Solution Assessment

Unclear as an implementation, but keeping the issue open is the right triage outcome. A stable SDK seam is plausible, but maintainers should choose the contract shape before a fix PR because plugin API changes are durable compatibility and security surface.
