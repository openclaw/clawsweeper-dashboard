- `impact:session-state`: The request affects how agent memory files are indexed and kept current for custom memory backends.

## Summary

Keep open. Current main has active-memory runtime hooks and shared file helper subpaths, but the scan/watch/hash/stale-cleanup loop still lives inside memory-core and there is no public file-ingestion event API for third-party memory backends.

## What This Changes

Requests a public Memory Plugin API so third-party memory backends can consume the built-in file scan/watch/incremental-sync/stale-cleanup pipeline and `extraPaths` indexing.

## Best Possible Solution

Define a narrow, permissioned SDK ingestion contract that lets external memory backends reuse scanning without moving optional storage backends into core.

## Reproduction Assessment

Yes for the source-level API gap: current main shows the memory-core scan/watch/sync pipeline and public memory runtime hooks, but no file-ingestion event contract. This is not a runtime bug reproduction.

## Solution Assessment

Unclear whether the proposed callback shape is the best API. The best path is a maintainer-designed memory SDK contract with explicit file-access and plugin trust semantics before implementation.
