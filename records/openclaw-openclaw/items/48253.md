## Summary

Keep open. Current main and v2026.6.10 still lack a memory file-ingestion event/API surface: memory-core owns scan, watch, hash, chunk, sync, and stale-cleanup behavior, while the public memory capability only exposes runtime/search/prompt/public-artifact hooks. This needs maintainer SDK and security design rather than cleanup closure.

## What This Changes

Expose the built-in memory file scanning, watching, hashing, chunking, and stale-cleanup pipeline as a memory plugin API for external memory backends.

## Best Possible Solution

Define a narrow documented memory ingestion SDK seam with explicit path consent, event delivery, chunking ownership, deletion and stale-cleanup semantics, and tests against memory-core plus an external-style memory plugin.

## Reproduction Assessment

Yes at source level: current capability/runtime/docs lack file event hooks, and the scanner/watcher remains memory-core internal. No live repro was needed because this is an API absence rather than a runtime failure report.

## Solution Assessment

Unclear as proposed; a file-event seam is plausible, but maintainers should choose callbacks versus a shared scanner/manager protocol and settle the security semantics first.

## Risks / Open Questions

- Opening the memory scan pipeline to third-party plugins changes the local-file trust boundary: configured MEMORY.md, memory/, and extraPaths contents could be delivered to plugin code that does not currently receive them through a documented host event contract.
- A public file-ingestion hook would become SDK compatibility surface, so callback shape, batch semantics, chunking ownership, delete/stale cleanup behavior, backpressure, and error handling need maintainer design before implementation.
- The existing helper exports can list and read memory files, but treating them as the complete feature would still leave plugins without consistent startup scan, watcher delivery, deletion handling, and stale index cleanup.
