## Summary

Keep open. Current main and v2026.6.10 still do not expose a memory file-ingestion event API: memory-core owns scanning, watching, hash diffing, chunking, sync, and stale cleanup, while public memory plugins get runtime/search/prompt/public-artifact surfaces only.

## What This Changes

Request to expose memory-core's scan/watch/incremental file-ingestion pipeline as a public memory plugin API so external memory backends can index configured files and extraPaths.

## Best Possible Solution

Define a narrow documented memory ingestion SDK seam with explicit path consent, event delivery, chunking ownership, deletion/stale-cleanup semantics, and contract tests against memory-core plus an external-style memory plugin.

## Reproduction Assessment

Yes, at source level: the current capability/runtime/docs lack file event hooks, and the scanner/watcher/sync pipeline remains memory-core internal. No live repro is needed because this is an API absence rather than a runtime failure report.

## Solution Assessment

Unclear. A file-event seam is plausible, but maintainers should choose callbacks versus a shared scanner/manager protocol and define the security semantics before implementation.
