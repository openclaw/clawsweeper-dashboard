## Summary

Keep open. Current main and the latest v2026.6.6 release still keep memory file scan/watch/hash/stale-cleanup behavior inside memory-core, while the public memory plugin capability has runtime/search/public-artifact surfaces but no file-ingestion event API.

## What This Changes

The issue requests a public memory plugin API for memory file scan/watch/change/delete events so external memory backends can index `MEMORY.md`, `memory/`, and `extraPaths` without reimplementing the pipeline.

## Best Possible Solution

Define a narrow, documented memory file-ingestion SDK seam with explicit plugin trust semantics and regression coverage for startup scan, watch updates, `extraPaths`, hash skip, deletion cleanup, memory-core, and one external-style memory plugin.

## Reproduction Assessment

Yes for the source-level API gap: current main and v2026.6.6 show memory-core owns scan/watch/hash/stale cleanup, while the public memory plugin capability lacks file-ingestion hooks. This is not a runtime bug reproduction.

## Solution Assessment

Unclear as an implementation, but keeping the issue open is the best current triage outcome. Maintainers should choose the SDK contract and file-access model before implementation because this becomes durable plugin API and security surface.
