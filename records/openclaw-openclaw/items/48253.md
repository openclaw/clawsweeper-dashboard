## Summary

Keep this open. Current main still lacks a host-owned memory file scan/watch/change-event API for active memory plugins, and the request is a real plugin SDK gap rather than a ClawHub-only plugin idea because VISION explicitly welcomes plugin API extensions when useful capabilities cannot yet be built externally.

## What This Changes

Expose the built-in memory file scan, watch, incremental sync, chunking, and stale-cleanup pipeline as an API consumable by third-party memory plugins.

## Best Possible Solution

Design an additive memory file-indexing capability for the active memory plugin: a host-owned scanner/watcher service that can deliver full-scan batches and add/change/delete events built on existing list/chunk helpers, with contract tests, docs, and a memory-core dogfood path before third-party plugins rely on it.

## Reproduction Assessment

Yes for the API gap. Inspecting current main’s memory plugin types, public plugin API, memory-core watcher/sync implementation, and memory-lancedb registration path is enough to verify that no third-party plugin can consume the host-owned file pipeline today.

## Solution Assessment

Unclear at the interface level. The requested direction matches the plugin-boundary policy, but the best implementation should be a generic, additive memory SDK/runtime contract rather than a LanceDB-specific workaround.
