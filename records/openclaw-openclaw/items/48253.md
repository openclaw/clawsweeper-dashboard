## Summary

Keep open. Current main and v2026.5.22 still expose memory plugins through prompt, runtime, corpus, embedding, and artifact surfaces, while the file scan/watch/hash/stale-cleanup pipeline remains private to memory-core, so the requested plugin API gap is still real.

## What This Changes

The issue asks OpenClaw to expose the existing memory file scan/watch/change pipeline as an additive Memory Plugin API for third-party memory backends.

## Best Possible Solution

Design an additive host-managed memory file-ingestion contract for the active memory plugin, dogfood it in memory-core, and document/test scan, watch, chunking, extraPaths, deletion, failure, and security semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection on current main gives high confidence that public memory capability registration has no scanner/change-event lifecycle while memory-core owns that lifecycle privately.

## Solution Assessment

Unclear at the exact interface level. The direction fits OpenClaw's plugin API expansion guidance, but maintainers should choose payload shape, chunking ownership, permissions, security boundaries, and failure semantics before implementation.
