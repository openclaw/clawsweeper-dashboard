## Summary

Keep open. Current main still has the memory-core file scan/watch/hash/index/stale-cleanup pipeline, but the public memory plugin contract exposes manager/runtime/public-artifact hooks rather than file scan/change ingestion hooks for third-party memory backends; the open memory role-slot PR is related but does not own this narrower ingestion API gap.

## What This Changes

The issue requests a public Memory Plugin API that exposes OpenClaw's file scan/watch/hash/chunk/index pipeline to third-party memory backends.

## Best Possible Solution

Keep this as a maintainer-reviewed memory plugin API design item and define a narrow, documented file-ingestion seam that lets external memory backends reuse core scanning without moving optional backends into core.

## Reproduction Assessment

Yes for the source-level API gap: current main shows the file scan/watch/sync pipeline in memory-core and no public memory file-event ingestion hook. This is not a runtime bug reproduction.

## Solution Assessment

Unclear as an implementation, but keeping the issue open is the best triage outcome. Maintainers should choose a narrow SDK contract and permission model before implementation because plugin API changes are durable compatibility and security surface.
