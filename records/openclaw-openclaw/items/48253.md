## Summary

Keep open. Current main exposes memory capability/runtime registration and helper file-listing exports, but the host-owned scan/watch/incremental/stale-cleanup lifecycle remains private to memory-core, so this is still a public API/product design gap rather than a closeable duplicate or ClawHub-only request.

## What This Changes

Request a public Memory Plugin API that lets third-party memory plugins consume OpenClaw-managed memory file scans, watch events, chunking, hashes, stale cleanup, and extraPaths.

## Best Possible Solution

Design an additive, versioned host-managed memory file-ingestion contract for the selected memory plugin, dogfood it in memory-core, and document/test scan, watch, chunking, deletion, extraPaths, and security boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection on current main shows the public memory capability contract has no scanner/change-event lifecycle while memory-core owns watcher and sync privately.

## Solution Assessment

Unclear at the exact interface level. The direction matches OpenClaw's plugin API expansion guidance, but maintainers should choose event payload, chunking ownership, permissions, and failure semantics before implementation.
