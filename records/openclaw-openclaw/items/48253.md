## Summary

Keep open. Current main has memory runtime registration and shared file-listing/index helper exports, but the host-owned scan, watch, hash-diff, and stale-cleanup lifecycle is still private to memory-core, so this remains a real public SDK/product design gap rather than a closeable ClawHub-only request.

## What This Changes

The issue requests an additive Memory Plugin API that exposes the built-in file scanning, watching, incremental indexing, and deletion pipeline to third-party memory plugins.

## Best Possible Solution

Design an additive, versioned host-managed memory file-ingestion contract for the active memory plugin, dogfood it in memory-core, and document/test scan, watch, chunking, deletion, extraPaths, and security semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection on current main shows the public memory capability contract has no scanner/change-event lifecycle while memory-core owns watcher and sync privately.

## Solution Assessment

Unclear at the exact interface level. The direction is consistent with OpenClaw's plugin API expansion guidance, but maintainers should choose payload shape, chunking ownership, permissions, and failure semantics before implementation.
