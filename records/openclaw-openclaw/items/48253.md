## Summary

Keep open. Current main and the latest release still lack a public memory file scan/change ingestion hook; existing helpers expose file listing, reads, runtime, and artifacts, but not the shared watch/incremental/stale-cleanup pipeline, so this remains a maintainer API/security design item.

## What This Changes

Request to expose the built-in memory file scan/watch/chunk/sync pipeline as a public Memory Plugin API for third-party memory backends.

## Best Possible Solution

Keep this open for maintainer API design of a narrow, documented memory file-ingestion seam that reuses the scanner without moving optional backends into core.

## Reproduction Assessment

Yes for the source-level API gap: current main has memory-core scan/watch/sync/stale cleanup code, but the public MemoryPluginRuntime/MemoryPluginCapability types do not expose file event ingestion hooks. This is not a runtime bug reproduction.

## Solution Assessment

Unclear as an implementation: the proposed event hooks are plausible, but maintainers should choose the durable contract shape and security model before a fix PR.
