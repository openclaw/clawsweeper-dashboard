## Summary

Keep open. Current main still has memory runtime registration and shared file-listing helpers, but the host-owned scan, watch, hash-diff, chunking, and stale-cleanup lifecycle is not exposed as a public memory plugin API, so this remains a real SDK/product design gap rather than a ClawHub-only request.

## What This Changes

The issue requests an additive public Memory Plugin API so third-party memory backends can reuse host-managed file scanning, watching, chunking, hash diffing, stale cleanup, and extraPaths ingestion.

## Best Possible Solution

Design an additive, versioned host-managed memory file-ingestion contract for the active memory plugin, dogfood it in memory-core, and document/test scan, watch, chunking, deletion, extraPaths, and security semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection on current main gives high confidence that the public memory capability contract has no scanner/change-event lifecycle while memory-core owns that lifecycle privately.

## Solution Assessment

Unclear at the exact interface level. The direction fits OpenClaw's plugin API expansion guidance, but maintainers should choose payload shape, chunking ownership, permissions, security boundaries, and failure semantics before implementation.
