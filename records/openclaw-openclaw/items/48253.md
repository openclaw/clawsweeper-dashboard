## Summary

Keep open. Current main still has the memory-core scan/watch/hash/stale-cleanup pipeline and shared file helpers, but it does not expose a public memory-plugin file scan/change ingestion contract for third-party memory backends; the related open memory-role and corpus-supplement PRs do not own this narrower API gap.

## What This Changes

The issue requests a public Memory Plugin API that exposes the built-in memory file scan/watch/change-detection pipeline to third-party memory backends.

## Best Possible Solution

Keep this as a maintainer-reviewed plugin API design item and define a narrow documented memory file-ingestion seam only if maintainers want external memory backends to reuse core scanning without moving optional backends into core.

## Reproduction Assessment

Yes for the source-level API gap, not as a runtime bug: current main shows the scanner/watch/sync pipeline inside memory-core and no public file-event ingestion hook in MemoryPluginRuntime. I did not run a live third-party plugin reproduction because this is a feature/API request rather than a broken shipped behavior report.

## Solution Assessment

Unclear as an implementation, but keeping the issue open is the right triage outcome. A stable SDK seam is plausible, but maintainers should choose the contract shape before implementation because plugin API changes and third-party file access are durable compatibility and security surfaces.
