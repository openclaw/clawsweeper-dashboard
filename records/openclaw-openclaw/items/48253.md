- `impact:session-state`: The requested API affects memory indexing and recall state for agents using custom memory backends.
- `impact:security`: The requested ingestion hook would expose workspace memory files and extraPaths contents across a plugin trust boundary.

## Summary

Keep open. Current main and latest v2026.6.6 still keep the memory file scan/watch/hash/stale-cleanup pipeline inside memory-core, while the public memory plugin API exposes runtime/search/public-artifact hooks rather than file ingestion events.

## What This Changes

Request to expose the built-in memory file scanning, watching, hashing, chunking, and stale cleanup pipeline as a memory plugin API for third-party backends.

## Best Possible Solution

Define a narrow, documented memory file-ingestion SDK seam with explicit plugin trust semantics and regression coverage for startup scan, watch updates, extraPaths, hash skip, deletion cleanup, memory-core, and one external-style memory plugin.

## Reproduction Assessment

Yes for the source-level API gap: current main shows the scan/watch/sync pipeline in memory-core and no public memory file-event ingestion hook. This is not a runtime bug reproduction.

## Solution Assessment

Unclear as an implementation, but keeping the issue open is the best current triage outcome. Maintainers should choose the SDK contract and file-access model before implementation because this becomes durable plugin API and security surface.
