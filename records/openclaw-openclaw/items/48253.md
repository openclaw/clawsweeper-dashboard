## Summary

Keep open. Current main has memory file-listing helpers and an active memory capability/runtime slot, but the host-owned scan, watch, incremental sync, and stale-delete lifecycle is still private to memory-core, so this remains a real plugin API/product design gap rather than a duplicate, stale support item, or ClawHub-only request.

## What This Changes

Request an additive Memory Plugin API so memory plugins can consume host-managed scan, watch, incremental indexing, chunking, and stale-delete events for MEMORY.md, memory/, and memorySearch.extraPaths.

## Best Possible Solution

Design an additive, documented host-managed memory file-ingestion lifecycle for the active memory plugin, dogfood it in memory-core, and test scan, watch, chunking, deletion, extraPaths, permissions, and failure semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection of current main shows memory-core owns watch/sync privately while the public plugin API lacks memory file event hooks.

## Solution Assessment

Unclear at the exact proposed interface level. The direction fits OpenClaw's plugin API expansion policy, but maintainers should choose payload shape, chunking ownership, permissions, and failure semantics before implementation.
