## Summary

Keep open. Current main still lacks the requested public memory file-event ingestion hook; the scan/watch/hash/stale-cleanup path exists inside memory-core and shared host helpers, while the related broader backend issue is closed stale rather than a canonical owner for this narrower API gap.

## What This Changes

The issue requests a public Memory Plugin API that lets third-party memory backends consume OpenClaw’s built-in file scan, watch, hash, chunking, and stale-cleanup pipeline.

## Best Possible Solution

Keep this as a maintainer-reviewed memory plugin API design item and define a narrow, documented file-ingestion seam with explicit file-access semantics for external memory backends.

## Reproduction Assessment

Yes for a source-level API gap: current memory-core owns scan/watch/sync logic, while the public memory capability type does not include file event ingestion hooks. This is not a runtime bug reproduction.

## Solution Assessment

Unclear as an implementation shape; keeping the issue open is the best triage outcome because maintainers need to choose the SDK contract and file-access model before code.
