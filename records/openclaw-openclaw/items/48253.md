## Summary

Keep open. Current main still has memory runtime registration and shared file-listing/chunking helpers, but the host-owned scan, watch, hash-diff, chunking, and stale-cleanup lifecycle is not exposed as a public memory plugin API, so this remains a real SDK/product design gap.

## What This Changes

The issue requests an additive Memory Plugin API that exposes OpenClaw's host-managed memory file scanning, watching, incremental sync, chunking, and stale cleanup pipeline to third-party memory backends.

## Best Possible Solution

Design an additive, documented host-managed memory file-ingestion contract for the active memory plugin, then dogfood it through memory-core with tests for initial scan, watcher updates, extraPaths, chunking, deletion cleanup, and backend failure semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection gives high confidence that memory-core owns the scan/watch lifecycle privately while the public memory plugin contract lacks scanner or file-change hooks.

## Solution Assessment

Unclear at the exact interface level. The direction fits OpenClaw's plugin API expansion guidance, but maintainers should choose payload shape, chunking ownership, permissions, security boundaries, and failure semantics before implementation.
