## Summary

Keep open. Current main has adjacent cleanup for stuck processing/running states, but the remaining request is still real: active stuck-session recovery is warning-only by default, normal Discord direct runner failures still use generic failure copy, and fresh stored sessions can still be reused without a timeout-dead or forced-fresh health gate. The linked #72434 and cross-referenced #73581 are adjacent timeout/stuck-lane reports, not a canonical replacement for this Discord partial-success/session-health policy gap.

## What This Changes

The issue requests unambiguous Discord session health and user-facing partial-success or late-failure handling after a routed embedded-agent timeout.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that either marks the affected chat session failed/dead, rotates it before later routing, or explicitly requires /new, and pair it with channel-appropriate copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes for the remaining behavior gap: current source and tests show generic Discord direct failure copy, warning-only active recovery, and stored-session reuse without a timeout-dead gate. A live Discord plus Claude CLI timeout run would strengthen end-to-end timing evidence, but the source-level gap is already visible.

## Solution Assessment

Unclear because no patch is proposed and the issue allows several acceptable outcomes. The narrow maintainable path is to choose one shared session-health policy first, then implement timeout/partial-success copy and tests around that policy.
