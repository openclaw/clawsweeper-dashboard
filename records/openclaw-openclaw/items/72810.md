## Summary

Keep open. Current main has adjacent timeout and lifecycle cleanup, but the reported session-health policy gap remains: active stuck recovery is still warning-only by default, timed-out session entries can still be reused by freshness alone, and normal Discord direct runner failures still use generic copy rather than partial-success/timeout-specific state.

## What This Changes

The issue asks OpenClaw to make Discord-routed embedded-agent timeouts and partial-success failures leave an unambiguous session-health state instead of keeping the timed-out session routable with generic failure copy.

## Best Possible Solution

Add a shared terminal timeout/session-health transition in the embedded-agent reply/session pipeline that records timeout/partial-success state and prevents blind reuse of that conversation until the selected policy is satisfied. Pair it with Discord-appropriate user copy that distinguishes possible completed side effects from total failure, and cover both recovery and next-turn routing in focused tests.

## Reproduction Assessment

Yes for the current behavior gap: the issue includes a sanitized timeout/stuck-session log shape, and current source/tests show generic Discord direct failure copy, warning-only active recovery, and fresh stored-session reuse without a timeout-dead health gate. A live Discord/Claude CLI timeout would strengthen timing proof but is not required to verify the remaining source-level gap.

## Solution Assessment

Unclear because no patch is proposed and the issue intentionally lists multiple acceptable outcomes. The narrow maintainable direction is a shared timeout health transition plus channel-specific copy after maintainers choose whether timed-out active sessions should be killed, rotated, or require /new.
