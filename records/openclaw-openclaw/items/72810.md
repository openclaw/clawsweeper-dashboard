## Summary

Keep open. Current main has adjacent cleanup that can clear active embedded runs, but it still does not implement the reported terminal timeout/session-health policy: a timed-out Discord route is not marked dead, rotated, or given partial-success-specific copy before later reuse.

## What This Changes

The issue requests that Discord-routed embedded-agent timeouts make session health unambiguous and distinguish possible partial success from total failure.

## Best Possible Solution

Add a shared terminal timeout/session-health transition in the embedded-agent reply/session pipeline, then have Discord routing and copy consume that state before the route is reused.

## Reproduction Assessment

Yes for the current source-level behavior gap: current code and tests show active-run diagnostics are observe-only by default, generic Discord raw-runner failure copy remains expected, and stored session reuse lacks a timeout-dead gate. A live Discord/Claude CLI timeout would add timing proof but is not required to confirm the missing policy.

## Solution Assessment

Unclear because no patch is proposed and the issue lists multiple acceptable behaviors. The narrow maintainable direction is a shared session-health state plus Discord-appropriate timeout/possible-partial-success copy, rather than a channel-only special case.
