## Summary

Keep open. Current main has adjacent cleanup for timeout and stale-lane recovery, but it still leaves the remaining product/runtime decision unresolved: active stuck-session recovery is warning-only by default, Discord direct runner failures still use generic copy, and docs describe active embedded/reply/lane work as serialized rather than failed, rotated, or forced fresh.

## What This Changes

The issue requests clearer Discord session health and routing after an embedded-agent timeout or pre-reply failure, including avoiding later routing to a wedged session and distinguishing possible partial success from total failure.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that either marks the affected chat session failed/dead, rotates it before later routing, or explicitly requires /new, and pair that with channel-appropriate copy warning that prior side effects may have completed. Keep it generic across chat channels with Discord regression coverage rather than a Discord-only workaround.

## Reproduction Assessment

Yes for source-level reproduction: current tests still assert generic Discord direct failure copy, and diagnostic recovery tests/docs still prove active work is warning-only by default. A live Discord plus Claude CLI timeout run would strengthen timing evidence, but it is not required to see the current-main behavior gap.

## Solution Assessment

Unclear because no patch is proposed and the remaining decision is runtime policy rather than a mechanical one-line fix. The narrow maintainable direction is a shared session-health transition plus timeout/partial-success copy once maintainers choose whether active side-effectful runs should be aborted, rotated, or marked dead.
