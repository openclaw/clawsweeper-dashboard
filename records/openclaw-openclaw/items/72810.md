## Summary

Keep open. Current main has adjacent timeout cleanup and terminal lifecycle backstops, but it still does not implement the reported Discord session-health policy: active stalled work is observe-only by default, Discord has no channel-owned timeout, fresh stored sessions can still be reused, and direct raw-runner failures still use generic copy.

## What This Changes

The issue asks OpenClaw to make Discord-routed embedded-agent timeouts unambiguous by marking or rotating dead sessions and distinguishing possible partial side effects from total failure.

## Best Possible Solution

Add a shared terminal timeout/session-health transition that marks or rotates the affected chat route after a fatal embedded-agent timeout, with Discord-facing copy that warns prior side effects may have completed.

## Reproduction Assessment

Yes for a source-level reproduction: current code classifies active stalled work as non-recovery-eligible, reuses fresh stored sessions, and keeps generic Discord direct raw-runner failure copy. A live Discord/Claude CLI timeout would strengthen timing proof but is not required to see the remaining policy gap.

## Solution Assessment

Unclear because no patch is proposed and the issue intentionally allows several policies. The narrow maintainable solution is to choose one shared terminal timeout transition and thread it through session initialization/routing plus Discord copy, rather than adding a Discord-only watchdog.
