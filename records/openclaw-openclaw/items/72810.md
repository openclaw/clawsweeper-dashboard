## Summary

Keep open. Current main has adjacent liveness classification and diagnostic cleanup, but it still does not implement the reported terminal timeout/session-health policy: active embedded runs remain non-recoverable by default, Discord direct raw-runner failures still use generic copy, and fresh stored sessions can still be reused without a timeout-dead or fresh-route gate.

## What This Changes

The issue requests clearer Discord embedded-agent session health and routing behavior after terminal timeouts or pre-reply failures that may have already produced side effects.

## Best Possible Solution

Add a shared terminal timeout/session-health transition in the embedded agent reply/session pipeline that makes the affected chat route unambiguous after fatal timeout: either mark it failed/dead, rotate it before the next turn, or explicitly require /new, with Discord-facing copy that warns prior side effects may have completed.

## Reproduction Assessment

Yes for the current source-level behavior gap: the code and tests show active-run diagnostics do not recover by default, Discord direct raw-runner failures still use generic copy, and session initialization can reuse a fresh stored session without a timeout-dead gate. A live Discord/Claude CLI timeout would strengthen timing proof but is not required to see the remaining policy gap.

## Solution Assessment

Unclear because no patch is proposed and the issue lists multiple acceptable behaviors. The narrow maintainable direction is a shared session-health transition plus Discord-appropriate timeout/partial-success copy after maintainers choose the active-run policy.
