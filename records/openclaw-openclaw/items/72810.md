## Summary

Keep open. Current main still logs active stalled Discord/CLI work as non-recoverable observation, only releases stale sessions with no active work, reuses fresh session entries without a timeout-dead gate, and still uses generic Discord direct-chat failure copy for raw runner failures.

## What This Changes

The issue requests clearer session-health handling for Discord-routed embedded agent timeouts so wedged sessions are no longer reused silently and partial success is not reported as a generic failure.

## Best Possible Solution

Add a shared terminal timeout/session-health transition that marks or rotates the affected chat route before the next turn, with Discord-facing copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes. Current source and docs show active stalled work is observed rather than recovered, reusable stored sessions lack a terminal timeout gate, and Discord direct raw-runner failures still use generic copy.

## Solution Assessment

Unclear because no patch is proposed and the issue lists several acceptable recovery policies. The narrow maintainable direction is a shared terminal timeout health transition plus explicit partial-success/timeout copy after maintainers choose the active-run policy.
