## Summary

Keep open. Current main has adjacent cleanup that makes some embedded-run failures return sessions to idle, but it still does not implement the requested terminal timeout/session-health policy: active stuck embedded runs remain observe-only by default, Discord direct runner failures still use generic copy, and the stored session reuse path has no failed/dead/fresh-route gate after a timeout.

## What This Changes

The issue asks OpenClaw to make Discord-routed embedded-agent session health unambiguous after terminal timeout or pre-reply failure, instead of leaving later routing ambiguous with generic failure text.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that either marks the affected chat session failed/dead, rotates it before later routing, or explicitly requires `/new`, with channel-appropriate copy warning that previous side effects may have completed.

## Reproduction Assessment

Yes for the current behavior gap: source and tests show generic Discord direct failure copy, warning-only active stuck-session recovery, and stored-session reuse without a timeout-dead health gate. A live Discord/Claude CLI timeout run would strengthen timing evidence but is not required to see the missing policy transition.

## Solution Assessment

Unclear because no patch is proposed and the issue intentionally allows multiple outcomes. The maintainable direction is a shared session-health transition plus timeout/partial-success copy once maintainers choose whether active side-effectful runs should be aborted, rotated, or marked dead.
