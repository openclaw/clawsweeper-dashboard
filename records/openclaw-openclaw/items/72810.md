## Summary

Keep open. Current main has adjacent cleanup for liveness diagnostics and terminal backstops, but it still does not implement the reported timeout/session-health policy: active stalled work is not recovery-eligible by default, fresh session initialization can reuse the same stored session, and Discord direct raw-runner failures still use generic copy unless verbose mode is enabled.

## What This Changes

The issue requests that Discord-routed embedded-agent timeouts make session health unambiguous and distinguish partial success from total failure.

## Best Possible Solution

Add a shared terminal timeout/session-health transition in the embedded agent reply pipeline that marks or rotates the affected chat session and gives Discord users timeout copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes for the current source-level gap: active stalled embedded work is not recovery-eligible by default, and Discord direct raw-runner failures still assert generic copy. A live Discord/Claude CLI timeout would add timing proof but is not required to show the missing policy.

## Solution Assessment

Unclear because no patch is proposed and the issue lists several acceptable behaviors. The narrow maintainable direction is a shared session-health transition plus Discord-facing timeout/partial-success copy after maintainers choose the active-run policy.
