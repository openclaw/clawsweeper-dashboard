## Summary

Keep open. Current main contains adjacent cleanup that helps terminal embedded runs return to idle, but the reported policy gap remains: active stuck-session recovery is still warning-only by default, Discord direct raw runner failures still use generic copy, and fresh stored sessions are reused without a timeout-dead/fresh-route health gate.

## What This Changes

The issue asks OpenClaw to make Discord-routed embedded-agent timeout state unambiguous after a partial-success or late-failure turn instead of leaving the session apparently routable with generic failure copy.

## Best Possible Solution

Add a shared terminal timeout/session-health transition in the embedded agent reply/session pipeline that either marks the affected chat session dead, rotates it before the next turn, or explicitly requires `/new`, and pair that with channel-appropriate copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes for the current behavior gap: source and tests show generic Discord direct failure copy, docs/source show active stuck-session recovery is warning-only, and session initialization still has no timeout-dead health gate. A live Discord plus Claude CLI timeout would strengthen timing proof but is not required to see the remaining policy gap.

## Solution Assessment

Unclear because no patch is proposed and the issue names several acceptable outcomes. The narrow maintainable direction is a shared session-health transition plus timeout/partial-success copy after maintainers choose the active-run policy.
