## Summary

Keep open. Current main still observes active Discord/CLI work as stalled rather than terminal, only recovers stale lanes without active work, reuses fresh stored sessions without a timeout-dead gate, and keeps normal Discord direct raw-runner failures on generic copy; the related closed stuck-session work is conservative and does not cover this active timeout/partial-success routing outcome.

## What This Changes

The issue asks OpenClaw to make Discord-routed agent sessions unambiguous after fatal timeout or pre-reply embedded-agent failure, especially when side effects may already have completed.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for active Discord/CLI runs that marks or rotates the affected route before the next turn and uses explicit partial-success/timeout copy.

## Reproduction Assessment

Yes, at source level: current main still classifies active stalled work as non-recoverable, keeps active abort opt-in, reuses fresh session rows without a timeout-dead gate, and tests normal Discord direct raw-runner failures against generic copy. A live Discord/Claude timeout would strengthen end-to-end timing proof but is not required to see the code-path gap.

## Solution Assessment

Unclear because no patch is proposed and the issue intentionally lists several acceptable policies. The narrow maintainable direction is a shared terminal timeout health transition plus explicit partial-success/timeout messaging once maintainers choose the active-run policy.
