## Summary

Keep open. Current main has conservative stale-lane recovery for #73581, but active embedded Discord/CLI work is still classified as non-recoverable, fresh stored sessions can still be reused after terminal state, and normal Discord direct raw-runner failures still use generic failure copy, so the timeout routing and partial-success state gap remains.

## What This Changes

Make Discord-routed embedded-agent timeouts leave session health/routing unambiguous and distinguish possible partial side effects from total failure.

## Best Possible Solution

Add a shared terminal-timeout/session-health transition that marks or rotates the affected chat route before the next turn, with Discord-facing copy that warns prior side effects may have completed.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: active embedded work is classified as stalled/long-running and not recovered, session reuse does not gate on terminal timeout health, and Discord direct raw-runner failures still expect generic copy. A live Discord/Claude CLI timeout would add timing proof but is not required to see the remaining behavior gap.

## Solution Assessment

Unclear because no patch is proposed and the issue lists several acceptable recovery policies. The narrow maintainable direction is shared terminal-timeout health handling plus explicit partial-success timeout copy after maintainers choose the active-run abort/rotation semantics.
