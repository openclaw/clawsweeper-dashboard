## Summary

Keep open. Current main has adjacent cleanup for terminal lifecycle events and stuck-session diagnostics, but it still does not implement the reported policy: active stuck sessions are warning-only by default, Discord direct raw runner failures still use generic copy, and fresh stored sessions can be reused without a timeout-dead or fresh-route health gate.

## What This Changes

The issue asks OpenClaw to stop routing Discord follow-up work to a session after a fatal embedded-agent timeout and to surface partial-success timeout state rather than generic failure text.

## Best Possible Solution

Add a shared terminal timeout/session-health transition in the embedded agent reply/session pipeline that either marks the affected chat session dead, rotates it before the next turn, or explicitly requires `/new`, and pair that with Discord-appropriate copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes for the current source-level behavior gap: source and tests show generic Discord direct failure copy, docs/source show active stuck-session recovery is warning-only, and session initialization still has no timeout-dead health gate. A live Discord plus Claude CLI timeout would strengthen timing proof, but it is not required to see the remaining policy gap.

## Solution Assessment

Unclear because no patch is proposed and the issue names several acceptable outcomes. The narrow maintainable direction is a shared session-health transition plus timeout/partial-success copy after maintainers choose the active-run policy.
