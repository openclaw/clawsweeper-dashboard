## Summary

Keep open. Current main has adjacent cleanup that reduces stuck `processing`/`running` states after embedded-run failures, but it still preserves the behavior this report asks maintainers to decide: active stuck-session recovery is warning-only by default, Discord direct runner failures still use generic copy, and there is no failed/dead/fresh-route session-health transition after a terminal timeout.

## What This Changes

The issue requests unambiguous Discord session health and user-facing partial-success/late-failure handling after a routed embedded-agent timeout.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that either marks the affected chat session failed/dead, rotates it before later routing, or explicitly requires `/new`, and pair that with channel-appropriate copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes for the current behavior gap: source and tests show generic Discord direct failure copy, docs/code show active stuck-session recovery is warning-only, and session initialization still reuses a fresh stored session without a timeout-dead health gate. A live Discord plus Claude CLI timeout run would strengthen timing evidence but is not required to see the remaining policy gap.

## Solution Assessment

Unclear because no patch is proposed and the issue offers several acceptable policy outcomes. The narrow maintainable direction is a shared session-health transition plus timeout/partial-success copy once maintainers choose whether active side-effectful runs should be aborted, rotated, or marked dead.
