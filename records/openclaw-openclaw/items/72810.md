## Summary

Keep open. Current main still observes active stalled Discord/CLI work without recovery, reuses fresh stored sessions without a timeout-dead gate, and keeps Discord direct raw-runner failures on generic copy; the closed #73581 context is related but does not cover this terminal timeout routing outcome.

## What This Changes

The issue asks OpenClaw to make Discord/CLI agent session health unambiguous after fatal timeouts or pre-reply failures, especially when side effects may already have completed.

## Best Possible Solution

Add a shared terminal timeout/session-health transition that marks or rotates the affected chat route before the next turn, plus Discord-facing copy that warns prior side effects may have completed.

## Reproduction Assessment

Yes for a source-level reproduction path: active no-progress work classifies as non-recoverable, the heartbeat only recovers eligible `session.stuck`, and fresh session entries are still reusable without a timeout-dead gate. A live Discord/Claude CLI timeout would add timing proof but is not required to see the current code gap.

## Solution Assessment

Unclear because no patch is proposed and the issue lists multiple acceptable policies. The narrow maintainable direction is a shared terminal timeout health transition plus explicit partial-success/timeout copy after maintainers choose the active-run policy.
