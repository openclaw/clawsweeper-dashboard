## Summary

Keep open. Current main still has the reported session-health policy gap: active stuck-session recovery is warning-only by default, Discord direct runner failures still fall back to generic copy, and fresh stored sessions can be reused without a timeout-dead or forced-fresh routing state. The closed Claude CLI regression in #72434 explains one trigger, but it does not resolve the Discord timeout/routing outcome described here.

## What This Changes

The issue requests clearer Discord embedded-agent timeout handling so a terminal failure after possible side effects marks or rotates the session instead of leaving routing ambiguous with generic failure copy.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that either marks the affected chat session failed/dead, rotates it before later routing, or explicitly requires `/new`, and pair it with channel-appropriate copy warning that prior side effects may have completed.

## Reproduction Assessment

Yes for the current behavior gap: source and tests show generic Discord direct failure copy, warning-only active stuck-session recovery, and stored-session reuse without a timeout-dead health gate. A live Discord/Claude CLI timeout run would strengthen timing proof but is not required to see the missing policy transition.

## Solution Assessment

Unclear because no patch is proposed and the issue lists several acceptable outcomes. The narrow maintainable direction is a shared session-health transition plus partial-success/timeout copy once maintainers choose whether active side-effectful runs should be aborted, rotated, or marked dead.
