## Summary

Keep open. The patch targets a real gateway restart race in current main and I found no code-level blocking finding, but the supplied real behavior proof covers an ordinary ready/restart/ready cycle rather than a restart signal arriving during startup.

## What This Changes

This PR queues gateway lifecycle signals that arrive before startup has returned a server handle, lets shutdown override a queued startup restart, and adds focused run-loop regression tests.

## Best Possible Solution

Land a narrow run-loop fix with focused regression tests, after the PR body includes redacted real terminal/log proof or a maintainer-approved explanation for why the exact startup-window race cannot be safely exercised live.

## Reproduction Assessment

Yes. Source inspection of current main shows a startup-window SIGUSR1 can be accepted before `server` and `restartResolver` exist, and the PR adds focused tests that force that timing window.

## Solution Assessment

Yes for the code direction. A single pending startup lifecycle request flushed after valid loop state is a narrow owner-boundary fix, with shutdown override preserved; the remaining blocker is proof quality, not a different implementation shape.
