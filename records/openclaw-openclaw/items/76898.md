## Summary

Keep open: the PR is draft and has the protected `maintainer` label, and the patch still misses the documented channel-prefixed Telegram target path, so it would not fully avoid Telegram CLI preload for explicit Telegram sends.

## What This Changes

The PR changes message CLI helpers to skip configured-channel plugin preload for non-dry-run Telegram gateway-owned actions, adds helper tests, and adds a changelog fix entry for #75477.

## Best Possible Solution

Use the same explicit-channel scope inputs for the preload guard as the existing preload resolver, keep dry-runs on the local preload path, and cover channel-prefixed Telegram sends in tests.

## Reproduction Assessment

Yes, source-reproducible: current main always preloads configured channel plugins before dispatch, and the PR only bypasses that when `opts.channel` is Telegram. The docs and current resolver show `telegram:...` target prefixes are another explicit Telegram path.

## Solution Assessment

No; the direction is right for `--channel telegram`, but the guard should reuse the existing channel/target/targets scope logic so the documented channel-prefixed form is fixed too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Honor channel-prefixed Telegram targets:** `src/cli/program/message/helpers.ts:79`
  - body: The new preload guard only resolves `opts.channel`, but `openclaw message` also treats channel-prefixed targets as explicit channel selection. A send like `--target telegram:123` will still call `ensurePluginRegistryLoaded({ onlyChannelIds: ["telegram"] })`, so the CLI can still enter the local Telegram preload path for a gateway-owned send. Reuse the existing resolver inputs (`channel`, `target`, `targets`) and add a regression for that form.
