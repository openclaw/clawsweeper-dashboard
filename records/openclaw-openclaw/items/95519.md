## Summary

Keep open: current main and v2026.6.9 still classify embedded provider error payloads for fallback only when they map to auth, auth_permanent, or billing. The open partial work at https://github.com/openclaw/openclaw/pull/94062 covers a bare `LLM request failed.` matcher path, but it does not clearly cover this returned `upstream_error` payload boundary.

## What This Changes

The issue asks OpenClaw to treat provider `upstream_error` / `LLM request failed` as fallbackable when model fallbacks are configured.

## Best Possible Solution

Extend the existing embedded result fallback classifier to reuse the transient upstream/server failover rules for provider error payloads, with focused tests that preserve the existing non-fallback guards.

## Reproduction Assessment

Yes at source level: the current classifier path can be traced to a null classification for non-auth upstream error payloads, but I did not run a live gateway/provider repro.

## Solution Assessment

Yes: the narrowest maintainable fix is to update the existing embedded result fallback classification path, not add a config option or a separate string allowlist.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/95519 by making the embedded agent model fallback result classifier treat transient provider upstream/server error payloads as fallback-worthy when a run returns an error result without visible or delivered output. Keep hook_block, aborts, committed delivery, side-effecting/replay-invalid results, and schema/format terminal errors out of fallback. Prefer reusing the shared failover classifier; do not add a new config option or broad `LLM request failed` substring match. Add focused regression coverage for an error payload shaped like `{"error":{"type":"upstream_error","message":"Upstream request failed","code":null}}` advancing to the next fallback and recording the failed candidate reason, while preserving existing business-denial and non-transient payload tests.
