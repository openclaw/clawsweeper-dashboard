## Summary

Keep open: current `main` still does not classify the reported embedded `upstream_error` provider payload as a fallback-worthy result, and several open PRs, especially https://github.com/openclaw/openclaw/pull/95542, are candidate fixes rather than merged implementations.

## What This Changes

The issue reports that configured model fallbacks are skipped when the primary provider returns an `upstream_error` / `LLM request failed` transient failure.

## Best Possible Solution

Land one canonical focused PR that updates the embedded result fallback classifier and structured upstream error mapping, then close this issue and supersede overlapping PRs.

## Reproduction Assessment

Yes at source level: current main routes embedded error payloads through a helper that drops server_error/timeout-style classifications, so the reported payload can stop fallback rotation. I did not run a live provider outage reproduction in this read-only review.

## Solution Assessment

Yes: the best fix belongs at the existing embedded result fallback classifier plus structured provider-error normalization, without adding config or a provider-specific workaround.
