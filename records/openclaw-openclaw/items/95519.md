## Summary

Keep open: current main and v2026.6.9 still route embedded error payload fallback through an auth/auth_permanent/billing-only classifier, so transient upstream/server provider payloads can fall through without trying configured model fallbacks. The open related PR only covers the bare `LLM request failed.` matcher and does not clearly fix this embedded payload filter.

## What This Changes

The issue requests that gateway/embedded runs with configured model fallbacks advance to the next model when the primary provider returns `upstream_error` / `LLM request failed`.

## Best Possible Solution

Extend the existing embedded result fallback classifier to treat shared transient provider-error classifications as fallback-worthy while preserving the current guards for visible output, hook blocks, side effects, and non-transient schema/format failures.

## Reproduction Assessment

Yes at source level: current main can be traced from embedded error payload classification to a null result for non-auth transient provider payloads, which prevents `runWithModelFallback` from advancing. I did not run a live gateway/provider repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to update the existing embedded result fallback classification path and shared transient signal coverage, not add a new config option or broad string allowlist.
