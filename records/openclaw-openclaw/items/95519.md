## Summary

Keep open: current main and v2026.6.9 still do not route the reported embedded provider `upstream_error` payload into model fallback, and several open PRs now explicitly own candidate fixes for this issue.

## What This Changes

The issue requests that configured model fallbacks run when the primary provider returns `upstream_error` / `Upstream request failed` instead of ending the turn as `LLM request failed`.

## Best Possible Solution

Land one narrow classifier fix that maps structured provider `upstream_error` / `Upstream request failed` into the existing transient failover path while preserving non-transient errors and adding focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an embedded error payload classified as `server_error` or `timeout` is filtered out by the current business-denial-only result classifier, and the tests still assert HTTP 500 payloads do not trigger fallback.

## Solution Assessment

Yes, the best direction is to fix the shared provider-error classification boundary used by embedded model fallback, not add a new config knob or downstream retry workaround.
