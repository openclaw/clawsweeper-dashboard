- `impact:message-loss`: The configured fallback chain can be skipped, leaving the user with a failed turn instead of a fallback model response.
- `impact:auth-provider`: The affected behavior is provider/model fallback routing after a primary provider returns a transient upstream error.

## Summary

Keep open: current main still does not classify the reported embedded provider `upstream_error` payload as fallback-worthy, and several open linked PRs are candidate fixes rather than merged implementation evidence.

## What This Changes

The issue asks OpenClaw to continue through configured model fallbacks when an embedded provider returns `upstream_error` / `Upstream request failed` instead of ending the turn with `LLM request failed`.

## Best Possible Solution

Land one narrow classifier fix that lets embedded transient provider payloads such as `upstream_error` rotate to configured fallback models while preserving existing guards for visible output, hook blocks, side effects, schema/format failures, and unknown internal errors.

## Reproduction Assessment

Yes, at source level. Current main can be traced from embedded error payload classification to a null result for non-auth transient provider payloads, which prevents `runWithModelFallback` from advancing.

## Solution Assessment

Yes, the maintainable fix is in the existing embedded result fallback classifier plus shared transient signal coverage. A broad generic `LLM request failed` allowlist would be riskier because some variants are intentionally terminal.
