## Summary

Keep open: current main still does not classify the reported embedded provider `upstream_error` / `Upstream request failed` payload as a fallback-worthy result, and several linked PRs reference this issue but none has merged yet.

## What This Changes

Fallback model chains should engage when the primary provider returns a transient upstream error instead of ending the embedded/gateway turn with `LLM request failed`.

## Best Possible Solution

Land one canonical fix that recognizes the reported transient provider error through the embedded result fallback classifier while preserving existing guards for visible output, hook blocks, side effects, and non-transient format/schema failures.

## Reproduction Assessment

Yes at source level: current main routes embedded provider error payloads through a narrow helper that drops transient server/upstream classifications, and a null result classification stops fallback advancement. I did not run a live provider outage reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable solution is to update the existing shared/embedded fallback classification path, not add a new config option or broad string allowlist. The remaining maintainer step is selecting and landing one proven PR from the linked candidates.
