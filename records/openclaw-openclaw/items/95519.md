## Summary

Keep open: current main still drops transient embedded provider-error payloads at the result fallback classifier, and several open PRs explicitly reference this issue as candidate fixes. The right cleanup is to keep the issue as the canonical report until one candidate lands, with https://github.com/openclaw/openclaw/pull/95542 currently looking like the strongest reviewed path from the linked discussion.

## What This Changes

The issue reports that configured model fallbacks are skipped when the primary provider returns a structured `upstream_error` / `Upstream request failed` payload.

## Best Possible Solution

Land one canonical fix that recognizes structured upstream provider failures in the shared classifier and embedded result-payload boundary, then close this issue and supersede the overlapping PRs.

## Reproduction Assessment

Yes. Source inspection traces the reported embedded provider error payload through a classifier that still returns null for server/upstream payloads, and the fallback loop treats null classification as a completed attempt rather than advancing candidates.

## Solution Assessment

Yes, the best fix belongs in the existing shared failover classifier plus embedded result-payload classifier boundary. It should not add a config knob or broadly treat every `LLM request failed` variant as transient.
