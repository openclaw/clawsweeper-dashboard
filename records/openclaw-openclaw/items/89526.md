- `P2`: This is a normal-priority gateway/auth diagnostic fix with limited blast radius, but it still needs a correctness repair before merge.

## Summary

Keep open: the diagnostic direction is useful, but the patch still ties fresh runtime-config drift checks to admin/sensitive callers while normal `openclaw health` uses the read-scoped health method, so the central operator path can still see a cached false-ok result; the supplied live proof also does not show the claimed ok-to-drift transition.

## What This Changes

The PR adds a `runtimeConfig` live-vs-disk drift summary to gateway health output, gates fingerprints/details behind sensitive snapshots, bypasses the health cache for admin callers, and adds regression tests around drift formatting, auth/model paths, and cache behavior.

## Best Possible Solution

Keep the diagnostic split from the restart-policy PR, but make the standard health path refresh or invalidate runtime-config drift without exposing fingerprints/details, then require exact-head redacted terminal proof showing normal health changes from ok to drift.

## Reproduction Assessment

Yes, from source inspection: normal health calls resolve to `operator.read`, while the PR bypasses the health cache only when `operator.admin` is present. I did not run a live gateway because this review is read-only.

## Solution Assessment

No: the runtimeConfig summary is a plausible diagnostic, but tying cache freshness to sensitive/admin output leaves the normal operator path stale. The better fix separates freshness from fingerprint/detail redaction or explicitly changes and tests the CLI scope contract.
