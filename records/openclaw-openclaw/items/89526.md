## Summary

Keep open. The diagnostic is useful, but the current merge result can still give misleading runtime-config health: unreadable or invalid disk config can be treated as drift, and normal cached health can hide fresh drift or admin fingerprint output for up to the health refresh interval; the latest force-push also is not covered by exact-head real behavior proof.

Adds `runtimeConfig` drift diagnostics to gateway health output, gates config fingerprints on sensitive snapshots, and adds health/config-reload regression tests.

## Best Possible Solution

Keep the health diagnostic, but make invalid/unreadable disk config an `unknown` state, ensure cached health cannot hide fresh admin/runtime-config drift, and add exact-head proof for admin and non-admin health output.

## Solution Assessment

No. The drift summary is the right layer, but this implementation needs to preserve invalid-config semantics and refresh or merge runtime-config drift before serving cached admin/non-probe health.

- **[P2] Treat unreadable disk config as unknown:** `src/commands/health.ts:1151`
  - body: `readSourceConfigBestEffort()` returns `{}` when the config file is missing, malformed, or cannot be read. Feeding that into the new drift comparison makes health report `state: "drift"` with restart-required wording instead of saying the disk source is unknown/invalid, so operators can get the wrong recovery instruction for a bad config file. Use a source read that preserves validity/read status, or treat the best-effort empty result as unknown unless it is known to be a real empty config.
  - confidence: 0.86
- **[P2] Refresh runtime-config drift before serving cached health:** `src/commands/health.ts:799`
  - body: The new `runtimeConfig` summary is added to the cacheable health payload, but the gateway health handler still serves a fresh-enough non-probe `healthCache` immediately and only refreshes in the background. Since `openclaw health` sends no `probe` unless verbose, a normal admin health check can miss fresh disk-vs-live drift, and can miss the admin-only fingerprints, until the next refresh. Force a fresh runtime-config check for this diagnostic or merge current drift state into cached responses.
  - confidence: 0.82
