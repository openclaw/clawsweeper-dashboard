## Summary

This PR should remain open because its central diagnostic does not represent the gateway’s adopted runtime state: the submitted real run remained `ok`, matching the current snapshot lifecycle rather than the divergent snapshots constructed by the tests.

## What This Changes

The PR adds live-versus-disk model, provider, and auth configuration diagnostics to gateway health, gates fingerprints to sensitive snapshots, bypasses cached health for admin callers, and adds regression tests.

## Best Possible Solution

Compare the latest validated disk source with a gateway-owned snapshot of configuration actually adopted by the running process, updating that snapshot only when the runtime applies the corresponding changes; retain admin-only fingerprints and keep hot-mode restart policy in the companion PR.

No. Gateway health is an appropriate diagnostic surface, but the live side must be the configuration actually adopted by the running gateway rather than source state promoted by the same file-watch event.
