## Summary

Keep open: source inspection confirms a real stale-runtime drift bug, but this PR is not merge-ready because it changes the documented `hot` reload-mode contract and lacks after-fix proof from a real gateway setup.

## What This Changes

The PR queues restart-required gateway reload plans from `hot` mode, adds runtime-vs-disk config drift diagnostics to health output, and extends gateway reload and health snapshot tests.

## Best Possible Solution

Preserve `hot` as the in-process-only mode by default, fix the stale drift with comparison/diagnostic handling, and only redefine reload-mode semantics with explicit maintainer approval and upgrade documentation.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main advances the reloader comparison snapshot before planning, then returns in hot mode for restart-required plans such as `gateway.auth.token`. That leaves a clear stale-runtime path after disk config changes.

## Solution Assessment

No. Queueing restarts fixes the drift, but it does so by making `hot` behave like `hybrid`; the safer fix preserves the existing mode contract and handles drift through diagnostics, retry/comparison state, or an explicit maintainer-approved migration.
