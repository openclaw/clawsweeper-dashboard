## Summary

Keep open. The exec/message boundary is the right fix direction, but PR head still lets direct path-like shell scripts with arbitrary dotted basenames skip shell-file preflight, so a shebang script can still run OpenClaw message delivery through exec.

## What This Changes

The PR adds exec preflight/message-delivery parsing, tests, docs, and a changelog entry to reject OpenClaw message delivery commands from agent exec while preserving non-delivery message commands.

## Best Possible Solution

Keep the exec guard, inspect path-like direct shell scripts regardless of dotted basename or by shebang, add a regression test, then rebase and rerun the targeted exec checks.

## Reproduction Assessment

Yes. Source inspection shows current main has no message-delivery guard before exec process launch, and PR head still skips direct file preflight for an executable shell script named like ./send.v1.

## Solution Assessment

No. The exec preflight/message-delivery boundary is the right place to solve this, but the implementation is incomplete until dotted path-like shell scripts are inspected before execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Inspect dotted direct shell scripts:** `src/agents/bash-tools.exec.ts:543-546`
  - body: The new direct-script detector only inspects known shell suffixes or path-like basenames with no dot. An executable `./send.v1` with a shell shebang skips `validateScriptFileForShellBleed`, so an `openclaw message send` inside can still run through exec. Please treat path-like direct script targets as inspectable regardless of arbitrary dotted basenames, or inspect the shebang before deciding to skip.
  - confidence: 0.88
