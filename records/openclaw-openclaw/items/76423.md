## Summary

Keep open. The PR addresses a real exec boundary bug that is still present on current main, but the current head still allows direct path-like shell scripts with arbitrary dotted names to skip shell-file preflight and run message delivery through exec.

## What This Changes

The PR adds exec preflight parsing, tests, docs, and a changelog entry to reject OpenClaw message delivery commands from agent `exec` while preserving non-delivery message commands.

## Best Possible Solution

Keep the exec preflight boundary, inspect direct path-like shell script targets regardless of arbitrary dotted names or by shebang before execution, and land the change with a focused dotted-script regression test.

## Reproduction Assessment

Yes. Source inspection shows current main lets message CLI delivery commands reach `messageCommand` from exec, and PR head still has a clear `./send.v1` direct-script bypass through the line 543-546 predicate.

## Solution Assessment

No. Blocking message delivery at the exec preflight boundary is the right approach, but the submitted implementation is incomplete until direct path-like scripts with dotted names are inspected or shebang-checked.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Inspect dotted direct shell scripts:** `src/agents/bash-tools.exec.ts:543-546`
  - body: The new direct-script guard only inspects commands with a known shell suffix or path-like names with no dot. An executable `./send.v1` with a shell shebang skips `validateScriptFileForShellBleed`, so an `openclaw message send` inside can still reach `runExecProcess` through exec. Please treat path-like direct script targets as inspectable regardless of arbitrary dotted names, or check the shebang before skipping.
  - confidence: 0.88
