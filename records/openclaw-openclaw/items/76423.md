## Summary

Keep open. The PR targets the right exec boundary and the follow-up commit fixes the earlier extensionless-script gap, but PR head still misses direct executable shell scripts with arbitrary dotted path names, leaving a concrete message-send bypass before merge.

## What This Changes

The PR adds exec preflight parsing, tests, docs, and a changelog entry to reject OpenClaw message delivery commands from agent `exec` while preserving non-delivery message commands.

## Best Possible Solution

Keep the exec preflight boundary, inspect direct path-like shell script targets regardless of arbitrary dotted names or by shebang, add a focused regression test, then rebase and rerun the targeted exec checks.

## Reproduction Assessment

Yes. Source inspection shows current main lets `openclaw message send` reach the message CLI delivery path from exec, and PR head still lets a direct `./send.v1` shell script skip shell-file inspection because the new predicate rejects dotted names outside the suffix allowlist.

## Solution Assessment

No. The exec preflight boundary is the right fix, but this implementation is still incomplete until direct path-like shell scripts with arbitrary dotted names are inspected before execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Inspect dotted direct shell scripts:** `src/agents/bash-tools.exec.ts:543-546`
  - body: The new direct-script guard only inspects commands with a known shell suffix or path-like names with no dot. An executable `./send.v1` with a shell shebang skips `validateScriptFileForShellBleed`, so an `openclaw message send` inside can still reach `runExecProcess` through exec. Please treat path-like direct script targets as inspectable regardless of suffix, or check the shebang before deciding to skip.
  - confidence: 0.88
