## Summary

Keep open. The PR targets the right exec preflight boundary and covers the main message-delivery paths, but review found a blocking bypass for direct executable shell scripts without a shell-like suffix; GitHub also reports the branch as dirty before it can land.

## What This Changes

The PR adds exec preflight parsing to reject OpenClaw message delivery commands from direct CLI, package-runner, shell-wrapper, heredoc/here-string, and shell-script forms, with tests, exec docs, and a changelog entry.

## Best Possible Solution

Keep the exec preflight boundary, extend shell-script inspection to cover direct executable/shebang shell scripts without requiring a suffix, then land after rebase and exact-head checks.

## Reproduction Assessment

Yes. Source inspection shows current main lets `openclaw message send` reach the message CLI delivery path from exec, and the PR-head bypass follows directly from the suffix-only direct shell-script target extractor.

## Solution Assessment

No. The exec preflight boundary is the right place to solve this, but the current implementation is not complete enough because direct no-extension executable shell scripts skip the new shell-file inspection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Inspect direct executable shell scripts:** `src/agents/bash-tools.exec.ts:537`
  - body: The new direct-script path only treats commands with `.bash`, `.fish`, `.ksh`, `.sh`, `.zsh`, or `.command` suffixes as shell scripts. A generated executable named `./send` with a shell shebang is a normal shell-script form, but it produces no target for `validateScriptFileForShellBleed`, so an `openclaw message send ...` inside still reaches `runExecProcess`.
  - confidence: 0.88
