## Summary

Keep open. The PR targets a real exec boundary bug and adds the right kind of guard, but current PR head still leaves a concrete dotted direct shell-script bypass and is dirty against base.

## What This Changes

The PR adds exec preflight parsing, tests, docs, and a changelog entry to reject OpenClaw message delivery commands from agent `exec` while preserving non-delivery message commands.

## Best Possible Solution

Keep the exec preflight boundary, inspect path-like direct shell script targets regardless of arbitrary dotted basenames or by shebang, add a focused regression test, then rebase and rerun the targeted exec checks.

## Reproduction Assessment

Yes. Source inspection shows current main lets `openclaw message send` reach the message CLI delivery path from exec, and PR head still lets a direct `./send.v1` shell script skip shell-file inspection.

## Solution Assessment

No. The exec preflight boundary is the right fix, but this implementation is incomplete until direct path-like shell scripts with arbitrary dotted names are inspected before execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Inspect dotted direct shell scripts:** `src/agents/bash-tools.exec.ts:543-546`
  - body: The new direct-script guard only inspects commands with a known shell suffix or path-like names with no dot. An executable `./send.v1` with a shell shebang skips `validateScriptFileForShellBleed`, so an `openclaw message send` inside can still reach exec. Please treat path-like direct script targets as inspectable regardless of suffix, or check the shebang before deciding to skip.
  - confidence: 0.88
