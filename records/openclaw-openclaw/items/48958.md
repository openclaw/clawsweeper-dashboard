## What This Changes

The PR adds a relevant-memory block stripper before embedded-runner prompt image detection and adds tests for ignoring media markers inside those blocks.

## Best Possible Solution

Rebase the PR onto current main, keep or replace the scanner-level defense only if it still protects a real current path, and add focused regression coverage plus redacted runtime proof showing stale memory media is not hydrated.

## Reproduction Assessment

Yes at the source-helper level: `detectImageReferences()` matches `[media attached: ...]` anywhere in the prompt string, and memory auto-recall can generate `<relevant-memories>` text via `prependContext`. I did not run a live memory or Telegram setup, and current embedded-runner prompt-boundary code may prevent the normal runtime path.

## Solution Assessment

Unclear. Scanner-level stripping is a narrow defense for the older failure mode, but current main now separates runtime context before prompt image loading, so the PR should be rebased and validated against that architecture before merge.
