## Summary

Keep open: this is a real prompt/media bug and the PR is directionally useful, but the current diff is not merge-ready because it only sanitizes one embedded path and leaves other injected-memory image paths open.

## What This Changes

This PR strips `<relevant-memories>` / `<relevant_memories>` blocks before embedded runner prompt-image detection and adds regression tests in `src/agents/pi-embedded-runner/run/images.test.ts`.

## Best Possible Solution

Land a shared prompt-image sanitization boundary that strips injected memory and untrusted-context scaffolding before any embedded or CLI prompt image detection, while keeping current-turn attachments outside those blocks intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: prepend hook memory context containing `[media attached: /tmp/photo.png (image/png)]` to a vision-capable embedded or CLI prompt, and current main scans that injected text through `detectImageReferences()` before loading prompt images.

## Solution Assessment

No. The PR’s direction is reasonable, but the maintainable fix should sanitize the shared prompt-image scanning boundary used by embedded and CLI paths, cover active-memory untrusted context, and fix the case-sensitive precheck/test assertion issues.
