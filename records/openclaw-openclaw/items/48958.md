## Summary

Keep open: this is a plausible narrow bug fix, but the external PR still lacks after-fix real behavior proof and has two small review issues to clean up before merge.

## What This Changes

The PR strips `<relevant-memories>` / `<relevant_memories>` blocks before prompt image-reference detection and adds focused tests for stale media markers in those blocks.

## Best Possible Solution

Land a narrow image-loading boundary fix that ignores injected memory/runtime context before resolving prompt image refs, preserves current-turn attachment hydration, and leaves the broader recent-history policy to https://github.com/openclaw/openclaw/issues/66702.

## Reproduction Assessment

Yes, at source level: `detectImageReferences()` scans `[media attached: ...]` markers from the prompt string, and hook `prependContext` can prepend injected context into that prompt path. I did not run a live memory or Telegram setup in this read-only review.

## Solution Assessment

Mostly yes: scanner-level stripping is a narrow defensive fix for the reported relevant-memory path. The PR should align the guard with the regex, remove or fix the ineffective test, and provide real after-fix behavior proof before merge.
