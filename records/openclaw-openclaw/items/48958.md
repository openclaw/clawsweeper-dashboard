## Summary

Keep this PR open for maintainer handling: the bug is real and the branch is useful, but the proposed patch is incomplete for the prompt/media boundary because it only sanitizes the embedded runner path and leaves the analogous CLI prompt-image path exposed. The newer open #66702 report also tracks the broader contamination family, including untrusted-context and recent-history image bleed, so this PR should not be treated as a complete fix as-is.

## What This Changes

The PR strips `<relevant-memories>` / `<relevant_memories>` blocks before embedded-runner prompt image detection and adds regression coverage in `src/agents/pi-embedded-runner/run/images.test.ts`.

## Best Possible Solution

Land a shared prompt-image sanitization boundary that removes injected memory and untrusted-context scaffolding before any current-turn prompt image detection, applies consistently to embedded and CLI prompt-image loaders, and leaves current-turn attachments outside those blocks intact. Keep #66702 as the broader follow-up for recent-history image pruning semantics if that needs a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: prepend a `<relevant-memories>` block containing `[media attached: /tmp/photo.png (image/png)]` to a vision-capable embedded or CLI prompt, and current main scans that injected text through `detectImageReferences()` before loading prompt images.

## Solution Assessment

No. The PR's direction is reasonable, but the best fix is to share the sanitizer across both embedded and CLI prompt-image loaders, remove the case-sensitive precheck mismatch, and avoid implying the broader #66702 contamination paths are solved.
