## Summary

Keep open: the PR is a plausible narrow bug fix for prompt image hydration, but current main still lacks this specific relevant-memory stripping, the external contributor has not provided real after-fix behavior proof, and two small review issues remain in the branch.

## What This Changes

The PR strips `<relevant-memories>` and `<relevant_memories>` blocks before prompt image-reference detection and adds focused tests for stale media markers in those blocks.

## Best Possible Solution

Land a rebased narrow image-loading boundary fix that ignores injected memory/runtime context before resolving prompt image refs, preserves current-turn attachment hydration, and leaves broader recent-history media policy to https://github.com/openclaw/openclaw/issues/66702.

## Reproduction Assessment

Yes, at source level: current main still has a prompt-string image scanner, and a prompt containing `<relevant-memories>` with `[media attached: ...]` markers can flow into `detectImageReferences()`. I did not run a live memory or Telegram setup in this read-only review.

## Solution Assessment

Mostly yes: scanner-level stripping is a narrow defensive fix for the relevant-memory path. The branch should fix the guard/test issues, provide real after-fix proof, and be validated against current main's newer prompt/runtime-context split.
