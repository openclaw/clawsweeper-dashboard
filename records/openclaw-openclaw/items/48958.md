## Reproduction Assessment

Yes, at source level: a prompt string containing `<relevant-memories>` with `[media attached: ...]` markers can still be scanned by `detectImageReferences()` when passed to `detectAndLoadPromptImages()`. I did not run a live memory or Telegram setup in this read-only review.

## Solution Assessment

Mostly yes: scanner-level stripping is a narrow defensive fix for this relevant-memory path. The branch should fix the guard/test issues, provide real after-fix proof, and validate against current `main`'s prompt/runtime-context split.

- **[P3] Make the memory-block guard case-insensitive:** `src/agents/pi-embedded-runner/run/images.ts:26-30`
  - body: `stripInjectedMemoryBlocks` uses a case-sensitive `text.includes("relevant")` guard before a case-insensitive regex. Mixed-case relevant-memory tags would skip stripping even though the regex is written to match them, so lowercase the guard input or remove the fast path.
  - confidence: 0.86
- **[P3] Replace the vacuous parser assertion:** `src/agents/pi-embedded-runner/run/images.test.ts:316`
  - body: `expect(refs.length).toBeGreaterThanOrEqual(0)` passes for every array length, so this test does not verify parser behavior or caller-level stripping. Assert the raw parser behavior or remove the test and rely on the integration cases.
  - confidence: 0.91
