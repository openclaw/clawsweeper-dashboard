## Summary

Keep this PR open. Current main still has the phantom prompt-image path, and the proposed branch is directionally useful but incomplete because it only sanitizes the embedded runner path and has an introduced case-variant bypass in the new helper.

## What This Changes

This PR strips `<relevant-memories>` / `<relevant_memories>` blocks before embedded-runner prompt image detection and adds regression tests for memory-context image references.

## Best Possible Solution

Land a shared prompt-image sanitization boundary that excludes injected memory scaffolding before any current-turn prompt image detection, and apply it consistently to embedded and CLI prompt-image paths while preserving legitimate current-turn attachments outside the memory block.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: create a prompt with `<relevant-memories>` containing `[media attached: /path/to/image.jpg]`, pass it to the embedded image loader or CLI prompt-ref loader with a vision-capable model, and the current code scans that injected block for image refs.

## Solution Assessment

No. The proposed approach is a useful start, but the best fix should share the sanitizer across prompt-image loaders and remove the case-sensitive fast-path mismatch before it can be treated as a complete fix.
