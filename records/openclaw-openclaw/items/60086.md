## Summary

Keep open. The PR addresses a plausible auto-reply UX bug, but the current head regresses stale active-run drain recovery by restarting only after accepted enqueues, and it still lacks required real behavior proof.

## What This Changes

The PR adds structured follow-up enqueue results, queued/backlog-full auto-reply acknowledgements, stale `/approve` recovery copy, and focused tests.

## Best Possible Solution

Keep the detailed enqueue and stale-approval UX direction, but restart stale drains whenever the active snapshot is stale, add rejected-enqueue stale-snapshot coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection reproduces the PR-introduced liveness bug by combining `isRunActive?.() === false` with a `duplicate` or `queue_full` detailed enqueue result, while current main's e2e test defines stale snapshots as immediate drains.

## Solution Assessment

No. The direction is maintainable, but the patch should keep stale-drain recovery independent of enqueue acceptance and should prove the visible behavior in a real channel setup.
