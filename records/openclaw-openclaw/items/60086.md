## Summary

Keep open. The PR is useful, but the current head still has a blocking liveness regression in the active-run follow-up path and only reports Vitest verification, not after-fix real behavior proof.

## What This Changes

The PR adds structured follow-up enqueue results, queued/backlog-full auto-reply acknowledgements, stale `/approve` recovery copy for eligible channels, and focused tests.

## Best Possible Solution

Keep the detailed enqueue and clearer stale-approval UX direction, but restart stale drains whenever the active snapshot is stale, add rejected-enqueue stale-snapshot coverage, and require redacted real channel proof before merge.

## Reproduction Assessment

Yes. Source inspection reproduces the PR-introduced liveness bug by combining `isRunActive?.() === false` with a `duplicate` or `queue_full` detailed enqueue result, while current main's e2e test defines stale snapshots as immediate drains.

## Solution Assessment

No. The direction is maintainable, but stale-drain recovery should not depend on enqueue acceptance, and the visible behavior still needs real channel proof.
