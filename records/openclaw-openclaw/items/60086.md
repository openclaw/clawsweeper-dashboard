## Summary

Keep open: the branch is a focused bug-fix direction, but it still introduces a stale-drain liveness regression and has only test-based verification for user-visible reply behavior.

## What This Changes

The PR adds stale `/approve` recovery copy, structured follow-up enqueue results, queued/backlog-full auto-reply acknowledgements, and focused tests around those paths.

## Best Possible Solution

Keep the detailed enqueue and clearer stale-approval UX direction, but restart stale drains regardless of enqueue acceptance and require redacted real channel proof before merge.

## Reproduction Assessment

Yes. Source inspection shows the PR-introduced regression by combining `isRunActive?.() === false` with a `duplicate` or `queue_full` detailed enqueue result, while current main and its e2e test require stale active snapshots to start draining immediately.

## Solution Assessment

No. The overall direction is maintainable, but the stale-drain recovery must not depend on enqueue acceptance, and the user-visible acknowledgement behavior still needs real setup proof.
