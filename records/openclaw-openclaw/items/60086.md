## Summary

Keep open. The PR addresses a real gap that current main still has, but the proposed patch is not mergeable as-is: it gates stale-active follow-up drain recovery on the new enqueue result and can strand already queued work when a duplicate or full-queue enqueue is rejected.

## What This Changes

The PR adds structured follow-up enqueue results, explicit active-run queue acknowledgements, stale `/approve` recovery copy, and focused auto-reply queue and approval tests.

## Best Possible Solution

Rebase or reapply the useful pieces onto current main, keep the structured enqueue result and user acknowledgements, preserve stale-active drain restart regardless of enqueue acceptance, add a regression for rejected enqueue plus stale `isRunActive() === false`, and include the required changelog entry for the user-visible reply changes.

## Reproduction Assessment

Yes. The PR body gives direct repro paths for stale `/approve`, active-run follow-up enqueueing, and backlog overflow, and the introduced blocker can be reproduced with `isRunActive()` false plus a duplicate or `queue_full` detailed enqueue result.

## Solution Assessment

No. The direction is maintainable, but the current implementation couples user acknowledgement to drain liveness; the safer solution is to keep stale snapshot recovery unconditional while separately returning queued, duplicate, or backlog-full reply payloads.
