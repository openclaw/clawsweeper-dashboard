## Summary

Keep open: current main still lacks the requested structured queue acknowledgements and stale `/approve` recovery copy, but this PR remains merge-blocked by a liveness regression in the proposed enqueue path and by missing real behavior proof.

## What This Changes

This PR adds structured follow-up enqueue results, queued/backlog-full reply acknowledgements, stale `/approve` recovery copy, and focused auto-reply tests.

## Best Possible Solution

Reapply the useful behavior on current main, keep stale-drain recovery independent of enqueue acceptance, add a rejected-enqueue stale-snapshot regression, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. The merge blocker is source-reproducible by combining `isRunActive?.() === false` with a duplicate or `queue_full` detailed enqueue result, which skips the PR’s accepted-only drain restart path.

## Solution Assessment

No. The feature direction is maintainable, but the current PR implementation couples stale-run liveness to accepting the latest message; the safer solution returns detailed acknowledgements while preserving the existing unconditional stale-drain restart after enqueue attempts.
