## Summary

Keep open: current main still lets `config.patch` merge against the runtime-shaped snapshot and validate the result without the default-stripping guard, while prior fixes only covered the Control UI and full `config.set`/`config.apply` writes.

## What This Changes

The issue asks for gateway `config.patch` to strip runtime-injected built-in provider defaults before validation so unrelated patches do not fail when `baseUrl` was omitted.

## Best Possible Solution

Keep this open and land a narrow gateway `config.patch` normalization/test change that preserves strict provider validation while sharing the existing runtime-default stripping behavior.

## Reproduction Assessment

Yes at source level: current main materializes missing built-in provider `baseUrl` as `""`, and `config.patch` validates the runtime-shaped merge without stripping that value. I did not run a live RPC repro in this read-only pass.

## Solution Assessment

Yes: the best fix is to apply the existing source/runtime projection and `stripBundledProviderRuntimeDefaults` behavior to `config.patch`, not to relax the core non-empty `baseUrl` schema.
