## Summary

Keep open: current main still lets `config.patch` merge against the runtime-shaped snapshot and validate the result without the bundled-provider default stripping guard, while the previous fixes only covered Control UI/full-config write paths.

## What This Changes

The issue reports that gateway `config.patch` rejects unrelated patches when built-in provider config has an injected empty `baseUrl` that was not authored in `openclaw.json`.

## Best Possible Solution

Land a narrow gateway `config.patch` normalization and regression test that reuses the existing source/runtime projection plus bundled provider default stripping without relaxing the provider schema.

## Reproduction Assessment

Yes at source level: current main materializes missing built-in provider `baseUrl` as `""`, and `config.patch` validates the runtime-shaped merge without stripping that value. I did not run a live RPC repro in this read-only pass.

## Solution Assessment

Yes: the maintainable fix is to apply the existing source/runtime projection and `stripBundledProviderRuntimeDefaults` behavior to `config.patch`, not to loosen the core non-empty `baseUrl` schema.

Add focused regression coverage in `src/gateway/server.config-patch.test.ts` for a built-in provider entry with `agentRuntime.id` and no authored `baseUrl`, then patch an unrelated field through `config.patch` and verify success/persistence without `baseUrl`. Keep the change scoped to gateway config write handling and tests; no docs or changelog edit expected.
