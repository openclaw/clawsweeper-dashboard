## Summary

Keep open: the PR targets a real Tavily SecretRef fallback bug, but the submitted fix is too broad because any unavailable configured SecretRef can fall through to ambient `TAVILY_API_KEY`.

## What This Changes

The PR changes Tavily API-key normalization to inspect unresolved SecretRefs and fall through to the existing `TAVILY_API_KEY` environment fallback.

## Best Possible Solution

Use inspect mode with Firecrawl-style guardrails: only resolve the matching allowed `TAVILY_API_KEY` env ref, block non-env/non-matching/disallowed refs, add focused tests, and land one canonical PR for the linked issue.

## Reproduction Assessment

Yes, from source inspection: current main and `v2026.6.8` strictly resolve the configured Tavily SecretRef before evaluating the existing `TAVILY_API_KEY` fallback. I did not run the live CLI command in this read-only review.

## Solution Assessment

No. Inspect mode is the right primitive, but this implementation is too broad because it collapses all unavailable configured refs into a missing value instead of gating fallback to the matching Tavily env ref.
