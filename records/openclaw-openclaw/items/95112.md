## Summary

Keep open: the PR targets a real Tavily SecretRef/env fallback bug, but the patch treats every unavailable configured SecretRef as missing, so Tavily can silently use ambient `TAVILY_API_KEY` over an explicit file, exec, custom-provider, or non-matching env ref.

## What This Changes

The PR changes Tavily API-key normalization to inspect SecretRef values and let unavailable configured refs fall through to the `TAVILY_API_KEY` environment fallback.

## Best Possible Solution

Land a guarded Tavily inspect-mode resolver that follows the Firecrawl/xAI pattern, preserves explicit SecretRef boundaries, adds focused regression tests, and includes redacted source-branch behavior proof.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.8` strictly resolve the configured Tavily SecretRef before evaluating the existing `TAVILY_API_KEY` fallback, so an unresolved env ref can throw first. I did not run the CLI in this read-only review.

## Solution Assessment

No; inspect mode is the right direction, but treating every `configured_unavailable` result as missing is too broad. The safer fix is Firecrawl/xAI-style guard logic that only falls back for the supported `TAVILY_API_KEY` env ref and preserves blocked behavior for other refs.
