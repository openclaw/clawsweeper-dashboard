## Summary

Keep open: the PR uses the right non-throwing SecretRef primitive, but it currently treats every unavailable configured Tavily SecretRef as missing, which can route requests through ambient `TAVILY_API_KEY` and still lacks after-fix runtime proof.

## What This Changes

The PR changes Tavily API-key normalization to use `resolveSecretInputString({ mode: "inspect" })` so unavailable configured SecretRefs fall through to the existing `TAVILY_API_KEY` environment fallback.

## Best Possible Solution

Use inspect mode with Firecrawl-style guardrails: only resolve the allowed `TAVILY_API_KEY` env SecretRef through the configured provider/defaults, preserve blocked unavailable refs, add focused tests, and provide redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection: current main and `v2026.6.8` strictly resolve the configured Tavily SecretRef before evaluating the existing `TAVILY_API_KEY` fallback. I did not run the CLI because this review is read-only.

## Solution Assessment

No. Inspect mode is the right primitive, but this implementation is too broad because it collapses all unavailable configured refs into a missing value instead of gating fallback to the matching Tavily env ref.
