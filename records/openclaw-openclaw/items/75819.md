## Summary

Keep open. Current main has the service-marker guard but not the PPID-ancestry guard needed for #75691, so this PR is still useful; however, the patch should be revised because the new ancestry block is reached only for restartful updates and the user-facing fix is missing a changelog entry.

## What This Changes

The PR exports the existing restart-stale PID ancestry walker, uses it in `openclaw update` to reject managed-service package updates when the active Gateway runtime PID is in the caller ancestry, and adds CLI regression coverage.

## Best Possible Solution

Keep the ancestry-based guard, but run it for any package-manager update from inside the active Gateway process tree before package mutation, including the `--no-restart` path, while preserving ordinary external-shell updates and adding focused regression coverage plus a single-line Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. #75691 provides concrete systemd reproduction steps, current main shows only an env-marker guard, and the PR test can model the active Gateway runtime PID in the caller ancestry; the remaining gap can be reproduced by the same mock setup with `updateCommand({ yes: true, restart: false })`.

## Solution Assessment

No as written. Reusing the bounded ancestry walker is the right seam, but placing the block after the no-restart return leaves a package-mutation path open; moving that check earlier in the managed-service preflight is the safer narrow fix.
