## Summary

Keep open. Current main has the service-marker guard from #75729 but still lacks the no-marker PPID-ancestry guard this PR adds, and maintainer discussion explicitly says this is a distinct path to keep open; the diff is focused, with no functional or security findings, but merge still needs conflict/CI reconciliation.

## What This Changes

This PR exports the existing bounded PID ancestry walker and uses it in `openclaw update` to refuse restartful package-manager updates when the active Gateway runtime PID is in the caller ancestry, with a regression test and changelog entry.

## Best Possible Solution

Land the narrow PPID-ancestry guard after rebase and exact-head CI, preserving the already-merged #75729 service-marker guard and the changelog coverage for #75691.

## Reproduction Assessment

Yes. #75691 gives a systemd plus agent-exec reproduction for a no-marker Gateway-owned child; current main only has the service-marker guard, while this PR adds a direct regression test and live VPS proof for the active runtime PID appearing in the caller ancestry.

## Solution Assessment

Yes. Reusing the existing bounded ancestry walker and failing before service stop or package replacement is the narrowest maintainable fix for the reported restart footgun; a detached restart helper would be a larger separate design.
