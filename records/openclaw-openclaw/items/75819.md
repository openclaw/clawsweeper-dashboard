## Summary

Keep open. Current main still lacks the PPID-ancestry guard and maintainer context says this PR is distinct from #75729, but the branch still lets the no-marker `--no-restart` ancestry path reach package replacement before the new guard runs.

## What This Changes

The PR exports the bounded PID ancestry walker, adds an updater block when the active Gateway PID is in the caller ancestry, adds a regression test, and records the fix in the changelog.

## Best Possible Solution

Land a narrow ancestry guard that blocks package-manager updates from the active Gateway process tree before service stop or package replacement, preserving the merged #75729 marker guard and covering both restart and no-restart cases.

## Reproduction Assessment

Yes. #75691 gives a systemd plus agent-exec reproduction, and this review finding can be reproduced with a mocked running service PID in the caller ancestry while invoking `updateCommand({ yes: true, restart: false })` without service-marker env.

## Solution Assessment

No, not as written. Reusing the bounded ancestry walker is the right seam, but the block should run before the `--no-restart` early return once the active runtime PID is known.
