## Summary

Keep open. The PR is a plausible narrow fix for the updater safety gap: it reuses the existing ancestor-PID walker, checks the active managed Gateway runtime PID before stopping the service or replacing the package tree, and keeps external-shell managed-service updates working. I found no blocking functional or security defect, but the user-facing CLI fix is missing the required changelog entry and the PR head still has failing/pending checks that need normal merge-gate handling.

## What This Changes

The PR changes `openclaw update` package-manager flow to block Gateway-owned update invocations using service-marker and PID-ancestry checks, strips Gateway service markers from post-core update respawns, exports the existing ancestor PID helper, and adds CLI regression tests.

## Best Possible Solution

Keep the conservative ancestor/service-marker guard, marker stripping, and regression coverage, then add the required Unreleased changelog entry and merge only after the failing checks are green or clearly unrelated.

Yes. The linked issue gives concrete Gateway-owned `openclaw update` steps, and the PR adds focused tests for both inherited service markers and active Gateway PID ancestry; I verified the source path but did not run live systemd reproduction in this read-only review.

## Solution Assessment

Yes, with the changelog gap fixed. Reusing the existing bounded ancestor walker and failing before service stop/package replacement is the narrowest maintainable fix for the managed-service updater footgun; a detached helper remains a larger separate design path.
