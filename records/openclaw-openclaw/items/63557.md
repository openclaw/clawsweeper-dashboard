## Summary

Keep open. The PR is still the canonical implementation path for gateway-level tool-call audit logging, but the branch is conflicting with current main and still writes direct audit rows in the old HTTP handler instead of the shared `invokeGatewayTool` seam that now backs both HTTP and RPC.

## What This Changes

The PR adds a redacted `gateway-tool-audit.jsonl` writer, threads gateway audit context through OpenResponses and embedded tool execution, and adds focused gateway audit tests plus unrelated guardrail-test updates.

## Best Possible Solution

Centralize one outcome-aware, redacted audit write in the shared direct-invoke helper and embedded tool wrapper, covering allowed, denied, and failed gateway-originated attempts without adding query UI, retention, signing, or config policy in this PR.

## Reproduction Assessment

Yes for the PR review findings via source inspection. Current main routes both HTTP and RPC direct invokes through `invokeGatewayTool`, while the PR head appends only in the old HTTP handler and places embedded appends after blocked paths have already exited.

## Solution Assessment

No. The JSONL helper and redaction direction are reasonable, but the audit write belongs at the current shared helper and embedded outcome boundary so allowed, denied, RPC, and HTTP attempts are all covered consistently.
