## Summary

Keep this PR open: the audit-log direction is useful and tied to the canonical gateway audit request, but the branch is stale against current main and misses important direct-RPC and denied-attempt audit paths.

## What This Changes

The PR adds a gateway tool-call JSONL audit helper, redaction, OpenResponses audit context propagation, direct `/tools/invoke` audit writes, focused tests, and some unrelated test/cleanup updates.

## Best Possible Solution

Rebase into a focused audit PR that records redacted, outcome-aware HTTP, RPC, and OpenResponses tool attempts from the shared direct invoke helper and embedded wrapper.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes HTTP and RPC direct invocation through `invokeGatewayTool`, while the PR head writes only in the old HTTP handler and only after blocked returns. I did not run runtime tests because this was a read-only review.

## Solution Assessment

No. The proposed JSONL sink and redaction helper are a reasonable direction, but the maintainable solution needs to be integrated with current main's shared invoke path and must record denied outcomes as well as allowed executions.
