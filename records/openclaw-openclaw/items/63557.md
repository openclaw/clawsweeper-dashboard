## Summary

Keep open: current main still lacks the gateway tool-call audit trail, but this branch has blocking audit-coverage gaps for denied calls and is stale against the current shared tool-invoke path.

## What This Changes

The PR adds a gateway tool-call audit JSONL helper and threads audit context through direct Gateway tool invocation and OpenResponses-originated embedded tool execution.

## Best Possible Solution

Land the audit trail at the current shared invocation and embedded hook boundaries, recording allowed and denied outcomes with redacted args and status metadata; keep approval-decision query work in https://github.com/openclaw/openclaw/pull/72769 separate.

## Reproduction Assessment

Yes. Source inspection shows current main's shared helper has blocked-return paths without an audit sink, and the PR head writes audit rows only after the blocked direct and embedded branches.

## Solution Assessment

No. The direction is useful, but the implementation should be rebased onto `invokeGatewayTool` and should write success plus denied audit records with status/reason metadata from the shared paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Audit denied direct tool calls:** `src/gateway/tools-invoke-http.ts:297-303`
  - body: The new direct audit row is written only after `hookResult.blocked` returns. A before-tool veto, approval refusal, or loop block responds 403 and leaves no JSONL row, which weakens the incident-audit trail for denied attempts; write a redacted denied record before returning.
  - confidence: 0.89
- **[P2] Audit denied embedded tool calls:** `src/agents/pi-tools.before-tool-call.ts:443-448`
  - body: The embedded audit append runs after the blocked branch throws or returns. OpenResponses-originated vetoes and approval denials will skip the audit log, so record the denied outcome and reason in the blocked path too.
  - confidence: 0.86
- **[P2] Move audit into the shared invoke helper:** `src/gateway/tools-invoke-http.ts:297-303`
  - body: Current main routes both HTTP `/tools/invoke` and SDK/RPC `tools.invoke` through `invokeGatewayTool`, but this PR writes the direct audit row only in the older HTTP handler. After rebase, SDK/RPC direct tool calls would have no audit row; put the audit context/write in `src/gateway/tools-invoke-shared.ts` so both surfaces stay covered.
  - confidence: 0.84
