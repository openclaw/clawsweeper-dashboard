## Summary

Keep open: the PR is a useful gateway audit feature, but the current diff is stale against the shared HTTP/RPC tool-invoke path and misses denied tool attempts, which is a security-sensitive gap for an audit log.

## What This Changes

Adds a gateway tool-call JSONL audit helper, threads audit context through direct `/tools/invoke` and OpenResponses embedded tool execution, and updates focused tests plus several shared test guardrails.

## Best Possible Solution

Land a rebased gateway audit trail in `src/gateway/tools-invoke-shared.ts` and the embedded hook path that records allowed and denied outcomes with redacted args plus status/reason metadata, while keeping approval-decision querying in the separate related PR.

## Reproduction Assessment

Yes, source-reproducible. The PR head appends audit rows only after allowed outcomes, while current main routes direct HTTP and RPC calls through `invokeGatewayTool` and has no gateway audit sink symbols.

## Solution Assessment

No. The direction is useful, but the implementation should be rebased onto the shared invoke helper and should record denied outcomes as first-class audit records.
