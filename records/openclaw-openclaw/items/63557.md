## Summary

Keep open: current main still lacks the requested gateway tool-call audit sink, but this branch is not merge-ready because it writes the direct audit row in an old HTTP-only path and still misses denied direct and embedded tool attempts.

## What This Changes

The PR adds a gateway tool-call audit JSONL helper, threads audit context through OpenResponses and agent tool execution, audits direct `/tools/invoke` calls, and includes focused tests plus unrelated guardrail cleanup.

## Best Possible Solution

Implement one focused, outcome-aware gateway audit feature on current main that records redacted allowed, denied, and failed gateway tool attempts from the shared direct-invoke helper and embedded OpenResponses tool wrapper, while leaving query UI, rotation, retention, and signing for separate work.

## Reproduction Assessment

Yes for the review findings: current main routes HTTP `/tools/invoke` and RPC `tools.invoke` through `invokeGatewayTool`, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL helper and redaction direction are reasonable, but the audit write should move to shared gateway seams and record denied/error outcomes instead of only allowed HTTP and embedded success paths.
