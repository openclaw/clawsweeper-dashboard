Keep open; the audit helper direction is useful, but the branch is stale against current main's shared direct-invoke seam and would miss RPC plus denied tool attempts.

## What This Changes

The branch adds a redacted gateway tool-audit JSONL helper, threads audit context through `/tools/invoke` and OpenResponses tool execution, and updates focused tests plus unrelated guardrail tests.

## Best Possible Solution

Implement the audit sink at the shared direct-invoke helper and embedded tool wrapper so allowed, denied, and failed gateway-originated tool attempts get one redacted record without adding query UI or retention policy in this PR.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes HTTP and RPC direct invokes through `invokeGatewayTool`, while the PR head appends only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL helper and redaction approach are reasonable, but the implementation should move direct audit writes into the shared invoke path and record denied embedded/direct attempts before returning or throwing.
