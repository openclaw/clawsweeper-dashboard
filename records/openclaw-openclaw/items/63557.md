## Summary

Keep open: current main still lacks the requested gateway tool-call audit sink and the linked feature issue was already closed in favor of this PR, but the branch is not mergeable as-is because current main now routes direct HTTP and SDK/RPC tool invocation through a shared helper that this patch does not audit completely.

## What This Changes

The PR adds a gateway tool-call audit JSONL helper, threads redacted audit context through direct `/tools/invoke` and OpenResponses embedded execution, and carries unrelated memory/test cleanup from shared CI fixes.

## Best Possible Solution

Rebase into a focused audit implementation that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct invoke helper and the embedded OpenResponses wrapper, while keeping approval-query work separate.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP `/tools/invoke` and RPC `tools.invoke` through `invokeGatewayTool`, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation needs to move to the shared direct-invoke seam and record denied/error outcomes instead of only allowed HTTP and embedded success paths.
