## Summary

Keep open: the PR targets a real gateway audit gap and remains the canonical follow-up for the closed gateway audit issue, but the branch is stale against current main's shared direct-invoke helper and still misses RPC and denied-attempt audit coverage.

## What This Changes

The PR adds a redacted `gateway-tool-audit.jsonl` helper, threads audit context through gateway/OpenResponses tool execution, and updates focused tests plus some unrelated guardrail tests.

## Best Possible Solution

Centralize one redacted audit record per gateway-originated attempt at the current shared direct-invoke helper and embedded tool wrapper, covering allowed, denied, and failed outcomes without adding query UI or retention policy in this PR.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes HTTP and RPC direct invokes through `invokeGatewayTool`, while the PR head appends only in the old HTTP handler and after blocked paths return. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL helper and redaction approach are reasonable, but the implementation should move direct audit writes into the shared invoke path and emit outcome-aware denied rows before direct or embedded blocked paths exit.
