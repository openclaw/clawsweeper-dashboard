## Summary

Keep open: current main still lacks the requested gateway tool-call audit sink, and this PR is the canonical implementation path, but the branch is not mergeable as written because it misses current-main RPC/direct-invoke and denied-attempt audit paths.

## What This Changes

The PR adds a redacted `gateway-tool-audit.jsonl` helper, threads audit context through OpenResponses and agent tool execution, writes direct `/tools/invoke` audit records, and includes focused tests plus unrelated test/guardrail cleanup.

## Best Possible Solution

Implement one focused gateway audit feature on current main that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct-invoke helper and the embedded OpenResponses tool wrapper, leaving query UI, retention, rotation, and signing for separate work.

## Reproduction Assessment

Yes for the review findings: current main routes both HTTP `/tools/invoke` and RPC `tools.invoke` through `invokeGatewayTool`, while the PR writes only in the HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation should move direct audit writes to the shared helper and record denied/error outcomes instead of only allowed HTTP and embedded success paths.
