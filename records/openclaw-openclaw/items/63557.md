## Summary

Keep open. Current main still lacks the requested gateway tool-call audit sink, and https://github.com/openclaw/openclaw/issues/13131 was closed as superseded by this PR, but the branch needs changes because it misses the current shared HTTP/RPC direct-invoke seam and denied tool attempts.

## What This Changes

The PR adds a JSONL gateway tool-call audit helper, threads audit context through direct tools-invoke and OpenResponses execution, adds redaction coverage, and carries some unrelated test cleanup.

## Best Possible Solution

Rebase into a focused audit PR that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct invoke helper and the embedded OpenResponses wrapper.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP /tools/invoke and RPC tools.invoke through invokeGatewayTool, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation needs to live at the shared direct-invoke seam and record denied outcomes instead of only allowed HTTP and embedded success paths.
