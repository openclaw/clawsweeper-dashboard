## Summary

Keep open: the audit feature is useful, but the branch is not mergeable against current main because direct tool invocation now goes through a shared HTTP/RPC helper while this PR audits the stale HTTP-only path and misses denied outcomes.

## What This Changes

The PR adds a gateway tool-call audit JSONL helper, threads audit context through direct and OpenResponses tool execution, and includes tests plus unrelated shared CI/memory cleanup.

## Best Possible Solution

Rebase into a focused audit implementation that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct-invoke helper and the embedded OpenResponses wrapper, while keeping approval-query and retention/query UI work separate.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP /tools/invoke and RPC tools.invoke through invokeGatewayTool, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation needs to move to the shared direct-invoke seam and record denied/error outcomes instead of only allowed HTTP and embedded success paths.
