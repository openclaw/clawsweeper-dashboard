## Summary

Keep open: the audit-log direction is useful, but the current branch is not mergeable as-is because direct tool invocation has moved to a shared HTTP/RPC helper on main while this patch writes only from the old HTTP handler and only after blocked outcomes have already returned.

## What This Changes

The PR adds a gateway tool-call JSONL audit helper with redaction, threads audit context through direct HTTP and OpenResponses tool execution, adds focused tests, and also carries unrelated shared test cleanup.

## Best Possible Solution

Rebase into a focused audit PR that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct invoke helper and embedded OpenResponses wrapper, while dropping unrelated cleanup.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP `/tools/invoke` and SDK/RPC `tools.invoke` through `invokeGatewayTool`, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation needs to live at the shared direct-invoke seam and record denied outcomes instead of only allowed HTTP and embedded success paths.
