## Summary

Keep this PR open: the audit-log feature is useful and still not present on current main, but the branch is dirty against current main and the proposed implementation misses the newer RPC direct-invoke path plus denied tool attempts.

## What This Changes

The branch adds a gateway tool-call JSONL audit helper, redacts persisted arguments, threads audit context through OpenResponses and embedded tool execution, and updates focused gateway/agent tests plus some guardrail tests.

## Best Possible Solution

Rebase onto current main and record gateway audit entries from the shared direct invoke helper and embedded tool wrapper so allowed and denied HTTP, RPC, and OpenResponses attempts emit redacted records with outcome/reason metadata; keep approval-decision querying separate in https://github.com/openclaw/openclaw/pull/72769.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR appends after blocked returns/throws, while current main routes both HTTP and RPC direct invokes through the shared invokeGatewayTool helper. I did not run runtime commands because this was a read-only review.

## Solution Assessment

No. The audit direction is useful, but an HTTP-only success-path append is not the maintainable fix now that direct tool invocation is shared with RPC and security audit trails need blocked-attempt coverage.
