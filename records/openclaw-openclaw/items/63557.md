## Summary

Keep open: current main still lacks gateway tool-call audit logging, and this active PR remains the canonical implementation candidate for the closed gateway audit request, but the branch should not merge as-is because it misses current shared HTTP/RPC invocation and denied-attempt audit paths.

## What This Changes

The PR adds a gateway tool-audit JSONL writer, redacted argument records, audit context threading for `/tools/invoke` and `/v1/responses`, and focused tests plus several guardrail test updates.

## Best Possible Solution

Rebase onto current main and implement audit writes in the shared direct invoke helper plus the embedded tool wrapper so allowed and denied HTTP, RPC, and OpenResponses attempts are recorded with redacted args and status/reason metadata; keep approval-decision querying separate in https://github.com/openclaw/openclaw/pull/72769.

## Reproduction Assessment

Yes for the review findings. Source inspection of PR head shows audit appends after blocked returns/throws, and current main shows HTTP and RPC direct invokes share `invokeGatewayTool` with no audit path; no runtime command was run because this was a read-only review.

## Solution Assessment

No. The audit direction is useful, but the PR should move direct auditing into the shared invoke helper and decide how denied outcomes are represented instead of keeping an HTTP-only success-path append.
