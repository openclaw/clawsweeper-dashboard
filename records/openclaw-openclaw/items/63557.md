## Summary

Keep open: current main still lacks the requested gateway tool-call audit sink, but this PR is not merge-ready because it writes direct audit rows in the old HTTP handler while current main routes both HTTP and RPC direct tool calls through the shared `invokeGatewayTool` helper, and denied paths are still missed.

## What This Changes

The PR adds a redacted `gateway-tool-audit.jsonl` helper, threads audit context through `/tools/invoke` and OpenResponses tool execution, and updates gateway/agent tests plus unrelated guardrail test files.

## Best Possible Solution

Centralize one outcome-aware, redacted audit write in the shared direct-invoke helper and embedded tool wrapper, covering allowed, denied, and failed gateway-originated attempts without adding query UI, retention, or config policy in this PR.

## Reproduction Assessment

Yes for the PR review findings via source inspection. Current main routes HTTP and RPC direct invokes through `invokeGatewayTool`, while the PR head appends only in the old HTTP handler and places embedded appends after blocked paths have already exited.

## Solution Assessment

No. The JSONL helper and redaction direction are reasonable, but the audit writes should move to the current shared helper and emit outcome-aware denied rows before direct or embedded blocked paths return.
