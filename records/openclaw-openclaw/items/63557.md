## Summary

Keep open: the audit feature is useful, but the proposed patch is not correct against current main because it misses the SDK/RPC direct tool surface and denied tool attempts, which are central to a security audit log.

## What This Changes

The PR adds a gateway JSONL tool-audit helper, redaction, OpenResponses/agent audit context propagation, direct `/tools/invoke` audit writes, focused tests, and some unrelated cleanup/test changes.

## Best Possible Solution

Rebase into a focused audit PR that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct invoke helper and the embedded OpenResponses tool wrapper.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP `/tools/invoke` and SDK/RPC `tools.invoke` through `invokeGatewayTool`, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation needs to live at the shared direct-invoke seam and record denied outcomes instead of only allowed HTTP and embedded success paths.
