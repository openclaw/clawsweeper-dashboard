## Summary

Keep open: current main still has no gateway tool-call audit sink, but this PR is stale against the shared HTTP/RPC direct-invoke path and would create an incomplete security audit trail by missing RPC and denied attempts.

## What This Changes

The PR adds a gateway tool-call JSONL audit helper, threads audit context through OpenResponses embedded tool execution, and writes audit rows for direct `/tools/invoke` calls.

## Best Possible Solution

Rework this into a focused audit PR that records redacted, outcome-aware allowed, denied, and failed gateway tool attempts from the shared direct-invoke helper and the embedded OpenResponses wrapper.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP `/tools/invoke` and SDK/RPC `tools.invoke` through `invokeGatewayTool`, while the PR writes only in the old HTTP handler and after blocked returns. I did not run tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the implementation needs to live at the shared direct-invoke seam and record denied outcomes instead of only allowed HTTP and embedded success paths.
