## Summary

Keep open: current main still lacks the requested gateway tool-call audit sink, and this PR contains useful work, but the branch is not mergeable as-is because it targets the old HTTP-only direct invoke path and misses denied attempts that matter for audit reconstruction.

## What This Changes

The PR adds a gateway tool-call JSONL audit helper, threads audit context into OpenResponses and embedded tool execution, writes allowed direct `/tools/invoke` audit records, and updates focused tests plus some unrelated test/cleanup files.

## Best Possible Solution

Rebase into the current shared `invokeGatewayTool` and embedded-wrapper paths, then emit redacted, outcome-aware audit records for allowed and denied HTTP, RPC, and OpenResponses tool attempts.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes HTTP and RPC direct invocation through `invokeGatewayTool`, while the PR head writes only in the old HTTP handler and only after blocked returns. I did not run runtime tests because this was a read-only review.

## Solution Assessment

No. The JSONL sink and redaction helper are a reasonable direction, but the maintainable fix must be wired through the current shared direct-invoke helper and must record denied outcomes as well as allowed executions.
