## Summary

Keep open: current main still lacks a gateway tool-call audit sink, and this PR is useful, but it now misses the shared `tools.invoke` RPC path and denied/refused attempts after the direct-invoke refactor on main.

## What This Changes

The branch adds a redacted `gateway-tool-audit.jsonl` path for gateway tool calls across `/tools/invoke` and OpenResponses, plus unrelated memory-core and test guardrail updates.

## Best Possible Solution

Centralize one outcome-aware, redacted audit write in the shared direct-invoke helper and embedded tool wrapper, covering allowed, denied, and failed gateway-originated attempts without adding query UI, retention, or config policy in this PR.

## Reproduction Assessment

Yes for the PR review findings. Source inspection shows current main routes both HTTP and RPC direct invokes through `invokeGatewayTool`, while the PR appends direct audit records only in the stale HTTP handler and appends embedded records only after blocked paths have exited.

## Solution Assessment

No. The JSONL helper and redaction approach are reasonable, but the implementation should move direct audit writes to the current shared invoke seam and emit outcome-aware denied rows before direct or embedded blocked paths return.
