## Summary

Keep open. The feature is useful, but the current branch is conflict-marked and its audit writes are still attached to stale/partial paths: current main now routes both HTTP and RPC direct tool calls through a shared helper, and the PR misses denied direct and embedded attempts.

## What This Changes

The PR adds a gateway tool-call audit JSONL helper, threads audit context through direct HTTP tool invocation and OpenResponses embedded tool execution, and updates focused tests plus several unrelated guardrail expectations.

## Best Possible Solution

Rebase onto current main and implement tool-call audit in the shared direct invoke helper plus the embedded tool wrapper so allowed and denied HTTP, RPC, and OpenResponses attempts are recorded with redacted args and status/reason metadata; keep approval-decision querying separate in https://github.com/openclaw/openclaw/pull/72769.

## Reproduction Assessment

Yes. Source inspection is enough here: current main routes HTTP and RPC direct calls through `invokeGatewayTool`, while the PR writes only on the old HTTP success path and appends embedded audit records after the blocked branch has already returned or thrown.

## Solution Assessment

No. The feature direction is reasonable, but the mergeable shape should target the shared direct invoke helper and record denied outcomes instead of adding a success-only HTTP/OpenResponses audit path.
