## Summary

Keep this PR open, but it is not merge-ready: current main now routes direct HTTP and RPC tool invocation through a shared helper, while this branch audits only the old HTTP success path and misses denied attempts.

## What This Changes

The PR adds a gateway tool-call JSONL audit sink, threads audit context through gateway and agent tool execution paths, and updates focused gateway/agent tests plus unrelated test guardrail sync.

## Best Possible Solution

Rebase onto current main and implement gateway tool-call auditing in the shared direct invoke helper and embedded tool wrapper so allowed and denied HTTP, RPC, and OpenResponses attempts emit redacted outcome records; keep approval-decision querying in the separate approval-audit work.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main routes both HTTP and RPC direct invokes through `invokeGatewayTool`, while the PR appends only in the old HTTP path and after blocked returns/throws. I did not run tests because this review is read-only.

## Solution Assessment

No. The audit direction is useful, but the proposed placement is no longer the narrow maintainable solution after current main introduced shared HTTP/RPC direct tool invocation, and it omits denied attempts from the audit trail.
