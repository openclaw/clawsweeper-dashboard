## Summary

Keep open: current main still lacks the requested gateway tool-call audit sink, and the linked feature issue was closed as superseded by this PR, but the branch is stale against the merged `tools.invoke` RPC/shared-helper path and still misses RPC plus denied-attempt audit rows.

## What This Changes

The PR adds a redacted `gateway-tool-audit.jsonl` sink, threads gateway audit context through `/tools/invoke` and `/v1/responses`, and updates gateway/agent tests plus unrelated guardrail-test files.

## Best Possible Solution

Centralize one outcome-aware, redacted audit write in the shared direct-invoke helper and embedded tool wrapper, covering allowed, denied, and failed gateway-originated attempts without adding query UI, retention, or config policy in this PR.

## Reproduction Assessment

Yes for the PR review findings. Source inspection shows current main routes both HTTP and RPC direct invokes through `invokeGatewayTool`, while PR head appends only in the old HTTP handler and appends embedded records only after blocked paths have already exited.

## Solution Assessment

No. The JSONL helper and redaction approach are reasonable, but the implementation should move direct audit writes to the current shared invoke seam and emit outcome-aware denied rows before direct or embedded blocked paths return.
