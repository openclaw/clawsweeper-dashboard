## Summary

Keep open: the startup guard is useful, but against current main it misses the newer memory embedding provider startup contract reads, so the same malformed manifest can still abort configured memory-search provider planning.

## What This Changes

The PR adds guarded manifest contract reads for startup provider ownership checks and a regression test for a malformed contracts accessor beside a healthy Brave web-search provider.

## Best Possible Solution

Refresh the branch on current main, route memoryEmbeddingProviders and embeddingProviders through the same guarded contract reader, and add a regression beside a configured memorySearch provider.

## Reproduction Assessment

Yes. From source inspection, current main's startup loop directly reads manifest contract arrays, and configured memorySearch provider planning still dereferences memoryEmbeddingProviders and embeddingProviders without the new guard; I did not run the repro because this review is read-only.

## Solution Assessment

No, not yet. Guarding speech, web, generation, and voice reads is the right shape, but current main also has memory embedding startup provider contract reads that need the same helper before this is the best fix.
