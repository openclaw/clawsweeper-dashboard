## Summary

Keep open: the reported Groq formatter gap is still valid on current main, but this PR is not ready because the matcher misses known argument-bearing Groq strings, the test does not prove sandbox-policy rewriting, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a Groq-specific unknown-tool regex to `formatAssistantErrorText` and a formatter test for the bare Groq attempted-call error string.

## Best Possible Solution

Land a rebased matcher that extracts the leading tool name from Groq attempted-call validation strings, including inline argument forms, with sandbox-policy regression coverage and redacted live Groq proof.

## Reproduction Assessment

Yes, at source level. Current main's regex chain cannot match the Groq `attempted to call tool ... which was not in request.tools` format, and a related Groq report shows argument-bearing forms; I did not run a live Groq request.

## Solution Assessment

No, not yet. `formatAssistantErrorText` is the right seam, but this PR's matcher only handles bare names and the added test does not prove sandbox extraction or blocked-tool guidance.

  - confidence: 0.9
