## Summary

Keep open: the reported ACP/ACPX failure is source-reproducible and the warm-restore direction fits the existing session contract, but the PR still treats null or missing persisted PIDs as reusable and lacks observed after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation to restore resolved persistent sessions, adds an ACPX PID-liveness check and regression tests, and updates the changelog.

## Best Possible Solution

Tighten ACPX stable reuse so missing or dead persisted PIDs force a fresh launch, keep the resolved-session warm restore, and land only after targeted validation plus redacted patched restart proof.

## Reproduction Assessment

Yes, at source level. Current main skips resolved persistent ACP identities during startup reconciliation and ACPX stable reuse lacks a process-liveness gate; I did not run the docker restart path in this read-only review.

## Solution Assessment

No. The warm-restore approach is narrow and placed in the right owner boundary, but the PID predicate must reject missing PIDs before it can reliably force the fresh-launch path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Treat missing persisted PIDs as not reusable:** `extensions/acpx/src/runtime.ts:706`
  - body: The PR's own live repro shows a post-restart record can have no persisted PID, but this guard only rejects numbered dead PIDs. When `readRecordAgentPid` returns `undefined`, the function falls through to `return true`, suppressing the fresh launch that is needed for a dead or missing subprocess. Return false for missing PID records and add coverage for that shape.
  - confidence: 0.84
