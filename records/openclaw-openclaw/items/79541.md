## Summary

Keep open: the bug is source-reproducible on current main and the PR is pointed at the right ACP/ACPX path, but the current head still has a PID-guard correctness bug and lacks after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation to warm-restore resolved sessions, adds an ACPX PID-liveness check before stable reuse, exports the process-liveness helper, adds ACP/ACPX regression tests, and updates the changelog.

## Best Possible Solution

Keep the warm-restore direction, but reject missing or dead persisted PIDs before suppressing fresh launch, add coverage for the no-PID record shape, and require redacted after-fix docker restart proof before landing.

## Reproduction Assessment

Yes. Current main source shows resolved persistent ACP identities are skipped during startup reconciliation and ACPX stable reuse lacks a process-liveness gate; I did not run the live docker restart path in this read-only review.

## Solution Assessment

No. The warm-restore approach is narrow and in the right owner boundary, but the current PID predicate still treats records with no persisted PID as reusable, so it needs tightening before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Treat missing persisted PIDs as stale:** `extensions/acpx/src/runtime.ts:706`
  - body: The new liveness guard only rejects a record when `readRecordAgentPid` returns a numbered PID that is dead. A post-restart record with no PID has no live subprocess proof, but this still falls through to `return true`, disabling the fresh-launch lease that this fix relies on. Return false when a resumable persistent record lacks a PID and cover that case.
  - confidence: 0.84
