## Summary

Keep open: the reported ACP/ACPX restart bug is source-reproducible and the PR is aimed at the right area, but the current patch still accepts missing-PID records on the reuse path and lacks after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation to warm-restore resolved persistent sessions, adds ACPX PID-liveness gating before stable session reuse, exports the process liveness helper, and adds ACP/ACPX regression tests plus a changelog entry.

## Best Possible Solution

Reject missing, invalid, or unrecognized persisted PIDs before disabling the fresh-launch path, cover that no-PID record shape in ACPX tests, then land only after targeted checks and redacted after-fix restart proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main skips resolved ACP identities during startup reconciliation, and ACPX stable reuse can accept matching records without process-liveness proof. I did not run the Docker restart flow or tests in this read-only review.

## Solution Assessment

No. The warm-restore direction is narrow and appropriate, but the proposed PID guard still lets missing or unrecognized PID records reuse the stale path and the PR lacks after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Reject missing PID records before reuse:** `extensions/acpx/src/runtime.ts:706`
  - body: The new guard only rejects a record when `readRecordAgentPid(existing)` returns a PID and that PID is dead. The PR's own deployed-container evidence shows `agent_pid: null`, and the helper only reads `pid`/`processId`, so this falls through to `return true`, disables the fresh-launch lease, and can still reattach to stale state. Treat missing or unrecognized PID data as not reusable and add coverage for that shape.
  - confidence: 0.9
