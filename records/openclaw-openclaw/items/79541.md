## Summary

Keep open: the bug is source-reproducible on current main and the PR targets the right ACP/ACPX path, but the branch still accepts no-PID records on the stale reuse path and lacks after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation for resolved persistent sessions, adds an ACPX PID-liveness reuse guard and tests, exports the liveness helper, and updates the changelog.

## Best Possible Solution

Reject missing or dead persisted PIDs before suppressing ACPX fresh launch, keep the resolved-session warm-restore path, add no-PID regression coverage, then merge only after targeted checks and redacted real restart proof pass.

## Reproduction Assessment

Yes, source-level. Current main filters out resolved ACP identities during startup reconciliation and ACPX stable reuse is metadata-only; I did not run the Docker restart path in this read-only review.

## Solution Assessment

No. The warm-restore direction is narrow and maintainable, but this PR still lets missing or null PID records reuse the stale path and does not yet provide after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject records without a persisted PID:** `extensions/acpx/src/runtime.ts:706`
  - body: The new guard only returns `false` when a parsed PID exists and is dead. A restart record with `pid: null`, no `pid`, or an unparsed legacy PID field falls through to `return true`, disables the fresh-launch lease path, and can reuse the stale existing record instead of launching a child. Treat a missing PID as not reusable and add coverage for that case.
  - confidence: 0.84
