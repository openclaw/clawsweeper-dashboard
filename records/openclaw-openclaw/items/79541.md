## Summary

Keep open: this is a plausible ACP/ACPX bug fix that is not fully present on current main, but the PR still has a blocking PID-reuse bug and lacks after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation to resume resolved persistent sessions, adds an ACPX PID-liveness guard/export, adds ACP/ACPX regression tests, and updates CHANGELOG.md.

## Best Possible Solution

Keep the warm-restore direction, but reject missing or dead persisted PIDs before reuse, cover that case, and require redacted patched docker-restart proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main skips resolved ACP identities during startup reconciliation and ACPX reuse can return a matching persisted record without PID liveness; I did not run the live docker restart path in this read-only review.

## Solution Assessment

No. The proposed direction is narrow and in the right owner boundary, but the current PR still treats no-PID records as reusable and therefore misses the exact reported restart shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Treat missing persisted PIDs as not reusable:** `extensions/acpx/src/runtime.ts:706-709`
  - body: The new guard only rejects numbered dead PIDs. The reported restart record has no persisted PID, so this falls through to `return true`; without a launch-lease scope, acpx's metadata-only reuse path can return the stale record instead of starting/loading a fresh process. Return false for missing PID on this resume path and cover it.
