## Summary

Keep this PR open: the reported ACP warm-restore bug is valid and the direction is useful, but the current patch still accepts the no-PID restart record shape described by the contributor and lacks after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation to warm-restore resolved persistent sessions, adds ACPX PID-liveness gating before persistent reuse, exports the process liveness helper, adds ACP/ACPX regression tests, and updates the changelog.

## Best Possible Solution

Reject missing or dead persisted PIDs before suppressing the fresh-launch path, cover that no-PID case in ACPX tests, and land only after targeted validation plus redacted after-fix restart proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main skips resolved ACP identities during startup reconciliation, and ACPX stable reuse can accept matching records without process-liveness proof. I did not run the Docker restart path or tests because this review is read-only.

## Solution Assessment

No. The warm-restore approach is the right narrow direction, but the PR is not yet the best fix because the PID guard still lets missing-PID records reuse the stale path and after-fix real behavior proof is absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject missing PID records before reuse:** `extensions/acpx/src/runtime.ts:706`
  - body: The new guard only rejects when `readRecordAgentPid(existing)` returns a PID and that PID is dead. The deployed-container evidence in the PR shows `agent_pid: null`; that parses to `undefined`, falls through to `return true`, disables `runWithLaunchLease`, and lets ACPX reuse the stale record instead of launching a fresh child. Treat missing or invalid PID as not reusable and add coverage for that case.
  - confidence: 0.86
