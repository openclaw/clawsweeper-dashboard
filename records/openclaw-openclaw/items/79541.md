## Summary

Current main still has the reported ACP/ACPX restart gap, and this PR is directionally narrow, but the latest head still reuses records with no persisted PID and lacks after-fix real behavior proof from a patched setup.

## What This Changes

The PR widens ACP startup reconciliation to warm-restore resolved persistent sessions, adds an ACPX process-liveness check before stable reuse, exports the liveness helper, adds ACP/ACPX regression tests, and updates the changelog.

## Best Possible Solution

Tighten the ACPX reuse predicate so records without a live persisted PID take the fresh-launch path, add coverage for that no-PID case, and land after redacted after-fix docker restart proof plus targeted ACP/ACPX checks.

## Reproduction Assessment

Yes, at source level. Current main skips already-resolved ACP identities during startup reconciliation and ACPX stable reuse checks only metadata, matching the reported post-restart bound-session failure path; I did not run the live docker restart flow in this read-only review.

## Solution Assessment

No. The warm-restore direction is the right narrow owner-boundary fix, but the PR head still treats records with no persisted PID as reusable, which contradicts the deployed record shape the author reported.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject reuse when the persisted PID is missing:** `extensions/acpx/src/runtime.ts:706-709`
  - body: The new guard only rejects a record when `readRecordAgentPid` returns a numbered PID and that process is dead. The live repro in the PR shows the post-restart record has `agent_pid: null`, and `readRecordAgentPid` converts a missing/null PID to `undefined`, so this path falls through to `return true` and suppresses the fresh-launch lease. Return false for resumable persistent records with no persisted PID, and add a no-PID regression case.
  - confidence: 0.82
