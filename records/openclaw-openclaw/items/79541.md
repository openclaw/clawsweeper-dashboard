## Summary

Keep open: the reported bug and warm-restore direction are credible, but the PR still accepts no-PID restart records matching the supplied deployed repro and lacks after-fix real behavior proof.

## What This Changes

The PR widens ACP startup reconciliation to warm-restore resolved persistent sessions, adds an ACPX PID-liveness check before stable reuse, exports the liveness helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Reject missing or dead persisted PIDs before suppressing fresh launch, add missing-PID coverage, then land after targeted ACP/ACPX validation and redacted after-fix restart proof.

## Reproduction Assessment

Yes. Current-main source and tests show resolved ACP identities are skipped at startup, and ACPX stable reuse can accept a persisted record using only metadata; I did not run the Docker restart path in this read-only review.

## Solution Assessment

No. The warm-restore approach is narrow and maintainable, but the current PID guard still lets no-PID records reuse the stale path instead of forcing a fresh launch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject records without a persisted PID:** `extensions/acpx/src/runtime.ts:706`
  - body: The new guard only returns false when a parsed PID exists and is dead. A restart record with `agent_pid: null` or no `pid`, which is the deployed-container shape described in the PR, falls through to `return true`, disables the fresh-launch lease, and lets acpx return the stale existing record. Treat a missing PID as not reusable and add coverage for that case.
  - confidence: 0.86
