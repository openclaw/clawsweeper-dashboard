## Summary

Keep open: this is a plausible ACP restart bug fix, but the PR still needs after-fix real behavior proof and the new PID guard appears to keep treating no-PID records as reusable.

## What This Changes

The PR widens ACP startup reconciliation to warm-restore resolved persistent sessions, adds an ACPX PID-liveness check before stable reuse, exports the liveness helper, adds ACP/ACPX regression tests, and updates the changelog.

## Best Possible Solution

Reject missing or dead persisted PIDs before suppressing the fresh-launch path, then add redacted after-fix docker restart proof and land after targeted ACP/ACPX validation is green.

## Reproduction Assessment

Yes. Current main source shows resolved persistent ACP identities are skipped at startup and ACPX stable reuse lacks a process-liveness gate; I did not run the live docker restart path in this read-only review.

## Solution Assessment

No. The warm-restore direction is narrow and well placed, but the current PID predicate still treats records with no persisted PID as reusable, so it should be tightened before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Treat missing persisted PIDs as not reusable:** `extensions/acpx/src/runtime.ts:706`
  - body: The new liveness guard only rejects a record when `readRecordAgentPid` returns a numbered PID that is dead. A post-restart record with no PID has no live subprocess proof, but this still falls through to `return true`, suppressing the fresh launch lease this PR is meant to force. Return false when a resumable persistent record lacks a PID, and cover that case.
