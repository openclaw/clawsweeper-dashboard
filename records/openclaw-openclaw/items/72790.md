## Summary

Keep open. Current main still exposes these compaction token settings as absolute numeric values across the public config type, Zod validator, generated schema, docs, tests, and Pi runtime resolver, so the requested percentage-based config is not implemented. The request is narrow enough for a focused repair PR, with one small design choice around the exact token-field scope.

## What This Changes

Request support for percentage-based compaction token settings so values such as `reserveTokensFloor` and `keepRecentTokens` can scale with each model's effective context window.

## Best Possible Solution

Add a shared typed token-budget config value that preserves existing numeric behavior, accepts documented percentage strings, resolves them against the effective model context budget in Pi compaction paths, and updates generated schema, help text, docs, tests, and changelog together.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence static verification path. Current main's type, Zod schema, generated schema, docs, and runtime resolver all accept absolute numbers only for the relevant compaction token settings.

## Solution Assessment

Yes, the requested explicit percentage-string support is the narrow maintainable direction. Silently auto-scaling existing numeric configs would change established behavior and should not be used as the primary fix.
