## Summary

Keep open. Current main still exposes the compaction token knobs as numeric-only through the public type, Zod validation, generated schema, docs, and runtime threshold math, so the requested percentage-based config remains unimplemented. The earlier ClawSweeper comment remains consistent with current-main evidence and this is narrow enough for a focused repair PR.

## What This Changes

Add support for percentage-based compaction token settings so configured reserve and recent-token budgets can scale with the active model context window.

## Best Possible Solution

Introduce a shared typed token-budget value contract that preserves existing numeric configs, accepts documented percentage strings, resolves them against the effective model context budget at runtime, and reuses that resolver anywhere compaction/preflight math needs a concrete token count. Keep generated schema, docs, tests, and changelog aligned.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence static reproduction path: current type definitions, Zod validation, generated schema, docs, and runtime resolvers all accept or consume numeric token settings only.

## Solution Assessment

Yes. Explicit percentage-string support is the narrow maintainable direction because it keeps existing absolute numeric configs stable; auto-scaling existing numbers would silently change established behavior.
