## Summary

Keep open. Current main still rejects or ignores percentage string values for the relevant compaction token settings across the public type, Zod validation, generated schema, docs, and runtime resolver, so the requested enhancement remains valid and unimplemented.

## What This Changes

The issue requests percentage-based compaction token settings, such as `reserveTokensFloor: "10%"` and `keepRecentTokens: "15%"`, resolved relative to each model context window.

## Best Possible Solution

Add a shared typed token-budget value contract that preserves existing numeric configs, accepts documented percentage strings, resolves them once against the effective model context budget, and uses that resolver consistently in Pi settings plus preflight memory/compaction threshold paths. Update generated schema, docs, tests, and changelog in the same PR.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence static reproduction path: the current type, Zod schema, generated schema, docs, and Pi resolver all accept or consume numeric token settings only.

## Solution Assessment

Yes. Explicit percentage-string support is the narrowest maintainable direction because it keeps existing absolute numeric configs stable; auto-scaling existing numbers would silently change established behavior.
