## Summary

Keep open. Current main still exposes, validates, documents, and consumes the relevant compaction token-budget settings as numbers, so percent-string values remain unimplemented.

## What This Changes

Request support for percentage values such as `"10%"` on compaction token-budget settings so they scale with the effective model context window.

## Best Possible Solution

Extend the existing compaction token-budget contract with a shared number-or-percent resolver, resolved against the effective context window while preserving current numeric semantics.

## Reproduction Assessment

Yes, via source-level reproduction. Current main validates these knobs as numbers and the runtime readers ignore or cannot accept non-number values, so a config value like `"10%"` cannot reach context-aware resolution.

## Solution Assessment

Yes. Explicit percent-string support resolved against the effective context window is narrower and safer than auto-scaling existing numeric values, which would silently change current user configs.

## Repair Work Prompt

Implement percentage-based compaction token budgets for the existing `agents.defaults.compaction.reserveTokens`, `keepRecentTokens`, and `reserveTokensFloor` settings. Add a shared value type/resolver that accepts the current numeric values plus percent strings like `"10%"`, resolving percentages against the effective model context window where that budget is available. Preserve existing numeric semantics, do not auto-scale numeric configs, and do not add a new config key. Cover parsing, rounding, invalid values, zero/nonzero rules, and small-context clamping. Update OpenClaw config types, Zod schema, generated schema/help/docs, runtime resolution in `pi-settings`, memory preflight threshold handling, manual compaction recent-tail handling if needed, and changelog/docs for the user-facing feature.
