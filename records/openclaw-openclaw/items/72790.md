## Summary

Keep open. Current main still treats the relevant compaction token-budget knobs as numbers end-to-end; the prior ClawSweeper note remains accurate and this is a narrow but unimplemented config/runtime/docs feature.

## What This Changes

Request support for percent-string compaction token-budget values so settings scale with the effective model context window.

## Best Possible Solution

Add a shared number-or-percent token-budget resolver against the effective context window, preserve current numeric semantics, and cover the affected compaction fields in schema, runtime, docs, and tests.

## Reproduction Assessment

Yes, as a source-level feature gap. Current main validates the requested fields as numbers and runtime resolution ignores non-number values, so a config value like "10%" cannot validate or reach compaction math.

## Solution Assessment

Yes. Explicit percent-string support is narrower and safer than auto-scaling existing numeric values, which would silently change current user configs.

## Repair Work Prompt

Implement percentage-string support for the compaction token-budget settings requested in https://github.com/openclaw/openclaw/issues/72790. Add a shared resolver for number-or-percent values such as "10%" that resolves against the effective model context window, preserves all existing numeric semantics, and is used consistently for reserveTokensFloor, keepRecentTokens, and any directly related compaction budget path that currently reads these values. Pin validation, rounding, clamping, and unavailable-context behavior in tests. Update public config types, Zod schema, generated schema/docs checks, user docs, and CHANGELOG.md. Do not silently auto-scale existing numeric configs and do not introduce a new config option unless maintainers explicitly ask for one.
