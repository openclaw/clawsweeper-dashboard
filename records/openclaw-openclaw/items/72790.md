## Summary

Keep open. Current main still has numeric-only compaction token budgets, and the same feature is already being explored in two open PRs that need maintainer API/review handling before this issue can resolve.

## What This Changes

The issue requests context-window-relative compaction settings so reserve and keep-recent token budgets can be configured as percentages or shares instead of fixed token counts.

## Best Possible Solution

Settle and land one backward-compatible relative-budget config API across schema, runtime compaction readers, memory preflight/flush planning, generated config metadata, tests, and public docs while preserving current numeric-token semantics.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main only accepts numeric absolute compaction budgets and lacks percent or share-based budget resolution.

## Solution Assessment

Unclear until maintainers choose the public config API shape. Additive share fields look safer than overloading existing numeric keys with percent strings, but the implementation needs to cover all compaction and memory threshold consumers before this issue is resolved.
