## Summary

Keep open. Current main still exposes and consumes the relevant compaction token-budget settings as numeric-only, so percentage strings remain a valid but unimplemented config/runtime/docs feature.

## What This Changes

The issue requests percentage-string support for compaction token settings such as `reserveTokensFloor` and `keepRecentTokens`.

## Best Possible Solution

Extend the existing compaction token-budget config contract with a shared number-or-percent resolver, resolved against the effective model context window, while preserving current numeric semantics.

## Reproduction Assessment

Yes, via source-level reproduction. Current main validates these knobs as numbers and the runtime readers ignore non-number values, so a config like `"10%"` cannot validate or reach context-aware resolution.

## Solution Assessment

Yes. Explicit percent-string support resolved against the effective context window is narrower and safer than auto-scaling existing numeric values, which would silently change current user configs.

Add a shared resolver for non-negative/positive token-budget values that accepts numbers or strings like `"10%"`, resolves percentages against the effective model context window where available, clamps/rounds consistently, and stops if unsupported formats require a product decision. Update public config types, Zod schema, generated schema/help metadata, runtime application in `src/agents/pi-settings.ts`, the memory preflight path in `src/auto-reply/reply/agent-runner-memory.ts`, docs, focused regression tests, and a user-facing changelog entry.
