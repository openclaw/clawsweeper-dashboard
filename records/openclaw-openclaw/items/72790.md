## Summary

Keep open. Current main still exposes, validates, documents, and consumes these compaction token-budget settings as numeric-only values, so percentage strings remain unimplemented and there is no linked implementation PR in the provided discussion.

## What This Changes

Add support for percentage values such as `"10%"` on compaction token-budget settings like `reserveTokensFloor` and `keepRecentTokens`.

## Best Possible Solution

Extend the existing compaction token-budget contract with a shared number-or-percent resolver against the effective model context window, while preserving current numeric semantics and documenting the new accepted value form.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main validates these knobs as numbers and runtime readers ignore non-number values, so a config value like `"10%"` cannot validate or reach context-aware resolution.

## Solution Assessment

Yes. Explicit percent-string support resolved against the effective context window is narrower and safer than auto-scaling existing numeric values, which would silently change current user configs.
