## What This Changes

The PR adds an owner-only Telegram `/providers` DM setup flow, a provider setup runtime with a private SDK subpath, Telegram secret-reply handling, and provider auth/config replay coverage.

## Best Possible Solution

Land this flow only after credential/profile persistence is atomic with the final setup commit, the private SDK boundary is accepted, and artifact-backed Telegram proof shows callbacks, secret handling/deletion, restart persistence, and cancellation behavior.

## Reproduction Assessment

Yes, from source inspection: enter a provider secret, stop at the default-model confirmation, then cancel. `runProviderPluginAuthMethod` has already written auth profiles before the setup runtime reports cancellation.

## Solution Assessment

No. The product direction is plausible, but the implementation should make credential/profile writes atomic with final setup commit, or make post-secret steps non-cancellable, before this is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Defer profile writes until setup commits:** `src/provider-setup/runtime.ts:278-291`
  - body: After `runProviderPluginAuthMethod` returns, provider profiles have already been persisted, but this flow still presents a cancellable default-model prompt before `updateConfig` runs. If the owner pastes a key and then taps Cancel, Telegram reports setup cancelled while the credential/profile can remain on disk without the matching config commit. Defer profile persistence until the final save, or remove the post-secret cancel path so cancellation cannot leave stored credentials behind.
  - confidence: 0.86
