## Summary

Keep open. The Telegram provider setup direction is plausible, but this draft has a blocking abandoned-session bug in the new DM secret flow and the PR body explicitly lacks real Telegram proof.

## What This Changes

The PR adds an owner-only Telegram `/providers` flow backed by a provider setup runtime, exposes that runtime through the plugin SDK, and wires provider auth/profile/config writes into Telegram callbacks and DM text replies.

## Best Possible Solution

Keep the provider setup flow behind maintainer review until active sessions expire safely, sensitive text consumption is covered by regression tests, and redacted live Telegram proof shows the DM setup path working end to end.

## Reproduction Assessment

Yes from source inspection: start setup through a sensitive text step, abandon it, then send a later non-slash DM from the same owner binding; the new runtime still matches the old text-step session and the Telegram handler returns before normal routing. I did not run a live Telegram reproduction.

## Solution Assessment

No as written: the feature should add active-session expiry/cleanup and regression coverage before merge, then prove the resulting behavior in a real Telegram DM with secrets redacted.
