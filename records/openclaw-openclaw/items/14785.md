## Summary

Keep open. Current main has useful partial mitigations such as Tool Search and local-model lean defaults, but the central fixed tool-schema and prompt overhead is still present: message schemas still include broad cross-channel sections, tool definitions still pass descriptions and parameters into session custom tools, the system prompt still emits Tooling text, and cron still carries long always-sent guidance.

## What This Changes

Request to reduce fixed per-session tool schema token overhead by pruning message schemas, deferring or narrowing tool loading, shortening descriptions, and splitting or compressing large tool surfaces.

## Best Possible Solution

Keep this as the canonical tracker and pursue incremental, compatibility-aware core changes: prune high-footprint message schema groups by active action/capability, reduce duplicated Tooling text, shorten long descriptions, avoid empty tool-definition payloads, and design any slim/lazy mode with explicit maintainer approval.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still builds broad message schemas, sends tool descriptions and parameters into session custom tools, and reports tool schema JSON as context cost. I did not run a live agent turn in this read-only review.

## Solution Assessment

No, a single automated repair is not the best way to solve the issue. The maintainable path is an incremental maintainer-sequenced design across prompt text, schema projection, channel compatibility, Tool Search, cron guidance, and plugin/declarative tool behavior.
