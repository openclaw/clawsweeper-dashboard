## Summary

Keep open: current main still has the fixed tool-schema overhead paths, and newer related reports continue to point back to this issue as the canonical tracker rather than replacing it. Tool Search and existing allow/deny policy are useful partial mitigations, but direct tool exposure, broad message schema fields, duplicated Tooling text, and long cron guidance remain on the default path.

## What This Changes

The issue requests reducing fixed per-session tool-schema context overhead through channel-aware pruning, lazy or on-demand loading, shorter descriptions, smaller tool surfaces, and general schema compression.

## Best Possible Solution

Keep this as the canonical tool-schema overhead tracker and sequence incremental core changes for duplicate Tooling text, empty effective tool lists, channel/capability-aware message schema trimming, shorter descriptions, and compatible slim or lazy schema behavior.
