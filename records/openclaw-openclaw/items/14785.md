## Summary

Keep open. Current main still exposes broad tool-schema overhead by default, while the newer Tool Search mitigation is opt-in and experimental rather than a complete replacement for the requested channel-aware, slim, or lazy schema behavior.

## What This Changes

The issue requests reducing fixed per-session tool schema token overhead through channel-aware pruning, lazy loading, moving tutorials out of descriptions, splitting large tools, and general description compression.

## Best Possible Solution

Keep this as the canonical tool-schema overhead tracker and sequence small maintainer-reviewed changes for duplicate Tooling text, empty effective tool lists, channel/capability-aware message schema trimming, shorter descriptions, and compatible slim or lazy schema behavior.

## Reproduction Assessment

Yes for a source-level current-state check: current main still builds broad message schemas, emits full tool definitions, includes Tooling prompt text, and keeps long cron guidance. I did not rerun a live Telegram token-count measurement in this read-only review.

## Solution Assessment

Yes for the umbrella direction, but no as one monolithic automated change. The maintainable path is incremental design across prompt policy, provider schema contracts, channel/plugin compatibility, and lazy or slim schema semantics.
