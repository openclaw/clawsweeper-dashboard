## Summary

Current main still pays the central fixed tool-context cost: the message tool retains broad parameter groups, cross-channel actions remain intentionally available for cron and isolated sessions, and model-facing definitions still carry schemas and descriptions. The merged metadata compaction is useful partial progress, but the requested reductions remain meaningful and this issue is still the appropriate umbrella for maintainer sequencing.

## What This Changes

Reduce baseline context consumed by tool schemas and duplicated tool guidance while preserving compatible tool availability and validation.

## Best Possible Solution

Keep this as the canonical tool-overhead tracker, prioritize measured deduplication and canonical schema compression first, then make explicit compatibility decisions for cross-channel pruning and any lazy discovery protocol.

## Reproduction Assessment

Yes, at high confidence from source and reported measurements. Current main exposes deterministic schema and prompt-report paths, while fresh-session reports show that the fixed cost remains material and increases with registered tools.

## Solution Assessment

Partly: an incremental tracker is the correct approach, and deduplicating or compacting the canonical definitions is the safest first step. Lazy loading, mega-tool splitting, and aggressive channel filtering are not automatically the best fixes because they require explicit provider and cross-channel compatibility design.
