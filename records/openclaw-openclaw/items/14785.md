## Summary

Keep open. Current main has useful mitigations such as send-only message schema scoping, experimental Tool Search, and shorter prompt summaries, but the default/core paths still send broad message and cron schemas, full model-facing descriptions and parameters, and prompt-side Tooling text; the discussion still treats this as the canonical design-sensitive performance tracker.

## What This Changes

The issue requests reducing fixed per-session tool schema/context overhead through channel-aware pruning, lazy or slim tool loading, shorter tool descriptions, and splitting or compressing broad tool schemas.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes for duplicated Tooling text, empty effective tool lists, channel/capability-aware message schema trimming, concise descriptions, and compatible slim or lazy schema modes.

## Reproduction Assessment

Yes for a source-level current-main check: the broad message schema, full model-facing descriptions and parameters, duplicated Tooling prompt text, and long cron guidance remain. I did not run a live Telegram token-count measurement in this read-only review.

## Solution Assessment

Yes for the umbrella direction, but not as a single automated fix. The maintainable path needs incremental design across provider schema contracts, prompt policy, channel/plugin compatibility, cron and isolated-agent behavior, and default versus opt-in schema compaction.
