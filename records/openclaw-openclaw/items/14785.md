## Summary

Keep open. Current main has useful mitigations such as opt-in Tool Search and shorter descriptions, but the default/core paths still send broad message and cron tool schemas, full model-facing tool descriptions and parameters, and Tooling prompt text; the discussion still treats this as a canonical design-sensitive performance tracker.

## What This Changes

Request to reduce fixed per-session tool schema and tool-list token overhead through channel-aware pruning, lazy loading, description trimming, and possible tool splitting.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes for duplicated Tooling text, empty effective tool lists, channel/capability-aware message schema trimming, concise descriptions, and a compatible slim or lazy schema mode.

## Reproduction Assessment

Yes for a source-level current-main check: the broad message schema, full model-facing descriptions and parameters, duplicated Tooling prompt text, and long cron guidance remain. I did not run a live Telegram token-count measurement in this read-only review.

## Solution Assessment

Yes for the umbrella direction, but not as a single automated fix. The maintainable path needs incremental design across provider schema contracts, prompt policy, channel/plugin compatibility, cron and isolated-agent behavior, and default versus opt-in schema compaction.
