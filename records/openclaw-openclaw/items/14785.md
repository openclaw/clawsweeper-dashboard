## Label Justifications

- `P2`: The issue describes a reproducible, broad context and latency cost, but not an urgent outage or broken core workflow.

## Summary

Current main has partial mitigations, including opt-in Tool Search and schema metadata compaction, but default runs can still carry broad message schemas plus separate Tooling prompt text; active discussion also treats this as the canonical, design-sensitive performance tracker.

## What This Changes

Reduce fixed per-session tool-definition context overhead through schema pruning, description compression, or compatible on-demand tool loading.

## Best Possible Solution

Keep this as the canonical tool-schema performance tracker, sequence focused measured changes for default Tool Search eligibility, empty-tool omission, prompt-text deduplication, and compatibility-aware message-schema pruning, and track non-tool bootstrap controls separately.

## Reproduction Assessment

Yes, at high source confidence: current context reporting exposes both schema JSON and Tooling text, and multiple commenters supplied real-run token measurements; no new live run was performed in this read-only review.

## Solution Assessment

The incremental direction is best, but no single proposed mechanism is universally safe; maintainers should choose provider-compatible defaults and split channel pruning, prompt deduplication, and lazy loading into focused changes.
