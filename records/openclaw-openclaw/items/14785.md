## Summary

This remains the canonical open tracker for a measurable fixed context cost that current main only partially reduces; the remaining work is meaningful, actively discussed, and requires maintainer sequencing across prompt, provider, channel, and plugin contracts.

## What This Changes

Reduce per-session token overhead from full tool schemas, broad multi-channel message parameters, duplicated Tooling text, and long always-loaded tool descriptions.

## Best Possible Solution

Keep this issue as the umbrella and land small benchmarked changes: remove avoidable Tooling duplication, omit definitions when the effective list is empty, compact descriptions, extend capability-owned schema pruning, and evaluate lazy discovery separately with explicit provider and cross-channel compatibility proof.

## Reproduction Assessment

Yes, at high confidence from source: current main still sends full effective tool descriptions and schemas and separately renders tool inventory text; multiple comments provide real-environment size and token measurements.

## Solution Assessment

The umbrella goal is correct, but the proposed techniques are not interchangeable; incremental compaction, duplicate removal, and capability-aware pruning are the narrowest maintainable first steps, while lazy loading and mega-tool splitting need separate product and compatibility design.
