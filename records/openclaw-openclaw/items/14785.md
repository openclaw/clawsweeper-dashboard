## Summary

Keep open. Current main still has the fixed model-facing tool-schema overhead paths described in the report, and the discussion continues to treat this as the canonical multi-step performance tracker rather than a completed or superseded item.

## What This Changes

Request to reduce fixed per-session tool-schema token overhead through channel-aware pruning, lazy or slim tool schemas, moving long descriptions out of tool payloads, splitting high-footprint tools, and compressing redundant descriptions.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid unnecessary tool payloads, prune high-footprint message fields by active channel and capability without breaking cross-channel cron or isolated runs, shorten long descriptions where behavior stays covered, and design a compatible slim or lazy schema mode for plugins and declarative tools.

Yes. A live Telegram measurement would refine the savings estimate but is not required to verify the remaining fixed overhead.

## Solution Assessment

No single automated patch is the best way to solve the issue. The requested direction is valid, but the maintainable solution needs maintainer-sequenced incremental work because pruning and lazy loading must preserve provider contracts, plugin tools, cron delivery, isolated runs, and prompt-cache behavior.
