## Summary

Keep open. Current main still sends broad tool schemas by default, including the large message schema, model-facing descriptions and parameters, Tooling prompt text, and long cron guidance; existing Tool Search, Code Mode, tool deny, skills trimming, and localModelLean paths are useful mitigations but not a default replacement for the requested slim, channel-aware, or lazy schema behavior.

## What This Changes

The issue requests reducing fixed per-session tool schema overhead through channel-aware pruning, lazy loading, shorter descriptions, and splitting or slimming high-footprint tools.

## Best Possible Solution

Keep this as the canonical tool-schema overhead tracker and sequence maintainer-reviewed changes for duplicate Tooling text, empty effective tool lists, channel/capability-aware message schema trimming, shorter descriptions, and compatible slim or lazy schema behavior.

## Reproduction Assessment

Yes for a source-level current-state check: current main still builds broad message schemas, emits full model-facing tool definitions, includes Tooling prompt text, and keeps long cron guidance. I did not rerun a live Telegram token-count measurement in this read-only review.

## Solution Assessment

Yes for the umbrella direction, but no as one monolithic automated change. The maintainable path is incremental and compatibility-aware across prompt policy, provider schema contracts, channel/plugin message contracts, and slim or lazy schema semantics.
