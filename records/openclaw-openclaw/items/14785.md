## Summary

Keep open. Current main has useful mitigations such as opt-in Tool Search and shorter descriptions, but the default/core paths still send broad message and cron tool schemas, full model-facing tool descriptions and parameters, and prompt-side Tooling text; the issue discussion still treats this as the canonical design-sensitive performance tracker.

## What This Changes

The issue requests reducing fixed per-session tool-schema token overhead through channel-aware schema pruning, lazy or slim tool loading, shorter descriptions, and possible tool-surface restructuring.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes for duplicated Tooling text, empty effective tool lists, channel/capability-aware message schema trimming, concise descriptions, and a compatible slim or lazy schema mode.

## Solution Assessment

Yes for the umbrella direction, but not as a single automated fix. The maintainable path needs incremental design across provider schema contracts, prompt policy, channel/plugin compatibility, cron and isolated-agent behavior, and default versus opt-in schema compaction.
