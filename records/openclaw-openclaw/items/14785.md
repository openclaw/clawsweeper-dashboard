## Summary

Keep open. Current main still has the central fixed-overhead paths: the message tool builds a broad schema, tool definitions still carry descriptions and parameters into sessions, the system prompt still emits Tooling text, and cron still embeds long always-sent guidance; the discussion also treats this as the canonical tracker with remaining follow-up work.

## What This Changes

Reduce fixed per-session token overhead by pruning, compressing, or lazily loading model-facing tool schemas, especially high-footprint tools such as message, browser, cron, and nodes.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes for Tooling text duplication, empty effective tool lists, channel/capability-aware message schema trimming, shorter long descriptions, and compatible lazy or slim schema behavior for plugins and declarative tools.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction of the remaining implementation state: broad message schemas, full customTools definitions, Tooling prompt text, and long cron guidance are still present.

## Solution Assessment

Unclear as a single implementation. The requested direction is valid, but the maintainable path needs incremental design across prompt policy, provider schema contracts, channel/plugin compatibility, cron behavior, and lazy/slim schema semantics.
