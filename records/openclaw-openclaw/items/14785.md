## Summary

Keep open. Current main and v2026.5.22 include partial mitigations such as send-only message schemas and opt-in Tool Search, but the default direct-tool path still sends model-facing descriptions and parameters, the non-send message schema remains broad, the system prompt still emits Tooling text, and cron still carries long always-sent guidance; the related discussion keeps this as the canonical design-sensitive tracker rather than a completed or superseded item.

## What This Changes

Request to reduce fixed per-session tool-schema token overhead through channel/action-aware pruning, lazy or on-demand tools, moving long descriptions out of always-sent tool payloads, splitting large tools, and general schema-description compression.

## Best Possible Solution

Keep this as the canonical tracker and land incremental, compatibility-aware core changes: prune message schemas by channel/action capability, decide default versus opt-in Tool Search or lazy schema behavior, remove duplicated Tooling text safely, shorten always-sent cron guidance, and preserve existing cron/isolated cross-channel validation paths with focused tests.

Yes, at source level. Current main still has broad non-send message schemas, model-facing tool definitions with descriptions and parameters, Tooling prompt text, and long cron guidance; I did not run a live token-count scenario because this review is read-only.

No for closing or for a single autonomous patch. The current mitigations are useful partials, but the durable fix needs incremental maintainer sequencing around compatibility, defaults, and provider/channel contracts.
