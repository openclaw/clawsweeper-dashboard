## Summary

Keep open. Current main has partial mitigations, including Tool Search, local-model lean mode, and a send-only message schema path, but the central fixed tool-schema/context overhead remains in normal sessions.

## What This Changes

The issue requests reducing fixed per-session tool-schema token overhead through channel-aware pruning, lazy or slim schema loading, description compression, and possible tool splitting.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence measured, compatibility-aware reductions for prompt deduplication, empty-tool-list suppression, message schema pruning, description trimming, and a deliberate slim or lazy schema mode.

## Reproduction Assessment

Yes. Current source shows selected tools still become model-facing definitions with descriptions and parameter schemas, and normal prompts still render Tooling text; I did not run a live token measurement in this read-only review.

## Solution Assessment

Yes for keeping this open as the canonical tracker, no for a one-shot automated fix. The remaining slices cross provider payloads, channel/plugin contracts, prompt policy, and optional config/default behavior, so maintainers should choose the next slice before implementation.
