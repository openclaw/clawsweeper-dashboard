## Summary

Current main has partial mitigations, including send-only message-schema trimming, schema metadata compaction, tool policy filtering, and experimental Tool Search, but ordinary direct-tool runs still pay for broad schemas, duplicated tool-list text, and long descriptions. This remains the canonical, actively supported tracker for several compatibility-sensitive optimization steps rather than an implemented or superseded request.

## What This Changes

Reduce fixed per-session context overhead from full tool schemas, broad cross-channel message parameters, duplicated tool summaries, and long embedded descriptions.

## Best Possible Solution

Keep this issue as the canonical measured roadmap, narrow it to gaps not covered by current Tool Search, and land reversible changes separately: remove safe prompt duplication, omit empty tool payloads, shorten descriptions, add capability-aware message-schema grouping, and extend provider-compatible deferred discovery with focused fresh-install and upgrade proof.

## Reproduction Assessment

Yes at source level: current main still exposes broad direct-tool schemas, long descriptions, and separately measured schema and tool-list prompt costs. Multiple discussion comments also provide real deployment measurements, although the precise current-main token count depends on provider and policy.

## Solution Assessment

No single original proposal is the best complete fix. The maintainable path is incremental compaction and deduplication plus capability-aware pruning and provider-supported deferred discovery, with explicit compatibility coverage for cross-channel, cron, plugin, MCP, and prompt-cache behavior.
