## Summary

Keep open. Current main still has the fixed model-facing tool-schema overhead paths described in the report, while the discussion treats this as the canonical multi-step performance tracker with only partial mitigations landed.

## What This Changes

Reduce fixed per-session tool schema/context overhead by pruning or lazily loading broad tool schemas and moving verbose tool guidance out of always-sent payloads.

## Best Possible Solution

Keep this as the canonical tracker and land incremental, measured reductions that preserve provider tool contracts, channel/plugin behavior, cron and isolated-session compatibility, and prompt-cache stability.

## Reproduction Assessment

Yes. Source inspection on current main shows the message, cron, system-prompt, and tool-definition paths still send broad descriptions and parameters; a live Telegram measurement would refine the estimate but is not needed to verify the remaining fixed overhead.

## Solution Assessment

The requested direction is valid, but no single automated patch is the best solution. The maintainable path is maintainer-sequenced incremental work because pruning and lazy loading must preserve provider contracts, plugin tools, cron delivery, isolated runs, and prompt-cache behavior.
