## Summary

Keep open: the PR fixes the reported explicit provider-qualified collision directionally, but the latest patch still treats any non-empty fallback provider as explicit provider intent, which regresses bare alias resolution from the CLI path.

## What This Changes

The PR guards model alias fallback resolution against cross-provider overrides for explicit provider-qualified model refs, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Carry explicit provider intent or the raw model override into fallback candidate resolution so provider-qualified selections are preserved while bare aliases still resolve through the existing alias-first contract.

## Reproduction Assessment

Yes. Current-main source shows the alias-first fallback path, and https://github.com/openclaw/openclaw/issues/79325 includes live OpenClaw 2026.5.7 output demonstrating the wrong provider selection.

## Solution Assessment

No. The PR is close, but using a non-empty `params.provider` as the explicit-intent signal is not the narrowest correct fix because `agentCommand` default-fills that provider for bare model aliases.
