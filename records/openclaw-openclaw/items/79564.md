## Summary

Keep open: the bug is real and this PR is close, but the guard currently treats any non-empty provider as explicit intent and would block existing slash-form alias behavior that current tests cover.

## What This Changes

The PR adds guarded model-alias fallback resolution, regression tests, and a changelog entry for provider-qualified model refs being redirected by cross-provider aliases.

## Best Possible Solution

Teach fallback resolution to distinguish exact configured provider/model selections from alias inputs, then keep both the issue collision and existing slash-form alias behavior covered by regression tests.

## Reproduction Assessment

Yes. Current-main source shows alias-first fallback candidate construction, and the linked issue supplies live OpenClaw 2026.5.7 output demonstrating the wrong provider selection.

## Solution Assessment

No. The direction is right, but using non-empty params.provider as the intent signal is too broad; the narrower fix should preserve exact configured provider/model refs without disabling slash-form aliases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve slash-form aliases when guarding providers:** `src/agents/model-fallback.ts:597-603`
  - body: The guard treats every non-empty `params.provider` as explicit provider intent. Existing fallback behavior intentionally resolves slash-form aliases before provider/model parsing, for example `provider: "xiaomi"`, `model: "mimo-v2-pro-mit"` resolving to `openai/xiaomi/mimo-v2-pro-mit`; with this guard the providers differ and the alias is skipped. Gate the block on a direct configured provider/model match or carry raw explicit-provider intent instead.
