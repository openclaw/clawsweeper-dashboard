## Summary

Keep open: the PR addresses the explicit provider collision, but it still has a blocking correctness issue because the real bare `--model` alias path supplies a default provider before fallback resolution, so the new guard can skip cross-provider aliases it is meant to preserve.

## What This Changes

The PR adds a guard in `src/agents/model-fallback.ts`, two fallback regression tests, and a changelog entry for the DeepSeek cross-provider alias collision.

## Best Possible Solution

Land a fix that distinguishes provider-qualified user input from defaulted provider values, preserving bare alias resolution while blocking cross-provider aliases only for explicit provider refs.

## Reproduction Assessment

Yes. Current-main source shows the alias-first resolution path, and https://github.com/openclaw/openclaw/issues/79325 includes live 2026.5.7 output where a provider-qualified `opencode-go` selection ran as OpenRouter.

## Solution Assessment

No. The direction is right, but the current patch uses a normalized provider argument as a proxy for explicit user intent; carry raw/provider-qualified intent into fallback resolution or resolve bare aliases before default provider splitting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve bare aliases in the real CLI path:** `src/agents/model-fallback.ts:595-598`
  - body: This guard treats any non-empty `params.provider` as a user-supplied provider, but `agentCommand` parses a bare `--model DeepSeek-V4-Pro` with the current/default provider before calling fallback. In that real path the alias target provider differs from the default provider, so the patch skips the configured alias and runs `<default>/DeepSeek-V4-Pro` instead of the alias target. Carry explicit-provider intent or the raw override into fallback resolution and add command-level coverage.
