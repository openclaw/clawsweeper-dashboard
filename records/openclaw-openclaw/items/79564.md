## Summary

Keep open: the PR targets a real provider/model alias bug and includes useful proof, but the current head still derives explicit provider intent from any non-empty fallback provider, which breaks existing slash-form alias behavior and should not merge as-is.

## What This Changes

The PR guards model fallback alias resolution against cross-provider redirects, adds DeepSeek alias collision regression tests, and adds a changelog entry.

## Best Possible Solution

Carry raw model/provider intent, or an explicit-provider flag, into fallback candidate resolution so explicit provider-qualified refs are protected without disabling real alias inputs.

## Reproduction Assessment

Yes. Current-main source shows alias-first fallback candidate resolution, and the linked issue provides live OpenClaw 2026.5.7 output for the wrong-provider selection.

## Solution Assessment

No. The PR direction is right, but using a non-empty parsed provider as the intent signal is too broad because fallback callers can receive provider values that came from alias parsing or defaults rather than explicit user provider selection.
