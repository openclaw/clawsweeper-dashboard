## Summary

Keep open: the PR fixes the reported explicit-provider alias collision in one path, but the current guard still conflates a populated provider value with user-specified provider intent and can break bare alias selection from the CLI.

## What This Changes

The PR guards model fallback alias resolution so configured aliases cannot redirect an explicitly provider-qualified model ref to a different provider, with regression tests and a changelog entry.

## Best Possible Solution

Carry explicit provider intent or the raw model override into fallback candidate resolution, preserve alias lookup for actual bare alias inputs, and add command-level regression coverage for both paths.

## Reproduction Assessment

Yes. Current-main source plus the linked live issue output give a high-confidence reproduction path: configure both DeepSeek refs, select opencode-go/deepseek-v4-pro, and the alias-first fallback path can select the OpenRouter ref before execution.

## Solution Assessment

No. The PR is aimed at the right bug, but using non-empty params.provider as the intent signal is not the best fix because command parsing can populate that provider for a bare alias the user did not provider-qualify.
