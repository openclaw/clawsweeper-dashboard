## Summary

Keep open: the PR addresses the reported cross-provider alias collision in helper-level fallback tests, but the current patch still regresses providerless alias use in the real `openclaw agent --model ...` path.

## What This Changes

The PR adds guarded model-alias fallback resolution, regression tests, and a changelog entry for provider-qualified model refs being redirected by aliases.

## Best Possible Solution

Land a small fix that distinguishes provider-qualified input from default-filled provider values, preserves bare alias resolution, and adds command-level regression coverage.

## Reproduction Assessment

Yes. Current-main source shows the alias-first resolution path, and https://github.com/openclaw/openclaw/issues/79325 includes live OpenClaw 2026.5.7 output demonstrating the wrong provider selection.

## Solution Assessment

No. The direction is right, but using a non-empty `params.provider` as the intent signal is not the narrowest correct fix because the CLI default-fills that provider for bare aliases.
