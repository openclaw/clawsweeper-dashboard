## Summary

Keep open. Current main defines session lifecycle and reset/compact controls, but the central requested behavior remains a product/API decision because cache-ttl pruning is still an in-memory, tool-result-only prompt optimization, not normal conversation-history expiry or an explicit prune-context method.

## What This Changes

Request explicit session-boundary semantics plus TTL-based mid-session context pruning/reset controls, including proposed prune-context and reset API methods.

## Best Possible Solution

Choose the intended contract, then either document cache-ttl as tool-result-only with /compact, /new, /reset, and sessions.reset as supported controls, or add an authenticated additive prune-context/session-boundary API with Pi/context-engine behavior and tests.

## Reproduction Assessment

Yes, at source level. With contextPruning.mode set to cache-ttl, current main only prunes during Pi context assembly and pruneContextMessages skips non-toolResult messages, so normal user/assistant history older than TTL remains in context.

## Solution Assessment

No. Current main is only a partial solution for the requested direction; the maintainable next step is a product/API decision on whether TTL remains a tool-result cache optimization or becomes a conversation-history/session-boundary mutation surface.
