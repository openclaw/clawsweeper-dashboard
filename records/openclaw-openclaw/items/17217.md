## Summary

Keep open: current main and v2026.5.22 define session lifecycle and reset/compact controls, but cache-TTL pruning is still request-time, in-memory, and tool-result-only, with no explicit prune-context API for normal conversation-history TTL pruning.

## What This Changes

The issue requests explicit session-boundary documentation plus user/API controls to prune context or reset sessions without restarting OpenClaw.

## Best Possible Solution

Define the product contract first, then either document TTL as tool-result-only with existing reset/compact controls or add an authenticated prune-context/session-boundary API with Pi, Gateway, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level for the feature gap. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and only rewrites eligible toolResult messages, so normal user/assistant history older than TTL remains in context.

## Solution Assessment

Unclear. A prune-context endpoint may be viable, but maintainers first need to choose whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation contract.
