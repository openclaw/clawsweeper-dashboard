## Summary

Keep open. Current main covers session lifecycle docs and reset/compact controls, but cache-TTL pruning remains an in-memory prompt-view optimization for tool results only, with no prune-context/session-boundary API for age-expiring normal conversation history.

## What This Changes

The issue requests explicit session boundary documentation, wall-clock TTL pruning for long sessions, and manual session prune/reset controls.

## Best Possible Solution

Define the product contract first, then either document TTL as tool-result-only with /compact, /new, /reset, and sessions.reset as the supported controls, or add an authenticated prune-context/session-boundary API with Pi and Gateway coverage.

## Reproduction Assessment

Yes, at source level. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and pruneContextMessages only rewrites eligible toolResult messages, so user/assistant history older than TTL remains in context.

## Solution Assessment

No, current main is only a partial answer. The maintainable next step is a product/API decision on whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation surface.
