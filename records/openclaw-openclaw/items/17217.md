## Summary

Keep open. Current main and the latest release still define session resets and manual compaction, but cache-TTL pruning remains request-time and tool-result-only, with no explicit prune-context/session-boundary API for wall-clock conversation-history expiry.

## What This Changes

The issue requests explicit session boundary documentation plus wall-clock TTL pruning of conversation context and manual prune/reset controls.

## Best Possible Solution

Define the product contract first, then either document TTL as tool-result-only with existing reset/compact controls or add an authenticated prune-context/session-boundary API with Pi/context-engine, Gateway, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level for the feature gap. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and only rewrites eligible toolResult messages, so normal user/assistant history older than TTL remains in context.

## Solution Assessment

Unclear. A prune-context endpoint may be viable, but maintainers first need to choose whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation contract.
