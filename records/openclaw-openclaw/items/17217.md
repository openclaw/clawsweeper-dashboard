## Summary

Keep open. Current main partially addresses the request by documenting session lifecycle and exposing reset/compact controls, but cache-TTL pruning is still tool-result-only prompt-view pruning and there is no explicit prune-context/session-boundary API; the related open items cover narrower control surfaces rather than fully superseding this issue.

## What This Changes

The issue requests explicit session boundary documentation, wall-clock TTL pruning of active context, and user/API controls to prune or reset session context without restarting OpenClaw.

## Best Possible Solution

Have maintainers choose the contract, then either document TTL as tool-result-only with /compact, /new, /reset, and sessions.reset as supported controls, or add an authenticated prune-context/session-boundary API with Pi and Gateway coverage.

## Reproduction Assessment

Yes, at source level. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and pruneContextMessages only rewrites eligible toolResult messages, so user/assistant history older than TTL remains in context.

## Solution Assessment

No, current main is only a partial answer. The maintainable next step is a product/API decision on whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation surface.
