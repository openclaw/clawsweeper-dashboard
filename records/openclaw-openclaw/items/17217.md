## Summary

Keep open. Current main and v2026.6.6 define sessions and expose `/new`, `/reset`, `sessions.reset`, and `sessions.compact`, but `contextPruning.ttl` is still a cache-touch-gated, in-memory tool-result pruning path rather than wall-clock expiry or summarization of normal conversation history; related open issues cover pieces of the cost problem but do not replace this broader session-boundary/API decision.

## What This Changes

Request to define session lifecycle and TTL semantics, then add explicit mid-session pruning/reset controls to reduce long-session token growth.

## Best Possible Solution

Define the canonical session/context lifecycle: either document cache-TTL as tool-result-only and point users to reset/compact controls, or add an explicit authorized prune/reset or active-pruning surface with schema, docs, and tests.

## Reproduction Assessment

Yes, source-level. Current main shows `cache-ttl` pruning is gated by last cache touch, only rewrites `toolResult` messages, and lacks a prune-context protocol method.

## Solution Assessment

No, not as an unconditional reinterpretation of existing `contextPruning.ttl`. The safer path is an additive, maintainer-approved lifecycle/API or config mode, or a docs clarification that the current TTL only controls tool-result pruning cadence.
