## Summary

Keep open. Current main and v2026.6.6 document session boundaries and provide reset and compaction controls, but cache-TTL pruning remains request-time and tool-result-only; wall-clock conversation-history expiry and an explicit manual prune operation remain unresolved product and security decisions.

## What This Changes

Define explicit session boundaries, expire old conversation context by TTL, and provide manual context-pruning and session-reset controls.

## Best Possible Solution

Define one canonical contract distinguishing ephemeral tool-result pruning, persisted conversation compaction, session reset, and any new retention policy, then implement approved mutations through the context-engine lifecycle and typed gateway session protocol.

## Reproduction Assessment

No, there is not a high-confidence bug reproduction under the documented contract: cache TTL is explicitly a tool-result pruning trigger. Source inspection does confidently reproduce the absence of wall-clock conversation-history expiry and a manual prune operation.

## Solution Assessment

No, the suggested standalone HTTP endpoints are not yet the best solution because they bypass the repository's typed gateway and context-engine ownership model. Maintainers should first approve retention, persistence, summarization, and authorization semantics.
