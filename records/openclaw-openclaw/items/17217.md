## Summary

Keep open. Current main still only partially satisfies the request: session lifecycle and reset/compact controls exist, but cache-TTL pruning remains tool-result-only during context assembly and there is no explicit prune-context/session-boundary API.

## What This Changes

The issue requests explicit session boundaries, immediate TTL-based pruning of aged context, and user/API controls to prune or reset context mid-session.

## Best Possible Solution

Define the product contract first, then either document TTL as tool-result-only with existing reset/compact controls or add an authenticated prune-context/session-boundary API with Pi and Gateway coverage.

## Reproduction Assessment

Yes, at source level for the feature gap. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and only rewrites eligible toolResult messages, so normal user/assistant history older than TTL remains in context.

## Solution Assessment

Unclear. A prune-context endpoint may be viable, but maintainers first need to choose whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation contract.
