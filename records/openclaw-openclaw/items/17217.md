## Summary

Keep open. Current main defines session lifecycle and supports reset/compact controls, but cache-ttl pruning is still tool-result-only during prompt assembly and there is no explicit prune-context gateway/API surface, so the central request remains unresolved.

## What This Changes

Request to define session boundaries, make cache-ttl expire old conversation context during long sessions, and add manual prune/reset controls.

## Best Possible Solution

Make a maintainer product decision on TTL semantics, then either add an additive prune-context/session-context API with Pi/context-engine tests or explicitly document cache-ttl as tool-result-only and direct users to sessions.compact, /new, and /reset.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the current behavior: configure cache-ttl and continue a session, and normal user/assistant history will not be TTL-expired because the pruner only mutates toolResult messages. I did not run a live timed gateway session in this read-only review.

## Solution Assessment

No, the current implementation is not the best complete solution for the request because it only supplies reset/compact controls and tool-result pruning. The maintainable path is an explicit product choice between new conversation-history pruning semantics and clearer docs that TTL is not a session-history expiry mechanism.
