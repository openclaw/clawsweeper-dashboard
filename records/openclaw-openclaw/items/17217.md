## Summary

Keep open. Current main documents session lifecycle and exposes reset/compact controls, but `cache-ttl` remains a tool-result-only prompt-assembly optimization and the prior maintainer discussion already kept the remaining product/API gap open.

## What This Changes

The issue requests explicit session-boundary documentation, TTL-based mid-session context pruning, and API/user controls to prune or reset session context without restarting OpenClaw.

## Best Possible Solution

Make an explicit maintainer product/API decision, then either add an additive authenticated prune-context/session-context surface with Pi/context-engine tests or clarify docs that `cache-ttl` never expires conversation history and point users to `/compact`, `/new`, `/reset`, and `sessions.reset`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `contextPruning.mode: "cache-ttl"` and continue a normal session after TTL; `pruneContextMessages` only mutates `toolResult` messages, so user/assistant history remains unless another mechanism compacts or resets it.

## Solution Assessment

No. The current reset/compact controls and tool-result pruning are useful but do not fully solve the requested TTL conversation-history expiry or explicit prune-context API; the maintainable path starts with a product/API choice.
