## Summary

Keep open. Current main now documents session lifecycle and exposes reset/compact controls, but `contextPruning.ttl` is still a prompt-cache/tool-result pruning mechanism, not wall-clock conversation-history expiry, and there is still no explicit prune-context session mutation surface.

## What This Changes

The issue asks OpenClaw to define session boundaries and add explicit TTL-based context pruning/reset controls for active sessions.

## Best Possible Solution

Make a maintainer product/API decision, then either document the current TTL semantics more explicitly or add an additive authenticated prune-context/session-boundary surface with Pi/context-engine implementation and focused tests.

## Reproduction Assessment

Yes, at source level for the reported gap: configure `contextPruning.mode: "cache-ttl"` and continue after TTL, and current main will only prune eligible `toolResult` entries during context assembly. I did not run a live gateway scenario because this review is read-only and the source path is direct.

## Solution Assessment

No, current main is not the full solution: `/new`, `/reset`, `sessions.reset`, and `/compact` cover useful controls but not TTL-based conversation-history expiry or an explicit prune-context API. The maintainable path starts with a product/API choice because the requested behavior changes the documented pruning contract.
