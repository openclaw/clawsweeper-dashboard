## Summary

Keep open. Current main and v2026.5.12 still treat `contextPruning.ttl` as prompt-view pruning for old tool results, not wall-clock conversation-history expiry, and the requested explicit prune/session-boundary API remains product and security design work.

## What This Changes

Request explicit session-boundary documentation, wall-clock TTL pruning of old conversation context, and user-triggered prune/reset controls without restarting OpenClaw.

## Best Possible Solution

Define the product contract first, then either document TTL as tool-result-only with existing reset/compact controls or add an authenticated prune-context/session-boundary API with Pi and Gateway coverage.

## Reproduction Assessment

Yes, at source level. With `contextPruning.mode: "cache-ttl"`, current main only considers pruning during context assembly and only rewrites eligible `toolResult` messages, so normal user/assistant history older than TTL remains in context.

## Solution Assessment

Unclear. A prune-context endpoint may be viable, but maintainers first need to choose whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation contract.
