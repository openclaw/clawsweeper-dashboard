## Summary

Keep open. Current main and the latest release define session reset/compaction controls, but cache-TTL pruning still runs only during context assembly, only rewrites old tool results, and does not provide a wall-clock conversation-history prune/reset API.

## What This Changes

Request to define session boundaries, make TTL pruning affect conversation history mid-session, and add explicit manual prune/reset controls.

## Best Possible Solution

Define the product contract first, then either document TTL as tool-result-only with existing reset/compact controls or add an authenticated prune-context/session-boundary API with Pi/context-engine, Gateway, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and only rewrites eligible toolResult messages, so normal user/assistant history older than TTL remains in context.

## Solution Assessment

Unclear. A prune-context endpoint may be viable, but maintainers first need to choose whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation contract.
