## Summary

Keep open. Current main and v2026.6.6 define session lifecycle and expose `/new`, `/reset`, `/compact`, `sessions.reset`, and `sessions.compact`, but `cache-ttl` remains request-time, tool-result-only pruning; there is still no manual prune-context operation or wall-clock conversation-history expiry, and adding one is a product and security decision.

## What This Changes

Requested behavior: define OpenClaw session boundaries, make TTL pruning apply during long sessions, and add manual session/context reset or prune controls.

## Best Possible Solution

Define one canonical contract distinguishing ephemeral tool-result pruning, persisted conversation compaction, session reset, and any new retention policy, then implement approved mutations through the typed gateway/session protocol and context-engine lifecycle.

## Reproduction Assessment

Yes, at source level for the missing requested capability: current main has reset and compaction, but no manual prune-context gateway method and no wall-clock conversation-history TTL expiry. This is not a high-confidence bug reproduction under the documented contract because docs define cache-TTL pruning as tool-result-only.

## Solution Assessment

No, the suggested standalone POST endpoints are not yet the best way to solve it because session mutation is already governed by the typed gateway protocol and context-engine/session ownership. Maintainers should approve retention, persistence, summarization, and authorization semantics before implementation.
