## Summary

Keep open. Current main documents session lifecycle and has reset/compaction controls, but cache-TTL pruning is still scoped to in-memory tool-result trimming and does not provide wall-clock expiry of normal conversation history or an explicit prune-context API.

## What This Changes

The issue requests explicit session boundary documentation plus TTL-based mid-session context pruning, manual prune controls, and session reset APIs.

## Best Possible Solution

Make an explicit maintainer product choice: either keep `contextPruning.ttl` as a prompt-cache/tool-result optimization and clarify supported `/compact`, `/new`, `/reset`, and `sessions.reset` controls, or add an additive session pruning/compaction surface that summarizes or rotates normal conversation history with protocol, auth, docs, and tests aligned.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: with `contextPruning.mode: "cache-ttl"`, pruning only runs during context assembly after a cache touch and `pruneContextMessages` skips non-`toolResult` messages. That means old user/assistant conversation remains until reset or compaction under current main.

## Solution Assessment

Unclear. The suggested endpoints are plausible, but existing `sessions.reset`, `sessions.compact`, `/reset`, `/new`, and `/compact` already cover part of the control surface; normal-history TTL pruning should be designed through the session/context-engine/compaction model rather than bolted onto the current tool-result pruner.
