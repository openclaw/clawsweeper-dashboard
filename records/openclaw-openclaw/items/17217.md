## Summary

Keep open. Current main covers part of the request with documented session reset and manual compaction controls, but cache-TTL pruning is still scoped to in-memory tool-result trimming during context assembly and does not provide wall-clock expiry of normal conversation history or an explicit prune-context API.

## What This Changes

The issue requests explicit session boundary documentation, wall-clock TTL pruning of conversation context, and manual API/user controls to prune or reset a session.

## Best Possible Solution

Make an explicit maintainer product choice: either keep `contextPruning.ttl` as a prompt-cache/tool-result optimization and clarify the docs around `/compact`, `/new`, `/reset`, and `sessions.reset`, or add an additive session pruning/compaction surface that summarizes or rotates normal conversation history with protocol, auth, docs, and tests aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `contextPruning.mode: "cache-ttl"`, pruning is only invoked during context assembly after a cache touch and `pruneContextMessages` skips non-`toolResult` messages, so old user/assistant messages remain in model context until reset or compaction.

## Solution Assessment

Unclear. The suggested endpoints are plausible, but existing `sessions.reset`, `sessions.compact`, `/reset`, `/new`, and `/compact` already cover part of the control surface; normal-history TTL pruning should likely be designed through the context-engine/compaction model rather than bolted onto the current tool-result pruner.
