## Label Justifications

- `impact:session-state`: The request changes retention, summarization, reset, and active conversation-context behavior.

## Summary

Keep open. Current main and v2026.6.5 define session lifecycle and expose reset/compaction controls, but cache-TTL pruning remains request-time, in-memory, and tool-result-only; wall-clock conversation-history expiry and an explicit prune operation still need product and security design.

## What This Changes

The issue requests explicit session-boundary semantics, wall-clock TTL pruning of conversation context, and manual prune/reset controls without restarting OpenClaw.

## Best Possible Solution

Define one generic, authorized context-engine/session-maintenance contract for optional conversation retention and summarization, or explicitly keep cache-TTL scoped to tool-result prompt pruning and direct users to `/compact`, `/new`, `/reset`, `sessions.compact`, and `sessions.reset`.

## Reproduction Assessment

Yes at source level: current main only evaluates cache-TTL during context assembly and `pruneContextMessages` only considers eligible `toolResult` messages. No gateway protocol prune method exists.

## Solution Assessment

No, the proposed standalone HTTP endpoints are not yet the best-established shape. Any implementation should extend existing gateway RPC, context-engine, SessionManager, and authorization boundaries rather than create a parallel mutation path.
