## Summary

Keep open. Current main covers part of the request with documented daily, idle, and manual session resets, but `contextPruning.ttl` is still documented and implemented as in-memory old-tool-result pruning during context assembly, not wall-clock expiry of normal conversation history, and there is no explicit prune-context mutation API.

## What This Changes

Request to define session boundaries, make TTL pruning affect old context mid-session, and add explicit manual prune/reset controls.

## Best Possible Solution

End state should either document TTL as tool-result-only with `/compact`, `/new`, `/reset`, and `sessions.reset` as supported controls, or add an additive authenticated prune-context/session-boundary API with Pi/context implementation and regression coverage.

## Reproduction Assessment

Yes, at source level: configure `contextPruning.mode: "cache-ttl"` and continue after TTL; current main only considers pruning during context assembly and only rewrites eligible `toolResult` messages, not normal user/assistant history. I did not run a live session because this review is read-only, but the source path is direct.

## Solution Assessment

No, current main is only a partial solution: reset and compact controls exist, but wall-clock TTL expiry for normal conversation history and a manual prune-context API are absent. The safer next step is a maintainer product/API decision before implementation.
