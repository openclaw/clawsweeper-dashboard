## Summary

Keep open. Current main covers part of the request with documented session lifecycle, `/new`, `/reset`, `sessions.reset`, and `sessions.compact`, but it still does not provide wall-clock TTL expiry for normal conversation history or an explicit prune-context/session-boundary mutation API.

## What This Changes

The issue requests explicit session-boundary documentation, wall-clock TTL pruning of old conversation history, and manual API controls to prune or reset session context.

## Best Possible Solution

Make a maintainer API/product decision, then either document current TTL semantics more explicitly or add an additive authenticated prune-context/session-boundary surface with Pi/context-engine behavior and regression coverage.

## Reproduction Assessment

Yes, at source level: configure `contextPruning.mode` as `cache-ttl` and continue after TTL; current main only prunes eligible `toolResult` messages during context assembly, not normal conversation history on a wall clock.

## Solution Assessment

No, current main is not the full requested solution: reset/compact controls are useful, but TTL-based conversation-history expiry and a manual prune-context API remain absent and need product/API design first.
