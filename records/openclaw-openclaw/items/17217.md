## Summary

Keep open: current main and the latest release define session lifecycle and expose reset/compact controls, but TTL pruning is still documented and implemented as request-time, in-memory tool-result pruning, with no approved manual prune-context surface or conversation-history expiry contract.

## What This Changes

The issue requests explicit session boundaries plus immediate TTL-based context pruning and manual/session API controls for pruning and reset without restarting OpenClaw.

## Best Possible Solution

Define one canonical session/context lifecycle contract distinguishing reset, compaction, tool-result pruning, manual clearing, and any retention/summarization policy, then implement only the maintainer-approved typed gateway/context surfaces with docs and focused tests.

## Reproduction Assessment

Yes, at source level for the missing requested capability: current main has reset and compaction, but no manual prune-context gateway method and no wall-clock conversation-history TTL expiry. This is not a high-confidence bug reproduction under the documented contract because docs define cache-TTL pruning as tool-result-only.

## Solution Assessment

No, not as standalone POST endpoints over new logic. Session mutation already belongs to typed gateway/session protocol and context-engine lifecycle seams, so maintainers should approve retention, persistence, summarization, and authorization semantics before implementation.
