## Summary

Keep open. Current main covers session lifecycle and reset/compact controls, but `cache-ttl` remains documented and implemented as in-memory tool-result pruning, with no explicit prune-context API for conversation-history TTL expiry.

## What This Changes

The issue requests explicit session-boundary docs, TTL-based conversation-history pruning, and manual session prune/reset controls.

## Best Possible Solution

Make a maintainer product/API decision, then either add an authenticated prune-context/session-context surface with Pi/context-engine coverage or clarify that `cache-ttl` never expires normal conversation history and direct users to `/compact`, `/new`, `/reset`, and `sessions.reset`.

## Reproduction Assessment

Yes, by source inspection. Configure `contextPruning.mode: "cache-ttl"` and continue after TTL; the hook only runs during context assembly and the pruner only mutates `toolResult` messages, so user/assistant history remains in context.

## Solution Assessment

Unclear until maintainers choose the product contract. Adding prune-context endpoints may be appropriate, but the requested TTL behavior changes documented semantics and should not be treated as a narrow bug fix.
