## Summary

Keep open. Current main documents session lifecycle and exposes reset/compact controls, but `cache-ttl` is still documented and implemented as in-memory tool-result pruning, while the prior maintainer review already kept the remaining TTL/session API gap open.

## What This Changes

The issue requests explicit session-boundary documentation, wall-clock TTL pruning of old conversation context, and manual session prune/reset controls.

## Best Possible Solution

Make a maintainer product/API decision, then either add an additive authenticated prune-context/session-context surface with Pi/context-engine tests or clarify docs that `cache-ttl` never expires conversation history and point users to `/compact`, `/new`, `/reset`, and `sessions.reset`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `contextPruning.mode: "cache-ttl"` and continue after TTL; the context hook only prunes during context assembly and the pruner only mutates `toolResult` messages, so user/assistant history remains.

## Solution Assessment

No. The current reset/compact controls are useful but do not solve TTL conversation-history expiry or an explicit prune-context API; the maintainable path starts with a product/API decision.
