## Summary

Keep open. Current main and v2026.6.10 still define session lifecycle plus reset/compact controls, but pruning remains request-time, in-memory, tool-result-only behavior and there is no authorized prune-context API or conversation-history TTL contract.

## What This Changes

Issue requests explicit session-boundary documentation, wall-clock TTL pruning of context, and user/API controls for mid-session pruning and reset.

## Best Possible Solution

Define one canonical session/context contract that separates tool-result pruning, conversation compaction, manual reset, TTL or rolling retention, and authorized API controls, then update code and docs consistently.

## Reproduction Assessment

Yes, at source level. Current main has reset and compaction controls, but no prune-context method and no wall-clock conversation-history TTL expiry; cache-TTL pruning is tool-result-only by source and docs.

## Solution Assessment

No, not as standalone POST endpoints yet. The maintainable path is a maintainer-approved typed session/context contract with authorization, persistence, summarization, recovery, and docs before transport adapters are added.
