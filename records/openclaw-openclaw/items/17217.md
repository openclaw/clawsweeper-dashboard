## Summary

Keep open. Current main defines session lifecycle plus reset/compact controls, but cache-TTL pruning is still in-memory tool-result trimming during context assembly, and no explicit prune-context/session-boundary API exists; related open issues cover narrower slices rather than fully superseding this request.

## What This Changes

The issue requests explicit session-boundary documentation and controls, including TTL-based active-session context pruning and manual prune/reset APIs.

## Best Possible Solution

Choose the intended contract, then either document TTL as tool-result-only with /compact, /new, /reset, and sessions.reset as supported controls, or add an authenticated prune-context/session-boundary API with Pi/context implementation and tests.

## Reproduction Assessment

Yes, at source level. With contextPruning.mode set to cache-ttl, current main only considers pruning during context assembly and pruneContextMessages only rewrites eligible toolResult messages, so user/assistant history older than TTL remains in context.

## Solution Assessment

No, current main is only a partial answer to the requested direction. The maintainable next step is a product/API decision on whether TTL remains a tool-result prompt-cache optimization or becomes a conversation-history/session-boundary mutation surface.
