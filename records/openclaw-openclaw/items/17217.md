## Summary

Keep open. Current main and v2026.6.5 define session boundaries and provide reset/compaction controls, but cache-TTL remains request-time, tool-result-only prompt pruning; wall-clock conversation-history expiry and an explicit prune operation remain unimplemented product and security decisions.

## What This Changes

The issue requests explicit session lifecycle documentation, wall-clock TTL expiry of conversation context, and manual prune/reset controls.

## Best Possible Solution

Define one generic, authorized context-engine maintenance contract for conversation retention and summarization if wall-clock expiry is desired; otherwise keep cache-TTL scoped to prompt pruning and clearly direct users to `/compact`, `/new`, `/reset`, `sessions.compact`, and `sessions.reset`.

## Reproduction Assessment

Yes at source level: current main only evaluates cache-TTL when assembling a request and prunes eligible tool results, while normal conversation history is not expired on a wall clock and no explicit prune operation exists.

## Solution Assessment

No, the proposed standalone HTTP endpoints are not yet the best-established shape. Any implementation should extend the existing gateway RPC, context-engine, SessionManager, and authorization boundaries rather than create a parallel session mutation path.
