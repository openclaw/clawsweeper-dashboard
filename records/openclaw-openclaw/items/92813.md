## Label Justifications

- `P1`: The PR targets agent credential storage on mounted filesystems and can affect whether real users can start agents or keep auth tokens private.
- `merge-risk: 🚨 compatibility`: The branch intentionally changes upgrade behavior for existing agent DB locations that cannot prove private modes or ACLs.

- `merge-risk: 🚨 availability`: Fail-closed credential-store policy can stop agents from opening on affected network-backed or chmod-less volumes.

## Summary

Keep open: the branch fixes the POSIX chmod-less secret-store gap, but the Windows chmod-success path still trusts mode bits that cannot prove credential privacy for the per-agent SQLite credential database.

## What This Changes

The branch replaces raw agent DB chmod calls with a private-or-refuse permission gate, applies it before read-only auth-profile SQLite loads, and adds agent DB chmod-less volume regression tests.

## Best Possible Solution

State/security ownership should choose ACL-aware Windows verification or explicit fail-closed Windows behavior, keep the POSIX mode-after-chmod checks, and land only with focused coverage and proof for the chosen policy.

## Reproduction Assessment

Yes, source-reproducible: PR head returns on Windows after successful `chmodSync`, while the Node v22.19 fs contract says Windows mode bits do not implement owner/group/others distinctions. I did not run a native Windows ACL reproduction in this read-only review.

## Solution Assessment

No. The POSIX mode-after-chmod design is a good part of the fix, but the complete solution still needs ACL-aware Windows privacy verification or an explicit fail-closed Windows policy accepted by the state/security owner.
