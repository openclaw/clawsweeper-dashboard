## Label Justifications

- `P1`: The PR targets agent credential storage on mounted filesystems and can affect whether real users can start agents or keep auth tokens private.
- `merge-risk: 🚨 compatibility`: The branch intentionally changes upgrade behavior for existing agent DB locations that cannot prove private modes or ACLs.

- `merge-risk: 🚨 availability`: Fail-closed credential-store policy can stop agents from opening on affected network-backed or chmod-less volumes.

## Summary

Keep open: the PR now covers the POSIX silent-success and read-only auth-profile paths, but it still treats a successful Windows chmod as credential-privacy proof even though Node's Windows mode contract cannot prove owner/group/other isolation.

## What This Changes

The PR adds a private-or-refuse permission gate for the per-agent SQLite credential database, applies it to normal and read-only auth-profile opens, and adds chmod-less volume regression tests.

## Best Possible Solution

State/security ownership should choose ACL-aware Windows verification or an explicit fail-closed Windows policy, keep the POSIX mode-after-chmod checks, and land only with focused coverage/proof for the chosen policy.

## Reproduction Assessment

Yes, source-reproducible: PR head returns on the Windows chmod-success path, while the Node v22.19 filesystem contract says Windows file modes do not implement owner/group/other distinctions. I did not run a native Windows ACL reproduction.

## Solution Assessment

No. The POSIX mode-after-chmod design is a good part of the fix, but the Windows success path is not the best complete solution because it neither verifies ACLs nor fails closed.
