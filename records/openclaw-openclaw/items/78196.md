## Summary

Keep open: current main and latest release v2026.5.28 still require hook capability metadata or explicit hook-policy config before a legacy external hook plugin enters the gateway startup set, and the linked fix path at https://github.com/openclaw/openclaw/pull/78229 is still open/unmerged in the provided GitHub context.

## What This Changes

The issue asks for the gateway daemon to load explicitly enabled external hook plugins from configured plugin paths, invoke their hooks, and report load failures instead of silently skipping them.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78229 with proof that explicitly enabled legacy external hook plugins register in the gateway while merely allowlisted plugins remain ambient-skipped, then close this issue as implemented.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: an external hook-only plugin with `plugins.allow` plus `plugins.entries.<id>.enabled = true`, but no `activation.onCapabilities: ["hook"]` and no `hooks.*` policy, is omitted by the current startup-intent gate. I did not run the Docker gateway flow, but the issue's logs and current source match the failure mode.

## Solution Assessment

Yes, the narrow linked fix direction is the best fit: preserve the ambient allowlist guard while treating explicitly enabled non-bundled plugins as legacy hook startup intent. It should be completed with real gateway proof before the issue is closed.
