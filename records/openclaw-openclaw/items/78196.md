## Summary

Keep open: current main and v2026.5.7 still keep explicitly enabled legacy external hook-only plugins out of the Gateway startup load set unless they declare hook capability metadata or hook policy config, and https://github.com/openclaw/openclaw/pull/78229 is an open linked implementation path.

## What This Changes

The issue asks for the long-running Gateway daemon to load explicitly enabled external legacy hook plugins from configured plugin paths, with visible diagnostics when a plugin is skipped or rejected.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78229 or an equivalent narrow startup-planner fix that starts explicitly enabled non-bundled legacy hook plugins while preserving disable, deny, allowlist, and bundled manifest gates.

## Reproduction Assessment

Yes, source-level. A non-bundled plugin with `plugins.entries.<id>.enabled=true` but no `activation.onCapabilities: ["hook"]` or `plugins.entries.<id>.hooks.*` config is filtered out before it reaches Gateway `onlyPluginIds`, and the reporter supplied matching Docker logs and hook side-effect checks.

## Solution Assessment

Yes, the linked startup-planner direction is the narrow maintainable fix. The issue should stay open until that PR, or an equivalent fix with real gateway proof, merges.
