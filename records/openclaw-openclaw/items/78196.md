## Summary

Keep open: current main and the latest release still filter explicitly enabled legacy external hook plugins unless they declare hook capability metadata or hook-policy config, and https://github.com/openclaw/openclaw/pull/78229 is the open fix path.

## What This Changes

The issue reports that the v2026.5.3+ Gateway daemon skips explicitly configured external hook plugins from `plugins.load.paths` even though CLI plugin commands load them.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78229 or an equivalent narrow startup-planner fix after real Gateway proof confirms explicitly enabled legacy external hook plugins load and fire while disable, deny, allowlist, and bundled manifest gates remain intact.

## Reproduction Assessment

Yes, source-level. Current main rejects a non-bundled plugin with `plugins.entries.<id>.enabled=true` but no hook capability or hook-policy config before it can enter the Gateway startup load set.

## Solution Assessment

Yes, the linked startup-planner direction is the narrow maintainable fix. It should restore legacy external hook startup only for explicitly enabled plugins while preserving the existing safety gates, then be proven in a real Gateway/container setup.
