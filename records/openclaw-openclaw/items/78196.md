## Label Justifications

- `P1`: The issue describes a regression that can silently disable live Gateway security and audit hooks for external-plugin users.
- `impact:security`: The affected `before_tool_call` hook is used for tool-call scanning, blocking, and audit logging, so skipping it weakens an operator security boundary.

## Summary

Keep open: current main and the latest v2026.6.10 release still only startup-load external hook plugins when they declare hook capability metadata or explicit hook policy, while the linked fix PR remains open and unmerged.

## What This Changes

The issue requests that the Gateway daemon load explicitly enabled extension hook plugins from `plugins.load.paths` and report rejects instead of silently skipping them.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/78229 with a startup-planner fix that preserves disable, deny, allowlist, and `activation.onStartup` boundaries, backed by redacted Gateway proof.

## Reproduction Assessment

Yes, source-level: current main excludes enabled-only legacy external hook plugins before Gateway runtime loading unless they declare hook activation metadata or hook policy config. I did not run the Docker reproduction in this read-only review.

Yes for the issue direction: the startup planner is the narrow runtime boundary to repair. The open candidate PR still needs opt-out preservation and real Gateway proof before it is the best merge path.
