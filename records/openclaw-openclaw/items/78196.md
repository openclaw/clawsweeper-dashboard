## Label Justifications

- `P1`: The issue describes a regression that can silently disable live Gateway security and audit hooks for real external-plugin users.

## Summary

Keep open: current main and v2026.6.8 still require hook capability metadata or explicit hooks policy before startup-loading external hook plugins, and the linked fix PR remains open and unmerged.

## What This Changes

The issue reports a v2026.5.3+ regression where the Gateway daemon silently skips explicitly enabled external hook plugins from `plugins.load.paths` even though CLI plugin commands load them.

## Best Possible Solution

Land or replace the linked startup-planner fix so explicitly enabled legacy external hook plugins load in the Gateway while preserving disable, deny, allowlist, and startup opt-out behavior.

## Reproduction Assessment

Yes, source-level: current main excludes enabled-only legacy external hook plugins before Gateway runtime loading, and the issue body includes concrete Docker logs plus hook side-effect checks. I did not run the live container reproduction in this read-only review.

## Solution Assessment

Yes for the direction: the startup planner/load boundary is the right place to restore daemon loading. The linked PR is plausible but not sufficient as-is until it preserves opt-outs and includes real Gateway proof.
