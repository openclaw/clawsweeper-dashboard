## Summary

Keep open: current main and the latest release still filter explicitly enabled legacy external hook plugins unless they declare hook capability metadata or hook-policy config, and https://github.com/openclaw/openclaw/pull/78229 is the open closing fix path.

## What This Changes

The issue reports a regression where the Gateway daemon skips explicitly enabled external hook-only plugins from `plugins.load.paths` while CLI plugin inspection still loads them.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78229 or an equivalent narrow startup-planner fix after real Gateway proof shows explicitly enabled legacy external hook plugins load and fire while disable, deny, allowlist, and bundled manifest gates remain intact.

## Reproduction Assessment

Yes, source-level. Current main rejects a non-bundled plugin with `plugins.entries.<id>.enabled=true` but no hook capability or hook policy before it can enter the Gateway startup load set, matching the reporter's Docker logs and hook side-effect checks.

## Solution Assessment

Yes, the linked startup-planner direction is the narrow maintainable fix. It should restore legacy external hook startup only for explicitly enabled plugins while preserving existing safety gates, then be proven in a real Gateway/container setup.
