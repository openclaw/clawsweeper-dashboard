## Summary

Keep open: current main still filters the reported explicitly enabled legacy external hook-plugin case out of the gateway startup load set, and an open PR with closing syntax already owns the implementation path.

## What This Changes

The issue reports that explicitly enabled external hook plugins load in short-lived CLI commands but are silently skipped by the long-running gateway daemon in v2026.5.3+.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78229 or an equivalent narrow startup-planner fix with real gateway proof, then close this issue as fixed.

## Reproduction Assessment

Yes, source-level: an allowlisted non-bundled plugin with `plugins.entries.<id>.enabled=true` but no hook capability metadata or hook-policy config is filtered out before it reaches the Gateway daemon load set. The reporter also supplied Docker logs and side-effect checks showing CLI load succeeds while daemon hooks do not run.

## Solution Assessment

Yes, the linked startup-planner direction is the narrow maintainable fix because it restores documented legacy external hook compatibility without owner-specific plugin IDs or ambient-loading merely allowlisted plugins.
