## Summary

Keep open: current `main` still has a source-visible gateway startup gap for legacy external hook plugins. The reporter's CLI-vs-daemon symptoms match the code path where the Gateway only loads IDs from the startup planner, while the planner admits hook plugins only when activation metadata or hook-policy config makes that intent explicit.

## What This Changes

The issue reports that custom plugins from `plugins.load.paths` load in CLI plugin commands but are silently skipped by the long-running Gateway daemon starting in v2026.5.3.

## Best Possible Solution

Restore daemon startup compatibility for explicitly configured external hook plugins without ambient-loading unrelated bundled plugins, and add regression coverage that proves Gateway startup registers the legacy hook-only case.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: an allowlisted non-bundled plugin from `plugins.load.paths` with `plugins.entries.<id>.enabled=true` but no activation metadata or hook-policy config is filtered out of `startup.pluginIds`, so Gateway never calls its `register(api)`. The reporter also supplied concrete Docker commands and logs showing the CLI path loads while the daemon path skips.

## Solution Assessment

No, current `main` is not the best solution because it preserves only explicit hook activation metadata and misses the documented external hook compatibility baseline. The narrow fix should be in the startup planner/registry path with tests, not in an owner-specific plugin workaround.

## Repair Work Prompt

Fix the Gateway startup regression tracked by https://github.com/openclaw/openclaw/issues/78196. Custom non-bundled OpenClaw plugins discovered through `plugins.load.paths` and explicitly enabled/allowlisted can be omitted from Gateway startup when they are legacy hook-only plugins with no `activation.onStartup`, no `activation.onCapabilities`, and no hook-policy config, even though CLI runtime inspection loads them. Keep core extension-agnostic. Restore external hook-only compatibility in the startup planner/registry path without ambient-loading unrelated bundled plugins. Add regression coverage for a config-path plugin that registers `before_tool_call` or `message_sending` via `register(api)` and lacks activation metadata: `resolveGatewayStartupPluginIds` should include it and Gateway startup should call `register`; disabled, denied, and unrelated ambient plugins should remain skipped. Update docs only if the public behavior text needs clarification, and add a user-facing changelog entry. Do not add owner-specific plugin IDs or require a new config option.
