## Summary

Keep open: current main and the latest release still preserve the gateway startup gap for explicitly enabled legacy external hook plugins, and an open linked PR with closing syntax is already the implementation path.

## What This Changes

Restore gateway daemon startup loading for explicitly configured external hook plugins from `plugins.load.paths` when CLI plugin inspection already loads them successfully.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78229 or an equivalent narrow startup-planner fix with real gateway proof, then close this issue as fixed.

## Reproduction Assessment

Yes, source-level: an allowlisted non-bundled plugin with `plugins.entries.<id>.enabled=true` but no `activation.onCapabilities` or hook-policy config is filtered out before it reaches Gateway `onlyPluginIds`. The reporter also supplied concrete Docker logs and side-effect checks showing CLI load succeeds while daemon hooks do not run.

## Solution Assessment

Yes, the linked startup-planner direction is the narrow maintainable fix because it restores the documented legacy external hook compatibility without owner-specific plugin IDs. The issue should stay open until that PR, or an equivalent fix, merges with real behavior proof.
