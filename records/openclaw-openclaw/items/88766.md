## Label Justifications

- `P2`: This is a bounded Mattermost/Codex channel behavior gap that affects long tool-heavy turns but does not block final reply delivery.
- `impact:other`: The issue affects in-flight progress visibility for Mattermost operators, which is maintainer-visible but not message loss, session state, auth, security, or availability.

## Summary

Keep open. Current main and the latest release support Mattermost draft progress in general, but Mattermost does not opt into dispatch's source-suppressed progress callback path, so Codex message-tool-only turns can still lose the requested in-flight preview.

## What This Changes

Request Mattermost tool-progress preview streaming during Codex harness turns even when final visible replies are delivered by the message tool instead of automatic source delivery.

## Best Possible Solution

Mattermost should opt into channel-owned progress callbacks when draft progress is active under message-tool-only source delivery, without changing final reply delivery mode.

## Reproduction Assessment

No live Mattermost run was performed. Source inspection gives a high-confidence failing path: Mattermost passes progress callbacks, dispatch suppresses them under message_tool_only unless an explicit allow flag is set, and Mattermost does not set that flag.

## Solution Assessment

Yes, the narrowest maintainable fix is to mirror the existing Discord/Telegram pattern in the Mattermost replyOptions path and add coverage for message-tool-only progress drafts. A broader config or visibleReplies change would risk changing final delivery semantics unnecessarily.
