## Summary

Keep open: current main still has the shared MCP loopback teardown bug, and this PR targets the right area, but it is not merge-ready because the creator-only ownership bit does not model active shared users, a cleanup seam test is left inconsistent, live A2A proof is still missing, and the branch now conflicts with main.

## What This Changes

The branch changes the shared MCP loopback ensure API to report creator ownership, carries that through CLI run preparation, and gates run-end loopback cleanup on that ownership with focused tests.

## Best Possible Solution

Use a lease or refcount for the shared MCP loopback server so every active run releases it and only the last release closes it, then refresh the branch and provide live A2A proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes by source inspection: current main shares a singleton MCP loopback server and cleanup-enabled `runCliAgent` calls close it unconditionally. I did not run a live A2A repro in this read-only review.

## Solution Assessment

No: the creator bit is a plausible mitigation for the reported parent-child ordering, but a lease/refcount is the safer lifecycle model for a shared server with overlapping active users.
