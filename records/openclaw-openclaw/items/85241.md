## Summary

Keep open: the PR targets a real shared MCP loopback teardown bug, but the creator-only ownership bit is still too narrow for overlapping cleanup-enabled runs, and the branch also needs conflict refresh plus stronger real behavior proof or an explicit maintainer override.

## What This Changes

The PR changes `ensureMcpLoopbackServer()` to return ownership metadata, threads it through prepared CLI run context, and gates MCP loopback cleanup on that ownership bit.

## Best Possible Solution

Replace creator-only ownership with an active-user lease or refcount release model so each run releases its own use and only the last release closes the shared loopback server, while preserving unconditional gateway shutdown.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and `v2026.6.9` share one process-wide MCP loopback server, and cleanup-enabled `runCliAgent` callers can close it globally while other runs may still use it. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible partial mitigation for the parent-created child-reuse ordering, but the maintainable fix is active-user lifecycle tracking rather than a single creator ownership bit.
