## Summary

Keep open: the PR targets a real current-main MCP loopback lifecycle bug, but the proposed creator-only cleanup model is still unsafe for overlapping cleanup-enabled runs and the branch is currently conflict-dirty with insufficient live behavior proof.

## What This Changes

The PR changes `ensureMcpLoopbackServer()` to return ownership metadata, threads that through CLI run preparation, and only closes the shared MCP loopback server when the run recorded itself as creator.

## Best Possible Solution

Use a per-run lease or refcount for the shared MCP loopback server so each run releases only its own use and the server closes only after the last active user drains, while preserving unconditional gateway shutdown.

## Reproduction Assessment

Yes, at source level: current main has a global MCP loopback singleton, prepare injects it into CLI runs, cleanup-enabled runs close it unconditionally, and subagent/gateway one-shot callers pass the cleanup flag. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No. Creator ownership is a plausible mitigation for the parent-created child-reuse ordering, but the best fix is active-user lifecycle tracking so any run still using the shared loopback server is protected.
