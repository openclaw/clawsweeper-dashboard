## Label Justifications

- `P1`: The PR targets a regression that can break agent-to-agent bundle MCP tool calls during overlapping parent/child runs.
- `merge-risk: 🚨 availability`: Merging an incorrect loopback lifecycle change could leave Node processes alive or tear down a shared MCP server while another run is still using it.

## Summary

Keep open: this is a plausible fix for the shared MCP teardown regression, but the creator-only ownership bit still leaves overlapping cleanup-true runs able to close a server another run is using, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR changes the MCP loopback singleton API to report whether a caller created the server, threads that ownership through CLI preparation, and gates run-end loopback cleanup on that flag with focused tests.

## Best Possible Solution

Use an active-run lease or refcount for the shared MCP loopback server so parent sessions, child runs, and parallel one-shot runs release it only when no active user remains, then prove the real A2A failure path is fixed.

## Reproduction Assessment

Yes, by source inspection: current main lets a non-session subagent request cleanupBundleMcpOnRunEnd and runCliAgent then closes the shared MCP loopback server unconditionally. I did not run a live A2A repro in this read-only review.

## Solution Assessment

No, not yet: carrying a creation fact is the right direction, but a creator bit is weaker than a lifecycle lease and does not protect cleanup-true runs that overlap each other. The PR also needs after-fix real behavior proof before merge.

Please track active run leases/refcounts or otherwise close only after the last active user releases the loopback server.
