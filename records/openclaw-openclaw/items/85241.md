## Summary

Keep open: the PR targets a real MCP loopback availability regression, but the proposed creator-only ownership bit is still too narrow and the external proof gate is not cleared.

## What This Changes

The PR changes MCP loopback server startup to return ownership metadata, threads that through CLI run preparation, and only closes the shared loopback server when the run recorded itself as the creator.

## Best Possible Solution

Use an active-user lease or refcount for the shared MCP loopback server so each run releases only its own use and the last release closes it, while preserving unconditional gateway shutdown.

## Reproduction Assessment

Yes, at source level: current main has a global MCP loopback singleton and cleanup-enabled CLI runs close it unconditionally, while subagent and one-shot paths pass the cleanup flag. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No. Creator ownership is a plausible mitigation for the parent-created child-reuse case, but the best fix is active-user lifecycle tracking so any run still using the shared loopback server is protected.
