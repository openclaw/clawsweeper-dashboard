## Summary

Keep open: current main and v2026.6.9 still have the shared MCP loopback cleanup bug, but this PR's creator-only ownership model is too narrow and can still close the singleton while another cleanup-enabled run is active.

## What This Changes

The PR changes MCP loopback server ownership tracking so only the run that created the shared MCP loopback server closes it during CLI run cleanup.

## Best Possible Solution

Use active-user lease or refcount semantics for the shared MCP loopback server so each run releases its own use and only the last release closes it, while preserving unconditional gateway shutdown.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.9 share one loopback server and cleanup-enabled `runCliAgent` closes it unconditionally, while subagent and one-shot paths pass that cleanup flag. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No. Creator ownership is a plausible mitigation for the parent-created child-reuse case, but the best fix is active-user lifecycle tracking so any run still using the shared loopback server is protected.
