## Summary

Keep open: current main and v2026.6.8 still have the unconditional shared MCP loopback close path, so this PR targets a real agent-to-agent availability bug, but the branch is not merge-ready because creator-only ownership is too narrow, cleanup seam coverage needs repair, live A2A proof or a maintainer proof override is still missing, and the PR is currently conflicting.

## What This Changes

The PR changes MCP loopback server ownership tracking so only the run that created the shared loopback server closes it during CLI run cleanup.

## Best Possible Solution

Use a lease or refcount-style lifecycle for the shared MCP loopback server so each active run releases its use and only the final release closes it, while preserving standalone one-shot cleanup and unconditional gateway shutdown.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.8 share one loopback server and cleanup-enabled runCliAgent closes it unconditionally, while non-session subagent runs pass that cleanup flag. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No. Creator ownership is a plausible mitigation for the parent-created child-reuse case, but the best fix is active-user lifecycle tracking so any run still using the shared loopback server is protected.
