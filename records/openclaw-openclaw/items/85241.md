## Summary

Keep open: current main still has the reported shared MCP loopback premature-close path, but this PR is not merge-ready because the creator-only flag leaves another active-user teardown ordering unprotected and the external real-behavior proof is still insufficient without a maintainer override.

## What This Changes

The PR changes the shared MCP loopback ensure API to return creator ownership metadata, threads that into CLI run preparation, and gates run-end loopback cleanup on that ownership with focused tests.

## Best Possible Solution

Use a lease or refcount-style lifecycle for the shared MCP loopback server so each active run releases its use and only the final release closes it, while preserving standalone one-shot cleanup and gateway shutdown behavior.

## Reproduction Assessment

Yes by source inspection: current main shares one loopback server and cleanup-enabled `runCliAgent()` closes it unconditionally. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No: creator ownership is a plausible mitigation for the parent-created child-reuse case, but it is not the best lifecycle model for a shared server because it does not protect every active user.
