## Summary

Keep this PR open: it targets a real source-reproducible MCP loopback lifecycle bug, but the branch still lacks after-fix real A2A proof and the creator-bit cleanup model leaves overlapping cleanup-enabled runs unsafe.

## What This Changes

The PR changes the shared MCP loopback server API to return creation ownership, threads that fact through CLI run preparation, and gates run-end cleanup on that ownership flag with focused tests.

## Best Possible Solution

Use an active-run lease or refcount for the shared MCP loopback server so every user releases it and only the last release closes it, then prove or explicitly override the real A2A parent/child behavior before merge.

## Reproduction Assessment

Yes, by source inspection: current main reuses a singleton MCP loopback server and `runCliAgent` closes it unconditionally for cleanup-enabled runs. I did not run a live A2A repro in this read-only review.

## Solution Assessment

No, not yet: the creator bit fixes the reported parent-session ordering, but a lease/refcount is the narrower durable lifecycle model for shared active users, and the existing cleanup seam test also needs to cover owned cleanup under the new guard.
