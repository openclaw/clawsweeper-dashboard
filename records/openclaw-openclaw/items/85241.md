## Summary

Keep open: the PR targets a real current-main and latest-release availability bug in shared MCP loopback cleanup, but the creator-only ownership model is not merge-ready because it still permits an active shared server to be closed in another overlap ordering, the existing owned-cleanup seam test is not updated for the new context flag, the branch is currently conflicting, and external real gateway A2A proof or a maintainer proof override is still missing.

## What This Changes

The PR changes MCP loopback ensure to return creator metadata, threads it through CLI preparation, gates run-end loopback closure on that flag, and adds loopback lifecycle tests.

## Best Possible Solution

Use a lease or refcount-style lifecycle for the shared MCP loopback server so each active run releases its use and only the final release closes it, while preserving standalone one-shot cleanup and unconditional gateway shutdown.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.8 share one loopback server and cleanup-enabled runCliAgent closes it unconditionally, while non-session subagent runs pass that cleanup flag. I did not run a live gateway A2A reproduction in this read-only review.

## Solution Assessment

No. Creator ownership is a plausible mitigation for the parent-created child-reuse case, but the best fix is an active-user lifecycle so any run that still uses the shared loopback server is protected.
