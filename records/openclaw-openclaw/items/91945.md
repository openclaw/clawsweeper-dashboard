## Summary

Keep open: current main and the latest release still use the legacy Cloudflare gateway-domain Anthropic route and provider-id keyed `cf-aig-authorization`, while Cloudflare's current docs make the REST direction credible. The remaining work is not a strict bug fix because changing the default API family and credential meaning needs maintainer product direction and live Cloudflare validation.

## What This Changes

The issue requests updating the built-in Cloudflare AI Gateway provider to default to Cloudflare's REST `/ai/v1` API while preserving legacy gateway-domain behavior.

## Best Possible Solution

Add REST `/ai/v1` support in the Cloudflare provider plugin after maintainers choose the supported API families and credential model, preserve existing gateway-domain configs, update docs/onboarding/tests, and require live Cloudflare proof.

## Reproduction Assessment

Not applicable: this is a provider-default modernization request, not a broken existing-behavior bug. Source inspection confirms current main still uses the legacy gateway-domain Anthropic route and id-keyed gateway auth behavior.

## Solution Assessment

No, not as written: switching the default specifically to `openai-completions` is not clearly the best fix because Cloudflare REST also offers Anthropic Messages, which matches OpenClaw's current default. The best path needs a maintainer decision on REST API families, credential semantics, and upgrade behavior before implementation.

## Risks / Open Questions

- Changing the default from the existing Anthropic gateway route to one REST API family can silently alter configured model IDs, auth headers, and expected upstream billing semantics for new setups.
- REST Unified Billing uses a Cloudflare account API token, while current OpenClaw onboarding labels `CLOUDFLARE_AI_GATEWAY_API_KEY` as the provider API key for requests through the gateway; that credential meaning needs an explicit migration plan.
- The recent user comment about `@cf/moonshotai/kimi-k2.6` overlaps the still-open Workers AI model-selection issue, so maintainers should decide whether REST migration and Workers AI onboarding are one design or separate follow-ups.
