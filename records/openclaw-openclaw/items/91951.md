## Summary

Keep open: current main still has no memory-search retention/window config, and the builtin sync path treats every discovered memory file as active until the source file disappears. This is useful but it adds a new config/product surface, so it needs maintainer direction rather than conservative cleanup closure.

## What This Changes

The issue requests a configurable memory-index retention window that skips or prunes dated memory files older than a configured age while preserving evergreen memory paths.

## Best Possible Solution

If maintainers approve the knob, add one canonical optional retention policy to memory-core config, schema/help, docs, and builtin sync so full reindex prunes aged dated memory files while preserving evergreen files and extraPaths by explicit contract.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current config cannot express a retention window, and the sync path indexes all discovered memory files until they are absent. I did not run a live large-index repro because this review is read-only.

## Solution Assessment

Unclear as a final product shape: the requested maxAgeDays-style knob is plausible, but maintainers should first decide the retention semantics and backend scope. Current temporal decay only changes ranking; it does not bound indexed rows.
