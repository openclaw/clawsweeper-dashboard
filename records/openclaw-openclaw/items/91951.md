## Summary

Keep open: current main and v2026.6.8 still lack a memory-search age-retention window, and adding one would create a new public config and pruning contract that needs maintainer product review before implementation.

## What This Changes

The issue requests an opt-in `memorySearch` retention setting that only indexes recent daily memory files, prunes older indexed chunks on reindex, and exempts evergreen memory files.

## Best Possible Solution

Define a memory-core owned retention contract, then update config types/schema/docs and the canonical sync/reindex pruning path so aged-out daily notes are excluded and deleted atomically while the unset default keeps current behavior.

## Reproduction Assessment

Yes, at source level: current main indexes every discovered memory file and only deletes rows whose path disappears from the active set. I did not recreate the reporter's exact 118 MB database, but the unbounded indexing path is clear from source.

## Solution Assessment

Unclear as proposed: an opt-in retention window is plausible, but maintainers still need to choose the public config name, age source, pruning scope, and interaction with existing temporal-decay and QMD retention settings.
