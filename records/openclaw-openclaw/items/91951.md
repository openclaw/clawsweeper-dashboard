## Label Justifications

- `P3`: This is an opt-in storage and performance enhancement rather than a regression against an existing documented retention contract.
- `impact:other`: The issue reports sustained local SQLite growth and potential memory-search slowdown for long-running agents.

## Summary

Current main still indexes every discovered memory file and removes rows only when a file is no longer in the active path set, so bounded indexing remains missing. The request fits the memory plugin and performance direction, but the new configuration and retention semantics need owner judgment.

## What This Changes

Add an optional age-based retention window for the builtin SQLite memory index that prunes old dated daily-memory chunks while preserving evergreen memory sources.

## Best Possible Solution

Define a plugin-owned retention contract for dated daily notes, reuse established date and evergreen classification where valid, and make canonical sync/reindex atomically exclude and delete aged-out rows while leaving behavior unchanged when unset.

## Reproduction Assessment

Yes at source level: current main indexes every discovered memory path and only removes paths that disappear, so the absence of age retention is directly traceable. The reporter's exact database size and growth rate were not independently reproduced.

## Solution Assessment

Unclear as proposed: retention belongs in the existing sync/reindex lifecycle, but maintainers must first settle the public config name, age calculation, exemption rules, and composition with temporal decay.
