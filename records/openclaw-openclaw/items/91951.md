## Summary

Keep open: current main and v2026.6.6 still lack a memory-search retention window, and the remaining work is a memory-area product/config decision rather than a stale or duplicate report.

## What This Changes

The issue requests a memory index retention setting that limits dated memory files by age while preserving evergreen memory files and extra paths.

## Best Possible Solution

Define a plugin-owned retention contract for dated memory notes, then implement the cutoff in the canonical sync/reindex path so aged-out rows are pruned while unset configs keep current behavior.

## Reproduction Assessment

Yes. Source inspection shows current main indexes every discovered memory path and only removes rows when a path disappears from the active set; the exact database size growth was not independently recreated.

## Solution Assessment

Unclear as proposed. Retention belongs in the existing sync/reindex lifecycle, but maintainers need to choose the config name, age calculation, exemption semantics, and relationship to temporal decay before implementation.
