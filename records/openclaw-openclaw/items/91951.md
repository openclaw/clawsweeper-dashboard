## Summary

Keep open: current main and v2026.6.8 still lack a daily memory-file age-retention window, and current indexing only prunes rows when source paths disappear. The request is useful, but it adds public config and pruning semantics that need maintainer product review before implementation.

## What This Changes

The issue requests a `maxAgeDays` or `retentionDays` setting to limit the indexed window for daily memory files while exempting evergreen memory files.

## Best Possible Solution

Define one documented memory-core retention contract, then implement it in the canonical discovery/reindex cleanup path with schema, help text, docs, and regression coverage while preserving the unset default as current behavior.

## Reproduction Assessment

Yes, at source level: current main indexes all discovered memory files and only deletes rows for paths no longer active. I did not recreate the reporter's exact 118 MB database growth measurement.

## Solution Assessment

Unclear as proposed: a retention knob is plausible and belongs near the existing sync/reindex lifecycle, but maintainers still need to choose the public config name, age source, exemption rules, QMD parity, and cleanup semantics.
