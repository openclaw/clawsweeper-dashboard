## Summary

Current main still lacks an age-retention setting or pruning path for durable memory-file index rows, so this feature request remains necessary. The request is product- and compatibility-sensitive because it would add public config and delete indexed memory chunks, so it should stay open for maintainer direction rather than being queued as an automatic repair.

## What This Changes

Request a `maxAgeDays` or `retentionDays` memory-search configuration that limits indexing and reindex pruning to a recent daily-memory window while exempting evergreen memory files and `extraPaths`.

## Best Possible Solution

Define a maintainer-approved retention contract for durable memory indexes, then implement it through the memory config schema, docs, sync pruning path, and regression coverage for dated files, evergreen files, and `extraPaths`.

## Reproduction Assessment

Yes for the source-level gap: current main has no memory-search retention config and the sync path only prunes missing files, not old daily files. I did not recreate the reporter's 80-day database locally because the request is for a new bounded-retention capability rather than a failing existing contract.

## Solution Assessment

Unclear until maintainers choose the product contract. A retention config may be the right direction, but the best fix needs an explicit decision on config location, age basis, default-off behavior, `extraPaths` exemption, and pruning semantics before code is written.
