## Summary

Keep open: current main and the latest release still lack a memory-search age-retention window, and the source path still indexes every discovered memory file while pruning only paths that disappear. The request is useful, but it adds a public config contract and pruning semantics that need memory-area product review before an implementation is queued.

## What This Changes

Requests an opt-in memory-search retention window that skips indexing and prunes old dated memory files while preserving evergreen memory content and extra-path behavior.

## Best Possible Solution

Define a plugin-owned retention contract for builtin memory-core, then update config/schema/docs and the canonical sync/reindex pruning path so aged-out daily notes are atomically excluded and removed while the unset default preserves current behavior.

## Reproduction Assessment

Yes, at source level: current main indexes every discovered memory path and only deletes rows whose path disappears from that active set. I did not independently recreate the reporter's exact 118 MB database growth measurement.

## Solution Assessment

Unclear as proposed: the retention direction is plausible and belongs in the existing sync/reindex lifecycle, but maintainers still need to choose the public config name, age source, exemption policy, and interaction with temporal decay.
