## Summary

Keep open. The PR targets a real, source-reproducible Anthropic transport bug, but the proposed `> 0` guard still allows positive `budget_tokens` values below Anthropic's documented minimum and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR changes the legacy Anthropic thinking payload builder to use a nullish budget fallback and omit `thinking` when the resolved budget is zero.

## Best Possible Solution

Tighten the legacy Anthropic thinking branch to emit `budget_tokens` only when the resolved budget is valid for Anthropic, omit thinking when no valid budget can be formed, and cover zero/default/sub-minimum cases with targeted transport tests plus redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection. Current main can resolve `thinkingBudgetTokens` to zero or a positive sub-minimum value, then the legacy payload builder either coerces zero to 1024 or would accept the PR's sub-minimum value.

## Solution Assessment

No, not yet. The nullish fallback is the right direction for explicit zero, but the guard should enforce Anthropic's 1,024-token minimum or omit legacy thinking below that threshold, with regression tests and real behavior proof.
