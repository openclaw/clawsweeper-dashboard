## Summary

Keep open. The underlying current-main bug is real, but this PR remains incomplete because the proposed `> 0` guard still emits positive manual thinking budgets below Anthropic's documented 1,024-token minimum, and the legacy budget path has no regression coverage. ([docs.anthropic.com](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking))

## What This Changes

The PR changes Anthropic legacy thinking payload construction to use a nullish fallback for `thinkingBudgetTokens` and skip the `thinking` block when the resolved budget is zero.

## Best Possible Solution

Land a narrow Anthropic transport fix that emits legacy manual thinking only when the resolved budget is valid for Anthropic, preserves adaptive-thinking behavior, keeps the default fallback only when valid, and covers zero, sub-minimum, default, and valid-budget cases in colocated tests.

## Reproduction Assessment

Yes. A static reproduction is clear from current code: for a non-adaptive reasoning model with `model.maxTokens` around 1500 and `reasoning: "high"`, `adjustMaxTokensForThinking` can compute `thinkingBudgetTokens: 476`, and the PR would still send that invalid manual budget.

## Solution Assessment

No. The PR direction is useful for preserving explicit zero, but the `budgetTokens > 0` guard is not the best fix because Anthropic requires a minimum manual budget of 1,024 tokens; the safer approach is a validity guard/helper plus focused tests. ([docs.anthropic.com](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking))
