## Summary

Keep open. The PR addresses a real current-main Anthropic transport bug, but its `budgetTokens > 0` guard is still too weak because it permits positive budgets below Anthropic's documented 1,024-token minimum; it also has no focused regression coverage for the legacy manual-thinking path.

## What This Changes

The PR changes Anthropic legacy thinking request construction to use a nullish `thinkingBudgetTokens` fallback and omit the `thinking` block when the resolved budget is zero.

## Best Possible Solution

Land a narrow Anthropic transport fix that preserves adaptive-thinking behavior, emits legacy `thinking: { type: "enabled", budget_tokens }` only when the resolved manual budget is valid for Anthropic, keeps the undefined-budget 1024 fallback only when valid, and covers zero, sub-minimum, default, and valid-budget paths in colocated tests.

## Reproduction Assessment

Yes. A static reproduction is clear from current code: use a non-adaptive reasoning model with a constrained token envelope, for example `model.maxTokens` around 1500 and `reasoning: "high"`, and `adjustMaxTokensForThinking` can produce a positive budget below 1024 that the PR would still send.

## Solution Assessment

No. The PR direction is right for preserving explicit zero, but `budgetTokens > 0` is not the narrowest maintainable fix because Anthropic requires manual thinking budgets to meet the documented minimum; the safer fix is a validity helper or guard with focused tests.
