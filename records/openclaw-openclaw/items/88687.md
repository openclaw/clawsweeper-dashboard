## Summary

Keep open for maintainer review: current main and v2026.6.1 still write embedded gap-fill rows with CLI child metadata, while this PR matches the existing delivery-mirror replay contract and has strong writer-to-replay proof. The remaining blocker is maintainer acceptance of the persisted raw transcript metadata change and the linked issue scope.

## What This Changes

The PR changes embedded assistant gap-fill transcript persistence so display-only assistant rows are written as `openai-responses` / `openclaw` / `delivery-mirror` and adds a regression assertion.

## Best Possible Solution

Land the narrow writer-side delivery-mirror tagging if maintainers accept the raw transcript metadata compatibility change, and keep the linked issue and companion hardening PR open for the remaining defense-in-depth decision.

## Reproduction Assessment

Yes at source level: current main writes gap-fill rows with `api: "cli"` and child provider/model metadata, while replay filtering only drops `openclaw` delivery mirrors. I did not run a live Anthropic restart replay.

## Solution Assessment

Yes: tagging the row correctly at write time is the narrowest primary fix because the replay and transcript mirror code already define `openclaw/delivery-mirror` as transcript-only. The assistant-turn merge path remains useful defense-in-depth, not a replacement for this writer fix.
