## Summary

Keep open. Current main still lacks the requested outbound metadata seam, so this PR is not obsolete, but the branch is not merge-ready: GitHub reports it as dirty, the metadata still is not forwarded into the channel handler, Hermes routing policy is hardcoded into core delivery paths, and the diff includes an auth-profile credential bootstrap change that needs to be split or explicitly reviewed.

## What This Changes

The PR adds Hermes arbiter metadata plumbing through sendMessage, gateway send, ChannelOutboundContext, delivery queue persistence/recovery, task/media/exec direct notifications, Swift gateway models, and tests, plus an external CLI auth-profile bootstrap change.

## Best Possible Solution

Keep the PR open and ask for a focused rebase/split. The maintainable end state is a generic opaque outbound metadata seam through gateway send, sendMessage, direct delivery, ChannelOutboundContext, queue persistence, and recovery, with Hermes-specific construction and routing owned by config or a plugin-facing layer rather than hardcoded core defaults; the auth-profile bootstrap change should be handled separately.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by rejecting metadata on gateway send validation and by exposing no metadata field through sendMessage, the delivery queue, or ChannelOutboundContext; the PR-specific adapter drop is visible on the PR head because createChannelHandler is still called without metadata.

## Solution Assessment

No. The proposed direction is useful, but this patch is not the best way as-is because it misses the final handler forwarding step, hardcodes Hermes routing in core notification paths, and mixes a security-sensitive auth change into a protocol/delivery PR; the safer fix is a narrower generic metadata seam plus plugin/config-owned Hermes metadata construction.
