## Summary

Keep open. Current main still lacks the outbound metadata path this PR is trying to add, but the PR branch is not merge-ready because the metadata is not forwarded into the final channel handler, Hermes routing values are hardcoded in core delivery paths, a new test leaks global spies, and the branch mixes in auth-profile credential behavior that belongs in a separate security review or should be rebased away.

## What This Changes

The PR branch adds Hermes arbiter metadata types and threads metadata through sendMessage, gateway send, direct delivery, channel outbound context, delivery queue persistence/recovery, task/media/exec-approval notifications, Swift gateway models, and tests, while also changing external CLI auth bootstrap code.

## Best Possible Solution

Keep the PR open and ask for a focused rebase/split: land a generic opaque outbound metadata seam through sendMessage, gateway send, direct delivery, ChannelOutboundContext, queue persistence, and recovery with adapter-level tests proving plugins receive it. Keep Hermes metadata construction and topic/bot selection configurable or plugin-owned unless maintainers explicitly want core-owned Hermes defaults, and remove the unrelated auth changes from this branch.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by rejecting metadata on gateway send validation, exposing no metadata field in sendMessage or ChannelOutboundContext, and persisting no metadata in delivery queue entries; the PR-specific drop is reproducible with a channel adapter assertion because createChannelHandler is still called without metadata.

## Solution Assessment

No. The proposed direction is useful, but this patch is not the best solution as-is because it misses the final handler forwarding step, hardcodes deployment-specific Hermes routing in core, leaks test globals, and mixes in unrelated auth behavior; a narrower generic metadata seam plus plugin/config-owned Hermes construction is safer.
