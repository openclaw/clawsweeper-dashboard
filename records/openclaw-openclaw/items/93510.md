The PR broadens the shared silent-reply exact matcher to treat token-only replies with terminal punctuation or wrapping quotes as silent and adds unit tests for those cases.

## Best Possible Solution

Land the shared matcher change after redacted real behavior proof shows the literal token no longer reaches a real delivery path and the substantive-reply boundary from the prior regression remains intact.

## Reproduction Assessment

Yes, source-reproducible: current `main` builds the exact silent-token regex with only whitespace around tokens, so `NO_REPLY.` and quoted token-only replies fail the silent check. The shared normalizer then depends on that check before delivery suppression.

## Solution Assessment

Yes, the shared exact silent-token matcher is the narrow owner-boundary for this token-only punctuation and quote leak. Broader mixed-content `NO_REPLY` suppression is a separate policy-sensitive problem and should stay out of this focused PR.
