## Summary

Keep open for maintainer review. The patch is focused, has sufficient Telegram proof, and I found no blocking correctness defect, but it intentionally changes a shared exported silent-reply helper used across plugin SDK and multiple delivery paths.

## What This Changes

The PR broadens `isSilentReplyText` to suppress token-only silent replies wrapped in quotes or followed by terminal punctuation, with unit coverage for those cases.

## Best Possible Solution

Land the focused shared matcher change if maintainers accept that quoted and punctuated token-only sentinels are globally non-deliverable through the exported helper.

## Reproduction Assessment

Yes. Current main and `v2026.6.9` use a whitespace-only exact matcher, and inspected Mantis baseline proof shows `NO_REPLY.` delivered in Telegram while the candidate proof shows the prompt no longer produces a new token reply.

## Solution Assessment

Yes for implementation shape: the central exact silent-token matcher is the narrow owner boundary, while per-channel fixes would duplicate policy. The remaining question is maintainer acceptance of the exported helper contract change.
