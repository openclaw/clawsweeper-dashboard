Full review comments:

- **[P1] Do not consume unprocessed Discord drops:** `extensions/discord/src/monitor/message-run-queue.ts:107-110`
  - body: This commits replay keys when the queue cap is exceeded, and the stale-job path does the same before the job reaches `processDiscordMessage`. Current skipped-job cleanup releases replay claims for accepted but not-started work, so committing here can make an unprocessed Discord delivery look consumed and suppress retry.
  - confidence: 0.92
- **[P1] Preserve skipped Discord cleanup:** `extensions/discord/src/monitor/message-run-queue.ts:138-140`
  - body: `deactivate()` now clears pending jobs and deactivates the run queue without running the current cleanup that releases replay claims and cleans reply typing feedback for accepted but not-started jobs. On shutdown or abort, those jobs can leave stale replay or typing state that affects later delivery behavior.
  - confidence: 0.9
- **[P1] Do not add a default Codex RPC timeout:** `extensions/codex/src/app-server/client.ts:541-543`
  - body: When `timeoutMs` is omitted, this helper now gives every app-server RPC a 60s deadline. Current main only times out calls that explicitly pass a timeout, and the checked sibling Codex app-server source does not define a universal 60s JSON-RPC deadline, so legitimate long-running operations can fail without caller opt-in.
  - confidence: 0.9
- **[P2] Keep transcript idempotency in owned storage:** `src/config/sessions/transcript.ts:449-452`
  - body: This writes a runtime `${transcriptPath}.idempotency.json` sidecar next to the transcript. Current main routes keyed transcript dedupe through the shared transcript append path, and OpenClaw-owned runtime state/cache needs clear storage ownership, cleanup, and rebuild semantics before adding another file format.
  - confidence: 0.86
- **[P2] Regenerate Discord config metadata:** `src/config/zod-schema.providers-core.ts:714-715`
  - body: The PR accepts `maxPendingPerSession` and `maxQueuedAgeMs` in the runtime schema, but bundled generated channel config metadata still only exposes `inboundWorker.runTimeoutMs`. Packaged config/help surfaces can drift from runtime unless the generated metadata is updated or the new config surface is removed.
  - confidence: 0.82
