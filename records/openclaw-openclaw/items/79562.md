## Summary

Keep open: current main does not implement the central Discord queue caps, but this PR is not merge-ready because it can send queued Codex RPCs after timeout, the new Discord drop controls are undocumented and need maintainer approval, and the supplied proof is only CI/unit output.

## What This Changes

This PR adds Discord inbound queue caps and stale-job drops, coalesces Discord preview/media work, caches transcript idempotency checks, serializes Codex app-server writes with RPC timeouts, and adds targeted async session-store reads.

## Best Possible Solution

Land a rebased branch that cancels timed-out queued Codex writes, documents and approves the Discord drop controls, and includes redacted runtime proof for the affected Discord/Codex paths.

## Reproduction Assessment

No high-confidence live reproduction is present for the original bottleneck claims. Source inspection gives a deterministic path for the PR-introduced queued RPC timeout bug: a prior backpressured write can hold writeChain past the request timeout, then the queued write still sends later.

## Solution Assessment

No. The direction is plausible, but this is not the best mergeable form until timed-out queued writes are canceled and the new Discord drop controls are documented and approved.
