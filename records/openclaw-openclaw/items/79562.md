- **[P1] Do not consume unprocessed Discord drops:** `extensions/discord/src/monitor/message-run-queue.ts:107-110`
  - body: This branch commits replay keys when a queue cap is exceeded, and the stale-job path below repeats the same pattern, before the job ever enters `processDiscordMessage`. Current skipped-job cleanup releases replay claims for accepted but not-started work, so committing here can make an unprocessed Discord delivery look consumed and suppress a retry instead of preserving delivery semantics.
  - confidence: 0.92
- **[P1] Preserve skipped Discord cleanup:** `extensions/discord/src/monitor/message-run-queue.ts:138-140`
  - body: `deactivate()` now clears `pendingBySession` and deactivates the run queue without running the current skipped-job cleanup that releases replay claims and cleans reply typing feedback for accepted but not-started jobs. On shutdown or abort, those jobs can leave stale replay/typing state and block later delivery behavior.
  - confidence: 0.9
- **[P1] Do not add a default Codex RPC timeout:** `extensions/codex/src/app-server/client.ts:541-543`
  - body: When `timeoutMs` is omitted, this helper now gives every app-server RPC a 60s deadline. Current main only times out calls that explicitly pass a timeout, and the checked Codex app-server transport/tests do not define a universal 60s JSON-RPC deadline, so legitimate long-running app-server operations can start failing without a caller opting in.
  - confidence: 0.9
- **[P2] Keep transcript idempotency in owned storage:** `src/config/sessions/transcript.ts:449-452`
  - body: This writes a runtime `${transcriptPath}.idempotency.json` sidecar next to the transcript. OpenClaw-owned runtime state/cache is expected to live in owned SQLite/session storage or be rebuildable from the canonical transcript, so this new sidecar lacks clear cleanup, migration, and ownership semantics.
  - confidence: 0.86
- **[P2] Regenerate Discord channel config metadata:** `src/config/zod-schema.providers-core.ts:714-715`
  - body: The PR accepts `maxPendingPerSession` and `maxQueuedAgeMs` in the schema, but the generated bundled channel config metadata still only exposes `inboundWorker.runTimeoutMs`. Packaged config/help surfaces can drift from the runtime schema unless the generated metadata is updated or the new config surface is removed.
  - confidence: 0.82
