## Summary

Keep open. The PR targets a real reset/queue bug and is close, but the current head can still wait behind an active run when the reply-run registry is active before the embedded session lane has depth; it also needs the required changelog entry and a small test-name cleanup.

## What This Changes

The PR adds resetTriggered-aware queue policy and call-site handling for `/new` and `/reset`, suppresses steer/followup paths for reset turns, and adds queue-policy tests.

## Best Possible Solution

Land an updated fix that carries reset-triggered interrupt intent through both active-run policy and prepared queue handling, independent of session-lane depth, while preserving heartbeat drops and normal non-reset queue behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a same-session active ReplyOperation with `messages.queue.mode: steer`, `resetTriggered: true`, and `getQueueSize(sessionLaneKey) === 0`; this PR returns `run-now` but can still wait because prepared queue aborting sees the original queue mode.

## Solution Assessment

No. The direction is right, but the patch should pass reset-triggered interrupt intent into `resolvePreparedReplyQueueState` or compute an effective interrupt queue mode before both abort paths run.
