## Summary

Keep open. The PR targets a valid reset/queue bug and moves the policy in the right direction, but the diff still leaves reset-triggered turns waiting when the active run is present only in the reply-run registry and the session lane depth is zero.

## What This Changes

The PR adds reset-triggered handling to active-run queue policy, suppresses steer/followup flags for reset turns, extends lane clearing for reset turns with lane depth, passes reset context into the agent runner, and adds queue-policy tests.

## Best Possible Solution

Update this PR or land a narrow equivalent so reset-triggered turns use interrupt semantics across both active-run policy and prepared queue waiting, independent of lane depth. Preserve heartbeat drops and non-reset queue behavior, add integration coverage for an active ReplyOperation with no session-lane depth, and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active same-session ReplyOperation with zero command-lane depth, queue mode steer, and a reset-triggered turn; the PR makes policy return run-now but still passes steer into the prepared queue helper, which waits instead of aborting.

## Solution Assessment

No. The direction is right, but the current patch does not carry reset as an aborting condition into resolvePreparedReplyQueueState or convert the prepared queue path to an effective interrupt mode before waiting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Abort reset turns without lane depth:** `src/auto-reply/reply/get-reply-run.ts:822-826`
  - body: This reset abort path still requires `laneSize > 0`. A reply operation is registered before `runEmbeddedPiAgent` enqueues the session lane, so `isActive` can be true while this lane depth is zero. In steer/followup/collect modes the PR then calls `resolvePreparedReplyQueueState` with the original queue mode, whose abort path only runs for `interrupt`, so the reset can still wait behind the active run.
