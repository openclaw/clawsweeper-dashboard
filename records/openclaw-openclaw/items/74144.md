## Summary

Keep this PR open and do not merge it as-is. The reported reset queue bug is valid and the patch moves in the right direction, but the proposed abort path is still lane-depth gated and can leave reset-triggered turns waiting behind an active run; the PR also lacks the required user-facing changelog entry.

## What This Changes

The PR adds reset-triggered handling to active-run queue policy, prepared-reply, and agent-runner paths, plus queue-policy tests, so `/new` and `/reset` are intended to bypass steer/followup-style active-run queues.

## Best Possible Solution

Land an updated version that carries reset-triggered interrupt intent through both active-run policy and prepared queue handling, independent of session-lane depth, while preserving heartbeat drops and non-reset queue behavior. Add focused regression coverage for the zero-lane-depth active-run case and a single-line `CHANGELOG.md` Fixes entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active same-session `ReplyOperation` with `messages.queue.mode: steer`, `resetTriggered: true`, and `getQueueSize(sessionLaneKey) === 0`; this PR returns `run-now` but can still wait because prepared queue aborting only sees the original queue mode.

## Solution Assessment

No. The direction is right, but the patch should pass reset-triggered interrupt intent into `resolvePreparedReplyQueueState` or compute an effective interrupt queue mode before both abort paths run, rather than relying on lane depth.
