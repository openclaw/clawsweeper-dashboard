## Summary

Keep open. The reported queue bug is real and this PR moves the active-run policy in the right direction, but the current patch can still wait instead of interrupting when a reset turn sees an active reply operation before session-lane depth is visible, and it is missing the required changelog entry.

## What This Changes

The PR adds `resetTriggered` handling to active-run queue policy, prepared-reply steering/followup decisions, agent-runner queue policy calls, and queue-policy tests for `/new` and `/reset` turns.

## Best Possible Solution

Land an updated patch that carries reset-triggered interrupt intent through both active-run policy and prepared queue abort/wait handling, with focused regression coverage and a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is an active same-session `ReplyOperation` with `messages.queue.mode: steer` or `collect`, `resetTriggered: true`, and `getQueueSize(sessionLaneKey) === 0`; the PR returns `run-now` but can still wait because prepared queue handling sees the original queue mode.

## Solution Assessment

No. The direction is right, but the narrow maintainable fix is to pass reset-triggered interrupt intent into `resolvePreparedReplyQueueState` or compute an effective interrupt queue mode before both abort paths run.
