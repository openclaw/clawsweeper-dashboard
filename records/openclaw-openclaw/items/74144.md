## Summary

Keep open. The underlying reset/queue bug is valid and current main still lacks reset awareness in active-run queue policy; this PR moves in the right direction, but its abort semantics remain incomplete because reset-triggered turns can still wait behind an active reply operation when the session command lane has no depth.

## What This Changes

The PR adds reset-triggered handling to active-run queue policy and runner paths so `/new` and `/reset` should bypass steer/followup-style queues, plus queue-policy regression tests.

## Best Possible Solution

Land an updated version that gives reset-triggered `/new` and `/reset` turns interrupt semantics across active-run policy and prepared queue handling, independent of command-lane depth, while preserving heartbeat drops and non-reset queue behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is an active same-session `ReplyOperation` with no session-lane depth, `messages.queue.mode: steer`, and a reset-triggered turn; this PR returns `run-now` but can still wait because the prepared queue helper only aborts for `interrupt`.

## Solution Assessment

No. The proposed direction is right, but the implementation should pass reset-triggered interrupt intent into `resolvePreparedReplyQueueState` or compute an effective interrupt queue mode before both abort paths run.
