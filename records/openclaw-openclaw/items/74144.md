## Summary

Keep open. The underlying report is valid and this PR moves in the right direction, but the patch still leaves a reset-triggered turn able to wait behind an active reply operation when the session lane depth is zero, and it is missing the required user-facing changelog entry.

## What This Changes

The PR adds a `resetTriggered` override to active-run queue policy and call sites, suppresses steer/followup handling for reset turns in prepared replies, expands an interrupt cleanup branch, and adds queue-policy tests.

## Best Possible Solution

Land an updated version that carries reset-triggered interrupt intent through both active-run policy and prepared queue handling, independent of session-lane depth, while preserving heartbeat drops and normal non-reset queue behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a same-session active `ReplyOperation` with `messages.queue.mode: steer`, `resetTriggered: true`, and `getQueueSize(sessionLaneKey) === 0`; the PR returns `run-now` but can still wait because prepared queue aborting sees the original queue mode.

## Solution Assessment

No. The direction is right, but the patch should pass reset-triggered interrupt intent into `resolvePreparedReplyQueueState` or compute an effective interrupt queue mode before both abort paths run, rather than relying on lane depth.
