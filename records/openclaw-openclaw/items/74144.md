## Summary

Keep open. The reported queue/reset bug is valid and current main still lacks reset-triggered queue-policy context, but this PR branch still has a reset interrupt gap when an active reply operation exists before the session lane has depth, and it is missing the required changelog entry.

## What This Changes

The PR adds reset-triggered handling to active-run queue policy and runner call sites, suppresses steer/followup handling for reset turns, extends an early abort path, and adds queue-policy tests for reset commands under active queue modes.

## Best Possible Solution

Land an updated version that treats reset-triggered /new and /reset turns as effective interrupt requests through both active-run policy and prepared queue handling, independent of lane depth, while preserving heartbeat drops and existing non-reset queue behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active same-session ReplyOperation with no session-lane depth, messages.queue.mode set to steer, and a reset-triggered turn; this PR can return run-now but then wait because the prepared queue helper only aborts for interrupt mode.

## Solution Assessment

No. The direction is right, but the implementation should pass reset-triggered interrupt intent into resolvePreparedReplyQueueState or compute an effective interrupt queue mode before both abort/wait paths run.
