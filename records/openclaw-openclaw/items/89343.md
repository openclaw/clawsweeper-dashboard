## Summary

Keep open: the PR is aimed at the right security boundary, but the implementation gates on model capability metadata instead of the resolved per-request thinking state, so explicit disabled-thinking requests can still emit reasoning blocks. The PR also lacks real after-fix provider proof.

## What This Changes

The PR adds reasoning-enabled checks to the direct OpenAI completions parser and the agent transport stream parser so reasoning deltas are not appended when model reasoning is disabled.

## Best Possible Solution

Use one resolved per-request thinking-enabled state for both parsers, suppress every thinking-output path when that state is disabled, preserve strict reasoning-tag filtering, and require focused regressions plus real affected-provider proof before merge.

## Reproduction Assessment

Yes from source inspection: current request builders can disable thinking while `model.reasoning` remains true, and the proposed parser gate checks only `model.reasoning`. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No. The proposed gate is a plausible location but not the best fix because it needs resolved per-request thinking state shared across direct and transport output parsing.
