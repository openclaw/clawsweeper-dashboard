## Summary

Keep open: this PR is a useful security-boundary direction, but it is not merge-ready because the new gate uses model capability metadata rather than the effective request thinking state and misses structured-content reasoning paths; the real behavior proof is still synthetic.

## What This Changes

The PR adds a `model.reasoning`-based guard in `src/agents/openai-transport-stream.ts` so non-text OpenAI-compatible completions reasoning deltas are skipped when the model is marked as not reasoning-capable.

## Best Possible Solution

Thread one effective reasoning-output decision from the resolved request/thinking level into the completions stream parser, apply it to every hidden-reasoning route while preserving visible text and strict partitioning, then add focused tests plus redacted real affected-provider proof.

## Reproduction Assessment

Yes, from source inspection: current `processOpenAICompletionsStream` appends provider `reasoning_content` as thinking output and the PR's new guard still cannot see the effective request thinking-off state. I did not run a live affected-provider reproduction in this read-only review.

## Solution Assessment

No; the current patch is only a plausible partial fix because it gates on catalog metadata and only one output loop. The best fix is a single effective request-state gate shared by all hidden-reasoning output paths with regression tests and real gateway/provider proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate reasoning output on the effective request state:** `src/agents/openai-transport-stream.ts:2916`
  - body: `model.reasoning` is catalog capability metadata, not the resolved request state. For example, the OpenRouter thinking-off wrapper removes `payload.reasoning` when `thinkingLevel === "off"`, but the model can still be `reasoning: true`; if that provider still emits `reasoning_content`, this guard lets it through and the reported leak remains. Please thread the effective request/thinking state into the stream parser and gate hidden reasoning on that instead.
  - confidence: 0.9
- **[P1] Apply the gate before structured thinking is routed:** `src/agents/openai-transport-stream.ts:2923`
  - body: The new guard is placed after `choiceDelta.content` has already routed typed reasoning content through `appendRoutedContentDelta`, and `getCompletionsContentDeltas` treats `type: "reasoning"` or `type: "thinking"` as hidden thinking. That means structured provider reasoning can still become a thinking block even when the new flag is false; the same effective-output gate needs to cover content-derived thinking too.
  - confidence: 0.86
