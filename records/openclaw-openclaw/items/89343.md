## Summary

Keep open: the PR is aimed at a real reasoning-field leak, but the latest patch still gates on raw `options.reasoningEffort`, which can suppress legitimate default-enabled reasoning output, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR gates OpenAI-compatible streamed reasoning deltas when reasoning is disabled in the direct OpenAI-completions provider and the transport stream parser.

## Best Possible Solution

Carry the resolved reasoning-enabled/disabled state from payload construction into both stream parsers, preserve strict partitioning and post-tool buffering, add focused regression tests, and require redacted after-fix provider output before merge.

## Reproduction Assessment

Yes from source inspection: current main appends hidden reasoning fields without a disabled-reasoning guard, and the PR patch would suppress default-enabled reasoning because it checks raw options instead of the resolved effort. I did not run a live affected provider in this read-only review.

## Solution Assessment

No; parser gating is the right layer, but this implementation is too narrow because it does not use the resolved reasoning state that actually drives the provider request.
