## Summary

Keep open: the security hardening request is still useful, but this PR is not mergeable as-is because it patches a stale collector location, overlaps the auth-warning scope of another open PR, and still lacks external contributor real behavior proof.

## What This Changes

The branch adds startup dangerous-config warnings for explicit `agents.defaults.sandbox.mode=off`, `gateway.auth.mode=none`, and empty trusted-proxy `allowUsers` in `src/security/dangerous-config-flags.ts`.

## Best Possible Solution

Rebase a focused sandbox-off startup warning into `src/security/core-dangerous-config-flags.ts` with regression coverage, and handle auth warnings through the related auth PR or a combined follow-up that treats omitted and empty `allowUsers` consistently.

## Reproduction Assessment

Yes, by source inspection. Current startup logs the dangerous-flag collector output, current main lacks the sandbox-off startup flag in the active core collector, and trusted-proxy allowUsers semantics are traceable in runtime and audit code.

## Solution Assessment

No for the current branch shape. The sandbox warning should move into the current core collector with tests, while the auth-warning scope should either be dropped here or coordinated with the related auth PR and fixed for omitted `allowUsers`.
