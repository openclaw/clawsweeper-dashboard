## Summary

Keep open: current main still lacks the requested sandbox-off startup warning, and the PR is useful but not mergeable as-is because the added trusted-proxy warning misses the omitted `allowUsers` case that runtime and audit already treat as allow-all.

## What This Changes

The PR adds startup dangerous-config warnings for explicit `agents.defaults.sandbox.mode="off"`, `gateway.auth.mode="none"`, and `gateway.auth.trustedProxy.allowUsers=[]`.

## Best Possible Solution

Add the explicit sandbox-off warning in the current core dangerous-flag helper with focused tests, and either align the trusted-proxy warning with `allowUsers ?? []` semantics or split the auth-warning expansion into its own reviewed change.

## Reproduction Assessment

Yes. By source inspection, a config with `gateway.auth.mode="trusted-proxy"`, a configured `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` skips the PR warning while runtime and audit treat the omitted value as allow-all.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the patch should be rebased onto the current helper split and the trusted-proxy warning must match runtime/audit semantics or be removed from this PR.
