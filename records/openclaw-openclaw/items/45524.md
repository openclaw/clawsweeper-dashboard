## Summary

Keep open. The requested sandbox-off startup warning is still missing from current main, but the PR is not merge-ready because it is based on the older helper layout and its added trusted-proxy warning misses the omitted `allowUsers` case that runtime, configure, and audit all treat as allow-all.

## What This Changes

The PR updates `src/security/dangerous-config-flags.ts` to add startup dangerous-flag strings for explicit sandbox-off defaults, unauthenticated gateway mode, and trusted-proxy `allowUsers` set to an empty array.

## Best Possible Solution

Rebase onto current main, add the explicit sandbox-off startup warning in `collectCoreInsecureOrDangerousFlags`, add focused tests for absent sandbox config and explicit `mode="off"`, and either remove the auth-warning expansion or normalize trusted-proxy allowlist detection to `allowUsers ?? []` so startup warnings match runtime and audit semantics.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="trusted-proxy"`, a configured `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` reproduces the PR bug by static inspection: the PR condition skips the warning, while runtime accepts all proxy-authenticated users.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the current PR is not the best solution until it is rebased and the trusted-proxy warning either matches runtime/audit behavior or is split out of this sandbox-focused change.
