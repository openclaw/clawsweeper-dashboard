- `P1`: The PR targets an active Telegram session-state bug and currently has merge-blocking config reload regressions affecting real channel workflows.

## Summary

Keep open: the PR is still a useful candidate for the Telegram hot-reload bug, but it is not merge-ready because the new reload map removes the safe Telegram fallback and marks authorization/policy keys as no-op even though the running bot keeps those values in startup-captured state.

## What This Changes

The PR narrows Telegram reload rules, adds wildcard-aware matching, and expands gateway reload-plan tests so selected runtime-only Telegram config edits do not restart the channel.

## Best Possible Solution

Keep the targeted runtime-only no-op behavior, but restore a safe Telegram restart fallback and only no-op keys that the running bot actually rereads from fresh config, with focused tests for omitted keys and authorization-policy edits.

## Reproduction Assessment

Yes, at source level: current main's broad `channels.telegram` reload prefix routes runtime-only Telegram edits to channel restart, while the PR head shows omitted keys and startup-captured policy paths remain unsafe. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

No, not yet. The plugin-owned reload metadata and gateway matcher are the right boundary, but the best fix must preserve fallback behavior and avoid no-op classification for keys the running bot does not reread from fresh config.

  - body: `dmPolicy`, `allowFrom`, and related account policy paths are now classified as no-op, but the running Telegram bot captures those values during startup and passes them into message processing.   - confidence: 0.9
