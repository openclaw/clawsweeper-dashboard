## Summary

Keep open: the direction is useful and the supplied terminal proof covers the root-key reload-planner path, but the patch is not merge-ready because the catch-all Telegram no-op rule suppresses restarts for lifecycle/security-sensitive config while the account prefix still restarts account-scoped runtime policy edits.

## What This Changes

The PR changes Telegram reload metadata so selected transport-affecting Telegram config paths restart the channel while other `channels.telegram.*` paths are treated as hot no-ops, with a gateway reload-plan regression test.

## Best Possible Solution

Classify only proven dynamic Telegram policy/read paths as no-op, keep lifecycle, credential, webhook, network, and security-sensitive account changes restart-backed, and add root plus account-scoped reload-plan coverage.

## Reproduction Assessment

Yes. Current main maps `channels.telegram.*` through `plan.restartChannels`, and the PR's own before/after terminal probe shows the `historyLimit` restart before the patch and no-op after the patch.

## Solution Assessment

No. The proposed direction is plausible for root runtime policy keys, but the current prefix shape is too broad for lifecycle/security root keys and too coarse for account-scoped runtime policy keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep lifecycle keys out of the catch-all no-op:** `extensions/telegram/src/shared.ts:199`
  - body: `noopPrefixes: ["channels.telegram"]` makes every unlisted Telegram path a dynamic no-op. But `channels.telegram.enabled` is used to decide whether the account should run, and webhook mode/host/port/path/cert are consumed when the channel starts. After this change those edits will not populate `plan.restartChannels`, so a running bot can stay enabled or keep the old polling/webhook listener until a manual restart. Please keep lifecycle-affecting Telegram keys restart-backed or make the no-op list explicit to proven runtime-only paths.
  - confidence: 0.92
- **[P2] Do not restart account-scoped runtime policy edits:** `extensions/telegram/src/shared.ts:197`
  - body: The new `channels.telegram.accounts` restart prefix matches every per-account field before the broad noop rule. Telegram account configs can carry runtime policy like `dmPolicy` and `historyLimit`, so `channels.telegram.accounts.work.dmPolicy` still restarts the channel and loses the same in-memory state this PR is meant to protect. Please add account-scoped coverage and a rule shape that restarts identity/lifecycle account fields without restarting runtime policy edits.
  - confidence: 0.87
