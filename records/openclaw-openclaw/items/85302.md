## What This Changes

The PR changes `scripts/crabbox-wrapper.mjs` to parse providers from `crabbox run --help` and reject selected providers not advertised by that output.

## Best Possible Solution

Keep the dynamic canonical provider parsing, but validate against the binary's accepted provider names and aliases or otherwise preserve alias pass-through while still rejecting truly unsupported providers.

## Reproduction Assessment

Yes, from source inspection: the PR accepts only names parsed from `providerHelpAll()` while Crabbox registers aliases such as `docker` for `local-container`. I did not run a live Crabbox binary in this checkout because no binary was available.

## Solution Assessment

No. Parsing canonical providers from help is a maintainable direction, but using that list as the complete acceptance set misses Crabbox's alias contract and creates a compatibility regression.
