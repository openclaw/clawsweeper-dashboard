Full review comments:

- **[P2] Require timeout evidence before hard-timing provider errors:** `src/agents/agent-run-terminal-outcome.ts:86`
  - body: This treats any provider-started `status: "error"` as `hard_timeout`, even without a hard `timeoutPhase`. `providerStarted` only proves dispatch began, so auth, quota, or model errors after dispatch will be projected as timeouts in task and session lifecycle paths instead of failures.
  - confidence: 0.93
- **[P2] Keep SDK wait provider errors failed without timeout attribution:** `packages/sdk/src/client.ts:64-68`
  - body: `statusAlreadyTimeoutAttributed` includes `status === "error"`, so an SDK wait payload with `{status: "error", providerStarted: true}` returns `timed_out` before falling through to the existing failed path. Non-timeout provider errors should remain failed unless the payload carries hard timeout evidence.
  - confidence: 0.9
- **[P2] Keep normalized lifecycle errors failed without timeout attribution:** `packages/sdk/src/normalize.ts:89-90`
  - body: For `phase === "error"`, this passes `true` into `hasHardTimeoutMetadata`, which makes `providerStarted: true` sufficient to emit `run.timed_out`. That changes ordinary provider error events into timeout events; only hard timeout metadata should override `run.failed`.
  - confidence: 0.9
