This PR should stay open because the shared direction is good, but the patch currently misclassifies provider-started non-timeout errors as timeouts across terminal projections. The protected maintainer label also rules out cleanup closure, and the remaining blocker is a narrow correctness repair.

## What This Changes

The PR routes task, session lifecycle, SDK wait, and normalized SDK event terminal projections through shared timeout/cancel precedence and adds regression coverage for provider-attributed timeouts.

## Best Possible Solution

Keep the shared terminal outcome consolidation, but require explicit timeout attribution, such as timeout status or a hard timeoutPhase, before mapping provider-started errors to timed_out.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Require timeout evidence before timing out provider errors:** `src/agents/agent-run-terminal-outcome.ts:86`
  - body: `providerStarted` only means the provider request began, so this branch turns ordinary provider-side failures without `timeoutPhase`, such as auth, quota, or model errors, into `hard_timeout`. That misprojects task and session terminal state as timed out instead of failed; require actual timeout status or a hard timeout phase before taking this path.
  - confidence: 0.9
- **[P2] Keep SDK wait errors failed without timeout attribution:** `packages/sdk/src/client.ts:65-69`
  - body: The SDK wait mapper makes `status === "error"` count as already timeout-attributed, so `providerStarted: true` alone returns `timed_out`. Existing SDK callers would see non-timeout provider failures as timeouts; keep status:error failed unless the payload carries timeout status or a hard timeout phase.
  - confidence: 0.86
- **[P2] Do not time out every provider-started lifecycle error:** `packages/sdk/src/normalize.ts:88`
  - body: For lifecycle `phase: "error"`, this path calls the timeout metadata check with attribution forced true, so any provider-started lifecycle error becomes `run.timed_out` even without a timeout phase. That should remain `run.failed` unless the event carries explicit timeout attribution.
  - confidence: 0.86
