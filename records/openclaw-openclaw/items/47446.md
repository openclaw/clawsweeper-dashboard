## Summary

Keep open: the proxy fix is useful in direction, but the current patch still has blocking implementation and security issues, and the PR body does not include after-fix real Discord proxy proof.

## What This Changes

The PR adds Discord gateway and REST env-proxy fallback plus disables proxy keep-alive on selected Undici proxy agents.

## Best Possible Solution

Rework this as a current-main Discord proxy fix that uses public SDK proxy helpers, applies target-aware NO_PROXY checks for Discord API and Gateway hosts, configures Undici proxy tunnel policy through `proxyTls`, drops unrelated node-command edits, and includes redacted real local-proxy proof.

## Reproduction Assessment

Yes for the patch defects: the PR diff, current extension boundary rules, current proxy helpers, and Undici 8.2.0 source/types reproduce the blockers by inspection. The original live Discord ECONNRESET report is still not reproduced because the manual local-proxy check is unchecked.

## Solution Assessment

No. The direction is valid, but this implementation is not the best fix because it crosses the plugin boundary, ignores NO_PROXY when turning env state into explicit Discord agents, and uses `connect` instead of `proxyTls` for proxy tunnel policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the SDK proxy helper instead of core src imports:** `extensions/discord/src/monitor/gateway-plugin.ts:7`
  - body: This relative import reaches from the Discord plugin into core `src/**`, which violates the extension boundary and can break packaged plugin loading. Use the public `openclaw/plugin-sdk/fetch-runtime` subpath for these helpers.
  - confidence: 0.95
- **[P1] Honor NO_PROXY before creating Discord env proxy agents:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: `resolveEnvHttpProxyUrl("https")` only selects a proxy URL; it does not check whether `discord.com` or `gateway.discord.gg` is excluded by `NO_PROXY`. With `HTTPS_PROXY` set and Discord bypassed, auth-bearing traffic is still forced through the proxy.
  - confidence: 0.92
- **[P2] Set env proxy tunnel policy through proxyTls:** `src/infra/net/undici-global-dispatcher.ts:96`
  - body: Undici applies proxy-connection transport options through `proxyTls`; `connect.keepAlive` does not configure the proxy tunnel for `EnvHttpProxyAgent`. Local proxies that close reused tunnels can still surface `ECONNRESET`.
  - confidence: 0.88
- **[P2] Set gateway metadata proxy policy through proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata `ProxyAgent` also puts `keepAlive: false` under `connect`, which Undici does not use for the proxy connection. This leaves metadata fetches vulnerable to stale proxy tunnel reuse.
  - confidence: 0.86
- **[P2] Set Discord REST proxy policy through proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:18`
  - body: The REST proxy agent uses the same unsupported `connect.keepAlive` shape, so explicit Discord REST proxy calls can still reuse closed CONNECT tunnels. Put the proxy transport policy on `proxyTls`.
  - confidence: 0.86
