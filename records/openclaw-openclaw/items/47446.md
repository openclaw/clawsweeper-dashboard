## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import proxy helpers through the SDK boundary:** `extensions/discord/src/monitor/gateway-plugin.ts:9`
  - body: This new relative import makes the Discord plugin depend on core src/** internals, which violates the extension boundary and can break packaged plugin loading. Use a public SDK subpath such as openclaw/plugin-sdk/fetch-runtime instead.
  - confidence: 0.95
- **[P1] Honor NO_PROXY before applying env proxy fallback:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: resolveEnvHttpProxyUrl only selects HTTPS_PROXY/HTTP_PROXY; it does not check whether discord.com or gateway.discord.gg is excluded by NO_PROXY. With HTTPS_PROXY set and Discord bypassed, this routes bot gateway traffic and tokens through the proxy anyway.
  - confidence: 0.92
- **[P2] Apply env proxy tunnel policy through proxyTls:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Undici applies proxy-connection transport options through proxyTls, not this connect object, so connect.keepAlive does not disable reuse of the proxy tunnel for EnvHttpProxyAgent. Local proxies that close reused tunnels can still surface ECONNRESET.
  - confidence: 0.88
- **[P2] Apply gateway metadata proxy policy through proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The gateway metadata ProxyAgent also puts keepAlive under connect, which Undici does not use for the proxy connection. If this metadata proxy path is kept, it needs the shared proxy helper or proxyTls-based tunnel policy.
  - confidence: 0.86
- **[P2] Apply Discord REST proxy policy through proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:15`
  - body: The REST proxy agent uses the same unsupported connect.keepAlive shape, so explicit Discord REST proxy calls can still reuse stale CONNECT tunnels. Use the shared HTTP/1 ProxyAgent helper or put the proxy transport policy on proxyTls.
  - confidence: 0.86
