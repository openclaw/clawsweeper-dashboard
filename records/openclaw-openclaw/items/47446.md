## Summary

Keep open: the proxy bug is plausible and useful to fix, but the current PR head is still proof-gated and has concrete correctness/security issues around plugin boundaries, NO_PROXY handling, and Undici proxy tunnel options. Current main appears to cover only the gateway-wide env-proxy bootstrap portion, not the full Discord WebSocket/REST behavior this PR is trying to change.

## What This Changes

The PR bootstraps gateway env-proxy dispatching, adds Discord env-proxy fallback for gateway/REST paths, disables proxy keep-alive on Undici agents, and updates related tests plus small gateway command cleanup.

## Best Possible Solution

Land a current-main Discord proxy fix that uses SDK proxy helpers, checks target-aware NO_PROXY before env-derived Discord routing, puts proxy tunnel options under proxyTls, and includes focused tests plus redacted real local-proxy proof.

## Reproduction Assessment

Yes for the patch defects: source inspection of the PR diff, plugin boundary policy, current proxy helpers, and Undici 8.2.0 source/types shows the boundary, NO_PROXY, and proxyTls issues. The original Discord local-proxy ECONNRESET behavior is not live-reproduced because the manual verification remains unchecked.

## Solution Assessment

No. The fix direction is useful, but this implementation is not the best solution because it crosses the plugin boundary, ignores NO_PROXY for env-derived Discord routing, and configures proxy tunnel policy through `connect` instead of `proxyTls`.

Full review comments:

- **[P1] Use the SDK proxy helper instead of core src imports:** `extensions/discord/src/monitor/gateway-plugin.ts:9`
  - body: This new relative import makes the Discord plugin depend on core `src/**` internals, which violates the plugin boundary and can break packaged plugin loading. Import the proxy helper from `openclaw/plugin-sdk/fetch-runtime` instead.
  - confidence: 0.95
- **[P1] Honor NO_PROXY before creating Discord env proxy agents:** `extensions/discord/src/monitor/gateway-plugin.ts:274`
  - body: `resolveEnvHttpProxyUrl("https")` only selects a proxy URL; it does not check whether `discord.com` or `gateway.discord.gg` is excluded by `NO_PROXY`. With `HTTPS_PROXY` set and Discord bypassed, auth-bearing traffic is still forced through the proxy.
  - confidence: 0.92
- **[P2] Set env proxy tunnel policy through proxyTls:** `src/infra/net/undici-global-dispatcher.ts:130`
  - body: Undici applies proxy-connection transport options through `proxyTls`; `connect.keepAlive` does not configure the proxy tunnel for `EnvHttpProxyAgent`. Local proxies that close reused tunnels can still surface `ECONNRESET`.
  - confidence: 0.88
- **[P2] Set gateway metadata proxy policy through proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata `ProxyAgent` also puts `keepAlive: false` under `connect`, which Undici does not use for the proxy connection. This leaves metadata fetches vulnerable to stale proxy tunnel reuse.
  - confidence: 0.86
- **[P2] Set Discord REST proxy policy through proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:15`
  - body: The REST proxy agent uses the same unsupported `connect.keepAlive` shape, so explicit Discord REST proxy calls can still reuse closed CONNECT tunnels. Put the proxy transport policy on `proxyTls`.
  - confidence: 0.86
