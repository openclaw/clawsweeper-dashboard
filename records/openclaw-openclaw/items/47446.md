## Summary

Keep this PR open, but it is not merge-ready: the direction is useful, yet the current diff violates the plugin boundary, bypasses target-aware NO_PROXY handling, and puts the proposed Undici proxy keep-alive policy on option fields that do not control the proxy tunnel.

## What This Changes

The PR adds Discord gateway/REST env-proxy fallback and disables proxy tunnel keep-alive for gateway startup, Discord metadata, and Discord REST traffic.

## Best Possible Solution

Revise this into a current-main Discord proxy fix that uses public SDK proxy helpers, checks Discord API and Gateway hosts with target-aware NO_PROXY semantics, configures Undici proxy tunnel policy through `proxyTls`, drops unrelated node-command refactors, and adds a changelog entry if user-facing behavior changes.

## Reproduction Assessment

Yes for the patch defects: the PR diff, current plugin boundary rules, current proxy-env helpers, and Undici v8.1.0 types/source reproduce the merge blockers by inspection. The original live Discord proxy ECONNRESET report is still unclear without the unchecked manual local-proxy validation.

## Solution Assessment

No. The requested fix is valid in direction, but this implementation is not the narrowest maintainable solution because it bypasses the plugin SDK boundary, ignores NO_PROXY when converting env proxy state into explicit Discord agents, and uses the wrong Undici option surface for proxy tunnel policy.
