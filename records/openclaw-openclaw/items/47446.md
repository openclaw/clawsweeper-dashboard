## Summary

Keep this PR open: the Discord proxy reliability direction is still useful, but the current branch has blocking implementation issues around plugin boundaries, NO_PROXY handling, and Undici proxy tunnel options, and it still lacks real local-proxy behavior proof.

## What This Changes

The PR adds Discord env-proxy fallback for gateway/REST paths, changes proxy agents to disable keep-alive, bootstraps Undici env-proxy dispatching in gateway startup, and updates focused proxy tests.

## Best Possible Solution

Refresh this as a current-main Discord proxy fix that stays behind SDK/plugin boundaries, uses target-aware NO_PROXY checks for Discord hosts, configures proxy tunnel policy through `proxyTls` or shared dispatcher helpers, coordinates with the newer metadata-proxy work, and includes redacted real local-proxy proof.

## Reproduction Assessment

Yes for the patch defects: source inspection of the PR diff, current plugin boundary rules, current proxy helpers, and Undici 8.3.0 source show the boundary, NO_PROXY, and proxyTls problems. The original Discord local-proxy ECONNRESET path was not live-reproduced because the PR still lacks manual proof.

## Solution Assessment

No. The fix direction is useful, but this implementation is not the best way because it imports core source from plugin code, ignores target-aware NO_PROXY handling, and configures proxy tunnel policy through `connect` instead of `proxyTls`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the SDK proxy helper from Discord plugin code:** `extensions/discord/src/monitor/gateway-plugin.ts:9`
  - body: This new relative import makes the Discord plugin depend on core `src/**` internals, which violates the plugin boundary and can break packaged plugin loading. Import proxy helpers from `openclaw/plugin-sdk/fetch-runtime` instead.
  - confidence: 0.95
- **[P1] Honor NO_PROXY before creating Discord env proxy agents:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: `resolveEnvHttpProxyUrl` only selects a proxy URL; it does not check whether `discord.com` or `gateway.discord.gg` is excluded by `NO_PROXY`. With `HTTPS_PROXY` set and Discord bypassed, bot traffic is still forced through the proxy.
  - confidence: 0.92
- **[P2] Put env-proxy tunnel policy under proxyTls:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Undici applies proxy-connection transport options through `proxyTls`; `connect.keepAlive` does not configure the proxy tunnel for `EnvHttpProxyAgent`. Local proxies that close reused tunnels can still surface `ECONNRESET`.
  - confidence: 0.88
- **[P2] Put gateway metadata proxy policy under proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata `ProxyAgent` also puts `keepAlive: false` under `connect`, which Undici does not use for the proxy connection. This leaves metadata fetches vulnerable to stale proxy tunnel reuse.
  - confidence: 0.86
- **[P2] Put Discord REST proxy policy under proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:15`
  - body: The REST proxy agent uses the same unsupported `connect.keepAlive` shape, so explicit Discord REST proxy calls can still reuse closed CONNECT tunnels. Put the proxy transport policy on `proxyTls` or use the shared HTTP/1 proxy helper.
  - confidence: 0.86
