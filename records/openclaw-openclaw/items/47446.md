## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the SDK proxy helper from Discord plugin code:** `extensions/discord/src/monitor/gateway-plugin.ts:9`
  - body: This relative import makes the Discord plugin depend on core src/** internals, which violates the extension boundary and can break packaged plugin loading. Import the proxy helper from an SDK subpath instead.
  - confidence: 0.95
- **[P1] Honor NO_PROXY before creating Discord env proxy agents:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: resolveEnvHttpProxyUrl only selects a proxy URL; it does not check whether discord.com or gateway.discord.gg is excluded by NO_PROXY. With HTTPS_PROXY set and Discord bypassed, bot traffic is still forced through the proxy.
  - confidence: 0.92
- **[P2] Put env-proxy tunnel policy under proxyTls:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Undici applies proxy-connection transport options through proxyTls; connect.keepAlive does not configure the proxy tunnel for EnvHttpProxyAgent. Local proxies that close reused tunnels can still surface ECONNRESET.
  - confidence: 0.88
- **[P2] Put gateway metadata proxy policy under proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata ProxyAgent also puts keepAlive: false under connect, which Undici does not use for the proxy connection. This leaves proxied metadata fetches vulnerable to stale tunnel reuse if this path is kept.
  - confidence: 0.86
- **[P2] Put Discord REST proxy policy under proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:15`
  - body: The REST proxy agent uses the same unsupported connect.keepAlive shape, so explicit Discord REST proxy calls can still reuse closed CONNECT tunnels. Put the proxy transport policy on proxyTls or use the shared helper.
  - confidence: 0.86
