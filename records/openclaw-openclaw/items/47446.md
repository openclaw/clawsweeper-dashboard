## Summary

Keep open. The PR targets a plausible Discord proxy bug, but the current branch still has blocking implementation issues and lacks real behavior proof; the existing ClawSweeper review concerns remain supported by current main and Undici 8.2.0 source.

## What This Changes

The PR bootstraps gateway env-proxy dispatching, adds Discord env-proxy fallback for gateway and REST paths, disables proxy keep-alive on Undici proxy agents, and updates related tests plus small gateway command cleanup.

## Best Possible Solution

Land a current-main Discord proxy fix that uses SDK proxy helpers, checks target-aware NO_PROXY before env-derived Discord routing, applies proxy tunnel options through proxyTls, and includes focused tests plus redacted real local-proxy proof.

## Reproduction Assessment

Yes for the patch defects: source inspection of the PR diff, extension boundary policy, current proxy helpers, and Undici 8.2.0 source/types shows the boundary, NO_PROXY, and proxyTls issues. The original Discord local-proxy ECONNRESET behavior was not live-reproduced because the PR still lacks manual verification proof.

## Solution Assessment

No. The fix direction is useful, but this implementation is not the best solution because it crosses the plugin boundary, ignores target-aware NO_PROXY handling, and configures proxy tunnel policy through connect instead of proxyTls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the SDK proxy helper from Discord plugin code:** `extensions/discord/src/monitor/gateway-plugin.ts:9`
  - body: This new relative import makes the Discord plugin depend on core `src/**` internals, which violates the plugin boundary and can break packaged plugin loading. Import the proxy helper from `openclaw/plugin-sdk/fetch-runtime` instead.
  - confidence: 0.95
- **[P1] Honor NO_PROXY before creating Discord env proxy agents:** `extensions/discord/src/monitor/gateway-plugin.ts:274`
  - body: `resolveEnvHttpProxyUrl("https")` only selects a proxy URL; it does not check whether `discord.com` or `gateway.discord.gg` is excluded by `NO_PROXY`. With `HTTPS_PROXY` set and Discord bypassed, auth-bearing traffic is still forced through the proxy.
  - confidence: 0.92
- **[P2] Put env-proxy tunnel policy under proxyTls:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Undici applies proxy-connection transport options through `proxyTls`; `connect.keepAlive` does not configure the proxy tunnel for `EnvHttpProxyAgent`. Local proxies that close reused tunnels can still surface `ECONNRESET`.
  - confidence: 0.88
- **[P2] Put gateway metadata proxy policy under proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata `ProxyAgent` also puts `keepAlive: false` under `connect`, which Undici does not use for the proxy connection. This leaves metadata fetches vulnerable to stale proxy tunnel reuse.
  - confidence: 0.86
- **[P2] Put Discord REST proxy policy under proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:15`
  - body: The REST proxy agent uses the same unsupported `connect.keepAlive` shape, so explicit Discord REST proxy calls can still reuse closed CONNECT tunnels. Put the proxy transport policy on `proxyTls`.
  - confidence: 0.86
