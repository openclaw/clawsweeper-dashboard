## Summary

Keep open. Current main already covers the gateway-owned fetch bootstrap part, but it does not fully implement the PR's Discord WebSocket env-proxy fallback or explicit proxy tunnel reuse mitigation; the proposed patch is still not mergeable because it crosses the plugin boundary, ignores target-aware NO_PROXY semantics, and puts undici proxy tunnel policy on unsupported option paths.

## What This Changes

The PR adds env-proxy fallback for Discord gateway and REST traffic, installs an EnvHttpProxyAgent during gateway startup, changes proxy-agent keep-alive options, updates focused proxy tests, and includes small gateway node-command refactors.

## Best Possible Solution

Revise the branch into a narrow Discord proxy fix that uses public plugin SDK exports, chooses env proxies with target-aware NO_PROXY checks for Discord API and Gateway hosts, configures undici proxy tunnel policy through supported proxyTls/requestTls option surfaces, rebases onto current main, leaves unrelated gateway node-command refactors out, and adds the required user-facing changelog entry if the behavior change remains.

## Reproduction Assessment

Yes for the merge-blocking patch defects: they reproduce by diff and source inspection against the extension boundary rules, the current proxy-env helper contract, and undici v8.1.0 source. The original live Discord proxy failure is still unclear because the manual local-proxy verification remains unchecked.

## Solution Assessment

No. The direction is useful, but this is not the best implementation because it bypasses the public plugin boundary, applies env proxy fallback without target-aware NO_PROXY handling, and places proxy tunnel keep-alive policy on unsupported undici option paths.
