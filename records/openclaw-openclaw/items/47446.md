## Summary

Keep open. Current main has global env-proxy bootstrap support, but the Discord gateway WebSocket path still does not implement the PR's env-proxy fallback, and this PR is not mergeable because it introduces plugin-boundary, NO_PROXY, and Undici proxy-option regressions.

## What This Changes

The PR adds env-proxy fallback for Discord gateway/REST traffic, bootstraps EnvHttpProxyAgent during gateway startup, changes proxy keep-alive options, updates focused proxy tests, and includes small gateway node-command refactors.

## Best Possible Solution

Revise this into a narrow current-main Discord proxy fix that uses public SDK proxy helpers, honors target-aware NO_PROXY for Discord API/Gateway hosts, configures Undici proxy tunnel policy through supported proxyTls/requestTls surfaces, drops unrelated refactors, and adds the required user-facing changelog entry if behavior changes.

## Reproduction Assessment

Yes for the patch defects: the diff, current plugin boundary rules, current proxy-env helpers, and repo Undici proxy tests reproduce the merge blockers by inspection. The original live Discord proxy failure is still unclear because the manual local-proxy check remains unchecked.

## Solution Assessment

No. The direction is useful, but this is not the best implementation because it bypasses the public plugin boundary, ignores NO_PROXY when converting env proxy state into explicit Discord agents, and puts tunnel keep-alive policy on unsupported Undici option paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop importing core src from the Discord plugin:** `extensions/discord/src/monitor/gateway-plugin.ts:7`
  - body: This new relative import reaches from a bundled plugin into core src, which violates the plugin boundary and can break packaged plugin loading. Current main already exposes the needed proxy helpers through `openclaw/plugin-sdk/fetch-runtime`, so use that public subpath instead.
  - confidence: 0.95
- **[P1] Honor NO_PROXY for Discord env proxy fallback:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: `resolveEnvHttpProxyUrl("https")` only selects a proxy URL; it does not check whether `discord.com` or `gateway.discord.gg` is excluded by `NO_PROXY`. With `HTTPS_PROXY` set and Discord bypassed, this forces auth-bearing gateway traffic through a proxy the operator asked to skip.
  - confidence: 0.92
- **[P2] Put EnvHttpProxyAgent tunnel policy on proxyTls:** `src/infra/net/undici-global-dispatcher.ts:96`
  - body: Undici's env proxy path applies proxied HTTPS transport settings through `proxyTls`, while `connect` is used for the direct/no-proxy agent. Placing `keepAlive: false` under `connect` leaves env-proxied HTTPS tunnel reuse unchanged, so the claimed ECONNRESET mitigation can remain ineffective.
  - confidence: 0.88
- **[P2] Configure gateway metadata ProxyAgent via proxyTls:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata fetch agent also puts `keepAlive: false` under `connect`, but `ProxyAgent` uses `proxyTls` for the proxy connection. Metadata fetches through the proxy can still reuse stale CONNECT tunnels and keep surfacing ECONNRESET.
  - confidence: 0.86
- **[P2] Configure Discord REST ProxyAgent via proxyTls:** `extensions/discord/src/monitor/rest-fetch.ts:18`
  - body: The explicit Discord REST proxy agent uses the same unsupported `connect.keepAlive` shape. REST calls through `channels.discord.proxy` can still reuse a closed proxy tunnel, so this path does not enforce the proposed local-proxy ECONNRESET fix.
  - confidence: 0.86
