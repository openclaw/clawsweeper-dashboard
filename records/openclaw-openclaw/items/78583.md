## Summary

Keep this PR open: current main does not have AgentKit/World HITL, and the PR still has a blocking security/API-boundary issue where a public plugin SDK subpath exports operator-admin gateway helpers while the verified resolver trusts a caller-supplied plugin id.

## What This Changes

Adds an opt-in bundled AgentKit plugin with World ID HITL approvals, proof-backed gateway approval resolution, TUI/channel approval actions, docs, changelog, dependencies, protocol/client updates, and tests.

## Best Possible Solution

Keep AgentKit opt-in, but change the proof-backed approval seam so only AgentKit's verified proof path can resolve allow decisions without exporting general operator.admin helpers to the public plugin SDK.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is a feature/security-hardening PR. The blocker is source-reproducible from the PR diff: public SDK admin exports combine with a resolveVerified path that trusts caller-declared pluginId.

## Solution Assessment

No. The feature direction may be reasonable, but exporting operator.admin gateway helpers through a public plugin SDK subpath is not the narrowest maintainable proof-backed approval boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Keep admin gateway helpers out of the public SDK:** `src/plugin-sdk/gateway-runtime.ts:38-41`
  - body: This export makes `withOperatorAdminGatewayClient` available from the public `openclaw/plugin-sdk/gateway-runtime` subpath. `plugin.approval.resolveVerified` only checks a caller-supplied `pluginId` against the pending request, so another plugin can use the same helper/API shape to resolve AgentKit proof-required approvals without going through AgentKit's World verification path; keep the admin client internal or add a caller-bound proof resolver.
  - confidence: 0.82
