## Summary

Keep open: the feature direction and supplied TUI/gateway proof are plausible, but the PR has a blocking authorization issue in the chat approval command path.

## What This Changes

The PR adds an opt-in bundled AgentKit plugin with World-backed HITL approvals, gateway proof-backed plugin approval resolution, TUI/channel approval rendering, CLI helpers, docs, config metadata, and tests.

## Best Possible Solution

Keep the AgentKit proof-backed approval design, but require chat `/agentkit approvals` and `/agentkit approve` to be current-session scoped unless the invoker is explicitly owner/admin, while preserving host CLI/operator access for cross-session inspection.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: an authorized plugin-command sender can invoke `/agentkit approve <id>`, the handler searches all pending AgentKit approvals, and no session/owner check occurs before the admin-scoped verified resolver is used.

## Solution Assessment

No. The overall feature can be the right shape, but the current command implementation needs a narrower authorization boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope AgentKit chat approvals to the current session:** `extensions/agentkit/src/command.ts:163-174`
  - body: The runtime `/agentkit approve` command fetches all pending AgentKit approvals and accepts any explicit approval ID without checking it belongs to `ctx.sessionKey` or requiring an owner/admin command scope. Since the follow-up path resolves through an operator.admin gateway client, an authorized sender in another chat/session could start World verification and release someone else's blocked tool call. Filter chat approvals to the invoking session, or require owner/admin scope for cross-session IDs.
  - confidence: 0.9
