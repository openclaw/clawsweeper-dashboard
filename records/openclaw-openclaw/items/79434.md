## Summary

Keep open: the PR addresses a real Telegram bug but remains incomplete because the inbound authorization path still resolves raw account groups and can reject the reported group before delivery. It also needs a changelog entry and stronger runtime proof before merge.

## What This Changes

The PR changes Telegram account config merging so explicit empty account-level groups fall back to root Telegram groups for single-account bots, with regression coverage for the merge helper.

## Best Possible Solution

Share the empty-as-absent single-account group inheritance rule with the runtime group-policy authorization path, add regression coverage there plus the merge helper test, add a changelog entry, and require redacted gateway or maintained runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main resolves accounts.default.groups: {} as a defined empty runtime map, so resolveChannelGroupPolicy skips root channels.telegram.groups and evaluateTelegramGroupPolicyAccess can return group-chat-not-allowed before delivery. I did not run a live Telegram gateway scenario in this read-only review.

## Solution Assessment

No. Updating mergeTelegramAccountConfig is necessary but not sufficient because inbound authorization still uses the raw shared group-policy resolver; the narrower maintainable fix is to align that resolver and cover the runtime path.
