## Summary

Keep open: the PR fixes the merge helper path, but the Telegram runtime authorization path still resolves the group allowlist from the raw config, so the linked allowlist rejection can still happen. The after-fix proof is also helper-only and does not show the gateway/group-policy path delivering an inbound group update.

## What This Changes

This PR changes Telegram account config merging so an explicit empty account-level `groups` map falls back to root groups for single-account bots, and adds account-config regression tests.

## Best Possible Solution

Share the empty-as-absent single-account group inheritance rule with the runtime group-policy resolver or a common helper used by both merge and authorization, then add a runtime authorization regression while preserving multi-account isolation.

## Reproduction Assessment

Yes, source-reproducible: current main treats `accounts.default.groups: {}` as a defined account map in both the merge helper and raw runtime policy lookup, so the reported root `channels.telegram.groups` entry is skipped. I did not run a live Telegram gateway scenario in this read-only review.

## Solution Assessment

No. Updating `mergeTelegramAccountConfig` is necessary, but it is not the complete narrow fix because Telegram inbound authorization still calls `resolveChannelGroupPolicy` on raw config and can reject the reported group before delivery.
