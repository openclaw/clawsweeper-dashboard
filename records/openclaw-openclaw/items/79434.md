## Summary

Keep open: this is a plausible Telegram bug fix, but the current PR still only changes the merged account config and does not align the raw runtime group-policy resolver used by inbound authorization, so the reported group can still be rejected before delivery. It also still needs after-fix proof of the real gateway/group-policy path.

## What This Changes

This PR treats empty Telegram account-level `groups` as unset for single-account root group fallback, adds account-config regression tests, and includes an unrelated qa-lab test lint cleanup.

## Best Possible Solution

Share the empty-as-absent single-account group inheritance rule between Telegram account merging and runtime group-policy authorization, add a runtime authorization regression and changelog entry, then require redacted gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main treats `accounts.default.groups: {}` as a defined account map in both the merge helper and raw runtime policy lookup, so root `channels.telegram.groups` is skipped. I did not run a live Telegram gateway scenario in this read-only review.

## Solution Assessment

No. The helper change is necessary but incomplete because inbound authorization still resolves group policy from raw config, so the same empty object can reject the reported chat before delivery.
