## Summary

Keep open: the PR fixes part of the Telegram account merge helper, but current inbound authorization still resolves group policy from the raw config, so the reported group update can still be rejected before delivery. It also still needs a changelog entry and stronger real behavior proof for the affected runtime path.

## What This Changes

The branch treats an explicit empty Telegram account-level `groups` map as unset for single-account root group fallback and adds merge-helper regression tests.

## Best Possible Solution

Align the empty-as-absent single-account inheritance rule in the runtime group-policy path, keep the merge-helper coverage, add runtime authorization coverage and a changelog entry, then merge after redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main resolves `accounts.default.groups: {}` as a defined empty map in both the merge helper and shared group-policy resolver, and Telegram can return `group-chat-not-allowed` before delivery.

## Solution Assessment

No. Changing `mergeTelegramAccountConfig` is necessary but not sufficient because inbound authorization still calls `resolveChannelGroupPolicy` on the raw config; the maintainable fix should share or centralize the same empty-as-absent rule there.
