## Summary

Keep open: the reported Telegram bug is valid, but this PR is still incomplete because the runtime inbound authorization path can keep using the empty account group map, the user-facing fix lacks a changelog entry, and the branch mixes in an unrelated dependency override that needs security/dependency-owner handling.

## What This Changes

The branch changes Telegram account group inheritance and adds merge-helper tests, while also pinning transitive `fast-uri` through root pnpm overrides and the lockfile.

## Best Possible Solution

Apply the empty-as-absent rule consistently to Telegram single-account group inheritance in both merge and shared group-policy resolution, preserve multi-account isolation, add regression/changelog coverage, and split the dependency pin into an approved security/dependency change.

## Reproduction Assessment

Yes, by source inspection. Current main shows both the merge helper and raw group-policy resolver treating `{}` as defined, and Telegram inbound authorization calls the raw resolver before returning `group-chat-not-allowed`.

## Solution Assessment

No. Updating `mergeTelegramAccountConfig` is necessary but not sufficient because the gateway inbound path uses `resolveChannelGroupPolicy` on raw config; the unrelated dependency override should be handled separately.
