## Summary

Keep open: the reported Telegram bug is valid, but this PR only changes the merged account config path. Runtime group authorization still resolves policy from raw config, where `accounts.default.groups: {}` suppresses root `channels.telegram.groups`, so the reported `not-allowed` drop can remain.

## What This Changes

The PR changes Telegram account config merging so an explicit empty account `groups` map falls back to root groups in single-account setups and adds account-config regression tests.

## Best Possible Solution

Unify Telegram single-account group inheritance in the runtime group-policy resolver or a shared helper used by both paths, then add a runtime authorization regression while preserving multi-account isolation.

## Reproduction Assessment

Yes, source-reproducible: current main shows both the merge helper and runtime group-policy lookup treating `groups: {}` as a defined account map, which blocks root group fallback. I did not run a live Telegram gateway scenario in this read-only review.

## Solution Assessment

No. Updating `mergeTelegramAccountConfig` is necessary, but it is not the narrow complete fix because Telegram runtime authorization still uses the raw group-policy path.
