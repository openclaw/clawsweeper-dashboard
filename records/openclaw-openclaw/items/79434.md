## Summary

Keep open: the branch fixes the merge-helper half of a real Telegram bug, but the latest diff still leaves inbound authorization resolving raw account groups and needs changelog plus stronger runtime proof before merge.

## What This Changes

The PR changes Telegram account config merging so explicit empty account-level `groups` falls back to root groups in single-account setups and adds two merge-helper tests.

## Best Possible Solution

Apply the empty-as-absent single-account inheritance rule in both Telegram account merging and runtime group-policy resolution, preserve multi-account isolation, add regression and changelog coverage, then merge after redacted gateway-path proof.

## Reproduction Assessment

Yes, by source inspection. A config with root `channels.telegram.groups`, single `accounts.default.groups: {}`, and `groupPolicy: "allowlist"` still makes `resolveChannelGroupPolicy` see an empty account map and reject the root-configured group.

## Solution Assessment

No. Updating `mergeTelegramAccountConfig` is necessary but incomplete because the inbound authorization path calls `resolveChannelGroupPolicy` on raw config; the same empty-as-absent single-account rule should be shared there.
