## Summary

Keep open: the PR addresses the Telegram merge helper but is still incomplete because the runtime group-policy authorization path keeps treating `accounts.default.groups: {}` as a configured empty map, so the reported group update can still be rejected.

## What This Changes

The PR changes Telegram account group merging so empty account-level groups fall back to root groups in single-account bots, adds regression tests, and adjusts one qa-lab assertion.

## Best Possible Solution

Share the empty-as-absent single-account group inheritance rule between Telegram account merging and runtime group-policy authorization, add a runtime authorization regression and changelog entry, then require redacted gateway or maintained runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main and this PR head both let raw group-policy resolution treat `accounts.default.groups: {}` as a defined empty account map, while Telegram inbound authorization uses that raw resolver before delivery. I did not run a live Telegram gateway scenario in this read-only review.

## Solution Assessment

No. The helper change is necessary but incomplete because inbound authorization still resolves group policy from raw config, so the same empty object can reject the reported chat before delivery.
