## Summary

Keep open: the report describes a plausible current-main Telegram regression path. Partial preview streaming disables block streaming, while final-payload suppression still only keys off the block-reply pipeline or directly sent block keys, so multiple final payloads can survive to Telegram final delivery despite the earlier canonical-final-answer fix for https://github.com/openclaw/openclaw/issues/79621.

## What This Changes

Fix duplicate Telegram final replies when `channels.telegram.streaming.mode="partial"` is enabled and block streaming is off.

## Best Possible Solution

Add preview-stream-aware final dedupe/finalization so Telegram partial previews produce one visible final reply while preserving unsent media, errors, and fallback behavior.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection shows the reported path is plausible on current main: partial preview streaming disables block streaming, and final-payload suppression does not use preview-stream state.

## Solution Assessment

Unclear as written. Extending the dedupe gate is directionally plausible, but the maintainable fix should be preview-finalization-aware and covered by Telegram dispatch tests so it does not drop unsent media or error payloads.
