## Summary

Keep open: current main still lets Telegram partial-preview turns flow into final-payload delivery with block streaming disabled, so multiple assistant text payloads can be delivered after the edited preview. This is a narrow user-visible regression, not a feature request or duplicate of the earlier shipped duplicate-reply fix.

## What This Changes

The issue asks to stop Telegram from sending multiple final replies when partial draft streaming is enabled while block streaming is off.

## Best Possible Solution

Carry an explicit preview-stream-finalization fact into final-payload suppression or Telegram final delivery, and suppress already-previewed text finals without changing block streaming defaults.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: Telegram partial preview sets block streaming off, while buildReplyPayloads only suppresses final payloads for block streaming or directly sent block keys. A focused unit test can reproduce this with multiple final text payloads and an active preview stream.

## Solution Assessment

Yes, the best fix is a narrow dedupe/suppression change on the preview-streaming final path, not enabling blockStreaming globally or changing user config defaults.
