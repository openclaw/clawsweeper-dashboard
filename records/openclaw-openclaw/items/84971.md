## Summary

Keep open: the one-line validator change is a plausible data-preservation fix for the linked cron-store loss report, but it still needs full persisted-store proof and a maintainer decision on whether preserving non-canonical kinds without migration is acceptable.

## What This Changes

This PR relaxes `src/cron/persisted-shape.ts` so persisted cron jobs with any non-empty string `payload.kind` pass validation while known `systemEvent` and `agentTurn` payloads keep structural checks.

## Best Possible Solution

Land a lossless cron-store preservation fix with regression coverage and real reload/write proof, while keeping execution limited to canonical payload kinds unless a separate migration intentionally converts legacy kinds.

## Reproduction Assessment

Yes from source: seed `jobs.json` with an unsupported string `payload.kind`, then current main's validator marks it invalid, `ensureLoaded` omits it, and a later full persist can serialize the filtered store. I did not run the live write cycle because this review is read-only.

## Solution Assessment

Partially yes: relaxing persisted-shape validation is a narrow preservation fix, but the PR still needs real reload/write proof and maintainer confirmation that preserving non-canonical kinds without converting them is the intended boundary.
