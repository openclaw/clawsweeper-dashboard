## Summary

Keep open: the validator change is source-consistent, but it is compatibility-sensitive and the supplied proof still stops at the helper rather than the cron store reload/write path that caused the reported data loss.

## What This Changes

The PR changes cron persisted-shape validation to accept any non-empty string `payload.kind` and adds unit tests for canonical, legacy, arbitrary, and invalid payload kinds.

## Best Possible Solution

Land a lossless persistence fix that keeps legacy rows through reload and persist, with store-cycle regression coverage and a separate doctor migration for executable legacy conversions.

Yes from source: seed `jobs.json` with an unsupported string `payload.kind`, and current main marks it invalid, omits it from `state.store.jobs`, and can later persist only the filtered list. I did not run the live store cycle because this review is read-only.

## Solution Assessment

Partially yes: relaxing persisted validation is the narrow preservation fix, but the PR still needs store-cycle proof/coverage and maintainer confirmation that unknown kinds should be retained without becoming runnable.
