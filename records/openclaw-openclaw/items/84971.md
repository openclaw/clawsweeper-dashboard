## Summary

## What This Changes

This PR changes `getInvalidPersistedCronJobReason` so persisted cron payloads with any non-empty string `payload.kind` pass validation while known `systemEvent` and `agentTurn` shapes keep their structural checks.

## Best Possible Solution

Land a narrow lossless persistence fix with store-reload regression coverage and real proof, while keeping execution limited to canonical payload kinds unless a separate doctor migration defines legacy-kind conversion.

## Reproduction Assessment

Yes from source: seed `jobs.json` with an unsupported string `payload.kind`, then the current validator marks it invalid and the load path omits it from `state.store.jobs`. I did not run a live reload because this review is read-only.

## Solution Assessment

Partially yes: relaxing persisted-shape validation is a narrow preservation fix, but the PR still needs real reload proof and either regression coverage or an explicit follow-up for runnable legacy-kind migration.
