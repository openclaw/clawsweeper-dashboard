## Summary

Keep open: current main still suppresses final assistant text in explicit message-tool-only group/channel delivery, and the docs/tests describe that as the configured shared-room contract. The report is useful, but the requested fallback or visible warning changes that visibility policy and needs maintainer judgment before implementation.

## What This Changes

The issue requests a guard so message-tool-only group/channel turns do not appear successful but silent when the model emits final assistant text instead of calling `message(action=send)`.

## Best Possible Solution

If maintainers accept the policy change, add one shared fallback predicate for normal group/channel user requests with no successful message-tool delivery evidence, exclude room events and send-policy denies, and cover normal dispatch, follow-up dispatch, ACP dispatch, docs, and duplicate-delivery tests.

## Reproduction Assessment

Yes, source-level: current tests and dispatch code show a high-confidence path where a group/channel final reply is suppressed under `message_tool_only` when it is not marked as message-tool delivery. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

Unclear: the proposed fallback would address the silent-room symptom, but current docs and tests define strict tool-only privacy as intentional. The narrow maintainable path is a maintainer-approved policy helper, not an ad hoc bypass in one dispatch branch.
