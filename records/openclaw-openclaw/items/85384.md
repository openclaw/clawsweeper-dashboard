## Summary

Keep open: current main still suppresses unmarked final assistant replies under explicit group/channel message-tool-only delivery, and the docs/tests describe that as the current shared-room contract. The reported failure is source-reproducible, but changing it to fallback delivery or a visible warning needs a maintainer visibility-policy decision.

## What This Changes

The issue asks OpenClaw to avoid silent group/channel replies when `message_tool_only` is configured but the model emits a final response instead of using the `message` tool.

## Best Possible Solution

Have maintainers choose the shared-room visibility contract, then implement one policy helper across normal dispatch, queued follow-up dispatch, and ACP if fallback or warning behavior is accepted.

## Reproduction Assessment

Yes, at source level: current code and tests show that explicit group/channel `message_tool` mode suppresses an ordinary final reply when no message-tool delivery marker exists. I did not run a live Feishu group reproduction in this read-only review.

## Solution Assessment

Unclear: the proposed fallback would heal the silent-room symptom, but it changes a documented tool-only privacy contract. The maintainable solution should be one maintainer-approved policy helper, not an ad hoc bypass in one dispatch branch.
