## Summary

Keep open: current main and v2026.5.20 still intentionally suppress unmarked final assistant text in explicit message-tool-only group/channel delivery. The report is source-reproducible, but fixing it would change the shared-room visibility contract and needs maintainer policy review.

## What This Changes

Requests a fallback or deterministic warning when explicit `message_tool` group/channel turns finish with sendable final content but no successful message-tool delivery.

## Best Possible Solution

Maintainers should first choose the visibility policy; if approved, implement one shared guard or diagnostic that respects room events, send-policy denies, hook-blocks, and duplicate message-tool evidence, with docs and regression coverage across normal dispatch, follow-ups, and ACP.

## Reproduction Assessment

Yes, at source level: the resolver returns `message_tool_only` for explicit group/channel `visibleReplies: "message_tool"`, and current dispatch/follow-up tests show final payloads are not sent when the model misses the message tool. I did not run a live Feishu reproduction.

## Solution Assessment

Unclear: a fallback would address the silent-room symptom, but it conflicts with the current documented strict privacy semantics. The best path is a maintainer-approved shared policy helper or warning, not an ad hoc bypass in one delivery branch.
