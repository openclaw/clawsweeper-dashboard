## Summary

Keep open: current main still suppresses ordinary final assistant text for explicit message-tool-only group/channel delivery, and current docs/tests describe that as the configured shared-room contract. The report is useful and source-reproducible, but changing it to automatic fallback or visible warning needs maintainer policy judgment before implementation.

## What This Changes

The issue asks OpenClaw to avoid silent shared-room turns when message-tool-only group/channel delivery receives a final assistant response but no successful message tool delivery evidence.

## Best Possible Solution

Maintainers should first decide the shared-room policy; if they accept a fallback, implement one shared predicate for normal group/channel user requests with no successful message-tool delivery evidence, excluding room events and send-policy denies, and cover normal dispatch, follow-up dispatch, ACP dispatch, docs, and duplicate-delivery tests.

## Reproduction Assessment

Yes, source-level: current resolver, dispatch, follow-up dispatch, and tests show a group/channel final reply is suppressed under `message_tool_only` when it is not marked as message-tool delivery. I did not run live Feishu proof in this read-only review.

## Solution Assessment

Unclear: the proposed fallback would address the silent-room symptom, but current docs and tests intentionally define strict tool-only privacy. The narrow maintainable path is a maintainer-approved policy helper or diagnostic, not an ad hoc bypass in one branch.
