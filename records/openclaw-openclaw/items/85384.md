## Summary

Keep open: current main still makes explicit message-tool-only group/channel final replies private when the model misses `message(action=send)`, and changing that to a fallback or visible warning is a delivery-policy decision rather than a clean automatic cleanup close.

## What This Changes

The issue requests a last-mile fallback or visible warning when an explicit `messages.groupChat.visibleReplies: "message_tool"` group/channel turn completes with sendable final output but no successful message-tool delivery.

## Best Possible Solution

If maintainers accept the policy change, add one shared fallback or warning predicate for normal group/channel user requests with no successful message-tool delivery evidence, excluding room events and send-policy denies, then cover normal dispatch, follow-up dispatch, ACP dispatch, docs, and duplicate-delivery tests.

## Reproduction Assessment

Yes, source-level: current resolver, dispatch code, and tests show that explicit `message_tool` group/channel turns suppress unmarked final assistant text when the message tool is available but unused. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

Unclear: the requested fallback would address the silent-room symptom, but current docs and tests define strict tool-only privacy as intentional. The maintainable path is a maintainer-approved shared policy helper, not an ad hoc bypass in one dispatch branch.
