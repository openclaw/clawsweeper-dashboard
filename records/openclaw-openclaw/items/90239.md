## Summary

Keep open. The family-history API is useful and well evidenced, but embedded `chat.history(includeFamily)` still reapplies the current-session start filter after flattening family targets while Gateway disables that final filter for family reads, so embedded mode can silently drop requested ancestor/reset `subagent_announce` rows.

## What This Changes

The PR adds a default-off `includeFamily` selector to Gateway `chat.history` and `sessions_history`, updates generated client/docs/prompt snapshots, and adds bounded reset-family transcript/archive discovery tests.

## Best Possible Solution

Ship one default-off, bounded, header-checked `includeFamily` contract after embedded and Gateway filtering match and maintainers accept the archived-history visibility boundary.

Yes at source level: current main has no family-history selector, the PR adds the intended retrieval path, and the embedded parity defect is visible by comparing its unconditional final filter with Gateway's `!includeFamily` guard. I did not run live transport proof in this read-only review.

## Solution Assessment

Mostly yes: a bounded default-off `includeFamily` path is the narrowest current candidate, but it is not the best mergeable solution until embedded and Gateway filtering match and maintainers accept the archive visibility boundary.
