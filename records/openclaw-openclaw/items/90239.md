## What This Changes

The PR adds opt-in `includeFamily` history retrieval to Gateway and embedded `chat.history`, forwards it from `sessions_history` with a `resetAncestors` alias, and updates protocol/schema/docs/tests/snapshots.

## Best Possible Solution

Reserve the current session target before capped ancestors in both Gateway and embedded mode, keep bounded/header-checked archive discovery, and settle one public history selector with real terminal proof before merge.

## Reproduction Assessment

Yes. At source level, a session entry with more than 32 ancestor archive/active targets in `usageFamilySessionIds` will hit the cap before the current session id is processed, so `chat.history(includeFamily)` can omit current history.

## Solution Assessment

No. The owner boundary is plausible, but the target resolver must preserve current-session history under the cap and the public selector/proof needs maintainer-ready treatment before this is the best fix.
