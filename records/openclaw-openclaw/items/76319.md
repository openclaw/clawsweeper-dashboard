## Summary

Keep open: the patch uses the right compatibility helper for default CLI-runtime cron routing, but it does not pass the effective agent id, so cron jobs for agents with per-agent CLI runtime config can still resolve to the canonical API provider.

## What This Changes

The PR routes resolved isolated cron model providers through `resolveCliRuntimeExecutionProvider`, adds cron model-formatting regressions, and records a changelog fix.

## Best Possible Solution

Thread the effective cron agent id into model selection and use the shared compatibility helper after normal model precedence, preserving the mismatched-provider guard for overrides.

## Reproduction Assessment

Yes. A focused unit case can call cron model selection for a job targeting an `agents.list` entry with `agentRuntime.id: "claude-cli"` and an Anthropic model; with the current PR shape the helper lacks `agentId` and cannot select the per-agent CLI runtime.

## Solution Assessment

No, not yet. The helper-after-precedence approach is the right maintainable direction, but it must receive the effective agent id to cover both defaults-level and per-agent runtime policies.
