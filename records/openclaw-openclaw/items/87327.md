## Summary

Current source still records the cron timeout at the phase level, and the runtime-plugin bootstrap path delegates to the plugin registry without a cron diagnostic shape that can identify the in-flight plugin. The issue should stay open because the central operator problem remains meaningful and is not covered by the existing phase-only watchdogs.

## What This Changes

The issue asks for runtime-plugin phase diagnostics, timeouts, retry handling, and cron failure-rate surfacing for isolated agentTurn runs that stall before execution starts.

## Best Possible Solution

Keep this open and split the implementation into a maintainer-approved core plan: first add plugin-attribution diagnostics for the runtime-plugins phase, then separately decide timeout/retry policy and rolling cron health metrics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run, but the issue body provides production evidence and current source confirms the central diagnostic gap: cron only reports the last execution phase, not the in-flight plugin.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a single bundle; plugin attribution is the narrowest first fix, while retry-on-stall, per-plugin budgets, and rolling failure-rate state need maintainer decisions because they affect runtime policy and public cron schema.
