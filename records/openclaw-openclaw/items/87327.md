## Summary

Keep open: current main still has a source-level path where isolated cron runs can time out with only the last `runtime-plugins` phase after runtime plugin loading has completed, and the open related PR only addresses the named-plugin diagnostic subset while newer comments report a provider-discovery recurrence on 2026.6.5.

## What This Changes

The issue requests better attribution, containment, retry policy, failure-rate observability, and related-root-cause review for isolated cron/agent pre-execution stalls reported as `runtime-plugins`.

## Best Possible Solution

Keep this as the canonical runtime-plugins pre-execution stall issue, review the focused diagnostic PR, then make pre-execution phase attribution name the actual in-flight hook/provider/plugin work and ensure discovery failures settle or time out without introducing default-breaking retry behavior.

## Reproduction Assessment

Yes at source level, but not with a live current-main run. Current main reports `runtime_plugins` before hook/provider/model resolution completes, so a stall in those awaits can reproduce the phase-only timeout shape described in the issue.

## Solution Assessment

No single small fix is proven best yet. The diagnostic PR is a useful partial, but the broader issue needs owner decisions around phase attribution, provider-discovery failure handling, per-plugin budgets, retry policy, and cron failure observability.
