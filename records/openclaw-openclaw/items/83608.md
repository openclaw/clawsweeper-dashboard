## Summary

Keep open. The PR targets a real workspace bug, but the patch changes global workspace seeding semantics inside `ensureAgentWorkspace`, which is also used by setup/onboard/agent creation paths, and it still leaves the linked wrong-workspace/`TOOLS.md` part of the report unresolved.

## What This Changes

The PR makes `ensureAgentWorkspace` skip SOUL, IDENTITY, USER, and HEARTBEAT creation whenever the target workspace is not brand new, and updates workspace tests for that behavior.

## Best Possible Solution

Fix the subagent workspace inheritance or add a runtime-only bootstrap seeding mode so spawned runs do not write templates into the repository root, while preserving explicit setup seeding behavior and adding focused regression plus real-behavior proof.

## Reproduction Assessment

Yes, at source level: current main can pass a spawned workspace override into `agent-command`, which then calls `ensureAgentWorkspace` on that path before state checks. I did not run a live Telegram/subagent reproduction.

## Solution Assessment

No. The broad brand-new-workspace guard is not the best fix because it changes setup semantics and does not address the wrong workspace handoff or `TOOLS.md`; the narrower solution belongs in spawned workspace routing or a runtime-only seeding mode.
