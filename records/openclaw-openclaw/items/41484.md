## Summary

Keep open. Current main has useful mitigations, but it still does not provide the requested per-job elevated approval model for cron and heartbeat, and the remaining design crosses security, config, cron, and elevated-exec policy.

## What This Changes

Request adds per-job elevated permission scoping for cron jobs and heartbeat so scheduled/internal runs do not rely on wildcard elevated allowlists.

## Best Possible Solution

Define a first-class cron/heartbeat elevated trust model, then update cron schema/tool/CLI/UI/docs and elevated runtime checks so privileged scheduled work is explicit, auditable, and defaults to non-elevated.

## Reproduction Assessment

Not applicable as a feature request. Source inspection does confirm current main has no per-job elevated flag and still resolves elevated availability through provider allowlists such as cron-event and heartbeat.

## Solution Assessment

Unclear. The proposed per-job flag is plausible, but the best fix needs maintainer security/product judgment because it must cover cron creation authority, main-session heartbeat events, isolated runs, existing wildcard configs, and migration/docs.
