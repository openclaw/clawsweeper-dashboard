## Summary

Keep open. Current main has adjacent hardening for cron and heartbeat, but it still does not implement the requested persisted per-job elevated/trust flag, owner-approved mutation rule, or heartbeat provenance/explicit elevated marker.

## What This Changes

The issue requests per-cron-job and heartbeat-scoped elevated-exec trust metadata so internal cron or heartbeat events do not require wildcard elevated privileges.

## Best Possible Solution

Design and land an additive cron/heartbeat trust model that persists job elevated/provenance metadata, gates `elevated: true` creation or mutation on an owner/elevated sender, removes reliance on broad internal event allowlists, and gives heartbeat an explicit provenance or elevated marker with docs and migration-safe defaults.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: the cron schemas reject unknown properties, the persisted job type has no elevated/trust field, and heartbeat context has provider identity but no persisted provenance check.

## Solution Assessment

Unclear until maintainer/secops design. The proposed per-job flag is a plausible direction, but the maintainable fix needs explicit mutation semantics and a heartbeat equivalent rather than only adding a boolean to cron jobs.
