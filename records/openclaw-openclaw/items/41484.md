## Summary

Keep open. Current main still gates elevated exec by provider/sender and session state, while cron job schemas, gateway protocol, CLI add/edit surfaces, and docs do not expose a per-job elevated trust flag; the linked cron feature issues are adjacent but do not supersede this privilege-boundary request.

## What This Changes

Add per-job elevated execution scoping for cron jobs and heartbeat-triggered work so approved jobs can run elevated without granting wildcard elevated access to all internal cron or heartbeat events.

## Best Possible Solution

Define a durable cron/heartbeat elevated trust contract, then update cron storage, gateway protocol, agent tool schema, CLI/UI/docs, and regression tests so elevated execution is explicit per approved job or heartbeat provenance rather than granted by internal-provider wildcard.

## Reproduction Assessment

Not applicable. This is a feature/security hardening request, and source inspection on current main gives high-confidence evidence that the requested per-job elevated state is absent.

## Solution Assessment

Unclear. Per-job scoping is the right boundary to evaluate, but the owner-approval rules and heartbeat modification provenance need maintainer product/security design before implementation.
