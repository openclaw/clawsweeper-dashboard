## Summary

Keep open. Current main still lacks per-job elevated scoping for cron or heartbeat, and the requested change is a security-sensitive new privilege/provenance contract that needs maintainer product and security review before implementation.

## What This Changes

Requests per-job elevated privilege/provenance controls for cron and heartbeat so internal scheduled turns no longer rely on wildcard elevated access for cron-event or heartbeat providers.

## Best Possible Solution

Design one canonical cron/heartbeat privilege provenance model, then update the cron protocol, runtime types, normalization/service paths, elevated resolver, heartbeat handling, docs, and regression tests together.

## Reproduction Assessment

Yes at source level, not as a runtime bug repro. Current main accepts no per-job elevated field in the cron schema/types and still resolves elevated availability from provider/sender allowlists.

## Solution Assessment

Unclear. A per-job flag is plausible, but the safest solution needs maintainer product and security review for setter authority, inheritance, revocation, migration, heartbeat provenance, and docs.
