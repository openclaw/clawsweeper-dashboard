## Summary

Keep open: current main still lacks per-job elevated/provenance scoping for cron and heartbeat. There are partial mitigations for isolated cron sessions, but elevated access is still resolved through provider/session allowlists rather than a persistent job-approval boundary, so this needs maintainer security/product design.

## What This Changes

Add per-job and heartbeat provenance scoping so scheduled jobs can use elevated exec only when explicitly approved by an elevated sender.

## Best Possible Solution

Define one fail-closed cron/heartbeat elevation trust contract, then update schema, persistence, runtime checks, docs, migration/doctor guidance, and regression tests together.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that the requested per-job elevated flag and heartbeat provenance model are absent on current main, while current elevation is provider/session allowlist based.

## Solution Assessment

Unclear. The proposed per-job flag is plausible, but the best solution needs maintainer security/product decisions for job ownership, heartbeat edit provenance, existing job compatibility, and fail-closed defaults.
