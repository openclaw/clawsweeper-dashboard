## Label Justifications

- `P1`: The issue concerns elevated tool access during autonomous cron and heartbeat runs, which can affect real scheduled-agent safety now.

## Summary

Keep open: current main still grants elevated execution through provider/sender allowlists, and the cron/heartbeat paths do not carry the per-job or last-modified provenance needed to make internal scheduled runs safe by source. This remains a security-sensitive product/API decision rather than a conservative cleanup close.

## What This Changes

The issue requests provenance-aware per-job elevated execution scoping for cron and heartbeat so internal scheduled runs cannot inherit wildcard elevated privileges without an explicit trusted source.

## Best Possible Solution

Define a maintainer-approved provenance-aware elevated-exec model for cron and heartbeat, including job creation/update authority, stored job schema or migration, heartbeat directive provenance, and docs/tests for upgrade behavior.

## Reproduction Assessment

Yes from source inspection: heartbeat and cron-event runs use provider keys for elevated allowlists, while cron payloads and heartbeat directives lack per-job or last-modified elevated provenance; no live gateway scenario was run because this was a read-only review.

## Solution Assessment

A per-job flag is plausible but incomplete on its own; the best fix needs a security-reviewed trust model spanning cron creation, persisted job updates, execution, and heartbeat file provenance.
