## Summary

Keep open. Current main still lacks per-job elevated scoping for cron or heartbeat, and the prior ClawSweeper discussion remains accurate: this is a security-sensitive privilege/provenance contract that needs maintainer product and security review before implementation.

## What This Changes

Request per-job elevated execution scoping for cron jobs and heartbeat so internal scheduled or heartbeat turns do not rely on broad wildcard elevated allowlists.

## Best Possible Solution

Design one canonical cron/heartbeat elevated-provenance model, then update the cron protocol, runtime types, normalization/service paths, elevated resolver inputs, heartbeat handling, docs, migration/doctor behavior, and regression tests together.

## Reproduction Assessment

Yes at source level, not as a runtime bug repro. Current main accepts no per-job elevated field in the cron schema/types and still resolves elevated availability from provider/sender allowlists.

## Solution Assessment

Unclear. A per-job flag is plausible, but the safest solution needs maintainer product and security review for setter authority, inheritance, revocation, migration, heartbeat provenance, and documentation.
