## Summary

Keep open. Current main still lacks per-job elevated/provenance state for cron or heartbeat, related searches did not find a narrower canonical duplicate, and the remaining work is a security-sensitive authorization design rather than conservative cleanup.

## What This Changes

Add per-job cron and heartbeat provenance scoping so unattended automation can use elevated exec only when the specific job or heartbeat state was approved by an elevated sender.

## Best Possible Solution

Define one canonical elevated-provenance contract for cron and heartbeat, then update schema, runtime persistence, elevated resolution, docs, doctor or migration behavior, and regression coverage together after maintainer security review.

## Reproduction Assessment

Yes, at source level. Current main has no per-job elevated/provenance field in cron schema, cron runtime types, cron persistence, heartbeat context, or elevated resolution, and unattended turns still resolve elevated availability from provider/sender allowlists.

## Solution Assessment

Unclear. A per-job flag is plausible, but the best fix needs an explicit authority design for approval, updates, revocation, migration, and heartbeat provenance before implementation.
