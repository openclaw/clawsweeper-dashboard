## Summary

Keep open: current main and v2026.5.12 still lack a persisted per-job elevated trust marker for cron/heartbeat automation, and the request changes unattended elevated host-exec authorization rather than cleanup-close material.

## What This Changes

Request adds per-job/per-heartbeat elevated-exec trust scoping so approved automation can run elevated without granting wildcard elevated privileges to all cron/heartbeat events.

## Best Possible Solution

Design an additive cron/heartbeat elevated-trust contract covering create/update authorization, persisted provenance, heartbeat trust semantics, Gateway schema, CLI/UI/docs, audit output, migration behavior, and focused security tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request for a new persisted trust marker. Source inspection confirms current main lacks a cron elevated field and still resolves elevated availability from provider-scoped allowlists.

## Solution Assessment

Unclear until maintainers settle the trust model. The per-job flag is plausible, but the maintainable fix needs coordinated cron persistence, heartbeat provenance, protocol validation, docs, migration, and security review rather than a narrow schema field alone.
