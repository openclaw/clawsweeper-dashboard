## Summary

Keep open. Current main still lacks a per-job elevated authority/provenance contract for cron and heartbeat, and the remaining work is a security/product design choice rather than conservative cleanup.

## What This Changes

The issue requests a per-job elevated flag and provenance rules so cron and heartbeat automation can use elevated exec without wildcard internal-event allowlists.

## Best Possible Solution

Define one canonical elevated-provenance contract for cron and heartbeat, then update protocol schema, persistence, elevated resolution, docs, migration/doctor behavior, and regression coverage after security review.

## Reproduction Assessment

Yes, at source level. Current main has no cron elevated/provenance field in protocol, runtime types, create/patch persistence, or heartbeat/elevated resolution.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers need to choose approval, update, revocation, audit, migration, and heartbeat provenance semantics before implementation.
