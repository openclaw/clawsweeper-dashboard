## Summary

Keep open: current main still lacks persisted per-job elevated trust for cron and heartbeat automation, and the remaining work is a security/product design decision rather than cleanup-close material. Related cron direct-exec, cache-control, and scoped-agent items overlap implementation surfaces but do not own this specific elevated authorization decision.

## What This Changes

Request a per-cron-job and heartbeat elevated trust marker so unattended automation can use host exec only when explicitly approved.

## Best Possible Solution

Design an additive cron/heartbeat elevated-trust contract covering create/update authorization, persisted provenance, heartbeat trust semantics, Gateway schema, CLI/UI/docs, audit output, migration behavior, and focused security tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a security-design feature request for a new persisted trust marker. Source inspection gives high confidence that current main lacks a cron elevated field and still resolves elevated availability from provider-scoped allowlists.

## Solution Assessment

Unclear until maintainers settle the trust model. The per-job flag is plausible, but the maintainable fix needs coordinated cron persistence, heartbeat provenance, protocol validation, docs, migration, and security review rather than a schema field alone.
