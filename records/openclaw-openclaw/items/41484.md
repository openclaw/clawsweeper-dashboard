## Summary

Keep open: current main still lacks a persisted per-job elevated trust marker for cron and heartbeat automation, and this is a security-sensitive feature/design request rather than cleanup-close material. The prior ClawSweeper keep-open review remains consistent with current source inspection, and the related cron direct-exec, cache-control, and scoped-agent trackers do not supersede this elevated authorization problem.

## What This Changes

The issue requests per-job elevated authorization for cron and heartbeat automation so only approved persisted jobs can inherit elevated exec privileges.

## Best Possible Solution

Design an additive cron/heartbeat elevated-trust contract covering create/update authorization, persisted provenance, heartbeat trust semantics, Gateway schema, CLI/UI/docs, audit output, migration behavior, and focused security tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config/security-design request for a new persisted trust marker. Source inspection gives high confidence that current main lacks a cron elevated field and still resolves elevated availability from provider-scoped allowlists.

## Solution Assessment

Unclear until maintainers settle the trust model. The per-job flag is plausible, but the maintainable fix needs coordinated cron persistence, heartbeat provenance, protocol validation, docs, migration, and security review rather than a narrow schema field alone.
