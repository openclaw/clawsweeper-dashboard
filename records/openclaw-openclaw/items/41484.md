## Label Justifications

- `impact:security`: The reported behavior concerns elevated exec, wildcard allowFrom, and whether internal scheduled events can inherit host-level authority.

## Summary

Keep open: current main shipped command cron and fresh isolated cron sessions reduce some risky workarounds, but it still has no persisted per-job or heartbeat elevation trust model, and scheduled heartbeat/main cron turns still rely on synthetic-provider elevated allowFrom gates.

## What This Changes

Request per-job and heartbeat elevation provenance so scheduled/internal event turns can use elevated exec only when explicitly approved, without wildcard internal-event privileges.

## Best Possible Solution

Add a maintainer-reviewed scheduler/heartbeat trust model that stores approved elevation intent, gates setting it by a real elevated sender or operator admin, defaults new untrusted jobs to non-elevated, and documents/migrates existing wildcard configs safely.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current cron and heartbeat schemas do not store per-job/provenance elevation state, while scheduled turns still feed synthetic providers into elevated allowFrom resolution.

## Solution Assessment

Unclear as an implementation, but the requested direction is plausible. The best fix needs maintainer and security design around the persisted trust model, not a quick automated addition of an elevated boolean.
