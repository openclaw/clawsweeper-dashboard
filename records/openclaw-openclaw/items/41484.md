## Summary

Keep open: current main and v2026.6.6 still lack the requested per-job or heartbeat provenance model for elevated exec; the existing direct command cron path is operator-admin automation and does not solve synthetic `cron-event`/`heartbeat` elevated allowFrom scoping.

## What This Changes

The issue requests per-job elevated-exec authorization for cron jobs and heartbeat edits so scheduled/internal work no longer depends on wildcard elevated allowFrom entries.

## Best Possible Solution

Design a maintainer-approved scheduler and heartbeat trust model that stores approved elevation intent, defaults untrusted scheduled work to non-elevated, and migrates/docs existing wildcard setups safely.

## Reproduction Assessment

Yes, for source-level gap proof: current cron types, Gateway protocol, SQLite schema, and heartbeat provider routing have no per-job or heartbeat provenance field, while elevated authorization still keys off provider allowFrom. I did not run a live exploit scenario because the missing contract is directly visible in current source.

## Solution Assessment

Unclear as proposed: the direction is valid, but a simple boolean is not the best standalone fix for a host-exec trust boundary. The safer solution needs maintainer security/product design for persisted trust, owner approval, migration, and docs.
