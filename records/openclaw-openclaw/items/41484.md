## Summary

Keep open: current main and the latest release still resolve elevated execution from provider/sender allowlists, while cron jobs and heartbeat directives do not carry the per-job or last-modified provenance needed for the requested trust boundary. This remains a security-sensitive product/API design item, not a cleanup close.

## What This Changes

The issue requests per-job elevated execution provenance for cron jobs and heartbeat so internal scheduled runs no longer need wildcard elevated privileges.

## Best Possible Solution

Define a maintainer-approved provenance-aware elevated-exec model for cron and heartbeat, including job creation/update authority, stored job schema or migration, heartbeat directive provenance, and docs/tests for upgrade behavior.

## Reproduction Assessment

Yes from source inspection: heartbeat and cron-event runs still use provider keys for elevated allowlists, while cron payloads and heartbeat directives lack per-job or last-modified elevated provenance. No live gateway scenario was run because this review is read-only.

## Solution Assessment

A per-job flag is plausible but incomplete on its own; the best fix needs a security-reviewed trust model spanning cron creation, persisted job updates, execution, and heartbeat file provenance.
