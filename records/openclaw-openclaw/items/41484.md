## Label Justifications

- `P1`: The issue describes a plausible elevated exec privilege-escalation path through cron and heartbeat in multi-user gateway setups.
- `impact:security`: The affected behavior is about who can cause scheduled or heartbeat-triggered work to inherit elevated host command execution privileges.

## Summary

Keep open: current main and the latest release still have source/channel-scoped elevated exec authorization for cron and heartbeat, without the requested per-job elevated flag or heartbeat last-modified provenance gate. The related scheduling-provenance RFC is useful design context, but it is broader, still open, and does not replace this narrower elevated-exec trust-model request.

## What This Changes

Request per-job elevated exec scoping for cron jobs and heartbeat so owner-created automation can run elevated without granting wildcard elevated privileges to internal cron or heartbeat sources.

## Best Possible Solution

Design one provenance-backed scheduling and heartbeat trust model that keeps owner-created elevated automation possible while preventing newly added users or prompt-injected jobs from inheriting wildcard elevated exec.

## Reproduction Assessment

Yes from source inspection: cron jobs and heartbeat currently enter execution through source/channel-scoped contexts while the elevated permission path has no per-job provenance gate. I did not run a live privilege-escalation scenario because this is a read-only cleanup review.

## Solution Assessment

Unclear as a final design: the requested per-job flag targets the right trust problem, but it should be reconciled with the broader scheduling-provenance RFC before changing cron storage, Gateway protocol, and heartbeat behavior.
