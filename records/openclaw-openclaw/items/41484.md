## Summary

Keep open. Current main still lacks per-job elevated/provenance fields for cron or heartbeat, no merged duplicate appears to own the specific privilege-scoping request, and implementing it changes the host-exec security boundary enough to need product and security review first.

## What This Changes

Request adds per-job elevated privilege scoping so trusted cron jobs and heartbeat automation can use elevated exec without wildcard internal allowFrom entries.

## Best Possible Solution

Design one canonical cron and heartbeat elevated-provenance model, then update the protocol schema, runtime types, normalization/service paths, elevated resolver inputs, heartbeat handling, docs, migration or doctor behavior, and regression tests together.

## Reproduction Assessment

Yes at source level, not as a runtime bug repro. Current main accepts no per-job elevated field in cron schema/types and elevated availability still resolves from provider/sender allowlists.

## Solution Assessment

Unclear. A per-job flag is plausible, but the maintainable fix needs an explicit privilege-provenance design covering who can set it, how it is updated or revoked, how old wildcard setups migrate, and how heartbeat authority is represented.
