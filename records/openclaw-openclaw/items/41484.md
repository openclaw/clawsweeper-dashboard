- `P1`: The report concerns a high-impact elevated execution boundary for cron and heartbeat workflows, not just ordinary cron ergonomics.
- `impact:security`: The issue is about who can authorize elevated exec from scheduled and heartbeat-triggered work.

## Summary

Keep open: current main and v2026.6.6 still lack per-job elevated/provenance fields for cron and heartbeat-triggered work, and the shipped command-cron path is adjacent rather than a replacement for elevated agent/heartbeat scoping. The related #71712 RFC should inform the trust model, but it does not supersede this narrower elevated cron/heartbeat boundary.

## What This Changes

The issue requests per-job elevated execution scoping for cron and heartbeat-triggered work so internal cron/heartbeat sources no longer require wildcard elevated access.

## Best Possible Solution

Keep this as a maintainer-owned security/product design item: store immutable creator/elevation provenance for scheduled and heartbeat-triggered work, authorize elevated execution from that provenance, and remove the need for internal-source wildcard elevated allowlists.

## Reproduction Assessment

Yes from source inspection: cron job schemas have no per-job elevated/provenance field, while heartbeat and cron-event runs resolve elevated access through the internal provider allowlist. I did not run a live gateway scenario because this read-only review avoided mutable/runtime proof commands.

## Solution Assessment

Unclear as written: a per-job flag is plausible, but the best fix needs gateway-injected non-forgeable provenance and heartbeat edit provenance, not only a mutable job boolean. #71712 should be considered during design, but it does not replace this issue.
