## Summary

Keep open: current main and v2026.6.6 have useful command cron and fresh-isolated-session mitigations, but they still do not provide a persisted per-job or heartbeat elevation trust model, and scheduled heartbeat/main cron turns still route through synthetic provider allowFrom gates.

## What This Changes

The issue requests per-job elevated execution scoping for cron jobs and heartbeat so scheduled/internal events do not rely on wildcard elevated permissions.

## Best Possible Solution

Design a maintainer-reviewed scheduler and heartbeat trust model that stores approved elevation intent, gates setting it by a real elevated sender or operator admin, defaults untrusted scheduled work to non-elevated, and documents/migrates existing wildcard configs safely.

## Reproduction Assessment

Yes, at source level. Current cron and Gateway schemas do not store per-job/provenance elevation state, while scheduled heartbeat/main cron turns still feed synthetic providers into elevated allowFrom resolution.

## Solution Assessment

Unclear as a concrete implementation, but the requested direction is plausible. The best fix is a maintainer-approved persisted trust model, not a quick boolean added to one cron schema surface.
