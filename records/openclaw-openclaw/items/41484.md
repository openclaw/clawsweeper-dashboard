## Summary

Keep open. Current main and the latest release still lack per-job cron elevated authority and heartbeat provenance, and the adjacent related issues do not supersede this exact elevated-exec boundary.

## What This Changes

Request per-job elevated exec authorization for cron jobs and heartbeat so unattended internal events do not rely on wildcard elevated allowlists.

## Best Possible Solution

Define one canonical elevated-provenance contract for cron and heartbeat, then update protocol schema, persistence, elevated resolution, docs, migration or doctor behavior, and regression coverage together.

Yes, at source level. Current main has no cron elevated/provenance field in the protocol, runtime types, create/patch persistence, or heartbeat/elevated resolution path.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers need to choose the approval, update, revocation, audit, migration, and heartbeat provenance contract before implementation.
