## Summary

Keep open: current main still resolves elevated execution from provider/sender allowlists, while cron jobs and heartbeat directives lack the per-job or last-modified approval provenance requested here. This remains security-sensitive product/API design work, not cleanup-closeable.

## What This Changes

The issue requests per-job elevated-exec provenance for cron jobs and heartbeat runs so internal events cannot inherit broad wildcard privileges.

## Best Possible Solution

Define a maintainer-approved provenance-aware elevated-exec model for cron and heartbeat, including job creation/update authority, stored job schema or migration, heartbeat directive provenance, and docs/tests for upgrade behavior.

## Reproduction Assessment

Yes from source inspection: heartbeat and cron-event runs still use provider keys for elevated allowlists, while cron payloads and heartbeat directives lack per-job or last-modified elevated provenance. No live gateway scenario was run because this review is read-only.

## Solution Assessment

Unclear: a per-job flag is plausible but incomplete on its own. The best fix needs a security-reviewed trust model spanning cron creation, persisted job updates, execution, and heartbeat file provenance.
