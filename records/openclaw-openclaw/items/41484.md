## Summary

Keep open: current main still has provider-scoped elevated allowlists for heartbeat/cron events and no persisted per-job elevated trust contract, so this remains a security-sensitive feature/design request rather than cleanup-close material.

## What This Changes

The issue asks for per-job elevated-exec scoping for cron jobs and heartbeat so internal events do not rely on wildcard elevation.

## Best Possible Solution

Design an additive cron/heartbeat elevated-trust contract covering create/update authorization, persisted job provenance, HEARTBEAT.md trust semantics, Gateway protocol/schema, CLI/UI/docs, audit output, and focused security tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new persisted trust marker and policy. Source inspection gives high confidence that current main lacks that field and still uses provider-scoped elevated allowlists for cron/heartbeat events.

## Solution Assessment

Unclear until maintainers settle the trust model. The proposed per-job flag is plausible, but the maintainable fix needs coordinated cron persistence, heartbeat-file provenance, protocol validation, docs, and secops review rather than a narrow schema field alone.
