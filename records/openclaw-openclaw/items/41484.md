## Summary

Keep open: current main has useful mitigations, including isolated cron sessions dropping ambient elevation and operator-admin command cron payloads, but it still does not provide the requested per-job elevated flag or heartbeat provenance gate. The open scheduling-provenance RFC overlaps but does not fully replace this narrower cron/heartbeat elevated-exec trust problem.

## What This Changes

Requested per-job elevated execution scoping for cron jobs and heartbeat so internal scheduled work cannot inherit broad elevated exec privileges from wildcard event-source allow rules.

## Best Possible Solution

Keep this as a maintainer-owned security/product design item, either as this specific tracker or explicitly folded into a broader scheduling-provenance design that also covers heartbeat and elevated exec.

## Reproduction Assessment

Yes for source-level gap proof, not as a failing bug reproduction. Current main shows provider-scoped elevated allowlists and heartbeat/cron-event internal providers, but no per-job cron elevation or HEARTBEAT.md provenance gate.

## Solution Assessment

Unclear until maintainers choose the trust model. The requested direction is plausible, but the best implementation should be a designed core cron/heartbeat security contract with migration and upgrade behavior, not a narrow local flag bolted onto one path.
