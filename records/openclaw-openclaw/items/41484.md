## Label Justifications

- `P1`: The source supports a plausible multi-user path where scheduled internal origins can retain durable sandbox-escape authority.
- `impact:security`: The issue concerns authorization for elevated exec, which bypasses the sandbox and runs on the configured host target.

## Summary

Current main and v2026.6.5 still authorize elevated exec from runtime provider/sender allowlists and persist no trusted creator or approver provenance on cron jobs, so the requested security boundary remains missing. The related direct-exec and scoped-agent issues overlap operationally but do not own this authorization requirement.

## What This Changes

Add per-job elevated authorization for cron and a trusted modification-based authorization model for heartbeat so scheduled internal origins no longer require wildcard elevated access.

## Best Possible Solution

Introduce an unforgeable, auditable authorization grant owned by the scheduler mutation boundary, revoke or require reapproval on security-relevant edits, default existing and imported jobs to non-elevated, and design a separate explicit authorization mechanism for heartbeat content rather than inferring trust from an unaudited file timestamp.

## Reproduction Assessment

Yes, at source level: current cron records carry no creator or approver authority, while elevated permission is resolved from the later runtime provider and sender context. A live host-exec exploit was intentionally not attempted during this read-only review.

## Solution Assessment

No, the proposed bare boolean is not yet the best safe solution; the goal is valid, but authorization must be server-issued, mutation-sensitive, auditable, migration-aware, and separately defined for heartbeat.

## Risks / Open Questions

- A client-supplied `elevated: true` field would be forgeable unless only a trusted server-side mutation boundary can create the authorization grant.
- An elevated job could retain privilege after an unauthorized user edits its command or prompt unless every security-relevant mutation revokes or reauthorizes the grant.
- Config-backed jobs, RPC callers, imports, backups, and upgrades need explicit handling so authorization metadata cannot be injected, copied, or silently broadened.
- Heartbeat is file-backed behavior rather than a structured cron job, so “last modified by an elevated sender” needs a separate durable and auditable ownership model.
