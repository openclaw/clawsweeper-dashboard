## Summary

Keep open: current main still lacks persisted per-job elevated trust for cron and heartbeat automation, and the remaining work is a security-sensitive feature/design decision rather than cleanup-close material.

## What This Changes

Requests per-cron-job and heartbeat elevated trust scoping so approved automation can run elevated without granting wildcard elevated access to all cron or heartbeat events.

## Best Possible Solution

Design an additive cron/heartbeat elevated-trust contract covering create/update authorization, persisted provenance, heartbeat trust semantics, Gateway schema, CLI/UI/docs, audit output, migration behavior, and focused security tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/security-design request for a new persisted trust marker. Source inspection gives high confidence that current main lacks a cron elevated field and still resolves elevated availability from provider-scoped allowlists.

## Solution Assessment

Unclear until maintainers settle the trust model. The per-job flag is plausible, but the maintainable fix needs coordinated cron persistence, heartbeat provenance, protocol validation, docs, migration, and security review rather than a narrow schema field alone.
