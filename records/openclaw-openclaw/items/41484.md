## Label Justifications

- `P1`: The issue describes a plausible elevated exec privilege-escalation path through scheduled cron and heartbeat work in multi-user gateway setups.
- `impact:security`: The affected behavior controls who can cause scheduled or heartbeat-triggered work to inherit elevated host command execution privileges.

## Summary

Keep open: current main and v2026.6.8 still authorize elevated exec from the current message/internal provider allowlist and do not have the requested per-job cron elevation flag or heartbeat last-modified provenance gate. The related scheduling provenance RFC is useful design context, but it is broader and does not replace this narrower elevated-exec trust-model request.

## What This Changes

Request adds per-job elevated exec scoping for cron jobs and provenance-aware heartbeat elevation so internal cron/heartbeat sources do not rely on wildcard elevated allowlists.

## Best Possible Solution

Design one provenance-backed cron and heartbeat elevation model that preserves owner-created privileged automation while preventing newly added users or prompt-injected scheduled work from inheriting wildcard elevated exec.

## Reproduction Assessment

Yes, from source inspection: cron jobs and heartbeat currently enter execution through source/provider-scoped contexts while elevated permission resolution has no per-job or heartbeat-provenance gate. I did not run a live privilege-escalation scenario because this is a read-only cleanup review.

## Solution Assessment

Unclear as a final design: the requested per-job flag targets the right trust problem, but it should be reconciled with the broader scheduling-provenance RFC at https://github.com/openclaw/openclaw/issues/71712 before changing cron storage, Gateway protocol, and heartbeat behavior.
