## What This Changes

The issue requests per-job elevated execution authority and provenance for cron jobs and heartbeat automation.

## Best Possible Solution

Design one canonical cron and heartbeat elevated-provenance model, then update protocol schema, runtime types, cron normalization/service paths, elevated resolver inputs, heartbeat handling, docs, doctor or migration behavior, and regression tests together.

## Reproduction Assessment

Yes at source level, not as a runtime bug repro. Current main accepts no per-job elevated field in cron schema/types, cron create/patch do not persist privilege metadata, and elevated availability still resolves from provider/sender allowlists.

## Solution Assessment

Unclear. A per-job flag is plausible, but the maintainable fix needs explicit privilege-provenance design covering who can set it, how it is updated or revoked, how old wildcard setups migrate, and how heartbeat authority is represented.
