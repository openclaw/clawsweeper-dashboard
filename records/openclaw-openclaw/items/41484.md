## Summary

Keep open: current main and the latest release still lack a cron/heartbeat elevated-provenance contract, and the request changes the elevated host-exec authorization boundary rather than a narrow repair.

## What This Changes

Request per-job elevated scoping for cron jobs and heartbeat so trusted automation can run elevated without wildcard internal-event allowlists.

## Best Possible Solution

Design one canonical cron and heartbeat elevated-provenance contract, then update protocol schema, runtime types, cron service paths, elevated resolution, heartbeat handling, docs, migration or doctor behavior, and regression tests together.

## Reproduction Assessment

Yes, at source level. Current main has no per-job elevated/provenance field in cron schema/types or persistence, and elevated availability still resolves from provider/sender allowlists.

## Solution Assessment

Unclear. A per-job flag is plausible, but the maintainable fix needs an explicit privilege-provenance design for approval, update, revocation, migration, and heartbeat authority.
