## Summary

Keep open: current main still lacks per-job elevated/provenance scoping for cron and heartbeat. Existing mitigations clear ambient elevation for fresh isolated cron sessions, but elevation is still resolved through provider/session allowlists rather than a persistent job-approval boundary, so this needs security/product design before implementation.

## What This Changes

Request per-cron-job elevated execution approval and heartbeat provenance so scheduled automation can opt into elevated exec without wildcard internal-event allowlists.

## Best Possible Solution

Define one fail-closed cron/heartbeat elevation trust contract, then update schema, persistence, runtime checks, docs, migration/doctor guidance, and regression tests together.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that the requested per-job elevated flag and heartbeat provenance model are absent on current main, while current elevation is provider/session allowlist based.

## Solution Assessment

Unclear. The proposed per-job flag is plausible, but the best solution needs maintainer decisions for job ownership, heartbeat edit provenance, existing job compatibility, and fail-closed defaults.
