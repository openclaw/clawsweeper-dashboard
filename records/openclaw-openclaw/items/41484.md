## Summary

Keep open: current main still lacks a persistent per-job elevation approval boundary for cron and heartbeat. The existing code has useful partial mitigations for isolated cron sessions, but elevated exec availability is still resolved through provider/session allowlists and internal event providers rather than job-level or heartbeat-edit provenance.

## What This Changes

The issue requests a per-job elevated flag and heartbeat provenance rules so scheduled automation can run elevated only when owner-approved instead of relying on wildcard internal event providers.

## Best Possible Solution

Define one fail-closed cron/heartbeat elevation trust contract, then update schema, persistence, runtime checks, docs, migration or doctor guidance, and regression tests together.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that the requested per-job elevated flag and heartbeat provenance model are absent on current main, while current elevation remains provider/session allowlist based.

## Solution Assessment

Unclear. The proposed per-job flag is plausible, but the best solution needs maintainer security/product decisions for job ownership, heartbeat edit provenance, existing job compatibility, and fail-closed defaults.
