## Summary

This should stay open for maintainer product/security review: current main still models elevated exec as a provider/source allowlist, while cron job storage and heartbeat dispatch do not carry the requested per-job or last-modified trust boundary.

## What This Changes

The issue requests per-cron-job and heartbeat-scoped elevated execution so scheduled/internal events no longer need wildcard elevated access.

## Best Possible Solution

Define a first-class persisted trust model for cron jobs and heartbeat edits, then migrate/document wildcard internal-event elevated allowlists without broadening non-owner privilege.

## Reproduction Assessment

Yes for the source-level gap: current main still routes elevated authorization through provider/source allowlists and the cron schema has no per-job trust field. I did not run a live exploit scenario because this is a feature/security design request rather than a narrow regression with an existing expected-behavior contract.

## Solution Assessment

Unclear pending maintainer design review. The proposed per-job flag is plausible, but the final solution also needs heartbeat provenance, config compatibility, docs, and migration behavior for existing wildcard internal-event setups.
