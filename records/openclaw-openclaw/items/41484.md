## Summary

Keep open. Current main does not implement per-job elevated scoping for cron or heartbeat; the request is a security-sensitive new cron/heartbeat privilege contract rather than stale cleanup, and the existing labels correctly point to maintainer product and security review.

## What This Changes

Request per-job elevated execution scoping so approved cron or heartbeat work can use elevated exec without granting wildcard elevated access to all internal cron or heartbeat turns.

## Best Possible Solution

Design one canonical cron/heartbeat privilege provenance model, then update the cron protocol, types, normalization, tool surface, docs, and regression tests so elevated jobs can be approved explicitly without wildcard internal providers.

## Reproduction Assessment

Not applicable as a bug repro, but source inspection gives high confidence in the current behavior: the cron schema/types have no per-job elevated field, while elevated checks are still provider/sender allowlist based.

## Solution Assessment

Unclear. A per-job flag is a plausible direction, but the safest solution needs maintainer product and security review for cron update authority, heartbeat file provenance, migration, and related scoped-execution work.
