## Label Justifications

- `P1`: The issue is a source-backed elevated-exec trust-boundary gap for multi-user cron and heartbeat workflows, with host-command security impact.
- `impact:security`: The central concern is preventing scheduled internal sources from inheriting host elevated exec without per-job or provenance-scoped approval.

## Summary

Keep open: current main and v2026.6.6 have useful mitigations, including command cron jobs and agent-facing cron tool scoping, but they still do not provide the requested per-job elevated flag or heartbeat provenance gate. The open scheduling-provenance RFC overlaps, but it is broader and does not replace this narrower elevated-exec trust problem.

## What This Changes

Requested per-job elevated-exec scoping for cron jobs and heartbeat so scheduled work cannot inherit broad wildcard elevation without provenance or owner approval.

## Best Possible Solution

Define a core cron and heartbeat provenance model that records non-forgeable authorship or approval for scheduled sources and uses that provenance to decide elevated exec, with migration, docs, and security review before implementation.

## Reproduction Assessment

Yes for source-level reproduction: current cron schema, persistence, elevated allowlist resolution, and heartbeat file handling show the requested per-job elevation/provenance gate is absent. I did not run a live privilege-escalation scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security model. A per-job elevated flag is plausible, but the best fix likely needs a broader provenance contract for cron and heartbeat rather than a standalone boolean that could drift from persisted schedule authorship.
