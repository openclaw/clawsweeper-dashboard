## Label Justifications

- `P1`: The issue describes a plausible elevated host-exec escalation path through scheduled cron and heartbeat work in multi-user gateway setups.
- `impact:security`: The affected behavior controls who can cause scheduled or heartbeat-triggered work to inherit elevated command execution privileges.

## Summary

Keep open: current main and the latest release still lack a per-job cron elevation flag, heartbeat last-modified/elevated provenance, or any replacement for source-scoped elevated allowlists on internal cron/heartbeat runs. This remains a security-sensitive product/API decision, not a cleanup-close candidate.

## What This Changes

The issue requests per-job elevated-exec trust scoping for cron jobs and heartbeat so scheduled or heartbeat-triggered work does not inherit wildcard elevated access from internal event sources.

## Best Possible Solution

Design one provenance-backed cron and heartbeat elevation model that preserves owner-created privileged automation while preventing newly added users or prompt-injected scheduled work from inheriting wildcard elevated exec.

## Reproduction Assessment

Yes. Source inspection shows cron jobs and heartbeat enter execution through source/provider-scoped contexts while elevated permission resolution has no per-job or heartbeat-provenance gate; no live privilege-escalation run was attempted in this read-only review.

## Solution Assessment

Unclear as a final design. The proposed per-job flag targets the right trust problem, but it should be reconciled with the broader scheduling-provenance RFC before changing cron storage, Gateway protocol, and heartbeat behavior.
