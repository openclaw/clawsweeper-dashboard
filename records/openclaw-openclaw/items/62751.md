## Summary

Keep open. The prior ClawSweeper assessment still matches current main: `sessions.patch` emits `session:patch`, but the reported `/model` and `session_status(model=...)` paths still persist model overrides through direct session-store mutation without that hook dispatch path.

## What This Changes

The issue asks for session model changes made through `/model` or `session_status(model=...)` to emit a `session:patch` hook event with the model patch.

## Best Possible Solution

Route user-driven model-selection persistence through a shared hook-emitting session patch helper while preserving existing allowlist resolution, exact-selection behavior, live-switch pending state, and queued-followup retargeting.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register a `session:patch` listener, then use `/model` or `session_status(model=...)`; those paths currently mutate session state without the hook dispatch used by `sessions.patch`.

## Solution Assessment

Yes. Option A is the narrow maintainable fix because it restores the existing `session:patch` contract instead of adding a second model-specific event surface.
