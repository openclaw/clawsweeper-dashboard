## Summary

Keep open. Current main still has the reported hook consistency gap: `sessions.patch` emits `session:patch`, but `/model` and `session_status(model=...)` persist user model selections through direct session-store updates without the same hook dispatch path. The docs describe those model-selection paths as equivalent exact user session selections, so the observable hook gap remains valid.

## What This Changes

The issue requests that session model changes made through `/model` or `session_status(model=...)` emit a `session:patch` hook event, matching `sessions.patch`.

## Best Possible Solution

Route model-selection persistence through a shared session patch emission helper or shared patch pipeline used by `sessions.patch`, directive-only `/model`, and `session_status(model=...)`. Preserve existing live-switch and persistence semantics, fire `session:patch` only after a successful changed model write, and add focused regression coverage for both reported non-RPC paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register a `session:patch` listener, then follow the current `/model` and `session_status(model=...)` paths, which persist model changes directly without calling the hook dispatch used by `sessions.patch`.

## Solution Assessment

Yes. Option A is the narrowest maintainable fix: share the existing patch-hook behavior instead of adding a separate `session:model_changed` event before restoring `session:patch` consistency.
