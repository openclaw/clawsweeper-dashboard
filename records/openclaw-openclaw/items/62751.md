## Summary

Keep open. Current main still has the reported consistency gap: `sessions.patch` emits `session:patch`, but `/model` and `session_status(model=...)` persist model changes through direct session-store writes without dispatching that hook.

## What This Changes

The issue requests that user-driven model changes through `/model` and `session_status(model=...)` emit the existing `session:patch` hook event with a model patch.

## Best Possible Solution

Route successful user-driven model mutations through a shared hook-emitting session patch path, preserving live-switch, auth-profile, visibility, and queued-followup semantics.

## Reproduction Assessment

Yes, at source level. Follow `sessions.patch` versus `/model` or `session_status(model=...)`: only the RPC path dispatches `session:patch`, while the reported model-change paths persist directly.

## Solution Assessment

Yes. Option A is the narrow maintainable fix because the existing hook contract and model patch helper already exist; adding a new `session:model_changed` event would expand the API unnecessarily.
