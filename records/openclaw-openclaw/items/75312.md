## Summary

Keep this open. Current main still has the reported unguarded shared-memory search path, and an open upstream PR already references this issue with closing syntax, so cleanup should wait for that fix or an equivalent patch to merge.

## What This Changes

The issue requests that memory-wiki tolerate unavailable or malformed shared memory search managers so `wiki_search` preserves wiki results for shared `memory` or `all` corpus searches.

## Best Possible Solution

Land #75435, or an equivalent narrow patch, so memory-wiki validates that shared search is callable, treats shared-search failures as an empty optional corpus, preserves wiki results for `corpus="all"`, and adds query/tool regression coverage without changing the public config contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure memory-wiki with `search.backend="shared"`, call `searchMemoryWiki` with corpus `all` or `memory`, and mock `getActiveMemorySearchManager` to return a truthy manager object without `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/memory-wiki/src/query.ts`, where the optional shared memory manager is consumed, rather than changing third-party memory plugin internals or the public config schema.
