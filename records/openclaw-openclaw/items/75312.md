## Summary

Keep this issue open. Current main at 85cfc91 still has the reported memory-wiki path: shared search is enabled for `search.backend="shared"` with `corpus="memory"` or `"all"`, and `searchMemoryWiki` calls `sharedMemoryManager.search(...)` without a callable guard or search-time fallback. Open PR #75435 already references this issue with closing syntax, so the maintainer path is to review and land that PR or an equivalent narrow fix.

## What This Changes

Fix `wiki_search` so shared-backend searches for the `memory` or `all` corpus tolerate an unavailable or malformed shared memory manager and preserve wiki results.

## Best Possible Solution

Land #75435 or an equivalent narrow memory-wiki patch that validates shared search is callable, catches shared-search failures, preserves already-computed wiki results for `corpus="all"`, and adds query/tool regression coverage without changing the public config contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure memory-wiki with `search.backend="shared"`, call `searchMemoryWiki` with `corpus="all"` or `"memory"`, and have `getActiveMemorySearchManager` return a truthy manager without callable `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. The narrowest maintainable solution is in `extensions/memory-wiki/src/query.ts`, where optional shared-memory results are consumed, rather than changing memory-core or third-party provider internals; the open PR’s stated boundary matches that approach.
