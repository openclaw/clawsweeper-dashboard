## Summary

Keep this issue open. Current main still has the unguarded memory-wiki shared-search call, and open PR #75435 already references this issue as the focused fix candidate.

## What This Changes

The issue requests `wiki_search` to tolerate unavailable or malformed shared memory search managers so wiki results are preserved when `corpus` includes `memory` or `all`.

## Best Possible Solution

Review and land #75435, or an equivalent narrow memory-wiki patch, so optional shared memory failures cannot discard already-computed wiki results.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure memory-wiki with `search.backend="shared"`, call `searchMemoryWiki` with `corpus="all"` or `"memory"`, and have `getActiveMemorySearchManager` return a truthy manager without callable `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. A narrow guard/fallback at the memory-wiki consumer boundary is the best maintainable fix because shared memory search is optional augmentation and may be supplied by third-party plugins crossing the runtime contract.
