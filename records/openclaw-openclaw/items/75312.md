## Summary

Keep this issue open: current main still routes shared memory corpus searches to an unguarded `sharedMemoryManager.search(...)` call, and https://github.com/openclaw/openclaw/pull/75435 is an open closing PR rather than a landed fix.

## What This Changes

Report that `wiki_search` crashes instead of preserving wiki results when shared memory search is selected and the active manager lacks a callable `search` method.

## Best Possible Solution

Land or replace the focused memory-wiki guard from https://github.com/openclaw/openclaw/pull/75435 so unavailable shared search is treated as empty optional augmentation while wiki results survive.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `search.backend="shared"` plus `corpus="memory"` or `"all"` reaches a truthy shared manager and calls `.search` without a structural guard.

## Solution Assessment

Yes. Guarding and catching at the memory-wiki consumer boundary is the narrowest maintainable fix because shared memory is optional augmentation supplied through the memory plugin/runtime contract.
