## Summary

Keep this issue open: current main still has the unguarded shared-memory search path, and #75435 is an open, unmerged closing PR candidate rather than an implemented fix.

## What This Changes

The issue requests that `wiki_search` preserve wiki results when shared memory search is unavailable, malformed, or throws under `search.backend="shared"` with `corpus="memory"` or `"all"`.

## Best Possible Solution

Land or reject #75435; the desired end state is a narrow `memory-wiki` fallback that treats unavailable shared memory search as optional augmentation while preserving wiki results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: shared backend plus `corpus="memory"` or `"all"` reaches an unguarded `sharedMemoryManager.search(...)` call when the resolved manager is truthy.

## Solution Assessment

Yes. Guarding and catching at the `memory-wiki` consumer boundary is the narrowest maintainable solution because shared memory search is optional augmentation supplied through plugin/runtime contracts.
