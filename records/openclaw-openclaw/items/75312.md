## Summary

Keep this issue open. Current main still has the reported `wiki_search` shared-memory failure path, and open PR #75435 already references this issue as the focused implementation candidate, so cleanup should wait for that PR or an equivalent patch to merge.

## What This Changes

The issue requests that `wiki_search` tolerate unavailable or malformed shared memory search when `memory-wiki` uses `search.backend="shared"` and a call includes the memory corpus.

## Best Possible Solution

Land #75435 or an equivalent narrow memory-wiki patch that validates shared search is callable, catches shared-search failures, preserves already-computed wiki results for `corpus="all"`, and adds query/tool regression coverage without changing the public config contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure memory-wiki with `search.backend="shared"`, call `searchMemoryWiki` with `corpus="all"` or `"memory"`, and have `getActiveMemorySearchManager` return a truthy manager without callable `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. The narrowest maintainable fix is in the memory-wiki consumer boundary, because shared memory is optional augmentation and bundled or third-party memory runtimes can still return unavailable, malformed, or throwing search managers.
