## Summary

Keep this issue open: current main still has the unguarded shared-memory `wiki_search` call, and #75435 is an open, unmerged closing PR candidate rather than an implemented fix.

## What This Changes

The issue reports that `wiki_search` throws when `search.backend="shared"` and a `memory` or `all` corpus call receives a truthy shared memory manager without a callable `search` method.

## Best Possible Solution

Review and land or supersede #75435 with a narrow memory-wiki consumer fallback that treats unavailable shared search as optional augmentation while preserving wiki results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: shared backend plus `corpus="memory"` or `"all"` reaches `sharedMemoryManager.search(...)` with no structural guard or search-time catch.

## Solution Assessment

Yes. Guarding and catching at the `memory-wiki` consumer boundary is the narrowest maintainable fix because shared memory search is optional augmentation supplied through runtime/plugin contracts.
