## Summary

Keep this issue open. Current main still reaches the reported unguarded shared-memory search call, and open PR #75435 already references this issue as the focused implementation candidate, so cleanup should wait for that PR or an equivalent patch to merge.

## What This Changes

Make `wiki_search` tolerate unavailable or malformed shared memory managers when memory-wiki uses the shared backend and `corpus` includes `memory`, while preserving wiki results.

## Best Possible Solution

Land #75435 or an equivalent narrow memory-wiki patch that validates shared search is callable, catches shared-search failures, preserves already-computed wiki results for `corpus="all"`, and adds query/tool regression coverage without changing the public config contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure memory-wiki with `search.backend="shared"`, call `searchMemoryWiki` with `corpus="all"` or `"memory"`, and have `getActiveMemorySearchManager` return a truthy manager without callable `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. The narrowest maintainable fix is in memory-wiki where the optional shared memory manager is consumed, because bundled and third-party memory providers both cross that consumer boundary.
