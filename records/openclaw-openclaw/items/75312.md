## Summary

Keep this issue open: current main still reaches the shared-memory branch and directly calls `sharedMemoryManager.search(...)` without a structural guard, and the linked fix PR is still an open implementation candidate rather than an implemented fix.

## What This Changes

The issue asks `memory-wiki` `wiki_search` to tolerate unavailable or malformed shared memory search when `search.backend="shared"` and the selected corpus includes `memory`.

## Best Possible Solution

Land a narrow `memory-wiki` fallback that treats malformed or throwing shared memory search as empty optional augmentation while preserving wiki results, then let the closing PR resolve this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: shared backend plus `corpus="memory"` or `"all"` with a truthy manager whose `search` is missing or throws reaches the direct call in `extensions/memory-wiki/src/query.ts`.

## Solution Assessment

Yes. Guarding and catching at the `memory-wiki` consumer boundary is the narrowest maintainable fix because shared memory search is optional augmentation supplied through plugin/runtime contracts.
