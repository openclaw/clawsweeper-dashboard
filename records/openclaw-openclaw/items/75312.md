## Summary

Keep this issue open. Current main still has the reported `wiki_search` failure path, and the provided timeline shows open PR #75435 already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent patch to merge.

## What This Changes

The issue reports that `wiki_search` can throw when `memory-wiki` uses `search.backend="shared"` and a per-call or effective corpus includes `memory`, because the resolved shared memory manager may not have a callable `search` method.

## Best Possible Solution

Land #75435 or an equivalent narrow memory-wiki consumer-boundary fix that validates the shared manager, contains shared-search failures, preserves wiki results for `corpus="all"`, and adds regression coverage for both query and tool surfaces.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `memory-wiki` with `search.backend="shared"`, call `searchMemoryWiki` or `wiki_search` with `corpus="all"` or `"memory"`, and mock `getActiveMemorySearchManager` to return a truthy manager without `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. The best fix belongs in `extensions/memory-wiki` at the optional shared-search consumer boundary, because both built-in and third-party memory plugins flow through that runtime contract and wiki-only results should survive optional shared-memory failure.
