## Summary

Keep this issue open: current main still reaches the shared-memory branch and directly calls a truthy manager's `search` method, while #75435 is an open, unmerged closing PR candidate rather than an implemented fix.

## What This Changes

The issue asks `wiki_search` to tolerate malformed or failing shared memory search when `memory-wiki` uses `search.backend="shared"` and a call selects `corpus="memory"` or `"all"`, preserving wiki results.

## Best Possible Solution

The end state should be a narrow `memory-wiki` fallback that validates shared search and treats unavailable shared memory as empty optional augmentation while preserving wiki results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: shared backend plus `corpus="memory"` or `"all"` with `getActiveMemorySearchManager` returning a truthy manager whose `search` is missing or throws reaches `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. Guarding and catching at the `memory-wiki` consumer boundary is the narrowest maintainable fix because shared memory search is optional augmentation supplied through plugin/runtime contracts.
