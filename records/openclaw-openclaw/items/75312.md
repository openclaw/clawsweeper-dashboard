## Summary

Keep this issue open: current main still takes the shared memory path and calls a truthy manager's `search` method without validating it, while #75435 is an open closing PR candidate rather than a merged fix.

## What This Changes

Make `wiki_search` tolerate malformed or failing shared memory managers when `search.backend=shared` and the requested corpus includes memory, preserving wiki results.

## Best Possible Solution

Land #75435 or an equivalent narrow `memory-wiki` boundary guard that treats unavailable shared search as optional and preserves already-computed wiki results.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `memory-wiki` with shared backend plus `corpus="all"` or `"memory"`, with `getActiveMemorySearchManager` returning a truthy object whose `search` is missing or throws; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. A defensive guard and failure boundary at the `memory-wiki` consumer is the narrowest maintainable fix because shared memory search is optional augmentation supplied by runtime/plugin contracts.
