## Summary

Keep this issue open: current main still calls the shared memory manager's `search` method without a structural guard, and #75435 is an open closing PR rather than a merged fix.

## What This Changes

The issue asks `wiki_search` to tolerate malformed or failing shared memory search when `memory-wiki` uses `search.backend="shared"` and `corpus` includes `memory` or `all`, preserving wiki results.

## Best Possible Solution

Land #75435 or an equivalent narrow memory-wiki boundary guard that validates shared search and preserves wiki corpus results when optional memory search is unavailable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure shared search with `corpus="all"` or `"memory"` and have `getActiveMemorySearchManager` return a truthy manager without callable `search`; current main reaches the unguarded call at `extensions/memory-wiki/src/query.ts:1217`.

## Solution Assessment

Yes. The best fix is a defensive memory-wiki consumer boundary that treats shared memory search as optional augmentation, rather than changing config schema or relying on every third-party memory runtime to be well-formed.
