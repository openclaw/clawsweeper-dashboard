## Summary

Keep this issue open: current main still takes the shared-memory search branch and directly calls `sharedMemoryManager.search(...)`, and the linked closing PR #75435 is still an open implementation candidate rather than a merged fix.

## What This Changes

The issue requests that `wiki_search` tolerate unavailable or malformed shared memory search managers when `search.backend="shared"` and the selected corpus includes memory.

## Best Possible Solution

Review and land #75435, or apply the same narrow memory-wiki fallback that validates and catches optional shared search while preserving wiki results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: shared backend plus `corpus="memory"` or `"all"` reaches `extensions/memory-wiki/src/query.ts:1217`, where a truthy manager without callable `search` throws before wiki results are returned.

## Solution Assessment

Yes. Guarding and catching at the memory-wiki consumer boundary is the narrowest maintainable fix because shared memory search is optional augmentation supplied through plugin/runtime contracts.
