## Summary

Keep open: current main does not include the guardrails plugin, and the PR is still useful as a concrete plugin-layer implementation, but the latest head still has blocking security and availability issues around credential handling and unbounded connector initialization. The related #72287/#73384/#74483 discussion is relevant context for the generic SDK seams, but it does not replace this plugin implementation.

## What This Changes

The PR adds a bundled `extensions/guardrails` plugin with blacklist, HTTP-provider, and dynamic-import connectors for `before_dispatch` allow/block checks, plus static asset packaging, dependency lockfile updates, and tests.

## Best Possible Solution

Decide the ownership model first: land it as an official bundled security plugin only after SecretRef-aware credentials, bounded initialization fallback, docs/changelog alignment, dependency review, and secops approval; otherwise ask the contributor to publish it as an external ClawHub/native plugin using the existing hook surface.

## Reproduction Assessment

Yes for the blocking review defects: configure a guardrails HTTP provider key as a structured SecretRef and the PR's manifest/type path only accepts strings; configure a custom HTTP provider or import module whose `init()` never resolves and the first matching `before_dispatch` path can wait before the configured timeout/fallback is applied. For the feature itself, reproduction is not applicable because current main lacks this bundled plugin.

## Solution Assessment

No, not as-is. The plugin-shaped approach uses an appropriate current hook seam, but the current bundled implementation is not the narrowest maintainable solution until credential resolution, initialization timeouts, and maintainer/secops ownership are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Make guardrails keys SecretRef-aware:** `extensions/guardrails/openclaw.plugin.json:26-28`
  - body: The new HTTP connector accepts provider API keys from plugin config, but `http.apiKey` and channel override `apiKey` are schema/type strings. That forces guardrail provider tokens into normal plugin config instead of the supported SecretRef/redaction path, and structured SecretRef objects will not validate or resolve for this bundled plugin.
  - confidence: 0.93
- **[P2] Bound HTTP provider initialization:** `extensions/guardrails/index.ts:178`
  - body: If a custom provider's `init()` never settles, the first matching `before_dispatch` event awaits `entry.initPromise` before any guardrails timeout or fallback is applied. That can stall inbound dispatch indefinitely instead of returning the configured fail-open or fail-closed result.
  - confidence: 0.88
- **[P2] Bound import connector initialization:** `extensions/guardrails/index.ts:322`
  - body: The import connector waits for module load and optional `init(args)` before the per-check timeout wrapper exists. A bad trusted checker can therefore hang external-channel dispatch during initialization rather than honoring `timeoutMs` and `fallbackOnError`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds credential-bearing outbound checks and trusted dynamic code execution without fully integrating the existing SecretRef pipeline or bounding initialization waits.

Concerns:

- **[high] Plaintext provider API keys:** `extensions/guardrails/openclaw.plugin.json:26`
  - body: `http.apiKey` and per-channel override keys are declared and resolved as plain strings, so guardrail provider tokens would bypass the supported SecretRef target surface and redaction expectations if this lands as a bundled plugin.
  - confidence: 0.93
- **[medium] Unbounded initialization can stall dispatch:** `extensions/guardrails/index.ts:178`
  - body: HTTP provider and import connector initialization promises are awaited from `before_dispatch` before the plugin's configured timeout applies, making a hung provider/module an availability risk at an external-channel ingress boundary.
  - confidence: 0.88
- **[medium] Trusted local code execution needs ownership clarity:** `extensions/guardrails/src/import-connector.ts:23`
  - body: The import connector executes operator-specified local modules through `jiti`. That may be acceptable for trusted deployments, but bundled inclusion needs explicit docs and threat-model clarity around trusted paths, hot reload, and code execution boundaries.
  - confidence: 0.82
