## Summary

Keep open: current main has the generic `before_dispatch` seam but not this bundled guardrails plugin, and the current PR head still has concrete security and availability blockers for an official bundled plugin. Prior discussion also correctly notes that #72287/#73384/#74483 cover adjacent generic SDK seams, not this concrete plugin implementation.

## What This Changes

This PR adds a bundled `extensions/guardrails` plugin that registers `before_dispatch` allow/block checks with blacklist, HTTP-provider, and dynamic-import connectors, plus runtime asset packaging, lockfile updates, and tests.

## Best Possible Solution

Decide the ownership model first. If this is bundled, require SecretRef-aware credential contracts, config schema/runtime resolution that preserves SecretRefs, bounded initialization fallback, security documentation for trusted local imports, changelog and labeler updates, dependency review, and secops/product approval; otherwise route the plugin to external ClawHub/native plugin distribution using the existing hook seam.

## Reproduction Assessment

Yes for the review defects: configure `http.apiKey` as a structured SecretRef object and the PR's string-only schema/resolver path will reject or drop it; configure a custom HTTP provider or import module whose `init()` never settles and the first matching `before_dispatch` path waits on that promise. For the feature request itself, reproduction is not applicable because current main does not include the bundled plugin.

## Solution Assessment

No, not as-is. The plugin-shaped approach is a reasonable use of the existing hook seam, but the current bundled implementation is not the narrowest maintainable solution until credential handling, initialization timeouts, docs/metadata, and ownership review are resolved.

Full review comments:

- **[P1] Make guardrails API keys SecretRef-aware:** `extensions/guardrails/openclaw.plugin.json:26-28`
  - body: The new HTTP connector sends provider credentials, but the manifest declares `http.apiKey` and channel override `apiKey` as plain strings, and the config resolver only accepts string values. A structured SecretRef for these bundled-plugin credentials would fail schema validation or be coerced to empty, bypassing the existing SecretRef/redaction pipeline used by other credential-bearing bundled plugins.
  - confidence: 0.93
- **[P2] Bound HTTP provider initialization waits:** `extensions/guardrails/index.ts:178`
  - body: The first matching dispatch awaits `entry.initPromise` before any connector timeout or `fallbackOnError` decision is applied. A custom provider whose `init()` never settles can stall inbound external-channel dispatch indefinitely instead of returning the configured fail-open or fail-closed result.
  - confidence: 0.88
- **[P2] Bound import connector initialization waits:** `extensions/guardrails/index.ts:322`
  - body: The import backend waits for module load and user `init(args)` to finish before the per-check timeout wrapper exists. A bad or wedged local checker can therefore hang `before_dispatch` during initialization instead of honoring `timeoutMs` and `fallbackOnError`.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `extensions/guardrails/openclaw.plugin.json:4`
  - body: This adds a user-facing feature, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to have a changelog entry with contributor attribution before landing.

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds credential-bearing outbound checks and trusted dynamic code execution without fully integrating SecretRef contracts or bounding initialization waits.

Concerns:

- **[high] Plaintext provider API keys:** `extensions/guardrails/openclaw.plugin.json:26`
  - body: `http.apiKey` and per-channel override keys are string-only and no `configContracts.secretInputs` paths are declared, so guardrail provider tokens would not use the bundled plugin SecretRef resolution/redaction pipeline if this lands.
  - confidence: 0.93
- **[medium] Unbounded initialization can stall inbound dispatch:** `extensions/guardrails/index.ts:178`
  - body: HTTP provider and import connector initialization promises are awaited from the `before_dispatch` path before the plugin's configured timeout/fallback behavior applies, making a hung provider or module an availability issue at a high-risk ingress boundary.
  - confidence: 0.88
- **[medium] Trusted local code execution surface:** `extensions/guardrails/src/import-connector.ts:23`
  - body: The import connector executes operator-specified local modules through `jiti`. That may be acceptable for trusted deployments, but bundled inclusion needs explicit documentation and threat-model clarity around absolute paths, hot reload, and trusted-code boundaries.
  - confidence: 0.84
