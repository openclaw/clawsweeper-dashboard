## Summary

Keep open: current main has the generic `before_dispatch` hook but not the proposed bundled guardrails plugin, and the linked SDK seam PRs do not replace this concrete plugin work. The PR still needs maintainer/security review because it adds an official bundled security plugin with provider API keys, outbound HTTP checks, dynamic local-code execution, new runtime dependencies, and missing release/label metadata.

## What This Changes

The PR adds a bundled `extensions/guardrails` plugin with blacklist, HTTP-provider, and dynamic-import connectors, per-channel config, static asset packaging, lockfile dependency updates, and tests.

## Best Possible Solution

Maintainers should choose the ownership model first. If guardrails are accepted as a bundled official plugin, land it only after SecretRef-aware credential contracts, bounded initialization fallback, security docs for trusted imports, changelog and labeler updates, dependency review, and secops/product approval; otherwise ask for this to ship as an external ClawHub/native plugin using the existing `before_dispatch` seam.

## Reproduction Assessment

Yes for the review defects: configure `http.apiKey` as a SecretRef object and the proposed string-only schema/resolver rejects or drops it; configure a custom HTTP provider or import module whose `init()` never settles and the first matching `before_dispatch` path waits on that promise. For the feature itself, reproduction is not applicable because current main has the hook seam but not the bundled plugin.

## Solution Assessment

No, not as-is. A plugin built on `before_dispatch` is a plausible narrow architecture, but the bundled implementation still misses credential, availability, metadata, and ownership requirements for a security-sensitive official plugin.

  - body: The new HTTP connector sends provider credentials, but the manifest declares global and per-channel `http.apiKey` as string-only and the resolver only preserves strings.   - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds credential-bearing outbound checks and trusted dynamic code execution without fully integrating SecretRef contracts or bounding initialization waits.

Concerns:

- **[high] Plaintext provider API keys:** `extensions/guardrails/openclaw.plugin.json:26`
  - body: `http.apiKey` and per-channel override keys are string-only and no `configContracts.secretInputs` paths are declared, so guardrail provider tokens would not use the bundled plugin SecretRef resolution/redaction pipeline if this lands.
  - confidence: 0.93
- **[medium] Unbounded initialization can stall inbound dispatch:** `extensions/guardrails/index.ts:178`
  - body: HTTP provider and import connector initialization promises are awaited from the `before_dispatch` path before the plugin's configured timeout/fallback behavior applies, making a hung provider or module an availability issue at a high-risk ingress boundary.
  - confidence: 0.88
- **[medium] Trusted local code execution surface:** `extensions/guardrails/src/import-connector.ts:23`
  - body: The import connector executes operator-specified local modules through `jiti`. That may be acceptable for trusted deployments, but bundled inclusion needs explicit documentation and threat-model clarity around absolute paths, hot reload, and trusted-code boundaries.
  - confidence: 0.84
