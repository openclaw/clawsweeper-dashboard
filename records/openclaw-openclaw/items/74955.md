## Summary

Keep open. Current main still has the reported gap in the message-tool send path, and this PR is pointed at the right seam, but the current patch drops agent/account context when invoking the TTS runtime and adds a regression test that can be affected by local TTS prefs.

## What This Changes

The PR adds a tagged-TTS pass to `handleSendAction` for message-tool/CLI/gateway sends, adds a focused outbound regression test, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land a narrow outbound-send hook that matches the auto-reply TTS contract end to end: run the existing lazy TTS runtime before delivery when no explicit media is present, pass the same agent/account/channel context into both the gate and runtime call, isolate the new test from host prefs, and leave short-text TTS behavior to #73911/#73758.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: configure tagged TTS, send `[[tts:text]]...[[/tts:text]]` through the message-tool/CLI/gateway send path, and current main reaches outbound send without invoking the TTS runtime hook.

## Solution Assessment

No, not as written. The hook placement and existing-media guard are the right shape, but the runtime call must receive `agentId` and `accountId`, and the test must isolate TTS prefs before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass TTS context into the runtime hook:** `src/infra/outbound/message-action-runner.ts:596-600`
  - body: The new hook gates with `agentId` and `accountId`, but the `maybeApplyTtsToPayload` call itself only passes `cfg`, `channel`, and `kind`. That helper resolves auto mode and provider config from those same context fields, so scoped agent/account TTS settings can be skipped or synthesized with the global config on the message-tool path. Pass `agentId` and `accountId: accountId ?? undefined` into the runtime call, matching auto-reply.
  - confidence: 0.9
- **[P3] Isolate local TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:26-30`
  - body: `shouldAttemptTtsPayload` reads local TTS prefs before `messages.tts.auto`. The new test config does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, so a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook. Point the test config at a test-local nonexistent prefs path or isolate the env.
  - confidence: 0.82
