## Summary

Keep open: the reported outbound tagged-TTS gap is real on current main and the PR is the right shape, but the patch needs scoped context, a dry-run guard, and test isolation before merge.

## What This Changes

The PR adds a gated `maybeApplyTtsToPayload` call to outbound send actions, a tagged-TTS regression test, and a changelog entry.

## Best Possible Solution

Land the outbound-send TTS hook after it preserves dry-run as no-side-effect, passes the same scoped context as auto-reply, and keeps directive parsing/runtime policy unchanged.

## Reproduction Assessment

Yes, by source inspection. Current main sends message-tool text through `handleSendAction` and `executeSendAction` without a TTS hook, while the final-reply path invokes the same TTS normalizer with scoped context.

## Solution Assessment

No, not as written. The hook belongs in this path, but it must skip dry-run provider calls, pass `agentId` and `accountId` into the runtime hook, and isolate local TTS prefs in the regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass scoped TTS context into the hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. Since `maybeApplyTtsToPayload` re-resolves auto mode, provider, voice, and prefs from those same fields, scoped overrides can pass the gate and then synthesize with the wrong effective config or skip unexpectedly.
  - confidence: 0.91
- **[P2] Skip TTS synthesis during dry-run sends:** `src/infra/outbound/message-action-runner.ts:593-597`
  - body: This hook runs before `sendMessage` reaches its dry-run return, so `openclaw message send --dry-run` can still call a configured external TTS provider and create an audio artifact. Add `!dryRun` to the gate so dry-run remains side-effect-free.
  - confidence: 0.87
- **[P3] Isolate TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Because the new test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook.
  - confidence: 0.82
