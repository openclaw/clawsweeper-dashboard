## Summary

Keep open: the underlying outbound/message-tool TTS bug is valid and current main still lacks a TTS normalization pass on message sends, but this PR is not mergeable as written because it can synthesize during dry-run and drops scoped TTS context.

## What This Changes

The PR adds a gated tagged-TTS normalization call to outbound message sends, a regression test, and a changelog entry.

## Best Possible Solution

Land the outbound-send TTS hook after it preserves dry-run as side-effect-free, passes scoped TTS context, and keeps tests isolated from local TTS prefs.

## Reproduction Assessment

Yes, by source inspection. Current main sends message-tool text through the outbound send path without a TTS hook, while final replies call the scoped TTS normalizer before delivery.

## Solution Assessment

No, not as written. Reusing the existing TTS hook in this path is the right direction, but the PR must skip dry-run provider calls, pass `agentId` and `accountId`, and isolate TTS prefs in the regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass scoped TTS context into the hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. Since `maybeApplyTtsToPayload` re-resolves auto mode, provider, voice, and prefs from those same fields, scoped overrides can pass the gate and then synthesize with the wrong effective config or skip unexpectedly.
  - confidence: 0.91
- **[P2] Skip TTS synthesis during dry-run sends:** `src/infra/outbound/message-action-runner.ts:593-597`
  - body: This hook runs before the existing dry-run return path, so `openclaw message send --dry-run` can still call a configured external TTS provider and synthesize the message body. Add the dry-run guard before loading or invoking the TTS runtime.
  - confidence: 0.88
- **[P3] Isolate TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Because the new test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook.
  - confidence: 0.82
