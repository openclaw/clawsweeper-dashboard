## Summary

Keep open: the bug is source-reproducible on current main and the hook belongs in this path, but the PR still has two blocking correctness/security issues around scoped TTS context and dry-run side effects.

## What This Changes

The PR adds a `shouldAttemptTtsPayload`-gated `maybeApplyTtsToPayload` call to outbound message sends, a tagged-TTS regression test, and a changelog entry.

## Best Possible Solution

Land the outbound send hook after it skips dry-run, passes the same scoped TTS context as auto-reply, preserves the current `asVoice` plumbing, and keeps TTS parsing/runtime policy unchanged.

## Reproduction Assessment

Yes, by source inspection. On current main, tagged text sent through `handleSendAction` reaches outbound send without `maybeApplyTtsToPayload`, while auto-reply invokes that hook with scoped context.

## Solution Assessment

No, not as written. The hook is the right narrow fix location, but it must skip dry-run sends, pass `agentId` and `accountId` into the runtime hook, and isolate local TTS prefs in the regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass TTS context into the runtime hook:** `src/infra/outbound/message-action-runner.ts:597-602`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. `maybeApplyTtsToPayload` re-resolves auto mode, provider, voice, prefs, and `textToSpeech` from those fields, so scoped overrides can pass the gate and then synthesize with the wrong effective config. Pass `agentId` and `accountId: accountId ?? undefined` into this call.
  - confidence: 0.91
- **[P2] Skip TTS synthesis during dry-run sends:** `src/infra/outbound/message-action-runner.ts:591-595`
  - body: This hook runs before `sendMessage` reaches its dry-run return, so `openclaw message send --dry-run` can still call a configured TTS provider and create an audio artifact. Add `!dryRun` to the gate so dry-run stays a planning-only path with no provider side effects.
  - confidence: 0.87
- **[P3] Isolate TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Because this test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook. Use a test-local nonexistent prefs path or isolate the env.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete dry-run side effect by allowing message text to be sent to the configured external TTS provider before delivery is skipped.

Concerns:

- **[medium] Dry-run can leak message text to TTS provider:** `src/infra/outbound/message-action-runner.ts:591`
