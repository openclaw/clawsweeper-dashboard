## Summary

Keep open: the reported bug is valid and the TTS hook belongs on the message-tool send path, but this PR still has two merge-blocking behavior/security defects around dry-run side effects and scoped TTS resolution.

## What This Changes

The PR adds a gated tagged-TTS hook to outbound message-tool sends, a regression test, and a changelog entry.

## Best Possible Solution

Land a rebased outbound-send TTS hook that preserves dry-run as side-effect-free, passes scoped TTS context, and keeps focused regression coverage isolated from local prefs.

## Reproduction Assessment

Yes. Source inspection shows current main sends message-tool text through the outbound send path without the TTS hook, while auto-replies call the scoped TTS normalizer before delivery; the PR body also includes after-fix live Telegram proof from a hot-patched install.

## Solution Assessment

No, not as written. Reusing the existing TTS hook is the narrow maintainable fix, but this branch must skip dry-run synthesis, pass agent/account context into the runtime call, and isolate TTS prefs in the test before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip TTS synthesis during dry-run sends:** `src/infra/outbound/message-action-runner.ts:593-597`
  - body: This hook runs before the existing no-delivery dry-run boundary, so `openclaw message send --dry-run` can still send the message body to a configured external TTS provider. Gate the whole block on `!dryRun` before loading or invoking the TTS runtime.
  - confidence: 0.9
- **[P2] Pass scoped TTS context into the hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. Because `maybeApplyTtsToPayload` re-resolves auto mode, provider, voice, and prefs from those fields, scoped overrides can pass the gate and then synthesize with the wrong effective config or skip unexpectedly.
  - confidence: 0.91
- **[P3] Isolate TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Because this test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds a new external TTS-provider path from message sends and currently does not preserve dry-run or scoped-provider boundaries.

Concerns:

- **[medium] Dry-run can disclose text to a TTS provider:** `src/infra/outbound/message-action-runner.ts:593`
  - body: The new hook runs before the dry-run send path and does not check `dryRun`, so a no-delivery send can still make an external synthesis request containing the message body.
  - confidence: 0.88

  - confidence: 0.78
