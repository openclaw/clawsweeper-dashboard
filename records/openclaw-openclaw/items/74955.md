## Summary

Keep open. Current main still lacks tagged-TTS handling on the message-tool send path, and the PR targets the right seam, but the patch is not correct as written because the new runtime call drops agent/account TTS context after using that same context for the guard.

## What This Changes

The PR adds tagged-TTS processing to shared outbound `send` actions, with a new regression test and changelog entry.

## Best Possible Solution

Keep the outbound hook approach, but pass the same `agentId` and `accountId` context into `maybeApplyTtsToPayload` that the guard uses, and add a focused regression for agent/account-scoped tagged TTS config before merge.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: configure tagged TTS only through an agent or account override, send `[[tts:text]]...[[/tts:text]]` through `runMessageAction`, and the proposed hook can pass the guard but call the runtime without the context needed to resolve that override.

## Solution Assessment

No, not as written. The hook location and explicit-media short circuit are the right shape, but the runtime call must preserve agent/account TTS context to match the documented and auto-reply behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass TTS context into the runtime hook:** `src/infra/outbound/message-action-runner.ts:602-607`
  - body: The new guard checks `shouldAttemptTtsPayload` with `agentId` and `accountId`, but the `maybeApplyTtsToPayload` call omits both. The TTS runtime resolves `agents.list[].tts` and account overrides from those fields, so a send with tagged TTS enabled only for a specific agent/account can pass the guard and then resolve global TTS as off or use the wrong provider config. Pass the same context into the runtime call and cover a scoped-config case.
  - confidence: 0.9
