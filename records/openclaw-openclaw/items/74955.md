## Summary

Keep open: the reported outbound tagged-TTS bug is source-reproducible and the PR is aimed at the right seam, but the current patch still has blocking correctness/security issues around dry-run side effects and scoped TTS config.

## What This Changes

The PR adds a gated tagged-TTS normalization call to outbound message-tool sends, a regression test, and a changelog entry.

## Best Possible Solution

Land a rebased outbound-send TTS hook that preserves dry-run as side-effect-free, passes the same scoped TTS context as auto-replies, and keeps focused regression coverage independent of local TTS prefs.

## Reproduction Assessment

Yes. Source inspection shows current main sends message-tool text through the outbound send path without the TTS hook, while final auto-replies call the scoped TTS normalizer before delivery.

## Solution Assessment

No, not as written. Reusing the existing TTS hook is the narrow maintainable fix, but this patch must skip dry-run synthesis and pass agent/account context into the runtime call before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip TTS synthesis during dry-run sends:** `src/infra/outbound/message-action-runner.ts:593-597`
  - body: This hook runs before the existing no-delivery dry-run boundary, so `openclaw message send --dry-run` can still send the message body to a configured external TTS provider. Add the dry-run guard before loading or invoking the TTS runtime.
  - confidence: 0.9
- **[P2] Pass scoped TTS context into the hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. Because `maybeApplyTtsToPayload` re-resolves auto mode, provider, voice, and prefs from those fields, scoped overrides can pass the gate and then synthesize with the wrong effective config or skip unexpectedly.
  - confidence: 0.91
- **[P3] Isolate TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Because this test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch can send dry-run or wrongly scoped message text to an external TTS provider before the normal delivery boundary.

Concerns:

- **[medium] Dry-run can disclose text to a TTS provider:** `src/infra/outbound/message-action-runner.ts:593`
  - body: The new hook runs before the dry-run send path and does not check `dryRun`, so a no-delivery send can still make an external synthesis request containing the message body.
  - confidence: 0.88
- **[medium] Scoped provider context is dropped:** `src/infra/outbound/message-action-runner.ts:599`
  - body: The runtime call omits `agentId` and `accountId`, so account- or agent-scoped TTS settings can be bypassed after the gate passes, potentially routing text to a different configured provider than intended.
  - confidence: 0.78
