## Summary

Keep open. The PR addresses a real gap on current main, where message-tool/CLI/gateway send paths still bypass the tagged-TTS hook, but the proposed hook is not yet correct because it drops the agent/account context that TTS config resolution depends on and the new test can be affected by local TTS prefs.

## What This Changes

The PR adds a tagged-TTS pass in `handleSendAction`, a focused outbound regression test, and a changelog entry for message-tool sends honoring `[[tts:text]]...[[/tts:text]]`.

## Best Possible Solution

Land a narrow outbound-send hook that matches the auto-reply TTS contract, including agent/account/channel context and isolated regression coverage, while keeping explicit media behavior unchanged and leaving short tagged speech text tracking to #73911.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: configure tagged TTS and send `[[tts:text]]...[[/tts:text]]` through the message tool, CLI, or gateway send path; current main reaches the normal send path without invoking the TTS runtime hook.

## Solution Assessment

No, not as written. The hook placement and existing-media guard are the right direction, but the runtime call must pass the same scoped TTS context used by auto-reply, and the test should isolate host prefs before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass TTS context into the runtime hook:** `src/infra/outbound/message-action-runner.ts:596-600`
  - body: The new hook gates with `agentId` and `accountId`, but the `maybeApplyTtsToPayload` call itself only passes `cfg`, `channel`, and `kind`. That helper resolves auto mode and provider config from those same context fields, so agent/account overrides can be skipped or synthesized with global settings on the message-tool path. Pass `agentId` and `accountId: accountId ?? undefined` into the runtime call, matching auto-reply.
  - confidence: 0.9
- **[P3] Isolate TTS prefs in the new test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:26-30`
  - body: `shouldAttemptTtsPayload` reads local TTS prefs before `messages.tts.auto`. The new test config does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, so a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook. Point the test config at a test-local nonexistent prefs path or isolate the env.
  - confidence: 0.82
