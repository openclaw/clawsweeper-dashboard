## Summary

Keep open: current main still lacks tagged-TTS normalization for message-tool sends, but this PR has blocking correctness and security issues around dry-run behavior, scoped TTS config, and test isolation.

## What This Changes

The PR adds a tagged-TTS hook to outbound message-tool sends, a regression test file, and a changelog entry.

## Best Possible Solution

Land a rebased outbound-send TTS hook in the current payload-building path that preserves dry-run as side-effect-free, passes scoped TTS context, and keeps focused regression coverage isolated from local prefs.

## Reproduction Assessment

Yes. Source inspection on current main shows message-tool sends build outbound payloads without the TTS hook, while auto-replies call the scoped TTS normalizer before delivery.

## Solution Assessment

No, not as written. Reusing the existing TTS hook is the right direction, but this branch must skip dry-run synthesis, pass agent/account context into the runtime call, isolate TTS prefs in the test, and port cleanly to current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip TTS synthesis during dry-run sends:** `src/infra/outbound/message-action-runner.ts:593-597`
  - body: This hook runs before the existing no-delivery dry-run boundary, so `openclaw message send --dry-run` can still send the message body to a configured external TTS provider. Gate the whole block on `!dryRun` before loading or invoking the TTS runtime.
  - confidence: 0.9
- **[P2] Pass scoped TTS context into the hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. Because `maybeApplyTtsToPayload` re-resolves auto mode, provider, voice, and prefs from those fields, scoped overrides can pass the gate and then synthesize with the wrong effective config or skip unexpectedly.
  - confidence: 0.91
- **[P3] Isolate TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Because this test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff opens a new external TTS-provider path from message sends without preserving dry-run and scoped-provider boundaries.

Concerns:

- **[medium] Dry-run can disclose text to a TTS provider:** `src/infra/outbound/message-action-runner.ts:593`
  - body: The new hook runs before the dry-run send path and does not check `dryRun`, so a no-delivery send can still make an external synthesis request containing the message body.
  - confidence: 0.88

  - confidence: 0.78
