## Summary

Keep open. The PR targets a real current-main gap in the message-tool send path, but the proposed patch is not merge-ready because it gates TTS with scoped agent/account context and then calls the runtime hook without that same context, plus the new regression test can be affected by host-local TTS prefs.

## What This Changes

The PR adds a tagged-TTS pass to outbound message-tool sends, a focused regression test, and a changelog entry for synthesizing `[[tts:text]]...[[/tts:text]]` markup from `mcp__openclaw__message`/CLI/gateway sends.

## Best Possible Solution

Land a narrow outbound-send hook that mirrors the auto-reply TTS contract: run the lazy TTS runtime before delivery when no explicit media is present, pass the same agent/account/channel context into both the gate and runtime call, isolate the new test from host-local prefs, and leave short explicit tagged text behavior to the separate TTS short-text work.

## Reproduction Assessment

Yes. Configure tagged TTS and send `[[tts:text]]...[[/tts:text]]` through the message-tool/CLI/gateway send path; current main reaches outbound send without invoking the TTS runtime hook, while the auto-reply path does invoke it.

## Solution Assessment

No, not as written. The hook placement and explicit-media guard are the right shape, but the runtime call must receive `agentId` and `accountId`, and the test must isolate local TTS prefs before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass scoped TTS context to the runtime hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The new gate checks `shouldAttemptTtsPayload` with `agentId` and `accountId`, but the subsequent `maybeApplyTtsToPayload` call drops both fields. Scoped agent/account configs can therefore pass the gate and then be skipped or synthesized with global provider/voice settings in the runtime. Pass `agentId` and `accountId: accountId ?? undefined` into the hook, matching auto-reply.
  - confidence: 0.91
- **[P3] Isolate local TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads local prefs before `messages.tts.auto`, but the new test config does not set `prefsPath` or isolate `OPENCLAW_TTS_PREFS`. A developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook. Point the test at a test-local nonexistent prefs path or isolate the env.
  - confidence: 0.84
