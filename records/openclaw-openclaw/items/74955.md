## Summary

Keep open. The PR targets a real current-main outbound/TTS gap, but the patch still needs narrow fixes: the new hook calls the TTS runtime without the context it used for gating, can synthesize audio during dry runs, and leaves the regression test sensitive to local TTS prefs.

## What This Changes

The PR adds a tagged-TTS pass in `handleSendAction`, a focused outbound regression test, and a changelog entry for message-tool sends honoring `[[tts:text]]...[[/tts:text]]`.

## Best Possible Solution

Land a narrow outbound-send hook that skips dry-run sends, runs only when no explicit media is present, passes the same agent/account/channel context into the TTS runtime as the gate, preserves current `asVoice` behavior, and isolates the regression test from host prefs.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: configure tagged TTS, send `[[tts:text]]...[[/tts:text]]` through the message-tool/CLI/gateway send path, and current main reaches outbound send without invoking the TTS runtime hook.

## Solution Assessment

No, not as written. The hook belongs in this path, but it must pass scoped TTS context, avoid dry-run side effects, and make the test deterministic before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass TTS context into the runtime hook:** `src/infra/outbound/message-action-runner.ts:599-604`
  - body: The gate resolves TTS with `agentId` and `accountId`, but the runtime call omits both. Because `maybeApplyTtsToPayload` resolves auto mode, provider, voice, and prefs from those same fields, agent/account overrides can pass the gate and then synthesize with the wrong effective config. Pass `agentId` and `accountId: accountId ?? undefined` into this call.
  - confidence: 0.91
- **[P2] Preserve dry-run sends by skipping synthesis:** `src/infra/outbound/message-action-runner.ts:593-597`
  - body: This hook runs before `sendMessage` reaches its dry-run return, so `openclaw message send --dry-run` can still call a configured TTS provider and create an audio artifact. Add `!dryRun` to this gate, or otherwise keep dry-run as a planning-only path with no provider side effects.
  - confidence: 0.84
- **[P3] Isolate local TTS prefs in the regression test:** `src/infra/outbound/message-action-runner.tagged-tts.test.ts:20-30`
  - body: `shouldAttemptTtsPayload` reads the local TTS prefs path before `messages.tts.auto`. Since this test does not pin `prefsPath` or isolate `OPENCLAW_TTS_PREFS`, a developer or CI image with local prefs set to `off` can make the tagged case skip the mocked hook. Use a test-local nonexistent prefs path or isolate the env.
  - confidence: 0.82
