## Summary

Keep open: this maintainer-labeled draft still contains a unique provider-discovery guard that is not on current main or the latest release, but it needs a short fallback-policy comment, focused validation, and coordination with the adjacent provider-discovery metadata PR before merge.

## What This Changes

The PR wraps provider-discovery credential metadata probing in a guard and adds a regression for poisoned `setup` and `providerAuthEnvVars` manifest metadata while preserving later env-backed plugin fallback selection.

## Best Possible Solution

Land the narrow helper guard after adding the fallback-policy comment, running the focused provider-discovery regression, and coordinating the metadata-guard series so malformed plugin rows skip consistently without broadening runtime loading.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main and `v2026.6.8` dereference `setup` and `providerAuthEnvVars` during selective fallback selection, so a throwing accessor can abort before a later healthy credentialed plugin is considered.

## Solution Assessment

Is this the best way to solve the issue? Yes, with a small maintainability follow-up: guarding `hasProviderAuthEnvCredential` is the narrow owner-boundary fix for selective `onlyPluginIds` fallback selection, but the deliberate no-credential skip policy should be documented and coordinated with the adjacent metadata guard.
