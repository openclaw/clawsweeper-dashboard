## Summary

Keep open: current main still has the unguarded credential metadata read, and this maintainer-labeled draft contains a focused useful fix rather than stale or superseded work. It needs maintainer validation, the requested fallback-policy comment, and coordination with the adjacent provider-discovery metadata draft before landing.

## What This Changes

The PR wraps provider-discovery credential metadata aggregation in a guard and adds a regression where poisoned credential metadata is skipped while a later healthy env-backed plugin remains eligible for scoped full loading.

## Best Possible Solution

Land a coordinated provider-discovery metadata-hardening path after focused regression proof, keeping the helper-level guard and documenting the deliberate no-credential-evidence fallback.

## Reproduction Assessment

Yes from source inspection, though I did not execute the failing path. Current main dereferences `setup` and `providerAuthEnvVars` in selective fallback selection without a guard, so a throwing accessor can abort before a later healthy credentialed plugin is considered.

## Solution Assessment

Yes, with one small maintainability follow-up. Guarding the credential helper is the narrow owner-boundary fix for the scoped full-load decision, but the deliberate no-credential fallback should be documented and coordinated with the adjacent metadata guard.
