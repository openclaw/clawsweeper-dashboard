## Summary

Keep open: current main still reads provider-discovery credential metadata without a guard, and this maintainer draft is a focused useful fix, but it needs normal maintainer validation, focused test proof, and coordination with the adjacent provider-discovery hardening PR before merge.

## What This Changes

The branch guards provider-discovery credential metadata aggregation so unreadable `setup` or `providerAuthEnvVars` metadata counts as no env credential evidence, and adds a poisoned-metadata regression that preserves a later healthy env-backed fallback plugin.

## Best Possible Solution

Land a coordinated provider-discovery metadata-hardening path after focused regression proof, keeping the helper-level guard and documenting the deliberate no-credential-evidence fallback.

## Reproduction Assessment

Yes. Source inspection shows current main reads `setup` and `providerAuthEnvVars` inside scoped fallback selection without a guard; a poisoned accessor on either field would throw before healthy later credentialed plugins can be selected.

## Solution Assessment

Yes, with one small maintainability follow-up. Guarding the credential helper is the narrow owner-boundary fix because `resolveSelectiveFullPluginIds` is its only caller, while the model-catalog sibling case is already covered by the related PR.
