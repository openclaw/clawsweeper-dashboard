## Summary

Keep open: this maintainer-labeled draft is still useful because current main and the latest release still read provider credential metadata without a guard, while the PR is a narrow fix with no blocking correctness or security finding found. It needs maintainer validation and coordination with the adjacent provider-discovery metadata guard before merge.

## What This Changes

The PR wraps provider-discovery credential metadata probing in a guard and adds a regression for poisoned `setup` and `providerAuthEnvVars` metadata while preserving later healthy env-backed plugin fallback selection.

## Best Possible Solution

Land the narrow credential-metadata guard after focused provider-discovery test proof, coordinated with https://github.com/openclaw/openclaw/pull/89945 so provider-discovery metadata failures have one consistent policy.

## Reproduction Assessment

Yes from source inspection: current main calls `hasProviderAuthEnvCredential` during selective full-plugin fallback and directly dereferences `plugin.setup?.providers` plus `plugin.providerAuthEnvVars`, so a throwing accessor aborts provider discovery. I did not execute the repro in this read-only review.

## Solution Assessment

Yes: guarding inside `hasProviderAuthEnvCredential` is the narrowest surface for treating unreadable credential metadata as no env evidence while preserving later healthy plugin fallback selection. The broader model-catalog metadata guard is complementary work in https://github.com/openclaw/openclaw/pull/89945 and should be coordinated rather than duplicated here.
