## Summary

## What This Changes

The PR wraps provider-discovery credential metadata reads in a guard and adds a regression for skipping poisoned `setup`/`providerAuthEnvVars` rows while still full-loading a later env-backed provider plugin.

## Best Possible Solution

Land the narrow helper-level guard only after maintainers accept the no-credential-evidence fallback policy, add the explanatory catch comment, and rerun focused provider-discovery proof.

## Reproduction Assessment

Yes. From source inspection, current main dereferences `setup` and `providerAuthEnvVars` in the selective fallback path, so a throwing accessor can abort before later healthy credentialed plugins are considered.

## Solution Assessment

Yes, with one maintainability follow-up. Guarding the credential helper is the narrow owner-boundary fix; scattering guards at callers would duplicate policy, but the deliberate no-credential fallback should be documented.
