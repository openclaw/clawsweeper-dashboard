## What This Changes

The PR wraps provider-discovery credential metadata aggregation in a guard and adds a regression where poisoned credential metadata is skipped while a later healthy env-backed plugin remains eligible for scoped full loading.

## Best Possible Solution

Land the helper-level guard after the catch documents the no-credential-evidence fallback, maintainers accept that compatibility behavior, and the focused provider-discovery regression passes.

## Reproduction Assessment

Yes, from source inspection. Current main dereferences `setup` and `providerAuthEnvVars` in the selective fallback path, so a throwing accessor can abort before later healthy credentialed plugins are considered.

## Solution Assessment

Yes, with one maintainability follow-up. Guarding the credential helper is the narrow owner-boundary fix; scattering guards at callers would duplicate policy, but the deliberate no-credential fallback should be documented.
