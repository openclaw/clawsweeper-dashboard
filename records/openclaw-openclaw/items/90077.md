## Summary

Keep open: current main and the latest release still have the unguarded credential metadata read, and this maintainer draft is a focused fix, but it remains protected maintainer work that needs the fallback-policy comment, focused validation, and coordination with the adjacent provider-discovery metadata draft before landing.

## What This Changes

The branch wraps provider-discovery credential metadata reads in a guard that treats unreadable `setup` or `providerAuthEnvVars` as no env credential evidence and adds a poisoned-metadata regression test.

## Best Possible Solution

Land the helper-level guard after adding the short fallback-policy comment, running the focused provider-discovery regression, and coordinating with the adjacent metadata guard so malformed plugin rows skip consistently without broadening runtime loading.

## Reproduction Assessment

Yes from source inspection, though I did not execute a failing run. Current main and `v2026.6.8` dereference `setup` and `providerAuthEnvVars` during selective fallback selection, so a throwing accessor can abort before a later healthy credentialed plugin is considered.

## Solution Assessment

Yes, with a small maintainability follow-up. Guarding `hasProviderAuthEnvCredential` is the narrow owner-boundary fix because that helper feeds the selective `onlyPluginIds` fallback decision, but the deliberate no-credential fallback should be documented and coordinated with the adjacent metadata guard.
