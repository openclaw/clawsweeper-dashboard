## Label Justifications

- `P2`: The RFC addresses a real cron/yield workflow gap with limited blast radius, but it is a product/API decision rather than an urgent regression fix.
- `impact:session-state`: The requested contract controls whether a yielded parent session receives state needed to resume after an isolated cron run terminates.
- `impact:message-loss`: Without an explicit completion wake, the parent-visible terminal notification for the cron wait can be absent even though the cron run completed.

## Summary

Keep open: current main still exposes isolated cron completion through run history, delivery state, and silent task records rather than an opt-in parent-session wake contract. The request would add new cron/session API and config surface, so it needs maintainer product direction before implementation.

## What This Changes

The issue requests an opt-in cron completion notification surface so a parent session waiting after an isolated cron run can wake when that specific cron finishes.

## Best Possible Solution

Keep isolated cron completion silent and pollable by default; if maintainers accept the RFC, add a narrow explicit notification target with schema, docs, and focused session-delivery tests.

## Reproduction Assessment

Yes for the current behavior: source and docs show isolated cron completion is poll/log/delivery state, while sessions_yield does not wait for cron completion. This is not a strict bug reproduction because the requested change is a new opt-in product contract.

## Solution Assessment

Unclear until maintainers choose the surface. The RFC's explicit opt-in target is safer than always-on parent fan-out, but current docs already define polling as the supported cron completion path.
