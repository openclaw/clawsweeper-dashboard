## Label Justifications

- `P2`: The RFC addresses a real cron/yield workflow gap with limited blast radius, but it is a product/API decision rather than an urgent shipped regression fix.
- `impact:session-state`: The requested behavior controls whether a yielded parent session receives state needed to resume after an isolated cron run terminates.
- `impact:message-loss`: Without an explicit completion wake, the parent-visible terminal notification for the cron wait can be absent even though the cron run completed.

## Summary

Keep open: current main and v2026.6.10 still lack an isolated-cron completion parent-wake contract, and the requested opt-in cron/session API needs maintainer product review rather than an automated bug-fix branch.

## What This Changes

Request an opt-in `cron.jobs[].notifyOnCompletion` or per-run notify target so a parent waiting after an isolated cron run can receive a bounded completion wake.

## Best Possible Solution

Keep isolated cron completion silent/polled by default, and add only a narrow explicit completion target if maintainers accept the new cron/session contract.

## Reproduction Assessment

Not applicable as a feature RFC; source and docs confirm the current intentional behavior is polling/logged cron completion plus silent detached task records, not a parent wake contract.

## Solution Assessment

Unclear until maintainers choose the surface; an explicit opt-in target is safer than always-on parent fan-out, but the permanent API boundary is still a product decision.
