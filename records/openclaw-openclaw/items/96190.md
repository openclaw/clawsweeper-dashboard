## Label Justifications

- `P2`: The RFC covers a real cron/yield workflow gap with limited blast radius, but it is a product/API decision rather than an urgent regression fix.
- `impact:session-state`: The requested behavior controls whether a yielded parent session receives state needed to resume after an isolated cron run terminates.
- `impact:message-loss`: Without an explicit completion wake, the parent-visible terminal notification for the cron wait can be absent even though the cron run completed.

## Summary

Keep open: current main and v2026.6.10 still expose isolated cron completion through run logs, delivery state, and polling, while the requested parent wake would add a new opt-in cron/session delivery contract that needs maintainer product and API review.

## What This Changes

The issue requests an opt-in `cron.jobs[].notifyOnCompletion` or per-spawn notification target so a parent session can wake after a specific isolated cron run completes.

## Best Possible Solution

Keep isolated cron completion silent and pollable by default, and add a narrow explicit notification target only if maintainers accept the new cron/session contract.

## Reproduction Assessment

Yes, source-level: current main keeps `cron.run` completion semantics polling-based and detached cron completion does not enqueue or heartbeat a parent session. This is not a strict bug reproduction because the issue is framed as an RFC for a new explicit contract.

## Solution Assessment

Unclear until maintainers choose the surface. The proposed explicit opt-in target is safer than always-on parent fan-out, but the existing supported cron completion path remains `cron.runs` polling or CLI `--wait`.
