## Label Justifications

- `P2`: The RFC addresses a real cron/yield workflow gap with limited blast radius, but it is a product/API decision rather than an urgent runtime regression fix.
- `impact:session-state`: The requested behavior controls whether a yielded parent session receives state needed to resume after an isolated cron run terminates.
- `impact:message-loss`: Without an explicit completion wake, the parent-visible terminal notification for the cron wait can be absent even though the cron run completed.

## Summary

Keep open: current main and the latest release still expose isolated cron completion through run history/delivery state, while the requested parent wake would add a new opt-in cron/session delivery contract that needs maintainer product and API review.

## What This Changes

Request an RFC decision on an opt-in cron completion notification flag or target so a yielded parent session can be woken after a specific isolated cron run completes.

## Best Possible Solution

Keep isolated cron completion silent and pollable by default, and add only a narrow explicit completion target or wait API if maintainers accept that cron/session contract.

## Reproduction Assessment

Not applicable as a feature RFC. Source confirms the current behavior: cron completion is exposed through run history/delivery state, detached isolated jobs do not enqueue a parent wake, and `sessions_yield` has no cron wait selector.

## Solution Assessment

Unclear until maintainers choose the API. An explicit opt-in target is safer than always-on parent fan-out, but the permanent shape could also be a `cron.runs` waiter or `sessions_yield` selector rather than a cron job config flag.
