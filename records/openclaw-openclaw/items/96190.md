## Label Justifications

- `P2`: The RFC addresses a real but bounded cron/session workflow gap and requires product/API design rather than urgent runtime repair.
- `impact:session-state`: The affected behavior is whether a yielded parent session remains parked or receives an explicit completion wake for an isolated cron run.

## Summary

Keep open: this is a real, unimplemented product/API decision for isolated cron completion wake semantics, not a stale bug report and not already covered by current `main`. The older cron-yield issue tracks the observable gap and docs/workaround path, while this issue is the dedicated RFC for the opt-in runtime contract.

## What This Changes

Requests an opt-in cron job or ad-hoc spawn completion notification surface that wakes a named parent session after an isolated cron run finishes.

## Best Possible Solution

Make a maintainer product decision first: either keep cron completion waits on `cron.runs`/CLI polling with clearer tool guidance, or design a narrow explicit completion-notification contract with default-false behavior and upgrade-safe docs/tests.

## Reproduction Assessment

Yes, at source level. Current `sessions_yield` requires a later wake event, detached cron execution does not use the main-session enqueue/heartbeat path, and isolated delivery awareness is guarded away from default main-session mirroring.

## Solution Assessment

Unclear as an implementation choice. `notifyOnCompletion` is a plausible opt-in design, but maintainers should compare it with the existing `cron.runs`/CLI `--wait` path and a narrower docs/tool-contract update before adding config and protocol surface.
