## Label Justifications

- `P2`: The RFC covers a real cron/yield workflow gap with limited blast radius, but it is a product/API decision rather than an urgent regression fix.
- `impact:session-state`: The requested contract controls whether a yielded parent session receives state needed to resume after an isolated cron run terminates.
- `impact:message-loss`: Without an explicit completion wake, the parent-visible terminal notification for a cron wait can be absent even though the cron run completed.

## Summary

Keep open: current main still has no `notifyOnCompletion` or equivalent isolated-cron parent wake contract, and the requested change would add new cron/session API plus config surface that needs maintainer product direction.

## What This Changes

Request an opt-in cron job or per-spawn completion notification surface that wakes a named parent session after an isolated cron run finishes.

## Best Possible Solution

Keep isolated cron completion silent and pollable by default; if maintainers accept the RFC, add a narrow explicit notification target with schema, protocol/CLI/docs coverage, and focused session-delivery tests.

## Reproduction Assessment

Yes for the current behavior: source and docs show isolated cron completion is poll/log/delivery state, while sessions_yield has no cron completion wait selector. This is not a strict bug reproduction because the requested outcome is a new opt-in product contract.

## Solution Assessment

Unclear until maintainers choose the surface. The RFC's explicit opt-in target is safer than always-on parent fan-out, but current docs already define polling as the supported cron completion path.
