## Label Justifications

- `P2`: The issue addresses a real cron/yield workflow gap with limited blast radius, but it is a product/API decision rather than an urgent regression fix.
- `impact:session-state`: The requested contract controls whether a yielded parent session receives state needed to resume after an isolated cron run terminates.
- `impact:message-loss`: Without an explicit completion wake, the parent-visible terminal notification for the cron wait can be absent even though the cron run completed.

## Summary

Keep open: current main and v2026.6.10 still lack an opt-in cron completion notification target for waking a yielded parent session after an isolated cron run. Existing cron completion semantics are polling/run-history, webhook/completion delivery, or explicitly gated awareness, so this needs maintainer product/API direction before implementation.

## What This Changes

The issue requests an opt-in cron completion notification surface so a parent session waiting after an isolated cron run can be woken explicitly.

## Best Possible Solution

Keep isolated cron completion silent and pollable by default; if maintainers accept the RFC, add a narrow explicit notification target with schema, docs, and focused session-delivery tests.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron wait selector on sessions_yield, and detached cron execution does not enqueue a parent-session wake on completion. This is not a strict bug reproduction because the issue requests a new opt-in product contract.

## Solution Assessment

Unclear until maintainers choose the public surface. The RFC's explicit opt-in target is safer than always-on parent fan-out, but existing cron completion waits are currently polling-based and the new API/config contract needs deliberate design.
