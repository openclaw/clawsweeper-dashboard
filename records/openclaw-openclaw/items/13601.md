## Summary

Keep open. Current main implements the durable JSONL-backed cron run-history subset, but the central requested product surface still lacks timestamp/range filters, richer `cron logs` or `cron history` CLI behavior, follow/export options, success-rate aggregates, and day/manual retention controls.

## What This Changes

Request to add persistent cron execution history with filtering, CLI log/history access, aggregates, export/follow, and retention/prune controls.

## Best Possible Solution

Scope an additive cron-history design on top of the existing JSONL-backed `cron.runs` model, then add timestamp filters, CLI/API/UI/docs coverage, aggregates, and explicit retention semantics while preserving access-control and bounded-read constraints.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement request. Source inspection on current main gives high confidence that durable run history exists but the requested time filters, richer CLI/history commands, aggregates, export/follow behavior, and day/manual retention controls remain absent.

## Solution Assessment

Unclear for the full proposal. Extending the existing JSONL-backed `cron.runs` contract with additive timestamp filters is the narrowest likely first step, while CLI shape, aggregates, follow/export behavior, and retention policy need maintainer scoping.
