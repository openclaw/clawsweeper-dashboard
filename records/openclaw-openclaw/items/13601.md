## Summary

Keep open. Current main has durable JSONL cron run history, but it still lacks the requested time-window filters, richer `cron logs`/`cron history` CLI, success/failure aggregates, export/follow behavior, and manual retention/prune/vacuum workflow.

## What This Changes

Request persistent cron execution history with filtering, export/follow, aggregates, and retention/pruning controls.

## Best Possible Solution

Extend the existing JSONL-backed history path with additive timestamp filters and documented API/CLI/UI behavior, then separately decide whether aliases, aggregates, export/follow, and manual retention controls belong in core.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives a high-confidence verification path: durable run history exists, while the requested filters, commands, aggregates, and retention workflow remain absent.

## Solution Assessment

Unclear for the full proposal. Persistence is already solved with JSONL, so the narrowest maintainable direction is additive filtering/API/CLI/docs work, with SQLite/manual vacuum treated as a separate product decision.
