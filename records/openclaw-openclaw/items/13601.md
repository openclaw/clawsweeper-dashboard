## Summary

Keep open. Current main has durable JSONL-backed cron run history and several filters, but the central requested product surface is still missing: timestamp/range filters, a richer `cron logs` or `cron history` CLI, follow/export behavior, success-rate aggregates, and explicit day/manual retention controls.

## What This Changes

The issue requests persistent cron execution history with status/time filtering, CLI log/history commands, JSON export, success-rate summaries, and retention/pruning controls.

## Best Possible Solution

Scope an additive cron-history design on top of the existing JSONL-backed `cron.runs` model, then add timestamp filters, CLI/API/UI/docs coverage, aggregates, and explicit retention semantics while preserving access-control and bounded-read constraints.

Not applicable as a bug reproduction: this is an enhancement request. Source inspection on current main gives high confidence that durable run history exists but timestamp filters, richer CLI/history commands, aggregates, export/follow behavior, and day/manual retention controls remain absent.

## Solution Assessment
