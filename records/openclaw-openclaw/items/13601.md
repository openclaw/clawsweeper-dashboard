## Summary

Keep open. Current main covers the durable JSONL-backed run-history subset, but it still lacks the central requested timestamp filters, richer `cron logs`/`cron history` CLI, follow/export workflow, success-rate aggregates, and explicit history retention/prune/vacuum controls.

## What This Changes

Request persistent cron execution history with status/time filtering, CLI log/export/follow commands, run aggregates, and retention/prune controls.

## Best Possible Solution

Scope an additive cron-history design on the existing `cron.runs` model, then add timestamp filters, CLI/API/UI/docs coverage, aggregates, and explicit retention semantics while preserving access-control and bounded-read constraints.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement request. Source inspection on current main gives high confidence that durable history exists but the requested time filters, richer CLI/history commands, aggregates, export/follow behavior, and manual/day-based retention controls remain absent.

## Solution Assessment

Unclear for the full proposal. Extending the existing JSONL-backed `cron.runs` contract with additive timestamp filters is the narrowest likely first step, while SQLite, vacuum/prune commands, aggregates, follow/export behavior, and retention policy need maintainer scoping.
