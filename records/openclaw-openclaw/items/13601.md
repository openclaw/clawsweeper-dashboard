## Summary

Keep open. Current main has durable JSONL-backed cron run history and status/delivery/query filtering, but the requested product surface still has substantial missing pieces: time/range filters, `cron logs`/`cron history` commands, follow/export behavior, success-rate aggregates, and explicit retention/prune workflow.

## What This Changes

Request persistent cron execution history with richer CLI filtering, export/follow, aggregates, and retention controls.

## Best Possible Solution

Scope an additive cron-history design on top of the existing JSONL-backed `cron.runs` model, then add timestamp filters, CLI/API/UI/docs coverage, aggregates, export/follow behavior, and explicit retention semantics without weakening bounded reads or access controls.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement request. Source inspection on current main gives high confidence that durable run history exists but timestamp filters, richer CLI/history commands, aggregates, export/follow behavior, and day/manual retention controls remain absent.

## Solution Assessment

Unclear for the full proposal. Extending the existing JSONL-backed `cron.runs` contract with additive timestamp filters looks like the narrowest first step, while CLI shape, aggregates, follow/export behavior, and retention policy need maintainer scoping.
