## Summary

Keep open. Current main has durable JSONL cron run history with status, delivery, query, pagination, sort, UI filtering, and automatic size/line pruning, but it still lacks the central requested time-window filters, richer `cron logs`/`cron history` CLI, success/failure aggregates, export/follow behavior, and manual retention/prune/vacuum workflow.

## What This Changes

Add persistent cron execution history with status and time filtering, CLI log/history commands, JSON export, live follow, list aggregates, and retention controls.

## Best Possible Solution

Keep this open until maintainers either implement the remaining additive JSONL-backed history surface or explicitly narrow the request; timestamp filters should flow through the run-log reader, protocol schema, Gateway handler, UI state, CLI/docs, and focused regression tests before considering a storage migration.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence source verification path: read the run-log options/filtering, protocol schema, Gateway handler, CLI registration, UI controls, and cron docs on current main to confirm which requested surfaces are still missing.

## Solution Assessment

Unclear for the full proposal. Persistence is already solved with JSONL, so the narrowest maintainable path is additive filtering, CLI/API/UI summaries, and documented retention semantics; SQLite/manual vacuum should be a separate product decision if JSONL cannot meet scale needs.
