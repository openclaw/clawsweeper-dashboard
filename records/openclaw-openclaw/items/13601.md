## Summary

Keep open. Current main has durable JSONL-backed cron run history plus status, delivery, query, sort, run-id, and pagination filters, but the central requested product surface is still missing timestamp/range filters, richer `cron logs` or `cron history` CLI behavior, follow/export options, success-rate aggregates, and day/manual retention controls.

## What This Changes

Request to add durable cron execution history with status/time filtering, JSON export or follow mode, reliability aggregates, and retention/prune controls.

## Best Possible Solution

Scope an additive cron-history design on top of the existing JSONL-backed `cron.runs` model, then add timestamp filters, CLI/API/UI/docs coverage, aggregates, and explicit retention semantics while preserving access-control and bounded-read constraints.

Not applicable as a bug reproduction: this is an enhancement request.
