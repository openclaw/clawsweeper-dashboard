## Summary

Keep open. Current main has durable JSONL-backed cron run history plus status, delivery, query, pagination, sorting, and UI filtering, but it still lacks the central requested CLI/API product surface: timestamp range filters, `cron logs` or `cron history` commands with status/time/follow/export options, success-rate/failure aggregates in `cron list`, and manual history retention/prune/vacuum workflow. Related #16799 and #16831 are status UX work and do not supersede this request.

## What This Changes

Request persistent cron execution history with filtering, CLI log/history commands, success-rate summaries, and retention/pruning controls.

## Best Possible Solution

Keep this open until the remaining cron history surface is either implemented or explicitly narrowed. A maintainable path would add timestamp window filters to the run-log reader, protocol schema, Gateway handler, UI state, and tests; extend the CLI with a documented `cron logs` or richer `cron runs` command for status/time/export/follow behavior; add a dedicated summary or `cron list` aggregate path; and document whether manual prune/vacuum/history-retention-days is in core scope or intentionally out of scope.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence source verification path. Current main can be checked by reading `ReadCronRunLogPageOptions`, `CronRunsParamsSchema`, the `cron.runs` handler, the CLI registration, and cron docs to confirm the missing timestamp, CLI, aggregate, and manual retention surfaces.

## Solution Assessment

Unclear for the whole proposal because part of the requested persistence is already solved with JSONL, while the remaining CLI/API/UI/retention scope needs maintainer product judgment. The narrowest likely implementation is additive JSONL-backed filters and summaries before considering a SQLite migration unless performance requirements demand it.
