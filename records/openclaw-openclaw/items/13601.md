## Summary

Keep open. Current main has durable JSONL-backed cron run history and status/query/delivery filtering, but still lacks the requested time-range filters, richer `cron logs` or `cron history` CLI surface, follow/export behavior, success-rate aggregates, and manual/day-based retention controls.

## What This Changes

Add persistent cron execution history with filtering, CLI log/history commands, aggregate status reporting, and retention controls.

## Best Possible Solution

Keep this issue open until maintainers scope an additive cron history design on the existing JSONL-backed run-log model, including timestamp filters, CLI/API/docs coverage, aggregates, and explicit retention semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement request. Source inspection on current main verifies partial durable history exists while the requested time filters, richer CLI/history commands, aggregates, export/follow behavior, and manual/day-based retention controls remain absent.

## Solution Assessment

Unclear for the full proposal. Extending the existing JSONL-backed `cron.runs` contract with additive timestamp filters is the narrowest likely first step, while SQLite, vacuum/prune commands, aggregates, follow/export behavior, and retention policy need maintainer scoping.
