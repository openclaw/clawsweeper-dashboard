## Summary

The request is partially implemented on current main: cron run history is durable in SQLite, automatically pruned, and filterable through the Gateway/UI, but the central CLI time-range/audit workflow, aggregates, follow behavior, and manual maintenance remain absent. Keep it open for a maintainer product/security decision rather than queueing one broad automated implementation.

## What This Changes

Add durable cron execution history with status and time filtering, CLI export/follow support, reliability aggregates, and retention-management controls.

## Best Possible Solution

Keep the SQLite-backed history and automatic row retention, then scope approved follow-ups: expose existing status/query/sort filters in the CLI, add timestamp bounds through the run-log/API/UI/CLI contract, and separately decide whether aggregates, live follow, output export/redaction, and manual database maintenance belong in core.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken established behavior. Current source and docs provide a high-confidence inventory of the implemented and missing capabilities.

## Solution Assessment

No, not as one monolithic change. SQLite and bounded automatic retention are the correct foundation, but the remaining filter, export, follow, aggregate, and maintenance features should be split around explicit product and sensitive-data policy decisions.
