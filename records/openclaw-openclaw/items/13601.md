## Summary

Current main and v2026.6.5 provide durable SQLite-backed cron run history, automatic per-job pruning, and API/UI filtering by status, delivery, query, and sort order. The issue remains necessary because the central user-facing workflow is incomplete: the CLI exposes only id/run-id/limit, timestamp-range filters do not exist, and success-rate summaries, follow mode, and manual retention administration remain absent; the broad remainder requires maintainer product and security decisions.

## What This Changes

Add persistent cron execution history with status/time filtering, export and follow workflows, reliability summaries, and retention administration.

## Best Possible Solution

Keep the shipped SQLite store as the single canonical history source, then approve a narrow first increment that adds timestamp bounds and ergonomic status/query flags end-to-end across the RPC, CLI, UI, tests, and docs; track aggregates, follow/export behavior, and manual maintenance separately after defining sensitive-output policy.

## Reproduction Assessment

Not applicable: this is a multi-part request for new CLI, query, reporting, export, and retention capabilities rather than a broken documented contract. Source and release inspection establish which portions already exist.

## Solution Assessment

No, the original all-in-one proposal is not the narrowest maintainable solution. The existing SQLite-backed path is correct, but the remaining capabilities should be split into security-reviewed, end-to-end increments rather than adding JSON files or a parallel history system.
