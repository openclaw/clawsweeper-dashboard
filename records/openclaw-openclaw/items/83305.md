## What This Changes

The PR adds a pull_request CI job that checks out the repository and runs a new Node regression-detector gate script with tests for bug-fix evidence and trading-sensitive approval.

## Best Possible Solution

Make this a trusted base-branch metadata gate with focused bypass tests, then define or reuse a maintained approval label before fail-closing trading-sensitive fixes.

## Reproduction Assessment

Yes, by source inspection for the blocking findings: the new pull_request job uses default checkout behavior and runs policy code from the PR merge ref, while the required approval label is absent from the live repository. I did not run tests in this read-only review.

## Solution Assessment

No. A metadata enforcement gate should run trusted base code and use a documented approval signal; this PR still runs PR-controlled policy code and adds an undefined fail-closed label path.
