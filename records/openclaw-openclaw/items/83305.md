## Summary

Keep open: the direction is useful, but the patch should not merge because the new policy gate runs branch-controlled code and its trading-sensitive detector would match the default PR template's risk section while requiring a label that does not exist.

## What This Changes

The PR adds a CI job and Node script that classify bug-fix PRs, require reproduction/verification/regression evidence in the PR body, and require a `risk-officer-approved` label for trading-sensitive fixes.

## Best Possible Solution

Implement this as a trusted metadata-only gate using base-branch code, define or reuse an approval-label contract explicitly, and keep the market-sensitive matcher narrow enough not to collide with the default PR template.

## Reproduction Assessment

Yes for the review findings by source inspection: the new script matches the template's `Risks and Mitigations` heading via the bare `risk` term, and the workflow runs the gate after checking out PR-controlled code. I did not run CI because this was a read-only review.

## Solution Assessment

No. A PR metadata enforcement gate should run trusted base code and use a narrow, documented approval signal; the current implementation is both bypassable and overbroad.

## Review Findings

  - body: This metadata gate checks out the PR merge ref and then runs `scripts/check-regression-detector-gate.mjs` from that checkout. A PR that omits evidence can change this script or workflow to exit 0, so the gate does not enforce the policy it is meant to enforce. Please follow the existing real-behavior-proof pattern: run trusted base code while reading the PR event payload.

  - confidence: 0.92
