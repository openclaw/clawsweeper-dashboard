## Summary

Keep open. The PR targets a real CI policy gap and includes usable terminal proof, but it is not merge-ready because the new enforcement gate can be controlled by the PR being checked and the approval matcher can block ordinary bug-fix PRs.

## What This Changes

The PR adds a CI job and `scripts/check-regression-detector-gate.mjs` to require bug-fix evidence and a `risk-officer-approved` label for detected trading-sensitive fixes.

## Best Possible Solution

Make this a trusted base-branch metadata gate with focused tests for bypass and template-collision cases, plus an explicit maintainer-owned approval-label contract.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the workflow runs the policy script from the PR checkout, and the repository PR template contains `Risk` text that the regex matches. I did not run CI in this read-only review.

## Solution Assessment

No. A metadata enforcement gate should run trusted base code and use a narrow documented approval signal; this PR currently uses PR-controlled code and an undefined broad approval policy.
