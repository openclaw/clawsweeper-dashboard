## Summary

Keep open. Current main and the latest release still resolve the default TUI Gateway URL from config/env/defaults without reading the active Gateway runtime port, and an open linked PR remains the focused fix path.

## What This Changes

The issue asks for `openclaw tui` to use the active Gateway runtime port, such as `48789`, instead of falling back to `18789` when no `--url` is passed.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with lock-backed port metadata consumed only for default local TUI resolution, preserving explicit `--url` and remote config precedence.

## Reproduction Assessment

Yes. The issue gives concrete `openclaw gateway --port 48789` then `openclaw tui` steps, and current-main source still resolves the default local URL without active runtime-port metadata.

## Solution Assessment

Yes at the direction level. The maintainable fix is to persist the resolved runtime port in the existing Gateway lock and consume it only for default local TUI resolution; the open linked PR is already attempting that path.
