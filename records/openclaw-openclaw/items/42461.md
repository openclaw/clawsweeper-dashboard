## Summary

Current main and the latest release still lack the active Gateway runtime-port handoff for default TUI connections, and an open same-repo PR already owns the fix path, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that `openclaw tui` connect to the active local Gateway runtime port instead of defaulting to `18789` when the Gateway was started with a custom port.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73338, or an equivalent lock-backed fix that preserves explicit `--url` and remote Gateway precedence while settling the local port precedence contract.

## Reproduction Assessment

Yes, at source level: current main builds default TUI Gateway URLs from config/env/default port data, while the active Gateway lock still does not persist a runtime port. I did not run the live two-process CLI repro in this read-only review.

## Solution Assessment

Yes, pending PR review: recording the active runtime port in Gateway-owned lock metadata and having default local TUI resolution read it is the narrow owner-boundary fix. The remaining maintainer choice is the intended precedence against `OPENCLAW_GATEWAY_PORT`.
