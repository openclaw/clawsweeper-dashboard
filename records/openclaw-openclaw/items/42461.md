Keep open. Current main and the latest release still lack the active Gateway lock-port handoff, and the open linked PR remains the focused implementation path for this regression.

## What This Changes

The issue reports that default `openclaw tui` still connects to `ws://127.0.0.1:18789` instead of the active custom Gateway port, such as `48789`.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, remote config, and stale-lock safety.

## Reproduction Assessment

Yes. The issue gives concrete `openclaw gateway --port 48789` then `openclaw tui` steps, and current-main source still resolves the default local URL from env/config/defaults without reading live lock-port metadata.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in the existing Gateway lock and consuming it only for default local TUI resolution is the narrow maintainable fix, while explicit URL and remote config precedence remain intact.
