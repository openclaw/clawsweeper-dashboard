## Summary

Keep open. Current main and the latest release still resolve the default local TUI Gateway target from env/config/defaults rather than active Gateway lock metadata, and the linked open PR at https://github.com/openclaw/openclaw/pull/73338 remains the focused implementation vehicle.

## What This Changes

The issue reports that `openclaw tui` defaults to `ws://127.0.0.1:18789` instead of the active custom Gateway `--port` unless `--url` is passed.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, remote config, and stale-lock safety.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a Gateway with `--port 48789`, then run `openclaw tui` without `--url`; current main can still build the local target from env/config/defaults instead of active lock metadata.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in the existing Gateway lock and using it only for default local TUI resolution is the narrow maintainable fix, and the linked PR remains the right merge vehicle.
