## Summary

Keep open. Current main and the latest release still build the default TUI Gateway target from config/env/defaults, not the active Gateway runtime port, and https://github.com/openclaw/openclaw/pull/73338 is still the open focused fix path with closing syntax for this issue.

## What This Changes

The issue reports that `openclaw tui` targets the default Gateway port `18789` instead of following a running Gateway started on a custom runtime port such as `48789`.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with lock-backed port metadata consumed only for default local TUI resolution, preserving explicit `--url` and remote config precedence.

## Reproduction Assessment

Yes, source-reproducible. The issue gives concrete `openclaw gateway --port 48789` then `openclaw tui` steps, and current-main source still resolves the default local TUI URL without active runtime-port metadata.

## Solution Assessment

Yes at the direction level. The maintainable fix is to persist the resolved runtime port in the existing Gateway lock and consume it only for default local TUI resolution; the open linked PR is already attempting that path.
