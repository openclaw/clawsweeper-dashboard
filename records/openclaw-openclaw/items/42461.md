## Summary

Keep open. Current main still resolves the default TUI Gateway URL from env/config/defaults and does not consume active Gateway lock port metadata, while the open fix PR https://github.com/openclaw/openclaw/pull/73338 is explicitly paired to this issue with closing syntax.

## What This Changes

Make `openclaw tui` follow the active local Gateway runtime port instead of defaulting to `18789` when Gateway was launched with a custom port.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, env URL, remote config, and stale/unverified-lock behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` with default config/env; current main can still build `ws://127.0.0.1:18789` because it lacks a live lock-port source.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in existing Gateway lock metadata and consuming it only for default local TUI resolution is the narrow maintainable fix, but this issue should stay open until the linked PR lands or is superseded.
