## What This Changes

The issue asks for `openclaw tui` to follow the active local Gateway runtime port when Gateway is started with a custom `--port`.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, remote config, and stale-lock safety.

## Reproduction Assessment

Yes, source-reproducible. The issue gives concrete `openclaw gateway --port 48789` then `openclaw tui` steps, and current main still resolves the default local TUI URL from env/config/defaults without reading active lock-port metadata.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in the existing Gateway lock and consuming it only for default local TUI resolution is the narrow maintainable fix; broad environment mutation would be less clean than injecting the active port into the resolver.
