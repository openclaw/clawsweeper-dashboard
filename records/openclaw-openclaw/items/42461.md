## Summary

Keep open. Current main still resolves the default TUI Gateway target from config/env/defaults without reading active Gateway lock-port metadata, and the open linked fix PR is explicitly paired to this issue with closing syntax.

## What This Changes

The issue asks for `openclaw tui` to follow the active local Gateway runtime port, such as `48789`, instead of falling back to the default `18789`.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, env URL, remote config, and stale or unverifiable lock behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` with default config/env; current main can still build `ws://127.0.0.1:18789` because it lacks a live lock-port source.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in existing Gateway lock metadata and consuming it only for default local TUI resolution is the narrow maintainable fix, but this issue should stay open until the linked PR lands or is superseded.
