## What This Changes

The issue reports that `openclaw tui` ignores an active Gateway started with a custom runtime `--port` and still connects to the default local port.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, env URL, remote config, and stale or unverifiable lock behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` without `--url`; current main can still build `ws://127.0.0.1:18789` because it lacks an active lock-port readback path.

## Solution Assessment

Yes at the direction level. Persisting and verifying the active Gateway lock port, then using it only for default local TUI resolution, is the narrow maintainable fix while the linked PR remains the right merge vehicle.
