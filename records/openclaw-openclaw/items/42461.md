## Summary

Keep open: current main still builds the default TUI Gateway URL from config/env/defaults and the lock payload still does not persist the resolved listener port; #73338 is the open target-repo fix linked with closing syntax, so maintainers should review or supersede it rather than close this issue.

## What This Changes

The issue asks for `openclaw tui` to resolve its default local Gateway URL from the active running Gateway port instead of falling back to 18789 when the Gateway was started with `--port`.

## Best Possible Solution

Land or supersede #73338 with lock-backed active-port discovery limited to local default TUI connections, preserving explicit `--url`, env URL, and remote config precedence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` in a separate default environment; the TUI resolver still has no active runtime-port source and can build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. The narrow maintainable solution is to persist the resolved active Gateway port in trusted runtime/lock metadata and have only the local default TUI path consume it, while explicit URL and remote configuration precedence remain unchanged.
