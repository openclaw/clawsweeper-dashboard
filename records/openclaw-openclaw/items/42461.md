## Label Justifications

- `P2`: The regression breaks normal TUI connectivity for local Gateways started on custom ports, although an explicit `--url` is an available workaround.
- `impact:other`: The issue prevents a core CLI surface from reaching an otherwise healthy active local Gateway.

## Summary

Current main and the latest release still lack a handoff from the running Gateway’s CLI-only port to default TUI connection resolution. The open, proof-positive linked PR remains the canonical implementation path, so this issue is still necessary until that PR lands or is rejected.

## What This Changes

Make `openclaw tui` follow the active local Gateway’s runtime `--port` value instead of falling back to port 18789.

## Best Possible Solution

Land the linked lock-owned discovery design with its liveness and precedence guards, keeping the running Gateway as the single owner of its runtime port without adding configuration or environment compatibility paths.

## Reproduction Assessment

Yes at the source level. Current main has no path carrying the Gateway’s CLI-only runtime port into default TUI URL resolution, and the linked PR documents a real before-and-after connection scenario on port 48789.

## Solution Assessment

Yes. Persisting the resolved port in the existing active Gateway lock and consulting it only for local default resolution is narrower and more owner-correct than parsing supervisor definitions, adding configuration, or globally mutating the environment.
