## Summary

Keep open: current main and the latest release still build the default local TUI Gateway URL from config/env/defaults and the lock payload does not publish the active runtime port. The open linked PR is the active fix candidate, so this issue should stay open until that PR lands or maintainers choose a different fix.

## What This Changes

Requests that `openclaw tui` follow the active local Gateway runtime port instead of falling back to `ws://127.0.0.1:18789` when the Gateway was started with a custom port.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73338, or an equivalent lock-backed fix that records the active Gateway port, verifies the lock owner before trusting it, and preserves explicit `--url` plus remote-config precedence.

## Reproduction Assessment

Yes, source-level: current main resolves the default TUI Gateway URL from config/env/defaults and the active lock payload has no port field. I did not run a live two-terminal Gateway/TUI repro in this read-only review.

## Solution Assessment

Yes, the linked PR's lock-backed direction appears to be the best maintainable fix because it uses the existing active Gateway ownership seam without mutating config or env and preserves explicit URL and remote precedence.
