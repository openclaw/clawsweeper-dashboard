## Summary

Keep open. Current main still resolves the default TUI Gateway URL from config/env/default port state, while the active Gateway lock does not expose the running listener port; the open closing PR should be reviewed or superseded before this issue closes.

## What This Changes

The issue requests that `openclaw tui` automatically connect to the active local Gateway runtime port, such as `48789`, instead of falling back to `18789` after `openclaw gateway --port 48789`.

## Best Possible Solution

Land or supersede the open closing PR with a narrow lock-backed runtime-port discovery path: record the resolved Gateway port, have default local TUI resolution consult that active metadata only after explicit URL/env/remote precedence, and cover custom-port plus stale-lock cases in tests and docs.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: `openclaw gateway --port 48789` resolves the port in the Gateway process, but a later `openclaw tui` without `--url` resolves local loopback through config/env/defaults and can still build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. The narrow maintainable fix is to expose the active Gateway listener port through the existing lock/runtime metadata seam and make TUI consume it for local defaults without changing explicit URL or remote connection precedence.
