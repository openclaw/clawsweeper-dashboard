## Summary

Keep open. Current main and the latest release still lack the active Gateway runtime-port handoff, and the open linked PR is the focused implementation path for this regression.

## What This Changes

The issue asks for `openclaw tui` to follow the active Gateway runtime port instead of falling back to port 18789.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, remote config, and stale-lock safety.

## Reproduction Assessment

Yes. The issue gives concrete `openclaw gateway --port 48789` then `openclaw tui` steps, and current-main source still resolves the default local URL from env/config/defaults without reading active lock-port metadata.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in the existing Gateway lock and consuming it only for default local TUI resolution is the narrow maintainable fix; mutating broad process env is less clean than injecting the active port into the resolver.
