## Summary

Keep open. Current main still lacks a live Gateway runtime-port handoff for default TUI connections, and the open paired fix at https://github.com/openclaw/openclaw/pull/73338 uses closing syntax, so this issue should remain open until that PR lands or is superseded.

## What This Changes

The issue reports that `openclaw tui` still targets the default local Gateway port instead of following the active Gateway runtime port when the Gateway was started with `--port`.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/73338 with a verified lock-backed active-port handoff that preserves explicit `--url`, env URL, remote config, and stale or unverifiable lock behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a Gateway with a custom port such as `48789`, then run `openclaw tui` without `--url`; current main can still build `ws://127.0.0.1:18789` because it does not read active runtime-port metadata.

## Solution Assessment

Yes at the direction level. Persisting and verifying the active Gateway lock port, then using it only for default local TUI resolution, is the narrow maintainable fix while the linked PR remains the right merge vehicle.
