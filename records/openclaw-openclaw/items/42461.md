## What This Changes

The issue asks for `openclaw tui` to follow the active local Gateway runtime port instead of defaulting to `18789` when Gateway was started with a custom `--port`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73338, or an equivalent fix that records the active Gateway runtime port and uses it for default local TUI resolution while preserving explicit `--url` and remote config precedence.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.8` shows TUI default URL resolution still comes from config/env/default port data while the Gateway lock payload does not persist the active runtime `--port`; I did not run the live two-process repro in this read-only review.

## Solution Assessment

Yes, the linked PR's lock-backed runtime-port approach appears to be the right owner-boundary fix because Gateway already owns the active lock and TUI only needs it for default local connections; maintainers still need to accept the active-lock precedence decision before merge.
