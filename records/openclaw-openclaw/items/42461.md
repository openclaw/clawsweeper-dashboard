## Summary

Keep open. Current main still lacks active runtime-port discovery in the default TUI Gateway connection path, and an open target-repo PR already references this issue as its closing target, so maintainers should review, land, or supersede that PR rather than close this issue or queue duplicate repair work.

## What This Changes

The issue asks for `openclaw tui` to automatically follow the active local Gateway runtime port, such as `ws://127.0.0.1:48789`, instead of falling back to `18789`.

## Best Possible Solution

Land or supersede #73338 with a narrow lock-backed runtime-port metadata path that preserves explicit `--url`, env URL, and remote-config precedence, plus custom-port and stale-lock regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` in a separate process with default config/env; the TUI resolver has no active-runtime metadata source and can still build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. The narrow maintainable solution is to expose the resolved Gateway port through existing active lock/runtime metadata and have the TUI consume it only for local defaults, while leaving explicit URL and remote precedence unchanged.
