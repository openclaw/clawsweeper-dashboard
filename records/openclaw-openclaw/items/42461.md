## Summary

Keep open. Current main still lacks a cross-process active Gateway port handoff for default TUI connections, and the live open PR https://github.com/openclaw/openclaw/pull/73338 is already paired to this issue with closing syntax.

## What This Changes

The issue asks `openclaw tui` to use the active local Gateway runtime port, such as 48789, instead of falling back to 18789 when no explicit `--url` is passed.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit URL, env URL, remote config, and stale-lock behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` with default config/env; the resolver still has no active lock-port source and can build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in existing Gateway lock metadata and consuming it only for default local TUI resolution is the narrow maintainable fix, but the issue should stay open until the linked PR lands.
