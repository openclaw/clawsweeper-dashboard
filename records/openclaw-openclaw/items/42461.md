## Summary

Keep open. Current main still builds the default TUI Gateway URL from env/config/defaults, the Gateway lock payload still omits the runtime port, and https://github.com/openclaw/openclaw/pull/73338 remains the focused implementation path for this regression.

## What This Changes

Make `openclaw tui` connect to the running local Gateway's active runtime port, such as 48789, instead of falling back to 18789 when no `--url` is passed.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/73338 with a lock-backed active-port handoff that preserves explicit `--url`, remote config, and stale-lock safety.

## Reproduction Assessment

Yes, at source level. The issue gives concrete `openclaw gateway --port 48789` then `openclaw tui` steps, and current main still resolves the default TUI URL without reading live lock-port metadata.

## Solution Assessment

Yes at the direction level. Recording the resolved runtime port in the existing Gateway lock and consuming it only for default local TUI resolution is the narrow maintainable fix while explicit URL and remote config precedence remain intact.
