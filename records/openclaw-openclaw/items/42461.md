## Summary

Keep open. Current main still builds the default TUI Gateway URL from config/env/defaults instead of active runtime lock metadata, and open target-repo PR #73338 already references this issue as its closing target, so maintainers should review, land, or supersede that PR rather than close this issue or queue duplicate work.

## What This Changes

The issue asks for `openclaw tui` to follow a running local Gateway's active custom runtime port instead of defaulting to `18789`.

## Best Possible Solution

Review and land or supersede #73338 with a narrow lock-backed active-port metadata path that preserves explicit `--url`, environment URL, and remote-config precedence, plus stale-lock and custom-port regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` in a separate process with default config/env; the resolver still has no active runtime-port source and can build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. The narrow maintainable fix is to persist the resolved active Gateway port in existing lock/runtime metadata and have the TUI consume it only for local default resolution while preserving explicit and remote overrides.
