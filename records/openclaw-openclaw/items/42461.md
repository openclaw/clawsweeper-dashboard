## Summary

Keep open. Current main still resolves the TUI default Gateway URL from config/env/defaults rather than active runtime port metadata, and open PR #73338 already closes this issue, so maintainers should review, land, or supersede that PR instead of closing this issue or queueing duplicate work.

## What This Changes

The issue asks for `openclaw tui` to follow the active local Gateway runtime port, such as `ws://127.0.0.1:48789`, instead of falling back to `18789`.

## Best Possible Solution

Review and land or supersede #73338 with a lock-backed active-port metadata path, preserving explicit `--url`, env URL, and remote-config precedence plus stale-lock regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` in a separate default environment; current main has no cross-process active-port source before it can build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. The narrow maintainable fix is to persist or validate active local Gateway port metadata and use it only for default local TUI resolution while preserving explicit and remote overrides.
