## Summary

Keep open: current main and the latest release still build the default local TUI Gateway URL from config/env/defaults, while the active Gateway lock has no runtime port metadata. The open same-repo fix PR already closes this issue, so this should stay open until that PR lands or is closed.

## What This Changes

The issue reports that `openclaw tui` targets the default Gateway port instead of the active runtime port when the Gateway is started with a custom `--port`.

## Best Possible Solution

Review, refresh checks for, and land `https://github.com/openclaw/openclaw/pull/73338`, or close that PR with maintainer rationale before reconsidering a replacement fix.

## Reproduction Assessment

## Solution Assessment

Yes, the open PR's lock-backed active-port read is the narrow maintainable fix because it uses the existing Gateway ownership seam, preserves explicit `--url` and remote-config precedence, and avoids adding config surface.
