## Summary

Current main and the latest release still lack a runtime-port source in the default TUI resolver, and the Gateway lock payload still has no port field. This issue should stay open because the remaining fix is already represented by the open same-repository closing PR.

## What This Changes

The issue requests that `openclaw tui` connect to the active local Gateway runtime port, such as 48789, instead of falling back to 18789 when no `--url` is passed.

## Best Possible Solution

Land `https://github.com/openclaw/openclaw/pull/73338` or an equivalent lock-backed fix that records the active Gateway port, preserves explicit URL and remote-config precedence, and keeps regression coverage for the 48789 case.

## Reproduction Assessment

Yes, source-level: current main builds the default TUI URL from config/env/defaults and the current Gateway lock payload does not store a runtime port. I did not run a live two-terminal repro because this review is read-only and tests/runtime commands could write artifacts.

## Solution Assessment

Yes for the linked direction: the best fix is to use the existing Gateway lock seam to publish a verified active runtime port for default local TUI resolution while preserving explicit `--url` and remote config precedence.
