## Summary

Keep open. Current main still lacks active-port discovery in the TUI resolver, and the linked open fix PR is the right path to resolve the remaining bug.

## What This Changes

Make `openclaw tui` follow the currently running Gateway's custom runtime port instead of falling back to `18789` when no `--url` is passed.

## Best Possible Solution

Review and land or supersede #73338 with a lock-backed active-port path that preserves explicit `--url`, env URL, remote-config precedence, and stale-lock handling.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: start `openclaw gateway --port 48789`, then run `openclaw tui` with default config/env; the resolver has no cross-process active-port source and can still build `ws://127.0.0.1:18789`.

## Solution Assessment

Yes at the direction level. The narrow maintainable fix is to persist the resolved active Gateway port in existing lock/runtime metadata and have the TUI consume it only for local default resolution.
