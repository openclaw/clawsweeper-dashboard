## Summary

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send preflight to it, and drains selected QQBot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Land the PR after outbound/channel owners accept the resolver and reconnect-drain risk on the exact current head, keeping the shared resolver ungated and the QQBot drain narrowly filtered.

Yes. The issue logs plus current source show a source-level path where setup-only outbound metadata can reach handler construction and produce `Outbound not configured for channel` when no send-capable message/outbound handler is resolved.

## Solution Assessment

Yes. The current patch shape is the best bounded fix I found because it keeps the shared resolver available for action/setup callers, gates only delivery preflight, and adds a narrowly selected QQBot recovery drain rather than making setup-only plugins globally invisible.
