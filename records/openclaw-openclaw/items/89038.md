## Summary

The PR is still useful, but it should not merge as-is because the delivery resolver's send-method gate drops the existing gateway-mode outbound path before `sendMessage` can reach core gateway RPC.

## What This Changes

This PR adds a send-capable delivery resolver for outbound channel plugins, switches message send/poll paths to it, and adds QQBot reconnect pending-delivery drains plus resolver/drain tests.

## Best Possible Solution

Keep the QQBot setup-only shadowing fix, but treat gateway-mode outbound plugins as deliverable and restore gateway-only regression coverage before merge.

## Reproduction Assessment

Yes for the PR-introduced regression from source: current main has a gateway-only sendMessage path, while the PR's delivery resolver only accepts local send methods. The original QQBot issue also has credible live proof in the PR body, but I did not run a live QQBot account in this read-only review.

## Solution Assessment

No. Splitting shared and delivery resolvers is the right shape, but the delivery-capability predicate is too narrow because gateway-mode delivery is an existing supported path.
