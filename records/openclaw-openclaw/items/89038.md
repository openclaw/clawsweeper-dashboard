## Summary

Keep open: current main does not yet contain the QQBot reconnect drain or delivery-only send preflight, but this PR head is still not merge-ready because a poll-only direct plugin can pass text-send preflight and later fail in the direct text handler.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, routes message send and poll preflight through it, drains matching QQBot pending deliveries on ready/resume, and adds focused outbound/QQBot tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell skip, but split or parameterize delivery resolution so text sends require text or gateway capability while poll sends can accept poll capability independently.

## Reproduction Assessment

Yes for the review finding from source: `sendMessage` uses the shared delivery resolver, that resolver accepts poll-only direct plugins, and the downstream direct handler requires text send capability. I did not rerun a live QQBot 4009 timeout against current main.

## Solution Assessment

No, not yet: the setup-shell skip and reconnect drain are the right shape, but the resolver needs text-versus-poll intent so the fix does not leave a false-positive text-send path.
