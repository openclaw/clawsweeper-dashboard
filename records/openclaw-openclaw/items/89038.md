## Summary

Keep open: the PR is still the active fix candidate for the QQBot reconnect message-loss report, but the current patch has a blocking resolver predicate bug and still needs stronger real callback-drain proof or explicit maintainer proof acceptance.

## What This Changes

The PR adds a send-capability-gated outbound resolver for delivery paths and drains matching QQBot pending deliveries from gateway ready/resumed callbacks.

## Best Possible Solution

Keep the QQBot reconnect recovery direction, but make the delivery resolver operation-specific so text sends require gateway or text-capable adapters, then add callback-drain proof or get explicit maintainer acceptance before merge.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows a non-text plugin can pass the new delivery predicate while the downstream text handler still rejects it. The original QQBot bug has strong linked logs and source evidence, but this read-only review did not run a fresh live QQBot reconnect reproduction.

## Solution Assessment

No. The QQBot reconnect recovery goal is valid, but the shared delivery predicate is too broad for text sends and should be narrowed or split by operation before merge.
