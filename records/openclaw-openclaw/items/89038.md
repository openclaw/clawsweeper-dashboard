## Summary

Keep open: the PR has strong QQBot reconnect proof and useful drain/setup-shell work, but the latest head still uses one generic delivery resolver for both text and poll sends, leaving a merge-blocking message-delivery defect.

## What This Changes

The PR adds a send-capability-filtered outbound resolver for delivery preflight, uses it from outbound message/poll sends, and adds QQBot reconnect pending-delivery drains with resolver and reconnect-drain tests.

## Best Possible Solution

Split delivery resolution by operation: text sends should require text or gateway capability, poll sends should require poll capability, and the QQBot reconnect drain should stay covered by focused setup-shell and pending-drain regressions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: the linked issue supplies live QQBot reconnect logs, and PR head source shows the generic resolver can still select a poll-only entry for text or a text-only entry for poll before later usable senders are considered.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the QQBot drain and setup-shell filtering are useful, but the delivery resolver should be operation-specific before merge.
