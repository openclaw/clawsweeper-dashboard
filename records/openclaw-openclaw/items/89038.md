## Summary

Keep open: the patch appears to address a real QQBot outbound delivery regression, but it changes compatibility-sensitive send-path resolution and reconnect replay behavior, so the remaining gate is owner/maintainer acceptance rather than conservative cleanup closure.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips non-send setup shells, switches message send/poll preflight to that resolver, and drains matching QQBot pending deliveries on gateway ready/resumed callbacks.

## Best Possible Solution

Land this only after outbound or QQBot owners accept the send-capability gate and reconnect replay semantics, keeping the fix bounded to delivery resolution and QQBot recovery without adding new config.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still lacks the QQBot reconnect drain, and the linked issue plus PR body provide live logs for the outbound-not-configured failure. I did not run a live QQBot repro in this read-only review.

## Solution Assessment

Yes, this is an acceptable best-fix shape for the resolver: it gates only delivery while preserving read/action resolution. The reconnect drain is a reasonable upgrade recovery path, but owner acceptance is needed because it changes replay timing.
