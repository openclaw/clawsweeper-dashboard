## Summary

Keep this PR open: the QQBot reconnect fix and proof are useful, but the current head still uses one delivery-capability predicate for both text sends and polls, so a poll-only direct plugin can pass `sendMessage` preflight and then fail later with the same outbound-not-configured class of error.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches outbound send preflight to it, drains QQBot pending deliveries on gateway ready/resumed, and expands outbound/QQBot regression tests.

## Best Possible Solution

Keep the QQBot resolver/drain direction, but split or parameterize delivery capability by intent so text sends require text/gateway capability while polls can require poll/gateway capability, with focused regression coverage before merge.

## Reproduction Assessment

Yes, from source inspection for the introduced PR defect: PR head lets `outbound.sendPoll` satisfy `sendMessage` preflight, while current delivery code requires text capability to build a handler. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No; the PR is a plausible QQBot fix, but the best fix needs intent-specific deliverability so text sends and polls do not share a predicate with incompatible capability requirements.
