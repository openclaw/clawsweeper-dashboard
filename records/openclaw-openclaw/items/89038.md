## Summary

Keep this PR open: the code path looks like the right fix candidate for the linked QQBot outbound reconnect regression, but the real behavior proof still does not directly show a pending delivery recovered through the new onReady/onResumed drain path.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, uses it for message send preflight, and drains matching QQBot pending deliveries from ready/resumed callbacks.

## Best Possible Solution

Land this after maintainers either accept the seeded/shared-drain proof gap or receive redacted real logs/artifacts showing callback-path pending delivery recovery, then let the linked issue close after merge.

## Reproduction Assessment

Yes, source-reproducible: the linked QQBot issue has real v2026.5.28 4009 reconnect logs, and current delivery code leaves pre-platform send failures pending for recovery. I did not run a fresh live QQBot timeout against current main in this read-only review.

## Solution Assessment

Mostly yes: splitting shared resolution from delivery send-capability checks is the right boundary, and QQBot-owned reconnect draining matches existing Telegram/WhatsApp recovery patterns. Resetting bootstrap state alone would not recover already-pending entries, while gating the shared resolver would risk action-only callers.
