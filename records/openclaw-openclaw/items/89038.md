## Summary

Keep open: this is an active bug-fix PR with useful QQBot reconnect proof, but the latest head still needs one correctness repair because text/gateway and poll delivery are resolved through the same broad capability gate.

## What This Changes

The PR adds a delivery-only outbound resolver that skips setup-only channel plugins, wires message and poll send preflight through it, and drains QQBot pending deliveries on ready/resumed reconnect.

## Best Possible Solution

Split delivery resolution by send operation, keep the shared resolver ungated for non-send callers, and retain the QQBot reconnect drain with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: the linked issue has live 2026.5.28 QQBot reconnect logs, and the current source still has the outbound-not-configured throw path when no send-capable handler is resolved.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the QQBot drain and setup-shell skip are the right direction, but the delivery resolver should be operation-specific before this is the narrow maintainable fix.
