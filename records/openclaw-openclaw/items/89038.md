## Summary

Keep open for maintainer review: the patch appears to address the linked QQBot reconnect outbound failure with solid source and live-log proof, but it intentionally changes shared delivery plugin selection and QQBot reconnect queue-drain timing.

## What This Changes

The PR adds operation-aware outbound delivery plugin resolution, routes send and direct delivery through that resolver, and drains matching QQBot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Land this PR or an equivalent owner-approved fix after accepting the shared resolver and QQBot reconnect-drain compatibility tradeoff, then close the linked QQBot regression issue with merge provenance.

## Reproduction Assessment

Yes at source level: current main still has a capability-blind direct adapter load path and the linked issue has live QQBot 4009 reconnect logs ending in `Outbound not configured for channel: qqbot`. I did not run a fresh live QQBot timeout against current main.

## Solution Assessment

Yes with maintainer acceptance: the delivery-only resolver fixes setup-shell shadowing where sends are resolved, and the reconnect drain covers pending entries left by the broken upgrade path. The remaining question is the intentional compatibility tradeoff, not a concrete patch defect.
