## Summary

Keep open: this PR addresses a real QQBot reconnect message-loss regression with strong proof, but the operation-aware poll resolver is only wired into `sendPoll` preflight and not the gateway poll endpoint that actually calls `outbound.sendPoll`. The central QQBot text-delivery fix is credible, but one shared delivery surface still needs a narrow repair before this is merge-ready.

## What This Changes

The PR adds operation-aware outbound delivery plugin resolution, rewires message/direct delivery to skip non-send setup shells, and adds QQBot reconnect pending-delivery drains plus regression tests.

## Best Possible Solution

Land a version that uses the operation-aware delivery resolver anywhere an actual send operation is selected, keeps the shared resolver for non-send/action surfaces, and has maintainer acceptance of the QQBot reconnect queue-drain behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: the linked issue has reconnect logs showing `Outbound not configured for channel: qqbot`, current main still has the capability-blind direct adapter load, and the PR adds a focused regression for that path; I did not run a fresh live QQBot 4009 timeout myself.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the resolver split is the right direction for send paths, but the gateway poll endpoint still uses the shared resolver, so the best fix is to align every actual send-operation selector while preserving the shared resolver for action/read callers.
