## Summary

Keep open: the QQBot reconnect fix is useful and the supplied proof is strong, but the PR still uses one aggregate delivery resolver for both text and poll paths, so mixed plugin registrations can select a plugin that cannot perform the requested send operation.

## What This Changes

The branch adds a send-capability-gated outbound delivery resolver, switches sendMessage/sendPoll preflight to it, and drains QQBot pending deliveries on gateway ready/resumed with resolver and QQBot channel tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell filtering, but make delivery plugin resolution operation-specific so text sends require text-or-gateway support and poll sends require poll support.

## Reproduction Assessment

Yes at source level. A poll-only plugin registered ahead of a text sender satisfies the PR's aggregate predicate, while sendMessage later needs a text-capable handler; the inverse text-only-before-poll path reaches the poll capability check.

## Solution Assessment

No. The QQBot drain and setup-shell skip are the right direction, but the resolver should be split by required send operation before merge.
