## Summary

Keep open: the PR is a useful direction, but it is not merge-ready because the new delivery resolver only protects `sendMessage`/`sendPoll` preflight while the linked heartbeat and cron announcement paths still call the durable delivery core directly.

## What This Changes

This PR adds a delivery-specific outbound channel resolver used by message send/poll and adds QQBot gateway ready/resumed drains for pending deliveries, with resolver and QQBot drain tests.

## Best Possible Solution

Move the send-capability/bootstrap guard into the shared durable delivery path used by `sendDurableMessageBatch`, then keep the QQBot reconnect drain only with owner-accepted replay semantics and direct heartbeat/cron regression proof.

## Reproduction Assessment

Yes by source inspection, not by a live QQBot run: heartbeat and cron call `sendDurableMessageBatch` directly and can still reach the durable handler without the new `sendMessage` preflight.

## Solution Assessment

No: the current patch is a plausible partial fix, but the best fix needs the send-capability/bootstrap decision in the shared durable delivery path used by all outbound senders.
