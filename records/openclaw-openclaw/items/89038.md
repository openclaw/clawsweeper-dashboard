## Summary

Needs changes before merge: the QQBot reconnect drain and setup-shell filtering are useful, but the shared delivery resolver still lets poll-only direct plugins qualify for text send preflight and can leave the hot text delivery path failing with `Outbound not configured`.

## What This Changes

The PR adds a delivery-only outbound channel resolver, routes message send preflight through it, drains QQBot pending deliveries on reconnect callbacks, and adds outbound/QQBot regression tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell filtering, but make delivery resolution intent-specific so text sends require text/gateway capability while polls require poll/gateway capability, with ordered fallback regression coverage.

## Reproduction Assessment

Yes, source-level: the linked issue provides live QQBot 4009 reconnect logs, and current/PR source shows the outbound-not-configured path through send preflight and `createPluginHandler`. I did not run a fresh live QQBot timeout in this read-only review.

## Solution Assessment

No: the QQBot drain plus delivery-only setup-shell filtering are the right direction, but the latest patch is not the best final fix until text and poll capability checks are separated at the resolver boundary.
