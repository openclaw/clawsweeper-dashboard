## Summary

## What This Changes

The branch adds a send-capability-gated outbound delivery resolver, switches sendMessage/sendPoll preflight to it, and drains QQBot pending deliveries on gateway ready/resumed with focused tests.

## Best Possible Solution

Merge after the QQBot/outbound owner accepts the replay scope, keeping the shared resolver ungated for non-send callers and the delivery-only resolver gated by send capability.

## Reproduction Assessment

Yes. Current main still has the shared-resolver send preflight path, and the linked issue plus PR body give concrete QQBot reconnect logs and live patched proof for the outbound-not-configured failure mode.

## Solution Assessment

Yes, with maintainer acceptance of the replay tradeoff. Splitting shared lookup from delivery lookup is the narrowest maintainable fix for send preflight while preserving action/read callers; the reconnect drain is a reasonable upgrade recovery safeguard but needs owner signoff.
