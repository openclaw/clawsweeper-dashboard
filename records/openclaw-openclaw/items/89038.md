## Summary

Keep open: the PR targets a real QQBot reconnect message-loss regression and has strong live proof, but the current diff still has a blocking resolver bug where poll-only plugins can satisfy ordinary text-send preflight and then fail later with the same outbound-not-configured path.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches `sendMessage`/`sendPoll` preflight to it, drains QQBot pending deliveries on gateway ready/resumed, and expands outbound/QQBot tests.

## Best Possible Solution

Make delivery resolution intent-specific while preserving the shared ungated resolver for non-send callers, then land the QQBot reconnect drain after maintainer acceptance of the replay semantics and exact-head checks.

## Reproduction Assessment

Yes, at source level. `sendMessage` and `sendPoll` now share `resolveOutboundChannelPluginForDelivery`, the new predicate accepts `outbound.sendPoll`, and the delivery core still requires a text sender for ordinary message sends.

## Solution Assessment

No, not yet. The PR's overall shape is the right layer, but the resolver must distinguish text-send and poll-send capability before it is the best fix.
