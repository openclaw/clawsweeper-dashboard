## Summary

Keep open: the QQBot reconnect proof is strong and the PR is still the active candidate fix for the linked user report, but the latest head still uses one delivery-capability predicate for both text sends and polls, leaving a shared outbound compatibility blocker before merge.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips setup-only plugins, wires it into message sends and polls, drains QQBot pending deliveries on ready/resumed reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell fix, but split delivery resolution by requested operation so text sends require text or gateway capability and polls require poll or gateway capability before merge.

## Reproduction Assessment

Yes, source-level. The PR head lets a poll-only direct plugin pass the same preflight used by text sends, while current delivery handler construction still requires a text sender and otherwise throws `Outbound not configured`.

## Solution Assessment

No. The direction is right, but the best fix is operation-specific delivery resolution rather than one send-capability predicate shared by text and poll paths.
