## Summary

Keep this PR open: current main still has the QQ Bot outbound-resolution/reconnect recovery gap and this patch is a plausible owner-boundary fix, but the new reconnect-drain behavior is still not directly proven in a real queued-delivery scenario.

## What This Changes

The PR adds a send-capability-aware outbound delivery resolver, switches core message send paths to it, and drains matching QQ Bot pending deliveries from gateway ready/resumed callbacks.

## Best Possible Solution

Keep the shared resolver for non-send callers, land the delivery resolver and QQ Bot-owned reconnect drain only after queued-drain proof is added or maintainers explicitly split/override that proof gap.

## Reproduction Assessment

Yes, source-reproducible for the core failure mode: current main can resolve a non-sending channel plugin for delivery, and delivery then errors when no send handler exists. I did not run a live QQ Bot reconnect repro in this read-only review.

## Solution Assessment

Mostly yes: the split delivery resolver is the right shape because it avoids breaking non-send callers, and the reconnect drain belongs in the QQ Bot plugin. It is not proof-complete until the queued-drain path is shown in a real reconnect scenario or removed from this PR.
