## Summary

Keep open: the latest head still has a source-provable delivery-resolution bug because text sends and poll sends share one capability gate that accepts poll-only direct plugins.

## What This Changes

The PR adds a send-capability-gated outbound channel resolver for delivery paths and drains QQBot pending deliveries on ready/resumed reconnect callbacks.

## Best Possible Solution

Split delivery qualification by operation: text sends should require text or gateway capability, while poll sends should use a poll-capable path.

## Reproduction Assessment

Yes. Source inspection shows a poll-only direct plugin satisfies the PR's shared delivery resolver, and `sendMessage` uses that resolver before entering a direct text path that requires text send capability.

## Solution Assessment

No. The PR is close, but the best fix needs operation-specific resolver qualification instead of one delivery predicate shared by text and poll sends.
