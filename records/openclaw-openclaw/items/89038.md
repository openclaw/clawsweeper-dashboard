## Summary

Keep open: the PR has strong QQBot proof and addresses the right area, but the actual direct delivery path can still load a pinned setup-only outbound adapter before the active send-capable runtime fallback, leaving the reported `Outbound not configured` failure reachable.

## What This Changes

The PR adds a delivery-only outbound resolver, wires message/poll preflight through it, and drains matching QQBot pending deliveries on gateway ready/resumed reconnect.

## Best Possible Solution

Make the actual direct delivery adapter selection operation-aware or text-capability-aware, then add an unmocked `sendMessage` or `deliverOutboundPayloads` regression for a pinned setup shell with `outbound: { deliveryMode: "direct" }` and an active `sendText` runtime fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a pinned setup shell with `outbound: { deliveryMode: "direct" }` can pass the PR preflight while `loadChannelOutboundAdapter` still returns that non-text outbound object, causing `createPluginHandler` to throw `Outbound not configured`.

## Solution Assessment

No. The PR is a plausible partial fix, but the best fix must align the real direct delivery adapter loader with the send-capability check instead of only changing the `sendMessage`/`sendPoll` preflight.
