## Summary

Keep this PR open: it addresses the right QQBot reconnect/message-loss bug with useful proof, but the current head still uses one delivery resolver predicate for both text sends and polls, which can select a plugin that cannot perform the requested operation.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips setup-only channel plugins, wires it into message/poll sends, and drains matching QQBot pending deliveries on ready/resumed reconnect.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell fix, but split delivery resolution by requested operation so message/text sends and poll sends each require the capability they actually call.

## Reproduction Assessment

Yes, source-level. The PR head lets text and poll sends share one capability predicate, while current delivery handler construction requires text capability and `sendPoll` requires poll capability.

## Solution Assessment

No. The direction is right, but the best fix is operation-specific delivery resolution rather than one send-capability predicate shared by text and poll paths.
