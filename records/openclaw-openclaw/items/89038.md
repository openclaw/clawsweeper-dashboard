## Summary

Keep open for maintainer review: the PR targets a real QQBot outbound-delivery regression and the patch reads as a focused fix, but it changes channel delivery resolution policy and adds reconnect replay behavior that maintainers should explicitly accept before merge.

## What This Changes

The branch adds a send-capable delivery resolver for outbound channel sends, switches message delivery paths to it, and drains matching QQBot pending deliveries on gateway ready/resumed callbacks.

## Best Possible Solution

Land after maintainers accept the send-capability-gated delivery resolver and QQBot reconnect replay semantics, keeping non-send callers on the ungated shared resolver.

## Reproduction Assessment

Yes. The linked report provides QQBot reconnect logs, and current source shows the ungated resolver can return a non-sending plugin before delivery handler construction throws and leaves the queued entry pending.

## Solution Assessment

Yes, this appears to be the right layer: delivery paths get a send-capability resolver while action/read callers keep the shared ungated resolver. The reconnect drain is plugin-owned and scoped by channel/account, with the remaining question limited to maintainer acceptance of replay proof breadth.
