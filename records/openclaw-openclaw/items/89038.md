## Summary

Keep open: the QQBot reconnect fix is useful and the supplied live/log proof is strong, but the current head still lets poll-only direct plugins satisfy text-send preflight and then fail later, so it needs a small delivery-intent split before merge.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, routes message send/poll preflight through it, and drains matching QQBot pending deliveries on gateway ready/resume with regression tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell skip, but split or parameterize delivery resolution so text sends require text/gateway capability while poll sends can independently accept poll capability.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source for the review finding: `sendMessage()` uses the single delivery resolver, that resolver accepts `outbound.sendPoll`, and direct text delivery still requires a text sender; the original QQBot bug also has supplied live reconnect logs, but I did not rerun a live QQBot 4009 timeout.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the general shape is right, but the resolver needs text-versus-poll intent so the fix does not create a competing false-positive send path.
