## Label Justifications

- `P1`: The linked regression breaks QQBot outbound deliveries after reconnect for heartbeat, cron, and related real-user channel workflows.
- `merge-risk: 🚨 compatibility`: The PR changes channel plugin resolution semantics before delivery, which can affect existing setup shells, external runtimes, gateway-only plugins, and poll-capable plugins.
- `merge-risk: 🚨 message-delivery`: The PR changes both outbound plugin selection and reconnect queue replay, so merge behavior can still drop, retry, or duplicate messages if the gate is wrong.

## Summary

Keep open: this targets a real QQBot message-loss regression, but the final diff still lets poll-only direct plugins satisfy the shared delivery preflight for ordinary text sends, leaving a shadowing path that can still fail with `Outbound not configured`.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches send preflight to it, adds QQBot reconnect pending-delivery drains, and expands outbound/QQBot regression tests.

## Best Possible Solution

Make the delivery resolver intent-specific while keeping the shared resolver ungated for non-send callers, then land the QQBot reconnect drain after exact-head validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the PR head accepts `outbound.sendPoll` in the shared delivery preflight, while the direct text delivery handler still requires text capability and throws `Outbound not configured` without it.

## Solution Assessment

Is this the best way to solve the issue? No, not yet; the resolver layer is appropriate, but it needs text-vs-poll intent separation before the fix is safe to merge.
