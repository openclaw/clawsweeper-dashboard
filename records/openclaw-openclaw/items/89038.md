## What This Changes

The branch adds a send-capability-gated outbound delivery resolver, switches message send/poll preflight to it, and drains selected QQBot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Land the focused resolver/drain fix once maintainers accept the delivery compatibility and reconnect retry timing, keeping the shared resolver ungated for action/setup callers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the linked issue has timed QQBot 4009 reconnect logs and current source shows the exact outbound-not-configured throw after delivery queue write-ahead when no send-capable handler resolves; I did not run a live QQBot repro myself.

## Solution Assessment

Is this the best way to solve the issue? Yes, subject to maintainer acceptance of the merge risks: splitting shared resolution from delivery-only sendability avoids breaking action/setup callers while the QQBot drain covers pending entries left by the bad resolver state.
