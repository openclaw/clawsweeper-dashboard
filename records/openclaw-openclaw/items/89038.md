## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches send/poll preflight to it with bootstrap support, preserves gateway-mode delivery, and drains pending QQBot deliveries on gateway ready/resumed callbacks with focused tests.

## Best Possible Solution

Land this fix after current-head QQBot reconnect proof or an explicit maintainer proof override, with owner acceptance that the shared resolver stays ungated while delivery paths require a send-capable or gateway-mode plugin.

## Reproduction Assessment

Yes from source and supplied issue logs, but not by a live run in this read-only review. The failure path is clear: a setup-only QQBot shell can satisfy the shared resolver while lacking send methods, leading the delivery core to throw Outbound not configured for channel: qqbot.

## Solution Assessment

Yes, conditionally: splitting shared resolution from delivery resolution is the maintainable fix because action/read callers keep the old resolver while send paths require deliverability. The reconnect drain is a reasonable recovery layer for pending entries, but it needs current-head proof or a maintainer override before merge.
