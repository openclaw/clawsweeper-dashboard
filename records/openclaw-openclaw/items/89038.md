## Summary

Keep this PR open because it is the active candidate for a real QQBot outbound delivery regression, but the current branch still has a blocking resolver bug: one aggregate delivery predicate is used for both text and poll sends even though those operations require different plugin capabilities.

## What This Changes

The PR adds a send-capability-gated outbound channel resolver, routes message and poll preflight through it, and drains QQBot pending deliveries on gateway ready/resumed callbacks.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell filtering, but make delivery plugin resolution operation-specific so text sends require text-or-gateway support and poll sends require poll support.

## Reproduction Assessment

Yes. The linked issue provides concrete QQBot reconnect logs, and source inspection shows the PR head still lets one aggregate resolver feed both text and poll send paths without operation-specific capability checks.

## Solution Assessment

No. The QQBot drain and setup-shell filtering are useful, but the resolver should be split by required send operation before merge.
