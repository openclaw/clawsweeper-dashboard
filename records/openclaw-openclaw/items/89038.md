## Summary

Keep open: the PR targets a real QQBot reconnect delivery regression, but the shared outbound resolver is send-gated in a way that can break action-only channel plugins, and the new QQBot reconnect drain still lacks real transport proof.

## What This Changes

The PR adds send-capability filtering to outbound channel plugin resolution, drains QQBot pending deliveries on gateway ready/resumed callbacks, and adds unit coverage for both paths.

## Best Possible Solution

Keep the broad resolver available for action/read callers, add send-capability filtering only to delivery/message-send resolution, then provide redacted live QQBot reconnect-drain proof or split that drain change out.

## Reproduction Assessment

Yes by source and issue evidence: current main returns the first matching channel plugin, setup-only entries are documented, and https://github.com/openclaw/openclaw/issues/88955 includes QQBot reconnect logs with outbound delivery failures.

## Solution Assessment

No; the PR is a plausible fix for setup-only send shadowing, but send-gating the shared resolver is broader than the plugin contract supports and the reconnect-drain half still needs real proof.
