## Summary

Keep open: the branch is a valuable fix for the linked QQBot reconnect delivery loss, but the gateway poll endpoint still bypasses the new operation-aware delivery resolver and can reject before reaching a poll-capable runtime plugin.

## What This Changes

The PR adds an operation-aware outbound delivery resolver, routes message/direct delivery through it, drains QQBot pending deliveries on reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Wire every concrete delivery entry point that needs a send operation, especially gateway polls, through the operation-aware delivery resolver while keeping the shared resolver for action/read paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: the linked issue has live QQBot reconnect logs, and the remaining review finding follows directly from the gateway poll handler still using the shared resolver before checking `sendPoll`.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the resolver split is the right ownership shape, but the PR must route the gateway poll entry point through that same operation-aware resolver before it is the best fix.
