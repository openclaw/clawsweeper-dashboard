## Summary

Keep open: the PR targets a real QQBot outbound message-loss regression and includes useful proof, but the current head still has a blocking text-send capability bug in the new delivery resolver.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send/poll preflight to it, drains pending QQBot deliveries on reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Split or parameterize delivery resolution by intent so sendMessage requires gateway/text capability while sendPoll can accept poll-only plugins, while keeping the QQBot reconnect drain and focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: a direct plugin with only outbound.sendPoll passes the new resolver, while the downstream direct text delivery handler still requires message.send.text or outbound.sendText and throws Outbound not configured.

## Solution Assessment

Is this the best way to solve the issue? No: the PR is in the right area, but the delivery resolver needs separate text and poll capability intent before it is the narrow maintainable fix.
