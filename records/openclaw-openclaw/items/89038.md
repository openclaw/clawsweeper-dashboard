## Summary

Keep open: the QQBot reconnect fix is useful and the supplied live/log proof is strong, but latest head still has a blocking outbound resolver defect because the new delivery gate treats poll-only plugins as valid for text send preflight.

Keep the QQBot reconnect drain and setup-shell filtering, but split delivery resolution by send intent so text sends require text/gateway capability while polls can use outbound.sendPoll, with ordered fallback regression coverage.

Full review comments:

- **[P1] Split text and poll delivery qualification:** `src/infra/outbound/channel-resolution.ts:218`
  - body: `channelPluginCanSend` treats `outbound.sendPoll` as enough for the shared delivery resolver, but `sendMessage` now uses that resolver before text delivery. If a loaded/setup plugin has only `sendPoll` and a later runtime plugin can send text, preflight stops on the poll-only plugin and `createPluginHandler` still returns null because it requires `message.send.text` or `outbound.sendText`, so text sends can fail with `Outbound not configured`. Make the resolver intent-specific and add the ordered poll-only-before-text fallback regression.
  - confidence: 0.91
