## Summary

Keep open: the PR targets a credible QQBot reconnect message-loss regression and has useful proof, but the new delivery predicate still uses one send-capability decision for both text and poll preflight, leaving a concrete hot-path false-positive risk before merge.

## What This Changes

The PR adds a delivery-only outbound channel resolver, routes send preflight through it, and drains matching QQBot pending deliveries on gateway ready/resume with regression tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell skip, but split or parameterize delivery resolution so text sends require text or gateway capability while poll sends can accept poll capability.

## Reproduction Assessment

Yes, source-level: the PR-head predicate accepts `outbound.sendPoll`, `sendMessage()` uses that same resolver, and direct delivery still requires text capability in `createPluginHandler()`. I did not rerun a live QQBot 4009 timeout.

## Solution Assessment

No, not yet: the overall QQBot repair shape is reasonable, but the capability predicate should distinguish text-send preflight from poll preflight so the fix does not add another ambiguous delivery selection path.
