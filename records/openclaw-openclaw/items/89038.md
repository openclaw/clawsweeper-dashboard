## Summary

Keep open for maintainer review: current main and the latest release do not include this fix, the linked QQBot message-loss bug is source-reproducible, and I found no discrete blocking patch defect, but the plugin-resolution compatibility contract and queued replay behavior need maintainer acceptance before merge.

## What This Changes

The branch adds a delivery-only send-capable outbound resolver, switches message send/poll preflight to it, drains QQBot pending deliveries on ready/resumed, and adds regression coverage.

## Best Possible Solution

Have maintainers explicitly accept the delivery resolver compatibility contract and QQBot queued-replay semantics, then land this PR to close https://github.com/openclaw/openclaw/issues/88955.

## Reproduction Assessment

Yes, source inspection plus the linked issue logs give a high-confidence reproduction path: current main can resolve a setup-only QQBot shell for send preflight, then durable delivery throws `Outbound not configured for channel: qqbot` after queue enqueue.

## Solution Assessment

Yes, this is the best reviewed shape: split shared/read resolution from delivery-only send-capable resolution, preserve gateway-mode plugins, and use the existing channel-owned drain helper for QQBot reconnects.
