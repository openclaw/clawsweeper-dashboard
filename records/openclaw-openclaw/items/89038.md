## Summary

Keep open: the QQBot reconnect fix is still needed, and the supplied live proof is strong, but the current branch still has a blocking shared resolver bug because text sends and polls use the same send-capability predicate.

## What This Changes

The PR adds a delivery-only outbound channel resolver, wires it into outbound message/poll sends, drains QQBot pending deliveries on ready/resumed reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell fix, but split delivery resolution by requested operation so text/message sends require text or gateway capability and polls require poll or gateway capability before merge.

## Reproduction Assessment

Yes, source-level. The PR head lets a poll-only direct plugin pass the same preflight used by text sends, while current delivery handler construction still requires a text sender and otherwise throws `Outbound not configured`.

## Solution Assessment

No. The direction is right, but the best fix is operation-specific delivery resolution rather than one send-capability predicate shared by text and poll paths.
