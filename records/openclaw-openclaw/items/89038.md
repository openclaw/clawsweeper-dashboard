- `merge-risk: 🚨 message-delivery`: The change affects outbound channel resolution and unproven reconnect queue draining, both of which can suppress or defer user-visible deliveries.

## Summary

Keep open: the live proof now supports the reported QQBot outbound-resolution symptom, but the patch applies a send-only predicate to a shared resolver that current action-dispatch paths also use for non-send plugin actions, so it is not merge-ready.

## What This Changes

The PR adds send-capability filtering to outbound channel plugin resolution, drains QQBot pending deliveries on gateway ready/resumed callbacks, and adds unit coverage for both paths.

## Best Possible Solution

Keep plugin resolution broad for action/read callers, add a delivery-specific send-capable selection path that scans candidates, and either live-prove the QQBot drain or split it into a follow-up.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the first loaded/active/bundled plugin without checking send capability, and https://github.com/openclaw/openclaw/issues/88955 includes 4009 reconnect logs; I did not run a live QQBot repro.

## Solution Assessment

No. The send-capability guard is directionally useful, but it is applied at the wrong layer; the best fix keeps shared plugin resolution available for action/read callers and narrows send filtering to delivery paths.
