## Summary

Keep this PR open: current main still lacks the send-capability-gated delivery resolver and QQBot reconnect drain, and the patch addresses a P1 message-loss regression with strong supplied live proof. I found no discrete blocking correctness finding, but the reconnect replay and plugin-resolution semantics remain compatibility/message-delivery sensitive enough to need QQBot or outbound owner signoff before merge.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips non-send setup shells, switches send preflight to it, and drains matching QQBot pending deliveries on gateway ready/resumed with focused tests.

## Best Possible Solution

Land only after a QQBot or outbound owner accepts the reconnect replay scope and delivery resolver compatibility semantics; keep the shared resolver for read/action callers and use the send-capability resolver only for delivery paths.

## Reproduction Assessment

Yes, from source and the linked issue logs: current main can resolve a setup-only QQBot shell for send preflight while the setup plugin has no send surface, leading to the reported outbound-not-configured path after reconnect. I did not run a live QQBot repro in this read-only review.

## Solution Assessment

Mostly: the resolver split is the right narrow layer for setup-shell shadowing and keeps read/action callers on the shared resolver. The reconnect replay piece is plausible but needs owner acceptance because it intentionally changes pending-delivery retry timing and duplicate-risk tradeoffs.
