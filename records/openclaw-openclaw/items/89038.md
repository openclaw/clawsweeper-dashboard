## Summary

Keep open for maintainer review: the PR addresses a real QQ Bot message-loss regression, current main does not contain the fix, and the contributor supplied strong live/log proof, but the resolver split and reconnect drain remain compatibility- and delivery-sensitive surfaces.

## What This Changes

The branch adds a send-capability-filtered outbound resolver for delivery paths, switches message send/poll resolution to it, and drains matching QQ Bot pending deliveries on gateway ready/resumed with regression tests.

## Best Possible Solution

Land the resolver split and QQ Bot drain after maintainers explicitly accept the plugin-selection compatibility boundary and the remaining QQ Bot topology proof gap.

## Reproduction Assessment

Yes. The linked report supplies concrete QQ Bot reconnect logs on OpenClaw 2026.5.28, and source inspection shows current main can throw `Outbound not configured for channel` after write-ahead enqueue when resolution returns a non-sending plugin; I did not run a live QQ Bot repro locally.

## Solution Assessment

Yes, the split between shared resolution and delivery-only resolution is the right owner boundary for this bug. The reconnect drain is a reasonable upgrade recovery path, with compatibility review needed before merge rather than an automated repair.
