## Summary

Keep open: this is a credible QQBot reconnect message-loss fix, but the current PR head still has a blocking resolver defect where one generic delivery predicate can accept a poll-only direct plugin for text sends.

## What This Changes

The PR adds a send-capability-gated outbound resolver for delivery paths and drains QQBot pending deliveries on gateway ready/resumed callbacks, with outbound resolver and QQBot reconnect-drain tests.

## Best Possible Solution

Split delivery resolution by operation so text sends require text or gateway capability, poll sends require poll or gateway capability, and the QQBot reconnect drain/setup-shell recovery remains covered by focused regressions.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the QQBot drain and setup-shell filtering are useful, but the shared text/poll delivery predicate is still too broad and should be split before merge.
