## Summary

Keep open: the QQBot fix is aimed at the right root cause, but the current patch can reject setup-shell-only channels in `sendMessage` before the existing lazy runtime bootstrap path runs.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, routes direct message send paths through it, and drains matching QQBot pending deliveries on gateway ready/resumed with regression tests.

## Best Possible Solution

Preserve the split resolver, but make direct send/poll preflight bootstrap-aware or defer send-capability gating until the delivery path so setup shells do not block runtime promotion.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the patch regression: a setup-only loaded channel plugin makes `resolveOutboundChannelPluginForDelivery` return undefined in `message.ts` before `loadBootstrappedOutboundAdapter` can bootstrap the runtime sender. I did not run a live QQBot reproduction in this read-only review.

## Solution Assessment

No: the split delivery resolver is the right direction, but the current `message.ts` use is not the safest fix because it moves send-capability gating ahead of lazy bootstrap. The safer solution keeps shared action/read resolution unchanged and makes send preflight bootstrap-aware.
