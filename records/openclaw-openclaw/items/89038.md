## What This Changes

The PR adds a delivery-only outbound channel resolver, uses it for send/poll preflight, preserves gateway-mode routing, and drains matching QQBot pending deliveries on ready/resumed reconnect callbacks.

## Best Possible Solution

Land the delivery-only resolver and QQBot reconnect drain only after current-head live QQBot proof and channel/outbound owner acceptance confirm the resolver scope and retry timing are safe.

## Reproduction Assessment

Yes, from source and the linked report: current main can throw "Outbound not configured for channel" when handler creation receives a non-send setup shell, and the issue logs show that path after QQBot WebSocket reconnect. I did not run a live QQBot reproduction in this read-only review.

## Solution Assessment

Yes, this is the right shape if current-head live proof checks out: keep the shared resolver available for read/action callers, add a delivery-only send-capability path, and put reconnect draining in the QQBot plugin via the SDK drain surface. Gating the shared resolver would be a worse boundary because action-only/setup callers still need non-send plugin metadata.
