## Summary

Keep open: the PR targets a real QQBot reconnect message-loss bug and has useful proof, but the current head is not merge-ready because one delivery predicate still qualifies poll-only plugins for the text-send preflight path.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, routes message send preflight through it, drains QQBot pending deliveries on ready/resumed, and adds resolver plus QQBot tests.

## Best Possible Solution

Split delivery resolution by send intent so text sends require text or gateway capability, poll sends require poll capability, while preserving setup-shell skipping, external bootstrap, and QQBot reconnect drain behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: a loaded direct plugin with only `outbound.sendPoll` is accepted by the PR resolver, then direct text delivery reaches `createPluginHandler`, which still requires a text sender and returns null.

## Solution Assessment

Is this the best way to solve the issue? No as written; the setup-shell skip and QQBot drain are the right shape, but text and poll send qualification need separate intent-aware checks before merge.

Full review comments:

- **[P1] Split text and poll delivery qualification:** `src/infra/outbound/channel-resolution.ts:218`
  - body: This predicate treats `outbound.sendPoll` as enough for the delivery resolver used by `sendMessage`. Because `resolveCurrent()` returns a loaded plugin before checking runtime registries, a loaded/setup plugin with only `sendPoll` can shadow a later text-capable runtime; `sendMessage` then takes the direct path and `createPluginHandler` returns null unless `message.send.text` or `outbound.sendText` exists, producing `Outbound not configured` for ordinary text sends. Keep poll qualification on the poll path, but require text/gateway capability for text sends and cover that ordering case.
