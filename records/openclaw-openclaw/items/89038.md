## Summary

Keep open: the latest head still has a blocking resolver-intent bug where a poll-only direct plugin can satisfy text-send preflight even though downstream text delivery still requires a text sender.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send/poll preflight to it, drains pending QQBot deliveries on reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Split or parameterize delivery resolution by intent so text sends require gateway/text capability while polls can accept poll-only plugins, keeping the QQBot reconnect drain and focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a direct plugin with only `outbound.sendPoll` passes the new resolver, while the downstream direct text handler still requires `message.send.text` or `outbound.sendText` and throws `Outbound not configured`.

## Solution Assessment

Is this the best way to solve the issue? No: the PR is in the right owner boundary, but one undifferentiated delivery predicate cannot safely represent both text-send and poll-send intent.
