## Summary

Needs changes before merge: the QQBot drain and setup-shell skip are useful and the real-behavior proof is strong, but the PR still uses one delivery-capability gate for both text sends and poll sends, leaving a source-reproducible path where normal text delivery can select a direct poll-only plugin and fail with the same outbound-not-configured class of error.

## What This Changes

The PR adds a delivery-only outbound channel resolver, routes message send preflight through it, drains QQBot pending deliveries on ready/resumed reconnect callbacks, and adds regression tests.

## Best Possible Solution

Keep the resolver split, but make delivery qualification operation-specific: text sends require text-capable or gateway delivery, while poll sends can accept poll-capable adapters.

## Reproduction Assessment

Yes from source, not from a live run in this review. The linked issue has live QQBot reconnect logs, and the PR head still has a clear source path where `sendMessage` can accept a poll-only direct plugin while the delivery handler requires text capability.

## Solution Assessment

No, not as written. The split between shared and delivery resolution is the right layer, but the best fix must split text and poll delivery qualification instead of using one `channelPluginCanSend` predicate for both operations.
