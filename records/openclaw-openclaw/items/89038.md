## Summary

Keep open: this PR still targets a real QQBot reconnect message-loss bug, but the current patch needs a capability-filter fix before it is merge-ready, and the reconnect-drain proof still needs maintainer acceptance or stronger callback-drain evidence.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send preflight to it, and drains QQBot pending deliveries from gateway ready/resumed callbacks.

## Best Possible Solution

Land only after the delivery resolver preserves operation-specific text capability for text sends, keeps existing plugin activation behavior intact, and maintainers accept or receive stronger reconnect-drain proof.

## Reproduction Assessment

Yes for the linked bug at source/log level, but not as a fresh live current-main reproduction by this review. The canonical issue has real QQBot reconnect logs and current source shows the outbound-not-configured path when no text-capable handler resolves.

## Solution Assessment

No as currently written. Splitting shared and delivery resolution is the right direction, but the new delivery gate must be operation-specific so non-text send surfaces cannot pass text-send preflight.
