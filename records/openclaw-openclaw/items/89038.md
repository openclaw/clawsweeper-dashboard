## Summary

Keep open: the QQBot reconnect fix is useful and has strong proof, but the latest head still has a blocking resolver defect because a poll-only direct plugin can satisfy text-send preflight even though direct delivery later requires a text sender.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, routes message send/poll preflight through it, and drains matching QQBot pending deliveries on gateway ready/resume with regression tests.

## Best Possible Solution

Split the delivery resolver by intent so text sends require text or gateway capability while poll sends require poll capability, preserving setup-shell skipping and external bootstrap behavior with focused regression coverage.

## Reproduction Assessment

Yes, source-level: the linked issue has live QQBot 4009 reconnect logs, and the remaining PR defect is traceable from the new resolver predicate through `sendMessage` to the direct handler's text-sender requirement. I did not run a fresh live QQBot reproduction in this read-only review.

## Solution Assessment

No: the setup-shell skip and QQBot drain are the right direction, but a single send-capability predicate for both text and poll is not the best fix because the downstream contracts differ.
