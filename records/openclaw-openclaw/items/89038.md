## Summary

Keep open for maintainer or QQBot owner review. The resolver change looks like a valid fix and the supplied live proof is strong, but the reconnect drain intentionally changes queued-message replay behavior and still needs explicit owner acceptance before merge.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver for send preflight and drains matching QQBot pending delivery entries from ready/resumed callbacks, with focused outbound and QQBot tests.

## Best Possible Solution

Land the resolver and QQBot recovery only after a channel owner accepts the replay semantics; otherwise narrow the reconnect drain while keeping the setup-shell delivery fix.

## Reproduction Assessment

Yes, source and report evidence are high-confidence: the linked QQBot report shows repeated 4009 reconnects followed by outbound-not-configured failures, and the PR body includes live patched QQBot proof. I did not rerun live QQBot credentials in this read-only review.

## Solution Assessment

Mostly yes for the resolver: splitting shared read/action lookup from send-capability delivery lookup is the right ownership boundary. The reconnect drain is a plausible upgrade recovery path, but it needs owner acceptance because replay semantics are a product and delivery-safety decision.
