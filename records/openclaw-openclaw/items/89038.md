## Summary

Keep open: this PR addresses a real QQBot reconnect message-loss report, but the diff still send-gates a shared channel resolver used by action-only plugin paths, and the new reconnect-drain behavior is explicitly not live-proven.

## What This Changes

The PR adds send-capability filtering to outbound channel plugin resolution, drains QQBot pending deliveries on gateway ready/resumed callbacks, and adds focused unit tests for both paths.

## Best Possible Solution

Keep the broad plugin resolver usable for action/read facets, add send-capability filtering only to send/delivery resolution, and either provide redacted live QQBot reconnect-drain proof or split that drain change into a follow-up.

## Reproduction Assessment

Yes by source and issue evidence: current main returns the first matching loaded channel plugin, while https://github.com/openclaw/openclaw/issues/88955 includes timed QQBot reconnect logs with the exact outbound error. I did not independently rerun a live QQBot 4009 reconnect in this read-only review.

## Solution Assessment

No; the setup-only send shadowing problem is real, but the current fix puts send capability gating in a broad resolver that action paths also use. The safer fix is a send-only predicate at delivery/message-send resolution plus action-only regression coverage.
