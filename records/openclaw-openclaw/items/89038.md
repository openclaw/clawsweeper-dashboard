## Summary

Keep open: the PR is still the active fix candidate for the linked QQBot reconnect message-loss regression, but the latest head still has a blocking outbound resolver compatibility defect and the reconnect callback drain proof remains weaker than the changed behavior deserves.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches `sendMessage`/`sendPoll` preflight to it, and drains QQBot pending deliveries from gateway ready/resumed callbacks.

## Best Possible Solution

Preserve the existing shared resolver external-channel bootstrap behavior in the delivery resolver, keep the shared/action resolver ungated, and have maintainers either accept the seeded recovery proof or ask for callback-drain logs before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review finding by source: PR head line 279 bypasses the current external-channel bootstrap helper, and current tests explicitly protect that bootstrap behavior for sibling resolvers. For the linked QQBot production bug, the issue has real logs and the source path is plausible, but I did not run a fresh live QQBot 4009 reproduction.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the split shared-versus-delivery resolver is the right shape for setup shells, but the delivery resolver should reuse the existing external-channel normalization/bootstrap path rather than creating a narrower one. The QQBot drain also needs direct callback proof or explicit maintainer acceptance of seeded recovery proof.
