## Summary

Keep open: current main still lacks the delivery-only resolver and qqbot reconnect drain, and this PR is the active fix path for the linked P1 message-loss regression. ## What This Changes

The PR adds a send-capability-gated outbound delivery resolver used by message send/poll and delivery adapter lookup, plus qqbot onReady/onResumed pending-delivery drains with focused tests.

## Best Possible Solution

Land this PR after maintainer review and CI with the resolver split intact, then treat broader multi-account QQBot live coverage as follow-up unless owners require it before merge.

## Reproduction Assessment

Yes. Source inspection plus the linked issue logs show current main can resolve a non-sending qqbot stub and then throw `Outbound not configured for channel: qqbot` before acking the queued delivery; I did not run a local live QQBot repro.

## Solution Assessment

Yes. Splitting shared and delivery-only resolution is the best fix shape because gating the shared resolver would risk action/target/read callers, and the qqbot drain follows the existing plugin SDK reconnect-drain pattern.
