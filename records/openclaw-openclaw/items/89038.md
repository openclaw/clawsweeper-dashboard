## Summary

Keep open; this is a focused fix for a P1 QQBot message-loss regression, but it is not merge-ready because the new reconnect drain misses the exact pending-error string left by the reported failure, so recently failed entries can be skipped by backoff instead of drained.

## What This Changes

This PR splits outbound channel resolution into shared and send-capable delivery paths, routes message send and poll delivery through the delivery resolver, and adds QQBot reconnect pending-delivery drains with tests.

## Best Possible Solution

Merge only after the QQBot reconnect drain explicitly handles the reported pending-error path and the PR body includes redacted proof or a maintainer-accepted explanation for any remaining live-drain proof gap.

## Reproduction Assessment

Yes for the PR finding by source inspection: the selector in `extensions/qqbot/src/channel.ts` does not match the reported lastError, and `src/infra/outbound/delivery-queue-recovery.ts` applies backoff when `bypassBackoff` is false. I did not run a live QQBot repro in this read-only review.

## Solution Assessment

No: the resolver split is the right owner-boundary shape, but the reconnect drain should also bypass the exact `Outbound not configured for channel: qqbot` pending-error path before merge.
