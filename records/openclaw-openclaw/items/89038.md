## Summary

Keep open: this is a useful P1 QQBot message-loss fix attempt with strong live proof, but the patch is not merge-ready because it gates the wrapper preflight while the queued direct-delivery/reconnect recovery path can still select a non-sending setup outbound adapter and hit the same `Outbound not configured` failure.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches `sendMessage`/`sendPoll` preflight to it, drains QQBot pending deliveries on ready/resumed reconnect callbacks, and adds focused resolver/QQBot tests.

## Best Possible Solution

Fix the actual direct delivery and reconnect recovery path so non-sending setup outbound adapters cannot shadow send-capable runtime plugins, then keep the shared action/target resolver behavior intact and land with focused delivery/drain regression coverage.

## Reproduction Assessment

Yes at source level. The PR's own test uses a setup shell shaped as `{ outbound: { deliveryMode: "direct" } }`, while the real delivery loader returns any non-undefined pinned outbound object and the real handler still throws when that object has no text sender.

## Solution Assessment

No, not yet. The resolver split is a good direction, but the best fix must also cover the actual queued direct-delivery/reconnect path instead of only the mocked `sendMessage` preflight.
