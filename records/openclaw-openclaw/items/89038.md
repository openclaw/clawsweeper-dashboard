## Summary

Keep this PR open for maintainer review: the patch appears to address the QQBot outbound message-loss regression with no discrete blocking code finding, but it intentionally changes send-path plugin resolution and adds reconnect recovery behavior that should be accepted by the channel/runtime owners before merge.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips non-sending channel plugin entries, switches send paths to that resolver, and drains matching QQBot pending deliveries on gateway ready/resumed with tests.

## Best Possible Solution

Land the scoped delivery resolver plus QQBot reconnect drain after channel/runtime owners accept the compatibility and reconnect-drain scope.

## Reproduction Assessment

Yes, source and report evidence give a high-confidence reproduction path: current main still resolves send paths through the shared plugin resolver and lacks QQBot reconnect drain, while the linked issue reports QQBot close-code 4009 reconnect followed by outbound heartbeat/cron failures.

## Solution Assessment

Yes, this is the best observed fix shape: filtering only the delivery resolver avoids breaking shared action/metadata resolution, and the QQBot drain covers pending entries left by the shipped failure mode.
