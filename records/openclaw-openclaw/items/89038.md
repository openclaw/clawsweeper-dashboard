## Summary

Keep open: the QQBot repair is useful and proof-backed, but the PR still lets the text-send preflight accept a poll-only direct plugin, which can select a plugin that the downstream text delivery handler later rejects.

## What This Changes

The branch adds a send-capability-gated outbound resolver for delivery paths, routes message sends through it, drains QQBot pending deliveries on reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Keep the QQBot reconnect drain and setup-shell skip, but split or parameterize delivery resolution so text sends require text/gateway capability while poll sends can accept poll capability.

## Reproduction Assessment

Yes, source-level: the PR-head predicate accepts outbound.sendPoll, sendMessage uses the same resolver, and direct delivery still requires text capability in createPluginHandler. I did not rerun a live QQBot 4009 timeout.

## Solution Assessment

No, not yet: the QQBot repair shape is reasonable, but the capability predicate should distinguish text-send preflight from poll preflight so the fix does not add another ambiguous delivery selection path.
