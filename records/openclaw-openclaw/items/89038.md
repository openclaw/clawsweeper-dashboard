## Summary

Keep open: the PR is still the active candidate for the linked QQBot reconnect message-loss bug, and current main does not have the QQBot reconnect drain. It is not merge-ready because the new delivery predicate is still broader than the downstream text-send contract, and the supplied live proof does not directly show the new onReady/onResumed drain recovering a pending delivery.

## What This Changes

The PR adds a send-capability-gated outbound channel resolver for delivery preflight, wires sendMessage/sendPoll through it, and adds QQBot onReady/onResumed pending-delivery drains plus tests.

## Best Possible Solution

Land a narrow fix that keeps one operation-aware outbound resolution path, preserves shared resolver semantics for non-send callers, and either provides direct callback-drain proof or records explicit maintainer acceptance of the seeded recovery proof.

## Reproduction Assessment

Yes for the PR-introduced resolver defect: source inspection shows non-text send methods can satisfy the new delivery predicate while createPluginHandler still requires text support for direct text sends. The original QQBot reconnect bug has linked live logs, but I did not run a fresh live QQBot reproduction in this read-only review.

## Solution Assessment

No. The reconnect recovery direction is useful, but the new delivery resolver should be folded into the existing activated resolver or split by operation before merge.
