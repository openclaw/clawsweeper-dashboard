## What This Changes

The PR adds operation-aware outbound delivery plugin resolution for send paths, aligns direct delivery adapter loading with that resolver, and drains matching QQBot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Merge this PR or an equivalent maintainer-approved fix after maintainers accept operation-aware send resolution as the intended shared delivery behavior and QQBot reconnect drain timing as the right recovery policy.

## Reproduction Assessment

Yes at source level. Current main still has a capability-blind direct adapter path that can select a non-send outbound shell and then throw Outbound not configured, and the linked issue provides live QQBot 4009 reconnect logs for the same failure.

## Solution Assessment

Yes, this appears to be the best bounded fix. It keeps the shared resolver available for non-send callers while making send preflight and direct delivery share one operation-aware send-capable resolver, avoiding a QQBot-only workaround.
