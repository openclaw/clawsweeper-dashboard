## Summary

Keep open: this PR is the active fix candidate for the linked QQBot message-loss regression, but the new delivery gate is broader than the handler contract and the supplied logs still do not directly prove the reconnect-drain callback path.

## What This Changes

The PR adds a send-capability-gated outbound resolver for delivery, switches message send/poll preflight to it, and drains matching QQBot pending deliveries from gateway ready/resumed callbacks with focused tests.

## Best Possible Solution

Keep the focused QQBot fix path, but make the delivery resolver capability-specific and land only after direct redacted reconnect-drain proof or explicit maintainer acceptance of that proof gap.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a fresh live current-main repro in this read-only review; the linked issue has real QQBot 4009 logs and the current source still has the matching outbound-not-configured throw path.

## Solution Assessment

Is this the best way to solve the issue? Not yet: splitting shared resolution from delivery resolution is the right shape, but the resolver must be capability-specific and the reconnect drain still needs direct live proof or maintainer acceptance.
