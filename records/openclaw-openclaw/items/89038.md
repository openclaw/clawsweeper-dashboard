## Label Justifications

- `P1`: The PR targets a reported qqbot reconnection regression that can stop outbound heartbeat/cron delivery and leave pending messages undelivered for real users.
- `merge-risk: 🚨 compatibility`: The patch changes shared channel plugin resolution semantics for delivery paths, which can affect setup-only, action-only, external, or gateway-mode plugin registrations.
- `merge-risk: 🚨 message-delivery`: The patch changes outbound plugin selection and reconnect queue draining, so an incorrect merge could still drop, defer, or duplicate channel messages.

## Summary

Keep open for normal merge review: current main still has the resolver/reconnect gap, while this PR has sufficient live-log proof and no blocking review findings; the remaining caution is that it changes a shared channel delivery seam and qqbot queue recovery behavior.

## What This Changes

The PR adds a send-capability-filtered outbound resolver for delivery paths, switches message delivery callers to it, and drains matching qqbot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Land the delivery-only resolver split and scoped qqbot reconnect drain after maintainer review confirms the shared channel delivery compatibility risk is acceptable.

## Reproduction Assessment

Yes, from source and reporter logs, though I did not live-reproduce QQ Bot here. Current main still resolves the first plugin for message adapters and has no qqbot reconnect drain, matching the linked failure path.

## Solution Assessment

Yes. The delivery-only resolver is the narrow fix because it avoids gating action and metadata callers while making send paths require a real send adapter; the reconnect drain is a scoped upgrade safeguard for already-pending qqbot entries.
