## Summary

Keep open: this is a plausible focused bug-fix PR, but it still needs maintainer review because the supplied live proof does not directly show the new QQBot onReady/onResumed reconnect-drain callback recovering a pending delivery.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send preflight to it, and drains matching QQBot pending deliveries from ready/resumed callbacks.

## Best Possible Solution

Land the resolver split and QQBot drain only after maintainers are satisfied that activation-aware channel behavior is preserved and queued QQBot entries recover safely on reconnect.

## Reproduction Assessment

Yes, source-reproducible: the linked reports show real QQBot 4009 reconnect failures, and current delivery code leaves pre-platform send failures pending for recovery. I did not run a fresh live QQBot timeout in this read-only review.

## Solution Assessment

Mostly yes: splitting shared resolution from delivery-only send capability is the right boundary, but the reconnect-drain half still needs direct callback-path proof or explicit maintainer acceptance before merge.
