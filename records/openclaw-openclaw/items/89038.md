## Summary

Keep open for maintainer review: the patch appears correct with no line-level blocking findings and sufficient real behavior proof, but the delivery-qualification and QQBot replay semantics are compatibility-sensitive enough to need owner acceptance before merge.

## What This Changes

The PR adds delivery-only outbound channel resolution that filters non-send setup shells, uses it from send/poll paths, and drains QQBot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Land only after outbound or QQBot owners accept the delivery resolver contract and reconnect replay policy, keeping the focused tests and live proof attached to the PR.

## Reproduction Assessment

Yes. Source inspection plus the linked report gives a high-confidence path for the reported failure: queued QQBot outbound sends can hit the current outbound-not-configured throw when resolution returns a setup-only or non-send plugin; I did not run a live QQBot repro in this read-only review.

## Solution Assessment

Yes, this is an acceptable best-fit repair if owners accept the replay policy: it keeps read/action resolution broad, gates only delivery paths, preserves gateway-mode plugins, and adds focused reconnect drain handling for queued QQBot deliveries.
