## Summary

Keep open: this PR targets a real QQBot reconnect message-loss regression, but the new delivery resolver still bypasses the current external-channel bootstrap path and the reconnect-drain proof remains indirect.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send preflight to it, and drains QQBot pending deliveries from gateway ready/resumed callbacks.

## Best Possible Solution

Reuse the existing external-channel normalization/bootstrap behavior in the delivery resolver, keep setup-only send gating, and settle the QQBot callback-drain proof or maintainer acceptance before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review finding by source: PR head rejects non-deliverable external channel ids before the current bootstrap path, while current tests protect that behavior for the shared resolver. The linked QQBot production bug has real logs, but I did not run a fresh live QQBot 4009 reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the shared-versus-delivery resolver split is the right shape for setup shells, but the delivery resolver should reuse the existing external-channel bootstrap path. The reconnect drain also needs direct callback proof or explicit maintainer acceptance of the seeded recovery proxy.
