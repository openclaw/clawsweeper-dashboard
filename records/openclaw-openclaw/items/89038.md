## Summary

Keep this PR open: the resolver split looks like the right boundary and has focused tests, but the branch also adds QQBot reconnect-time queue replay and the PR body explicitly says that new drain behavior was not live-verified.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches outbound message delivery paths to it, and triggers QQBot pending-delivery drains from gateway ready/resumed callbacks with tests.

## Best Possible Solution

Land the resolver fix with the reconnect-drain behavior only after redacted live QQBot evidence shows an actual queued pending delivery drains after reconnect, or split the unverified drain hook into a follow-up.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves the first matching channel plugin without send-capability filtering, and the linked report shows QQBot reconnect followed by `Outbound not configured for channel: qqbot`. I did not run a live QQBot reproduction in this read-only review.

## Solution Assessment

Unclear until proof is complete: the resolver split is the narrowest maintainable boundary I found, but the reconnect drain addition is only mock-tested. The best path is to prove the drain live or split it out.
