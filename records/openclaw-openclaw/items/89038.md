## Summary

Keep this PR open: the patch addresses a real QQBot outbound reconnect regression and has strong live-log proof, but merge still needs maintainer acceptance of reconnect replay and plugin-deliverability compatibility risk.

## What This Changes

Adds a delivery-only outbound channel resolver, switches sendMessage/sendPoll preflight to it, and drains matching QQBot pending deliveries on gateway ready/resumed.

## Best Possible Solution

Merge after QQBot/outbound owner acceptance, keeping shared lookup ungated for read/action callers and delivery lookup gated by send capability.

## Reproduction Assessment

Yes. The linked issue provides concrete QQBot reconnect logs, and current main still has the shared-resolver preflight path that can let setup-only entries shadow send-capable runtime entries.

## Solution Assessment

Yes, with maintainer acceptance of replay risk. Splitting shared lookup from delivery lookup is the narrowest maintainable fix for send preflight while preserving action/read callers; the reconnect drain is a reasonable upgrade recovery safeguard but needs owner signoff.
