## Summary

Keep this PR open: the code direction looks like a reasonable fix for the QQBot setup-shell delivery regression, but the external real-behavior proof still does not show the current PR head delivering the reported heartbeat/cron path after a QQBot reconnect.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips setup-only send-ineligible plugins, switches send/poll preflight to it, and drains matching QQBot pending deliveries on gateway ready/resumed callbacks.

## Best Possible Solution

Merge this shape only after current-head QQBot reconnect delivery proof or an explicit owner proof override, keeping setup/action/read resolution separate from delivery-only send resolution.

## Reproduction Assessment

We have a source-level reproduction path, not a live current-main run: current main can return a setup shell from the shared resolver and the delivery core throws `Outbound not configured for channel` when no send handler is available. The linked issue logs tie that failure to QQBot reconnects after 4009 timeouts.

## Solution Assessment

Yes, this is the best code shape I saw: adding a delivery-only resolver is safer than gating the shared resolver because action/read/setup callers still need setup-only or action-only plugins. The remaining blocker is proof/owner confidence, not a better implementation location.
