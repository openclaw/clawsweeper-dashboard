## Summary

Keep this PR open with a blocking fix required: the send-capability idea is pointed at the right bug, but the branch only gates the `sendMessage` preflight while the durable delivery and reconnect-drain path still reloads `plugin.outbound` through the unchanged adapter loader.

## What This Changes

The branch adds a send-capability delivery resolver, switches message send/poll preflight to it, adds QQBot ready/resumed pending-delivery drains, and adds focused outbound/QQBot tests.

## Best Possible Solution

Apply the same send-capability/fallback rule at the actual outbound adapter resolution used by direct delivery and queue recovery, while keeping the shared non-send resolver ungated for setup/action callers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the linked issue has live QQBot 4009 reconnect logs, and the current/PR source path shows a setup-only outbound object can still reach the durable delivery `Outbound not configured` throw.

## Solution Assessment

Is this the best way to solve the issue? No: splitting shared and delivery resolution is the right direction, but the gate must be applied where direct delivery and queue recovery actually load the outbound adapter.
