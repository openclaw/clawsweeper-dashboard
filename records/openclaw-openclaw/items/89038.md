## Summary

## What This Changes

The PR adds a delivery-only send-capability resolver for outbound channel selection, switches message send/poll delivery paths to it, and drains qqbot pending deliveries from ready/resumed reconnect callbacks with focused tests.

## Best Possible Solution

Land the focused resolver split and qqbot reconnect drain after maintainer acceptance of the compatibility and replay risk, keeping non-send callers on the shared resolver.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main resolves the first loaded/pinned channel plugin without send-capability checks, then delivery throws `Outbound not configured for channel` when that entry has no send handler. The linked issue and PR body add live qqbot logs for the same observable failure and after-fix path.

## Solution Assessment

Yes, this is the best observed fix shape: add a delivery-only send-capability resolver while keeping the shared resolver ungated for action/target/read callers, then drain upgrade-path pending qqbot entries through the existing delivery queue runtime. The safer alternative, gating the shared resolver, was already avoided because it would risk action-only plugin compatibility.
