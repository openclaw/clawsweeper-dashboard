## Summary

Keep this PR open, but the current head is not merge-ready: the live diff still applies send-capability filtering to the shared resolver, which can break non-send action/metadata callers, and the QQBot reconnect-drain path remains unproven in a real reconnect.

## What This Changes

## Best Possible Solution

Preserve the shared resolver contract, add a delivery-only resolver for actual send/message-adapter paths, update callers/tests around that boundary, and require redacted live proof that a queued QQBot delivery drains after reconnect or split the drain into a later PR.

## Solution Assessment
