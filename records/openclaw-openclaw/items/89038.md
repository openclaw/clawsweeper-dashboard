## Summary

Keep this PR open for maintainer review: source inspection supports the reported QQBot resolver-shadowing/message-loss regression, and the patch is a bounded fix with strong default-account live proof, but core delivery resolver compatibility and QQBot multi-account reconnect-drain scope still need maintainer acceptance before merge.

## What This Changes

The PR adds a delivery-only outbound channel resolver that skips non-sending channel plugin entries, switches send paths to it, and drains matching QQBot pending deliveries on gateway ready/resumed with tests.

## Best Possible Solution

Land the resolver split and QQBot reconnect drain after maintainers accept the plugin-selection compatibility risk and the QQBot proof scope, or ask for targeted multi-account/shard proof before merging.

## Reproduction Assessment

Yes. The linked issue provides QQBot reconnect logs, and current main source shows a setup-only plugin can be returned before send capability is checked, leading to the observed outbound-not-configured throw and pending queue entry.

## Solution Assessment

Yes, with maintainer acceptance. Splitting shared resolution from delivery-only send-capable resolution is the narrowest way to preserve action callers while fixing delivery, and the QQBot drain is in the plugin owner boundary via the SDK seam.
