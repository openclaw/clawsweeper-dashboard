## Summary

Keep this PR open for maintainer review and likely merge: current main and the latest shipped release still have the setup-only resolver/drain gap, and the proposed patch is focused with sufficient live QQBot log proof and no blocking code findings. The remaining concern is merge safety because outbound plugin resolution and queued delivery replay are compatibility-sensitive message-delivery paths.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send/poll delivery paths to it, and drains matching qqbot pending deliveries on gateway ready/resumed callbacks with focused tests.

## Best Possible Solution

Merge this bounded fix after maintainer review confirms the delivery-only resolver boundary and qqbot replay scope are acceptable for existing channel/plugin setups.

## Reproduction Assessment

Yes, from source and issue logs: current main and v2026.5.28 can resolve a setup-only channel plugin through the shared outbound message path, and qqbot reconnect callbacks do not drain queued deliveries. I did not run a live QQBot reproduction in this read-only review.

## Solution Assessment

Yes, this appears to be the best bounded fix: it gates only delivery resolution, leaves shared non-send resolution intact, and uses the existing SDK drain primitive rather than adding new queue machinery.
