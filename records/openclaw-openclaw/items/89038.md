## Summary

Keep open: the PR is a plausible fix for the linked QQBot reconnect message-loss bug, but the current head still has a blocking text-send preflight regression where poll-only direct plugins can satisfy the shared delivery resolver and shadow a text-capable runtime sender.

## What This Changes

The PR adds a send-capability-gated outbound delivery resolver, switches message send/poll preflight to it, drains pending QQBot deliveries on reconnect, and adds outbound/QQBot regression tests.

## Best Possible Solution

Land the PR only after the delivery preflight becomes capability-intent aware so text sends require text/gateway capability while poll sends can still use poll-only channels, with focused regression coverage for both paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the linked issue has live QQBot reconnect logs, and this PR's remaining blocker follows directly from `sendMessage` and `sendPoll` sharing a resolver that accepts `sendPoll` for text preflight.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the delivery-only resolver is the right layer, but it needs text-versus-poll intent so fixing QQBot setup-shell shadowing does not preserve the same failure mode for poll-only direct plugins.
