## Summary

Keep this PR open, but the current head is not merge-ready: the diff still applies the send-capability guard to the shared channel resolver, so non-send callers can lose access to action/metadata/gateway-routing plugins, and the added QQBot reconnect drain still lacks real queued-delivery proof.

## What This Changes

The PR filters outbound channel plugin resolution by send capability and adds QQBot pending-delivery drains from gateway ready/resumed callbacks with unit coverage.

## Best Possible Solution

Keep the shared resolver compatible, add a delivery-only resolver for actual send/queue recovery paths, and either prove the QQBot reconnect drain with redacted live queued-delivery evidence or split that drain into a separately proven change.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: the patched resolver rejects plugins without send methods, while unchanged action and send-routing callers still use that shared resolver. I did not run tests because this review is read-only.

## Solution Assessment

No. The maintainable fix is to preserve the shared resolver for metadata/action paths and add a delivery-specific resolver for send paths, then prove or split the QQBot drain change.
