## Summary

## What This Changes

The PR splits outbound channel plugin resolution into shared and delivery-only paths, updates message send/poll delivery lookups, and adds qqbot reconnect draining for matching pending deliveries with focused tests.

## Best Possible Solution

Land the existing PR once required checks pass, keeping the shared resolver ungated for non-send callers and using the delivery-only resolver plus qqbot reconnect drain for stale pending outbound entries.

## Reproduction Assessment

Yes, from source inspection and the linked report: current main resolves message adapters through an ungated first-plugin resolver, then throws `Outbound not configured for channel: qqbot` when that resolved plugin cannot create a send handler. I did not run a live qqbot reproduction in this read-only review, but the current code path and provided logs match.

## Solution Assessment

Yes, this is the best fix shape I found: splitting shared and delivery-specific resolution avoids the earlier one-sided risk of breaking action-only plugins while fixing send-path shadowing. The reconnect drain is a reasonable upgrade safeguard for pending entries left by the broken build.
