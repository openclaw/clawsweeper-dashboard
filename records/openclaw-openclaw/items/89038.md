## Summary

## What This Changes

The branch adds a send-capability-gated outbound delivery resolver, routes message sends through it, and has QQBot drain matching pending deliveries on ready/resumed callbacks with tests.

## Best Possible Solution

Land a scoped fix that keeps shared channel resolution available for metadata/actions, gates only delivery on send capability, and deliberately owns the QQBot reconnect-drain semantics with maintainer approval.

## Reproduction Assessment

## Solution Assessment

Yes, this appears to be the best bounded fix: core owns generic delivery capability resolution and QQBot owns its reconnect recovery hook. Re-registering only QQBot outbound on reconnect would be narrower but would not protect other setup-shell shadowing paths.
