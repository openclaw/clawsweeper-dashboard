## Label Justifications

- `P1`: The PR targets a reported regression where QQBot reconnects can break outbound delivery and lose pending messages for real users.

## Summary

Keep open: the patch now looks like a focused fix for the resolver shadowing bug and I found no new blocking code defect, but the reconnect-drain half still lacks after-fix proof showing an actual pending qqbot queue entry being drained.

## What This Changes

The PR adds send-capability-gated outbound delivery resolution, routes delivery send paths through it, and drains matching QQBot pending deliveries on gateway ready/resumed with focused tests.

## Best Possible Solution

Land the resolver split and reconnect drain only after maintainers either receive redacted pending-drain proof or explicitly accept the source-backed rationale for why the live drain path cannot be naturally reproduced on the fixed build.

## Reproduction Assessment

Yes at source level: current main can resolve a setup-only QQBot plugin through the shared outbound resolver, and `deliver.ts` then throws `Outbound not configured for channel: qqbot`. I did not run a live QQBot repro in this read-only review.

## Solution Assessment

Mostly yes: splitting shared and delivery resolvers is the right owner-boundary fix because action/metadata callers keep the ungated resolver while send paths require send capability. The reconnect drain is a reasonable upgrade safeguard, but its real pending-entry behavior still needs maintainer acceptance or proof.
