## Summary

Keep open: this PR is still useful for the QQBot reconnect delivery bug, but the shared delivery resolver now accepts non-text send surfaces for text delivery and the supplied real behavior proof still does not show the new onReady/onResumed drain path recovering a pending delivery.

## What This Changes

The PR adds a delivery-only outbound channel resolver, switches message send preflight to it, and drains matching QQBot pending deliveries from gateway ready/resumed callbacks.

## Best Possible Solution

Preserve the QQBot reconnect recovery intent, but make the delivery resolver operation-specific so text sends require gateway or text-capable adapters, then add callback-drain proof or explicit maintainer acceptance before merge.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows a plugin with only outbound.sendMedia, sendPayload, sendPoll, formatted send, or message media/payload/poll support passes the new delivery resolver, while the direct text handler still requires text support. The original QQBot bug has strong report logs and source evidence, but not a fresh current-main live repro in this review.

## Solution Assessment

No. The QQBot-specific recovery direction is plausible, but the shared resolver predicate is too broad for sendMessage and should be split by operation or narrowed to the downstream text-delivery contract before merge.
