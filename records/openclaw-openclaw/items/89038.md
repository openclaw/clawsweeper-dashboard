## Summary

Keep this PR open: current main does not have the QQBot reconnect drain or the delivery-specific resolver, and the branch remains the active candidate fix for the linked QQBot outbound-delivery regression. I found no discrete line-level patch defect, but the shared send/poll preflight change is compatibility-sensitive and should get maintainer sign-off plus exact-head CI before merge.

## What This Changes

The PR adds a delivery-only outbound channel resolver for sendMessage/sendPoll, skips setup-only channel plugins on delivery paths, drains QQBot pending deliveries on ready/resumed reconnect callbacks, and adds focused outbound/QQBot tests.

## Best Possible Solution

Land this PR after exact-head CI and outbound/QQBot maintainer acceptance of the shared delivery-resolver compatibility risk, then close https://github.com/openclaw/openclaw/issues/88955 while keeping the separate QQBot RESUME and heartbeat target-routing items on their own tracks.

## Solution Assessment

Yes, this is the best bounded fix shape I found: keep shared resolution for non-send callers, gate only send/poll delivery preflight, reuse existing normalization/bootstrap, and drain QQBot pending entries on reconnect.
