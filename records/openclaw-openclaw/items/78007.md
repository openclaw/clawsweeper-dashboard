## Summary

Keep open: the report is plausible and current `main` still has gateway-level Undici dispatcher bootstrapping that can change plugin `fetch()` behavior, but the exact failure needs the gateway proxy environment/config confirmed before deciding whether this is a core regression or an expected proxy-routing failure.

## What This Changes

The issue reports that outbound `fetch()` from external channel plugins fails inside the OpenClaw 2026.5.4 gateway process while the same requests succeed outside the gateway.

## Best Possible Solution

Preserve normal outbound HTTPS for external channel plugins while surfacing the active gateway proxy environment clearly; if a valid proxy setup still breaks plugin fetch, fix the shared Undici/proxy runtime rather than adding a WeChat-specific bypass.

## Reproduction Assessment

No, not for the exact reported Weixin failure: the issue lacks the gateway proxy environment/config and I did not run the external plugin. Source inspection does show a plausible current-main path where gateway startup installs an env-proxy Undici dispatcher, making in-gateway fetch behave differently from standalone Node fetch when proxy env is present.

## Solution Assessment

Unclear. A direct plugin-fetch bypass is probably not the best first fix because current code intentionally routes provider and plugin network requests through gateway proxy env; the narrower path is to confirm the active proxy inputs, improve diagnostics/docs, and only change shared proxy handling if valid external HTTPS requests fail under a valid proxy setup.
