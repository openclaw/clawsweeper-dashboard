## Summary

Keep this PR open, but do not merge it as-is: the reported claude-cli reset is real, yet the current patch can under-detect real Claude CLI OAuth account changes and still exposes an unresolved public plugin SDK auth mode.

## What This Changes

The PR adds a `host-only` CLI backend auth epoch mode to the plugin SDK, applies it to Anthropic `claude-cli`, skips selected auth-profile credentials in CLI auth epoch calculation, and adds docs/tests.

## Best Possible Solution

Narrow the fix so unchanged Claude CLI host identity preserves resume continuity, but changed or unprovable host identity still resets; keep `host-only` internal unless auth/API owners explicitly approve it as a public plugin SDK contract.

## Reproduction Assessment

Yes. Source inspection plus supplied logs and QA comments show current main mixes the selected auth profile into the CLI auth epoch, so cosmetic profile rotation can produce `invalidatedReason: "auth-profile"` and drop the stored claude-cli session.

## Solution Assessment

No. Narrowing the over-broad reset trigger is the right direction, but this implementation makes Claude CLI OAuth host identity unprovable while also exposing a new public plugin SDK mode before that contract is approved.
