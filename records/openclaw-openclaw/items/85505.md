## Summary

Keep this PR open: the reported Claude CLI auth-profile reset is real and the branch addresses the right area, but the current patch reopens the same partial credential-read reset class that current main intentionally protects.

## What This Changes

This PR changes Claude CLI auth-epoch calculation to hash the host OAuth refresh token and exclude selected auth-profile credentials when that local credential is considered identity-bearing, with regression tests for cosmetic auth-profile rotation and refresh-token changes.

## Best Possible Solution

Land a narrow Claude CLI fix that preserves resume continuity across cosmetic profile rotation only when host identity is stable, while keeping partial-read stability and failing closed when identity cannot be proven.

## Reproduction Assessment

Yes, source inspection and the PR proof show the cosmetic profile rotation path: current main includes the selected profile in the epoch, and session reuse resets when the profile or epoch changes. I did not run the gateway, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No. Narrowing the over-broad auth-profile trigger is the right direction, but this implementation reintroduces partial-read auth-epoch resets and needs an explicit Claude CLI identity strategy before merge.

  - body: Current main deliberately makes full OAuth reads and token-shaped partial reads produce the same Claude CLI epoch because a transient missing `refreshToken` can occur during keychain rewrites.
