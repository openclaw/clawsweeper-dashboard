## Summary

Keep this PR open: the root cause and fix direction are credible, and the updated terminal proof is enough for this non-visual CLI path, but the new public `authEpochMode` value still needs public plugin documentation before merge.

## What This Changes

This PR adds a `host-only` CLI backend auth epoch mode, wires it through CLI run preparation, marks the Anthropic claude-cli backend to ignore auth-profile credentials for epoch calculation, and adds focused regression tests.

## Best Possible Solution

Land the host-only claude-cli fix with public CLI backend docs that define all auth epoch modes and clearly warn that `host-only` is only safe when the backend never injects OpenClaw profile credentials.

## Reproduction Assessment

Yes, from source: current main computes the CLI auth epoch from both local and selected profile credentials, and `resolveCliSessionReuse` resets when the profile id changes without a matching epoch. I did not run a live current-main gateway repro, but the PR's production logs and deterministic path make the failure high-confidence.

## Solution Assessment

Mostly yes: a backend-owned auth epoch mode is the right seam for a CLI whose spawned process uses host login rather than injected OpenClaw profile credentials. The implementation still needs public docs and maintainer acceptance of the API/auth-boundary semantics before merge.
