## Summary

Keep open. The PR addresses a real stale heartbeat/cron timestamp bug that still exists on current main, but the proposed regex refresh is too broad and would rewrite user-authored cron reminder content that starts with `Current time:`; it also misses the required user-facing changelog entry.

## What This Changes

The PR changes `appendCronStyleCurrentTimeLine` to refresh or collapse existing `Current time:` lines and adds colocated Vitest coverage for empty input, append, refresh, and duplicate-collapse cases.

## Best Possible Solution

Keep the shared-helper refresh approach, but constrain replacement to OpenClaw-generated timestamp lines or a backward-compatible internal marker, preserve user-authored prompt content, add preservation regression coverage, and add an Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. The original bug is reproducible by feeding helper output with an old generated `Current time:` line back into `appendCronStyleCurrentTimeLine` with a newer `nowMs`; the PR regression is reproducible with cron reminder text whose first content line starts `Current time:`.

## Solution Assessment

No. The helper-level boundary is the right place to fix the stale timestamp, but the current PR is not the best implementation because it cannot distinguish generated timestamp lines from user-authored content and it omits release-note coverage.
