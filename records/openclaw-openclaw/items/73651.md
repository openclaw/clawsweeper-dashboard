## Summary

Keep this PR open. Current main still sends Feishu markdown and structured cards without catching the reported 230099/11310 table-limit failure, while this PR is an active, mergeable implementation that adds a narrow outbound-only fallback and focused coverage. I did not find a blocking correctness or supply-chain issue in the proposed diff; maintainers should decide whether to land this outbound scope separately from the broader reply-dispatcher work in #43690 and #50429.

## What This Changes

The PR adds a Feishu CardKit table-limit error classifier, outbound plain-text fallback for markdown/structured card sends, and parser/fallback tests under the Feishu plugin.

## Best Possible Solution

Land the narrow outbound fallback if maintainers accept the scope, then handle the remaining reply-dispatcher/streaming-card table-limit behavior through the related Feishu follow-up rather than broadening this PR opportunistically.

## Reproduction Assessment

Yes. Current-main inspection gives a high-confidence reproduction path: make sendMarkdownCardFeishu or sendStructuredCardFeishu reject with the reported 230099/11310 table-limit shape on a card-worthy outbound message, and the rejection propagates instead of falling back.

## Solution Assessment

Yes for this PR's stated outbound scope. The proposed path catches only the known CardKit table-limit rejection, resends through the existing sendMessageFeishu path, preserves non-table-limit failures by rethrowing, and leaves broader reply-dispatcher behavior to the related follow-up.
