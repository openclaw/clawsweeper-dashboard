## Summary

Keep this PR open. Current main still has no Feishu table-limit fallback on the direct outbound card paths, and the PR is an active, focused implementation for that outbound surface rather than work already shipped or superseded; the broader reply-dispatcher/streaming work remains tracked separately by #43690 and #50429.

## What This Changes

The PR adds a Feishu CardKit table-limit error classifier, wraps outbound markdown and structured card sends with table-limit-only plain-text fallback, threads reply metadata through fallback/media sends, and adds focused tests.

## Best Possible Solution

Land or adapt a narrow Feishu outbound fallback that catches only the known CardKit table-count rejection, resends through the existing plain-text Feishu path with reply/thread metadata preserved, and keeps broader reply-dispatcher or streaming-card handling coordinated through the related follow-up work.

## Reproduction Assessment

Yes. On current main, configure Feishu card rendering and make sendMarkdownCardFeishu or sendStructuredCardFeishu reject with the reported 230099/11310 table-limit shape; outbound.ts currently returns or awaits those card sends directly, so the rejection propagates instead of falling back.

## Solution Assessment

Yes for this PR's outbound scope. The patch reuses the existing sendMessageFeishu plain-text path, narrows fallback to a table-limit classifier, and rethrows other card failures rather than masking unrelated Feishu errors.
