## Summary

Keep open: the endpoint choice is supported by Feishu/Lark docs and SDK types, but the new handler ignores the API response code and the PR still lacks live Feishu recall proof.

## What This Changes

The PR adds Feishu `unsend` to the message action surface, calls `client.im.message.delete({ path: { message_id } })`, and adds focused unit coverage.

## Best Possible Solution

Keep the plugin-local recall endpoint, but validate the Feishu `{ code, msg }` response, add a negative regression test, and require live proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, by source inspection: current main has generic `unsend` action naming but the Feishu handler has no `unsend` branch and falls through to the unsupported-action error. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted: using `im.message.delete` is the right endpoint, but the handler should check the returned Feishu API code before reporting success. The narrow repair is response validation plus a regression test, not a different endpoint.
