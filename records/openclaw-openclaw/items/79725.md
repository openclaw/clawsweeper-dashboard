## Summary

Keep this PR open: the proposed direction is plausible, but the fallback is reached only after an activity record exists, so it can still miss the reported race; the external PR also lacks inspectable real behavior proof.

## What This Changes

The PR adds a diagnostic activity callback from the embedded-run registry so session snapshots can report active embedded work, plus a changelog entry for the false stuck-session report.

## Best Possible Solution

Make the active-handle fallback produce an `embedded_run` snapshot even when no `SessionActivity` exists yet, then add regression coverage for that queue/state-before-activity race.

## Reproduction Assessment

Yes, from source inspection with medium confidence. The queued/session-state path can exist without a diagnostic activity record, so the proposed patch still has a route to an empty activity snapshot during the reported race.

## Solution Assessment

No. Bridging to active handle state is likely the right narrow fix, but it needs to run before the no-activity early return or otherwise synthesize the `embedded_run` snapshot for that path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Check active handles before returning an empty snapshot:** `src/logging/diagnostic-run-activity.ts:293-296`
  - body: The new fallback only runs after `resolveSessionActivity(params)` succeeds. Move the handle check ahead of the no-activity return or synthesize an `embedded_run` snapshot there.
