## Summary

Keep open: current main and the latest release still generate dotted provider callback data that the Telegram model-list parser rejects, and the direct fix PRs are closed unmerged.

## What This Changes

Allow Telegram `/models` provider-list callbacks to parse provider IDs containing dots so providers like `hf.co` render model buttons instead of plain text.

## Best Possible Solution

Land a narrow Telegram plugin fix that accepts dots in `mdl_list_{provider}_{page}`, adds `hf.co` parser and provider-keyboard regression coverage, and records a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A direct current-main regex check reproduces the rejected `mdl_list_hf.co_1` callback, and source inspection shows the provider keyboard emits that callback before the handler branches on parser success.

## Solution Assessment

Yes. The narrowest maintainable fix is to expand only the Telegram model-list provider capture and add dotted-provider regression tests; no core model catalog or provider configuration change is needed.

## Repair Work Prompt

Fix the Telegram `/models` model-list callback bug for dotted provider IDs. In `extensions/telegram/src/model-buttons.ts`, make `parseModelCallbackData()` accept provider IDs containing `.` for `mdl_list_{provider}_{page}` callbacks while preserving existing hyphen/underscore behavior and page validation. Add regression coverage in `extensions/telegram/src/model-buttons.test.ts` for parsing `mdl_list_hf.co_1` and for `buildProviderKeyboard([{ id: "hf.co", count: ... }])` emitting callback data that parses back to the same provider. Add a single-line user-facing fix entry to `CHANGELOG.md`. Do not edit generated `dist` bundles, do not change model catalog/provider normalization, and do not broaden unrelated callback parsing.
