## Summary

Keeping this open. Current main has stable tab aliases and docs telling agents to use `suggestedTargetId`, but it does not resolve the central report: flattened `kind=fill` with `ref`/`text` can still reach runtime and fail `fill requires fields`, and stale-target recovery for mutating `click`/`type`/`fill` remains intentionally limited rather than automatic.

## What This Changes

The issue reports browser plugin failures in long chrome-mcp runs, including stale target mismatches, aborts, screenshot temp-file errors, and `fill`/`type` argument validation drift.

## Best Possible Solution

Split this into a focused fill schema/runtime repair plus a maintainer decision on safe target-mismatch recovery for mutating browser actions, while keeping `suggestedTargetId` as the durable tab handle.

## Reproduction Assessment

Partially yes from source: current main still rejects `kind=fill` with only `ref` and `text`, and current tests explicitly avoid retrying mutating stale-target actions. I did not reproduce the full long-running chrome-mcp cron failure against a live browser.

## Solution Assessment

No as one automated fix: the fill-schema drift is narrow, but browser-layer recovery for mutating actions needs a maintainer-approved safety boundary because retries can duplicate sends or form mutations.
