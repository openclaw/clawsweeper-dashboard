## Summary

Keep open. Current main has stable tab-reference docs and intentionally limited stale-target retry, but the central report is not fully solved: flat `kind=fill` with `ref`/`text` is still exposed by the model-facing schema and still rejected by runtime normalization unless `fields` is provided, while mutating stale-target recovery remains a product/owner decision.

## What This Changes

The issue reports long-running chrome-mcp browser sessions losing valid targets/refs and rejecting fill/type/click actions during cron-driven browsing flows.

## Best Possible Solution

Split the work into a narrow fill contract repair plus an explicit browser-owner decision on safe recovery behavior for mutating stale-target failures, backed by live chrome-mcp repro logs.

## Reproduction Assessment

Yes for the fill-schema subproblem from source: flat `kind=fill` with `ref` and `text` is still schema-exposed but runtime-rejected unless `fields` is present. No for the full long-running chrome-mcp AbortError/ENOENT sequence; that still needs a live capture or focused repro.

## Solution Assessment

Unclear. A narrow fill normalization/schema repair is likely the best first fix, but automatic recovery for mutating target mismatches is not obviously safe and needs an explicit browser-layer policy decision.
