## Summary

Keep open. Current main has fixed the diagnostics, bundled provider registration, and per-agent TTS override parts, but automatic TTS still intentionally follows the documented provider fallback chain; making a configured ElevenLabs provider fail closed would be a product/config decision rather than a narrow bug fix.

## What This Changes

The issue requests Discord automatic TTS to honor the configured ElevenLabs provider, log fallback failures, and support per-agent voice configuration.

## Best Possible Solution

Keep this as the canonical maintainer decision point for TTS provider stickiness: either document fallback as intentional for pinned providers or add a narrow strict-provider option while preserving fallback by default.

## Reproduction Assessment

Yes for the remaining provider-stickiness behavior from source inspection: automatic reply paths call textToSpeech without disableFallback, and synthesis expands the provider order unless disableFallback is true. The already-fixed diagnostics and per-agent voice portions are covered by docs/tests on current main.

## Solution Assessment

Unclear. Current main fixes observability and per-agent voice support while preserving documented fallback semantics, so a fail-closed provider lock needs an explicit product/config decision before implementation.
