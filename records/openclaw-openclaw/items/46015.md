## Summary

Keep open. Current main has implemented the diagnostics, provider registration, and per-agent TTS override portions, but strict provider stickiness or fail-closed auto TTS would change the documented fallback contract and was explicitly left open in the issue discussion.

## What This Changes

Request that Discord automatic TTS honor configured ElevenLabs without silent Edge fallback and support per-agent voice or voiceId overrides.

## Best Possible Solution

Keep this as the canonical maintainer decision point for automatic TTS provider stickiness: either document fallback as intentional for pinned providers or add a narrow, documented strict-provider option while preserving fallback by default.

## Reproduction Assessment

Yes for the remaining fallback-selection behavior. Current source shows automatic replies call textToSpeech without disableFallback, and synthesis walks the resolved fallback provider list.

## Solution Assessment

Unclear. Current main fixes observability and per-agent voice support while preserving documented fallback semantics, so a fail-closed provider lock needs an explicit product/config decision before implementation.
