## Summary

Keep open. Current main has resolved the diagnostics and per-agent voice portions, but automatic TTS still treats the configured ElevenLabs provider as the primary member of a fallback chain rather than a fail-closed provider lock, and the issue discussion explicitly left provider stickiness open.

## What This Changes

Discord automatic TTS is expected to honor a configured ElevenLabs provider, surface fallback errors, and support per-agent voice selection like other TTS paths.

## Best Possible Solution

Keep this as the canonical maintainer decision point for automatic TTS provider stickiness: either document fallback as intentional for explicitly configured providers, or add a narrow opt-in strict-provider contract that fails closed when a provider or voice is pinned while preserving fallback behavior by default.

## Reproduction Assessment

Yes for the remaining fallback contract. Current main shows automatic replies call textToSpeech without disableFallback, and synthesizeSpeech walks the resolved fallback provider list, so the behavior can be verified from source without a live ElevenLabs account.

## Solution Assessment

Unclear. Forcing fail-closed automatic TTS would satisfy the report, but current docs, types, and tests preserve fallback as expected behavior, so the maintainable fix needs a product decision or opt-in strict-provider setting.
