## Summary

Current main and shipped releases now cover the reported Discord auto-TTS behavior: ElevenLabs/Microsoft speech providers are registered, configured providers are tried first with visible fallback diagnostics, and per-agent TTS voice overrides are implemented through `agents.list[].tts`. The remaining hard no-fallback/provider-stickiness idea is a separate product-policy feature, not a live regression in this report.

## What This Changes

The issue asks Discord auto TTS to honor a configured ElevenLabs provider, expose fallback failures, and support per-agent voice IDs like Telegram.

## Best Possible Solution

Keep the shipped TTS implementation: configured ElevenLabs is registered and tried first, fallback attempts are visible, and per-agent voices use `agents.list[].tts`; any strict no-fallback policy should stay separate from this bug.

## Reproduction Assessment

Yes. The original failure is source- and history-reproducible from the pre-fix provider registration/diagnostics path, and current main has source evidence showing that path now resolves provider, agent, channel, and account TTS context before synthesis.

## Solution Assessment

Yes. The current implementation is the maintainable fix for this report because it uses the shared TTS config resolution path, bundled provider activation metadata, and `/tts status` diagnostics instead of adding Discord-only special cases.
