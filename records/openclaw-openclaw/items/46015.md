## Summary

Keep open. Current main has resolved the diagnostics and per-agent TTS configuration portions, but automatic TTS still treats the configured ElevenLabs provider as the primary member of a fallback chain rather than an exclusive, fail-closed provider; the issue discussion also explicitly left provider stickiness as remaining scope.

## What This Changes

The report asks Discord automatic TTS to respect a configured ElevenLabs provider, surface fallback errors, and support per-agent voiceId selection like Telegram.

## Best Possible Solution

Keep this as the canonical maintainer decision point for automatic TTS provider stickiness: either document fallback as intentional for explicitly configured providers, or add a narrow opt-in strict-provider contract that fails closed when a provider or voice is pinned while preserving fallback behavior by default.

## Reproduction Assessment

Yes for the remaining fallback contract. Current main shows automatic replies call textToSpeech without disableFallback, and synthesizeSpeech walks the resolved fallback provider list, so the contract can be verified from source without a live ElevenLabs account.

## Solution Assessment

Unclear. Forcing fail-closed automatic TTS would satisfy the report, but current docs, types, and tests preserve fallback as expected behavior, so the maintainable fix needs a product decision or opt-in strict-provider setting.
