## Summary

Keep open. Current main still has config validation/write backups and invalid-config recovery, but not the requested transactional config-apply flow that health-checks the restarted gateway and rolls back a schema-valid but operationally bad config.

## What This Changes

Request a configurable automatic backup, health-check, and rollback mechanism for `config.apply`/`config.patch` changes that make the gateway fail or degrade after restart.

## Best Possible Solution

Define a gateway-owned transactional config write contract that reuses existing validation/backups and health-gates restart-requiring writes before promotion, with explicit rollback scope, timeout behavior, and documentation.

## Reproduction Assessment

No high-confidence runtime reproduction is available from the report because it does not include a concrete schema-valid config that breaks current main. Source inspection does confirm the requested health-check and rollback path is absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the best maintainable solution needs maintainer decisions on what counts as healthy, how long to observe, what state can be rolled back, and what happens if rollback fails.
