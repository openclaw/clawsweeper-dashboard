- `maturity:stable`: The issue affects the M4 Stable gateway-runtime surface, including config RPC and gateway lifecycle behavior.

## Summary

Keep open. Current main and v2026.6.10 still provide validation, last-known-good/manual doctor recovery, and restart health diagnostics, but they do not implement a health-gated config.apply/config.patch commit with automatic rollback after an unhealthy gateway restart.

## What This Changes

Request automatic backup, health check, and rollback safeguards around gateway config.apply/config.patch writes when a new config makes the gateway unhealthy.

## Best Possible Solution

Define a bounded health-gated config commit contract that builds on existing last-known-good, doctor repair, and restart-health primitives without adding broad config knobs before the policy is settled.

## Reproduction Assessment

Yes, at source level: config.apply/config.patch persist validated config and can schedule restart without a post-restart health result or rollback branch. I did not run a live schema-valid crash-loop scenario in this read-only review.

## Solution Assessment

Unclear as an implementation. The reliability goal is sound, but maintainers need to choose health semantics, observation window, rollback scope, operator notification, and whether a new config surface is acceptable.
