## Label Justifications

- `P2`: This is a normal-priority reliability improvement with real downtime impact but it needs product design before implementation.
- `impact:crash-loop`: The reported failure mode is a schema-valid config causing the gateway to fail or repeatedly restart until manual recovery.
- `impact:message-loss`: When the gateway is down after a bad config, channel messages cannot be delivered through OpenClaw until the operator recovers it.

## Summary

Keep open. Current main still has config validation, backup rotation, last-known-good promotion, doctor/manual restore, and restart scheduling, but it does not implement the requested post-apply health gate or automatic rollback for a schema-valid config that makes the gateway unhealthy.

## What This Changes

The issue requests automatic pre-apply config backup, post-restart gateway health checking, and rollback for schema-valid configuration changes that make the gateway unhealthy.

## Best Possible Solution

Define a core-owned safe config apply contract, then implement opt-in health-gated rollback with tests for fresh apply, upgrade/default behavior, failed health, and rollback failure semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence missing-capability path, but this read-only review did not run a live broken-config gateway failure.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an immediate implementation; the reliability goal is aligned, but maintainers should first choose the health predicate, timeout, rollback scope, and failure semantics.
