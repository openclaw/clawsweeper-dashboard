## Summary

Keep open. Current main still validates and writes config, keeps backup/last-known-good primitives, and schedules restart, but it does not implement the requested post-apply health gate or automatic rollback for schema-valid config that leaves the Gateway unhealthy; the reporter's follow-up also explicitly defers this to maintainer reliability design.

## What This Changes

The issue requests automatic backup, post-apply Gateway health checking, and rollback for `config.apply`/`config.patch` writes when a schema-valid config leaves the Gateway unhealthy.

## Best Possible Solution

Define a Gateway-owned transactional config write policy that reuses existing validation/backups, health-gates promotion for restart-requiring writes, and restores only within explicit rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence evidence that the requested post-apply health-check and rollback path is missing, but no concrete schema-valid failing config was provided for runtime reproduction.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation needs maintainer decisions on health definition, timeout behavior, rollback scope, config surface, and what happens if rollback itself fails.

## Security Review

Status: needs_attention

Summary: The requested rollback feature is security-sensitive because it would persist and restore config that can affect auth, plugin loading, gateway exposure, tools, and secret references.

Concerns:

- **[medium] Define rollback scope for security-critical config:** `src/gateway/server-methods/config.ts:529`
  - body: Automatic rollback after `config.apply` or `config.patch` could silently restore or undo auth, TLS, plugin, tool, or gateway exposure changes unless the design explicitly limits what can be reverted and how operators are notified.
  - confidence: 0.82
- **[medium] Protect backup and restore handling:** `src/config/io.ts:2242`
  - body: Config writes already maintain backup files with restrictive modes, but an automatic runtime rollback path still needs explicit permissions, redaction, SecretRef, retention, and audit rules before it restores prior config bytes.
