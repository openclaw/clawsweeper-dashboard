## Summary

Keep open. Current main has validation, SecretRef preflight, backup rotation, and last-known-good/doctor recovery primitives, but it still lacks the requested configurable post-apply health check and automatic rollback for config writes that pass schema validation but leave the gateway unhealthy.

## What This Changes

The issue requests automatic backup, health checking, and rollback around gateway `config.apply`/`config.patch` writes so bad config changes do not leave the gateway down.

## Best Possible Solution

Define a gateway-owned transactional config write policy that reuses existing validation/backups and health-gates promotion or restoration for restart-requiring writes with explicit rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested post-apply health-check/rollback path is missing, but the report does not provide a concrete schema-valid config that reproduces a current-main gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but the current code and docs intentionally keep whole-file restore doctor-owned, so maintainers need to define health criteria, timeout behavior, rollback scope, and the new config surface.
