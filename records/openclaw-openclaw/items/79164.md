## Summary

Keep open. Current main still validates and writes config, maintains backups, promotes last-known-good snapshots, and offers doctor-owned recovery for invalid config, but it does not health-gate `config.patch`/`config.apply` promotion or auto-rollback a schema-valid config that leaves the gateway unhealthy; the reporter’s follow-up also confirms this needs maintainer reliability/product design before implementation.

## What This Changes

The issue requests a configurable automatic backup, post-apply health check, and rollback path for gateway config writes that break runtime health.

## Best Possible Solution

Define a gateway-owned transactional config write policy that reuses existing validation/backups, observes an explicit post-restart health contract, and restores or promotes config within documented rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence evidence that the requested post-apply health-check and rollback path is missing, but the issue does not include a concrete schema-valid config that reproduces a current-main gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but choosing health criteria, timeout behavior, rollback scope, failure semantics, and the new config surface requires maintainer design before a narrow implementation is safe.
