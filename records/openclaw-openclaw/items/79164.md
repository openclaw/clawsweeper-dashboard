## Summary

Keep open. Current main validates and writes config changes, keeps backup/last-known-good machinery, and reserves invalid-config recovery for doctor/manual repair, but it still lacks the requested configurable post-apply health check and automatic rollback for schema-valid config changes that leave the Gateway unhealthy.

## What This Changes

Request automatic safeguards around `config.apply` and `config.patch`: pre-write rollback point, post-apply health check, configurable timeout, and rollback on Gateway failure.

## Best Possible Solution

Define a Gateway-owned transactional config-write policy that records a rollback point, defines health criteria and timeout behavior, restores or promotes explicitly for restart-requiring writes, and documents how it differs from doctor-managed last-known-good repair.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main lacks a post-apply health-check/rollback path, but the report does not include a concrete schema-valid config that reproduces a current-main Gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but health definition, rollback scope, timeout defaults, operator notification, and the new config surface need maintainer reliability/product design before implementation.
