## Label Justifications

- `P2`: This is a normal-priority core reliability feature with meaningful operator impact, but it is not a confirmed urgent regression or emergency.
- `impact:message-loss`: The reported failure mode can take the gateway down and stop message-based access such as Telegram or other channels.
- `impact:crash-loop`: A bad config followed by restart can leave the gateway down or repeatedly restarting until manual recovery.

## Summary

Keep open. Current main and v2026.6.6 still validate and persist `config.apply` / `config.patch` writes, then hot-reload or schedule restart work, but they do not health-gate the post-restart result or restore a schema-valid config that leaves the gateway operationally unhealthy.

## What This Changes

The issue requests automatic safeguards around gateway config writes: pre-apply backup, post-apply health observation, and rollback when the resulting config leaves the gateway unhealthy.

## Best Possible Solution

Define a maintainer-owned config safety contract, then implement it in the core gateway config write/reload/restart path by reusing last-known-good and doctor repair mechanisms with focused tests for healthy writes, unhealthy writes, rollback failure, and upgrade defaults.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken documented behavior. Source, docs, tests, and the latest release provide a high-confidence absence check; a live schema-valid runtime-bad config scenario would still be needed before implementation.

## Solution Assessment

Unclear: the direction is plausible and core-owned, but the proposed config knobs and auto-rollback semantics are not yet the best fix until maintainers define the health signal, observation window, rollback scope, and failure semantics.
