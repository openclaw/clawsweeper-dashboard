## Summary

Keep open. Current main still validates and backs up config writes, and doctor can restore invalid snapshots, but there is no post-apply gateway health observation or automatic rollback for schema-valid config that leaves the gateway unhealthy; the reporter’s latest production context also confirms this needs maintainer reliability/product design.

## What This Changes

Request automatic safeguards around `config.apply` and `config.patch`, including post-restart gateway health checks and rollback when a schema-valid config leaves the gateway unhealthy.

## Best Possible Solution

Define a gateway-owned transactional config write contract that reuses existing validation and backups, then health-gates promotion or restoration for restart-requiring writes with explicit health criteria, timeout, rollback scope, and failure semantics.

## Reproduction Assessment

No, not as a bug reproduction path. This is a feature request: source inspection confirms the requested health-check and rollback path is missing, but the issue does not provide a concrete schema-valid config that fails on current main.

## Solution Assessment

Unclear. The requested direction is plausible, but the best solution depends on maintainer choices for health definition, timeout behavior, rollback scope, and whether the policy should be configurable or defaulted.
