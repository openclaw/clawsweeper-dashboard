## Label Justifications

- `P2`: Configuration safety is important but still a bounded reliability feature rather than an emergency runtime outage.
- `impact:message-loss`: A bad gateway config can take messaging channels offline until manual recovery.
- `impact:crash-loop`: The reported failure mode is a schema-valid config that makes gateway startup or restart loop fail.

## Summary

Keep open. Current main has meaningful last-known-good and manual doctor recovery safeguards, but it still documents and implements config.apply/config.patch as validated writes plus restart scheduling, not as health-gated commits with automatic rollback after an unhealthy gateway restart.

## What This Changes

The issue requests automatic backup, post-apply health checking, and automatic rollback for config.apply/config.patch changes that leave the gateway unhealthy after restart.

## Best Possible Solution

Keep this issue as the product/design anchor for a health-gated config commit and rollback contract that builds on current last-known-good and doctor repair behavior.

## Reproduction Assessment

Yes from source for the missing contract, not from a live crash reproduction: config.apply/config.patch persist validated config and schedule restart without a post-restart health result gate.

## Solution Assessment

No final implementation path yet; maintainers need to define health semantics, rollback scope, observation window, and rollback-failure behavior before a fix PR can be safe.
