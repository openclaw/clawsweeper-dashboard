## Summary

Keep open. Current main has config validation, SecretRef preflight, pre-replacement backup rotation, and doctor/manual invalid-config recovery, but it still lacks the requested configurable post-apply health check and automatic rollback for schema-valid config changes that leave the gateway unhealthy.

## What This Changes

Request automatic config safeguards around `config.apply` and `config.patch`: backup, post-apply health check, rollback on failure, and new `gateway.configSafety` settings.

## Best Possible Solution

Define a gateway-owned transactional config safety policy that reuses existing validation and backups, then health-gates promotion or restoration for restart-requiring writes with explicit rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested post-apply health-check and rollback path is missing, but there is no concrete schema-valid config provided to reproduce a current-main gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but the health definition, rollback scope, timeout semantics, and new config surface need maintainer reliability/product design before implementation.
