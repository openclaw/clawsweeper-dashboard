## Summary

Keep open. Current main has config validation, write-time `.bak` rotation, SecretRef preflight, and narrow last-known-good recovery for invalid snapshots, but it does not implement the requested configurable post-apply health check and rollback for schema-valid config changes that leave the gateway unhealthy.

## What This Changes

The issue requests opt-in automatic config backup, post-apply gateway health checking, and rollback around `config.apply` and `config.patch` writes.

## Best Possible Solution

Define a gateway-owned transactional config write policy that reuses existing validation and backups, then health-gates promotion or restoration for restart-requiring writes with explicit rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested post-apply health-check/rollback path is missing, but the report does not include a concrete schema-valid config that reproduces a current-main gateway crash.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact config knobs, health definition, timing, and rollback scope need maintainer design because current docs intentionally reserve whole-file recovery for doctor-managed repair.
