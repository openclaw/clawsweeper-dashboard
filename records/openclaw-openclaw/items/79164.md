## Summary

Keep open. Current main has schema validation, backup rotation, SecretRef preflight, and doctor-managed last-known-good repair, but it still lacks the requested configurable post-apply health check and automatic rollback for schema-valid config changes that leave the gateway unhealthy.

## What This Changes

The issue requests automatic backup, post-apply gateway health checking, and rollback for `config.apply`/`config.patch` changes that make the gateway fail after restart.

## Best Possible Solution

Define a gateway-owned transactional config write policy that reuses existing validation and backups, then health-gates promotion or restoration for restart-requiring writes with explicit rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested post-apply health-check/rollback path is missing, but the report does not provide a concrete schema-valid config that reproduces a current-main gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but health criteria, timeout behavior, rollback scope, and the proposed config knobs need maintainer design because current docs reserve whole-file restore for doctor-managed repair.
