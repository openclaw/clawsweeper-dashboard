## Summary

Keep open. Current main still validates and backs up config writes, but it does not provide the requested configurable post-apply health check and automatic rollback for schema-valid config changes that make the gateway unhealthy; the reporter’s follow-up confirms this needs reliability/product design rather than a narrow bug fix.

## What This Changes

Request automatic safeguards around `config.apply` and `config.patch`: pre-apply backup, post-restart health checking, and rollback for operationally bad config changes.

## Best Possible Solution

Define a gateway-owned transactional config write policy that reuses existing validation/backups, records a pre-change restore point, health-gates restart-requiring writes, and restores only within explicitly documented rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested post-restart health-check/rollback path is missing, but there is no concrete schema-valid current-main config that reproduces a gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but health definition, observation window, rollback scope, failure semantics, and config knobs need maintainer design before implementation.
