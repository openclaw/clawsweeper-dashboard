## Summary

Keep open. Current main has write-time backups, schema/clobber rejection, and invalid-config last-known-good recovery, but it does not implement the requested configurable post-apply health check and rollback for schema-valid changes that make the gateway unhealthy.

## What This Changes

Request automatic safeguards around `config.apply` and `config.patch`: backup, post-apply gateway health check, and rollback with configurable settings.

## Best Possible Solution

Design a gateway-owned config transaction policy that preserves existing validation/backups and, for opted-in restart-requiring writes, verifies a defined health signal before promoting last-good or restoring the prior config.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the missing health-check/rollback path, but no concrete schema-valid bad config is supplied to reproduce a current-main crash.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact `configSafety` knobs, health definition, and rollback boundaries need maintainer design because current docs intentionally keep whole-file recovery narrow.
