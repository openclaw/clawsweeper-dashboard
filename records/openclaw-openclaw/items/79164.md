## Summary

Keep open. Current main and the latest shipped release still validate and commit config writes before scheduling reload/restart, but they do not provide the requested post-apply health gate or automatic rollback for schema-valid config that leaves the gateway unhealthy.

## What This Changes

Request automatic backup, post-apply gateway health checking, and rollback for `config.apply` / `config.patch` changes that leave the gateway unhealthy.

## Best Possible Solution

Define a gateway-owned transactional config write contract that reuses existing backups and last-known-good state, stages restart-requiring writes, observes documented health criteria, and promotes or rolls back with explicit timeout and scope semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested health-check/rollback path is absent, but no schema-valid config sample has been provided to reproduce a runtime-health failure on current main.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers need to define health, timeout, rollback boundaries, and failure semantics before implementation is the narrowest maintainable path.
