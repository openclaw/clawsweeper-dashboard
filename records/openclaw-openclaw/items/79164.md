## Summary

Keep open. Current main still validates and writes `config.apply`/`config.patch` changes and then schedules restart work without a post-restart health observation or rollback branch; existing last-known-good recovery is invalid-config/doctor scoped, and the reporter has confirmed this needs maintainer reliability design.

## What This Changes

Request automatic backup, post-apply health checking, configurable rollback, and recovery behavior for config writes that leave the gateway unhealthy.

## Best Possible Solution

Define a gateway-owned transactional config write policy that reuses existing validation and backups, then health-gates promotion or restoration for restart-requiring writes with explicit rollback boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested health-gated rollback path is missing, but the thread does not provide a concrete schema-valid config that reproduces a current-main gateway crash.

## Solution Assessment

Unclear. The direction is plausible, but the health definition, observation window, rollback scope, config knobs, and rollback-failure behavior need maintainer design because current docs reserve whole-file restore for doctor/manual repair.
