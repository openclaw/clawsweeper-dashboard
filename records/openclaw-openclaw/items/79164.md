## Summary

Keep open. Current main has validation, write-time backup rotation, and doctor-driven last-known-good recovery, but it still lacks the requested configurable post-apply health gate and automatic rollback for schema-valid config changes that make the gateway unhealthy.

## What This Changes

Request automatic safeguards around `config.apply` and `config.patch`: pre-write backup, post-apply health checking, and configurable rollback when the gateway fails after a config change.

## Best Possible Solution

Define a gateway-owned transactional config write contract that reuses existing validation/backups, health-gates restart-requiring writes before promotion, and documents explicit rollback scope and timeout semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested health-check and rollback path is absent, but there is no concrete schema-valid config sample that reproduces a current-main gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution needs maintainer design for health definition, timeout behavior, rollback boundaries, and config/API surface before code work starts.
