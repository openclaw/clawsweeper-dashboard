## Summary

Keep open: the PR targets a real Google memory embedding regression, but the current patch is still a runtime-only `authProviderId` alias that can miss cold plugin capability resolution and changes the accepted `memorySearch.provider` contract without maintainer-approved semantics.

## What This Changes

The PR adds an `authProviderId` fallback to the shared memory embedding provider resolver and updates the memory-core embedding test mock with a `google` to `gemini` regression case.

## Best Possible Solution

Choose one memory provider-id contract, then either support `google` end to end through manifest/startup/runtime diagnostics and tests or keep `gemini` canonical with doctor or diagnostic guidance for existing `google` configs.

## Reproduction Assessment

Yes, by source inspection: current main preserves `memorySearch.provider: "google"` into memory provider initialization, while the Google memory adapter and manifest use `gemini`, matching the unknown-provider failure path. I did not run a live CLI reproduction because this review is read-only.

## Solution Assessment

No, not yet: centralizing the fallback is cleaner than duplicated call-site scans, but it is still a runtime-only alias that misses cold activation and makes `authProviderId` a config contract without owner approval.
