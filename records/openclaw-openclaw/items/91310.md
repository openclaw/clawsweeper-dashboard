## Summary

Keep open: the centralized fallback is a plausible mitigation, but it only scans already-registered adapters and implicitly makes every adapter `authProviderId` an accepted memory provider id without a maintainer-approved startup/docs/contract path.

## What This Changes

The PR adds an `authProviderId` fallback to the shared memory embedding provider resolver and a memory-core regression test for resolving `google` to the registered `gemini` adapter.

## Best Possible Solution

Choose one memory provider-id contract, then either support `google` end to end through manifest/startup/runtime/docs/tests or keep `gemini` canonical with doctor or diagnostic guidance for affected configs.

## Reproduction Assessment

Yes. Source inspection shows `memorySearch.provider: "google"` reaches memory provider initialization while the Google memory adapter and manifest are keyed as `gemini`, matching the linked unknown-provider failure path.

## Solution Assessment

No. Centralizing the fallback is cleaner than duplicated call-site scans, but it is still a runtime-only alias that misses cold activation and turns `authProviderId` into a memory provider contract without owner approval.

- **[P1] Resolve aliases before cold plugin lookup:** `src/plugins/memory-embedding-provider-runtime.ts:66-67`
  - body: The fallback scans only adapters already registered in the process. In a cold memory CLI or Gateway path, `memorySearch.provider: "google"` does not match the Google plugin manifest's `gemini` memory contract, so the plugin may not be loaded and there may be no registered adapter to scan. Move the alias into manifest/startup owner resolution or use an explicit plugin-owned alias with cold-path coverage.
  - confidence: 0.84

Either document and test this alias semantics end to end, or keep `gemini` canonical and add migration or diagnostic guidance.
