## Summary

Keep open: the PR targets the reported Google memory embedding failure, but the current patch is not merge-ready because the fallback only scans already-registered adapters and makes `authProviderId` an implicit `memorySearch.provider` contract without cold-start, docs, or owner-confirmed semantics.

## What This Changes

The PR updates the shared memory embedding provider resolver to fall back from adapter-id lookup to registered adapters whose `authProviderId` matches, with a memory-core regression test.

## Best Possible Solution

Choose one documented memory provider-id contract, then either support `google` end to end through manifest/startup/runtime/docs/tests or keep `gemini` canonical with doctor or diagnostic guidance for affected configs.

## Reproduction Assessment

Yes. Source inspection shows `memorySearch.provider: "google"` is preserved into provider initialization while the Google memory adapter and manifest are keyed as `gemini`, matching the reported unknown-provider throw path.

## Solution Assessment

No. Centralizing the fallback is cleaner than duplicated call-site scans, but it is still a runtime-only alias that misses cold activation and makes `authProviderId` a config contract without owner approval.
