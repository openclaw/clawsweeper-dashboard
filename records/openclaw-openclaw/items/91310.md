## Summary

Keep open: the PR is aimed at a real Google memory regression and includes useful resolver proof, but the patch is not merge-ready because the fallback only helps after an adapter is already registered and it broadens `memorySearch.provider` alias semantics without an explicit owner decision or end-to-end coverage.

## What This Changes

The branch adds a shared `getMemoryEmbeddingProvider` fallback that scans registered memory adapters by `authProviderId` and updates the memory-core embedding test mock plus a regression test for `google` resolving to Gemini.

## Best Possible Solution

Choose one documented memory provider-id contract, then either support `google` end to end through manifest/startup/runtime diagnostics and tests or keep `gemini` canonical with doctor or diagnostic guidance for existing `google` configs.

## Reproduction Assessment

Yes, by source inspection: current main preserves `memorySearch.provider: "google"` into memory provider initialization, while the Google memory adapter and manifest use `gemini`, matching the reported unknown-provider failure path. I did not run a live CLI reproduction because this review is read-only.

## Solution Assessment

No, not yet: centralizing the fallback is cleaner than duplicated call-site scans, but it is still a runtime-only alias that misses cold activation and makes `authProviderId` a config contract without owner approval.
