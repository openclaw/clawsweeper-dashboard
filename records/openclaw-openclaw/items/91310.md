## Summary

Keep open: this PR targets a real Google memory embedding regression, but the runtime-only `authProviderId` fallback is not merge-ready because it changes provider-id semantics without covering cold plugin activation, docs, diagnostics, or the explicit memory provider contract.

## What This Changes

The branch adds an `authProviderId` fallback to `getMemoryEmbeddingProvider` and updates a memory-core regression test so `google` can resolve to the registered `gemini` adapter after registration.

## Best Possible Solution

Choose one memory provider-id contract, then support it consistently through manifest/cold startup resolution, runtime lookup, docs, diagnostics or migration guidance, and focused tests.

## Reproduction Assessment

Yes, by source inspection. `memorySearch.provider: "google"` reaches provider initialization while current main and `v2026.6.9` declare and register only the `gemini` memory adapter for Google, matching the unknown-provider path.

## Solution Assessment

No. Centralizing the fallback is better than duplicated call-site scans, but this version is too broad and does not cover the cold startup, manifest, docs, and diagnostics contract needed for a durable fix.
