## Summary

Keep open: this targets a real shipped Google memory regression, but the branch is not merge-ready because the new alias fallback only covers already registered adapters and broadens the `memorySearch.provider` contract without cold-start activation, diagnostics, or docs coverage.

## What This Changes

The PR adds an `authProviderId` fallback to memory embedding provider lookup and a regression test for resolving a `google` memory provider selection to the Gemini adapter.

## Best Possible Solution

Resolve `google` memory provider config through one documented end-to-end provider-id contract that covers startup activation, runtime lookup, diagnostics, and cold-path regression tests.

## Reproduction Assessment

Yes from source: current main resolves memory adapters by provider/configured API ids, while the Google memory adapter is registered as `gemini` with `authProviderId: "google"`. I did not run a live `memory status --index` path because this review is read-only.

## Solution Assessment

No, not yet: the centralized resolver fallback is plausible for already registered adapters, but it does not cover cold startup/manifest ownership and makes `authProviderId` aliases a config contract without maintainer direction.

Full review comments:

- **[P1] Resolve auth-provider aliases before cold capability lookup:** `src/plugins/memory-embedding-provider-runtime.ts:63-67`
  - body: The fallback only scans adapters that are already registered. In a cold CLI/Gateway path, `memorySearch.provider: "google"` is still sent through capability resolution as `google`, while the Google manifest declares its memory embedding provider as `gemini`; if the Google plugin was not loaded for some other reason, this can preserve the reported `Unknown memory embedding provider: google` failure. Please move the alias into the lookup/owner-resolution path or add equivalent manifest/startup coverage.
  - confidence: 0.84
- **[P1] Make the new memory provider alias an explicit contract:** `src/plugins/memory-embedding-provider-runtime.ts:63-67`
  - body: Matching `memorySearch.provider` against `authProviderId` silently accepts values that the memory config docs currently describe as adapter ids such as `gemini`. If `google` is meant to be supported, the PR needs docs/diagnostics and tests for that contract; otherwise this runtime alias should be rejected in favor of migration or a targeted diagnostic.
  - confidence: 0.74
