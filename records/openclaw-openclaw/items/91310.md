## Summary

Keep this PR open, but do not merge it yet: the central resolver fallback fixes the warm registered-adapter path, while cold plugin capability/startup resolution still keys the Google memory adapter as `gemini`, not `google`. That means the reported Google memory configuration can still miss the owner plugin unless Google is already loaded for another reason.

## What This Changes

The PR adds an `authProviderId` fallback to memory embedding provider runtime lookup and a memory-core regression test for resolving `google` to the Gemini adapter.

## Best Possible Solution

Decide whether `memorySearch.provider: "google"` is a supported alias; if yes, make alias resolution cover cold capability loading, startup diagnostics, docs, and focused regression tests, otherwise migrate or guide users back to `gemini`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main resolves memory embedding adapters by adapter id/configured API owner, while the Google memory adapter id is `gemini` and its auth provider is `google`. I did not run the CLI because this review is read-only.

## Solution Assessment

No, not yet: centralizing the lookup is the right owner boundary, but the patch only covers already registered adapters. The best fix must also resolve the alias before cold manifest/capability loading and settle the config contract.
