## Label Justifications

## Summary

Keep open: the PR is a plausible centralized mitigation with sufficient resolver-path proof, but it is not merge-ready because it makes `authProviderId` an implicit `memorySearch.provider` contract and can still miss cold plugin activation where the Google manifest declares only `gemini`. The canonical issue remains open, and the sibling alias PR is also open with blockers, so this PR is not safely superseded.

## What This Changes

The PR adds an `authProviderId` fallback in the shared memory embedding provider resolver and a memory-core regression test for resolving `google` to the Gemini adapter.

## Best Possible Solution

Choose one memory provider-id contract, then support it through manifest, cold startup, runtime lookup, docs, diagnostics, and focused tests, or keep `gemini` canonical with doctor/status guidance for affected configs.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.6.8 preserve `memorySearch.provider: "google"` into provider initialization while the Google memory adapter and manifest are keyed as `gemini`, matching the reported unknown-provider throw path.

## Solution Assessment

No. Centralizing the fallback is cleaner than duplicated call-site scans, but this runtime-only alias is too broad and misses cold activation unless the provider-id contract is chosen and implemented end to end.
