## Summary

Keep open: this targets a real memory/Gateway regression, but the patch adds call-site `authProviderId` fallback scans while shared config and startup provider ownership still treat `google` as a separate id. ## What This Changes

This PR adds `authProviderId` fallback lookups for memory embedding provider resolution in memory-core and the Gateway, plus a memory-core regression test for resolving `google` to the Gemini adapter.

## Best Possible Solution

If maintainers approve `google` as a compatibility alias, put that mapping in the shared memory embedding provider resolver and startup/config owner path, then cover memory manager, Gateway, docs, and upgrade behavior together.

## Reproduction Assessment

Yes from source inspection: current main looks up memory embedding providers by adapter id or configured API owner, while the Google memory adapter is `gemini` with `authProviderId: "google"`. I did not run the live Google API key/index rebuild flow in this read-only review.

## Solution Assessment

No, not yet. The fallback is a plausible mitigation, but the best fix needs one canonical provider-resolution contract so config, startup loading, memory-core, and Gateway do not diverge.
