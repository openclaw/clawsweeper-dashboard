## Summary

Keep open: the PR addresses a real Google/Gemini memory-provider regression, but the resolver-only fallback is not yet a safe best fix because it can miss cold plugin activation and it broadens accepted memory provider ids without a maintainer-approved contract.

## What This Changes

The PR adds an `authProviderId` fallback to the shared memory embedding provider resolver and updates a memory-core regression test for resolving `google` to the Gemini adapter.

## Best Possible Solution

Choose one memory provider-id contract, then support it through manifest/cold startup resolution, runtime lookup, diagnostics or docs, and focused tests; otherwise keep `gemini` canonical with migration or diagnostic guidance.

## Reproduction Assessment

Yes, by source inspection. Current main and `v2026.6.9` expose the Google memory adapter as `gemini` while the reported `memorySearch.provider: "google"` can reach memory provider initialization by id.

## Solution Assessment

No. Centralizing the fallback is cleaner than duplicated call-site scans, but this runtime-only alias is too broad and misses cold activation unless the provider-id contract is chosen and implemented end to end.
