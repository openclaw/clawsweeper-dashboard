## Summary

Keep open: the patch is aimed at the right queue/Gateway/CLI boundary and includes useful real proof, but the new busy rejection still fires only when the session lane drains, so a long active lane can still outlive the Gateway final wait before any busy payload is returned.

## What This Changes

The PR adds command-queue wait details, a queue rejection hook, Gateway lane-busy final payloads, and CLI rendering/tests for blocked same-session agent dispatches.

## Best Possible Solution

Make lane-busy finalization timeout-driven while the session entry is still queued, keep non-CLI lane serialization unchanged, and prove both short-drain and long-active-lane cases.

## Reproduction Assessment

Yes, at source level: current main serializes same-session runs through the session lane and only emits lane-wait facts from the queue drain path, while the CLI waits for a generic Gateway final response. I did not run a new live repro in this read-only review, but the PR body includes before/after live output for a short held-lane case.

## Solution Assessment

No: the patch is a plausible boundary, but not the best fix yet because the rejection remains dequeue-time instead of threshold-time. The safer fix is a queue-owned wait timer or equivalent Gateway finalization while the entry is still queued.
