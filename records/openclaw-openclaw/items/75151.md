## Summary

Keep open. Current main still has the reset-path durability gap: the active session store can be persisted to a newly generated transcript path before that JSONL exists, and the replay helper can return without creating the target transcript, leaving history readers with an empty result when no candidate file exists.

## What This Changes

The issue reports that context-overflow or compaction reset can update a direct Telegram session’s `sessions.json` entry to a new nonexistent `sessionFile`, orphaning the previous real transcript and making history appear empty.

## Best Possible Solution

Enforce a reset-time invariant for embedded/direct sessions: create or exclusively prewarm the successor JSONL before updating the active store, keep the previous valid mapping if successor creation fails, ensure no replay early-exit can leave the new mapping without a durable transcript, and cover the missing-source, assistant-only-tail, and normal replay cases with focused regression tests.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert the persisted successor `sessionFile` exists; the current replay test already proves one of those branches leaves no target file.

## Solution Assessment

Yes. The best fix is a narrow durability guard at the reset boundary, possibly sharing the header helper being discussed in PR #55817; cleanup/doctor repair is useful afterward but too late for the active-session invariant.
