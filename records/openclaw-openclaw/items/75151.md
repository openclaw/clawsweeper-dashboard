## Summary

Keep open. Current main still has the reset durability gap: the reset path can publish a new `sessionFile` before the successor transcript is guaranteed to exist, and the replay helper still has source paths that return without creating that target.

## What This Changes

The issue requests that context-overflow or reset rotation never leave `sessions.json` pointing at a nonexistent transcript while the real prior history remains orphaned.

## Best Possible Solution

Enforce a reset-time durability invariant: create a non-clobbering header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation or replay fails.

## Reproduction Assessment

Yes, source-level. A focused test can call `resetReplyRunSession` with a missing or assistant-only previous transcript and assert that the persisted successor `sessionFile` exists instead of allowing history to resolve to `[]`.

## Solution Assessment

Yes. Fixing reset publication durability is the narrow maintainable path; cleanup or read-path repair may help recovery, but they do not prevent the store from pointing at a missing successor transcript.
