## Summary

Keep open. Current main still has the reset durability gap: it can publish a new reset `sessionFile` before the successor transcript is durable, replay can return without creating that target, and the related fix PRs are still open rather than merged.

## What This Changes

The issue reports that context-overflow reset/rotation can persist a Telegram direct session mapping whose `sessionFile` points to a nonexistent transcript, causing history lookup to return empty while the previous transcript still exists.

## Best Possible Solution

Land one coordinated session durability fix that creates an exclusive header-only successor transcript before publishing the reset mapping, or keeps the previous valid mapping if successor creation or replay fails, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. A focused test can call `resetReplyRunSession` with a missing or assistant-only previous transcript, then assert that the persisted successor `sessionFile` exists instead of allowing history to resolve to `[]`.

## Solution Assessment

Yes. The narrow maintainable fix is a write-path durability invariant; read-path repair and diagnostics can help recovery, but they should not be the primary prevention mechanism.
