## Summary

Keep open. Current main still permits the context-overflow reset path to persist a successor `sessionFile` before the successor JSONL is durable, and the replay helper still has early-return branches that leave the target transcript absent.

## What This Changes

The issue reports that context-overflow reset can map an active Telegram direct session to a newly generated `sessionFile` that does not exist, orphaning the real prior transcript from history lookup.

## Best Possible Solution

Enforce a reset-time durability invariant: create or prewarm the successor JSONL before publishing the new active store entry, keep the prior valid mapping if creation fails, and make replay unable to early-return without a durable target once reset is going to persist the successor. Keep cleanup/doctor as repair for already-corrupt stores, not the primary continuity mechanism.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert that the persisted successor `sessionFile` exists; current replay tests already show the assistant-only branch returns `0` and leaves the target absent.

## Solution Assessment

Yes. The narrowest maintainable fix is at the reset/replay durability boundary; read-path cleanup is too late for active continuity, and the related helper work does not cover this reset path unless it is explicitly reused here.
