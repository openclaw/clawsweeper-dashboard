## Summary

Keep open. Current main still has a reset/replay durability gap where a new `sessionFile` can be published before any successor transcript is guaranteed to exist, matching the reported orphaned-history failure mode.

## What This Changes

The issue requests durable context-overflow reset handling so an active session key never points at a missing transcript while the real prior transcript remains orphaned.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only or replay-populated successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation fails.

## Reproduction Assessment

Yes, from source inspection. A focused regression can call `resetReplyRunSession` with a missing or assistant-only previous transcript and assert the persisted successor `sessionFile` exists; existing replay coverage already proves the assistant-only target-absent branch.

## Solution Assessment

Yes. The narrow maintainable fix is a reset/replay write-path guard before store publication; cleanup and read-path repair are useful diagnostics but too late to preserve active continuity.
