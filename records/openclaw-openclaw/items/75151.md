## Summary

Keep open. Current main still has the reset durability gap: the reset path persists a new `sessionFile` before the successor transcript is guaranteed to exist, and replay still has early-return branches that leave the target JSONL absent.

## What This Changes

Prevent context-overflow session reset from remapping an active session key to a nonexistent transcript file and orphaning the previous real history.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation or replay fails.

## Reproduction Assessment

Yes from source inspection. A focused unit test can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert that the persisted successor `sessionFile` exists.

## Solution Assessment

Yes. The narrow maintainable fix is a write-path durability guard in reset/replay; read-path cleanup remains useful recovery tooling but is too late to preserve active continuity.

## Repair Work Prompt

Fix the missing-transcript reset invariant for https://github.com/openclaw/openclaw/issues/75151. In `resetReplyRunSession`, do not persist or publish a successor `sessionFile` unless the successor transcript has been durably created; prefer a header-only JSONL when no user/assistant replay tail is available, and preserve the previous valid mapping if creation fails. Update `replayRecentUserAssistantMessages` or add a small helper so missing source, empty kept records, and assistant-only tails still leave a valid header-only target when reset needs one. Add regression coverage for missing-source and assistant-only reset paths, keep changes scoped to reset/replay/session history behavior, and do not add a config option or broad transcript rotation feature. Add a user-facing changelog fix entry.
