## Summary

Keep open. Current main still allows the reset path to persist a new `sessionFile` before the successor transcript is durable, and replay can return without creating that file, matching the reported orphaned-history failure mode.

## What This Changes

The issue reports that context-overflow reset can persist a new session mapping whose transcript file was never created, orphaning the real prior transcript.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation fails.

## Reproduction Assessment

Yes. A focused unit test can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only and assert the persisted successor `sessionFile` exists; current replay coverage already proves one branch leaves the target absent.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the reset/replay path before the store entry is published; read-path cleanup is useful repair tooling but too late to preserve active continuity.
