## Summary

Keep open. Current main still allows the reset path to persist a new `sessionFile` before the successor transcript is guaranteed to exist, and replay has early-return paths that leave the published target absent.

## What This Changes

The issue asks OpenClaw to preserve session continuity by never rotating a live session mapping to a missing transcript after context-overflow reset or compaction failure.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation/replay fails.

## Reproduction Assessment

Yes, from source inspection. A focused regression test can call `resetReplyRunSession` with a missing or assistant-only previous transcript and assert the persisted successor `sessionFile` exists; the existing replay test already proves one target-absent branch.

## Solution Assessment

Yes. The narrow maintainable fix is a write-path durability guard in reset/replay; cleanup and read-path repair are useful but too late to preserve active continuity.

## Repair Work Prompt

Fix openclaw/openclaw#75151. Current main can persist a reset successor `sessionFile` before the transcript exists: `resetReplyRunSession` writes the next entry, then `replayRecentUserAssistantMessages` may return before creating a header when the source transcript is missing, empty, or assistant-only. Implement the narrow durability invariant: reset must not publish a `sessions.json` mapping to a nonexistent transcript. Prefer creating a header-only successor transcript before updating the store, or keeping the previous valid mapping if successor creation fails. Preserve existing replay behavior for valid tails, add regression coverage for missing-source and assistant-only replay/reset cases, and do not add a config option or broad transcript-rotation feature. Coordinate with #55817 conceptually but fix the reset/replay path directly. Add a user-facing CHANGELOG.md fix entry.
