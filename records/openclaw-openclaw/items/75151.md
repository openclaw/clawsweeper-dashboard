## Summary

Keep open. Current main still permits the reset path to persist a new `sessionFile` before a durable successor JSONL exists; replay can return before header creation in missing-source, empty-tail, and assistant-only-tail paths, and history readers return empty when no candidate transcript exists. Existing cleanup and doctor paths are remedial, while open PR #55817 is not merged and does not cover this reset path on current main.

## What This Changes

The issue requests a durability guard so context-overflow reset or rotation never leaves `sessions.json` pointing a direct Telegram session at a nonexistent transcript while prior history remains orphaned.

## Best Possible Solution

Enforce a reset-time invariant for direct/embedded session rotations: create or prewarm the successor JSONL before publishing the new store mapping, keep or restore the previous valid mapping if successor creation fails, and leave cleanup/doctor repair as a fallback rather than the primary data-integrity mechanism.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert the persisted successor `sessionFile` exists; the current replay test already proves the assistant-only branch leaves no target file.

## Solution Assessment

Yes. The narrowest maintainable fix is at the reset/replay durability boundary, optionally sharing a header helper with PR #55817; a cleanup-only or doctor-only repair would happen after the active session has already been remapped to a missing transcript.
