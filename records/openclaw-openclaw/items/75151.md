## Summary

Keep open. Current main still has the reset durability gap: the reset path publishes a new `sessionFile` before the successor JSONL is durable, and replay can return without creating that file, matching the reported orphaned-history failure mode.

## What This Changes

The issue asks OpenClaw to make context-overflow session reset durable so `sessions.json` never points at a missing successor transcript and previous history remains recoverable.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation or replay fails.

## Reproduction Assessment

Yes. A focused unit test can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only and assert the persisted successor `sessionFile` exists; current replay coverage already proves one branch leaves the target absent.

## Solution Assessment

Yes. The narrowest maintainable fix is in the reset/replay path before the store entry is published; read-path cleanup is useful as repair tooling but too late to preserve active continuity.

## Repair Work Prompt

Fix the reset durability bug from https://github.com/openclaw/openclaw/issues/75151. In current main, `resetReplyRunSession` persists a new `sessionFile` before `replayRecentUserAssistantMessages` creates the successor JSONL, and replay can return without creating it when the source is missing/empty/assistant-only. Add a reset-time invariant so the successor transcript exists before the store mapping is published, or keep the previous valid mapping if successor creation/replay fails. Prefer a header-only transcript helper shared with existing transcript append/session utilities if that fits local patterns. Add regression coverage for missing-source and assistant-only replay through `resetReplyRunSession`, and preserve the existing stale-transcript fallback behavior. Do not broaden compaction policy, provider behavior, or unrelated session cleanup semantics. Add a `CHANGELOG.md` Fixes entry because this is user-facing history loss. Coordinate with related PR #55817 so duplicate initSessionState work is not reimplemented if that PR lands first.
