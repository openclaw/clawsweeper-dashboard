## Summary

Keep open. Current main still allows the reset path to persist a new `sessionFile` before the successor transcript is guaranteed to exist, and replay has early-return branches that leave the published target absent.

## What This Changes

The issue asks OpenClaw to preserve session-history continuity when context-overflow reset rotates a session by ensuring `sessions.json` never points at a nonexistent transcript.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation or replay fails.

## Reproduction Assessment

Yes from source inspection. A focused regression test can call `resetReplyRunSession` with a missing or assistant-only previous transcript and assert the persisted successor `sessionFile` exists; existing replay coverage already proves one target-absent branch.

## Solution Assessment

Yes. The narrow maintainable fix is a write-path guard in reset/replay durability, while cleanup and read-path repair remain useful but too late to preserve active continuity.

## Repair Work Prompt

Fix the context-overflow reset durability gap reported in https://github.com/openclaw/openclaw/issues/75151. In `resetReplyRunSession`, do not publish a successor `sessionFile` that can remain nonexistent: create a header-only successor transcript before updating the session store, or keep the previous valid mapping if successor creation/replay fails. Update `replayRecentUserAssistantMessages` or add a small shared helper as needed so missing/assistant-only/empty source transcripts still leave a durable header for the new session. Add regression coverage for missing source and assistant-only source through the reset path. Preserve existing stale-session read fallback behavior in `resolveSessionTranscriptCandidates`; do not add config knobs or broad compaction policy changes. Coordinate with #55817 if its header helper lands first. Add a single-line `CHANGELOG.md` fix entry because this is user-facing session continuity behavior.
