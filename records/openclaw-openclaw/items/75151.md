## Summary

Keep open. Current main still has the reset durability gap: the reset path publishes a new `sessionFile` before the successor transcript is durable, and replay can return without creating the target JSONL, matching the reported empty-history failure mode.

## What This Changes

The issue requests durable handling of context-overflow session resets so `sessions.json` does not point at a nonexistent transcript and recent history remains recoverable.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation/replay fails.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert the persisted successor `sessionFile` exists; current replay tests already prove one branch leaves the target absent.

## Solution Assessment

Yes. The narrowest maintainable fix is in `resetReplyRunSession` and `replayRecentUserAssistantMessages`; read-path cleanup is useful repair tooling but too late to preserve active continuity.
