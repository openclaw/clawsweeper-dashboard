## Summary

Keep open. Current main still has a source-reproducible reset durability gap: the reset path persists a new sessionFile before the successor transcript is guaranteed to exist, and the replay helper can return without creating that file.

## What This Changes

The issue reports that context-overflow reset can persist a session mapping whose sessionFile points to a nonexistent transcript, orphaning the prior valid history.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation or replay fails.

## Reproduction Assessment

Yes, source-level. A focused test can call resetReplyRunSession with a missing or assistant-only previous transcript and then assert that the persisted successor sessionFile exists instead of allowing history to resolve to [].

## Solution Assessment

Yes. The narrow maintainable fix is the write-path durability invariant; cleanup and read-path repair are useful diagnostics, but they should not be the primary prevention mechanism.
