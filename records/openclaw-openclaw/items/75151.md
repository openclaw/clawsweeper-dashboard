## Summary

Keep open. Current main still has the reset/replay durability gap described in the report: the reset path can publish a new active `sessionFile` before any successor transcript is durable, and the replay helper still has early-return paths that leave the target JSONL absent.

## What This Changes

The issue requests that context-overflow or compaction reset paths stop remapping an active Telegram direct session to a nonexistent transcript while the prior real transcript remains orphaned.

## Best Possible Solution

Enforce a reset-time durability invariant: create or prewarm the successor JSONL before publishing the new active store entry, keep the previous valid mapping if target creation fails, and make replay unable to early-return without a durable target once reset is going to persist the successor.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert that the persisted successor `sessionFile` exists; current replay tests already show the assistant-only branch returns `0` and leaves the target absent.

## Solution Assessment

Yes. The narrowest maintainable fix is at the reset/replay durability boundary; read-path cleanup and the related `initSessionState()` rollover work are useful but too late or too narrow to guarantee active continuity for this reset path.

## Repair Work Prompt

Fix the reset/replay durability gap for https://github.com/openclaw/openclaw/issues/75151. Current `resetReplyRunSession` can persist a new active `sessionFile` before the successor JSONL exists, and `replayRecentUserAssistantMessages` can early-return without creating the target when the source is missing, empty, or assistant-only. Implement the narrow reset-time invariant: before publishing or before considering reset complete, ensure the successor transcript file is durably created with a valid session header; if creation fails, do not leave `sessions.json`, the active in-memory store, or queued followups pointing at a nonexistent target. Keep cleanup/doctor as remediation only, not the primary continuity mechanism. Add regression coverage for missing source and assistant-only source through `resetReplyRunSession`, and update the replay tests so target durability is guaranteed without replaying unsafe assistant-only content. Do not broaden this into transcript size rotation or the open #70472 umbrella. Add a user-facing changelog fix entry unless maintainers classify the change as internal-only.
