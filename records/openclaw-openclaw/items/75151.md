## Summary

Keep open. Current `main` still permits the reported durability gap: the auto-reply reset path can persist a new active `sessionFile` before any successor JSONL exists, and the replay helper has early-return paths that leave that target file absent, so history readers can still return an empty result for the active mapping.

## What This Changes

The issue asks OpenClaw to prevent context-overflow reset/session rotation from persisting a `sessionFile` that points to a nonexistent transcript, which can make history appear empty while the prior transcript remains on disk.

## Best Possible Solution

Enforce a reset-time invariant for embedded/direct session rotation: the successor transcript must be durably created before the active session store points at it, or the old valid mapping must remain in place. Replay should preserve recent continuity when possible, but every reset outcome should leave history tooling with a real current transcript rather than an empty lookup.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert the persisted successor `sessionFile` exists; current replay coverage already proves the assistant-only branch leaves no target file.

## Solution Assessment

Yes. The narrowest maintainable fix is a durability guard at the auto-reply reset/replay boundary, coordinated with #55817 if that helper lands; cleanup/doctor repair is useful but does not prevent the active mapping from becoming invalid.
