## Summary

Keep open. Current main still has the reset durability gap: `resetReplyRunSession` persists a new `sessionFile` before the successor transcript is durable, and replay still has early-return paths that leave the target JSONL absent.

## What This Changes

Prevent context-overflow or reset recovery from mapping an active session to a nonexistent transcript file and losing visible continuity.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new mapping, or keep the previous valid mapping if successor creation or replay fails.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose source transcript is missing or assistant-only, then assert the persisted successor `sessionFile` exists; current replay tests already prove one branch leaves the target absent.

## Solution Assessment

Yes. The narrowest maintainable fix is in `resetReplyRunSession` and `replayRecentUserAssistantMessages`; read-path cleanup is useful repair tooling but too late to preserve active continuity.

## Repair Work Prompt

Fix the session reset durability bug described in this issue. Current main lets `resetReplyRunSession` persist a new `sessionFile` before the successor transcript exists, while `replayRecentUserAssistantMessages` can return without creating the target when the source is missing, empty, or assistant-only. Implement the narrow invariant that a reset-created successor transcript is durable before the active store/follow-up mapping switches, or keep the previous valid mapping and surface a recoverable error if creation fails. Preserve existing replay-tail behavior and role-ordering safeguards. Check overlap with open PR #55817 and related issue #70472 before changing shared rollover code; do not take on transcript size-cap policy or broad compaction redesign in this PR. Add a user-facing changelog entry unless the final patch is test-only.
