## Summary

Keep open. Current main still persists the reset mapping before the successor transcript is durable, and replay has early-return branches that can leave the published `sessionFile` absent.

## What This Changes

The issue asks OpenClaw to preserve session continuity by never mapping a reset or rotated session to a nonexistent transcript file.

## Best Possible Solution

Enforce a reset-time durability invariant: create a header-only successor transcript before publishing the new store entry, or keep the previous valid mapping if successor creation/replay fails.

## Reproduction Assessment

Yes, from source inspection. Calling `resetReplyRunSession` with a missing or assistant-only previous transcript can persist a new `sessionFile` before replay returns without creating the target file.

## Solution Assessment

Yes. The narrow maintainable fix is in reset/replay durability; cleanup and read-path repair are useful backstops but happen too late to preserve active continuity.
