## Summary

Keep open: current main still has only an internal TUI streaming watchdog override, while the production TUI path, config schema, CLI docs, and env-var surface do not expose a way for users to disable the chat-log message.

## What This Changes

Request a public config or env knob to disable or tune the TUI streaming watchdog message while preserving the current 30s default.

## Best Possible Solution

Add a documented public TUI setting, thread it into `createEventHandlers`, preserve the 30000ms default, treat `0` as disabled, and cover config parsing plus the production `runTui` wiring in tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: any active TUI chat run that receives a delta and then stays silent for the 30000ms watchdog window will append the system message. Existing tests also prove the internal handler can disable the watchdog with `streamingWatchdogMs: 0`, but current production wiring exposes no public way to set it.

## Solution Assessment

Unclear until maintainers choose the public surface; the existing internal handler option is the right low-level seam, but adding a user-facing config key or env var is a product/config contract decision.
