## Summary

Current main and the shipped v2026.5.2 release already implement the requested Slack outbound target-account routing through a safer shared resolver, while this branch would be unsafe to merge because it scans raw bindings and can ignore agent-scoped peer collisions.

## What This Changes

The PR adds a raw `cfg.bindings` peer-id scan in `runMessageAction` before the existing agent-keyed account fallback.

## Best Possible Solution

Keep the released resolver-based implementation and close this superseded branch instead of merging its raw binding scan.

## Reproduction Assessment

Yes for the historical bug. The PR body gives a concrete two-account Slack reproduction, and current source shows the old fallback was replaced by target-aware resolution with focused regression coverage.

## Solution Assessment

Yes for current main, no for this branch. The shipped resolver is the narrower maintainable fix because it is route-only, agent-scoped, and peer-aware; this branch's raw scan can choose the wrong credential-bound account.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Filter out ACP bindings before account selection:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: This loop iterates raw `cfg.bindings`, so ACP bindings with the same channel and peer id can be selected for outbound sends even though account routing helpers intentionally use route bindings only. That can route a message through the wrong Slack bot account.
  - confidence: 0.88
- **[P1] Keep peer lookup scoped to the resolved agent:** `src/infra/outbound/message-action-runner.ts:854-866`
  - body: The new fast path matches only channel plus peer id and exits before the agent-keyed fallback. If two agents intentionally bind different accounts to the same Slack peer, the later agent can be forced onto whichever binding appears first.
  - confidence: 0.86
