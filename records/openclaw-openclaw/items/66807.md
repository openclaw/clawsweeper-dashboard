## Summary

Keep open: the reported outbound routing gap still exists on current main, but this PR branch is not safe to merge because its peer lookup can select non-route bindings and can become order-dependent when the same peer is bound for multiple agents.

## What This Changes

The PR adds a pre-fallback account lookup in `runMessageAction` that scans configured bindings by channel and target peer before using the existing agent-keyed account fallback.

## Best Possible Solution

Add a typed outbound account resolver for message actions that keeps explicit `--account` and `defaultAccountId` precedence, uses route-only binding data, chooses a peer-bound account only when the match is unique or agent-safe, and covers the Slack multi-account CLI case with focused regression tests.

## Reproduction Assessment

Yes. The PR body gives a concrete Slack multi-account config, and current main's source path confirms that a CLI send without `--account` uses the default agent's channel account without looking at the target peer.

## Solution Assessment

No. The proposed direction is valid, but this implementation is not the safest fix because it scans raw bindings and ignores ambiguity across agents; a route-only, unique-or-agent-scoped resolver is the safer alternative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter peer lookup to route bindings:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: This loop scans raw `cfg.bindings`, but that array also contains `type: "acp"` bindings. If an ACP binding has the same channel, peer id, and accountId, this fast path can select it for an outbound message even though the existing account fallback filters through route bindings only.
  - confidence: 0.92
- **[P2] Keep peer lookup agent-safe when bindings collide:** `src/infra/outbound/message-action-runner.ts:852-865`
  - body: The new fast path matches only channel plus `peer.id` and runs before the current agent-keyed fallback. If two agents intentionally bind different accounts to the same Slack channel, the second agent can be forced onto the first matching binding, making account routing depend on config order.
  - confidence: 0.86
