## Summary

Current main already contains a safer shipped fix for the same Slack outbound account-routing bug, and the PR branch is now obsolete and still has the raw-binding/account-collision issues identified in review.

## What This Changes

This PR adds a peer-id scan in `runMessageAction` to select a Slack account from configured bindings before the existing agent-keyed fallback.

## Best Possible Solution

Keep the released `v2026.5.7` implementation, which solves the routing bug through the shared route-binding resolver with regression coverage, and close this superseded branch.

## Reproduction Assessment

Yes. The PR body gives a concrete two-account Slack config and the old source path ignored the outbound target peer; current main has since replaced that path with target-aware resolution.

## Solution Assessment

Yes for current main, no for the PR branch. The shipped fix is narrower and safer because it uses route-only, agent-scoped, peer-aware resolver logic instead of scanning raw bindings by channel and peer id only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter peer lookup to route bindings:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: The new loop scans raw `cfg.bindings`, which can include `type: "acp"` entries. An ACP binding with the same channel, peer id, and account id could be selected for outbound message routing even though account routing should only consider route bindings.
  - confidence: 0.92
- **[P2] Preserve agent scoping when peer bindings collide:** `src/infra/outbound/message-action-runner.ts:852-865`
  - body: The fast path matches only channel plus `peer.id` before the existing agent fallback runs. If two agents bind different accounts to the same Slack channel, sends can become config-order dependent and use the wrong account.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR branch changes credential-bound Slack account routing and can select an unintended bot identity; current main's shipped fix avoids that path.

Concerns:

- **[medium] Wrong Slack account can be selected:** `src/infra/outbound/message-action-runner.ts:849`
  - body: The raw binding scan and lack of agent-safe ambiguity handling can route an outbound message through an unintended Slack account. Because account ids map to bot tokens and permissions, this is a credential-boundary regression.
  - confidence: 0.86
