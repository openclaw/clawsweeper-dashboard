## Summary

Current main and release v2026.5.7 already fix the multi-account Slack outbound routing bug with a safer route-only, agent-scoped, peer-aware resolver and regression coverage; the PR branch is superseded and still has credential-bound account-selection defects.

## What This Changes

The PR adds a peer-id scan over config bindings in `runMessageAction` to choose a Slack account for outbound message actions before the existing agent-keyed fallback.

## Best Possible Solution

Keep the released v2026.5.7 resolver-based implementation and close this superseded branch.

## Reproduction Assessment

Yes. The PR body gives a concrete two-account Slack setup, and the old account fallback ignored the target peer; current main has target-aware route resolution and a focused regression test for that path.

## Solution Assessment

Yes for current main, no for the PR branch. The shipped implementation is the narrower maintainable fix because it reuses the route-binding resolver instead of adding a raw binding scan inside `runMessageAction`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve agent scope when matching peer bindings:** `src/infra/outbound/message-action-runner.ts:849-852`
  - body: The new fast path selects the first binding matching only channel and peer id before the existing agent-keyed fallback. If two agents/accounts are intentionally bound to the same Slack peer, sends for the later agent can be forced through the first bot and fail with permission or `channel_not_found` errors.
  - confidence: 0.87
- **[P1] Exclude ACP bindings from outbound route lookup:** `src/infra/outbound/message-action-runner.ts:849-852`
  - body: Scanning raw `cfg.bindings` includes `type: "acp"` bindings, while the existing account-routing helpers only consider route bindings. A matching ACP persistence binding could incorrectly choose a delivery account for an outbound send.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR branch changes Slack account routing and can select an unintended bot identity; current main's route-only resolver avoids that credential-boundary regression.

Concerns:

- **[medium] Wrong Slack account can be selected:** `src/infra/outbound/message-action-runner.ts:849`
  - body: The raw binding scan and lack of agent-safe ambiguity handling can route outbound messages through an unintended Slack account. Since account ids map to bot tokens and workspace permissions, this is a credential-boundary risk if the branch were merged.
  - confidence: 0.86
