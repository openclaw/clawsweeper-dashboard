## Summary

Current main and the shipped v2026.5.2 release already implement the central Slack outbound target-account routing through a safer shared resolver, while this PR branch is unsafe because its raw binding scan can select the wrong credential-bound account.

## What This Changes

The PR adds a peer-aware raw binding scan in `runMessageAction` before the existing agent-keyed account fallback.

## Best Possible Solution

Keep the released resolver-based implementation and close this superseded branch instead of merging its raw binding scan.

## Reproduction Assessment

Yes for the historical bug from source and the PR's concrete two-account Slack scenario. I did not rerun live Slack, but current source and regression coverage verify that the old target-blind fallback has been replaced.

## Solution Assessment

Yes for current main, no for this branch. The shipped resolver is the narrower maintainable fix because it is route-only, agent-scoped, peer-aware, and already covered by regression tests; this branch's raw scan can choose the wrong account.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve agent scoping in peer account lookup:** `src/infra/outbound/message-action-runner.ts:866`
  - body: This fast path matches only `channel` plus `peer.id` and assigns `accountId` before the existing agent-keyed fallback can run. If two agents intentionally bind different Slack accounts to the same channel, sends from the later agent will use whichever binding appears first, causing cross-account routing and permission failures.
  - confidence: 0.9
- **[P2] Filter outbound lookup to route bindings:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: Scanning raw `cfg.bindings` includes ACP bindings, while the existing account helpers use route bindings only. An ACP binding with the same channel and peer can be selected for message sends even though it is not an outbound route binding.
  - confidence: 0.86

## Security Review
