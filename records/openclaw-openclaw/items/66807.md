## Summary

Close: current main and v2026.5.2 already ship the safer route-only, agent/peer-aware Slack account resolver with focused regression coverage, while this branch is superseded and risky to merge.

## What This Changes

This PR adds a peer-aware account lookup in `runMessageAction` that scans bindings by channel and peer target before the existing agent-keyed fallback.

## Best Possible Solution

Keep the released resolver-based implementation and close this superseded branch instead of merging its raw binding scan.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the historical bug from the PR's two-account Slack scenario; current main no longer reproduces that path because it routes through the target-aware resolver and has regression coverage.

## Solution Assessment

Is this the best way to solve the issue? Yes for current main, no for this PR branch. The shipped resolver is route-only, agent-scoped, and peer-aware; the branch's raw scan can choose the wrong credential-bound account.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter out non-route bindings before selecting accounts:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: The new lookup iterates raw `cfg.bindings`, so ACP bindings with a matching channel and peer can be selected for outbound message routing even though the existing route helpers exclude them. Account ids map to bot credentials, so this can silently send through the wrong Slack identity.
  - confidence: 0.88
- **[P1] Preserve agent scoping when peer matches collide:** `src/infra/outbound/message-action-runner.ts:864-867`
  - body: This loop chooses the first binding matching only channel and peer id, before the existing agent-keyed fallback runs. If two agents intentionally bind different Slack accounts to the same peer, sends from the later agent will be forced onto the first account in config order.
  - confidence: 0.9
