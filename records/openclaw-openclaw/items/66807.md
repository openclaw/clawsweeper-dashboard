## Summary

Keep open: current main still has the reported peer-blind outbound account fallback, and the PR addresses a real bug but is not safe to merge as written because it can select ACP bindings or the wrong account when peer bindings collide.

## What This Changes

The PR adds a peer-id scan before the existing agent-keyed account fallback in `runMessageAction` so outbound message actions can choose an account from bindings that match the target peer.

## Best Possible Solution

Use the existing route-binding resolver semantics for outbound message actions: explicit account/defaults still win, peer-bound route bindings are honored with agent-safe precedence, ACP bindings are ignored, and the Slack multi-account CLI path gets regression coverage.

## Reproduction Assessment

Yes. The PR body gives a concrete multi-account Slack reproduction, and current main confirms the CLI send path supplies the default agent and then falls back through an agent/channel account map that ignores the target peer.

## Solution Assessment

No. The proposed direction is right, but the implementation is not the safest fix because it scans the raw binding union and chooses the first channel/peer match without preserving route-only and agent-safe account scoping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter peer lookup to route bindings:** `src/infra/outbound/message-action-runner.ts:846-849`
  - body: This loop scans raw `cfg.bindings`, but that array also includes `type: "acp"` entries with the same `match` shape. An ACP binding with the same channel, peer id, and accountId can be selected for an outbound message even though the existing fallback only considers route bindings.
  - confidence: 0.92
- **[P2] Keep peer lookup agent-safe when bindings collide:** `src/infra/outbound/message-action-runner.ts:850-864`
  - body: The new fast path matches only channel plus `peer.id` and runs before the agent-keyed fallback. If two agents bind different accounts to the same Slack channel, sends from the second agent can be forced onto the first matching binding, making account routing depend on config order.
  - confidence: 0.86
