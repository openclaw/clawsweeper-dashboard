## Summary

Keep this PR open. The reported multi-account Slack routing bug is still present on current main, but the proposed patch is not merge-safe because it bypasses route-binding filtering and can override an existing agent-specific account choice when multiple bindings share a peer.

## What This Changes

The PR branch adds a pre-fallback peer-aware account lookup in `runMessageAction` that scans configured bindings by channel and target peer before using the existing agent-keyed fallback.

## Best Possible Solution

Add a typed outbound account resolver for message actions that preserves explicit `--account` and `defaultAccountId` precedence, uses route-only binding data, applies peer-aware matching only when it can do so without cross-agent ambiguity, and covers the Slack multi-account case with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with two Slack account bindings and a CLI send that omits `--account`, `messageCommand` supplies the default agent and current `runMessageAction` chooses an agent-keyed account without considering the target peer.

## Solution Assessment

No. The proposed direction is valid, but this patch is not the best implementation because it scans raw config bindings and ignores agent scope when peer matches collide; a safer fix should reuse or adapt the route-only peer-aware resolver and add regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Filter peer lookup to route bindings:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: This loop scans raw `cfg.bindings`, but that array also contains `type: "acp"` bindings. If an ACP binding has the same channel, peer id, and accountId, this new fast path can select it for an outbound message even though existing route account logic filters through `listRouteBindings`, causing sends to use the wrong account.
  - confidence: 0.92
- **[P2] Preserve agent scoping for duplicate peer bindings:** `src/infra/outbound/message-action-runner.ts:852-865`
  - body: The new fast path matches only channel plus `peer.id` and runs before the current agent-keyed fallback. For configs with two accounts bound to the same Slack channel for different agents, a send with a real second-agent `agentId` now takes the first matching peer binding instead of that agent's account, so routing becomes order-dependent.
  - confidence: 0.86
