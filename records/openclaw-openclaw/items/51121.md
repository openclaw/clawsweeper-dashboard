## Summary

Keep this PR open, but it still needs changes before merge. Current main still has the un-cleared iMessage stop timeout, so the contribution is aimed at a real leak, but the proposed `finally` cleanup can cancel the fallback kill on a reused `IMessageRpcClient` because the close promise is instance-scoped and remains resolved after the first child exits.

## What This Changes

The PR changes `extensions/imessage/src/client.ts` so `IMessageRpcClient.stop()` stores the 500 ms timeout handle and clears it in a `finally` block after the close race finishes.

## Best Possible Solution

Keep the timer cleanup direction, but make the close waiter state per spawned child or otherwise bind the stop race to the child being stopped. Add focused fake-timer coverage for early child close, hung child fallback kill, and restart-then-stop reuse before merging.

## Reproduction Assessment

Yes. A high-confidence static and fake-timer reproduction is to let one `IMessageRpcClient` child close, call `start()` again on the same instance, then call `stop()` on a hung second child; on the PR branch, stale `this.closed` wins immediately and `clearTimeout(stopTimer)` cancels the fallback `SIGTERM`.

## Solution Assessment

No. Clearing the orphaned timer is the right goal, but doing it without resetting or scoping the close promise per child is not the narrowest safe fix because it can remove the only fallback cleanup on reused clients.
