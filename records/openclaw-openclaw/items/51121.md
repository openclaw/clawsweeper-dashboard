## Summary

Keep this PR open: the timer cleanup goal is valid, but the current patch can cancel the fallback kill on a reused client because the close promise is instance-scoped and remains resolved after the first child exits.

## What This Changes

The PR changes `IMessageRpcClient.stop()` to store the 500 ms timeout handle and clear it in a `finally` block after the shutdown race.

## Best Possible Solution

Keep the timer cleanup, but scope or recreate the close waiter per spawned child, then add fake-timer coverage for early close, hung-child fallback, and client reuse plus the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: let one child close, call `start()` again on the same `IMessageRpcClient`, then call `stop()` on a hung second child; on the PR branch, stale `this.closed` wins immediately and the new fallback timer is cleared.

## Solution Assessment

No. Clearing the orphaned timer is the right direction, but the patch is not yet the narrowest safe fix because it does not reset or scope close state to the child whose timeout is being managed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:
