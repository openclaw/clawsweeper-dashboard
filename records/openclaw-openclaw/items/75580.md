## Summary

Keep this PR open, but it still needs one correctness fix before merge. The patch addresses the real multi-client-tool-call loss on current main, yet the new capture array is populated when each async client-tool execute path reaches the callback, so hook or approval delays can reorder calls under pi-agent-core's default parallel tool execution.

## What This Changes

The PR changes embedded-runner client-tool metadata from one call to an array, maps all detected calls into `pendingToolCalls`, emits all `/v1/responses` function-call items in JSON and SSE modes, and adds runner/gateway regression tests plus a changelog entry.

## Best Possible Solution

Keep the end-to-end multiple-call repair, but make client-tool capture deterministic against the assistant's source tool-call order and add a delayed-hook regression test before landing. The unrelated WhatsApp dependency drift should stay outside this PR.

## Reproduction Assessment

Yes. The original issue reproduces from source on current main with a multi-tool `/v1/responses` turn flowing through one overwritten runner slot and one emitted pending call. The remaining PR bug can be reproduced with multiple client tools where the first call's before-tool-call hook or approval resolves after a later call under parallel tool execution.

## Solution Assessment

No, not fully. The array-based shape is the right fix for preserving every call, but pushing into the array only after awaited client-tool hook work does not maintain the ordered API contract; the safer repair is to carry source order through capture or force delegated client tools through deterministic sequential execution.
