## Summary

Keep this PR open. Current main still has the original single-call loss, and the PR is the right general shape, but the latest patch still records client tool calls in async completion order under the default parallel tool runner, which can violate the ordered `/v1/responses` output contract it is trying to preserve.

## What This Changes

The PR changes embedded-runner client-tool metadata from one call to an array, emits all pending `/v1/responses` `function_call` items in JSON and SSE modes, adds multi-call regression tests, and updates the changelog.

## Best Possible Solution

Keep the end-to-end array repair, but make the capture order deterministic with respect to the assistant message's source tool-call order, add a delayed-hook regression test for that case, then land after targeted runner/gateway tests and the normal changed gate are clean.

## Reproduction Assessment

Yes. The original bug has a source-level reproduction on current main through one overwritten runner slot plus two `pendingToolCalls[0]` response emitters; the remaining PR bug can be reproduced with multiple client tools where an earlier tool's before-tool-call hook or approval path resolves after a later one.

## Solution Assessment

No, not fully. The array-based pipeline is the maintainable direction, but pushing into the array after awaited delegated execution is not the best way to guarantee ordered API output under parallel tool execution; source-order indexing or sequential client-tool execution would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve source order through async hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1470`
  - body: This records each client call only when that delegated tool reaches the callback. `pi-agent-core` runs prepared tool executions in parallel by default, and `toClientToolDefinitions` awaits before-tool-call hook/approval work before invoking the callback; if the first client tool waits longer than the second, `/v1/responses` will return function calls out of the assistant's source order. Please carry source order through the capture path or force delegated client tools through sequential execution.
  - confidence: 0.84
