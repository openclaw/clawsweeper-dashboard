## Summary

Keep this PR open. Current main still has the original multi-client-tool-call bug and the PR is the active implementation candidate, but the latest patch still records client tools in callback completion order; with parallel tool execution and awaited before-tool-call hooks, that can differ from the assistant message's source order.

## What This Changes

The PR changes embedded-runner client-tool metadata from one call to an array, maps it into multiple pending tool calls, emits all `/v1/responses` `function_call` items in JSON and SSE modes, and adds regression tests plus a changelog entry.

## Best Possible Solution

Keep the end-to-end array and multi-item response shape, but make client-tool capture deterministic against the assistant tool-call source order and add a delayed-hook regression before merging.

## Reproduction Assessment

Yes. The original issue is reproducible from current main by following the single-slot runner capture plus `pendingToolCalls[0]` JSON/SSE emitters; the remaining PR bug is reproducible with multiple client tools where an earlier tool's before-tool-call hook or approval path resolves after a later one.

## Solution Assessment

No, not fully. The array pipeline is the right direction, but appending when callbacks finish is not the narrowest maintainable way to guarantee ordered API output under parallel tool execution; source-order indexing or deliberately sequential client-tool execution would be safer.
