## Summary

Keep this PR open. It addresses a real `/v1/responses` bug and follows the useful prior #52287 shape, but the current patch should not merge until client-tool capture preserves the assistant's source order under async hooks or approval delays.

## What This Changes

The PR changes embedded-runner client-tool metadata from one stored call to an array, emits all pending `/v1/responses` function-call items in JSON and SSE responses, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the end-to-end array repair, but make the capture path deterministic before merge. A narrow fix should either force delegated client tools through pi-agent-core sequential execution or carry source-order metadata through the callback path, then add a delayed-hook regression test.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original bug: a `/v1/responses` turn with multiple client tools is collapsed by one runner slot and later by `pendingToolCalls[0]`. The PR-specific regression can be reproduced by delaying the first client tool's before-tool-call hook while pi-agent-core executes the batch in parallel.

## Solution Assessment

No, not fully. The array-based repair is the right broad shape, but appending callbacks after awaited hook work is not the narrowest maintainable way to guarantee the OpenResponses output order contract under parallel execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve source order through async hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1470`
  - body: This appends client calls when each delegated tool execution reaches the callback, but client tools still run under pi-agent-core's default parallel execution. Because `toClientToolDefinitions` awaits before-tool-call hook or approval work before invoking the callback, a slower first tool can be pushed after a faster later tool, causing `/v1/responses` to return function calls out of the assistant's source order. Please make the capture deterministic, for example by making client tool definitions sequential or by carrying source order through the capture path.
  - confidence: 0.82
