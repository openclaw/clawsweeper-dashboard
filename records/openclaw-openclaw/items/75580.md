## Summary

Keep this PR open. It fixes the current-main single-client-tool-call loss across the embedded runner and `/v1/responses` JSON/SSE emitters, and the latest push addressed the required changelog entry, but the proposed ordered-array capture still records client calls after async hook execution, so parallel tool execution can reorder multiple client calls before they reach callers.

## What This Changes

The PR changes embedded-runner client-tool metadata from a single call to an array, emits all pending `/v1/responses` `function_call` items in JSON and SSE responses, and adds runner/gateway regression tests plus a changelog entry.

## Best Possible Solution

Keep the end-to-end array repair, but make client-tool capture deterministic with respect to the assistant's source tool-call order, add a delayed-hook regression test for that edge, then land after the targeted runner/gateway suites and the normal changed gate are green.

## Reproduction Assessment

Yes. The original bug has a high-confidence source reproduction: a `/v1/responses` turn with three client function tools flows through one overwritten runner slot and one emitted pending call on current main. The remaining PR issue can be reproduced by adding an async before-tool-call hook or approval delay for the first of multiple client tools while pi-agent-core uses its default parallel execution.

## Solution Assessment

No, not fully. The array-based fix is the right shape for preserving every call, but recording calls only when the delegated tool execution reaches the callback is not the narrowest maintainable way to guarantee the ordered API contract under async hooks; the PR should either force client tools through sequential execution or carry source order through the capture path.
