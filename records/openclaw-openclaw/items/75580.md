## Summary

Keep open. The PR is the active implementation candidate for a valid multi-client-tool-call bug, but the latest patch still records calls when each async client-tool callback completes, so parallel tool execution can return `/v1/responses` function_call items out of the assistant message's source order.

## What This Changes

The PR changes embedded-runner client-tool metadata from a single call to an array, emits all pending `/v1/responses` function_call items in JSON and SSE modes, adds multi-call regression tests, and updates the changelog.

## Best Possible Solution

Keep the end-to-end array repair, but make client-tool capture deterministic with respect to assistant source order, preferably by carrying an ordinal or reserving a source-order slot before awaited hook work and then filling it after adjusted params are known. Add a delayed-hook regression test, keep the existing changelog entry, and land only after targeted runner/gateway tests plus the normal changed gate are clean.

## Reproduction Assessment

Yes. The original bug is source-reproducible on current main through one overwritten runner slot plus `pendingToolCalls[0]` emitters; the remaining PR bug can be reproduced with two client tools where the first tool's before-tool-call hook or approval path resolves after the second.

## Solution Assessment

No, not fully. The array-based pipeline is the right direction, but pushing into the array after awaited delegated execution is not the narrowest correct ordering fix under pi-agent-core's default parallel tool execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve source order through async hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1470`
  - body: This records each client call only when the delegated tool reaches the callback. The pinned runner executes prepared tool bodies in parallel by default, and `toClientToolDefinitions` awaits before-tool-call hook work before invoking the callback; if an earlier client tool waits longer than a later one, `/v1/responses` will return function calls out of the assistant's source order. Please carry source order through the capture path or force delegated client tools through sequential execution.
