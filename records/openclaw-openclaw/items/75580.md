## Summary

Keep this PR open. Current main still has the multi-client-tool bug from #52288, and prior PR #52287 was closed unmerged, but the latest patch still records calls in callback completion order, which can differ from assistant source order under parallel tool execution.

## What This Changes

The branch changes embedded-runner client-tool metadata from one call to an array, emits all pending `/v1/responses` `function_call` items in JSON and SSE responses, adds multi-call regression tests, and updates the changelog.

## Best Possible Solution

Keep the end-to-end array repair, make capture deterministic against assistant source order, add delayed-hook coverage, then land after targeted runner/Gateway tests and the normal changed gate pass.

## Reproduction Assessment

Yes. The original bug has a source-level reproduction on current main through one overwritten runner slot plus two `pendingToolCalls[0]` response emitters; the remaining PR bug can be reproduced with multiple client tools where an earlier before-tool-call hook resolves after a later one.

## Solution Assessment

No, not fully. The array-based pipeline and HTTP iteration are the right direction, but callback completion order is not a stable API order under parallel tool execution; carrying source index/tool-call identity through capture is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve source order through async client-tool hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1470`

Please key the capture by source order or run delegated client tools sequentially.
