## Summary

Keep open: current main and v2026.6.8 still cancel an older foreground delivery as soon as a newer visible generation wins, and three open linked PRs explicitly close this issue but none has merged yet.

## What This Changes

Requests changing foreground reply fencing so a newer visible delivery does not cancel an older generation that is still active/in-flight.

## Best Possible Solution

Land one reviewed foreground-fence fix that prevents in-flight reply truncation while preserving stale-suppression and ordering invariants, then close this issue from the merged PR.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still return true once a newer visibleDeliveryGeneration exceeds the older snapshot, and the buffered dispatcher creates that fence before the deeper admission path. I did not run a live Telegram bridge reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation: the proposed active-generation guard directly addresses truncation, but related issue https://github.com/openclaw/openclaw/issues/93858 shows the final fix should also prove ordering and stale-suppression behavior. The issue itself is still necessary because no linked fix has merged.
