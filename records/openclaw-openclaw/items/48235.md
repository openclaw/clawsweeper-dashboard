## Summary

Keep open: current main still lacks the requested continuity behavior, but this PR is not merge-ready because the patch has source-reproducible prompt delivery, prompt-safety, and type-contract blockers, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds a session-entry field, transcript-tail assistant reader, scheduled daily/idle reset capture, and one-shot continuity hint for the next reply after automatic session rollover.

## Best Possible Solution

Revise the feature so bounded, clearly delimited scheduled-rollover context reaches both immediate and queued prompt paths, reserves the new session field, clears only after actual delivery, and keeps explicit `/new` and `/reset` as blank starts.

## Reproduction Assessment

Not applicable as a strict bug reproduction: current docs define daily/idle expiry as a new session. The PR defects are source-reproducible from the diff plus current prompt queue and SessionEntry projection code.

## Solution Assessment

No, not as written. The safer path is to put the hint in a prompt-envelope input used by both command and queued runs, delimit transcript-derived text as untrusted context, reserve the new SessionEntry key, and clear the field only after the prompt that includes it is actually delivered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reserve the new session field:** `src/config/sessions/types.ts:95`
  - body: Adding `lastAssistantMessageBeforeReset` to `SessionEntry` also needs the field in `SESSION_ENTRY_RESERVED_SLOT_KEY_LIST`. That list has a type-level assertion covering every SessionEntry key; without this update the plugin slot-key contract fails type checking and leaves the new core field available for plugin projection collisions.
  - confidence: 0.9
- **[P2] Keep reset hints in queued prompt paths:** `src/auto-reply/reply/body.ts:48`
  - body: `applySessionHints` prepends and clears the reset hint before prompt bodies are rebuilt, but queued/followup/collect/steer runs use `queuedBody` built from `effectiveBaseBody`, not the prefixed body. If the first post-reset message waits behind an active run, the hint can be cleared without ever reaching the model.
  - confidence: 0.88
- **[P2] Delimit prior assistant text before injecting it:** `src/auto-reply/reply/body.ts:47`
  - body: `resetHintMsg` comes from the prior transcript and can contain newlines or instruction-shaped text reflected from earlier turns. Embedding it directly in the wrapper makes prior output hard to distinguish from the surrounding hint instructions.
  - confidence: 0.84
- **[P3] Clear the in-memory hint without store context:** `src/auto-reply/reply/body.ts:49-50`
  - body: The hint fires whenever `params.sessionEntry` carries `lastAssistantMessageBeforeReset`, but clearing is gated on `sessionStore` and `sessionKey`. A caller that supplies only the entry can inject the same one-shot hint repeatedly.
  - confidence: 0.72
