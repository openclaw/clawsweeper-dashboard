## Summary

Keep open: current main does not implement this continuity feature, and the PR is not merge-ready because it changes scheduled-reset semantics, has source-reproducible session/prompt blockers, is conflicting, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a SessionEntry field and transcript tail reader to carry the last assistant message from a scheduled daily/idle reset into a one-shot next-turn prompt hint.

## Best Possible Solution

Revise the feature so scheduled-rollover context is explicitly approved, safely delimited, delivered through both immediate and queued prompt paths, and cleared only after delivery with real reset proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because current docs and code define daily/idle reset as a session rollover. The PR blockers are source-reproducible from the current prompt queue path, SessionEntry slot-key contract, and PR diff.

## Solution Assessment

No, not as written. The safer path is to carry the hint through the shared prompt envelope used by immediate and queued runs, reserve the field, delimit transcript-derived text, and clear it only after actual delivery.

Full review comments:

- **[P2] Reserve the new session field:** `src/config/sessions/types.ts:95`
  - body: Adding `lastAssistantMessageBeforeReset` to `SessionEntry` also needs the key in `SESSION_ENTRY_RESERVED_SLOT_KEY_LIST`; that list has a type-level assertion covering every SessionEntry key, so the new core field is otherwise unreserved and can break the plugin slot-key contract.
  - confidence: 0.9
- **[P2] Carry reset hints into queued runs:** `src/auto-reply/reply/body.ts:47`
  - body: `applySessionHints` prepends and clears the reset hint before prompt bodies are rebuilt, but deferred followup/collect/steer paths send `queuedBody`, which is built from `effectiveBaseBody` rather than the prefixed body. If the first post-reset message is queued behind an active run, the hint is cleared without reaching the model.
  - confidence: 0.88
- **[P2] Delimit prior assistant text before injecting it:** `src/auto-reply/reply/body.ts:47`
  - body: `resetHintMsg` comes from the prior transcript and can contain newlines or instruction-shaped text reflected from earlier turns. Embedding it directly in the wrapper makes prior output hard to distinguish from the surrounding reset instructions.
  - confidence: 0.84
- **[P3] Clear the in-memory hint without store context:** `src/auto-reply/reply/body.ts:49`
  - body: The hint fires whenever `sessionEntry` carries `lastAssistantMessageBeforeReset`, but clearing is gated on `sessionStore` and `sessionKey`. A caller that supplies only the entry can inject the same one-shot hint repeatedly.
  - confidence: 0.72
