## Summary

Keep this PR open: current main does not implement the continuity hint, but the proposed patch still has source-reproducible correctness and prompt-safety blockers, lacks contributor real behavior proof, and changes scheduled-reset behavior enough to need maintainer product review.

## What This Changes

The PR adds a one-shot scheduled-reset continuity hint by storing the previous assistant transcript tail in SessionEntry and prepending it to the next reply.

## Best Possible Solution

Revise the feature so bounded, clearly delimited scheduled-rollover context reaches both immediate and queued prompt paths, reserves the new SessionEntry key, clears only after delivery, and keeps explicit resets as clean starts.

## Reproduction Assessment

Not applicable as an existing bug reproduction because current docs define daily and idle expiry as new sessions. The PR blockers are source-reproducible from the diff plus the current prompt queue and SessionEntry slot-key contract.

## Solution Assessment

No, not as written. The safer path is to carry the hint through the prompt envelope used by both command and queued runs, delimit transcript-derived text as untrusted context, reserve the new field, and clear it only after actual delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reserve the new session field:** `src/config/sessions/types.ts:95`
  - body: Adding `lastAssistantMessageBeforeReset` to `SessionEntry` also needs the key in `SESSION_ENTRY_RESERVED_SLOT_KEY_LIST`; that list has a type-level assertion covering every SessionEntry key, so the new core field is otherwise unreserved and can break the plugin slot-key contract.
  - confidence: 0.9
- **[P2] Carry reset hints into queued runs:** `src/auto-reply/reply/body.ts:47`
  - body: `applySessionHints` prepends and clears the reset hint before prompt bodies are rebuilt, but deferred followup/collect/steer paths send `queuedBody`, which is built from `effectiveBaseBody` rather than the prefixed body. If the first post-reset message is queued behind an active run, the hint is cleared without reaching the model.
  - confidence: 0.88
- **[P2] Delimit prior assistant text before injecting it:** `src/auto-reply/reply/body.ts:46`
  - body: `resetHintMsg` comes from the prior transcript and can contain newlines or instruction-shaped text reflected from earlier turns. Embedding it directly in the wrapper makes prior output hard to distinguish from the surrounding reset instructions.
  - confidence: 0.84
- **[P3] Clear the in-memory hint without store context:** `src/auto-reply/reply/body.ts:48-49`
  - body: The hint fires whenever `sessionEntry` carries `lastAssistantMessageBeforeReset`, but clearing is gated on `sessionStore` and `sessionKey`. A caller that supplies only the entry can inject the same one-shot hint repeatedly.
  - confidence: 0.72
