## Summary

Keep open: the PR is a focused session-continuity feature, but it is not merge-ready because the head changes reset semantics by default, lacks real scheduled/idle reset proof, and still has source-reproducible prompt delivery, SessionEntry contract, and prompt-trust blockers.

## What This Changes

The PR adds a `SessionEntry` field, reads the last assistant transcript message during scheduled daily/idle reset rollover, and injects that text once into the next reply prompt.

## Best Possible Solution

Revise this as an explicitly approved scheduled-rollover continuity design that preserves compatibility, reserves the session-store contract, safely delimits transcript-derived text, delivers through both immediate and queued prompt paths, and is proven in a real daily/idle reset setup.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a new continuity feature. The PR blockers are source-reproducible from the PR diff and current-main prompt queue, SessionEntry slot-key, and untrusted-context code paths.

## Solution Assessment

No, not as written. The safer path is to make the reset continuity semantics explicit, carry the hint through the shared prompt envelope used by queued and immediate runs, reserve the field, delimit transcript-derived text, and clear only after actual delivery.
