## Summary

Keep this PR open. Current main still lacks a scheduled daily/idle reset continuity hint, and the linked amnesia report makes the user problem still relevant, but the proposed patch needs maintainer follow-up because the hint can be cleared before queued prompts receive it and the carried transcript text needs safer delimiting.

## What This Changes

The PR adds a one-shot last-assistant-message continuity hint after daily/idle session rollover by storing a new session field, reading the prior transcript tail, and prepending/clearing the hint in reply prompt setup.

## Best Possible Solution

Keep this PR or a replacement open until OpenClaw has a rebased implementation that captures a bounded assistant-only tail during scheduled daily/idle rollover, treats it as untrusted prior output with clear delimiters, delivers it through both immediate and queued prompt paths, clears it only after actual delivery, preserves explicit /new and /reset as fresh starts, and adds targeted tests/docs for that behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: create a stale daily/idle session with a prior assistant transcript line, run the next inbound turn through initSessionState/get-reply prompt building, and current main creates a new session/archive without any assistant-tail hint field or prompt injection.

## Solution Assessment

No, not as written. The product direction is plausible, but the implementation should move the hint through the same prompt body that queued runs use, clear only after delivery, and delimit/sanitize the carried assistant text before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep reset hints in queued prompts before clearing:** `src/auto-reply/reply/body.ts:48-50`
  - body: `applySessionHints` adds the continuity hint only to `prefixedBody`, but queued modes build `queuedBody` from `effectiveBaseBody` instead. Since this block also clears the session field, followup/collect/steer queued runs can lose the scheduled-reset hint before the model ever sees it.
  - confidence: 0.89
- **[P2] Delimit prior assistant text before injecting it:** `src/auto-reply/reply/body.ts:47`
  - body: The previous assistant message is inserted directly into the wrapper text. If it contains newlines or instruction-shaped text, it can escape the intended quoted region and read like part of the new instruction; treat it as untrusted prior output with explicit delimiters or normalization.
  - confidence: 0.84
- **[P3] Clear the in-memory hint when persistence is unavailable:** `src/auto-reply/reply/body.ts:49-50`
  - body: The hint fires whenever `sessionEntry` has `lastAssistantMessageBeforeReset`, but it is only cleared inside the `sessionStore && sessionKey` persistence guard. If a caller provides a session entry without store context, repeated calls with the same object will keep prepending the hint.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, package, or release-script changes, but it introduces a prompt-injection concern by embedding prior transcript text into an instruction wrapper without strong delimiters.

Concerns:

- **[medium] Raw prior assistant text enters prompt instructions:** `src/auto-reply/reply/body.ts:47`
  - body: `resetHintMsg` comes from the previous transcript and is interpolated directly into the new prompt wrapper. Prior assistant output can reflect untrusted earlier user or web content, and multiline/instruction-shaped text can blur into the wrapper instruction unless it is delimited or normalized.
  - confidence: 0.84
