## Summary

Keep open. Current main still has the reported failure path, and an open unmerged PR already references this issue with closing syntax, so the issue should stay open until that PR or an equivalent focused fix lands.

## What This Changes

Report that Lobster workflows invoking the optional `llm-task` tool fail with an empty explicit-allowlist error when inherited tool allowlists are present and the embedded run intentionally disables tools.

## Best Possible Solution

Land #74139 or an equivalent focused fix that lets intentional no-tool `llm-task` embedded runs ignore inherited config allowlists while preserving fail-closed behavior for explicit runtime tool requests; coordinate the shared guard semantics with #73031 and #74810.

## Reproduction Assessment

Yes. The issue provides concrete config, Lobster, and `/tools/invoke` steps, and current main statically confirms the same path from `llm-task` through disabled tools, collected allowlists, and the empty-callable-tools guard.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix is to distinguish inherited/config allowlists from caller-supplied runtime tool constraints when tools are intentionally disabled, rather than removing the guard for ordinary tool-enabled runs.
