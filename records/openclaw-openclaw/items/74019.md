## Summary

Keep open: current main still has the reported false-positive empty-tool allowlist path, and #74139 is an open unmerged implementation candidate that explicitly closes this issue.

## What This Changes

The issue requests that `llm-task` JSON-only embedded runs avoid failing when inherited `tools.alsoAllow` or agent tool allowlists are non-empty while tools are intentionally disabled.

## Best Possible Solution

Review and land #74139 or an equivalent focused fix that distinguishes inherited config allowlists from runtime tool constraints, with regression coverage for `llm-task` plus inherited allowlist policy.

## Reproduction Assessment

Yes. The report gives concrete config, Lobster, and `/tools/invoke` steps, and current-main source inspection shows the same path from `llm-task` through disabled tools, inherited allowlist-source collection, and the empty-callable-tools guard.

## Solution Assessment

Yes, if the fix is source-aware. The narrow maintainable solution is to skip the empty allowlist guard for intentional no-tool runs caused by inherited config policy while preserving runtime `toolsAllow` fail-closed behavior.
