## Summary

Keep open. Current main and the latest release still match the reported failure path: `llm-task` intentionally disables tools, inherited allowlist sources are still collected, and the empty-allowlist guard still errors. An open PR already references this issue with closing syntax, so this should remain open until that PR or an equivalent narrowed fix lands.

## What This Changes

The issue asks for Lobster workflows invoking `llm-task` through `/tools/invoke` to allow JSON-only embedded model calls when inherited `tools.alsoAllow` includes `lobster` and `llm-task`.

## Best Possible Solution

Land #74139 or an equivalent focused fix that skips inherited/config empty-allowlist failures for intentional no-tool `llm-task` embedded runs while preserving runtime `toolsAllow` conflicts, then close this issue as implemented after the fix is merged and released or proven on main.

## Reproduction Assessment

Yes. The issue provides concrete config, Lobster workflow, and `/tools/invoke` steps, and current main statically confirms the same failing path without needing a live gateway run in this read-only review.

## Solution Assessment

Yes for the requested direction, but the implementation should be narrowed. The maintainable fix should distinguish inherited config allowlists from explicit runtime `toolsAllow` so JSON-only no-tool runs proceed without weakening caller-requested tool constraints.
