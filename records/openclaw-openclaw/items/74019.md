## Summary

Keep open: current main still has the false-positive empty-tool allowlist path, and #74139 is an open unmerged PR that explicitly closes this issue.

## What This Changes

The issue requests that Lobster-driven `llm-task` runs with inherited `tools.alsoAllow` entries proceed as intentional no-tool JSON model calls instead of failing the empty explicit allowlist guard.

## Best Possible Solution

Land #74139 or an equivalent focused fix that distinguishes inherited config allowlists from runtime tool constraints, with regression coverage for `llm-task` and generic no-tool runs.

## Reproduction Assessment

Yes. The report gives exact config and invocation steps, and source inspection on current main shows `llm-task` sets `disableTools`, the embedded runner empties the callable tool set, and the guard still errors when inherited allowlist sources remain.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the empty allowlist guard or embedded-runner policy path, while preserving the runtime `toolsAllow` fail-closed contract already covered by tests.
