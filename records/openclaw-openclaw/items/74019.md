Keep open: current main still has the reported false-positive empty-tool allowlist path, and #74139 is an open unmerged implementation PR with closing syntax for the same Lobster/llm-task bug.

## What This Changes

Allow Lobster workflows invoking llm-task to run JSON-only embedded model calls with tools disabled even when inherited tools.alsoAllow includes lobster/llm-task.

## Best Possible Solution

Review and land #74139 or an equivalent narrow fix that skips the empty allowlist guard only for intentional no-tool runs from inherited config, with regression coverage for llm-task/disableTools plus inherited tools.alsoAllow.

## Reproduction Assessment

Yes. The report gives concrete config, Lobster, and /tools/invoke steps, and static inspection of current main confirms the path from llm-task disableTools through inherited allowlist collection to the guard error.

## Solution Assessment

Yes for the requested direction, but current main is not fixed. The maintainable solution should distinguish inherited config allowlists from caller-supplied runtime tool constraints so runtime toolsAllow remains fail-closed.
