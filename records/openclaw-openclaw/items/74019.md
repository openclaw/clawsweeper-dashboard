## Summary

Keep open. Current `main` still has the reported false-positive empty-tool allowlist path, and #74139 is an open unmerged PR with closing syntax for the same Lobster/`llm-task` bug, so the issue should remain open until that fix or an equivalent one lands.

## What This Changes

The issue requests that Lobster workflows invoking `llm-task` continue to run JSON-only embedded model calls when inherited `tools.alsoAllow` entries are present and tools are intentionally disabled.

## Best Possible Solution

Land #74139 or an equivalent focused fix that lets intentional no-tool `llm-task` embedded runs ignore inherited config allowlists while preserving fail-closed behavior for explicit runtime tool requests, with a targeted regression test for inherited `tools.alsoAllow` plus `disableTools`.

## Reproduction Assessment

Yes. The issue provides concrete config, Lobster, and `/tools/invoke` steps, and current `main` statically confirms the same path from `llm-task` through disabled tools, inherited allowlist-source collection, and the empty-callable-tools guard.

## Solution Assessment

Yes for the requested direction, but not yet solved on current `main`. The safest fix is to distinguish inherited config allowlists from caller-supplied runtime tool constraints when tools are intentionally disabled, rather than removing the guard for ordinary tool-enabled runs.
