## Summary

Keep open. Current main still has the reported false-positive path: `llm-task` intentionally disables tools, the embedded runner still collects inherited tool allowlist sources, and the guard still errors when those sources resolve to an empty callable tool set. #74139 is an open, unmerged PR with closing syntax for this same bug, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for Lobster workflows that invoke `llm-task` to allow the tool's intentional no-tool embedded model run even when global or agent `tools.alsoAllow` entries include `lobster` and `llm-task`.

## Best Possible Solution

Land #74139 or an equivalent focused fix that treats inherited config allowlists as non-fatal for intentional no-tool embedded model runs, while keeping caller-supplied runtime tool constraints fail-closed and adding regression coverage for the reported `llm-task` plus `tools.alsoAllow` case.

## Reproduction Assessment

Yes. The issue provides concrete config, Lobster, and `/tools/invoke` steps, and current main statically confirms the same path from `llm-task` through `disableTools`, inherited allowlist-source collection, an empty tool set, and the empty-callable-tools guard.

## Solution Assessment

Unclear for the exact open PR until reviewed, but the requested direction is sound. The narrow maintainable solution is to distinguish inherited config allowlists from runtime tool requests when tools are intentionally disabled, rather than removing the guard for all disabled-tool cases.
