## Summary

Keep open: current main still contains the false-positive empty explicit tool allowlist path for Lobster-driven `llm-task` runs, and open PR #74139 explicitly closes this issue but has not merged.

## What This Changes

The issue reports that Lobster workflows invoking `llm-task` fail when inherited `tools.alsoAllow` entries meet an embedded no-tool model run.

## Best Possible Solution

Review and land #74139 or an equivalent focused fix with regression coverage for inherited config allowlists during `llm-task` and generic no-tool model runs.

## Reproduction Assessment

Yes. The report gives exact config and invocation steps, and static source inspection on current main shows `llm-task` disables tools while the embedded runner still evaluates inherited allowlists against an empty tool set.

## Solution Assessment

Yes with a caveat. The narrow fix belongs in the allowlist guard or embedded-runner policy path, but it should preserve runtime `toolsAllow` fail-closed behavior already covered by tests.
