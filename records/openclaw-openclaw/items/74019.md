## Summary

Keep open. Current main still has the reported false-positive empty-tool allowlist path, and PR #74139 is open, unmerged, and explicitly references this issue as its fix candidate.

## What This Changes

The issue asks for Lobster workflows invoking `llm-task` to run JSON-only embedded model calls with tools disabled without inherited `tools.alsoAllow` entries causing a “No callable tools remain” failure.

## Best Possible Solution

Finalize a focused fix that lets intentional no-tool embedded runs ignore inherited config allowlist entries while preserving fail-closed behavior for caller-supplied runtime tool constraints, with targeted guard coverage for `disableTools: true` plus inherited `tools.alsoAllow`.

## Reproduction Assessment

Yes. The issue provides concrete config, workflow, and `/tools/invoke` steps, and current main statically confirms the same path from `llm-task` through inherited allowlist collection, empty disabled-tool materialization, and the guard error.

## Solution Assessment

Yes for the requested direction, with one important boundary. The maintainable fix should distinguish inherited config allowlists from explicit runtime tool requests instead of broadly weakening the guard for ordinary tool-enabled or caller-constrained runs.
