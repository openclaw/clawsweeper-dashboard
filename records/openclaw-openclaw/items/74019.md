## Summary

Keep open: current main still has the reported false-positive empty-tool allowlist path, and #74139 is an open unmerged PR with closing syntax for this issue.

## What This Changes

The issue asks OpenClaw to let Lobster workflows invoke `llm-task` even when inherited `tools.alsoAllow` entries are present, because the embedded model run intentionally disables tools.

## Best Possible Solution

Review and land #74139 or an equivalent source-aware fix that distinguishes inherited config allowlists from runtime tool constraints, with regression coverage for `llm-task` and generic no-tool model runs.

## Reproduction Assessment

Yes. The report gives concrete config and Lobster `/tools/invoke` steps, the latest release has a confirming repro comment, and current-main source inspection shows the same path from `llm-task` through disabled tools, inherited allowlist collection, and the empty-callable-tools guard.

## Solution Assessment

Yes, if the fix preserves runtime policy. The narrow maintainable solution is to skip the empty allowlist guard for intentional no-tool runs caused by inherited config policy while keeping explicit runtime `toolsAllow` fail-closed.
