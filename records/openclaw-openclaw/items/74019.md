## Summary

Keep open. Current main still has the reported false-positive empty-tool allowlist path, and an open unmerged PR already references this issue with closing syntax, so the issue should stay open until that fix or an equivalent one lands.

## What This Changes

The issue asks for Lobster workflows invoking `llm-task` to run their JSON-only embedded model call without inherited `tools.alsoAllow` entries for `lobster` and `llm-task` triggering the empty-callable-tools guard.

## Best Possible Solution

Land #74139 or an equivalent focused fix that lets intentional no-tool `llm-task` embedded runs ignore inherited config allowlists while preserving fail-closed behavior for explicit runtime tool requests, with a targeted guard test for inherited `tools.alsoAllow` plus disabled tools.

## Reproduction Assessment

Yes. The issue provides concrete config, Lobster, and `/tools/invoke` steps, and current main statically confirms the same path from `llm-task` through disabled tools, inherited allowlist-source collection, and the empty-callable-tools guard.

## Solution Assessment

Yes for the requested direction, with one boundary to preserve. The safest fix is to distinguish inherited config allowlists from caller-supplied runtime tool constraints when tools are intentionally disabled, rather than removing the guard for ordinary tool-enabled runs.
