## Summary

Current main already solves the central failure mode: root-level edit oldText/newText arguments are normalized into the canonical edits array before schema validation, and that behavior is present in the latest release. The exact prompt-description example requested was not added, but the observed timeout loop from that wrong argument shape should no longer occur on the current implementation.

## What This Changes

The issue asks for an explicit example and warning in the edit tool description so reasoning MoE models avoid repeatedly sending root-level oldText/newText arguments instead of the required edits array.

## Best Possible Solution

Keep the released runtime normalization for this exact edit-argument shape, and use https://github.com/openclaw/openclaw/issues/86454 for the broader repeated failed tool-call loop guard if that still needs product work.

## Reproduction Assessment

No high-confidence live reproduction was run because the report depends on stochastic behavior from specific MoE models and provider routing. Source inspection does confirm the reported wrong root-level edit shape and shows current main normalizes it before validation.

## Solution Assessment

Yes, current main uses a stronger fix than prompt wording alone for the central failure mode: it accepts the observed malformed root-level oldText/newText shape and converts it to canonical edits before validation. A literal prompt example may still be an ergonomic choice, but it is no longer necessary to prevent this specific validation retry loop.
