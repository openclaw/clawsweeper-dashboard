## Summary

Keep open: current main still has the policy-refresh fast path that can trust an incomplete persisted `plugins[]` array when no `policyPluginIds` are supplied, and this PR is a focused store-level fix with regression coverage. The patch looks correct, but the external PR only supplies Vitest output, so real CLI or gateway behavior proof is still a merge gate.

## What This Changes

This PR adds a persisted-index guard and regression test so `policy-changed` refreshes rebuild source metadata when `installRecords` reference plugins missing from `plugins[]`.

## Best Possible Solution

Land the store-level invariant after real behavior proof and CI: keep `installRecords` as durable source state and treat `plugins[]` as rebuildable metadata when policy refresh sees an incomplete persisted view.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: persist matching `installRecords` with an empty `plugins[]`, then run `refreshPersistedInstalledPluginIndex` with `reason: "policy-changed"` and no `policyPluginIds`; current main takes the persisted fast path and leaves the plugin absent. I did not run it because this review was read-only.

## Solution Assessment

Yes, this is the best small fix shape: enforcing the invariant in `canRefreshPersistedPolicyState` covers every policy-changed caller, while a caller-only `policyPluginIds` fix would leave other no-target refreshes exposed.
