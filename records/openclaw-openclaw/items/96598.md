## Label Justifications

- `P1`: The issue describes a current Telegram channel workflow where one inbound message can produce duplicate user-visible assistant replies despite final-only configuration.
- `impact:message-loss`: The affected behavior is channel message delivery: an intermediate assistant message is delivered when the documented final-only mode should suppress it.

## Summary

Keep open: the docs promise final-only Telegram delivery for streaming.mode "off", and current source still has a plausible path that can emit block replies from an intermediate assistant tool-use turn before the terminal answer.

## What This Changes

The issue reports that Telegram with channels.telegram.streaming.mode = "off" sends assistant text from an intermediate tool-use turn and then sends the terminal final answer.

## Best Possible Solution

Make the final-only Telegram source-reply path suppress assistant text from non-terminal tool-use turns while preserving explicit message-tool sends, media, approvals, errors, and intentional preview/progress/block modes.

## Reproduction Assessment

Yes from source inspection and the reporter's session record, but not from a live Telegram run here. A tool-use assistant turn with text can reach text_end/message_end before the terminal stop turn, and the embedded block-reply path remains present even when Telegram preview streaming is off.

## Solution Assessment

No to a pure identical-message de-dupe as the primary fix; it would miss non-identical intermediate text. The cleaner fix is to gate non-terminal assistant text at the final-only source-delivery boundary and keep de-dupe only as a secondary safety net if needed.
