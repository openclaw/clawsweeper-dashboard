## Summary

Keep open. Current main still implements an append-only transparency note and preserves unsupported reminder/follow-up language in the delivered reply; existing tests assert that behavior. The current plugin hooks can revise or rewrite text, but they still do not expose the per-turn scheduling evidence needed to enforce the requested scheduling-backed choice outside core.

## What This Changes

The issue requests a pre-send gate that blocks unsupported future-commitment language unless the agent removes it or creates a concrete scheduled action.

## Best Possible Solution

Implement an opt-in pre-send finalization gate that reuses the existing commitment detector, successfulCronAdds, and same-session cron coverage, with tests proving unsupported commitment text is revised or backed by a real schedule before delivery. If maintainers choose plugin-first, first extend the finalization/message hook API to expose the scheduling evidence needed for a plugin to enforce the same contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction is present in the existing runReplyAgent test path: a reply payload of “I'll remind you tomorrow morning.” with successfulCronAdds set to 0 returns the same text plus the transparency note. I did not execute tests because this was a read-only review.

## Solution Assessment

No. The current append-only disclaimer is not the best solution for the requested trust contract because it still delivers the unsupported promise; the maintainable fix is a deliberate opt-in gate or an explicit public hook contract that can make the same scheduling-backed decision.
