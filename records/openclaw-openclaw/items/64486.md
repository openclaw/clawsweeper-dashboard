## Summary

Keep open. Current main still preserves unsupported reminder/follow-up language and appends the unscheduled-reminder note after the reply payload has already been built; the newer inferred-commitments feature is opt-in, post-reply background extraction and explicitly skips exact reminder/scheduling requests. The requested pre-send gate remains a product/API decision rather than something already implemented or safely closeable.

## What This Changes

Request an opt-in pre-send gate that forces unsupported future-commitment language to be removed or backed by a concrete scheduled action before delivery.

## Best Possible Solution

Implement a deliberate opt-in pre-send finalization gate that reuses the existing commitment detector plus successfulCronAdds and same-session cron coverage, with focused tests proving unsupported commitment text is revised or backed by a real schedule before delivery. If maintainers choose plugin-first, first extend the public finalization hook contract to expose the scheduling evidence needed for a plugin to make the same decision.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists in the current runReplyAgent test path: a mocked reply of "I'll remind you tomorrow morning." with successfulCronAdds set to 0 returns the same promise plus the transparency note. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No. The current append-only disclaimer is not the best solution for the requested trust contract because it still sends the unsupported promise; the narrow maintainable direction is an opt-in gate or an explicit hook/API contract that can make the same scheduling-backed decision before delivery.
