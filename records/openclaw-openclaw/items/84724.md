## Summary

Keep open: current main and the latest release still deny the requested self-disable path, and the remaining work is a small but security-sensitive expansion of isolated cron self-scope rather than an implemented or duplicate item.

## What This Changes

The issue requests a narrow built-in `cron` tool path that lets an isolated cron run disable its own current job without using `exec`.

## Best Possible Solution

Add a tightly guarded self-disable path for the active cron job only, with tests proving cross-job and multi-field updates remain denied.

## Reproduction Assessment

Yes. Source inspection shows a self-scoped `cron` tool call with `action: "update"`, the current `jobId`, and `patch: { enabled: false }` is denied before gateway dispatch; the existing test suite also asserts that denial.

## Solution Assessment

Yes, conditionally. The best shape is a dedicated self-disable action or a strictly constrained self-scoped update that only permits `enabled:false` for the active job, not broad `cron.update` access.
