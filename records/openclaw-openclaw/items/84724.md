## What This Changes

The issue requests allowing isolated cron runs to disable their own current cron job through the built-in cron tool without falling back to exec.

## Best Possible Solution

Add a tightly guarded self-disable path for the active cron job only, after maintainer/security approval, while preserving existing self-remove and read-only self-scope behavior.

## Reproduction Assessment

Yes. Source inspection on current origin/main shows the self-scoped cron gate rejects update, and the adjacent test explicitly locks in denial for `patch: { enabled: false }`.

## Solution Assessment

Yes, conditionally. The narrow maintainable solution is a dedicated self-disable action or a strictly constrained self-scoped update that only permits `enabled:false` for the active job, not broad `cron.update` access.
