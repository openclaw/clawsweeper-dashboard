Keep open: current main and the latest release still deny the requested self-disable path, no open or merged PR owns it, and the remaining work is a small but security-sensitive product decision.

## What This Changes

The issue requests a restricted built-in cron-tool operation that lets an isolated cron run disable its own current job without using exec.

## Best Possible Solution

Add a tightly guarded self-disable path for the active cron job only, after maintainer/security approval, while preserving existing self-remove and read-only self-scope behavior.

## Reproduction Assessment

Yes. Source inspection shows the self-scoped cron gate allows only status/list plus get/remove/runs for the current job, and an adjacent test explicitly expects `update` with `{ enabled: false }` to be denied.

## Solution Assessment

Yes, conditionally. The narrow maintainable shape is a dedicated self-disable action or strictly constrained self-scoped update that only permits `enabled:false` for the active job, not broad `cron.update` access.
