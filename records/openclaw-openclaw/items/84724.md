## Summary

Keep open: current main and v2026.6.9 still deny self-scoped `cron.update` with `{ enabled: false }`, and the supportive third-party comment confirms the gap affects reusable headless cron jobs. The requested fix is small, but it changes a restricted cron mutation boundary and needs maintainer product/security approval before implementation.

## What This Changes

The issue requests a restricted built-in cron tool path that lets an isolated cron run disable its own current job without using `exec`.

## Best Possible Solution

Approve and implement a minimal self-disable-only path for the current cron job, preserving all existing self-remove/read-only guards and adding denial tests for other jobs and non-`enabled:false` patches.

## Reproduction Assessment

Yes. Source inspection shows the current self-scope gate allows only `get`, `remove`, and `runs` for the matching job id, and the current test suite explicitly denies self-scoped `update` with `{ enabled:false }`.

## Solution Assessment

Yes, the proposed narrow self-disable path is the right layer because the Gateway update contract already supports `enabled:false` and the agent cron tool is the blocking authorization wrapper. Maintainers still need to choose whether this should be a dedicated `disable` action or a constrained `update` subset.
