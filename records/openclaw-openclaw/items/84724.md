## Summary

Keep open: current main and v2026.6.11 still deny scoped cron update/disable, while the issue discussion shows recurring-job workflows where remove and exec are poor workarounds. The fix shape is bounded, but it widens an isolated cron authorization boundary and needs maintainer/security approval first.

## What This Changes

Allow isolated cron runs to disable their own current cron job without falling back to exec, while preserving the job record and run history.

## Best Possible Solution

Add a narrowly guarded self-disable path for the active cron job only, with tests proving same-job `enabled:false` succeeds and all other scoped mutations remain denied.

## Reproduction Assessment

Yes. Source inspection on current origin/main shows the self-scoped cron gate rejects update, and the adjacent test explicitly denies `patch: { enabled: false }`.

## Solution Assessment

Yes, conditionally. The best fix is a dedicated self-disable action or tightly constrained scoped update for `enabled:false` on the active job, not broad `cron.update` access.
