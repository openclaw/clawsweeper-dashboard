## Label Justifications

- `P1`: The report describes a broken Control UI/Gateway chat workflow where a provider outage can make an active conversation disappear from the visible history.
- `impact:session-state`: The affected behavior is chat.history/session transcript state drifting from the active conversation after an agent-run failure.
- `impact:message-loss`: The user-visible assistant error response is not durably served back through chat.history, causing prior chat messages to disappear from the UI view.

## Summary

Keep open: current main still has the reported Gateway agent-run error path, and the shipped v2026.5.28 code has the same pattern. The issue is source-reproducible enough for a focused fix PR, but I did not run a live network-outage reproduction in this read-only review.

## What This Changes

The issue asks for Gateway chat.send failures from unreachable model providers to leave a consistent persisted assistant error in chat.history so Control UI does not lose the visible conversation.

## Best Possible Solution

Persist exactly one display-safe assistant error through the transcript parent-chain path when an agent run ends with Gateway-delivered error payloads and no runtime-owned assistant error was persisted, while keeping broadcastChatError for live UI notification.

## Reproduction Assessment

Yes, at source level: chat.send has a returnedAgentErrorPayloads path that broadcasts an error without its own assistant transcript append, and embedded-run failure paths can return isError payloads. I did not run a full live network-disconnect reproduction in this read-only review.

## Solution Assessment

Yes, the requested direction is the right fix shape if implemented through SessionManager-backed transcript append logic with idempotency. The fix should not patch dist files or add raw JSONL writes, and it should skip appending when the runtime already persisted an assistant error turn.
