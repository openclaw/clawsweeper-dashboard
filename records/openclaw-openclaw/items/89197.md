## Label Justifications

- `P1`: A provider/runtime outage can break an active Gateway/Control UI chat workflow by making refreshed visible history incomplete.
- `impact:session-state`: The issue is about durable transcript and `chat.history` state drifting from live run delivery after an agent-run failure.
- `impact:message-loss`: The assistant error turn is not durably returned through `chat.history`, so the visible failure response can disappear from the Control UI view.

## Summary

Keep open: current main and the latest release still broadcast started-run agent error payloads without persisting a durable assistant error turn, and the open closing PR is the right implementation path to review and land before this issue closes.

## What This Changes

The issue reports that a started Gateway agent run with no reachable model can leave `chat.history` without a persisted assistant error turn, causing the Control UI to show an incomplete refreshed conversation.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/89483 after maintainer approval, keeping the fix on the existing transcript append wrapper with sanitization and idempotency coverage.

## Reproduction Assessment

Yes, at source level: current main's returned-agent-error branch broadcasts the error without appending an assistant transcript entry. I did not execute the live network/provider outage path in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to persist exactly one sanitized assistant error through the existing transcript append wrapper when the runtime did not already persist one, rather than patching `dist/` or masking the state gap in the UI.
