## Summary

Keep open: current main and the latest release still leave first-party native Talk/Wake ASR on platform recognizers, while the existing media-audio and realtime-provider surfaces only cover adjacent uploaded-audio, browser/Gateway relay, Android relay, or plugin-provider paths. This remains a product and security-sensitive feature request, not an implemented, duplicate, stale, or ClawHub-only cleanup item.

## What This Changes

Request an opt-in custom ASR server/provider setting for voice input with URL, auth, model, and language options while preserving platform speech as the default.

## Best Possible Solution

Add a first-class native ASR provider/config boundary for the chosen Talk/Wake paths, keep platform recognition as the default, and reuse existing realtime-transcription plus provider-request security contracts rather than adding a one-off macOS-only HTTP preference.

## Reproduction Assessment

Not applicable; this is a feature request, not broken documented behavior. Source inspection gives high confidence that native Talk/Wake still use platform recognizers and no native custom ASR endpoint config exists.

## Solution Assessment

Unclear until maintainer product/security scoping. The maintainable direction is a shared native ASR provider/config seam that reuses existing provider security contracts, with ClawHub/provider plugins used only for engines that fit the existing relay/plugin surfaces.
