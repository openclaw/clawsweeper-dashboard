## Summary

Keep open. Current main still lacks a custom ASR provider for native voice input: macOS, iOS, and Android voice paths instantiate platform speech recognizers directly, while the existing OpenAI-compatible transcription support applies to inbound media, Voice Call, QQBot/STT, or standalone skill surfaces rather than the app voice wake/talk/push-to-talk settings requested here.

## What This Changes

Request to add an opt-in ASR provider setting so voice wake, talk mode, or push-to-talk can use a custom OpenAI-compatible HTTP transcription endpoint instead of platform speech recognition.

## Best Possible Solution

Define a native voice transcription provider boundary first, keep platform speech as the default provider, then add an OpenAI-compatible HTTP provider that reuses existing request/auth policy where practical and updates settings, docs, and tests across the selected macOS/iOS/Android surfaces.

## Reproduction Assessment

Not applicable as a regression, but there is a high-confidence static verification path: inspect current native voice code and settings to confirm only platform speech recognizers and locale/mic controls exist for app voice input.

## Solution Assessment

Unclear as proposed. A custom OpenAI-compatible HTTP provider is a reasonable implementation, but the maintainable path is a provider abstraction plus a scoped product/security decision rather than adding one-off endpoint fields directly inside each recognizer runtime.
