## Summary

Keep open: current main still hardwires native Talk and Voice Wake ASR to Apple/Android platform recognizers, while existing transcription/plugin surfaces cover inbound audio files or TTS, not configurable live device ASR. The request is valid, but it needs a new native provider/config design rather than conservative cleanup.

## What This Changes

Allow users to configure a custom OpenAI-compatible ASR endpoint with optional auth, model, and language instead of relying only on platform speech recognition.

## Best Possible Solution

Define a native ASR provider abstraction for Talk, Voice Wake, and push-to-talk, keep platform speech as the default, then add an OpenAI-compatible HTTP provider plus settings and docs for the agreed scope.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current main has no configurable ASR provider and starts platform recognizers directly in the native voice paths.

## Solution Assessment

Unclear until maintainers choose scope, but a one-off URL setting would not be the best implementation. The maintainable path is a provider seam first, then an HTTP ASR provider and settings wired through that seam.
