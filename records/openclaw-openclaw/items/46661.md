## Summary

Keep open. The request is valid remaining product work: current main has shared realtime/transcription provider seams, but native macOS Talk and the mobile Talk paths still use platform speech recognizers and the macOS Talk config surface only carries speech locale/TTS fields, not a custom ASR endpoint.

## What This Changes

The issue requests an opt-in custom ASR server setting for voice transcription, including URL, optional auth, model, language hint, test connection, and platform recognizer fallback.

## Best Possible Solution

Add a dedicated Talk transcription provider/config surface that native Talk clients can consume, keep platform speech as the default fallback, and reuse the existing provider/session seams where they fit instead of adding one-off endpoint handling in each client.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: native Talk still uses platform speech recognition, and the existing Gateway relay does not yet provide the requested first-party ASR settings surface.

## Solution Assessment

No as a small config patch; the maintainable path is a scoped provider/config boundary for Talk transcription with explicit fallback behavior. Existing realtime/transcription relay pieces should be reused where possible, but they do not by themselves solve the native preferences request.
