## Summary

Keep open: current main still lacks a custom ASR provider for native Talk, Voice Wake, and push-to-talk input; existing HTTP transcription support is limited to inbound media and Voice Call surfaces, so this remains a valid feature request needing maintainer scoping.

## What This Changes

Request to let users configure a custom OpenAI-compatible ASR HTTP endpoint, including URL, auth, model, and language settings, instead of relying only on native platform speech recognition for app voice input.

## Best Possible Solution

Add a native ASR provider abstraction for app voice capture, keep platform speech as the default provider, then add a guarded OpenAI-compatible HTTP provider with settings, docs, and tests for the selected surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that native app voice input still uses platform recognizers, while custom HTTP transcription exists only for media/Voice Call paths.

## Solution Assessment

Unclear until maintainers choose the covered surfaces and security policy. The maintainable direction is a provider seam first; directly wiring endpoint fields into each recognizer path would duplicate platform behavior and network policy.
