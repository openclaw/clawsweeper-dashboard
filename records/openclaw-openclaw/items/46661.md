## Summary

Keep open: current main and v2026.6.10 still route native macOS speech capture through Apple Speech, while the existing `tools.media.audio` and Talk transcription paths are separate Gateway/provider surfaces. https://github.com/openclaw/openclaw/issues/87140 is a useful narrower macOS Push-to-Talk follow-up, but it does not cover the full ASR settings, credential, fallback, and platform-scope decision requested here.

## What This Changes

Request to add an opt-in ASR/STT provider setting for native voice input so users can route speech capture to custom OpenAI-compatible or local endpoints instead of Apple Speech.

## Best Possible Solution

Design a maintainer-scoped native ASR/STT boundary that preserves Apple Speech as the default, reuses Gateway media-audio or Talk provider contracts where appropriate, and documents credential, network, fallback, and platform scope before implementation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and latest-release inspection give high confidence that native macOS voice input still uses Apple Speech and has no custom ASR endpoint selector.

## Solution Assessment

Unclear until maintainers choose the product and security scope. Reusing existing Gateway media-audio, Talk session, provider request, and plugin contracts is safer than adding a parallel native HTTP preference by default.
