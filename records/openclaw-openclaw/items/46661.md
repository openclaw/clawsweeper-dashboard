## Summary

Keep open: current main and the latest release still route native Talk and Voice Wake through platform speech recognizers, while existing media-understanding and realtime provider paths do not provide the requested native custom ASR endpoint setting.

## What This Changes

The issue requests an opt-in custom ASR endpoint setting for native voice input, with URL, optional auth, model, language hint, test connection, fallback behavior, and the current platform recognizer remaining the default.

## Best Possible Solution

Add a shared native ASR provider/config boundary only after maintainers choose the Talk/Wake/platform scope, keep platform recognition as the default, and reuse existing provider request, SecretRef, private-network, proxy, TLS, and redaction contracts.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection gives high confidence that native Talk/Wake still use platform recognizers and no native custom ASR endpoint setting exists.

## Solution Assessment

Unclear until maintainers choose product and security scope. A one-off macOS HTTP preference is not the best shape; the maintainable path is a shared native ASR provider boundary or an explicit decision to route native voice through plugin-backed Gateway relay.
