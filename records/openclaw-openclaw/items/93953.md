## Summary

Keep open: current `origin/main` and v2026.6.9 still route both `openclaw config` and `openclaw configure` into the shared configure wizard without a non-TTY guard, and open linked PRs already own the fix path.

## What This Changes

The issue asks for `openclaw config` and `openclaw configure` to fail early with a clear non-TTY error instead of entering the interactive wizard when stdin/stdout are not interactive.

## Best Possible Solution

Land one canonical shared configure guard that requires stdin and stdout TTYs, preserves normal interactive behavior, adds focused regression coverage, and closes this issue after the chosen PR merges.

## Reproduction Assessment

Yes from source inspection: both public aliases reach `configureCommandFromSectionsArg`, which launches the wizard without a non-TTY guard on current `origin/main`. I did not run the pipe command because this review is read-only and avoids commands that may create artifacts.

## Solution Assessment

Yes in direction: a shared stdin/stdout guard at the configure helper is the narrowest maintainable fix because both aliases and sectioned/full flows reach that helper. The exact implementation should be chosen by reviewing the open linked PRs rather than queueing another fix.
