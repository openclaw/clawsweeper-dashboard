## Summary

Keep open: current main and the latest release still route both `openclaw configure` and bare `openclaw config` into the interactive wizard without a TTY guard, and open closing PRs already own the fix path.

## What This Changes

The issue asks for `openclaw configure` and bare `openclaw config` to fail fast with a clear non-TTY error instead of starting the interactive wizard on piped input.

## Best Possible Solution

Review and land one existing linked PR that adds a shared stdin/stdout TTY guard before the configure wizard, then close this issue after the fix merges and is verified against both CLI aliases.

## Reproduction Assessment

Yes. Source inspection shows both public command aliases route to the same helper, and that helper starts the interactive wizard without a TTY guard; I did not run the pipe command in this read-only review.

## Solution Assessment

Yes. A shared guard before `runConfigureWizard` is the narrow maintainable fix because both aliases and sectioned/full flows reach the same helper; the remaining action is to pick and land one existing PR.
