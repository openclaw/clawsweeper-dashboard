## Summary

Keep open: current main and v2026.6.9 still route both `openclaw config` and `openclaw configure` into the interactive wizard without a non-TTY guard, and open linked PRs already own the fix path.

## What This Changes

The issue asks for `openclaw config` and `openclaw configure` to fail fast with a clear non-TTY error instead of entering the interactive configure wizard on piped stdin/stdout.

## Best Possible Solution

Land one canonical shared configure guard that requires stdin and stdout TTYs, preserves normal interactive behavior, and closes this issue after the chosen PR merges.

## Reproduction Assessment

Yes from source inspection, without running the pipe command because this review is read-only. Current main routes both public aliases into the same wizard helper with no TTY guard.

## Solution Assessment

Yes. A shared stdin/stdout guard at the configure helper is the narrowest maintainable fix because both aliases and sectioned/full flows reach that helper.
