## What This Changes

The issue asks for `openclaw config` and `openclaw configure` to fail closed with a clear non-TTY error instead of entering the interactive configure wizard.

## Best Possible Solution

Review and land one narrow linked fix PR that adds a shared TTY guard for both `configure` and no-subcommand `config`, with focused regression coverage and a clear non-interactive alternative message.

## Reproduction Assessment

Yes, from source inspection: both CLI paths call `configureCommandFromSectionsArg`, which currently calls `runConfigureWizard` before any TTY guard. I did not run the piped command because this review is read-only and avoids commands that may create runtime artifacts.

Yes, a shared guard at the configure wizard entry point appears to be the narrowest maintainable fix because both `openclaw configure` and no-subcommand `openclaw config` route there.
