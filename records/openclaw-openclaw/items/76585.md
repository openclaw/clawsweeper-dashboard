## Summary

Keep open. The report is fresh, severe, and adds narrower workspace-bloat evidence to the open #76562 regression family; current source confirms the 630s lane-timeout mechanism and some synchronous bootstrap reads, but the exact “arbitrary 234MB binaries in workspace” root cause is not fully proven against current code because bootstrap loading is already limited to recognized files with a 2 MiB open guard.

## What This Changes

The issue requests hardening cron/background runs against event-loop saturation from bloated agent workspaces by limiting, skipping, or yielding during bootstrap-context file loading and by reconsidering cron lane timeout behavior.

## Best Possible Solution

Use this issue or a maintainer-selected canonical issue to pin a minimal bloated-workspace cron repro, then move any blocking prep scan/read/hash work off the Gateway event loop or skip large/binary inputs before they enter cron prep.

## Reproduction Assessment

## Solution Assessment

Unclear. Increasing timeouts alone is not the best fix; the maintainable path is to identify the blocking prep stage and cap, skip, yield, or async offload that work while keeping existing timeout controls as a safety net.
