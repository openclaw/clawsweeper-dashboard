## Summary

Keep open. This is a fresh severe cron/event-loop regression report with useful workspace-bloat evidence; current source explains the 630s lane timeout and has several heavy prep surfaces, but the exact arbitrary-binary bootstrap root cause is not proven on current main and the issue carries distinct diagnostics beyond the broader #76562 report.

## What This Changes

Cron jobs on OpenClaw 2026.5.2 time out under event-loop saturation after large binary workspace files inflate agent preparation work.

## Best Possible Solution

Pin a minimal bloated-workspace cron repro here or in the canonical #76562 investigation, then harden the specific prep path so cron and health timers are not blocked by heavy workspace work.

## Reproduction Assessment

No. The report provides concrete field evidence and repro steps, but current-main source shows default bootstrap reads only named files with a 2MiB guard and lightweight cron can skip injection, so the exact failing path still needs focused live or unit reproduction.

## Solution Assessment

Unclear. Raising the 630s timeout is not the best primary fix; the maintainable solution is to identify the blocking prep stage and cap, skip, yield, or offload that work while keeping lane timeouts as a safety net.
