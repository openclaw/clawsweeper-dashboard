## Summary

Keep open: current main and v2026.6.10 still expose Slack files as placeholders plus media path/type facts, not as the requested structured file id, filename, MIME type, and size metadata. The fileId placeholder/download fixes partially address older related reports, but the remaining work is a shared channel/plugin attachment contract decision.

## What This Changes

The issue requests structured Slack inbound file attachment metadata, including file id, filename, MIME type, and size, in agent-delivered message context.

## Best Possible Solution

Define a shared inbound attachment metadata contract across channel/plugin SDK surfaces, then populate it from Slack while preserving existing placeholders, MediaPath/MediaPaths fields, and download-file behavior.

## Reproduction Assessment

Yes at source level: SlackFile carries id, name, mimetype, and size, but current Slack and shared inbound media normalization drops those fields before agent context. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

No for a Slack-only text or field patch; the maintainable path is a shared channel/plugin SDK attachment model that Slack can populate while preserving existing media fields and placeholders.
