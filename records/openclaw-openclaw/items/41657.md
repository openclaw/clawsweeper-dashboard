## Summary

Keep open: current main and v2026.5.20 expose Slack files as placeholders plus generic media path/type context, with fileId visible in the placeholder, but they still do not deliver durable agent-facing id/name/MIME/size attachment metadata. The remaining work is a public channel/plugin context contract and redaction-policy decision.

## What This Changes

Request Slack inbound file attachment metadata such as file id, name, MIME type, and size be delivered to agents instead of only text and generic media path context.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events carry id/name/mimetype/size, while current main and v2026.5.20 expose placeholders and generic media path/type fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only fields would not settle the broader attachment model.
