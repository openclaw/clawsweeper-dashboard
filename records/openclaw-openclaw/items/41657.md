## Summary

Keep open: current main and v2026.6.8 still deliver Slack files as placeholders plus generic media path/type facts, but they do not provide the requested structured file id, name, MIME type, and size metadata. The remaining work is a cross-channel/plugin SDK attachment contract decision, not a narrow Slack-only bug fix.

## What This Changes

Request structured Slack file attachment metadata in inbound agent delivery instead of only text placeholders and generic media context.

## Best Possible Solution

Define a shared inbound attachment metadata contract in the channel/plugin SDK, then map Slack, Discord, Telegram, and other media-capable channels into that shape while preserving the existing MediaPath/MediaPaths and placeholder behavior.

## Reproduction Assessment

Yes for a source-level reproduction: a Slack message file can contain id/name/mimetype/size, but current main normalizes inbound media to placeholder plus path/url/contentType/kind/messageId only. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No for a Slack-only field; the maintainable solution is a shared inbound attachment contract because Discord and Telegram have the same class of metadata. The current placeholder/download-file behavior should remain as compatibility while the structured model is added.
