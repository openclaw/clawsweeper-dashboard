## Summary

Keep open: current Slack inbound handling no longer strips files entirely, but it still delivers placeholders and media paths rather than the requested structured file metadata. The remaining work is an additive attachment contract/API decision, not stale cleanup.

## What This Changes

Request to deliver Slack inbound file metadata such as id, name, MIME type, and size to agents instead of only text, placeholders, and media paths.

## Best Possible Solution

Define an additive cross-channel inbound attachment metadata schema, map Slack file id/name/mimetype/size into it while preserving existing `MediaPath(s)` and placeholders, and add Slack plus shared contract coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current Slack delivery uses placeholders and media path fields, not a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the contract shape. A Slack-only text append would be narrow, but the existing discussion points toward a shared cross-channel `InboundAttachments`-style model as the more maintainable direction.
