## Summary

Keep open: current main still lacks an agent-facing structured per-file attachment metadata contract for Slack files; it only exposes Slack file names/file IDs in placeholders plus generic media path/url/type fields. The closed file-ID PR was superseded by this broader issue, and the open Slack file-access PR addresses media download reliability rather than the shared metadata shape.

## What This Changes

Request that Slack inbound messages deliver per-file attachment metadata such as ID, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model through the channel/SDK context, map Slack id/name/mimetype/size into it, and preserve current placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. Slack messages carry file metadata before delivery, but current main only exposes placeholders and generic media path/url/type fields to agents; I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the broader attachment model.
