## Summary

Keep open: current main preserves Slack file inputs and can download accessible files, but it still exposes them to agents as placeholders plus media fields rather than the requested file ID/name/MIME/size metadata contract. The related open Slack hydration PR is about file access reliability, not the structured metadata surface.

## What This Changes

Request to deliver Slack inbound file metadata such as file ID, name, MIME type, and byte size to agents.

## Best Possible Solution

Define an additive inbound attachment metadata contract, populate Slack file ID/name/MIME/size/download state into it, and preserve existing BodyForAgent placeholders plus MediaPath/MediaTypes compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Slack file events reach delivery, but the finalized agent context contains BodyForAgent placeholders and MediaPath/MediaTypes rather than a structured file metadata list.

## Solution Assessment

Unclear. A Slack-only placeholder expansion would be the smallest local change, but the issue discussion and current channel context types point to a shared additive metadata contract that needs maintainer API direction first.
