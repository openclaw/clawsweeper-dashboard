## Summary

Keep open: current main still does not deliver the requested Slack per-file metadata contract to agents; it only exposes file name/id in text placeholders and generic media path/url/type fields. The closed fileId PR was superseded by this issue, while the open Slack file-access PR addresses download reliability rather than the shared attachment metadata shape.

## What This Changes

Request that Slack inbound messages deliver file attachment metadata such as id, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model, map Slack id/name/mimetype/size into it, and preserve current placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. Slack messages carry file metadata before delivery, but current main only exposes placeholders and generic media path/url/type fields to agents.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the broader attachment model.
