## Summary

Keep open: current main has improved Slack file ingestion, placeholders, file IDs, and media path delivery, but it still does not expose a structured inbound attachment metadata object with Slack file id, name, MIME type, and size. The related open PR addresses media hydration/access reliability rather than the metadata contract requested here.

## What This Changes

Request a Slack inbound attachment metadata contract so agents receive file id, name, MIME type, and size when Slack messages include files.

## Best Possible Solution

Add a backward-compatible structured inbound attachment model in the channel/plugin SDK context, map Slack id/name/mimetype/size into it, and keep existing placeholders plus MediaPath/MediaPaths behavior for compatibility.

## Reproduction Assessment

Yes, by source inspection. A Slack message with files flows into placeholders and MediaPath/MediaPaths, while the finalized inbound context has no structured per-file attachment metadata field.

## Solution Assessment

Unclear until maintainers choose the cross-channel contract. A structured additive context field is likely cleaner than provider-specific inline text, while auto-download alone would not satisfy the metadata request.
