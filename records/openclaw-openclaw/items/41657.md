## Summary

Keep open: current main and v2026.5.20 still expose Slack files through placeholders plus generic media path/type fields, but they do not provide durable agent-facing id/name/MIME/size attachment metadata. The discussion and related PRs narrow this to a shared channel/plugin context contract and redaction-policy decision, not a cleanup close.

## What This Changes

Request Slack inbound file attachments expose agent-facing metadata such as file id, name, MIME type, and size instead of relying only on text placeholders and generic media fields.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events and OpenClaw's SlackFile type carry id/name/mimetype/size, while current main and v2026.5.20 expose placeholders and generic media path/type fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the public channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only fields would not settle the broader attachment model.
