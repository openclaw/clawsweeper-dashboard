## Summary

Keep open: current main and the latest release still expose Slack inbound files through placeholders plus generic media path/type fields, but not as durable agent-facing id/name/MIME/size metadata. The remaining work changes the public channel/plugin SDK context contract and needs a maintainer product/redaction decision rather than stale cleanup or an automated Slack-only patch.

## What This Changes

Request agents receive Slack file attachment metadata such as file ID, name, MIME type, and size when Slack inbound messages include files.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing BodyForAgent placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events carry id/name/mimetype/size, while current agent context exposes placeholders and generic media path/url/type fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the broader attachment model.
