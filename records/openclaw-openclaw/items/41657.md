## Summary

Keep open: current main and the latest release still expose Slack file attachments as placeholders plus media path/type fields, but not as the requested structured file id, filename, MIME type, and size metadata. The remaining work is a shared channel/plugin SDK attachment contract decision, not a narrow Slack-only bug fix.

## What This Changes

The issue requests Slack inbound attachment metadata so agents receive file id, filename, MIME type, and size instead of only text placeholders and media path fields.

## Best Possible Solution

Define one shared inbound attachment metadata contract across channel/plugin SDK surfaces, populate it from Slack, and preserve current BodyForAgent placeholders plus MediaPath/MediaPaths compatibility.

## Reproduction Assessment

Yes for source-level behavior: current main shows Slack file placeholders and media path/type fields, but no structured attachment metadata object with file id, filename, MIME type, and size. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No for a Slack-only metadata patch; the best path is a shared channel/plugin SDK attachment model that Slack can populate while preserving existing media fields and placeholders.
