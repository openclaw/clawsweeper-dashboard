## Summary

Keep this open: current main partially helps by putting Slack file names and file IDs into agent-visible placeholders and media path/type fields, but it still does not provide the requested structured attachment metadata block with file id, name, MIME type, and size. The prior discussion correctly narrows this from a Slack-only stripping bug into a cross-channel inbound attachment contract decision.

## What This Changes

Request structured Slack inbound file attachment metadata in agent delivery so agents can see file id, name, MIME type, and size without polling Slack separately.

## Best Possible Solution

Design a shared inbound attachment metadata contract in the channel/plugin SDK, populate it from Slack, and preserve existing BodyForAgent placeholders plus MediaPath/MediaPaths compatibility with cross-channel regression coverage.

## Reproduction Assessment

Yes for source-level behavior: current main shows Slack file placeholders and MediaPath/MediaTypes, but no structured attachment metadata fields for file id, name, MIME type, or size. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No for a Slack-only patch; the best path is a shared channel/plugin SDK attachment metadata model that Slack can populate while preserving current media fields and placeholders.
