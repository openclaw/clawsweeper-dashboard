## Summary

Keep open. Current main still has the requested Slack file fields at the event boundary, but agent delivery is limited to name/fileId placeholders plus legacy media path fields; it does not populate a structured inbound attachment metadata list with id, name, MIME type, and size.

## What This Changes

Request to expose Slack inbound file attachment metadata, including file ID, filename, MIME type, and size, to agents when Slack messages include files.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract, populate it from Slack message.files as the first implementation, preserve existing BodyForAgent placeholders and MediaPath/MediaPaths behavior, render the metadata as untrusted structured context for agents, and document/test the contract.

## Reproduction Assessment

Yes. A synthetic Slack message event with files containing id, name, mimetype, and size follows the current prepare path into placeholder text and MediaPath/MediaTypes only; current main has no structured metadata field that would carry the full file record to agents.

## Solution Assessment

Yes, the keep-open direction is the best path: add an additive generic metadata contract rather than encoding more provider-specific text into BodyForAgent or relying only on auto-download. The related #62792 media hydration work is adjacent but does not replace passive metadata delivery.
