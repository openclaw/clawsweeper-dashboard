## Summary

Keep this open: current main handles Slack files better than the original wording suggests, but it still does not expose the requested structured attachment metadata or MIME/size metadata to agents.

## What This Changes

Request that Slack inbound file messages deliver file metadata such as file ID, filename, MIME type, and size to agents.

## Best Possible Solution

Define an additive inbound attachment metadata contract, preferably shared across channels, then map Slack file id/name/mimetype/size into it while preserving current placeholders and MediaPath behavior.

## Reproduction Assessment

Yes. Source inspection shows a Slack file event with id/name/mimetype/size is reduced to BodyForAgent/RawBody placeholders plus MediaPath(s)/MediaType(s), with no structured attachment metadata field.

## Solution Assessment

Unclear. The request is valid, but maintainers should choose between a Slack-only placeholder expansion and the broader shared InboundAttachment-style model discussed on the issue.
