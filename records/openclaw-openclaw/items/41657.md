## Summary

Keep open: current main handles Slack file downloads and exposes Slack file IDs in placeholders, but it still does not deliver the requested per-file metadata contract with MIME type and size to agents. The closed fileId PR was explicitly superseded by this issue, and the open Slack file-access PR addresses download reliability rather than the shared attachment metadata shape.

## What This Changes

The issue requests that inbound Slack messages expose file attachment metadata such as file ID, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible cross-channel inbound attachment metadata model, map Slack id/name/mimetype/size into it, and preserve current placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. Slack messages carry file metadata before delivery, but current main only exposes placeholders and media path/type fields, with no structured attachment metadata or MIME type/size delivery to agents.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the broader attachment model.
