## Summary

Keep open: current main still does not deliver structured Slack attachment metadata to agents. Slack file IDs are now visible in placeholders and files can flow through the generic media path, but file name, MIME type, size, and provider file ID are not preserved as a durable attachment object.

## What This Changes

Request Slack inbound message delivery expose attachment metadata such as file ID, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK context, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events carry file metadata before delivery, while current agent context only exposes placeholders and generic media path/url/type fields; no live Slack workspace was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the broader attachment model.
