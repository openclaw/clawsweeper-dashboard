## Summary

Keep open: current main has improved Slack file ingestion, files.info URL refresh, placeholders, file IDs, and media path/type delivery, but it still does not expose the requested structured inbound attachment metadata with Slack file id, name, MIME type, and size. The related open Slack media PR is adjacent to file hydration/access reliability, not the channel/SDK metadata contract needed here.

## What This Changes

Request structured Slack file attachment metadata in inbound messages delivered to agents, instead of only text/media-path delivery.

## Best Possible Solution

Add a backward-compatible structured inbound attachment model in the channel/plugin SDK context, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus media-path behavior for compatibility.

## Reproduction Assessment

Yes, by source inspection. A Slack message with files flows through placeholder and MediaPath/MediaPaths handling, while the finalized inbound context has no structured per-file attachment metadata field.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive context field is likely the cleanest path, while inline Slack-only text or auto-download alone would not satisfy the metadata request.
