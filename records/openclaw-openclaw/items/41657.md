## Summary

Keep open: current main and latest release still expose Slack files through placeholders, media paths, and content types, but not the requested structured file id, filename, MIME type, and size metadata. The remaining work is a shared channel/plugin attachment-context contract, not a stale Slack-only bug fix.

## What This Changes

The issue requests that Slack inbound file messages deliver structured attachment metadata such as file id, filename, MIME type, and size to agents.

## Best Possible Solution

Define one shared inbound attachment metadata contract and have Slack populate it from existing file payload fields while preserving current placeholders and MediaPath/MediaPaths behavior.

## Reproduction Assessment

Yes, at source level: SlackFile carries id, name, mimetype, and size, but SlackMediaResult and the shared inbound media payload drop those fields before agent context. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

Not yet: a Slack-only text append or one-off MediaFileIds field would be too narrow. The maintainable solution is a shared channel/plugin attachment model that Slack can populate while preserving existing media fields.
