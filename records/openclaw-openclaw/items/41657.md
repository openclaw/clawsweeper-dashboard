## Summary

Keep open: current main already handles Slack files as media paths and filename/file-id placeholders, but the requested file metadata surface is still missing. The related open PR is about making Slack media hydration more reliable, not adding the structured id/name/MIME/size attachment metadata requested here.

## What This Changes

Request that Slack inbound file messages expose at least file id, name, MIME type, and size to agents instead of only text/media placeholders.

## Best Possible Solution

Add an additive, backward-compatible inbound attachment metadata surface that preserves existing `BodyForAgent` placeholders and `MediaPath` behavior while exposing id/name/MIME/size in a documented structured form.

## Reproduction Assessment

Not applicable as a feature/API gap rather than a documented bug. Source inspection on current main shows Slack consumes `message.files`, but agents only receive placeholders and media path/type fields, not structured file id/name/MIME/size metadata.

## Solution Assessment

Unclear until maintainers choose the public surface. The safest direction is an additive structured metadata model, possibly shared across channels, while retaining existing Slack placeholders and media paths for compatibility.
