## Summary

Keep open: current main and the latest shipped release still expose Slack file events as placeholders plus generic media path/type context, not durable id/name/MIME/size attachment metadata. The source path is clear, but the remaining work is a public channel/plugin context contract and redaction policy decision rather than a stale cleanup close.

## What This Changes

Requests Slack inbound file attachment id, name, MIME type, and size metadata be delivered to agents instead of only text/media placeholders.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, preserve existing BodyForAgent placeholders plus MediaPath/MediaType behavior, and document redaction rules for provider IDs, URLs, and local paths.

## Reproduction Assessment

Yes, by source inspection. Slack events and OpenClaw's SlackFile type carry the requested fields, while current main and v2026.5.20 only expose placeholders and generic MediaPath/MediaType-style facts to the agent context.

## Solution Assessment

Unclear until maintainers choose the public channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only plumbing would leave the broader attachment model unresolved.
