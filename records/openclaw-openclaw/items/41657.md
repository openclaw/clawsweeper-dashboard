## Summary

Keep open: current main and v2026.6.10 still expose Slack attachments through visible file placeholders plus media path/content-type fields, but not as the requested structured file id, filename, MIME type, and size metadata. The remaining work is a shared channel/plugin SDK attachment contract decision rather than a narrow Slack-only bug fix.

## What This Changes

The issue requests agent-visible Slack inbound file attachment metadata, including file id, filename, MIME type, and size.

## Best Possible Solution

Define one shared inbound attachment metadata contract across channel/plugin SDK surfaces, then populate Slack file id, name, MIME/content type, size, and local/download facts while preserving existing placeholders and media path behavior.

## Reproduction Assessment

Yes for source-level behavior: current main accepts Slack file id, name, MIME type, and size at the event boundary, but the delivered agent media/context shapes do not carry a structured attachment metadata object. No live Slack workspace reproduction was run in this read-only review.

## Solution Assessment

No for a Slack-only field patch; the maintainable solution is a shared channel/plugin SDK attachment model that Slack can populate while preserving existing media fields and placeholders.
