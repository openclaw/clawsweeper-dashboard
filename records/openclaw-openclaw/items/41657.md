## Summary

Keep open: current main no longer strips Slack files entirely, but it still does not provide the requested structured or minimum file metadata to agents; the related open PR improves file hydration rather than deciding the metadata contract.

## What This Changes

Add Slack inbound attachment metadata such as file ID, name, MIME type, and size to the agent-delivered message context.

## Best Possible Solution

Define an additive inbound attachment metadata contract, map Slack file ID/name/MIME/size/download state into it, and preserve existing placeholders plus `MediaPath`/`MediaTypes` compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Slack `files` are read, but current agent delivery only exposes name/fileId placeholders plus media path/type arrays, with no structured attachment metadata or size field.

## Solution Assessment

Unclear. A Slack-only inline placeholder expansion would be narrow, but the existing comment and SDK shape point toward an additive shared attachment metadata contract, which needs maintainer product/API choice first.
