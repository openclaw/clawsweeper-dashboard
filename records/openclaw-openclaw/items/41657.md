## Summary

Keep open: current main and v2026.6.10 still expose Slack files as placeholders plus media path/type facts, not as the requested structured file id, filename, MIME type, and size metadata. The remaining work is a shared channel/plugin SDK attachment contract decision, so this should stay with maintainers rather than close as implemented or queue an automated Slack-only fix.

## What This Changes

The issue requests structured Slack inbound file attachment metadata, including file ID, filename, MIME type, and size, to be delivered to agents.

## Best Possible Solution

Define one shared inbound attachment metadata contract across channel/plugin SDK surfaces, then populate it from Slack while preserving existing placeholders and media path fields.

## Reproduction Assessment

Yes for source-level behavior: current main carries Slack raw file metadata only until normalization, then delivers placeholders plus path/type media facts without a structured attachment object. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

No for a Slack-only body-text or fileId-only patch; the maintainable solution is a shared channel/plugin SDK attachment model that Slack, Discord, and Telegram can populate consistently.
