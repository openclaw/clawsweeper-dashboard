## Summary

Keep open: current main still lacks a durable structured attachment metadata field for Slack inbound files. The shipped behavior exposes Slack files through placeholders plus generic media path/type fields, so the remaining work is an agent-facing channel/plugin SDK contract decision rather than stale cleanup.

## What This Changes

Request that Slack inbound file attachments deliver metadata such as file ID, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events carry id/name/mimetype/size, while current agent context exposes only placeholders and generic media path/url/type fields.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the broader attachment model.
