## Summary

Keep open: current main and latest release still expose Slack files as placeholders plus generic media path/type context, but they do not deliver durable id/name/MIME/size attachment metadata to agents. The remaining work is a public cross-channel inbound attachment contract and redaction-policy decision, so this is not a conservative cleanup close.

## What This Changes

Request to include Slack file attachment metadata such as file id, name, MIME type, and size in inbound agent message delivery.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model, map Slack id/name/MIME/size into it, and preserve the existing placeholder plus MediaPath/MediaPaths behavior.

## Reproduction Assessment

Yes, by source inspection: Slack file events and OpenClaw's SlackFile model carry the requested fields, while current main maps inbound media to generic path/url/type fields and placeholder text only. I did not run a live Slack scenario in this read-only cleanup pass.

## Solution Assessment

Unclear until maintainers choose the public channel/plugin SDK contract and redaction rules. Directionally, an additive structured cross-channel field is more maintainable than Slack-only inline text or a fileId-only patch.
