## Summary

Keep open: current main has Slack file ingestion, fallback placeholders with name/fileId, files.info hydration, and MediaPath/MediaTypes delivery, but it still does not expose a structured inbound attachment metadata object containing Slack file id, name, MIME type, and size. The related open Slack media PR is about hydration/access reliability, not the metadata contract requested here.

## What This Changes

The issue requests that inbound Slack messages expose file attachment metadata such as file id, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible structured inbound attachment metadata model in the channel/plugin SDK context, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaPaths behavior for compatibility.

## Reproduction Assessment

Yes, by source inspection. Slack files flow into placeholders and media path/type context, while current main has no structured per-file attachment metadata field for agents.

## Solution Assessment

Unclear until maintainers choose the contract. A structured additive context field is likely cleaner than expanding provider-specific inline text, while media download reliability alone does not satisfy the metadata request.
