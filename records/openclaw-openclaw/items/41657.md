## Summary

Keep open: current main preserves and downloads Slack files, but agent delivery still exposes placeholders plus media path/type fields rather than the requested structured file ID/name/MIME/size metadata; #62792 is related hydration work, not the metadata contract.

## What This Changes

The issue asks Slack inbound delivery to include file attachment metadata such as file ID, file name, MIME type, and size when messages contain files.

## Best Possible Solution

Define an additive inbound attachment metadata contract, populate Slack file ID/name/MIME/size/download state into it, and preserve the existing placeholders plus MediaPath/MediaTypes compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Slack event with files reaches delivery, but the finalized agent context contains BodyForAgent placeholders and MediaPath/MediaTypes rather than a structured attachment list or size metadata.

## Solution Assessment

Unclear. Expanding the Slack placeholder would be the narrowest Slack-only change, but the issue discussion and shared channel types point toward an additive cross-channel metadata contract that needs maintainer API direction first.
