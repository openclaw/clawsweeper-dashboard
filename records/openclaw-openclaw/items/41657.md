## Summary

Keep open: current main still reads Slack file events and exposes placeholders plus media path/type fields, but it does not expose the requested structured file id, filename, MIME type, and size metadata to agents. The discussion correctly narrows this from a Slack-only stripping claim into a shared inbound attachment contract decision.

## What This Changes

Request that Slack inbound messages deliver file attachment metadata such as file id, name, MIME type, and size to agents instead of only text placeholders and media paths.

## Best Possible Solution

Define a shared inbound attachment metadata contract in the channel/plugin SDK, populate it from Slack, and preserve current BodyForAgent placeholders plus MediaPath/MediaPaths compatibility with cross-channel regression coverage.

## Reproduction Assessment

Yes for source-level behavior: current main shows Slack file placeholders and MediaPath/MediaTypes, but no structured attachment metadata fields for file id, name, MIME type, or size. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No for a Slack-only patch; the best path is a shared channel/plugin SDK attachment metadata model that Slack can populate while preserving current media fields and placeholders.
