## Summary

Keep open: current main still exposes Slack files as placeholders plus legacy media path/type context, but not as durable agent-facing id/name/MIME/size attachment metadata. The remaining work is a channel/plugin SDK contract and redaction decision, and the related Slack file-access PRs do not close this request.

## What This Changes

The issue requests agent-facing Slack file attachment metadata such as file ID, file name, MIME type, and size for inbound Slack messages.

## Best Possible Solution

Add an additive shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events carry file fields, but current main converts inbound files into placeholders and legacy media path/type fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the channel/plugin SDK contract and redaction rules. A backward-compatible cross-channel attachment field is the maintainable direction; Slack-only inline text or fileId-only fields would not settle the broader model.
