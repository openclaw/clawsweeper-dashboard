## Summary

Keep open: current main preserves Slack file arrays, downloads accessible files, and exposes fileId placeholders plus media fields, but it still does not deliver the requested structured file ID/name/MIME/size metadata to agents. The related open Slack hydration PR addresses file access reliability, not this metadata contract.

## What This Changes

Request structured Slack inbound file attachment metadata, at minimum file ID, file name, MIME type, and size, in agent-delivered messages or context.

## Best Possible Solution

Define an additive inbound attachment metadata contract, populate Slack file ID/name/MIME/size/download state into it, and preserve existing BodyForAgent placeholders plus MediaPath/MediaTypes compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Slack event with files reaches delivery, but the finalized agent context contains BodyForAgent placeholders and MediaPath/MediaTypes rather than a structured attachment list with name/MIME/size metadata.

## Solution Assessment

Unclear. Expanding the Slack placeholder is the narrowest Slack-only change, but the issue discussion and shared channel context types point toward an additive cross-channel metadata contract that needs maintainer API direction first.
