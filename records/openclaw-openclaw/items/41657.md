## Summary

Keep open: current main and v2026.5.20 still expose Slack attachments through placeholders plus generic media path/type context, but not as durable agent-facing id/name/MIME/size attachment metadata. The remaining work needs a public channel/plugin context contract and metadata redaction decision before implementation.

## What This Changes

Request Slack inbound file attachments to deliver structured metadata such as file id, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events and file info types carry id/name/MIME/size, while current main and v2026.5.20 expose placeholders and generic media path/type fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only fields would not settle the broader attachment model.
