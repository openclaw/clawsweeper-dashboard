## Summary

Keep open. Current main and the latest release no longer totally strip Slack files and do expose Slack file IDs in placeholders plus the `download-file` action, but they still do not provide the requested structured per-file metadata fields such as file name, MIME type, size, and provider file ID.

## What This Changes

The issue requests Slack inbound file metadata, at least file ID, name, MIME type, and size, to be delivered to agents instead of relying on placeholders or manual polling.

## Best Possible Solution

Design a shared inbound attachment metadata contract, keep existing `MediaPath`/`MediaPaths` and placeholder behavior for compatibility, then let Slack populate provider ID, file name, MIME type, size, download state, and media references through that contract.

## Reproduction Assessment

Yes for source-level reproduction: Slack receives `files[]` with the requested fields, but the delivered OpenClaw media context only carries path/URL/type facts and text placeholders. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

No for the Slack-only inline or partial-field approaches as the final fix. The cleaner solution is a shared cross-channel attachment metadata model because the issue discussion and sibling channel behavior point to the same contract gap.
