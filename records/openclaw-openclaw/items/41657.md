## Summary

Current main has substantially improved Slack file handling: agents receive downloaded media plus filename and Slack file ID placeholders. The remaining request is still valid because file size and a structured per-attachment metadata object are not delivered, and choosing that shared inbound contract requires maintainer product/API judgment.

**Likely related people**
- @vincentkoc — high confidence; recent contributor to the current Slack file-reference and media-delivery implementation.
- scoootscooob — medium confidence; carried the major Slack channel move into the bundled plugin boundary.

## What This Changes

Add structured agent-visible metadata for files attached to inbound Slack messages, including filename, MIME type, size, and Slack file ID.

## Best Possible Solution

Define a minimal typed inbound attachment descriptor at the shared channel-context boundary, populate Slack filename, MIME type, size, and file ID, retain existing MediaPaths and placeholders for compatibility, and add focused Slack plus shared-contract coverage with an explicit sibling-channel impact decision.

## Reproduction Assessment

Yes, at source level. Current main demonstrably supplies media paths/types and filename/fileId placeholders, while the shared context and Slack media result expose no structured attachment object or file size.

## Solution Assessment

No, adding more Slack-only text is not the best permanent solution. A narrow shared typed attachment descriptor is more maintainable, but maintainers must approve the public contract and sibling-channel scope first.
