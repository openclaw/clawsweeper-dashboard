## Summary

Keep open: current main and v2026.5.22 still carry Slack files as placeholders plus generic media path/type facts, not structured attachment metadata with id/name/MIME/size. The related open Slack file-access PR at https://github.com/openclaw/openclaw/pull/62792 targets media materialization, and the closed fileId-only PR at https://github.com/openclaw/openclaw/pull/64429 did not land the shared metadata contract this issue needs.

## What This Changes

The issue asks OpenClaw to deliver structured Slack file attachment metadata such as file id, name, MIME type, and size to agents for inbound Slack messages.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in channel/plugin context, map Slack id/name/MIME/size into it, preserve existing placeholder plus MediaPath/MediaType behavior, and redact private URLs unless maintainers approve exposing them.

## Reproduction Assessment

Yes, by source inspection. Slack events and OpenClaw's SlackFile model carry id/name/MIME/size, while current main and v2026.5.22 expose only placeholders and generic media fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the public channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only plumbing would not settle the broader attachment model.
