## Summary

Keep open: current main and v2026.5.20 still expose Slack file attachments as placeholder text plus generic media path/type context, but they do not deliver durable id/name/MIME/size attachment metadata to agents. The remaining work is a shared inbound context contract and redaction-policy decision, not a cleanup close.

## What This Changes

Request to deliver Slack file attachment metadata such as file id, filename, MIME type, and size to agents when Slack inbound messages include files.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in channel/plugin context, map Slack id/name/MIME/size into it, and preserve existing placeholder plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events and OpenClaw's SlackFile model carry id/name/MIME/size, while current main and v2026.5.20 expose only placeholders and generic media fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the public channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only plumbing would not settle the broader attachment model.
