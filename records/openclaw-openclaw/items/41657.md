## Summary

Keep open: current main and v2026.5.20 carry Slack files as placeholders plus generic media path/type context, but they still do not expose structured agent-facing file id/name/MIME/size metadata. The remaining work is an additive channel/plugin context contract and redaction-policy decision, not a cleanup close.

## What This Changes

Request that Slack inbound file attachments expose agent-readable metadata such as file id, name, MIME type, and size.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model through the channel/plugin SDK, map Slack id/name/mimetype/size into it, and preserve existing placeholders plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events and OpenClaw's SlackFile type carry id/name/mimetype/size, but the prepared inbound context currently exposes only text placeholders and generic media path/type fields.

## Solution Assessment

Unclear until maintainers choose the SDK contract and redaction policy. An additive shared attachment metadata field is the maintainable direction; Slack-only inline text or auto-download alone would not settle the cross-channel model.
