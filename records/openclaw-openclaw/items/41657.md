## Summary

Keep open: current main and v2026.6.8 still deliver Slack files as visible placeholders plus media path/type fields, but not as the requested structured file id, filename, MIME type, and size metadata. The remaining work is a shared channel/plugin SDK attachment contract decision, not a narrow Slack-only bug fix.

## What This Changes

The issue requests structured Slack inbound file metadata, including file ID, file name, MIME type, and size, in agent-delivered message context.

## Best Possible Solution

Define one shared inbound attachment metadata contract across channel/plugin SDK surfaces, then populate it from Slack while preserving existing placeholders and media path fields.

## Reproduction Assessment

Yes for source-level behavior: current main carries Slack raw file fields only until normalization, then delivers placeholders plus media path/type fields without a structured attachment metadata object. No live Slack workspace repro was run in this read-only review.

## Solution Assessment

No for a Slack-only text or field patch; the maintainable solution is a shared channel/plugin SDK attachment model that Slack can populate while preserving existing media fields and placeholders.
