## Summary

Keep open: current main downloads Slack files when possible and exposes file IDs in placeholders, but it still lacks the requested per-file metadata contract and does not surface Slack MIME type or size to agents.

## What This Changes

Request Slack inbound delivery to provide agents with file attachment metadata such as file ID, name, MIME type, and size.

## Best Possible Solution

Add a backward-compatible cross-channel inbound attachment metadata model, map Slack id/name/mimetype/size into it, and preserve current placeholders plus MediaPath/MediaTypes behavior.

## Reproduction Assessment

Yes, by source inspection. A Slack message with files reaches placeholder and media path/type delivery, but the finalized context has no structured per-file metadata field and the fallback text omits MIME type and size.

## Solution Assessment

Unclear until maintainers choose the channel/SDK contract. A structured additive context field looks cleaner than Slack-only inline text or auto-download alone, but that is a product/API decision.
