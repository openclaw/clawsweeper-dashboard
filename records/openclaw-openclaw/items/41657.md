## Summary

Keep open: current main and v2026.5.22 still expose Slack files as fileId-bearing placeholders plus generic media path/type fields, not durable structured attachment metadata. The related open Slack file-access PR targets media materialization, and the closed fileId-only PR was routed back to this issue, so the remaining API/redaction decision still belongs here.

## What This Changes

The issue requests that inbound Slack file attachments deliver structured metadata such as file id, name, MIME type, and size to agents.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata model in channel/plugin context, map Slack id/name/MIME/size into it, and preserve existing placeholder plus MediaPath/MediaType behavior.

## Reproduction Assessment

Yes, by source inspection. Slack events and OpenClaw's SlackFile model carry id/name/MIME/size, while current main and v2026.5.22 expose only placeholders and generic media fields without a structured attachment metadata object.

## Solution Assessment

Unclear until maintainers choose the public channel/plugin SDK contract and redaction rules. A structured additive cross-channel field is the maintainable direction; Slack-only inline text or fileId-only plumbing would not settle the broader attachment model.
