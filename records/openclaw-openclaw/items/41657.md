## Summary

Keep open: current main and v2026.6.10 still provide Slack file placeholders, media paths, and content types, but not the requested structured file id, filename, MIME type, and size metadata. The remaining work is a shared channel/plugin attachment-context contract, not a stale Slack-only bug fix.

## What This Changes

The issue requests that Slack inbound file attachments deliver structured metadata such as file ID, name, MIME type, and size to agents.

## Best Possible Solution

Define a shared inbound attachment metadata contract across channel/plugin SDK and agent context surfaces, then populate Slack from its existing file metadata while preserving current placeholders and MediaPath/MediaPaths behavior.

## Reproduction Assessment

Yes, at source level: SlackFile carries id, name, mimetype, and size, but current Slack and shared inbound media normalization drop those fields before agent context. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

No for a Slack-only text append or one-off MediaFileIds field; the maintainable fix is a shared channel/plugin SDK attachment model that Slack can populate while preserving existing media fields.
