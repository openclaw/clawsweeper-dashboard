## Summary

Keep open. The PR is a focused candidate for the QMD embed-interval issue, but the live head still marks blank or whitespace `embedInterval` values as configured, changes the documented lexical/BM25 contract without docs, and lacks real OpenClaw/QMD runtime proof.

## What This Changes

The PR adds an internal resolved QMD update flag and uses it so an explicitly configured `memory.qmd.update.embedInterval` can trigger periodic QMD embedding in lexical search mode.

## Best Possible Solution

If maintainers accept explicit lexical-mode embedding, land a version that normalizes configured-interval intent from the same trimmed/valid input path as interval parsing, updates the memory docs/tests for the new contract, and includes real QMD runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main resolves an embed interval but returns before it can affect scheduling or execution in lexical `search` mode. I did not run a live QMD end-to-end reproduction in this read-only review.

## Solution Assessment

No as submitted: carrying explicit interval intent through resolved config is the right boundary, but the live patch still treats empty and whitespace values as configured and leaves docs/proof incomplete for the behavior change.
