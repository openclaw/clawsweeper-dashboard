## Summary

Keep this PR open: it is the active candidate for the QMD lexical-mode embedInterval contract, but the current head still has a zero/invalid-duration override bug, leaves public docs inconsistent, lacks real QMD runtime proof, and needs memory/QMD owner direction before merge.

## What This Changes

The PR adds an internal resolved QMD `embedIntervalConfigured` flag and uses it to let explicit `memory.qmd.update.embedInterval` schedule and run `qmd embed` in lexical `search` mode, with focused tests.

## Best Possible Solution

Have memory/QMD owners decide the lexical-mode contract; if accepted, gate only on a successfully parsed positive explicit interval, update docs/help/tests, and require redacted real QMD runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main resolves `embedInterval`, but `shouldRunEmbed()` and `shouldScheduleEmbedTimer()` return in lexical `search` mode before the interval can affect embedding.

## Solution Assessment

No as submitted. Carrying explicit interval intent through resolved config is the right boundary if maintainers accept the behavior, but the flag must represent a valid positive parsed interval and docs/proof must be aligned.
