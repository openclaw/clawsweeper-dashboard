## Summary

Keep open: the PR addresses a real Telegram non-streaming duplicate-media bug and includes useful happy-path proof, but the patch still records block media after non-throwing skipped deliveries, which can remove the final fallback attachment.

## What This Changes

The PR adds Telegram non-streaming media deduplication by tracking block-delivered media URLs, filtering final reply payloads, adding helper tests, and updating the changelog.

## Best Possible Solution

Keep the Telegram-local dedup approach, but gate sentBlockMediaUrls updates on actual block delivery success and add a regression for non-throwing skipped block sends.

## Reproduction Assessment

Yes, source-reproducible. Current main has no block-to-final media dedup in the Telegram non-streaming dispatch path, and the linked bug report gives concrete Telegram steps, though I did not run a live current-main repro locally.

## Solution Assessment

No. The direction is right, but the implementation should only add URLs to sentBlockMediaUrls when the block delivery result was actually sent, not when it completed with `skipped` or `delivered:false`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Track block media only after a sent result:** `extensions/telegram/src/bot-message-dispatch.ts:1143`
  - body: `trackBlockMedia()` runs whenever the block callback completes, even when the send path returned `skipped`/`false`. `deliverLaneText` maps a false `sendPayload` to `skipped`, and `deliverReplies` can return `delivered:false` when a message_sending hook cancels, so this marks the URL as already sent and strips it from the final fallback even though no block media landed.
  - confidence: 0.86
