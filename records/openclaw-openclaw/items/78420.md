## Summary

Keep open: the PR targets a real Telegram non-streaming duplicate-media bug and includes real Telegram proof, but the latest head still records block media after non-throwing skipped/no-send deliveries, which can make the final fallback reply drop media that was never sent.

## What This Changes

The PR adds Telegram turn-scoped block-media URL tracking, a pure dedupe helper with tests, and a changelog entry so non-streaming final replies do not resend MEDIA attachments already delivered by block replies.

## Best Possible Solution

Keep the Telegram-local dedupe approach, but gate block-media tracking on an observed visible send result across text-lane, suppressed-reasoning, and direct payload branches, with regression coverage for no-send/skipped paths.

## Reproduction Assessment

Yes, source-reproducible. Current main directly sends non-streaming media block replies while final media suppression only uses blockReplyPipeline when streaming is enabled, and https://github.com/openclaw/openclaw/issues/78372 provides live Telegram evidence.

## Solution Assessment

No. The proposed Telegram-local dedupe is the right narrow direction, but this implementation must use actual delivery results before adding URLs to sentBlockMediaUrls so skipped block deliveries do not remove final fallback media.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate block media tracking on visible sends:** `extensions/telegram/src/bot-message-dispatch.ts:1134-1139`
  - body: `trackBlockMedia` has no signal that the block reply was actually delivered. It is called after delivery calls complete, but `sendPayload` can complete with `handled_no_send`, `delivered:false`, or `skipped`, so this can mark media as sent and make the final reply drop fallback media that never reached Telegram. Thread the delivery result through and only record media after a visible block send.
  - confidence: 0.88
