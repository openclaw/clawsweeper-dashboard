## Summary

Keep this PR open: it addresses a real Telegram non-streaming duplicate-media bug and has useful proof, but the patch has a blocking correctness issue because it marks block media as sent before the block delivery succeeds.

## What This Changes

The PR adds Telegram dispatch-side deduplication for final MEDIA attachments already delivered through non-streaming block replies, plus a helper, tests, and changelog entry.

## Best Possible Solution

Land the Telegram-specific dedupe after changing it to track only successfully delivered block media and adding a regression test for block send failure preserving final media fallback.

## Reproduction Assessment

Yes, source-level: current main direct-sends non-streaming media-bearing block replies and only uses blockReplyPipeline sent media for final dedupe when blockStreamingEnabled is true. The linked user report also gives concrete Telegram non-streaming reproduction steps and trajectory evidence.

## Solution Assessment

No, not as currently written: the fix direction is narrow and appropriate, but this implementation records block media before delivery succeeds. The safer solution is to mark URLs sent only after a successful block delivery, or remove them on block send failure, so final fallback can still carry media.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Track block media only after successful delivery:** `extensions/telegram/src/bot-message-dispatch.ts:1032`
  - body: The new set is populated before the block payload is actually sent. If that block media send throws, the dispatcher catches the block failure and continues to the later final delivery, but the final payload will be deduped against a media URL that was never delivered, so Telegram users can lose the attachment instead of getting the final fallback. Move the record after a successful block send or remove it on failure.
  - confidence: 0.86
