## Summary

Keep open: this targets a valid Telegram duplicate-delivery bug, but the current patch can still resend fully filtered captioned media through the legacy `mediaUrl` fallback and lacks the required changelog entry plus real Telegram proof.

## What This Changes

The branch adds Telegram turn-local tracking for block-delivered media, a pure dedupe helper, and unit tests to filter duplicate final-reply MEDIA attachments in non-streaming mode.

## Best Possible Solution

Ship a mediaUrl-aware dedupe fix with regression coverage, preferably at the shared reply-media finalization boundary or with Telegram integration coverage, plus a changelog entry and real non-streaming Telegram proof.

## Reproduction Assessment

Yes, source-level: non-streaming media block replies can be direct-sent while final media suppression only sees block-streaming pipeline media. The linked issue also has a user confirmation on v2026.5.5, but this review did not perform a live Telegram run.

## Solution Assessment

No. The direction is narrow, but this implementation must clear or realign legacy `mediaUrl`, add the changelog entry, and include real Telegram proof before it is the best maintainable fix.
