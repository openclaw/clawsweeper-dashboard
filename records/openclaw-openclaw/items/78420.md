## Summary

Keep open: the PR addresses the right Telegram non-streaming MEDIA duplicate path and includes real Telegram proof, but the latest diff still records block media after non-throwing skipped/no-send block delivery, which can strip the final fallback attachment.

## What This Changes

The branch adds Telegram-local tracking of block-delivered media URLs, filters duplicates from final reply payloads, adds a pure dedupe helper with tests, and updates the changelog.

## Best Possible Solution

Keep the Telegram-local dedupe approach, but record sentBlockMediaUrls only for block replies that actually report a visible send and cover non-throwing skipped/no-send results in regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main sends non-streaming media blocks directly while only streaming block-pipeline media contributes to final media suppression, and the linked issue has live Telegram confirmation; I did not run a live current-main bot.

## Solution Assessment

No. The proposed direction is narrow and maintainable, but this implementation still treats non-throwing skipped/no-send block deliveries as sent media; it should gate tracking on a true visible-delivery result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate media tracking on visible block sends:** `extensions/telegram/src/bot-message-dispatch.ts:1179`
  - body: `trackBlockMedia()` is called after `sendPayload(effectivePayload, ...)` without checking its boolean result, and the same helper is also called after text-lane delivery without checking for `skipped`. `sendPayload` can return false for `handled_no_send` or `deliverReplies({ delivered: false })`, so the final reply can drop media even though the block media was never sent.
