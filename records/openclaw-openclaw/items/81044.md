## Summary

Keep open for normal PR handling: the patch addresses a source-reproducible Feishu media cap bug, the updated PR body includes real Feishu live-output proof, and I found no blocking correctness or security findings.

## What This Changes

The PR forwards the configured Feishu inbound media byte cap into message-resource downloads and adds bounded-read regression coverage for over-limit Feishu SDK response shapes.

## Best Possible Solution

Merge the narrow Feishu plugin fix after required checks pass, keeping saveMediaBuffer as defense-in-depth and treating deeper SDK-level download cancellation as separate follow-up work if Feishu exposes a suitable API.

## Reproduction Assessment

Yes. Current main computes the Feishu media cap, but source inspection shows message-resource downloads fully materialize SDK responses before saveMediaBuffer enforces that cap; the PR body also reports before/after test and live over-limit evidence.

## Solution Assessment

Yes. Passing the existing cap into the Feishu download seam and enforcing it across the SDK’s Buffer/data, stream, async-iterable, and writeFile shapes is the narrow maintainable fix; full network-level cancellation for writeFile would require a different SDK capability.
