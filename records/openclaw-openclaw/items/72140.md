## Summary

Keep open. Current main hides stale pre-compaction totals by rendering unknown context when `totalTokensFresh` is false, but it still does not use the latest compaction checkpoint `tokensAfter` as the requested post-compaction context value; open PR #72285 remains the active implementation path.

## What This Changes

The issue requests that shared `/status` output show the current post-compaction context size instead of a stale pre-compaction total after manual or automatic compaction.

## Best Possible Solution

Land a narrow shared status formatter change that keeps fresh totals authoritative, selects the newest compaction checkpoint before validating `tokensAfter`, shows that value only when valid, leaves context unknown otherwise, and includes regression coverage plus the required changelog entry for the user-facing `/status` fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders `Context: ?/...` instead of the checkpoint post-compaction size.

## Solution Assessment

Yes in direction. The maintainable fix belongs in the shared status formatter using the existing freshness marker and checkpoint data, not in a WhatsApp-specific workaround or a broad session migration.
