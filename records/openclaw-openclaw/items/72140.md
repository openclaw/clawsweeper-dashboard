## Summary

Keep open. Current main avoids presenting a stale `totalTokens` value as current context, and newer compaction accounting can persist `tokensAfter`, but the shared status formatter still ignores compaction checkpoints and renders unknown when totals are stale; open PR #72285 remains the active unmerged implementation path for the remaining checkpoint-backed display.

## What This Changes

The issue requests that shared status output show the current post-compaction context size instead of stale pre-compaction totals.

## Best Possible Solution

Land or replace #72285 with a narrow shared formatter change that keeps fresh totals authoritative, selects the newest checkpoint by `createdAt`, uses finite non-negative `tokensAfter` only when totals are stale or unavailable, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main ignores that checkpoint and renders unknown context instead of the post-compaction size.

## Solution Assessment

Yes in direction. The remaining fix belongs in the shared status formatter using existing session freshness and checkpoint state, with the safer detail that newest-checkpoint selection should happen before validating optional `tokensAfter`.
