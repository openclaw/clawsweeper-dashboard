## Summary

Keep open. Current main prevents the worst stale pre-compaction display by rendering unknown context when totals are marked stale, but it still does not use the latest compaction checkpoint `tokensAfter` as the requested post-compaction context value; open PR #72285 remains the focused implementation path.

## What This Changes

The issue requests that shared `/status` output show the current post-compaction context size instead of a stale pre-compaction total after manual or automatic compaction.

## Best Possible Solution

Review and land PR #72285, or an equivalent narrow shared formatter change that keeps fresh totals authoritative and falls back to the latest valid compaction checkpoint `tokensAfter` only when current totals are stale or missing, with regression coverage for the post-compaction status display.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders `Context: ?/...` instead of the checkpoint post-compaction size.

## Solution Assessment

Yes. The best fix belongs in the shared status formatter using existing freshness and checkpoint data, not in a WhatsApp-specific workaround or broad session migration.
