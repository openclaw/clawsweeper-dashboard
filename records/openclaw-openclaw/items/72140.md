## Summary

Keep open. Current main avoids showing a known-stale total as current context, but it still renders unknown instead of falling back to compaction checkpoint `tokensAfter`, so the requested post-compaction context display remains unimplemented while #72285 is the active implementation path.

## What This Changes

Request that `/status` and shared status cards show the current post-compaction context size, with the pre-compaction peak kept as compaction metadata.

## Best Possible Solution

Land or replace #72285 with a narrow shared formatter change that keeps fresh totals authoritative, uses the newest checkpoint by `createdAt` only when its `tokensAfter` is finite and non-negative, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main ignores that checkpoint and renders unknown context.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in the shared status formatter using existing freshness and checkpoint state, not in channel-specific status rendering.
