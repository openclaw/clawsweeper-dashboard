## Summary

Keep open. Current main no longer displays the stale pre-compaction total when `totalTokensFresh` is false, but it still does not show the latest post-compaction `tokensAfter` value in the shared status formatter. Open PR #72285 is the paired implementation path, but it is not merged and has review feedback outstanding.

## What This Changes

The issue asks for `/status` and shared post-message status cards to display the current post-compaction context size, with pre-compaction peak data kept on the compaction line instead of the Context line.

## Best Possible Solution

Land PR #72285 after its narrow review fixes, or an equivalent shared formatter change that uses fresh totals when available, otherwise uses the latest valid compaction checkpoint `tokensAfter` for stale/missing totals, and adds regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context instead of the checkpoint post-compaction size.

## Solution Assessment

Yes, the shared formatter is the right place to solve this because `/status` and channel status cards use the same rendering path. The PR direction is correct, but final implementation should preserve latest-checkpoint semantics and avoid reusing older checkpoint totals.
