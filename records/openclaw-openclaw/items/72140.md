## Summary

Keep open. Current main suppresses stale pre-compaction totals when session totals are marked stale, but it still renders unknown context instead of the latest post-compaction checkpoint value; open PR #72285 remains the active unmerged implementation path.

## What This Changes

Issue requests that shared status output show the current post-compaction context size rather than stale pre-compaction totals after compaction.

## Best Possible Solution

Land a narrow shared status formatter change that keeps fresh totals authoritative, selects the newest compaction checkpoint by `createdAt`, uses a finite non-negative `tokensAfter` only when current totals are stale or unavailable, otherwise leaves context unknown, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main does not read that checkpoint and renders unknown context.

## Solution Assessment

Is this the best way to solve the issue? Yes. The fix belongs in the shared status formatter using the existing session freshness and checkpoint fields; a startup migration or channel-specific workaround would be broader than needed.
