## Summary

Keep open. Current main still prevents stale pre-compaction totals from being shown as current context, but it falls back to unknown context rather than using the latest compaction checkpoint `tokensAfter`; open PR #72285 remains the active implementation path and is not merged into main.

## What This Changes

The issue requests that shared `/status` and post-message status cards show the current post-compaction context size instead of stale pre-compaction totals after compaction.

## Best Possible Solution

Land a narrow shared status formatter fix that keeps fresh totals authoritative, selects the newest compaction checkpoint before validating `tokensAfter`, displays that post-compaction value only when valid, otherwise leaves context unknown, and includes focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context instead of that checkpoint-backed post-compaction size.

## Solution Assessment

Yes in direction. The fix belongs in the shared status formatter using existing freshness and checkpoint state, not in a WhatsApp-specific workaround or a broad compaction/session migration.
