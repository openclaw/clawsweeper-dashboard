## Summary

Keep open. Current main avoids showing a known-stale pre-compaction total as current context, but it still does not use stored compaction checkpoint `tokensAfter` values to show the requested post-compaction size, and open PR #72285 remains the active implementation path.

## What This Changes

The issue asks `/status` and shared status cards to display the current post-compaction context size instead of the pre-compaction total.

## Best Possible Solution

The shared status formatter should keep fresh totals authoritative, otherwise fall back to the newest valid compaction checkpoint `tokensAfter` and keep the compaction delta on the compactions line where appropriate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main ignores the checkpoint and renders unknown context instead of the requested post-compaction size.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in the shared status formatter using existing session freshness and checkpoint state, with focused regression coverage for stale totals plus checkpoint fallback.
