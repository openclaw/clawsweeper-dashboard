## Summary

Keep open. Current main suppresses the stale pre-compaction total by rendering unknown context when `totalTokensFresh` is false, but it still does not use the latest compaction checkpoint `tokensAfter` value for the shared status formatter. Open PR #72285 remains the narrow implementation path for the remaining behavior.

## What This Changes

The issue requests that shared `/status` and status-card output show the current post-compaction context size instead of a stale pre-compaction token total.

## Best Possible Solution

Review and land PR #72285, or an equivalent narrow shared formatter change that prefers fresh totals and falls back to the latest valid compaction checkpoint `tokensAfter` only when displayed totals are stale or missing, with regression coverage for post-compaction status display.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context instead of the checkpoint post-compaction size.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the shared status formatter using the existing freshness marker and checkpoint data, not in a WhatsApp-specific workaround or a broader session migration.
