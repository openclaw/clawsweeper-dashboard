## Summary

Keep open. Current main no longer displays the stale pre-compaction total as current context when `totalTokensFresh` is false, but it still renders unknown context instead of using the latest valid compaction checkpoint `tokensAfter`; open PR #72285 remains the active, unmerged implementation path for the remaining behavior.

## What This Changes

The issue requests that shared `/status` and status-card context output show the current post-compaction context size instead of stale pre-compaction token totals after manual or automatic compaction.

## Best Possible Solution

Land a narrow shared status formatter change that keeps fresh totals authoritative, selects the newest compaction checkpoint by `createdAt`, uses that checkpoint's finite non-negative `tokensAfter` only when current totals are stale or unavailable, otherwise leaves context unknown, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context rather than that checkpoint-backed post-compaction size.

## Solution Assessment

Yes in direction. The fix belongs in the shared status formatter using existing session freshness and checkpoint state, with the safer detail that the newest checkpoint should be selected before optional `tokensAfter` is validated.
