## Summary

Keep open. Current main avoids showing the stale pre-compaction total by rendering unknown context when `totalTokensFresh` is false, but it still does not use the latest compaction checkpoint `tokensAfter` as the requested post-compaction context value; open PR #72285 remains the active implementation path.

## What This Changes

The issue requests that `/status` and shared post-message status cards show the current post-compaction context size rather than stale pre-compaction totals after compaction.

## Best Possible Solution

Land a narrow shared status formatter change that keeps fresh totals authoritative, selects the newest compaction checkpoint before validating `tokensAfter`, shows that value only when valid, leaves context unknown otherwise, and includes regression coverage plus the required changelog entry for the user-facing `/status` fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context because the formatter has no checkpoint fallback.

## Solution Assessment

Yes in direction. The maintainable fix belongs in the shared status formatter using the existing freshness marker and checkpoint data, not in a WhatsApp-specific workaround or a broad session migration.
