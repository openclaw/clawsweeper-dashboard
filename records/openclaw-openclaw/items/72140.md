## Summary

Keep open. Current main avoids presenting known-stale totals as current context, but the shared status formatter still does not fall back to the latest compaction checkpoint `tokensAfter`, and open PR #72285 remains the active implementation path.

## What This Changes

The issue asks `/status` and shared post-message status cards to show the current post-compaction context size instead of stale pre-compaction totals.

## Best Possible Solution

Land or replace PR #72285 with a narrow shared formatter fix that prefers fresh totals, falls back to the newest valid checkpoint `tokensAfter` only when totals are stale or unavailable, and adds focused regression coverage plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context rather than the checkpoint post-compaction size.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The narrow maintainable solution belongs in the shared status formatter because `/status` and post-message status cards flow through `buildStatusMessage()`.
