## Summary

Current main and the shipped v2026.4.27 release now preserve Discord explicit target prefixes, so the reported `user:<id>` to bare numeric to `channel:<id>` failure chain no longer matches the implementation. The earlier discussion and linked PRs identified the right root cause, but the equivalent fix and regression coverage are already present on main.

## What This Changes

Fix Discord cron and related scheduled delivery so explicit DM targets such as `user:<id>` keep their prefix instead of being routed as Discord channels.

## Best Possible Solution

Keep the shipped/current normalized-target behavior and close this issue as fixed; maintainers can handle the now-redundant open PR separately if it still needs cleanup.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: parse `user:<id>` through Discord explicit target parsing, verify it stays `user:<id>`, then confirm cron/heartbeat delivery uses that parsed value before outbound send. I did not exercise live Discord API delivery in this read-only review.

## Solution Assessment

Yes. Preserving `target.normalized` is the narrowest maintainable fix because it keeps explicit DM targets prefixed while leaving the documented bare-numeric channel default intact.
