## Summary

Keep open. The linked bug is source-reproducible and current main/latest release still lack the exec marker filtering, but this PR is conflicting, lacks real behavior proof, and its broad `OPENCLAW_SERVICE_` prefix block would strip documented service repair controls.

## What This Changes

The PR updates the host exec environment security policy and macOS generated policy to strip `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_*` from inherited exec environments, plus adds a sanitizer regression test preserving `OPENCLAW_SHELL=exec`.

## Best Possible Solution

Land a rebased sanitizer fix that strips exact internal Gateway identity markers at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, regenerates parity artifacts, and proves clean CLI exit in a service-backed exec run.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, the current sanitizer leaves them intact, and credential resolution changes when `OPENCLAW_SERVICE_KIND=gateway`; I did not run live systemd/Gateway proof in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; exact runtime marker stripping or an explicit preservation rule for documented service controls is safer.
