## Summary

Keep open. The bug is source-reproducible and this PR is a plausible fix path, but the branch is currently conflicting, lacks real service-backed after-fix proof, and its broad `OPENCLAW_SERVICE_` prefix block would also strip the documented `OPENCLAW_SERVICE_REPAIR_POLICY=external` control.

## What This Changes

The branch updates host env sanitizer policy, generated Swift policy, and a regression test to strip `OPENCLAW_SYSTEMD_UNIT` and all `OPENCLAW_SERVICE_*` values from exec-spawned child processes.

## Best Possible Solution

Land a rebased sanitizer fix that blocks the exact internal runtime marker keys at the host exec boundary, preserves documented service controls and exec markers, and includes real service-backed proof of clean CLI exit.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, the sanitizer policy leaves them intact, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; block exact runtime marker keys or explicitly preserve documented service controls.
