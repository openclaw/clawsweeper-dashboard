## Summary

Keep open. The linked #61095 report gives a concrete exec-tool reproduction and current main still lacks service-marker filtering, but this PR should not merge as-is because the broad `OPENCLAW_SERVICE_` prefix also strips documented service-repair controls.

## What This Changes

The PR adds `OPENCLAW_SYSTEMD_UNIT` and the `OPENCLAW_SERVICE_` prefix to host exec environment filtering, regenerates the macOS policy, and adds a regression test for leaked gateway markers.

## Best Possible Solution

Land a rebased fix that filters exact internal gateway/node runtime markers in the shared policy and generated macOS policy while preserving documented operator controls and covering both behaviors in tests.

## Reproduction Assessment

Yes. The source-level path is high-confidence: gateway service startup exports the markers, agent exec builds child env from `process.env`, current main does not filter those markers, and credentials branch on `OPENCLAW_SERVICE_KIND`. The PR regression follows directly from `OPENCLAW_SERVICE_REPAIR_POLICY` sharing the proposed blocked prefix.

## Solution Assessment

No. The sanitizer boundary is the right place to fix the leak, but blocking the entire `OPENCLAW_SERVICE_` namespace is overbroad; exact marker keys or an explicit preservation rule is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the service repair policy:** `src/infra/host-env-security-policy.json:140`
  - body: Adding `OPENCLAW_SERVICE_` to `blockedPrefixes` also strips `OPENCLAW_SERVICE_REPAIR_POLICY=external`. Doctor documents and reads that key to skip service lifecycle repair when another supervisor owns it, so exec-spawned `openclaw doctor --repair` can silently fall back to `auto`. Block the specific runtime marker keys instead, or explicitly preserve the repair-policy key.
  - confidence: 0.91

## Security Review

Status: needs_attention
