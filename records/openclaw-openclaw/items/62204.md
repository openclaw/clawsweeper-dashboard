## Summary

Keep open: current main and the latest release still have the reported source path, but this PR's broad `OPENCLAW_SERVICE_` prefix block is overbroad and the external PR still lacks real service-backed exec proof.

## What This Changes

The PR adds `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_` filtering to the host exec environment policy, regenerates the macOS policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased sanitizer fix that blocks exact internal Gateway runtime markers at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, and includes real service-backed proof of clean CLI exit.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, the current sanitizer leaves them intact, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; block exact runtime marker keys or explicitly preserve documented service controls.
