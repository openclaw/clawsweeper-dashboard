## Summary

The PR targets a real source-proven agent exec bug, but it should stay open because the broad service-prefix filter can suppress documented service repair controls, the branch conflicts with the current host-env policy schema, and no after-fix real service-backed proof is present.

## What This Changes

The PR adds `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_` filtering to the host exec environment policy, regenerates the macOS policy file, and adds a sanitizer regression test preserving `OPENCLAW_SHELL=exec`.

## Best Possible Solution

Land a rebased sanitizer fix that strips exact internal Gateway runtime marker keys at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, updates the generated macOS policy, and proves clean CLI exit in a real service-backed exec run.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, the current sanitizer leaves them intact, and credential resolution changes when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; the safer fix is exact runtime marker stripping or an explicit preservation rule for documented service controls.
