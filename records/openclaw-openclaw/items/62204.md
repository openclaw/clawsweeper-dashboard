## Summary

Keep open. The underlying exec-env bug is still present on current main and the PR targets the right boundary, but the proposed broad prefix block would strip a documented service-repair control and the PR still lacks real service-backed after-fix proof.

## What This Changes

The PR adds host exec filtering for `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_*`, regenerates the macOS policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased sanitizer fix that blocks exact internal Gateway runtime markers at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, and includes real service-backed proof of clean CLI exit.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, current sanitizer policy leaves them intact, and credential resolution changes when `OPENCLAW_SERVICE_KIND=gateway`.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; use exact runtime marker keys or explicitly preserve documented service controls.
