## Summary

Keep open: the reported bug path is still present on current main and the PR targets the right sanitizer boundary, but the current diff should not merge because the broad service prefix strips a documented doctor repair guard and the PR lacks real service-backed exec proof.

## What This Changes

The branch adds `OPENCLAW_SYSTEMD_UNIT` to blocked keys, adds `OPENCLAW_SERVICE_` to blocked prefixes, regenerates the macOS generated policy, and adds a host env sanitizer regression test.

## Best Possible Solution

Land a rebased host exec sanitizer fix that strips exact internal Gateway runtime markers, preserves documented service controls and `OPENCLAW_SHELL=exec`, and proves clean CLI exit in a real service-backed exec session.

## Reproduction Assessment

Yes. Source inspection shows Gateway services set the reported markers, host exec inherits `process.env`, current main does not filter those markers, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; block exact runtime marker keys or explicitly preserve documented repair policy controls.
