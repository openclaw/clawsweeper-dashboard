## Summary

Keep open: this PR targets a valid source-reproducible env/auth bug, but the branch is conflicting, lacks real behavior proof, and its broad `OPENCLAW_SERVICE_` block can remove the documented `OPENCLAW_SERVICE_REPAIR_POLICY=external` guard from exec children.

## What This Changes

The PR adds host exec env filtering for `OPENCLAW_SYSTEMD_UNIT` and all `OPENCLAW_SERVICE_*` variables, regenerates the macOS policy artifact, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased sanitizer fix that strips exact internal Gateway runtime marker keys at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, updates generated parity artifacts, and proves clean CLI exit in a real service-backed exec run.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, current sanitizer leaves those markers intact, and credential resolution changes when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; the safer fix is exact runtime marker stripping or an explicit preservation rule for documented service controls.
