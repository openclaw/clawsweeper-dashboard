## Summary

Keep open. The underlying bug is credible and current main still has the source-level leak path, but this PR's denylist is too broad and it still lacks real after-fix proof that an exec-spawned CLI command succeeds and exits cleanly.

## What This Changes

The PR adds host exec env filtering for `OPENCLAW_SYSTEMD_UNIT` and all `OPENCLAW_SERVICE_*` variables, regenerates the macOS policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased host exec sanitizer fix that strips exact internal gateway/runtime service markers, preserves `OPENCLAW_SHELL=exec` and documented service repair controls, and proves both marker stripping and clean CLI exit.

## Reproduction Assessment

Yes. Source inspection shows gateway services set the reported markers, host exec inherits `process.env`, the current sanitizer policy does not filter those markers, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is not the narrowest maintainable fix because it removes documented service-repair policy from child processes.
