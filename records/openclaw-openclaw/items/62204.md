## Summary

Keep open. The PR targets a real current-main bug and its linked open issue, but the broad `OPENCLAW_SERVICE_` prefix blocks a documented repair-policy env var, the branch is conflicting, and the external contributor has not supplied real service-backed after-fix proof.

## What This Changes

The branch adds host env sanitizer policy entries and tests to strip `OPENCLAW_SYSTEMD_UNIT` and all `OPENCLAW_SERVICE_*` variables from exec-spawned child processes while preserving `OPENCLAW_SHELL=exec`.

## Best Possible Solution

Land a rebased sanitizer fix that strips exact Gateway runtime markers at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, and includes real service-backed proof of clean CLI exit.

## Reproduction Assessment

Yes. Source inspection shows Gateway services set the reported markers, host exec inherits `process.env`, the current policy does not filter those markers, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; block exact runtime marker keys or explicitly preserve documented repair policy controls.
