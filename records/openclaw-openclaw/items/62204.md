## Summary

Keep open. The PR targets a valid source-reproducible exec/auth bug, but the branch is now conflicting, lacks real service-backed proof, and its broad `OPENCLAW_SERVICE_` filter can suppress documented service lifecycle controls.

## What This Changes

The PR adds `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_` filtering to the host exec environment policy, updates the generated macOS policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased sanitizer fix that strips exact internal Gateway runtime marker keys at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, updates generated parity artifacts, and proves clean CLI exit in a real service-backed exec run.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, agent host exec inherits `process.env`, the current sanitizer leaves those markers intact, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; the safer fix is exact runtime marker stripping or an explicit preservation rule for documented service controls.
