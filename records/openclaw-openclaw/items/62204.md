## Summary

Keep this PR open, but do not merge it as-is. The linked bug still appears valid on current main, while the proposed prefix block would also remove a documented service-repair control that `openclaw doctor` relies on under externally managed supervisors.

## What This Changes

The PR updates host exec environment sanitization to strip gateway service runtime markers, regenerates the macOS Swift policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased fix that strips only internal gateway/node runtime markers from exec-spawned host environments, preserves documented operator controls such as OPENCLAW_SERVICE_REPAIR_POLICY, keeps the macOS generated policy in parity, and adds regression proof for marker stripping plus the reported CLI command completing cleanly.

## Reproduction Assessment

Yes. The linked bug has a high-confidence source-level reproduction path: gateway service startup exports the markers, exec builds from process.env, current main does not strip them, and credential resolution branches on OPENCLAW_SERVICE_KIND=gateway. The PR regression is also reproducible from the policy match itself because OPENCLAW_SERVICE_REPAIR_POLICY shares the proposed blocked prefix.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole OPENCLAW_SERVICE_ prefix is overbroad; the safer fix is to block the specific internal runtime marker keys or explicitly preserve the documented repair-policy key.

## Review Findings
