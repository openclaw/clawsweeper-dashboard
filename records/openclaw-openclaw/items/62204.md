## Summary

Keep open: the underlying exec env leak is still present on current main, but this PR is not ready because the broad service-prefix filter strips a documented service-repair control and no real after-fix behavior proof has been posted.

## What This Changes

The branch adds host exec environment filtering for Gateway service marker variables, updates the generated macOS policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased host exec sanitizer fix that strips exact internal Gateway runtime markers, preserves documented service controls and `OPENCLAW_SHELL=exec`, and proves clean CLI exit in a real service-backed exec session.

## Reproduction Assessment

Yes. Source inspection shows Gateway services set the reported markers, host exec inherits `process.env`, the current sanitizer policy does not filter them, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; block exact runtime marker keys or explicitly preserve documented repair policy controls.
