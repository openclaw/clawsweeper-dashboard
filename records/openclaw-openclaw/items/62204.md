## Label Justifications

- `P2`: This is a real agent exec and CLI-to-gateway bug with a bounded env/auth surface, but it is not a core-runtime emergency.

## Summary

Keep open: the reported bug is source-reproducible, but this PR is not merge-ready because it is conflicting, lacks real service-backed proof, and its broad service-prefix filter can suppress a documented service repair guard.

## What This Changes

The PR adds host exec environment filtering for OpenClaw service markers, regenerates the macOS generated policy, and adds sanitizer regression coverage.

## Best Possible Solution

Land a rebased sanitizer fix that strips exact internal gateway runtime marker keys at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, updates generated macOS policy, and proves clean CLI exit in a real service-backed exec run.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, the current sanitizer leaves them intact, and credential resolution changes when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; the safer fix is exact runtime marker stripping or an explicit preservation rule for documented service controls.
