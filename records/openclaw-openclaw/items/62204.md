- `merge-risk: 🚨 availability`: Dropping the external service repair guard could let exec-spawned doctor commands rewrite, start, or restart gateway services owned by another supervisor.

## Summary

Keep open because the linked exec/CLI bug is still present on current main and the latest release, but this PR cannot merge as-is: it is conflicting, lacks real service-backed proof, and its broad service-prefix block would remove documented service controls.

## What This Changes

The PR adds host exec environment filtering for `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_`, regenerates the macOS host env policy, and adds a sanitizer regression test that preserves `OPENCLAW_SHELL=exec`.

## Best Possible Solution

Land a rebased sanitizer fix that blocks exact internal Gateway runtime marker keys at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, and includes real service-backed proof of clean CLI exit.

## Reproduction Assessment

Yes, source-level. Gateway service env sets the markers, host exec inherits `process.env`, the current sanitizer leaves them intact, and credential resolution changes when `OPENCLAW_SERVICE_KIND=gateway`; I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; the safer fix is exact runtime marker stripping or an explicit preservation rule for documented service controls.
