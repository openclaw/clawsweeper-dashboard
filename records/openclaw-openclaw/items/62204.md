## Summary

Keep this PR open. It targets a real current-main bug path, but the patch is not safe as written because the broad `OPENCLAW_SERVICE_` prefix would also strip documented service-repair policy controls, and the branch is stale against the current host-env policy schema.

## What This Changes

The PR adds `OPENCLAW_SYSTEMD_UNIT` and the `OPENCLAW_SERVICE_` prefix to host exec environment filtering, regenerates the macOS generated policy, and adds a regression test for stripping gateway service markers while preserving `OPENCLAW_SHELL=exec`.

## Best Possible Solution

Land a rebased fix that strips only the internal gateway/node runtime markers from exec-spawned host environments, preserves documented controls such as `OPENCLAW_SERVICE_REPAIR_POLICY`, keeps the generated macOS policy in parity, and adds regression proof for both marker stripping and preserved repair-policy behavior.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: gateway service startup exports the marker variables, host exec builds child env from `process.env`, current main does not filter those markers, and credential resolution branches on `OPENCLAW_SERVICE_KIND=gateway`. The PR regression is also reproducible from the policy match itself because `OPENCLAW_SERVICE_REPAIR_POLICY` shares the proposed blocked prefix.

## Solution Assessment

No. The host exec sanitizer is the right boundary, but blocking the whole `OPENCLAW_SERVICE_` prefix is overbroad; the safer approach is to block exact internal runtime marker keys or explicitly preserve documented operator controls.

## Review Findings
