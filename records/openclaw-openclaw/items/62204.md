## Summary

Keep open. The linked bug is still present on current main and the host exec sanitizer is the right boundary, but this PR's `OPENCLAW_SERVICE_` prefix is overbroad because it also strips documented service-repair controls from exec-spawned child processes.

## What This Changes

The PR adds `OPENCLAW_SYSTEMD_UNIT` and `OPENCLAW_SERVICE_` filtering to host exec env sanitization, regenerates the macOS policy, and adds a sanitizer regression test.

## Best Possible Solution

Land a rebased host exec sanitizer fix that strips exact internal gateway/runtime service markers, preserves documented service repair controls and `OPENCLAW_SHELL=exec`, and proves both marker stripping and clean CLI exit.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: gateway services set the markers, host exec inherits `process.env`, the current policy does not filter those markers, and credential resolution branches on `OPENCLAW_SERVICE_KIND=gateway`. I did not run a live systemd/Gateway repro in this read-only review.

## Solution Assessment

No. The host exec sanitizer is the right place to fix the leak, but blocking the whole `OPENCLAW_SERVICE_` prefix is not the narrowest safe solution because it removes documented service-repair policy from child processes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid blocking the service repair policy:** `src/infra/host-env-security-policy.json:140`
