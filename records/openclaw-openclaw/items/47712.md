## Summary

Keep open: current main still lacks the shared config key and cross-surface emoji filtering; the only emoji stripping I found is provider/channel-local, so the reported behavior remains unresolved.

## What This Changes

Request to add a `messages.tts.filterEmoji` setting so TTS input filters emoji before generating speech.

## Best Possible Solution

Add a shared TTS text-preparation helper controlled by a documented `messages.tts.filterEmoji` policy, apply it before synthesis across auto-TTS, the agent tool, and Gateway conversion, and keep visible reply text unchanged.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the strict `messages.tts` schema and public TTS type do not accept `filterEmoji`, and shared TTS call sites either strip markdown only or pass raw text to synthesis.

## Solution Assessment

Unclear until maintainers confirm the product default. A shared text-preparation helper is the narrowest maintainable direction; provider-local emoji stripping alone leaves other TTS surfaces inconsistent.
