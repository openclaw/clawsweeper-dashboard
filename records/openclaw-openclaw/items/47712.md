## Summary

Keep open. Current main still does not accept or honor `messages.tts.filterEmoji`; the shared TTS config, docs, generated schema, auto-TTS path, agent tool, and `tts.convert` path all omit it, while emoji stripping is only provider-local in Local CLI and Discord voice code.

## What This Changes

Add a `messages.tts.filterEmoji` option that filters emoji before TTS synthesis, defaulting to emoji filtering.

## Best Possible Solution

Add a shared TTS text-preparation helper controlled by `messages.tts.filterEmoji?: boolean`, apply it consistently before synthesis, update config schema/help/labels/generated schema/docs/changelog, and keep provider-local sanitizers aligned with the shared behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a requested config/runtime feature. The current-main source still gives high-confidence proof of the gap because the strict config contract and all shared synthesis entry points omit `filterEmoji`.

## Solution Assessment

Yes, the maintainable direction is a shared TTS text-preparation path rather than separate provider-local filters. The implementation should cover auto-TTS, the agent tool, and `tts.convert` so all user-facing TTS entry points behave consistently.
