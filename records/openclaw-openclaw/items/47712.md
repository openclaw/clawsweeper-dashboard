## Summary

Keep open. Current main still rejects the requested shared TTS config key and only has provider/channel-local emoji stripping, so the user-visible TTS behavior remains inconsistent.

## What This Changes

Add a `messages.tts.filterEmoji` option so TTS can filter emoji from spoken text before synthesis.

## Best Possible Solution

Add a shared TTS text-preparation helper controlled by a documented `messages.tts.filterEmoji?: boolean` default, apply it before synthesis across auto-TTS, the agent tool, Gateway conversion, and CLI/directive TTS paths, and keep visible reply text unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `messages.tts.filterEmoji` is outside the strict config schema and the shared TTS call sites either pass raw text or apply markdown-only preparation before synthesis.

## Solution Assessment

Yes. A shared config/runtime/docs/test change is the narrowest maintainable fix; provider-local emoji stripping alone leaves other providers and one-shot TTS surfaces inconsistent.
