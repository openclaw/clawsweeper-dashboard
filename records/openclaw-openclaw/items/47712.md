## Summary

Keep open. Current main still lacks the public `messages.tts.filterEmoji` config key and a shared TTS text-preparation path; the only emoji stripping I found is provider-specific and not controlled by `messages.tts`.

## What This Changes

Add a `messages.tts.filterEmoji` option so TTS can skip emoji before speech generation, with the requested default behavior of filtering emoji.

## Best Possible Solution

Add a documented shared TTS text-preparation helper controlled by `messages.tts.filterEmoji` after deciding the default, and apply it consistently to auto-TTS, the agent `tts` tool, and `tts.convert` without mutating visible chat/transcript text.

## Reproduction Assessment

Yes, by source inspection: current main has a strict `messages.tts` schema without `filterEmoji`, and the TTS synthesis entry points pass text without a shared emoji filter. I did not run live TTS because this review is read-only and source proof is sufficient for the gap.

## Solution Assessment

Unclear as requested because the shared helper boundary is maintainable, but the requested default-on behavior needs a maintainer product decision. The safer implementation would document explicit default semantics and preserve visible replies/transcripts unchanged.
