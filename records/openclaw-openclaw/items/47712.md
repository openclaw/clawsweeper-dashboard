## Summary

Keep open: current main still does not accept or honor `messages.tts.filterEmoji`. The gap is valid, but adding a public config key with default-on spoken-text filtering changes TTS policy, so maintainers should confirm the semantics before implementation.

## What This Changes

The issue requests a `messages.tts.filterEmoji` option so TTS skips emoji before speech synthesis.

## Best Possible Solution

Define a shared TTS text-preparation policy controlled by a documented config field, apply it consistently before synthesis across auto-TTS, the agent `tts` tool, and Gateway `tts.convert`, and keep visible reply text unchanged.

## Reproduction Assessment

Yes. Current-main source inspection is enough: the strict shared TTS config schema and public type do not accept `filterEmoji`, while shared synthesis call sites either strip markdown only or pass normalized/raw text to TTS.

## Solution Assessment

Unclear. A shared text-preparation helper is the narrowest maintainable direction, but the requested default-on behavior should not be chosen without maintainer product confirmation.
