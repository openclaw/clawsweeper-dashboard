## Summary

Keep open: current main still does not accept or honor `messages.tts.filterEmoji`; the shared config contract, docs, and main TTS call paths lack config-controlled emoji filtering, while existing emoji stripping is provider/channel-local only.

## What This Changes

Add a `messages.tts.filterEmoji` option, defaulting to filtering emoji before TTS synthesis.

## Best Possible Solution

Add a shared spoken-text preparation helper controlled by documented `messages.tts.filterEmoji` policy, apply it before synthesis across shared TTS surfaces, and leave visible reply text unchanged.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `filterEmoji` has no matches, the strict shared schema/type/docs omit it, and the main TTS paths either strip markdown only or pass raw text to synthesis.

## Solution Assessment

Unclear until maintainers confirm the product default. A shared text-preparation helper is the narrowest maintainable direction; provider-local emoji stripping leaves other TTS surfaces inconsistent.
