## Summary

Keep open. Current main still does not accept or honor `messages.tts.filterEmoji`; the shared TTS config contract is closed without that key, docs omit it, and the main synthesis paths pass unfiltered text except for provider-local sanitizers.

## What This Changes

Add support for a `messages.tts.filterEmoji` option that filters emoji before text-to-speech synthesis, defaulting to filtered speech per the request.

## Best Possible Solution

Add a shared TTS text-preparation helper controlled by `messages.tts.filterEmoji?: boolean`, document the default, apply it before provider synthesis in auto-TTS, the agent `tts` tool, and Gateway `tts.convert`, and align provider-local sanitizers behind that shared behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: `messages.tts.filterEmoji` is absent from the strict config schema and type, and the shared synthesis paths do not call a config-controlled emoji filter before `textToSpeech`.

## Solution Assessment

Yes. The best fix is a narrow shared config/runtime/docs/test change; provider-local sanitizers alone are not sufficient because they leave other providers and one-off TTS paths inconsistent.
