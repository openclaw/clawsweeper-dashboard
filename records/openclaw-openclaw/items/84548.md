## Summary

Keep open: current main still has a plausible source-level path for the reported downgrade, and no relevant fix after v2026.5.18 was found. The cron runner validates the resolved thinking level against the loaded catalog, while the shared thinking resolver treats catalog `reasoning: false` as off-only before the Google provider profile can advertise Gemini 3 Flash levels.

## What This Changes

The issue reports that isolated cron sessions in v2026.5.18 downgrade `thinkingDefault: low` to `off` for `google/gemini-3-flash-preview`, despite the same configuration working in v2026.4.23.

## Best Possible Solution

Make Google Gemini 3 thinking support authoritative at the provider profile/catalog boundary, then add regression coverage for isolated cron with `google/gemini-3-flash-preview` and `thinkingDefault: low` when catalog metadata would otherwise mark the row non-reasoning.

## Reproduction Assessment

Yes, at source level: a catalog row with `reasoning: false` makes `low` unsupported before Google’s provider profile is considered, and isolated cron then downgrades to `off`. I did not run the live cron path against Google credentials.

## Solution Assessment

Yes, a narrow fix at the provider-owned thinking/catalog boundary is the maintainable direction. The safer repair should avoid weakening explicit `reasoning:false` semantics for unrelated custom models.
