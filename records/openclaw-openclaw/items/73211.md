## Summary

Keep open. Current main still matches the report: docs advertise Mattermost preview streaming modes, but the Mattermost schema/types do not accept `channels.mattermost.streaming`, and the monitor still creates and updates the draft preview unconditionally; open PR #73252 is the direct closing implementation candidate and is still unmerged.

## What This Changes

Allow Mattermost draft-preview streaming to be controlled by the documented channel streaming configuration instead of being hardcoded on.

## Best Possible Solution

Land a Mattermost-local fix that accepts the existing canonical `streaming.mode` contract at channel and account scope, preserves today’s default, and gates draft callbacks plus preview finalization when mode is `off`.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure `channels.mattermost.streaming` as documented, then the strict Mattermost schema/type reject or omit that field while the inbound monitor still creates and updates the draft preview path. I did not run a live Mattermost API smoke in this read-only review.

## Solution Assessment

Yes. Using the existing shared `streaming.mode` contract is the narrowest maintainable fix; adding a Mattermost-only `draftPreview` boolean would create a second public model for the same behavior.
