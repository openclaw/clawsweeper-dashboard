## Summary

Keep this open for maintainer review: current main and the latest release still have a source-reproducible Telegram reply-chain path that can hydrate bot-authored media, but current tests intentionally expect that hydration when visibility allows it, so the remaining decision is the Telegram reply-media contract rather than a safe cleanup close.

## What This Changes

The issue requests that Telegram reply-media resolution skip bot-authored media so a user reply to a bot photo does not reattach the bot’s prior media as inbound user media.

## Best Possible Solution

Make the Telegram reply-chain media contract explicit; if bot-authored media should be skipped, land a narrow guard that keeps bot-authored reply text context and user-authored media hydration covered by tests, otherwise close this as intended behavior with the current test contract cited.

## Reproduction Assessment

Yes. Source inspection shows reply-chain nodes with media can call bot.api.getFile without checking whether node.sourceMessage.from?.id is ctx.me?.id, and current tests include a bot-authored generated-photo reply ancestor that hydrates when allowlisted.

## Solution Assessment

Unclear. A self-authored media skip is the narrow implementation, but current tests and prior review make the intended Telegram reply-chain media contract a maintainer decision before a fix should land.
