## Summary

Keep this open: current main and the latest release still fetch reply-chain media from bot-authored Telegram messages, while the only open linked PR is a conflicting fork PR against an older two-file implementation. The remaining maintainer decision is whether to change the current reply-chain test contract that still expects bot-authored generated media to be fetched.

## What This Changes

The issue asks Telegram reply-media handling to skip media fetches from bot-authored replied-to messages so the agent does not receive its own prior media as inbound user attachments.

## Best Possible Solution

Update the current reply-chain media resolver to skip media downloads for bot-authored source messages while preserving reply-chain text/context, then adjust the Telegram regression test and require real Telegram proof on the PR.

## Reproduction Assessment

Yes from source, but not by running a live repro: current main and `v2026.6.5` fetch reply-chain media solely from `resolveInboundMediaFileId` plus `hasInboundMedia`, and the adjacent test expects `getFileSpy` to fetch `generated-photo-1` from a bot-authored reply target.

## Solution Assessment

The narrow self-authored guard is the likely best code-level fix, but the current fork PR is not the best landing path as-is because it conflicts and targets removed code; maintainers should confirm the current test contract before a current-main PR lands.
