Keep open: current main implements Discord typing heartbeat and event-driven progress drafts, but it still does not post an unconditional speculative Discord placeholder within the inbound ACK window. The remaining behavior is a maintainer UX decision because it changes visible channel noise and cleanup/finalization expectations.

## What This Changes

Request Discord to add an early speculative "Thinking..." placeholder plus sustained typing and optional tool-progress updates for parity with Slack-style perceived-latency feedback.

## Best Possible Solution

Make a product decision on whether Discord progress mode should start the existing editable draft at inbound acceptance or reply start, then implement it in the existing draft lifecycle with final edit/delete cleanup and live Discord timing proof.

## Reproduction Assessment

Yes, at source level: current main's Discord progress draft starts only after progress/work events and the tests explicitly keep text-only accepted turns from starting a draft. A live Discord run would still be needed to measure the exact <=3s visible timing.

## Solution Assessment

No, not as the bundled ask is written: typing heartbeat and tool-progress drafts are already present, while the best remaining fix is a narrow decision to start the existing Discord draft earlier rather than build a parallel placeholder system.
