## Summary

Keep open. The PR addresses a real current-main restore failure path, but the proposed rollback is incomplete because retry bookkeeping in `resumedRuns` can still poison the next init cycle, and the branch is dirty against current main.

## What This Changes

The PR snapshots subagent registry state during durable restore, resets the restore latch on failures, adds restore-retry coverage, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Land a rebased narrow fix that rolls back all restore/resume bookkeeping touched during a failed durable restore, covers failure after at least one restored run is marked resumed, and adds the active-version changelog entry.

## Reproduction Assessment

Yes. A focused unit test can restore two active runs, make a later resume step throw after the first run is marked in `resumedRuns`, call `initSubagentRegistry()` again, and assert that the first run is retried instead of skipped by the guard.

## Solution Assessment

No. The PR targets the right function and failure path, but the rollback should include `resumedRuns` state and the patch still needs a changelog entry plus a current-main rebase.
