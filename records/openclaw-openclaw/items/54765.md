## Summary

Keep open: the PR targets a real subagent restore retry bug, but the current patch still misses resumed-run rollback and the external PR lacks real after-fix behavior proof before merge.

## What This Changes

The PR snapshots `subagentRuns` during subagent registry restore, resets the restore latch on failure, adds restore-retry coverage, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Merge a repaired version that rolls back both registry state and restore-time resume bookkeeping, adds focused `resumedRuns` retry coverage, and leaves release-note handling to maintainers.

## Reproduction Assessment

Yes. From source inspection, a focused test can restore multiple runs, throw after one restore/resume path marks a run in `resumedRuns`, call `initSubagentRegistry()` again, and assert that the earlier run is retried instead of skipped.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting the latch is aimed at the right function, but the durable fix also needs to snapshot or clear `resumedRuns` entries touched during the failed restore pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Roll back resumed-run state on restore failure:** `src/agents/subagent-registry.ts:818-823`
  - body: The catch block restores `subagentRuns` and resets `restoreAttempted`, but a failed restore pass can already have added earlier ids to `resumedRuns` before a later restore-path exception. On the next `initSubagentRegistry()` retry, `resumeSubagentRun()` skips those ids, so cleanup or wait recovery is not re-run. Snapshot or clear the touched resumed ids and cover a failure after one resumed run.
  - confidence: 0.86
