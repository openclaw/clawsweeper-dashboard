## Summary

Keep open: current main still has the durable restore failure mode, and this PR is a plausible fix direction, but the branch has a blocking rollback gap because failed restore attempts can also poison `resumedRuns`; it also lacks required real behavior proof.

## What This Changes

The PR snapshots subagent registry state during restore, resets the restore latch on failure, adds a restore-retry test, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after at least one resumed run, then require redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let one restore/resume path add a run id to `resumedRuns`, then throw later in restore; retrying init skips the stale id because `resumeSubagentRun()` returns when the id is already marked resumed.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to rollback or clear restore-time `resumedRuns` ids and cover that failure mode.

## Review Findings
