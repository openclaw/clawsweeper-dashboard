## Summary

Keep open: the PR targets a real restore-retry bug still present on main, but its rollback only restores `subagentRuns` and can leave `resumedRuns` poisoned after a failed restore pass; it also lacks contributor-owned real behavior proof.

## What This Changes

The PR snapshots `subagentRuns` during subagent registry restore, resets the restore latch after failure, and adds restore-retry plus related test cleanup changes.

## Best Possible Solution

Update the PR to roll back or clear restore-time `resumedRuns` state along with `subagentRuns`, add regression coverage for failure after one resumed run, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let the first resumed run add its id to `resumedRuns`, throw before the restore pass completes, then call `initSubagentRegistry()` again and the stale `resumedRuns` entry will make `resumeSubagentRun()` skip it.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting the latch is the right area, but the durable fix also needs to restore the resume bookkeeping that controls retry eligibility.
