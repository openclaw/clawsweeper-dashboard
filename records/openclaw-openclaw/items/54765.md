## Summary

Keep open: the PR targets a real subagent restore bug, but it is not merge-ready because the rollback only restores `subagentRuns` and the latch while leaving restore-time `resumedRuns` ids stale; the external PR also lacks real behavior proof and is currently merge-conflicting.

## What This Changes

The PR snapshots `subagentRuns` during durable subagent registry restore, resets the restore latch on failure, and adds/updates Vitest restore-retry and cleanup mocks.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add a regression case for failure after a run is marked resumed, rebase, and include redacted real restore-retry proof.

## Reproduction Assessment

Yes, at source level. A restore path that marks a run id in `resumedRuns` and then throws leaves that id gated off, so the next `initSubagentRegistry()` retry skips `resumeSubagentRun()` for it.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting the restore latch is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.
