Keep this PR open, but it is not merge-ready: the durable-restore bug is real, while the patch still misses restore-time `resumedRuns` rollback, lacks contributor real behavior proof, and is currently merge-conflicting.

## What This Changes

The branch snapshots subagent registry restore state, resets the restore latch on failure, adds a restore-retry test, and hardens related Vitest mocks and cleanup.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add a regression case for failure after a run is marked resumed, rebase, and include redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. A restore that marks a run id in `resumedRuns` and then throws leaves that id gated off, so the next `initSubagentRegistry()` retry skips `resumeSubagentRun()` for it.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting the restore latch is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.
