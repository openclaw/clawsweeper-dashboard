## Summary

Keep open: current main still has the durable restore retry bug, and this PR targets the right path, but the patch leaves restore-time `resumedRuns` bookkeeping behind and lacks required real behavior proof.

## What This Changes

The PR snapshots subagent registry state during durable restore, resets the restore latch after failure, and adds retry-focused subagent registry tests and test cleanup hardening.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after one run is marked resumed, then require redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let an earlier `resumeSubagentRun()` add its id to `resumedRuns`, throw during a later restore/resume step, then retry init; the stale id makes `resumeSubagentRun()` return early.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Roll back resumed-run state on restore failure:** `src/agents/subagent-registry.ts:818-822`
  - body: The catch block restores `subagentRuns` and resets the latch, but a failed restore can already have added earlier run IDs to `resumedRuns` before a later exception. The next init then skips those IDs in `resumeSubagentRun()`, so cleanup or wait recovery is not retried; snapshot or clear the touched ids and cover failure after one resumed run.
  - confidence: 0.88
