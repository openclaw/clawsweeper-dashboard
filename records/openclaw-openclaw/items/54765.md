## Summary

Keep this PR open: it targets a real, source-reproducible subagent restore bug, but the current patch still misses rollback of `resumedRuns` after a partially completed restore pass and the external PR has only test/format proof rather than real after-fix behavior proof.

## What This Changes

The PR snapshots `subagentRuns` during restore, resets the restore latch on catch, adds a restore-retry unit test, and adjusts registry test mocks/cleanup.

## Best Possible Solution

Merge a repaired version that rolls back both registry and resume bookkeeping on failed restore attempts, adds focused `resumedRuns` retry coverage, and leaves landing-time release-note handling to maintainers.

## Reproduction Assessment

Yes, from source inspection. A focused test can restore multiple runs, make a later restore/resume step throw after one id is marked in `resumedRuns`, call `initSubagentRegistry()` again, and assert that the earlier id is retried instead of skipped.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting the latch is aimed at the right function, but the narrow durable fix also needs to snapshot or clear `resumedRuns` touched during the failed restore pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Roll back resumed-run state on restore failure:** `src/agents/subagent-registry.ts:815-820`
  - body: The catch block restores `subagentRuns` and resets `restoreAttempted`, but a failed restore pass can already have added earlier ids to `resumedRuns` before a later restore-path exception. On the next `initSubagentRegistry()` retry, `resumeSubagentRun()` skips those ids, so cleanup or wait recovery is not re-run. Snapshot or clear the touched resumed ids and cover a failure after one resumed run.
  - confidence: 0.86
