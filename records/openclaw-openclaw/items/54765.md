## Summary

Keep open. The PR targets a real durable-restore bug, but the proposed rollback only restores `subagentRuns` and the restore latch; it can still leave restore-time `resumedRuns` entries behind, so a retry may skip cleanup/wait recovery for runs already touched before the transient failure.

## What This Changes

The PR snapshots `subagentRuns` during subagent registry restore, resets the restore latch on failure, and adds focused restore-retry plus related test cleanup changes.

## Best Possible Solution

Update the PR to roll back or clear restore-time `resumedRuns` state along with `subagentRuns`, add regression coverage for a failure after one resumed run, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence focused path: restore multiple runs, fail after one run has been marked in `resumedRuns`, call `initSubagentRegistry()` again, and assert the first run is retried instead of skipped.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting the latch is the right area, but the narrow durable fix must also restore the resume bookkeeping that controls retry eligibility.

## Review Findings

- **[P2] Roll back resumed-run state on restore failure:** `src/agents/subagent-registry.ts:818-823`
