## Summary

Keep open: the PR targets a real subagent restore retry bug, but the patch still leaves restore-time `resumedRuns` state behind after a partial restore failure and lacks required real behavior proof from a real setup.

## What This Changes

The PR snapshots `subagentRuns`, resets `restoreAttempted` after restore failures, adds a restore-retry test, and hardens related subagent registry test mocks and cleanup.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after one resumed run, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let the first `resumeSubagentRun()` add its id to `resumedRuns`, throw during a later restore/resume step, then retry init; the stale id makes `resumeSubagentRun()` return early.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to rollback or clear restore-time `resumedRuns` ids and cover that failure mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
