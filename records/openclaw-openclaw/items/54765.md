## Summary

Keep open: this targets a real current-main session restore bug, but the patch still misses `resumedRuns` rollback, lacks real behavior proof, and the live PR is conflicting.

## What This Changes

The PR snapshots `subagentRuns` during durable restore, resets `restoreAttempted` after a caught restore failure, and adds restore-retry tests plus test cleanup/mocks.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after one run is marked resumed, rebase, and require redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. A restore that marks a run id in `resumedRuns` and then throws can leave that id gated off, so the next `initSubagentRegistry()` retry skips `resumeSubagentRun()` for it.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.
