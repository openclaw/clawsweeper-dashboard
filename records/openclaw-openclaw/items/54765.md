## Summary

Keep open: the PR targets a real subagent registry restore bug, but the current patch is not mergeable because the rollback only restores `subagentRuns`, leaves restore-time `resumedRuns` bookkeeping poisoned, and still lacks contributor-owned real behavior proof.

## What This Changes

The PR snapshots `subagentRuns` around restore, resets `restoreAttempted` on failure, adds a restore-retry test, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Roll back both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after one resumed run, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let the first resumed run add its id to `resumedRuns`, throw before the restore pass completes, then call `initSubagentRegistry()` again and `resumeSubagentRun()` will skip that stale id.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the PR also needs to snapshot or clear restore-time `resumedRuns` ids and cover failure after one resumed run.
