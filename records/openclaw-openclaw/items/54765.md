## Summary

Keep open: this PR targets a real subagent durable-restore bug on current main, but the patch still misses `resumedRuns` rollback, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

This PR snapshots `subagentRuns` during durable restore, resets `restoreAttempted` after restore failure, and adds restore-retry tests plus shared test mocks and cleanup hardening.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add a regression case for failure after one run is marked resumed, rebase, and include redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. A restore that marks a run id in `resumedRuns` and then throws can leave that id gated off, so the next `initSubagentRegistry()` retry skips `resumeSubagentRun()` for it.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.
