## Summary

Keep open: the durable restore retry bug is still valid and not superseded, but this PR still leaves restore-time `resumedRuns` state stale after a failure and lacks real behavior proof.

## What This Changes

The PR snapshots subagent registry state during restore, resets the restore latch after failure, and adds retry-focused registry tests plus test cleanup hardening.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` on restore failure, cover failure after at least one resumed run, and merge only after redacted real restore-retry proof is supplied.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let an earlier `resumeSubagentRun()` add its id to `resumedRuns`, throw during a later restore/resume step, then retry init; the stale id makes `resumeSubagentRun()` return early.

## Solution Assessment

No, not as-is. Snapshotting `subagentRuns` and resetting the restore latch is the right direction, but the patch also needs to rollback restore-time `resumedRuns` ids and prove the retry path in a real setup.
