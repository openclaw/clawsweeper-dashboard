## Summary

Keep this PR open: it targets a real subagent restore bug, but the proposed rollback still leaves restore-time `resumedRuns` state behind, so a retry can skip recovery work for runs processed before the failure.

## What This Changes

The PR snapshots `subagentRuns`, resets the restore latch after a failed restore, adds a restore-retry test, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after one resumed run, and then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple runs, let the first resumed run add its id to `resumedRuns`, throw before the restore pass completes, then call `initSubagentRegistry()` again and `resumeSubagentRun()` will skip that stale id.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the PR also needs to snapshot or clear restore-time `resumedRuns` ids and cover failure after one resumed run.
