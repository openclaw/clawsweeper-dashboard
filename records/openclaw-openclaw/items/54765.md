## Summary

Keep open: this PR targets a real subagent session-restore bug, but the current patch still leaves restore-time `resumedRuns` state stale after a later failure, lacks real behavior proof, and is merge-conflicting with current main.

## What This Changes

The PR snapshots subagent registry durable restore state, resets the restore latch after transient restore failure, and adds focused restore-retry tests plus test cleanup/mocks.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add a regression case for failure after a run is marked resumed, rebase, and include redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. A restore that adds a run id to `resumedRuns` and then throws leaves that id gated off, so the next `initSubagentRegistry()` retry skips `resumeSubagentRun()` for it.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.
