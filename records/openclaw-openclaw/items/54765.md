## Summary

Keep this PR open: the underlying durable subagent restore failure is still source-reproducible on current main, but the proposed rollback is incomplete because it does not restore `resumedRuns`, and the external PR still lacks real behavior proof.

## What This Changes

The PR snapshots `subagentRuns` during durable restore, resets `restoreAttempted` after a catch, and adds restore-retry tests plus registry test cleanup/mocks.

## Best Possible Solution

Rollback both `subagentRuns` and restore-time `resumedRuns` state on restore failure, add regression coverage for failure after one run is marked resumed, rebase the branch, and require redacted real restore-retry proof before merge.

## Reproduction Assessment

Yes, at source level. Restore multiple persisted runs, let an earlier resume path add a run id to `resumedRuns`, then throw during a later restore/resume step; the next init retry can skip that stale id.

## Solution Assessment

No. Snapshotting `subagentRuns` and resetting `restoreAttempted` is the right area, but the patch also needs to snapshot or clear restore-time `resumedRuns` ids and cover that failure mode.
