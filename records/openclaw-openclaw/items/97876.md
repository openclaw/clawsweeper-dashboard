## Summary

Keep open: the test idea is useful, but this patch is not merge-ready because the added test observes the SDK's final `globalThis.fetch` call, which would still happen if the guarded fetch wiring were removed.

## What This Changes

The PR adds `src/llm/providers/openai-completions.fetch-proof.test.ts`, a Vitest that stubs `globalThis.fetch` and exercises the real OpenAI Chat Completions SDK request path.

## Best Possible Solution

Revise the test so it asserts behavior only `buildGuardedModelFetch` can produce, such as SSRF blocking before the final global fetch stub is called.

## Reproduction Assessment

Yes. Source and dependency inspection show that without the custom fetch option `openai@6.39.1` uses ambient global fetch, while the guarded path rejects the link-local URL before that final fetch call; I did not execute the test in this read-only review.

## Solution Assessment

Yes. For the stated coverage gap, asserting SSRF blocking before `globalThis.fetch` is the narrowest maintainable proof that the SDK is using `buildGuardedModelFetch` rather than its default fetch.
