## Summary

This PR should stay open: it contains plausible CLI/post-delivery liveness work, but the current branch still has a source-level hook-isolation bug, adds host-specific proxy/runtime Compose overlays, and lacks real Telegram proof for the visible Telegram behavior it changes.

## What This Changes

The PR defers post-turn hooks until source reply delivery drains, cleans up one-shot Codex CLI harness sessions, adds durable post-turn worker state, changes Telegram guest/draft/final-delivery behavior, adds Docker Compose overlays, and updates tests.

## Best Possible Solution

Split this into a narrow CLI/post-delivery liveness fix with exact single-handler post-turn isolation, and move Telegram guest/recovery behavior plus Docker overlays into separately reviewed PRs with real Telegram and security proof.

## Reproduction Assessment

Yes for the PR-introduced hook bug: source inspection shows a plugin with two agent_end or llm_output handlers would schedule two jobs, and each isolated worker would reload and run both handlers. No runtime repro was run in this read-only review, and the Telegram behavior still lacks after-fix real-environment proof.

## Solution Assessment

No. The hook worker should execute exactly the scheduled registration, and the liveness fix should not be bundled with new Telegram modes or host-specific Compose/proxy surfaces without separate owner review.

## Review Findings

Overall confidence: 0.86

Full review comments:

- **[P2] Run only the scheduled hook in the worker:** `src/agents/post-turn/isolated-plugin-hook-worker-entry.ts:59-63`
