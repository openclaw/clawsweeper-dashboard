## Summary

Keep open: the updated head still has source-proven typecheck blockers, failing CI, a regression in foreground reply freshness, duplicated isolated hook execution, and unreviewed Telegram/deployment security surface. The supplied terminal proof helps for the CLI/Gateway liveness claim, but it does not cover the visible Telegram behavior added by this PR.

## What This Changes

The PR changes post-delivery hook scheduling, Codex CLI embedded harness cleanup, Telegram guest/delivery recovery behavior, root Docker Compose overlays, and related tests.

## Best Possible Solution

Split the work into a narrow liveness/post-delivery fix that preserves existing reply freshness behavior, and move Telegram guest-mode plus deployment overlays into separately reviewed PRs with schema/docs, live Telegram proof, and a documented trust model.

## Reproduction Assessment

Yes for the blocking PR defects: CI logs and source inspection show the missing export, type errors, removed foreground reply freshness behavior, and hook-worker registration mismatch. The claimed runtime liveness behavior is only partially covered by the PR body's terminal proof and does not reproduce the Telegram-visible surfaces.

## Solution Assessment

No. The PR is not the best way to solve the liveness issue because it combines a narrow cleanup with broad Telegram guest-mode, recovery, and deployment changes while breaking existing type/tests and adding security-sensitive compose surfaces.

  - body: `dispatch.ts` imports `AfterSourceReplyDeliveryCallback` from `./types.js`, but that barrel still does not export the type, so `check-prod-types` fails with TS2305 on this head. Export the type from the barrel or import it from `get-reply-options.types.js`.

  - body: The PR removes the foreground reply fence that suppresses stale foreground finals after a newer inbound turn starts. Current CI shows `dispatch.freshness.test.ts` failures where older finals remain queued/delivered, which can put stale answers back into the chat after a newer turn.
  - confidence: 0.94
- **[P1] Keep maintenance params compatible with call sites:** `src/agents/pi-embedded-runner/context-engine-maintenance.ts:273-282`
  - body: `buildContextEngineMaintenanceRuntimeContext`/maintenance params no longer include `agentId`, but existing runner call sites still pass `agentId`; `check-prod-types` reports TS2353 in `run.ts` and `run/attempt.ts`. Either retain the parameter in the contract or update all call sites together.
  - confidence: 0.9
- **[P1] Use supported JSON write options:** `src/agents/post-turn/durable-job-state.ts:137`
  - body: `writeJsonAtomic` does not accept `ensureDirMode`, and CI reports TS2353 at this call. Use the supported option name/contract from the JSON file helper so the new durable job store typechecks.
  - confidence: 0.92
- **[P1] Type hook event records before reading fields:** `src/plugins/hooks.ts:641-645`
  - body: The generic `eventRecord`/`ctxRecord` values are narrowed to `{}` here, so reading `runId`, `sessionId`, `sessionKey`, and related fields fails production typecheck. Narrow or cast them to a string-keyed record before calling `firstString`.
  - confidence: 0.88
- **[P2] Preserve the selected hook registration in the worker:** `src/agents/post-turn/isolated-plugin-hook-worker-entry.ts:59-64`
  - body: The parent creates one durable job per hook registration, but the child worker reloads every registration for the same plugin/hook pair and runs all of them. A plugin with multiple `agent_end` or `llm_output` handlers would run each handler once per registration, duplicating post-turn work and corrupting job accounting.
  - confidence: 0.9
- **[P2] Remove the unreviewed proxy compose overlay:** `docker-compose.ai-proxy.yml:3`
  - body: This root overlay adds an unpinned proxy image and routes Gateway/CLI HTTP(S) traffic through it, which is a new deployment trust surface unrelated to the liveness fix. Move it to a separate reviewed change with pinned images and a documented trust model, or remove it from this PR.
  - confidence: 0.88
