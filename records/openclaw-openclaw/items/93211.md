## Summary

This should stay open: current `main` still lacks a classifier for the reported Zhipu/GLM 1305 overload text, and there is no linked or searched fix PR for this exact payload. The fallback loop is more tolerant of unknown thrown errors now, but the provider-owned overload classification and embedded error-payload path remain actionable.

## What This Changes

The issue reports that Zhipu/GLM error code 1305 with Chinese overload text is not classified as a failover signal, so configured model fallback can be skipped.

## Best Possible Solution

Add a focused Z.AI/provider-owned failover classifier for code 1305 and `访问量过大`, with regression tests proving the configured fallback chain advances and reports `overloaded` rather than `unknown`.

## Reproduction Assessment

Yes at source level: a focused classifier test using the reported `[1305][该模型当前访问量过大，请您稍后再试]` text should currently miss `overloaded`. I did not run a live Zhipu saturation repro.

## Solution Assessment

Yes, but the best fix should use the provider-owned `classifyFailoverReason` hook when the pattern is Z.AI-specific; broad shared CJK overload matching is acceptable only if tests show the wording is provider-generic.
