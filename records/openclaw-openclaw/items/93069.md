## Summary

Keep open: current main and latest release still accept or advertise `diagnostics.otel.protocol: "grpc"` while diagnostics-otel only starts OTLP export for `http/protobuf`; the open linked PR is the current fix candidate but is still unmerged and conflicting.

## What This Changes

The issue asks OpenClaw to make the diagnostics OTel config contract, generated help/reference docs, handwritten docs, runtime behavior, and tests agree on whether `grpc` is supported.

## Best Possible Solution

Land a refreshed version of https://github.com/openclaw/openclaw/pull/93087 or equivalent that narrows the canonical config/docs/help contract and repairs shipped `grpc` configs through doctor, unless maintainers choose the larger real gRPC exporter implementation instead.

## Reproduction Assessment

Yes. Source inspection on current main shows the config type/schema/help/reference paths accept or advertise `grpc`, while diagnostics-otel exits before OTLP exporter startup unless the protocol is `http/protobuf`.

## Solution Assessment

Unclear until maintainers accept the compatibility tradeoff. The lower-risk repair is to reject `grpc` in canonical config/docs with doctor migration for shipped configs; implementing real gRPC export would be a larger feature path.
