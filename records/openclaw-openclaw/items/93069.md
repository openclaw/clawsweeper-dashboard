## Summary

Keep open: current main and v2026.6.9 still accept or advertise `diagnostics.otel.protocol: "grpc"`, while diagnostics-otel returns before OTLP exporter startup for any protocol other than `http/protobuf`; the active linked fix is still open and currently conflicting.

## What This Changes

Request to align the `diagnostics.otel.protocol: "grpc"` config/docs/schema contract with diagnostics-otel runtime support by either rejecting unsupported `grpc` early or implementing real gRPC export.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/93087 or an equivalent narrow PR that removes `grpc` from the canonical config/docs/help contract and repairs shipped `grpc` configs through doctor, unless maintainers choose the larger real gRPC exporter implementation.

## Reproduction Assessment

Yes. Source inspection on current main shows schema/type/help/reference paths accept or advertise `grpc`, while diagnostics-otel exits before OTLP exporter startup unless the protocol is `http/protobuf`; the issue discussion also includes a validation probe confirming `grpc` is accepted.

## Solution Assessment

Unclear until maintainers choose the product contract. The narrower maintainable path is to reject `grpc` in canonical config/docs with a doctor migration for shipped configs; implementing real gRPC export is a larger feature path.
