- `impact:other`: The concrete impact is an observability configuration foot-gun where a validated protocol can disable telemetry export at runtime.

## Summary

Keep open: current main and the latest release still accept or advertise `diagnostics.otel.protocol: "grpc"`, while the diagnostics-otel runtime returns before exporter startup for any protocol other than `http/protobuf`; the linked closing PR is still open and has a compatibility blocker around shipped config migration.

## What This Changes

Align the diagnostics OTel protocol contract across config schema, help/docs, runtime behavior, and tests so `grpc` either works or fails early with a clear operator-facing path.

## Best Possible Solution

Land one narrow contract-alignment fix: either implement real gRPC exporters, or remove/reject `grpc` with doctor migration, aligned docs/help/tests, and proof for both fresh validation and upgrade behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows schema/type/docs accepting or advertising `grpc`, while the diagnostics-otel runtime emits `unsupported_protocol` and returns before exporter setup for non-`http/protobuf`.

## Solution Assessment

Unclear until maintainers choose the shipped config contract. Rejecting `grpc` with doctor migration looks like the narrowest repair, while implementing real gRPC export is a larger feature path.
