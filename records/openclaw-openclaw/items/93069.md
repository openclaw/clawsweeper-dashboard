## Summary

Keep open: current main and the latest release still accept or advertise `diagnostics.otel.protocol: "grpc"`, while diagnostics-otel does not start OTLP exporters for non-`http/protobuf`; the linked fixing PR is open but unmerged and currently conflicting.

## What This Changes

The issue asks OpenClaw to align the `diagnostics.otel.protocol` config contract, docs, help text, and diagnostics-otel runtime behavior for unsupported `grpc` values.

## Best Possible Solution

Land a clean version of https://github.com/openclaw/openclaw/pull/93087 that narrows the canonical config/docs/help contract and repairs shipped `grpc` configs through doctor, unless maintainers choose the larger real gRPC exporter implementation instead.

## Reproduction Assessment

Yes. Source inspection on current main shows the config type/schema/help/reference paths accept or advertise `grpc`, while diagnostics-otel exits before OTLP exporter startup unless the protocol is `http/protobuf`.

## Solution Assessment

Unclear until maintainers accept the compatibility tradeoff. The lower-risk repair is to reject `grpc` in canonical config/docs with doctor migration for shipped configs; implementing real gRPC export would be a larger feature path.
