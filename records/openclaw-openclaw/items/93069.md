## Summary

Keep open: current `main` and `v2026.6.8` still accept or advertise `diagnostics.otel.protocol: "grpc"`, while diagnostics-otel still exits before exporter startup unless the protocol is `http/protobuf`. The linked candidate fix at https://github.com/openclaw/openclaw/pull/93087 should stay paired with this issue, but it is open, unmerged, and currently conflicting, so this issue remains the canonical tracker.

## What This Changes

The issue asks OpenClaw to align the `diagnostics.otel.protocol` config contract with diagnostics-otel runtime support so `grpc` either works or is rejected before startup.

## Best Possible Solution

Land a conflict-resolved version of the linked migration/schema/docs/test fix; if maintainers reject disabling legacy `grpc` telemetry during doctor repair, keep this issue open and choose the larger real gRPC exporter implementation instead.

## Reproduction Assessment

Yes. Source inspection on current `main` shows config/schema/help/docs still accept or advertise `grpc`, while diagnostics-otel returns before exporter startup unless the protocol is `http/protobuf`.

## Solution Assessment

Unclear until maintainers accept the shipped-config contract change. The smaller maintainable fix is to remove `grpc` from the canonical config contract with doctor migration while keeping the env-var runtime guard; implementing real gRPC exporters is the larger alternative.
