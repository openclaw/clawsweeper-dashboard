- **[P2] Match lint recursion to runtime discovery depth:** `src/skills/loading/local-loader.ts:225`
  - body: `skills lint` stops descending after depth 4, but runtime grouped-skill discovery allows depth 6 and checks for `SKILL.md` before stopping. A malformed skill under a still-supported deep group can make runtime discovery consider that candidate while lint exits clean, contradicting the new docs; align the budget/depth with runtime discovery or share the resolver and add a deepest-depth regression test.
  - confidence: 0.91
- **[P2] Honor configured skill symlink targets in lint:** `src/skills/discovery/status.ts:103-107`
  - body: Runtime workspace/project/extra discovery can load symlinked skills outside the root when `skills.load.allowSymlinkTargets` trusts the target, but `lintSkillRoots` does not receive that config and the loader still reads through the original root boundary. In that setup, malformed runtime-loadable symlinked skills can be silently omitted from lint; thread the same symlink policy through lint or reuse the runtime candidate resolver with coverage.
  - confidence: 0.87
- **[P3] Align missing-name lint wording with fallback behavior:** `docs/cli/skills.md:127-130`
  - body: The new docs say lint reports missing required fields `name` and `description`, but current runtime falls back to the directory basename when `name` is absent, and the PR keeps that fallback before the missing-name branch. Either make this an explicit validation warning separate from failed-load reporting, or document only the true load-failure cases so users are not promised an unreachable `missing required field: name` result.
  - confidence: 0.8
