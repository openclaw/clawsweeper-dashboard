## Summary

Keep open. The Feishu plugin is the right boundary and the raw-content plumbing is useful, but this PR still has blocking ordering and bounding defects before it is mergeable.

## What This Changes

The PR preserves raw Feishu message content and prepends media resolved from recent pending Feishu group-history entries to the inbound media payload.

## Best Possible Solution

Land a Feishu-owned fix that resolves bounded pending-history media before skip/audio decisions, preserves rawContent, adds focused regressions, and coordinates with #66432's thread-history work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mention-required Feishu group where a screenshot is buffered into chatHistories, then a bare bot mention triggers handling; current main records only placeholder history and this PR resolves it too late.

## Solution Assessment

No, not yet. The Feishu plugin boundary is correct, but the media merge must happen earlier, rich-post downloads must be capped before resolution, and audio indexes must match the final MediaPaths order.

Full review comments:

- **[P2] Resolve history media before the empty-message skip:** `extensions/feishu/src/bot.ts:758-761`
  - body: A bare bot mention after a screenshot leaves the triggering message with empty content and no triggering-message media, so the handler returns at the empty-message guard before this added history-media block can run. Move pending-history media resolution before that guard, or include resolved history media in the skip decision.
  - confidence: 0.86
- **[P2] Enforce the history media cap before resolving posts:** `extensions/feishu/src/bot.ts:791-802`
  - body: The 10-item guard runs only after resolveFeishuMediaList returns, but that helper downloads every embedded image/media in a rich post before returning. A single pending rich post can still fetch far more than 10 assets, defeating the latency bound promised by the PR.
  - confidence: 0.91
- **[P2] Recompute audio indexes after prepending history media:** `extensions/feishu/src/bot.ts:814-827`
  - body: Audio preflight computes MediaTranscribedIndexes from the triggering-message media order before this block prepends history media and rebuilds mediaPayload. If the triggering message is audio, the final MediaPaths order shifts and the transcribed index can point at the wrong item.
  - confidence: 0.82

  - body: This is a user-facing Feishu fix, but the PR does not update CHANGELOG.md. Repo policy requires user-facing fix, feat, and perf changes to add a changelog entry before merge.
