## Summary

Keep open: the Feishu pending-history media problem still exists on current main, but this PR is not merge-ready because the new block would run too late after current empty-message/audio decisions, and its media cap is enforced after rich-post downloads already happened.

## What This Changes

The PR preserves raw Feishu fetched-message content and prepends media resolved from recent pending Feishu group-history entries to the inbound media payload.

## Best Possible Solution

Land a rebased Feishu plugin fix that preserves raw fetched content, resolves pending-history media before empty-message and audio preflight decisions, enforces download caps before resolving rich posts, and adds focused regressions plus a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a mention-required Feishu group where a user posts an image that is buffered into chatHistories, then sends a bare bot mention; current main records only pending-history text and the PR's current insertion point would still miss or mis-index key paths after rebase.

## Solution Assessment

No, not yet. Feishu-owned rawContent/media lookup is the right boundary, but the merge point and resource cap need to move earlier and the audio metadata must be recomputed after any media ordering change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve history media before empty-message skips:** `extensions/feishu/src/bot.ts:759-762`
  - body: On current main, a bare bot mention after a pending screenshot can leave stripped content empty and triggering-message mediaList empty, so the handler returns at the empty-message guard before this later history-media block can run. Move pending-history media resolution before that guard or include resolved history media in the skip predicate.
  - confidence: 0.86
- **[P2] Cap history media before downloading rich posts:** `extensions/feishu/src/bot.ts:801-804`
  - body: The 10-item guard runs after `resolveFeishuMediaList` returns, but that helper downloads every embedded image/media in a rich post before returning. A single pending post with many assets can still trigger far more than 10 downloads, defeating the promised latency/cost bound.
  - confidence: 0.91
- **[P2] Recompute audio indexes after prepending history media:** `extensions/feishu/src/bot.ts:827`
  - body: Current main computes the audio preflight index from the triggering-message mediaList before this block prepends history media and rebuilds MediaPaths. If the triggering message is audio, the final MediaPaths order shifts and MediaTranscribedIndexes can point at the wrong item.
  - confidence: 0.82
