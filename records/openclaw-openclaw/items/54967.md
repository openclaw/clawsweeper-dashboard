## Summary

Keep open. Current main still does not attach media from Feishu pending group history, and this PR is a useful Feishu plugin repair, but the patch is not ready to merge against current main because it can miss bare-mention triggers, over-download rich history posts before enforcing the cap, and misalign audio transcript indexes after prepending history media.

## What This Changes

The PR preserves raw Feishu message body content and prepends best-effort media resolved from recent pending Feishu group-history entries to the inbound agent media payload.

## Best Possible Solution

Land a rebased Feishu plugin fix that preserves fetched rawContent, resolves pending-history media before any empty-message return or audio preflight, enforces the media-item bound before downloading rich post assets, keeps MediaTranscribedIndexes aligned with final MediaPaths, and adds focused Feishu regressions for those cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mention-required Feishu group where a screenshot or rich post is buffered before a bare bot mention; current main has only text pending history, and the PR as written can still return early or over-download. A second focused path is pending history media plus a triggering audio message, which shifts final MediaPaths after the audio index is computed.

## Solution Assessment

No, not yet. The rawContent plumbing and Feishu-owned media lookup are the right direction, but the implementation should move the history media merge earlier and enforce bounds before downloads rather than rebuilding mediaPayload after audio preflight.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve history media before empty-message returns:** `extensions/feishu/src/bot.ts:759-761`
  - body: On current main, a bare @bot trigger after a screenshot leaves `ctx.content` empty and `mediaList` empty, so the handler returns at the empty-message guard before this added history-media block can run. Move the pending-history media resolution before that guard, or include resolved history media in the skip decision.
  - confidence: 0.86
- **[P2] Cap rich history media before downloading:** `extensions/feishu/src/bot.ts:802-805`
  - body: The 10-item guard runs after `resolveFeishuMediaList` returns, but that helper downloads every embedded image/media in a rich post before returning. A single pending post can still fetch far more than 10 assets, defeating the latency and cost bound this change promises.
  - confidence: 0.91
- **[P2] Keep audio indexes aligned after prepending history media:** `extensions/feishu/src/bot.ts:815-827`
  - body: Audio preflight computes `MediaTranscribedIndexes` from the triggering-message `mediaList` before this block prepends history media and rebuilds `mediaPayload`. If the triggering message is audio and history media is prepended, the final `MediaPaths` order shifts and the transcribed index can point at the wrong item.
  - confidence: 0.82
