## Summary

Keep open: the PR targets a real Feishu pending-history media gap, but the current patch is not merge-ready because it resolves history media too late, applies the media cap after rich-post downloads, and can misalign audio metadata. Related #66432 covers Feishu topic thread history media, not this mention-gated pending-history path.

## What This Changes

The PR preserves raw Feishu message content and tries to prepend media resolved from recent pending Feishu group-history entries to inbound agent media context.

## Best Possible Solution

Land a Feishu-owned fix that resolves bounded pending-history media before skip and audio-preflight decisions, preserves rawContent, adds focused regressions and a changelog entry, and keeps #66432's thread-history work coordinated but separate unless a shared helper is extracted.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mention-required Feishu group where a screenshot is buffered into chatHistories, then a bare bot mention triggers handling; current main records only placeholder history, and this PR's block is placed after the empty-message return path.

## Solution Assessment

No, not yet. The Feishu plugin is the right boundary and rawContent plumbing is useful, but the media merge must happen earlier and the rich-post cap must be enforced before downloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve history media before the empty-message skip:** `extensions/feishu/src/bot.ts:758-761`
  - body: A bare bot mention after a screenshot leaves the triggering message with empty content and no triggering-message media, so the handler returns at the empty-message guard before this added history-media block can run. Move pending-history media resolution before that guard, or include resolved history media in the skip decision.
  - confidence: 0.86
- **[P2] Enforce the history media cap before resolving posts:** `extensions/feishu/src/bot.ts:791-802`
  - body: The 10-item guard runs only after `resolveFeishuMediaList` returns, but that helper downloads every embedded image/media in a rich `post` before returning. A single pending rich post can still fetch far more than 10 assets, defeating the latency bound promised by the PR.
  - confidence: 0.91
- **[P2] Recompute audio indexes after prepending history media:** `extensions/feishu/src/bot.ts:814-827`
  - body: Audio preflight computes `MediaTranscribedIndexes` from the triggering-message media order before this block prepends history media and rebuilds `mediaPayload`. If the triggering message is audio, the final `MediaPaths` order shifts and the transcribed index can point at the wrong item.
  - confidence: 0.82
