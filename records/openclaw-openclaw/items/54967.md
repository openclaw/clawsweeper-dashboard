## Summary

Keep this PR open: current main still only attaches media from the triggering Feishu message, and the PR addresses a real pending-history gap, but it is not merge-ready because its stated 10-item download bound is enforced after `resolveFeishuMediaList` can already download every embedded item in a rich post. The adjacent open PR #66432 is related Feishu history-media work, but it targets thread-history rendering rather than this PR’s pre-mention pending-history media path.

## What This Changes

The PR adds raw Feishu message content to `FeishuMessageInfo` and best-effort resolves media from recent pending group-history entries into the inbound agent media payload.

## Best Possible Solution

Rebase or replace the PR with a current-main patch that preserves raw Feishu content for fetched messages, resolves pending-history media before any empty-trigger return, enforces media-count limits before downloads, adds focused Feishu regression coverage, and includes the required changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is clear: in mention-required Feishu groups, current main records unmentioned media messages as pending history but builds `MediaPaths` only from the later triggering message. A focused unit test can model an image before a bot mention and assert that current main lacks the pending image path.

## Solution Assessment

No. The direction is useful, but the current patch is not the best merge shape until it enforces the download cap before resolving rich-post media and is rebased against current main’s empty-message handling.
