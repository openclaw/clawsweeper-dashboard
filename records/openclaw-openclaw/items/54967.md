## Summary

Keep open. Current main still does not attach media from Feishu pending group history, and the PR is useful but not merge-ready because it can be bypassed by the current empty-message return, over-download rich post media before enforcing its cap, and misalign audio transcript indexes after prepending history media.

## What This Changes

The PR preserves raw Feishu message body content and prepends best-effort media resolved from recent pending Feishu group-history entries to the inbound agent media payload.

## Best Possible Solution

Land a rebased Feishu plugin fix that resolves pending-history media before empty-message and audio preflight decisions, enforces item caps before downloads, preserves rawContent, and adds focused regressions plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mention-required Feishu group where an image or rich post is buffered into chatHistories, then a bare bot mention triggers handling; current main records only text placeholders and the proposed branch would still return before its history-media block in that path.

## Solution Assessment

No, not yet. Feishu-owned rawContent/media lookup is the right boundary, but the merge must run earlier and use pre-download caps instead of rebuilding mediaPayload after skip/audio metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve history media before empty-message returns:** `extensions/feishu/src/bot.ts:759-761`
  - body: On current main, a bare bot mention after a screenshot leaves ctx.content empty and mediaList empty, so the handler returns at the empty-message guard before this added history-media block can run. Move pending-history media resolution before that guard, or include resolved history media in the skip decision.
  - confidence: 0.86
- **[P2] Cap rich history media before downloading:** `extensions/feishu/src/bot.ts:802-805`
  - body: The 10-item guard runs after resolveFeishuMediaList returns, but that helper downloads every embedded image/media in a rich post before returning. A single pending post can still fetch far more than 10 assets, defeating the latency and cost bound this change promises.
  - confidence: 0.91
- **[P2] Keep audio indexes aligned after prepending history media:** `extensions/feishu/src/bot.ts:815-827`
  - body: Audio preflight computes MediaTranscribedIndexes from the triggering-message mediaList before this block prepends history media and rebuilds mediaPayload. If the triggering message is audio and history media is prepended, the final MediaPaths order shifts and the transcribed index can point at the wrong item.
  - confidence: 0.82
