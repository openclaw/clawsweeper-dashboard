## Summary

Keep open: current main still does not attach media from Feishu pending mention history, and this PR is aimed at the right Feishu plugin boundary, but it is not merge-ready because the history-media merge must be rebased around current empty-message/audio ordering and must enforce its media cap before downloads. The related open PR #66432 covers Feishu thread history media, not this pending group-mention history path.

## What This Changes

This PR preserves raw Feishu fetched-message content and prepends best-effort media resolved from recent pending Feishu group-history entries to inbound agent media payloads.

## Best Possible Solution

Land a rebased Feishu plugin fix that preserves rawContent, resolves bounded pending-history media before empty-message and audio-preflight decisions, keeps media indexes aligned, adds focused regressions and a changelog entry, and coordinates any shared rawContent pieces with #66432.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mention-required Feishu group where an image or rich post is buffered into chatHistories, then a bare bot mention triggers handling; current main records only placeholder text and the PR's merge point must run before the current empty-message return.

## Solution Assessment

No, not yet. Feishu-owned rawContent/media lookup is the right boundary, but the current patch needs earlier integration with current main and a pre-download cap before it is the narrow maintainable fix.

  - body: On current main, a bare bot mention after a pending screenshot can have empty `ctx.content` and no triggering-message media, so the handler returns at the empty-message guard before this later history-media block would run.   - body: The `10`-item guard runs after `resolveFeishuMediaList` returns, but that helper downloads every embedded image/media in a rich `post` before returning. A single pending rich post can still fetch far more than the promised cap, defeating the latency/resource bound.

  - body: Current main computes audio preflight and `MediaTranscribedIndexes` from the triggering-message media list before this block rebuilds `mediaPayload`. If history media is prepended ahead of a triggering audio message, the final `MediaPaths` order shifts and the transcribed index can point at the wrong item.
  - confidence: 0.82
