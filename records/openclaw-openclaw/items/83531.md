## Summary

Keep open: this is useful Feishu reaction-tool work, but the current diff has blocking issues. The new default-enabled tool bypasses the existing `actions.reactions` policy, the emoji fallback corrupts valid mixed-case Feishu emoji types, and the supplied real-world proof is too indirect to verify the new `feishu_reaction` tool itself.

## What This Changes

The PR adds a default-enabled Feishu `feishu_reaction` agent tool with config/schema/manifest wiring, emoji alias normalization, tests, and docs.

## Best Possible Solution

Expose the native Feishu reaction tool through one coherent reaction policy path, preserve official emoji type casing after alias normalization, and add direct real Feishu add/list/remove proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR registers the new tool through `tools.reactions` only, while current main gates reaction actions through `actions.reactions`. The official Feishu docs and current `Typing` usage show valid mixed-case emoji types that the PR would uppercase.

## Solution Assessment

No; the current implementation is not the safest policy fit because it creates a second reaction gate and case-mangles valid Feishu emoji names. The maintainable fix is to reuse or explicitly reconcile the existing reaction action gate and normalize against the official emoji vocabulary while preserving canonical casing.
