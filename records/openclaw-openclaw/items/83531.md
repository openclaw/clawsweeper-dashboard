## What This Changes

This PR adds a bundled Feishu `feishu_reaction` tool, config and manifest wiring, emoji alias normalization, tests, and docs for add/remove/list reaction actions.

## Best Possible Solution

Keep one Feishu reaction contract: either improve the existing `message` reaction path and docs, or add a dedicated tool that delegates to the same gate and canonical normalization with focused tests and real Feishu proof.

## Reproduction Assessment

Yes, source-level: current main already gates Feishu `message` reaction actions with `actions.reactions`, while the PR registers `feishu_reaction` from a separate default-on `tools.reactions` gate without checking that existing disable path.

## Solution Assessment

No, not as written. A dedicated Feishu reaction tool may be acceptable, but it should share the existing action gate and canonical reaction normalization instead of creating a second default-on reaction surface.
