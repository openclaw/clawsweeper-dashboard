## Summary

Close as implemented. The reported Docker topology is now covered on current main by running the OpenClaw runtime image under `tini` and by keeping the bundled Compose path on Docker init, so the gateway no longer has to act as PID 1 for adopted Chromium/crashpad descendants.

## What This Changes

The issue reports Chromium and chrome_crashpad zombies accumulating under the OpenClaw gateway process in long-running Docker containers until PID exhaustion breaks tool execution.

## Best Possible Solution

Keep the image-level Tini and Compose init coverage; leave separate sandbox PID-1 reaping work tracked by https://github.com/openclaw/openclaw/issues/68691.

## Reproduction Assessment

Yes. The issue gives strong production diagnostics for a Docker gateway running as PID 1, and the later canonical Docker-runtime issue and PR verified the same PID-1 orphan-reaping failure mode from source and Docker smoke output.

## Solution Assessment

Yes. For the reported container topology, an init/subreaper at the Docker runtime boundary is the narrow maintainable fix because Chromium/crashpad descendants adopted by PID 1 are then reaped outside the Node gateway process.
