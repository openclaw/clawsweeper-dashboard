## Summary

Keep open. Current main has improved Docker PID-1 handling, but the browser plugin still supervises only the direct Chromium `ChildProcess` and has no browser-owned process-tree or descendant reaping path for the reported normal-exit Chromium/crashpad leak.

## What This Changes

The issue requests a fix for managed Chromium and chrome_crashpad processes accumulating as zombies after normal browser exits in PID-1 gateway containers.

## Best Possible Solution

Harden the bundled browser plugin lifecycle so OpenClaw-owned Chromium exits, explicit stops, restarts, bootstrap retries, and crashpad/adopted descendants are supervised and reaped with focused Linux regression coverage.

## Reproduction Assessment

Yes. The issue gives a high-confidence production diagnostic path in browser-heavy PID-1 gateway containers, and current source still matches a direct-child-only lifecycle gap; I did not run a live long-duration container repro in this read-only pass.

## Solution Assessment

No current main solution is sufficient for this issue. The Docker `tini` work is useful related mitigation, but the narrow maintainable fix still belongs in the browser plugin lifecycle, with a shared SDK process helper only if needed for plugin-boundary safety.
