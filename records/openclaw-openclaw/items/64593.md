## Summary

Keep open. Current main and the latest release still supervise only the direct Chromium `ChildProcess`, while the report describes normal-exit `chromium` and `chrome_crashpad` zombies accumulating under the live OpenClaw gateway process in PID-1 containers.

## What This Changes

The issue reports Chromium and crashpad child processes becoming unreaped zombies after normal browser-tool use in long-running gateway containers.

## Best Possible Solution

Fix the bundled browser plugin lifecycle so OpenClaw-owned Chromium is supervised and reaped across normal exits, explicit stops, restarts, startup retry cleanup, and crashpad/adopted descendants, with a focused Linux container regression.

## Reproduction Assessment

Yes, at source level. The issue gives concrete production diagnostics for browser-heavy PID-1 gateway containers, and current source still has the direct-child-only lifecycle gap; I did not run a live container stress repro in this read-only review.

## Solution Assessment

No current solution on main is sufficient. The requested direction is right in principle, but the maintainable fix should be a guarded current-path browser lifecycle change that handles normal exits and descendants, not a broad unsafe process killer.
