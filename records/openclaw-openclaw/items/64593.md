## Summary

Keep open. Current main still launches and tracks only the direct managed Chromium child process, while the reported normal-exit leak involves chromium and chrome_crashpad zombies under the live OpenClaw parent; the startup-failure, sandbox PID-1, and gateway-orphan related items do not supersede this browser-plugin lifecycle gap.

## What This Changes

The issue requests fixing Chromium and chrome_crashpad zombie accumulation after normal managed-browser exits in containerized gateway deployments.

## Best Possible Solution

Fix the bundled browser plugin lifecycle so OpenClaw-owned Chromium is supervised and reaped across normal exits, explicit stops, restarts, startup retry cleanup, and crashpad/adopted descendants, with focused unit and Linux/container regression coverage.

## Reproduction Assessment

Yes. The issue gives a high-confidence production reproduction and diagnostic path: run browser-heavy OpenClaw gateway containers as PID 1, then count defunct chromium/chrome_crashpad children whose PPID is the OpenClaw process; current source still matches that lifecycle gap.

## Solution Assessment

No current solution on main is sufficient. The requested browser-service boundary is correct, but waiting on only the direct Chromium ChildProcess is unlikely to cover crashpad or adopted descendants; a scoped browser lifecycle supervisor or subreaper strategy is safer than process-group signaling alone.
