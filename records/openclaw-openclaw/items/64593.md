## Summary

Keep open. Current main still launches and tracks only the direct managed Chromium child process, while the reported failure is normal browser use leaving chromium and chrome_crashpad zombies under the OpenClaw parent in long-running containers. The related startup-failure, sandbox PID-1, and gateway-orphan items do not supersede this browser-plugin lifecycle gap.

## What This Changes

Issue requests fixing Chromium and chrome_crashpad zombie accumulation after normal managed-browser exits in containerized gateway deployments.

## Best Possible Solution

Fix the bundled browser plugin lifecycle so managed Chromium is container-safe across normal exits, explicit stops, restarts, and startup retry cleanup. The desired end state is a scoped supervisor/subreaper or equivalently safe tree-reaping strategy for the OpenClaw-owned browser, focused unit coverage, a Linux/container regression that repeats managed browser exits and checks for no defunct chromium or chrome_crashpad descendants, and docs only if operators must set a container option.

## Reproduction Assessment

Yes. The issue gives a high-confidence production reproduction and diagnostic path: run browser-heavy OpenClaw gateway containers as PID 1, then count defunct chromium/chrome_crashpad children whose PPID is the OpenClaw process. I did not run that live scenario here, but current source still matches the reported lifecycle gap.

## Solution Assessment

No current solution on main is sufficient. The requested browser-service boundary is correct, but waiting on only the direct Chromium ChildProcess is unlikely to cover crashpad or adopted descendants; a scoped browser lifecycle supervisor/subreaper is the safer maintainable fix.
