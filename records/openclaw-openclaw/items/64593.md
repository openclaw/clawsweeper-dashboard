## Summary

Keep open. Current main still tracks and stops only the direct managed Chromium ChildProcess, while this report concerns normal-exit chromium and chrome_crashpad zombies whose parent remains the live OpenClaw gateway process.

## What This Changes

The issue requests fixing Chromium and chrome_crashpad zombie accumulation after normal managed-browser exits in containerized gateway deployments.

## Best Possible Solution

Fix the bundled browser plugin lifecycle so OpenClaw-owned Chromium is supervised and reaped across normal exits, explicit stops, restarts, startup retry cleanup, and crashpad/adopted descendants.

## Reproduction Assessment

Yes. The issue gives a high-confidence production diagnostic path in browser-heavy PID-1 gateway containers, and current source still matches the reported direct-child-only lifecycle gap.

## Solution Assessment

No current solution on main is sufficient. The right boundary is the browser plugin lifecycle, but the implementation needs owned-process supervision or reaping rather than only signaling the direct Chromium ChildProcess.
