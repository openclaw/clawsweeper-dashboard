## Summary

Close as implemented: the supported Docker runtime now runs under Tini as PID 1, with Compose init coverage too, so the reported container topology that reparented Chromium/crashpad descendants to the OpenClaw gateway is fixed and shipped in v2026.5.18.

## What This Changes

The issue asks OpenClaw to stop Chromium and chrome_crashpad zombies from accumulating under the gateway process after normal browser exits in long-running PID-1 containers.

## Best Possible Solution

Keep the Tini-backed runtime image and Compose init coverage, and treat any v2026.5.18-or-newer browser zombie repro under Tini as a new focused browser lifecycle report.

## Reproduction Assessment

Yes for the original topology: the issue gives production PID-1 container diagnostics, and the pre-fix image lacked an init wrapper. Current main should not reproduce that same path because Tini is now PID 1 in the supported image.

## Solution Assessment

Yes. Moving PID-1 reaping to Tini is the narrow runtime-boundary fix for adopted Chromium/crashpad descendants in supported containers; browser-service changes should wait for a current-image repro where Tini is already PID 1.
