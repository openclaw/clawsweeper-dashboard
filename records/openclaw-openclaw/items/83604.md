## Summary

Keep open: current main and the latest release still have the memory-wiki public-artifacts bridge and memory-core provider, but memory-lancedb registers only LanceDB tools and no `publicArtifacts` provider, so the requested parity remains source-actionable and is not superseded by the Honcho or Active Memory issues.

## What This Changes

Request to add `publicArtifacts.listArtifacts` support to the official memory-lancedb plugin so memory-wiki bridge can import public memory artifacts when LanceDB owns the memory slot.

## Best Possible Solution

Register a memory-lancedb memory capability that preserves existing runtime capability fields and exposes a public-artifacts provider, preferably via a shared memory-host artifact helper or a plugin-local helper that avoids importing memory-core private internals.

## Reproduction Assessment

Yes, by source inspection: with `plugins.slots.memory = "memory-lancedb"` and memory-wiki bridge artifact reads enabled, the bridge calls `listActiveMemoryPublicArtifacts`, but memory-lancedb registers no `publicArtifacts.listArtifacts` provider so the shared state returns `[]`. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Yes, adding the capability at the memory-lancedb plugin boundary is the narrow maintainable path. The implementation should not special-case memory-wiki or import memory-core private internals, and it should coordinate with the runtime-registration PR so one capability registration carries all fields.
