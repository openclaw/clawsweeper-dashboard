- `impact:session-state`: The issue is about memory/wiki bridge state being starved of public memory artifacts when `memory-lancedb` owns the memory slot.

## Summary

Keep open: current main has the generic public-artifacts bridge and `memory-core` implements it, but `memory-lancedb` still only registers LanceDB tools, so bridge mode has no provider to list artifacts when LanceDB owns the memory slot. The related Honcho reports are external-plugin scope and the active-memory report is a different tool-propagation problem, so they do not supersede this issue.

## What This Changes

Add `publicArtifacts.listArtifacts` support to the official `memory-lancedb` plugin so `memory-wiki` bridge mode can import public memory artifacts when LanceDB owns the memory slot.

## Best Possible Solution

Register a `memory-lancedb` memory capability with a public-artifacts provider, preferably through a shared memory-host helper reused by `memory-core`, with focused bridge/import tests and docs only if maintainers intentionally decide the plugin should not support bridge export.

## Reproduction Assessment

Yes, by source inspection: configure `plugins.slots.memory = "memory-lancedb"` with `memory-wiki` bridge mode and the bridge calls `listActiveMemoryPublicArtifacts`, but the active plugin registers no `publicArtifacts` provider, so the shared state returns `[]`. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Yes, adding the capability at the `memory-lancedb` plugin boundary is the narrow maintainable path; the safer implementation should reuse or promote the existing workspace artifact listing logic rather than importing `memory-core` private internals.
