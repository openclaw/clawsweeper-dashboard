## Summary

Keep open: current main still has the public-artifacts bridge and memory-core parity implementation, but memory-lancedb does not register a memory capability or public artifact provider, so bridge mode remains starved when that plugin owns the memory slot. The related Honcho items are external-plugin scope and the active-memory thread is a distinct embedded-tool problem, so they do not supersede this request.

## What This Changes

Add publicArtifacts.listArtifacts support to @openclaw/memory-lancedb so memory-wiki bridge mode can import public memory artifacts when memory-lancedb owns the memory slot.

## Best Possible Solution

Register a memory-lancedb memory capability with a public-artifacts provider, preferably by sharing or promoting the existing workspace artifact listing logic; document unsupported bridge behavior only if maintainers decide the plugin should not export public artifacts.

## Reproduction Assessment

Yes, by source inspection: configure plugins.slots.memory to memory-lancedb with memory-wiki bridge mode enabled, and the bridge calls listActiveMemoryPublicArtifacts while the active plugin has no publicArtifacts provider, so the shared helper returns an empty list. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Yes: adding the capability at the memory-lancedb plugin boundary is the narrow maintainable solution. The safer implementation should reuse or promote the existing public workspace-artifact listing logic rather than import memory-core private internals.
