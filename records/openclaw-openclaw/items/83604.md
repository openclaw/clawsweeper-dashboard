## Summary

Keep open: current main still has the public-artifacts bridge and `memory-core` provider, but `memory-lancedb` only declares and registers tools, so bridge mode still has no `publicArtifacts.listArtifacts` provider when LanceDB owns the memory slot. The Honcho reports are external-plugin scope, and the active-memory issue is a different embedded-tool problem, so neither supersedes this issue.

## What This Changes

Request to add `publicArtifacts.listArtifacts` support to the official `memory-lancedb` plugin so `memory-wiki` bridge mode can import public memory artifacts when LanceDB owns the memory slot.

## Best Possible Solution

Register a `memory-lancedb` memory capability with a public-artifacts provider, preferably via a shared memory-host helper reused with `memory-core`; document the lack of bridge export only if maintainers intentionally decide not to support it.

## Reproduction Assessment

Yes, by source inspection: configure `plugins.slots.memory = "memory-lancedb"` with `memory-wiki` bridge mode and the bridge calls `listActiveMemoryPublicArtifacts`, but the active plugin has no `publicArtifacts` provider, so the shared state returns `[]`. I did not run a live Gateway scenario because this review is read-only.

## Solution Assessment

Yes, adding the capability at the `memory-lancedb` plugin boundary is the narrow maintainable path. The safer implementation should reuse or promote the existing workspace artifact listing logic rather than importing private `memory-core` internals directly.
