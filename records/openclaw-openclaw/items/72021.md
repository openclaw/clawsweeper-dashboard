## Summary

Keep open: current main and v2026.6.9 still let aggregate daily/session/grounded signals satisfy the promotion `minRecallCount` gate, and fresh production comments report the same zero-recall MEMORY.md bloat, so this is not conservative cleanup-closeable.

## What This Changes

The issue asks memory-core to stop treating daily/session/grounded signal counts as real recalls for short-term promotion eligibility.

## Best Possible Solution

Maintainers should define the memory-core signal contract for organic recalls versus daily/session/grounded evidence, then align rank/apply gates, scoring/audit output, docs, and regression tests around that policy.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.9 gate on aggregate `signalCount`, and the current test suite explicitly ranks a `recallCount=0`, `dailyCount=3` candidate under default gates; no live workspace replay was run in this read-only review.

## Solution Assessment

Unclear. A recallCount-only gate is a plausible fix for the false-positive path, but current tests intentionally preserve daily-only and grounded-signal promotion, so the best fix needs a maintainer signal-policy decision before code changes.
