## Summary

Keep this issue open. Current main still uses aggregate short-term signal counts for the promotion recall gate, and open PR #74114 is already the active closing-syntax implementation path that maintainers need to review, land, revise, or reject.

## What This Changes

Separate true user recall counts from daily/session/grounded aggregate signal counts in short-term memory promotion so automated dreaming signals do not satisfy or distort real-recall promotion thresholds.

## Best Possible Solution

Review PR #74114 as the canonical implementation path, then land or revise one memory-core change that clearly separates true user recalls from automated daily/session evidence where intended, updates the promotion tests, and aligns the memory/dreaming docs with the chosen threshold contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the mixed-signal gate: current source and tests show `recallCount: 0` plus `dailyCount: 3` can clear default promotion gates. The exact reported score values require the reporter's private recall-store fixture.

## Solution Assessment

Is this the best way to solve the issue? Unclear until PR #74114 is reviewed against the existing daily and grounded promotion contracts. A narrow recall-only gate is plausible, but the maintainable fix must deliberately preserve or update the tested dreaming lanes instead of silently invalidating them.
