## Summary

Keep open: current main and v2026.6.10 still gate deep-promotion ranking and apply on aggregate signalCount, and current tests intentionally preserve daily-only promotion with recallCount=0, so this remains an active memory-core signal-policy bug rather than cleanup.

## What This Changes

The issue reports that short-term memory promotion uses combined signalCount from recall, daily, and grounded signals for the minRecallCount gate, allowing daily/session-only entries to pass recall gates and crowd out real recalls.

## Best Possible Solution

Define the promotion threshold contract so minRecallCount is not satisfied by daily/session-only signals, preserve or split any intentional grounded-evidence path explicitly, then update rank/apply/explain/docs/tests together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: daily/session ingestion writes daily signals, the ranker sums them into signalCount, and the rank/apply paths compare that aggregate to minRecallCount. I did not run tests because this review is read-only, but current tests already codify a daily-only candidate with recallCount=0 passing default promotion gates.

## Solution Assessment

No final fix is ready without a maintainer signal-policy decision. A recallCount-only gate would address the reported daily/session false positives, but current grounded-only tests show maintainers need to decide whether grounded evidence gets its own explicit path or threshold instead of sharing minRecallCount semantics.
