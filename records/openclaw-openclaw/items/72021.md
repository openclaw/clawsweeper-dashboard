## Summary

Keep open: current main and the v2026.6.9 tag still let aggregate signalCount satisfy the deep-promotion minRecallCount gate, and current tests intentionally preserve daily-only promotion, so this remains a real memory-core signal-policy bug/decision rather than cleanup.

## What This Changes

The issue asks memory-core short-term promotion to stop treating daily/session/grounded signal counts as real recalls for promotion eligibility and scoring because recallCount=0 entries can promote while real recall entries can be hidden.

## Best Possible Solution

Maintainers should define whether deep promotion gates require organic recalls or aggregate short-term evidence, then align rank/apply gates, scoring, audit output, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level. Current main still lets aggregate daily/grounded signals satisfy minRecallCount, and an existing test explicitly ranks a recallCount=0, dailyCount=3 candidate under default gates; no live workspace command was run in this read-only review.

## Solution Assessment

Unclear. A recallCount-only gate is a plausible repair for the false-positive path, but current tests intentionally preserve aggregate signal promotion, so maintainers should settle the signal policy before code changes.
