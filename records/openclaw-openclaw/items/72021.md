## Summary

Keep open. Current main and the latest v2026.6.6 release still gate memory-core promotion on aggregate signalCount, while the previously linked fix PR is now closed unmerged and current tests intentionally cover daily-only promotion, so this needs a maintainer signal-contract decision rather than cleanup close.

## What This Changes

The issue asks memory-core short-term promotion to stop treating daily/session/grounded signal counts as real recall counts for the minRecallCount promotion gate.

## Best Possible Solution

Settle one memory-core promotion signal contract, then update rank/apply gates, scoring or annotations, docs, and regression tests so daily/session/grounded evidence cannot silently promote recallCount=0 noise unless that path is explicitly named and intended.

## Reproduction Assessment

Yes. Source inspection on current main shows rank and apply both gate minRecallCount on aggregate signalCount, and the current test suite explicitly expects recallCount=0,dailyCount=3 to clear default gates; I did not run live promotion because this review is read-only.

## Solution Assessment

Unclear. A recallCount-only gate fixes the false-eligibility symptom, but the best fix must deliberately preserve, rename, or retire daily/session and grounded promotion semantics with aligned tests and docs.
