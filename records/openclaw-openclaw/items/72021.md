## Summary

Keep open. Current main still gates `minRecallCount` on aggregate `signalCount`, and an existing daily-only fixture encodes the reported false-candidate behavior; open PR #74114 explicitly closes this issue and is the active implementation path.

## What This Changes

Request that short-term promotion distinguish real `recallCount` from daily/session/grounded signals for promotion gating and scoring.

## Best Possible Solution

Review and land or revise #74114 so promotion gate semantics are explicit, with aligned tests for recall, daily/session, and grounded signals.

## Reproduction Assessment

Yes for the mixed-signal gate from source inspection: current code gates on aggregate `signalCount`, and the fixture shows three daily-only signals can rank with `recallCount` 0. The exact score-distribution example still depends on the reporter's private recall-store data.

## Solution Assessment

Yes for keeping this issue open: #74114 is the active closing PR, so the issue should not close before that PR merges. The narrow recallCount-only gate matches the CLI docs and `promote-explain` threshold semantics better than the current aggregate gate.
