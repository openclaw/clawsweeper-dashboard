## Summary

Keep open: current main and the latest release still gate deep promotion on aggregate signalCount, and current tests intentionally allow daily-only promotion, so the reported behavior is unresolved and needs maintainer product judgment rather than cleanup closure.

## What This Changes

The issue asks memory-core short-term promotion to separate real recall counts from daily/session/grounded signal counts when applying promotion gates and scoring.

## Best Possible Solution

Maintainers should decide the durable promotion signal contract, then align rank/apply gates, audit output, docs, and regression tests around one definition of recall versus aggregate signal count.

## Reproduction Assessment

Yes. Source inspection and an existing test show current main can rank a default candidate with `recallCount=0` and `dailyCount=3`; issue comments add real workspace examples, but I did not run a live workspace command in this read-only review.

## Solution Assessment

Unclear. A recallCount-only gate would address the false-positive path, but current tests intentionally support daily/grounded signal promotion, so maintainers need to settle the policy before a fix is safest.
