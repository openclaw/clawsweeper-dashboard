## Summary

Keep this open. Current main still uses aggregate short-term signal counts for the promotion recall gate, and open PR #74114 is already the active closing-syntax implementation path that maintainers need to land, revise, or reject.

## What This Changes

The issue asks memory-core short-term promotion to distinguish real user recallCount signals from automated daily/session-derived signals when applying promotion gates and scoring candidates.

## Best Possible Solution

Codify the promotion threshold contract, then land one policy-compatible PR that separates true user recalls from automated daily/session evidence where intended and updates memory-core tests plus memory/Dreaming docs before closure.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the mixed-signal gate: current source and tests show `recallCount: 0` plus `dailyCount: 3` can clear default promotion gates. The exact reported score values need the reporter's private recall-store fixture to reproduce numerically.

## Solution Assessment

Is this the best way to solve the issue? Unclear. PR #74114 addresses the core report with a recall-only gate, but the maintainable fix must reconcile existing daily/session and grounded promotion behavior rather than silently invalidating tested paths.
