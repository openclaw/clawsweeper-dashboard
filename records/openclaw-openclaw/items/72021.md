## Summary

Keep open. Current main and the latest shipped tag still gate memory-core promotion on aggregate signal count, the latest real-world comment reproduces the dailyCount-only false-positive path, and the open closing PR remains the focused implementation path while maintainers decide the recall/daily/grounded signal contract.

## What This Changes

The issue asks for short-term promotion to stop treating daily/session promotion signals as equivalent to real recallCount signals when enforcing recall gates and scoring candidates.

## Best Possible Solution

Settle one memory-core promotion signal contract, then land the gate logic, regression tests, diagnostics, docs, and release notes through the linked implementation path.

## Reproduction Assessment

Yes. Source inspection on current main shows the rank and apply paths still gate on aggregated recallCount + dailyCount + groundedCount, and the May 22 real-world comment provides a current dailyCount-only false-positive candidate list; I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a pure recallCount-only swap. That path addresses the reported false eligibility, but the best fix must explicitly decide what daily and grounded signals are allowed to do in promotion gates and update the existing daily-only and grounded-only eligibility tests accordingly.
