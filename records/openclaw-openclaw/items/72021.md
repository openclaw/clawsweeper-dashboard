## Summary

Keep open: current main and v2026.6.9 still use aggregate signalCount for the deep-promotion minRecallCount gate, and current tests explicitly preserve daily-only promotion with recallCount=0, so this remains a real memory-core signal-policy bug/maintainer decision rather than cleanup.

## What This Changes

The issue asks memory-core short-term promotion to stop treating daily/session/grounded signals as real recalls for minRecallCount promotion eligibility.

## Best Possible Solution

Maintainers should define whether deep promotion gates require organic recalls or aggregate short-term evidence, then align rank/apply gates, scoring, audit output, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level. Current main still lets aggregate daily/grounded signals satisfy minRecallCount, and an existing test explicitly ranks a recallCount=0, dailyCount=3 candidate under default gates.

## Solution Assessment

Unclear. A recallCount-only gate is a plausible repair for the false-positive path, but current tests intentionally preserve aggregate signal promotion, so maintainers should settle the signal policy before code changes.
