## What This Changes

The issue reports a Gateway startup regression where a manually cloned source-root memory plugin is listed as enabled but rejected by `plugins.slots.memory` validation when a persisted installed-plugin index already exists.

## Best Possible Solution

Review and land one narrow registry-boundary fix that makes Gateway validation and startup derive a fresh registry when a configured source-root plugin is missing from the persisted index, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: current main can reuse a persisted installed-plugin index, build validation `knownIds` from that index path, and reject an explicit `plugins.slots.memory` id that only exists in the global source-root discovery path. I did not run the live Windows Gateway flow.

## Solution Assessment

Yes, the best fix shape is a narrow registry-boundary stale-index invalidation for configured source-root plugins, not a broad per-call scan. The open linked PR is the current candidate path and should be reviewed rather than starting another fix.
