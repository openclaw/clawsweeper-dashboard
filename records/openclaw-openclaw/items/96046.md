## Summary

Keep open: current main still lacks the exact stale-index check for a configured global source-root plugin, and the open closing PR is the active implementation candidate.

## What This Changes

Gateway startup rejects a configured memory-slot plugin cloned into the global extensions root when a persisted plugin index already exists.

## Best Possible Solution

Land one narrow registry freshness fix that invalidates the persisted index when a configured global source-root plugin is present on disk but missing from the index, with regression coverage for the stale-index path.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can return a non-empty persisted index without a configured global source-root stale check, then validates `plugins.slots.memory` against registry IDs from that index. I did not run the live Windows Gateway scenario.

## Solution Assessment

Yes, the best fix belongs at the registry freshness/discovery boundary, not in slot validation itself. The implementation should stay narrow enough to avoid broad per-call scans while making Gateway validation see configured source-root plugins that discovery already finds.
