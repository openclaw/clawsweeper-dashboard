## Summary

This should stay open: current main still has the release-era path where config validation can reject an explicit memory slot because the manifest registry is narrowed to the persisted installed-plugin index, and the existing staleness checks do not cover a newly added source-root plugin missing from that index.

## What This Changes

The issue reports a 2026.6.9 Gateway startup regression where an enabled local `extensions/` memory plugin appears in plugin inventory but `plugins.slots.memory` validation fails with `plugin not found`.

## Best Possible Solution

Make the persisted registry freshness path rederive when a configured or source-root plugin is discoverable but missing from the persisted index, then add regression coverage for installing one plugin, adding a later local memory plugin, and starting with `plugins.slots.memory` set to that id.

## Reproduction Assessment

Yes at source level: the validation path derives `knownIds` from a metadata snapshot, and the current snapshot code uses an index-only manifest registry when the persisted index is non-empty. I did not run the full Windows/npm package scenario in this read-only review.

## Solution Assessment

Yes, the best fix is in persisted-registry freshness or validation metadata derivation, not a memory-plugin-specific exception. A broad Gateway hot-path scan would conflict with the process-stable plugin metadata boundary.
