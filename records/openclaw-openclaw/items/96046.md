## What This Changes

The issue reports a 2026.6.9 Gateway startup regression where `plugins.slots.memory` rejects a source-root plugin that `plugins list` discovers after a persisted installed-plugin index already exists.

## Best Possible Solution

Review and land one narrow fix that makes Gateway config validation see configured source-root plugins missing from the persisted index, with regression coverage for stale index plus global extension root behavior.

## Reproduction Assessment

Yes, source-reproducible: current main still returns a non-empty persisted plugin index when no stale check trips, then validates `plugins.slots.memory` against registry IDs derived from that index. I did not run the live Windows Gateway flow in this read-only review.

## Solution Assessment

Yes, the requested direction is the right bug boundary: registry freshness or source-root discovery for configured plugins. The best implementation should avoid a broad per-call scan and add a focused regression test around stale persisted indexes.
