## Summary

Keep open. Current main and the latest release still gate cross-agent session listing and sending through `tools.sessions.visibility` plus `tools.agentToAgent`, and the Telegram-specific one-way asymmetry still needs an Agent A/B policy table or live proof before changing the authorization boundary.

## What This Changes

The issue reports asymmetric Telegram group cross-agent session visibility where one agent can list/send to the peer group session key but the other cannot.

## Best Possible Solution

Define an explicit cross-agent session contract that separates safe A2A sends from list/history/status visibility, preserves default privacy, and proves the Telegram group A/B flow with tests plus redacted live evidence.

## Reproduction Assessment

No high-confidence exact repro yet. Source inspection shows current main can hide cross-agent rows and sends unless visibility is `all` and A2A policy allows the target, but the issue lacks the Agent A/B policy table or live Telegram proof needed to distinguish a policy denial from an index asymmetry.

## Solution Assessment

Unclear until maintainers choose the authorization contract. The safest direction is not implicit same-group trust, but an explicit send/read visibility split or allowlisted policy with regression coverage and live Telegram proof.
