## Summary

Keep open. The requested Slack behavior is useful, but this external feature PR is not merge-ready because it introduces a new config-driven prompt/history policy, routes raw recent Slack messages into prompt fields that can be parsed as inline directives, and does not include inspectable real behavior proof.

## What This Changes

The branch adds `slashCommand.contextAware` to Slack config and changes empty Slack slash commands to fetch the last 10 channel messages and send that history as the prompt.

## Best Possible Solution

Implement the feature in the current Slack plugin path with bounded, sanitized or enveloped untrusted history kept separate from directive input, plus focused Slack slash tests, docs/schema updates, and redacted real Slack proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible for review: current main rejects empty Slack slash prompts, and the PR source shows the proposed empty-prompt history path and directive fallback risk without needing a live Slack workspace.

## Solution Assessment

Is this the best way to solve the issue? No. The feature should keep fetched history as explicit untrusted context and preserve a clean command/directive input boundary instead of placing raw Slack history into `Body` and `BodyForAgent`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep recent history out of directive input:** `src/slack/monitor/slash.ts:583`
  - body: In context-aware mode this line makes raw channel history the prompt sent as `Body`/`BodyForAgent`, while `RawBody` and `CommandBody` stay empty. `resolveDirectiveCommandText` falls back to prompt text when command text is empty, so a prior Slack message containing inline directives such as `/model`, `/think`, or `/elevated` can change the invoker's slash request. Carry history as bounded untrusted context and keep directive parsing tied to the invoker's command text.
  - confidence: 0.9
