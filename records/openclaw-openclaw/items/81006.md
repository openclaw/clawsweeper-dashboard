## Summary

Keep open: this PR is not correct as-is. It skips `timeout` only inside the ACPX runtime wrapper, but current `main` can reject the auto-injected `timeout` option in the ACP manager before `AcpxRuntime.setConfigOption` is reached.

## What This Changes

The PR hoists the `timeout`/`timeout_seconds` skip in `AcpxRuntime.setConfigOption` out of the Codex-only branch and updates ACPX runtime tests for non-Codex timeout and mode forwarding.

## Best Possible Solution

Fix the ACP manager runtime-options/control application path so auto-injected timeout controls are skipped or fail-softed when unsupported, while preserving strict behavior for explicit user config-option writes.

## Reproduction Assessment

Yes. Current main source and related live reports show `timeoutSeconds` becomes a `timeout` config option and can be rejected before a Claude ACP turn starts when Claude advertises only `mode`, `model`, and `effort`.

## Solution Assessment

No. The proposed guard is downstream of current main's manager rejection path; the fix needs manager/runtime-options coverage plus a live Claude ACP proof after rework.
