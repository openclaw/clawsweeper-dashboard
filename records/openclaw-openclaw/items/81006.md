## Summary

Keep open: the PR addresses a real ACPX timeout-forwarding failure, but current main has already moved to a narrower Codex/Claude guard and this branch is now conflicting; the proposed universal timeout drop is broader than the current advertised-config-option contract.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` filtering so those keys are ignored for every ACP agent and updates ACPX runtime tests for non-Codex sessions.

## Best Possible Solution

Keep the fix capability-aware: skip timeout for known adapters that reject it or when the backend does not advertise it, while preserving advertised timeout aliases and landing through a rebased branch or current-main follow-up.

## Reproduction Assessment

Yes for the original shipped/base failure: source and PR logs show `timeout` reached Claude ACP and that Claude ACP rejects unknown config ids. Against current main, the exact Claude path has already been narrowed by a targeted guard, so I did not establish a fresh live failure there.

## Solution Assessment

No, not in its current broad form. The narrow Claude/Codex skip matches the observed dependency failure, but unconditionally dropping `timeout` for every ACPX agent bypasses the existing advertised-config-option contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve advertised timeout config options:** `extensions/acpx/src/runtime.ts:897-899`
  - body: This guard drops `timeout` and `timeout_seconds` even when the active ACP backend explicitly advertises one of those config ids.
