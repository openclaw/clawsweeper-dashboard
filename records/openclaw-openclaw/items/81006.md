## Summary

Keep open for maintainer handling, but this PR should not merge as written. The original Claude timeout failure has been superseded on current main by the merged advertisement-aware fix in https://github.com/openclaw/openclaw/pull/80812, while the remaining all-backend/OpenCode timeout rejection is tracked by https://github.com/openclaw/openclaw/issues/81250 and https://github.com/openclaw/openclaw/pull/81603; this branch is now conflicting and its unconditional ACPX timeout skip would regress advertised timeout support.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` suppression out of the Codex-only branch so all ACPX agents ignore those keys, and updates runtime tests for non-Codex behavior.

## Best Possible Solution

Keep current main's advertisement-aware timeout filtering, land or close the focused OpenCode follow-up as appropriate, and then resolve this conflicted branch together with its linked issue.

## Reproduction Assessment

Yes, source-level reproduction exists for the timeout-rejection class: current main can still emit `timeout` when adapter keys are unknown, and https://github.com/openclaw/openclaw/issues/81250 shows OpenCode rejecting it. The original Claude path is no longer a current-main failure because https://github.com/openclaw/openclaw/pull/80812 landed.

## Solution Assessment

No, this PR is no longer the best fix as written. The narrower current-main approach preserves advertised timeout support, while this PR's universal ACPX skip would make supported backend timeout controls ineffective.
