## Summary

Keep open: the bug is real, but this PR is not ready to merge because the new ACPX wrapper guard is too late for the current managed `sessions_spawn` path. Current main can build and reject the auto-injected `timeout` config option in manager runtime-control handling before `AcpxRuntime.setConfigOption` is called.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` filtering from the Codex-specific branch to all ACPX agents and updates ACPX runtime tests for timeout skipping and non-timeout forwarding.

## Best Possible Solution

Move the optional auto-injected timeout handling to the manager runtime-control construction/application boundary so unsupported backend-advertised key sets skip or fail-soft only that timeout mirror, while keeping explicit user `set_config_option` writes and permission/model controls strict.

## Reproduction Assessment

Yes. A source-level current-main path is clear: a Claude ACP session that advertises `mode`, `model`, and `effort` plus `runtimeOptions.timeoutSeconds` causes `applyManagerRuntimeControls` to build `timeout` and reject it before the PR's ACPX wrapper guard is reached.

## Solution Assessment

No. The wrapper no-op is reasonable as defense in depth, but it is not the best fix by itself because the managed spawn path can fail earlier in manager runtime-control handling; the narrower maintainable fix belongs at the runtime-option pair/application boundary.
