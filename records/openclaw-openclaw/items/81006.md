## Summary

Keep open: the PR has sufficient real behavior proof for the released Claude ACP failure, but its current universal timeout drop is broader than current main's capability-aware ACP config contract and should not merge as-is.

## What This Changes

The PR changes `AcpxRuntime.setConfigOption` to ignore `timeout` and `timeout_seconds` for every ACPX backend instead of only known unsupported commands.

## Best Possible Solution

Keep the capability-aware path: omit timeout when a backend does not advertise it or is a known stale unsupported adapter, while preserving `timeout` and `timeout_seconds` for adapters that explicitly advertise them.

## Reproduction Assessment

Yes for the released bug path: the PR body shows real OpenClaw 2026.5.7 Claude ACP before/after logs, and dependency source confirms Claude ACP rejects unknown config IDs. Current main has since moved to a narrower capability-aware fix, so the remaining review question is the PR diff, not whether the original report was real.

## Solution Assessment

No. The branch fixes the observed Claude failure, but suppressing timeout for every ACPX backend is broader than the current-main advertised-key contract; keeping the guard capability-aware is the narrower maintainable solution.
