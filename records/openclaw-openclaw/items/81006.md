## Summary

Keep open: the PR addresses a real ACPX Claude failure and includes sufficient real log proof, but current main has since moved this area toward a capability-aware timeout contract and the PR branch still drops timeout config for every ACPX backend.

## What This Changes

The branch hoists ACPX `timeout`/`timeout_seconds` filtering out of the Codex-only path and updates ACPX runtime tests for non-Codex config forwarding.

## Best Possible Solution

Keep the capability-aware path: omit timeout for adapters that do not advertise it, preserve it for adapters that explicitly advertise it, and keep narrow known-adapter guards for stale records where needed.

## Reproduction Assessment

Yes. Source inspection plus the PR's before/after logs give a high-confidence path: the released ACPX path forwarded timeout to Claude ACP, and Claude ACP rejects unknown config ids.

## Solution Assessment

No. The branch fixes the observed Claude failure, but the universal early return is broader than the current-main contract that preserves timeout when an adapter explicitly advertises support.
