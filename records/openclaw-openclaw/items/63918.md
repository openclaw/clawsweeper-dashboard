## Summary

Current main and shipped releases no longer have the reported cron-to-OpenAI `thinking=none` path: cron preserves explicit `payload.thinking`, forwards it into the embedded run, and OpenAI GPT-5 reasoning normalization resolves explicit `minimal` to a supported effort.

## What This Changes

The issue reports isolated cron `agentTurn` jobs against `openai/gpt-5-nano` failing because explicit `payload.thinking=minimal` was sent to OpenAI as unsupported `none`.

## Best Possible Solution

Keep the shipped cron forwarding and provider-normalization path; reopen only with a fresh current-version cron payload showing `minimal` still becomes `none`.

## Reproduction Assessment

No. The old failure path is clear from the report, but current main preserves explicit cron thinking and resolves GPT-5 reasoning efforts without sending unsupported `none`.

## Solution Assessment

Yes. The maintainable fix is the shared provider-effort normalization plus cron forwarding, not a cron-only config workaround or model-specific special case.
