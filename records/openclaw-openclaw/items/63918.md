## Summary

Close: current main preserves explicit cron `payload.thinking`, forwards the resolved thinking level into the cron runner, and the OpenAI transport no longer falls back to `reasoning.effort: none`; the fix is included in v2026.4.9 and later releases.

## What This Changes

The issue reports isolated cron `agentTurn` jobs for `openai/gpt-5-nano` sending unsupported `thinking=none` despite `payload.thinking=minimal`, causing an OpenAI 400.

## Best Possible Solution

Keep the shipped v2026.4.9+ OpenAI reasoning/default handling and close this issue as implemented; open a new report only if the same payload fails on a current release.

## Reproduction Assessment

Yes for the original shipped path: v2026.4.8 source sent `reasoning.effort: "none"` when no OpenAI reasoning option reached the transport, and the reporter confirmed v2026.4.9 resolved the same cron payload. Current main does not show a remaining failure path.

## Solution Assessment

Yes. The shipped fix keeps explicit cron thinking forwarding and changes the OpenAI fallback away from unsupported disabled reasoning, which is the narrowest maintainable solution without new config or product policy.
