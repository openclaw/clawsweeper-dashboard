## Summary

Close: current main and the shipped v2026.4.9 release preserve cron payload thinking and the OpenAI transport no longer falls through to the unsupported disabled reasoning value for this path; the reporter also confirmed v2026.4.9 resolved the same failing cron jobs.

## What This Changes

## Best Possible Solution

Keep the v2026.4.9+ OpenAI reasoning handling and close this issue; a new report should include a current release version, full redacted cron job payload, and redacted OpenAI 400 if this reappears.

## Reproduction Assessment

No high-confidence current-main failing repro remains. I did not run a live OpenAI call, but source inspection plus shipped tests show cron payload.thinking is forwarded and the OpenAI transport no longer maps this path to unsupported none; the reporter confirmed v2026.4.9 resolved the same jobs.

## Solution Assessment

Yes. The durable fix is at the existing cron/provider boundary: cron preserves the explicit thinking value, the harness forwards it as shared reasoning, and the OpenAI transport honors it without adding a new config knob.
