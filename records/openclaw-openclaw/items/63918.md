## Summary

Current main and shipped releases already solve the reported cron OpenAI thinking failure: cron preserves `payload.thinking`, forwards the resolved `thinkLevel`, and OpenAI Responses request shaping no longer forces unsupported `none` for enabled reasoning; the reporter also confirmed the same payloads completed after upgrading to v2026.4.9.

## What This Changes

The issue reports cron `agentTurn` jobs sending OpenAI `thinking=none` despite `payload.thinking=minimal`, causing `gpt-5-nano` 400 failures.

## Best Possible Solution

Keep the shipped OpenAI reasoning-effort implementation and close this issue; any new failure on current releases should come in as a fresh report with current version, cron payload, and provider debug logs.

## Reproduction Assessment

No, not against current main: source inspection shows the cron `payload.thinking` value is preserved and the OpenAI transport no longer forces unsupported `none`. The historical v2026.4.8 failure is source-proven by the tag diff and reporter-confirmed fixed after v2026.4.9.

## Solution Assessment

Yes; fixing the shared OpenAI Responses reasoning-effort path and preserving cron `thinkLevel` is the maintainable solution, rather than adding a cron-specific config override or model-specific workaround.
