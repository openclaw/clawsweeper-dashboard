## Summary

Current main and the v2026.4.9 release already cover the reported cron OpenAI reasoning failure: cron preserves the explicit payload thinking level, and the OpenAI transport no longer falls back to unsupported disabled reasoning for GPT-5 reasoning models.

## What This Changes

The issue reports cron agentTurn requests for openai/gpt-5-nano failing because the OpenAI payload used unsupported thinking none despite payload.thinking being minimal.

## Best Possible Solution

Keep the shared OpenAI reasoning request shaping from v2026.4.9 and close this reopened issue; any new failure should be filed with a current-release payload/log showing reasoning none still being emitted.

## Reproduction Assessment

Yes for the historical shipped bug: the v2026.4.8 source path emitted reasoning effort none for reasoning models without explicit options, and the reporter confirmed the same cron jobs succeeded after v2026.4.9. I did not run a live OpenAI call in this read-only review.

## Solution Assessment

Yes. The fix belongs in the shared OpenAI request-shaping path while cron continues forwarding the resolved thinkLevel, avoiding a cron-specific workaround or new config surface.
