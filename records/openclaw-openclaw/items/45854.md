Keep open. Current main implements the linked call log, SMS search, and per-device Android session-isolation pieces, but the remaining device-control, ADB-style, and dynamic Kotlin execution capabilities are still absent and need product/security design before implementation.

## What This Changes

The issue requests an Android node roadmap covering device control, ADB-style tooling, dynamic Kotlin execution, and already-merged call log, SMS search, and per-device session isolation.

## Best Possible Solution

Split the roadmap into narrower proposals or PRs: Android device control with explicit permission UX, ADB/node execution aligned to exec approvals and node policy, and dynamic Kotlin execution only after a security-reviewed sandbox design exists.

Concerns:

- **[high] Dynamic Android code execution needs a sandbox design:**
  - body: Distributing and executing Kotlin/script code on an Android node would let an agent run new code on the device, so it needs a trust model, permission boundary, audit trail, and rollback story before implementation.
  - confidence: 0.92
- **[medium] ADB-style tools need exec-policy alignment:** `docs/nodes/index.md:56`
