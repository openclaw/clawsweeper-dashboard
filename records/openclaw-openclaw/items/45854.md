Status: needs_attention

Summary: The issue is not a patch, but the remaining ADB-style and dynamic Kotlin execution requests are security-sensitive and need explicit design before implementation.

Concerns:

- **[high] Design a sandbox before dynamic Android code execution:**
  - body: Distributing Kotlin or script code to an Android node would let an agent run new code on a user device, so this needs a trust model, permission boundary, audit trail, and rollback story before implementation.
  - confidence: 0.9
- **[medium] Align ADB-style access with node command approvals:** `src/gateway/node-command-policy.ts:63`
  - body: ADB-style file, process, app, log, and device access can expose broad user data and device state, so it should be designed around command allowlists, pairing scopes, and approval semantics rather than added as an ordinary Android command.
