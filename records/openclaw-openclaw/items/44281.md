Keep open: the PR targets a source-reproducible Telegram media/reply-tag bug, but the branch is not merge-ready because it broadens whitespace normalization, still needs a current Telegram plugin port, and lacks required real Telegram proof.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, Telegram reply delivery wiring, and Telegram tests so inline MEDIA directives and reply tags are normalized before reply sends.

## Best Possible Solution

Land a narrow current-main Telegram plugin fix that strips only reply tags before media extraction, preserves ordinary whitespace, updates focused outbound and Telegram coverage, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows a same-line `[[reply_to:42]] MEDIA:./photo.jpg` token is parsed for media before reply tags are stripped, and media extraction only recognizes line-start `MEDIA:` tokens.

## Solution Assessment

No. The parser-order goal is valid, but this branch should be narrowed to strip only reply tags before media extraction, preserve whitespace, be ported to the current Telegram plugin path, and include real Telegram proof.
