## Summary

Keep open: this targets a real Telegram reply-media gap that still appears source-reproducible on current main, but the proposed patch is not merge-ready because it introduces whitespace normalization for ordinary replies and has only mocked/unit validation rather than real Telegram proof.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, Telegram reply delivery/dispatch, and Telegram tests so inline MEDIA directives and reply tags are normalized before Telegram sends.

## Best Possible Solution

Adapt the fix to current main’s extensions/telegram delivery and shared outbound payload plan, strip reply tags before MEDIA extraction without normalizing tag-free text, add focused regression coverage, and provide redacted real Telegram proof.

## Reproduction Assessment

Yes, from source inspection: current main parses MEDIA before stripping reply tags, while media extraction only recognizes lines beginning with MEDIA, so a same-line reply tag before MEDIA remains ordinary text. I did not run tests because this was a read-only review.

## Solution Assessment

No. The current PR direction fixes the ordering issue too broadly by normalizing all raw reply text; the safer solution is to strip only reply tags or otherwise preserve existing tag-free formatting before media extraction.

## Review Findings

Overall correctness: patch is incorrect
