## Summary

Keep open: the Telegram reply-media bug still appears valid on current main, but this PR should not merge as-is because it lacks real Telegram proof, is stale against the current extension path, and its parser change can rewrite ordinary reply whitespace.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, Telegram delivery wiring, native-command delivery options, and Telegram tests so inline MEDIA directives and reply tags are normalized before Telegram sends replies.

## Best Possible Solution

Port a narrow current-main fix that strips only reply tags before media extraction, preserves tag-free whitespace, updates the extensions/telegram path and focused tests, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows current main parses MEDIA before reply tags are stripped, and MEDIA extraction only recognizes lines whose trimmed start begins with MEDIA:, so a same-line reply tag plus MEDIA directive is kept as text; I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The parser-order goal is valid, but this branch should be narrowed to strip reply tags before media extraction without normalizing tag-free text and should be ported to the current extensions/telegram delivery path.
