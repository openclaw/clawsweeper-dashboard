## Summary

Keep open: this targets a real Telegram media/reply-directive gap, but the branch is stale against current main, the parser change would corrupt whitespace across `parseReplyDirectives()` callers, and the PR still lacks required real Telegram proof.

## What This Changes

The PR changes reply directive parsing and Telegram reply delivery so inline `MEDIA:` and reply tags are normalized before sending, with tests for media delivery and reply threading.

## Best Possible Solution

Refresh the PR onto current main, fix `parseReplyDirectives()` with a whitespace-preserving pre-media reply-tag strip, keep Telegram delivery on the shared outbound plan, and attach real Telegram proof for media plus reply threading.

## Reproduction Assessment

Yes, source-level. Current main splits `MEDIA:` before stripping reply tags, and the media parser only extracts lines whose trimmed start is `MEDIA:`, so a same-line `[[reply_to:42]] MEDIA:...` payload remains text.

## Solution Assessment

No. The PR fixes the same-line parsing direction, but doing it by running `parseInlineDirectives(raw)` on every payload normalizes ordinary whitespace; the safer fix is a narrow, whitespace-preserving reply-tag strip before media extraction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve whitespace when stripping reply tags before media parsing:** `src/auto-reply/reply/reply-directives.ts:20`
  - body: `parseInlineDirectives(raw, ...)` always runs `normalizeDirectiveWhitespace`, even when there is no reply tag, and this result is then fed into media parsing. Any caller of `parseReplyDirectives()` with aligned prose, multiple spaces, or extra blank lines can lose user-visible formatting; strip reply tags before media extraction without reformatting text that does not need tag removal.
