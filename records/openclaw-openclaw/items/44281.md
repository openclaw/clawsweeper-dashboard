## Summary

Keep open: the underlying Telegram reply-media bug is still source-reproducible on current main, but this PR is not merge-ready because it targets the removed legacy Telegram path, broadens parser whitespace normalization, is merge-conflicting, and lacks the live Telegram proof required for reply-context behavior.

## What This Changes

The branch changes reply directive parsing, outbound payload normalization, legacy Telegram dispatch/native-command delivery plumbing, and Telegram delivery tests to normalize inline `MEDIA:` and reply tags before send.

## Best Possible Solution

Replace the branch with a current-main fix in the shared reply parser and `extensions/telegram` delivery path that strips reply tags only for media detection, preserves reply text whitespace, threads current message context where needed, and includes focused regression coverage plus live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: on current main, `[[reply_to:42]] MEDIA:./photo.jpg` is parsed before reply-tag stripping, while the media parser only extracts `MEDIA:` when it starts the trimmed line.

## Solution Assessment

No. The PR targets the right bug, but the current patch changes the wrong Telegram tree for current main and broadens parser whitespace normalization beyond the narrow media-detection fix.

## Review Findings

  - confidence: 0.9

  - confidence: 0.88
