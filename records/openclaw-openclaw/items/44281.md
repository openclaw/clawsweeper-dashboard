## Summary

Keep open: current main still has the same source-level Telegram MEDIA/reply-tag parsing gap, but this PR should not merge as-is because it introduces a whitespace-normalization regression, lacks real Telegram proof, and needs adaptation to the current Telegram plugin path.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, Telegram reply dispatch/native command delivery, and Telegram tests so inline MEDIA directives and reply tags are normalized before Telegram replies are sent.

## Best Possible Solution

Adapt only the parser-ordering fix to current main by stripping reply tags before MEDIA extraction while preserving tag-free whitespace, keep the existing extensions/telegram payload-plan path, add focused regression coverage, and attach redacted Telegram proof.

## Reproduction Assessment

Yes, from source inspection. Current main parses MEDIA before stripping reply tags, and MEDIA extraction only recognizes line-leading tokens, so a payload like [[reply_to:42]] MEDIA:./photo.jpg remains text instead of media.

## Solution Assessment

No. The PR fixes the ordering issue too broadly by normalizing all raw reply text; the safer fix is to strip only reply tags, or keep a tag-free fast path, before media extraction and port the change to extensions/telegram.
