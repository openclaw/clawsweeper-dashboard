## Summary

Keep open: the parser-order bug still appears valid on current main, but this PR is not merge-ready because it is merge-conflicting, lacks real Telegram proof, still edits the removed pre-plugin Telegram path, and introduces a whitespace regression in shared reply parsing.

## What This Changes

The PR changes shared reply directive parsing, outbound reply normalization, legacy Telegram delivery wiring, and Telegram delivery tests for inline `MEDIA:` directives combined with reply tags.

## Best Possible Solution

Port a narrow parser-order fix to current main that strips only reply tags before media extraction, preserves tag-free whitespace, covers the current `extensions/telegram` path, and includes redacted real Telegram proof.

## Reproduction Assessment

Yes, from source inspection: current main parses media before removing reply tags, while media extraction only recognizes lines whose trimmed start is `MEDIA:`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The intended parser-order fix is valid, but this PR applies it through unconditional whitespace normalization and stale legacy Telegram files; the safer solution is a narrower current-plugin patch with real Telegram proof.
