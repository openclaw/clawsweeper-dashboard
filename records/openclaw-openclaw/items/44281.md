## Summary

Keep open: the reported Telegram media/reply-tag gap is still source-reproducible on current main, but this PR is not ready because it lacks real Telegram proof, is stale against the current plugin path, and introduces a whitespace regression in shared reply parsing.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, Telegram reply delivery wiring, and Telegram delivery tests so inline `MEDIA:` directives and reply tags are normalized before Telegram sends.

## Best Possible Solution

Port a narrow current-main fix that strips only reply tags before media extraction, preserves tag-free whitespace, covers `extensions/telegram`, and includes redacted real Telegram proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `[[reply_to:42]] MEDIA:./photo.jpg` reaches media extraction before reply tags are stripped, and extraction only recognizes lines that start with `MEDIA:` after trimming. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The parser-order goal is valid, but this branch applies it through unconditional whitespace normalization and stale legacy Telegram wiring; a narrower patch on the current plugin path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve whitespace when stripping reply tags:** `src/auto-reply/reply/reply-directives.ts:20`
