## Summary

Keep open: the underlying Telegram MEDIA-plus-reply-tag bug is still source-reproducible on current main, but this PR should not merge as-is because it is conflicting, edits the old `src/telegram` path, introduces a whitespace regression, and lacks required real Telegram proof.

## What This Changes

The PR updates reply directive parsing, outbound payload normalization, Telegram reply delivery wiring, and delivery tests for inline MEDIA directives combined with reply tags.

## Best Possible Solution

Patch current main by stripping reply tags before media extraction without normalizing tag-free text, keep the fix in the current `extensions/telegram` plugin path, add focused regression tests, and attach redacted Telegram proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main parses MEDIA before removing reply tags, while media extraction only recognizes line-leading `MEDIA:` tokens. A Telegram reply payload like `[[reply_to:42]] MEDIA:./photo.jpg` will therefore remain text instead of media.

## Solution Assessment

No. The useful parser-ordering fix should be narrowed to strip reply tags before media extraction while preserving tag-free whitespace, and the Telegram wiring should be adapted to the current `extensions/telegram` implementation.

## Review Findings
