## Summary

Keep this PR open for now: the reported Telegram MEDIA/reply-tag bug is still source-reproducible on current main, but this branch is not merge-ready because it lacks real Telegram proof and introduces a whitespace-normalization regression in the parser path.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, legacy Telegram delivery wiring, and Telegram tests so inline MEDIA directives and reply tags are normalized before Telegram sends.

## Best Possible Solution

Port a narrow current-main fix that strips only reply tags before media extraction, preserves tag-free whitespace, updates the current extensions/telegram path and focused tests, and includes redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a same-line `[[reply_to:42]] MEDIA:./photo.jpg` reaches media extraction before reply tags are stripped, while media extraction only recognizes lines whose trimmed start is `MEDIA:`; I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The parser-order goal is valid, but this branch applies it through broad inline directive parsing before media extraction and stale legacy Telegram wiring; a smaller current-main patch that strips only reply tags first is safer.
