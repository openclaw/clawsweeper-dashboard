## Summary

Keep open. The underlying Telegram MEDIA plus reply-tag bug still appears present on current main, but this PR is not merge-ready because the parser-order change can corrupt intentional whitespace and the Telegram hunks target the old pre-plugin path instead of the current plugin layout.

## What This Changes

The PR changes shared reply directive parsing, outbound payload normalization, and Telegram reply delivery/tests so inline MEDIA directives and reply tags are normalized before Telegram sends replies.

## Best Possible Solution

Implement a narrow current-main fix that strips reply tags only for MEDIA detection while preserving ordinary reply text, code-block whitespace, and existing MEDIA line-start semantics; then add parser/outbound and Telegram delivery regressions for same-line `[[reply_to:<id>]] MEDIA:...`, with scoped current-message threading only if it fits the current Telegram plugin flow.

## Reproduction Assessment

Yes. The source trace is enough: on current main, `[[reply_to:42]] MEDIA:./photo.jpg` reaches media extraction before reply-tag stripping, and the media parser only extracts lines whose trimmed start is `MEDIA:`.

## Solution Assessment

No. The requested direction is valid, but this PR’s implementation is not the best fix because it pre-parses the whole raw text and can normalize unrelated whitespace; a smaller tag-stripping step before MEDIA detection is safer.
