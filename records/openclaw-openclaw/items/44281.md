## Summary

Keep open. The underlying Telegram reply-media bug is still present on current main, but this branch should not merge as-is because it normalizes full reply whitespace before media parsing and still targets the old `src/telegram` layout instead of the current Telegram plugin path.

## What This Changes

The branch changes shared reply directive parsing, outbound reply normalization, legacy Telegram dispatch/native-command delivery plumbing, and Telegram delivery tests so inline `MEDIA:` and reply tags are normalized before Telegram sends replies.

## Best Possible Solution

Implement a current-main repair that strips reply tags only for MEDIA detection, threads the current message id through the outbound/Telegram normalization path where needed, ports coverage to `extensions/telegram`, and adds the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: `[[reply_to:42]] MEDIA:./photo.jpg` reaches `splitMediaFromOutput` before reply-tag stripping, and that parser only extracts media when the trimmed line starts with `MEDIA:`.

## Solution Assessment

No. The PR is aimed at the right bug, but parsing the full raw reply through `parseInlineDirectives` before media extraction is broader than needed because it normalizes unrelated whitespace, and the branch no longer targets the current Telegram plugin files.
