## Summary

Keep open. The PR targets a real Telegram reply-media bug, but the branch is not correct against current main: it changes the legacy `src/telegram` path while current Telegram delivery lives under the plugin path, and its parser change can corrupt intentional reply whitespace.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, legacy Telegram dispatch/native-command delivery plumbing, and Telegram delivery tests to normalize inline `MEDIA:` and reply tags before Telegram sends replies.

## Best Possible Solution

Implement a current-main replacement that strips reply tags only for media detection, keeps reply text whitespace stable, wires any needed current-message-id context through `extensions/telegram`, adds focused regression coverage, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: `[[reply_to:42]] MEDIA:./photo.jpg` is parsed before reply-tag stripping, and the media parser only extracts `MEDIA:` when it starts the trimmed line.

## Solution Assessment

No. The PR is aimed at the right bug, but the current patch is broader than necessary in the parser and targets legacy Telegram files instead of the current plugin implementation.
