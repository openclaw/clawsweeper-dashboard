## Summary

Keep open: the underlying Telegram MEDIA plus reply-tag bug is still source-reproducible on current main, but this PR is not merge-ready because it lacks real Telegram proof and the parser change can corrupt whitespace in ordinary replies.

## What This Changes

The PR normalizes Telegram reply payloads before delivery and changes reply directive parsing so reply tags are removed before MEDIA extraction.

## Best Possible Solution

Keep the shared outbound payload plan, add a narrow reply-tag stripping step before media extraction that preserves tag-free text exactly, update current extensions/telegram coverage, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows current main splits MEDIA directives before stripping reply tags, while MEDIA extraction only recognizes line-leading MEDIA tokens, so [[reply_to:42]] MEDIA:./photo.jpg remains text.

## Solution Assessment

No. The intended parser-order fix is valid, but this implementation is too broad because it normalizes all raw reply text and also needs adaptation to the current extensions/telegram delivery path.
