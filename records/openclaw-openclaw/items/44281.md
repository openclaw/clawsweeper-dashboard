## Summary

Keep this PR open, but it needs changes before merge. Current main still has the same-line reply-tag plus MEDIA parsing gap, while this branch is stale against the current Telegram plugin layout and its parser change would corrupt ordinary reply whitespace before delivery.

## What This Changes

The PR changes shared reply directive parsing, outbound reply normalization, and Telegram reply delivery/tests so inline MEDIA directives and reply tags are normalized before Telegram sends replies.

## Best Possible Solution

Implement a current-main fix that strips only reply tags from a MEDIA-detection copy before calling splitMediaFromOutput, preserves ordinary reply text and code-block whitespace, ports any needed Telegram delivery tests to extensions/telegram, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current source gives a high-confidence reproduction: a reply such as [[reply_to:42]] MEDIA:./photo.jpg reaches media extraction before the reply tag is stripped, and the media parser ignores lines that do not trim-start with MEDIA:.

## Solution Assessment

No. The PR is pointed at the right bug, but pre-parsing the full raw reply with parseInlineDirectives before media extraction is not the narrowest maintainable fix because that helper normalizes unrelated whitespace.
