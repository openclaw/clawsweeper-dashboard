## Summary

Keep this PR open, but it is not mergeable as-is: current main still has the source-reproducible same-line reply-tag/MEDIA gap, while this branch is conflicting, lacks real Telegram proof, and introduces a whitespace-normalization regression in the shared parser.

## What This Changes

The branch modifies shared reply directive parsing, outbound payload normalization, Telegram reply delivery/dispatch, and Telegram tests to normalize inline MEDIA directives and reply tags before Telegram sends replies.

## Best Possible Solution

Adapt the parser fix to current main by stripping only reply tags before MEDIA extraction without normalizing tag-free text, keep the current extensions/telegram payload-plan path, add focused regression coverage, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes. From source inspection, current main parses MEDIA before stripping reply tags, while media extraction only recognizes lines beginning with MEDIA:, so [[reply_to:42]] MEDIA:./photo.jpg remains ordinary text instead of media.

## Solution Assessment

No. The PR fixes the ordering issue too broadly by normalizing all raw reply text; the safer solution is to strip only reply tags, or otherwise preserve tag-free formatting, before media extraction and adapt the change to current extensions/telegram code.
