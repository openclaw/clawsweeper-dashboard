## Summary

Keep open: the same-line reply-tag plus MEDIA case is still source-reproducible on current main, but this PR is stale/conflicting, lacks required live Telegram proof, and its parser change can rewrite ordinary reply whitespace.

## What This Changes

The PR changes reply directive parsing, outbound normalization, Telegram reply delivery wiring, native command/message dispatch options, and Telegram tests so inline MEDIA directives and reply tags are normalized before Telegram sends replies.

## Best Possible Solution

Port a narrow current-main fix that strips only reply tags before media extraction, preserves tag-free whitespace, updates focused extensions/telegram coverage, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows parseReplyDirectives() still parses MEDIA before stripping reply tags, and MEDIA extraction only recognizes lines whose trimmed start begins with MEDIA:, so a same-line reply tag plus MEDIA directive remains text on current main.

## Solution Assessment

No. The parser-order goal is valid, but this branch should be narrowed to strip only reply tags before media extraction, preserve tag-free whitespace, and be ported to the current extensions/telegram path.
