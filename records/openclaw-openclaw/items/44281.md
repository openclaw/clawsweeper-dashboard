## Summary

Keep open: the same-line reply-tag/MEDIA bug is still source-reproducible on current main, but this PR is not merge-ready because it lacks required real Telegram proof, conflicts with current main, still targets removed legacy Telegram paths, and introduces a whitespace regression.

## What This Changes

This PR changes reply directive parsing, outbound payload normalization, Telegram reply delivery wiring, native-command delivery options, and Telegram tests so inline MEDIA directives and reply tags are normalized before sending replies.

## Best Possible Solution

Port a narrow current-main fix that strips only reply tags before media extraction, preserves tag-free whitespace, updates the current extensions/telegram path and focused tests, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows parseReplyDirectives() calls splitMediaFromOutput(raw) before reply tags are stripped, while splitMediaFromOutput() only extracts lines whose trimmed start begins with MEDIA:, so [[reply_to:42]] MEDIA:./photo.jpg is treated as text on current main; I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. The parser-order goal is valid, but this branch applies it through broad inline directive parsing before media extraction and stale legacy Telegram wiring; a smaller current-main patch that strips only reply tags first is safer.
