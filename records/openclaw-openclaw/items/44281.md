## Summary

Keep open: the underlying same-line reply-tag plus MEDIA delivery bug is still source-reproducible on current main, but this PR is not merge-ready because the proposed parser-order change would normalize user-visible whitespace before delivery and the Telegram files have since moved under the plugin path.

## What This Changes

The PR strips reply tags before MEDIA extraction, normalizes Telegram reply payloads before delivery, threads current message IDs through Telegram reply delivery, and adds Telegram/directive regression tests.

## Best Possible Solution

Port the parser-order fix to current main by stripping reply tags before MEDIA detection without applying broad whitespace normalization, keep Telegram on the shared outbound payload plan, and add regressions for same-line reply-tag plus MEDIA and whitespace preservation.

## Reproduction Assessment

Yes by source inspection: current main parses MEDIA before stripping reply tags, and the media parser only extracts lines whose trimmed start is MEDIA:. A payload like '[[reply_to:42]] MEDIA:./photo.jpg' therefore remains a plain text line unless the parser order is fixed.

## Solution Assessment

No, not as written. Parsing reply tags before media extraction is the right direction, but this diff must preserve ordinary message whitespace and be ported to the current extensions/telegram delivery path before it is mergeable.
