## Summary

Keep open: the Telegram media-reply bug is still source-reproducible on current main, but this PR is not merge-ready because it lacks real Telegram proof, is stale against the extensions/telegram layout, and introduces a whitespace regression in directive parsing.

## What This Changes

The PR changes reply directive parsing, outbound payload normalization, Telegram dispatch wiring, and Telegram delivery tests so inline MEDIA directives and reply tags are normalized before Telegram sends replies.

## Best Possible Solution

Land a narrow current-main fix that strips only reply tags before media extraction, preserves tag-free whitespace, updates focused extensions/telegram coverage, and includes redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows [[reply_to:42]] MEDIA:./photo.jpg reaches splitMediaFromOutput() before reply tags are stripped, and splitMediaFromOutput() only extracts line-start MEDIA tokens.

## Solution Assessment

No. The parser-order goal is valid, but this branch should be narrowed to strip only reply tags before media extraction, preserve whitespace, be ported to extensions/telegram, and include real Telegram proof.
