## Label Justifications

- `P1`: The report describes a production WhatsApp channel workflow where inbound delivery can remain frozen for hours while the gateway appears connected.
- `impact:message-loss`: The central impact is inbound WhatsApp messages not reaching OpenClaw despite transport-level health signals.
- `impact:session-state`: The reported zombie state is a mismatch between connection/session health and actual inbound application delivery.

## Summary

Keep open: current main and v2026.6.6 still make WhatsApp recovery decisions from passive transport/app timestamps, while the requested active inbound-path verification remains unimplemented and needs maintainer design plus live WhatsApp proof.

## What This Changes

Request to add opt-in active WhatsApp liveness verification that performs a server round trip during idle or reconnect recovery before trusting the inbound path.

## Best Possible Solution

Keep this open for a maintainer-approved WhatsApp-plugin design: choose a low-side-effect Baileys probe, add opt-in config only if approved, feed failures into the existing reconnect/backoff/status path, and prove it with live WhatsApp recovery evidence.

## Reproduction Assessment

Yes at source level: current main shows passive timestamp/frame-based liveness and no active probe path after reconnect. I did not establish the reported multi-hour WhatsApp zombie state in a live environment during this read-only review.

## Solution Assessment

Unclear: active verification is a plausible direction, but the best fix depends on maintainers choosing the WhatsApp probe primitive, privacy behavior, config contract, and relationship to the existing reconnect trackers.
