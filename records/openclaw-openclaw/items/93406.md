## Summary

Keep open: current main and v2026.6.9 still make WhatsApp recovery decisions from passive WebSocket frame and message timestamps, while the requested active inbound-path verification is not implemented or owned by a merged related PR.

## What This Changes

The issue requests opt-in active WhatsApp liveness probes that verify inbound delivery after idle windows or reconnects instead of trusting passive transport and message timestamps alone.

## Best Possible Solution

Have maintainers choose a WhatsApp-plugin liveness design, then implement opt-in active probes inside the existing controller/reconnect/status path with dependency-backed probe semantics, config/docs/tests, and live WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main uses passive WebSocket frame and app-message timestamps, and reconnect starts a fresh passive watchdog window. I did not run a live multi-hour WhatsApp zombie-state reproduction in this read-only review.
