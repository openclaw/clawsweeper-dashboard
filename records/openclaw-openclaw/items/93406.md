## Summary

Keep open: current main still uses passive WhatsApp transport/app-silence signals and gives reconnects a fresh passive window without active inbound-path verification; related passive tuning, hook, and backfill work does not own this active-liveness product decision.

## What This Changes

The issue requests opt-in active WhatsApp liveness probes so reconnects prove inbound delivery recovered instead of trusting transport reconnect alone.

## Best Possible Solution

Have maintainers choose one WhatsApp-plugin recovery design, then implement opt-in active liveness verification in the owned controller/reconnect/status path with config/docs/tests and live WhatsApp proof.

## Reproduction Assessment

Yes for the source-level gap: current main shows passive timestamp/frame-based liveness and no active probe after reconnect. I did not establish a live current-main reproduction of the reported multi-hour WhatsApp zombie state.

## Solution Assessment

Unclear: active verification is a plausible owner-plugin direction, but the best fix depends on maintainers choosing the probe primitive, privacy behavior, config contract, and relationship to existing watchdog/backfill work.
