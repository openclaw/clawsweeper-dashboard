## Summary

Keep open: current main and latest v2026.6.9 still rely on passive WhatsApp Web timestamps and frame activity, while the requested opt-in active server-roundtrip verification remains unimplemented and needs a maintainer product/API decision.

## What This Changes

Request an opt-in WhatsApp active liveness probe that confirms inbound-path recovery during idle and reconnect windows before trusting a recovered connection.

## Best Possible Solution

Decide the owner boundary first, then implement a narrowly scoped opt-in WhatsApp liveness probe in the channel owner layer with bounded probe frequency, reconnect integration, docs/config updates, regression tests, and live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main shows passive timestamp/frame-based liveness and no active recovery probe. I did not run a live WhatsApp zombie-state reproduction.

## Solution Assessment

Unclear: active probing is a plausible fix, but the best primitive, cadence, and owner boundary need maintainer confirmation and live Baileys proof before a PR should be queued.
