## Summary

Keep open: current main and v2026.6.6 still rely on passive WhatsApp transport/app timestamps, and no related open or merged item safely owns the requested active inbound-path liveness verification.

## What This Changes

The issue requests opt-in active WhatsApp liveness probes that verify the inbound path after idle periods and reconnects instead of relying only on passive transport and inbound timestamps.

## Best Possible Solution

Keep this open for a maintainer-approved WhatsApp-plugin liveness design that validates a low-side-effect probe primitive, keeps the default compatible, and feeds failures into the existing reconnect/backoff/status path with live WhatsApp proof.
