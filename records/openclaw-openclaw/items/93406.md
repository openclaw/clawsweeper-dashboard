## Summary

Keep open: current main and v2026.6.8 still use passive WhatsApp transport/app-silence signals and do not actively verify that inbound delivery recovered after reconnect; the related watchdog-tuning and health-hook items are complementary rather than a replacement.

## What This Changes

The issue requests an opt-in WhatsApp liveness probe that performs a server round trip during idle/reconnect windows and feeds failures into the existing reconnect path.

## Best Possible Solution

Adopt a maintainer-approved WhatsApp-plugin design that uses a low-side-effect active probe, defaults off if configured, feeds failures into the existing controller reconnect/backoff/status path, and includes focused tests plus live WhatsApp proof.

## Reproduction Assessment

Yes at source level, but not as a live current-main reproduction. Current main shows passive timestamp/frame-based liveness and no active probe path after reconnect; the reported multi-hour zombie state still needs live WhatsApp proof.

## Solution Assessment

Unclear: active verification is a plausible direction, but the best fix depends on maintainers choosing the WhatsApp probe primitive, privacy behavior, config contract, and relationship to passive watchdog tuning.
