## Summary

Keep open: current main and latest release still use passive WhatsApp transport/app-silence watchdog signals, while the requested active inbound-path verification is not implemented and is not replaced by the open passive-watchdog tuning or hook-observability work.

## What This Changes

Request opt-in WhatsApp connection-liveness probes that confirm the inbound path recovered after reconnects and distinguish quiet accounts from dead inbound delivery.

## Best Possible Solution

Have maintainers choose one WhatsApp-plugin liveness/recovery design, then implement active verification in the existing controller/monitor/status path with a privacy-safe probe primitive, config/docs/tests, and live WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main shows passive timestamp/frame-based liveness and no active proof that reconnect restored inbound delivery. I did not establish the reported multi-hour WhatsApp zombie state in a live environment during this read-only review.

## Solution Assessment

Unclear: active verification is a plausible owner-boundary fix, but the best solution depends on maintainer choices about the probe primitive, privacy behavior, config contract, and relationship to existing watchdog tuning and health/status work.
