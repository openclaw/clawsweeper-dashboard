## Summary

Keep open: current main and v2026.6.9 still rely on passive WhatsApp watchdog signals, while the requested active inbound-path verification remains a product/API decision and is not superseded by the passive tuning or hook-event work.

## What This Changes

Requests opt-in active WhatsApp connection-liveness probes that verify the inbound path after idle windows and reconnects before treating the connection as recovered.

## Best Possible Solution

Have maintainers choose a WhatsApp-plugin liveness design, then implement opt-in active probes in the existing controller/reconnect/status path with a validated Baileys round-trip primitive, config/docs/tests, and live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main shows passive timestamp/frame-based liveness and no active probe after reconnect. I did not establish the reported multi-hour WhatsApp zombie state in a live environment during this read-only review.

## Solution Assessment

Unclear: active verification is a plausible direction, but the best fix depends on maintainers choosing the probe primitive, privacy behavior, config contract, and relationship to passive watchdog tuning and health hooks.
