## Summary

Keep open: current main and v2026.6.9 still use passive WhatsApp timestamp/frame watchdogs and fresh reconnect windows; the related watchdog-tuning and health-hook work does not implement active inbound-path verification.

## What This Changes

The issue requests opt-in WhatsApp active liveness probes after idle periods and reconnects so recovery is trusted only after a server round trip confirms the inbound path.

## Best Possible Solution

Have maintainers choose a WhatsApp-plugin liveness contract, then implement opt-in active probes in the controller/reconnect/status path with a privacy-safe Baileys primitive, config/docs/tests, and live WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main shows passive timestamp/frame-based liveness, reconnects reset the passive window, and no active probe is wired. I did not reproduce the reported multi-hour WhatsApp zombie state in a live environment.

## Solution Assessment

Unclear: active verification is a plausible direction in the right WhatsApp plugin boundary, but maintainers still need to choose the probe primitive, privacy behavior, config contract, and relationship to passive watchdog tuning.
