## Summary

Keep open: current main and v2026.6.9 still rely on passive WhatsApp watchdog timestamps and WebSocket frame activity, with no active inbound-path verification after reconnect. The related watchdog-config and health-hook PRs are adjacent, but neither implements or supersedes this requested liveness decision.

## What This Changes

The issue requests opt-in active WhatsApp connection-liveness verification so reconnects prove the inbound path recovered instead of trusting passive transport and silence timers.

## Best Possible Solution

Have maintainers choose a WhatsApp-plugin liveness design, then implement active verification in the owned controller/reconnect/status path with a dependency-validated probe primitive, config/docs/tests, and live WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main shows passive timestamp/frame-based liveness, a fresh watchdog window after reconnect, and no active probe after reconnect. I did not establish the reported multi-hour WhatsApp zombie state in a live environment during this read-only review.

## Solution Assessment

Unclear. Active verification is plausible, but the best fix depends on maintainers choosing the probe primitive, privacy behavior, config contract, and relationship to existing watchdog tuning and health/status paths.
