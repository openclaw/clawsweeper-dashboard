## Summary

Keep open: current main and v2026.6.8 still make WhatsApp recovery decisions from passive transport/app timestamps, with no active post-reconnect or idle inbound-path proof; the related items are useful but do not implement this request.

## What This Changes

Request an opt-in WhatsApp active liveness probe that verifies the inbound path during idle windows and after reconnects before trusting passive healthy status.

## Best Possible Solution

Keep this open for a maintainer-approved WhatsApp plugin design that validates a low-side-effect active probe, feeds failures into the existing reconnect/backoff/status path, and proves recovery with live WhatsApp evidence.

## Reproduction Assessment

Yes at source level, but not as a live current-main repro. Current code shows passive frame/timestamp-based liveness and no active proof path after reconnect; the reported multi-hour production zombie state still needs live WhatsApp reproduction.

## Solution Assessment

Unclear. Active verification is a plausible solution, but the best fix depends on maintainers choosing the probe primitive, privacy behavior, config contract, and coordination with related recovery work.
