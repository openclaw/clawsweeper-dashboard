## Summary

Keep open: current main and v2026.6.8 still make WhatsApp recovery decisions from passive transport/app timestamps and do not actively verify that inbound delivery recovered after reconnect. The related watchdog-tuning and health-hook items are adjacent, but they do not implement the requested active inbound-path liveness check.

## What This Changes

The issue requests an opt-in WhatsApp active liveness probe that verifies inbound delivery after idle periods and reconnects.

## Best Possible Solution

Keep this as the active-liveness design tracker: choose a low-side-effect WhatsApp probe primitive, feed failed probes into the existing reconnect/backoff/status path, add an opt-in config surface only if maintainers approve it, and prove recovery with a real WhatsApp setup.

## Reproduction Assessment

Yes at source level for the missing feature: current main still uses passive frame/app timestamp liveness and lacks active post-reconnect verification. No live WhatsApp zombie-state reproduction was established in this read-only review.

## Solution Assessment

Unclear: active verification is a plausible direction, but it is not proven as the best fix until maintainers select the probe primitive, privacy behavior, and config/API contract. The linked watchdog-tuning work is useful but does not solve this request.
