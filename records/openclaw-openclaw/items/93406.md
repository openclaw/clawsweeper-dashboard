## Label Justifications

- `P1`: The report describes a real WhatsApp channel workflow where inbound delivery can remain frozen for hours while the gateway appears connected.
- `impact:message-loss`: The central impact is inbound WhatsApp messages not reaching OpenClaw despite transport-level health signals.
- `impact:session-state`: The reported zombie state is a mismatch between connection/session health and actual inbound application delivery.

## Summary

Keep open: current main and the latest release still make WhatsApp recovery decisions from passive transport/app timestamps, while the requested active inbound-path verification is not implemented and is not safely superseded by the related watchdog-tuning, health-hook, or reconnect-backfill items.

## What This Changes

The issue requests an opt-in WhatsApp active liveness probe that confirms the inbound path recovered after idle periods and reconnects.

## Best Possible Solution

Keep the request open for a maintainer-approved WhatsApp-plugin design: validate a low-side-effect probe primitive, add an opt-in config surface only if needed, feed failures into the existing reconnect/backoff/status path, and prove it with real WhatsApp recovery evidence.

## Reproduction Assessment

Yes at source level: current main shows passive timestamp/frame-based liveness and no active probe path after reconnect. I did not establish the reported multi-hour WhatsApp zombie state in a live environment during this read-only review.

## Solution Assessment

Unclear: active verification is a plausible direction, but the best fix depends on maintainers choosing the WhatsApp probe primitive, privacy behavior, config contract, and relationship to the existing reconnect/backfill trackers.
