## Summary

Keep open. Current main and v2026.5.19 still parse managed browser evaluate input as a parenthesized expression, so statement bodies with `var`, `let`, or `const` still fail and the closed broader evaluate issue explicitly left this as a separate follow-up.

## What This Changes

The issue asks for browser evaluate to accept JavaScript statement bodies such as `var`, `let`, and `const` without requiring a manual IIFE wrapper.

## Best Possible Solution

Extend the existing browser evaluate implementation to accept statement bodies while preserving current expression/function behavior across page, ref, batch, and existing-session paths.

## Reproduction Assessment

Yes. Source inspection plus a direct parser check show the current `eval("(" + fnBody + ")")` wrapper rejects declaration statements and accepts the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is compatibility-preserving statement support in the existing evaluate path, not a separate browser exec surface.
