## Summary

Keep open. Current main and the latest release still parse managed browser evaluate input as a parenthesized expression, so `var`, `let`, and `const` statement bodies fail while the reported IIFE workaround succeeds; the broader browser exec/evaluate request was closed with this narrower statement-body issue explicitly left as follow-up.

## What This Changes

The issue requests that `browser act kind=evaluate` and `openclaw browser evaluate` accept JavaScript statement bodies such as `var`, `let`, and `const` without manual IIFE wrapping.

## Best Possible Solution

Extend the browser plugin evaluate implementation to accept statement bodies through a compatibility-preserving async wrapper while keeping current expression/function behavior, evaluate gating, timeout bounds, and navigation/SSRF guards across page, ref, and existing-session paths.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with `eval("(" + fnBody + ")")`, which rejects `var`, `let`, and `const` statement bodies while accepting the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is to treat evaluate input as a compatibility-preserving function body when needed, rather than adding a separate browser exec surface or changing the existing security gate.
