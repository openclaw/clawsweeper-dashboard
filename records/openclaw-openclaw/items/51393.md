## Summary

Keep open. Current main still parses managed Playwright `act:evaluate` input as a parenthesized expression, so declaration statements remain blocked; related #55598 was closed around the existing evaluate surface and does not supersede this statement-support gap.

## What This Changes

Allow `browser act kind=evaluate` / `openclaw browser evaluate` to accept JavaScript statement bodies with declarations instead of only expression/function source.

## Best Possible Solution

Add compatibility-preserving statement-body support inside the browser plugin evaluate implementation, centralize the parsing/fallback logic, and cover page/ref evaluate without weakening timeout, abort, evaluateEnabled, or navigation/SSRF guards.

## Reproduction Assessment

Yes, at source level. Current main uses the same parenthesized eval wrapper in both managed Playwright evaluate paths, and a direct parser check confirms that wrapper rejects `var`, `let`, and `const` statement bodies.

## Solution Assessment

Yes. The narrowest maintainable direction is a focused browser plugin evaluate change with regression tests, not a new core browser tool or a duplicate `browser.exec` surface.
