## Summary

Keep open: current main and v2026.5.20 still parse browser evaluate input as a parenthesized expression, so declaration statement bodies with var, let, or const fail. The broader browser exec/evaluate issue is closed as implemented but explicitly leaves this narrower statement-body request open, and the other related open issue tracks force-click behavior rather than this gap.

## What This Changes

The issue requests allowing browser evaluate input to include JavaScript statement bodies such as var, let, and const declarations instead of only expressions.

## Best Possible Solution

Extend the existing browser evaluate path with compatibility-preserving statement/function-body support, update CLI/docs/tool guidance, and add focused tests for expressions, functions, IIFEs, declarations with return, ref/page targets, timeout/abort, and navigation guards.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with eval("(" + fnBody + ")"), which rejects var/let/const statement bodies while accepting the reported IIFE workaround.

## Solution Assessment

Yes, if it is implemented in the existing evaluate path without changing security defaults. The maintainable fix is compatibility-preserving statement/function-body support, not a separate browser.exec command or a looser evaluate gate.
