## Summary

Keep open: current main and the latest shipped release still parse managed browser evaluate input as a parenthesized expression, so statement bodies with declarations fail. The related browser exec/evaluate issue was closed with this issue explicitly left as the narrower follow-up, and the open force-click issue does not own this statement-body request.

## What This Changes

The issue asks for browser evaluate input to accept JavaScript statement bodies such as `var`, `let`, and `const` instead of only expression input.

## Best Possible Solution

Add compatibility-preserving statement/function-body support in the existing browser evaluate path while keeping the current gate, timeout, navigation, and security boundaries intact.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main and `v2026.5.20` still wrap evaluate input with `eval("(" + fnBody + ")")`, which rejects declaration statement bodies while accepting the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is compatibility-preserving statement/function-body support in the existing evaluate implementation, not a separate browser exec command or a change to evaluate security defaults.
