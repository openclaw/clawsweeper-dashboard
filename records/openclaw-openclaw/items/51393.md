## Summary

Keep open. Current main still parses managed Playwright browser evaluate input as a parenthesized expression in both ref-based and page-level paths, so statement bodies beginning with `var`, `let`, or `const` still fail before execution, and the public tool/CLI surface has no statement mode or replacement command that supersedes this request.

## What This Changes

The issue requests browser `act` evaluate support for JavaScript statement bodies such as `var`, `let`, and `const` instead of requiring manual IIFE wrapping.

## Best Possible Solution

Implement compatibility-preserving statement support in the browser plugin evaluate path, likely by centralizing expression/function parsing with a safe statement-body fallback, then add focused coverage for page and ref evaluate with `var`, `let`, `const`, async statement bodies, existing expressions, and function inputs.

## Reproduction Assessment

Yes. The current managed Playwright path is source-reproducible: `fn` is parsed with the same parenthesized eval wrapper that a direct Node check shows rejects `var`, `let`, and `const` statement bodies before execution.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the browser plugin evaluate implementation, with deliberate expression-vs-statement handling rather than a broad new core tool or an unconditional wrapper that changes return behavior.

## Repair Work Prompt

Fix browser evaluate statement support for https://github.com/openclaw/openclaw/issues/51393. Current managed Playwright evaluate parses `fn` with `eval("(" + fnBody + ")")`, which rejects statement bodies such as `var x = 1; x`, `let x = 1; x`, and `const x = 1; x`. Implement a compatibility-preserving solution in the browser plugin evaluate path, likely around `evaluateViaPlaywright`, so page and ref evaluate accept statement bodies while existing function-expression and expression inputs keep their current return semantics. Do not broaden core/plugin boundaries, do not add a separate core tool unless absolutely necessary, and do not weaken `browser.evaluateEnabled`, timeout clamping, abort disconnect handling, or navigation/SSRF guards. Add or extend colocated Vitest coverage for page evaluate and ref evaluate with `var`, `let`, `const`, async statement bodies, existing expression inputs, and existing function inputs. Consider whether existing-session Chrome MCP evaluate should be documented or covered separately if its behavior differs. Because this is user-facing browser behavior, add a single-line `CHANGELOG.md` entry under the active version with `Thanks @cpc1986`; do not credit bot or maintainer handles.
