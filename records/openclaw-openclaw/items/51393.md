## Summary

Keep open. Current main still parses managed Playwright browser evaluate input as a parenthesized expression, so `var`/`let`/`const` bodies reproduce the reported syntax failure; the closed related browser exec request explicitly left this statement-support gap as a separate follow-up.

## What This Changes

Allow `browser act` and `openclaw browser evaluate` to accept statement-style JavaScript bodies with declarations instead of requiring IIFE wrapping.

## Best Possible Solution

Add compatibility-preserving statement-body support inside the browser plugin evaluate implementation, with shared parsing/fallback logic and regression coverage for page and ref evaluate while preserving timeout, abort, evaluateEnabled, target, navigation, and SSRF guards.

## Reproduction Assessment

Yes, at source level. Current main uses the same parenthesized eval wrapper in both managed Playwright evaluate paths, and a direct parser check rejects `var`, `let`, and `const` statement bodies.

## Solution Assessment

Yes. The narrowest maintainable direction is a focused browser plugin evaluate change with regression tests, not a new core tool or a duplicate `browser.exec` surface.
