## Summary

Keep open. Current main still treats managed Playwright evaluate input as a parenthesized expression in both page and ref paths, so raw declaration statements remain unsupported; the closed #55598 discussion does not supersede this narrower statement-body request.

## What This Changes

Support `browser act kind=evaluate` inputs that contain JavaScript statements such as `var`, `let`, or `const` declarations instead of requiring an IIFE workaround.

## Best Possible Solution

Add compatibility-preserving statement-body support inside the browser plugin evaluate implementation, centralize the parse/fallback logic for page and ref evaluation, and document the accepted `--fn` forms.

## Reproduction Assessment

Yes, at source level. Current main uses the same parenthesized eval wrapper in both managed Playwright evaluate paths, and a direct parser check confirms that wrapper rejects `var`, `let`, and `const` statement bodies.

## Solution Assessment

Yes. The narrowest maintainable direction is a focused browser plugin evaluate change with regression tests, not a new core browser tool or a duplicate `browser.exec` surface.

## Repair Work Prompt

Implement statement-body support for `browser act kind=evaluate` / `openclaw browser evaluate --fn` in the browser plugin. Keep existing expression and function-source behavior compatible; add a shared helper for page and ref evaluation so raw JavaScript statement bodies such as `var x = 1; return x` or `let x = 1; return x` can run without requiring callers to wrap an IIFE. Preserve `browser.evaluateEnabled`, evaluate timeout/abort behavior, ref argument handling, and post-evaluate navigation/SSRF guards. Do not add a separate `browser.exec` command or a new config option. Add focused regression coverage for page and ref evaluate statement inputs, and update user-facing docs/changelog if behavior changes.
