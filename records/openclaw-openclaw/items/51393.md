## Summary

Keep open. Current main and the latest release still parse managed browser evaluate input as a parenthesized expression, so statement bodies with `var`, `let`, or `const` still require the reported IIFE workaround; the related browser exec/evaluate issue was closed while explicitly leaving this statement-body request as separate follow-up.

## What This Changes

Request support for statement-style JavaScript bodies in `browser act kind=evaluate` / `openclaw browser evaluate` without manual IIFE wrapping.

## Best Possible Solution

Extend the browser plugin evaluate implementation to accept statement bodies while preserving current expression/function behavior across page, ref, and existing-session paths.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with `eval("(" + fnBody + ")")`, which rejects declaration statements and accepts the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is to extend the existing evaluate parsing/execution path with compatibility-preserving statement support, not add a separate browser exec surface.
