## Summary

Keep open: current main and the latest release still reject statement-style evaluate bodies because the managed browser path parses input as a parenthesized expression, and the broader browser exec thread explicitly left this narrower request as follow-up.

## What This Changes

The issue requests that browser evaluate accept JavaScript statement bodies such as var, let, and const declarations instead of requiring expression or IIFE input.

## Best Possible Solution

Extend the existing browser evaluate implementation to accept statement bodies while preserving current expression/function compatibility, gating, timeout bounds, and navigation guards across page, ref, batch, and existing-session behavior.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with `eval("(" + fnBody + ")")`, which rejects statement bodies with `var`, `let`, or `const` while accepting the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is compatibility-preserving statement support in the existing browser evaluate path, not a separate browser exec command or a change to security defaults.
