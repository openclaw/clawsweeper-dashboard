## Summary

Keep open. Current main still treats managed-browser evaluate input as a parenthesized expression/function, and the related arbitrary-JavaScript issue was closed while explicitly leaving this statement-body request as a separate follow-up.

## What This Changes

The issue requests support for JavaScript statements such as var/let/const in browser evaluate actions instead of requiring callers to wrap them in an IIFE.

## Best Possible Solution

Define a backward-compatible statement execution mode for browser evaluate, or explicitly document expression-only behavior; if implemented, cover managed and existing-session profiles with tests for expressions, functions, ref arguments, async bodies, and statement bodies with explicit returns.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: managed-browser evaluate wraps the supplied text in parentheses before `eval`, so declarations such as `var x = 1` are invalid syntax in that parser shape. I did not run the browser server because this review is read-only and avoids artifact-producing commands.

## Solution Assessment

Unclear. Auto-wrapping statement bodies is plausible, but the existing field is named `fn`, CLI/docs describe function source, and the Chrome MCP path forwards a `function` payload, so maintainers should decide the cross-driver API semantics first.
