## Summary

Keep open: current main and v2026.5.22 still parse browser evaluate input as a parenthesized expression, so `var`/`let`/`const` statement bodies remain unsupported. The broader browser exec/evaluate request at https://github.com/openclaw/openclaw/issues/55598 was closed with this narrower behavior explicitly left as follow-up, and no open PR now owns it.

## What This Changes

The issue asks for `browser act kind=evaluate` to accept JavaScript statement bodies with declarations such as `var`, `let`, and `const`, instead of only expression-shaped input.

## Best Possible Solution

Add compatibility-preserving statement/function-body support in the existing browser evaluate paths, with focused tests and docs, while keeping current security gates and timeout/navigation guards.

## Reproduction Assessment

Yes. Source inspection shows the current `eval("(" + fnBody + ")")` wrapper rejects declaration statements, and a direct parser check confirms `var`/`let`/`const` inputs fail while IIFEs and function expressions parse.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing evaluate implementation to accept statement/function bodies without adding a separate browser exec command or weakening evaluate security defaults.
