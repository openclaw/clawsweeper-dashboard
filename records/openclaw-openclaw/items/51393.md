## Summary

Keep open: current main and v2026.5.22 still parse managed browser evaluate input as a parenthesized expression, so `var`/`let`/`const` statement bodies remain unsupported. The closed browser exec/evaluate request explicitly left this narrower behavior as follow-up, and the FSB comment gives useful prior art for treating input as a function body while preserving the existing timeout wrapper.

## What This Changes

The issue requests support for JavaScript statement bodies in browser `act kind=evaluate`, so `var`/`let`/`const` declarations work without manual IIFE wrapping.

## Best Possible Solution

Add compatibility-preserving statement/function-body support in the existing browser evaluate paths, with focused tests and docs, while keeping current security gates and timeout/navigation guards.

## Reproduction Assessment

Yes. Source inspection shows the current `eval("(" + fnBody + ")")` wrapper rejects declaration statements, and a direct parser check confirms `var`/`let`/`const` inputs fail while IIFEs and function expressions parse.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing evaluate implementation to accept statement/function bodies without adding a separate browser exec command or weakening evaluate security defaults.
