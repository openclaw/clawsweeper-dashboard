## Summary

Keep open: current main and v2026.5.20 still parse browser evaluate input as a parenthesized expression, so declaration statements fail; no canonical PR or duplicate issue has taken over this narrower request.

## What This Changes

The issue requests that browser evaluate accept JavaScript statement or function-body input such as var/let/const declarations instead of only expression/function sources.

## Best Possible Solution

Extend the existing browser evaluate implementation to accept statement/function-body input while preserving current expression/function compatibility, gating, timeout bounds, and navigation guards across page, ref, batch, and existing-session behavior.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with `eval("(" + fnBody + ")")`, which rejects statement bodies with `var`, `let`, or `const` while accepting the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is compatibility-preserving statement/function-body support in the existing browser evaluate path, not a separate browser exec command or a change to security defaults.
