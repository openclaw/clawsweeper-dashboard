## Summary

Keep open: current main still evaluates browser input by parenthesizing the submitted source before `eval`, so declaration statements such as `var`, `let`, and `const` remain invalid. The closed browser exec/evaluate request at https://github.com/openclaw/openclaw/issues/55598 explicitly left this narrower statement-body behavior as follow-up, and duplicate search did not find an open PR or better canonical item owning it.

## What This Changes

The issue requests browser evaluate support for JavaScript statement bodies such as `var`/`let`/`const` declarations instead of accepting only expression-shaped inputs.

## Best Possible Solution

Add compatibility-preserving statement/function-body support in the existing browser evaluate path, with focused tests and docs, while keeping current security gates, timeouts, and navigation guards.

## Reproduction Assessment

Yes. Source inspection and a direct parser check show the current `eval("(" + fnBody + ")")` wrapper rejects `var`/`let`/`const` statement bodies while the documented IIFE workaround remains expression-shaped.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing evaluate implementation to accept statement/function bodies without adding a separate browser exec command or weakening evaluate security defaults.
