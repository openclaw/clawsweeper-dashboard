## Summary

Keep open: current main and latest v2026.5.22 still parse browser evaluate input as a parenthesized expression, so `var`/`let`/`const` statement bodies still fail; no merged PR or canonical duplicate has taken over this narrower request.

## What This Changes

Request `browser act kind=evaluate` and `openclaw browser evaluate` to accept statement/function-body JavaScript, including declarations, without requiring callers to wrap code in an IIFE.

## Best Possible Solution

Add compatibility-preserving statement/function-body support in the existing browser evaluate path, with focused tests and docs, while keeping the current security gates and timeout/navigation guards.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with `eval("(" + fnBody + ")")`, which rejects `var`, `let`, and `const` statement bodies while accepting the reported IIFE workaround.

## Solution Assessment

Yes. The maintainable fix is a narrow compatibility-preserving extension of the existing evaluate implementation, not a new browser exec command or a change to evaluate security defaults.
