## Summary

Keep open. Current main and v2026.5.12 still parse managed browser evaluate input as parenthesized expressions, so declaration statements like `var`, `let`, and `const` remain unsupported without the reported IIFE workaround; the related browser exec/evaluate thread was closed while explicitly leaving this statement-body request as a follow-up.

## What This Changes

Request browser evaluate support for JavaScript statement bodies, especially `var`/`let`/`const` declarations, without manual IIFE wrapping.

## Best Possible Solution

Add compatibility-preserving statement-body support in the browser plugin evaluate implementation with shared parsing/fallback logic, focused page/ref regression coverage, docs, and an explicit existing-session decision.

## Reproduction Assessment

Yes. Current main wraps managed Playwright evaluate input with `eval("(" + fnBody + ")")`, and a direct parser check rejects `var`, `let`, and `const` bodies while accepting the reported IIFE workaround.

## Solution Assessment

Yes. The narrowest maintainable solution is to extend the existing browser plugin evaluate path and tests with compatibility-preserving statement support, not add a separate `browser.exec` surface.

Implement statement-body support for browser evaluate in the browser plugin without adding a new core tool, config option, or force-click behavior. Preserve existing expression and function-source semantics first: function sources should still be called with the element for ref evaluate and without arguments for page evaluate; expressions and IIFEs should still return the same values; runtime errors should not be hidden by a statement fallback; async results must keep the current Promise.race timeout behavior. Add shared private parsing/execution logic for the Playwright page and ref evaluators, support declaration statement bodies such as `var x = 1; return x`, `let x = 1; return x`, and `const x = 1; return x`, and clarify whether existing-session Chrome MCP evaluate can support the same behavior or should be documented/rejected consistently.
