## Summary

Keep open. Current main and the latest release still parse managed browser evaluate input as a parenthesized expression, so statement bodies with `var`, `let`, or `const` still fail while the reported IIFE workaround succeeds; the closed browser exec/evaluate thread explicitly left this narrower request as follow-up.

## What This Changes

The issue requests `browser act kind=evaluate` support for JavaScript statement bodies such as `var`, `let`, and `const` without requiring a manual IIFE wrapper.

## Best Possible Solution

Extend the browser plugin evaluate implementation to accept statement bodies while preserving current expression/function behavior, evaluate gating, timeout bounds, and navigation/SSRF guards across page, ref, and existing-session paths.

## Reproduction Assessment

Yes, by source inspection and a direct parser check. Current main wraps evaluate input with `eval("(" + fnBody + ")")`, which rejects `var`/`let`/`const` statement bodies and accepts the reported IIFE workaround.

## Solution Assessment

Yes. The narrow maintainable path is to extend the existing evaluate parsing/execution path with compatibility-preserving statement support, not add a separate browser exec surface.
