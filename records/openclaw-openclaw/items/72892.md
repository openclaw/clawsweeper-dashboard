## Summary

Keep open. Current main still keys inbound dedupe on route plus `MessageSid`, while Control UI/WebChat `chat.send` maps each client idempotency key into `MessageSid`; two dispatches with different run ids remain runtime-distinct and would not collapse. The discussion correctly cross-links #72314, but that PR is scoped to missing message IDs and does not close this identified-message sender/run-id mismatch.

## What This Changes

The issue reports duplicate Control UI/WebChat inbound user messages because two dispatch paths use different sender/run identifiers and bypass inbound dedupe.

## Best Possible Solution

Keep this issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat backend guard that catches that fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids would bypass dedupe, but there is no current-main duplicate-pair artifact proving where the second dispatch is produced.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable path is to first identify the double-dispatch source and fix or guard that specific Control UI/WebChat path.
