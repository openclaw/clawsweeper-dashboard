## Summary

Keep open. Current main still keys accepted WebChat sends and inbound dedupe on the `idempotencyKey`/`MessageSid`, so distinct run ids can bypass the existing duplicate guard; the related maintainer PR #76446 targets this area but is not merged in the provided context.

## What This Changes

The issue reports that Control UI/WebChat can inject one user message twice into the same session under different sender identities, causing two agent runs and replies.

## Best Possible Solution

Land or revise #76446, or an equivalent narrow Gateway/WebChat dedupe, with regression coverage for same-content distinct-id sends and then verify the duplicate sender metadata no longer appears.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; source inspection shows two accepted WebChat sends with different idempotency keys bypass current dedupe, but the full one-browser-action duplicate path still needs logs or session metadata.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the open fix is reviewed; a narrow active-run WebChat dedupe looks maintainable, but it should be checked against the exact sender-mismatch evidence before closing.
