## Summary

Keep open. Current main has a Control UI in-flight submit guard for duplicate browser submits, but Gateway and inbound dedupe still treat distinct `chat.send` run IDs as distinct inbound messages; the related Gateway-side active-send fix is still an open PR in the provided context, not merged main behavior.

## What This Changes

The issue reports duplicate Control UI/WebChat inbound user turns when the same browser message enters one session under different sender/run identities.

## Best Possible Solution

Review and land #76446 or an equivalent Gateway-side active WebChat dedupe, then close this after a regression test or captured session metadata proves one browser send cannot create two agent runs.

## Reproduction Assessment

No, not for the exact one-browser-action duplicate path. Source inspection does show that two accepted WebChat `chat.send` calls with different idempotency keys produce different `MessageSid` values and bypass inbound dedupe, but current UI also has a partial in-flight submit guard.

## Solution Assessment

Unclear. The Control UI guard is useful, but the maintainable complete fix should be Gateway-side if duplicate sends can reach `chat.send` through distinct client/run identities; #76446 is the active candidate to review.
