## Summary

Keep open. Current main still accepts distinct WebChat `chat.send` run ids as distinct inbound message ids, and the active WebChat duplicate-send fix is still represented by an open related PR rather than merged main behavior.

## What This Changes

The issue asks OpenClaw to prevent Control UI/WebChat from injecting the same user message twice under different sender identities and triggering duplicate agent replies.

## Best Possible Solution

Keep this issue open until #76446 or an equivalent fix lands, then verify the duplicate sender/run-id shape with focused Gateway regression coverage or captured session metadata.

## Reproduction Assessment

Partially. Source inspection gives a clear current-main path where two accepted WebChat `chat.send` calls with different idempotency keys become different inbound `MessageSid` values and bypass dedupe, but the full one-browser-action duplicate path still needs logs or `sessions_history` evidence.

## Solution Assessment

Unclear. The active-run dedupe proposed by #76446 is a plausible narrow fix for duplicated WebChat submits, but the exact sender-mismatch path should be verified so a separate Control UI bridge relay is not left behind.
