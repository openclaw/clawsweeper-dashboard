## Summary

Keep open. Current main still leaves the reported bypass plausible: inbound dedupe keys by route plus `MessageSid`, `chat.send` turns each client idempotency key into `MessageSid`, and the Control UI/Gateway guards only cover the same in-flight submit or reused idempotency key. The related #72314 work is useful context but is scoped to missing message ids, not two identified dispatches with different ids.

## What This Changes

Ensure Control UI/WebChat user messages trigger only one agent run even when duplicate sender identities or distinct run ids would bypass inbound dedupe.

## Best Possible Solution

Keep this open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat guard that catches that fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids bypass dedupe, but there is no current-main duplicate-pair artifact proving where the second dispatch originates.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable path is to identify the double-dispatch source and fix or guard that specific Control UI/WebChat path.
