## Summary

Keep open. Current main still makes the reported bypass plausible: inbound dedupe keys on route plus `MessageSid`, `chat.send` maps each client idempotency key to `MessageSid`, and the Control UI guard only coalesces same in-flight submits before distinct run ids exist. The related #72314 PR is useful prior art for idless retries, but it is open and does not cover two identified dispatches with different ids.

## What This Changes

The issue asks OpenClaw to prevent Control UI WebChat from injecting one user message through both the direct WebChat path and the `openclaw-control-ui` sender path so one user message triggers only one agent run.

## Best Possible Solution

Keep this open, collect one duplicate pair from debug logs or `sessions_history`, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat guard that catches that fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids bypass dedupe, but there is no current-main duplicate-pair artifact proving the second dispatch origin.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable solution is to identify the double-dispatch source and fix or guard that specific Control UI/WebChat path.
