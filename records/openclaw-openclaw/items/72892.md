## Summary

Keep open. Current `main` still has the bypass shape described in the discussion: WebChat `chat.send` turns use the client run id as `MessageSid`, inbound dedupe keys on that id, and current UI/Gateway guards only collapse same-submit or same-id retries rather than two distinct accepted sends with identical content.

## What This Changes

The issue reports duplicate Control UI/WebChat inbound user messages caused by two distinct sender identities and message ids entering the same agent session.

## Best Possible Solution

Capture one current-main duplicate pair with full session metadata, then remove the upstream second dispatch path if identifiable; use a narrowly scoped secondary dedupe key only if the duplicate shape cannot be eliminated at the source.

## Reproduction Assessment

Unclear. Static source gives a high-confidence bypass path for two distinct WebChat run ids, but the issue still needs current-main logs or `sessions_history` metadata proving one browser send produces two dispatch origins.

## Solution Assessment

Unclear. A content fingerprint fallback is plausible defense-in-depth, but the safer maintainable fix is to identify why two dispatches are created before changing normal repeated-message semantics.
