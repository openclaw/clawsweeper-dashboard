## Summary

Keep open. Current main still has the structural gap: WebChat dispatches use the generated run id as `MessageSid`, inbound dedupe only skips the same dedupe key, and the UI/Gateway guards only cover duplicate in-flight submits or reused idempotency keys. The linked #72314 PR is scoped to missing message ids, and #75737 is broader rather than a canonical replacement for this narrower duplicate-inbound report.

## What This Changes

The issue reports duplicate Control UI/WebChat inbound user messages caused by distinct sender/run identities bypassing inbound dedupe and triggering two agent turns.

## Best Possible Solution

Keep this as the narrow tracking issue, collect one duplicate pair with provider, originatingChannel, accountId, peerId, senderId, messageSid, and runId, then remove the upstream double dispatch or add a narrowly scoped WebChat/Control UI guard for that exact shape.

## Reproduction Assessment

Unclear. The code gives a high-confidence explanation for why two distinct WebChat run ids bypass dedupe, but the report still needs current-main debug logs or `sessions_history` metadata proving the two dispatch origins in one duplicate pair.

## Solution Assessment

Unclear. A content fingerprint fallback might catch the symptom, but the safer maintainable fix is to first identify the second-dispatch path and guard that path without changing normal repeated-message semantics.
