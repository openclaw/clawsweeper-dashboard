## Summary

Keep open. Current main still has the same observable boundary described in the report: inbound dedupe is route-plus-message-id based, `chat.send` maps each client idempotency key to `MessageSid`, and the Control UI guards only coalesce the same in-flight submit before a new run id is generated. The related #72314 PR is useful context for idless retries, but it does not cover two identified WebChat/Control UI dispatches with different ids.

## What This Changes

The issue asks OpenClaw to stop Control UI/WebChat from injecting one user message twice through different sender/run identities so it triggers only one agent turn.

## Best Possible Solution

Keep this open, capture one duplicate pair with provider, originatingChannel, accountId, peerId, senderId, messageSid, runId, and client path metadata, then remove the upstream double dispatch or add a narrowly scoped WebChat/Control UI guard that does not change normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter provides plausible v2026.4.24 steps and current source explains how two different run ids bypass dedupe, but there is no current-main duplicate-pair log or `sessions_history` artifact proving where the second dispatch originates.

## Solution Assessment

Unclear. A content fingerprint fallback could catch the symptom, but the safer maintainable solution is to identify the specific WebChat/Control UI double-dispatch path and guard that path without broad content dedupe.
