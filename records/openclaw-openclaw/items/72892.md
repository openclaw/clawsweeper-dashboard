## Summary

Keep open. Current main still has the structural gap described in the discussion: WebChat dispatch uses a generated run id as `MessageSid`, inbound dedupe only collapses identical provider/route/message-id keys, and the existing Control UI/Gateway guards cover same-submit or same-idempotency retries rather than two distinct run ids from different sender paths.

## What This Changes

Report asks OpenClaw to prevent Control UI/WebChat from injecting the same user message twice under different sender/run identities.

## Best Possible Solution

Keep this as the narrow tracking issue, collect one duplicate pair with provider, originatingChannel, accountId, peerId, senderId, messageSid, and runId, then remove the upstream double dispatch or add a narrowly scoped guard for that exact shape.

## Reproduction Assessment

Unclear. Static source explains how two distinct WebChat run ids bypass inbound dedupe, but the issue still needs current-main debug logs or `sessions_history` metadata proving the two dispatch origins for one duplicate pair.

## Solution Assessment

Unclear. A content fingerprint secondary key could catch the symptom, but the safer maintainable fix is to identify the second dispatch path first and guard or remove that path without changing normal repeated-message semantics.
