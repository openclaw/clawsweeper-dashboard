## Summary

Keep open. Current main still dedupes inbound WebChat dispatches by route plus `MessageSid`, while `chat.send` uses the generated idempotency key as that message id, so a second path with a different run id would still bypass the cache; the related #72314 PR is scoped to missing ids and #75737 is broader rather than a canonical replacement.

## What This Changes

The issue asks WebChat/Control UI to stop injecting one user message twice under distinct sender/run identities so one send produces one agent run.

## Best Possible Solution

Keep this as the narrow tracker, capture one duplicate pair with provider/originatingChannel/accountId/peerId/senderId/messageSid/runId, then remove the upstream double dispatch or add a narrowly scoped WebChat/Control UI guard for that exact shape.

## Reproduction Assessment

Unclear. The source gives a high-confidence explanation for why two distinct WebChat run ids bypass current dedupe, but this review still does not have current-main logs or `sessions_history` metadata proving the two dispatch origins for one duplicate pair.

## Solution Assessment

Unclear because there is no proposed patch. A content fingerprint fallback may reduce the symptom, but the safer maintainable fix is to first identify the second-dispatch path and guard or remove that path without changing normal repeated-message semantics.
