## Summary

Keep open. Current main still makes the reported bypass plausible: inbound dedupe requires a stable `MessageSid`, `chat.send` turns each distinct idempotency key into a distinct `MessageSid`, and the Control UI guard only coalesces duplicate submits before a second request is made. The discussion correctly identifies #72314 as adjacent missing-id dedupe work, but that PR does not close the reported two-identified-dispatch case.

## What This Changes

The issue requests fixing Control UI/WebChat duplicate user-message injection where two sender identities and different message IDs bypass inbound dedupe and trigger two agent turns.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat backend guard that catches this fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source shows why two different run IDs would bypass dedupe, but there is no current-main duplicate-pair artifact proving where the second dispatch is produced.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable solution is to first identify the double-dispatch source and fix or guard that specific Control UI/WebChat path.
