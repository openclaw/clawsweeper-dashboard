## Summary

Keep open. Current main still has the structural bypass described in the issue discussion: inbound dedupe keys on provider/route/message id, `chat.send` uses each Control UI idempotency key as `MessageSid`, and the UI/Gateway guards only collapse same-submit or same-id retries. The missing piece is current-main metadata proving where the second dispatch originates.

## What This Changes

Fix Control UI WebChat so one browser user message enters the agent session once instead of twice under different sender/message identities.

## Best Possible Solution

Keep this as the narrow tracking issue, capture one current-main duplicate pair with full provider/origin/account/peer/session/messageSid/sender/runId metadata, then remove the upstream second dispatch path or add a narrowly scoped guard for that exact shape.

## Reproduction Assessment

Unclear. Static source shows that two `chat.send` calls with different idempotency keys would bypass inbound dedupe, but the report still lacks current-main logs or `sessions_history` metadata proving one browser send creates both dispatches.

## Solution Assessment

Unclear. Removing the second dispatch source would be the narrowest fix if identified; a content-fingerprint backstop should only be considered after preserving normal repeated-message semantics.
