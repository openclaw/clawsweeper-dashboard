## Summary

Keep open. Current main still makes the reported bypass plausible: inbound dedupe requires a stable message id and does not include content, `chat.send` maps each client idempotency key to `MessageSid`, and the Control UI guard only coalesces duplicate submits while the same submit is still in flight. The related #72314 work is adjacent but remains scoped to missing message ids, so it does not close two identified dispatches with different ids.

## What This Changes

The issue asks OpenClaw to prevent Control UI/WebChat user messages from being injected twice into the same agent session when the duplicate dispatches carry different sender or run ids.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat backend guard that catches the fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids would bypass dedupe, but there is no current-main duplicate-pair artifact or log proving where the second dispatch is produced.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable path is to first identify the double-dispatch source and fix or guard that specific Control UI/WebChat path.
