## Summary

Current main already blocks the reported WebChat/Control UI duplicate active-send path before inbound message-id dedupe is reached, and the fix is merged and contained in the latest release.

## What This Changes

WebChat/Control UI duplicate inbound sends with different sender identities and message ids should trigger only one agent run.

## Best Possible Solution

Keep the shipped active WebChat send dedupe; any fresh duplicate-processing logs from a different non-`chat.send` path should be tracked as a new narrow report or under the broader duplicate-delivery tracker.

## Reproduction Assessment

No current-main failing reproduction remains. The pre-fix shape was source-reproducible from two active WebChat text sends with different idempotency keys, and current main now has a regression test proving that shape dispatches only once.

## Solution Assessment

Yes. The narrow upstream `chat.send` active-run guard solves the reported duplicate active internal WebChat dispatch without adding broad content-based inbound dedupe that could suppress intentional later messages.
