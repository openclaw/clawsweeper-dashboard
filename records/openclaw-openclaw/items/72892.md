## Summary

Keep open. Current main still has the structural gap described in the issue discussion: WebChat dispatch uses a per-send idempotency key as the inbound `MessageSid`, inbound dedupe keys on that message id rather than content, and the existing UI/Gateway guards only collapse same-submit or same-id retries. Review target: https://github.com/openclaw/openclaw/issues/72892

## What This Changes

The issue asks OpenClaw to stop Control UI/WebChat from injecting one user message twice under different sender/run identities and triggering duplicate agent turns.

## Best Possible Solution

Keep this as the narrow tracking issue, capture one current-main duplicate pair with full session metadata, then either remove the upstream second dispatch path or add a narrowly scoped guard for that exact shape.

## Reproduction Assessment

Unclear. Static source shows how two distinct WebChat run ids would bypass inbound dedupe, but the issue still needs current-main logs or `sessions_history` metadata proving the two dispatch origins for one browser send.

## Solution Assessment

Unclear. A content fingerprint fallback could reduce duplicate replies, but the best maintainable fix is to identify the second dispatch source first and avoid changing normal repeated-message semantics.
