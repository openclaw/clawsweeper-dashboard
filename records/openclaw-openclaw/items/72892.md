## Summary

Keep open. Current `main` still treats distinct WebChat `chat.send` idempotency keys as distinct inbound `MessageSid` values, and inbound dedupe remains message-id based; the related active-send fix is an open PR, not merged main behavior.

## What This Changes

The issue asks OpenClaw to prevent Control UI/WebChat from injecting one user message twice under different sender identities and triggering two agent runs.

## Best Possible Solution

Keep this as the narrow tracking issue until PR #76446 or an equivalent fix lands, then verify the duplicate sender shape with a regression test or captured session metadata before closing.

## Reproduction Assessment

Partially. Source inspection shows two accepted WebChat sends with different idempotency keys would bypass inbound dedupe, but the full one-browser-action duplicate path still needs current-main logs or `sessions_history` metadata.

## Solution Assessment

Unclear until the open fix is reviewed. A narrow active-run WebChat dedupe looks plausible for rapid duplicate sends, but the best fix should be confirmed against the exact second-dispatch source before this issue closes.
