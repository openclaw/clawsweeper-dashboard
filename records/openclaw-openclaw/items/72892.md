## Summary

Keep open. Current main still has the message-id-based inbound dedupe shape described in the report, and the current Control UI/WebChat guards only catch reused idempotency keys or duplicate in-flight submits before a second distinct run id exists. The related idless-retry PR is still open and scoped to missing message IDs, while the newer WebChat/Control UI report is broader rather than a canonical replacement for this narrower duplicate-inbound-message bug.

## What This Changes

The issue asks for Control UI/WebChat to inject each user message into the agent session only once, instead of creating distinct inbound turns with different sender/run identities that bypass inbound dedupe.

## Best Possible Solution

Keep this as the narrow WebChat/Control UI duplicate-inbound tracker, capture one real duplicate pair with full routing metadata, then remove the upstream double dispatch or add a tightly scoped guard for that specific pair without changing normal repeated-message semantics.

## Reproduction Assessment

Unclear. Source review gives a high-confidence mechanism for why two distinct run ids bypass inbound dedupe, but the issue still needs current-main gateway logs or sessions_history metadata proving the two dispatch origins for the same user send.

## Solution Assessment

Unclear. Avoiding a second injection is the right end state, but the safest implementation should first identify whether the duplicate comes from a second WebChat client/send path, session resume behavior, or a bridge relay; a generic content-hash dedupe is not yet the best proven fix.
