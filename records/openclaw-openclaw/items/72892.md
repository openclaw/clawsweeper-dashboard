## Summary

Keep open. Current main still has the structural gap described in the report: Control UI/WebChat sends use generated run IDs as `MessageSid`, inbound dedupe keys include that message ID, and the existing UI/Gateway guards only collapse duplicate submits that reuse the same in-flight submit key or idempotency key. The related idless-retry PR #72314 is useful adjacent work but does not cover two distinct run IDs, and #75737 is broader rather than a canonical replacement for this narrower duplicate-inbound report.

## What This Changes

The issue reports that Control UI/WebChat can inject one user message twice with different sender/run identities, bypassing inbound dedupe and causing duplicate agent replies.

## Best Possible Solution

Keep this issue as the narrow tracking item, capture one duplicate pair with full dispatch/session metadata, then remove the upstream double dispatch or add a narrowly scoped WebChat/Control UI guard that collapses only that duplicate pair pattern without changing normal repeated-message semantics.

## Reproduction Assessment

Unclear. The source gives a high-confidence explanation for why two distinct run IDs bypass dedupe, but the issue still lacks current-main debug logs or `sessions_history` metadata proving the two dispatch origins for a reproducible duplicate pair.

## Solution Assessment

Unclear. A content-fingerprint secondary key could catch the symptom, but the maintainable fix should first identify whether the duplicate comes from a second WebChat client/send path, session resume behavior, or a Control UI bridge relay, then guard that path specifically.
