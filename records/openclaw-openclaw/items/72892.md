## Summary

Keep open. Current main still keys inbound dedupe on route plus `MessageSid`, while `chat.send` maps each client idempotency key into `MessageSid`; the discussion correctly notes that two Control UI/WebChat dispatches with different ids would still bypass the cache. The related #72314 work is relevant prior art, but it is scoped to missing message ids and does not directly cover two identified dispatches with different ids.

## What This Changes

The issue asks OpenClaw to stop Control UI/WebChat from injecting one user message twice under different sender identities or message ids, which causes duplicate agent replies.

## Best Possible Solution

Keep this open, capture one duplicate pair with provider, originatingChannel, accountId, peerId, senderId, messageSid, and runId, then either remove the upstream double dispatch or add a narrow Control UI/WebChat guard that catches that fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current main explains why two distinct run ids would bypass dedupe, but there is no current-main duplicate-pair artifact proving where the second dispatch originates.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the narrower maintainable fix is to identify the second dispatch path and fix or guard that specific Control UI/WebChat route.
