## Summary

Keep open. Current main still matches the reported failure mode: inbound dedupe is keyed by route plus `MessageSid`, `chat.send` maps each client idempotency key into `MessageSid`, and the existing Control UI/Gateway guards only catch the same in-flight submit or reused idempotency key. The discussion and the related #72314 PR are useful context, but that PR is scoped to missing message ids, not duplicate Control UI/WebChat dispatches that both carry different ids.

## What This Changes

Ensure Control UI/WebChat user messages trigger only one agent run when duplicate sender identities or distinct run ids would otherwise bypass inbound dedupe.

## Best Possible Solution

Keep this open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat guard that catches that fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids bypass dedupe, but there is no current-main duplicate-pair log or `sessions_history` artifact proving where the second dispatch originates.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable path is to identify and fix or guard the specific Control UI/WebChat double-dispatch source.
