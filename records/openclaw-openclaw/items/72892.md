## Summary

Keep open. Current main still makes the reported bypass plausible when the same WebChat content reaches the backend as two distinct `chat.send` runs with different idempotency keys; the existing dedupe and UI guards cover same-id or in-flight duplicate submits, not a proven two-origin fan-out. The related #72314 work is useful adjacent context but is scoped to missing provider message ids and does not close this sender/run-id mismatch.

## What This Changes

The issue asks OpenClaw to stop Control UI WebChat from injecting one user message twice under distinct sender/run identities and causing duplicate agent replies.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat backend guard that catches that fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids bypass dedupe, but there is no current-main duplicate-pair log or `sessions_history` metadata proving where the second dispatch originates.

## Solution Assessment

Unclear. A secondary content fingerprint could catch the symptom, but the safer maintainable fix is to identify the second dispatch source first and fix or guard that specific Control UI/WebChat path.
