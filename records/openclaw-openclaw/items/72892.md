## Summary

Keep open. Current main still has the same structural gap described in the report: Control UI/WebChat sends map each generated run id to `MessageSid`, inbound dedupe keys include that message id, and the UI/Gateway guards only catch reused idempotency keys or duplicate in-flight submits before a second distinct run id exists. The discussion correctly distinguishes this from the open idless-retry PR, and the newer #75737 report is related but broader rather than a canonical replacement for this narrower duplicate-inbound-message bug.

## What This Changes

The issue reports that a Control UI WebChat user message can be dispatched twice into the same agent session with different sender/run identities, causing duplicate agent replies because inbound dedupe cannot match the two turns.

## Best Possible Solution

Preserve this as the narrower tracking issue, capture one duplicate pair with provider, originatingChannel, accountId, peerId, senderId, messageSid, and runId, then either remove the upstream double dispatch or add a narrowly scoped WebChat/Control UI guard that collapses the duplicate pair without changing normal repeated-message semantics.

## Reproduction Assessment

Unclear. The source path gives a high-confidence explanation for why two distinct run ids bypass dedupe, but the issue still lacks current-main debug logs or `sessions_history` metadata proving the two dispatch origins for a reproducible duplicate pair.

## Solution Assessment

Unclear. A content fingerprint secondary key could catch the symptom, but the maintainable fix should first identify whether the duplicate comes from a second WebChat client/send path, session resume behavior, or a bridge relay and then guard that specific path.
