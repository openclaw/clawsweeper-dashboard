## Summary

Keep open. Current main has partial coverage for the field report, but it still leaves meaningful requested work: invalid `tools.exec.host` config values get strict enum/allowed-value feedback without nearest-value guidance, runtime fanout visibility is split across task, process, MCP, and Gateway diagnostics surfaces, and the deterministic local maintenance-runner pattern is not documented.

## What This Changes

The issue requests better guidance for the `tools.exec.host` enum footgun, broader operator observability into runtime/process/MCP/session fanout, documentation for model-free local maintenance runners, and consideration of compact inline log review in dashboards.

## Best Possible Solution

Split this field report into focused follow-ups: a small config-validation UX decision for invalid enum suggestions, a designed operator status surface that joins task/subagent/session/process/MCP/runtime-load signals, a docs page or section for deterministic local maintenance runners, and a separate Control UI decision for inline log review.

## Reproduction Assessment

Partially yes: the `tools.exec.host` footgun is reproducible from the current strict schema and validation path. The RAM/fanout observation does not yet have a high-confidence minimal reproduction, so it is best treated as an observability request.

## Solution Assessment

No as a single repair lane. The maintainable solution is to split the report by surface before implementation, because each part has different owners, validation paths, and product tradeoffs.
