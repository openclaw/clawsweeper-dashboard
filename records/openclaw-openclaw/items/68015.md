## Summary

Keep this issue open. Current main has partial improvements for the field report, but the requested work is still not complete: invalid `tools.exec.host` values get strict allowed-value feedback without nearest-value guidance, fanout visibility remains split across task/process/diagnostic surfaces, and the deterministic local maintenance-runner pattern is not documented as an endorsed ops pattern.

## What This Changes

The issue requests improved guidance for an invalid `tools.exec.host` enum value, better runtime fanout observability, documentation for deterministic local maintenance runners, and more compact in-band operator log review.

## Best Possible Solution

Split this field report into focused maintainer-triaged follow-ups: decide whether config validation should add nearest-value enum suggestions for `tools.exec.host`, design a unified operator status surface for task/session/process/MCP/runtime pressure, document when deterministic local scripts or systemd user timers are preferable to model turns, and handle inline dashboard log review as a separate Control UI decision.

## Reproduction Assessment

Partially yes. The `tools.exec.host` footgun is reproducible from the current strict schema and validation helper; the RAM/fanout observation has no high-confidence minimal reproduction and is best treated as an observability request.

## Solution Assessment

No, not as one repair lane. The maintainable path is to split the report by surface because the enum guidance, diagnostics view, docs pattern, and dashboard UX have different owners and validation paths.
