## Summary

Keep open: current main still has the reported source shape, and the prior shipped reconnect-drain fix does not address repeated poll-success drains against a live delivery that is still holding the active queue claim.

## What This Changes

The issue asks for Telegram reconnect recovery to stop re-entering the same in-flight pending delivery while the live send is still claimed and stalled.

## Best Possible Solution

Add a narrow Telegram/outbound recovery change that suppresses or backs off redundant reconnect drains for entries still owned by a live send, while preserving active-claim duplicate protection and crash-replay recovery.

## Reproduction Assessment

Yes for a source-level reproduction: current main drains after every Telegram poll success, and the shared queue logs/skips when the pending delivery is still held by withActiveDeliveryClaim. I did not run a live Telegram reproduction of the 129-second stall in this read-only pass.

## Solution Assessment

No current implementation is not yet the best end state: the active claim is correct, but reconnect drain scheduling needs to avoid repeatedly re-entering the same live-owned pending delivery. The safer fix is to coalesce/defer the redundant drain rather than hiding logs or replaying the entry.
