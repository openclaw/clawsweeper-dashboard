## Summary

Keep open: current main still has the reported shape. Telegram schedules reconnect recovery after every poll success, while the shared delivery queue claim skips an entry that is still owned by a live send, so the repeated `already being recovered` loop remains a plausible current-main bug rather than an implemented fix.

## What This Changes

The issue reports that Telegram reconnect polling repeatedly drains the same pending media delivery while its original send is still in flight, causing repeated recovery-skip logs and long user-visible stalls.

## Best Possible Solution

Add a narrow reconnect-drain single-flight or active-claim backoff so poll-success recovery does not repeatedly reprocess the same live-owned queue entry, while preserving replay for genuinely stranded Telegram pending deliveries and verifying the Telegram send timeout/retry boundary.

## Reproduction Assessment

Yes, at source level: hold an enqueued delivery with `withActiveDeliveryClaim`, trigger repeated Telegram poll-success drains, and current code will skip the same entry as `already being recovered` until the live claim releases. I did not run a live Telegram setup in this read-only review.

## Solution Assessment

No, current main is not the best end state because it preserves repeated poll-success recovery without suppressing the same active live-delivery entry. The best fix should target reconnect-drain scheduling or per-entry active-claim backoff, not remove the active claim that prevents duplicate sends.
