## Summary

Keep open. Current main has useful partial mitigations for allowlisted embedded runs, but it still does not implement the requested memory-only fast path: Active Memory still enters the normal embedded runner, the core coding-tool registry is constructed before final allowlist filtering, and the standard prompt/resource-loader/session/stream setup path still runs. The related broad latency reports and #75591 overlap, but they do not supersede this narrower active-memory/embedded-run design request.

## What This Changes

The issue requests an opt-in fast path for active-memory and other memory-only embedded runs so allowlisted tools and lightweight setup do not pay the full embedded runner initialization cost.

## Best Possible Solution

Add a generic allowlist-aware embedded-run profile that can construct only explicitly allowed tools while preserving tool policy, sandbox, auth, and plugin contracts. Treat provider/session warm-pooling and ephemeral session-index behavior as separately reviewed follow-ups unless benchmarks show they are required after construction-time and prompt/resource-loader work is reduced.

## Reproduction Assessment

Yes for the source-level reproduction: current main still sends Active Memory through runEmbeddedPiAgent with a memory toolsAllow list, constructs the coding-tool registry before final filtering, and runs normal prompt/resource/stream setup. The exact latency numbers should be reproduced with `[trace:embedded-run] prep stages` on a representative workspace before landing a fix.

## Solution Assessment

Unclear. The proposed `fastPath: true` flag is directionally plausible, but the safer maintainable solution is likely a named allowlist-aware embedded profile or tool factory first, with security-sensitive provider/session reuse designed and benchmarked separately.
