## Summary

Keep open. Current main has useful partial mitigations for explicit memory-tool allowlists, bundle-MCP skipping, Active Memory live-config removal, and abort cleanup, but it still does not provide a construction-time fast path for memory-only embedded runs and still builds the broad embedded tool/system-prompt/session setup path before streaming.

## What This Changes

The issue requests a fast-path opt-in for memory-only embedded Active Memory runs so they avoid full tool registry, prompt, resource-loader, and provider setup overhead on each invocation.

## Best Possible Solution

Add a generic embedded-run lightweight construction path that builds only explicitly allowed core and plugin tools while preserving sandbox, owner-only, provider, auth, and plugin policy checks. Treat system-prompt slimming and provider/session reuse as separately benchmarked follow-ups unless the first change still leaves a measured setup floor.

## Reproduction Assessment

Yes for the source-level root cause: current main runs Active Memory with a memory-only `toolsAllow`, then constructs the broad embedded tool registry before filtering it. The live timing numbers are supplied by the issue and corroborating comment, but were not rerun here.

## Solution Assessment

Unclear. A fast-path concept is directionally right, but the safest maintainable solution is probably a construction-time explicit tool allowlist or lightweight embedded profile first, with resource-loader and stream warm-pool caching reviewed separately.
