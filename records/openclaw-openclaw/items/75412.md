## Summary

Keep open. Current main still has only partial mitigations for allowlisted embedded runs: Active Memory still calls the normal embedded runner, the runner still constructs the OpenClaw coding-tool registry before final allowlist filtering, and the normal prompt/resource/stream setup stages still run. The issue discussion adds independent Active Memory and dreaming traces, so this remains a valid performance report that needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests an opt-in memory-only embedded-run fast path for Active Memory and similar callers to avoid fixed setup overhead by constructing only allowed memory tools and reducing prompt/resource/provider setup work.

## Best Possible Solution

Add a generic allowlist-aware embedded-run profile that can construct only explicitly allowed tools while preserving tool policy, sandboxing, auth, plugin contracts, and diagnostics. Treat provider/session warm pooling and ephemeral one-shot session registry behavior as separately reviewed follow-ups unless benchmarking shows they are required for the first fix.

## Reproduction Assessment

Yes for the source-level reproduction: current main still runs Active Memory through the normal embedded attempt path, constructs the tool registry before final `toolsAllow` filtering, and proceeds through the standard prompt/resource/stream setup stages. The exact 16-18s timing still needs a representative live workspace trace.

## Solution Assessment

Unclear. The proposed `fastPath: true` flag is plausible, but the safer maintainable direction is a typed allowlist-aware embedded profile or tool factory first, with security-sensitive provider/session caching designed and benchmarked separately.
