## Summary

Keep open. Current main has partial mitigations for allowlisted embedded runs, but it still does not implement the requested Active Memory or memory-only embedded fast path: Active Memory uses the normal embedded runner, the runner still builds the broad OpenClaw tool registry before final `toolsAllow` filtering, and the usual prompt/resource/session/stream setup path still runs per attempt.

## What This Changes

Request an opt-in fast path for Active Memory and other memory-only embedded runs so allowlisted memory tools can avoid broad tool construction, heavyweight prompt setup, resource-loader setup, and repeated stream/session initialization overhead.

## Best Possible Solution

Add a generic allowlist-aware embedded-run profile that constructs only explicitly allowed tools while preserving tool policy, sandbox, auth, and plugin contracts; benchmark that first for Active Memory and memory-only jobs, then consider provider/session warm pooling as a separately reviewed optimization if setup still exceeds the target budget.

## Reproduction Assessment

Yes. The issue includes concrete v2026.4.29 trace timings from several environments, and current main still follows the same source-level path: Active Memory calls the normal embedded runner, broad tool construction occurs before final allowlist filtering, and normal prompt/resource/session/stream setup still runs.

## Solution Assessment

Unclear. The proposed `fastPath: true` flag is directionally plausible, but the safest maintainable fix is likely a named allowlist-aware embedded profile and construction-time tool filtering first, with lifecycle caching handled as a separate design step.
