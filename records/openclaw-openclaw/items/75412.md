## Summary

Keep open. Current main has useful partial mitigations for allowlisted embedded runs, but the reported performance/design gap remains: Active Memory still enters the normal embedded runner, full coding-tool construction still happens before final allowlist filtering, and the regular prompt, resource-loader, and stream setup phases still run per attempt. The issue discussion includes multiple independent 2026.4.29 traces and adjacent latency reports, so this is a concrete current performance item rather than cleanup noise.

## What This Changes

The issue requests an opt-in fast path for Active Memory and similar memory-only embedded runs so they avoid full embedded-run setup overhead on each invocation.

## Best Possible Solution

Add a generic allowlist-aware embedded-run profile that constructs only explicitly allowed tools while preserving existing policy, sandbox, auth, and plugin contracts; benchmark Active Memory and dreaming after that before adding provider/session warm-pool reuse.

## Reproduction Assessment

Yes for the source-level reproduction: current main still routes Active Memory through the normal embedded attempt path, constructs the full tool set before filtering to memory tools, and runs the standard prompt/resource/stream setup. The exact latency numbers still need a live benchmark on a representative workspace.

## Solution Assessment

Unclear. The proposed fastPath flag is directionally plausible, but the safer maintainable fix is an explicit allowlist-aware construction/profile seam first, with session/provider caching treated as a separate security-sensitive follow-up.
