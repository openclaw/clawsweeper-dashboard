## Summary

Keep open. Current main contains partial mitigation for memory-only embedded runs, but it still does not implement the requested fast path or remove the fixed setup floor described by the report and corroborating comments.

## What This Changes

The issue requests a fast-path opt-in for memory-only embedded runs, especially active-memory, to avoid full tool, prompt, resource-loader, and stream setup on every invocation.

## Best Possible Solution

Design a generic embedded-run lightweight profile that constructs only explicitly allowed tools while preserving all existing policy gates, then benchmark active-memory and other memory-only embedded callers. Treat provider/resource-loader warm pooling as a separate measured design unless maintainers explicitly approve the broader cache semantics.

## Reproduction Assessment

Yes for the source-level reproduction: current main still routes active-memory through the standard embedded attempt path and still performs tool construction, prompt build, resource-loader reload, session creation, and stream setup before streaming. The exact latency numbers need live timing proof on a representative workspace.

## Solution Assessment

Unclear. The proposed `fastPath` is directionally aligned with the observed cost, but the safer maintainable solution is likely a narrower construction-time allowlist or lightweight embedded profile first, with provider/session reuse handled separately after benchmark and security review.

## Risks / Open Questions

- The exact 16-18s setup timing was not live-benchmarked during this read-only review, so the remaining performance magnitude should be validated on a representative workspace.
- A fast path that bypasses tool construction can accidentally bypass sandbox, owner-only, provider, group, plugin disablement, or SecretRef semantics if it is not designed as a generic policy-preserving construction path.
- Provider/session warm pools are broader than tool filtering and need explicit invalidation rules for auth profile changes, model changes, plugin runtime state, and per-session transcript/resource state.
- The issue overlaps still-open broader latency reports such as #75591, so maintainers should avoid splitting a shared root-cause fix into incompatible special cases.
