## Summary

Keep open. Current main still builds the normal embedded-run tool registry before applying the memory-only allowlist, and the remaining request is a real runtime performance/design issue rather than a stale or already-implemented cleanup item.

## What This Changes

The issue requests a fast embedded-run path so Active Memory and other memory-only embedded runs avoid fixed setup overhead from full tool construction and repeated prompt/session setup.

## Best Possible Solution

Add a generic embedded-run lightweight construction path or construction-time allowlist that can materialize only explicitly allowed core/plugin tools while preserving existing policy and sandbox guarantees; treat prompt slimming and provider/session reuse as separate measured follow-ups.

## Reproduction Assessment

Yes for the source-level root cause: current main still constructs the normal tool registry before applying the memory-only allowlist, and Active Memory uses that path on eligible turns. The exact timing reproduction needs a live benchmark workspace with trace:embedded-run prep-stage output.

## Solution Assessment

Unclear. A fast path is directionally right, but a broad boolean that also changes prompt composition and caches provider/session state is riskier than first adding a policy-preserving construction-time allowlist or named lightweight embedded profile.
