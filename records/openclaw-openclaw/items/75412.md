## Summary

Keep open. Current main contains useful partial mitigations for allowlisted embedded runs, but it still does not implement the requested active-memory fast path: memory-only runs still enter the full embedded attempt pipeline, build the OpenClaw coding-tool registry before final allowlist filtering, and then run the normal system-prompt, resource-loader, agent-session, and stream setup stages.

## What This Changes

The issue requests a lightweight or fast embedded-run path for active-memory and other memory-only embedded invocations to avoid paying full setup overhead on every turn.

## Best Possible Solution

Add a generic allowlist-aware embedded-run profile that can construct only explicitly allowed tools while preserving policy, sandbox, auth, and plugin contracts; treat provider/session warm-pooling as a separately benchmarked follow-up if the construction-time and prompt/resource work does not get active-memory under its timeout budget.

## Reproduction Assessment

Yes for the source-level reproduction: current main still runs Active Memory through the normal embedded attempt path, constructs tools before final `toolsAllow` filtering, and proceeds through the standard prompt/resource/stream setup stages. The exact timing regression still needs a live benchmark on a representative workspace.

## Solution Assessment

Unclear. The proposed `fastPath: true` flag is directionally plausible, but the narrowest maintainable solution is likely an allowlist-aware embedded profile or tool factory first, with security-sensitive caching or warm-pool reuse designed separately.
