## Summary

Keep open. Current main has a relevant partial mitigation from #74592: `toolsAllow` is now forwarded into plugin-tool materialization so memory plugin tools can be exposed, but the embedded attempt still constructs the broad core/OpenClaw tool registry, builds a system prompt, reloads the resource loader, and sets up an agent session before streaming. The specific fixed setup-latency request is therefore not implemented and is narrower than the broader Active Memory reliability trackers.

## What This Changes

The issue requests a fast-path opt-in for memory-only embedded agent runs so Active Memory can avoid per-invocation setup work that is unrelated to memory recall.

## Best Possible Solution

Keep this as a specific embedded-run performance/design issue. The maintainable end state is a generic lightweight embedded-run construction path that uses the runtime allowlist at construction time, preserves all tool-policy and plugin boundaries, benchmarks prompt/resource/session setup separately, and lets Active Memory opt in only after those invariants are covered by tests.

## Reproduction Assessment

Yes for the source-level reproduction: Active Memory still invokes the embedded runner with a memory-only allowlist, and current main still builds the broad core tool registry plus fixed prompt/resource/session stages before streaming. The exact wall-clock timings need a representative live benchmark after the #74592 partial mitigation.

## Solution Assessment

Unclear as proposed. A fast path is directionally appropriate, but a single opt-in boolean plus provider/session warm pooling is not obviously the safest first step; a construction-time tool allowlist with benchmarked prompt/resource-loader follow-ups would be easier to validate.
