## Summary

Keep open. Current main has useful partial mitigations for allowlisted embedded runs, but it still does not implement the requested memory-only embedded-run fast path: Active Memory still enters the normal embedded runner, full tool construction still happens before final allowlist filtering, and the normal prompt/resource-loader/session/stream setup path still runs.

## What This Changes

The issue requests a memory-only embedded-run fast path for Active Memory and related one-shot embedded runs to avoid full tool, prompt, session, and provider setup overhead.

## Best Possible Solution

Add a generic allowlist-aware embedded-run profile that constructs only explicitly allowed tools while preserving policy, sandbox, auth, and plugin contracts; handle prompt minimization and ephemeral one-shot session behavior in the same design, and treat provider/session warm pooling as a separately benchmarked follow-up if needed.

## Reproduction Assessment

Yes for the source-level reproduction: current main still runs Active Memory through the normal embedded attempt path, constructs the coding tool registry before final allowlist filtering, and proceeds through the standard prompt/resource/stream setup. The exact latency numbers still need live trace timing on a representative v2026.4.29 or current-main workspace.

## Solution Assessment

Unclear. The proposed `fastPath: true` flag is plausible, but the safer maintainable shape is likely an allowlist-aware embedded profile or tool factory first, with session/provider reuse designed separately because it crosses auth, sandbox, and lifecycle boundaries.
