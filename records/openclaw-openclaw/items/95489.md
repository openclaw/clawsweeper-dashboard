## Summary

Keep open: current main and v2026.6.9 still contain the source-level bypass where claude-cli generic failure text becomes visible non-error output and the fallback classifier returns `null` before model fallback can advance; two open linked PRs now target the fix.

## What This Changes

The issue requests that claude-cli generic out-of-credits failure output trigger the configured model fallback chain instead of being delivered as the final assistant response.

## Best Possible Solution

Land one guarded classifier fix that treats only the exact generic external runner failure text as fallback-worthy while preserving visible-output, hook-block, abort, replay, and committed-delivery guards; then close this issue and supersede duplicate PRs.

## Reproduction Assessment

Yes at source level: current main wraps CLI `output.text` as a normal `{ text }` payload, `hasVisibleAgentPayload` counts that text as visible, and the fallback classifier returns `null` before fallback classification. I did not run a live exhausted Claude Code account.

## Solution Assessment

Yes. The classifier is the right owner boundary because the fallback runner already advances when result classification returns a failure; CLI result-shaping could also work, but a guarded classifier fix is the narrowest current path.
