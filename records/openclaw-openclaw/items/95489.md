## Summary

Keep open: current main still accepts the reported claude-cli generic failure text as visible non-error output before the model-fallback classifier reaches its billing/error checks, while the docs promise configured fallback candidates for billing and provider failures.

## What This Changes

The issue reports that claude-cli credit-exhaustion output is treated as a final assistant response instead of triggering the configured model fallback chain.

## Best Possible Solution

Teach the existing fallback path to treat claude-cli's generic external-run failure copy as a failed attempt, then add focused regression coverage without adding config or changing explicit user model-selection policy.

The bug is that a claude-cli primary can emit `GENERIC_EXTERNAL_RUN_FAILURE_TEXT` as a normal visible payload, causing `classifyEmbeddedAgentRunResultForModelFallback` to return `null` before configured fallbacks are tried. Keep the fix narrowly in the embedded-agent fallback classifier or CLI result shaping; preserve no-retry safeguards for aborts, hook blocks, replay-invalid side effects, committed outbound delivery evidence, and real visible assistant replies. Do not add config options, do not change fallback policy for explicit user `/model` selections, and do not edit `CHANGELOG.md`.
