## Summary

Keep open: current main and the latest release still limit Dreaming light-phase ingestion to daily memory files, redacted session transcripts, and existing recall entries; workspace file-creation events remain a useful but product-sensitive memory feature request.

## What This Changes

Request Dreaming light phase to ingest workspace file-creation signals, such as newly created tools or scripts, so they can become promotion candidates after compaction.

## Best Possible Solution

Design an opt-in workspace-event memory source with allowlisted paths, redaction, scoring, docs, and tests before wiring it into light Dreaming and deep promotion.

## Reproduction Assessment

Yes, at source level. Current main still feeds light Dreaming from daily memory, session transcripts, and recall entries only; I did not run a live workspace reproduction in this read-only review.

## Solution Assessment

Unclear. A workspace.watchPaths-style source is plausible, but the best fix needs maintainer agreement on memory-source ownership, safety filters, and whether this belongs in memory-core config or a narrower plugin/API seam.
