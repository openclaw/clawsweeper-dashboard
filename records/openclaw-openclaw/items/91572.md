## Summary

Keep open: current main and the latest release still do not ingest workspace file-creation events into dreaming; the merged recall diagnostic work explains excluded memory hits but does not add the requested workspace-event source.

## What This Changes

The issue requests that memory-core dreaming light phase ingest workspace file creation events from configurable workspace paths so newly created tools/config/data can become memory promotion candidates.

## Best Possible Solution

Define an opt-in workspace-event memory source with path allowlists, redaction, and clear plugin/API ownership, then wire it into short-term promotion with focused docs and tests.

## Reproduction Assessment

Yes. Source inspection shows current main only feeds light dreaming from daily memory files, session transcripts, and existing recall entries; no live workspace run was performed in this read-only review.

## Solution Assessment

Unclear. `workspace.watchPaths` is plausible, but the best fix needs a maintainer decision on memory-source ownership, safety filters, and whether to extend the plugin API instead of adding a broad watcher.
