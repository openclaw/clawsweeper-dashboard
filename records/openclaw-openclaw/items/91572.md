## Summary

Keep open: current main and v2026.6.6 still do not ingest workspace file-creation events into dreaming; the related merged recall diagnostic work only explains excluded memory hits and deliberately leaves promotion eligibility unchanged.

## What This Changes

The issue requests that dreaming ingest workspace file-creation events so newly created tools, config files, and data files can become memory promotion candidates instead of relying only on memory notes and transcripts.

## Best Possible Solution

Define an opt-in workspace-event memory source with path allowlists, redaction, and clear memory-core or plugin-API ownership, then feed approved events into short-term promotion with focused docs and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows light dreaming only ingests daily memory files and session transcripts before reading existing short-term recall entries; no live workspace run was performed in this read-only review.

## Solution Assessment

Unclear. A workspace watchPaths-style source is plausible, but the best fix needs a maintainer decision on safety filters, config/API ownership, and whether existing memorySearch.extraPaths should remain search-only.
