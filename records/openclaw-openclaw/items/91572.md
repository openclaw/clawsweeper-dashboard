## Summary

Keep open: current main and v2026.6.8 still limit light dreaming to daily memory files, session transcripts, and existing recall entries, so workspace file-creation events remain a real missing memory signal. The request is a feature/product boundary decision rather than a strict bug because docs and config do not currently define workspace watcher ingestion.

## What This Changes

Request to have memory-core dreaming ingest workspace file-creation events, such as new tools, scripts, and config files, as short-term signals eligible for later memory promotion.

## Best Possible Solution

Define an opt-in workspace-event memory source with path allowlists, redaction, and clear memory plugin/API ownership, then connect it to short-term promotion with focused docs and tests.

## Reproduction Assessment

Yes. Source inspection shows current main only feeds light dreaming from daily memory files, session transcripts, and existing recall entries; no live workspace run was performed in this read-only review.

## Solution Assessment

Unclear. A workspace.watchPaths-style source is plausible, but the best fix needs maintainer agreement on memory-source ownership, safety filters, and whether to extend the plugin API instead of adding a broad watcher.
