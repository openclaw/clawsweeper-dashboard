## Summary

Keep open: current main still limits dreaming light-phase ingestion to daily memory notes, session transcript corpus entries, and existing recall traces. The requested workspace file-creation signal is a new memory source and likely new configuration/API surface, so it needs maintainer product review rather than cleanup closure.

## What This Changes

Request that memory-core dreaming ingest workspace file-creation events from paths such as tools/ or scripts/ so created tools can become promotion candidates.

## Best Possible Solution

Define an opt-in, memory-core-owned workspace-event source that records minimal redacted file-creation metadata into the short-term recall store with path and file-type allowlists, or explicitly route the capability to a plugin/API extension if maintainers do not want it in the bundled memory plugin.

## Reproduction Assessment

Yes, from source inspection: current main's light phase only ingests daily memory files, session transcript corpus entries, and existing recall traces. I did not run a live workspace scenario, but the missing workspace-file event path is clear in the current code.

## Solution Assessment

Unclear: workspace.watchPaths is plausible, but the best fix needs a maintainer decision on memory-source ownership, safety filters, and whether to reuse or extend the existing memory event journal. Adding a broad watcher directly to dreaming would be too easy to make noisy or privacy-sensitive.
