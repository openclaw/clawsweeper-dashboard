## Summary

Keep open. Current main still has diagnostic liveness warnings independent of the health-monitor startup grace, and the existing liveness test path warns after 30s whenever diagnostic work is queued or active; related #76360 may reduce the cold-start trigger but does not resolve the warning policy.

## What This Changes

The issue asks OpenClaw to suppress or grace diagnostic liveness event-loop-delay warnings during gateway cold start.

## Best Possible Solution

Define an explicit diagnostic liveness startup grace that preserves useful telemetry while avoiding WARN-level cold-start noise, and handle the underlying Control UI cold-start latency in #76360 separately.

## Reproduction Assessment

Yes, from source and the existing test harness: start the diagnostic heartbeat, queue diagnostic work, return an `event_loop_delay` sample, and advance the 30s timer; current main expects a visible `liveness warning:` log. I did not execute the test because this review was read-only.

## Solution Assessment

Unclear. Reusing the health-monitor grace directly is probably the wrong seam because that grace only gates channel restart checks; a narrower diagnostic heartbeat warn-level grace would better match the reported cold-start noise without hiding operational samples.
