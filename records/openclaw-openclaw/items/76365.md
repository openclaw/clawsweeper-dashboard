## Summary

Keep open. Current main still lacks startup-grace suppression for diagnostic liveness sampling, and #76401 is open and unmerged with closing syntax for this issue.

## What This Changes

Suppress diagnostic liveness warnings during the existing gateway startup-grace window so normal cold-start event-loop spikes do not warn.

## Best Possible Solution

Review and land #76401 or an equivalent focused change that suppresses diagnostic liveness emission during startup grace while preserving post-grace warnings.

## Reproduction Assessment

Yes, source-level. Current main starts diagnostic liveness sampling without a grace option, gateway startup passes only getConfig, and existing tests show queued work turns an event_loop_delay sample into a WARN-level liveness message.

## Solution Assessment

Yes. Sharing the existing startup-grace policy with diagnostic liveness emission is the narrow maintainable fix, and the open closing PR is the active implementation path to review.
