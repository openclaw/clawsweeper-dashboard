## Summary

Keep open. Current main still starts the diagnostic liveness sampler without a startup-grace guard, and open PR #76401 already uses closing syntax for this issue, so the right maintainer path is to review and land or replace that PR before closing.

## What This Changes

The issue asks for diagnostic liveness sampling to honor the gateway startup-grace window so cold-start `models.list` and `sessions.list` delays do not produce WARN-level liveness spam.

## Best Possible Solution

Review and land #76401, or an equivalent focused diagnostics change, so liveness sampling skips the startup-grace window and gains regression coverage in `src/logging/diagnostic.test.ts`.

## Reproduction Assessment

Yes from source inspection: current main starts diagnostics at gateway startup, samples liveness on the first 30s heartbeat, and has no grace guard while diagnostic work is queued. I did not run a live cold-start, but the code and existing fake-timer test shape provide a high-confidence reproduction path.

## Solution Assessment

Yes. Applying a `startupGraceMs` guard to `startDiagnosticHeartbeat` and passing the existing 60s gateway startup-grace is the narrowest maintainable fix; the issue should stay open until the linked PR or equivalent merges.
