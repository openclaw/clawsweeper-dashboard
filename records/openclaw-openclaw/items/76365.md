## Summary

Keep open. Current main still starts diagnostic liveness sampling without a startup-grace guard, and the linked closing PR is still open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Apply the gateway startup-grace window to diagnostic liveness sampling so normal cold-start event-loop delay does not produce WARN-level liveness messages.

## Best Possible Solution

Review and land #76401 or an equivalent focused change that shares the existing 60s startup grace with diagnostic liveness sampling and keeps post-startup warnings intact.

## Reproduction Assessment

Yes, source-level. Current main starts diagnostic liveness sampling without a grace option, samples on the heartbeat, and existing tests show queued work turns an event_loop_delay sample into a WARN-level liveness message.

## Solution Assessment

Yes. Applying the existing startup-grace policy to diagnostic liveness emission is the narrow maintainable fix, and the open closing PR is the active implementation path to review.
