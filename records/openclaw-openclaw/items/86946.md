## Summary

Keep this open: current main has substantial WebChat-side protection for pending and optimistic sends, but the durable `chat.send` boundary is still unresolved because the gateway still acknowledges `started` before the user-turn recorder and dispatch/fallback persistence path have completed.

## What This Changes

The issue reports silent WebChat user-message loss when an optimistic send is reconciled against stale `chat.history` before the durable transcript includes the submitted user turn.

## Best Possible Solution

Define and implement a gateway-side success boundary where a WebChat send is either durably visible/progressing after ACK or fails visibly and auditably, while keeping the current UI pending-message preservation.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. Source inspection does show the timing class: UI preservation is stronger now, but `chat.send` can still ACK before the durable/progress boundary this issue asks for.

## Solution Assessment

Unclear as a final product contract. The current UI fix is necessary but incomplete for the reported durable loss; the remaining best fix likely belongs at the gateway persistence/progress boundary and needs maintainer approval.
