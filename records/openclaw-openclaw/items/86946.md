## Label Justifications

- `P1`: The report describes silent loss of a user-submitted WebChat message on the primary chat workflow.
- `impact:data-loss`: The reported user input can fail to appear in the durable transcript after the UI reports it as sent.
- `impact:message-loss`: The central failure is a sent user message disappearing or never becoming an agent turn.
- `impact:session-state`: The issue depends on stale `chat.history` reconciliation and missing durable session transcript state.

## Summary

Keep open: current main and the latest release preserve more WebChat UI state, but `chat.send` still acknowledges `started` before the user turn is durably persisted or dispatch progress is guaranteed, so the reported silent user-message loss class is not proven fixed.

## What This Changes

The issue requests that WebChat user sends not disappear or be silently dropped when `chat.send` returns before the durable transcript/history path catches up.

## Best Possible Solution

Keep the UI pending-message preservation, and make `chat.send` success mean the user turn has either durable/observable run progress or a visible failure path.

## Reproduction Assessment

No live current-main reproduction was run, but the source path still reproduces the acceptance window: `chat.send` ACKs before transcript persistence and dispatch completion. The reporter also supplied live v2026.5.22 evidence for the timing-dependent failure.

## Solution Assessment

Unclear: the current UI preservation is necessary but not sufficient for the reported durable no-turn loss. The best fix likely needs a server-side acceptance/progress/error boundary, with maintainers deciding how it should interact with the non-blocking ACK contract.
