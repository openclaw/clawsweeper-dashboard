## Summary

Keep open: current main and the latest release protect more of the WebChat UI from stale-history replacement, but `chat.send` still acknowledges `started` before the user-turn recorder and fallback persistence boundary is complete, and the open gateway-side candidate PR is unresolved.

## What This Changes

The issue reports WebChat losing accepted optimistic user messages when `chat.send` returns before the durable transcript and a later history reconcile lacks the user turn.

## Best Possible Solution

Make the gateway `chat.send` contract explicit: after ACK, a WebChat user turn is durably visible/progressing or fails visibly and auditably, while keeping the current UI pending-message preservation.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. Source inspection still shows the timing class: UI preservation is stronger, but `chat.send` can ACK before the durable/progress boundary this issue asks for.

## Solution Assessment

Unclear as a final product contract. The UI-side preservation is necessary but not sufficient; the remaining best fix likely belongs at the gateway recorder/session-accessor boundary and needs maintainer approval.
