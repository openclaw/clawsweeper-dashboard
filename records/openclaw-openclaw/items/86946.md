## Label Justifications

- `P1`: The report describes silent loss of a WebChat user message on the primary control surface, leaving users without an agent turn or failure signal.
- `impact:message-loss`: The affected behavior is a sent user message disappearing and not being processed as a chat turn.
- `impact:session-state`: The failure involves stale `chat.history` reconciliation and a user turn missing from the durable session transcript.
- `impact:data-loss`: The reported user input is not preserved in the durable transcript, making it lost user data rather than a rendering-only issue.

## Summary

Keep open: current main still has a real chat.send timing window where WebChat reports `started` before the user message is durably visible, and the existing UI merge protection only preserves display state rather than guaranteeing the agent turn/transcript path.

## What This Changes

The issue requests that WebChat stop losing optimistic user messages when `chat.send` returns before the durable transcript is visible to `chat.history`.

## Best Possible Solution

Make `chat.send` success mean the user send is either durably accepted for the transcript/turn path or fails visibly, while preserving before_agent_run privacy gates and keeping the UI pending bubble until canonical history catches up.

## Reproduction Assessment

Yes in source form: `chat.send` currently returns `started` before the later transcript update path, and the reporter gives live v2026.5.22 steps for the stale-history window. I did not independently run a live WebChat capture in this read-only pass.

## Solution Assessment

Unclear for the proposed local fix: preserving pending UI bubbles helps display, but the report says the durable turn and agent processing can be missing. The maintainable fix likely needs server-side acceptance/durability semantics plus the existing UI reconciliation, with explicit handling for before_agent_run privacy gates.
