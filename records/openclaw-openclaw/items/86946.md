## Label Justifications

- `P1`: The issue reports a primary WebChat send workflow accepting a user message without durable transcript progress or an agent run.
- `impact:data-loss`: The reported failure can silently drop a submitted user turn from durable session history.
- `impact:message-loss`: The central symptom is a user message accepted by chat.send that never reaches agent processing or a visible reply.
- `impact:session-state`: The failure sits at the boundary between WebChat local state, chat.history reconciliation, and session transcript persistence.

## Summary

Keep open: current main and v2026.6.8 mitigate the UI full-replace/optimistic-tail race, but `chat.send` still acknowledges `started` before the durable user-turn transcript/progress boundary is guaranteed, and the open gateway PR is still an unmerged candidate with review blockers.

## What This Changes

The issue requests a fix for WebChat user messages that can remain non-durable after a successful `chat.send` ACK and then disappear or never run when history reconciliation catches an older transcript snapshot.

## Best Possible Solution

Review or replace the open gateway fix so WebChat user turns use the current recorder/session-accessor idempotency path and either become durable before success/progress is reported or surface a visible failure without duplicate transcript writes.

## Reproduction Assessment

No high-confidence live current-main repro was run. Source inspection does reproduce the core ACK-before-durable-transcript/progress boundary, and the reporter provided live logs for the original timing-dependent loss.

## Solution Assessment

Unclear as an implementation choice: preserving UI optimistic messages is already in current main, but the remaining server-side fix needs a maintainer decision on whether chat.send success means durable user-turn progress or only accepted dispatch.
