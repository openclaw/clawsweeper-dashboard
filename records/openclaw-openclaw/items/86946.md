## Label Justifications

- `P1`: The issue reports a broken WebChat send path where accepted user messages can disappear or fail to become durable, affecting the primary agent workflow.
- `impact:data-loss`: The report is about user messages accepted by `chat.send` not reliably landing in the durable transcript.

- `impact:session-state`: The bug sits at the boundary between optimistic UI state, `chat.history` hydration, run state, and durable session transcript state.

## Summary

Current main contains substantial WebChat-side preservation for pending and optimistic messages, so the earlier full-replace UI race is partly mitigated, but the reported durable boundary is still open: `chat.send` can return `started` before user-turn persistence or dispatch progress is guaranteed, and the documented contract remains intentionally non-blocking.

## What This Changes

Prevent WebChat optimistic user messages from being lost when a stale `chat.history` reconcile lands before the durable transcript catches up.

## Best Possible Solution

Keep the current UI pending-message preservation and define a server-side boundary where `chat.send` success means durable user turn, observable run progress, or visible/auditable failure.

## Reproduction Assessment

No high-confidence live current-main reproduction was established here; source inspection shows the ACK-before-dispatch window remains and current tests cover UI preservation rather than durable send progress.

## Solution Assessment

The existing UI preservation is necessary but not the complete best fix; the remaining issue needs a maintainer-approved server-side progress or failure boundary for successful `chat.send` acknowledgements.
