## Summary

Keep open. Current main still routes ordinary custom OpenAI-compatible onboarding through Chat Completions only, while the supported `openai-responses` adapter is available only through manual config or Azure-specific handling; related #67669 adds a diagnostic command but does not fix initial onboarding.

## What This Changes

Custom OpenAI-compatible setup should be able to verify and persist Responses-style endpoints instead of failing when the backend does not support Chat Completions.

## Best Possible Solution

Custom provider onboarding should expose Chat Completions and Responses as distinct OpenAI-compatible choices, verify against the selected endpoint, persist the matching `ModelApi`, and document the choice while preserving current defaults.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: choose custom OpenAI-compatible onboarding for a non-Azure Responses-only `/v1` backend and OpenClaw builds a `/chat/completions` probe, then persists `openai-completions` with no ordinary custom Responses option to select.

## Solution Assessment

Yes. The narrow maintainable fix is to extend onboarding/config parsing, verification, tests, and docs for `openai-responses`; the existing runtime adapter means a broader provider runtime rewrite is unnecessary.
