## Summary

Keep open. Current main still routes non-Azure custom OpenAI-compatible onboarding through Chat Completions and cannot select or persist the existing `openai-responses` adapter, so the reporter's Responses-only custom endpoint failure remains source-reproducible.

## What This Changes

The issue asks custom OpenAI-compatible provider onboarding to validate and save a Responses-style backend instead of failing with a 503 when the backend rejects Chat Completions probes.

## Best Possible Solution

Extend custom-provider onboarding to choose or detect `openai-responses`, verify `/responses`, and persist the existing adapter while keeping Chat Completions and Anthropic flows stable.

## Reproduction Assessment

Yes, at source level: configure a non-Azure `/v1` backend that supports `/responses` but rejects `/chat/completions`, then choose custom OpenAI-compatible or Unknown onboarding. Current main builds a Chat Completions probe and saves `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend custom onboarding/config parsing to select the already-supported `openai-responses` adapter; a provider runtime rewrite is unnecessary.
