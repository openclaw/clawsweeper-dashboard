## Summary

Keep open. Current main still verifies ordinary custom OpenAI-compatible onboarding with Chat Completions only and persists `openai-completions`, even though manual config already supports `openai-responses`; the related Azure work only covers Azure OpenAI endpoints.

## What This Changes

Allow custom OpenAI-compatible onboarding to verify and persist Responses-style providers instead of failing when `/chat/completions` is unavailable.

## Best Possible Solution

Extend the custom-provider onboarding seam to choose or detect `openai-responses`, verify `/responses`, and persist the existing adapter while preserving Chat Completions for current OpenAI-compatible flows.

## Reproduction Assessment

Yes, at source level: configure a non-Azure `/v1` backend that supports `/responses` but rejects `/chat/completions`, then choose OpenAI-compatible custom onboarding. Current main builds a Chat Completions probe and saves `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend custom onboarding/config parsing to select the already-supported `openai-responses` adapter; a provider runtime rewrite is unnecessary.
