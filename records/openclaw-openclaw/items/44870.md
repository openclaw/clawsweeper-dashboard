## Summary

Keep open. Current main and the latest release still support manual `openai-responses` config, but custom-provider onboarding only offers OpenAI Chat Completions or Anthropic and verifies non-Azure OpenAI-compatible endpoints with `/chat/completions`, so a Responses-only backend can still fail the reporter's setup path.

## What This Changes

Allow custom OpenAI-compatible onboarding and verification to work with backends that support `/v1/responses` instead of `/v1/chat/completions`.

## Best Possible Solution

Extend custom-provider onboarding to select or detect `openai-responses`, verify `/responses`, and persist the existing adapter while preserving the current Chat Completions and Anthropic flows.

## Reproduction Assessment

Yes, at source level. Configure a non-Azure `/v1` backend that supports `/responses` but rejects `/chat/completions`, then choose custom OpenAI-compatible or Unknown onboarding; current main builds a Chat Completions probe and saves `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend custom onboarding and non-interactive parsing to select the already-supported `openai-responses` adapter; a provider runtime rewrite is unnecessary.
