## Summary

Keep open. Current main still routes non-Azure custom OpenAI-compatible onboarding and verification through Chat Completions, while the reporter specifically describes a backend that exposes Responses-style OpenAI behavior and rejects that path.

## What This Changes

The issue asks OpenClaw custom model onboarding to validate and persist OpenAI-compatible backends that support `/v1/responses` rather than `/v1/chat/completions`.

## Best Possible Solution

Extend custom-provider onboarding to choose or detect the existing `openai-responses` adapter, verify `/responses`, and persist that adapter while keeping Chat Completions and Anthropic flows stable.

## Reproduction Assessment

Yes, at source level: configure a non-Azure `/v1` backend that supports `/responses` but rejects `/chat/completions`, then choose custom OpenAI-compatible or Unknown onboarding. Current main builds a Chat Completions probe and saves `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend custom onboarding/config parsing to select the already-supported `openai-responses` adapter; a provider runtime rewrite is unnecessary.
