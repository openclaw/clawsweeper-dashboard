## Summary

Keep open. Current main still routes custom non-Azure OpenAI-compatible onboarding through Chat Completions verification and persistence, while the existing runtime/config surface already supports `openai-responses` for backends that expose `/v1/responses`.

## What This Changes

The issue asks for custom OpenAI-compatible provider verification to work for a backend that exposes Responses-style OpenAI models but rejects the Chat Completions probe.

## Best Possible Solution

Extend custom-provider onboarding to expose or detect the existing `openai-responses` adapter, verify `/responses`, and persist that API while leaving Chat Completions, Anthropic, Azure OpenAI Responses, and Azure AI Foundry behavior stable.

## Reproduction Assessment

Yes, at source level: configure a non-Azure `/v1` backend that accepts `/responses` but rejects `/chat/completions`, then choose OpenAI-compatible or Unknown in custom onboarding. Current main sends the Chat Completions probe and maps the provider to `openai-completions`.

## Solution Assessment

Yes. The narrow fix is in custom onboarding/config parsing because the runtime adapter already exists; a provider runtime rewrite is unnecessary.
