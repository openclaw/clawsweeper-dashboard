## Summary

Keep open. Current main still routes ordinary custom OpenAI-compatible onboarding through Chat Completions only; manual config already supports `openai-responses`, and the linked diagnostic PR remains open and does not change onboarding.

## What This Changes

The issue asks custom OpenAI-compatible provider onboarding to validate and persist a Responses-compatible custom backend instead of failing through the default Chat Completions probe.

## Best Possible Solution

Custom-provider onboarding should let users choose or detect the existing `openai-responses` adapter, verify the matching `/responses` endpoint, and keep Chat Completions as the default for existing OpenAI-compatible flows.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a non-Azure `/v1` custom backend that supports `/responses` but not `/chat/completions`, then choose OpenAI-compatible onboarding; current main builds a Chat Completions probe and persists `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing custom onboarding/config parsing seam to select the already-supported `openai-responses` adapter; a provider runtime rewrite is unnecessary.
