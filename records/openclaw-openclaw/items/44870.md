## Summary

Keep open. Current main and the latest release still route non-Azure custom OpenAI-compatible onboarding through Chat Completions only, while the runtime/config contract already supports `openai-responses`; the related open PR adds a diagnostic command rather than fixing onboarding selection.

## What This Changes

Custom OpenAI-compatible onboarding fails for backends that support the Responses API but not Chat Completions because verification and saved compatibility use `openai-completions` only.

## Best Possible Solution

Expose and verify the existing `openai-responses` adapter in custom-provider interactive and non-interactive onboarding while preserving current Chat Completions, Anthropic, and Azure behavior.

## Reproduction Assessment

Yes, at source level. Configure a non-Azure custom OpenAI-compatible provider whose `/v1/responses` path works but `/v1/chat/completions` returns 503; current onboarding will probe Chat Completions and save `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to expose or detect the already-supported `openai-responses` adapter in custom onboarding and parsing, not to add a separate diagnostic command or rewrite provider runtime transport.
