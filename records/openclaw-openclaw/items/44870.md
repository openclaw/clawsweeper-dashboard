## Summary

Keep open. Current main still lacks an ordinary custom OpenAI Responses onboarding path: non-Azure OpenAI-compatible custom providers are verified and persisted as Chat Completions even though manual config supports `openai-responses`; related #67669 is an open diagnostic command and does not change onboarding.

## What This Changes

Allow custom OpenAI-compatible onboarding to verify and persist Responses-compatible custom providers instead of forcing Chat Completions.

## Best Possible Solution

Custom-provider onboarding should expose Chat Completions and Responses as distinct OpenAI-compatible choices, verify the selected endpoint, persist the matching existing `ModelApi`, and update tests/docs while preserving current defaults.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: choose custom OpenAI-compatible onboarding for a non-Azure `/v1` backend that supports Responses only, and current main builds a `/chat/completions` probe and persists `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing onboarding/config parsing seam to select the already-supported `openai-responses` adapter; a provider runtime rewrite is unnecessary.
