## Summary

Keep open. Current main and the latest release still route custom non-Azure OpenAI-compatible onboarding through Chat Completions, while the existing runtime/config contract already supports `openai-responses` for backends that expose `/v1/responses`.

## What This Changes

The issue requests custom provider onboarding to verify and save OpenAI Responses-compatible endpoints instead of always using Chat Completions for OpenAI-compatible custom providers.

## Best Possible Solution

Expose and verify the existing `openai-responses` adapter in custom-provider onboarding and non-interactive parsing while preserving current Chat Completions, Anthropic, and Azure-specific behavior.

## Reproduction Assessment

Yes, at source level. Configure a non-Azure `/v1` backend that supports `/responses` but rejects `/chat/completions`, then choose custom OpenAI-compatible or Unknown onboarding; current main builds a Chat Completions probe and saves `openai-completions`.

## Solution Assessment

Yes. The narrow maintainable fix is to expose or detect the already-supported `openai-responses` adapter in custom onboarding and parsing, rather than adding a separate diagnostic command or rewriting runtime transport.
