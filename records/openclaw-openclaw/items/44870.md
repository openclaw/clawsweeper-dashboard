## Summary

Keep open. Current main still lacks a first-class custom-provider onboarding path for ordinary OpenAI Responses endpoints: custom OpenAI-compatible setup verifies and persists Chat Completions, while `openai-responses` is supported only through manual config or Azure-specific handling.

## What This Changes

The issue reports that custom OpenAI-compatible provider verification fails with HTTP 503 for a backend whose `/v1/models` endpoint works, and asks why it does not validate like the Anthropic-compatible path.

## Best Possible Solution

Custom provider onboarding should expose Chat Completions and Responses as distinct OpenAI-compatible API choices, verify against the selected endpoint, persist the matching `ModelApi`, and document the choice while preserving the current Chat Completions default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: choose custom OpenAI-compatible onboarding for a non-Azure Responses-only `/v1` backend and OpenClaw builds a `/chat/completions` probe, then persists `openai-completions` with no ordinary custom Responses option to select.

## Solution Assessment

Yes. The narrowest maintainable fix is to extend onboarding/config parsing, verification, tests, and docs for `openai-responses`; the runtime and config schema already support that adapter, so a broader provider runtime rewrite is unnecessary. The related #67669 `custom-test` command may be useful, but it would not fix initial onboarding for this report.
