## Summary

Keep open: current main still passes a package-root cwd into the managed update handoff, and early CLI startup can call `process.cwd()` before the later update-command cwd fallback runs. The related issues are useful context, but this report has a narrower production handoff failure that is still actionable.

## What This Changes

The issue reports a macOS LaunchAgent-managed update handoff failure where `openclaw update --yes` crashes with `uv_cwd` after the service is stopped.

## Best Possible Solution

Harden the managed package-update handoff so the helper and spawned CLI use a durable working directory before service stop/restart, and cover the early CLI `uv_cwd` path with focused regression tests.

## Reproduction Assessment

No: I did not run a live macOS LaunchAgent managed update against current main. Source inspection matches the reported stack because current main can spawn the handoff from a package-root cwd while early entrypoint code can call `process.cwd()` before update fallback logic executes.

## Solution Assessment

Yes: the requested cwd sanitization is the right fix direction, but the maintainable boundary should cover both the handoff helper cwd and early CLI bootstrap rather than only later update-command handling.
