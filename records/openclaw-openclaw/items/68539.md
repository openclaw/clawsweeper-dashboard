## Summary

Keep open. Current main still has inbound audio transcription, but the `openai-codex` media provider is image-only and the batch audio path requires a provider-owned `transcribeAudio` implementation; the existing docs also describe OpenAI speech-to-text through the `openai` REST/API-key path, not Codex OAuth.

## What This Changes

Request support for inbound voice/audio transcription through the `openai-codex` ChatGPT/Codex-authenticated path without a separate OpenAI API key, API charges, or coupling to Codex text-message limits.

## Best Possible Solution

Keep this as provider-owned OpenAI plugin work: either add and document an `openai-codex` audio transcription implementation after proving the Codex-authenticated endpoint, billing, and limit behavior, or document Codex OAuth STT as unsupported and point users to the existing API-key and CLI transcription providers.

## Reproduction Assessment

Yes. Configure `tools.media.audio.models` with `provider: "openai-codex"`, or rely on an `openai-codex` active model for audio, and current main will not produce Codex audio transcription because the provider is image-only and the runner requires `transcribeAudio`.

## Solution Assessment

Unclear. The requested behavior is plausibly the right user-facing outcome, but the safest implementation cannot be chosen until the upstream Codex-authenticated transcription contract and its billing/limit semantics are confirmed.
