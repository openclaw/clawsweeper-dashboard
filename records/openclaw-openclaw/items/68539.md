## Summary

Keep open: current main still lacks Codex-authenticated audio transcription, and the request depends on maintainer-approved proof of the upstream Codex OAuth STT endpoint, billing, and limit semantics.

## What This Changes

The issue requests inbound voice/audio transcription through the `openai-codex` ChatGPT/Codex-authenticated path without an OpenAI API key, API charges, or normal Codex message-limit coupling.

## Best Possible Solution

Keep the work provider-owned in the bundled OpenAI plugin: add and document `openai-codex` audio transcription only after proving the supported auth, endpoint, billing, and limit contract, or document it as unsupported and route users to existing API-key or local CLI STT paths.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives a high-confidence current-state check: `openai-codex` is image-only and all audio provider paths require `transcribeAudio`.

## Solution Assessment

Unclear. The OpenAI plugin is the right ownership boundary, but the specific Codex OAuth transcription implementation should wait for maintainer-approved proof of the upstream endpoint and its billing/limit semantics.
