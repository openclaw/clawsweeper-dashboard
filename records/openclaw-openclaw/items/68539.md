## Summary

Keep open. Current main still requires an audio media-understanding provider to implement `transcribeAudio`, while the OpenAI Codex OAuth provider is registered for image understanding only; docs and the latest release surface also exclude Codex OAuth from speech-to-text support. The prior ClawSweeper keep-open comment remains accurate after checking current source, docs, tests, release state, and adjacent Codex OAuth history.

## What This Changes

The issue requests voice/audio transcription through the OpenAI Codex OAuth transport without an OpenAI API key, API billing, or coupling to normal Codex text-message limits.

## Best Possible Solution

Keep this as provider-owned OpenAI plugin work: add a documented `openai-codex` audio transcription provider only after confirming the supported Codex-authenticated STT endpoint and billing/limit behavior, or document Codex OAuth STT as unsupported and point users to the existing API-key and local-CLI transcription providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `tools.media.audio.models` with `provider: "openai-codex"`; the entry can reach the audio runner, but current main rejects it because the provider has no `transcribeAudio`, while auto-detection skips Codex because its metadata is image-only.

## Solution Assessment

Unclear. The narrow maintainable implementation would live in `extensions/openai`, but reusing the existing OpenAI REST STT path would not meet the no-API-key/no-API-billing requirement; the Codex-authenticated path needs maintainer or live upstream confirmation first.
