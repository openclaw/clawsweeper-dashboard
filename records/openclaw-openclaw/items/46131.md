## Summary

Keep open. The model bug remains on current main and the model-fix direction is useful, but this branch is not mergeable as-is because it mixes in unrelated Telegram retry changes that can replay non-idempotent sends.

## What This Changes

The branch applies provider-level API defaults to inline provider models with tests and also adds Telegram retry handling for 429 and grammY failed-after errors.

## Best Possible Solution

Land a current-main, model-only fix in the inline provider builder that applies provider-policy API defaults before transport resolution, with focused regression tests and no Telegram retry changes.

## Reproduction Assessment

Yes, at source level. An Anthropic inline provider with models but no explicit api still produces an inline model with api undefined on current main, and explicit model resolution then skips the inline branch because it requires inlineMatch?.api.

## Solution Assessment

No, not as this PR branch. The model fix is the right direction, but the maintainable solution is a narrow current-main patch that uses the provider-policy normalization seam and leaves Telegram strict-send retry behavior unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep failed-after envelopes out of strict send retries:** `src/telegram/network-errors.ts:216-218`
  - body: isSafeToRetrySendError gates non-idempotent Telegram sends, but this change treats a generic grammY "failed after N attempts" envelope as safe without nested pre-connect evidence. That wrapper can still represent a post-send transport failure, so retrying here can duplicate visible messages; keep 429 handling separate and require pre-connect codes for strict sends.
