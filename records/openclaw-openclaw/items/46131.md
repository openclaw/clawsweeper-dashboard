## Summary

Keep open. The inline-provider model bug is still present on current main, but this PR branch should not merge as-is because it mixes the valid model fix with an unrelated Telegram retry change that can make non-idempotent sends unsafe.

## What This Changes

The PR applies provider-level API fallback to inline provider models, adds regression tests, and also changes Telegram send retry classification for 429 and grammY failed-after errors.

## Best Possible Solution

Land a current-main, model-only fix in `src/agents/pi-embedded-runner/model.inline-provider.ts` that applies provider-policy API defaults before inline transport construction, with focused tests and no Telegram retry changes.

## Reproduction Assessment

Yes. A source-level reproduction is an Anthropic inline provider with `models` and no explicit `api`: current `buildInlineProviderModels` leaves `api` undefined, and `resolveExplicitModelWithRegistry` then skips the inline branch because `inlineMatch?.api` is false.

## Solution Assessment

No, not as this PR branch. The model-fix direction is right, but the maintainable path is a narrow current-main patch using the provider-policy normalization seam and leaving Telegram retry behavior unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep failed-after envelopes out of strict send retries:** `src/telegram/network-errors.ts:216-218`
  - body: `isSafeToRetrySendError` gates non-idempotent Telegram sends, but this change treats a generic grammY `failed after N attempts` envelope as safe without nested pre-connect evidence. That wrapper can still represent a post-send transport failure, so retrying here can duplicate user-visible messages; keep 429 handling separate and require pre-connect codes for strict sends.
  - confidence: 0.92
