## Summary

Keep this PR open but do not merge it as-is: the provider API inheritance bug is still source-reproducible on current main, but the branch also broadens non-idempotent Telegram send retries in a way current main intentionally avoids and it lacks real behavior proof.

## What This Changes

The branch applies provider-level API defaults to inline provider model entries and adds Telegram send retry handling for 429 and grammY failed-after envelopes.

## Best Possible Solution

Port a model-only fix to current main by applying provider-policy normalization in model.inline-provider.ts with focused regression tests, while leaving Telegram strict-send retry behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current main’s inline builder drops the Anthropic provider-policy API default before explicit model resolution checks inlineMatch?.api, so the reported fallthrough is traceable without running a test in this read-only review.

## Solution Assessment

No, not as this PR branch. The model fix direction is right, but the maintainable solution is a narrow current-main patch using the provider-policy seam and dropping the unrelated Telegram failed-after retry change.

## Review Findings
