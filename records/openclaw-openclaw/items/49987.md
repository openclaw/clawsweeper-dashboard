## Summary

Keep open: the PR is a plausible gateway/CLI improvement, but source review shows the current patch can miss the startup sentinel gate, can write into the wrong state directory, and lacks routing context for the user/session it claims to notify.

## What This Changes

The PR changes `openclaw gateway restart` to write a `restart-sentinel.json` payload after `runServiceRestart()` reports success.

## Best Possible Solution

If bare CLI restart notifications are desired, implement a daemon-env-aware pre-restart handoff with rollback and explicit session routing; otherwise keep `/restart` and the Gateway tool as the supported routed notification paths.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after post-restart checks, while startup only schedules sentinel delivery if the file exists during the post-ready sidecar check. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart success avoids one stale no-op sentinel case, but the maintainable solution needs daemon environment, pre-restart write/rollback timing, and explicit routing semantics or a maintainer decision to use the existing routed restart paths instead.

  - confidence: 0.9

  - body: The payload only contains a generic message and stats. Without `sessionKey`, `deliveryContext`, `threadId`, or continuation data, the sentinel reader falls back to the main session and cannot reliably notify the user or active chat that initiated the restart.
