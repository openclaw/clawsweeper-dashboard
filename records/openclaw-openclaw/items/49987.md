## Summary

Keep open: the restart-notification idea is plausible, but this PR is not merge-ready because the sentinel can be written after the restarted gateway has already skipped the startup check, it writes through the CLI environment instead of the daemon environment, it lacks routable session context, it is merge-conflicting, and no real after-fix behavior proof was provided.

## What This Changes

The PR adds a restart-sentinel write to `openclaw gateway restart` after `runServiceRestart()` succeeds so the restarted gateway can emit a post-restart CLI notification.

## Best Possible Solution

If CLI restart notifications are desired, implement a daemon-env-aware pre-restart handoff with rollback and explicit session routing; otherwise keep the existing Gateway tool and slash-command restart paths as the routed post-restart notification surfaces.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after post-restart checks, startup only checks for an existing sentinel during its sidecar pass, and the payload omits routable session fields. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the earlier no-op stale-sentinel bug, but the maintainable solution needs daemon environment ownership, earlier write/rollback timing, and explicit routing semantics.
