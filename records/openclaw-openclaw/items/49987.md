## Summary

Keep open: the feature direction is useful, but this patch can miss the restart notification because it writes the sentinel after the restarted gateway may already have passed startup sentinel detection, and it can write to the CLI state path instead of the daemon state path.

## What This Changes

The PR changes `openclaw gateway restart` to write an existing restart-sentinel payload after `runServiceRestart()` returns true.

## Best Possible Solution

Add a narrow lifecycle seam that writes the sentinel after restart preflight commits to a real restart but before the new gateway can pass startup sentinel detection, resolves the path with the daemon service environment, cleans up failed or no-op paths, and adds focused coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection is enough: current main has no CLI restart sentinel, and the PR writes it after `runServiceRestart()` even though startup only schedules delivery when the file already exists during `sidecars.restart-sentinel`.

## Solution Assessment

No. The requested direction is sound, but this implementation writes too late and with the wrong environment; the safer fix is to hook the sentinel write into the restart lifecycle before daemon startup can observe the file and use daemon env path resolution.
