## Summary

Keep open: current main still lacks CLI restart sentinel behavior, but this PR’s late write can miss startup delivery and uses the CLI environment for the sentinel path. The PR also lacks contributor-supplied real behavior proof for a real CLI gateway restart notification.

## What This Changes

The PR imports restart-sentinel helpers and writes a `restart` sentinel after `runServiceRestart()` succeeds in `runDaemonRestart()`.

## Best Possible Solution

Add a daemon-aware restart hook that records the sentinel in the daemon state after restart preflight commits to a real restart but before replacement startup can pass the sentinel gate, with cleanup for failed or no-op paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR defects: the branch writes the sentinel only after `runServiceRestart()` returns, while startup checks for the file during the post-ready sidecar; I did not run a live restart in this read-only review.

## Solution Assessment

No. Moving the write after `runServiceRestart()` avoids the stale no-op sentinel, but the durable signal needs to be written early enough for startup to see it and with the daemon service environment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Write the sentinel before startup can pass the gate:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new `writeRestartSentinel(...)` call runs only after `runServiceRestart()` has returned. That helper can restart the service, run `postRestartCheck`, inspect loaded state, and emit success before returning, while startup schedules sentinel delivery only when `restart-sentinel.json` already exists during its post-ready sidecar check; the replacement gateway can pass that gate before this line writes the file, so no notification is delivered.
  - confidence: 0.92
- **[P2] Resolve the sentinel path with the daemon environment:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: This call does not pass an environment, so `writeRestartSentinel` resolves the file path from the CLI process `process.env`. Installed gateways can run with persisted `OPENCLAW_STATE_DIR` and `OPENCLAW_CONFIG_PATH` from the service command; a CLI shell using a different profile or state dir can write a sentinel the restarted daemon will never read.
  - confidence: 0.9
