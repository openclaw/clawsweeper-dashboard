## Summary

Keep this PR open, but it still needs changes before merge: current main still lacks a CLI restart sentinel, while this branch writes the sentinel after the restart path can already have passed startup sentinel detection and uses the invoking CLI environment instead of the daemon environment. The PR also lacks after-fix real behavior proof.

## What This Changes

The PR imports restart-sentinel helpers into the daemon CLI lifecycle and writes a `Gateway restarted via CLI` sentinel after `runServiceRestart()` returns true.

## Best Possible Solution

Add a narrow restart lifecycle seam that prepares the sentinel after restart preflight commits to a real restart but before the replacement gateway can pass sentinel startup detection, resolves the path with daemon service environment, and cleans up failed or no-op paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has no CLI restart sentinel, and the PR writes its sentinel after `runServiceRestart()` even though gateway startup only schedules delivery when the file is already present.

## Solution Assessment

No. The requested direction is useful, but this implementation is not the best fix because it writes too late in the lifecycle and resolves the sentinel path from the CLI environment instead of the daemon environment.

  - confidence: 0.9
