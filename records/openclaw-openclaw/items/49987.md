## Summary

Keep open for maintainer review: the idea is plausible, but the current patch is not merge-ready because it can miss the startup sentinel gate, writes from the CLI environment instead of the daemon environment, lacks routable session context, and still has no real behavior proof.

## What This Changes

The PR changes `openclaw gateway restart` to write a restart sentinel after `runServiceRestart()` returns success so the restarted gateway can notify the user.

## Best Possible Solution

If bare CLI restart notifications are desired, implement a daemon-env-aware pre-restart handoff with rollback and explicit session routing; otherwise keep `/restart` and the Gateway tool as the supported post-restart notification paths.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after post-restart checks, while startup only schedules sentinel delivery if the file exists during the post-ready sidecar check. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the stale no-op sentinel, but the maintainable path needs daemon environment, earlier write/rollback timing, and explicit routing semantics or a decision to use the existing routed restart paths instead.
