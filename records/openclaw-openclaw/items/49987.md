## Summary

Keep open: current main does not implement this exact CLI restart notification, but this PR is not merge-ready because source review shows the sentinel can be written too late, with the wrong state directory, and without routable session context; required real behavior proof is also missing.

## What This Changes

The PR changes `openclaw gateway restart` to write a restart sentinel after `runServiceRestart()` reports success.

## Best Possible Solution

If maintainers want bare CLI restarts to notify users, add a daemon-aware pre-restart sentinel or handoff hook with rollback, daemon env, routed session context, focused tests, and real CLI restart proof; otherwise keep the existing Gateway tool restart continuation as the supported notification path.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after post-restart checks, while startup only schedules sentinel delivery if the file already exists during the post-ready sidecar check. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the stale no-op sentinel, but the maintainable fix needs daemon env, earlier write/rollback timing, and explicit routing semantics or an explicit decision to use the existing Gateway tool continuation instead.
