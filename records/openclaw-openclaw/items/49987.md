## Summary

Keep open: current main still lacks this exact CLI restart sentinel hook, and the PR is not merge-ready because the sentinel is written too late for startup consumption, uses the invoking CLI environment, lacks routable session context, and has no real behavior proof.

## What This Changes

The PR changes `openclaw gateway restart` to await `runServiceRestart()` and then write a generic restart sentinel when the helper returns true.

## Best Possible Solution

If maintainers want bare CLI restarts to notify users, add a daemon-aware restart lifecycle hook that writes or rolls back a routed sentinel before restart emission/startup can pass the sentinel gate; otherwise close this in favor of the existing Gateway tool continuation path.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR writes the sentinel after `runServiceRestart()` returns, while startup only schedules sentinel delivery if the file already exists during the post-ready sidecar check. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the stale no-op sentinel, but the maintainable fix needs daemon env, earlier write/rollback timing, and explicit routing semantics or an explicit decision to use the existing Gateway tool continuation instead.
