## Summary

Keep open: current main still lacks a CLI restart sentinel hook, but this PR’s current hook can miss startup delivery, can write to the wrong state directory, and lacks real behavior proof.

## What This Changes

The PR modifies CLI gateway restart lifecycle code to write a restart sentinel after `runServiceRestart()` returns true.

## Best Possible Solution

Add a daemon-aware restart lifecycle hook that records a routable sentinel after restart preflight commits to a real restart and before replacement startup can pass the sentinel gate, using the daemon state environment and cleanup for no-op or failed paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR defects: the branch writes the sentinel after `runServiceRestart()` returns, while startup checks for the file during the post-ready sidecar; I did not run a live restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the stale no-op sentinel, but the durable signal must be written early enough for startup to see it, into the daemon state directory, and with a routable session/delivery context if it is meant to notify a user.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Write the sentinel before startup can pass the gate:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new `writeRestartSentinel(...)` call runs only after `runServiceRestart()` returns. That helper can restart the service, run the post-restart check, inspect loaded state, and emit success before returning, while startup schedules sentinel delivery only if the file already exists during its post-ready sidecar check; the replacement gateway can pass that gate before this line writes the file, so the notification is missed.
  - confidence: 0.92
- **[P2] Resolve the sentinel path with the daemon environment:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: This call does not pass an environment, so `writeRestartSentinel` resolves the file path from the CLI process `process.env`. Installed gateways can run with persisted `OPENCLAW_STATE_DIR` and `OPENCLAW_CONFIG_PATH` from the service command, so a CLI shell using a different profile or state dir can write a sentinel the restarted daemon will never read.
  - confidence: 0.9
- **[P2] Persist a routable session with the sentinel:** `src/cli/daemon-cli/lifecycle.ts:273-278`
  - body: The payload only contains the generic restart message and stats, with no `sessionKey`, `deliveryContext`, or thread id. The sentinel reader falls back to enqueuing a main-session system event when `sessionKey` is absent, so this does not deliver the promised notification to the initiating chat/user and may not wake any active session.
  - confidence: 0.82
