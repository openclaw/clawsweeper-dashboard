## Summary

Keep this PR open because current main still lacks the requested CLI restart notification behavior, but the proposed patch is not mergeable as written. The PR writes the sentinel after `runServiceRestart()` returns and with the CLI environment, while the gateway only schedules sentinel delivery if the file already exists during startup and installed services may read a different state directory.

## What This Changes

This PR changes `src/cli/daemon-cli/lifecycle.ts` to write a best-effort `Gateway restarted via CLI` restart sentinel after `openclaw gateway restart` reports success.

## Best Possible Solution

Add a narrow CLI lifecycle seam for gateway restarts: complete config/service/no-op preflight first, resolve the sentinel path with the daemon service environment when available, write the sentinel before the restart is scheduled or signaled, and remove it on rejected or failed restart paths. Keep the behavior generic to the gateway lifecycle and cover it with focused lifecycle and sentinel tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no CLI restart sentinel write, and the PR writes the file after `runServiceRestart()` even though startup only schedules sentinel delivery when the file already exists during `sidecars.restart-sentinel`.

## Solution Assessment

No. The PR addresses the right behavior gap, but writing only after `runServiceRestart()` returns is not the best fix because startup can miss the sentinel and custom daemon state directories can write/read different files; a pre-restart write after preflight with cleanup is safer.

- **[P2] Write the sentinel before startup can miss it:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new success sentinel is written only after `runServiceRestart()` returns. Gateway startup only schedules `scheduleRestartSentinelWake()` when `hasRestartSentinel()` is true during `sidecars.restart-sentinel`, so a fast service-manager restart can pass that check before this write lands; the file then waits for a later unrelated start and the CLI restart notification is missed or delayed.

For profile/custom-state services this writes into the CLI state dir instead of the daemon state dir, so the restarted gateway never sees it.
  - confidence: 0.9
