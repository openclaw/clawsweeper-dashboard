## Summary

Keep this PR open, but the current patch is not merge-ready: it adds the useful CLI restart notification path, yet writes the sentinel after the restarted gateway may already have passed the startup sentinel gate and uses the invoking CLI environment instead of the daemon service environment.

## What This Changes

The PR adds restart-sentinel imports to `src/cli/daemon-cli/lifecycle.ts` and writes a `Gateway restarted via CLI` sentinel after `runServiceRestart()` returns true.

## Best Possible Solution

Add a narrow daemon restart lifecycle hook that prepares the sentinel after restart preflight commits to a real restart but before the replacement gateway can pass startup's sentinel gate, using the daemon service environment and cleaning up no-op or failed paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR writes the sentinel only after `runServiceRestart()` returns, while startup only schedules delivery if the sentinel file exists during the post-ready `sidecars.restart-sentinel` check; I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. The requested behavior is useful, but this implementation is not the best fix because it writes too late for startup delivery and resolves the sentinel path from the CLI process environment instead of the daemon's persisted environment.
