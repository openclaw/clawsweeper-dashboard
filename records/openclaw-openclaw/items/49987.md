## Summary

Keep open: the requested CLI restart notification is still not implemented on current main, but this PR's current patch writes the sentinel too late and with the wrong environment, so it should not merge as-is.

## What This Changes

The PR imports restart-sentinel helpers into the daemon CLI restart path and writes a `Gateway restarted via CLI` sentinel after `runServiceRestart()` returns true.

## Best Possible Solution

Add a narrow daemon restart lifecycle seam that writes the sentinel after restart preflight commits to a real restart but before the replacement gateway's startup sentinel gate, using the daemon service environment and cleanup on failed or no-op paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main has no CLI restart sentinel, and the PR writes its sentinel after `runServiceRestart()` even though startup only checks for the file during the restart-sentinel sidecar. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. The requested direction is useful, but this implementation is not the best fix because it writes too late in the lifecycle and resolves the sentinel path from the CLI environment instead of the daemon environment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Write the sentinel before startup can pass the gate:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new success sentinel is written only after `runServiceRestart()` returns. For completed or unmanaged restarts, that path has already waited for the replacement gateway, while startup only schedules restart-sentinel delivery if the file exists during `sidecars.restart-sentinel`; this can leave the notification for a later unrelated start.
  - confidence: 0.92
- **[P2] Use the daemon environment for the sentinel path:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: `writeRestartSentinel(sentinelPayload)` defaults to the invoking CLI `process.env`. Installed gateways can run with persisted `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`, so profile or custom-state installs may write the sentinel where the CLI looks instead of where the restarted daemon reads.
  - confidence: 0.9
