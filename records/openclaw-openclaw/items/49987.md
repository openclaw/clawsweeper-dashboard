## Summary

Keep open for maintainer review: the PR targets a real restart-notification gap, but it is merge-conflicting, lacks required real behavior proof, and the current patch can miss or misroute the sentinel because it writes a generic payload after restart completion.

## What This Changes

The PR changes `runDaemonRestart` to write a best-effort restart sentinel after `runServiceRestart()` succeeds so a restarted gateway can announce `Gateway restarted via CLI`.

## Best Possible Solution

If bare CLI restart notifications are desired, implement a daemon-env-aware pre-restart handoff with rollback and explicit routing; otherwise keep routed restart surfaces as the supported notification path.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after post-restart checks, while startup only schedules sentinel delivery if the file exists during the post-ready sidecar check. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the stale no-op sentinel, but the maintainable path needs daemon environment, earlier write/rollback timing, and explicit routing semantics or a decision to use existing routed restart paths instead.
