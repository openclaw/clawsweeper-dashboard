## Summary

Keep open. The PR is focused, but it is not merge-ready because the sentinel is written after the restarted gateway's startup sentinel check, uses the invoking CLI environment rather than the daemon environment, omits routable session context, and still lacks real after-fix behavior proof.

## What This Changes

The PR imports restart-sentinel helpers into daemon lifecycle code and writes a generic `cli.restart` sentinel after `runServiceRestart()` returns true.

## Best Possible Solution

Implement bare CLI restart notifications only if maintainers want them, using a daemon-env-aware pre-restart handoff with rollback and explicit target/session semantics, or keep post-restart notices limited to the existing routed Gateway tool and chat command paths.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after restart/post-check work, startup only schedules a sentinel wake if the file exists during that sidecar pass, and the payload omits routable session fields. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Reusing the sentinel subsystem is plausible, but writing after restart completion with no daemon environment or routing context is not the narrow maintainable fix.
