## Summary

Keep this PR open. Current main still lacks the requested CLI restart sentinel for `openclaw gateway restart`, but the PR patch should not land as-is because it writes the sentinel after the restarted gateway may already have skipped startup sentinel scheduling, and it stores the sentinel using the invoking CLI environment instead of the daemon service environment.

## What This Changes

The PR imports restart-sentinel helpers into `src/cli/daemon-cli/lifecycle.ts` and writes a best-effort `Gateway restarted via CLI` sentinel after `runServiceRestart()` returns `true`.

## Best Possible Solution

Add a narrow lifecycle seam for CLI gateway restarts: complete config/no-op preflight first, resolve the sentinel location with the daemon service environment when available, write the sentinel before the restart is scheduled or signaled, and remove it on rejected or failed restart paths. Cover managed, scheduled, unmanaged, no-op, failure-cleanup, and custom-state-dir cases with focused lifecycle tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no CLI restart sentinel write, and the PR writes the sentinel after `runServiceRestart()` even though the gateway only schedules sentinel delivery if the file exists during startup; the daemon-env mismatch follows directly from `writeRestartSentinel` defaulting to `process.env`.

## Solution Assessment

No. Moving the write after `runServiceRestart()` fixes the original no-op false-positive review comment but is not the best fix because it can miss startup consumption and use the wrong state directory; the safer solution is a pre-restart write after preflight/no-op checks with cleanup on failure.
