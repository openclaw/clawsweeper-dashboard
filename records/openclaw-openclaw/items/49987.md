## Summary

Keep open: current main still lacks the CLI restart-sentinel behavior, and this PR targets the right gateway lifecycle gap, but the current patch has two concrete delivery defects that should block merge.

## What This Changes

The PR changes `openclaw gateway restart` to write a best-effort `Gateway restarted via CLI` restart sentinel after `runServiceRestart()` reports success.

## Best Possible Solution

Add a narrow CLI lifecycle seam that writes the sentinel after restart preflight but before scheduling or signaling the restart, uses the daemon service environment when available, removes the file on rejected or failed restart paths, and covers the behavior with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection is enough: current main has no CLI restart sentinel write, and the PR writes the sentinel after `runServiceRestart()` even though startup only schedules delivery when the file exists during startup.

## Solution Assessment

No. The requested direction is sound, but this implementation should write before the restarted gateway can pass the sentinel check and should resolve the sentinel path from the daemon environment rather than the CLI shell.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Write the sentinel before startup can miss it:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new success sentinel is written only after `runServiceRestart()` returns. Gateway startup only schedules restart-sentinel delivery when the file already exists during `sidecars.restart-sentinel`, so a fast service-manager restart can pass that check before this write lands and the notification will be delayed until a later unrelated start.
  - confidence: 0.92
- **[P2] Resolve the sentinel path with daemon env:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: `writeRestartSentinel(sentinelPayload)` defaults to `process.env`, but installed gateways can run with `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` from the persisted service command. In profile/custom-state services this writes the sentinel where the CLI would look, not where the restarted daemon will read it.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/cli/daemon-cli/lifecycle.ts:267`
  - body: This is a user-visible CLI/gateway notification feature, but the PR only edits the lifecycle code. Repo policy requires a `CHANGELOG.md` entry for user-facing feature, fix, and perf changes.
  - confidence: 0.86
