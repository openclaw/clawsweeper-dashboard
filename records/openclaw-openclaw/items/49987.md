## Summary

Keep open: current main does not implement the CLI restart sentinel, and this PR still has blocking lifecycle problems around timing, state-directory resolution, routable delivery context, and missing real behavior proof.

## What This Changes

The PR changes `openclaw gateway restart` to write a restart sentinel after `runServiceRestart()` returns true so startup can send a post-restart notification.

## Best Possible Solution

Add a daemon-aware restart lifecycle hook that writes or cleans a routed sentinel after restart preflight commits and before replacement startup can pass the sentinel gate, with focused tests and real CLI restart proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: PR head writes the sentinel after `runServiceRestart()` returns, while startup only schedules delivery if the file exists during the post-ready sidecar check. I did not run a live restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the stale no-op sentinel, but the durable signal must be written early enough for startup to see it, into the daemon state directory, and with routable session/delivery context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Write the sentinel before startup can pass the gate:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new write runs only after `runServiceRestart()` returns. That helper already waits through post-restart checks and emits success before returning, while startup only schedules sentinel delivery if the file exists during `sidecars.restart-sentinel`; the replacement gateway can pass that check before this line writes the file, dropping the notification.
  - confidence: 0.92
- **[P2] Write the sentinel into the daemon state directory:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: `writeRestartSentinel` defaults to the CLI process environment, but installed gateways can run with persisted `OPENCLAW_STATE_DIR`/`OPENCLAW_CONFIG_PATH` from the service command. In profile or custom-state installs, this writes into the CLI state dir instead of the daemon state dir, so the restarted gateway never reads it.
  - confidence: 0.9
- **[P2] Include routed session data in the sentinel:** `src/cli/daemon-cli/lifecycle.ts:273-278`
  - body: The payload contains only a generic message and stats, with no `sessionKey`, `deliveryContext`, `threadId`, or continuation. The sentinel reader falls back to a main-session system event when `sessionKey` is absent, so this does not reliably notify the initiating user or active session.
  - confidence: 0.82
