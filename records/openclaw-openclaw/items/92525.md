## Summary

Current main and v2026.6.6 intentionally reserve goal clearing and replacement for operator/session controls, while the two contributor implementations closed unmerged to request this exact product decision. The issue remains necessary because no open or merged work owns the proposed authority and persistent-history change.

## What This Changes

Add a model-callable `clear_goal` operation for non-active goals and atomically retain a bounded cleared-goal history in canonical session state.

## Best Possible Solution

Make an explicit owner decision on whether models may clear a narrowly defined terminal-state subset; if approved, use one atomic canonical session-store transition with bounded history and document retention, reset, visibility, and upgrade semantics before reopening an implementation PR.

## Reproduction Assessment

Not applicable: source, docs, tests, and v2026.6.6 agree that model-side clearing is intentionally unsupported, so this is a proposed contract change rather than a reproducible bug.

## Solution Assessment

Unclear pending the product decision. Atomic canonical-store rotation is substantially safer than the rejected JSONL sidecar, but it is not yet the best complete solution because model authority, clearable states, history visibility, retention, and reset behavior remain undecided.

## Risks / Open Questions

- Model-callable clearing would reverse a documented operator/session authority boundary and allow the agent to remove goal state without an explicit operator action.
- A `clearedGoals` collection would add a durable session data model whose retention, reset, export, synchronization, visibility, and upgrade semantics are not yet decided.
- Treating paused or limit-reached goals as clearable may discard resumable workflow state; the safe state set is a product choice rather than a mechanical implementation detail.
