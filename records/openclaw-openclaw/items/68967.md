## Summary

Keep open: current main still lacks Google Chat per-thread session isolation, and this PR is a plausible implementation path, but the submitted patch currently bypasses the shared replyToMode policy by forcing inbound threads after that policy has cleared implicit replies.

## What This Changes

The PR adds optional `channels.googlechat.sessionThread` config, thread-scoped Google Chat session keys, forced reply threading, docs/generated metadata, changelog text, and unit coverage.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session keys, and preserves replyToMode plus explicit reply-target semantics while coordinating with the thread-delivery path.

## Reproduction Assessment

Yes, at source level. PR head computes `threadForReply` from `payload.replyToId ?? forcedThreadName`, while current shared reply filtering intentionally clears implicit `replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but inbound-thread fallback needs to respect shared reply policy instead of being restored unconditionally in channel delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
