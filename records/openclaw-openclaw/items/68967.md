## Summary

Keep open. Current main still lacks `sessionThread` and this PR is useful, but the latest branch is not merge-ready because it targets pre-turn-kernel Google Chat context wiring and still bypasses explicit Google Chat `replyToMode: "off"` with the forced inbound-thread fallback.

## What This Changes

This PR adds optional Google Chat `sessionThread` config, derives thread-scoped session keys from inbound `thread.name`, forces replies into the inbound thread, updates schema/docs/generated metadata/changelog, and adds helper tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that adds opt-in per-thread sessions, preserves explicit `replyToMode: "off"`, routes through the current turn kernel, updates config/docs/changelog, and leaves broader streamed/follow-up thread propagation to #69422.

## Reproduction Assessment

Yes. Source inspection on current main shows Google Chat sessions are still keyed by space id and `sessionThread` is absent; the PR body also gives a concrete two-thread Google Chat reproduction, though I did not rerun live Chat in this read-only pass.

## Solution Assessment

No as submitted. The hashed thread-session direction is maintainable, but the PR needs turn-kernel adaptation and an explicit `replyToMode: "off"` guard before it is the best mergeable fix.
