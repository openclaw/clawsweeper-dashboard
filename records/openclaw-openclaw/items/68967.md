## Summary

Keep open. The per-thread Google Chat session feature is not on current main and the PR has sufficient real-behavior proof, but the submitted diff still bypasses Google Chat `replyToMode` by restoring the inbound thread after shared reply filtering.

## What This Changes

Adds a Google Chat `sessionThread` config option, thread-derived session keys, forced reply threading, config metadata, docs, changelog, and session-key tests.

## Best Possible Solution

Land a rebased Google Chat implementation that preserves space-based routing, adds opt-in per-thread session storage, and uses a policy-preserving thread delivery path from https://github.com/openclaw/openclaw/pull/74235 or an equivalent fix.

## Reproduction Assessment

Yes, source-level. Current main strips implicit `replyToId` in `replyToMode: "off"`, while the PR re-adds the inbound Google Chat thread through `forcedThreadName` after that filtering.

## Solution Assessment

No, not as submitted. The per-thread session-key direction is reasonable, but outbound thread fallback must honor `replyToMode` or delegate to an equivalent policy-aware Google Chat thread-routing path.
