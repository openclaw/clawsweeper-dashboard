## Summary

Keep open. The PR is useful and current main still lacks Google Chat per-thread sessions, but the latest diff still forces the inbound thread after the shared reply policy has intentionally removed implicit `replyToId` for `replyToMode: "off"`, so one merge-blocking behavior issue remains.

## What This Changes

Adds a Google Chat `sessionThread` boolean, derives thread-scoped session keys, forces replies into the inbound thread, updates config/schema/generated docs/changelog, and adds session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that adds opt-in per-thread sessions, respects explicit flat reply mode, updates config/docs/changelog, and leaves broader streamed/follow-up thread propagation to #69422.

## Reproduction Assessment

Yes for source-level behavior. Current main has no `sessionThread` config and uses the space-derived route session key; the PR body also provides a concrete two-thread Google Chat scenario, though I did not rerun live Chat in this read-only pass.

## Solution Assessment

No as submitted. The hashed `dispatchSessionKey` direction is maintainable, but the forced reply-thread fallback must respect `replyToMode: "off"` before the PR is mergeable.
