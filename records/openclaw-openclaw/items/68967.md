## Summary

Keep open. Current main has threaded Google Chat reply plumbing, but it still does not expose the opt-in per-thread Google Chat session mode; this PR addresses a real remaining gap and incorporated several prior review fixes, but the branch is not merge-ready against current main and has changelog/JSDoc defects.

## What This Changes

The branch adds a Google Chat `sessionThread` option that hashes inbound thread names into session keys, threads replies via inbound `thread.name`, updates config/schema/docs/changelog metadata, and adds a session-key unit test.

## Best Possible Solution

Rebase the PR onto current main, keep the session partitioning inside the Google Chat plugin, use the current turn-kernel route/context fields to dispatch and persist the derived session key without changing space-based route matching, move reply fallback changes into the split reply-delivery module, and fix the changelog/JSDoc before maintainer review.

## Reproduction Assessment

Yes. The PR body gives a concrete two-thread Google Chat space reproduction, current main source confirms no `sessionThread` option exists, and the proposed helper test covers the key session-splitting behavior; live Google Chat verification was not rerun during this read-only pass.

## Solution Assessment

Unclear as currently submitted. The plugin-owned opt-in session-key suffix is a plausible narrow solution, but the branch targets a pre-refactor monitor path and should be adapted to current main before deciding whether the boolean `sessionThread` name is the final config contract.
