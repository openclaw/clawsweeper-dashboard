## Summary

Keep open: the per-thread Google Chat session feature is still absent from current main and this PR remains the active implementation, but the current head is conflicting and still bypasses Google Chat flat-reply policy by restoring a thread target after shared reply filtering clears it.

## What This Changes

The PR adds an opt-in Google Chat `sessionThread` config, derives thread-scoped session keys, forces replies toward inbound Chat threads, and updates config metadata, docs, changelog, and tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps routing space-based, adds opt-in per-thread session storage, and delegates thread delivery to the policy-preserving path from https://github.com/openclaw/openclaw/pull/74235 or an equivalent fix.

## Reproduction Assessment

Yes, at source level: the PR restores an inbound Google Chat thread when `payload.replyToId` is absent, while current main intentionally clears implicit reply targets for Google Chat accounts in flat reply mode. I did not run live Google Chat in this read-only review.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but outbound thread fallback must respect post-policy reply semantics or be replaced by a thread-routing fact that the shared policy can clear.
