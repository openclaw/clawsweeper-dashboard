## Summary

Keep this PR open: current main still lacks the requested Google Chat per-thread session option, but the submitted patch has a blocking reply-threading correctness issue and overlaps the active Google Chat thread-delivery work.

## What This Changes

The PR adds a `channels.googlechat.sessionThread` boolean, derives Google Chat session keys from inbound thread names when enabled, forces replies toward inbound threads, and updates config/docs/generated metadata plus tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session keys, preserves explicit reply targets, and applies inbound-thread fallback only when reply policy allows it.

## Reproduction Assessment

Yes, at source level. Current main lacks the `sessionThread` surface, and PR head restores `forcedThreadName` after shared reply threading can clear `payload.replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but delivery fallback must respect reply policy and be reconciled with the related Google Chat thread-delivery work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
