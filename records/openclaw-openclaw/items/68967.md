## Summary

Keep open: the PR contains useful Google Chat per-thread session work that is not on current main, but the branch still has a correctness bug because it can force replies back into the inbound thread after `replyToMode: "off"` has cleared the reply target.

## What This Changes

Adds a Google Chat `sessionThread` boolean, thread-derived session keys, inbound-thread reply fallback, config/docs/generated metadata/changelog updates, and session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that keeps route matching space-based, adds an approved opt-in per-thread session key, preserves explicit reply targets, and only applies inbound-thread fallback when reply policy allows it.

## Reproduction Assessment

Yes, source-level. Current main lacks `sessionThread`, and the PR head restores `message.thread?.name` after shared reply filtering can clear `payload.replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but the delivery fallback must respect `replyToMode` and stay coordinated with the active Google Chat thread-delivery work.
