## Summary

Keep open: the per-thread session idea is useful and has real Google Chat proof, but the branch still has a blocking reply-policy regression because it can re-add an inbound thread after shared `replyToMode` filtering removed it. The branch also overlaps active Google Chat thread-delivery work, so it needs a focused repair and rebase rather than cleanup closure.

## What This Changes

The branch adds an optional Google Chat `sessionThread` config with hashed per-thread session keys, config/docs/test updates, and fallback delivery to the inbound Chat thread.

## Best Possible Solution

Land this as a clean opt-in session-isolation feature after rebasing onto the final Google Chat thread-delivery path, preserving `replyToMode` compatibility and documenting/tests for both default space-scoped and opt-in thread-scoped sessions.

## Reproduction Assessment

Yes for the blocking review finding at source level: current main filters `replyToId` according to `replyToMode`, and the PR reintroduces the inbound thread when that filtered field is absent. I did not run a live Google Chat webhook in this read-only review.

## Solution Assessment

No, not as currently patched. The per-thread session primitive is reasonable, but reply delivery must preserve existing `replyToMode` semantics and the branch should be reconciled with the active Google Chat thread-delivery PR before merge.
