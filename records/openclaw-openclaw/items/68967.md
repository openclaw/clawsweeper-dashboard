## Summary

Keep open: the Google Chat per-thread session feature is still useful and not implemented on current main, but the PR is not mergeable as submitted because its forced inbound-thread fallback can bypass flat reply mode after shared reply filtering has cleared `payload.replyToId`.

## What This Changes

Adds `channels.googlechat.sessionThread`, derives thread-hashed session keys, forces replies toward the inbound Google Chat thread, updates config/docs/generated metadata/changelog, and adds session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that adds opt-in per-thread sessions, keeps route matching space-based, preserves explicit reply targets, and only applies inbound-thread fallback when flat reply mode has not opted out.

## Reproduction Assessment

Yes, at source level. Current main lacks `sessionThread`, and the PR head clearly shows `threadForReply = payload.replyToId ?? forcedThreadName` after shared reply filtering can clear `payload.replyToId`.

## Solution Assessment

No as submitted. The hashed session-key approach is maintainable, but forced inbound-thread fallback must not bypass an account configured for flat replies; a narrow delivery gate is the safer fix.
