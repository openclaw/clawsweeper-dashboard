## Summary

Keep open: the Google Chat per-thread session idea has useful real-world proof, but the branch is not merge-ready against current main because it misses the active schema file, can reintroduce unsafe DM thread metadata, bypasses replyToMode when falling back to the inbound thread, widens a public SDK helper for one channel, and edits release-owned changelog content.

## What This Changes

The PR adds an opt-in Google Chat `sessionThread` setting, thread-scoped session key derivation, reply-thread delivery fallback, config/docs/generated metadata, and a focused session-key test.

## Best Possible Solution

Rebase and narrow the PR so `sessionThread` is added only after maintainer approval, validated in `src/config/zod-schema.providers-googlechat.ts`, preserves current DM and `replyToMode` behavior, avoids accidental public SDK expansion, and lets the narrower Google Chat reply-delivery PR own thread-target preservation.

## Solution Assessment

No. The per-thread session approach is plausible, but this branch is not the best current-main solution until it is rebased, keeps reply delivery separate or aligned with the narrower active PR, and gets maintainer direction on the new config and SDK surfaces.
