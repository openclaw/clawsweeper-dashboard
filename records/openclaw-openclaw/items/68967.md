## Summary

Keep open: the Google Chat per-thread session feature is not on current main, but this PR is not mergeable as submitted because it can bypass Google Chat `replyToMode: "off"` and it currently conflicts with main while overlapping active thread-delivery work.

## What This Changes

The branch adds a `channels.googlechat.sessionThread` boolean, hashes inbound Google Chat thread names into session keys, forces replies toward inbound threads, updates config/docs/generated metadata/changelog, and adds session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that keeps route matching space-based, adds opt-in per-thread session keys, preserves explicit reply targets, and applies inbound-thread fallback only when reply policy allows it.

## Reproduction Assessment

Yes, at source level. Current main lacks `sessionThread`, and the PR head has `threadForReply = payload.replyToId ?? forcedThreadName` after shared reply filtering can clear `replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but the delivery fallback must respect `replyToMode` and the broader thread-delivery work should stay aligned with https://github.com/openclaw/openclaw/pull/74235.
