## Summary

Keep open: the per-thread Google Chat session feature is not on current main, but the PR still has a blocking reply-policy regression because it restores the inbound thread after shared `replyToMode` filtering has intentionally cleared it.

## What This Changes

The PR adds an optional `channels.googlechat.sessionThread` config, derives thread-scoped Google Chat session keys, forces replies toward the inbound thread, and updates config metadata, docs, changelog, and tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session storage, and lets policy-preserving thread delivery come from https://github.com/openclaw/openclaw/pull/74235 or an equivalent shared-thread fact.

## Reproduction Assessment

Yes, at source level for the review finding: PR head restores `forcedThreadName` when the filtered payload has no `replyToId`, while current main clears implicit replies for Google Chat flat mode.

## Solution Assessment

No as submitted. The per-thread session-key direction is useful, but outbound thread fallback must respect the post-policy payload or be replaced by the thread-routing model from the related delivery PR.
