## Summary

Keep open. The per-thread Google Chat session work is useful and has sufficient real Google Chat screenshot proof, but the branch is merge-conflicting and the reply-delivery half still bypasses existing reply-threading policy; it also overlaps the focused Google Chat thread-delivery PR that should own that behavior.

## What This Changes

The branch adds `channels.googlechat.sessionThread`, derives hash-suffixed Google Chat session keys per inbound thread, forwards thread metadata on replies, and updates config schema, docs, generated baseline, changelog, and tests.

## Best Possible Solution

Land a rebased Google Chat change that keeps per-thread session storage opt-in and default-off, preserves space-based route matching, and delegates outbound thread placement to a replyToMode-aware Google Chat thread-routing path or splits that work to the canonical thread-delivery PR.

## Reproduction Assessment

Yes, at source level: the PR restores an inbound Google Chat thread when `payload.replyToId` is absent, while current main's reply threading policy can intentionally remove that field. I did not run live Google Chat in this read-only review.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but outbound thread fallback must respect the existing reply policy or be split into the canonical Google Chat thread-delivery work.
