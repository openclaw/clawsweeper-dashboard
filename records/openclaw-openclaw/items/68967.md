## Summary

Keep open: current main still lacks the Google Chat `sessionThread` feature, and this PR is the only live implementation found, but the patch still has a concrete reply-policy bypass that should be repaired before merge.

## What This Changes

The PR adds a Google Chat `sessionThread` option, thread-derived session keys, forced inbound-thread reply delivery, config/docs metadata, changelog text, and unit coverage.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session storage, and preserves `replyToMode` plus explicit reply-target semantics.

## Reproduction Assessment

Yes, at source level for the review finding. PR head restores `forcedThreadName` when `payload.replyToId` is absent, while current main intentionally clears implicit `replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but forced thread delivery should be gated on post-policy reply semantics or replaced by an explicit thread-routing fact that does not bypass `replyToMode`.
