## Summary

Keep open: the per-thread Google Chat session feature is useful and has real screenshot proof, but the current branch is not mergeable as-is because its `forcedThreadName` fallback can bypass existing `replyToMode` policy and it conflicts with current main plus overlapping Google Chat thread-delivery work.

## What This Changes

The PR adds an opt-in Google Chat `sessionThread` config, derives a thread-scoped session key from inbound Chat thread metadata, updates config/docs/tests, and forces outbound replies toward the inbound thread.

## Best Possible Solution

Preserve the opt-in per-thread session isolation idea, but rebase it onto current main and route thread placement through policy-aware Google Chat delivery plumbing before settling the public config name.

## Reproduction Assessment

## Solution Assessment

No, not as proposed. The per-thread session key is a maintainable direction, but outbound thread delivery must stay policy-aware and be reconciled with the active Google Chat thread-delivery PR before merge.
