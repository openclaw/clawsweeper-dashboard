## Summary

Keep this PR open: current main still lacks the per-thread Google Chat session option, but the branch has a blocking reply-policy regression where the new forced inbound-thread fallback can re-thread replies after shared `replyToMode` filtering intentionally cleared the thread target.

## What This Changes

The PR adds a Google Chat `sessionThread` option, per-thread session-key derivation, inbound-thread reply delivery, config/docs/generated metadata, changelog text, and session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that keeps route matching space-based, adds opt-in per-thread session keys, preserves explicit reply targets, and applies inbound-thread fallback only when reply policy allows it.

## Reproduction Assessment

Yes, at source level. PR head passes `forcedThreadName` after the shared reply filter can clear `payload.replyToId`, so `replyToMode: "off"` or later single-use-mode payloads can still be sent in-thread.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but the delivery fallback must respect reply policy and be reconciled with the active Google Chat thread-delivery work.
