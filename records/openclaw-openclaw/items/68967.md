## Summary

Keep this PR open because current main still lacks the per-thread Google Chat session option, but the branch needs changes before merge: its forced inbound-thread fallback can bypass `replyToMode: "off"`, and the thread-delivery portion overlaps active Google Chat reply-routing work.

## What This Changes

The PR adds a `channels.googlechat.sessionThread` boolean, derives Google Chat session keys from inbound thread names when enabled, forces Google Chat replies toward inbound threads, and updates config/docs/generated metadata plus tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session keys, preserves explicit reply targets, and applies inbound-thread fallback only when reply policy allows it.

## Reproduction Assessment

Yes, at source level. Current main lacks the `sessionThread` surface, and PR head restores `forcedThreadName` after shared reply threading can clear `payload.replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but the delivery fallback must respect reply policy and be reconciled with the related Google Chat thread-delivery work.
