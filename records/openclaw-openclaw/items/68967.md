## Summary

Keep open: the per-thread Google Chat session feature is not on current main and remains useful, but the PR still has a blocking correctness issue where forced inbound-thread delivery can override the existing flat reply mode contract.

## What This Changes

The PR adds a Google Chat `sessionThread` config option, derives per-thread session keys, sends replies back to inbound Google Chat threads, updates config/docs/generated metadata/changelog, and adds session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that adds opt-in per-thread sessions, keeps route matching space-based, honors explicit flat reply mode, and leaves broader streamed/follow-up routing to the existing Google Chat threading work.

## Reproduction Assessment

Yes at source level. Current main lacks `sessionThread`, and the PR head clearly shows the remaining blocker with `threadForReply = payload.replyToId ?? forcedThreadName` after shared reply filtering.

## Solution Assessment

No as submitted. The hashed session-key approach is maintainable, but forced inbound-thread fallback must not bypass an account configured for flat replies.
