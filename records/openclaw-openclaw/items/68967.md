## Summary

Keep this PR open: it is a focused and useful Google Chat threading change, but the latest head still bypasses the existing flat-reply policy when it falls back to the inbound thread.

## What This Changes

The PR adds an optional Google Chat `sessionThread` boolean, threads outbound replies from inbound thread metadata, updates config schema/docs/changelog/generated metadata, and adds session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that adds opt-in per-thread sessions, keeps route matching space-based, honors explicit flat reply mode, and leaves broader streamed/follow-up routing to https://github.com/openclaw/openclaw/issues/69422.

## Reproduction Assessment

Yes for source-level behavior. Current main lacks `sessionThread`, the PR body provides a concrete two-thread Google Chat scenario with screenshots, and the remaining `replyToMode: "off"` bug is reproducible with a focused delivery-helper test using `forcedThreadName`.

## Solution Assessment

No as submitted. The hashed dispatch-session-key approach is maintainable, but the forced inbound-thread fallback must skip implicit fallback when the account explicitly sets flat reply mode.
