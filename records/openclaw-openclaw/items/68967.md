## Summary

Keep open: current main still lacks the Google Chat per-thread session option, but this PR has a blocking reply-policy regression where the new forced thread fallback can re-thread replies after shared `replyToMode` filtering intentionally cleared them.

## What This Changes

The PR adds an optional Google Chat `sessionThread` config, derives thread-scoped session keys from inbound `thread.name`, forces outbound replies to the inbound thread, and updates config metadata, docs, changelog, and tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session storage, and either reuses https://github.com/openclaw/openclaw/pull/74235 for thread delivery or otherwise preserves `replyToMode` and explicit reply-target semantics.

## Reproduction Assessment

Yes, at source level for the review finding. The PR restores `forcedThreadName` when `payload.replyToId` is absent, while current main's reply-mode filter intentionally clears implicit reply targets for flat Google Chat accounts.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but outbound thread fallback must be gated by the same post-policy reply semantics or replaced by a thread-routing fact that the shared policy can clear.
