- **[P1] Move sessionThread to the active Google Chat schema:** `src/config/zod-schema.providers-core.ts:868`
  - body: This adds `sessionThread` to `zod-schema.providers-core.ts`, but current main exports Google Chat config from `zod-schema.providers-googlechat.ts` and the plugin consumes that active schema. After rebase, a documented `channels.googlechat.sessionThread` config would still be rejected unless the option moves to the active schema and metadata is regenerated from it.
  - confidence: 0.92

  - body: `threadForSend` is derived from `message.thread?.name` for every Google Chat message, and the forced thread is later used for delivery. Current main intentionally suppresses thread metadata for DMs because Google Chat rejects DM thread resource names, so this needs to stay group-only on rebase.
  - confidence: 0.9
- **[P1] Respect replyToMode before forcing inbound threads:** `extensions/googlechat/src/monitor-reply-delivery.ts:32`
  - body: Falling back to `forcedThreadName` whenever `payload.replyToId` is absent bypasses the dispatcher's replyToMode decision. When replyToMode has intentionally removed the reply target, this sends back into the inbound thread anyway and changes current message-routing semantics.
  - confidence: 0.9
- **[P1] Keep the session override out of the SDK contract:** `src/plugin-sdk/inbound-envelope.ts:42`
  - body: Adding `sessionKey?: string` to the plugin-facing inbound-envelope helper turns a Google Chat implementation detail into public SDK surface. Unless maintainers explicitly approve and document that API, keep the override local to the channel or route it through an existing internal session path.
  - confidence: 0.86
