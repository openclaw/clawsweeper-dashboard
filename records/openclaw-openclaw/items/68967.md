## Summary

Keep open: the PR addresses a real Google Chat session/threading gap, but the latest head still has a merge-blocking reply-threading regression because it reintroduces the inbound thread after the shared reply policy has removed it.

## What This Changes

The PR adds an optional `channels.googlechat.sessionThread` boolean with hashed thread-scoped session keys, propagates the inbound Google Chat thread into reply delivery, updates config/schema/docs/changelog, and adds session-key coverage.

## Best Possible Solution

A mergeable version should keep the hashed session dispatch key but make Google Chat thread fallback policy-aware so the shared `payload.replyToId` result stays authoritative.

## Reproduction Assessment

Yes, source-level. Current main has no `sessionThread` option and routes Google Chat sessions by space, while the PR finding follows directly from the diff plus existing reply-mode policy tests.

## Solution Assessment

No as submitted. The per-thread session approach is narrow, but forced reply threading must not override `replyToMode` suppression or single-use consumption.
