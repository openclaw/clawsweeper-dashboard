## Summary

Keep open. Current main still lacks the Google Chat per-thread session option, but this PR should not merge as submitted because the forced inbound-thread fallback bypasses existing `replyToMode` policy after the shared reply pipeline has stripped implicit thread targets.

## What This Changes

The PR adds an opt-in Google Chat `sessionThread` setting, derives thread-scoped session keys, forces outbound reply delivery into the inbound thread, updates config/docs/generated metadata, and adds Google Chat session-key tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session keys, and preserves shared reply policy plus explicit reply-target semantics while coordinating with https://github.com/openclaw/openclaw/pull/74235.

## Reproduction Assessment

Yes, at source level for the patch regression. The PR restores `forcedThreadName` after the shared reply pipeline has intentionally cleared `payload.replyToId` for Google Chat `replyToMode: "off"`; I did not live-run Google Chat in this read-only review.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but forced inbound-thread delivery needs to respect post-policy `replyToId` and coordinate with the existing Google Chat thread-delivery lane.
