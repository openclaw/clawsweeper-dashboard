## Summary

Keep open: the PR adds useful Google Chat per-thread session work with credible proof, but the current head still has a blocking reply routing bug that can bypass `replyToMode: "off"`.

## What This Changes

This PR adds an opt-in Google Chat `sessionThread` config, derives thread-hashed session keys, updates config/docs/generated metadata/changelog, and adds session-key coverage.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that adds opt-in per-thread sessions, keeps route matching space-based, preserves explicit reply targets, and only uses inbound-thread fallback when resolved reply mode allows implicit threading.

## Reproduction Assessment

Yes, at source level. Current main lacks `sessionThread`, and the PR head clearly shows `threadForReply = payload.replyToId ?? forcedThreadName` after shared reply filtering can clear `payload.replyToId`.

## Solution Assessment

No as submitted. The hashed session-key approach is maintainable, but the forced inbound-thread fallback must not bypass an account configured for flat replies.
