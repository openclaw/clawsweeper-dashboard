## Summary

Keep open: current main still lacks the requested Google Chat per-thread session option, and the PR has a source-level correctness bug that makes `replyToMode: "off"` ineffective for implicit inbound threads.

## What This Changes

The branch adds a `channels.googlechat.sessionThread` boolean, derives per-thread Google Chat session keys from inbound thread names, forces replies toward inbound threads, updates config/docs/generated metadata/changelog, and adds session-key tests.

## Best Possible Solution

Land a rebased Google Chat-owned implementation that keeps route matching space-based, adds opt-in per-thread session keys, preserves explicit reply targets, and applies inbound-thread fallback only when reply policy allows it.

## Reproduction Assessment

Yes, at source level. Current main lacks `sessionThread`, and PR head restores `forcedThreadName` after the shared reply filter can clear `payload.replyToId` for `replyToMode: "off"`.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but the delivery fallback must respect reply policy and stay aligned with the related thread-delivery work in https://github.com/openclaw/openclaw/pull/74235.
