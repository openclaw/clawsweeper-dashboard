## Summary

Keep open: current main still lacks the per-thread Google Chat session mode, and the branch has useful real Google Chat proof, but the patch reintroduces inbound threading after shared reply policy has filtered it away.

## What This Changes

The branch adds a `channels.googlechat.sessionThread` option, derives per-thread Google Chat session keys, threads Google Chat replies toward inbound threads, updates config/docs/generated metadata, and adds focused session-key tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session keys, preserves explicit reply targets, and applies inbound-thread fallback only when the resolved reply policy allows it.

## Reproduction Assessment

Yes, at source level. Current main lacks the per-thread Google Chat session surface, and PR head restores `forcedThreadName` after shared reply policy can clear `payload.replyToId`.

## Solution Assessment

No as submitted. The per-thread session-key direction is plausible, but delivery must respect the post-policy `payload.replyToId` contract and be reconciled with the active Google Chat thread-routing work.
