## Summary

Keep open: the per-thread Google Chat session feature is useful and the PR has real Google Chat proof, but the current diff can force replies back into the inbound thread after reply-policy filtering and it overlaps active thread-delivery work, so it is not merge-ready.

## What This Changes

The PR adds an optional Google Chat `sessionThread` config that derives per-thread session keys, threads outbound replies to the inbound Chat thread, and updates config metadata, docs, changelog, and tests.

## Best Possible Solution

Land a rebased implementation that keeps `sessionThread` as an opt-in session-storage partition, routes outbound Google Chat threads through a policy-aware delivery fact, and finalizes the config name with maintainer approval.

## Reproduction Assessment

Yes. Source inspection shows the PR restores `message.thread?.name` in delivery when `payload.replyToId` is absent, while current main's reply-mode filter can intentionally remove `replyToId` before Google Chat delivery.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but outbound thread delivery should flow through policy-aware Google Chat thread plumbing rather than a fallback that bypasses shared reply-mode semantics.
