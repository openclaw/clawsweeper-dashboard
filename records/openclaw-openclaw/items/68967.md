## Summary

Keep open: the per-thread Google Chat session mode is useful and the PR includes real Google Chat screenshot proof, but the patch is not merge-ready because it can bypass existing `replyToMode` behavior and it overlaps the open Google Chat thread-delivery work at https://github.com/openclaw/openclaw/pull/74235.

## What This Changes

The PR adds an optional Google Chat `sessionThread` config, derives thread-scoped session keys, updates config/docs/generated metadata/tests, and forces outbound replies toward the inbound Google Chat thread.

## Best Possible Solution

Preserve the opt-in per-thread session isolation idea, but rebase it onto current main, keep outbound thread placement inside policy-aware Google Chat delivery plumbing, and settle the public config shape before merge.

Yes, source-level: the PR restores an inbound Google Chat thread when `payload.replyToId` is absent, while current main's reply policy uses that absence to suppress implicit threading for `replyToMode` modes. I did not run a live Google Chat webhook in this read-only review.

## Solution Assessment

No, not as proposed. The per-thread session key direction is maintainable, but outbound thread delivery must remain policy-aware and be reconciled with active Google Chat thread-delivery work before merge.
