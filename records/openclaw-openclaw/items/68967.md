## Summary

Keep open: the per-thread Google Chat session feature is not on current main, but this PR still needs revision because its forced inbound-thread fallback bypasses existing flat reply-mode semantics and overlaps active thread-delivery work in https://github.com/openclaw/openclaw/pull/74235.

## What This Changes

The branch adds an optional Google Chat `sessionThread` config, derives thread-scoped session keys from inbound `thread.name`, forces outbound replies to the inbound thread, and updates config metadata, docs, changelog, and tests.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps routing space-based, adds opt-in per-thread session storage, and delegates thread delivery to the policy-preserving path from https://github.com/openclaw/openclaw/pull/74235 or an equivalent fix.

## Reproduction Assessment

Yes, at source level for the review finding: the PR restores an inbound thread when `payload.replyToId` is absent, while current main intentionally clears implicit reply targets for Google Chat accounts in flat reply mode.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but outbound thread fallback must respect the same post-policy reply semantics or be replaced by a thread-routing fact that the shared policy can clear.
