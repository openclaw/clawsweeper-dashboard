## Summary

Keep open. The PR addresses a real Google Chat per-thread session feature and includes sufficient real Google Chat proof, but the current diff still has a blocking reply-policy regression: it can force replies back into an inbound thread after shared reply filtering intentionally cleared that target.

## What This Changes

The PR adds a `channels.googlechat.sessionThread` option, thread-scoped Google Chat session keys, forced thread delivery, config/docs metadata, changelog text, and unit coverage.

## Best Possible Solution

Land a rebased Google Chat implementation that keeps route matching space-based, adds opt-in per-thread session storage, and preserves `replyToMode` plus explicit reply-target semantics while coordinating with the active thread-delivery PRs.

## Reproduction Assessment

Yes, at source level for the review finding. PR head restores `forcedThreadName` when `payload.replyToId` is absent, while current main's reply-mode filter intentionally clears implicit `replyToId` for flat reply mode.

## Solution Assessment

No as submitted. The per-thread session-key direction is reasonable, but the delivery fallback should be gated on the same post-policy reply semantics or replaced with an explicit thread-routing fact that does not bypass `replyToMode`.
