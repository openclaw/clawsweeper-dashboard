## Summary

Keep open. Current main still excludes the Responses-family transport IDs from malformed tool-call argument repair, and the open #75281 PR is already the active implementation candidate with closing syntax.

## What This Changes

The issue asks whether malformed tool-call argument repair should be enabled for Codex/Responses transport IDs such as `openai-codex-responses`, `openai-responses`, and `azure-openai-responses`.

## Best Possible Solution

Review and either land or reject #75281's narrow gate change: enable repair only for Responses transports whose active stream emits compatible `toolcall_delta` partial JSON, with focused gate tests and a changelog entry if behavior changes.

## Reproduction Assessment

Unclear. The current exclusion is reproducible from the gate and guard tests on current main, but the report explicitly says no malformed Codex arguments were observed, so there is not yet a high-confidence runtime failure repro.

## Solution Assessment

Unclear. A transport-specific expansion is likely the narrowest maintainable fix, but #75281 should settle the verified `openai-codex-responses` and `azure-openai-responses` policy while keeping WebSocket-capable generic `openai-responses` guarded unless proven safe.
