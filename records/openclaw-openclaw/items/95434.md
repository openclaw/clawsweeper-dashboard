## Summary

Keep open: this PR targets the right bug, but the patch puts model-override authorization in the `agent` request payload where the gateway neither declares nor trusts it, so the proposed fix can fail before the child run starts.

## What This Changes

The PR forwards `resolvedModel` to the subagent gateway `agent` call and tries to persist/authorize model override fields during subagent spawn planning.

## Best Possible Solution

Forward the resolved model through the child `agent` launch using the existing gateway client/internal or synthetic authorization seam, remove the unsupported request payload flag, and add focused coverage for the authorized launch path.

## Reproduction Assessment

Yes, at source level. Current main resolves and stores subagent model metadata, and the PR's proposed launch path can be checked against the strict `AgentParamsSchema` plus the gateway client authorization guard; no live QQ/Qwen run was executed in this read-only review.

## Solution Assessment

No. Forwarding the model is the right direction, but the authorization flag must not be a public `agent` request field, and the session-patch addition duplicates current main's derived override behavior.
