## Summary

No blocking findings. The patch addresses the linked Telegram message-loss regression by making the shared transcript final-text reader skip OpenClaw transcript-only assistant rows before Telegram can use them as final reply candidates.

## What This Changes

The PR updates the shared session transcript reader to ignore `provider=openclaw` `delivery-mirror` and `gateway-injected` assistant rows when selecting latest assistant text, with focused regression tests.

## Best Possible Solution

Land a narrow shared-reader fix that excludes transcript-only OpenClaw mirror/injected rows from final-text candidate selection while preserving mirror persistence and provider-boundary behavior.

## Reproduction Assessment

Yes at source level: current main reads the newest assistant row from the transcript and Telegram uses that reader for current-turn final text, so a later `delivery-mirror` row can become the selected candidate. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes. Filtering transcript-only OpenClaw rows at the shared transcript reader is the narrowest maintainable fix for the candidate-selection bug, and the provider-boundary test avoids suppressing external providers that happen to use similar model names.
