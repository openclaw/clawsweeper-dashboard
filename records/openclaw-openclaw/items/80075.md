## Summary

Keep this draft PR open for maintainer handling. It targets a valid payload fan-out bug and has useful proof, but the proposed canonicalization still misses media-only final replies because it requires normalized text before choosing the canonical final answer.

## What This Changes

The PR changes embedded runner payload selection to prefer a canonical final assistant reply over accumulated assistant text when pre-tool progress, status text, or raw-looking output is also present.

## Best Possible Solution

Revise the payload selection so canonical raw final answers with media/audio directives are used even when their cleaned text is empty, then land after targeted payload and incomplete-turn regression tests plus maintainer review.

## Reproduction Assessment

Yes for the source-level bug: current main selects every non-empty accumulated `assistantTexts` entry and final dispatch sends each payload item. I did not live-test Telegram, but the linked issue evidence and deterministic helper path are enough to reproduce the code path.

## Solution Assessment

No, not yet. The shared payload boundary is the right place to fix this, but the current PR condition excludes media-only canonical final replies and should prefer raw final media/audio directives even without text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

- **[P2] Handle media-only canonical final replies:** `src/agents/pi-embedded-runner/run/payloads.ts:372`
