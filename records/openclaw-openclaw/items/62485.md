## Summary

Keep this PR open for changes: the proposed lazy import is aimed at a real CLI help/banner path, but the current branch still has a cache regression that drops configured banner modes, and it lacks contributor real behavior proof before merge.

## What This Changes

The PR adds `src/cli/banner-config-lite.runtime.ts` and changes `readCliBannerTaglineMode` to dynamic-import the config reader with a process-level tagline-mode cache.

## Best Possible Solution

Keep a narrow CLI banner/help fix that avoids the heavy config path during help generation while preserving documented `cli.banner.taglineMode` behavior.

## Reproduction Assessment

Yes for the PR regression: the PR head source shows the fallback cache is set before the dynamic import callback can persist the resolved config value. The original CPU spin has a clear command and source path, but I did not live-run the potentially hanging command.

## Solution Assessment

No. The runtime-boundary direction is plausible, but this patch should not cache the cold fallback as completed state; it needs a resolved-result cache or another lightweight reader that preserves the existing config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
