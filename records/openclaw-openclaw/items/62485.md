## Summary

Keep open: this is a plausible fix for a real CLI help hang, but the PR branch is not merge-ready because the async cache can permanently ignore `cli.banner.taglineMode`, real terminal proof is missing, and current main still keeps the help banner module graph tied to the static config import.

## What This Changes

The PR adds a CLI banner runtime facade and changes `readCliBannerTaglineMode` to dynamically import config, cache the tagline mode, and fall back to the random tagline on cold help-path reads.

## Best Possible Solution

Land or replace this with a narrow CLI banner fix that removes the help-time heavy config import from the help module graph, preserves `cli.banner.taglineMode` for normal banners, and includes real terminal proof.

## Reproduction Assessment

Yes for the PR regression: PR head source plus a Node dynamic-import timing probe show the fallback cache path is taken before callbacks can persist the resolved config. The original hang remains source-reproducible from the help-to-banner static import path, but I did not execute the potentially hanging command in this read-only review.

## Solution Assessment

No. The runtime-boundary direction is plausible, but this implementation caches fallback before the promise result and current main's `mode: "default"` help hook is only a partial mitigation because the static config helper is still imported.
