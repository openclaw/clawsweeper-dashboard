## Summary

Keep open. This PR still targets a real CLI help-hang path, but PR head has a cache bug that drops configured banner tagline modes, and the external contributor has not provided real after-fix behavior proof; current main has a later configless help-hook change, but it does not remove the static banner config import enough to make this PR safely close as implemented.

## What This Changes

The PR adds a `banner-config-lite.runtime.ts` re-export and changes CLI banner config reading to use a dynamic import/cache instead of a static `createConfigIO` import.

## Best Possible Solution

Land or retain a narrow CLI help/banner fix that keeps `openclaw tui -h` off the heavy config-load path while preserving documented `cli.banner.taglineMode` behavior, then close the linked issue and this PR if current main already proves sufficient.

## Reproduction Assessment

Yes for the PR regression: source inspection of PR head plus the Node dynamic-import timing probe show the callback cannot populate local state before the fallback cache assignment. The original CPU spin has a clear source path through TUI help and banner rendering, but it was not live-run here.

## Solution Assessment

No. The runtime-boundary direction is plausible, but this patch should not cache the cold fallback as completed state; it needs a resolved-result cache or another lightweight reader that preserves the existing config contract.
