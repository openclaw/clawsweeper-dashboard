## Summary

Keep open. Current main still has the implicated static CLI banner config import, and this PR remains a plausible fix path, but the latest head still has a source-proven cache regression and lacks the required real behavior proof.

## What This Changes

The PR adds a `banner-config-lite.runtime.ts` boundary and changes `readCliBannerTaglineMode` to dynamically import config and cache a tagline mode for CLI banner rendering.

## Best Possible Solution

Land or replace with a corrected narrow CLI banner config fix that removes the help-time heavy config import/load path while preserving documented `cli.banner.taglineMode` behavior and proving the help command exits in a real terminal setup.

## Reproduction Assessment

Yes for the PR regression: PR head source plus a Node dynamic-import timing probe show the fallback cache path is taken before callbacks can persist the resolved config. The original hang has a clear source path, but I did not execute the potentially hanging command in this read-only review.

## Solution Assessment

No. The runtime-boundary direction is plausible, but relying on synchronous dynamic-import callbacks and caching `undefined` breaks the existing banner config contract; use a resolved-result cache or another lightweight reader that preserves config semantics.
