## Summary

Keep open. Current main still uses the synchronous heavy config path in CLI banner tagline resolution, so the reported help hang is not implemented on main; however, this PR head is not safe to merge because its dynamic-import cache path permanently returns the fallback value and drops configured banner tagline modes.

## What This Changes

The PR adds a `banner-config-lite.runtime.ts` runtime boundary and rewrites CLI banner tagline-mode loading to use a lazy dynamic import with a process-wide cache.

## Best Possible Solution

Land a corrected CLI banner-config fix that removes the heavy synchronous config import from Commander help while preserving `cli.banner.taglineMode` semantics and `env`-scoped reads, then let the linked bug close through the merge.

## Reproduction Assessment

Yes. The PR regression is reproducible from source inspection and a focused Promise timing check: the proposed cache is set to `undefined` before the import callback can run, so configured tagline modes are skipped after the first call. I did not run `openclaw tui -h` or Vitest because this sweep is read-only and must avoid artifact-producing commands.

## Solution Assessment

No. The proposed runtime boundary is directionally plausible, but relying on dynamic-import `.then()` to run synchronously is not a maintainable fix; a corrected implementation should use a real lightweight synchronous config selector or an async cache that does not regress first-render and env-specific behavior.
