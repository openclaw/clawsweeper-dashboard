## Summary

Keep open: current main still has the reported static banner config import, but this PR is not merge-ready because PR head caches the fallback before the dynamic import settles, can ignore existing banner config for the process, and lacks real terminal proof.

## What This Changes

The PR changes the CLI banner config helper to lazy import `createConfigIO` through a new runtime boundary and cache the banner tagline mode.

## Best Possible Solution

Land or replace this with a narrow CLI banner fix that removes the help-time heavy config import while preserving `cli.banner.taglineMode` and proving `openclaw tui -h` exits in a real terminal.

## Reproduction Assessment

Yes for the PR regression: PR head source plus a Node dynamic-import timing probe show the fallback cache path is taken before callbacks can persist the resolved config. The original hang has a clear source path, but I did not execute the potentially hanging command in this read-only review.

## Solution Assessment

No. The runtime-boundary direction is plausible, but this implementation caches the fallback before the promise result and needs a resolved-result cache or another lightweight reader that preserves config semantics.
