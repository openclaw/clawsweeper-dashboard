## Summary

Keep open. This PR still has a concrete cache regression in the banner config reader and lacks after-fix real behavior proof; current main also has not replaced the heavy static config import with the PR's intended runtime boundary.

## What This Changes

The PR changes the CLI banner config reader to dynamic-import the config module through a new runtime boundary so help rendering for `openclaw tui -h` avoids the synchronous heavy config path.

## Best Possible Solution

Land a corrected narrow CLI banner/help fix that keeps help rendering off the heavy config import path, preserves documented `cli.banner.taglineMode` behavior, and proves `openclaw tui -h` exits in a real setup.

## Reproduction Assessment

Yes for the PR regression: PR head source shows the fallback cache is set before promise callbacks can persist a resolved config value. The original hang has a clear command and source path, but I did not live-run the potentially hanging CLI.

## Solution Assessment

No. The runtime-boundary direction is plausible, but this patch should not cache the cold fallback as the completed result; it needs a resolved-result cache or another lightweight reader that preserves the existing config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
