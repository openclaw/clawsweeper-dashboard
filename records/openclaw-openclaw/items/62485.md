## Summary

Keep open. The PR is still the active implementation path for the linked CLI hang, but current main has not implemented the runtime-boundary change, the PR head still has a source-proven banner config cache regression, and the external contributor has not provided required real behavior proof.

## What This Changes

The PR replaces the CLI banner config helper’s static config import with a dynamic `*.runtime.ts` import and caches a tagline mode result for help/banner rendering.

## Best Possible Solution

Land or replace with a corrected narrow CLI banner config fix that removes the help-time heavy import/load path while preserving `cli.banner.taglineMode` and proving `openclaw tui -h` exits in a real setup.

## Reproduction Assessment

Yes for the PR regression: PR head source plus a Node dynamic-import timing probe show the fallback cache path is taken before callbacks can persist the resolved config. The original hang has a clear command and current source path, but I did not live-run a potentially hanging CLI in this read-only review.

## Solution Assessment

No. The runtime-boundary direction is plausible, but relying on synchronous dynamic-import callbacks and caching `undefined` breaks the existing banner config contract; use a resolved-result cache or another lightweight reader that preserves the config semantics.
