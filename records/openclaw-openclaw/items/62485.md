## Summary

Keep open. Current main still has the synchronous banner config path implicated by the linked issue, and this PR remains a plausible narrow fix direction, but the current PR head still has a cache bug that can ignore configured banner tagline mode values after the cold fallback.

## What This Changes

The PR moves CLI banner tagline config loading behind a dynamic runtime import to avoid a help-time hang in `openclaw tui -h`.

## Best Possible Solution

Land a corrected CLI banner-config fix that removes the help-time heavy config import while preserving configured `cli.banner.taglineMode` values and `env`-scoped reads, with focused regression coverage.

## Reproduction Assessment

Yes for the PR regression: source inspection plus the Node Promise timing probe show the dynamic import callback cannot populate the local result before the cold-path cache assignment. The original `openclaw tui -h` CPU spin has a clear source path but was not live-run here.

## Solution Assessment

No. The runtime-boundary direction is plausible, but this implementation should not cache the cold fallback as completed state; it needs a resolved-result cache or another lightweight reader that preserves the existing config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Populate the tagline cache after import resolution:** `src/cli/banner-config-lite.ts:66`
  - body: On the cold help path, `syncResolved` is still false when this assignment runs, so `cachedTaglineMode` becomes `undefined`. Because `readCliBannerTaglineMode()` treats any non-`null` value, including `undefined`, as completed cache state, later calls never observe the resolved config and configured `cli.banner.taglineMode` values are ignored for the rest of the process.
  - confidence: 0.93
