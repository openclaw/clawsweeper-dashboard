## Summary

Keep open: current main and the latest release still expose node gateway targeting as host/port/TLS only and build `ws(s)://host:port` with no path component. The open linked pull request https://github.com/openclaw/openclaw/pull/97679 is the right implementation candidate to review, not a reason to close this issue before merge.

## What This Changes

Request adding a `--context-path` option for `openclaw node run` and `openclaw node install`, persisting it in node host config and daemon arguments for reverse-proxied Gateway paths.

## Best Possible Solution

Review the linked implementation candidate, settle the canonical node endpoint API, then wire that one surface through node run, node install, `node.json`, daemon arguments, docs, and focused fresh-install plus persisted-config tests.

## Reproduction Assessment

Yes. Source inspection shows node run/install expose host and port only, the runner constructs `ws(s)://host:port`, and the daemon install args persist only that same host/port shape.

## Solution Assessment

Unclear. The capability belongs in core node CLI, but maintainers should decide whether the best permanent API is a path-only flag, a full endpoint URL, or reuse of an existing Gateway URL contract.
