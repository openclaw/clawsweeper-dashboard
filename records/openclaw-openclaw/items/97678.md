- `impact:other`: The issue affects remote node-to-gateway connectivity for deployments where the WebSocket endpoint is mounted below a path prefix.
- `maturity:stable`: The request maps to M4 Stable Gateway runtime and CLI maturity areas from the shortlist.

## Summary

Keep open: current `main` and the latest release still expose node host gateway targeting as host/port/TLS only, and the node host still builds `ws(s)://host:port` with no path component. A prior pull request attempted the same feature but closed unmerged, so it is useful context rather than a safe canonical fix.

## What This Changes

Add a node host gateway context-path option for `openclaw node run` and `openclaw node install`, persist it in node host config, and reuse it for installed services.

## Best Possible Solution

Define one canonical node gateway endpoint surface, wire it through foreground run, service install, `node.json`, daemon argument generation, docs, and focused tests for both fresh and previously persisted node config.

## Reproduction Assessment

Yes, from source inspection: node run/install expose host and port only, and the runner constructs `ws(s)://host:port` without a path. I did not run a live reverse-proxy repro because the missing representation is clear in current source and shipped release source.

## Solution Assessment

Unclear whether the exact `--context-path` flag is the best permanent API. The capability belongs in core node CLI, but maintainers should decide whether a path-only flag or a full endpoint URL is the cleaner long-term surface.
