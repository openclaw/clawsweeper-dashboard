## Summary

Keep open: the patch addresses real installer failure modes, but it only updates `scripts/install.sh` while the documented `install-cli.sh` git path still has the same tag-fetch and noninteractive pnpm behavior; the PR also still needs attached real behavior proof rather than a prose-only result summary.

## What This Changes

The PR changes `scripts/install.sh` so git installs of `main` fetch without tags and so the git dependency install defaults `CI=true` for non-TTY pnpm runs.

## Best Possible Solution

Mirror both installer fixes into `scripts/install-cli.sh`, add focused checks for both shell installers, and attach redacted terminal or log proof for the git-main fetch and non-TTY pnpm install paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `main` fetches all tags before handling `main`, and both public shell installers run the same git/pnpm install flow. I did not run a full installer repro because this was a read-only review.

## Solution Assessment

No, not yet. The direction is narrow and maintainable, but the duplicate public CLI installer needs the same treatment before the fix covers the documented surface.

## Review Findings

- **[P2] Mirror the main fetch fix into install-cli.sh:** `scripts/install.sh:1964`
  - body: This fixes the moved-tag failure only for `scripts/install.sh`. The documented `scripts/install-cli.sh` git path has the same `checkout_git_openclaw_ref` flow and still runs `git fetch --tags origin` before checking whether the requested ref is `main`, so CLI-only git installs can still fail before reaching the branch checkout. Please mirror this change in the CLI installer path.
  - confidence: 0.86
