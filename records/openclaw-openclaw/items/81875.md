## Summary

Keep this PR open: the `scripts/install.sh` change is pointed at a real installer failure, but the matching published `scripts/install-cli.sh` git path still has the same tag-fetch and non-TTY pnpm install behavior, and the PR still lacks inspectable real behavior proof.

## What This Changes

The PR changes `scripts/install.sh` so git installs of `main` fetch/pull without tags and run the git dependency install with `CI=true` by default.

## Best Possible Solution

Update both published Bash installers consistently, then attach artifact-backed proof for the moved-tag git path and non-TTY pnpm module purge behavior.

## Reproduction Assessment

Yes. Current main contains the failing tag-fetch and non-TTY pnpm paths, the prior ClawSweeper comment reports a moved-tag repro, and pnpm 11.1.0 source confirms why `CI=true` bypasses the purge prompt.

## Solution Assessment

No, not yet. The `scripts/install.sh` edit is the right shape for that script, but the best fix should mirror the same handling in `scripts/install-cli.sh` and include inspectable after-fix behavior proof.

## Security Review

Status: needs_attention

Summary: The installer diff is supply-chain sensitive because `--no-frozen-lockfile` broadens dependency resolution for all git installs, including immutable tag installs.

Concerns:

- **[medium] Unfrozen git installs broaden dependency resolution:** `scripts/install.sh:2372`
  - body: Changing the shared git install step to `pnpm install --no-frozen-lockfile` allows tag and semver git installs to resolve or rewrite dependencies instead of using the checked-in lockfile. That should be scoped to moving branch installs or explicitly justified before merge.
