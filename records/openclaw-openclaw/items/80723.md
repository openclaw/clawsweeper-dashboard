## Summary

Keep open: current main still uses `node:sqlite` for the default memory-core store and treats FTS5/sqlite-vec as runtime capabilities, so the requested portable bundled SQLite backend is not implemented and needs maintainer product/dependency review rather than cleanup closure.

## What This Changes

Request to bundle a WASM SQLite build with FTS5 and vector support for memory search instead of relying on `node:sqlite`.

## Best Possible Solution

Maintainers should decide whether default memory-core should gain a portable SQLite runtime, enforce/probe Node SQLite capabilities more strictly, or steer affected users to an alternate memory plugin while preserving the memory plugin contracts.

## Reproduction Assessment

No high-confidence runtime reproduction was established. Source inspection does confirm that current main still depends on `node:sqlite` and would mark FTS/vector unavailable if those capabilities fail to load.

## Solution Assessment

Unclear. Bundling WASM SQLite could address portability, but it is a broad runtime/dependency decision; a narrower maintainable path may be capability gating, clearer diagnostics, Node version policy, or a memory-plugin backend choice.
