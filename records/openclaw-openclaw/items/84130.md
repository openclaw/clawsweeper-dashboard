## What This Changes

The issue asks OpenClaw to suppress a repeated Discord split-message batch when the same long reply was already delivered via `message(action=send)` in the same turn.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80445 or an equivalent narrow fix that dedupes same-route logical explicit-send/final-delivery repeats, handles split batches idempotently, and preserves cross-target, media, and legitimate near-duplicate replies.

## Reproduction Assessment

Not fully; I did not run a live Discord reproduction, but the source path and reporter-provided Discord message IDs make the bug source-reproducible. Current main has generic message-tool suppression and Discord chunking but lacks the same-turn final-delivery guard targeted by the report.

## Solution Assessment

Yes directionally: the right end state is a narrow same-route logical-message dedupe plus split-batch idempotency. A broad final-output suppression would risk hiding legitimate follow-up or cross-target replies.
