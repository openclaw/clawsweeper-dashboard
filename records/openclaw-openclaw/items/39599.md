## Summary

Keep this open: current main and the latest release still accept the reported Feishu/Lark announce cron shape without an explicit target, and the linked closing PRs are still open/unmerged.

## What This Changes

The issue asks OpenClaw to reject Feishu/Lark announce cron jobs at creation time when no delivery target is available.

## Best Possible Solution

Review and land one focused fix that enforces Feishu/Lark announce target validation at the cron service/gateway boundary, keeps CLI feedback aligned, and adds regression coverage for missing and valid targets.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron add` and gateway/service validation can still accept `delivery: { mode: "announce", channel: "feishu" }` without `to`, while the outbound runtime can later fail with Feishu's missing-target error.

## Solution Assessment

Yes. Fail-fast create/update validation, with CLI feedback and focused tests, is the narrow maintainable fix; the issue should stay open until one linked fix is consolidated, reviewed, and merged.
