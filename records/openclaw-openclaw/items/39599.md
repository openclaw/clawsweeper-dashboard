## Label Justifications

- `P1`: The issue describes accepted scheduled jobs later failing user-visible Feishu delivery, which breaks an active channel workflow.
- `impact:message-loss`: The affected behavior is scheduled announce output not reaching the configured Feishu destination after an invalid no-target job is accepted.

## Summary

Keep open: current main and latest release still validate configured announce channels and blank fields, but they do not reject a Feishu/Lark announce cron job when `delivery.to` is omitted. The issue remains canonical because same-repo fix attempts are closed unmerged and the open closing candidates are cross-repository PRs that still need maintainer review.

## What This Changes

The issue requests create-time validation for Feishu/Lark announce cron jobs that lack `delivery.to` or another deterministic delivery target.

## Best Possible Solution

Land one focused validation fix at the gateway/service boundary for Feishu/Lark announce delivery, with regression coverage for omitted targets and explicit valid targets; keep audit/migration hints and clearer legacy diagnostics as follow-up UX unless maintainers deliberately include them.

## Reproduction Assessment

Yes. Source inspection on current main shows `delivery: { mode: "announce", channel: "feishu" }` can pass create-time validation while runtime target resolution can still produce the Feishu missing-target error.

## Solution Assessment

Yes. Gateway/service create and update validation is the narrowest maintainable fix because it prevents invalid stored jobs from every creation path; CLI/UI feedback can mirror that validation but should not be the only safety boundary.
