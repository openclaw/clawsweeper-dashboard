## Summary

Keep open: current main still has a source-reproducible create-time validation gap, and open closing PRs (#39602 and #39683) are already paired with the report, so the issue should wait for one reviewed fix to land rather than be closed early.

## What This Changes

The issue asks cron creation/update validation to reject Feishu/Lark announce jobs unless a delivery target is explicit or deterministically resolved.

## Best Possible Solution

Land one narrow fix that validates announce delivery against the plugin-owned target contract at the cron service boundary, avoids hard-coding bundled plugin IDs in core, and adds regression coverage for Feishu/Lark missing-target rejection and explicit-target acceptance.

## Reproduction Assessment

Yes, source-reproducible: the CLI can construct `delivery: { mode: "announce", channel: "feishu" }` with no `to`, and the service create/update validation does not reject it. The later outbound resolver has the Feishu target requirement that produces the runtime missing-target failure.

## Solution Assessment

Yes on the fail-fast direction, but the maintainable implementation should use generic/plugin-owned target validation instead of hard-coding Feishu/Lark policy into core. The existing open PRs should be reviewed with that boundary in mind.
