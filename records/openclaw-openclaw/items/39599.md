## Summary

Keep open: current main and latest release still allow a Feishu/Lark announce cron delivery with no `delivery.to`, while runtime outbound delivery can later fail with the reported missing-target error. Two open closing-syntax fork PRs still target this bug but are conflicting and unmerged, so they are implementation candidates rather than close evidence.

## What This Changes

Require cron creation/update to reject Feishu/Lark announce delivery without an explicit or deterministically resolved target instead of persisting jobs that fail at runtime.

## Best Possible Solution

Land a narrow gateway/service validation fix that rejects Feishu/Lark announce jobs without an explicit or proven resolved target, with regression coverage for create/update and a separate follow-up for auditing legacy jobs.

## Reproduction Assessment

Yes, at source level. The CLI/RPC can create announce delivery with `channel: "feishu"` and no `to`, current validation permits it, and runtime target resolution later fails when no session/default target exists.

## Solution Assessment

Yes for the central direction: fail fast before invalid Feishu/Lark announce jobs persist. The best fix should share gateway/service validation and preserve any truly deterministic context-derived target behavior already documented for active-chat reminders.
