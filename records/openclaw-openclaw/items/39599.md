## Label Justifications

- `P1`: Accepted scheduled Feishu/Lark reminders can run without delivering the expected message, breaking a real channel workflow.
- `impact:message-loss`: The issue is about scheduled outbound messages being accepted and later failing delivery, which users experience as missed reminders.

## Summary

Current main and the latest stable release still allow an announce cron job to be created with a Feishu/Lark channel but no resolvable target, while runtime delivery can later fail with the missing-target path. The issue should stay open as the canonical tracker because two linked fork PRs still reference it with closing syntax but are open and conflicting, so no merged implementation owns the fix yet.

## What This Changes

The issue requests create-time validation for Feishu/Lark announce cron jobs so jobs without an explicit or deterministically resolved delivery target are rejected before scheduling.

## Best Possible Solution

Land one current-main fix that validates create and update through the canonical cron delivery target resolver, adds CLI feedback and regression coverage, and tracks stored-job audit guidance separately.

## Reproduction Assessment

Yes, at source level. The CLI can send announce delivery with `channel` and no `to`, the gateway/service validate channel shape but not target resolvability, and runtime delivery later has an unresolved-target error path; I did not run a live Feishu job.

## Solution Assessment

Yes in direction, but the best fix is not a Feishu-only CLI guard. The maintainable solution is a shared create/update validation path that rejects only when no explicit or deterministic runtime target can be resolved.
