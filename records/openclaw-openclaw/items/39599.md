## Label Justifications

- `P1`: Accepted Feishu/Lark cron jobs can later miss scheduled announce delivery at runtime, which is an active broken channel workflow.
- `impact:message-loss`: The issue is specifically about scheduled Feishu announce output not reaching the intended chat because the job was accepted without a delivery target.

## Summary

Keep open: current main still accepts an announce cron delivery with `channel: "feishu"` and omitted `delivery.to`, while the same-repo fix PRs are closed unmerged and the two open fork PRs that close this issue are conflicting candidate work rather than a landed fix.

## What This Changes

The issue requests create-time validation so Feishu/Lark announce cron jobs without an explicit or deterministically resolved `delivery.to` are rejected instead of failing at runtime.

## Best Possible Solution

Land a focused gateway/service create and update validation that rejects Feishu/Lark announce delivery without an explicit or deterministically resolved target, with regression coverage and a separate audit/diagnostic follow-up for already-persisted jobs if maintainers want it.

## Reproduction Assessment

Yes. Source inspection shows `cron.add` can still pass `delivery: {mode: "announce", channel: "feishu"}` without `to`, and current gateway/service validation does not reject that shape before persistence.

## Solution Assessment

Yes for the central request. The maintainable fix belongs at the gateway/service validation boundary, with CLI/UI mirroring only for better feedback rather than as the sole safety check.
