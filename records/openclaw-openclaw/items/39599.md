## Summary

Keep open. Current main still accepts Feishu/Lark announce cron delivery without `delivery.to` through the protocol, gateway validation, cron service, and CLI creation path, and open PRs already explicitly target this issue, so it should wait for one canonical fix to merge.

## What This Changes

The issue requests create-time validation so Feishu/Lark announce cron jobs without an explicit or deterministically resolved delivery target are rejected instead of failing at runtime.

## Best Possible Solution

Land one focused fix that validates Feishu/Lark announce cron delivery on create and update when neither explicit `delivery.to` nor a deterministic resolved target exists, with regression coverage at the service/gateway boundary and CLI/UI feedback kept as a user-experience layer.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `{ mode: "announce", channel: "feishu" }` remains schema-valid and passes the inspected gateway, service, and CLI creation paths; runtime resolution can then fail when no target or default target exists.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: enforce the safety boundary server-side for create/update first, then mirror the validation in CLI/UI without changing valid implicit-target flows for other channels.
