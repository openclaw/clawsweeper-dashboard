## What This Changes

Require create-time rejection of Feishu/Lark announce cron jobs that lack a resolvable delivery target.

Land one current-main fix that validates create and update through the canonical channel target-resolution contract, adds aligned CLI feedback and regression coverage, and tracks audit guidance for already-stored invalid jobs separately.

## Solution Assessment

Yes in direction, but the best fix is a shared service-boundary resolvability check covering create and update, not duplicated Feishu-only CLI rules or an unconditional explicit-field requirement.
