## Summary

Keep this open: current main and the latest release still accept Feishu/Lark announce cron delivery with no target, and outbound resolution can still fail later with the reported missing-target error. The issue also has an open closing PR, so it should remain the canonical tracker until that fix is landed, replaced, or explicitly superseded.

## What This Changes

The issue asks for cron creation/update validation to reject Feishu/Lark announce delivery when no explicit or deterministically resolved `delivery.to` target exists.

## Best Possible Solution

Land one consolidated current-main fix that rejects unresolved Feishu/Lark announce cron delivery at the cron service/gateway boundary, keeps CLI feedback aligned, preserves legacy cleanup edits, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main can create announce delivery with `channel: "feishu"` and no `to`, and outbound resolution can later return the Feishu missing-target error when no plugin default target exists.

## Solution Assessment

Yes, the requested fail-fast validation is the right direction. The maintainable version belongs at the cron service/gateway boundary with CLI feedback and compatibility-aware handling for existing invalid jobs.
