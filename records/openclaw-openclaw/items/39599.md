## Summary

Keep open: current main and the latest release still allow an announce cron delivery with `channel: "feishu"` and no `delivery.to`, while runtime Feishu delivery later needs a concrete target. The open linked fork PRs are not merged and are currently conflicting, so they are implementation candidates, not grounds to close this issue.

## What This Changes

The issue asks cron creation to reject Feishu/Lark announce jobs without an explicit or deterministically resolved delivery target before they are stored and scheduled.

## Best Possible Solution

Land a narrow gateway/service validation change that rejects Feishu/Lark announce delivery without `delivery.to` unless the same creation context can prove a valid target, with focused regression tests and separate follow-up for legacy job audit UX.

## Reproduction Assessment

Yes, at source level. Current CLI/RPC can construct announce delivery with `channel: "feishu"` and no `to`, current create-time validation permits it, and runtime target resolution later fails when no session/default target is available.

## Solution Assessment

Yes for the central direction: fail fast before invalid Feishu/Lark announce jobs persist, using gateway/service validation plus tests. The linked branches still need rebase/proof, and the audit/error-message ideas are follow-up UX rather than the minimal fix.
