## Summary

Keep this open: current main and the latest tagged release still accept Feishu/Lark announce cron delivery without an effective target, and runtime outbound resolution can fail later. The open fixing PR that closes this issue is still conflicted and not merged, so the issue should remain the canonical tracker.

## What This Changes

The issue requests create-time validation that rejects Feishu/Lark announce cron jobs when no explicit or resolver-backed delivery target exists.

## Best Possible Solution

Land one consolidated current-main fix that rejects unresolved Feishu/Lark announce cron delivery at the cron service/gateway boundary, keeps CLI feedback aligned, preserves legacy cleanup edits, and includes focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main can accept `delivery: { mode: "announce", channel: "feishu" }` without `to`, and outbound resolution can later return the Feishu missing-target error when no default target exists.

## Solution Assessment

Yes. Fail-fast validation at the cron service/gateway boundary, with CLI feedback and regression tests, is the narrow maintainable fix; this issue should stay open until a linked fix lands or is superseded.
