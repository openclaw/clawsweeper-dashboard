## Summary

Keep open: current main and latest release still validate blank cron delivery fields and configured announce channels, but they do not reject a Feishu/Lark announce cron job when `delivery.to` is omitted. The issue has source-level reproduction plus later operator evidence, while the linked fix attempts are unmerged or open conflicting fork PRs rather than implemented mainline work.

## What This Changes

Reject Feishu/Lark announce cron jobs at creation or update time when no explicit or deterministically resolved delivery target exists.

## Best Possible Solution

Land one focused current-main fix at the gateway/service validation boundary for Feishu/Lark announce delivery, with regression tests for omitted targets and explicit valid targets; treat cron audit and clearer legacy diagnostics as follow-up UX unless included deliberately.

## Reproduction Assessment

Yes. Source inspection shows current main rejects blank `delivery.to` but still permits omitted `delivery.to` for configured Feishu announce delivery, which reaches the runtime missing-target error path.

## Solution Assessment

Yes for the central bug: create/update validation at the gateway/service boundary is the narrowest maintainable fix because it prevents invalid stored jobs from every creation path. CLI/UI feedback can mirror that validation, while audit/migration UX is a separate operator follow-up.
