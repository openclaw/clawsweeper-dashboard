## Summary

Keep this open: current main still accepts Feishu/Lark announce cron delivery without an effective target at create/update validation time, while runtime outbound resolution can still fail later. Open fix PRs already reference this issue with closing syntax, so the issue should remain the canonical tracker until one of those fixes lands or is explicitly superseded.

## What This Changes

The issue requests create-time validation that rejects Feishu/Lark announce cron jobs unless `delivery.to` is explicit or a deterministic target can be resolved.

## Best Possible Solution

Review and land one focused fix that rejects Feishu/Lark announce delivery without an explicit or resolver-backed target at the cron service/gateway boundary, with aligned CLI feedback and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main can accept `delivery: { mode: "announce", channel: "feishu" }` without `to`, and outbound resolution can later return the Feishu missing-target error when no plugin default target exists.

## Solution Assessment

Yes. Fail-fast validation at the cron service/gateway boundary, with CLI feedback and regression tests, is the narrow maintainable fix; this issue should stay open until a linked fix lands or is superseded.
