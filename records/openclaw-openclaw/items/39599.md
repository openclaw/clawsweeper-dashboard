## Summary

Keep open: current main and v2026.6.10 still allow a Feishu/Lark announce cron delivery object with `channel` but no `to`, while the runtime target resolver still fails that shape at delivery time. The issue is already paired with open fork PR candidates, so no new repair lane should be queued until maintainers adopt, refresh, or close those candidates.

## What This Changes

The issue asks cron creation/update validation to reject Feishu/Lark announce jobs when `delivery.to` is missing and no deterministic target can be resolved.

## Best Possible Solution

Land one focused mainline fix at the gateway/service validation boundary for Feishu/Lark announce delivery, with regression tests for omitted targets and explicit valid targets; keep audit and diagnostic wording as follow-up UX unless maintainers intentionally broaden scope.

## Reproduction Assessment

Yes, source-level. The CLI can submit `{ mode: "announce", channel: "feishu" }` with no `to`, the protocol and gateway validation allow it, and the runtime resolver still produces the Feishu missing-target failure.

## Solution Assessment

Yes for the central bug. Gateway/service validation is the narrowest maintainable fix because it blocks invalid stored jobs from every creation path; CLI/UI checks can mirror it for better feedback.
