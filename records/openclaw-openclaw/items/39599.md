- `impact:message-loss`: The reported failure mode stores a scheduled message job that reaches runtime without a deliverable Feishu target.

## Summary

Keep open: current main and latest v2026.6.10 still validate webhooks and isolated session support, but do not reject Feishu/Lark announce cron delivery when `delivery.to` is missing. Two open fork PRs still claim to close this issue, so this should remain the canonical issue while maintainers adopt, import, or reject those candidates.

## What This Changes

The issue asks for create-time rejection of Feishu announce cron jobs that omit `delivery.to`, because accepted jobs later fail delivery at runtime.

## Best Possible Solution

Review and land or import a narrow gateway/service validation fix for Feishu/Lark announce delivery without an explicit or deterministic target, while preserving targetless channels such as Telegram that can resolve context differently.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `cron add --announce --channel feishu` can construct delivery with no `to`, and `createJob` validates only webhook URLs and session-target support before accepting the job. I did not run the CLI or a live Feishu delivery path in this read-only review.

## Solution Assessment

Yes, a gateway/service validation rule is the best safety boundary, with CLI validation as an earlier user-facing layer. Existing stored bad jobs should be handled by a separate audit or doctor-style follow-up rather than making runtime delivery guess a target.
