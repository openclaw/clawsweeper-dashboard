## Label Justifications

- `P1`: The issue can cause scheduled Feishu reminders or reports to be accepted but fail at delivery time, producing missed user-visible messages.
- `impact:message-loss`: The reported failure mode is a cron announce message that is scheduled and run but not delivered to the Feishu destination.

## Summary

Keep open: current main and latest release still accept an isolated Feishu/Lark announce cron delivery with a configured channel but no `delivery.to`, while runtime target resolution later fails with the Feishu missing-target error. The report is still necessary, and the existing linked fix attempts are closed or external candidates rather than merged mainline behavior.

## What This Changes

Reject Feishu/Lark announce cron jobs without an explicit or deterministically resolved delivery target at creation/update time.

## Best Possible Solution

Add gateway/service create and update validation that rejects Feishu/Lark announce delivery without explicit or deterministically resolved `delivery.to`, with focused CLI/gateway regression tests and a separate audit/diagnostic follow-up for already-stored jobs.

## Reproduction Assessment

Yes, source-level: the current `cron.add` path can accept announce delivery with channel `feishu` and no `delivery.to`, while runtime target resolution later requires a Feishu target. I did not run a live Feishu delivery repro in this read-only review.

## Solution Assessment

Yes. Fail fast at the gateway/service creation and update boundary is the narrowest maintainable fix; CLI/UI feedback should mirror that validation, while auditing old jobs can be a follow-up.
