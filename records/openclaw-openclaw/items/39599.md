## Summary

Keep this open: current main still accepts targetless Feishu/Lark announce cron delivery, and runtime target resolution can still fail later with the reported missing-target error. There are open linked fix PRs, so the issue should close only after one consolidated fix lands or is explicitly superseded.

## What This Changes

The issue asks for create-time validation so Feishu/Lark announce cron jobs without an explicit or deterministically resolvable delivery target are rejected before scheduling.

## Best Possible Solution

Land one focused current-main fix that validates Feishu/Lark announce delivery against an explicit or runtime-resolvable target, preserves legacy cleanup edits, and adds regression coverage for missing, valid, and reactivation paths.

## Reproduction Assessment

Yes, at source level. Current main can build and validate `delivery: { mode: "announce", channel: "feishu" }` without `to`, while outbound resolution can later return the Feishu missing-target error.

## Solution Assessment

Yes. Fail-fast validation at the cron service/gateway boundary, with aligned CLI feedback and regression tests, is the narrow maintainable fix; the issue should stay open until a linked fix PR lands or is superseded.
