## Label Justifications

- `P1`: Cron can accept a scheduled Feishu announce job that later fails to deliver, breaking an active channel workflow for users.
- `impact:message-loss`: The reported behavior is scheduled announce output not reaching the Feishu destination after an invalid no-target job is accepted.

## Summary

Keep open: current main and the latest release still allow a Feishu/Lark announce cron job with an explicit channel but omitted `delivery.to` to reach creation validation, while runtime delivery can still fail with the reported missing-target error. The issue remains canonical, and the open cross-repository closing PRs should be reviewed or retired before starting a replacement fix lane.

## What This Changes

Reject Feishu/Lark announce cron job creation when no explicit or deterministically resolved delivery target exists.

## Best Possible Solution

Land one focused gateway/service validation fix for Feishu/Lark announce delivery without a target, mirror it in CLI/UI feedback where useful, and track legacy audit/diagnostic wording as follow-up UX if maintainers want it.

## Reproduction Assessment

Yes. Current source shows `delivery: { mode: "announce", channel: "feishu" }` can pass create-time validation while runtime target resolution still has the Feishu missing-target error path.

## Solution Assessment

Yes. Gateway/service create and update validation is the narrowest maintainable fix because it prevents invalid stored jobs from every creation path; CLI/UI validation should mirror that boundary rather than replace it.
