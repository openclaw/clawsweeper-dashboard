## Summary

Keep this open: current main still accepts targetless Feishu/Lark announce cron delivery, and runtime target resolution can later fail; the linked open PRs should be consolidated rather than closing the issue before merge.

## What This Changes

This issue asks for Feishu/Lark announce cron jobs to be rejected at create/update time when no explicit or deterministically resolvable delivery target exists.

## Best Possible Solution

Land one focused current-main fix that validates Feishu/Lark announce delivery against a real explicit or runtime-resolvable target, preserves safe cleanup of legacy invalid jobs, and adds regression coverage for missing, valid, and reactivation paths.

## Reproduction Assessment

Yes, at source level. Current main can build and validate `delivery: { mode: "announce", channel: "feishu" }` without `to`, while outbound resolution can later return the Feishu missing-target error.

## Solution Assessment

Yes. Fail-fast validation at the cron service/gateway boundary, with aligned CLI feedback and regression tests, is the narrow maintainable fix; the issue should stay open until a linked fix PR lands or is superseded.
