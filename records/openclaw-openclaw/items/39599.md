## Label Justifications

- `P1`: Accepted Feishu announce cron jobs can later miss scheduled delivery at runtime, breaking an active channel workflow for real users.
- `impact:message-loss`: The issue is specifically about scheduled Feishu announce messages not being delivered after an invalid no-target job is accepted.

## Summary

Keep open: current main and the latest release still reject blank cron delivery fields and ambiguous channels, but they do not reject a configured Feishu/Lark announce cron job when `delivery.to` is omitted; later runtime delivery can still fail with the Feishu missing-target error, and operator evidence confirms this happens in real jobs.

## What This Changes

Reject Feishu/Lark announce cron jobs that have no explicit or deterministically resolved delivery target before they are stored or scheduled.

## Best Possible Solution

A focused current-main fix should prevent unresolved Feishu/Lark announce jobs from being stored while still accepting explicit or truly resolved targets; legacy audit and clearer diagnostics can remain a separate follow-up UX path.

## Reproduction Assessment

Yes, at source level: current main lets the CLI construct `{ mode: "announce", channel: "feishu" }`, gateway validation only checks configured channel/blank fields, and runtime target resolution can later produce the Feishu missing-target error. I did not run a live Feishu job in this read-only review.

## Solution Assessment

Yes for the central bug: create/update validation at the gateway/service boundary is the narrowest maintainable fix because it blocks invalid stored jobs from every creation path. CLI/UI feedback can mirror that validation, while audit and diagnostic wording are separate operator follow-ups.
