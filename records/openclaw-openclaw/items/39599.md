## Summary

Keep this open: current main still accepts targetless Feishu/Lark announce cron delivery while runtime target resolution can fail later, and open closing PRs already cover the fix path.

## What This Changes

Request fail-fast validation for Feishu/Lark announce cron jobs that omit an explicit or deterministically resolvable delivery target.

## Best Possible Solution

Land one focused current-main fix that validates Feishu/Lark announce delivery against an explicit or runtime-resolvable target, preserves legacy cleanup edits, and adds regression coverage for missing, valid, and reactivation paths.

## Reproduction Assessment

Yes, at source level. Current main can construct and validate `delivery: { mode: "announce", channel: "feishu" }` without `to`, while outbound resolution later returns a missing-target error when no default target exists.

## Solution Assessment

Yes. Fail-fast validation at the cron service/gateway boundary, with aligned CLI feedback and regression tests, is the narrow maintainable direction; the issue should stay open until a linked fix PR lands or is explicitly superseded.
