## Summary

Keep open. The prior ClawSweeper comment remains accurate on current main: WhatsApp still accepts debounce only at channel/account scope, the group config/schema path does not include `debounceMs`, and the live runtime still passes one scalar debounce window into the listener while excluding media messages from the debounce path.

## What This Changes

Request to allow per-group `debounceMs` overrides for WhatsApp groups, with possible later parity for other group-capable chat channels.

## Best Possible Solution

Implement WhatsApp first: accept `groups.<jid>.debounceMs` at root and account scopes, resolve exact group before wildcard before account/channel/global fallback, use the existing per-item debounce resolver, and update batching so earlier media in the same group/sender burst is preserved when the later text triggers the turn. Treat other channels as follow-up unless maintainers want a shared cross-plugin config contract.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a WhatsApp config containing `groups.<jid>.debounceMs` is outside the strict group schema, and even if injected manually the runtime only receives one scalar debounce value and skips media messages from debounce.

## Solution Assessment

Yes for a scoped WhatsApp fix. The narrow maintainable path is to wire the existing debouncer hook through WhatsApp group config and tests, while leaving broad cross-channel parity to later plugin-specific work or an explicit shared-contract design.
