## Summary

Keep open: current main and v2026.5.22 still do not support `debounceMs` inside WhatsApp group config, and the adjacent timestamp, media, composing, and adaptive debounce threads do not supersede this narrower per-group override request.

## What This Changes

The issue requests `debounceMs` inside per-group WhatsApp config, with eventual parity for similar group-capable channels, so one group can wait longer without changing the whole account or channel.

## Best Possible Solution

Define a WhatsApp-owned per-group debounce contract with exact-group-before-wildcard fallback, schema/types/docs/tests, and an explicit media scope; keep cross-channel parity as a separate product/API decision.

## Reproduction Assessment

Yes, source-level. Current main's strict WhatsApp group schema omits `debounceMs`, runtime resolves only one account/channel debounce scalar before listener creation, and I did not run a live WhatsApp group session.

## Solution Assessment

Unclear for the full request. A scoped WhatsApp per-group override is plausible, but maintainers still need to choose fallback semantics, media behavior, and whether cross-channel parity belongs in this change.
