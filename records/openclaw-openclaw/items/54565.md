## Best Possible Solution

Approve a narrow WhatsApp-owned group-level debounce config only if maintainers want the new config surface; then wire it through the existing per-item debouncer with explicit immediate-flush rules and focused docs/tests.

## Reproduction Assessment

Yes at source level: current main has strict WhatsApp group schema entries without debounceMs, while debounceMs is supported only at channel/account or global inbound scope. I did not run a live WhatsApp repro because the config gap is directly visible in source.

## Solution Assessment

Unclear until maintainers decide the config contract. The narrowest maintainable implementation would keep this WhatsApp-owned, reuse the existing per-item debouncer, and avoid turning it into a cross-channel config expansion in the same change.
