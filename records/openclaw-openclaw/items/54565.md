## Summary

Keep open. Current main still lacks group-level WhatsApp debounce schema/runtime wiring, and the prior ClawSweeper review remains accurate; this is a narrow WhatsApp config/runtime feature gap, while cross-channel parity should be separate follow-up.

## What This Changes

Request to allow `debounceMs` in per-group WhatsApp config, with possible parity for other group-capable channels.

## Best Possible Solution

Implement WhatsApp first with `groups.<jid>.debounceMs` at root and account scope, exact-before-wildcard fallback, per-message debounce resolution, and batching that preserves earlier media in the same group/sender burst; leave broad cross-channel parity to explicit follow-up.

## Reproduction Assessment

Yes. Source inspection is enough: `groups.<jid>.debounceMs` is rejected by the strict WhatsApp group schema, the runtime passes only one scalar debounce value into the listener, and media messages bypass the current debounce predicate.

## Solution Assessment

Yes for a scoped WhatsApp repair. The maintainable path is to extend the bundled WhatsApp config/runtime and tests using the existing debouncer hook, not to redesign every channel in the same PR.
