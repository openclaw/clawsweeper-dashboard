## Label Justifications

- `P1`: Gateway can report ready while a configured channel never initializes, breaking real channel send/receive workflows after plugin load or compatibility failures.
- `impact:message-loss`: A configured channel missing from the loaded registry can make inbound or outbound messages unavailable while the Gateway process continues running.

## Summary

Keep open: current main and v2026.6.8 still start and readiness-check only channel plugins that successfully enter the loaded registry. Main has partial improvements for configured startup-plugin planning, plugin compatibility diagnostics, status/read-only resolution, and doctor repair guidance, but the central configured-versus-loaded Gateway startup audit requested here is still missing.

## What This Changes

The issue requests clearer Gateway startup diagnostics when configured channels are skipped after plugin/runtime mismatch, missing generated modules, Feishu setup failure, API timeout, or heavy startup/plugin scans.

## Best Possible Solution

Add a focused Gateway startup/readiness audit that compares configured channel intent against the loaded channel registry and surfaces skipped or failed channel owners with actionable diagnostics; track hot reload, fail-fast policy, Feishu API warning severity, and broad startup performance as separate follow-ups.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection gives a credible path: configured channel plugins are planned for startup, but startup/readiness visibility still depends on successfully loaded channel registry entries.

## Solution Assessment

Partially yes: the narrow configured-versus-loaded audit is the maintainable first fix for the silent skip symptom. Fail-fast compatibility behavior, hot reload, Feishu external API health severity, and startup performance need separate maintainer decisions or narrower issues.
