## Label Justifications

- `P1`: Gateway can report ready while a configured channel never initializes, breaking real channel send/receive workflows after plugin load or compatibility failures.
- `impact:message-loss`: When a configured channel never enters the loaded registry, messages for that channel can be unavailable while the Gateway process keeps running.

## Summary

Keep open: current main and the latest release still start/readiness-check only loaded channel plugins, while this issue is about configured channels disappearing before that loaded registry exists. Main has useful partial diagnostics for plugin compatibility, status, and missing external plugins, but not the requested Gateway startup configured-versus-loaded channel audit.

## What This Changes

The issue requests clearer Gateway diagnostics and recovery when configured channel plugins fail to load or initialize, including compatibility checks, channel init audit logs, reload/rescan behavior, missing-module guidance, startup performance improvements, and Feishu timeout health warnings.

## Best Possible Solution

Add a focused Gateway startup/readiness audit that compares configured channel/plugin ownership with the active loaded registry and surfaces skipped or failed channels with actionable loader diagnostics, while tracking hot reload, broad performance, and Feishu SDK timeout work separately.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection gives a credible reproduction path: configure a channel whose plugin fails before registration, and Gateway startup/readiness only iterate loaded channel plugins, so the configured channel can be absent from both the startup attempts and readiness failures.

## Solution Assessment

Partially yes: the narrow startup/readiness audit is the maintainable first fix. Fail-fast compatibility policy, hot plugin reload, startup metadata caching, and Feishu external API health behavior need separate maintainer decisions or linked fixes.
