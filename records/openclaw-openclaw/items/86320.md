## Label Justifications

- `P1`: Gateway can report ready while a configured channel never initializes, breaking real channel send/receive workflows after plugin load or compatibility failures.
- `impact:message-loss`: When a configured channel never enters the loaded registry, messages for that channel can become unavailable while the Gateway process keeps running.

## Summary

Keep open: current main has useful status/doctor diagnostics for configured channel load failures, but Gateway startup and readiness still operate on the loaded channel registry only, so a configured channel that never registers can remain absent from the startup/readiness audit.

## What This Changes

The issue asks Gateway startup/readiness to clearly surface configured channels that fail to load or initialize, including plugin/runtime mismatch, missing generated modules, external API timeouts, hot reload, and startup performance concerns.

## Best Possible Solution

Add a focused Gateway startup/readiness audit that compares configured channel intent with the loaded channel registry and reports skipped or failed channels with actionable loader diagnostics; split hot reload, Feishu timeout policy, and startup performance into separate owner-scoped work.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection gives a credible path: configured channel intent and status diagnostics exist, but Gateway startup/readiness still iterate only loaded channel plugins, so a plugin that fails before registration can be absent from that surface.

## Solution Assessment

Partially yes: the existing status/doctor diagnostics are useful, but they are not the complete fix for the reported Gateway startup/readiness silence. The maintainable fix is a narrow configured-versus-loaded startup/readiness audit, with fail-fast compatibility, hot reload, Feishu timeout handling, and performance tracked separately.
