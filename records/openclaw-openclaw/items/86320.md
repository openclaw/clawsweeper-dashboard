## Label Justifications

- `P1`: Gateway can report ready while a configured channel never initializes, breaking real channel send/receive workflows after plugin load or compatibility failures.
- `impact:message-loss`: When a configured channel never enters the loaded registry, messages for that channel can become unavailable while the Gateway process keeps running.

## Summary

Keep open: current main and the latest release still start and readiness-check only the loaded channel registry, so a configured channel that fails before registration can still disappear from the startup/readiness surface. Related issues cover adjacent generated-module warnings or transport-specific symptoms, but not this configured-versus-loaded Gateway audit.

## What This Changes

The issue requests clearer Gateway startup diagnostics and safer behavior when configured channel plugins are incompatible, missing generated modules, or fail before channel registration.

## Best Possible Solution

Add a focused Gateway startup/readiness audit that compares configured channel intent with the loaded channel registry and reports skipped or failed channels with actionable loader diagnostics; split hot reload, startup performance, and Feishu API timeout policy into separate work.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection gives a credible path: configure a channel whose plugin fails before registration, and Gateway startup/readiness only iterate loaded channel plugins, so the configured channel can be absent from both startup attempts and readiness failures.

## Solution Assessment

Partially yes: a narrow configured-versus-loaded startup/readiness audit is the maintainable first fix. Fail-fast compatibility policy, hot plugin reload, startup metadata caching, and Feishu external API health behavior should be separate maintainer decisions or linked fixes.
