## Label Justifications

- `P1`: Gateway startup can report ready while a configured channel is unavailable, breaking a real channel workflow after plugin/version/dependency issues.
- `impact:message-loss`: When the channel is not initialized, outbound or inbound channel communication cannot be delivered even though the gateway remains up.

## Summary

Keep open: current main has partial plugin-load and bundled-channel diagnostics, but it still does not provide the requested channel initialization audit or a clear configured-versus-loaded failure summary when a channel is absent from the active registry.

## What This Changes

The issue requests clearer gateway diagnostics and compatibility handling when channel plugins or bundled channel modules fail to initialize.

## Best Possible Solution

Keep the existing doctor/minHost recovery paths, then add a focused gateway channel-start audit that reports configured, attempted, loaded, skipped, and failed channels with reasons; split hot reload and broad plugin-scan performance into separate design work if still needed.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the core diagnostic gap is reproducible from the startup path: only loaded channel plugins are started, and current code lacks a configured-versus-loaded audit summary.

## Solution Assessment

Partially yes: an audit log and clearer plugin/channel health surface are the right narrow fix. Fail-fast compatibility behavior, hot reload, and broad lazy-loading changes need maintainer scoping because current behavior intentionally continues startup after channel failures.
