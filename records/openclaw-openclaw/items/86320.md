## Label Justifications

- `P1`: Gateway can appear ready while a configured channel is unavailable, breaking real channel send/receive workflows after plugin or dependency load failures.
- `impact:message-loss`: When a configured channel never initializes, messages for that channel can be unavailable or undeliverable while the Gateway process keeps running.

## Summary

Keep open: current main has partial plugin-load and bundled-channel diagnostics, but Gateway startup/readiness still operate from the loaded channel registry only and do not provide the requested configured-versus-loaded channel initialization audit, so the central silent-skip diagnostic gap remains.

## What This Changes

Request clearer Gateway diagnostics when channel plugins are skipped or fail to initialize after plugin/runtime mismatches, missing generated modules, external API timeouts, or heavy startup load.

## Best Possible Solution

Add a focused Gateway channel-start/readiness audit that compares configured channel ownership with the active loaded registry and reports skipped or failed channels with reasons, while splitting hot reload and broad startup-performance work into separate design decisions.

## Reproduction Assessment

No high-confidence live current-main repro was run. Source inspection gives a credible reproduction for the diagnostic gap: startup and readiness only see loaded channel plugins, so configured-but-unregistered channels can be absent from both paths.

## Solution Assessment

Partially yes: clearer audit and health diagnostics are the right narrow fix. Fail-fast startup, hot reload, and performance changes are broader policy choices and should not be folded into the first repair without maintainer scoping.
