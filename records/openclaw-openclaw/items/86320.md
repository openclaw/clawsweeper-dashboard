## Label Justifications

- `P1`: A configured messaging channel can be unavailable after startup with no channel-level startup diagnosis, breaking real channel workflows.
- `impact:message-loss`: The reported failure leaves configured message channels unavailable or unable to send replies.

## Summary

Keep open: current main has better plugin-level and loaded-channel startup diagnostics than the reported 2026.5.12 build, but it still does not fully solve the central gap where a configured channel can fail before registration and never appear in the gateway channel startup path.

## What This Changes

The issue asks for clearer gateway diagnostics when channel plugins are skipped because of plugin/runtime mismatch, missing generated bundled modules, API timeouts, or heavy startup discovery.

## Best Possible Solution

Keep the existing plugin diagnostics, then add an owner-approved gateway channel initialization audit that includes configured channels that failed to register or load before `listChannelPlugins()`, while tracking hot reload and broad performance caching separately if maintainers want them.

## Reproduction Assessment

Yes, source-level: plugin registration failures are recorded in plugin diagnostics, but the gateway channel startup loop only sees already loaded channel plugins. I did not run the exact Windows Feishu mismatch scenario in this read-only review.

## Solution Assessment

Unclear as a single implementation: channel audit logging is the likely core fix, but fail-fast compatibility checks, hot reload, and startup caching are policy-sensitive and should not be bundled into an automated patch without maintainer direction.
