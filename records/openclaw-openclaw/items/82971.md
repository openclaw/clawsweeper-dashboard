## Summary

Keep open: the release-check idea is useful, but the patch wires the new scanner into a path that the official npm release workflow currently skips, so it does not yet protect the publish/preflight path that produced the stale Control UI package.

## What This Changes

The PR adds a packed Control UI JS protocol scanner to `scripts/openclaw-npm-release-check.ts` and unit tests for stale and current bundle fixtures.

## Best Possible Solution

Put the protocol comparison in the pack/tarball validation lane that actually runs during npm preflight, preferably shared with `release-check.ts` or the prepared tarball verifier, and prove it against a real stale and current package artifact.

## Reproduction Assessment

Yes, source-level: the PR adds the guard only under `collectPackedTarballErrors`, while the npm release workflow runs this script with `OPENCLAW_NPM_RELEASE_SKIP_PACK_CHECK=1`; the linked issue supplies concrete package mismatch evidence. I did not run the release workflow in this read-only review.

## Solution Assessment

No. Scanning packed Control UI assets is the right direction, but placing the scan only in the skipped `openclaw-npm-release-check` pack path is not the narrowest reliable fix for the publish/preflight failure class.
