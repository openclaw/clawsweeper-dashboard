## Summary

Keep open. Current main still has the reported Discord voice prompt gap: the voice transcript path can resolve the Discord channel override for authorization, but the successful voice agent run never forwards the per-channel `systemPrompt`, while Discord text/native ingress already builds and forwards that prompt through `GroupSystemPrompt` and `extraSystemPrompt`.

## What This Changes

Discord voice transcript agent runs should honor a per-channel `systemPrompt` override the same way Discord text ingress does.

## Best Possible Solution

Update the Discord plugin voice ingress so successful guild voice transcript runs reuse the trusted resolved channel config, build the same per-channel prompt used by text/native ingress, and pass it via `extraSystemPrompt` while keeping STT transcript text and channel metadata out of trusted system prompt content.

## Reproduction Assessment

Yes. Configure `channels.discord.guilds.<guild>.channels.<voiceChannelId>.systemPrompt`, drive `processDiscordVoiceSegment` through the existing mocked manager path, and inspect the `agentCommandFromIngress` options; current main has no path that can include that prompt.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Discord plugin voice path: carry or re-resolve the trusted channel config, reuse `buildDiscordGroupSystemPrompt`, and pass the result through the existing agent ingress `extraSystemPrompt` field rather than changing the shared agent runner.
