## Label Justifications

- `P1`: The report describes a current-release Control UI action killing and restarting the Gateway on a real systemd user install.
- `impact:crash-loop`: The affected behavior is process-level availability: selected Gateway PIDs are SIGTERM/SIGKILLed and systemd restarts the service.

## Summary

Keep open: current main has added a protected-PID path for service startup, but the stale Gateway cleanup still treats any non-excluded OpenClaw listener PID as killable and does not prove that Control UI/service-triggered cleanup cannot terminate live Gateway descendants or siblings.

## What This Changes

The issue reports that a Control UI cleanup action for two short-lived processes kills the live Gateway process tree on Linux/WSL systemd installs.

## Best Possible Solution

Define and implement a process-tree-safe cleanup boundary that never kills the live Gateway tree from Control UI or in-process restart paths, while preserving service-start protection against truly stale listener PIDs.

## Reproduction Assessment

Yes at source level, but not as a full live UI reproduction: the current cleanup path still kills broad non-excluded OpenClaw listener PIDs, while the issue provides concrete WSL/systemd SIGKILL restart logs.

## Solution Assessment

Unclear. The safe direction is to prevent live process-tree cleanup, but maintainers should choose whether that means removing the UI action, adding dry-run/confirmation, or tightening process ownership filters without breaking stale listener recovery.
