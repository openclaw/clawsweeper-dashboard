## Label Justifications

- `P1`: The report describes a broken ACP runtime workflow and gateway crash loop in a released Desktop build.
- `impact:crash-loop`: The issue reports repeated app-ready to polling-stopped gateway looping when trying the codex ACP backend.
- `impact:auth-provider`: The issue is about ACP backend/provider routing failing to register for configured acpx or codex backends.

## Summary

Current main and shipped releases after v2026.4.23 already address the reported ACPX backend-registration regression: the acpx plugin now registers the `acpx` ACP runtime backend during startup, and v2026.4.27 is the first stable release I found with that lazy registration path.

## What This Changes

The issue reports that AutoClaw Desktop v2026.4.23 cannot configure ACP through acpx or codex because the bundled acpx plugin loads without registering an ACP runtime backend.

## Best Possible Solution

Keep the shipped acpx plugin-owned backend registration path and direct users on v2026.4.23 to upgrade to v2026.4.27 or newer.

## Reproduction Assessment

Yes at source level: v2026.4.23 lacked the lazy backend registration body, and the registry throws the exact reported error when no backend is registered. Current main and v2026.4.27+ contain the acpx registration path that addresses that failure.

## Solution Assessment

Yes; the implemented fix keeps ACP backend ownership in the acpx plugin while exposing only the generic ACP runtime registry to core, which matches the repository's plugin boundary guidance.
