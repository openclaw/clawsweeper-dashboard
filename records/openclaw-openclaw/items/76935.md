## Summary

Keep open: current main and latest v2026.5.12 still have the QQ Bot path where ordinary setup leaves `streaming` unset, outbound dispatch maps that to block streaming enabled, and core then emits block replies as normal channel messages. Two users have now reported the same 2026.5.x regression, but the default was introduced intentionally, so the fix needs QQ Bot owner/product confirmation rather than cleanup closure.

## What This Changes

The issue reports that the external QQ Bot plugin in OpenClaw 2026.5.x sends multiple or repetitive replies compared with the bundled 2026.4.29 behavior.

## Best Possible Solution

Decide the intended QQ Bot default, then add regression coverage: either restore final-only delivery for unset config or keep streaming by default and fix the QQ Bot chunking/context path so one user turn cannot produce repetitive or stale-content replies.

## Reproduction Assessment

Yes for the multi-message path at source level: normal setup leaves `streaming` unset, QQ Bot maps that to `disableBlockStreaming: false`, and core enables block streaming. No live QQ run was performed, and the context-bleeding symptom is not separately isolated with high confidence.

## Solution Assessment

Unclear. A narrow code fix could treat unset QQ Bot streaming as final-only, but because the current default was intentionally introduced in https://github.com/openclaw/openclaw/pull/63746, the maintainable solution needs an owner decision before implementation.
