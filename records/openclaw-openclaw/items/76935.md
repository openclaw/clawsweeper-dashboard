## Summary

This should stay open: current main still has the QQ Bot default setup path that can enable block streaming for otherwise unset external-plugin configs, and the reporter's follow-up about cross-turn content is not resolved by an obvious shipped fix.

## What This Changes

The issue asks to fix a QQ Bot 2026.5.2 regression where the external plugin sends chunked, repetitive replies and may mix prior conversation context after upgrading from 2026.4.29.

## Best Possible Solution

Align QQ Bot's default delivery and conversation isolation with the intended channel contract: default installs should not spam or repeat replies, explicit streaming should remain configurable, and regression coverage should lock the chosen default.

## Reproduction Assessment

Yes at source level for the multi-message path: default setup leaves `streaming` unset, and current QQ Bot dispatch turns that into block streaming on. The context-bleed part is plausible from the report but not independently reproduced here.

## Solution Assessment

Unclear: the narrow code change would stop treating unset QQ Bot streaming as block streaming on, but that may reverse the documented #63746 default. Maintainer or QQ Bot owner confirmation should decide the default before implementation.
