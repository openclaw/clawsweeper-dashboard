## Summary

This should stay open: current main still has a QQ Bot default path that can enable block-style multi-message replies for unset streaming config, and the newer context-bleeding report has not been ruled out by current source or live verification.

## What This Changes

The issue reports that external QQ Bot in OpenClaw 2026.5.2 sends multiple/repetitive replies and may mix prior turn content after upgrading from 2026.4.29.

## Best Possible Solution

Align QQ Bot's default delivery contract with owner intent: unset/default installs should either produce one final reply or clearly document and configure intentional partial delivery, with regression coverage for the chosen behavior and a separate check for true context bleed.

## Reproduction Assessment

Yes at source level for the multi-message path: default setup leaves `streaming` unset, and the dispatcher does not disable block streaming for unset config. No live QQ credential test or transcript-based reproduction was available for the context-bleeding part.

## Solution Assessment

Unclear: the narrow code change would be to stop enabling block delivery for unset QQ Bot streaming config, but that may reverse the recent #63746 default-partial behavior. Maintainers should confirm the intended default before a repair PR changes runtime behavior.
