## Summary

This should stay open: current main still has a source-level path where default QQ Bot installs can emit multiple normal channel messages through block streaming, and the reporter's context-bleed update is plausible enough to need QQ Bot owner follow-up rather than cleanup closure.

## What This Changes

The issue reports that the external QQ Bot channel in 2026.5.2 sends multiple repetitive replies and bleeds prior conversation context after the channel moved from bundled to external packaging.

## Best Possible Solution

Align QQ Bot's default streaming contract across setup, runtime, command status, docs, and tests so default installs do not surprise users with repeated block replies unless maintainers explicitly choose that UX.

## Reproduction Assessment

Yes at source level for the multi-message part: unset QQ Bot streaming flows to `disableBlockStreaming: false`, which enables block replies. No for the full context-bleed claim yet, because I did not have live QQ transcripts showing cross-conversation leakage beyond normal session/group history.

## Solution Assessment

Unclear. The narrow code fix would stop treating an unset QQ Bot streaming config as block streaming on, but that may reverse the explicit default chosen in PR #63746, so the default needs maintainer/QQ Bot owner confirmation first.
