## Summary

Keep this open: current main still has a source-level QQ Bot path where a default setup leaves streaming unset and outbound dispatch forces block streaming on, matching the reported multi-message regression; the newer context-bleeding note needs QQ Bot owner confirmation and live/log detail before choosing the default behavior.

## What This Changes

The issue asks OpenClaw to stop QQ Bot replies after the 2026.5.2 external-plugin upgrade from splitting into repeated multi-message responses and mixing prior-turn context.

## Best Possible Solution

Align QQ Bot with a clear delivery contract: default installs should not emit repeated multi-bubble replies unless an operator explicitly opts into QQ Bot streaming, with tests covering unset, `off`, `partial`, `c2cStreamApi`, and group-history boundaries.

## Reproduction Assessment

Yes at source level: default setup leaves `streaming` unset, then QQ Bot passes `disableBlockStreaming: false`, which core resolves to block streaming on. I did not run a live QQ Bot credential test.

## Solution Assessment

Unclear: the narrow code fix is to stop treating unset QQ Bot streaming as block-streaming opt-in, but that may reverse the recently documented `partial` default from #63746 and should be confirmed by the QQ Bot owners.
