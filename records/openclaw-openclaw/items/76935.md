## Summary

Keep open: current main still has a source-level QQ Bot path that can enable block replies when `streaming` is unset, matching the multi-message symptom, and the context-bleeding update is not proven resolved or out of scope.

## What This Changes

The issue reports that the externalized QQ Bot plugin in 2026.5.2 sends 2-4 repetitive messages per prompt and may mix prior turn context after upgrading from the bundled 2026.4.29 channel.

## Best Possible Solution

Align QQ Bot's default streaming and session-context behavior with the intended channel contract: upgrade/default installs should avoid duplicate multi-message delivery unless the operator explicitly enables streaming, while preserving explicit `/bot-streaming` and official C2C streaming paths.

## Reproduction Assessment

Partly yes: the multi-message symptom is reproducible from source because default setup leaves `streaming` unset and the outbound path then enables block replies. The context-bleeding symptom is not high-confidence without live QQ logs or a C2C/group scope.

## Solution Assessment

Unclear: the narrow code direction is to stop treating unset QQ Bot `streaming` as block-streaming enabled, but that may reverse the recent #63746 default and needs QQ Bot owner confirmation.
