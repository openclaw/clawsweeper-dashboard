## Summary

Keep open: current main and v2026.5.7 still contain the QQ Bot default path that enables block streaming when setup leaves `streaming` unset, matching the reported multi-message regression; however, changing that default would reverse a recently intentional QQ Bot streaming change, so this needs owner/product confirmation rather than automatic closure.

## What This Changes

The issue reports that QQ Bot replies became multi-message, repetitive, and context-bleeding after upgrading to OpenClaw 2026.5.2 with the external `@openclaw/qqbot` plugin.

## Best Possible Solution

Decide the intended QQ Bot default first: either restore final-only delivery for unset config with tests and docs, or keep streaming-on-by-default and fix the QQ Bot streaming/chunking path so it cannot repeat or bleed prior content.

## Reproduction Assessment

Yes at source level for the multi-message behavior: default setup leaves `streaming` unset, QQ Bot maps that to `disableBlockStreaming: false`, and core turns that into block streaming. No live QQ Bot run was performed, and the context-bleeding part is not isolated with high confidence.

## Solution Assessment

Unclear. The narrow code change would be to stop forcing block streaming on for unset QQ Bot config, but that reverses the recently merged default-partial behavior, so the safer path is an owner decision followed by a focused regression test.
