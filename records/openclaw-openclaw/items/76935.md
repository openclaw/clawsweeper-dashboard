## Summary

Keep open: current main still owns the official QQ Bot plugin, and source inspection shows a plausible current-main repetition path where QQ Bot C2C streaming ignores upstream partial-reply `replace` semantics and appends prior text on prefix mismatch.

## What This Changes

The issue requests restoring QQ Bot replies to a single concise response after the 2026.5.2 externalized official plugin started sending multiple repetitive messages.

## Best Possible Solution

Fix QQ Bot's official streaming delivery to honor upstream partial-reply replacement semantics and keep the change plugin-local unless a shared SDK contract is missing.

## Reproduction Assessment

Yes from source inspection: emit one QQ C2C streaming partial followed by an upstream partial with `replace: true` and non-prefix text, and the current QQ streamer appends the prior text instead of replacing it. I did not live-run QQ, so this is source-reproducible rather than end-to-end reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is in `extensions/qqbot` to preserve/consume the `replace` signal or otherwise reset streaming state on replacement, with regression coverage around `StreamingController` and `dispatchOutbound`.
