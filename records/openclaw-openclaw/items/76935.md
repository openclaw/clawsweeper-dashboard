## Summary

This should stay open because current main still has a source-level QQ Bot default path that can emit multiple block replies for an unset `streaming` config, and the newer context-bleed report needs QQ Bot owner confirmation rather than cleanup closure.

## What This Changes

The issue reports verbose, repeated, and cross-turn QQ Bot replies after upgrading from 2026.4.29 to 2026.5.2 with the external `@openclaw/qqbot` plugin.

## Best Possible Solution

Align QQ Bot's default streaming contract across code, docs, commands, and tests, likely making unset config final-only unless owners confirm `partial` is still intended, then separately verify whether group/session context is bleeding beyond the documented history behavior.

## Reproduction Assessment

Yes for the multi-message part at source level: default setup leaves `streaming` unset, and dispatch then enables block streaming. No live QQ Bot run was performed, and the newer context-bleed claim is not yet independently reproduced.

## Solution Assessment

Unclear: the narrow code fix is to stop treating unset QQ Bot streaming as block-streaming-on, but that reverses a recently shipped/defaulted QQBot streaming behavior and should be confirmed by the QQ Bot owners.
