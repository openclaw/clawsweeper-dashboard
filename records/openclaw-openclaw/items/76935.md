## Summary

Keep open: current main and v2026.6.6 still contain the QQ Bot path where unset `streaming` leaves block streaming enabled, so the multi-message behavior is still source-reproducible. The reported repetition/context-bleeding part is not isolated enough to close as fixed or queue a safe automatic repair, especially because a QQ Bot maintainer noted that one reply split into multiple parts can be expected and asked for screenshots.

## What This Changes

The issue reports that the external QQ Bot plugin in OpenClaw 2026.5.x sends split, repetitive replies and may mix previous conversation context after upgrading from the bundled 2026.4.29 channel.

## Best Possible Solution

Choose and document the intended QQ Bot default first, then add focused regression coverage so one user turn either stays final-only when `streaming` is unset or uses streaming without repeating or mixing prior context.

## Reproduction Assessment

Yes for the multi-message portion at source level: unset QQ Bot streaming reaches `disableBlockStreaming: false`, and core maps that to block streaming on. No for the full repetition/context-bleeding claim: the reports need screenshots or live logs on current main to distinguish expected split blocks from duplicated or mixed replies.

## Solution Assessment

Unclear. A narrow default change could be plausible, but the best fix depends on a QQ Bot owner decision about whether unset `streaming` should preserve old final-only delivery or keep current split/block behavior with stronger dedupe/context safeguards.
