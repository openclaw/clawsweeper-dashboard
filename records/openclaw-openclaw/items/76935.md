## Summary

Keep open: current main still leaves a normal QQ Bot setup with `streaming` unset, and that path enables block-streamed channel replies unless the account explicitly sets `streaming: false` or `streaming.mode: "off"`. The later context-bleeding update is not fully proven from source, but the multi-message regression remains credible and the intended default conflicts with recent QQ Bot streaming work.

## What This Changes

The issue asks OpenClaw to restore QQ Bot to single concise replies after the 2026.5.2 external-plugin upgrade and investigate reported cross-turn content bleed.

## Best Possible Solution

QQ Bot should have a documented, test-covered default that either preserves single-message replies unless explicitly enabled or clearly opts users into streaming, with separate coverage for session/context isolation.

## Reproduction Assessment

Yes at source level for the multi-message part: default setup leaves `streaming` unset and the outbound dispatcher then allows block streaming. No live QQ Bot test was run, and the cross-turn context-bleed claim is not independently reproduced.

## Solution Assessment

Unclear: changing the QQ Bot default to final-only would be a narrow technical fix for multiple replies, but it may undo the recently intentional #63746 `partial` default. The default policy should be confirmed by the QQ Bot owners before implementation.
