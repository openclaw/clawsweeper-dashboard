## Summary

Keep open: current main and the latest release still contain the QQ Bot path where ordinary setup leaves `streaming` unset, QQ Bot maps that to block streaming enabled, and core can split one reply into multiple channel messages. The later context-bleeding symptom is not isolated, but the multi-message regression remains source-reproducible and needs a QQ Bot owner/default-policy decision rather than conservative cleanup closure.

## What This Changes

The issue reports that QQ Bot replies became split, repetitive, and sometimes context-mixed after upgrading from 2026.4.29 to 2026.5.x externalized QQ Bot.

## Best Possible Solution

Choose the intended QQ Bot default first: restore final-only delivery when `streaming` is unset, or keep streaming enabled and fix the QQ Bot chunking/context path so one user turn cannot produce repeated or context-mixed visible replies.

## Reproduction Assessment

Yes at source level for the multi-message behavior: default setup can leave QQ Bot streaming unset, current QQ Bot maps that to `disableBlockStreaming: false`, and core turns that into block streaming. No live QQ Bot run was performed, and the context-bleeding part is not isolated with high confidence.

## Solution Assessment

Unclear. The narrow code change would make undefined QQ Bot streaming behave like `off`, but that may reverse recent intentional streaming work; the safer solution is an owner decision followed by focused regression coverage.
