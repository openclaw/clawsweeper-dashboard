## Summary

Keep open: current main still has a source-level QQ Bot path where fresh/default account config leaves `streaming` unset, then `dispatchOutbound` passes `disableBlockStreaming: false`, which forces block replies and matches the reported multi-message regression. The right product default is still ambiguous because #63746 intentionally made QQBot partial/block streaming the default, while the later context-bleed report needs QQ Bot owner triage rather than cleanup closure.

## What This Changes

The issue reports that external `@openclaw/qqbot` in 2026.5.2 emits multiple or repeated replies and may mix prior conversation context after upgrading from the bundled 2026.4.29 behavior.

## Best Possible Solution

Make the QQ Bot streaming default explicit and consistent across setup, schema, docs, and dispatch behavior, then add focused regression coverage for default external installs and any confirmed context-bleed path.

## Reproduction Assessment

Yes for the multi-message regression at source level: default setup leaves `streaming` unset, and current dispatch forces block streaming on. The context-bleeding detail still needs live or targeted fixture reproduction to identify whether it is C2C session history, group pending history, or streaming duplication.

## Solution Assessment

Unclear: the narrow code change would stop forcing block streaming on for unset QQ Bot config, but #63746 intentionally made partial streaming the default. Owner confirmation should decide whether to restore final-only default, migrate configs, or document the new default with an explicit `streaming: false` workaround.
