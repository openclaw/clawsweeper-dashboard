## Summary

Keep open: current main and v2026.5.12 still leave ordinary QQ Bot setup with `streaming` unset, and that path enables OpenClaw block streaming, matching the reported multi-message regression; the newer context-bleeding symptom is credible but not isolated to one source path yet.

## What This Changes

The issue asks for QQ Bot replies after the 2026.5.2 external-plugin upgrade to return as a single concise answer without repeated or prior-turn content.

## Best Possible Solution

Decide the intended QQ Bot default first, then either restore final-only delivery for unset config with tests/docs or keep streaming-on-by-default and fix the QQ Bot streaming/history path so it cannot repeat or mix prior-turn content.

## Reproduction Assessment

Yes for the multi-message path at source level: ordinary setup leaves `streaming` unset, QQ Bot maps that to block streaming enabled, and core can send block chunks as channel messages. No live QQ Bot run was performed, and the context-bleeding part remains only user-reported/source-adjacent rather than fully reproduced.

## Solution Assessment

Unclear. A narrow code change could make unset QQ Bot config final-only again, but that would reverse an intentional default-partial change, so the safer solution is owner confirmation followed by focused regression coverage.
