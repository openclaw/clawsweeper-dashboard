## Summary

Keep open: current main and the latest release still contain the QQ Bot path where normal setup can leave `streaming` unset, QQ Bot maps that to block streaming enabled, and core can emit one reply as multiple channel messages. The later context-bleeding report is not fully isolated, but the split/repeated-message regression remains source-reproducible and is supported by two external user reports.

## What This Changes

Investigate a reported QQ Bot regression where OpenClaw 2026.5.x sends multiple or repeated replies after the channel moved to the external `@openclaw/qqbot` plugin.

## Best Possible Solution

Decide the intended QQ Bot default first, then either restore final-only delivery when `streaming` is unset or keep streaming enabled and fix the QQ Bot chunking/context path with focused regression coverage.

## Reproduction Assessment

Yes, at source level for the multi-message behavior: default setup can leave QQ Bot `streaming` unset, current QQ Bot maps that to `disableBlockStreaming: false`, and core turns that into block streaming. No live QQ Bot run was performed, and the context-bleeding part is not isolated with high confidence.

## Solution Assessment

Unclear. A narrow code change could make undefined QQ Bot streaming behave like `off`, but that may reverse recent intentional streaming work; the safer path is an owner decision followed by targeted tests.
