## Summary

Keep open: current main still has a source-level QQ Bot path where default setup leaves `streaming` unset while outbound dispatch enables block replies, which matches the reported multi-message regression; changing that default conflicts with the documented #63746 partial-streaming default and needs QQ Bot owner confirmation.

## What This Changes

The issue asks OpenClaw/QQ Bot to stop sending multiple or repetitive replies, including possible context bleed, after upgrading to 2026.5.2 and using external `@openclaw/qqbot`.

## Best Possible Solution

Define the QQ Bot streaming default explicitly, then align setup, `/bot-streaming` status, docs, and regression tests with that contract so default installs either send one final reply or clearly opt into streamed block replies.

## Reproduction Assessment

Yes at source level for the multi-message behavior: default setup leaves `streaming` unset, and the outbound path enables block streaming for that state. The context-bleeding follow-up is not live-reproduced and remains less certain.

## Solution Assessment

Unclear: the narrow code fix would restore final-only delivery for unset QQ Bot streaming, but #63746 documented `partial` as the intended default. A maintainer should choose the contract before a patch changes released behavior.
