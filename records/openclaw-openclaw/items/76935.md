## Summary

This should stay open: current main still has the QQ Bot default path that can produce multiple normal channel messages after setup, and the reporter's newer context-bleeding note needs owner follow-up rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to restore concise, single-reply QQ Bot behavior after the 2026.5.2 external plugin upgrade and investigate possible cross-turn context bleed.

## Best Possible Solution

Align QQ Bot defaults, status commands, and docs so a fresh install has one well-defined delivery mode; if partial streaming remains intended, make opt-out and coalescing behavior explicit and fix any separate context bleed with focused QQ Bot coverage.

## Reproduction Assessment

Yes at source level for the multi-message symptom: default setup leaves streaming unset, QQ Bot passes disableBlockStreaming false, and core enables block replies from that value. The context-bleeding symptom is not yet reproduced without live QQ Bot traffic and transcript/session evidence.

## Solution Assessment

Unclear: the narrow code change would stop treating unset QQ Bot streaming as block streaming on, but that may reverse the #63746 default-partial behavior without owner confirmation. A safer path is to first decide the intended default, then update outbound logic, /bot-streaming status, docs, and tests together.
