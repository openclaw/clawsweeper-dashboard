## Summary

This should stay open: current main still has a QQ Bot delivery path that can force block streaming on for default external-plugin installs, matching the reported multi-message regression, but the right default needs QQ Bot owner confirmation because recent changelog text describes partial streaming as intentional.

## What This Changes

The issue asks OpenClaw to restore concise single-message QQ Bot replies after upgrading to 2026.5.2 with the external `@openclaw/qqbot` plugin.

## Best Possible Solution

Align QQ Bot with the intended channel streaming contract: default installs should produce one final reply unless the operator explicitly enables QQ Bot streaming, while explicit C2C official streaming should continue using one replace-style stream message.

## Reproduction Assessment

Yes at source level: default QQ Bot setup leaves `streaming` unset, then `dispatchOutbound` forces block streaming on through `disableBlockStreaming: false`. I did not run a live QQ Bot credential test.

## Solution Assessment

Unclear without QQ Bot owner input: the narrow code fix is to stop forcing block streaming on for unset config, but that may reverse the recently documented `partial` default for QQBot.
