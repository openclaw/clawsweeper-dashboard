## Summary

Keep open: current `main` and v2026.5.12 still contain the QQ Bot path that turns unset `streaming` config into block-streaming delivery, matching the reported multi-message regression; the context-bleeding part is plausible but not isolated to one path. Because that default came from a deliberate QQ Bot streaming change, this needs owner/product confirmation rather than cleanup closure.

## What This Changes

The issue reports that external QQ Bot replies became multi-message, repetitive, and sometimes cross-contaminated between turns after upgrading from 2026.4.29 to 2026.5.2.

## Best Possible Solution

Decide the intended QQ Bot default first, then either restore final-only delivery for unset config or keep streaming-on-by-default and fix the chunking/repetition/context path with docs and regression coverage.

## Reproduction Assessment

Yes for the multi-message part at source level: default setup can leave `streaming` unset, QQ Bot maps that to `disableBlockStreaming: false`, and core enables block streaming. No live QQ Bot run was performed, and the context-bleeding symptom is not isolated with high confidence.

## Solution Assessment

Unclear. The narrow code change would be to restore final-only behavior when QQ Bot `streaming` is unset, but that would reverse the intentional default from https://github.com/openclaw/openclaw/pull/63746, so owner confirmation should come first.
