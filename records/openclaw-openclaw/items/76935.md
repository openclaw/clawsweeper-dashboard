## Summary

Keep open: current main still maps a default QQ Bot install with no `streaming` config to block streaming enabled, which matches the reported multi-message regression; the newer context-bleed update is plausible but needs narrower QQ Bot owner diagnosis.

## What This Changes

The issue asks to restore concise single-message QQ Bot replies after 2026.5.2 and prevent prior-topic context from bleeding into current QQ replies.

## Best Possible Solution

Have QQ Bot owners align the external plugin with the channel streaming contract, then add regression coverage for unset streaming, explicit streaming on/off, and any confirmed context-boundary bug.

## Reproduction Assessment

Yes at source level for the multi-message part: default setup leaves `streaming` unset, and runtime turns that into block streaming enabled. The newer context-bleed claim is unclear without a focused live or unit reproduction that distinguishes C2C, group history, and normal session memory.

## Solution Assessment

Unclear: the narrow technical fix is likely to stop treating unset QQ Bot streaming as block streaming enabled, but that may reverse the recently documented `partial` default from #63746. Owner confirmation should settle the default before implementation.
