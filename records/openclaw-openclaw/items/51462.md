## Summary

Keep this PR open: current main still skips the assistant update for textless tool-call-only assistant messages, and the PR’s focused presence check is still useful. The remaining blocker is review hygiene for an external PR, especially inspectable real behavior proof and a current-main rebase/validation pass.

## What This Changes

The PR changes the embedded message-end handler to emit an assistant update when an assistant message contains only structured tool-call blocks and no text or media.

## Best Possible Solution

Land a rebased, current-main version that emits the assistant update for textless structured tool-call messages with focused regression coverage, and track relay-visible tool-call metadata separately if maintainers want that UX.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `handleMessageEnd` with an assistant message whose content is only a `toolCall`/`toolUse`/`functionCall` block and observe that current main's `(cleanedText || hasMedia)` guard does not emit an assistant event.

## Solution Assessment

Yes for the scoped subscription-handler state bug. A presence-only block-type check is narrower than adding synthetic relay markers, while using the existing ID-based extraction helper would be too strict for providers that omit tool-call IDs.
