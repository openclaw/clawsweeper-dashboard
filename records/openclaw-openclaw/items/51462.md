## Summary

Keep this PR open. Current main still lacks the tool-call-only assistant update path, and the PR targets a real gap, but the branch is stale against the current handler shape and still needs maintainer review for regression coverage and the empty-event relay behavior.

## What This Changes

The PR adds a local structured tool-call-block presence check to `handleMessageEnd` and uses it to emit an assistant stream update even when the assistant message has no text or media.

## Best Possible Solution

Adapt the focused handler fix to current main, add regression coverage for textless tool-call-only assistant messages across `toolCall`/`toolUse`/`functionCall` and blank or missing ids, then make an explicit maintainer call on whether relay consumers should receive structured tool-call metadata or a non-empty progress marker.

## Reproduction Assessment

Yes. A focused unit-level reproduction is to call `handleMessageEnd` with an assistant message whose `content` contains only a structured tool-call block and no text/media, then assert that current main does not call `onAgentEvent` while the intended fix should.

## Solution Assessment

Unclear. The presence-only check is a narrow and plausible fix for the subscription-handler state path, but the PR is stale against current main and does not settle whether empty assistant events are sufficient for relay consumers.
