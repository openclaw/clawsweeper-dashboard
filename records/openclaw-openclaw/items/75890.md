## Summary

Keep open. The PR targets a real Signal linked-device Note-to-Self gap, but the patch is not mergeable as written because it reads through `syncMessage: unknown` and changes documented self-message handling without an explicit opt-in, tests, docs, or changelog coverage.

## What This Changes

The PR promotes self-addressed Signal `syncMessage.sentMessage` envelopes to `dataMessage` so linked-device Note-to-Self messages can reach normal inbound processing.

## Best Possible Solution

Land a Signal-plugin-owned fix that narrows and validates `syncMessage.sentMessage`, preserves sent-transcript loop protection, gates Note-to-Self behavior behind a documented maintainer-approved default, and adds focused regression tests plus changelog/docs updates.

## Reproduction Assessment

Yes. A source-level reproduction is clear: feed `createSignalEventHandler` a Signal receive event whose own-account `syncMessage.sentMessage` has matching destination/source; current main returns before dispatch, and the PR attempts to route that envelope.

## Solution Assessment

No. The destination/source check is the right core guard, and upstream signal-cli supports those fields, but this implementation needs a typed payload guard, opt-in/default decision, regression tests, and docs/changelog alignment before it is the maintainable fix.

## Review Findings

  - body: `SignalEnvelope.syncMessage` is typed as `unknown`, so this optional-chain property access will fail strict TypeScript checking. Add a local type guard/schema for the sync sent-message shape before reading `sentMessage`, `destinationNumber`, or `destinationUuid`.
  - confidence: 0.94
- **[P2] Gate Note-to-Self ingestion behind an explicit opt-in:** `extensions/signal/src/monitor/event-handler.ts:555-556`
  - body: Current Signal docs promise that personal-account self messages are ignored for loop protection, but this unconditional promotion would start sending private Note-to-Self traffic to the agent for existing linked-device configs whose own sender passes `open` or allowlist access. Add a documented config gate or get maintainer approval for the default behavior change before routing these messages automatically.
  - confidence: 0.78
