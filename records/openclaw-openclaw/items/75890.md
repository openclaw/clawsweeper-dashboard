## Summary

Keep open. The PR targets a real source-visible Signal Note-to-Self limitation, but the branch still admits self-addressed `syncMessage.sentMessage` traffic across the existing loop-protection boundary without an echo/replay discriminator, focused tests, or inspectable real behavior proof.

## What This Changes

The PR promotes self-addressed Signal linked-device `syncMessage.sentMessage` envelopes into inbound `dataMessage` when the operator number is in `channels.signal.allowFrom`, and updates Signal docs plus the changelog.

## Best Possible Solution

Implement Signal self-chat inside the Signal plugin with an explicit opt-in, a tested outbound echo/replay discriminator such as source-device or outbound timestamp handling, regression coverage for both admission and rejection, and matching public docs.

## Reproduction Assessment

Yes for the current limitation by source inspection: a receive envelope from the configured Signal account with a self-addressed `syncMessage.sentMessage` returns at the own-account guard and also matches the blanket sync-message drop. I did not run a live linked-device Signal reproduction in this read-only review.

## Solution Assessment

No for this branch as written. The `allowFrom` opt-in is a reasonable direction, but `destination === source` alone does not distinguish user-authored Note-to-Self messages from OpenClaw-authored or replayed sentTranscripts before the loop guard is bypassed.
