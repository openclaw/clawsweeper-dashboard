## Summary

Keep open. The Note-to-Self problem is valid and the PR now includes docs and changelog coverage, but the current diff still routes any self-addressed `syncMessage.sentMessage` in self-chat mode, which matches signal-cli sentTranscript sync traffic and can re-ingest an OpenClaw Note-to-Self reply if that outbound is replayed.

## What This Changes

The branch updates Signal's event handler to opt self-allowlisted linked-device accounts into Note-to-Self syncMessage ingestion, and adds Signal docs plus an Unreleased changelog entry.

## Best Possible Solution

Land Signal-owned Note-to-Self support after the self-addressed sync path can distinguish user-authored Note-to-Self messages from OpenClaw's own sentTranscript echoes, with regression coverage for both paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enable self-chat by putting the account in `allowFrom` and feed the PR handler a self-addressed `syncMessage.sentMessage`; current main drops the envelope while the PR promotes it to `dataMessage`.

## Solution Assessment

No. The `allowFrom` opt-in and docs/changelog updates are a good direction, but destination/source equality is not a sufficient origin guard for Signal sync traffic; the safer fix needs signal-cli echo semantics or an internal outbound-replay guard plus tests.

- **[P2] Keep self sentTranscripts out of self-chat:** `extensions/signal/src/monitor/event-handler.ts:573-585`
  - body: In self-chat mode this branch promotes every self-addressed `syncMessage.sentMessage` into `dataMessage`. signal-cli's JSON-RPC docs show that sync sentMessage payloads use the same source/destination-equal shape, and current main filters syncMessage presence to avoid replaying sentTranscript payloads.
