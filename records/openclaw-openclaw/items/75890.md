## Summary

Keep open. The PR now includes the requested Signal docs and changelog updates, but the code path still appears to bypass the existing sentTranscript loop guard for self-addressed sync traffic when self-chat mode is enabled.

## What This Changes

The PR changes the Signal handler to opt into linked-device Note-to-Self syncMessage ingestion via `allowFrom`, updates Signal docs, and adds a changelog entry.

## Best Possible Solution

Land Signal-owned Note-to-Self support only after the self-addressed sync path can distinguish user-authored Note-to-Self messages from OpenClaw's own sentTranscript echoes, with regression tests for both cases.

## Reproduction Assessment

Yes. A source-level reproduction is clear: feed the PR handler a self-chat-enabled, self-addressed `syncMessage.sentMessage`; the PR promotes it to `dataMessage`, while current main drops all sync envelopes before dispatch.

## Solution Assessment

No. The allowFrom opt-in and destination/source guard are a plausible start, but the implementation must preserve the existing sentTranscript loop guard before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep self sentTranscripts out of self-chat:** `extensions/signal/src/monitor/event-handler.ts:573-585`
  - body: In self-chat mode this branch promotes every self-addressed `syncMessage.sentMessage` before the own-account drop can run. That same self-addressed sync shape is used by signal-cli sentTranscript traffic, and current main explicitly filters syncMessage presence to prevent replaying the bot's own sent messages. If an OpenClaw Note-to-Self reply is echoed or replayed as a sentTranscript, the agent can ingest its own outbound and loop; add an origin/echo guard plus coverage before routing this path.
  - confidence: 0.84
