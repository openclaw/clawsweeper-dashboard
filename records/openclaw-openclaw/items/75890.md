## Summary

Keep open. The PR addresses a real Signal Note-to-Self gap and the latest head adds an `allowFrom` self-chat gate, but it still changes documented Signal behavior while only modifying the handler, leaving docs, changelog, and focused regression coverage missing.

## What This Changes

This PR modifies the Signal event handler to promote self-addressed `syncMessage.sentMessage` envelopes to inbound `dataMessage` when the operator's own E.164 number appears in `allowFrom`.

## Best Possible Solution

Land a Signal-plugin-owned fix that keeps self-chat opt-in through explicit self allowlisting, preserves sentTranscript loop protection, and includes matching Signal docs, changelog, and targeted tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: feed `createSignalEventHandler` an own-account `syncMessage.sentMessage` whose destination matches the source; current main drops it before dispatch, while the PR routes it when self-allowlisted.

## Solution Assessment

No, not yet. The handler approach is plausible after the allowFrom gate, but the PR should update the documented number model and add release notes plus regression coverage before merge.
