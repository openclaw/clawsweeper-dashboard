## Summary

Keep open. The PR targets a real Signal linked-device Note-to-Self gap and includes docs/changelog updates, but the current guard still treats destination/source equality as sufficient to route self-addressed sentTranscript sync traffic into the agent pipeline.

## What This Changes

The PR changes the Signal event handler to promote self-addressed `syncMessage.sentMessage` envelopes in `allowFrom` self-chat mode, updates Signal docs, and adds an Unreleased changelog entry.

## Best Possible Solution

Land Signal-owned Note-to-Self support only after the self-addressed sync path can distinguish user-authored Note-to-Self messages from OpenClaw's own sentTranscript echoes, with focused regression coverage for the allowed and denied sync shapes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: feed the PR handler a self-chat-enabled, self-addressed `syncMessage.sentMessage`; current main drops the sync envelope, while the PR promotes it to `dataMessage`, and that same shape also matches the sentTranscript risk.

## Solution Assessment

No. The `allowFrom` opt-in is a reasonable direction, but destination/source equality is not the narrowest maintainable guard until the own-reply echo path is excluded or proven impossible.

  - confidence: 0.86

  - body: `isSelfChatMode` can only turn on when an E.164 `account` matches `allowFrom`, but the new Note-to-Self check also claims to support `destinationUuid === sourceUuid` and current main supports UUID-only own-account loop protection. A UUID-only Signal account or `uuid:<id>` allowlist will still drop self-chat sync messages; either support that opt-in or document/test the E.164-only boundary.
  - confidence: 0.8
