## Summary

Keep open; current main still drops the reported Signal Note-to-Self shape, but the PR is not merge-ready because the new self-chat branch accepts sentTranscript-shaped self-sync traffic under the same condition that prior loop protection was built to reject.

## What This Changes

The PR updates Signal's event handler to promote self-addressed linked-device sync messages into inbound messages when the account is self-allowlisted, and adds matching Signal docs plus a changelog entry.

## Best Possible Solution

Implement Signal Note-to-Self support with a tested guard that accepts user-authored self-chat only while preserving sentTranscript and own-message loop protection, then land with docs and changelog.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main drops own-account messages and any `syncMessage`, while signal-cli documents self-addressed `syncMessage.sentMessage` receive payloads. I did not run a live Signal linked-device test in this read-only review.

## Solution Assessment

No. The allowFrom opt-in, docs, and changelog are useful, but destination/source equality is not a sufficient loop guard for sentTranscript-shaped self-sync traffic; the safer path needs an origin/echo discriminator plus regression tests.

## Review Findings

Full review comments:

- **[P2] Keep self sync echoes out of self-chat:** `extensions/signal/src/monitor/event-handler.ts:573-585`
  - body: This branch promotes every self-addressed `syncMessage.sentMessage` in self-chat mode before the own-account guard can run. signal-cli documents sent-message sync traffic with matching source and destination identifiers, and the existing code drops sync envelopes specifically to avoid replaying sentTranscripts.
