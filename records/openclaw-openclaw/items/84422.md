## Summary

Keep open: this is a plausible gateway availability fix, but the branch still shuts down the unsupervised SSH/TUI path it says should stay online, and the supplied proof exercises a simulator/unit harness rather than a real gateway receiving SIGHUP.

## What This Changes

The PR adds a gateway run-loop SIGHUP handler that ignores SIGHUP under detected supervision, gracefully stops otherwise, and adds two signal-handling tests.

## Best Possible Solution

Confirm the real SSH/TUI signal path, then land an explicit SIGHUP policy that preserves gateway availability where intended, with focused tests and redacted live terminal or log proof from the actual gateway process.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for unhandled SIGHUP: current main lacks a SIGHUP run-loop listener and a no-handler Node process exits 129. No high-confidence live reproduction is present for the full SSH/TUI or supervised gateway outage path.

## Solution Assessment

Is this the best way to solve the issue? No: the current PR prevents the default unhandled crash but still stops the unsupervised gateway, so it does not yet match the availability problem stated in the PR. The safer path is to prove the real signal and make the SIGHUP stop-versus-ignore policy explicit.
