## Summary

Keep open: the PR targets a plausible missing gateway SIGHUP handler, but it is not merge-ready because the direct SSH/TUI path still shuts the gateway down and the linked outage evidence points to SIGTERM rather than SIGHUP.

## What This Changes

The PR adds a gateway run-loop SIGHUP listener that ignores SIGHUP under detected supervisors, otherwise routes it through graceful stop, and adds two run-loop signal tests.

## Best Possible Solution

Confirm the real SSH disconnect signal path, then land a SIGHUP policy that keeps supervised or daemon gateways available without surprising existing stop/restart behavior, with live proof and focused tests.

## Reproduction Assessment

Source-level yes for unhandled SIGHUP: current main lacks a SIGHUP run-loop listener and Node documents default SIGHUP termination. No high-confidence live reproduction is present for the linked SSH/systemd outage, whose only journal evidence shows SIGTERM.

## Solution Assessment

No. Adding a SIGHUP handler is the right code boundary, but stopping the unsupervised SIGHUP path does not preserve the gateway for the direct SSH/TUI case and the linked SIGTERM evidence remains unaddressed.
