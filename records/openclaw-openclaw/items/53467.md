## Best Possible Solution

Keep the opt-in Slack parity feature, but gate other-mention drops on Slack-ID-specific bot mention certainty, cover degraded identity with non-empty mention regexes in tests, align docs to that contract, and require redacted live Slack proof.

## Reproduction Assessment

Yes, by source inspection. With `ignoreOtherMentions=true`, empty `ctx.botUserId` from the non-fatal Slack `auth.test` path, and non-empty mention regexes, a real `<@BOT_ID>` message can satisfy `hasAnyMention && !wasMentioned` and hit the new drop branch.

## Solution Assessment

No, not as currently written. The config/schema/resolver shape is reasonable, but the drop gate should require Slack-ID-specific bot mention certainty instead of treating generic mention regexes as enough to classify Slack `<@...>` tokens as non-bot mentions.
