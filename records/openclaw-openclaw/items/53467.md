## Summary

Keep open: the Slack parity feature is useful and not on current main, but the latest diff can still drop legitimate bot-directed Slack messages when botUserId is unavailable and regex mention detection is configured, and the PR lacks inspectable live Slack proof.

## What This Changes

This PR adds a Slack per-channel `ignoreOtherMentions` config key, schema/docs/generated metadata, resolver wiring, inbound drop logic, tests, and a changelog entry.

## Best Possible Solution

Keep the Slack parity feature, but require Slack-ID explicit mention certainty, or an equivalent Slack-ID-specific classifier, before applying the other-mention drop; align docs/tests with that contract and add redacted live Slack proof.

## Reproduction Assessment

Yes, by source inspection against the PR diff. Enable `ignoreOtherMentions`, let Slack `auth.test` leave `botUserId` empty, configure a non-matching mention regex or agent identity so `canDetectMention` is true, and a real `<@BOT_ID>` channel ping can satisfy `hasAnyMention && !wasMentioned` and be dropped.

## Solution Assessment

No, not as currently written. The config/schema/resolver shape is maintainable, but the drop gate should require Slack-ID certainty instead of the broad mention-regex detectability used for normal mention gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require Slack ID certainty before dropping mentions:** `extensions/slack/src/monitor/message-handler/prepare.ts:967`
  - body: The new drop branch treats `canDetectMention` as enough certainty, but that is true when mention regexes exist even if Slack `auth.test` left `ctx.botUserId` empty. In that supported fallback, a real `<@BOT_ID>` ping will not set `wasMentioned` unless a regex happens to match the text, so enabling this option can silently discard bot-directed channel messages. Gate this filter on a resolved Slack bot user id or an equivalent Slack-ID explicit mention signal.
  - confidence: 0.88
