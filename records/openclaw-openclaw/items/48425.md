## Summary

Keep this PR open, but it is not mergeable as written: current main already strips the bulky runtime-only `resolvedSkills` field, while this PR removes the whole `skillsSnapshot` key and can drop unrelated nested data with the same name. It also still lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds a shared JSON.stringify replacer that removes `skillsSnapshot` from session-store serialization and disk-budget byte estimates, plus serialization tests.

## Best Possible Solution

Preserve the current lightweight `skillsSnapshot` metadata contract, keep bulky runtime-only data out of persistence with a scoped serializer or normalization step, align disk-budget estimates, and require redacted real-world size/runtime proof before merge.

## Reproduction Assessment

Yes for source-level behavior: current main preserves lightweight `skillsSnapshot` fields and strips only `resolvedSkills`, while the PR's global replacer would remove the whole key anywhere in the JSON graph. I did not run a live size reproduction in this read-only review.

## Solution Assessment

No. A global key-name replacer is not the narrowest maintainable fix because it loses metadata current main intentionally preserves and can remove unrelated nested persisted data with the same key name.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve lightweight skill snapshot metadata:** `src/config/sessions/store-serialization.ts:10`
  - body: The replacer removes every `skillsSnapshot` from session-store serialization. Current main only treats `resolvedSkills` as runtime-only: tests preserve `prompt`, `skills`, `skillFilter`, and `version`, and `agent-command.ts` uses those fields to avoid rebuilding snapshots when version/filter still match. This PR would make resumed sessions lose that metadata after the next save/restart.
  - confidence: 0.93
- **[P2] Constrain filtering to the intended session field:** `src/config/sessions/store-serialization.ts:10`
  - body: `JSON.stringify` applies this replacer to every nested property, so any persisted object containing a key named `skillsSnapshot` is silently dropped. ACP `backendExtras` accepts arbitrary safe keys and is persisted under `entry.acp.runtimeOptions`, so a backend option with this name would be lost after save/restart; scope the serializer by path or avoid a global key-name filter.
  - confidence: 0.9
