## Summary

Keep open: the bug is valid and the PR is on the right path, but the current head globally drops any `skillsSnapshot` key during JSON serialization, which can lose nested ACP/plugin-owned data after save and restart.

## What This Changes

The PR adds a shared session-store serialization replacer, applies it to session saves and disk-budget sizing, and adds tests for omitting `skillsSnapshot` while preserving `systemPromptReport`.

## Best Possible Solution

Land a scoped session-store serialization helper that removes only top-level `SessionEntry.skillsSnapshot` from disk writes and disk-budget sizing, preserves `systemPromptReport`, and leaves nested ACP/plugin-owned data intact.

## Reproduction Assessment

Yes. Source inspection shows current main persists full `SessionEntry.skillsSnapshot`, and the PR regression is reproducible by serializing a session entry with nested `acp.runtimeOptions.backendExtras.skillsSnapshot`.

## Solution Assessment

No, not as written. The intended persistence fix is narrow and maintainable, but the global key-name replacer should be replaced with a path-aware sanitizer or store-copy serialization that only omits the top-level session entry field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Scope the snapshot filter to session entries:** `src/config/sessions/store-serialization.ts:10`

ACP `backendExtras` can legally persist arbitrary string keys, including `skillsSnapshot`, and those values would disappear after save/restart; filter only the top-level `SessionEntry.skillsSnapshot` field.
