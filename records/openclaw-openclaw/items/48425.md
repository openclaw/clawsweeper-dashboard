## Summary

Keep open: the storage-bloat problem is real, but this PR still lacks inspectable real behavior proof and its global serializer conflicts with current main's narrower `resolvedSkills` persistence contract.

## What This Changes

The PR adds a shared JSON.stringify replacer to omit `skillsSnapshot` from session-store writes and disk-budget byte estimates, plus serialization tests.

## Best Possible Solution

Preserve lightweight `skillsSnapshot` metadata, omit or dedupe only truly bulky runtime-only data with path-aware serialization, keep disk-budget estimates aligned, and attach redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main persists lightweight `skillsSnapshot` fields after normalization, and the PR's replacer would remove the whole key anywhere in the JSON graph; I did not run a live size repro in this read-only review.

## Solution Assessment

No. A global key-name replacer is not the narrowest maintainable fix because current main intentionally preserves lightweight snapshot metadata and nested persisted data can legally reuse the same key name.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve lightweight skill snapshot metadata:** `src/config/sessions/store-serialization.ts:10`
  - body: The replacer removes every `skillsSnapshot` from session-store serialization. Current main only treats `resolvedSkills` as runtime-only: tests preserve `prompt`, `skills`, `skillFilter`, and `version`, and `agent-command.ts` uses those fields to avoid rebuilding snapshots when version/filter still match. This PR would make resumed sessions lose that metadata after the next save/restart.
  - confidence: 0.92
- **[P2] Constrain the replacer to the session-entry field:** `src/config/sessions/store-serialization.ts:10`
  - body: `JSON.stringify` applies this replacer to every nested property, so any persisted object containing a key named `skillsSnapshot` is silently dropped. ACP `backendExtras` accepts arbitrary safe keys and is persisted under `entry.acp.runtimeOptions`, so a backend option with this name would be lost after save/restart; scope the serializer by path or avoid a global replacer.
  - confidence: 0.9
