## Summary

The central storage-bloat fix is already shipped on main in a narrower form that strips runtime-only resolvedSkills while preserving lightweight snapshot metadata. The remaining branch behavior is risky because it uses a global key-name serializer, so this older PR should close rather than merge.

## What This Changes

The branch adds a session-store JSON.stringify replacer that drops skillsSnapshot during save and disk-budget estimation, plus serialization tests.

## Best Possible Solution

Keep the shipped resolvedSkills stripping and hydration behavior, leave broader session-storage pruning or lazy-loading work to canonical open reports, and avoid a global key-name serializer.

## Reproduction Assessment

Yes for source-level behavior: current main preserves lightweight skillsSnapshot metadata and strips only resolvedSkills, while this PR would remove the whole key anywhere in the JSON graph. I did not run a live size reproduction in this read-only review.

## Solution Assessment

No. The best current solution is the shipped narrower persistence normalization; this PR's global key-name replacer is not the narrowest maintainable fix because it drops useful metadata and unrelated nested data with the same key name.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P2] Preserve lightweight skill snapshot metadata:** `src/config/sessions/store-serialization.ts:10`
  - body: The replacer removes every skillsSnapshot from session-store serialization. Current main preserves prompt, skills, skillFilter, and version and uses version/filter on resume, so this would lose durable metadata after the next save/restart.
  - confidence: 0.93

  - confidence: 0.9
