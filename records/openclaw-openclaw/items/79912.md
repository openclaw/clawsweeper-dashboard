## Summary

Keep open: the PR targets a real Active Memory status bug, but the proposed formatter maps every internal `empty` result to `none`, including unavailable memory-search and generic embedded-run failure paths that should not look like a benign no-memory result.

## What This Changes

The PR adds a display formatter so Active Memory status lines render internal `empty` recall results as `status=none` and updates active-memory status tests accordingly.

## Best Possible Solution

Split benign no-memory recall from unavailable or failed recall before formatting, then render only the benign `NONE` or zero-hit outcome as `none` while preserving visible failure-oriented statuses for broken memory paths.

## Reproduction Assessment

Yes. Current main source and tests show a normal `NONE` recall becomes internal `status: "empty"` and is printed directly in the user-visible Active Memory status line; I did not run tests under the read-only review constraint.

## Solution Assessment

No. The display-only mapping is too broad because it also turns unavailable memory-search and generic embedded-run failures into `status=none`; the safer fix is to classify result reasons first and map only benign no-memory outcomes.
