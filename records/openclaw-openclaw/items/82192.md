## Summary

Keep open: the performance direction is useful and the PR supplies real benchmark output, but the diff makes public completion cache writes and doctor cache repair skip plugin command registration, which would produce core-only cached completions for installed shells. The linked regression report is also broader than this completion path and still includes health/gateway cold-start evidence.

## What This Changes

The PR skips plugin CLI registration during completion cache writes and doctor completion cache generation, adds focused tests, and adds a startup benchmark case.

## Best Possible Solution

Keep explicit/manual completion cache rebuilds plugin-aware, keep updater/package smoke refreshes on the existing env-scoped fast path, and only make doctor core-only if maintainers intentionally document that tradeoff.

## Reproduction Assessment

Source-reproducible: I did not run the benchmark, but current main shows `completion --write-state` loads plugin registration unless the skip env is set, and the PR diff prevents it for all write-state paths. The linked report supplies real timing output for the slow completion path.

## Solution Assessment

No. Skipping plugin startup is the right optimization boundary for updater/package smoke refreshes, but making public and doctor cache writes core-only is not the narrowest maintainable fix because installed completion relies on the cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep plugins in explicit completion cache writes:** `src/cli/completion-cli.ts:108`
  - body: This skips plugin command registration whenever `--write-state` is present. That is the user/manual cache path used by installed shell completion; current docs say completion generation eagerly loads command trees, and update already has a separate env-only fast path that leaves full plugin-command rebuilds to explicit `openclaw completion --write-state`. After this change, regenerating the cache after installing a plugin writes a core-only completion file, so plugin commands disappear from installed shell completion.
  - confidence: 0.93
- **[P2] Avoid writing core-only caches from doctor:** `src/commands/doctor-completion.ts:36-38`
  - body: Doctor now forces `OPENCLAW_COMPLETION_SKIP_PLUGIN_COMMANDS=1` while regenerating or installing a missing completion cache. That repair path is documented as a normal cache regeneration, so the sourced cache would omit plugin commands even after doctor reports success. Keep this skip scoped to updater/package smoke paths, or make the reduced cache behavior an explicit maintainer-approved contract with matching docs/tests.
  - confidence: 0.84
