## Summary

Keep open: this is a focused sessions bug-fix PR, but it still lacks after-fix real ACP proof and the overlay should not hard-code the ACP backend when persisted metadata carries a backend id.

## What This Changes

The PR overlays ACP control-plane session rows in `openclaw sessions` so model fields show an ACP sentinel, adds regression coverage, and records a changelog entry.

## Best Possible Solution

Keep the call-site overlay, but carry the persisted ACP backend through the row and require redacted after-fix ACP output before merge.

## Reproduction Assessment

Yes, at source level: current main emits `resolveSessionDisplayModelRef` directly and that resolver falls back to the configured agent model when stored model fields are absent. The contributor also supplied before-fix deployed-container output, but this review did not live-run ACP.

## Solution Assessment

No, not yet: the overlay is the right narrow shape, but it should use the persisted ACP backend instead of always returning `acpx`. Merge should also wait for after-fix real behavior proof from the contributor's setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Use the stored ACP backend in the overlay:** `src/commands/sessions.ts:87`
  - body: Control-plane ACP metadata already records the backend that created the session, and docs allow ACP backend overrides. Returning a literal `acpx` means sessions from any other ACP backend will still be listed as acpx, which is the same kind of misleading runtime display this PR is fixing. Thread the stored backend into the overlay, falling back to acpx only if absent, and add a custom-backend regression case.
  - confidence: 0.84
