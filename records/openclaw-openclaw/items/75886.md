## Summary

Current main still has the reported failure path, so this issue should stay open and is a good narrow fix candidate.

## What This Changes

Harden cron scheduler/store reloads so malformed persisted job schedules do not abort scheduler ticks or block unrelated due jobs.

## Best Possible Solution

Make cron store reload schedule comparison non-throwing and isolate malformed persisted jobs with a clear warning or repair path, while preserving due jobs and existing schedule-change invalidation semantics.

## Reproduction Assessment

Yes. A malformed persisted job can be loaded once, then a later force reload compares it through cronSchedulingInputsEqual, which calls the throwing cronScheduleIdentity before onTimer collects due jobs.

## Solution Assessment

No, the current implementation is not the best way to handle this. The narrow maintainable fix is to use a non-throwing schedule identity for reload comparison and add regression coverage; string-schedule normalization can be added as compatibility but should not be the only defense.

## Repair Work Prompt

Fix openclaw/openclaw#75886. Current main can abort a cron scheduler tick when a malformed persisted job has a string-shaped `schedule` field: `ensureLoaded(... forceReload ...)` calls `invalidateStaleNextRunOnScheduleChange`, which calls `cronSchedulingInputsEqual`, which calls the throwing `cronScheduleIdentity`. Make reload schedule comparison non-throwing so one malformed persisted job cannot prevent unrelated due jobs from running. Prefer using `tryCronScheduleIdentity` in `cronSchedulingInputsEqual` or otherwise treating unidentifiable schedules as changed without throwing; add per-job tolerance/logging in `ensureLoaded` if needed. Consider compatibility normalization for legacy string schedules, but do not rely on that alone as the safety boundary. Add regression coverage for a disabled malformed row with `schedule: "0 17 * * *"` and no id or missing id-like fields alongside a valid due job: repeated force reload/timer execution must not throw, and the valid due job must still run or remain selectable. Preserve existing behavior that formatting/key-order-only schedule rewrites do not clear `nextRunAtMs`, and that real schedule/enabled changes do clear stale pending slots. Add a user-facing changelog entry. Do not refactor unrelated cron execution, delivery, provider, or docs surfaces.
