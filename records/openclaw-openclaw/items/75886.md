## Summary

Current main still has the reported scheduler-abort path, and the linked open PR #75896 uses closing syntax for this issue, so this should stay open until that PR merges or is superseded.

## What This Changes

Make cron scheduler reloads resilient to malformed persisted job entries so one bad schedule cannot abort the whole scheduler tick.

## Best Possible Solution

Land or supersede the linked fix PR with a narrow non-throwing reload comparison and regression coverage that preserves due-job processing when one persisted row is malformed.

## Reproduction Assessment

Yes. A string-shaped persisted schedule can be loaded, then a later force reload reaches ensureLoaded -> invalidateStaleNextRunOnScheduleChange -> cronSchedulingInputsEqual -> cronScheduleIdentity before due jobs are collected, and that helper throws on the malformed schedule.

## Solution Assessment

No for current main. The maintainable fix is to make reload schedule comparison non-throwing and cover this persisted malformed shape; the open PR is the natural candidate to review for that behavior.
