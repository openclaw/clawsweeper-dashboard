## Summary

Keep open: the report describes broken existing heartbeat scheduling after a SIGUSR1 gateway restart, and current main does not clearly supersede the reported release because the central heartbeat scheduler/wake blobs are identical to v2026.5.4. Existing SIGUSR1 coverage protects a related in-process wake-handler race, but it does not prove the macOS LaunchAgent/full-process gap reported here is fixed.

## What This Changes

Heartbeat polling should continue its configured 30-minute cadence after a SIGUSR1 gateway restart instead of silently stopping while the gateway and cron remain alive.

## Best Possible Solution

Add a regression-backed runtime fix that proves heartbeat timers resume after SIGUSR1 supervised and in-process restarts, with diagnostics when heartbeat is blocked by a stuck run, cron lane, or active-job state.
