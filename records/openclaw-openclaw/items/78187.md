## Label Justifications

- `P2`: The original report describes a normal-priority reliability bug where heartbeat-driven checks stopped for hours while the gateway remained otherwise usable.
- `impact:message-loss`: The failure suppressed heartbeat-driven checks, reminders, and notification polling.
- `impact:session-state`: The failure mode involves heartbeat scheduler and wake-handler state across gateway restart/session lifecycle boundaries.

## Summary

Close: current main and shipped releases now cover the central silent heartbeat-stop failure. Heartbeat wake state is reset on new lifecycle registration, heartbeat turns no longer inherit the long interactive timeout, and the related ack-only pending-delivery loop is also fixed.

## What This Changes

Heartbeat polling should resume after a SIGUSR1 gateway restart instead of silently stopping while cron continues.

## Best Possible Solution

Close this issue and rely on the shipped heartbeat scheduler fixes in v2026.6.1 and newer; reopen or file a new report only if the same SIGUSR1 sequence reproduces on a current release with logs.

## Reproduction Assessment

No current-main failing reproduction remains: source and history show the stale lifecycle, long-running heartbeat, and related pending-ack loop paths have been fixed. The original older-version behavior was concrete enough to trace through the merged scheduler fixes.

## Solution Assessment

Yes. The current solution fixes the scheduler and heartbeat runtime boundaries rather than adding a SIGUSR1-specific workaround: lifecycle state resets, heartbeat turns are bounded, and ack-only pending state no longer refreshes the in-flight skip loop.
