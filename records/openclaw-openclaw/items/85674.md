## Label Justifications

- `P3`: This is a speculative prompt-cost and ergonomics feature, not a regression or broken current workflow.
- `impact:session-state`: The requested activation would persist per session and change which skill context is injected into the agent prompt.

## Summary

Close as ClawHub/plugin-scope: the requested Claude-style `paths:` conditional skill loading is useful, but current OpenClaw policy and plugin hooks route optional runtime skill behavior outside core, and a near-identical request was already closed on that basis.

## What This Changes

Add an optional `paths:` field to `SKILL.md` frontmatter so skills stay out of the prompt until matching file paths are touched in the session.

## Best Possible Solution

Keep core `SKILL.md` metadata unchanged; implement path-triggered skill prompt behavior as a ClawHub plugin/bundle using `before_tool_call` and `before_prompt_build`, and raise a narrow plugin-API gap on the broader prompt-overhead tracker only if that path blocks.

## Reproduction Assessment

Not applicable: this is a request for a new skill-loading mode, not a broken current behavior. Source inspection shows current visibility is controlled by eligibility, allowlists, prompt budgeting, and `disable-model-invocation`, not path-triggered activation.

## Solution Assessment

Yes for cleanup: ClawHub/plugin routing is the best current outcome because a near-identical core `paths:` schema request was already closed that way and current plugin hooks cover the proposed runtime behavior. A core implementation would require reversing that product/API decision.
