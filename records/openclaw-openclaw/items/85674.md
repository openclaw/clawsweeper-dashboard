## Summary

Keep open: current main and the latest shipped release have static eligibility, visibility, and prompt-budget controls for skills, but no path-triggered conditional promotion; this is a new public skill/session contract needing maintainer product and API judgment, not conservative cleanup.

## What This Changes

Request adds optional `paths:` SKILL.md frontmatter so matching skills stay out of `<available_skills>` until a touched file path matches, plus conditional/active CLI visibility.

## Best Possible Solution

Define a first-class conditional-skill contract in skill metadata and prompt snapshot assembly, covering parsing, activation events, session persistence, CLI/status output, docs, and tests while preserving existing allowlist and `disable-model-invocation` behavior.

## Reproduction Assessment

Not applicable: this is a new skill-loading mode request rather than a broken current behavior. Source and release inspection show current behavior is controlled by eligibility, allowlists, prompt budgets, and `disable-model-invocation`, not file-path-triggered activation.

## Solution Assessment

Unclear: the path-conditioned activation direction is plausible, but maintainers should choose the public SKILL.md field, activation events, persistence, CLI/status, and prompt-cache semantics before implementation. Coordinating it with the open runtime discovery, triggers, and tool/toolset activation threads is safer than landing a one-off contract.
