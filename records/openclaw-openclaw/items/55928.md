## Summary

Keep open. Current main still serves authenticated HTTP session-history reads without an internal-session guard, and the PR is a focused, clean fix for that exposed HTTP surface; the only actionable merge blocker I found is the missing required user-facing changelog entry.

## What This Changes

The PR adds an internal gateway session-key predicate, returns 403 for HTTP session-history reads targeting raw or canonical subagent, cron, or ACP keys, and adds regression coverage while preserving WebSocket `chat.history`.

## Best Possible Solution

Land this HTTP-only access-control guard after adding a single Unreleased changelog entry, keeping the WebSocket runtime path available until a separate caller-trust model exists for internal WS flows.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed `agent:main:cron:daily` with transcript data, then request `/sessions/cron%3Adaily/history` with `operator.read` on current main and observe that the handler canonicalizes and reads the transcript instead of returning 403.

## Solution Assessment

Yes, aside from the changelog omission. Checking both the raw request key and resolved canonical key before transcript/SSE reads is the narrowest maintainable HTTP fix, and preserving WebSocket history avoids breaking existing subagent-control runtime flows.
