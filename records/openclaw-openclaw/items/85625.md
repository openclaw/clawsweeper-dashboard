## Summary

Keep open: the linked bug is credible and the fix direction is useful, but this PR is not merge-ready because it skips cumulative `result` usage before teaching the parser to read the real assistant-message usage shape. Official Claude docs describe final result usage as cumulative and assistant usage as message-level data, while current `main` only reads top-level `usage`/`stats`. ([docs.anthropic.com](https://docs.anthropic.com/en/docs/claude-code/sdk/sdk-cost-tracking))

## What This Changes

The PR changes the CLI JSONL streaming parser to ignore usage on `result` events and adds a regression test expecting assistant-event usage to be retained.

## Best Possible Solution

Land the accounting fix only after the parser reads documented assistant-message usage, ignores cumulative result usage for session snapshots, and is proven with a real redacted Claude stream-json run.

## Reproduction Assessment

Yes for the review finding: current source only reads top-level usage fields, while the Claude SDK schema places assistant usage under the nested message object. I did not run live Claude CLI because this review is read-only.

## Solution Assessment

No. Ignoring cumulative result usage is the right direction, but the patch should first parse the real assistant-message usage shape and test that shape rather than a top-level assistant fixture.
