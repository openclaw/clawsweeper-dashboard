## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require a selected Token Plan region before catalog activation:** `extensions/xiaomi/index.ts:97-99`
  - body: `requireConfiguredProvider` currently accepts any configured object, but this PR also allows a `xiaomi-token-plan` timeout-only overlay. With `XIAOMI_TOKEN_PLAN_API_KEY` or an auth profile, that path returns the manifest provider with the Singapore base URL instead of requiring the user-selected regional URL, so Token Plan traffic can silently route to the wrong region. Gate catalog resolution on a non-empty regional `baseUrl` or persist the selected region in auth/profile state.
  - confidence: 0.9
- **[P2] Reject keys without the expected Xiaomi prefix:** `extensions/xiaomi/index.ts:126-134`
  - body: `buildXiaomiKeyMismatchMessage` only errors when the key starts with the opposite prefix, so an arbitrary value like `bad-key` can pass Token Plan onboarding and persist the provider. Since the new onboarding/docs promise `tp-...` versus `sk-...` validation, require the expected prefix for each path or change the documented contract.
  - confidence: 0.86
- **[P3] Align the Token Plan config example costs:** `docs/providers/xiaomi.md:201-210`
  - body: The manual Token Plan config example still sets both model costs to zero, while the manifest and PR body define paid tiered cache-read pricing. Users copying this example would disable cost accounting for the new provider, so either mirror the manifest costs or omit the cost block from the example.
  - confidence: 0.78
