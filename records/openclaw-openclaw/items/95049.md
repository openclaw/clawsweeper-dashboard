## Summary

Keep open: the PR still routes hook-level model overrides through the default provider as an explicit provider/model pair, so provider-qualified refs and aliases can be misrouted, and the PR body still does not include real after-fix hook-run proof.

## What This Changes

## Best Possible Solution

Resolve the hook override through the alias-aware model-selection path, document or explicitly narrow the config contract, add focused regression coverage, and require redacted real hook-run proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main reads `hookConfig` and gates `llmSlug`, but the handler does not forward `hookConfig.model` and the generator resolves only the agent default model. I did not run a live hook path in this read-only review.

## Solution Assessment

No. Forwarding the value is the right area, but keeping the default provider while overriding only the model bypasses alias/provider-qualified resolution and is not the best OpenClaw-shaped fix.
