## Summary

Keep open: the linked source bug is real, but this PR is still not merge-ready because the updated model override path can keep routing provider-qualified or alias values through the default provider, and the PR still lacks after-fix real hook-run proof.

## What This Changes

The PR forwards the session-memory hook model string into LLM slug generation and uses it as the embedded slug agent model name while keeping the default provider.

## Best Possible Solution

Resolve the hook model through the existing alias-aware model-selection path, add focused handler/generator regression tests for provider-qualified and alias values, and require redacted real hook-run proof before merge.

## Reproduction Assessment

Yes from source inspection. Current `main` reads `hookConfig` and gates `llmSlug`, but calls `generateSlugViaLLM({ sessionContent, cfg })`, while the generator resolves only the agent default model.

## Solution Assessment

No. Forwarding the hook value is the right owner boundary, but keeping the default provider while replacing only the model is not the best fix because `runEmbeddedAgent` skips alias/provider resolution when both provider and model are explicit.
