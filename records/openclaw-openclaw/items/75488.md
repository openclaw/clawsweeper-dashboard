## Summary

Keep open. Current main still classifies BlueBubbles attachment placeholders by MIME only, while this PR makes the narrow UTI-aware audio classification in the existing placeholder path, adds focused tests and a changelog entry, and has no blocking correctness or security finding. The `maintainer` label also means this should proceed through explicit maintainer review rather than cleanup closure.

## What This Changes

The PR adds a BlueBubbles audio-attachment predicate for `audio/*` MIME types plus a narrow set of Apple audio UTIs, uses it for inbound `<media:audio>` placeholders, adds monitor-normalize tests, and updates the changelog.

## Best Possible Solution

Land a narrow BlueBubbles-owned fix that keeps the existing placeholder contract, recognizes the confirmed UTI-only audio cases before agent dispatch, and preserves the broader transcript-enrichment work as closed or separately justified rather than adding speculative API calls or config.

## Reproduction Assessment

Yes. On current main, a BlueBubbles attachment with `uti: "public.mpeg-4-audio"` and no `mimeType` reaches a MIME-only `allAudio` check and therefore cannot produce `<media:audio>`; the PR adds direct unit coverage for that path.

## Solution Assessment

Yes. The proposed change is the narrowest maintainable fix: it changes only the BlueBubbles placeholder classification helper, uses fields already present in the normalized attachment type, keeps mixed/non-audio attachments generic, and avoids new config, network calls, SDK seams, or dependencies.
