## Summary

Keep open: current main and the latest release still carry Discord thread parent metadata and manual pin-reading actions, but neither automatically injects parent-channel pinned messages into thread session context. The remaining request is a real Discord session-context feature that needs maintainer policy on defaults, context visibility, and whether a new config surface is acceptable.

## What This Changes

The request asks for Discord thread sessions to automatically receive parent-channel pinned-message context or metadata when the session starts.

## Best Possible Solution

Choose the Discord parent-pin inheritance policy first, then implement it in the Discord plugin with existing read-target/context-visibility gates, focused tests, and docs for default versus opt-in behavior.

## Reproduction Assessment

Yes for source-level behavior, but not as a failing bug repro. Current main clearly carries Discord thread parent metadata and supports manual pin reads, while the inbound thread context path does not fetch or inject parent-channel pin content.

## Solution Assessment

Unclear until maintainers choose the product policy. Automatic injection is plausible, but the safest maintainable solution depends on default behavior, opt-in config, and supplemental-context filtering decisions.
