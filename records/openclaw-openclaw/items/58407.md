## Summary

Keep this open: current main supports Discord parent-thread routing, thread starter context, parent/model session inheritance, channel topic context, and manual pin listing, but it does not automatically fetch parent-channel pins for a thread-bound session. The requested behavior is a new Discord context/config policy choice rather than a broken existing contract.

## What This Changes

Request automatic injection or metadata exposure of a Discord thread's parent-channel pinned messages when a thread-bound session starts.

## Best Possible Solution

Add a maintainer-approved Discord thread context feature that fetches parent-channel pins, wraps them as untrusted supplemental context or explicit metadata, documents the config/default, and keeps the existing manual `list-pins` action available.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. Source inspection gives a high-confidence current-main path: thread sessions carry parent metadata and starter/session context, while pin fetching only exists as a manual message action.

## Solution Assessment

Unclear as proposed. Raw system-context injection is probably not the safest default; a config-gated untrusted-context or metadata design would be the maintainable direction if maintainers want this behavior.
