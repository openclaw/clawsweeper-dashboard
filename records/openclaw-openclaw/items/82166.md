## Summary

Keep open: the linked browser bug is real and this PR targets the right code path, but the current head still has a blocking correctness issue. Its default-port preservation path relies on WHATWG URL serialization, which drops explicit `:80`/`:443`, so the new explicit-default-port expectation is not actually satisfied.

## What This Changes

The PR changes browser CDP URL parsing and profile resolution so explicit URL ports win, portless URLs preserve configured `cdpPort`, and unit tests cover port-precedence cases.

## Best Possible Solution

Keep the proposed precedence rule, but build port-preserving normalized URLs without relying on WHATWG URL serialization for default ports, with regression coverage for portless, explicit default-port, IPv6, and userinfo URLs.

## Reproduction Assessment

Yes. Current main is source-reproducible through `resolveProfile` and `parseBrowserHttpUrl`, and PR head can be probed directly to show default ports cannot be preserved by assigning `URL.port`.

## Solution Assessment

No, not yet. The intended fix direction is narrow and maintainable, but the current `normalizedWithPort` implementation needs to preserve explicit default ports without WHATWG reserialization dropping them.
