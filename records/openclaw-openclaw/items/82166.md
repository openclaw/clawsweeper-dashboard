## Summary

Keep open: this PR targets a valid browser config bug and includes real before/after proof, but the current patch introduces a CDP URL preservation regression by rebuilding `cdpUrl` from only protocol, host, and port.

## What This Changes

The PR changes browser profile resolution so a configured `cdpPort` is preserved when `cdpUrl` has no explicit port, with a focused regression test.

## Best Possible Solution

Revise `resolveProfile` to preserve all URL components while only replacing the derived default port when the URL omits a port and profile `cdpPort` is set, with tests for both the bare-loopback case and token/path-preserving remote URLs.

## Reproduction Assessment

Yes. Current main source shows a profile `cdpUrl` without an explicit port is parsed to the protocol default and assigned over the configured `cdpPort`, and the linked report provides the matching Chrome port-80 failure.

## Solution Assessment

No. The intended port-precedence fix is the right boundary, but rebuilding the URL from only protocol/host/port is not; the safer solution is to preserve `parsed.normalized` or mutate a URL object only for the missing-port case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve URL components when setting the CDP port:** `extensions/browser/src/browser/config.ts:514`
  - body: The new assignment rebuilds `cdpUrl` as only `protocol//host:port`, so any configured path, query string, username, or password is dropped. This module already supports tokenized and direct CDP URLs, so a profile such as `wss://connect.example.com?apiKey=abc` or `http://host:9222/path?token=...` would lose the data needed to connect. Preserve the parsed URL and only change the port when the raw URL omitted one and `cdpPort` was configured.
  - confidence: 0.93
