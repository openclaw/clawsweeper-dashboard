## Summary

Keep this PR open, but it still has a blocking correctness issue. The latest patch fixes the earlier default-port normalization concern, but its raw regex misclassifies supported portless URL forms such as bracketed IPv6 loopback and numeric user-info credentials as explicit ports, so the same `cdpPort` overwrite can still happen.

## What This Changes

The PR changes browser profile resolution and tests so a portless `profiles.*.cdpUrl` preserves an explicit `cdpPort` while explicit URL ports continue to win.

## Best Possible Solution

Keep the narrow precedence fix, but detect explicit ports from the URL authority in a way that ignores userinfo and bracketed IPv6 host syntax.

## Reproduction Assessment

Yes. Current main is source-reproducible for the linked bug, and the remaining PR bug is reproduced by probing the new explicit-port regex against supported portless URL forms.

## Solution Assessment

No, not as currently implemented. The intended precedence rule is right, but the raw regex should be replaced with authority-aware explicit-port detection and regression coverage for IPv6 and userinfo cases.
