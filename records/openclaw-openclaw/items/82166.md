## Summary

Keep open: this is a useful narrow browser bug fix, but the current branch still has a blocking parser bug in its explicit-port detection for supported URL forms with bracketed IPv6 hosts or numeric userinfo.

## What This Changes

The PR changes browser profile resolution and tests so a portless profile `cdpUrl` preserves an explicit `cdpPort` while explicit URL ports continue to win.

## Best Possible Solution

Keep the narrow precedence fix, but replace the raw regex with authority-aware explicit-port detection and add regression coverage for bracketed IPv6 and userinfo cases.

## Reproduction Assessment

Yes. Current main is source-reproducible from `resolveProfile` through `parseBrowserHttpUrl`, and a focused Node probe reproduces the PR branch's remaining false positive on `http://[::1]` and numeric userinfo URLs.

## Solution Assessment

No, not as currently implemented. The intended precedence rule is right, but the branch should parse the URL authority instead of scanning the raw URL text with a broad colon regex.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Parse explicit ports from the URL authority:** `extensions/browser/src/browser/config.ts:510`

In those cases `resolveProfile` still overwrites the configured `cdpPort` with the parser default 80/443 for supported URL forms; parse the authority while ignoring userinfo and bracketed IPv6 host colons.
