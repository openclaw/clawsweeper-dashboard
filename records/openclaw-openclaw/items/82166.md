## Summary

Keep open: the PR addresses a real browser profile bug with sufficient real-behavior proof, but the current head has a blocking edge-case regression in explicit default-port handling.

## What This Changes

The PR changes browser profile resolution to preserve an explicit `cdpPort` when a profile `cdpUrl` omits a port, normalizes the URL with that port, and adds a focused regression test.

## Best Possible Solution

Land a corrected browser config fix that preserves `cdpPort` only when the raw `cdpUrl` truly omits a port, while explicit URL ports including `:80` and `:443` still win, with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current `main` follows `cdpPort: 18800` plus `cdpUrl: "http://127.0.0.1"` through `resolveProfile` to parser default port 80, and the linked report includes the matching Chrome bind failure.

## Solution Assessment

No. The precedence fix belongs in `resolveProfile`, but detecting explicit ports via `URL.port` is not sufficient because default ports are normalized away; the parser or resolver should retain raw explicit-port metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Detect explicit default ports before URL normalization:** `extensions/browser/src/browser/config.ts:509`
  - body: `URL.port` is empty not only when the URL omits a port, but also when the user explicitly writes the protocol default, such as `http://host:80` or `https://host:443`. With this branch, `cdpUrl: "http://host:80"` plus `cdpPort: 18800` now resolves to 18800 instead of honoring the explicit URL port, which contradicts the stated URL-port precedence and the PR's escape hatch for users who really want port 80. Please detect explicit ports from the raw URL, or have `parseBrowserHttpUrl` expose that fact, and add coverage for explicit default ports.
  - confidence: 0.92
