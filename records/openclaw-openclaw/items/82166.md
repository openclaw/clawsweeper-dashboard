## Summary

Keep open: the intended browser CDP precedence fix is useful, but the current PR head still has a blocking parser bug because `normalizedWithPort` relies on WHATWG URL serialization, which drops explicitly written default ports and makes the new `:80` regression expectation fail.

## What This Changes

The PR changes browser CDP URL parsing, profile resolution, and regression tests so explicit URL ports should win over `cdpPort`, portless `cdpUrl` values preserve configured `cdpPort`, and protocol defaults remain when no port is configured.

## Best Possible Solution

Keep the proposed precedence rule, but build the port-preserving normalized URL without WHATWG reserialization dropping default ports, with focused coverage for portless URLs, explicit default ports, IPv6, and userinfo.

## Reproduction Assessment

Yes. The current-main source path shows `resolveProfile` overwriting a configured `cdpPort` with the parser's default 80 for a portless HTTP URL, matching the linked Chrome startup failure.

## Solution Assessment

No, not yet. The precedence direction is the right narrow fix, but the current parser helper still serializes explicit default ports through WHATWG URL and loses them before merge.
