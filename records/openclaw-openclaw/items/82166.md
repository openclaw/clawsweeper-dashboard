## Summary

Keep open: the linked browser startup bug is valid and the PR targets the right code path, but the patch rebuilds `cdpUrl` from only protocol, host, and port, which would drop existing path/query/auth details for documented remote CDP URLs such as Browserless and Browserbase token URLs.

## What This Changes

The PR changes browser profile resolution so a profile-level `cdpPort` is preserved when `cdpUrl` omits an explicit port, and adds a focused config test for that case.

## Best Possible Solution

Preserve the intended `cdpPort` precedence while keeping the parsed CDP URL's path, query, and auth components intact, and extend regression coverage for both the reported loopback case and tokenized no-port remote CDP URLs.

## Reproduction Assessment

Yes. The original issue is source-reproducible through `resolveProfile` and `parseBrowserHttpUrl`, and the PR-introduced regression is source-reproducible against the existing no-port WSS query-token config test shape.

## Solution Assessment

No. The proposed precedence rule is the right narrow direction, but reconstructing the URL string is not safe; a better fix should preserve `parsed.normalized` or clone the URL and adjust only the port when needed.

## Review Findings

Overall correctness: patch is incorrect
