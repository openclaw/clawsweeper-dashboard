## Summary

Keep open: the PR fixes the reported IPv4 loopback case and supplies convincing live proof, but the latest diff still misdetects explicit ports by scanning raw URL text, so portless bracketed IPv6 and userinfo URLs can still overwrite a configured cdpPort with 80/443.

## What This Changes

The PR updates browser profile resolution and unit tests so a portless profile cdpUrl preserves an explicit cdpPort while explicit URL ports continue to win.

## Best Possible Solution

Keep the cdpPort precedence rule, but replace the raw regex with authority-aware explicit-port detection and add regression coverage for portless IPv6 and userinfo URLs plus explicit default ports.

## Reproduction Assessment

Yes, at source/probe level. Current main still overwrites the explicit profile cdpPort through parseBrowserHttpUrl, and the PR's new regex can be probed directly to show portless IPv6 and userinfo URLs are misclassified.

## Solution Assessment

No, not as currently implemented. The intended precedence rule is the right direction, but explicit-port detection should parse the URL authority while ignoring userinfo and bracketed IPv6 host colons.

## Repair Work Prompt

Repair the browser profile URL explicit-port detection in https://github.com/openclaw/openclaw/pull/82166. Replace the raw `urlHasExplicitPort` regex in `extensions/browser/src/browser/config.ts` with authority-aware detection that treats a port as explicit only when it appears after the host portion, while ignoring userinfo colons and bracketed IPv6 host colons. Preserve these behaviors: portless `cdpUrl` plus explicit `cdpPort` keeps the configured port, explicit URL ports including `:80` and `:443` win over `cdpPort`, and remote portless CDP URLs with no configured `cdpPort` keep protocol defaults. Add regression tests in `extensions/browser/src/browser/config.test.ts` for `http://[::1]` plus `cdpPort`, `http://user:123@127.0.0.1` plus `cdpPort`, the existing IPv4 portless case, and explicit default-port cases. Do not change Chrome launch behavior or remote CDP policy. No contributor changelog request is needed; maintainer landing handles user-facing changelog credit if this lands.
