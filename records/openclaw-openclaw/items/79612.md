## Summary

Keep open: current main still has the reported split between diagnostic active-work state and reply-run state, and an open closing PR already owns the focused fix path.

## What This Changes

The issue reports false `queued_work_without_active_run` gateway diagnostics when the detector misses active reply work that recovery later sees for the same session key.

## Best Possible Solution

Review and land the focused reply-run diagnostic lifecycle fix, or an equivalent narrow fix, then close this issue after it reaches main.

## Reproduction Assessment

Yes, at source level: current main can have an active reply operation in `replyRunState` while the diagnostic activity snapshot remains empty, which makes the detector emit `queued_work_without_active_run` before recovery separately sees active reply work. I did not establish a live channel reproduction in this read-only pass.

## Solution Assessment

Yes, the open linked PR's approach is the narrow maintainable fix because it mirrors reply-operation lifecycle into the existing diagnostic active-work tracker without adding channel-specific policy or disabling real stale-lane recovery.
