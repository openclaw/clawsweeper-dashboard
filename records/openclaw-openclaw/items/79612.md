## Summary

Close: current main now mirrors reply operations into diagnostic activity, so queued sessions with active reply work are no longer classified as `queued_work_without_active_run`. I could not prove this has shipped in v2026.5.12, so this is a current-main-only fix.

## What This Changes

The issue asks the gateway stuck-session detector to stop emitting `queued_work_without_active_run` when an active reply run exists for the same session key.

## Best Possible Solution

Keep the current-main reply-run diagnostic ownership model, and handle the narrower session-reset orphaned-reply scenario in https://github.com/openclaw/openclaw/pull/80427 if that follow-up lands.

## Reproduction Assessment

Yes for the pre-fix behavior: the issue logs and earlier source review show diagnostics could miss active reply work while recovery saw it. On current main, that failing source path no longer reproduces because reply operations populate diagnostic active work when created, rotated, and cleared.

## Solution Assessment

Yes. The current-main implementation is the narrow maintainable fix because it makes the detector and recovery share reply-run active-work facts without disabling real stale-lane recovery.
