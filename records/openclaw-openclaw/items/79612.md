## Summary

Keep open: current main still has split diagnostic and recovery views of active reply work, so the reported false `queued_work_without_active_run` warning is a valid narrow gateway diagnostics bug.

## What This Changes

The issue reports repeated false `queued_work_without_active_run` diagnostics when the same gateway session has active reply work.

## Best Possible Solution

Make the stuck-session detector use the same non-mutating active-work facts as recovery before emitting `session.stuck`, with regression coverage for active reply work and unchanged coverage for truly stale queued lanes.

## Reproduction Assessment

Yes, at source level: current main can classify a queued processing session as `queued_work_without_active_run` from diagnostic activity while recovery separately sees `replyRunRegistry` active work and keeps the lane. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No current implementation is not the best solution because the warning is emitted before the same active-work check that recovery uses to skip. The narrow fix is to share or preflight the active-work snapshot without adding Telegram-specific behavior or disabling real stale-lane recovery.
