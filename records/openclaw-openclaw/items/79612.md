## What This Changes

The issue asks the gateway stuck-session detector to stop emitting `queued_work_without_active_run` when active reply work exists for the same session key.

## Best Possible Solution

Keep the current shared diagnostic activity integration and leave broader performance or channel-latency work to their narrower tracking issues.

## Reproduction Assessment

Yes. The original 2026.5.7 logs plus pre-fix source path showed the detector reading diagnostic activity while recovery separately saw reply-run state; current main regression tests now model queued active work and assert no `session.stuck`.

## Solution Assessment

Yes. Mirroring reply-run operations into the existing diagnostic embedded-run activity tracker is the narrow maintainable fix because detector and recovery now share the same active-work facts without Telegram-specific suppression.
