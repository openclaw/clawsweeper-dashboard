## Summary

Keep open: this maintainer draft is useful TUI wiring for the harness prewarm stack, but the current diff can put gateway-mode TUI into a submit-blocking warmup state even when no prewarm actually exists.

## What This Changes

The PR wires the TUI to start an optional agent runtime prewarm during the first history load, shows a warming runtime busy state, blocks submit while that state is active, and adds focused unit coverage.

## Best Possible Solution

Gate the busy state on an actual prewarm promise, add no-prewarm backend coverage, and land this TUI wiring after the harness prewarm base is accepted.

## Reproduction Assessment

Yes for the review finding by source inspection: use a TUI backend without `prewarmAgentRuntime`, such as `GatewayChatClient`, and the PR still sets `warming runtime` on the first history load. Because that status blocks submit, a startup history retry can block the first send without any prewarm running.

## Solution Assessment

No, not yet. The TUI prewarm path is plausible, but the warmup status should be tied to an actual prewarm call so gateway/no-prewarm backends keep their existing first-send behavior.
