## Repair Work Prompt

Fix persistent provider auth cooldown recovery for https://github.com/openclaw/openclaw/issues/70903. Current main still has two gaps: billing-disabled primaries with fallback candidates are suspended far from expiry even when the fallback chain does not provide a real alternate provider/profile/billing path, and configure/plugin auth refresh via runProviderPluginAuthMethod writes fresh credentials without clearing stale usageStats for the affected provider/profile. Preserve persisted cooldown protection; do not drop disabledUntil globally, do not weaken auth_permanent handling, do not add a new config option or CLI reset, and avoid changing unrelated provider policy. Implement targeted billing recovery by probing on the existing throttle when the configured fallback chain has no real alternate auth/billing path, and add provider/profile-scoped stale cooldown cleanup after successful configure/plugin auth refresh using the existing clearAuthProfileCooldown/store helpers. Coordinate with https://github.com/openclaw/openclaw/pull/74909 if touching failure-policy code; current main already filters format from shared auth-profile health, so do not regress OpenAI Codex WHAM handling. Add focused regression coverage for the same-provider/same-profile billing fallback case and configure/plugin re-auth cleanup, plus a single-line changelog entry because this is user-visible recovery behavior.
