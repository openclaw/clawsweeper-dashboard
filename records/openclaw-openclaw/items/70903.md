## Summary

Keep open. Current main still only partially mitigates the report: single-provider billing cooldowns can be probed, but billing-disabled primaries with configured fallback candidates are still skipped until near expiry, and configure/plugin auth refresh can still write fresh credentials without clearing the matching stale usage entry.

## What This Changes

The issue requests faster recovery from persisted provider auth-profile cooldowns after billing recovery or successful provider re-authentication.

## Best Possible Solution

Keep persisted cooldown protection for unresolved provider-health failures, but add targeted recovery: allow throttled billing probes when the configured fallback chain does not provide a real alternate billing/auth path, and clear provider/profile-scoped stale usage state after successful configure/onboarding/plugin auth refresh while preserving unrelated provider state.

## Reproduction Assessment

Yes. The issue and follow-up comment give concrete Anthropic billing and GitHub Copilot re-auth scenarios, and current source/tests still encode the remaining behaviors without needing live provider credentials.

## Solution Assessment

Yes. The narrow maintainable fix is targeted recovery rather than dropping persisted cooldowns wholesale: recover user-fixed states promptly while keeping cooldowns for unresolved failures and preserving cross-provider fallback preference.
