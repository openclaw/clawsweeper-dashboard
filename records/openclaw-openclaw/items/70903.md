## Summary

Keep open. Current main still has only partial mitigation: billing-disabled profiles can remain skipped when fallback candidates exist, and successful configure/plugin auth refresh still writes fresh credentials without clearing the matching stale usage state.

## What This Changes

The issue requests provider cooldown recovery after billing top-up or successful re-auth so users are not blocked by persisted auth-state lockouts.

## Best Possible Solution

Keep persisted cooldown protection, but add targeted recovery: throttled billing probes when the fallback chain does not provide a real alternate auth/billing path, and provider/profile-scoped usage-state cleanup after successful configure/onboarding auth refresh.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete Anthropic billing and GitHub Copilot re-auth scenarios, and current source/tests still encode the same remaining skip and cleanup gaps.

## Solution Assessment

Yes. The maintainable fix is targeted recovery rather than dropping persisted cooldowns wholesale, preserving protection for unresolved provider-health failures while clearing user-fixed states promptly.
