## Summary

Keep open. Current main has partial mitigation for direct `models auth login` and immutable disable windows, but the reported same-auth fallback-chain recovery and `openclaw configure` plugin-auth stale-cooldown cleanup gaps are still present in source.

## What This Changes

The issue requests that persisted provider auth cooldown state stop blocking users for hours after billing, token, or re-auth recovery.

## Best Possible Solution

Keep persisted cooldown protection, but add targeted recovery for same-auth billing-disabled chains and clear matching usage state after successful configure/plugin auth refresh.

## Reproduction Assessment

Yes, source-level. Current main still has a path where fallback candidates sharing a disabled auth profile suppress far-from-expiry billing probes, and configure/plugin auth can save fresh credentials without clearing stale usage state.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing probe throttle and `clearAuthProfileCooldown` helper, not to remove persisted cooldowns or add a new user-facing reset mode first.
