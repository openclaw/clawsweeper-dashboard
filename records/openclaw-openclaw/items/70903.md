## Summary

Keep open. Current main has partial mitigation for single-provider billing probes and direct `models auth login`, but the reported same-auth fallback chain and configure/plugin-auth recovery gaps are still present in source.

## What This Changes

The issue reports that persisted provider/auth-profile cooldown state can keep users blocked for hours after billing recovery or credential refresh.

## Best Possible Solution

Keep persisted cooldown protection, but add targeted recovery for billing-disabled chains with no real alternate auth path and clear matching usage state after successful configure/plugin auth refresh.

## Reproduction Assessment

Yes, source-level. Current main still has a clear path where two same-provider candidates sharing disabled auth profiles count as fallbacks and suspend rather than probing, and plugin configure auth upserts fresh credentials without clearing stale usage state.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing probe throttle and cleanup helpers, not to remove persisted cooldowns or add a new user-facing reset mode first.
