## What This Changes

The issue reports that an Android node shown by `openclaw nodes status` cannot be removed with `openclaw nodes remove --node <id>` because the gateway returns `unknown nodeId`.

## Best Possible Solution

Review and land, revise, or reject the open linked gateway fix so every node row returned by `node.list` has a matching authorized removal path that preserves unrelated device roles and tokens.

## Reproduction Assessment

Yes. The reporter's output plus current source show a node ID displayed by `nodes status` can be backed by device pairing while `node.pair.remove` only deletes the legacy node-pairing store; I did not run a live Android flow in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the credential semantics. The repair belongs in the gateway node/device pairing removal boundary, not as a CLI-only workaround, because the server listing and removal surfaces currently disagree about the backing store.
