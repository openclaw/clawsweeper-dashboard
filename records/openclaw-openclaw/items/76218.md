## Summary

Keep open. Current main still has no cron `promptCacheKey`/`cacheKey` field and OpenAI prompt-cache routing is still derived from the isolated cron run's fresh session id; the separate model-override diagnostic concern is already fail-closed on main.

## What This Changes

Allow cron `agentTurn` payloads to provide a stable prompt cache key independent of the per-run isolated session id.

## Best Possible Solution

Add a validated `promptCacheKey` cron `agentTurn` field, propagate it as a provider prompt-cache override through embedded OpenAI execution, and leave `sessionTarget` transcript isolation unchanged.

## Reproduction Assessment

Yes. Static tracing is high-confidence: isolated cron creates a fresh session id per run, passes that id to embedded execution, and OpenAI transports use the session id as `prompt_cache_key`; live OpenAI billing would only quantify the cost impact.

## Solution Assessment

Yes. An explicit prompt-cache key is the narrowest maintainable fix because it stabilizes provider cache routing without reusing cron transcripts or changing custom-session semantics.
