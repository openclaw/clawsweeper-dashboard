## Summary

Keep open. Current main still derives OpenAI `prompt_cache_key` from the isolated cron run session id, and there is no accepted cron payload or runner option that can supply a stable independent key.

## What This Changes

Allow scheduled cron agent-turn payloads to set a stable OpenAI prompt cache key decoupled from the per-run isolated session id.

## Best Possible Solution

Add a trimmed `promptCacheKey` field for cron `agentTurn` payloads, propagate it to OpenAI stream options as `promptCacheKey ?? sessionId`, document the behavior, and keep existing `cacheRetention` and compatibility gates.

## Reproduction Assessment

Yes. Code inspection plus existing cron tests reproduce the behavior: isolated cron generates a fresh session id per run, and the OpenAI transport uses that session id as the only prompt cache key source; a live billing run was not needed for this triage.

## Solution Assessment

Yes for the cron scope. A narrow explicit `promptCacheKey` field is more maintainable than using `sessionTarget: "session:<stable-id>"`, because it avoids sharing transcript history while preserving provider cache routing.
