## Summary

Keep open. Current main still ties OpenAI `prompt_cache_key` to the per-run cron session id, while isolated cron intentionally creates a fresh session id for each execution; the related model-override diagnostic concern is already addressed on main.

## What This Changes

Allow cron agent-turn payloads to provide a stable prompt cache key so isolated recurring OpenAI runs can reuse prompt cache without reusing transcript history.

## Best Possible Solution

Add a validated `promptCacheKey` cron agent-turn field, carry it through the isolated embedded-run stream options as an OpenAI prompt-cache override, and keep `sessionTarget` semantics unchanged so transcript isolation remains intact.

## Reproduction Assessment

Yes. Static tracing is high-confidence: an isolated cron run forces a fresh session id, that id is passed into the embedded run, and OpenAI transport uses it as `prompt_cache_key`; live OpenAI billing would only quantify the impact.

## Solution Assessment

Yes. An explicit cache key is narrower and more maintainable than reusing a stable cron session, because it preserves clean per-run transcripts while stabilizing only the provider cache routing key.

Add a `promptCacheKey` field (prefer camelCase in OpenClaw payloads) to cron `agentTurn` create/update schemas and runtime types, preserve it through normalization/storage, and pass it from isolated cron execution into the embedded agent run/stream options so OpenAI Responses and compatible Completions transports use it instead of the per-run `sessionId` when cache retention is not `none`.
