## Summary

Keep open. Current main still has no cron `promptCacheKey` payload field, isolated cron still creates a fresh session id per run, and OpenAI transport still derives `prompt_cache_key` from that session id; the separate cron model-override diagnostic concern is already addressed.

## What This Changes

Allow recurring cron agent-turn payloads to provide a stable OpenAI prompt-cache key without reusing the same transcript session.

## Best Possible Solution

Add a validated `promptCacheKey` cron agent-turn field, propagate it as an OpenAI prompt-cache override while preserving isolated transcript semantics, and document the cron-specific usage.

## Reproduction Assessment

Yes. Source tracing is high-confidence: isolated cron forces a fresh session id, passes it into the embedded run, and OpenAI transport uses that id as `prompt_cache_key`; live billing would only quantify the cost impact.

## Solution Assessment

Yes. A dedicated cache-key override is narrower than reusing `sessionTarget: "session:<stable-id>"` because it stabilizes provider cache routing without accumulating cron transcript history.
