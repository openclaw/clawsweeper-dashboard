## Summary

Keep open. Current main still derives OpenAI `prompt_cache_key` from isolated cron's fresh per-run session id and has no cron payload override, so this remains a focused feature repair rather than implemented cleanup.

## What This Changes

Allow cron `agentTurn` payloads to carry an explicit stable prompt cache key so recurring OpenAI runs can share cache routing without reusing transcripts.

## Best Possible Solution

Add a validated `promptCacheKey` cron `agentTurn` payload field, forward it through the embedded runner to OpenAI-compatible transport options, keep `cacheRetention` and provider compatibility gates intact, and document the cron pattern.

## Reproduction Assessment

Yes. Source tracing on current main shows isolated cron forces a new session id per run and OpenAI transports use that id as `prompt_cache_key`; live billing would only quantify the already-clear cache miss impact.

## Solution Assessment

Yes. An explicit cache-key override is the narrowest maintainable direction because it avoids conversation-history accumulation and avoids changing default cache routing for ordinary sessions.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/76218. Add a validated explicit `promptCacheKey` field to cron `agentTurn` payloads and normalize/store/patch it consistently. Propagate it through isolated cron execution and embedded runner stream options so OpenAI Responses and opted-in OpenAI-compatible completions use the override instead of the per-run `sessionId` when `cacheRetention` is not `none`; preserve existing sessionId behavior when no override is set. Do not reuse isolated cron transcripts, do not derive a global cache key from full prompt content, and do not weaken provider compatibility or OpenAI-Codex sanitization behavior. Update docs and add a user-facing changelog entry.
