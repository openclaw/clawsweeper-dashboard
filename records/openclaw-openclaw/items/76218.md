## Summary

Keep open. Current main still lacks a cron or agent-turn `promptCacheKey` payload field, isolated cron still creates a fresh session id per run, and OpenAI transports still derive `prompt_cache_key` from that session id; the optional invalid-model diagnostic has already been addressed.

## What This Changes

Allow cron `agentTurn` payloads to carry a stable prompt cache key independent of the isolated run session id.

## Best Possible Solution

Add an additive, validated `promptCacheKey` payload and runner option that only affects provider cache routing, preserves fresh isolated transcripts, respects `cacheRetention` and provider compatibility, and documents cron usage limits.

## Reproduction Assessment

Yes. Source tracing is high-confidence: cron rejects or drops any cache-key field, isolated jobs force a fresh session id, and OpenAI request construction uses that session id as `prompt_cache_key`.

## Solution Assessment

Yes. A dedicated cache-key override is the narrowest maintainable fix because it stabilizes OpenAI cache routing without reusing a cron transcript and accumulating history.

## Repair Work Prompt

Implement the stable cron prompt cache key requested in https://github.com/openclaw/openclaw/issues/76218. Add an additive `promptCacheKey` field to cron `agentTurn` payloads and patches, normalize and persist it like the existing model/fallbacks/lightContext/tools fields, propagate it through isolated cron execution into the embedded runner/OpenAI stream options, and use it as the OpenAI `prompt_cache_key` only when cache retention is not `none` and the target provider path supports prompt cache keys. Preserve isolated cron transcript semantics: do not make `sessionTarget: "isolated"` reuse a stable session id, and do not change provider defaults or OpenAI-compatible opt-in policy. Add regression coverage for schema validation, normalization/create/patch persistence, isolated cron forwarding, and OpenAI Responses/completions payload construction. Update `docs/automation/cron-jobs.md`, `docs/reference/prompt-caching.md`, and `CHANGELOG.md` with user-facing behavior and provider limits. Be aware of related discussion in https://github.com/openclaw/openclaw/issues/76413: do not broaden prompt_cache_key forwarding to unsupported `openai-codex` or third-party endpoints.
