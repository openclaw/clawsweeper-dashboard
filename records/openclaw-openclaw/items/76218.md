## Summary

Keep open. Current main still has no cron payload cache-key field, isolated cron still creates a fresh session id per run, and OpenAI transports still derive `prompt_cache_key` from that session id; the separate cron model-override diagnostic concern is already addressed.

## What This Changes

Request to let cron `agentTurn` payloads set a stable `promptCacheKey` independent of the per-run isolated `sessionId`.

## Best Possible Solution

Add a validated `promptCacheKey` cron agent-turn field, propagate it as a provider cache-key override without changing isolated transcript semantics, and document the cron-specific usage.

## Reproduction Assessment

Yes. Source tracing is high-confidence: isolated cron forces a new session id, passes it into the embedded run, and OpenAI transport uses that id as `prompt_cache_key`; live billing would only quantify the cost impact.

## Solution Assessment

Yes. A dedicated cache-key override is the narrowest maintainable direction because `sessionTarget: "session:<stable-id>"` stabilizes the key only by also accumulating transcript history.
