## Summary

Keep open. Current main still has no cron `promptCacheKey` payload field, isolated cron still generates a fresh session id per run, and OpenAI transports still derive `prompt_cache_key` from that per-run `sessionId`; the optional model-override diagnostic concern is already addressed.

## What This Changes

Request adding a cron or agentTurn payload cache-key override so recurring cron runs can use a stable OpenAI `prompt_cache_key` without reusing the transcript session.

## Best Possible Solution

Add a validated `promptCacheKey` agent-turn payload field, propagate it to OpenAI prompt-cache routing while preserving isolated transcript semantics, and document the cron usage pattern and provider limits.

## Reproduction Assessment

Yes. Source tracing is high-confidence: isolated cron forces a fresh session id, the executor passes that id into the embedded run, and OpenAI transport uses that id as `prompt_cache_key`; live billing would only quantify the cost impact.

## Solution Assessment

Yes. A dedicated cache-key override is the narrowest maintainable solution because it stabilizes provider cache routing without reusing a session transcript and accumulating cron history.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76218. Add a validated `promptCacheKey` field to cron `agentTurn` payloads so isolated recurring cron jobs can set a stable OpenAI `prompt_cache_key` independent of the per-run `sessionId`. Preserve isolated transcript semantics and do not make `sessionTarget: "session:<stable-id>"` the solution. Propagate the field through cron normalization/types, isolated cron execution, embedded agent stream options, and OpenAI Responses plus opted-in OpenAI-compatible Completions transport. Keep `cacheRetention: "none"` suppressing the key and preserve existing `compat.supportsPromptCacheKey` and OpenAI Responses payload-policy stripping; be careful not to send unsupported prompt-cache params to native `openai-codex` paths given the adjacent #76413 report. Update cron and prompt-caching docs and add a user-facing changelog entry. Stop and ask for maintainer input if solving this requires a broader generic agentTurn API redesign beyond the cron payload path.
