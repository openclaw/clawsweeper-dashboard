## Summary

Keep this PR open with changes requested: the safety-net direction is reasonable for the linked gateway crash, but the classifier still trusts the wrong part of Node's ESM stack and can route a third-party/plugin runtime export mismatch into the clean-restart path that skips fatal diagnostics.

## What This Changes

The PR adds an unhandled-rejection classifier that treats OpenClaw bundled dist `*.runtime.js` export/missing-module failures as in-place-upgrade rotation and exits for a clean restart without fatal hooks, with classifier and handler tests.

## Best Possible Solution

Keep the runtime safety net, but extract the actual ESM importer from Node's stack/code-frame and only skip fatal hooks for proven OpenClaw-owned dist rotations; ambiguous plugin or dependency failures should stay on the fatal path.

## Reproduction Assessment

Yes for the blocking review finding: a Node 22 scratch ESM import reproduces the stack shape that makes the PR's ownership gate choose the OpenClaw async caller instead of the plugin importer. The linked upgrade crash itself is supported by logs and source, but not live-reproduced here.

## Solution Assessment

No: the restart safety net is a plausible fix, but this implementation needs a stricter importer ownership check before it is the maintainable solution. The diagnostic-bundle bypass also needs explicit maintainer acceptance after the classifier is safe.
