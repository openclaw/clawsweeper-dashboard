## Summary

Keep open: the safety-net direction is reasonable, but the new stack ownership check can overmatch third-party relative runtime export mismatches when an OpenClaw dist frame appears later in the async stack, so this needs a narrow classifier/test repair before merge.

## What This Changes

Adds an unhandled-rejection classifier and tests that route OpenClaw bundled dist runtime rotation errors to a clean restart that skips fatal-error hooks.

## Best Possible Solution

Keep a top-level restart safety net for verified OpenClaw own-dist rotations, but require the classifier to prove the failing relative import's actual importer is under OpenClaw dist and preserve fatal hooks for dependency/plugin failures.

## Reproduction Assessment

Yes for the review finding: a Node 24 ESM probe produced a third-party relative runtime export-mismatch stack that also contains an OpenClaw dist async caller, which the new stack-wide ownership check would overmatch. The original live upgrade race remains source-level rather than end-to-end reproduced here.

## Solution Assessment

No: the restart safety net is the right layer for a broken in-memory module graph, but the current relative-specifier ownership test is too broad. The safer fix is to match the actual importing module frame, then keep dependency/plugin failures on the normal fatal path.
