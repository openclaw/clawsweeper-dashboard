## Summary

Keep open. Current main and the latest release still force Feishu thread delivery from inbound `thread_id`/`root_id` even when `replyInThread` is configured as `"disabled"`, so the PR is not implemented or obsolete; however, the branch is stale against the current `bot-content.ts` split, overlaps with open PR #73958, and needs maintainer judgment on root/thread semantics before it can land cleanly.

## What This Changes

The PR changes Feishu group session resolution and one bot test so non-topic group quote/thread metadata no longer sets `replyInThread` or `threadReply` when `replyInThread` is disabled, while preserving thread context for topic session scopes.

## Best Possible Solution

Settle the Feishu semantics explicitly: if disabled is meant to be a hard off switch for normal groups, port the resolver change to `extensions/feishu/src/bot-content.ts`, preserve topic-mode anchoring and direct-send thread delivery, update the current regression tests and docs/changelog as needed, and coordinate that outcome with #73958 and #73951.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a normal Feishu group with `replyInThread: "disabled"`, provide inbound `thread_id`, and `resolveFeishuGroupSession` still returns `replyInThread: true`/`threadReply: true`; the current unit test at `extensions/feishu/src/bot.test.ts:2668` asserts exactly that behavior.

## Solution Assessment

No. The PR direction is plausible, but the submitted branch is stale and still needs a current-main port plus a check that direct send paths do not lose topic/thread delivery when only `threadReply` remains true.
