## Summary

Keep open: current main still appears to force Feishu thread replies when disabled config sees inbound thread metadata, but this PR is conflicting and should not merge as-is because the active resolver moved and the root_id/thread_id semantics still need maintainer review.

## What This Changes

The PR changes Feishu group-session routing and one bot test so disabled `replyInThread` no longer treats incoming `thread_id` or `root_id` metadata as thread replies in non-topic groups.

## Best Possible Solution

Land one current-main Feishu plugin fix that updates `bot-content.ts`, dispatcher behavior, and focused tests together while explicitly deciding the `root_id` versus `thread_id` semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows inbound `threadId` or `rootId` still forces `replyInThread` true despite disabled config, and the current bot test asserts that behavior.

## Solution Assessment

No. The bug-fix direction is plausible, but this branch edits the stale resolver location, can drop thread delivery for topic-scoped direct sends, and needs reconciliation with https://github.com/openclaw/openclaw/pull/73958 before it is the best fix.
