## Summary

Keep open. Current main still overrides Feishu `replyInThread: "disabled"` when inbound `thread_id` or `root_id` is present, so the PR is not obsolete, but the submitted branch is stale against the current resolver split and has a concrete topic-thread delivery regression before it can merge.

## What This Changes

The PR changes Feishu group session resolution and one bot test so non-topic group quote/thread metadata no longer sets `replyInThread` or `threadReply` when `replyInThread` is disabled, while preserving topic-scope thread context.

## Best Possible Solution

Land one coordinated Feishu-plugin patch that updates `resolveFeishuGroupSession` in `bot-content.ts`, keeps separate concepts for session topic context and outbound `reply_in_thread`, preserves topic-mode/direct-send anchoring, updates the regression tests, and records the user-facing behavior change in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a normal Feishu group with `replyInThread: "disabled"`, provide inbound `thread_id`, and the resolver/test path still produces `replyInThread: true` and `threadReply: true`.

## Solution Assessment

No. The PR direction is plausible, but the current patch is not the best fix as written because it makes `replyInThread` config-only while direct send paths still use that field for topic delivery, and it has not been ported to the current `bot-content.ts` resolver.
