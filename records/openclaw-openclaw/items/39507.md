## Summary

Keep open. Current main still forces Feishu thread delivery from inbound `thread_id` or `root_id` even when `replyInThread` is disabled, so this PR is not obsolete; however, the branch targets the old resolver location, has a concrete topic-thread direct-send regression, and overlaps with open #73958, so it needs maintainer coordination before merge.

## What This Changes

The PR changes Feishu group session resolution and one bot test so non-topic group quote/thread metadata no longer sets `replyInThread` or `threadReply` when `replyInThread` is disabled, while preserving topic-scope thread context.

## Best Possible Solution

Settle the Feishu semantics explicitly, then land one coordinated Feishu-plugin patch that updates `resolveFeishuGroupSession` in `bot-content.ts`, preserves topic-mode anchoring and direct-send thread delivery, updates the regression tests, and adds user-facing docs or changelog coverage if behavior changes.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a normal Feishu group with `replyInThread: "disabled"`, provide inbound `thread_id`, and the resolver/test path still produces `replyInThread: true` and `threadReply: true`.

## Solution Assessment

No. The PR direction is plausible, but the submitted patch is stale against current main and can drop topic-thread delivery for direct send paths that use `groupSession.replyInThread` rather than the dispatcher override.
