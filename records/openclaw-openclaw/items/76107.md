## Summary

Keep open. Current main contains partial prompt/tool caching and usable OAuth-token reuse, but the embedded runner still performs auth initialization for each non-plugin-harness run and still marks the reported `core-plugin-tools`, `system-prompt`, and `stream-setup` stages per attempt; the MiniMax provider registration does not show a provider-owned runtime-auth exchange hook that would prove the reported MiniMax auth delay is fixed.

## What This Changes

The report asks to reduce per-message MiniMax/Feishu latency by caching or reusing auth, system-prompt, and plugin/tool preparation across warm sessions.

## Best Possible Solution

Capture the same current-main stage trace first, then split any fixes between provider-owned MiniMax runtime-auth reuse and generic prompt/tool hot-path reductions without putting MiniMax policy into core.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map directly to current embedded-runner telemetry, and another user corroborates the symptom, but the issue lacks raw trace logs/config and I did not run a live MiniMax-M2.7/Feishu message.

## Solution Assessment

Unclear. The requested caching direction is plausible, but the maintainable fix needs a measured current-main trace and should use the existing provider runtime-auth and prompt/tool cache seams rather than a broad session-level shortcut that could break expiry, profile rotation, plugin disablement, or dynamic prompt inputs.
