## Summary

Keep open. Current main has partial post-release work for stable system-prompt prefix caching and plugin tool descriptor reuse, but the embedded runner still runs auth initialization during startup and MiniMax does not expose a visible runtime-auth exchange/reuse hook, so the reported auth bottleneck is not proven fixed.

## What This Changes

The issue asks OpenClaw to cache or reuse provider auth, system-prompt construction, and plugin-tool setup to reduce per-message MiniMax-M2.7/Feishu latency.

## Best Possible Solution

Capture the same stage trace on current main, then split any fix into provider/session auth reuse and remaining measured prompt/tool hot paths without moving MiniMax-specific policy into core.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map to current embedded-run telemetry, but the issue does not include raw trace logs/config and I did not run a live MiniMax-M2.7/Feishu message to reproduce the exact timings.

## Solution Assessment

Unclear. Stable prompt-prefix and plugin descriptor caches are useful partial fixes, but auth reuse needs a narrower design that preserves token expiry, profile rotation, provider hooks, and plugin disablement behavior.
