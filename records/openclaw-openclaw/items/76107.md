## Summary

Keep open. Current main has partial prompt/tool-startup optimizations after the reported release, but embedded auth still runs during each PI startup and MiniMax has no visible runtime-auth reuse hook, so the reported auth bottleneck is not proven fixed.

## What This Changes

Request to cache provider auth, system-prompt preparation, and plugin-tool preparation so MiniMax-M2.7 Feishu message runs avoid repeated auth and fixed prep overhead.

## Best Possible Solution

Profile the reported MiniMax/Feishu path on current main, then split any remaining work into generic provider/session auth reuse and narrowly scoped prompt/tool hot-path improvements.

## Reproduction Assessment

No high-confidence reproduction yet. The reported stage names map to current embedded-run startup/prep markers, but the issue lacks raw trace logs/config and I did not run the live MiniMax-M2.7/Feishu message path on current main.

## Solution Assessment

Unclear. The caching direction is plausible, but the maintainable fix should first identify which stages remain hot on current main and avoid MiniMax-specific policy in the core runner.
