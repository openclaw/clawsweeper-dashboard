## Summary

Keep open. Current main has partial post-release prompt/tool caching, but the central bottlenecks are not proven fixed: auth is still initialized in the embedded run startup path and the reported tool, prompt, and stream stages still run as per-attempt preparation.

## What This Changes

The issue asks for cached/reused auth plus session-level prompt and plugin-tool warm paths to reduce MiniMax/Feishu message-pipeline latency.

## Best Possible Solution

Capture the same stage trace on current main, then split fixes into generic provider/session auth reuse and any remaining prompt/tool prep hot paths without MiniMax-specific policy in core.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map directly to current embedded-runner timing markers, but the report lacks raw trace/config and I did not run a live MiniMax-M2.7/Feishu message.

## Solution Assessment

Unclear. The caching direction is plausible, but the best fix needs current-main profiling plus a narrower auth/session-cache design that preserves expiry, profile rotation, provider hooks, and plugin disablement behavior.
