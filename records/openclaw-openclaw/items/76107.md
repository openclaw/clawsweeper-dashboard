## Summary

Keep open. Current main still runs embedded auth initialization before the `auth` startup stage and still has per-attempt `core-plugin-tools`, `system-prompt`, and `stream-setup` prep stages; the stable prompt-prefix cache is a partial improvement, and open PR #76277 addresses related latency churn but does not prove the reported MiniMax auth bottleneck is fixed.

## What This Changes

The issue reports message-pipeline latency from per-message provider auth plus per-message prompt and plugin-tool preparation overhead.

## Best Possible Solution

Capture the same stage trace on current main, land or revise #76277 for the hot paths it actually fixes, then address any remaining bottleneck with provider/session auth reuse and prompt/tool caching that preserve OAuth expiry, profile rotation, and plugin-owned provider hooks.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map directly to current embedded-runner telemetry and another user corroborated the symptom, but there are no raw logs/config and I did not run a live MiniMax-M2.7/Feishu message.

## Solution Assessment

Unclear. The requested caching direction is plausible, but the maintainable fix needs a narrower design than generic token reuse so it respects provider runtime hooks, profile rotation, plugin disablement, and OAuth expiry.
