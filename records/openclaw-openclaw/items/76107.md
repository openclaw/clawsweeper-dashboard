## Summary

Keep open. Current main has post-2026.4.29 work that caches the stable system-prompt prefix and plugin tool descriptors, but the runner still initializes auth per run and I did not find a session-level MiniMax auth reuse path that proves the reported auth bottleneck is solved.

## What This Changes

The issue asks to reduce per-message MiniMax/Feishu reply latency by caching provider auth and avoiding repeated system-prompt plus core/plugin tool setup.

## Best Possible Solution

Keep this open until maintainers can capture the same stage trace on current main, then split the work into targeted fixes for durable provider auth reuse and any remaining core tool/system-prompt hot paths.

## Reproduction Assessment

Partial. The named stage labels map directly to current embedded-run telemetry, but the issue lacks raw trace logs/config and I did not run a live MiniMax/Feishu message, so the exact 6.2s and 16.4s numbers are not independently reproduced.

## Solution Assessment

Unclear/partly. Prompt and tool caching are already partly addressed on current main after v2026.4.29, while the auth proposal still needs a narrower design that preserves OAuth expiry, profile rotation, provider hooks, and plugin disablement behavior.
