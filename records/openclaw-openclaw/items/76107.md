## Summary

Keep open. Current main has partial hot-path improvements for stable prompt prefixes and plugin tool descriptors, but the embedded runner still performs auth initialization during each run and MiniMax does not show a provider runtime-auth reuse hook, so the reported auth and fixed-overhead bottlenecks are not proven solved.

## What This Changes

Reduce message-run auth and fixed prompt/tool preparation latency by caching provider auth, session prompt state, and plugin/tool initialization.

## Best Possible Solution

Capture a current-main stage trace for the same MiniMax-M2.7/Feishu path, then land targeted generic fixes for provider/session auth reuse and any remaining prompt/tool hot paths without MiniMax-specific policy in core.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map directly to embedded-run telemetry and another user corroborated the symptom, but the issue does not include raw trace logs/config and I did not run a live MiniMax-M2.7/Feishu message on current main.

## Solution Assessment

Unclear. The requested caching direction is plausible, but current main only partially mitigates prompt/tool overhead and durable auth reuse needs a narrower design that preserves OAuth expiry, profile rotation, and provider runtime hook behavior.
