## Summary

Keep open. Current main has partial prompt/tool hot-path improvements, but the reported auth and fixed-prep bottlenecks are not proven fixed: auth still initializes in the embedded runner startup path and the same `core-plugin-tools`, `system-prompt`, and `stream-setup` stages still exist per attempt.

## What This Changes

The issue reports message-pipeline latency from per-message provider auth plus per-message prompt and plugin tool preparation overhead.

## Best Possible Solution

Capture a current-main stage trace for the same MiniMax-M2.7/Feishu path, then split durable provider/session auth reuse from any remaining generic prompt and plugin-tool hot-path reductions without moving MiniMax policy into core.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map directly to current embedded-runner telemetry and another user corroborated the symptom, but the issue lacks raw logs/config and I did not run a live MiniMax-M2.7/Feishu message against current main.

## Solution Assessment

Unclear. The stable prompt-prefix and prompt-report caches are useful partial fixes, but the auth proposal needs a narrower design that preserves OAuth expiry, profile rotation, provider runtime hooks, and plugin disablement behavior.
