## Summary

Keep open. Current main includes several prompt, plugin-tool, and runtime-cache improvements, but it still does per-run auth startup and per-attempt prompt/tool preparation, and there is no current-main proof that the reported MiniMax auth bottleneck or remaining fixed overhead is solved.

## What This Changes

The issue asks OpenClaw to reduce message-pipeline latency by caching or reusing authentication, system-prompt preparation, and plugin/tool initialization across messages.

## Best Possible Solution

Capture the same stage trace on current main, then split any remaining work into provider/session auth reuse and narrowly scoped prompt/tool hot-path reductions without baking MiniMax-specific policy into core.

## Reproduction Assessment

Partial, not high confidence. The reported stage names map to current embedded-run telemetry and another user confirmed seeing the issue, but the issue lacks raw trace logs and I did not run a live MiniMax-M2.7/Feishu message to verify the exact timings.

## Solution Assessment

Unclear. The existing prompt-prefix and plugin descriptor caches are directionally right, but the auth proposal needs a design that preserves provider OAuth expiry, profile rotation, runtime hook isolation, and plugin disablement behavior.
