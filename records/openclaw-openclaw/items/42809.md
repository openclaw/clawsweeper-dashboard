## Summary

Keep open: the PR targets a real Feishu card rendering bug, but the branch is stale against current main's structured-card paths and still introduces merge-blocking regressions around card width config, fenced code, table-limit fallback, and quoted-card extraction.

## What This Changes

This PR adds a Feishu markdown-card helper that converts Markdown pipe tables into native Card JSON 2.0 table elements, wires buildMarkdownCard to it, and adds unit/integration tests.

## Best Possible Solution

Implement a shared Feishu card content builder that covers markdown, structured, streaming/update, and quoted-card extraction paths while preserving width_mode and falling back when Feishu table limits are exceeded.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu send. Current main routes renderMode=card or auto table-like text through sendStructuredCardFeishu/buildStructuredCard, which emits one markdown element, so Feishu card markdown receives raw pipe-table text.

## Solution Assessment

No. Native table conversion is a useful direction, but hooking only buildMarkdownCard is not the narrowest maintainable fix for current main because structured cards, streaming updates, quote extraction, width config, and table-limit fallback all need to agree.

  - body: The splitter emits one native table component for every matched Markdown table. Related Feishu reports show cards can fail with the platform table-count limit, so multi-table messages need a cap or fallback instead of blindly pushing every table element.
