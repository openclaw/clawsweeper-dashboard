## Summary

Keep this PR open, but it is not merge-ready. The native-table direction addresses a real Feishu card bug, yet the latest head still misses current main's structured and streaming card paths, regresses the schema-2.0 width config, leaves table/code and quote-extraction edge cases, and lacks after-fix real Feishu proof.

## What This Changes

The branch adds a Feishu markdown-card helper that converts Markdown pipe tables into Card JSON 2.0 table elements, delegates buildMarkdownCard to it, and adds Feishu tests.

## Best Possible Solution

Build one Feishu card table normalization and fallback layer that covers markdown cards, structured outbound/reply cards, streaming updates, quoted-card extraction, and platform table-count limits while preserving width_mode.

## Reproduction Assessment

Yes, from source inspection. Current main treats pipe tables as card-worthy content, but card builders emit markdown elements unchanged and the card paths skip the plain-text table conversion; no live Feishu send was performed in this read-only review.

## Solution Assessment

No. The conversion idea is useful, but the PR hooks only buildMarkdownCard while current main primarily routes card/auto replies and outbound sends through structured and streaming card paths, and the branch still leaves width, tilde-fence, quote-extraction, and table-limit issues.

  - confidence: 0.86

  - body: The splitter emits one native table component for every matched Markdown table. Related Feishu reports show cards can fail with the platform table-count limit, so multi-table messages need a cap or fallback instead of blindly pushing every table element.
  - confidence: 0.84

  - body: After this change, outgoing cards can contain native table elements, but current interactive-card extraction only reads text-like elements. Replies quoting a table-only card can lose that context and fall back to generic interactive-card text.
