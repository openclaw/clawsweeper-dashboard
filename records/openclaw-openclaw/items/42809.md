## Summary

Keep open. The PR targets a real Feishu card-table rendering gap that is not implemented on current main, but the branch still needs real Feishu behavior proof and fixes for current card paths, width config, fenced-code handling, table-count fallback, and quoted-card extraction before merge.

## What This Changes

The branch adds a Feishu markdown-card helper that converts Markdown pipe tables into Card JSON 2.0 table elements, delegates buildMarkdownCard to it, and adds Feishu tests.

## Best Possible Solution

Build one Feishu card table normalization and fallback layer that covers markdown cards, structured outbound/reply cards, streaming updates, quoted-card extraction, and platform table-count limits while preserving the schema-2.0 width_mode contract.

## Reproduction Assessment

Yes, at source level. Current main treats Markdown tables as card-worthy content, but card builders emit markdown elements unchanged and the card paths skip the plain-text table conversion; no live Feishu send was performed in this read-only review.

## Solution Assessment

No. The native table conversion idea is useful, but this PR wires only the old markdown-card helper and leaves current structured/reply/outbound/streaming card paths plus fallback and extraction behavior incomplete.

  - confidence: 0.86

  - body: The splitter emits one native table component for every matched Markdown table. Related Feishu reports show cards can fail with the platform table-count limit, so multi-table messages need a cap or fallback instead of blindly pushing every table element.
  - confidence: 0.84
- **[P2] Extract native table text when quoting card replies:** `extensions/feishu/src/markdown-card.ts:133-145`
  - body: After this change, outgoing cards can contain native table elements, but current interactive-card extraction only reads text-like elements. Replies quoting a table-only card can lose that context and fall back to generic interactive-card text.
