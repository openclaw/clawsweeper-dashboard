## Summary

Keep open: current main still does not render Markdown pipe tables as native Feishu card tables, but this PR head is not merge-ready because it only wires one card builder, conflicts with the current Schema 2.0 width_mode contract, lacks table-limit and quoted-reply coverage, and has no real Feishu/Lark proof.

## What This Changes

The PR adds a Feishu markdown-card helper that parses Markdown pipe tables into native Card JSON 2.0 table elements and wires buildMarkdownCard plus tests to that helper.

## Best Possible Solution

Implement one Feishu card table and fallback layer shared by markdown, structured, reply, outbound, streaming-close, and quoted-card paths, preserving Schema 2.0 width_mode and platform limits, then verify it with redacted real Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level. Current main routes Markdown pipe tables into Feishu card markdown paths while Feishu documents native table components and no pipe-table syntax for card Markdown; I did not run a live Feishu send.

## Solution Assessment

No for the current PR head. Native table conversion is a reasonable ingredient, but the branch must cover structured card surfaces, preserve width_mode, respect table limits, retain quoted-reply context, and include real Feishu/Lark proof.
