## What This Changes

The PR adds a Feishu markdown-card helper that converts Markdown pipe tables into native Card JSON 2.0 table elements and delegates buildMarkdownCard to that helper with new tests.

## Best Possible Solution

Implement one Feishu card table and fallback layer shared by markdown, structured, reply, outbound, streaming-close, and quoted-card paths, preserving Schema 2.0 width_mode and Feishu platform limits, then verify it with redacted real Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level. Current main detects Markdown pipe tables for card mode and sends card replies through markdown/structured card elements without native table conversion, while Feishu docs list a limited markdown subset and a separate native table component; I did not run a live Feishu send.

## Solution Assessment

No for the current PR head. Native table conversion is a reasonable ingredient, but the branch must cover structured card surfaces, preserve width_mode, respect table limits, retain quoted-reply context, and include real Feishu/Lark proof.

  - confidence: 0.88
