## Label Justifications

- `P1`: Default Telegram polling can silently lose its configured stall recovery path for real channel workflows.
- `impact:message-loss`: A silent isolated polling worker can leave inbound Telegram updates undelivered without the configured watchdog firing.

## Summary

Keep open: current main and v2026.5.18 still route default Telegram polling through isolated ingress while the configured stall threshold is only used by the embedded polling branch; https://github.com/openclaw/openclaw/pull/83505 fixed stalled spool handlers but not this parent-worker liveness gap.

## What This Changes

The issue requests that `channels.telegram.pollingStallThresholdMs` be honored when Telegram isolated polling ingress is enabled by default.

## Best Possible Solution

Honor the existing documented `pollingStallThresholdMs` in isolated ingress by tracking worker message liveness and restarting the worker/bot cycle on the configured threshold, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main defaults into isolated ingress, and the only `#stallThresholdMs` consumer is the embedded polling watchdog. I did not run a live Telegram stall because this review is read-only.

## Solution Assessment

Yes. Wiring the existing documented setting into the default isolated path is narrower and more maintainable than documenting it as embedded-only or adding a public opt-out just to recover old behavior.
