## Summary

Keep open: the latest head fixes the prior live-base-metadata concern by preserving known live cron base keys, but it still misses orphaned base rows in non-default agent stores after the last job for that agent is deleted. The PR should be repaired before merge; the linked canonical issue remains open until a fix lands.

## What This Changes

The PR broadens cron session retention so the reaper can prune stale unowned base cron session keys, wires live-job ownership keys through the cron timer and gateway, and adds reaper regression tests.

## Best Possible Solution

Visit every configured per-agent session store needed for cron retention, preserve live job-owned base rows, prune stale deleted-job orphans with regression coverage, and align docs if retention semantics now cover orphaned base rows.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron uses stable base keys while the reaper filters only :run keys; source inspection of the PR also shows non-default stores with only deleted-job orphans are skipped because store paths are derived from live jobs.

## Solution Assessment

No as written. The guarded orphan-only design is better than broad age-pruning live base rows, but store discovery has to cover agent stores that no live job references before it fully solves deleted-job orphan cleanup.
