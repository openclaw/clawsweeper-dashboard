## Summary

Keep open: the bug is real and this PR is a plausible guarded repair, but the current head still makes stale isolated cron base rows retention-deletable even though current code and tests intentionally preserve user-selected model/auth/label metadata in those rows. Because the PR author is a repository member and the remaining blocker is session-state upgrade semantics, this needs maintainer review rather than cleanup closure.

## What This Changes

The PR broadens cron session retention sweeping to include stale unprotected base cron session keys, passes a protected non-isolated key set from the cron timer, and adds regression tests for isolated or orphaned base sessions.

## Best Possible Solution

Preserve job-owned cron base metadata while pruning or resetting only disposable isolated-run handles and transcripts, with replacement-time orphan transcript cleanup either included or explicitly split to a canonical follow-up.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable base keys while the reaper only accepts :run-shaped keys, so stale isolated base entries skip the retention cutoff; I did not run a mutating live gateway repro in this read-only review.

## Solution Assessment

No as written. The guarded approach is better than a broad predicate swap, but deleting the whole isolated base row is not the safest lifecycle operation because current code intentionally carries user model/auth metadata through that row.
