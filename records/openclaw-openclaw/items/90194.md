## Summary

Keep open; the branch fixes the isolated-base retention gap in principle, but it still has a blocking compatibility/session-state defect because documented custom `session:cron:*` targets can be swept and archived as unsiblinged base cron keys.

## What This Changes

The PR changes the cron session reaper to prune stale cron base keys without `:run:` siblings, preserve base keys with run siblings, and add focused retention and transcript-archival tests.

## Best Possible Solution

Restrict retention to generated isolated cron base keys and existing `:run:` children while preserving persistent custom session targets; split deleted-job orphan cleanup unless generated ownership is explicitly tracked.

## Reproduction Assessment

Yes. Source inspection of current main shows `sweepCronRunSessions` only accepts `isCronRunSessionKey`, while isolated jobs use the base `agent:<id>:cron:<jobId>` key, so the linked bug is source-reproducible; I did not run tests in this read-only review.

## Solution Assessment

No. The reaper is the right owner path, but the proposed no-sibling base-key rule is too broad because persistent custom `session:cron:*` targets have the same key shape and no `:run:` sibling.
