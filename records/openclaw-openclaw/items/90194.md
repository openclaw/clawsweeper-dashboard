## Summary

Keep open: current main and the latest release still need an isolated cron retention fix, but this PR is not merge-ready because it makes stale isolated cron base rows fully deletable and can drop durable model/auth/label metadata for long-cadence jobs.

## What This Changes

The branch broadens cron retention sweeping to prune stale unprotected base cron keys, passes protected non-isolated keys from the timer, and adds regression tests around isolated or orphaned base sessions.

## Best Possible Solution

Preserve job-owned cron base metadata while pruning or resetting only disposable isolated-run session handles and transcripts, and explicitly include or split replacement-time orphan transcript cleanup.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable base keys while the reaper only accepts :run-shaped keys, so stale isolated base entries skip the retention cutoff; I did not run a mutating live gateway repro in this read-only review.

## Solution Assessment

No as written. The guarded approach is better than a broad predicate swap, but deleting the whole isolated base row is not the safest lifecycle operation because current code intentionally carries user model/auth metadata through that row.
