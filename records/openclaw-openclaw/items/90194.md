## Summary

Keep open: this is a focused bug-fix PR for a real cron/session retention gap, but the latest head is still not merge-ready because it only sweeps stores derived from current live jobs or the default store, leaving stale base rows in non-default agent stores when the last job for that agent was deleted.

## What This Changes

The PR broadens cron session retention to prune stale orphaned base cron session keys, wires live-job ownership through the cron timer and gateway, and updates cron retention tests and docs.

## Best Possible Solution

Visit every configured per-agent session store needed for cron retention, preserve live job-owned base rows, prune stale deleted-job orphans with focused regression coverage, and keep the docs aligned with the final orphan-only behavior.

## Reproduction Assessment

Yes, at source level. Current main writes isolated cron state to stable `agent:<id>:cron:<jobId>` base keys while the reaper only accepts `:run:` keys, and the PR-head gap is also source-reproducible from store paths derived only from live jobs/default store.

## Solution Assessment

No as written. The guarded orphan-only design is the right direction compared with broad age-pruning live base rows, but store discovery must cover non-default agent stores that no live job currently references before this fully solves deleted-job orphan cleanup.
