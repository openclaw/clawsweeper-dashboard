## Summary

Keep open: the refreshed PR targets a real cron session-retention bug, but the current patch can delete stale job-owned isolated cron base rows wholesale, which conflicts with current code that carries user-selected model/auth metadata through fresh isolated cron runs.

## What This Changes

The PR broadens cron session reaping to prune stale unprotected base cron keys, passes protected non-isolated cron session keys from the timer, and adds regression tests for isolated or orphaned base sessions.

## Best Possible Solution

Preserve job-owned cron base metadata while pruning or resetting only disposable isolated-run session handles/transcripts, and explicitly split or include replacement-time orphan transcript cleanup.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable base keys while the reaper only accepts :run-shaped keys, so stale isolated base entries skip the retention cutoff; I did not run a mutating live gateway repro in this read-only review.

## Solution Assessment

No as written. The guarded approach is better than a broad predicate swap, but deleting the whole stale isolated base row is not the safest lifecycle operation because current code intentionally carries user model/auth metadata through that row.
