## Summary

Keep open: this PR targets a real shipped cron retention bug and is member-authored, so it is not cleanup-closeable. The current head still over-protects stale base cron rows after a job changes away from a base-row-owning target, so it needs a narrow repair before merge.

## What This Changes

The PR extends cron retention sweeping to prune stale orphaned base cron session rows, wires agent-aware ownership keys through the cron timer, and adds regression coverage.

## Best Possible Solution

Narrow protected-key construction to only jobs that actually persist the base row, keep run pruning unchanged, and add target-change regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 only prune :run: cron keys while isolated cron jobs persist stable base keys that skip the retention cutoff.

## Solution Assessment

No, not yet. The orphan-only design is the right direction, but the current helper over-approximates ownership and can preserve base rows for jobs that no longer write them.
