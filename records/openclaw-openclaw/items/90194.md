## Summary

Keep open: the PR fixes the original unscoped isolated cron base-key leak, but current head still has a blocking compatibility/session-state defect for persistent custom cron targets that are already agent-scoped or case-normalized by runtime session-key resolution.

## What This Changes

The PR broadens cron session retention sweeping to prune stale isolated cron base session keys while preserving persistent non-isolated cron session keys with a new protected-key set.

## Best Possible Solution

Use the same canonical session-key resolution for known persistent cron keys that the runtime uses when writing sessions, then cover agent-scoped and case-normalized custom cron targets before merging.

## Reproduction Assessment

Yes. Source inspection shows current main filters cron retention through isCronRunSessionKey, which returns false for base cron keys such as agent:main:cron:<jobId>, so isolated base sessions are source-reproducible as unpruned.

## Solution Assessment

No, not yet. The PR is the right ownership area, but current head manually builds protected session keys instead of reusing runtime canonicalization, leaving supported persistent custom targets vulnerable to the new base-key sweep.
