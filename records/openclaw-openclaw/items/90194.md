## Summary

Keep open: this PR targets a real cron retention bug that still exists on current main and the latest release, but the current patch makes stale isolated cron base rows deletion-eligible and can drop persisted model/auth/label metadata. Because this is a member-authored, compatibility-sensitive session-state PR, cleanup closure is not appropriate.

## What This Changes

The PR broadens cron session retention sweeping to delete stale unprotected base cron session keys, passes known persistent cron session keys from the timer, and adds reaper regression tests.

## Best Possible Solution

Preserve job-owned cron base metadata while pruning or resetting only disposable isolated-run session handles and transcripts, and either fix or explicitly split replacement-time orphan transcript cleanup.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron jobs persist stable base keys while the reaper only accepts :run-shaped keys, so stale isolated base entries skip the retention cutoff; I did not run a mutating live gateway repro in this read-only review.

## Solution Assessment

No as written. The guarded approach is safer than a broad predicate swap, but deleting the whole isolated base row is not the best lifecycle operation because current code intentionally carries user model/auth metadata through that row.
