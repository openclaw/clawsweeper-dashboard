## Summary

Keep open: the guarded orphan-only design is useful, but current PR head still skips non-default agent session stores once the last job for that agent has been deleted, so part of the deleted-job orphan cleanup remains unresolved before merge.

## What This Changes

The PR extends cron session retention to prune stale orphaned base cron session rows, wires live-job ownership into the cron timer and gateway, adds regression tests, and updates cron retention docs.

## Best Possible Solution

Visit every configured per-agent session store needed for cron retention, preserve live job-owned base rows, prune stale deleted-job orphans with regression coverage, and keep the docs aligned with that final behavior.

Yes. Source inspection on current main shows isolated cron writes stable base keys while the reaper filters only `:run:` keys; PR source inspection also shows non-default orphan-only stores are skipped because store paths are derived from live jobs.

## Solution Assessment

No as written. The guarded orphan-only design is better than broad age-pruning live base rows, but store discovery must cover agent stores that no live job references before this fully solves deleted-job orphan cleanup.
