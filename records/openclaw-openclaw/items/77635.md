## Summary

Keep open. This is a fresh, detailed gateway reliability report, and current main still routes HTTP webhooks and WebSocket upgrades through one Node HTTP server; diagnostics expose event-loop starvation but do not self-heal or prove the reported failure is fixed.

## What This Changes

The issue reports gateway HTTP webhook requests becoming unresponsive after upgrading to 2026.5.3 while existing WebSocket clients continue to work.

## Best Possible Solution

First isolate the blocking workload from diagnostics/stability data, then make the narrowest gateway or owner-plugin change that keeps liveness probes and webhook ACK paths responsive or adds a deliberate recovery path.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the reported shape because HTTP webhooks and WebSocket upgrades share one Node event loop, but the specific 30-60 minute stall still needs a current-main live repro or diagnostics bundle.

## Solution Assessment

Unclear. The requested self-heal direction may be useful, but the best fix is likely to identify and remove or yield the blocking work before adding restart policy.
