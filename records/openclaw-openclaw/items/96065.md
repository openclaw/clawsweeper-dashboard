## Summary

Keep open: this targets a real security-sensitive install bug, but the proposed managed-key fix is not mergeable because it can remove config SecretRef credentials from the generated service instead of moving them to the existing EnvironmentFile path.

## What This Changes

The PR adds config SecretRef env refs to `OPENCLAW_SERVICE_MANAGED_ENV_KEYS`, updates one install-plan assertion, and adds a standalone repro script.

## Best Possible Solution

Preserve the SecretRef value in the install plan, mark the intended Google/Gemini credential entries as file-backed through `environmentValueSources`, and prove staged service output has no inline plaintext while the generated env file still delivers the key.

## Reproduction Assessment

Yes from source inspection. The diff adds config SecretRef keys to the managed-key list, and current `writeManagedServiceEnvKeysToEnvironment` deletes managed keys from the plan before systemd file-backed rendering can write them.

## Solution Assessment

No. The maintainable fix is to preserve the secret value and set file-backed source metadata for the service renderer, not to use `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` as the delivery mechanism.
