## Summary

Keep open: the PR targets a real security-sensitive daemon install bug, but the proposed managed-key approach can remove the config SecretRef value from the install plan without writing it to the systemd EnvironmentFile, so it is not merge-ready.

## What This Changes

The PR adds config SecretRef env keys to `OPENCLAW_SERVICE_MANAGED_ENV_KEYS`, adjusts one daemon install test, and adds a standalone repro script for the `GEMINI_API_KEY` service-env case.

## Best Possible Solution

Use the existing `environmentValueSources: "file"` path for the affected Google/Gemini config SecretRef values, then prove the staged systemd unit omits plaintext and the generated 0600 EnvironmentFile still contains the redacted runtime value.

## Reproduction Assessment

Yes. Source inspection shows config SecretRef values enter the install-plan environment as inline service values on current main, and systemd renders inline values as `Environment=` assignments.

## Solution Assessment

No. The PR removes the plaintext value from the plan but does not preserve runtime delivery through the systemd EnvironmentFile; the maintainable fix is to mark the value as file-backed instead.
