## Summary

Keep open: this is a focused fix attempt for a real credential-exposure bug, but the patch is not merge-ready because it removes the config SecretRef value from the install plan without preserving runtime delivery through the renderer contract.

## What This Changes

The PR adds config SecretRef env refs to `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` during gateway install planning, updates one regression assertion, and adds a repro script.

## Best Possible Solution

Replace the managed-key removal approach with a file-backed `environmentValueSources` fix for the relevant config SecretRef provider secrets, then prove staged service output omits inline plaintext while preserving runtime delivery.

## Reproduction Assessment

Yes for a source-level path: current main adds config SecretRef env values to the install-plan environment with default inline source, and the PR then puts those keys into a managed-key policy that deletes the value before file-backed delivery. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is the file-backed `environmentValueSources` approach used by the current renderer contract, not adding config SecretRef values to `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` and proving only that the plan no longer contains the inline value.
