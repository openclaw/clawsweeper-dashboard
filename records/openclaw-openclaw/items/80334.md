## Summary

Keep open and do not merge as-is: the PR addresses a real Ollama failure mode, but its blanket `options.think` request shape conflicts with upstream Ollama’s current chat API contract and current OpenClaw tests, and the PR supplies only unit-test proof.

## What This Changes

The PR moves Ollama native chat `think` values from top-level request fields into `options.think` and updates Ollama stream tests to expect that placement.

## Best Possible Solution

Preserve top-level Ollama `think` for models with native thinking support, and add a capability-aware guard or clamp so non-thinking models cannot receive truthy top-level `think` from defaults, overrides, or explicit params unless live upstream proof supports another contract.

## Reproduction Assessment

Yes, partially from source: current main can emit top-level truthy `think` from runtime or explicit model params, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact default-thinking route from the linked issue still needs live/config confirmation because current OpenClaw also has provider thinking-profile validation.

## Solution Assessment

No. Moving all Ollama thinking control into `options.think` is not the narrowest maintainable fix because upstream Ollama models thinking control as a top-level chat request field; the safer path is capability-aware guarding while preserving top-level `think` for supported models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep native Ollama thinking on the chat request field:** `extensions/ollama/src/stream.ts:242`

Moving runtime thinking into `options.think` means `/think off` and effort levels for reasoning-capable Ollama models will not be seen by the server, so keep top-level `think` for supported models and guard the non-thinking path separately.
