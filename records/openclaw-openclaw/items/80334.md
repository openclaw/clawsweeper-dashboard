## Summary

Keep this PR open but do not merge it as written: the diff is focused, but it moves Ollama `think` into `options.think`, which current Ollama source treats as an ordinary model option and ignores rather than as thinking control. The PR also has only mocked/unit proof, not after-fix live Ollama behavior proof.

## What This Changes

The PR moves Ollama `think` values from top-level `/api/chat` request fields into `options.think` and updates related stream tests to expect that placement.

## Best Possible Solution

Preserve Ollama's top-level `think` for thinking-capable models and explicit supported model params, while preventing truthy top-level `think` from reaching non-reasoning models; back it with focused regression tests and live Ollama proof.

## Reproduction Assessment

Yes, from source: current main and the PR diff show the changed payload placement, and upstream Ollama source shows `options.think` is not the request field that controls thinking. I did not run a live Ollama endpoint for the reporter's exact models.

## Solution Assessment

No. Moving all thinking control into `options` is not the best fix because Ollama currently reads `think` from the top-level chat request; a safer fix should capability-gate or clamp top-level truthy `think` for non-thinking models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve top-level think for supported Ollama models:** `extensions/ollama/src/stream.ts:239`
  - body: Ollama's current API reads thinking control from top-level `ChatRequest.Think`; `options` is parsed as `api.Options`, which has no `think` field and ignores unknown keys. After this change, `/think off|low|medium|high` and configured `params.thinking` no longer affect thinking-capable Ollama models, so users cannot disable or set the reasoning effort. Please gate or clamp top-level `think` for non-thinking models instead of moving it wholesale into `options`.
  - confidence: 0.92
