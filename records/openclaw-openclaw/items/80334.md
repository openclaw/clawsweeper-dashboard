## Summary

Keep this PR open: the patch conflicts with Ollama's current chat request contract for native thinking control and the contributor supplied only unit-test proof, not live Ollama behavior proof.

## What This Changes

The PR moves Ollama `think` values from the top-level `/api/chat` payload into `options.think` and updates Ollama stream tests to expect that placement.

## Best Possible Solution

Keep top-level `think` for reasoning-capable Ollama models and fix non-thinking failures with capability-aware validation or suppression backed by live Ollama proof.

## Reproduction Assessment

No, not as a complete live current-main reproduction. Source inspection shows current main can emit top-level truthy `think` and upstream Ollama rejects truthy `req.Think` for non-thinking models, but the exact OpenClaw default-thinking route still needs live/config confirmation.

## Solution Assessment

No. Moving all native Ollama thinking control into `options.think` is not the narrowest maintainable fix because upstream Ollama reads native thinking from top-level `ChatRequest.Think`; a capability-aware guard is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep native Ollama think on the chat request field:** `extensions/ollama/src/stream.ts:242`
  - body: Ollama's chat request contract reads native thinking from top-level `ChatRequest.Think`, while `Options` is runner/model options and does not define `think`. Moving runtime and configured values into `options.think` can make `/think off` and effort controls ineffective for reasoning-capable Ollama models, so preserve top-level `think` for supported models and guard the non-thinking path instead.
  - confidence: 0.9
