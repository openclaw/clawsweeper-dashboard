## Summary

Keep this PR open, but block merge: it addresses a real Ollama 400 path, but the blanket move to `options.think` conflicts with Ollama's current chat request contract and would regress native reasoning-model thinking controls. The supplied proof is only mocked/unit-test proof, not a real Ollama after-fix run.

## What This Changes

The PR moves Ollama native `think` request parameters from the top-level chat payload into `options.think` and updates the Ollama stream tests accordingly.

## Best Possible Solution

Keep Ollama `think` on the upstream top-level field for models that support thinking, and prevent truthy top-level `think` from reaching non-thinking models using catalog capability/validation plus live Ollama proof.

## Reproduction Assessment

Yes, partially from source: current main can emit top-level truthy `think`, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact default-thinking route from the report still needs live/config confirmation because current OpenClaw also has supported-level validation.

## Solution Assessment

No. Moving all native Ollama thinking control into `options.think` is not the narrowest maintainable fix because upstream Ollama reads `think` from the top-level chat request; the safer fix is capability-aware guarding while preserving top-level `think` for reasoning models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep native Ollama thinking on the chat request field:** `extensions/ollama/src/stream.ts:242`
  - body: Ollama's native chat API reads thinking control from top-level `ChatRequest.Think`; `options` is modeled as runner/predict options and has no `think` field.
