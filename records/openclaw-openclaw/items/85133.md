## Label Justifications

- `P1`: The reported failure leaves the macOS gateway down after update until manual launchctl recovery, breaking real channel and automation workflows.
- `impact:crash-loop`: The issue is about process-level availability after launchd supervision is lost.
- `impact:message-loss`: When the gateway is down, Telegram and other integrations can silently stop responding.

## Summary

Current main now addresses the central launchd deregistration failure: the reload handoff waits for bootout completion before re-bootstrap and has a fallback path, while the update flow already rechecks health and re-bootstraps installed-but-unloaded LaunchAgents. The fix is on main only after the latest release, so users still on v2026.5.20 need the next release or a build from current main.

## What This Changes

The issue reports that macOS self-update can rewrite the gateway LaunchAgent plist but leave `ai.openclaw.gateway` unloaded, causing the gateway to stay down until manual `launchctl bootstrap`.

## Best Possible Solution

Keep the current main launchd reload hardening and ship it in the next release; any recurrence after that should be filed with fresh `gateway-restart.log`, `launchctl print`, and `/healthz` evidence.

## Reproduction Assessment

Yes from source and reported logs, but not as a live current-main failure: the affected release path bootout/bootstrap race is documented in the related report and current main now changes that exact reload handoff. I did not run a macOS launchd self-update reproduction in this read-only review.

## Solution Assessment

Yes: fixing the launchd reload handoff and preserving the existing post-update health/re-bootstrap checks is the narrow maintainable repair for the reported missing registration. The remaining operator-facing improvements such as alerts or extra exit logging can be handled separately if still desired.
