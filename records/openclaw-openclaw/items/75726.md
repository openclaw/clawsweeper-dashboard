## Summary

Keep open. The reported header/notice leak has a shipped preface mitigation, but #75755 still references this issue with closing syntax and current source inspection shows the async-completion wrapper body named in the report is still emitted and not covered by the preface-only stripper.

## What This Changes

The issue requests that user-facing sanitization strip the modern OpenClaw runtime-context wrapper variant when models echo it into Discord, Slack, Telegram, or similar channel output.

## Best Possible Solution

Review #75755 against the shipped preface mitigation and either land an equivalent narrow shared-sanitizer regression fix for echoed async/body wrappers or explicitly close that PR if maintainers accept the current mitigation as sufficient.

## Reproduction Assessment

Yes. A high-confidence source-level path is to pass the modern header/notice followed by the async-completion prompt body and a visible reply through `sanitizeUserFacingText`; current main removes the preface but leaves the unmarked async body paragraphs.

## Solution Assessment

Yes, the shared sanitizer/runtime-context helper is the right layer because multiple channel and display surfaces use it. The open question is whether #75755's broader paragraph consumption is safer than keeping the shipped preface-only mitigation.
