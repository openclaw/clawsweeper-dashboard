## Summary

Keep open. Current main has several useful Windows plugin-loader mitigations, including native loading on Windows, native-error fallback, and root-alias fast-path cleanup, but I still did not find direct current-main proof that the packaged `minimax`, `google`, and `talk-voice` dist entrypoints load on Windows under the default Node stack.

## What This Changes

The issue reports Windows packaged bundled plugins failing to load at Gateway startup with `RangeError: Maximum call stack size exceeded` in OpenClaw v2026.4.26.

## Best Possible Solution

Prove or fix the packaged runtime path with a Windows package smoke that loads the three named dist entrypoints under default Node stack, then close with that run evidence or land the narrow loader/build change it exposes.

## Reproduction Assessment

Yes for the reported v2026.4.26 release: the issue gives exact environment, affected plugin ids, logs, and a stack-size-controlled check. Unclear for current main because the relevant fixes are generic and no exact Windows packaged-entrypoint smoke was found for all three plugins.

## Solution Assessment

Unclear. The native-require, fallback, root-alias, and activation changes are plausible mitigations, but the best closure path is direct Windows package proof for the reporter's exact dist entrypoints rather than inferring from adjacent startup traces.
