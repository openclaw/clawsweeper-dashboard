## Summary

Keep open. Current main still has Feishu outbound paths that send visible text separately before voice media, so the reported duplicate-message behavior remains valid and is not covered by the narrower TTS-only duplicate fix discussed in #70092.

## What This Changes

Request suppressing redundant Feishu text delivery when a reply sends voice audio through `MEDIA:` or `audioAsVoice`, while preserving normal media captions.

## Best Possible Solution

Add a narrow Feishu outbound guard that treats voice-note media as media-only for visible delivery, with regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. By source inspection, invoking Feishu `sendMedia` with non-empty text, a media URL, and `audioAsVoice: true` reaches `sendOutboundText` before `sendMediaFeishu` on current main.

## Solution Assessment

Yes, if scoped narrowly. The maintainable fix is in Feishu outbound/reply delivery: suppress separate text only for voice-note media while leaving non-voice captions and comment-thread fallback behavior intact.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76440 in the Feishu plugin only. Current main sends a separate visible text message before voice media when `text` and `mediaUrl` are combined with `audioAsVoice: true` or a native voice-audio media path. Suppress that separate visible text only for Feishu voice-note media, while preserving normal captions for non-voice media, comment-thread text fallback, reply/thread targeting, media failure fallback, and existing `audioAsVoice` forwarding. Cover both direct `feishuOutbound.sendMedia` and the reply dispatcher/directive delivery path. Add focused regression tests in `extensions/feishu/src/outbound.test.ts` and `extensions/feishu/src/reply-dispatcher.test.ts`; keep `extensions/feishu/src/media.ts` behavior intact unless a small helper is needed. Add a single-line user-facing `CHANGELOG.md` fix entry. Do not move this policy into core or change other channel behavior.
