## Summary

Keep open. Current main still has Feishu paths where a voice media reply with text can send visible text separately from the media, #76450 is an open unmerged implementation PR tied to this issue, and the later testing comment reports a possible extra voice-send path that still needs confirmation.

## What This Changes

The issue requests deduplicating Feishu voice replies so text plus voice media does not produce separate text and duplicate voice/audio messages.

## Best Possible Solution

Review and either land or revise #76450 with regression tests for direct `sendMedia` and reply-dispatcher voice-note delivery, then investigate the newer single-directive duplicate-voice report if it remains reproducible.

## Reproduction Assessment

Yes by source inspection for the original bug: a Feishu `sendMedia` payload with non-empty text, `mediaUrl`, and `audioAsVoice: true` reaches the text send before `sendMediaFeishu`. The later 1 text plus 2 voice report is not fully reproduced without Feishu logs or a live trace.

## Solution Assessment

Unclear for the full issue: suppressing visible text for voice-note media is the narrow fix for the original split, but the newer single-directive duplicate-voice report may require broader pipeline tracing before #76450 is sufficient.
