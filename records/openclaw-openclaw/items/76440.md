## Summary

Keep open. Current main still has a Feishu path where a voice media reply with text can produce separate visible text and media sends, and the related TTS-tool fix in #70092 does not cover this adapter/directive sendMedia path.

## What This Changes

The issue requests suppressing Feishu's separate text send when an outbound media reply is a voice/audio message.

## Best Possible Solution

Suppress the separate visible text only for Feishu voice-note media, while preserving captions for non-voice media and comment targets, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes by source inspection: a Feishu `sendMedia` call with non-empty text, `mediaUrl`, and `audioAsVoice: true` reaches `sendOutboundText` before `sendMediaFeishu`. I did not run a live Feishu API send in this read-only review.

## Solution Assessment

Yes, if scoped narrowly: the maintainable fix is in Feishu outbound/reply delivery, suppressing text only when the media is actually a voice-note send and leaving normal media captions unchanged.
