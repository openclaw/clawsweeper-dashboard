## Label Justifications

- `P2`: This is a meaningful multi-channel continuity feature with clear user value, but it is not an urgent regression or outage.
- `impact:session-state`: The request changes which voice, channel, webhook, and CLI turns share transcripts, context, queue state, and session lifecycle.

## Summary

Keep open: current main has explicit session-key entry points and DM identity links, but it still lacks identity-anchored routing across non-DM, voice, webhook, and programmatic inputs; the remaining work needs a maintainer product/security design rather than cleanup closure.

## What This Changes

Request identity-based session unification so the same user can share one OpenClaw session across voice, Telegram, WhatsApp, webhooks, and programmatic CLI injection.

## Best Possible Solution

Design one opt-in canonical session alias or identity-routing contract with validated identities, isolated defaults, origin-preserving delivery metadata, docs, config/schema coverage, and coordinated scope with the open shared-session trackers.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source and docs give a high-confidence feature-gap check: current routing only links direct-message peers and cannot express identity anchoring for voice, webhook, or programmatic inputs.

## Solution Assessment

Unclear as proposed: explicit --session-key is already supported, but an automatic identityAnchor map should be folded into a maintainer-approved session alias/routing design rather than added as a one-off config surface.
