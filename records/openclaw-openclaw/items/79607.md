## Summary

Keep open: current main has lower-level `sessionKey` routing for Gateway agent calls, hooks, SDK/API callers, and voicewake targets, but it still lacks the requested `openclaw agent --session-key` CLI surface and a general identity-anchored routing config for voice, webhooks, programmatic injection, and chat channels. The related issues and open implementation PR cover adjacent channel-group or binding-level sharing, but they do not supersede this issue's programmatic injection plus identity-anchor scope.

## What This Changes

The issue requests identity-based session routing plus an `openclaw agent --session-key` CLI override so one user can share a session across voice, Telegram, WhatsApp, and programmatic input.

## Best Possible Solution

Design an explicit, opt-in session identity or alias contract that extends current `dmScope`/`identityLinks`, exposes a guarded CLI/API way to target stable sessions, preserves isolated defaults, and documents the privacy and trust-boundary tradeoffs.

## Reproduction Assessment

Not applicable: this is a feature/config request rather than a bug against an existing contract. Source and docs inspection on current main gives high confidence that the requested CLI flag and identity-anchor config do not exist today.

## Solution Assessment

Unclear as a final implementation path: the requested behavior is valid, but the raw `sessionKey` and identity-anchor shapes need maintainer-approved routing, validation, and privacy semantics before code work.
