## Summary

Keep open: current main and the latest release still only partially cover this request. Lower-level Gateway, hooks, and voicewake paths can accept explicit session keys, but `openclaw agent` still has no `--session-key` CLI surface on main and there is no identity-anchored routing config for voice, webhooks, programmatic injection, and channel traffic.

## What This Changes

Request identity-based session routing so one user can share a stable agent session across Telegram, voice, WhatsApp, webhooks, and programmatic CLI injection.

## Best Possible Solution

Design an additive, opt-in identity/session-target contract that extends current `dmScope` and `identityLinks`, then land the CLI `--session-key` piece with shared validation while preserving isolated defaults.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug against an existing contract. Source and docs inspection give high confidence that the requested CLI flag and identity-anchor config are absent on current main.

## Solution Assessment

Unclear as an implementation plan: the CLI flag fits existing lower-level plumbing, but the broader identity-anchored routing needs maintainer-approved privacy and routing semantics before a core change.
