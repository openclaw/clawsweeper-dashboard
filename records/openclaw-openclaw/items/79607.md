## Label Justifications

- `P2`: This is a meaningful multi-channel continuity feature with clear user impact, but it is not an urgent regression or outage.
- `impact:session-state`: The request changes which voice, channel, webhook, WebChat, and CLI turns share transcripts, context, queue state, and lifecycle.
- `impact:security`: Identity-based merging can expose private context across transports unless authorization and trust boundaries are explicit.

## Summary

Keep open: current main and v2026.6.8 support explicit session-key entry points plus DM identity links, but they still do not provide identity-anchored routing across non-DM, voice, webhook, WebChat, or programmatic inputs.

## What This Changes

Request identity-anchored routing so the same user can share one OpenClaw session across voice, Telegram, WhatsApp, WebChat/programmatic, and other input surfaces.

## Best Possible Solution

Design an additive, opt-in identity-routing/session-anchor contract with explicit trust boundaries, validated session targets, origin-preserving delivery metadata, schema/docs updates, and coordination with #92653, #19929, and #7524.

## Reproduction Assessment

Yes at source level: current routing applies dmScope and identityLinks only to direct-message peers, while CLI, HTTP, and voice paths provide explicit session keys but no general identity-anchor map.

## Solution Assessment

Unclear as proposed: --session-key is already present and useful, but a config identityAnchor requires maintainer design for identity proof, non-DM surfaces, and safe origin/delivery handling.
