## Summary

Keep open: current main and v2026.6.10 cover explicit CLI session-key injection and direct-message identity links, but they still lack the requested identity-anchored routing contract across voice, webhook, WebChat, group/channel, and programmatic input surfaces.

## What This Changes

The issue requests one canonical user session across input channels, including an explicit `openclaw agent --session-key` path and a config-driven identity-to-session anchor.

## Best Possible Solution

Design one additive, opt-in identity/session routing contract with validated identity proof, canonical session targets, origin-preserving delivery metadata, docs, security guidance, and tests that reconcile the related shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current main deterministically supports direct-message identity links and explicit session-key injection, but the strict schema and routing path cannot express an identity-to-canonical-session anchor for voice, hook, WebChat, group/channel, or arbitrary programmatic inputs.

## Solution Assessment

Unclear as proposed. `openclaw agent --session-key` is already the right narrow injection path, but the identity-anchor config needs a maintainer product/security decision and should be reconciled with the shared-session and named-inbound-session designs before implementation.
