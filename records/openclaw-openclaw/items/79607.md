## Summary

Keep open: current main still lacks the requested `openclaw agent --session-key` CLI surface and a general identity-anchored session-routing config, even though lower-level Gateway and hook `sessionKey` paths exist.

## What This Changes

Request identity-based session routing so one user can share a single session across voice, Telegram, WhatsApp, webhooks, and programmatic `openclaw agent` injection.

## Best Possible Solution

Design a guarded session-alias or identity-routing contract that extends current `dmScope`/`identityLinks`, exposes a stable CLI/API target, preserves isolated defaults, and documents the cross-surface privacy boundary.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug against an existing contract. Source and docs inspection confirms the requested CLI flag and identity-anchor config are absent on current main.

## Solution Assessment

Unclear as proposed: a stable session-key selector is plausible, but the identity-anchor shape and raw routing semantics need maintainer-approved validation, docs, and privacy policy before implementation.
