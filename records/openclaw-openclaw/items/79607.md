## Summary

Keep open: current main still lacks both requested surfaces, while existing lower-level sessionKey plumbing and related channel-group trackers do not cover identity-anchored routing across voice, CLI injection, hooks, and messaging channels.

## What This Changes

The issue requests identity-based session routing so one user can share a canonical OpenClaw session across Telegram, voice, WhatsApp, webhooks, and CLI injection.

## Best Possible Solution

Design an additive opt-in session identity or alias-routing contract that extends current dmScope/identityLinks, exposes a guarded CLI/API route to stable session keys, and preserves isolated defaults unless explicitly configured.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a bug against an existing contract. Source and docs inspection on current main confirms the requested CLI flag and identity-anchor config do not exist.

## Solution Assessment

Unclear as a concrete implementation: --session-key is a plausible narrow CLI addition, but identityAnchor needs a maintainer-approved routing and privacy contract across CLI, Gateway RPC, hooks, voicewake, and channel routing.

## Security Review

Status: needs_attention

Summary: The requested feature is privacy-sensitive because it intentionally merges user context across channels and input surfaces.

Concerns:

- **[medium] Define merged-context disclosure:** `docs/concepts/session.md:28`
  - body: Session docs already warn that shared DM context can expose one user's messages to another; identity-anchored cross-surface routing needs equivalent opt-in semantics and operator-facing disclosure.
  - confidence: 0.9
- **[medium] Treat sessionKey as routing, not authorization:** `docs/gateway/security/index.md:108`
  - body: Gateway security docs state that sessionKey is routing/context selection rather than per-user auth, so any identity-to-session mapping must avoid implying an authorization boundary.
  - confidence: 0.86
- **[medium] Define runtime posture for merged groups and channels:** `docs/channels/groups.md:144`
  - body: Group docs currently rely on group sessions using non-main keys for sandbox posture; routing those identities into main or shared sessions needs explicit tool and sandbox semantics.
  - confidence: 0.84
