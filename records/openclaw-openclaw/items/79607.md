## Summary

Keep open. Current main partially covers this with explicit session keys and direct-message identity links, but identity-anchored routing for non-DM, voice, webhook, and programmatic inputs is still missing and needs maintainer product/security design.

## What This Changes

Request one canonical user session across channels and programmatic or voice inputs, likely through explicit session-key injection plus identity-anchored routing configuration.

## Best Possible Solution

Design a generalized, security-reviewed session alias or identity routing model that builds on identityLinks, explicit sessionKey, and Voice Call sessionKey support without adding an unsafe parallel config path.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request. Source and docs do confirm the current behavior that non-DM, webhook, and voice surfaces do not automatically route through a shared identity anchor.

## Solution Assessment

Unclear until maintainers choose the product boundary. The explicit --session-key path is a good supported workaround, but an automatic identityAnchor-style config needs a broader session-routing and security design rather than a narrow patch.
