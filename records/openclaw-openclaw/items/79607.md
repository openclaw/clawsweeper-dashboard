## Summary

Keep open: current main supports explicit session-key entry points and DM identity links, but it still lacks the requested identity-anchored routing contract across voice, webhook, WebChat, group/channel, and programmatic input surfaces.

## What This Changes

The issue requests identity-based session unification so one known user can share a single OpenClaw session across channels and programmatic inputs.

## Best Possible Solution

Design an additive, opt-in identity-routing contract with explicit trust boundaries, validated session targets, origin-preserving metadata, schema/docs updates, and coordination with the related shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current routing applies identityLinks only to direct-message scopes and the strict session schema has no identityAnchor or general inbound session target.

## Solution Assessment

Unclear as proposed. The explicit CLI --session-key part is already present and useful, but the identityAnchor config requires maintainer design for identity proof, non-DM surfaces, and safe origin/delivery handling.
