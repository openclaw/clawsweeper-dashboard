## Summary

Keep open: current main has lower-level `sessionKey` paths for Gateway agent calls, hooks, and voicewake routing, but it still lacks the requested `openclaw agent --session-key` CLI surface and the proposed identity-anchored routing config.

## What This Changes

Add stable session-key and identity-anchored routing so programmatic, voice, and chat inputs from the same known user can land in one canonical session.

## Best Possible Solution

Design a core session-alias or identity-routing contract that extends current `dmScope`/`identityLinks` and exposes a guarded CLI/API path while preserving safe per-channel defaults.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug against an existing contract. Source and docs inspection confirms the CLI flag and identity-anchor config do not exist today.

## Solution Assessment

No final implementation path is established yet: the proposed shape is plausible, but the safest fix needs a maintainer-approved session identity contract across CLI, Gateway RPC, hooks, voicewake, and channel routing.
