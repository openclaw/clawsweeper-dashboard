## Summary

Keep open: current main has lower-level `sessionKey` plumbing for Gateway agent calls and voicewake routing, but it still lacks the requested `openclaw agent --session-key` CLI surface and any identity-anchored session routing config. The related group/channel trackers are useful context, but they do not fully cover this issue's programmatic injection plus identity-anchor scope.

## What This Changes

The issue requests identity-based session unification so one user can share a canonical agent session across voice, CLI/programmatic injection, Telegram, WhatsApp, and other input channels.

## Best Possible Solution

Design an additive identity/session-alias routing contract that extends the existing `dmScope` and `identityLinks` model, exposes a guarded CLI/API path for explicit session targeting, preserves isolated defaults, and documents privacy boundaries.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug against an existing contract. Source and docs inspection on current main confirm the requested CLI flag and identity-anchor config are absent while lower-level session-key paths already exist.

## Solution Assessment

Unclear as a concrete implementation: the proposed `--session-key` flag is plausible, but identity-anchored routing needs a maintainer-approved session identity contract across CLI, Gateway RPC, hooks, voicewake, routing, docs, and privacy policy.
