## Summary

Keep open: current main still has only partial lower-level session-key plumbing, while the requested `openclaw agent --session-key` selector and identity-anchored routing config are absent. Related open items cover binding/channel grouping, but they do not fully replace this issue's programmatic injection, voice input, and identity-anchor scope.

## What This Changes

The issue requests identity-based session unification so programmatic, voice, Telegram, WhatsApp, and other inputs from the same known user route into one canonical session.

## Best Possible Solution

Design an opt-in core session alias or identity-routing contract that extends `dmScope`/`identityLinks`, exposes a guarded CLI/API path, and preserves isolated defaults unless an operator explicitly links surfaces.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug against an existing contract. Source and docs inspection on current main gives high confidence that the requested CLI flag and identity-anchor config surface do not exist today.

## Solution Assessment

Unclear as a final implementation shape. A raw `sessionKey` CLI escape hatch and identity-anchor map are plausible, but the maintainable solution needs a validated session-routing contract and explicit privacy policy across CLI, Gateway, hooks, voicewake, and channel routing.
