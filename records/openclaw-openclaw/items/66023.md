## Summary

Keep open. Current main still lacks add-time custom cron job IDs, and this PR is a useful implementation candidate for the open feature request, but the proposed patch leaves a concrete case-variant run-log aliasing bug before merge.

## What This Changes

The PR adds optional `--id` support to `openclaw cron add`, threads the field through CLI params, normalization, Gateway cron schema, cron create types/service logic, and focused tests for custom ID validation and duplicate rejection.

## Best Possible Solution

Land the feature after making custom cron job IDs canonical for file-backed state, preferably by accepting only lowercase slug IDs or rejecting case-insensitive duplicates before persistence, then update the CLI/protocol/service tests, public cron docs, and `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path for the missing feature: `cron add` exposes no `--id`, the Gateway add schema rejects unknown `id`, and the service always generates a UUID. The PR-specific bug is reproducible by creating `Daily` and `daily` on a case-insensitive filesystem and observing that both resolve to the same run-log file name.

## Solution Assessment

No. The PR’s end-to-end direction is right, but the current patch is not the best final shape because the new user-controlled file-backed identifier is not canonicalized or checked case-insensitively, and the user-facing docs/changelog coverage is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-variant cron job IDs:** `src/cron/service/jobs.ts:519`
  - body: `createJob` only rejects exact duplicate IDs, but the new custom ID pattern allows uppercase letters. On case-insensitive filesystems, `Daily` and `daily` both pass this check and resolve to the same `runs/<jobId>.jsonl` run-log path, so run history writes and pruning can collide between jobs. Canonicalize IDs or reject case-insensitive duplicates before accepting the job.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a low-severity local history integrity concern by allowing user-controlled case-variant cron IDs to alias the same run-log file on common case-insensitive filesystems.

Concerns:

- **[low] Prevent case-variant run-log aliasing:** `src/cron/service/jobs.ts:519`
  - body: Custom cron IDs are user-controlled and may include uppercase letters; because run logs are stored as `runs/<jobId>.jsonl`, case-variant IDs can share one underlying file on case-insensitive filesystems and mix or corrupt per-job run history.
  - confidence: 0.86
