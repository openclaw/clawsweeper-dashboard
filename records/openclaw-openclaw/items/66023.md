## Summary

Keep open. The feature targets a real missing cron capability, but this PR is not ready to merge because the diff leaves case-variant run-log aliasing, duplicate-ID Gateway error mapping, generated protocol clients, branch freshness, and contributor real behavior proof unresolved.

## What This Changes

The PR adds an optional `--id`/`id` field for cron job creation across the CLI, normalization, Gateway schema, service creation, and focused tests.

## Best Possible Solution

Land one custom cron-ID implementation that validates or canonicalizes file-safe IDs, maps duplicate collisions as `INVALID_REQUEST`, updates generated clients and docs, keeps UUIDs as the default, and includes redacted real CLI or Gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows no `cron add --id`, no Gateway `cron.add` `id` field, and service creation still using `crypto.randomUUID()` unconditionally; the PR blockers are also source-reproducible from the diff and adjacent code.

## Solution Assessment

No. The end-to-end direction is maintainable, but this patch is not the best merge shape until it fixes case-safe ID collisions, Gateway duplicate-error mapping, generated protocol clients, branch freshness, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-variant cron job IDs:** `src/cron/service/jobs.ts:519-520`
  - body: `createJob` only checks exact ID equality, but the new pattern allows uppercase letters. On case-insensitive filesystems, `Daily` and `daily` can resolve to the same `runs/<jobId>.jsonl` file and mix run history, so custom IDs need canonical casing or case-insensitive collision checks before persistence.
  - confidence: 0.9
- **[P2] Return duplicate IDs as invalid requests:** `src/cron/service/jobs.ts:520-521`
  - body: The duplicate guard throws a plain `Error`, while Gateway `cron.add` only converts `TypeError` and `RangeError` from `context.cron.add` into `INVALID_REQUEST`. A duplicate ID through Gateway or CLI can surface as an internal failure instead of the promised validation response.
  - confidence: 0.86
- **[P2] Regenerate cron protocol clients:** `src/gateway/protocol/schema/cron.ts:293`
  - body: Adding `id` to `CronAddParamsSchema` changes the Gateway wire contract, but the committed Swift `CronAddParams` model still has no `id` property. Typed clients cannot send the new field until the generated protocol artifacts are refreshed.
  - confidence: 0.86
