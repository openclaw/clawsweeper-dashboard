## Summary

Keep open. Current main still lacks supported `cron add --id` behavior, and this PR is relevant, but it needs real behavior proof plus focused fixes before merge.

## What This Changes

The PR adds optional `--id` support to `cron add`, threads the id through normalization, Gateway validation, and cron service creation, and adds focused tests for validation and duplicate rejection.

## Best Possible Solution

Land one end-to-end custom cron id implementation after canonical/path-safe validation, correct Gateway error mapping, regenerated protocol clients, and redacted real CLI/runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--id` option, a Gateway add schema that rejects top-level `id`, a create type that omits `id`, and a service create path that always calls `crypto.randomUUID()`. The PR-specific blockers are also source-reproducible from the exact-case duplicate check, run-log filename pathing, Gateway error mapping, and missing generated Swift model updates.

## Solution Assessment

No. The end-to-end direction is useful, but this patch is not the best merge shape until duplicate ids are collision-safe, duplicate errors map cleanly through Gateway/CLI, generated protocol clients are updated, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-variant cron job IDs:** `src/cron/service/jobs.ts:519`
  - body: `createJob` only rejects exact `job.id === id`, but the new custom id pattern allows uppercase letters. On case-insensitive filesystems, `Daily` and `daily` can resolve to the same `runs/<jobId>.jsonl` file, mixing or clobbering run history between jobs.
  - confidence: 0.9
- **[P2] Refresh generated cron protocol clients:** `src/gateway/protocol/schema/cron.ts:293`
  - body: Adding `id` to `CronAddParamsSchema` changes the Gateway wire contract, but the committed Swift `CronAddParams` model is not updated in this PR and still has no `id` field. Typed native clients cannot send the new value until the generated protocol artifacts are refreshed.
  - confidence: 0.86
- **[P2] Return duplicate IDs as validation errors:** `src/cron/service/jobs.ts:519-520`
  - body: The duplicate-id branch throws a plain `Error`, while the Gateway `cron.add` handler rethrows non-`TypeError` and non-`RangeError` failures from `context.cron.add`. A duplicate id through Gateway or CLI will bypass the `INVALID_REQUEST` validation response instead of returning the explicit user-facing error promised by the PR.
  - confidence: 0.85
