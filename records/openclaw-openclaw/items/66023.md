## Summary

## What This Changes

The branch adds optional `--id` support to `openclaw cron add`, threads the value through cron normalization, Gateway validation, cron create types/service logic, exact duplicate rejection, and focused tests.

## Best Possible Solution

Land one canonical custom cron-ID implementation that keeps omitted IDs on UUIDs, validates a conservative canonical ID format, rejects collisions consistently through Gateway/CLI, updates generated protocol clients and docs, and includes focused tests plus redacted real runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `cron add --id`, rejects top-level `id` in `CronAddParamsSchema`, omits `id` from `CronJobCreate`, and always calls `crypto.randomUUID()` in `createJob`; the PR-specific blockers are also source-reproducible from the diff and current run-log/Gateway handler code.

## Solution Assessment

No. The end-to-end direction is reasonable, but this patch is not the best merge shape until IDs are case-safe, duplicate IDs return a validation response, generated protocol clients are updated, and contributor real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-variant cron job IDs:** `src/cron/service/jobs.ts:519`
  - body: `createJob` rejects only exact `job.id === id`, while the new custom-ID pattern accepts uppercase letters. On case-insensitive macOS/Windows filesystems, `Daily` and `daily` can both be created but resolve to the same `runs/<jobId>.jsonl` file, mixing run history; canonicalize IDs or reject case-insensitive duplicates before persisting.
  - confidence: 0.9
- **[P2] Regenerate the cron add protocol clients:** `src/gateway/protocol/schema/cron.ts:293`
  - body: Adding `id` to `CronAddParamsSchema` changes the Gateway wire contract, but the PR does not update generated Swift protocol models, and current `CronAddParams` still has no `id` property. Typed clients cannot send the new field until the generated artifacts are refreshed.
  - confidence: 0.84
- **[P2] Map duplicate IDs as invalid cron.add requests:** `src/cron/service/jobs.ts:519`
  - body: The duplicate guard throws a plain `Error`, but the Gateway `cron.add` handler maps only `TypeError` and `RangeError` from `context.cron.add` to `INVALID_REQUEST` and rethrows other failures. Duplicate IDs through Gateway or CLI will be treated as server failures instead of the explicit validation error promised by the PR.
  - confidence: 0.86
