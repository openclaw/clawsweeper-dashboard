## Summary

Keep open. Current main still lacks the requested custom cron job ID create path, and this PR is a plausible implementation candidate, but it is not merge-ready because the new user-controlled ID can alias run-log files on case-insensitive filesystems and the Gateway protocol artifacts are not fully synchronized.

## What This Changes

The PR adds optional `--id` support to `openclaw cron add`, threads the value through normalization, Gateway cron validation, cron create types/service logic, and focused tests for validation and duplicate rejection.

## Best Possible Solution

Land a single custom-ID implementation after making IDs canonical and file-safe, updating generated protocol clients and focused tests, and adding real CLI or runtime proof with private details redacted.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the missing feature because the CLI has no `--id`, the Gateway add schema rejects `id`, `CronJobCreate` omits `id`, and `createJob` always generates a UUID; the PR-specific run-log collision is source-reproducible from the exact-case duplicate check plus `runs/<jobId>.jsonl` pathing.

## Solution Assessment

No. The end-to-end direction is right, but the current patch is not the best final shape because it accepts uppercase IDs without case-insensitive collision protection and changes the Gateway schema without updating generated client models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-variant cron job IDs:** `src/cron/service/jobs.ts:519`
  - body: `createJob` only rejects `job.id === id`, but the new custom ID pattern allows uppercase letters. On case-insensitive filesystems, `Daily` and `daily` both pass this check and resolve to the same `runs/<jobId>.jsonl` run-log file, so run history can be mixed or overwritten. Canonicalize custom IDs or reject case-insensitive duplicates before persisting.
  - confidence: 0.9
- **[P2] Regenerate the cron add protocol clients:** `src/gateway/protocol/schema/cron.ts:293`
  - body: Adding `id` to `CronAddParamsSchema` changes the Gateway wire contract, but the committed generated `CronAddParams` model in `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift` still has no `id` field. Typed clients cannot send the new value through that model until the generated protocol artifacts are refreshed.
  - confidence: 0.82
