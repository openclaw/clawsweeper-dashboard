## Summary

Keep open. Current main still lacks custom cron job IDs on `cron add`, and this PR remains the focused implementation path for the open feature request, but the diff needs a case-insensitive ID collision guard before merge because run logs are stored under `runs/<jobId>.jsonl`.

## What This Changes

The PR adds optional `--id` support to `openclaw cron add`, threads the field through CLI params, normalization, Gateway cron schema, cron create types/service logic, and focused tests for safe IDs and duplicate rejection.

## Best Possible Solution

Land the feature after making custom cron job IDs canonical for file-backed state, preferably by lowercasing the accepted slug or rejecting case-insensitive duplicates before persistence, then document `openclaw cron add --id` and add the required changelog entry.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path for the missing feature: the CLI has no `--id`, the Gateway schema rejects it, and the service always generates a UUID. The PR-specific collision risk is reproducible by creating case-variant IDs on a case-insensitive filesystem and observing that both map to the same run-log pathname.

## Solution Assessment

No. The end-to-end direction is right, but the current patch is not the best final shape because it leaves case-variant run-log aliasing unresolved and omits the expected user-facing docs/changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject case-variant cron id duplicates:** `src/cron/service/jobs.ts:519`
  - body: `createJob` only rejects exact duplicate IDs, but the new pattern allows uppercase letters. On case-insensitive filesystems, `Daily` and `daily` both pass this check yet resolve to the same `runs/<jobId>.jsonl` run-log file, so writes/pruning/history can collide between jobs. Canonicalize custom IDs or reject duplicates case-insensitively before accepting the job.
  - confidence: 0.9
