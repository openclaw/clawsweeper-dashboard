## Summary

Keep open: this PR addresses a real memory-core rollback/cache loss, but the current head still drops current main's temp-DB cleanup callback, so failed safe reindexes can clean up temp sqlite files before the open handle is closed.

## What This Changes

The PR snapshots and restores memory-core reindex retry state, mirrors successful embedding cache batches across safe reindex rollback, adds focused regression tests, and updates the changelog.

## Best Possible Solution

Land the rollback recovery design after wiring an idempotent temp DB close through `beforeTempCleanup`; keep transport retry and batch-splitting work in https://github.com/openclaw/openclaw/issues/44166.

## Reproduction Assessment

Yes. Current main is source-reproducible for the cache-loss/retry-state gap, and the PR body supplies A/B terminal proof showing 0 cache rows on main versus 109 preserved rows on the branch after an interrupted force reindex.

## Solution Assessment

Mostly yes. Snapshotting rollback-relevant state and mirroring successful cache batches is a narrow maintainable fix, but this PR head must restore the current main `beforeTempCleanup` cleanup ordering before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Close the temp DB before atomic cleanup:** `extensions/memory-core/src/memory/manager-sync-ops.ts:1270-1273`
