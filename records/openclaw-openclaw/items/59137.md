## Summary

Keep open. Current main still has the rollback damage this PR targets: safe reindex restores only DB/vector/FTS handles after failure, non-batch embedding cache writes are deferred until every missing batch succeeds, and session retry state has no full-retry sentinel; the source PR is useful but still needs to be ported/rebased onto current main and given the required changelog entry.

## What This Changes

The PR changes memory-core reindex rollback to preserve retry state, keep full-session retries pending, mirror successful embedding-cache writes during safe reindex, and add focused regression coverage.

## Best Possible Solution

Port the rollback/cache behavior into current memory-core helpers, keep the scope limited to rollback retry state and embedding-cache durability, add focused regression coverage, and add a one-line CHANGELOG.md Fixes entry with appropriate attribution.

## Reproduction Assessment

Yes. The PR body gives a concrete offline force-reindex reproduction, and current-main source inspection confirms the two key mechanics: delayed cache persistence and no full-session retry sentinel after rollback.

## Solution Assessment

Yes, with adaptation. Snapshotting retry state and mirroring successful cache batches is the narrow maintainable fix; the safer path is to rebase/port it onto current main and leave broader transport retry work to #44166.
