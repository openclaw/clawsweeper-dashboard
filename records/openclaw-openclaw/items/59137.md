## Summary

Keep open: the reported memory-core rollback bug is real and this PR carries useful source work, but the current branch is not merge-ready because it misses current-main temp-DB cleanup handling and leaves the batch-mode cache path outside the new mirror write.

## What This Changes

The PR adds memory-core reindex rollback recovery for sync retry state, a full-session retry sentinel, per-batch embedding-cache mirroring during safe reindex, an older-cache-table guard, focused tests, and a changelog entry.

## Best Possible Solution

Repair the two narrow memory-core gaps, keep the existing no-new-config/provider-API boundary, then rerun the memory-core tests and changed gate before maintainer review.

## Reproduction Assessment

Yes. Source inspection on current main shows delayed direct cache persistence and no rollback-state snapshot/restore, matching the PR body's real offline force-reindex reproduction.

## Solution Assessment

No, not yet. The direction is right, but the best fix must also preserve current main's `beforeTempCleanup` temp-handle close path and route batch-mode cache success writes through the same mirror helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve temp DB cleanup before rollback removal:** `extensions/memory-core/src/memory/manager-sync-ops.ts:1270-1273`
  - body: Current main passes `beforeTempCleanup` into `runMemoryAtomicReindex` so the temp SQLite handle is closed before the helper removes `*.tmp` files. This rewritten safe reindex path omits that callback, so on platforms that reject removing open SQLite files the cleanup can fail with an aggregate error before normal rollback recovery, leaving temp files and obscuring the original failure. Reintroduce the current `tempDbClosed`/`beforeTempCleanup` handling around the new snapshot and mirror logic.
  - confidence: 0.86
- **[P2] Mirror cache writes from batch mode too:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:293-300`
  - body: The new mirror write is only used by `embedChunksInBatches`; when `providerRuntime.batchEmbed` succeeds, this path still writes cache rows only to the temp DB via `upsertMemoryEmbeddingCache`. If a later safe-reindex step fails, those provider-returned embeddings are still discarded with the temp DB for batch-mode users. Use the new cache persistence helper here as well so both embedding paths have the same rollback behavior.
  - confidence: 0.79
