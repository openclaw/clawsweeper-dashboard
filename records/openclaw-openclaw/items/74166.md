## Summary

Keep open. Current main and the latest release still hardcode the Gateway chat-delta throttle at 150 ms with no requested resolver, and this is a new core Gateway operator setting rather than a broken existing contract.

## What This Changes

Request to make the Gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default and defining fallback plus `0` semantics.

## Best Possible Solution

If accepted, add a Gateway-owned documented throttle setting/resolver that preserves the 150 ms default, defines invalid/negative/empty and `0` behavior, and updates focused tests plus docs/changelog for the chosen operator surface.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: `emitChatDelta` still uses a fixed 150 ms guard and no requested env/config resolver exists.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but current Gateway operator-timeout precedent includes documented config plus env precedence, so maintainers should choose the supported surface before implementation.
