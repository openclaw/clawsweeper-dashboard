## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms and has no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver, docs, or config surface; the linked implementation PR is closed and unmerged, and the Telegram streaming request is channel-scoped rather than a replacement for this shared Gateway WebSocket path.

## What This Changes

Add an operator-configurable Gateway chat-delta broadcast throttle via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the existing 150 ms default.

## Best Possible Solution

If maintainers accept the knob, add a small Gateway-owned resolver with the 150 ms default, strict invalid/negative fallback behavior, explicit `0` semantics, updated comments/tests, and docs or changelog coverage appropriate for a supported runtime setting.

## Reproduction Assessment

Yes. Emit two visible assistant text deltas for the same chat run less than 150 ms apart in the existing `server-chat.agent-events` harness; current main suppresses the second delta until a flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the durable public surface for this Gateway tuning control needs maintainer agreement first.
