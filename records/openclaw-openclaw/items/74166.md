## Summary

Keep open. Current main and the latest release still hardcode the Gateway chat-delta broadcast throttle at 150 ms, and the exact implementation PR is closed unmerged, so the remaining question is maintainer acceptance of a new core Gateway operator knob.

## What This Changes

Request to make the Gateway chat-delta broadcast throttle configurable through `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle resolver or config field with explicit invalid/empty/negative and `0` semantics, preserve the 150 ms default, document the operator contract, and cover it with focused Gateway tests.

## Reproduction Assessment

Yes. Emit two assistant text deltas for the same chat run less than 150 ms apart; current `emitChatDelta` suppresses the second broadcast until a final/tool/item flush path sends the buffered text.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but this creates a new supported core Gateway operator setting, so maintainers should choose env-only, config-backed, or no knob.
