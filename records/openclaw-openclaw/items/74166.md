## Summary

Keep open: current main still hardcodes the Gateway chat-delta throttle at 150 ms and lacks the requested resolver, but this is a new core operator tuning contract that needs maintainer product/API approval rather than cleanup-close handling.

## What This Changes

Request to add `OPENCLAW_CHAT_DELTA_THROTTLE_MS` so the Gateway chat-delta broadcast throttle can be tuned while preserving the 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle resolver with the 150 ms default, explicit invalid/negative/empty/`0` semantics, focused tests, and docs/changelog for the chosen operator surface.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: `emitChatDelta` uses a fixed 150 ms threshold and no requested env/config resolver exists.

## Solution Assessment

Unclear pending maintainer product/API review. The env resolver is narrow and default-preserving, but current Gateway operator-timeout precedent includes documented config plus env precedence, so maintainers should choose the supported surface and bounds before implementation.
