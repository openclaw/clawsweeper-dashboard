## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms with no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver or docs, the linked implementation PR is closed and unmerged, and the related Telegram request covers a separate channel preview-streaming surface. The remaining decision is whether maintainers want a new core Gateway operator knob and, if so, whether it should be env-only or config-backed.

## What This Changes

Request to make the Gateway chat-delta WebSocket broadcast throttle configurable while preserving the existing 150 ms default unless operators opt in.

## Best Possible Solution

If accepted, add a small Gateway-owned resolver or config field that preserves the 150 ms default, falls back on unset/empty/non-numeric/negative values, defines explicit `0` semantics, updates the nearby comments, and covers the behavior with focused Gateway tests plus docs or changelog coverage appropriate for a supported operator setting.

## Reproduction Assessment

Yes. Source and existing unit-test inspection give a high-confidence reproduction path: emit two assistant text deltas for the same chat run less than 150 ms apart, and the second visible update is suppressed until the flush path before final or a boundary event broadcasts the buffered text.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver shape is narrow and default-preserving, but the best supported surface for a new Gateway tuning control needs maintainer agreement before implementation work should proceed.
