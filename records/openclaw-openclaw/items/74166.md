## Summary

Keep open. Current main and the latest release still use a hardcoded 150 ms Gateway chat-delta throttle with no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver, and accepting this would add a new Gateway operator setting that needs maintainer product/API review.

## What This Changes

Request to make the Gateway chat event delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle setting with explicit unset, empty, invalid, negative, fractional, and `0` semantics while keeping the 150 ms default and updating focused tests plus operator-facing docs/changelog as maintainers choose.

## Reproduction Assessment

Yes for the current limitation: source inspection shows `emitChatDelta` still uses a literal 150 ms threshold, and exact search found no resolver or env-var support. This is a feature request rather than a broken documented contract.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but the supported configuration surface should be accepted before implementation.
