## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms and has no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver, config entry, or docs surface; the linked implementation PR is closed and unmerged, and the related Telegram request is channel-scoped rather than a replacement for this shared Gateway WebSocket path.

## What This Changes

Request to make the Gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

If maintainers accept the operator knob, add a small centralized resolver with the 150 ms default, strict invalid/negative fallback behavior, explicit `0` semantics, updated comments/tests, and docs or changelog coverage appropriate for a supported Gateway runtime setting.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence reproduction path: emit two visible assistant text deltas for the same chat run less than 150 ms apart, and the second delta is buffered until a final or tool-boundary flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the best public surface for a new Gateway tuning control needs maintainer agreement before an automated fix PR should be queued.
