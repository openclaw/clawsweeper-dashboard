## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms, has no resolver for the requested environment variable, and the linked implementation PR is closed and unmerged. The request is valid and narrow, but it adds a supported core Gateway operator configuration surface, so maintainer product/API approval should come before an automated fix branch.

## What This Changes

Add an operator-configurable Gateway chat-delta broadcast throttle via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

If maintainers accept the feature, add a small centralized resolver with the 150 ms default, strict invalid/negative fallback behavior, explicit `0` semantics, use it in `server-chat.ts`, update the fixed-throttle comment, and add focused unit plus agent-events coverage with docs/changelog treatment appropriate for a supported Gateway runtime setting.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: emit two visible assistant text deltas for the same chat run less than 150 ms apart, and the second delta is buffered until a final/tool-boundary flush because `emitChatDelta` returns at the fixed 150 ms check.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, and existing env resolver patterns make it feasible, but the best supported public surface for a new Gateway tuning control needs maintainer agreement.
