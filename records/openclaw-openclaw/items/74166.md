## Summary

Keep open. Current main and v2026.5.2 still hardcode the Gateway chat-delta throttle at 150 ms with no env resolver, while the requested knob is a new core Gateway tuning contract rather than broken existing behavior.

## What This Changes

The issue requests an operator setting to configure the Gateway chat-delta broadcast throttle via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while keeping the 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle resolver with the 150 ms default and explicit invalid, negative, empty, and `0` semantics, then update focused tests plus docs/changelog for the chosen operator surface.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: `emitChatDelta` suppresses broadcasts inside 150 ms and no requested env/config resolver exists.

## Solution Assessment

Unclear pending maintainer product/API review. The env resolver is narrow and default-preserving, but current Gateway operator-timeout precedent includes documented config plus env precedence, so maintainers should choose the supported surface before implementation.
