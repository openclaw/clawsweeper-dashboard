## Summary

Keep open: current main still hardcodes the Gateway chat-delta throttle at 150 ms and has no requested resolver, but this is a new Gateway operator tuning contract that needs maintainer product/API approval rather than cleanup-close handling.

## What This Changes

Request to make Gateway chat-delta broadcast throttling configurable with `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle setting/resolver with the 150 ms default, explicit invalid-value semantics, focused regression tests, and docs/changelog for the chosen operator surface.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: `emitChatDelta` uses a fixed 150 ms threshold and the requested env/config resolver is absent.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but existing Gateway timeout precedent uses documented config plus env precedence, so maintainers should choose the supported surface and bounds before implementation.
