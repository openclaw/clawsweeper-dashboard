## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms with no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver or docs surface, and the linked implementation PR #74152 is closed unmerged; the remaining blocker is maintainer acceptance of a new core Gateway operator knob.

## What This Changes

Request to make Gateway chat-delta broadcast throttling configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default.

## Best Possible Solution

If maintainers accept the capability, add a Gateway-owned resolver or config field that preserves the 150 ms default, defines invalid/negative/empty and `0` semantics, updates nearby comments and operator docs/changelog as appropriate, and covers the behavior with focused Gateway tests.

## Reproduction Assessment

Yes. Source and existing test inspection give a high-confidence path: emit two assistant text deltas for the same chat run less than 150 ms apart, and current `emitChatDelta` suppresses the second broadcast until a flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but the best supported surface may be env-only, config-backed, or no supported knob.
