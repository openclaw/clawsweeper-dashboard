## Summary

Keep open. Current main and the latest release still hardcode the Gateway chat-delta broadcast throttle at 150 ms with no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver, and the linked implementation PR is closed unmerged, so this remains a new core Gateway operator-setting decision for maintainers.

## What This Changes

Request to add `OPENCLAW_CHAT_DELTA_THROTTLE_MS` so Gateway chat-delta broadcasts can override the current 150 ms throttle while preserving the default and fallback behavior.

## Best Possible Solution

If accepted, keep the 150 ms default, add a Gateway-owned resolver with documented invalid and `0` semantics, update focused tests, and document/changelog the operator setting according to maintainer policy.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection confirms the current fixed behavior through the hardcoded 150 ms guard and the 100 ms suppressed-delta test path.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but it creates a supported core Gateway setting and maintainers should choose the public surface first.
