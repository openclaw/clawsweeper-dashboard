## Summary

Keep open: current main still hardcodes the Gateway chat-delta broadcast throttle, and the requested env/config surface is a new core operator tuning contract that needs maintainer product/API approval before implementation.

## What This Changes

Request to make the Gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

If maintainers accept the feature, add a Gateway-owned throttle resolver with the 150 ms default, explicit invalid/empty/negative/zero semantics, focused regression coverage, and docs/changelog for the chosen operator surface.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: `emitChatDelta` still uses a fixed 150 ms threshold and the requested resolver/env var is absent.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but current Gateway operator timeout precedent includes documented config plus env precedence, so maintainers should choose the supported surface first.
