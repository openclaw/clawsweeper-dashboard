## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms, the requested env resolver/config surface is absent, and the linked implementation PR was closed unmerged pending maintainer acceptance of the new core operator knob.

## What This Changes

Request to make the Gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the existing 150 ms default.

## Best Possible Solution

If maintainers accept the capability, add a Gateway-owned resolver or config field with the 150 ms default, clear invalid/empty/negative and `0` semantics, focused Gateway tests, and any needed operator docs/changelog coverage.

## Reproduction Assessment

Yes. Emit two assistant text deltas for the same run less than 150 ms apart; current `emitChatDelta` suppresses the second broadcast until a final/tool/item flush path runs, and existing tests exercise that timing.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env var is narrow and default-preserving, but the best supported surface may be env-only, config-backed, or no supported knob.
