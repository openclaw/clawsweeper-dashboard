## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms, there is no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver or documented setting, the implementation PR linked from the issue is closed and unmerged, and the Telegram-related request covers a different channel preview-streaming surface. The remaining blocker is maintainer product/API acceptance for a new core Gateway operator knob and whether it should be env-only or config-backed.

## What This Changes

Request to make Gateway chat-delta WebSocket broadcasts use a configurable `OPENCLAW_CHAT_DELTA_THROTTLE_MS` throttle while preserving the current 150 ms default and honoring `0` as throttle-disabled.

## Best Possible Solution

A maintainer should first decide the supported surface for this Gateway tuning control. If accepted, add a Gateway-owned resolver or config field that preserves the 150 ms default, rejects unset/empty/non-numeric/negative values, defines explicit `0` semantics, updates nearby comments and operator docs, and covers the default and override behavior with focused Gateway tests.

## Reproduction Assessment

Yes. Source inspection and the existing agent-events test give a high-confidence reproduction path: emit two assistant text deltas for the same run less than 150 ms apart and the second visible update is suppressed until a flush path before final or another boundary event.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but a supported Gateway operator knob may need a config-backed surface and docs/schema alignment rather than an env-only contract.
