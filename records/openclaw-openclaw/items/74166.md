## Summary

Keep open: current main still hardcodes the Gateway chat-delta throttle at 150 ms and has no requested resolver, but exposing it is a new core operator tuning contract that needs maintainer product/API approval rather than cleanup-close handling.

## What This Changes

Request to make the Gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the existing 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle resolver with the 150 ms default, explicit invalid/negative/zero semantics, focused tests, docs, and changelog for the chosen operator surface.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: `emitChatDelta` uses a fixed 150 ms threshold and no requested env/config resolver exists.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but existing Gateway timeout tuning uses documented config plus env precedence, so maintainers should choose the supported surface and bounds before implementation.
