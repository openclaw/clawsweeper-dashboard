## Summary

Keep open. Current main and the latest release still hardcode the Gateway chat-delta broadcast throttle at 150 ms, the requested resolver/env knob is absent, and linked PR #74152 is closed unmerged; the remaining question is maintainer acceptance of a new core Gateway operator setting.

## What This Changes

Make Gateway chat-delta broadcast throttling configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default.

## Best Possible Solution

Have a maintainer choose the supported Gateway surface; if accepted, add a documented and tested Gateway-owned throttle setting with the 150 ms default preserved and explicit invalid, negative, empty, and `0` semantics.

## Reproduction Assessment

Yes. Emit two assistant text deltas for the same chat run less than 150 ms apart; current `emitChatDelta` suppresses the second broadcast until a final/tool/item flush path sends the buffered text.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but a config-backed field or no supported knob may be the intended product surface.
