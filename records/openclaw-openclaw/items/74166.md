## Summary

Keep open. Current main still has a hardcoded Gateway chat-delta broadcast throttle and no resolver for the requested environment variable; the linked implementation PR is closed and unmerged, and the related Telegram request covers a separate channel preview-streaming surface. The remaining blocker is maintainer product/API approval for a new core Gateway runtime knob.

## What This Changes

Request an opt-in `OPENCLAW_CHAT_DELTA_THROTTLE_MS` setting for Gateway chat-delta broadcasts while preserving the current 150 ms default.

## Best Possible Solution

If maintainers accept the operator control, add a small centralized resolver with the 150 ms default, strict invalid/negative fallback behavior, explicit `0` semantics, updated comments/tests, and appropriate docs or changelog coverage for a supported Gateway runtime setting.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence reproduction path: emit two visible assistant text deltas for the same chat run less than 150 ms apart, and the second delta is buffered until a final or tool-boundary flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the best public surface for a new Gateway tuning control needs maintainer agreement before automation implements it.
