## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms, has no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver or docs/config surface, and the linked implementation PR is closed and unmerged. The requested change is narrow, but it creates a supported Gateway operator tuning surface, so maintainer product/API approval should come before an automated repair branch.

## What This Changes

Expose the Gateway chat-delta broadcast throttle through `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the existing 150 ms default and allowing `0` to disable throttling.

## Best Possible Solution

If maintainers accept the knob, add a small centralized resolver with the 150 ms default, strict fallback for unset/empty/non-numeric/negative values, explicit `0` semantics, updated comments, focused unit and Gateway agent-event tests, and user-facing docs/changelog coverage for the supported runtime setting.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: emit two visible assistant text deltas for the same Gateway chat run less than 150 ms apart, and the second delta is buffered until a final or tool-boundary flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but the best supported public surface for a new Gateway tuning control should be decided before a repair branch is queued.
