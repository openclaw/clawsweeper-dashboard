## Summary

Keep open. Current main and the latest release still hardcode the Gateway chat-delta throttle at 150 ms, the requested env resolver is absent, and linked PR #74152 is closed unmerged; the remaining decision is whether maintainers want this new core Gateway operator knob.

## What This Changes

Request to make Gateway chat-delta broadcast throttling configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default and honoring `0` as throttle-disabled.

## Best Possible Solution

If accepted, add a Gateway-owned throttle resolver or config field with explicit invalid/negative/empty and `0` semantics, preserve the 150 ms default, update operator docs/changelog as appropriate, and cover the behavior with focused Gateway tests.

## Reproduction Assessment

Yes. Emit two assistant text deltas for the same chat run less than 150 ms apart; current `emitChatDelta` suppresses the second broadcast until a final/tool/item flush path sends the buffered text.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but maintainers should decide whether the supported surface is env-only, config-backed, or no knob.
