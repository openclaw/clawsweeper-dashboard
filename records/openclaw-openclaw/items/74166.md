## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms, the requested env resolver is absent, and the linked implementation PR was closed unmerged pending maintainer acceptance of a new Gateway operator knob.

## What This Changes

Make the Gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

If maintainers accept the capability, add a Gateway-owned resolver or config field that preserves the 150 ms default, defines invalid/negative/empty and `0` semantics, and covers the behavior with focused Gateway tests plus any required docs/changelog entry.

## Reproduction Assessment

Yes. Emit two assistant text deltas for the same chat run less than 150 ms apart; current `emitChatDelta` suppresses the second broadcast until a final/tool/item flush path runs, and the existing agent-events test models that path.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but the supported surface may be env-only, config-backed, or no public knob.
