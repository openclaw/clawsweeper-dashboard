## Summary

Keep open. Current main and latest release still hardcode the Gateway chat-delta broadcast throttle at 150 ms, and the requested operator knob is a new core Gateway setting that needs maintainer product/API acceptance.

## What This Changes

The issue requests an operator-configurable Gateway chat-delta broadcast throttle via `OPENCLAW_CHAT_DELTA_THROTTLE_MS`, preserving the current 150 ms default.

## Best Possible Solution

If accepted, add a Gateway-owned throttle setting with explicit invalid, negative, empty, and `0` semantics, keep the 150 ms default, and update focused tests plus operator-facing docs/changelog as maintainers prefer.

## Reproduction Assessment

Yes for the current behavior, though this is a feature request rather than a broken contract: source and tests show assistant updates inside the 150 ms window are buffered until a flush path. I did not execute tests because this was a read-only review.
