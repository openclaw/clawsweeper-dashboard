## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms with no env/config resolver, and the linked implementation PR #74152 is closed unmerged; the remaining decision is maintainer acceptance of a new core Gateway operator knob.

## What This Changes

The issue requests an opt-in `OPENCLAW_CHAT_DELTA_THROTTLE_MS` Gateway broadcast throttle while preserving the existing 150 ms default.

## Best Possible Solution

If maintainers accept the capability, add a Gateway-owned throttle resolver or config setting with explicit default, invalid-value fallback, `0` semantics, focused Gateway tests, and operator-facing docs/changelog as appropriate.

## Reproduction Assessment

Yes. Source and existing test inspection give a high-confidence path: emit two assistant text deltas for the same chat run less than 150 ms apart, and current `emitChatDelta` suppresses the second broadcast until a flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed env resolver is narrow and default-preserving, but a config-backed Gateway setting may be preferable if maintainers want discoverability and schema documentation.
