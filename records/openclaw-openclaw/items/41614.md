## Summary

Keep open. The PR targets a real stale cron-cache problem, but current main stores runtime fields like `runningAtMs` in `jobs-state.json` while the branch only watches `jobs.json`, so it still misses the reported manual recovery path; the external contributor also has not supplied real behavior proof.

## What This Changes

The branch adds mtime-based reload checks to `CronService.ensureLoaded()` and tests external `jobs.json` edits that clear a running marker or delete jobs.

## Best Possible Solution

Rework the fix on current main so cache invalidation covers both `jobs.json` and the derived `jobs-state.json` state sidecar, distinguishes missing files from non-file paths, and includes focused regression tests plus redacted real gateway proof.

## Reproduction Assessment

Yes, source-reproducible. A current-main cron service can have `runningAtMs` merged from `jobs-state.json`; after clearing only that sidecar, this PR's non-force fast path would stat unchanged `jobs.json` and keep the stale in-memory marker.

## Solution Assessment

No. The fix direction is useful, but the implementation is not the best current-main solution until it follows the split-store contract and handles missing store files separately from non-file paths.

- **[P1] Watch the state sidecar before trusting the cache:** `src/cron/service/store.ts:32-33`
  - body: `runningAtMs` no longer lives in `jobs.json` for current stores; `loadCronStore` merges it from the derived `jobs-state.json` sidecar.   - confidence: 0.92

That means deleting the store file to clear wedged jobs keeps the old in-memory jobs instead of reloading the empty store; distinguish ENOENT from non-file paths before skipping reload.
  - confidence: 0.84
