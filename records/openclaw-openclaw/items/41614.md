## Summary

Keep open: the PR is a focused cron cache-reload fix, but it is not merge-ready on current main because cron runtime state now lives in `jobs-state.json` while the patch only checks `jobs.json` mtime before reusing the cache. That leaves the advertised manual recovery path for `runningAtMs` stale, and the external contributor still has not provided real behavior proof.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` plus tests for seeing manual `jobs.json` edits before `cron.run` and read operations.

## Best Possible Solution

Make cron cache freshness sidecar-aware by tracking both definition and runtime-state files, reload under the existing cron lock before read/manual-run operations, and prove external runtime-state recovery with focused tests plus redacted real gateway evidence.

## Reproduction Assessment

Yes, from source inspection: current main caches `state.store` for non-force operations, and the PR's proposed freshness check only observes `jobs.json` even though current runtime state is merged from `jobs-state.json`. I did not run tests because this review was read-only.

## Solution Assessment

No. The mtime check is pointed at only one of the files that make up the current cron store, so the narrow maintainable fix is a combined freshness token or equivalent reload trigger for both definition and state sidecar paths.
