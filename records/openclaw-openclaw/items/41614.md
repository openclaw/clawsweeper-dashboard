## Summary

Keep open: the reported cron cache bug is valid and still present on current main, but this PR is not merge-ready because the branch misses current sidecar state-file freshness, uses a stale test dependency, and lacks required real behavior proof.

## What This Changes

This PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` plus tests for externally clearing or deleting cron jobs from `jobs.json`.

## Best Possible Solution

Rebase the fix on current main so cache freshness covers both `jobs.json` and resolved `jobs-state.json`, records freshness after loading, updates tests to `requestHeartbeat`, and includes redacted live gateway/manual-edit proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns cached cron state on non-force `ensureLoaded()`, and manual `cron.run` checks `runningAtMs` after that path.

## Solution Assessment

No. Mtime invalidation is a reasonable direction, but this branch is incomplete against current main because it ignores the state sidecar, records the mtime before loading, and uses a stale test fixture dependency.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` now merges runtime fields from `jobs-state.json`, and state-only writes can update `runningAtMs` without touching `jobs.json`.   - confidence: 0.91
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main this object literal will not typecheck; update the fixtures to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:42-43`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
