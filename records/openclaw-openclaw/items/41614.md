## Summary

Keep open: this is a focused cron bug-fix PR, but the current branch still misses the runtime state sidecar, uses an outdated test dependency name, records the cache token before loading, and lacks required real behavior proof.

## What This Changes

The PR adds mtime-based cache invalidation to cron service store loading plus regression tests for live edits to the cron job store.

## Best Possible Solution

A rebased fix should make cron cache freshness reflect both jobs.json and jobs-state.json, record freshness after loading, use current CronServiceDeps, and cover sidecar-only manual edits with focused regression tests.

## Reproduction Assessment

Yes, by source inspection: current main reuses cached cron state on list/status/run paths while runtime markers can live in jobs-state.json, so sidecar-only external edits can remain invisible until force reload or restart.

## Solution Assessment

No. Mtime invalidation is a maintainable direction, but this PR only checks jobs.json, records the mtime before loading, and its tests use an outdated heartbeat dependency name.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Track the state sidecar when checking cache freshness:** `src/cron/service/store.ts:34-35`
  - body: loadCronStore() merges runtime fields from jobs-state.json, and stateOnly writes can update fields like runningAtMs without touching jobs.json. This fast path only stats state.deps.storePath, so clearing a stuck marker in the sidecar can still leave cron.run returning already-running until restart or a config-file edit.
  - confidence: 0.91
- **[P2] Use the current heartbeat dependency in the tests:** `src/cron/service.external-store-reload.test.ts:25`
  - body: Current CronServiceDeps requires requestHeartbeat, but the added tests pass requestHeartbeatNow. After rebasing onto current main, these fixtures will not typecheck; update them to the current dependency shape or the shared harness.
  - confidence: 0.92
- **[P3] Record the cache mtime after loading the store:** `src/cron/service/store.ts:42`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. If an external write races with the read, the service can store a newer mtime with older content and later treat the cache as fresh; stat after loadCronStore() instead.
  - confidence: 0.74
