## Summary

Keep this PR open because it targets a real cron cache freshness bug, but the current branch is not merge-ready: it predates the split cron state sidecar, still has cache invalidation holes, and lacks real behavior proof.

## What This Changes

The PR adds mtime-based reload logic to cron's cached store loader and a new test file for manual external edits to persisted cron jobs.

## Best Possible Solution

Rebase onto the current split cron store, track freshness for both jobs.json and jobs-state.json, reload on missing or non-file backing paths, record freshness after loading, update tests, and attach redacted live gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main reuses cached cron state for normal operations, and manual cron.run checks runningAtMs after that cached load path. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime-based invalidation is the right direction, but this branch must account for the current split state sidecar, missing backing files, stat-after-read ordering, and current service dependencies before it is the maintainable fix.

Full review comments:

- **[P2] Track the state sidecar before reusing cache:** `src/cron/service/store.ts:34-35`
  - body: loadCronStore() merges runtime fields from jobs-state.json, and state-only saves can update only that sidecar.   - confidence: 0.92

  - body: loadCronStore() treats a missing store file as an empty cron store, but this branch returns early when fileMtimeMs is null. After a populated store is loaded, externally deleting jobs.json or replacing it with a non-file path would keep the old jobs cached indefinitely.
  - confidence: 0.87

  - body: Current main requires CronService deps to include requestHeartbeat, but the added fixtures pass requestHeartbeatNow. After rebasing, these object literals will not match the current service dependency shape.
  - confidence: 0.93

  - confidence: 0.75
