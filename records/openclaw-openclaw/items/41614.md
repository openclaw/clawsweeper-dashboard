## Summary

Keep open: current main still caches non-force cron operations, and the PR direction is useful, but the patch only keys invalidation on `jobs.json` while current main stores runtime fields such as `runningAtMs` in the `jobs-state.json` sidecar. That leaves the manual recovery case incomplete after the split-store changes.

## What This Changes

The PR adds an mtime check to `CronService.ensureLoaded()` so a loaded cron service reloads `jobs.json` before normal operations, plus regression tests for externally clearing `runningAtMs` and deleting jobs.

## Best Possible Solution

Rebase the fix onto current main and make the cron cache freshness check cover both `jobs.json` and its state sidecar before returning the cached store.

## Reproduction Assessment

Yes. By source inspection, current main's normal list/run paths call `ensureLoaded()` without `forceReload`, and `ensureLoaded()` returns cached state; the PR-specific gap is also source-reproducible because runtime state is loaded from the sidecar while the new fast path checks only `jobs.json`.

## Solution Assessment

No. The mtime invalidation direction is right, but the current patch is not the narrow complete fix for current main because it ignores the sidecar file that stores the reported `runningAtMs` recovery state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track the cron state sidecar mtime:** `src/cron/service/store.ts:35-36`
  - body: `loadCronStore()` merges runtime fields from the `jobs-state.json` sidecar when it exists, and `saveCronStore()` keeps fields like `runningAtMs` there without rewriting `jobs.json`. This fast path only stats `state.deps.storePath`, so clearing `runningAtMs` in the sidecar leaves the cached in-memory job returning `already-running` until restart or a config-file edit. Include the sidecar in the reload check and add a regression that edits only that file.
