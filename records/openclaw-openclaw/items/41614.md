## Summary

Keep open: the stale-cache bug is valid, but this branch is not merge-ready against current main because it misses the split runtime state sidecar, can keep deleted/non-file stores cached, still records freshness before loading, and lacks live behavior proof.

## What This Changes

The PR adds mtime-based invalidation to cron's cached store loader and adds unit tests for manually edited persisted cron jobs.

## Best Possible Solution

Rebase onto current main, track freshness for both jobs.json and jobs-state.json, reload on missing or non-file backing paths, record mtimes after loading, update the tests, and attach redacted live gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main reuses cached cron state for normal operations, and manual cron.run checks runningAtMs after that cached load path. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime-based invalidation is the right direction, but this branch must account for the current split state sidecar, missing backing files, stat-after-read ordering, and current service dependencies before it is the maintainable fix.

- **[P2] Track the state sidecar before reusing cache:** `src/cron/service/store.ts:34-35`
  - body: loadCronStore() merges runtime fields from jobs-state.json, and stateOnly saves can update only that sidecar.   - confidence: 0.87

  - confidence: 0.93

  - confidence: 0.75
