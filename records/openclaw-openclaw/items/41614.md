## Summary

Keep open: current main still has the stale in-memory cron cache path, but this PR is incomplete because it validates only `jobs.json`, uses a pre-read mtime as cache proof, and lacks the required changelog entry.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` and a new external-store reload test file for manual cron store edits before `cron.run` and `cron.list`.

## Best Possible Solution

Refresh the patch so cache reuse validates every file read by `loadCronStore()`, records mtimes only after successful reads, preserves missing/non-file safeguards, and adds focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes, source-reproducible: populate the service cache, externally edit only `jobs-state.json` to clear `runningAtMs` while leaving `jobs.json` unchanged, then call `cron.list` or `cron.run`; this PR's `jobs.json`-only fast path can still return cached state. I did not execute a live test in this read-only review.

## Solution Assessment

No. Mtime invalidation is the right narrow direction, but this implementation must track both cron store files and record freshness after reads before it is a reliable fix.
