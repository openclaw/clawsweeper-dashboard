## Summary

Keep this PR open as a useful but incomplete cron bug fix: current main still has the stale cached-store path, but this branch is not merge-ready against the current sidecar-backed cron store and lacks real behavior proof.

## What This Changes

The PR adds mtime-based cache invalidation to cron store loading plus tests for manual `jobs.json` edits that clear `runningAtMs` or delete jobs.

## Best Possible Solution

Rebase the fix so cache freshness covers both `jobs.json` and the resolved state sidecar, handles missing-file transitions, records freshness after loading, and proves the manual recovery flow in a real gateway.

## Reproduction Assessment

Yes. Source inspection shows current main returns cached cron state on non-force `ensureLoaded()`, and manual `cron.run` checks `runningAtMs` after that path; no live repro was run because this was a read-only review.

## Solution Assessment

No. Mtime invalidation is a reasonable direction, but this branch must be updated for the current sidecar store, missing-file freshness, post-read freshness recording, and current test dependency names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:33-35`
  - body: `loadCronStore()` now merges runtime fields from `jobs-state.json`, and state-only writes can update `runningAtMs` without touching `jobs.json`. Since this fast path only stats `storePath`, clearing a stuck marker in the sidecar can still leave `cron.run` returning `already-running` until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Reload when the store file disappears:** `src/cron/service/store.ts:33-34`
  - body: `loadCronStore()` treats a missing store file as an empty cron store, but this fast path returns early whenever `stat()` returns `null`. After a populated store is loaded, deleting `jobs.json` externally would leave the old jobs cached indefinitely instead of reflecting the empty store.
  - confidence: 0.85
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:24-25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main this object literal will not typecheck; update the fixtures to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:33-35`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
