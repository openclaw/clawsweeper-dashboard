## Summary

Keep open: the cron cache bug is real and this PR is focused, but the patch is not merge-ready because it only watches `jobs.json` while current main stores active runtime markers in `jobs-state.json`, and the contributor has not provided real behavior proof.

## What This Changes

The PR adds an mtime check to `CronService.ensureLoaded()` plus unit coverage so `cron.list` and `cron.run` reload external `jobs.json` edits before reusing cached cron state.

## Best Possible Solution

Update cache invalidation to account for both `jobs.json` and the derived `jobs-state.json` runtime sidecar, add regression coverage for sidecar-only `runningAtMs` recovery, and then provide redacted real gateway or CLI proof.

## Reproduction Assessment

Yes, from source inspection: current main stores active cron markers in `jobs-state.json`, while the PR fast path only stats `jobs.json` before reusing memory. A focused repro would edit the sidecar `runningAtMs` while the gateway is live and then call `cron.run` without changing `jobs.json`.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but the implementation must include the split-store runtime sidecar and the tests must exercise the current operator recovery file.
