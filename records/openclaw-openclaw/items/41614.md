## Summary

Keep open: the PR is pointed at a real cron cache bug, but the current patch is incomplete against current main because runtime cron state is sidecar-backed and the new tests still use a stale heartbeat dependency name.

## What This Changes

The PR adds mtime-based cache invalidation to cron store loading and regression tests for external edits to persisted cron jobs.

## Best Possible Solution

Rebase the fix onto current main and make cron cache freshness cover both jobs.json and the state sidecar, with regression coverage for a sidecar-only runningAtMs edit.

## Reproduction Assessment

Yes. Source inspection shows current main reuses cached cron state on ordinary list/run paths, while current persistence can update runningAtMs in the state sidecar without rewriting jobs.json.

## Solution Assessment

No. The mtime invalidation direction is right, but this patch is not the narrow complete fix for current main because it checks only the config store file and its new tests are stale against CronServiceDeps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Track the cron state sidecar mtime:** `src/cron/service/store.ts:35-36`
  - body: `loadCronStore()` merges runtime fields from the `jobs-state.json` sidecar, and `saveCronStore()` can update fields like `runningAtMs` there without rewriting `jobs.json`. This fast path only stats `state.deps.storePath`, so clearing `runningAtMs` in the sidecar still leaves the cached job returning `already-running` until restart or a config-file edit. Include the sidecar in the reload check and add a sidecar-only regression.
  - confidence: 0.9
- **[P2] Use the current heartbeat dependency in the tests:** `src/cron/service.external-store-reload.test.ts:23`
  - body: On current main `CronServiceDeps` requires `requestHeartbeat`, but the new tests construct `CronService` with `requestHeartbeatNow` in the object literal. After rebase this will not typecheck; update the new constructors to use the current dependency name or the shared harness.
  - confidence: 0.92
