## Summary

Keep open: current main still has a source-reproducible stale cron cache path, but this PR is not ready to merge because it misses the split `jobs-state.json` sidecar and its new tests are stale against current dependencies.

## What This Changes

The PR adds mtime-based cache invalidation to cron store loading and regression tests for manual external edits to the cron store.

## Best Possible Solution

Rebase the fix onto current main, track freshness for both `jobs.json` and `jobs-state.json`, update the cache marker after loading, add sidecar-only regression coverage, and use the current `requestHeartbeat` fixture shape.

## Reproduction Assessment

Yes. Source inspection shows current main reuses cached cron state on list/status/run paths, while runtime markers are loaded from and can be written only to `jobs-state.json`; I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The mtime invalidation direction is maintainable, but this PR only checks `jobs.json` and its tests no longer match current `CronServiceDeps`; the safer fix should cover the sidecar contract too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Track the state sidecar when checking cache freshness:** `src/cron/service/store.ts:35-36`
  - body: `loadCronStore()` merges runtime fields from `jobs-state.json`, and `saveCronStore(..., { stateOnly: true })` can update fields like `runningAtMs` without rewriting `jobs.json`. This fast path only stats `state.deps.storePath`, so clearing a stuck marker in the sidecar can still leave `cron.run` returning `already-running` until restart or a config-file edit.
  - confidence: 0.91
- **[P2] Use the current heartbeat dependency in the tests:** `src/cron/service.external-store-reload.test.ts:23`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added tests pass `requestHeartbeatNow`. After rebasing onto current main, these fixtures will not typecheck; update them to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record the cache mtime after loading the store:** `src/cron/service/store.ts:35`
  - body: The patch makes the mtime a cache-validity token, but the load path still captures it before reading the file. If an external write races with the read, the service can store the newer mtime with older content and then treat later calls as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
