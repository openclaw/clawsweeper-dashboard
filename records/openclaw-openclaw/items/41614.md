## Summary

Keep open: the cron cache bug is valid and current main still reuses in-memory state on ordinary paths, but this PR is incomplete for the current sidecar-backed store contract and its added tests use a stale dependency name. The PR also lacks after-fix real behavior proof beyond unit/CI output.

## What This Changes

The PR adds an mtime check to `CronService.ensureLoaded()` so live cron instances reload changed `jobs.json` files and adds external-edit regression tests.

## Best Possible Solution

Rebase the fix onto current main and make cron cache freshness cover both `jobs.json` and `jobs-state.json`, with a sidecar-only regression and current `CronServiceDeps` fixtures.

## Reproduction Assessment

Yes. Source inspection shows current main reuses cached cron state on ordinary run/list paths, while runtime fields are merged from and can be written only to `jobs-state.json`.

## Solution Assessment

No. The mtime invalidation direction is right, but this patch only watches `jobs.json` and its tests are stale against the current `requestHeartbeat` dependency.

## Review Findings
