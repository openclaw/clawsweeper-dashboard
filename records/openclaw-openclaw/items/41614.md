## Summary

Keep this PR open, but do not merge it as-is. Current main still has the stale cron cache path the PR is trying to fix, while the branch misses the current `jobs-state.json` freshness boundary, uses an outdated test dependency name, and lacks external real behavior proof.

## What This Changes

The branch adds mtime-based cron store cache invalidation and regression tests for manual external `jobs.json` edits.

## Best Possible Solution

Rebase the fix so cron cache freshness covers both `jobs.json` and `jobs-state.json`, records freshness after loading, updates tests to the current `requestHeartbeat` dependency, and includes redacted real gateway/manual-edit proof.

## Reproduction Assessment

Yes, by source inspection. Current main's non-force `ensureLoaded` path returns cached state, manual `cron.run` checks `runningAtMs` after that path, and current runtime fields can live in `jobs-state.json`.

## Solution Assessment

No. Mtime invalidation is a maintainable direction, but this branch needs to be sidecar-aware, update its test fixtures for current main, and avoid recording freshness before the load before it is the narrow safe fix.

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` merges runtime fields from `jobs-state.json`, and `saveCronStore({ stateOnly: true })` can update fields like `runningAtMs` without touching `jobs.json`.   - confidence: 0.91

  - confidence: 0.92

  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
