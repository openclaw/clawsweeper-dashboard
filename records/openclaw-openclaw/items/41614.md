## Summary

Keep open: this is a plausible cron stale-cache fix, but the current PR is not merge-ready because it misses the split runtime-state sidecar, treats missing store stats as cache-fresh, has stale test dependencies, and lacks real gateway proof.

## What This Changes

The branch changes `CronService.ensureLoaded()` to stat `jobs.json` before reusing cached cron state and adds tests for external running-marker clearing and job deletion.

## Best Possible Solution

Rebase onto the current split cron store: track freshness for both `jobs.json` and `jobs-state.json`, invalidate on missing backing files, record freshness after loading, update tests, and attach redacted live gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main returns early from `ensureLoaded()` once `state.store` is populated, and `cron.run` checks `runningAtMs` after that cached load path. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime invalidation is the right direction, but this patch only tracks the primary store file, treats a missing stat as fresh, records freshness before reading, and uses stale test dependencies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Track the state sidecar before reusing cache:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` merges runtime fields from `jobs-state.json`, and state-only saves can update only that sidecar. Since this fast path stats only `state.deps.storePath`, clearing a stuck `runningAtMs` marker in the sidecar can still leave `cron.run` using stale memory until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Reload when the store stat disappears:** `src/cron/service/store.ts:35`
  - body: `loadCronStore()` treats a missing store file as an empty cron store, but this branch returns early when `fileMtimeMs === null`. After a populated store is loaded, externally deleting or replacing `jobs.json` with a non-file path would keep the old jobs cached indefinitely.
  - confidence: 0.86
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main, these object literals will not match the service dependency shape.
  - confidence: 0.92
- **[P3] Record freshness after loading the file:** `src/cron/service/store.ts:42-43`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
