## Summary

Keep open: the underlying stale cron cache bug still exists on current main, but this branch is incomplete after the split-store change because its fast path can still miss sidecar-only edits to `jobs-state.json`.

## What This Changes

The PR adds an mtime-based cache check in `CronService.ensureLoaded()` plus regression tests so live cron services reload external `jobs.json` edits before list/run operations.

## Best Possible Solution

Rebase the fix onto current main and make cron cache freshness cover both `jobs.json` and the derived state sidecar, with sidecar-only regression coverage for manual runtime-state recovery.

## Reproduction Assessment

Yes. By source inspection, normal `cron.list`/`cron.status`/`cron.run` paths call `ensureLoaded()` without `forceReload`, and current main stores `runningAtMs` in `jobs-state.json` while the PR checks only `jobs.json`.

## Solution Assessment

No. Mtime invalidation is the right narrow direction, but the current patch does not cover the sidecar file that now holds the reported recovery state.
