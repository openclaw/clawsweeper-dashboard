## Summary

Keep this PR open, but do not merge it as-is. The reported cron cache-staleness problem is real and the PR is aimed at the right surface, but current main splits cron data across `jobs.json` and `jobs-state.json`, while the patch only validates the main store file and records the cache mtime before the read it is meant to prove.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` for the cron store file and adds Vitest coverage for external `jobs.json` edits before `cron.run` and `cron.list`.

## Best Possible Solution

Refresh the implementation so non-force cron cache reuse validates every file read by `loadCronStore()` (`jobs.json` and the derived `jobs-state.json`), records observed mtimes only after successful reads, preserves the non-file/missing-path guard, and includes focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Populate the service cache, externally edit only `jobs-state.json` to clear `runningAtMs` while leaving `jobs.json` unchanged, then call `cron.list` or `cron.run`; the PR’s jobs.json-only fast path can still return cached state.

## Solution Assessment

No. Mtime invalidation is the right narrow direction, but this patch is incomplete because it validates only the config file and lets a pre-read mtime become the cache proof for later hits.

- **[P2] Track both cron store files before cache hits:** `src/cron/service/store.ts:37-39`
  - body: The new fast path only stats `state.deps.storePath`, but current `loadCronStore()` also reads and merges `jobs-state.json`, and `saveCronStore()` can update that sidecar without touching `jobs.json`. Clearing `runningAtMs` or other runtime state in the sidecar can still leave `cron.list` and `cron.run` using stale in-memory state.
  - confidence: 0.93
- **[P2] Record cache mtimes after successful reads:** `src/cron/service/store.ts:38`
  - body: The cache check treats `state.storeFileMtimeMs` as proof that memory matches disk, but the load path records an mtime captured before `loadCronStore()` reads file contents. If a new mtime is observed before stale contents are read, future cache hits can keep stale data until another write changes the mtime again.
  - confidence: 0.8
