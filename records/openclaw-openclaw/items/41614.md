## Summary

Keep open: current main still has the stale cron cache behavior, and this PR is a useful narrow fix, but it is incomplete because it watches only the config store file, records the cache mtime before the read, and lacks the required user-facing changelog entry.

## What This Changes

This PR adds mtime-based cron store cache invalidation in `ensureLoaded()` plus external-store reload tests for manual edits before cron list/run paths.

## Best Possible Solution

Refresh the implementation so non-force cron cache reuse validates the full store footprint (`jobs.json` and the derived `jobs-state.json`), records observed mtimes only after successful reads, and ships focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Populate the service cache, externally edit only `jobs-state.json` to clear `runningAtMs` while leaving `jobs.json` unchanged, then call `cron.list` or `cron.run`; the PR's jobs.json-only fast path can still return cached state.

## Solution Assessment

No. Mtime invalidation is the right narrow direction, but this patch needs to track both files read by `loadCronStore()` and avoid treating a pre-read mtime as proof that memory matches disk.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Track both cron store files before cache hits:** `src/cron/service/store.ts:37-39`
  - body: The new fast path only stats `state.deps.storePath`, but `loadCronStore()` also reads and merges the derived `jobs-state.json`, and `saveCronStore()` can update that sidecar without touching `jobs.json`. Clearing `runningAtMs` or other runtime state in the sidecar can still leave `cron.list` and `cron.run` using stale in-memory state.
  - confidence: 0.93
- **[P2] Record cache mtimes after successful reads:** `src/cron/service/store.ts:38`
  - body: The cache check treats `state.storeFileMtimeMs` as proof that memory matches disk, but the load path records an mtime captured before `loadCronStore()` reads file contents. If a new mtime is observed before stale contents are read, future cache hits can keep stale data until another write changes the mtime again.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/cron/service/store.ts:33-39`
  - body: This is a user-facing cron recovery fix, but the PR only changes cron code and tests. Repo policy requires a `CHANGELOG.md` Fixes entry for user-facing fixes, with appropriate contributor credit.
  - confidence: 0.94
