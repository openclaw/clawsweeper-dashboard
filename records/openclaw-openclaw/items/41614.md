## Summary

Keep open: the stale cron cache bug is valid, but this branch is not merge-ready against current main because cron runtime state now lives in a sidecar that the patch does not track, missing-file transitions can stay cached, and contributor-owned real gateway proof is still absent.

## What This Changes

The PR adds mtime-based cache invalidation to `ensureLoaded()` plus unit tests for external cron store edits that clear `runningAtMs` or remove jobs.

## Best Possible Solution

Update the PR so cache freshness covers both `jobs.json` and the resolved `jobs-state.json`, treats missing/deleted files as reloads, records freshness after loading, updates the tests to current deps, and includes redacted live gateway proof.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main's non-force `ensureLoaded()` returns cached state, and manual `cron.run` checks `runningAtMs` after that path. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime invalidation is a reasonable direction, but this branch must be updated for the sidecar store, missing-file freshness, post-read freshness recording, and current test dependency names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Track the state sidecar in cache freshness:** `src/cron/service/store.ts:32-35`
  - body: `loadCronStore()` now merges runtime fields from `jobs-state.json`, and state-only writes can update `runningAtMs` without touching `jobs.json`. Since this fast path only stats `storePath`, clearing a stuck marker in the sidecar can still leave `cron.run` returning `already-running` until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Reload when the store file disappears:** `src/cron/service/store.ts:33-34`
  - body: `loadCronStore()` treats a missing store file as an empty cron store, but this fast path returns early whenever `stat()` returns `null`. After a populated store is loaded, deleting `jobs.json` externally would leave the old jobs cached indefinitely instead of reflecting the empty store.
  - confidence: 0.85
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:24-25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main this object literal will not typecheck; update the fixtures to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:32-35`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
