## Summary

Keep open: the reported cron stale-cache bug is source-reproducible, but this PR is not ready because it misses the current `jobs-state.json` sidecar, treats deleted stores as fresh cache hits, uses an obsolete test dependency name, and still lacks contributor-owned real behavior proof.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` and unit tests for manual cron store edits that clear `runningAtMs` or remove jobs.

## Best Possible Solution

Rebase the branch to make cache freshness cover both `jobs.json` and `jobs-state.json`, treat missing files as reloads, record freshness after loading, update tests to current deps, and attach redacted live gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current main reuses cached cron state on non-force loads, and manual `cron.run` checks `runningAtMs` after that cached path; I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime invalidation is a reasonable direction, but this PR only checks the primary store, treats missing files as fresh cache hits, records freshness before the read, and has stale test fixture deps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Track the state sidecar in cache freshness:** `src/cron/service/store.ts:32-35`
  - body: `loadCronStore()` now merges runtime fields from `jobs-state.json`, and `saveCronStore(..., { stateOnly: true })` can update `runningAtMs` without touching `jobs.json`. Because this fast path only stats `storePath`, clearing a stuck marker in the sidecar can still leave `cron.run` returning `already-running` until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Reload when the store file disappears:** `src/cron/service/store.ts:33-34`
  - body: `loadCronStore()` treats a missing store file as an empty cron store, but this fast path returns early whenever `stat()` returns `null`. After a populated store is loaded, deleting `jobs.json` externally would leave the old jobs cached indefinitely instead of reflecting the empty store.
  - confidence: 0.85
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:24-25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main, these object literals will not typecheck; update the fixtures to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:32-35`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
