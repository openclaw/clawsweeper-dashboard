## Summary

Keep this PR open, but it is not mergeable as-is: the stale-cache bug still exists on current main, and the branch needs sidecar-aware freshness, current test dependencies, post-read timestamping, and real behavior proof before normal review can continue.

## What This Changes

The PR adds mtime-based cron store cache invalidation in `CronService.ensureLoaded()` plus regression tests for external `jobs.json` edits before `cron.list` and `cron.run`.

## Best Possible Solution

Rebase the fix so cron cache freshness covers both `jobs.json` and `jobs-state.json`, records freshness after loading, updates tests to current dependencies, and includes redacted real gateway/manual-edit proof.

## Reproduction Assessment

Yes, by source inspection: current main’s non-force `ensureLoaded` path returns cached cron state, and `cron.run` checks `runningAtMs` after that path. The sidecar contract means the stale marker can live in `jobs-state.json`, which this branch does not track.

## Solution Assessment

No. Mtime invalidation is a maintainable direction, but this implementation must be sidecar-aware, use the current dependency names in tests, and record freshness after the load before it is the narrow safe fix.

## Review Findings

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` merges runtime fields from `jobs-state.json`, and `saveCronStore({ stateOnly: true })` can update fields like `runningAtMs` without touching `jobs.json`.   - confidence: 0.91

  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. On current main this test file will not typecheck after rebase; update both constructors to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record the cache timestamp after loading:** `src/cron/service/store.ts:42-48`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
