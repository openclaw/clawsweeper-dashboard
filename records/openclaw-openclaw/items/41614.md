## Summary

Keep open: the bug is real and the patch is useful, but it is not merge-ready on current main because runtime cron state now lives in a sidecar that the new cache check does not watch.

## What This Changes

The PR adds mtime-based cron store cache invalidation in `ensureLoaded()` and new tests for external `jobs.json` edits before `cron.run` and `cron.list`.

## Best Possible Solution

Update the cron store freshness check to cover both definition and runtime sidecar changes, then add a regression that clears `runningAtMs` in `jobs-state.json` while the gateway has stale in-memory state.

## Reproduction Assessment

Yes. Source inspection shows current main reuses cached cron state on normal `list`/`run` paths, and the PR's proposed fix can be checked against the current `jobs-state.json` sidecar contract without needing a live gateway.

## Solution Assessment

No. Watching `jobs.json` mtime is the right general direction, but the maintainable fix must use a composite freshness signal for the config file and runtime sidecar so manual recovery edits to active markers are observed.

## Review Findings

Overall correctness: patch is incorrect
