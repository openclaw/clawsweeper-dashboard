## Summary

Keep open: current main still has a source-reproducible stale cron cache path, but this PR is not merge-ready because the patch misses the current sidecar store contract, mishandles missing-file freshness, uses an obsolete test dependency name, and lacks real behavior proof.

## What This Changes

The PR adds mtime-based cache invalidation to cron store loading and tests for manual edits to `jobs.json`.

## Best Possible Solution

Rebase into a sidecar-aware freshness check for both `jobs.json` and `jobs-state.json`, handle deletion and non-file transitions, record freshness after loading, and include focused tests plus redacted live gateway proof.

## Reproduction Assessment

Yes, source-reproducible. Current main returns cached cron state on non-force `ensureLoaded()`, and manual `cron.run` reaches that path before checking `runningAtMs`; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. Mtime invalidation is a reasonable direction, but this implementation needs to cover the state sidecar, missing-file transitions, and post-read freshness recording before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` now merges runtime fields from `jobs-state.json`, and state-only writes can change `runningAtMs` without touching `jobs.json`. Since this fast path stats only `storePath`, clearing a stuck marker in the sidecar can still leave `cron.run` returning `already-running` until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Reload when the store file disappears:** `src/cron/service/store.ts:35-36`
  - body: `loadCronStore()` treats a missing store file as an empty cron store, but this fast path returns when `stat()` yields `null`. After a populated store is loaded, deleting `jobs.json` externally would leave the old jobs cached indefinitely instead of reflecting the empty store.
  - confidence: 0.86
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main these object literals will not typecheck; update the fixtures to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:42-48`
  - body: The patch uses mtime as the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
