## Summary

Keep open: the stale cron cache bug is still source-reproducible on current main, but this PR is not merge-ready because it misses the runtime sidecar, treats missing primary stores as cache hits, uses stale test deps, and lacks real behavior proof.

## What This Changes

The PR adds mtime-based cron store cache invalidation plus tests for manual `jobs.json` edits that clear `runningAtMs` or delete jobs.

## Best Possible Solution

Rebase the branch to track both `jobs.json` and `jobs-state.json` freshness, handle missing-file transitions as reloads, record freshness after loading, update tests to current deps, and attach redacted live gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `ensureLoaded()` returns immediately when `state.store` is set, and `cron.run` calls it before checking `runningAtMs`; manual disk edits can therefore leave stale in-memory state. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime invalidation is directionally right, but this PR only tracks `jobs.json`, returns early when stat is null, records the freshness token before reading, and its tests use the old `requestHeartbeatNow` dependency.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Track the state sidecar in cache freshness:** `src/cron/service/store.ts:32-35`
  - body: `loadCronStore()` merges runtime fields from `jobs-state.json`, and `saveCronStore(..., { stateOnly: true })` can update only that sidecar. Because this fast path stats only `state.deps.storePath`, clearing a stuck `runningAtMs` marker in the sidecar can still leave `cron.run` using stale memory until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Reload when the store file disappears:** `src/cron/service/store.ts:33-34`
  - body: `loadCronStore()` treats a missing store file as an empty cron store, but this branch returns early when `fileMtimeMs === null`. After a populated store is loaded, externally deleting `jobs.json` would keep the old jobs cached indefinitely instead of reflecting the empty store.
  - confidence: 0.86
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:24-25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main, these object literals will not typecheck; update the fixtures to the current dependency shape or the shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:32-35`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
