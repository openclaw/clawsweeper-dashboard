## Summary

Keep open: the stale cron cache path is still source-reproducible on current main, but this PR is not ready because it only watches jobs.json, uses stale test dependencies, records the cache mtime before loading, and lacks real behavior proof.

## What This Changes

The branch adds mtime-based cache invalidation to cron store loading and regression tests for live edits to cron jobs.json.

## Best Possible Solution

A rebased fix should make cron cache freshness reflect both jobs.json and the derived jobs-state.json sidecar, record freshness after loading, update tests to current CronServiceDeps, and include redacted real gateway/manual-edit proof.

## Reproduction Assessment

Yes, by source inspection: current main's ensureLoaded returns cached state for non-force calls, and cron.run uses that path before checking runningAtMs while runtime markers can live in jobs-state.json.

## Solution Assessment

No. Mtime invalidation is a maintainable direction, but this PR does not include the sidecar in freshness, uses a stale dependency name in tests, and records the mtime before the read.

  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. If an external write races with the read, the service can store a newer mtime with older content and later treat the cache as fresh; stat after loadCronStore() instead.
  - confidence: 0.74
