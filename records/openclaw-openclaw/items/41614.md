## Summary

Keep open: the reported stale cron cache path is still source-reproducible on current main, but this PR needs rebase fixes for the current sidecar-backed store and real behavior proof before merge.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` plus unit tests for manual `jobs.json` edits that clear `runningAtMs` or delete jobs.

## Best Possible Solution

Rebase the fix so cache freshness covers `jobs.json` and the resolved `jobs-state.json`, records freshness after loading, updates the tests to current deps, and verifies the manual edit flow in a real gateway.

## Reproduction Assessment

Yes. Source inspection shows current main returns cached cron state on non-force `ensureLoaded()`, and manual `cron.run` checks `runningAtMs` after that path.

## Solution Assessment

No. Mtime invalidation is a reasonable direction, but this branch is incomplete against current main because it ignores the state sidecar, records the mtime before loading, and uses a stale test dependency.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` now merges runtime fields from `jobs-state.json`, and state-only writes can update `runningAtMs` without touching `jobs.json`.   - confidence: 0.91

  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:42-43`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
