## Solution Assessment

No. Mtime-based invalidation is the right direction, but this branch must account for the current split state sidecar, missing backing files, stat-after-read ordering, and current service dependencies before it is the maintainable fix.

Full review comments:

- **[P2] Track the state sidecar before reusing cache:** `src/cron/service/store.ts:34-35`
  - body: loadCronStore() merges runtime fields from jobs-state.json, and stateOnly saves can update only that sidecar. Since this fast path stats only state.deps.storePath, clearing a stuck runningAtMs marker in the sidecar can still leave cron.run using stale memory until restart or a primary-file edit.
  - confidence: 0.92

  - body: loadCronStore() treats a missing store file as an empty cron store, but this branch returns early when fileMtimeMs is null. After a populated store is loaded, externally deleting jobs.json or replacing it with a non-file path would keep the old jobs cached indefinitely.
  - confidence: 0.87
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:25`
  - body: Current main requires CronService deps to include requestHeartbeat, but the added fixtures pass requestHeartbeatNow. After rebasing, these object literals will not match the current service dependency shape.
  - confidence: 0.93
- **[P3] Record freshness after loading the file:** `src/cron/service/store.ts:42-48`
  - body: The cache token is captured before loadCronStore() reads the file. A concurrent external edit can pair a newer mtime with older loaded content and make the fast path treat stale state as fresh until another write occurs.
  - confidence: 0.75
