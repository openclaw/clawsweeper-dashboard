## Summary

Keep open: the PR is aimed at a real stale cron cache bug, but the current branch is not merge-ready because it misses the `jobs-state.json` freshness boundary, uses an outdated heartbeat test dependency, records freshness before loading, and lacks external real behavior proof.

## What This Changes

The PR adds mtime-based `jobs.json` cache invalidation in `CronService.ensureLoaded()` plus tests for external clearing and deletion of stored cron jobs.

## Best Possible Solution

Rework the fix so freshness covers both `jobs.json` and resolved `jobs-state.json`, records mtimes after loading, updates tests to the current `requestHeartbeat` dependency, and includes redacted live gateway/manual-edit proof.

## Reproduction Assessment

Yes. The issue is source-reproducible by inspection: current main returns cached state for non-force `ensureLoaded()`, and manual run checks `runningAtMs` after that path; I did not run a live gateway in this read-only review.

## Solution Assessment

No. Mtime invalidation is a maintainable direction, but this branch must become sidecar-aware, current-main-compatible, and post-load freshness based before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Track the state sidecar when checking freshness:** `src/cron/service/store.ts:34-35`
  - body: `loadCronStore()` merges runtime fields from `jobs-state.json`, and state-only writes can update `runningAtMs` without touching `jobs.json`. Since this fast path only stats `storePath`, clearing a stuck marker in the sidecar can still leave `cron.run` returning `already-running` until restart or a primary-file edit.
  - confidence: 0.91
- **[P2] Use the current heartbeat dependency in tests:** `src/cron/service.external-store-reload.test.ts:25`
  - body: Current `CronServiceDeps` requires `requestHeartbeat`, but the added fixtures pass `requestHeartbeatNow`. After rebasing onto current main this object literal will not typecheck; update the fixtures to the current dependency shape or shared harness.
  - confidence: 0.92
- **[P3] Record freshness after loading the store:** `src/cron/service/store.ts:39-40`
  - body: The patch makes mtime the cache-validity token, but the load path captures it before reading the file. A concurrent external edit can pair a newer mtime with older loaded content and then make the fast path treat stale state as fresh; stat after `loadCronStore()` instead.
  - confidence: 0.74
