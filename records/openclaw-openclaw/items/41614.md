## Summary

Keep this PR open, but do not merge it as-is. Current main still has the non-force cron cache path that can miss external store edits, and the PR is useful, but the branch is stale against the current `jobs.json` plus `jobs-state.json` split and still has concrete P2 correctness issues.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` and a new Vitest file covering external `jobs.json` edits before `cron.run` and `cron.list`.

## Best Possible Solution

Refresh the implementation on current main so cron cache freshness covers every file read by `loadCronStore()` (`jobs.json` and `jobs-state.json`), records observed mtimes after successful reads, preserves the non-file guard, and adds focused coverage for config edits and sidecar-only edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to populate a `CronService` cache, externally edit `jobs-state.json` to clear `runningAtMs` while leaving `jobs.json` unchanged, then call `cron.list` or `cron.run`; the PR fast path can still return the cached state.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but this patch is not the best current fix because it watches only the config store path and turns a pre-read mtime into the future cache validator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Track the cron state sidecar before cache hits:** `src/cron/service/store.ts:37-39`
  - body: The new fast path only stats `state.deps.storePath`. Current `loadCronStore()` also reads `jobs-state.json`, and `saveCronStore()` can update that sidecar without touching `jobs.json`, so clearing `runningAtMs` in the sidecar can leave `cron.run` and `cron.list` using stale in-memory state.
  - confidence: 0.92
- **[P2] Record cache mtimes after successful reads:** `src/cron/service/store.ts:38`
  - body: The cache check now treats `state.storeFileMtimeMs` as proof that the cached store matches disk, but `ensureLoaded()` still records the mtime before `loadCronStore()` reads file contents. If a write's new mtime is observed before the old contents are read, future reloads can be skipped with stale data cached.
  - confidence: 0.78
