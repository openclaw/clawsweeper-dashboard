## Summary

Keep open: the stale cron cache problem is still valid on current main, but this patch is incomplete after the split `jobs-state.json` store and still records freshness before the file read.

## What This Changes

The PR adds mtime-based cache invalidation to cron store loading plus tests for external cron store edits before `cron.run` and `cron.list`.

## Best Possible Solution

Refresh the patch so cache reuse validates every file read by `loadCronStore()`, records mtimes only after successful reads, preserves missing/non-file behavior, and includes focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: populate the service cache, externally edit only `jobs-state.json` to clear `runningAtMs` while leaving `jobs.json` unchanged, then call `cron.list` or `cron.run`; the PR's `jobs.json`-only cache check can still return cached state.

## Solution Assessment

No. Mtime invalidation is the right narrow approach, but this implementation is not reliable until it tracks both cron store files and records freshness after the successful reads it is certifying.
