## Summary

Keep open: current main and v2026.6.10 still use one multi-page `clawpdf` image fallback call, while the PR is a focused candidate fix for the linked scanned-PDF starvation bug. The patch looks correct, but merging it changes the practical `maxPixels` budget from aggregate-per-call to per-page, so maintainers should accept or document that compatibility tradeoff before merge.

## What This Changes

The branch replaces the document-extract PDF image fallback’s single multi-page `clawpdf` call with one image extraction call per selected page and updates extractor tests for that call shape.

## Best Possible Solution

Land the plugin-owner fix after maintainers explicitly accept or document the per-page `maxPixels` semantics for PDF fallback rendering.

## Reproduction Assessment

Yes at source level: current main batches fallback image extraction, and published `clawpdf@0.3.0` shares one `remainingPixels` budget across pages in that call. The PR body also includes before/after terminal-style output from a real 3-page render.

## Solution Assessment

Yes, with a merge-safety caveat: per-page `extract({ mode: "images", pages: [n] })` keeps page validation and render sizing in clawpdf while fixing starvation. The safer pre-merge state is to document or explicitly accept that `maxPixels` now behaves as a per-page fallback render budget.
