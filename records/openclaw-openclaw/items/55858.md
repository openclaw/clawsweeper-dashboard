## Summary

Keep open. Current main has per-channel mode support and per-channel queue debounce support, but cap and drop remain global/session-only in the public config type, strict schema, resolver, and docs, so the narrowed feature request is still actionable.

## What This Changes

The issue requests per-channel queue debounce, cap, and drop settings with fallback to global queue defaults, while acknowledging that per-channel mode is already supported.

## Best Possible Solution

Add a backward-compatible core queue config extension for channel-specific cap and drop fallback, resolve those options in the shared queue settings path, document the precedence, and cover mode/debounce/cap/drop fallback order with focused tests.

Not applicable as a runtime bug report; this is a feature gap. The gap is high-confidence from static verification of current main: schema and docs expose per-channel debounce but resolver/schema/types keep cap and drop global-only.

## Solution Assessment

Yes, a narrow shared queue config/runtime change is the best way to solve this. Channel-specific hacks in Telegram, Slack, or TUI would duplicate behavior that already belongs in resolveQueueSettings and the public messages.queue contract.
