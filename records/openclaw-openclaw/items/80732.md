## Summary

Keep open. Current main still does not auto-trust non-loopback private `baseUrl` hosts for generic OpenAI-compatible model transports, and changing that would relax the model-provider SSRF boundary that current tests deliberately keep opt-in-only.

## What This Changes

Allow generic `openai-completions` and `openai-responses` providers to trust configured private `baseUrl` hostnames automatically instead of requiring per-provider `request.allowPrivateNetwork`, with any current WebSocket path handled consistently.

## Best Possible Solution

Decide the model-provider SSRF trust boundary first; if exact configured-host trust is approved, implement it narrowly in the current model transport with redirect/host-scope tests, docs or doctor guidance, and a separate current-path decision for WebSocket behavior.

## Reproduction Assessment

Yes, by source inspection: a non-loopback private provider `baseUrl` without `request.allowPrivateNetwork` resolves to a policy with no `allowedHostnames` and no private-network opt-in, so the guarded fetch path remains subject to the default private-IP block. I did not run a live LM Studio or vLLM server in this read-only review.

## Solution Assessment

No, not as an automatic cleanup one-liner. The requested default exact-host trust conflicts with current tests and the recently landed fake-IP-only policy, so the maintainable path is a maintainer-reviewed SSRF policy change with focused regression coverage.
