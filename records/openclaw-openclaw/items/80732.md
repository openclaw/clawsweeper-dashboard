## Summary

Keep open. Current main still requires explicit `models.providers.<id>.request.allowPrivateNetwork` for non-loopback private model provider hosts, and the requested automatic exact-host trust would change the SSRF security boundary that current tests and a recent fake-IP-only landing preserve.

## What This Changes

Request auto-allowlisting configured OpenAI-compatible model provider `baseUrl` hostnames in SSRF policy, plus honoring private-network opt-in for the OpenAI Responses WebSocket path.

## Best Possible Solution

Decide the SSRF trust boundary first; if exact configured `baseUrl` host trust is approved, implement it narrowly in the model transport with regression coverage and align docs or doctor guidance for the remaining explicit opt-in cases.

## Reproduction Assessment

Yes, by source inspection: non-loopback private model-provider hosts without `request.allowPrivateNetwork` resolve to a policy that lacks both `allowPrivateNetwork` and `allowedHostnames`, so the guarded fetch path rejects literal private IP hosts. I did not run a live LM Studio/vLLM server in this read-only review.

## Solution Assessment

No, not as a cleanup one-liner. The requested fix conflicts with current tests and a recent fake-IP-only security-policy landing, so the safer path is a maintainer-reviewed SSRF policy change with targeted tests and docs.
