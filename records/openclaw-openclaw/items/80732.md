## Summary

Keep open. Current main and the latest shipped release still keep non-loopback generic OpenAI-compatible provider baseUrl endpoints behind explicit private-network opt-in, while https://github.com/openclaw/openclaw/pull/80751 is the active implementation path and still needs maintainer/security policy approval.

## What This Changes

The issue asks OpenClaw to trust configured generic OpenAI-compatible provider baseUrl origins on private networks without requiring broad `request.allowPrivateNetwork`, and to align Responses transport behavior.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80751 or an equivalent exact-origin policy after security approval, then close this issue as implemented and split any remaining Codex provider-id collision into a focused follow-up.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-loopback private generic provider baseUrl without explicit `request.allowPrivateNetwork` reaches the guarded fetch without `allowedHostnames` or origin trust, so the SSRF guard blocks the private host; I did not run a live LM Studio, vLLM, or Ollama server.

## Solution Assessment

Unclear until maintainer/security policy review signs off. The open exact-origin PR is a safer direction than a plain hostname allowlist because it scopes trust by origin and preserves metadata/redirect guardrails, but the trust boundary is a product/security decision.
