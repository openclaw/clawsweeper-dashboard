## Summary

Keep open. Current main still keeps non-loopback private custom model-provider hosts behind explicit `request.allowPrivateNetwork`, while the active draft fix PR directly addresses the exact-origin SSRF boundary and needs maintainer/security review before this issue can close.

## What This Changes

Request automatic private `baseUrl` trust for generic OpenAI-compatible LLM providers, plus consistent private-network handling for the Responses transport.

## Best Possible Solution

Have a maintainer/security owner decide the exact-origin custom/local provider trust boundary, land or revise the active PR if approved, and split any bundled `codex` provider-id warning into a focused follow-up if needed.

## Reproduction Assessment

Yes, by source inspection. A non-loopback private generic provider `baseUrl` without `request.allowPrivateNetwork` currently gets fake-IP hostname policy only, while current tests assert non-loopback private provider hosts remain denied by default.

## Solution Assessment

Unclear until maintainer/security review. Exact-origin trust is narrower than broad `allowPrivateNetwork`, but it is still a deliberate SSRF policy change and the active PR is the right place to review that boundary.
