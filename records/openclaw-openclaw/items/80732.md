## Summary

Keep open. Current main still keeps non-loopback private model-provider hosts opt-in-only, while the active draft fix in https://github.com/openclaw/openclaw/pull/80751 is still awaiting maintainer/security-policy review rather than being merged or shipped.

## What This Changes

The issue asks OpenClaw to trust explicitly configured private/LAN OpenAI-compatible provider base URLs without requiring broad `models.providers.<id>.request.allowPrivateNetwork`.

## Best Possible Solution

Keep this issue open until maintainers approve and land a scoped SSRF policy change, preferably through the active draft PR or an equivalent narrower replacement, then close it with the merge commit and release provenance.

## Reproduction Assessment

Yes. Source inspection shows a non-loopback private provider `baseUrl` without `request.allowPrivateNetwork` reaches the guarded fetch path with no `allowedHostnames` or `allowPrivateNetwork`, so the SSRF guard blocks it before the model call.

## Solution Assessment

Not yet. The one-line `allowedHostnames` direction changes the SSRF boundary; the safer path is maintainer-approved exact-origin/custom-local scoping with redirect, metadata, DNS-rebinding, and docs coverage.
