## Summary

Keep open. Current main still keeps LAN, tailnet, and private-DNS model provider endpoints behind explicit `request.allowPrivateNetwork` and does not implement the exact-origin trust requested here; the active fix path is the open security-policy PR, so closing this issue before that lands would be premature.

## What This Changes

The issue requests exact configured-baseUrl SSRF trust for generic OpenAI-compatible model providers, plus related handling for Responses/Codex private-endpoint setups.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80751 or an equivalent exact-origin SSRF policy after security approval, then split the bundled Codex provider-id collision into a focused follow-up if it still reproduces.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-loopback private provider baseUrl without `request.allowPrivateNetwork` resolves to a policy without `allowPrivateNetwork` or `allowedHostnames`, so the SSRF guard rejects the literal private host; I did not run a live LM Studio or vLLM server.

## Solution Assessment

Unclear until maintainer/security review signs off. The open exact-origin PR is a much safer shape than a one-line hostname allowlist because it scopes by origin and adds metadata/redirect coverage, but the trust boundary is still a product/security decision.
