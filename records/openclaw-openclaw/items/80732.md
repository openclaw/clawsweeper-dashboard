## Summary

Keep open. Current main still lacks exact-origin private-network trust for configured generic OpenAI-compatible provider baseUrls, and the active fix path is the open SSRF policy PR at https://github.com/openclaw/openclaw/pull/80751, which needs maintainer/security approval rather than cleanup closure.

## What This Changes

The request is to let generic OpenAI-compatible model providers reach a configured private/local baseUrl without requiring broad allowPrivateNetwork, and to align the Responses/Codex-related paths.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/80751 with a maintainer-approved exact-origin custom/local provider policy, then handle the Codex provider-id warning as docs/doctor or a separate issue if still needed.

## Reproduction Assessment

Yes. Source inspection shows a non-loopback private OpenAI-compatible baseUrl without request.allowPrivateNetwork reaches the guarded fetch with only hostnameAllowlist/fake-IP policy, so private IP checks still block it; I did not run a live LM Studio or vLLM server.

## Solution Assessment

Unclear until maintainer/security review signs off. The active exact-origin PR is a plausible narrow fix, but accepting private-origin trust and any SDK/docs surface is a policy decision, not a cleanup one-liner.
