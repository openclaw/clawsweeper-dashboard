## Summary

Keep open: this targets a real current-main Copilot image-runtime auth bypass, but the patch is not mergeable as written because it ignores the endpoint returned with the exchanged Copilot token and the supplied after-fix proof still shows an image-tool error.

## What This Changes

The PR exchanges GitHub Copilot image-tool OAuth credentials for a short-lived Copilot API token in `resolveImageRuntime`, updates Copilot IDE header constants, and adjusts one header expectation.

## Best Possible Solution

Use the same prepared-runtime auth semantics for image requests as the GitHub Copilot provider uses for chat, carrying both the exchanged API token and entitlement-specific `baseUrl`, then add focused coverage and redacted live proof.

## Reproduction Assessment

Yes for source-level reproduction: current main's image runtime uses the raw provider key while the Copilot provider auth path returns a prepared token and base URL, and the linked report plus before screenshot show the integrator-scope 400. I did not run a live Copilot subscription request in this read-only review.

## Solution Assessment

No as written. Exchanging the token is the right direction, but the fix must also apply `copilotToken.baseUrl` to the model used by `complete()` or route through the provider runtime auth helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Apply the exchanged Copilot base URL to image requests:** `src/media-understanding/image.ts:180`
  - body: `resolveCopilotApiToken` returns the endpoint derived from the Copilot token, and the runtime chat path passes that `baseUrl` along. This PR only copies `copilotToken.token` into `apiKey` and keeps using the original `model`, so `complete()` will still build the request from `model.baseUrl`; proxy-routed or entitlement-specific Copilot accounts can continue sending image requests to the wrong endpoint. Update the returned/runtime model when the token exchange succeeds.
  - confidence: 0.91
