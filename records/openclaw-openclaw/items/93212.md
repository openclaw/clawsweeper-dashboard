## Summary

Keep this PR open, but not as-is: the overload classification is a plausible useful fix, yet the Z.AI-specific `[1305]` regex is added to a global legacy fallback list and the supplied proof does not show the changed fallback behavior running.

## What This Changes

The PR adds `访问量过大` and a `[1305]`/Zhipu regex to failover classifiers so Zhipu GLM overloads classify as `overloaded`.

## Best Possible Solution

Move the Z.AI numeric-code mapping into the Z.AI provider plugin's `classifyFailoverReason` hook, keep the generic Chinese overload phrase only with focused coverage, and add proof that the classifier/fallback path now behaves correctly.

## Reproduction Assessment

Partially: source inspection shows current main does not classify the exact Zhipu `[1305]` / `访问量过大` shape as `overloaded`. I do not have a high-confidence live reproduction of the original immediate-failure symptom on current main because `v2026.6.6` already continues past unrecognized primary errors when fallbacks remain.

## Solution Assessment

No: the generic Chinese overload phrase may be acceptable, but the Z.AI-specific numeric code belongs in the provider-owned hook or must otherwise be provider-guarded. The current patch is a plausible workaround, not the best owner-boundary fix.
