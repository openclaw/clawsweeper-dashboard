## Summary

Keep open for normal PR handling. ## What This Changes

The PR makes unsupported Gateway service platforms return a read-only service adapter, keeps status rendering an unknown runtime detail, and adds regression tests that start/stop/restart fail at the service-check boundary before fallback handling.

## Best Possible Solution

Land the adapter-boundary fix once maintainers accept the unsupported-platform compatibility change and required checks are acceptable, while keeping supported launchd/systemd/schtasks adapters unchanged.

## Reproduction Assessment

Yes from source inspection: current main throws in resolveGatewayService() on unsupported platforms before status can render, and lifecycle-core fails early when service.isLoaded() rejects. I did not run a live unsupported-host repro in this read-only review.

## Solution Assessment

Yes, this is the best fix shape I found: moving unsupported behavior into the service adapter lets read-only status use readRuntime() while keeping mutating lifecycle commands rejected at the shared service-check boundary. A downstream status special case would duplicate the ownership boundary and leave lifecycle behavior easier to drift.
