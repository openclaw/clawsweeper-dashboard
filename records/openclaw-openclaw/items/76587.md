## Summary

Keep this issue open: current main still compares Discord command descriptions verbatim during reconcile, and an open paired PR already uses closing syntax for this exact bug, so the issue should be resolved by reviewing and landing or closing that PR rather than by cleanup closure.

## What This Changes

Request to normalize Discord native slash-command description comparison for server-collapsed whitespace and CJK newlines so reconcile does not PATCH unchanged commands and trigger 429s.

## Best Possible Solution

Review PR #76588 and land a comparator-only normalization fix with focused regression coverage, preserving authored descriptions on writes and not relying on the deploy hash cache to mask equality drift.

## Reproduction Assessment

Yes at source level: a live command whose returned description is `第一行。第二行。` and a desired command whose description is `第一行。\n第二行。` flow through commandsEqual as different strings, causing editApplicationCommand during reconcile. I did not run a live Discord startup in this read-only pass.

## Solution Assessment

Yes, the paired PR direction appears to be the narrowest maintainable fix: normalize descriptions only inside the comparison projection and add regression tests for multiline/CJK equality plus substantive differences. Mutating descriptions before writes or relying on the persisted hash cache would either change authored command text or leave the in-process comparator wrong.
