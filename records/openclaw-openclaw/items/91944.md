## Summary

Keep this open: current main and the latest release still have the unbounded cron missed-slot replay path, and https://github.com/openclaw/openclaw/pull/92398 is the open, mergeable closing PR that now owns the fix with real Gateway proof.

## What This Changes

A cron job whose schedule was updated through the API can be replayed at the wrong time after gateway restart because startup catch-up infers a missed slot from stale schedule history.

## Best Possible Solution

Review and land the linked cron activation-boundary PR, then let the merge close this issue.

## Reproduction Assessment

Yes, source-reproducible. Current main recomputes nextRunAtMs on schedule update but startup catch-up still replays when the current expression's previous slot is newer than lastRunAtMs, with no activation boundary.

## Solution Assessment

Yes, the best current path is the linked PR's schedule activation boundary with the legacy fallback for already-updated jobs. The issue should stay open until that PR lands or maintainers choose a replacement fix.
