Keep this open: current main and v2026.6.6 still have the unbounded cron missed-slot replay path, and the open closing PR now owns the fix path with real Gateway proof.

## What This Changes

The issue reports that a cron job updated through `cron.update` can fire at the wrong time after an upgrade/restart because startup catch-up replays a slot from before the schedule update.

## Best Possible Solution

Land one schema-covered schedule-activation implementation with an accepted upgrade-safe fallback, then close this issue through the merged https://github.com/openclaw/openclaw/pull/92398.

Yes, conditionally. The linked PR's dedicated schedule activation timestamp plus legacy `updatedAtMs` fallback is the narrowest maintainable fix path I found for both post-fix edits and already-updated jobs during upgrade.
