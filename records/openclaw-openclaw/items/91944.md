## What This Changes

The issue reports cron jobs updated through the cron API firing at the wrong time after gateway restart because startup catch-up can treat a pre-update cron slot as missed.

## Best Possible Solution

Review and land the focused startup catch-up fix from https://github.com/openclaw/openclaw/pull/92398 or an equivalent narrow patch, then let the closing reference close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows the current-main startup catch-up path can classify `computeJobPreviousRunAtMs(job, now) > lastRunAtMs` as missed even when `cron.update` already moved `nextRunAtMs` into the future, and the open PR demonstrates that path against current-main `timer.ts`.

## Solution Assessment

Yes. A scheduler-side bound using existing schedule-definition metadata is the best fit; changing doctor to arbitrate stale `.migrated` files would address the reporter's initial theory, but current source shows archives are not active inputs and same-id current rows already win.
