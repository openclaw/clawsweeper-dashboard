## Summary

The reported upgrade-time cron misfire is serious enough to keep open, but the proposed `.migrated` re-import root cause does not match v2026.6.5 or current `main`: doctor only detects active legacy paths and same-ID SQLite rows win over legacy JSON. The remaining task is to reproduce the actual schedule/state mutation during update preflight or gateway startup rather than implement timestamp arbitration between two stores.

## What This Changes

Prevent an upgrade-time doctor preflight from causing an API-updated cron schedule to regress to stale legacy JSON state and fire at the wrong time.

## Best Possible Solution

Keep SQLite as the sole authority, reproduce the update with a newer API-updated SQLite row plus stale active and archived legacy artifacts, then fix the first doctor or scheduler path that changes the canonical schedule and add an upgrade regression test proving the job neither regresses nor becomes immediately due.

## Reproduction Assessment

No. The reporter provides a concrete state sequence, but the claimed `.migrated` read is contradicted by both release and current source, and the generic missed-job logs do not establish the mutation path.

## Solution Assessment

No. Adding file-mtime arbitration or another dual-source rule would conflict with SQLite ownership; the best fix is to reproduce and repair the actual preflight, projection-backfill, or scheduler path while ignoring completed archives.
