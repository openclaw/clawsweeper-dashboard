## Summary

## What This Changes

The issue asks OpenClaw to prevent API-updated cron schedules from replaying stale pre-update slots after gateway restart or upgrade.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92398, or replace it with an equivalent scheduler-owned activation boundary plus doctor migration that preserves the same upgrade behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron.update moves nextRunAtMs without recording schedule activation, and restart catch-up replays a previous cron slot solely because it is newer than lastRunAtMs.

## Solution Assessment

Yes, with maintainer acceptance required. The linked PR’s scheduler-owned activation boundary plus doctor backfill is the best current fix shape I found, but it changes persisted cron state and conservative upgrade replay behavior.
