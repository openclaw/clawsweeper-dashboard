## Summary

Keep open: current `main` still has the cron restart catch-up heuristic that can classify a pre-update cron slot as missed, and two open closing PRs now own the fix path. The original `.migrated` overwrite theory is not supported by current doctor code, but the central wrong-time cron replay remains source-reproducible.

## What This Changes

The issue reports that a cron job updated through `cron.update` can fire at the wrong time after gateway restart because startup catch-up replays a stale pre-update slot.

## Best Possible Solution

Land one canonical cron restart catch-up fix, preferably the smaller proof-supplied PR if maintainer review agrees, then let the merged closing PR close this issue.

## Reproduction Assessment

Yes, source-level: after `cron.update` moves `nextRunAtMs` into the future, restart catch-up still classifies a cron job as missed when the new expression's previous slot is later than `lastRunAtMs`. I did not execute a live gateway restart because this review is read-only, but the linked PR proof exercises that path.

## Solution Assessment

Yes, the best fix direction is to bound startup replay by the job definition/update time in the cron timer path, not to add timestamp arbitration for `.migrated` archives. Current doctor code already avoids reimporting archived files and preserves same-ID SQLite rows, so the runtime catch-up heuristic is the narrow failing surface.
