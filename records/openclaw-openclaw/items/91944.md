## Summary

Keep this open: current main and v2026.6.8 still lack the cron schedule activation boundary needed to prevent restart catch-up from replaying a slot that predates an API schedule update, and https://github.com/openclaw/openclaw/pull/92398 is the open closing PR that owns the fix but is currently conflicting.

## What This Changes

The issue reports that after a cron schedule is changed via `cron.update`, a restart can replay a missed slot from the previous schedule and fire an outbound job at the wrong time.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/92398 or an equivalent activation-boundary fix, then let the merged PR close this issue.

## Reproduction Assessment

Yes, source-reproducible. Current main recomputes `nextRunAtMs` on schedule update, but restart catch-up still replays when the current expression's previous slot is newer than `lastRunAtMs`, with no activation boundary.

## Solution Assessment

Yes, the best current shape is a cron-owned schedule activation boundary plus doctor-owned migration/backfill for existing rows. The issue should stay open until that fix lands or maintainers choose a replacement implementation.
