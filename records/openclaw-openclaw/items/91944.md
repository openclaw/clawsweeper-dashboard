## Summary

Keep this open: current main still has the cron startup catch-up path that can classify a slot from the updated cron expression as missed using only `lastRunAtMs`, and two open PRs already reference this issue as their closing target. The original `.migrated` JSON re-import theory is not supported by current source, but the reported wrong-time execution remains a valid core cron bug until one canonical fix lands.

## What This Changes

The issue reports that cron schedules updated through `cron.update` can still fire at the wrong startup catch-up time after upgrade/restart instead of staying on the SQLite-updated future schedule.

## Best Possible Solution

Land one canonical cron startup-catch-up fix that bounds missed-slot replay to the job definition/write timestamp across both immediate replay and backoff deferral, then close this issue from the merged PR.

## Reproduction Assessment

Yes, source-reproducible without running a local repro: seed a cron job with an old `lastRunAtMs`, update its cron expression so `nextRunAtMs` moves into the future, then restart with startup catch-up enabled; current main only compares the new expression's previous slot against `lastRunAtMs`. The linked fix PRs also include before/after proof for this state sequence.

## Solution Assessment

Yes on direction, but not yet solved on main. The best fix is in cron startup catch-up, not doctor timestamp arbitration over `.migrated` archives: replay should ignore candidate slots that predate the schedule definition update while preserving genuine missed-slot recovery.
