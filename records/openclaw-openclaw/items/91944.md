## Summary

Keep this issue open. Current main still has the source-level startup replay path that can treat a slot from the updated cron expression as missed, and the only viable fixing PR is still open with closing syntax, so the issue should close only after that PR lands.

## What This Changes

The issue reports that after a cron schedule is updated via API, gateway restart can replay a pre-update cron slot and fire the job at the wrong time.

## Best Possible Solution

Land the linked cron startup replay fix so replay is bounded by schedule activation time for both startup catch-up and backoff missed-slot deferral, then close this issue through the merged PR.

## Reproduction Assessment

Yes, at source level. Current main still computes a previous slot from the current cron expression during startup replay and compares it only to `lastRunAtMs`; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes, the linked PR is the best current solution path because it targets the startup replay heuristic rather than the unsupported stale-archive theory. The issue should stay open until that PR is merged or replaced by an equivalent fix.
