## What This Changes

The issue requests preventing cron restart catch-up from replaying stale pre-update schedule slots after API schedule changes and cron state migration.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92398, or replace it with an equivalent scheduler-owned activation boundary plus doctor migration that preserves the same upgrade behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron.update moves nextRunAtMs without recording schedule activation, and restart catch-up replays a previous cron slot solely because it is newer than lastRunAtMs.

## Solution Assessment

Yes, the linked PR’s scheduler-owned activation boundary plus doctor backfill is the best current fix shape I found. It still needs maintainer acceptance because it changes persisted cron state and conservative upgrade replay behavior.
