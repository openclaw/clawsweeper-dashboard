## Summary

Keep open: current main still has the unguarded cron restart catch-up predicate, the latest tagged release lacks a landed fix, and two open closing PRs already own candidate repairs.

## What This Changes

The issue reports that after `cron.update` changes a SQLite-backed schedule, gateway restart can classify an older inferred cron slot as missed and fire an isolated agent job during startup deferral.

## Best Possible Solution

Land one canonical cron replay-boundary fix, likely https://github.com/openclaw/openclaw/pull/92398 or the alternate https://github.com/openclaw/openclaw/pull/92305, then close this issue and retire the companion PR.

## Reproduction Assessment

Yes, by source inspection. Current main can replay a cron job on restart when the new expression's previous slot is newer than `lastRunAtMs`, even if an API update already moved `nextRunAtMs` into the future; the companion PRs provide matching after-fix proof, but this read-only review did not run tests locally.

## Solution Assessment

Yes, keeping the issue open is the right state until a linked fix merges. The best repair is a single scheduler-side replay-boundary fix, not timestamp arbitration with `.migrated` legacy archives.
