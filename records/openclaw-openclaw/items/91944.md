## Summary

Keep this open: current main and v2026.6.8 still have the unbounded cron missed-slot replay path, while the open linked PR remains the active candidate fix for the schedule-activation boundary.

## What This Changes

Cron jobs updated through the API can be replayed at the wrong time after upgrade/restart because startup catch-up can infer an old missed slot from the current cron expression and prior run state.

## Best Possible Solution

Land one cron activation-boundary fix that covers both field-present jobs and already-updated legacy rows, then let the merged PR close this issue.

## Reproduction Assessment

Yes. Source inspection shows schedule updates recompute `nextRunAtMs`, while startup catch-up still compares the current cron expression's previous slot to `lastRunAtMs` without an activation boundary.

## Solution Assessment

Yes, conditionally. The linked PR is the right current path because it adds scheduler-owned activation state plus legacy-row handling; this issue should stay open until that PR merges or maintainers choose a replacement.
