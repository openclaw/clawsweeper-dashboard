## Summary

Keep open. Current main still lacks `openai-codex` audio support, but this PR should not merge as-is because it bypasses configured audio endpoints and updates stale default-model plumbing instead of the current runtime and manifest metadata path.

## What This Changes

The PR adds audio capability and a Codex transcription wrapper to the OpenAI media-understanding provider, adds an `openai-codex` audio default, and extends the OpenAI plugin test assertion.

## Best Possible Solution

Update the OpenAI plugin runtime and manifest metadata together: add Codex audio capability, audio default, auto priority, a transcription handler that preserves explicit endpoints, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. On current main, configure or select `openai-codex` for audio media understanding: the provider is image-only and lacks `transcribeAudio`, and the audio resolver rejects providers without that handler before any transcription call.

## Solution Assessment

No. The PR moves in the right direction, but it is stale against current metadata-driven selection and its wrapper discards explicit audio endpoint overrides; the safer path is a metadata-aware Codex audio handler that only strips known ChatGPT backend URLs.
