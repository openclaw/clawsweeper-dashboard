## Summary

Keep open. Current main still lets `openclaw models set` update only the global default while an explicit default-agent model continues to win at runtime; the later comment also correctly notes that subagents spawned by that agent can inherit the same stale agent model through the current spawn fallback chain.

## What This Changes

The issue requests that default-model changes warn about or reconcile explicit `agents.list.main.model` overrides that keep the active/default agent on the previous model.

## Best Possible Solution

Keep model precedence unchanged, but make `openclaw models set` detect explicit default-agent overrides that still point elsewhere and print a clear warning with the affected agent id and config path to update.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: set `agents.defaults.model.primary` to a new model while `agents.list` contains `id: "main"` with its own old `model.primary`; current resolution uses the agent model first.

## Solution Assessment

Yes for a narrow CLI diagnostic fix. Automatic reconciliation or precedence changes would need explicit opt-in or maintainer product judgment because per-agent model overrides are documented behavior.

## Repair Work Prompt

Fix the confusion reported in openclaw/openclaw#76128: `openclaw models set <model>` updates `agents.defaults.model.primary` but can leave the active/default agent pinned to an older `agents.list[].model.primary`. Keep existing per-agent override precedence intact. Add a warning/reporting path to `models set` that detects configured default/active agent overrides whose explicit `model.primary` differs from the newly written default, and prints a clear warning naming the affected agent id(s) and the config path (`agents.list[].model.primary`) to update or clear. Include the subagent inheritance implication only if the agent lacks an explicit `subagents.model`; do not silently rewrite `agents.list[].model`, do not change runtime precedence, and do not broaden this into the larger reconcile/session-reset proposal. Add focused tests for the warning and non-warning cases, update CLI/model docs if user-facing wording changes, and add a CHANGELOG.md fix entry.
