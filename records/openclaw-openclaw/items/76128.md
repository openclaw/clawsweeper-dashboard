## Summary

Keep open. Current main still lets `openclaw models set` update `agents.defaults.model` and print success while an explicit default-agent `agents.list[].model.primary` continues to win for the active agent, with the same shadow also feeding subagent spawn defaults.

## What This Changes

The issue asks for default-model changes to warn, reconcile, or expose when the main/default agent has an explicit per-agent model override that keeps using the old model.

## Best Possible Solution

Keep per-agent override precedence intact, but make `openclaw models set` detect default-agent model shadows and print a clear warning that names the affected agent id and config path, including the subagent inheritance consequence.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `agents.defaults.model.primary` to a new model while `agents.list` contains the default agent with an old `model.primary`, then current resolution returns the agent override first.

## Solution Assessment

Yes for a narrow warning/reporting repair. Automatic reconciliation would be broader product behavior, while a warning preserves the documented override contract and addresses the confusing success path.

## Repair Work Prompt

Fix the confusion reported in https://github.com/openclaw/openclaw/issues/76128. Keep agent override precedence unchanged. Make `openclaw models set <model>` detect configured default/active agent entries whose explicit `agents.list[].model.primary` differs from the newly written `agents.defaults.model.primary`, and print a clear warning that names the affected agent id(s), the shadowing config path such as `agents.list[].model.primary`, and that subagents spawned by that agent may also inherit the old model unless `agents.list[].subagents.model` is set. Do not silently rewrite per-agent models without an explicit maintainer-approved flag. Add focused tests for the warning/no-warning cases and update user-facing docs plus `CHANGELOG.md` because this changes CLI output.
