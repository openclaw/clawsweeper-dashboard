## Summary

Keep open: the patch is a plausible targeted fix for the central `fetchJson` CDP reader, but the external PR still needs real local behavior proof and maintainer awareness that it introduces a hard 16 MiB cap while a separate browser CDP `/json/version` reader remains uncapped.

## What This Changes

The branch imports the shared response-limit reader, caps `fetchJson` CDP JSON response bodies at 16 MiB, parses the bounded buffer, and updates the related unit test mock to use a real `Response`.

## Best Possible Solution

Land a focused browser plugin fix that uses the shared bounded reader at the central `fetchJson` path after real oversized-response proof and maintainer acceptance of the 16 MiB cap, with `readChromeVersion` either intentionally tracked separately or included in a broader follow-up.

## Reproduction Assessment

Yes from source, but not from a live run: current main calls raw `response.json()` in `fetchJson`, and Node's `Response.json()` consumes the body through the platform reader without OpenClaw's byte cap. I did not run an oversized local CDP/HTTP repro in this read-only review.

## Solution Assessment

Yes as a targeted mitigation: the central `fetchJson` helper is the right owner boundary for its callers and reusing `readResponseWithLimit` is narrow. It is not complete browser-CDP OOM hardening because `readChromeVersion` remains a separate uncapped path, and the PR still needs real proof plus cap acceptance.
