## Summary

Keep open: current main and the latest release still truncate assistant chat-history display text after only inline directive stripping, while the direct fix candidate remains open at https://github.com/openclaw/openclaw/pull/90924.

## What This Changes

The issue reports that WebChat history reloads after heartbeat can reproject long assistant messages with visible `<thinking>` scaffolding/private reasoning and an `...(truncated)...` suffix.

## Best Possible Solution

Land a focused gateway history projection fix that strips assistant-visible reasoning/final scaffolding before truncation while preserving exact tool payloads; keep history-length configurability separate unless maintainers choose it.

## Reproduction Assessment

Yes, source-reproducible: current main routes `chat.history` through gateway display projection that strips only inline directive tags before truncating assistant text. I did not run the original Windows WebChat plus Gemini heartbeat session.

## Solution Assessment

Yes, the best fix direction is the linked gateway projection change: sanitize assistant-visible history text before truncation at the server boundary. Raising or configuring the cap alone would still allow malformed private scaffolding to reach history consumers.
