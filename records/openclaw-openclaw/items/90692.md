## Summary

Keep open: current main and the latest release still project WebChat history by directive-stripping and truncating assistant text without the shared assistant-visible history sanitizer, and the focused closing PR is still open rather than merged.

## What This Changes

The issue reports that WebChat history reloads after heartbeat or a new message can expose raw assistant reasoning tags/content and replace a completed long assistant answer with an 8,000-character truncated projection.

## Best Possible Solution

Land the linked focused gateway/shared sanitizer fix, or an equivalent patch that applies assistant-visible reasoning sanitization before history truncation while preserving exact tool payloads.

## Reproduction Assessment

Yes at source level: current main's `chat.history` path still reads transcript messages and projects assistant text through directive-only stripping followed by truncation, while UI tests show unpaired thinking tags keep inner text visible. No runtime reproduction was attempted in this read-only review.

## Solution Assessment

Yes: the best fix is server-side in gateway history projection before truncation, reusing the assistant-visible history sanitizer and preserving exact tool payloads. Raising the 8k cap or relying only on the UI parser would leave raw malformed scaffolding in history payloads.
