## What This Changes

Prevent WebChat history reloads from exposing raw assistant reasoning/final scaffolding or replacing completed long assistant messages with truncated display text after heartbeat/history refresh.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90924 or an equivalent gateway history-projection fix that sanitizes assistant-visible text before truncation while preserving exact tool payloads.

## Reproduction Assessment

No live end-to-end Windows heartbeat reproduction was run. The source-level reproduction path is high-confidence: chat.history reloads call projectChatDisplayMessages, which currently directive-strips and truncates assistant text before reasoning sanitization.

## Solution Assessment

Yes, the best fix is server-side gateway history projection sanitization before truncation. A UI-only parser change or configurable cap would not prevent malformed private scaffolding from reaching all history consumers.
