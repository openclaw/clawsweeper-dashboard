## Summary

Keep open: current main still projects WebChat history by truncating assistant text after directive-only stripping, so malformed or cut-off reasoning tags can still reach the history UI after reload.

## What This Changes

The issue reports that long WebChat history messages can expose raw thinking tags and append a truncation marker after a later heartbeat or message reloads history.

## Best Possible Solution

Sanitize assistant-visible reasoning and final-tag scaffolding in the gateway history projection before applying the history text cap, without adding a new config option or changing tool payload semantics.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run. The source path is high-confidence: current history projection truncates assistant text before applying the shared reasoning sanitizer, and the UI keeps text from unpaired thinking tags.

## Solution Assessment

Yes, a narrow server-side history projection fix is the best current direction. It prevents malformed reasoning scaffolding from reaching all history consumers instead of relying on the UI after truncation has already broken tag structure.
