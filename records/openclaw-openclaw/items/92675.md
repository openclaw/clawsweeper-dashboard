## Summary

Current main and v2026.6.6 already implement the requested stale auth JSON cleanup: runtime auth reads the SQLite store, and doctor imports, verifies, backs up, and removes legacy `auth-profiles.json` / `auth-state.json` when the import succeeds.

## What This Changes

Issue reports recurring GitHub Copilot HTTP 401s after v2026.6.6, attributed to legacy JSON auth files remaining alongside migrated SQLite auth state.

## Best Possible Solution

Keep the shipped SQLite auth store and doctor migration path; fresh reports should focus on any reproducible 401 that remains after `openclaw doctor --fix` and gateway restart on v2026.6.6 or newer.

## Reproduction Assessment

No high-confidence current-main failure reproduction remains. The reported stale JSON shape is covered by current source and tests, which show SQLite-only runtime reads and verified doctor cleanup of legacy auth JSON.

## Solution Assessment

Yes. The existing doctor-owned migration is the maintainable fix: import legacy auth JSON into SQLite, verify read-back before cleanup, and keep runtime auth on the canonical SQLite store.
