## Summary

Keep open: the report is a credible auth-provider regression after the latest release, and current main does not contain a newer fix for the reviewed auth migration paths. Source review shows runtime auth is meant to be SQLite-only and doctor already removes legacy JSON when run, so the remaining work is to identify why the reporter still hit stale-token behavior after upgrade.

## What This Changes

The issue reports recurring GitHub Copilot HTTP 401s after upgrading to OpenClaw 2026.6.6, attributed to stale legacy `auth-profiles.json` and `auth-state.json` coexisting with the SQLite auth store.

## Best Possible Solution

Keep runtime auth SQLite-only, then add a narrow doctor/auth-status/on-login or gateway-snapshot repair once maintainers confirm which stale state path makes Copilot reuse an invalid GitHub token after upgrade.

## Reproduction Assessment

No, not yet with high confidence. The issue gives a detailed real-host reproduction and workaround, but current-main source review did not prove a runtime path that rereads stale `auth-profiles.json` once SQLite auth exists.

## Solution Assessment

Unclear. The narrowest maintainable direction is not a runtime JSON fallback; it is to confirm whether upgrade cleanup, `doctor --fix`, auth-list diagnostics, gateway auth snapshot invalidation, or Copilot token-cache handling owns the remaining failure.
