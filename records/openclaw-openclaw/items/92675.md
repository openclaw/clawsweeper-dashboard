## Summary

Current main and the shipped v2026.6.6 tree already implement the SQLite auth store boundary and legacy JSON cleanup path: Copilot auth resolves through the SQLite-backed auth store, legacy `auth-profiles.json` is not a runtime fallback, and `openclaw doctor --fix` imports then removes legacy JSON after verification.

## What This Changes

The issue reports recurring GitHub Copilot HTTP 401s after the SQLite auth migration and asks OpenClaw to remove or surface stale `auth-profiles.json` / `auth-state.json` files.

## Best Possible Solution

Keep the shipped SQLite auth-store behavior and use `openclaw doctor --fix` as the supported cleanup path for leftover legacy auth JSON files.

## Reproduction Assessment

No: I did not reproduce the host-level HTTP 401, and current source plus shipped tests show the reported stale `auth-profiles.json` path is not read by the Copilot runtime. The supported cleanup path is reproducible in source through the doctor migration tests.

## Solution Assessment

Yes: the best current solution is the existing SQLite-only runtime plus doctor-owned legacy JSON import/removal, because root policy keeps runtime on canonical state and reserves legacy state cleanup for doctor migrations.
