## What This Changes

The issue reports recurring GitHub Copilot 401s after upgrade because stale legacy `auth-profiles.json` / `auth-state.json` files were left beside the SQLite auth store.

## Best Possible Solution

Keep the shipped SQLite auth store and doctor migration path; any remaining Copilot 401 after `openclaw doctor --fix` should be filed as a fresh current-state auth report with redacted logs.

## Reproduction Assessment

No current-main reproduction remains. Source and tests show stale legacy JSON is not a runtime auth source and the reported duplicate-store shape is handled by doctor cleanup after SQLite verification.

## Solution Assessment

Yes. The existing doctor-owned migration is the right boundary: import and verify into SQLite, remove legacy files after backup, and keep runtime reads on the canonical SQLite store.
