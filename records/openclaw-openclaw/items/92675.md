## Summary

Current main and the v2026.6.6 release already cover the requested auth cleanup: doctor imports legacy auth JSON/state into SQLite, verifies read-back, backs up and removes the old files, and runtime auth reads SQLite rather than stale auth-profiles.json.

## What This Changes

The issue reports recurring GitHub Copilot 401s after upgrading to v2026.6.6, attributed to legacy auth-profiles.json/auth-state.json remaining beside the SQLite auth store.

## Best Possible Solution

Keep SQLite as the canonical runtime auth store and use doctor to import, verify, back up, and remove legacy JSON; any remaining live 401 after doctor on v2026.6.6 or newer should be reported with fresh diagnostics.

## Reproduction Assessment

No. I did not reproduce the reporter's live HTTP 401, and current source plus tests show stale auth-profiles.json is not read at runtime while doctor removes legacy auth-profiles.json/auth-state.json after verified SQLite import.

## Solution Assessment

Yes. The best current solution is the existing doctor-owned migration plus SQLite-only runtime store; adding runtime fallback or another cleanup path would reintroduce duplicate auth sources.
