## Summary

Current main and the shipped v2026.6.6 tag already use SQLite as the canonical auth store, avoid runtime reads from legacy `auth-profiles.json`, and back up/remove legacy auth JSON after verified doctor migration, so the reported stale-file failure mode is already covered by shipped code.

## What This Changes

The issue reports recurring GitHub Copilot HTTP 401s after v2026.6.6, attributed to stale legacy auth JSON files remaining alongside migrated SQLite auth state.

## Best Possible Solution

Keep the SQLite-only runtime and shipped doctor migration; users who still have host-local legacy files should rerun `openclaw doctor --fix` and re-authenticate if doctor reports unrecoverable legacy OAuth state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No current-main failure path was reproduced; source and shipped regression tests instead show that runtime ignores legacy JSON and doctor removes it after SQLite verification.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable fix is the current SQLite-only runtime plus verified doctor migration, not a fallback that can keep stale JSON credentials alive.
