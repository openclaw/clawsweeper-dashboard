## Summary

Current main and the shipped v2026.6.6 tree already implement the central fix: runtime auth reads SQLite, and doctor migrates/removes stale legacy auth JSON only after verified SQLite read-back.

## What This Changes

The issue reports recurring GitHub Copilot HTTP 401s after OpenClaw 2026.6.6, attributed to stale legacy `auth-profiles.json` and `auth-state.json` files remaining after SQLite auth migration.

## Best Possible Solution

Keep the shipped SQLite-only runtime plus doctor-owned legacy JSON migration; ask for a fresh issue only if Copilot 401s still reproduce after `openclaw doctor --fix` on v2026.6.6 or newer.

## Reproduction Assessment

No. I did not reproduce a failing current-main path; source and regression tests show current main ignores stale legacy auth JSON at runtime and removes it through doctor after verified SQLite import.

## Solution Assessment

Yes. The best solution is the shipped doctor-owned migration boundary: keep runtime on canonical SQLite, verify import before destructive cleanup, and avoid a runtime fallback to stale JSON.
