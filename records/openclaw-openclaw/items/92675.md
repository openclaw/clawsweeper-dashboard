## Summary

Current main already has the canonical SQLite auth repair path, and the matching cleanup/verification behavior shipped in v2026.6.6; the remaining supported action for affected hosts is to run doctor repair rather than keep this item open.

## What This Changes

Issue reports recurring GitHub Copilot HTTP 401s after upgrading to v2026.6.6, attributed to stale legacy auth JSON files remaining beside migrated SQLite auth state.

## Best Possible Solution

Keep SQLite as the canonical runtime store and use `openclaw doctor --fix` to import, back up, and remove leftover legacy auth JSON before restarting the gateway.

## Reproduction Assessment

Yes, source-reproducible. Current tests model stale legacy JSON beside a fresh SQLite credential and prove doctor preserves SQLite while removing the stale file; the live Copilot 401 itself was not replayed without credentials.

## Solution Assessment

Yes. The shipped doctor migration is the right boundary: runtime stays SQLite-only, while doctor owns legacy JSON import, backup, verification, and cleanup.
