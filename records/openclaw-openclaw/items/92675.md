## Summary

Current main and the v2026.6.6 release already implement the requested SQLite auth-store boundary: runtime auth reads SQLite, while doctor imports, verifies, backs up, and removes legacy auth JSON. The reported 401 may need a new trace if it still reproduces after doctor, but stale `auth-profiles.json` / `auth-state.json` is not a remaining runtime source in current code.

## What This Changes

The issue reports recurring GitHub Copilot HTTP 401s after upgrade and requests cleanup or warning behavior for stale legacy auth JSON left alongside SQLite auth credentials.

## Best Possible Solution

Keep SQLite as the canonical runtime auth store and use the doctor-owned migration to back up/remove legacy auth JSON; investigate only fresh 401s that reproduce after doctor on v2026.6.6 or newer.

## Solution Assessment

Yes. The best existing solution is the current doctor migration plus SQLite-only runtime auth path, which avoids adding runtime JSON fallbacks or duplicate credential sources.
