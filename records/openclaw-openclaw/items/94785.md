## Summary

## What This Changes

Adds QQ Bot metadata to prefer session-based announce target lookup and records non-C2C inbound delivery routes during QQ Bot outbound dispatch.

## Best Possible Solution

Land this narrow QQ Bot route-persistence fix after proof shows a group inbound writes QQ deliveryContext and a later sessions_send or announce reaches the same QQ group while C2C does not overwrite that route.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 lack the QQ Bot route write and metadata opt-in while the announce resolver requires that opt-in to read stored session deliveryContext. I did not run a live QQ Bot sessions_send reproduction in this read-only review.

## Solution Assessment

Mostly yes: persisting typed QQ Bot deliveryContext in the plugin and using the existing metadata-gated announce lookup is the right owner-boundary fix. The remaining blocker is proof of the actual announce route, not evidence of a better code location.
