## Summary

Keep open: the PR targets the right QQ Bot group-route bug, but the patch persists `updateLastRoute` for every inbound QQ Bot turn, so default direct C2C/DM traffic can overwrite the main session route; the supplied proof is also unit/compile output rather than real QQ Bot or equivalent transport proof.

## What This Changes

The PR opts QQ Bot into stored announce-target lookup and persists QQ Bot inbound route metadata into session last-route state during outbound dispatch.

## Best Possible Solution

Gate QQ Bot route persistence to group/guild inbound sessions, keep the stored-session announce lookup, add focused group/direct regression coverage, and require redacted live or maintainer-accepted transport proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main builds the exact QQ group target but does not persist it through `updateLastRoute`, and QQ Bot is not opted into stored announce lookup. No live QQ Bot API reproduction was run in this review.

## Solution Assessment

No, not as submitted: the plugin-owned group route persistence plus metadata-gated announce lookup is the right shape, but the persistence must be narrowed so direct QQ Bot turns do not overwrite main-session delivery state.
