## Summary

Current main and shipped releases already fail closed when an isolated cron `payload.model` is rejected, with cron-preflight diagnostics and error-state persistence instead of silent fallback.

## What This Changes

The issue asks cron runs with a rejected explicit `payload.model` to fail loudly or warn instead of silently falling back to the agent's primary model.

## Best Possible Solution

Keep the shipped fail-closed cron model validation and diagnostics path; any remaining report should provide a redacted job definition that reaches a different runtime path.

## Reproduction Assessment

No current-main failing reproduction was established. Source and history provide a high-confidence path for the old fall-through behavior and show that current main now rejects the model before execution.

## Solution Assessment

Yes. Failing the cron run during preflight is the narrow maintainable fix for cost-sensitive model rejection, and the later diagnostics work makes the error visible without preserving silent fallback.
