## Summary

Current main already fails rejected cron `payload.model` selections before the agent runner, records the run as an error with diagnostics and consecutive-error state, and emits scheduler diagnostics instead of silently falling back to the agent primary model.

## What This Changes

The issue asks cron agent-turn jobs with `agentId: main` and a rejected `payload.model` to fail loudly or warn instead of silently falling back to the agent primary model.

## Best Possible Solution

Keep the current fail-closed cron model validation and diagnostics on main, then let the next release carry it to stable users.

## Reproduction Assessment

No against current main: the reported allowlist-rejection path now returns a `cron-preflight` error before provider execution, then records error state and diagnostics. The reported v2026.5.7 behavior is older/release behavior rather than a current-main failing path.

## Solution Assessment

Yes. Failing closed with stored diagnostics and warning logs is the narrowest maintainable fix because it prevents hidden expensive fallback without introducing a new fallback policy or config knob.
