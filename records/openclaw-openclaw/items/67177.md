## Summary

Keep open: current main still returns raw Teams `file.download.info` `content.downloadUrl` values before the existing Graph shares rewrite, while docs and tests show SharePoint/OneDrive Graph downloads are intended but only covered for `contentUrl`/reference attachments.

## What This Changes

Inbound msteams DM file attachments with `file.download.info` should download by rewriting SharePoint/OneDrive `downloadUrl` values through the Graph shares endpoint instead of yielding placeholder-only media.

## Best Possible Solution

Normalize both `contentUrl` and `file.download.info` `downloadUrl` SharePoint/OneDrive links through the same Graph shares helper, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Pass a `file.download.info` attachment whose `content.downloadUrl` is SharePoint/OneDrive and a fetch mock that only succeeds for `graph.microsoft.com/v1.0/shares/...`; current main resolves the candidate to the raw URL before the existing Graph rewrite can run.

## Solution Assessment

Yes. Reusing `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the existing early return is the narrowest maintainable fix; broader logging or DM inline-image behavior should remain separate unless a focused gap remains after this branch is fixed.
