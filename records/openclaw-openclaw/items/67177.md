## Label Justifications

- `P1`: The issue reports a user-facing MSTeams DM attachment workflow where inbound file content is dropped and only a placeholder reaches the agent.
- `impact:message-loss`: The reported failure silently drops inbound attachment content from a channel message before it reaches the agent as usable media.

## Summary

Keep open: current main and v2026.6.8 still return the raw MSTeams file.download.info downloadUrl, and https://github.com/openclaw/openclaw/pull/85845 is the open linked fix candidate rather than a merged implementation.

## What This Changes

The issue requests routing MSTeams file.download.info SharePoint/OneDrive downloadUrl attachments through the existing Graph shares endpoint so DM file contents are downloaded instead of dropped.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/85845, then close this issue once main contains the resolver rewrite and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Teams file.download.info attachment with a SharePoint/OneDrive downloadUrl is treated as downloadable, then resolveDownloadCandidate() returns the raw URL instead of a Graph shares URL before the direct download path runs.

## Solution Assessment

Yes. Reusing tryBuildGraphSharesUrlForSharedLink() in the missing file.download.info branch, with fallback to the original URL for non-shared links, is the narrow maintainable fix and matches the open linked PR.
