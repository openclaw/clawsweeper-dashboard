## Summary

Keep open: current main still returns `file.download.info` `content.downloadUrl` values as raw SharePoint/OneDrive URLs, while the existing Graph shares rewrite only runs for `contentUrl` attachments. The issue is narrow, source-reproducible, and already has a matching helper plus adjacent tests to extend.

## What This Changes

The issue requests that msteams inbound `file.download.info` attachments rewrite SharePoint/OneDrive `content.downloadUrl` values through the Microsoft Graph shares content endpoint instead of fetching the raw URL directly.

## Best Possible Solution

Ship a narrow msteams fix that applies the existing Graph shares rewrite to `content.downloadUrl`, preserves allowlist/auth fallback behavior, and adds focused regression coverage without broadening into unrelated DM media paths.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: a `file.download.info` attachment with a SharePoint `content.downloadUrl` will currently be fetched at the raw URL, while a focused mock can make only the Graph shares URL succeed. I did not run a live Teams DM tenant repro in this read-only review.

## Solution Assessment

Yes. Reusing `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` inside the early `file.download.info` branch is the narrowest maintainable fix; broader DM inline-media work should stay with https://github.com/openclaw/openclaw/issues/65329.
