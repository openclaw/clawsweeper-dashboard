## Summary

Keep open. Current main still leaves the reported MS Teams `file.download.info` path unfixed: the branch returns the raw `content.downloadUrl` before the existing SharePoint/OneDrive Graph shares rewrite runs, and no open PR found in the related search closes this narrow bug.

## What This Changes

Fix MS Teams inbound `file.download.info` attachments so SharePoint/OneDrive `content.downloadUrl` values are rewritten through the Graph shares endpoint instead of being fetched directly.

## Best Possible Solution

Patch only the `file.download.info` branch to apply `tryBuildGraphSharesUrlForSharedLink(downloadUrl)`, preserve existing allowlist/auth fallback behavior, and add regression coverage for SharePoint/OneDrive `content.downloadUrl`.

## Reproduction Assessment

Yes, source-level. A `file.download.info` attachment with a SharePoint/OneDrive `content.downloadUrl` currently returns a raw download candidate URL before the existing Graph shares rewrite can run; I did not run a live Teams tenant reproduction.

## Solution Assessment

Yes. Reusing the existing Graph shares helper in the early `file.download.info` branch is the narrowest maintainable fix, while broader DM media fallback and logging behavior should remain separate follow-ups.
