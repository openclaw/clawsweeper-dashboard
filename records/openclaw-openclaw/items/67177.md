- `P1`: A supported MS Teams inbound file workflow can leave real users' agents with placeholders instead of file content.

## Summary

Keep open: current main and the latest release still return the raw MS Teams `file.download.info` `content.downloadUrl` before the existing SharePoint/OneDrive Graph shares rewrite can run, so the reported attachment drop remains source-proven.

## What This Changes

The issue asks OpenClaw to rewrite MS Teams `file.download.info` `downloadUrl` values through the existing Microsoft Graph shares endpoint so DM file attachments download correctly.

## Best Possible Solution

Land a narrow MS Teams plugin fix that applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the `file.download.info` branch, preserves existing allowlist/auth fallback behavior, and adds regression coverage for SharePoint/OneDrive `downloadUrl` values.

## Reproduction Assessment

Yes, source-level: a `file.download.info` attachment with a SharePoint/OneDrive `content.downloadUrl` enters the early branch and returns the raw URL before the Graph shares rewrite. I did not run a live Teams tenant reproduction in this read-only pass.

## Solution Assessment

Yes: reusing the existing Graph shares helper in the early branch is the narrowest maintainable fix. Broader personal-DM media fallback behavior should stay with https://github.com/openclaw/openclaw/issues/65329.
