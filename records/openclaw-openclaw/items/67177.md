## Summary

Keep open. Current main still returns the raw MS Teams `file.download.info` `content.downloadUrl` before the existing SharePoint/OneDrive Graph shares rewrite runs, so the reported DM file-download failure remains source-proven.

## What This Changes

The issue requests rewriting MS Teams `file.download.info` `downloadUrl` values through the Graph shares endpoint and surfacing attachment-download failures instead of dropping them silently.

## Best Possible Solution

Land a narrow MS Teams plugin fix that applies the existing Graph shares helper to `file.download.info` `downloadUrl` values while preserving current allowlist, auth fallback, and non-shared URL behavior.

## Reproduction Assessment

Yes, source-level. A `file.download.info` attachment with a SharePoint/OneDrive `content.downloadUrl` enters the early branch in `resolveDownloadCandidate()`, returns the raw URL, and cannot reach the existing Graph shares rewrite; no live Teams tenant was exercised.

## Solution Assessment

Yes. Reusing `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the early branch is the narrowest maintainable fix; broader personal-DM media fallback work should stay with https://github.com/openclaw/openclaw/issues/65329.
