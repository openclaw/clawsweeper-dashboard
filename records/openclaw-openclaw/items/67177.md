## Summary

Keep open: current main still returns Teams `file.download.info` `content.downloadUrl` values as raw SharePoint/OneDrive URLs, while the existing Graph shares rewrite only runs for `contentUrl` attachments. The bug is narrow, source-reproducible, and no open closing PR currently owns this exact fix.

## What This Changes

The issue requests that msteams `file.download.info` attachments rewrite SharePoint/OneDrive `content.downloadUrl` values through the existing Graph shares endpoint before download.

## Best Possible Solution

Ship a narrow msteams fix that applies the existing Graph shares rewrite to `content.downloadUrl`, preserves allowlist/auth fallback behavior, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-level. A `file.download.info` attachment with a SharePoint/OneDrive `content.downloadUrl` currently produces a raw download candidate URL, and a focused mock can make only the Graph shares URL succeed.

## Solution Assessment

Yes. Reusing `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` inside the early `file.download.info` branch is the narrowest maintainable fix; broader DM media fallback work should stay with the separate open issue.
