## Summary

Keep open. Current main and the latest release still return raw Teams `file.download.info` `content.downloadUrl` values, while the Graph shares rewrite only applies to the separate `contentUrl` branch; the prior ClawSweeper comment and linked PR context remain accurate.

## What This Changes

Rewrite Microsoft Teams `file.download.info` attachment `downloadUrl` values through the Graph shares endpoint for SharePoint/OneDrive links and improve regression coverage for that path.

## Best Possible Solution

Land a narrow msteams plugin fix that applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the file.download.info branch, keeps existing fetch-policy enforcement intact, and adds a regression test where only the Graph shares endpoint succeeds.

## Reproduction Assessment

Yes. A focused unit test can create a `file.download.info` attachment whose `content.downloadUrl` is a SharePoint URL and use a fetch mock that succeeds only for `graph.microsoft.com/v1.0/shares/...`; current main will fetch the raw SharePoint URL and return no media.

## Solution Assessment

Yes. Reusing the existing Graph shares helper in the early file.download.info branch is the narrowest maintainable fix; no core API or broader Teams media redesign is needed for this specific bug.
