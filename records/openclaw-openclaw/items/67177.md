## Summary

Keep open: current main still returns Teams `file.download.info` `content.downloadUrl` values as raw SharePoint/OneDrive URLs, while the existing Graph shares rewrite only runs for the later `contentUrl` path. The issue is narrow, source-reproducible, and no open closing PR currently owns this exact fix.

## What This Changes

The issue asks for msteams inbound `file.download.info` attachment `downloadUrl` values to be rewritten through the existing Microsoft Graph shares endpoint before download.

## Best Possible Solution

Ship a focused msteams fix that applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` inside the `file.download.info` branch, preserves allowlist/auth fallback behavior, and adds a regression test for SharePoint/OneDrive `content.downloadUrl`.

## Reproduction Assessment

Yes, source-level. A `file.download.info` attachment with a SharePoint/OneDrive `content.downloadUrl` currently produces a raw download candidate URL before the existing Graph shares rewrite can run; I did not run a live Teams tenant reproduction.

## Solution Assessment

Yes. Reusing the existing Graph shares helper in the early `file.download.info` branch is the narrowest maintainable fix; broader DM media fallback behavior should stay with the separate open issue unless maintainers choose a larger pass.
