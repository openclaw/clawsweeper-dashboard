## Label Justifications

- `P1`: The issue describes a user-facing MSTeams DM attachment workflow where inbound file content is silently lost before the agent can use it.
- `impact:message-loss`: The affected attachment content is dropped from an inbound channel message and only a placeholder reaches the agent.

## Summary

Keep open: current main and v2026.6.10 still return the raw MSTeams `file.download.info` `downloadUrl`, and the linked fix PR is open/draft rather than merged. This issue remains the canonical tracker for a source-reproducible MSTeams message-loss bug.

## What This Changes

The issue requests routing MSTeams `file.download.info` SharePoint/OneDrive `downloadUrl` attachments through the existing Graph shares download path instead of fetching the raw URL directly.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/85845; close this issue only after main contains the resolver rewrite and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main treats `file.download.info` attachments as downloadable but returns their raw SharePoint/OneDrive `downloadUrl` before the direct download path runs; I did not perform a live Teams tenant replay.

## Solution Assessment

Yes. Reusing `tryBuildGraphSharesUrlForSharedLink()` in the missing resolver branch, with fallback to the original URL for non-shared links, is the narrow maintainable fix and matches the open linked PR.
