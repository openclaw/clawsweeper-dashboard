## Summary

Keep open: current main still returns raw Teams `file.download.info` `content.downloadUrl` values, while the existing Graph shares rewrite and regression coverage only cover the separate `contentUrl`/reference attachment path.

## What This Changes

The issue requests rewriting msteams `file.download.info` SharePoint/OneDrive `downloadUrl` attachments through the Graph shares endpoint, plus better visibility for download failures.

## Best Possible Solution

Apply `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the `file.download.info` branch, preserve existing media/auth allowlist behavior, and add a focused regression test plus changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence focused reproduction: pass a `file.download.info` attachment whose `content.downloadUrl` is SharePoint/OneDrive and a fetch mock that only succeeds for `graph.microsoft.com/v1.0/shares/...`; current main will fetch the raw URL instead.

## Solution Assessment

Yes. Reusing the existing Graph shares helper at the early `file.download.info` return is the narrowest maintainable fix; the broader logging concern should stay scoped because the main remote download catch already warns.
