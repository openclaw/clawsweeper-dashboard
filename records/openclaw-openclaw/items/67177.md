## Summary

Current main still leaves the reported Teams `file.download.info` path unfixed: the early `downloadUrl` branch returns the raw SharePoint/OneDrive URL before the existing Graph shares rewrite runs. This remains a narrow, source-reproducible msteams bug and should stay open for a focused fix.

## What This Changes

The issue requests that Teams `file.download.info` attachment `content.downloadUrl` values be rewritten through the existing Microsoft Graph shares endpoint before download.

## Best Possible Solution

Ship a focused msteams fix that applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the `file.download.info` branch, preserves current allowlist/auth fallback behavior, and adds regression coverage for a SharePoint/OneDrive `content.downloadUrl`.

## Reproduction Assessment

Yes, source-level. A `file.download.info` attachment with a SharePoint/OneDrive `content.downloadUrl` currently returns a raw download candidate URL before the existing Graph shares rewrite can run; I did not run a live Teams tenant reproduction.

## Solution Assessment

Yes. Reusing the existing Graph shares helper in the early `file.download.info` branch is the narrowest maintainable fix; broader DM media fallback and logging behavior should stay separate unless maintainers choose a larger pass.
