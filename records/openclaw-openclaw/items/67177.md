## Summary

Keep open. Current main still returns raw `file.download.info` `content.downloadUrl` values, while the existing Graph shares rewrite and regression coverage only cover the separate `contentUrl`/reference path.

## What This Changes

The issue asks the msteams plugin to rewrite SharePoint/OneDrive `file.download.info` `content.downloadUrl` values to Microsoft Graph shares download URLs before fetching DM attachments.

## Best Possible Solution

Land a narrow msteams plugin fix that applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` in the `file.download.info` branch, preserves the existing media/auth host policy checks, and adds a regression test where only the Graph shares URL succeeds.

## Reproduction Assessment

Yes. A focused unit test can pass a `file.download.info` attachment whose `content.downloadUrl` is a SharePoint URL and a fetch mock that succeeds only for `graph.microsoft.com/v1.0/shares/...`; current main will fetch the raw SharePoint URL instead.

## Solution Assessment

Yes. Reusing the existing Graph shares helper at the early `file.download.info` return is the narrowest maintainable fix for this bug; no core API or broader Teams media redesign is needed.
