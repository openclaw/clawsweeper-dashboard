## Summary

Keep open: current main still returns Teams `file.download.info` `content.downloadUrl` values as raw SharePoint/OneDrive URLs, while the existing Graph shares rewrite only runs for the later `contentUrl` path. The report remains narrow, source-reproducible, and already has the helper and adjacent tests needed for a focused fix.

## What This Changes

Rewrite MS Teams `file.download.info` attachment `content.downloadUrl` SharePoint/OneDrive links through the Graph shares endpoint so DM file uploads download instead of producing only placeholders.

## Best Possible Solution

Ship a narrow msteams fix that applies `tryBuildGraphSharesUrlForSharedLink()` to the `file.download.info` `downloadUrl` branch, preserves allowlist/auth fallback behavior, and adds focused regression coverage.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: a `file.download.info` attachment with a SharePoint `content.downloadUrl` currently exits before the Graph shares rewrite and will be fetched at the raw URL. I did not run a live Teams DM tenant repro in this read-only review.

## Solution Assessment

Yes. Reusing the existing Graph shares helper inside the early `downloadUrl` branch is the narrowest maintainable fix; broader DM inline-media fallback work should stay with https://github.com/openclaw/openclaw/issues/65329.
