## Summary

Current `main` still has the reported MSTeams `file.download.info` raw-download path, and the issue is already paired with an open closing PR, so it should stay open until that fix or an equivalent one lands.

## What This Changes

The issue asks for MSTeams `file.download.info` attachment `downloadUrl` values from SharePoint/OneDrive to be routed through the existing Graph shares download path instead of being fetched directly.

## Best Possible Solution

Land or intentionally replace https://github.com/openclaw/openclaw/pull/85845, then close this issue once current `main` contains the resolver rewrite and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Teams `file.download.info` attachment with a SharePoint/OneDrive `downloadUrl` reaches `resolveDownloadCandidate()` and current `main` returns the raw URL rather than a Graph shares URL.

## Solution Assessment

Yes. Reusing `tryBuildGraphSharesUrlForSharedLink()` in the `file.download.info` branch, with fallback to the original URL for non-shared links, is the narrow maintainable fix; the linked PR targets that boundary.
