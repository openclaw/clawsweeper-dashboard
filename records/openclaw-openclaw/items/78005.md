## Summary

Keep open: the patch would make deep status report normal third-party plugin versions as drift and can read plugin install records from the CLI state instead of the daemon state; external real behavior proof is also not attached yet.

## What This Changes

The PR adds a deep-only `openclaw [gateway] status --deep` diagnostic that reports npm/ClawHub plugin install records whose recorded version differs from the gateway version.

## Best Possible Solution

Report version drift only for plugin packages that are explicitly expected to track the gateway, read install records using the daemon/service environment, and include actual after-fix status output before merge.

## Reproduction Assessment

Yes, by source inspection: an npm/ClawHub plugin with an independent version such as `0.1.0` is reported as drift against gateway `2026.5.4`, and the gatherer reads install records without the daemon env. I did not run the PR branch because this review is read-only, but the code path is direct.

## Solution Assessment

No. The diagnostic belongs plausibly in deep status, but the current implementation needs a narrower compatibility signal and daemon-state record lookup before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Limit drift checks to gateway-versioned plugin packages:** `src/plugins/plugin-version-drift.ts:82`
  - body: The helper treats every npm/ClawHub install as expected to equal the gateway version. ClawHub/npm are documented as general plugin distribution paths with pinned exact/tag versions, so a normal third-party plugin at `0.1.0` will show `expected 2026.5.4` on every deep status run. Scope this to OpenClaw-owned/date-versioned plugins or another explicit compatibility signal.
  - confidence: 0.9
- **[P2] Read plugin install records from the daemon state:** `src/cli/daemon-cli/status.gather.ts:503`
  - body: `gatherDaemonStatus` already merges the service environment and resolves the daemon config from it, but this new call omits that env and falls back to `process.env` via the installed-index reader. When the installed service uses a different profile/state directory than the shell, `--deep` will inspect the wrong `plugins/installs.json` and miss or invent drift for the running gateway.
  - confidence: 0.87
