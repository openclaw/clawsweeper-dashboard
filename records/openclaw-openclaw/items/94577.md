## Summary

Keep this issue open: current main and the latest release still use the fixed 512-byte recent-session header probe, and the linked open pull request is the active fix candidate rather than a merged fix.

## What This Changes

The issue reports that `continueRecent()` can start a new session when recent-session discovery skips a valid JSONL transcript whose first header line is longer than 512 bytes.

## Best Possible Solution

Review and land the linked focused PR, or an equivalent bounded first-line scanner fix with regression coverage, then let this issue close after merge.

## Reproduction Assessment

Yes. Source inspection shows a header longer than 512 bytes can be loadable by `loadEntriesFromFile()` but rejected by `findMostRecentSession()` before `continueRecent()` decides whether to resume; I did not run tests because this review is read-only.

## Solution Assessment

Yes in shape. The narrow maintainable fix belongs in the recent-session header validation helper with a bounded first-line reader and regression coverage, rather than loading whole transcripts during directory scans.
