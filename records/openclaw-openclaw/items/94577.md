## Summary

Keep this issue open because current main and the latest release still use the fixed-size recent-session header scan, and the linked open PR is the active fix candidate.

## What This Changes

The issue reports that recent-session resume can skip a valid JSONL transcript when its first header line exceeds the recent scan window.

## Best Possible Solution

Review and land the linked focused PR, or an equivalent bounded first-line scanner fix with regression coverage, then close this issue after the fix merges.

## Reproduction Assessment

Yes. Source inspection shows a header longer than 512 bytes can be loadable by the full-file loader but rejected by findMostRecentSession before continueRecent decides whether to resume; I did not run a writable repro in this read-only review.

## Solution Assessment

Yes in shape. The narrow maintainable fix belongs in recent-session header validation with bounded first-line reading and focused regression coverage, not in a broad full-transcript directory scan.
