## Summary

Keep this issue open because current main and the latest release still use a fixed 512-byte recent-session header scan, while the linked open PR is the active fix candidate and has not merged yet.

## What This Changes

continueRecent should resume valid JSONL session files even when the first session header line exceeds 512 bytes.

## Best Possible Solution

Review and land the linked focused PR, or an equivalent bounded first-line scanner fix with regression coverage, then close this issue after the fix merges.

## Reproduction Assessment

Yes. Source inspection shows loadEntriesFromFile() reads the whole JSONL file while findMostRecentSession() validates candidates by parsing only the first 512 bytes, so a valid header longer than that can be skipped before continueRecent() resumes.

## Solution Assessment

Yes in shape. The maintainable fix belongs in recent-session header validation with a bounded first-line reader and regression coverage, rather than loading whole transcripts during directory scans.
