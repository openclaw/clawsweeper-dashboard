## Summary

Keep this issue open: current main and the latest release still have the fixed 512-byte recent-session header scan, and the linked open PR is the active fix candidate rather than a reason to close before merge.

## What This Changes

The issue reports that `SessionManager.continueRecent()` can start a new session when `findMostRecentSession()` rejects a valid JSONL transcript whose first header line exceeds 512 bytes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94578, or an equivalent bounded first-line scanner fix with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows a header longer than 512 bytes can be loadable by `loadEntriesFromFile()` but rejected by `findMostRecentSession()` before `continueRecent()` decides whether to resume.

## Solution Assessment

Yes in shape. The narrow maintainable fix belongs in the recent-session header validation helper with a bounded first-line reader and regression coverage, rather than loading whole transcripts during directory scans.
