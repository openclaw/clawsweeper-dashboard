## Summary

Keep open: the latest PR head fixes the earlier conflict-marker and undeclared-variable problems, but the retry guard is still unsafe because it can treat a pre-prompt compaction precheck as if the inbound message was already submitted.

## What This Changes

The PR tracks `currentMessageId` in the embedded PI run loop, switches same-message overflow-compaction retries to the continuation prompt with user-message persistence suppression, adds a regression test, and adds a Telegram changelog entry.

## Best Possible Solution

Repair the PR so continuation/suppression is tied to an actually persisted or submitted current user message, with regression coverage for both post-submit overflow and pre-prompt compaction precheck retries.

## Reproduction Assessment

Yes. The linked issue has concrete Telegram duplicate-message logs, and current main source shows the non-mid-turn overflow retry can reuse the original prompt/current message id; this review established the path from source rather than a live Telegram run.

## Solution Assessment

No. The intended same-message guard is a narrow maintainable direction, but this patch marks the id as submitted before the attempt proves a user message was persisted, so a pre-prompt compaction retry can suppress the real inbound prompt.

## Review Findings

- **[P1] Track submitted IDs only after the prompt is persisted:** `src/agents/pi-embedded-runner/run.ts:1058`

Track this from the persisted-user callback or otherwise exclude precheck/no-submit attempts.

## Repair Work Prompt

Repair PR #76820 for the Telegram/currentMessageId overflow-compaction retry bug. Keep the fix scoped to the embedded PI run-loop and tests. The same-currentMessageId continuation prompt and `suppressNextUserMessagePersistence` must only apply after the current user message was actually submitted/persisted; do not mark `lastSubmittedCurrentMessageId` before an attempt that can exit through `promptErrorSource: "precheck"` without sending the prompt. Prefer wiring this to the existing `onUserMessagePersisted` callback or an equivalent submitted/persisted signal, and add regression coverage for a currentMessageId pre-prompt compaction path that must retry with the original prompt exactly once. Also strengthen the post-submit overflow test to assert the retry prompt is the continuation prompt. Keep the existing changelog entry; do not change unrelated compaction policy or Telegram delivery behavior.
