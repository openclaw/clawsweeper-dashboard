## Summary

Keep open: the PR targets a real Telegram duplicate replay bug, but the current head still marks the current message id as submitted before proving the user message was persisted, so a pre-prompt compaction retry can suppress the first real prompt after compaction.

## What This Changes

The PR adds run-loop state to switch same-`currentMessageId` overflow-compaction retries to the continuation prompt with user-message persistence suppression, plus a regression test and changelog entry.

## Best Possible Solution

Tie same-message retry suppression to an actual persisted user message, preferably via `onUserMessagePersisted` or an explicit attempt result, and cover both precheck/no-submit and post-submit overflow retries.

## Reproduction Assessment

Yes, from source and linked logs. The linked Telegram report shows duplicate external message ids, and current main shows non-mid-turn compaction retries can reuse the original prompt and `currentMessageId` without a continuation override.

## Solution Assessment

No. The intended same-message guard is narrow, but this implementation records submission too early; the safer fix is to mark the id only after user-message persistence or to exclude precheck/no-submit attempts.
