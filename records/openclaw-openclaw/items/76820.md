## Summary

Keep open: the bug is valid and the production direction is narrow, but the patch still lets the retry submit the original channel transcript prompt/current message metadata, so it does not yet reliably prevent duplicate Telegram transcript entries.

## What This Changes

The PR tracks the last submitted current message id in the embedded runner, switches same-message overflow-compaction retries to a continuation prompt, adds loop coverage, and updates the changelog.

## Best Possible Solution

Repair the PR so same-message compaction retries become a runtime-only continuation that cannot append the original inbound prompt/message id again, then land it as the implementation for #76424.

## Reproduction Assessment

Yes. Source inspection shows current main resubmits the same currentMessageId/transcriptPrompt on overflow-compaction retry, and the PR's mocked path exercises that retry without needing live Telegram state.

## Solution Assessment

No, not as written. The intended strategy is narrow, but changing only the runner prompt override leaves attempt.ts free to submit transcriptPrompt as the visible user message again.
