## Summary

Keep open: the PR targets a valid Telegram context-overflow replay bug, but the latest head still only changes the runner prompt while leaving the original transcript submission metadata available to the attempt layer, so the duplicate inbound message can still be persisted.

## What This Changes

The PR tracks the last submitted currentMessageId, switches unchanged-message compaction retries to a continuation prompt, adds an overflow-compaction regression test, and updates the changelog.

## Best Possible Solution

Make same-message compaction retry a true continuation path that cannot persist the original inbound transcript/currentMessageId again, then cover the second-attempt payload contract in the regression test.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main keeps the same currentMessageId/transcriptPrompt across compaction retry, and the linked issue includes direct-session plus group-topic duplicate Telegram message_id evidence. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

No, not as written. The PR's intended direction is narrow, but setting only nextAttemptPromptOverride leaves attempt.ts able to submit the original transcriptPrompt as the visible user message again.
