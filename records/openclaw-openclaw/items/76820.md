## Summary

Keep open: this PR targets a real Telegram overflow-compaction replay bug, but the latest diff still does not prevent the duplicate submission. It only changes the runner prompt override, while the attempt layer still receives the original non-empty transcript prompt and can submit it as the visible user message again.

## What This Changes

The PR tracks the last submitted currentMessageId, switches same-message overflow-compaction retries to a continuation prompt, adds a regression test, and updates the changelog.

## Best Possible Solution

Repair the PR so same-message compaction retries cannot persist the original inbound prompt/message id again, then land it as the implementation for #76424 with payload-level regression coverage.

## Reproduction Assessment

Yes. Current main source shows the retry keeps the same currentMessageId/transcriptPrompt, and #76424 includes duplicate Telegram message_id logs from both direct and group-topic sessions.

## Solution Assessment

No, not as written. The intended direction is narrow, but changing only nextAttemptPromptOverride leaves attempt.ts free to submit the original transcriptPrompt as the visible user message again.
