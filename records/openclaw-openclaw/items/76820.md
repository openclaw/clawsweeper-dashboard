## Summary

Keep this PR open: it targets a valid Telegram compaction-retry bug, but the current diff is not mergeable because the run-loop changes introduce TypeScript errors and the new regression test does not assert the retry inputs it claims to cover.

## What This Changes

The branch adds a Telegram compaction-retry guard in the embedded PI run loop, a regression test and changelog entry, plus an unrelated core plugin-tools runtime-snapshot caching change for #76295.

## Best Possible Solution

Land a narrow run-loop fix that uses a continuation prompt and one-shot user-persistence suppression for same-currentMessageId compaction retries, with assertions on the second attempt inputs; split or explicitly accept the unrelated #76295 cache change.

## Reproduction Assessment

Yes. The linked report #76424 gives concrete Telegram logs, and current main’s run loop shows non-mid-turn overflow compaction retrying with the original prompt/currentMessageId instead of a continuation prompt.

## Solution Assessment

No. The direction is narrow and maintainable, but this diff needs typed retry-local state, the mutable suppression flag wired into the next attempt, and regression assertions before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Declare and pass the retry suppression flag:** `src/agents/pi-embedded-runner/run.ts:1644`
  - body: The new compaction branch assigns `suppressNextUserMessagePersistence = true`, but the run loop has no local variable by that name and the attempt still passes `params.suppressNextUserMessagePersistence`. This makes the branch fail typechecking and, once declared, still needs to pass the mutable one-shot flag into `runEmbeddedAttemptWithBackend` so the compacted retry does not persist the replayed user prompt.
  - confidence: 0.96
- **[P1] Allow numeric current message ids in retry state:** `src/agents/pi-embedded-runner/run.ts:776`
  - body: `params.currentMessageId` is declared as `string | number | undefined`, but the new `lastSubmittedCurrentMessageId` state is only `string | undefined`. Assigning the parameter into that state fails strict TypeScript and would mishandle numeric channel message ids unless both sides are deliberately normalized.
  - confidence: 0.93
- **[P2] Assert the retried attempt inputs:** `src/agents/pi-embedded-runner/run.overflow-compaction.loop.test.ts:121-130`
  - body: The added test only checks that the generic `retrying prompt` log appears, which current main already emits for successful overflow compaction. Please assert the second mocked attempt receives the continuation prompt and the one-shot persistence suppression flag, otherwise this regression test can pass without the intended behavior.
  - confidence: 0.9
