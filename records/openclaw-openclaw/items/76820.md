## Summary

Keep open: the latest PR still leaves the duplicate Telegram replay bug unfixed because the new same-message guard compares against state that is never populated, and the added test would not catch that failure.

## What This Changes

The branch adds local retry-loop state that compares currentMessageId across overflow compaction retries, switches matching retries to a continuation prompt with persistence suppression, and adds a regression test.

## Best Possible Solution

Repair this PR so the runner records the submitted or persisted currentMessageId, uses a continuation prompt with one-shot persistence suppression on same-id compaction retries, proves the second attempt inputs, and adds the required changelog entry.

## Reproduction Assessment

Yes. #76424 has concrete direct-session and group-topic Telegram logs showing repeated external message ids after overflow auto-compaction, and current main's retry path still reuses params.prompt/currentMessageId for non-mid-turn retries unless an override is already set.

## Solution Assessment

No. The intended direction is narrow and maintainable, but this diff never populates the state it compares and the test does not assert the continuation/suppression inputs that would prove the fix.
