## Summary

Keep open for maintainer routing: current main already covers the generic channelRuntime contract and the related Weixin fetch/Content-Length regression was fixed through the earlier runtime-fetch work, but the issue still bundles external `openclaw-weixin` package behavior with a remaining in-repo Codex Computer Use status/remediation improvement.

## What This Changes

The issue reports three post-2026.5.7 problems: an external Weixin startup runtime race, an Undici `Content-Length` fetch failure, and unclear Codex Computer Use `marketplace_missing` status in isolated Codex homes.

## Best Possible Solution

Split the owner paths: keep the Weixin runtime/header fixes with the external plugin or the already-merged runtime-fetch provenance, and handle OpenClaw only with a narrow Codex Computer Use status/remediation improvement if maintainers want `/status` to detect an existing bundled marketplace without writing config.

## Reproduction Assessment

Partly. Source inspection reproduces the Codex Computer Use `marketplace_missing` path because status uses `installPlugin: false` and bundled marketplace registration is gated behind `allowAdd`, but I did not establish a live Weixin or isolated Codex-home end-to-end failure on current main.

## Solution Assessment

No single in-repo fix is clearly best. The Content-Length/Weixin fetch failure is superseded by https://github.com/openclaw/openclaw/issues/78007 and https://github.com/openclaw/openclaw/pull/78143, the Weixin runtime race belongs in the external plugin, and the OpenClaw-side change should be a focused Codex status/remediation decision.
