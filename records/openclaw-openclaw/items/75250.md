## Summary

Keep open. Current main has mitigations for stale plugin dependency staging, plugin min-host skew, LaunchAgent path stability, and update restart/profile repair, but the central macOS runtime-ownership problem remains: local mode is still documented and implemented as an external Node plus global CLI LaunchAgent, so Homebrew shared-library ABI churn can still break the service or CLI before OpenClaw can repair itself. A maintainer also explicitly confirmed the plugin-runtime-deps part is superseded while the broader Homebrew/runtime policy remains unresolved.

## What This Changes

Make macOS updates robust against mixed Homebrew Node, global OpenClaw package, LaunchAgent runtime, plugin SDK, and plugin dependency cache drift so the service and CLI keep working after brew or OpenClaw upgrades.

## Best Possible Solution

Maintainers should choose the macOS Gateway runtime authority. If external Homebrew/global npm remains supported, doctor/status/update should validate and repair CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host compatibility, and legacy dependency roots as one coherent health check. If OpenClaw should own local service runtime, installer/update should move toward a pinned managed runtime and package layout with atomic service updates.

## Reproduction Assessment

No, not from this checkout alone. The issue includes concrete macOS/Homebrew logs that map to current code paths, but reproducing the Homebrew shared-library ABI break and mixed launchd/global package state requires a live affected macOS Homebrew environment.

## Solution Assessment

Unclear. The existing current-main mitigations are useful for cache drift and plugin-version skew, but the requested runtime ownership change is broader than a narrow bug fix and needs maintainer platform direction before implementation is safely scoped.
