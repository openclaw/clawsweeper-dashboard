## Summary

Keep open. Current main has useful mitigations for stale plugin runtime-deps roots, stable Homebrew Cellar paths, and update restart/version checks, but macOS local mode still documents and implements an external Node plus global CLI service layout. The reported Homebrew Node ABI churn and atomic runtime ownership problem remains a platform/update-policy decision rather than something current main fully implements.

## What This Changes

Request to make macOS Homebrew/global npm installs resilient to Node ABI churn, LaunchAgent/package version skew, and stale plugin runtime-deps cache drift after Homebrew or OpenClaw updates.

## Best Possible Solution

Maintainers should decide the macOS Gateway runtime authority first. If external Homebrew/global npm remains supported, doctor/status/update should explicitly validate CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage keys, then repair or give precise instructions. If OpenClaw should own the service runtime, the installer and service manager should move toward a pinned managed runtime and package layout.

## Reproduction Assessment

No. The report includes concrete macOS/Homebrew logs that match current service and plugin-runtime paths, but there is no high-confidence deterministic reproduction for the combined brew upgrade ABI churn plus service/package/plugin-cache skew sequence in this checkout.

## Solution Assessment

Unclear. A bundled/pinned OpenClaw-managed runtime may be the broadest fix, but that is a macOS platform ownership decision; the narrower maintainable first step would be diagnostics and repair around whichever runtime policy maintainers choose.
