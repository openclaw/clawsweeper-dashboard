## Summary

Keep open. Current main has useful mitigations for plugin runtime-deps drift, LaunchAgent path drift, and update restart verification, but the macOS local-mode product boundary still uses an external Node/global CLI service layout, so the Homebrew Node ABI and atomic runtime ownership problem remains real and distinct from the broader #74630 runtime cluster.

## What This Changes

The issue requests robust macOS update behavior so Homebrew Node, the global OpenClaw package, plugin SDK/runtime-deps cache, and LaunchAgent runtime state cannot drift after Homebrew or OpenClaw updates.

## Best Possible Solution

Maintainers should decide the macOS Gateway runtime authority first. If Homebrew/global npm remains supported, doctor/status/update should validate and repair the CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host compatibility, and runtime-deps stage keys before startup; if OpenClaw should own the service runtime, the installer should move toward a pinned managed runtime and package layout.

## Reproduction Assessment

No. The report provides concrete macOS/Homebrew logs that map to current service and runtime-deps code paths, but this checkout does not provide a high-confidence deterministic reproduction for Homebrew shared-library ABI churn plus service/package/cache skew.

## Solution Assessment

Unclear. The current mitigations are appropriate for cache drift and update verification, but the suggested bundled/pinned runtime direction is a platform ownership decision; a narrower first step would be maintainer-approved diagnostics and repair around the chosen runtime policy.
