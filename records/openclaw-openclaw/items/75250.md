## Summary

Keep open. Current main has useful mitigations for stale runtime-deps roots, versioned runtime-deps stage keys, service entrypoint drift, and plugin-version-skew degradation, but the macOS local service is still documented and implemented around an external global OpenClaw CLI plus an external Node runtime, so the Homebrew ABI and mixed runtime/package/cache drift reported here is not fully resolved.

## What This Changes

The issue requests a more robust macOS update/runtime path so Homebrew Node, the OpenClaw package, the LaunchAgent runtime, plugin SDK files, and plugin runtime-deps caches cannot drift after Homebrew or OpenClaw updates.

## Best Possible Solution

Maintainers should decide the supported macOS runtime authority first. If the external Homebrew/global npm layout remains supported, OpenClaw should explicitly validate and report CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage keys, then repair or prune stale state before startup. If OpenClaw should own the service runtime, the macOS service manager should move toward a pinned managed runtime/package layout.

## Reproduction Assessment

No. The issue includes concrete macOS/Homebrew logs that line up with current code paths, but there is no high-confidence repo-local reproduction for the full brew ABI churn plus service/package/plugin-cache skew sequence.

## Solution Assessment

Unclear. A bundled or pinned runtime would address the broadest failure class, but that is a platform ownership decision; a narrower maintainable first step would be maintainer-approved diagnostics and repair around whichever runtime policy remains supported.
