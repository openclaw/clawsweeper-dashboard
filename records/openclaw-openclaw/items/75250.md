## Summary

Keep open. Current main has retired the old plugin-runtime-deps staging path and has several mitigations for stale service/package paths, but it still intentionally documents and builds macOS local mode around an external Node plus global CLI launchd service. The maintainer comment also explicitly keeps this open because the broader Homebrew runtime-ownership and shared-library ABI problem is unresolved.

## What This Changes

The issue reports macOS update breakage when Homebrew-managed Node, global OpenClaw packages, LaunchAgent service paths, plugin host requirements, and plugin runtime dependency caches drift out of sync.

## Best Possible Solution

Maintainers should decide the macOS Gateway runtime authority first. If external Homebrew/global npm remains supported, doctor/status/update should validate CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host compatibility, and legacy dependency cleanup as one coherent health check; if OpenClaw should own the service runtime, installer/update should move toward a pinned managed runtime and package layout.

## Reproduction Assessment

No high-confidence deterministic reproduction path is available from this checkout. The supplied macOS/Homebrew logs are concrete and map to current source paths, but reproducing the dylib ABI churn plus mixed service/CLI state requires an affected macOS Homebrew environment.

## Solution Assessment

Unclear. Current main’s dependency-staging cleanup and path-stability mitigations are useful, but the requested durable fix is a platform/runtime ownership decision rather than a narrow code-only repair.
