## Summary

Keep open. Current main has useful mitigations for plugin runtime-deps drift, plugin min-host skew, LaunchAgent path stability, and update restart verification, but it still documents and implements macOS local mode as an external Node/global CLI launchd service. The Homebrew Node shared-library ABI failure and the broader atomic runtime-ownership question remain real and are distinct from the broader runtime regression cluster.

## What This Changes

The issue requests a more robust macOS update/runtime flow so Homebrew Node, the OpenClaw CLI package, Gateway service command, plugin SDK files, and plugin runtime-deps cache cannot drift after Homebrew or OpenClaw upgrades.

## Best Possible Solution

Maintainers should first choose the macOS Gateway runtime authority. If external Homebrew/global npm remains supported, doctor/status/update should validate and repair CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host compatibility, and runtime-deps stage keys as one coherent health check; if OpenClaw should own the service runtime, installer/update should move toward a pinned managed runtime and package layout.

## Reproduction Assessment

No high-confidence deterministic reproduction path is available from this checkout. The supplied macOS/Homebrew logs map to current service, update, min-host, and runtime-deps code paths, but reproducing Homebrew dylib ABI churn plus mixed launchd/global package state requires a live macOS Homebrew environment in the affected state.

## Solution Assessment

Unclear. Current main's mitigations are useful and narrow for cache drift and restart verification, but they do not settle the external-runtime ownership problem; the best next implementation depends on maintainer platform policy before an automated repair can be safely scoped.
