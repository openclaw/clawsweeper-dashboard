## Summary

Keep open. Current main has meaningful runtime-deps and service-drift mitigations, but macOS local mode still intentionally depends on an external Node/global CLI service layout, so the reported Homebrew Node ABI and atomic runtime ownership problem remains unresolved and is not fully covered by the broader open #74630 regression cluster.

## What This Changes

The issue asks OpenClaw to survive macOS Homebrew/OpenClaw update drift by keeping the CLI, Gateway service Node runtime, host package, plugin SDK, and plugin runtime-deps cache in sync.

## Best Possible Solution

Maintainers should decide the macOS Gateway runtime authority first. If Homebrew/global npm remains supported, doctor/status/update should validate CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage keys before startup; if OpenClaw should own the service runtime, the installer and service manager should converge on a pinned managed runtime/package layout.

## Reproduction Assessment

No. The issue has concrete macOS/Homebrew logs that match current code paths, but there is not a high-confidence deterministic reproduction for the combined Homebrew ABI churn plus service/package/plugin-cache skew sequence in this checkout.

## Solution Assessment

Unclear. A bundled/pinned service runtime could solve the broadest failure class, but it is a platform ownership decision; the safer first step is maintainer-approved diagnostics and repair around whichever runtime policy OpenClaw wants to support.
