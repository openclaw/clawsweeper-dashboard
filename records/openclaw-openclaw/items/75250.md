## Summary

Keep open. Current main contains useful runtime-deps, package update, and launchd path-drift mitigations, but the central macOS Homebrew/runtime ownership problem remains: docs and service code still support an external Node plus global OpenClaw package layout, so Homebrew Node ABI churn and non-atomic service/runtime/package updates are not fully resolved or superseded by the broader open regression cluster.

## What This Changes

The issue asks OpenClaw to keep the macOS Gateway, CLI, Node runtime, plugin SDK, and bundled plugin runtime-deps cache from drifting apart after Homebrew or OpenClaw updates.

## Best Possible Solution

Maintainers should choose the macOS Gateway runtime authority first. If external Homebrew/global npm remains supported, doctor/status/update should explicitly validate and repair the CLI Node, service Node, package root/version, LaunchAgent entrypoint, plugin min-host requirements, and runtime-deps stage root before startup; if OpenClaw should own the service runtime, the installer and service manager should move to a pinned managed runtime/package layout.

## Reproduction Assessment

No. The report includes concrete macOS/Homebrew logs and they match current runtime and plugin-cache surfaces, but this checkout does not provide a high-confidence deterministic way to reproduce Homebrew shared-library ABI churn plus service/package/plugin-cache skew.

## Solution Assessment

Unclear. A bundled or pinned runtime would likely solve the broadest class of failures, but that is a platform ownership decision; a narrower maintainable first step would be maintainer-approved diagnostics and repair around whichever runtime policy remains supported.
