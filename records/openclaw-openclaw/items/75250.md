## Summary

Keep open. Current main contains several useful mitigations for stale plugin runtime-deps roots, launchd entrypoint drift, and post-update version verification, but it still documents and implements macOS local mode around an external Node/global CLI service layout. The remaining Homebrew Node ABI and atomic runtime ownership problem is not fully covered by the broader open #74630 regression cluster.

## What This Changes

The issue requests a more robust macOS update/runtime layout so Homebrew Node, the Gateway service, the OpenClaw package, plugin SDK files, and plugin runtime-deps caches cannot drift independently after `brew upgrade` or OpenClaw updates.

## Best Possible Solution

Maintainers should first decide the macOS Gateway runtime authority. If Homebrew/global npm remains supported, doctor/status/update should explicitly validate CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage keys, then repair or prune stale state before startup. If OpenClaw should own the service runtime, the installer/service manager should move toward a pinned managed runtime and package layout.

## Reproduction Assessment

No. The issue includes concrete macOS/Homebrew logs matching current service and runtime-deps code paths, but there is no high-confidence deterministic reproduction for the combined `brew upgrade` ABI churn plus service/package/plugin-cache skew sequence in this checkout.

## Solution Assessment

Unclear. The suggested bundled/pinned runtime direction may be the broadest fix, but it is a platform ownership decision; a narrower maintainer-approved first step would be explicit diagnostics and repair around the chosen runtime policy.
