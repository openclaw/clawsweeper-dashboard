## Summary

Keep open. Current main has meaningful mitigations for service entrypoint drift, update restart verification, and bundled plugin runtime-deps cache drift, but macOS local mode still intentionally uses an external global CLI plus Homebrew/system Node runtime, so the Homebrew native ABI and atomic runtime ownership problem remains unresolved and distinct from the broader runtime cluster.

## What This Changes

The issue asks OpenClaw to survive macOS Homebrew/OpenClaw update cycles where Homebrew Node, the global OpenClaw package, the LaunchAgent command, plugin SDK files, and plugin runtime-deps caches can drift out of sync.

## Best Possible Solution

Maintainers should decide the macOS Gateway runtime authority first. If external Homebrew/system Node remains supported, doctor/status/update should explicitly validate and repair CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host compatibility, and runtime-deps stage keys; if OpenClaw should own the service runtime, the installer should move toward a pinned managed runtime/package layout.

## Reproduction Assessment

No. The report includes concrete macOS/Homebrew logs that map to current service and runtime-deps code paths, but there is no high-confidence deterministic reproduction in this checkout for Homebrew shared-library ABI churn combined with service/package/cache skew.

## Solution Assessment

Unclear. Current main mitigates stale paths and plugin runtime-deps drift, but the suggested bundled/pinned runtime direction is a macOS platform ownership decision rather than a narrow code-only repair.
