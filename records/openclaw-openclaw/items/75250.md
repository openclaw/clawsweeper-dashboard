## Summary

Keep open. Current main has meaningful mitigations for stale plugin runtime-deps roots, post-update gateway version checks, and plugin min-host skew, but the macOS documented and implemented local-mode path still depends on an external Node/global CLI service layout, so the reported Homebrew Node ABI and atomic runtime ownership problem is not fully resolved.

## What This Changes

The issue asks OpenClaw to make macOS Homebrew/global npm Gateway updates resilient across Node runtime, service entrypoint, host/plugin version, and bundled plugin runtime-deps cache drift.

## Best Possible Solution

Maintainers should choose the macOS Gateway runtime authority first. If external Homebrew/global npm remains supported, doctor/status/update should explicitly validate CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage keys, then repair or prune stale state before startup. If OpenClaw should own the service runtime, the installer and service manager should move toward a pinned managed runtime and package layout.

## Reproduction Assessment

No. The report provides concrete logs and paths that match current macOS service and plugin-runtime code paths, but this checkout does not provide a high-confidence deterministic reproduction for Homebrew shared-library ABI churn combined with service/package/plugin-cache skew.

## Solution Assessment

Unclear. A bundled or pinned runtime may be the most complete fix, but that is a platform ownership decision; the narrower maintainable first step is maintainer-approved diagnostics and repair around the chosen runtime policy.
