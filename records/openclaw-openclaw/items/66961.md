## Summary

Keep open. Current main still has a source-reproducible path where an unresolved custom provider with custom models can be serialized into generated `models.json`, and session model admission still reports catalog-visible but non-allowlisted models with a terse `model not allowed` error.

## What This Changes

The issue asks OpenClaw to keep generated `models.json` valid when custom providers lack resolved auth and to clarify or align session admission for discoverable Codex models.

## Best Possible Solution

Keep the explicit session model allowlist, make `models.json` planning omit or neutralize Pi-invalid unresolved provider rows, and improve blocked session/subagent model errors to explain the catalog-versus-allowlist distinction.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: configure a non-built-in provider with custom `models` and no resolvable `apiKey`, then `models.json` planning can serialize a row the pinned Pi registry rejects; a catalog-visible model outside `agents.defaults.models` still reaches the unchanged allowlist error path.

## Solution Assessment

Yes. The narrow maintainable fix is to filter only dependency-invalid unresolved custom-provider rows and improve diagnostics; automatically allowing every model from the `models list --all` catalog would weaken the documented allowlist policy.
