## Summary

Keep open. Current `main` still preserves explicit custom providers with models but no resolved `apiKey` into the generated `models.json` plan, and session model overrides still enforce `agents.defaults.models` with the terse `model not allowed` error even when a model is catalog-visible.

## What This Changes

The issue asks for `models.json` generation to omit or neutralize unresolved custom-provider rows and for session/subagent model admission to explain catalog-visible models blocked by the allowlist.

## Best Possible Solution

Keep the explicit model allowlist as the policy boundary, but make `models.json` planning emit only pi-valid provider rows and make blocked session/subagent model overrides explain the allowlist/catalog distinction.

## Reproduction Assessment

Yes. A source-level reproduction can use an explicit remote custom provider with `models` but no resolvable auth to show the generated plan preserves the invalid row, then use `sessions.patch` or `sessions_spawn` with a catalog-visible model absent from `agents.defaults.models` to get the allowlist rejection.

## Solution Assessment

Yes, the narrow repair is to filter or neutralize unresolved pi-invalid provider rows and improve diagnostics; silently allowing every model shown by `models list --all` would weaken the documented allowlist policy.
