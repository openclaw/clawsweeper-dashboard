## Summary

Keep open. Current main still permits unresolved custom-provider rows to reach the generated `models.json` plan, and session model overrides still collapse catalog-visible but non-allowlisted models to the terse `model not allowed` error.

## What This Changes

The issue requests that generated `models.json` omit unresolved custom providers and that session/subagent model rejection explain or align with catalog-visible Codex models.

## Best Possible Solution

Keep the explicit model allowlist as the session boundary, make `models.json` planning emit only Pi-valid provider rows, and improve blocked session/subagent model errors to explain the catalog-versus-allowlist distinction.

## Reproduction Assessment

Yes. Source inspection shows a non-built-in custom provider with `models` and no resolvable `apiKey` can survive into the generated plan, and a catalog-visible model absent from `agents.defaults.models` still reaches `model not allowed` through `sessions.patch`.

## Solution Assessment

Yes. The narrow maintainable fix is to filter or neutralize only Pi-invalid unresolved provider rows and improve diagnostics; automatically allowing every model from the all-catalog discovery view would weaken the documented allowlist policy.
