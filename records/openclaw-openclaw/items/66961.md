## Summary

Keep open. Current main still serializes explicit custom providers into generated models.json after auth normalization can leave custom-model providers without apiKey, and session model admission still returns the same terse allowlist rejection for catalog-visible but non-allowlisted models.

## What This Changes

The issue requests that generated models.json avoid unresolved custom-provider rows that poison Pi model discovery, and that session/subagent model admission either allow catalog-visible Codex models or explain the allowlist mismatch.

## Best Possible Solution

Keep the explicit session allowlist, filter or neutralize only Pi-invalid unresolved custom-provider rows before writing models.json, and improve session/subagent errors to explain catalog visibility versus agents.defaults.models admission.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a non-built-in provider with custom models and no resolvable apiKey, then the generated models.json can contain a row the pinned Pi registry rejects; a catalog-visible model outside agents.defaults.models still reaches the unchanged allowlist error path.

## Solution Assessment

Yes. The narrow fix is to preserve the allowlist policy while preventing invalid generated provider rows and making the rejection message actionable; automatically allowing every models.list --all row would be broader and less maintainable.
