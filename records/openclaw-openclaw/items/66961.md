## Summary

Keep open. Current main still leaves an explicit custom provider with models but no resolvable apiKey unchanged after provider normalization, which can still trip pi-coding-agent's custom-provider validation. The session side also still enforces `agents.defaults.models` for overrides and returns the terse `model not allowed` path, so the reported allowlist/catalog divergence is not fully resolved.

## What This Changes

Request that generated `models.json` exclude or neutralize unresolved custom providers and that session/subagent model admission handle discoverable Codex models consistently with the configured allowlist.

## Best Possible Solution

Keep the explicit model allowlist as the policy boundary, but make `models.json` planning produce only pi-valid provider rows and make session/subagent override errors explain when a catalog-visible model is blocked because it is absent from `agents.defaults.models`.

## Reproduction Assessment

Yes. A focused source-level reproduction can use a config with a non-built-in provider that has `models` but no resolvable auth to show the generated provider survives into `models.json`, and a `sessions.patch`/`sessions_spawn` model override with `agents.defaults.models` excluding `codex/gpt-5.4` to show the allowlist rejection.

## Solution Assessment

No, current main is not the best final state. The maintainable fix is targeted filtering or neutralization of unresolved invalid custom-provider rows plus clearer allowlist diagnostics, not silently allowing every model shown by `models list --all`.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/66961. Implement a narrow repair for two related model-runtime issues. First, make generated `models.json` omit or neutralize provider entries that would be invalid for pi-coding-agent: non-built-in/custom providers with `models.length > 0` and no resolvable request auth. Preserve intentional keyless/marker cases, including `auth: "aws-sdk"`, local providers, override-only rows (`models: []`), provider synthetic auth markers such as Codex, env/SecretRef marker strings, and any configured provider whose auth can be resolved from env or auth profiles. Second, keep `agents.defaults.models` as the session override allowlist, but improve the error path used by `sessions.patch`/`sessions_spawn` so a catalog-visible model rejected by the allowlist tells the user to add it to `agents.defaults.models`, clear the allowlist, or pick from the configured model list. Add focused unit coverage for the generator behavior and the session admission message. Do not auto-allow every `models list --all` row. Update docs only if user-facing wording changes, and add a `CHANGELOG.md` fix entry with `Thanks @drjontam` if behavior or user-facing diagnostics change.
