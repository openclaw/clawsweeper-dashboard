## Summary

Keep open. Current main still has both reported edges: unresolved explicit custom-provider rows can survive into the generated models.json plan, and session model overrides still return a terse allowlist rejection for catalog-visible models outside agents.defaults.models.

## What This Changes

The issue asks for models.json generation to omit or neutralize custom providers without resolved auth and for session/subagent model admission to clarify why models listed by discovery can still be blocked by the allowlist.

## Best Possible Solution

Keep the explicit model allowlist as the admission boundary, make models.json planning emit only Pi-valid provider rows, and improve blocked session/subagent model errors to explain the catalog-versus-allowlist distinction.

## Reproduction Assessment

Yes. A source-level reproduction can define a non-built-in custom provider with models but no resolvable apiKey and inspect the generated plan, then request a catalog-visible model absent from agents.defaults.models through sessions.patch or sessions_spawn to hit the allowlist rejection.

## Solution Assessment

Yes. The narrow maintainable fix is to filter or neutralize only Pi-invalid unresolved provider rows and improve diagnostics; automatically allowing every model from the all-catalog discovery view would weaken the documented allowlist policy.
