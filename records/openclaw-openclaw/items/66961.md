## Summary

Keep open. Current main still has both implicated paths: generated models.json can serialize unresolved non-built-in custom providers with models but no apiKey, and session model admission still reports a terse allowlist rejection for otherwise catalog-visible model refs.

## What This Changes

The issue requests models.json generation to avoid poisoning PI discovery with unresolved custom providers and asks session/subagent model admission to align with or clearly explain live model catalog visibility.

## Best Possible Solution

Filter or neutralize only PI-invalid unresolved custom-provider rows before writing models.json, keep the explicit agents.defaults.models allowlist, and make session/subagent rejection explain catalog visibility versus admission policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure a non-built-in provider with custom models and no resolvable apiKey, then generated models.json can contain a row the pinned PI registry rejects. A catalog-visible ref outside agents.defaults.models still reaches the existing allowlist rejection path.

## Solution Assessment

Yes. The maintainable fix is narrow: preserve the allowlist policy, prevent invalid generated provider rows from poisoning PI discovery, and improve the session/subagent error instead of allowing every models.list --all entry.
