## What This Changes

The issue asks OpenClaw to omit PI-invalid unresolved custom-provider rows from generated models.json and clarify session/subagent rejection for catalog-visible Codex models outside the configured allowlist.

## Best Possible Solution

Filter or neutralize only PI-invalid unresolved custom-provider rows before writing models.json, keep the explicit agents.defaults.models allowlist, and make session/subagent errors explain catalog visibility versus admission policy.

## Reproduction Assessment

Yes, at source level. Configure a non-built-in custom provider with models and no resolvable apiKey: current main can serialize that row to models.json, and the pinned PI registry rejects it; a catalog-visible but non-allowlisted ref still reaches the model not allowed branch.

## Solution Assessment

Yes. The narrow maintainable fix is to stop writing PI-invalid provider rows and improve the existing rejection message, not to bypass the configured agents.defaults.models allowlist or add a new policy knob.
