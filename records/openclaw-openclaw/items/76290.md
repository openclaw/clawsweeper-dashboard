## Summary

Keep open. Current main still has a plausible double-unset path for indexed arrays: the CLI removes the requested array element, then the write layer applies the same indexed unset path again before persisting.

## What This Changes

Fix `openclaw config unset 'agents.list[1]'` so it removes only the selected array element and preserves later entries.

## Best Possible Solution

Make indexed config unsets idempotent across the CLI and write-preparation layers, then add a regression that persists `[agent-a, agent-c]` after unsetting `agents.list[1]`.

## Reproduction Assessment

Yes. The source path is high-confidence: current main first splices the array in `runConfigUnset`, then write preparation receives and splices the same indexed `unsetPaths` entry again.

## Solution Assessment

Yes, a narrow config CLI/write-path fix is the best direction. The fix should preserve the existing runtime-default and include/env-ref protections while preventing indexed array unsets from being applied twice.

Fix https://github.com/openclaw/openclaw/issues/76290. Current main first removes `agents.list[1]` in `runConfigUnset`, then passes the same parsed path via `writeOptions.unsetPaths`; the write layer applies that indexed unset again, so `[agent-a, agent-b, agent-c]` persists as `[agent-a]` instead of `[agent-a, agent-c]`. Keep the change scoped to config CLI/write-preparation behavior. Preserve prototype-key blocking, runtime-default suppression, include/env-ref handling, and existing object unset semantics. Do not change general config path syntax. Add a regression through the public CLI command or write path proving that unsetting `agents.list[1]` from a 3-entry list persists the first and third entries only, and include a user-facing `CHANGELOG.md` fix entry unless maintainers classify the patch as internal-only.
