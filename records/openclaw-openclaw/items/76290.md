## Summary

Keep open. Current main still has the double-apply path for indexed array unsets, and the linked open PR #76294 is the active implementation candidate rather than a reason to close this issue early.

## What This Changes

`openclaw config unset 'array[index]'` should remove only the selected array element and preserve later entries.

## Best Possible Solution

Land a narrow fix like #76294 that makes indexed array unsets apply once while preserving the existing object-key unset write protections, with a regression for `agents.list[1]` persisting `[agent-a, agent-c]`.

## Reproduction Assessment

Yes. Source inspection on current main shows `runConfigUnset` splices the indexed array path, then passes the same path to write preparation, which can splice the shifted array again during persistence.

## Solution Assessment

Yes. The narrow maintainable fix is to keep object-key `unsetPaths` handling but avoid replaying indexed leaf array deletions after the CLI has already carried the spliced array into `nextConfig`; #76294 appears aligned and should be reviewed or landed before this issue closes.
