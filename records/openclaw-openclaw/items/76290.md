## Summary

Keep open. The report matches current main: `config unset` removes the array element in the CLI clone, then passes the same path to the config writer, where unset paths are applied again and can delete the shifted successor element.

## What This Changes

The issue asks for `openclaw config unset 'array[index]'` to delete only the selected array element while preserving later entries.

## Best Possible Solution

Change the config unset/write contract so array-index unsets are applied exactly once while object-key unset paths still suppress runtime defaults from being reintroduced.

## Reproduction Assessment

Yes. A static reproduction is clear: with `[agent-a, agent-b, agent-c]`, the CLI splice produces `[agent-a, agent-c]`, then the writer applies the same index unset again and can persist `[agent-a]`, matching the report.

## Solution Assessment

Yes, the reporter's jq-like single-element deletion is the right behavior; the narrow repair should prevent duplicate write-time array removal rather than reject supported bracket paths.
