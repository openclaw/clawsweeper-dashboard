## Summary

Keep open. Current main still skips oversized parent-session inheritance for channel/thread session init by logging a warning and starting the child isolated; it does not auto-compact the parent or emit a channel-visible warning for the reported Discord-style cold starts. The legacy `session.parentForkMaxTokens` knob has been removed and subagent forks now surface a note, but those changes do not close the user-visible amnesia gap described here.

## What This Changes

Request warning, auto-tuning, or parent compaction when parent-session inheritance is skipped because the parent exceeds the fork token cap.

## Best Possible Solution

Add a session-owned, rate-limited visible warning when parent inheritance is skipped, while leaving automatic parent compaction and model-aware cap calculation for a separate maintainer-reviewed design pass.

## Reproduction Assessment

Yes. Existing tests reproduce the current behavior by creating parent sessions above the threshold and verifying that current main starts a fresh child and marks it forked; this read-only review inspected those tests and the live branch source rather than executing them.

## Solution Assessment

No for the current behavior. The overflow guard is necessary to avoid inheriting unusable context, but log-only isolation is not the best user experience; the narrowest maintainable next fix is a visible warning, with parent auto-compaction treated as a larger compaction-contract change.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/63141: when `initSessionState()` skips parent-session forking because `resolveParentForkDecision()` returns `parent-too-large`, surface a rate-limited visible warning to the originating channel/thread so users know the child is starting without inherited context. Keep the existing overflow guard and fresh-child behavior. Do not implement automatic parent compaction or model-aware cap changes in this PR; those need maintainer review because of #43661. Prefer a session-owned notice field/result that is emitted once per skipped child session, and reuse the existing `Parent context is too large to fork...` wording or a concise user-facing variant. Update focused tests for channel/thread session init and any get-reply plumbing, keep subagent fallback behavior working, update public docs if behavior text changes, and add a user-facing `CHANGELOG.md` fix entry without bot attribution.
