## Summary

Keep open. Current main still keeps the overflow guard but does not solve the user-visible cold-start problem for channel/thread forks: oversized parents are skipped with a subsystem log and isolated child context, while only the subagent path surfaces the fallback note.

## What This Changes

Request automatic parent-fork recovery or a visible warning when an oversized parent session prevents child sessions from inheriting context.

## Best Possible Solution

Add a rate-limited channel-visible parent-fork fallback notice for oversized parent sessions; defer parent auto-compaction and model-aware cap changes to separate compaction design work.

## Reproduction Assessment

Yes. Source inspection and existing tests show a reproducible current-main path: create a parent session over 100,000 tokens, initialize a child with `ParentSessionKey`, and the child starts isolated while the channel path only logs the skip.

## Solution Assessment

No for the current behavior. The overflow guard is necessary to avoid inheriting unusable context, but log-only isolation does not meet the requested warning/recovery behavior; the narrow maintainable next step is a visible, rate-limited fallback notice.

## Repair Work Prompt

Implement the narrow visible-warning path for https://github.com/openclaw/openclaw/issues/63141. When `initSessionState()` skips a parent fork because `resolveParentForkDecision()` returns `status: "skip"`, propagate a sanitized fallback notice to the reply/delivery path and send it once or rate-limited to the originating channel/thread so users understand the child started with isolated context. Do not auto-compact the parent, do not make the 100k cap model-aware, and do not reintroduce `session.parentForkMaxTokens`. Preserve the existing subagent `forkFallbackNote` behavior. Add regression coverage for an oversized channel/thread parent producing a visible warning on first skip and not repeatedly warning on later turns. Update docs and `CHANGELOG.md` because this is user-facing.
