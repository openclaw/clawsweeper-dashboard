## Summary

Keep open. Current main still uses the static parent-fork cap and still marks oversized parents as already forked after only a log warning, so the reported cold child-session behavior remains present despite the related compaction provider and checkpoint infrastructure.

## What This Changes

Request to prevent oversized parent-session fork skips from silently creating cold child sessions by auto-compacting the parent, warning visibly, or deriving a model-aware cap.

## Best Possible Solution

Keep the issue open and implement a session-owned remediation: at minimum, emit a rate-limited visible/operator notice when parent inheritance is skipped, and optionally add bounded parent pre-compaction or a model-aware cap only if it can reuse existing context-window and compaction infrastructure without expanding startup/runtime coupling.

## Reproduction Assessment

Yes. The existing session test constructs a parent with totalTokens above the default cap and verifies the child is marked forked but starts with a fresh session/file; the current source still follows that path.

## Solution Assessment

No. The current skip guard prevents immediate context overflow, but it is not the best end-user behavior because it silently drops inherited context; the narrowest maintainable improvement is a visible, rate-limited warning, with parent pre-compaction as a more invasive follow-up.
