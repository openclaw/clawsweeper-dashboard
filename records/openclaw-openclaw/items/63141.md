## Summary

Keep open. Current main still uses a static parent-fork token cap and still handles oversized parents by logging a warning, marking the child as forked, and starting it fresh; the related compaction provider and checkpoint work exists, but it is not wired into this overflow branch.

## What This Changes

The issue requests auto-compaction, a visible warning, or model-aware tuning when parent-session forking is skipped because `session.parentForkMaxTokens` is exceeded.

## Best Possible Solution

Keep the issue open and implement a session-owned remediation first: emit a rate-limited visible warning when parent inheritance is skipped, while leaving auto-compaction and model-aware cap changes for a maintainer-reviewed follow-up that can account for compaction timeout behavior and provider contracts.

## Reproduction Assessment

Yes. The existing session tests build parent sessions above the default cap and verify current main marks the child as forked while starting it with a fresh session/file; source inspection shows the same branch remains active.

## Solution Assessment

No for the current behavior. The skip guard prevents inherited-context overflow, but it is not the best user experience because it drops context with only a log warning; the narrowest maintainable repair is a visible, rate-limited warning, with automatic parent compaction as a larger design choice.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/63141 with the narrow visible-warning path only. Current main skips parent-session inheritance when `parentTokens > session.parentForkMaxTokens`, logs a warning, sets `forkedFromParent`, and starts the child fresh. Add a session-owned, rate-limited or persisted user/operator-visible warning so the originating channel understands that the child thread started without parent context. Do not auto-compact the parent, change the default `100000` cap, or derive the cap from model context windows unless a maintainer explicitly expands scope. Prefer a small contract such as a typed skip/notice result from `initSessionState()` or a persisted notice signature on `SessionEntry`, then surface it through the reply path without duplicate spam. Update focused tests for the oversized parent skip and the delivery/reply path, update docs if the visible behavior is documented, and add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @Hugo0`. Preserve existing behavior where setting `session.parentForkMaxTokens: 0` disables the guard and where larger configured caps allow forking.
