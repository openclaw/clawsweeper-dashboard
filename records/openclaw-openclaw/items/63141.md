## Label Justifications

- `P2`: The issue reports real production context-continuity pain, but the safest next step is a bounded user-visible warning rather than urgent runtime recovery.
- `impact:session-state`: Oversized parent-fork skips start child sessions without inherited context, making the agent appear to forget recent conversation state.

## Summary

Keep open. Current main and v2026.5.19 still use the static 100,000-token parent-fork cap, and the channel/thread `session-init` path still handles an oversized parent by logging a warning and starting the child with isolated context, without a visible channel warning, pre-fork compaction, or model-aware cap sizing.

## What This Changes

The issue requests that oversized parent session forks stop failing silently by adding a visible warning, pre-fork compaction, or model-aware token cap sizing.

## Best Possible Solution

Add a session-owned, deduped visible warning when channel/thread parent-fork inheritance is skipped, while leaving automatic compaction and model-aware cap sizing to separate maintainer-approved work.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence reproduction path: create a parent session above 100,000 tokens, initialize a child with `ParentSessionKey`, and current main marks the child forked while using fresh isolated context after a log-only skip.

## Solution Assessment

Yes for the narrow warning slice. A deduped visible channel/thread warning is the smallest maintainable improvement; automatic parent compaction and dynamic cap sizing are broader compaction/model-budget policy changes.
