## Summary

Keep open. Current main still tracks the ignored generated A2UI hash, the dev update preflight build still starts with A2UI bundling and can rewrite that hash, and the fallback loop still reuses the same preflight worktree with plain detached checkouts after candidate failures. The linked minimal PR was closed unmerged, and the related open A2UI reproducibility issue does not cover the updater checkout-conflict hardening.

## What This Changes

The issue asks OpenClaw to stop `openclaw update` preflight walkback failures caused by a tracked generated A2UI bundle hash becoming dirty between fallback checkouts.

## Best Possible Solution

Harden the git-install updater so each disposable preflight candidate starts from a clean worktree before checkout, then add a regression test that simulates a failed candidate leaving a tracked generated file dirty before the next fallback. Re-untracking the A2UI hash can still be a companion cleanup, but updater resilience is the durable fix for this class of generated-file conflicts.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current main tracks an ignored generated hash, the preflight build can rewrite it, and the fallback loop attempts the next candidate with a plain checkout in the same worktree after failures.

## Solution Assessment

Yes, queueing a narrow updater robustness fix is the best path. Re-untracking the hash addresses this artifact, but cleaning or resetting the disposable preflight worktree before each candidate is the maintainable fix for similar future generated-file dirtiness.

## Repair Work Prompt

Fix the `openclaw update` dev preflight walkback bug described in openclaw/openclaw#64818. Current main still tracks the ignored generated `src/canvas-host/a2ui/.bundle.hash`; the preflight build can rewrite it; then `src/infra/update-runner.ts` reuses one preflight worktree and tries the next candidate with plain `git checkout --detach`, which can fail with local-change conflicts and end in `preflight-no-good-commit`.

Implement a narrow updater-side robustness fix: before each fallback candidate checkout in the disposable preflight worktree, clean/reset the worktree state enough that generated tracked files from the previous failed candidate cannot block the next checkout. Keep this scoped to the preflight worktree; do not reset or clean the user's real checkout. Preserve existing Windows cleanup behavior and package-manager preflight flow.

Add or update a regression test in `src/infra/update-runner.test.ts` that simulates a candidate failing after leaving a tracked generated file dirty, then verifies the updater still attempts and can select a later fallback candidate instead of bailing from checkout conflicts. A companion cleanup to untrack `src/canvas-host/a2ui/.bundle.hash` is acceptable only if it stays aligned with `.gitignore`, but the updater resilience is the required fix.

Likely files: `src/infra/update-runner.ts`, `src/infra/update-runner.test.ts`; optionally `src/canvas-host/a2ui/.bundle.hash` only for a deliberate generated-artifact tracking cleanup. Validation: `pnpm test src/infra/update-runner.test.ts`; run `pnpm check:changed` in Testbox before handoff if code/test/runtime files changed. Changelog: add a user-facing fix entry if the updater behavior changes, with proper non-bot attribution. Do not modify unrelated A2UI bundle reproducibility work from #67547 or the open signal-polyfill PR #68934.
