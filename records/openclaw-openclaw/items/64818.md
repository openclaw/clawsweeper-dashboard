## Summary

Keep open. Current main still has the reported failure shape: the generated A2UI hash is tracked while ignored, the dev preflight build can rewrite it, and the fallback loop reuses the same worktree with plain checkouts and no per-candidate cleanup. The linked minimal PR is closed unmerged, and the remaining related A2UI issue does not cover updater walkback resilience.

## What This Changes

The issue asks OpenClaw to prevent `openclaw update` dev preflight walkback from being blocked by a dirty tracked A2UI bundle hash generated during failed fallback candidates.

## Best Possible Solution

Harden the git-install updater so every disposable preflight candidate starts from a clean worktree before checkout, then add a regression test for a failed candidate leaving a tracked generated file dirty. Re-untracking the hash file is useful cleanup, but the updater should be resilient to the broader class of generated-file dirtiness.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: dirty a tracked `.bundle.hash`, then attempt a checkout of another commit with a different blob; the current preflight loop performs that same plain checkout after build or lint failures in a reused worktree.

## Solution Assessment

Yes, if the fix targets updater robustness rather than only the A2UI artifact. Cleaning or resetting the disposable preflight worktree before each candidate is the narrow maintainable fix; untracking `.bundle.hash` alone would only address this specific file.

## Repair Work Prompt

Fix the git-install `openclaw update` dev preflight walkback bug from https://github.com/openclaw/openclaw/issues/64818. Current main tracks `src/canvas-host/a2ui/.bundle.hash` even though it is ignored, `pnpm build` can rewrite it through `scripts/bundle-a2ui.mjs`, and `src/infra/update-runner.ts` reuses one preflight worktree while trying plain `git checkout --detach <sha>` for each fallback candidate. Implement the fix in the updater path so each disposable preflight candidate starts from a clean worktree before checkout, or otherwise makes checkout robust against dirty tracked generated files left by the previous failed candidate. Add a focused regression test in `src/infra/update-runner.test.ts` that simulates a failed candidate dirtying a tracked generated file and proves the next candidate can still be checked out and selected. Prefer updater robustness over relying only on re-untracking `.bundle.hash`; do not change dependency versions or broad A2UI bundle behavior. Add a user-facing `CHANGELOG.md` fix entry with a valid non-bot/non-maintainer attribution if the repair changes runtime behavior. Related context: closed unmerged #64822 proposed untracking the hash; open #67547 covers A2UI reproducibility and should not be treated as the same fix.
