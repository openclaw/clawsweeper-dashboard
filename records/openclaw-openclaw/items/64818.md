## Summary

Keep open. The original `src/canvas-host/a2ui` path is gone on current main, but the same failure class still exists under the Canvas plugin: the plugin A2UI hash is tracked while ignored, a read-only recompute shows the checked-in hash is stale, and the dev preflight loop still reuses one worktree with plain checkout attempts after failed candidates.

## What This Changes

The issue asks OpenClaw update preflight to survive generated A2UI hash dirtiness during candidate walkback instead of ending with `preflight-no-good-commit`.

## Best Possible Solution

Harden the disposable dev preflight worktree so failed candidates cannot leave tracked generated-file dirt that blocks later checkout attempts; hash refreshes or path-specific untracking can be separate cleanup, but they should not be the only updater fix.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main has a stale tracked Canvas A2UI hash that the build hook would rewrite, and the preflight walkback still performs plain checkout attempts in the same worktree after failed candidates.

## Solution Assessment

Yes, updater hardening is the best narrow fix. Re-untracking or refreshing one hash marker is brittle because the Canvas plugin hash has already drifted again after a merged refresh-only PR.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/64818 by hardening dev update preflight walkback against generated tracked-file dirtiness. Current main creates one disposable preflight worktree in `src/infra/update-runner.ts`, runs `git checkout --detach` for each candidate, then runs deps/build/lint and continues on failure without resetting the worktree before the next checkout. The Canvas A2UI build hook can dirty tracked `extensions/canvas/src/host/a2ui/.bundle.hash`; a read-only recompute showed the current hash is already stale. Add a narrow fix that resets or otherwise cleans only the disposable preflight worktree before retrying/checking out the next candidate, without resetting the user's main checkout and without making hash untracking the only solution. Add focused coverage in `src/infra/update-runner.test.ts` that simulates a failed candidate leaving a tracked generated file dirty and verifies a later candidate checkout can still proceed. Include a changelog entry if the updater behavior change is user-visible.
