## Label Justifications

- `impact:session-state`: The issue is about the first outbound message missing from the routed agent's persisted session transcript and context.

## Summary

Keep open: current main and v2026.5.19 still choose explicit, session, or default agent ids before channel/account route bindings can select the routed agent, so the outbound-first WhatsApp session continuity request remains unimplemented.

## What This Changes

Add support for outbound WhatsApp sends to create or resume the routed agent session and persist the first outbound message into that transcript.

## Best Possible Solution

Add a shared outbound agent/session resolver that honors explicit agent or session first, then channel/account/peer route bindings, then the default agent, and reuse it across CLI, Gateway, and outbound action runner paths before transcript mirroring.

## Reproduction Assessment

Yes, by source inspection. Configure a non-default WhatsApp account route, send with `openclaw message send --channel whatsapp --account <bound-account>`, and current main selects explicit/session/default agent ids before account route bindings can choose the session to mirror into.

## Solution Assessment

Yes, if the fix is the shared resolver described here. A WhatsApp-only implementation or mandatory new flag would duplicate routing policy and risk drifting from inbound routing behavior.
