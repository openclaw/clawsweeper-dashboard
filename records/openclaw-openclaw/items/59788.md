- `impact:session-state`: The issue is about the first outbound message missing from the routed agent's persisted session transcript and context.

## Summary

Keep open: current main and latest release v2026.6.6 still resolve outbound sends to an explicit/session/default agent before account bindings can select the routed non-default agent, so the outbound-first WhatsApp transcript continuity request remains unimplemented.

## What This Changes

Requests session-aware outbound channel sends so `openclaw message send` can create or resume the routed agent session and persist the first outbound WhatsApp message into that transcript.

## Best Possible Solution

Add a shared outbound agent/session resolver that honors explicit agent or session first, then channel/account/peer route bindings, then the default agent, and reuse it across CLI, Gateway, and outbound transcript mirroring.

## Reproduction Assessment

Yes. Source inspection shows a non-default WhatsApp account route plus `openclaw message send --channel whatsapp --account <bound-account>` still selects the default agent before account bindings can choose the routed session.

## Solution Assessment

Unclear until maintainers choose automatic route inference versus explicit opt-in. If accepted, the narrow maintainable path is a shared outbound resolver rather than a WhatsApp-only implementation.
