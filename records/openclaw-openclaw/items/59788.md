## Summary

Keep open: current main and shipped v2026.5.19 still preselect an explicit/session/default agent before account route bindings can choose the routed non-default agent, so the outbound-first WhatsApp session continuity request remains unimplemented.

## What This Changes

The issue requests that outbound `openclaw message send` for a routed WhatsApp account create or resume the correct routed agent session and persist the first outbound message into that transcript.

## Best Possible Solution

Add a shared outbound agent/session resolver that honors explicit agent or session first, then channel/account/peer route bindings, then the default agent, and reuse it across CLI, Gateway, and outbound action-runner transcript mirroring.

## Reproduction Assessment

Yes, by source inspection. Configure a non-default WhatsApp account route, send with `openclaw message send --channel whatsapp --account <bound-account>`, and current main selects explicit/session/default agent ids before account route bindings can choose the session to mirror into.

## Solution Assessment

Yes, if the fix is the shared resolver described here. A WhatsApp-only implementation or mandatory new flag would be less maintainable unless maintainers explicitly choose opt-in semantics.
