## Summary

Keep open: current main still mirrors outbound sends only after an agent has already been selected, and the CLI/Gateway paths still fall back to the default or explicit session agent before consulting channel/account bindings. The non-default WhatsApp account-routed outbound-first workflow remains unimplemented.

## What This Changes

The issue requests outbound `openclaw message send` and similar sends to create or resume the routed agent session and persist the first outbound turn in that transcript.

## Best Possible Solution

Add a shared outbound agent resolver that honors explicit agent/session first, then channel/account/peer route bindings, then the default agent, and use it before outbound session-route derivation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure a non-default agent bound to a WhatsApp account, run `openclaw message send --channel whatsapp --account <bound-account>`, and current main still selects the default agent before outbound mirroring can use account bindings.

## Solution Assessment

Yes. The maintainable fix is a generic route-based outbound agent resolver reused by CLI, Gateway, and the action runner, not a WhatsApp-only branch or a required new flag.
