## Summary

Keep open: current main and v2026.6.8 still do not infer the routed non-default agent from an outbound WhatsApp account send, so the first outbound turn is mirrored only for the selected/default agent path. The remaining work is real but needs maintainer product direction on automatic route inference versus explicit opt-in.

## What This Changes

Request to have outbound `openclaw message send` through routed WhatsApp accounts create or resume the routed agent session and persist the first outbound turn.

## Best Possible Solution

Add a maintainer-approved generic outbound agent/session resolver shared by CLI, Gateway send, and message-tool paths that preserves explicit session or agent first, then applies routed channel/account/peer bindings under the chosen opt-in or default semantics.

## Reproduction Assessment

Yes, at source level. Current main selects the default agent before outbound account routing and mirrors only for that selected agent, so a WhatsApp account bound to a non-default agent will not populate that agent's transcript from `openclaw message send`.

## Solution Assessment

Unclear until maintainers choose automatic account-route inference versus an explicit opt-in flag or command. The maintainable implementation should be a shared outbound resolver, not a WhatsApp-only special case.
