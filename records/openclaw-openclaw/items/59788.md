## Summary

Keep open: current main can mirror outbound sends once an agent is already selected, but it still chooses the default CLI/Gateway agent before resolving an explicit WhatsApp account binding. That leaves the requested non-default, account-routed outbound-first workflow unimplemented.

## What This Changes

The issue requests session-aware outbound WhatsApp sends that resolve the routed non-default agent, create or resume the matching DM session, and persist the opener before the first inbound reply.

## Best Possible Solution

Add a generic outbound agent resolver before outbound session-route derivation. Explicit `agentId` and `sessionKey` should keep precedence, then channel/account/peer route bindings should select the agent, and only unmatched sends should fall back to the default agent; avoid a WhatsApp-only special case.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure a non-default agent with a WhatsApp account binding, run `openclaw message send --channel whatsapp --account <bound-account>`, and current main still passes the default agent into outbound routing without reverse-resolving that account binding.

## Solution Assessment

Yes, the requested session-aware outbound flow is the right direction, but the maintainable fix is a shared route-based resolver rather than only adding a WhatsApp-specific branch or an explicit flag.
