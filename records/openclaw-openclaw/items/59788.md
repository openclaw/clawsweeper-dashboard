- `impact:session-state`: The issue is about the first outbound message missing from the routed agent's persisted session transcript and context.

## Summary

Keep open: current main and the latest shipped release still select explicit/session/default agent ids before consulting channel/account route bindings, so the outbound-first non-default WhatsApp account workflow remains unimplemented.

## What This Changes

The issue requests outbound `openclaw message send` and similar channel sends to resolve the routed agent/session and persist the first outbound WhatsApp message into that session transcript.

## Best Possible Solution

Add a shared outbound agent/session resolver that honors explicit agent or session first, then channel/account/peer route bindings, then the default agent, and reuse it across CLI, Gateway, and outbound action runner paths before transcript mirroring.

## Reproduction Assessment

Yes, by source inspection. Configure a non-default WhatsApp account route, send with `openclaw message send --channel whatsapp --account <bound-account>`, and current main selects explicit/session/default agent ids before account route bindings can choose the session to mirror into.

## Solution Assessment

Yes, if the fix is the shared resolver described above. A WhatsApp-only branch or mandatory new flag would duplicate routing policy and risk drifting from inbound routing behavior.
