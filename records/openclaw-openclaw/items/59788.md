## Summary

Keep open: current main still mirrors outbound sends only after an agent has already been selected, and the CLI/Gateway paths still choose explicit/session/default agent ids before consulting channel/account route bindings. The outbound-first non-default WhatsApp account workflow remains unimplemented.

## What This Changes

Add route-aware outbound message sends that create or resume the bound agent session and persist the first outbound turn before a reply arrives.

## Best Possible Solution

Add a generic outbound agent/session resolver that honors explicit agent or session first, then channel/account/peer route bindings, then the default agent, and reuse it before outbound mirror route derivation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a non-default agent bound to a WhatsApp account, run `openclaw message send --channel whatsapp --account <bound-account>`, and current main selects the default agent before outbound mirroring can use account bindings.

## Solution Assessment

Yes. The maintainable fix is a shared generic route-based outbound resolver reused by CLI, Gateway, and the action runner, not a WhatsApp-only branch or a required new flag.
