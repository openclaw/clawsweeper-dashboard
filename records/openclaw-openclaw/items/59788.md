## Summary

Keep open: current main and v2026.6.10 still select the default, explicit, or session agent before account-route inference, so an outbound-first WhatsApp send can still miss the routed non-default agent transcript.

## What This Changes

The issue requests session-aware outbound `openclaw message send` behavior that creates or resumes the routed agent session and persists the first outbound WhatsApp turn before a reply arrives.

## Best Possible Solution

Add a maintainer-approved shared outbound agent/session resolver for CLI, Gateway send, and message-tool paths that preserves explicit agent/session precedence and then applies account/peer route bindings under the chosen automatic or opt-in semantics.

## Reproduction Assessment

Yes, at source level: current main selects the default or explicit/session agent before outbound account routing and mirrors only for that selected agent. I did not run a live WhatsApp scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose automatic account-route inference versus explicit opt-in semantics. The narrow maintainable shape is a shared outbound resolver, not a WhatsApp-only special case.
