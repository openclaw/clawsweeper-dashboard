## Summary

Keep open: current main and v2026.6.8 still do not infer the routed non-default agent from an outbound WhatsApp account send, so the outbound-first transcript continuity request remains unimplemented and needs maintainer product direction.

## What This Changes

Request to make outbound WhatsApp `openclaw message send` create or resume the routed agent session and persist the first outbound message in that transcript.

## Best Possible Solution

Add a generic outbound agent/session resolver shared by CLI, Gateway send, and message-tool paths that honors explicit session or agent first, then routed channel/account/peer bindings under maintainer-approved default or opt-in semantics.

## Reproduction Assessment

Yes. Source inspection against current main shows CLI supplies the default agent to `runMessageAction` before account bindings can infer a non-default routed agent; no live WhatsApp send was run.

## Solution Assessment

Unclear until maintainers choose automatic route inference versus explicit opt-in. If accepted, the maintainable fix is a shared outbound resolver rather than a WhatsApp-only path.
