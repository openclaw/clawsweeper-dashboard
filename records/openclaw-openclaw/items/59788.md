## Summary

Keep open: current main and v2026.6.8 still select the default agent before outbound session routing, so an account-only WhatsApp send cannot infer the routed non-default agent and persist the opener into that agent's transcript.

## What This Changes

The issue requests outbound WhatsApp message sends to create or resume the routed agent session and persist the initial outbound message in the transcript.

## Best Possible Solution

Add a generic outbound agent/session resolver shared by CLI, Gateway send, and message-tool paths that honors explicit session or agent first, then routed channel/account/peer bindings under maintainer-approved default or opt-in semantics.

## Reproduction Assessment

Yes, source-level: current CLI sends provide the default agent before outbound session routing, so account-only sends cannot infer the non-default routed agent from the WhatsApp account binding. I did not run a live WhatsApp send in this read-only checkout.

## Solution Assessment

Unclear until maintainers choose automatic inference versus explicit opt-in. The best implementation shape appears to be a shared outbound resolver rather than a WhatsApp-only special case.
