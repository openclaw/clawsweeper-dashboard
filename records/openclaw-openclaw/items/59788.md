## Summary

Keep open: current main and v2026.6.10 still anchor outbound `message send` to an explicit/session/default agent before account-route inference, so a WhatsApp account bound to a non-default agent is not selected as the first outbound transcript owner. The issue is valid source-reproducible work, but it still needs maintainer direction on automatic route inference versus an explicit opt-in send mode.

## What This Changes

The issue requests outbound WhatsApp sends to create or attach to the routed agent session and persist the first outbound message in that transcript.

## Best Possible Solution

Add a maintainer-approved shared outbound agent/session resolver for CLI, Gateway send, and message-tool paths that preserves explicit agent/session precedence, then applies account/peer route bindings under the chosen automatic or opt-in semantics.

## Reproduction Assessment

Yes, at source level: current main selects the default or explicit/session agent before outbound account routing and mirrors only for that selected agent. I did not run a live WhatsApp scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose automatic account-route inference versus explicit opt-in semantics. The narrow maintainable shape is a shared outbound resolver, not a WhatsApp-only special case.
