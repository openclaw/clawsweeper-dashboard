- `impact:session-state`: The reported impact is a delivered first outbound message missing from the routed agent's persisted session transcript and later context.

## Summary

Keep open: current main and v2026.6.8 still choose an explicit/session/default agent before outbound routing, so a WhatsApp account bound to a non-default agent is not inferred as the transcript owner for the first outbound send. The remaining work is valid but needs maintainer direction on automatic account-route inference versus explicit opt-in semantics.

## What This Changes

Request outbound `openclaw message send` and similar channel sends to create or resume the WhatsApp-routed agent session and persist the first outbound message in that transcript.

## Best Possible Solution

Add a maintainer-approved shared outbound agent/session resolver for CLI, Gateway send, and message-tool paths that preserves explicit session or agent precedence, then applies account/peer route bindings under the chosen automatic or opt-in semantics.

## Reproduction Assessment

Yes, at source level: current main selects the default or explicit/session agent before outbound account routing and mirrors only for that selected agent. I did not run a live WhatsApp scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose automatic account-route inference versus an explicit opt-in flag or command. The maintainable fix should be a shared outbound resolver rather than a WhatsApp-only special case.
