## Summary

Keep open: current main has outbound transcript mirroring, but the CLI and Gateway outbound paths still choose the default or explicitly supplied agent before consulting channel/account bindings, so the non-default WhatsApp account-routed workflow remains unimplemented.

## What This Changes

Request routed outbound sends, especially `openclaw message send --channel whatsapp --account <bound-account>`, to create or resume the bound agent session and persist the first outbound turn before an inbound reply arrives.

## Best Possible Solution

Add a shared outbound agent resolver that honors explicit agent/session first, then channel/account/peer route bindings, then the default agent, and use it before outbound session-route derivation.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure a non-default agent bound to a WhatsApp account, run `openclaw message send --channel whatsapp --account <bound-account>`, and current main still passes the default agent into outbound mirroring before account bindings can select the routed agent.

## Solution Assessment

Yes. The requested automatic session-aware outbound flow is the maintainable direction, but it should be implemented as a generic route-based resolver rather than a WhatsApp-only branch or a required new flag.
