## Label Justifications

- `P1`: The report describes silent loss of user-visible assistant content on a reasoning-provider streaming path observed by a real user.
- `impact:message-loss`: Visible reply text can be hidden from the user when native reasoning and visible content split an inline-code span.
- `impact:session-state`: The persisted assistant message and adjacent thinking block no longer match the intended visible transcript.

## Summary

Keep open: current main and the latest release still route native `reasoning_content` separately from partitioned visible `content`, so the reported inline-code boundary problem is not implemented away. No merged or open canonical PR owns this exact case, and the remaining fix needs a maintainer-approved rule that avoids trading message loss for reasoning leakage.

## What This Changes

The issue requests fixing OpenAI-compatible reasoning stream routing so `content` and `reasoning_content` in the same chunk cannot split an inline code span and hide visible reply text in a thinking block.

## Best Possible Solution

Define and implement one shared provider/agent streaming rule that preserves visible inline-code continuations without exposing arbitrary native thinking, with regression coverage in both OpenAI-compatible stream paths.

## Reproduction Assessment

Yes at source level: current routing still sends native `reasoning_content` through thinking-block appenders while the inline-code-aware partitioner only sees `content`. I did not run a live provider or local test in this read-only cleanup review.

## Solution Assessment

Unclear as an implementation choice: routing native reasoning through the partitioner is plausible, but maintainers should first confirm the privacy and ordering contract for same-chunk visible content plus native reasoning.
