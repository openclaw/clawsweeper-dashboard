- `impact:session-state`: The issue is about stale isolated cron session transcripts and session entries accumulating after completed runs.

## Summary

Keep open: current main still treats `deleteAfterRun` cleanup as a delivery-path side effect, so isolated cron runs that skip delivery or end through terminal error paths can leave transcript/session files for delayed retention cleanup instead of deleting them at run completion.

## What This Changes

The issue reports that isolated `agentTurn` cron jobs with `deleteAfterRun: true` leave per-run session `.jsonl` files on disk after completion.

## Best Possible Solution

Centralize `deleteAfterRun` transcript cleanup in the terminal isolated-run lifecycle so every completed isolated cron run cleans up its own transcript while preserving retention reaping, one-shot job deletion, and active-descendant safety.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run an isolated `agentTurn` cron job with `deleteAfterRun: true` where delivery is disabled, skipped, or the run returns through the terminal catch path, and the current delivery-local cleanup helper is not reached.

## Solution Assessment

Yes. The narrow maintainable fix is to move the existing cleanup contract to a run-final isolated cron path rather than adding a new public cleanup option or relying on the 24-hour retention reaper.
