- `impact:session-state`: The report is about stale isolated cron session transcripts and metadata accumulating after completed runs.

## Summary

Keep open: current `main` and latest `v2026.5.19` still only run `deleteAfterRun` transcript deletion from selected direct-delivery paths, so isolated cron runs with no delivery, skipped delivery, or terminal errors can leave their run session files for retention cleanup instead of deleting them at completion.

## What This Changes

Isolated cron jobs with `deleteAfterRun: true` should delete their per-run session entry and transcript file after the run completes.

## Best Possible Solution

Move `deleteAfterRun` transcript/session cleanup into an isolated-run finalization path that executes for successful, silent/no-delivery, skipped-delivery, and terminal error outcomes while preserving retention reaping as a delayed fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run an isolated `agentTurn` cron job with `deleteAfterRun: true` and no direct delivery or a terminal run error, and the current delivery-local cleanup is not reached.

## Solution Assessment

Yes. The narrow maintainable fix is to share or move the existing `sessions.delete` cleanup semantics into a run-final cleanup path instead of adding a new setting or changing global session retention.
