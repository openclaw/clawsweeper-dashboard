## Summary

Keep open: current main has a delivery-path cleanup for `deleteAfterRun`, but isolated cron runs with no delivery or terminal error can still leave the run transcript/session entry for the retention reaper instead of deleting it when the run finishes.

## What This Changes

The issue requests that isolated cron jobs with `deleteAfterRun: true` remove their per-run session transcript files after completion.

## Best Possible Solution

Move `deleteAfterRun` transcript cleanup to a run-final cleanup path for isolated cron sessions so every terminal run removes its own transcript while leaving one-shot job deletion and retention reaping semantics intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run an isolated `agentTurn` cron job with `deleteAfterRun: true` and no delivery, or a terminal error, and the current delivery-only cleanup is not reached.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing isolated cron cleanup contract beyond direct delivery paths rather than changing global session retention or adding a new option.

## Repair Work Prompt

Fix isolated cron `deleteAfterRun` session cleanup. Current main only calls `sessions.delete` from direct/silent delivery handling, so `delivery.mode: "none"` and terminal error paths can leave `agents/<agent>/sessions/<run-id>.jsonl` until the 24-hour reaper. Keep cleanup limited to isolated cron session keys, preserve one-shot job deletion behavior in `src/cron/service/timer.ts`, do not delete main/current/custom reusable sessions, and keep cleanup best-effort so the run result still wins. Add focused regression coverage for no-delivery and error outcomes, and add a single user-facing `CHANGELOG.md` fix entry if the repair changes shipped behavior.
