## Summary

Keep open: the underlying active-memory bug is valid, but this PR should not merge yet because the new helper reads the wrong registry field and introduces a plugin-boundary violation.

## What This Changes

The PR changes active-memory to compute the recall sub-agent `toolsAllow` list at call time and updates its test to expect `memory_recall` to be absent when no registry provides it.

## Best Possible Solution

Preserve the intended fallback, but implement tool availability through registered `names` and a public runtime/SDK seam or core-side allowlist normalization, with coverage for both memory-core-only and memory-lancedb-active cases.

## Reproduction Assessment

Yes, at source level: current main hardcodes `memory_recall` into the active-memory embedded-runner allowlist, while the manifests show that only memory-lancedb provides that tool and memory-core provides the documented fallback pair. The linked issue also includes concrete runtime logs and steps.

## Solution Assessment

No. The PR direction is narrow, but the implementation reads `t.name` instead of `t.names` and reaches from plugin production code into core internals; a safer fix should use the existing registry shape through a supported runtime seam or resolve optional tool filtering inside core.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.9
