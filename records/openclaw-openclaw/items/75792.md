## Summary

Keep open. Current main still has an actionable iMessage startup timeout gap: the readiness probe uses `chats.list`, failed readiness enters the generic channel restart path, and the runtime timeout field is present in TypeScript/runtime code but is not accepted by the strict config schema or documented for users.

## What This Changes

The report asks OpenClaw to stop iMessage from restarting in a 5s loop when `chats.list` times out on a large Messages database, starting with a configurable startup/RPC timeout.

## Best Possible Solution

Expose a supported, documented, bounded per-account iMessage startup/RPC timeout using the existing `probeTimeoutMs` runtime contract, with schema, generated metadata, docs, and regression tests kept in sync. Treat off-thread `imsg` supervision as a follow-up only if the configurable timeout does not resolve large-database startup behavior.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: make the iMessage startup probe return `ok: false` with `imsg rpc timeout (chats.list)` during `monitorIMessageProvider`, and the channel task will fail into the generic auto-restart path; the issue body supplies matching live logs from a large `chat.db`.

## Solution Assessment

Yes for a first repair. Exposing and documenting the existing timeout contract is the narrowest maintainable fix; renaming it to a new `chatsListTimeoutMs` field or moving supervision off-thread should wait unless maintainers want a broader product/API change.
