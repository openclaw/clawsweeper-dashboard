## Summary

Keep open. Current main still has an actionable iMessage startup timeout gap: the startup readiness probe calls `chats.list`, failed readiness flows into the generic channel auto-restart loop, and the only existing iMessage timeout field is used by runtime code but is not accepted by the strict config schema or documented for users.

## What This Changes

Request to make iMessage startup tolerant of large `chat.db` files whose `imsg rpc` `chats.list` readiness probe exceeds the current timeout and triggers repeated provider auto-restarts.

## Best Possible Solution

Add a supported, documented, bounded per-account iMessage startup/RPC timeout setting, preferably by exposing the existing `probeTimeoutMs` contract through schema, generated config metadata if required, docs, and tests. Keep broader async/off-thread iMessage supervision as a follow-up design item if large databases still stall the gateway after users can raise the startup budget.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: make `probeIMessage` return an `imsg rpc timeout (chats.list)` during `monitorIMessageProvider` startup, and the channel manager will record the error and schedule auto-restart; the issue body provides the live large-`chat.db` logs.

## Solution Assessment

Yes for a first repair. Exposing and documenting the existing iMessage timeout path is the narrowest maintainable fix; moving `imsg` supervision off-thread is broader and should follow only with live proof that timeout configurability is insufficient.
