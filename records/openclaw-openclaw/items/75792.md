## Summary

Keep open. Current main still has an actionable iMessage startup timeout gap: the monitor probes readiness through `chats.list`, failed startup tasks feed the generic channel auto-restart loop, and the only apparent timeout knob is read by runtime code but is not accepted by the strict iMessage config schema or documented for users.

## What This Changes

The issue reports that iMessage startup loops when `imsg rpc` `chats.list` times out on a large `chat.db`, and asks for a configurable or asynchronous startup path that avoids disrupting the gateway.

## Best Possible Solution

Add a supported, documented iMessage startup/RPC timeout setting that is accepted by the schema, works per account, is bounded to sane values, and is used consistently by the initial `chats.list` readiness probe and related startup checks. A later follow-up can decouple iMessage readiness from full provider failure if large databases still make startup fragile.

## Reproduction Assessment

Yes. The source-level path is clear: make `probeIMessage` return a timeout for `chats.list` during `monitorIMessageProvider` startup, and the channel manager will record the error and schedule an auto-restart; a live large `chat.db` would confirm the timing and event-loop metrics.

## Solution Assessment

Yes for a first repair. Exposing and documenting the existing timeout concept through the iMessage schema/docs/tests is the narrowest maintainable short-term fix; moving subprocess supervision off-thread is broader and should be a separate design follow-up if metrics still show event-loop stalls.
