## Summary

Keep open: this active maintainer-labeled PR targets a real Control UI session-picker bug, but the current patch still leaves large histories hidden because `limit: 0` is omitted by the UI helper and `sessions.list` falls back to its 100-row default.

## What This Changes

The PR removes active-minutes and current-agent filters from Control UI chat session-list refreshes and updates the related UI tests.

## Best Possible Solution

Define the chat picker history contract explicitly, then implement it with coverage for over-100-row stores and a bounded/paginated path that still keeps selected or older channel conversations reachable.

## Reproduction Assessment

Yes via source inspection: current main sends `sessions.list` with active-minutes and agent filters from chat refresh paths, while the branch still falls back to the gateway's 100-row cap when it sets `limit` to 0. I did not run UI proof because this review is read-only.

## Solution Assessment

No: removing the time and agent filters is a plausible direction, but the PR leaves the default row cap implicit and untested for large histories. A safer fix needs explicit picker-limit or pagination semantics and upgrade/performance proof.
