## Summary

Keep open for maintainer review: the PR is protected by the maintainer label and the source pass found the requested full-history behavior is incomplete because omitted session limits still fall back to 100 rows.

## What This Changes

This PR changes Control UI chat session refreshes to request unscoped, untimed session lists for chat pickers and updates the related UI tests.

## Best Possible Solution

Keep the unscoped, untimed picker direction, but make the picker history contract explicit with a maintainer-approved unbounded mode or a documented and tested high picker cap.

## Reproduction Assessment

Yes via source inspection: current main sends sessions.list with activeMinutes, limit, and agentId from the chat refresh paths, and the gateway filters by agentId and activeMinutes. I did not run the UI because this review is read-only.

## Solution Assessment

No: the PR removes the time and agent filters, but setting the limit constant to 0 only omits the parameter and the gateway then applies its 100-row default. The safer fix needs explicit picker-limit semantics and regression coverage for stores beyond the default cap.
