## Summary

This should stay open: current main has a partial bounded-watchdog mitigation from https://github.com/openclaw/openclaw/pull/84949, but it does not fully implement the reported cleanup behavior, and https://github.com/openclaw/openclaw/pull/84353 is still the open issue-closing fix path for abandoning stuck in-flight write locks.

## What This Changes

The issue reports that post-run auto-compaction can leave a live session JSONL write lock held after timeout, causing later Discord turns in the same session to fail before replying.

## Best Possible Solution

Land or otherwise resolve the open in-flight lock cleanup PR so attempt teardown abandons stuck compaction-held locks with transcript fencing, then close this issue once the fix is merged and verified.

## Reproduction Assessment

Yes, source-level with medium confidence: the report shows repeated lock timeouts on the same live lock, and current main still lacks an in-flight lock abandonment path when a post-prompt compact operation never settles. I did not establish a fresh live Discord/Opus reproduction against current main.

## Solution Assessment

No, current main is not yet the full solution: the merged watchdog-budget change is useful but only bounds the stale hold window. The better fix is the open cleanup path that releases or abandons stuck in-flight write locks during attempt teardown while preserving transcript fencing.
