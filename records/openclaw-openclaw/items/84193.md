## Summary

Keep open: current main has a partial mitigation that bounds the embedded compaction lock watchdog, but it does not include the in-flight lock abandonment needed for the reported post-compaction lock leak. ## What This Changes

The issue reports that post-run auto-compaction can leave a session JSONL write lock held after timeout, causing later Discord turns in the same session to fail with `SessionWriteLockTimeoutError`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/84353 for the compaction-held lock abandonment path, while keeping same-session lane races and same-PID restart cleanup on their narrower canonical issues.

## Reproduction Assessment

Yes at source level: current main still lets a reacquired session write lock live until the wrapped compaction or hook operation settles, while later contenders time out at 60000ms. I did not run a live Discord/Opus reproduction in this read-only review.

## Solution Assessment

Yes, the active fix direction is the right boundary: the embedded attempt session-lock controller owns the reacquired locks and cleanup handoff. The safer merge path is to review the open PR’s abandonment plus transcript-fence behavior rather than close this issue from the partial max-hold mitigation.
