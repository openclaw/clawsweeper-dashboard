## Summary

Keep this PR open for maintainer review: the current patch appears to address the MCP OAuth and schema failures without a blocking code finding, but it changes auth-provider fallback behavior and still needs live or maintainer-accepted proof for the automatic redirect fallback path.

## What This Changes

The PR updates MCP OAuth login/fetch handling and JSON Schema normalization to preserve the legacy loopback default, retry rejected DCR registrations with localhost, surface OAuth error bodies, persist the chosen redirect URI, and repair unicode-invalid schema patterns before TypeBox compilation.

## Best Possible Solution

Land a scoped core MCP fix that preserves the legacy default, falls back to localhost only after DCR redirect rejection, persists the selected redirect for code exchange, and has redacted live OAuth proof or explicit maintainer acceptance of that proof gap.

## Reproduction Assessment

Yes, source and dependency inspection give a high-confidence reproduction path: current main uses 127.0.0.1 for DCR, returns foreign body-less responses unchanged to the MCP SDK, and passes draft schemas into TypeBox paths that compile patterns with the unicode flag. The linked reporter also provided live Calendly and PayPal symptoms.

## Solution Assessment

Yes, this is a strong candidate fix shape: it preserves the shipped default, makes localhost an error-triggered fallback instead of a default flip, normalizes the Response boundary the MCP SDK expects, and repairs only patterns that currently fail unicode compilation. The remaining question is proof and maintainer acceptance for auth-provider compatibility, not a clear better code path.
