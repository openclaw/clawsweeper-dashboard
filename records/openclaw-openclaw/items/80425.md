## Summary

Keep open: the reported auth-epoch bug is real, but this PR fixes it by bypassing the existing authEpochMode/skipLocalCredential ownership seam and can preserve CLI sessions across real local account changes. A narrower related PR tracks the safer epoch-policy approach, while the status-label change appears to be separate work for https://github.com/openclaw/openclaw/issues/80184.

## What This Changes

The PR changes CLI auth epoch hashing to prefer auth-profile credentials over local CLI credentials, adds regression tests, and moves CLI OAuth status labels ahead of env API-key labels.

## Best Possible Solution

Fix https://github.com/openclaw/openclaw/issues/80178 through the prepare/backend auth-epoch policy seam, or review the narrower related PR, while handling the status-label issue as a separate focused change.

## Reproduction Assessment

Yes. Source inspection shows current main hashes both the local credential fingerprint and profile epoch part, and the session reuse path invalidates on a versioned auth-epoch mismatch; I did not run a live credential-rotation test in this read-only review.

## Solution Assessment

No. The resolver-level fix suppresses local credentials whenever a profile exists, but the maintainable path should respect authEpochMode/skipLocalCredential or explicitly change that contract with maintainer approval.
