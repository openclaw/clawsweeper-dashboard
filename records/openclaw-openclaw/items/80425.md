## Summary

Keep open and do not merge as-is: the bug is valid and the PR contains useful tests, but the central resolver change collapses the documented `combined` versus `profile-only` CLI auth-epoch policy, and the external real-behavior proof is still only mock/unit-test evidence.

## What This Changes

The PR updates CLI auth epoch hashing to prefer auth-profile credentials over local CLI fingerprints when present, moves CLI OAuth status-label checks ahead of aliased environment API-key lookup, adds focused tests, and regenerates a Swift gateway model field.

## Best Possible Solution

Keep the backend auth-epoch policy contract intact, fix the reported Claude/auth-profile storage-flip case through the narrow policy seam or an explicitly reviewed backend policy change, and prove it with focused regression tests plus redacted runtime evidence.

## Reproduction Assessment

Yes. Current main is source-reproducible by mocking the Claude local credential reader from `null` to an OAuth credential while the auth-profile credential stays unchanged; the resolver hashes different part lists and the session path treats an epoch mismatch as `auth-epoch` invalidation.

## Solution Assessment

No. The status-label ordering change looks useful, but the auth-epoch fix is too broad because it makes any present profile credential override the documented `combined` local-plus-profile contract instead of preserving the existing policy boundary.
