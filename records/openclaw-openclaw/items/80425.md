## Summary

Keep open: the linked auth-epoch bug is real, but this PR drops local CLI fingerprints whenever an auth profile exists, bypassing the documented combined/profile-only policy and risking session reuse across real local account changes. It also still needs real after-fix behavior proof beyond mocked tests.

## What This Changes

The PR changes CLI auth-epoch hashing to prefer auth-profile credentials over local CLI fingerprints, adds regression tests, moves CLI OAuth status labels ahead of env API keys, and updates a generated Swift protocol field.

## Best Possible Solution

Fix the storage-presence flip through the backend/prepare authEpochMode ownership seam, or another maintainer-approved ownership rule, with regression coverage that preserves invalidation for real account/profile changes.

## Reproduction Assessment

Yes. The current-main path is source-reproducible by mocking the Claude local credential reader from null to an OAuth credential while the auth-profile credential stays unchanged; the resolver hashes different part lists and cli-session maps the mismatch to auth-epoch invalidation.

## Solution Assessment

No. The PR fixes the reported flip by changing the resolver globally, but the narrower maintainable solution should respect the documented combined/profile-only auth ownership seam or explicitly change that contract with maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor auth ownership before dropping local credentials:** `src/agents/cli-auth-epoch.ts:202`
  - body: This skips the local CLI fingerprint whenever an auth profile exists, even for default/combined backends where the contract says the epoch includes both host credential and profile state. In that mode the local CLI credential is still execution auth, so changing the local Claude account while the profile is unchanged can keep reusing the old session instead of invalidating it; route this through the existing `authEpochMode`/`skipLocalCredential` policy seam or update that contract explicitly.
