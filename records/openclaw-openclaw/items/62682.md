## Summary

Keep this PR open: the fix direction matches the reported fallback waste, but the current branch still has blocking abort-classification bugs that can suppress legitimate provider fallback, and the real-behavior proof is still partial for the cron/run-budget paths.

## What This Changes

The PR threads abort signals into model fallback paths, tags HTTP disconnects, adds terminal abort detection and run-budget timeout metadata, and updates tests, changelog, and trajectory metadata.

## Best Possible Solution

Keep the PR open, constrain terminal detection to explicit abort signals or AbortError cause chains, match structured or prefix-based cron timeout reasons, then require redacted real proof for the run-budget/cron path before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a direct provider TimeoutError would now be rethrown before failover normalization, and a phase-suffixed cron timeout signal does not match the PR's terminal string set.

## Solution Assessment

No, not as-is. The proposed direction is narrow, but the implementation should only classify terminal run aborts from explicit abort signals or abort causes and should recognize current cron timeout variants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit terminal timeout checks to abort causes:** `src/agents/model-fallback.ts:215-233`
  - body: `isTerminalAbortFromError` scans the thrown error itself, and `runFallbackCandidate` calls it before `coerceToFailoverError`. A provider or SDK error with `name === "TimeoutError"` is a normal retryable timeout on current main, but this branch would rethrow it and skip configured fallback models. Restrict the helper to AbortError wrappers/cause chains or the explicit run-budget marker.
  - confidence: 0.91
- **[P2] Match phase-suffixed cron timeout reasons:** `src/agents/model-fallback.ts:123-164`
  - body: The terminal string set exact-matches only `cron: job execution timed out`, while the cron timer can abort with `cron: job execution timed out (last phase: ...)`. Those real cron budget expirations will still fall through into fallback retries; match the stable prefix or use a structured abort reason.
  - confidence: 0.89
