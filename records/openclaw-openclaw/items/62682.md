## Summary

Keep this PR open, but do not merge it as-is: the latest head still risks turning ordinary provider `TimeoutError`s into terminal aborts and still misses current cron timeout strings with phase metadata; it also needs concrete after-fix runtime proof before merge.

## What This Changes

The PR threads abort signals into model-fallback callers, adds terminal abort detection for run-budget, client-disconnect, and cron timeouts, marks embedded run-budget timeouts in failover metadata, updates trajectory metadata, tests, and the changelog.

## Best Possible Solution

Repair this PR so terminal abort detection is limited to explicit abort signals, abort causes, or structured run-budget markers, preserves provider timeout fallback, matches cron timeout variants, and includes redacted real runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main treats TimeoutError as retryable timeout-shaped input, while the PR's latest terminal-abort helper would rethrow top-level TimeoutError and still exact-matches only the base cron timeout string.

## Solution Assessment

No, not as-is. The direction is narrow, but the latest implementation should avoid treating top-level provider timeouts as terminal and should recognize phase-suffixed cron timeout reasons.

## Review Findings

Only treat timeout as terminal when it comes from an abort wrapper/cause or an explicit run-budget marker so ordinary provider timeouts still fall back.
  - confidence: 0.9
