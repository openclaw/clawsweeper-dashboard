## Summary

Keep open. The PR addresses a real fallback bug that is still present on current main, but the latest head still misclassifies direct provider TimeoutError failures as terminal, misses current phase-suffixed cron timeout reasons, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds terminal-abort detection to model fallback, threads abort signals through agent/cron/auto-reply/gateway callers, tags HTTP disconnect aborts, records run-budget timeout metadata, and adds regression tests plus a changelog entry.

## Best Possible Solution

Repair this PR or a replacement so terminal aborts are represented by precise abort causes or structured run-budget markers, cron timeout variants are recognized, provider timeouts still fall back, and real runtime proof is attached before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a fallback chain whose first candidate throws a direct TimeoutError, and a cron abort whose reason is the current-main `cron: job execution timed out (last phase: ...)` string.

## Solution Assessment

No, not as-is. The direction is narrow and maintainable, but the implementation must avoid treating top-level provider TimeoutError as terminal and must match current cron timeout reason variants before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Limit terminal matching to abort causes:** `src/agents/model-fallback.ts:207-225`
  - body: `isTerminalAbortFromError` includes the thrown error itself in the candidate list, so a provider or SDK that throws a direct `TimeoutError` is rethrown before `coerceToFailoverError` can classify it as a retryable timeout. Only treat timeout as terminal when it comes from an abort wrapper/cause or an explicit run-budget marker so legitimate provider timeouts still fall back.
  - confidence: 0.88
- **[P2] Match phase-suffixed cron timeout reasons:** `src/agents/model-fallback.ts:156`
  - body: The string-reason path exact-matches only `cron: job execution timed out`, but current main can abort cron runs with `cron: job execution timed out (last phase: ...)`. Match the stable prefix or use a structured abort reason, otherwise those terminal cron timeouts still retry fallback candidates.
  - confidence: 0.86
