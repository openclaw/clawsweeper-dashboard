## Summary

Keep open. The latest branch fixes the earlier normalization and cron string-shape review comments, but it still does not solve the central embedded-run timeout case because it checks the caller abort signal while the embedded budget timer aborts a private run controller.

## What This Changes

The PR adds terminal-abort detection to model fallback, plumbs optional abort signals through agent/auto-reply/cron callers, tags HTTP client disconnect aborts, and adds model-fallback tests.

## Best Possible Solution

Expose or preserve an explicit embedded run-budget-timeout marker at the failover decision boundary, update the failover policy and embedded fallback tests around that marker, keep the HTTP disconnect tagging if maintainers want it, and add the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is an embedded run with a short `timeoutMs` and multiple fallback candidates: `scheduleAbortTimer` aborts the private `runAbortController`, while the PR checks `opts.abortSignal`, and current policy still maps the resulting LLM-phase timeout to fallback.

## Solution Assessment

No. The caller-signal approach is useful for HTTP disconnects and cron wrapper aborts, but it is not sufficient for the embedded runner's private run-budget timeout; the safer fix is an explicit embedded-run timeout contract or failover-policy marker.
