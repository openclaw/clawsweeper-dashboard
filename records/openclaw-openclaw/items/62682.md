## Summary

Keep open: the PR improves terminal-abort handling for caller-provided signals, but it still does not cover the embedded runner's own run-budget timeout described by #60388 because that timeout aborts a private internal controller.

## What This Changes

The PR adds terminal-abort detection to model fallback, threads optional abort signals through agent/cron/auto-reply callers, tags HTTP client disconnect aborts, and adds model-fallback unit tests.

## Best Possible Solution

Expose or preserve an explicit embedded-run budget-timeout marker at the failover decision boundary so run-budget timeouts stop fallback without conflating provider/request timeouts; keep the HTTP disconnect tagging if maintainers want that behavior and add the changelog entry.

## Reproduction Assessment

Yes. A focused source-level reproduction is an embedded agent run with a short `timeoutMs` and multiple fallbacks: `scheduleAbortTimer` aborts the internal run controller, while the PR only checks the caller signal passed as `opts.abortSignal`.

## Solution Assessment

No. Passing the caller abort signal is appropriate for HTTP disconnects and cron wrapper aborts, but it is not sufficient for the embedded runner's private run-budget timeout; the safer fix is an explicit embedded-run timeout contract or failover-policy guard.
