## Summary

Keep open. The PR is a useful implementation candidate for the linked timeout-fallback bug, but the latest head still has blocking timeout-classification issues and lacks contributor real-behavior proof for the central run-budget/cron fallback-skip path.

## What This Changes

The PR changes model fallback, HTTP disconnect handling, and embedded-runner timeout metadata so terminal run aborts skip further fallback candidates.

## Best Possible Solution

Revise the PR to use a precise terminal-abort contract that preserves provider-timeout fallback, handles current cron timeout reason variants or a structured marker, and adds redacted current-head run-budget/cron proof before maintainer review.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: scheduleAbortTimer aborts the private embedded run as a timeout, failover policy treats non-compaction/non-tool timeouts as retryable, and model fallback continues timeout-shaped failures through candidates.

## Solution Assessment

No, not as-is. The direction is narrow, but the current patch skips fallback for direct provider TimeoutError values and misses current phase-suffixed cron timeout reasons after rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep direct TimeoutError values fallback-eligible:** `src/agents/model-fallback.ts:224-225`
  - body: `isTerminalAbortFromError` includes the top-level thrown error in its terminal scan and runs before `coerceToFailoverError`. A provider or SDK that throws a direct `TimeoutError` would now be rethrown immediately instead of being normalized to the existing `reason: "timeout"` fallback path, so restrict this to abort-wrapped terminal reasons or an explicit run-budget marker.
  - confidence: 0.86
- **[P2] Match current cron timeout reason variants:** `src/agents/model-fallback.ts:155-157`
  - body: The terminal string check only accepts the exact base cron timeout text. Current main can abort with `cron: job execution timed out (last phase: ...)`, so after rebase those cron timeouts can miss this guard and still continue through fallback; use a stable prefix/marker or structured abort reason.
  - confidence: 0.82
