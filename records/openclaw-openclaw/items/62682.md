## Summary

Keep this PR open: it addresses a real fallback bug that is still visible in current main, and the author has already responded to earlier bot review notes. It should not merge as-is because the latest patch overmatches direct provider timeout errors and misses current-main cron timeout reasons that include phase details.

## What This Changes

The PR threads abort signals into model fallback, tags HTTP disconnects, adds terminal-abort detection plus run-budget timeout metadata, and adds tests/changelog coverage so terminal run aborts skip further fallback attempts.

## Best Possible Solution

Update the PR to use a precise terminal-abort contract that preserves direct provider timeout fallback, handles current cron timeout reason variants or a structured marker, and includes real run-budget/cron proof before maintainer review against the related cron-budget PR.

## Reproduction Assessment

Yes, at source level. Current main has a run-budget timer that aborts the embedded run and fallback code that still treats timeout-shaped failures as fallback-eligible, so the unwanted retry path is traceable without needing a full live run.

## Solution Assessment

No, not as-is. The direction is narrow and maintainable, but the patch should not classify every thrown TimeoutError as terminal and must handle current cron timeout reasons that include last-phase detail.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep direct provider timeouts fallback-eligible:** `src/agents/model-fallback.ts:224-225`
  - body: `isTerminalAbortFromError` returns true for any top-level `TimeoutError` before `coerceToFailoverError` runs. Current failover code intentionally maps direct `TimeoutError` values to `reason: "timeout"`, so a provider/SDK request timeout would now be rethrown and skip the configured fallback candidates. Please restrict this helper to abort-wrapped terminal reasons or an explicit run-budget marker.
  - confidence: 0.86
- **[P2] Match current cron timeout reasons:** `src/agents/model-fallback.ts:115-119`
  - body: The terminal string set only contains the exact `cron: job execution timed out` value, but current main can abort with `cron: job execution timed out (last phase: ...)`. After rebase, those cron timeouts will miss this guard and can still fall through the fallback chain, including through the `abortReason()` error thrown in the cron executor. Use a stable marker or match the current reason shape.
  - confidence: 0.82
