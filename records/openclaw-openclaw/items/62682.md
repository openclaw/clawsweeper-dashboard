Keep open. The PR is a well-scoped fix with strong runtime proof, but the new terminal cron reason allowlist misses the existing pre-execution watchdog timeout string, so one already-aborted cron path can still cascade through fallback candidates.

## What This Changes

The PR threads abort signals through agent fallback callers and adds terminal-abort detection so run-budget timeouts, HTTP client disconnects, and known cron timeout reasons stop model fallback retries.

## Best Possible Solution

Land the terminal-abort approach after adding the missing cron pre-execution timeout prefix and a regression test that proves it short-circuits while ordinary provider timeouts still cascade.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lacks terminal abort context in runWithModelFallback, while cron and embedded runners can abort a whole run with timeout reasons that currently look retryable to the fallback loop.

## Solution Assessment

No, not quite as proposed. The overall design is the right narrow boundary, but the PR should include the existing cron pre-execution watchdog timeout reason before merge so the terminal cron coverage is complete.
