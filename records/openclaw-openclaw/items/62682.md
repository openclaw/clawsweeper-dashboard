## Summary

Keep open: the latest branch still classifies a direct provider `TimeoutError` as a terminal run abort before failover normalization, which would skip configured fallback models for ordinary retryable provider timeouts. The real behavior proof is also incomplete for the latest head and the run-budget/cron timeout paths.

## What This Changes

The PR adds terminal-abort detection to model fallback, propagates abort signals through agent, gateway, cron, and auto-reply paths, records run-budget timeout metadata, and updates tests, changelog, and trajectory metadata.

## Best Possible Solution

Keep the PR open, narrow terminal detection to explicit abort signals or `AbortError` cause chains, add a regression test for direct provider `TimeoutError` fallback, and require redacted latest-head runtime proof for the run-budget/cron path before merge.

## Reproduction Assessment

Yes, source-reproducible: with configured fallbacks, a run callback that throws an `Error` named `TimeoutError` would hit `isTerminalAbortFromError(err)` and rethrow before normalization on PR head. Current main would normalize that same error to a timeout failover and continue the fallback chain.

## Solution Assessment

No, not as-is. The direction is maintainable, but terminal run-abort detection must not include the top-level provider error itself because provider timeouts remain a supported retry trigger.
