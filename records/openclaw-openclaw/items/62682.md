## Summary

Keep open: this is a plausible bug fix, but the current PR head still has blocking correctness issues in terminal-abort classification and still lacks contributor-supplied real behavior proof for the changed runtime path.

## What This Changes

The PR adds terminal abort detection to model fallback, propagates abort signals through agent, auto-reply, cron, and gateway callers, tags HTTP disconnect aborts, adds run-budget timeout metadata, updates failover policy and trajectory metadata, and adds tests plus a changelog entry.

## Best Possible Solution

Repair the branch so terminal aborts are identified by a precise abort-cause or structured run-budget marker, cron timeout reasons match the current stable contract, provider timeouts still fall back, and redacted runtime proof is added before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a fallback chain where the first candidate throws a direct TimeoutError, and a cron timeout whose abort reason is the phase-suffixed current-main string.

## Solution Assessment

No, not as-is. The proposed direction is narrow and useful, but the implementation must avoid classifying top-level provider TimeoutError as terminal and must match current cron timeout reason variants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Limit terminal matching to abort causes:** `src/agents/model-fallback.ts:207-225`
  - body: `isTerminalAbortFromError` includes the thrown error itself in the candidate list, so a provider or SDK that throws a direct `TimeoutError` is rethrown before `coerceToFailoverError` can classify it as a retryable timeout. Only treat timeout as terminal when it comes from an abort wrapper/cause or an explicit run-budget marker, so legitimate provider timeouts still fall back.
  - confidence: 0.88
- **[P2] Match phase-suffixed cron timeout reasons:** `src/agents/model-fallback.ts:156`
  - body: Current main can abort cron runs with `cron: job execution timed out (last phase: ...)`, but this exact `Set.has` check only recognizes the base string. Match the stable prefix or use a structured abort reason, otherwise those terminal cron timeouts still retry fallback candidates.
  - confidence: 0.86
