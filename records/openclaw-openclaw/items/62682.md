## Summary

Keep open. The PR targets a valid source-reproducible timeout/fallback bug, but the current patch still has blocking correctness issues around provider timeout fallback and cron timeout reason matching, and it still needs contributor-supplied real behavior proof.

## What This Changes

The PR adds terminal abort detection to model fallback, propagates abort signals through agent, auto-reply, cron, and gateway callers, tags HTTP disconnect aborts, adds run-budget timeout metadata, and updates tests plus changelog text.

## Best Possible Solution

Repair this PR or a replacement branch so terminal run-budget aborts use a precise marker or abort-cause contract, cron timeout strings are matched by stable structure or prefix, provider timeouts still fall back, and redacted current-head runtime proof is posted.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an embedded or cron agent run with fallback candidates where the run-budget timer aborts the run and current fallback/failover logic treats the timeout as fallback-eligible.

## Solution Assessment

No, not as-is. The direction is narrow, but the terminal detection must avoid top-level direct TimeoutError matches and must cover current cron timeout reason variants without losing legitimate provider-timeout fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Restrict terminal timeout detection to abort causes:** `src/agents/model-fallback.ts:224-225`
  - body: `isTerminalAbortFromError` includes the top-level thrown error before failover normalization. A provider or SDK that throws a direct `TimeoutError` is intentionally classified as a retryable timeout by current fallback code, but this guard would rethrow before trying the next candidate; only treat timeout as terminal when it comes from an abort wrapper/cause or an explicit run-budget marker.
  - confidence: 0.88
- **[P2] Match phase-suffixed cron timeout reasons:** `src/agents/model-fallback.ts:156`
  - body: Current main's cron timeout message can include `(last phase: ...)`, but the new terminal-string path uses an exact `Set.has` check against only the base text. Those timeout aborts will miss terminal detection when cron progress metadata is present; match a stable prefix or use a structured abort reason.
  - confidence: 0.84
