## Summary

Keep this PR open: it targets a real fallback bug and now includes sufficient real-behavior proof, but the latest head still has a blocking regression where direct provider `TimeoutError` failures would bypass normal fallback.

## What This Changes

The PR adds terminal-abort detection and abort-signal propagation across model fallback, embedded runner timeout handling, gateway disconnects, cron execution, auto-reply paths, trajectory metadata, tests, and changelog text.

## Best Possible Solution

Keep the terminal-abort design, but limit thrown-error terminal detection to explicit abort wrappers or known terminal reason strings while preserving provider timeout fallback, then rebase and land with the regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a direct provider `Error` named `TimeoutError` reaches `isTerminalAbortFromError` before failover normalization on PR head, while current main classifies that shape as retryable timeout failover.

## Solution Assessment

No, not as-is. The overall terminal-abort direction is maintainable, but the helper must not treat the top-level provider error itself as terminal without an explicit abort wrapper or terminal signal/reason context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Restrict terminal detection to abort wrappers:** `src/agents/model-fallback.ts:230`
  - body: `isTerminalAbortFromError` starts by checking `err` itself, so a provider or SDK error whose top-level `name` is `TimeoutError` is rethrown before `coerceToFailoverError` can convert it into retryable timeout failover. Current main intentionally lets ordinary provider timeouts continue through configured fallback models. Start from an explicit `AbortError` wrapper's cause chain, or otherwise prove the thrown error is a terminal run abort, and add a regression for direct provider `TimeoutError` fallback.
