## Summary

Keep open: the PR improves caller-signal abort handling, but it is still not mergeable for the core #60388 path because the embedded run timer aborts a private controller and current failover policy still treats LLM-phase run timeouts as fallback-capable; it also lacks the required changelog entry.

## What This Changes

The PR adds terminal-abort detection to model fallback, threads optional abort signals through agent, cron, and auto-reply callers, tags HTTP client disconnect aborts, and expands model-fallback tests.

## Best Possible Solution

Solve #60388 at the embedded-run failover decision boundary so the run's own budget timeout cannot be converted into another model attempt; keep the HTTP disconnect tagging if desired, then add focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is an embedded agent run with a short `timeoutMs` and configured fallbacks: `scheduleAbortTimer` aborts the private `runAbortController`, current failover policy still rotates or falls back for LLM-phase `timedOut`, and the PR only checks the caller signal.

## Solution Assessment

No. Checking the caller signal is useful for HTTP disconnects and cron wrapper aborts, but it is too far outside the embedded timer to solve the run-budget timeout path; the safer fix is an explicit embedded-run timeout marker or failover-policy guard.
