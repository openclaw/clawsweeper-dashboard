## Summary

Keep open: the latest head still has blocking abort-classification defects and the external contributor proof gate is not satisfied. The useful direction is clear, but ordinary provider timeouts must still fall back and cron timeout variants must be recognized before this can merge.

## What This Changes

The PR changes model fallback abort classification, propagates abort signals through agent, cron, auto-reply, and gateway callers, tags HTTP disconnect aborts, records run-budget timeout metadata, and updates tests plus the changelog.

## Best Possible Solution

Repair this PR so terminal abort detection is based on explicit abort signals, abort causes, or structured run-budget markers, preserves provider timeout fallback, matches cron timeout variants, and includes real runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main classifies provider `TimeoutError` as retryable timeout, while the PR rethrows it as terminal, and current cron timeout strings can include phase metadata that the PR exact-match misses.

## Solution Assessment

No, not as-is. The approach is directionally narrow, but the implementation must avoid treating top-level provider timeouts as terminal and must recognize current cron timeout reason variants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Limit terminal timeout checks to abort causes:** `src/agents/model-fallback.ts:215-233`
  - body: `isTerminalAbortFromError` includes the thrown error itself, so a provider or SDK direct `TimeoutError` is rethrown before `coerceToFailoverError` can classify it as a retryable timeout. Only treat timeout as terminal when it comes from an abort wrapper/cause or an explicit run-budget marker so ordinary provider timeouts still fall back.
  - confidence: 0.88
- **[P2] Match phase-suffixed cron timeout reasons:** `src/agents/model-fallback.ts:164`
  - body: The string branch exact-matches only `cron: job execution timed out`, but current cron timeout messages can include `(last phase: ...)`. Those terminal cron timeouts will still fall through into fallback retries; match the stable prefix or use a structured abort reason.
  - confidence: 0.9
