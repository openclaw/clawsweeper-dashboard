## Summary

Keep open. The latest branch usefully handles HTTP disconnect tagging and cron string abort reasons, but it still does not carry the embedded runner's private run-budget timeout into the fallback layer, so the central #60388 behavior remains unproven and likely unfixed; the user-facing fix also still needs a changelog entry.

## What This Changes

The branch adds optional abort-signal-aware terminal abort handling to model fallback, propagates signals through agent, auto-reply, cron, and HTTP-disconnect callers, tags disconnects with `ClientDisconnectError`, and adds model-fallback unit tests.

## Best Possible Solution

Revise the PR so embedded run-budget timeouts surface an explicit terminal marker to the outer fallback decision without changing provider/request timeout fallback semantics, keep the useful HTTP disconnect and cron-string handling, add embedded-run regression coverage, and document the fix in Unreleased Fixes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure an embedded PI run with a short `timeoutMs` and multiple model fallbacks, let `scheduleAbortTimer` fire, and observe the assistant failover path escalate `timedOut` to model fallback; this PR's current tests only abort a synthetic caller signal.

## Solution Assessment

No. Passing the caller `AbortSignal` is a reasonable mechanism for HTTP disconnects and cron wrapper timeouts, but it does not reach the embedded runner's private timeout controller, so the central #60388 fix needs a more direct runner-to-fallback contract.
