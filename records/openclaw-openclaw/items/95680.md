Keep open; the reset hint and usage-cost warning parts are useful CLI polish, but the timeout warning is not merge-ready because it says an invalid Gateway RPC timeout is falling back even though current code rejects that value.

## What This Changes

The branch adds recovery hints to `openclaw reset` errors, warning output for invalid gateway option fallback paths, and reset command tests for the new messages.

## Best Possible Solution

Keep the reset hints and the actual `--days` fallback warning, but remove or rework the timeout warning so invalid Gateway RPC timeouts either consistently reject or consistently fall back with focused tests and live CLI proof.

## Reproduction Assessment

Yes, from source inspection rather than executing commands: invalid Gateway RPC `--timeout` is rejected by the shared parser before transport, while the PR warns later that a default is used.

## Solution Assessment

No. The reset hints and `--days` warning are reasonable, but the timeout warning is in the wrong path for the current invalid-timeout contract and should be removed or covered by an intentional behavior change.
