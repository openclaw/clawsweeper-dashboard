## Summary

Keep open. Current main and the latest release still hardcode the bundled `session-memory` hook to write direct children of `<workspace>/memory/` with a `YYYY-MM-DD-slug.md` filename, while the requested configurable output path and filename templates are not implemented or documented. The related startup-context work only made root-level slugged memory files readable; it does not solve this customization request.

## What This Changes

The issue asks for configurable output path and filename templates for the bundled `session-memory` hook so generated memory files can follow workspace conventions instead of a flat `memory/YYYY-MM-DD-slug.md` layout.

## Best Possible Solution

Add a narrow, backward-compatible `session-memory` session artifact template contract: keep the existing default output exactly, support validated workspace-relative output path and filename templates with documented date/time/slug placeholders, update bundled and public hook docs, and preserve existing startup-context loading unless a separate product decision expands it.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main always builds `<workspace>/memory`, always writes `${dateStr}-${slug}.md`, and the tests/docs still expect that fixed root-level filename shape.

## Solution Assessment

Yes, for a narrowed repair. Explicit hook configuration is the maintainable way to support user workspace conventions; parsing freeform `AGENTS.md` instructions, adding a separate daily writer, or auto-cleaning directories should remain separate unless maintainers choose that scope.
