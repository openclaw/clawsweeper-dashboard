## Summary

Keep open. Current main and the latest release still write bundled session-memory artifacts to a fixed workspace `memory/` directory with a fixed date-plus-slug basename, while the only related output-directory PR is closed unmerged and the broader path/template/daily/cleanup request still needs maintainer product scope.

## What This Changes

The issue requests configurable `session-memory` output paths and filename templates so `/new` and `/reset` artifacts can follow workspace conventions instead of always writing flat files under `memory/`.

## Best Possible Solution

Define a backward-compatible session-memory artifact-template contract with root-safe workspace-relative output handling, preserving the current default while explicitly deciding whether daily files and cleanup belong in scope.

## Reproduction Assessment

Yes, at source level. Current main still hardcodes the session-memory output directory to workspace `memory` and derives a fixed date-plus-slug basename, with docs and tests asserting that behavior.

## Solution Assessment

Unclear until maintainers choose the config contract. Configurable templates are plausible, but the requested daily and cleanup behavior should not be bundled without explicit scope approval.
