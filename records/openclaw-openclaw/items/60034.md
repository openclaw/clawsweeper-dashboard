## Summary

Keep open. Current main and the latest release still write bundled session-memory artifacts to a fixed workspace `memory/` directory with a fixed date-plus-slug filename, while the related output-directory PRs are closed unmerged and the broader path/template/daily/cleanup scope still needs maintainer product review.

## What This Changes

The issue requests configurable `session-memory` output path and filename-template settings so generated memory files can follow workspace conventions instead of a fixed flat `memory/` layout.

## Best Possible Solution

Define a backward-compatible, root-safe session-memory artifact-template contract that preserves the current default and explicitly scopes daily output and cleanup separately.

## Reproduction Assessment

Yes, at source level. Current main still hardcodes the session-memory output directory and date-plus-slug basename, with docs and tests asserting the fixed layout; I did not run a live `/new` or `/reset` flow because this review is read-only and the source path is clear.

## Solution Assessment

Unclear until maintainers choose the config contract. Configurable templates are plausible, but the proposed daily output and cleanup behavior should not be bundled without explicit scope and filesystem-safety decisions.
