## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope node policy to the run:** `src/gateway/server-node-events.ts:597`
  - body: This records the authenticated node under only the canonical session key, and the resolver later reads that session-global entry for any tool resolution during the 15-minute TTL. If two node-originated turns for the same session overlap, or a later non-node turn uses that session before expiry, it can inherit the wrong node's allow/deny policy; this security boundary needs to be run-scoped, consumed/cleared with the run, or passed through existing run context.
  - confidence: 0.89
- **[P2] Keep the legacy extension driver out of runtime schema:** `src/config/zod-schema.ts:671`
  - body: Current browser config treats `driver: "extension"` as legacy input that doctor migrates to `existing-session`, and docs/tests preserve that latest-shape contract. Adding it back to the normal Zod union makes retired config valid again and conflicts with the current runtime schema boundary.
  - confidence: 0.92
- **[P2] Document gateway.tools.byNode before exposing it:** `src/config/zod-schema.ts:1081`
  - body: `gateway.tools.byNode` is a new public, fail-closed security config option, but the config help/labels and gateway docs still describe only `gateway.tools.allow` and `gateway.tools.deny`. Please add the matching help, label, and public docs entries so operators can discover absent-vs-empty `allow` and the authenticated-node scope.
  - confidence: 0.86
