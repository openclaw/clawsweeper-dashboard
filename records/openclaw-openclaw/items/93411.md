## Summary

Keep this PR open, but it is not merge-ready: the per-node tool restriction is a useful security primitive, but the patch currently lets session-scoped mutable state decide a security boundary and also changes an unrelated legacy browser-driver schema path.

## What This Changes

The branch adds `gateway.tools.byNode`, records node-originated `agent.request` node ids, applies node allow/deny policy in gateway and embedded tool resolution, adds tests, and changes browser profile schema validation.

## Best Possible Solution

Carry authenticated node identity as per-run scoped context into tool resolution, include it in any cache/inheritance keying, remove the unrelated browser-driver schema drift, and prove the result with focused tests plus a redacted live multi-node gateway run.

## Reproduction Assessment

Yes for the patch defects by source inspection: the PR stores node identity by session key only, and the MCP tool cache key lacks node policy. I did not run a live multi-node gateway scenario in this read-only review.

## Solution Assessment

No. The maintainable shape is to pass authenticated node identity through the run/tool context and cache contract, while keeping legacy browser-driver compatibility in doctor migration rather than raw runtime schema validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Bind node policy to the run, not the session:** `src/gateway/session-node-id-registry.ts:41`
  - body: This records one hosting node per session key for 15 minutes, but node dispatch is fire-and-forget and tool construction happens later. A second node request for the same session can overwrite the entry before the first run resolves tools, and later non-node turns can inherit stale policy; if the later node has a broader allow list, this defeats the byNode security boundary. Carry the authenticated node id on the run/tool context or key and clear it by run/session id instead of using a session-wide map.
  - confidence: 0.88
- **[P1] Include node scope in MCP tool caching:** `src/gateway/tool-resolution.ts:126`
  - body: Adding `resolveNodeScopedToolPolicy()` makes the resolver depend on hidden mutable node state, but the MCP loopback cache is still keyed only by session/channel/config context. A tool list resolved for one node can be served to another node on the same session for the cache TTL, bypassing the new per-node restriction. Include the authenticated node or resolved node policy in the cache key, or bypass the cache when byNode policy applies.
  - confidence: 0.84
- **[P2] Do not re-accept the legacy browser extension driver:** `src/config/zod-schema.ts:671-683`
  - body: This unrelated schema change lets `browser.profiles.*.driver = "extension"` pass raw validation and exempts it from `cdpPort`/`cdpUrl`, while current types/docs expose only `openclaw`, legacy `clawd`, and `existing-session`, and doctor migrates `extension` to `existing-session`. Runtime then treats non-`existing-session` drivers as `openclaw` and can fail later or use the wrong browser path. Drop this from the PR or route it through the existing doctor migration contract.
  - confidence: 0.82
