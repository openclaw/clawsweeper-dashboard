## Summary

Keep open: the PR targets a real security boundary, but the proposed state is not merge-ready because node identity is stored by session instead of by run, a retired browser driver value is reintroduced, and the new config surface lacks public docs/help alignment.

## What This Changes

The PR adds `gateway.tools.byNode` with an in-memory session-to-node registry, applies that policy in gateway and embedded tool resolution, adds tests, and also changes browser profile schema validation to accept `driver: "extension"`.

## Best Possible Solution

Land only after node identity is carried in a run-scoped authenticated context, the retired browser driver value stays doctor-only, and the new gateway config contract is documented and proven in a real multi-node run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection is enough for the review findings: the PR records node identity by session key and current tool construction receives run context separately; no live multi-node repro was run.

## Solution Assessment

Is this the best way to solve the issue? No. The right shape is to carry authenticated node identity as run-scoped gateway context and keep legacy config normalization in doctor rather than broadening the runtime schema.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope the node policy to the run, not the session:** `src/gateway/server-node-events.ts:597`
  - body: This records the authenticated node under only the canonical session key, and the resolver later reads that session-global entry for any tool resolution during the 15-minute TTL. If two node-originated turns for the same session overlap, or a non-node turn/tool-inventory request uses that session before expiry, it can inherit the wrong node's allow/deny policy; for a security boundary this needs to be run-scoped, cleared/consumed with the run, or passed through the existing run tool context.
  - confidence: 0.89
- **[P2] Keep the legacy extension driver out of runtime schema:** `src/config/zod-schema.ts:671`
  - body: The current browser config contract treats `driver: "extension"` as legacy input that doctor migrates to `existing-session`, and current tests reject it as an unsupported profile driver. Adding it back to the normal Zod union makes retired config valid again and conflicts with the latest-shape runtime schema.
  - confidence: 0.92
- **[P2] Document the new gateway.tools.byNode contract:** `src/config/zod-schema.ts:1081`
  - body: `gateway.tools.byNode` is a new public, fail-closed security config option, but the config help/labels and public gateway docs still describe only `gateway.tools.allow` and `gateway.tools.deny`. Please add the matching help/label/docs entries so operators can discover the key, understand absent-vs-empty `allow`, and know that it is node-auth scoped.
  - confidence: 0.86
