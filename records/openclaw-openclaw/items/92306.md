## Summary

The report contains credible evidence of a scheduled DeepSeek run stalling for about 380 seconds, entering stuck-session recovery, aborting the embedded run, and surfacing “LLM request failed.” Current source shows cron-specific timeout/abort handling around the shared agent runtime, but the initiating stream stall has not been reproduced on current main and no merged fix or canonical replacement was found, so the issue remains necessary for maintainer diagnosis.

## What This Changes

Fix cron-triggered DeepSeek agent runs that stall mid-stream and terminate with “LLM request failed” while comparable interactive chat runs usually complete.

## Best Possible Solution

Preserve the real timeout signal from the cron watchdog through embedded-run failure and fallback classification, then fix any confirmed shared DeepSeek stream termination defect so both cron and interactive runs complete or fail promptly with an accurate error.

## Reproduction Assessment

No, not yet on current main. The screenshot provides high-confidence evidence of the v2026.6.1 cron abort path, and source inspection confirms the relevant path, but a live current-main DeepSeek reproduction is still missing.

## Solution Assessment

Unclear at the final code level. Extending the timeout is not the best fix; the narrow solution is to identify the first cron-versus-chat divergence and preserve timeout semantics through the shared provider and fallback path.
