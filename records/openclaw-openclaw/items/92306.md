## Summary

Current main fixes the cron-specific timeout propagation failure class shown by this v2026.6.1 report: explicit cron timeouts now remain explicit through the embedded runner instead of being replaced by the model idle watchdog’s implicit cap. The fix is present on main but not in latest release v2026.6.5.

## What This Changes

Investigate cron-triggered agent runs that stall and end with “LLM request failed” when using DeepSeek models, despite similar chat requests completing more reliably.

## Best Possible Solution

Keep the current canonical timeout propagation and ship it in the next release so explicit cron timeout intent reaches the embedded-runner watchdog without a new config option or provider-specific workaround.

## Reproduction Assessment

Yes at source level: the screenshot establishes the cron abort sequence, and current-main regression coverage identifies the cron timeout signal loss that caused the embedded runner to apply the wrong idle watchdog cap.

## Solution Assessment

Yes: preserving explicit timeout intent at the cron entry point and forwarding it through the existing runner contract is narrower and more maintainable than provider-specific parsing changes, fallback behavior, or another config option.
