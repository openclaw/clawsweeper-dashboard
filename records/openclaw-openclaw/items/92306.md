## Label Justifications

- `P1`: Released builds can abort recurring scheduled-agent workflows after several minutes even though equivalent interactive model requests may succeed.
- `impact:auth-provider`: The observable failure occurs during the configured DeepSeek provider request and is surfaced as an LLM request failure.
- `impact:crash-loop`: Scheduled runs stall for several minutes and are forcibly aborted, materially affecting cron-run availability.

## Summary

Current main already addresses the reported cron-only abort by preserving an explicitly configured job timeout through the shared embedded-agent runner, preventing it from being replaced by the runner’s implicit LLM idle limit. The fix is not included in v2026.6.5, so users of v2026.6.1 still need a future release or current main.

## What This Changes

Investigate cron-triggered DeepSeek agent runs that stall and end with “LLM request failed” while comparable chat runs usually complete.

## Best Possible Solution

Keep the current canonical timeout propagation implementation, retain its focused regression coverage, and ship commit 60e818f5 in the next OpenClaw release.

## Reproduction Assessment

Yes at source level. The attached diagnostic shows the aborted cron request, and current regression coverage explicitly describes the lost cron timeout signal and resulting implicit LLM idle limit; a live DeepSeek replay was not required to establish the already-fixed source defect.

## Solution Assessment

Yes. Preserving explicit timeout intent through the canonical shared runner fixes the owner boundary without adding DeepSeek-specific policy, a second cron execution path, or another configuration option.
