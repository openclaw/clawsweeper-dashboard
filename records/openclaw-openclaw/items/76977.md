## Summary

Keep open: current main still has a source-traceable path where Control UI PDF attachments are staged through the media store, which unconditionally fsyncs the temp file and wraps an `EPERM` from that point as the reported `MediaOffloadError`. The related #52093 cleanup covered Windows `rename` failures in JSON state writes, not this media-store `fsync` path.

## What This Changes

The issue reports a Windows regression where Control UI PDF attachments fail during chat.send media staging with `MediaOffloadError: EPERM: operation not permitted, fsync`.

## Best Possible Solution

Patch the media-store buffer write helper so transient or unsupported Windows file fsync failures do not block attachment staging, while preserving temp-file cleanup, rename semantics, and failure behavior for other write errors.

## Reproduction Assessment

Yes. The current source path matches the reported stack: Control UI `chat.send` parses the PDF attachment, `saveMediaBuffer` stages it, and any `EPERM` from `handle.sync()` is wrapped into the reported `MediaOffloadError`; live Windows validation was not run in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `src/media/store.ts` around the temp-file sync step, not in the Control UI, PDF tool, or model/provider path.
