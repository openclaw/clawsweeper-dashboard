## Summary

This is a useful partial fix, but it is not merge-ready: it relaxes validation and doctor pruning for explicit `main` bindings while the runtime route path can still remap those bindings to the configured default or first listed agent.

## What This Changes

The PR adds `DEFAULT_AGENT_ID` to config validation and doctor compatibility-pruning allowed sets, plus regression tests for preserving `main` route bindings with a populated `agents.list`.

## Best Possible Solution

Preserve exact `main` consistently across schema validation, doctor migration, runtime route resolution, and configured binding compilation, with focused regression coverage proving the documented ACP fallback bindings still target the implicit default agent.

## Reproduction Assessment

Yes, by source inspection. The documented ACP config with populated `agents.list` and `agentId: "main"` bindings reaches the PR's relaxed schema/doctor predicates, but current runtime `pickFirstExistingAgentId()` can still fall back from `main` to the configured/default listed agent; I did not run tests because this review is read-only.

## Solution Assessment

No. Preserving `main` in schema and doctor is necessary, but the best fix must also preserve the same target in runtime route resolution and configured binding compilation so the accepted config does not silently deliver to a different agent.
