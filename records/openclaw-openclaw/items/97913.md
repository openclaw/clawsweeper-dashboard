## Summary

Keep open: current main still lets the Codex app-server dynamic-tool bridge pass a raw tool return object through to `convertToolContents(result.content)`, so the reported `reduce` failure remains source-reproducible and narrowly repairable.

## What This Changes

The issue requests hardening Codex dynamic tool execution so plain plugin payload objects are normalized into `AgentToolResult` envelopes before middleware and response conversion.

## Best Possible Solution

Add a small Codex dynamic-tool result normalizer at the bridge boundary, reuse existing `AgentToolResult` envelope conventions, and cover both raw payload and already-structured results in focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a dynamic tool returning a plain object without `content[]` can pass through to `convertToolContents(result.content)`, which calls `reduce` on undefined. I did not run a live Mac-control repro in this read-only cleanup pass.

## Solution Assessment

Yes, normalizing at the Codex dynamic-tool bridge before middleware/extensions and Codex response conversion is the narrowest maintainable fix. Changing Codex or adding plugin-specific handling would leave other OpenClaw runtime tools exposed to the same envelope mismatch.
