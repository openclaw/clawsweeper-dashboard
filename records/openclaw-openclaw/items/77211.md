## Summary

Keep this PR open for maintainer review, but the current patch is not merge-ready: current main documents and tests Telegram `streaming.mode: "off"` as final-only, while this PR would allow generic progress messages to be sent as standalone Telegram messages again.

## What This Changes

The PR changes Telegram dispatch so default tool progress is not suppressed when answer preview streaming is disabled, and updates the related regression test.

## Best Possible Solution

Keep Telegram `streaming.mode: "off"` final-only unless maintainers explicitly decide to restore standalone progress; any reversal should update the channel docs, streaming docs, tests, and changelog together.

## Reproduction Assessment

Yes for the PR-introduced behavior change by source inspection: the patch removes the suppression flag for `streamMode: "off"`, and core dispatch only suppresses generic progress when that flag is true. I did not run tests because this was a read-only review and `pnpm` was not available on PATH.

## Solution Assessment

No. The proposed change conflicts with the current documented Telegram final-only contract; the maintainable path is either to keep suppression when no answer draft stream exists or make an explicit maintainer-approved product/docs change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep `off` mode suppressing standalone progress:** `extensions/telegram/src/bot-message-dispatch.ts:1170`
  - body: Current docs and the current Telegram regression test define `streaming.mode: "off"` as final-only: no preview edits and no standalone generic progress. With this change, `suppressDefaultToolProgressMessages` is undefined when preview streaming is off, so `dispatchReplyFromConfig` can send plan, working-status, or tool-result progress messages into Telegram anyway. That regresses users who disabled streaming to get only final replies.
  - confidence: 0.9
- **[P3] Assert the dispatcher call before reading mock calls:** `extensions/telegram/src/bot-message-dispatch.test.ts:910-913`
  - body: The new optional-chain read from `.mock.calls.at(-1)` returns `undefined` if the dispatcher is never called, so the test can pass without exercising Telegram dispatch. Add an explicit `toHaveBeenCalled()` or `toHaveBeenCalledTimes(1)` before checking the option.
  - confidence: 0.82
