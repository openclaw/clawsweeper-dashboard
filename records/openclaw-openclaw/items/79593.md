Keep open for normal PR review and CI. The patch addresses a source-reproducible CLI display bug, the supplied real behavior proof is sufficient, and I did not find a blocking correctness or security issue in the diff.

## What This Changes

Threads the message CLI `--limit` value into shared text formatting for read, pins, and search output, adds pagination hints, and updates docs, changelog, and formatter tests.

## Best Possible Solution

Land a reviewed version of this renderer-limit fix after CI passes, and keep deeper provider pagination gaps as separate follow-up work if maintainers want to close the broader linked issue fully.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction path: pass a read, pins, or Discord search payload with more than 25 message-like rows into `formatMessageCliText`, and the shared renderer truncates through `messages.slice(0, 25)` despite CLI `--limit` options.

## Solution Assessment

Yes. Passing the existing caller limit into the shared formatter with focused formatter tests is the narrowest maintainable fix for the display cap; provider-specific pagination expansion can remain separate.
