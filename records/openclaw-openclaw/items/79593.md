## Summary

Keep open. Current main still has the shared message formatter cap, and this active PR is a focused fix that threads the caller limit into CLI rendering with matching formatter tests; I found no blocking correctness issue in the diff.

## What This Changes

The PR makes `message read`, `message pins`, and Discord `message search` text output honor the caller display limit, adds pagination hints, documents `pins --limit`, adds formatter tests, and updates the changelog.

## Best Possible Solution

Land the shared formatter/CLI limit fix after normal maintainer merge checks, and keep provider-specific pagination work tracked separately.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass a read, list-pins, or Discord search payload with more than 25 message-like rows through current-main `formatMessageCliText`, and `renderMessageList` slices the output to 25 while ignoring the caller's CLI limit.

## Solution Assessment

Yes. The PR takes the narrow maintainable path: parse the existing CLI limit for display, preserve the default 25-row cap when omitted, and surface already-returned pagination metadata without broad provider rewrites.
