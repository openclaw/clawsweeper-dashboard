## Summary

Keep open. The PR addresses a real current-main formatter cap and includes usable live proof, but the new full-page hint is applied too broadly and can print a history-specific hint for pins/search exact-count payloads.

## What This Changes

The PR threads a caller display limit into CLI message text formatting for read, pins, and search, adds pagination hints, and updates docs, tests, and changelog.

## Best Possible Solution

Land the formatter fix with action-aware pagination hints that preserve JSON payloads, keep the default display cap when no limit is supplied, and avoid history-specific guidance for non-history actions.

## Reproduction Assessment

Yes. Source inspection on current main shows `renderMessageList` slices to 25 while the CLI exposes `--limit`; a focused formatter unit payload with more than 25 rows reproduces the displayed-row cap without live provider access.

## Solution Assessment

Mostly yes. Threading the caller limit through the shared formatter is the narrow maintainable fix, but the new full-page heuristic should be scoped or worded by action before merge.
