## Summary

Keep open. The shared formatter fix is pointed at the right bug, and the supplied terminal screenshot gives real after-fix proof for Discord read, but the PR currently overstates the search behavior because the Discord search provider still clamps requests to 25 before the formatter sees them.

## What This Changes

Threads the message CLI `--limit` into text rendering for read, pins, and search, adds pagination hints and formatter tests, and updates docs/changelog.

## Best Possible Solution

Keep the formatter limit threading, but reconcile the search scope before merge so user-visible docs/changelog/tests only claim behavior the real CLI can deliver.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: build a read, pins, or Discord-style search payload with more than 25 rows and `formatMessageCliText` truncates via `messages.slice(0, 25)` while the CLI exposes `--limit`.

## Solution Assessment

No. Threading the display limit through the shared formatter is the right narrow fix for read and pins, but the PR should not say Discord search honors larger caller limits while the provider still clamps search requests to 25.
