## Summary

Keep open: this addresses a real source-reproducible daily reset bug with sufficient terminal proof, but the scheduled sweep still calls the generic reset service after inspecting a specific store row, so it can mutate a fresher alias/store instead of the stale row it evaluated.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session resets, wires it into gateway lifecycle cleanup, extends reset reason typing, adds focused scheduler tests, and updates session reset docs.

## Best Possible Solution

Land a target-bound proactive daily reset scheduler that mutates exactly the scanned store row, includes duplicate-store or legacy-alias regression coverage, and documents the accepted proactive rollover timing.

## Reproduction Assessment

Yes, source-reproducible: current main only evaluates daily freshness when a requested session is loaded on the inbound agent path, while gateway maintenance has no daily reset sweep. I did not run a live multi-hour rollover.

## Solution Assessment

No, not yet: the proactive scheduler is the right direction, but it should bind the mutation to the scanned store row instead of re-resolving by key through the generic reset service. Maintainers should also explicitly accept the proactive rollover timing change.
