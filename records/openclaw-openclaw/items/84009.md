## Summary

The PR fixes a real source-reproducible daily-reset gap, but it is not merge-ready because the scheduled reset path can still mutate a different session store than the row it scanned.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session resets, wires timer lifecycle cleanup, extends reset reasons, adds focused gateway/session tests, and updates session docs.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the exact scanned store target and matched store key with duplicate/custom-store coverage before accepting proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main lacks a maintenance sweep and only evaluates daily freshness during session use; the remaining PR bug is source-reproducible because the scheduler scans one store but the reset service re-resolves the backing store from the key.

## Solution Assessment

No. The proactive scheduler is the right direction for the linked bug, but the implementation should mutate the scanned store target instead of re-resolving by logical key; maintainer acceptance is also needed for the upgrade from lazy to proactive rollover.
