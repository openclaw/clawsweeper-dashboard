## Summary

Keep this PR open: the real-behavior proof is sufficient and the direction matches the linked bug, but the patch still has a blocking session-store targeting defect and needs maintainer acceptance of the proactive rollover upgrade behavior before merge.

## What This Changes

The PR adds a gateway maintenance scheduler that proactively resets stale daily sessions, wires its timer into gateway startup/shutdown, updates reset reason handling, and documents proactive daily rollover.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes, by source inspection. Current main has no daily reset maintenance scheduler and only evaluates daily freshness when a requested session is used, while the PR-specific wrong-store risk follows directly from scanning `target.storePath` and then calling a key-only reset.

## Solution Assessment

No. The scheduler direction fits the linked bug, but the current implementation should mutate the exact scanned store target instead of re-resolving by session key, and maintainers still need to accept proactive rollover as the daily reset contract.
