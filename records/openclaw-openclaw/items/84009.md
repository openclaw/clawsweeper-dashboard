## Summary

Keep open: the reported daily-reset gap is real and the PR includes strong terminal proof, but the patch is not merge-ready because the new scheduler scans a specific store and then calls a key-only reset path that can mutate a different backing store when duplicate logical session keys exist.

## What This Changes

The PR adds a proactive daily session reset scheduler, wires it into gateway maintenance/startup/shutdown, extends reset reason handling, and updates focused gateway tests plus session reset docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Current main has no daily reset maintenance scheduler and only the inbound/session-use path evaluates daily freshness, while the PR-specific wrong-store risk is visible from the scheduler to reset-service call chain.

## Solution Assessment

No as currently written. Proactive maintenance is a plausible fix, but the mutation must be bound to the store row the scheduler scanned before this is merge-safe.
