## Summary

Keep this PR open: the proactive daily reset direction matches the linked bug and the supplied terminal proof is useful, but the current patch still has a P1 session-state defect because scheduled resets can mutate a different backing store than the stale row selected by the sweep.

## What This Changes

The PR adds a gateway maintenance scheduler for stale daily session resets, threads timer/config cleanup through gateway runtime handles, expands reset tests, and updates session reset docs.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the scanned store target and matched key, add duplicate/custom-store regression coverage, and preserve the proactive rollover contract intentionally.

## Reproduction Assessment

Yes. Source inspection shows current main evaluates daily freshness only when a requested session is used, and the PR-introduced target-binding defect is source-reproducible from the scheduler's key-only reset call into a reset service that re-resolves store targets.

## Solution Assessment

No. The proactive maintenance scheduler is the right fix direction, but this implementation is not safe until reset mutation is bound to the scanned store and duplicate-store coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reset the scanned store instead of re-resolving by key:** `src/gateway/session-daily-reset-scheduler.ts:84`
