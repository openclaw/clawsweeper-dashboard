## Summary

Keep open: the PR addresses a real source-reproducible daily reset gap and includes useful terminal proof, but the branch still resets by key after scanning concrete store paths, so duplicate or custom-store setups can rotate the wrong session or leave the stale row unchanged. The proactive rollover behavior also remains a compatibility decision for existing daily-reset deployments.

## What This Changes

The PR adds a gateway maintenance scheduler for stale daily session resets, wires timer cleanup/runtime config refresh, extends reset reasons, adds focused scheduler tests, and updates session docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main only checks daily freshness on the session-use path, and the branch's blocking defect follows from scanning `target.storePath` but mutating through a key-only reset service that re-resolves the target.

## Solution Assessment

No, not as patched. The scheduler is the right general direction for wall-clock daily reset, but the mutation must be bound to the scanned store target before this is a maintainable fix.
