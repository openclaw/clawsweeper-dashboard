## Summary

Keep open: the proof and overall direction are strong, but the scheduler still resets by re-resolving a key instead of mutating the exact row it scanned, which is a blocking session-state bug. Maintainers should also explicitly accept the upgrade behavior change from lazy rollover to proactive rollover.

## What This Changes

The PR adds a gateway maintenance timer that scans session stores for stale daily-reset sessions, skips active and implicit provider-owned CLI sessions, invokes the reset service, and updates session reset docs.

## Best Possible Solution

Land a target-bound proactive daily reset scheduler that mutates exactly the scanned store row, has duplicate-store or legacy-alias regression coverage, and documents the accepted proactive rollover timing.

## Reproduction Assessment

Yes, by source inspection: current main evaluates daily reset freshness on the session-use path and has no gateway maintenance daily-reset sweep. The PR blocker is also source-reproducible because the new scheduler scans a target store but calls a reset service that re-resolves by key.

## Solution Assessment

No: the proactive scheduler direction is reasonable, but the current implementation is not the safest fix because it does not bind the reset mutation to the row it evaluated. A target-bound reset helper with duplicate-store coverage is the maintainable path.
