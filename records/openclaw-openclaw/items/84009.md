## Summary

Keep open: this PR addresses a real daily-reset bug and includes sufficient terminal proof, but the scheduler still resets by logical key only after scanning a concrete store row, so duplicate stores or legacy aliases can mutate the wrong session state. The proactive rollover timing is also an observable upgrade behavior that maintainers should explicitly accept before merge.

## What This Changes

The PR adds proactive gateway maintenance for daily session resets, wires the new timer into gateway lifecycle cleanup, extends reset-service reason handling, adds focused scheduler tests, and documents proactive daily rollover.

## Best Possible Solution

Land the proactive scheduler only after reset mutation is bound to the scanned store path, target agent id, and matched key, with duplicate-store regression coverage and explicit maintainer acceptance of proactive rollover.

## Reproduction Assessment

Yes. Current main evaluates daily freshness in the inbound session-use path and the maintenance timer has no daily reset scheduler, which matches the linked report where no message arrives after the reset boundary.

## Solution Assessment

No. A proactive scheduler is the right direction, but this implementation should bind mutation to the scanned store path, agent id, and matched key before merge; otherwise duplicate stores or aliases can be reset incorrectly.
