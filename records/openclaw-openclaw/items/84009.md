## Summary

Keep this PR open: the proactive daily-reset direction is well scoped and the supplied terminal proof is useful, but the new scheduler can evaluate a stale row from one session store and then reset a different store because it only passes `sessionKey` into the existing reset service.

## What This Changes

The PR adds a gateway maintenance scheduler for stale daily session resets, wires it into lifecycle cleanup and hot config reads, extends reset reason plumbing, and updates focused tests and docs.

## Best Possible Solution

Fix the scheduler to reset the exact store target that failed freshness, then land it if maintainers accept proactive wall-clock daily rollover as the intended contract.

## Reproduction Assessment

Yes, source-level. Current main evaluates daily freshness in the session-use path and the gateway maintenance timers have no daily reset scheduler, so a stale session can persist until traffic arrives; I did not run a live boundary-crossing gateway wait.

## Solution Assessment

No as currently written. The PR uses the right reset policy and freshness helpers, but the scheduled mutation must be bound to the concrete store target that was scanned before the implementation is safe to merge.

## Repair Work Prompt

On https://github.com/openclaw/openclaw/pull/84009, fix the daily session reset scheduler target-binding bug. The scheduler scans `resolveAllAgentSessionStoreTargetsSync(cfg)` and evaluates staleness from `target.storePath`, but it calls `performReset(sessionKey)`, while `performGatewaySessionReset` re-resolves the store from current config and can mutate a different configured/freshest store. Change the implementation so scheduled daily resets mutate the exact store target and matched key that failed freshness, preserving active-run skip, implicit provider-owned CLI exemption, hooks/archive behavior, and current reset service semantics as much as practical. Add a regression with a custom `session.store` and a stale old default discovered store sharing the same session key, proving the fresh configured store is not reset because of the stale old row. Do not add a config option or change docs unless the public behavior wording changes.
