- `merge-risk: 🚨 compatibility`: The PR intentionally changes existing lazy daily rollover into proactive rollover, which can affect upgrade behavior for configured daily-reset users.

## Summary

The PR is useful and has strong after-fix proof, but it is not merge-ready because the scheduled sweep scans a concrete session-store row and then resets only by logical key, which can mutate a different row or leave the scanned stale row unchanged when duplicate keys, aliases, or discovered stores exist.

## What This Changes

The branch adds a gateway maintenance daily-session reset scheduler, wires timer lifecycle and config reload handling, extends reset reason handling, adds focused tests, and updates session docs.

## Best Possible Solution

Keep the proactive scheduler direction, but make the reset mutation target-bound to the scanned store path, agent id, and matched key, then add duplicate-store/alias coverage before maintainers accept the rollover behavior.

## Reproduction Assessment

Yes. Source inspection shows current main only evaluates daily freshness during session initialization, while the PR body supplies terminal proof for the new maintenance-timer path; I did not run a live multi-hour gateway crossing the boundary.

## Solution Assessment

No. The proactive scheduler is the right direction for the reported bug, but the implementation needs a target-bound reset path before it is the safest maintainable fix.
