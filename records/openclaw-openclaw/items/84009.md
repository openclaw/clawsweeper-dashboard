## Summary

Keep this PR open: the proactive daily-reset shape is useful and the supplied terminal proof covers the main bug, but the scheduled mutation is still not bound to the store row it scanned, so duplicate stores or aliases can reset the wrong session or leave stale state behind.

The PR adds a gateway maintenance scheduler for stale daily sessions, wires its timer into gateway lifecycle cleanup, extends reset-service reason handling, adds focused tests, and updates session docs.

## Best Possible Solution

Land the proactive scheduler only after scheduled resets mutate the exact scanned store path, target agent id, and matched key, with duplicate-store and alias regression coverage plus explicit maintainer acceptance of proactive daily rollover.

## Solution Assessment

No. The proactive scheduler is the right general direction, but the current implementation should bind reset mutation to the scanned store row and add duplicate-store/alias coverage before merge.
