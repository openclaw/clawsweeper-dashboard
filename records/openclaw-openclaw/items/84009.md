## Risks / Open Questions

- Scheduled resets can mutate a different canonical/fresher row than the stale row scanned, leaving the stale row behind for repeated sweeps and causing session-state drift.
- Existing users with daily reset configured may see sessions rotate shortly after the reset boundary on upgrade instead of lazily on the next inbound message; that compatibility change needs maintainer acceptance.
- The gateway adds a recurring filesystem scan across configured and discovered session stores every minute, which is new operational cost even though the PR guards overlapping sweeps.
