## Summary

Keep open: the PR targets the right gateway/session bug and includes useful proof, but the latest scheduler predicate can still leave stale daily sessions unreset when background bookkeeping updates `updatedAt` after the boundary.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session rollover, reset-service race guards, focused gateway/session tests, and docs for daily reset behavior.

## Best Possible Solution

Land the proactive daily rollover only after the scheduler proves daily-boundary staleness from the recovered session start time, ignores bookkeeping `updatedAt` for freshness, preserves the existing guards, and includes regression coverage for a post-boundary bookkeeping write.

## Reproduction Assessment

Yes, at source level: a daily session with `sessionStartedAt` before the reset boundary but a bookkeeping `updatedAt` after the boundary returns false from the PR's `isDailyBoundaryStale`. That contradicts current reset policy and docs, which make `sessionStartedAt` the daily freshness source.

## Solution Assessment

No: proactive maintenance is the right shape, but using `updatedAt` as daily activity is not the maintainable fix. The safer repair is to gate scheduled resets on daily-boundary staleness from the lifecycle start timestamp while preserving the PR's active-run, CLI, alias, store-target, and expected-row guards.
