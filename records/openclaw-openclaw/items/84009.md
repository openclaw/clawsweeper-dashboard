## Summary

Keep open. The PR targets a real, source-reproducible daily reset gap and has strong terminal proof, but the scheduled reset path still drops the scanned store target before mutation, so it can rotate or migrate a different duplicate/legacy row than the one it judged stale.

## What This Changes

Adds a gateway maintenance daily-session reset scheduler, wires its timer into gateway lifecycle cleanup, expands reset reason handling, adds focused scheduler tests, and updates session docs.

## Best Possible Solution

Land proactive daily rollover only after scheduled resets are target-bound to the scanned store path, agent id, and matched row, with duplicate-store coverage and explicit maintainer acceptance of the compatibility change.

## Reproduction Assessment

Yes by source inspection. Current main evaluates daily freshness in the inbound session-use path and has no daily reset maintenance timer, so a stale session can remain untouched until another message arrives.

## Solution Assessment

No as proposed. A proactive scheduler is a reasonable fix shape, but the mutation must stay bound to the scanned target instead of reusing key-only reset resolution that intentionally folds duplicate and legacy rows.
