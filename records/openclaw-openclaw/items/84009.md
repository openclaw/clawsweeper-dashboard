## Summary

The PR has strong terminal proof for the intended daily-reset behavior, but it is not merge-ready because the scheduled sweep scans one concrete session-store row and then resets through a key-only resolver that can mutate a different row.

## What This Changes

The PR adds proactive gateway maintenance for daily session resets, clears the new timer on shutdown, extends reset reason handling, adds scheduler coverage, and updates session docs.

## Best Possible Solution

Land the proactive scheduler only after scheduled resets are bound to the exact scanned store path, agent id, and matched row, with duplicate-store or legacy-alias coverage and explicit maintainer acceptance of proactive rollover.

## Reproduction Assessment

Yes by source inspection. Current main evaluates daily freshness in session-use paths and has no maintenance timer that resets stale daily sessions when no inbound message arrives; I did not run a live multi-hour rollover because this review is read-only.

## Solution Assessment

No, not yet. A gateway maintenance scheduler is a reasonable owner boundary, but the reset mutation must be target-bound instead of re-resolving by logical key, and maintainers should explicitly accept the proactive rollover compatibility change.
