## Summary

Keep open: the PR has strong terminal proof for the intended proactive daily reset behavior, but the current patch still has a blocking session-store correctness bug. The scheduler scans a concrete store target, then resets by session key through the generic reset service, which can re-resolve the same key to a different fresher store and rotate the wrong session or leave the scanned stale row intact.

## What This Changes

The PR adds a gateway maintenance scheduler for daily session resets, wires the new timer through gateway lifecycle cleanup and runtime config reloads, extends reset reason handling, adds focused tests, and updates session docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is the accepted daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main evaluates daily freshness on session use, and the PR's remaining defect follows directly from the scheduler's key-only reset call plus the reset service's freshest-store re-resolution.

## Solution Assessment

No, not yet. The scheduler direction is appropriate for the linked bug, but the implementation needs to mutate the exact scanned store target and the proactive rollover compatibility choice should be explicit before merge.
