## Summary

Keep open: the intended proactive daily reset behavior is well motivated and has strong terminal proof, but the patch still has a blocking store-target bug that can reset the wrong persisted session.

## What This Changes

The PR adds a gateway maintenance scheduler that proactively resets stale daily sessions, wires the timer into gateway lifecycle cleanup, extends reset reason handling, and updates session docs/tests.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned store target and add duplicate/custom-store regression coverage, then land only if maintainers accept proactive wall-clock rollover as the intended daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main only checks daily freshness in the session-use path and gateway maintenance has no daily reset scheduler, so the linked no-inbound-after-boundary scenario can leave rollover waiting for traffic.

## Solution Assessment

No, not as currently patched. The proactive scheduler is the right direction for the linked report, but it must reset the exact scanned store target before the implementation is merge-safe.
