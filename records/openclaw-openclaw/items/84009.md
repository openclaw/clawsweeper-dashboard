## Summary

## What This Changes

The PR adds a gateway maintenance scheduler that scans session stores for stale daily sessions, resets them through the existing reset service with race guards, updates shutdown handles/tests/docs, and restores Docker package parser guards.

## Best Possible Solution

Emit a `sessions.changed` update for each successful scheduled reset, keep the existing reset-service and race-guard path, then land only after maintainers accept proactive daily rollover as the configured contract.

## Reproduction Assessment

Yes, at source level. Current main has no daily reset maintenance timer and evaluates reset freshness on session-use paths, matching the linked report that stale daily sessions do not roll over without inbound traffic.

## Solution Assessment

No, not quite. Reusing gateway maintenance plus the existing reset service is the right owner boundary, but the PR should also emit the established session-change notification for successful background resets before it is merge-ready.
