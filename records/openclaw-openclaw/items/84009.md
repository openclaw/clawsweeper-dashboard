## Summary

Keep open: this addresses a real daily-session reset bug and has sufficient terminal proof, but the scheduled reset still re-resolves by session key and can mutate a different duplicate, discovered-store, or legacy-alias row than the stale row it scanned.

## What This Changes

Adds a gateway maintenance scheduler for proactive daily session resets, wires it into gateway lifecycle cleanup, extends reset reason handling, adds focused scheduler tests, and updates session docs.

## Best Possible Solution

Bind scheduled resets to the exact scanned store path, agent id, and matched row, add duplicate-store or legacy-alias regression coverage, then land proactive rollover with the documented upgrade behavior.

## Reproduction Assessment

Yes by source inspection. Current main evaluates daily freshness in the inbound session-use path and does not start any daily-reset maintenance timer, so a stale session can remain untouched until another message arrives.

## Solution Assessment

No. The proactive scheduler is the right direction, but the implementation should mutate the scanned row directly instead of rediscovering a potentially different store target from the session key.
