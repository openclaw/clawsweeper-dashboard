## Summary

Keep this PR open with changes requested: the user problem is source-reproducible and the real-behavior proof is strong, but the scheduler still calls the reset service by logical key only after scanning a specific store path, so aliases, duplicate logical keys, configured stores, or discovered stores can reset the wrong row or leave the scanned stale row unchanged.

## What This Changes

The PR adds a gateway maintenance scheduler that proactively scans session stores for stale daily sessions, skips active and implicit provider-owned CLI sessions, wires timer cleanup/config reload handling, and updates session docs.

## Best Possible Solution

Keep the proactive gateway-maintenance scheduler direction, but make the reset mutation target-bound to the scanned store path, agent id, and matched key before maintainers accept the rollover behavior.

Yes. Source inspection shows current main only evaluates daily freshness in the session-use path and current maintenance timers do not scan session stores; the linked issue also gives a concrete daily-reset config and observed stale context. I did not run a multi-hour live gateway crossing the configured boundary.

## Solution Assessment

No for the current patch. The scheduler belongs in gateway maintenance and reuses the right freshness policy, but the reset operation must be bound to the scanned target instead of rediscovering by key before this is the narrow maintainable fix.
