## Summary

Keep open: this PR addresses a real source-reproducible daily reset gap and includes sufficient terminal proof, but the proposed scheduler needs a current-entry/active-run recheck before mutating session state. Maintainers also need to accept the compatibility change from lazy rollover to proactive background rollover.

## What This Changes

The PR adds a gateway maintenance scheduler that scans configured session stores for stale daily sessions, resets them through the existing reset service, wires timer cleanup, and updates session docs/tests.

## Best Possible Solution

Land proactive daily rollover only with a mutation-time guard that proves the same session is still stale and inactive before resetting, then keep the docs explicit about the upgrade-visible timing change.

## Reproduction Assessment

Yes from source: current main evaluates daily freshness only when a requested session is used, and gateway maintenance has no daily reset sweep. I did not run a live multi-hour gateway wait.

## Solution Assessment

No, not yet. A gateway maintenance scheduler is the right boundary for wall-clock daily resets, but the reset must be conditional on the current session still being the scanned stale session before it mutates state.
