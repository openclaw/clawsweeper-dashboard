## Summary

Keep open. The PR addresses the reported lazy daily reset gap and has credible terminal proof, but the scheduler still decides staleness in one backing store and then mutates by key through a reset service that can choose a different store.

## What This Changes

The PR adds proactive gateway maintenance for daily session resets, lifecycle cleanup for the new timer, focused scheduler tests, and docs for proactive rollover and implicit CLI-session continuity.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is the accepted daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main evaluates daily freshness when a session is used, and the PR's remaining defect follows from the scheduler's key-only reset call plus the reset service's freshest-store re-resolution.

## Solution Assessment

No, not yet. The PR is the right direction for the reported bug, but scheduled reset mutation should be bound to the scanned store target instead of reusing the generic key lookup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reset the scanned store instead of re-resolving by key:** `src/gateway/session-daily-reset-scheduler.ts:84`
  - body: The scheduler decides staleness while iterating a concrete `target.storePath`, but it only passes `sessionKey` into `performReset`. Carry the scanned store target into the mutation or add a store-bound helper, and cover the duplicate-store case.
