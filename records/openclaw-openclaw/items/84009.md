## Summary

Keep open: this is a valid source-reproducible gateway session bug with strong terminal proof, but the current PR is not merge-ready because the scheduled sweep scans a specific store row and then resets by key only, allowing the reset service to re-resolve and mutate a different duplicate or legacy-alias row. Maintainers also need to explicitly accept that daily-reset users will see proactive rollover shortly after the boundary instead of lazy rollover on the next inbound message.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session resets, lifecycle cleanup wiring, focused scheduler tests, and session documentation updates.

## Best Possible Solution

Land a target-bound proactive daily reset scheduler that mutates exactly the scanned store row, covers duplicate-store and legacy-alias cases, and documents the accepted proactive rollover timing.

## Reproduction Assessment

Yes, by source inspection: current main evaluates daily freshness in the inbound session-use path and gateway maintenance has no daily reset timer. I did not run a live overnight boundary crossing in this read-only review.

## Solution Assessment

No, not yet: the proactive scheduler is the right direction, but key-only reset after a target-row scan is not the safest mutation path. The maintainable fix is to bind scheduled resets to the scanned store target and row, then have maintainers accept the proactive timing change.
