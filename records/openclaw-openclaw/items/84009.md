## Summary

Keep open: the bug and proof are credible, but this PR still has a blocking session-store correctness issue where the scheduler scans one backing store and resets by logical key through global resolution, which can mutate the wrong session when stores collide.

## What This Changes

The PR adds a gateway maintenance scheduler for daily session resets, wires its timer lifecycle through gateway startup/shutdown, adds focused tests, and updates session reset docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Current main has no daily reset maintenance scheduler and evaluates daily freshness in the inbound session-use path, so the no-message-after-boundary bug is source-reproducible without waiting for a live 4 AM boundary.

## Solution Assessment

No as written. A gateway maintenance sweep is the right implementation shape, but the reset mutation must be bound to the scanned store target and the proactive-upgrade behavior needs maintainer acceptance before merge.
