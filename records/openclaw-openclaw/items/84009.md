## Summary

The PR should stay open because the scheduler shape addresses the linked bug, but the current patch still resets by session key after scanning a concrete store. That can rotate the wrong session or leave the stale row untouched when the same logical key exists in multiple configured/discovered stores.

## What This Changes

The PR adds a gateway maintenance scheduler for stale daily session resets, wires its timer lifecycle into gateway startup/shutdown, and updates session reset docs/tests.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Current main only evaluates daily freshness from the session-use path, while gateway maintenance has no daily reset scheduler; the PR finding is also source-reproducible from the key-only reset call after target scanning.

## Solution Assessment

No as written. A gateway maintenance sweep is the right shape, but the reset mutation must be bound to the scanned store target before this is the safest maintainable fix.
