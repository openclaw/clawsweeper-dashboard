## Summary

Keep open: the PR has strong proof for the single-store happy path, but it still has a P1 correctness bug because the scheduled sweep scans one backing store and then resets by re-resolving only the logical session key, which can mutate a different store when duplicate keys exist.

## What This Changes

The branch adds a gateway daily reset scheduler, passes runtime config into maintenance timers, clears the new timer on shutdown/restart, updates reset reason plumbing, adds focused tests, and updates session docs.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the exact scanned store target and matched store key with duplicate/custom-store coverage before accepting proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main only evaluates daily freshness during later session resolution and has no maintenance reset scheduler; the PR's remaining store-target bug is also source-reproducible from the scheduler dropping `target.storePath` before calling the reset service.

## Solution Assessment

No, not yet. The scheduler is the right shape for the linked bug, but the implementation should mutate the exact scanned store target before maintainers decide whether proactive rollover is the desired upgrade behavior.
