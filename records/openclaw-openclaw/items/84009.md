## Summary

Keep this PR open, but it is not merge-ready: the single-store proof is strong, yet the scheduler still scans a specific store row and then resets by key through a resolver that can choose a different store or legacy alias before mutating session state.

## What This Changes

The PR adds a gateway maintenance timer that proactively resets stale daily sessions, clears the timer during gateway shutdown, extends the reset reason, adds focused scheduler coverage, and updates session docs.

## Best Possible Solution

Land the proactive scheduler only after scheduled resets mutate the exact scanned store path, target agent id, and matched key, with duplicate-store or legacy-alias coverage and explicit maintainer acceptance of proactive daily rollover.

Yes, by source inspection: current main evaluates daily session freshness only on the inbound requestedSessionKey path, and the PR-introduced row-binding defect follows from the scheduler calling a key-only reset service after scanning a concrete row. ## Solution Assessment

No, not as currently implemented. The proactive scheduler is the right direction, but the reset operation needs to be bound to the scanned store target before the compatibility change is safe to merge.
