## Best Possible Solution

## Reproduction Assessment

## Solution Assessment

No. A proactive scheduler is the right direction, but this patch must bind the reset mutation to the evaluated row or alias group before it is the best maintainable fix.

- **[P1] Bind scheduled resets to the evaluated row:** `src/gateway/session-daily-reset-scheduler.ts:84`
  - body: `resetStaleDailySessions` decides that a specific row from a specific store is stale, but then calls `performReset(sessionKey)`, whose reset service re-runs canonical/legacy resolution and promotes the freshest matching row before mutating. With a stale legacy or case-variant row beside a fresh canonical row, or an active canonical run beside a stale alias, the sweep can reset the wrong session instead of the row it evaluated.
