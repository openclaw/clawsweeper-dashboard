## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Respect session-change subscriptions for scheduled resets:** `src/gateway/server-maintenance.ts:88`
  - body: This new scheduled reset path uses broad `broadcast()`, so every read-scope operator client receives `sessions.changed` even if it never called `sessions.subscribe`. Existing session-change emitters target `getSessionEventSubscriberConnIds()`/`broadcastToConnIds`, and the protocol docs describe `sessions.subscribe` as the gate. Route this operator event through the same subscriber registry or shared emitter.
  - confidence: 0.9
- **[P2] Use global-aware node delivery for scheduled resets:** `src/gateway/server-maintenance.ts:89`
  - body: For a scheduled reset of a non-default global session, this sends the node `sessions.changed` payload to bare `global`. Existing node/chat delivery maps selected-agent globals to `agent:<agentId>:global` and includes bare `global` only for the default agent, so non-default global subscribers can miss the reset update. Reuse the existing global-aware delivery-key mapping and add coverage for that case.
  - confidence: 0.9
- **[P2] Keep agent identity in active global reset guards:** `src/gateway/server-maintenance.ts:361-366`
  - body: The scheduler reduces active runs to a `Set` of sessionKey strings, but selected-agent global runs are stored as `sessionKey: "global"` with their owning `agentId`. In `session.scope: "global"` with per-agent stores, one active work/global run makes the guard skip unrelated stale global sessions for other agents. Carry `{ sessionKey, agentId }` or scoped global keys into the guard and add a two-agent regression test.
  - confidence: 0.86
