## Summary

Keep open: the problem and after-fix proof are solid, but the patch still re-resolves a scanned session by key before mutating it, so duplicate stores or legacy aliases can rotate a different row than the stale row selected by the scheduler.

## What This Changes

The PR adds a gateway maintenance scheduler that scans session stores for stale daily sessions, resets them through gateway reset plumbing, threads the timer through gateway lifecycle cleanup, adds focused scheduler coverage, and updates session reset docs.

## Best Possible Solution

Land a target-bound scheduler that mutates exactly the scanned store row, has duplicate-store or legacy-alias regression coverage, and records maintainer acceptance of proactive rollover timing.

## Reproduction Assessment

Yes, source-reproducible: current main evaluates daily freshness in inbound session initialization, while gateway maintenance has no daily reset sweep. The linked issue and PR body also provide a concrete daily-reset scenario and after-fix terminal proof.

## Solution Assessment

No, not yet: a proactive scheduler is a plausible fix, but this implementation should not re-resolve by session key after scanning a specific store row. The safer solution is to bind the mutation to the scanned target and cover duplicate-store or legacy-alias cases.
