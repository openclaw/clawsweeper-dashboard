## Summary

Keep this PR open: the scheduler approach and proof are useful, but the patch can still reset the wrong backing store because it drops the scanned store target before mutating the session. The proactive rollover upgrade behavior also remains a maintainer-visible compatibility choice.

## What This Changes

The PR adds a gateway daily-session-reset scheduler, wires it into gateway maintenance lifecycle/config reload paths, extends reset reason handling, adds focused scheduler tests, and updates session docs.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the exact scanned store target and matched key with duplicate/custom-store coverage before accepting proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main computes daily freshness only when a requested session is processed and gateway maintenance has no daily reset timer; the PR's remaining target-binding defect is also source-reproducible from the key-only reset call.

## Solution Assessment

No. The scheduler is the right shape, but the mutation must stay bound to the scanned store target before merge; maintainers should also explicitly accept the proactive rollover upgrade behavior.
