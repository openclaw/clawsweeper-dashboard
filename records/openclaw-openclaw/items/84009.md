## Summary

Keep this PR open. The maintenance-scheduler direction and proof are solid, but the current branch can reset a different backing store than the one it scanned when duplicate logical session keys exist; the proactive rollover behavior also remains an upgrade-visible maintainer choice.

## What This Changes

This PR adds a gateway maintenance daily-session reset scheduler, wires its timer lifecycle through gateway startup/shutdown, extends reset reason handling, adds focused tests, and updates session reset docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Current main has no daily reset maintenance scheduler and only evaluates daily freshness in the session-use path, so the linked bug is source-reproducible without a live 4 AM wait.

## Solution Assessment

No as written. A gateway maintenance sweep is the right implementation shape, but it must bind reset mutation to the scanned store target before it is a safe maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reset the scanned store instead of re-resolving by key:** `src/gateway/session-daily-reset-scheduler.ts:84`
  - body: The scheduler scans a specific `target.storePath`, but when it finds a stale row it calls `performReset(sessionKey)` only. The production reset path re-runs `resolveGatewaySessionStoreTarget`, whose lookup selects the freshest duplicate logical key across candidate stores, so a stale row in one store can reset a newer row in another store and leave the stale row untouched. Carry the scanned store path/key into the mutation or add a target-bound reset helper and cover duplicate stores.
  - confidence: 0.93
