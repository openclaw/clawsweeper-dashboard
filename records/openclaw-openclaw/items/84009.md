## Summary

Keep open: the PR is the active fix for unattended daily resets, but the scheduled-reset notification path should be repaired before merge because it currently sends a session-specific `sessions.changed` event to every node with any session subscription.

## What This Changes

Adds a gateway maintenance scheduler that proactively rotates stale daily sessions, hardens reset-service row guards, updates session docs, and adds gateway/session regression coverage.

## Best Possible Solution

Keep the gateway-owned scheduler shape, but scope scheduled reset node fanout to the reset session and preserve the existing active-run/provider-owned guardrails before merge.

## Reproduction Assessment

## Solution Assessment

No as submitted. A gateway scheduler that reuses the reset service is the right boundary, but the node notification fanout must be session-scoped before this is the best fix.
