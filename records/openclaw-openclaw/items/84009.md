## Summary

Keep this PR open: the main idea is well supported and the supplied terminal proof covers the primary scheduled rollover path, but the patch still has a blocking store-target bug where a sweep over one session store can reset a different store for the same key.

## What This Changes

The PR adds a gateway maintenance scheduler for stale daily session resets, wires timer lifecycle/config reload handling, extends reset handling for scheduled daily resets, and updates session docs/tests.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main evaluates daily freshness only during session use, and the P1 review finding is source-reproducible from the scheduler's key-only reset call plus the reset service's freshest-store re-resolution.

## Solution Assessment

No, not yet. The scheduler is the right boundary for proactive rollover, but it must mutate the scanned store target instead of re-resolving by key, and maintainers still need to accept the upgrade behavior change.
