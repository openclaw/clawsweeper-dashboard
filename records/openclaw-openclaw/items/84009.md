The PR adds a gateway maintenance scheduler for daily session resets, wires the timer into gateway lifecycle cleanup, updates reset reason plumbing, adds focused gateway/session tests, and revises session reset docs.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the exact scanned store target and matched store key with duplicate/custom-store coverage before accepting proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main only evaluates daily freshness during later session use and the PR's store-target bug is source-reproducible from the scheduler dropping the scanned target before calling the reset service.

## Solution Assessment

No. The scheduler is a plausible fix direction, but this branch should first bind reset mutation to the scanned store target and then get maintainer acceptance of the proactive rollover upgrade behavior.

- **[P1] Reset the scanned store instead of re-resolving by key:** `src/gateway/session-daily-reset-scheduler.ts:84`
