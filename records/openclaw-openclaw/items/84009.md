## Summary

Keep open: this is a valid gateway/session bug fix with strong real-behavior proof, but the patch still has a blocking store-target defect. The scheduler scans a specific session store and then calls the key-only reset service, which can re-resolve and rotate a different fresher duplicate/custom store instead of the stale row it found.

## What This Changes

The PR adds a gateway daily-session-reset scheduler in maintenance timers, wires timer cleanup, extends reset reason handling, adds scheduler/maintenance coverage, and updates session docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main only checks daily freshness in the session-use path, and the PR branch's blocking defect follows from scanning `target.storePath` but mutating through a key-only reset service that re-resolves the target.

## Solution Assessment

No, not yet. Proactive gateway maintenance is the right shape for wall-clock daily resets, but the mutation must target the scanned store and maintainers still need to accept the upgrade-visible rollover timing.
