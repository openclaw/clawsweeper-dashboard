## Summary

Keep this PR open: it targets the valid stale daily-session bug from the linked issue, but the scheduler currently captures a startup config snapshot and can keep resetting on stale settings after `session.*` hot reloads.

## What This Changes

The PR adds a gateway maintenance scheduler that scans configured session stores every minute and invokes the existing reset service for daily sessions that are stale past the configured reset boundary.

## Best Possible Solution

Keep the gateway-maintenance sweep shape, but have each sweep use the active runtime config and guard resets against the specific stale entry before requiring real gateway proof across the daily boundary.

## Reproduction Assessment

Yes. Source inspection and the linked issue give a high-confidence path: a daily-reset session can pass the wall-clock boundary with no inbound turn, and current main only evaluates freshness when session-use paths run.

## Solution Assessment

No, not yet. A gateway maintenance sweep is the right direction, but this implementation needs active-config reads, a stale-entry guard, and real gateway proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the active config during each sweep:** `src/gateway/session-daily-reset-scheduler.ts:87`
  - body: The scheduler captures `cfg` when maintenance starts and reuses it forever. Current reload planning treats `session.*` changes as no-restart because runtime paths read the active config, so after a user changes `session.reset.atHour`, `mode`, or `store`, this timer can keep scanning/resetting according to the old startup snapshot until the gateway restarts. Pass a config getter into the scheduler or restart/update the timer when session config changes.
  - confidence: 0.91
- **[P2] Recheck the entry before resetting it:** `src/gateway/session-daily-reset-scheduler.ts:66`
  - body: The sweep decides a session is stale from a store snapshot, then calls `performReset(sessionKey)` later. The reset service mutates whatever entry is current by the time its store writer runs, so an inbound turn or lazy reset that refreshes the same key during hook/cleanup awaits can still be reset by the sweep even though it is no longer stale. Carry the expected session id/timestamps into a guarded reset or re-evaluate freshness inside the locked mutation.
  - confidence: 0.82
