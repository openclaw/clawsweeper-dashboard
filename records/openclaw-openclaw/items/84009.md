## What This Changes

The PR adds a gateway maintenance scheduler that proactively resets stale daily sessions, wires its timer into gateway startup/shutdown, and updates session reset docs and focused tests.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Current main has no daily reset maintenance scheduler and only evaluates daily freshness in the inbound session-use path, so the no-message-after-boundary bug is source-reproducible without waiting for a real 4 AM boundary.

## Solution Assessment

No as written. A gateway maintenance sweep is the right implementation direction, but the reset mutation must be bound to the scanned store target before the proactive rollover behavior is safe to merge.
