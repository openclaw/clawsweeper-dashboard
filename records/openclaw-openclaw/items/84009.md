## Summary

Keep open: the latest branch appears to address the earlier row-binding race and has strong terminal proof, but the scheduled reset path still uses combined daily+idle freshness, so `mode: "daily"` configs with `idleMinutes` can be reset proactively by idle expiry instead of only after the daily boundary.

## What This Changes

The PR adds a gateway maintenance scheduler for daily session rollover, conditional reset-service row binding, timer cleanup, docs, and focused gateway/session tests.

## Best Possible Solution

Land the proactive daily rollover after the scheduled path proves daily-boundary staleness specifically, preserves the conditional row-binding guards, and has maintainer acceptance for the documented upgrade timing.

## Reproduction Assessment

Yes, at source level: configure `session.reset` with `mode: "daily"`, a future `atHour`, and a small `idleMinutes`; the PR scheduler calls `evaluateSessionFreshness`, which returns stale on idle expiry even though the daily boundary has not passed.

## Solution Assessment

No, not yet: the row-binding fix is the right direction, but the scheduler should check daily-boundary staleness specifically or explicitly add and document proactive idle expiry as a separate product decision.
