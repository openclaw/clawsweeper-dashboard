## Summary

Keep this PR open: the scheduled daily reset direction is plausible and the supplied terminal proof is strong, but the current patch can reset a different backing store than the stale row it just scanned when duplicate logical session keys exist.

## What This Changes

The PR adds a gateway maintenance scheduler for stale daily session resets, wires timer cleanup and hot-config reads, adds focused gateway/session tests, and updates session docs.

## Best Possible Solution

Keep the scheduler approach, but make the reset mutation target-bound to the scanned store and matched key, add duplicate/custom-store regression coverage, and have maintainers explicitly accept proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Current main only checks daily freshness during inbound session setup, and the PR-introduced target-binding defect is source-reproducible from the scheduler's key-only reset call into a reset service that re-resolves store targets.

## Solution Assessment

No, not yet. The scheduler is the right general shape, but it should mutate the exact scanned store target before maintainers decide whether proactive rollover is the desired upgrade behavior.
