## Summary

Keep open: the scheduler is aimed at the right bug and the PR body includes strong terminal proof, but the branch still resets by re-resolving only the session key after scanning a concrete store row, which can mutate the wrong duplicate or legacy row before merge.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session resets, wires its interval into gateway lifecycle cleanup, broadens reset reason typing, adds focused scheduler tests, and updates session reset docs.

## Best Possible Solution

Land a target-bound proactive daily reset scheduler that mutates exactly the scanned store row, includes duplicate-store or legacy-alias regression coverage, and documents the accepted proactive rollover timing.

## Reproduction Assessment

Yes, source-reproducible: current main evaluates daily freshness only when a requested session is loaded on the inbound agent path, and gateway maintenance has no daily reset sweep. I did not run a live multi-hour 4 AM rollover.

## Solution Assessment

No. The scheduler direction is reasonable, but the current mutation path re-resolves by key instead of mutating the inspected row; a target-bound reset helper is the safer maintainable fix.
