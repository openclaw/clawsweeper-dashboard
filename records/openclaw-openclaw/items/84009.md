## Summary

Keep open: this is a valid gateway/session bug fix with strong terminal proof, but the patch is not merge-ready because scheduled resets are not bound to the scanned store row and can reset a fresher duplicate session while leaving the stale row behind.

## What This Changes

Adds a gateway maintenance scheduler for stale daily sessions, wires timer cleanup and reload-aware config handling, extends reset reasons, adds focused session tests, and updates session docs.

## Best Possible Solution

Land the proactive scheduler only after scheduled resets mutate the exact scanned store path, target agent id, and matched key, with duplicate-store and alias regression coverage plus explicit maintainer acceptance of proactive daily rollover.

## Reproduction Assessment

Yes. Source inspection shows current main only evaluates daily freshness when an inbound agent request loads a session, and the linked issue describes the no-message-after-boundary scenario; the PR's row-binding defect is also source-reproducible from the duplicate-store resolver contract.

## Solution Assessment

No, not as currently implemented. The scheduler is the right boundary, but it must call a target-bound reset path instead of re-entering the key-only resolver; after that, proactive rollover is maintainable if maintainers accept the compatibility change.
