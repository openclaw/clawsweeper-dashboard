## Summary

Keep open: the scheduler direction is plausible, but current head still sends a single-session reset notification through broad operator and node fanout, so unrelated subscribers can receive the wrong session-state event.

## What This Changes

## Best Possible Solution

Keep the proactive daily reset scheduler, but route scheduled reset notifications through the same targeted operator session-event and node session-fanout contracts used by existing session mutations.

## Solution Assessment

No, not yet. The scheduler and reset-service reuse are an acceptable fix shape, but the notification path needs to use targeted session-event subscribers and `nodeSendToSession` before merge.
