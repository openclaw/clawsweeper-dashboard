## Summary

Keep open: the PR is aimed at a real session-state bug and has strong terminal proof, but the latest branch still has a blocking target-binding bug where a scheduled sweep can reset a different backing-store row than the stale row it scanned.

## What This Changes

The PR adds a gateway maintenance scheduler that proactively scans session stores for stale daily sessions, resets them through the gateway reset service, wires timer cleanup/config reloads, adds tests, and updates session docs.

## Best Possible Solution

Keep the scheduler approach, but bind the reset mutation to the exact scanned store target and matched key, add duplicate-store regression coverage, and explicitly accept or document the proactive rollover upgrade contract before merge.

## Reproduction Assessment

Yes, from source inspection: current main has no maintenance sweep for daily resets and only checks daily freshness during inbound session use. I did not run a live current-main reproduction in this read-only review.

## Solution Assessment

No, not as-is. The scheduler direction is maintainable, but the reset mutation must be bound to the scanned store/key before the PR is safe to merge.
