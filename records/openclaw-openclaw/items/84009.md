## Summary

Keep open: the PR addresses a real gateway/session-state gap and includes sufficient terminal proof, but the scheduled reset path still re-resolves by session key instead of mutating the exact row it scanned, which can reset the wrong persisted session in duplicate-store or legacy-alias cases.

## What This Changes

The PR adds gateway maintenance-driven daily session reset scheduling, lifecycle cleanup, focused tests, and documentation for proactive daily rollover.

## Best Possible Solution

Land a target-bound scheduler that preserves existing reset semantics while passing the scanned store path, agent id, and matched key into the mutation path, with explicit maintainer acceptance of proactive rollover timing.

## Reproduction Assessment

Yes by source inspection. Current main evaluates daily freshness in the inbound/request session-use path and does not start a daily-reset maintenance timer, so a stale session can persist until another message loads it.

## Solution Assessment

No. The scheduler belongs in gateway maintenance, but the implementation should bind the reset mutation to the scanned row before maintainers accept the proactive timing change.
