## Summary

Keep open: the PR addresses a real daily reset gap and includes strong terminal proof, but the new scheduler still has a blocking session-state hazard around duplicate rows across session-store targets. The PR adds a gateway maintenance scheduler for daily session rollover, wires timer cleanup and config reloads, extends reset reason handling, and updates session docs and tests.

## Best Possible Solution

Land proactive daily rollover only after the scheduler revalidates freshness against the authoritative selected store/key immediately before mutation, with regression coverage for cross-store duplicate rows and maintainer acceptance of the upgrade timing behavior.

## Solution Assessment

No, not yet. Reusing the reset service from maintenance is the right shape if proactive rollover is accepted, but the scheduler must bind each reset to the authoritative selected store/key before mutation.
