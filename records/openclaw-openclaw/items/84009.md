## Summary

Keep open: the proactive daily-reset scheduler is the right repair shape, but the current PR head still gates scheduled daily resets on row activity instead of the existing session-start daily freshness contract, so it can leave stale context active after bookkeeping or system-event writes.

## What This Changes

The PR adds a gateway maintenance scheduler for daily session resets, wires its timer into gateway lifecycle cleanup, adds reset-service race guards and tests, and updates session docs.

## Best Possible Solution

Land the proactive gateway maintenance rollover after scheduled staleness uses recovered sessionStartedAt versus dailyResetAt and keeps the active-run, provider-owned CLI, alias/store-target, and race guards intact.

## Solution Assessment

No. The proactive scheduler is the maintainable fix shape, but it must reuse the existing sessionStartedAt daily contract instead of adding an activity-based daily rule.
