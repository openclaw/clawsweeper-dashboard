## Summary

Keep open: the PR has strong proof for the linked daily-reset bug, but the current patch still has a P1 correctness issue because the scheduler scans a specific session store and then resets by key through the generic resolver, which can mutate a different store when duplicate/custom/discovered stores exist.

## What This Changes

This PR adds a gateway maintenance timer and daily session reset scheduler, clears the new timer on shutdown, tests scheduler behavior, and updates session documentation.

## Best Possible Solution

Keep the gateway scheduler approach, but make scheduled reset mutation target the exact scanned session-store row with duplicate/custom-store regression coverage before maintainers accept proactive daily rollover as the daily reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main lacks a daily reset maintenance sweep, and the PR finding follows from scanning `target.storePath` before calling a key-only reset that re-resolves the backing store.

## Solution Assessment

No, not yet. A gateway-owned sweep is the right boundary, but the implementation must mutate the scanned store target and include duplicate/custom-store coverage before the behavior is merge-safe.

- **[P1] Reset the scanned store instead of re-resolving by key:** `src/gateway/session-daily-reset-scheduler.ts:84`
