## Summary

Keep open: the PR addresses a real daily-reset scheduling gap and includes useful terminal proof, but the scheduled reset path still re-resolves by session key instead of mutating the exact scanned store row. That can rotate the wrong current session or leave the stale row untouched in duplicate-key, legacy-alias, or discovered-store cases, so the branch is not merge-ready yet.

## What This Changes

Adds a gateway maintenance daily-session reset scheduler, lifecycle cleanup handles, a daily reset reason path, focused tests, and session docs for proactive daily rollover.

## Best Possible Solution

Land a target-bound proactive daily reset scheduler with duplicate-store and legacy-alias regression coverage, then have maintainers explicitly accept the proactive rollover timing.
