## Risks / Open Questions

- Scheduled reset mutation can target a different backing store than the stale row selected by the scan when duplicate logical session keys exist across configured or discovered stores.
- Merging this changes daily reset deployments from lazy rollover on the next inbound message to proactive rollover shortly after the boundary, so existing users may see session state rotate earlier after upgrade.
- The gateway will scan configured and discovered session stores every minute; overlapping sweeps are guarded, but maintainers should accept that operating cost for larger state directories.
