## Summary

Keep open: the PR addresses a real daily-reset scheduling gap and has sufficient terminal proof, but the scheduler still resets by logical key after scanning a concrete store row, which can rotate the wrong session or leave the stale row unchanged when aliases or duplicate store targets exist.

## What This Changes

The PR adds a gateway maintenance timer that scans session stores for stale daily sessions, resets them through the gateway reset service, wires timer cleanup handles, adds focused tests, and documents proactive daily rollover.

## Best Possible Solution

Land the proactive scheduler only after reset mutation is bound to the scanned store path, target agent id, and matched key, with regression coverage for duplicate store or alias cases and explicit maintainer acceptance of proactive rollover.

## Reproduction Assessment

Yes, by source inspection. Current main evaluates daily reset freshness only when a session is loaded for use and has no maintenance scheduler, matching the linked report where no inbound message arrives after the reset boundary.

## Solution Assessment

No, not yet. The proactive scheduler is a plausible fix shape, but the mutation must be target-bound before merge and maintainers still need to accept the observable proactive rollover behavior.
