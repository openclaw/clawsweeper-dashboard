## Summary

Keep this PR open, but do not merge it yet: the scheduled reset path is useful and well evidenced, but it can still reset the wrong backing session store when duplicate keys exist across configured or discovered stores.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session resets, threads the new timer through gateway lifecycle cleanup, extends reset reason handling, adds focused tests, and updates session docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned store target with duplicate/custom-store regression coverage, then land only if maintainers accept proactive wall-clock rollover as the intended daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main evaluates daily freshness only when a session is used, and the PR's remaining defect follows from the scheduler's key-only reset call plus the reset service's freshest-store re-resolution.

## Solution Assessment

No, not yet. The maintenance scheduler is the right shape for the reported wall-clock reset behavior, but it needs to mutate the scanned store target instead of re-resolving by key before it is merge-safe.
