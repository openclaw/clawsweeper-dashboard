## Summary

Keep open: the PR fixes the right gateway/session area, but the scheduler still uses combined freshness, so a daily+idle config can be reset by idle expiry before the configured daily boundary.

## What This Changes

This PR adds a gateway maintenance timer that proactively resets stale daily session rows, threads conditional reset guards through the reset service, updates timer cleanup, tests, and session docs.

## Best Possible Solution

Land proactive daily rollover only after the scheduled path proves daily-boundary staleness specifically, preserves the row-binding and active-run guards, and documents the accepted upgrade timing.

## Reproduction Assessment

Yes, at source level: configure session.reset with mode daily, a future atHour, and a small idleMinutes; the PR scheduler calls evaluateSessionFreshness, which returns stale on idle expiry even though the daily boundary has not passed.

## Solution Assessment

No, not yet: the row-binding fix is the right direction, but the scheduler should check daily-boundary staleness specifically or explicitly make proactive idle expiry a separate documented product decision.
