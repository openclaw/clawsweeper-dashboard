## Summary

The PR addresses a valid gateway session-state bug with strong terminal proof, but it still has a merge-blocking correctness issue: the new scheduler scans a concrete session store and then resets only by logical key, which can mutate a different backing store when duplicate keys exist.

## What This Changes

The PR adds a gateway maintenance timer that proactively resets stale daily sessions, wires timer cleanup through gateway lifecycle handles, extends reset reason plumbing, adds focused scheduler tests, and updates session reset docs.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the exact scanned store target and matched key with duplicate/custom-store coverage before accepting proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Source inspection shows current main only evaluates daily freshness during later session use and has no daily reset maintenance timer; the PR's remaining store-target bug is also source-reproducible from the scheduler dropping the scanned target before calling the reset service.

## Solution Assessment

No. The scheduler is a plausible fix direction, but this branch should first bind reset mutation to the scanned store target and then get maintainer acceptance of the proactive rollover upgrade behavior.
