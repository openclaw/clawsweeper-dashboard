## Summary

Keep open: the PR addresses a real daily-reset gap with strong terminal proof, but scheduled resets still pass only the session key into a reset service that reselects the backing store, so duplicate or custom-store setups can rotate the wrong session or leave the stale row untouched.

## What This Changes

The PR adds a gateway maintenance timer that scans session stores for stale daily sessions, calls the reset service, wires cleanup handles, adds scheduler and maintenance tests, and updates session docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main only evaluates daily freshness in the session-use path, and the PR's blocking defect follows from resetStaleDailySessions scanning target.storePath while performGatewaySessionReset re-resolves the freshest store from the key.

## Solution Assessment

No, not yet. The maintainable fix is to carry the scanned storePath, agentId, and matched key into an exact-store reset mutation while preserving the existing reset service safety guards.
