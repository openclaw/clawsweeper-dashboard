## Summary

## Best Possible Solution

Keep the scheduler approach, but make the reset mutation target-bound to the scanned store and matched key, cover duplicate-store cases, and explicitly approve the proactive rollover contract before merge.

## Reproduction Assessment

Yes. Current main source-reproduces the issue because daily freshness is evaluated when a session is used, while gateway maintenance has no daily reset sweep; the PR body also supplies after-fix terminal proof for the proposed timer path.

## Solution Assessment

No, not yet. The scheduler is the right boundary for the wall-clock behavior, but the reset operation must be bound to the scanned store row before this is the safest implementation.
