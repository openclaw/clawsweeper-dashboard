## Summary

Keep open: the PR has strong real behavior proof and the gateway-maintenance direction is plausible, but the latest head still has a P1 session-row targeting defect because the scheduler scans one store row and then resets by logical key through the generic resolver.

## What This Changes

The PR adds a gateway maintenance daily session reset scheduler, wires its timer lifecycle into gateway startup/shutdown, adjusts reset-service reason typing, adds focused scheduler/maintenance tests, and updates session docs.

## Best Possible Solution

Land a target-bound scheduler that preserves existing reset semantics while carrying the scanned store path, agent id, and matched key into the mutation path, with explicit maintainer acceptance of proactive rollover timing.

## Reproduction Assessment

Yes, by source inspection. Current main evaluates daily reset freshness in the inbound agent request path and has no daily reset maintenance scheduler, so a stale session can persist until another message loads it.

## Solution Assessment

No. The scheduler belongs in gateway maintenance, but the implementation should bind the reset mutation to the scanned row before maintainers accept the proactive timing change.
