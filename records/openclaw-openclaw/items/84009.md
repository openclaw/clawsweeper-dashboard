## Summary

Keep open: the real-behavior proof is sufficient, but the branch still has a blocking row-binding defect where scheduled reset scans one concrete store row and then mutates through the normal key-only resolver, which can select a different duplicate row/store.

## What This Changes

The PR adds a gateway maintenance scheduler for proactive daily session resets, wires its timer cleanup into gateway lifecycle paths, extends reset reasons, adds focused scheduler tests, and updates session docs.

## Best Possible Solution

Land the proactive scheduler only after scheduled resets mutate the exact scanned store path, target agent id, and matched key, with duplicate-store regression coverage and explicit maintainer acceptance of proactive daily rollover.

## Reproduction Assessment

Yes. Source inspection shows current main evaluates daily freshness only when an inbound agent request loads a session, and the PR-introduced row-binding defect is also reproducible from the resolver contract.

## Solution Assessment

No. The proactive scheduler is the right direction for the reported wall-clock behavior, but it must bind mutation to the scanned row before this is the maintainable fix.
