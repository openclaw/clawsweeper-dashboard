## Summary

This PR is useful and the linked bug remains valid, but it is not merge-ready because scheduled resets still call the generic key-only reset service after scanning a specific store row, which can reset a different backing store when duplicate logical session keys exist.

## What This Changes

The PR adds a gateway maintenance timer that proactively sweeps stale daily sessions, wires timer cleanup and hot-config reads, adds focused scheduler tests, and updates session reset documentation.

## Best Possible Solution

Keep the scheduler approach, but bind reset mutation to the scanned store target and matched key, add duplicate/custom-store regression coverage, and preserve the documented proactive rollover contract intentionally.

## Reproduction Assessment

Yes. Current main evaluates daily freshness only when a session-use path loads the entry, and the PR-introduced target-binding defect is source-reproducible from the scheduler's key-only reset call into a reset service that re-resolves store targets.

## Solution Assessment

No, not yet. The scheduler is the right direction for the linked wall-clock reset bug, but the mutation must be target-bound before this is the narrow maintainable fix.
