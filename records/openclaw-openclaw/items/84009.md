## Summary

Keep this PR open: it fixes a real daily reset scheduling gap, but the current patch can reset the wrong backing session store when the scanned stale row shares a key with a fresher row in another configured/discovered store.

## What This Changes

The PR adds a gateway maintenance scheduler that proactively resets stale daily sessions, clears the new timer on shutdown/restart, extends reset service reasons, adds focused tests, and updates session docs.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is the accepted daily reset contract.

## Reproduction Assessment

Yes, at source level. Current main evaluates daily freshness when a session is used, and the PR's remaining defect follows from the scheduler's key-only reset call plus the reset service's freshest-store re-resolution.

## Solution Assessment

No, not yet. Proactive gateway maintenance is the right direction for the linked bug, but the reset mutation must be bound to the scanned store before this is the maintainable fix.
