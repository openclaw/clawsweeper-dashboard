## Summary

Keep open: the PR addresses a source-reproducible daily reset gap and includes strong terminal proof, but the scheduled reset still mutates by re-resolving the session key through global runtime config instead of the exact store row it scanned, so duplicate logical keys or discovered stores can reset the wrong row while leaving the stale row unchanged.

## What This Changes

The PR adds a gateway daily-session reset scheduler, wires its timer into gateway maintenance and cleanup, extends reset reason handling, adds focused scheduler tests, and updates session docs.

## Best Possible Solution

Keep the scheduler direction, but bind the mutation to the exact scanned store and key, add duplicate-store regression coverage, and make maintainer acceptance of proactive rollover explicit before merge.

## Reproduction Assessment

Yes, by source inspection: current main evaluates daily freshness during inbound/requested-session handling and has no gateway maintenance daily reset timer. I did not establish a live current-main wait across the reset hour.

## Solution Assessment

No, not yet. The scheduler is the right direction for wall-clock daily reset semantics, but the mutation needs to be bound to the scanned store/key before maintainers decide whether proactive rollover is the accepted upgrade behavior.
