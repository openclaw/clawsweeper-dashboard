## Summary

Keep open: this PR appears to be a concrete fix with sufficient real behavior proof, but it deliberately changes when configured daily sessions rotate, so the remaining blocker is maintainer compatibility signoff rather than cleanup closure or an automated repair.

## What This Changes

The branch adds a gateway maintenance scheduler that proactively resets stale daily sessions through the existing reset service, updates race/shutdown coverage, and documents the new rollover timing.

## Best Possible Solution

Land or revise the PR only after maintainer acceptance of proactive wall-clock daily reset semantics, with the active-run, selected-row, and implicit provider-owned CLI safeguards preserved.

## Reproduction Assessment

Yes, from source: current main evaluates daily freshness in the inbound session-use path and gateway maintenance has no daily reset scheduler, so a stale daily session with no inbound message has no current-main reset trigger. I did not run a live current-main repro in this read-only review.

## Solution Assessment

Yes, this is the right layer if maintainers accept proactive daily semantics: it reuses gateway maintenance and the existing reset service instead of adding a parallel reset path. The safer alternative, if compatibility is prioritized, is preserving lazy timing by default and making proactive rollover explicit.
