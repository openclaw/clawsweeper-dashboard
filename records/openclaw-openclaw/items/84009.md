## Summary

The PR addresses a real gateway/session-state gap and has strong terminal proof for the common path, but it is not merge-ready because the scheduled reset path can re-resolve and mutate a different matching session row than the stale row it scanned.

## What This Changes

The branch adds a gateway daily-session reset scheduler, wires it into maintenance/shutdown handles, extends reset reason handling, adds scheduler tests, and updates session docs for proactive rollover.

## Best Possible Solution

Land proactive daily rollover only after scheduled resets are target-bound to the exact scanned store path, agent id, and matched row, with duplicate-store coverage and explicit maintainer acceptance of the compatibility change.

## Reproduction Assessment

Yes by source inspection. Current main evaluates daily freshness only when a session is used and has no daily reset maintenance timer; the remaining PR defect is also visible because the scheduler drops the scanned store target before mutation.

## Solution Assessment

No. The proactive scheduler is the right direction for the linked report, but it should bind mutation to the scanned row before merge and maintainers still need to accept the lazy-to-proactive compatibility change.
