## Summary

Keep open: the branch addresses a real source-reproducible daily reset gap and includes sufficient terminal proof, but it still has a blocking store-target bug that can reset a different session row than the one the scheduler scanned, plus a maintainer-visible proactive rollover compatibility choice.

## What This Changes

The PR adds a gateway maintenance scheduler that scans session stores for stale daily sessions, invokes the gateway reset service, reloads config per sweep, clears timer handles, and updates session docs/tests.

## Best Possible Solution

Bind scheduled daily resets to the exact scanned session-store target with duplicate/custom-store regression coverage, then land only if proactive wall-clock rollover is accepted as the daily reset contract.

## Reproduction Assessment

Yes. Current main only evaluates daily freshness from the session-use path, so the linked no-message-after-boundary issue is source-reproducible; the PR finding is also source-reproducible from the key-only reset call.

## Solution Assessment

No as written. A gateway maintenance sweep is a plausible fix, but the reset mutation must be target-bound before merge and maintainers should explicitly accept proactive wall-clock rollover as the contract.

- **[P1] Reset the scanned store instead of re-resolving by key:** `src/gateway/session-daily-reset-scheduler.ts:84`

Pass the scanned target/store key into the mutation or use a target-bound reset helper.
