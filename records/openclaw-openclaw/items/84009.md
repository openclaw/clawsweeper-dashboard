## Summary

Keep open: the linked daily-reset bug is real and this PR is on the right track with strong proof, but the current branch still has a P1 correctness issue because the scheduler scans a concrete store target and then resets by logical key, allowing duplicate/discovered stores to mutate the wrong session row.

## What This Changes

This PR adds a gateway maintenance scheduler for stale daily session resets, wires timer cleanup and hot-config reads, adds focused gateway/session tests, and updates session docs.

## Best Possible Solution

Keep the scheduler approach, but make the reset mutation target-bound to the scanned store and matched key, add duplicate/custom-store regression coverage, and have maintainers explicitly accept proactive daily rollover as the reset contract.

## Reproduction Assessment

Yes. Current main only evaluates daily freshness during session use, and the PR-introduced target-binding defect is source-reproducible from the scheduler's key-only reset call into a reset service that re-resolves store targets.

## Solution Assessment

No. The scheduler direction is maintainable, but it needs a target-bound reset path plus explicit maintainer acceptance of proactive daily rollover before merge.
