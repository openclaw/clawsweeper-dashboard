## Summary

## What This Changes

The PR adds a gateway maintenance timer that scans session stores every minute and resets stale daily sessions, with reset-service reason plumbing and focused gateway tests.

## Best Possible Solution

Keep proactive daily reset, but make the scheduler read current runtime config or restart/recreate on session config changes, update public session docs, and validate with real gateway crossing-boundary proof.

## Solution Assessment

No, not yet: using the existing reset service and freshness helpers is the right direction, but the scheduler must honor dynamic session config and the public docs/proof need to match the new timing behavior.
