## Summary

Keep open: the PR has strong real-environment proof for the happy path, but the latest head still drops the scanned store target before mutation, so scheduled daily rollover can reset the wrong session row in duplicate/custom-store layouts.

## What This Changes

The PR adds a gateway maintenance scheduler that proactively scans session stores for stale daily sessions, resets them through the gateway reset service, and updates session docs and focused tests.

## Best Possible Solution

Bind scheduled resets to the exact scanned store target with duplicate/custom-store regression coverage, then land only if maintainers accept proactive wall-clock daily rollover as the intended contract.

## Reproduction Assessment

Yes, at source level. Current main evaluates daily freshness in the session-use path, while the PR adds a scheduled path; the remaining PR defect is also source-reproducible from the scheduler's key-only reset call and the reset service's freshest-store re-resolution.

## Solution Assessment

No, not yet. The scheduler shape is plausible, but the mutation must be bound to the scanned store target; otherwise the fix creates a competing source of truth for duplicate/custom stores.
