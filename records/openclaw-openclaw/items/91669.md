## Summary

Keep open: current `main` still has a Feishu/shared dispatch path where an empty `replyResult` produces `queuedFinal=false, replies=0`; the newer generic no-visible fallback does not prove the assistant final text is delivered, and no merged PR or release currently owns this exact group final-reply loss.

## What This Changes

The issue reports Feishu dispatch returning `queuedFinal=false, replies=0` after the agent writes a final assistant response to the session transcript, causing no final message to be sent back to the Feishu group.

## Best Possible Solution

Keep this open and capture a current-main Feishu run with dispatch metadata, then fix the earliest incorrect shared reply-payload or Feishu fallback gate without adding transcript-file scraping to steady-state runtime.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows the reported `replies=0` shape remains possible when the reply resolver returns no final payload despite agent transcript state, but the exact current-main trigger still needs Feishu runtime metadata.

## Solution Assessment

No, the transcript-scraping workaround in the issue is not the best shape. The maintainable fix should preserve final-answer payloads at the embedded-run/reply-dispatch boundary and keep intentional `message_tool_only` suppression intact.
