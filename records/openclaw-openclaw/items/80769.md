## Summary

Keep open: current main still has static exec policy, allowlists, durable approvals, and human/chat approval forwarding, but no isolated verifier-agent approval mode. The prior ClawSweeper comment reached the same conclusion, and a fresh check against current main confirms this is a security-sensitive product/config design request rather than cleanup-close material.

## What This Changes

The issue requests a new exec approval mode where an isolated verifier agent evaluates raw shell commands against policy before allowing execution.

## Best Possible Solution

Design a conservative, fail-closed verifier approval contract in core, with explicit payload isolation, policy schema, fallback semantics, auditability, docs, and regression tests after maintainer security review.

## Reproduction Assessment

Not applicable: this is a new security feature request, not a broken current behavior. Source inspection shows current main lacks the requested verifier-agent mode.

## Solution Assessment

Unclear until maintainers decide the trust model. The raw-command verifier concept is plausible, but the maintainable path is a core approval-policy design rather than an automated bug-fix PR.
