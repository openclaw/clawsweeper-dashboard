## Summary

Close as ClawHub/plugin scope: current main already exposes a pre-tool-call hook plus plugin approval/blocking semantics that can implement a command-only AI verifier outside core, while adding `approvals.exec.mode: "agent"` would be a new optional core policy/config surface.

## What This Changes

The issue requests an agent-backed exec approval mode that sends raw commands to a verifier agent before allowing execution.

## Best Possible Solution

Publish this as a ClawHub guardrail/plugin that uses `before_tool_call` to inspect only the exec command, call a verifier model or endpoint, and block or require human approval; open a narrow SDK/API issue only if that plugin cannot get a necessary command fact.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; the source check shows current main lacks a core `agent` exec-approval mode but already has the plugin hook and approval path needed for an external verifier.

## Solution Assessment

Is this the best way to solve the issue? Yes for maintainer cleanup: keep core on the generic hook/approval contract and route the optional verifier policy to ClawHub/plugin distribution rather than adding a one-off core approval mode.
