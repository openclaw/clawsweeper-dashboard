## Summary

Keep open: current main still only supports static exec policy, allowlists, durable approvals, and human/chat approval forwarding; there is no verifier-agent approval mode. This changes the core exec approval trust model and config contract, and related guardrail/risk-classifier threads do not fully supersede this narrower proposal.

## What This Changes

Request an exec approval mode where an isolated verifier agent receives only the raw command and approves or denies it under configurable policy and fallback settings.

## Best Possible Solution

Define a conservative verifier approval contract, or explicitly route this use case to the plugin guardrail path, then update schema, gateway protocol, execution flow, docs, and tests with fail-closed defaults.

## Reproduction Assessment

Not applicable: this is a feature/security-design request, not a broken existing behavior report. The high-confidence check is source inspection on current main, which shows no verifier-agent mode in exec approval policy, config, protocol, or gateway handling.

## Solution Assessment

Unclear until maintainers decide the trust boundary. The proposed command-only verifier direction is plausible, but the safest maintainable solution needs a fail-closed core design or an explicit decision to keep verifier logic in plugin guardrails.
