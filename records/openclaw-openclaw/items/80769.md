## Summary

Keep open: current main has static exec policy, allowlists, human/chat approval routing, and plugin approval hooks, but it does not implement an isolated verifier-agent mode for exec approvals. This is a new security-sensitive approval policy and config surface, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests a new exec approval mode that sends raw commands to an isolated verifier agent with policy, fallback, timeout, and optional denial notification settings.

## Best Possible Solution

Design a fail-closed verifier contract for exec approvals, decide whether it belongs in core policy or a documented approval hook seam, then implement schema, docs, threat-model tests, and runtime coverage after maintainer security review.

## Reproduction Assessment

Not applicable: this is a feature/security-design request, not a report of broken existing behavior. Source inspection gives a high-confidence state check that current main lacks the requested verifier-agent approval mode.

## Solution Assessment

Unclear until maintainers decide the trust model. The proposed direction is plausible, but it should be shaped as a core approval-policy or documented approval-hook design with fail-closed defaults rather than attempted as a narrow fix.
