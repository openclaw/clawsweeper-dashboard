## Summary

Keep open: current main still does not provide a clear supported path for a memory corpus supplement registered from a plugin service start callback, and the related repair PRs were closed unmerged. The exact failure mode has shifted because late registration-only API calls are now guarded after synchronous register returns, but the central user problem remains unresolved.

## What This Changes

The issue reports that third-party `registerMemoryCorpusSupplement` registrations made from `api.registerService({ start })` can disappear or fail to participate in `memory_search` after plugin reload/lazy-load lifecycle changes.

## Best Possible Solution

Define the plugin SDK lifecycle contract, then either support service-start-owned memory corpus supplements with provenance-aware preservation/replay and regression tests, or document and diagnose that corpus supplements must register synchronously in `register(api)`.

## Reproduction Assessment

Yes, source-level: current main closes registration-only API calls after synchronous `register(api)` returns, services start later, and memory supplement state is still cleared/restored without service-start provenance. I did not run the full live gateway/plugin reproduction.

## Solution Assessment

Unclear until maintainers decide the SDK lifecycle contract. A loader-only preservation patch is plausible but must distinguish service-start-owned supplements from intentionally dropped register-time supplements, while the alternative is docs and diagnostics for synchronous-only registration.
