## Summary

Keep open: current main still has unresolved plugin SDK lifecycle behavior for memory corpus supplements registered from service startup, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to preserve or clearly support MemoryCorpusSupplement registrations made from plugin service startup across lazy plugin reloads so `memory_search` can query them.

## Best Possible Solution

Review one canonical fix that defines the supported lifecycle for memory corpus supplement registration, preserves valid registrations across lazy reloads, and adds regression coverage for the service-start case.

## Reproduction Assessment

Yes, at source level: a service can be registered during `register(api)`, service startup runs later, delayed calls through the captured registration API are guarded, and `memory_search` returns empty results when no corpus supplements remain registered. The exact reporter hypothesis about a successful service-start registration being wiped still needs maintainer confirmation.

## Solution Assessment

Unclear. The narrow fix depends on whether maintainers want `registerMemoryCorpusSupplement` to be supported from `service.start()`, or whether the SDK should require synchronous registration and emit clearer diagnostics/docs for delayed registration attempts.
