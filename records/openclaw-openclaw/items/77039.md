## Summary

Keep open: current main still shows an unresolved plugin SDK lifecycle problem around memory corpus supplements registered from service startup, and open PRs already reference this issue with closing syntax.

## What This Changes

The issue reports that memory corpus supplements registered from a plugin service start callback can be lost when lazy memory_search loading re-runs plugin registration without restarting the service.

## Best Possible Solution

Review and land one canonical fix that defines the supported lifecycle for memory corpus supplement registration and adds regression coverage for lazy memory_search reloads.

## Reproduction Assessment

Yes. Source inspection on current main plus the reporter's trace gives a high-confidence path: service.start runs after register(api), late use of the guarded api cannot reliably mutate registration state, and memory_search returns no supplement hits when the global supplement list is empty.

## Solution Assessment

Unclear. The narrow fix depends on whether maintainers want registerMemoryCorpusSupplement to be supported from service.start, or want the SDK contract to require synchronous register(api) registration with clearer diagnostics/docs.
