## Summary

Keep open: the report is recent and source inspection shows a real SDK/runtime ambiguity around registering memory corpus supplements from a service start callback, even though current main already has safeguards for ordinary non-activating tool-discovery loads.

## What This Changes

The issue asks for MemoryCorpusSupplement registrations made from a plugin service start callback to survive later lazy memory_search tool loading.

## Best Possible Solution

Define the SDK contract for service-start runtime registration, then either document/enforce synchronous registration during register(api) or add an explicit supported runtime seam for service-owned corpus supplements with regression coverage for lazy tool discovery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the observable failure: service.start runs after register() and a captured registration API is closed by then, so a supplement registered only there will not be present when memory_search reads listMemoryCorpusSupplements(). I did not run a live/plugin test because this review was required to stay read-only.

## Solution Assessment

Unclear; preserving snapshot state is already handled for ordinary non-activating tool discovery, but supporting registration from service.start would change or clarify the plugin SDK lifecycle contract. A safer end state is an explicit contract plus a focused regression test, rather than only changing snapshot restore behavior.
