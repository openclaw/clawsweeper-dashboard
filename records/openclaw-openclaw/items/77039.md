## Summary

Keep open: current main still does not define or safely support memory corpus supplement registration from a service-start callback, and the closed fix PRs did not land a canonical loader or SDK lifecycle fix.

## What This Changes

The issue asks OpenClaw to preserve or support `registerMemoryCorpusSupplement` registrations made from a plugin service startup path across lazy memory tool reloads.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration, then land a focused SDK/loader fix or a clear diagnostic/docs update with regression coverage for lazy `memory_search` reloads and failed-register rollback.

## Reproduction Assessment

Yes, source-level. Current main shows that service-start callbacks do not receive a registration API, captured registration methods are closed after synchronous `register()`, and `memory_search` returns no supplement results when the global supplement list is empty.

## Solution Assessment

Unclear. A loader-only merge/preserve fix is not obviously the best answer because the reported plugin pattern crosses the current guarded registration lifecycle; maintainers should first settle the SDK contract and then implement the narrowest matching fix.
