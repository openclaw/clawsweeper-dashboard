## Summary

Keep open: current main still has an unresolved plugin SDK lifecycle gap around corpus supplements registered from `service.start()`, and the linked closing PRs are open rather than merged.

## What This Changes

The issue asks OpenClaw to preserve non-bundled plugin `MemoryCorpusSupplement` registrations made from a service start callback across lazy `memory_search` reloads.

## Best Possible Solution

Define the SDK lifecycle for memory corpus supplements, then land one canonical fix or docs/diagnostic change so service-start registrations are either supported reliably or rejected clearly.

## Reproduction Assessment

Yes. The issue has a concrete live trace, and current source shows the service callback runs after synchronous registration while `memory_search` returns empty when the corpus supplement list is empty; I did not run a live gateway reproduction.

## Solution Assessment

Unclear. The narrow code repair depends on whether maintainers want `service.start()` to be a supported runtime registration point, and the open closing PRs should be reviewed before this issue is closed.
