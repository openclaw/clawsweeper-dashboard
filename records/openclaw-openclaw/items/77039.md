## Summary

Keep open: current main still has a credible plugin SDK lifecycle failure for memory corpus supplements registered from service startup, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to preserve MemoryCorpusSupplement registrations made from a plugin service.start callback across lazy plugin reloads so memory_search can query them.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration, then land one regression-tested loader/SDK fix that keeps third-party plugins and bundled memory-wiki behavior aligned.

## Reproduction Assessment

Yes, at source level. A plugin service starts after register(api), captured registration API calls are closed after register returns, and memory_search returns no supplement results when the supplement registry is empty.

## Solution Assessment

Unclear. The open PRs target plausible loader-level fixes, but current source also raises an SDK contract question about whether registration APIs should work from service.start at all.
