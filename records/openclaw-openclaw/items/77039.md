## Summary

Keep open: current main still has a credible plugin SDK lifecycle gap for memory corpus supplements registered from service startup, and two open fixing PRs already reference this issue with closing syntax.

## What This Changes

Third-party plugins that register a memory corpus supplement from a service start callback should keep that supplement available after lazy memory_search plugin reloads.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration, then land one canonical fix with regression coverage for lazy memory_search reloads.

## Reproduction Assessment

Yes. Source inspection plus the reporter's trace gives a high-confidence path: service.start runs after register(api), memory supplement state is global, and memory_search returns empty results when no supplements are listed.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on whether maintainers want registerMemoryCorpusSupplement to be valid from service.start or want docs/diagnostics to require synchronous register(api) registration.
