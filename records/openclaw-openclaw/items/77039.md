## Summary

Keep open: source inspection still shows a real SDK lifecycle ambiguity for memory corpus supplements registered from a service start callback, while current main already has safeguards for ordinary non-activating lazy tool-discovery loads.

## What This Changes

The issue asks OpenClaw to preserve third-party memory corpus supplements registered from a plugin service start callback across lazy memory_search/tool-discovery loads.

## Best Possible Solution

Define the plugin SDK lifecycle, then either require registerMemoryCorpusSupplement during synchronous register(api) with docs/diagnostics or add an explicit service/runtime registration path with regression coverage for lazy memory_search.

## Reproduction Assessment

Yes. Source inspection plus the reported trace gives a high-confidence path: service.start runs after register(api), and current main closes the captured registration API before service.start can call registerMemoryCorpusSupplement.

## Solution Assessment

Unclear. Preserving snapshot state is already handled for ordinary non-activating lazy loads, so the narrowest maintainable solution depends on whether maintainers want service.start to be a supported runtime registration point.
