## Summary

Keep open: current main still has a credible memory supplement lifecycle failure, and open PR #77065 explicitly targets this issue with closing syntax but is not merged.

## What This Changes

The issue reports that third-party plugins registering a `MemoryCorpusSupplement` from `registerService().start` lose or fail to expose that supplement when lazy `memory_search` loading re-enters plugin registration.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration, then either make service-start registrations survive lazy reloads with regression coverage or emit clear diagnostics and document synchronous register-only usage.

## Reproduction Assessment

Yes. Source inspection plus the reporter’s trace gives a high-confidence path: service-start registration runs after synchronous plugin registration, while loader reload and memory state clearing can leave `memory_search` with an empty supplement list.

## Solution Assessment

Unclear. The linked PR targets the active reload wipe path, but current main also closes captured registration APIs before `service.start`, so maintainers should settle the SDK lifecycle and then land the matching fix.
