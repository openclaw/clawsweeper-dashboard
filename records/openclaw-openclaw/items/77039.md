## Summary

Keep open: current main still has a source-reproducible plugin SDK lifecycle gap around memory corpus supplements, and open PRs already reference this issue as their fix target.

## What This Changes

The issue reports that memory corpus supplements registered from a plugin service start callback are lost after plugin re-registration during lazy memory_search loading.

## Best Possible Solution

Keep the issue open until one canonical fix lands with regression coverage, either preserving service-start corpus supplements across reloads or explicitly documenting and diagnosing late registration as unsupported.

## Reproduction Assessment

Yes, by source inspection rather than live execution. Current main clears global memory plugin state on activating loads and memory_search returns no supplement results when the list is empty; service.start runs outside the guarded registration window.

## Solution Assessment

Unclear. The issue is valid, but #77065 and #77098 take different approaches, and the maintainable fix depends on the intended SDK lifecycle for service-start registration.
