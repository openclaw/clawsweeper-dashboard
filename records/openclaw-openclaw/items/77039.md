- `impact:session-state`: The issue concerns process memory supplement registration state becoming unavailable to `memory_search`, causing expected corpus context to disappear.

## Summary

Keep open: current main and latest v2026.6.8 still do not provide a supported service-start path for `registerMemoryCorpusSupplement`; the original wipe path has shifted into a broader SDK lifecycle decision because registration-only API calls are closed after synchronous `register(api)` returns. The related repair PRs were closed unmerged, so there is no implemented or viable canonical fix to close against.

## What This Changes

The issue asks OpenClaw to preserve memory corpus supplements registered from a plugin service start callback across lazy non-activating plugin re-registers.

## Best Possible Solution

Define the plugin SDK lifecycle for memory corpus supplements, then either support service-start-owned supplement registration with provenance-aware preservation and regression coverage, or document and diagnose that supplements must register synchronously in `register(api)`.

## Reproduction Assessment

Yes, source-level: current main closes registration-only API calls after synchronous `register(api)` returns, services start later with no API in their context, and the non-activating restore path still has no service-start provenance. I did not run the full live gateway/plugin reproduction.

## Solution Assessment

Unclear until maintainers decide the SDK lifecycle contract. A loader-only merge was already shown to be too broad for rollback safety, while docs/diagnostics may be the better answer if late registration is intentionally unsupported.
