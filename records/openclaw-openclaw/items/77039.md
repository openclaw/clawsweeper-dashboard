## Label Justifications

- `P2`: This is a normal-priority plugin SDK/runtime bug affecting memory retrieval from third-party corpus supplements, with a workaround and limited blast radius.
- `impact:session-state`: The issue is about in-process memory supplement registration state being lost or unavailable, which can make memory_search miss expected corpus context.

## Summary

Keep open: current main and the latest release do not contain a merged fix for the reported plugin memory lifecycle, and the closed related PRs were unmerged repair attempts. Source still shows memory state cleared/restored without service-start provenance, while current synchronous-registration guards make the late service-start API contract ambiguous rather than solved.

## What This Changes

The issue reports that third-party memory corpus supplements registered from a plugin service start callback disappear after a lazy memory_search-triggered plugin reload.

## Best Possible Solution

Define the SDK lifecycle contract, then either support service-start-owned memory corpus supplements with provenance-aware restoration/replay and regression tests, or document and diagnose that corpus supplements must register synchronously in register(api).

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a credible reproduction boundary: activating loads clear memory plugin state, service registration can be idempotent across reloads, and current late registration-only API calls are closed after register().

## Solution Assessment

Unclear until maintainers decide the plugin lifecycle contract. A loader-only preservation patch is plausible, but current synchronous-registration guards mean the clean fix may instead be docs plus diagnostics for late registration attempts.
