## Summary

Keep open: current main still shows a concrete plugin SDK lifecycle problem for memory corpus supplements registered from service startup, and two open PRs already reference this issue with closing syntax. The issue should stay open until maintainers choose and land a canonical fix or explicitly document that this registration pattern is unsupported.

## What This Changes

The issue requests that external plugin memory corpus supplements registered from service startup continue to participate in `memory_search` after lazy plugin reloads.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration, then land one canonical loader/API fix with regression coverage for lazy memory_search reloads.

## Reproduction Assessment

Yes. Source inspection plus the reporter's trace gives a high-confidence reproduction path: the registration API is closed after `register(api)`, services start later, and `memory_search` returns no supplement results when the supplement list is empty.

## Solution Assessment

Unclear. Preserving supplements across reloads may be the right fix, but current source also raises the contract question of whether `registerMemoryCorpusSupplement` should be valid from `service.start()` or restricted to synchronous `register(api)`.
