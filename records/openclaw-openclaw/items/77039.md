## Summary

Keep open: current main and the latest release still lack the service-start preservation repair and regression test, while the SDK continues to expose corpus supplements as a supported plugin API.

## What This Changes

Fix a plugin-loader lifecycle bug where MemoryCorpusSupplement registrations can be lost when lazy memory_search tool loading re-runs plugin registration.

## Best Possible Solution

Add a provenance-aware loader/lifecycle fix that preserves service-start-owned corpus supplements across lazy reloads while keeping failed-register rollback exact and allowing register-time supplements to be intentionally removed.

## Reproduction Assessment

Yes, at source level: current main still clears memory plugin state during activating reloads, services start outside the loader, and memory_search reads the global supplement list at query time. I did not run a live packaged reproduction in this read-only pass.

## Solution Assessment

No, the issue is not solved on current main. The best fix is a narrow loader or lifecycle change with regression tests, not a helper-level idempotency change or a blanket merge of restored supplement arrays.
