## Summary

Keep open: current main still has a source-backed SDK lifecycle ambiguity for memory corpus supplements registered from a service start callback, and open PRs #77065 and #77200 both use closing syntax for this issue.

## What This Changes

The issue asks OpenClaw to preserve or clearly support `registerMemoryCorpusSupplement` registrations made from `registerService.start` across lazy plugin reloads.

## Best Possible Solution

Review and land one canonical fix with regression coverage for the chosen lifecycle contract, then close the duplicate fix PR and let this issue close after merge.

## Reproduction Assessment

Yes. Source inspection plus the reporter's trace gives a high-confidence path: service `start` runs outside synchronous `register(api)`, memory supplement state is cleared/restored by loader paths, and `memory_search` returns empty when the supplement list is empty.

## Solution Assessment

Unclear. The current open PRs address related loader preservation paths, but maintainers still need to decide whether service-start registration is supported runtime behavior or should be rejected with docs and diagnostics.
