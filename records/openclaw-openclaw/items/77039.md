## Summary

Keep open: current main and v2026.6.10 still leave the reported memory supplement lifecycle unresolved. The exact shipped wipe path has shifted into a plugin SDK decision because registration-only API calls are closed after synchronous register returns, while the related fix PRs are closed unmerged.

## What This Changes

The issue reports memory corpus supplements registered from a plugin service start callback becoming unavailable after lazy plugin reloads, causing memory_search to miss that corpus.

## Best Possible Solution

Define the SDK lifecycle for memory corpus supplements, then either support service-start-owned registration with provenance-aware preservation and regression coverage, or document and diagnose that supplements must register synchronously in register(api).

## Reproduction Assessment

Yes, source-level: current main closes registration-only API calls after synchronous register(api), services start later without plugin API context, and the loader has no service-start provenance or replay path. I did not run the full live gateway/plugin reproduction.

## Solution Assessment

Unclear until maintainers decide the SDK lifecycle contract. A safe fix should either make service-start-owned memory supplement registration an explicit, tested lifecycle path or reject that pattern with clear docs and diagnostics.
