## Summary

Keep open: current main still has the plugin SDK lifecycle and memory-state restore shape that can leave a service-start corpus supplement unavailable, and the related fix PRs were closed without merging. The remaining fix needs maintainer direction because late registration from `service.start()` conflicts with the guarded synchronous `register(api)` contract.

## What This Changes

The issue reports that `registerMemoryCorpusSupplement` registrations made from a plugin service startup are lost or unavailable after lazy `memory_search` plugin re-registration.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration, then land one focused loader or SDK-contract fix with regression coverage for lazy `memory_search` reloads and failed-register rollback behavior.

## Reproduction Assessment

Yes, source-level. The reporter's trace matches current source where service startup runs after guarded registration, memory supplement state is global, non-activating loads restore memory snapshots wholesale, and `memory_search` returns empty when the supplement list is empty.

## Solution Assessment

Unclear. A loader-only preservation fix may address one failure mode, but the captured-API-from-`service.start()` pattern conflicts with current guarded registration semantics, so maintainers should settle the SDK contract before accepting a repair.
