## Summary

Keep open: current main still shows a source-reproducible SDK lifecycle problem around memory corpus supplements registered from a service start callback, and two open PRs explicitly reference this issue as the implementation path.

## What This Changes

A third-party plugin reports that `registerMemoryCorpusSupplement` registrations made from `api.registerService({ start })` are unavailable to `memory_search` after lazy plugin loader re-entry.

## Best Possible Solution

Define whether memory corpus supplements may be registered from service/runtime startup, then land one canonical PR with regression coverage or document and diagnose the sync-register-only contract.

## Reproduction Assessment

Yes, by source inspection rather than a live run: service `start()` executes after synchronous `register()`, the captured registration API is closed, and activating reloads clear memory supplement state while same-plugin service registration is idempotent.

## Solution Assessment

Unclear. The open PRs target preservation of runtime state, but current source also shows a closed captured API after `register()`, so the narrowest maintainable fix depends on the intended SDK lifecycle contract.
