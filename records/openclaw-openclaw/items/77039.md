## Summary

Keep open: current main still shows a credible plugin SDK lifecycle problem for memory corpus supplements registered from service startup, and two open PRs already reference this issue with closing syntax.

## What This Changes

The issue reports that third-party plugins can lose or fail to expose MemoryCorpusSupplement registrations made from a registerService start callback when lazy memory_search loading re-runs plugin registration.

## Best Possible Solution

Define the supported lifecycle for memory corpus supplement registration and land one canonical regression-tested loader/SDK fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence failing path: service.start runs after register(api), the registration API is closed by then, and memory_search returns no supplement results when the global supplement list is empty.

## Solution Assessment

Unclear. A maintainable fix should first decide whether service.start may register memory corpus supplements, then either preserve those registrations across reloads or reject that lifecycle with clear docs and diagnostics.
