## Summary

Keep open: current main and v2026.6.10 still leave the memory supplement lifecycle unresolved; the related fix PRs are closed unmerged, and current source either makes late service-start registration a no-op or restores memory snapshots without service-start provenance.

## What This Changes

The issue asks OpenClaw to preserve or explicitly define memory corpus supplement registrations made from plugin service startup across lazy non-activating plugin re-registers.

## Best Possible Solution

Define the SDK lifecycle for memory corpus supplements, then either support service-start-owned registration with provenance-aware preservation and regression coverage, or reject that pattern with clear docs and diagnostics.

## Reproduction Assessment

Yes, source-level: current main closes registration-only API calls after synchronous register(api), services start later without plugin API context, and memory_search returns no supplement results when the global supplement list is empty. I did not run the full live gateway/plugin reproduction.

## Solution Assessment

Unclear until maintainers decide the SDK lifecycle contract. The maintainable path is either an explicit supported service-start registration lifecycle or a clear register-time-only contract with diagnostics and docs.
