## Summary

Keep open: current main still shows a real plugin SDK lifecycle problem around memory corpus supplements registered from service startup, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue reports that memory corpus supplements registered from a plugin service start callback can disappear or become unavailable after lazy plugin re-registration.

## Best Possible Solution

Review and land one canonical fix that defines the supported lifecycle for memory corpus supplement registration and adds regression coverage for lazy memory_search reloads.

## Reproduction Assessment

Yes, source inspection plus the reporter's trace gives a high-confidence reproduction path: service.start runs after register(api), memory supplement state is global, and memory_search returns empty results when the supplement list is empty.

## Solution Assessment

Unclear. The narrow code fix depends on whether maintainers want registerMemoryCorpusSupplement to be valid from service.start, or want docs/diagnostics to require synchronous register(api) registration instead.
