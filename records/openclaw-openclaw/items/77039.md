## Summary

Keep open: current main already protects ordinary non-activating tool-discovery loads from replacing active memory state, but the reported service-start registration pattern still exposes an SDK lifecycle ambiguity that needs maintainer direction.

## What This Changes

A third-party plugin asks OpenClaw to keep `registerMemoryCorpusSupplement` registrations made from a registered service start callback available across lazy `memory_search` tool discovery reloads.

## Best Possible Solution

Define the plugin SDK lifecycle contract for runtime service callbacks, then either document/enforce synchronous corpus supplement registration during `register(api)` or add an explicit runtime-safe registration seam with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the observable zero-result behavior: a service callback runs after `register(api)` and calls through a closed captured API, so no supplement reaches the list read by `memory_search`. I did not execute a live plugin scenario in this read-only review.

## Solution Assessment

Unclear; changing snapshot restore alone would not address the current source path where service-start API calls are already closed. The safer solution is to settle the SDK contract and cover the chosen behavior with focused loader/service/memory tests.
