## Summary

Keep open: current main still exposes the reported plugin SDK lifecycle gap, and two open PRs already reference this issue with closing syntax, so maintainers should converge on one fix before cleanup.

## What This Changes

The issue reports that memory corpus supplements registered from plugin service startup are lost or unavailable after lazy memory_search reloads re-run plugin registration without restarting the service.

## Best Possible Solution

Review and land one canonical lifecycle fix that preserves supported memory corpus supplement registrations across lazy plugin reloads, with regression coverage for the service.start path.

## Reproduction Assessment

Yes, from source inspection plus the reporter's trace. register() receives a guarded API, service.start runs later, restore paths still replace memory supplement state, and memory_search returns empty results when the supplement list is empty.

## Solution Assessment

Unclear. A narrow loader fix is likely, but maintainers should first decide whether registerMemoryCorpusSupplement is supported from service.start or whether the SDK should require synchronous register(api) registration with diagnostics.
