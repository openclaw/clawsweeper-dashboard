## Summary

Keep open: current main still has a real plugin SDK lifecycle ambiguity around memory corpus supplements registered from service.start, and two open PRs already reference this issue with closing syntax, so the issue should stay open until one of those fixes is reviewed and landed or explicitly rejected.

## What This Changes

The issue reports that a third-party plugin's registerMemoryCorpusSupplement call from a registered service start callback stops contributing results to memory_search after lazy plugin re-registration.

## Best Possible Solution

Review the open fix PRs together and land one regression-tested lifecycle decision that keeps corpus supplements available across lazy memory_search loads without adding a private bundled-plugin-only path.

## Reproduction Assessment

Yes. Source inspection plus the issue's timestamped trace gives a high-confidence path: service.start runs after register(api), current main closes the captured registration API after register returns, and memory_search returns empty when no corpus supplement remains registered.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on the intended SDK contract for service.start: either support an explicit runtime registration path with regression coverage or fail/document late registration clearly while preserving active supplement state across lazy reloads.
