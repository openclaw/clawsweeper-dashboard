## Summary

Keep open. Current main already has Gateway and browser realtime Talk support, but native macOS Talk Mode still uses the legacy speech-recognition, chat-send/history-polling, and TTS playback loop; maintainer discussion also explicitly narrowed this to native macOS integration on top of the existing realtime session path.

## What This Changes

Request to add native macOS Talk Mode support for OpenAI Realtime speech-to-speech, preserving the existing TTS fallback while reaching parity with the voice-call plugin's realtime phone experience.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer, mic capture, streaming playback, overlay/cancel/barge-in handling, and fallback to the current talk.speak path, reusing talk.realtime.session and the existing relay/WebRTC contracts before adding any new Gateway methods.

## Reproduction Assessment

Not applicable as a feature request; the current behavior gap is high-confidence from source inspection. Native macOS Talk still uses SFSpeechRecognizer plus chat.send/history polling and TTS, while Gateway/browser realtime Talk support exists on current main.

## Solution Assessment

Yes for the narrowed maintainer direction, but not for the original new talk.realtime.open/audio/close API as the default. The maintainable route is to reuse talk.realtime.session and existing realtime transports first, adding new gateway PCM methods only if native macOS cannot use the current contract.
