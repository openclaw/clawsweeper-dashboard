## Summary

Keep open. Current main has the shared gateway/browser realtime Talk contracts, but native macOS Talk still uses local speech recognition into `chat.send` plus `talk.speak`/TTS playback, and the issue discussion explicitly narrowed the remaining work to native macOS integration rather than closure.

## What This Changes

Add native macOS Talk Mode support for realtime speech-to-speech using the existing realtime Talk/provider contracts while preserving the current STT/chat/TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: build a Swift session layer over the existing Talk realtime contracts, wire mic/audio/barge-in/overlay state into Talk Mode, preserve `talk.speak` fallback, and split the ElevenLabs PCM fallback concern separately.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-confirmed: native macOS Talk still uses `SFSpeechRecognizer`, `chatSend`/history, and `talk.speak`/TTS while gateway/browser realtime Talk contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing `talk.client.create` or `talk.session.create` from macOS. No for the original parallel `talk.realtime.open/audio/close` API unless native macOS proves the existing contracts are insufficient.
