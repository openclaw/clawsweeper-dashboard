## Summary

Keep open. Current main already has Gateway/browser realtime Talk session support and a PCM gateway relay, but native macOS Talk still follows the legacy speech-recognition to chat to TTS path. Maintainer discussion also explicitly narrowed this to macOS client integration with the existing realtime session flow, not a new backend voice stack or the ElevenLabs PCM side issue.

## What This Changes

Add native macOS Talk Mode speech-to-speech realtime support, preferably by consuming the existing `talk.realtime.session` flow while keeping the current `talk.speak` path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer around the existing `talk.realtime.session` result contracts, wire microphone capture and incremental assistant audio playback into Talk Mode state/cancellation, preserve provider-secret and ephemeral-session boundaries, and fall back to the current `talk.speak` flow when realtime is unavailable.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS still uses `SFSpeechRecognizer` plus `chat.send`/assistant text/TTS, and no macOS realtime Talk wrapper or client flow exists on current main.

## Solution Assessment

Yes for the narrowed direction of reusing `talk.realtime.session`; no for adding a parallel `talk.realtime.open/audio/close` backend API as the first step. Gateway PCM methods should only be added if native macOS proves the existing session and relay contracts are insufficient.
