## Summary

Keep open. Current main has the Gateway/browser realtime Talk contracts and PCM relay support, but native macOS Talk still uses local speech recognition, `chat.send`, response waiting, and `talk.speak`; the issue discussion explicitly narrows the remaining work to native macOS integration over the existing Talk session/client contracts.

## What This Changes

Add native macOS Talk Mode support for OpenAI Realtime-style speech-to-speech sessions while keeping the existing STT to chat to TTS path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer over the final `talk.client.*` or `talk.session.*` contracts, wire mic/audio/barge-in into Talk Mode, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is source-confirmed: native macOS still follows local STT to `chat.send` to `talk.speak`, while Gateway/browser realtime Talk contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing `talk.client.*` or `talk.session.*`. No for adding the originally sketched parallel `talk.realtime.open/audio/close` API unless macOS proves the existing contracts are insufficient.
