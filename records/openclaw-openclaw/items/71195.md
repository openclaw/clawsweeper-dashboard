## Summary

Keep open. Current main has the shared Gateway/browser realtime Talk path, including OpenAI WebRTC and Gateway relay support, but native macOS Talk still uses the legacy Apple speech recognition to `chat.send` to `talk.speak` loop; maintainer discussion already narrowed this issue to native macOS integration rather than a new backend voice stack.

## What This Changes

Add native macOS Talk Mode support for a realtime speech-to-speech path, preferably by consuming the existing `talk.realtime.session` flow while preserving the current TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer around the existing `talk.realtime.session` contracts, wire mic capture and incremental assistant audio playback into Talk Mode state/cancellation, preserve provider-secret boundaries, and fall back to `talk.speak` when realtime is unavailable.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS still uses Apple speech recognition plus `chat.send`/`chat.history`/TTS, while current main already has Gateway and browser realtime Talk support.

## Solution Assessment

Yes for the narrowed direction of reusing `talk.realtime.session`; no for adding a parallel `talk.realtime.open/audio/close` API as the default first step. New Gateway PCM methods should wait until native macOS proves the existing session and relay contracts are insufficient.
