## Summary

Keep open. Current main has shared realtime Talk gateway/browser contracts, but native macOS Talk still uses the local speech-recognition to chat to TTS path, and live maintainer discussion narrowed this issue to native macOS integration rather than closure.

## What This Changes

Add native macOS Talk Mode support for OpenAI Realtime speech-to-speech while retaining the current STT/chat/TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: implement a Swift realtime session layer over the existing Talk realtime contracts, wire mic/audio/barge-in into Talk Mode, preserve talk.speak fallback, and split any remaining ElevenLabs PCM fallback problem into its own narrow issue.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is source-confirmed: native macOS Talk still uses SFSpeechRecognizer, chatSend/history, and talk.speak/TTS while gateway/browser realtime Talk contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing talk.client.create or talk.session.create. No for the original parallel talk.realtime.open/audio/close API unless native macOS proves the existing contracts are insufficient.
