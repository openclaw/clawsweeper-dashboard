Keep open: current main and the latest release still leave native macOS Talk on the local STT -> chat -> TTS path, while realtime Talk is implemented for browser/client-owned and Gateway-relay surfaces; the live discussion also narrows this to a maintainer-sponsored macOS client integration rather than a new gateway voice stack.

## What This Changes

The issue requests native macOS Talk Mode support for OpenAI Realtime speech-to-speech, with parity to the voice-call phone experience.

## Best Possible Solution

Implement native macOS realtime Talk on the current talk.client.* or talk.session.* contracts, wire AVAudioEngine/WebRTC or PCM relay playback into the Talk runtime and overlay, keep the existing STT/chat/talk.speak fallback, and track the ElevenLabs PCM 403 behavior separately.
