## Summary

Keep open. Current main has the Gateway/browser realtime Talk surfaces and PCM relay support, and the broad realtime voice tracker was closed in favor of this narrower Mac slice; native macOS Talk still follows the local speech recognition to `chat.send` to `talk.speak` path and lacks a native realtime client flow. Maintainer discussion already narrowed the useful direction to native macOS integration over the existing Talk realtime/session contracts, not another backend voice stack.

## What This Changes

Request native macOS Talk Mode support for realtime speech-to-speech parity with voice-call while preserving the existing STT/chat/TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer over the final `talk.client.*` or `talk.session.*` contracts, wire mic/audio/barge-in into Talk Mode, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is high-confidence from source inspection: native macOS still uses `SFSpeechRecognizer`, `chatSend`, response waiting, and `talk.speak`, while Gateway/browser realtime Talk already exists.

## Solution Assessment

Yes for the narrowed direction of reusing the final `talk.client.*` and `talk.session.*` surfaces. No for adding a parallel `talk.realtime.open/audio/close` API by default unless native macOS proves the existing client-owned or Gateway-relay contracts are insufficient.
