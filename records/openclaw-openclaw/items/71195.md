## Summary

Keep open. Current main has the shared realtime voice provider seam, Gateway `talk.realtime.session`, browser WebRTC/relay Talk transports, and relay tests, but native macOS Talk still runs the legacy speech-recognition to chat to TTS loop. Maintainer discussion already narrowed this to native macOS integration with the existing realtime Talk session flow, not a new backend voice stack.

## What This Changes

The issue requests native macOS Talk Mode support for OpenAI Realtime speech-to-speech parity with Voice Call while preserving the current `talk.speak` TTS path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer around the existing `talk.realtime.session` result contracts, wire microphone capture and incremental assistant audio playback into Talk Mode state/cancellation, preserve provider-secret boundaries, and fall back to the current `talk.speak` flow when realtime is unavailable.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS still uses Apple speech recognition plus `chat.send`/`chat.history`/TTS, while current main already has Gateway and browser realtime Talk support.

## Solution Assessment

Yes for the narrowed direction of reusing `talk.realtime.session`; no for adding a parallel `talk.realtime.open/audio/close` API as the default first step. New Gateway PCM methods should wait until native macOS proves the existing session and relay contracts are insufficient.
