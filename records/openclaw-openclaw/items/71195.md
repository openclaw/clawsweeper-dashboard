## Summary

Keep open. Current main already has Gateway/browser realtime Talk APIs and PCM relay support, but native macOS Talk still uses local speech recognition, `chat.send`, and `talk.speak`; issue discussion also explicitly narrowed this to native macOS integration rather than a new backend voice stack.

## What This Changes

Request adds native macOS Talk realtime speech-to-speech using the shared realtime voice contracts while preserving the current STT-to-chat-to-TTS fallback.

## Best Possible Solution

Implement a native Swift realtime Talk session over the existing `talk.client.*` or `talk.session.*` contracts, wire mic/audio/barge-in into `TalkModeRuntime`, and keep `talk.speak` as the fallback.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is high-confidence from source inspection: native macOS still uses `SFSpeechRecognizer`, `chatSend`, response waiting, and `talk.speak`, while Gateway/browser realtime Talk already exists.

## Solution Assessment

Yes for the narrowed direction of reusing the final `talk.client.*` and `talk.session.*` surfaces. No for adding a parallel `talk.realtime.open/audio/close` API by default unless native macOS proves the existing contracts are insufficient.
