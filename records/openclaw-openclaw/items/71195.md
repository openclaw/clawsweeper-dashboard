## Summary

Keep open. Current main has shared gateway/browser realtime Talk contracts, but native macOS Talk still follows local speech recognition into chat and talk.speak/TTS playback, and the live discussion explicitly narrowed the work to native macOS integration rather than closure.

Add native macOS Talk Mode support for OpenAI Realtime speech-to-speech while preserving the existing STT to chat to TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: build a Swift session layer over the existing Talk realtime contracts, wire mic/audio/barge-in into Talk Mode, and preserve talk.speak fallback.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-confirmed: native macOS Talk still uses SFSpeechRecognizer, chatSend/history, and talk.speak/TTS while gateway/browser realtime Talk contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing talk.client.create or talk.session.create. No for the original parallel talk.realtime.open/audio/close API unless native macOS proves the existing contracts are insufficient.
