## Summary

Keep open: current main and v2026.6.10 already have shared Talk realtime APIs, browser realtime Talk, and Android realtime relay, but native macOS Talk still runs the local speech-recognition to chat.send to talk.speak/TTS loop.

## What This Changes

Request native macOS Talk Mode realtime speech-to-speech integration using the shared realtime voice path while preserving the current STT-to-chat-to-TTS fallback.

## Best Possible Solution

Sponsor a focused native macOS integration over existing talk.client.create or talk.session.create APIs, add Swift audio/session UI work plus signed-app proof, and keep the current STT/chat/TTS path as fallback.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.10 shows native macOS Talk still uses SFSpeechRecognizer, chat.send, assistant text waiting, and talk.speak/local TTS instead of a realtime Talk session.

## Solution Assessment

No for the original API sketch. The best path is to reuse current talk.client.create or talk.session.create contracts and add the native Swift session/audio work, not introduce parallel talk.realtime.open/audio/close methods.
