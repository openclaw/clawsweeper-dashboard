## Summary

Keep open. Current main and the latest release still leave native macOS Talk on local speech recognition, Gateway chat, and `talk.speak`, while the realtime Talk contracts already exist for browser, Gateway relay, Android, and iOS; the remaining work is the maintainer-narrowed macOS client integration, not a new gateway voice stack.

## What This Changes

Add a native macOS Talk Mode speech-to-speech realtime path using OpenAI Realtime or the existing Talk realtime contracts while keeping the current STT/chat/`talk.speak` fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: build Swift client/session audio wiring over the existing `talk.client.create` or `talk.session.create` contracts, preserve `talk.speak` fallback, and split the ElevenLabs PCM 403 issue separately.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the current gap: native macOS Talk still uses local speech recognition, `chat.send`/history, and `talk.speak` while realtime Talk session contracts exist elsewhere.

## Solution Assessment

Yes for keeping this open around the maintainer-narrowed native macOS goal. No for adding a parallel `talk.realtime.open/audio/close` API unless the existing `talk.client.create` or `talk.session.create` contracts prove insufficient.
