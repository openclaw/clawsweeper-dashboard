Keep open. ## What This Changes

The issue requests native macOS Talk Mode support for OpenAI Realtime speech-to-speech while preserving the current Talk TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift session layer over the existing Talk realtime contracts, wire mic/audio/barge-in/overlay state, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-confirmed: native macOS Talk still uses local speech recognition, `chat.send`/history, and `talk.speak` while gateway/browser/Android realtime contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing `talk.client.create` or `talk.session.create`. No for the original parallel `talk.realtime.open/audio/close` API unless native macOS proves the existing contracts are insufficient.
