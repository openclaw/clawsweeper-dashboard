## Summary

Keep open. Current main has shared Talk realtime contracts for browser and gateway-relay flows, but native macOS Talk still uses local speech recognition, chat send/history, and `talk.speak`; the live discussion also explicitly narrowed this to native macOS integration rather than another gateway voice stack.

## What This Changes

Add native macOS Talk Mode support for OpenAI Realtime speech-to-speech, reusing existing Talk realtime session contracts while preserving the current `talk.speak` fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: build a Swift session layer over the existing Talk realtime contracts, wire mic/audio/barge-in/overlay state, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-confirmed: native macOS Talk still uses local speech recognition, `chat.send`/history, and `talk.speak` while gateway/browser realtime contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing `talk.client.create` or `talk.session.create`. No for the original parallel `talk.realtime.open/audio/close` API unless native macOS proves the existing contracts are insufficient.
