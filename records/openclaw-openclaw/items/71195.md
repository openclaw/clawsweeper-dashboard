## Summary

Keep open. Current main has the shared realtime Talk gateway/browser contracts, but native macOS Talk still follows the local STT to `chat.send` to `talk.speak` path, and maintainer discussion explicitly narrowed this issue to native macOS client integration rather than closing the feature.

## What This Changes

Add native macOS Talk Mode support for an OpenAI Realtime speech-to-speech path that reuses existing realtime Talk/provider contracts while preserving the current TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer over the existing Talk realtime contracts, wire mic/audio/barge-in into Talk Mode, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is source-confirmed: native macOS Talk still uses local STT, `chat.send`, and `talk.speak`, while gateway/browser realtime Talk contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing the existing `talk.client.*` or `talk.session.*` contracts. No for adding the originally sketched parallel `talk.realtime.open/audio/close` API unless native macOS proves those existing contracts are insufficient.
