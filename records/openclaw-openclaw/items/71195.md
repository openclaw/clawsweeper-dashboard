## Summary

Keep open. Current main has the Gateway and browser realtime Talk pieces, but the native macOS Talk path still runs the legacy speech recognition, chat send/history polling, and TTS playback loop; maintainer discussion also explicitly narrowed this issue to native macOS integration rather than another backend voice stack.

## What This Changes

The issue requests native macOS Talk Mode realtime speech-to-speech support, with phone-like latency and barge-in while preserving the current `talk.speak` fallback path.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer that reuses `talk.realtime.session`, streams microphone audio, plays incremental assistant audio, maps overlay/cancel/barge-in state correctly, and falls back to the existing `talk.speak` path when realtime is unavailable.

## Reproduction Assessment

Not applicable as a feature request; the current behavior gap is high-confidence from source inspection. Native macOS Talk still uses Apple speech recognition plus `chat.send`/`chat.history`/TTS, while current main already has Gateway and browser realtime Talk support.

## Solution Assessment

Yes for the narrowed direction of reusing `talk.realtime.session`; no for adding a parallel `talk.realtime.open/audio/close` API as the default first step. New Gateway PCM methods should be added only if native macOS cannot use the existing session and relay contracts.
