## Summary

Keep open. Current main has the shared gateway/browser realtime Talk contracts, but native macOS Talk still runs the local speech-recognition to chat to TTS loop, and the live issue discussion explicitly narrowed the remaining work to native macOS integration rather than another backend voice stack.

## What This Changes

The issue requests native macOS Talk Mode support for OpenAI Realtime speech-to-speech parity with the voice-call plugin experience.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer over talk.client.create or talk.session.create, wire mic/audio/barge-in into Talk Mode, and preserve talk.speak fallback.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is high-confidence from source inspection: native macOS still uses SFSpeechRecognizer, chatSend, response waiting, and talk.speak while gateway/browser realtime Talk already exists.

## Solution Assessment

Yes for the narrowed direction of reusing the existing talk.client.* or talk.session.* contracts. The original parallel talk.realtime.open/audio/close API sketch is not the best default unless macOS proves the existing client-owned or gateway-relay contracts are insufficient.
