## Summary

Keep open. Current main still has the backend and browser realtime Talk pieces, but native macOS Talk Mode remains on the legacy speech-recognition, chat-send/history-polling, and TTS playback loop; maintainer discussion explicitly narrowed the remaining work to native macOS integration rather than closing this as already implemented.

## What This Changes

Add a native macOS Talk Mode realtime speech-to-speech path using the existing realtime voice infrastructure, while preserving the current STT to chat to TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer, GatewayConnection support for the existing realtime session/relay contract, mic capture, streaming playback, overlay/cancel/barge-in handling, and fallback to the current talk.speak path before introducing any new Gateway PCM methods.

## Reproduction Assessment

Not applicable as a feature request; the current behavior gap is high-confidence from source inspection. Native macOS Talk still uses SFSpeechRecognizer plus chat.send/history polling and TTS, while Gateway/browser realtime Talk support exists on current main.

## Solution Assessment

Yes for the narrowed direction, but no for the original new talk.realtime.open/audio/close API as the default path. The maintainable solution is to reuse talk.realtime.session and existing realtime transports first, adding new Gateway methods only if native macOS cannot use the current contract.
