## Summary

Keep open: current main and v2026.6.10 still leave native macOS Talk on the local speech-recognition to chat.send to talk.speak/TTS loop, while the shared realtime session APIs and browser/gateway-relay paths already exist. The remaining work is a maintainer-sponsored native macOS integration, not a new gateway voice stack.

## What This Changes

Add a native macOS Talk Mode realtime speech-to-speech path using OpenAI Realtime/shared Talk session APIs while preserving the existing STT/chat/TTS fallback.

## Best Possible Solution

Sponsor a focused native macOS integration that consumes the existing talk.client/talk.session APIs, adds Swift audio capture/playback and overlay state, and keeps the current STT/chat/TTS path as fallback.

## Reproduction Assessment

Yes. Source inspection shows native macOS Talk still uses SFSpeechRecognizer, chat.send, assistant response waiting, and talk.speak/local TTS, while realtime Talk sessions exist only through browser/gateway-relay paths.

## Solution Assessment

No for the original API sketch; the current best path is to reuse the existing talk.client.create or talk.session.create contracts rather than adding parallel talk.realtime.open/audio/close methods.
