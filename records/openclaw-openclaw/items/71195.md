## Summary

Keep open. Current main still leaves native macOS Talk on the local STT -> Gateway chat -> `talk.speak` path, while the existing realtime Talk contracts are implemented for browser/Gateway relay and sibling native clients; the issue discussion also explicitly narrows this to native macOS integration rather than a new gateway voice stack.

## What This Changes

Request native macOS Talk Mode support for OpenAI Realtime speech-to-speech, preserving the existing STT/TTS fallback path.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: build a Swift session layer over the existing Talk realtime contracts, wire mic/audio/barge-in/overlay state, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the current gap: native macOS Talk still uses local speech recognition, `chat.send`/history, and `talk.speak` while realtime Talk session contracts already exist elsewhere.

## Solution Assessment

Yes for keeping the issue open around the maintainer-narrowed native macOS goal; no for adding a parallel `talk.realtime.open/audio/close` API unless the existing `talk.client.create` or `talk.session.create` contracts prove insufficient.
