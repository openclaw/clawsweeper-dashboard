## Summary

Keep open. Current main has the reusable gateway/browser realtime Talk contracts, but native macOS Talk still runs the local speech-recognition to chat to TTS flow, and the live discussion explicitly narrowed the remaining work to native macOS integration.

## What This Changes

Requests a native macOS Talk Mode speech-to-speech Realtime path that matches the phone/voice-call experience while preserving the existing Talk TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift session layer over the existing realtime Talk contracts, wire mic/audio/barge-in/overlay state, and preserve the current `talk.speak` fallback instead of adding a parallel gateway API unless the existing contracts prove insufficient.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-confirmed: native macOS Talk still uses local speech recognition, `chat.send`/chat events or history, and TTS playback while gateway/browser realtime contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing `talk.client.create` or `talk.session.create` from native macOS. No for the original parallel `talk.realtime.open/audio/close` API unless macOS proves those existing contracts are insufficient.
