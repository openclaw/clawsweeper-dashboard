## Summary

Keep open. Current main has gateway/browser realtime Talk contracts, but native macOS Talk still runs local speech recognition to `chat.send` to `talk.speak`, and the issue discussion explicitly narrows the remaining work to native macOS client integration rather than closing the feature.

## What This Changes

Add a native macOS Talk Mode realtime speech-to-speech path using the existing realtime Talk provider/session surface while preserving the current TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up: add a Swift realtime session layer over the existing `talk.client.create` or `talk.session.create` contracts, wire mic/audio/barge-in into Talk Mode, and preserve `talk.speak` fallback.

## Reproduction Assessment

Not applicable as a feature request. The current behavior gap is source-confirmed: native macOS still follows local STT to `chat.send` to `talk.speak`, while Gateway/browser realtime Talk contracts already exist.

## Solution Assessment

Yes for the maintainer-narrowed direction of reusing `talk.client.*` or `talk.session.*`. No for adding the originally sketched parallel `talk.realtime.open/audio/close` API unless macOS proves the existing contracts are insufficient.
