## Summary

Keep open. Current main has the shared Gateway and browser realtime Talk path, including a Gateway relay, but native macOS Talk still does not call `talk.realtime.session` and remains on the classic speech-recognition, `chat.send`/history polling, and TTS playback loop. Maintainer discussion has already narrowed this to native macOS integration rather than closing it as implemented.

## What This Changes

Request to add native macOS Talk Mode speech-to-speech realtime audio, ideally reusing the existing OpenAI/realtime voice infrastructure while preserving the current `talk.speak` fallback path.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up. The desired end state is Swift-side realtime session integration, AVAudioEngine capture, streaming playback, overlay phase/cancel handling, barge-in behavior, and fallback to the existing `talk.speak` path for non-realtime providers; the ElevenLabs PCM 403 behavior should stay split into a smaller Talk TTS issue if it still reproduces.

## Reproduction Assessment

Not applicable as a feature request; the current behavior gap is high-confidence from source inspection. Native macOS Talk still uses `SFSpeechRecognizer` plus `chat.send`/history plus TTS, while Gateway/browser realtime Talk support exists on current main.

## Solution Assessment

Yes for the narrowed maintainer direction, but not for the original new `talk.realtime.open/audio/close` API as the default path. The maintainable route is to reuse `talk.realtime.session` and existing realtime transports first, adding new Gateway PCM methods only if native macOS cannot use the current contract.
