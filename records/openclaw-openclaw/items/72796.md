## Summary

Keep open: current main still lacks the requested public hook identity fields, and this PR is useful but not mergeable as-is because it misses live inline agent hook contexts and needs to preserve newer channel normalization. The related #73343 and #72797 context supports the same user-identity extensibility direction, but this PR should stay scoped to agent hook context rather than tool exec environment injection.

## What This Changes

This PR adds optional sender/chat fields to `PluginHookAgentContext`, adds `SenderSessionKey` to `MsgContext`, and threads those values through selected hook/session routing paths.

## Best Possible Solution

Land a rebased additive SDK change that exposes normalized sender/chat identity consistently through all agent hook invocation paths, keeps the current channel normalization helper, and documents the public semantics.

## Reproduction Assessment

Not applicable as a feature PR. There is a high-confidence source-inspection path: current main lacks the requested fields, and the PR adds them only to selected hook/session paths while leaving other live inline hook contexts without identity.

## Solution Assessment

No, not as currently patched. The additive API is a reasonable direction, but the implementation should be rebased and completed so sender/chat fields are consistently populated without regressing current channel normalization.
