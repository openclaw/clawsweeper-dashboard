## Summary

Keep open: this is a plausible additive plugin SDK feature, but the current PR is not mergeable as patched. It addresses the earlier Greptile senderSessionKey guard concern, but it still lacks external real behavior proof, predates current channel-context normalization, and does not populate sender/chat identity across the live agent hook contexts that the feature is meant to unblock.

## What This Changes

The PR adds optional sender/chat fields to PluginHookAgentContext, adds a SenderSessionKey message context field, and gives sender-scoped session routing priority in session initialization.

## Best Possible Solution

Land a rebased additive SDK contract that defines approved sender/chat fields, threads them through every relevant agent hook context while preserving channel normalization, updates focused docs/tests, and keeps exec-env identity to the separate related work.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high-confidence feature-gap evidence: current main omits sender/chat fields from PluginHookAgentContext and live agent hook context builders while run params already carry sender metadata.

## Solution Assessment

No, not as currently patched. The additive API direction is reasonable, but the patch should be rebased, complete agent hook propagation, preserve buildAgentHookContextChannelFields semantics, and document the public contract.
