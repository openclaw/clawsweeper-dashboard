## Summary

Keep open: current main still lacks the requested sender/chat hook context, but this PR is incomplete as an SDK change and still needs contributor proof plus maintainer API review before merge.

## What This Changes

The branch adds optional sender/chat fields to plugin agent hook context, adds `SenderSessionKey` to message context, and gives that key priority in session routing for reply handling.

## Best Possible Solution

Land a rebased additive SDK contract that defines the approved sender/chat fields, threads them through all agent-turn and reply hook contexts via shared helpers, and updates focused docs/tests while keeping exec-env identity in the separate related work.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high-confidence feature-gap evidence: current main omits sender/chat fields from agent hook contexts even though run params already carry sender metadata.

## Solution Assessment

No, not as currently patched. The additive direction is plausible, but the PR must preserve current channel normalization, cover all agent hook context builders, and document the public SDK semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Populate identity in all agent hook contexts:** `src/plugins/hook-types.ts:172-177`
  - body: Adding these fields to `PluginHookAgentContext` is not enough if the main agent-turn hook contexts never carry them. The embedded runner still builds the shared `hookCtx` for `agent_turn_prepare`, `before_prompt_build`, and legacy `before_agent_start` without `params.senderId`, so plugins would see the new public fields as undefined on the primary path.
  - confidence: 0.88
- **[P2] Preserve normalized channel context:** `src/auto-reply/reply/get-reply.ts:584-590`
  - body: This hunk builds `before_agent_reply` context with `channelId: hookMessageProvider`. Current main derives `messageProvider` and `channelId` through `buildAgentHookContextChannelFields`, so rebasing this literally would collapse `channelId` back to the provider instead of the conversation target.
  - confidence: 0.86
- **[P3] Document the new SDK context fields:** `src/plugins/hook-types.ts:172-177`
  - body: These fields and `SenderSessionKey` become public plugin SDK/session contracts, but the hook docs currently define sender context only for message hooks and channel context for agent hooks. Plugin authors need the approved meaning and availability before relying on this API.
  - confidence: 0.9
