## Summary

Keep open: the PR is a plausible Control UI UX improvement, but the current branch still regresses existing protocol-mismatch login guidance and lacks inspectable after-fix real behavior proof.

## What This Changes

The PR changes the Control UI login gate to use a new login-required predicate, keeps transient disconnected states in the dashboard shell with a `Connecting...` topbar pill, and adds a mobile CSS override for that label.

## Best Possible Solution

Keep the reconnect/dashboard-shell split, but include existing high-confidence protocol-mismatch/config failures in the gate predicate with focused coverage before merging with real behavior proof.

## Reproduction Assessment

Yes, at source level. The PR-introduced regression follows from `connected=false`, a configured `gatewayUrl`, `lastError='protocol mismatch'`, and `lastErrorCode=null` returning false from `shouldRenderLoginGate` while current main has protocol-mismatch login-gate recovery and tests.

## Solution Assessment

No, not yet. Splitting transient reconnects from login-required states is the right local surface, but the predicate must preserve existing protocol-mismatch recovery and the PR still needs inspectable real behavior proof.
