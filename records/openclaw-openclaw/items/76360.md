## Summary

Keep this issue open. Current main has request-path caching and bounded fallbacks for `models.list` and `sessions.list`, but I did not find a current-main fix that prewarms the Control UI WebSocket path after Gateway restart; the reporter's v2026.5.2 cold-start measurements are concrete and describe a remaining first-call latency gap after the shipped performance fixes.

## What This Changes

The issue requests fixing or explicitly accounting for >20s first `models.list` and `sessions.list` Control UI WebSocket responses after a Gateway restart on v2026.5.2.

## Best Possible Solution

Keep the issue open until Gateway makes the first Control UI `models.list` and `sessions.list` calls after restart responsive, preferably through bounded post-ready/idle cache warmup or a request path that does not require cold full catalog/store work.

## Reproduction Assessment

No live reproduction was established in this review. Source inspection does show cold process caches and first-request store/catalog loading on the current `models.list` and `sessions.list` paths, which matches the reporter's warm-cache-after-first-call observation.

## Solution Assessment

Unclear whether the reporter's suggested prewarm alone is the best fix. The safest direction is a bounded, non-blocking warmup or faster configured/default view path with observability, so maintainers should choose the performance tradeoff before implementation.
