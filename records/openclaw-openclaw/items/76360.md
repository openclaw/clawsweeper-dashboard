## Summary

Keep this issue open. Current main has newer unreleased `sessions.list` event-loop mitigations and bounded model-catalog fallbacks, but I did not find source, tests, docs, or release provenance proving the first Control UI `models.list` and `sessions.list` calls after a gateway restart are now consistently responsive.

## What This Changes

The issue requests responsive first `models.list` and `sessions.list` Control UI WebSocket calls after a gateway restart, instead of the reported 23-24s cold-cache delay on v2026.5.2.

## Best Possible Solution

Keep the issue open until Gateway has a bounded post-ready/idle warm path, or an equivalently fast cold request path, with observability that distinguishes cold cache warmup from a hung gateway.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source inspection does show cold process caches and request-path session-store/catalog loading, while the report gives concrete v2026.5.2 timings; newer unreleased mitigations make the exact current-main latency uncertain.

## Solution Assessment

Unclear whether unconditional prewarming is the best fix. The narrow maintainable direction is bounded, non-blocking post-ready or idle warmup, or further request-path work that avoids cold synchronous store/catalog pressure without delaying gateway readiness.
