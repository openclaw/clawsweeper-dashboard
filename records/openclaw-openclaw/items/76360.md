## Summary

Keep this issue open. Current main still has cold process caches and request-path session-store/model-catalog loading for the Control UI list RPCs, and the post-v2026.5.2 commits do not add a startup or post-ready prewarm for this path.

## What This Changes

Request to make first Control UI `models.list` and `sessions.list` calls after a Gateway restart responsive instead of taking more than 20 seconds until caches are warm.

## Best Possible Solution

Keep the issue open until Gateway makes first post-restart Control UI `models.list` and `sessions.list` calls responsive through bounded post-ready warmup, cheaper cold paths, or clear cold-cache observability.

## Reproduction Assessment

No live reproduction was established in this review. Source inspection shows cold process caches plus first-request model catalog and session-store loading, which plausibly reproduces the reporter's warm-after-first-call behavior but does not quantify the 20s latency by itself.

## Solution Assessment

Unclear whether prewarming alone is the best way to solve the issue. The maintainable direction is a bounded non-blocking warmup or a cheaper cold request path, with observability so startup work does not simply move into another visible stall.
