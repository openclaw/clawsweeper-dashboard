## Summary

Keep this issue open. Current main still has cold process caches and request-time session-store/model-catalog work on the Control UI list RPC paths, so the v2026.5.2 first-call latency report is not clearly implemented or obsolete.

## What This Changes

The issue reports cold first-call latency for Control UI `models.list` and `sessions.list` after a Gateway restart on v2026.5.2.

## Best Possible Solution

Make the first post-restart Control UI `models.list` and `sessions.list` responses responsive through a bounded post-ready or idle warmup, or by removing cold full catalog/store work from those request paths while preserving readiness safety.

## Reproduction Assessment

No, there is not a high-confidence live reproduction in this review. Source inspection does show cold process caches and first-request store/catalog loading that matches the reporter's warm-after-first-call behavior.

## Solution Assessment

Unclear whether prewarming alone is the best fix. A bounded, non-blocking warmup or a cheaper cold request path with observability is safer than moving unbounded catalog/session work onto startup readiness.
