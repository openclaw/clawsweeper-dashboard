## Summary

Keep open: the branch is pointed at the right Telegram offset-store boundary, but it still treats shipped v2 same-bot offset files as valid and removes stale state asynchronously, so it does not reliably fix the reported token-rotation drop path.

## What This Changes

Adds Telegram token fingerprints to persisted long-poll offset state, exposes rotation inspection and handler helpers, logs and deletes detected stale offsets at polling startup, and adds focused tests.

## Best Possible Solution

Keep the fingerprinted v3 design, but distrust or repair shipped v2 same-bot offset state before polling, await stale-file cleanup, and prove that exact path with regression coverage plus redacted live logs.

## Reproduction Assessment

Yes, at source level: a released v2 offset with the same bot id and a high lastUpdateId is restored as valid, then lower post-rotation update_id values are rejected by the tracker as accepted-watermark. I did not run a live BotFather rotation in this read-only review.

## Solution Assessment

No. Token fingerprinting is the right narrow direction, but this PR preserves the exact released v2 same-bot state that the linked issue reports and should make stale cleanup deterministic before polling can write a new offset.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Handle fingerprint-less v2 offsets:** `extensions/telegram/src/update-offset-store.ts:169-172`
  - body: The linked bug is against released version 2 offset files, which contain only botId and lastUpdateId. This branch only detects same-bot rotation when parsed.tokenFingerprint is already present, so an upgraded v2 same-bot file is classified as valid and still seeds the stale high watermark. Treat tokened fingerprint-less v2 state as stale or add an equivalent startup repair for that upgrade path.
  - confidence: 0.93
- **[P3] Await stale-offset cleanup before polling:** `extensions/telegram/src/offset-rotation-handler.ts:86-88`
  - body: handle() starts deletion in the background, but monitor continues into polling immediately. If polling writes a fresh v3 offset before the unlink resolves, the late unlink can delete the new file and leave disk state behind the running session. Make the startup cleanup awaitable before polling can write a replacement offset.
  - confidence: 0.74
