## Summary

Keep open: this PR is aimed at a valid Telegram stale-offset bug, but current PR head still treats fingerprint-less version 2 same-bot offsets as valid, so it misses the shipped failure path; it also starts stale-offset deletion in the background before polling can write a fresh offset.

## What This Changes

The PR changes the Telegram plugin offset store to persist token fingerprints, expose rotation inspection and handler helpers, log/delete stale offsets on startup, and add focused tests.

## Best Possible Solution

Keep the token-fingerprint direction, but handle fingerprint-less version 2 state safely and await startup cleanup before polling can persist a replacement offset.

## Reproduction Assessment

Yes, at source level: create a version 2 offset with the same bot id and a high lastUpdateId, then inspect or read it with a rotated same-bot token at PR head; it remains valid and seeds the accepted-watermark path. I did not run a live BotFather rotation in this read-only review.

## Solution Assessment

No. Token fingerprinting is the right narrow direction, but this implementation preserves shipped version 2 same-bot state and deletes stale files asynchronously; treating or repairing fingerprint-less version 2 state and awaiting cleanup is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle fingerprint-less v2 offsets:** `extensions/telegram/src/update-offset-store.ts:169-172`
  - body: The reported bug is against released version 2 offset files, which only contain botId and lastUpdateId. This branch only detects same-bot rotation when parsed.tokenFingerprint already exists, so an upgraded v2 same-bot file is classified as valid and still passes the stale high watermark into the tracker. Please treat tokened fingerprint-less v2 state as stale or add an equivalent startup repair for that upgrade path.
  - confidence: 0.91
- **[P3] Await stale-offset cleanup before polling starts:** `extensions/telegram/src/offset-rotation-handler.ts:86-88`
  - body: handle() starts the stale offset deletion in the background, but monitor proceeds immediately to polling and can persist a fresh v3 offset before the unlink finishes. If the unlink resolves after that write, it can delete the new offset file and leave disk state behind the running session. Make the rotation cleanup awaitable on the startup path so the old file is removed before any fresh offset can be written.
  - confidence: 0.72
