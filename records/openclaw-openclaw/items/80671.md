## Summary

Keep open: the PR is aimed at the right Telegram stale-offset bug, but the current branch still trusts released v2 same-bot offset files and can race stale-offset deletion against fresh offset writes.

## What This Changes

The PR adds fingerprinted Telegram offset state, rotation inspection/handler exports, startup warning and cleanup behavior, and focused Telegram offset tests.

## Best Possible Solution

Keep the fingerprinted offset-state approach, but handle tokened v2 same-bot offsets before polling, await stale cleanup, and prove the corrected shipped-state path with regression tests plus redacted live logs.

## Reproduction Assessment

Yes. At source level, current main stores v2 bot-id-only offsets, PR head preserves same-bot v2 offsets as valid, and the tracker skips lower update IDs against that restored watermark.

## Solution Assessment

No. Token fingerprinting is the right narrow direction, but this implementation misses the released v2 same-bot state and should make stale cleanup deterministic before polling can persist a fresh offset.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle fingerprint-less v2 offsets:** `extensions/telegram/src/update-offset-store.ts:169-172`
  - body: The reported bug is against released version 2 offset files, which only contain `botId` and `lastUpdateId`. This branch only detects same-bot rotation when `parsed.tokenFingerprint` already exists, so an upgraded v2 same-bot file is classified as valid and still seeds the stale high watermark. Please treat tokened fingerprint-less v2 state as stale or add an equivalent startup repair for that upgrade path.
  - confidence: 0.91
- **[P3] Await stale-offset cleanup before polling starts:** `extensions/telegram/src/offset-rotation-handler.ts:86-88`
  - body: `handle()` starts deleting the stale offset file in the background, but monitor proceeds immediately to polling and can write a fresh v3 offset before unlink finishes. If the unlink resolves after that write, it can delete the new offset and leave disk state behind the running session. Make rotation cleanup awaitable on the startup path.
  - confidence: 0.72
