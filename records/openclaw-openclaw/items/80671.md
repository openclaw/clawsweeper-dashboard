## Summary

Keep open: the PR is aimed at the right Telegram bug, but the current branch still misses the released v2 same-bot token-rotation path and makes stale-offset cleanup asynchronous before polling starts.

## What This Changes

The PR adds Telegram token fingerprints to persisted update-offset state, introduces rotation inspection/cleanup helpers, wires monitor startup logging/deletion, refactors polling lease fingerprinting, and adds focused tests.

## Best Possible Solution

Keep the token-fingerprint design, but repair or distrust tokened v2 same-bot offset files before polling, make stale deletion awaitable, and add focused regression plus redacted live logs for the shipped rotation path.

## Reproduction Assessment

Yes, at source level: a version 2 offset with the same bot id and high lastUpdateId remains valid at PR head because it has no fingerprint, then the tracker skips lower updates as accepted-watermark. I did not run a live BotFather rotation in this read-only review.

## Solution Assessment

No. Token fingerprinting is the right narrow direction, but this implementation still trusts released v2 same-bot state and should await cleanup before polling can write a fresh offset.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle fingerprint-less v2 offsets:** `extensions/telegram/src/update-offset-store.ts:169-172`
  - body: The reported failure starts from released version 2 offset files, which only store `botId` and `lastUpdateId`. This branch normalizes those files with `tokenFingerprint: null`, and the token-rotation check only runs when `parsed.tokenFingerprint` is truthy, so a same-bot `/revoke` upgrade still returns the stale high watermark. Treat tokened fingerprint-less v2 state as stale or add an equivalent startup repair.
  - confidence: 0.91
- **[P3] Await stale-offset cleanup before polling starts:** `extensions/telegram/src/offset-rotation-handler.ts:86-88`
  - body: `handle()` fires `deleteTelegramUpdateOffset()` in the background, but monitor continues into polling immediately. If a new update is persisted before the unlink resolves, the late unlink can remove the fresh v3 file and leave the next restart without the new state. Make the rotation cleanup awaitable before polling begins.
  - confidence: 0.72
