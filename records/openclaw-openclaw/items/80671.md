## Summary

Keep open: the fingerprint direction is sound, but the latest PR head still trusts released v2 same-bot offset files and the new stale-offset cleanup can race with fresh persistence.

## What This Changes

The PR adds Telegram token fingerprints to persisted long-poll offset state, exports rotation inspection and handler helpers, logs and deletes stale offset files on startup, and adds focused store and handler tests.

## Best Possible Solution

Keep the v3 fingerprint approach, but distrust or repair tokened v2 same-bot offset files before polling, make stale deletion awaitable, and prove that path with regression tests plus redacted live logs.

## Reproduction Assessment

Yes, at source level: create a version 2 offset with the same bot id and a high lastUpdateId, then inspect or read it with a rotated same-bot token at PR head; it is still treated as valid. I did not run the live BotFather path in this read-only review.

## Solution Assessment

No. Token fingerprinting is the right narrow direction, but this implementation misses the released v2 same-bot state and should make startup cleanup deterministic before polling can write a fresh offset.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle fingerprint-less v2 offsets:** `extensions/telegram/src/update-offset-store.ts:169-172`
  - body: The reported bug is against released version 2 offset files, which only contain `botId` and `lastUpdateId`. This branch only detects same-bot rotation when `parsed.tokenFingerprint` already exists, so an upgraded v2 same-bot file is classified as valid and still passes the stale high watermark into the tracker. Please treat tokened fingerprint-less v2 state as stale or add an equivalent startup repair for that upgrade path.
  - confidence: 0.91
- **[P3] Await stale-offset cleanup before polling starts:** `extensions/telegram/src/offset-rotation-handler.ts:86-88`
  - body: `handle()` starts the stale offset deletion in the background, but monitor proceeds immediately to polling and can persist a fresh v3 offset before the unlink finishes. If the unlink resolves after that write, it can delete the new offset file and leave disk state behind the running session. Make the rotation cleanup awaitable on the startup path so the old file is removed before any fresh offset can be written.
  - confidence: 0.72
