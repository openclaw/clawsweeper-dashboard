## Summary

Keep open: the patch is pointed in the right direction, but it does not fix the shipped v2 same-bot token-rotation state from the linked issue because fingerprint-less v2 offsets with a matching bot id are still trusted.

## What This Changes

The PR adds Telegram offset token fingerprinting, rotation inspection/handler surfaces, startup logging and stale-offset deletion, plus focused offset-store and handler tests.

## Best Possible Solution

Keep the fingerprinted v3 direction, but classify or repair tokened fingerprint-less v2 state so upgraded installs do not continue trusting a stale same-bot offset after token rotation.

## Reproduction Assessment

Yes, at source level: create a version 2 offset with the same bot id and a high `lastUpdateId`, then read it with a rotated same-bot token at PR head; it is classified valid and still seeds the tracker watermark. I did not run the live BotFather path in this read-only review.

## Solution Assessment

No. Token fingerprinting is the right narrow fix, but the current PR preserves fingerprint-less v2 same-bot offsets and therefore misses the shipped state shape reported by the linked issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle fingerprint-less v2 offsets:** `extensions/telegram/src/update-offset-store.ts:171-172`
  - body: The linked issue was reported against v2026.5.7, whose offset file is version 2 and has no `tokenFingerprint`. This guard only treats a same-bot token as rotated when a stored fingerprint already exists, so a v2 same-bot file is classified `valid` and its stale high watermark is still passed into the tracker after upgrade/restart. Please treat tokened fingerprint-less v2 state as stale or add an equivalent startup repair for this path.
  - confidence: 0.88
