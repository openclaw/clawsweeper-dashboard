## Summary

Keep open: the patch is aimed at the right Telegram offset boundary, but it still trusts shipped v2 bot-id-only offset files, so existing 2026.5.7 users can remain stuck after same-bot token rotation. It also has only source/unit proof, not real Telegram/BotFather after-fix proof.

## What This Changes

The PR adds Telegram bot-token fingerprints to long-poll offset state, rotation classification/logging, stale offset cleanup, and regression tests for bot identity and token changes.

## Best Possible Solution

Make the Telegram offset reader treat bot-id-only state as unverifiable when a current token is supplied, or add an explicit migration/doctor cleanup path, then demonstrate a redacted real token-rotation receive flow.

## Reproduction Assessment

Yes, at source level: current main writes v2 offsets keyed only by bot id, restores same-bot offsets, and the update tracker skips lower update_id values from the restored watermark. I did not run a live BotFather rotation in this read-only review.

## Solution Assessment

No. Fingerprinting future v3 writes is the right owner boundary, but preserving v2 same-bot offsets leaves the shipped-state failure path alive for existing users.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Invalidate unverifiable v2 offsets:** `extensions/telegram/src/update-offset-store.ts:169-174`
  - body: Latest release writes version 2 bot-id-only offset files. If a user rotates a token before this new fingerprint has been written, `parsed.tokenFingerprint` is null, so this condition skips rotation handling and the function later returns the stale `lastUpdateId` as valid. That leaves the linked same-bot rotation path broken for existing installs; treat bot-id-only state as unverifiable or clear it through an explicit repair path before restoring it.
  - confidence: 0.87
