## Summary

Keep open: this PR is aimed at the right Telegram offset bug, but the current head still preserves shipped v2 same-bot offset files without a token fingerprint and therefore misses the reported upgrade/restart path; CI also has a clear bundled-extension lint failure.

## What This Changes

The PR adds Telegram bot-token fingerprints to persisted long-poll offsets, rotation inspection/handler helpers, startup logging plus stale-offset deletion, and focused Telegram offset tests.

## Best Possible Solution

Keep the fingerprinted v3 direction, but classify or repair tokened fingerprint-less v2 state before it seeds the tracker, add regression coverage for that shipped state shape, fix the lint failure, and then provide live proof for the same-bot rotation path.

## Reproduction Assessment

Yes, at source level: a v2 offset with the same bot id and a high `lastUpdateId` is still classified valid at PR head, and current tracker code skips lower incoming update IDs from that restored watermark. I did not run a live BotFather `/revoke` in this read-only review.

## Solution Assessment

No. Token fingerprinting is the right narrow direction, but this implementation preserves fingerprint-less v2 same-bot offsets and therefore misses the shipped state shape reported by the linked issue; it also needs the lint failure fixed before merge.
