## Summary

Keep open. Current `main` still gates successful `deleteAfterRun` removal on `kind: "at"`, while the API/schema preserve an explicit boolean flag independently of schedule kind, and a linked open PR already targets this issue with closing syntax.

## What This Changes

The issue requests that `deleteAfterRun: true` delete a job after its first successful run for `every` and `cron` schedules as well as `at` schedules.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64375 as the narrow fix with regression coverage, or explicitly reject/document recurring `deleteAfterRun` as unsupported across API, CLI, and docs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard makes `shouldDelete` false and the persisted job is not removed.

## Solution Assessment

Yes, if maintainers accept the existing common-field semantics. Removing the schedule-kind guard while keeping the successful-run check is the narrowest maintainable fix; the alternative is to make recurring `deleteAfterRun` invalid or clearly documented as unsupported.
