## Summary

Keep open. Current `main` still restricts successful `deleteAfterRun` deletion to `kind: "at"`, and the issue is already paired with open unmerged PR #64375 using closing syntax, so it should remain open until that PR lands or maintainers intentionally narrow the contract.

## What This Changes

Honor top-level `deleteAfterRun` for successful `every` and `cron` schedules so recurring jobs can self-delete after their first successful run.

## Best Possible Solution

Review and either land PR #64375 as the narrow canonical fix with regression coverage, or explicitly make recurring `deleteAfterRun` unsupported by tightening the API, CLI, and docs contract.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: apply a successful result to an `every` or `cron` job with `deleteAfterRun: true`; current `main` returns false from `applyJobResult` because of the `kind === "at"` guard, so the store removal path is skipped.

## Solution Assessment

Yes, if maintainers accept the reported semantics. Removing the schedule-kind guard while keeping the successful-run check is the narrowest maintainable fix; the safer alternative is to reject or document recurring `deleteAfterRun` as unsupported.
