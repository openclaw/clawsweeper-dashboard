- `P1`: The bug can make self-deleting cron jobs keep firing and repeatedly dispatch agent or system-event work for real users.
- `impact:message-loss`: The reported failure can duplicate scheduled message or system-event delivery after the job should have been removed.
- `impact:session-state`: The cron store retains jobs despite an accepted self-delete flag, leaving scheduler state inconsistent with the requested lifecycle.

## Summary

Keep open. Current main still gates successful `deleteAfterRun` removal on `schedule.kind === "at"`, and the active closing-syntax fix remains https://github.com/openclaw/openclaw/pull/64375.

## What This Changes

Honor `deleteAfterRun: true` for successful `every` and `cron` schedules instead of only `at` schedules.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64375 as the narrow guard-removal fix with regression coverage, or explicitly enforce and document an at-only contract.

## Reproduction Assessment

Yes. Source inspection is high confidence: a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard makes `shouldDelete` false and the stored job is not removed.

## Solution Assessment

Yes. Removing only the schedule-kind guard while keeping the successful-run check is the narrowest maintainable fix if the common-field contract is accepted.
