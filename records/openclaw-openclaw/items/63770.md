## Summary

Keep open. Current main still gates successful `deleteAfterRun` removal on `schedule.kind === "at"`, while the cron API preserves `deleteAfterRun` as a common job field and an open PR already targets this issue with closing syntax.

## What This Changes

Honor `deleteAfterRun: true` for `every` and `cron` schedules instead of only deleting successful `at` jobs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64375 as the narrow fix with regression coverage, or explicitly reject recurring `deleteAfterRun` and enforce/document an at-only contract.

## Reproduction Assessment

Yes. Source inspection is high confidence: a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard makes `shouldDelete` false and the stored job is not removed.

## Solution Assessment

Yes. Removing only the schedule-kind guard while keeping the successful-run check is the narrowest maintainable fix if the existing common-field contract is accepted; the safer alternative is to reject/document recurring `deleteAfterRun` explicitly.
