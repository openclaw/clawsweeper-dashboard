## Label Justifications

- `P1`: Recurring jobs that should self-delete can continue dispatching scheduled agent or system events for real users.
- `impact:message-loss`: The scheduler can duplicate scheduled message or system-event delivery after the job should have been removed.
- `impact:session-state`: The cron store retains a job despite an accepted self-delete lifecycle flag, leaving persisted scheduler state inconsistent with user intent.

## Summary

Keep open. Current main still ignores explicit `deleteAfterRun: true` for successful `every` and `cron` schedules because the post-run deletion guard remains limited to `schedule.kind === "at"`, and an open closing-syntax PR is already the active implementation path.

## What This Changes

The issue requests that cron `deleteAfterRun: true` remove jobs after the first successful run for `every` and `cron` schedules, not only `at` schedules.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64375 as the narrow guard-removal fix with regression coverage, or explicitly reject and document an at-only `deleteAfterRun` contract.

## Reproduction Assessment

Yes. Source inspection is high confidence: a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard makes `shouldDelete` false and the stored job is retained.

## Solution Assessment

Yes. Removing the schedule-kind guard while preserving deletion only after `result.status === "ok"` is the narrowest maintainable fix if the common-field contract is accepted.
