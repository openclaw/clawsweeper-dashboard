## Label Justifications

- `P1`: The bug can make cron jobs intended to self-delete keep firing, dispatching repeated agent/system events for real users.
- `impact:message-loss`: The reported failure can duplicate scheduled message or system-event delivery after the job should have been removed.
- `impact:session-state`: The cron store retains jobs despite an accepted self-delete flag, leaving scheduler state inconsistent with the requested job lifecycle.

## Summary

Keep open. Current main still gates successful `deleteAfterRun` removal on `schedule.kind === "at"`, while the cron API accepts `deleteAfterRun` as a common job field and https://github.com/openclaw/openclaw/pull/64375 remains the active open closing-syntax implementation path.

## What This Changes

Requests that cron `deleteAfterRun: true` delete successful `every` and `cron` jobs after their first run instead of only `at` jobs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64375 as the narrow guard-removal fix with regression coverage, or explicitly reject recurring `deleteAfterRun` by enforcing and documenting an at-only contract.

## Reproduction Assessment

Yes. Source inspection is high confidence: a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard makes `shouldDelete` false and the stored job is not removed.

## Solution Assessment

Yes. Removing only the schedule-kind guard while keeping the successful-run check is the narrowest maintainable fix if the common-field contract is accepted; the alternative is to reject and document recurring `deleteAfterRun` explicitly.
