## Summary

Keep open. Current main and the latest release still restrict successful `deleteAfterRun` deletion to `kind: "at"`, while the API and job creation/update paths accept an explicit top-level `deleteAfterRun` boolean for recurring jobs. The open linked PR #64375 is the implementation candidate, so this issue should stay open until that PR lands or maintainers deliberately narrow the documented contract.

## What This Changes

The reporter asks for `deleteAfterRun: true` to delete successful `every` and `cron` jobs after their first run instead of only deleting successful `at` jobs.

## Best Possible Solution

Review and either land PR #64375 as the canonical narrow fix with regression coverage and changelog credit, or explicitly make recurring `deleteAfterRun` unsupported by tightening CLI/API/docs so users cannot set a no-op field.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: create or patch an `every` or `cron` job with `deleteAfterRun: true`, then apply a successful run result; current main returns `false` from `applyJobResult` because of the `kind === "at"` guard, so the stored job is not removed.

## Solution Assessment

Yes, if maintainers accept the reported semantics. The narrow maintainable fix is to make deletion depend on explicit `deleteAfterRun === true` plus successful completion and add focused tests proving recurring successes delete while recurring failures remain.
