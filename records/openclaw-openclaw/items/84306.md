## Summary

Keep this PR open: it fixes the main abort-controller registration race in the right Gateway layer, but the new pre-dispatch abort path can still lose idempotency if the client retries the same send after abort and before the original preprocessing settles.

## What This Changes

Moves Gateway `chat.send` abort-controller registration before awaited attachment preprocessing and adds pre-dispatch media cleanup plus abort/duplicate regression coverage.

## Best Possible Solution

Keep the early abort-controller registration, but preserve pre-dispatch abort idempotency until the original send settles and add a regression for abort-then-duplicate-before-dispatch.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.5.28 register the abort controller only after awaited attachment preprocessing, so an abort during that await can miss the run; I did not run the live repro in this read-only pass.

## Solution Assessment

No. Early registration is the right fix shape, but the PR needs an idempotency guard for the abort-then-duplicate window before it is the best maintainable solution.
