## Summary

Keep open: current main and the latest release still have the WhatsApp outbound quote-metadata gap, and https://github.com/openclaw/openclaw/pull/94879 is the open closing fix candidate rather than a merged implementation.

## What This Changes

Fix WhatsApp quote metadata so Desktop renders bot replies when a user swipe-replies to an OpenClaw-authored group message.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94879 after redacted live WhatsApp Desktop proof or an explicit maintainer decision to accept source-level proof.

## Reproduction Assessment

Yes at source level, but not live-reproduced in this review: current main skips the bot's own sent-message echo before quote metadata is cached, then falls back to fromMe:false and the replying participant when building the quote. The reported Desktop rendering symptom still needs live WhatsApp Desktop proof.

## Solution Assessment

Yes, the narrow best fix is to cache bot-authored outbound quote metadata at the existing send-tracking point so later swipe-reply lookups resolve fromMe:true and the bot participant. Guessing later in deliverWebReply would be a weaker boundary because the send path is where the actual outbound message id and bot identity are available.
