## Summary

The issue remains necessary: current main still filters tracked bot-authored echoes before the only quote-metadata cache write, then builds a quoted reply with `fromMe: false` and the replying user as participant when that cache misses. The only linked repair, https://github.com/openclaw/openclaw/pull/91541, closed unmerged because it could not provide live WhatsApp Desktop proof, and no other canonical fix exists.

## What This Changes

Fix WhatsApp Desktop suppressing bot replies when a user swipe-replies to an OpenClaw-authored group message.

## Best Possible Solution

Cache the successful outbound Baileys message key and canonical bot identity at `sendTrackedMessage`, reuse that metadata when constructing later quotes, add focused cache/delivery regression tests, and record a real Android swipe-reply to WhatsApp Desktop rendering proof.

## Reproduction Assessment

No, we do not have a high-confidence live current-main reproduction in this review; however, current source deterministically reproduces the metadata cache miss and incorrect quote-key fallback described by the reporter.

## Solution Assessment

Yes in architecture: recording outbound metadata at the successful send owner is narrower and more maintainable than guessing during reply delivery, but the implementation must use Baileys’ canonical identity and prove the observable Desktop result.
