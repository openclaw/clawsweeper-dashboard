## What This Changes

The PR adds session quota-suspension state, lane concurrency suspension, handoff briefing injection, and diagnostic failover events across agent failover paths.

## Best Possible Solution

If maintainers want this feature, implement it as a documented failover state machine with explicit resume/reject behavior, typed diagnostics, real handoff snapshot generation, regression tests, a changelog entry, and live quota-failover proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of current main plus the latest diff shows the params.config type error, missing diagnostic union member, and zero-concurrency queue path. I did not run tests because this review was read-only.

## Solution Assessment

No; changing cooldown skip behavior into lane suspension without a documented resume/reject state machine is not the narrowest maintainable solution. A safer direction is to preserve current fallback progression unless maintainers approve and test a complete suspension model.

Full review comments:

- **[P1] Pass params.cfg into suspension:** `src/agents/model-fallback.ts:915`
  - body: runWithModelFallback exposes cfg, not config. This new call uses params.config, so the branch will not typecheck and the suspension path cannot receive the session config; pass params.cfg here.
  - confidence: 0.98
- **[P1] Add failover events to the diagnostic union:** `src/infra/diagnostic-events.ts:842`
  - body: DiagnosticEventInput is derived from DiagnosticEventPayload, but the new DiagnosticFailoverEvent is never added to that union. emitTrustedDiagnosticEvent({ type: "model.failover", ... }) is therefore not assignable and consumers cannot narrow the new event type.
  - confidence: 0.96
- **[P2] Reject or resume suspended lanes:** `src/process/command-queue.ts:331`
  - body: Allowing maxConcurrent to become 0 makes enqueueCommandInLane keep accepting work into a lane that drainLane can no longer start. Since the new LaneSuspendedError is unused and the patch does not restore concurrency, commands queued after suspension can wait forever.
  - confidence: 0.92
- **[P2] Transition sessions before handoff injection:** `src/agents/session-suspension.ts:40`
  - body: suspendSession stores state: "suspended", while runEmbeddedAttempt injects the handoff only for state: "resuming". With no transition to resuming and no snapshotRef generation, the promised handoff briefing will not be injected after suspension.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/agents/session-suspension.ts:1`
  - body: This is a user-facing feat that changes failover behavior, but the PR does not update CHANGELOG.md. Add a single-line entry under the active release section before this can ship.
  - confidence: 0.9
