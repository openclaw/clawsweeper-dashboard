## Summary

Keep open: the latest PR addresses some earlier type and diagnostic wiring feedback, but source review still shows blocking behavior defects and the PR still lacks real after-fix runtime proof.

## What This Changes

The PR adds quota-suspension state, command-lane throttling, handoff-message injection, diagnostics, and session-store maintenance around model failover.

## Best Possible Solution

Implement this as a documented failover state machine that only suspends when no immediate fallback can continue, records a real handoff summary, transitions and resumes state on the normal run path, adds regression tests and changelog/docs, and includes live quota-failover proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows lane suspension happens before continuing to later fallback candidates, and normal attempts check for resuming without running the maintenance transition. I did not execute runtime tests because this review was read-only.

## Solution Assessment

No; bolting lane suspension onto cooldown skipping while fallback continues is not the narrowest maintainable solution. A safer path is an explicit failover state machine with resume semantics, real summary generation, tests, and live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Avoid suspending after a recoverable fallback skip:** `src/agents/model-fallback.ts:914-922`
  - body: This branch suspends the lane when one provider candidate is already in cooldown, then immediately continues to the next fallback. If the next candidate succeeds, future commands in the same lane still wait behind maxConcurrent=0 until the timer fires, so a successful fallback can freeze the next turn.
  - confidence: 0.94
- **[P1] Make the handoff state reachable on normal attempts:** `src/agents/pi-embedded-runner/run/attempt.ts:2189-2192`
  - body: The attempt path loads the session store without maintenance and only injects a handoff when state is "resuming", but suspendSession writes "suspended" and the auto-resume timer only restores queue concurrency. Normal next attempts therefore never hit this injection branch.
  - confidence: 0.93
- **[P2] Generate a real handoff summary before injection:** `src/agents/pi-embedded-runner/run/attempt.ts:2200-2202`
  - body: The new summary helper has no call site and all suspension calls omit summary, so even if the state reaches "resuming" the injected handoff falls back to "No recovery briefing was captured" rather than the promised session context.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/agents/session-suspension.ts:1`
  - body: This is a user-facing feature that changes failover and queue behavior, but the PR does not update CHANGELOG.md. Add a single-line entry under the active release section before shipping.
  - confidence: 0.88
