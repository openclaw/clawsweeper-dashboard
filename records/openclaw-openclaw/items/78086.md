## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Advance suspension state on the resume path:** `src/agents/pi-embedded-runner/run/attempt.ts:2229-2232`
  - body: This resume hook reads the session store without running or persisting quota-suspension maintenance, then injects only when the record is already `resuming`. A normally persisted `suspended` record can stay in that state past its TTL, so the promised fallback handoff may never be injected.
  - confidence: 0.95
- **[P2] Restore configured lane concurrency:** `src/agents/session-suspension.ts:20-23`
  - body: The auto-resume timer always restores suspended lanes to `1`. Gateway setup configures the main and subagent lanes from config, with current defaults of 4 and 8, so a resumed lane can remain under-capacity until some later config pass reapplies the real limit.
  - confidence: 0.91
- **[P2] Pass a real summary to suspension writes:** `src/agents/pi-embedded-runner/run.ts:1882-1892`
  - body: This suspension call, and the other new call sites, do not pass `summary`. The resume path therefore falls back to `No recovery briefing was captured.` instead of giving the fallback model the structured recovery briefing promised by the PR body.
  - confidence: 0.9
- **[P2] Export failover diagnostics through OTLP:** `extensions/diagnostics-otel/src/service.ts:2424-2425`
  - body: The diagnostics-otel subscriber explicitly returns for `model.failover`, so the new event is not exported through this plugin even though the PR describes OTLP visibility as part of the feature.
  - confidence: 0.88
- **[P2] Preserve cooldown reasons in suspension state:** `src/agents/model-fallback.ts:946`
  - body: Every non-billing cooldown suspension is persisted as `quota_exhausted`. Transient conditions such as overloaded or timeout cooldowns can then be reported to the fallback model, diagnostics, and users as quota exhaustion.
  - confidence: 0.84
- **[P3] Align the suspended-lane API with queue behavior:** `src/process/command-queue.ts:44-48`
  - body: The PR exports `LaneSuspendedError` as if suspended lanes reject commands, but `enqueueCommandInLane` still queues work when `maxConcurrent` is 0. Either wire this error into the suspended-lane path or remove the exported contract so callers do not depend on behavior that never happens.
  - confidence: 0.79
- **[P3] Add the required changelog entry:** `src/agents/session-suspension.ts:1`
  - body: This is a user-facing feature that changes failover and queue behavior, but the PR does not update `CHANGELOG.md`. Add a single-line entry under the active release section before shipping.
  - confidence: 0.86
