## Summary

Keep open and do not merge yet; the branch has blocking type-level defects, can leave suspended lanes queued indefinitely, and lacks the required real behavior proof for an external PR.

## What This Changes

The PR adds session quota-suspension state, lane concurrency suspension, handoff briefing injection, and diagnostic failover events across agent failover paths.

## Best Possible Solution

If maintainers want this feature, implement it as a documented failover state machine with explicit resume/reject behavior, typed diagnostics, regression tests for cooldown fallback and queue behavior, and live quota-failover proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the diff shows compile-time type errors and a no-resume queue suspension path. I did not run tests because this review was read-only.

## Solution Assessment

No; setting command-lane concurrency to 0 without rejecting or resuming queued work is not the narrowest maintainable way to handle quota failover. A safer direction is an explicit session/lane state machine that preserves existing fallback behavior and surfaces suspended commands predictably.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass the real config into suspension:** `src/agents/model-fallback.ts:915`
  - body: `runWithModelFallback` receives `cfg`, not `config`. This new call reads a non-existent `params.config`, so the branch should fail typechecking and suspension would not receive the session config; pass `params.cfg` here.
  - confidence: 0.98
- **[P1] Add failover events to the diagnostic union:** `src/infra/diagnostic-events.ts:842`
  - body: `DiagnosticEventInput` is derived from `DiagnosticEventPayload`, but the new `DiagnosticFailoverEvent` is never included in that union. As written, `emitTrustedDiagnosticEvent({ type: "model.failover", ... })` is not assignable and consumers cannot narrow the new event type.
  - confidence: 0.96
- **[P2] Reject or resume suspended lanes:** `src/process/command-queue.ts:331`
  - body: Allowing `maxConcurrent` to become 0 makes `enqueueCommandInLane` keep accepting work into a lane that `drainLane` can no longer start. Since the new `LaneSuspendedError` is unused and the patch does not restore concurrency, commands queued after quota suspension can wait forever.
  - confidence: 0.91
