## Summary

Keep this PR open: the Upstash hosting guide is useful, but the current steps introduce a security-sensitive LAN bind in an SSH-tunnel flow and do not document a persistent runtime path that matches the page's keep-alive claim.

## What This Changes

This PR adds a new Upstash Box hosting guide under `docs/install/upstash.md` and registers it in the Hosting navigation in `docs/docs.json`.

## Best Possible Solution

Keep the guide, but revise it so the SSH-tunnel path stays on loopback and the Upstash Box persistence step is explicit, verified, and consistent with OpenClaw Gateway service guidance.

## Reproduction Assessment

Yes for the review findings from source inspection: the added guide combines a loopback SSH tunnel with `gateway.bind lan`, and it uses foreground `openclaw gateway run` while promising persistent keep-alive. I did not run a live Upstash Box in this read-only review.

## Solution Assessment

No. The maintainable docs path is to keep the default SSH-tunnel setup on loopback and add a clear persistent supervisor/init verification step rather than making LAN binding the default.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep the SSH tunnel path on loopback:** `docs/install/upstash.md:52`
  - body: This guide forwards local port 18789 to the Box's `127.0.0.1:18789`, so the Gateway does not need to listen on every interface. Please remove this command from the default tunnel flow or make LAN binding an explicit advanced option with firewall guidance.
  - confidence: 0.91
- **[P2] Document a persistent startup path:** `docs/install/upstash.md:53`
  - body: The page promises a persistent keep-alive Gateway, but the only startup command is foreground `openclaw gateway run`. If the Upstash Box flow relies on the daemon installed by onboarding, the guide should verify that service with status/restart commands; if it relies on Upstash init scripts, document that path. As written, readers can finish with a Gateway tied to the active SSH session instead of a persistent host process.
  - confidence: 0.82
