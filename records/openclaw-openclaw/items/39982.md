## Summary

Keep this PR open, but it still needs changes before merge: current main still uses numeric config backup rotation, while the PR introduces timestamped rotation with a data-retention bug in legacy migration and a collision-allocation bug under same-second bursts.

## What This Changes

The PR changes config backup rotation from numbered `.bak.1` slots to UTC datetime-stamped `.bak.YYYYMMDD-HHmmss` files, updates pruning and permission hardening, and expands backup-rotation tests.

## Best Possible Solution

Preserve the timestamped-backup direction in the config backup module, but repair migration retention, collision suffix allocation, suffix validation, and changelog coverage before maintainers choose between this PR and the competing implementation.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows legacy numbered backups are enqueued as orphans and shows the collision loop reuses the first missing same-second suffix. I did not run tests because the review was read-only.

## Solution Assessment

No, not yet. The module/test boundary is appropriate, but the PR should preserve existing rollback history during migration, choose collision suffixes from the highest existing value plus one, tighten suffix matching, and add the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve legacy numbered backups during migration:** `src/config/backup-rotation.ts:150-153`
  - body: On the first post-upgrade maintenance cycle, existing `.bak.1` through `.bak.4` files fall into the `orphans` bucket and are all unlinked. Current main keeps those files as the recovery ring, and the linked request explicitly calls out migration safety, so this can discard several known-good rollback states immediately after upgrading. Treat legacy numeric slots as valid until they are converted or naturally pruned under the same total backup cap.
  - confidence: 0.9
- **[P2] Choose the next suffix from the highest existing collision:** `src/config/backup-rotation.ts:69-73`
  - body: When cleanup has already pruned lower same-second suffixes, this loop reuses the first missing value such as `-02` even if newer `-03..-06` backups remain. The next cleanup then sorts that freshly rotated backup as the oldest and deletes it, so rapid writes can stop preserving recent rollback states. Parse the existing suffixes for this timestamp and append max+1 instead.
  - confidence: 0.91
- **[P3] Reject unproduced zero collision suffixes:** `src/config/backup-rotation.ts:32`
  - body: The regex accepts `-00` and `-01` because `\d{2,}` includes them, even though the allocator starts at `-02`. User-created files with those suffixes would be treated as managed backups and count toward pruning, so tighten the pattern and add assertions for `-00` and `-01`.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `src/config/backup-rotation.ts:46-49`
  - body: This changes user-visible config backup filenames and upgrade cleanup behavior, but the PR only modifies code and tests. OpenClaw policy requires a `CHANGELOG.md` entry for user-facing feature changes before merge.
  - confidence: 0.87
