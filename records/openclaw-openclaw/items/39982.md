## Summary

Keep this PR open. Current main still uses numeric config backup rotation, so the proposed datetime-suffix work is not obsolete, but the PR branch still has a collision-retention bug under burst writes and overlaps an open sibling implementation for the same request.

## What This Changes

This PR changes config backup rotation from numeric `.bak.1`-style slots to UTC datetime-stamped `.bak.YYYYMMDD-HHmmss` files, updates cleanup/permission handling, and expands backup-rotation tests.

## Best Possible Solution

Merge one canonical datetime-backup implementation only after it allocates collision suffixes monotonically from the highest existing suffix, has deterministic regression coverage for burst writes, keeps retention bounded, hardens all retained backup files, and has an explicit legacy numbered-backup migration policy.

## Reproduction Assessment

Yes. A static trace is enough: with same-second entries like `openclaw.json.bak.T-03` through `T-06`, the PR chooses the first missing `T-02`, and the cleanup sorter then treats `T-02` as oldest and removes the freshly rotated backup.

## Solution Assessment

No. The PR direction is plausible, but this branch is not yet the best fix because collision numbering should continue after the highest retained suffix, and maintainers still need to settle the canonical design against #40392 and the legacy-backup migration policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Continue after the highest collision suffix:** `src/config/backup-rotation.ts:69-73`
  - body: When the lower same-second suffixes have already been pruned, this loop reuses the first missing suffix such as `-02` even if newer siblings like `-03..-06` still exist. The subsequent cleanup sorts by numeric suffix and removes that newly rotated backup as the oldest, so recent rollback points stop advancing during burst writes. Pick `max(existingSuffix) + 1` instead.
  - confidence: 0.92
- **[P3] Stabilize the collision regression test:** `src/config/config.backup-rotation.test.ts:231-244`
  - body: This test learns the current timestamp from one rotation and then expects the next rotation to use the same second. If the test crosses a one-second boundary between those calls, the second rotation gets a new timestamp and the assertion for the old `-03` suffix fails. Inject a clock or timestamp parameter for this regression path.
  - confidence: 0.78
