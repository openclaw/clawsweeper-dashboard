## Summary

Keep open: current main and the latest release still use the numbered backup ring, while this PR still has blocking migration and collision-handling defects plus missing real behavior proof. The related timestamp-backup issue and another open implementation PR mean maintainers should choose one canonical migration path rather than close or auto-repair this branch now.

## What This Changes

This PR changes config backup rotation from numeric `.bak.1` slots to UTC datetime-stamped `.bak.YYYYMMDD-HHmmss` files, updating cleanup, permission hardening, and focused backup-rotation tests.

## Best Possible Solution

Land one canonical timestamp-backup implementation that preserves or converts legacy numbered backups, allocates collision names deterministically from the maximum existing suffix, keeps bounded retention and owner-only permissions, and includes deterministic tests plus real config-write proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head classifies legacy numbered backups as orphans, reuses the first missing collision suffix, accepts `-00`/`-01`, and has a wall-clock-dependent collision test. The underlying request is a feature change, not a current-main bug reproduction.

## Solution Assessment

No, not yet. The backup-rotation module is the right boundary, but the mergeable path needs safe legacy preservation or conversion, max-suffix collision allocation, tighter suffix matching, deterministic tests, real behavior proof, and a maintainer decision on the canonical related implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve legacy backups during migration:** `src/config/backup-rotation.ts:153-159`
  - body: On the first post-upgrade maintenance cycle, existing `.bak.1` through `.bak.4` files are classified as orphans and unlinked. Current main treats those files as the recovery ring, and the linked request calls out migration safety, so preserve or convert them under the same total cap instead of deleting rollback snapshots.
  - confidence: 0.92
- **[P2] Choose collision suffixes from the max existing value:** `src/config/backup-rotation.ts:69-73`
  - body: When cleanup has pruned lower same-second suffixes, this loop can reuse `-02` while newer `-03..-06` backups remain. The next cleanup can then sort that freshly rotated backup as oldest and delete it immediately, so parse existing suffixes for the timestamp and append max plus one.
  - confidence: 0.9
- **[P3] Reject zero-padded collision suffixes below 02:** `src/config/backup-rotation.ts:32`
  - body: The matcher accepts `-00` and `-01` because `\d{2,}` includes them, even though the allocator starts at `-02`. User-created files with those suffixes would be treated as managed backups and count toward pruning, so tighten the pattern and test those values.
  - confidence: 0.86
- **[P3] Make the collision test deterministic:** `src/config/config.backup-rotation.test.ts:231-243`
  - body: The test learns a timestamp from the first rotation, then calls `rotateConfigBackups` again using the wall clock. If the second changes between those calls, the second rotation uses a different timestamp and the test fails even when collision handling is correct; freeze time or inject the timestamp source.
  - confidence: 0.78
