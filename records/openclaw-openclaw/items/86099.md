## Summary

Keep open: current main has Qwen payload shaping for vLLM/OpenAI-compatible requests, but vLLM still does not expose a provider-owned binary off/on thinking profile, so the UI/slash-command mismatch described by the PRD remains.

## What This Changes

Request model-aware thinking controls so vLLM Qwen-style models expose binary off/on choices while OpenAI-style reasoning models keep level-based effort controls.

## Best Possible Solution

Add a vLLM-owned binary thinking profile for explicitly configured Qwen-style models, reuse the existing durable thinking state and payload mapper, and cover session/directive/UI/Pi/docs behavior with focused tests.

## Reproduction Assessment

Yes, at source level: current main has Qwen request payload mapping but vLLM does not expose a resolveThinkingProfile or binary-thinking hook, so control surfaces still fall back to generic level choices. I did not run a live vLLM/Pi backend.

## Solution Assessment

Unclear until maintainers choose the profile-selection contract. The requested provider-owned direction matches the existing architecture, but the implementation should avoid model-id heuristics when explicit model params already carry the stronger Qwen-thinking fact.
