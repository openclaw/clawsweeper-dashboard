## Summary

Keep open: current main and the latest release have the vLLM Qwen payload mapper, but vLLM still does not expose a provider-owned binary thinking profile, so the central UI/slash-command off/on mismatch remains.

## What This Changes

The issue requests model-aware vLLM/Pi thinking controls so Qwen-style vLLM models expose binary off/on thinking while OpenAI-style reasoning models keep level-based controls.

## Best Possible Solution

Add a vLLM-owned thinking profile that exposes `off` and labeled `on` for explicitly configured Qwen-style models, reuses the existing stored thinking/default machinery, preserves the current payload mapper, and covers Pi/session/UI/docs behavior with focused tests.

## Reproduction Assessment

Yes, at source level: current vLLM maps configured Qwen thinking payloads but does not expose a vLLM binary provider profile, so the command/control surface still has no vLLM-owned off/on profile. I did not run a live vLLM server.

## Solution Assessment

Unclear until maintainers choose the vLLM profile-selection contract. The requested provider-owned profile direction matches the existing architecture, but the safest implementation should avoid brittle model-id guesses when explicit model params or compat metadata already carry the stronger fact.
