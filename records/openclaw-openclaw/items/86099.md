## Summary

Keep open: current main already has vLLM/Qwen payload shaping and Pi extra-param propagation, but it does not yet have the requested vLLM-owned binary thinking profile, so UI/slash/session choices can still expose level-style controls for configured Qwen-like vLLM models.

## What This Changes

Request model-aware vLLM and Pi harness thinking controls that expose binary off/on Qwen thinking while preserving level-based reasoning for models that support it.

## Best Possible Solution

Add a vLLM-owned binary thinking profile for configured Qwen formats using the existing durable thinking state and provider profile APIs, extend context only if needed, and cover directive/session/UI rows, Pi propagation, payload mapping, and docs together.

## Reproduction Assessment

Yes, from source inspection: vLLM registers only a stream wrapper while the shared thinking resolver needs a provider profile hook to expose off/on choices. The current payload mapper exists, but the user-facing profile gap remains on current main.

## Solution Assessment

Unclear until maintainers approve the profile/API shape; the requested provider-owned binary profile is directionally aligned, but the broad PRD should be narrowed into a focused design that reuses existing profile and Pi extra-param seams.
