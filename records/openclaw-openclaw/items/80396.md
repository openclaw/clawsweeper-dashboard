## Summary

Keep open: the PR is a useful warn-only fix, but the current wiring emits the warning from `parseReplyDirectives()`, which is also used by planning and comparison paths, so one fenced `MEDIA:` example can produce duplicate or misleading delivery warnings before actual outbound delivery.

## What This Changes

Adds a fenced-MEDIA skip callback to the media parser, wires reply directive parsing to log a warning, and adds parser regression tests.

## Best Possible Solution

Keep the parser signal, but move `logWarn()` to the real outbound delivery boundary so each skipped fenced media token warns once only when a delivery attempt is being prepared.

Yes. Source inspection shows current main keeps fenced `MEDIA:` lines as visible text without a diagnostic, and the PR's remaining defect is also source-reproducible through repeated `parseReplyDirectives()` callers.

## Solution Assessment

No, not yet. The parser callback is a good low-risk signal, but logging from `parseReplyDirectives()` is too broad; the warning should be emitted once at the actual outbound delivery boundary.
