## Summary

Keep open. The warn-only direction is a plausible narrow follow-up to the linked media issue, but the PR currently pulls Node-only logging into a parser that is also bundled by Control UI, and the external real-behavior proof gate is still unmet.

## What This Changes

The PR adds a `logWarn()` when `splitMediaFromOutput()` sees a `MEDIA:` line inside a fenced code block and adds parser tests for that warning while preserving extraction behavior.

## Best Possible Solution

Keep the warn-only diagnostic, but route it through a browser-safe boundary or injected callback instead of importing the Node logger from the shared parser, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main skips fenced lines before `MEDIA:` matching, and the existing parser test preserves that skipped-line behavior; I did not run tests because this review is read-only.

## Solution Assessment

No as currently implemented. Warn-only is a reasonable narrow direction, but the warning must not add Node-only logging dependencies to the shared parser used by Control UI.
