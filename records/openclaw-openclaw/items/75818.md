## Summary

Keep this open. Current main still has the reported packaged dependency-resolution gap: Telegram and the likely related bundled channel plugins keep runtime dependencies in plugin-local package manifests, while packaged OpenClaw excludes nested extension node_modules and current postinstall/release policy prunes or rejects the old dependency-staging layouts. The related closed reports cover loader, SDK-alias, or channel-runtime fixes, not this remaining first-run package dependency contract.

## What This Changes

The issue asks for bundled channel plugins such as Telegram to resolve their runtime dependencies out of the box after a fresh global OpenClaw install, without requiring doctor repair.

## Best Possible Solution

Pick and implement one supported packaging contract for heavy optional bundled channels: either move them to normal downloadable plugin packages with package-local dependencies, or deliberately change the bundled package policy with release/package-acceptance coverage. The final behavior should make doctor repair optional cleanup, not a required first-run dependency installer.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: a fresh global package lacks extension-local node_modules while the bundled Telegram channel load path still reaches a bare grammy runtime import. A live package-activation smoke would be the best executable proof.

## Solution Assessment

No. The issue's doctor workaround is not the best long-term solution because current main treats generated bundled-plugin dependency roots as legacy debris. The maintainable fix should align first-run channel activation with the documented plugin dependency model before changing package artifacts.
