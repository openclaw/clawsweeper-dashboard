## Summary

Keep this open. Current main does not appear to implement a safe out-of-box dependency path for the reported packaged Telegram failure: the Telegram plugin still owns `grammy` and related runtime dependencies locally, the root package does not declare those dependencies, and package/release checks deliberately exclude or reject `dist/extensions/**/node_modules` and legacy runtime-dependency staging. The related closed reports cover different loader/SDK-alias failures and do not resolve this remaining packaged dependency-resolution gap.

## What This Changes

The issue asks for bundled channel plugins such as Telegram, Discord, Nostr, and Slack to resolve their runtime dependencies immediately after a global OpenClaw install, without requiring `openclaw doctor --fix`.

## Best Possible Solution

Resolve the packaging contract so a freshly installed OpenClaw either installs these heavy channel plugins as normal downloadable packages with package-local dependencies, or explicitly changes the bundled package policy with release/package-acceptance coverage for first-run channel activation. The end state should make `doctor --fix` a cleanup/repair tool, not a required first-run dependency installer for bundled channels.

## Reproduction Assessment

Yes. There is a high-confidence static reproduction path: a packaged install lacks root and extension-local `node_modules` for these plugin-owned dependencies, while the Telegram channel plugin load path still reaches a runtime bare `grammy` import.

## Solution Assessment

No, the workaround described in the issue is not the best long-term solution on current main because doctor now treats runtime-dependency staging as legacy debris. A maintainable fix should align packaged channel activation with the documented plugin dependency model before changing release artifacts.
