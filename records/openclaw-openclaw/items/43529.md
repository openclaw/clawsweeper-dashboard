## Summary

Keep this PR open, but do not merge it as-is: current main still lacks timeout-triggered reply session reset, while the PR still has blocking correctness gaps and no after-fix real behavior proof. Prior review discussion improved the branch with a one-reset guard, meta.error coverage, heartbeat skipping, and transient HTTP retry preservation, but the latest head still misses payload-only timeout failures and can over-reset mixed-cause fallback failures.

## What This Changes

The PR adds timeout-triggered session reset handling to reply-agent fallback execution and adds focused runReplyAgent tests for timeout reset behavior.

## Best Possible Solution

Revise the patch so timeout recovery is driven by structured per-attempt fallback reasons, handles payload-only aborted timeout results before success handling, preserves heartbeat/transient-HTTP guards, updates direct callers and regression coverage, and includes redacted real setup proof.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main can emit timeout isError payloads with meta.aborted and no meta.error, while the reply runner has no timeout-reset path for that surface. The linked issue also describes the all-model timeout death spiral that the code path can currently fall through to generic failure handling.

## Solution Assessment

No. The direction is right, but the PR should rely on structured fallback attempts all having timeout reasons and should classify payload-only timeout failures before treating the embedded run as successful.
