## Summary

Keep open: current main still lacks timeout-driven session reset for the reported reply-agent death spiral, but this PR is not merge-ready because it misses a current timeout result surface, can reset on mixed-cause fallback failures, needs rebase/caller updates, and lacks contributor real behavior proof.

## What This Changes

The PR adds a timeout-specific session reset callback to reply-agent fallback execution, wires it from `runReplyAgent`, and adds focused timeout-reset tests.

## Best Possible Solution

Rebase and revise the fix so timeout recovery handles payload-only aborted timeout results, resets only when structured fallback attempts all timed out, updates direct callers/tests, and includes redacted real setup proof.

## Reproduction Assessment

Yes by source inspection, not by a live run in this read-only review. Current main can emit prompt-timeout `isError` payloads with `meta.aborted` and no `meta.error`, while the reply runner and fallback classifier have no timeout-reset path for that result.

## Solution Assessment

No. The direction is right, but the current patch should use structured per-attempt timeout data and classify payload-only timeout failures before success handling instead of relying on `meta.error` or aggregate error text.
