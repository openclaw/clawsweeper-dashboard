## Summary

Keep open: the PR addresses a valid timeout death-spiral bug, but the patch is not merge-ready because it misses a current-main timeout payload path and can reset sessions for mixed-cause fallback failures.

## What This Changes

The PR adds timeout detection to reply fallback handling, resets the active session after timeout failures, returns friendlier retry copy, and adds focused timeout-reset tests.

## Best Possible Solution

Land a rebased, current-main-compatible timeout recovery that resets only after definite all-attempt timeout exhaustion and covers thrown, `meta.error`, and payload-only timeout surfaces with regression tests.

## Reproduction Assessment

Yes, from source inspection rather than a live test: current main can return a timeout `isError` payload with `meta.aborted` and no `meta.error`, and the fallback classifier treats that as non-failing. The reply runner then has no timeout reset hook on current main.

## Solution Assessment

No. The direction is right, but the patch should use structured fallback-attempt reasons and handle payload-shaped timeout failures while preserving the heartbeat and transient-HTTP guards.
