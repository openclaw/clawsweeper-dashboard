## Summary

Keep this PR open: it targets a real timeout death-spiral bug that still appears present on current main, but the branch is not merge-ready because it misses a current timeout payload shape, can reset sessions for mixed-cause fallback failures, conflicts with current callers, and lacks real behavior proof.

## What This Changes

The PR adds timeout-driven session reset handling to `runReplyAgent`/`runAgentTurnWithFallback` plus focused tests for thrown, embedded, unavailable-reset, and heartbeat timeout paths.

## Best Possible Solution

Revise and rebase the fix so timeout recovery handles payload-only timeout failures, resets only when structured fallback attempts are all timeouts, updates current direct callers/tests, and includes redacted real setup proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main can emit prompt timeout failures as `isError` payloads with `meta.aborted` and no `meta.error`, while the fallback classifier treats aborted results as non-fallback success and the reply runner has no timeout reset path.

## Solution Assessment

No. The direction is right, but the patch should use structured per-attempt timeout data and classify payload-only timeout failures before success handling instead of relying on `meta.error` or aggregate error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle payload-only timeout failures before success:** `src/auto-reply/reply/agent-runner-execution.ts:478-480`
  - body: The embedded runner can return a timeout as an `isError` payload with `meta.aborted` and no `meta.error`. Because this branch only checks `runResult.meta?.error`, that timeout can still exit as a successful run and keep the stuck session.
  - confidence: 0.91
- **[P2] Require every fallback attempt to be a timeout:** `src/auto-reply/reply/agent-runner-execution.ts:588-590`
  - body: The catch path applies `isTimeoutError(err)` to aggregate fallback error text. A chain with one timeout plus an auth, billing, or format failure can still match and reset the session even though this recovery should only apply when all model attempts timed out.
  - confidence: 0.87
- **[P2] Update all direct callers for the new timeout callback:** `src/auto-reply/reply/agent-runner-execution.ts:91`
  - body: The PR makes `resetSessionAfterTimeout` a required `runAgentTurnWithFallback` parameter, but current main has many direct test callers that only pass the existing reset callbacks. After rebase, those callers need a no-op timeout callback or the type-check lane will fail.
  - confidence: 0.78
