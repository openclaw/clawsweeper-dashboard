## Summary

Keep open. The underlying timeout death-spiral bug is still present on current main, but this PR is not ready because it misses payload-only timeout failures, can reset sessions for mixed-cause fallback failures, is currently conflicted, and lacks contributor real behavior proof.

## What This Changes

This PR adds timeout-triggered session reset handling in reply-agent fallback execution and focused timeout-reset tests.

## Best Possible Solution

Rebase and revise timeout recovery to handle payload-only aborted timeout results, reset only when structured fallback attempts all timed out, keep heartbeat and transient-HTTP guards, update callers/tests, and add redacted real setup proof.

## Reproduction Assessment

Yes by source inspection, not by a live run in this read-only review. Current main can emit timeout isError payloads with meta.aborted and no meta.error, while the reply runner has no timeout-reset path for that surface.

## Solution Assessment

No. The direction is right, but the current patch should rely on structured per-attempt timeout reasons and classify payload-only timeout failures before success handling instead of relying on meta.error or aggregate error text.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle payload-only timeout failures before success:** `src/auto-reply/reply/agent-runner-execution.ts:478-480`
  - body: The embedded runner can return a timeout as an `isError` payload with `meta.aborted` and no `meta.error`. This branch only watches `runResult.meta?.error`, so that timeout can still break out as a successful run and keep the stuck session.
  - confidence: 0.91

  - confidence: 0.87

  - body: The PR makes `resetSessionAfterTimeout` required, but current main has many direct `runAgentTurnWithFallback` test callers that still pass only the existing reset callbacks; after rebase those callers need a no-op timeout callback or type checking will fail.
  - confidence: 0.78
