## Summary

Keep this PR open because the reported timeout death spiral is still a real source-level session-state problem on current main, but the proposed patch is not merge-ready. It still misses the payload-only aborted timeout surface, can reset on mixed fallback failures by parsing aggregate text, is stale against current runner test call sites, and has no after-fix real behavior proof.

## What This Changes

The PR adds timeout-triggered reply session resets, timeout retry copy, and focused runReplyAgent tests for thrown and `meta.error` timeout paths.

## Best Possible Solution

Implement timeout recovery from structured all-attempt fallback reasons, classify aborted payload-only timeout results before success handling, preserve heartbeat/transient/mixed-failure behavior, and include redacted real runtime proof.

## Reproduction Assessment

Yes, source-level. Current main lacks timeout reset handling in `runAgentTurnWithFallback`, and the embedded runner has a payload-only aborted timeout result that can keep the same transcript active.

## Solution Assessment

No. The direction is right, but the current patch should use structured fallback attempt data and handle payload-only timeout failures before treating the run as successful.
