## Summary

Keep open: current main still lacks the requested timeout-driven session reset, and this PR is useful, but the patch is not merge-ready because it misses a current timeout payload surface and can reset sessions for mixed failure chains that are not all-timeout failures.

## What This Changes

The PR adds timeout detection to runReplyAgent fallback handling, wires a timeout session-reset callback, returns retry-oriented user copy, and adds focused runReplyAgent tests.

## Best Possible Solution

Keep the useful PR open, update it to cover every current timeout surface with structured all-attempt timeout checks, preserve heartbeat and transient-HTTP safeguards, then rebase and validate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: runEmbeddedPiAgent can return a timeout isError payload with meta.aborted and no meta.error, and the current/PR fallback handling treats that as success instead of rotating the session.

## Solution Assessment

No. The direction is right, but the maintainable fix should use structured fallback attempt reasons and timeout payload classification instead of matching any timeout substring in aggregate errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle timeout error payloads before treating the run as success:** `src/auto-reply/reply/agent-runner-execution.ts:478-484`
  - body: The embedded runner can return `{ payloads: [{ isError: true, text: "...timed out..." }], meta: { aborted: true } }` without `meta.error`. Because this branch only checks `runResult.meta?.error`, that timeout still falls through as a successful run and keeps the same bloated session, so the reported death spiral remains for this current timeout surface.
  - confidence: 0.91
- **[P2] Require all fallback attempts to be timeouts before resetting:** `src/auto-reply/reply/agent-runner-execution.ts:588-592`
  - body: The catch path uses `isTimeoutError(err)` on the aggregate fallback error. Since `FallbackSummaryError` concatenates every failed attempt, a mixed chain with one timeout plus an auth, billing, or format failure can still match and rotate the session even though the blocking cause is not an all-model timeout death spiral. Use the structured attempts/reasons to reset only when every exhausted model failed for timeout.
  - confidence: 0.86
