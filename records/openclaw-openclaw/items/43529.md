## Summary

Keep open. The run-reply timeout-reset problem still appears valid on current main, but this PR is not safe to merge as written because it classifies aggregate fallback failures by substring, misses timeout-only error payloads without `meta.error`, and is stale against the current split test/caller surface.

## What This Changes

The PR adds timeout-triggered session reset handling to run-reply fallback execution, wires a timeout reset helper through `runReplyAgent`, and adds focused timeout-reset tests.

## Best Possible Solution

Land a rebased implementation that rotates the run-reply session only after definitive all-model timeout exhaustion on a user turn, handles thrown, `meta.error`, and payload-only timeout surfaces, and preserves existing heartbeat, transient HTTP retry, billing, rate-limit, auth, context-overflow, role-ordering, external-chat, and lifecycle behavior.

## Reproduction Assessment

Yes. Current main source inspection shows no timeout reset path, and focused unit coverage can reproduce both the intended all-model timeout reset and the patch regressions: mixed fallback attempts must not reset, while an embedded timeout returned only as an `isError` payload should reset.

## Solution Assessment

No, not as written. The direction is maintainable, but the safer implementation should use structured fallback attempts for all-timeout proof, classify timeout payloads before accepting a run, and update the current split test/direct-caller surface after rebase.
