## Summary

Keep open: the PR targets a real reply-agent timeout death spiral that current main still does not reset, but the current patch is not mergeable as written because it misses payload-only timeout failures, can reset on mixed fallback failures, needs rebase caller updates, and lacks external real behavior proof.

## What This Changes

The PR adds timeout-driven session reset handling to `runReplyAgent`/`runAgentTurnWithFallback` plus focused tests for thrown, `meta.error`, unavailable-reset, and heartbeat timeout paths.

## Best Possible Solution

Revise and rebase the PR so timeout recovery handles payload-only timeout failures before success handling, resets only when every fallback attempt is a timeout, preserves heartbeat/transient safeguards, and includes redacted real setup proof.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main can emit a timeout as an `isError` payload with `meta.aborted` and no `meta.error`, and the classifier can treat that shape as success without session rotation.

## Solution Assessment

No. The direction is right, but the patch should use structured per-attempt timeout checks and classify payload-only timeout failures before success handling, rather than relying on `meta.error` or aggregate error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle payload-only timeout failures before returning success:** `src/auto-reply/reply/agent-runner-execution.ts:478-480`
  - body: The embedded runner can return a timeout as an `isError` payload with `meta.aborted` and no `meta.error`. Because this branch only checks `runResult.meta?.error`, that timeout can still exit as a successful run and keep the stuck session.
  - confidence: 0.91
- **[P2] Check each fallback attempt before resetting sessions:** `src/auto-reply/reply/agent-runner-execution.ts:584-590`
  - body: The catch path applies `isTimeoutError(err)` to the aggregate `FallbackSummaryError` text. A chain with one timeout plus an auth, billing, or format failure can still match and reset the session even though this recovery is for all models timing out.
  - confidence: 0.87
- **[P2] Update direct callers for the required timeout callback:** `src/auto-reply/reply/agent-runner-execution.ts:91`
  - body: The PR makes `resetSessionAfterTimeout` a required `runAgentTurnWithFallback` parameter, but current main has many direct test callers that only pass the existing reset callbacks. After rebase, those callers need a no-op timeout callback or the type-check lane will fail.
  - confidence: 0.78
