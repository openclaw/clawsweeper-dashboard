## Summary

Keep open: the timeout death-spiral bug is still source-reproducible on current main, but this PR still misses payload-only timeout results, can reset sessions on mixed-cause fallback failures, needs rebase caller updates, and lacks external real behavior proof.

## What This Changes

The PR adds timeout-driven session reset handling to `runReplyAgent`/`runAgentTurnWithFallback` and focused tests for thrown and `meta.error` timeout paths.

## Best Possible Solution

Revise and rebase the PR so timeout recovery handles payload-only failures, resets only when every fallback attempt is a timeout, preserves heartbeat/transient safeguards, updates direct callers/tests, and includes redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection shows current main can emit a prompt-timeout as an `isError` payload with `meta.aborted` and no `meta.error`, and the fallback classifier can return success for that shape without rotating the session.

## Solution Assessment

No. The direction is right, but the patch should use structured per-attempt timeout checks and classify payload-only timeout failures before success handling, rather than relying on `meta.error` or aggregate error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle timeout payloads before returning success:** `src/auto-reply/reply/agent-runner-execution.ts:478-480`
  - body: The embedded runner can return a timeout as an `isError` payload with `meta.aborted` and no `meta.error`. Because this branch only checks `runResult.meta?.error`, that timeout can still exit as a successful run and keep the stuck session.
  - confidence: 0.91
- **[P2] Require all fallback attempts to be timeouts:** `src/auto-reply/reply/agent-runner-execution.ts:584-590`

  - confidence: 0.87
- **[P2] Update direct runner callers for the new callback:** `src/auto-reply/reply/agent-runner-execution.ts:91`
  - body: The PR makes `resetSessionAfterTimeout` a required `runAgentTurnWithFallback` parameter, but current main has many direct test callers that only pass the existing reset callbacks. After rebase, those callers need a no-op timeout callback or the type-check lane will fail.
  - confidence: 0.78
