## Summary

Keep open: the supplied terminal proof is sufficient and the alias mechanics are tested, but the patch still needs maintainer security/product judgment because it makes code-mode `exec` look like shell `exec` to existing hook policies without a hook-visible discriminator.

## What This Changes

The PR adds a `command` alias for code-mode `exec`, mirrors and reconciles `code`/`command` through `before_tool_call` paths, and updates code-mode docs and regression tests.

## Best Possible Solution

Keep the merged context fix from https://github.com/openclaw/openclaw/pull/83481, then choose a hook contract that either adds an explicit code-mode discriminator before the alias or deliberately keeps code-mode policies separate.

## Reproduction Assessment

Yes for the review blocker at source level: current hook events expose only `toolName` and `params`, while this PR adds `command` to the same `exec` tool shape. I did not run tests because this review is read-only, but the PR body includes terminal proof for the alias behavior.

## Solution Assessment

No. The alias-only design proves the mechanics, but the safer contract is to add a hook-visible code-mode discriminator or distinct tool identity before making code-mode cells exec-compatible for policy authors.
