## Summary

Keep open for maintainer review: the PR appears to address the source-level code-mode exec hook gap with focused tests and real hook-runtime proof, but it adds a public hook alias/discriminator contract that affects compatibility and authorization policy semantics.

## What This Changes

The PR adds a code-mode exec `command` alias, hook-visible `toolKind`/`toolInputKind` metadata, alias reconciliation around hook rewrites, docs, and regression coverage for outer code-mode exec hooks.

## Best Possible Solution

Land the alias/discriminator path only after maintainer signoff on the hook contract and upgrade impact, keeping regression coverage for marked code-mode exec, plain exec, trusted rewrites, and divergent aliases.

## Reproduction Assessment

Yes at source level. Current main's code-mode exec reads only `code` and the adapter passes raw params to `before_tool_call`, so a hook policy looking for `params.command` will not see code-mode source; I did not execute it because this review is read-only.

## Solution Assessment

Yes, if maintainers accept the contract. Mirroring aliases only for marked code-mode control tools plus `toolKind`/`toolInputKind` is narrower than renaming the tool, while still leaving the public hook-shape decision to maintainers.
