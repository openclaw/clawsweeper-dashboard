## Summary

Keep open for maintainer review: the patch is focused and proof-backed with no blocking correctness finding, but it intentionally changes public authorization-hook inputs for outer code-mode exec and needs explicit signoff on the compatibility and security-boundary semantics.

## What This Changes

The PR adds a code-mode exec `command` alias, hook-visible `toolKind`/`toolInputKind` discriminators, normalization around `before_tool_call` and trusted rewrites, docs, and regression tests.

## Best Possible Solution

Land the alias/discriminator contract only if maintainers accept the compatibility and authorization-hook semantics, preserving the regression coverage for marked code-mode exec, plain exec, trusted rewrites, and divergent aliases.

## Reproduction Assessment

Yes at source level. Current main's code-mode exec reads only `params.code`, and the Pi adapter passes raw params to `before_tool_call`, so a hook policy looking only at `params.command` cannot see the code-mode source without this new alias path.

## Solution Assessment

Yes, if maintainers want shared exec-shaped policy coverage. The alias plus host-authoritative discriminator is the narrowest reviewed shape here; if maintainers do not want shell policies matching code-mode by default, a distinct tool name or the current `code`-only contract is safer.
