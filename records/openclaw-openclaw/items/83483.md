## Summary

Keep open: the patch is coherent, the new runtime proof is sufficient, and I found no discrete blocking code finding. The remaining blocker is maintainer acceptance of a public hook/security contract change: code-mode `exec` would now expose source as `params.command` while relying on `toolKind` to keep shell-style policies distinct.

## What This Changes

The PR adds a `command` alias plus `toolKind`/`toolInputKind` metadata for outer OpenClaw code-mode `exec` hook dispatches, with docs and focused regression coverage.

## Best Possible Solution

Land this only after maintainer acceptance that `params.command` is the supported exec-compatible alias for code-mode source and `toolKind`/`toolInputKind` are the supported separation mechanism for shell-versus-code policies.

## Reproduction Assessment

Yes, at source level: current main documents and implements code-mode `exec` as `code`-only and the plugin hook types lack `toolKind`/`toolInputKind`. I did not run tests because this cleanup review is read-only.

## Solution Assessment

Yes, if maintainers accept the contract: marking only code-mode control tools and adding additive hook discriminators is a maintainable way to resolve the shared `exec` name. The product/security choice is whether `command` should also be model-facing or kept hook-only.
