## Summary

Keep open: current `main` still has the cron cleanup gap, but this PR is not merge-ready because it drops the removed-job hook payload and does not handle active-run or pending run-log write races; the external contributor also has not provided after-fix real behavior proof.

## What This Changes

The PR adds fire-and-forget cleanup after `cron.remove` to delete matching cron session-store entries and the per-job run-log file, plus focused cleanup tests and small unrelated formatting/test cleanup.

## Best Possible Solution

Repair or replace the branch with a narrow cron removal lifecycle fix that preserves removed-event snapshots, cleans deleted-job base and run session rows after active runs settle, drains pending run-log writes before deletion, and includes focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or seed a cron session and run log, call `cron.remove`, and current `main` only removes the cron job record while session-store rows and run logs are separate artifacts.

## Solution Assessment

No. The cleanup direction is right, but the proposed branch is not the best merge path until it preserves hook payloads and coordinates cleanup with active runs and pending run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the removed job snapshot:** `src/cron/service/ops.ts:344`
  - body: Current main and the documented `cron_changed` contract carry the deleted job snapshot on removed events. This PR emits `removed` without `job`, so hook consumers lose the payload they need for scheduler reconciliation.
  - confidence: 0.91
- **[P2] Re-clean sessions after active runs settle:** `src/cron/service/ops.ts:348`
  - body: The cleanup runs once immediately after removal. If the deleted job is already running, the isolated run can persist the canonical cron session after this cleanup, and the existing reaper keeps base cron sessions, so the visible ghost session can return.
  - confidence: 0.87
- **[P2] Drain pending run-log writes before unlinking:** `src/cron/service/ops.ts:406`
  - body: `appendCronRunLog` queues per-path writes and creates the file, but `cleanupRunLog` unlinks without waiting for that queue. A pending finished-event append can recreate the removed job's run log after cleanup.
  - confidence: 0.84
