## Summary

Keep open: the linked bug is still valid on current main, but this PR is not safe to merge because it regresses the removed-event hook payload, leaves in-flight cleanup races, and lacks real behavior proof.

## What This Changes

The PR adds fire-and-forget cleanup of cron session-store entries and per-job run logs after `cron.remove`, plus focused cleanup tests and small unrelated test/formatting edits.

## Best Possible Solution

Repair or replace this with a narrow cron removal lifecycle fix that preserves removed-event job snapshots, removes base and run session keys after active runs settle, drains pending run-log writes before unlinking, and keeps focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main removes only the cron job record while cron sessions and run logs are written through separate paths. I did not run the Control UI flow because this was a read-only review.

## Solution Assessment

No, not as-is. The cleanup direction is right, but the branch must preserve the removed-event job snapshot and coordinate cleanup with active runs and queued run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the removed job snapshot:** `src/cron/service/ops.ts:344`
  - body: Current main emits the deleted job snapshot, and the documented `cron_changed` contract says removed events still carry it for external scheduler reconciliation. This branch emits `removed` without `job`, so hook consumers lose the deletion payload they currently rely on.
  - confidence: 0.91
- **[P2] Clean up again after active runs settle:** `src/cron/service/ops.ts:348`
  - body: The cleanup runs once immediately after removal. If the deleted job is already running, isolated cron completion can persist the canonical `agent:<agent>:cron:<jobId>` key afterward, and the existing reaper keeps that base key, so the visible ghost session can return.
  - confidence: 0.87
- **[P2] Drain run-log writes before unlinking:** `src/cron/service/ops.ts:406`
  - body: `cleanupRunLog` unlinks the JSONL file without waiting for `appendCronRunLog`'s per-path queue. A pending or later finished-event append can recreate the removed job's run log after this unlink, leaving the stale artifact behind.
  - confidence: 0.84
