## Summary

Keep open. The cron cleanup bug still exists on current main, but this PR is not mergeable as-is because it regresses removed-event hook payloads and its one-shot cleanup can still lose races with active cron runs or queued run-log writes.

## What This Changes

The PR adds fire-and-forget `cron.remove` cleanup for matching session-store entries and per-job run-log files, plus focused cleanup tests and small unrelated test/formatting changes.

## Best Possible Solution

Revise the source work so cron artifact deletion is final: preserve current removed-event metadata, clean the correct per-agent session store, and coordinate cleanup with active run completion plus pending run-log writes.

## Reproduction Assessment

Yes. Create a cron job, let it create a session/run log, call `cron.remove`, and current main only removes the job config; the PR-specific races are reproducible from code paths where an active run or pending log append writes after the immediate cleanup.

## Solution Assessment

No, not as-is. The narrow cleanup direction is right, but it should be integrated with the finished-run/run-log write paths or repeated after active jobs settle while preserving the current `cron_changed` removed-event contract.

- **[P2] Keep the removed job in cron events:** `src/cron/service/ops.ts:349`
  - body: Current main includes the deleted job snapshot on `removed` events so `cron_changed` hooks can receive `evt.job`, and `server-cron.test.ts` asserts that contract. This branch emits `removed` without `job`, so merging it would regress plugin hook payloads for job deletion.
  - confidence: 0.91
- **[P2] Re-run cleanup after active cron jobs finish:** `src/cron/service/ops.ts:353-355`
  - body: The cleanup runs once immediately after `cron.remove`. If the job was already executing, isolated cron completion can persist the canonical `agent:<agent>:cron:<jobId>` key after this point, and the existing reaper preserves that base key, so the ghost session can come back.
  - confidence: 0.87
- **[P2] Drain pending run-log writes before unlinking:** `src/cron/service/ops.ts:416`
  - body: `cleanupRunLog` unlinks the run-log file without coordinating with `appendCronRunLog`'s queued writes. Finished events append through a fire-and-forget promise, so a pending or later append can recreate `{runs}/{jobId}.jsonl` after deletion.
  - confidence: 0.84
