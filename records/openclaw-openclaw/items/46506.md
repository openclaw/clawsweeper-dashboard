## Summary

Keep open: current main still leaves cron sessions/run logs behind on job deletion, and this PR is useful source work, but the patch still has concrete merge blockers around removed-event payloads and cleanup races.

## What This Changes

The PR adds asynchronous `cron.remove` cleanup for cron session-store entries and per-job run logs, adds cleanup tests, and includes small unrelated test/formatting edits.

## Best Possible Solution

Revise the cron removal path so artifact cleanup is final after active runs and pending log writes settle, while preserving removed-event job snapshots and adding regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Current main source shows `cron.remove` removes only the job config, while session persistence and fire-and-forget run-log writes can recreate artifacts after a one-time immediate cleanup.

## Solution Assessment

No, not as-is. The cleanup direction is narrow and maintainable, but it needs to preserve the removed-event contract and coordinate with active cron completion plus queued run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep the removed job in cron events:** `src/cron/service/ops.ts:349`
  - body: Current main emits the deleted job snapshot on removed events, and `cron_changed` hooks/tests rely on `evt.job` for deletion reconciliation. This branch emits `removed` without `job`, so merging it would regress plugin hook payloads for job deletion.
  - confidence: 0.91
- **[P2] Re-run cleanup after active cron jobs finish:** `src/cron/service/ops.ts:353-355`
  - body: The cleanup runs once immediately after removal. If the job is already executing, isolated cron completion can persist the canonical cron session key afterward, and the existing reaper keeps that base key, so the visible ghost session can come back.
  - confidence: 0.87
- **[P2] Drain pending run-log writes before unlinking:** `src/cron/service/ops.ts:416`
  - body: `cleanupRunLog` unlinks the file without coordinating with queued `appendCronRunLog` writes. Since finished events append fire-and-forget, a pending or later append can recreate the deleted job's run-log file after cleanup.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cron/service/ops.ts:353-355`
  - body: This is a user-facing fix for ghost cron sessions, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing fixes to include a changelog entry before landing.
  - confidence: 0.86
