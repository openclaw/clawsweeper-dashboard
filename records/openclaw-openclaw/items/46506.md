## Summary

Keep open. The linked bug is still valid on current main, but this PR is not mergeable as-is because it drops the removed-event job snapshot, leaves active-run and queued run-log races, and still lacks real behavior proof.

## What This Changes

The PR adds asynchronous cleanup of cron session-store entries and per-job run-log files after `cron.remove`, plus focused cleanup tests and small unrelated test/formatting edits.

## Best Possible Solution

Repair or replace this with a narrow cron removal lifecycle fix that preserves removed-event job snapshots, removes deleted-job base and run session keys after active runs settle, drains pending run-log writes before unlinking, and verifies the real UI or CLI flow.

## Reproduction Assessment

Yes, from source inspection: current main removes only the cron job record while cron sessions and run logs are written through separate paths. I did not run the Control UI flow because this was a read-only review.

## Solution Assessment

No, not as-is. The cleanup direction is correct, but the branch needs to preserve the removed-event snapshot and coordinate cleanup with active runs and queued run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the removed job snapshot:** `src/cron/service/ops.ts:344`
  - body: Current main emits the deleted job snapshot, and the documented `cron_changed` contract says removed events still carry it for external scheduler reconciliation. This branch emits `removed` without `job`, so hook consumers lose the deletion payload they currently rely on.
  - confidence: 0.91
- **[P2] Re-clean removed cron sessions after active runs settle:** `src/cron/service/ops.ts:348-350`
  - body: The cleanup runs once immediately after removal. If the deleted job is already running, isolated cron completion can persist the canonical `agent:<agent>:cron:<jobId>` key afterward, and the existing reaper keeps that base key, so the visible ghost session can return.
  - confidence: 0.87
- **[P2] Wait for pending cron run-log writes:** `src/cron/service/ops.ts:406-410`
  - body: `cleanupRunLog` unlinks the JSONL file without waiting for `appendCronRunLog`'s per-path queue. A pending or later finished-event append can recreate the removed job's run log after this unlink, leaving the stale artifact behind.
  - confidence: 0.84
