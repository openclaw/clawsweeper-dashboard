## Summary

Keep open: the linked cron cleanup bug is still source-reproducible on current main, and this PR is useful but not mergeable as-is because it drops the removed-event job snapshot, races active runs and queued run-log writes, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds fire-and-forget cleanup after `cron.remove` to delete matching session-store entries and the per-job run-log file, plus focused cleanup tests and small unrelated test/formatting edits.

## Best Possible Solution

Repair or replace the PR with a narrow cron removal lifecycle fix that preserves removed-event snapshots, cleans base and run session keys after active runs settle, drains run-log writes before unlinking, and includes focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, from source inspection: current main’s `cron.remove` removes only the job record, while cron sessions and run logs are persisted through separate paths. I did not run the Control UI flow in this read-only review.

## Solution Assessment

No. The cleanup direction is right, but this branch regresses removed-event hook payloads and does not coordinate with active runs or queued run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the removed job snapshot:** `src/cron/service/ops.ts:344`
  - body: Current main emits the deleted job snapshot, and the documented `cron_changed` contract says removed events still carry it for external scheduler reconciliation. This branch emits `removed` without `job`, so hook consumers lose the deletion payload they currently rely on.
  - confidence: 0.91
- **[P2] Re-clean sessions after active runs settle:** `src/cron/service/ops.ts:348-350`
  - body: The cleanup runs once immediately after removal. If the deleted job is already running, isolated completion can persist the stable cron session key afterward, and the existing reaper keeps base cron sessions, so the visible ghost session can return.
  - confidence: 0.87
- **[P2] Drain pending run-log writes before unlinking:** `src/cron/service/ops.ts:406-410`
  - body: `cleanupRunLog` unlinks the JSONL file without waiting for `appendCronRunLog`'s per-path queue. A pending or later finished-event append can recreate the removed job's run log after this unlink, leaving the stale artifact behind.
  - confidence: 0.84
