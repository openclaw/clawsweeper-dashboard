## Summary

Keep open: the ghost-session bug is real and this PR is useful, but the current head would regress removed-event hook payloads and can still leave artifacts after in-flight cron runs or queued run-log writes.

## What This Changes

The PR adds asynchronous `cron.remove` cleanup for cron session-store entries and per-job run logs, with cleanup tests and small unrelated test/formatting edits.

## Best Possible Solution

Revise the cron removal path so artifact cleanup is final after active runs and pending log writes settle, while preserving removed-event job snapshots and adding regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current `cron.remove` removes only the job config, and the PR's one-time cleanup can be defeated by active-run session persistence and queued run-log appends.

## Solution Assessment

No, not as-is. The cleanup direction is narrow, but the implementation needs to preserve the removed-event contract and coordinate cleanup with active cron completion plus pending log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep the removed job in cron events:** `src/cron/service/ops.ts:344`
  - body: This changes the removed event from the current deleted-job snapshot contract to omit `job`. Docs and tests say removed `cron_changed` events carry the deleted job so external schedulers can reconcile deletions, so plugin hooks would lose required context.
  - confidence: 0.91
- **[P2] Re-run cleanup after active cron jobs finish:** `src/cron/service/ops.ts:348`
  - body: The cleanup runs only once at remove time. If the job is already executing, isolated cron completion can persist the canonical `agent:<id>:cron:<jobId>` key afterward, and the existing reaper keeps that base key, so the visible ghost session can return.
  - confidence: 0.87
- **[P2] Drain pending run-log writes before unlinking:** `src/cron/service/ops.ts:406`
  - body: `cleanupRunLog` unlinks the file without coordinating with fire-and-forget `appendCronRunLog` writes. A pending or later finished-event append can recreate the deleted job's log file after cleanup.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cron/service/ops.ts:348`
  - body: This is a user-facing fix for ghost cron sessions, but the branch does not update `CHANGELOG.md`. Repo policy requires user-facing fixes to be recorded before landing.
  - confidence: 0.86
