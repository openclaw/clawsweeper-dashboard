## Summary

Keep open: current main still lacks cron.remove artifact cleanup, but this PR is not merge-ready because it regresses the removed-event snapshot contract, leaves active-run and run-log races, and has no real after-fix behavior proof.

## What This Changes

The PR adds asynchronous cron.remove cleanup for cron session-store entries and per-job run-log files, plus cleanup tests and small unrelated test/formatting edits.

## Best Possible Solution

Update the cron removal path so cleanup is final after active runs and queued run-log writes settle, while preserving removed-event job snapshots and adding focused regression coverage.

## Reproduction Assessment

Yes. Source inspection at the provided current-main SHA shows cron.remove only deletes the job config, while session persistence and fire-and-forget run-log append paths can leave or recreate the reported artifacts.

## Solution Assessment

No, not as-is. The cleanup direction is narrow, but this implementation drops a documented event payload and performs one-shot cleanup without coordinating with active runs or pending run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep the removed job snapshot in events:** `src/cron/service/ops.ts:344`
  - body: Current main emits the deleted job snapshot on removed events, and the public cron_changed contract says removed events carry that snapshot for external scheduler reconciliation. This branch emits removed without job, so hook consumers lose the deletion payload they currently rely on.
  - confidence: 0.91
- **[P2] Make cleanup final after active runs finish:** `src/cron/service/ops.ts:348`
  - body: The cleanup is launched once immediately after removal. If the deleted job is already running, isolated cron completion can persist the canonical cron session key afterward, and the existing reaper keeps that base key, so the visible ghost session can come back.
  - confidence: 0.87
- **[P2] Drain queued run-log writes before unlinking:** `src/cron/service/ops.ts:406`
  - body: cleanupRunLog unlinks the file without coordinating with appendCronRunLog's per-path queue. A pending or later finished-event append can recreate the removed job's JSONL file after this unlink, leaving the stale run-log artifact behind.
  - confidence: 0.84
