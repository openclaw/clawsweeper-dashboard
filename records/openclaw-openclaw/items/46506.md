## Summary

Keep open. The PR targets a real cron cleanup bug that still exists on current main, but the branch is stale against current cron event behavior and its one-shot cleanup can still leave sessions or run logs behind when removal races with an active or just-finished run.

## What This Changes

The PR adds asynchronous `cron.remove` cleanup for cron session-store entries and per-job run log files, adds focused cleanup tests, and includes small unrelated test/formatting cleanups.

## Best Possible Solution

Keep the source work, but revise it so cron artifact deletion is final: preserve current removed-event metadata, clean the correct per-agent session store, and coordinate cleanup with active run completion plus pending run-log writes before landing it for #46369.

## Reproduction Assessment

Yes. The original bug has a direct source-backed reproduction path: create a cron job, let it create a cron session/run log, call `cron.remove`, and current main only removes the job config. The remaining PR issues are also reproducible from code paths where an already-running cron job or queued run-log append writes artifacts after the PR's immediate cleanup.

## Solution Assessment

No, not as-is. The narrow fix direction is right, but the cleanup should be integrated with the finished-run/run-log write path or repeated after active jobs settle, while preserving current cron_changed removed-event metadata.

Full review comments:

- **[P2] Preserve the removed job in cron events:** `src/cron/service/ops.ts:349`
  - body: Current main includes the deleted job snapshot on `removed` events so `cron_changed` hooks can receive `evt.job`, and `server-cron.test.ts` asserts that contract. This branch emits `removed` without `job`, so merging it would regress plugin hook payloads for job deletion. Please keep `job: removedJob` when adding the cleanup call.
  - confidence: 0.91
- **[P2] Re-run cleanup after active cron jobs finish:** `src/cron/service/ops.ts:353-355`
  - body: The cleanup runs once immediately after `cron.remove`. If the job was already executing, the isolated runner can persist `agent:<agent>:cron:<jobId>` after this point, and the existing reaper preserves canonical cron keys, so the ghost session can come back. Please defer or repeat artifact cleanup from the finished-run path when the job was removed while active.
  - confidence: 0.87
- **[P2] Drain pending run-log writes before deleting the log:** `src/cron/service/ops.ts:416`
  - body: `cleanupRunLog` unlinks the file without coordinating with `appendCronRunLog`'s queued writes. Finished events append through a fire-and-forget promise, so a pending or later append can recreate `{runs}/{jobId}.jsonl` after deletion. Please add a run-log delete helper that drains pending writes, or otherwise wait for the final append before unlinking.
  - confidence: 0.84
