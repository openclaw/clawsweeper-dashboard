## Summary

Keep this PR open, but it should not merge as-is: current main still has the linked cleanup bug, while this branch fixes the simple idle-delete case but leaves in-flight session and run-log races, lacks real behavior proof, and is stale against current cron event behavior.

## What This Changes

This PR adds fire-and-forget cleanup on `cron.remove` to delete matching cron session-store entries and the per-job run log, plus cron cleanup tests and small unrelated test/formatting changes.

## Best Possible Solution

Implement the cleanup on current main so confirmed job removal purges canonical and run cron sessions in the correct per-agent store, deletes run logs after pending writes settle, preserves removed-event job metadata, and covers idle plus in-flight cases with focused regression tests.

## Reproduction Assessment

Yes, from source inspection: current main `cron.remove` removes only the job record, while the existing reaper intentionally excludes base cron session keys. I did not run an end-to-end UI reproduction in this read-only review.

## Solution Assessment

No. The PR is the right general direction for idle deletion, but the one-shot fire-and-forget cleanup is not the narrowest complete fix because active runs and queued run-log appends can recreate the artifacts it deletes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve the removed job in cron events:** `src/cron/service/ops.ts:344`
  - body: Current main carries the removed job snapshot on the `removed` event so gateway cron hooks can still populate `job`, `sessionTarget`, and `agentId` after `getJob()` starts returning undefined. This branch emits only `{ jobId, action }`, so rebasing the cleanup as-is would drop removed-job metadata for `cron_changed` hooks.
  - confidence: 0.86
- **[P2] Coordinate cleanup with in-flight cron runs:** `src/cron/service/ops.ts:348`
  - body: `cron.remove` starts cleanup immediately, but an isolated run that was already active can later persist `agent:<agent>:cron:<jobId>` again. The existing session reaper skips base cron keys, so deleting a running job can still leave the visible ghost session indefinitely; cleanup needs to run after the active run settles or suppress late persistence for removed jobs.
  - confidence: 0.88
- **[P2] Drain pending run-log writes before unlinking:** `src/cron/service/ops.ts:406`
  - body: `cleanupRunLog` unlinks the JSONL file immediately, but finished events append through the queued fire-and-forget `appendCronRunLog` path. If an append is already pending, it can recreate the file after this unlink, leaving the deleted job's run log behind.
  - confidence: 0.85
