## Summary

Keep open. Current main still lacks the cron artifact cleanup, and this PR is a useful but currently incorrect fix: it lacks real behavior proof, drops the removed-event job snapshot, and can leave artifacts behind around active runs or queued run-log writes.

## What This Changes

The PR adds asynchronous cleanup of cron session-store entries and per-job run logs after `cron.remove`, plus focused cleanup tests and small unrelated test/formatting edits.

## Best Possible Solution

Repair or replace this PR with a cron removal lifecycle fix that preserves removed-event job snapshots, removes base and run session keys after active runs settle, drains pending run-log writes before unlinking, and keeps focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main removes only the cron job record while cron sessions and run logs are maintained through separate paths. I did not run the UI flow because this was a read-only review.

## Solution Assessment

No, not as-is. The cleanup direction is right, but the branch regresses the removed-event payload and performs one-shot cleanup without coordinating with active runs or queued run-log writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep the removed job snapshot in events:** `src/cron/service/ops.ts:344`
  - body: Current main emits the deleted job snapshot on removed events, and the public `cron_changed` contract says removed events carry that snapshot for external scheduler reconciliation. This branch emits `removed` without `job`, so hook consumers lose the deletion payload they currently rely on.
  - confidence: 0.91
- **[P2] Make cleanup final after active runs finish:** `src/cron/service/ops.ts:348`
  - body: The cleanup is launched once immediately after removal. If the deleted job is already running, isolated cron completion can persist the canonical cron session key afterward, and the existing reaper keeps that base key, so the visible ghost session can come back.
  - confidence: 0.87
- **[P2] Drain queued run-log writes before unlinking:** `src/cron/service/ops.ts:406`
  - body: `cleanupRunLog` unlinks the file without coordinating with `appendCronRunLog`'s per-path queue. A pending or later finished-event append can recreate the removed job's JSONL file after this unlink, leaving the stale run-log artifact behind.
  - confidence: 0.84
