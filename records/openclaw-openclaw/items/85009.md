## What This Changes

The issue reports Anthropic gateway cron runs failing because a user-defined OpenClaw tool reaches the Anthropic Messages API as `tools[].type: "custom"`.

## Best Possible Solution

Land or rework the linked provider-boundary fix so Anthropic Messages runs always use an Anthropic-compatible tool payload shape without changing OpenClaw's internal tool policy.

## Reproduction Assessment

Yes, at source level: current main can still return a custom Anthropic session stream instead of the boundary-aware transport when there is no pre-resolved API key, while the Anthropic transport is the path that emits `input_schema` tool payloads. I did not run a live Anthropic cron job in this read-only review.

## Solution Assessment

Yes: the best fix belongs at the embedded-run/provider transport boundary so Anthropic requests use an accepted wire schema while OpenClaw's internal tool policy remains intact. The linked PR appears to target that boundary directly, but the issue should remain open until that fix lands or maintainers choose another implementation.
