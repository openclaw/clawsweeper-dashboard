## Summary

Keep open. Current main preserves the reported `user:ou_...` Feishu target routing, but it still loses useful Feishu/Lark response details when the SDK throws HTTP 400 errors, so the user-visible failure remains actionable as a plugin diagnostic bug rather than a completed core routing fix.

## What This Changes

The issue requests fixing Feishu direct-message sends through the shared `message` tool for `user:ou_...` targets that return HTTP 400 while group sends work.

## Best Possible Solution

Keep the existing Feishu target contract, add plugin-local normalization for Lark/Axios API failures so message send and `feishu_chat` preserve `code`, `msg`, `log_id`, and troubleshooter details, then use those diagnostics to decide whether the remaining `9499` delivery failure is a code bug or Feishu app/tenant setup guidance.

## Reproduction Assessment

Partially yes. The live DM delivery failure is not high-confidence reproducible here because no Feishu tenant credentials were available, but the diagnostic gap is reproducible with mocks by making `client.im.message.create` or `client.contact.user.get` reject with an Axios-shaped HTTP 400 containing `response.data`.

## Solution Assessment

Yes for the next repair: plugin-local Feishu/Lark error normalization is the narrowest maintainable fix because current main already maps `user:ou_...` to `open_id`. A core routing change or a claim that `9499` delivery is fixed would be premature without tenant-backed proof.
