## Summary

Keep open. Current main already supports the reported Feishu `user:ou_...` target form, but the message-send and `feishu_chat` member lookup paths still allow Lark/Axios HTTP 400 rejections to surface through generic error text, so the reporter's Feishu `code`, `msg`, `log_id`, and troubleshooter details can still be hidden.

## What This Changes

The issue requests that Feishu direct user sends through the shared message tool stop returning opaque HTTP 400 errors and expose actionable Feishu diagnostics for user-target failures.

## Best Possible Solution

Keep the issue open and add a Feishu-plugin error normalizer around message create/reply/media sends and `feishu_chat` member lookups so thrown Lark/Axios responses preserve Feishu `code`, `msg`, HTTP status, top-level or nested `log_id`, and troubleshooter URL while leaving the existing `user:ou_...` routing contract unchanged.

## Reproduction Assessment

Partially yes. The exact live DM `9499` failure needs Feishu tenant credentials, but the diagnostic bug has a high-confidence unit reproduction by mocking `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped HTTP 400 containing `response.data`.

## Solution Assessment

Yes for the next repair. A plugin-local Feishu API error normalizer is the narrowest maintainable fix because current main already maps the reported user-open-id target correctly; claiming the vendor `9499` delivery failure is fixed should wait for live Feishu proof.
