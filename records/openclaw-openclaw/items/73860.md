## Summary

Keep open. Current main still supports the reported `user:ou_...` target form, but the Feishu direct-send and `feishu_chat` member lookup paths still let Lark/Axios HTTP 400 rejections surface as generic errors instead of preserving Feishu `code`, `msg`, and `log_id`.

## What This Changes

Request to fix Feishu message-tool direct sends to `user:ou_...` targets where HTTP 400 errors lose vendor diagnostics while group sends work.

## Best Possible Solution

Keep target routing unchanged and add plugin-local Feishu/Lark Axios error normalization for message send, media send, and `feishu_chat` lookup paths so users get vendor diagnostics before pursuing any tenant-permission fix.

## Reproduction Assessment

Partially yes. Live DM delivery needs Feishu tenant credentials, but the diagnostic bug is reproducible with a focused unit test that mocks `client.im.message.create` or `client.contact.user.get` rejecting with an Axios-shaped HTTP 400 containing `response.data.code`, `msg`, and nested `error.log_id`.

## Solution Assessment

Yes for the next repair. Reusing the existing Feishu formatter locally is the narrowest maintainable fix because current main already maps `user:ou_...` to `open_id`, and changing core routing would risk breaking working group/direct behavior.
