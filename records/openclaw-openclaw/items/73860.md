## Summary

Keep open. Current main preserves the reported `user:ou_...` target shape well enough to send as Feishu `open_id`, but the remaining Feishu HTTP 400 path still loses the vendor error body in both message send and `feishu_chat`, so the report is still actionable.

## What This Changes

The issue asks for Feishu direct-message sends through the `message` tool to work for `user:ou_...` targets instead of returning generic HTTP 400 failures.

## Best Possible Solution

Keep Feishu `user:ou_...` sends routed as `open_id`, add Feishu-plugin error normalization for thrown Lark/Axios API failures, and then use the preserved Feishu code/log/troubleshooter details to decide whether the remaining 9499 delivery failure is setup guidance or a separate code fix.

## Reproduction Assessment

Unclear for the live DM delivery failure without Feishu tenant credentials. Yes for the confirmed diagnostic bug: mock `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped 400 containing `response.data`, and current code reports only the generic error message.

## Solution Assessment

Yes for the next narrow repair: plugin-local Feishu error normalization is the safest maintainable path because target parsing already maps `user:ou_...` to `open_id`. A core routing change or claim that 9499 is fixed would be premature without tenant-backed evidence.
