## Summary

Keep open. Current `main` still routes Feishu `user:ou_...` targets as `open_id`, so this does not look like a core target-parsing regression, but the reported 400 diagnostic gap remains in the Feishu plugin: thrown Lark/Axios responses from message send and `feishu_chat` member lookup paths still lose Feishu `code`, `msg`, and `log_id` details behind generic error text.

## What This Changes

The issue asks OpenClaw to fix Feishu message-tool direct sends to `user:ou_...` targets returning generic HTTP 400 errors and to preserve useful Feishu API diagnostics.

## Best Possible Solution

Keep this issue open and add a Feishu-plugin error normalizer used by message send, media send if affected, and `feishu_chat` so thrown Lark/Axios responses preserve Feishu `code`, `msg`, nested `error.log_id`, `troubleshooter`, and HTTP status while leaving the existing `user:ou_...` routing contract unchanged.

## Reproduction Assessment

Partially yes. The live DM delivery failure needs Feishu tenant credentials, but the diagnostic bug has a high-confidence unit reproduction by mocking `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped HTTP 400 containing `response.data`.

## Solution Assessment

Yes for the next repair. A plugin-local Feishu API error normalizer is the narrowest maintainable fix because current `main` already maps the reported user open_id target correctly; a core routing change or a claim that Feishu `9499` delivery is fixed would be premature without live tenant proof.
