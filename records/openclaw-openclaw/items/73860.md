## Summary

Keep open. Current main already maps `user:ou_...` targets correctly, but the Feishu plugin still lets rejected Lark/Axios HTTP 400s from direct sends and member lookups collapse to generic text, leaving the reported vendor code/log_id diagnostics unavailable.

## What This Changes

The issue reports Feishu `message` tool direct sends to `user:ou_<open_id>` failing with HTTP 400 while group sends work.

## Best Possible Solution

Keep the existing target-routing behavior and add Feishu plugin-level normalization for rejected Lark/Axios responses, then use the surfaced `log_id` and troubleshooter link to decide whether `code: 9499` is a separate delivery-permission bug.

## Reproduction Assessment

Partially yes. Live DM delivery requires Feishu tenant credentials, but the diagnostic failure is reproducible with a focused unit test that mocks `client.im.message.create` or `client.contact.user.get` rejecting with an Axios-shaped HTTP 400 containing Feishu `response.data`.

## Solution Assessment

Yes for the next repair. A plugin-local error normalizer is the narrowest maintainable path because current main already maps `user:ou_...` to `open_id`, and changing core routing would risk regressing working group/direct target behavior.
