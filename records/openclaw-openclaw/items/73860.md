## Summary

Keep open. Current main still supports `user:ou_...` Feishu routing, but the reported diagnostic failure remains: Lark/Axios HTTP 400 responses thrown from Feishu send and member lookup paths are not unwrapped, so Feishu `code`, `msg`, `log_id`, and troubleshooter details can be lost behind generic error text.

## What This Changes

The issue requests fixing Feishu direct-message sends through the shared `message` tool for `user:ou_<open_id>` targets and preserving useful Feishu API error details when failures occur.

## Best Possible Solution

Keep the issue open and add a Feishu-plugin error normalizer used by message send and `feishu_chat` so thrown Lark/Axios responses preserve Feishu `code`, `msg`, `log_id`, and troubleshooter details while leaving the existing `user:ou_...` routing contract unchanged.

## Reproduction Assessment

Partially yes. The live DM delivery failure needs Feishu tenant credentials, but the diagnostic gap is reproducible with a unit mock that makes `client.im.message.create` or `client.contact.user.get` reject with an Axios-shaped HTTP 400 containing `response.data`.

## Solution Assessment

Yes for the next repair: plugin-local Feishu error normalization is the narrowest maintainable step because current main already maps the reported user open_id target correctly. A core routing change or a claim that Feishu 9499 delivery is fixed would be premature without live tenant proof.
