## Summary

Keep open. Current main still supports the reported `user:ou_...` target form, but Feishu direct sends and `feishu_chat` member lookups still let Lark/Axios HTTP 400 rejections surface through generic error text instead of preserving the Feishu `code`, `msg`, `log_id`, and troubleshooter details the reporter captured.

## What This Changes

The issue requests reliable Feishu message-tool direct sends to `user:ou_...` targets, or at least actionable Feishu API error details when those sends fail with HTTP 400.

## Best Possible Solution

Add a Feishu-plugin error normalizer to the message send/media/chat paths so rejected Lark/Axios responses preserve vendor diagnostics, then use the exposed Feishu `log_id` to decide whether a separate delivery-permission fix is needed.

## Reproduction Assessment

Partially yes. The live DM delivery failure needs Feishu tenant credentials, but the diagnostic failure is reproducible with a focused unit test that mocks `client.im.message.create` or `client.contact.user.get` rejecting with an Axios-shaped HTTP 400 containing `response.data`.

## Solution Assessment

Yes for the next repair. A plugin-local error normalizer is the narrowest maintainable path because current main already maps `user:ou_...` to `open_id`, and changing core routing would risk breaking working group/direct target behavior.
