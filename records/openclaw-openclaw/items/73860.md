## Summary

Keep open. Current main preserves the obvious `user:ou_...` Feishu target routing, but the reported diagnostic failure is still present: Lark/Axios HTTP 400 responses can bypass Feishu response normalization and collapse to generic error text, while the live `code:9499` delivery root cause remains unproven.

## What This Changes

The issue requests a fix for Feishu direct-message sends through the shared `message` tool when targeting `user:ou_<open_id>`, plus better handling of related Feishu API 400 failures.

## Best Possible Solution

Keep the `user:ou_...` target contract, add Feishu-plugin error normalization for Lark/Axios API failures in message send and `feishu_chat`, and then use preserved `code`, `msg`, `log_id`, and troubleshooter details to decide whether the remaining 9499 delivery failure needs code changes or setup guidance.

## Reproduction Assessment

Unclear for the live DM delivery failure because this review did not have Feishu tenant credentials. Yes for the confirmed diagnostic gap: mock `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped 400 containing `response.data`, and current main reports only the generic error message.

## Solution Assessment

Yes for the next narrow repair: plugin-local Feishu error normalization is the maintainable first step because current target parsing already maps `user:ou_...` to `open_id`. A core routing change or a claim that 9499 is fixed would be premature without tenant-backed proof.
