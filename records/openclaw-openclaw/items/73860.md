## Summary

Keep open. Current main still supports the reported Feishu `user:ou_...` target form, but Feishu message sends and `feishu_chat` member lookups can still surface Lark/Axios HTTP 400 failures as generic error text instead of preserving the Feishu `code`, `msg`, `log_id`, and troubleshooter details captured by the reporter.

## What This Changes

Request to fix Feishu message-tool direct sends to `user:ou_...` targets returning HTTP 400 and preserve Feishu API diagnostics for message/member lookup failures.

## Best Possible Solution

Add plugin-local Feishu API error normalization for direct text/card/media sends and `feishu_chat` lookups, then use the exposed code/log id to decide whether a separate Feishu permission or targeting fix is needed.

## Reproduction Assessment

Partially yes. The diagnostic failure can be reproduced with a focused unit test that mocks `client.im.message.create` or `client.contact.user.get` rejecting with an Axios-shaped HTTP 400 containing `response.data`; full live DM delivery requires a Feishu tenant and would send real messages.

## Solution Assessment

Yes for the next repair. A Feishu-plugin-local normalizer is the narrowest maintainable fix because current main already maps `user:ou_...` to `open_id`, and changing core routing would risk breaking working group/direct targets.
