## Summary

Keep this open: current main and the latest release still lack Telegram-specific cron announce `To` validation, and two open PRs already claim this issue as their closing work.

## What This Changes

The issue asks the cron dashboard to validate and suggest Telegram announce targets so invalid `To` values are not saved and later fail delivery.

## Best Possible Solution

Land one Control UI fix that mirrors Telegram runtime target parsing, adds clear Telegram-specific helper/error UI, and only adds configured-group suggestions if maintainers accept that UX scope.

## Reproduction Assessment

Yes, source-level: current main validates only webhook `deliveryTo`, then forwards Telegram announce `delivery.to` to runtime where an unresolvable target can throw the reported recipient error. I did not run a live dashboard or Telegram cron in this read-only review.

## Solution Assessment

Yes for save-time validation as the core fix, but the final implementation should preserve the Telegram runtime target contract. Rich configured-group suggestions are useful but need maintainer UX scope confirmation.
