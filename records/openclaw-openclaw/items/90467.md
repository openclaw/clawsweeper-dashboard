## Summary

Keep this open: current main and the latest release still allow the Control UI to save arbitrary Telegram announce `To` values, and two open PRs already claim this issue as their closing work.

## What This Changes

The issue requests safer Control UI validation and suggestions for Telegram cron announce delivery targets so invalid values fail before runtime delivery.

## Best Possible Solution

Land one linked Control UI fix that validates Telegram announce `To` against the runtime target contract, adds clear inline/helper copy, and closes this issue after merge.

## Reproduction Assessment

Yes, source-level: current main validates only webhook `deliveryTo`, then forwards Telegram announce `delivery.to` to runtime where an unresolvable nonnumeric target can throw the reported error. I did not run a live dashboard or Telegram cron in this read-only review.

## Solution Assessment

Yes for the core bug: save-time validation and visible helper/error text in the Control UI are the right layer. The best implementation should mirror the Telegram plugin's target contract and leave any richer configured-group suggestions to the same PR only if maintainers accept the UX scope.
