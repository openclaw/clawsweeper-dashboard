## Summary

Keep this open: current main and the latest release still accept arbitrary Telegram cron announce `To` values from the dashboard, and the remaining work is already owned by an open closing PR that needs maintainer review rather than a duplicate fix job.

## What This Changes

The issue requests Control UI validation and safer suggestions for Telegram cron announce `To` targets so unsendable values are not saved and configured groups/topics are easier to choose.

## Best Possible Solution

Land one narrow fix, likely through the open closing PR after repair, that mirrors the Telegram plugin target contract, displays Telegram-specific helper/error copy, and avoids suggesting agent names unless they resolve to a deliverable Telegram target.

## Reproduction Assessment

Yes at source level: current main validates `deliveryTo` only for webhook delivery, forwards announce `delivery.to` raw, and Telegram delivery rejects unresolved nonnumeric targets. I did not run a live dashboard/Telegram cron flow in this read-only review.

## Solution Assessment

Unclear as a final implementation: save-time Control UI validation is the right boundary, but maintainers still need the PR to preserve the plugin-owned Telegram target contract and decide whether configured-group suggestions are in scope.
