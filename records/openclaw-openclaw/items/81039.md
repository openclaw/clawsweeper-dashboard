## Summary

Keep open: current main supports the reported CLI trajectory gap and the recorder integration is the right direction, but the latest PR head still skips trajectory flushing when cleanup throws and the supplied proof is mock-only.

## What This Changes

The PR wires `createTrajectoryRuntimeRecorder` into `executePreparedCliRun` to emit CLI lifecycle events and adds mocked lifecycle tests for prep-failure handling.

## Best Possible Solution

Land a narrow CLI-runner trajectory integration that protects recorder flushing across cleanup failures and is backed by redacted real pure `claude-cli` executor proof.

## Reproduction Assessment

Yes, source-reproducible: current main has no recorder calls in `executePreparedCliRun`, while the docs and embedded runner establish default runtime lifecycle events. I did not run a live `claude-cli` session in this read-only review.

## Solution Assessment

No, not yet. The recorder seam is the narrow maintainable fix, but the current `finally` ordering lets cleanup rejections bypass the new flush path, and the PR still needs real CLI executor proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Flush trajectory records after cleanup failures:** `src/agents/cli-runner/execute.ts:818-825`
  - body: The new `finally` awaits `claudeSkillsPlugin.cleanup()`, `systemPromptFile.cleanup()`, and `cleanupImages()` before reaching the trajectory `flush()`. If any cleanup rejects after events were queued, JavaScript exits the `finally` immediately and the new recorder never flushes, so a successful CLI run can still leave no sidecar or a half-written one. Move the recorder flush into a protected cleanup path that still runs when earlier cleanup fails.
  - confidence: 0.88
