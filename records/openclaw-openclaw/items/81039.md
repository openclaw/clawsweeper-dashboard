## Summary

Keep open: this is a valid narrow bug-fix direction, but the current PR head is still incorrect because cleanup rejections can prevent the trajectory recorder from flushing, and the supplied proof does not exercise the patched OpenClaw CLI executor path.

## What This Changes

The PR wires the runtime trajectory recorder into `executePreparedCliRun` and adds lifecycle tests for CLI-runner trajectory events.

## Best Possible Solution

Land the CLI-runner trajectory integration after moving recorder flushing into a protected cleanup path, adding cleanup-rejection coverage, and supplying redacted real pure `claude-cli` executor proof.

## Reproduction Assessment

Yes. Current-main source inspection shows the CLI executor has no recorder calls, while the docs and embedded runner establish default lifecycle events; the linked issue also includes a v2026.5.7 install audit.

## Solution Assessment

No, not yet. The recorder seam is the right narrow fix, but this PR still lets cleanup rejections bypass the new flush path and the supplied proof does not run the actual CLI executor flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Flush trajectory even when cleanup rejects:** `src/agents/cli-runner/execute.ts:818-825`
  - body: The new `finally` awaits `claudeSkillsPlugin.cleanup()`, `systemPromptFile.cleanup()`, and `cleanupImages()` before reaching `trajectoryRecorder.flush()`. If any cleanup rejects after events were queued, JavaScript exits the `finally` immediately and the sidecar never flushes, so successful or terminal events can still be lost. Move the recorder flush into a protected cleanup path that runs even when earlier cleanup fails.
  - confidence: 0.9
