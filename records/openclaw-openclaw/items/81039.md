Keep this PR open: it targets a valid source-reproducible CLI trajectory bug, but the latest head still has a blocking cleanup-order flaw and the supplied proof does not exercise the actual OpenClaw `claude-cli` executor path.

## What This Changes

The PR adds runtime trajectory recorder calls and lifecycle tests to `executePreparedCliRun` so pure CLI executor runs emit lifecycle sidecars.

## Best Possible Solution

Land the narrow CLI-runner trajectory integration after moving recorder flushing into a protected cleanup path, adding cleanup-rejection coverage, and supplying redacted real CLI executor proof.

## Reproduction Assessment

Yes, source-reproducible: current main has no recorder calls in `executePreparedCliRun`, while docs and the embedded runner establish default runtime lifecycle events. I did not run a live `claude-cli` session in this read-only review.

## Solution Assessment

No, not yet. The recorder seam is the right narrow fix, but the current PR still lets cleanup rejections bypass the new flush path and the proof does not run the actual CLI executor flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Flush trajectory even when cleanup rejects:** `src/agents/cli-runner/execute.ts:818-825`
  - body: The new `finally` still awaits `claudeSkillsPlugin.cleanup()`, `systemPromptFile.cleanup()`, and `cleanupImages()` before reaching `trajectoryRecorder.flush()`. If any cleanup rejects after events were queued, JavaScript exits the `finally` immediately and the sidecar never flushes, so successful or terminal events can still be lost. Move the recorder flush into a protected cleanup path that runs even when earlier cleanup fails.
  - confidence: 0.9
