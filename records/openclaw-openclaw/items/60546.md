## Summary

Keep open: current main still lets Microsoft Foundry discovery carry Claude deployments into an OpenAI-only provider route, and the linked fix PR at https://github.com/openclaw/openclaw/pull/72446 is still open and unmerged.

## What This Changes

The issue reports that built-in `microsoft-foundry` discovery can select Claude deployments and then persists/routes them through OpenAI-compatible Foundry URLs instead of Anthropic Messages endpoints.

## Best Possible Solution

Keep this issue open until the guardrail in https://github.com/openclaw/openclaw/pull/72446 or an equivalent fix lands; treat native Anthropic Foundry routing as a separate provider enhancement unless maintainers choose that larger path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a succeeded Claude deployment can flow from discovery into selection and persistence, then connection/runtime code resolves it to OpenAI-compatible Foundry routes rather than an Anthropic Messages route.

## Solution Assessment

Yes for the immediate bug boundary. Filtering or clearly warning on Anthropic deployments during `microsoft-foundry` onboarding is the narrowest maintainable fix; native Anthropic Foundry support would be a larger additive provider feature.
