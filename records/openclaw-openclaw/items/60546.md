## Summary

Keep open: current main still allows Microsoft Foundry discovery to select and persist Claude deployments while the provider only builds OpenAI-compatible Foundry routes, and the linked closing PR remains open and unmerged.

## What This Changes

The issue asks Microsoft Foundry onboarding to either support Azure Foundry Claude deployments correctly or stop offering them when only OpenAI-compatible routes are wired.

## Best Possible Solution

Land or replace the provider-local guardrail in https://github.com/openclaw/openclaw/pull/72446; keep first-class Azure Claude onboarding separate under https://github.com/openclaw/openclaw/pull/47181.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: succeeded Claude deployments can flow through discovery, selection, and persistence, then connection/runtime code resolves them to OpenAI-compatible Foundry routes instead of an Anthropic Messages route.

## Solution Assessment

Yes for the immediate bug boundary. Filtering or warning on unsupported Anthropic deployments during Microsoft Foundry onboarding is narrower than adding full native Anthropic Foundry support in this bug fix.
