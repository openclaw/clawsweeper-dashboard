## Summary

Keep open: current main still exposes and persists all succeeded Microsoft Foundry deployments while the built-in provider only has OpenAI-compatible routing, and the linked closing PR remains open and unmerged.

## What This Changes

The issue asks the built-in microsoft-foundry provider to stop offering Anthropic/Claude Foundry deployments unless it can route them through the Anthropic Messages API.

## Best Possible Solution

Keep this issue open until PR #72446 or an equivalent microsoft-foundry guardrail lands; native Anthropic Foundry routing should be a deliberate follow-up provider enhancement if maintainers want the built-in provider to own that API family.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: discovery can select any succeeded deployment, then config persistence, connection testing, and runtime auth resolve it to OpenAI-compatible Foundry routes rather than Anthropic Messages routes.

## Solution Assessment

Yes for the immediate bug boundary. Filtering or warning on Anthropic deployments during onboarding is the narrowest maintainable short-term fix; full native Anthropic Foundry support would need a larger provider API, auth header, routing, and test implementation.
