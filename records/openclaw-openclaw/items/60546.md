## Summary

Keep open: current main still lets the built-in Microsoft Foundry provider select and persist any succeeded deployment while the provider routes only OpenAI-compatible Foundry APIs, and an open closing PR already targets the guardrail.

## What This Changes

The issue asks the built-in microsoft-foundry provider to either support Anthropic Foundry routing or avoid selecting Anthropic deployments it cannot route correctly.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/72446 or an equivalent provider-local guardrail lands; treat full Azure Claude onboarding as separate feature work in https://github.com/openclaw/openclaw/pull/47181.

## Reproduction Assessment

Yes, with a high-confidence source-level path: discovered succeeded deployments can be selected and persisted, then the built-in provider resolves them through OpenAI-compatible API values and /openai/v1 routes. I did not run a live Azure Foundry account in this read-only review.

## Solution Assessment

Yes for the immediate bug boundary: filtering or warning on unsupported Anthropic deployments during Microsoft Foundry onboarding is the narrowest maintainable fix. Native Anthropic Foundry support is larger provider feature work and should remain separate.
