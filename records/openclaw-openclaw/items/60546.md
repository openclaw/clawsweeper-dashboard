## What This Changes

The issue reports that Microsoft Foundry onboarding can select Claude deployments even though the built-in provider routes only through OpenAI-compatible Foundry endpoints.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/72446 or an equivalent provider-local guardrail lands; treat full first-class Azure Claude onboarding as separate feature work tracked by https://github.com/openclaw/openclaw/pull/47181.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a succeeded Claude deployment can flow from Azure discovery into selection and persistence, while connection/runtime code resolves it to OpenAI-compatible Foundry routes instead of Anthropic Messages routes.

## Solution Assessment

Yes for the immediate bug boundary. Filtering or warning on unsupported Claude deployments during Microsoft Foundry onboarding/catalog persistence is the narrowest maintainable fix; native Anthropic Foundry support would be a larger additive provider feature.
