## Summary

Keep open. Current main still lets microsoft-foundry discovery surface all succeeded deployments while the provider API, base URL builder, runtime auth, and connection test remain OpenAI-compatible only; the linked guardrail PR #72446 is open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The request is for the built-in microsoft-foundry provider to either route Azure AI Foundry Claude deployments through the Anthropic Messages endpoint or stop offering them during onboarding until that support exists.

## Best Possible Solution

Keep this issue open while maintainers review and either land PR #72446 or merge an equivalent microsoft-foundry guardrail. Native Anthropic Foundry support should be handled as a deliberate provider enhancement if maintainers want the built-in provider to own that API family.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: discovery can select any succeeded deployment, then config persistence, runtime auth, and connection testing resolve that selection to OpenAI-compatible Foundry routes instead of Anthropic Messages routes.

## Solution Assessment

Yes for the immediate bug boundary. Filtering or warning on Anthropic deployments during onboarding is the narrowest maintainable short-term fix; full native Anthropic routing would require a larger runtime/auth/API implementation.
