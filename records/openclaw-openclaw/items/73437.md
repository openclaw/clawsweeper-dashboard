## Summary

Keep open. Current main still only provides loop-time `assemble()` for context engines that own compaction, while non-owning engines use the tool-result guard path and there is no public per-model-call full-message rewrite hook. The related `transform_context` PR is closed unmerged, so the remaining work is an API and trust-contract decision rather than cleanup.

## What This Changes

The issue requests a per-LLM-call context assembly or transform surface so context engines/plugins can inspect and rewrite the full message history during multi-call agent loops.

## Best Possible Solution

Define a stable, documented per-model-call context rewrite contract, preferably additive unless maintainers intentionally change `assemble` semantics. The contract should specify prompt-content access, returned-message validation, ordering relative to budget enforcement, and coverage for owning engines, non-owning engines, and multi-tool-call loops.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: inspect the setup-time `assembleAttemptContextEngine()` call, the later `activeSession.prompt()` calls, the `ownsCompaction` branch that gates loop-time assembly, and the absence of any public `transform_context` hook.

## Solution Assessment

Unclear. The additive hook direction from PR #57585 appears safer than globally changing `assemble` timing, but the API shape and trust model should be chosen by maintainers before implementation.

## Risks / Open Questions

- Hook ordering relative to Pi's private `transformContext`, OpenClaw budget enforcement, tool-result guards, and context-engine compaction is still a product/API choice.
