## Label Justifications

- `P2`: This is a normal-priority context-engine/plugin API feature gap with real downstream impact but no active core outage.
- `impact:session-state`: The request changes when session context messages are assembled and potentially rewritten during multi-call agent loops.
- `impact:security`: A per-call full-context mutation contract would grant plugins raw conversation access and model-input rewrite authority.

## Summary

Keep open: current main and v2026.6.8 still only provide a specialized per-iteration `assemble()` path for `ownsCompaction` engines, not the requested general per-LLM-call full-context assembly or mutation contract. This remains a security-sensitive plugin/context-engine API design decision rather than stale cleanup.

## What This Changes

Request a public context-engine or plugin hook contract that can inspect and update model-bound context before each LLM call inside multi-call agent loops.

## Best Possible Solution

Define a backward-compatible, maintainer-owned per-model-call context contract with explicit security gates, validation, failure behavior, and parity expectations across embedded and harness runtimes.

## Reproduction Assessment

Yes, at source level: current main shows the ordinary `assemble` call before `activeSession.prompt(...)`, while the only loop-time reassembly path is the specialized `ownsCompaction` transformContext hook. I did not run a live multi-call model scenario because this is a feature/API cadence review rather than a documented regression.

## Solution Assessment

Unclear as an implementation direction: the issue identifies a real remaining API question, but maintainers need to choose between changing ContextEngine cadence, adding a full-context hook, or coordinating with modifying LLM hooks.
