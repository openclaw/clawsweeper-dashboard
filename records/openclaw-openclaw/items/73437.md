## Summary

Keep open. Current main only partially addresses the request: owning context engines get a loop-time `assemble()` path, but non-owning engines still go through the tool-result guard and there is no public per-LLM-call full-message rewrite hook. The related additive `transform_context` PR is closed unmerged, so the remaining work is still a public API and trust-contract decision.

## What This Changes

The issue requests a per-model-call context assembly or transform surface so context engines/plugins can rewrite the full message history during multi-call agent loops.

## Best Possible Solution

Define a stable, documented per-model-call context rewrite contract, preferably additive unless maintainers intentionally change `assemble` semantics. The contract should specify prompt-content access, returned-message validation, ordering relative to budget enforcement, and coverage for owning engines, non-owning engines, and multi-tool-call loops.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: inspect the setup-time `assembleAttemptContextEngine()` call, the later `activeSession.prompt()` call, the `ownsCompaction` branch that gates loop-time assembly, and the absence of any public `transform_context` hook.

## Solution Assessment

Unclear. The additive hook direction from PR #57585 appears safer than globally changing `assemble` timing, but the API shape and trust model should be chosen by maintainers before implementation.

## Risks / Open Questions

- Changing `assemble` to fire for every loop-time model call could alter public context-engine semantics for existing third-party plugins.
- A full-message rewrite hook exposes prompt and transcript content to plugin code, so hook gating, conversation-access policy, and validation need explicit maintainer review.
- Hook ordering relative to Pi's private `transformContext`, OpenClaw budget enforcement, tool-result guards, and context-engine compaction is still an API/product choice.
