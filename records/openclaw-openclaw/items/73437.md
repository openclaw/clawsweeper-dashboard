## Label Justifications

- `P2`: This is a normal-priority plugin/context-engine API feature gap with real downstream impact but no active core outage.
- `impact:session-state`: The request changes when and how session context messages are assembled and rewritten during multi-call agent loops.

## Summary

Keep open: current main and v2026.6.8 still do not provide the requested public per-LLM-call full-context assembly or mutation contract. The existing ContextEngine loop hook is limited to `ownsCompaction` engines, and the related open LLM-hook PR is adjacent rather than a replacement for this context-engine cadence decision.

## What This Changes

Request a per-LLM-call full-context ContextEngine or plugin hook so context engines can inspect and rewrite messages before each model call inside tool loops.

## Best Possible Solution

Define a backward-compatible, maintainer-owned per-model-call context mutation contract with explicit security gates, validation, failure behavior, and runtime parity across embedded and Codex harness paths.

## Reproduction Assessment

Yes, at source level: current main shows ordinary `assemble` before `activeSession.prompt(...)`, while the only loop-time reassembly path is the specialized `ownsCompaction` transformContext hook. I did not run a live multi-call LLM scenario because this is a feature/API cadence review rather than a documented behavior regression.

## Solution Assessment

Unclear as an implementation direction: the issue identifies a real remaining API question, but maintainers need to choose between changing ContextEngine cadence, adding a full-context hook, or coordinating with modifying LLM hooks.
