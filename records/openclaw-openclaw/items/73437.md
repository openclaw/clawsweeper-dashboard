## Label Justifications

- `P2`: This is a normal-priority plugin/context-engine API feature gap with real downstream impact but no active core outage.
- `impact:session-state`: The request changes when and how session context messages are assembled and rewritten during multi-call agent loops.
- `impact:security`: A per-call full-context mutation contract would grant plugins raw conversation access and model-input rewrite authority.

## Summary

Keep open: current main and v2026.6.8 still do not provide a general public per-LLM-call ContextEngine assembly or full-context mutation contract; the only loop-time reassembly path remains the specialized `ownsCompaction` path.

## What This Changes

Request a ContextEngine or plugin hook path that can inspect and rewrite the full model-bound message context before each LLM call in an agentic tool loop.

## Best Possible Solution

Define a backward-compatible per-model-call context mutation contract with explicit conversation-access policy, payload validation, handler ordering, failure behavior, and embedded/Codex parity.

## Reproduction Assessment

Yes, at source level: current main shows ordinary `assemble` before `activeSession.prompt(...)`, while the only loop-time reassembly path is the specialized `ownsCompaction` transformContext hook. I did not run a live multi-call LLM scenario because this is a feature/API cadence review rather than a documented behavior regression.

## Solution Assessment

Unclear as an implementation direction: the issue identifies a real remaining API question, but maintainers need to choose between changing ContextEngine cadence, adding a full-context hook, or coordinating with modifying LLM hooks. Any solution needs explicit security gates and host parity.
