## Summary

Keep open. Current main partially addresses the problem for context engines that set `ownsCompaction: true`, but non-owning engines still take the tool-result guard path and there is still no public `transform_context` hook or other full-message rewrite hook before every model call. The related additive-hook PR is closed unmerged, so the remaining work is a public API and prompt-content trust decision rather than a completed implementation.

## What This Changes

The issue requests per-LLM-call context assembly or an additive `transform_context` hook so context engines/plugins can rewrite the full message array during multi-call tool loops.

## Best Possible Solution

Define a stable, documented per-model-call context rewrite contract that covers owning and non-owning context engines, or explicitly revise `assemble` semantics with compatibility notes. The design should specify prompt-content access permissions, message validation, hook ordering relative to budget enforcement and transcript sanitization, and tests for multi-tool-call loops.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure or reason about a non-owning active context engine, then follow the attempt runner path where only the tool-result guard is installed after the one setup-time `assembleAttemptContextEngine()` call; exact searches also confirm there is no public `transform_context` hook on current main.

## Solution Assessment

Unclear. The additive hook direction from #57585 appears safer than changing `assemble` for every engine, but the public API shape and prompt-content trust contract need maintainer approval before implementation.
