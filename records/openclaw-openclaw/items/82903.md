## Summary

Keep open: the linked bug is source-reproducible, but this PR changes a fresh-only helper globally and would make runtime compaction/fork paths trust stale token totals as fresh; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/config/sessions/types.ts` so stale preserved `totalTokens` are returned by `resolveFreshSessionTotalTokens` and edits `CHANGELOG.md`.

## Best Possible Solution

Keep `resolveFreshSessionTotalTokens` fresh-only for runtime decisions and use a separate preserved-total display path for `/context`, session list, status, and Control UI callers, with regression tests for both display and fresh-only paths.

## Reproduction Assessment

Yes. Source inspection shows current main filters finite stale totals through the fresh-only helper, and existing fork/preflight tests show why changing that helper globally regresses fresh-only runtime callers.

## Solution Assessment

No. The requested display behavior is valid, but this PR is not the safest implementation because it changes the shared fresh-only helper instead of routing only display consumers through preserved totals.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep fresh-only callers from using stale token totals:** `src/config/sessions/types.ts:553-556`
  - body: Removing the freshness guard makes `resolveFreshSessionTotalTokens` return stale preserved values to runtime callers. `resolveParentForkTokenCountRuntime` exits early on that helper, so a stale parent row like the existing fixture with `totalTokens: 1` and `totalTokensFresh: false` would return `1` instead of the transcript-derived count; preflight compaction similarly skips transcript estimation when the helper returns a number. Keep this helper fresh-only and add a separate preserved/display-total path for `/context` and UI callers.
  - confidence: 0.94
