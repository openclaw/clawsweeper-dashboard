## Summary

Keep open: current main still gates QQBot debug logging with `!!process.env.QQBOT_DEBUG`, so `QQBOT_DEBUG=0` remains enabled. The report is source-backed, narrow, and not covered by an existing merged fix or canonical duplicate.

## What This Changes

The issue requests that QQBot debug logging only enable for explicit truthy `QQBOT_DEBUG` values and stay disabled for common false-like values.

## Best Possible Solution

Replace the QQBot-local debug guard with explicit truthy parsing and add regression tests for false-like `QQBOT_DEBUG` values without importing core internals into plugin production code.

## Reproduction Assessment

Yes. Source inspection shows the current `isDebug` guard uses string truthiness, and a Node probe with `QQBOT_DEBUG='0'` confirms that expression evaluates as enabled.

## Solution Assessment

Yes. The narrow maintainable fix is to change only the QQBot logger guard to an explicit allow-list, then cover `0`, `false`, `off`, `no`, blank, unset, and true-like values in `log.test.ts`.
