## Summary

Keep open. The PR targets a real heartbeat leak, but the current diff is not merge-ready because it applies generic message-mode token stripping to heartbeat payloads, would regress current-main heartbeat tool-block sanitation, and lacks the required user-facing changelog entry.

## What This Changes

The PR changes `buildReplyPayloads` to strip `HEARTBEAT_OK` from heartbeat final payloads and adds focused payload-builder regression tests for OK suppression, alert preservation, media preservation, and heartbeat error formatting scope.

## Best Possible Solution

Land a rebased payload-layer fix that suppresses exact OK-only heartbeat final payloads while preserving heartbeat-mode `ackMaxChars`, current tool-block sanitation, focused regression coverage, and a single-line changelog entry.

## Reproduction Assessment

Yes. Source inspection shows the original leak at `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` on current main, and the PR regression is also source-reproducible from applying message-mode stripping before heartbeat-mode `ackMaxChars` handling can preserve short internal acks.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but the maintainable fix needs heartbeat-aware OK suppression that composes with `sanitizeHeartbeatPayload` and existing heartbeat ack semantics.
