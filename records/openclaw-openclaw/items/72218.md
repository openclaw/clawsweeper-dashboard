## Summary

Keep open. The PR targets a real heartbeat payload leak, but the current patch is not merge-ready because it uses message-mode token stripping for heartbeat payloads, is stale against current main's heartbeat tool-block sanitizer, and lacks the required changelog entry.

## What This Changes

The PR changes `buildReplyPayloads` so heartbeat final payloads run through `HEARTBEAT_OK` stripping, keeps Bun socket error formatting out of heartbeat runs, and adds focused payload-builder tests.

## Best Possible Solution

Land a narrow payload-layer fix that suppresses exact OK-only heartbeat final payloads while preserving heartbeat-mode `ackMaxChars`, current heartbeat payload sanitization, focused regression coverage, and the required changelog entry.

## Reproduction Assessment

Yes. The original leak is reproducible at the source level with `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })`, and the PR regression follows from `HEARTBEAT_OK all good` being stripped in message mode instead of heartbeat mode.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but heartbeat payloads need heartbeat-mode semantics and the current sanitizer rather than generic message-mode stripping alone.
