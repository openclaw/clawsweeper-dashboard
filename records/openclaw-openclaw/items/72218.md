## Summary

Keep open. The PR targets a real current-main heartbeat leak, but the branch is not merge-ready because it applies message-mode token stripping to heartbeat payloads before heartbeat-mode `ackMaxChars` suppression can run, which can make short ack text visible. It also lacks the required user-facing changelog entry.

## What This Changes

The PR changes `buildReplyPayloads` so heartbeat final payloads run through `HEARTBEAT_OK` sanitation, keeps Bun socket error formatting scoped away from heartbeat payloads, and adds focused payload-builder regression tests.

## Best Possible Solution

Keep the narrow payload-layer fix direction, but preserve heartbeat-mode ack semantics: exact OK-only heartbeat final payloads should be suppressed before channel delivery, while token-prefixed short heartbeat acks must still go through `ackMaxChars` handling or an equivalent local heartbeat-mode check with focused coverage.

## Reproduction Assessment

Yes. Current main can be proved through `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })`, which returns the raw payload because of the heartbeat bypass; the branch regression follows from `HEARTBEAT_OK all good` being stripped in message mode before heartbeat-mode suppression can run.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but applying message-mode stripping to all heartbeat final payloads is not the narrowest safe fix because it changes the documented heartbeat ack contract.
