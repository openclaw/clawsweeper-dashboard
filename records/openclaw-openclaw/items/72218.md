## Summary

Keep open: this PR targets a valid heartbeat leak, but the branch should not merge as written because it applies message-mode stripping to heartbeat payloads, which can expose short ack text and bypass the current heartbeat-specific tool-block sanitizer. It also still needs real delivery-path proof before merge.

## What This Changes

The PR changes `buildReplyPayloads` to strip `HEARTBEAT_OK` from final heartbeat payloads and adds focused payload-builder regression tests.

## Best Possible Solution

Rebase the branch and implement heartbeat-specific payload sanitation that drops OK-only heartbeat final payloads while preserving `sanitizeHeartbeatPayload`, heartbeat-mode `ackMaxChars`, media retention, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` still bypasses token stripping, and the PR regression is source-reproducible from message-mode stripping before heartbeat-mode `ackMaxChars` can apply.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but the fix should add OK-only suppression inside the heartbeat-aware sanitizer path rather than replacing it with the generic message branch.
