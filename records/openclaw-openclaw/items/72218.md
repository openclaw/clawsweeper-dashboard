## Summary

Keep open: this targets a real heartbeat payload bug, but the branch is conflicting, lacks real delivery-path proof, and its generic message-mode stripping would regress current-main heartbeat-specific sanitation if rebased mechanically.

## What This Changes

The PR changes `buildReplyPayloads` so final heartbeat payloads go through generic `HEARTBEAT_OK` token stripping and adds focused payload-builder tests.

## Best Possible Solution

Rebase or replace with a small patch that performs `HEARTBEAT_OK` suppression inside heartbeat-specific payload sanitation while retaining legacy tool-block cleanup, media retention, heartbeat-mode `ackMaxChars` behavior, regression tests, and live delivery proof.

## Reproduction Assessment

Yes, at source level. Current main maps heartbeat final payloads only through `sanitizeHeartbeatPayload`, so `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` still leaves a renderable payload at that layer.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but the maintainable fix should compose OK-only suppression with the heartbeat-aware sanitizer and heartbeat-mode `ackMaxChars` semantics.
