## Summary

Keep open: the reported bug is valid, but this PR is not correct as written because it applies message-mode `HEARTBEAT_OK` stripping to heartbeat payloads and does not preserve current heartbeat-specific sanitation; it also still lacks real delivery-path proof.

## What This Changes

The PR changes `buildReplyPayloads` so heartbeat final payloads containing `HEARTBEAT_OK` go through token stripping and adds focused payload-builder tests.

## Best Possible Solution

Land a rebased payload-layer fix that suppresses OK-only heartbeat final payloads while preserving `sanitizeHeartbeatPayload`, heartbeat-mode `ackMaxChars` semantics, focused regression tests, and redacted real channel proof.

## Reproduction Assessment

Yes from source inspection. On current main, `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` goes through the heartbeat sanitizer path without `HEARTBEAT_OK` stripping, and the PR regression is source-reproducible from using message-mode stripping for heartbeat payloads.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but the fix should add OK suppression inside the heartbeat-aware sanitizer path instead of replacing heartbeat-specific sanitation and ack handling with the generic message branch.
