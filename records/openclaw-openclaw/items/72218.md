## Summary

Keep open: the bug remains on current main and this PR targets the right payload layer, but the branch conflicts with main, regresses heartbeat-specific sanitation/ack semantics, and still lacks after-fix real delivery proof.

## What This Changes

The PR changes `buildReplyPayloads` to run heartbeat final payloads through `HEARTBEAT_OK` stripping and adds focused payload tests for OK-only, alert, media, and Bun error cases.

## Best Possible Solution

Compose OK-only suppression inside the heartbeat-aware sanitizer path, preserving legacy tool-block cleanup, heartbeat-mode `ackMaxChars`, media retention, and focused regression coverage, then add redacted real channel proof.

## Reproduction Assessment

Yes, at source level. On current main, `buildReplyPayloads` routes `isHeartbeat: true` payloads through `sanitizeHeartbeatPayload` without calling `stripHeartbeatToken`, so a token-only final heartbeat payload remains renderable at that layer.

## Solution Assessment

No, not as written. Removing the raw bypass is directionally right, but the fix should compose OK suppression with heartbeat-specific sanitation and heartbeat-mode stripping rather than switching heartbeat payloads to generic message-mode behavior.
