## Summary

Keep open. The PR targets a real heartbeat leak, and the earlier Greptile test-assertion concern was addressed, but the current head is still not merge-ready because it applies message-mode stripping to heartbeat payloads, can fail to preserve current-main heartbeat tool-block sanitation, and lacks the required changelog entry.

## What This Changes

The PR changes `buildReplyPayloads` so heartbeat final payloads run through `HEARTBEAT_OK` stripping and adds focused payload-builder tests.

## Best Possible Solution

Land a rebased payload-layer fix that suppresses exact OK-only heartbeat final payloads while preserving heartbeat-mode `ackMaxChars`, current tool-block sanitation, focused regression coverage, and a changelog entry.

## Reproduction Assessment

Yes. The original leak is source-reproducible from `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` on current main, and the PR regression is source-reproducible from stripping heartbeat payloads in message mode before heartbeat-mode `ackMaxChars` semantics can apply.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but the fix needs heartbeat-aware suppression that composes with current-main sanitizer behavior instead of generic message-mode stripping for every heartbeat payload.
