## Summary

Keep open. Current main still has the heartbeat payload bypass this PR targets, but the proposed diff is not merge-ready because it routes all heartbeat final payload token handling through message-mode stripping before the heartbeat runner can apply heartbeat-mode `ackMaxChars` semantics.

## What This Changes

The PR removes the heartbeat early return in `buildReplyPayloads`, keeps Bun socket error formatting scoped to non-heartbeat runs, and adds focused payload-builder regression tests for `HEARTBEAT_OK` final payload sanitation.

## Best Possible Solution

Keep the narrow payload-layer direction, but make the final-payload sanitizer heartbeat-aware: suppress exact OK-only heartbeat payloads before channel delivery while preserving prefixed or short heartbeat acknowledgments for the runner's `ackMaxChars` logic, then add regression coverage and the required changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced at the payload layer with `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })`, and the PR regression is visible by reasoning from `[openclaw] HEARTBEAT_OK all good` becoming `all good` before `normalizeHeartbeatReply` can apply heartbeat-mode suppression.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but applying message-mode stripping to every heartbeat final payload is broader than the documented contract; exact OK-only suppression or equivalent heartbeat-mode handling is safer.
