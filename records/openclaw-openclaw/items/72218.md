## Summary

This PR targets a real heartbeat delivery bug, but the current patch still has correctness blockers: it applies generic message-mode stripping to heartbeat payloads and drops the heartbeat-specific sanitizer path now present on main. It also needs real behavior proof before merge because the PR body only shows focused test/check output.

## What This Changes

The PR changes `buildReplyPayloads` to strip `HEARTBEAT_OK` from heartbeat final payloads and adds focused payload-builder tests.

## Best Possible Solution

Land a rebased payload-layer fix that suppresses exact OK-only heartbeat final payloads while preserving heartbeat-specific tool-block cleanup, heartbeat-mode `ackMaxChars`, and focused regression coverage.

## Reproduction Assessment

Yes. The original leak is source-reproducible from `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` on current main, and the PR regression is source-reproducible from using message-mode stripping before heartbeat-mode `ackMaxChars` can apply.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but the fix should add OK-only suppression inside the heartbeat-aware sanitizer path rather than replacing it with the generic message branch.
