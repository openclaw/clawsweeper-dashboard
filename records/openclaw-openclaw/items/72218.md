## Summary

Keep open: current main still has the source-level heartbeat final-payload leak, but this PR is not mergeable as written because it would route heartbeat payloads through generic message-mode stripping and bypass the newer heartbeat-specific sanitizer. It also still lacks after-fix real channel proof.

## What This Changes

The PR changes `buildReplyPayloads` to run heartbeat final payloads through `HEARTBEAT_OK` stripping and adds focused tests for OK-only suppression, alert preservation, media retention, and Bun error scope.

## Best Possible Solution

Rebase and implement exact OK-only suppression inside the heartbeat-specific sanitizer path, preserving legacy tool-block cleanup and heartbeat-mode ack behavior, then add focused regression coverage plus redacted real Telegram or equivalent channel proof.

## Reproduction Assessment

Yes, at source level. Current main routes `isHeartbeat: true` payloads through `sanitizeHeartbeatPayload` without calling `stripHeartbeatToken`, so an exact `HEARTBEAT_OK` final payload remains renderable at the payload-builder layer.

## Solution Assessment

No, not as written. The direction is right, but the patch should compose OK suppression into the heartbeat sanitizer instead of switching heartbeat payloads to generic message-mode stripping and losing current sanitizer behavior.
