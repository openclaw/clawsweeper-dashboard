## Summary

Keep open. The PR fixes a real leak direction, but the current branch is not merge-ready because it applies message-mode token stripping to heartbeat final payloads, which changes the documented `ackMaxChars` heartbeat-ack contract, and it now also needs to compose with current main's heartbeat tool-block sanitizer and changelog policy.

## What This Changes

The PR changes `buildReplyPayloads` so heartbeat final payloads run through `HEARTBEAT_OK` stripping, keeps Bun socket error formatting out of heartbeat runs, and adds focused payload-builder tests.

## Best Possible Solution

Land a narrow payload-layer fix that suppresses exact OK-only heartbeat final payloads while preserving heartbeat-mode `ackMaxChars`, current heartbeat tool-block sanitation, focused regression coverage, and the required changelog entry.

## Reproduction Assessment

Yes. The source-level repro is `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` for the original leak, and the PR regression follows from `HEARTBEAT_OK all good` being stripped in message mode before heartbeat-mode `ackMaxChars` suppression can run.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but heartbeat payloads need heartbeat-mode semantics or an equivalent local check rather than generic message-mode stripping.
