## Summary

Keep this PR open, but it should not merge in its current shape: the latest head still has P1 security blockers in Harvester ingress, PostgreSQL TLS, and child-process environment handling, plus incomplete real behavior proof for most claimed behavior.

## What This Changes

The branch adds an AlpaCore MCP bridge/editor config, devices CLI fallback behavior, Control UI origin persistence, Azure Container Apps deploy assets, and a new Harvester backend/control UI stack.

## Best Possible Solution

Split the narrow existing-behavior fixes into focused PRs, and handle the new Harvester/ACA/AlpaCore product surface only after explicit auth, CORS, TLS, environment-sanitization, image-provenance, ownership, and runtime-proof decisions are resolved.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restrict Harvester ingress before exposing mutations:** `infra/main.bicep:831-832`
  - body: The Harvester Container App is external and allows any browser origin, while the server exposes unauthenticated POST routes for cleanup, target creation, and worker scaling. Restrict CORS to expected Control UI origins and require auth for deployed mutation routes.
  - confidence: 0.94
- **[P1] Keep PostgreSQL certificate verification enabled:** `harvester-server.mjs:400`
  - body: Every non-disable SSL mode becomes `{ rejectUnauthorized: false }`, so the default cloud database path skips server certificate validation. Keep verification on by default and reserve insecure TLS for an explicit local/debug setting.
  - confidence: 0.95
- **[P1] Sanitize the deploy helper environment before spawning:** `scripts/azd-deploy-openclaw-from-package.mjs:147-149`
  - body: The helper forwards raw `process.env` plus overrides into `az`, `docker`, and `azd`. This child-process boundary should use the existing host exec sanitizer with explicit safe overrides to avoid inherited environment injection.
  - confidence: 0.9
- **[P2] Use reproducible base images for AlpaCore builds:** `infra/images/alpa-engine/Dockerfile:1`
  - body: The AlpaCore Dockerfiles depend on local unqualified `latest` images, so clean CI or ACR builds will fail unless those tags happen to exist on the builder. Pin registry images, build from source, or make the base image an explicit input.
  - confidence: 0.91
- **[P2] Point Cursor MCP at the serving AlpaCore port:** `.cursor/mcp.json:9`
  - body: Cursor is configured for port 5143, but the VS Code config and bridge default use port 8080. Cursor users will connect to the wrong endpoint unless the port is aligned or the difference is documented and backed by the service config.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security and supply-chain concerns in public ingress, database TLS, child-process env handling, and image provenance.

Concerns:

- **[high] Wildcard CORS on public mutation surfaces:** `infra/main.bicep:831`
  - body: The new Harvester Container App permits any origin while the server exposes unauthenticated state-mutating endpoints, creating cross-origin abuse risk for deployed instances.

- **[high] PostgreSQL TLS skips certificate verification:** `harvester-server.mjs:400`
  - body: The Harvester database connection treats SSL as required but sets `rejectUnauthorized: false`, so the default cloud database path does not verify the server certificate.

  - body: The new helper passes raw process environment values to external tools instead of using the existing host exec sanitizer, leaving a command-execution boundary exposed to inherited env injection.
  - confidence: 0.9
- **[medium] Unqualified latest images weaken build provenance:** `infra/images/alpa-engine/Dockerfile:1`
  - body: The added AlpaCore Dockerfiles depend on local `latest` image tags rather than pinned or source-built inputs, making the deploy path non-reproducible and vulnerable to accidental or untrusted local image substitution.
