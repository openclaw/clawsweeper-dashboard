## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restrict Harvester ingress before exposing mutations:** `infra/main.bicep:743-745`
  - body: The new Harvester app is externally reachable with wildcard CORS, while the server adds unauthenticated POST routes for cleanup, target creation, and worker scaling. Any browser origin can drive those deployed mutation endpoints; restrict CORS to expected Control UI origins and add auth for non-local deployments.
  - confidence: 0.94
- **[P1] Keep PostgreSQL certificate verification enabled:** `harvester-server.mjs:294`
  - body: `sslMode` defaults to `require`, but every non-`disable` value becomes `{ rejectUnauthorized: false }`. That encrypts the Harvester database connection while skipping server certificate validation by default; verification should stay on unless an explicit insecure local mode is selected.
  - confidence: 0.95
- **[P1] Sanitize the deploy helper environment before spawning:** `scripts/azd-deploy-openclaw-from-package.mjs:147-150`
  - body: The helper forwards raw `process.env` plus overrides into `az`, `docker`, and `azd`. OpenClaw already has `sanitizeHostExecEnv` for this child-process boundary, and the PR has a GHAS warning for this pattern; build the env through that sanitizer with explicit safe overrides.
  - confidence: 0.9
- **[P2] Use reproducible base images for azd services:** `infra/images/alpa-engine/Dockerfile:1`
  - body: The AlpaCore service Dockerfiles depend on local unqualified `latest` base images, so a clean CI or ACR build will fail unless those images happen to exist on the builder. Use pinned registry images, build from source in this repo, or make the base image an explicit build arg.
  - confidence: 0.91
- **[P2] Point Cursor MCP at the serving AlpaCore port:** `.cursor/mcp.json:9`
  - body: Cursor is configured with `ALPACORE_URL` on port 5143, while the bridge default and VS Code config use port 8080. Unless the service really listens on 5143, Cursor users will connect to the wrong port; align the editor configs with the serving endpoint.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security and supply-chain concerns in public ingress, database TLS, child-process env handling, and image provenance.

Concerns:

- **[high] Wildcard CORS on public mutation surfaces:** `infra/main.bicep:743`
  - body: The new Harvester Container App permits any origin while the server exposes unauthenticated state-mutating endpoints, creating cross-origin abuse risk for deployed instances.
  - confidence: 0.94
- **[high] PostgreSQL TLS skips certificate verification:** `harvester-server.mjs:294`
  - body: The Harvester database connection treats SSL as required but sets `rejectUnauthorized: false`, so the default cloud database path does not verify the server certificate.
  - confidence: 0.95
- **[high] Deploy helper forwards unsanitized environment:** `scripts/azd-deploy-openclaw-from-package.mjs:147`
  - body: The new helper passes raw process environment values to external tools instead of using the existing host exec sanitizer, leaving workspace or inherited env injection risk at a command-execution boundary.
  - confidence: 0.9
- **[medium] Unqualified latest images weaken build provenance:** `infra/images/alpa-engine/Dockerfile:1`
  - body: The added AlpaCore Dockerfiles depend on local `latest` image tags rather than pinned or source-built inputs, making the deploy path non-reproducible and vulnerable to accidental or untrusted local image substitution.
  - confidence: 0.9
