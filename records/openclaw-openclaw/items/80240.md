## Summary

Keep this PR open, but it is not mergeable as-is: the latest head still has blocking security/runtime defects, mixes several product surfaces into one branch, and only supplies real behavior proof for one Harvester slice.

## What This Changes

The PR adds an AlpaCore MCP bridge, local devices fallback, Control UI origin seeding, Azure Container Apps deployment assets, and a new Harvester backend/control UI stack.

## Best Possible Solution

Split the narrow devices fallback and Control UI origin persistence changes into focused PRs, and handle AlpaCore/Harvester/Azure as a separately designed integration with explicit auth, CORS, TLS, env sanitization, image provenance, docs, and real runtime proof.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR head shows wildcard public CORS around unauthenticated Harvester mutations, disabled PostgreSQL certificate verification, raw child-process env forwarding, local latest image bases, and a Cursor port mismatch. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR contains useful narrow fixes, but bundling them with a new Harvester/Azure product surface and unresolved security/runtime blockers is not the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restrict Harvester ingress before exposing mutations:** `infra/main.bicep:743-745`
  - body: The new Harvester app is externally reachable with `allowedOrigins: ['*']`, while `harvester-server.mjs` exposes unauthenticated POST routes for cleanup, target creation, and worker scaling. Any browser origin can drive those mutations against a deployed app; restrict CORS to expected Control UI origins and add auth for non-local deployments.
  - confidence: 0.94
- **[P1] Keep PostgreSQL certificate verification enabled:** `harvester-server.mjs:294`
  - body: `sslMode` defaults to `require`, but every non-`disable` value becomes `{ rejectUnauthorized: false }`. That encrypts the Harvester database connection while skipping server certificate validation by default; verification should be on unless an explicit insecure local option is selected.
  - confidence: 0.95
- **[P1] Sanitize the deploy helper environment before spawning:** `scripts/azd-deploy-openclaw-from-package.mjs:147-150`
  - body: The new helper forwards raw `process.env` plus overrides into `az`, `docker`, and `azd`. OpenClaw already has `sanitizeHostExecEnv` for this child-process boundary, and the PR has a GHAS warning for this pattern; build the env through that sanitizer with explicit safe overrides.
  - confidence: 0.9
- **[P2] Use reproducible base images for azd services:** `infra/images/alpa-engine/Dockerfile:1`
  - body: The AlpaCore service Dockerfiles use local unqualified `latest` base images, so a clean CI or ACR build will fail unless those images happen to exist on the builder. Use pinned registry images, build from source in this repo, or make the base image an explicit build arg.
  - confidence: 0.91
- **[P2] Point Cursor MCP at the serving AlpaCore port:** `.cursor/mcp.json:9`
  - body: The Cursor config points `ALPACORE_URL` at port 5143, but the bridge default, VS Code config, and Harvester service use port 8080. Cursor users will connect to an unused port unless the config or service listener is aligned.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security and supply-chain concerns in public ingress, database TLS, child-process env handling, and image provenance.

Concerns:

- **[high] Wildcard CORS on public mutation surfaces:** `infra/main.bicep:743`
  - body: The new Harvester Container App permits any origin while the server exposes unauthenticated state-mutating endpoints, creating cross-origin abuse risk for deployed instances.
  - confidence: 0.94
- **[high] PostgreSQL TLS skips certificate verification:** `harvester-server.mjs:294`
  - body: The Harvester database connection treats SSL as required but sets `rejectUnauthorized: false`, so the default cloud database path does not verify the server certificate.
  - confidence: 0.95

  - body: The new helper passes raw process environment values to external tools instead of using the existing host exec sanitizer, leaving workspace or inherited env injection risk at a command-execution boundary.
  - confidence: 0.9
- **[medium] Unqualified latest images weaken build provenance:** `infra/images/alpa-engine/Dockerfile:1`
  - body: The added AlpaCore Dockerfiles depend on local `latest` image tags rather than pinned or source-built inputs, making the deploy path non-reproducible and vulnerable to accidental or untrusted local image substitution.
  - confidence: 0.9
