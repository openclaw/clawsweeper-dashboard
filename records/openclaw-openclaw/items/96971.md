## Summary

Keep open: the linked gateway TLS SAN bug is still present on current main and v2026.6.10, and this PR is a real candidate fix, but the latest head can still rewrite explicit CN-only certificate deployments and changes TLS identity scope without maintainer/security confirmation.

## What This Changes

The PR adds subjectAltName entries to gateway auto-generated self-signed certificates and adds startup-time regeneration of older CN-only cert/key pairs with backup and rollback handling.

## Best Possible Solution

Land the SAN generator fix after maintainers confirm the supported identities and ensure legacy repair cannot rewrite explicit/custom certificate material, preferably by limiting runtime upgrade to provably OpenClaw-owned default paths or moving repair to doctor.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 generate CN-only gateway certificates, and Node v24 `checkServerIdentity` requires IP SAN entries for IP hostnames. I did not run a full gateway handshake in this read-only review.

## Solution Assessment

No. Adding SANs in the gateway TLS generator is the right layer, but this PR is not the best final shape until certificate ownership and the supported SAN identity set are constrained.
