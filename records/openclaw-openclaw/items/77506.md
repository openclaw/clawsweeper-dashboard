## Summary

Keep open: current main still has the active-memory embedded recall run asking for `memory_recall`, while the bundled `memory-core` plugin only advertises and registers `memory_search` and `memory_get`. The exact session-wipe side effect was not live-reproduced, and `memory_recall` is documented for a non-core memory backend, so the fix should avoid unavailable names without removing backend compatibility.

## What This Changes

Active-memory should not request unavailable memory tools in its embedded recall sub-agent when the selected memory backend is memory-core.

## Best Possible Solution

Resolve active-memory's embedded recall tool allowlist from the selected or available memory plugin tools, preserving `memory_recall` support when a backend provides it and omitting it for memory-core-only runs.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the active-memory allowlist includes `memory_recall` while bundled memory-core exposes only `memory_search` and `memory_get`, which is enough to justify keeping this open.

## Solution Assessment

No, blindly removing `memory_recall` is probably not the best fix because current docs reserve it for memory-lancedb. A safer fix is to build the allowlist from the selected backend's registered or manifest-declared memory tools and add regression coverage for memory-core.
