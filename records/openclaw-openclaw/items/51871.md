## Summary

Keep this issue open. The discussion narrowed the regression to malformed persisted cron rows, and current main still has Control UI paths that dereference `schedule.kind`, `payload.kind`, and `payload.message` without guarding against rows that survived store load in an incomplete shape.

## What This Changes

The issue requests that the Control UI Cron dashboard continue displaying cron jobs instead of blanking when persisted `jobs.json` contains malformed or legacy cron rows.

## Best Possible Solution

Harden cron store loading so malformed persisted rows are either safely normalized from legacy fields or quarantined with clear diagnostics, then make the Control UI cron list/edit/render paths defensive so one bad row cannot blank the dashboard.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a persisted `jobs.json` containing at least one row missing `schedule` or a valid `payload.kind`, then loading the Control UI Cron page; current source still dereferences those fields directly.

## Solution Assessment

Yes. The narrow maintainable solution is store-boundary validation or quarantine plus defensive Control UI rendering; editing generated `dist` files or adding optional chaining at only one UI call site would leave the underlying persisted-state contract broken.
