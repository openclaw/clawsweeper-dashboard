## Summary

Keep this issue open: current main still has a source-reproducible malformed persisted cron row path, and open PR #75970 is the active closing implementation rather than a reason to close before merge.

## What This Changes

Cron dashboard and cron list surfaces should continue showing valid jobs when one persisted `jobs.json` row is malformed or missing schedule/payload fields.

## Best Possible Solution

Cron store loading should validate or quarantine unrecoverable persisted rows before runtime/UI consumers, preserve raw data for repair, and keep defensive UI guards as a second line of protection.

## Reproduction Assessment

Yes. A persisted cron row with a valid payload but missing or non-object `schedule` can still reach current-main cron read/UI paths where `job.schedule.kind` is dereferenced.

## Solution Assessment

Yes. The narrow maintainable direction is to validate or quarantine malformed persisted rows at the cron store boundary, then keep UI guards so one bad row cannot blank the dashboard.
