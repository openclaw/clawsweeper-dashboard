## Summary

Keep this issue open. Current main still has a source-reproducible cron dashboard/backend failure path for malformed persisted cron rows, and open PR #75970 is the active closing implementation rather than a reason to close before merge.

## What This Changes

The report asks for the Control UI cron dashboard to display existing cron jobs instead of going blank when persisted legacy or hand-edited cron rows are malformed.

## Best Possible Solution

Review and land or replace #75970 so malformed persisted cron rows are validated or quarantined at the store boundary and the Control UI cannot be blanked by one bad row.

## Reproduction Assessment

Yes. A persisted cron row with a valid payload but missing or non-object `schedule` can still reach current-main cron list/dashboard paths, where `job.schedule.kind` is dereferenced in backend maintenance and UI filtering/edit hydration.

## Solution Assessment

Yes. The best path is the linked PR direction: validate or quarantine malformed persisted rows at the cron store boundary, preserve user data for repair, and keep defensive UI guards as a second line of protection.
