## Summary

Keep this issue open. Current main still has a source-reproducible path where a persisted cron row with a valid payload but missing schedule can reach Control UI schedule dereferences, and the linked closing PR is still open rather than merged.

## What This Changes

The issue reports that malformed persisted cron jobs can blank the Control UI cron dashboard even though cron jobs continue running and CLI listing may work.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75970 with store-boundary validation/quarantine that preserves raw malformed rows for repair, plus defensive UI schedule guards.

## Reproduction Assessment

Yes. A persisted cron row with a valid `payload` but missing or non-object `schedule` can pass the Control UI payload-only filter and then reach `job.schedule.kind` or `formatCronSchedule` on current main.

## Solution Assessment

Yes. Store-boundary quarantine/validation with UI schedule guards is the narrow maintainable direction; broad optional chaining in built artifacts would be weaker and could leave malformed runtime state circulating.
