## Summary

Keep open: current main and the latest release still do not provide a claude-cli MCP path to `exec` or `process`, and the requested default-off exposure is a security-sensitive command-execution boundary that needs maintainer product/security direction before implementation.

## What This Changes

The issue requests an experimental opt-in that exposes `exec` and possibly `process` through the bundled claude-cli MCP surface while preserving OpenClaw's existing exec approval flow.

## Best Possible Solution

Design a maintainer-approved, default-off MCP exposure path that constructs `exec` and any allowed `process` subset through the existing exec approval machinery, with schema/docs/tests and live claude-cli proof; if that boundary is rejected, document embedded runtime as the supported workaround.

## Reproduction Assessment

Yes, source-reproducible: current main explicitly excludes `exec` and `process` from the loopback MCP tool surface, and the strict config schema has no opt-in key to restore them. I did not run a live Claude CLI turn.

## Solution Assessment

Unclear: the requested default-off flag is plausible, but it is not a safe autonomous fix because exposing shell tools over MCP needs maintainer security/product agreement and likely more design than removing names from the exclude set.
