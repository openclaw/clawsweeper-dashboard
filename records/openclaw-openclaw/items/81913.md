## Summary

Keep open: current main still exposes only refresh-state helpers from `skills-runtime` and omits the requested host-env helpers from `sandbox`; a linked open PR proposes part of the SDK surface, while later comments and the published ClawHub plugin show remaining SDK/security boundary questions.

## What This Changes

The issue requests a stable public plugin SDK surface for third-party plugins that inspect installed skills, parse `SKILL.md` metadata, resolve skill-local paths, read skill config/env, and sanitize subprocess env overlays.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/81921 or replace it with a maintainer-shaped SDK slice that documents and tests narrow skills metadata, config/path, and env-sanitization contracts, using the published ClawHub plugin as a reference case.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof of the current public SDK export gap on main.

## Solution Assessment

Unclear until maintainer/security review. A small additive SDK surface may be right, but security-sensitive env and path behavior should likely use narrow documented wrappers rather than broad internal helper promotion.
