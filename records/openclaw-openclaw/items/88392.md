## Summary

Keep open: current main and the latest release still expose only catalog-shaped `models.list` rows to RPC clients, while the CLI has the credential-aware `available` computation the issue asks to share.

## What This Changes

Request to expose the CLI’s per-model availability signal to third-party plugins and Gateway RPC clients.

## Best Possible Solution

Add a narrow, documented availability surface that reuses the CLI’s single availability computation and updates protocol, SDK/docs, and focused tests together.

## Reproduction Assessment

Yes for the API gap: current source shows `models.list` schemas and tests omit `available` and reject `view: "available"`. This is a feature request rather than a failing existing runtime behavior.

## Solution Assessment

Unclear until maintainers choose the public contract shape. The narrowest maintainable direction is to centralize the CLI availability logic and expose it through an additive RPC field or focused SDK subpath, not duplicate heuristics downstream.
