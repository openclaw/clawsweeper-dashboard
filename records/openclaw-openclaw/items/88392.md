## Summary

Current main and the v2026.6.6 release now expose model availability on `models.list`, so the requested RPC path is implemented and downstream plugins/RPC clients no longer need to rederive the flag for that surface.

## What This Changes

The issue requests exposing the CLI's per-model `available` model signal to plugins or Gateway RPC clients.

## Best Possible Solution

Keep the shipped additive `models.list.available` RPC contract; handle narrower availability-correctness bugs as separate follow-up issues or PRs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection verifies current main and v2026.6.6 now expose the requested optional `available` field on `models.list` rows.

## Solution Assessment

Yes. The shipped solution uses the issue's preferred additive RPC path, avoids downstream SDK algorithm duplication for RPC clients, and keeps the shared result field optional for compatibility.
