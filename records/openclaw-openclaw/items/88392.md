## Summary

Current main and the shipped v2026.6.6 release already implement the central request by exposing a credential-aware optional `available` field on `models.list` rows, with focused coverage for the ChatGPT/Codex routing divergence described in the issue.

## What This Changes

The issue requests exposing OpenClaw's per-model availability signal to plugins and Gateway RPC clients so external model pickers do not reimplement CLI-only heuristics.

## Best Possible Solution

Keep the shipped additive `models.list` availability field as the canonical public surface; open a narrower follow-up only if maintainers want a direct in-process SDK helper beyond the RPC path.

## Reproduction Assessment

Not applicable as a current bug reproduction: this was an API feature request, and current main plus v2026.6.6 now expose the requested `available` signal through `models.list`. The original source gap is no longer present in the protocol schema or Gateway handler.

## Solution Assessment

Yes. The implemented Option 1 is the issue's preferred path and keeps availability single-sourced through Gateway RPC instead of exporting broad CLI/model-registry internals through the plugin SDK.
