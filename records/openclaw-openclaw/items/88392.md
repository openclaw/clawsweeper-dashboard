## Summary

Current main already implements the requested public model-availability surface: `models.list` now returns an optional credential-aware `available` flag, including coverage for the ChatGPT/Codex availability divergence, and the implementation is present in the v2026.6.6 release.

## What This Changes

Expose OpenClaw’s credential-aware per-model availability signal to external model-list consumers so third-party pickers can match the usable model set.

## Best Possible Solution

Keep the shipped additive `models.list` availability field as the canonical public surface; file a narrower follow-up only if a direct in-process SDK helper is still needed beyond the RPC path.

## Reproduction Assessment

Not applicable as a current-main bug reproduction: this was an API feature request, and current main now exposes `available` through `models.list`. The old source-level gap is no longer present in the checked source or the v2026.6.6 tag.

## Solution Assessment

Yes. The shipped additive RPC field matches the issue’s preferred Option 1 and avoids publishing broad CLI/registry internals through the plugin SDK.
