## Summary

Keep this issue open. Current main still lacks the requested lightweight workspace/user `subagents/<label>.md` role-file convention for native `sessions_spawn`; the code path only accepts explicit spawn parameters, configured agent/subagent defaults, inline attachments, and broad subagent tool policy. The feature is plausible but needs maintainer API/security judgment because frontmatter-driven model and tool changes would affect spawn precedence and capability policy.

## What This Changes

Request persistent workspace-scoped and user-scoped markdown role files for `sessions_spawn` subagents, selected by label and merged into subagent system context, model, and tool policy.

## Best Possible Solution

Design the feature deliberately before implementation: if accepted for core, add a native resolver in the `sessions_spawn`/`spawnSubagentDirect` path with workspace-before-user precedence, explicit spawn params overriding frontmatter, tool-policy narrowing by default, safe model override validation, docs, and focused tests for missing files, invalid frontmatter, run vs session mode, fork vs isolated context, and unchanged behavior when no role file exists.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing feature. The static reproduction path is high-confidence: on current main, `sessions_spawn(label: "researcher", ...)` only uses the label as metadata/prompt context and never resolves `subagents/researcher.md`.

## Solution Assessment

Unclear because there is no proposed patch. The requested direction is plausible, but the best maintainable solution needs product/API and security policy decisions for lookup scope, precedence, model override trust, and whether role-file tool policy may only narrow or also broaden capabilities.
