## Summary

Keep open. The OpenAI-compatible max-token fallback addresses a real current-main gap, but this PR is not merge-ready because the QMD change makes the documented `onSessionStart` sync contract a no-op for QMD and leaves existing QMD tests expecting the removed update path; it also needs the required user-facing changelog entry.

## What This Changes

The branch falls OpenAI-compatible Responses/Completions max-token params back to `model.maxTokens` and removes QMD session-start sync from `maybeWarmSession`.

## Best Possible Solution

Keep the max-token fallback with regression coverage, then resolve the QMD reader/writer collision without silently breaking the documented session-start sync contract, or explicitly update the contract if maintainers choose that behavior.

## Reproduction Assessment

Yes, by source inspection: current main omits `model.maxTokens` in both OpenAI-compatible builders, and the PR removes the QMD session-start update path while current tests still require it. I could not run the tests because `pnpm` is unavailable.

## Solution Assessment

No. The transport fallback is a narrow fix, but the QMD change is not the best complete solution until it preserves or intentionally revises the documented `onSessionStart` behavior and updates tests/changelog accordingly.
