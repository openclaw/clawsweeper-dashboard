## Summary

Keep open. The OpenAI-compatible max-token fallback is a valid narrow fix, but this PR is not merge-ready because the QMD hunk removes documented and tested session-start sync behavior instead of making the update/query race reader-safe.

## What This Changes

This PR forwards `model.maxTokens` into OpenAI-compatible stream payloads when request options omit `maxTokens` and removes QMD session-start sync from first-search warmup.

## Best Possible Solution

Split or land the OpenAI-compatible max-token fallback with regression coverage and changelog, then fix the QMD race by coordinating update/query access while preserving configured session-start freshness.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current `main` omits `model.maxTokens` in both OpenAI-compatible builders when options are absent, and the PR's QMD hunk would make the documented `onSessionStart` path a no-op.

## Solution Assessment

No. The max-token fallback is narrow, but disabling QMD session-start sync is not the best fix for a race; the reader/writer interaction should be serialized, deferred, or otherwise made safe while preserving the documented hook.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve QMD session-start sync behavior:** `extensions/memory-core/src/memory/qmd-manager.ts:1587-1594`

Please fix the reader/writer race without making the configured session-start hook a no-op, or change the documented contract with maintainer signoff.
