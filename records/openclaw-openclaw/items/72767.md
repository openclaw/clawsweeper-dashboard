## Summary

Keep open. Current main still does not implement `agent.capabilities`, and the PR is a reasonable additive Gateway RPC, but the latest head still misreports inherited agent skills and lacks the required changelog entry for a user-facing feature.

## What This Changes

The PR adds a READ-scoped `agent.capabilities` Gateway RPC with protocol schemas, generated Swift models, method registration, and a handler summarizing an agent's model, tools, MCP servers, skills, bound channels, and session count.

## Best Possible Solution

Ship the additive Gateway RPC after it reports skills through the existing effective agent skill-filter contract, covers inherited defaults, per-agent replacement, and explicit empty skills in focused tests, and records the user-facing method in the Unreleased changelog.

## Reproduction Assessment

Yes. Configure `agents.defaults.skills: ["github"]` and an agent entry that omits `skills`; the PR handler takes the resolved config object's own `skills: undefined` property and returns `[]` instead of the inherited default.

## Solution Assessment

No. The RPC direction is maintainable, but the skill portion should reuse `resolveAgentSkillsFilter` or the same raw-entry effective helper instead of inspecting the resolved config object's own properties.
