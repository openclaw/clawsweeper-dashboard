## Summary

Keep open. The additive Gateway RPC is useful and current main does not already provide `agent.capabilities`, but the PR still misreports inherited agent skills and is missing the required changelog entry for a user-facing Gateway feature.

## What This Changes

The PR adds a READ-scoped `agent.capabilities` Gateway RPC with protocol schemas/generated Swift models and a handler that summarizes model, tool, MCP, skill, channel, and session-count data for an agent.

## Best Possible Solution

Ship the additive Gateway RPC after it reports skills through the existing effective skill-filter contract, covers inherited defaults and explicit replacement in focused tests, and records the user-facing method in the Unreleased changelog.

## Reproduction Assessment

Yes. With `agents.defaults.skills: ["github"]` and an agent entry that omits `skills`, the PR handler takes the resolved config object's own `skills: undefined` property and returns an empty list, while current helper tests expect inherited defaults.

## Solution Assessment

No. The RPC direction is maintainable, but the skill portion should reuse `resolveAgentSkillsFilter` or the effective helper instead of inspecting the resolved config object's own properties.
