## Summary

Keep open: current main still routes classic voice-call replies through a fully awaited embedded run, so post-turn compaction can delay TTS even after the related `tools.allow` fix is present.

## What This Changes

The issue requests that voice-call deliver the embedded model’s spoken response to TTS before post-turn compaction blocks realtime playback.

## Best Possible Solution

Use the existing embedded-runner block-reply/flush seam to hand parsed `{spoken}` output to voice-call TTS before compaction waits, while preserving transcript persistence and avoiding duplicate speech.

## Reproduction Assessment

No, not as a live Twilio/Ollama current-main reproduction in this review. The reporter provided live logs, and current source still shows a deterministic blocking path from `handleInboundResponse()` through the fully awaited embedded run before TTS.

## Solution Assessment

Yes, the narrow maintainable direction is early voice delivery through the embedded runner’s existing reply callback/flush path. A global compaction disable or new user-facing config would be a broader policy change than this bug needs.
