## Summary

Keep open: current main has the prerequisite voice agent tool-allowlist fix, but the reported voice-call latency path remains source-backed because TTS is still reached only after `generateVoiceResponse()` returns from the full embedded run, and the embedded runner still waits for post-turn compaction unless a block-reply flush path is used.

## What This Changes

The issue asks for voice-call embedded responses to be spoken as soon as the model reply is available, without waiting for post-turn compaction.

## Best Possible Solution

Add a narrow voice-call early-delivery handoff, likely by using the embedded runner block-reply/flush callback path or an equivalent voice-local callback, so spoken text reaches TTS before the compaction wait while final result cleanup remains intact.

## Reproduction Assessment

Yes for source-level reproduction, though not as a live Twilio run. Current main now includes the `tools.allow` prerequisite fix, and the remaining call path still awaits the full embedded run before `manager.speak()` while the runner waits on post-turn compaction when no early flush callback is wired.

## Solution Assessment

Yes, a narrow early handoff is the best direction. It should not disable compaction globally or change tool-policy semantics; it should deliver the voice reply before the existing compaction wait and cover that ordering with focused tests.
