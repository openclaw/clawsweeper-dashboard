## Summary

Keep open. Current main and the latest release still have voice-call consume only the completed embedded-run payloads after the runner's compaction wait, and https://github.com/openclaw/openclaw/pull/94015 is an open unmerged candidate fix rather than landed behavior.

## What This Changes

The issue reports that classic voice-call waits for post-turn embedded-agent compaction before handing the generated spoken response to TTS, causing long call latency and possible session-context contamination.

## Best Possible Solution

Land a narrow voice-call fix that uses the existing embedded-runner early reply callback seam or an equivalent nonblocking maintenance policy, with regression coverage, then close this issue when that fix is merged.

## Reproduction Assessment

Yes, at source level. Current main still has voice-call await the completed embedded-agent result and read result.payloads, while the runner waits on compaction retry before returning; I did not replay a live Twilio call.

## Solution Assessment

Yes, the best fix direction is narrow: have voice-call consume early block replies before post-turn compaction blocks the returned result. The broader sentence-level streaming TTS work in https://github.com/openclaw/openclaw/issues/8355 is related latency work but not required to solve this bug.
