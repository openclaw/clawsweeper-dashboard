## Summary

Keep open: the report is consistent with current main source. Voice-call waits for `generateVoiceResponse()`, that function awaits the embedded runner result, and the runner still waits for compaction retry/maintenance after the assistant response path unless a streaming/block flush callback has already delivered the reply.

## What This Changes

The issue asks for voice-call embedded responses to hand assistant text to TTS before post-turn compaction completes and to avoid contaminating later voice context.

## Best Possible Solution

Keep the issue open and implement a narrow voice-call low-latency handoff that delivers spoken output before post-turn compaction while preserving safe transcript/compaction cleanup.

## Reproduction Assessment

No full current-main live reproduction was run. The supplied live Twilio/Ollama logs plus current source make the blocking path source-reproducible, but current main also lacks the prerequisite tools-allow fix from https://github.com/openclaw/openclaw/pull/79508.

## Solution Assessment

No current fix is present on main. The safest direction is to deliver the voice reply through an early callback or equivalent voice-call handoff before the compaction wait, not to disable or globally weaken compaction.
