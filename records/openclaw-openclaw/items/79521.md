## Summary

Keep open: the current main source still matches the reported latency path. Voice-call waits for the full embedded run result before calling TTS, while the embedded runner can still wait on compaction retry/maintenance after the assistant response is available.

## What This Changes

The issue requests that voice-call deliver the embedded model response to TTS before post-turn compaction can add long latency or contaminate later voice context.

## Best Possible Solution

Add a narrow voice-call low-latency handoff that delivers the spoken assistant output before post-turn compaction waits, while keeping transcript persistence and compaction cleanup safe for later turns.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main phone-call reproduction was run, but the reporter's real logs plus current source make the blocking path source-reproducible.

## Solution Assessment

Is this the best way to solve the issue? The requested direction is sound, but the safest fix is a narrow early voice delivery callback or equivalent handoff rather than disabling compaction globally or adding a new config knob.
