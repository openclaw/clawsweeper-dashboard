## Summary

Keep open: current main still has the blocking path. Voice-call awaits the full embedded run before speaking, while the embedded runner still waits for post-turn compaction/retry cleanup after the assistant response unless a flush callback has already delivered the reply.

## What This Changes

The issue reports that voice-call waits for embedded-run post-turn compaction before handing a generated response to TTS, causing long phone-call latency and polluted voice context.

## Best Possible Solution

Add a narrow voice-call low-latency handoff that delivers the spoken response before post-turn compaction waits, using the existing embedded-runner flush/streaming seam or a small equivalent callback while preserving transcript and compaction cleanup.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: mock or instrument the voice-call embedded run so the assistant payload is available before a delayed compaction wait, then observe that manager.speak() cannot run until generateVoiceResponse() resolves. A full Twilio/Ollama live repro was not rerun in this review.

## Solution Assessment

No current fix is present on main. The maintainable fix is a narrow early-delivery path for voice-call responses, not a global compaction timeout change or disabling compaction for all embedded runs.
