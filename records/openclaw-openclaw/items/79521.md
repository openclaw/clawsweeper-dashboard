## Summary

Keep open. Current main and the latest release still make voice-call wait for the completed embedded-agent result while the existing early block-reply path is not wired into `generateVoiceResponse`; the open closing PR is the right candidate path but has not landed.

## What This Changes

Issue reports that voice-call waits for embedded-agent post-turn compaction before returning text to TTS, causing long call latency and voice-session context contamination.

## Best Possible Solution

Refresh, review, and land or replace the open voice-call early-reply PR so `generateVoiceResponse` consumes `onBlockReply` text before compaction waits while preserving the existing embedded-runner compaction behavior for other callers.

## Reproduction Assessment

Yes. Source inspection shows `generateVoiceResponse` waits for `runEmbeddedAgent()` to finish and reads only aggregate payloads, while the embedded runner has an early block-reply callback path that can flush text before compaction waits; I did not run a live Twilio call in this read-only review.

## Solution Assessment

Yes, the best direction is to reuse the existing embedded-agent `onBlockReply`/`blockReplyBreak` contract in voice-call rather than adding a voice-specific compaction bypass. The open PR appears to target that seam, but it needs refresh and proof before merge.
