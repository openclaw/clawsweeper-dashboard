## Summary

Keep open: current main still makes voice-call wait for the full embedded run before speaking, while the embedded runner still waits for post-turn compaction unless an early block-reply flush path is wired. The prerequisite tools-allow bug is now fixed on main, but this separate latency path remains source-backed.

## What This Changes

The issue asks OpenClaw to hand voice-call/TTS the embedded model's spoken response before post-turn compaction can add phone-call latency or contaminate later voice context.

## Best Possible Solution

Implement a narrow voice-call early handoff that delivers the parsed spoken payload before the post-turn compaction wait, while leaving global compaction behavior intact and adding regression coverage for the delayed path.

## Reproduction Assessment

No high-confidence live current-main reproduction was executed. The issue is source-reproducible because voice-call awaits generateVoiceResponse(), generateVoiceResponse() awaits the embedded runner, and the runner still awaits compaction unless an early block-reply flush path is supplied.

## Solution Assessment

Yes, the requested direction is the right fix boundary. The safest implementation is an early voice-call delivery callback or equivalent handoff before compaction wait, not disabling or weakening compaction globally.
