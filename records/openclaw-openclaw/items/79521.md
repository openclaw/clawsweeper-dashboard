## Summary

Keep open: the report has credible live after-fix voice-call evidence, and current source still routes voice-call TTS through the embedded runner's final return value while the runner waits on post-turn compaction. The earlier generic delivery fix only flushes block-reply callbacks, which this voice-call path does not use.

## What This Changes

The issue asks for voice-call embedded responses to reach TTS before post-turn compaction can add 20s+ latency or contaminate the voice session.

## Best Possible Solution

Extend the existing deliver-before-compaction invariant to the voice-call TTS path, preferably by exposing the completed spoken response before the compaction wait or by applying a narrow low-latency voice-run compaction policy without changing global compaction behavior.

## Reproduction Assessment

Yes for the post-tools.allow-fix path: the reporter supplied live Twilio/Ollama logs, and source inspection shows voice-call waits on `runEmbeddedPiAgent()` while that runner waits for compaction. I did not run a live Twilio call against current main, and current main still needs the separate tools.allow fix to reach the same symptom cleanly.

## Solution Assessment

Yes, the requested direction matches the already-established delivery-before-compaction behavior from the prior channel fix. The safest implementation should be narrow to voice-call or generic result delivery, not a broad compaction rewrite or new user config knob.
