## Summary

Keep open: current main includes the prerequisite routed-agent `tools.allow` fix, but the central voice latency bug still appears present because voice-call waits for the full embedded run before speaking and does not use the runner’s pre-compaction reply-flush path.

## What This Changes

The issue reports that voice-call embedded responses wait for post-turn compaction before TTS receives the already-generated spoken response, causing long call latency and polluted voice context.

## Best Possible Solution

Implement a voice-call low-latency delivery path that speaks the parsed spoken output through an early embedded-runner callback before compaction retry or afterTurn waits, while keeping transcript persistence and compaction cleanup safe and nonblocking for TTS.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. The reporter supplied live Twilio/Ollama logs, and current source still makes the blocking path source-reproducible now that the related tool-allowlist prerequisite is on main.

## Solution Assessment

Yes, the best direction is a narrow early voice handoff before the embedded runner waits on compaction, not a global compaction disable or a new user-facing configuration knob.
