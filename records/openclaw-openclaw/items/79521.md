## Summary

Keep open: current main still has the blocking shape described by the report. The tools.allow prerequisite is now present, but voice-call still waits for the full embedded run result before calling TTS, and the embedded runner still waits on post-turn compaction after the assistant response path unless a block-reply delivery hook is used.

## What This Changes

Voice-call should hand the spoken response to TTS before post-turn compaction blocks the embedded runner.

## Best Possible Solution

Add a narrow voice-call low-latency delivery path that hands the spoken assistant text to TTS before post-turn compaction waits, while preserving transcript persistence and safe compaction cleanup.

## Reproduction Assessment

Yes for source-level reproduction, but not for a full live call in this review. Current main still has the exact await chain from webhook to response generator to embedded runner compaction wait, and the reporter supplied live Twilio/Ollama logs after the tools.allow prerequisite fix.

## Solution Assessment

Yes, the maintainable direction is a focused early voice delivery hook or equivalent voice-call handoff before compaction waits. Globally disabling or weakening compaction would be broader and riskier than fixing the voice-call delivery timing.
