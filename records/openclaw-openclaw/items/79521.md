## Summary

Keep open: current main includes the prerequisite voice `tools.allow` forwarding fix, but the voice-call path still awaits the full embedded run before calling TTS, and the embedded runner still waits on post-turn compaction unless an early block-reply flush callback is provided.

## What This Changes

The issue requests that voice-call conversation mode speak the embedded model response before post-turn compaction completes, avoiding 20s+ call latency and unrelated compaction context leakage.

## Best Possible Solution

Implement a narrow voice-call early spoken-output handoff before the embedded-run compaction wait, using the existing block-reply flush seam where possible while preserving transcript and compaction cleanup.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main awaits the embedded result before TTS, and the embedded result path awaits compaction unless an early flush callback is provided. I did not run the live Twilio/Ollama call path.

## Solution Assessment

No current fix is present on main. The best direction is a scoped early voice delivery callback or equivalent handoff before the compaction wait, not disabling compaction globally.
