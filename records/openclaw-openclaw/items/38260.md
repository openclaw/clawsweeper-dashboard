## Summary

Keep open: current main still has a WebChat/media image path that can reach in-process sharp/libvips, and the shipped fallback only handles thrown optional-optimizer errors, not a process-terminating SIGILL. The related work at https://github.com/openclaw/openclaw/pull/74243 explicitly excluded this native crash, and https://github.com/openclaw/openclaw/pull/75732 remains open for a different Sharp/SSE2 compatibility path.

## What This Changes

The report requests preventing openclaw-gateway from core-dumping with SIGILL in libvips-cpp during a WebChat image-message flow.

## Best Possible Solution

Keep this as the native libvips/SIGILL crash tracker until maintainers reproduce on latest release or current main, collect coredump/backtrace plus CPU and binary details, then choose a guarded dependency, backend, packaging, or process-isolation fix with focused WebChat image-flow coverage.

## Reproduction Assessment

No high-confidence crash reproduction is available. Source inspection shows the reported WebChat image flow can still reach in-process sharp/libvips, but reproducing the SIGILL needs the original payload, CPU flags, loaded binary paths, and a coredump or gdb backtrace on current main or the latest release.

## Solution Assessment

No safe fix is established yet. The shipped optional-optimizer fallback is useful for thrown sharp availability errors, but a process-terminating SIGILL needs crash proof before maintainers choose packaging, backend fallback, or process isolation.
