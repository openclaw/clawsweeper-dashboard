## Summary

Keep open: current main still has a WebChat image path that can reach in-process sharp/libvips, and the merged fallback only handles thrown missing-optimizer errors, not a process-terminating SIGILL. The related optimizer-fallback work at https://github.com/openclaw/openclaw/pull/74243 explicitly excluded this native crash, and the open SSE2 fallback at https://github.com/openclaw/openclaw/pull/75732 targets a different Sharp load-failure issue.

## What This Changes

The issue reports that openclaw-gateway exits with a SIGILL/core dump in libvips-cpp after a WebChat image-flow conversation.

## Best Possible Solution

Keep this as the native libvips SIGILL crash tracker, collect current-release crash proof, then implement a guarded backend, dependency, or process-isolation fix with WebChat image-flow regression coverage.

## Reproduction Assessment

No, there is no high-confidence way to reproduce the crash itself yet. Source inspection shows the reported WebChat image flow can still reach in-process sharp/libvips, but reproducing the SIGILL needs the original payload, CPU flags, loaded binary paths, and a coredump or gdb backtrace on current main or the latest release.

## Solution Assessment

No safe fix is established yet. The shipped optional-optimizer fallback is useful for thrown Sharp availability errors, but a process-terminating SIGILL needs crash proof before maintainers choose packaging, backend fallback, or process isolation.
