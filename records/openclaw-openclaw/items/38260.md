## Summary

Keep open: current main still has a WebChat/media image path that can reach in-process sharp/libvips, and the shipped fallback only handles thrown optimizer-unavailable errors, not a process-terminating SIGILL. The related open work at https://github.com/openclaw/openclaw/pull/75732 targets a similar Sharp/SSE2 compatibility family but does not close this native libvips crash.

## What This Changes

The issue asks OpenClaw to stop `openclaw-gateway` from crashing with SIGILL in `libvips-cpp` during a WebChat image conversation flow.

## Best Possible Solution

Keep this as the native libvips/SIGILL crash tracker until maintainers reproduce on latest release or current main, collect coredump/backtrace plus CPU and binary details, then choose a dependency, backend, packaging, or process-isolation fix with focused WebChat image-flow coverage.

## Reproduction Assessment

No. Source inspection shows the reported WebChat image flow can still reach in-process sharp/libvips on current main, but a high-confidence current-main SIGILL reproduction needs the original payload, CPU details, loaded binary paths, and a coredump or gdb backtrace.

## Solution Assessment

No current safe fix is established. The thrown-error optimizer fallback is useful for missing or unavailable sharp, but a process-terminating SIGILL needs crash proof before choosing packaging, backend fallback, or isolation work.
