## Summary

Keep open: current main and the latest release have useful unavailable-processor fallbacks, but the default image path is still Sharp-first and in-process, so a native libvips SIGILL can still terminate the gateway before JavaScript fallback handling runs.

## What This Changes

Prevent openclaw-gateway from crashing with SIGILL in libvips during WebChat image attachment processing.

## Best Possible Solution

Keep this as the native image-processing crash tracker until current crash artifacts identify whether the right fix is backend selection, CPU/package gating, or isolating Sharp/libvips work outside the gateway process.

## Reproduction Assessment

No high-confidence way to reproduce the crash yet. Source inspection shows current main still routes WebChat image refs into in-process Sharp/libvips, but current crash proof needs the triggering image or sanitized reproducer plus coredump/backtrace and CPU flags.

## Solution Assessment

Unclear. The current fallback chain is useful for thrown missing/unavailable processor errors, but a native SIGILL likely needs a maintainer decision about backend defaulting, packaging/CPU gating, or process isolation.
