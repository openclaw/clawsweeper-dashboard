## Summary

Current main and shipped releases no longer use the in-process Sharp/libvips image backend named in the SIGILL crash, so the central gateway crash path is already removed.

## What This Changes

The issue reports a gateway SIGILL/core dump in libvips during a webchat image message flow on v2026.2.22-2.

## Best Possible Solution

Keep the shipped Rastermill/Photon image backend; any current-release gateway crash should be reported as a new issue with the exact image, CPU flags, selected backend, package architecture, and native stack trace.

## Reproduction Assessment

No current-main reproduction remains for the reported in-process libvips path because Sharp/libvips is no longer in the core image backend. The historical crash is credible from the supplied kernel and systemd logs, but I did not run the old v2026.2.22-2 image flow.

## Solution Assessment

Yes; replacing the in-process Sharp/libvips backend is the narrowest durable fix for a fatal native library crash named in the report, and current code keeps image limits and media coverage on the replacement path.
