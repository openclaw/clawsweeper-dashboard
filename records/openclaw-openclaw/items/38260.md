## Best Possible Solution

Keep this as the native image-processing crash tracker until fresh crash artifacts show whether the fix belongs in backend defaulting, packaging/CPU gating, or isolating Sharp/libvips work outside the gateway process.

No high-confidence way to reproduce the SIGILL itself yet. Source inspection does show the reported WebChat image flow can still reach in-process Sharp/libvips on current main and v2026.5.20, but the crash needs a triggering image or sanitized reproducer plus coredump/backtrace and CPU details.

## Solution Assessment

Unclear. The current fallback and OPENCLAW_IMAGE_BACKEND path are useful for thrown optimizer or missing-backend failures, but a native invalid-opcode crash likely needs a maintainer decision about backend policy, packaging/CPU compatibility, or process isolation.
