Keep open: current main still has an in-process WebChat/media image path that reaches sharp/libvips, and the related optimizer-unavailable fallback only handles JavaScript errors, not process-terminating SIGILL.

## What This Changes

The issue reports a native SIGILL/libvips crash in openclaw-gateway after a WebChat image message followed by a short prompt.

## Best Possible Solution

Keep this as the native libvips crash tracker until maintainers reproduce on latest release or current main, collect coredump/backtrace plus CPU and binary details, then choose a dependency, backend, packaging, or process-isolation fix with focused WebChat image-flow coverage.

No. Source inspection shows the reported WebChat image flow can still reach in-process sharp/libvips, but a high-confidence current-main SIGILL reproduction needs the original payload, CPU/libvips binary details, and a crash backtrace.

## Solution Assessment

No current safe fix is established. The optimizer-unavailable fallback is useful for thrown JavaScript errors, but it cannot catch a process-terminating SIGILL; the fix should follow crash proof rather than guessing between packaging, backend, or isolation changes.
