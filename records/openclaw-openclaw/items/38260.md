## Summary

Keep open: current main and v2026.5.20 still route the reported WebChat image path through in-process Sharp/libvips by default, and the existing JavaScript fallback only handles thrown optimizer-unavailable errors, not a native SIGILL that terminates the gateway process.

## What This Changes

The issue reports a gateway SIGILL/core dump in libvips during a WebChat image attachment follow-up flow.

## Best Possible Solution

Keep this as the native image-processing crash tracker until fresh crash artifacts show whether the fix belongs in backend defaulting, package/CPU gating, or isolating Sharp/libvips work outside the gateway process.

## Reproduction Assessment

No high-confidence way to reproduce the SIGILL itself yet. Source inspection does show the reported WebChat image flow can still reach in-process Sharp/libvips on current main and v2026.5.20, but the crash needs a triggering image or sanitized reproducer plus coredump/backtrace and CPU details.

## Solution Assessment

Unclear. The current fallback and OPENCLAW_IMAGE_BACKEND path are useful for thrown optimizer or missing-backend failures, but a native invalid-opcode crash likely needs a maintainer decision about backend policy, packaging/CPU compatibility, or process isolation.
