## Summary

Keep open: current main has not solved the linked heartbeat/model-initialization diagnostic issue, and this PR remains a meaningful candidate, but it is not merge-ready because it adds a second replay-visible sentinel without updating all runtime consumers and still does not classify the provider/Ollama failure modes reported by the issue.

## What This Changes

The PR adds a `[model initialization failed]` assistant-content sentinel, passes an `isModelInitError` flag through embedded-runner unhandled-stop-reason recovery, and updates replay-history trimming for both sentinel strings.

## Best Possible Solution

Classify provider/model initialization failures at the provider, fallback, or heartbeat boundary and use one shared sentinel predicate for replay, prompt, gateway display, OpenAI-compatible conversion, and heartbeat delivery suppression.

## Reproduction Assessment

Yes at source level for the PR blockers: current main has old-sentinel-only consumers and the PR's model-init path is gated by the `Unhandled stop reason:` matcher. I did not reproduce the live heartbeat/Ollama-cloud provider failure in this read-only review.

## Solution Assessment

No. The patch is a plausible diagnostic attempt, but the best fix should classify the actual provider/fallback/heartbeat failure boundary and keep internal sentinel handling shared across all runtime consumers.
