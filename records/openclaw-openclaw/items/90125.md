## Summary

Keep open: the PR points at a real diagnostic problem, but it still emits a new replay-visible sentinel without updating all current consumers, and its model-init classification only covers `Unhandled stop reason:` stream-wrapper failures rather than the timeout/rate-limit/Ollama-proxy provider failures described by the linked issue.

## What This Changes

The PR adds a second stream-error sentinel for model initialization failures, threads an `isModelInitError` flag through unhandled-stop-reason recovery, and updates replay-history tail detection.

## Best Possible Solution

Classify provider/model initialization failures at the existing provider/fallback/heartbeat boundary, share one replay-sentinel predicate across gateway and replay consumers, and land with focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes at source level for the PR blockers: the patch emits a second sentinel while current prompt/gateway consumers recognize only the old sentinel, and the new model-init branch is gated by the `Unhandled stop reason:` matcher. ## Solution Assessment

No. The patch is a plausible small mitigation, but the best fix belongs at the provider/fallback/heartbeat failure boundary and needs a shared sentinel predicate across all replay and prompt consumers.
