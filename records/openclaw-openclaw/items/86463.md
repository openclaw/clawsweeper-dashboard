## Summary

Keep this PR open. The warn-only mitigation is a reasonable conservative direction, but the current patch can emit false stranded-reply warnings from appended metadata and the new QA scenario does not actually assert the WARN it claims to prove.

## What This Changes

The PR adds a stranded `message_tool_only` reply guard/logger, wires it into `runReplyAgent`, and adds unit/integration coverage plus a qa-channel scenario.

## Best Possible Solution

The end state should be a narrow warn-only implementation that keys off actual assistant final text, proves the WARN in a live gateway log, and keeps recovery semantics for the linked issue as an explicit maintainer decision.

## Reproduction Assessment

Yes. The source path is clear: `message_tool_only` suppresses automatic final delivery, and a non-empty final payload with no successful message-tool delivery is currently dropped from the source channel.

## Solution Assessment

No, not yet. Warn-only observability may be an acceptable conservative mitigation, but this implementation should distinguish actual assistant reply text from appended metadata and the QA proof should assert the WARN before merge.

- **[P2] Detect stranded replies before metadata appends:** `src/auto-reply/reply/agent-runner.ts:2144-2155`

That was the pre-existing silent-drop behavior, so this scenario can pass even if the warning path is removed. Add a `markGatewayLogCursor()` plus `readGatewayLogs()` assertion for `source-reply/stranded` after the inbound turn.
  - confidence: 0.9
