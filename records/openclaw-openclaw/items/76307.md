## Summary

Keep open. The fresh regression still matches current main: the OpenAI completions path can run under a guarded provider fetch timeout, partial assistant text is converted into user-facing payloads, and the explicit timeout fallback only handles zero-payload timeouts; the merged session-repair PR is complementary rather than a fix for this stream-abort path.

## What This Changes

The issue reports deterministic long OpenClaw agent replies being cut mid-word after a fetch timeout even though the same OpenAI model and prompt complete through a direct API call.

## Best Possible Solution

Fix the shared provider streaming timeout path so active SSE chunks do not produce a false total-body timeout, and ensure timeout-after-partial text is retried or surfaced as an error instead of a final truncated reply.

Yes, for source-level triage: the issue gives concrete CLI steps/logs and current main shows a configured provider fetch timeout can abort an OpenAI stream after partial chunks. I did not run the live OpenAI/WSL reproduction in this read-only pass.
