## Summary

Keep open. The report is a fresh, detailed regression with a credible cross-channel reproduction, and current main still has timeout paths where partial assistant text can become the visible payload instead of a completed response or explicit timeout failure.

## What This Changes

Investigate deterministic truncation of long agent replies across CLI, Discord, and Telegram on OpenClaw 2026.4.29 with large bootstrap context and OpenAI Chat Completions.

## Best Possible Solution

Fix the shared embedded runner/provider timeout path so a long OpenAI stream either completes with the provider's natural stop reason or is retried/reported as a timeout instead of delivering a partial fragment as the final reply.

Yes, for triage: the issue gives concrete OpenClaw CLI steps, heavy bootstrap conditions, channel confirmations, and a direct OpenAI API control. A high-confidence automated regression should mock a timed-out run with non-empty `assistantTexts` to capture the current partial-fragment path.

## Solution Assessment

Unclear. The current implementation is not sufficient because it only synthesizes timeout errors when no assistant payload exists; the narrow maintainable fix should start in the shared runner/provider timeout handling, not in Discord or Telegram chunking.

The observable bug is that long agent replies can abort after a small assistant fragment and that fragment is delivered as if it were the final answer across CLI/Discord/Telegram. Add a regression test where `runEmbeddedAttempt` returns `timedOut: true` or `idleTimedOut: true`, no tool execution/compaction timeout, and non-empty `assistantTexts` such as a mid-line MEMORY.md fragment; assert the final payload is not treated as a successful partial answer and that timeout/retry metadata is preserved.
