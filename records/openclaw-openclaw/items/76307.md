## Summary

Keep open. This is a fresh, detailed regression report, and current main still has a shared timeout path where a timed-out run with non-empty assistant text can be returned as a normal partial reply instead of an explicit timeout/retry outcome.

## What This Changes

The issue asks OpenClaw to stop deterministically truncating long agent replies after tiny fragments across CLI, Discord, and Telegram when the direct provider call completes normally.

## Best Possible Solution

Fix the shared embedded runner/provider timeout path so timeout-after-partial assistant output is retried or reported as a timeout, not delivered as a completed reply.

## Reproduction Assessment

Yes for triage. The issue gives concrete user steps and controls, and a high-confidence regression can mock a timedOut attempt with non-empty assistantTexts to reproduce the current partial-fragment path without live provider access.

## Solution Assessment

Yes, the narrow maintainable direction is the shared runner/provider timeout handling rather than Discord or Telegram chunking. The fix should preserve intentional user-abort partials while preventing timeout fragments from becoming successful replies.
