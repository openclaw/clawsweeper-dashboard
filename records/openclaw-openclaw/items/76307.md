## Summary

Keep open. This is a fresh, detailed regression report, and current `main` still has the shared timeout-after-partial-output path described in the issue: the guarded OpenAI completions fetch can abort an active stream on wall-clock timeout, while accumulated assistant text can still be retained and delivered as the visible reply.

## What This Changes

The issue reports deterministic truncation of long OpenAI-backed agent replies after small partial fragments across Discord, Telegram, and agent turns under heavy bootstrap context.

## Best Possible Solution

Fix the shared streaming timeout semantics so active SSE streams are not aborted solely because request wall-clock age exceeds the configured timeout, and ensure timeout-after-partial-output is retried or reported as incomplete rather than delivered as a final answer.

## Reproduction Assessment

Yes for triage from source and report evidence: the issue gives concrete heavy-bootstrap steps, and current `main` has a clear timeout-after-partial-output failure path. I did not run the live OpenAI/WSL2 scenario, so this is source-reproducible rather than directly reproduced here.

## Solution Assessment

Unclear until a patch is proposed, but the narrow maintainable direction is shared provider fetch/OpenAI stream timeout handling plus runner regression coverage. A Discord or Telegram chunking change would not address the reported channel-agnostic path.
