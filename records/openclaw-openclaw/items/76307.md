## Summary

Keep open. The report is fresh, detailed, and source inspection of current main still shows a shared timeout path where a timed-out run with non-empty assistant text can be surfaced as the final user-visible reply instead of a completed response or explicit timeout error.

## What This Changes

Long agent replies should complete normally or report an explicit timeout instead of exposing tiny mid-line fragments across CLI, Discord, and Telegram.

## Best Possible Solution

Add a shared runner/provider regression for timeout-after-partial-assistant-text and make that path retry or surface an explicit timeout instead of finalizing the fragment.

## Reproduction Assessment

Yes, as a source-level reproduction path: the issue gives concrete CLI/live steps and current main has a mocked-testable path where `timedOut` plus non-empty `assistantTexts` bypasses timeout synthesis. I did not execute a live OpenAI run in this read-only pass.

## Solution Assessment

Unclear for the exact root trigger, but channel-specific fixes are not the best direction. The maintainable fix should start in shared embedded-runner timeout/failover/payload handling with regression coverage for non-empty partial output.
