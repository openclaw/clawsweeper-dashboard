## Summary

Keep open. Current main still has the source path described in the report and follow-up discussion: an OpenAI-compatible stream can collect partial assistant text, hit the guarded fetch timeout, and then build a visible reply from the partial text because the explicit timeout payload only covers the zero-payload case; #76420 is complementary session-repair work, not a fix for this first-delivery truncation.

## What This Changes

Long-output agent replies should complete naturally or fail with an explicit timeout instead of surfacing tiny deterministic partial fragments across channels.

## Best Possible Solution

Long streaming replies should either finish with the provider's natural stop reason or produce an explicit timeout/retry failure, with regression coverage for non-empty partial text followed by a timeout.

## Reproduction Assessment

Yes, source-reproducible with high confidence: a focused test can mock an OpenAI-compatible stream that emits assistant text and then aborts from the guarded fetch timeout. I did not run a live OpenAI reproduction in this read-only review, but the current source and existing test gap match the report.

## Solution Assessment

Yes: the maintainable fix belongs in shared provider/runner timeout handling, not in Discord or Telegram chunking. A safer patch should first make timeout-after-partial-output explicit, then decide whether streaming progress should reset the provider request timeout and update docs if that semantic changes.
