## Summary

Keep this draft PR open: the `chat.inject` user-role/idempotency direction addresses a real Gateway gap, but the current branch still drops unrelated current-main Gateway behavior around stop matching, provider-auth aborts, image-model routing, TTS projection, and public protocol docs. It also overlaps the open Gateway metadata API work, so the final additive contract should be coordinated before merge.

## What This Changes

This PR extends Gateway `chat.inject` with optional `role` and `idempotencyKey`, adds idempotent transcript append behavior, updates Swift Gateway models, and expands transcript append tests.

## Best Possible Solution

Land one additive Gateway `chat.inject` contract that supports user-role/idempotent replay while preserving current stop matching, provider-auth aborts, image-model routing, TTS projection, generated clients, and public docs.

## Reproduction Assessment

Yes. Current main’s schema rejection of the new fields and the PR-introduced regressions are source-reproducible by comparing current main with PR head `09ec74efdd6dce5f11d2f01c8e064f7666ff1153`; no live test was run during this read-only review.

## Solution Assessment

No. The locked idempotency approach is reasonable, but this branch is not the best merge shape until it restores unrelated current-main Gateway behavior and coordinates the public protocol with the overlapping metadata PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve canonical session matching for stop commands:** `src/gateway/server-methods/chat.ts:1609`
  - body: The branch now resolves abort candidates only when `active.sessionKey === params.sessionKey`. Current main also matches canonical aliases and the backing `sessionId`, so a stop sent through an alias such as `main` can miss the active run and fail to abort or persist the partial.
  - confidence: 0.93
- **[P1] Keep provider metadata on active chat runs:** `src/gateway/server-methods/chat.ts:2216`
  - body: The `registerChatAbortController` call no longer records `providerId` or `authProviderId`, and the selected-model callback no longer updates those fields after fallback. `abortChatRunsForProvider` relies on this metadata, so provider logout or credential revocation can leave active chat sends running.
  - confidence: 0.92
- **[P1] Keep the configured image-model override path:** `src/gateway/server-methods/chat.ts:2134-2145`
  - body: Replacing `resolveImageModelOverridePlan` with a session-model-only check ignores `agents.defaults.imageModel`. Text-only sessions with image attachments can no longer switch to the configured vision model and will route those images as media paths instead.
  - confidence: 0.94
- **[P2] Preserve TTS supplement markers on media replies:** `src/gateway/server-methods/chat-transcript-inject.ts:156-165`
  - body: The refactor removes the `ttsSupplement` parameter from injected assistant transcript appends, so delayed media-only TTS replies are persisted without `openclawTtsSupplement`. `projectRecentChatDisplayMessages` uses that marker to merge audio into the original assistant row, so history can show a separate audio assistant message.
  - confidence: 0.88
- **[P2] Document the expanded chat.inject contract:** `src/gateway/protocol/schema/logs-chat.ts:69-70`
  - body: The schema adds `role` and `idempotencyKey`, but the public Gateway/WebChat docs still describe `chat.inject` as assistant-only transcript injection. Please document user-role appends, idempotent replay responses, returned message IDs, and duplicate-broadcast behavior.
  - confidence: 0.86
