## Summary

Keep open: this PR targets a real, still-open Gateway protocol feature request, but it needs after-fix real behavior proof and the new idempotency path is not concurrency-safe enough to merge as-is.

## What This Changes

The PR extends Gateway `chat.inject` with optional `role` and `idempotencyKey`, adds a user-role transcript append path, treats deduped replays as successful, and adds unit coverage for user transcript appends.

## Best Possible Solution

Land an additive, documented `chat.inject` replay contract that supports user and assistant synthetic turns with durable, atomic idempotency through the transcript append pipeline.

## Reproduction Assessment

Not applicable as a bug report. The current feature gap is source-verifiable: current main exposes a closed assistant-only `chat.inject` schema and handler.

## Solution Assessment

No. The direction matches the related request, but idempotency needs to be atomic under concurrent retries and the public protocol/docs surface should be updated before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Make chat.inject idempotency atomic:** `src/gateway/server-methods/chat.ts:1466-1470`
  - body: The new `chat.inject` idempotency check scans the transcript before the append path takes the transcript write lock. Two retries with the same key can both scan before either write lands, then serialize through `appendSessionTranscriptMessage` and append duplicate user or assistant entries, so the replay contract is not actually idempotent under retry races. Move the key check inside the locked append path or add an in-flight dedupe guard for `chat.inject` keys.
  - confidence: 0.82
