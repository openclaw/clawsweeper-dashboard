## Summary

Keep open: this is a plausible new Gateway protocol feature, but it is not mergeable as submitted because the generated protocol artifacts/docs are out of sync and the external PR has no real behavior proof. It also needs maintainer API approval for model-visible user-message insertion through `chat.inject`.

## What This Changes

The PR adds optional `role` and `idempotencyKey` parameters to `chat.inject`, routes `role: "user"` through a new transcript append helper, and adds unit coverage for direct injected user-message appends.

## Best Possible Solution

Accept this API only after maintainers confirm the contract, regenerate protocol clients, document the new params, add handler-level replay coverage, and attach redacted real Gateway proof.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection shows current main only supports assistant-only `chat.inject`, while this PR adds a new public protocol mode rather than fixing broken existing behavior.

## Solution Assessment

No, not as submitted. The schema and shared append-helper direction is plausible, but the maintainable path needs generated protocol artifacts, docs, handler-level replay coverage, real behavior proof, and explicit API approval before merge.
