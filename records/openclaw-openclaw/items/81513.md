## Summary

Keep open: the branch addresses the open user-role replay request and includes sufficient live proof, but latest head still checks replay idempotency before the serialized transcript append path, so concurrent replays can duplicate transcript rows.

## What This Changes

The PR extends Gateway `chat.inject` with optional `role` and `idempotencyKey`, adds a user-role transcript append path, returns deduped success for replays, regenerates Swift models, and adds focused transcript coverage.

## Best Possible Solution

Land an additive, documented `chat.inject` contract after making idempotency atomic under transcript append serialization while preserving the assistant default and admin-only scope.

## Reproduction Assessment

Yes for the review finding by source inspection: PR head checks `idempotencyKey` before the append helper enters `withTranscriptAppendQueue`. The broader item is a feature request rather than a current-main bug reproduction.

## Solution Assessment

No. Extending `chat.inject` is a plausible direction for the related request, but replay dedupe must move into the serialized append path or an equivalent in-flight guard, and the public contract should be documented before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make chat.inject idempotency atomic:** `src/gateway/server-methods/chat.ts:1471-1475`
  - body: The replay path checks the transcript for an existing `idempotencyKey` before the append helper enters the per-transcript queue/write lock. Two concurrent calls with the same key can both observe no match and then serialize into duplicate user turns while the API advertises deduped replay semantics; move the key check into the locked append path or add an in-flight guard covering both check and append.
  - confidence: 0.86
- **[P3] Document the new chat.inject contract:** `src/gateway/protocol/schema/logs-chat.ts:69-70`
  - body: Adding `role` and `idempotencyKey` changes the public Gateway RPC, but the current public docs still describe `chat.inject` as assistant-only. Update the Gateway/WebChat docs for the assistant default, user-role append semantics, deduped replay result, and unchanged admin scope so client authors do not rely on stale behavior.
  - confidence: 0.78
