## Summary

Keep open: this PR addresses the open user-role replay request and now includes credible sequential live Gateway proof, but the proposed idempotency contract is still not safe under concurrent retries because the duplicate-key scan happens before the transcript append queue/write lock.

## What This Changes

The branch adds optional `role` and `idempotencyKey` to `chat.inject`, adds a user-role transcript append path, returns deduped success for replayed keys, and adds unit coverage.

## Best Possible Solution

Land an additive, documented `chat.inject` replay contract that supports synthetic user and assistant turns with atomic durable idempotency through the transcript append pipeline.

## Reproduction Assessment

Not applicable as a bug report. The feature gap is source-verifiable on current main: the closed schema and handler expose only assistant-style injection.

## Solution Assessment

No. The direction matches https://github.com/openclaw/openclaw/issues/79820, but the idempotency guarantee must be atomic and the public docs/protocol surface should be aligned before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Make chat.inject idempotency atomic:** `src/gateway/server-methods/chat.ts:1471-1475`
  - body: The new replay path scans the transcript for an existing key before the append path takes the transcript append queue/session write lock. Two concurrent retries with the same key can both pass this check, then serialize through `appendSessionTranscriptMessage` and append duplicate entries, so the advertised idempotency contract is not reliable under retry races. Move the key check inside the locked append path or add an in-flight guard that covers the check and append together.
  - confidence: 0.83
