## Summary

Keep open: this is a plausible Gateway protocol feature PR, but it still lacks required real behavior proof and the proposed replay idempotency is non-atomic under concurrent retries.

## What This Changes

The PR extends Gateway `chat.inject` with optional `role` and `idempotencyKey`, adds user-role transcript append support, returns deduped replay success, and adds unit coverage.

## Best Possible Solution

Land an additive, documented `chat.inject` replay contract that supports user and assistant synthetic turns with durable atomic idempotency through the transcript append pipeline.

## Reproduction Assessment

Not applicable as a bug report. The feature gap is source-verifiable on current main: the schema and handler expose only assistant-style injection.

## Solution Assessment

No. The direction matches the related request, but the idempotency guarantee needs to be atomic and the public protocol/docs surface should be aligned before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Make chat.inject idempotency atomic:** `src/gateway/server-methods/chat.ts:1468-1472`
  - body: The new `chat.inject` replay path scans the transcript for an existing key before the append path takes the transcript write lock. Two retries with the same key can both pass this check, then serialize through `appendSessionTranscriptMessage` and append duplicate entries, so the replay contract is not actually idempotent under retry races. Move the key check inside the locked append path or add an in-flight dedupe guard for `chat.inject` keys.
  - confidence: 0.82
