## Summary

Keep open: this PR addresses a real Gateway feature gap and includes live after-fix proof, but the proposed replay contract is not atomic, so concurrent retries with the same key can still append duplicate transcript entries.

## What This Changes

The PR extends Gateway `chat.inject` with optional `role` and `idempotencyKey`, adds a user-role transcript append path, treats replayed keys as deduped success, and adds focused transcript coverage.

## Best Possible Solution

Land an additive, documented `chat.inject` contract that supports synthetic user and assistant transcript turns with durable atomic idempotency through the shared transcript append pipeline.

## Reproduction Assessment

Not applicable as a bug report. The current-main feature gap is source-verifiable from the closed `chat.inject` schema and assistant-only handler, and the PR race is source-verifiable by comparing the pre-append scan with the append helper's lock boundary.

## Solution Assessment

No. The direction matches https://github.com/openclaw/openclaw/issues/79820, but the patch should make idempotency atomic and align the public docs/protocol surface before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Make chat.inject idempotency atomic:** `src/gateway/server-methods/chat.ts:1468-1475`
  - body: The replay path scans the transcript for an existing key before the append helper enters the per-transcript queue/session write lock. Two concurrent retries with the same `idempotencyKey` can both observe no match and then serialize through `appendInjectedUserMessageToTranscript`, appending duplicate user turns while still advertising deduped replay semantics. Move the key check into the locked append path or add an in-flight guard covering both the check and append.
  - confidence: 0.84
