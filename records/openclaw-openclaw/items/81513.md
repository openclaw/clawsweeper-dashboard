## Summary

Keep open: the requested Gateway capability is valid and the submitted proof is sufficient, but the current branch still drops existing stop matching, provider-auth abort, image attachment routing, TTS projection, and protocol documentation behavior that should be preserved before merge.

## What This Changes

The PR expands Gateway `chat.inject` with optional `role` and `idempotencyKey`, adds locked transcript append dedupe for injected messages, updates generated Swift models, and extends focused transcript tests.

## Best Possible Solution

Land an additive Gateway protocol update that keeps existing abort, provider-auth, image attachment, and TTS projection behavior while documenting the expanded `chat.inject` contract and reconciling it with the related metadata work.

## Reproduction Assessment

Yes. Current main's closed `chat.inject` schema rejects the requested fields and always appends assistant messages; the PR regressions are source-reproducible by comparing the latest head against current main.

## Solution Assessment

No. The locked idempotency direction is reasonable, but this branch is not the best landing path until it restores unrelated current-main Gateway behavior and coordinates the public API shape with the overlapping metadata PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve canonical session matching for stop commands:** `src/gateway/server-methods/chat.ts:1615`
  - body: The branch now matches abortable runs only when `active.sessionKey === params.sessionKey`. Current main also checks canonical aliases and the backing `sessionId`, so a stop sent through an alias such as `main` can miss the active run and fail to abort or persist the partial.
  - confidence: 0.92
- **[P1] Keep provider metadata on active chat runs:** `src/gateway/server-methods/chat.ts:2213-2223`
  - body: `registerChatAbortController` no longer records `providerId` or `authProviderId`, and the branch no longer updates those fields after model fallback. `abortChatRunsForProvider` relies on that metadata, so provider logout or credential revocation can leave active chat sends running.
  - confidence: 0.91
- **[P1] Keep the configured image-model override path:** `src/gateway/server-methods/chat.ts:2148-2151`
  - body: This path now routes image attachments only when the session model itself supports images. Current behavior can switch to `agents.defaults.imageModel` for image attachments on text-only sessions; with this diff those sends fall back to media paths instead of the configured vision model.
  - confidence: 0.95
- **[P2] Preserve TTS supplement markers on media replies:** `src/gateway/server-methods/chat.ts:2471-2480`
  - body: The refactor stops passing the TTS supplement marker when appending delayed media-only TTS replies. `projectRecentChatDisplayMessages` depends on `openclawTtsSupplement` to merge the audio supplement into the original assistant message, so history can show a separate audio assistant row.
  - confidence: 0.88
- **[P2] Document the expanded chat.inject contract:** `src/gateway/protocol/schema/logs-chat.ts:69-70`
  - body: Adding `role` and `idempotencyKey` changes the public Gateway protocol, but the docs still describe `chat.inject` as an assistant-note-only path. Please document user-role appends, idempotent replay responses, returned message IDs, and duplicate-broadcast behavior.
  - confidence: 0.86
