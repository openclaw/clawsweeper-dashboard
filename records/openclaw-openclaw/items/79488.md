## Summary

Keep open: current main still has the source path that can stringify object-shaped skill config requirements as `[object Object]`, and an open linked PR explicitly targets this issue with closing syntax, so the issue should stay open until that PR is reviewed, landed, or closed.

## What This Changes

The issue requests correct handling of object-shaped `metadata.openclaw.requires.config` entries in skill status/info output instead of rendering them as `[object Object]`.

## Best Possible Solution

Settle the skill metadata contract, then land a focused fix that either supports object entries by `path` with tests/docs or reports unsupported object entries without silently satisfying the requirement.

## Reproduction Assessment

Yes from source inspection: current main sends `requires.config` array entries through a normalizer that calls `String(entry)`, and an object entry becomes `[object Object]`. I did not run a full `openclaw skills info` fixture because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the contract: extracting `path` is the useful compatibility direction, but current docs/types only define string config paths. The open linked PR is the right active place to review that decision rather than closing this issue now.
