## Summary

Keep open: current main and v2026.6.10 still route skill `requires.config` through a string-only requirement path, and the linked fixing PR is still open and conflicting. Maintainers need to decide whether object entries are a supported skill metadata contract or should be rejected with a clear unsatisfied diagnostic.

## What This Changes

The issue asks OpenClaw to handle object-shaped skill `requires.config` entries without rendering `[object Object]` or blocking valid skills.

## Best Possible Solution

Maintainers should decide the skill requirement contract, then either support object entries end to end across parser, evaluator, docs, and tests or keep string-only docs and reject object entries with a clear unsatisfied diagnostic.

## Reproduction Assessment

Yes, source-level. Object entries in `requires.config` flow through `normalizeStringList` into `String(entry)` and then into skill status checks as config paths, which explains the reported `[object Object]` output.

## Solution Assessment

Unclear as requested. Current docs and evaluator types define string config paths, so supporting `{path, access, purpose}` is only the best fix if maintainers confirm that as the intended skill metadata contract.
