## Summary

Keep open: current main and the latest release still stringify object-shaped `requires.config` entries, and the issue is paired with an open but conflicting fix PR. Maintainers should decide whether object entries are a supported skill metadata contract or should be rejected with a clearer diagnostic.

## What This Changes

The issue asks OpenClaw to handle object-shaped skill `requires.config` entries without rendering `[object Object]` or incorrectly blocking the skill.

## Best Possible Solution

Keep one explicit skill requirement contract: either support object entries consistently in parsing, status, docs, and tests, or keep string-only docs and reject object entries without `[object Object]` diagnostics.

## Reproduction Assessment

Yes. Source inspection shows object entries in `requires.config` still flow through `String(entry)` and then into skill status checks as string paths, which explains the reported `[object Object]` output.

## Solution Assessment

Unclear as a product decision. If object-shaped config requirements are intended, the best fix is a narrow shared parser/evaluator/docs/test update; if not, the best fix is to keep the string-only contract and improve validation/diagnostics.
