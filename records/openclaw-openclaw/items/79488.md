## Summary

Keep open: source inspection on current main still shows object-shaped `metadata.openclaw.requires.config` entries being normalized as strings, which matches the reported `[object Object]` failure, and an open linked PR already uses closing syntax for this issue. The issue should remain open until that fix is reviewed and merged or replaced.

## What This Changes

The issue requests support for object-shaped skill `requires.config` entries so `openclaw skills info` and eligibility checks use the entry `path` instead of rendering `[object Object]`.

## Best Possible Solution

Review and land the linked focused fix or an equivalent parser/docs/tests change that either supports object entries by `path` or rejects them explicitly without producing `[object Object]`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an object entry in `requires.config` flows through `normalizeStringList()`, which calls `String(entry)` and feeds `[object Object]` into config eligibility checks.

## Solution Assessment

Unclear. Extracting `.path` is a narrow technical fix for the visible bug, but maintainers should first settle whether `path`/`access`/`purpose` objects are a supported skill metadata contract and update docs/tests accordingly.
