## Summary

Keep open: current main and v2026.6.10 still route object-shaped skill config requirements through string-list normalization, while the linked candidate fix remains open and unmerged. The remaining decision is whether OpenClaw should support object entries end to end or keep the documented string-only contract with a better diagnostic.

## What This Changes

This issue requests that object-shaped `metadata.openclaw.requires.config` entries render and evaluate by their structured values instead of becoming `[object Object]` and blocking skills.

## Best Possible Solution

Settle one canonical skill config requirement contract, then land a narrow fix or cleaned/split linked PR that keeps parser, evaluator, CLI/doctor status, docs, and regression tests aligned.

## Reproduction Assessment

Yes. Source inspection shows object entries in `requires.config` flow through `String(entry)` and then into skill status config checks as paths, which explains the reported `[object Object]` output without running a mutating CLI repro.

## Solution Assessment

Unclear as a product decision. If object entries are intended, the best fix is end-to-end parser/evaluator/docs/test support; if not, the best fix is to keep the string-only contract and reject or report objects without `[object Object]` diagnostics.
