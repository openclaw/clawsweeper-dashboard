## Summary

Keep open: current main still routes `metadata.openclaw.requires.config` through the loose string normalizer, so object entries can become `[object Object]`, and an open fixing PR already references this issue. The exact object-entry contract still needs maintainer review because current docs define `requires.config` as string config paths, not filesystem/path/access/purpose objects.

## What This Changes

Support object-shaped `metadata.openclaw.requires.config` entries without showing `[object Object]` or incorrectly blocking skills.

## Best Possible Solution

Settle the skill metadata contract, then land a narrow fix that either supports object entries by extracting `path` with tests/docs or rejects them with a clear diagnostic instead of displaying `[object Object]`.

## Reproduction Assessment

Yes from source inspection: object array entries in `requires.config` still flow through `String(entry)` before the skill status evaluator checks config paths. I did not run a live CLI fixture because this review was read-only.

## Solution Assessment

Unclear: extracting the `path` field is a plausible narrow way to eliminate `[object Object]`, but accepting `access`/`purpose` object metadata changes the documented string-array contract and should be confirmed by maintainers.
