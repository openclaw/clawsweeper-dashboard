## Summary

Keep open: current main still has the source path that turns object-shaped config requirements into `[object Object]`, and an open PR already references this issue with closing syntax, so it should stay open until that PR lands or is replaced.

## What This Changes

Request support or clear handling for object-shaped `metadata.openclaw.requires.config` entries so skill status does not show `[object Object]` or incorrectly block the skill.

## Best Possible Solution

Review and land a focused fix, either through the open PR or a narrower replacement, that settles the `requires.config` contract and adds regression coverage for object entries without silently dropping requirements.

## Reproduction Assessment

Yes for the `[object Object]` symptom: current source sends object array entries through `String(entry)`, and a direct Node check confirms the resulting string. I did not run a full `openclaw skills info` fixture during this read-only review.

## Solution Assessment

Unclear: accepting `{ path, access, purpose }` objects may be the right compatibility direction, but current docs and evaluators define string config keys. The maintainable solution should update parser, evaluator/status behavior, tests, and docs together after that contract decision.
