## Summary

Keep open: current main still has the source path that turns object-shaped skill config requirements into `[object Object]`, and https://github.com/openclaw/openclaw/pull/79538 is already an open closing PR for this issue. The issue should remain open until that PR is reviewed and merged, or maintainers choose a different skill metadata contract.

## What This Changes

Support or clearly reject object-shaped `metadata.openclaw.requires.config` entries so `openclaw skills info --json` no longer reports `[object Object]` or a misleading missing config path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79538 if maintainers accept the object-format contract, or replace it with a narrower fix that rejects unsupported object entries with a clear unsatisfied diagnostic; close this issue only after that fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows object entries flow through `normalizeStringList()` and then `String(entry)`, and a direct Node check confirms that produces `[object Object]`; I did not run the full CLI in this read-only review.

## Solution Assessment

Unclear. End-to-end object support may be right, but current docs and evaluators define `requires.config` as string OpenClaw config paths, so the maintainable solution needs a contract decision before changing status behavior.
