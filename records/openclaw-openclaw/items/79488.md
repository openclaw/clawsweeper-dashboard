## Summary

Keep open. Current main still has a source-reproducible path that turns object-shaped `requires.config` entries into `[object Object]`, but supporting `{path, access, purpose}` with filesystem-access semantics extends the currently documented string-array config-key contract and needs maintainer schema judgment first.

## What This Changes

Allow object-shaped skill `requires.config` entries with `path`, `access`, and `purpose` to display and validate correctly instead of surfacing `[object Object]`.

## Best Possible Solution

Define one canonical schema for skill config requirements, then update parser/types/status/doctor/docs/tests so object entries either work intentionally or produce a clear unsupported-shape diagnostic without `[object Object]`.

## Reproduction Assessment

Yes, source-level. Structured YAML metadata can become JSON, `requires.config` is normalized as a string list, and object entries pass through `String(entry)`, which produces `[object Object]`; I did not need a full CLI run to confirm that path.

## Solution Assessment

Unclear. Supporting object/file-access semantics may be the right fix, but current docs and types only promise string config-key checks, so maintainers should first decide whether to extend the manifest contract or reject the object shape clearly.
