## Summary

Keep open: source inspection confirms object entries in `metadata.openclaw.requires.config` are coerced into `[object Object]`, but current OpenClaw docs and types define that field as string config paths, so supporting `path`/`access`/`purpose` objects needs a maintainer contract decision before a narrow fix.

## What This Changes

Support or clearly reject `SKILL.md` `metadata.openclaw.requires.config` entries written as objects with `path`, `access`, and `purpose` instead of only strings.

## Best Possible Solution

Decide the skill metadata contract, then either normalize object entries by their `path` field with tests and docs or reject them with a clear diagnostic instead of emitting `[object Object]`.

## Reproduction Assessment

Yes from source inspection for the reported `[object Object]` symptom: `resolveOpenClawManifestRequires()` sends `requiresRaw.config` through a normalizer that calls `String(entry)` for array items. I did not run a live CLI fixture because this was a read-only review.

## Solution Assessment

Unclear: accepting `path`/`access`/`purpose` objects may be the right compatibility direction, but current docs and types only define string OpenClaw config paths. The maintainable fix should first settle that contract, then update parser, status output, tests, and docs together.
