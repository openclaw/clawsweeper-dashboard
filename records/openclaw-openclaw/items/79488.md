## Summary

Keep open: current main and the latest release still route object-shaped `requires.config` entries through string normalization, while the linked fix PR remains open and conflicting. The remaining decision is whether object entries are a supported skill metadata contract or should be rejected with a clear unsatisfied diagnostic.

## What This Changes

The issue asks OpenClaw to handle object-shaped skill `requires.config` entries without rendering `[object Object]` or incorrectly blocking skill eligibility/status checks.

## Best Possible Solution

Keep one explicit skill requirement contract: either support object entries end to end in parser, evaluator, docs, and tests, or keep the string-only contract and surface unsupported object entries as clear unsatisfied diagnostics.

## Reproduction Assessment

Yes. Source inspection shows object entries in `requires.config` still flow through `String(entry)` and then into skill status checks as string paths; a live CLI reproduction was not run in this read-only review.

## Solution Assessment

Unclear. Extracting `.path` is only the best fix if maintainers approve object-shaped skill config requirements; otherwise the safer fix is to keep the string-only contract and make unsupported object entries clearly unsatisfied instead of rendering `[object Object]`.
