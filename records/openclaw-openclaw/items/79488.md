## Summary

Keep open: current main still stringifies object-shaped `requires.config` entries, and the linked closing PR is open rather than merged. This should stay open until https://github.com/openclaw/openclaw/pull/79538 or a narrower replacement settles the config metadata contract and lands.

## What This Changes

The issue asks for object-shaped skill config requirements to display and validate by their `path` value instead of becoming `[object Object]`.

## Best Possible Solution

Decide whether object-shaped `requires.config` entries are supported; if yes, normalize by `path` and align parser types, docs, and regression tests, and if no, reject them with a clear diagnostic instead of `[object Object]`.

## Reproduction Assessment

Yes for the reported `[object Object]` symptom: source inspection shows object entries flow through `String(entry)` and then into config status checks as the path. I did not run the CLI because this review is read-only and avoids commands that may write artifacts.

## Solution Assessment

Unclear: extracting `.path` is a plausible narrow compatibility fix, but the maintainable solution should first confirm the object-entry contract and then update parser, status output, docs, and tests together.
