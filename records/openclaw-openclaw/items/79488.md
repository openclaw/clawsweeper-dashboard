## What This Changes

The issue requests support for object-shaped `metadata.openclaw.requires.config` entries so skill status and `openclaw skills info --json` use the entry `path` instead of rendering `[object Object]`.

## Best Possible Solution

Settle the skill metadata contract, then land a focused implementation that aligns parser behavior, eligibility checks, tests, and docs around either supported object entries or a clear unsupported-entry diagnostic.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the visible symptom: object entries flow through `normalizeStringList()` and `String(entry)`, which produces `[object Object]`. I did not run a live `openclaw skills info` fixture because this was a read-only review.

## Solution Assessment

Unclear: extracting `path` is the narrow fix if maintainers accept object-shaped config requirements, but a clear unsupported diagnostic is safer if `requires.config` remains string-only. The open linked PR is the right place to resolve that choice.
