## Summary

Keep open: current main and the latest release still coerce object-shaped `requires.config` entries through the generic string normalizer, while the linked fix PR is open and currently conflicting. Current docs still describe `requires.config` as string paths, so maintainers should confirm the object-entry contract before landing or replacing the fix.

## What This Changes

The issue requests proper handling of object-shaped skill `requires.config` entries instead of showing `[object Object]` and marking the skill blocked.

## Best Possible Solution

Maintain one canonical contract for skill config requirements: either support object entries consistently in parsing, status, docs, and tests, or keep string-only docs and reject object entries without `[object Object]` diagnostics.

## Reproduction Assessment

Yes. Source inspection shows object entries in `requires.config` flow through `String(entry)` and then into skill status checks as paths, which explains the reported `[object Object]` output without needing to run a mutating checkout command.

## Solution Assessment

Unclear as a product decision. If object-shaped config requirements are intended, the best fix is a narrow shared parser/evaluator/docs/test update; if not, the best fix is to keep the string-only contract and improve validation/diagnostics so objects do not render as `[object Object]`.
