## Summary

Keep open: current main still finalizes ACP-visible text after a terminal `done` event without completed-action evidence, so the Telegram/ACP false-completion path remains plausible. The newer ACP event ledger is useful for session-load replay, but it does not verify that a claimed file-write or message-send side effect happened.

## What This Changes

The issue asks OpenClaw to prevent Telegram/ACP agents from narrating completed file-write or message-send actions before those actions actually happen.

## Best Possible Solution

Define a generic ACP completed-action evidence contract plus a final-answer revision or holdback gate that chat channels can use while preserving legitimate text-only replies and current Telegram streaming transport.

## Reproduction Assessment

Yes. From source, a mock ACP runtime can emit visible text such as “I wrote it” followed by `done`, and the dispatcher/projector path will deliver that text without completed-action evidence; I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. Current final-only buffering and the ACP event ledger improve delivery timing and replay history, but they do not verify action claims; a generic ACP evidence/finalization contract is safer than Telegram-specific phrase filtering.
