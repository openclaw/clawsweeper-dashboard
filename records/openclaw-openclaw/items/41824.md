- `P1`: A chat-visible agent workflow can falsely report completed side effects, which is urgent for operator trust and real channel workflows.
- `impact:message-loss`: The report includes a claimed message/file-write action that was not actually delivered or persisted when the agent said it was done.
- `impact:session-state`: The visible chat transcript can drift from actual ACP/tool side-effect state.

## Summary

Keep open: current main still lets ACP text deltas become Telegram-visible final replies without a typed tool result or side-effect receipt, so the core false-completion problem remains despite adjacent task-completion fixes.

## What This Changes

The issue asks OpenClaw to prevent Telegram-visible agents from claiming file or message side effects were completed when no corresponding tool action or side-effect evidence actually occurred.

## Best Possible Solution

Define a generic ACP finalization contract that can distinguish ordinary final text from completed side-effect evidence before visible chat delivery, without relying on Telegram-specific phrase filtering.

## Reproduction Assessment

Yes, source-reproducible: current ACP dispatch tests can emit text_delta followed by done, and current main delivers that text as a Telegram final reply without side-effect evidence. I did not run a live Telegram/file-system replay.

## Solution Assessment

Unclear until maintainers choose the ACP contract. The maintainable direction is a generic evidence/finalization gate, not Telegram-specific phrase filtering or an after-the-fact hook that only logs mismatches.
