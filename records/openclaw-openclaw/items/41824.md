## Summary

Keep this issue open. Current main has adjacent ACP improvements, including final-only projection and a terminal done-event guard, but it still does not verify Telegram/ACP final replies that claim file or message actions against observed completed mutating actions before delivery.

## What This Changes

The issue requests stronger Telegram/ACP harness behavior so agents cannot report file or message actions as completed when those actions did not execute.

## Best Possible Solution

Keep the issue open for a runtime/harness-boundary design: record completed mutating file/message/tool events where OpenClaw can observe them, expose that evidence to ACP/Telegram final delivery, and start with a narrow opt-in warning, revision, or block when final text claims an unobserved action.

## Reproduction Assessment

Yes. The remaining gap can be reproduced with a focused ACP dispatch simulation: have runTurn emit a text_delta such as “I wrote/sent it” followed by done, with no completed tool_call evidence, and current dispatch will persist and deliver that text.

## Solution Assessment

No. Current main’s terminal done-event guard and final-only projection are useful adjacent fixes, but they do not solve the central safety property; the safer path is an explicit evidence ledger plus a final-answer gate.
