## Label Justifications

- `P1`: A chat-visible agent workflow can falsely report completed side effects, which breaks real operator/channel workflows.
- `impact:message-loss`: The report includes a claimed message/file-write action that was not actually delivered or persisted when the agent said it was done.
- `impact:session-state`: The visible Telegram transcript can diverge from ACP/tool side-effect state and mislead the operator about completed work.

## Summary

Keep open: current main and v2026.5.22 still allow ACP output text to become a Telegram-visible final reply after a terminal event without a typed completed-action receipt or side-effect evidence, so the reported false-completion class is not fixed.

## What This Changes

The issue asks OpenClaw to prevent Telegram-visible ACP agents from claiming file/message side effects before the underlying tool or action actually completed.

## Best Possible Solution

Define a generic ACP finalization contract that distinguishes ordinary final text from completed side-effect evidence before chat-visible delivery, with tests across ACP projector, dispatch, hooks, and Telegram delivery.

## Reproduction Assessment

Yes, source-reproducible: current ACP dispatch can emit text_delta followed by done, and current main delivers that text as a Telegram final reply without side-effect evidence. I did not run a live Telegram/file-system replay.

## Solution Assessment

Unclear until maintainers choose the ACP contract. The maintainable direction is a generic evidence/finalization gate, not Telegram-specific phrase filtering or an after-the-fact hook that only logs mismatches.
