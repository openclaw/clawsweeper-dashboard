## Summary

Keep open: current main still allows ACP-visible text to become the final Telegram/chat reply after a terminal `done` event without any completed-action evidence, so the reported false-completion path remains plausible and needs maintainer API/product direction.

## What This Changes

The issue reports ACP-backed Telegram turns claiming file/message side effects before the action happened, and asks whether OpenClaw can enforce tool-first or verified final-response behavior on chat surfaces.

## Best Possible Solution

Define a generic ACP completed-action evidence and final-answer hold/revision contract across runtime events, projector delivery, and dispatch finalization while preserving legitimate text-only replies.

## Reproduction Assessment

Yes, from source: a mock ACP runtime can emit visible text like “I wrote it” followed by `done`, and current main will accept and deliver that text as final without completed-action evidence. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the ACP contract, but a generic evidence/finalization gate is the best likely direction. A Telegram-specific phrase filter or extra final-message workaround would be less maintainable and could conflict with intentional streaming behavior.
