## Summary

Keep open: current main still has a source-level path where ACP text can become a Telegram/chat-visible final reply after terminal completion without evidence that the claimed side effect happened. The related broader truthfulness/fail-closed issues are useful context, but they do not supersede this narrower Telegram ACP final-delivery contract gap.

## What This Changes

The issue reports Telegram-visible ACP replies claiming completed actions before the underlying tool or file-write work is proven complete.

## Best Possible Solution

Define a generic ACP runtime/final-delivery contract that carries completed-action evidence and can hold, revise, or clearly mark unsupported success claims before chat delivery.

## Reproduction Assessment

Yes, source-reproducible: a mock ACP runtime can emit a text_delta such as a completed-action claim followed by done, and current main will deliver that text as a final chat reply without completed-action evidence. I did not run a live Telegram/file-system replay of the reporter's private session.

## Solution Assessment

Unclear until maintainers choose the ACP contract. The maintainable direction is a generic evidence/finalization gate, not Telegram-specific phrase filtering or an after-the-fact hook that only logs the mismatch.
