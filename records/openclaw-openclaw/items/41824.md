## Summary

Keep this issue open. Current main has narrowed adjacent ACP failure modes, including final-only ACP projection, a terminal done-event guard, and a changeloged fix for the related premature Claude ACP completion path, but it still lacks the requested safety property: a Telegram/ACP final answer that claims a file or message action completed is not checked against an observed completed mutating action before delivery.

## What This Changes

The issue requests stronger Telegram/ACP harness behavior so agents cannot report file/message actions as complete when those actions did not execute.

## Best Possible Solution

Keep this open for maintainer design. The desired end state is a runtime/harness-boundary verified-action mechanism that records completed mutating file/message/tool events where OpenClaw can observe them, exposes that evidence to ACP/Telegram final delivery, and applies an opt-in or narrowly scoped warning, revision, or block when a final answer claims an unobserved completed action.
