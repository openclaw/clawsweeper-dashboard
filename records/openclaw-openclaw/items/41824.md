## Summary

The issue remains necessary: current main still accepts an ACP terminal `done` event as a completed turn without requiring that a requested action produced tool activity. The narrower fix PR closed unmerged, and the related Telegram report remains open; a current-main Claude-over-ACP trace is still needed before defining cross-runtime strict-execution behavior.

## What This Changes

Prevent Telegram-visible agents from reporting file writes, messages, or other requested actions as completed when no corresponding tool execution occurred.

## Best Possible Solution

Reproduce the failure on current main with Claude over ACP, capture structured ACP events and the delivered Telegram reply, then enforce a runtime-owned action-intent/outcome invariant that distinguishes ordinary chat from requested execution and is consumed consistently by every channel.

## Reproduction Assessment

No, not yet with high confidence on current main. The issue and https://github.com/openclaw/openclaw/issues/60955 provide credible historical session evidence, and current source permits completed ACP text-only turns, but a live Claude ACP trace is still required.

## Solution Assessment

No, a Telegram-specific or rendered-text matcher is not the best general fix. The maintainable direction is structured shared-runtime enforcement, but maintainers must decide how action intent is represented without forcing tools for ordinary conversation.
