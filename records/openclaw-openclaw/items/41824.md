## Summary

Keep open: current main still permits ACP-visible text to become the final chat reply after a terminal `done` event without any completed-action evidence, so the Telegram/ACP false-completion path remains plausible and needs maintainer API/product direction.

## What This Changes

The issue asks OpenClaw to prevent or expose Telegram/ACP false-completion claims where an agent reports file-write or message-send actions that did not execute.

## Best Possible Solution

Define a generic ACP completed-action evidence and final-answer revision/holdback contract across runtime events, dispatch, and projector delivery while preserving legitimate text-only replies and Telegram’s single visible preview/finalization behavior.

## Reproduction Assessment

Yes, from source: a mock ACP runtime can emit visible text such as “I wrote it” followed by `done`, and current main will accept and deliver that final text without completed-action evidence. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract, but a generic ACP evidence/finalization gate is the best likely direction. A Telegram-specific phrase filter or extra final-message workaround would be narrower but less maintainable and would fight the existing streaming decision.
