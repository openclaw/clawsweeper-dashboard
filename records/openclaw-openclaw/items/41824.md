## Summary

Keep open: current main still lets ACP-visible text become the final Telegram/chat reply after a terminal `done` event without completed-action evidence, so the false-completion path remains source-reproducible and needs maintainer API/product direction.

## What This Changes

The issue asks OpenClaw to stop Telegram/ACP agents from delivering unverified completed-action claims and to clarify or enforce tool-first execution semantics on chat surfaces.

## Best Possible Solution

Define a generic ACP runtime and final-delivery contract that carries completed-action evidence and can hold or revise unsupported action-completion claims before chat delivery, while preserving valid text-only replies and Telegram’s single visible streaming/final message behavior.

## Reproduction Assessment

Yes, source-reproducible: a mock ACP runtime can emit visible text such as `I wrote it` followed by `done`, and current main will flush that text as the final chat reply without completed-action evidence. I did not live-replay the reporter’s private Telegram/file session.

## Solution Assessment

Unclear until maintainers choose the ACP contract, but a generic completed-action evidence and finalization gate is the best likely direction. A Telegram-specific phrase filter or extra final-message workaround would be less maintainable and could conflict with intentional streaming behavior.
