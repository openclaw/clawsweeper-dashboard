## Summary

Keep open: current main still permits ACP text to become a Telegram/chat-visible final reply after terminal `done` without any completed-action evidence, so the reported false-success failure mode is not implemented away.

## What This Changes

The issue asks OpenClaw to prevent ACP/Telegram agents from reporting completed actions that were not actually executed.

## Best Possible Solution

Define a generic ACP runtime/final-delivery contract that carries completed-action evidence and can hold or revise unsupported success claims before chat delivery.

## Reproduction Assessment

Yes, source-reproducible: a mock ACP runtime can emit a visible text claim followed by `done`, and current main will send that text as the Telegram/chat final reply without completed-action evidence. I did not run a live Telegram/file-system replay of the reporter's private session.

## Solution Assessment

Unclear until maintainers choose the ACP contract; the maintainable direction is a generic evidence/finalization gate rather than Telegram-specific phrase filtering or hook-only after-the-fact detection.
