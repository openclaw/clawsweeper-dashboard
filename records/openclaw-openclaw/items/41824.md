## Label Justifications

- `P1`: A chat-visible agent workflow can falsely report completed side effects, which is urgent for operator trust and real channel workflows.
- `impact:message-loss`: The report includes a claimed message/file-write action that was not actually delivered or persisted when the agent said it was done.
- `impact:session-state`: The visible chat transcript can drift from the actual ACP/tool side-effect state.

## Summary

Keep open: current main still permits an ACP runtime to deliver completed-action-looking final text to Telegram without a typed tool or side-effect receipt, and the merged progress-only task fix does not cover direct chat final replies.

## What This Changes

Prevent Telegram/ACP chat replies from claiming file or message side effects completed unless the claim is backed by real execution or finalization evidence.

## Best Possible Solution

Define a generic ACP finalization contract that can distinguish final text from completed side-effect evidence before visible chat delivery, while preserving legitimate text-only replies and plugin-level policy extensions.

## Reproduction Assessment

Yes, source-reproducible: a mock ACP runtime can emit completed-action-looking text_delta output followed by done, and current main will deliver that text as a final chat reply without side-effect evidence. I did not run a live Telegram/file-system replay.

## Solution Assessment

Unclear until maintainers choose the ACP contract. The maintainable direction is a generic evidence/finalization gate, not Telegram-specific phrase filtering or an after-the-fact hook that only logs mismatches.
