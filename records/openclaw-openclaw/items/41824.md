## Label Justifications

- `P1`: A chat-visible agent workflow can falsely report completed side effects, which is urgent for operator trust and real channel workflows.
- `impact:message-loss`: The report includes a claimed message/file-write action that was not actually delivered or persisted when the agent said it was done.
- `impact:session-state`: The issue is about drift between the visible chat transcript and the actual tool/session side-effect state.

## Summary

Keep open: current main still allows ACP final text to be delivered to Telegram without a typed completed-action or side-effect evidence contract, and the adjacent progress/delivery issues do not supersede this specific false-success gap.

## What This Changes

The issue reports Telegram/ACP agents can visibly claim completed file or message side effects before OpenClaw has evidence those actions actually ran.

## Best Possible Solution

Define a generic ACP finalization contract that carries completed tool or side-effect evidence, can hold or revise unsupported success claims before chat delivery, preserves legitimate text-only replies, and is proven with focused ACP tests plus real Telegram validation.

## Reproduction Assessment

Yes, source-reproducible: a mock ACP runtime can emit completed-action-looking text_delta output followed by done, and current main will deliver that text as a final chat reply without completed-action evidence. I did not run a live Telegram/file-system replay of the reporter's private session.

## Solution Assessment

Unclear until maintainers choose the ACP contract. The maintainable direction is a generic evidence/finalization gate, not Telegram-specific phrase filtering or an after-the-fact hook that only logs mismatches.
