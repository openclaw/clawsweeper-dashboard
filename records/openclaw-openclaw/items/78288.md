## Summary

Keep open: this is the active canonical PR for the linked exec-node display request, but it should not merge as-is because the new formatter can show a node label even when an explicit non-node host would run elsewhere.

## What This Changes

This PR appends a trimmed exec `node` value to transparency details, adds formatter tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow formatter fix that shows node context for node-target exec calls, preserves cwd/raw/explain formatting, and adds a regression for explicit non-node hosts carrying a stray `node` arg.

## Reproduction Assessment

Yes. Source inspection shows current main routes exec transparency through `resolveExecDetail` and ignores `args.node`, and the review finding is source-reproducible from the runtime contract that only `host=node` consumes `params.node`.

## Solution Assessment

No as submitted. Updating `resolveExecDetail` is the right surface, but the node label should not be rendered for explicit gateway or sandbox execution because that would make the transparency message false.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate the node label to node-target exec calls:** `src/agents/tool-display-exec.ts:401-402`
  - body: `node` is only meaningful for `host=node`, and the runtime only forwards `params.node` inside the node-host branch. As written, args like `{ host: "gateway", node: "raspberrypi" }` would run on the gateway but display `node: raspberrypi`, making the transparency message misleading; gate this on node-target args and add a regression test.
