## Summary

Keep open. Current main still lacks cursor pagination for both requested RPCs: `sessions.list` accepts only the existing filters plus `limit` and returns no `nextCursor` or pre-slice `total`, while `agents.files.list` accepts only `agentId` and returns a fixed workspace bootstrap/MEMORY file set. The related artifact cursor PR remains useful context, but the cursor helper/contract it describes is not present in current main.

## What This Changes

Add cursor-style pagination fields to `agents.files.list` and `sessions.list`, following the pattern proposed for artifact listing.

## Best Possible Solution

Define an additive cursor contract for `sessions.list` across schema, runtime, shared result types, SDK/client models, generated Swift models, docs, and focused tests. For `agents.files.list`, make an explicit product/API call: either add compatible pagination metadata for consistency or document that the fixed workspace-file surface remains intentionally unpaginated.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape issue: strict validators reject a `cursor` field for both RPCs, and the runtime responses provide no `nextCursor` or pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a plausible narrow solution for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that method is currently fixed-scope rather than an unbounded listing.
