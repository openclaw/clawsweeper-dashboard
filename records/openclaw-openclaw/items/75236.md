## Summary

Keep open. Current main still lacks a cursor/nextCursor/total contract for both requested RPCs: `sessions.list` only applies a `limit` slice and reports the sliced count, while `agents.files.list` remains a strict `agentId`-only RPC over a fixed bootstrap/MEMORY file set. The referenced artifact pagination work is useful context, but #75208 is still open and current main does not contain the proposed cursor helper contract.

## What This Changes

The issue requests additive cursor pagination fields for `agents.files.list` and `sessions.list` so gateway clients can page large file/session result sets consistently.

## Best Possible Solution

Define an additive Gateway pagination contract for `sessions.list` across schema, runtime, shared result types, generated clients, docs, and focused tests. For `agents.files.list`, maintainers should either intentionally apply the same fields for API consistency or document that the fixed bootstrap/MEMORY file set remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: strict schemas reject `cursor` for both requested RPCs, `sessions.list` returns no continuation metadata after limit slicing, and `agents.files.list` always returns its fixed file array.

## Solution Assessment

Unclear for the exact requested scope. Cursor pagination is a reasonable direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that RPC is fixed-scope today rather than a broad workspace file listing.
