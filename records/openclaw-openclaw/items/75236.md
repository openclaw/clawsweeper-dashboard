## Summary

Keep open. Current main still lacks cursor pagination for both requested RPCs: `sessions.list` has a strict non-cursor schema and only limit-slices results, while `agents.files.list` remains a strict fixed bootstrap/MEMORY file listing. The request is valid, but the exact public protocol scope needs maintainer judgment because the referenced artifact cursor contract is still in an open PR and `agents.files.list` is not currently an unbounded workspace listing.

## What This Changes

Request additive cursor pagination fields for `agents.files.list` and `sessions.list`, following the artifact RPC cursor pattern proposed in #75208.

## Best Possible Solution

Define an additive pagination contract for `sessions.list` across schema, runtime, shared types, SDK/UI/generated clients, docs, and tests. For `agents.files.list`, either intentionally add the same fields for API consistency or document that the fixed bootstrap file surface remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: strict schemas reject `cursor` for both requested RPCs, `sessions.list` only supports a `limit` slice, and neither response shape includes `nextCursor` or `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a reasonable direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that RPC currently returns a bounded bootstrap file set rather than a large workspace listing.
