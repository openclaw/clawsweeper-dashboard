## Summary

Keep open: current main and latest release v2026.6.10 still lack cursor/nextCursor pagination for both requested Gateway RPCs. The request remains useful, but it changes public protocol shape and needs maintainer API direction rather than cleanup closure.

## What This Changes

Request cursor pagination for `agents.files.list` and `sessions.list` Gateway RPCs so large workspaces can page those results consistently.

## Best Possible Solution

Have maintainers choose the Gateway list pagination standard first, then implement an additive protocol/server/client/docs/test change only for the RPCs that should become pageable.

## Reproduction Assessment

Yes at source level: current Gateway schemas and handlers show no `cursor` or `nextCursor` contract for either requested RPC. This is a missing feature/API contract rather than a live bug reproduction.

## Solution Assessment

Unclear. Cursor pagination is plausible for `sessions.list`, but current main deliberately uses offset metadata, and `agents.files.list` may not need pagination while it returns a fixed editable workspace-file set.
