## Summary

Keep open. Current main still lacks cursor pagination on both requested Gateway RPC surfaces, and the referenced artifact cursor work is an open PR rather than a merged contract on main. The request is valid enough to preserve, but the full scope needs maintainer API judgment because `sessions.list` is an unbounded listing while `agents.files.list` currently exposes a small, fixed bootstrap-file surface.

## What This Changes

The issue requests adding cursor-style pagination fields to `agents.files.list` and `sessions.list`, following the cursor pattern proposed for artifact RPC listing.

## Best Possible Solution

Define an additive cursor contract for `sessions.list` across schema, runtime, shared result types, SDK/client models, generated Swift models, docs, and focused tests. For `agents.files.list`, make an explicit maintainer API call: either add compatible pagination metadata for consistency or document that the fixed bootstrap-file surface is intentionally unpaginated.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape issue: strict validators reject `cursor` for both requested RPCs, and current responses provide no `nextCursor` or pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination looks like a maintainable additive direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that method is fixed-scope today.
