## Summary

Keep open. Current main still lacks cursor pagination for both requested Gateway RPCs: the strict protocol schemas do not accept `cursor`, the runtime responses do not return `nextCursor` or a pre-slice `total`, and the referenced artifact cursor helper from the related PR is not present on main.

## What This Changes

Add cursor-style pagination fields to `agents.files.list` and `sessions.list`, including cursor input and paginated result metadata.

## Best Possible Solution

Define an additive cursor contract for `sessions.list` across protocol schema, runtime, SDK/client types, generated clients, docs, and focused tests. For `agents.files.list`, first make the product/API call explicit: either add compatible pagination metadata for consistency or document that the fixed bootstrap-file surface is intentionally unpaginated.

## Reproduction Assessment

Yes. Source inspection against current main is enough: strict validators reject a `cursor` field for both RPCs, and the runtime responses provide no `nextCursor` or pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a plausible narrow solution for `sessions.list`, but applying it to fixed-scope `agents.files.list` should wait for maintainer confirmation or an explicit contract decision.
