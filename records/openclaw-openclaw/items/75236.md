## Summary

Keep open. Current main still lacks cursor pagination for both requested Gateway RPCs: `sessions.list` has strict params without `cursor` and returns only the sliced session rows, while `agents.files.list` accepts only `agentId` and returns the fixed workspace bootstrap/MEMORY file set. The request is valid for `sessions.list`, but the `agents.files.list` half needs maintainer API-scope judgment because that RPC is not currently an unbounded workspace listing.

## What This Changes

Request to add cursor-style pagination parameters and result metadata to the Gateway `agents.files.list` and `sessions.list` RPCs, following the artifact-list pattern proposed in related work.

## Best Possible Solution

Add an additive cursor contract for `sessions.list` across protocol schema, runtime result shape, shared types, SDK/generated clients, docs, and focused tests. For `agents.files.list`, maintainers should first decide whether the fixed bootstrap-file RPC should intentionally gain pagination fields for consistency or remain documented as a small unpaginated surface.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape gap: both requested schemas reject `cursor`, and the current runtimes return no `nextCursor` or `total` metadata.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a maintainable direction for `sessions.list`, but copying it to `agents.files.list` should wait for maintainer confirmation because that endpoint is currently fixed-scope rather than an unbounded file listing.
