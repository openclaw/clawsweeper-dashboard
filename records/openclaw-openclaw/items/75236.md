## Summary

Keep open. Current main still lacks cursor pagination on both requested Gateway RPCs, and the referenced artifact cursor work remains an open PR rather than a main-branch contract. This is valid API consistency/performance work, but it is not implemented or superseded yet.

## What This Changes

Request cursor-style pagination fields for `agents.files.list` and `sessions.list` to avoid unbounded list responses and align them with the proposed artifact listing contract.

## Best Possible Solution

Define an additive Gateway pagination contract for `sessions.list` across schema, runtime, shared result types, SDK/client models, generated Swift artifacts, docs, and focused tests. For `agents.files.list`, make the API decision explicit: either add compatible pagination metadata for consistency or document the fixed-scope file list as intentionally unpaginated.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape issue: strict validators reject `cursor` on both requested RPCs, and current responses expose neither `nextCursor` nor a pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a good candidate for `sessions.list`, but applying the same contract to fixed-scope `agents.files.list` and reusing an unmerged artifact PR’s helper needs maintainer API direction first.
