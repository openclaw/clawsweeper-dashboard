## Summary

Keep open: current main and the latest release still lack the requested `cursor`/`nextCursor` contract. `sessions.list` now has bounded offset pagination, while `agents.files.list` remains a fixed editable workspace-file surface, so this is a Gateway API direction decision rather than cleanup-close material.

## What This Changes

The issue requests adding cursor-based pagination fields to the Gateway `sessions.list` and `agents.files.list` RPCs.

## Best Possible Solution

Have maintainers choose the Gateway list pagination contract first: standardize `sessions.list` on cursor semantics only if it is better than the current offset contract, and page `agents.files.list` only if that fixed file surface is expected to expand.

## Reproduction Assessment

Yes at source level: current schemas and handlers show no `cursor` or `nextCursor` contract for either requested RPC. This is a missing feature/API contract rather than a bug needing live reproduction.

## Solution Assessment

Unclear. Cursor pagination is plausible for `sessions.list`, but current main deliberately uses offset metadata, and `agents.files.list` may not need pagination while it returns a fixed bootstrap-file set.
