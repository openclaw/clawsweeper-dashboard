## Summary

Keep open: current main and v2026.6.10 still do not expose cursor/nextCursor pagination for either requested Gateway RPC, and adding it would be a public protocol/API decision rather than conservative cleanup.

## What This Changes

Request cursor pagination for `agents.files.list` and `sessions.list` so large workspaces can page those Gateway RPC results consistently.

## Best Possible Solution

Have maintainers choose the Gateway list pagination contract first, keeping or deliberately evolving the current sessions offset contract and only paginating agent files if that surface is expected to grow beyond the fixed editable-file set.

## Reproduction Assessment

Yes, at source level: current schemas and handlers show no `cursor` or `nextCursor` contract for either requested RPC. This is a missing API feature rather than a bug needing live reproduction.

## Solution Assessment

Unclear. Cursor pagination is plausible for `sessions.list`, but current main deliberately uses offset metadata, and `agents.files.list` may not need pagination while it returns a fixed editable workspace-file set.
