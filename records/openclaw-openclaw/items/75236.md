## Summary

Keep open. Current main still lacks cursor pagination for both requested RPCs, and the issue is recent, concrete, and not superseded by the open artifact PR or the closed broader performance report.

## What This Changes

Request additive cursor pagination fields for `agents.files.list` and `sessions.list`, following the artifact-listing pattern proposed in related gateway work.

## Best Possible Solution

Define an additive cursor contract for `sessions.list` across schema, runtime, shared result types, SDK/client models, generated Swift models, docs, and focused tests. For `agents.files.list`, make an explicit API decision: either add compatible pagination metadata for consistency or document that the fixed workspace-file surface remains intentionally unpaginated.

## Reproduction Assessment

Yes. Source inspection against current main is enough to reproduce the API-shape issue: strict validators reject `cursor` for both RPCs, and the runtime responses provide no `nextCursor` or pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a plausible narrow solution for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that method is currently fixed-scope rather than an unbounded listing.
