## Summary

Keep open. Current main still lacks cursor pagination on both requested Gateway RPC surfaces, and the referenced artifact cursor helper is not available on main because the related PR remains open. The request is valid, but the full scope needs maintainer API judgment because `sessions.list` is an unbounded listing while `agents.files.list` currently exposes a small fixed bootstrap-file surface.

## What This Changes

Add cursor-based pagination fields to the `agents.files.list` and `sessions.list` Gateway RPC params and results, following the artifact list cursor pattern.

## Best Possible Solution

Define an additive cursor contract for `sessions.list` across protocol schema, runtime behavior, SDK/client types, generated Swift models, docs, and focused tests. For `agents.files.list`, make an explicit maintainer API decision: either add compatible pagination metadata for consistency or document that the fixed bootstrap-file surface is intentionally unpaginated.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape gap: both strict validators omit `cursor`, and current responses do not expose `nextCursor` or a pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a maintainable additive direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that method is fixed-scope today and the proposed shared cursor helper is not on main.
