## Summary

Keep open: current main and v2026.6.10 still do not expose cursor/nextCursor pagination for either requested RPC, and the remaining work is a Gateway protocol direction decision rather than cleanup closure.

## What This Changes

The issue requests cursor pagination for Gateway `agents.files.list` and `sessions.list`, following the artifact-list cursor pattern proposed in a related PR.

## Best Possible Solution

Have maintainers choose the Gateway list pagination contract first: keep or extend the current sessions offset contract deliberately, and only add agent-file pagination if that file surface is expected to grow beyond the fixed bootstrap set.

## Reproduction Assessment

Yes, at source level: current schemas and handlers show no `cursor` or `nextCursor` contract for either requested RPC. This is a missing API feature rather than a bug needing live reproduction.

## Solution Assessment

Unclear. Cursor pagination is plausible for `sessions.list`, but current main deliberately uses offset metadata, and `agents.files.list` may not need pagination while it returns a fixed editable workspace-file set.
