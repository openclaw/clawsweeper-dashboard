## Summary

Keep open. Current main still has strict, non-cursor schemas for both named RPCs; `sessions.list` only supports a limit slice and returns no `nextCursor` or pre-slice `total`, while `agents.files.list` accepts only `agentId` and returns a fixed bootstrap/MEMORY file set. The related artifact work is relevant context, but current main’s artifact RPCs do not establish the requested shared cursor contract, and `agents.files.list` needs maintainer scope judgment before adding pagination fields.

## What This Changes

Request cursor pagination fields for Gateway `agents.files.list` and `sessions.list`, following the artifact pagination pattern referenced from the related artifact RPC work.

## Best Possible Solution

Define an additive cursor contract for `sessions.list` across schema, runtime, shared types, SDK/client models, generated Swift models, docs, and focused tests. For `agents.files.list`, maintainers should either intentionally add the same fields for protocol consistency or document that the fixed bootstrap/MEMORY file set remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: strict schemas reject `cursor` on both requested RPCs, and the sessions runtime returns only a limit-truncated list without `nextCursor` or a pre-slice `total`.

## Solution Assessment

Unclear for the exact requested scope. Cursor pagination is a reasonable direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer confirmation because that RPC is currently fixed-scope rather than an unbounded workspace listing.
