## Summary

Keep open. Current main still lacks cursor pagination on both requested Gateway RPC surfaces: `sessions.list` accepts filters and `limit` only, while `agents.files.list` still returns a fixed bootstrap-file list with no cursor metadata. The request is valid, but the remaining work needs maintainer protocol judgment because the artifact cursor helper referenced by the issue is not present on main and `agents.files.list` is intentionally small-scope today.

## What This Changes

The issue requests adding cursor-style pagination fields to `agents.files.list` and `sessions.list`, following the cursor pagination pattern proposed for artifact RPC listing.

## Best Possible Solution

Define an additive Gateway list-pagination contract first, then apply it to `sessions.list` across schema, runtime, shared types, SDK/client models, generated Swift models, docs, and focused tests. For `agents.files.list`, make an explicit API decision: either add compatible `limit`/`cursor`/`nextCursor`/`total` metadata for consistency or document that the fixed bootstrap-file surface intentionally remains unpaginated.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape issue: strict validators reject `cursor` for both requested RPCs, and current responses provide neither `nextCursor` nor a pre-slice `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a reasonable additive direction for `sessions.list`, but applying it to fixed-scope `agents.files.list` should wait for maintainer confirmation and a settled cursor contract.
