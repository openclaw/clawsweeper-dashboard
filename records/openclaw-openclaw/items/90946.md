## Label Justifications

- `P1`: The PR addresses a gateway availability loop where chat requests can fail with HTTP 000 after a child gateway kills the supervised parent.
- `merge-risk: 🚨 availability`: Changing stale-process cleanup can leave a supervised gateway alive or prevent a child from freeing the port, so the invariant needs maintainer acceptance beyond green tests.

## Summary

This PR should stay open for maintainer review: current main still overwrites the inherited gateway service PID before stale cleanup, while the branch applies a focused service-mode exclusion with strong real-host proof. The remaining question is maintainers accepting the availability-sensitive invariant that a valid inherited supervised gateway PID must not be killed by service-mode startup cleanup.

## What This Changes

The PR strictly parses an inherited gateway service PID before gateway startup overwrites the env var, threads it into stale gateway PID cleanup, and adds parser plus reparent-regression tests.

## Best Possible Solution

Merge the bounded service-mode exclusion if maintainers accept that startup cleanup must not kill an inherited supervised parent, while keeping explicit restart/update/doctor cleanup paths able to terminate the running gateway.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from source and supplied real-host proof: launchd-managed gateway, Codex 0.135 zsh shell snapshot sourcing a legacy `.zshrc` auto-start block, a reparented child with `process.ppid === 1`, and inherited `OPENCLAW_GATEWAY_SERVICE_PID` matching the victim. I did not rerun the live host scenario in this read-only review.

## Solution Assessment

Yes, this appears to be the best bounded fix: capture the inherited service PID before overwriting the env var and pass it only to the cleanup owner that decides the kill set. Broader host-config cleanup would not harden the OpenClaw primitive, and changing explicit restart/update cleanup would be too broad.
