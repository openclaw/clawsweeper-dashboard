## Summary

Keep open: current main still drops the inherited service PID before cleanup, while this PR adds a narrow opt-in protection with focused tests and sufficient real-host proof. I found no blocking patch defect, but the cleanup semantic is availability-sensitive and still needs normal maintainer merge judgment.

## What This Changes

The PR captures an inherited gateway service PID before `gateway run` overwrites it and passes that PID through stale gateway cleanup so a reparented child gateway does not kill the supervised parent.

## Best Possible Solution

Land the opt-in inherited-PID protection after maintainer acceptance and normal required-check gating, while keeping explicit update, doctor, and daemon restart cleanup paths unprotected as the PR does.

## Reproduction Assessment

Yes. Current main overwrites the inherited gateway PID before cleanup and only excludes visible ancestors, while the PR body supplies live macOS launchd logs for the failing and fixed behavior; I did not rerun the host-specific scenario locally.

## Solution Assessment

Yes. The PR fixes the narrow owner boundary by capturing the existing runtime PID before it is overwritten and making the protection opt-in at the service-mode gateway call site instead of weakening explicit restart/update cleanup paths.
