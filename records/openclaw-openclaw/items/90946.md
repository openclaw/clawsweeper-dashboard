## Summary

Keep open for maintainer review: the PR is still useful, current main does not contain the inherited-PID protection, and the patch has no blocking correctness finding. The remaining merge question is whether maintainers accept the availability-sensitive service-mode cleanup invariant.

## What This Changes

The PR captures an inherited `OPENCLAW_GATEWAY_SERVICE_PID` before `gateway run` overwrites it, threads that PID through stale gateway cleanup, and adds parser plus reparent-regression tests.

## Best Possible Solution

Land the opt-in inherited-PID protection after maintainer acceptance and normal required-check gating, while keeping update, doctor, and daemon restart cleanup paths intentionally unprotected.

## Reproduction Assessment

Yes. Source inspection shows current main overwrites the inherited gateway PID before cleanup and only excludes visible ancestors, while the PR body/comment provide macOS launchd before/after logs; I did not rerun the host-specific scenario locally.

## Solution Assessment

Yes. The PR fixes the narrow owner boundary by preserving the inherited runtime PID at `gateway run` and passing it through cleanup only for that service-mode startup path, instead of weakening explicit update/doctor/restart cleanup.
