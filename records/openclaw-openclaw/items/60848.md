## Summary

Keep this PR open: current main still does not implement the inbound message_id fallback, but the branch is not merge-ready because the fallback can persist unsafe current-message anchors and the current native payload path remains outside the shared reply-mode fix; the PR also lacks real Feishu behavior proof.

## What This Changes

The PR updates the Feishu plugin to persist a message_id-backed MessageThreadId for topic sessions without root_id and to pass replyInThread through threadId fallback text/card/media paths, with Feishu unit coverage.

## Best Possible Solution

Update the Feishu plugin on current main so MessageThreadId is only persisted from a safe root/message anchor, thread_id-only topics are resolved or left unanchored safely, and one trimmed reply-mode helper covers text, markdown-card, structured-card, native payload, media, and local-image sends with focused tests and real Feishu proof.

## Reproduction Assessment

Yes, at source level. The code path is group_topic or replyInThread enabled with no root_id, plus either a thread_id-only topic event or a native payload send using whitespace replyToId with a valid threadId fallback.

## Solution Assessment

No. The PR is in the right Feishu plugin surface, but it needs a current-main rebase with guarded anchor selection, native payload reply-mode propagation, and real Feishu behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid current-message anchors for thread_id-only topics:** `extensions/feishu/src/bot.ts:1017-1019`
  - body: This fallback stores the current message_id as MessageThreadId for every topic session without root_id when replyInThread is enabled. Feishu topic events can have thread_id but no root_id, so later async or subagent replies can use the current message as the delivery thread anchor and start a new topic instead of staying in the existing one. Gate this to true starter/no-thread events or resolve the om_* root from thread_id before persisting it.
  - confidence: 0.88
- **[P2] Propagate thread replies through native payload sends:** `extensions/feishu/src/outbound.ts:572-589`
  - body: The current-main sendPayload path resolves a reply target but does not pass replyInThread to payload media sends, and its final native card still uses the old untrimmed ctx.threadId check. With whitespace replyToId plus a valid threadId, native Feishu card/media payloads can still post outside the intended topic thread; use the same reply-mode helper here and add coverage.
  - confidence: 0.86
