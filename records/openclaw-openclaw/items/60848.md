## Summary

Keep open. Current main still has the Feishu topic-thread anchoring gap this PR targets, but the proposed patch is incomplete because the native payload/card path can still use `threadId` as the reply target without consistently propagating `replyInThread`; related PR #67069 also overlaps inbound Feishu thread metadata and should be coordinated before landing.

## What This Changes

The PR updates Feishu inbound topic context to persist a message-level `MessageThreadId` fallback and propagates `replyInThread` through outbound threadId fallback text, card, media, and local-image sends with focused tests.

## Best Possible Solution

Update the PR on current main, keep the guarded no-root `message_id` fallback for topic sessions, centralize one trimmed `replyInThread` decision, apply it to text, markdown-card, structured-card, native payload, media, and local-image sends, and add regression coverage for native payload `threadId` fallback.

## Reproduction Assessment

Yes. The issue is reproducible at unit level by configuring Feishu `groupSessionScope: "group_topic"` with `replyInThread: "enabled"`, sending a no-`root_id` topic starter, and exercising native payload sends with whitespace `replyToId` plus `threadId` fallback.

## Solution Assessment

No. The PR is the right narrow area and fixes several paths, but it is not yet the best complete fix because `sendPayload` still needs the same trimmed `replyInThread` propagation and coverage.
