## Summary

Keep open: this is a real Feishu thread-affinity fix area, but the PR is not ready because the current fallback can persist the current message as a thread anchor for thread_id-only topic events, native payload sends still miss the same replyInThread mode, and the external PR has only mocked/unit-test proof.

## What This Changes

The PR changes the Feishu plugin to persist a message_id-backed MessageThreadId for topic sessions without root_id and to pass replyInThread through threadId fallback text, card, media, and local-image outbound paths with tests.

## Best Possible Solution

Update the Feishu plugin on current main so thread anchors are only persisted when they are safe, thread_id-only topics resolve or omit unsafe root anchors, and one trimmed reply-mode decision covers text, markdown-card, structured-card, native payload, media, and local-image sends before merge.

## Reproduction Assessment

Yes. A high-confidence source-level path exists with group_topic or replyInThread enabled, no root_id, and either a thread_id-only topic event or a native payload send using whitespace replyToId plus threadId fallback.

## Solution Assessment

No. The PR is in the right Feishu plugin surface and fixes useful paths, but it needs a current-main rebase that avoids unsafe message_id anchors for thread_id-only topics, covers native payload sends, and adds real Feishu behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid current-message anchors for thread_id-only topics:** `extensions/feishu/src/bot.ts:1017-1019`
  - body: This fallback stores the current message_id as MessageThreadId for any topic session without root_id. Feishu topic messages can have thread_id but no root_id, so later async/subagent replies can use the current message as the delivery thread anchor and create a new topic instead of staying in the existing one. Gate the fallback to starter/no-thread cases or resolve the root om_* from thread_id before persisting it.
  - confidence: 0.88
- **[P2] Propagate thread replies through native payload sends:** `extensions/feishu/src/outbound.ts:572-589`
  - body: The current-main sendPayload path resolves a reply target but does not pass replyInThread to payload media sends, and the final native card still uses the old untrimmed ctx.threadId check. With whitespace replyToId plus a valid threadId, native Feishu card/media payloads can still post outside the intended topic thread; use the same reply-mode helper here and add coverage.
  - confidence: 0.86
