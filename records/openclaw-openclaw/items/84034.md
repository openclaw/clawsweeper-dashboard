## Summary

Keep open: current main and the latest release still accept arbitrary nonempty active-memory recall text unless it matches a narrow no-recall or timeout list, so the reported assistant chatter can become hidden prompt context.

## What This Changes

The issue reports that active-memory recall can accept assistant-style no-recall chatter as a memory summary and inject it into the hidden active-memory prompt context.

## Best Possible Solution

Harden `normalizeActiveSummary` now with focused parser-side rejection for assistant chatter on both normal and timeout-partial paths, and leave a stricter structured recall protocol as a later design follow-up.

## Reproduction Assessment

Yes, source-level: a reported assistant-style recall string that is not in the no-recall or timeout lists will be normalized into a summary and wrapped into the active-memory prompt prefix. I did not run a live GLM reproduction, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes, parser-side validation in `normalizeActiveSummary` is the narrowest maintainable first fix because both normal recall and timeout-partial recovery already funnel through it. A JSON-only recall protocol would be stronger but is larger than this immediate bug fix.
