## Summary

Keep open: current main and the latest release still treat arbitrary nonempty recall output as a valid summary unless it matches known NONE or timeout values; the reporter's fresh live examples fit that parser gap.

## What This Changes

The issue requests active-memory recall to reject assistant-style no-recall chatter instead of injecting it into hidden active_memory_plugin context.

## Best Possible Solution

Fail closed in normalizeActiveSummary on assistant chatter and metadata-restatement variants across normal and timeout-partial paths, with regression coverage; leave strict JSON recall protocol hardening as a follow-up.

## Reproduction Assessment

Yes, as a source-level reproduction: any reported chitchat string that is not a known NONE or timeout value will be returned by normalizeActiveSummary and can be wrapped as active-memory context. I did not run a live GLM turn in this read-only pass.

## Solution Assessment

Yes, parser-side fail-closed validation is the narrowest immediate fix because both normal recall and timeout-partial recovery already share normalizeActiveSummary. A structured JSON protocol would be a stronger later hardening step, not a prerequisite for containment.
