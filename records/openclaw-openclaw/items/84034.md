## Summary

Keep open: current main still accepts arbitrary nonempty active-memory recall text after only no-recall and timeout-boilerplate filtering, so the reported assistant chitchat can still become hidden prompt context. No open or merged PR appears to cover this exact parser hardening, while the related injected-context registry RFC is broader and not a substitute for the narrow bug fix.

## What This Changes

The issue reports that active-memory recall can inject assistant-style boilerplate into hidden memory context instead of returning NONE or a compact factual summary.

## Best Possible Solution

Harden normalizeActiveSummary to fail closed on obvious assistant chatter before metadata injection, with regression tests for both normal recall and timeout-partial recovery; leave a structured JSON recall protocol as a separate follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: any nonempty assistant-style recall output that is not a known no-recall value or timeout string is normalized into a summary and injected. I did not run the live GLM provider path in this read-only review.

## Solution Assessment

Yes, parser-side hardening is the narrowest maintainable immediate fix because both normal and timeout-partial recall already funnel through normalizeActiveSummary. A structured response protocol would be stronger but is larger than needed for the first regression fix.
