## Summary

Keep open: current main still accepts arbitrary nonempty recall text unless it is a known no-recall value or timeout boilerplate, so the reported assistant-style chatter can pass source-level normalization into the hidden prompt prefix.

## What This Changes

The issue requests active-memory recall validation so assistant boilerplate from the recall subagent is rejected instead of wrapped in `<active_memory_plugin>`.

## Best Possible Solution

Harden `normalizeActiveSummary` with focused parser-side rejection for assistant chatter now, with regression tests for normal and timeout-partial paths; consider a strict structured recall protocol as a later follow-up.

## Reproduction Assessment

Yes, from source inspection: current main normalizes raw subagent output by rejecting only no-recall and timeout values, then injects any remaining nonempty single-line text. I did not run a live GLM reproduction, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes, the recommended parser-side hardening is the narrowest maintainable first fix because both normal and timeout-partial paths already funnel through `normalizeActiveSummary`. A structured JSON protocol would be stronger but is larger than the immediate regression fix.
