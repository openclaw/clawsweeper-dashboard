## Summary

Keep open: this PR is the active implementation candidate for https://github.com/openclaw/openclaw/issues/80234, but the current diff can suppress an explicit Discord mention, lacks contributor-provided real behavior proof, and adds a new Discord config API that needs maintainer review.

## What This Changes

The PR adds `channels.discord.implicitReplyMentions` with `fromUsers` and `fromBots`, threads it through Discord preflight implicit reply detection, and updates config schema, generated metadata, docs, tests, and changelog.

## Best Possible Solution

Fix the explicit-mention suppression case, add regression coverage for a bot-authored native reply that also contains `<@botId>`, then land only after maintainer API review and redacted real Discord proof.

## Reproduction Assessment

Yes, source-level: current main treats `referencedAuthorId === botId` as `reply_to_bot`, and `allowBots="mentions"` can accept that implicit mention. I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No for the current patch: the config direction is plausible, but this implementation checks `baseText` after mention normalization and can suppress a real explicit mention. The safer path is to preserve raw-content explicit mentions, add the missing regression test, and get maintainer API review plus real behavior proof.
