## Summary

Keep this PR open. The supplied live Discord proof is now adequate, but the current PR head still has a blocking correctness issue: reply-only suppression checks normalized `baseText`, so a native reply that also contains a real textual bot mention can be misclassified as reply-only and dropped.

## What This Changes

The PR adds `channels.discord.implicitReplyMentions` for Discord so deployments can separately allow user-authored and bot-authored native replies to count as implicit mentions.

## Best Possible Solution

Preserve raw textual bot mentions before suppression, add reply-plus-explicit-mention regression coverage, then land only if maintainers accept the new Discord config API.

## Reproduction Assessment

Yes, source-level: current main treats `referencedAuthorId === botId` as `reply_to_bot`, and the PR discussion includes live baseline proof that a bot-authored native reply-only message woke the target bot. I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No for the current patch: the config direction is plausible, but checking normalized `baseText` can suppress a real explicit mention on a native reply. The safer fix is to use raw message content or an equivalent pre-normalized signal and add the missing regression test.
