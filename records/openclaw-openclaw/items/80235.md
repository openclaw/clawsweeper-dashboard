## Summary

Keep open: the config direction is plausible and the supplied real Discord proof is sufficient, but the current PR can still drop a native reply that contains a real textual bot mention because the suppression predicate checks normalized text.

## What This Changes

Adds `channels.discord.implicitReplyMentions` config, schema, docs, generated metadata, changelog, and Discord preflight tests for user- and bot-authored native reply activation.

## Best Possible Solution

Preserve raw textual bot mentions before reply-only suppression, add native-reply-plus-explicit-mention coverage, then decide whether the new Discord config API is acceptable.

## Reproduction Assessment

Yes, source-level and supported by supplied live proof: current main treats a Discord native reply whose referenced author is the bot as an implicit `reply_to_bot` mention. I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

No for the current patch: the config direction is plausible, but checking normalized `baseText` can suppress a real explicit `<@botId>` mention on a native reply. The safer fix is to check raw `message.content` or an equivalent pre-normalized signal and add regression coverage.
