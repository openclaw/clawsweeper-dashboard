## Summary

Keep this PR open: the contributor has now provided usable real Discord proof for the central multi-bot behavior, but the latest head still has a correctness blocker in the explicit-mention preservation path.

## What This Changes

The PR adds `channels.discord.implicitReplyMentions` with `fromUsers` and `fromBots`, threads it into Discord preflight mention handling, and updates config schema, generated metadata, docs, tests, and changelog.

## Best Possible Solution

Preserve raw textual bot mentions while suppressing only reply-generated mentions, add the missing regression test, and then land the config API only after maintainer approval.

## Reproduction Assessment

Yes, source-level: current main routes `referencedAuthorId === botId` into `reply_to_bot`, and the latest PR head still checks normalized `baseText` when deciding whether a reply mention is textual. I did not run a live Discord gateway repro in this read-only review.

## Solution Assessment

No for the current patch: the config direction is plausible, but the implementation still suppresses a real explicit `<@botId>` mention when it appears on a native reply. The safer fix is to use raw message content or an equivalent raw explicit-mention signal for the suppression decision.
