## Summary

Keep open. Current main has a structured workspace-status doctor check, but the underlying legacy-workspace detector currently always returns no legacy directories, while the docs still promise that doctor warns about extra workspaces; the reported upgrade/data-loss path is not proven fixed on main.

## What This Changes

The issue asks OpenClaw to prevent upgrade-created phantom legacy/rebranded workspace directories and keep file tools, memory flush writes, exec cwd, and UI paths pointed at the same active workspace.

## Best Possible Solution

Keep this open for a focused maintainer investigation that makes the active workspace/state root canonical during upgrades, detects parallel legacy/rebranded directories, and verifies memory-flush writes cannot target a different root from the runtime workspace.

## Reproduction Assessment

No high-confidence runtime repro is available from current main; the issue's upgrade timing is explicitly hard to trigger. Source inspection does show the current doctor workspace-status path cannot detect extra legacy workspace directories because its detector always returns an empty list.

## Solution Assessment

Unclear. The right fix likely combines real legacy/rebrand directory detection with a memory-flush/root invariant, but the exact creator of the phantom directory still needs investigation before choosing migration versus refusal behavior.
