## Summary

Keep open. Current main and v2026.6.10 already provide implicit Linux no-display headless fallback plus actionable text for explicit headed mode, but the browser route/client path still reduces this case to a string-only error instead of the requested machine-readable reason/remediation contract.

## What This Changes

The issue requests structured degradation or policy-controlled headless fallback when managed browser profiles request headed launch in an environment without a display server.

## Best Possible Solution

Add a narrow typed no-display profile-unavailable result for explicit headed no-display launches while preserving the existing implicit fallback unless maintainers approve broader fallback policy.

## Reproduction Assessment

Yes. Source inspection shows explicit headed local managed launches on Linux without DISPLAY/WAYLAND_DISPLAY throw an actionable browser-domain error, then route/client helpers expose only a string error instead of the requested structured reason/remediation object.

## Solution Assessment

No, current main is only a partial solution. The default fallback path is useful, but the explicit headed no-display case still lacks a machine-readable result and the fallback-vs-typed-error choice needs maintainer API/product direction.
