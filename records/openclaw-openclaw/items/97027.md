## Summary

Keep open. Current main already has implicit Linux no-display headless fallback plus actionable explicit-headed error text, but the browser route/client/tool surface still reduces this case to string-only errors instead of the requested machine-readable degradation result.

## What This Changes

The issue requests structured degradation or policy-controlled fallback when a managed headed browser profile runs without a display server.

## Best Possible Solution

Add a narrow typed no-display profile-unavailable result for explicit headed no-display launches while preserving the existing implicit headless fallback unless maintainers approve a broader fallback policy.

## Reproduction Assessment

Yes, source-level. On Linux without `DISPLAY` or `WAYLAND_DISPLAY`, explicit headed configuration reaches `BrowserProfileUnavailableError`, then route/client error handling reduces the result to string-only JSON errors rather than a typed reason/remediation shape.

## Solution Assessment

No, current main is only a partial solution. The default fallback path is useful, but the explicit headed no-display case still lacks a machine-readable result and the fallback-vs-typed-error choice needs maintainer API/product direction.
