## Summary

Keep open. Current main and v2026.6.10 already have implicit Linux no-display headless fallback plus actionable text for explicit headed launches, but the browser route, gateway, and client surfaces still collapse the case to string errors rather than the requested machine-readable no-display reason and remediation contract.

## What This Changes

The issue requests clean degradation for managed browser launches when a headed profile runs without a display server, either via controlled headless fallback or a structured no-display error.

## Best Possible Solution

Add a narrow typed no-display unavailable result and remediation for explicit headed no-display launches while preserving the existing implicit fallback unless maintainers approve a broader fallback policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on Linux without DISPLAY/WAYLAND_DISPLAY, explicitly set `browser.headless=false`, `browser.profiles.<name>.headless=false`, or `OPENCLAW_BROWSER_HEADLESS=0`, then hit a browser start/tool route and observe a message-only no-display failure.

## Solution Assessment

Unclear. Current main is a useful partial solution, but the safest remaining path is a typed error/remediation contract unless maintainers explicitly approve policy-controlled auto-fallback for explicit headed profiles.
