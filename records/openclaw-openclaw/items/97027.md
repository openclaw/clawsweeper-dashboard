## Summary

Keep open. Current main and v2026.6.10 already provide implicit Linux no-display headless fallback plus actionable error text for explicit headed mode, but browser HTTP/tool errors still collapse this case to `{ error: string }`, so the requested machine-readable reason/remediation remains unimplemented and needs maintainer API/product direction.

## What This Changes

The issue requests structured degraded behavior or a policy-controlled headless fallback when a managed headed browser profile starts on a host without `DISPLAY` or `WAYLAND_DISPLAY`.

## Best Possible Solution

Add a narrow typed no-display profile-unavailable result for explicit headed no-display launches while preserving the existing implicit headless fallback and avoiding silent fallback unless maintainers approve that policy.

## Reproduction Assessment

Yes, source-level. On Linux without `DISPLAY`/`WAYLAND_DISPLAY`, explicit headed configuration reaches `getManagedBrowserMissingDisplayError`, then the route layer maps the resulting browser error to a plain `{ error: string }` response rather than the requested typed reason/remediation shape.

## Solution Assessment

No, current main is only a partial solution. The default fallback path is good, but the explicit headed no-display case still lacks a machine-readable result and the fallback-vs-typed-error choice needs maintainer API/product direction.
