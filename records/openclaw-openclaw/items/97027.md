## Summary

Keep open. Current main and v2026.6.10 partially handle this by falling back to headless when no display is present by default and by producing an actionable text error for explicit headed mode, but the browser HTTP/client/tool surfaces still collapse that case to a plain string error rather than a machine-readable reason/remediation shape.

## What This Changes

Request a structured managed-browser no-display degraded result, or a policy-controlled headless fallback, when headed launch is configured without a display server.

## Best Possible Solution

Add a narrow typed no-display unavailable result with remediation for explicit headed no-display launches, preserve the existing implicit headless fallback, and add silent fallback only if maintainers approve that policy.

## Reproduction Assessment

Yes, at source level. On Linux without DISPLAY/WAYLAND_DISPLAY, explicit headed configuration reaches BrowserProfileUnavailableError, but the route and client layers reduce that to a plain string error instead of the requested typed reason/remediation result.

## Solution Assessment

No, current main is only a partial solution. The implicit fallback and actionable text are useful, but the stable structured error shape and fallback policy still need maintainer direction.
