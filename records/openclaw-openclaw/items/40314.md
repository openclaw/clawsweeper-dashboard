## Summary

Keep open. Current main still drops messages added by internal `message:received` and `message:sent` hooks, and this PR is a plausible fix path, but it still needs real behavior proof, conflict/check recovery, and a maintainer scope decision for `message:sent` auto-delivery before merge.

## What This Changes

The branch drains `message:received` hook messages through routed replies, drains `message:sent` hook messages through outbound delivery with hook-loop suppression, persists `skipMessageHooks`, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land one maintained, rebased fix that drains hook messages only on the accepted replyable surface contract, preserves loop suppression and policy/session context, updates docs/tests to match, and closes https://github.com/openclaw/openclaw/issues/39688 from the merge.

## Reproduction Assessment

Yes, at source level. Registering an internal `message:received` or `message:sent` hook that pushes into `event.messages` exercises current-main paths that trigger the hook but never drain those messages into delivery.

## Solution Assessment

Mostly yes, but not fully settled. The PR’s session, policy, queue, and loop-suppression mechanics are aligned with the bug, while maintainers still need to choose whether `message:sent` should auto-deliver hook replies or whether the narrower replyable-surface contract in https://github.com/openclaw/openclaw/pull/41375 is preferred.
