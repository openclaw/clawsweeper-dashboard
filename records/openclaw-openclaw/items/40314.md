## Summary

Keep open. Current main still drops internal `message:*` hook reply messages, and this PR is a plausible fix path, but it still needs real behavior proof plus maintainer reconciliation with the narrower open hook-reply PR and current main’s durable-send routing.

## What This Changes

The PR delivers `message:received` hook replies through routed replies, delivers `message:sent` hook replies through outbound delivery, and persists hook-loop suppression through queued recovery.

## Best Possible Solution

Land one reconciled hook-reply contract on current main that drains supported `event.messages` with session/send-policy context, suppresses hook loops durably, updates docs/tests, and then closes the linked bug.

## Reproduction Assessment

Yes, at source level. Registering a `message:received` or `message:sent` internal hook that pushes text into `event.messages` exercises current-main paths that trigger the hook but never drain those messages into reply or outbound delivery.

## Solution Assessment

Mostly yes. Draining at the existing reply/outbound boundaries with session policy and `skipMessageHooks` is narrow, but the final solution should reconcile the `message:sent` scope with the open maintainer PR and current durable-send path.
