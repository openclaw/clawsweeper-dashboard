## Summary

Keep open. Current main still drops messages added by internal message hooks, and this PR is a plausible implementation path, but merge should wait for real behavior proof, CI/conflict cleanup, and maintainer scope reconciliation with the narrower open hook-reply PR.

## What This Changes

The PR drains `message:received` and `message:sent` internal hook `event.messages` into reply/outbound delivery, suppresses recursive message hooks, persists that suppression through queued recovery, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land one reconciled hook-reply contract on current main that drains supported `event.messages` with session/send-policy context, suppresses hook loops durably, updates docs/tests, and then closes the linked bug.

## Reproduction Assessment

Yes, at source level. Registering a `message:received` or `message:sent` internal hook that pushes into `event.messages` exercises current-main paths that emit the hook but never drain those messages into delivery.

## Solution Assessment

Mostly yes. Draining at reply/outbound boundaries with session policy and `skipMessageHooks` is narrow, but maintainers should reconcile this PR's broader `message:sent` auto-delivery with https://github.com/openclaw/openclaw/pull/41375 before landing.
