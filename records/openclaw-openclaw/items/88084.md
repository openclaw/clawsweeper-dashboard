## What This Changes

The PR changes core auto-reply command routing so authorized text `/approve` can bypass active reply-operation admission and enter the native slash fast path, with focused tests for the new approval path.

## Best Possible Solution

Refresh the branch and land a narrow fix that only bypasses for `/approve` turns the command dispatcher will actually handle, with regression coverage for `commands.text=false` native surfaces and direct redacted behavior proof.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current main only fast-paths native slash commands and still uses `dispatchOperationSessionKey` for active-run admission, so text `/approve` can be blocked behind the active approval turn. The PR's unsafe fallback is also source-proven when command-source metadata is missing and text commands are disabled.

## Solution Assessment

No, not as written: the owner boundary is plausible, but the bypass and fast path must be gated to approval commands the text-command dispatcher will actually handle. The safer fix is to reuse the existing `shouldHandleTextCommands` decision and add focused tests for the disabled-text native-surface case.
