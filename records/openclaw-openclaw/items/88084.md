## Summary

Keep open: current main still does not implement the approval reply-lane bypass, and this PR is not merge-ready because the source-missing `ctx.CommandAuthorized` fallback can bypass reply-operation admission even when the text command router will not handle `/approve`. The branch also lacks direct current-branch real behavior proof and is currently conflicting with `main`.

## What This Changes

This PR adds a `/approve` reply-operation bypass, allows authorized text `/approve` through the native slash fast path, and adds focused regression tests.

## Best Possible Solution

Land a rebased narrow approval-command bypass that only skips admission when the normal approve handler will consume the command, with focused tests and direct redacted real-channel proof.

## Reproduction Assessment

Yes from source inspection, but not from a live run in this read-only review: current main only fast-paths native slash commands and still uses the active reply-operation lane for text approval replies. The PR bug path is source-reproducible through `commands.text=false` plus missing `CommandSource`, but needs live proof before merge.

## Solution Assessment

No as written: the narrow bypass is the right kind of fix, but it must share the text-command routing gate so source-missing authorized `/approve` texts that handlers will not consume do not bypass admission. A rebased patch with that guard, focused tests, and direct proof would be the maintainable path.
