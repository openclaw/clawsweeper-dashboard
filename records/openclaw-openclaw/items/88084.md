## Summary

Keep open: the branch targets a real source-reproducible approval-command lane problem, but it is not merge-ready because the bypass is too broad, the head currently conflicts with `main`, and direct after-fix real behavior proof is still missing.

## What This Changes

The branch adds a `/approve`-specific reply-operation bypass, lets authorized text `/approve` enter the slash fast path, and adds focused approval-routing tests.

## Best Possible Solution

Rebase onto current `main` and land a narrow approval-command bypass only for command turns the command router will actually handle, with focused tests and redacted real-channel proof.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current code serializes reply admission by session and the `/approve` handler only runs when text commands are allowed. The PR’s failure mode is also source-reproducible from the new bare `CommandAuthorized` fallback plus the existing `shouldHandleTextCommands` gate.

## Solution Assessment

No as submitted: the owner boundary is plausible, but the bypass must be gated to command turns that will actually be handled and rebased over the newer fast-path code on `main`. The safer fix is to thread the text-command routing check into both the admission bypass and fast-path decision.
