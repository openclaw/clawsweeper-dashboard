## Summary

Keep open: the topic routing and cutoff pieces address real current-main gaps, but the patch is not merge-ready because the new cancellation predicate only reaches `deliverReplies()` while default Telegram final sends try durable delivery first. The PR also still lacks a fresh after-fix Telegram topic stop run.

## What This Changes

The PR updates Telegram topic detection, native stop cutoff metadata, in-flight reply delivery cancellation, runtime import retry caching, and related Telegram/abort regression tests.

## Best Possible Solution

Thread cancellation through the default durable Telegram final-delivery path or fall back when superseded, keep the narrow topic/cutoff fixes, and require redacted live Telegram proof before landing.

## Reproduction Assessment

Yes, source-level: current main ignores message-level `is_topic_message` in message/native command forum resolution while existing sequential-key routing already uses it as a topic signal. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not yet: the topic-routing and cutoff approach is maintainable, but the cancellation fix needs to cover durable final delivery, which is the default Telegram final-send path when supported.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Carry cancellation into durable final delivery:** `extensions/telegram/src/bot-message-dispatch.ts:1052`

Pass the fence signal or predicate through the durable delivery path, or explicitly fall back when superseded.
