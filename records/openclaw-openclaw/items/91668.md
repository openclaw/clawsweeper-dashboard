## Summary

Keep this PR open, but do not merge it yet: reusing the canonical run-liveness policy is the right fix boundary, while the new stale branch can strand a run if terminal finalization fails, and the supplied proof does not execute the changed recovery behavior.

## What This Changes

The PR applies the existing subagent liveness policy during restart recovery, finalizes stale orphaned runs, adds three regression tests, and documents the new behavior.

## Best Possible Solution

Retain the canonical liveness gate, but give stale-run finalization exactly one owner that remains retryable on failure, then prove an actual restart-recovery flow reaches terminal state and notifies the parent without resurrecting stale work.

## Reproduction Assessment

Yes, from source inspection: pass a stale restart-aborted run and make `finalizeInterruptedSubagentRun` reject; the PR returns `failed=0` with no `failedRuns` entry, so the scheduler does not retry or finalize it later. The originally reported stale-resume behavior is also directly visible on current main because recovery lacks the shared liveness gate.

## Solution Assessment

No, not yet. Reusing `isLiveUnendedSubagentRun` is the narrowest maintainable policy choice, but the stale finalization failure must remain under a retryable single owner rather than being swallowed.
