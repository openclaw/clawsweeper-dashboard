## Summary

Keep open. Current main and the linked test-only PR make the originally suspected cache-missing paired-token path look already handled or unreproduced, but the reporter’s release symptom shifted to gateway service or duplicate listener state and this issue does not yet have the post-reboot diagnostics needed to prove it is fully covered by another open issue.

## What This Changes

The issue asks for `openclaw tui` to reconnect after reboot without re-approving scopes when the device remains paired and `paired.json` already contains valid approved tokens.

## Best Possible Solution

Keep this open for the focused post-reboot gateway lifecycle diagnostics; if those logs match an existing systemd stale-listener issue, route this as a duplicate, otherwise add a targeted regression and fix in the implicated auth or daemon path.

## Reproduction Assessment

No, not for the full reported reboot symptom on current main. The targeted cache-missing paired-token path has live proof in the related test PR that it does not reproduce, while the newer stale/duplicate gateway theory still needs the requested post-reboot diagnostics.

## Solution Assessment

Unclear. Loading directly from `paired.json` is not the safest first fix because current code separates client token cache from server pairing state; the narrower maintainable path is to identify whether the active gateway listener or systemd ownership state is wrong before changing auth recovery.
