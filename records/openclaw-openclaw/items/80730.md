## Summary

Keep open. The report is plausible against current source: the TUI client only reloads its local `identity/device-auth.json` cache, while the gateway’s approved token lives in `devices/paired.json`; the exact post-reboot state still needs a focused reproduction before changing auth recovery behavior.

## What This Changes

The issue reports that `openclaw tui` asks for scope re-approval after reboot even though the existing paired device entry still has approved tokens.

## Best Possible Solution

Add a focused regression for the TUI restart/cache-missing state, then fix the approved local recovery path without weakening scope-upgrade approval or device-token boundaries.

## Reproduction Assessment

Yes, but only from source inspection so far. The high-value repro is a paired TUI identity whose `devices/paired.json` has an approved operator token while the local `identity/device-auth.json` cache is absent or stale, then starting `openclaw tui` after restart.

## Solution Assessment

Unclear. The issue’s proposed direction of loading from `paired.json` may solve the symptom, but the safer solution needs maintainer sign-off because client-token recovery and server pairing storage are separate auth surfaces.
