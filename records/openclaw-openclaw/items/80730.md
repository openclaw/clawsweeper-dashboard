## Summary

Keep open. The cache-missing token theory is weakened by current source and the related draft test PR, but the reporter's latest evidence points to a stale or duplicate gateway listener after reboot/update, and that path still needs the requested post-reboot diagnostics before it can be closed or routed to a canonical lifecycle issue.

## What This Changes

The issue reports that `openclaw tui` asks for renewed scope approval after reboot even though the device remains paired and has approved tokens in `paired.json`.

## Best Possible Solution

Keep this open until the requested reboot diagnostics identify the gateway owner/listener state; then either route it to the matching lifecycle issue or add a focused regression and fix for the confirmed TUI/gateway auth path.

## Reproduction Assessment

No. The reporter gave a concrete v2026.5.7 sequence and current source explains both the token-recovery path and stale-PID diagnostics, but there is not yet a high-confidence current-main or post-reboot reproduction tying the TUI re-approval to one root cause.

## Solution Assessment

Unclear. Loading directly from `paired.json` is probably not the right first fix because current main already reissues and stores paired device tokens through `hello-ok`; the safer next step is to prove or disprove the stale/duplicate gateway listener path before changing auth recovery.
