## Summary

Keep open. The original TUI token-cache theory is not proven against current source and is covered by a draft test-only PR, but the reporter’s latest evidence shifts the unresolved symptom toward Linux gateway service ownership or duplicate listener state after boot.

## What This Changes

The issue reports that after a Linux reboot, `openclaw tui` asks for scope re-approval even though the device fingerprint and approved token state appear to persist in `devices/paired.json`.

## Best Possible Solution

Keep this issue as the canonical thread, collect the post-reboot gateway/service diagnostics, then make a narrow fix to the proven gateway ownership or auth recovery path with focused regression coverage.

## Reproduction Assessment

Unclear. The reporter supplied a release-state reboot sequence and stable device fingerprint, but the targeted current-source cache-missing path was reported unreproducible in https://github.com/openclaw/openclaw/pull/80742 and the newer duplicate-listener hypothesis still needs post-reboot status/process/journal output.

## Solution Assessment

Unclear. Loading directly from `devices/paired.json` is not established as the safest fix because current source can reissue approved device tokens; the next maintainable step is to prove whether gateway service ownership or duplicate listener state is causing the scope-upgrade request.
