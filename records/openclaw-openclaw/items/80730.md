## Summary

Keep open. Current source and the related test-only PR make the originally suspected cache-missing TUI token path look unlikely, but the reporter later supplied fresh release context pointing toward gateway service ownership or duplicate listener state after reboot, which is not yet reproduced or ruled out on current main.

## What This Changes

The issue reports that `openclaw tui` asks for scope re-approval after reboot even though the TUI device remains paired and has tokens in `devices/paired.json`.

## Best Possible Solution

Capture the requested post-reboot gateway and systemd diagnostics, then fix the specific service ownership, duplicate listener, state-dir drift, or auth-cache mismatch that causes TUI to reach a pairing-required state while preserving explicit scope-upgrade approval boundaries.

## Reproduction Assessment

No, we do not yet have a high-confidence current-main reproduction. The cache-missing paired-token path was investigated in https://github.com/openclaw/openclaw/pull/80742 and not reproduced, while the remaining duplicate-listener theory needs the post-reboot `gateway status --deep`, `systemctl`, `pgrep`, and journal output requested in the discussion.

## Solution Assessment

Unclear. Loading tokens directly from `devices/paired.json` is probably not the safest fix because server pairing state and client credentials are separate auth surfaces; the narrower path is to prove whether the failing state is stale listener, state-dir drift, stale local `identity/device-auth.json`, or approved-scope mismatch first.
