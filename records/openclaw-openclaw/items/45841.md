## Summary

Keep open: current main and v2026.5.22 still intentionally block sandboxed requester sessions from spawning ACP sessions, and enabling a bridge, sidecar, or nested ACP placement crosses a sandbox-to-host security/product boundary.

## What This Changes

Request enabling sandboxed OpenClaw sessions to spawn ACP sessions through a safe bridge, sidecar, or nested sandbox design instead of the current documented block.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit/rate-limit controls, schema/docs/tests, and live Docker/acpx proof before relaxing the current fail-closed guard.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a sandboxed requester calling sessions_spawn with runtime="acp" or /acp spawn is blocked by availability and runtime-policy guards, and focused tests assert the denial paths.

## Solution Assessment

Unclear. The issue lists plausible bridge, sidecar, and nested-sandbox directions, but current main has no accepted sandbox-to-host ACP contract; the maintainable next step is a maintainer security/product decision before code changes.
