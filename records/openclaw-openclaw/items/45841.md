## Summary

Keep this open. Current main still deliberately blocks ACP spawns from sandboxed requester sessions, documents ACP as host-side rather than sandbox-wrapped, and has regression tests asserting the block; the related earlier report was closed by its author, not replaced by a shipped implementation.

## What This Changes

Request to let sandboxed OpenClaw sessions spawn or control ACP harness sessions without giving up the Docker sandbox boundary, likely through an opt-in bridge/proxy, nested sandbox, or in-sandbox ACP mode.

## Best Possible Solution

Define and implement an explicit ACP sandbox compatibility mode that preserves the default block, adds a narrow audited spawn/steer/cancel path or nested sandbox runtime, updates config/schema/docs, and proves with tests that sandboxed sessions cannot gain general host execution through ACP.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode="all"` and ACP enabled, `spawnAcpDirect` or `/acp spawn` returns the documented forbidden error before ACP runtime initialization; current tests assert that path. This read-only review inspected the tests rather than executing them.

## Solution Assessment

No proposed patch is present to assess. The current blanket block is the safest default but does not solve the requested workflow; the maintainable fix needs an explicit maintainer-approved sandbox/ACP boundary design rather than a small automatic code change.

  - body: A host-side ACP bridge or proxy could become a sandbox escape if it accepts broad commands or unaudited session control from a container. Any implementation should be opt-in, narrowly scoped to ACP spawn/steer/cancel, authenticated to the agent/session, allowlisted, rate-limited, and logged.
  - confidence: 0.9
