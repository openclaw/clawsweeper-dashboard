## Summary

Keep open: current main and the latest release still intentionally block ACP spawns from sandboxed requester sessions, and the docs/tests describe that as the current security boundary. The prior closed issue preserves useful context but did not land a fix, while the newer execution-backend issue is broader than this targeted sandbox/ACP design request.

## What This Changes

The issue asks for sandboxed OpenClaw sessions to spawn ACP-backed sessions through a secure sandbox-to-host boundary.

## Best Possible Solution

Define an explicit ACP placement or bridge contract with operator opt-in, allowlisting, audit/rate-limit controls, config/schema/docs/tests, and live Docker/acpx proof before relaxing the current fail-closed guard.

## Reproduction Assessment

Yes, source-level: a sandboxed requester calling sessions_spawn with runtime="acp" is blocked by availability and runtime-policy guards before ACP initialization. Existing tests cover the denial path, but this read-only review did not run tests.

## Solution Assessment

Unclear; the issue identifies plausible bridge, sidecar, and nested/container execution approaches, but current main has no accepted sandbox-to-host ACP contract. The narrowest maintainable next step is maintainer security/product design for that boundary before code changes.
