## Summary

Keep open: current main intentionally blocks ACP spawns from sandboxed sessions and documents/tests that as the current security boundary; neither the self-closed earlier report nor the broader execution-backend proposal supersedes this narrower design request.

## What This Changes

The issue requests an opt-in way for sandboxed OpenClaw sessions to start ACP-backed coding harness sessions through a bridge, nested sandbox, or containerized execution model.

## Best Possible Solution

Define a security-reviewed ACP bridge or containerized ACP placement contract first, then update config/schema, spawn policy, docs, and focused sandbox/ACP regression plus live validation around that contract.

## Reproduction Assessment

Yes for the current limitation, source-level: a sandboxed requester calling sessions_spawn with runtime="acp" is blocked by availability and runtime-policy guards before ACP initialization. Existing tests cover the same denial path.

## Solution Assessment

Unclear: no accepted implementation exists yet. The host proxy/bridge idea is plausible, but the best next step is maintainer security/product design for the sandbox-to-host contract, not a narrow code cleanup.
