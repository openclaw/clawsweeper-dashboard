## Summary

Keep open: current main and v2026.6.10 still intentionally block ACP spawns from sandboxed requester sessions, so the requested sandbox-to-ACP bridge or sandboxed ACP placement remains unresolved and needs maintainer product/security design.

## What This Changes

Request an opt-in, auditable way for sandboxed OpenClaw sessions to start or control ACP harness sessions without dropping sandbox isolation.

## Best Possible Solution

Keep the fail-closed guard until maintainers define an explicit opt-in sandbox-to-ACP bridge or execution-placement contract with allowlisting, audit/rate-limit controls, schema/docs/tests, and live Docker/acpx proof.

## Reproduction Assessment

Yes, at source level. A sandboxed requester calling sessions_spawn with runtime="acp" is denied before dispatch, and the direct /acp spawn path returns the same runtime-policy error.

## Solution Assessment

Unclear. Bridge, sidecar, and nested-sandbox designs are plausible, but current main has no maintainer-approved sandbox-to-host ACP contract, so the best next step is product/security design rather than an autonomous patch.
