## Summary

Keep open. Current main still intentionally blocks ACP spawns from sandboxed requesters and documents that ACP runs on the host; solving this needs a new opt-in bridge or containerized ACP design across a security boundary, not cleanup closure.

## What This Changes

Sandboxed OpenClaw sessions should be able to spawn or bridge ACP sessions without giving up container isolation.

## Best Possible Solution

Define a narrow security-reviewed ACP bridge or containerized ACP runtime contract, then update config/schema, spawn policy, docs, and focused regression plus live sandbox/ACP validation around that contract.

## Reproduction Assessment

Yes, source-level. A sandboxed requester calling `sessions_spawn` with `runtime: "acp"` reaches the current availability/runtime-policy guards and is blocked; existing tests cover the same path.

## Solution Assessment

No concrete best implementation is established yet. The host proxy/bridge idea is plausible, but maintainers need to choose the security contract before an implementation PR is safe.
