## Summary

Keep open: current main and the latest shipped release still intentionally block ACP spawns from sandboxed requester sessions, and the remaining request is a security/product design change rather than stale cleanup or a duplicate. The related closed issue documents a user workaround, while the newer execution-backend issue and ACP RFC are broader adjacent work, not a merged replacement for this targeted sandboxed-ACP compatibility request.

## What This Changes

Allow sandboxed OpenClaw sessions to use ACP through a safe bridge, nested placement, or opt-in host-side execution contract.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit and rate-limit controls, config/schema/docs/tests, and live Docker/acpx proof before relaxing the fail-closed guard.

## Reproduction Assessment

Yes, source-level: a sandboxed requester calling sessions_spawn with runtime="acp" is blocked by availability and runtime-policy guards before ACP initialization. Existing tests cover the denial path, but this review did not run live Docker/acpx proof.

## Solution Assessment

Unclear; the issue lists plausible bridge, sidecar, and nested-sandbox directions, but current main has no accepted sandbox-to-host ACP contract. The narrowest maintainable next step is a maintainer security/product decision for that boundary before code changes.
