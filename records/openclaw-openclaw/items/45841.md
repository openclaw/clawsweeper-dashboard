## Summary

Keep open. Current main intentionally documents and enforces the sandboxed-requester ACP block; the related earlier issue was closed only because that reporter changed setup, while the merged security hardening in https://github.com/openclaw/openclaw/pull/32254 explicitly treated host-side ACP spawning from a sandbox as a sandbox-boundary risk and a later comment called it a stopgap.

## What This Changes

The issue requests an opt-in way for sandboxed OpenClaw sessions to spawn or control ACP coding-harness sessions without fully dropping sandbox isolation.

## Best Possible Solution

Define a maintainer-approved, opt-in ACP sandbox bridge or sandbox-owned ACP runtime contract that preserves the fail-closed default and documents the audit/security boundary before implementation.

## Reproduction Assessment

Yes, from source and tests rather than a live Docker run. Current main rejects `runtime="acp"` for sandboxed requesters through both tool availability and spawn runtime policy, with regression tests covering the failure path.

## Solution Assessment

Unclear. The requested direction is plausible, and the issue discussion usefully outlines proxy, sidecar, and nested-sandbox options, but maintainers need to choose the security and API boundary before code changes are safe.
