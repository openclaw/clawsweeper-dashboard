## Summary

Keep open: current main and the latest release still intentionally block ACP spawns from sandboxed requester sessions, and the remaining work is a security/product design change rather than cleanup. The older duplicate-looking issue was closed by its reporter after a local workaround, while the broader execution-backend issue is related but does not replace this specific sandbox-to-ACP bridge request.

## What This Changes

The issue requests an opt-in way for sandboxed OpenClaw sessions to spawn or control ACP sessions without giving up the sandbox boundary.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit/rate-limit controls, config/schema/docs/tests, and live Docker/acpx proof before relaxing the existing fail-closed guard.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a sandboxed requester calling `sessions_spawn` with `runtime: "acp"` is blocked by availability and runtime-policy guards, and focused tests cover the denial path; I did not run live Docker/acpx proof in this read-only review.

## Solution Assessment

Unclear. The issue lists plausible bridge, sidecar, and nested-sandbox directions, but current main has no accepted sandbox-to-host ACP contract; the maintainable next step is a maintainer security/product decision before code changes.
