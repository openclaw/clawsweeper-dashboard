## Summary

Keep open: current main still deliberately blocks ACP spawns from sandboxed requester sessions, documents that as the current security boundary, and has regression coverage for the denial path. The linked closed report was a user workaround and the newer execution-backend issue is broader, so neither is a canonical replacement for this targeted sandbox/ACP design request.

## What This Changes

Allow sandboxed OpenClaw sessions to spawn ACP sessions through a safe bridge, proxy, or containerized ACP execution model instead of the current host-runtime block.

## Best Possible Solution

Define a security-reviewed ACP bridge or containerized execution placement contract first, then update config/schema, spawn policy, docs, regression tests, and live Docker/acpx validation around that contract.

## Reproduction Assessment

Yes, source-level: a sandboxed requester calling sessions_spawn with runtime="acp" is blocked by availability and runtime-policy guards before ACP initialization. Existing tests cover the same denial path, but I did not run tests because this review was read-only.

## Solution Assessment

Unclear because no accepted implementation exists yet. A host proxy/bridge or containerized ACP placement model is plausible, but the next step should be maintainer security/product design for the sandbox-to-host contract.
