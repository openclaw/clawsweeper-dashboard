## Summary

Keep open: current main and the shipped v2026.5.20 release still intentionally block sandboxed requester sessions from spawning ACP sessions, and enabling a bridge, sidecar, or nested ACP placement would be a security/product design change rather than conservative cleanup.

## What This Changes

Request to let sandboxed OpenClaw sessions spawn or bridge ACP harness sessions without giving up sandbox isolation.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit/rate-limit controls, schema/docs/tests, and live Docker/acpx proof before relaxing the current fail-closed guard.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a sandboxed requester calling `sessions_spawn` with `runtime: "acp"` or `/acp spawn` is blocked by availability and runtime-policy guards, and focused tests assert the denial paths.
