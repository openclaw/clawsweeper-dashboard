## Summary

Keep open: current main and v2026.6.6 still deliberately block ACP spawns from sandboxed requester sessions, and the useful discussion here needs a sandbox-to-host or sandboxed-ACP execution contract rather than cleanup closure.

## What This Changes

Request to let sandboxed OpenClaw sessions spawn and control ACP harness sessions through an opt-in bridge, nested sandbox, or other audited placement model.

## Best Possible Solution

Keep the fail-closed guard until maintainers define an explicit opt-in sandbox-to-ACP placement or bridge contract with allowlisting, audit/rate-limit controls, schema/docs/tests, and live Docker/acpx proof.

## Reproduction Assessment

Yes, at source level. A sandboxed requester calling sessions_spawn with runtime="acp" is denied before dispatch, and the direct ACP spawn path returns a forbidden runtime-policy error.

## Solution Assessment

Unclear. Bridge, sidecar, and nested-sandbox designs are plausible, but current main has no maintainer-approved sandbox-to-host ACP contract, so the best next step is product/security design rather than an autonomous patch.
