## Summary

Keep open: current main still intentionally blocks ACP spawns from sandboxed requester sessions, and the requested bridge/nested-sandbox path is a real feature/security design question rather than stale cleanup.

## What This Changes

Sandboxed OpenClaw sessions should be able to spawn or orchestrate ACP sessions through a controlled, auditable boundary.

## Best Possible Solution

Define an opt-in, audited ACP-from-sandbox design in the core ACP/session runtime contract, then update config, docs, and regression coverage around that explicit boundary.

## Reproduction Assessment

Yes for the current limitation: source, docs, and tests on current main all show sandboxed requester sessions cannot use runtime="acp". This is not a broken existing contract; it is a requested expansion of that contract.

## Solution Assessment

Unclear for implementation. The proposed bridge/proxy direction is plausible, but the narrow maintainable solution needs a maintainer-owned security design rather than simply relaxing the existing block.
