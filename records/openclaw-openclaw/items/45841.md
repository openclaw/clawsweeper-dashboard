## Summary

Keep open: current main and the latest release still intentionally block ACP spawns from sandboxed requester sessions, so the requested sandbox-to-ACP bridge or sandboxed ACP placement remains unresolved and needs maintainer product/security design.

## What This Changes

Allow sandboxed OpenClaw sessions to spawn or control ACP harness sessions without giving up the sandbox boundary.

## Best Possible Solution

Keep the fail-closed guard until maintainers define an explicit opt-in sandbox-to-ACP bridge or placement contract with allowlisting, audit/rate-limit controls, schema/docs/tests, and live Docker/acpx proof.

## Reproduction Assessment

Yes, at source level. A sandboxed requester calling sessions_spawn with runtime="acp" is denied before dispatch, and the direct ACP spawn path returns a forbidden runtime-policy error.

## Solution Assessment

Unclear. Bridge, sidecar, and nested-sandbox designs are plausible, but current main has no maintainer-approved sandbox-to-host ACP contract, so the best next step is product/security design rather than an autonomous patch.
