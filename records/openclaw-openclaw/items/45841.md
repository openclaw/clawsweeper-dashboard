## Summary

Keep open: current main and v2026.5.19 still intentionally block ACP spawns from sandboxed requester sessions, and the request remains a security/product design change rather than stale cleanup or a duplicate. The open execution-backend PR and ACP RFC are useful adjacent work, but they do not yet implement this targeted sandboxed-ACP bridge.

## What This Changes

Allow sandboxed OpenClaw sessions to spawn or control ACP sessions through a safe sandbox-to-host or sandboxed execution boundary.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit and rate-limit controls, config/schema/docs/tests, and live Docker/acpx proof before relaxing the fail-closed guard.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a sandboxed requester calling `sessions_spawn` with `runtime: "acp"` is blocked by availability and runtime-policy guards, and existing tests cover the denial path; I did not run live Docker/acpx proof in this read-only review.

## Solution Assessment

Unclear. The issue lists plausible bridge, sidecar, and nested-sandbox directions, but current main has no accepted sandbox-to-host ACP contract; the maintainable next step is a maintainer security/product decision before code changes.
