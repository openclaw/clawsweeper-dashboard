## Summary

Keep open: current main and the latest release still deliberately block ACP spawns from sandboxed requester sessions, and enabling a bridge, sidecar, or alternate placement would be a security/product design change rather than cleanup. The earlier host-proxy issue was closed by its reporter after a local workaround, while the broader execution-backend and tool-level sandbox issues are adjacent but do not replace this narrow sandbox-to-ACP request.

## What This Changes

Request an opt-in, auditable way for sandboxed OpenClaw sessions to spawn or control ACP sessions without dropping sandbox isolation.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit and rate-limit controls, config/schema/docs/tests, and live Docker/acpx proof before relaxing the existing fail-closed guard.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a sandboxed requester calling `sessions_spawn` with `runtime: "acp"` or `/acp spawn` is blocked by availability and runtime-policy guards, and focused tests cover the denial paths; I did not run live Docker/acpx proof in this read-only review.

## Solution Assessment

Unclear. The issue lists plausible bridge, sidecar, and nested-sandbox directions, but current main has no accepted sandbox-to-host ACP contract; the maintainable next step is a maintainer security/product decision before code changes.
