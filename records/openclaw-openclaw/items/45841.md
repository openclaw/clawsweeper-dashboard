## Summary

Keep open: current main and v2026.5.12 still intentionally block ACP spawns from sandboxed requester sessions, and the remaining request is a security/product feature rather than stale cleanup. https://github.com/openclaw/openclaw/issues/82017 is broader execution-placement work, not a merged replacement for this targeted sandboxed-ACP compatibility request.

## What This Changes

The issue requests an opt-in sandbox-to-ACP bridge or placement model so sandboxed OpenClaw sessions can spawn ACP harness sessions without giving up sandbox isolation.

## Best Possible Solution

Define an explicit sandbox-to-ACP placement or bridge contract with operator opt-in, allowlisting, audit/rate-limit controls, config/schema/docs/tests, and live Docker/acpx proof before relaxing the fail-closed guard.

## Reproduction Assessment

Yes, source-level: a sandboxed requester calling sessions_spawn with runtime="acp" is blocked by availability and runtime-policy guards before ACP initialization. Existing tests cover the denial path, but this read-only review did not run live Docker/acpx proof.

## Solution Assessment

Unclear; the issue lists plausible bridge, sidecar, and nested-sandbox directions, but current main has no accepted sandbox-to-host ACP contract. The narrowest maintainable next step is maintainer security/product design for that boundary before code changes.
