## Summary

Keep open: current main still intentionally blocks ACP spawns from sandboxed sessions and documents/tests that as the current security boundary. The earlier self-closed report is a user workaround, and the newer execution-backend issue is broader and explicitly distinct rather than a canonical replacement.

## What This Changes

Request sandboxed OpenClaw sessions to spawn ACP sessions through an opt-in bridge, nested/containerized ACP placement, or another narrow audited execution boundary.

## Best Possible Solution

Define a security-reviewed ACP bridge or containerized ACP placement contract first, then update config/schema, spawn policy, docs, and focused sandbox/ACP regression plus live validation around that contract.

## Reproduction Assessment

Yes for the current limitation, source-level: a sandboxed requester calling sessions_spawn with runtime="acp" is blocked by availability and runtime-policy guards before ACP initialization. Existing regression tests cover the same denial path.

## Solution Assessment

Unclear because no accepted implementation exists yet. A host proxy/bridge or containerized placement model is plausible, but the next step should be maintainer security/product design for the sandbox-to-host contract.
