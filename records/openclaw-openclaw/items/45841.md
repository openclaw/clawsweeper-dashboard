## Summary

Keep open: current main and the latest stable release still intentionally block sandboxed requester sessions from spawning ACP sessions, and enabling this needs a sandbox-to-host execution contract rather than cleanup closure.

## What This Changes

Issue requests an opt-in way for sandboxed OpenClaw sessions to spawn or control ACP-backed coding-agent sessions without dropping the sandbox boundary.

## Best Possible Solution

Define an explicit opt-in sandbox-to-ACP placement or bridge contract with allowlisting, audit/rate-limit controls, schema/docs/tests, and live Docker/acpx proof before relaxing the current fail-closed guard.

## Reproduction Assessment

Yes, at source level. A sandboxed requester calling sessions_spawn with runtime="acp" or /acp spawn is blocked by availability and runtime-policy guards, and focused tests assert those denial paths.

## Solution Assessment

Unclear. The bridge, sidecar, and nested-sandbox directions are plausible, but current main has no accepted sandbox-to-host ACP contract, so the best next step is maintainer security/product design before code changes.
