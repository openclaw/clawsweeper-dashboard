## Summary

Keep open: current main and the latest release still intentionally block ACP spawns from sandboxed requester sessions, so the requested sandbox-to-ACP bridge or sandboxed ACP placement remains unresolved. The discussion has useful bridge, sidecar, and nested-sandbox design context, but this still needs maintainer product and security design before implementation.

## What This Changes

The issue requests a secure way for sandboxed OpenClaw sessions to use ACP orchestration without dropping the sandbox boundary.

## Best Possible Solution

Keep the fail-closed guard until maintainers define an explicit opt-in sandbox-to-ACP bridge or execution-placement contract with allowlisting, audit controls, docs, tests, and live Docker/acpx proof.

## Reproduction Assessment

Yes, at source level: a sandboxed requester calling `sessions_spawn` with `runtime="acp"` is denied before ACP dispatch, and `/acp spawn` calls the same fail-closed policy path. I did not run a live Docker/acpx repro in this read-only review.

## Solution Assessment

Unclear as an implementation: bridge, sidecar, nested-sandbox, and broader execution-placement designs are plausible, but current main has no maintainer-approved sandbox-to-host ACP contract. The best next step is product/security design rather than an autonomous patch.
