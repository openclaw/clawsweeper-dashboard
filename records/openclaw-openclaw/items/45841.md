## Summary

Keep this open. Current main still intentionally blocks ACP spawns from sandboxed requester sessions, documents that limitation, and has regression coverage for the fail-closed path; solving the requested workflow would require a new, security-sensitive sandbox/ACP compatibility design.

## What This Changes

Request to let sandboxed OpenClaw sessions spawn ACP sessions through an opt-in bridge/proxy, nested sandbox, or in-sandbox ACP mode instead of the current blanket block.

## Best Possible Solution

Define an explicit ACP sandbox compatibility mode that keeps the default block, adds a narrow authenticated and audited spawn/steer/cancel path or nested sandbox runtime, and proves the boundary with config/schema/docs/test coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with a sandboxed requester, `sessions_spawn({ runtime: "acp" })` and `/acp spawn` hit the documented forbidden path before ACP runtime initialization, and current tests assert that behavior.

## Solution Assessment

No proposed patch is present to assess. The current fail-closed behavior is the safest default, but it does not satisfy the requested workflow; the maintainable fix needs a maintainer-approved sandbox-to-host ACP boundary design.

## Security Review

Status: needs_attention

Summary: Needs attention because the requested feature would let a sandboxed requester trigger host-side ACP harness execution.

Concerns:

- **[high] Define the host ACP bridge as a controlled privilege boundary:** `src/agents/acp-spawn.ts:189`
  - body: A compatibility bridge could become a sandbox escape if it accepts broad commands or unaudited session control from a container. Any design should be opt-in, authenticated to the agent/session, allowlisted, rate-limited, logged, and limited to ACP spawn/steer/cancel semantics.
  - confidence: 0.9
- **[medium] Keep the default ACP sandbox behavior fail-closed:** `docs/tools/acp-agents.md:721`
  - body: Current code and docs fail closed for sandboxed requesters. A compatibility mode should preserve that default and require explicit operator configuration before any host-side ACP execution is reachable from a sandbox.
  - confidence: 0.86
