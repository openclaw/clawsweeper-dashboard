## Summary

Keep open. Current main and the latest release still expose a single exact memory-flush write target, so a custom flush prompt that asks for `SESSION_HANDOFF.md` will still be rejected; the default bundled prompt now targets only the daily memory file, but the configurable additional-target bug remains valid. PR #66765 is closed unmerged, #74198 was closed as a duplicate of this issue, and #63664/#68995 are adjacent broader policy requests rather than replacements.

## What This Changes

Allow pre-compaction memory-flush runs to write `SESSION_HANDOFF.md` or other explicitly configured additional targets instead of only appending to `memory/YYYY-MM-DD.md`.

## Best Possible Solution

Keep this issue open as the canonical `SESSION_HANDOFF.md` and additional-target tracker. The maintainable end state is a typed memory-flush write-target contract in config or `MemoryFlushPlan`, with the daily memory file kept append-only, explicitly validated overwrite targets for files such as `SESSION_HANDOFF.md`, traversal and sibling-prefix rejection, docs/schema updates, and regression tests across the plan, runner, tool wrapper, and config surfaces.

## Reproduction Assessment

Yes. Configure `agents.defaults.compaction.memoryFlush.prompt` to request `SESSION_HANDOFF.md`, trigger a memory run, and the current wrapper will reject that path because only `memory/YYYY-MM-DD.md` is forwarded as `memoryFlushWritePath`.

## Solution Assessment

No for the prompt-derived permission approach described in the issue. The safer fix is an explicit allowlist or target contract, not granting write permission to arbitrary paths mentioned in free-form prompt text.
