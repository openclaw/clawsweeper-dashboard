## Summary

Keep open. Current main still enforces one exact append-only memory-flush write target, while the requested behavior would expand sandboxed write authority and needs product/security policy review; the closed configurable-path duplicate points back here, and the broader write-lock discussion remains separate at https://github.com/openclaw/openclaw/issues/63664.

## What This Changes

The issue requests allowing pre-compaction memory flush to write `SESSION_HANDOFF.md` or other explicitly configured prompt paths in addition to `memory/YYYY-MM-DD.md`.

## Best Possible Solution

Add an explicit memory-flush write-target contract with normalized workspace-root validation, traversal and sibling-prefix rejection, per-target append or overwrite modes, docs/schema alignment, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `memoryFlush.prompt` to request `SESSION_HANDOFF.md`, trigger memory flush, and the wrapper rejects anything other than `activeMemoryFlushPlan.relativePath`; I did not run a live compaction turn.

## Solution Assessment

No, not as prompt-derived permission. The maintainable fix is an explicit configured or plugin-provided allowlist with validated paths and per-target write modes.
