## Summary

Keep open. Current main still enforces one exact append-only memory-flush write target, while this request would expand a sandboxed write surface and needs maintainer policy and security review.

## What This Changes

The issue requests allowing `SESSION_HANDOFF.md` or other explicitly configured memory-flush prompt targets alongside the daily memory note.

## Best Possible Solution

Add an explicit memory-flush write-target contract with normalized workspace-root validation, traversal and sibling-prefix rejection, per-target append or overwrite modes, docs/schema alignment, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure a flush prompt to request `SESSION_HANDOFF.md`, trigger memory flush, and the wrapper rejects anything other than `activeMemoryFlushPlan.relativePath`.

## Solution Assessment

No, not as prompt-derived permission. The maintainable fix is an explicit configured or plugin-provided allowlist with validated paths and per-target write modes.
