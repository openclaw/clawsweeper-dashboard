## Summary

Keep open. Current main still permits exactly one append-only memory-flush write target and rejects `SESSION_HANDOFF.md`; the related implementation PR is closed unmerged, and the safe repair needs an explicit allowlist contract rather than prompt-derived write authority.

## What This Changes

Request to let memory-flush turns write `SESSION_HANDOFF.md` or other explicitly allowed prompt-configured paths in addition to the daily memory note.

## Best Possible Solution

Add a typed memory-flush write-target allowlist with workspace-root normalization, traversal and sibling-prefix rejection, explicit append/overwrite modes, schema/docs updates, and regression coverage; do not treat free-form prompt text as permission.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure the memory-flush prompt to request `SESSION_HANDOFF.md`, trigger a memory flush, and current main still forwards only `relativePath` while the write wrapper rejects any other resolved path.

## Solution Assessment

No for the prompt-derived permission approach in the original request and closed PR. The safer maintainable solution is an explicit typed allowlist/target contract with validated paths and per-target write modes.
