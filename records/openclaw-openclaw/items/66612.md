## Summary

Keep open. Current main still gives memory-flush runs exactly one append-only write target and rejects `SESSION_HANDOFF.md`; the closed fix PR did not merge, and the remaining change needs a deliberate write-allowlist design because it relaxes a flush-time sandbox.

## What This Changes

Request to let pre-compaction memory flush write `SESSION_HANDOFF.md` or other explicitly configured handoff paths in addition to the daily memory note.

## Best Possible Solution

Add an explicit typed memory-flush write-target contract with workspace-root normalization, traversal and sibling-prefix rejection, and per-target write modes, while keeping the daily note append-only and updating config schema, docs, and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure the flush prompt to request `SESSION_HANDOFF.md`, trigger a memory flush, and the current wrapper will reject that path because only `activeMemoryFlushPlan.relativePath` is allowed.

## Solution Assessment

Partly. Allowing a handoff file through an explicit validated allowlist is the right direction, but granting write permission from free-form prompt text is not the safest solution.
