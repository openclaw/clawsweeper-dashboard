## Summary

Keep open. Current main still forwards exactly one memory-flush write target into the write wrapper, and that wrapper rejects any other path such as `SESSION_HANDOFF.md`; config/docs expose prompt text but no explicit additional write-target contract. The linked fix PR #66765 is closed unmerged, and a PR body search only found that closed implementation attempt, so there is no active merged or open implementation candidate.

## What This Changes

Allow pre-compaction memory flush turns to write both the daily memory note and `SESSION_HANDOFF.md` or another explicitly configured additional target instead of being restricted to one daily path.

## Best Possible Solution

Keep this issue open as the canonical additional memory-flush write-target tracker. The maintainable end state is a typed allowlist in config or `MemoryFlushPlan`, daily notes kept append-only, validated overwrite targets for files such as `SESSION_HANDOFF.md`, traversal and sibling-prefix rejection, docs/schema updates, and regression coverage through the plan, runner, embedded tool context, and write wrapper.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `memoryFlush.prompt` to request `SESSION_HANDOFF.md`, trigger a memory run, and current main will still pass only `activeMemoryFlushPlan.relativePath` to the write wrapper, which rejects any path that does not resolve to that exact target.

## Solution Assessment

No for the previously proposed prompt-derived path detection. The safer solution is an explicit typed allowlist/target contract with normalized workspace-root validation and per-target append/overwrite behavior, rather than treating free-form prompt text as write permission.
