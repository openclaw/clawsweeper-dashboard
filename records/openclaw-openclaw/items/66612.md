## Summary

Keep open. Current main intentionally constrains memory-flush writes to one append-only target, while this asks for a broader write-target contract that affects a sandboxed write surface and still needs maintainer policy judgment.

## What This Changes

The issue asks memory flushes to write both the daily memory note and `SESSION_HANDOFF.md`, or to honor explicitly configured flush-prompt write targets.

## Best Possible Solution

Add an explicit memory-flush write-target contract with normalized workspace-root validation, traversal and sibling-prefix rejection, clear append versus overwrite modes, docs/schema alignment, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure the flush prompt to request `SESSION_HANDOFF.md`, trigger memory flush, and the write wrapper rejects anything other than the single `activeMemoryFlushPlan.relativePath`.

## Solution Assessment

No, not as prompt-derived permission. The maintainable fix is an explicit configured or plugin-provided allowlist with validated paths and per-target write modes.
