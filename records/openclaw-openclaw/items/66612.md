## Summary

Keep open. Current main still enforces a single append-only memory-flush write target, while allowing `SESSION_HANDOFF.md` or prompt-derived extra paths would add new write authority and needs an explicit product/security decision.

## What This Changes

Request to let memory-flush turns write `SESSION_HANDOFF.md` or other approved prompt/config paths in addition to `memory/YYYY-MM-DD.md`.

## Best Possible Solution

Keep this open until maintainers choose the write-target policy; if accepted, implement a typed validated memory-flush target contract with append/overwrite modes across the memory plugin plan, runner, tool wrapper, schema, docs, and tests.

## Reproduction Assessment

Yes, at source level. A memory-triggered run still forwards one `memoryFlushWritePath`, and the wrapped write tool rejects every resolved path other than that target; I did not run a live compaction turn.

## Solution Assessment

No, not as prompt-derived permission. The maintainable fix is an explicit configured or plugin-provided allowlist with normalized path validation and per-target write modes.
