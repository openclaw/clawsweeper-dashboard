## Summary

Keep this PR open. Current main still only protects the foreground active session during session maintenance, while this PR targets a real subagent data-loss path; however, the proposed helper still misses persisted active subagent runs and the branch needs a small repair before it is safe to merge.

## What This Changes

The PR threads protected session keys through pruning, entry-count capping, and disk-budget enforcement so sessions backing active subagent runs are skipped, with targeted pruning tests and skip logging.

## Best Possible Solution

Land a rebased fix that reuses the current preserveKeys-style maintenance seam, derives active subagent child session keys from the persisted-plus-memory run snapshot, protects prune/cap/disk-budget entry eviction, and adds regression coverage plus a changelog entry before closing the related open bug.

## Reproduction Assessment

Yes. The product bug has a clear path with session maintenance in enforce mode, a tight maxEntries or disk budget, and an active subagent child session older than retained entries; current main only preserves opts.activeSessionKey, not active child sessions. The PR-specific blocker is reproducible by having an active run visible only through the persisted registry snapshot while getActiveSubagentSessionKeys reads only the in-memory map.

## Solution Assessment

No, not as written. The direction is maintainable, but the helper must read through getSubagentRunsSnapshotForRead(subagentRuns), and the patch should be rebased onto current main’s preserveKeys API with required changelog coverage.
