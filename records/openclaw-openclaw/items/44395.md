## Summary

Current main and v2026.6.5 still use size-based chunks that can blend short Markdown sections. The issue remains necessary because the overlapping implementation PR is conflicting, proof-gated, and much broader than the focused heading-boundary problem.

## What This Changes

Add Markdown heading-aware memory chunking and explore entity extraction, graph retrieval, reranking, and consolidation improvements.

## Best Possible Solution

Adopt one memory-core-owned heading-aware chunking rule with deterministic line ranges, explicit reindex behavior, and measured before/after retrieval proof; track entity extraction, graph retrieval, reranking, and consolidation as separate canonical architecture items.

## Reproduction Assessment

Yes at source level: current main deterministically combines adjacent short sections until its character budget is reached, and memory-core embeds the combined result. This read-only review did not run a controlled live retrieval benchmark.

## Solution Assessment

No, not as one bundled implementation: heading-aware chunking is a focused solution to section contamination, while entity extraction, graph retrieval, reranking, and consolidation require separate product and architecture decisions; the current six-strategy PR is also broader than necessary.
