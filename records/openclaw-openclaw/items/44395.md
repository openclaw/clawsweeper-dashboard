## Summary

Keep open: current main and v2026.6.6 still use fixed token/overlap memory chunking with no heading-aware or section-aware strategy, and the entity/wiki synthesis part remains broader than chunk boundary selection. The open multi-strategy chunking PR overlaps this request but is conflicting and still needs maintainer API, compatibility, proof, and security review, so it is not a safe cleanup-close target.

## What This Changes

The issue requests heading-aware memory chunking, with stretch work for entity extraction and graph-style memory search, to reduce blended embeddings across unrelated markdown sections.

## Best Possible Solution

Choose a memory-core chunking API, land a focused heading-aware or section-aware path with config/schema/reindex coverage, and track entity extraction or wiki synthesis separately through the memory-wiki/MVP design threads.

## Reproduction Assessment

Yes. Source inspection shows headed sections under the default 400-token budget are accumulated into the same chunk until the size limit because the active chunker has no heading-boundary logic.

## Solution Assessment

Unclear. Heading-aware chunking is plausible, but maintainers need to decide the default versus opt-in strategy shape, and the entity extraction stretch should be split into separate memory-wiki or REM pipeline work.
