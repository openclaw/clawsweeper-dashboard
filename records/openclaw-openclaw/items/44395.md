## Summary

Keep open: current main and v2026.6.6 still use fixed token/overlap memory chunking with no heading-aware strategy, and the entity/wiki synthesis part is only partially adjacent to open memory-wiki work. The open multi-strategy chunking PR overlaps the heading-aware request, but it is conflicting and still needs maintainer product/API review, so it is not a safe cleanup-close target.

## What This Changes

The issue requests heading-aware markdown memory chunking and a larger entity-extraction or graph-memory path so memory search does not blend distinct markdown sections into one embedding chunk.

## Best Possible Solution

Choose a memory-core chunking strategy/API, land a focused heading-aware or section-aware path with config/schema/reindex coverage, and track entity extraction or wiki synthesis separately through the memory-wiki/MVP design threads.

## Reproduction Assessment

Yes. Source inspection shows headed sections under the default 400-token budget are accumulated into the same chunk until the size limit because the active chunker has no heading-boundary logic.

## Solution Assessment

Unclear. Heading-aware chunking is plausible, but the best fix needs a maintainer decision on default behavior versus an opt-in strategy, and entity extraction should be split into a separate memory-wiki or REM pipeline decision.
