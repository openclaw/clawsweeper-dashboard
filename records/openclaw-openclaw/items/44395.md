## Summary

Keep open. Current main still indexes memory through the fixed token/overlap chunker with no heading-aware strategy, no strategy config, and no strategy-aware reindex metadata; the chunking work is already represented by open PR #68019, while entity extraction, graph memory, reranking, and consolidation need separate maintainer scoping.

## What This Changes

Requests heading-aware or section-aware Markdown memory chunking, with broader follow-up ideas for entity extraction, graph memory, reranking, and nightly consolidation.

## Best Possible Solution

Ship a maintained, documented heading-aware or section-aware strategy in the shared memory-host indexing path with strategy-aware reindex metadata, then track entity extraction, graph memory, reranking, and consolidation as separate memory-wiki/QMD follow-ups.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: several short `##` sections whose combined estimated size stays under `tokens * 4` will remain in one chunk because current main flushes on size, not heading boundaries. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Unclear for the full scope. Heading-aware chunking is a narrow maintainable direction, but #68019's broader six-strategy design and the entity/graph/reranking stretch work need maintainer architecture review before choosing the final shape.
