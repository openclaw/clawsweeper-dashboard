## Summary

Keep open: current main and the latest release still use fixed-size token/overlap memory chunking, the open multi-strategy chunking PR remains the current implementation candidate, and the entity/wiki synthesis portion is a broader product/privacy/API decision rather than shipped behavior.

## What This Changes

Request heading/section-aware memory chunking plus later entity extraction or graph-style synthesis so memory search retrieves distinct markdown sections instead of blended fixed-size chunks.

## Best Possible Solution

Land a maintainer-approved section-aware chunking strategy through the memory-host SDK and memory-core indexing/reindex path, then track entity extraction, wiki synthesis, reranking, and consolidation as opt-in follow-ups with explicit privacy, cost, provenance, and reindex controls.

## Reproduction Assessment

Yes for the chunking portion by source inspection: adjacent short heading sections under the token budget stay in one chunk because current code flushes only on size. The entity-extraction and wiki-synthesis portions are new capabilities rather than broken existing contracts.

## Solution Assessment

Unclear for the full issue. Heading-aware chunking is a plausible narrow direction and https://github.com/openclaw/openclaw/pull/68019 is the current candidate, but entity extraction and automated wiki synthesis need product, privacy, cost, and API decisions first.
