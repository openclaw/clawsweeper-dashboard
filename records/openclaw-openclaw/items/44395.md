## Summary

Keep open: current main and the latest release still use fixed token/overlap memory chunking, with no heading-aware strategy and no automated entity-extraction path. The open multi-strategy chunking PR overlaps the heading-aware part but is broad, conflicting, and still needs maintainer review, so it is not a safe cleanup-close target.

## What This Changes

Request heading-aware or section-aware memory chunking plus longer-term entity extraction and graph memory improvements for memory search.

## Best Possible Solution

Choose a memory-core chunking strategy/API, land a focused heading-aware or section-aware path with config/schema/reindex coverage, and track entity extraction or wiki synthesis as a separate memory-wiki or REM pipeline decision.

## Reproduction Assessment

Yes. Source inspection shows headed sections under the default 400-token budget are accumulated into the same chunk until the size limit because the active chunker has no heading-boundary logic.

## Solution Assessment

Unclear. Heading-aware chunking is plausible, but the best fix needs a maintainer decision on default behavior versus an opt-in strategy and should split entity extraction into a separate memory-wiki/REM design.
