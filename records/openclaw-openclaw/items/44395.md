## Summary

Keep open. Current main still chunks memory with the fixed token/overlap path and has no heading-aware strategy or strategy-aware reindex metadata; the comments add useful real-world voice-memory and memory-wiki bridge cases, but the chunking work is still represented by open PR #68019 and the entity/graph/consolidation scope needs separate maintainer design.

## What This Changes

Request heading-aware Markdown memory chunking, plus later entity extraction, graph memory, reranking, and consolidation improvements for memory search quality.

## Best Possible Solution

Ship a maintained heading/section-aware chunking strategy in memory-host and memory-core with reindex metadata and docs, then track entity extraction, graph memory, and consolidation as separate scoped memory-wiki or QMD follow-ups.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: several short `##` sections whose combined estimated size stays under `tokens * 4` will remain in one chunk because current main flushes by size, not heading boundaries.

## Solution Assessment

Unclear for the full scope. Heading-aware chunking is a narrow maintainable direction, but choosing a small heading-only change versus #68019's broader strategy framework, and scoping entity extraction, needs maintainer architecture review.
