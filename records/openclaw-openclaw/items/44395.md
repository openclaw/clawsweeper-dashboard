## Summary

Keep open: current main and v2026.6.10 still use fixed token/overlap memory chunks for memory_search, with no heading-aware strategy or merged entity-extraction path. The related implementation PRs are closed unmerged, so this remains useful as the canonical product/design tracker.

## What This Changes

Request heading-aware Markdown chunking for memory search, plus broader entity extraction and graph-style memory improvements for better recall precision.

## Best Possible Solution

Keep this as the canonical tracker, split heading-aware chunking from entity/graph-memory work, and have maintainers choose the memory-core API/default plus index identity contract before implementation.

## Reproduction Assessment

Yes by source inspection. Indexing a Markdown memory file with several short headed sections under the current token/overlap chunker can place unrelated sections in one embedding chunk until the size budget flushes; I did not run a live index in this read-only review.

## Solution Assessment

Unclear. Heading-aware chunking is a plausible narrow slice, but the best solution still needs maintainer decisions on default versus opt-in behavior, index invalidation, and separation from the larger entity-extraction design.
