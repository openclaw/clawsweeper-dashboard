## Summary

Keep open. Current main still routes memory indexing through fixed token/overlap chunking, with no heading-aware strategy or automatic wiki/entity extraction on the shipped path; related multi-strategy chunking work remains an open implementation candidate that needs maintainer API/security review.

## What This Changes

Request heading-aware memory chunking plus broader entity extraction, graph memory, reranking, and consolidation improvements for memory search.

## Best Possible Solution

Land a maintainer-approved section-aware chunking strategy through the shared memory-host indexing path with strategy-aware reindex metadata, then track entity extraction, graph/wiki memory, cross-encoder reranking, and consolidation as narrower follow-ups.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create several short adjacent markdown heading sections whose combined estimated size stays under the fixed chunk budget, and chunkMarkdown will keep them together because it only flushes on size.

## Solution Assessment

Unclear for the full issue. Heading-aware or section-aware chunking is a plausible narrow solution, but the final strategy/config/reindex shape and the broader entity/wiki scope need maintainer architecture and security review.
