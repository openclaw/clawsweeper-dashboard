## Summary

Keep open. Current main still routes memory-search indexing through the fixed token/overlap chunker with no heading-aware strategy, no strategy-aware reindex metadata, and no docs/config surface; the related implementation candidate remains open, while entity extraction and graph/reranking work need separate maintainer scoping.

## What This Changes

The issue requests heading-aware memory-search chunking for Markdown sections, plus broader entity extraction, graph memory, reranking, and nightly consolidation ideas.

## Best Possible Solution

Ship a maintained, documented heading-aware or section-aware strategy in the shared memory-host indexing path with strategy-aware reindex metadata, then track entity extraction, graph memory, reranking, and consolidation as separate memory-wiki/QMD follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: several short `##` sections whose combined estimated size stays under the token-derived character budget remain in one chunk because current main flushes on size, not heading boundaries.

## Solution Assessment

Unclear for the full scope. Heading-aware chunking is a narrow maintainable fix for the blended-vector failure, but the entity/graph/reranking scope and the broad multi-strategy implementation need maintainer product and architecture review before choosing the final shape.
