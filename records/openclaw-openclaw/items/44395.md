## Summary

Keep open. Current main and the latest release still use fixed token/overlap memory chunking with no shipped heading-aware strategy or automatic entity-extraction path, while the chunking portion is already covered by an open implementation PR that needs maintainer review.

## What This Changes

Request heading-aware or section-aware memory chunking to avoid blended markdown sections, with entity extraction and graph-style memory as stretch goals.

## Best Possible Solution

Land a maintainer-approved section-aware chunking strategy through the shared memory-host indexing path with strategy-aware reindex metadata, then track entity extraction, graph/wiki memory, reranking, and consolidation as narrower follow-ups.

## Reproduction Assessment

Yes for the chunking failure mode by source inspection: several short adjacent markdown heading sections whose combined estimated size stays under the token budget will be kept in one chunk because the current chunker flushes only on size. The entity-extraction portion is not a broken existing contract; it is a requested new capability.

## Solution Assessment

Unclear for the full issue. Heading-aware chunking is a plausible narrow fix, but the strategy/config/reindex shape and the broader entity/wiki scope need maintainer architecture and product review, with https://github.com/openclaw/openclaw/pull/68019 as the current chunking implementation candidate.
