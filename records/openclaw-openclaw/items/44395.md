## Summary

Keep open: current main and the latest release still use fixed token/overlap memory chunking with no heading-aware strategy, and Memory Wiki still requires explicit source/apply workflows rather than automatic entity or synthesis extraction. The chunking portion has an open implementation candidate, but the full request still needs maintainer product/API review.

## What This Changes

Request heading-aware memory chunking plus broader entity extraction, graph/wiki memory, reranking, and consolidation improvements for memory search precision.

## Best Possible Solution

Land a maintainer-approved section-aware chunking strategy through the shared memory-host indexing path, then track entity extraction, graph/wiki memory, reranking, and consolidation as opt-in follow-ups with explicit privacy, cost, and provenance controls.

## Reproduction Assessment

Yes for the chunking failure mode by source inspection: several short adjacent heading sections whose combined size stays under the token budget will remain in one chunk because current code flushes only on size. The entity-extraction portion is a requested new capability rather than a broken existing contract.

## Solution Assessment

Unclear for the full issue. Heading-aware chunking is a plausible narrow fix, but the strategy/config/reindex shape and the broader entity/wiki automation scope need maintainer architecture and product review, with https://github.com/openclaw/openclaw/pull/68019 as the current chunking candidate.
