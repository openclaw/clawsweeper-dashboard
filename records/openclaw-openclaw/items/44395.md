## Summary

Keep open: current main and the latest release still use fixed token/overlap memory chunking with no heading-aware strategy or automated entity extraction path. The related multi-strategy chunking PR is open and broad, so this remains useful maintainer/product work rather than cleanup-close material.

## What This Changes

Request heading-aware memory chunking plus broader entity extraction/graph-memory improvements to reduce blended memory-search retrieval chunks.

## Best Possible Solution

Choose a plugin-owned memory chunking API, land a narrow heading-aware or section strategy with index metadata/reindex coverage, and keep entity extraction/synthesis as a separately reviewed memory-wiki or REM pipeline effort.

## Reproduction Assessment

Yes. Source inspection shows small headed sections under the default 400-token budget are accumulated together until the size limit, because chunkMarkdown has no heading-boundary branch.

## Solution Assessment

Unclear. Heading-aware chunking is a plausible fix, but the best solution needs a maintainer decision on default behavior versus an opt-in strategy and on keeping entity extraction as separate memory-wiki/REM work.
