## Summary

Keep this open. Current main still routes memory-search indexing through the fixed token/overlap chunker, with no heading-aware strategy in config, reindex metadata, or docs; the newer daily-dreaming snippet code preserves heading context for promotion candidates, but it does not change the `memory_search` embedding chunks described by this issue. The related comments and #68019 show active demand and an open unmerged implementation candidate, while entity extraction, graph memory, and cross-encoder reranking remain broader product work.

## What This Changes

The issue requests heading-aware or section-aware memory chunking for Markdown memory files, plus broader follow-up ideas for entity extraction, graph memory, reranking, and nightly consolidation.

## Best Possible Solution

Land a maintained heading-aware or section-aware strategy in the shared memory-host indexing path used by memory-core, expose and document the config, persist strategy-aware metadata so strategy changes trigger reindexing, and split entity extraction, graph memory, cross-encoder reranking, and nightly consolidation into separately scoped memory-wiki/QMD follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: several short `##` sections whose combined estimated size stays under the token-derived character budget remain in one chunk because current main flushes on size, not heading boundaries.

## Solution Assessment

Unclear for the full scope. Heading-aware chunking is a narrow maintainable fix for the blended-vector failure, but the entity/graph/reranking scope and the broad multi-strategy PR need maintainer product and architecture review before choosing the final implementation shape.
