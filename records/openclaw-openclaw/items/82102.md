## Summary

Keep open: current main still routes Slack modal lifecycle submissions only into compact agent-visible system events, while plugin interactive dispatch is wired for block actions. This is a valid Slack/plugin SDK capability gap, but it needs a new backward-compatible modal lifecycle seam and product/security design rather than conservative cleanup close.

## What This Changes

The issue requests that Slack plugin-owned modal submissions expose full view_submission state to plugins before the agent-visible event is compacted.

## Best Possible Solution

Add a backward-compatible Slack-owned modal lifecycle handler contract that routes matching callback_id or namespace submissions to plugins with full view state, while keeping agent system events compact and reference-based.

## Reproduction Assessment

Yes, source-reproducible: submit an oversized Slack modal and the current lifecycle path enqueues only a compact Slack interaction event, while plugin dispatch exists only for block actions. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

Unclear pending maintainer design: the preferred full-state plugin dispatch plus compact reference direction fits the boundary, but the exact SDK contract, ack timing, summary/reference return, and PII handling need product/API review. Raising the compact limit alone is not the best long-term solution.
