## Summary

Keep open. Current main still has no `tools.exec.denyPathPatterns` surface, and this PR is the active implementation candidate for the open #74379 incident report, but the latest PR head still leaves documented hard-deny cases bypassable in primary exec paths.

## What This Changes

The PR adds opt-in `tools.exec.denyPathPatterns` config/schema/docs/changelog support, a new infra glob/token matcher, and enforcement before exec allowlist/approval checks in both `createExecTool` and node-host `system.run` policy evaluation.

## Best Possible Solution

Land a maintainer-reviewed version that evaluates one shared deny-path policy against parsed argv plus existing shell-wrapper inner payload extraction before every gateway, sandbox, elevated, and node execution path, and that resolves cwd-relative sensitive filenames before filtering candidates. Keep the config/schema/docs/changelog aligned with focused tests for direct argv, wrapped shell payloads, global-plus-agent union, allowlist/full bypass resistance, and bare `.env`/`credentials.json` cases.

## Reproduction Assessment

Yes. Configure `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` and run `bash -c "cat ~/.openclaw/secrets/foo.env"` through the gateway/sandbox/elevated path; the PR scans raw wrapper text at `src/agents/bash-tools.exec.ts:1622`, so the tilde pattern is missed. A second static reproduction is `tools.exec.denyPathPatterns: ["**/.env"]` with `cat .env`, which is skipped before cwd resolution in `src/infra/exec-deny-path.ts`.

## Solution Assessment

No. The feature direction is appropriate for #74379, but the current patch is not the narrow maintainable fix because it duplicates shell-wrapper parsing with a lossy tokenizer and filters out bare filenames before the documented resolved-path match can run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before deny matching:** `src/agents/bash-tools.exec.ts:1622`
  - body: The pre-dispatch gate passes the full command string as `shellPayload`. For `bash -c "cat ~/.openclaw/secrets/foo.env"`, the tokenizer keeps the inner command as one token prefixed by `cat `, so a tilde-based deny pattern does not match and gateway/sandbox/elevated execution can continue.
  - confidence: 0.9
- **[P2] Resolve bare sensitive filenames before skipping:** `src/infra/exec-deny-path.ts:211-212`
  - body: The matcher rejects bare args such as `.env` and `credentials.json` because they have no slash, tilde, or absolute path shape. Those args never get resolved against `cwd`, so documented patterns like `**/.env` and `**/credentials.json` miss common commands such as `cat .env`.
  - confidence: 0.88
