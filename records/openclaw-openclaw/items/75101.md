## Summary

Keep open. Current main does not implement `tools.exec.denyPathPatterns`, and the PR is a valid direction for the linked security issue, but latest head still has concrete bypass and cross-host matching bugs in the new exec deny boundary.

## What This Changes

The PR adds an opt-in `tools.exec.denyPathPatterns` hard-deny config with schema/help/docs/changelog updates, a shared matcher helper with unit tests, and gateway/node-host exec policy wiring.

## Best Possible Solution

Revise the PR so one shared deny-path evaluation uses parsed argv and extracted shell payload before every execution path, honors documented cwd-relative matching, avoids gateway-cwd assumptions for remote node defaults, and adds focused integration tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: `bash -c 'cat ~/.openclaw/secrets/foo.env'` with a tilde secret pattern misses the local raw-command gate, `cat .env` with `**/.env` is skipped before cwd resolution, and exact-file patterns miss when a shell operator is glued to the path.

## Solution Assessment

No. The hard-deny control is the right direction for the linked security issue, but this implementation is not the best mergeable fix until the deny gate uses the existing parsed shell-wrapper seams and matches the documented path semantics consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before local deny checks:** `src/agents/bash-tools.exec.ts:1623-1625`
  - body: The local pre-dispatch gate scans the whole command as raw `shellPayload` with no argv. For wrapper forms such as `bash -c 'cat ~/.openclaw/secrets/foo.env'`, the inner path remains glued to preceding command text, so gateway/sandbox/elevated execution can miss a configured deny before dispatch.
  - confidence: 0.92
- **[P1] Split shell operators around path tokens:** `src/infra/exec-deny-path.ts:127-134`
  - body: `tokenizeShellPayload()` only separates on whitespace, so exact deny patterns can miss common shell forms where a path is followed by `;`, `>`, or `&&`. This leaves the new hard-deny gate bypassable for documented exact-file patterns such as `~/.aws/credentials`.
  - confidence: 0.88
- **[P2] Resolve bare cwd paths before filtering:** `src/infra/exec-deny-path.ts:210-212`
  - body: The matcher rejects candidates that are not already path-like before it resolves them against `cwd`, so documented patterns like `**/.env` and `**/credentials.json` do not block `cat .env` or `cat credentials.json` from the current directory.
  - confidence: 0.91
- **[P2] Avoid gateway cwd matching for node defaults:** `src/agents/bash-tools.exec.ts:1625`
  - body: For `host=node` without an explicit workdir, `workdir` is intentionally undefined because the gateway cwd is not the remote node cwd. Passing that through this preflight still lets relative paths resolve against the gateway process before the node-host policy sees the real execution context, causing wrong-machine denials or matches.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The patch adds a useful security control, but latest head still leaves concrete denylist bypass and cross-host misapplication paths in the new exec boundary.

Concerns:

- **[high] Shell wrappers bypass local denyPathPatterns:** `src/agents/bash-tools.exec.ts:1623`
  - body: The gateway/sandbox/elevated pre-dispatch gate scans raw command text instead of parsed argv plus extracted shell-wrapper payload, so quoted wrapper forms can avoid tilde-based secret-path denies.
  - confidence: 0.92
- **[high] Shell operators can evade exact path patterns:** `src/infra/exec-deny-path.ts:127`
  - body: The shell-payload tokenizer does not split control or redirection operators from adjacent path tokens, so exact-file deny patterns can miss ordinary shell syntax such as appending `;` or redirecting after a protected path.
  - confidence: 0.88
- **[medium] Bare sensitive filenames are skipped:** `src/infra/exec-deny-path.ts:210`
  - body: Patterns such as `**/.env` and `**/credentials.json` do not block `cat .env` or `cat credentials.json` because the matcher rejects bare filenames before cwd resolution.
  - confidence: 0.91
