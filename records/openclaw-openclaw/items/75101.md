## Summary

Keep open. Current main still lacks the requested deny-path exec policy and the linked security incident is valid, but this PR should not merge as-is because PR head still has concrete denylist bypasses and a node-host cwd mismatch.

## What This Changes

The PR adds an opt-in `tools.exec.denyPathPatterns` hard-deny exec policy with config/schema/help/docs/changelog wiring, runtime gates, and matcher tests.

## Best Possible Solution

Revise the PR so one shared deny-path gate evaluates parsed argv and shell-wrapper payload segments consistently across gateway, sandbox, elevated, and node execution paths, with regression tests for operator-adjacent paths and bare cwd-relative secrets.

## Reproduction Assessment

Yes. Static reproduction is high-confidence against the PR diff: `cat ~/.openclaw/secrets/foo.env; echo ok` keeps `;` attached in the whitespace tokenizer, and `cat .env` is skipped before cwd resolution despite a `**/.env` pattern.

## Solution Assessment

No. The requested hard-deny policy is the right direction, but the current implementation is not the best fix until it reuses the existing shell parser and resolves documented cwd-relative candidates before filtering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Split shell operators before path matching:** `src/infra/exec-deny-path.ts:121-128`
  - body: `tokenizeShellPayload` only splits on whitespace, so common shell forms like `cat ~/.openclaw/secrets/foo.env; echo ok`, `cat ~/.openclaw/secrets/foo.env>/tmp/leak`, or `cat ~/.openclaw/secrets/foo.env&&true` keep the operator attached to the path token and miss a `~/.openclaw/secrets/**` deny glob. Reuse the existing shell segment parser or split recognized operators before matching.
  - confidence: 0.92
- **[P2] Resolve bare sensitive names before skipping:** `src/infra/exec-deny-path.ts:198-199`
  - body: The matcher skips non-path-like tokens before resolving them against `cwd`, so documented patterns like `**/.env` and `**/credentials.json` do not block common commands such as `cat .env` or `cat credentials.json`. Resolve cwd-relative candidates first or treat configured sensitive basenames as match candidates.
  - confidence: 0.9
- **[P2] Avoid gateway cwd matching for node defaults:** `src/agents/bash-tools.exec.ts:1623`
  - body: For `host="node"` with no explicit workdir, `workdir` is intentionally undefined because the gateway cwd is not meaningful on the remote node, but the new pre-dispatch matcher still lets relative paths fall back to `path.resolve()` on the gateway process. That can deny a node run based on the wrong directory before the node-host policy phase evaluates the real context.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch adds a useful security control, but the current matcher leaves concrete bypasses in the boundary it introduces.

Concerns:

- **[high] Shell metacharacters bypass path denial:** `src/infra/exec-deny-path.ts:121`
  - body: Operator-adjacent path tokens are not normalized before glob matching, so ordinary shell syntax can avoid a configured hard-deny pattern while still reading the denied file.
  - confidence: 0.92
- **[medium] Bare secret filenames are skipped:** `src/infra/exec-deny-path.ts:198`
  - body: Patterns advertised for common secret files such as `**/.env` and `**/credentials.json` do not catch bare cwd-relative reads because the matcher rejects those tokens before resolution.
