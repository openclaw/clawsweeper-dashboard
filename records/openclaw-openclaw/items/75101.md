## Summary

Keep open. Current main still lacks the requested deny-path exec policy, and the latest PR head is a useful implementation attempt, but it leaves concrete bypasses in the security control it adds: local gateway/sandbox/elevated execution scans raw wrapper text instead of parsed inner shell payloads, and the matcher skips bare sensitive filenames before cwd resolution.

## What This Changes

The PR adds an opt-in `tools.exec.denyPathPatterns` exec hard-deny config with schema, docs, changelog, helper tests, and enforcement before allowlist, approval, and safe-bin checks.

## Best Possible Solution

Land a maintainer-reviewed version where one shared deny-path policy evaluates parsed command segments or inner shell payloads before every execution path, treats cwd-resolvable sensitive filenames as candidates, keeps schema/docs aligned, and includes gateway, sandbox/elevated, and node integration coverage.

## Reproduction Assessment

Yes. Configure `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` and run local gateway/sandbox/elevated `bash -c "cat ~/.openclaw/secrets/foo.env"`; the new pre-dispatch gate scans raw wrapper text and misses the tilde pattern. A second static reproduction is `tools.exec.denyPathPatterns: ["**/.env"]` with `cat .env`, which is skipped before cwd resolution.

## Solution Assessment

No. The deny-path gate is the right product direction for the linked incident, but this implementation needs to reuse parsed shell-wrapper payloads on local/elevated paths and broaden candidate selection for bare cwd-relative sensitive filenames before it is the narrow maintainable fix.
