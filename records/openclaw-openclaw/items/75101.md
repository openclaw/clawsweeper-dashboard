## Summary

Keep open. Current main does not have this exec path-deny surface, and the linked secret-exfiltration report remains valid, but this PR should not merge as-is because the latest head still misses wrapped shell commands and bare sensitive filenames while widening a security review with unrelated package/channel changes.

## What This Changes

The branch adds opt-in `tools.exec.denyPathPatterns` config, schema, docs, tests, and exec-policy enforcement, while also changing Matrix DM owner resolution and removing the WhatsApp `qrcode` dependency.

## Best Possible Solution

Land a maintainer-reviewed version that evaluates one shared deny-path policy against parsed argv plus extracted shell-wrapper inner payloads before every gateway, sandbox, elevated, and node execution path. Resolve cwd-relative bare sensitive filenames before candidate filtering, remove duplicate helper files and unrelated package/channel changes, and keep config, schema, docs, changelog, and tests aligned with the supported v1 scope.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: configure `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` and send `bash -c 'cat ~/.openclaw/secrets/foo.env'` through the `createExecTool` path; the PR scans raw wrapper text at `src/agents/bash-tools.exec.ts:1622`. A second reproduction is `tools.exec.denyPathPatterns: ["**/.env"]` with `cat .env`, which is skipped before cwd resolution in `src/infra/exec-deny-path.ts`.

## Solution Assessment

No. The requested direction is the right core fix for the linked report, but this implementation is not the narrowest maintainable solution until it reuses existing wrapper extraction, resolves bare filenames before filtering, and splits unrelated package/channel changes out of the security PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before deny matching:** `src/agents/bash-tools.exec.ts:1622`
  - body: The pre-dispatch gate passes the full raw command as `shellPayload` with no argv. For `bash -c 'cat ~/.openclaw/secrets/foo.env'`, tokenization keeps the inner command as one token prefixed by `cat `, so a `~/.openclaw/secrets/**` pattern does not match and gateway/sandbox/elevated execution can continue. Feed the matcher parsed argv plus the existing shell-wrapper inner payload extraction before dispatch.
  - confidence: 0.9
- **[P2] Resolve bare filenames before filtering:** `src/infra/exec-deny-path.ts:198-199`
  - body: The matcher rejects bare args such as `.env` and `credentials.json` before resolving them against `cwd`. That means documented patterns like `**/.env` and `**/credentials.json` miss common commands such as `cat .env`; resolve candidates first or treat configured sensitive basenames as path candidates.
  - confidence: 0.88
- **[P3] Remove the duplicate node-host matcher:** `src/node-host/exec-deny-path.ts:1`
  - body: The live imports now use `src/infra/exec-deny-path.ts`, but this PR still adds the old `src/node-host/exec-deny-path.ts` and its tests. Keeping two deny matcher implementations makes future accidental imports and behavior drift likely, so delete the stale copy as part of the move.
  - confidence: 0.78
