## Summary

Keep open: the feature is still absent from current main, and the latest PR head is useful but not mergeable because the new deny gate still has concrete shell-wrapper and cwd-relative path gaps in the security boundary it introduces.

## What This Changes

Adds an opt-in `tools.exec.denyPathPatterns` exec hard-deny config with schema/help/docs/changelog updates, a shared matcher helper with unit tests, and gateway/node-host policy wiring.

## Best Possible Solution

Revise the PR so one shared deny-path path evaluates parsed argv plus extracted shell-wrapper payload before every execution path, resolves sensitive cwd-relative basenames correctly, avoids gateway-cwd assumptions for remote node defaults, and adds focused integration tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: on PR head, `bash -c 'cat ~/.openclaw/secrets/foo.env'` with `~/.openclaw/secrets/**` misses in the local gate, and `cat .env` with `**/.env` is skipped before cwd resolution.

## Solution Assessment

No. The requested hard-deny control is the right direction for #74379, but this implementation is not the narrowest maintainable fix until it reuses the parsed shell-wrapper seam and matches relative sensitive filenames according to the documented cwd semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before local deny checks:** `src/agents/bash-tools.exec.ts:1623-1624`
  - body: The local pre-dispatch gate scans the whole command string as `shellPayload` with no argv. For `bash -c 'cat ~/.openclaw/secrets/foo.env'`, tokenization keeps the inner command as one token prefixed by `cat `, so a `~/.openclaw/secrets/**` deny pattern is missed before gateway, sandbox, or elevated-full dispatch. Feed parsed argv plus the extracted shell-wrapper payload into the matcher here.
  - confidence: 0.92
- **[P2] Resolve bare cwd paths before filtering:** `src/infra/exec-deny-path.ts:210-212`
  - body: The matcher drops candidates that are not considered path-like before it resolves them against `cwd`, so documented patterns like `**/.env` and `**/credentials.json` do not block common commands such as `cat .env` or `cat credentials.json`. Resolve relative candidates first, or explicitly treat configured sensitive basenames as path candidates.
  - confidence: 0.91
- **[P2] Avoid gateway cwd matching for node defaults:** `src/agents/bash-tools.exec.ts:1625`
  - body: When `host=node` has no explicit workdir, `workdir` is intentionally left undefined because the gateway cwd is not meaningful on the remote node. Passing that through still lets `path.resolve()` use the gateway process cwd, so relative node commands can be denied against the wrong machine before the node-host policy sees the real context. Defer relative matching to the node host or disable cwd-based matching in this preflight case.
  - confidence: 0.83
