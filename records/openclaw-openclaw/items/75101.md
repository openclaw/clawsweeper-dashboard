## Summary

Keep open. Current main does not implement the requested exec path denylist, and the latest PR still has blocking bypasses in the new security gate. The direction matches the linked #74379 report and related wrapper-denial discussion, but the patch should not merge until shell wrappers are parsed before deny matching, cwd-resolved bare sensitive filenames are covered, and unrelated WhatsApp dependency churn is split or explicitly justified.

## What This Changes

The PR adds opt-in `tools.exec.denyPathPatterns` config, schema, docs, tests, and pre-exec deny checks, while also removing the WhatsApp plugin's `qrcode` dependency and lockfile entries.

## Best Possible Solution

Land a maintainer-reviewed exec denylist that evaluates one shared policy against parsed argv and extracted shell-wrapper payloads before every gateway, sandbox, elevated, and node execution path. It should resolve cwd-relative bare sensitive filenames, keep config/schema/docs/changelog/tests aligned, and split unrelated plugin dependency cleanup into a separate PR.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` plus gateway/sandbox/elevated `bash -c 'cat ~/.openclaw/secrets/foo.env'`, where the PR scans the raw wrapper text. A second reproduction is `tools.exec.denyPathPatterns: ["**/.env"]` plus `cat .env`, which is skipped before cwd resolution.

## Solution Assessment

No. The requested core direction is appropriate for the linked security report, but this implementation is not the best fix until it reuses existing shell-wrapper extraction, resolves bare sensitive filenames before filtering them out, and separates the WhatsApp package change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before deny matching:** `src/agents/bash-tools.exec.ts:1619-1623`
  - body: The pre-dispatch gate passes the full raw command as `shellPayload` with no argv. For `bash -c 'cat ~/.openclaw/secrets/foo.env'`, tokenization keeps the inner command as one token prefixed by `cat `, so a `~/.openclaw/secrets/**` pattern does not match and gateway/sandbox/elevated execution can continue. Feed the matcher parsed argv plus extracted shell-wrapper inner payloads before dispatch.
  - confidence: 0.91
- **[P2] Resolve bare filenames before filtering:** `src/infra/exec-deny-path.ts:210-212`
  - body: The matcher rejects bare args such as `.env` and `credentials.json` before resolving them against `cwd`. That means documented patterns like `**/.env` and `**/credentials.json` miss common commands such as `cat .env`; resolve candidates first or treat configured sensitive basenames as path candidates.
  - confidence: 0.89
