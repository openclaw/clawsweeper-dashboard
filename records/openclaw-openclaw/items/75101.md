## Summary

Keep open: current main does not include this security config, and the PR is a plausible implementation path, but latest head still has concrete bypasses/false-denial issues in the new deny boundary that should block merge.

## What This Changes

The PR adds `tools.exec.denyPathPatterns` config/schema/docs/changelog support and runtime path-deny matching before exec allowlist/approval handling in gateway and node-host paths.

## Best Possible Solution

Land the default-off deny-path config after the gateway and node-host gates share the existing shell-wrapper resolution semantics and regression tests cover Windows paths, `cmd`/PowerShell payloads, POSIX line continuations, and comments.

## Reproduction Assessment

Yes. Source inspection on latest PR head gives high-confidence repro paths for Windows backslash paths, `cmd`/PowerShell wrapper payloads, POSIX backslash-newline continuation, and unquoted comment handling, though I did not execute those cases live.

## Solution Assessment

No, not in the current form. The config direction matches the linked security request, but the mergeable solution should reuse the established wrapper resolver and make the deny tokenizer match the shell path forms it claims to protect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1416`
  - body: The gateway deny gate parses the command with `splitShellArgs()`, which treats backslashes as POSIX escapes. On a Windows gateway, `type C:\Users\alice\.ssh\id_rsa` is reduced before `denyPathPatterns` sees it, so configured Windows secret paths can bypass the new gate.
  - confidence: 0.9
- **[P1] Scan cmd and PowerShell payloads in the gateway gate:** `src/agents/bash-tools.exec.ts:1427-1430`
  - body: The new gateway gate extracts inner payloads only with the local POSIX-shell helper, while the existing shared resolver already handles `cmd.exe /c` and `pwsh -Command`. A denied file can still be reached by wrapping the read in `cmd.exe` or PowerShell because that nested payload is never tokenized for denial.
  - confidence: 0.88
- **[P1] Match shell backslash semantics before denying paths:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` treats every backslash as an escape and appends escaped newlines. That corrupts Windows paths inside shell payloads and misses POSIX `\`-newline continuations, where the shell joins the denied path before execution but the matcher sees a token containing a newline.
  - confidence: 0.86
- **[P2] Ignore unquoted shell comments during payload tokenization:** `src/infra/exec-deny-path.ts:149-155`
  - body: The payload tokenizer never stops at an unquoted `#` that begins a shell comment, so `bash -c 'echo ok # ~/.openclaw/secrets/foo.env'` is denied even though the shell ignores the path. This should follow the existing shell-argv comment behavior to avoid false denials.
  - confidence: 0.8
