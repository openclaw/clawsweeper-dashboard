## Summary

Keep open: current main does not have this hard-deny config surface, and the PR targets a real security gap from #74379, but latest head still leaves concrete denylist bypasses before it is mergeable.

## What This Changes

The PR adds an opt-in `tools.exec.denyPathPatterns` exec hard-deny gate with schema/help/docs/changelog updates, global/per-agent unioning, and helper tests for argv and shell-payload matching.

## Best Possible Solution

Land a revised hard-deny control that uses the shared parsed argv/shell-wrapper seams before every exec host, treats sensitive cwd-relative basenames as match candidates, handles ordinary shell token boundaries, and adds integration coverage for gateway, sandbox/elevated, and node-host policy order.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: `bash -c 'cat ~/.openclaw/secrets/foo.env'` is missed by the local raw-text gate, `cat .env` is skipped before cwd resolution for `**/.env`, and an exact pattern such as `~/.aws/credentials` can be bypassed by an adjacent shell operator.

## Solution Assessment

No. A path denylist is the right direction for #74379, but this implementation is not the narrowest safe fix until it normalizes commands through the existing parser seams and closes the remaining token-matching gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before local deny matching:** `src/agents/bash-tools.exec.ts:1621-1625`
  - body: The local pre-dispatch gate scans raw `params.command` with an empty argv. For `bash -c 'cat ~/.openclaw/secrets/foo.env'`, the inner command becomes one token prefixed with `cat `, so a `~/.openclaw/secrets/**` deny pattern is missed before gateway, sandbox, or elevated-full dispatch. Feed parsed argv plus extracted shell-wrapper payload into this gate.
  - confidence: 0.91
- **[P2] Resolve bare filenames before filtering:** `src/infra/exec-deny-path.ts:210-212`
  - body: The matcher rejects bare args such as `.env` or `credentials.json` before resolving them against `cwd`. That means documented patterns like `**/.env` and `**/credentials.json` do not block common commands such as `cat .env`, leaving the configured hard deny incomplete.
  - confidence: 0.9
- **[P2] Tokenize shell operators before matching paths:** `src/infra/exec-deny-path.ts:127-134`
  - body: `tokenizeShellPayload()` only splits on whitespace, so exact documented paths can stay glued to shell syntax. For example, `cat ~/.aws/credentials; echo ok` produces `~/.aws/credentials;`, which does not match the exact `~/.aws/credentials` deny pattern and still reads the file.
  - confidence: 0.84

## Security Review

  - body: Gateway, sandbox, and elevated-full local execution depend on the new pre-dispatch gate, but that gate scans raw shell text instead of parsed argv and extracted inner payloads, so wrapped secret reads can avoid a tilde-based deny pattern.
  - confidence: 0.91

  - body: Patterns like `**/.env` and `**/credentials.json` do not block `cat .env` or `cat credentials.json` because the matcher rejects bare filenames before resolving them against the working directory.

  - body: The tokenizer keeps adjacent shell syntax on path tokens, so exact deny entries such as `~/.aws/credentials` can be bypassed with ordinary shell separators while still executing the sensitive read.
