## Summary

Keep open. The linked request is a real core exec-security gap and current main still has no deny-path config or enforcement, but this PR is not landable because the latest head still has two concrete bypasses in the new hard-deny gate and mixes unrelated Matrix/WhatsApp changes into the security patch.

## What This Changes

The branch adds opt-in `tools.exec.denyPathPatterns` config, schema, docs, tests, and exec-deny enforcement, while also changing Matrix DM owner resolution and removing the WhatsApp plugin's `qrcode` dependency.

## Best Possible Solution

Land a maintainer-reviewed version that evaluates one shared deny-path policy against parsed argv plus extracted shell-wrapper inner payloads before every gateway, sandbox, elevated, and node execution path. Resolve cwd-relative bare sensitive filenames before filtering, split unrelated Matrix/WhatsApp changes into separate PRs, and keep config, schema, docs, changelog, and targeted tests aligned with the documented v1 scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` plus `bash -c 'cat ~/.openclaw/secrets/foo.env'` through `createExecTool`; the PR scans the full raw wrapper text, so the inner path remains part of a single `cat ...` token. A second reproduction is `tools.exec.denyPathPatterns: ["**/.env"]` plus `cat .env`, which is skipped before cwd resolution.

## Solution Assessment

No. The requested direction is the right core fix for the linked report, but this implementation is not the narrowest maintainable solution until it reuses existing wrapper extraction, resolves bare filenames before filtering, and removes unrelated channel/package changes from the security PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before deny matching:** `src/agents/bash-tools.exec.ts:1622`
  - body: The pre-dispatch gate passes the full raw command as `shellPayload` with no argv. For `bash -c 'cat ~/.openclaw/secrets/foo.env'`, tokenization keeps the inner command as one token prefixed by `cat `, so a `~/.openclaw/secrets/**` pattern does not match and gateway/sandbox/elevated execution can continue; feed the matcher parsed argv plus extracted shell-wrapper inner payloads before dispatch.
  - confidence: 0.9
- **[P2] Resolve bare filenames before filtering:** `src/infra/exec-deny-path.ts:211-212`
  - body: The matcher rejects bare args such as `.env` and `credentials.json` before resolving them against `cwd`. That means documented patterns like `**/.env` and `**/credentials.json` miss common commands such as `cat .env`; resolve candidates first or treat configured sensitive basenames as path candidates.
  - confidence: 0.88
