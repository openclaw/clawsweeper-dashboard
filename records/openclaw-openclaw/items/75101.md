## Summary

Keep this PR open: it adds the right hard-deny surface for #74379 and includes real after-fix output, but the latest head still has concrete deny bypasses in Windows command parsing, Windows shell-wrapper payload extraction, and shell backslash handling.

## What This Changes

Adds `tools.exec.denyPathPatterns` config/schema/docs/changelog support plus shared glob matching and pre-approval exec denial in gateway and node-host paths.

## Best Possible Solution

Land a default-off deny-path config only after the gateway and node-host gates share the existing wrapper-resolution semantics and the matcher has regression coverage for Windows paths, cmd/PowerShell payloads, POSIX line continuations, and comments.

## Reproduction Assessment

Yes, source inspection gives high-confidence failing paths on latest PR head: Windows-style paths are corrupted before matching, cmd/PowerShell payloads are not extracted in the gateway gate, and POSIX backslash-newline continuations leave a denied path unmatched. I did not run a live Windows or shell execution test in this read-only review.

## Solution Assessment

No, not in the current form. The config direction matches #74379, but the mergeable fix should reuse the established shell-wrapper resolver and make the deny tokenizer match the shell path forms it claims to protect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1416`
  - body: The gateway deny gate parses the command with `splitShellArgs()`, which treats backslashes as POSIX escapes. On a Windows gateway, `type C:\Users\alice\.ssh\id_rsa` is reduced before `denyPathPatterns` sees it, so configured Windows secret paths can bypass the new gate.
  - confidence: 0.9
- **[P1] Scan cmd and PowerShell payloads in the gateway gate:** `src/agents/bash-tools.exec.ts:1426-1430`
  - body: The new gateway gate extracts inner payloads only with the local POSIX-shell helper, while the existing shared resolver already handles `cmd.exe /c` and `pwsh -Command`. A denied file can still be reached by wrapping the read in `cmd.exe` or PowerShell because that nested payload is never tokenized for denial.
  - confidence: 0.88
- **[P1] Match shell backslash semantics before denying paths:** `src/infra/exec-deny-path.ts:124-132`
  - body: `tokenizeShellPayload()` treats every backslash as an escape and appends escaped newlines. That corrupts Windows paths inside shell payloads and misses POSIX `\`-newline continuations, where the shell joins the denied path before execution but the matcher sees a token containing a newline.
  - confidence: 0.86
- **[P2] Ignore unquoted shell comments during payload tokenization:** `src/infra/exec-deny-path.ts:149-155`
  - body: The payload tokenizer never stops at an unquoted `#` that begins a shell comment, so `bash -c 'echo ok # ~/.openclaw/secrets/foo.env'` is denied even though the shell ignores the path. This should follow the existing shell-argv comment behavior to avoid false denials.
  - confidence: 0.8
