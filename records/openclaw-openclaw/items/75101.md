## Summary

Keep open. The linked security problem is valid and the feature direction is appropriate, but the current PR still leaves concrete denylist bypasses in the primary exec paths and includes stale/unrelated package-surface churn that should be cleaned up before maintainer review can proceed.

## What This Changes

The PR adds opt-in `tools.exec.denyPathPatterns` config, schema, docs, changelog, matcher tests, and exec-policy enforcement, while also removing `qrcode` from the WhatsApp plugin package metadata and lockfile.

## Best Possible Solution

Land a maintainer-reviewed version that evaluates one shared deny-path policy against parsed argv plus existing shell-wrapper inner payload extraction before every gateway, sandbox, elevated, and node execution path. Resolve cwd-relative bare sensitive filenames before candidate filtering, remove duplicate helper files and unrelated package changes, and keep config/schema/docs/changelog/tests aligned with direct argv, wrapped shell payload, global-plus-agent union, allowlist/full bypass resistance, and `.env`/`credentials.json` coverage.

## Reproduction Assessment

Yes. Configure `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` and send `bash -c 'cat ~/.openclaw/secrets/foo.env'` through the gateway/sandbox/elevated `createExecTool` path; the PR scans raw wrapper text at `src/agents/bash-tools.exec.ts:1622`, so the tilde pattern is missed. A second static reproduction is `tools.exec.denyPathPatterns: ["**/.env"]` with `cat .env`, which is skipped before cwd resolution in `src/infra/exec-deny-path.ts`.

## Solution Assessment

No. The requested direction is the right core fix for #74379, but this diff is not the best implementation because it duplicates shell-wrapper parsing with a lossy tokenizer, filters out bare filenames before resolved-path matching, and mixes unrelated dependency cleanup into the security change.

  - body: The pre-dispatch gate passes the full raw command as `shellPayload`.
