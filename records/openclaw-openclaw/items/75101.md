## Summary

Keep open and do not merge yet. The PR addresses a real exec security gap, but latest head still has source-reproducible bypasses and false-denial behavior in the new deny-path boundary.

## What This Changes

The PR adds `tools.exec.denyPathPatterns` config/schema/docs/tests and runtime enforcement before exec allowlist, approval, and safe-bin handling.

## Best Possible Solution

Land a default-off deny-path config only after the gateway and node-host gates share existing shell-wrapper semantics and regression tests cover Windows paths, cmd/PowerShell payloads, POSIX line continuations, and comments.

## Reproduction Assessment

Yes. Source inspection of PR head gives concrete repro paths for Windows backslash corruption, cmd/PowerShell wrapper payload bypasses, POSIX backslash-newline bypasses, and shell-comment false denials.

## Solution Assessment

No, not in the current form. The config direction matches the linked security request, but the mergeable solution should reuse the established wrapper resolver and make deny tokenization match the shell path forms it claims to protect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1416`
  - body: The gateway deny gate parses command text with `splitShellArgs()`, which treats backslashes as POSIX escapes. On a Windows gateway, `type C:\Users\alice\.ssh\id_rsa` is reduced before `denyPathPatterns` sees it, so configured Windows secret paths can bypass the new gate.
  - confidence: 0.9
- **[P1] Scan cmd and PowerShell payloads in the gateway gate:** `src/agents/bash-tools.exec.ts:1427-1430`
  - body: The gateway gate extracts inner payloads only for POSIX shells instead of using the shared shell-wrapper resolver. Denied paths embedded in `cmd.exe /c` or `pwsh -Command` payloads can remain a single command-string argument, so home-expanded patterns are missed instead of being checked against the real payload tokens.
  - confidence: 0.87
- **[P1] Match shell backslash semantics before denying paths:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` treats every backslash as an escape and appends escaped newlines. That corrupts Windows paths inside shell payloads and misses POSIX `\`-newline continuations, where the shell joins the denied path before execution but the matcher sees a token containing a newline.
  - confidence: 0.86
- **[P2] Ignore unquoted shell comments during payload tokenization:** `src/infra/exec-deny-path.ts:142-156`
  - body: The payload tokenizer never stops at an unquoted `#` that begins a shell comment, so `bash -c 'echo ok # ~/.openclaw/secrets/foo.env'` is denied even though the shell ignores the path. This should follow the existing shell-argv comment behavior to avoid false denials.
  - confidence: 0.8
