## Summary

This PR is still useful because current main still requires Slack `member-info` `userId` even though the shared message schema allows omission; the patch looks like the right narrow Slack dispatcher fix, but it should stay open until the external live Slack proof gate is satisfied.

## What This Changes

The PR defaults Slack `member-info` action `userId` to the trusted inbound sender when omitted and adds dispatch tests for fallback, missing-sender error, and explicit-user precedence.

## Best Possible Solution

Land the Slack plugin-layer fallback after redacted live Slack proof shows omitted `userId` resolves to the inbound sender and explicit `userId` still wins.

## Reproduction Assessment

Yes, from source: current main exposes `userId` as optional in the shared message action schema while Slack dispatch still requires it and throws before invoking Slack.

## Solution Assessment

Yes, pending live proof: the Slack dispatcher is the narrow plugin-owned translation point, while the lower Slack runtime can continue requiring an explicit `memberInfo` `userId`. The Discord parity note can be handled as a separate follow-up instead of broadening this PR.
