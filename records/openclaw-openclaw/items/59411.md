## Summary

Keep this PR open: current main still lacks timezone-aware session reset and the proposed idleGuard API, and https://github.com/openclaw/openclaw/pull/71400 only covers the daily-plus-idle AND semantics. The branch is not mergeable as-is because it has timezone correctness bugs, changes omitted-timezone fallback behavior, overlaps a competing public API, and lacks external real behavior proof.

## What This Changes

The PR adds optional `session.reset.timezone` and `session.reset.idleGuard` fields, wires user-timezone fallback through session reset policy resolution, updates help/schema/types, and adds focused reset tests.

## Best Possible Solution

Choose one public API for daily-plus-idle AND semantics, keep timezone-aware daily boundaries as a separately validated config addition if accepted, and port the final change onto current reset-policy lifecycle semantics.

## Reproduction Assessment

Yes. Current main can be verified by source inspection, and the PR-specific blockers are reproducible from the proposed schema and timezone math: invalid IANA strings throw at runtime and valid UTC+13/+14 offsets are normalized incorrectly.

## Solution Assessment

No, not as-is. The useful timezone idea should be validated and ported separately, while the AND-semantics API needs maintainer choice against the competing adaptive-mode PR before repair automation is appropriate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A config typo such as `UTC+8` throws `RangeError` during session freshness evaluation, so this needs schema or policy-boundary validation before runtime use.
  - confidence: 0.91
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The proposed offset normalization treats differences above 12 hours as day wrap, but valid IANA zones such as `Pacific/Tongatapu` and `Pacific/Kiritimati` exceed +12. Those users would get inverted offsets and reset boundaries on the wrong day.
  - confidence: 0.92
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-187`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently skipped and the config behaves as daily-only, which is misleading for an idle guard.
  - confidence: 0.84
- **[P2] Keep omitted reset timezones host-local:** `src/config/sessions/reset.ts:132-133`
  - body: Using `agents.defaults.userTimezone` when `session.reset.timezone` is omitted changes existing daily reset behavior for users who configured prompt timezone but expected reset boundaries to stay on the gateway host clock, contradicting the new help text and compatibility note.
  - confidence: 0.8
- **[P2] Regenerate the config docs baseline:** `src/config/zod-schema.session.ts:21-22`
  - body: This changes the public config schema/help surface but does not update the generated config baseline hash. Current main's config docs check reports that drift and asks contributors to run `pnpm config:docs:gen` for intentional config changes.
  - confidence: 0.86
