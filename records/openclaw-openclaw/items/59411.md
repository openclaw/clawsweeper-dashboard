## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested timezone-aware session reset support, and #71400 only covers the daily-plus-idle AND semantics, but this patch has concrete correctness issues and needs maintainer API judgment before it is ported onto the current reset-policy implementation.

## What This Changes

The PR adds optional `timezone` and `idleGuard` fields to session reset config, threads user timezone fallback into session policy resolution call sites, updates config schema/help/types, and adds reset policy tests.

## Best Possible Solution

Decide the reset API shape first, then port only the accepted behavior onto `src/config/sessions/reset-policy.ts`: keep existing daily/idle semantics stable, add validated IANA timezone support if desired, reconcile AND semantics with #71400, and update generated config/docs/changelog plus focused reset-policy tests.

## Reproduction Assessment

Yes. The current-main gap is reproducible by inspecting the strict schema and reset policy, and the PR-specific defects have concrete checks: invalid IANA IDs throw `RangeError`, and UTC+13 zones produce offsets above the PR's wrap threshold.

## Solution Assessment

No, not as-is. The timezone capability is useful, but the proposed implementation needs validation, real-world offset handling, documented fallback semantics, and a port onto the current reset-policy module; the AND behavior should be reconciled with #71400 rather than landing a competing flag casually.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A typo such as `UTC+8` throws `RangeError` during session freshness evaluation, so this needs schema or policy-boundary validation before runtime use.
  - confidence: 0.9
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The proposed offset normalization treats differences above 12 hours as day wrap, but valid IANA zones such as `Pacific/Tongatapu` and `Pacific/Kiritimati` exceed +12. Those users would get inverted offsets and reset boundaries on the wrong day.
  - confidence: 0.91
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-187`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently skipped and the config behaves as daily-only, which is misleading for an idle guard.
  - confidence: 0.84
- **[P2] Keep omitted timezone configs host-local:** `src/auto-reply/reply/session.ts:382`
  - body: Passing `agents.defaults.userTimezone` as a fallback changes existing daily reset behavior for users who configured prompt timezone but did not set `session.reset.timezone`. That contradicts the current host-local session docs and the PR's compatibility claim.
  - confidence: 0.79
