## Summary

Keep this PR open for maintainer handling, but it is not merge-ready: it proposes a useful timezone reset feature plus a competing daily+idle AND API, while the branch has known timezone correctness bugs, missing real behavior proof, and overlap with the adaptive-mode implementation thread.

## What This Changes

The PR adds optional `session.reset.timezone` and `session.reset.idleGuard` fields, wires timezone fallback through session reset policy resolution, updates help/schema/types, and adds focused session reset tests.

## Best Possible Solution

Split the accepted reset contract into one coherent API: use the canonical adaptive-mode path for AND semantics if maintainers choose it, and land timezone-aware daily boundaries only after validation, offset math, docs, generated config metadata, and lifecycle semantics are correct.

## Reproduction Assessment

Yes. Current main is source-reproducible as host-local daily reset with OR daily/idle semantics, and the PR-specific blockers are visible in the proposed schema/timezone math plus the local `Intl.DateTimeFormat` RangeError check.

## Solution Assessment

No, not as-is. The timezone idea is useful, but this patch needs validation and math fixes, and the AND-semantics API should be resolved against https://github.com/openclaw/openclaw/pull/71400 before polishing this branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A config typo such as `UTC+8` throws `RangeError` during session freshness evaluation, so this needs schema or policy-boundary validation before runtime use.
  - confidence: 0.91
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The proposed offset normalization treats differences above 12 hours as day wrap, but valid IANA zones such as `Pacific/Tongatapu` and `Pacific/Kiritimati` exceed +12. Those users would get inverted offsets and reset boundaries on the wrong day.
  - confidence: 0.92
- **[P2] Keep omitted reset timezones host-local:** `src/config/sessions/reset.ts:132-133`
  - body: When `session.reset.timezone` is omitted, falling back to `agents.defaults.userTimezone` changes existing daily reset behavior for users who configured a prompt timezone but expected reset boundaries to stay on the gateway host clock. That contradicts the help text and the PR's compatibility note.
  - confidence: 0.84
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-187`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently skipped and the config behaves as daily-only, which is misleading for an idle guard.
  - confidence: 0.84
- **[P2] Update the generated config docs baseline:** `src/config/zod-schema.session.ts:21-22`
  - body: Adding `timezone` and `idleGuard` changes the public config schema/help surface, but the PR only edits source schema/help. The config docs baseline hash must be regenerated for this intentional config-surface change.
  - confidence: 0.82
