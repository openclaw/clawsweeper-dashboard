## Summary

Keep open, but do not merge as-is: the feature is plausible and current main does not provide timezone-aware reset or daily-plus-idle AND semantics, but the branch has correctness blockers, lacks external real behavior proof, and overlaps with the open adaptive reset API work.

## What This Changes

The PR adds optional `session.reset.timezone` and `session.reset.idleGuard`, wires user-timezone fallback into session reset policy callers, updates schema/help/types, and adds focused session reset tests.

## Best Possible Solution

Choose one reset contract, then land a coherent implementation with runtime semantics, exported types, Zod schema, generated config baseline, docs, and focused tests aligned; timezone-aware daily boundaries may need to be split from the adaptive/AND API if maintainers prefer the narrower mode.

## Reproduction Assessment

Yes for the current behavior and PR blockers. Current main is source-reproducible as host-local daily reset with OR daily/idle semantics, and the proposed timezone validation and UTC+13/+14 math bugs are visible from source plus focused Node checks.

## Solution Assessment

No, not as-is. The timezone idea is useful, but this patch needs validation and math fixes, and the AND-semantics API should be resolved against https://github.com/openclaw/openclaw/pull/71400 before polishing this branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A config typo such as `UTC+8` throws `RangeError` during session freshness evaluation, so this needs schema or policy-boundary validation before runtime use.
  - confidence: 0.91
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The offset normalization treats differences above 12 hours as day wrap, but valid IANA zones such as `Pacific/Tongatapu` and `Pacific/Kiritimati` exceed +12. Those users would get inverted offsets and reset boundaries on the wrong day.
  - confidence: 0.92
- **[P2] Keep omitted reset timezones host-local:** `src/config/sessions/reset.ts:135-136`
  - body: When `session.reset.timezone` is omitted, falling back to `agents.defaults.userTimezone` changes existing daily reset behavior for users who configured a prompt timezone but expected reset boundaries to stay on the gateway host clock. That contradicts the help text and the PR's compatibility note.
  - confidence: 0.84
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-181`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently skipped and the config behaves as daily-only, which is misleading for an idle guard.
  - confidence: 0.84
- **[P2] Update the config docs baseline:** `src/config/zod-schema.session.ts:21-22`
  - body: Adding `timezone` and `idleGuard` changes the public config schema/help surface, but the PR only edits source schema/help. The config docs baseline hash must be regenerated after the API shape is finalized.
  - confidence: 0.82
