## Summary

Keep open: this PR targets a valid source-reproducible context-window bug, but the current diff still misses the session metadata and `/status` override paths and needs adaptation to current-main provider lookup semantics before it can merge.

## What This Changes

The PR caps positive context-token overrides in the shared agent resolver by configured model context windows, adds resolver tests, adjusts one model-config fixture, and raises one plugin-loader test timeout.

## Best Possible Solution

Rebase or replace this with a current-main patch that caps known effective context limits consistently in the shared resolver, session persistence, and status summary while preserving provider lookup semantics and unknown-model override behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by passing a positive `contextTokensOverride` larger than a known configured provider/model window into the shared resolver, session-store persistence, or status-summary path; current main returns or persists the override first.

## Solution Assessment

No, not yet. Capping in the shared resolver is the right direction, but this PR misses the session metadata and `/status` bypasses and should reuse the fuller current-main provider context-token lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P2] Cap every reported override path:** `src/agents/context.ts:288-289`
  - body: The PR body calls out session metadata and `/status`, but the patch only caps the shared resolver. `session-store.ts` still persists `params.contextTokensOverride` before consulting the model window, and the status-summary runtime still has its own override-first resolver, so both reported paths can keep over-reporting after this change. Thread the capped resolution through those callers, or apply the same min logic there, with regressions for both paths.
  - confidence: 0.9

Current main also supports `contextTokens`, provider-level defaults, exact-provider precedence before alias fallback, and slash-model guards, so after rebase this cap can miss or misread known limits unless it reuses or mirrors the existing provider context-token lookup.
  - confidence: 0.84
