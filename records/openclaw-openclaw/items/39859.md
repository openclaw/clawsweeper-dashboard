## What This Changes

The PR caps context-token overrides inside `resolveContextTokensForModel`, adds regression coverage for configured model windows and provider aliases, and adjusts nearby test fixtures/timeouts.

## Best Possible Solution

Rebase or replace this with a narrow current-main fix that caps overrides by the effective known model limit across shared resolution, session persistence, and status-summary reporting while preserving override behavior when the model limit is unknown.

## Reproduction Assessment

Yes. Source inspection shows current main returns a positive override before consulting known model limits in the shared resolver, session persistence, and status-summary paths.

## Solution Assessment

No, not yet. The direction is right, but the PR needs to use current main's effective `contextTokens` precedence and cover the raw session/status override paths, not only the old shared resolver branch.

Because the linked bug includes persisted metadata and `/status`, those paths need the same cap and regression coverage.
