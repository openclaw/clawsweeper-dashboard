## Summary

Keep open: the shared resolver change is in the right direction, but the branch still needs real behavior proof and does not yet cover all session/status paths that can over-report an oversized context override.

## What This Changes

The PR changes `resolveContextTokensForModel` to cap a positive context-token override by a known model window, adds resolver regression tests including provider aliases, adjusts one provider fixture, and increases one plugin-loader test timeout.

## Best Possible Solution

Rebase or replace the branch with a current-main fix that applies the min cap consistently in the shared resolver, session-store persistence, and status-summary/reporting paths while preserving current provider lookup semantics and adding focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by passing a positive `contextTokensOverride` larger than a known configured model window into current main's resolver, session-store, or status-summary paths; those paths still return or persist the override first.

## Solution Assessment

No, not yet. Capping in the shared resolver is the right core direction, but the current PR does not cover the persistence/status bypasses and should reuse current main's fuller provider context-token lookup semantics.
