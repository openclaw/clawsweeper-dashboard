## Summary

Keep open: the underlying bug is still valid on current main, but this PR is not merge-ready because its resolver change is stale relative to current provider-aware resolution and it lacks required after-fix real behavior proof.

## What This Changes

The PR changes `resolveContextTokensForModel` to cap a positive override by a configured or discovered model window, adds resolver regressions, and adjusts related test fixtures.

## Best Possible Solution

Revise against current main by applying configured overrides as ceilings after the existing provider-aware model-window resolution, then cover session persistence and status/reporting callers with focused regressions.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: configure `agents.defaults.contextTokens` above a configured or discovered selected-model window, then follow `resolveContextTokensForModel` or session persistence where the larger override still wins first.

## Solution Assessment

No. The `min(override, knownWindow)` direction is right, but this PR should layer that cap over current main's configured-provider resolver and update remaining raw session/status override paths instead of replacing the resolver with narrower lookup logic.
