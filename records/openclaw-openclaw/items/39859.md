## Summary

Keep open: this PR targets a real source-reproducible context-window bug, but the patch still leaves reported session persistence and status-summary override paths uncapped and uses a narrower provider lookup than current main now supports.

## What This Changes

The PR changes `resolveContextTokensForModel` to cap positive context-token overrides by a known model window, adds resolver regression tests, adjusts one model-config test fixture, and raises one plugin-loader test timeout.

## Best Possible Solution

Rebase or replace the branch with a current-main fix that applies the min cap consistently in shared resolution, session persistence, and status summary while preserving current provider lookup semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by passing a positive `contextTokensOverride` larger than a known configured provider/model window into the shared resolver, session-store persistence, or status-summary path; current main returns or persists the override first.

## Solution Assessment

No, not yet. Capping in the shared resolver is the right direction, but this PR does not cover the session metadata and `/status` bypasses and should reuse the fuller current-main provider context-token lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Cap every reported override path:** `src/agents/context.ts:288-289`
  - body: The PR body calls out session metadata and `/status`, but this patch only caps the shared resolver. `session-store.ts` still persists `params.contextTokensOverride` before consulting the model window, and the status-summary runtime still has its own override-first resolver, so both reported paths can keep over-reporting after this change. Thread the override through capped resolution, or apply the same min logic in those callers, with regressions for both paths.
  - confidence: 0.9
- **[P2] Reuse the full provider context-token lookup:** `src/agents/context.ts:273-279`
  - body: The new config scan only reads `models[].contextWindow` through normalized provider matching. Current main also supports `contextTokens`, provider-level defaults, and exact-provider precedence before alias fallback, so after rebase this cap can miss or misread known limits unless it reuses the shared provider context-token lookup.
  - confidence: 0.84
