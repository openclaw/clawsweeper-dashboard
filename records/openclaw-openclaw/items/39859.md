## Summary

Keep open. The intended cap is valid, but this PR is stale against current resolver semantics and remains incomplete: it misses configured effective-cap behavior, bypasses newer provider-aware lookup safeguards, and does not fix the raw session/status/reply paths named by the PR body.

## What This Changes

This PR changes `resolveContextTokensForModel` and related tests to cap a positive context-token override by a configured model `contextWindow`, adds provider-alias coverage, adjusts one model-config test fixture, and increases one plugin loader test timeout.

## Best Possible Solution

Refresh or replace the branch with a narrow fix that resolves the known provider/model effective window first, applies configured or persisted `contextTokens` only as ceilings when that window is known, preserves unknown-model override behavior, and covers resolver, reply setup, session persistence, follow-up accounting, and `/status` display together.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.contextTokens` above a configured or discovered model window and follow the resolver, session-store, reply setup, or non-fallback `/status` paths; current main returns or stores the override before applying the smaller known limit.

## Solution Assessment

No, not as currently proposed. The `min(override, knownWindow)` direction is correct, but the branch should be rebuilt on current resolver helpers and should cover the raw persistence/status/reply callers instead of adding a partial resolver branch.
