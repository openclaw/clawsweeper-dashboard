## Summary

Keep open: current main still has the reported over-reporting path, and this PR is useful but incomplete because it caps only part of the resolver while missing current configured-cap semantics and raw status/session callers.

## What This Changes

This PR changes `resolveContextTokensForModel` and related tests to cap a positive context-token override by a configured model `contextWindow`, adds provider-alias coverage, adjusts one model-config fixture, and increases one plugin-loader test timeout.

## Best Possible Solution

Make configured context tokens act as ceilings across resolver, session metadata, reply setup, follow-up accounting, and `/status`, while preserving provider-qualified lookup safeguards and unknown-model override behavior.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.contextTokens` above a configured or discovered model window and follow the resolver, status, or session persistence paths; current main can return or store the larger override before applying the known smaller cap.

## Solution Assessment

No, not as proposed. The `min(override, knownWindow)` direction is right, but the patch should reuse current configured-provider resolution, preserve provider-aware lookup safeguards, and cover raw status/session/reply callers together.

## Review Findings
