## Summary

Keep open: the linked bug still exists on current main and this PR is the active implementation candidate, but the patch needs changes because the new cap ignores configured `contextTokens` effective caps, and the contributor proof is only tests/CI rather than real behavior output.

## What This Changes

The PR changes `resolveContextTokensForModel()` to cap a positive context override by a known model window, adds resolver regression tests, and adjusts related test fixtures.

## Best Possible Solution

Land a narrow resolver fix that treats overrides as caps when a selected model limit is known, using the same `contextTokens`/provider-default precedence as the existing config contract, with regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a positive override returns before any configured or discovered model window can cap it, so an override above a known model limit can still be persisted or reported.

## Solution Assessment

No, not yet. The direction is the right shared-resolver fix, but the implementation must honor configured `contextTokens` effective caps and provider-level caps before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor configured context token caps:** `src/agents/context.ts:277-279`
  - body: The new configured-window scan only reads `match.contextWindow`, but model config already treats `contextTokens` as the effective runtime cap and prefers it over `contextWindow`. With a model configured as `contextWindow: 1_050_000, contextTokens: 200_000` and an override of `1_048_000`, this branch returns the oversized override instead of the 200k effective cap, so the over-reporting bug remains for valid configs. Please mirror the existing `contextTokens ?? contextWindow ?? provider` precedence when computing `modelTokens`.
  - confidence: 0.88
