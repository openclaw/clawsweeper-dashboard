## Summary

Keep open: the underlying bug is still source-reproducible on current main, and this PR is a plausible but incomplete/conflicting fix that misses the session metadata and `/status` override paths, does not preserve the current provider context-token lookup semantics, and lacks external real behavior proof.

## What This Changes

The PR caps positive context-token overrides in `resolveContextTokensForModel`, adds resolver/model-config regression coverage, and raises one plugin-loader test timeout.

## Best Possible Solution

Rebase or replace this with a current-main patch that caps known effective context limits consistently in the shared resolver, session persistence, and status summary while preserving unknown-model override behavior and current provider lookup semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by passing a positive `contextTokensOverride` larger than a known configured provider/model window into the shared resolver, session-store persistence, or status-summary path; current main returns or persists the override first.

## Solution Assessment

No, not yet. Capping in the shared resolver is the right direction, but this PR misses the session metadata and `/status` bypasses and should reuse the fuller current-main provider context-token lookup.
