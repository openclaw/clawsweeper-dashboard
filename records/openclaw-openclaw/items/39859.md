## Summary

Keep open: the underlying context-window bug is still source-reproducible on current main, and this PR is the linked implementation candidate, but the branch is not merge-ready because it misses the reported session/status paths, uses stale configured-context lookup semantics, and lacks after-fix real behavior proof.

## What This Changes

The PR caps `resolveContextTokensForModel()` overrides by configured model context windows, adds resolver regression tests, adjusts a model-config fixture, and raises one legacy plugin-loader test timeout.

## Best Possible Solution

Rebase or replace this with a current-main fix that caps known effective context limits consistently in the shared resolver, session persistence, and status summary while preserving unknown-model override behavior and current provider lookup semantics.

## Reproduction Assessment

Yes. Source inspection shows that passing a positive `contextTokensOverride` above a lower configured provider/model window through the shared resolver, session-store persistence, or status-summary path can still return or persist the override first on current main.

## Solution Assessment

No. Capping in the shared resolver is the right direction, but this PR needs to reuse the current configured-context lookup and cover the session-store and status-summary bypasses named in the report.
