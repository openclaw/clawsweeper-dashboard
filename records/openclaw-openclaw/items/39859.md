## Summary

Keep open: the PR targets a real context-window over-reporting bug, but the branch is stale/conflicting and its resolver rewrite no longer preserves current main's provider-aware lookup semantics; it also lacks required after-fix real behavior proof.

## What This Changes

The PR changes context-token resolution and tests to cap configured overrides by a known model window, with small fixture and plugin-loader test-timeout adjustments.

## Best Possible Solution

Revise or replace the branch so configured overrides are applied as ceilings after the current provider-aware model-window resolver, session persistence writes the capped effective window, and /status remains consistent with focused regressions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set agents.defaults.contextTokens above a known configured or discovered model window, then follow resolveContextTokensForModel or session persistence where the larger override can still win before the smaller known limit caps it.

## Solution Assessment

No. The min(override, knownWindow) direction is right, but this PR should layer that cap over current main's provider-aware resolver and update raw session persistence rather than replacing lookup semantics with a narrower branch.
