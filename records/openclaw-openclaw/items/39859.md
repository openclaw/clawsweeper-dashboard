## Summary

Keep open: this PR targets a real, source-reproducible context-window bug linked from https://github.com/openclaw/openclaw/issues/39857, but it is not merge-ready because the branch only caps the shared resolver, misses current-main bypass paths, and needs real behavior proof.

## What This Changes

The PR changes context-token resolution to cap a positive override by a known model context window, normalizes provider aliases for configured model windows, adds resolver tests, and adjusts small test fixtures/timeouts.

## Best Possible Solution

Rework this on current main so the override cap is applied at the shared resolver and the persistence/status callers that bypass it, while preserving the current provider lookup order and posting redacted real /status or session-metadata proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: pass a positive contextTokensOverride larger than a known configured model window to current main's resolver or bypassing session/status paths, and the override is returned before the smaller known model limit is considered.

## Solution Assessment

No, not as-is. The direction is right, but the proposed patch needs to preserve current-main provider/model lookup semantics and cover the session/status paths that still read the raw override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Route bypassing callers through the cap:** `src/agents/context.ts:292`
  - body: The new Math.min only runs when callers use resolveContextTokensForModel, but session persistence and the status-summary runtime still take a positive contextTokensOverride before consulting model-aware resolution. With a 1,048,000 override and a known 900,000 model window, those paths can still store or report 1,048,000, so route those callers through the capped resolution too.
  - confidence: 0.88
- **[P2] Preserve provider-specific lookup ordering:** `src/agents/context.ts:278-287`
  - body: The new modelTokens path uses findNormalizedProviderValue and then always tries `${ref.provider}/${ref.model}`. Current main intentionally uses exact-provider-before-alias matching, provider-level context caps, and guarded provider-qualified cache lookup so model-only slash IDs do not pick up another provider's window. Keep that helper and lookup order when adding the override cap.
  - confidence: 0.86
