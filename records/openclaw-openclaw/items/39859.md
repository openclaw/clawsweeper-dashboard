## Summary

Keep open: this PR is the linked implementation candidate for https://github.com/openclaw/openclaw/issues/39857, and current main still has override-first context-token paths, but the branch needs revision against the current provider-aware resolver and still lacks real behavior proof.

## What This Changes

The PR changes context-token resolution and tests to cap a configured override by a known model window, with small fixture and plugin-loader test-timeout adjustments.

## Best Possible Solution

Revise against current main by applying configured overrides as ceilings after the existing provider-aware model-window resolution, then cover session persistence and status/reporting callers with focused regressions while coordinating any broader token-limit work with https://github.com/openclaw/openclaw/issues/71136.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: configure agents.defaults.contextTokens above a configured or discovered model window, then follow resolveContextTokensForModel or session persistence where the larger override is returned before the known smaller window can cap it.

## Solution Assessment

No. The min(override, knownWindow) direction is right, but this PR should layer that cap over current main's configured-provider resolver and update remaining raw session/status override paths instead of replacing the resolver with narrower lookup logic.
