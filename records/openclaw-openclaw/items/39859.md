## Summary

Keep open: current main still has the override-first resolver path, and this PR's implementation is stale/incomplete against the newer provider-aware context resolution and status/session paths.

## What This Changes

The branch changes context-token resolution and tests to cap configured overrides by model windows, adjusts a model-config fixture, and increases one plugin-loader test timeout.

## Best Possible Solution

Apply the override as a ceiling after resolving the current main effective model window, then align status, session persistence, reply setup, and follow-up accounting with focused regression coverage.

## Reproduction Assessment

Yes. By source inspection, configure agents.defaults.contextTokens above a configured or discovered model window and follow resolveContextTokensForModel or its status/session callers; current main can still return the larger override before applying the known cap.

## Solution Assessment

No. The min(override, knownWindow) direction is right, but this PR should layer that cap on top of the current configured-provider resolver and update caller paths that can still use raw configured context tokens.
