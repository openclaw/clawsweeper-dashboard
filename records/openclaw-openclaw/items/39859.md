## Summary

Keep open. The bug remains valid, but this PR is stale/conflicting, lacks real behavior proof, and its resolver change bypasses current main's provider-aware context-window safeguards while leaving session/status override paths incomplete.

## What This Changes

This PR changes context-token resolution and tests to cap a configured override by a known model window, with small fixture and plugin-loader test-timeout adjustments.

## Best Possible Solution

Revise against current main by applying configured overrides as ceilings after the existing provider-aware model-window resolution, then cover resolver, session persistence, and /status callers with focused regressions while coordinating broader token-limit work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure agents.defaults.contextTokens above a known provider/model window, then follow resolveContextTokensForModel or session persistence where the raw override can still win first.

## Solution Assessment

No. The min(override, knownWindow) direction is right, but this PR should layer that cap over current main's configured-provider resolver and update remaining raw session/status override paths instead of replacing the resolver with narrower lookup logic.
