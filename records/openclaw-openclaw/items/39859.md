## Summary

Keep open. Current main at 7969f1f07ccc0d3526dc97c4975c1184b2c147d7 still returns positive contextTokensOverride values before resolving known model context windows, so the linked over-reporting bug remains. The PR has the right resolver-level direction, but it is stale against newer current-main resolver semantics, leaves at least one session metadata path able to persist the raw override, and carries unrelated security-sensitive baseline/CI churn that should not be merged with this product fix.

## What This Changes

The PR changes context-window resolution and tests to cap a configured override by a known model window with provider-alias coverage, plus small model-fixture/test-timeout updates and unrelated CI/secrets-baseline follow-ups.

## Best Possible Solution

Refresh or replace this PR with a narrow current-main fix that resolves the known model window before honoring a positive override, returns min(override, knownWindow) when both are known, preserves override behavior when the model window is unknown, and applies the same effective value to status, runtime/session accounting, and session-store persistence without carrying unrelated CI or baseline changes.

Yes. No. The proposed min(override, modelWindow) direction is correct, but the branch is not the best merge shape as-is because it bypasses current configured-context helper semantics, leaves direct session-store persistence able to over-report, and mixes unrelated security-baseline/workflow changes into the fix.
