## Summary

Keep open. Current main still has the override-first resolver and raw override persistence/display paths, while this PR has the right direction but is stale against newer resolver semantics and does not cover all session/status paths named in the report.

## What This Changes

The PR changes context-window resolution and tests so a positive context-token override is capped by a known configured model window, adds provider-alias coverage, adjusts one model-config fixture, and increases one plugin loader test timeout.

## Best Possible Solution

Refresh or replace the branch with a narrow current-main fix that resolves the effective known provider/model window first, treats configured or persisted contextTokens as ceilings when that window is known, preserves override behavior for unknown model windows, and updates resolver, reply setup, session persistence, follow-up/runtime accounting, and /status coverage together.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure agents.defaults.contextTokens above a configured or discovered model window and follow the current resolver, session-store, reply setup, or /status paths; current main returns or stores the override before applying the smaller known limit.

## Solution Assessment

No, not as currently proposed. The min(override, knownWindow) direction is correct, but the PR should be refreshed around current resolver helpers and should cover the raw session/status/reply callers instead of only adding a partial resolver branch.
