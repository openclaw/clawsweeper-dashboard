## What This Changes

The PR changes effective session context-token resolution so known model windows cap larger user overrides, with resolver regression tests plus small fixture and test-timeout maintenance.

## Best Possible Solution

Refresh this PR on current main so the shared resolver and the remaining session-store/status-summary callers all cap effective windows by known model limits while preserving current provider alias, contextTokens, and provider-default semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: on current main, a positive contextTokensOverride larger than a known configured model window is returned before the resolver checks model limits, and session-store/status-summary have separate override-first branches.

## Solution Assessment

No. The PR is aimed at the right shared behavior, but it is not yet the best current-main fix because it misses session-store/status-summary override-first paths and needs to preserve current contextTokens/provider-default lookup semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route session overrides through the model-aware cap:** `src/agents/command/session-store.ts:100-102`
  - body: `updateSessionStoreAfterAgentRun` still persists `params.contextTokensOverride` before it calls the resolver. A run with `agents.defaults.contextTokens` above the known model window can therefore keep writing the inflated value into session metadata even after this PR's resolver change; pass the override through model-aware resolution instead of returning it directly.
  - confidence: 0.9
- **[P2] Apply the cap in status summary resolution:** `src/commands/status.summary.runtime.ts:206-207`
  - body: The status summary runtime has its own resolver that returns `contextTokensOverride` before checking configured provider/model caps. Since `status.summary.ts` passes defaults and session entries through that override slot, the summary/status JSON path can still over-report the effective window unless this local resolver gets the same cap behavior.
  - confidence: 0.87
- **[P2] Preserve configured contextTokens precedence:** `src/agents/context.ts:277-279`
  - body: The PR head's new configured lookup reads only `match.contextWindow`. Current config supports `contextTokens` as the effective runtime cap, and current main already has helper logic that prefers `contextTokens` over `contextWindow`; refresh this branch to reuse that semantics so oversized overrides are capped by configured effective limits too.
  - confidence: 0.83
