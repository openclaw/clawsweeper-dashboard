## What This Changes

The PR rediscovers the current xAI token endpoint when a persisted OAuth credential still points at the retired `https://auth.x.ai/oauth/token`, then adds a regression test for that refresh path.

## Best Possible Solution

Land the xAI plugin-owned refresh fix after adding redacted terminal/live output or a maintainer override showing a stale persisted endpoint is rediscovered and normalized.

## Reproduction Assessment

Yes for the source-level and dependency-contract path: current main posts refresh to the cached endpoint, and live xAI probes show the retired endpoint returns HTML while discovery points to `/oauth2/token`. A full real stale-token OpenClaw refresh was not run in this review.

## Solution Assessment

Yes, this is the narrowest maintainable fix shape I found: keep the behavior inside the xAI plugin, only special-case the known retired endpoint, and keep all other cached endpoints untouched. A doctor migration alone would not help users who hit refresh before running doctor.
