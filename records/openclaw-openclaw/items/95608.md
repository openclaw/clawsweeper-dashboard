- `merge-risk: 🚨 security-boundary`: Mis-associated caller-supplied telemetry identifiers can be forwarded to trusted exporters and trace storage under the wrong run.

## Summary

Keep open: the feature is useful and the happy-path proof is now sufficient, but the branch still stores caller attribution in mutable session diagnostic state, so a later same-session request can overwrite or clear the active run's trusted telemetry context before that run finishes.

## What This Changes

The PR adds optional gateway/SDK/Swift `clientContext`, normalizes it, stores it in diagnostic session state, and forwards it to trusted diagnostics private data on lifecycle events.

## Best Possible Solution

Make attribution run-owned, preferably by carrying the normalized bag with the admitted run or indexing by `runId`, then add same-session overlap coverage and get maintainer approval for the telemetry contract.

## Reproduction Assessment

Yes from source: two differently attributed gateway requests for one session can mutate the same diagnostic session `clientContext` before the earlier run finishes emitting lifecycle events.

## Solution Assessment

No: the trusted private-data channel is a plausible boundary, but the implementation should bind attribution to the admitted run rather than to a reusable session record.
