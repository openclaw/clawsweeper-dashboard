## Summary

Keep open: the latest PR head still stores caller-supplied attribution on mutable diagnostic session state, so same-session overlap can mis-associate trusted telemetry, and the new gateway/SDK/Swift/plugin private-data contract still needs maintainer approval.

## What This Changes

The PR adds optional `clientContext` input to gateway agent runs, forwards it through the TypeScript SDK and Swift model, normalizes it, and delivers it to trusted diagnostic listeners as private data on lifecycle events.

## Best Possible Solution

Make attribution run-owned, preferably by carrying the normalized bag with the admitted run or indexing it by `runId`, then land only after the telemetry API and privacy contract are maintainer-approved.

## Reproduction Assessment

Yes, from source: the gateway accepts and yields before async dispatch, while latest head writes `clientContext` into one mutable diagnostic session slot that later lifecycle events read. Two differently attributed same-session requests can therefore overwrite or clear each other's trusted attribution before the earlier run finishes emitting events.

## Solution Assessment

No: the trusted private-data channel is a plausible boundary, but this implementation should bind attribution to the admitted run rather than reusable session state. The safer shape is run-owned or `runId`-keyed attribution plus focused overlap coverage.
