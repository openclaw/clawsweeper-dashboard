## Label Justifications

- `P2`: This is a normal-priority diagnostics/gateway feature PR with blocking correctness and compatibility concerns, not an active production outage.
- `merge-risk: 🚨 compatibility`: The diff expands gateway protocol, SDK, Swift generated model, and plugin-visible diagnostic contracts that downstream clients may rely on.
- `merge-risk: 🚨 session-state`: The implementation stores caller attribution on mutable diagnostic session state, which can mis-associate data across same-session runs.
- `merge-risk: 🚨 security-boundary`: Caller-supplied attribution can be exposed to trusted diagnostics consumers and may be attached to the wrong run if session state is overwritten.

## Summary

Keep this PR open, but the current patch is not merge-ready: the new attribution bag is still written to mutable session-scoped diagnostic state, so a later same-session request can overwrite or clear the active run's trusted attribution before lifecycle events emit.

## What This Changes

The branch adds optional gateway/SDK `clientContext`, normalizes it, stores it on diagnostic session state, forwards it to trusted diagnostic private data on lifecycle events, updates Swift protocol models, and adds focused tests plus generated API hashes.

## Best Possible Solution

Carry the normalized context as run-owned trusted private data, preferably through the admitted run or a `runId`-keyed store, preserve Swift/source compatibility, and merge only after gateway/SDK/plugin-diagnostics contract approval.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR writes caller context to session-scoped diagnostic state before dispatch, while same-session run serialization happens later in the embedded runner lane. This is feature work rather than a current-main bug reproduction.

## Solution Assessment

No; the trusted private-data direction is plausible, but the current session-scoped storage is not the best fix for per-run attribution. The maintainable shape is a run-owned carrier or `runId` lookup, plus explicit compatibility handling for generated Swift callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind clientContext to the run before forwarding it:** `src/gateway/server-methods/agent.ts:2735-2738`
  - body: This writes the caller bag into one mutable diagnostic session record before the run is admitted to the embedded runner's same-session lane. A second request for the same session can reach this line while the first run is active or queued, replacing or clearing the value that later `session.state` / `message.queued` emits will forward as trusted private data. Carry the normalized context with the admitted run, or key it by `runId`, and cover two differently attributed same-session requests.
  - confidence: 0.94
- **[P2] Preserve Swift AgentParams source compatibility:** `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:836`
  - body: The new optional wire field is added as a non-defaulted trailing initializer parameter, so existing OpenClawKit source that constructs `AgentParams` without `clientcontext:` will stop compiling. If this model is public source API, add a defaulted parameter/compatibility overload or update the generator policy before treating the gateway field as additive.
  - confidence: 0.82
