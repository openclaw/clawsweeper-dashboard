## Summary

Keep open: latest head still has a P1 run-scoping and telemetry-boundary blocker because caller attribution is stored on mutable diagnostic session state, while the durable gateway/SDK/plugin private-data contract still needs maintainer approval.

## What This Changes

The PR adds optional gateway, SDK, and Swift `clientContext` input and forwards normalized attribution metadata to trusted diagnostic private data on lifecycle events.

## Best Possible Solution

Make attribution run-owned, preferably by carrying the normalized bag with the admitted run or indexing it by `runId`, then add same-session overlap coverage and land only after the telemetry contract is maintainer-approved.

## Reproduction Assessment

Yes, from source: the gateway accepts and yields before async dispatch, while latest head writes `clientContext` into one mutable diagnostic session slot that later lifecycle events read. Two differently attributed same-session requests can therefore overwrite or clear each other's trusted attribution before the earlier run finishes emitting events.

## Solution Assessment

No: the trusted private-data channel is a plausible boundary, but the implementation should bind attribution to the admitted run rather than to reusable session state. The safer shape is run-owned or `runId`-keyed attribution plus focused overlap coverage.
