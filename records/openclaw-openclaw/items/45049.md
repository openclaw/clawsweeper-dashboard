## Summary

Keep open: current main has partial safeguards, but the central embedded-agent protocol-boundary decision remains unresolved. Gateway HTTP `tool_choice` enforcement is present on main, while the embedded runner still promotes some standalone text-form tool calls into executable tool calls and only logs other shaped text as unsupported text.

## What This Changes

The issue asks OpenClaw to prevent model-written fake tool invocations in assistant text from being accepted as successful tool use and to enforce real structured tool-call boundaries in the agent loop.

## Best Possible Solution

Make an explicit maintainer/security decision for the embedded agent protocol boundary: remove text-form tool promotion, or gate it behind a clearly named audited compatibility mode while keeping structured tool calls as the only default executable path.

## Reproduction Assessment

Yes from source, though I did not run a live LiteLLM/vLLM reproduction in this read-only pass. Current main still wraps embedded streams with standalone text-tool-call promotion, and current tests explicitly expect that promotion behavior.

## Solution Assessment

No, the current partial solution is not the full fix. Gateway `tool_choice` enforcement is useful for OpenAI-compatible HTTP client tools, but the embedded runner still needs a maintainer-approved strict boundary or explicit compatibility design.
