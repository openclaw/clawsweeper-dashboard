## Label Justifications

- `impact:security`: The report concerns preserving the structured tool-call boundary used by policy hooks, approvals, and provenance.
- `impact:message-loss`: Users can receive normal-looking assistant output while the intended tool-backed action was skipped or fabricated.
- `impact:session-state`: Assistant text can diverge from the turn ledger when tool invocation details are recorded only as text instead of structured toolCall/toolResult entries.

## Summary

Keep open. Current main and the latest release still only warn when assistant text looks like a tool call; the agent reply path then continues delivering that text, so the strict protocol-boundary behavior remains unresolved and needs maintainer security/product direction.

## What This Changes

The issue requests agent-loop enforcement so models cannot satisfy tool requests by writing simulated tool calls or tool results in assistant text instead of emitting structured tool_calls.

## Best Possible Solution

Define a core protocol-boundary policy for suspicious assistant text that blocks or retries only through structured tool_calls, with documented default versus opt-in behavior for local providers and tests proving hook and ledger preservation.

## Reproduction Assessment

Yes, at source level. Current main detects tool-call-shaped assistant text and logs a warning, then continues visible assistant-text delivery without a strict block, retry, or forced tool-choice policy.

## Solution Assessment

Unclear. Enforcing the structured tool-call boundary is the right direction, but maintainers still need to choose the default behavior, retry/routing semantics, and local-provider compatibility policy.
