## Label Justifications

- `P1`: The issue affects real agent/tool workflows and has security/session-integrity consequences, but it still needs a bounded maintainer decision rather than emergency handling.
- `impact:security`: The report and comments describe tool protocol bypass risk around `before_tool_call`, policy checks, provenance, and fabricated tool results.
- `impact:session-state`: The reported failure creates transcript and tool-ledger drift where assistant text claims execution that the session did not actually record.
- `impact:message-loss`: When agents claim tool or channel actions happened without a real invocation, expected outbound work can be silently absent while the user sees a completed reply.

## Summary

Keep open: current main has partial safeguards, but it does not fully enforce the requested agent-loop protocol boundary. Gateway HTTP `tool_choice` enforcement is merged, while the embedded runner still promotes some standalone text-form tool calls and only warns for other shaped text, so the central reliability/security issue remains.

## What This Changes

The issue asks OpenClaw to require real structured tool invocations instead of accepting or delivering simulated text-form tool calls and fabricated tool-result claims.

## Best Possible Solution

Settle the core protocol-boundary policy, then align the embedded runner, provider compatibility wrappers, diagnostics, docs, and regression tests so simulated tool text cannot masquerade as verified execution.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main wraps the embedded agent stream with standalone text-call promotion, tests expect that promotion, and docs describe warning-only handling for other shaped text.

## Solution Assessment

No, current main is only a partial solution. Gateway `tool_choice` enforcement and diagnostics help, but the best fix needs a consistent agent-runtime protocol-boundary decision that accounts for local-model compatibility.
