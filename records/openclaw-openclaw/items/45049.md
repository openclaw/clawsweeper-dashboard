## Label Justifications

- `P1`: The issue affects real agent/tool workflows by allowing text-only completion where a structured tool call was required, with user-visible false action confidence.
- `impact:security`: The requested fix concerns the protocol boundary for tool execution, policy hooks, and provenance rather than trusting model-written text.
- `impact:session-state`: The transcript can end with assistant text claiming or implying tool activity while the session ledger has no matching tool call or tool result.
- `impact:message-loss`: When the required action is a tool-backed message or external action, the turn can end with no real tool invocation or delivered action despite user-facing text.

## Summary

Keep open: current main still has warning-only handling for fake tool-call-shaped assistant text in ordinary agent turns. The merged Gateway HTTP tool_choice work is useful, but it only covers caller-supplied HTTP client tools and does not implement the reported agent-loop enforcement behavior.

## What This Changes

The issue requests stricter agent-loop handling when models describe or fabricate tool use in assistant text instead of emitting structured tool calls.

## Best Possible Solution

Keep this issue open as the canonical narrow tool-call-boundary bug and design a scoped, security-reviewed enforcement path that never executes assistant text as tools, but can fail or retry when an explicit required-tool contract exists.

## Reproduction Assessment

Yes, at source level: current main executes only structured toolCall blocks and warns-but-delivers tool-shaped assistant text without forcing a retry or blocking final text. I did not live-run the reporter's LiteLLM/vLLM setup in this read-only review.

## Solution Assessment

No complete solution exists on main. The best fix is not to parse assistant text into tools, but to add a scoped protocol-boundary gate or retry path for explicit required-tool contracts while preserving compatibility for ordinary text turns.
