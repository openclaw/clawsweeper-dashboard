## Label Justifications

- `P1`: The issue affects real agent/tool workflows and has current source-level evidence plus user reports of apparent action completion without actual tool invocation.
- `impact:security`: The report concerns preserving the structured tool-call boundary that policy hooks, approvals, and provenance rely on.

- `impact:session-state`: Assistant text can diverge from the turn ledger and transcript state when a fabricated tool invocation is recorded only as text.

## Summary

Keep open. Current main still treats simulated tool-call-shaped assistant text as assistant output after a diagnostic warning, and the issue has protected security/maintainer/product-review labels plus fresh source-backed reports, so it needs maintainer policy work rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to enforce real structured tool invocation when models emit simulated tool calls or action claims as text.

## Best Possible Solution

Define a protocol-boundary enforcement policy for suspicious assistant text, with any retry/routing path entering the existing structured tool hook and approval pipeline and with documented default versus opt-in behavior for local providers.

## Reproduction Assessment

Yes, at source level. Current main detects tool-call-shaped assistant text and logs a warning, then continues through the normal assistant-text delivery path without a strict block or retry.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but the exact default, retry, routing, and provider-compatibility behavior needs maintainer policy before implementation.
