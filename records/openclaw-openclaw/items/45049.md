## Summary

Keep open. Current main has useful diagnostics, visible-text cleanup, narrow standalone text-tool promotion, and merged Gateway `tool_choice` enforcement, but the mixed prose/JSON fake-call path is still treated as text rather than a maintainer-approved hard enforcement or retry policy.

## What This Changes

The issue asks OpenClaw to enforce real structured tool invocations when a model emits text that simulates a tool call.

## Best Possible Solution

Keep this as the canonical strict tool-call boundary item: maintainers should define a generic policy for fake-call text, forced tool-choice retry, and response gating while using existing per-model `extra_body.tool_choice` overrides for model-specific workarounds.

## Reproduction Assessment

Yes, at source level: current tests construct a text-only JSON tool-call-looking assistant reply and expect a warning that it is treated as text. I did not run the original LiteLLM/vLLM stack live in this read-only review.

## Solution Assessment

Unclear for the exact implementation: warning and per-model forced `tool_choice` are only partial mitigations, while automatic routing or fail-closed retry needs maintainer-approved protocol and compatibility semantics. A generic enforcement boundary is safer than one-off JSON parsing or deterministic URL-to-tool rules.
