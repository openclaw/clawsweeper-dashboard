## Summary

Keep open. Current main still treats simulated tool-call text as assistant text after logging a warning; the shipped mitigations improve diagnostics and local-provider defaults, but they do not implement the requested generic protocol-boundary enforcement, forced retry, or deterministic routing for LiteLLM/vLLM-style local model failures.

## What This Changes

The issue asks OpenClaw to prevent models from satisfying tool-required tasks with plain text that resembles or claims a tool call, especially for local OpenAI-compatible providers.

## Best Possible Solution

Define a maintainer-approved strict protocol policy where executable tool activity originates from structured tool events, simulated tool-call text is blocked or surfaced under an explicit mode, provider-scoped forced retries are narrowly configured, and any provider-local text-promotion path enters the normal hook, approval, provenance, docs, and test surfaces.

## Reproduction Assessment

Yes. A source-level reproduction is an assistant message containing JSON/XML/ReAct-style tool-call text and no structured tool block: current `handleMessageEnd` logs the diagnostic and then proceeds with normal reply emission. Fresh issue discussion also adds a concrete vLLM/Hermes run with no ledgered tool calls.

## Solution Assessment

No. Current main's warning, local-model docs, `tool_choice: "auto"` default, per-model `required` workaround, strict-agentic GPT-5 lane, and LM Studio exception are useful partial mitigations, but they do not provide the requested generic protocol boundary for local OpenAI-compatible providers.
