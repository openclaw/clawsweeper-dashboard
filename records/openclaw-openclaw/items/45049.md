## Summary

Keep open. Current main still treats tool-call-shaped assistant text as a diagnostic warning rather than a strict protocol-boundary failure, while the requested retry/routing/enforcement behavior remains security-sensitive policy work with an open companion PR and related reports.

## What This Changes

The issue requests stricter agent-loop handling when models emit simulated tool invocations or success claims as assistant text instead of structured tool calls.

## Best Possible Solution

Define a strict structured-tool boundary for suspicious assistant text, decide the provider/config defaults for retries and deterministic routing, and ensure any enforced call still enters the normal hook, approval, and provenance pipeline.

## Reproduction Assessment

Yes, at source level: an assistant message containing registered tool-call-shaped text without structured tool content reaches the message-end handler, emits a warning, and continues as assistant text. I did not live-run the reporter's LiteLLM/vLLM setup in this read-only review.

## Solution Assessment

Unclear. Strict protocol enforcement is the right security boundary, but forced retries, deterministic routing, and default strictness need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
