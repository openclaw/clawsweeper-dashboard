## Label Justifications

- `P1`: Current main confirms that malformed tool-call-shaped output can still be delivered as text while a requested action is skipped in real agent workflows.
- `impact:session-state`: Fabricated invocation and result text can enter the transcript without corresponding structured tool ledger events.
- `impact:security`: Tool policy hooks, approvals, and provenance checks only protect invocations that enter the structured execution pipeline.
- `impact:message-loss`: A required fetch, search, exec, or outbound action can be omitted while the user receives a misleading normal response.

## Summary

Keep open: current main detects tool-call-shaped assistant text but explicitly treats it as ordinary text, and the merged Gateway enforcement only applies when an API client already requests `tool_choice`; the embedded agent runtime still lacks a maintainer-approved contract for when to require, retry, or fail a missing structured invocation. The narrower Telegram report and broader pre-response-gate proposal are related but do not fully replace this provider-agnostic agent-loop report.

Likely related people
- @vincentkoc — high confidence; introduced the current diagnostic and has the strongest recent history on the central embedded-agent path.
- @steipete — medium confidence; substantial prior feature history across the embedded message handler and tool-call execution boundary.

## What This Changes

Request structural handling when a model emits tool-call-like prose instead of a real structured tool invocation.

## Best Possible Solution

Define one runtime-owned required-tool or pre-response gate that validates actual structured tool-call and result ledger events, uses bounded retries or a deterministic failure outcome, and leaves ordinary turns unchanged unless an explicit policy requires execution.

## Reproduction Assessment

Yes, at high source confidence. Current main contains the exact branch for tool-call-shaped assistant text with no structured invocation and deliberately treats it as text; this review did not rerun the reporter's external LiteLLM/vLLM stack.

## Solution Assessment

No, the suggested automatic URL/search routing and generic prose-pattern retry are not the best general fix. The maintainable solution is an explicit required-execution contract checked against structured turn state, with maintainer-defined activation and failure semantics.
