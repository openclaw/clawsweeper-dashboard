## Summary

Keep open. Current main has diagnostics, local-model documentation, a per-model `tool_choice: "required"` workaround, a narrow strict-agentic path for OpenAI/Codex GPT-5-family runs, and an LM Studio compatibility promotion path, but it still lacks the generic protocol-boundary enforcement, provider-scoped forced retry behavior, or deterministic routing requested for LiteLLM/vLLM local-model tool-selection failures.

## What This Changes

The issue requests stricter agent-loop handling so assistant text that describes or mimics tool use is not accepted as task progress when a real structured tool call is required.

## Best Possible Solution

Define a maintainer-approved strict protocol policy where executable tool activity originates from structured tool events, simulated tool-call text is surfaced or blocked according to an explicit mode, forced retries are provider/model scoped, and any provider-local text-promotion path still enters the normal hook, approval, provenance, docs, and test surfaces.

## Reproduction Assessment

Yes. The original LiteLLM/vLLM stack was not live-run in this read-only review, but current main has a high-confidence source-level reproduction: pass an assistant message containing JSON/XML/ReAct-style tool-call text and no structured tool block into the embedded message-end path; it logs the warning and proceeds as a normal reply.

## Solution Assessment

No. Current main's warning, docs workaround, OpenAI/Codex GPT-5 strict-agentic lane, and LM Studio exception are useful partial mitigations, but they are not the requested generic strict boundary for local OpenAI-compatible providers.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the remaining fix changes the trust boundary between assistant text, structured tool calls, policy hooks, approvals, and user-visible completion claims.

Concerns:

- **[medium] Decide strict text-to-tool boundary semantics:** `src/agents/pi-embedded-subscribe.tool-text-diagnostics.ts:75`
  - body: Current main logs tool-call-shaped assistant text and treats it as a reply. Any future block, retry, or forced-tool path needs an explicit policy so assistant prose cannot be mistaken for real execution or progress.
  - confidence: 0.9
- **[medium] Preserve hook and approval guarantees:** `src/agents/pi-tool-definition-adapter.ts:233`
  - body: `before_tool_call` runs inside the structured tool adapter before execution. Deterministic routing or text-promotion must enter this adapter path rather than bypassing plugin policy, approvals, and provenance.
  - confidence: 0.88
- **[medium] Reconcile LM Studio text promotion:** `extensions/lmstudio/src/stream.ts:159`
  - body: LM Studio currently promotes a narrow bracketed text format into structured tool-call blocks for registered tools. A strict protocol policy should decide whether that provider-local exception remains allowed and under which constraints.
  - confidence: 0.82
- **[medium] Scope forced tool choice carefully:** `docs/gateway/local-models.md:239`
  - body: Docs already warn that `tool_choice: "required"` should only be used where every normal turn should call a tool. A generic forced retry could cause unintended tool calls if applied broadly.
  - confidence: 0.8
