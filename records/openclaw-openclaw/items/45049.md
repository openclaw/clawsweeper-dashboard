## Label Justifications

- `P1`: The issue affects real agent/tool workflows and current source still allows warning-only delivery after simulated tool-call text is detected.
- `impact:security`: The report concerns preserving the structured tool-call boundary used by policy hooks, approvals, provenance, and auditability.
- `impact:message-loss`: Users can receive normal-looking assistant output while the intended tool-backed action was skipped or fabricated.
- `impact:session-state`: Assistant text can diverge from the turn ledger when tool invocation details exist only as text instead of structured toolCall/toolResult entries.

## Summary

Keep open. Current main and v2026.5.22 still detect assistant text that looks like a tool call only as a warning, then continue through the normal assistant reply path instead of blocking, retrying, or requiring a structured tool_calls invocation.

## What This Changes

The issue asks OpenClaw to stop accepting assistant text that imitates tool calls and enforce or retry tool use only through structured tool_calls.

## Best Possible Solution

Define a protocol-boundary policy that blocks or retries suspicious assistant text only through structured tool_calls, with documented default versus opt-in behavior and tests for hook and ledger preservation.

## Reproduction Assessment

Yes, at source level. Current main detects tool-call-shaped assistant text and logs a warning, then continues visible assistant-text processing without a strict block, retry, or forced tool-choice policy.

## Solution Assessment

Unclear. Enforcing the structured tool-call boundary is the right direction, but maintainers still need to choose default behavior, retry/routing semantics, and local-provider compatibility policy.
