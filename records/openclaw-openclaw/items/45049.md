## Summary

Current main still handles generic JSON/XML/ReAct-style simulated tool calls as warning-only assistant text, so this security-relevant issue should stay open for maintainer policy and implementation work.

## What This Changes

The issue requests agent-loop safeguards so models cannot satisfy tool-use requests by emitting simulated tool calls or completion claims as plain assistant text.

## Best Possible Solution

Define a maintainer-approved strict protocol policy where executable tool activity originates from structured tool events, suspicious tool-call text is blocked or surfaced explicitly, provider exceptions are documented, and retry/routing paths preserve hook, approval, provenance, docs, and tests.

## Reproduction Assessment

Yes, at source level: an assistant message containing JSON/XML/ReAct-style tool-call text and no structured tool block reaches `handleMessageEnd`, which logs a diagnostic and continues normal reply handling. I did not run a live vLLM/LiteLLM reproduction in this read-only review.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but generic deterministic routing and forced-tool retries need maintainer-scoped policy so they do not bypass hooks or force unsafe tool calls.
