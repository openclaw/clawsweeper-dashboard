## Label Justifications

- `P1`: The report affects real agent/tool workflows and has fresh evidence of users receiving apparent task completion without real tool invocation.
- `impact:security`: The issue is about preserving the tool-call protocol boundary so policy hooks and approvals are not bypassed by simulated tool text.
- `impact:session-state`: Assistant output can diverge from the structured turn ledger, creating transcript and state drift.
- `impact:message-loss`: Tool-backed actions can be skipped while the user still receives a normal-looking reply.

## Summary

Keep open. Current main has diagnostic coverage for simulated tool-call-shaped assistant text, but it still leaves that text on the normal assistant-output path and has not made the requested strict enforcement, retry, or routing policy decision.

## What This Changes

The issue asks OpenClaw to prevent agents from presenting simulated or prose-only tool usage as successful work when no real structured tool invocation occurred.

## Best Possible Solution

Define a strict structured-tool boundary for suspicious assistant text, choose default versus opt-in behavior for local providers, and ensure any retry or routing path enters the existing hook, approval, and provenance pipeline.

## Reproduction Assessment

Yes, at source level. Current main routes tool-call-shaped assistant text through a diagnostic warning and then continues the normal assistant-text path; I did not live-run the reporter's LiteLLM/vLLM setup.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but forced retries and deterministic routing need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
