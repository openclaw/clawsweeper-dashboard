## Label Justifications

- `P1`: This is an urgent agent reliability/security boundary issue where users can receive or act on fabricated tool execution instead of real tool results.
- `impact:security`: The issue concerns whether tool execution and policy hooks are reached only through the structured tool-call protocol boundary.
- `impact:message-loss`: When the model emits prose or simulated tool output instead of an invocation, the intended tool-backed response can be silently replaced by an ordinary assistant message.
- `impact:session-state`: Fabricated tool-call text or results can enter the transcript as session state without a corresponding real tool-call ledger entry.

## Summary

Keep open: current main and the latest release contain useful partial fixes for OpenAI-compatible Gateway `tool_choice`, but the core/provider runtime still has intentional plain-text tool-call promotion paths, so the strict protocol-boundary request is not fully implemented and needs maintainer security/product review.

## What This Changes

The issue asks OpenClaw to enforce real structured tool invocation when models emit simulated tool-call text instead of OpenAI-style `tool_calls`.

## Best Possible Solution

Define a maintainer-approved strict protocol boundary that separates provider-native tool calls from any optional local-model repair, then update the agent/provider/gateway paths and tests to match that policy.

## Reproduction Assessment

Yes, at source level: current tests and provider wrappers still show standalone text tool-call promotion into executable `toolCall` blocks. I did not run the reported LiteLLM/vLLM live stack, and the exact JSON example is now covered by diagnostics rather than automatic retry.

## Solution Assessment

Unclear as an implementation path: current main has a good Gateway `tool_choice` partial fix, but it is not the complete agent-loop/protocol-boundary policy this issue requests. The best fix needs maintainer agreement on strict enforcement versus local-model compatibility repair.
