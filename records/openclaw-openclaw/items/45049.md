## Label Justifications

- `P1`: The issue affects real agent/tool workflows and has current source-level evidence plus fresh reports of normal-looking replies without backed tool invocation.
- `impact:security`: The report concerns preserving the structured tool-call boundary that policy hooks, approvals, and provenance rely on.
- `impact:message-loss`: Users can receive normal-looking assistant output while the intended tool-backed action was skipped.
- `impact:session-state`: Assistant text can diverge from the turn ledger and transcript state when fabricated tool invocation details are recorded only as text.

## Summary

Keep open. Current main detects tool-call-shaped assistant text and logs it, but still delivers it as assistant text; deciding whether to block, retry, or force structured tool routing remains a security-sensitive product decision.

## What This Changes

The issue asks OpenClaw to prevent assistant text that simulates tool calls and to require or recover real structured tool invocations for local-model tool requests.

## Best Possible Solution

Define a protocol-boundary policy for suspicious assistant text, with any retry or routing path using structured tool calls and a documented default versus opt-in behavior for local providers.

## Reproduction Assessment

Yes, at source level. Current main detects tool-call-shaped assistant text and logs a warning, then continues visible assistant-text delivery without a strict block, retry, or forced tool-choice policy.

## Solution Assessment

Unclear. A structured tool-call boundary is the right security direction, but the exact default, retry/routing behavior, and local-provider compatibility policy need maintainer approval before implementation.
