## Summary

Keep open. Current main still treats tool-call-shaped assistant text as a warning-only diagnostic and then continues normal assistant-text delivery; the remaining enforcement, retry, and routing behavior is security-sensitive policy work rather than a safe autonomous cleanup close.

## What This Changes

The issue asks OpenClaw to enforce real structured tool calls instead of accepting tool-call-shaped assistant text or unbacked success claims as normal output.

## Best Possible Solution

Define a strict structured-tool boundary for suspicious assistant text, decide the default/provider/config policy for retries or routing, and ensure every enforced call still enters the normal hook, approval, and provenance pipeline.

## Reproduction Assessment

Yes, at source level: tool-call-shaped assistant text without structured tool content reaches handleMessageEnd, emits the diagnostic warning, and continues through assistant-text handling. I did not live-run the reporter's LiteLLM/vLLM setup.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but forced tool_choice retries and deterministic URL/search routing need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
