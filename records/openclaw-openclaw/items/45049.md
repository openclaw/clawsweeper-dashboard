## Label Justifications

- `P1`: The report describes a broken agent/tool workflow that can deliver fabricated tool results or omit required actions in real tool-enabled sessions.
- `impact:security`: The issue concerns protocol-boundary enforcement around tool invocation and prompt-injection-resilient tool policy.
- `impact:message-loss`: When the model returns prose instead of making a real tool call, the intended tool-mediated action or reply can be skipped.
- `impact:session-state`: The failure mode leaves transcripts with assistant claims but no matching tool-call/tool-result ledger entries.

## Summary

Keep open: current main and v2026.6.10 still have warning-only handling for JSON/XML/ReAct-style fake tool-call text in ordinary agent turns. The merged Gateway HTTP tool-choice work is useful, but it is scoped to caller-supplied HTTP client tools and does not implement the broader agent-loop enforcement requested here.

## What This Changes

The issue requests structural agent-loop enforcement so assistant prose cannot substitute for a real protocol-level tool call when tool use is required.

## Best Possible Solution

Define a maintainer-approved enforcement contract for agent turns: keep structured `toolCall` as the only execution path, decide which fake-tool-text cases should warn, suppress, fail closed, or retry, and align local-model compatibility, docs, and tests with that contract.

## Reproduction Assessment

Yes from source inspection, though I did not live-run the reporter's LiteLLM/vLLM setup in this read-only review. Current main still has a warning-only JSON/XML/ReAct fake-tool-text path and no general agent-loop retry or fail-closed gate when no structured `toolCall` block is present.

## Solution Assessment

Unclear. A hard protocol boundary is the right layer for security, but the exact behavior needs maintainer policy because forced tool choice, fail-closed retries, visible-output suppression, and local-model compatibility have different upgrade impacts.
