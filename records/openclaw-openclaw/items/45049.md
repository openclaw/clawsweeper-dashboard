## Summary

Keep open: current main has partial safeguards, but the central protocol-boundary question remains unresolved. The Gateway HTTP API now rejects unsatisfied `tool_choice` requests, while the embedded agent path still has compatibility promotion for some standalone text-form tool calls and only logs other shaped text as unsupported provider output.

## What This Changes

The issue requests stronger agent-loop enforcement so models cannot claim or simulate tool use in assistant text instead of producing structured tool-call protocol events.

## Best Possible Solution

Make an explicit maintainer/security decision for the embedded agent protocol boundary: either remove/gate text-form tool promotion behind a clearly named compatibility mode, or add a strict pre-response/tool-ledger enforcement path with focused docs and upgrade tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main reproduction path for the remaining class: the embedded runner still promotes some standalone text-form tool calls, while other shaped text is warning-only rather than a forced retry or hard failure. I did not run a live vLLM/LiteLLM reproduction in this read-only pass.

## Solution Assessment

No, the current partial solution is not the complete fix for this issue. Gateway `tool_choice` enforcement is useful, but the embedded agent-loop policy still needs a maintainer-approved strict boundary or compatibility-gated design.
