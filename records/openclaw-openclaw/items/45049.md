## Summary

Current main still treats generic JSON/XML/ReAct-style simulated tool-call text as warning-only assistant prose, so the security-relevant protocol-boundary work is not implemented and should stay open for maintainer policy direction.

## What This Changes

The issue asks OpenClaw to enforce real structured tool invocations instead of allowing model text that simulates tool calls or tool results.

## Best Possible Solution

Define a maintainer-approved strict protocol boundary where executable work originates from structured tool events, suspicious assistant tool text is blocked or surfaced explicitly, provider exceptions are documented, and retry/routing paths preserve hooks, approvals, provenance, docs, and tests.

## Reproduction Assessment

Yes, at source level: an assistant message containing `{"tool":"web_fetch","args":{"url":"https://nos.nl"}}` and no structured tool block reaches `handleMessageEnd`, logs the simulated-tool warning, and then continues normal assistant text delivery. I did not run a live vLLM/LiteLLM reproduction in this read-only review.

## Solution Assessment

Unclear as proposed. Strict protocol enforcement is the right boundary, but deterministic URL-to-tool routing and forced-tool retries need maintainer security/product policy so they do not create unsafe or compatibility-breaking tool calls.
