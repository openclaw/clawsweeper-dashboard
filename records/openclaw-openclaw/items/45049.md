## Summary

Keep open. Current main still treats JSON/XML/ReAct-style simulated tool calls as assistant text with warning/cleanup paths, while strict protocol enforcement, forced retry behavior, and deterministic routing remain unimplemented policy decisions.

## What This Changes

Request stricter agent-loop handling so tool-required prompts cannot complete as simulated tool-call text instead of structured tool invocation.

## Best Possible Solution

Define a strict structured-tool boundary where suspicious assistant text is blocked or surfaced explicitly, any retry goes through structured tool events with hooks and approvals, provider exceptions are documented, and local-model docs/tests are updated.

## Reproduction Assessment

Yes, at source level: an assistant message containing JSON/XML/ReAct-style tool-call text and no structured tool block reaches handleMessageEnd, logs a warning, and continues normal assistant reply handling. I did not live-run the reporter's LiteLLM/vLLM stack.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but forced tool_choice retries and deterministic routing need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
