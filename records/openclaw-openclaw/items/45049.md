## Summary

Keep open. Current main has warning-only diagnostics, visible-text sanitizing for some tool-shaped formats, and documented forced-tool workarounds, but it still does not enforce a strict structured-tool protocol or retry/block when a tool-required turn ends as assistant text.

## What This Changes

Require the agent loop to handle simulated textual tool calls as a protocol-boundary failure and preserve real tool execution through structured tool-call events.

## Best Possible Solution

Define a maintainer-approved strict tool-call boundary where executable tool activity originates from structured tool events, suspicious tool-call text is blocked or surfaced explicitly, provider exceptions are documented, and any retry/routing path goes through the normal hook, approval, provenance, docs, and test surfaces.

## Reproduction Assessment

Yes, at source level: an assistant message containing JSON/XML/ReAct-style tool-call text and no structured tool block reaches handleMessageEnd, logs a warning, and continues normal assistant reply handling. I did not live-run the reporter's vLLM/LiteLLM setup in this read-only review.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but forced tool_choice retries and rule-based routing need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
