## Label Justifications

- `P1`: The issue affects real agent/tool workflows and has current source-level evidence plus user reports of apparent action completion without real tool invocation.
- `impact:security`: The report concerns preserving the structured tool-call boundary that policy hooks, approvals, and provenance rely on.
- `impact:message-loss`: Users can receive normal-looking assistant output while the intended tool-backed action was skipped.
- `impact:session-state`: Assistant text can diverge from the turn ledger and transcript state when fabricated tool invocation details are recorded only as text.

## Summary

Keep open. Current main detects tool-call-shaped assistant text and logs a warning, but it still treats that content as assistant text and proceeds through normal visible delivery; strict blocking, retrying, or forced tool routing remains an unresolved security/product decision.

## What This Changes

The issue asks OpenClaw to enforce structured tool invocation when an assistant simulates a tool call in text instead of emitting real `tool_calls`.

## Best Possible Solution

Define a protocol-boundary policy for suspicious assistant text, with any retry or routing path going through the existing structured tool hook pipeline and with documented default versus opt-in behavior for local providers.

## Reproduction Assessment

Yes, at source level. Current main detects tool-call-shaped assistant text and logs a warning, then continues through normal assistant-text delivery without a strict block, retry, or forced tool-choice policy.

## Solution Assessment

Unclear. A strict structured-tool boundary is the right security direction, but the exact default, retry/routing behavior, and provider compatibility policy need maintainer approval before implementation.
