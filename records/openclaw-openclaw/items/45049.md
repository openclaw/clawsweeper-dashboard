## Label Justifications

- `P1`: The issue affects real agent/tool workflows now and can produce user-visible success text without the expected tool invocation.
- `impact:security`: The remaining work is about preserving the tool-call protocol boundary and policy hook guarantees before execution.
- `impact:session-state`: Assistant text can claim tool activity that is absent from the structured turn ledger, creating transcript and state drift.
- `impact:message-loss`: Tool-backed actions, including messaging or external writes, can be skipped while the user receives a normal-looking reply.

## Summary

Keep open. Current main detects several tool-call-shaped assistant-text patterns and logs that they are being treated as text, but it still does not define the requested strict enforcement, retry, or routing policy for models that omit structured tool calls.

## What This Changes

Request to prevent prose-simulated tool calls from being accepted as normal successful agent output and to define forced-tool or routing behavior for local model stacks.

## Best Possible Solution

Define a strict structured-tool boundary for suspicious assistant text, decide provider/default/config semantics for retries or routing, and ensure every enforced call still enters the normal hook, approval, and provenance pipeline.

## Reproduction Assessment

Yes, at source level: tool-call-shaped assistant text without structured tool content reaches handleMessageEnd, emits only the diagnostic warning, and continues through assistant-text handling. I did not live-run the reporter's LiteLLM/vLLM setup.

## Solution Assessment

Unclear. Strict protocol enforcement is the right boundary, but forced tool_choice retries and deterministic URL/search routing need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
