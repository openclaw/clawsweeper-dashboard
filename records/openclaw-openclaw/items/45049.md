## Summary

Keep open: current main has partial mitigation and diagnostics, but it does not fully enforce real tool invocation for the reported agent-loop problem. The merged Gateway fix only covers OpenAI-compatible HTTP client `tool_choice` contracts, while the core agent/provider paths still need a maintainer-approved security/product decision for simulated or plain-text tool-call behavior.

## What This Changes

The issue asks OpenClaw to prevent agents from treating simulated tool-call text as successful work and to enforce real structured tool invocation when a tool is required.

## Best Possible Solution

Define one core policy boundary for agent turns: real structured tool calls enter the tool pipeline and hooks, while simulated text-shaped calls are rejected, retried under a constrained tool-choice policy, or kept only behind an explicit narrow compatibility path.

## Reproduction Assessment

Yes, at source level rather than from a live vLLM/LiteLLM run: current main logs JSON/XML/ReAct-shaped tool text as text and still contains plain-text tool-call compatibility paths, while issue comments provide concrete v2026.3.12 and v2026.4.9 reproductions.

## Solution Assessment

Unclear; the narrow Gateway `tool_choice` fix is useful but not the best complete solution for this issue because internal agent-loop enforcement and provider compatibility policy remain undecided.
