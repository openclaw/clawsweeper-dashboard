## Summary

Keep open. Current main still detects simulated tool-call text only as a warning path, while the requested strict protocol boundary, forced retry behavior, and deterministic routing remain unresolved and security-sensitive policy work.

## What This Changes

The issue asks OpenClaw to stop treating simulated tool-call text as acceptable assistant output and to add stronger enforcement or routing when tools are clearly required.

## Best Possible Solution

Define a strict structured-tool boundary for suspicious assistant text, decide the provider/config defaults for retries and routing, and ensure any enforced call still flows through the normal tool hook, approval, and provenance pipeline.

## Reproduction Assessment

Yes, at source level: an assistant message containing registered JSON/XML/ReAct-style tool-call text but no structured tool block reaches the message-end handler, produces a warning, and continues as assistant text. I did not live-run the reporter's LiteLLM/vLLM setup in this read-only review.

## Solution Assessment

Unclear. Strict protocol enforcement is the right security boundary, but the proposed forced retry and deterministic routing behavior need maintainer policy so they do not bypass hooks, approvals, provider compatibility, or user intent.
