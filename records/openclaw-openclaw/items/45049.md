## Summary

Keep open. Current main has warning-grade detection, display/history sanitization, a per-model `tool_choice: "required"` workaround, a narrow OpenAI/Codex GPT-5 strict-agentic path, and an LM Studio bracketed-text promotion exception, but it still does not provide the generic protocol-boundary enforcement or provider-scoped retry/routing behavior requested for LiteLLM/vLLM local-model failures.

## What This Changes

The issue asks OpenClaw to prevent models from satisfying tool-required tasks by emitting simulated tool-call text instead of structured `tool_calls` invocations.

## Best Possible Solution

Define a maintainer-approved strict protocol policy: simulated tool-call text should be blocked, surfaced, or retried according to explicit mode/provider rules; any deterministic routing or promotion must enter the normal hook, approval, provenance, and tool-result ledger path; docs and tests should cover LiteLLM/vLLM, LM Studio, and unsupported-provider behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by passing an assistant message containing JSON/XML/ReAct-style tool-call text and no structured tool block into `handleMessageEnd`; current main logs the warning and continues through visible-reply handling. The issue also has fresh live vLLM/Hermes evidence in the discussion.

## Solution Assessment

No. Current main’s warning, display sanitization, per-model `tool_choice: "required"` workaround, LM Studio exception, and narrow GPT-5 strict-agentic path are partial mitigations, not the generic local-provider protocol-boundary enforcement requested here.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the remaining design controls the boundary between assistant text, structured tool execution, policy hooks, approvals, provenance, and user-visible completion claims.

Concerns:

- **[medium] Decide strict text-to-tool boundary semantics:** `src/agents/pi-embedded-subscribe.tool-text-diagnostics.ts:75`
  - body: Current main logs tool-call-shaped assistant text and treats it as text. A block, retry, or promotion path needs explicit maintainer policy so assistant prose cannot be confused with real execution.
  - confidence: 0.9
- **[medium] Preserve hook and approval guarantees:** `src/agents/pi-tool-definition-adapter.ts:233`
  - body: Real tools run through `before_tool_call` before execution. Any deterministic route or text-promotion path must enter this same adapter boundary rather than bypassing plugin policy, approvals, and provenance.
  - confidence: 0.88
- **[medium] Scope forced tool choice carefully:** `docs/gateway/local-models.md:239`
  - body: Docs already warn that `tool_choice: "required"` should only be used where every normal turn should call a tool. Generic forced retries could cause unintended tool calls if applied across normal chat turns.
  - confidence: 0.82
- **[medium] Reconcile LM Studio text promotion:** `extensions/lmstudio/src/stream.ts:159`
  - body: LM Studio currently promotes a narrow bracketed text format into structured tool-call blocks for registered tools. A strict policy should decide whether that exception remains allowed and under which constraints.
  - confidence: 0.78
