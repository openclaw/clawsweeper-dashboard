## Summary

Keep open: the PR targets a real Telegram partial-preview duplicate-final regression, but the implementation suppresses matching final payloads for every channel that uses `onPartialReply`, which can bypass channel-specific preview finalization paths such as Slack draft final edits.

## What This Changes

The PR records the latest partial-preview text in `runReplyAgent()`, suppresses matching final reply payloads in `buildReplyPayloads()`, and adds focused regression tests.

## Best Possible Solution

Scope the dedupe to the Telegram/finalizable-preview contract boundary, or pass an explicit durable-preview signal into payload building, so Telegram suppresses duplicate finals without starving other channel finalizers.

## Reproduction Assessment

Yes for source-level reproducibility: current main leaves final payloads intact when partial preview streaming runs with block streaming off, and the supplied proof shows duplicate final payloads before the patch. I did not run local tests because this review is read-only.

## Solution Assessment

No, not as currently implemented: the dedupe fix belongs behind a Telegram or explicit finalizable-preview contract, not as a blanket rule for every `onPartialReply` user.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope preview dedupe to finalizable preview channels:** `src/auto-reply/reply/agent-runner-payloads.ts:432-434`
  - body: `previewStreamedText` now makes `buildReplyPayloads()` drop final text for any channel whose `opts.onPartialReply` fired. Slack preview streaming also uses `onPartialReply`, but its final payload is what runs `deliverWithFinalizableLivePreviewAdapter()` and marks the draft as committed; without that payload, cleanup can discard the pending draft. Please scope this dedupe to Telegram or an explicit finalizable-preview contract instead of every partial-preview channel.
  - confidence: 0.88
