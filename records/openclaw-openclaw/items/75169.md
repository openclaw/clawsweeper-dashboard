## Summary

Keep open. Current main still has the GitHub skill setup docs gap, but the PR is not merge-ready because it presents a fresh session as sufficient after installing `gh` even though host binary detection is cached in the gateway process.

## What This Changes

The PR adds GitHub skill setup guidance with gateway restart/status commands, `gh auth status`, and Linux/systemd PATH troubleshooting after installing or authenticating `gh`.

## Best Possible Solution

Keep the docs-only fix, but make the install/PATH-change path require `openclaw gateway restart` and reserve fresh-session language for snapshot-only cases where it is actually sufficient.

## Reproduction Assessment

Yes. Current main reproduces the docs gap, and code inspection shows the PR wording issue: a cached missing `gh` result survives a new agent session in the same gateway process.

## Solution Assessment

No. A docs-only change is the right scope, but this wording should distinguish newly installed binaries from authentication or session-snapshot changes so it matches the runtime contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require a gateway restart after installing gh:** `skills/github/SKILL.md:67`
  - body: This says a fresh session is an alternative after installing `gh`, but `hasBinary()` caches a missing binary until the gateway process `PATH`/`PATHEXT` changes or the process restarts. If the gateway already evaluated `gh` as absent, a new agent session reuses that cached false result, so this preserves the setup failure the docs are trying to fix.
