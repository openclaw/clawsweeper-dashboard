## Summary

Keep open: the docs-only direction is useful for the linked GitHub CLI setup problem, but the added wording still says a fresh session can substitute for a gateway restart after installing `gh`, which does not match the current binary-eligibility cache behavior.

## What This Changes

The PR adds GitHub skill setup guidance for restarting and checking the OpenClaw gateway after installing or authenticating the GitHub CLI, plus service PATH troubleshooting text.

## Best Possible Solution

Revise the docs so install or PATH changes require `openclaw gateway restart`, keep fresh-session language only where it matches session snapshots, and preserve the current GH_CONFIG_DIR troubleshooting guidance from main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the GitHub skill requires `gh`, eligibility calls `hasBinary`, and `hasBinary` caches a missing binary until `PATH`/`PATHEXT` changes or the gateway process restarts.

## Solution Assessment

No. A docs-only fix is the right scope, but this PR should not tell users that a fresh session is enough after installing `gh`; the install/PATH case needs gateway restart wording and the auth/HOME cases should stay separate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require restart after installing gh:** `skills/github/SKILL.md:67`
  - body: The added setup text says a fresh session is an alternative after installing `gh`, but GitHub skill eligibility flows through `hasBinary()`, which caches a missing binary until `PATH`/`PATHEXT` changes or the gateway process restarts. A new agent session can reuse that cached false result, so this guidance can leave the skill unavailable after install; make install/PATH changes require `openclaw gateway restart` and reserve fresh-session wording for cases where the session snapshot is the actual issue.
