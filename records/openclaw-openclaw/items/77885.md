## Summary

Keep open. Current main only covers the bundled Compose and Podman launchers with an init process; the root image and several cloud/manual deployment examples still start Node directly as PID 1, so the reported container zombie risk is not already resolved.

## What This Changes

The issue requests wrapping OpenClaw container starts with a PID 1 init process such as `tini` or Docker/Podman `--init` so orphaned child processes are reaped.

## Best Possible Solution

Add an image-level init wrapper or verified equivalent for long-running OpenClaw containers, then align the Docker/Fly/VM docs and regression checks while preserving the existing Compose and Podman init coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live-reproduced here, but the source path is clear: the root Dockerfile starts Node as PID 1 with no entrypoint init, while related launchers explicitly opt into init handling.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: fix the container launch boundary with an init wrapper or Docker/Podman init setting, rather than adding OpenClaw-specific process-reaping logic inside the Node app.
