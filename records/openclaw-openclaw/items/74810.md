## Summary

Keep open. Current main still matches the reported failure shape: `modelRun` and `llm-task` intentionally remove callable tools, inherited explicit allowlist sources are still collected, and the empty-allowlist guard still errors when the callable tool list is empty. Related open PRs #73031 and #74139 cover the likely fix surface but have not landed.

## What This Changes

The issue asks for intentional LLM-only model runs to ignore inherited tool allowlists instead of failing before provider execution.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global or agent `tools.allow`/`tools.alsoAllow`; keep allowlist validation strict for normal tool-enabled runs and explicit runtime tool requests.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` reproductions, and source inspection on current main shows the same path: tools are removed, inherited allowlist sources remain, and the guard still errors before provider execution.

## Solution Assessment

Yes at the product level, but the implementation should be scoped carefully. The safest fix is to neutralize inherited allowlist checks for intentional LLM-only calls without weakening explicit allowlist failure behavior for runs that are supposed to expose tools.
