## Summary

Keep open. Current main still has the reported failure shape: raw model runs and `llm-task` intentionally run without tools, inherited `tools.alsoAllow` becomes an explicit allowlist source, and the empty-callable-tools guard still errors when `disableTools` is true. Related open PRs cover overlapping fix paths, but nothing is merged into current main.

## What This Changes

The issue requests that LLM-only `modelRun` and `llm-task` executions ignore inherited tool allowlists instead of failing before provider execution when tools are intentionally disabled.

## Best Possible Solution

Land one focused fix that treats intentional LLM-only provider calls as no-tool runs even when inherited config allowlists exist, while preserving fail-closed validation for explicit tool-enabled allowlist requests. Add regression coverage for both gateway `modelRun` and `llm-task` under global `tools.profile` or `tools.alsoAllow`, then include the user-facing changelog entry.

## Reproduction Assessment

Yes. The issue provides concrete CLI repros and a later comment provides a raw `/tools/invoke` repro; current-main source inspection shows the same path still exists without needing a live provider run.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if it is scoped to intentional LLM-only runs. The safer implementation should neutralize inherited allowlist checks for those no-tool calls without suppressing the guard for ordinary tool-enabled or explicit-runtime-allowlist runs.
