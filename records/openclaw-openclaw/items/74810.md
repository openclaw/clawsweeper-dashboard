## Summary

Keep open. Current main still matches the reported failure shape: raw `modelRun` and `llm-task` intentionally disable tools, but inherited explicit allowlist sources are still collected and the empty-callable-tools guard still errors instead of treating the call as an intentional no-tool provider run. Related PRs #73031 and #74139 are open and overlap the fix surface, but neither has landed on current main.

## What This Changes

The issue requests that intentional LLM-only `modelRun` and `llm-task` runs bypass inherited tool allowlist empty-callable failures while preserving fail-closed allowlist behavior for normal tool-enabled runs.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global `tools.profile` or `tools.alsoAllow`; include a user-facing changelog entry when the fix lands.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` repros, and current-main source inspection shows the same path: tools are disabled, inherited allowlist sources remain, and the guard still errors before provider execution.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if scoped to intentional LLM-only runs. The safer implementation should skip or neutralize inherited allowlist checks for those no-tool calls without weakening validation for ordinary tool-enabled runs or explicit runtime `toolsAllow` mistakes.
