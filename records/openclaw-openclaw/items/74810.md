## Summary

Keep open: current main still has the reported empty-callable-tools precheck for intentional no-tool runs with inherited allowlist sources, and the related open fix PRs have not landed.

## What This Changes

LLM-only `modelRun` and `llm-task` runs should proceed as no-tool provider calls even when global or agent tool allowlists are non-empty.

## Best Possible Solution

Land one focused fix that treats inherited ambient allowlists as irrelevant for intentional no-tool model calls while preserving runtime `toolsAllow` validation, with regression coverage for gateway modelRun and the embedded `llm-task`/heartbeat shape.

## Reproduction Assessment

Yes. The reporter provided concrete CLI and `/tools/invoke` paths, and current-main source still shows `modelRun`/`disableTools` producing zero tools while inherited allowlist sources trigger the pre-provider guard.

## Solution Assessment

Yes at the product level, but the implementation boundary matters. The narrow maintainable solution is to exempt intentional no-tool runs from inherited ambient allowlist failures without allowing an explicit per-run `toolsAllow` plus `disableTools` contradiction.
