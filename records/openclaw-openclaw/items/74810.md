## Summary

Keep open. Current main still matches the reported failure shape: intentional no-tool `modelRun` and `llm-task` paths can reduce callable tools to an empty set, while inherited explicit allowlist sources are still collected and the empty-allowlist guard still returns an error before provider execution. Related open PRs #73031 and #74139 overlap the likely fix surface, so this issue should stay open until one reconciled fix lands.

## What This Changes

LLM-only `modelRun` and `llm-task` runs should ignore inherited tool allowlists when tools are intentionally disabled.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited config allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global or agent tool allowlists; keep explicit runtime allowlist mistakes and normal tool-enabled allowlist misses fail-closed.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` repros, and current-main source inspection shows the same path: tools are disabled, inherited allowlist sources remain, and the guard can still error before provider execution.

## Solution Assessment

Yes at the product level, but the implementation scope needs care. The safest fix should neutralize inherited allowlist checks for intentional LLM-only runs without weakening validation for tool-enabled runs or contradictory explicit runtime `toolsAllow` requests.
