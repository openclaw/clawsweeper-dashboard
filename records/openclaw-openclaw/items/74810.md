## Summary

Keep open. Current `main` still has the reported failure shape: LLM-only runs clear callable tools, inherited allowlist sources are still collected, and the empty-allowlist guard still turns that into a preflight error before provider execution. The related open PRs cover overlapping parts of the fix surface, so this should be reconciled rather than closed or duplicated.

## What This Changes

Intentional no-tool `modelRun` and `llm-task` executions should proceed as LLM-only provider calls even when global or inherited tool allowlists are non-empty.

## Best Possible Solution

Land one focused fix that treats intentional LLM-only provider calls as no-tool runs even when inherited config allowlists exist, with regression coverage for both gateway `modelRun` and `llm-task`; preserve fail-closed behavior for ordinary tool-enabled allowlist misses and explicit runtime allowlist conflicts.

## Reproduction Assessment

Yes. The issue provides concrete CLI reproductions, a follow-up comment provides a direct `/tools/invoke` reproduction, and current-main source inspection shows the same preflight path from empty callable tools plus inherited allowlist sources to provider-skipping error.

## Solution Assessment

Yes at the product level, but the implementation should be scoped carefully. The safest fix is to neutralize inherited allowlist checks only for intentional LLM-only runs while preserving validation for normal tool-enabled runs and explicit runtime allowlist mistakes.
