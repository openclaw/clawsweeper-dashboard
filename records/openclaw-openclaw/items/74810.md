## Summary

Keep open. Current main still matches the reported failure shape: intentional no-tool `modelRun` and `llm-task` runs clear the callable tool set, but inherited explicit allowlist sources are still collected and the guard still preflights them as an error before provider execution. Related PRs #73031 and #74139 overlap the fix surface but neither is merged into current main.

## What This Changes

The issue requests that intentional LLM-only `modelRun` and `llm-task` executions ignore inherited non-empty tool allowlists instead of failing the empty-callable-tools guard.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited config allowlists exist, with regression coverage for both gateway `modelRun` and `llm-task`; keep ordinary tool-enabled allowlist misses and explicit runtime allowlist mistakes fail-closed.

## Reproduction Assessment

Yes. The issue provides concrete CLI repros, a follow-up comment provides a direct `/tools/invoke` repro, and current-main source inspection shows the same preflight path: empty callable tools plus non-empty inherited allowlist sources yields an error before provider execution.

## Solution Assessment

Yes at the product level, but the implementation should be scoped carefully. The safest fix is to neutralize inherited allowlist checks only for intentional LLM-only runs while preserving validation for normal tool-enabled runs and explicit runtime allowlist conflicts.
