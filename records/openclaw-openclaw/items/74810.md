## Summary

Keep open: current main still has the reported failure path for intentional no-tool runs with inherited allowlist entries, and the issue adds the generic gateway `modelRun` path alongside the related `llm-task` workflow report.

## What This Changes

LLM-only `modelRun` and `llm-task` runs should ignore inherited tool allowlists instead of failing the empty-callable-tools guard.

## Best Possible Solution

Land one focused fix, likely by reconciling #73031 and #74139, that treats intentional no-tool provider calls as no-tool runs while preserving fail-closed allowlist validation for ordinary tool-enabled runs.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` reproductions, and current-main source inspection shows the same empty-tools plus inherited-allowlist precheck path before provider execution.

## Solution Assessment

Yes at the product level, but the implementation boundary matters. The safest fix should neutralize inherited allowlist checks only for intentional LLM-only runs and add regression coverage for both gateway `modelRun` and `llm-task`.
