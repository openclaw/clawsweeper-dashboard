## What This Changes

Fix LLM-only `modelRun` and `llm-task` execution so inherited tool allowlists do not abort intentional no-tool provider calls.

## Best Possible Solution

Land one focused fix, likely by reconciling #73031 and #74139, that treats intentional no-tool provider calls as no-tool runs while preserving fail-closed allowlist validation for ordinary tool-enabled runs.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` reproductions, and current-main source inspection confirms the same empty tools plus inherited allowlist guard path before provider execution.

## Solution Assessment

Yes at the product level, but the implementation boundary matters. The safest fix is to neutralize inherited allowlist checks only for intentional LLM-only runs and add regression coverage for both generic `modelRun` and `llm-task`.
