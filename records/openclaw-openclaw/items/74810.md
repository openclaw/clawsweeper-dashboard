## Summary

Keep open. Current main still matches the reported failure shape: raw `modelRun` and `llm-task` intentionally disable tools, but inherited allowlist sources from `tools.profile` / `tools.alsoAllow` are still collected and the empty-callable-tools guard still turns `disableTools` into a pre-provider error. The follow-up comment adds an independent `/tools/invoke` reproduction through `llm-task`, and open PRs #73031 and #74139 cover overlapping fix surfaces but are not merged into current main.

## What This Changes

LLM-only one-shot model runs and `llm-task` calls should proceed as no-tool provider calls even when the runtime inherits non-empty global tool allowlists.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global `tools.profile` or `tools.alsoAllow`; include a user-facing changelog entry when it lands.

## Reproduction Assessment

Yes. The issue and follow-up comment give concrete CLI and `/tools/invoke` reproductions, and current-main source inspection shows the same path: tools are disabled, inherited allowlist sources remain, and the pre-provider guard still errors on an empty callable set.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if scoped to intentional LLM-only runs. The safer implementation should skip or neutralize inherited allowlist checks for those no-tool calls without weakening validation for ordinary tool-enabled or explicit runtime-allowlist runs.
