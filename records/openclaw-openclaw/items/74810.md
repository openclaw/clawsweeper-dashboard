## Summary

Keep open. Current main still matches the reported failure shape: intentional no-tool `modelRun` and `llm-task` paths zero out callable tools, but inherited explicit allowlist sources are still collected and the empty-allowlist guard still returns an error before provider execution. Related open PRs #73031 and #74139 overlap the fix surface, but neither has landed on current main.

## What This Changes

The issue requests that intentional LLM-only `modelRun` and `llm-task` runs ignore inherited tool allowlists instead of failing the empty-callable-tools guard.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global `tools.profile` or `tools.alsoAllow`; preserve fail-closed behavior for ordinary tool-enabled allowlist misses and explicit runtime allowlist mistakes.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` repros, and current-main source inspection shows the same path: tools are disabled, inherited allowlist sources remain, and the guard still errors before provider execution.

## Solution Assessment

Yes at the product level, but the safest implementation should be scoped carefully. It should skip or neutralize inherited allowlist checks for intentional LLM-only runs without weakening validation for normal tool-enabled runs or contradictory explicit runtime `toolsAllow` requests.
