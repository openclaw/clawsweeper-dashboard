## Summary

Keep open. Current main still matches the reported failure shape: raw gateway `modelRun` and plugin-owned `llm-task` intentionally run with tools disabled, but inherited explicit allowlist sources are still collected and the empty-callable-tools guard still returns the reported pre-provider error. The follow-up production reproduction strengthens the report, and related PRs #73031 and #74139 overlap the fix surface but remain open rather than landed.

## What This Changes

The issue requests that intentional LLM-only `modelRun` and `llm-task` runs bypass inherited tool allowlist empty-callable failures while preserving fail-closed behavior for normal tool-enabled runs.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as valid no-tool runs when inherited allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global `tools.profile` or `tools.alsoAllow`; add a user-facing changelog entry when it lands.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete CLI and `/tools/invoke` repros, and current-main source inspection shows the same structural path: tools are disabled, inherited allowlist sources remain, and the guard returns the reported error before provider execution.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if scoped to intentional LLM-only runs; the implementation should skip or neutralize inherited allowlist checks for those no-tool calls without weakening validation for normal tool-enabled runs or accidental explicit runtime `toolsAllow` conflicts.
