## Summary

Keep open: current main still has the reported failure shape, and the follow-up comment adds an independent `/tools/invoke` reproduction for the related `llm-task` path. The linked open PRs are implementation candidates, but none has merged into main yet.

## What This Changes

Intentional LLM-only `modelRun` and `llm-task` runs should proceed without tools even when inherited tool allowlists are non-empty.

## Best Possible Solution

Land one focused fix, likely reconciling #73031 and #74139, that treats intentional no-tool provider calls as no-tool runs while preserving allowlist validation for ordinary tool-enabled runs.

## Reproduction Assessment

Yes. The issue provides CLI repro steps, the follow-up comment provides a raw `/tools/invoke` repro, and current-main source inspection shows the same empty-tools plus inherited-allowlist precheck path before provider execution.

## Solution Assessment

Yes at the product level, but the implementation boundary matters. The safest fix is to neutralize inherited allowlist checks only for intentional LLM-only runs and add regression coverage for both gateway `modelRun` and `llm-task`.
