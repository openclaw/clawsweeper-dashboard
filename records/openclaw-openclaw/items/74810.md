## Summary

Keep open. Current main still matches the reported failure shape: intentional LLM-only runs can empty the callable tool set while inherited allowlist sources remain non-empty, and the guard still turns that into a preflight error before provider submission. The related open PRs are repair candidates to reconcile, not evidence that the issue is fixed.

## What This Changes

LLM-only `modelRun` and `llm-task` runs should proceed as no-tool provider calls even when global or inherited tool allowlists are configured.

## Best Possible Solution

Reconcile or replace the existing fix candidates with one focused change that treats intentional no-tool provider calls as valid no-tool runs while preserving fail-closed allowlist behavior for normal tool-enabled runs and explicit runtime allowlist mistakes. Add regression coverage for both gateway `modelRun` and `llm-task` paths.

## Reproduction Assessment

Yes. The issue provides concrete CLI and curl reproductions, a second reporter confirmed the `/tools/invoke` path, and current-main source inspection shows the same preflight path still exists without needing a live provider call.

## Solution Assessment

Yes at the product level, but the implementation must be scoped carefully. The safest solution is to bypass inherited allowlist emptiness only for intentional LLM-only runs while keeping ordinary allowlist misses and explicit tool-enabled conflicts fail-closed.
