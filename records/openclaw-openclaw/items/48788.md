## Summary

Keep open: current main and v2026.6.6 still lack the requested central multi-encoding Content-Disposition filename decoder, and the linked shared-decoder PR is open, unmerged, and explicitly scoped as partial follow-up work.

## What This Changes

The issue requests a shared filename decoder for RFC 5987 Content-Disposition values, common charset-specific filenames, and mojibake recovery across channel adapters.

## Best Possible Solution

Review or replace the linked shared-decoder PR with a narrow media/plugin-safe decoder, keep the shipped Feishu mitigation, and track any remaining channel migrations under this issue or a maintainer-approved follow-up.

## Reproduction Assessment

Yes, for the source-level architectural gap: current main has local or limited filename parsers but no shared multi-encoding Content-Disposition utility. I did not run live channel downloads during this read-only review.

## Solution Assessment

Yes directionally: a narrow shared media or plugin-safe decoder is the right end state compared with more channel one-offs. The broad architecture fork discussed in comments is too large for this issue, and the open PR still needs maintainer boundary review before it can be canonical.
