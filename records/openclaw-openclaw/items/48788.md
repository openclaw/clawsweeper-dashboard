## Summary

Keep this open: current main still has separate narrow filename parsers and no shared multi-encoding Content-Disposition decoder, while https://github.com/openclaw/openclaw/pull/71517 is the existing implementation candidate for this feature.

## What This Changes

The issue requests a shared filename decoder for Content-Disposition headers with RFC 5987 and multiple legacy encoding recovery across channel adapters.

## Best Possible Solution

Review and land a rebased, focused shared decoder implementation that keeps the Feishu workaround intact, centralizes reusable Content-Disposition decoding in `src/media` or an agreed SDK seam, and leaves narrower channel migrations as follow-up work when needed.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/architecture follow-up. Source inspection gives high confidence that current main lacks the requested shared utility: remote media and Feishu still use separate narrow parsers, and no `src/media/filename.ts` exists.

## Solution Assessment

Yes in direction, but it is not complete on main. The maintainable path is to review or rework https://github.com/openclaw/openclaw/pull/71517 rather than start a separate repair job while that implementation candidate is open.
