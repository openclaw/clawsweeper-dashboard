## Summary

Keep open: current main and v2026.6.10 still use local remote-media and Feishu filename parsers rather than the requested shared multi-encoding Content-Disposition decoder. The open shared-decoder PR is useful context, but it is partial/conflicting and does not safely supersede this issue yet.

## What This Changes

The issue requests a central reusable filename decoder for Content-Disposition headers, mojibake recovery, and non-UTF-8 charset handling across media/channel adapters.

## Best Possible Solution

Choose one plugin-safe shared decoder seam, land conservative charset-aware Content-Disposition parsing with recovery/sanitization tests, and migrate the first media/channel consumers without duplicating core and Feishu implementations.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 still have local parsers and Feishu-specific recovery, while the requested shared multi-encoding utility is absent. I did not run live channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API boundary. The shared utility direction fits the problem, but the maintainable solution should be one focused plugin-safe seam rather than a broad architecture fork or duplicated Feishu/core decoders.
