## Summary

Keep open; current main still only has private fetch/Feishu filename helpers, and the active shared-decoder implementation candidate remains open and unmerged.

## What This Changes

The issue requests a shared Content-Disposition filename decoder with RFC 5987, mojibake recovery, and multi-encoding support for reuse across channel adapters.

## Best Possible Solution

A shared media filename decoder should live in the core/plugin SDK media surface with tests for RFC 5987, Latin-1/Windows-1252 mojibake recovery, explicit CJK charset vectors, and incremental migration away from private channel helpers.

## Reproduction Assessment

Not applicable: this is a feature/API consolidation request rather than a single failing runtime path. Source inspection gives high confidence that the requested central exported utility is absent on current main.

## Solution Assessment

Yes: the maintainable direction is a shared media filename decoder exported through the existing plugin/runtime surface, with Feishu as an initial consumer and broader channel migration added incrementally. Current main is not sufficient yet, and PR #71517 should be reviewed before creating duplicate repair work.
