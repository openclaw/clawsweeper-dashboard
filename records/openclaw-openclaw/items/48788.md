## Summary

Keep open: current main and the latest release still have Feishu-local and remote-media-local filename parsing, not the requested shared multi-encoding decoder. The open shared-decoder PR is useful context, but it is partial, currently conflicting, and still needs maintainer direction on the shared plugin-safe boundary.

## What This Changes

The issue requests a centralized filename decoding utility for RFC 5987, mojibake recovery, and multi-encoding Content-Disposition handling across channel adapters.

## Best Possible Solution

Choose one plugin-safe shared decoder seam, land charset-aware Content-Disposition parsing with conservative recovery and sanitization tests, and migrate the first media/channel consumers without duplicating Feishu-local and core implementations.

## Reproduction Assessment

Yes, by source inspection: current main has only local Content-Disposition parsers and Feishu-specific recovery, while the requested shared multi-encoding utility is absent. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API boundary. The shared utility direction fits the problem, but the best implementation should be a focused plugin-safe seam rather than a broad architecture fork or duplicated Feishu/core decoders.
