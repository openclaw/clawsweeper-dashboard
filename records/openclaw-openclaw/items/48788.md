## Summary

Keep open: current main and v2026.6.10 still have separate remote-media and Feishu filename parsers rather than the requested shared multi-encoding Content-Disposition decoder. The open shared-decoder PR is useful implementation context, but it remains open/conflicting and explicitly partial, so it is not a safe close or supersession target yet.

## What This Changes

The issue requests a reusable filename decoding utility for RFC 5987 and multi-encoding Content-Disposition filename handling across media/channel adapters.

## Best Possible Solution

Have maintainers choose one plugin-safe shared media filename decoder seam, then refresh and land https://github.com/openclaw/openclaw/pull/71517 or a narrower replacement that preserves the Feishu fix without duplicating decoder logic.

## Reproduction Assessment

Yes, at source level: current main still lacks a shared multi-encoding decoder and keeps separate remote-media and Feishu filename parsers. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API boundary. A shared decoder is the right direction, but the maintainable solution should be one plugin-safe media seam rather than duplicated core and plugin helpers.
