## Summary

Keep open. Current main only has narrow Feishu and remote-media filename parsing, while the requested shared multi-encoding utility and cross-channel rollout are still represented by an open partial implementation PR plus unresolved SDK/API scope decisions.

## What This Changes

The issue requests a shared filename decoding utility for Content-Disposition filenames across channel adapters, including RFC 5987 and common CJK encoding recovery.

## Best Possible Solution

Land a scoped shared decoder in `src/media`, wire the common fetch and Feishu inbound save paths first, settle any public SDK export through docs/contracts, then migrate remaining channel filename paths incrementally under this tracker.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: feed current main a language-tagged or non-UTF-8 `filename*` header through `fetchRemoteMedia`, or a Feishu `file_name` payload through `resolveFeishuMediaList`; those paths do not use a shared multi-encoding decoder today.

## Solution Assessment

Yes on direction, not yet on closure. A shared media decoder with incremental consumers is the maintainable path, but the exact SDK/API surface and remaining channel rollout should be coordinated through the open implementation PR or a rebased successor.
