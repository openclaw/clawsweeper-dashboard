## Summary

Keep open: current main and the latest release still have local Feishu and remote-media filename parsing, not the requested shared multi-encoding Content-Disposition decoder. The open shared-decoder PR is useful context, but it is partial, marked dirty, and still needs a maintainer boundary decision rather than a new cleanup close.

## What This Changes

The issue requests a shared filename decoder that handles RFC 5987 Content-Disposition headers, common mojibake recovery, and explicit encodings across channel adapters.

## Best Possible Solution

Choose one plugin-safe shared decoder seam, land charset-aware Content-Disposition parsing with conservative recovery and sanitization tests, and migrate the first media/channel consumers without duplicating Feishu-local and core implementations.

## Reproduction Assessment

Yes, at source level: current main has only local Content-Disposition parsers and Feishu-specific recovery, while the requested shared multi-encoding utility is absent. I did not run live channel reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API boundary. The shared utility direction fits the problem, but the best implementation should be a focused plugin-safe seam rather than a broad architecture fork or duplicated Feishu/core decoders.
