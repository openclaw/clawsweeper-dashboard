## Summary

Keep open: current main and v2026.6.6 still lack the requested shared multi-encoding Content-Disposition filename decoder; the existing implementation is split between a Feishu-local CJK recovery path and a limited remote-media parser, while https://github.com/openclaw/openclaw/pull/71517 is still open and conflicting rather than a merged fix.

## What This Changes

The issue requests a centralized filename utility that parses RFC 5987 Content-Disposition filenames, recovers common mojibake, supports non-UTF-8 charsets, and can be reused across channel adapters.

## Best Possible Solution

Finish review of the linked shared-decoder PR or replace it with a narrow equivalent, then track channel-by-channel migrations and the Feishu JSON filename path as follow-up work where needed.

## Reproduction Assessment

Yes, for the source-level architectural gap: current main has local or limited filename parsers but no shared multi-encoding utility. I did not run live Feishu or remote-media downloads during this read-only review.

## Solution Assessment

Unclear until the shared boundary is reviewed. A narrow shared media/plugin-safe decoder is the maintainable direction, while the broader architecture fork discussed in comments is too broad for this issue.
