## Summary

Keep this open: current main has the Feishu-specific Latin-1/CJK Content-Disposition repair, but it still lacks the requested shared multi-encoding filename decoder and an open PR is already carrying the likely implementation path.

## What This Changes

Request a centralized filename decoding utility for RFC 5987, mojibake recovery, and explicit charset handling across channel/media filename paths.

## Best Possible Solution

Review and land a rebased, narrow shared decoder implementation that keeps the Feishu workaround intact, centralizes reusable Content-Disposition decoding in the media layer or an agreed SDK seam, and adds focused tests for RFC 5987 language tags, explicit charsets, basename sanitization, and mojibake recovery.

## Reproduction Assessment

Not applicable as a strict bug reproduction, because this is a feature/architecture follow-up. Source inspection gives high confidence that current main lacks the requested shared utility: remote media and Feishu still use separate narrow parsers, and no `src/media/filename.ts` exists.

## Solution Assessment

Yes in direction, but not complete on main. The maintainable path is to review or rework the existing shared-decoder PR rather than start a separate repair job while that implementation candidate is open.
