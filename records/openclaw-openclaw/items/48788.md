## Summary

Keep this open. Current main still has only a private fetch-local Content-Disposition parser plus a Feishu-local mojibake recovery helper; it does not yet provide the requested shared, exported, multi-encoding filename decoder or broad channel migration. PR #71517 is the active implementation candidate, but it is still open and explicitly scoped as partial work, so it is not merged proof that this tracker can close.

## What This Changes

The issue requests a centralized filename utility for Content-Disposition parsing, RFC 5987 handling, and multi-encoding filename recovery across channel adapters.

## Best Possible Solution

Land or revise the shared decoder work through a narrow media/plugin-SDK surface, with focused tests for RFC 5987 language tags, Latin-1/Windows-1252 mojibake recovery, explicit CJK charset vectors, path basename safety, and incremental migration of the relevant channel download paths.

## Reproduction Assessment

Not applicable as a runtime reproduction for this feature-tracking issue. Source inspection gives high confidence that the requested central exported utility is still absent on current main while only partial local helpers exist.

## Solution Assessment

Yes, the requested shared media filename decoder is the maintainable direction; keeping separate private parsers in fetch and Feishu would keep the behavior inconsistent. PR #71517 is the current candidate to review rather than spawning duplicate work.
