## Summary

This PR should stay open: the bug and fix direction are valid, and the author supplied real Gateway/CLI proof, but the current patch still has two P2 correctness blockers around retry-backoff preservation and malformed cron schedule isolation.

## What This Changes

The PR adds cron maintenance logic and tests to repair persisted future `nextRunAtMs` values that are invalid cron schedule slots, plus a changelog entry for #77867.

## Best Possible Solution

Revise the maintenance repair so it validates future cron slots inside the existing schedule-error handling path and preserves only active retry backoff windows, with regression tests for malformed future-slot jobs and expired error-state jobs.

## Reproduction Assessment

Yes. Current main source preserves still-future `nextRunAtMs` values, and the PR body includes before/after Gateway evidence for the Asia/Shanghai stale-future slot; I did not rerun it because this review is read-only.

## Solution Assessment

No. The repair direction is narrow and appropriate, but the current implementation needs to tie retry preservation to active backoff and keep all cron schedule probes under the existing schedule-error isolation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Limit retry preservation to active backoff windows:** `src/cron/service/jobs.ts:164-165`
  - body: This guard skips stale-future repair for every job with any prior error, even after the retry backoff has expired. A daily cron job with `lastStatus: "error"`, `lastRunAtMs` yesterday, and an invalid week-ahead `nextRunAtMs` would still be preserved, leaving the bug in place for that state; compare the existing `isErrorBackoffPending()` windowed logic in `timer.ts`.
  - confidence: 0.9
- **[P2] Catch schedule errors before probing future cron repairs:** `src/cron/service/jobs.ts:168`
  - body: `shouldRepairFutureCronNextRunAtMs()` calls cron schedule computation before entering `recomputeJobNextRunAtMs()`'s try/catch. A malformed persisted cron job with a future `nextRunAtMs` can now throw through maintenance/list/startup instead of recording `scheduleErrorCount` and continuing with other jobs.
  - confidence: 0.93
