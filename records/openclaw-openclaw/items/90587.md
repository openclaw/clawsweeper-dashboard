## Summary

Keep open: current main and v2026.6.8 still route `openai/gpt-5.5` agent turns through the managed Codex app-server, while the bundled and latest upstream `@openai/codex` packages still publish only x64/arm64 launcher targets and throw on linux/riscv64.

## What This Changes

The issue asks OpenClaw to handle or document the `codex app-server exited` failure on linux/riscv64 when the managed upstream Codex launcher rejects that architecture.

## Best Possible Solution

Make the unsupported Codex platform path explicit in OpenClaw, document the existing custom app-server/WebSocket escape hatches, and track full managed riscv64 support on upstream Codex binary availability or a maintainer-approved fallback policy.

## Reproduction Assessment

Yes from source, though not from a live RISC-V64 host. Current OpenClaw still launches the managed `@openai/codex` wrapper, and both the packaged 0.139.0 launcher and current sibling Codex launcher deterministically reject linux/riscv64.

## Solution Assessment

Unclear as an implementation choice. A narrow OpenClaw preflight/docs improvement is maintainable, but full managed RISC-V64 Codex execution depends on upstream binary support or an explicit maintainer-approved fallback policy.
