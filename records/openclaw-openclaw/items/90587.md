## Summary

Keep open: current main and v2026.6.8 still route default Codex-backed OpenAI agent turns through the managed @openai/codex app-server, and current upstream Codex still has no linux/riscv64 target in its launcher or npm platform-package matrix. This remains a maintainer product-boundary decision rather than conservative cleanup.

## What This Changes

Request for OpenClaw to handle, work around, or document Codex app-server startup failure on linux/riscv64 where @openai/codex reports an unsupported platform.

## Best Possible Solution

Make the unsupported-platform path explicit in OpenClaw docs and diagnostics while tracking full managed linux/riscv64 Codex support upstream; use the existing custom app-server override only when a compatible Codex binary is supplied.

## Reproduction Assessment

Yes from source, though not from a live RISC-V64 host. Current OpenClaw still starts the managed @openai/codex launcher by default, and upstream Codex deterministically throws for linux/riscv64.

## Solution Assessment

Unclear as an implementation choice. A narrow OpenClaw preflight/error and docs improvement is maintainable, but full managed RISC-V64 Codex execution depends on upstream binary support or a maintainer-approved fallback policy.
