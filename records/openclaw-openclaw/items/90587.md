## Summary

Keep open: current main and the latest release still route the managed Codex app-server through the upstream @openai/codex wrapper, and that wrapper still has no linux/riscv64 target. The report is credible, but the remaining decision is whether OpenClaw should add a clearer unsupported-platform path, document a runtime-routing workaround, or wait for upstream Codex RISC-V64 binaries.

## What This Changes

The issue reports that OpenClaw on Ubuntu RISC-V64 fails when starting the Codex app-server because the bundled OpenAI Codex wrapper rejects linux/riscv64.

## Best Possible Solution

Make the unsupported Codex platform path explicit: either document and surface the existing custom app-server/runtime-policy workarounds, or add a preflight guard that tells linux/riscv64 users that managed Codex needs an upstream-supported binary before agent turns can use the native Codex runtime.

## Reproduction Assessment

Yes from source, though not from a live RISC-V64 host. Current OpenClaw still launches the managed @openai/codex wrapper, and upstream Codex deterministically throws the reported unsupported-platform error for linux/riscv64.

## Solution Assessment

Unclear as an implementation choice. Full RISC-V64 managed Codex support depends on upstream OpenAI Codex binaries, while OpenClaw can still provide a better guard, docs, or runtime-routing workaround.
