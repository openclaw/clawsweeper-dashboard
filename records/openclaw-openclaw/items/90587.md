## Summary

Keep open: current main and v2026.6.8 still route default OpenAI/Codex agent turns through the managed @openai/codex app-server, while current upstream Codex still has no linux/riscv64 launcher target or npm platform package. This remains a maintainer product-boundary/support decision, not conservative cleanup.

## What This Changes

The issue reports linux/riscv64 OpenClaw TUI agent turns failing because the managed Codex app-server launcher rejects `linux (riscv64)`, and asks for a workaround or support plan.

## Best Possible Solution

Make the unsupported-platform path explicit in OpenClaw diagnostics and docs while tracking full managed linux/riscv64 Codex support upstream; use the existing `appServer.command` or `OPENCLAW_CODEX_APP_SERVER_BIN` override only when a compatible Codex binary is supplied.

## Reproduction Assessment

Yes, from source rather than a live RISC-V64 host. Current OpenClaw still starts the managed @openai/codex launcher by default for OpenAI/Codex agent turns, and upstream Codex deterministically throws for linux/riscv64 because no target triple is selected.

## Solution Assessment

Unclear as an implementation choice. A narrow OpenClaw preflight/error and docs improvement is maintainable, but full managed RISC-V64 Codex execution depends on upstream binary support or a maintainer-approved fallback policy.
