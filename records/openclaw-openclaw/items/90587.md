## Summary

Keep open: current main and the latest release still delegate managed Codex app-server startup to upstream @openai/codex, and current upstream Codex still has no linux/riscv64 launcher target. The remaining work is a maintainer product-boundary decision: clearer OpenClaw unsupported-platform handling and docs, a documented custom app-server/runtime workaround, or waiting for upstream Codex RISC-V64 binaries.

## What This Changes

A RISC-V64 user reports that OpenClaw TUI turns using openai/gpt-5.5 fail because the managed Codex app-server exits with upstream `Unsupported platform: linux (riscv64)`.

## Best Possible Solution

Make the unsupported Codex platform path explicit in OpenClaw while tracking upstream RISC-V64 binary support: either add a clear preflight/error and docs for custom app-server or direct OpenClaw runtime routing, or defer full managed support until upstream publishes a compatible binary.

## Reproduction Assessment

Yes from source, though not from a live RISC-V64 host. Current OpenClaw still launches the managed @openai/codex wrapper, and current upstream Codex deterministically throws for linux/riscv64.

## Solution Assessment

Unclear as an implementation choice. A narrow OpenClaw guard/docs improvement is maintainable, but full managed RISC-V64 Codex execution depends on upstream binary support or an explicit maintainer-approved fallback policy.
