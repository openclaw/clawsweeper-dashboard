## Summary

Keep open: current main and v2026.6.8 still route official OpenAI agent turns through the managed Codex app-server, while OpenClaw and upstream Codex still have no linux/riscv64 managed launcher target. This remains a maintainer platform/support decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to handle linux/riscv64 Codex app-server startup instead of failing with the upstream @openai/codex unsupported-platform exit.

## Best Possible Solution

Keep this issue open until maintainers choose the platform policy: improve unsupported-platform diagnostics/docs, support an explicit custom app-server path, or wait for upstream Codex linux/riscv64 binaries.

## Reproduction Assessment

Yes, from source rather than a live RISC-V64 host. Current OpenClaw still starts the managed @openai/codex launcher by default for official OpenAI agent turns, and upstream Codex deterministically throws for linux/riscv64 because no target triple is selected.

## Solution Assessment

Unclear as an implementation choice. A narrow diagnostic/docs improvement is maintainable, but actual successful turns on linux/riscv64 require upstream binary support or a maintainer-approved fallback/custom app-server policy.
