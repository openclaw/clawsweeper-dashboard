## Summary

Keep open: current main and v2026.6.8 still use the managed @openai/codex app-server path without a linux/riscv64 platform package, and current upstream Codex still throws before startup on linux/riscv64. The remaining action is a maintainer product-boundary choice, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to handle or support linux/riscv64 systems where the managed Codex app-server launcher exits with `Unsupported platform: linux (riscv64)`.

## Best Possible Solution

Make the unsupported-platform path explicit in OpenClaw and document the supported custom app-server/binary override path, while tracking full managed linux/riscv64 support upstream.

## Reproduction Assessment

Yes from source, though not from a live RISC-V64 host. Current OpenClaw still starts the managed @openai/codex launcher, and upstream Codex deterministically throws for linux/riscv64.

## Solution Assessment

Unclear as an implementation choice. A narrow OpenClaw preflight/error and docs improvement is maintainable, but full managed RISC-V64 Codex execution depends on upstream binary support or an explicit maintainer-approved fallback policy.
