## Summary

Keep open: this is a valid voice-call status bug fix candidate, but the submitted patch is too broad because it changes shared manager lookups from active-only to active-or-history while those lookups are used by live call guards.

## What This Changes

The PR adds voice-call persisted-store lookup helpers, wires `CallManager.getCall` and `getCallByProviderCallId` to fall back to those helpers, and adds restore tests for completed and multi-snapshot calls.

## Best Possible Solution

Keep manager lookups active-only, add a newest-first persisted lookup behind read-only status surfaces, and provide focused tests plus redacted runtime proof for the completed-call status flow.

## Reproduction Assessment

Yes, source-level: terminal calls are persisted and removed from active indexes, while current status paths only query active manager lookups before returning `found:false`. I did not run a live telephony provider flow in this read-only review.

## Solution Assessment

No. The latest-snapshot store lookup is useful, but the fallback should be status-scoped instead of changing shared manager methods that other runtime paths use as active-call guards.
