## Summary

## What This Changes

The PR adds persisted-store fallback lookups for voice-call status read paths while keeping shared manager lookups active-only and projecting status responses through a redacted DTO.

## Best Possible Solution

Land a status-only persisted lookup fix that preserves active-only manager semantics and safe redacted status projection, with real behavior proof for completed-call status after eviction or restart.

## Reproduction Assessment

Yes, from source inspection: complete or restart past a voice call so active maps no longer contain it, then query status while persisted call history still has the record. I did not run a live provider call in this read-only review.

## Solution Assessment

Yes, with one proof gap: status-only fallback is the best boundary because it fixes the existing read surface without letting terminal persisted calls back into manager, webhook, realtime, or auto-response lookups. The broader manager-level fallback attempted elsewhere is riskier because those lookup methods are live-path inputs.
