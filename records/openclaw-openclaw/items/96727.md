## Summary

Keep open: the second commit fixes the stale-snapshot problem, but the PR still broadens active-call manager lookups into active-or-history lookups and only provides Vitest output as proof.

## What This Changes

This PR adds persisted voice-call store lookup helpers, wires `CallManager.getCall` and `getCallByProviderCallId` to fall back to that store, and adds restore tests for completed and multi-snapshot calls.

## Best Possible Solution

Keep `CallManager.getCall` and `getCallByProviderCallId` active-only, add newest-first persisted lookup only for the status/read surfaces, and require redacted runtime proof of the completed-call status path.

## Reproduction Assessment

Yes from source inspection: complete or restart past a call so active maps no longer contain it, then query status while plugin-state history still has the record. I did not run a live provider call in this read-only review.

## Solution Assessment

No. The newest-snapshot store helper is now on the right track, but the PR should scope persisted fallback to status reads instead of changing all manager lookups from active-only to active-or-history.
