## Summary

Keep open: latest head fixes the stale-snapshot issue, but it still broadens shared manager lookups from active-only to active-or-history, which can make terminal persisted calls visible to webhook/realtime paths that use those lookups as active-call guards. The proof is still focused Vitest output rather than a redacted runtime `get_status` flow after completion or restart.

## What This Changes

The PR adds persisted-store lookup helpers and changes `CallManager.getCall` and `getCallByProviderCallId` to fall back to persisted call records, with restore tests for completed and multi-snapshot calls.

## Best Possible Solution

Keep manager lookups active-only, add newest-first persisted lookup behind read-only status surfaces, and prove the completed-and-evicted status flow without changing active-call guards.

## Reproduction Assessment

Yes from source inspection: current status paths read active manager maps only, while terminal calls are persisted and then removed from active indexes. I did not run a live telephony provider flow in this read-only review.

## Solution Assessment

No. The PR fixes the symptom but does it at the shared manager lookup layer, which is too broad for callers that use those methods as active-call guards; a status-scoped persisted lookup is the safer fix.
