## Summary

Keep open: current main and the latest release still have the voice-call status miss, but this PR is not merge-ready because it fixes the bug by broadening shared manager lookups from active-only to active-or-history, which affects webhook and realtime callers that use those methods as active-call guards. The contributor proof is still Vitest/test-store output rather than a redacted completed-and-evicted or restarted `get_status` runtime flow.

## What This Changes

The PR adds persisted voice-call store lookup helpers, wires `CallManager.getCall` and `getCallByProviderCallId` to fall back to those helpers, and adds restore tests for completed and multi-snapshot calls.

## Best Possible Solution

Keep manager lookups active-only and add a newest-first persisted lookup only behind read-only status surfaces, then prove the completed-and-evicted status flow with redacted runtime evidence.

## Reproduction Assessment

Yes, source-level: terminal calls are persisted and removed from active indexes, while current status paths only query active manager lookups before returning `found:false`. I did not run a live telephony provider flow in this read-only review.

## Solution Assessment

No. The bug should be fixed at the status-read boundary; changing shared manager lookup methods is too broad for callers that rely on active-only semantics.
