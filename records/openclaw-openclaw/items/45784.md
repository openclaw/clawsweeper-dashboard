## Summary

Keep open: the path bug is real enough to continue reviewing, but this PR is not safe to merge because it changes only `resolveConfigDir`, splitting config/state behavior and moving trusted runtime roots for existing `OPENCLAW_HOME` service installs.

## What This Changes

The PR adds an `OPENCLAW_HOME` branch to `resolveConfigDir` that returns the resolved home directory itself before the existing `.openclaw` default.

## Best Possible Solution

Define a coordinated path contract that avoids `.openclaw/.openclaw` only for explicit homes that intentionally name the OpenClaw state directory, then update `resolveStateDir`, `resolveConfigDir`, focused tests, docs if needed, and the changelog together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR regression path: `OPENCLAW_HOME=/srv/openclaw-home` moves `resolveConfigDir` consumers to `/srv/openclaw-home` while `resolveStateDir` and config IO still use `/srv/openclaw-home/.openclaw`.

## Solution Assessment

No. Changing every explicit `OPENCLAW_HOME` in `resolveConfigDir` is broader than the reported double-nesting case and does not update the state/config resolver used by onboarding.
