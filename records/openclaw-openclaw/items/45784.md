## Summary

Keep open. The underlying OPENCLAW_HOME nesting bug is real, but this PR is not correct as-is because it changes all explicit OPENCLAW_HOME config roots and leaves the state-dir path still nesting.

## What This Changes

The PR adds an explicit OPENCLAW_HOME branch in `src/utils.ts` so `resolveConfigDir()` returns the resolved home directory directly before appending `.openclaw`.

## Best Possible Solution

Revise or replace the PR with a narrow basename guard in both `resolveStateDir()` and `resolveConfigDir()`, plus regression tests that fix `OPENCLAW_HOME=~/.openclaw` while preserving `/srv/openclaw-home/.openclaw` defaults.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=~/.openclaw` becomes the effective home and the current state/config helpers append `.openclaw` again; the PR branch also source-reproduces the new `/srv/openclaw-home` config-root regression.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly is broader than the documented home-override contract, and changing only `resolveConfigDir()` does not fix state/config paths that use `resolveStateDir()`.

## Review Findings
