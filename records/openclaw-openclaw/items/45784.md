## Summary

Keep open: the underlying OPENCLAW_HOME nesting bug is real, but this PR is not safe to merge as-is because it changes every explicit OPENCLAW_HOME value into the final config directory and only updates the config-dir helper, leaving the state-dir path unresolved.

## What This Changes

The PR adds an OPENCLAW_HOME branch in src/utils.ts so resolveConfigDir() returns the resolved home path directly whenever that env var is set.

## Best Possible Solution

Land a narrow basename guard in both resolveStateDir() and resolveConfigDir(), preserving parent-home semantics and adding regression coverage for .openclaw, legacy state dir names, tilde expansion, and the /srv/app -> /srv/app/.openclaw negative case.

## Reproduction Assessment

Yes, at source level. OPENCLAW_HOME=/home/user/.openclaw is preserved by the home resolver, then current main's state and config helpers append another .openclaw segment.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly is broader than the documented home-override contract, and changing only resolveConfigDir() leaves resolveStateDir() unresolved.
