## Summary

Keep open: the underlying `OPENCLAW_HOME=~/.openclaw` nesting bug is real, but this PR broadens `OPENCLAW_HOME` semantics for every explicit home value, leaves `resolveStateDir()` unresolved, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an explicit `OPENCLAW_HOME` branch to `resolveConfigDir()` that returns the resolved home path directly before appending `.openclaw`.

## Best Possible Solution

Land or replace this with a narrow basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()`, with regression tests for `.openclaw`, legacy state names, tilde expansion, and non-state-dir parent homes.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME=/home/user/.openclaw`, current main preserves that value as the effective home and then both state and config helpers append another `.openclaw` segment; I did not run onboarding in this read-only pass.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly changes the documented parent-home contract and changing only `resolveConfigDir()` leaves `resolveStateDir()` unresolved.
