## Summary

Keep open: the bug is real, but this PR is not merge-ready because it changes the `OPENCLAW_HOME` contract too broadly, leaves the state-dir path unresolved, is currently conflicting, and lacks required after-fix real behavior proof.

## What This Changes

This PR changes `resolveConfigDir()` so any explicit `OPENCLAW_HOME` is returned directly as the config directory before the normal `.openclaw` suffix logic runs.

## Best Possible Solution

Replace this with a narrow shared basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()`, preserving parent-home semantics and adding regression tests for state-dir names, tilde expansion, and parent homes.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME=/home/user/.openclaw`, the effective-home resolver preserves that path and both current state/config helpers append another `.openclaw` segment.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly changes the documented parent-home contract, and changing only `resolveConfigDir()` leaves `resolveStateDir()` and canonical config paths unresolved.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Preserve parent OPENCLAW_HOME defaults:** `src/utils.ts:296-307`
  - body: This branch returns `resolvedHome` for any explicit `OPENCLAW_HOME`.   - confidence: 0.94

Changing only `resolveConfigDir()` leaves the state path and canonical config path nested for `OPENCLAW_HOME` values that already end in `.openclaw`.
  - confidence: 0.91
