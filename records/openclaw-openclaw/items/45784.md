## Summary

Keep open: the reported bug is still source-reproducible on current main, but this PR is too broad and incomplete because it changes every explicit `OPENCLAW_HOME` into the final config directory and does not fix the state-dir path that drives canonical config placement.

## What This Changes

The PR changes `resolveConfigDir()` so any explicit `OPENCLAW_HOME` is returned directly as the config directory.

## Best Possible Solution

Use a narrow state-directory basename guard in both `resolveStateDir()` and `resolveConfigDir()`, with regression coverage for `.openclaw`, legacy state-dir names, tilde expansion, and the `/srv/app -> /srv/app/.openclaw` negative case.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` is preserved by the home resolver, then current main's state and config helpers append another `.openclaw`.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly is broader than the documented home-override contract, and changing only `resolveConfigDir()` leaves the state-dir path unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve .openclaw for parent OPENCLAW_HOME values:** `src/utils.ts:296-307`
  - body: This branch returns `OPENCLAW_HOME` itself for every explicit home override, so `OPENCLAW_HOME=/srv/openclaw-home` stops using `/srv/openclaw-home/.openclaw` for config-derived roots. That conflicts with current docs/tests and can move global `.env` and plugin roots to a different directory; only skip the suffix when the resolved basename already names a state directory.
  - confidence: 0.94

  - body: The reported onboarding/config placement still flows through `resolveStateDir()` and `resolveCanonicalConfigPath()`, but this PR only changes `resolveConfigDir()`. Add the same narrow guard in the state-dir helper with regression coverage, otherwise canonical state/config paths can still be nested.
  - confidence: 0.91
