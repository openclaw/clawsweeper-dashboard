## Summary

Keep open. The underlying OPENCLAW_HOME double-nesting bug is real and still present on current main and v2026.4.29, but this PR is not safe to merge because it redefines every explicit OPENCLAW_HOME as the config directory and only changes resolveConfigDir, splitting config-dir consumers from resolveStateDir and config I/O.

## What This Changes

The PR adds an explicit OPENCLAW_HOME branch in src/utils.ts so resolveConfigDir returns the resolved home directly instead of the default .openclaw child.

## Best Possible Solution

Revise or replace this PR with a coherent path-contract fix that avoids double-nesting only when an explicit home already names the OpenClaw state directory, keeps service homes under their .openclaw child, and adds focused regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. The PR diff itself provides a high-confidence reproduction: with OPENCLAW_HOME=/srv/openclaw-home, the added branch returns /srv/openclaw-home from resolveConfigDir while current docs/tests and resolveStateDir expect /srv/openclaw-home/.openclaw.

## Solution Assessment

No. Changing only resolveConfigDir and treating every explicit OPENCLAW_HOME as the config directory is broader than the bug; the safer solution is a coordinated, narrow compatibility rule across resolveStateDir and resolveConfigDir with regression tests.
