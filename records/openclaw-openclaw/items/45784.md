## Summary

Keep open: the reported path bug is real, but this PR is not safe to merge because it changes every explicit `OPENCLAW_HOME` value into the final config directory and leaves the state-dir path unresolved. It also lacks after-fix real behavior proof from an onboarding/config run.

## What This Changes

The PR adds a branch in `src/utils.ts` so `resolveConfigDir()` returns the resolved `OPENCLAW_HOME` directly whenever it is explicitly set.

## Best Possible Solution

Land a narrow state-directory basename guard in both `resolveStateDir()` and `resolveConfigDir()`, preserve parent-home semantics, and add regression coverage for `.openclaw`, legacy state dir names, tilde expansion, and the `/srv/app -> /srv/app/.openclaw` negative case.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` is preserved by the home resolver, then current main's state and config helpers append another `.openclaw`.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly is broader than the documented home-override contract, and changing only `resolveConfigDir()` leaves `resolveStateDir()` unresolved.

## Review Findings

Overall correctness: patch is incorrect
