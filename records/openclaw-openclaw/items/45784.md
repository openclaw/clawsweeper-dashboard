## Summary

Keep open. The PR targets a real, source-reproducible path bug, but the patch is not safe to merge because it broadens `OPENCLAW_HOME` semantics for parent-directory values, misses `resolveStateDir()`, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/utils.ts` so `resolveConfigDir()` returns any explicit `OPENCLAW_HOME` directly instead of deriving a `.openclaw` config directory under it.

## Best Possible Solution

Replace this patch with a narrow basename guard in both `resolveStateDir()` and `resolveConfigDir()`, plus regression tests that fix `OPENCLAW_HOME=~/.openclaw` while preserving `/srv/openclaw-home/.openclaw` parent-home behavior.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=~/.openclaw` becomes the effective home, then the current state/config helpers append `.openclaw` again; this read-only review did not run a live onboarding command.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly is broader than the documented home-override contract, and changing only `resolveConfigDir()` leaves state-derived config paths nested.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve .openclaw for parent OPENCLAW_HOME values:** `src/utils.ts:301-307`
  - body: This branch returns `OPENCLAW_HOME` itself for every explicit home override, so `OPENCLAW_HOME=/srv/openclaw-home` moves config, plugin, and `.env` lookup from `/srv/openclaw-home/.openclaw` to `/srv/openclaw-home`. That conflicts with current tests, docs, and `resolveStateDir()` semantics; the fix should only skip the suffix when the resolved basename is already a state directory.
  - confidence: 0.94
- **[P2] Update state-dir resolution as part of this fix:** `src/utils.ts:296-297`
  - body: The reported onboarding/config path still goes through `resolveStateDir()` in `src/config/paths.ts`, which this PR does not change. With `OPENCLAW_HOME=~/.openclaw`, state and canonical config paths still become `~/.openclaw/.openclaw/...`; add the same narrow guard there with regression coverage.
  - confidence: 0.91
