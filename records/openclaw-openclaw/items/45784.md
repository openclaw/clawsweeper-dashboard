## Summary

Keep open: this PR targets a real, source-reproducible `OPENCLAW_HOME` nesting bug, but the patch is not merge-ready because it changes every explicit `OPENCLAW_HOME` into the config directory, leaves `resolveStateDir()` unresolved, is currently dirty/conflicting, and lacks real behavior proof. The canonical bug thread at https://github.com/openclaw/openclaw/issues/45765 remains open and the broader related report at https://github.com/openclaw/openclaw/issues/54014 is still open too.

## What This Changes

The PR changes `resolveConfigDir()` so an explicit `OPENCLAW_HOME` is returned directly as the config directory instead of appending `.openclaw`.

## Best Possible Solution

Replace this with a narrow shared guard in both `resolveStateDir()` and `resolveConfigDir()` that only treats `OPENCLAW_HOME` as the state dir when its basename is a known state directory, preserves parent-home defaults, adds regression tests, and includes redacted real onboarding/config proof.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME=/home/user/.openclaw`, current main preserves that effective home and then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly changes the documented parent-home contract and changing only `resolveConfigDir()` leaves the state/canonical config path unresolved.
