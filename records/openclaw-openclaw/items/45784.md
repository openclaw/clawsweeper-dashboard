## Summary

This PR should stay open because it targets a real `OPENCLAW_HOME=~/.openclaw` nesting bug, but the branch changes the path contract too broadly and still lacks real after-fix behavior proof. The current review comments already point to the main blockers, and the canonical bug remains open at https://github.com/openclaw/openclaw/issues/45765.

## What This Changes

The PR changes `src/utils.ts` so `resolveConfigDir` returns any explicit `OPENCLAW_HOME` directly instead of appending `.openclaw`.

## Best Possible Solution

Replace this branch with a shared, narrow state-directory-name guard for config and state path helpers, preserving parent-home semantics and proving both fresh install and upgrade behavior.

## Reproduction Assessment

Yes, at source level. Current main appends `.openclaw` after `resolveRequiredHomeDir`, so `OPENCLAW_HOME` values that already end in `.openclaw` can produce the nested config/state shape described by the linked bug.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly fixes one case but changes the established parent-directory behavior and leaves state resolution on a different contract.
