## Summary

Keep open: the underlying nested `OPENCLAW_HOME=~/.openclaw` bug is real, but this PR is not mergeable because it broadens parent-home semantics, leaves the state-dir path unresolved, lacks after-fix real behavior proof, and is currently conflicting.

## What This Changes

The PR changes `resolveConfigDir()` so any explicit `OPENCLAW_HOME` is returned directly as the config directory.

## Best Possible Solution

Land or replace this with a narrow guard in both `resolveStateDir()` and `resolveConfigDir()` that only uses `OPENCLAW_HOME` directly when its basename is a known state directory, with regression tests for `.openclaw`, legacy names, tilde expansion, and non-state-dir parent homes.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME=/home/user/.openclaw`, current main preserves that value as the effective home and then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly changes the documented parent-home contract and changing only `resolveConfigDir()` leaves `resolveStateDir()` unresolved.
