## Summary

Keep open: the underlying OPENCLAW_HOME path problem is real and still tracked by related reports, but this PR is not mergeable because it moves every explicit home override to the config root and diverges from current state/config defaults, docs, and regression coverage.

## What This Changes

The branch adds an explicit OPENCLAW_HOME branch in resolveConfigDir that returns the resolved home directory itself before the .openclaw default.

## Best Possible Solution

Replace or revise this with a coordinated path-contract fix that avoids double nesting only for explicit homes that already name the OpenClaw state directory, preserves service-home defaults, and updates focused tests, docs if needed, and the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR regression path: with OPENCLAW_HOME=/srv/openclaw-home and no explicit state/config overrides, the patch returns /srv/openclaw-home from resolveConfigDir while current resolveStateDir and tests expect /srv/openclaw-home/.openclaw.

## Solution Assessment

No. Treating every explicit OPENCLAW_HOME as the config directory is broader than the reported double-nesting case; a safer fix should align resolveStateDir, resolveConfigDir, and their consumers around a narrow compatibility rule.
