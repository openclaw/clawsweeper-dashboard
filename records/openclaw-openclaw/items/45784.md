## Summary

Keep open: the reported path bug is real on current main, but this PR is too broad and incomplete because it changes every explicit `OPENCLAW_HOME` config directory while leaving `resolveStateDir()` unchanged and lacks after-fix real behavior proof.

## What This Changes

The PR changes `resolveConfigDir()` so any explicit `OPENCLAW_HOME` is returned as the config directory, avoiding an extra `.openclaw` suffix when that value already names `.openclaw`.

## Best Possible Solution

Land a narrow state-directory basename guard in both `resolveStateDir()` and `resolveConfigDir()`, preserve parent-home semantics, and add regression coverage for `.openclaw`, legacy state dir names, tilde expansion, and the `/srv/app -> /srv/app/.openclaw` negative case.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME=/home/user/.openclaw`, the home resolver preserves that path and current main's state/config helpers append another `.openclaw`.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly is broader than the documented home-override contract, and changing only `resolveConfigDir()` leaves the state-dir path unresolved.

## Review Findings

  - confidence: 0.91
