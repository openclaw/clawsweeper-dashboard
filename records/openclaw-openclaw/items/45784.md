## Label Justifications

- `P2`: This is a focused path-resolution bug fix with limited blast radius, but it affects state/config location for users who set OPENCLAW_HOME.
- `merge-risk: 🚨 compatibility`: Merging this patch would change existing parent OPENCLAW_HOME deployments from `$OPENCLAW_HOME/.openclaw` to `$OPENCLAW_HOME` for config-derived paths.
- `merge-risk: 🚨 auth-provider`: Global `.env` loading can move provider credentials and auth-related environment from the existing state directory to a different root.
- `merge-risk: 🚨 security-boundary`: The patch changes which trusted `.env` files and plugin roots OpenClaw reads, which is a sensitive path-trust boundary.

## Summary

Keep open: the bug being targeted is real, but this patch changes the existing parent-directory OPENCLAW_HOME contract, leaves state-dir resolution unresolved, has no real behavior proof, and the live branch is currently conflicting.

## What This Changes

The PR changes `src/utils.ts` so `resolveConfigDir()` returns an explicit `OPENCLAW_HOME` directly instead of appending `.openclaw`.

## Best Possible Solution

Replace this branch with a narrow known-state-dir basename guard shared by `resolveStateDir()` and `resolveConfigDir()`, preserving parent-home semantics with regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME=/home/user/.openclaw`, the effective-home resolver preserves that value and current state/config helpers append another `.openclaw` segment.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly changes the documented parent-home contract, and changing only `resolveConfigDir()` leaves `resolveStateDir()` unresolved.
