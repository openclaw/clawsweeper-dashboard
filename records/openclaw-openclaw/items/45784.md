## Summary

Keep open. The reported nested-path bug is real, but this PR is not safe to merge because it changes only `resolveConfigDir`, which does not drive the active `openclaw.json` path, and it would also move existing explicit-home installs away from the documented `$OPENCLAW_HOME/.openclaw` layout.

## What This Changes

The PR adds an explicit `OPENCLAW_HOME` branch to `resolveConfigDir` that returns the resolved home directory directly instead of appending `.openclaw`.

## Best Possible Solution

Use a narrow state-dir basename guard in both `resolveStateDir` and `resolveConfigDir`, preserving `/srv/home -> /srv/home/.openclaw` behavior, and add regression tests for both the `~/.openclaw` positive case and a non-state-dir negative case.

## Reproduction Assessment

Yes, at source level. Current main resolves `OPENCLAW_HOME=~/.openclaw` as the effective home and then both `resolveStateDir` and `resolveConfigDir` append `.openclaw`, but I did not run the onboarding flow because this review is read-only.

## Solution Assessment

No. This PR is not the best fix because it changes only `resolveConfigDir` and applies a broad semantic change; the safer fix is a narrow basename guard shared by the state and config helpers with regression coverage.

That contradicts the current docs/tests that treat `OPENCLAW_HOME` as a replacement home, not always the final state directory.
