## Summary

Keep open. The reported bug is real, but this patch is broader than the documented path contract, leaves the state-dir path unresolved, and still lacks contributor real-behavior proof.

## What This Changes

This PR adds a `src/utils.ts` branch that makes `resolveConfigDir()` return the resolved `OPENCLAW_HOME` directly whenever it is set.

## Best Possible Solution

Replace this patch with a narrow basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()`, plus regression coverage for an explicit state-directory home and for parent-home preservation.

## Reproduction Assessment

Yes, at source level. If `OPENCLAW_HOME` already names the `.openclaw` state directory, current main resolves that as the effective home and then appends another `.openclaw` in both state/config helpers.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly is broader than the docs and tests allow, and changing only `resolveConfigDir()` leaves the state-dir path that drives canonical config placement unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve .openclaw for parent OPENCLAW_HOME values:** `src/utils.ts:296-307`
  - body: This branch returns `OPENCLAW_HOME` itself for every explicit home override, so a dedicated service parent directory stops using its `.openclaw` child for config, plugin roots, and global `.env` lookup. That conflicts with the current tests, docs, and `resolveStateDir()` semantics; the guard should only skip the suffix when the resolved basename is already a state directory.
  - confidence: 0.94
- **[P2] Fix state-dir resolution with the config-dir change:** `src/utils.ts:296-297`
  - body: The reported onboarding/config path still flows through `resolveStateDir()` and `resolveCanonicalConfigPath()`, but this PR only changes `resolveConfigDir()`. Add the same narrow guard in the state-dir helper with regression coverage, otherwise state-derived config paths can still be nested.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: Needs attention: the broad path change also moves trusted global `.env` lookup to the `OPENCLAW_HOME` root for parent-home setups.

Concerns:

- **[low] Keep trusted .env lookup inside the state directory:** `src/utils.ts:296`
  - body: `loadGlobalRuntimeDotEnvFiles()` builds the global `.env` path from `resolveConfigDir()`. Returning the parent home directly can make OpenClaw ingest a different `.env` file than existing installs intended, so the path fix should preserve the `.openclaw` child except when the home already names a state directory.
