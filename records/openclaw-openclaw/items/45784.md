## Summary

Keep this PR open, but it is not merge-ready: it targets a real OPENCLAW_HOME nesting bug, while the patch broadens existing parent-home semantics, leaves state-dir resolution unresolved, lacks real behavior proof, and currently conflicts with main.

## What This Changes

The PR adds a resolveConfigDir branch that returns the resolved OPENCLAW_HOME directly whenever OPENCLAW_HOME is explicitly set.

## Best Possible Solution

Replace this branch with a narrow known-state-dir basename guard shared by resolveStateDir and resolveConfigDir, preserving parent-home semantics with regression tests and real onboarding/config proof.

## Reproduction Assessment

Yes, at source level. With OPENCLAW_HOME set to a path ending in .openclaw, current main preserves that as the effective home and then both state and config helpers append another .openclaw segment.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly changes the documented parent-home contract, and changing only resolveConfigDir leaves resolveStateDir unresolved.
