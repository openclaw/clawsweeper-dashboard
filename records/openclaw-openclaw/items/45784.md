## Summary

Keep open. The underlying OPENCLAW_HOME nesting bug is valid, but this PR is not safe as-is because it returns every explicit OPENCLAW_HOME value as the config root, only changes `resolveConfigDir()`, and conflicts with current state/config path semantics and tests; external real behavior proof is also missing.

## What This Changes

The PR adds an OPENCLAW_HOME branch in `resolveConfigDir()` that returns the resolved home directory directly before the default `.openclaw` suffix logic.

## Best Possible Solution

Replace this broad config-only branch with a narrow basename guard shared by `resolveStateDir()` and `resolveConfigDir()`, preserving `/srv/app -> /srv/app/.openclaw` while avoiding `.openclaw/.openclaw` when the explicit home already names a state directory.

## Reproduction Assessment

Yes. Source tracing on current main is enough: `OPENCLAW_HOME=/home/user/.openclaw` becomes the effective home, then both state and config helpers append another `.openclaw` when direct overrides are unset.

## Solution Assessment

No. The current PR is broader than the bug and only patches `resolveConfigDir()`; the maintainable fix is a narrow state-dir-name guard in both affected helpers with regression coverage for the positive and negative cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve .openclaw defaults for general OPENCLAW_HOME values:** `src/utils.ts:298-307`
  - body: This branch returns `OPENCLAW_HOME` itself for every explicit value, so `OPENCLAW_HOME=/srv/openclaw-home` resolves config paths under `/srv/openclaw-home` instead of the existing `/srv/openclaw-home/.openclaw` contract. That diverges from `resolveStateDir()` and current tests, and moves `.env` loading plus global plugin roots away from existing installs. Please only skip the suffix when the resolved home already names a state directory, and apply the same guard to the state helper.
  - confidence: 0.93
