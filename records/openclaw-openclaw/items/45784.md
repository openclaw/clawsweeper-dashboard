## Label Justifications

- `P2`: This is a focused path-resolution bug fix with limited blast radius, but it affects state/config location for users who set `OPENCLAW_HOME`.
- `merge-risk: 🚨 compatibility`: Merging this patch would change existing parent `OPENCLAW_HOME` deployments from `$OPENCLAW_HOME/.openclaw` to `$OPENCLAW_HOME` for config-derived paths.
- `merge-risk: 🚨 auth-provider`: Global `.env` loading can move provider credentials and auth-related environment from the existing state directory to a different root.
- `merge-risk: 🚨 security-boundary`: The patch changes which trusted `.env` files and plugin roots OpenClaw reads, which is a sensitive path-trust boundary.

## Summary

Keep open: the reported bug is valid, but this PR is not merge-ready because it changes all explicit `OPENCLAW_HOME` config roots instead of only the already-state-dir case, leaves state-dir resolution unresolved, and lacks real behavior proof.

## What This Changes

The PR adds a `resolveConfigDir()` branch that returns the resolved `OPENCLAW_HOME` directly whenever it is explicitly set.

## Best Possible Solution

Replace this branch with a narrow basename/state-dir guard shared by `resolveStateDir()` and `resolveConfigDir()`, preserving parent-home semantics and adding regression tests plus real onboarding/config proof.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set to a path ending in `.openclaw`, current main preserves that value as the effective home and then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly changes the documented parent-home contract, and changing only `resolveConfigDir()` leaves `resolveStateDir()` unresolved.
