## Summary

Keep open, but this PR is not mergeable as-is. The reported nested-path bug is real and source-reproducible, yet this branch only changes `resolveConfigDir()` and also changes every explicit `OPENCLAW_HOME` from a home replacement into the final config directory, which conflicts with current state-dir behavior and tests.

## What This Changes

The branch adds an explicit `OPENCLAW_HOME` branch in `src/utils.ts` so `resolveConfigDir()` returns the explicit home path directly instead of appending `.openclaw`.

## Best Possible Solution

Revise or replace the PR with a narrow basename guard shared by `resolveStateDir()` and `resolveConfigDir()`, preserving `/srv/openclaw-home -> /srv/openclaw-home/.openclaw`, adding regression tests, and posting redacted real run proof.

## Reproduction Assessment

Yes. Source tracing on current main shows `OPENCLAW_HOME=~/.openclaw` is treated as the effective home and then the state/config helpers append `.openclaw`; I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR is too broad for general `OPENCLAW_HOME` values and incomplete for the onboard path because it leaves `resolveStateDir()` unchanged; the safer fix is a targeted guard in both affected helpers.
