## Summary

Keep open: the PR targets a real source-reproducible path bug, but the patch is not safe to merge because it changes parent `OPENCLAW_HOME` semantics, leaves state resolution unchanged, lacks real behavior proof, and currently conflicts with main.

## What This Changes

The PR changes `resolveConfigDir` so any explicit `OPENCLAW_HOME` is returned directly as the config directory instead of appending `.openclaw`.

## Best Possible Solution

Replace this branch with a narrow known-state-directory basename guard shared by `resolveConfigDir` and `resolveStateDir`, preserving parent-home semantics with regression tests and real onboarding/config proof.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set to a path ending in `.openclaw` and no direct state/config override, `resolveRequiredHomeDir` preserves that path and both current config/state helpers append another `.openclaw` segment.

## Solution Assessment

No. Returning every explicit `OPENCLAW_HOME` directly is broader than the reported bug and leaves `resolveStateDir` unresolved, so the maintainable fix is a shared narrow guard for already-state-directory homes.
