## Summary

Keep open. The linked path problem is worth resolving, but this PR is not mergeable as-is because it changes the `OPENCLAW_HOME` contract for every explicit home root while current main, docs, and regression tests still treat `OPENCLAW_HOME` as a home directory whose default state/config live under `.openclaw`.

## What This Changes

The PR adds an `OPENCLAW_HOME` branch in `src/utils.ts` so `resolveConfigDir` returns the resolved home directory itself instead of the default `.openclaw` child.

## Best Possible Solution

Revise or replace this with a coordinated path-contract fix that avoids double nesting only when an explicit home already names the OpenClaw state directory, preserves service homes under their `.openclaw` child, and adds focused regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `OPENCLAW_HOME=/srv/openclaw-home` and no state/config overrides, the PR returns `/srv/openclaw-home` from `resolveConfigDir` while current `resolveStateDir` and tests expect `/srv/openclaw-home/.openclaw`.

## Solution Assessment

No. Treating every explicit `OPENCLAW_HOME` as the config directory is broader than the reported double-nesting case; a safer fix should align `resolveStateDir`, config path resolution, and config-dir consumers around a narrow compatibility rule.
