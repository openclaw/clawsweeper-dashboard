## Summary

Keep open: the PR targets a real OPENCLAW_HOME nesting bug, but the patch is not merge-ready because it changes parent-home semantics, leaves state-dir behavior unresolved, and lacks real behavior proof.

## What This Changes

The PR changes `src/utils.ts` so `resolveConfigDir` returns an explicit `OPENCLAW_HOME` directly instead of appending `.openclaw`.

## Best Possible Solution

Replace this branch with a narrow known-state-directory basename guard shared by `resolveConfigDir` and `resolveStateDir`, preserving parent-home semantics with regression tests and real onboarding/config proof.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set to a path ending in `.openclaw` and no direct state/config override, `resolveRequiredHomeDir` preserves that path and both current config/state helpers append another `.openclaw` segment.

## Solution Assessment

No, not as written. Returning every explicit `OPENCLAW_HOME` directly is broader than the reported bug and leaves `resolveStateDir` unresolved, so the maintainable fix is a shared narrow guard for already-state-directory homes.
