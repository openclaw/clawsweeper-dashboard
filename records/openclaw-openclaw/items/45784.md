## Label Justifications

- `P2`: This is a focused path-resolution bug fix with limited blast radius, but it affects state/config location for users who set OPENCLAW_HOME.
- `merge-risk: 🚨 compatibility`: Merging this patch would change existing parent OPENCLAW_HOME deployments from using $OPENCLAW_HOME/.openclaw to using $OPENCLAW_HOME directly.
- `merge-risk: 🚨 auth-provider`: Config-derived .env loading can move provider credentials and auth-related environment from the existing state directory to a different root.
- `merge-risk: 🚨 security-boundary`: The patch changes which trusted .env files and plugin roots OpenClaw reads, which is a sensitive path-trust boundary.

## Summary

Keep open: the bug is real, but this branch is not mergeable because it broadens the documented OPENCLAW_HOME contract, leaves the state-dir path unresolved, is currently conflicting, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an explicit OPENCLAW_HOME branch to resolveConfigDir() that returns the resolved home path directly as the config directory.

## Best Possible Solution

Replace this branch with a narrow basename/state-dir guard in both resolveStateDir() and resolveConfigDir(), preserving parent-home semantics and adding regression tests plus real onboarding/config proof.

## Reproduction Assessment

Yes, at source level. With OPENCLAW_HOME set to a path ending in .openclaw, current main preserves that value as the effective home and then both state and config helpers append another .openclaw segment.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly changes the documented parent-home contract, and changing only resolveConfigDir() leaves resolveStateDir() unresolved.
