## Summary

Keep open. Current main still has the OPENCLAW_HOME double-nesting behavior for the state/config path, but this PR is unsafe as written because it changes every explicit OPENCLAW_HOME from a home root into the config directory itself and only updates resolveConfigDir, leaving state/config I/O on the old path contract.

## What This Changes

The PR adds an explicit OPENCLAW_HOME branch in src/utils.ts so resolveConfigDir returns OPENCLAW_HOME directly instead of appending .openclaw.

## Best Possible Solution

Keep this PR open for revision or maintainer consolidation with the related OPENCLAW_HOME reports. The durable fix should update the state/config path contract coherently, avoid double-nesting only when the explicit value already names the OpenClaw state directory, preserve service-home behavior, and add focused regression tests.

## Reproduction Assessment

Yes. The PR diff itself provides a high-confidence reproduction: with OPENCLAW_HOME=/srv/openclaw-home, the added branch returns /srv/openclaw-home from resolveConfigDir while current docs/tests and resolveStateDir expect /srv/openclaw-home/.openclaw.

## Solution Assessment

No. Changing only resolveConfigDir and treating every explicit OPENCLAW_HOME as the config directory is broader than the bug; the safer solution is a coordinated, narrow compatibility rule across resolveStateDir and resolveConfigDir with regression tests.
