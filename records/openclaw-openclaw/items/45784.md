## Summary

Keep open: the reported OPENCLAW_HOME nesting bug is real on current main, but this PR is too broad and incomplete because it changes parent-home semantics and only patches resolveConfigDir(). It also still lacks after-fix real behavior proof.

## What This Changes

The PR changes resolveConfigDir() so any explicit OPENCLAW_HOME is returned directly as the config directory.

## Best Possible Solution

Land a narrow basename/state-dir guard in both resolveStateDir() and resolveConfigDir(), preserving parent-home behavior and adding regression coverage for .openclaw, legacy names, tilde expansion, and non-state-dir negative cases.

## Reproduction Assessment

Yes. Source inspection of current main shows OPENCLAW_HOME=/home/user/.openclaw is preserved by the home resolver, then both state and config helpers append another .openclaw segment.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly changes the documented parent-home contract and changing only resolveConfigDir() leaves resolveStateDir() unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep parent OPENCLAW_HOME values under .openclaw:** `src/utils.ts:296-307`
  - body: This branch returns OPENCLAW_HOME itself for every explicit home override, so OPENCLAW_HOME=/srv/openclaw-home changes config-derived directories from /srv/openclaw-home/.openclaw to /srv/openclaw-home. Current docs/tests still define OPENCLAW_HOME as a home replacement, and resolveConfigDir() consumers such as global .env and plugin roots will stop reading existing state; only skip the suffix when the basename already names a state directory.
  - confidence: 0.94
- **[P2] Fix state-dir resolution too:** `src/utils.ts:296-297`
  - body: The reported onboarding/config placement also flows through resolveStateDir() and resolveCanonicalConfigPath(), but this PR only changes resolveConfigDir(). With OPENCLAW_HOME already ending in .openclaw, canonical state/config paths can still nest unless the same narrow guard is added in src/config/paths.ts with regression coverage.
  - confidence: 0.91
