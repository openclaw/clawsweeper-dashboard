## Summary

Keep open: the underlying OPENCLAW_HOME nesting bug is real, but this PR is not merge-ready because it broadens the documented home-dir contract, leaves the state-dir path unresolved, is currently conflicting, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an explicit OPENCLAW_HOME branch in src/utils.ts so resolveConfigDir() returns the resolved home directory directly.

## Best Possible Solution

Replace this with a narrow basename/state-dir guard shared by resolveStateDir() and resolveConfigDir(), with regression coverage for .openclaw, legacy state names, tilde expansion, and parent-home cases.

## Reproduction Assessment

Yes, at source level. With OPENCLAW_HOME=/home/user/.openclaw and no direct state/config overrides, current main preserves that value as the effective home and both state/config helpers append another .openclaw segment.

## Solution Assessment

No. Returning every explicit OPENCLAW_HOME directly breaks the parent-home contract and only changes resolveConfigDir(); a narrow state-dir-basename guard in both helpers is the safer fix.
