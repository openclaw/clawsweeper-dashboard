## Summary

Keep open. The PR targets a real source-reproducible OPENCLAW_HOME nesting bug, but the current branch is not merge-ready because it returns every explicit OPENCLAW_HOME directly, changes existing parent-home behavior, leaves state resolution inconsistent, and still lacks real behavior proof.

## What This Changes

The PR changes `resolveConfigDir` so any explicit `OPENCLAW_HOME` is used directly as the config directory instead of appending `.openclaw`.

## Best Possible Solution

Replace the broad direct-return rule with a shared narrow guard that uses `OPENCLAW_HOME` directly only when its basename already names a known OpenClaw state directory, and apply it consistently to config and state path helpers with regression tests.

## Reproduction Assessment

Yes, at source level. Current main preserves explicit `OPENCLAW_HOME` in the effective-home resolver, then config and state helpers append `.openclaw`, so `OPENCLAW_HOME=/home/user/.openclaw` can resolve to `/home/user/.openclaw/.openclaw`.

## Solution Assessment

No for this PR as written. Returning every explicit `OPENCLAW_HOME` directly fixes one case but breaks the existing parent-directory contract and leaves state resolution on a different rule.
