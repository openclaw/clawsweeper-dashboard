## Label Justifications

- `merge-risk: 🚨 compatibility`: The PR changes a provider-controlled catalog read from accepting any successful response size to refusing responses above 16 MiB, which can affect future OpenRouter dynamic capability metadata.

## Summary

Keep open: current main still reads the OpenRouter runtime model catalog success body with unbounded `response.json()`, while this PR applies the existing bounded-reader pattern with focused regression coverage and no line-level findings. The remaining merge decision is maintainer acceptance or tuning of the new 16 MiB fail-closed ceiling.

## What This Changes

The PR replaces the OpenRouter runtime model catalog `response.json()` read with a 16 MiB `readResponseWithLimit` read and adds streamed overflow plus SQLite cache regression tests.

## Best Possible Solution

Land the focused runtime bounded-read fix once maintainers accept or tune the 16 MiB cap, and keep the separate model-scan hardening in https://github.com/openclaw/openclaw/pull/95418 separate.

## Reproduction Assessment

Yes, source-reproducible. Current main reaches an unbounded `response.json()` on the OpenRouter runtime catalog success body, and the PR body plus proof checks show bounded cancellation after the patch.

## Solution Assessment

Yes, with a cap-size tradeoff. Reusing the existing bounded-response helper at the runtime catalog read boundary is the narrow maintainable fix; the cap value is the maintainer acceptance point.
