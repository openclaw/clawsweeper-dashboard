## Label Justifications

- `P2`: This is normal-priority agent resource and security hardening for a provider catalog fetch with limited runtime blast radius.
- `merge-risk: 🚨 compatibility`: The PR changes a provider-controlled catalog read from accepting any successful response size to refusing responses above 16 MiB, which can affect future OpenRouter dynamic capability metadata.

## Summary

This PR is still necessary: current main and the latest release still read the embedded OpenRouter capability catalog with unbounded `response.json()`, while the PR applies the existing bounded-reader pattern and adds focused regression coverage. I found no line-level correctness or security findings; the remaining maintainer question is whether the new 16 MiB fail-closed cap is the intended compatibility ceiling.

## What This Changes

The PR replaces the embedded OpenRouter capability catalog success-body `response.json()` with a 16 MiB `readResponseWithLimit` parse and adds streamed overflow plus SQLite cache regression tests.

## Best Possible Solution

Land the focused bounded-read fix once maintainers accept or tune the 16 MiB cap, while keeping the separate model-scan hardening in https://github.com/openclaw/openclaw/pull/95418 on its own path.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.9 reach an unbounded `response.json()` on the OpenRouter runtime catalog success body, and the PR body provides terminal proof for bounded cancellation after the patch.

## Solution Assessment

Yes, with a cap-size tradeoff. Reusing the existing bounded-response helper at the runtime catalog read boundary is the narrow maintainable fix; an ad hoc reader would duplicate the helper already used by the sibling pricing path.
