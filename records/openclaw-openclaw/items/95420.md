## Summary

Keep open for maintainer review: the patch is a focused hardening fix with sufficient terminal proof and no line-level findings, but the new 16 MiB OpenRouter catalog cap is a compatibility tradeoff maintainers should accept before merge.

## What This Changes

The PR changes the OpenRouter embedded runtime model-capability catalog fetch to read the `/models` success body through a 16 MiB bounded helper and adds streamed overflow plus SQLite cache regression tests.

## Best Possible Solution

Land the focused runtime bounded-read fix after maintainers accept the 16 MiB catalog cap, and keep the separate model-scan hardening in https://github.com/openclaw/openclaw/pull/95418 separate.

## Reproduction Assessment

Yes. Current main and v2026.6.9 reach an unbounded `response.json()` on the OpenRouter runtime catalog success body, and the PR body includes a negative control plus after-fix terminal output showing bounded cancellation.

## Solution Assessment

Yes, with one explicit merge tradeoff. Reusing `readResponseWithLimit` at the runtime catalog boundary is the narrow maintainable fix; the 16 MiB cap size is the maintainer acceptance point.
