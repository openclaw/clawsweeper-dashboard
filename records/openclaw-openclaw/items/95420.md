## Summary

Keep open for maintainer review: current main still uses an unbounded successful `response.json()` read for the OpenRouter runtime model-capability catalog, and this PR is a focused, review-clean bounded-read fix with convincing terminal proof. The same-endpoint CLI scan hardening is now tracked separately by https://github.com/openclaw/openclaw/pull/95418, so it does not need to be folded into this PR.

## What This Changes

The PR routes the OpenRouter runtime model-capability catalog success body through `readResponseWithLimit` with a 16 MiB cap and adds streamed overflow plus SQLite cache regression tests.

## Best Possible Solution

Land this focused runtime bounded-read hardening after normal exact-head checks, while reviewing the separate CLI and SDK catalog-bound PRs on their own surfaces.

## Reproduction Assessment

Yes. Current main has a direct `response.json()` success-body read that can be driven with an oversized injected stream, and the PR body includes negative-control plus after-fix terminal output showing bounded cancellation.

## Solution Assessment

Yes for the runtime capability path. Reusing the existing bounded-response helper at the catalog read boundary is the narrow maintainable fix; same-class CLI and SDK catalog hardening is already split into separate PRs.
