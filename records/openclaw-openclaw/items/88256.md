## Summary

Keep open: current main and the latest release still use the reported Telegram polling restart policy and there is no stop-timeout circuit breaker. The related open sendChatAction issue overlaps operationally, but it does not own this narrower polling restart backoff and stop-dedup work.

## What This Changes

The issue requests hardening Telegram long-polling restart behavior so network instability does not produce rapid restart cycles or repeated stop attempts.

## Best Possible Solution

Add focused Telegram polling restart hardening: longer bounded backoff, per-cycle stop dedupe, stop-timeout burst cooldown, healthy-poll reset, and fake-timer regression coverage for both storms and normal recovery.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection confirms the reported 2s/30s restart policy and stop-timeout path, and the next proof should add a fake-timer regression for consecutive polling failures.

## Solution Assessment

Yes, hardening the existing Telegram polling restart path is the right boundary. The exact constants should be validated with tests rather than only raising maxMs, because normal short outage recovery still matters.
