## Summary

Keep open: the latest PR patch still has narrow correctness blockers in self-chat suppression, fallback send bookkeeping, core/plugin ownership, and required changelog coverage.

## What This Changes

The PR updates BlueBubbles monitor, send, probe, and status handling to suppress self-chat echoes, surface Private API helper disconnects, and add an `imsg` fallback for helper-disconnected sends.

## Best Possible Solution

Keep the helper-disconnect/probe direction, but preserve the assistant-outbound self-chat guard, make fallback sends non-actionable for reply-cache purposes, keep BlueBubbles status policy plugin-owned or behind a generic seam, and add a user-facing changelog entry.

## Reproduction Assessment

Yes, for the PR-introduced regressions: source inspection shows the patch drops the confirmed-assistant guard, returns `imsg-fallback` as an actionable send id, and adds BlueBubbles-specific fallback status parsing to core. I did not run tests under the read-only constraint.

## Solution Assessment

No: the PR addresses real BlueBubbles reliability problems, but this merge shape would regress user-authored self-chat prompts, poison outbound id bookkeeping, and duplicate plugin-owned status policy in core.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve user-authored self-chat prompts:** `extensions/bluebubbles/src/monitor-processing.ts:878-886`
  - body: This broadens self-chat duplicate caching from confirmed assistant outbounds to every `fromMe` self-chat copy. Current main explicitly keeps a user-authored self-chat prompt dispatchable when there is no confirmed assistant outbound, so this would drop valid prompts before they reach the agent.
  - confidence: 0.86
- **[P2] Do not cache fallback sends as real message IDs:** `extensions/bluebubbles/src/send.ts:187`
  - body: The fallback returns `messageId: "imsg-fallback"`, but the monitor caches every send id except `ok` and `unknown` as a real BlueBubbles id. A successful fallback would create fake reply/reaction/edit context that BlueBubbles cannot resolve; return a non-actionable result or mark it so cache announcement is skipped.
  - confidence: 0.91
- **[P2] Keep BlueBubbles status policy out of core:** `src/infra/channels-status-issues.ts:18-36`
  - body: This adds BlueBubbles-specific probe parsing and warning copy to the shared status collector even though the current design delegates status issue collection to plugin collectors and BlueBubbles already has one. Route this through a plugin-owned or generic unloaded-plugin seam so core does not drift from `extensions/bluebubbles/src/status-issues.ts`.
  - confidence: 0.8
- **[P3] Add the user-facing changelog entry:** `extensions/bluebubbles/src/send.ts:703`
  - body: The PR changes user-visible BlueBubbles send, probe, and status behavior but does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix changes to be recorded in the active Fixes section.
  - confidence: 0.96
