## Summary

Keep open: the PR addresses plausible BlueBubbles bugs, but the fallback path currently publishes a synthetic message ID that downstream code treats as a real BlueBubbles GUID, and it also needs boundary/changelog cleanup before merge.

## What This Changes

The PR changes BlueBubbles monitor/send/probe/status handling to suppress self-chat echoes, report Private API helper disconnects, and fall back to `imsg` when helper-backed sends fail.

## Best Possible Solution

Keep the PR open, make fallback sends non-actionable for BlueBubbles message-ID caching, keep helper-disconnect issue collection behind plugin-owned or generic status seams, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes, for the PR-introduced regression: source inspection shows the fallback returns `imsg-fallback`, and current main's monitor caches any non-`ok`/`unknown` result as an actionable BlueBubbles message ID. I did not run tests because this review was read-only.

## Solution Assessment

No: the self-chat and helper-status direction looks useful, but the fallback result must not be exposed as a real BlueBubbles message ID, and plugin-specific status policy should stay plugin-owned or use a generic seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not cache fallback sends as real message IDs:** `extensions/bluebubbles/src/send.ts:187`
  - body: The fallback returns `messageId: "imsg-fallback"`, but the monitor treats every result other than `ok`/`unknown` as a real BlueBubbles GUID and caches/announces it for later replies, reactions, and edits. A successful fallback would give the agent a short ID for a message BlueBubbles cannot act on, and repeated fallback sends reuse the same fake ID. Return a non-actionable result or add metadata so this path skips `maybeEnqueueOutboundMessageId`.
  - confidence: 0.91
- **[P2] Keep plugin-specific status policy out of core:** `src/infra/channels-status-issues.ts:17`
  - body: This adds BlueBubbles-specific parsing and warning text to `src/infra/channels-status-issues.ts`, while current core aggregation delegates channel policy to plugin collectors. Duplicating the BlueBubbles helper warning in core makes the shared status layer own plugin behavior and can drift from `extensions/bluebubbles/src/status-issues.ts`; route this through the plugin contract or a generic unloaded-plugin seam instead.
  - confidence: 0.78
- **[P3] Add the user-facing changelog entry:** `extensions/bluebubbles/src/send.ts:610`
  - body: The PR changes user-visible BlueBubbles send, monitor, and status behavior, but the file list has no `CHANGELOG.md` entry. OpenClaw policy requires user-facing fix changes to be recorded, so add a single-line active-version Fixes entry without inventing bot-only attribution.
  - confidence: 0.96
