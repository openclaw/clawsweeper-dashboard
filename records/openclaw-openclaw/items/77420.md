## Summary

Keep open: the PR targets useful BlueBubbles fixes, but the current branch still introduces blocking regressions in self-chat prompt handling, fallback send-ID caching, and core/plugin status ownership, plus a missing user-facing changelog entry.

## What This Changes

The PR changes BlueBubbles monitoring, sending, probing, and status issue collection to suppress self-chat reflections, report helper disconnects, and fall back to the local `imsg` CLI for some disconnected-helper sends.

## Best Possible Solution

Keep the useful helper-disconnect direction, but land it with fallback sends marked non-actionable, the confirmed-assistant self-chat guard preserved, status warnings routed through plugin-owned or generic seams, and changelog coverage added.

## Reproduction Assessment

Yes, for the PR-introduced regressions: source inspection shows the self-chat guard removal, the synthetic fallback ID flowing into current outbound-ID caching, and core gaining BlueBubbles-specific status policy. I did not run tests under the read-only review constraint.

## Solution Assessment

No: the PR is directionally useful, but this is not the best merge shape until the fallback is non-actionable, user-authored self-chat prompts stay dispatchable, status ownership remains plugin/generic, and the changelog is updated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve user-authored self-chat prompts:** `extensions/bluebubbles/src/monitor-processing.ts:878-879`
  - body: This records every self-chat `fromMe` row as a duplicate marker. Current main explicitly keeps a user-authored self-chat prompt dispatchable when no assistant outbound was confirmed, so the reflected inbound copy can be dropped before it reaches the agent.
  - confidence: 0.86
- **[P2] Do not cache fallback sends as real message IDs:** `extensions/bluebubbles/src/send.ts:187`
  - body: The fallback returns `messageId: "imsg-fallback"`, but the monitor treats non-`ok`/`unknown` IDs as actionable BlueBubbles GUIDs. Successful fallback sends would cache and announce the same fake ID for replies, reactions, and edits that BlueBubbles cannot resolve.
  - confidence: 0.91
- **[P2] Keep BlueBubbles status policy plugin-owned:** `src/infra/channels-status-issues.ts:17-37`
  - body: This adds BlueBubbles-specific payload parsing and warning text to the shared status collector even though current main delegates status policy to plugin collectors and BlueBubbles already has one. Move this behind the plugin contract or a generic unloaded-plugin seam.
  - confidence: 0.8
- **[P3] Add the user-facing changelog entry:** `extensions/bluebubbles/src/send.ts:706-717`
  - body: The PR changes BlueBubbles send, monitor, probe, and status behavior, but the diff has no `CHANGELOG.md` entry. OpenClaw policy requires user-facing fix changes to be recorded in the active Fixes section.
  - confidence: 0.96
