## Summary

Keep this PR open: the latest head still has blocking correctness issues around synthetic BlueBubbles message IDs, broadened self-chat suppression, plugin-specific status policy in core, and missing changelog coverage.

## What This Changes

The PR changes BlueBubbles monitor, send, probe, and status handling to suppress self-chat echoes, report Private API helper disconnects, and use `imsg` as a helper-disconnected fallback.

## Best Possible Solution

Keep the useful helper-disconnect and self-chat direction, but make fallback sends non-actionable, preserve the current self-chat prompt guard, keep status policy behind plugin-owned or generic seams, and document the user-visible fix.

## Reproduction Assessment

Yes, for the PR-introduced regressions: source inspection shows the fallback returns `imsg-fallback` and current main caches non-`ok`/`unknown` send IDs, and current self-chat tests show prompts without confirmed assistant outbound should still dispatch. I did not execute tests under the read-only constraint.

## Solution Assessment

No: the direction is useful, but this patch is not the best merge shape until fallback sends skip message-ID caching, self-chat suppression stays scoped to confirmed assistant outbounds, core avoids BlueBubbles-specific policy, and the changelog is updated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve user-authored self-chat prompts:** `extensions/bluebubbles/src/monitor-processing.ts:875`
  - body: This removes the confirmed-assistant-outbound guard and records every self-chat `fromMe` row as a duplicate marker. Current main explicitly keeps a user-authored self-chat prompt dispatchable when no assistant outbound was confirmed; with this change, the reflected inbound copy is dropped before it can reach the agent. Keep the assistant confirmation or otherwise distinguish local user sends from assistant sends.
  - confidence: 0.86
- **[P2] Do not cache fallback sends as real message IDs:** `extensions/bluebubbles/src/send.ts:187`
  - body: The fallback returns `messageId: "imsg-fallback"`, but the monitor treats every result other than `ok` or `unknown` as an actionable BlueBubbles message GUID. A successful fallback would cache and announce the same fake ID for replies, reactions, and edits that BlueBubbles cannot resolve; return a non-actionable result or add metadata so this path skips outbound message-ID caching.
  - confidence: 0.91
- **[P2] Keep BlueBubbles status policy out of core:** `src/infra/channels-status-issues.ts:18`
  - body: This adds BlueBubbles-specific payload parsing and warning text to the shared status collector even though the current design delegates status policy to plugin collectors and BlueBubbles already has one. Duplicating plugin behavior in core can drift from `extensions/bluebubbles/src/status-issues.ts`; route this through the plugin contract or a generic unloaded-plugin seam instead.
  - confidence: 0.8
- **[P3] Add the user-facing changelog entry:** `extensions/bluebubbles/src/send.ts:703`
  - body: The PR changes user-visible BlueBubbles send, monitor, probe, and status behavior, but the diff has no `CHANGELOG.md` entry. OpenClaw policy requires user-facing fix changes to be recorded in the active Fixes section.
