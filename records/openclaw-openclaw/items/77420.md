## Summary

Keep open; the PR has useful BlueBubbles direction, but the current head still introduces blocking regressions in self-chat prompt handling, outbound message-id caching, and plugin/core status ownership.

## What This Changes

The PR updates BlueBubbles monitor, send, probe, and status handling to suppress self-chat echoes, report Private API helper disconnects, and attempt an `imsg` fallback for helper-disconnected sends.

## Best Possible Solution

Keep the helper-disconnect and echo-dedupe improvements, but preserve existing prompt semantics, avoid actionable synthetic send IDs, keep channel-specific status policy plugin-owned, and document the user-visible fix.

## Reproduction Assessment

Yes, for the PR-introduced regressions: current main tests and source show user-authored self-chat prompts should still dispatch, and current monitor code caches every non-`ok`/`unknown` send id. I did not execute tests under the read-only review constraint.

## Solution Assessment

No. The direction is useful, but the current patch is not the best merge shape until fallback sends are non-actionable, self-chat suppression remains scoped to confirmed assistant outbounds, status policy stays behind the plugin boundary, and the changelog is updated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve user-authored self-chat prompts:** `extensions/bluebubbles/src/monitor-processing.ts:875`
  - body: This records every self-chat `fromMe` row as a duplicate marker. Current main explicitly keeps a user-authored self-chat prompt dispatchable when no assistant outbound was confirmed, so the reflected inbound copy would be dropped before it can reach the agent.
  - confidence: 0.88
- **[P2] Do not cache fallback sends as real IDs:** `extensions/bluebubbles/src/send.ts:187`
  - body: The fallback returns `messageId: "imsg-fallback"`, but the monitor treats any non-empty result other than `ok` or `unknown` as a real BlueBubbles message GUID. A fallback send would cache and announce a fake id that replies, reactions, and edits cannot resolve.
  - confidence: 0.92
- **[P2] Keep BlueBubbles status policy out of core:** `src/infra/channels-status-issues.ts:18`
  - body: This adds BlueBubbles-specific payload parsing and warning text to the shared status collector even though the plugin already owns `collectStatusIssues`. Duplicating channel policy in core can drift from the plugin collector and weakens the extension boundary.
  - confidence: 0.82
- **[P3] Add the user-facing changelog entry:** `extensions/bluebubbles/src/send.ts:703`
  - body: The PR changes user-visible BlueBubbles send, probe, monitor, and status behavior, but the patch has no `CHANGELOG.md` entry. OpenClaw policy expects user-facing fix changes in the active Fixes section.
  - confidence: 0.96
