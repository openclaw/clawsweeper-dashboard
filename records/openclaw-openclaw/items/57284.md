## Summary

## What This Changes

The issue asks for `openclaw agent --deliver` to send the original message directly like `openclaw message send` and also proposes direct session injection.

## Best Possible Solution

Keep `openclaw message send` as the literal notification command, keep `agent --deliver` scoped to agent replies, and split this into clearer suppressed-status handling plus a separate session-injection proposal if maintainers want that feature.

## Solution Assessment

No, not as originally requested. Maintainer discussion rejected redefining `agent --deliver` as literal direct send, so the maintainable path is narrower status surfacing plus a separate session-injection feature decision.
