## Summary

Keep open: current main and the latest release still define `openclaw agent --deliver` as agent-reply delivery, while direct channel notifications already belong to `openclaw message send`; the remaining request needs maintainer product-boundary review rather than conservative cleanup closure.

## What This Changes

The issue asks for `openclaw agent --deliver` to send the provided text directly, and also asks for a direct session-injection path that avoids a model turn.

## Best Possible Solution

Keep `openclaw message send` as the literal outbound notification command, keep `agent --deliver` as agent-reply delivery unless maintainers explicitly change that contract, and separately design any direct session-injection CLI/API surface.

## Reproduction Assessment

Yes, at source level: current main sends `--message` into the agent run and only delivers normalized reply payloads afterward. I did not run a live Signal delivery in this read-only review.

## Solution Assessment

No, redefining `agent --deliver` as literal send is not clearly the best fix because current docs and maintainer discussion point direct notifications to `openclaw message send`. A separate session-injection surface or clearer suppressed-delivery status handling would be safer if maintainers want to pursue the remaining behavior.
