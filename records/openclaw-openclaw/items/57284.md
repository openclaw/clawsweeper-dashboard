## Summary

Keep open. Current main and the latest release still route `agent --deliver` through an agent-reply delivery path, while target-repo PRs that changed it into literal direct send were closed on a command-boundary decision, so the remaining work needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks for `openclaw agent --deliver` to deliver user-authored text directly to a selected channel without spawning an LLM turn, and also raises direct session-injection behavior for `--session-id`.

## Best Possible Solution

Keep `openclaw message send` as the literal outbound notification command, keep `agent --deliver` scoped to agent replies, and split any remaining work into clearer suppressed-status handling plus a separate direct session-injection proposal if maintainers want that feature.

## Reproduction Assessment

Yes, at source level. Current main still sends `--deliver` through the Gateway agent run path and current tests show empty or `NO_REPLY` agent payloads are treated as suppressed no-visible delivery rather than direct sending the original message.

## Solution Assessment

No, not as originally requested. Redefining `agent --deliver` as literal send duplicates `message send` and was rejected in target-repo review; the maintainable path is scoped status surfacing and a separate session-injection decision.
