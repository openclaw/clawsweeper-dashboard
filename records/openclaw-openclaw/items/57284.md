## Summary

Keep open: current main and latest release still implement `openclaw agent --deliver` as agent-reply delivery, not direct original-message send, and the same-repo direct-delivery attempts were closed unmerged after boundary review. The remaining direct-notification/session-injection behavior needs maintainer product direction, especially with an open fork PR still referencing this issue.

## What This Changes

The issue requests deterministic direct channel delivery for `openclaw agent --deliver` and a direct session injection path instead of routing notification text through an agent turn.

## Best Possible Solution

Keep `message send` as the direct channel notification path, preserve `agent --deliver` as reply delivery unless maintainers explicitly change that contract, and separately decide direct session injection or suppressed-status UX.

## Reproduction Assessment

Yes, at source level: current main sends `message` into the Gateway agent run with only a `deliver` boolean, then delivers normalized result payloads after the run; `NO_REPLY` payloads are covered as suppressed/no-visible delivery. I did not run a live Signal recipient in this read-only review.

## Solution Assessment

No, redefining `agent --deliver` as literal send is not currently the best confirmed fix because it overlaps the documented `message send` command and prior same-repo PR review closed that direction as a command-boundary change. A separate direct session injection surface or clearer suppressed-delivery status needs maintainer direction.
