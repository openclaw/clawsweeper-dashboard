## Summary

Keep open. Current main still treats `openclaw agent --deliver` as agent-reply delivery rather than a literal send, and the related direct-delivery PRs were closed unmerged on a maintainer boundary decision, so the remaining work needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks for `openclaw agent --deliver` to send the provided message directly and also asks for direct session injection for `--session-id` workflows.

## Best Possible Solution

Keep `openclaw message send` as the literal notification surface, keep `agent --deliver` as agent-reply delivery, and split any remaining work into clearer suppressed-status surfacing plus a separate session-injection feature if maintainers want it.

## Reproduction Assessment

Yes, from source inspection. Current main still routes `agent --deliver` through the Gateway/agent run and suppresses NO_REPLY or empty payloads as `no_visible_payload` without outbound send.

## Solution Assessment

No for the requested literal-send change. The related maintainer decision keeps direct sends on `openclaw message send`; the safer path is status surfacing for suppressed agent replies and a separate session-injection design.
