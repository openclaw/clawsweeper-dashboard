## Summary

Keep open. Current main still treats `openclaw agent --deliver` as an agent-run command that delivers only the agent-produced reply, while both related direct-delivery PRs were closed unmerged and the latest maintainer discussion says the direct-send boundary should remain `openclaw message send`.

## What This Changes

The issue asks for `openclaw agent --deliver` to send the original message directly instead of running an agent turn, plus a direct session injection surface.

## Best Possible Solution

Keep `openclaw message send` as the direct notification surface, decide whether `agent --deliver` needs clearer non-JSON no-visible-result surfacing, and track direct session injection as a separate explicit CLI/API feature if maintainers want it.

## Reproduction Assessment

Yes, from source inspection. Current main still routes `agent --deliver` through an agent run and returns a suppressed no-visible-payload status when the agent produces `NO_REPLY`, so no outbound channel send occurs in that path.

## Solution Assessment

No for the requested direct-send approach. The closed related PR discussion says redefining `agent --deliver` as literal send is the wrong boundary; a narrower status/docs fix plus separate session-injection design is safer.
