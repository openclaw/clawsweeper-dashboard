## Summary

Current main still routes `openclaw agent --deliver` through a full agent turn and delivers the model result, while the latest release still documents reply delivery rather than direct notification delivery. This should stay open because the proposed direct-delivery behavior is compatibility-sensitive and an open linked fork PR is still acting as a candidate fix path.

## What This Changes

The issue requests that `openclaw agent --deliver` send the provided notification text directly to the selected channel/target and add a direct session-injection path instead of running an LLM turn.

## Best Possible Solution

Maintainers should choose the public contract first: either sponsor an explicit direct notification/session-injection path with upgrade-aware docs and tests, or keep `--deliver` as reply delivery and document `openclaw message send` as the supported direct-send path.

## Reproduction Assessment

Yes, at source level: the CLI and Gateway pass the message into `agentCommand`, `agentCommand` runs `runAgentAttempt`, and delivery sends `result.payloads` rather than the original message. I did not run a live Signal delivery path in this read-only review.

## Solution Assessment

Unclear. Direct delivery is a plausible fix for notification loss, but it changes the documented reply-delivery contract, so the best solution needs maintainer approval of the public CLI/session behavior first.
