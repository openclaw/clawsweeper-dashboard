## Summary

Keep open: current main and v2026.6.8 still treat `openclaw agent --deliver` as agent-reply delivery, while the direct-notification/session-injection behavior requested here remains a maintainer product/API decision.

## What This Changes

Request direct notification delivery for `openclaw agent --deliver` and direct session injection instead of routing the text through a normal agent turn.

## Best Possible Solution

Keep `message send` as the deterministic notification path, preserve `agent --deliver` as reply delivery unless maintainers explicitly change that contract, and design direct session injection as a separate explicit API if needed.

## Reproduction Assessment

Yes, at source level: current main sends the CLI message into a Gateway agent run with `deliver: true`, then delivers only normalized reply payloads afterward. I did not run live Signal delivery in this read-only pass.
