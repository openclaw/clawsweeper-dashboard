## Summary

Keep open. Current main still treats `agent --deliver` as delivery of an agent-produced reply: the CLI and Gateway route through the agent run path, and empty/NO_REPLY payloads return without outbound delivery. Open PR #57300 directly tracks the same fix with closing syntax, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests that `openclaw agent --deliver` directly deliver the provided message to the selected channel instead of running an LLM turn that can produce no outbound payload.

## Best Possible Solution

Implement the narrow direct-delivery behavior on current main: when `--deliver` is set and there are no `internalEvents`, build a delivery result from `opts.message` before any model/ACP attempt and pass it through the existing channel resolution and `deliverAgentCommandResult` path. Preserve the agent-run path for internal-event announcements, add regression coverage for direct delivery and empty/NO_REPLY cases, and treat no-LLM session injection as a separate API decision unless maintainers choose to couple it.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: run `openclaw agent --deliver` with a model result that produces no payload, and current main reaches `deliverAgentCommandResult` with an empty payload list, logs `No reply from agent.`, and never calls outbound delivery.

## Solution Assessment

Yes for the channel-delivery part: #57300's direction of short-circuiting `--deliver` without `internalEvents` and reusing the existing delivery helper is the narrow maintainable fix. The session-injection request is related but should likely be split unless maintainers want one combined product change.
