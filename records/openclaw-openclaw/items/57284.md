## Summary

Keep open. Current main and v2026.5.22 still treat `agent --deliver` as agent-reply delivery, the target-repo direct-delivery PRs were closed unmerged on a command-boundary decision, and the remaining status/session-injection scope needs maintainer scoping.

## What This Changes

The issue asks for `openclaw agent --deliver` to send the original message directly and for a direct session-injection surface that does not start a model turn.

## Best Possible Solution

Keep `openclaw message send` as the literal notification command, keep `agent --deliver` scoped to agent replies, and split any remaining work into explicit suppressed-status/exit behavior plus a separate session-injection design if maintainers want that feature.

## Reproduction Assessment

Yes, as a source-level reproduction: current main still routes `agent --deliver` through an agent run and normalizes agent-produced `NO_REPLY` or empty payloads into suppressed no-payload delivery rather than sending the original message.

## Solution Assessment

No, not as originally requested. Maintainer discussion rejected redefining `agent --deliver` as literal direct send, so the maintainable path is narrower status behavior plus a separate session-injection feature decision.
