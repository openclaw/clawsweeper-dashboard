## Summary

Keep open: current main and v2026.6.8 still treat `openclaw agent --deliver` as agent-reply delivery, not literal notification send, and the same-repo direct-delivery PRs were closed unmerged after compatibility and command-boundary review. The remaining work needs maintainer product direction around suppressed-delivery UX and any explicit direct session-injection surface.

## What This Changes

The issue asks for `openclaw agent --deliver` to deliver the original message directly and for direct session injection instead of routing the text through an agent turn.

## Best Possible Solution

Keep `message send` as the deterministic notification path, preserve `agent --deliver` as reply delivery unless maintainers explicitly change that contract, and decide any direct session-injection API as a separate explicit surface.

## Reproduction Assessment

Yes, at source level: current main sends the CLI message into the Gateway as an agent run with `deliver: true`, then only delivers normalized reply payloads afterward. Current tests show `NO_REPLY` can normalize to no outbound payload with suppressed delivery status.

## Solution Assessment

No, redefining `agent --deliver` as a literal-send command is not the best confirmed fix because `message send` already owns direct outbound sends and a same-repo PR was closed on boundary/compatibility grounds. A maintainer should choose between better suppressed-status UX, docs, or an explicit new session-injection/direct-send API.
