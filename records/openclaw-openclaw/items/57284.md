## Summary

Keep open: current main and v2026.6.10 still define `openclaw agent --deliver` as agent-reply delivery, while literal notification delivery and direct session injection remain command/API boundary decisions with closed unmerged fix attempts and one open conflicting fork PR.

## What This Changes

The issue asks for `openclaw agent --deliver` to send the provided message directly to a channel target instead of treating it as an LLM prompt, and also asks for a direct session-injection path.

## Best Possible Solution

Keep `message send` as the deterministic outbound notification path unless maintainers explicitly redefine `agent --deliver`; design direct session injection as a separate API if maintainers want that capability.

## Reproduction Assessment

Yes, at source level: current main sends `--deliver` into the agent/Gateway path as reply delivery, then downstream delivery uses only agent-produced payloads and treats empty or `NO_REPLY` output as suppressed terminal delivery.

## Solution Assessment

No for the broad requested rewrite as-is: current docs and maintainer discussion point to `message send` as the direct notification surface, while direct session injection needs a separate API/security decision.
