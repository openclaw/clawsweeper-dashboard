## Summary

Keep open: current main and v2026.6.8 still route `openclaw agent --deliver` through an agent turn and deliver only reply payloads, while the same-repo direct-delivery PRs were closed unmerged after maintainer boundary review. The remaining suppressed-delivery/session-injection question needs maintainer product handling rather than conservative cleanup closure.

## What This Changes

The issue asks for `openclaw agent --deliver` and session-key sends to inject user-authored text directly instead of routing it through an LLM turn that can suppress delivery.

## Best Possible Solution

Keep `openclaw message send` as the deterministic channel notification path, preserve `agent --deliver` as reply delivery unless maintainers explicitly change that contract, and separately decide whether to add direct session injection or stronger suppressed-status UX.

## Reproduction Assessment

Yes, at source level: current main sends the message into an agent run with a `deliver` boolean, then delivers only normalized reply payloads afterward; `NO_REPLY` can normalize to no outbound payload. I did not run a live Signal delivery in this read-only review.

## Solution Assessment

No, redefining `agent --deliver` as literal send is not currently the best confirmed fix because it overlaps the documented `message send` path and was rejected in a same-repo PR as a command-boundary change. A separate session-injection surface or improved suppressed-delivery status/docs needs maintainer direction.
