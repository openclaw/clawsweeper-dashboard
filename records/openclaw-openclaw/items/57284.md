## Summary

Keep open. Current main and the latest release still implement `agent --deliver` as agent-reply delivery, while the direct-send PRs were closed unmerged after a maintainer boundary decision, so the remaining issue needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks for `openclaw agent --deliver` to send the original message directly and for a direct session-injection path.

## Best Possible Solution

Keep direct channel notifications on `openclaw message send`, keep `agent --deliver` as agent-reply delivery, and split the remaining work into clearer suppressed-status behavior plus a separately designed session-injection or notify feature if maintainers want it.

## Reproduction Assessment

Yes, from source inspection. Current main routes `agent --deliver` through an agent/Gateway run and suppresses empty or `NO_REPLY` result payloads as terminal `no_visible_payload` without synthesizing a direct outbound message from the original CLI text.

## Solution Assessment

No for the requested direct-send change as written. The maintainable path is to preserve the existing `message send` boundary, improve suppressed-status visibility if needed, and design direct session injection as a separate feature.
