## Summary

Current main already fixes the MiMo replay failure: MiMo model ids now preserve valid `reasoning_content` through OpenAI-compatible replay and the custom-provider fallback backfills it for thinking-mode assistant messages. The fix was merged through the Xiaomi MiMo PR and shipped in `v2026.5.16-beta.1`, while the reported stable `v2026.5.12` does not contain it.

## What This Changes

The issue asks OpenClaw to allow MiMo `mimo-v2.5-pro` OpenAI-compatible tool-call history to retain `reasoning_content` instead of stripping it globally.

## Best Possible Solution

Keep the provider-boundary fix and direct affected `v2026.5.12` users to a release containing it, currently `v2026.5.16-beta.1` or newer current main.

## Reproduction Assessment

Yes for the affected released behavior: the reporter and the merged Xiaomi PR provide a concrete second-turn MiMo tool-call replay path with `400 Param Incorrect` when `reasoning_content` is absent. I did not re-run a live Xiaomi credentialed call in this read-only pass, but current main source and tests show that failing replay shape is now covered.

## Solution Assessment

Yes. The merged fix is narrower than adding a user-facing compat knob: it preserves/backfills `reasoning_content` only for MiMo/DeepSeek-style replay contracts while keeping stock OpenAI/Qwen stripping behavior intact.
