## Summary

Keep open: the reported error-handling concern is plausible, but this branch is too broad because it rethrows unreachable-gateway transport failures that current docs say should still use the config-only fallback. The PR also still lacks after-fix real CLI proof from a real setup.

## What This Changes

The PR adds a probe-mode rethrow in `channelsStatusCommand` and a focused command-flow regression test for a mocked probe failure.

## Best Possible Solution

Rethrow only non-transport request/RPC failures from `channels status --probe`, keep `GatewayTransportError` on the documented config-only fallback path, and cover both branches with focused tests plus redacted CLI proof.

## Reproduction Assessment

Yes, at source level. A focused harness can mock `callGateway` with a plain post-connect request error for the intended rethrow and with a `GatewayTransportError` for the fallback-preservation regression; I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR rethrows every probe-mode gateway failure, but the narrow maintainable fix is to rethrow only when `!isGatewayTransportError(err)` and let unreachable-gateway failures continue through the existing fallback.
