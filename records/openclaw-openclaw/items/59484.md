## Summary

Keep open. The reported behavior is still plausible on current main, but this PR is too broad because it would remove the documented config-only fallback when `channels status --probe` cannot reach the gateway. A narrower fix should preserve fallback for typed transport failures and only rethrow post-handshake/request/probe failures.

## What This Changes

The PR changes `channelsStatusCommand` to rethrow `callGateway` failures when `--probe` is set and adds a regression test for that path.

## Best Possible Solution

Keep the PR open but revise it so `channels status --probe` rethrows non-transport request/probe failures after gateway dispatch while preserving config-only fallback for `GatewayTransportError` reachability failures. Add tests for both cases and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. The existing command-flow test harness can mock `callGateway` failures; one mock can reproduce the intended post-gateway probe error, and a typed `GatewayTransportError` mock can reproduce the regression introduced by the current PR guard.

## Solution Assessment

No. Rethrowing every probe-mode `callGateway` failure is not the narrowest maintainable fix; using the existing transport-error type to keep unreachable gateways on the fallback path is safer and matches documented behavior.
