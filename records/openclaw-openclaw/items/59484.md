## Summary

Current main and v2026.6.6 already address the useful per-account probe-failure problem by returning partial channel status payloads; the stale PR's remaining unconditional probe rethrow would remove the documented config-only fallback for actual gateway reachability/auth failures.

## What This Changes

The PR adds a probe-mode rethrow in `channelsStatusCommand` and a regression test so `channels status --probe` does not use config-only fallback after a gateway call rejects.

## Best Possible Solution

Keep the shipped gateway-side partial-status implementation and close this stale CLI fallback branch rather than merging a broader rethrow.

## Reproduction Assessment

No current-main failure reproduces from source inspection: the gateway now converts throwing or timed-out per-account probes into partial status payloads with warnings and tests cover that path. I did not run a live CLI repro because this review was read-only.

## Solution Assessment

No; this PR is no longer the best way to solve the issue because the unconditional probe rethrow is broader than the stated failure and removes the documented fallback for unreachable gateways. The shipped gateway-side partial-status handling is the narrower maintainable solution.
