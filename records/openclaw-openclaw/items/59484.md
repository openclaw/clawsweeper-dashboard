## Summary

Keep open. The PR targets a plausible CLI status bug, but the current diff is broader than the stated after-gateway scope because it rethrows every `channels status --probe` gateway-call failure, including typed unreachable-gateway transport failures that current docs and code keep on the config-only fallback path; it also lacks contributor-provided real behavior proof.

## What This Changes

The PR adds an early rethrow for `channels status --probe` `callGateway` failures and a focused command-flow regression test.

## Best Possible Solution

Revise the PR so `channels status --probe` rethrows non-transport request/RPC failures while preserving config-only fallback for `GatewayTransportError`, then cover both paths and provide redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence harness path: mock `callGateway` with a plain request error for the intended rethrow behavior and with a `GatewayTransportError` for the fallback-preservation regression; I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR is too broad; the narrow maintainable fix is to gate the probe-mode rethrow with `!isGatewayTransportError(err)` so unreachable-gateway failures keep the documented fallback.
