## Summary

Keep open: the requested direction is plausible, but the current branch is too broad because it turns unreachable-gateway transport errors into hard failures despite the documented config-only fallback, and the external PR still lacks real CLI proof.

## What This Changes

The PR adds an early `--probe` rethrow in `channels status` plus a focused command-flow regression test.

## Best Possible Solution

Gate the probe-mode rethrow to non-transport gateway request/RPC failures, preserve config-only fallback for `GatewayTransportError`, cover both paths, and add redacted terminal proof.

## Reproduction Assessment

Yes, at source level. A focused harness can mock `callGateway` with a plain request error for the intended rethrow and with a `GatewayTransportError` for the fallback-preservation regression; I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR is too broad; the narrow maintainable fix is to rethrow only when `!isGatewayTransportError(err)` so unreachable-gateway failures keep the documented fallback.
