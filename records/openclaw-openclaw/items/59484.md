## Summary

Keep open: the reported problem is plausible, but the current PR overcorrects by rethrowing every probe-mode gateway failure, including unreachable-gateway transport failures that current docs say should use config-only fallback.

## What This Changes

This PR changes `channelsStatusCommand` to rethrow `channels status --probe` `callGateway` failures and adds a focused regression test for that path.

## Best Possible Solution

Keep the PR open and revise it so probe mode rethrows only non-transport request/probe failures while preserving config-only fallback for unreachable gateway transport errors, with tests for both paths and a changelog entry.

## Reproduction Assessment

Yes. The existing command-flow test harness can mock `callGateway` with a plain probe/request error for the intended fix and with a `GatewayTransportError` for the regression introduced by the PR guard.

## Solution Assessment

No. Rethrowing every probe-mode `callGateway` error is broader than the PR’s after-gateway scope; using `isGatewayTransportError(err)` to preserve reachability fallback is the narrower maintainable fix.
