## Summary

Keep this PR open but not merge-ready: it targets a real `channels status --probe` false-fallback path, but the patch rethrows all probe-mode gateway failures, which would break the documented config-only fallback for unreachable gateways; it also lacks real CLI proof and is currently conflicting.

## What This Changes

The PR rethrows `callGateway` failures in `channels status --probe` and adds a command-flow regression test for a mocked probe-mode rejection.

## Best Possible Solution

Rethrow only non-transport request/RPC failures from `channels status --probe`, preserve `GatewayTransportError` on the config-only fallback path, and cover both branches with focused tests plus redacted CLI proof.

## Reproduction Assessment

Yes, at source level. Current main catches all `callGateway` failures, while `GatewayClient.request` can reject post-connect request failures and `callGateway` can reject typed transport failures for unreachable gateway paths.

## Solution Assessment

No. The PR should not rethrow every probe-mode failure; the narrow maintainable fix is to rethrow non-transport request/RPC failures while preserving unreachable-gateway fallback.
