## Summary

Keep open: the PR targets a plausible `channels status --probe` bug, but the current patch rethrows every probe-mode gateway call failure and would bypass the documented config-only fallback for unreachable-gateway transport errors. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR changes `channels status --probe` so `callGateway` failures are rethrown instead of falling back to config-only status, and adds a focused regression test.

## Best Possible Solution

Revise the PR so `channels status --probe` rethrows non-transport RPC/request failures while preserving config-only fallback for `GatewayTransportError`, then add focused tests for both paths and contributor-provided runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence harness path: mock `callGateway` with a plain request error for the intended rethrow behavior and with a `GatewayTransportError` for the fallback-preservation regression; tests were not run because this review is read-only.

## Solution Assessment

No. The current PR is broader than the stated after-gateway scope; the narrower maintainable fix is to keep typed transport errors on the documented fallback path and rethrow only non-transport request/probe failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve fallback for gateway transport failures:** `src/commands/channels/status.ts:309-311`
