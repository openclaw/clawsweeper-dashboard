## Summary

Keep open: current `main` still does not distinguish post-gateway request/probe failures from unreachable-gateway transport failures, but this PR’s guard is too broad and would also bypass the documented config-only fallback for closed/timeout transport errors.

## What This Changes

The branch adds a probe-mode guard in `channelsStatusCommand` that rethrows `callGateway` errors and adds a regression test for a rejected probe status call.

## Best Possible Solution

Revise the PR so `channels status --probe` rethrows non-transport RPC/request failures while preserving config-only fallback for `GatewayTransportError`, with focused tests for both paths.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence harness path: mock `callGateway` with a plain request error for the intended behavior and with a `GatewayTransportError` for the regression introduced by the PR guard; tests were not run because this review is read-only.

## Solution Assessment

No. Rethrowing every probe-mode `callGateway` error is broader than the requested after-gateway scope; the narrower fix is to keep typed transport errors on the documented fallback path and rethrow only non-transport request failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve fallback for gateway transport failures:** `src/commands/channels/status.ts:309-311`
