## Summary

Keep open: this targets a real `channels status --probe` reporting bug, but the branch rethrows every probe-mode gateway failure and would remove the documented config-only fallback for unreachable gateways.

## What This Changes

The PR changes `channels status --probe` to rethrow `callGateway` failures in probe mode and adds a focused command-flow regression test.

## Best Possible Solution

Rethrow only non-transport request/RPC failures from `channels status --probe`, preserve `GatewayTransportError` on the documented config-only fallback path, and cover both branches with focused tests plus redacted CLI proof.

## Reproduction Assessment

Yes, at source level. Current main catches all `callGateway` failures, while `GatewayClient.request` can reject with a plain post-connect `Error` and `callGateway` can reject with typed `GatewayTransportError` for unreachable-gateway close/timeout paths.

## Solution Assessment

No. The PR should not rethrow every probe-mode failure; the narrow maintainable fix is to rethrow only non-transport request/RPC failures and keep unreachable gateways on the existing fallback path.
