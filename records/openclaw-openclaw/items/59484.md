## Summary

Current main already handles per-account `channels status --probe` failures at the gateway layer, preserving account-level probe diagnostics without entering the config-only fallback; the PR branch's remaining CLI rethrow is broader than the maintained fallback contract.

## What This Changes

The PR rethrows `channels status --probe` `callGateway` failures from `channelsStatusCommand` and adds a command-flow regression test for that path.

## Best Possible Solution

Keep the current main gateway-level probe failure handling and close this branch rather than merging the broader CLI rethrow.

## Reproduction Assessment

Yes. Source-level reproduction is clear: mock a throwing `probeAccount` in the gateway status test to verify current main returns `probe.ok: false`, and mock `callGateway` with a `GatewayTransportError` on the PR branch to see the broad guard skip the documented fallback.

## Solution Assessment

Yes for current main, no for the PR diff. Handling probe hook failures inside the gateway status method is the narrow maintainable fix because it preserves unreachable-gateway config-only fallback while still surfacing probe failures in status output.
