## Summary

Keep open: the PR targets a real source-level bug, but the current diff rethrows typed gateway transport failures that the docs say should keep the config-only fallback. It also lacks after-fix real CLI proof and is currently conflicting with main.

## What This Changes

The PR adds a probe-mode rethrow in `channelsStatusCommand` and a command-flow test for a mocked post-connect probe failure.

## Best Possible Solution

Rethrow only non-transport request/RPC failures from `channels status --probe`, preserve `GatewayTransportError` on the documented config-only fallback path, and cover both branches with focused tests plus redacted CLI proof.

## Reproduction Assessment

Yes, at source level. Current main catches all `callGateway` failures, while `GatewayClient.request` can reject with a plain post-connect `Error` and `callGateway` can reject with typed `GatewayTransportError` for unreachable-gateway close/timeout paths.

## Solution Assessment

No. The PR should not rethrow every probe-mode failure; the narrow maintainable fix is to rethrow only when `!isGatewayTransportError(err)` and keep unreachable gateways on the existing fallback path.
