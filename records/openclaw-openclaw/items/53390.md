## Summary

Keep open: current main still has a source-level path where browser text snapshots come from page/root accessibility trees without any viewport or scroll-bounds contract, so the reported scroll mismatch is not resolved and needs maintainer product direction before changing default snapshot semantics.

## What This Changes

The issue asks for `browser snapshot` to reflect the currently visible viewport after scrolling instead of returning top-of-page or full-tree content.

## Best Possible Solution

Define a viewport-aware snapshot contract, then implement it consistently across the tool schema, action query, route plan, Playwright/CDP/Chrome MCP backends, docs, and long-page regression coverage while preserving full-tree access if maintainers want both modes.

## Reproduction Assessment

Yes, at source level. The issue gives a concrete long-page scroll-then-snapshot sequence, and current main still has no viewport or scroll-bounds data before reading page/root accessibility snapshots.

## Solution Assessment

Unclear until maintainers choose the snapshot contract. A backend-only tweak would be incomplete; the maintainable fix needs one documented behavior across all snapshot implementations, with a compatibility path for full-tree snapshots if needed.
