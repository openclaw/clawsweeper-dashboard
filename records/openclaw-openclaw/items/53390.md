## Summary

Keep open: current main and the latest release still have browser text snapshot paths built around page/root/full accessibility snapshots with no viewport scope, scroll position, or visible-viewport metadata. The FSB comment is useful supporting context because it points to the same unresolved contract choice: visible viewport reads should be distinct from full DOM reads.

## What This Changes

The issue requests viewport-aware browser text snapshots after scrolling, with scroll/viewport state reflected separately from full-page DOM reads.

## Best Possible Solution

Define a browser snapshot contract that exposes visible-viewport reads and scroll/viewport metadata while retaining an explicit full-tree mode, then update the schema, CLI/API, routes, backends, docs, and long-page regression proof together.

## Reproduction Assessment

Yes, at source level: the reported scroll-then-snapshot flow maps to current snapshot paths that neither accept nor return viewport/scroll state for text snapshots. I did not run a live Chromium reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the snapshot contract. A safe solution likely needs an explicit viewport/full-tree scope across all browser snapshot backends rather than silently replacing existing full-tree behavior in one backend.
