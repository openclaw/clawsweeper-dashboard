## Summary

Keep open: current main still has browser text snapshot paths built around page/root/full accessibility snapshots, while the issue asks for visible-viewport semantics and scroll/viewport state after scrolling. This remains a real browser automation contract gap, but it should not be auto-fixed until maintainers choose the API shape because silently changing snapshot defaults could break users who rely on full-tree output.

## What This Changes

The issue requests that browser `snapshot` reflect the currently visible viewport after scrolling rather than returning top/full-page snapshot content.

## Best Possible Solution

Define an explicit browser snapshot contract that supports visible-viewport reads and scroll/viewport metadata while preserving an explicit full-tree mode, then update schema, CLI/docs, route planning, all snapshot backends, and long-page regression proof together.

## Reproduction Assessment

Yes, at source level: the reported scroll-then-snapshot flow maps to current snapshot paths that neither accept nor return visible viewport or scroll state for text snapshots. I did not run a live Chromium reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the snapshot contract. The safer direction is an explicit viewport/full-tree scope across all browser snapshot backends rather than silently replacing current full-tree behavior in one path.
