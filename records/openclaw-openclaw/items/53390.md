## Summary

Keep open. Current main still routes browser snapshots through full AI/ARIA/CDP/Chrome-MCP tree reads without a defined visible-viewport or scroll-position contract, and the linked FSB comment supports treating this as a browser read-contract decision rather than cleanup-close material.

## What This Changes

Request that browser snapshots reflect the current visible viewport after scrolling, or surface scroll and viewport state so agents know what part of the page they are reading.

## Best Possible Solution

Define the browser read contract first: keep full-tree/raw reads explicit, add or select a viewport-aware visible read with scroll metadata, then cover Playwright, raw CDP, and Chrome MCP paths with focused tests and live proof.

## Reproduction Assessment

No, not as a high-confidence live current-main repro. Source inspection shows the snapshot paths omit scroll and viewport metadata, but the reported after-scroll GitHub page output still needs live browser proof.

## Solution Assessment

Unclear. A direct flip to viewport-only snapshots may break full-tree ref workflows, so the best fix is a maintainer-approved read contract with explicit full-tree and visible-viewport behavior.
