## Summary

Keep open: current main still routes browser text snapshots through page/root/full-tree paths with no viewport or scroll-state contract, and the recent FSB comment is useful supporting context for a maintainer-visible visible-viewport versus full-DOM decision.

## What This Changes

The issue asks for `browser snapshot` to reflect the current visible viewport after scrolling instead of continuing to report top-of-page content.

## Best Possible Solution

Define and document one cross-backend snapshot scope contract, ideally with visible viewport text plus scroll/viewport metadata and an explicit full-tree mode if maintainers want to preserve both behaviors.

## Reproduction Assessment

Yes, at source level: the reported scroll-then-snapshot flow maps to snapshot paths that do not accept, filter by, or report viewport/scroll state for text snapshots. I did not establish a live Chromium reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the snapshot contract. A backend-only fix would be risky; the maintainable path is a documented cross-backend viewport/full-tree scope decision with regression coverage.
