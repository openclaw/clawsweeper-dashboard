## Label Justifications

- `P2`: This is a meaningful browser automation correctness/design issue with below-the-fold workflow impact, but it is not a crash, security bypass, or urgent production regression.
- `impact:session-state`: The issue is about browser viewport state drifting from the page state returned to the agent after scroll.

## Summary

Keep open: current main and the latest release still have browser snapshot schema, route planning, and backend paths without a visible-viewport scope or returned scroll/viewport metadata. The FSB comment is useful supporting context for solving this as an explicit visible-vs-full snapshot contract instead of silently changing existing full-tree defaults.

## What This Changes

Make browser snapshots reflect the current scrolled viewport, including below-the-fold content, instead of returning top/full-page content after scroll.

## Best Possible Solution

Define an explicit browser snapshot scope that preserves full-tree/raw reads while adding visible-viewport reads with scroll/viewport metadata across schema, CLI/docs, route planning, Playwright, Chrome MCP, CDP fallback, and long-page regression proof.

## Reproduction Assessment

Yes, at source level: the reported scroll-then-snapshot flow maps to snapshot request, plan, and backend paths that have no scroll/viewport fields and use page/root/full accessibility reads. I did not run a live Chromium reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API review. The safer solution is an explicit visible/full snapshot scope with viewport metadata rather than silently replacing current full-tree behavior in one backend.
