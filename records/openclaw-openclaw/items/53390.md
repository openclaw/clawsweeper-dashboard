## Summary

Keep open. Current main still lacks a viewport-aware browser text snapshot contract: the public schema, client/action query, route plan, docs, and Playwright/CDP/Chrome MCP snapshot paths expose format, refs, selector/frame, labels, URLs, and size limits, but no current-viewport or visible-only mode, and the implementations still read page/root accessibility snapshots without viewport filtering.

## What This Changes

The issue requests that browser text snapshots reflect the currently visible viewport after scrolling instead of continuing to show top-of-page content.

## Best Possible Solution

Define and document a viewport-aware browser text snapshot contract, then implement it consistently through the browser tool schema, action query, client API, snapshot route plan, Playwright path, CDP fallback, Chrome MCP path, CLI, and docs. Add a long-page regression that scrolls first and asserts the next text snapshot reflects visible content after the scroll.

## Reproduction Assessment

Yes. The issue gives a concrete long-page browser-tool sequence, and current main’s source path does not inspect scroll offsets or viewport bounds before producing text snapshots; I did not run a live browser because this review is read-only.

## Solution Assessment

Unclear because there is no proposed patch. The maintainable solution needs a product decision on default behavior versus an explicit viewport-scoped mode before an automated fix should change the browser snapshot contract.
