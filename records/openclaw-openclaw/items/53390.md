## Summary

Keep open: current main still routes browser text snapshots through page/root accessibility snapshots with no viewport, scroll offset, visible-only flag, or bounds contract, so the reported scroll-then-snapshot mismatch is still source-reproducible and needs a maintainer decision on snapshot semantics.

## What This Changes

The issue requests browser snapshot output to reflect the currently visible viewport after scrolling instead of continuing to return top-of-page or full-tree content.

## Best Possible Solution

Define one documented viewport-aware snapshot contract across the tool schema, action query, route plan, Playwright, CDP, Chrome MCP, docs, and long-page regression coverage while preserving full-tree access if maintainers want both behaviors.

## Reproduction Assessment

Yes, at source level: the reported long-page scroll-then-snapshot flow maps to current code paths that read page/root accessibility snapshots without viewport or scroll-bounds state. I did not establish a live Chromium trace in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the snapshot contract. A backend-only tweak would be incomplete; the maintainable fix needs consistent behavior across all snapshot implementations, with a compatibility path if full-tree snapshots remain supported.
