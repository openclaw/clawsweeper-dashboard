## Summary

Keep open: current main still has no viewport-aware text snapshot contract, and the recent FSB comment reinforces that this needs a deliberate visible-viewport versus full-tree design choice rather than cleanup closure.

## What This Changes

The issue asks for browser snapshots after scrolling to reflect the currently visible viewport instead of returning the initial/top-page content.

## Best Possible Solution

Define and document a single viewport-aware snapshot contract across the tool schema, CLI/API, route plan, Playwright, CDP, Chrome MCP, docs, and long-page regression coverage while preserving full-tree access through an explicit mode if maintainers want both behaviors.

## Reproduction Assessment

Yes, at source level: the reported scroll-then-snapshot flow maps to current snapshot paths that do not accept, filter by, or report viewport/scroll state for text snapshots. I did not establish a live Chromium reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the snapshot contract. A safe fix likely needs an explicit viewport/full-tree scope across all browser snapshot backends rather than a backend-only change that silently replaces existing full-tree behavior.
