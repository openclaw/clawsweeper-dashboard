## Summary

Keep open: the one-line resolver change is a plausible piece for Gateway/getStatusSummary, but it does not fix the reported CLI `openclaw status --json` path because that path disables channel summary before `buildChannelSummary` is called. The PR also needs stronger real behavior proof for the affected CLI command before merge.

## What This Changes

The PR enables setup-fallback channel plugin resolution inside `listChannelSummaryPlugins` and adds a unit assertion that `buildChannelSummary` forwards that option.

## Best Possible Solution

Update the JSON status path and regression coverage so configured external or setup-based channels appear in `channelSummary` for `openclaw status --json` without starting channel runtimes, then include redacted real CLI output from a configured channel.

Yes, from source inspection: `statusCommand({ json })` reaches `scanStatusJsonFast`, which sets `includeChannelSummary: false`, so the CLI path can produce an empty `channelSummary` before this PR's changed function runs. I did not perform a live Telegram run.

## Solution Assessment
