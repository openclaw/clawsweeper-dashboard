## Summary

Keep open. Current main still validates original stripped `--resources` tokens against lowercase allowed names, so valid mixed-case resource names fail; `v2026.4.27` carries the same parser, and the related fix PRs in the provided context are closed and unmerged.

## What This Changes

The issue requests case-insensitive parsing for `skills/skill-creator/scripts/init_skill.py --resources` while preserving validation and order-preserving deduplication.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase resource names, and add focused `parse_resources()` regression tests for mixed-case valid input, case-insensitive duplicates, empty-token handling, and invalid-resource rejection.

## Reproduction Assessment

Yes. A read-only `python3 -B` import of `parse_resources()` on current main shows lowercase values succeed, while mixed-case valid values and case-only duplicates exit with unknown-resource errors.

## Solution Assessment

Yes. A narrow parser normalization plus focused Python unittest coverage is the smallest maintainable fix; no core architecture, plugin API, security, or product decision is needed first.
