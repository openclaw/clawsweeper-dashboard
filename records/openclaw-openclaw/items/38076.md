## Summary

Keep open. Current main and the latest release still validate stripped `--resources` tokens against lowercase allowed names without normalizing case, and the related implementation attempts are closed and unmerged, so no conservative close reason applies.

## What This Changes

The issue asks for the skill-creator initializer to accept `--resources` values case-insensitively while preserving validation and order-preserving deduplication.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase resource names, and add focused parser regression coverage.

## Reproduction Assessment

Yes. A read-only Python probe against current main shows lowercase resources succeed while mixed-case valid values and case-only duplicates exit with unknown-resource errors.

## Solution Assessment

Yes. A narrow normalization inside `parse_resources()` with isolated tests is the smallest maintainable path; no core API, plugin architecture, security, or config change is needed.
