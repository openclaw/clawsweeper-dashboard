## Summary

Keep open. Current main still rejects mixed-case `--resources` values because `parse_resources()` validates stripped original tokens against lowercase allowed names, and the related fix PRs are closed and unmerged with no open canonical replacement.

## What This Changes

Request case-insensitive parsing and regression coverage for `skill-creator` `init_skill.py --resources` values while preserving validation and order-preserving deduplication.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase names, and add focused parser tests.

## Reproduction Assessment

Yes. A read-only in-memory probe against current `origin/main` shows lowercase resources succeed while mixed-case valid values and case-only duplicates exit with unknown-resource errors.

## Solution Assessment

Yes. A narrow normalization inside `parse_resources()` with isolated regression coverage is the smallest maintainable path; no core API, plugin architecture, security, or config change is needed.
