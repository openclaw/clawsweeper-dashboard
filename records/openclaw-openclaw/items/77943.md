## Summary

Keep open. Current main still emits divergent list-style `--json` shapes, and I found no merged canonical PR implementing the proposed envelope; because this would add a new public CLI JSON contract and migration flag, it needs maintainer product/API review rather than cleanup closure.

## What This Changes

The issue requests a standardized, versioned JSON envelope for list-style CLI subcommands such as agents, sessions, models, cron, nodes, and skills.

## Best Possible Solution

Define one supported CLI list-output contract, add an opt-in compatibility path for the new envelope, document the migration timeline, and cover the affected commands with focused JSON shape tests before any default changes.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the premise: current main still serializes the named list commands with bare-array and command-specific object shapes rather than one shared envelope.

## Solution Assessment

Unclear. The proposed versioned envelope is plausible, but it is a public CLI output change and the best solution depends on maintainer decisions about schema fields, opt-in flag naming, default timing, and legacy availability.
