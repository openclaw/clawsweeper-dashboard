## Summary

Keep open: this PR targets a real source-backed gateway hot-reload bug, but the plugin pre-replace rollback can miss a channel that was partially stopped before `stopChannel` rejected, and the PR still needs real behavior proof beyond mocked tests/CI.

## What This Changes

The PR adds per-channel gateway hot-reload failure collection, plugin pre-replace stop rollback, focused reload regression tests, and a changelog entry for the linked channel-restart bug.

## Best Possible Solution

Keep the aggregate-failure approach, track rollback candidates before awaiting `stopChannel`, add a regression for partial-stop rejection, then land one canonical fix for https://github.com/openclaw/openclaw/issues/83054 after real behavior proof is added.

## Reproduction Assessment

Yes, source-reproducible: current main awaits each channel restart in a plain loop, so a rejecting first channel prevents later channel restarts. I did not run tests because this was a read-only checkout review.

## Solution Assessment

No: the final restart isolation is the right direction, but the plugin pre-replace rollback should track channels before awaiting `stopChannel` so partial stop failures are also rolled back.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Track pre-stop rollbacks before awaiting stopChannel:** `src/gateway/server-reload-handlers.ts:358-360`
  - body: During plugin reload pre-replace, the rollback list is populated only after `stopChannel` resolves. The real stop path marks the channel manually stopped and aborts it before awaiting plugin shutdown, so a rejection there can leave the channel down while this rollback restarts only earlier successful stops. Track the channel before awaiting `stopChannel`, as the secrets reload rollback path already does, and add a partial-stop failure test.
  - confidence: 0.91
