## Summary

Keep open: the PR fixes the main restart-loop failure shape, but the latest rollback path can still start channels that were not running before an aborted plugin pre-replace stop. The `maintainer` label also means this should stay in explicit maintainer handling rather than cleanup-close.

## What This Changes

The PR wraps gateway hot-reload channel stop/start loops with per-channel failure collection, adds plugin pre-replace rollback, focused regression tests, and a changelog entry.

## Best Possible Solution

Keep the aggregate per-channel restart behavior, but make aborted plugin pre-replace rollback restore only channels/accounts that were actually running before the attempted stop.

## Reproduction Assessment

Yes, source-reproducible: current main awaits `restartChannel(channel)` in a plain loop, so a rejecting first channel prevents later channel restarts. The rollback-state defect in the PR is also source-reproducible from `stopChannel` manual-stop marking and `startChannel` clearing that state.

## Solution Assessment

No: the final restart isolation is the right direction, but the plugin pre-replace rollback should preserve pre-reload running/manual-stop state instead of starting every recorded stop candidate.
