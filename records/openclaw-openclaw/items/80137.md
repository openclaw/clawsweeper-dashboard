## Summary

Keep this PR open: the `id` to `sessionId` compatibility fix is still needed and now has real live-gateway proof, but the patch also accepts a `sessionId`-only thread by writing it into `thread.id`, which conflicts with the upstream contract and OpenClaw's later use of the real thread id.

## What This Changes

The PR adds Codex app-server thread response normalization before AJV validation and adds focused validator tests.

## Best Possible Solution

Keep the Codex app-server boundary normalizer, but only fill missing `sessionId` from a present `thread.id`; missing `thread.id` should remain invalid with regression coverage.

## Reproduction Assessment

Yes, by source inspection and supplied live logs: current main validates raw `thread/start` responses against a schema requiring `thread.sessionId`, so an id-only response is rejected before the Codex turn can start.

## Solution Assessment

No: the one-way `id` to `sessionId` normalization is the narrow fix, but accepting `sessionId` as a replacement for missing `id` is not supported by the upstream type or OpenClaw's binding semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not synthesize thread.id from sessionId:** `extensions/codex/src/app-server/protocol-validators.ts:160-161`
  - body: The reported compatibility gap is an id-only thread, but this branch also accepts the opposite shape and stores `sessionId` as `thread.id`. Upstream documents `sessionId` as shared across a session tree, while OpenClaw persists `response.thread.id` for later resume/turn calls, so this can bind future calls to the wrong identifier instead of rejecting a malformed response.
  - confidence: 0.9
