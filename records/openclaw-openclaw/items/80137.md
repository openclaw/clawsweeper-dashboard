## Summary

Keep open: the PR targets a real Codex app-server validation regression and now has live proof, but it still accepts the unsupported `sessionId`-without-`id` shape and can persist the wrong thread identifier for later resume/turn calls.

## What This Changes

The PR adds pre-validation normalization for Codex thread start/resume responses and a new validator test file for `id`/`sessionId` cross-fill cases.

## Best Possible Solution

Keep the Codex app-server response-boundary normalization, but only fill missing `sessionId` from a present `thread.id`; responses missing `thread.id` should still fail validation.

## Reproduction Assessment

Yes. Source inspection shows current main validates raw `thread/start` and `thread/resume` responses against a schema requiring `sessionId`, and the supplied live logs show the reported `id`-only Codex response failing before the patch.

## Solution Assessment

No. Filling `sessionId` from `id` is the narrow compatibility fix, but filling missing `id` from `sessionId` crosses two upstream-distinct identifiers that OpenClaw later uses for thread resume and turn requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep sessionId from replacing the thread id:** `extensions/codex/src/app-server/protocol-validators.ts:161-162`
  - body: The live failure is an `id`-only thread, but this branch also accepts a missing `id` by storing `sessionId` as `thread.id`. Upstream documents `sessionId` as shared across a session tree, while OpenClaw persists `response.thread.id` and sends it as `threadId` for later resume and turn calls, so this can poison the binding instead of rejecting a malformed response. Please remove the reverse fill and keep missing `id` invalid unless upstream proves the fields are interchangeable.
  - confidence: 0.9
