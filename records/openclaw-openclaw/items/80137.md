## Summary

Keep open: the PR addresses a real Codex app-server regression and has sufficient live proof, but the latest patch still accepts a `sessionId`-only response by manufacturing `thread.id`, which can persist the session-tree id as OpenClaw's thread id.

## What This Changes

The branch adds Codex app-server `thread/start` and `thread/resume` response normalization before AJV validation, focused validator tests, and a changelog entry.

## Best Possible Solution

Keep the Codex app-server boundary normalizer, but only fill missing `thread.sessionId` from a present `thread.id`; missing `thread.id` should remain invalid with regression coverage.

## Reproduction Assessment

Yes. Source inspection plus supplied live logs show current main validates raw `thread/start` responses against a schema requiring `thread.sessionId`, so an id-only live response fails before the Codex turn can start.

## Solution Assessment

No. One-way `id` to `sessionId` normalization is the narrow fix, but accepting `sessionId` as a replacement for missing `id` is not supported by the upstream type or OpenClaw's binding semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not synthesize thread.id from sessionId:** `extensions/codex/src/app-server/protocol-validators.ts:160-161`
  - body: The reported compatibility gap is an id-only thread, but this branch also accepts the opposite shape and stores `sessionId` as `thread.id`. Upstream documents `sessionId` as shared across a session tree, while OpenClaw persists `response.thread.id` for later resume/turn calls, so this can bind future calls to the wrong identifier instead of rejecting a malformed response.
  - confidence: 0.9
