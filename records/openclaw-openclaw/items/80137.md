## Summary

Keep open: the reported Codex app-server compatibility bug is valid, but this PR currently has a correctness issue because it also fabricates `thread.id` from `sessionId`, and the proof is only a unit test rather than real app-server behavior.

## What This Changes

The PR normalizes Codex app-server thread start/resume responses by cross-filling missing `id` and `sessionId` before AJV validation and adds validator tests.

## Best Possible Solution

Normalize only missing `sessionId` from a present `thread.id`, keep missing `thread.id` invalid, add focused regression coverage, and require redacted real Codex app-server proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main feeds the raw `thread/start` response into AJV, and the generated schema requires `thread.sessionId`, so an otherwise valid `thread.id`-only response is rejected before turn start. I did not live-reproduce the Codex service variant because this is a read-only review.

## Solution Assessment

No: filling missing `sessionId` from `id` is the narrow boundary fix, but filling missing `id` from `sessionId` conflates distinct upstream identifiers that OpenClaw later uses for thread resume and turn requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep sessionId from replacing the thread id:** `extensions/codex/src/app-server/protocol-validators.ts:161-162`
  - body: The reported compatibility gap is a `thread.id` value without `thread.sessionId`, but this branch also accepts the opposite shape and stores `sessionId` as `thread.id`. Upstream describes `sessionId` as shared across a session tree, while OpenClaw persists `response.thread.id` for later resume and turn requests, so accepting a missing id can bind future calls to the wrong identifier instead of rejecting a malformed response. Please remove the reverse fill and keep only `id` to `sessionId` normalization unless upstream proves these fields are interchangeable.
  - confidence: 0.88
