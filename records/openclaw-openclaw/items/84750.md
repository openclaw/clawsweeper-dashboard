The issue reports that stale device-store `.tmp` files may cause `openclaw devices list` to time out after `JsonFileReadError` on `paired.json`.

## Best Possible Solution

Treat this as part of the device-pairing JSON read/write race cluster and fix the canonical read path with focused regression coverage before closing both related reports.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does show the device list path reading the pairing JSON files without the same serialization used by write operations, which plausibly explains the reported `File changed during read` error.

## Solution Assessment

No: a startup `.tmp` cleanup hook is not the best primary fix unless a test proves stale siblings can affect reads. The safer repair is to make the device-pairing read/write path tolerate or serialize canonical atomic replacements and then add a regression test.

Fix the device-pairing JSON read failure cluster reported by https://github.com/openclaw/openclaw/issues/84750 and https://github.com/openclaw/openclaw/issues/84751. Current source reads `devices/pending.json` and `devices/paired.json` through `loadState` while write paths use atomic replacement; `listDevicePairing` and read helpers are not serialized with writes. Add focused regression coverage that simulates a canonical `paired.json`/`pending.json` replacement during a device list/read path and verifies the command/store recovers or retries instead of surfacing `JsonFileReadError`/timing out. Prefer a bounded fix in the device-pairing store or shared pairing-files helper; do not add a configurable TTL or directory-wide stale `.tmp` cleanup as the only fix unless the test proves orphan files are actually involved. Include a user-facing changelog fix entry if the repair changes runtime behavior.
