## What This Changes

The issue requests avoiding device-management timeouts by cleaning or ignoring stale `.tmp` files under `~/.openclaw/devices` during pairing JSON reads.

## Best Possible Solution

Land or sequence a single bounded async JSON read retry at the fs-safe/OpenClaw helper boundary, then verify the device-pairing command path and close this issue together with the related live-write race if the timeout is resolved.

## Reproduction Assessment

Partially: there is no high-confidence reproduction for stale `.tmp` siblings corrupting reads, but current source and the open retry PRs provide a clear reproduction path for the canonical atomic-rename `File changed during read` race. I did not run a live concurrent gateway/device-store repro in this read-only review.

## Solution Assessment

No: startup `.tmp` cleanup or skipping directory scans is not the best primary fix unless a test proves stale siblings affect reads. The safer solution is a bounded retry or equivalent serialization for canonical pairing JSON reads while preserving fail-closed behavior for real corruption.
