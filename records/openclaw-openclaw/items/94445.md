## Summary

Keep open: the patch is a narrow source-level fix for cron cloud runs without an explicit timeout, but this PR has a protected `maintainer` label, changes existing no-timeout cron behavior, and still lacks after-patch real cron runtime proof from a real setup.

## What This Changes

The PR removes the cron-wide no-idle-timeout bypass from the embedded LLM idle-timeout resolver and updates focused tests so cron cloud calls use the default watchdog while local-provider cron calls still opt out.

## Best Possible Solution

Land the narrow resolver change only after maintainers accept the no-explicit-timeout cloud-cron default and the PR includes redacted real cron runtime proof showing the changed watchdog path and retained local-provider opt-out.

## Reproduction Assessment

Yes at source level: current main returns 0 for `resolveLlmIdleTimeoutMs({ trigger: "cron" })`, and the production cron path passes `trigger: "cron"` into the embedded runner. I did not establish a packaged live cron reproduction.

## Solution Assessment

Yes for the narrow resolver bug: removing the cron-wide bypass lets the existing cloud/local provider classification decide the implicit watchdog. This is not the full answer for broader timeout/fallback policy, so maintainer acceptance and runtime proof remain important.
