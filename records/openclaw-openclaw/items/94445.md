## Summary

Keep open for maintainer review: the patch is a narrow, source-backed fix with sufficient real-behavior proof and no blocking code findings, but it intentionally changes default cron cloud timeout behavior and the PR has a protected maintainer label.

## What This Changes

The PR removes the cron-wide no-timeout idle-watchdog opt-out for embedded cloud model calls, adds regression coverage for cloud/local cron paths, and updates the agent-loop timeout docs.

## Best Possible Solution

Merge only after a maintainer explicitly accepts restoring the default cloud cron idle fuse while preserving explicit timeout precedence and the local-provider opt-out.

## Reproduction Assessment

Yes at source level: current main returns 0 for resolveLlmIdleTimeoutMs({ trigger: "cron" }), and cron embedded runs pass trigger "cron" into the stream wrapper path that only activates for positive idle timeouts. I did not run a live forced stalled-provider replay.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff; removing the cron-wide opt-out is narrower than adding a new config surface and keeps explicit run/provider/agent timeout precedence plus the local-provider exception. If upgrade compatibility is preferred, the safer alternative is a default-preserving mode with an explicit strict watchdog policy.
