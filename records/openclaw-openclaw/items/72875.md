## What This Changes

The issue reports that `memorySearch.provider = "local"` can make memory CLI/runtime initialization fail with `Unknown memory embedding provider: local` even though the capability embedding CLI can use the same local provider.

## Best Possible Solution

Reconcile #72937 first: merge it if maintainers still want explicit manager-side registration, or close it as superseded by the shipped manifest-backed provider fallback, then close this issue once no separate live runtime failure remains.

## Reproduction Assessment

Yes. The issue gives a concrete 2026.4.23 config and `openclaw memory status --deep` failure path, and current source traces that path through `MemoryIndexManager.loadProviderResult` into the exact provider lookup that would throw if `local` cannot resolve.

## Solution Assessment

Unclear while #72937 remains open. The manifest-backed capability lookup is the more architecture-aligned fix because it follows the plugin contract documented for standalone CLI paths, but maintainers should decide whether the PR's explicit registration is useful defense in depth or redundant.
