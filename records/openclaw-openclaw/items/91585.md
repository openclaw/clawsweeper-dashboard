## Label Justifications

- `P2`: This is a concrete integration bug affecting OAuth-backed MCP servers in one Codex app-server workflow, with limited blast radius but real user impact.
- `impact:auth-provider`: The failure is caused by authentication and credential ownership not surviving into the Codex-native MCP tool-discovery path.

## Summary

Keep open: current main still projects an OAuth-backed HTTP MCP server into Codex without transferring OpenClaw’s stored OAuth session or tool filter, and no merged or open canonical replacement owns the remaining work. Codex has its own supported OAuth configuration and credential store, so choosing between a bridge, Codex-native authentication, or an explicit unsupported-projection diagnostic requires maintainer judgment.

## What This Changes

Ensure OAuth-authenticated OpenClaw MCP servers remain usable as native tools in Codex app-server sessions, or report clearly when safe native projection is unavailable.

## Best Possible Solution

Choose one documented owner boundary: securely materialize the OpenClaw-authenticated and filtered MCP surface for Codex, map users into Codex’s native OAuth lifecycle when contracts are equivalent, or omit the native server and emit an actionable diagnostic instead of projecting an unusable URL.

## Reproduction Assessment

Yes, at source level with high confidence: an OAuth HTTP server without explicit projected headers becomes a Codex server containing its URL but none of OpenClaw’s usable OAuth session state. The live Robinhood OAuth flow was not rerun during this read-only review.

## Solution Assessment

Unclear: the report correctly identifies the broken boundary, but the best repair depends on whether OpenClaw or Codex should own OAuth login, credential storage, token refresh, and tool filtering for native sessions; unsupported fields should not be passed through blindly.
