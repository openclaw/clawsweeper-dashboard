## Summary

Keep open: current main still projects the reported OAuth-backed HTTP MCP server into Codex native `mcp_servers` with only Codex-supported launch/header/approval fields, while OpenClaw's own MCP runtime consumes OAuth and tool-filter metadata. The open candidate PR is the right active path, but it is not merged and remains a maintainer config/auth contract decision.

## What This Changes

The issue asks OpenClaw to stop projecting OAuth-backed HTTP MCP servers into Codex app-server sessions as unusable bare URL-only native MCP entries.

## Best Possible Solution

Land a rebased, maintainer-approved version of the linked Codex OAuth projection PR, or choose an explicit unsupported-projection diagnostic if native Codex OAuth handoff is not the desired contract.

## Reproduction Assessment

Yes from source: current main normalizes an OAuth HTTP MCP server with no headers into a Codex native MCP entry containing only `url`. I did not run the live Robinhood OAuth flow in this read-only review.

## Solution Assessment

Unclear: the issue is real, but the best fix depends on maintainer choice between Codex-native OAuth descriptor mapping, an OpenClaw-authenticated bridge/materializer, or a diagnostic for unsupported native projection.
