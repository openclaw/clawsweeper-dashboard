The issue asks OpenClaw to preserve, bridge, or diagnose OAuth-backed HTTP MCP server metadata when projecting `mcp.servers` into native Codex app-server `mcp_servers`.

## Best Possible Solution

Keep this issue open as the canonical tracker and resolve it through an owner-reviewed version of https://github.com/openclaw/openclaw/pull/87478 or an equivalent bridge/diagnostic design that preserves shipped MCP OAuth behavior.

## Reproduction Assessment

Yes from source. Current main projects the reporter's HTTP MCP server through `normalizeCodexMcpServerConfig`, which carries URL/header fields but not the Codex OAuth fields that sibling Codex source accepts for streamable HTTP servers.

## Solution Assessment

Unclear until maintainers choose the config/auth contract. Passing `oauth_resource` and `oauth.client_id` is plausible because Codex supports them, but the best fix must decide whether those descriptors live in shared MCP config, Codex-specific metadata, or an OpenClaw-authenticated bridge/diagnostic path.
