## Summary

Keep open: current main and v2026.6.1 still project the reported user MCP server through a Codex-native config helper that copies URL/header launch fields but not OpenClaw OAuth/session/filter metadata, while the OpenClaw runtime path does use that metadata successfully.

## What This Changes

The issue reports that OAuth-backed HTTP MCP servers configured in OpenClaw are projected into native Codex app-server sessions as a bare URL, so Codex cannot discover the authenticated tools.

## Best Possible Solution

Choose one owner-approved behavior for OAuth user MCP servers: bridge OpenClaw-authenticated tools into the Codex surface, map only Codex-supported OAuth config with clear login diagnostics, or omit the native projection with an explicit diagnostic instead of sending an unusable bare URL.

## Reproduction Assessment

Yes from source: the current projection helper turns `mcp.servers.*` into Codex `mcp_servers` using only common launch/header fields, so an OAuth HTTP server with no headers becomes a bare URL. I did not run the live Robinhood OAuth flow in this read-only review.

## Solution Assessment

Unclear: the report identifies a real failure, but the best fix needs an owner decision because OpenClaw OAuth state and Codex-native MCP OAuth state are separate contracts. A diagnostic-only fix may be safer than silently passing unsupported fields through.
