## Summary

Keep open: current main still sends the reported user MCP server through a Codex projection that preserves URL/common launch fields, headers, and Codex approval mode only, so an OAuth HTTP server with no supported headers still becomes a bare URL. The related open PR is the right canonical work item, but it still needs maintainer config/auth contract review before this issue can close.

## What This Changes

The issue requests that OAuth-backed HTTP MCP servers configured in OpenClaw survive Codex app-server projection or produce an explicit diagnostic instead of an unusable bare URL.

## Best Possible Solution

Review and rebase https://github.com/openclaw/openclaw/pull/87478 or a successor that preserves shipped OpenClaw MCP OAuth fields while adding only owner-approved Codex descriptor projection or diagnostics.

## Reproduction Assessment

Yes from source: current main projects an enabled user HTTP MCP server through normalizeCodexMcpServerConfig, which emits only the URL when no supported headers or Codex approval metadata are present. I did not run the live Robinhood OAuth flow in this read-only review.

## Solution Assessment

Unclear: the bug is real, but the best fix requires maintainer choice between an OpenClaw-authenticated bridge, Codex-native OAuth descriptor projection, or an explicit unsupported-projection diagnostic.
