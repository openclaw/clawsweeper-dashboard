## What This Changes

## Best Possible Solution

Recover selected-version compatibility from the version endpoint when artifact metadata is sparse, keep package-level fallback only for selected latest, and require redacted after-fix CLI proof before merge.

## Reproduction Assessment

Yes, by source inspection plus live ClawHub API responses: the artifact endpoint can omit compatibility while the selected version endpoint still advertises it. I did not run the local CLI in this read-only review.

## Solution Assessment

No. The latest-only package fallback is a plausible mitigation for the original pinned symptom, but the best fix must preserve selected-version compatibility rather than treating sparse artifact metadata as unrestricted.
