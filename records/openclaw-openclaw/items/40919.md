## Summary

Keep open: current main and the latest release still use the full session reindex path, while the open replacement PR for incremental session sync is conflicting and waiting on design/repair work.

## What This Changes

The issue reports that growing session JSONL memory files are fully reindexed by deleting and reinserting all session chunks instead of using incremental indexing.

## Best Possible Solution

Land or replace the open memory-core incremental session indexing PR with a bounded design that preserves chunk-overlap correctness, provider batching behavior, and storage upgrade safety.

## Reproduction Assessment

Yes, at source level: changed session transcripts reach `indexFile`, and `writeChunks` clears existing rows before rewriting regenerated chunks. I did not run a live long-session benchmark because this review is read-only.

## Solution Assessment

No final fix is merged yet. The open PR is a plausible direction, but the best solution needs maintainer review for source-wide batching, chunk-overlap correctness, and persistent storage/API contract safety.
