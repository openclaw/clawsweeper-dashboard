## Summary

This issue is still necessary: current main still rebuilds dirty session transcript entries and the shared writer still deletes existing chunk/vector/FTS rows before inserting the rebuilt chunks. An open candidate PR now owns the same problem, but it is unmerged and conflicting, so this issue should stay open until that PR lands or a narrower replacement is chosen.

## What This Changes

The issue requests incremental indexing for session transcript memory sync so growing JSONL files are not fully deleted, rebuilt, and re-embedded on each dirty sync.

## Best Possible Solution

Land a rebased memory-core fix that incrementally indexes changed session transcripts across direct and source-wide batch sync paths while keeping storage and SDK surface changes intentionally reviewed.

## Reproduction Assessment

Yes, at source level: current main builds full dirty session entries, passes them to indexFile, and writeChunks clears existing indexed rows before reinserting chunks. I did not run a live 50-turn benchmark in this read-only review.

## Solution Assessment

Unclear for the exact implementation: incremental indexing is the right direction, but the current open PR still needs rebase plus source-wide batch and storage/API contract decisions before it is the best merge shape.
