## Label Justifications

- `P2`: This is a normal-priority memory performance bug with real embedding, CPU, and write-amplification cost but limited blast radius.
- `impact:session-state`: The affected code repeatedly rewrites session transcript chunks in the memory search index as session state grows.

## Summary

Keep open: current main and the latest release still rebuild changed session transcript entries through the full index path, and the only live implementation candidate is still an open PR with unresolved review/merge blockers.

## What This Changes

The issue asks OpenClaw memory sync to incrementally index appended session JSONL content instead of deleting and reinserting all chunks for a growing transcript.

## Best Possible Solution

Land a rebased memory-core fix that incrementally indexes appended session transcript content across direct and source-wide batch sync paths, with focused regression coverage and explicit storage/API contract review.

## Reproduction Assessment

Yes, source-level: current main builds a complete session entry for dirty transcript files, calls indexFile, and writeChunks clears all rows for that path/source before reinserting. I did not run a live timing benchmark, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No, not yet. The incremental-sync direction is right, but the current open PR still needs to cover direct and source-wide batch providers and settle storage/API contract concerns before it is the best merge shape.
