## Summary

Keep open: current `main` and latest release still rebuild changed session transcripts through the full session entry path, and the open candidate PR is not merged or merge-ready yet.

## What This Changes

The issue asks OpenClaw to avoid full delete-and-reinsert session JSONL indexing by adding incremental session memory sync for growing transcripts.

## Best Possible Solution

Land a reviewed memory-core fix that preserves existing session chunks when only appended transcript content changes, covers direct and source-wide batch providers, and keeps storage/API changes explicit and tested.

## Reproduction Assessment

Yes, at source level: changed session files still flow through `buildSessionEntry` and `indexFile`, and `writeChunks` clears existing chunk/vector/FTS rows before reinserting them. I did not run a live 50-turn benchmark in this read-only review.

## Solution Assessment

Unclear as proposed: incremental indexing is the right direction, but a line-offset-only fix is not enough without chunk-overlap, source-wide provider, and persisted-index upgrade decisions. The existing open PR is the best current lane, but it needs repair and maintainer review before it can close this issue.
