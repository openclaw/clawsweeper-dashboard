## Summary

Keep open. Current main still rebuilds changed session transcripts from the full JSONL file and writes them through the shared clear-and-reinsert path, while open PR #75179 remains the active implementation candidate.

## What This Changes

The issue requests incremental session memory indexing for growing session JSONL transcripts instead of repeatedly deleting and reinserting all chunks.

## Best Possible Solution

Review, land, or replace #75179 with a session-specific incremental indexing path that preserves valid existing chunks, records durable append or chunk metadata, and falls back to full reindex when a transcript is rewritten or unsafe to update incrementally.

## Reproduction Assessment

Yes, at source level. With session memory enabled and a transcript crossing `sync.sessions` thresholds, current main reaches `syncSessionFiles`, rebuilds the full session entry, and writes it through `indexFile`/`writeChunks`.

## Solution Assessment

Unclear for the issue's raw offset-based proposal because chunk overlap and transcript rewrites need explicit handling. The narrow maintainable solution is a session-specific incremental writer with durable metadata and full-reindex fallbacks, which should be evaluated through #75179 or a replacement.
