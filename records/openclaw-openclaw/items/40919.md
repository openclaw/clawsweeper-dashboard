## Summary

Keep open. Current main still gates which session transcripts are selected for sync, but a changed transcript is rebuilt from the full JSONL file and written through the shared clear-and-reinsert path; the currently open #75179 remains the implementation candidate for the remaining incremental-indexing work.

## What This Changes

The issue requests incremental session-memory sync for growing JSONL transcripts so unchanged chunks are not repeatedly deleted, re-embedded, and reinserted.

## Best Possible Solution

Review, land, or replace #75179 with a session-specific incremental indexing path that preserves valid existing chunks, records durable append or chunk metadata, and falls back to full reindex when a transcript is rewritten or unsafe to update incrementally.

## Reproduction Assessment

Yes. With session memory enabled and a growing JSONL transcript crossing `sync.sessions` thresholds, the source path deterministically reaches `syncSessionFiles`, rebuilds the full entry with `buildSessionEntry`, and writes it through `indexFile`/`writeChunks`; no runtime test was run because this was a read-only review.

## Solution Assessment

Unclear for the exact offset-based proposal because chunk overlap and transcript rewrites need careful handling. The maintainable direction is a session-specific incremental writer with hash or cursor metadata and explicit full-reindex fallbacks, not only dirty-file selection.
