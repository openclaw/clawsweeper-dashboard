## Summary

Keep open: current main and the latest release still rebuild dirty session transcript indexes through the full session-entry path, while the active replacement work is still the open memory-core PR and needs maintainer review/proof before the issue can close.

## What This Changes

The issue asks for session memory sync to avoid full delete-and-reinsert indexing for growing session JSONL transcripts, preferably by indexing only appended or changed content.

## Best Possible Solution

Land a reviewed, proof-backed memory-core implementation that preserves existing vector/FTS/session state while rewriting only the changed transcript chunks and retaining safe full-reindex fallbacks.

## Reproduction Assessment

Yes at source level. Current main still routes a dirty changed transcript through full `buildSessionEntry` and `indexFile`, whose write path clears indexed rows before writing regenerated chunks; I did not run a live sync in this read-only review.

## Solution Assessment

Unclear. Incremental indexing is the right direction, but the open PR still needs maintainer review for chunk overlap, stored metadata, vector/FTS consistency, and upgrade safety before it is clearly the best fix.
