## Label Justifications

- `P2`: This is a normal-priority performance bug in session memory sync with limited blast radius.
- `impact:session-state`: The report concerns session transcript memory indexing and the cost of keeping that session state searchable.

## Summary

Current main still rebuilds changed session transcript indexes end to end: it reads and flattens the whole JSONL file, rechunks it, and clears existing rows before writing new chunks. The issue should stay open while the paired implementation at https://github.com/openclaw/openclaw/pull/75179 remains unresolved.

## What This Changes

The issue requests incremental indexing for growing session JSONL memory files instead of periodically rebuilding the whole session index.

## Best Possible Solution

Review, narrow, or replace the open paired memory-core implementation so dirty session transcript sync can reuse unchanged chunks safely across overlap and compaction cases.

## Reproduction Assessment

Yes, at source level: a dirty session file still goes through full `buildSessionEntry` parsing and full `indexFile` replacement. I did not run a live long-session benchmark in this read-only review.

## Solution Assessment

Yes, incremental indexing is the right bug-fix direction, but it needs a careful memory-core implementation that preserves overlap, hash, and compaction behavior rather than a config-only workaround.
