## Summary

Keep open. Current main still only gates which session transcripts are selected for sync; once a changed session is selected, it is rebuilt from the full JSONL file and written through the shared clear-and-reinsert indexing path. The related #59577 implementation candidate remains open, so the remaining work is not implemented on main.

## What This Changes

The issue requests replacing session memory's full delete-and-reinsert sync for growing JSONL transcripts with incremental indexing, archival, or retention behavior.

## Best Possible Solution

Add a session-specific incremental indexing path that preserves valid existing chunks, stores durable append or chunk metadata, updates vector, FTS, chunk, and file state consistently, and falls back to full reindex when a transcript is truncated, compacted, rechunked, or otherwise unsafe to update incrementally.

## Reproduction Assessment

Yes. With session memory enabled and `sources` including `sessions`, appending enough JSONL data to cross the session delta threshold deterministically reaches `syncSessionFiles`, which rebuilds the selected session via `buildSessionEntry` and writes it through the clear-and-reinsert path.

## Solution Assessment

Unclear for the exact `lastSyncOffset` proposal because the issue discussion correctly flags chunk-overlap boundaries. The safer direction is a reviewed session-specific incremental writer backed by hash or chunk metadata with full-reindex fallbacks for unsafe changes.
