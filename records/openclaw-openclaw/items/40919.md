## Summary

Keep open. Current main still rebuilds changed session transcripts from the full JSONL content and writes them through the shared clear-and-reinsert path; the active fix path is the open implementation PR at https://github.com/openclaw/openclaw/pull/75179, which already has focused review blockers to resolve.

## What This Changes

The issue requests incremental session-memory indexing so growing JSONL transcripts do not repeatedly delete, rechunk, re-embed, and reinsert unchanged session content.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/75179 or a replacement that preserves valid existing session chunks, records durable chunk metadata, and falls back to full reindex when transcript rewrites or overlap make incremental reuse unsafe.

## Reproduction Assessment

Yes, at source level. With session memory enabled and a transcript crossing the configured delta thresholds, current main reaches syncSessionFiles, rebuilds the full session entry, and writes it through indexFile/writeChunks, which clears existing rows first.

## Solution Assessment

The incremental direction is the right fix, but the raw lastSyncOffset proposal is not sufficient because chunk overlap and transcript rewrites need explicit handling. The maintainable solution is a session-specific incremental writer with durable chunk metadata and safe full-reindex fallbacks.
