## Summary

Keep open. Current main still rebuilds a changed session transcript from the full JSONL file and writes it through the shared clear-and-reinsert path; the open #75179 remains the implementation candidate for the remaining incremental session-indexing work.

## What This Changes

Request incremental session memory indexing for growing JSONL transcripts to avoid full delete/reinsert work and re-embedding unchanged chunks.

## Best Possible Solution

Review, land, or replace #75179 with a session-specific incremental indexing path that preserves valid existing chunks, records durable append or chunk metadata, and falls back to full reindex when a transcript is rewritten or unsafe to update incrementally.

## Reproduction Assessment

Yes, at source level. With session memory enabled and a transcript crossing `sync.sessions` thresholds, `processSessionDeltaBatch` reaches `syncSessionFiles`, which rebuilds the full entry with `buildSessionEntry` and writes it through `indexFile`/`writeChunks`; no live performance run was executed in this read-only review.

## Solution Assessment

Unclear for the issue's raw offset-based proposal because chunk overlap and transcript rewrites need explicit handling. The maintainable fix is a session-specific incremental writer with durable metadata and full-reindex fallbacks, which is what #75179 or a replacement should be reviewed for.
