## Summary

Keep open: the missing `openclaw status` session row is documented as expected for a quiet Discord channel after restart, but the separate claim that tagged messages are not received after restart is a plausible behavior bug and is too recent/specific to close without live channel-health proof or a traced source root cause.

## What This Changes

Configured Discord channel sessions should continue receiving and processing messages after `openclaw gateway restart`.

## Best Possible Solution

Treat the session-row absence as expected diagnostics behavior, then verify the no-response symptom with `openclaw channels status --probe`, `openclaw status --deep`, and sanitized gateway diagnostics; if confirmed, add a focused Discord restart/inbound regression around provider lifecycle and message listener registration.

## Reproduction Assessment

No. The reporter gives concrete steps and logs, but this review could not run a live Discord gateway restart, and source inspection did not reveal a deterministic failing path on current main.

## Solution Assessment

Unclear. Pre-creating session rows is not the right fix because docs define session rows as conversation state, not liveness; the maintainable fix depends on proving why the post-restart Discord listener stops receiving messages, if it does.
