## What This Changes

Change the `openai-responses` adapter to send the system prompt through top-level `instructions` rather than as a system or developer item in `input`.

## Best Possible Solution

Define one canonical prompt-placement rule for the generic Responses adapter, then land the linked PR with request-body coverage proving the system prompt appears exactly once and with upgrade proof for both direct OpenAI and supported compatible endpoints.

## Reproduction Assessment

Yes for the reported payload shape: current main deterministically includes the system prompt in converted `input` and omits top-level `instructions`. The report does not include evidence that a live OpenAI or compatible endpoint rejects or misinterprets that validly constructed payload.

## Solution Assessment

Unclear. The linked two-field change is the narrowest way to match the existing Codex adapter, but maintainers must first decide whether that policy is safe for every custom provider using the generic `openai-responses` adapter and require payload-level proof.
