## What This Changes

The issue asks OpenClaw to send `openai-responses` system prompts through the top-level Responses API `instructions` field instead of as `system` or `developer` messages in `input`.

## Best Possible Solution

Keep this issue open while maintainers choose one canonical linked PR and land a compatibility-scoped request-shape fix that preserves providers requiring `system` input messages unless live provider proof warrants a broader switch.

## Reproduction Assessment

Yes for the serialized request shape: current source and the latest release still place non-Codex `openai-responses` system prompts in `input` and omit top-level `instructions`. No live provider failure has been reproduced from the issue data.

## Solution Assessment

No final fix direction is proven yet. An unconditional switch to top-level `instructions` is not clearly the best solution because current OpenAI docs allow system/developer input messages and OpenClaw has compatibility handling for providers without developer-role support.
