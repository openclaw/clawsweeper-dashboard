## Summary

Current `main` still emits the reported payload, but the official OpenAI contract does not establish that this is invalid: it supports both top-level `instructions` and developer/system input messages. Keep this issue open while the clean linked PR is reviewed as a compatibility/product-policy change, including whether generic, Azure, and Codex Responses adapters should intentionally use different system-instruction serialization.

## What This Changes

Change the custom `openai-responses` adapter to send its system prompt through top-level `instructions` instead of a developer/system item in `input`.

## Best Possible Solution

Define and document the intended instruction-serialization boundary for generic, Azure, and Codex Responses providers, then add emitted-request regression tests and change only the adapters whose provider contract or observed behavior requires top-level `instructions`.

## Reproduction Assessment

No. We can reproduce the reported payload shape from current source, but official OpenAI documentation accepts developer/system input messages as an instruction mechanism, and no live request failure or semantic regression is shown.

## Solution Assessment

No, not yet. The linked two-line change is plausible and matches the Codex-specific adapter, but it assumes a universal provider invariant that the official contract and Azure sibling do not establish.
