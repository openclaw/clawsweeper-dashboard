## Summary

Keep open: current main still contains the reported Chrome startup failure path, and the latest shipped release shows the same profile-resolution behavior. A host-only loopback profile CDP URL overrides the explicit configured CDP port to 80, then the managed Chrome launcher passes that privileged port to Chrome.

## What This Changes

The issue asks OpenClaw to start Chrome for a local browser profile configured with a host-only loopback CDP URL and explicit CDP port instead of failing on port 80.

## Best Possible Solution

Fix browser profile resolution so a local managed profile with a host-only loopback `cdpUrl` plus explicit `cdpPort` keeps the explicit port or fails early with a clear config error, with regression coverage for the reported shape.

## Reproduction Assessment

Yes. Source inspection on current main follows the reported config to `cdpPort = 80` before Chrome launch; I did not run a live macOS Chrome process in this read-only review.

## Solution Assessment

No. The current implementation is not the best solution because it converts a local managed profile config into a privileged-port bind failure; the narrower maintainable fix belongs in browser profile normalization or validation while preserving remote CDP defaults.

The reported browser profile has `cdpUrl: "http://127.0.0.1"` and `cdpPort: 18800`; current `resolveProfile` parses the host-only URL to port 80 and `launchOpenClawChrome` passes that to `--remote-debugging-port`, causing Chrome startup to fail.
