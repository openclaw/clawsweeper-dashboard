## Summary

Keep open: current main still resolves the reported managed browser profile to CDP port 80 before launching Chrome, so the reported startup failure remains source-reproducible and narrow enough for a focused browser plugin fix.

## What This Changes

The issue reports that `openclaw browser start` fails on macOS when a managed Chrome profile has `cdpPort: 18800` plus a host-only `cdpUrl`, producing a privileged port 80 bind error.

## Best Possible Solution

Normalize or validate browser profiles so local managed loopback profiles with an explicit `cdpPort` do not bind Chrome to default port 80, while remote CDP default-port behavior and stale WebSocket cleanup continue to work.

## Reproduction Assessment

Yes. From source inspection, the reported profile follows `resolveProfile` from `cdpPort: 18800` to `parseBrowserHttpUrl("http://127.0.0.1")`, yielding port 80 before the Chrome launch args are built.

## Solution Assessment

Not applicable as a proposed patch; the narrow maintainable fix belongs in browser profile normalization or early validation, not in Chrome launch retry logic.
