## Summary

Current `main` still has the reported failure path: a managed profile with a host-only loopback `cdpUrl` and explicit `cdpPort` resolves to port 80 before Chrome launch, so the issue should stay open for a focused browser-plugin fix.

## What This Changes

The issue reports that `openclaw browser start` cannot start managed Chrome because the configured profile launches CDP on port 80 instead of the configured port 18800.

## Best Possible Solution

Handle this in the browser plugin by preventing managed local profiles from turning a loopback host-only `cdpUrl` plus explicit `cdpPort` into a privileged port-80 launch, with regression coverage for the reported config.

## Reproduction Assessment

Yes. Source inspection on current `main` follows the reported config through `resolveProfile`, `parseBrowserHttpUrl`, and `buildOpenClawChromeLaunchArgs`, producing port 80 for Chrome; I did not run a live macOS Chrome launch in this read-only sweep.

## Solution Assessment

No. Current `main` is not the best solution because it converts a docs-disallowed local managed config into a low-level Chrome bind failure; the narrower fix is to normalize to the explicit local `cdpPort` or reject the config before spawning Chrome with a clear message.
