## Summary

Current `main` still reproduces the reported failure path in source: a managed profile with `cdpUrl: "http://127.0.0.1"` and `cdpPort: 18800` resolves to port 80 before Chrome launch, matching the macOS permission-denied error. The issue should stay open and is narrow enough for a focused browser-plugin fix PR.

## What This Changes

The issue reports that `openclaw browser start` fails to start managed Chrome on macOS when the profile config sets `cdpUrl` to `http://127.0.0.1` and `cdpPort` to `18800`.

## Best Possible Solution

Fix the browser plugin so managed local profiles cannot silently turn a loopback host-only `cdpUrl` plus explicit `cdpPort` into a privileged port-80 launch; either normalize to the explicit `cdpPort` or reject the config before spawning Chrome with a clear message.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the reported profile config enters `resolveProfile`, `parseBrowserHttpUrl` supplies port 80, and `buildOpenClawChromeLaunchArgs` passes that port to Chrome.

## Solution Assessment

No. Current main is not the best solution because it converts malformed docs-disallowed local profile config into a low-level Chrome bind failure; normalization or early validation would be narrower and more actionable.
