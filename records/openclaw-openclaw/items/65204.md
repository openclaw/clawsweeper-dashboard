## Summary

Keep open: current main still resolves the reported managed Chrome profile to port 80 by reparsing the host-only profile CDP URL, then passes that port into Chrome launch, and no open replacement PR or canonical issue owns the remaining fix.

## What This Changes

The issue reports that `openclaw browser start` cannot start Chrome when a managed `openclaw` profile has `cdpUrl: "http://127.0.0.1"` and `cdpPort: 18800`.

## Best Possible Solution

Normalize managed browser profile CDP fields so an explicit `cdpPort` is not overwritten by a host-only loopback `cdpUrl`, or fail early with an actionable config error before launching Chrome, while preserving remote CDP semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: the reported config enters `resolveProfile()`, `http://127.0.0.1` parses to port 80, and that port is passed to Chrome. I did not run a live macOS Chrome process in this read-only review.

## Solution Assessment

No, the current implementation is not the best solution because it silently turns a local managed profile's explicit port into a privileged port-80 bind attempt. The narrow maintainable fix belongs in profile normalization or validation with regression coverage for the reported config shape.
