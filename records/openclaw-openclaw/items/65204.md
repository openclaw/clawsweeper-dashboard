## Summary

Keep open: current main still routes the reported browser profile shape to CDP port 80 before launching Chrome, so the startup failure remains source-reproducible and is narrow enough for a focused browser plugin fix.

## What This Changes

The issue reports that `openclaw browser start` tries to launch Chrome CDP on privileged port 80 when a managed profile has `cdpUrl: "http://127.0.0.1"` and `cdpPort: 18800`.

## Best Possible Solution

Normalize or validate browser profiles so local managed loopback profiles with an explicit `cdpPort` keep that port, while remote CDP default-port behavior and stale WebSocket cleanup continue to work.

## Reproduction Assessment

Yes, from source inspection. The reported config follows `resolveProfile` from `cdpPort: 18800` to `parseBrowserHttpUrl("http://127.0.0.1")`, yielding port 80 before Chrome launch.

## Solution Assessment

No. The current implementation is not the best solution because it converts an ambiguous local managed profile into a privileged-port Chrome bind failure; the narrower maintainable fix belongs in browser profile normalization or early config validation.
