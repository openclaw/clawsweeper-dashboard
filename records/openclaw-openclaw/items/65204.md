## Summary

Keep open. Current main still has a source-level path that turns the reported `browser.profiles.openclaw.cdpUrl: "http://127.0.0.1"` into CDP port 80 before launching Chrome, which matches the exact macOS permission-denied failure in the report. The docs now say local managed `openclaw` profiles should not set `cdpUrl`, but the runtime still silently accepts this config and launches a privileged port instead of normalizing to `cdpPort` or failing with an actionable config error.

## What This Changes

The issue asks OpenClaw to handle or diagnose a browser config where a managed Chrome profile has both `cdpUrl: "http://127.0.0.1"` and `cdpPort: 18800`, but `openclaw browser start` tries to start Chrome on port 80.

## Best Possible Solution

Keep the issue open and fix the browser plugin so a managed local profile cannot silently turn a host-only loopback `cdpUrl` plus `cdpPort` into a privileged port-80 launch. The maintainable endpoint is either to normalize that config to the explicit `cdpPort` or reject it before launch with a clear config/doctor message and docs-aligned guidance.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with the reported profile config, `resolveProfile` takes the host-only `cdpUrl`, `parseBrowserHttpUrl` supplies port 80, and `launchOpenClawChrome` passes that resolved port to Chrome.

## Solution Assessment

No. The current behavior is not the best solution because it converts a common malformed local-profile config into a low-level Chrome bind failure; the safer fix is browser-config normalization or an early actionable validation error before spawning Chrome.
