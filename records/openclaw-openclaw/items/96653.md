## Summary

Keep open: current main still routes reply-time agent-runtime SecretRefs through gateway `secrets.resolve` before local fallback, and https://github.com/openclaw/openclaw/pull/96661 is the open candidate fix for the same log-flood bug. The closed https://github.com/openclaw/openclaw/pull/96914 was already superseded rather than merged.

## What This Changes

The issue asks OpenClaw to avoid noisy per-reply gateway `secrets.resolve` `UNAVAILABLE` logs for active exec-backed model-provider SecretRefs while preserving successful local fallback.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/96661 is either landed as the focused fix or rejected in favor of a narrower gateway-safe alternative.

## Reproduction Assessment

Yes at source level: queued replies still call `resolveCommandSecretRefsViaGateway` for agent-runtime targets, those targets include `models.providers.*.apiKey`, and current main only pre-skips gateway credential exec refs before attempting `secrets.resolve`. I did not run a live gateway reply turn in this read-only review.

## Solution Assessment

Yes for the main bug if maintainers accept the boundary: the shared command-secret resolver is the right layer to avoid an expected useless gateway RPC while preserving local fallback. The underlying-error visibility ask should remain redacted and may be handled as a follow-up if the skip fix lands first.
