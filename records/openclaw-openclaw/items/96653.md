## What This Changes

Issue requests that reply-path command-secret resolution stop producing per-turn gateway `secrets.resolve` `UNAVAILABLE` noise for exec-backed model-provider SecretRefs while preserving successful local fallback.

## Best Possible Solution

Land one canonical shared resolver fix that avoids the futile gateway RPC for active exec-backed agent-runtime/model-provider targets, preserves gateway resolution for active non-exec targets, and keeps no-local-exec callers safe.

## Reproduction Assessment

Yes, by source inspection: queued replies call the shared command-secret resolver for agent-runtime targets, and current main still attempts gateway `secrets.resolve` unless the narrow gateway-credential exec skip applies. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes: the narrow shared command-secret resolver boundary is the maintainable fix, not reply-only log suppression. The current best path is to review and land https://github.com/openclaw/openclaw/pull/96661 or an equivalent guarded replacement.
