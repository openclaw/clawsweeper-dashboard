## What This Changes

The issue asks OpenClaw to avoid noisy per-reply `secrets.resolve` `UNAVAILABLE` logs for exec-backed model-provider SecretRefs and to improve diagnosability of the underlying gateway failure.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/96661 or choose a different core resolver boundary; keep broader cross-context exec resolver work separate unless maintainers intentionally fold it in with redacted diagnostics.

## Reproduction Assessment

Yes at source level: queued replies call command-secret resolution for agent-runtime targets that include `models.providers.*.apiKey`, while current main only pre-skips gateway credential exec refs. I did not run a live macOS gateway turn in this read-only review.

## Solution Assessment

Likely yes for the main bug: the shared command-secret resolver is the right boundary, and the open candidate PR targets active exec refs while preserving inactive-target and no-local-exec safeguards. The diagnostic-detail ask should be handled as redacted operator-safe detail or a follow-up.
