## Summary

Keep open: current main and the latest release still route queued replies through gateway `secrets.resolve` for agent-runtime targets that include model-provider SecretRefs, and the only preflight skip remains limited to gateway credential exec refs. The open fix candidate at https://github.com/openclaw/openclaw/pull/96661 now owns the likely implementation path, but it has not merged and still needs maintainer acceptance of the local-exec resolution boundary.

## What This Changes

The issue reports noisy per-reply gateway `secrets.resolve` `UNAVAILABLE` logs for exec-backed model-provider SecretRefs, despite successful local fallback resolution.

## Best Possible Solution

Review and land a focused shared resolver fix, likely through https://github.com/openclaw/openclaw/pull/96661 if maintainers accept the local-exec boundary, while keeping any failure detail redacted and operator-safe.

## Reproduction Assessment

Yes at source level: queued replies still call `resolveCommandSecretRefsViaGateway` with agent-runtime targets that include `models.providers.*.apiKey`, and current main only pre-skips gateway credential exec refs before calling `secrets.resolve`. I did not run a live gateway reply turn in this read-only review.

## Solution Assessment

Yes for the central bug: the shared command-secret resolver is the narrow owner boundary, and skipping the expected useless RPC for active exec-backed command targets is cleaner than adding a config knob. The underlying-error visibility ask should be solved separately or with strictly redacted diagnostics.
