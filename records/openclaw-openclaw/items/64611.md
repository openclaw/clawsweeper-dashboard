This PR changes heartbeat task due calculation to treat future last-run timestamps as due and adds focused `isTaskDue` unit coverage.

## Best Possible Solution

## Reproduction Assessment

Yes. From source inspection on current main, `isTaskDue(now + 5 * 60 * 1000, "30m", now)` follows the elapsed-subtraction path and returns false because the elapsed value is negative; when all tasks fail this check, the runner skips as `no-tasks-due`.

## Solution Assessment

Yes. Updating the shared `isTaskDue` helper is the narrowest maintainable fix because due-task filtering and task timestamp advancement both depend on it, and the added tests preserve normal elapsed/not-elapsed behavior.
