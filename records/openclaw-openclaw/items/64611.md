## What This Changes

This PR updates `isTaskDue` to treat future `lastRunMs` values as due and adds unit coverage for never-run, elapsed, not-yet-elapsed, and clock-skew cases.

## Best Possible Solution

Land or cherry-pick the clock-skew guard with focused unit coverage after proof and current-main validation, accepting future heartbeat task timestamps as invalid last-run state.

## Reproduction Assessment

Yes. Source inspection of current main shows `isTaskDue(now + 5m, "30m", now)` falls through to a negative elapsed-time comparison, and the runner uses that result to skip task-mode heartbeats.

## Solution Assessment

Yes. Updating `isTaskDue` is the narrow shared fix because both due-task filtering and post-run timestamp updates call the helper; runner-local clamping would duplicate the timing invariant.
