## Summary

Keep open: current main and the latest release still lack the future-timestamp guard, and this is a focused heartbeat task scheduling bug fix. The patch looks correct, but the external PR still needs real behavior proof and a refresh against current main before merge.

## What This Changes

The PR treats future `heartbeatTaskState` last-run timestamps as due in `isTaskDue` and adds focused unit coverage for never-run, elapsed, not-yet-elapsed, and clock-skew cases.

## Best Possible Solution

Rebase or cherry-pick the clock-skew guard onto current main, keep the focused `isTaskDue` regression tests, and require redacted real heartbeat proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.9` compare `nowMs - lastRunMs >= intervalMs`, so a future `lastRunMs` is treated as not due. I did not run tests because this review is read-only.

## Solution Assessment

Yes, this is the best fix layer: both task selection and timestamp advancement share `isTaskDue`, so one helper guard preserves the scheduling invariant. A runner-only guard would duplicate the same state rule.
