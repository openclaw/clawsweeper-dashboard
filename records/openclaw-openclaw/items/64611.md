## What This Changes

This PR treats future `lastRunMs` values as due in `isTaskDue` and adds unit coverage for never-run, elapsed, not-yet-elapsed, and future timestamp cases.

## Best Possible Solution

Refresh or cherry-pick the one-line helper guard plus tests, then land after redacted runtime proof and explicit maintainer acceptance of the future timestamp behavior.

## Reproduction Assessment

Yes, source-reproducible. On current main, `isTaskDue(now + 5 * 60 * 1000, "30m", now)` follows the existing subtraction path and returns false, which then lets task-mode heartbeat skip as `no-tasks-due`.

## Solution Assessment

Yes. Updating the shared `isTaskDue` helper is the narrowest maintainable fix because both due-task selection and timestamp advancement use that helper; the remaining blockers are proof, refresh, and upgrade acceptance.
