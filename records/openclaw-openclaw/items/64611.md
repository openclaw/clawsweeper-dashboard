## What This Changes

The PR changes heartbeat task due calculation so a future `lastRunMs` is treated as due and adds focused unit coverage for never-run, elapsed, not-elapsed, and clock-skew cases.

## Best Possible Solution

Heartbeat task due checks should treat future last-run timestamps as due, keep normal elapsed/not-elapsed behavior intact, and merge only after maintainer review confirms the session-state semantics plus fresh proof.

## Reproduction Assessment

Yes, source-reproducible: current main compares `nowMs - lastRunMs` against the interval, so a future `lastRunMs` is negative and the task is skipped as not due. I did not run tests because this review is read-only.

## Solution Assessment

Yes, this is the best fix layer: `isTaskDue` is shared by task filtering and task timestamp updates, while docs and types define the stored value as last-run state rather than a future deferral. Runner-only guards or migrations would be broader and less direct.
