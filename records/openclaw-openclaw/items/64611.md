## What This Changes

The PR changes heartbeat task due checks to treat future last-run timestamps as due and adds unit coverage for normal and clock-skew cases.

## Best Possible Solution

Land a refreshed version of the narrow `isTaskDue` guard and focused tests after real behavior proof or maintainer runtime validation confirms skew recovery without changing normal due/not-due behavior.

## Reproduction Assessment

Yes, source-reproducible: on current main, `isTaskDue(now + 5 * 60 * 1000, "30m", now)` follows the subtraction path and returns false, so a skewed future last-run timestamp can leave a task not due.

## Solution Assessment

Yes, this is the best code location: both due-task selection and timestamp advancement use the shared `isTaskDue` helper, so a runner-only guard would duplicate the invariant.
