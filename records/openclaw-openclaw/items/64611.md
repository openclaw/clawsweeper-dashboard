Keep open: current main still has the source-reproducible heartbeat task clock-skew bug, and this PR is a narrow helper/test fix, but it remains proof-gated for an external contributor and needs a current-base validation refresh before merge.

## What This Changes

This PR changes heartbeat task due calculation to treat future `lastRunMs` timestamps as due and adds focused `isTaskDue` unit coverage.

## Best Possible Solution

Heartbeat task due checks should treat future last-run timestamps as clock skew, keep focused helper coverage, and merge or cherry-pick the guard after real proof and current-base validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: call `isTaskDue(now + 5 * 60_000, "30m", now)` on current main and it returns false because the elapsed delta is negative; I did not run a live NTP or VM-resume scenario in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes: the helper-level guard fixes both due-task filtering and timestamp semantics at the shared invariant, while a runner-local clamp would duplicate the timing rule.
