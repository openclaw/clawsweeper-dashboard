## Summary

Keep open: current main still has the callback-coupled producer bug, while this PR carries a narrow fix with regression coverage and sufficient real-behavior proof. The remaining issue is docs alignment, because the public reasoning docs still describe stream mode as Telegram-only.

## What This Changes

The PR decouples embedded Pi reasoning stream emission from the optional channel callback and adds regression tests for Gateway/WebSocket no-callback thinking events plus the thinking visibility gate.

## Best Possible Solution

Land this narrow callback-decoupling fix with docs alignment, and keep the broader capability-gated thinking-event design in https://github.com/openclaw/openclaw/pull/54821 as a separate follow-up if maintainers choose that boundary.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main gates streamReasoning on onReasoningStream, emitReasoningStream returns before the thinking event, and Gateway chat.send does not pass that callback. I did not run a live current-main Gateway failure in this read-only review.

## Solution Assessment

Yes for the runtime path: removing only the callback dependency while preserving canShowReasoning and the existing silentExpected guard is the narrow maintainable fix. The PR still needs docs alignment before it is ready to land cleanly.
