## Summary

Keep this PR open: it is a focused, source-backed fix for the Gateway/WebSocket reasoning-stream callback dependency, with supplied real behavior proof and green checks. The remaining issue is documentation alignment because public docs still describe `/reasoning stream` as Telegram-only.

## What This Changes

The PR decouples embedded Pi reasoning streaming from the optional channel callback, optional-chains the callback call, and adds no-callback plus `thinkingLevel: "off"` regression tests.

## Best Possible Solution

Land the narrow callback-decoupling fix after updating the public reasoning/WebChat docs; keep capability-gated thinking subscriptions and UI rendering as separate follow-up work.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: current main requires `onReasoningStream`, while Gateway `chat.send` does not provide that callback, so `stream: "thinking"` events are not produced for that path.

## Solution Assessment

Yes for the code path. Decoupling event emission from the optional channel callback while preserving `canShowReasoning` is the narrow fix; the safer remaining step is docs alignment, not a broader protocol redesign in this PR.
