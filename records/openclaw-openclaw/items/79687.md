## Summary

Keep this PR open: the runtime fix is focused, source-backed, proof-supplied, and CI is green, but the public reasoning docs still describe `stream` as Telegram-only and would become stale after merge.

## What This Changes

The PR decouples embedded Pi reasoning-stream emission from the optional `onReasoningStream` callback so Gateway/WebSocket clients can receive live `stream: "thinking"` events, and adds no-callback plus `thinkingLevel: "off"` regression tests.

## Best Possible Solution

Land the narrow callback-decoupling fix after updating the reasoning/WebChat docs; keep capability-gated subscriptions and UI rendering as separate coordinated follow-up work.

## Reproduction Assessment

Yes. Source tracing on current main shows `streamReasoning` requires `onReasoningStream`, while the Gateway `chat.send` path does not provide that callback; the PR also supplies after-fix Gateway/WebSocket proof.

## Solution Assessment

Yes for the runtime path. Decoupling `emitAgentEvent({ stream: "thinking" })` from the optional channel callback while preserving `canShowReasoning` is the narrow fix; the remaining change should be docs alignment, not a broader protocol redesign in this PR.
