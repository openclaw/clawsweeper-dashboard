## Summary

Keep open. Current main still resolves unset `memory.qmd.update.embedTimeoutMs` to 120000ms, uses that value for semantic QMD embedding, and logs only the raw timeout/backoff text without naming the config key; prior QMD timeout work made the value configurable but did not address this default or diagnostic gap.

## What This Changes

Request to raise or contextualize the QMD embed timeout default and make timeout warnings/docs point users to `memory.qmd.update.embedTimeoutMs`.

## Best Possible Solution

Implement a focused QMD-owned repair: preserve explicit overrides, make unset semantic embed timeouts local-GGUF-friendly, add a timeout warning that names the config key and effective value, and document the symptom with recommended values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text when the subprocess exceeds that limit.

## Solution Assessment

Yes. The narrow maintainable fix belongs in QMD config resolution, QMD embed failure diagnostics, focused QMD tests, and QMD docs; it should not change the builtin memory engine or lexical `search` behavior.
