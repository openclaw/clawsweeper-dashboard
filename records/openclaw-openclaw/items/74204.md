## Summary

Keep open. Current main still has the reported QMD semantic embed timeout gap: the unset `memory.qmd.update.embedTimeoutMs` default is 120000ms, semantic QMD modes pass that value directly to `qmd embed`, and timeout warnings do not name the config key that fixes it. Prior QMD hardening work added configurability but did not resolve the local-GGUF default or diagnostic problem.

## What This Changes

Request to make QMD semantic embed timeouts more suitable for local GGUF workloads and surface `memory.qmd.update.embedTimeoutMs` in timeout diagnostics and docs.

## Best Possible Solution

Implement a focused QMD-owned repair: make the unset semantic embed timeout local-GGUF-friendly, preserve explicit `embedTimeoutMs` overrides, enrich QMD embed timeout warnings with the config key and effective value, add regression coverage, and document the timeout symptom with recommended values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff message if the subprocess exceeds that limit.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in QMD config resolution, QMD embed failure diagnostics, QMD tests, and QMD docs; it should not broaden into unrelated QMD scheduling or builtin memory-engine behavior.
