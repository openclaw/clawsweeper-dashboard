## Summary

Keep open. Current main still matches the reported QMD embed timeout gap: unset `memory.qmd.update.embedTimeoutMs` resolves to 120000ms, semantic QMD modes pass that value to `qmd embed`, and the warning path logs only the raw timeout/backoff text without naming the config key. Prior QMD timeout work added configurability, but it does not address this local-GGUF default and diagnostic issue.

## What This Changes

Make QMD embedding timeouts more usable for local GGUF workloads by raising or tuning the default and surfacing `memory.qmd.update.embedTimeoutMs` in timeout guidance.

## Best Possible Solution

Implement a focused QMD-owned repair: preserve explicit overrides, make the unset semantic embed timeout local-GGUF-friendly, enrich timeout warnings with the config key and effective value, add regression coverage, and document the reported symptom with recommended values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text when the subprocess exceeds that limit.

## Solution Assessment

Yes. The narrowest maintainable path is in QMD config resolution, QMD embed failure diagnostics, focused QMD tests, and QMD docs; it should not change the builtin memory engine or unrelated QMD scheduling semantics.
