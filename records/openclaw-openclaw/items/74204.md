## Summary

Keep open. Current main still matches the reported QMD semantic-embed timeout gap: unset `memory.qmd.update.embedTimeoutMs` resolves to 120000ms, `query`/`vsearch` modes pass it directly to `qmd embed`, and timeout warnings do not name the config key that fixes the failure.

## What This Changes

Request to make QMD semantic embedding usable on local GGUF setups by increasing or contextualizing the 120s `memory.qmd.update.embedTimeoutMs` default and surfacing the config fix in diagnostics/docs.

## Best Possible Solution

Implement a focused QMD-owned repair: keep explicit overrides, make the unset semantic-embed timeout local-GGUF-friendly, include the config key and effective value in timeout warnings, and add QMD docs/tests plus changelog coverage.

## Reproduction Assessment

Yes. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, runs `qmd embed` with that timeout, then logs only the raw timeout/backoff text on failure.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in QMD config resolution, QMD embed diagnostics, focused QMD tests, and QMD docs; it should not change the builtin memory engine or unrelated scheduling behavior.
