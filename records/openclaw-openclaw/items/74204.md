## Summary

Keep open. Current main still defaults QMD embed maintenance to 120000ms, applies that timeout to semantic `qmd embed` runs, and logs timeout/backoff text without naming `memory.qmd.update.embedTimeoutMs`; the related HTTP embedding-provider work and the closed lexical-mode QMD fix do not resolve this local-GGUF semantic timeout case.

## What This Changes

Request that QMD semantic embed maintenance use a local-GGUF-friendly timeout default and surface `memory.qmd.update.embedTimeoutMs` in timeout diagnostics and docs.

## Best Possible Solution

Preserve explicit overrides, make the semantic QMD embed timeout default local-GGUF-friendly, and update timeout warnings plus QMD troubleshooting docs to name `memory.qmd.update.embedTimeoutMs`.

## Reproduction Assessment

Yes, from source inspection rather than a live GGUF run. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text on failure.

## Solution Assessment

Yes. A focused QMD config-default, diagnostic, docs, and regression-test repair is narrower than the adjacent HTTP embedding-provider feature work and directly addresses the in-process local-GGUF timeout report.
