## Summary

Keep open. Current main still matches the reported QMD timeout problem: an unset `memory.qmd.update.embedTimeoutMs` resolves to 120000ms, semantic QMD modes pass that value to `qmd embed`, and timeout warnings do not name the config key that fixes the problem. Related older QMD timeout issues fixed configurability and BM25-only skipping, but they do not close this semantic local-GGUF default/diagnostic gap, and no open PR currently references this issue.

## What This Changes

Request to make QMD semantic embed timeouts more suitable for local GGUF workloads and surface `memory.qmd.update.embedTimeoutMs` in timeout diagnostics and docs.

## Best Possible Solution

Implement a focused QMD-owned repair: make the unset semantic embed timeout local-GGUF-friendly, preserve explicit `embedTimeoutMs` overrides, enrich QMD embed timeout warnings with the config key and effective value, add regression coverage, and document the reported timeout symptom with recommended values.

Yes. ## Solution Assessment
