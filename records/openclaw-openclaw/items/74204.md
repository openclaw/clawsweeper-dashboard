## Summary

Keep open. Current main still has the reported QMD semantic-embed timeout and diagnostic gap: the unset default resolves to 120000ms, `query`/`vsearch` modes pass that timeout to `qmd embed`, and the warning does not name the config key that fixes it.

## What This Changes

The issue requests a more local-GGUF-friendly `memory.qmd.update.embedTimeoutMs` default and clearer timeout guidance when QMD embedding exceeds 120 seconds.

## Best Possible Solution

Resolve this in the QMD config and diagnostics path: make the unset semantic-embed timeout suitable for local GGUF workloads or context-aware, and surface `memory.qmd.update.embedTimeoutMs` in timeout warnings and QMD troubleshooting docs.

## Reproduction Assessment

Yes. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs the raw timeout/backoff without the fix key.

## Solution Assessment

Yes. A focused QMD-owned repair is the narrowest maintainable direction; changing the built-in memory engine or generic process timeout handling would be broader than the reported behavior.
