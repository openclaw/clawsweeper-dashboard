## Summary

Keep open. Current main still matches the reported QMD failure mode: the effective default embed timeout is 120000ms, vector-backed QMD modes pass that value directly to `qmd embed`, and the warning path does not point users to `memory.qmd.update.embedTimeoutMs` or the QMD troubleshooting docs.

## What This Changes

Request to make QMD embedding friendlier for local GGUF workloads by raising or adapting `memory.qmd.update.embedTimeoutMs`, improving timeout diagnostics, and documenting the recovery path.

## Best Possible Solution

Implement a focused QMD-owned fix: make the unset embed timeout local-GGUF-friendly for semantic QMD modes, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the symptom and recommended values in the QMD troubleshooting/config reference pages.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff when the subprocess exceeds that limit.

## Solution Assessment

Yes. The narrow maintainable solution is in QMD config resolution, QMD embed failure diagnostics, and QMD docs; the separate QMD scheduling reports in #63626 and #73432 should remain out of scope.
