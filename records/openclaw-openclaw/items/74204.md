## Summary

Keep open. Current main still resolves an unset QMD embed timeout to 120000ms, passes that budget directly to semantic `qmd embed` runs, and logs the raw timeout/backoff message without naming the config key that fixes it.

## What This Changes

Requests a QMD-owned fix so semantic `qmd embed` runs are not cut off by the 120s default and timeout warnings/docs point users to `memory.qmd.update.embedTimeoutMs`.

## Best Possible Solution

Implement a QMD-owned repair that preserves explicit overrides, uses a local-GGUF-friendly unset semantic embed timeout or equivalent default, and makes timeout warnings plus docs name `memory.qmd.update.embedTimeoutMs`.

## Reproduction Assessment

Yes, from source inspection rather than a live GGUF run. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text on failure.

## Solution Assessment

Yes. The narrow maintainable fix belongs in QMD config resolution, QMD embed diagnostics, focused tests, docs, and changelog, while leaving the builtin memory engine and lexical `searchMode: "search"` behavior unchanged.
