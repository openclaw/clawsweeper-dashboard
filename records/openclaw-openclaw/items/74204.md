## Summary

Keep open. Current `main` still matches the reported QMD embed timeout problem: an unset QMD embed timeout resolves to 120000ms, semantic QMD modes pass that value directly to `qmd embed`, and timeout warnings surface only the raw subprocess timeout/backoff text without naming `memory.qmd.update.embedTimeoutMs`. The fix is narrow and source-owned, with no open PR found that already handles this issue.

## What This Changes

The issue requests a larger or context-aware default for `memory.qmd.update.embedTimeoutMs`, clearer timeout diagnostics, and QMD troubleshooting docs for local GGUF embedding timeouts.

## Best Possible Solution

Implement a focused QMD-owned repair: make the unset embed timeout local-GGUF-friendly for semantic `vsearch`/`query` modes, keep explicit user overrides respected, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the reported symptom and recommended values in QMD docs.

Yes. ## Solution Assessment

Yes. The narrow maintainable solution belongs in QMD config resolution, QMD embed failure diagnostics, and QMD docs; the separate QMD scheduling issues should stay out of scope.
