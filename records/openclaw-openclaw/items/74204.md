## Summary

Keep open. Current main still resolves an unset QMD embed timeout to 120000ms, passes that value directly to `qmd embed`, and logs the raw timeout/backoff without naming `memory.qmd.update.embedTimeoutMs`; the QMD docs expose the knob but do not yet document the reported timeout symptom. https://github.com/openclaw/openclaw/issues/74204

## What This Changes

The issue requests a longer or smarter default for QMD embed timeout on local GGUF workloads, clearer timeout diagnostics, and QMD troubleshooting documentation for `memory.qmd.update.embedTimeoutMs`.

## Best Possible Solution

Implement a focused QMD-owned repair: make the unset embed timeout local-GGUF-friendly for semantic `vsearch` and `query` modes, keep explicit user overrides respected, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the symptom and recommended values in the QMD troubleshooting/config pages.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text if the subprocess exceeds that limit.

## Solution Assessment

Yes. The narrow maintainable solution belongs in QMD config resolution, QMD embed failure diagnostics, and QMD docs; the separate QMD scheduling issues should stay out of scope.
