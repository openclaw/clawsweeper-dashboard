## Summary

Keep open. Current main still uses a 120000ms unset QMD embed timeout for semantic modes and logs the raw timeout/backoff message without naming the config key that fixes it.

## What This Changes

The issue asks OpenClaw to make QMD semantic embedding timeouts friendlier for local GGUF workloads by raising or contextualizing the default and surfacing the `memory.qmd.update.embedTimeoutMs` fix in diagnostics and docs.

## Best Possible Solution

Implement a QMD-owned repair that preserves explicit overrides, uses a local-GGUF-friendly unset embed timeout for semantic QMD modes or an equivalent default, and makes timeout warnings plus docs name `memory.qmd.update.embedTimeoutMs`.

## Reproduction Assessment

Yes, from source inspection rather than a live GGUF run. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text on failure.

## Solution Assessment

Yes. The narrow maintainable fix belongs in QMD config resolution, QMD embed diagnostics, focused tests, docs, and changelog while leaving the builtin memory engine and lexical `searchMode: "search"` behavior unchanged.
