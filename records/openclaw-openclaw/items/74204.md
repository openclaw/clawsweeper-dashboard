## Summary

Keep open. Current main and the latest release still use the 120000ms QMD embed default and the warning still does not name the config key; the open closing PR at https://github.com/openclaw/openclaw/pull/87572 is the active fix path and has not merged.

## What This Changes

The issue asks OpenClaw to make QMD local GGUF embedding usable by raising or adapting the embed timeout default and making timeout diagnostics point to `memory.qmd.update.embedTimeoutMs`.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/87572 so explicit overrides remain honored, semantic QMD embedding becomes local-GGUF-friendly, diagnostics name the existing config key, and schema help/docs/tests match the accepted default.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for current main: enable QMD semantic search with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and OpenClaw resolves 120000ms and passes it to `qmd embed`. The reporter supplied matching timeout logs, but I did not live-run the GGUF workload in this read-only review.

## Solution Assessment

Unclear until the linked PR finishes maintainer review: raising the default and improving the warning is a narrow direct fix, but the longer default affects all unset QMD users and should be explicitly accepted or narrowed before merge.
