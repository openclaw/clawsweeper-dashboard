## Summary

Keep open. Current main still resolves an unset QMD embed timeout to 120000ms, passes that value directly to `qmd embed` in semantic QMD modes, and logs timeout/backoff text without naming `memory.qmd.update.embedTimeoutMs`; the docs expose the knob but still lack a symptom-specific troubleshooting entry.

## What This Changes

Request to make the QMD embed timeout friendlier for local GGUF semantic search and surface `memory.qmd.update.embedTimeoutMs` in timeout errors and docs.

## Best Possible Solution

Implement a focused QMD-owned repair: use a local-GGUF-friendly effective timeout for semantic `vsearch`/`query` embed runs while preserving explicit overrides, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the reported symptom with recommended values in QMD docs.

## Reproduction Assessment

Yes. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and any `qmd embed` run exceeding 120000ms will be killed by the subprocess timeout and logged without the config-key hint.

## Solution Assessment

Yes. The narrow maintainable fix belongs in QMD config resolution, QMD embed failure diagnostics, QMD docs, and focused tests; the separate QMD scheduling issues should stay out of scope.
