## Summary

Keep open. Current main still resolves an unset QMD embed maintenance timeout to 120000ms, uses that value for semantic `qmd embed` runs, and logs timeout/backoff text without naming `memory.qmd.update.embedTimeoutMs`; the related local-server embedding-provider work and the closed lexical-mode QMD fix do not resolve this local-GGUF semantic timeout case.

## What This Changes

The issue asks OpenClaw to raise or contextualize the QMD embed timeout default for local GGUF semantic search and make timeout diagnostics or docs point to `memory.qmd.update.embedTimeoutMs`.

## Best Possible Solution

Preserve explicit overrides, make semantic QMD embed timeout behavior local-GGUF-friendly, and update timeout warnings plus QMD troubleshooting docs to name `memory.qmd.update.embedTimeoutMs`.

## Reproduction Assessment

Yes, from source inspection rather than a live GGUF run. Configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff text on failure.

## Solution Assessment

Yes. A focused QMD config-default, diagnostic, docs, and regression-test repair is narrower than the adjacent HTTP embedding-provider feature work and directly addresses the in-process local-GGUF timeout report.
