## Summary

Keep open. Current main and v2026.6.9 provide generic transcript redaction plus plugin hooks that can rewrite or block message persistence, but they do not implement the requested protected-file definition, similarity matching, no-flash frontend blocked state, or audit workflow; related hook issues overlap but do not own this exact protected-content policy.

## What This Changes

The issue requests a protected-content filtering workflow that detects agent-read protected files before persistence and channel/UI output, replacing or blocking unsafe content.

## Best Possible Solution

Define a maintainer-approved protected-output policy first, then implement it at the smallest core/plugin boundary that guarantees no raw protected content is persisted or emitted to supported UI/channel surfaces, with config, docs, and focused security regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source review does show the capability gap: current main has generic hooks and redaction settings, but no protected-file matcher or protected-content output workflow.

## Solution Assessment

Unclear. The two-layer hook approach is plausible, but maintainers should decide the threat model, config surface, default behavior, frontend blocked semantics, and whether this belongs in a plugin using existing hooks or requires new core/protocol support.
