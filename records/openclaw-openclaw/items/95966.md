## Summary

Keep open. Current main and the latest release have generic transcript hooks, delivery hooks, live display normalization, and redaction primitives, but they still do not implement a protected-file/content-provenance policy or a live UI blocked-event workflow; the related security-profile, sanitizer, and guard-hook items only partially overlap.

## What This Changes

Request to add protected-content output filtering so agent-read protected files are not persisted, broadcast, or briefly shown in channel/UI output.

## Best Possible Solution

Define a maintainer-approved protected-output security policy and the minimal core/plugin/protocol seams for transcript persistence, delivery queue recovery, outbound cancellation, and live UI blocked-state rendering, with detection logic plugin-owned where possible.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a security feature/design request. Source inspection confirms current main lacks the requested protected-content policy and blocked UI event workflow, but no failing current-main live repro was run.

## Solution Assessment

Unclear. The two-layer hook plan uses relevant seams, but default behavior, protected-file provenance, similarity thresholds, false-positive handling, audit logging, and protocol/UI blocked semantics need maintainer security/product direction before it is the best implementation path.
