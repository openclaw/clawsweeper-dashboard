## Summary

Keep open. Current main has useful generic hooks, transcript redaction, and live transcript projection, but it does not implement protected-file/content-provenance policy, pre-model delivery filtering, or an explicit blocked-message UI/protocol workflow; the related guard-hook PR only partially overlaps and is not merge-ready.

## What This Changes

The issue requests a protected-content filtering workflow that lets agents read protected files while preventing matching content from being persisted, broadcast to channels, or flashed in the UI.

## Best Possible Solution

Define a maintainer-approved security design that keeps core lean: a narrow plugin/API guard seam plus explicit transcript/channel/UI blocked-message semantics, with protected-file policy and defaults reviewed before implementation.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/security design request. Source inspection gives high confidence that current main lacks the requested protected-file policy, pre-delivery filter, and blocked-event workflow.

## Solution Assessment

Unclear as proposed. The safest direction is a maintainer-approved guard/API and UI/protocol design rather than adding a broad default core DLP filter with new config knobs in one automated fix.
