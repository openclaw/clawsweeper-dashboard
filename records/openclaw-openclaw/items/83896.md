## Summary

Keep open as a low-risk CLI maintainability item: current main still mutates Commander's readonly `commands` array, but Commander 14.0.3 currently dispatches through that same array, so the reported stale-map runtime failure is not proven.

## What This Changes

The issue requests replacing or redesigning `removeCommand` so OpenClaw does not splice Commander's internal `commands` array directly.

## Best Possible Solution

Refactor lazy command replacement away from mutating Commander's readonly public array, or explicitly preserve the helper with tests proving removed names cannot dispatch and lazy replacement paths still work.

## Reproduction Assessment

No high-confidence runtime reproduction is established. Source inspection confirms direct mutation of Commander's readonly `commands` array, but Commander 14.0.3 currently dispatches removed names through `commands.find(...)`, not a separate name map.

## Solution Assessment

Unclear. The simple recommendation to use a public Commander removal API does not fit Commander 14.0.3, so the safer path is either a bounded lazy-registration refactor or a tested local helper contract.
