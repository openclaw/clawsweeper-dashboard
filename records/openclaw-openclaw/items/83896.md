The issue asks OpenClaw to stop removing lazy CLI placeholders by splicing Commander's internal `commands` array.

## Best Possible Solution

CLI owners should either redesign lazy command replacement around a supported registry path or keep the current Commander 14 internal contract explicit with parse-dispatch and parent-state coverage.

## Reproduction Assessment

Yes for the source-level dependency-contract concern: current main still casts and mutates Commander's readonly `commands` list. No for the specific stale dispatch claim in Commander 14.0.3: a focused proof produced `commander.unknownCommand` instead of routing to the removed action.

## Solution Assessment

Unclear. Commander 14.0.3 does not expose a public remove API, so the best fix is likely an owner-reviewed lazy-registration design or explicit tests around the current internal contract rather than a mechanical API swap.
