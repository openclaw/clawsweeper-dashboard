## Summary

Keep open as a CLI maintainability cleanup: current main still casts Commander's readonly command list to mutable and splices it, and the same helper shipped in v2026.5.18. The specific stale name-map routing failure is not established for Commander 14.0.3 because dispatch currently resolves through `commands.find`, but the dependency-contract fragility is real enough for CLI-area review.

## What This Changes

The issue requests replacing or redesigning a CLI helper that mutates Commander's readonly `commands` array to remove registered subcommands.

## Best Possible Solution

Have a CLI-area reviewer either refactor lazy command replacement away from mutating Commander's readonly array, or deliberately keep this helper with tests that prove removed names cannot dispatch and replacement paths still work.

## Reproduction Assessment

No high-confidence runtime reproduction is established. Source inspection confirms the direct mutation of a readonly Commander array, but Commander 14.0.3 currently dispatches removed names by searching that same array.

## Solution Assessment

Unclear. Commander 14.0.3 does not expose a public remove API, so the best fix is likely a bounded lazy-registration refactor or an explicitly tested helper contract rather than the issue's simple API-swap recommendation.
