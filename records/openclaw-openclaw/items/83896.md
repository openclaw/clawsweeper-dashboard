## Summary

Keep open as a low-priority CLI maintainability/design item: current main and v2026.5.18 still mutate Commander's readonly `commands` array, but Commander 14.0.3 dispatch currently searches that same array, so the reported stale name-map routing failure is not proven.

## What This Changes

The issue asks OpenClaw to stop removing Commander commands by casting and splicing Commander's readonly `commands` array.

## Best Possible Solution

Have the CLI owner either refactor lazy command replacement away from mutating Commander's readonly public array, or keep the helper intentionally with regression tests proving removed names cannot dispatch and lazy placeholder replacement still works.

## Reproduction Assessment

No high-confidence runtime reproduction is established. Source inspection confirms the direct splice of a readonly Commander array, but Commander 14.0.3 currently dispatches removed names through `commands.find`, so the stale-map parse failure is not proven.

## Solution Assessment

Unclear. Commander 14.0.3 does not expose a public removal API, so the maintainable path is likely a bounded lazy-registration refactor or an intentionally tested helper contract rather than a direct API swap.
