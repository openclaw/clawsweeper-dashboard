## Summary

Keep open as a low-priority CLI maintainability cleanup: current main and the latest release still mutate Commander's readonly `commands` array, while the claimed stale dispatch failure remains unproven for Commander 14.0.3.

## What This Changes

The issue asks OpenClaw to stop removing lazy Commander placeholders by directly splicing `program.commands`.

## Best Possible Solution

CLI owners should choose and test one supported lazy command replacement contract, either by avoiding Commander tree surgery or by explicitly preserving the internal helper with parse and parent-state coverage.

## Reproduction Assessment

No high-confidence user-facing failure is reproduced here. Source inspection does confirm the direct splice of Commander's readonly `commands` array, while Commander 14.0.3 lookup source makes the stale-dispatch claim uncertain.

## Solution Assessment

Unclear. The requested public-API swap is not available in Commander 14.0.3, so the maintainable fix is a CLI-owner design choice with focused lazy-registration regression tests.
