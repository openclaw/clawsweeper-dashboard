## Summary

Keep open as a low-priority CLI maintainability cleanup: current main and the latest release still splice Commander's readonly command list, but the reported stale dispatch failure remains unproven for Commander 14.0.3.

## What This Changes

The issue asks OpenClaw to stop removing Commander subcommands by casting and splicing `program.commands` directly.

## Best Possible Solution

CLI owners should either redesign lazy command replacement around a supported registry path or keep this helper as an explicit, tested Commander 14 contract with parse-dispatch and parent-state coverage.

## Reproduction Assessment

Yes for the source-level dependency-contract concern: current main still casts and mutates Commander's readonly `commands` list. No high-confidence runtime stale-dispatch failure was reproduced or source-proven for Commander 14.0.3.

## Solution Assessment

Unclear. Commander 14.0.3 does not expose a public removal API, so the best fix is likely an owner-reviewed lazy-registration design or explicit tests around the current internal contract rather than a mechanical API swap.
