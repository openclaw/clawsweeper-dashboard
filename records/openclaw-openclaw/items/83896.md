## Summary

Keep open as a CLI maintainability cleanup: current main and v2026.5.18 still splice Commander's readonly `commands` array, but Commander 14.0.3 dispatch currently searches that same array and exposes no public remove API, so the reported runtime routing failure is not proven.

## What This Changes

The issue requests replacing or reworking `removeCommand` so CLI command removal does not mutate Commander's readonly `commands` array directly.

## Best Possible Solution

Refactor lazy command replacement to avoid directly mutating Commander internals, or explicitly preserve the helper with regression tests that prove removed names cannot dispatch and replacement paths still work.

## Reproduction Assessment

No high-confidence runtime reproduction is established. Source inspection confirms the direct splice of a readonly Commander array, but Commander 14.0.3 currently dispatches removed names through `commands.find`, so the reported stale-map parse path is not proven.

## Solution Assessment

Unclear. Commander 14.0.3 does not expose a public remove API, so the maintainable fix is likely a bounded lazy-registration refactor or a narrower tested helper contract rather than a simple API swap.
