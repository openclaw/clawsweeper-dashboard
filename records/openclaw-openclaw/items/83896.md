## Summary

Keep open as a CLI cleanup/design item: current main and v2026.5.18 still splice Commander's readonly `commands` array, but Commander 14.0.3 dispatch currently searches that same array, so the specific stale name-map routing failure is not proven.

## What This Changes

The issue requests replacing direct mutation of Commander's command list with a public or registry-owned removal path.

## Best Possible Solution

Have the CLI owner either refactor lazy command replacement away from mutating Commander's readonly public array, or explicitly preserve the helper with tests that prove removed names cannot dispatch and replacement paths still work.

## Reproduction Assessment

No high-confidence runtime reproduction is established. Source inspection confirms the direct splice of a readonly Commander array, but Commander 14.0.3 currently dispatches removed names through `commands.find`, so the reported stale-map parse path is not proven.

## Solution Assessment

Unclear. Commander 14.0.3 does not expose a public `removeCommand` API, so the maintainable fix is likely a bounded lazy-registration refactor or a narrower tested helper contract rather than a simple API swap.
