## Summary

Keep open: the linked cron fallback bug is valid from current-main source and docs, but this PR has a blocking alias-resolution regression and the supplied proof is static/source-only rather than an after-fix real cron run.

## What This Changes

The PR makes isolated cron preflight probe configured model fallbacks before skipping a run, and adds cron preflight fallback tests.

## Best Possible Solution

Keep the preflight fallback idea, but resolve the preflight candidate chain through the same alias-aware model-selection path used by `runWithModelFallback`, add regression coverage for alias fallbacks, add the required changelog entry, and require real runtime proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main returns `skipped` immediately on primary preflight unavailability while the docs say cron job primaries can use configured fallbacks. I did not run a live cron scenario in this read-only review.

## Solution Assessment

No. The PR is aimed at the right code path, but it parses fallback refs with `parseModelRef` instead of the existing alias-aware fallback resolver, so valid alias-backed fallback chains can still be skipped.

## Review Findings
