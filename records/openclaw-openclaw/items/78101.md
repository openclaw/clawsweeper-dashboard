## Summary

Keep open: the PR addresses a real slow-filesystem gateway latency problem with sufficient runtime proof, but the proposed cache can cross plugin registry scopes because the key omits supplied candidates and only fingerprints install-record ids.

## What This Changes

The PR adds a 2-second in-memory coalescing cache around `loadPluginRegistrySnapshotWithMetadata` and a changelog entry for slow-filesystem gateway latency.

## Best Possible Solution

Keep the near-term coalescing only if every behavior-affecting registry input is isolated in the key, or bypass the cache for caller-supplied registry scopes and leave the deeper async work to the companion performance PR.

## Reproduction Assessment

Yes. The PR body provides concrete WSL2/WhatsApp after-fix logs and timings, and source inspection shows the current synchronous registry snapshot path is still reachable from plugin metadata loading.

## Solution Assessment

No, not as written. A short TTL can be a reasonable mitigation, but it must fingerprint supplied candidates and full install-record values or bypass caching for those scoped calls.

## Review Findings
