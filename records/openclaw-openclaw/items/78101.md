## Summary

Keep open: this PR targets a real slow-filesystem Gateway bug and includes useful WSL2 runtime proof, but the latest cache key still does not isolate all registry-affecting inputs, leaving a merge-blocking correctness and trust-scope issue.

## What This Changes

This PR adds a 2-second in-memory coalescing cache around `loadPluginRegistrySnapshotWithMetadata` and a changelog entry for slow-filesystem `channels.status` and channel-restart latency.

## Best Possible Solution

Keep the near-term coalescing only if every registry-affecting scope is fingerprinted, or bypass the cache for caller-supplied scopes that cannot be keyed cheaply; retain focused regression coverage and the changelog entry if the mitigation remains user-facing.

## Reproduction Assessment

Yes, with source-backed confidence rather than a local live rerun: the linked issue and PR body provide concrete WSL2/WhatsApp steps and before/after logs, and current source shows status/reporting flows call the synchronous snapshot loader.

## Solution Assessment

No, not as written. A short TTL may be a reasonable near-term mitigation, but this implementation is not safe until supplied candidates, install-record contents, and env-derived store scope are isolated or bypassed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fingerprint supplied registry inputs:** `src/plugins/plugin-registry-snapshot.ts:217-219`
  - body: The cache key only includes the names of `installRecords` and still omits `params.candidates`, but both inputs are passed into registry assembly and can change the plugin set, roots, package metadata, and install metadata. Two scoped calls with the same state/config inside the 2-second TTL can therefore receive the first caller's snapshot; fingerprint these inputs stably or bypass caching when they are supplied.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff touches plugin discovery and can reuse plugin registry results across caller scopes during the TTL window.

Concerns:

- **[medium] Cache can cross plugin trust scopes:** `src/plugins/plugin-registry-snapshot.ts:217`
  - body: Plugin registry snapshots determine which plugins are visible and enabled. Because the cache key omits supplied candidates and only keys install-record ids, a caller can receive another scoped registry result inside the 2-second window.
