## Summary

Keep open: the reported slow-filesystem gateway stall is well supported and the PR includes usable live proof, but the patch can reuse a plugin registry across different state-affecting inputs and should not merge as-is.

## What This Changes

The PR adds a 2-second module-level memoization layer around `loadPluginRegistrySnapshotWithMetadata` plus a changelog entry for slow-filesystem `channels.status` and channel-restart latency.

## Best Possible Solution

Keep the short-lived coalescing only if the cache key or bypass logic preserves every registry-affecting caller scope, then land it with focused regression tests and the existing changelog entry.

## Reproduction Assessment

Yes, with source-backed confidence rather than a local live rerun: the PR body gives concrete WSL2/WhatsApp steps and before/after logs, and current source shows `channels.status` reaches the synchronous registry snapshot path.

## Solution Assessment

No. Coalescing may be the right near-term mitigation, but the proposed global cache is too broad because it omits inputs that change plugin discovery and persisted-registry behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key all state-affecting snapshot inputs:** `src/plugins/plugin-registry-snapshot.ts:210-224`
  - body: The cache key ignores inputs that currently change registry output, including `env`, `workspaceDir`, `candidates`, and `installRecords`. A status call for one workspace/env can populate the module cache, and another call with the same state/config but a different workspace or persisted-registry env within 2 seconds will receive the wrong plugin registry, potentially hiding or exposing plugins from the wrong scope. Include those inputs in the key or bypass the cache when non-default/unhashable inputs are present.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch touches plugin discovery, and the cache-key omission can cross workspace/env plugin trust boundaries for up to two seconds.

Concerns:

- **[medium] Cache can cross plugin trust boundaries:** `src/plugins/plugin-registry-snapshot.ts:210`
  - body: Plugin registry results determine which plugin code is visible and enabled. Because the proposed cache does not distinguish `workspaceDir`, `env`, `candidates`, or `installRecords`, one caller can receive another scope's registry result inside the TTL window.
  - confidence: 0.86
