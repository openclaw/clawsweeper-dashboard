## Summary

Keep open: the performance problem is credible and the supplied proof is sufficient, but the PR still has a cache-isolation bug that can reuse a plugin registry snapshot across caller scopes during the TTL window.

## What This Changes

The PR adds a 2-second in-memory coalescing cache around `loadPluginRegistrySnapshotWithMetadata` and a changelog entry for slow-filesystem gateway latency.

## Best Possible Solution

Keep the near-term coalescing only if every behavior-affecting registry input is isolated, or bypass the cache for caller-supplied scopes; leave the deeper async fix to the companion PR.

## Reproduction Assessment

Yes, with source-backed confidence rather than a local live rerun: the linked issue and PR body provide concrete WSL2/WhatsApp steps and before-after logs, and current source still has synchronous registry snapshot work reachable from plugin metadata loading.

## Solution Assessment

No, not as written. A short TTL can be a reasonable mitigation, but this patch is not safe until supplied candidates and full install-record values are fingerprinted or those scoped calls bypass the cache.

## Review Findings

Overall correctness: patch is incorrect
