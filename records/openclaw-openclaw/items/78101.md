## Summary

Keep open: this is a useful, well-evidenced performance fix, but the current patch is not merge-ready because the new global TTL cache can reuse plugin registry results across different workspace, environment, candidate, or install-record scopes.

## What This Changes

The PR adds a 2-second in-memory cache around `loadPluginRegistrySnapshotWithMetadata` plus a changelog entry for slow-filesystem `channels.status` and channel-restart performance.

## Best Possible Solution

Keep the near-term coalescing only after every registry-affecting scope is keyed or uncacheable scopes bypass the cache, with regression coverage retained alongside the changelog entry.

## Reproduction Assessment

Yes, with source-backed confidence rather than a local live rerun: the linked issue and PR body provide concrete WSL2/WhatsApp steps and before/after logs, and current source shows status/reporting flows call the synchronous snapshot loader.

## Solution Assessment

No, not as written. A short TTL may be a reasonable near-term mitigation, but this implementation is not the narrowest safe form because its cache key omits inputs that change plugin discovery and persisted-registry behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Key all registry-affecting inputs:** `src/plugins/plugin-registry-snapshot.ts:209-220`
  - body: The cache key ignores inputs that change registry output, including `env`, `workspaceDir`, `candidates`, and `installRecords`. A call from one workspace or plugin scope can populate the module cache, and another call with the same state/config but a different scope within 2 seconds will receive the wrong registry snapshot; include stable fingerprints for these inputs or bypass the cache when they are supplied.
  - confidence: 0.9
