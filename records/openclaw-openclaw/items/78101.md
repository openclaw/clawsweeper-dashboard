## Summary

Keep open: the performance problem is credible and the PR includes real after-fix logs, but the proposed module cache can still return a registry snapshot from a different caller scope because its key omits registry-affecting inputs.

## What This Changes

The PR adds a 2-second in-memory cache around `loadPluginRegistrySnapshotWithMetadata` and a changelog entry about reducing `channels.status` and channel-restart registry scans on slow filesystems.

## Best Possible Solution

Keep the short-lived coalescing only if every registry-affecting caller scope is keyed or uncacheable scopes bypass the cache, with focused regression tests and the existing changelog entry retained.

## Reproduction Assessment

Yes, with source-backed confidence rather than a local live rerun: the linked issue and PR body give concrete WSL2/WhatsApp steps and before/after logs, and current source shows the synchronous registry snapshot path is used by status/plugin-reporting flows.

## Solution Assessment

No. The short TTL may be a reasonable near-term mitigation, but this implementation is not the best mergeable form because its cache key omits inputs that change plugin discovery and persisted-registry behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key all state-affecting snapshot inputs:** `src/plugins/plugin-registry-snapshot.ts:209-220`
  - body: The cache key ignores inputs that currently change registry output, including `env`, `workspaceDir`, `candidates`, and `installRecords`. A call from one workspace/env can populate the module cache, and another call with the same state/config but a different plugin scope within 2 seconds will receive the wrong registry snapshot. Include those inputs in the key or bypass the cache when the caller supplies non-default or unhashable scope inputs.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff touches plugin discovery, and the incomplete cache key can cross plugin visibility or trust scopes for the TTL window.

Concerns:

- **[medium] Cache can cross plugin trust boundaries:** `src/plugins/plugin-registry-snapshot.ts:209`
  - body: Plugin registry results determine which plugins are visible and enabled. Because the proposed cache does not distinguish `workspaceDir`, `env`, `candidates`, or `installRecords`, one caller can receive another scope's registry result inside the TTL window.
