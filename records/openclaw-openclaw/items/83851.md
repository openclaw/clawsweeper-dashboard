## Summary

Keep open. Current main and the latest release still lack the requested deterministic post-write config apply/reload contract; the merged reloadKind work is preflight metadata and the open related PR only targets narrower CLI wording.

## What This Changes

The issue requests deterministic, machine-readable post-write status for config patch/set/gateway writes, including write result, reload plan, hot-reload outcome, restart requirement, and effective-runtime verification.

## Best Possible Solution

Define a backward-compatible shared config-write acknowledgement contract that reuses the reload planner, reports persisted write result and changed paths, states restart requirement/reasons, and makes reload completion or verification semantics explicit for gateway RPC and CLI JSON output.

## Reproduction Assessment

Yes. Source inspection on current main shows config write responses and CLI success output still lack the requested deterministic changed-path, reload-plan, apply-outcome, and effective-runtime fields.

## Solution Assessment

No patch is under review. The best path is a maintainer-approved gateway/CLI response contract, not closing this as covered by preflight reloadKind lookup or the narrower CLI-hint PR.
