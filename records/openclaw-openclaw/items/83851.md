## Summary

Keep open. Current main and v2026.6.8 still do not expose the requested deterministic post-write config apply/reload contract; merged reloadKind work is preflight schema metadata, and the open related PR only addresses narrower CLI wording.

## What This Changes

The issue requests deterministic machine-readable post-write apply/reload status for CLI config writes and gateway config.patch/config.apply responses.

## Best Possible Solution

Define a shared config-write acknowledgement contract that reuses the existing reload planner, reports persisted write result and changed paths, states restart requirement/reasons, and makes reload completion or runtime verification semantics explicit for both gateway RPC and CLI JSON output.

## Reproduction Assessment

Yes. Source inspection on current main shows config write responses and CLI success output still lack the requested deterministic changed-path, reload-plan, apply-outcome, and effective-runtime fields.

## Solution Assessment

No patch is under review. The best path is a maintainer-approved gateway/CLI response contract that extends the existing write flow, not closing this as covered by preflight reloadKind lookup or the narrower CLI-hint PR.
