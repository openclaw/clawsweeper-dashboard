## Summary

Keep open. Current main still lacks the requested agent-facing task/team tools, so the PR is not obsolete, but the proposed patch has concrete P2 review findings and is stacked on the still-open filesystem hardening dependency.

## What This Changes

This PR adds owner-only `task` and `team` agent tools, registers them in OpenClaw tool creation and owner-only lists, adds an in-memory team registry, and includes filesystem permission-denied/error hardening with tests.

## Best Possible Solution

Keep the PR open for maintainer review, resolve or split the #67331 filesystem hardening dependency first, then rebase the task/team work onto current main with catalog registration, flat enum schemas, explicit action validation, `ToolInputError` handling for user mistakes, and a deliberate owner-scoped task/team lifecycle model.

## Reproduction Assessment

Not applicable as a bug report, but there is a high-confidence review path: current-main source inspection shows the requested tools are absent, and the PR diff plus `/tools/invoke` code paths expose the concrete review findings without needing generated artifacts.

## Solution Assessment

No, the current patch is not the best merge shape as-is. The intended feature may belong in core, but the safer path is a smaller rebased change that uses existing task owner-access helpers, registers the tools in the catalog, and resolves the schema/error-handling issues before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Register task and team in the core catalog:** `src/agents/openclaw-tools.ts:235-239`
  - body: The PR constructs `task` and `team` as core tools, but it does not add them to `CORE_TOOL_DEFINITIONS`. That leaves `group:openclaw`, profile allowlists, UI sections, and `isKnownCoreToolId()` unaware of the new tools, so the new tools can disappear under normal profile/policy resolution instead of being consistently available.
  - confidence: 0.86
- **[P2] Emit flat enum schemas for task:** `src/agents/tools/task-tool.ts:29-39`
  - body: `TaskToolSchema` uses `Type.Union(...Type.Literal...)` for `action`, `runtime`, and `notifyPolicy`, which emits `anyOf`. Existing tool schemas use `stringEnum`/`optionalStringEnum` because some providers reject `anyOf` in tool schemas, so this can make the new tool unusable with those providers.
  - confidence: 0.93
- **[P2] Emit a flat enum schema for team:** `src/agents/tools/team-tool.ts:19`
  - body: `TeamToolSchema.action` also uses `Type.Union(...Type.Literal...)`, producing the same `anyOf` shape that the repo's schema helper avoids for provider compatibility. Use `stringEnum(TEAM_ACTIONS)` here as well.
  - confidence: 0.92
- **[P2] Classify allowed-mount escapes as workspace boundaries:** `src/agents/fs-permission-denied.ts:22-26`
  - body: The classifier never matches `Sandbox path escapes allowed mounts`, but the same PR wraps that exact message from sandbox fs bridge paths. Those errors therefore surface with `reason=unknown` instead of `workspace_boundary`, breaking consumers that branch on the structured reason.
  - confidence: 0.91
- **[P2] Reject unknown task actions before output lookup:** `src/agents/tools/task-tool.ts:232-241`
  - body: After the explicit task actions, the handler falls through to the output lookup path. Direct surfaces such as `/tools/invoke` call `execute` without TypeBox validation, so a typo like `ouptut` becomes a state-dependent `task not found` error instead of a clear unsupported-action client error.
  - confidence: 0.9
- **[P2] Translate missing team updates to ToolInputError:** `src/agents/tools/team-tool.ts:88-94`
  - body: `updateTeam()` throws a plain `Error` for an unknown team id, and this branch does not translate it. `/tools/invoke` maps only `ToolInputError`/authorization errors to 4xx responses, so a user-correctable missing team update becomes a 500 `tool execution failed`.
  - confidence: 0.88
