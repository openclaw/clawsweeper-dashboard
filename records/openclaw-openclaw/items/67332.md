Full review comments:

- **[P1] Import Type from the declared package in task:** `src/agents/tools/task-tool.ts:1`
  - body: This new file imports `@sinclair/typebox`, but current `package.json` declares `typebox` and the surrounding tools import from `typebox`; the task tool can fail to load under the repo package layout.
  - confidence: 0.96
- **[P1] Import Type from the declared package in team:** `src/agents/tools/team-tool.ts:1`
  - body: This new file has the same undeclared `@sinclair/typebox` import. Use the declared `typebox` package so the team tool resolves consistently with the rest of the agent tools.
  - confidence: 0.95
- **[P2] Register task and team in the core catalog:** `src/agents/openclaw-tools.ts:235-240`
  - body: The PR constructs `task` and `team` as core tools, but it does not add them to `CORE_TOOL_DEFINITIONS`. Profiles, `group:openclaw`, tool sections, and `isKnownCoreToolId()` are catalog-driven, so these tools will not behave as first-class core tools.
  - confidence: 0.9
- **[P2] Use flat enum schemas for task:** `src/agents/tools/task-tool.ts:31-39`
  - body: `TaskToolSchema` uses `Type.Union(...Type.Literal...)` for `action`, `runtime`, and `notifyPolicy`, which emits `anyOf`. Existing tool schemas use `stringEnum`/`optionalStringEnum` because some providers reject `anyOf`.
  - confidence: 0.94
- **[P2] Use a flat enum schema for team:** `src/agents/tools/team-tool.ts:22`
  - body: `TeamToolSchema.action` also uses `Type.Union(...Type.Literal...)`, producing the same provider-incompatible `anyOf` shape. Use the shared flat enum helper here as well.
  - confidence: 0.93
- **[P2] Preserve missing-read errors before wrapping denials:** `src/agents/sandbox/fs-bridge-path-safety.ts:73-84`
  - body: `openReadableFile` now wraps every failed boundary open as `permission_denied`, including ordinary missing-file opens. Current sandbox read behavior preserves not-found diagnostics, so this changes normal ENOENT/no-such-file reads into permission failures.
  - confidence: 0.91
- **[P2] Classify allowed-mount escapes as workspace boundaries:** `src/agents/fs-permission-denied.ts:22-25`
  - body: The classifier never matches the bridge phrase `Sandbox path escapes allowed mounts`, while this PR wraps local and remote sandbox boundary errors using that exact cause text. Those denials become `reason=unknown` instead of `workspace_boundary`.
  - confidence: 0.92
- **[P2] Reject unknown task actions before output lookup:** `src/agents/tools/task-tool.ts:232-240`
  - body: After the explicit task actions, the handler falls through to the output lookup path. Direct invoke surfaces can call `execute` without TypeBox validation, so a typo can become a misleading `task not found` response instead of an unsupported-action error.
  - confidence: 0.9
- **[P2] Translate missing team updates to ToolInputError:** `src/agents/tools/team-tool.ts:88-93`
  - body: `updateTeam()` throws a plain `Error` for an unknown team id, and this branch does not translate it. Direct tool invocation maps only input/authorization errors to 4xx, so a user-correctable missing update can surface as a 500 tool failure.
  - confidence: 0.88
