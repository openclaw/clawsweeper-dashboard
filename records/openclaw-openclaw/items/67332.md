Full review comments:

- **[P1] Enforce owner-only checks at the call boundary:** `src/agents/tools/task-tool.ts:106`
  - body: Setting `ownerOnly` on the returned tool is only metadata here: current OpenClaw tool invocation does not consume that property before executing tools. Wire the authorization decision through the shared tool-resolution/invocation boundary or this does not actually enforce owner-only access.
  - confidence: 0.9
- **[P1] Use owner-scoped task accessors:** `src/agents/tools/task-tool.ts:3-12`
  - body: The task tool imports raw task-registry functions and accepts `ownerKey`/`sessionKey` from model input. Current main has caller-owner scoped helpers; using raw access can list or mutate unrelated task records if this tool is reachable.
  - confidence: 0.91
- **[P1] Import Type from the declared package:** `src/agents/tools/task-tool.ts:1`
  - body: This new tool imports `@sinclair/typebox`, but the repo declares and uses `typebox`. With pnpm's strict package resolution, the new task/team tools can fail to load unless they import the declared package.
  - confidence: 0.94
- **[P2] Register task and team in the core catalog:** `src/agents/openclaw-tools.ts:238-242`
  - body: The PR constructs `task` and `team` as core tools but does not add them to `CORE_TOOL_DEFINITIONS`. Profiles, `group:openclaw`, known-core-tool checks, and `tools.catalog` are catalog-driven, so these tools will not behave consistently as first-class core tools.
  - confidence: 0.9
- **[P2] Use flat enum schemas for task and team:** `src/agents/tools/task-tool.ts:31-39`
  - body: `TaskToolSchema` uses `Type.Union(...Type.Literal...)` for enum fields, producing `anyOf`. Existing agent tools use the shared flat enum helpers because some providers reject `anyOf` in tool schemas; apply the same fix to the team action schema.
  - confidence: 0.93
- **[P2] Preserve missing-file errors when wrapping denials:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: `openReadableFile` now wraps every failed boundary open as `permission_denied`, including ordinary missing-file reads. Current sandbox behavior preserves ENOENT/no-such-file diagnostics, so this changes normal missing-file reads into permission failures.
  - confidence: 0.9
- **[P2] Classify allowed-mount escapes as workspace boundaries:** `src/agents/fs-permission-denied.ts:22-25`
  - body: The classifier does not match the new bridge cause text `Sandbox path escapes allowed mounts`, so those denials become `reason=unknown` instead of `workspace_boundary`.
  - confidence: 0.92
- **[P2] Reject unsupported task actions before output lookup:** `src/agents/tools/task-tool.ts:232-240`
  - body: After the explicit task actions, the handler falls through to the output path. Direct tool invocation can bypass TypeBox validation, so a typo becomes a misleading `task not found` response instead of an unsupported-action error.
  - confidence: 0.89
- **[P2] Convert missing team updates to ToolInputError:** `src/agents/tools/team-tool.ts:88-93`
  - body: `updateTeam()` throws a plain `Error` for unknown team IDs, and this branch does not translate it. Direct invocation maps `ToolInputError` to client errors but plain errors to execution failures.
  - confidence: 0.87
