Keep this PR open. Current main still lacks the proposed task/team tools, and the PR remains useful source work, but the submitted patch has blocking review findings around core catalog registration, provider-compatible schemas, structured filesystem errors, and direct tool-invoke error handling.

## What This Changes

This PR adds owner-only task and team agent tools, registers their constructors and owner-only names, and bundles filesystem permission_denied/payload guard hardening with tests.

## Best Possible Solution

Keep the PR open, resolve or split #67331 first, then rebase task/team onto current main with catalog/profile/docs/changelog coverage, flat enum schemas, explicit action validation, ToolInputError handling, and a deliberate owner-scoped task/team lifecycle model.

Not applicable as a bug report, but there is a high-confidence review path: current main lacks the proposed task/team files and constructors, and the PR diff plus direct /tools/invoke code paths expose the concrete findings without running generated artifacts.

## Solution Assessment

No, the current patch is not the best merge shape as-is. The intended feature may belong in core, but the safer path is a smaller rebased change that uses existing catalog and task-owner access patterns and resolves the schema/error-handling issues first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Register task and team in the core catalog:** `src/agents/openclaw-tools.ts:235-239`
  - body: The PR constructs `task` and `team` as core tools, but it does not add them to `CORE_TOOL_DEFINITIONS`. Profile allowlists, `group:openclaw`, `tools.catalog`, and `isKnownCoreToolId()` are generated from that catalog, so the new tools will not be treated consistently as first-class core tools under normal policy/catalog resolution.
  - confidence: 0.88
- **[P2] Emit flat enum schemas for task:** `src/agents/tools/task-tool.ts:29-39`
  - body: `TaskToolSchema` uses `Type.Union(...Type.Literal...)` for `action`, `runtime`, and `notifyPolicy`, which emits `anyOf`. Existing tool schemas use `stringEnum`/`optionalStringEnum` because some providers reject `anyOf`, so this can make the new tool unusable with those providers.
  - confidence: 0.93
- **[P2] Emit a flat enum schema for team:** `src/agents/tools/team-tool.ts:19`
  - body: `TeamToolSchema.action` also uses `Type.Union(...Type.Literal...)`, producing the same `anyOf` shape that the repo's schema helper avoids for provider compatibility. Use `stringEnum(TEAM_ACTIONS)` here as well.
  - confidence: 0.92
- **[P2] Classify allowed-mount escapes as workspace boundaries:** `src/agents/fs-permission-denied.ts:22-26`
  - body: The classifier never matches `Sandbox path escapes allowed mounts`, but the same PR wraps that exact message from sandbox fs bridge paths. Those errors therefore surface with `reason=unknown` instead of `workspace_boundary`, breaking consumers that branch on the structured reason.
  - confidence: 0.91
- **[P2] Reject unknown task actions before output lookup:** `src/agents/tools/task-tool.ts:232-241`
  - body: After the explicit task actions, the handler falls through to the output lookup path. Direct surfaces such as `/tools/invoke` call `execute` without TypeBox validation, so a typo like `ouptut` becomes a state-dependent `task not found` error instead of a clear unsupported-action client error.
  - confidence: 0.9
- **[P2] Translate missing team updates to ToolInputError:** `src/agents/tools/team-tool.ts:88-94`
  - body: `updateTeam()` throws a plain `Error` for an unknown team id, and this branch does not translate it. `/tools/invoke` maps only `ToolInputError` and authorization errors to 4xx responses, so a user-correctable missing team update becomes a 500 `tool execution failed`.
  - confidence: 0.88
