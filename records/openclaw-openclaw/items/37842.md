## Label Justifications

- `P2`: The issue can cause costly runaway multi-agent activity, but it requires agent-to-agent tools and a cyclic topology rather than breaking the default runtime for all users.
- `impact:session-state`: The missing guard is about causal session state crossing multiple agent sessions without a shared visited-agent chain.
- `impact:message-loss`: A cyclic inter-agent cascade can flood useful communication with repeated `sessions_send` messages and consume delivery capacity.

## Summary

The issue is still necessary: current main still creates fresh inter-session provenance for each `sessions_send` and only bounds the built-in pairwise reply-back flow, so a three-agent A->B->C->A chain is not correlated across runs. The linked implementation was closed unmerged and no current merged issue or PR owns the remaining graph-aware guard.

## What This Changes

Request graph-aware loop detection for `sessions_send` so multi-agent cascades that revisit an agent are blocked instead of bypassing `maxPingPongTurns`.

## Best Possible Solution

Add a bounded canonical visited-agent chain to `InputProvenance`, propagate it through `sessions_send` recipient-run construction, and reject delivery when the normalized target agent already appears in the current chain while leaving `maxPingPongTurns` as the existing pairwise reply-round guard.

## Reproduction Assessment

Yes, at source level: enable agent-to-agent tools, have A call B, B call C, and C call A; each `sessions_send` starts a new provenance object without the earlier visited agents while the only built-in bound is the pairwise A2A loop. I did not run a live three-agent setup in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable direction is to carry canonical visited-agent IDs in the existing inter-session provenance boundary and block revisits before delivery. A new config knob or SOUL.md behavioral rule would be less reliable than enforcing the invariant in the tool path.
