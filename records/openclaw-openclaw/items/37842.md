## Summary

Keep open. Current main still has only source/provenance marking and pairwise A2A ping-pong limiting; it does not carry a visited-agent chain through nested `sessions_send` calls, so the reported A->B->C->A cascade remains plausible. The linked fix PR is closed unmerged, and #51832 covers a different channel error-cascade mechanism rather than this `sessions_send` graph-cycle gap.

## What This Changes

Request graph-aware `sessions_send` loop detection that blocks triangular or polygonal multi-agent cascades which bypass pairwise `maxPingPongTurns`.

## Best Possible Solution

Implement a narrow core guard for `sessions_send`: add an optional normalized visited-agent chain to inter-session provenance and the gateway schema, thread incoming provenance through tool construction and nested A2A steps, block delivery before targeting an agent already in the chain, and add regression coverage for self-send, A->B->C->A, legacy/non-agent session keys, schema acceptance, and unchanged pairwise `maxPingPongTurns`.

## Reproduction Assessment

Yes. The issue gives a concrete A->B->C->A path, and current-main source inspection shows there is no chain state that could stop that path; this read-only review did not run a live multi-agent deployment.

## Solution Assessment

Yes. A provenance-chain guard is the narrowest maintainable solution, but it should include additive gateway schema support and canonical agent-ID normalization rather than copying the earlier unmerged PR verbatim.
