## Summary

Keep open. Current main still has source-only inter-session provenance and pairwise A2A ping-pong limiting, while the linked implementation PR is closed unmerged and the related Slack error-cascade issue is a different mechanism.

## What This Changes

Add graph-aware provenance or cycle detection to `sessions_send` so multi-agent A->B->C->A cascades are blocked instead of bypassing `maxPingPongTurns`.

## Best Possible Solution

Implement a narrow core `sessions_send` guard that threads a normalized optional visited-agent chain through inter-session provenance, blocks delivery when the target is already in that chain, and preserves existing pairwise `maxPingPongTurns` behavior.

## Reproduction Assessment

Yes. The issue gives a concrete A->B->C->A path, and current-main source inspection shows no chain or visited-agent state in provenance, tool delivery, or nested A2A steps; this review did not run a live multi-agent deployment.

## Solution Assessment

Yes. The provenance-chain direction is the narrowest maintainable solution, provided the repair also updates the gateway schema and normalizes agent IDs so it does not repeat the closed PR's schema gap.
