## Summary

Keep open: current main and latest release still carry only source-session provenance plus the pairwise A2A ping-pong guard, not a propagated visited-agent chain. The open draft PR is the current candidate fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests graph-aware sessions_send cycle detection so multi-agent A-to-B-to-C-to-A cascades cannot bypass the existing pairwise maxPingPongTurns guard.

## Best Possible Solution

Land one canonical fix that propagates normalized visited-agent IDs through InputProvenance and blocks revisiting an agent before sessions_send delivery, using the open candidate PR as the review path unless maintainers replace it.

## Reproduction Assessment

Yes at source level: current main carries no chain/visited-agent state in runtime or protocol provenance, and the A2A helper only alternates one requester-target pair. I did not run a live multi-agent deployment in this read-only review.

## Solution Assessment

Yes: carrying normalized visited-agent state in InputProvenance and blocking before delivery is the right owner boundary because both gateway and in-process sessions_send handoffs already consume that contract. The open candidate PR should be reviewed and landed or closed before any duplicate fix work starts.
