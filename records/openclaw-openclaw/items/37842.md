## Summary

Keep open: current main and v2026.6.6 still have pairwise `sessions_send` reply-back limiting, but no propagated agent-chain/trace state to stop triangular or polygonal cascades. The linked implementation attempt captured the right shape but is closed unmerged, and the Slack cascade sibling was fixed through a different inbound-channel guard.

## What This Changes

Request graph-aware `sessions_send` cycle detection by propagating inter-session chain state and blocking 3+ agent loops that bypass `maxPingPongTurns`.

## Best Possible Solution

Add normalized agent-chain provenance through the runtime and gateway protocol, then block revisiting an agent before `sessions_send` delivery while preserving existing pairwise `maxPingPongTurns` behavior.

## Reproduction Assessment

Yes at source level: current `sessions_send` carries no chain state, and the A2A helper only alternates requester and target sessions. I did not run a live multi-agent deployment reproduction in this read-only review.

## Solution Assessment

Yes: a normalized chain in `InputProvenance` plus a pre-delivery `sessions_send` cycle guard is the narrow maintainable fix. Prompt-only rules or a new config knob would not reliably stop circular delivery paths.
