## Summary

This issue remains necessary: current main preserves only immediate inter-session provenance and limits the built-in two-session reply flow, but it does not carry prior participants into the next run’s tool construction, so A→B→C→A remains uncorrelated. The linked implementation at https://github.com/openclaw/openclaw/pull/37945 closed unmerged, and no merged replacement now owns the work.

## What This Changes

Add graph-aware cycle detection so a multi-agent `sessions_send` path stops when it revisits a prior participant.

## Best Possible Solution

Extend normalized inter-session provenance with a bounded canonical participant path, pass it through run tool construction, and reject a repeated target before delivery while retaining `maxPingPongTurns` for the existing pairwise reply flow.

## Reproduction Assessment

Yes, at source level with high confidence: A→B records only A as the immediate source, B's tool construction does not receive that provenance, and B→C therefore starts a fresh path that permits C→A. A live multi-agent deployment was not run because this review is read-only.

## Solution Assessment

Yes in direction: the existing input-provenance handoff is the narrowest maintainable owner boundary, but the implementation should use canonical identities and prove both cycle blocking and non-cycle coordination rather than copy the earlier PR's display-key assumptions.
