## Summary

The issue remains necessary: current `main` still bounds only the built-in pairwise reply flow, while each new `sessions_send` creates provenance without a visited-agent path, so a multi-agent A→B→C→A cascade is not correlated across runs. The linked fix PR closed unmerged and exposed one narrow canonical-ID defect, but no current implementation or viable replacement now owns the work.

## What This Changes

Add graph-aware cycle prevention for multi-agent `sessions_send` chains that bypass the direct pairwise `maxPingPongTurns` guard.

## Best Possible Solution

Propagate a bounded list of canonical visited agent IDs through the existing `InputProvenance` owner boundary, reject a send before delivery when the normalized target is already present, and retain `maxPingPongTurns` for its existing direct reply-round contract.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: enable agent-to-agent tools, have A call B, B call C, and C call A; each send starts with provenance that lacks the earlier agents, while the existing bound only governs one pairwise follow-up flow. I did not run the live three-agent scenario because this review is read-only.

## Solution Assessment

Yes, a provenance-owned visited-agent chain is the narrowest maintainable fix, provided agent IDs are canonicalized and the field is propagated through every recipient-run construction path. A new config knob or separate orchestration subsystem is not needed.
