## Summary

Keep open: current main still lacks graph-aware `sessions_send` chain provenance. The focused implementation PR was closed unmerged, and the open Slack cascade issue is related but covers a different gateway-error mechanism, so this remains a bounded safety feature rather than a documented `maxPingPongTurns` bug.

## What This Changes

Request graph-aware chain or trace provenance for `sessions_send` so multi-agent cycles like A->B->C->A can be detected instead of relying only on pairwise `maxPingPongTurns`.

## Best Possible Solution

Add typed, normalized A2A chain provenance through the current provenance schema and tool-construction path, reject explicit-agent target revisits in `sessions_send`, and preserve existing pairwise and non-agent behavior.

## Reproduction Assessment

Yes, by source inspection. Current main retains only immediate inter-session provenance, and `maxPingPongTurns` alternates requester/target in one pair, so A->B->C->A has no path-aware state.

## Solution Assessment

Yes in direction. Provenance-carried chain tracking is the narrow maintainable solution, but it should normalize agent IDs and be implemented against the current schema/tool path rather than reviving the closed PR verbatim.
