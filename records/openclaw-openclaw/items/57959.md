## Summary

Keep open: current main does not implement the requested fallback media-trust behavior, but this PR still needs real behavior proof and rework around provider media policy, skipped-candidate handling, prompt-image suppression, and partial-execution production.

## What This Changes

The branch adds failure-aware retry image forwarding, cross-provider prompt-image suppression, and partial-execution retry context plumbing with focused tests in agent fallback code.

## Best Possible Solution

Rebase or replace the branch with one provider-media trust policy that covers explicit images, prompt-detected refs, skipped candidates, format failures, config/docs, upgrade behavior, and real partial-execution producers.

## Reproduction Assessment

Yes, for source-level review: the PR head shows image stripping depends on callback-local retry state, while the fallback loop can skip candidates before the run callback. I did not run a live cross-provider fallback in this read-only pass.

## Solution Assessment

No. The PR is pointed at the right boundary, but the maintainable solution needs one provider media-trust policy plus producer-side partial-execution metadata, not callback-local image decisions and inert warning plumbing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Guard skipped cross-provider fallback attempts:** `src/agents/command/attempt-execution.ts:321-323`
  - body: `isFallbackRetry` is derived from how many run callbacks have executed, but `runWithModelFallback` can skip a primary candidate before calling `run`. In that case the first actual callback can be for a different provider with `isFallbackRetry=false`, so this early return forwards images and prompt-image suppression stays off across the provider boundary.
  - confidence: 0.86
- **[P1] Make provider media trust an explicit policy:** `src/agents/command/attempt-execution.ts:329-330`
  - body: This hard-codes image stripping whenever the provider changes, but the linked provider-trust issue calls for a documented trust policy and default/upgrade behavior. Landing the fail-closed policy without that maintainer decision can break or confuse existing cross-provider fallback expectations.
  - confidence: 0.82
- **[P2] Suppress prompt detection whenever retry images are stripped:** `src/agents/command/attempt-execution.ts:556`
  - body: `resolveRetryImages` strips explicit images for same-provider `format` failures too, but `suppressPromptImageDetection` is only enabled for cross-provider retries. A prompt that references a local image can therefore reload and send images after the explicit payload was stripped for the retry.
  - confidence: 0.8
- **[P1] Populate partial execution before threading it:** `src/agents/model-fallback.ts:818-831`
  - body: `runWithModelFallback` only emits the replay warning when a previous `FailoverError` carries `partialExecution`, but the production throw sites in this diff do not attach that metadata. As merged, the new partial-execution warning path would not fire for real fallback retries.
  - confidence: 0.78
