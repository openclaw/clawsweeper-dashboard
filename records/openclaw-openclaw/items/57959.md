## Summary

Keep open. This PR carries useful fallback-safety work from the earlier design discussion and the prompt-image issue, but the current head still has blocking correctness gaps, lacks contributor real behavior proof, and is now conflicting with current main.

## What This Changes

The PR adds failure-aware retry image handling, cross-provider prompt-image suppression plumbing, and partial-execution context threading for agent model fallback, with unit tests.

## Best Possible Solution

Rebase a narrower fallback-safety patch that uses one shared strip/suppression decision for explicit and prompt-detected images, and only ships replay warnings once partial-execution metadata is populated at real failure sites.

## Reproduction Assessment

Yes, source inspection is enough for the review findings: the PR head strips explicit images for same-provider format retries while suppressing prompt image detection only for cross-provider retries, and it forwards partialExecution without adding real production throw-site population.

## Solution Assessment

No. The image-safety direction is useful, but this patch should align prompt-image suppression with the same decision that strips retry images and either populate partial-execution metadata in production or defer the replay warning path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Suppress prompt images whenever retry images are stripped:** `src/agents/command/attempt-execution.ts:556`
  - body: `resolveRetryImages` strips explicit images after a `format` failure, but the embedded runner only suppresses prompt-based image detection on cross-provider retries. A same-provider format retry with a prompt-referenced local image will still load and send that image again, defeating the strip policy and likely re-triggering the same failure.
  - confidence: 0.88
- **[P2] Populate partial execution before warning retries:** `src/agents/model-fallback.ts:817-831`
  - body: The fallback loop forwards `previousPartialExecution`, but this PR adds no production `FailoverError` construction that sets `partialExecution`; only tests set it, and the author described throw-site population as future runner work. The replay warning therefore does not fire in real runs, so non-idempotent tool replay remains unfixed.
  - confidence: 0.88
