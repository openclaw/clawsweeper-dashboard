## Summary

Keep open: this PR is a plausible fallback-safety fix, but the current head still has two correctness blockers and no contributor-supplied real behavior proof. Current main has not implemented the central changes, so this should stay open for contributor/maintainer follow-up rather than cleanup closure.

## What This Changes

The PR changes agent fallback retry handling to selectively preserve or strip images, suppress prompt-detected images on cross-provider retries, and pass failure/partial-execution context into later fallback attempts.

## Best Possible Solution

Rebase a narrower fallback-safety patch that drives explicit-image stripping and prompt-image suppression from one shared decision, and add real partial-execution producers before shipping replay warnings.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough to reproduce the review findings: same-provider format retries strip explicit images but keep prompt image detection enabled, and partialExecution is consumed without any production producer.

## Solution Assessment

No. The image-safety direction is useful, but the current patch should align prompt-image suppression with the strip decision and either populate partial-execution metadata at real failure sites or defer the warning path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Suppress prompt detection whenever retry images are stripped:** `src/agents/command/attempt-execution.ts:556`
  - body: `resolveRetryImages` returns `undefined` for same-provider `format` retries, but `suppressPromptImageDetection` is only set for cross-provider retries. A retry after an image-format failure can still reload a local image referenced in the prompt and send it again, defeating the strip policy and likely causing the same failure.
  - confidence: 0.88
- **[P2] Populate partial execution before warning retries:** `src/agents/model-fallback.ts:817-831`
  - body: The fallback loop forwards `previousPartialExecution`, but the diff adds no production `FailoverError` throw site that sets `partialExecution`; the only construction with that field is in tests, and the author confirmed producer work is future runner work. The replay warning therefore never fires in real fallback runs.
  - confidence: 0.88
