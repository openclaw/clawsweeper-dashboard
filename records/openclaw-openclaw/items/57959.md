## Summary

Keep open: this PR still targets a real agents fallback gap that is not on current main, but it is not merge-ready because prompt-detected images can still be reintroduced on same-provider format retries, the partial-execution warning path is not populated in production, and external real behavior proof is absent.

## What This Changes

The branch adds failure-aware fallback image forwarding, cross-provider prompt-image suppression, partial-execution fields/context threading, and focused agents fallback tests.

## Best Possible Solution

Rebase and land a narrower fallback-safety patch that suppresses both explicit and prompt-detected images whenever the retry image policy strips them, and either populates partial-execution metadata from runner/CLI failure sites or removes the inactive warning behavior.

## Reproduction Assessment

Yes. Source inspection shows current main still strips explicit retry images unconditionally, still loads prompt-detected images, and lacks partial-execution payload fields; the PR diff also shows the same-provider format-retry prompt-image gap.

## Solution Assessment

No. The image-safety direction is useful, but the patch should align prompt-image suppression with the same policy that strips retry images and make partial-execution metadata real before claiming replay protection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Suppress prompt image loading when retry images are stripped:** `src/agents/command/attempt-execution.ts:552`
  - body: `resolveRetryImages` drops explicit images when the previous failure was `format`, but `suppressPromptImageDetection` is only set for cross-provider retries. A same-provider format retry with a prompt-referenced local image will still run prompt image detection and send images again, defeating the format-error strip and likely re-triggering the same failure.
  - confidence: 0.87
- **[P2] Populate partial execution before warning retries:** `src/agents/model-fallback.ts:817`
  - body: The fallback loop forwards `previousPartialExecution`, but the PR does not add any production `FailoverError` construction that sets `partialExecution`; the author also acknowledged this as future plumbing. The replay warning therefore never fires outside tests, so non-idempotent tools can still be replayed after a failed attempt.
  - confidence: 0.88
