## Label Justifications

- `P2`: The issue is a bounded but real exec-policy/session-state drift affecting Discord operators after config changes.
- `impact:session-state`: The core problem is durable session entry state carrying `exec*` values into later turns and restarts.
- `impact:security`: A retained `execSecurity` value can keep broader command-execution permission after an operator tightens global exec policy.

## Summary

Keep open. Current main and v2026.6.6 still store authorized `/exec` defaults in session entries and resolve those stored values ahead of global or agent exec policy; the merged ACP cache fix covers a separate stale runtime-handle path, so the security/config precedence question remains.

## What This Changes

The issue asks OpenClaw to stop persisted Discord session `/exec` overrides from silently shadowing later global exec-security config changes.

## Best Possible Solution

Define the intended lifetime and precedence of `/exec` session defaults, then update persistence, resolution, reset, and migration behavior so tightened operator exec policy cannot be silently shadowed by stale session fields.

## Reproduction Assessment

Yes, source-level: current main persists `exec*` fields into `SessionEntry` and later resolves them ahead of agent/global exec policy. I did not run a live Discord restart reproduction in this read-only review.

## Solution Assessment

Unclear. The narrow code paths are known, but choosing between non-persistence, different precedence, reset/migration behavior, or a deliberate durable override contract is a product/security decision.
