## Summary

Keep open. Current main and the latest release still persist authorized `/exec` session defaults and resolve stored session exec values ahead of later global or agent exec policy, while merged related work only fixed an adjacent ACP runtime-cache staleness path and current docs/tests intentionally describe durable per-session defaults.

## What This Changes

The issue requests that Discord `/exec` session overrides stop silently shadowing later global exec-security configuration changes after restart.

## Best Possible Solution

Define the intended lifetime and precedence of session-level `/exec` defaults, then align persistence/resolution, migration or reset behavior, focused tests, and docs so global operator policy changes cannot be silently shadowed by stale session state unless that durable override is explicitly the supported contract.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` still persist `exec*` fields into session entries and resolve them ahead of configured global or agent exec policy for later runs.

## Solution Assessment

Unclear. The reporter's desired behavior is plausible for security enforcement, but current docs and tests intentionally preserve authorized external-channel session defaults, so maintainers need to choose between non-persistence, different precedence, reset/migration behavior, or a deliberate durable override contract.
