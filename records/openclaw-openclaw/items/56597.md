## Summary

Keep open. Current main and v2026.6.8 still persist `/exec` fields in session entries and resolve stored values before agent/global exec defaults; docs and tests now describe that as per-session behavior, so this needs a maintainer security/product decision rather than cleanup close.

## What This Changes

The issue asks OpenClaw to stop Discord `/exec` session overrides from persisting indefinitely or from taking precedence over later global `tools.exec` security changes.

## Best Possible Solution

Define the canonical `/exec` precedence and reset policy, then implement a tested change that makes operator security policy enforceable without accidentally breaking documented session-default behavior.

## Reproduction Assessment

Yes by source inspection: current main writes `execHost`/`execSecurity`/`execAsk`/`execNode` into `SessionEntry` and later resolves them before agent/global defaults. I did not run a live Discord gateway restart in this read-only review.

## Solution Assessment

Unclear; changing resolver precedence alone is plausible but may violate documented `/exec` session defaults. The safer path is a maintainer-approved policy plus focused tests and migration or reset behavior for existing stored entries.
