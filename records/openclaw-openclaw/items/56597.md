## Summary

Keep open. Current main and the latest release still persist `/exec` fields in session entries and resolve them before agent/global exec defaults; docs/tests now treat that as intended per-session behavior, so this needs maintainer security/product decision rather than cleanup close.

## What This Changes

The issue asks OpenClaw to stop Discord `/exec` session overrides from persisting ahead of later global exec-security config changes.

## Best Possible Solution

Define the canonical `/exec` precedence and reset policy, then implement a tested change that makes hard operator security policy enforceable without accidentally breaking documented session-default behavior.

## Reproduction Assessment

Yes by source inspection: current main writes execHost/execSecurity/execAsk/execNode into `SessionEntry` and later resolves them before agent/global defaults. I did not run a live Discord gateway restart.

## Solution Assessment

Unclear; changing resolver precedence alone is plausible but may violate documented `/exec` session defaults. The safer path is a maintainer-approved policy plus focused tests and migration/reset behavior for existing stored entries.
