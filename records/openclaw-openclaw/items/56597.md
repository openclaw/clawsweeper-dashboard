## Summary

Keep open. Current main still persists `/exec` session defaults into durable session state and resolves those stored values before configured agent/global exec defaults; tests and docs also describe sticky per-session exec behavior, so this is an unresolved exec-policy decision rather than a cleanup close.

## What This Changes

The issue requests that Discord sessions stop letting persisted per-session exec directives override later global `tools.exec` config changes after a gateway restart.

## Best Possible Solution

Define the intended admin-policy precedence for exec settings, then update the generic session exec persistence/resolution path, session reset or config-version behavior if needed, docs, and focused regression tests so tightened operator policy takes effect for existing Discord sessions without a Discord-only workaround.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by resolving a persisted session entry with `execSecurity: "full"` after global config is changed to a stricter value; current main reads the stored session value before configured defaults.

## Solution Assessment

Unclear. The report identifies a real policy conflict, but the narrow maintainable fix depends on whether maintainers want global `tools.exec` to be a hard ceiling, host approvals to be the hard enforcement layer, or persisted session overrides to carry config-version/reset semantics.
