## Label Justifications

- `P2`: The issue describes a bounded but real session exec-policy drift affecting Discord/operator security posture without evidence of a current emergency.
- `impact:session-state`: The central problem is persisted session entry state carrying `exec*` values across later turns, resets, and restarts.
- `impact:security`: A retained session `execSecurity` value can keep broader command-execution permission after an operator tightens global exec policy.

## Summary

Keep open. Current main and the latest release still persist `/exec` session defaults and still resolve stored session exec values ahead of configured global/agent exec policy; the merged ACP cache fix only covers cached runtime-handle freshness, not this session-entry precedence question.

## What This Changes

The issue asks OpenClaw to stop Discord session-stored `exec*` directive defaults from silently overriding later global `tools.exec` security changes.

## Best Possible Solution

Define the intended lifetime and precedence of session-level exec overrides, then update persistence/resolution plus focused tests so operator global exec policy changes are not silently shadowed by old stored directives.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` still persist `exec*` fields into session entries and resolve them ahead of configured global/agent exec policy for later runs.

## Solution Assessment

Unclear. The mechanical fixes are plausible, but choosing non-persistence, different precedence, reset/migration behavior, or a deliberate durable override contract is a product/security decision.
