## Label Justifications

- `P2`: The issue is a bounded exec/session-state security-policy problem rather than an immediate outage or confirmed broad exploit.
- `impact:security`: Persisted exec overrides can preserve a looser requested host-exec policy after an operator tightens global configuration.
- `impact:session-state`: The reported behavior is caused by durable session state carrying exec override fields across later turns and restarts.

## Summary

Keep open. Current main still persists authorized `/exec` options into durable session state and resolves those stored exec fields before agent/global `tools.exec` settings; the related ACP cache fix only addressed stale runtime-handle reuse, not stored session override precedence.

## What This Changes

The issue asks for global `tools.exec` changes to take precedence over durable Discord session `/exec` overrides after restarts or config changes.

## Best Possible Solution

Define the exec policy contract first, then align persistence, resolver precedence, docs, migration or reset behavior, and focused tests with that contract.

## Reproduction Assessment

Yes, at source level. A session entry containing `execSecurity: "full"` is still resolved before stricter agent/global `tools.exec.security`, and current tests/docs still encode sticky session behavior.

## Solution Assessment

Unclear. The report identifies a real security-policy conflict, but the best fix depends on a maintainer decision about global config as a hard ceiling versus sticky authorized session defaults with reset or migration semantics.
