## Summary

Keep open. Current main and latest release v2026.6.10 still persist `/exec` fields in session entries and resolve stored session values before config-derived defaults, while docs/tests now describe that as per-session behavior; this needs a maintainer security/product decision rather than cleanup close.

## What This Changes

The issue reports that Discord session-stored `/exec` directives continue overriding later global exec security config after gateway restarts.

## Best Possible Solution

Define the canonical `/exec` precedence and reset policy, then implement a tested session-state migration or reset path that lets operator security policy take effect without accidentally breaking documented session defaults.

## Reproduction Assessment

Yes, by source inspection: current main writes `/exec` fields into `SessionEntry` and later resolves those stored fields before config-derived defaults. I did not run a live Discord gateway restart in this read-only review.

## Solution Assessment

Unclear. A simple resolver-order change is plausible, but current docs and tests describe `/exec` as persistent session state, so the safer solution needs maintainer-approved policy plus migration/reset handling for existing stored entries.
