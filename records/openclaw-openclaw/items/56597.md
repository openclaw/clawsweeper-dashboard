## Label Justifications

- `P2`: The issue is a normal-priority security-policy and session-state problem with a bounded exec/session surface rather than an immediate outage.
- `impact:security`: Persisted session exec overrides can preserve a looser requested exec policy after an operator tightens global configuration.
- `impact:session-state`: The reported behavior is caused by durable session state carrying exec override fields across later turns and restarts.

## Summary

Keep open. Current main and the latest release still persist authorized `/exec` defaults into durable session state and resolve those stored values before agent/global exec defaults; the related ACP cache fix only addressed a different stale-runtime path.

## What This Changes

The issue asks OpenClaw to stop persisted Discord session exec directives from overriding later global `tools.exec` security configuration changes.

## Best Possible Solution

Define whether global `tools.exec` is a hard ceiling or whether stored `/exec` defaults remain intentionally sticky, then update persistence, resolver precedence, docs, and focused upgrade tests to match that policy.

## Reproduction Assessment

Yes, at source level. A session entry containing `execSecurity: "full"` is still resolved before stricter agent/global `tools.exec.security` on later turns, and focused tests currently assert that sticky behavior.

## Solution Assessment

Unclear. The report identifies a real security-policy conflict, but the best fix depends on a maintainer decision about whether global exec config should cap stored session overrides or whether the documented sticky session model should remain with a separate reset/version mechanism.
