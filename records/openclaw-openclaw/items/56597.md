## Summary

Keep open. Current main and latest release v2026.6.10 still persist `/exec` fields in session entries and resolve stored session values before config-derived defaults, while docs/tests now describe that as per-session behavior; this needs a maintainer security/product decision, not cleanup close.

## What This Changes

Discord sessions are reported to retain stored per-session exec directives after global exec security config changes, causing old session state to override current policy.

## Best Possible Solution

Define the canonical `/exec` precedence and reset policy, then implement a tested session-state migration or reset path that makes operator security policy enforceable without accidentally breaking documented session defaults.

## Reproduction Assessment

Yes, by source inspection: current main writes `/exec` fields into `SessionEntry` and later resolves those stored fields before config-derived defaults. I did not run a live Discord gateway restart in this read-only review.

## Solution Assessment

Unclear; a simple resolver-order change is plausible but may violate documented `/exec` per-session defaults. The safer solution needs a maintainer-approved policy plus focused tests and migration/reset handling for existing stored entries.
