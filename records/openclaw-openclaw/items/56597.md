## Label Justifications

- `P2`: The issue is a bounded exec/session-state security-policy problem rather than an immediate outage or confirmed broad exploit.
- `impact:security`: Persisted exec overrides can preserve a looser requested host-exec policy after an operator tightens global configuration.
- `impact:session-state`: The reported behavior is caused by durable session state carrying exec override fields across later turns and restarts.

## Summary

Keep open. Current main and the latest tagged release still persist authorized per-session `/exec` options and resolve stored session exec fields before agent/global `tools.exec` settings; the related ACP cache fix addressed stale runtime-handle reuse, not this stored session override precedence.

## What This Changes

The issue reports that Discord sessions persist per-session exec directive fields and continue using them after global exec policy changes.

## Best Possible Solution

Define the exec policy contract first, then align persistence, resolver precedence, docs, migration or reset behavior, and focused tests with that contract.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.20` both persist `execSecurity` into session state and resolve it before stricter agent/global `tools.exec.security`.

## Solution Assessment

Unclear. The report identifies a real security-policy conflict, but current docs and tests encode sticky `/exec` session defaults, so the best fix depends on a maintainer decision about the desired contract.
