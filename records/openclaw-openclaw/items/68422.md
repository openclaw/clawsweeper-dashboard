## Summary

Keep open. The linked chat-width regression still appears valid on current main, but this PR should not merge as written because the fixed `.chat-thread-inner` cap would override the documented configurable grouped-message width, and the PR still lacks after-fix browser proof.

## What This Changes

The PR changes Control UI chat CSS to cap and center the chat thread and composer at 920px.

## Best Possible Solution

Constrain the chat welcome/thread/composer in `ui/src/styles/chat/layout.css` while preserving `gateway.controlUi.chatMessageMaxWidth`, then attach real wide-viewport `/chat` proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows the real chat view emits `.chat-thread-inner` and `.agent-chat__input`, while the stylesheet has no width or max-width centering for those selectors; I did not run a browser session.

## Solution Assessment

No as written. The PR is in the right CSS owner area, but it should either constrain only the welcome/composer surfaces or preserve the existing `--chat-message-max-width` contract for grouped messages.

## Review Findings

Overall correctness: patch is incorrect
