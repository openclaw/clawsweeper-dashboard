## Summary

Keep open: the linked chat-width regression is still valid, but this PR currently hard-caps the transcript wrapper in a way that conflicts with the documented configurable grouped-message width, and it still lacks after-fix browser proof.

## What This Changes

The PR changes Control UI chat layout CSS to cap and center the chat thread wrapper and composer at 920px.

## Best Possible Solution

Adjust `ui/src/styles/chat/layout.css` so the welcome/thread/composer stay readable on wide viewports while preserving `--chat-message-max-width` for grouped messages, then attach real wide-viewport browser proof.

## Reproduction Assessment

Yes from source inspection: current main renders `.chat-thread-inner` and `.agent-chat__input`, while current `layout.css` has no width or max-width centering for those selectors. I did not run a live browser session.

## Solution Assessment

No as written. The PR is in the right CSS owner area, but the fixed 920px `.chat-thread-inner` cap can override the existing `gateway.controlUi.chatMessageMaxWidth` contract for grouped messages.
