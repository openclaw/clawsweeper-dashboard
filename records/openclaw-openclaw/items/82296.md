## Summary

Keep open. The latest patch adds a useful TUI consumer for lifecycle `fallback_step` events, but it still does not make the local embedded TUI fallback path emit that event, so the live footer can remain stale; the protected `maintainer` label also prevents cleanup closure.

## What This Changes

The PR updates TUI agent-event handling and colocated tests so lifecycle `fallback_step` events update `sessionInfo.modelProvider` and `sessionInfo.model` for active or pending runs.

## Best Possible Solution

Emit run-scoped lifecycle `fallback_step` model updates from the embedded TUI `agentCommandFromIngress` fallback path, keep the TUI consumer/tests, and add redacted live runtime proof before merge.

## Reproduction Assessment

Yes, at source level: current main's footer reads `sessionInfo`, the embedded backend forwards only received agent events, and the local fallback path records fallback steps only to trajectory. The linked proof shows the new handler tests failing on origin/main, but I did not establish a live multi-provider TUI fallback run.

## Solution Assessment

No. Consuming `fallback_step` in the TUI is a useful piece, but the local TUI producer also needs to emit the run-scoped lifecycle event before this is the narrow complete fix.
