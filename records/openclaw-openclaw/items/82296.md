## Summary

Keep open: the PR adds a useful TUI consumer for fallback-step model updates, but current main still does not emit those fallback-step lifecycle events from the embedded TUI run path, so the live footer bug can remain.

## What This Changes

The PR updates `src/tui/tui-event-handlers.ts` and its tests so lifecycle `fallback_step` events can update `sessionInfo.modelProvider` and `sessionInfo.model` for active or pending TUI runs.

## Best Possible Solution

Emit run-scoped fallback-step lifecycle model updates from the embedded `agentCommandFromIngress` fallback path, keep the TUI consumer/tests, and add redacted runtime proof before merge.

## Reproduction Assessment

Yes, at source level: the footer renders from `sessionInfo`, the embedded backend only forwards received agent events, and the local fallback callback currently records fallback steps only to trajectory data. The linked artifacts also show the new handler tests failing on origin/main and passing on the PR branch.

## Solution Assessment

No. Consuming `fallback_step` in the TUI is a useful part of the fix, but the local embedded run path still needs to emit the run-scoped lifecycle event for live fallback attempts.
