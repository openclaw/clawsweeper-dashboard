## Summary

Keep open. The TUI handler/test direction is useful, but this PR is not merge-ready because it only consumes `lifecycle` `fallback_step` events while the TUI agent-command fallback path still records those steps to trajectory without emitting them as agent events; the submitted proof is also test-only rather than real TUI fallback behavior.

## What This Changes

The PR adds TUI event-handler logic and regression tests that update `sessionInfo.modelProvider` and `sessionInfo.model` when a lifecycle `fallback_step` event carries `fallbackStepToModel`.

## Best Possible Solution

Add a live, run-scoped fallback model signal on the actual TUI run path and consume that same contract in the footer handler, with focused producer/consumer tests and redacted real TUI proof.

## Reproduction Assessment

Yes, at source level: the footer reads `sessionInfo` while the current TUI fallback path records fallback steps without emitting a live fallback-step event to the handler. I did not run a live provider fallback in this read-only review.

## Solution Assessment

No. Handling `fallback_step` in the TUI is a useful consumer-side piece, but the fix needs the producer-side event on the actual TUI/gateway run path or it will not change the reported runtime behavior.
