## Summary

Keep open: the stale direct-bridge gateway fallback idea is useful, but this PR also broadens unavailable Codex PreToolUse handling into a no-op, which current main deliberately avoids at the native tool approval boundary. The branch is also currently conflicting and lacks real OpenClaw/Codex stale-bridge proof, while the canonical restart/drain issue remains open with a separate candidate PR that is not yet proof-positive enough to supersede this one.

## What This Changes

The branch makes stale or unusable direct native hook bridge errors fall through to gateway `nativeHook.invoke`, changes unavailable `PreToolUse` relay responses from denial to empty no-op output, and updates CLI relay tests.

## Best Possible Solution

Preserve the useful stale-bridge gateway fallback only if unknown PreToolUse and PermissionRequest unavailable states remain fail-closed and the explicit no-policy noop marker stays the only no-op path, or resolve the broader restart/drain bug through the canonical issue and a proofed successor PR.

## Reproduction Assessment

Yes at source level: current code and tests show stale direct-bridge errors stop before gateway fallback, and Codex source shows empty successful PreToolUse output does not block. I did not run a live WSL2/OpenClaw/Codex gateway restart scenario in this read-only review.

## Solution Assessment

No. Falling through from stale direct bridge to gateway is plausible, but making all unavailable PreToolUse events no-op is not the best fix because current main intentionally keeps unknown policy state fail-closed.
