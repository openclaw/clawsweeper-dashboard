## Label Justifications

- `P1`: The PR targets a broken Codex native hook relay workflow affecting native tool execution and modifies a permission-sensitive fallback path.
- `merge-risk: 🚨 compatibility`: Unavailable PreToolUse currently denies by default, and this PR changes existing Codex hook sessions to return no output on relay outage.
- `merge-risk: 🚨 security-boundary`: PreToolUse enforces OpenClaw before-tool policy and loop detection before native tool execution, and this PR can bypass that boundary during relay outage.

## Summary

The stale direct-bridge fallback idea is still useful, but this branch is not merge-ready because it also changes unavailable PreToolUse from fail-closed to no-op. Current `main` already has the safer explicit no-policy marker from the merged replacement for the related fallback work, and this PR now conflicts with that path.

## What This Changes

The PR changes native hook relay CLI handling so stale direct bridge errors fall through to the gateway path and unavailable PreToolUse relay responses return no output instead of a deny response.

## Best Possible Solution

Keep the stale direct-bridge-to-gateway fallback idea, but rebase it onto current `main` and preserve deny-by-default unavailable PreToolUse behavior except for the existing explicit no-policy marker.

## Reproduction Assessment

Yes, source-reproducible: current main still returns unavailable fallback for stale direct bridge registration errors before trying the gateway path. The PR's unsafe fail-open behavior is also visible in the diff and in Codex's PreToolUse parser contract.

## Solution Assessment

No. Falling through from stale direct bridge records to the gateway is a reasonable fix direction, but making unknown unavailable PreToolUse fail open is not the safest maintainable solution.
