## Label Justifications

- `P2`: This is a normal-priority plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Plugin command authors cannot currently send a private initial slash response plus a visible follow-up, so intended command output can be hidden from the conversation.

## Summary

Keep open: current main and v2026.6.6 still expose native plugin commands as a generic `PluginCommandContext` plus a returned `ReplyPayload`, while Discord responder primitives exist internally but are not bridged into command handlers.

## What This Changes

Request to add Discord slash interaction responder controls such as defer, reply, follow-up, and edit to native plugin command handlers.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel-capability contract, with a Discord implementation, while preserving the existing returned-`ReplyPayload` path and auth/routing behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the command handler receives only `PluginCommandContext` and returns one `PluginCommandResult`, while Discord responder methods are internal or interactive-handler-only.

## Solution Assessment

Unclear. The requested Discord-specific responder shape is plausible, but maintainers should decide whether the durable SDK surface is Discord-specific or a generic command responder/channel capability.
