## Summary

Keep open. Current main still lacks the requested Discord responder API for native plugin commands, and solving it is a public plugin SDK plus Discord interaction-lifecycle design item rather than a conservative close candidate.

## What This Changes

The issue requests Discord-specific responder controls for native plugin command handlers so plugins can defer, reply, follow up, and edit slash-command responses.

## Best Possible Solution

Add an additive typed Discord responder facade for native `api.registerCommand` handlers after authorization and route resolution, preserve the existing `ReplyPayload` return behavior, and cover guild/group-DM mixed-visibility flows in docs and tests.

## Reproduction Assessment

Yes, by source inspection. A Discord native plugin command reaches `executePluginCommand`, which only passes `PluginCommandContext`, and the handler result is delivered once afterward through `deliverDiscordInteractionReply`.

## Solution Assessment

Unclear until maintainers approve the API. The requested direction is valid, but the maintainable fix should expose a narrow typed responder facade rather than raw Discord interaction objects.
