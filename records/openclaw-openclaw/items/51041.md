## Summary

Keep open: current main and v2026.6.10 still lack plugin-command access to Discord defer, reply, follow-up, or edit-reply controls. The request is source-reproducible and fits the plugin API roadmap, but it changes a public SDK and Discord routing contract, so it needs maintainer API direction rather than cleanup closure.

## What This Changes

The issue requests Discord slash interaction response controls such as defer, reply, follow-up, and edit-reply in plugin command handlers.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel capability with a Discord implementation, while preserving current returned-payload behavior and command auth/routing.

## Reproduction Assessment

Yes. Source inspection shows current main still types plugin command handlers as one PluginCommandContext returning one PluginCommandResult, and the Discord native-command path only delivers that returned payload.

## Solution Assessment

Unclear. A responder capability is the right class of solution, but maintainers should choose whether the permanent SDK surface is generic or Discord-specific before implementation.
