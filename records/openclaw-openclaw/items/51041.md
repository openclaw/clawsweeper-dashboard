## Label Justifications

- `P2`: The request is a normal-priority plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Discord plugin command handlers cannot currently send a private initial slash response plus a visible follow-up, so intended command output can be suppressed from the conversation.

## Summary

Keep open: current main and v2026.6.6 still expose plugin commands as a generic command context plus one returned payload, while Discord interaction responder primitives remain internal or limited to separate interactive-handler paths.

## What This Changes

The issue requests Discord interaction response controls for plugin command handlers so commands can defer, reply, follow up, and edit slash-command responses.

## Best Possible Solution

Design a narrow, documented plugin-command responder contract that reuses Discord’s existing interaction primitives while preserving the current returned-payload path for existing plugins.

## Reproduction Assessment

Yes. Source inspection on current main shows `PluginCommandContext` has no responder API, `executePluginCommand` only passes that generic context, and Discord native plugin commands deliver one returned payload after the handler completes.

## Solution Assessment

Unclear. The requested `ctx.discord.respond` shape is plausible, but the best fix needs maintainer API design so this does not become a narrow Discord-only SDK surface when a generic command responder contract might be better.
