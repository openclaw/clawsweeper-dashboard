## Label Justifications

- `P2`: This is a bounded plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Native Discord plugin commands cannot send both private slash acknowledgements and visible follow-ups, so intended command output can be hidden from the conversation.
- `maturity:stable`: The affected Discord surface is scored M4 Stable in `qa/maturity-scores.yaml`, including native command and control behavior.

## Summary

Keep open: current main and v2026.6.10 still expose plugin commands as one generic `PluginCommandContext` plus one returned `PluginCommandResult`; Discord has internal interaction primitives, but they are not bridged into `registerCommand` handlers. This is useful and source-reproducible, but it changes the public plugin SDK and Discord command-routing semantics, so it needs maintainer API direction.

## What This Changes

Expose Discord slash interaction response controls such as defer, reply, follow-up, and edit-reply to native OpenClaw plugin command handlers.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel capability, likely with a Discord implementation, while preserving current returned-payload behavior and command auth/routing.

## Reproduction Assessment

Yes, at source level: current main still types command handlers as `PluginCommandContext` plus one returned `PluginCommandResult`, and the Discord path only delivers that returned payload. I did not run live Discord because the missing public API is directly visible in the type and dispatch path.

## Solution Assessment

Unclear. A responder capability is the right class of solution, but maintainers should decide whether the permanent SDK surface is generic or Discord-specific before implementation.
