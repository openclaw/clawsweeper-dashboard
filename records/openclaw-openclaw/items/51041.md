## Summary

Keep open. Current main still lacks the requested Discord responder API for native plugin commands, and the remaining work is a public Plugin SDK plus Discord interaction-lifecycle design decision rather than a conservative cleanup close.

## What This Changes

Expose Discord slash interaction response controls such as defer, reply, follow-up, and edit-reply to native OpenClaw plugin command handlers.

## Best Possible Solution

Add an additive typed Discord responder facade for native `api.registerCommand` handlers after authorization and route resolution, preserve the existing `ReplyPayload` return behavior, and cover guild/group-DM mixed-visibility flows in docs and tests.

## Reproduction Assessment

Yes. By source inspection, a Discord native plugin command reaches `executePluginCommand`, which passes only `PluginCommandContext`, then the dispatcher delivers one returned `ReplyPayload` afterward through `deliverDiscordInteractionReply`.

## Solution Assessment

Unclear until maintainers approve the public API shape. The requested direction is valid, but the maintainable fix should expose a narrow typed responder facade rather than raw Discord interaction objects.
