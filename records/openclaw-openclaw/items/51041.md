## Label Justifications

- `P2`: This is a bounded plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Native Discord plugin commands cannot send both private slash acknowledgements and visible follow-ups, so intended command output can be hidden from the conversation.

## Summary

Keep open: current main and v2026.6.10 still expose plugin command handlers as one `PluginCommandContext` plus one returned `PluginCommandResult`, with no Discord responder bridge for defer, reply, follow-up, or edit-reply. The request is useful and source-reproducible, but it extends the public plugin SDK and Discord command-routing boundary, so it needs maintainer API direction rather than automatic cleanup.

## What This Changes

The issue requests Discord interaction-level response controls for native `api.registerCommand` plugin command handlers.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel capability with a Discord implementation, while preserving current returned-payload behavior and command auth/routing.

## Reproduction Assessment

Yes, at source level: current main still types plugin command handlers as `PluginCommandContext` plus one returned `PluginCommandResult`, and the Discord native command path only delivers that returned payload. I did not run live Discord because the missing public API is directly visible in the type and dispatch path.

## Solution Assessment

Unclear. A responder capability is the right class of solution, but maintainers should choose whether the permanent SDK surface is generic or Discord-specific before implementation.
