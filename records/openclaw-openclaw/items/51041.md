## Summary

Keep open. Current main still lacks a Discord responder surface on native plugin command context, and the requested mixed ephemeral/public slash-command flow requires an additive public Plugin SDK and Discord lifecycle design rather than cleanup closure.

## What This Changes

Expose Discord interaction response controls such as defer, reply, followUp, and editReply to `api.registerCommand` handlers.

## Best Possible Solution

Add a narrow typed Discord responder facade to native plugin command handlers after authorization and route resolution, while preserving the existing returned `ReplyPayload` path for backward compatibility.

## Reproduction Assessment

Yes, at source level: current main shows native plugin commands receive only `PluginCommandContext` and return one `ReplyPayload`, while Discord sends that payload after the handler with one ephemeral setting. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers approve the public API and lifecycle shape. The requested direction is valid, but the maintainable solution should expose a narrow typed responder facade rather than raw Discord interaction objects.
