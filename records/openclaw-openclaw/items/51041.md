## Summary

Keep open: current main and v2026.6.10 still route Discord plugin slash commands through the generic `PluginCommandContext` plus one returned `PluginCommandResult`, while Discord responder primitives exist only in internal/native or interactive-callback paths. The request is source-reproducible and aligned with plugin API evolution, but it changes the public SDK and Discord routing semantics, so it needs maintainer API direction rather than conservative cleanup.

## What This Changes

Requested exposing Discord slash interaction response controls such as defer, reply, follow-up, and edit-reply to OpenClaw plugin command handlers.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel capability, with a Discord implementation, while preserving current returned-payload behavior and command authorization/routing.

## Reproduction Assessment

Yes, at source level: current main still types plugin command handlers as `PluginCommandContext` plus one returned `PluginCommandResult`, and the Discord native command path only delivers that returned payload. I did not run live Discord because the missing public API is directly visible in the type and dispatch path.

## Solution Assessment

Unclear. A responder capability is the right class of solution, but maintainers should choose whether the permanent SDK surface is generic or Discord-specific before implementation.
