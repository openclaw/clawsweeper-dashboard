## Label Justifications

- `P2`: This is a normal-priority plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Plugin command authors cannot currently send a private initial Discord slash response plus a visible follow-up, so intended command output can be hidden from the conversation.

## Summary

Keep open: current main and the latest release still expose native plugin commands as `PluginCommandContext` plus a returned `PluginCommandResult`; Discord response primitives exist internally and for component interactive handlers, but they are not bridged into `api.registerCommand` handlers.

## What This Changes

The issue requests Discord slash-command response controls, such as defer, reply, follow-up, and edit, on plugin command handler context.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel-capability contract, with a Discord implementation, while preserving existing returned-payload behavior and command auth/routing.

## Reproduction Assessment

Yes, at source level: plugin command handlers receive `PluginCommandContext` and return one `PluginCommandResult`, while Discord responder methods are internal or interactive-handler-only. I did not run a live Discord interaction because the missing API is visible in the current type and dispatch path.

## Solution Assessment

Unclear. The requested Discord-specific responder shape is plausible, but maintainers should decide whether the durable SDK surface is Discord-specific or a generic command responder/channel capability.
