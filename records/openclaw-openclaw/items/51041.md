## Summary

Keep open. Current main still routes Discord native plugin commands through the generic command context and one returned reply payload, so the requested responder facade is not implemented and needs public SDK plus Discord interaction-lifecycle design.

## What This Changes

The issue requests Discord-specific interaction response controls for native plugin command handlers so plugins can defer, reply, follow up, and edit slash-command responses.

## Best Possible Solution

Add an additive typed Discord responder facade for native command handlers after authorization and route resolution, keep the existing `ReplyPayload` return behavior for compatibility, and cover guild plus group-DM mixed-visibility flows in SDK docs and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a Discord native plugin command receives only `PluginCommandContext`, returns one `ReplyPayload`, and the Discord dispatcher delivers that payload afterward with a single ephemeral setting.

## Solution Assessment

Unclear until maintainers approve the public API shape. The requested direction is valid, but the maintainable fix should expose a narrow typed responder facade rather than raw Discord interaction objects.
