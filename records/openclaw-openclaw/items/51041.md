## Summary

Keep open. Current main still has no Discord responder on native plugin command context: Discord slash commands defer up front and can deliver the single returned payload, but plugin handlers cannot control mixed ephemeral/public follow-ups or edit the original interaction reply themselves.

## What This Changes

Expose Discord interaction responder controls such as defer, reply, followUp, and editReply to native plugin command handlers.

## Best Possible Solution

Add a narrow typed Discord responder facade to native plugin command handlers after authorization and route resolution, while preserving the existing returned ReplyPayload path for backward compatibility.

## Reproduction Assessment

Yes, at source level. Current main constructs PluginCommandContext without Discord responder fields and delivers only the handler's returned payload after execution, so a plugin cannot interleave ephemeral defer, public follow-up, or edit calls.

## Solution Assessment

Unclear until maintainers approve the public API and lifecycle shape. The requested direction is valid, but the maintainable path should expose a small typed responder facade rather than raw Discord interaction access.
