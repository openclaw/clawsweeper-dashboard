## Summary

Keep open. Current main still gives native plugin commands only the generic command context plus one returned reply payload, while Discord owns the interaction response after the handler returns; the requested responder facade remains unimplemented and needs SDK/lifecycle design.

## What This Changes

Expose Discord slash interaction response controls such as defer, reply, followUp, and editReply to native plugin command handlers.

## Best Possible Solution

Add an additive typed Discord responder facade for native command handlers after authorization and route resolution, keep the existing ReplyPayload return path for compatibility, and cover guild plus group-DM mixed-visibility flows in SDK docs and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a Discord native plugin command receives only PluginCommandContext, returns one ReplyPayload, and the Discord dispatcher delivers that payload afterward with a single responseEphemeral setting.

## Solution Assessment

Unclear until maintainers approve the public API shape. The requested direction is valid, but the maintainable fix should expose a narrow typed responder facade rather than raw Discord interaction objects.
