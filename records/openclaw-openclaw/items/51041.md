## Summary

Keep open. Current main and the latest release still expose only the generic plugin command context to `api.registerCommand` handlers; Discord native command dispatch defers the slash interaction before plugin execution, then sends the handler result as a single `ReplyPayload` under the configured slash-command ephemeral mode. The existing Discord `respond` helper is available for interactive component handlers, not native plugin command handlers.

## What This Changes

The issue requests Discord-specific defer, reply, follow-up, and edit-reply controls for native plugin command handlers so plugins can mix ephemeral confirmations with visible Discord follow-ups.

## Best Possible Solution

Design an additive, typed Discord responder seam for native `api.registerCommand` handlers, injected only after OpenClaw authorization and route resolution. Model it on the existing Discord interactive `respond` helper, keep raw Discord objects private, support defer/reply/followUp/edit semantics with per-message ephemeral flags, preserve the existing `ReplyPayload` return path for old handlers, and document/test the Discord native command behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is enough: register a plugin command on Discord and inspect the handler context path, which currently receives generic `PluginCommandContext` only while the native slash dispatcher defers up front and later sends one `ReplyPayload` through `deliverDiscordInteractionReply`.

## Solution Assessment

Unclear until maintainers approve the API shape. The requested direction is valid, but the maintainable fix should be a narrow typed responder facade that preserves OpenClaw auth/routing semantics, not direct access to raw Discord interaction objects.
