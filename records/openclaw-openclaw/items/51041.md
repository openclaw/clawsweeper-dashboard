## Summary

Keep open. Current main still lacks Discord-specific interaction responder controls in plugin command handlers, and the requested public SDK/lifecycle change needs maintainer design rather than conservative cleanup closure.

## What This Changes

The issue asks OpenClaw to expose Discord slash-interaction defer, reply, follow-up, and edit-reply controls to native plugin command handlers while preserving existing authorization and routing behavior.

## Best Possible Solution

Add a narrow typed Discord responder facade to authorized native plugin command handlers, preserve the existing returned ReplyPayload path for compatibility, and prove mixed ephemeral/public follow-up behavior with focused tests plus live or redacted runtime evidence.

## Reproduction Assessment

Yes, at source level. Current main constructs PluginCommandContext without a Discord responder and delivers only the handler's returned ReplyPayload after execution, so plugin code cannot interleave defer, mixed-visibility follow-up, or edit-reply calls.

## Solution Assessment

Unclear until maintainers approve the public API and lifecycle shape. The requested direction is valid, but the maintainable solution should be a small typed Discord responder facade rather than raw Discord interaction access.
