## Label Justifications

- `P2`: This is a normal-priority plugin SDK and Discord command ergonomics improvement with clear source evidence but no urgent production regression.
- `impact:message-loss`: The issue is about command handlers being unable to choose visible follow-up delivery after an ephemeral slash response, which can suppress the public message the plugin wants to send.

## Summary

Keep open. Current main and the latest release still route Discord plugin slash commands through the generic `PluginCommandContext` plus one returned `ReplyPayload`, so the requested defer/reply/follow-up/edit controls are not available to `api.registerCommand` handlers.

## What This Changes

Expose Discord slash interaction response controls to plugin command handlers while preserving existing command authorization and routing behavior.

## Best Possible Solution

Add a narrow, documented plugin-command responder contract that lets Discord native command handlers defer, send private/public replies, follow up, and edit the original response while keeping the current returned-payload path backward compatible.

## Reproduction Assessment

Yes, by source inspection: `PluginCommandContext` and `executePluginCommand` do not expose any Discord interaction responder, while Discord native commands only deliver the returned `ReplyPayload`. I did not run a live Discord repro because this read-only review can prove the API gap from current source and shipped tag inspection.

## Solution Assessment

Unclear. The requested `ctx.discord.respond` shape is plausible, but the best permanent fix needs maintainer API design so OpenClaw does not accidentally create an ad hoc Discord-only SDK surface where a generic command responder or channel capability contract would age better.
