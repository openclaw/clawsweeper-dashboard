## Label Justifications

- `P2`: This is a normal-priority plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Discord plugin command handlers cannot currently send a private initial slash response plus a visible follow-up, so intended command output can be suppressed from the conversation.

## Summary

Keep open. Current main and the latest release still expose plugin commands as a generic `PluginCommandContext` plus one returned `ReplyPayload`; Discord has internal defer/reply/follow-up/edit primitives, but there is no plugin-command SDK bridge for them.

## What This Changes

The issue requests Discord slash-command interaction responder controls for native `api.registerCommand` handlers, including defer, private/public replies, follow-ups, and edit-reply support.

## Best Possible Solution

Design a narrow, documented plugin-command responder contract, likely reusing the existing Discord interactive-handler responder shape where appropriate, while preserving the current returned-`ReplyPayload` path for existing plugins.

## Reproduction Assessment

Yes. Source inspection on current main shows `PluginCommandContext` lacks a Discord responder and `executePluginCommand` only passes the generic context to handlers and returns a single payload.

## Solution Assessment

Unclear. The requested `ctx.discord.respond` shape is plausible, but the best permanent fix needs maintainer API design so OpenClaw does not create a one-off Discord SDK surface where a generic command responder or channel capability contract would be better.
