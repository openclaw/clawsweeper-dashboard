## Label Justifications

- `P2`: This is a normal-priority plugin SDK and Discord command capability gap with clear source evidence but no urgent production regression.
- `impact:message-loss`: Plugin authors cannot send both private slash-command acknowledgements and visible Discord follow-ups from a native command handler, so intended command output can be hidden from the conversation.

## Summary

Keep open: current main and the latest release still do not expose Discord interaction responder controls to `api.registerCommand` handlers. The issue remains the canonical tracking item, but the API shape needs maintainer direction because it extends the public plugin SDK and Discord routing boundary.

## What This Changes

The issue requests a plugin command API for Discord slash interactions that can defer, reply, follow up, and edit replies with per-response ephemeral control.

## Best Possible Solution

Add a narrow documented plugin-command responder or channel-capability contract, with a Discord implementation, while preserving current returned-payload behavior and command auth/routing.

## Reproduction Assessment

Yes, at source level: current main still types plugin command handlers as `PluginCommandContext` plus one returned `PluginCommandResult`, with no Discord or generic responder capability on the context. I did not run a live Discord interaction because the missing API is visible in the current type and dispatch path.

## Solution Assessment

Unclear. A responder is the right kind of capability, but maintainers should choose whether the durable SDK surface is generic or Discord-specific before implementation.
