## Summary

Keep open: current main and the latest release still resolve allowed elevated sessions with no explicit default to `on`, then route absent-per-call exec through elevated gateway execution. The behavior is now documented and tested, so cleanup closure would be wrong; changing it needs an explicit maintainer security/product decision.

## What This Changes

The issue reports that enabling elevated exec changes non-elevated exec routing from sandbox to gateway and asks for sandbox-first, opt-in elevated exec behavior.

## Best Possible Solution

Maintainers should decide the elevated exec security contract explicitly; if sandbox-first is desired, change the implicit default to `off`, define whether `ask` remains an alias or becomes opt-in, and update source, tests, docs, and changelog together.

## Reproduction Assessment

Yes, source-level. With elevated allowed and no directive/session/config default, current main resolves the level to `on`; an omitted per-call `elevated` value then uses that default and `resolveExecTarget` selects `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off direction is mechanically narrow, but current shipped docs and tests intentionally encode default-on and `ask`-as-`on`, so the maintainable path starts with a security/UX decision.
