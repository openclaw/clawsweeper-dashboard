## Label Justifications

- `P1`: The issue affects exec routing across the sandbox/gateway boundary for users who enable elevated execution.
- `impact:security`: The report is about whether enabling elevated exec silently changes sandbox isolation defaults for command execution.

## Summary

Current main still routes omitted-flag exec calls through the elevated session default when elevated access is allowed, and that default remains gateway-oriented unless explicitly configured off; this is a real sandbox-boundary product/security decision, so the issue should stay open rather than be swept as implemented or stale.

## What This Changes

The issue requests sandbox-first exec routing when `tools.elevated.enabled` is on, with gateway elevation only for explicit elevated calls or explicit session/user intent.

## Best Possible Solution

Maintainers should explicitly choose the elevated exec contract, then align default resolution, `/elevated ask` semantics, system prompt text, docs, tests, and changelog around that decision.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.20` shows allowed elevated sessions default to `on`, omitted `exec.elevated` uses that session default, and elevated requests resolve to `gateway`.

## Solution Assessment

Unclear as an automated fix. The narrow default-off direction matches the reported sandbox-first expectation, but current docs and tests encode default-on and `/elevated ask` as host execution, so maintainers need to choose the security and compatibility contract first.
