## Summary

Keep open: current main and latest release still default allowed elevated sessions with no explicit elevated default to `on`, and active elevated exec still routes to the gateway host. The focused fix in https://github.com/openclaw/openclaw/pull/46908 was closed unmerged, and I did not find a merged or viable open canonical replacement for this behavior.

## What This Changes

Issue reports that enabling elevated exec changes ordinary sandboxed exec routing to gateway-host execution and lacks a sandbox-default, per-call elevation mode.

## Best Possible Solution

Keep the issue open, choose the safe elevated-default policy explicitly, add focused regression coverage for sandbox-default routing, and handle any new `/elevated ask` semantics as a maintainer-approved behavior change.

## Reproduction Assessment

Yes, for source-level reproduction: current main resolves allowed elevated sessions with no explicit default to `on`, then exec treats omitted `elevated` as that default and routes elevated requests to `gateway`. I did not run the reported Docker/Discord scenario in this read-only review.

## Solution Assessment

No, the current implementation is not the best safe-default behavior. The narrow fallback fix is maintainable, while changing `/elevated ask` into per-call opt-in needs maintainer product/security approval because current docs define it as an alias for `on`.
