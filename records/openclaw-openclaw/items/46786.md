## Summary

Keep open. Current main still matches the reported elevated exec routing chain, and the default-off PR that referenced this issue was closed unmerged, so this remains a security-sensitive maintainer contract decision rather than cleanup-close material.

## What This Changes

The issue asks for elevated mode to leave sandbox exec as the default unless an exec call explicitly requests elevation, and to avoid defaulting new allowed sessions to host exec.

## Best Possible Solution

Maintainers should explicitly choose the elevated exec contract. If sandbox-default opt-in elevation is desired, update default resolution, directive persistence/status, exec inheritance, prompt wording, docs, and regression tests together; if current behavior is intentional, keep the code but make the exec/security docs state the tradeoff more prominently.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a sandboxed allowed elevated session with no explicit default resolves to `on`, omitted `exec.elevated` inherits that active state, and elevated `auto` routing resolves to `gateway`.

## Solution Assessment

Unclear. The requested default-off or ask-as-opt-in behavior is plausibly safer, but it conflicts with current documented and tested behavior; a maintainer should approve the security contract before code changes.
