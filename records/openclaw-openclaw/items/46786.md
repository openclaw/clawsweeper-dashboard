Keep open: current main still has the reported source-level routing chain, and the behavior is now documented/tested as the elevated exec contract. Changing it crosses the sandbox/gateway security boundary and needs an explicit maintainer product/security decision, not cleanup closure.

## What This Changes

The issue reports that enabling elevated exec causes ordinary sandboxed `exec` calls to route to the gateway and asks for sandbox-by-default, opt-in elevation semantics.

## Best Possible Solution

Maintainers should explicitly decide the elevated exec contract, then align default resolution, exec inheritance, prompt/status wording, docs, and regression coverage around the chosen sandbox-default or host-default behavior.

Yes. Source inspection on current main gives a high-confidence path: enable elevated tools for an allowed sender, start a session without an explicit elevated directive, call `exec` without `params.elevated`, and the inherited elevated state routes through gateway.

Unclear. The reporter's requested sandbox-default behavior is plausibly safer, but it conflicts with current docs and tests, so the narrow maintainable fix needs maintainer approval before implementation.
