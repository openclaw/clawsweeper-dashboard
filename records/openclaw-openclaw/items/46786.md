## Label Justifications

- `P1`: Sandboxed exec can still route to the gateway host when elevated access is enabled and allowed, affecting a core agent workflow.
- `impact:security`: The issue concerns shell execution crossing the sandbox isolation boundary through elevated defaults.

## Summary

Keep open: current main and v2026.6.6 still default allowed elevated sessions with no explicit elevated default to `on`, and active elevated exec still routes to the gateway host. The direct fixes in https://github.com/openclaw/openclaw/pull/46908 and https://github.com/openclaw/openclaw/pull/19595 were closed unmerged, so the remaining safe-default and `/elevated ask` policy needs maintainer security/product review.

## What This Changes

The issue asks for elevated exec enablement to keep sandbox exec as the default unless elevation is explicitly requested.

## Best Possible Solution

Maintainers should choose the elevated policy explicitly, likely preserving explicit `agents.defaults.elevatedDefault` while making unset allowed sessions safe by default, then add focused routing regression tests and update elevated docs to match the chosen `/elevated ask` semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main resolves allowed elevated sessions with no explicit default to `on`, then exec treats omitted `elevated` as that default and routes elevated requests to `gateway`.

## Solution Assessment

No. The current implementation is not the best safe-default behavior; the narrow fallback fix is maintainable, but changing `/elevated ask` semantics or model-controllable elevation needs maintainer security/product approval.
