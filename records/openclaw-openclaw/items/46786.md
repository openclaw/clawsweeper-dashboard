## Summary

Keep this open: current main still defaults an allowed elevated session with no explicit level to active elevated routing, and active elevated routing still sends implicit sandboxed exec to the gateway host. The direct default-off fix PR is closed unmerged, while the newer exec policy PR only partially overlaps and is not a safe canonical replacement.

## What This Changes

The issue reports that enabling elevated exec changes implicit exec routing from sandbox to gateway instead of keeping elevation opt-in per call.

## Best Possible Solution

Make a maintainer-owned security/product decision, then align the elevated default, /elevated ask semantics, per-call elevation behavior, system prompt text, docs, and regression tests in one focused change.

## Reproduction Assessment

Yes. Source inspection on current main shows allowed elevated sessions fall back to "on", "on"/"ask" become active elevated mode, and active elevated mode resolves implicit exec to gateway instead of sandbox; I did not run the full Docker plus Discord scenario.

## Solution Assessment

Unclear pending maintainer decision. Defaulting the fallback to off is the narrow fix for the silent sandbox bypass, but redefining /elevated ask conflicts with current docs and should be decided with the exec policy-boundary work.
