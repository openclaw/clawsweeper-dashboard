## Summary

Keep this open: current main still has source-level evidence that an allowed elevated session without an explicit elevated default resolves to elevated host routing, and the latest release contains the same fallback. The linked fix PR was closed unmerged, and the requested `/elevated ask` behavior conflicts with current docs, so this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue reports that enabling elevated exec makes non-elevated exec calls run on the gateway instead of the sandbox and requests sandbox-by-default, opt-in elevation behavior.

## Best Possible Solution

Maintainers should decide the elevated exec security contract, then align default resolution, per-call elevation routing, status/system-prompt text, docs, and regression tests in one coherent fix.

## Reproduction Assessment

Yes, by source inspection: current main resolves an allowed elevated session with no explicit elevated level to `on`, and the exec tool treats `on`/`ask` as elevated host routing. I did not run the Docker/Discord repro path in this read-only review.

## Solution Assessment

No, not as a close: the narrow default-off repair is clear, but changing `/elevated ask` into sandbox-by-default opt-in behavior conflicts with current docs. The best fix needs a maintainer decision and coordinated code, docs, and tests.
