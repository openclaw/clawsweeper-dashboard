## Label Justifications

- `P1`: Sandboxed exec can still route to the gateway host when elevated access is enabled and allowed, affecting a core agent workflow.
- `impact:security`: The issue concerns shell execution crossing the sandbox isolation boundary through elevated defaults.

## Summary

Keep open: current main and the latest release still resolve an allowed elevated session with no explicit `elevatedDefault` to `on`, and exec maps that active elevated level to host execution. The direct fix PRs are closed unmerged, while the open exec schema PR addresses a neighboring model-policy boundary rather than the safe-default decision here.

## What This Changes

The issue asks for elevated exec to remain sandbox-by-default unless elevation is explicitly requested, and for unset `elevatedDefault` or `/elevated ask` behavior not to route all exec calls to the gateway host.

## Best Possible Solution

Maintainers should choose the elevated default and `/elevated ask` policy explicitly, then align the reply fallback, directive persistence, exec routing coverage, and elevated docs so sandboxed exec stays sandboxed unless elevation is intentionally requested.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main resolves an allowed elevated session with no explicit default to `on`, then exec treats ordinary omitted-flag calls as elevated and routes them to `gateway`.

## Solution Assessment

Partly. A narrow fallback-to-off fix is plausible for the safe-default bug, but changing `/elevated ask` semantics and upgrade behavior needs maintainer security/product approval first.
