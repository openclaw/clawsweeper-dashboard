## Summary

Keep this open: current main and the latest release still resolve an allowed elevated session with no explicit elevated default to active elevated routing, and active elevation still routes implicit exec to the gateway host. The earlier default-off fix PR was closed unmerged, while the newer exec policy PR is adjacent rather than a complete replacement for this issue.

## What This Changes

The issue asks for enabling `tools.elevated` to preserve sandbox exec by default and make host elevation opt-in or explicitly configured.

## Best Possible Solution

Make a maintainer-owned security/product decision that aligns the implicit elevated default, `/elevated ask`, per-call elevation policy, system prompt text, docs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: current main resolves an allowed elevated session with no explicit level to `on`, and `on`/`ask` become active elevated host routing. I did not run the full Docker plus Discord scenario in this read-only review.

## Solution Assessment

Unclear pending maintainer decision: defaulting the fallback to `off` is the narrow fix for the silent sandbox bypass, but redefining `/elevated ask` conflicts with current docs and should be decided with the exec policy-boundary work.
