## Label Justifications

- `P1`: Allowed sandboxed sessions can consistently execute ordinary commands on the gateway host without an explicit directive or per-call elevation request.
- `impact:security`: The implicit default crosses the command-execution isolation boundary from sandbox to gateway.

## Summary

Current main and v2026.6.5 still default an allowed elevated session to `on` when no directive, saved session level, or agent default exists, causing ordinary exec calls to route from sandbox to gateway. The narrow off-by-default repair remains unmerged; redefining `ask` as sandbox-first would conflict with its documented session-wide semantics and requires a separate product decision.

## What This Changes

Make elevated capability sandbox-safe by default while preserving explicit session-level and per-call host elevation.

## Best Possible Solution

Default only the no-override elevated session state to `off`, preserve explicit `agents.defaults.elevatedDefault`, saved `/elevated` session state, and per-call `elevated` behavior, then add fresh-session and explicit-override regression tests and update the documented default.

## Reproduction Assessment

Yes. Current main deterministically resolves an allowed fresh session with no explicit elevation setting to `on`, and downstream exec routing sends calls without `elevated` to the gateway.

## Solution Assessment

Partly. Changing the terminal fallback to `off` is the narrowest fix for the silent default, but redefining `ask` as per-call-only elevation is not the best fix because it conflicts with the current documented session-mode contract.
