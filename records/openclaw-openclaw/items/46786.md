## Summary

Keep open. Current main still matches the reported routing chain: allowed elevated sessions without an inline, session, or configured elevated default fall back to `on`; omitted `exec.elevated` calls inherit that active level; and elevated `auto` routing resolves to `gateway`. The linked default-off PR was closed unmerged, so this remains a security-sensitive behavior decision rather than cleanup-close material.

## What This Changes

The issue reports that enabling `tools.elevated.enabled` causes sandboxed `exec` calls to route to the gateway host by default, even when individual calls do not pass `elevated: true`.

## Best Possible Solution

Maintainers should explicitly choose the elevated-mode security contract. If sandbox-default opt-in elevation is desired, update default resolution, directive persistence/status, exec inheritance, prompt wording, docs, and regression tests together; if default-on host execution is intentional, keep the implementation but make the exec docs and security guidance state that tradeoff clearly.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a sandboxed, allowed elevated session with no explicit elevated default resolves to `on`, an omitted `exec.elevated` call inherits that state, and elevated `auto` routing resolves to `gateway`.

## Solution Assessment

Unclear. The requested default-off or ask-as-opt-in behavior is plausibly safer, but it conflicts with current documented and tested defaults; a maintainer should approve the contract before code changes.
