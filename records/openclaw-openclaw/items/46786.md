## Summary

Keep open: current main still defaults allowed elevated sessions to `on` and routes active elevated exec to the gateway, while current docs/tests now describe that behavior, so this needs maintainer/security product judgment rather than cleanup closure.

## What This Changes

The issue asks for elevated exec to be sandbox-first by default, with gateway execution only when explicitly requested per call or session.

## Best Possible Solution

Maintainers should explicitly decide the elevated exec contract; if sandbox-first is desired, change the implicit elevated default to `off`, preserve explicit `exec(elevated: true)`, and update docs/tests together.

## Reproduction Assessment

Yes, source-level. With elevated allowed and no directive/session/config default, current main resolves the level to `on`, then the exec path treats the missing per-call flag as active elevated mode and resolves the host to `gateway`.

## Solution Assessment

No, not as an automated cleanup fix. The one-line default-off change is mechanically clear, but current docs/tests intentionally describe default-on and `ask`-as-`on`, so changing it is a security/UX policy decision.
