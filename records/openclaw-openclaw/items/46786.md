## Label Justifications

- `P1`: The issue affects exec routing across the sandbox/gateway boundary for users who enable elevated execution.
- `impact:security`: The report is about whether enabling elevated exec silently changes sandbox isolation defaults for command execution.

## Summary

Keep open: current main and the latest release still codify allowed elevated sessions defaulting to host-oriented exec, so the report is neither stale nor implemented; changing it needs an explicit maintainer product/security decision.

## What This Changes

The issue asks for enabling `tools.elevated` without changing non-elevated exec routing from sandbox to gateway, and for elevation to be opt-in per exec call.

## Best Possible Solution

Maintainers should choose the elevated exec contract, then update default resolution, `/elevated ask` semantics, system prompt text, docs, tests, and changelog together.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main and `v2026.5.20` show allowed elevated sessions default to `on`, omitted `exec.elevated` uses that default, and elevated requests resolve to `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off direction matches the sandbox-first expectation, but current docs and tests encode host execution while elevated is active, so maintainers need to choose the security and compatibility contract first.
