## Label Justifications

- `P1`: The issue concerns a sandbox-to-gateway exec routing path that can affect users who enable elevated access.
- `impact:security`: The affected behavior controls whether agent shell execution stays in the sandbox or crosses to the gateway host.

## Summary

Keep this open: current main and latest release still default allowed elevated sessions to active elevated routing, and active elevated exec routes to the gateway host instead of the sandbox. The remaining question is product/security policy, not lack of evidence.

## What This Changes

The issue reports that enabling elevated exec makes implicit exec calls leave the sandbox and asks for sandbox-by-default elevation with opt-in host exec.

## Best Possible Solution

Make a maintainer-owned security/product decision for elevated defaults, then align runtime fallback, `/elevated ask`, model-facing/tool args, system prompt text, docs, and regression tests to that contract.

## Reproduction Assessment

Yes. Source inspection shows an allowed elevated session with no explicit level defaults to `on`, and omitted per-call `elevated` inherits that active mode before the host resolver maps it to `gateway`.

## Solution Assessment

Unclear pending maintainer decision. Defaulting the fallback to `off` is the narrowest way to stop silent elevation, but changing `/elevated ask` into a per-call-only mode would alter the current shipped contract.
