## Label Justifications

- `P1`: The issue affects exec routing across the sandbox and gateway boundary for users who enable elevated exec.
- `impact:security`: The issue is about whether elevated exec silently changes sandbox isolation defaults.

## Summary

Keep open: current main and the latest release still default allowed elevated sessions with no explicit elevated default to `on`, so omitted-flag `exec` calls become elevated and resolve to the gateway. The earlier default-off fix PR was closed unmerged, while current docs and tests now encode parts of the default-on and `/elevated ask` behavior, so this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue asks for elevated exec to preserve sandbox routing for non-elevated `exec` calls and make elevation opt-in or default-off when `tools.elevated.enabled` is enabled.

## Best Possible Solution

Maintainers should explicitly choose the elevated exec security contract; if sandbox-first is desired, update default resolution, `/elevated ask` semantics, system prompt text, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.20` shows allowed elevated sessions default to `on`, omitted-flag `exec` uses that default, and elevated requests resolve to `gateway`.

## Solution Assessment

Unclear as an automated fix. The closed default-off PR is mechanically narrow, but current docs and tests encode default-on and `/elevated ask` as host execution, so maintainers need to choose the security and compatibility contract first.
