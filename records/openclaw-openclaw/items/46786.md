## Summary

Keep open: current main still resolves allowed elevated sessions with no explicit elevated default to `on`, and omitted per-call `elevated` then routes exec through elevated gateway execution. That behavior is now documented and covered by tests, so changing it needs an explicit maintainer security/product decision rather than cleanup closure.

## What This Changes

The issue requests sandbox-first exec behavior when `tools.elevated` is enabled, with gateway execution used only when elevation is explicitly requested.

## Best Possible Solution

Maintainers should decide the elevated exec security contract explicitly; if sandbox-first is desired, update the default, `ask` semantics, source, tests, docs, and changelog together.

## Reproduction Assessment

Yes, source-level. With elevated allowed and no directive/session/config default, current main resolves the level to `on`; an omitted per-call `elevated` value then uses that default and `resolveExecTarget` selects `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off direction is mechanically narrow, but current shipped docs and tests intentionally encode default-on and `ask`-as-`on`, so the maintainable path starts with a security/UX decision.
