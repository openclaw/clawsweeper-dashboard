## Label Justifications

- `P1`: This affects elevated exec routing across the sandbox and gateway boundary for users who enable the feature.
- `impact:security`: The issue is about whether enabling elevated exec silently changes sandbox isolation defaults.

## Summary

Keep open: current main and the latest release still encode allowed elevated sessions with no explicit `elevatedDefault` as active host execution, and changing that now is a sandbox-boundary product/security decision rather than conservative cleanup.

## What This Changes

The issue asks for elevated exec to stay sandbox-first unless a tool call explicitly requests elevation, instead of routing omitted-elevated exec calls to the gateway when elevated mode is enabled or set to `ask`.

## Best Possible Solution

Maintainers should explicitly decide the elevated exec security contract; if sandbox-first is desired, update default resolution, `/elevated ask` semantics, tests, docs, and changelog together.

## Reproduction Assessment

Yes, source-level. With elevated allowed and no directive/session/config default, current main resolves the level to `on`; an omitted per-call `elevated` value then uses that default and `resolveExecTarget` selects `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off direction is mechanically narrow, but current shipped docs and tests intentionally encode default-on and `ask` as host execution with approvals, so the maintainable path starts with a security/UX decision.
