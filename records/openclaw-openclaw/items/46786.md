## Label Justifications

- `P1`: This affects exec routing across the sandbox and gateway boundary for users who enable elevated exec.
- `impact:security`: The issue is about whether elevated exec silently changes sandbox isolation defaults.

## Summary

Keep open: current main and the latest release still resolve allowed elevated sessions with no explicit default to active host execution, so omitted-elevated `exec` calls route as elevated. The related default-off PR was closed unmerged, and changing this now is a sandbox-boundary product/security decision because current docs and tests also encode the shipped default-on contract.

## What This Changes

The issue asks for elevated exec to stay sandbox-first by default, with gateway execution only when explicitly requested per call or session.

## Best Possible Solution

Maintainers should explicitly decide the elevated exec security contract; if sandbox-first is desired, update default resolution, `/elevated ask` semantics, system prompt text, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Source inspection shows elevated-allowed sessions without an explicit elevated level resolve to `on`; omitted `exec` `elevated` then uses that default and `resolveExecTarget` selects gateway.

## Solution Assessment

Unclear as an automated fix. Defaulting to `off` is mechanically narrow and aligns with per-call opt-in expectations, but current shipped docs and tests encode default-on and `ask` as a host-exec alias, so maintainers need to choose the contract first.
