## Summary

Keep open: current main and v2026.5.19 still default allowed elevated sessions with no explicit elevated default to `on`; omitted-flag `exec` then becomes elevated and routes to `gateway`. The related fix PR at https://github.com/openclaw/openclaw/pull/46908 was closed unmerged, and current docs/tests now encode the default-on and `/elevated ask` alias contract, so resolving this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue requests that enabling elevated exec not change sandbox-default exec routing, with host elevation used only through explicit session/default choice or per-call `exec(elevated: true)`.

## Best Possible Solution

Maintainers should explicitly choose the elevated exec security contract; if sandbox-first is desired, update default resolution, `/elevated ask` semantics, system prompt text, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Source inspection on current main shows elevated-allowed sessions without an explicit level resolve to `on`; omitted-flag `exec` uses that default and `resolveExecTarget` selects `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off change in https://github.com/openclaw/openclaw/pull/46908 is mechanically narrow, but current docs and tests now encode default-on and `/elevated ask` as host execution, so maintainers need to choose the contract first.
