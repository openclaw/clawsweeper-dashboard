## Label Justifications

- `P1`: The issue affects command execution across the sandbox/gateway boundary for users who enable elevated execution.
- `impact:security`: The report is about sandbox isolation and whether enabling elevated exec silently changes where shell commands run.

## Summary

Current main and the latest release still preserve the reported elevated exec contract problem: allowed elevated sessions fall back to an active `on` level, omitted `exec.elevated` inherits that session default, and elevated exec resolves outside the sandbox. This should stay open for maintainer product/security review because changing the default or redefining `/elevated ask` affects sandbox isolation and upgrade compatibility.

## What This Changes

The issue requests that enabling `tools.elevated` preserve sandbox exec by default and make gateway elevation opt-in per exec call or explicit session choice.

## Best Possible Solution

Maintainers should explicitly choose the elevated exec contract, then align default resolution, `/elevated ask` semantics, system prompt wording, docs, regression tests, and changelog around that decision.

## Reproduction Assessment

Yes, at source level. Current main resolves allowed elevated sessions with no explicit default to `on`, omitted `exec.elevated` inherits that default, and elevated requests resolve to `gateway` rather than sandbox.

## Solution Assessment

Unclear as an automated fix. The default-off direction matches the reported sandbox-first expectation, but current docs and tests encode default-on and `/elevated ask` as host execution, so maintainers need to choose the compatibility and security contract first.
