## Label Justifications

- `P1`: This affects elevated exec routing across sandbox and gateway boundaries for users who enable the feature.
- `impact:security`: The issue is about whether enabling elevated exec silently changes sandbox isolation defaults.

## Summary

Keep open: current main and the latest shipped release still resolve allowed elevated sessions with no explicit `elevatedDefault` to `on`, and omitted per-call `elevated` then routes exec through elevated host execution. That behavior is now documented and covered by tests, so changing it is a security/product decision rather than conservative cleanup.

## What This Changes

The issue requests sandbox-first exec routing when `tools.elevated` is enabled, with host elevation only when explicitly requested per call.

## Best Possible Solution

Maintainers should explicitly decide the elevated exec security contract; if sandbox-first is desired, update the default, `ask` semantics, source, tests, docs, and changelog together.

## Reproduction Assessment

Yes, source-level. With elevated allowed and no directive/session/config default, current main resolves the level to `on`; an omitted per-call `elevated` value then uses that default and `resolveExecTarget` selects `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off direction is mechanically narrow, but current shipped docs and tests intentionally encode default-on and `ask` as host execution with approvals, so the maintainable path starts with a security/UX decision.
