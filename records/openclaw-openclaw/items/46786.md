## Summary

Keep this open: current main and the latest release still resolve an allowed elevated session with no explicit elevated default to active elevated routing, while `/elevated ask` is documented as host execution with approvals. The linked default-off PR was closed unmerged, so this remains a maintainer security/product decision rather than cleanup closure.

## What This Changes

The issue asks for elevated exec to remain sandbox-first by default, with gateway routing only when elevation is explicitly selected or requested per call.

## Best Possible Solution

Keep the issue open for a maintainer-owned security/product decision that aligns default resolution, per-call elevation, status/system-prompt text, docs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: an allowed elevated session with no explicit elevated level resolves to `on`, and the exec tool treats `on`/`ask` as elevated host routing. I did not run the full Docker plus Discord scenario in this read-only review.

## Solution Assessment

Unclear pending maintainer decision: defaulting the fallback to `off` addresses the silent sandbox bypass, but redefining `/elevated ask` as sandbox-by-default per-call opt-in conflicts with current docs and needs an explicit product/security call.
