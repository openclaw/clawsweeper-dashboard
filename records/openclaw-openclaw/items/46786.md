## Summary

Keep open: current main and the latest release still resolve allowed elevated sessions without an explicit default to `on`, and active elevated exec still routes outside the sandbox. Because current docs and tests now describe that contract, changing it needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue asks for elevated exec to stay sandbox-first by default and route to host execution only when explicitly requested per call or session.

## Best Possible Solution

Maintainers should explicitly decide the elevated exec contract; if sandbox-first is desired, change the implicit default to `off`, define whether `ask` remains an alias or becomes opt-in, and update tests, docs, and changelog together.

## Reproduction Assessment

Yes, source-level. With elevated allowed and no directive/session/config default, current main resolves the level to `on`; missing per-call `elevated` then uses that default and `resolveExecTarget` selects `gateway`.

## Solution Assessment

Unclear as an automated fix. The default-off direction is mechanically narrow, but current shipped docs and tests intentionally encode default-on and `ask`-as-`on`, so the maintainable path starts with an explicit security/UX decision.
