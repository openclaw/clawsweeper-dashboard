## Label Justifications

- `P1`: Current main still contains a default that can route allowed elevated sessions to gateway-host exec instead of sandbox exec.
- `impact:security`: The issue is about command execution crossing from the sandbox boundary to the gateway host.

## Summary

Keep open: current main and the latest release still default allowed elevated sessions to `on` when no explicit `agents.defaults.elevatedDefault` is set, so the sandbox-default routing regression is not implemented on main.

## What This Changes

The issue reports that enabling elevated exec makes ordinary exec calls route to the gateway host instead of staying sandboxed unless elevation is explicitly requested or configured.

## Best Possible Solution

Change the implicit allowed elevated default to `off`, add regression coverage for the no-directive/no-session/no-config path, and leave `/elevated ask` semantics unchanged unless maintainers explicitly approve a separate behavior change.

## Reproduction Assessment

Yes. Source inspection shows allowed elevated sessions still fall back to `on`, and the exec target resolver routes elevated `auto` execution to `gateway` instead of sandbox.

## Solution Assessment

No, the current implementation is not the best end state for secure defaults. The narrow fallback fix is clear, but changing `/elevated ask` semantics should be a separate maintainer product decision because current docs define it as an alias for `on`.
