## Summary

Keep open: current main still queues accepted Discord messages before resolving current-message attachments, so CDN URLs can be fetched only after queue wait time and failures fall back into media context instead of a durable local file.

## What This Changes

The issue reports that queued Discord messages can lose usable Discord image attachments when CDN URLs expire before the agent processes the queued message.

## Best Possible Solution

Persist current-message Discord attachments before queue wait time or carry pre-resolved durable media facts through the queued job, while preserving existing SSRF policy, size/timeouts, forwarded/referenced media behavior, and history-media behavior.

## Reproduction Assessment

Yes, from source with medium confidence: an accepted Discord message can be queued in `message-handler.ts`, while attachment resolution is deferred until `message-handler.process.ts` runs later. I did not run a live Discord CDN-expiry repro.

## Solution Assessment

Yes, the best fix is a narrow Discord-plugin repair that stores durable media before queue wait time or carries pre-resolved media through the inbound job. It should not add a config option or change shared channel contracts unless a small Discord-local payload field is needed.
