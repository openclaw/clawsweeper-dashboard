## Label Justifications

- `P1`: Queued Discord messages can silently lose attachment media before the agent turn, breaking an active channel workflow.
- `impact:data-loss`: The attachment bytes may never be saved locally after the Discord CDN URL expires during queue delay.
- `impact:message-loss`: The agent can receive the inbound Discord message without the media content the user sent.
- `maturity:stable`: The issue affects the Discord channel surface, which qa/maturity-scores.yaml scores as M4 Stable.

## Summary

Keep open: current main and the latest release still resolve Discord attachment media after the inbound run queue, and the linked closing PR is open but unmerged.

## What This Changes

Discord queued attachment messages should preserve inbound media by downloading expiring CDN URLs at receipt/preflight time instead of during later queued processing.

## Best Possible Solution

Review and land the linked receipt-time Discord media resolution PR, or an equivalent focused Discord-plugin fix, after real Discord proof shows queued attachments survive the delay.

## Reproduction Assessment

Yes at source level: current main preflights and enqueues the Discord job, then resolves direct and forwarded media later inside processDiscordMessageInner. I did not run a live Discord delayed-queue CDN-expiry repro in this read-only review.

## Solution Assessment

Yes: resolving and persisting Discord media during authorized receipt-time preflight, then reusing those media facts during queued processing, is the narrow owner-boundary fix. The linked PR appears to target that shape but still needs review and real behavior proof before merge.
