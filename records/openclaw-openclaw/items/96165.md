## Summary

Keep open: current main still resolves normal Discord attachment media after the queued job starts, and the linked closing PR remains open and unmerged.

## What This Changes

The issue reports that Discord attachments sent while OpenClaw is busy can sit in the inbound run queue until Discord CDN URLs expire, leaving the agent without the media the user sent.

## Best Possible Solution

Review and land the linked receipt-time Discord media resolution PR, or an equivalent focused Discord-plugin fix, after proof shows queued direct and forwarded attachments survive the delay.

## Reproduction Assessment

Yes at source level: current main enqueues a preflight context and resolves direct and forwarded media later inside processDiscordMessageInner. I did not run a live Discord delayed-queue CDN-expiry repro in this read-only review.

## Solution Assessment

Yes: resolving and persisting Discord media during receipt-time preflight, then reusing those local media facts during queued processing, is the narrow owner-boundary fix; the open PR implements that shape but still needs live proof.
