- `impact:data-loss`: The report is about attachment bytes that may never be saved locally after the Discord CDN URL expires.

## Summary

Keep open: current main still resolves Discord attachment media after the inbound run queue, and the linked fix PR is open but unmerged. The issue remains necessary until https://github.com/openclaw/openclaw/pull/96183 or an equivalent fix lands.

## What This Changes

The issue reports that queued Discord messages can lose attachments because signed CDN URLs are fetched after the run queue delay instead of at message receipt time.

## Best Possible Solution

Review and land the linked receipt-time Discord media resolution PR, or an equivalent focused fix, after live proof shows queued attachments survive the delay.

## Reproduction Assessment

Yes, at source level: current main preflights the message before queue insertion, then downloads direct and forwarded media later inside `processDiscordMessageInner`. I did not run a live Discord CDN-expiry reproduction in this read-only pass.

## Solution Assessment

Yes: resolving and persisting Discord media during authorized receipt-time preflight, then reusing those local media facts during queued processing, is the narrow owner-boundary fix.
