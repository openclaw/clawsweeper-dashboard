## Summary

Keep open: current main and v2026.6.10 still resolve Discord attachment media after the inbound preflight context is queued, and the linked candidate fix remains open and unmerged.

## What This Changes

The issue reports that queued Discord messages can lose attachments because Discord CDN URLs may expire before OpenClaw downloads them during processing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96183, or an equivalent receipt-time Discord media resolution fix, after live proof shows queued attachments survive the delay.

## Reproduction Assessment

Yes, at source level: current main preflights and queues the context, then later resolves direct and forwarded attachment media inside processDiscordMessageInner. I did not run a live Discord delayed-queue CDN-expiry repro in this read-only review.

## Solution Assessment

Yes: resolving and persisting Discord media during receipt-time preflight, then reusing those local media facts during queued processing, is the narrow owner-boundary fix. The open linked PR proposes that shape but is not merged yet.
