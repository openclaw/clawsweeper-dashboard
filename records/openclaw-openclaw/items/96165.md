## Label Justifications

- `P1`: Queued Discord messages can silently lose attachment media before the agent turn, breaking an active channel workflow.
- `impact:data-loss`: The attachment bytes may never be saved locally when the delayed fetch fails.
- `impact:message-loss`: The inbound Discord message can reach the agent without the media content the user sent.

## Summary

Keep open: current main and the latest release still resolve Discord attachment URLs during queued message processing, while the linked fix PR is open and unmerged.

## What This Changes

The issue asks Discord to save inbound attachments at receipt-time preflight so queued messages do not depend on expired CDN URLs when processed later.

## Best Possible Solution

Review and land the linked receipt-time media resolution PR, or an equivalent focused Discord plugin fix, after redacted live proof shows queued attachments survive dequeue.

## Reproduction Assessment

Yes at source level: current main preflights and enqueues the message, then resolves direct and forwarded media only inside processDiscordMessageInner. I did not run a live Discord delayed-queue CDN-expiry repro.

## Solution Assessment

Yes: resolving and persisting Discord media during receipt-time preflight, then reusing those local media facts during queued processing, is the narrow owner-boundary fix.
