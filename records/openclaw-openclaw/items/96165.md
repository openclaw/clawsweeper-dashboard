## Label Justifications

- `P1`: Queued Discord messages can silently lose attachment media before the agent turn, breaking an active channel workflow.
- `impact:data-loss`: Attachment bytes may never be saved locally after the Discord CDN URL expires.
- `impact:message-loss`: The agent can receive an inbound Discord message without the media content the user sent.

## Summary

Keep open: current main and the latest release still resolve Discord attachment media after the inbound run queue, while the open linked PR is the candidate fix and has not merged.

## What This Changes

The issue reports that queued Discord messages with attachments can lose media because Discord CDN URLs are fetched only after queue delay.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96183, or an equivalent focused Discord-plugin fix, after real Discord proof shows queued attachments survive the delay.

## Reproduction Assessment

Yes, at source level: current main enqueues a preflight context and only resolves direct and forwarded media later inside processDiscordMessageInner. I did not run a live Discord delayed-queue CDN-expiry repro in this read-only review.

## Solution Assessment

Yes for the linked direction: resolving and persisting Discord media during authorized preflight, then reusing those media facts after dequeue, is the narrow owner-boundary fix.
