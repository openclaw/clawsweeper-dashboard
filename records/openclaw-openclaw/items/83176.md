The branch adds a stale-running session reconciler, wires it into session abort/cleanup and diagnostic recovery, extends session status metadata, and adds session/gateway regression tests.

## Best Possible Solution

Keep the reconciler direction, but preserve the active-lane guard before stale reply-turn closure, add regression coverage for that ordering, and require redacted real Telegram or gateway-restart proof before merge.

## Reproduction Assessment

Source-reproducible for the review finding: the PR calls stale reply-turn closure before the active lane snapshot check, so an active lane with no embedded-run handle can be marked failed. I did not live-reproduce the original restarted Telegram topic blockage in this review.

## Solution Assessment

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve active lane protection before closing stale turns:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:286-288`
  - body: This new recovery runs before the existing active-lane guard. If a session has an active command lane but no embedded-run handle, a stale `replyTurnState` will call `sendRecoveryFallback` and mark the entry failed instead of returning `keep_lane`, so an in-flight Telegram turn can get an interruption message and corrupted state. Check the lane snapshot before calling `closeStaleReplyTurnIfNeeded`, or fold `activeCount` into that guard.
