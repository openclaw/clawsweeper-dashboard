- `impact:message-loss`: The reported failure delivers raw envelope JSON instead of the intended channel reply text.
- `impact:auth-provider`: The failure is tied to provider routing through `openai-completions` and OpenAI-compatible upstreams.

## Summary

Keep open. Current main and v2026.5.22 still treat detected tool-shaped assistant text as visible text, and the channel delivery path can emit that text when no structured messaging-tool execution evidence exists, so this is not a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to stop leaking internal `message` tool envelope JSON to Discord when an agent uses an OpenAI-compatible `openai-completions` provider and to route it like the Ollama provider.

## Best Possible Solution

Fix the OpenAI-compatible tool-call path so real upstream `tool_calls` produce structured `message` tool execution, and add guarded regression coverage that raw message-tool envelope JSON cannot reach channel-visible block replies.

## Reproduction Assessment

Yes for the failure class by source inspection: detected tool-shaped assistant text is logged but still allowed to continue as visible text. I did not establish a live Discord/OpenAI-compatible proxy reproduction on current main.

## Solution Assessment

Unclear; the requested provider parity is correct, but the safest fix likely belongs at the OpenAI-compatible transport or tool-call normalization boundary rather than a broad raw-JSON unwrap fallback at channel delivery.
