## Summary

Keep open: this is a new security-sensitive config option rather than refactor-only cleanup, and the PR currently lacks real behavior proof while leaving docs/tests/changelog and lockfile hygiene unresolved.

## What This Changes

This PR adds a `bootstrapTrustExternal` hook opt-in that lets `bootstrap-extra-files` read recognized bootstrap files from outside the workspace, plus an unrelated `pnpm-lock.yaml` importer change.

## Best Possible Solution

If maintainers want this capability, ship it as an explicit documented trust-boundary feature with guarded size enforcement, regression coverage, changelog, and clean lockfile state; otherwise keep the workspace-only bootstrap contract.

## Reproduction Assessment

Not applicable; this PR proposes a new opt-in capability, not a reported bug. Current main is source-verifiable: extra bootstrap files are intentionally loaded through the workspace-boundary guard and outside traversal is covered by tests.

## Solution Assessment

Unclear/no as written; external bootstrap paths may be useful, but the current direct-read implementation and undocumented config key are not the narrowest maintainable way to change this trust boundary.

  - body: The opt-in branch checks `stat.size` and then reads the entire file with `fs.readFile`. If the file is replaced or grows after the stat, the promised bootstrap size cap is no longer enforced, so this should use a guarded/opened read or otherwise cap the actual bytes read.

  - confidence: 0.9
